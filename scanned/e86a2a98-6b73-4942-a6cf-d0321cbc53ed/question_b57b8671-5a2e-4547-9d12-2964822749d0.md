RocksDB and AccountsDB open and close files dynamically during compaction, flush, and shrink operations. If the FD limit is set correctly at startup but RocksDB's internal FD cache (`max_open_files` config) is set higher than the OS limit, RocksDB will silently fail to open files during compaction, causing compaction to stall. A stalled compaction causes write amplification to grow unboundedly, eventually causing RocksDB writes to block, halting blockstore writes and transaction confirmation. Can an attacker trigger this by sending transactions that maximize write amplification (e.g., many small writes to many different accounts)?
