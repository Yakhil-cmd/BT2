### Title
Delegators Can Enter or Add to Delegation Pools for Disabled BTC Tokens, Locking Funds With Zero Reward Accrual - (File: `src/pool/pool.cairo`)

---

### Summary

`enter_delegation_pool` and `add_to_delegation_pool` in `pool.cairo` do not verify that the pool's underlying token is currently active before accepting delegator funds. When a BTC token is disabled via `disable_token`, delegators can still deposit into pools backed by that token. Their funds are transferred to the staking contract, but because disabled tokens are excluded from staking power calculations, the delegators earn zero rewards. Funds remain locked for the full exit wait window (default: one week) before they can be recovered.

---

### Finding Description

The `Pool` contract's `enter_delegation_pool` and `add_to_delegation_pool` functions each call `assert_staker_is_active()`, which only checks `!self.staker_removed.read()`. [1](#0-0) [2](#0-1) [3](#0-2) 

Neither function checks whether the pool's token is currently active. The token address is available via `self.token_dispatcher.read().contract_address`. [4](#0-3) 

The downstream call `add_stake_from_pool` in `staking.cairo` also performs no token-activity check: [5](#0-4) 

By contrast, the reward-distribution path explicitly skips disabled tokens: [6](#0-5) 

The `disable_token` function marks a token inactive starting at `current_epoch + K`: [7](#0-6) 

`is_active_token` uses this stored state to gate reward calculations: [8](#0-7) 

Because `set_member_balance` records the delegator's balance at `epoch + K`: [9](#0-8) 

a delegator who enters a pool after `disable_token` is called will have their balance take effect at exactly the epoch when the token becomes inactive, earning zero rewards from the moment their delegation is effective.

---

### Impact Explanation

A delegator who enters or adds to a BTC pool after its token has been disabled (or while it is being disabled) will:

1. Have their BTC transferred to the staking contract immediately.
2. Have their delegated balance recorded at `current_epoch + K`, which coincides with or follows the token's deactivation epoch.
3. Earn **zero rewards** for the entire period their funds are locked, because disabled tokens are excluded from staking power and reward calculations.
4. Be unable to recover funds until after the exit wait window (default one week) elapses following an `exit_delegation_pool_intent` call.

This constitutes **temporary freezing of funds** (High impact per the allowed scope) and **loss of unclaimed yield** for the affected delegators.

---

### Likelihood Explanation

- The `disable_token` function is callable by the security agent at any time.
- After disabling, the pool contract continues to accept new delegations with no on-chain warning or revert.
- Any unprivileged delegator who calls `enter_delegation_pool` or `add_to_delegation_pool` on a BTC pool after its token is disabled will trigger this outcome.
- The delegator has no way to detect the disabled state from within the pool contract itself, since the pool stores only the token address and not its active status.

---

### Recommendation

Add an active-token check in both `enter_delegation_pool` and `add_to_delegation_pool` in `pool.cairo` before accepting funds. The pool contract can query the staking contract for the token's active status, or the staking contract's `add_stake_from_pool` can assert `is_active_token` for the relevant token at the current epoch before updating balances and accepting the transfer.

---

### Proof of Concept

1. Security agent calls `staking.disable_token(btc_token_address)` at epoch `E`. Token becomes inactive at epoch `E + K`.
2. Delegator calls `pool.enter_delegation_pool(reward_address, amount)` on a BTC pool at epoch `E`.
3. `assert_staker_is_active()` passes (staker is not removed).
4. Delegator's BTC is transferred to the staking contract via `transfer_to_staking_contract` → `add_stake_from_pool`.
5. `set_member_balance` records the delegator's balance at epoch `E + K`.
6. At epoch `E + K`, `is_active_token(btc_token_address, E + K)` returns `false`.
7. `calculate_staker_pools_rewards` skips the pool: `if !self.is_active_token(:token_address, epoch_id: curr_epoch) { continue; }`.
8. Delegator earns zero rewards. Funds remain locked until `exit_delegation_pool_intent` + exit wait window completes. [10](#0-9) [7](#0-6) [11](#0-10) [9](#0-8)

        fn disable_token(ref self: ContractState, token_address: ContractAddress) {
            self.roles.only_security_agent();
            let is_active_opt: Option<(Epoch, bool)> = self.btc_tokens.read(token_address);
            assert!(is_active_opt.is_some(), "{}", Error::TOKEN_NOT_EXISTS);
            let (is_active_first_epoch, is_active) = is_active_opt.unwrap();
            let curr_epoch = self.get_current_epoch();
            assert!(curr_epoch >= is_active_first_epoch, "{}", Error::INVALID_EPOCH);
            assert!(is_active, "{}", Error::TOKEN_ALREADY_DISABLED);
            let next_is_active_first_epoch = self.get_epoch_plus_k();
            self.btc_tokens.write(token_address, (next_is_active_first_epoch, false));
            self.emit(TokenManagerEvents::TokenDisabled { token_address });
