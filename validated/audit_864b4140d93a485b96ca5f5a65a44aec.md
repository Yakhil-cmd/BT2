### Title
Declare Transactions Bypass `max_l2_gas_amount` Gateway Admission Check — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces an upper bound on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips that check for `Declare` transactions. An unprivileged user can submit a `Declare` with `l2_gas.max_amount = u64::MAX`, bypass the stateless gate, and have the transaction admitted to the mempool — violating the invariant the `max_l2_gas_amount` config knob is meant to enforce.

### Finding Description

`StatelessTransactionValidator::validate_resource_bounds` contains the following asymmetric branch:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production config sets `max_l2_gas_amount = 1_210_000_000`. [2](#0-1) 

For `Invoke` and `DeployAccount`, the check is enforced and tested: [3](#0-2) 

For `Declare`, the test `valid_l2_gas_amount_on_declare` explicitly asserts that a `Declare` with `max_amount = 200` (above a limit of `100`) passes validation — confirming the bypass is intentional at the test level but unguarded at the policy level: [4](#0-3) 

The stateless validator is the only place this upper-bound is enforced. The downstream stateful path (`check_fee_bounds`) only checks a **lower** bound (`minimal_gas_amount <= max_amount`), not an upper bound: [5](#0-4) 

`verify_can_pay_committed_bounds` checks whether the account balance covers `max_amount × max_price_per_unit`. With `max_amount = u64::MAX` and `min_gas_price = 8_000_000_000`, the committed fee is `≈ 1.47 × 10²⁸ wei`, which no real account can cover — so that path rejects the transaction. However, with a moderately elevated `max_amount` (e.g., `2 × 10⁹`, just above the `1.21 × 10⁹` limit) and a sufficiently funded account, the transaction passes all downstream checks and enters the mempool. [6](#0-5) 

### Impact Explanation

A funded account can submit `Declare` transactions with `l2_gas.max_amount` exceeding the configured gateway ceiling (`1_210_000_000`). Such transactions pass stateless validation, pass stateful fee-balance checks (because the account can cover the committed fee), and are admitted to the mempool. This violates the admission invariant the `max_l2_gas_amount` knob is designed to enforce and constitutes **mempool admission of transactions the gateway is explicitly configured to reject**.

Impact category: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

Any user with a funded account can trigger this with a single crafted `Declare` RPC call. No special privilege, no race condition, no peer cooperation required. The bypass is unconditional for all `Declare` transactions.

### Recommendation

Remove the `Declare` exemption and apply the same `max_l2_gas_amount` upper-bound check uniformly:

```rust
// In validate_resource_bounds, replace:
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(...);
}

// With:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The existing TODO comment (`// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.`) confirms the developers already identified this gap. [1](#0-0) 

### Proof of Concept

1. Fund an account with enough STRK to cover `(max_l2_gas_amount + 1) × min_gas_price` (e.g., `1_210_000_001 × 8_000_000_000 ≈ 9.68 × 10¹⁸ fri`).
2. Craft an `RpcDeclareTransactionV3` with:
   - `resource_bounds.l2_gas.max_amount = 1_210_000_001` (one above the configured ceiling)
   - `resource_bounds.l2_gas.max_price_per_unit = 8_000_000_000` (at `min_gas_price`)
   - A valid Sierra contract class
3. Submit via `starknet_addDeclareTransaction`.
4. **Stateless validator**: the `Declare` branch is taken; the `max_l2_gas_amount` check is skipped → `Ok(())`.
5. **Stateful validator**: `check_fee_bounds` passes (lower-bound only); `verify_can_pay_committed_bounds` passes (account is funded); blockifier executes the Declare normally.
6. The transaction is accepted into the mempool with `l2_gas.max_amount` exceeding the configured limit — the same transaction would have been rejected if submitted as `Invoke` or `DeployAccount`.

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}
