### Title
Cross-Chain Replay of Unprotected Legacy Transactions via Missing Chain ID in Signing Hash - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

### Summary
ZKsync OS accepts pre-EIP-155 "unprotected" legacy transactions (v=27 or v=28) whose signing hash is computed over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` with no chain ID. Because the hash is chain-agnostic, any such transaction signed on another EVM chain can be replayed verbatim on ZKsync OS.

### Finding Description

In `try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155 protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);   // only [nonce, gasPrice, gasLimit, to, value, data]
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: chain_id is mixed into the hash
    ...
    apply_u64_encoding_to_hash(chain_id, &mut hasher);
    ...
};
``` [1](#0-0) 

`is_eip155()` returns `false` when `v == 27 || v == 28`, i.e., the pre-EIP-155 case. [2](#0-1) 

The caller in `parse_and_compute_signed_hash` does **not** reject the unprotected variant — it simply wraps it as `Self::Legacy(tx, sig_data)` and returns the chain-ID-free hash for signature verification: [3](#0-2) 

All typed transactions (EIP-2930, EIP-1559, EIP-4844, EIP-7702) carry their `chain_id` in the payload and are explicitly checked against `expected_chain_id` before acceptance: [4](#0-3) 

The unprotected legacy path has no equivalent guard.

### Impact Explanation

An attacker who observes an unprotected legacy transaction (v=27/28) broadcast on any other EVM chain (e.g., Ethereum mainnet, a testnet, or another L2) can submit the identical raw bytes to ZKsync OS. If the originating account has:
- the same nonce on ZKsync OS (trivially true for fresh accounts, nonce=0), and
- a non-zero balance on ZKsync OS,

the transaction will pass signature verification and execute, transferring funds or invoking contracts without the user's consent on ZKsync OS. This is a direct, unprivileged, externally-reachable path to unauthorized fund loss.

### Likelihood Explanation

Pre-EIP-155 transactions are still generated by legacy tooling, hardware wallets in compatibility mode, and certain DeFi protocols. The attack requires only that the attacker observe such a transaction on a public chain and submit it to ZKsync OS — no special access is needed. The nonce-matching constraint is easily satisfied for accounts that have never transacted on ZKsync OS (nonce=0), which is the common case for users migrating from Ethereum.

### Recommendation

Reject unprotected legacy transactions outright in `parse_and_compute_signed_hash`. After the `is_eip155()` check, return an `InvalidChainId` (or a new `MissingChainId`) error instead of accepting the `Self::Legacy` variant:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    // Reject: unprotected legacy txs have no chain ID in their signing hash
    // and are replayable across all EVM chains.
    return Err(InvalidTransaction::InvalidChainId.into());
};
``` [5](#0-4) 

### Proof of Concept

1. On Ethereum mainnet, broadcast a legacy ETH transfer with v=27 (no EIP-155), nonce=0, from address `A` to address `B`.
2. Observe the raw RLP-encoded transaction bytes `T`.
3. Fund address `A` on ZKsync OS (nonce is 0 on ZKsync OS since `A` has never transacted there).
4. Submit `T` to ZKsync OS unchanged.
5. ZKsync OS calls `try_parse_and_hash_for_signature_verification` → `is_eip155() == false` → signing hash is computed over `[nonce=0, gasPrice, gasLimit, to=B, value, data]` with no chain ID.
6. `ecrecover` recovers address `A` — signature is valid.
7. The transaction executes on ZKsync OS, transferring funds from `A` to `B` without `A`'s consent on ZKsync OS. [6](#0-5) [3](#0-2)
