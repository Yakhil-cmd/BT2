### Title
Missing Minimum Threshold Validation in FROST Signing Enables Sub-Threshold Signature Production — (File: `src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` do not enforce a minimum threshold of 2, creating an inconsistency with the DKG entry point `assert_key_invariants` in `src/dkg.rs` which does enforce this bound. A malicious coordinator or library caller can supply `threshold = 0` or `threshold = 1` to the FROST signing path, bypassing the quorum requirement that was enforced at key-generation time.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` explicitly rejects any threshold below 2:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The public validation helper `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound — it never checks the lower bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

The `presign` entry point in the same file has the identical gap — it validates only `args.threshold.value() > participants.len()` and never `args.threshold.value() < 2`: [3](#0-2) 

The `ReconstructionLowerBound` wrapper type itself imposes no lower-bound constraint, so `ReconstructionLowerBound::from(0usize)` is a valid value that passes every check in the signing path: [4](#0-3) 

The same gap exists in the OT-based ECDSA `sign` function: it checks `participants.len() < threshold` (which trivially passes when `threshold = 0`) but never enforces `threshold >= 2`, unlike `validate_triple_inputs` which does: [5](#0-4) 

---

### Impact Explanation

**Mapped impact: High — Corruption of signing outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs; or Critical — Unauthorized creation of a valid threshold signature.**

When `threshold = 0` or `threshold = 1` is passed to the FROST signing path:

- The `frost_core::aggregate` call receives a threshold that is lower than the one used during DKG. If `frost_core` honours the caller-supplied threshold, the coordinator can aggregate a valid signature using only its own share, without collecting shares from the required quorum. This violates the threshold security property and constitutes unauthorized signature creation (Critical).
- If `frost_core` internally rejects the inconsistent threshold, the signing round terminates with an unexpected protocol error, permanently denying signing to all honest participants who have already committed their nonces (High).

Either outcome is reachable through the same missing check.

---

### Likelihood Explanation

The entry points (`presign`, `assert_sign_inputs`) are public library API functions. Any caller — including a malicious coordinator who controls which threshold value is passed — can supply `threshold = 0` or `threshold = 1`. No key compromise, side-channel, or external assumption is required. The inconsistency between DKG validation and signing validation makes this easy to trigger accidentally or deliberately.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already uses, in both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same fix to the OT-based ECDSA `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs`.

---

### Proof of Concept

```rust
use threshold_signatures::{
    frost::assert_sign_inputs,
    participants::Participant,
    ReconstructionLowerBound,
};

let participants = vec![Participant::from(1u32), Participant::from(2u32)];
let me = participants[0];
let coordinator = participants[0];

// threshold = 0 passes all current validation in assert_sign_inputs
let result = assert_sign_inputs(
    &participants,
    ReconstructionLowerBound::from(0usize), // should be rejected, is not
    me,
    coordinator,
);
assert!(result.is_ok()); // succeeds — no lower-bound check exists

// A coordinator now calls the FROST sign function with threshold = 0,
// potentially aggregating a valid signature from a single share,
// bypassing the quorum that was enforced during DKG (threshold >= 2).
```
