### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Bridge Transfer - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

The `withdrawToNear` and `withdrawToEthereum` functions in `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** calling the exit precompile, but the return value of the low-level `call` to the precompile is captured in `res` and **never checked**. If the precompile call fails for any reason, the function does not revert. The tokens are permanently destroyed on the EVM side while the corresponding NEP-141 transfer to NEAR (or withdrawal to Ethereum) never occurs, resulting in permanent loss of user funds.

---

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER CHECKED — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) 

The same unchecked-return pattern appears in `withdrawToEthereum`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) 

The `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) returns `ExitError` — causing the EVM `call` to return `0` — in multiple reachable conditions. For example, `parse_recipient` fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID` when the recipient bytes are not a valid NEAR account ID: [4](#0-3) 

Similarly, `get_nep141_from_erc20` fails with `ERR_TARGET_TOKEN_NOT_FOUND` if the ERC-20 is not registered: [5](#0-4) 

Because `res` is never inspected, the Solidity function returns normally in all these failure cases. The ERC-20 total supply is reduced by `amount` (via `_burn`) but the NEAR-side transfer promise is never created.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 tokens are irreversibly burned (`_burn` reduces `totalSupply` and the user's balance to zero).
- No NEAR promise is scheduled; the corresponding NEP-141 tokens are never transferred.
- There is no recovery path: the burned ERC-20 tokens cannot be re-minted (only the admin can call `mint`), and the NEAR-side balance is unchanged.

The user's bridged assets are permanently destroyed with no recourse.

---

### Likelihood Explanation

**Medium.**

The `recipient` parameter of `withdrawToNear` is arbitrary user-supplied `bytes`. Any of the following inputs cause the precompile to return `0`:

- A byte string that is not valid UTF-8.
- A string that is not a valid NEAR account ID (e.g., too long, contains forbidden characters, or is empty).
- A string whose parsed account ID portion exceeds NEAR's 64-character limit.

These are realistic mistakes for users interacting directly with the contract ABI (e.g., via a script or a frontend bug). The `withdrawToEthereum` path is less likely to fail since the recipient is a fixed-size 20-byte Ethereum address, but the same structural defect exists there too.

---

### Recommendation

Check the return value of every low-level `call` to the exit precompiles and revert on failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions to call the precompile **before** burning, so that a precompile failure causes a revert before any state change occurs.

---

### Proof of Concept

1. A NEP-141 token `token.near` is bridged to Aurora; its ERC-20 mirror is deployed at address `ERC20_ADDR` via `deploy_erc20_token`. The user holds `1000` units.
2. The user calls `ERC20_ADDR.withdrawToNear(bytes("!!!invalid!!!account!!!"), 1000)` — the recipient string contains characters forbidden in NEAR account IDs.
3. Inside `withdrawToNear`, `_burn(msg.sender, 1000)` executes successfully; the user's balance drops to `0` and `totalSupply` decreases by `1000`.
4. The assembly `call` to the `ExitToNear` precompile is made. Inside the precompile, `parse_recipient` calls `receiver_account_id.parse()` which fails, returning `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`. The EVM `call` returns `res = 0`. [6](#0-5) 

5. Because `res` is never checked, the Solidity function returns without reverting.
6. No `ExitToNear` promise log is emitted; no NEAR-side transfer is scheduled.
7. The user has lost `1000` tokens permanently: they are gone from the ERC-20 ledger and never arrive on NEAR.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
