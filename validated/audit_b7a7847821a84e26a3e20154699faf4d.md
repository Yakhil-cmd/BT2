### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Unverified Shares to Coordinator - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator aggregates participant-supplied `(norm_big_y, norm_big_c)` values by simple summation with no cryptographic verification that each share was honestly computed. A single malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that honest parties accept as valid, permanently destroying the correctness of the derived confidential key.

### Finding Description
In `do_ckd_coordinator`, the coordinator receives a `CKDOutput` tuple from every other participant and unconditionally adds each component into the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The protocol specification requires each participant to compute:

- `big_y = y_i * G` (random blinding factor times generator)
- `big_c = (x_i * H(pk, app_id)) + (y_i * app_pk)` (masked partial signature)
- Apply Lagrange weighting: `norm_big_y = lambda_i * big_y`, `norm_big_c = lambda_i * big_c` [2](#0-1) 

There is no zero-knowledge proof, commitment, or any other mechanism to enforce that a participant's submitted `(norm_big_y, norm_big_c)` was computed using their actual private share `x_i` and a valid blinding factor `y_i`. The coordinator has no way to distinguish a correctly computed share from an arbitrary pair of group elements.

The correct aggregated output satisfies:
```
C_total - app_sk * Y_total = msk * H(pk, app_id)
```
because the blinding terms cancel. If any participant substitutes arbitrary `(Y', C')`, this cancellation breaks and the `unmask(app_sk)` call produces a wrong key. [3](#0-2) 

### Impact Explanation
A malicious participant sends a crafted `(norm_big_y, norm_big_c)` pair to the coordinator. The coordinator sums all contributions and returns a `CKDOutput` that every honest party accepts as the protocol result. When the TEE calls `unmask(app_sk)`, it obtains a value that is not equal to `msk * H(pk, app_id)`. The derived confidential key is permanently wrong for that protocol run. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation
Any single participant in the CKD session is sufficient to trigger this. Participants are untrusted by the threshold model; the entire point of the protocol is to tolerate malicious participants up to the threshold. The attack requires only that the malicious party deviate from the protocol when sending its private message to the coordinator — a trivial local modification with no detectable on-wire signature. Likelihood: **Medium**.

### Recommendation
Each participant must accompany their `(norm_big_y, norm_big_c)` submission with a non-interactive zero-knowledge proof of correct computation — specifically, a proof that `big_c` was formed as `x_i * H + y_i * app_pk` for the same `y_i` used to compute `big_y = y_i * G`, and that `x_i` corresponds to the participant's committed public share `x_i * G2` from the DKG output. A standard sigma protocol (e.g., a Chaum–Pedersen proof of discrete-log equality across two bases) suffices. The coordinator must verify all proofs before aggregating.

### Proof of Concept
1. Run a CKD session with participants `{P1, P2, P3}` and coordinator `P1`.
2. `P2` (malicious) replaces its honest `(norm_big_y, norm_big_c)` with `(G, G)` (the generator point in both slots).
3. `P2` sends `(G, G)` to `P1` via `chan.send_private`.
4. `P1`'s `do_ckd_coordinator` adds `G` to both running sums without any check.
5. The returned `CKDOutput` has `big_y = Y_honest + G` and `big_c = C_honest + G`.
6. `unmask(app_sk)` computes `(C_honest + G) - app_sk * (Y_honest + G)`, which equals `msk * H(pk, app_id) + G - app_sk * G` — a value that differs from the correct key by `(1 - app_sk) * G`.
7. All honest parties accept this corrupted output; there is no detection mechanism. [4](#0-3) [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
