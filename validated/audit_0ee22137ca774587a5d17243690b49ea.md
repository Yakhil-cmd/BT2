### Title
Publicly Callable `update_rewards` with `disable_rewards` Parameter Enables Permanent Freezing of Staker Yield — (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` in `StakingRewardsManagerImpl` is callable by any unprivileged address. It accepts a `disable_rewards: bool` parameter. When called with `disable_rewards: true`, it writes the current block number to the global `last_reward_block` storage slot **before** the early-return guard, consuming the one-call-per-block budget without distributing any rewards. An attacker can repeat this every block to permanently freeze consensus-era yield for all stakers.

---

### Finding Description

`update_rewards` is exposed as a public ABI entry point via `#[abi(embed_v0)]`: [1](#0-0) 

The only caller checks are `general_prerequisites()` (contract not paused, caller not zero address) and a per-block guard: [2](#0-1) 

Critically, `last_reward_block` is written to storage **before** the `disable_rewards` branch: [3](#0-2) 

Because `last_reward_block` is a single global slot, once it is set to the current block number, the assertion `current_block_number > self.last_reward_block.read()` will fail for every subsequent call in the same block. The attacker therefore consumes the entire per-block reward slot without any rewards being distributed.

The `_update_rewards` path that actually credits stakers is only reached when `disable_rewards` is `false` **and** the system is post-consensus: [4](#0-3) 

The attacker only needs to supply any valid, active staker address with non-zero balance — information that is fully public from on-chain events and the `stakers` vector: [5](#0-4) 

---

### Impact Explanation

In V3 consensus-rewards mode, `update_rewards` is the sole mechanism by which block rewards are distributed to stakers and their delegation pools. By calling it with `disable_rewards: true` at every block, an attacker permanently prevents all stakers from accumulating unclaimed yield. The rewards are not deferred — the per-block reward window is irrecoverably consumed. This matches **High: Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The attack requires no special privilege, no token holdings, and no flash loan. The attacker only needs to:
1. Know any valid active staker address (trivially available from `NewStaker` events or the public `stakers` vector).
2. Pay the gas cost of one transaction per block.

The attack is therefore cheap, permissionless, and fully predictable. Likelihood is **High**.

---

### Recommendation

Restrict `update_rewards` to callers with a trusted role (e.g., the staker themselves, or a designated rewards-manager role), mirroring how `update_rewards_from_attestation_contract` is gated to the attestation contract: [6](#0-5) 

At minimum, move the `self.last_reward_block.write(current_block_number)` call to **after** the `disable_rewards` guard so that a no-op call does not consume the per-block slot.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker reads any valid staker address `S` from on-chain state.
3. At every new block `B`, attacker sends: `update_rewards(S, disable_rewards: true)`.
4. Inside the call:
   - `current_block_number (B) > last_reward_block` → assertion passes.
   - Staker `S` is active with non-zero balance → assertions pass.
   - `last_reward_block` is written to `B`. ← **slot consumed**
   - `disable_rewards == true` → early return, zero rewards distributed.
5. Any legitimate staker who now calls `update_rewards` for block `B` hits `REWARDS_ALREADY_UPDATED` and reverts.
6. Block `B`'s reward window is permanently lost for all stakers and their pools.
7. Repeating step 3 every block freezes all staker yield indefinitely.
