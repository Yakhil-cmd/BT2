### Title
Missing Zero Address Validation for `reward_address` Allows Permanent Loss of Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
The `stake()`, `change_reward_address()` (staking), `enter_delegation_pool()`, and `change_reward_address()` (pool) functions accept a `reward_address` parameter without validating it against `address(0)`. A staker or delegator who passes or later sets `reward_address = 0` will have all future claimed rewards permanently transferred to the zero address, resulting in irreversible loss of yield.

### Finding Description
The vulnerability class from the external report is **missing input validation on critical state variables**. In the Starknet Staking system, `reward_address` is the address to which all accrued staking or delegation rewards are transferred when `claim_rewards()` is called. It is stored in `InternalStakerInfo` (staking contract) and `InternalPoolMemberInfo` (pool contract) and can be set at entry time or updated later.

Four entry points accept `reward_address` without a zero-address guard:

**1. `stake()` in `src/staking/staking.cairo`:** [1](#0-0) 
The only check on `reward_address` is that it is not a registered token address. There is no `assert!(reward_address.is_non_zero(), ...)`.

**2. `change_reward_address()` in `src/staking/staking.cairo`:** [2](#0-1) 
Same pattern — only the token-address guard is present.

**3. `enter_delegation_pool()` in `src/pool/pool.cairo`:** [3](#0-2) 
Only `amount != 0` and `token_address != reward_address` are checked.

**4. `change_reward_address()` in `src/pool/pool.cairo`:** [4](#0-3) 
Only the token-address guard is present.

The codebase already defines `GenericError::ZERO_ADDRESS` and uses it in other contexts, confirming the pattern is known but was not applied here: [5](#0-4) 

### Impact Explanation
When `claim_rewards()` is called for a staker or pool member whose `reward_address` is `address(0)`, the reward tokens are transferred to the zero address and are permanently unrecoverable. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

The `reward_address` field is the sole destination for reward payouts; there is no fallback or recovery mechanism once it is set to zero and rewards are claimed.

### Likelihood Explanation
The entry path is fully unprivileged: any staker calling `stake()` or `change_reward_address()`, and any delegator calling `enter_delegation_pool()` or `change_reward_address()` on the pool, can trigger this. Realistic causes include:

- A frontend bug that passes an uninitialized or empty address field.
- A user scripting a transaction with a zero-initialized struct.
- A griefing actor who deliberately sets their own reward address to zero to permanently burn their yield (no profit motive, but real damage to the user).

### Recommendation
Add a non-zero guard on `reward_address` in all four functions, consistent with the existing `ZERO_ADDRESS` error:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply this check in:
- `stake()` in `src/staking/staking.cairo`
- `change_reward_address()` in `src/staking/staking.cairo`
- `enter_delegation_pool()` in `src/pool/pool.cairo`
- `change_reward_address()` in `src/pool/pool.cairo`

### Proof of Concept

1. Deploy the system normally.
2. Call `stake(reward_address: 0, operational_address: <valid>, amount: <min_stake>)` — the call succeeds because the only `reward_address` check is `!does_token_exist(reward_address)`, and `address(0)` is not a registered token.
3. Advance epochs and attest to accrue rewards.
4. Call `claim_rewards(staker_address)` — the contract transfers the accrued STRK to `address(0)`.
5. The yield is permanently lost; the staker's `unclaimed_rewards_own` resets to zero with no tokens received.

The same sequence applies to a delegator via `enter_delegation_pool(reward_address: 0, amount: ...)` followed by `claim_rewards(pool_member)` on the pool contract. [6](#0-5) [7](#0-6)
