### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory owner enforces an admin fee ceiling through `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`, checked inside `setPoolAdminFees`. However, the parallel admin path `setPoolBinAdditionalFees` sets per-bin additional buy/sell fees with **no cap check at all**, allowing the pool admin to impose arbitrarily high per-bin fees (up to `uint16.max = 65 535 E6 ≈ 6.55 %` per bin) regardless of what the factory owner has set as the ceiling.

---

### Finding Description

`setPoolAdminFees` enforces the factory cap:

```solidity
// MetricOmmPoolFactory.sol  lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, callable by the same pool admin, forwards directly to the pool with **no cap validation**:

```solidity
// MetricOmmPoolFactory.sol  lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check:

```solidity
// MetricOmmPool.sol  lines 464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

These per-bin fees are applied **additively** to the oracle-derived base fee in every swap step:

```solidity
// MetricOmmPool.sol  line 910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

**Bypass sequence (analog to passport mint → transfer → mint):**

1. Factory owner lowers `maxAdminSpreadFeeE6` to a small value (e.g. `1 000` = 0.1 %) to protect traders.
2. Pool admin calls `setPoolAdminFees` — correctly blocked at 0.1 %.
3. Pool admin calls `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` — no check, succeeds.
4. Every swap touching that bin now pays an additional ≈ 6.55 % on top of the oracle spread, far exceeding the factory owner's intended ceiling.

The factory owner's cap is checked in one path (`setPoolAdminFees`) but is entirely absent from the parallel path (`setPoolBinAdditionalFees`) that achieves the same economic effect.

---

### Impact Explanation

Traders executing swaps in the targeted bin pay fees materially above the factory-owner-set ceiling. The excess fee accrues as surplus inside the pool and is collected by the pool admin via `collectPoolFees`. This is a direct, quantifiable loss of trader funds (up to ≈ 6.55 % of swap notional per bin) caused by the pool admin exceeding the factory-enforced boundary — an admin-boundary break with direct fund loss.

---

### Likelihood Explanation

The pool admin is a single EOA or contract that can call `setPoolBinAdditionalFees` at any time with no timelock, no co-signer, and no on-chain notice. The factory owner has no mechanism to detect or prevent this after pool deployment. Likelihood is **high** whenever a pool admin acts adversarially within the semi-trusted model.

---

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` mirroring the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the factory owner can set independently, and enforce it here.

---

### Proof of Concept

```
Setup:
  factory.setFeeCaps(200_000, 500, 1_000_000, 500_000)
  // maxAdminSpreadFeeE6 = 500 (0.05 %)

Step 1 — capped path blocked:
  factory.setPoolAdminFees(pool, 501, 0)
  // reverts: AdminFeeTooHigh ✓

Step 2 — uncapped path succeeds:
  factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)
  // succeeds; bin 0 now carries +6.5535 % buy/sell fee

Step 3 — trader swaps through bin 0:
  pool.swap(recipient, true, 1_000e18, 0, ...)
  // trader pays oracle spread + 6.5535 % extra
  // excess accrues as surplus, collected by pool admin
  // factory owner's 0.05 % cap is completely bypassed
```
