### Title
Underestimated Minimal Gas for Declare Transactions Causes Incorrect Pre-Validation Fee Bound Check - (File: crates/blockifier/src/fee/gas_usage.rs)

### Summary

`estimate_minimal_gas_vector` systematically underestimates the minimum gas required for `Declare` transactions by hardcoding `n_compiled_class_hash_updates: 0` instead of `1`. A `Declare` transaction always writes a `(class_hash → compiled_class_hash)` entry to state, so the DA cost for that entry is always incurred. The underestimation causes `check_fee_bounds` to accept `Declare` transactions whose resource bounds are below the true minimum, mirroring the external report's biased-estimator pattern: a denominator (or count) that is systematically too small produces a floor that is too low, and transactions that should be rejected are admitted.

### Finding Description

In `estimate_minimal_gas_vector` the state-change count for `Declare` is:

```rust
// crates/blockifier/src/fee/gas_usage.rs  lines 168-174
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,   // ← wrong; should be 1
    n_modified_contracts: 1,
},
``` [1](#0-0) 

The TODO comment confirms the developers know the correct value is `1`. Every V2/V3 `Declare` transaction writes one `compiled_class_hash` entry; `get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` felts to the DA segment:

```rust
// crates/blockifier/src/fee/gas_usage.rs  lines 33-34
onchain_data_segment_length += state_changes_count.n_compiled_class_hash_updates * 2;
``` [2](#0-1) 

With `n_compiled_class_hash_updates: 0` the minimum estimate is short by **2 felts** of DA cost (≈ `2 × DATA_GAS_PER_FIELD_ELEMENT` in KZG mode, or `2 × SHARP_GAS_PER_DA_WORD` in calldata mode).

`estimate_minimal_gas_vector` is called directly inside `check_fee_bounds`, which is the pre-validation gate that decides whether a transaction's resource bounds are sufficient to proceed:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 374-382
fn check_fee_bounds(&self, tx_context: &TransactionContext) -> ... {
    let minimal_gas_amount_vector = estimate_minimal_gas_vector(
        &tx_context.block_context,
        self,
        &tx_context.get_gas_vector_computation_mode(),
    );
    ...
``` [3](#0-2) 

`check_fee_bounds` is called unconditionally (when `charge_fee` is true) inside `perform_pre_validation_stage`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 363-366
if self.execution_flags.charge_fee {
    self.check_fee_bounds(tx_context)?;
    verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
}
``` [4](#0-3) 

Because the floor is too low, a `Declare` transaction whose `l1_data_gas` (or `l1_gas`) resource bound falls in the gap `[underestimated_min, actual_min)` passes `check_fee_bounds`, is admitted to the mempool, and is handed to the batcher. During execution the real DA cost—which includes the two felts for the compiled-class-hash entry—is computed and charged. The post-execution report then finds the actual gas exceeds the declared bound, reverts the transaction, and charges the user a revert fee while consuming their nonce.

The actual state-change count used at execution time is computed from the real state diff via `StateChanges::count_for_fee_charge`, which correctly counts `compiled_class_hashes.len()`:

```rust
// crates/blockifier/src/state/cached_state.rs  lines 926-927
n_compiled_class_hash_updates: self.state_maps.compiled_class_hashes.len(),
``` [5](#0-4) 

So the pre-validation estimate and the post-execution accounting are inconsistent for every `Declare` transaction.

### Impact Explanation

Any `Declare` transaction whose resource bounds are set to the value returned by `estimate_minimal_gas_vector` (the documented lower bound) will:
1. Pass `check_fee_bounds` and be admitted.
2. Fail post-execution because the actual DA cost is higher.
3. Cause the user to pay a revert fee and lose their nonce.

This is systematic economic harm to every user who relies on the published minimum estimate for `Declare` transactions. It also means the sequencer admits and executes transactions it should have rejected at the pre-validation stage, matching the "incorrect fee/gas accounting with economic impact" criterion.

### Likelihood Explanation

The bug is present in every code path that calls `estimate_minimal_gas_vector` for a `Declare` transaction with `charge_fee = true`. The gap is small (2 DA felts) but deterministic and always present. Any wallet or SDK that uses the blockifier's own minimum-gas helper to construct `Declare` resource bounds will produce a bound that is too low.

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` for `Declare` transactions in `estimate_minimal_gas_vector`, as the TODO comment already states:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1,  // was 0; Declare always writes compiled_class_hash
    n_modified_contracts: 1,
},
```

### Proof of Concept

1. Call `estimate_minimal_gas_vector` for a V3 `Declare` transaction with `GasVectorComputationMode::All` and `use_kzg_da = true`.
2. Observe the returned `l1_data_gas` component is `0` (no compiled-class-hash DA cost counted).
3. Submit a `Declare` transaction with `l1_data_gas.max_amount` set to exactly that value.
4. `check_fee_bounds` passes (underestimated floor ≤ user bound).
5. Execution proceeds; the real state diff includes one `compiled_class_hash` write → `2 × DATA_GAS_PER_FIELD_ELEMENT` additional L1 data gas.
6. `PostExecutionReport` finds `actual_l1_data_gas > max_amount`, reverts the transaction, and charges the revert fee.
7. User's nonce is incremented and fee is deducted despite the transaction failing.
