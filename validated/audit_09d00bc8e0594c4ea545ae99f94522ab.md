### Title
Malicious CKD Participant Can Submit Arbitrary Unverified Shares to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates per-participant contributions `(λ_i·Y_i, λ_i·C_i)` by simple addition with no zero-knowledge proof or algebraic verification that each contribution was honestly computed. Any single legitimate-but-malicious participant can substitute arbitrary group elements, permanently corrupting the `CKDOutput` `(Y, C)` delivered to the application. The app then derives a wrong or unusable confidential key `s`, with no ability to detect or recover from the corruption.

---

### Finding Description

**Root cause — missing proof-of-correct-computation on participant shares.**

In `do_ckd_coordinator` the coordinator collects every participant's `(norm_big_y, norm_big_c)` and unconditionally adds them:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The protocol specification requires each node `i` to compute:

- `Y_i = y_i · G` (fresh random blinding)
- `C_i = x_i · H(pk, app_id) + y_i · A` (ElGamal encryption of the BLS share)

and send `(λ_i · Y_i, λ_i · C_i)` to the coordinator. [2](#0-1) 

No Schnorr proof, DLEQ proof, or any other zero-knowledge argument is attached to the message. The coordinator has no mechanism to verify that the received pair is of the form `(λ_i · y_i · G, λ_i · (x_i · H(pk,app_id) + y_i · A))` for any consistent `y_i`. The `recv_from_others` call authenticates the *sender identity* (TLS-authenticated channel per `docs/network-layer.md`) but does not authenticate the *content* of the message. [3](#0-2) 

**Exploit path:**

1. A malicious participant `P_m` is a legitimate member of the CKD participant set.
2. Instead of computing `(λ_m · Y_m, λ_m · C_m)` honestly, `P_m` sends arbitrary group elements `(Y_m', C_m')` to the coordinator.
3. The coordinator sums all contributions including the malicious ones:
   - `Y_final = Y_honest_sum + Y_m'` (corrupted)
   - `C_final = C_honest_sum + C_m'` (corrupted)
4. The coordinator returns `CKDOutput { big_y: Y_final, big_c: C_final }`.
5. The TEE application calls `ckd_output.unmask(app_sk)` and obtains `C_final - a·Y_final`, which is not equal to `msk · H(pk, app_id)`. [4](#0-3) 

The application has no way to detect the corruption because it cannot verify the BLS signature `s` without an independent copy of `msk · H(pk, app_id)`.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

Every CKD request that includes the malicious participant produces a wrong confidential key `s`. The TEE application silently derives an incorrect secret, breaking the determinism guarantee of CKD (the same `app_id` will yield a different, wrong key on every run that includes the malicious node). Because the protocol requires all `n` participants to contribute (no threshold aggregation is applied in the CKD round), a single malicious participant can corrupt 100% of CKD requests indefinitely.

---

### Likelihood Explanation

**High.** Any one of the `n` MPC nodes participating in CKD can mount this attack. No privileged access, leaked keys, or cryptographic break is required — only the ability to send a crafted message over the already-authenticated channel, which every participant can do by construction. The attack is silent (no error is returned to honest parties) and persistent (every future CKD request is affected until the malicious node is removed).

---

### Recommendation

Attach a **DLEQ (Discrete-Log Equality) proof** to each participant's contribution, proving that the same secret scalar `y_i` was used in both `Y_i = y_i · G` and the `y_i · A` term inside `C_i`, and that `C_i - y_i · A = x_i · H(pk, app_id)` is consistent with the participant's public key share. Concretely:

1. Each participant computes and sends `(λ_i · Y_i, λ_i · C_i, π_i)` where `π_i` is a Schnorr/DLEQ proof of knowledge of `y_i` and consistency with the public share `X_i = x_i · G_2`.
2. The coordinator verifies every `π_i` before adding the contribution; any invalid proof causes the protocol to abort and identifies the malicious participant.

This is the standard approach used in threshold BLS and verifiable secret sharing to prevent malicious share substitution.

---

### Proof of Concept

**Setup:** 3-participant CKD session, participants `P_1`, `P_2` (honest), `P_3` (malicious). Coordinator is `P_1`.

**Honest execution (expected):**
- Each `P_i` computes `(λ_i · Y_i, λ_i · C_i)` per `compute_signature_share`.
- Coordinator sums: `Y = Σ λ_i Y_i`, `C = Σ λ_i C_i`.
- App computes `s = C - a·Y = msk · H(pk, app_id)`. ✓

**Malicious execution:**
- `P_3` replaces its contribution with `(Y_3', C_3') = (G_1, G_1)` (arbitrary non-zero points).
- Coordinator sums: `Y' = λ_1 Y_1 + λ_2 Y_2 + G_1`, `C' = λ_1 C_1 + λ_2 C_2 + G_1`.
- App computes `s' = C' - a·Y' = msk·H(pk,app_id) + G_1 - a·G_1 = msk·H(pk,app_id) + (1-a)·G_1`.
- `s' ≠ msk·H(pk,app_id)` for any `a ≠ 1`. The derived key is wrong and non-deterministic across sessions. No error is raised anywhere in the protocol.

The existing test at `src/confidential_key_derivation/protocol.rs` lines 210–283 confirms the honest path works; the same test modified to inject `(G_1, G_1)` from one participant would produce a mismatched `confidential_key` vs `expected_confidential_key`, demonstrating the corruption. [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
