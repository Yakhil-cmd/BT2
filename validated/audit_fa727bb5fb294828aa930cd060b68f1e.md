### Title
Gateway Admits Invoke Transaction with Invalid Signature via Validation-Skip Logic for Deployed Accounts with Nonce Zero - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the `__validate__` entry-point call (signature verification) for invoke transactions with nonce=1 when the account's on-chain nonce is 0 and the account has any transaction in the mempool. This UX feature exists to support the simultaneous `deploy_account + invoke` pattern. However, the check does not verify that the account is actually undeployed. An attacker with an already-deployed account whose on-chain nonce is 0 can submit a valid invoke(nonce=0) to seed their address into the mempool, then submit an invoke(nonce=1) with an invalid signature. The gateway skips `__validate__` for the second transaction and admits it to the mempool without signature verification.

---

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations` to decide whether to skip the blockifier `__validate__` entry point: [1](#0-0) 

`run_pre_validation_checks` calls `skip_stateful_validations` after the nonce and resource-bounds checks pass: [2](#0-1) 

`skip_stateful_validations` returns `true` (skip `__validate__`) when the transaction is an Invoke with nonce=1, the account's on-chain nonce is 0, and `account_tx_in_pool_or_recent_block` returns `true`: [3](#0-2) 

The comment states the intent: verify a `deploy_account` exists for the account. The check used is `account_tx_in_pool_or_recent_block`, which returns `true` if the account has **any** transaction in the pool or recent block — not specifically a `deploy_account`: [4](#0-3) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, causing the blockifier's `StatefulValidator` to skip the `__validate__` entry point entirely: [5](#0-4) 

The `StatefulValidator::perform_validations` path for Invoke transactions calls `validate_tx` only when `execution_flags.validate` is `true`: [6](#0-5) 

And `validate_tx` returns `Ok(None)` immediately when `validate` is `false`: [7](#0-6) 

The `validate_by_mempool` call that precedes the skip check only validates nonce ordering and fee escalation — it does not verify signatures: [8](#0-7) 

**Broken invariant**: The gateway must not admit an invoke transaction whose `__validate__` entry point would fail. The `skip_stateful_validations` function is supposed to apply only when the account is undeployed (deploy_account pending). It incorrectly also applies when the account is already deployed with on-chain nonce=0 and the attacker has seeded any transaction into the mempool.

---

### Impact Explanation

An attacker can cause the gateway to admit an invoke transaction with an invalid (or forged) signature into the mempool without running the account's `__validate__` entry point. This directly satisfies the High impact criterion: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The admitted invalid transaction will fail during batcher execution (since `execution_flags.validate` is `true` during block building), but the gateway's admission invariant — that every admitted invoke has passed its account's signature check — is broken. This enables mempool pollution, wasted batcher resources, and potential griefing of the sequencer.

---

### Likelihood Explanation

The preconditions are low-cost and fully attacker-controlled:
1. The attacker needs a deployed account with on-chain nonce=0 (trivially achievable by deploying a new account and never transacting).
2. The attacker submits one valid invoke(nonce=0) — this passes all checks including `__validate__` — to seed their address into the mempool.
3. The attacker immediately submits an invoke(nonce=1) with an invalid signature. The nonce check passes (`0 <= 1 <= 200`), `validate_by_mempool` passes (no duplicate, no fee escalation issue), and `skip_stateful_validations` returns `true` because `tx.nonce()==1`, `account_nonce==0`, and `account_tx_in_pool_or_recent_block` returns `true`.

No privileged access, no special network position, and no race condition is required.

---

### Recommendation

`skip_stateful_validations` must additionally verify that the account is **not yet deployed** before skipping `__validate__`. Concretely, before returning the result of `account_tx_in_pool_or_recent_block`, the function should check that no contract class is deployed at the sender address (i.e., `state.get_class_hash_at(sender_address) == ClassHash::default()`). Only if the account is undeployed should the skip be permitted.

Alternatively, `account_tx_in_pool_or_recent_block` should be replaced with a more specific check that verifies a `deploy_account` transaction (not just any transaction) is pending for the sender address.

---

### Proof of Concept

**Step-by-step trigger:**

1. Attacker deploys account contract `A` (on-chain nonce becomes 0).
2. Attacker submits `invoke_v3` with `sender_address=A`, `nonce=0`, valid signature → passes `validate_nonce` (0 ≤ 0 ≤ 200), passes `validate_by_mempool`, passes `run_validate_entry_point` (signature valid). Address `A` is now in the mempool.
3. Attacker submits `invoke_v3` with `sender_address=A`, `nonce=1`, **invalid/arbitrary signature**.
   - `validate_nonce`: `0 ≤ 1 ≤ 200` → passes. [9](#0-8) 
   - `validate_by_mempool`: no duplicate hash, no fee escalation issue → passes.
   - `skip_stateful_validations`: `tx.nonce()==1` ✓, `account_nonce==0` ✓, `account_tx_in_pool_or_recent_block(A)==true` ✓ → returns `true`. [10](#0-9) 
   - `run_validate_entry_point`: `execution_flags.validate = false` → `__validate__` is never called. [11](#0-10) 
4. The invalid-signature invoke(nonce=1) is forwarded to the mempool and admitted. The gateway's admission invariant is violated.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
