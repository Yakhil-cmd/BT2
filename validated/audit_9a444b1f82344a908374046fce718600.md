### Title
CKD Coordinator Blindly Aggregates Unverified Participant Shares, Allowing Any Malicious Participant to Corrupt the Derived Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function aggregates `(big_y, big_c)` ElGamal shares from all participants with no cryptographic verification of their correctness. A single malicious participant can inject arbitrary group elements, causing the coordinator to produce a corrupted `CKDOutput` that delivers an incorrect derived key to the application. This is the direct analog of H-02: just as `TransformController.sol` assumed only `outputToken` would be returned and did not account for other tokens emitted by transformers, `do_ckd_coordinator` assumes every participant's share is honestly computed and does not account for maliciously crafted shares.

---

### Finding Description

In `do_ckd_coordinator` (lines 44–57 of `src/confidential_key_derivation/protocol.rs`), the coordinator receives a `CKDOutput` from every other participant and unconditionally accumulates it:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each honest participant is supposed to compute (in `compute_signature_share`, lines 148–181):

```
norm_big_y_i = λ_i · y_i · G₁
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

so that the coordinator's aggregate satisfies:

```
big_y = Σ norm_big_y_i
big_c = Σ norm_big_c_i = msk · H(pk ‖ app_id) + (Σ λ_i · y_i) · app_pk
```

and the application can unmask (line 54 of `mod.rs`):

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar   // = msk · H(pk ‖ app_id)
}
```

**There is no verification step.** The coordinator holds only the master public key (`key_pair.public_key`), not the individual verification shares `pk_i = x_i · G₂`. It therefore cannot check whether a received `(norm_big_y_i, norm_big_c_i)` is consistent with participant `i`'s committed key share. A malicious participant simply sends:

```
(norm_big_y_i + Δ_y,  norm_big_c_i + Δ_c)   where Δ_c ≠ app_sk · Δ_y
```

The coordinator sums these without complaint. The final `CKDOutput` becomes:

```
big_y' = big_y + Δ_y
big_c' = big_c + Δ_c
```

and the application's unmask yields:

```
big_c' − app_sk · big_y' = msk · H(pk ‖ app_id) + (Δ_c − app_sk · Δ_y)  ≠  msk · H(pk ‖ app_id)
```

The derived key is silently wrong. The application has no way to detect this because it does not know `msk` directly.

The structural parallel to H-02 is exact:

| H-02 (0x) | This finding (CKD) |
|---|---|
| `transformERC20` only tracks `outputToken` balance delta | `do_ckd_coordinator` only sums received shares |
| Transformer sub-components can return other tokens that are never accounted for | Participant sub-components can return arbitrary group elements that are never verified |
| Untracked tokens silently alter the user's real balance | Unverified shares silently corrupt the derived key |

---

### Impact Explanation

A single malicious participant (one of the `n` nodes in the CKD call) can cause every honest party—including the application—to accept a corrupted `CKDOutput`. The application receives and uses a derived key that does not equal `msk · H(pk ‖ app_id)`. This is **corruption of CKD outputs so honest parties accept unusable cryptographic outputs**, matching the allowed High impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

The attacker needs only to be a registered participant in the `participants` list passed to `ckd()`. No key material, side-channel, or external compromise is required. The attack is a single-round, one-message deviation: replace the honest `(norm_big_y, norm_big_c)` with any other pair of `G1` points before calling `chan.send_private`. The coordinator has no mechanism to reject or detect this. Likelihood is **High** given any adversarial participant.

---

### Recommendation

Add per-share verification before accumulation. The standard approach for this ElGamal-over-BLS construction is a **DLEQ (Discrete Log Equality) proof** attached to each share, proving that the same secret `y_i` was used in both `big_y_i = y_i · G₁` and the `y_i · app_pk` term inside `big_c_i`. Alternatively, if individual verification keys `pk_i = x_i · G₂` are made available to the coordinator (e.g., stored alongside `KeygenOutput`), a pairing check can verify the BLS component:

```
e(big_c_i − big_y_i · (app_pk / G₁), G₂)  ==  e(H(pk ‖ app_id), pk_i)
```

Either approach ensures that a malicious participant's share is rejected before it contaminates the aggregate.

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
