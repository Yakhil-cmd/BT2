### Title
Missing Caller Authorization on `update_rewards` Allows Any User to Permanently Freeze Staker Yield - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in the `StakingRewardsManagerImpl` implementation is callable by any address, despite the protocol specification explicitly restricting it to "Only starkware sequencer." Because the function unconditionally writes `last_reward_block` to the current block number before distributing rewards, an unprivileged attacker can call it with `disable_rewards: true` every block, permanently preventing the legitimate sequencer from distributing block rewards to any staker.

### Finding Description
The spec at `docs/spec.md` lines 1644–1645 states:

> **access control**: Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` lines 1449–1507 contains no caller identity check whatsoever:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker existence checks ...

    // Update last block rewards.  <-- written unconditionally
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;   // <-- exits without distributing rewards
    }
    // ... reward distribution ...
}
```

`general_prerequisites()` only checks that the contract is not paused — it does not verify the caller is the sequencer.

**Attack path:**

1. Attacker monitors the mempool (or simply calls proactively each block).
2. Attacker calls `update_rewards(any_valid_staker, disable_rewards: true)`.
3. `last_reward_block` is set to `current_block_number` with no rewards distributed.
4. When the legitimate sequencer attempts `update_rewards` in the same block, the assertion `current_block_number > self.last_reward_block.read()` fails with `REWARDS_ALREADY_UPDATED`, and the call reverts.
5. The block's rewards are permanently lost — there is no mechanism to retroactively distribute rewards for a skipped block.
6. Repeating this every block permanently freezes all staker yield.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

`last_reward_block` is a single global storage variable. One attacker call per block is sufficient to deny rewards to all stakers for that block. Because there is no catch-up or retroactive reward mechanism, the lost yield is unrecoverable. An attacker with no stake and no privileged access can execute this attack indefinitely at the cost of transaction fees.

### Likelihood Explanation
**High.** The entry point is a public, permissionless function. No tokens, no stake, no privileged key, and no special setup are required. The attacker only needs to submit a transaction once per block. On Starknet, transaction fees are low, making sustained griefing economically viable.

### Recommendation
Add an explicit sequencer-only guard at the top of `update_rewards`, analogous to the checks used in other restricted functions (e.g., `update_current_epoch_block_rewards` in `reward_supplier.cairo` which asserts `get_caller_address() == staking_contract`):

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    // Add sequencer-only check:
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "ONLY_SEQUENCER"
    );
    self.general_prerequisites();
    // ...
}
```

If the sequencer address is not stored on-chain, an alternative is to use Starknet's `get_execution_info` to verify the transaction originates from the expected sequencer, or to introduce a dedicated role (e.g., `REWARDS_UPDATER`) that is granted only to the sequencer.
