### Title
Off-by-one in DKG threshold upper-bound check silently accepts `threshold == N`, enabling permanent denial of signing — (`src/dkg.rs`)

### Summary

`assert_key_invariants` uses a strict `>` comparison instead of `>=` when validating the threshold upper bound. The protocol specification explicitly requires `1 < threshold < N` (strictly less than N), but the code permits `threshold == N`. A library caller who passes `threshold = participants.len()` will silently produce a DKG output whose reconstruction requires every single participant, making signing permanently unavailable if any one party is absent.

### Finding Description

The protocol specification in `docs/dkg.md` states:

> 1.1 Each P_i asserts that `1 < threshold < N`.

The upper bound is strict: `threshold` must be **strictly less than** N.

The implementation in `assert_key_invariants` checks:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold,
        max: participants.len(),
    });
}
``` [1](#0-0) 

Because the guard uses `>` rather than `>=`, the value `threshold == participants.len()` (i.e., `threshold == N`) passes validation without error. The same permissive check is replicated in `validate_triple_inputs` for OT-based triple generation:

```rust
if threshold_value > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

When `threshold == N` is accepted, `do_keyshare` generates a polynomial of degree `threshold - 1 = N - 1`:

```rust
let degree = threshold.value().checked_sub(1)...;
let secret_coefficients = Polynomial::<C>::generate_polynomial(Some(secret), degree, rng)?;
``` [3](#0-2) 

A degree-`N-1` polynomial requires exactly N evaluation points to reconstruct, meaning every participant must be present for any future signing or resharing operation.

### Impact Explanation

This matches the **High** allowed impact: *Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions.*

When `threshold == N` is used:
- The DKG completes successfully and produces apparently valid key shares.
- Every subsequent signing, resharing, or refresh operation requires all N participants simultaneously.
- The absence of even one honest participant — due to network partition, node failure, or a single malicious party going offline — permanently blocks all signing. The key material cannot be recovered or reshared without all N parties.
- There is no in-protocol mechanism to detect or recover from this state after the fact.

### Likelihood Explanation

The entry path is a direct, unprivileged library API call. Any caller of `keygen` or `reshare` can supply `threshold = participants.len()`. The validation function `assert_key_invariants` is the sole guard and it silently passes the invalid value. No special privilege, leaked key, or cryptographic break is required. The misconfiguration is invisible at call time because the DKG protocol runs to completion without error. [4](#0-3) 

### Recommendation

Replace `>` with `>=` in both threshold upper-bound guards so that `threshold == N` is rejected:

In `assert_key_invariants` (`src/dkg.rs`):
```rust
// Before (off-by-one):
if threshold > participants.len() { ... }

// After (correct):
if threshold >= participants.len() { ... }
```

In `validate_triple_inputs` (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`):
```rust
// Before:
if threshold_value > participants.len() { ... }

// After:
if threshold_value >= participants.len() { ... }
```

The error variant `ThresholdTooLarge` already exists and carries the right semantics; only the comparison operator needs to change. [5](#0-4) 

### Proof of Concept

```rust
// N = 3 participants, threshold = 3 (== N, should be rejected per spec)
let participants = vec![p1, p2, p3];
let threshold = 3usize; // equals participants.len()

// assert_key_invariants passes silently: 3 > 3 is false, no error
let result = keygen::<MyCiphersuite>(&participants, p1, threshold, rng);
assert!(result.is_ok()); // DKG succeeds

// Now signing requires ALL 3 parties.
// If p3 goes offline, signing is permanently blocked.
// The polynomial has degree 2 (N-1), requiring 3 evaluation points to reconstruct.
```

The analogous off-by-one in the external report used `<=` where `<` was required, extending one phase by one unit. Here, `>` is used where `>=` is required, extending the accepted threshold range by one unit — admitting `threshold == N` and silently producing an unusable key configuration.

pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
