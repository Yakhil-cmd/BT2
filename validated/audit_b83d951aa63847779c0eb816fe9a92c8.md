### Title
Missing Zero `msg_hash` Validation in OT-Based ECDSA Signing Enables Private Key Recovery - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

### Summary
The OT-based ECDSA `sign` function accepts a zero `msg_hash` without rejection. When combined with presignature reuse (which the protocol does not prevent at the API level), a malicious coordinator can recover the aggregate private key by running two signing sessions — one with `msg_hash = 0` and one with `msg_hash ≠ 0` — over the same presignature.

### Finding Description
The Robust ECDSA `sign` function explicitly rejects a zero message hash:

```rust
// src/ecdsa/robust_ecdsa/sign.rs lines 91-95
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
``` [1](#0-0) 

The OT-based ECDSA `sign` function performs no equivalent check:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs lines 22-76
pub fn sign(
    ...
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    // No msg_hash == 0 check
    ...
}
``` [2](#0-1) 

The signature share computation in the OT-based scheme is:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs lines 155-158
let r = x_coordinate(&presignature.big_r);
Ok(msg_hash * k_i + r * sigma_i)
``` [3](#0-2) 

When `msg_hash = 0`, the `k_i` term vanishes entirely: `s_i = r * sigma_i`. The coordinator aggregates to `s = r * sigma`, where `sigma = k_nonce^{-1} * x` (the product of the inverse nonce and the private key), as established by the presign protocol. [4](#0-3) 

### Impact Explanation
A malicious coordinator who can reuse a presignature (no API-level enforcement prevents this; the library only documents it as a caller responsibility) can execute the following key-recovery attack:

**Session 1** — `msg_hash = 0` (accepted without error):
- Each participant computes `s_i = r * sigma_i`
- Coordinator aggregates: `s1 = r * sigma`

**Session 2** — same presignature, `msg_hash = h ≠ 0`:
- Each participant computes `s_j = h * k_j + r * sigma_j`
- Coordinator aggregates: `s2 = h * k_nonce^{-1} + r * sigma`

**Key recovery**:
- `s2 - s1 = h * k_nonce^{-1}` → `k_nonce^{-1} = (s2 - s1) / h`
- `sigma = s1 / r`
- `x = sigma / k_nonce^{-1} = (s1 / r) / ((s2 - s1) / h) = s1 * h / (r * (s2 - s1))`

This directly recovers the aggregate private key `x`. Impact: **Critical** — extraction of aggregate secret key material.

The security documentation for the robust ECDSA scheme explicitly documents this class of attack and mandates the `h = 0` rejection: [5](#0-4) 

### Likelihood Explanation
The entry path is straightforward: any caller of the public `sign` API can pass `msg_hash = Scalar::ZERO`. A malicious coordinator controlling the signing session can do so deliberately. The presignature reuse required for the second step is not prevented by any runtime check in the library — it is only a documented caller obligation. The attack requires no cryptographic breaks, no leaked keys, and no external assumptions.

### Recommendation
Add the same zero-hash guard present in the robust ECDSA `sign` function to the OT-based ECDSA `sign` function, immediately after the participant-count checks:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs — add after line 63
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0".to_string(),
    ));
}
``` [6](#0-5) 

### Proof of Concept
1. Generate a DKG key share set with `n = 2`, `threshold = 2`.
2. Run the presign protocol to obtain a `PresignOutput` with known `big_r`, `k`, `sigma` shares.
3. Call `ot_based_ecdsa::sign(...)` with `msg_hash = Scalar::ZERO` — the call succeeds and returns `s1 = r * sigma`.
4. Call `ot_based_ecdsa::sign(...)` again with the **same** `RerandomizedPresignOutput` and `msg_hash = h` (any nonzero scalar) — the call succeeds and returns `s2 = h * k_nonce^{-1} + r * sigma`.
5. Compute `x = s1 * h / (r * (s2 - s1))` and verify it equals the aggregate private key reconstructed via Lagrange interpolation of the key shares. [2](#0-1) [1](#0-0)

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
