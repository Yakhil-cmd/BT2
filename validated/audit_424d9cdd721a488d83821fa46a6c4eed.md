### Title
`PresignOutput` Reuse Enabled by Reference-Taking `rerandomize_presign` Allows Malicious Coordinator to Extract Private Key — (File: `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both the Robust ECDSA and OT-based ECDSA schemes expose `PresignOutput` as a `Clone`-able, serializable struct, and their `rerandomize_presign` functions accept the presignature by shared reference (`&PresignOutput`), never consuming it. This means a single `PresignOutput` can be rerandomized and fed into `sign` an arbitrary number of times. A malicious coordinator can exploit this by aborting signing sessions after collecting participants' signature shares, then initiating new sessions with different messages against the same presignature material. Two signing sessions sharing the same underlying nonce yield multiplicatively related nonces, enabling full private key extraction via standard ECDSA nonce-reuse techniques.

---

### Finding Description

**Root cause — `PresignOutput` is `Clone` and `rerandomize_presign` takes it by reference:**

In `src/ecdsa/robust_ecdsa/mod.rs`, `PresignOutput` derives `Clone` and `Serialize/Deserialize`: [1](#0-0) 

The `rerandomize_presign` function signature takes the presignature by shared reference, not by value: [2](#0-1) 

Because `presignature` is `&PresignOutput`, the caller retains full ownership of the `PresignOutput` after the call. The caller can invoke `rerandomize_presign` again with different `RerandomizationArguments` (different `(h, ε, ρ)`) to produce a second `RerandomizedPresignOutput` from the same underlying nonce material. Each resulting `RerandomizedPresignOutput` is then passed by value to `sign` (consuming it), but the original `PresignOutput` remains intact and reusable. [3](#0-2) 

The identical pattern exists in the OT-based ECDSA scheme: [4](#0-3) [5](#0-4) 

The documentation explicitly warns against this but provides no type-level or runtime enforcement: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The library's own security documentation states the consequence directly: [8](#0-7) 

Two signing sessions that share the same underlying `PresignOutput` nonce material but use different `(h, ε)` values produce signature shares with multiplicatively related nonces. A coordinator holding both sets of shares can apply standard ECDSA nonce-reuse algebra to reconstruct the aggregate private key. This maps to the **Critical** allowed impact: *Extraction, reconstruction, or disclosure of private signing shares or aggregate secret material*.

---

### Likelihood Explanation

A malicious coordinator controls the signing session lifecycle and can:

1. Initiate signing session 1 with message `m1`.
2. Wait for all participants to send their signature shares via `sign` (shares are sent to the coordinator only, per the protocol spec).
3. Abort the session — withhold the final aggregated signature from participants.
4. Initiate signing session 2 with a different message `m2`.
5. Participants, observing that session 1 produced no output, have no type-level signal that their `PresignOutput` was already consumed — because `rerandomize_presign` did not consume it. They call `rerandomize_presign` again on the same `PresignOutput` and participate in session 2.
6. The coordinator now holds two sets of signature shares for the same underlying nonce → full key extraction.

The library provides no mechanism for participants to detect that their presignature was already used in an aborted session. The `PresignOutput` is not consumed by `rerandomize_presign`, so participants have no type-level signal that their nonce material has been exposed. [9](#0-8) 

---

### Recommendation

1. **Make `rerandomize_presign` consume the `PresignOutput` by taking it by value** in both `src/ecdsa/robust_ecdsa/mod.rs` and `src/ecdsa/ot_based_ecdsa/mod.rs`:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // consumed — not reusable
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

2. **Remove the `Clone` derive from `PresignOutput`** in both schemes to prevent callers from circumventing single-use enforcement by cloning before passing. This enforces the documented invariant at the type level, analogous to the boolean-mapping fix recommended in the Connext report.

---

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
