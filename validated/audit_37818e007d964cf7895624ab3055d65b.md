### Title
WalletConnect `chia_getWalletAddresses` Fingerprint Access Control Bypass Allows Dapp to Enumerate Addresses of Unpaired Wallet Keys - (File: packages/gui/src/electron/commands/Commands.ts)

---

### Summary

The WalletConnect fingerprint access control in `dispatchPairRequest` only validates the singular `params.fingerprint` field. The `chia_getWalletAddresses` dapp command accepts a **plural** `fingerprints` array parameter that is never validated against the paired fingerprint. A dapp granted `chia_getWalletAddresses` can silently enumerate wallet addresses for every other key stored in the user's keyring, bypassing the per-fingerprint pairing restriction entirely.

---

### Finding Description

`dispatchPairRequest` enforces that a WalletConnect command runs only for the fingerprint the dapp was paired to: [1](#0-0) 

The guard destructures only `params.fingerprint` (singular) and compares it to `pair.fingerprint` and `loggedInFingerprint`. If `params.fingerprint` is absent, it falls back to `loggedInFingerprint`, which equals `pair.fingerprint`, so the check passes.

The `daemon.get_wallet_addresses` command, exposed as the dapp command `chia_getWalletAddresses`, accepts a **different** parameter named `fingerprints` (plural JSON array): [2](#0-1) 

The underlying daemon API accepts an arbitrary array of fingerprints: [3](#0-2) 

Because `dispatchPairRequest` never inspects `params.fingerprints` (plural), a dapp paired to fingerprint A can pass `{ fingerprints: [B, C, D] }`. The guard sees `params.fingerprint === undefined`, falls back to `loggedInFingerprint` (= A = `pair.fingerprint`), passes all checks, and the daemon returns derived wallet addresses for keys B, C, and D — keys the dapp was never authorized to access.

The command is also marked `allowConfirmationBypass: true`, so once the dapp has been granted `chia_getWalletAddresses` at pairing time, no per-request confirmation dialog is shown to the user: [4](#0-3) 

---

### Impact Explanation

A dapp paired to any single fingerprint can silently retrieve derived wallet addresses (across all HD derivation indices) for every other wallet key stored in the local keyring. This is a bypass of the WalletConnect fingerprint isolation guarantee. The user approved the dapp for one key; the dapp operates across all keys without further consent or notification. Wallet addresses, when linked to specific key identities, expose the full address space of other wallets on the machine, enabling cross-key surveillance and identity linkage.

---

### Likelihood Explanation

Any dapp that requests and is granted `chia_getWalletAddresses` during pairing can exploit this immediately. The permission label shown to the user ("Get wallet addresses for one or more wallet keys") does not indicate that other keys' addresses will be accessible. The bypass requires no additional privileges, no chaining, and no user interaction after initial pairing.

---

### Recommendation

In `dispatchPairRequest` (or in a command-specific handler for `chia_getWalletAddresses`), validate that every fingerprint in `params.fingerprints` (when present) equals `pair.fingerprint`. Alternatively, strip or override the `fingerprints` parameter before forwarding to the daemon, forcing it to only return addresses for the paired fingerprint. The same audit should be applied to any other command that accepts a fingerprint-like identifier in a field other than the singular `fingerprint` key.

---

### Proof of Concept

1. A malicious dapp initiates a WalletConnect session and requests the `chia_getWalletAddresses` command during pairing.
2. The user approves pairing for their active wallet key (fingerprint A). The user optionally enables "Allow without asking" for this command.
3. The dapp sends a WalletConnect session request:
   ```json
   {
     "method": "chia_getWalletAddresses",
     "params": {
       "fingerprints": [111111, 222222, 333333],
       "count": 10,
       "index": 0
     }
   }
   ```
4. `dispatchPairRequest` extracts `params.fingerprint` → `undefined`. Falls back to `loggedInFingerprint` (= A). Checks `A === pair.fingerprint` → passes. The second guard (`fingerprint !== undefined`) is skipped because `fingerprint` is `undefined`.
5. The daemon executes `get_wallet_addresses` with `fingerprints: [111111, 222222, 333333]` and returns all derived addresses for those keys.
6. The dapp receives wallet addresses for keys it was never authorized to access, with no confirmation dialog shown to the user. [5](#0-4) [2](#0-1)
