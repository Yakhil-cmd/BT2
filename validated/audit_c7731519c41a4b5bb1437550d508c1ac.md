Looking at the external report's vulnerability class — **lack of segregation of duties / overly permissive access control on state-mutating functions** — I need to find an analog in the Starknet Staking codebase where a function that should be restricted is callable by an overly broad set of callers, causing accounting damage or yield loss.

After examining `src/staking/staking.cairo`, `src/staking/interface.cairo`, and the test suite, I found a valid analog.

---

### Title
Unrestricted `update_rewards` with `disable_rewards: true` Allows Any Caller to Permanently Freeze All Stakers' Consensus Yield — (`src/staking/staking.cairo`)

### Summary
`IStakingRewardsManager::update_rewards` has no caller restriction. Any address can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` counter without distributing any rewards. Because `last_reward_block` is a single global field, one such call per block permanently blocks reward distribution for **all** stakers for that block.

### Finding Description
`update_rewards` is implemented at line 1449 of `src/staking/staking.cairo`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
```

There is no caller check beyond `general_prerequisites()` (which only enforces the pause flag). The function accepts an arbitrary `staker_address` and a `disable_rewards: bool` flag.

The storage field gating the call is:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
```

This is a **single global** `BlockNumber`, not a per-staker map. When `update_rewards` is called with `disable_rewards: true`, `last_reward_block` is advanced to the current block but no rewards are credited. Any subsequent call in the same block — including a legitimate call with `disable_rewards: false` — reverts with `REWARDS_ALREADY_UPDATED`.

This is confirmed by the test suite:

```cairo
staking_rewards_dispatcher.update_rewards(:staker_address, disable_rewards: true);
// Both subsequent calls fail:
let result = staking_rewards_safe_dispatcher
    .update_rewards(:staker_address, disable_rewards: true);
assert_panic_with_error(:result, expected_error: Error::REWARDS_ALREADY_UPDATED.describe());
let result = staking_rewards_safe_dispatcher
    .update_rewards(:staker_address, disable_rewards: false);
assert_panic_with_error(:result, expected_error: Error::REWARDS_ALREADY_UPDATED.describe());
```

The test calls `update_rewards` with no `cheat_caller_address_once`, confirming the function is callable by any address.

### Impact Explanation
An unprivileged attacker can call `update_rewards(any_active_staker, disable_rewards: true)` once per block. Because `last_reward_block` is global, this single call prevents **every** staker from receiving consensus block rewards for that block. Repeating this every block permanently freezes all consensus yield across the entire protocol. This matches the **High: Permanent freezing of unclaimed yield** impact category.

### Likelihood Explanation
The attack requires no special role, no token balance, and no prior state. The only cost is gas per block. Any address on Starknet can execute this continuously. Likelihood is high.

### Recommendation
Restrict `update_rewards` to be callable only by the staker's registered operational address, or separate the `disable_rewards: true` path into a dedicated function gated by `only_app_governor` or `only_operator`. The `disable_rewards: false` path (normal reward distribution) should similarly be restricted to the operational address to prevent griefing of individual stakers.

### Proof of Concept
1. Consensus rewards are active (after `set_consensus_rewards_first_epoch` takes effect).
2. In block N, attacker calls `update_rewards(any_active_staker_address, disable_rewards: true)`.
3. `last_reward_block` is set to N; no rewards are distributed to any staker.
4. Any legitimate call `update_rewards(staker, disable_rewards: false)` in block N reverts with `REWARDS_ALREADY_UPDATED`.
5. Stakers earn zero consensus rewards for block N.
6. Attacker repeats every block — all consensus rewards are permanently frozen.
