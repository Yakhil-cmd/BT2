### Title
Coordinator Can Present Different `(h, ε)` to Different Signers After Presignature Commitment, Enabling Secret Key Extraction — (`src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

In the robust ECDSA signing phase, the coordinator distributes the message hash `h` and rerandomization tweak `ε` to each participant individually, with no consensus mechanism enforcing that all participants receive identical values. A malicious coordinator can present different `(h, ε)` pairs to different subsets of signers who have already committed their presignature material. Because the presignature nonce is shared across these sessions, this constitutes nonce reuse, and the secret key can be recovered via standard ECDSA nonce-reuse algebra.

---

### Finding Description

The robust ECDSA signing phase is coordinator-driven: the coordinator sends `(h, ε)` to each participant, who then computes a linearized signature share `s_i` and returns it. Unlike the DKG rounds — which use echo broadcast (Byzantine Reliable Broadcast) to guarantee all parties deliver the same value — the signing phase has no such consensus step for `(h, ε)`.

The library's own security documentation explicitly acknowledges this:

> *"Before implementing or using the robust ECDSA scheme implemented here, be aware that it is vulnerable to **split-view attacks** in the robust setting when the signing parameters are not globally consistent. If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."* [1](#0-0) 

The documentation further warns:

> *"Ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers."* [2](#0-1) 

The phrasing "must not be able to" is a recommendation to the library caller, not an enforcement inside the library. The signing implementation in `src/ecdsa/robust_ecdsa/sign.rs` does not use echo broadcast or any cross-participant consistency check for `(h, ε)`. The coordinator is the sole sender of these values to each participant, and no participant can verify that others received the same inputs.

This is the direct analog to the external report's `setWhitelistReserve()` pattern: a privileged party (coordinator / owner) can silently present different protocol parameters to different participants after those participants have already committed irreversible cryptographic material (presignature shares), breaking the invariant that all parties operate under the same rules.

---

### Impact Explanation

**Impact: Critical — Extraction/reconstruction of private signing shares.**

If a malicious coordinator runs two overlapping signing sessions over the same presignature with different `(h₁, ε₁)` and `(h₂, ε₂)` presented to different signer subsets (each of size ≥ 2t+1), the resulting partial signatures expose the relationship between the nonce and the secret key. Standard ECDSA nonce-reuse recovery then yields the full aggregate secret key.

The documentation explicitly confirms this:

> *"a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."* [3](#0-2) 

---

### Likelihood Explanation

The coordinator is a single point of control in the signing phase. Any party operating as coordinator — whether compromised, malicious, or coerced — can trivially execute this attack by sending different `(h, ε)` bytes to different participants in two concurrent or sequential signing sessions over the same presignature. No cryptographic capability beyond coordinator role is required. The library provides no internal guard against this. [4](#0-3) 

---

### Recommendation

Enforce global consistency of `(h, ε)` inside the signing protocol itself, not as a caller-side obligation:

1. **Use echo broadcast for `(h, ε)`**: Before participants compute their signature shares, run a reliable broadcast round (as already used in DKG) where the coordinator broadcasts `(h, ε)` and all participants verify they received the same value. The echo broadcast infrastructure already exists in `src/protocol/echo_broadcast.rs`. [5](#0-4) 

2. **Bind `(h, ε)` to the presignature identifier**: Include a commitment to `(h, ε)` in the presignature output so that participants can detect reuse of a presignature with different parameters.

3. **Enforce N₁ = N₂ = 2t+1 inside the library**: The library currently documents this as a recommendation; it should be a hard invariant enforced at signing initialization.

---

### Proof of Concept

```
Setup:
  - N=5 participants, t=2 (MaxMalicious), threshold=3
  - DKG completes; all parties hold shares of secret key x
  - One presignature is generated by all 5 parties → nonce k is committed

Attack (malicious coordinator):
  Session A: Send (h₁, ε₁) to participants {P1, P2, P3}
    → Each computes s_i^A = linearized share under (h₁, ε₁, k)
    → Coordinator collects s^A = Σ s_i^A

  Session B (same presignature): Send (h₂, ε₂) to participants {P1, P2, P4}
    → Each computes s_i^B = linearized share under (h₂, ε₂, k)
    → Coordinator collects s^B = Σ s_i^B

  Both sessions use the same nonce k (same presignature).
  s^A and s^B are two ECDSA signatures with the same nonce k but different messages.
  Standard nonce-reuse recovery:
    x = (s^A · h₂ - s^B · h₁) / (s^B - s^A)  (mod q)
  → Full secret key x recovered.
```

No special capability beyond coordinator role is required. The library does not prevent the coordinator from sending different `(h, ε)` to different participants, and no participant can detect the inconsistency. [6](#0-5)

```markdown
# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

4. **Do not sign with $h = 0$** (the zero message hash).
   This input enables a related algebraic split-view attack in the modified scheme when
   $N_1 > 2t + 1$.
```
