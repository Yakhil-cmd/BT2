### Title
Zero `priceProviderTimelock` Bypasses Oracle Rotation Delay, Enabling Instant Malicious Price Provider Swap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`createPool` accepts `priceProviderTimelock = 0` without enforcing a minimum. Because `proposePoolPriceProvider` computes `executeAfter = block.timestamp + 0`, the guard in `executePoolPriceProviderUpdate` (`block.timestamp < execAfter`) is immediately false, so propose and execute can be called in the same block. A pool admin who is also a market maker can attract LP deposits, then atomically rotate to a malicious price provider and drain LP principal through manipulated-price swaps.

---

### Finding Description

`createPool` stores the caller-supplied `priceProviderTimelock` directly with no lower-bound check: [1](#0-0) 

`proposePoolPriceProvider` computes the unlock timestamp as: [2](#0-1) 

When `timelock == 0`, `executeAfter == block.timestamp`. The enforcement gate in `executePoolPriceProviderUpdate` is: [3](#0-2) 

`block.timestamp < block.timestamp` is always `false`, so the revert is never triggered. Both calls can be issued in the same transaction, making the timelock completely inoperative.

The only alternative that prevents rotation is `priceProviderTimelock == type(uint256).max` (immutable mode): [4](#0-3) 

Any value strictly less than `type(uint256).max` — including `0` — is treated as mutable mode with that many seconds of delay. There is no code path that rejects `0` or enforces a sensible minimum.

---

### Impact Explanation

The price provider is the sole source of bid/ask prices for every swap. A malicious provider can return `bidPrice = 1` and `askPrice = type(uint128).max`, allowing the pool admin to:

1. Drain all token1 from the pool by buying token0 at a near-zero ask price.
2. Drain all token0 from the pool by selling token0 at a near-infinite bid price.

LP principal — not just fees — is at risk. The pool solvency invariant (pool balances must cover LP claims) is broken.

---

### Likelihood Explanation

The pool admin is semi-trusted only within caps and timelocks. The timelock is the explicit protection mechanism for oracle rotation (see audit finding M01 in `AuditFindings.md`): [5](#0-4) 

Setting `priceProviderTimelock = 0` is a valid on-chain configuration that any pool creator can choose. A pool admin who is also a liquidity provider or market maker has direct financial incentive to exploit this. The attack requires no external conditions — only the pool admin's own transactions.

---

### Recommendation

Enforce a minimum non-zero timelock for mutable-mode pools. In `_validatePoolParameters` (or at the top of `createPool`), add:

```solidity
uint256 MIN_PRICE_PROVIDER_TIMELOCK = 1 days; // or protocol-chosen constant
if (params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

This mirrors the Teller recommendation: enforce a sensible minimum for the delay parameter so that the protection mechanism cannot be nullified by setting it to zero.

---

### Proof of Concept

```solidity
function test_ZeroTimelockBypassesOracleRotation() public {
    // Pool created with priceProviderTimelock = 0 (mutable, zero delay)
    PoolParameters memory params = _defaultParams();
    params.priceProviderTimelock = 0;  // NOT type(uint256).max → mutable mode
    params.salt = keccak256("zero_timelock_pool");
    address pool = factory.createPool(params);

    // LPs deposit funds into the pool
    // ... (addLiquidity calls) ...

    // Admin deploys a malicious oracle returning extreme prices
    MockOracle maliciousOracle = new MockOracle();
    maliciousOracle.setTokens(params.token0, params.token1);
    maliciousOracle.setBidAndAskPrice(1, type(uint128).max); // extreme spread

    // Propose and execute in the same block — no warp needed
    vm.startPrank(admin);
    factory.proposePoolPriceProvider(pool, address(maliciousOracle));
    // executeAfter == block.timestamp, so block.timestamp < executeAfter is false
    factory.executePoolPriceProviderUpdate(pool); // succeeds immediately
    vm.stopPrank();

    // Admin now swaps at manipulated prices, draining LP principal
    // ...
}
```

The key mechanic: `proposePoolPriceProvider` sets `pendingPriceProviderExecuteAfter[pool] = block.timestamp + 0 = block.timestamp`, and `executePoolPriceProviderUpdate` checks `block.timestamp < block.timestamp` which is `false`, so it never reverts. [6](#0-5)

```text
  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
```

```markdown
## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
```
