Audit Report

## Title
Cycles Minting Rate Limit Bypass via Pre-Accumulated Cycles from Failed Deposits - (`File: rs/nns/cmc/src/main.rs`)

## Summary

The `ensure_balance` function mints cycles and increments the rate-limit counter **before** the deposit to the target canister is attempted. When a deposit fails (e.g., target canister does not exist), the minted cycles remain in the CMC's balance but the ICP is refunded. After the sliding rate-limit window expires, a subsequent deposit for the same amount computes `cycles_to_mint = 0` via saturating subtraction, adds nothing to the limiter, and delivers the pre-accumulated cycles to the target — effectively bypassing the rate limit for that request.

## Finding Description

`ensure_balance` at `rs/nns/cmc/src/main.rs:2306–2325` computes:

```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;  // saturating sub → 0 when balance ≥ cycles
```

`Cycles::sub` is confirmed saturating at `rs/types/cycles/src/cycles.rs:133–139`. When `current_balance >= cycles`, `cycles_to_mint` is `Cycles(0)`, so `check_and_add_cycles(state, now, Cycles(0))` adds nothing to the limiter.

The call order in `deposit_cycles` (`rs/nns/cmc/src/main.rs:2116–2135`) is:
1. `ensure_balance(cycles, limiter_to_use)` — mints cycles, increments limiter
2. `call_with_payment128(management_canister, METHOD_DEPOSIT_CYCLES, ...)` — attempts deposit

If step 2 fails, `process_top_up` (`rs/nns/cmc/src/main.rs:2004–2010`) calls `refund_icp` to return ICP to the sender, but there is no mechanism to burn or return the cycles minted in step 1. They remain in the CMC's balance.

The `Limiter` uses a sliding time window (`max_age`). After the window expires, `purge_old` (`rs/nns/cmc/src/limiter.rs:83–92`) removes the old minting events from `total_count`, resetting the limiter — but the stranded cycles in the CMC balance are not removed.

Exploit flow:
- **T=0**: Attacker sends X ICP to CMC subaccount for non-existing canister. `ensure_balance` mints X cycles (limiter += X). Deposit fails. ICP refunded (minus `TOP_UP_CANISTER_REFUND_FEE`). CMC balance = X cycles.
- **T=1 hour**: Limiter window expires; `total_count` reset to 0.
- **T=1 hour**: Attacker sends X ICP for valid canister Y. `ensure_balance`: `current_balance = X`, `cycles_to_mint = 0`, limiter unchanged. CMC deposits X cycles to Y from existing balance. ICP burned.
- **T=1 hour**: Attacker also sends up to 150P ICP for canister Z. Full rate limit consumed normally.
- Net: X + 150P cycles delivered in one window; rate limit is 150P/hour.

## Impact Explanation

This is a concrete rate-limit bypass on the NNS Cycles Minting Canister, a core ICP financial integration component. The bypass allows any unprivileged user to exceed the `base_cycles_limit` (150 Peta-cycles/hour) by up to X cycles per window, where X is bounded only by the attacker's ICP budget and the small `TOP_UP_CANISTER_REFUND_FEE`. The attacker still pays ICP for the cycles ultimately delivered, but circumvents the protocol's intended cap on ICP-to-cycles conversion rate. This constitutes unauthorized excessive cycles minting and abnormal ICP burning beyond protocol-defined limits, matching the **High** impact class: significant NNS/financial integration security impact with concrete protocol harm.

## Likelihood Explanation

- Requires no privileged access; any principal can call `notify_top_up`.
- Cost per bypass batch is only `TOP_UP_CANISTER_REFUND_FEE` (small relative to cycles gained).
- Repeatable every rate-limit window (hourly for the base limiter).
- No victim interaction, social engineering, or external dependency required.
- The existing test `cmc_notify_top_up_not_rate_limited_by_invalid_top_up` confirms the mechanism (failed deposits leave cycles in CMC balance) but does not test the cross-window bypass scenario.

## Recommendation

Move the limiter increment and minting to **after** `call_with_payment128` succeeds, so only cycles actually delivered to a target canister are counted against the rate limit. Alternatively, on deposit failure, explicitly burn the stranded cycles (e.g., by calling `ic0_mint_cycles128` with a negative adjustment if the API supports it, or by transferring them to a burn address) so they do not persist in the CMC balance across rate-limit windows. A third option is to track the CMC's "reserved" balance separately and subtract it from `current_balance` in `ensure_balance` so pre-accumulated stranded cycles do not suppress `cycles_to_mint`.

## Proof of Concept

```
// Setup: base_cycles_limit = 150P cycles/hour, rate = 100 XDR/ICP, 1T cycles/XDR

// Step 1 (T=0): Attacker sends 1500 ICP to CMC subaccount for non-existing canister X
//   notify_top_up({block_index: B1, canister_id: X})
//   → ensure_balance(150P): current_balance=0, cycles_to_mint=150P, limiter=150P, mints 150P
//   → call_with_payment128 to X fails (X does not exist)
//   → refund_icp: attacker receives ~1500 ICP back (minus TOP_UP_CANISTER_REFUND_FEE)
//   → CMC balance = 150P cycles (stranded)

// Step 2 (T=1 hour): Limiter purge_old() removes the T=0 window; total_count = 0

// Step 3 (T=1 hour): Attacker sends 1500 ICP to CMC subaccount for valid canister Y
//   notify_top_up({block_index: B2, canister_id: Y})
//   → ensure_balance(150P): current_balance=150P, cycles_to_mint=0, limiter unchanged (0)
//   → call_with_payment128 deposits 150P to Y from existing balance — succeeds
//   → burn_and_log: 1500 ICP burned
//   → 150P cycles delivered to Y, 0 rate limit consumed

// Step 4 (T=1 hour): Attacker sends 1500 ICP for valid canister Z
//   notify_top_up({block_index: B3, canister_id: Z})
//   → ensure_balance(150P): current_balance=0, cycles_to_mint=150P, limiter=150P (at limit)
//   → 150P cycles delivered to Z normally

// Result: 300P cycles delivered in one hour window; rate limit is 150P/hour
// Bypass cost: one TOP_UP_CANISTER_REFUND_FEE from Step 1
```

    })?;

    Ok(())

fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();

    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    // unused because of check above
    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
