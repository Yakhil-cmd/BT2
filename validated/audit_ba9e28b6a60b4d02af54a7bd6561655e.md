### Title
`skip_stateful_validations` Hardcodes `Nonce(Felt::ONE)` and Ignores the `max_nonce_for_validation_skip` Config Field, Allowing Signature-Bypass Admission When the Feature Is Disabled - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidatorConfig` defines, documents, and serializes a `max_nonce_for_validation_skip` field that is intended to control the upper nonce bound for which the `__validate__` entry-point is skipped during gateway admission. The gateway function `skip_stateful_validations` ignores this field entirely and hardcodes the comparison to `Nonce(Felt::ONE)`. When an operator sets `max_nonce_for_validation_skip = 0` to disable the UX skip feature, the gateway still bypasses `__validate__` for every nonce-1 invoke transaction whose sender has any transaction in the mempool or a recent block, admitting transactions with invalid signatures.

---

### Finding Description

`StatefulTransactionValidatorConfig` declares and serializes `max_nonce_for_validation_skip`: [1](#0-0) 

The field is documented as *"Maximum nonce for which the validation is skipped"* and defaults to `Nonce(Felt::ONE)`: [2](#0-1) 

The gateway's `run_pre_validation_checks` calls `skip_stateful_validations` without passing the config: [3](#0-2) 

`skip_stateful_validations` hardcodes the nonce comparison to `Nonce(Felt::ONE)` and never consults `max_nonce_for_validation_skip`: [4](#0-3) 

The returned `skip_validate` flag is then used to suppress the `__validate__` entry-point call entirely: [5](#0-4) 

The old `PyValidator` code correctly used the configurable field: [6](#0-5) 

The new gateway code dropped this parameterization, creating a permanent hardcoded bypass.

The `account_tx_in_pool_or_recent_block` guard checks for **any** transaction from the sender, not specifically a `deploy_account`: [7](#0-6) 

This means any account that has ever had a transaction in the mempool or a recent committed block satisfies the guard, making the bypass reachable for a broad set of senders.

---

### Impact Explanation

When an operator sets `max_nonce_for_validation_skip = 0` (intending to disable the feature), the gateway ignores the setting and still skips `__validate__` for every nonce-1 invoke transaction whose sender appears in the mempool or a recent block. A transaction with an invalid or forged signature passes gateway admission and enters the mempool. The batcher will eventually attempt execution and the transaction will revert, but the mempool has accepted an unauthorized transaction, violating the admission invariant.

Impact category: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

Any operator who explicitly sets `max_nonce_for_validation_skip = 0` to harden their node is silently unprotected. The config field is public, documented, and serialized, so operators have a reasonable expectation that it is honored. An attacker who knows the field is ignored can submit nonce-1 invoke transactions with arbitrary signatures for any account that has ever appeared in the mempool.

---

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` and replace the hardcoded `Nonce(Felt::ONE)` equality check with a `<=` comparison against the configured value:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,   // <-- add parameter
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() <= max_nonce_for_validation_skip   // <-- use config
            && tx.nonce() > Nonce(Felt::ZERO)
            && account_nonce == Nonce(Felt::ZERO)
        {
            ...
        }
    }
    Ok(false)
}
```

Update `run_pre_validation_checks` to forward `self.config.max_nonce_for_validation_skip`.

---

### Proof of Concept

1. Operator deploys a sequencer node with `max_nonce_for_validation_skip = 0` in `StatefulTransactionValidatorConfig` (intending to disable the skip).
2. Attacker submits a valid `deploy_account` for address `A` → admitted to mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits an `Invoke V3` for address `A` with `nonce = 1` and a **garbage signature**.
4. Gateway calls `skip_stateful_validations`: `tx.nonce() == Nonce(Felt::ONE)` is `true`, `account_nonce == Nonce(Felt::ZERO)` is `true`, `account_tx_in_pool_or_recent_block(A)` is `true` → returns `true`.
5. `run_validate_entry_point` sets `execution_flags.validate = false` → `__validate__` (signature check) is **not called**.
6. The forged-signature invoke transaction is admitted to the mempool, bypassing account authorization at the gateway layer.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
