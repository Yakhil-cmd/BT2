Audit Report

## Title
Unbounded Ballot Iteration in SNS Governance `distribute_rewards` Exhausts Instruction Limit, Permanently Halting Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary

The `distribute_rewards` function in SNS governance iterates over all ballots of all `ReadyToSettle` proposals and all voting neurons in a single synchronous execution with no instruction-limit guard. At ceiling values of 700 proposals × 200,000 neurons, the ballot aggregation loop alone reaches 140 million iterations. When the IC instruction limit is exceeded inside the timer callback, state is rolled back and `latest_reward_event` is never updated, causing `should_distribute_rewards` to return `true` on every subsequent invocation and reproducing the same trap permanently.

## Finding Description

**Root cause — no instruction-limit guard in `distribute_rewards`:**

`distribute_rewards` is a synchronous function called directly from `run_periodic_tasks` with no batching or instruction-limit check anywhere in its body.

**Loop 1 — ballot aggregation (lines 5894–5931):**
```rust
for proposal_id in &considered_proposals {
    if let Some(proposal) = self.get_proposal_data(*proposal_id) {
        for (voter, ballot) in &proposal.ballots {
            // NeuronId::from_str + HashMap entry + Decimal arithmetic
        }
    }
}
```
`considered_proposals` is populated by `ready_to_be_settled_proposal_ids()` with no cap other than the governance parameter ceiling. `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700` and `MAX_NUMBER_OF_NEURONS_CEILING = 200_000` yield a worst-case of 700 × 200,000 = **140 million iterations**, each performing `NeuronId::from_str`, a `HashMap::entry` operation, and `Decimal` arithmetic. Even at a conservative 100 instructions per iteration this is 14 billion instructions, nearly 3× the IC's 5-billion-instruction per-message limit.

**Loop 2 — maturity distribution (lines 5954–5997):**
```rust
for (neuron_id, neuron_reward_shares) in neuron_id_to_reward_shares {
    let neuron: &mut Neuron = match self.get_neuron_result_mut(&neuron_id) { ... };
    // Decimal division + neuron mutation
}
```
Up to 200,000 neurons, each requiring a mutable neuron lookup and `Decimal` arithmetic, with no instruction-limit break.

**Additional amplifier — `new_rounds_count` loop (lines 5861–5872):**
For a new SNS, `latest_reward_event().end_timestamp_seconds.unwrap_or_default()` returns `0`, so `new_rounds_count = now / round_duration_seconds`. With the minimum `round_duration_seconds = 86400` and a Unix timestamp of ~1.7×10⁹, this yields ~19,675 iterations of `reward_rate_at` arithmetic on the first invocation.

**Permanent liveness failure mechanism:**
`should_distribute_rewards` checks `seconds_since_last_reward_event > round_duration_seconds` using `latest_reward_event().end_timestamp_seconds.unwrap_or_default()`. Because the timer trap rolls back all state mutations, `latest_reward_event` is never updated, so `should_distribute_rewards` returns `true` on every subsequent timer tick, reproducing the same trap indefinitely.

**Confirmed absence of any guard in SNS governance:**
A search for `is_message_over_threshold` or `DISTRIBUTION_MESSAGE_LIMIT` in `rs/sns/governance/**` returns zero matches. The NNS governance already fixed this exact class of bug using `is_message_over_threshold(DISTRIBUTION_MESSAGE_LIMIT)` inside `continue_processing`, breaking the loop when the instruction threshold is approached.

## Impact Explanation

When the instruction limit is exceeded inside a timer callback on the IC, execution traps and all state mutations are rolled back. Because the condition causing the overflow (large neuron count + large ballot count) is persistent state, every subsequent timer invocation reproduces the same trap. The SNS governance canister enters a permanent liveness failure: voting rewards are never distributed, neuron maturity never increases, and `latest_reward_event` is never updated. This freezes the economic incentive layer of the SNS, degrading governance participation and token holder trust.

This matches the **High ($2,000–$10,000)** impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

Any SNS that configures `max_number_of_neurons` near `MAX_NUMBER_OF_NEURONS_CEILING = 200,000` and `max_number_of_proposals_with_ballots` near `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700` is at risk. These are governance-controlled parameters that an SNS community may raise to support growth. An unprivileged participant who creates neurons and votes on proposals — normal, incentivized behavior — organically drives the system toward the failure threshold. No special privilege or coordination is required beyond normal participation at scale. The failure is self-sustaining once triggered.

## Recommendation

Apply the same pattern already used in NNS governance:

1. Move the per-neuron maturity increment out of `distribute_rewards` and into a separate batched timer task analogous to `RewardsDistributionStateMachine` in NNS governance.
2. Store the pending per-neuron reward map in stable memory after `distribute_rewards` computes the reward shares.
3. Process the map in chunks across multiple timer invocations, checking `is_message_over_threshold` after each neuron update and breaking when the limit is approached.
4. Cap `new_rounds_count` to a safe maximum per invocation, or initialize `reward_start_timestamp_seconds` from `genesis_timestamp_seconds` rather than defaulting to Unix epoch 0.

## Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_number_of_proposals_with_ballots = 700`.
2. Create 200,000 neurons by staking the minimum stake amount.
3. Submit 700 proposals and have all neurons vote on each via following (requires only the root neuron to vote explicitly).
4. Wait for `run_periodic_tasks` to fire (every `RUN_PERIODIC_TASKS_INTERVAL`).
5. `should_distribute_rewards` returns `true`; `distribute_rewards` is called synchronously.
6. The ballot aggregation loop at line 5894 iterates 700 × 200,000 = 140 million times, each calling `NeuronId::from_str` and a `HashMap::entry` operation.
7. The IC instruction counter exceeds 5 billion; the timer callback traps; state is rolled back.
8. `latest_reward_event` is never updated; `should_distribute_rewards` returns `true` again on the next tick.
9. The SNS governance canister is permanently unable to distribute voting rewards.

A deterministic integration test using PocketIC can reproduce this by populating `proto.proposals` with 700 proposals each containing 200,000 ballot entries, then calling `run_periodic_tasks` and asserting that `latest_reward_event` is never updated across repeated invocations.
