### Title
Malicious CKD Participant Can Submit Arbitrary Contribution Shares Without Proof of Correct Computation — (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` blindly aggregates `(norm_big_y, norm_big_c)` shares received from each participant with no proof that those values were derived from the participant's actual private share. A single malicious participant can inject arbitrary elliptic-curve points, causing the coordinator to output a corrupted confidential key that honest parties will accept as legitimate.

### Finding Description
Each participant is expected to compute:

```
norm_big_y = lambda_i * y_i * G
norm_big_c = lambda_i * (private_share_i * H(pk, app_id) + y_i * app_pk)
```

The coordinator in `do_ckd_coordinator` (lines 44–56) simply receives and sums every participant's `(norm_big_y, norm_big_c)` pair:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

There is no zero-knowledge proof, consistency check, or any other mechanism that binds `norm_big_c` to the participant's committed public key share. The coordinator cannot distinguish a correctly formed contribution from an arbitrary group element. The analog to the Prysm bug is exact: just as Prysm's inclusion-proof cache omits `kzg_commitments` from the key and therefore accepts sidecars with arbitrary commitments, the CKD coordinator omits verification of the relationship between `norm_big_c` and the participant's public key share, and therefore accepts arbitrary ciphertext contributions. [1](#0-0) 

The `compute_signature_share` helper that honest participants call does produce the correct values, but nothing in the protocol enforces that a participant actually called it. [2](#0-1) 

### Impact Explanation
A malicious participant sends `(norm_big_y′, norm_big_c′)` of their choice. The coordinator accumulates:

```
final_C = (sum over honest i of lambda_i*(x_i*H + y_i*A))
        + norm_big_c′          ← attacker-controlled addend
```

After the application unmasks with `app_sk`, it obtains a confidential key that differs from `msk * H(pk, app_id)` by an attacker-chosen offset. All honest parties accept this output as the protocol result. This is **corruption of CKD output** — a High-severity impact under the allowed scope.

### Likelihood Explanation
Any single participant in the CKD session can mount this attack. No special privilege, leaked key, or cryptographic break is required — the attacker only needs to deviate from the protocol by sending a crafted `CKDOutput` message. The attack is a single-round, single-message deviation with no detectable side-effect visible to honest parties.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
