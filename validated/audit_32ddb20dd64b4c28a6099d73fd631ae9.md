### Title
Unprivileged Caller Can Permanently Freeze Staker Yield via `update_rewards` with `disable_rewards: true` - (File: src/staking/staking.cairo)

### Summary

`update_rewards` in the `Staking` contract is callable by any non-zero address despite the specification requiring it to be restricted to the Starknet sequencer. Because the function writes to the global `last_reward_block` storage variable even when `disable_rewards: true` (returning without distributing rewards), any unprivileged caller can consume the per-block reward slot for every block, permanently denying yield to all stakers.

### Finding Description

The `update_rewards` function is gated only by `general_prerequisites`, which checks that the contract is not paused and the caller is non-zero. There is no role-based access control enforcing the "Only starkware sequencer" restriction stated in the specification. [1](#0-0) 

The function writes `last_reward_block` to the current block number unconditionally — before the `disable_rewards` branch — and then returns early without distributing any rewards when `disable_rewards` is `true`: [2](#0-1) 

`last_reward_block` is a single global storage slot shared across all stakers: [3](#0-2) 

Once `last_reward_block` equals the current block number, any subsequent call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The `general_prerequisites` function — the only guard — enforces no caller identity restriction: [5](#0-4) 

The specification explicitly states the intended access control: [6](#0-5) 

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` in a given block:

1. Advances `last_reward_block` to the current block number.
2. Causes the function to return before distributing any STRK or BTC rewards.
3. Prevents the sequencer from calling `update_rewards` again in the same block (reverts with `REWARDS_ALREADY_UPDATED`).
4. All stakers lose their block rewards for that block.

Repeating this every block results in **permanent freezing of unclaimed yield** for all stakers and their delegators. This matches the **High** impact category: *Permanent freezing of unclaimed yield or unclaimed royalties*.

### Likelihood Explanation

The entry path requires only a non-zero address and a valid active staker address (observable on-chain). The attacker needs no funds beyond gas. The attack is repeatable every block at negligible cost. While Starknet's sequencer controls transaction ordering within a block, the sequencer does not call `update_rewards` in every block, leaving windows where an attacker's call lands first. Even a single successful call per block is sufficient to deny that block's rewards indefinitely.

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, consistent with the specification. For example, introduce a stored `sequencer_address` and assert `get_caller_address() == self.sequencer_address.read()`, or use an existing role (e.g., a dedicated `REWARDS_MANAGER_ROLE`) enforced via the `RolesComponent`. This mirrors the fix applied in the referenced report: restricting who can trigger the state-modifying path that gates legitimate operations.

### Proof of Concept

```
// Attacker (any non-zero address) calls in block N:
staking.update_rewards(
    staker_address: any_active_staker,  // observable on-chain
    disable_rewards: true,              // no rewards distributed
);
// last_reward_block is now N, no rewards sent.

// Sequencer attempts its legitimate call in block N:
staking.update_rewards(
    staker_address: any_active_staker,
    disable_rewards: false,
);
// REVERTS: REWARDS_ALREADY_UPDATED (current_block_number == last_reward_block)

// Attacker repeats every block → all stakers permanently denied yield.
```
