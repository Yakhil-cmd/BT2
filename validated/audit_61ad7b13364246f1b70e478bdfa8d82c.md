### Title
Missing Participant-Set Binding Check in `presign()` Allows Mismatched Triple Reuse — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

### Summary

The `presign()` function in the OT-based ECDSA pipeline explicitly omits the check that the `participants` list supplied at presign time matches the participant set that was used during Beaver triple generation. The code acknowledges this with a comment but relies on an assumption ("presumably they need to have been present") that is not enforced. A malicious library caller or coordinator can invoke `presign` with a different participant set than the one used to generate the triples, causing the Lagrange-coefficient arithmetic inside the presign to be computed over the wrong domain. Every honest party that accepts the resulting `PresignOutput` will hold a cryptographically invalid presignature, and any subsequent signing attempt will produce an invalid signature, permanently denying signing for that round.

### Finding Description

**Root cause — explicit omission of the participant-set check:**

In `src/ecdsa/ot_based_ecdsa/presign.rs` lines 38–40, the code reads:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [1](#0-0) 

The only cross-check that is actually performed is a threshold equality check:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [2](#0-1) 

This verifies only that the *threshold scalar* matches; it does not verify that the *set of participant identifiers* matches. The `PresignArguments` struct carries the triple shares and their threshold, but no binding to the participant list that produced them.

**How the attack works:**

Beaver triples are generated for a specific participant set, e.g. `{P1, P2, P3}` with threshold 2. The Lagrange coefficients used inside triple generation are computed over this set. When `presign` is later called with a *different* participant set, e.g. `{P1, P2}` (still satisfying `threshold ≤ participants.len()`), the presign protocol computes Lagrange coefficients over `{P1, P2}` while the underlying triple shares were constructed for `{P1, P2, P3}`. The resulting presign output is cryptographically invalid: the secret-sharing arithmetic does not reconstruct the correct nonce or key material. Every honest participant who runs the protocol with the mismatched set will accept this corrupted `PresignOutput` because no binding check exists.

**Entry path:**

`presign()` is a public library API. Any caller — including a malicious participant or a malicious coordinator who controls which participant list is passed — can trigger this by supplying a `participants` slice that differs from the one used during triple generation while reusing the same `PresignArguments`. [3](#0-2) 

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs, leading to permanent denial of signing.**

When the participant set is mismatched, the Lagrange interpolation inside presign operates over the wrong domain. The resulting `PresignOutput` is structurally well-formed (it passes all type checks) but cryptographically invalid. Any signing round that consumes this presign output will produce a signature that fails verification against the group public key. Because each `PresignOutput` is one-time-use by design, the corrupted output cannot be retried; the signing round is permanently lost. If a malicious coordinator repeatedly triggers this mismatch, they can deny signing indefinitely for honest parties under valid protocol inputs.

This maps directly to the allowed High impact: *"Corruption of presign outputs so honest parties accept inconsistent or unusable cryptographic outputs"* and *"Permanent denial of signing for honest parties under valid protocol inputs."*

### Likelihood Explanation

The `presign()` function is a public API entry point. The caller supplies both the `participants` slice and the `PresignArguments` (which contains the triple shares) independently. There is no cryptographic binding between the two. A malicious participant or coordinator needs only to call `presign` with a participant list that differs from the triple-generation list — a trivially reachable condition requiring no privileged access, no leaked keys, and no cryptographic break.

### Recommendation

Record the participant set (as a sorted, canonical list or its hash) inside the triple share metadata at generation time, and verify in `presign()` that the supplied `participants` list matches this recorded set before proceeding. Concretely:

1. Extend the triple-share output type to include the participant-set identifier (e.g., a hash of the sorted participant list).
2. In `presign()`, after constructing the `ParticipantList`, compute the same hash and compare it against `args.triple0.1.participants_hash` and `args.triple1.1.participants_hash`, returning `InitializationError::BadParameters` on mismatch.
3. Remove the comment that justifies omitting this check.

### Proof of Concept

```
1. Run triple generation for participant set S1 = {P1, P2, P3}, threshold = 2.
   Each participant Pi holds triple shares (a_i, b_i, c_i) keyed to S1.

2. Construct PresignArguments using P1's triple shares from step 1.
   args.threshold = 2, args.triple0 and args.triple1 carry P1's shares.

3. Call presign(&[P1, P2], P1, args):
   - participants.len() = 2 >= threshold = 2  ✓  (threshold check passes)
   - args.threshold == args.triple0.1.threshold  ✓  (threshold equality passes)
   - NO check that {P1, P2} == {P1, P2, P3}     ✗  (missing check)

4. presign() proceeds. Lagrange coefficients are computed for {P1, P2},
   but the triple shares encode the secret-sharing over {P1, P2, P3}.
   The arithmetic is inconsistent; the PresignOutput is cryptographically invalid.

5. P2 runs the same presign with {P1, P2} and its own (also S1-keyed) shares.
   Both parties complete the protocol and hold a corrupted PresignOutput.

6. Signing with this PresignOutput produces a signature that fails
   verification. The signing round is permanently lost.
``` [4](#0-3)
