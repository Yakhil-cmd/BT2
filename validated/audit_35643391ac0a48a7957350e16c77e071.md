### Title
Missing Lower-Bound Validation on `gas_limit` and `native_price` in Block Metadata Initialization Allows Prover to Produce Valid Proofs for Blocks That Reject All L2 Transactions - (`basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

### Summary
The ZK-mode block metadata initialization function `metadata_op` reads `BlockMetadataFromOracle` from the oracle and validates only that `gas_limit` does not exceed `MAX_BLOCK_GAS_LIMIT`. It performs no lower-bound checks: `gas_limit = 0`, `native_price = 0`, and `pubdata_limit = 0` all pass initialization without error. A prover supplying `gas_limit = 0` causes every L2 transaction to be rejected with a per-transaction validation error (`CallerGasLimitMoreThanBlock`), leaving the block structurally valid and fully provable — an empty block whose ZK proof the settlement layer will accept. This is the direct analog of M-01 scenario 2 (zero supply causing all mints to revert): a critical block-level parameter is not validated at initialization, and the dysfunction surfaces only when transactions are processed.

### Finding Description

`metadata_op` in `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs` deserializes `BlockMetadataFromOracle` from the oracle and applies exactly one range check:

```rust
if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT
    || metadata.individual_tx_gas_limit() > MAX_TX_GAS_LIMIT
{
    return Err(internal_error!("block or tx gas limit is too high"));
}
``` [1](#0-0) 

No lower-bound check exists. `gas_limit = 0` satisfies `0 > MAX_BLOCK_GAS_LIMIT == false`, so it passes.

`BlockMetadataFromOracle` carries `gas_limit: u64`, `native_price: U256`, and `pubdata_limit: u64` as plain fields with no invariants enforced at deserialization time: [2](#0-1) 

The deserialization path (`UsizeDeserializable for BlockMetadataFromOracle`) also performs no validation: [3](#0-2) 

**`gas_limit = 0` path (most impactful):**

When any non-service L2 transaction is validated, `validation_impl.rs` checks:

```rust
require!(
    tx_gas_limit <= block_gas_limit,
    InvalidTransaction::CallerGasLimitMoreThanBlock,
    system
)?;
``` [4](#0-3) 

With `block_gas_limit = 0`, every L2 transaction with any non-zero gas limit is rejected with `InvalidTransaction::CallerGasLimitMoreThanBlock`. This is a *per-transaction validation error*, not an `InternalError`. The block itself remains structurally valid, all L1/service transactions are unaffected (the check is gated on `!transaction.is_service()`), and the prover can generate a valid ZK proof for the resulting empty block.

**`native_price = 0` path (causes block-level `InternalError`):**

The `native_price != 0` guard exists only inside per-transaction validation:

```rust
if native_price.is_zero() {
    return Err(internal_error!("Native price cannot be 0").into());
}
``` [5](#0-4) 

This is an `InternalError`, not a `TxError::Validation`. It propagates as a block-level failure rather than a per-transaction rejection. The check is absent from `metadata_op`, so the block is initialized successfully and the error only surfaces when the first L2 transaction is processed.

**`pubdata_limit = 0` path:**

Similarly, `pubdata_limit = 0` is never checked in `metadata_op`. Any L2 transaction that writes pubdata will immediately hit `BlockPubdataLimitReached`.

The oracle documentation explicitly states that oracle responses are untrusted and *must* be validated by the calling code: [6](#0-5) 

The `metadata_op` trait comment also states that the implementation "should … validate the metadata before block processing begins": [7](#0-6) 

### Impact Explanation

A prover supplying `gas_limit = 0` via the `BLOCK_METADATA_QUERY_ID` oracle response produces a block in which every L2 transaction is rejected with a per-transaction validation error. The block is structurally valid; the prover can generate and submit a valid ZK proof. The settlement layer accepts the proof and advances the chain. No L2 user transactions are included. Repeated across consecutive blocks this constitutes a sustained, provable denial-of-service against all L2 transaction senders. L1→L2 (service) transactions are unaffected because the gas-limit check is skipped for service transactions.

### Likelihood Explanation

The prover is an explicitly listed attacker class in the scope ("prover/forward execution input"). Block metadata is part of the oracle's non-deterministic input; the prover controls it. The `gas_limit` field is a plain `u64` with no enforced minimum. The attack requires only that the prover serialize `gas_limit = 0` into the `BLOCK_METADATA_QUERY_ID` response — a trivial oracle manipulation. No key compromise or governance attack is required.
