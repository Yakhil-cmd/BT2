### Title
Missing Cross-Phase Binding of `max_malicious` Enables Split-View Secret Key Extraction — (File: `src/ecdsa/robust_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/presign.rs`)

---

### Summary

The robust ECDSA `sign()` function accepts a fresh, caller-supplied `max_malicious` parameter that is never validated against the `max_malicious` used during the corresponding `presign()` phase. Because `PresignOutput` does not record the presigning threshold, a malicious coordinator can invoke `sign()` with a smaller `max_malicious` (N2 < N1), violating the documented N1 = N2 = 2t+1 invariant and enabling the codebase's own documented split-view attack that recovers the aggregate secret key.

---

### Finding Description

The robust ECDSA scheme requires that the number of participants in presigning (N1) and signing (N2) be identical and equal to `2 * max_malicious + 1`. The codebase's own security documentation states this explicitly:

> "Do **not** allow N1 > 2t+1, and Do **not** allow N2 < N1. Allowing larger presigning sets or smaller signing sets enables split-view and presignature-reuse attacks when a coordinator can run parallel or partially overlapping signing sessions." [1](#0-0) 

`presign()` correctly enforces `participants.len() == 2*max_malicious+1`: [2](#0-1) 

`sign()` also enforces `participants.len() == 2*max_malicious+1`, but against its **own** freshly supplied `max_malicious` argument — not the one used during presigning: [3](#0-2) 

The `PresignOutput` struct stores only cryptographic material (`big_r`, `alpha`, `beta`, `c`, `e`) and does **not** record the `max_malicious` value used during presigning: [4](#0-3) 

Because `sign()` takes `max_malicious` as an independent `impl Into<MaxMalicious>` argument with no cross-check against the presignature: [5](#0-4) 

…a malicious coordinator can supply `max_malicious = t-1` to `sign()` while the presignature was generated with `max_malicious = t`. Both individual calls pass their own local validation, but the cross-phase invariant N1 = N2 is silently broken.

---

### Impact Explanation

**Critical — Extraction of the aggregate secret signing key.**

The split-view attack proceeds as follows once N2 < N1 is achieved:

1. Presign with N1 = 2t+1 participants (set S1). Each participant i holds shares `(alpha_i, beta_i, c_i, e_i)` tied to the shared nonce k.
2. The coordinator rerandomizes the presignature twice: once bound to subset S2 ⊂ S1 of size 2t−1, once bound to subset S3 ⊂ S1 of size 2t−1 (S2 ≠ S3).
3. The coordinator runs two signing sessions: session A with S2 using `max_malicious = t-1`, session B with S3 using `max_malicious = t-1`. Both pass `sign()`'s local validation.
4. In each session, honest participants compute Lagrange-weighted signature shares using their respective participant sets (S2 or S3), producing different linear combinations of the same underlying secret material.
5. The coordinator collects the two sets of signature shares. Because both sessions derive from the same underlying nonce k (with only a public rerandomization delta difference), the coordinator can apply standard ECDSA nonce-reuse algebra across the two sessions to solve for the aggregate secret key x.

The documentation explicitly acknowledges this attack class:

> "a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions." [6](#0-5) 

The missing binding is the necessary and sufficient condition for the coordinator to trigger this path.

---

### Likelihood Explanation

**High.** The malicious coordinator is an explicitly modeled adversary in this library. The `sign()` public API accepts `max_malicious` as a plain caller-supplied value with no attestation. No out-of-band mechanism prevents a coordinator from passing a smaller `max_malicious` than was used in `presign()`. Honest participants have no way to detect the mismatch because `PresignOutput` carries no record of the presigning threshold. The attack requires only two signing sessions and control of the coordinator role — both are within the documented threat model.

---

### Recommendation

**Short term:** Embed `max_malicious` (or equivalently the presigning participant count) inside `PresignOutput` and `RerandomizedPresignOutput`. In `sign()`, assert that the stored value equals the `max_malicious` argument before proceeding:

```rust
// In PresignOutput / RerandomizedPresignOutput:
pub max_malicious: MaxMalicious,

// In sign():
if presignature.max_malicious != max_malicious.into() {
    return Err(InitializationError::BadParameters(
        "max_malicious must match the value used during presigning".to_string(),
    ));
}
```

**Long term:** Wherever protocol outputs cross phase boundaries (presign → sign, keygen → presign), bind all security-critical parameters into the output struct so that downstream phases cannot silently deviate from the values established upstream. This is the threshold-signatures analog of the Yield Protocol recommendation to introduce hard limits on all critical system variables.

---

### Proof of Concept

```rust

let presign_output = presign(&participants_5, me, PresignArguments {
    keygen_out,
    max_malicious: 2.into(),  // N1 = 2*2+1 = 5
}, rng).unwrap();

// ... coordinator collects all 5 PresignOutputs, rerandomizes for two subsets of 3 ...

// Sign with max_malicious = 1 → N2 = 3 < N1 = 5
// sign() only checks: participants.len() == 2*1+1 == 3 ✓
// sign() does NOT check: presignature was generated with max_malicious == 1
let sign_protocol = sign(
    &participants_3_subset_A,  // 3 participants
    coordinator,
    1usize,                    // max_malicious = 1, not 2 as used in presign
    me,
    public_key,
    rerandomized_presig_A,
    msg_hash,
).unwrap(); // succeeds — no cross-phase check

// Repeat with subset_B → coordinator now holds two sessions with related nonces
// → apply nonce-reuse algebra → recover secret key x
```
