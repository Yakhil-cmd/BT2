### Title
Presignature Single-Use Not Enforced at Type Level Enables Nonce-Reuse Secret Key Extraction — (`src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both the Robust ECDSA and OT-based ECDSA schemes require that each `PresignOutput` is consumed exactly once. However, `PresignOutput` derives `Clone` and `rerandomize_presign()` accepts it by shared reference (`&PresignOutput`), meaning the same presignature can be rerandomized and fed into multiple independent signing sessions. A malicious coordinator who controls session initiation can exploit this to obtain two valid signatures over different messages that share a multiplicatively related nonce, recovering the aggregate secret key via standard ECDSA nonce-reuse algebra.

---

### Finding Description

**Root cause — `PresignOutput` is freely cloneable and rerandomization does not consume it:**

In the Robust ECDSA module:

Because `PresignOutput` derives `Clone` and `rerandomize_presign` borrows rather than moves it, a caller can legally call `rerandomize_presign` twice on the same `PresignOutput` with two different `RerandomizationArguments` (different `(h, ε, ρ)` tuples), producing two distinct `RerandomizedPresignOutput` values — each with a different `delta`-scaled nonce `R' = δ·R` — and then drive two separate signing sessions to completion.

**The security documentation explicitly forbids this but provides no enforcement:** [5](#0-4) [6](#0-5) 

The presigning code comment says "it's crucial that a presignature is never reused," and the security doc says "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." Neither translates into a compile-time or runtime guard.

**Attack flow (malicious coordinator, Robust ECDSA):**

1. All `2t+1` participants complete the presigning protocol, each holding their share of `PresignOutput` `(R, α_i, β_i, c_i, e_i)`.
2. The coordinator calls `rerandomize_presign(&presign_out, args_M1)` → `rerandomized_1` (nonce `R₁ = δ₁·R`).
3. The coordinator calls `rerandomize_presign(&presign_out, args_M2)` → `rerandomized_2` (nonce `R₂ = δ₂·R`). The same `presign_out` is used; Rust permits this because it is borrowed, not moved.
4. The coordinator runs signing session 1 with `rerandomized_1` and message hash `h₁`, collecting all `s_i` shares. Participants have no record that this presignature was already used.
5. The coordinator runs signing session 2 with `rerandomized_2` and message hash `h₂`, collecting all `s_i` shares again.
6. The coordinator now holds `(R₁, s₁)` and `(R₂, s₂)` where `R₁ = δ₁·R`, `R₂ = δ₂·R`. The nonces are multiplicatively related by the known ratio `δ₁/δ₂`. Standard ECDSA nonce-reuse algebra recovers the aggregate secret key `x`.

The security documentation confirms this exact scenario:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [7](#0-6) 

---

### Impact Explanation

**Critical — Extraction of the aggregate secret key.**

Two signing sessions driven from the same `PresignOutput` yield signatures with multiplicatively related nonces. Given `(R₁, s₁, h₁)` and `(R₂, s₂, h₂)` and the known scalars `δ₁`, `δ₂`, the coordinator can solve for the master secret `x` algebraically. This is a full key compromise: the attacker can thereafter forge arbitrary signatures without any participant cooperation.

---

### Likelihood Explanation

The coordinator is an explicitly in-scope attacker role. The coordinator controls session initiation and knows both `delta` values (they computed them via HKDF). Honest participants have no per-presignature usage ledger — the library provides none — so they will participate in both sessions without objection. The only barrier is the documentation warning, which is not enforced in code. Any deployment where the coordinator role is held by a potentially adversarial party (e.g., a threshold-wallet service, an MPC node operator) is directly exposed.

---

### Recommendation

Enforce single-use semantics at the Rust type level, mirroring the analog fix of adding a tracking mechanism:

1. **Remove `Clone` from `PresignOutput`** in both `src/ecdsa/robust_ecdsa/mod.rs` and `src/ecdsa/ot_based_ecdsa/mod.rs`. Without `Clone`, the value can only be moved, not duplicated.
2. **Change `rerandomize_presign` to consume `PresignOutput` by value** (`presignature: PresignOutput` instead of `presignature: &PresignOutput`). This makes the Rust borrow checker statically enforce that each presignature is rerandomized at most once.
3. Optionally, also remove `Clone` from `RerandomizedPresignOutput` so that the rerandomized output itself cannot be duplicated before being passed to `sign()`.

These changes impose zero runtime cost and make the security invariant "one presignature, one signature" a compile-time guarantee rather than a documentation note.

### Proof of Concept

```rust
// Demonstrates that the same PresignOutput can be rerandomized twice
// (compiles and runs today — no Clone call needed, just two borrows)

let presign_out: PresignOutput = /* result of presign protocol */;

let args_m1 = RerandomizationArguments { big_r: presign_out.big_r, tweak: tweak1, ... };
let args_m2 = RerandomizationArguments { big_r: presign_out.big_r, tweak: tweak2, ... };

// Both calls succeed because rerandomize_presign takes &PresignOutput
let rerandomized_1 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args_m1).unwrap();
let rerandomized_2 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args_m2).unwrap();

// Coordinator now drives two signing sessions with the same underlying nonce R.
// Collecting (R1, s1) and (R2, s2) where R1 = δ1·R, R2 = δ2·R recovers x via:
//   s1 = δ1^{-1} * (h1 * k + Rx * σ)
//   s2 = δ2^{-1} * (h2 * k + Rx * σ)
// Eliminating (k, σ) yields x.
``` [8](#0-7) [9](#0-8)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
