### Title
`starknet_estimateFee` / `starknet_simulateTransactions` omit `l2_gas_consumed` while `overall_fee` silently includes L2 gas cost, making the authoritative description wrong and preventing correct `l2_gas.max_amount` resource-bound setting - (File: `crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is computed from the full three-component gas vector (L1 gas + L1 data gas + L2 gas), yet both the in-code docstring and the OpenAPI spec authoritatively describe `overall_fee` as equalling only `gas_consumed * gas_price + data_gas_consumed * data_gas_price`. The L2 gas term is silently baked into `overall_fee` with no corresponding quantity field, so callers cannot reconstruct the fee from the other response fields, cannot verify the breakdown, and—critically—cannot derive the `l2_gas.max_amount` they must supply in a V3 `AllResourceBounds` transaction.

### Finding Description

**`FeeEstimation` struct** (`crates/apollo_rpc_execution/src/objects.rs`, lines 94–113):

```rust

    pub gas_consumed: Felt,       // l1_gas
    pub l1_gas_price: GasPrice,
    pub data_gas_consumed: Felt,  // l1_data_gas
    pub l1_data_gas_price: GasPrice,
    // TODO(Tzahi): Add l2_gas_consumed. ...
    pub l2_gas_price: GasPrice,   // price present, quantity absent
    /// The total amount of fee. This is equal to:
    /// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
    pub overall_fee: Fee,         // ← description is wrong
    pub unit: PriceUnit,

```

**`tx_execution_output_to_fee_estimation`** (lines 161–183) sets:

```rust
overall_fee: tx_execution_output.execution_info.receipt.fee,
```

`receipt.fee` is computed in `TransactionReceipt::from_params` (`crates/blockifier/src/fee/receipt.rs`, line 114) as:

```rust
tx_context.tx_info.get_fee_by_gas_vector(
    &tx_context.block_context.block_info,
    gas,          // full GasVector: l1_gas + l1_data_gas + l2_gas
    tx_context.effective_tip(),
)
```

`gas` is the full `GasVector` including `l2_gas` (line 105–109 of `receipt.rs`). Therefore:

```
overall_fee = l1_gas × l1_gas_price
            + l1_data_gas × l1_data_gas_price
            + l2_gas × l2_gas_price
```

The OpenAPI spec (`crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json`, line 3650) repeats the same wrong formula:

```json
"description": "The estimated fee for the transaction … equals to
                gas_consumed*gas_price + data_gas_consumed*data_gas_price"
```

The L2 gas term is absent from both the description and the response fields.

### Impact Explanation

This matches **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

1. **Wrong authoritative value.** The `overall_fee` description is wrong: callers who compute `gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price` will obtain a value strictly less than `overall_fee` whenever L2 gas is non-zero. The discrepancy is invisible because `l2_gas_consumed` is absent from the response.

2. **Broken resource-bound derivation.** V3 transactions with `AllResourceBounds` require the caller to supply `l2_gas.max_amount`. The only authoritative source for this value is `starknet_estimateFee`. Because `l2_gas_consumed` is not returned, callers cannot derive a correct `max_amount`. Setting it too low causes the transaction to revert at execution; setting it arbitrarily high wastes funds and may exceed the gateway's `max_l2_gas_amount` cap (`StatelessTransactionValidator::validate_resource_bounds`, line 80–85 of `stateless_transaction_validator.rs`).

3. **Slippage analog.** The external report's invariant is: *a user cannot bound what they receive/pay because a key quantity is not exposed*. Here the key quantity is `l2_gas_consumed`: users cannot bound their L2 gas spend because the estimation endpoint withholds the consumed amount while embedding its cost in `overall_fee`.

### Likelihood Explanation

Every V3 transaction that consumes any L2 gas (i.e., any Cairo 1 contract call) triggers this discrepancy. The `starknet_estimateFee` and `starknet_simulateTransactions` endpoints are the primary tool wallets and SDKs use to populate resource bounds before submission. The issue is present in the production code path with no flag or configuration that corrects it.

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`.
2. Correct the `overall_fee` docstring in `FeeEstimation` to include the L2 gas term.
3. Update the `FEE_ESTIMATE` schema in `starknet_api_openrpc.json` to add `l2_gas_consumed` as a required field and fix the `overall_fee` description.

### Proof of Concept

1. Submit any Cairo 1 invoke transaction to `starknet_estimateFee`.
2. Receive `FeeEstimation { gas_consumed, l1_gas_price, data_gas_consumed, l1_data_gas_price, l2_gas_price, overall_fee, unit }`.
3. Compute `reconstructed = gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price`.
4. Observe `reconstructed < overall_fee`; the gap equals `l2_gas_consumed × l2_gas_price`, which is not disclosed.
5. Attempt to set `l2_gas.max_amount` for the corresponding V3 transaction submission — no field in the response provides this value, so the caller must guess or use an arbitrary upper bound.

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```
