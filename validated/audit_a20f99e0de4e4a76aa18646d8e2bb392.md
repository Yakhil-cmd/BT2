### Title
Multi-NFT Royalty Amount Understated in WalletConnect Signing Confirmation Dialog - (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` divides the fungible amount equally among all NFT royalty percentages before applying each percentage, producing a "Total Amount with Royalties" that is systematically understated in the WalletConnect signing confirmation dialog. The actual on-chain royalty charge applies each NFT's royalty to the **full** fungible amount, not a split portion. A user who accepts a multi-NFT offer via WalletConnect will approve a transaction displaying a lower total cost than what is actually deducted from their wallet.

### Finding Description

`formatAmountWithRoyalties` is called by `withRoyaltyTotals` to compute the `amountWithRoyalties` field shown in the Electron WalletConnect confirmation dialog (`Confirm.tsx`) for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` commands. [1](#0-0) 

The flawed logic:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length); // divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

For N NFTs with royalty percentages `r₁, r₂, …, rₙ` and fungible amount `A`:

| | Formula |
|---|---|
| **Displayed (wrong)** | `A + (A/N) * (r₁ + r₂ + … + rₙ) / 10000` |
| **Actual on-chain** | `A + A * r₁/10000 + A * r₂/10000 + … + A * rₙ/10000` |

The displayed total is exactly `N` times smaller in the royalty component than what the protocol charges.

The `OfferBuilderXCHSection` (used in the standard offer builder UI) correctly sums royalty payments from the backend's `calculate_royalties` RPC for each NFT against the full amount: [2](#0-1) 

This is the same two-metric inconsistency as the external report: one code path (WalletConnect confirmation) uses a split-amount basis; the other (actual protocol execution) uses the full-amount basis.

The understated `amountWithRoyalties` is rendered directly in the WalletConnect confirmation dialog: [3](#0-2) 

### Impact Explanation

A user accepting a multi-NFT offer via WalletConnect sees a "Total Amount with Royalties" that is lower than what will actually be deducted. With N NFTs each carrying royalty percentage R, the displayed royalty is `R/N` of the true royalty. The gap grows with both N and R. The user approves the transaction based on the understated figure; the wallet daemon then executes the spend at the correct (higher) amount, causing an unexpected balance reduction.

This matches the **High** impact class: WalletConnect state causes a user to approve the wrong amount.

### Likelihood Explanation

An unprivileged attacker can:
1. Mint NFTs with high royalty percentages (a normal, permissionless operation).
2. Construct an offer involving multiple such NFTs in exchange for XCH or CATs.
3. Deliver the offer string to a victim via any WalletConnect-connected dApp.
4. The victim's confirmation dialog shows the understated total; if they approve, more funds are spent than displayed.

No leaked keys, host compromise, or social engineering beyond presenting a legitimate-looking offer is required.

### Recommendation

Remove the `splitAmount` division. Apply each royalty percentage to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This matches the backend's `calculate_royalties` behavior and the `OfferBuilderXCHSection` accumulation logic. [4](#0-3) 

### Proof of Concept

The existing test in `parseCommandDisplay.test.ts` encodes the buggy behavior as the expected value:

- Amount: 100,000,000 mojos (0.0001 XCH)
- NFT 1 royalty: 500 bp (5%), NFT 2 royalty: 10 bp (0.1%)
- **Displayed** (current code): `splitAmount = 50,000,000`; royalty = `(50M×500 + 50M×10)/10000 = 2,550,000` mojos → total `0.00010255 XCH`
- **Correct on-chain**: royalty = `(100M×500 + 100M×10)/10000 = 5,100,000` mojos → total `0.0001051 XCH`

The test asserts `amountWithRoyalties: '0.00010255'`, confirming the dialog shows ~50% of the true royalty cost when two NFTs are involved. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
