### Title
Gateway Skips `__validate__` Signature Check for Nonce-1 Invoke Transactions When Deploy-Account Is Pending — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the account's `__validate__` entry-point call for any Invoke transaction with `nonce == 1` submitted while the sender's on-chain nonce is still `0` and the account appears in the mempool or a recent block. Because the only guard is a mempool-presence check (not a signature check), any unprivileged attacker can submit an Invoke transaction with an arbitrary payload and an invalid/empty signature for any account that has a pending `DeployAccount` transaction, and the gateway will admit it to the mempool without ever verifying the signature.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `validate: false` in `ExecutionFlags`: [2](#0-1) 

This means `StatefulValidator::perform_validations` reaches the branch: [3](#0-2) 

…and returns `Ok(())` without ever calling `tx.validate_tx(…)` — the account's `__validate__` Cairo entry point is never executed. The transaction's signature is therefore never verified at the gateway admission stage.

The only check that runs before this skip is `validate_by_mempool`, which calls `mempool_client.validate_tx(validation_args)`: [4](#0-3) 

The mempool's `validate_tx` checks for duplicate transaction hashes and nonce-range validity, but does **not** verify the cryptographic signature: [5](#0-4) 

The `account_tx_in_pool_or_recent_block` check only confirms that the account has *some* transaction in the mempool or a recently committed block — it does not bind the incoming transaction to any specific signer or payload: [6](#0-5) 

### Impact Explanation

An attacker who observes that account `A` has a pending `DeployAccount` transaction (nonce 0) in the mempool can submit an `Invoke` transaction for `A` with:
- `nonce = 1`
- arbitrary `calldata`
- an invalid or empty `signature`

The gateway will:
1. Pass the nonce range check (`0 ≤ 1 ≤ 200`).
2. Pass the mempool `validate_tx` check (no duplicate hash, valid nonce).
3. Detect that `A` is in the mempool via `account_tx_in_pool_or_recent_block`.
4. Skip `__validate__` entirely.
5. Admit the transaction to the mempool.

The admitted transaction carries an invalid signature and will revert during actual block execution (because `new_for_sequencing` sets `validate: true` and `strict_nonce_check: true`): [7](#0-6) 

However, before that revert, the attacker's transaction occupies the `(A, nonce=1)` slot in the mempool. If the attacker submits with a higher tip than Alice's legitimate nonce-1 transaction, the mempool's fee-escalation logic may replace Alice's valid transaction with the attacker's invalid one, permanently blocking Alice's first post-deploy action. Even without fee escalation, the sequencer wastes execution resources on a transaction that is guaranteed to revert.

This matches the **High** impact: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation

The trigger is fully unprivileged. The attacker only needs to observe the mempool (or the gateway's public API) to learn that a `DeployAccount` for a target address is pending. This is observable from any public RPC endpoint. The attack requires no special access, no privileged keys, and no on-chain funds beyond the gas needed to submit the transaction to the gateway.

### Recommendation

The `skip_stateful_validations` UX shortcut should not bypass signature verification entirely. Two options:

1. **Verify the signature against the deploy-account's expected class**: Before skipping `__validate__`, verify the transaction signature off-chain using the class hash declared in the pending `DeployAccount` transaction. Only skip the on-chain call if the off-chain check passes.

2. **Defer, do not skip**: Instead of skipping `__validate__`, defer the invoke transaction until the `DeployAccount` is committed, then run full validation. This eliminates the UX shortcut but closes the admission bypass.

At minimum, the gateway should reject any nonce-1 invoke for an undeployed account whose signature does not match the expected format for the declared class, rather than admitting it unconditionally.

### Proof of Concept

```
1. Alice submits DeployAccount(nonce=0, class_hash=C) → admitted to mempool.
   account_tx_in_pool_or_recent_block(Alice) == true.
   On-chain nonce for Alice == 0.

2. Attacker submits Invoke(sender=Alice, nonce=1, calldata=DRAIN_FUNDS, signature=0x0).
   Gateway stateful validation:
     - validate_nonce: 0 <= 1 <= 200 ✓
     - validate_by_mempool: no duplicate hash, nonce in range ✓
     - skip_stateful_validations:
         tx.nonce() == 1 ✓
         account_nonce == 0 ✓
         account_tx_in_pool_or_recent_block(Alice) == true ✓
         → returns true (skip __validate__)
     - run_validate_entry_point(skip_validate=true):
         ExecutionFlags { validate: false, ... }
         __validate__ NOT called ✓
   → Attacker's transaction admitted to mempool.

3. Batcher picks up DeployAccount then Invoke.
   Invoke execution calls __validate__ → signature invalid → tx reverts.
   But if attacker's tip > Alice's tip, Alice's legitimate nonce-1 tx was
   already evicted from the mempool by fee escalation.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
