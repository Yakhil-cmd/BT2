### Title
Consensus Divergence via Identity-Key-Gated Blocklist in `ProcessProposalHandler` — (`app/proposal.go`)

### Summary

`ProcessProposalHandler` and `SetBlockList` produce non-deterministic, validator-dependent results for the same proposal when the validator set contains a mix of nodes with and without an age identity key. A blocked address submitting a single unprivileged transaction is sufficient to trigger a permanent consensus split and chain halt.

---

### Finding Description

`ProposalHandler` is initialized with an empty `blocklist` map: [1](#0-0) 

`SetBlockList`, called every `EndBlock` via `app.RefreshBlockList`, silently returns `nil` and **never populates** `h.blocklist` when `h.Identity == nil`: [2](#0-1) 

For validators that do have an identity key, `SetBlockList` decrypts the stored blob and populates `h.blocklist` with the blocked addresses: [3](#0-2) 

`ProcessProposalHandler` then branches on `len(h.blocklist) == 0`: [4](#0-3) 

- **Validator without identity key**: `h.blocklist` is always empty → always returns `ACCEPT`.
- **Validator with identity key**: `h.blocklist` is populated → returns `REJECT` for any proposal containing a tx from a blocked address.

The raw encrypted blob is stored on-chain and is readable by anyone via the `BlockList` gRPC query: [5](#0-4) 

There is no enforcement anywhere that validators must have an identity key configured. The comment in the struct even acknowledges the nil case as a normal operating mode: [6](#0-5) 

---

### Impact Explanation

When the proposer has no identity key, their `PrepareProposal` also uses `ValidateTransaction` (via `ExtTxSelector`) with an empty blocklist, so they include blocked txs in the proposal. Validators with identity keys then call `ProcessProposalHandler`, find a blocked signer, and return `REJECT`. Validators without identity keys return `ACCEPT`. CometBFT requires all honest validators to agree on `ProcessProposal` for the same proposal; a split between `ACCEPT` and `REJECT` causes a consensus failure and chain halt.

This is a **Critical** impact: consensus divergence / chain halt triggered by an unprivileged transaction.

---

### Likelihood Explanation

- The blocklist feature is explicitly designed for production use (integration tests, `MsgStoreBlockList`, admin-gated).
- Nothing in the node startup, configuration validation, or `SetBlockList` warns or errors when `Identity == nil` while a non-empty blocklist exists on-chain.
- Any validator that misconfigures or omits their identity key silently becomes a divergence point.
- The attacker only needs to submit a standard transaction from a blocked address; no special privileges are required.

---

### Recommendation

`ProcessProposalHandler` must not silently skip enforcement when `h.Identity == nil` and a non-empty blocklist blob exists on-chain. Options:

1. **Fail closed**: if `h.Identity == nil` and the on-chain blob is non-empty, return `REJECT` (or panic/halt) rather than `ACCEPT`.
2. **Enforce identity at startup**: if a non-empty blocklist is stored on-chain and the node has no identity key, refuse to start or log a fatal error.
3. **Store the blocklist in plaintext** (or with a separate plaintext address list) so all validators can enforce it regardless of identity key, using the encrypted blob only for confidentiality of the full list contents.

---

### Proof of Concept

1. Start a two-validator local testnet. Give validator A an age identity key; leave validator B without one (`Identity == nil`).
2. Store a blocklist containing address `X` (encrypted to validator A's key) via `MsgStoreBlockList`.
3. After `EndBlock`, validator A's `h.blocklist` is populated; validator B's remains empty.
4. Submit a tx signed by address `X`.
5. Wait for validator B to be the proposer. Validator B's `PrepareProposal` has an empty blocklist → includes the tx.
6. Validator A calls `ProcessProposalHandler` → `h.blocklist` non-empty → `ValidateTransaction` returns error → returns `REJECT`.
7. Validator B calls `ProcessProposalHandler` → `h.blocklist` empty → returns `ACCEPT`.
8. Consensus split; the chain halts.

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}
