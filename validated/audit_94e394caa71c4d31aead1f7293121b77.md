### Title
`TurnBridge` msg handler is a no-op stub with no authorization check, permanently disabling the bridge circuit breaker - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` gRPC handler in `x/cronos/keeper/msg_server.go` is implemented as a bare stub that immediately returns `(nil, nil)` — no permission check, no state write. Any unprivileged account can submit a `MsgTurnBridge` transaction and receive a success response, while the bridge-enabled state is never actually mutated. The intended emergency circuit-breaker for the Gravity bridge is therefore permanently inoperative.

### Finding Description
The Cronos permission system explicitly defines `CanTurnBridge` as a guarded capability:

```go
// x/cronos/keeper/permissions.go
const (
    CanChangeTokenMapping uint64 = 1 << iota // 1
    CanTurnBridge                             // 2
    All = CanChangeTokenMapping | CanTurnBridge
)
```

`UpdateTokenMapping`, the sibling privileged handler, correctly enforces this gate:

```go
// x/cronos/keeper/msg_server.go  lines 69-82
func (k msgServer) UpdateTokenMapping(...) ... {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    ...
}
```

`TurnBridge`, however, is implemented as:

```go
// x/cronos/keeper/msg_server.go  lines 84-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

There is no call to `HasPermission`, no call to `k.GetParams`, and no KV-store write of any kind. The handler is fully registered in the gRPC service descriptor and reachable on-chain via the `MsgTurnBridge` proto message.

### Impact Explanation
Two distinct impacts arise from this stub:

1. **Authorization bypass (High):** The `CanTurnBridge` permission check — the exact guard the permission system was designed to enforce — is completely absent. Any unprivileged address can submit `MsgTurnBridge` and receive a `code: 0` success response, satisfying the "Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks" criterion.

2. **Permanent inability to halt bridge flows (High):** Because the handler writes nothing, the bridge-enabled state can never be set to `false`. The emergency circuit-breaker that the Cronos team explicitly designed (ADR-009, `CanTurnBridge` permission, `CmdTurnBridge` CLI) to stop Gravity bridge asset flows in a security incident is permanently broken. If a vulnerability in the bridge is discovered, operators have no on-chain mechanism to halt outbound `send_to_evm_chain` or inbound conversion flows, satisfying the "Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows" criterion.

### Likelihood Explanation
The entry path is fully open: `MsgTurnBridge` is a standard Cosmos SDK message, signed by any key, broadcast to any node. No special privilege, leaked key, or validator access is required. The handler is wired into the module's message router and will be executed on every node in the network.

### Recommendation
Implement `TurnBridge` with the same pattern used by `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `sdkerrors.ErrUnauthorized` on failure.
2. Write the bridge-enabled flag to the KV store (e.g., a dedicated key under the cronos store key).
3. Add a corresponding `IsBridgeEnabled(ctx)` keeper method and gate all Gravity bridge EVM-hook handlers on it.

### Proof of Concept
```
# Any unprivileged key
cronosd tx cronos turn-bridge false \
  --from attacker \
  --chain-id cronosmainnet_25-1 \
  -y

# Result: code: 0, raw_log: "[]"
# Bridge state: unchanged — bridge remains permanently active
# Admin's emergency circuit-breaker: silently swallowed
```

The `test_gravity_turn_bridge` integration test at `integration_tests/test_gravity.py:664-665` asserts `rsp["code"] == 0` after calling `turn_bridge("false", from_="validator")` and then expects subsequent bridge sends to fail — an expectation that can never be satisfied because the handler is a no-op.
