### Title
`MAX_NUMBER_OF_LOGS` Invariant Violated in Multiblock Batch Proving Path, Causing Prover Panic — (`basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`)

---

### Summary

The per-block L2→L1 log limit (`MAX_NUMBER_OF_LOGS = 16_384`) is enforced only at the individual block level. In the multiblock batch proving path, logs from every block in the batch are accumulated into a fixed-capacity `ArrayVec<Bytes32, 16384>`. If the combined log count across all blocks in a single batch exceeds 16,384, `ArrayVec::push` panics, making the batch permanently unprovable and halting the chain.

---

### Finding Description

**Constant and per-block check:**

`MAX_NUMBER_OF_LOGS = 16_384` is defined as the Merkle tree leaf capacity. [1](#0-0) 

The per-block enforcement in `check_for_block_limits` uses a strict `>` comparison, meaning a block is allowed to contain **exactly** 16,384 logs: [2](#0-1) 

This check is called in the ZK tx loop after each transaction: [3](#0-2) 

**Multiblock batch accumulation — the missing check:**

`ZKBatchDataKeeper` holds a fixed-capacity `ArrayVec<Bytes32, 16384>` for the batch-level log Merkle tree: [4](#0-3) 

In the multiblock batch proving post-tx op, after each block is finalized, all of that block's logs are pushed into this shared accumulator: [5](#0-4) 

The `apply_to_array_vec` implementation calls `array_vec.push(log.hash())` unconditionally for every log in the block, with no capacity check: [6](#0-5) 

`ArrayVec::push` panics when the array is full. There is no `try_push` or capacity guard anywhere in this path.

**The gap:** The per-block check ensures each block has ≤ 16,384 logs. But a batch with two blocks — block A with 16,384 logs and block B with even 1 log — has 16,385 total logs. When block B's logs are pushed into the already-full `batch_data.logs_storage`, the prover panics.

The `l2_logs_root` function that consumes this `ArrayVec` also assumes it fits within 16,384 leaves (height-14 tree): [7](#0-6) 

---

### Impact Explanation

When the prover processes a multiblock batch whose total L2→L1 log count exceeds 16,384, `apply_to_array_vec` panics in the RISC-V proving environment. The batch becomes permanently unprovable. Because ZKsync OS requires every committed batch to be proven before the next one can be settled on L1, this halts the chain's settlement layer — a critical liveness failure.

---

### Likelihood Explanation

Two log-emitting paths are reachable by unprivileged callers:

1. **L1→L2 priority transactions** — each accepted priority transaction unconditionally emits exactly one L2→L1 log via `emit_l1_l2_tx_log`: [8](#0-7) 

   Priority transactions are submitted on L1 and **must** be processed by the sequencer in order. An attacker can queue 16,384 priority transactions to saturate block A's log capacity, then queue additional priority transactions that land in block B of the same batch.

2. **L1 Messenger system hook** — any L2 user can call the L1 Messenger to emit user messages, which also add entries to `logs_storage` via `push_message`. [9](#0-8) 

The sequencer has no code-enforced mechanism to seal a batch before the cross-block total exceeds 16,384. The only protection is off-chain operator policy, which is not guaranteed.

---

### Recommendation

Add a per-batch log capacity check before calling `apply_to_array_vec`. For example, in the multiblock batch post-tx op, before accumulating a block's logs, verify:

```rust
let remaining = batch_data.logs_storage.remaining_capacity();
let block_logs = io.logs_storage.len() as usize;
if block_logs > remaining {
    // seal the batch before this block, or return an error
}
```

Alternatively, change the per-block limit to `MAX_NUMBER_OF_LOGS / max_blocks_per_batch`, or use `try_push` in `apply_to_array_vec` and propagate the error gracefully instead of panicking.

---

### Proof of Concept

1. Attacker submits 16,384 L1→L2 priority transactions on L1. These are queued in the priority queue.
2. The sequencer processes them into block A. After the last one, `system.io.logs_len() == 16_384`, which satisfies `logs_used > MAX_NUMBER_OF_LOGS` as `false` (exactly equal, not greater), so block A is accepted with 16,384 logs.
3. The sequencer starts block B in the same batch and processes one more priority transaction (or any L1-messenger call). Block B now has 1 log.
4. The sequencer finalizes the batch and submits it for proving.
5. The prover calls `apply_to_array_vec` for block A (fills `batch_data.logs_storage` to capacity 16,384), then calls it again for block B. The first `push` on the full `ArrayVec` panics.
6. The batch is permanently unprovable; chain settlement halts.
