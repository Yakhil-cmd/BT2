### Title
ERC-20 Mirror Tokens Permanently Burned Without Accounting for Partial NEP-141 Refund in Omni Exit Path - (File: `engine-precompiles/src/native.rs`)

### Summary

When a user exits bridged ERC-20 tokens to NEAR using the Omni message path (`withdrawToNear` with a `:msg` suffix), the ERC-20 tokens are burned upfront in the EVM. The `ExitToNear` precompile then schedules a `ft_transfer_call` on the NEP-141 contract. Per the NEP-141 standard, if the receiver's `ft_on_transfer` returns a non-zero amount (partial or full refund), the NEP-141 contract refunds those tokens back to Aurora's account. However, since `ft_transfer_call` itself resolves as a **successful** NEAR promise regardless of the refund amount, the `exit_to_near_precompile_callback` never triggers ERC-20 re-minting. The result is a permanent accounting inconsistency: the user's ERC-20 tokens are burned for the full amount, but only a partial (or zero) amount of NEP-141 tokens was actually transferred to the receiver, with the remainder silently accumulating in Aurora's NEP-141 balance with no corresponding ERC-20 counterpart.

### Finding Description

**Step 1 – ERC-20 burn is unconditional.**

In `EvmErc20V2.withdrawToNear`, the ERC-20 burn happens before the precompile call and is not conditional on the downstream NEAR promise succeeding: [1](#0-0) 

**Step 2 – Omni path sets `transfer_near_args = None`.**

In `exit_erc20_token_to_near`, the Omni branch (`Message::Omni`) selects `ft_transfer_call` as the method and explicitly sets `transfer_near_args = None`: [2](#0-1) 

**Step 3 – Callback only fires on total promise failure.**

Back in the precompile `run()`, `callback_args` is built with `refund` populated (when `error_refund` feature is on) and `transfer_near = None`. The callback is attached. However, inside `exit_to_near_precompile_callback`, the refund branch is only entered when the promise result is **not** `Successful`: [3](#0-2) 

**Step 4 – `ft_transfer_call` always resolves as `Successful` even on partial refund.**

The NEP-141 `ft_transfer_call` method resolves its promise with `Successful(refund_amount_bytes)` regardless of how many tokens the receiver's `ft_on_transfer` returns. The callback at line 214 matches on `PromiseResult::Successful(_)` and ignores the returned bytes entirely — it never reads the refund amount. So when the receiver returns `N` tokens back to Aurora, the NEP-141 contract credits Aurora's balance with `N` tokens, but the callback takes the success branch and does nothing: [4](#0-3) 

**Step 5 – No re-minting path exists for partial refund.**

`refund_on_error` (which re-mints ERC-20 tokens) is only reachable from the `else if` branch, which requires the promise to have failed entirely: [5](#0-4) 

**Net result:** User burns `X` ERC-20 tokens. Receiver's `ft_on_transfer` returns `R` tokens (0 < R ≤ X). Aurora's NEP-141 balance increases by `R`. The ERC-20 supply is permanently reduced by `X`. The `R` NEP-141 tokens in Aurora's account have no ERC-20 counterpart — they are orphaned and unclaimable by the user.

### Impact Explanation

**Permanent freezing of funds.** The NEP-141 tokens refunded to Aurora's account cannot be claimed by anyone: there are no ERC-20 tokens to burn in exchange for them, and no administrative recovery path is exposed. The user who initiated the exit loses their ERC-20 tokens permanently. Aurora's NEP-141 balance silently inflates relative to the ERC-20 total supply, creating a growing accounting gap that can affect all future bridge users (the "extra" NEP-141 tokens in Aurora's account could be drained by subsequent legitimate deposits, effectively socializing the loss).

### Likelihood Explanation

**Medium.** The Omni exit path is the production path used by the Aurora Omni bridge for cross-chain token routing. Any NEAR contract acting as a receiver that implements `ft_on_transfer` and returns a non-zero amount (e.g., due to slippage limits, insufficient liquidity, or intentional rejection) triggers this bug. A malicious or buggy receiver contract can cause this for any user who routes through it. The user does not need to be malicious — an honest user routing through a DEX or bridge aggregator that partially fills an order is sufficient.

### Recommendation

In `exit_to_near_precompile_callback`, when the promise result is `Successful`, read the returned bytes and interpret them as the NEP-141 refund amount. If the refund amount is non-zero, re-mint the corresponding amount of ERC-20 tokens to the original sender's address (using the same `refund_on_error` / `setup_refund_on_error_input` path already used for full failures). The `refund_address` and `erc20_address` are already encoded in `callback_args.refund` when `error_refund` is enabled, so the information needed for re-minting is available.

Additionally, the `ExitToNearPrecompileCallbackArgs` should be extended to carry the original sender's EVM address so that partial re-mints can be credited to the correct account.

### Proof of Concept

1. Alice holds 1000 units of ERC-20 token `T` on Aurora (backed by NEP-141 `t.near`).
2. Alice calls `T.withdrawToNear(recipient="dex.near:{"min_out":"900"}", amount=1000)`.
3. `EvmErc20V2.withdrawToNear` calls `_burn(Alice, 1000)` — Alice's ERC-20 balance drops to 0, ERC-20 `totalSupply` drops by 1000.
4. The `ExitToNear` precompile schedules `ft_transfer_call` on `t.near` with `receiver_id="dex.near"`, `amount=1000`, `msg={"min_out":"900"}`.
5. `dex.near.ft_on_transfer` determines it can only fill 800 units and returns `200` (refund).
6. `t.near` refunds 200 tokens back to Aurora's account. `ft_transfer_call` resolves `Successful(200)`.
7. `exit_to_near_precompile_callback` sees `PromiseResult::Successful(_)`, ignores the `200`, and returns `None`.
8. **Result:** Alice has 0 ERC-20 tokens. `dex.near` received 800 NEP-141 tokens. Aurora's NEP-141 balance has 200 extra tokens with no ERC-20 counterpart. Alice's 200 units are permanently lost. [6](#0-5) [7](#0-6) [3](#0-2)
