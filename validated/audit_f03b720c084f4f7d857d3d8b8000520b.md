### Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Liquidity Operations — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

### Summary

The `MetricOmmPoolLiquidityAdder` periphery contract exposes `addLiquidity` and `removeLiquidity` entry points for LPs but contains no `deadline` parameter or timestamp check on any of its 29 liquidity-operation call sites. The swap router (`MetricOmmSimpleRouter`) correctly enforces `_checkDeadline()` on every swap path, but the liquidity adder — the symmetric user-facing contract for LP operations — has no equivalent guard. A pending liquidity transaction can be held in the mempool and executed arbitrarily late, after the oracle price has moved and the bin layout relative to the oracle has shifted, causing the LP to add or remove liquidity at a price that no longer reflects their intent.

### Finding Description

`MetricOmmSimpleRouter` calls `_checkDeadline(params.deadline)` at the top of every public swap function: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The helper that enforces this is: [5](#0-4) 

A `deadline` field is present in every swap parameter struct: [6](#0-5) 

By contrast, a grep for `deadline` across all periphery contracts returns **zero hits** in `MetricOmmPoolLiquidityAdder.sol` or `IMetricOmmPoolLiquidityAdder.sol`, while those same files contain 29 references to `addLiquidity`/`removeLiquidity`. No `deadline` field exists in the liquidity parameter structs and no timestamp check is performed before the pool call.

The pool's `addLiquidity` and `removeLiquidity` are low-level functions with no deadline enforcement of their own: [7](#0-6) [8](#0-7) 

Because Metric OMM is an oracle-based AMM, the effective price of every bin is computed relative to the live oracle bid/ask at swap time: [9](#0-8) 

The bin layout (`curBinDistFromProvidedPriceE6`, `curBinIdx`, `curPosInBin`) is stored as a distance from the oracle's mid-price. When the oracle price moves, the same bin indices represent a completely different absolute price range. An LP who submitted a liquidity transaction targeting specific bins at time T will have that transaction execute against a shifted bin layout at time T+Δ, with no on-chain protection.

### Impact Explanation

**Direct loss of LP principal.** When a delayed `addLiquidity` transaction executes after the oracle price has moved:

- The LP deposits tokens into bins that now sit at a different absolute price relative to the oracle mid-price than intended.
- If the oracle price moved in the direction that makes the LP's targeted bins "in-the-money," the LP immediately suffers impermanent loss because incoming swaps will drain the more valuable token from those bins at the stale bin-relative price.
- For `removeLiquidity`, the LP receives a token ratio that reflects the post-drift bin state rather than the state at submission time, potentially receiving significantly more of the depreciated token.

This is a direct loss of user principal, not a gas or dust issue. The magnitude scales with oracle price movement during the mempool delay and the LP's position size.

### Likelihood Explanation

Network congestion events (gas spikes, sequencer delays on L2s) routinely cause transactions to sit in the mempool for minutes to hours. Oracle prices for volatile pairs can move several percent in that window. The trigger requires no privileged access — any LP using the `MetricOmmPoolLiquidityAdder` is exposed on every liquidity operation. The swap router's existing deadline pattern confirms the protocol is aware of this risk class and intentionally mitigated it for swaps; the omission in the liquidity adder is therefore an oversight rather than a design choice.

### Recommendation

Add a `uint256 deadline` field to the `addLiquidity` and `removeLiquidity` parameter structs in `IMetricOmmPoolLiquidityAdder.sol` and call `_checkDeadline(params.deadline)` (or an equivalent inline check `if (block.timestamp > params.deadline) revert TransactionExpired(...)`) at the top of each public function in `MetricOmmPoolLiquidityAdder.sol`, mirroring the pattern already used in `MetricOmmSwapRouterBase._checkDeadline`: [5](#0-4) 

### Proof of Concept

1. LP calls `MetricOmmPoolLiquidityAdder.addLiquidity(...)` targeting bins `[0, 1]` when the oracle mid-price is `P`. The transaction is submitted with a gas price that causes it to queue in the mempool.
2. The oracle price moves from `P` to `P * 1.05` (5% increase) while the transaction is pending.
3. The pool's `curBinDistFromProvidedPriceE6` and `curBinIdx` are updated by an intervening swap that re-centers the bin cursor against the new oracle price.
4. The LP's stale transaction executes: `addLiquidity` deposits tokens into bins `[0, 1]`, which now represent a price range 5% lower than the current oracle mid-price — i.e., the LP has added liquidity entirely below the current market price.
5. The very next swap drains the LP's token0 at the stale bin price, and the LP has effectively sold token0 at a 5% discount relative to the oracle price at execution time, with no recourse.
