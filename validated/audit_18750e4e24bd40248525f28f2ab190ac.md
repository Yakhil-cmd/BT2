### Title
Silent Error Swallowing in `executeProcessCallback` Causes Scheduled Transactions to Be Permanently Dropped - (File: `engine/execution/computation/computer/computer.go`)

### Summary
When the system process-callback transaction fails with an output error, `executeProcessCallback` logs the error as a "critical_error" but returns `nil` — no error is propagated, no state cleanup is performed, and all pending scheduled callback transactions are silently discarded. This is the direct Flow analog of the swap-router pattern: a callback failure is merely logged while the on-chain state is left inconsistent and user funds (fees paid to schedule transactions) are permanently lost.

### Finding Description
In `executeProcessCallback` at lines 653–662 of `engine/execution/computation/computer/computer.go`, after executing the system process-callback transaction, the code checks whether the transaction produced an output error:

```go
if txn.Output().Err != nil {
    // if the process transaction fails we log the critical error but don't return an error
    // so that block execution continues and only the scheduled transactions halt
    callbackLogger.Error().
        Err(txn.Output().Err).
        Bool("critical_error", true).
        Uint64("height", request.ctx.BlockHeader.Height).
        Msg("system process transaction output error")

    return nil, txnIndex, nil   // ← error swallowed; no cleanup
}
```

The function returns `(nil callbackTxs, txnIndex, nil error)`. Consequences:

1. `blueprints.ExecuteCallbacksTransactions` is never called — no executor transactions are generated for any pending scheduled callback.
2. No error is returned to the caller (`executeBlock`), so block execution proceeds normally.
3. The `ScheduledTransactionsIndex.Failed` path is never reached; every affected scheduled transaction remains in `ScheduledTxStatusScheduled` (pending) indefinitely.
4. Fees that users paid at scheduling time are not refunded — the `feesReturned` field tracked by `ScheduledTransactionsIndex.Cancelled` is never populated.

The contrast with the correct error path is visible in `module/chunks/chunkVerifier.go` lines 484–485, where the chunk verifier treats the same output error as fatal and propagates it:

```go
if processOutput.Err != nil {
    return nil, nil, fmt.Errorf("process callback transaction failed: %w", processOutput.Err)
}
```

The block computer deliberately diverges from this, choosing liveness over correctness, but does so without any compensating state cleanup.

### Impact Explanation
Any user who paid FLOW fees to schedule a transaction via `FlowTransactionScheduler` loses both the execution of their scheduled transaction and the fees paid to reserve that execution slot. If the scheduled transaction was itself a token transfer, bridge operation, or DeFi interaction, the underlying assets remain locked in whatever intermediate state they were in before the callback was supposed to fire. The scheduler index is left with stale "pending" entries that can never transition to a terminal state through normal protocol operation, creating permanent accounting drift between on-chain scheduler state and actual execution history.

### Likelihood Explanation
The process-callback transaction (`blueprints.ProcessCallbacksTransaction`) calls into the `FlowTransactionScheduler` Cadence contract. Any Cadence-level panic, computation-limit exhaustion, or storage error inside `scheduler.process()` produces a non-nil `txn.Output().Err`. An unprivileged transaction sender can influence this path by scheduling transactions: if the aggregate execution effort of all pending callbacks in a single block exceeds the system transaction's gas budget, the process transaction fails and every callback in that block is dropped. Because the scheduler accepts callbacks from any account, this is reachable without privileged access.

### Recommendation
Replace the silent-drop branch with explicit state cleanup:

1. **Mark affected scheduled transactions as failed** by calling `ScheduledTransactionsIndex.Failed` for each pending callback ID that was present in the scheduler queue at the time of failure.
2. **Refund scheduling fees** to the original payers, mirroring the logic in `ScheduledTransactionsIndex.Cancelled`.
3. **Emit failure events** so off-chain indexers and users can detect the dropped callbacks.
4. Alternatively, if the design intent is to silently skip callbacks on process failure, document and enforce a maximum per-block callback count so the process transaction cannot exceed its computation budget.

### Proof of Concept
1. Schedule enough callbacks in a single block so that `scheduler.process()` exceeds its computation limit, causing `txn.Output().Err != nil` in `executeProcessCallback`.
2. Observe that `executeProcessCallback` returns `nil, txnIndex, nil` — no executor transactions are appended to the system collection.
3. Query `ScheduledTransactionsIndex` for the affected IDs: they remain in `ScheduledTxStatusScheduled` with no `ExecutedTransactionID` and no fee refund recorded.
4. Confirm that the fees deducted at scheduling time are not returned to the payers.

```go
func (idx *ScheduledTransactionsIndex) Cancelled(
	lctx lockctx.Proof,
	rw storage.ReaderBatchWriter,
	scheduledTxID uint64,
	feesReturned uint64,
	feesDeducted uint64,
	transactionID flow.Identifier,
) error {
	if !lctx.HoldsLock(storage.LockIndexScheduledTransactionsIndex) {
		return fmt.Errorf("missing required lock: %s", storage.LockIndexScheduledTransactionsIndex)
	}

	key := makeScheduledTxPrimaryKey(scheduledTxID)
	var tx access.ScheduledTransaction
	if err := operation.RetrieveByKey(rw.GlobalReader(), key, &tx); err != nil {
		return fmt.Errorf("could not retrieve scheduled transaction %d: %w", scheduledTxID, err)
	}
	if tx.Status != access.ScheduledTxStatusScheduled {
		return fmt.Errorf("tx %d already in terminal state %s: %w", scheduledTxID, tx.Status, storage.ErrInvalidStatusTransition)
	}

	tx.Status = access.ScheduledTxStatusCancelled
	tx.FeesReturned = feesReturned
	tx.FeesDeducted = feesDeducted
	tx.CancelledTransactionID = transactionID

	if err := operation.UpsertByKey(rw.Writer(), key, tx); err != nil {
		return fmt.Errorf("could not update scheduled transaction %d: %w", scheduledTxID, err)
	}
	return nil
```

```go
func (idx *ScheduledTransactionsIndex) Failed(
	lctx lockctx.Proof,
	rw storage.ReaderBatchWriter,
	scheduledTxID uint64,
	transactionID flow.Identifier,
) error {
	if !lctx.HoldsLock(storage.LockIndexScheduledTransactionsIndex) {
		return fmt.Errorf("missing required lock: %s", storage.LockIndexScheduledTransactionsIndex)
	}

	key := makeScheduledTxPrimaryKey(scheduledTxID)

	var tx access.ScheduledTransaction
	if err := operation.RetrieveByKey(rw.GlobalReader(), key, &tx); err != nil {
		return fmt.Errorf("could not retrieve scheduled transaction %d: %w", scheduledTxID, err)
	}
	if tx.Status != access.ScheduledTxStatusScheduled {
		return fmt.Errorf("tx %d already in terminal state %s: %w", scheduledTxID, tx.Status, storage.ErrInvalidStatusTransition)
	}

	tx.Status = access.ScheduledTxStatusFailed
	tx.ExecutedTransactionID = transactionID

	if err := operation.UpsertByKey(rw.Writer(), key, tx); err != nil {
		return fmt.Errorf("could not update scheduled transaction %d: %w", scheduledTxID, err)
	}
	return nil
```
