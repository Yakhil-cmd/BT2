### Title
`uint32` Overflow in `_afterTimelock` Allows Admin to Bypass Stop-Loss Timelock — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` computes `uint32(block.timestamp + timelock)`. When `timelock` is set to any value ≥ `2^32 − block.timestamp` (currently ~2.54 billion seconds, ~80.6 years), the addition overflows `uint32`, producing an `executeAfter` that is numerically less than `block.timestamp`. The `_requireElapsed` guard then passes immediately, letting the pool admin execute drawdown, decay, or watermark changes with zero delay — bypassing the timelock that LPs rely on to react.

---

### Finding Description

`_afterTimelock` is the single source of truth for every timelocked parameter change in the extension:

```solidity
// metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol:297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space, then **silently truncated** to `uint32`. At the current epoch (~1 753 000 000):

```
block.timestamp + type(uint32).max
= 1_753_000_000 + 4_294_967_295
= 6_047_967_295
uint32(6_047_967_295) = 6_047_967_295 − 4_294_967_296 = 1_752_999_999
```

The result (~`block.timestamp − 1`) is stored as `executeAfter`. The guard:

```solidity
// metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol:301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

compares `uint256 block.timestamp` against the `uint32` (implicitly widened to `uint256`). Because `block.timestamp > executeAfter`, the revert is never reached and execution continues immediately.

**No validation exists** on the `timelock` value — neither in `initialize` nor in `proposeOracleStopLossTimelock`:

```solidity
// initialize — no _validateTimelock call
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock accepted without bounds check
```

```solidity
// proposeOracleStopLossTimelock — no bounds check on newTimelock
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    ...
    sched.pendingTimelock = newTimelock;   // type(uint32).max accepted
    ...
}
```

---

### Impact Explanation

Once the overflow timelock is active, the admin can call `proposeOracleStopLoss{Drawdown,Decay,HighWatermarks}` and immediately call the corresponding `execute*` function in the same block, with no delay. Concrete outcomes:

| Admin action | Consequence |
|---|---|
| `drawdownE6 = 0` | `floorMultiplier = 1e6`; stop-loss triggers on any metric decrease → **all swaps permanently blocked** |
| `drawdownE6 = 1e6` | `floorMultiplier = 0`; stop-loss never triggers → **LP value-loss protection silently disabled** |
| `watermarks = 0` | Stop-loss fires on the very next swap → **pool frozen** |
| `decayPerSecondE8 = 1e8` | Watermarks decay to zero in one second → **protection collapses** |

The first and third outcomes constitute broken core pool functionality (swap path permanently reverts). The second constitutes loss of the LP protection invariant the extension was deployed to enforce. Both are within the allowed impact gate.

---

### Likelihood Explanation

The trigger is the pool admin — a semi-trusted role. The admin must:

1. Propose `newTimelock = type(uint32).max` (or any value ≥ ~2.54 billion).
2. Wait for the **current** timelock to elapse (the proposal itself is correctly gated by the existing timelock).
3. Execute the timelock change.
4. From that point, all subsequent parameter proposals are immediately executable.

Step 2 means the attack is not instant from a fresh pool, but it is fully reachable by any pool admin after one timelock period. Because `proposeOracleStopLossTimelock` accepts `type(uint32).max` without complaint and emits a plausible-looking event (`OracleStopLossTimelockProposed`), the change may go unnoticed by LPs monitoring events.

---

### Recommendation

1. **Add a maximum timelock cap** in both `initialize` and `proposeOracleStopLossTimelock`. A reasonable ceiling (e.g., 365 days = 31 536 000 seconds) is well within `uint32` range and prevents overflow:

```solidity
uint32 private constant MAX_TIMELOCK = 365 days;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

2. **Use `uint256` for `executeAfter`** throughout `PoolStopLossSchedule` and `PendingHighWatermarks` to eliminate the truncation entirely. The storage cost is one extra slot but removes the class of overflow bugs permanently.

---

### Proof of Concept

```solidity
// Pool initialized with a 7-day timelock (legitimate setup)
extension.initialize(pool, abi.encode(uint32(500_000), uint32(0), uint32(7 days)));

// Step 1: admin proposes overflow timelock
vm.prank(admin);
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);

// Step 2: wait for current 7-day timelock
vm.warp(block.timestamp + 7 days);

// Step 3: execute — now oracleStopLossConfig[pool].timelock = type(uint32).max
vm.prank(admin);
extension.executeOracleStopLossTimelock(pool);

// Step 4: propose drawdown = 0 (should require another 7-day wait)
vm.prank(admin);
extension.proposeOracleStopLossDrawdown(pool, 0);
// _afterTimelock returns uint32(block.timestamp + type(uint32).max) ≈ block.timestamp - 1

// Step 5: execute IMMEDIATELY — no warp needed
vm.prank(admin);
extension.executeOracleStopLossDrawdown(pool); // succeeds; drawdown = 0
