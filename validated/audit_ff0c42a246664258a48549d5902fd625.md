### Title
Missing Zero-Address Validation in `reward_address` Setter Functions Causes Temporary Freezing of Unclaimed Yield and Staked Funds - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary

`Staking.change_reward_address`, `Staking.stake`, `Pool.change_reward_address`, and `Pool.enter_delegation_pool` all accept a `reward_address: ContractAddress` parameter without checking whether it equals `Zero::zero()`. If a staker or pool member sets their `reward_address` to the zero address, every subsequent attempt to transfer rewards to that address will revert, blocking `claim_rewards` and `unstake_action` until the address is corrected. The protocol already defines `GenericError::ZERO_ADDRESS` but never applies it to these paths.

### Finding Description

**Root cause — `Staking.change_reward_address`:**

```cairo
// src/staking/staking.cairo  L517-540
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // ← no assert!(reward_address.is_non_zero(), ...)
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
```

The same omission exists in `Staking.stake` (line 307–311), `Pool.enter_delegation_pool` (line 195), and `Pool.change_reward_address` (line 506–510).

**Downstream impact — `send_rewards_to_staker`:**

```cairo
// src/staking/staking.cairo  L1620-1625
let reward_address = staker_info.reward_address;
let amount = staker_info.unclaimed_rewards_own;
claim_from_reward_supplier(:reward_supplier_dispatcher, :amount, :token_dispatcher);
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

If `reward_address` is zero, the OpenZeppelin ERC20 `transfer` call reverts ("ERC20: transfer to the zero address"), causing the entire transaction to revert. The same pattern appears in `Pool.claim_rewards` (pool.cairo line 366).

**Downstream impact — `unstake_action`:**

```cairo
// src/staking/staking.cairo  L494-495
let token_dispatcher = strk_token_dispatcher();
self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
```

`unstake_action` calls `send_rewards_to_staker` before returning the principal. If `reward_address` is zero, `unstake_action` reverts, preventing the staker from completing their exit and recovering their staked STRK.

**Existing error constant — never applied here:**

```cairo
// src/errors.cairo  L39
GenericError::ZERO_ADDRESS => "Address is zero",
```

The protocol already has the guard constant; it is simply not used in the reward-address setters.

### Impact Explanation

A staker or pool member who sets `reward_address` to `Zero::zero()` (accidentally, via a buggy script, or a UI error) will find that:

1. `Staking.claim_rewards` / `Pool.claim_rewards` reverts — unclaimed yield is temporarily frozen.
2. `Staking.unstake_action` reverts — the staker cannot complete their exit and recover their principal, temporarily freezing staked funds.

The freeze persists until the user calls `change_reward_address` again with a valid address. Because the fix is available, the freeze is temporary rather than permanent. This matches **"Temporary freezing of funds"** in the allowed High impact scope, and at minimum **"Griefing / damage to users"** at Medium.

### Likelihood Explanation

Any staker or pool member can trigger this by passing `Zero::zero()` to `change_reward_address` or by staking/entering the pool with a zero reward address. No privileged role is required. Accidental triggering is realistic (scripting errors, zero-initialised structs, front-end bugs). The entry path is fully unprivileged and reachable on mainnet.

### Recommendation

Add a zero-address guard in every function that writes `reward_address` to storage:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply this to:
- `Staking.stake` — before writing `reward_address` to `staker_info`
- `Staking.change_reward_address` — alongside the existing token-address check
- `Pool.enter_delegation_pool` — alongside the existing token-address check
- `Pool.change_reward_address` — alongside the existing token-address check

### Proof of Concept

1. Staker calls `Staking.stake(reward_address: Zero::zero(), operational_address: X, amount: MIN_STAKE)` — accepted without revert.
2. Staker accrues rewards over several epochs via attestation.
3. Staker calls `Staking.claim_rewards(staker_address)` → `send_rewards_to_staker` → `checked_transfer(recipient: 0, amount: rewards)` → **reverts**.
4. Staker calls `Staking.unstake_intent()` — succeeds (no reward transfer here).
5. After the exit window, staker calls `Staking.unstake_action(staker_address)` → `send_rewards_to_staker` → **reverts** — principal is locked.
6. Staker calls `Staking.change_reward_address(valid_address)` — succeeds.
7. Staker retries `unstake_action` — succeeds. Freeze is lifted.

The same sequence applies to a pool member via `Pool.enter_delegation_pool(reward_address: Zero::zero(), amount)` followed by `Pool.claim_rewards`.
