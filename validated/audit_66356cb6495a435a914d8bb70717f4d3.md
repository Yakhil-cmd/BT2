Audit Report

## Title
Unbounded `ingress_expiries` Loop Allows OOM DoS via Attacker-Controlled `ingress_end` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary

The `construction_payloads` function in ICRC1 Rosetta accepts caller-supplied `ingress_start` and `ingress_end` values and enters an unbounded `while` loop that allocates one `EnvelopeContent::Call` and one `SigningPayload` per iteration. No guard bounds the range `ingress_end - ingress_start`, so an unauthenticated attacker can supply `ingress_end = u64::MAX` to trigger ~139 million iterations, exhausting process memory and crashing the Rosetta server. The identical structural defect exists in the ICP Rosetta variant.

## Finding Description

In `construction_payloads` (`rs/rosetta-api/icrc1/src/construction_api/services.rs`, L111–194), `ingress_interval` is computed as `(MAX_INGRESS_TTL − PERMITTED_DRIFT).as_nanos()` = (300s − 60s) = 240 × 10⁹ ns. [1](#0-0) 

The two existing guards only check ordering and minimum end time; neither bounds the *range* `ingress_end − ingress_start`: [2](#0-1) 

The loop that follows has no iteration cap: [3](#0-2) 

The per-iteration step is `ingress_interval − INGRESS_INTERVAL_OVERLAP` = 240s − 120s = 120s = 1.2 × 10¹¹ ns. [4](#0-3) 

With `ingress_start ≈ now` (≈ 1.7 × 10¹⁸ ns) and `ingress_end = u64::MAX` (≈ 1.84 × 10¹⁹ ns), the range is ≈ 1.67 × 10¹⁹ ns, yielding ≈ 139,000,000 iterations. Each iteration clones `canister_method_args`, a method name string, sender, nonce, and ingress expiry into `envelope_contents`, and pushes a `SigningPayload` into `signing_payloads`: [5](#0-4) 

The HTTP handler is fully public with no authentication or rate-limiting: [6](#0-5) 

The ICP Rosetta variant has the identical unbounded loop with the same step size (120s), also lacking any range guard: [7](#0-6) [8](#0-7) 

## Impact Explanation

A single unauthenticated HTTP POST causes the Rosetta server process to be killed by the OS OOM killer (or panic on allocation failure), dropping all in-flight requests. This matches the allowed ICP bounty impact: **Application/platform-level DoS, crash** against a Rosetta/financial-integration component with concrete user and protocol harm — **High ($2,000–$10,000)**. Any operator or user relying on the Rosetta API for transaction construction, submission, or monitoring is denied service until the process is manually restarted. The attack is repeatable with a single `curl` command.

## Likelihood Explanation

The endpoint is unauthenticated, network-reachable, and requires only a valid JSON body with a large `ingress_end` field. No privileged access, key material, or protocol-level position is needed. The exploit is trivially reproducible and can be repeated indefinitely to prevent recovery.

## Recommendation

Add an explicit range cap immediately after the existing guards in both `rs/rosetta-api/icrc1/src/construction_api/services.rs` and `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`:

```rust
const MAX_INGRESS_EXPIRIES: usize = 100; // ~200 minutes of coverage

let step = ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
if step == 0 || (ingress_end - ingress_start) / step > MAX_INGRESS_EXPIRIES as u64 {
    return Err(Error::processing_construction_failed("ingress window too large"));
}
```

Alternatively, break out of the loop after `MAX_INGRESS_EXPIRIES` entries are collected. Apply the same fix to the ICP Rosetta variant.

## Proof of Concept

```bash
NOW=$(python3 -c "import time; print(int(time.time()*1e9))")
INGRESS_END=18446744073709551615  # u64::MAX

curl -X POST http://rosetta-server:8080/construction/payloads \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\":\"Internet Computer\",\"network\":\"<ledger_id>\"},
    \"operations\": [<valid_transfer_op>],
    \"public_keys\": [<valid_pubkey>],
    \"metadata\": {
      \"ingress_start\": $NOW,
      \"ingress_end\": $INGRESS_END
    }
  }"
# Server enters ~139M-iteration loop, allocates tens of GB, crashes via OOM.
```

A deterministic unit test can confirm the issue by calling `services::construction_payloads` directly with `ingress_end = u64::MAX` and asserting that the returned `ingress_expiries` vector length is bounded, or that the function returns an error before the loop runs away.
