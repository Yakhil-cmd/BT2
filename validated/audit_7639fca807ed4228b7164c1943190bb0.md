### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Permanently Freeze Staker Yield - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no such caller check. Any unprivileged address can call it with `disable_rewards: true` for every block, consuming the per-block reward slot without distributing rewards, permanently denying consensus-phase yield to all stakers.

### Finding Description
The spec explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation in `StakingRewardsManagerImpl::update_rewards` performs no caller identity check whatsoever: [2](#0-1) 

The function only checks that the current block number is greater than `last_reward_block`, then unconditionally writes the current block number to `last_reward_block` before branching on `disable_rewards`: [3](#0-2) 

Because `last_reward_block` is a single global slot shared across all stakers, any caller who front-runs the sequencer with `disable_rewards: true` consumes the slot for that block. The legitimate sequencer's subsequent call reverts with `REWARDS_ALREADY_UPDATED`. [4](#0-3) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` on every Starknet block prevents the sequencer from ever distributing consensus-phase block rewards. Because `last_reward_block` is updated to the current block before the early return, the sequencer's call for the same block always reverts. Stakers accumulate zero `unclaimed_rewards_own` indefinitely. The attack is protocol-wide: a single attacker targeting any one active staker blocks reward distribution for every block.

### Likelihood Explanation
**High.** The function is publicly callable with no preconditions beyond the contract being unpaused and a valid staker address existing. Starknet transactions are cheap. An attacker needs only to submit one transaction per block with `disable_rewards: true` and any active staker address. No capital, no privileged role, and no special knowledge is required. The attack is trivially automatable.

### Recommendation
Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the configured sequencer address (or a dedicated role), mirroring the pattern already used for `update_rewards_from_attestation_contract`: [5](#0-4) 

Concretely, store the authorized sequencer address during initialization and assert it at the start of `update_rewards`, before the `last_reward_block` write.

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker monitors the mempool / block production.
3. At each new block, attacker calls:
   ```
   staking.update_rewards(staker_address: <any_active_staker>, disable_rewards: true)
   ```
4. `last_reward_block` is set to the current block number; the function returns early with no rewards distributed.
5. The legitimate sequencer calls `update_rewards` for the same block → reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeat every block. All stakers receive zero consensus rewards indefinitely.

The root cause is the absence of `assert!(get_caller_address() == self.sequencer_address.read(), ...)` before the `last_reward_block.write(current_block_number)` at line 1485. [6](#0-5)
