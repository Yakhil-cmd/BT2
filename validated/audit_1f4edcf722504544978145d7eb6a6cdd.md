### Title
Zero-Amount BTC Transfer in `transfer_to_pools_when_unstake` Can Block `unstake_action` — (File: src/staking/staking.cairo)

---

### Summary

The `transfer_to_pools_when_unstake` internal function in `staking.cairo` unconditionally transfers the delegated pool balance back to each pool contract during `unstake_action`, without guarding against a zero amount. When a staker holds a BTC pool with zero delegated balance (no delegators ever joined, or all exited), and the BTC token reverts on zero-value transfers, `unstake_action` permanently reverts, freezing the staker's STRK principal in the contract.

---

### Finding Description

`unstake_action` calls `transfer_to_pools_when_unstake` to return each pool's delegated stake: [1](#0-0) 

The loop iterates over every pool registered to the staker. For each pool it reads the current delegated balance and transfers it unconditionally:

```cairo
let pool_balance = self.get_delegated_balance(:staker_address, :pool_contract);
// ... no zero-check ...
token_dispatcher
    .checked_transfer(
        recipient: pool_contract,
        amount: pool_balance.to_native_amount(:decimals).into(),
    );
```

`pool_balance` is legitimately zero in two realistic situations:

1. **No delegators ever joined the BTC pool** — the staker enabled a BTC pool at stake time but no one delegated.
2. **All BTC delegators exited before `unstake_action`** — each `remove_from_delegation_pool_intent` call decrements the stored delegated balance; after all exits the balance reaches zero.

The same unconditional-transfer pattern appears in `send_rewards_to_staker` (STRK rewards path) and `pool.claim_rewards` (STRK rewards path): [2](#0-1) [3](#0-2) 

For those two sites the token is hardcoded to `STRK_TOKEN_ADDRESS`, and the OpenZeppelin STRK ERC-20 does not revert on zero-value transfers, so they are benign in practice. The BTC pool path in `transfer_to_pools_when_unstake` uses a configurable `token_address` per pool: [4](#0-3) 

If the deployed BTC token reverts on zero-value transfers (a documented behaviour of certain ERC-20 implementations), the entire `unstake_action` transaction reverts.

The `unstake_action` entry point is callable by any address: [5](#0-4) 

There is no fallback path to complete the withdrawal once this revert occurs.

        fn transfer_to_pools_when_unstake(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<InternalStakerPoolInfoV2>,
        ) {
            for (pool_contract, token_address) in staker_pool_info.pools {
                let pool_balance = self.get_delegated_balance(:staker_address, :pool_contract);
                let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
                let pool_dispatcher = IPoolDispatcher { contract_address: pool_contract };
                pool_dispatcher.set_staker_removed();
                self
                    .insert_staker_delegated_balance(
                        :staker_address, :pool_contract, delegated_balance: Zero::zero(),
                    );
                let decimals = self.get_token_decimals(:token_address);
                token_dispatcher
                    .checked_transfer(
                        recipient: pool_contract,
                        amount: pool_balance.to_native_amount(:decimals).into(),
                    );
