### Title
Gateway Skips `__validate__` Signature Verification for Invoke Transactions with Nonce=1 When a Deploy-Account Is Pending — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally bypasses the account's `__validate__` entry point (which performs signature verification) for any invoke transaction whose nonce is exactly `1` when the on-chain account nonce is `0` and the mempool reports a transaction for that address. An unprivileged attacker who observes a pending `deploy_account` transaction for any address can submit an invoke transaction with an **invalid or arbitrary signature** for that address and have it admitted to the mempool without any signature check.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip validation) when all three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }` and passes it to `StatefulValidator::validate`. Inside `validate_tx`, the very first check is:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [2](#0-1) 

This means the `__validate__` entry point — the sole place where the account contract verifies the transaction signature — is **never called**. The transaction is admitted to the mempool with no cryptographic check on its signature.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate hashes and nonce ordering; it does not verify signatures. [3](#0-2) 

The full gateway admission path is:

```
add_tx_inner
  → stateless_tx_validator.validate()   // size/format checks only
  → convert_rpc_tx_to_internal_and_executable_txs()
  → extract_state_nonce_and_run_validations()
      → run_pre_validation_checks()
          → validate_state_preconditions()  // nonce range, gas price
          → validate_by_mempool()           // duplicate/nonce-too-old only
          → skip_stateful_validations()     // ← returns true; __validate__ skipped
      → run_validate_entry_point(skip_validate=true)  // no-op
  → mempool_client.add_tx()             // invalid tx admitted
``` [4](#0-3) 

**Attack steps:**

1. Attacker observes that address `A` has a pending `deploy_account` transaction in the public mempool (or was recently committed, making `account_tx_in_pool_or_recent_block` return `true`).
2. Attacker crafts an `InvokeV3` transaction for address `A` with `nonce = 1` and an **invalid or zeroed signature**.
3. Attacker submits this transaction to the gateway.
4. The gateway reads `account_nonce = 0` from state, sees `tx_nonce = 1`, and finds the address in the mempool → `skip_stateful_validations` returns `true`.
5. The invalid transaction is admitted to the mempool without any signature verification.

The `max_nonce_for_validation_skip` config field exists in `StatefulTransactionValidatorConfig` but is **not used** by the gateway's `skip_stateful_validations`; the gateway hardcodes the check to `Nonce(Felt::ONE)`. [5](#0-4) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Invalid-signature transactions are admitted to the mempool. When the batcher later attempts to execute them, `__validate__` fails and the transactions are rejected (nonce not incremented, no fee charged, not included in the block). However:

- **DoS / resource exhaustion**: An attacker can flood the mempool with invalid transactions for every address that has a pending `deploy_account`, consuming mempool slots and batcher CPU.
- **Griefing via fee escalation**: If the attacker submits the invalid invoke with a higher tip than the legitimate user's invoke at the same `(address, nonce=1)`, the mempool's fee-escalation logic may replace the legitimate transaction with the attacker's invalid one. The legitimate user's transaction is evicted from the mempool and must be resubmitted.
- **Broken admission invariant**: The gateway's contract is that every transaction in the mempool has passed signature verification. This invariant is broken for the specific nonce=1 / account_nonce=0 window.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The mempool is publicly observable; any node can see pending `deploy_account` transactions.
- The attack requires no privileged access, no special contract, and no on-chain funds.
- The window is narrow (only nonce=1 while account_nonce=0), but the deploy_account + invoke UX pattern is the standard onboarding flow, so this window is hit by every new account.

---

### Recommendation

The skip should not bypass signature verification entirely. Two options:

1. **Preferred**: Keep the UX skip but still run `__validate__` against a simulated state where the deploy_account has already been applied (i.e., the account contract exists). This preserves the UX benefit while enforcing signature correctness.

2. **Conservative**: Remove the gateway-level skip entirely. The Python-layer `PyValidator::should_run_stateful_validations` already implements the same skip with the `deploy_account_tx_hash` parameter supplied by the caller; rely on that path instead, or require the client to supply the deploy_account tx hash as proof before skipping. [6](#0-5) 

At minimum, add a guard that the skip is only applied when the `deploy_account_tx_hash` is explicitly provided and verified, rather than inferred from the observable mempool state.

---

### Proof of Concept

```
// Precondition: address A has a pending deploy_account in the mempool.
// account_nonce(A) == 0 on-chain.

// 1. Craft an InvokeV3 for address A, nonce=1, signature=[0x0, 0x0] (invalid).
let invalid_invoke = RpcInvokeTransactionV3 {
    sender_address: A,
    nonce: Nonce(Felt::ONE),
    signature: TransactionSignature(vec![Felt::ZERO, Felt::ZERO].into()),
    // ... valid resource bounds, calldata, etc.
};

// 2. Submit to gateway.
gateway.add_tx(RpcTransaction::Invoke(RpcInvokeTransaction::V3(invalid_invoke)), None).await;

// 3. Gateway path:
//    skip_stateful_validations: tx_nonce==1, account_nonce==0,
//    account_tx_in_pool_or_recent_block(A)==true  →  returns true
//    run_validate_entry_point(skip_validate=true)  →  __validate__ NOT called
//    mempool_client.add_tx(...)  →  invalid tx admitted

// 4. Batcher later tries to execute the invalid invoke:
//    __validate__ is called → fails (invalid signature)
//    Transaction rejected, not included in block.
//    But legitimate user's invoke at nonce=1 may have been evicted via fee escalation.
```

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
