### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary
The `calculate_rewards` function in the Pool contract contains an explicitly acknowledged unbounded loop over a delegator's `pool_member_epoch_balance` trace. A delegator who repeatedly modifies their balance across many epochs without claiming rewards will grow this trace without bound. Once the trace is large enough, every call to `claim_rewards` (and `pool_member_info_v1`) will exceed the Starknet gas limit, permanently freezing the delegator's unclaimed yield with no recovery path.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over every entry in `pool_member_epoch_balance` that falls between the delegator's last checkpoint and the current epoch:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each call to `add_to_delegation_pool`, `enter_delegation_pool`, or `exit_delegation_pool_intent` invokes `set_member_balance`, which appends a new entry to the trace keyed at `current_epoch + K`:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

The `entry_to_claim_from` cursor is only advanced when `claim_rewards` succeeds. If a delegator makes one balance change per epoch and never claims rewards, the trace grows by one entry per epoch. After enough epochs, the loop in `calculate_rewards` will exceed the block gas limit on every invocation, making `claim_rewards` permanently uncallable.

`pool_member_info_v1` (a view function) also calls `calculate_rewards` with the same loop: [3](#0-2) 

There is no batched or paginated reward-claiming mechanism. Once the trace is too large, there is no on-chain recovery path.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A delegator whose `pool_member_epoch_balance` trace grows beyond the gas-limit threshold can never successfully call `claim_rewards`. Their accrued STRK rewards are locked in the pool contract forever. The pool contract holds no admin function to forcibly flush or paginate the trace. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**Medium.** The attack requires the delegator to make balance changes across many epochs without claiming rewards. This is a realistic pattern for long-term delegators who actively manage their position (e.g., dollar-cost averaging into the pool, or repeatedly partially exiting and re-entering). The protocol imposes no upper bound on how many epochs a delegator may go without claiming, and no cap on trace length. The developers themselves acknowledge the loop is unbounded in the source comment. [4](#0-3) 

---

### Recommendation

1. **Paginate reward claiming**: Add a `max_entries` parameter to `claim_rewards` so the loop processes at most N trace entries per call, updating `entry_to_claim_from` and accumulating partial rewards across multiple transactions.
2. **Bound trace growth**: Enforce a maximum number of unclaimed balance-change entries per pool member (e.g., require `claim_rewards` before a new balance change if the trace exceeds a threshold).
3. **Merge same-epoch entries**: Ensure that multiple balance changes within the same epoch always overwrite rather than append, limiting trace growth to at most one entry per epoch.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` in epoch E₀ — trace length = 1.
2. Delegator calls `add_to_delegation_pool` in epoch E₁ — trace length = 2.
3. Delegator calls `exit_delegation_pool_intent` (partial) in epoch E₂ — trace length = 3.
4. Repeat steps 2–3 across N epochs without ever calling `claim_rewards`.
5. After N epochs, `pool_member_epoch_balance` trace length = N+1.
6. Delegator (or reward address) calls `claim_rewards`. The loop at line 859 must iterate all N+1 entries in a single transaction.
7. For sufficiently large N (bounded only by Starknet's block gas limit), the transaction reverts with an out-of-gas error.
8. Every subsequent `claim_rewards` call also reverts — the delegator's accumulated yield is permanently frozen.

The entry path is entirely unprivileged: `add_to_delegation_pool` and `exit_delegation_pool_intent` are callable by any pool member. [5](#0-4) [6](#0-5) [7](#0-6)
