### Title
Silent Address Truncation in ERC-20 Bridge `receive_erc20_tokens` Mints Tokens to Unintended Recipient - (`File: engine/src/engine.rs`)

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` accepts any `ft_on_transfer` message whose hex-encoded address portion is **longer than** 40 characters and silently uses only the first 40 characters as the recipient address. The correct check is `!= 40`, not `< 40`. A user who supplies a malformed-length address string (e.g. a 21-byte hex address, or any trailing data appended to the address) will have their bridged ERC-20 tokens minted to a silently-derived, unintended address, permanently freezing those funds.

### Finding Description

In `engine/src/engine.rs`, `receive_erc20_tokens` parses the `msg` field of an `ft_on_transfer` callback to determine the EVM recipient:

```rust
let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
// Recipient - 40 characters (Address in hex without '0x' prefix)
if message.len() < 40 {
    return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
}
let mut address_bytes = [0; 20];
hex::decode_to_slice(&message[..40], &mut address_bytes)
    .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
Address::from_array(address_bytes)
``` [1](#0-0) 

The guard `message.len() < 40` only rejects messages that are **too short**. Any message longer than 40 hex characters passes the check, and `&message[..40]` silently truncates the input to the first 40 characters. The remaining characters are discarded without error.

By contrast, the sibling function `receive_base_tokens` delegates to `FtTransferMessageData::try_from`, which enforces an **exact** length of 40 characters for the new message format:

```rust
if message.len() == 40 { ... }
``` [2](#0-1) 

This inconsistency means the ERC-20 bridge path silently accepts and truncates oversized address strings, while the ETH bridge path correctly rejects them.

The `ft_on_transfer` NEAR entry point is publicly callable by any NEP-141 token contract: [3](#0-2) 

### Impact Explanation

When a user bridges a NEP-141 ERC-20 token to Aurora and supplies a `msg` field whose hex address portion is longer than 40 characters (e.g. `0x<42 hex chars>` — a 21-byte address), the engine:

1. Strips the `0x` prefix → 42-char string.
2. Passes the `< 40` guard (42 ≥ 40).
3. Decodes only `message[..40]` → a 20-byte address that is **not** the address the user intended.
4. Mints the full bridged ERC-20 token amount to that silently-derived address.

If the derived address is not controlled by the user, the minted tokens are permanently inaccessible. This constitutes **permanent freezing of funds** (Critical impact).

### Likelihood Explanation

Likelihood is **low**. The user must supply a malformed message — either a hex address that is too long (e.g. a 21-byte value with `0x` prefix, analogous to the ENS PoC), or any trailing data appended to a valid 40-char address. This is a user-side mistake, but the protocol should reject it with an error rather than silently truncating and minting to an unintended address. The ENS finding was rated Medium for the same class of mistake.

### Recommendation

Change the length guard in `receive_erc20_tokens` from a one-sided `< 40` check to an exact `!= 40` check:

```rust
// Before (vulnerable):
if message.len() < 40 {
    return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
}

// After (fixed):
if message.len() != 40 {
    return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
}
```

This makes the ERC-20 bridge path consistent with `FtTransferMessageData::try_from` used in `receive_base_tokens`, and ensures any oversized address string is rejected rather than silently truncated. [4](#0-3) 

### Proof of Concept

1. Alice holds 1000 units of a NEP-141 token (`token.near`) that has a corresponding ERC-20 on Aurora.
2. Alice calls `ft_transfer_call` on `token.near` with:
   - `receiver_id`: Aurora engine account
   - `amount`: `1000`
   - `msg`: `"0x9fD6E51AaD88f6f4CE6aB8827279CFFFb92266332265"` — a 21-byte (42 hex char) value with `0x` prefix
3. `token.near` calls `ft_on_transfer` on Aurora with the same `msg`.
4. Inside `receive_erc20_tokens`:
   - `strip_prefix("0x")` → `"9fD6E51AaD88f6f4CE6aB8827279CFFFb92266332265"` (42 chars)
   - `42 < 40` is **false** → guard passes
   - `message[..40]` → `"9fD6E51AaD88f6f4CE6aB8827279CFFFb9226633"` (first 40 chars)
   - Decoded address: `0x9fD6E51AaD88f6f4CE6aB8827279CFFFb9226633`
5. 1000 ERC-20 tokens are minted to `0x9fD6E51AaD88f6f4CE6aB8827279CFFFb9226633` — an address Alice does not control.
6. Alice's tokens are permanently frozen. [5](#0-4)
