Audit Report

## Title
Unsanitized Memo Bytes Enable Markdown Injection into ICRC-21 `GenericDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

The `add_memo` function in `ConsentMessage` converts ICRC-1 memo bytes to a UTF-8 string and interpolates it verbatim into a markdown-formatted `GenericDisplayMessage` string, wrapping it in a single backtick pair. A malicious dApp can supply a memo whose valid UTF-8 content begins with a backtick, collapsing the opening delimiter into an empty inline-code span and rendering everything that follows as raw markdown — including attacker-crafted bold field labels (`**Amount:**`, `**To:**`) that are visually indistinguishable from the ledger-generated fields above them. This directly undermines the security guarantee of ICRC-21, which exists precisely to protect users from malicious dApps.

## Finding Description

In `packages/icrc-ledger-types/src/icrc21/responses.rs`, `add_memo` at line 284 handles `GenericMemo::Icrc1Memo` by first attempting UTF-8 decoding (falling back to hex only on invalid UTF-8), then interpolating the result verbatim:

```rust
// responses.rs L288-294
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
// ...
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

The template `` `{memo_str}` `` wraps `memo_str` in a single backtick pair. Under CommonMark, if `memo_str` begins with a backtick, the two adjacent backticks form an **empty inline-code span** (`` `` ``), and all subsequent content is rendered as raw markdown outside any code fence. No sanitization or escaping of backticks (or other markdown-special characters) is performed anywhere in this path.

The entry point is fully unprivileged. In `packages/icrc-ledger-types/src/icrc21/lib.rs` at line 392, the memo from the caller-supplied `TransferArg` is passed directly to `with_memo` without any sanitization:

```rust
// lib.rs L392-394
if let Some(memo) = memo {
    display_message_builder =
        display_message_builder.with_memo(GenericMemo::Icrc1Memo(memo.0));
}
```

The `FieldsDisplayMessage` variant is not affected because the memo is stored as a structured `Value::Text` field at lines 296–299, not interpolated into a markdown string.

No existing guard checks memo content for markdown-special characters. The only existing check (`std::str::from_utf8`) is a prerequisite for the attack, not a defense against it — valid UTF-8 is required for the injection to work, and the attacker controls exactly that.

## Impact Explanation

ICRC-21 is the mechanism by which wallets display a ledger-generated, trustworthy consent message to users before they approve a transaction — specifically to defend against malicious dApps that control their own UI. By injecting fake bold-field markdown into the `GenericDisplayMessage`, a malicious dApp can cause the wallet to render attacker-controlled `**Amount:**` and `**To:**` lines that are visually identical to the ledger-generated fields. A user who approves based on the injected (lower) fields may authorize a transfer to an attacker-controlled account for an amount they did not intend. This constitutes a **significant ledger security impact with concrete user harm** — specifically, potential theft of user funds via deception of the consent mechanism — matching the High ($2,000–$10,000) impact tier: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."*

## Likelihood Explanation

- The entry point is fully unprivileged: any caller can invoke `icrc21_canister_call_consent_message` with an arbitrary `arg` blob containing a crafted `TransferArg`.
- The `Memo` type (`packages/icrc-ledger-types/src/icrc1/transfer.rs` line 32) is an unbounded `ByteBuf`; the 32-byte figure in the ICRC-1 spec is a minimum ledgers *should* support, not a hard cap. Even under a strict 32-byte limit, a payload such as `` `\n\n**To:** `attacker` `` (~20 bytes) is sufficient to inject a fake recipient field.
- The attack requires a malicious dApp — which is precisely the threat model ICRC-21 is designed to defend against.
- Any wallet that renders `GenericDisplayMessage` as markdown (the intended rendering per ICRC-21) is directly affected.
- The attack is repeatable, low-cost, and requires no special privileges or on-chain state.

## Recommendation

Escape all backtick characters in `memo_str` before interpolation in `packages/icrc-ledger-types/src/icrc21/responses.rs` at line 294:

```rust
let escaped = memo_str.replace('`', "\\`");
message.push_str(&format!("\n\n**Memo:**\n`{escaped}`"));
```

For more robust protection against multi-line content, use a fenced code block and escape any triple-backtick sequences inside the memo:

```rust
let escaped = memo_str.replace("```", "\\`\\`\\`");
message.push_str(&format!("\n\n**Memo:**\n```\n{escaped}\n```"));
```

The `FieldsDisplayMessage` path requires no change.

## Proof of Concept

**Minimal reproducible test (unit/integration):**

1. Construct a `TransferArg` with `memo` set to the UTF-8 bytes of `` `\n\n**To:** `attacker` `` (≤ 32 bytes).
2. Encode as Candid and call `icrc21_canister_call_consent_message` with `method = "icrc1_transfer"` and `device_spec = GenericDisplay`.
3. Decode the returned `ConsentInfo` and extract the `GenericDisplayMessage` string.
4. Assert that the string contains a line beginning with `**To:**` that was not generated by the ledger (i.e., the injected fake field appears after the `**Memo:**` section).
5. Render the string with any CommonMark-compliant renderer and observe that the injected `**To:**` line is visually indistinguishable from the ledger-generated `**To:**` line.

This test can be written against the existing `StateMachine`-based test harness in `rs/ledger_suite/tests/sm-tests/src/lib.rs` (see `test_icrc21_transfer_message` at line 3648 for the pattern), substituting the memo bytes with the crafted payload.
