### Title
Gateway validates L2 gas price against stale `l2_gas_price` instead of `next_l2_gas_price`, causing invalid transactions to be admitted or valid transactions to be rejected - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validator reads `block_N.l2_gas_price` from the previous block header for both the soft threshold check and the blockifier pre-validation context. However, the batcher builds block N+1 using `block_N.next_l2_gas_price` — the EIP-1559-adjusted price already computed and stored in the same block header. This mismatch means the gateway validates transactions against a stale price, admitting transactions that will fail execution when the price rises, and rejecting transactions that would succeed when the price falls.

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` from the latest finalized block header. It populates `strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price.price_in_fri` — the price that was used for the *current* block's transactions — and silently drops `block_header.next_l2_gas_price`, which is the EIP-1559-adjusted price that will govern the *next* block. [1](#0-0) 

This stale `BlockInfo` is then used in two places inside `StatefulTransactionValidator`:

1. **Soft threshold check** — `validate_resource_bounds` reads `strk_gas_prices.l2_gas_price` from this stale `BlockInfo` and compares it against the transaction's `max_price_per_unit`. The developer-authored TODO comment at line 229 explicitly acknowledges the bug: [2](#0-1) 

2. **Blockifier pre-validation context** — `run_validate_entry_point` builds a `BlockContext` from the same stale `BlockInfo` (incrementing only the block number, not the gas price) and runs `perform_pre_validation_stage` / `check_fee_bounds` against it: [3](#0-2) 

Meanwhile, the orchestrator's `try_sync` correctly reads `next_l2_gas_price` from the same block header and stores it as the price for the next block: [4](#0-3) 

`build_proposal` then forwards this value as `l2_gas_price_fri` in the `ProposalInit`, which becomes the actual gas price the batcher uses when executing transactions: [5](#0-4) 

The two prices diverge every block because `calculate_next_base_gas_price` adjusts the price based on gas consumption: [6](#0-5) 

### Impact Explanation

**When the L2 gas price is rising** (`block_N.next_l2_gas_price > block_N.l2_gas_price`):

- A transaction with `max_price_per_unit = block_N.l2_gas_price` passes both the gateway soft check and the gateway blockifier pre-validation (which uses the stale price).
- The transaction is admitted to the mempool.
- When the batcher executes it in block N+1 at `block_N.next_l2_gas_price`, `check_fee_bounds` fires `ResourceBoundsError::MaxGasPriceTooLow` and the transaction is dropped.
- **Result**: invalid transactions are admitted to the mempool and waste sequencer execution resources.

**When the L2 gas price is falling** (`block_N.next_l2_gas_price < block_N.l2_gas_price`):

- A transaction with `max_price_per_unit = block_N.next_l2_gas_price` fails the gateway soft check (threshold is computed from the higher stale price) and is rejected with `GAS_PRICE_TOO_LOW`.
- The same transaction would pass `check_fee_bounds` in the batcher at the lower actual price.
- **Result**: valid transactions are rejected at the gateway even though they would succeed in execution.

This matches the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The L2 gas price changes every block under normal EIP-1559 operation. Any block with gas consumption above or below the target causes `next_l2_gas_price ≠ l2_gas_price`. This is the common case, not an edge case. The discrepancy is bounded by `gas_price_max_change_denominator` per block, but it is always present and grows larger during sustained high or low load. No special attacker capability is required — any user submitting a transaction during a price-change period is affected.

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, replace `block_header.l2_gas_price.price_in_fri` with `block_header.next_l2_gas_price` when populating `strk_gas_prices.l2_gas_price` in the returned `BlockInfo`. This is exactly what the TODO comment at line 229 of `stateful_transaction_validator.rs` requests. The `next_l2_gas_price` field is already present in `BlockHeaderWithoutHash` and is populated by the state-sync layer, so no additional RPC calls are needed.

### Proof of Concept

1. Observe block N with `l2_gas_price = P` and `next_l2_gas_price = P' > P` (gas usage above target).
2. Submit an invoke V3 transaction with `AllResourceBounds { l2_gas: { max_price_per_unit: P, ... }, ... }`.
3. Gateway reads block N's `l2_gas_price = P` as the threshold; the transaction satisfies `max_price >= 100% * P` and passes `validate_tx_l2_gas_price_within_threshold`.
4. Gateway builds a `BlockContext` with `l2_gas_price = P` (stale); `check_fee_bounds` passes since `P >= P`.
5. Transaction is admitted to the mempool.
6. Batcher builds block N+1 with `l2_gas_price = P'`; `check_fee_bounds` fires `MaxGasPriceTooLow { max_gas_price: P, actual_gas_price: P' }`.
7. Transaction is dropped from the block; the mempool slot was wasted and the sequencer spent CPU on a doomed transaction.

Conversely, with `P' < P` (gas usage below target), a transaction with `max_price = P'` is rejected at step 3 with `GAS_PRICE_TOO_LOW` even though it would have executed successfully in block N+1.

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
