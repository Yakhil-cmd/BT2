### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear()` and `withdrawToEthereum()` Causes Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear()` and `withdrawToEthereum()` functions that irreversibly burn the caller's ERC-20 tokens **before** invoking the exit precompile via inline assembly. The return value of the low-level `call()` is captured in a local variable `res` but is never inspected. If the precompile call fails for any reason, the function returns successfully without reverting, leaving the user with permanently destroyed tokens and no corresponding NEAR or Ethereum tokens received.

---

### Finding Description

In `EvmErc20.sol` lines 53–63 and `EvmErc20V2.sol` lines 53–64, `withdrawToNear()` follows this sequence:

```solidity
// EvmErc20.sol / EvmErc20V2.sol
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // <-- tokens destroyed here, irreversibly

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — if call returns 0, execution continues silently
    }
}
``` [1](#0-0) 

The identical unchecked pattern appears in `withdrawToEthereum()` (lines 65–76 of both contracts) and in `EvmErc20V2.sol`: [2](#0-1) 

The exit precompile (`ExitToNear`) can return a failure (`ExitError`) in several reachable conditions. When it does, the EVM `call()` opcode returns `0` into `res`. Because `res` is never tested, the Solidity function returns normally. The `_burn()` that already executed is not rolled back — it is a permanent state change.

The `ExitToNear` precompile fails (returns `Err`) in the following reachable cases:

1. **ERC-20 not registered**: `get_nep141_from_erc20` returns an error if the ERC-20 address has no NEP-141 mapping. [3](#0-2) 
2. **Non-zero `apparent_value` for ERC-20 exit**: The precompile explicitly rejects this. [4](#0-3) 
3. **Invalid recipient account ID**: Parsing `parse_recipient` can fail on malformed input. [5](#0-4) 
4. **Engine paused**: The precompile checks engine state and can reject the call.

In all these cases, `_burn()` has already executed, the ERC-20 supply is reduced, and the user's balance is zero — but no NEP-141 `ft_transfer` promise is ever scheduled on NEAR.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user calling `withdrawToNear()` or `withdrawToEthereum()` under any of the failure conditions above will have their ERC-20 tokens permanently destroyed with no corresponding NEAR or Ethereum tokens received. The tokens are gone from the ERC-20 supply and cannot be recovered because there is no refund path: the function does not revert, so the burn is final. This is a direct permanent fund loss for the caller.

---

### Likelihood Explanation

The vulnerability is reachable by any unprivileged ERC-20 token holder. The most realistic trigger is passing a recipient NEAR account ID that is syntactically valid enough to pass Solidity's `bytes` parameter handling but is rejected by the precompile's `parse_recipient` or `get_nep141_from_erc20` logic. A user who makes a typo in the recipient account ID, or who calls `withdrawToNear` on an ERC-20 that was deployed but whose NEP-141 mapping was not yet registered, will silently lose all tokens passed to `amount`. No special privilege is required.

---

### Recommendation

Check the return value of the precompile `call()` and revert on failure. The burn should either be moved **after** a successful precompile call, or the assembly block must revert on `res == 0`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, restructure the function to call the precompile first (in a read-only simulation or with a revert-on-failure guard) and only burn tokens after confirming the precompile accepted the call.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) for a NEP-141 token.
2. Mint tokens to address `A`.
3. From address `A`, call `withdrawToNear(invalidRecipient, amount)` where `invalidRecipient` is a byte string that causes `get_nep141_from_erc20` or `parse_recipient` to fail inside the precompile.
4. Observe: the transaction succeeds (no revert), `balanceOf(A)` is now `0`, and no NEP-141 `ft_transfer` promise was scheduled on NEAR.
5. The tokens are permanently destroyed.

The root cause is structurally identical to the reported `safeTransferMany()` bug: a function whose name and interface imply a complete, successful asset transfer silently skips the critical success check — in the original report, the ERC721Receiver check; here, the precompile call result — leaving user funds permanently frozen.
