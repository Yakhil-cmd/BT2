### Title
ICA Precompile `submitMsgs` Declared `payable` But Silently Discards Sent ETH/CRO — (`x/cronos/keeper/precompiles/ica.go`)

### Summary

The `IICAModule` Solidity interface declares both `submitMsgs` and `registerAccount` as `payable`, meaning the EVM will accept and credit any ETH/CRO value sent with those calls to the precompile address. However, the precompile's `Run` implementation for `SubmitMsgsMethodName` never reads `contract.Value()` and never forwards it anywhere. Any native CRO sent by a caller is permanently locked at the precompile address with no recovery path.

### Finding Description

The Solidity interface binding declares:

```solidity
// x/cronos/events/bindings/src/ICA.sol
function submitMsgs(string calldata connectionID, bytes calldata data, uint256 timeout)
    external payable returns (uint64);
function registerAccount(string calldata connectionID, string calldata version, int32 ordering)
    external payable returns (bool);
``` [1](#0-0) 

The auto-generated Go binding confirms `submitMsgs` is a paid mutator:

```
// Solidity: function submitMsgs(string connectionID, bytes data, uint256 timeout) payable returns(uint64)
``` [2](#0-1) 

The `Run` implementation for `SubmitMsgsMethodName` unpacks only `connectionID`, `data`, and `timeout`, then calls `msgServer.SendTx`. There is no read of `contract.Value()` and no forwarding of the attached ETH value:

```go
case SubmitMsgsMethodName:
    args, err := method.Inputs.Unpack(contract.Input[4:])
    ...
    connectionID := args[0].(string)
    data := args[1].([]byte)
    timeout := args[2].(*big.Int)
    // contract.Value() is never read or forwarded
    execErr = stateDB.ExecuteNativeAction(precompileAddr, converter, func(ctx sdk.Context) error {
        msgServer := icacontrollerkeeper.NewMsgServerImpl(&ic.controllerKeeper)
        response, err := msgServer.SendTx(ctx, &icacontrollertypes.MsgSendTx{...})
        ...
    })
``` [3](#0-2) 

In the standard EVM execution model (including Ethermint), when a call with non-zero value reaches a precompile, the EVM transfers the value from the caller's balance to the precompile address **before** invoking `Run`. Because the precompile never reads or re-routes that value, it accumulates at `icaContractAddress` with no withdrawal mechanism.

This is the direct Cronos analog of the Llama M-01 bug class: a function is declared payable and the outer layer accepts value, but the value is never forwarded through the execution chain.

### Impact Explanation

Any unprivileged EVM user who calls `submitMsgs` (or `registerAccount`) with a non-zero `msg.value` permanently loses that CRO. The funds are credited to the precompile address and are irrecoverable because:
- The precompile has no `withdraw` or sweep function.
- The precompile address is not a module account with a governance-controlled sweep path.
- The ICA `SendTx` call on the host chain proceeds without the value, so the user's intent (funding the ICA operation) is also silently unmet.

This constitutes a **High** impact: permanent loss of CRO for honest users attempting to use the ICA precompile with attached value, and corruption of the precompile's native-token accounting state.

### Likelihood Explanation

Any unprivileged EVM user or smart contract can call the ICA precompile with value. The `payable` declaration in the public interface actively invites such calls. A contract that wraps ICA operations and forwards `msg.value` to `submitMsgs` would silently drain its callers.

### Recommendation

Either:
1. **Remove `payable`** from `submitMsgs` and `registerAccount` in `ICA.sol` and add a guard in `Run` that rejects calls with non-zero value (`if contract.Value().Sign() > 0 { return nil, errors.New("value not accepted") }`), or
2. **Forward the value** to the ICA owner's account or the ICA module account within `ExecuteNativeAction` if the intent is to allow funding the ICA account alongside the packet submission.

### Proof of Concept

1. Deploy any EVM contract on Cronos that calls the ICA precompile:
   ```solidity
   IICAModule ica = IICAModule(0x...icaContractAddress...);
   ica.submitMsgs{value: 1 ether}(connID, data, timeout);
   ```
2. Observe: the call succeeds (returns a sequence number), the caller's CRO balance decreases by 1 CRO, and the ICA precompile address balance increases by 1 CRO permanently.
3. No function exists on the precompile to recover the locked CRO. [4](#0-3) [5](#0-4)
