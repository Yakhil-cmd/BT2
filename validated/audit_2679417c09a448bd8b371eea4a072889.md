### Title
`set_eth_connector_contract_account` Replaces Connector Without Checking Old Connector Balance — (`engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` unconditionally overwrites the stored ETH connector account ID with no check on whether the old connector still holds user funds. After the switch, every connector-routed operation (`withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw`) is dispatched to the **new** connector. Any ETH (bridged as NEP-141 tokens) remaining in the old connector becomes permanently inaccessible through the Aurora Engine interface.

---

### Finding Description

`set_eth_connector_contract_account` is callable by the contract owner or via a private self-call: [1](#0-0) 

It delegates immediately to `set_connector_account_id`, which is a raw storage write with no precondition: [2](#0-1) 

Every connector-facing user operation is dispatched through `return_promise`, which reads the **current** connector account from storage at call time: [3](#0-2) 

For example, `withdraw` calls `return_promise` with method `"engine_withdraw"`, so after the account is replaced, all withdrawals are directed to the new connector: [4](#0-3) 

The same routing applies to `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, and `storage_withdraw` — all of which call `return_promise` targeting `get_connector_account_id`. [5](#0-4) 

There is no call to query the old connector's `ft_total_supply` or any per-account balance before the pointer is overwritten.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

If the old connector contract still holds any user ETH (represented as NEP-141 tokens), those tokens become permanently inaccessible through the Aurora Engine. Users cannot call `withdraw` to bridge back to Ethereum, cannot `ft_transfer` their balance, and cannot query their balance via the engine — all of these operations are now silently routed to the new connector, which has no record of the old balances. The old connector contract itself requires the Aurora Engine account to authorize `engine_withdraw`, so users cannot bypass the engine to recover funds directly either.

---

### Likelihood Explanation

A legitimate upgrade scenario — deploying a new connector contract to fix a bug or add features, then calling `set_eth_connector_contract_account` — is the natural trigger. The owner does not need to be malicious or compromised; they simply need to forget (or not know) that they must drain the old connector to zero before switching. This is exactly the class of mistake the Union Finance report describes: an administrative removal/replacement operation that lacks a safety guard on outstanding balances.

---

### Recommendation

Before overwriting the connector account ID, query the old connector's `ft_total_supply` (or equivalent balance view). Require it to be zero before allowing the switch:

```rust
// pseudocode
let old_account = get_connector_account_id(&io)?;
let supply = query_ft_total_supply(&old_account); // cross-contract view
require!(supply == 0, ERR_CONNECTOR_STILL_HOLDS_FUNDS);
set_connector_account_id(io, &args.account);
```

Alternatively, provide an explicit fund-migration step that moves all balances from the old connector to the new one atomically before the pointer is updated.

---

### Proof of Concept

1. Users bridge ETH to Aurora. The ETH connector contract (`old_connector.near`) now holds `N` wei of NEP-141 ETH on behalf of users.
2. The Aurora Engine owner deploys `new_connector.near` (e.g., an upgraded version) and calls:
   ```
   set_eth_connector_contract_account({ account: "new_connector.near", withdraw_serialize_type: Borsh })
   ``` [6](#0-5) 
3. `set_connector_account_id` writes `"new_connector.near"` to storage, overwriting `"old_connector.near"` with no balance check.
4. A user calls `withdraw` on the Aurora Engine. `return_promise` reads `get_connector_account_id` → `"new_connector.near"` and dispatches `engine_withdraw` there. [7](#0-6) 
5. `new_connector.near` has no record of the user's balance; the call fails or returns zero.
6. The user's funds remain locked in `old_connector.near` indefinitely. There is no engine-level path to recover them.

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
