### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged depositor to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and gates on `owner` (the position owner) instead. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` lets any caller freely specify an arbitrary `owner`, an unprivileged depositor can name an allowlisted address as `owner` and pass the guard unconditionally.

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` is declared as:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
```

The first `address` parameter — `sender`, the actual depositor/payer — is unnamed and **ignored**. The guard reads:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [1](#0-0) 

The mapping is named `allowedDepositor` and the admin setter is `setAllowedToDeposit(pool, depositor, allowed)`, making the design intent unambiguous: the guard is supposed to check **who is depositing**, not who owns the position. [2](#0-1) 

The pool correctly forwards both `sender` and `owner` to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` with no restriction beyond non-zero, and records `msg.sender` as the payer:

```solidity
return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
``` [4](#0-3) 

**Attack path:**
1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only address B.
2. Attacker A (not allowlisted) calls `addLiquidityExactShares(pool, owner=B, salt, deltas, max0, max1, "")`.
3. The pool calls `beforeAddLiquidity(sender=adder, owner=B, ...)`.
4. The extension evaluates `allowedDepositor[pool][B]` → `true` → no revert.
5. A's tokens are pulled; LP shares are minted into B's position.

The allowlist is fully bypassed. A also gains the ability to grief B by forcing an unwanted position into B's account.

### Impact Explanation

The `DepositAllowlistExtension` is the sole on-chain mechanism for restricting who may deposit into a pool. Bypassing it means:
- Any address can add liquidity to a pool configured as restricted, violating the core access-control invariant.
- LP shares are minted to an allowlisted owner without their consent, which can be used to grief or manipulate their position.
- The pool admin's configuration is rendered meaningless.

This is a broken core pool functionality / admin-boundary break with direct fund-flow consequences (unauthorized LP share minting).

### Likelihood Explanation

The bypass requires only a public call to `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with `owner` set to any known allowlisted address. No privileged access, no special tokens, and no complex setup is needed. Any observer of on-chain allowlist events can trivially identify a valid `owner` to supply.

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

Note: when the periphery adder is used, `sender` will be the `MetricOmmPoolLiquidityAdder` contract address. If per-user granularity is required through the adder, the adder must also forward the original `msg.sender` in `extensionData`, or the allowlist must be keyed on the adder contract itself. The fix above is correct for direct pool calls and for any adder that is itself allowlisted.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Foundry integration test sketch
// depositorA is NOT allowlisted; ownerB IS allowlisted.
// After the call, LP shares appear in ownerB's position — allowlist bypassed.

function test_allowlistBypass_ownerSpoofing() public {
    address depositorA = makeAddr("depositorA");
    address ownerB     = makeAddr("ownerB");

    // Only ownerB is allowlisted as depositor
    depositExtension.setAllowedToDeposit(address(pool), ownerB, true);

    // Fund and approve depositorA
    _setupUser(depositorA, /* caller */ depositorA, address(pool));

    LiquidityDelta memory d = _deltaAbovePrice(4, 10_000);

    // depositorA calls with owner=ownerB — should revert but does NOT
    vm.prank(depositorA);
    helper.addLiquidityExactShares(
        address(pool), ownerB, 99, d, type(uint256).max, type(uint256).max, ""
    );

    // LP shares minted into ownerB's position — bypass confirmed
    uint256 shares = stateView.positionBinShares(address(pool), ownerB, 99, int8(4));
    assertGt(shares, 0, "allowlist bypassed: shares minted for non-allowlisted depositor");
}
```
