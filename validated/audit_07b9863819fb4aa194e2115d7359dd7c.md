### Title
`pause_contract` Blocks All User Withdrawals, Temporarily Freezing ETH Funds - (File: engine/src/contract_methods/connector.rs)

### Summary
When the Aurora Engine contract is paused via `pause_contract`, the `withdraw` function (NEAR-side ETH bridge-out) and the `submit`/`submit_with_args` functions (EVM-level exits via `ExitToNear`/`ExitToEthereum` precompiles) are all blocked by `require_running`. This means users holding ETH on Aurora have no path to withdraw their funds during a pause, constituting a temporary fund freeze.

### Finding Description
The `withdraw` function in `engine/src/contract_methods/connector.rs` is the NEAR-callable entry point for users to bridge their ETH from Aurora back to Ethereum. It begins with:

```rust
pub fn withdraw<I: IO + Copy + PromiseHandler, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    require_running(&state::get_state(&io)?)?;  // <-- blocks when paused
    ...
}
``` [1](#0-0) 

`require_running` returns `ERR_PAUSED` whenever `state.is_paused == true`: [2](#0-1) 

The `pause_contract` admin function sets `state.is_paused = true`: [3](#0-2) 

The same `require_running` guard is present in `submit` and `submit_with_args`, which are the only entry points for EVM transactions (including calls to the `ExitToNear` and `ExitToEthereum` precompiles): [4](#0-3) 

The `ExitToNear` and `ExitToEthereum` precompiles are the EVM-level withdrawal mechanisms. When `submit` is blocked, these precompiles become unreachable: [5](#0-4) 

The `PrecompileFlags` only covers `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` — the two withdrawal precompiles — confirming these are the sole EVM-level exit paths: [6](#0-5) 

The result is that when `pause_contract` is called, **every withdrawal path** for user funds is simultaneously blocked:
- NEAR-side: `withdraw` → `require_running` → `ERR_PAUSED`
- EVM-side: `submit` → `require_running` → `ERR_PAUSED` → `ExitToNear`/`ExitToEthereum` unreachable

### Impact Explanation
**High — Temporary freezing of funds.** All user-accessible withdrawal mechanisms are blocked simultaneously when the contract is paused. Users holding ETH on Aurora cannot bridge assets back to Ethereum or NEAR during the pause window, regardless of urgency (e.g., to repay loans or respond to other on-chain commitments). The pause is owner-controlled and has no time limit, so the freeze duration is unbounded.

### Likelihood Explanation
**Medium.** The `pause_contract` function is an intentional admin feature callable by the contract owner at any time. Pausing is expected to be transient (e.g., during a security incident), but the design does not distinguish between blocking new deposits/interactions and blocking user withdrawals. The owner may legitimately pause the contract without intending to freeze user funds.

### Recommendation
Remove the `require_running` check from the `withdraw` function in `engine/src/contract_methods/connector.rs`. Users should always be able to withdraw their own funds, even when the contract is paused. Similarly, consider allowing `ExitToNear` and `ExitToEthereum` precompile calls to proceed through `submit` even when `state.is_paused == true`, or introduce a separate, narrower pause scope that does not affect withdrawal paths.

### Proof of Concept
1. A user deposits ETH into Aurora; their balance is recorded on-chain.
2. The contract owner calls `pause_contract` (e.g., in response to a suspected exploit).
3. The user calls `withdraw` on the Aurora Engine contract to bridge ETH back to Ethereum.
4. `connector::withdraw` calls `require_running`, which returns `ERR_PAUSED` because `state.is_paused == true`.
5. The user's withdrawal reverts. The user also cannot use `submit` to call `ExitToNear` or `ExitToEthereum` because `submit` also calls `require_running`.
6. The user's ETH is frozen for the entire duration of the pause, with no available withdrawal path. [7](#0-6) [8](#0-7)

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}

pub struct Precompiles<'a, I, E, H> {
    pub all_precompiles: BTreeMap<Address, AllPrecompiles<'a, I, E, H>>,
    pub paused_precompiles: BTreeSet<Address>,
}

impl<I, E, H> Precompiles<'_, I, E, H> {
    fn is_paused(&self, address: &Address) -> bool {
        self.paused_precompiles.contains(address)
    }
}

impl<I: IO + Copy, E: Env, H: ReadOnlyPromiseHandler> executor::stack::PrecompileSet
    for Precompiles<'_, I, E, H>
{
    fn execute(
        &self,
        handle: &mut impl PrecompileHandle,
    ) -> Option<Result<executor::stack::PrecompileOutput, PrecompileFailure>> {
        let address = Address::new(handle.code_address());

        if self.is_paused(&address) {
            return Some(Err(PrecompileFailure::Fatal {
                exit_status: ExitFatal::Other(prelude::Cow::Borrowed("ERR_PAUSED")),
            }));
        }

        let result = match self.all_precompiles.get(&address)? {
            AllPrecompiles::ExitToNear(p) => process_precompile(p, handle),
            AllPrecompiles::ExitToEthereum(p) => process_precompile(p, handle),
            AllPrecompiles::PredecessorAccount(p) => process_precompile(p, handle),
            AllPrecompiles::PrepaidGas(p) => process_precompile(p, handle),
            AllPrecompiles::PromiseResult(p) => process_precompile(p, handle),
            AllPrecompiles::CrossContractCall(p) => process_handle_based_precompile(p, handle),
            AllPrecompiles::Generic(p) => process_precompile(p.as_ref(), handle),
        };

        Some(result.and_then(|output| post_process(output, handle)))
    }
