### Title
`py_deploy_account` Accepts Caller-Supplied `sender_address` as `contract_address` Without Cryptographic Verification — (`File: crates/native_blockifier/src/py_deploy_account.rs`)

### Summary

In `crates/native_blockifier/src/py_deploy_account.rs`, the `py_deploy_account` function constructs a `DeployAccountTransaction` by reading `contract_address` directly from the Python object's `sender_address` attribute without verifying it equals the cryptographically derived address (`calculate_contract_address(class_hash, salt, constructor_calldata, deployer=0)`). Every downstream blockifier operation — contract deployment target, nonce slot, fee-balance slot, and `sender_address` in `TxInfo` — is keyed off this unverified field. An attacker who can influence the Python-layer transaction object can bind the execution to an arbitrary address, causing the contract to be deployed at the wrong storage slot, nonce to be consumed from the wrong account, and fees to be charged from the wrong balance.

### Impact Explanation

If the Python sequencer passes a `sender_address` that differs from the cryptographic derivation:

1. **Wrong deployment target**: The contract class is written to the attacker-chosen storage slot, not the address that the transaction hash commits to. State is permanently corrupted.
2. **Wrong nonce consumed**: The nonce at the attacker-chosen address is incremented, not the address that signed the transaction. This can replay or block transactions for the victim address.
3. **Wrong fee balance drained**: `verify_can_pay_committed_bounds` and the fee-transfer syscall debit the attacker-chosen address, enabling fee theft or denial-of-service against an arbitrary account.
4. **Wrong `sender_address` in `TxInfo`**: The `__validate_deploy__` entry point receives the spoofed address as `get_execution_info().contract_address`, breaking any contract-level self-checks.

This matches **Critical — Wrong state, receipt, storage value, or revert result from blockifier/syscall/execution logic for accepted input** and **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload**.

### Likelihood Explanation

The `native_blockifier` Python bindings are the execution path for the Python-based sequencer (section 13 of the project wiki). The Python layer reads `sender_address` from the user-submitted transaction object and passes it directly to `py_deploy_account` without independently recomputing `calculate_contract_address`. Any user who can submit a deploy-account transaction to the Python sequencer can supply a mismatched `sender_address`. No privileged access is required; the only precondition is the ability to call the sequencer's transaction-submission API.

### Recommendation

Replace the blind attribute read with a cryptographic derivation inside `py_deploy_account`, mirroring the `TransactionConverter` path:

```rust
// Instead of:
let contract_address =
    ContractAddress::try_from(py_attr::<PyFelt>(py_tx, "sender_address")?.0)?;

// Use:
let contract_address = tx.calculate_contract_address()
    .map_err(NativeBlockifierInputError::StarknetApiError)?;
```

Optionally assert equality with the Python-supplied value to surface mismatches early:

```rust
let py_supplied = ContractAddress::try_from(py_attr::<PyFelt>(py_tx, "sender_address")?.0)?;
assert_eq!(contract_address, py_supplied, "sender_address mismatch");
```

This is exactly the pattern used by `ExecutableDeployAccountTransaction::create` and `TransactionConverter::convert_rpc_tx_to_internal`.

### Proof of Concept

1. Construct a valid `DeployAccountTransactionV3` with fields `(class_hash=C, salt=S, constructor_calldata=D)`. The correct contract address is `A = calculate_contract_address(S, C, D, deployer=0)`.
2. Set `sender_address = V` where `V ≠ A` and `V` is a victim account with a known balance and nonce.
3. Submit the transaction to the Python sequencer. The Python layer calls `py_deploy_account(py_tx)`, which sets `contract_address = V`.
4. The blockifier executes with `storage_address = V`: the constructor runs at `V`'s storage slot (overwriting the victim's contract class), the nonce at `V` is incremented, and fees are deducted from `V`'s balance.
5. The contract class `C` is never deployed at `A`; the transaction hash (also taken verbatim from Python) does not commit to `V`, so the OS/proof layer may accept a hash computed over `A` while the state mutation targets `V`. [8](#0-7)
