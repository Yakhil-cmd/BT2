### Title
Setting a New ETH Connector Contract Account Breaks Existing ETH Withdrawals - (File: `engine/src/contract_methods/connector.rs`)

### Summary
The `set_eth_connector_contract_account` function allows the owner to replace the stored ETH connector account at any time with no migration guard. After the swap, every subsequent `withdraw` call routes to the new connector, which holds no ETH balance. All users who deposited ETH through the old connector lose their withdrawal path.

### Finding Description
`set_eth_connector_contract_account` (lines 418–437 of `engine/src/contract_methods/connector.rs`) unconditionally overwrites the connector account ID stored under `EthConnectorStorageId::EthConnectorAccount`:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I, env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        ...
        set_connector_account_id(io, &args.account);          // ← overwrites immediately
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}

After the connector account is replaced:

1. **Withdrawals break**: `withdraw` dispatches `engine_withdraw` to the new connector, which holds no NEP-141 ETH balance. The call fails or mints/burns against an empty ledger. The user's EVM balance is not debited, but the withdrawal never completes.
2. **In-flight deposits misroute**: Any `ft_on_transfer` callback arriving from the old connector is no longer recognized as the base-token connector, so it is treated as an ERC-20 transfer and fails or credits the wrong token.

The setter is callable by the owner or via a private self-call with no guard on outstanding balances:

```rust
let is_private = env.assert_private_call();
if is_private.is_err() {
    require_owner_only(&state, &env.predecessor_account_id())?;
}
``` [5](#0-4) 

The public entrypoint is exposed unconditionally:

### Impact Explanation
All ETH deposited through the old connector becomes unwithdrawable via the engine's normal `withdraw` path. The old connector still holds the NEP-141 ETH supply, but the engine now dispatches `engine_withdraw` to the new connector. Because `engine_withdraw` on the new connector is the only authorized burn-and-bridge path (it requires the engine as caller), users cannot bypass the engine to reclaim funds from the old connector directly. This constitutes **High – Temporary freezing of funds** (if the owner reverts the setter) escalating to **Critical – Permanent freezing of funds** if the old connector is decommissioned or its keys are deleted.

### Likelihood Explanation
The owner may legitimately upgrade the connector contract (e.g., to patch a bug or add features) without realizing that the swap atomically breaks all existing withdrawal paths. No on-chain check prevents the swap while ETH balances are outstanding. The action is a single owner transaction, requiring no special conditions beyond normal protocol operation.

### Recommendation
- Before allowing the connector account to be changed, assert that the old connector's total NEP-141 ETH supply is zero (i.e., all ETH has been withdrawn).
- Alternatively, make the setter one-time-only and introduce a separate flag to enable/disable new deposits, analogous to the suggestion in the original report.
- At minimum, emit a prominent warning and enforce a time-lock so users can withdraw before the new connector takes effect.

### Proof of Concept
1. Alice deposits 10 ETH to Aurora; the old connector (`old.near`) mints 10 NEP-141 ETH and credits Alice's EVM balance.
2. Owner calls `set_eth_connector_contract_account` with `new.near` (a freshly deployed connector with zero ETH balance).
3. Alice calls `withdraw` on the engine specifying her Ethereum address.
4. The engine reads `new.near` from storage and dispatches `engine_withdraw` to `new.near`.
5. `new.near` has no ETH balance; the call fails (or succeeds but burns nothing meaningful).
6. Alice's EVM balance is not debited; her 10 ETH is permanently locked — she cannot withdraw via the engine (routes to `new.near`) and cannot call `engine_withdraw` on `old.near` directly (requires the engine as caller).

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
