### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees to Front-Run Swappers — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` / `addFeeSellE6` to the pool with **no cap validation**. The `uint16` type permits values up to 65 535 E6 (~6.55 %), additive on top of the global spread fee (hard-capped at 20 %). A malicious or compromised pool admin can front-run a pending swap by instantly raising the active bin's additional fee to `type(uint16).max`, forcing the user to pay ~6.55 % more than expected, then resetting it — with no timelock and no on-chain guard.

### Finding Description

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` caps before updating global fees: [1](#0-0) 

`setPoolBinAdditionalFees` has **no analogous cap**: [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — only a bin-index range check: [3](#0-2) 

During swap execution the per-bin fee is added directly to the base spread fee: [4](#0-3) 

The same additive pattern appears in `getSellAndBuyPrices` and the data-provider lens: [5](#0-4) 

**Attack path:**
1. Pool admin observes a pending user swap targeting bin `k`.
2. Admin front-runs with `setPoolBinAdditionalFees(pool, k, 65535, 65535)` — `type(uint16).max` = 65 535 E6 ≈ 6.55 %.
3. User's swap executes; effective fee = `baseFeeX64 + 6.55 %` instead of the advertised `baseFeeX64`.
4. Admin resets the bin fee to 0 in the same or next block.

No timelock, no delay, no factory-owner co-signature required.

### Impact Explanation

Users pay up to **6.55 % more** per swap than the pool's published fee schedule. The excess accrues as LP fees. If the pool admin is also an LP (a common setup for bootstrapped pools), they directly extract the surplus. Even without LP participation, users suffer a direct loss of principal — the swap settles at a materially worse effective price than any pre-swap quote or slippage check would have shown, because the fee change happens between quote and execution.

The global spread fee is hard-capped at 20 % (`HARD_MAX_SPREAD_FEE_E6`): [6](#0-5) 

Per-bin additional fees have no corresponding ceiling, breaking the intended admin-boundary invariant.

### Likelihood Explanation

Low. Requires a malicious or compromised `poolAdmin[pool]`. However, the attack requires only a single permissioned call with no timelock, making it trivially executable once the admin is compromised.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` consistent with the existing global fee caps, e.g.:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinFeeE6` storage variable (settable by the factory owner, bounded by `HARD_MAX_SPREAD_FEE_E6`) and enforce it here.

### Proof of Concept

```
Pool deployed: adminSpreadFeeE6 = 1_000 (0.1%), maxAdminSpreadFeeE6 = 200_000 (20%)

1. User broadcasts: swap(pool, token1→token0, amountIn=1000 USDC, priceLimitX64=...)
2. Admin sees mempool tx, front-runs:
       factory.setPoolBinAdditionalFees(pool, curBinIdx, 65535, 65535)
   → bin.addFeeBuyE6 = 65535  (~6.5535 %)
3. User tx executes:
       effectiveFee = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
                    ≈ 0.1% + 6.55% = 6.65% total
   User receives ~6.45% fewer token0 than quoted.
4. Admin resets:
       factory.setPoolBinAdditionalFees(pool, curBinIdx, 0, 0)
```
