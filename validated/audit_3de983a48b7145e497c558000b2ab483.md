### Title
Constructor `replace_class` Bypasses `__validate_deploy__` Signature Verification in DeployAccount — (`crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

For `DeployAccount` transactions, the blockifier runs the constructor **before** `__validate_deploy__`. The constructor is permitted to call `replace_class` (not blocked in validate mode), which mutates the contract's class hash in shared state. When `validate_tx` subsequently resolves the class hash via `state.get_class_hash_at(storage_address)` and dispatches `__validate_deploy__` with `class_hash: None`, it executes against the **replaced** class — not the originally declared one. An attacker can pre-declare a permissive class whose `__validate_deploy__` returns `VALIDATED` unconditionally, then deploy an account whose constructor swaps to that class, bypassing all signature verification.

---

### Finding Description

**Step 1 — Constructor runs before `__validate_deploy__`.**

In `run_non_revertible`, the `DeployAccount` branch explicitly runs `run_execute` (constructor) first, then `validate_tx`: [1](#0-0) 

**Step 2 — `replace_class` is not blocked in validate mode.**

The constructor for a `DeployAccount` runs inside an `EntryPointExecutionContext::new_validate` context. The `replace_class` syscall handler in `syscall_base.rs` only checks that the target class is Cairo 1; it performs **no** execution-mode check: [2](#0-1) 

Compare this to `deploy` and `meta_tx_v0`, which explicitly call `reject_syscall_in_validate_mode`. `replace_class` has no such guard.

**Step 3 — `validate_tx` resolves class hash from post-constructor state.**

After the constructor has run (and potentially called `replace_class`), `validate_tx` reads the class hash from the **current** state and dispatches the validate call with `class_hash: None`: [3](#0-2) 

With `class_hash: None`, the execution engine resolves the class dynamically from `state.get_class_hash_at(storage_address)` — which now returns the **replaced** class hash, not the originally declared one.

**Step 4 — The OS mirrors this behavior intentionally.**

The OS-level `execute_deploy_account_transaction` in `transaction_impls.cairo` explicitly reads the newest state entry after constructor invocation and uses `state_entry.class_hash` for the `__validate_deploy__` execution context: [4](#0-3) 

The comment "Fetch the newest state entry, after constructor invocation" confirms the post-constructor class hash is used. The blockifier and OS are consistent — and both are vulnerable.

---

### Impact Explanation

An attacker can:

1. Declare **class A** (Cairo 1): constructor calls `replace_class(class_B_hash)`.
2. Declare **class B** (Cairo 1): `__validate_deploy__` unconditionally returns `VALIDATED`, ignoring the signature.
3. Pre-fund the deterministic contract address (computed from class A hash + salt).
4. Submit a `DeployAccount` transaction referencing class A with **any arbitrary signature**.

Execution:
- Constructor of class A runs → `replace_class(class_B)` succeeds → state now maps the address to class B.
- `validate_tx` reads class B hash → dispatches `__validate_deploy__` against class B's code.
- Class B's `__validate_deploy__` returns `VALIDATED` regardless of signature.
- Transaction is **accepted** with an invalid/empty signature.

The deployed account ends up with class B's code. The signature over the transaction hash (which commits to class A) is **never verified**. This violates the core security invariant of account abstraction for `DeployAccount`.

**Impact: Critical** — Invalid DeployAccount transaction accepted through account validation bypass.

---

### Likelihood Explanation

- Requires no privileged access; any user can declare classes and submit transactions.
- Requires pre-funding the contract address (trivially done by sending ETH/STRK to the deterministic address before deployment).
- The attack is fully deterministic and reproducible.
- No race conditions or timing dependencies.

---

### Recommendation

Block `replace_class` when the execution context is `ExecutionMode::Validate`, consistent with how `deploy`, `call_contract`, and `meta_tx_v0` are blocked. Add the guard in `syscall_base.rs`:

```rust
pub fn replace_class(&mut self, class_hash: ClassHash) -> SyscallResult<()> {
    if self.context.execution_mode == ExecutionMode::Validate {
        return self.reject_syscall_in_validate_mode("replace_class");
    }
    // ... existing checks
}
```

Alternatively, for `DeployAccount` specifically, snapshot the class hash **before** running the constructor and use that snapshot (not the post-constructor state) when dispatching `__validate_deploy__`. This would require a targeted fix in `validate_tx` or `run_non_revertible`.

The OS-level code in `transaction_impls.cairo` would also need to be updated to use the pre-constructor class hash for `__validate_deploy__`.

### Proof of Concept

```rust
// Pseudocode for a Rust integration test:

// 1. Declare class_b: __validate_deploy__ returns VALIDATED unconditionally.
// 2. Declare class_a: constructor calls replace_class(class_b_hash).
// 3. Compute contract_address = calculate_contract_address(salt, class_a_hash, [], 0).
// 4. Fund contract_address with enough balance for fees.
// 5. Build DeployAccount tx: class_hash=class_a, signature=[0xdead, 0xbeef] (invalid).
// 6. Execute the transaction.
// 7. Assert: tx succeeds (no ValidateTransactionError).
// 8. Assert: state.get_class_hash_at(contract_address) == class_b_hash.
// 9. Assert: validate_call_info.call.class_hash resolves to class_b_hash, NOT class_a_hash.
//    (Demonstrating __validate_deploy__ ran against class_b, not class_a.)
```

The key assertion is that `validate_call_info` reflects execution of class B's `__validate_deploy__`, while the transaction was signed (invalidly) against class A's hash — proving the signature was never checked against the declared class.
