### Title
Missing Verification of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly accumulates `(big_y, big_c)` contributions from participants without any proof of correctness. A single malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that decrypts to an attacker-influenced confidential key instead of the legitimate `msk · H(pk ‖ app_id)`.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `CKDOutput` and sums the components unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to compute and send:

- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

as implemented in `compute_signature_share`: [2](#0-1) 

However, no zero-knowledge proof, commitment, or consistency check is attached to the message sent by `do_ckd_participant`:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

The coordinator has no way to verify that the received `(big_y, big_c)` pair was honestly derived from the participant's actual key share `x_i` and a fresh random `y_i`. A malicious participant can substitute arbitrary group elements `(big_y', big_c')`.

The final output is used by the requester via `unmask`:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [4](#0-3) 

If a malicious participant sends `(big_y_i + Δ_Y, big_c_i + Δ_C)` for arbitrary group elements `Δ_Y, Δ_C`, the coordinator computes:

- `Y' = Y + Δ_Y`
- `C' = C + Δ_C`

The requester then obtains:

```
unmask(app_sk) = C' − app_sk · Y'
               = msk · H(pk ‖ app_id) + Δ_C − app_sk · Δ_Y
```

This is not the legitimate confidential key unless `Δ_C = app_sk · Δ_Y`, which the attacker cannot enforce without knowing `app_sk`. The result is a corrupted, attacker-influenced output.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept incorrect cryptographic outputs.**

The coordinator and requester have no mechanism to detect the corruption. The `CKDOutput` passes through `unmask` and yields a wrong key silently. Any downstream use of the derived key (e.g., decryption, authentication, deterministic secret derivation inside a TEE) will fail or produce attacker-influenced material. A single malicious participant out of the threshold set is sufficient to trigger this.

---

### Likelihood Explanation

Any participant in the CKD protocol can act maliciously. The attack requires only that one participant deviate from the protocol by sending wrong group elements — no cryptographic break, no external compromise, and no special privilege beyond being a legitimate protocol participant. The library's documented use case (industrial MPC, TEE key management) makes this a realistic threat.

---

### Recommendation

Add a zero-knowledge proof of correct construction alongside each participant's `(norm_big_y, norm_big_c)` contribution. Concretely, each participant should prove in zero knowledge that:

1. They know a scalar `y_i` such that `norm_big_y_i = λ_i · y_i · G`.
2. `norm_big_c_i − norm_big_y_i · (app_pk / G)` equals `λ_i · x_i · H(pk ‖ app_id)`, consistent with their committed public key share.

A Schnorr-style sigma protocol (already used elsewhere in the codebase for DKG proofs of knowledge in `src/dkg.rs`) can be adapted here. The coordinator must verify all proofs before summing contributions, and abort if any proof fails.

---

### Proof of Concept

```
Setup:
  - 3 participants, threshold = 2
  - Honest participants 1 and 3 compute correct (norm_big_y, norm_big_c)
  - Malicious participant 2 sends (norm_big_y_2 + Δ_Y, norm_big_c_2 + Δ_C)
    for arbitrary Δ_Y, Δ_C ∈ G1

Expected coordinator output:
  C - app_sk * Y = msk * H(pk || app_id)   [correct confidential key]

Actual coordinator output:
  (C + Δ_C) - app_sk * (Y + Δ_Y)
  = msk * H(pk || app_id) + Δ_C - app_sk * Δ_Y
  ≠ msk * H(pk || app_id)                  [corrupted key, no error raised]
```

The `do_ckd_coordinator` function returns `Ok(Some(ckd_output))` with the corrupted value. The requester calls `ckd_output.unmask(app_sk)` and silently receives the wrong key. No error is raised at any point in the protocol. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
