The `expires_at` is computed at enqueue time as `apply_state.block_height + config.wasm_config.limit_config.yield_timeout_length_in_blocks`. ### Title
PromiseYield Timeout Queue Ordering Broken by `yield_timeout_length_in_blocks` Protocol Upgrade — (File: `runtime/runtime/src/lib.rs`)

### Summary
`resolve_promise_yield_timeouts` assumes the `PromiseYieldTimeout` trie queue is monotonically ordered by `expires_at`. This invariant holds only when `yield_timeout_length_in_blocks` is constant across all enqueued entries. A protocol upgrade that reduces this parameter causes newer entries to carry a smaller `expires_at` than older ones already in the queue. The `break` on the first non-expired front entry then permanently blocks all newer (already-expired) entries behind it, so their `PromiseResult::Failed` callbacks are never dispatched.

### Finding Description

**Enqueue path** — `runtime/runtime/src/function_call.rs`:

```rust
enqueue_promise_yield_timeout(
    state_update,
    &mut promise_yield_indices,
    account_id.clone(),
    receipt.input_data_ids[0],
    apply_state.block_height
        + config.wasm_config.limit_config.yield_timeout_length_in_blocks,  // ← baked-in at creation time
);
``` [1](#0-0) 

Every `promise_yield_create` call appends a `PromiseYieldTimeout { expires_at }` to the back of the persistent trie queue. Because `block_height` is monotonically increasing and `yield_timeout_length_in_blocks` is assumed constant, the queue is expected to be sorted by `expires_at`.

**Dequeue path** — `runtime/runtime/src/lib.rs`:

```rust
// Queue entries are ordered by expires_at
if queue_entry.expires_at > apply_state.block_height {
    break;   // ← stops at first non-expired entry; never looks further
}
``` [2](#0-1) 

The loop pops from `first_index` forward and breaks the moment it sees an entry that has not yet expired. The code comment makes the ordering assumption explicit.

**The broken invariant**: `yield_timeout_length_in_blocks` is a field of `LimitConfig` inside `RuntimeConfig`, which is versioned per protocol version. [3](#0-2) 

If a protocol upgrade lowers this value (e.g., from 200 to 100 blocks), any `promise_yield_create` call made after the upgrade produces an entry with `expires_at = current_height + 100`, which is smaller than entries already in the queue that were created before the upgrade with `expires_at = old_height + 200`. The queue is no longer sorted. The `break` stops at the first old, not-yet-expired entry and never reaches the newer, already-expired entries behind it.

**Exact corrupted state**: The `PromiseYieldTimeout` entries for yields created after the protocol upgrade remain in the trie queue with `first_index` never advancing past the blocking older entry. The corresponding `PromiseYieldReceipt` keys remain in state indefinitely. No `PromiseResume` receipt with `data: None` is ever emitted for those yields, so the callback action receipt is never executed. [4](#0-3) 

### Impact Explanation

Any contract that called `promise_yield_create` after a protocol upgrade that reduced `yield_timeout_length_in_blocks` will have its timeout callback permanently suppressed as long as any older entry with a larger `expires_at` sits ahead of it in the queue. The callback (`PromiseResult::Failed`) is never delivered. The `PromiseYieldReceipt` and its associated state keys (`PromiseYieldStatus`, `YieldId` mappings) are never cleaned up. This is a receipt-causality violation: a receipt whose precondition (timeout expiry) is satisfied is never executed.

### Likelihood Explanation

`yield_timeout_length_in_blocks` is currently 200 in every deployed `RuntimeConfig` snapshot. However, the parameter is part of the versioned `LimitConfig` and can be changed in any future protocol upgrade. The bug is latent and silent — it would only manifest after such an upgrade, with no on-chain signal that yields are stuck.

### Recommendation

Replace the `break` with `continue` (skip the non-expired entry and keep scanning) or, better, sort the queue on dequeue by scanning all entries up to the compute limit rather than stopping at the first non-expired front entry. The simplest safe fix mirrors the recommendation in the external report:

```rust
// Instead of:
if queue_entry.expires_at > apply_state.block_height {
    break;
}
// Use:
if queue_entry.expires_at > apply_state.block_height {
    promise_yield_indices.first_index += 1;  // advance past this entry
    continue;
}
```

Alternatively, store entries in a heap or re-sort on protocol upgrade.

### Proof of Concept

1. Block 100: Contract A calls `promise_yield_create`. `yield_timeout_length_in_blocks = 200`. Entry appended: `{ expires_at: 300, data_id: A }`.
2. Protocol upgrade at block 150 sets `yield_timeout_length_in_blocks = 50`.
3. Block 160: Contract B calls `promise_yield_create`. Entry appended: `{ expires_at: 210, data_id: B }`. Queue is now `[{300, A}, {210, B}]` — **not sorted**.
4. Block 210: `resolve_promise_yield_timeouts` runs. `first_index` points to `{300, A}`. Check: `300 > 210` → `break`. Entry `{210, B}` is never reached.
5. Contract B's callback is never invoked with `PromiseResult::Failed`. Its `PromiseYieldReceipt` key remains in state forever (or until block 300 when A's entry finally unblocks the queue).
6. Any contract logic that depends on the timeout callback (e.g., releasing locked funds, retrying an operation) is permanently stalled.
