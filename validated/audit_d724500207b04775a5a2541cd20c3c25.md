Audit Report

## Title
Byzantine Dealer DoS via Out-of-Range Chunks Triggering Unbounded BSGS Table Allocation in `dec_chunks` — (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

## Summary

A single Byzantine dealer below the fault threshold can produce a dealing with out-of-range plaintext chunks that nonetheless passes all public verification steps (`verify_ciphertext_integrity`, `verify_chunking`, `verify_sharing`). When honest nodes call `load_transcript`, `dec_chunks` falls through to `CheatingDealerDlogSolver::new(n, m)`, which allocates up to 2 GiB of BSGS table memory and may stall for hours per dealing. All honest nodes in the subnet are affected simultaneously on every epoch transition that includes the malicious dealing.

## Finding Description

**Root cause — NIZK proof does not bound individual chunks to `[CHUNK_MIN, CHUNK_MAX]`.**

`verify_chunking` only checks that the `z_s` response scalars satisfy:

```
ss = n * m * (CHUNK_SIZE - 1) * CHALLENGE_MASK
zz = 2 * NUM_ZK_REPETITIONS * ss
z_sk < zz_big  (for each z_sk in nizk.z_s)
``` [1](#0-0) 

This bound is orders of magnitude larger than `CHUNK_MAX = 0xFFFF`. The proof proves knowledge of the plaintext chunks but does not constrain individual `s_{i,j}` values to `[CHUNK_MIN, CHUNK_MAX]`. The existence of `CheatingDealerDlogSolver` itself is the canonical confirmation: its doc comment explicitly states it handles "a malicious DKG participant whose NIZK chunking proof checks out." [2](#0-1) 

**`verify_ciphertext_integrity` also passes for out-of-range chunks.**

The function only checks pairing equations over group elements and has no visibility into the integer value of the plaintext chunks. [3](#0-2) 

The existing test `should_decrypt_correctly_for_cheating_dealer` explicitly confirms this: [4](#0-3) 

**`dec_chunks` unconditionally allocates the BSGS table on any out-of-range chunk.**

When `HonestDealerDlogLookupTable::solve_several` returns `None` for any chunk (because the chunk is outside `[CHUNK_MIN, CHUNK_MAX]`), `CheatingDealerDlogSolver::new(n, m)` is called with no prior guard: [5](#0-4) 

`CheatingDealerDlogSolver::new` is hardcoded to allow up to 2 GiB: [6](#0-5) 

The BSGS range `2*zz-1` for a 40-node subnet is on the order of hundreds of billions, requiring a table of ~20 million entries (~1 GiB) and tens of thousands of giant steps per chunk, each requiring a `Gt` group operation and hash lookup. [7](#0-6) 

**`load_transcript` calls `dec_chunks` for every dealing in the transcript.**

The call chain is: `load_transcript` → `compute_threshold_signing_key` → `decrypt` → `dec_chunks`. [8](#0-7) [9](#0-8) 

`load_transcript` is invoked in a background thread per transcript on every DKG summary block: [10](#0-9) 

The thread stalls indefinitely (hours per the code comment) trying to brute-force the discrete log, preventing the node from obtaining its threshold signing key for the new epoch.

**`verify_dealing` does not catch this.**

`verify_dealing` calls `verify_zk_proofs`, which runs `verify_ciphertext_integrity` and `verify_chunking` — both of which pass for out-of-range chunks as shown above. There is no check that individual plaintext chunks are within `[CHUNK_MIN, CHUNK_MAX]`. [11](#0-10) 

## Impact Explanation

This is a **High** severity subnet availability DoS. A single Byzantine dealer (below the fault threshold) can stall all honest nodes' threshold signing key derivation for a new epoch. Each affected node allocates up to 2 GiB of BSGS table memory and spends hours in computation per malicious dealing. With multiple cheating dealers (still below the fault threshold), the impact multiplies. The subnet loses its ability to produce threshold signatures for the affected epoch, disrupting all chain-key operations (ckBTC, ckETH, threshold ECDSA/BLS, etc.) that depend on the signing key being available. This matches the allowed impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."**

## Likelihood Explanation

- Requires only a single Byzantine dealer node (well below the fault threshold).
- The dealing passes all public verification steps (`verify_chunking`, `verify_sharing`, `verify_ciphertext_integrity`).
- The attack is deterministic and reproducible.
- No privileged access, key material, or network-level attack is required beyond being a registered dealer in a DKG round.
- The attack triggers on every epoch transition that includes the malicious dealing.

## Recommendation

1. **Bound check before BSGS**: After `HonestDealerDlogLookupTable::solve_several` returns `None`, verify that the decrypted group element is plausibly within the extended BSGS range before invoking `CheatingDealerDlogSolver`. If the element is provably outside the search range, return `Err(DecErr::InvalidChunk)` immediately.
2. **Reject cheating dealings at verification time**: Extend `verify_dealing` / `verify_zk_proofs` to detect and reject dealings whose NIZK proof witnesses imply out-of-range chunks, so such dealings never enter the transcript.
3. **Cap BSGS computation time**: Add a timeout or iteration limit to `CheatingDealerDlogSolver::solve` so that a single bad dealing cannot stall a node indefinitely.
4. **Rate-limit or sandbox transcript loading**: Run `load_transcript` with memory and CPU limits so that a malicious dealing cannot exhaust node resources.

## Proof of Concept

The existing test `should_decrypt_correctly_for_cheating_dealer` in `rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/tests/forward_secure.rs` already demonstrates the core mechanism: [12](#0-11) 

To demonstrate the full DoS path, extend this test to:
1. Call `prove_chunking` on the out-of-range chunks (to generate a valid NIZK proof).
2. Call `verify_chunking` to confirm the proof passes.
3. Call `verify_dealing` to confirm the dealing passes all public verification.
4. Measure wall-clock time and peak memory of `dec_chunks` for a subnet-sized parameter set (n=40, m=16) to confirm the 2 GiB allocation and multi-hour stall.
