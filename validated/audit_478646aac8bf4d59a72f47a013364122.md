### Title
Missing `msg_hash` Non-Zero Validation in OT-Based ECDSA `sign` Allows Presignature Secret Disclosure - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary
The OT-based ECDSA `sign` function accepts a zero `msg_hash` without validation. When `msg_hash = 0`, the signature component collapses to `s = r · σ`, directly revealing the presignature secret `σ = k · x` to any observer of the signature. The sibling robust ECDSA `sign` function explicitly rejects `msg_hash = 0` for exactly this reason, but the OT-based variant has no such guard.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/sign.rs`, the public `sign` entry point performs several input validations (participant count, duplicates, coordinator membership, threshold size) but contains **no check that `msg_hash ≠ 0`**: [1](#0-0) 

The signature share computation in `compute_signature_share` is:

```
s_i = msg_hash * k_i + r * sigma_i
``` [2](#0-1) 

When `msg_hash = 0`, this reduces to:

```
s_i = r * sigma_i
```

After the coordinator aggregates all shares, the final `s = r · σ`. Since `r` is the public x-coordinate of `R` (already in the signature), any observer can compute:

```
σ = s / r
```

`σ` is the presignature secret defined in `PresignOutput` as `sigma: sigma_i`, which is a share of `k · x` (nonce times private key): [3](#0-2) 

By contrast, the robust ECDSA `sign` function **explicitly rejects** `msg_hash = 0` with the comment "to avoid potential split view attacks": [4](#0-3) 

The OT-based ECDSA `sign` function has no equivalent guard, creating an inconsistency in the security posture of the two signing variants.

---

### Impact Explanation

**Impact: Critical — Disclosure of presign secret material.**

When `msg_hash = 0` is accepted:

1. Each participant computes `s_i = r · σ_i` and sends it to the coordinator.
2. The coordinator aggregates to `s = r · σ` where `σ = k · x` is the aggregate presignature secret.
3. Since `r` is public (it is the x-coordinate of `R` in the output signature), `σ = s / r` is directly computable by any party that sees the signature `(R, s)`.
4. `σ = k · x` is presign secret material — its disclosure violates the security model of the protocol.
5. Furthermore, if an attacker obtains a second signature `(R, s')` over a non-zero `msg_hash'` using the **same presignature** (nonce reuse, a separate but documented risk), they can recover the nonce `k = (s' − r · σ) / msg_hash'` and then the private key `x = σ / k`.

This matches the allowed impact: **Critical — Extraction or disclosure of presign secrets / nonce material**.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The `msg_hash` parameter is caller-supplied. Any application that passes a pre-hashed message could inadvertently or maliciously supply a zero scalar (e.g., hashing a specially crafted input, or a bug in the hashing layer).
- A malicious coordinator distributing the message to be signed could instruct participants to sign `msg_hash = 0`.
- The library provides no defense; the protocol runs to completion and produces a valid-looking signature without any error.
- The robust ECDSA variant's explicit rejection of this case confirms the library authors are aware of the risk in the signing context, making the omission in the OT-based variant a clear gap.

---

### Recommendation

Add a non-zero check for `msg_hash` in the OT-based ECDSA `sign` function, mirroring the existing check in the robust ECDSA variant:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0".to_string(),
    ));
}
```

This should be placed in the `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` alongside the other input validations, before the protocol is started. [5](#0-4) 

---

### Proof of Concept

**Setup:** Two participants run OT-based ECDSA presign to obtain `PresignOutput { big_r: R, k: k_i, sigma: σ_i }`.

**Attack:**
1. Caller invokes `sign(participants, coordinator, threshold, me, public_key, presignature, Scalar::ZERO)`.
2. No error is returned; the protocol proceeds.
3. Each participant computes `s_i = 0 · k_i + r · σ_i = r · σ_i`.
4. Coordinator aggregates: `s = r · σ` where `σ = Σ λ_i · σ_i = k · x`.
5. Output signature is `(R, s)` with `s = r · σ`.
6. Attacker computes `σ = s · r^{-1} mod q` — the presignature secret `k · x` is now fully disclosed.

**Verification:** The signature passes `sig.verify(&public_key, &Scalar::ZERO)` because `s · G = r · σ · G = r · k · x · G` and the verifier equation `0 · R + r · X = r · x · G = r · σ · k^{-1} · G` is satisfied, confirming the signature is accepted while leaking `σ`. [2](#0-1) [4](#0-3)

    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,

    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });

        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,

        });

        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );

}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<Scalar, ProtocolError> {
    // Round 1
    // Linearize ki
    // Spec 1.1
    let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
    let k_i = lambda * presignature.k;

    // Linearize sigmai
    // Spec 1.2
    let sigma_i = lambda * presignature.sigma;

    // Compute si = h * ki + Rx * sigmai
    // Spec 1.3
    let r = x_coordinate(&presignature.big_r);
    Ok(msg_hash * k_i + r * sigma_i)
}
