### Title
Presignature Participant-Set Mismatch in FROST EdDSA `sign_v2` Causes Permanent Signing Denial — (`src/frost/eddsa/sign.rs`)

---

### Summary

The `sign_v2` function in the FROST EdDSA module accepts a `PresignOutput` whose `commitments_map` was built during the presign phase with a specific participant set, but performs **no validation** that this participant set matches the `participants` argument supplied at signing time. When the participant set diverges between presign and sign — due to a participant dropping out, a reshare, or a malicious participant deliberately abandoning the protocol after presign — the signing protocol fails permanently for that presignature, wasting all offline work and denying signing to honest parties.

---

### Finding Description

In `src/frost/eddsa/sign.rs`, `sign_v2` accepts:

- `participants: &[Participant]` — the current signing participant set
- `presignature: PresignOutput` — contains `commitments_map: BTreeMap<frost_ed25519::Identifier, round1::SigningCommitments>` built during the presign phase with a (potentially different) participant set P₁ [1](#0-0) 

The only validation performed is `assert_sign_inputs`, which checks that `me` and `coordinator` are present in `participants`. It does **not** check that the identifiers in `presignature.commitments_map` match the identifiers in `participants`. [1](#0-0) 

In `do_sign_coordinator_v2`, the stale `commitments_map` is used directly to build the `SigningPackage`:

```rust
let signing_package =
    frost_ed25519::SigningPackage::new(presignature.commitments_map, message.as_slice());
``` [2](#0-1) 

In `do_sign_participant_v2`, each participant signs using the presignature's `commitments_map` and their own `nonces`:

```rust
let signing_package = SigningPackage::new(presignature.commitments_map, message);
let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
``` [3](#0-2) 

There are two concrete failure modes when P₁ ≠ P₂:

1. **A participant is in `participants` (P₂) but not in `commitments_map` (P₁):** `round2::sign` fails because the participant's identifier is absent from the signing package.
2. **`commitments_map` contains entries for participants not in `participants`:** `aggregate` fails because signature shares are missing for those entries, even though the remaining honest parties provided valid shares.

The analog to the external report is exact: just as `xezETH` is minted at time T₁ based on valuation V₁ and `ezETH` is minted at time T₂ based on valuation V₂ — with no enforcement that V₁ = V₂ — presignatures are computed at time T₁ with participant set P₁ and consumed at time T₂ with participant set P₂, with **no enforcement that P₁ = P₂**. The two-phase protocol accumulates inconsistent state that only surfaces as a runtime failure.

---

### Impact Explanation

This matches the allowed High impact: **"Permanent denial of signing… for honest parties under valid protocol inputs and documented trust assumptions."**

A malicious participant can deliberately exploit this:

1. Participate honestly in presign with P₁ = {p₁, p₂, p₃}, causing all parties to invest offline computation.
2. Refuse to participate in the subsequent sign round (drop out).
3. The remaining honest parties P₂ = {p₁, p₂} attempt `sign_v2` with the old presignature.
4. The `commitments_map` still contains p₃'s commitment; `aggregate` fails because no signature share for p₃ is provided.
5. The presignature is permanently wasted; a full new presign round is required.

Repeated griefing exhausts the presignature pool and permanently blocks signing for honest parties without any cryptographic capability on the attacker's part.

---

### Likelihood Explanation

This is realistically reachable because:

- Presignatures are computed **offline** (before the message is known), so participant availability can legitimately change between presign and sign.
- A reshare between presign and sign changes the participant set, making all prior presignatures incompatible.
- A single malicious participant can trigger this at will by participating in presign and then refusing to sign, with no cost beyond the presign round itself.
- The library provides no mechanism to detect or prevent the mismatch at initialization time; the error surfaces only deep inside the async protocol execution.

---

### Recommendation

Add a validation step inside `sign_v2` (before constructing the protocol future) that checks the identifiers in `presignature.commitments_map` exactly match the identifiers derived from `participants`:

```rust
// After building `participants: ParticipantList`
let presign_ids: BTreeSet<_> = presignature.commitments_map.keys().cloned().collect();
let sign_ids: BTreeSet<_> = participants
    .participants()
    .iter()
    .map(|p| p.to_identifier::<frost_ed25519::Ed25519Sha512>())
    .collect::<Result<_, _>>()?;
if presign_ids != sign_ids {
    return Err(InitializationError::BadParameters(
        "presignature participant set does not match signing participant set".to_string(),
    ));
}
```

This mirrors the pattern already used in `presign` for the OT-based scheme, which checks threshold consistency between the triple and the presign arguments. [4](#0-3) 

---

### Proof of Concept

1. Participants P₁ = {p₁, p₂, p₃} run the FROST EdDSA presign protocol. Each receives a `PresignOutput` with `commitments_map` containing entries for all three identifiers and their own `nonces`.
2. Before signing, p₃ becomes unavailable (network failure or deliberate drop-out).
3. p₁ and p₂ call `sign_v2` with `participants = [p₁, p₂]` and the `PresignOutput` from step 1.
4. `sign_v2` accepts the call — `assert_sign_inputs` passes because both p₁ and p₂ are in `participants`.
5. In `do_sign_coordinator_v2`, the `SigningPackage` is built from the stale `commitments_map` that still contains p₃'s commitment.
6. p₁ and p₂ each produce a valid `round2::SignatureShare`; p₃ produces none.
7. `aggregate` is called with two signature shares but a `SigningPackage` referencing three commitments. `aggregate` returns an error (missing share for p₃).
8. The coordinator propagates `ProtocolError::AssertionFailed`. The presignature is permanently consumed and signing fails for honest parties p₁ and p₂. [5](#0-4) [6](#0-5)

async fn do_sign_coordinator_v2(
    mut chan: SharedChannel,
    participants: ParticipantList,
    threshold: ReconstructionLowerBound,
    me: Participant,
    keygen_output: KeygenOutput,
    presignature: PresignOutput,
    message: Vec<u8>,
) -> Result<SignatureOption, ProtocolError> {
    // --- Round 1
    let signing_package =
        frost_ed25519::SigningPackage::new(presignature.commitments_map, message.as_slice());

    let mut signature_shares: BTreeMap<frost_ed25519::Identifier, round2::SignatureShare> =
        BTreeMap::new();

    let vk_package = keygen_output.public_key;

    let key_package =
        construct_key_package(threshold, me, keygen_output.private_share, &vk_package)?;

    let key_package = Zeroizing::new(key_package);
    let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)
        .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
    signature_shares.insert(me.to_identifier()?, signature_share);

    let sign_waitpoint = chan.next_waitpoint();
    for (from, signature_share) in
        recv_from_others(&chan, sign_waitpoint, &participants, me).await?
    {
        signature_shares.insert(from.to_identifier()?, signature_share);
    }

    // --- Signature aggregation.
    // * Converted collected signature shares into the signature.
    // * Signature is verified internally during `aggregate()` call.
    // We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
    // Feature "cheater-detection" only points to a malicious participant, if there's such.
    // It doesn't bring any additional guarantees.
    let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
    let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
        .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;

    Ok(Some(signature))

fn do_sign_participant_v2(
    mut chan: SharedChannel,
    threshold: ReconstructionLowerBound,
    me: Participant,
    coordinator: Participant,
    keygen_output: &KeygenOutput,
    presignature: PresignOutput,
    message: &[u8],
) -> Result<SignatureOption, ProtocolError> {
    // --- Round 1.
    // * Send our signature share.
    if coordinator == me {
        return Err(ProtocolError::AssertionFailed(
            "the do_sign_participant function cannot be called
            for a coordinator"
                .to_string(),
        ));
    }

    let vk_package = keygen_output.public_key;

    let key_package =
        construct_key_package(threshold, me, keygen_output.private_share, &vk_package)?;
    // Ensures the values are zeroized on drop
    let key_package = Zeroizing::new(key_package);

    let signing_package = SigningPackage::new(presignature.commitments_map, message);
    let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)
        .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;

    let sign_waitpoint = chan.next_waitpoint();
    chan.send_private(sign_waitpoint, coordinator, &signature_share)?;

    Ok(None)
