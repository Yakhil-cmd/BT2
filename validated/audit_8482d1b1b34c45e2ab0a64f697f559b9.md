### Title
Truncating Floor Division in `native_per_pubdata` Computation Undercharges Users for Pubdata — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` — the native resource cost per pubdata byte — is computed using floor (truncating) division (`wrapping_div`) instead of ceiling division (`div_ceil`). Because this value is used to charge users for every pubdata byte they consume, rounding it down systematically undercharges users, allowing them to consume more pubdata than they have paid for in native resources. The analogous `native_per_gas` ratio correctly uses `div_ceil` to protect the protocol, but the pubdata ratio does not.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (the L2 transaction validation path), `native_per_pubdata` is derived as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs integer floor division. The same truncating pattern appears in the public API helper:

```rust
// api/src/helpers.rs, line 427
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

By contrast, `native_per_gas` — the analogous ratio for gas — is computed with ceiling division:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 135
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

`native_per_pubdata` is subsequently used in two critical places:

1. **Intrinsic pubdata overhead deduction** — in `create_resources_for_tx` (`gas_helpers.rs`, line 352):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ```
2. **Actual pubdata charge** — in `get_resources_to_charge_for_pubdata` (`gas_helpers.rs`, line 430–432):
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ```

When `pubdata_price % native_price != 0`, the floor-divided `native_per_pubdata` is strictly less than the true ratio. Every pubdata byte is charged at a rate that is up to `(native_price − 1)` native units below the true cost. The shortfall scales linearly with the number of pubdata bytes consumed.

**Concrete example:**
- `pubdata_price = 11`, `native_price = 10`
- True `native_per_pubdata = ceil(11/10) = 2`
- Actual `native_per_pubdata = floor(11/10) = 1`
- For a transaction generating 1 000 pubdata bytes: user pays 1 000 native instead of 2 000 — a 50 % undercharge.

The same truncation is present in the `delta_gas` adjustment in `compute_gas_refund` (`refund_calculation.rs`, line 72):

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
```

Floor division here means `delta_gas` can be 0 when it should be 1, causing the user to avoid paying for the last fractional native unit consumed. This is a secondary, smaller-magnitude instance of the same rounding direction error.

---

### Impact Explanation

The protocol undercharges every transaction that generates pubdata when `pubdata_price % native_price != 0`. Users receive a larger effective native resource budget than they have paid for, and pay less for actual pubdata consumption. This:

- Allows users to submit transactions that would otherwise be rejected for insufficient native resources.
- Causes the protocol to subsidize the difference between the true pubdata cost and the truncated cost, eroding the economic model that ties native resource consumption to proving costs.
- Scales with pubdata volume: a transaction generating `N` pubdata bytes is undercharged by up to `(native_price − 1) × N` native units.

---

### Likelihood Explanation

The condition `pubdata_price % native_price != 0` is the common case. Both `pubdata_price` and `native_price` are arbitrary `U256` values set by the operator via oracle. There is no enforcement that `pubdata_price` be an exact multiple of `native_price`. Any unprivileged transaction sender can trigger this path simply by submitting a normal L2 transaction when the block's pricing parameters satisfy the condition.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` computation in both locations:

```rust
// validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

Similarly, the `delta_gas` adjustment in `compute_gas_refund` should use ceiling division:

```rust
// refund_calculation.rs, line 72
(native_used.div_ceil(native_per_gas)) as i64 - (gas_used as i64)
```

This mirrors the already-correct treatment of `native_per_gas` and ensures the protocol never loses native resource revenue due to rounding.

First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
