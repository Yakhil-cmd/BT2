### Title
Pool Admin Can Instantly Raise Fees to Maximum Without Timelock, Causing Direct Loss to Swapping Users — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

The `MetricOmmPoolFactory` enforces a timelock for price-provider changes (`priceProviderTimelock`) but applies **no timelock** to pool-admin fee changes (`setPoolAdminFees`, `setPoolBinAdditionalFees`). A pool admin can atomically raise fees to the factory-enforced cap and have those fees take effect on the very next swap, with no window for users to react.

### Finding Description

`setPoolAdminFees` and `setPoolBinAdditionalFees` are callable by the pool admin at any time with no delay: [1](#0-0) [2](#0-1) 

Both calls propagate immediately into pool state via `setPoolFees` and `setBinAdditionalFees` on the pool: [3](#0-2) [4](#0-3) 

By contrast, price-provider changes go through a two-step propose/execute flow gated by `priceProviderTimelock[pool]`: [5](#0-4) 

The design explicitly acknowledges that the pool admin is **semi-trusted only inside caps and timelocks** (factory comment and Smart Audit Pivot). Timelocks exist for provider changes but are entirely absent for fee changes, breaking that invariant.

The hard caps allow the admin to raise fees to:
- `adminSpreadFeeE6` up to `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20% spread)
- `adminNotionalFeeE8` up to `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` (1% notional)
- `addFeeBuyE6` / `addFeeSellE6` up to `uint16` max = 65 535 (~6.5% per-bin) [6](#0-5) 

### Impact Explanation

A pool admin can atomically:
1. Call `setPoolAdminFees(pool, 200_000, 1_000_000)` — raises spread fee to 20% and notional fee to 1%.
2. Call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` — raises per-bin buy/sell fees to ~6.5%.

Any swap executed in the same block (or any subsequent block before the user can react) pays the inflated fee. The user receives far fewer output tokens than the pre-change quote indicated. The excess is retained in the pool as fee surplus, claimable by the admin via `collectPoolFees`. This is a direct, quantifiable loss of user principal on every swap.

### Likelihood Explanation

The pool admin is a semi-trusted role, not a fully trusted protocol owner. The factory's own design distinguishes the two (separate `onlyOwner` vs `onlyPoolAdmin` paths). A compromised or malicious pool admin — a realistic threat for any permissioned DeFi pool — can execute this with a single transaction. No special conditions, no oracle manipulation, and no reentrancy are required.

### Recommendation

Apply the same two-step timelock pattern used for price-provider changes to fee changes:

1. Add a `pendingFeeConfig` and `pendingFeeConfigExecuteAfter` mapping in the factory.
2. `setPoolAdminFees` stores the proposed config and records `block.timestamp + feeChangeTimelock`.
3. A separate `executePoolAdminFees(pool)` function enforces the delay before calling `setPoolFees` on the pool.

The timelock duration should be long enough (e.g., 24–72 hours) for users to observe the pending change and exit positions before it takes effect.

### Proof of Concept

```solidity
// Attacker is pool admin
// Step 1: Raise fees to maximum — takes effect immediately
factory.setPoolAdminFees(pool, 200_000, 1_000_000); // 20% spread, 1% notional
factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535); // ~6.5% per-bin

// Step 2: Victim's swap executes in the same or next block
// Victim calls pool.swap(...) expecting ~0% admin fee
// Pool now charges 20% + 6.5% = ~26.5% total fee on the swap
// Victim receives ~26.5% fewer output tokens than quoted
// Admin collects the surplus via factory.collectPoolFees(pool)
``` [1](#0-0) [2](#0-1) [3](#0-2)

  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
