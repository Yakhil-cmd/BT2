### Title
Reverting `adminFeeDestination` permanently bricks fee collection and blocks protocol owner's `setPoolProtocolFee` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`, `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`setPoolAdminFees` and `setPoolProtocolFee` both call `collectFees` as a **mandatory prerequisite** before updating fee rates. `collectFees` performs a live ERC-20 transfer to the externally-controlled `adminFeeDestination` address. If that address is a contract that reverts on token receipt, every call to `collectFees` reverts, permanently blocking fee collection and preventing the protocol owner from ever changing protocol fees on that pool.

---

### Finding Description

**Invariant being broken**: The protocol owner must always be able to update protocol fees on any pool. The pool admin is semi-trusted and must not be able to block the protocol owner's governance functions.

**Step 1 — The mandatory collect-before-update pattern.**

Both `setPoolAdminFees` and `setPoolProtocolFee` call `collectFees` on the pool before writing new fee config:

`MetricOmmPoolFactory.sol` lines 418–425 (`setPoolAdminFees`): [1](#0-0) 

`MetricOmmPoolFactory.sol` lines 328–335 (`setPoolProtocolFee`): [2](#0-1) 

The permissionless `collectPoolFees` also follows the same path: [3](#0-2) 

**Step 2 — `collectFees` transfers tokens to the external `adminFeeDestination_`.**

Inside `MetricOmmPool.collectFees`, when admin fees are non-zero, tokens are transferred directly to the caller-supplied `adminFeeDestination_`: [4](#0-3) 

If `adminFeeDestination_` is a contract that reverts on ERC-20 receipt, the entire `collectFees` call reverts. Because `collectFees` is called before any state update in `setPoolAdminFees` and `setPoolProtocolFee`, those functions also revert atomically.

**Step 3 — The pool admin can set `adminFeeDestination` to any non-zero address.**

`setPoolAdminFeeDestination` only validates non-zero: [5](#0-4) 

There is no check that the address can actually receive tokens. A pool admin can point this to a contract with a reverting `receive`/fallback or a reverting ERC-20 `tokensReceived`-style hook.

**Step 4 — No bypass exists for the protocol owner.**

- The protocol owner cannot change `adminFeeDestination` (only the pool admin can).
- The protocol owner cannot call `collectFees` directly (pool enforces `onlyFactory`).
- The protocol owner cannot sweep fees from the pool (only from the factory via `collectTokens`).
- `setPoolProtocolFee` always calls `collectFees` with the current (non-zero) admin fee config first, so even setting protocol fees to 0 is blocked.

---

### Impact Explanation

1. **Protocol fees permanently stuck in pool** — `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` are never reset to 0; spread surplus fees are never extracted. All accrued protocol and admin fees are frozen in the pool contract with no recovery path.
2. **Protocol owner cannot change protocol fees for that pool** — `setPoolProtocolFee` is permanently bricked for the affected pool, breaking a core governance function.
3. **Admin-boundary break** — A semi-trusted pool admin blocks the protocol owner's fee management, exceeding the pool admin's intended authority.

---

### Likelihood Explanation

- **Malicious path**: A pool admin deliberately calls `setPoolAdminFeeDestination` with a reverting contract address after fees have accrued. The only guard is `address != 0`. This is a single transaction.
- **Accidental path**: A pool admin legitimately sets `adminFeeDestination` to a multisig or DAO contract that later becomes unable to receive tokens (paused, self-destructed, or bricked). The result is identical.
- Admin fee rates are non-zero in any economically meaningful pool deployment, so the `if (totalFee0ToAdmin > 0)` transfer branches will be reached.

Likelihood: **Low-Medium** (requires pool admin action, but the pool admin is semi-trusted and the function has no safety guard).

---

### Recommendation

Decouple fee collection from fee-rate updates. Two options:

1. **Pull-style (preferred)**: Remove the mandatory `collectFees` call from `setPoolAdminFees` and `setPoolProtocolFee`. Let fee collection remain a separate, independently callable operation (`collectPoolFees`). Fee-rate changes should only update the stored config and call `setPoolFees` on the pool.

2. **Try-catch isolation**: Wrap the `collectFees` call in a `try/catch` inside `setPoolAdminFees` and `setPoolProtocolFee` so that a reverting destination cannot block fee-rate governance. Emit an event on failure so the stuck fees are observable.

Additionally, consider validating in `setPoolAdminFeeDestination` that the new address is an EOA or passes a token-receipt probe, or at minimum document that a reverting destination permanently bricks fee management.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Malicious fee destination: reverts on any ERC-20 transfer
contract RevertingFeeDestination {
    // ERC-20 safeTransfer calls transfer(), which calls this fallback
    fallback() external { revert("blocked"); }
}

// Attack sequence:
// 1. Pool admin deploys RevertingFeeDestination
// 2. Pool admin calls factory.setPoolAdminFeeDestination(pool, address(revertingDest))
// 3. Fees accrue normally via swaps (adminSpreadFeeE6 > 0)
// 4. Protocol owner calls factory.setPoolProtocolFee(pool, newFee, newFee)
//    -> internally calls pool.collectFees(..., address(revertingDest))
//    -> pool calls IERC20(token0).safeTransfer(revertingDest, amount)
//    -> revertingDest reverts
//    -> setPoolProtocolFee reverts
// 5. Protocol owner is permanently blocked from changing protocol fees on this pool.
// 6. collectPoolFees(pool) also reverts for any caller.
// 7. All accrued fees (notionalFeeToken0Scaled, notionalFeeToken1Scaled, spread surplus)
//    are frozen in the pool with no recovery path.
```
