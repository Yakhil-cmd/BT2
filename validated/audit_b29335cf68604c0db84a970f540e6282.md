### Title
`require_running` in `exit_to_near_precompile_callback` Permanently Freezes User Funds When Contract Is Paused - (File: `engine/src/contract_methods/connector.rs`)

### Summary

The `exit_to_near_precompile_callback` function enforces `require_running` (the Aurora Engine's pause guard) at the start of its execution. This callback is a NEAR promise callback that is automatically scheduled by the `ExitToNear` precompile after a user's EVM transaction burns ERC-20 tokens. If the engine is paused between the precompile execution (which burns tokens) and the callback execution (which completes the transfer or refunds burned tokens), the callback fails with `ERR_PAUSED`, permanently freezing the user's funds.

### Finding Description

The `ExitToNear` precompile flow works as follows:

1. A user submits an EVM transaction that calls the `ExitToNear` precompile (e.g., to unwrap wNEAR ERC-20 back to native NEAR, or to bridge ERC-20 tokens back to NEP-141).
2. During EVM execution, the ERC-20 tokens are **burned** and a NEAR cross-contract promise is scheduled (e.g., `near_withdraw` on the wNEAR contract, or `ft_transfer` on the NEP-141 contract).
3. A callback to `exit_to_near_precompile_callback` is attached to that promise.
4. The callback is responsible for either:
   - Completing the wNEAR unwrap by forwarding the released NEAR tokens to the user (`transfer_near` branch), or
   - Refunding the burned ERC-20 tokens if the base promise failed (`refund` branch, when `error_refund` feature is enabled).

The vulnerability is at line 203 of `engine/src/contract_methods/connector.rs`:

```rust
pub fn exit_to_near_precompile_callback<...>(...) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;   // <-- blocks callback if contract is paused
        ...
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
            ...
        }
    })
}
```

If the owner calls `pause_contract` between the precompile execution and the callback execution, `require_running` returns `ERR_PAUSED` and the callback panics. In NEAR, a panicking callback reverts only its own state changes — the base promise (e.g., `near_withdraw`) has already committed. This leaves the system in an irrecoverable state:

- **wNEAR unwrap scenario**: The user's wNEAR ERC-20 tokens are burned, `near_withdraw` succeeds (NEAR is released to Aurora's account), but the callback that forwards those NEAR to the user fails. The NEAR is permanently stuck in Aurora's account and the user's ERC-20 is gone.
- **ERC-20 bridge-back with `error_refund` feature**: The user's ERC-20 tokens are burned, the `ft_transfer` on the NEP-141 contract fails, and the refund callback that would re-mint the ERC-20 tokens also fails. The user loses their tokens permanently.

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user's ERC-20 tokens are burned during EVM execution (before the callback fires). If the contract is paused before the callback executes, the callback is blocked by `require_running`, and the user can never recover their tokens. The NEAR tokens released by `near_withdraw` are also permanently stranded in Aurora's account. There is no recovery path: the callback is a one-time NEAR receipt that will not be retried.

### Likelihood Explanation

The pause window is narrow but realistic. The owner can call `pause_contract` at any time for emergency reasons (e.g., a discovered exploit). The `ExitToNear` precompile callback is an asynchronous NEAR receipt that executes in a subsequent block after the EVM transaction. Any pause that occurs in that inter-block window triggers the freeze. Given that pauses are typically triggered during incidents (when many users may be actively bridging), the probability of overlap is non-negligible.

### Recommendation

Remove `require_running` from `exit_to_near_precompile_callback`. This callback is not a user-initiated action — it is a system-scheduled receipt that must complete to preserve fund integrity. It should execute regardless of the contract's pause state, just as debt repayment should be allowed even when a lending protocol is paused.
