### Title
Gateway Admits Invoke Transactions with Arbitrary Signatures via `skip_stateful_validations` Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (the only place where an account's signature is verified) for any Invoke transaction with `nonce=1` whose sender address appears in the mempool. Because `account_tx_in_pool_or_recent_block` checks only for the *presence* of any transaction from that address — not that a deploy-account transaction specifically was submitted by the same sender — an unprivileged attacker can submit an Invoke with a completely arbitrary signature for any address that has a pending deploy-account in the pool, and the gateway will admit it without ever verifying the signature.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` — the incoming nonce is exactly 1.
3. `account_nonce == Nonce(Felt::ZERO)` — the on-chain nonce is 0 (account not yet deployed).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The comment claims this is safe because the presence of any transaction from that address in the pool implies a deploy-account was submitted. That reasoning is incorrect: `account_tx_in_pool_or_recent_block` returns `true` whenever *any* transaction from the address is in the pool or in the mempool's committed-block state — it does not verify the transaction type is `DeployAccount`, nor that the submitter of the pending deploy-account is the same party submitting the invoke. [2](#0-1) 

**Effect on the validation pipeline**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate = true`: [3](#0-2) 

This sets `validate: false` in `ExecutionFlags`. Inside `StatefulValidator::perform_validations`, the early-return path is taken before `__validate__` is ever called: [4](#0-3) 

Neither the stateless validator nor `validate_by_mempool` performs any cryptographic signature check — the stateless validator only checks signature *length*, and `validate_tx` in the mempool checks only for duplicate hashes and nonce ordering: [5](#0-4) [6](#0-5) 

The result: an Invoke transaction with a completely arbitrary signature is admitted to the mempool without any signature verification.

**Execution-time consequence**

When the batcher later executes the transaction, `new_for_sequencing` sets `validate: true`: [7](#0-6) 

The `__validate__` entry point is then called and fails (invalid signature → revert). If the account was pre-funded before deployment, the fee for the failed validation is charged to the account for a transaction the account owner never authorized.

---

### Impact Explanation

**Broken invariant**: Every transaction admitted to the mempool must have passed account-level signature verification via `__validate__`. This invariant is violated for Invoke transactions with `nonce=1` whenever any transaction from the same sender address exists in the mempool.

**Concrete impacts**:
- The gateway/mempool admits transactions with invalid signatures — matching the "High: Mempool/gateway/RPC admission accepts invalid transactions before sequencing" criterion.
- If the target account is pre-funded (a common pattern: fund the counterfactual address before deploying), the attacker causes the account to pay fees for an unauthorized transaction upon deployment.
- An attacker can flood the mempool with signature-invalid transactions for every address that has a pending deploy-account, consuming mempool capacity and block space.

---

### Likelihood Explanation

The trigger is entirely unprivileged and requires only:
1. Observing any pending deploy-account transaction in the mempool (public information in a distributed sequencer).
2. Submitting an Invoke with `nonce=1` for the same address with an arbitrary signature.

No special role, key, or prior relationship with the victim is required. The condition (`nonce=1`, on-chain nonce `0`, address in pool) is a normal, expected state during the deploy-account + invoke UX flow, making it reliably triggerable.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction for the sender address is present in the pool. Alternatively, add a separate mempool API `deploy_account_in_pool(address) -> bool` that only returns `true` when the pending transaction for that address is of type `DeployAccount`. This ensures the skip-validation path is only taken when the legitimate account owner submitted the deploy-account, not when an arbitrary third party exploits the presence of someone else's deploy-account.

---

### Proof of Concept

**Setup**: Victim V submits `deploy_account` for address `A` (class hash `C`, salt `S`). The transaction enters the mempool. On-chain nonce for `A` is `0`.

**Attack**:
```
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "sender_address": A,          // victim's counterfactual address
  "nonce": "0x1",               // exactly 1
  "calldata": [...],            // arbitrary
  "signature": ["0xdead", "0xbeef"],  // completely invalid
  "resource_bounds": { ... },   // valid bounds
  ...
}
```

**Gateway path**:
1. `StatelessTransactionValidator::validate` — passes (signature length ≤ max, resource bounds valid).
2. `get_nonce_from_state(A)` → `Nonce(0)` (account not deployed).
3. `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
4. `validate_by_mempool` → checks duplicate hash and nonce ordering → passes.
5. `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` (victim's deploy-account is in pool) → **returns `true`**.
6. `run_validate_entry_point(skip_validate=true)` → `validate=false` → `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`.
7. Transaction is admitted to the mempool. ✓

**Execution path** (batcher):
- `new_for_sequencing` sets `validate=true`.
- `__validate__` is called with the invalid signature → fails → transaction reverts.
- If `A` was pre-funded, fee is charged for the unauthorized transaction. [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
