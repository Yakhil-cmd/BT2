### Title
Ethereum Transaction Validation Returns `internal_error!` Instead of Validation Error for Oversized Gas Limit — (`basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`)

---

### Summary

In the Ethereum transaction validation path, when a transaction's `gas_limit` is large enough that `gas_limit * ERGS_PER_GAS >= u64::MAX`, the code returns an `internal_error!` (a fatal block-level error) instead of a proper `InvalidTransaction` validation error. Because this overflow check executes **before** the block-gas-limit check, any unprivileged user can submit a single Ethereum transaction with `gas_limit = u64::MAX` to trigger a fatal internal error in block processing rather than a clean per-transaction rejection.

---

### Finding Description

`MAX_BLOCK_GAS_LIMIT` is defined as `u64::MAX / ERGS_PER_GAS` (= `u64::MAX / 256`): [1](#0-0) 

In the **ZK transaction validation path**, the overflow guard correctly returns a `Validation`-class error: [2](#0-1) 

In the **Ethereum transaction validation path**, the identical guard instead returns an `internal_error!`: [3](#0-2) 

The block-gas-limit check that would otherwise catch the oversized value comes **after** this overflow guard: [4](#0-3) 

So for any `tx_gas_limit > MAX_BLOCK_GAS_LIMIT` (e.g., `u64::MAX`):

1. `u64::MAX.saturating_mul(256) == u64::MAX`, which is **not** `< u64::MAX`.
2. The overflow guard fires first and returns `TxError::Internal(InternalError(...))`.
3. The block-gas-limit check at line 129 is never reached.

By contrast, the ZK metadata initializer explicitly validates both the block and per-transaction gas limits against `MAX_BLOCK_GAS_LIMIT` / `MAX_TX_GAS_LIMIT` at block-init time: [5](#0-4) 

The Ethereum metadata initializer only checks the block-level gas limit, not the per-transaction limit: [6](#0-5) 

The Ethereum block header's `individual_tx_gas_limit()` simply delegates to `block_gas_limit()`, so the per-transaction limit is bounded only indirectly: [7](#0-6) 

---

### Impact Explanation

`TxError::Internal` is a fatal error class. In the ZK block-processing loop, internal errors abort block execution rather than skipping the offending transaction: [8](#0-7) 

The Ethereum block flow follows the same error-propagation contract. A single transaction with `gas_limit = u64::MAX` therefore causes the entire block to fail rather than being cleanly rejected as an invalid transaction. This is a **state-transition / forward-execution DoS**: the sequencer cannot finalize a block that contains such a transaction, and the prover would diverge from forward execution if the error is handled differently across modes.

---

### Likelihood Explanation

The `gas_limit` field of an RLP-encoded Ethereum transaction is a user-supplied `u64` with no pre-parse bound check. Any unprivileged sender can craft a legacy, EIP-1559, or EIP-4844 transaction with `gas_limit = u64::MAX` and submit it to the sequencer. No special privilege, key, or oracle influence is required. [9](#0-8) 

---

### Recommendation

Replace the `internal_error!` in the Ethereum validation path with the same `InvalidTransaction` variant used in the ZK path:

```rust
// ethereum/validation_impl.rs  (line 116-120)
require!(
    tx_gas_limit.saturating_mul(ERGS_PER_GAS) < u64::MAX,
-   internal_error!("TX gas limit overflows ergs counter"),
+   InvalidTransaction::CallerGasLimitTooHigh,   // matches ZK path
    system
)?;
```

Additionally, add an explicit `MAX_TX_GAS_LIMIT` guard in `EthereumMetadataOp::metadata_op` (mirroring the ZK metadata initializer) so that an oversized block-level gas limit is caught at block-init time before any transaction is processed.

---

### Proof of Concept

1. Construct an EIP-1559 transaction with `gas_limit = u64::MAX` and a valid signature.
2. Submit it to a ZKsync OS node running in Ethereum-block mode.
3. `validate_and_compute_fee_for_transaction` evaluates `u64::MAX.saturating_mul(256) == u64::MAX`, which is not `< u64::MAX`.
4. The `require!` macro returns `Err(TxError::Internal(InternalError("TX gas limit overflows ergs counter")))`.
5. The Ethereum block flow propagates this fatal error, aborting block finalization.
6. The ZK path, processing the same transaction, would return `Err(TxError::Validation(InvalidTransaction::CallerGasLimitTooHigh))` and simply skip the transaction — demonstrating the forward/proving divergence and the DoS impact on the Ethereum execution path.
