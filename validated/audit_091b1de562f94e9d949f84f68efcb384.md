### Title
Missing Access Control on `update_rewards()` Allows Anyone to Permanently Block Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary

The `update_rewards()` function in the `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged caller can invoke it with `disable_rewards: true` to advance the global `last_reward_block` checkpoint without distributing any rewards, permanently consuming the one-time-per-block reward slot and preventing the legitimate sequencer from ever distributing rewards for that block.

### Finding Description

The spec for `update_rewards` explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

However, the implementation only calls `general_prerequisites()`: [2](#0-1) 

`general_prerequisites()` only checks that the contract is unpaused and the caller is non-zero — there is no sequencer identity check: [3](#0-2) 

After validating that the provided `staker_address` is active and has a non-zero balance, the function unconditionally writes the current block number to the global `last_reward_block` storage slot: [4](#0-3) 

The `disable_rewards` early-return check comes **after** this write. So calling `update_rewards(valid_staker, disable_rewards: true)` marks the block as processed while distributing zero rewards.

Because `last_reward_block` is a single global value (not per-staker), one such call per block poisons the slot for **all** stakers: [5](#0-4) 

Any subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker calling `update_rewards(any_active_staker, disable_rewards: true)` once per block prevents every staker from accruing consensus-phase block rewards for that block. Repeated across all blocks, this permanently zeroes out all consensus reward accrual for the entire protocol. The lost rewards are not recoverable; `last_reward_block` can only advance, never retreat.

### Likelihood Explanation

**High.** The attacker needs only:
1. A valid, active staker address (trivially obtained from on-chain events).
2. To submit a transaction before the sequencer's reward-update transaction in each block.

No privileged access, no capital, and no signature from any other party is required. The call is cheap (a single storage write after a few reads).

### Recommendation

Add an explicit sequencer-identity check at the top of `update_rewards`, analogous to the existing `assert_caller_is_attestation_contract` pattern:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    ...
```

Store the sequencer address in contract storage (set at construction or by governance) and assert `get_caller_address() == self.sequencer_address.read()`.

Alternatively, move the `last_reward_block.write(...)` to **after** the `disable_rewards` guard so that a no-op call does not consume the block slot.

### Proof of Concept

```
1. Staker S is active with non-zero balance.
2. Attacker A calls:
       staking.update_rewards(staker_address: S, disable_rewards: true)
   in block N.
   → last_reward_block is set to N.
   → No rewards are distributed (early return).
3. Sequencer calls:
       staking.update_rewards(staker_address: S, disable_rewards: false)
   in the same block N.
   → Reverts: REWARDS_ALREADY_UPDATED.
4. All stakers lose block-N consensus rewards permanently.
5. Attacker repeats step 2 in every subsequent block.
```

The flow-test suite already demonstrates that a second call in the same block reverts with `REWARDS_ALREADY_UPDATED`, confirming the one-shot nature of the slot: [6](#0-5)
