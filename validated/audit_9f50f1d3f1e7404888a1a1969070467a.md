### Title
Unverified Participant Contributions in CKD Protocol Allow Malicious Participant to Silently Corrupt the Coordinator's Confidential Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

`do_ckd_coordinator` blindly accumulates `CKDOutput` values received from participants without any cryptographic verification that each participant used their correct key share. A single malicious participant can send arbitrary group elements, silently corrupting the final confidential key that the coordinator accepts as valid — directly analogous to the donation-attack pattern of using unverified external values instead of internal accounting.

---

### Finding Description

The external report's root cause is that `TermMaxMarket.provideLiquidity` uses manipulatable external values (`balanceOf`, `totalSupply`) instead of internal accounting, allowing an attacker to corrupt a calculation that honest parties then accept as correct. The analog here is that `do_ckd_coordinator` uses participant-supplied group elements without any proof of correctness, allowing a malicious participant to corrupt the aggregated CKD output that the coordinator accepts as valid.

**Correct per-participant computation** (in `compute_signature_share`, lines 148–182):

```
big_y  = λᵢ · yᵢ · G
big_c  = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · A)
```

where `xᵢ` is the participant's secret key share and `yᵢ` is a fresh random scalar.

**Coordinator aggregation** (`do_ckd_coordinator`, lines 50–55):

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is **no zero-knowledge proof, DLEQ proof, or any other verification** that the received `big_y` and `big_c` were computed from the participant's actual key share `xᵢ`. The coordinator simply sums whatever values arrive over the channel.

A malicious participant `P_m` can deviate from the protocol and send any pair `(big_y', big_c')` — for example `(0, 0)`, or values chosen to steer the aggregate toward a target — and the coordinator has no mechanism to detect or reject the substitution. The corrupted aggregate is then returned as `CKDOutput` and passed to the application's `unmask(app_sk)` call, producing an incorrect confidential key that the application accepts as legitimate.

This is structurally identical to the donation attack: just as `TermMaxMarket` trusts `balanceOf` (an externally manipulable value) instead of maintaining internal accounting, `do_ckd_coordinator` trusts participant-supplied group elements (externally manipulable values) instead of requiring cryptographic proofs of correct computation.

The `compute_signature_share` helper that honest participants call is entirely local and unreachable by the coordinator for verification purposes. [2](#0-1) 

---

### Impact Explanation

**High.** A single malicious participant causes the coordinator to accept a corrupted `CKDOutput`. The application then calls `unmask(app_sk)` on this corrupted output and obtains an incorrect confidential derived key, which it treats as valid. Honest parties have no way to detect the corruption. This matches the allowed impact: *"Corruption of CKD outputs so honest parties accept inconsistent… unusable cryptographic outputs."*

---

### Likelihood Explanation

**Medium.** Any participant enrolled in the CKD session can trigger this by simply sending arbitrary bytes in place of their correct contribution. No special privilege, leaked key, or external assumption is required — only participation in the protocol. The CKD protocol is an n-of-n aggregation scheme, so every participant is a reachable attacker-controlled entry point.

---

### Recommendation

Each participant should accompany their `(big_y, big_c)` contribution with a zero-knowledge proof of correct formation — for example a DLEQ proof demonstrating that `big_c - big_y_scaled` lies on the correct coset determined by the participant's public key share. The coordinator must verify all such proofs before accumulating contributions, replacing blind trust in external values with cryptographically enforced internal accounting.

---

### Proof of Concept

1. A threshold CKD session is initiated with participants `{P_1, P_2, P_3}` and coordinator `P_1`.
2. Honest participants `P_2` and `P_3` compute and send their correct `(big_y, big_c)` pairs.
3. Malicious participant `P_m` (any enrolled participant) sends `big_y = G·0 = identity` and `big_c = identity` (or any arbitrary elements) instead of their correct contribution.
4. `do_ckd_coordinator` at lines 50–55 adds these values to `norm_big_y` and `norm_big_c` without verification.
5. The returned `CKDOutput` is `(Σ_{j≠m} λⱼ·Yⱼ, Σ_{j≠m} λⱼ·Cⱼ)` — missing the malicious participant's legitimate share.
6. The application calls `ckd_output.unmask(app_sk)` and obtains a confidential key that does not equal `msk · H(pk ‖ app_id)`, silently accepting an unusable or attacker-influenced value as the derived secret. [3](#0-2)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
