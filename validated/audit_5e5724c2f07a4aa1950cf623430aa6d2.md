### Title
Incorrect NFT Net Proceeds Calculation Misleads Sellers on Offer Confirmation Screen - (File: packages/gui/src/components/offers/utils.ts)

### Summary
The `calculateNFTRoyalties` function in `packages/gui/src/components/offers/utils.ts` always returns `nftSellerNetAmount` equal to the full asking `amount`, with the correct deduction of royalties deliberately commented out. This causes the "Net Proceeds" figure displayed to an NFT seller reviewing a `TokenForNFT` offer to be overstated by the full royalty amount, misleading the seller into accepting an offer under false financial pretenses.

### Finding Description
In `calculateNFTRoyalties`, the `nftSellerNetAmount` field is hardcoded to `amount` (the full asking price):

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The commented-out block is the correct calculation. The royalty amount is computed correctly on line 312:

```typescript
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
``` [2](#0-1) 

But it is never subtracted from `nftSellerNetAmount`. The result is that `nftSellerNetAmount` always equals the full asking price regardless of the NFT's royalty percentage.

This incorrect value propagates into two user-facing surfaces:

**1. `NFTOfferViewer.tsx` — offer acceptance screen for imported offers:**

`nftSaleInfo.nftSellerNetAmount` is used to compute `overrideNFTSellerAmount`, which is passed to `NFTOfferSummary` to override the displayed seller amount:

```typescript
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [3](#0-2) 

The "Net Proceeds" label then renders this incorrect value prominently: [4](#0-3) 

**2. `NFTOfferEditor.tsx` — offer creation screen:**

The "They will receive" label (shown to the offer creator when in `TokenForNFT` mode) also renders `nftSellerNetAmount`: [5](#0-4) 

### Impact Explanation
When a seller views an imported `TokenForNFT` offer for an NFT with royalties, the GUI displays "Net Proceeds: X" where X equals the full asking price. The seller actually receives `X − royalty_amount` (royalties are enforced on-chain and deducted from the payment). For an NFT with a 30% royalty and a 10 XCH offer, the GUI shows "Net Proceeds: 10 XCH" while the seller actually receives 7 XCH. The seller approves the offer based on a materially incorrect amount — a direct financial loss caused by the GUI displaying the wrong amount at the offer confirmation step.

This matches the allowed High impact: *"display the wrong asset, identity, amount, destination, or status that causes a user to approve … the wrong … amount."*

### Likelihood Explanation
The bug triggers for every NFT that has a non-zero `royaltyPercentage` when viewed in a `TokenForNFT` offer context. No special attacker setup is required beyond the normal existence of royalty-bearing NFTs. An NFT creator who minted an NFT with a high royalty percentage (e.g., 30%) and subsequently sold it benefits each time the current owner accepts a `TokenForNFT` offer, because the royalty flows to the creator. The seller is systematically shown the wrong net amount on every such offer.

### Recommendation
Uncomment and restore the correct `nftSellerNetAmount` calculation in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

Apply this only for the `TokenForNFT` exchange type (for `NFTForToken`, the seller receives the full asking price because the royalty is paid by the taker on top of the asking price). Verify the corrected value is propagated correctly through `overrideNFTSellerAmount` in `NFTOfferViewer.tsx`.

### Proof of Concept
1. Mint an NFT with `royaltyPercentage = 3000` (30%).
2. Transfer the NFT to a second wallet (the "seller").
3. From a third wallet (the "buyer"), create a `TokenForNFT` offer: offer 10 XCH for the NFT.
4. Import the offer file into the seller's Chia GUI.
5. On the offer confirmation screen, observe "Net Proceeds: 10 XCH" displayed under the `TokenForNFT` path in `NFTOfferViewer`.
6. Accept the offer.
7. Observe that the seller's wallet receives 7 XCH (10 − 3 XCH royalty), not 10 XCH as displayed.

The root cause is `nftSellerNetAmount = amount` at [1](#0-0)  with the correct subtraction commented out, surfaced to the user via [4](#0-3) .
