### Title
Missing Participant-Set Validation in OT-Based ECDSA Presign Allows Coordinator to Corrupt Presign Outputs - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

---

### Summary

The `presign` function in `src/ecdsa/ot_based_ecdsa/presign.rs` explicitly omits the check that the presign participant set matches the participant set embedded in the `TriplePub` used as input. A malicious coordinator can supply Beaver triples generated by one participant set to a presign session running with a different participant set. Because each honest participant has no way to detect this mismatch, they will proceed through the presign protocol using arithmetically inconsistent triple shares, producing a corrupted presign output that all honest parties accept as valid.

---

### Finding Description

**Root cause ‚ÄĒ the omitted check:**

The `presign` function validates that the threshold of the supplied triples matches the presign threshold, but it explicitly skips the analogous participant-set check:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [1](#0-0) 

The `TriplePub` struct produced by triple generation already carries the participant set that was used during generation:

```rust
TriplePub {
    big_a,
    big_b,
    big_c,
    participants: participants.clone().into(),
    threshold,
}
``` [2](#0-1) 

So the information needed to perform the check is present in `args.triple0.1.participants` and `args.triple1.1.participants`, but the library never compares it against the `participants` argument passed to `presign`.

**Why the stated justification is wrong:**

The comment says "presumably they need to have been present in order to have shares." This is incorrect. A Beaver triple share `(a_i, b_i, c_i)` is a plain scalar. Participant `P_i` can legitimately hold a share from a triple generation session that used a completely different set of co-participants `{P_i, P_j, P_k}`, and then be handed that share by a malicious coordinator for a presign session running with `{P_i, P_j, P_l}`. The share is structurally valid for `P_i`'s index, so no local check on the share itself will detect the mismatch.

**Exploit flow:**

1. Malicious coordinator runs triple generation with participant set `S‚āĀ = {A, B, D}`, obtaining `TriplePub‚āĀ` (which records `participants = S‚āĀ`) and per-party shares.
2. Coordinator initiates a presign session with participant set `S‚āā = {A, B, C}` (where `C ‚Č† D`).
3. Coordinator distributes to each of `A` and `B` the triple shares from step 1 (which are valid scalars for their indices), and to `C` a share fabricated or taken from yet another session.
4. Each honest party calls `presign(S‚āā, me, args_with_triple_from_S‚āĀ, ...)`. The threshold check passes because `threshold` is the same. The participant-set check is absent.
5. All parties proceed through the presign protocol using triple shares whose polynomial structure is indexed over `S‚āĀ`, not `S‚āā`. The Lagrange interpolation and exponent-interpolation checks inside the presign use the identifiers of `S‚āā`, producing arithmetically inconsistent intermediate values.
6. The presign completes and returns a `PresignOutput` that all honest parties accept. The output is cryptographically corrupted: the implicit nonce `R` and the auxiliary shares `(őĪ_i, ő≤_i, c_i, e_i)` do not satisfy the algebraic relations required for a valid signature.

This is the direct analog of the "future period" bug: just as `getRewardForPeriod` was missing the check `_period <= currentPeriod`, `presign` is missing the check `presign_participants == triple_participants`. In both cases, a missing state-consistency guard allows an attacker to inject outputs from a mismatched context that honest parties accept without complaint.

---

### Impact Explanation

**Impact: High ‚ÄĒ Corruption of presign outputs so honest parties accept inconsistent cryptographic outputs.**

Every honest participant completes the presign protocol and stores a `PresignOutput` they believe is valid. In reality the output is corrupted: the nonce commitment `R` and the secret shares `(őĪ_i, ő≤_i)` are computed over an inconsistent participant set. Any subsequent signing attempt using this presignature will either produce an invalid ECDSA signature (detectable only at verification time, after the presignature has been consumed and must never be reused) or, in the worst case where the coordinator controls the interpolation inputs, produce a signature that leaks nonce material enabling private-key recovery.

The presignature-reuse constraint documented in the library means that even a failed signing attempt permanently invalidates the presignature, constituting a denial of signing for that session.

---

### Likelihood Explanation

**Likelihood: High.**

The malicious coordinator is an explicitly modeled adversary in this library (see `docs/ecdsa/robust_ecdsa/signing.md` and the `coordinator` parameter in `sign`). The attack requires no cryptographic capability: the coordinator simply routes triple shares from one generation session into a presign session with a different participant list. The omission is self-documented in the source code with a comment that provides an incorrect justification, making accidental triggering by a non-malicious but misconfigured coordinator equally plausible.

---

### Recommendation

Add an explicit participant-set equality check in `presign` immediately after the threshold check:

```rust
if participants.as_slice() != args.triple0.1.participants.as_slice()
    || participants.as_slice() != args.triple1.1.participants.as_slice()
{
    return Err(InitializationError::BadParameters(
        "Presign participant set must exactly match the participant set \
         used during triple generation".to_string(),
    ));
}
```

This mirrors the existing threshold check and closes the gap. The `TriplePub.participants` field already carries the required information; it only needs to be compared.

        args.triple0 = (shares‚āĀ[party], pub‚āĀ)   ‚Üź pub‚āĀ records S‚āĀ, not S‚āā
        args.threshold = 3

        ‚Üí threshold check: 3 == pub‚āĀ.threshold  ‚úď  (passes)
        ‚Üí participant-set check: ABSENT           ‚úó  (not performed)
        ‚Üí protocol proceeds with S‚āā identifiers but S‚āĀ triple shares

        The presignature is consumed and cannot be reused (per library invariant),
