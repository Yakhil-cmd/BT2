### Title
Missing Storage Collision Check in CREATE/CREATE2 Allows Deployment to Addresses with Pre-existing Attacker-Controlled Storage - (File: `evm_interpreter/src/ee_trait_impl.rs`)

### Summary
ZKsync OS's EVM interpreter explicitly skips the storage-emptiness check during CREATE/CREATE2 deployment. The EVM specification (confirmed by the developers' own comment and disabled test cases) requires that deployment fail if the target address has non-empty storage. An attacker can pre-populate storage at a deterministic CREATE2 address by deploying a contract, writing to storage, then self-destructing — leaving storage intact but clearing code and nonce. A subsequent CREATE2 deployment to that address succeeds on ZKsync OS (inheriting attacker-controlled storage) when it should fail, producing an EVM semantic mismatch and corrupted initial contract state.

### Finding Description

In `before_executing_frame` in `evm_interpreter/src/ee_trait_impl.rs`, the deployment collision check reads:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // burn all gas, emit CreateCollision, return Ok(false)
}
``` [1](#0-0) 

The guard checks only `deployee_code_len != 0 || deployee_nonce != 0`. The storage-emptiness condition is acknowledged in the comment but is entirely absent from the predicate. This is independently confirmed by the test index files, which disable the relevant Ethereum test vectors with explicit comments:

- `create2collisionStorageParis.json` → `enabled: false`, comment: `"We do not check storage for collision"`
- `RevertInCreateInInitCreate2Paris.json` → `enabled: false`, comment: `"We do not check for storage collisions"` [2](#0-1) [3](#0-2) 

**Exploit path:**

1. Attacker computes the deterministic CREATE2 address `X` for a known deployer, salt, and initcode.
2. Attacker deploys a contract to `X` (using the same deployer/salt/initcode or any other means that lands code at `X`).
3. The deployed contract writes attacker-chosen values to storage slots (e.g., slot 0 = attacker address, slot 1 = `type(uint256).max`).
4. The contract calls `SELFDESTRUCT` (available pre-Cancun / EIP-6780). This clears `code` and `nonce` but **leaves storage intact**.
5. Now `X` has: `code_len == 0`, `nonce == 0`, but non-zero storage.
6. A victim protocol executes CREATE2 with the same deployer/salt/initcode targeting `X`.
7. **On Ethereum (EIP-7610 / Paris spec):** deployment reverts — storage collision detected.
8. **On ZKsync OS:** the check at line 429 passes (`code_len == 0 && nonce == 0`), the constructor executes, and the new contract is deployed **inheriting the attacker's storage values**.

The constructor may initialize some slots (e.g., `owner = msg.sender`), but any slot the constructor does not explicitly write retains the attacker's pre-seeded value. For contracts that assume fresh-zero storage (proxy implementations, upgradeable contracts, contracts using storage for access control bitmaps, fee accumulators, etc.), this produces silently corrupted initial state.

### Impact Explanation

- **EVM semantic mismatch / state-transition bug**: ZKsync OS produces a different post-state than a conformant EVM for the same transaction sequence. The deployed contract exists on ZKsync OS with attacker-controlled storage, while on Ethereum the deployment reverts and no contract exists.
- **Corrupted initial contract state**: Any storage slot not overwritten by the constructor retains the attacker's value. This can silently bypass access-control initialization, corrupt accounting state, or pre-set reentrancy guards to "entered."
- **Integrity failure in the state transition function**: The STF accepts a state that a conformant EVM would reject, breaking the ZK rollup's equivalence guarantee and potentially invalidating proof assumptions about post-deployment storage invariants.

### Likelihood Explanation

- **Attacker preconditions**: No privileged access required. The attacker only needs to submit two ordinary transactions (deploy + selfdestruct) before the victim's CREATE2.
- **Address predictability**: CREATE2 addresses are fully deterministic and publicly computable from deployer address, salt, and initcode hash — all of which are observable on-chain or in mempool.
- **SELFDESTRUCT availability**: Required to clear code/nonce while preserving storage. This is available in Paris (the hardfork referenced by the disabled tests). Post-Cancun (EIP-6780), SELFDESTRUCT only clears the account when called in the same deployment transaction, limiting but not eliminating the attack surface (same-transaction storage seeding is still possible in some patterns).
- **Realistic targets**: Factory patterns, minimal proxy deployments (EIP-1167), and CREATE2-based counterfactual deployments are common and widely used; many assume zero initial storage.

### Recommendation

Implement the storage-emptiness check before executing the constructor frame. Since ZKsync OS cannot currently enumerate all storage slots of an address, the practical fix is to track whether any storage write has ever occurred at the target address (a single boolean flag per address in the account cache) and reject deployment if that flag is set. Alternatively, align with EIP-7610 semantics by querying the storage trie root for the target address and rejecting deployment if it is non-empty. The check must be placed in `before_executing_frame` alongside the existing `deployee_code_len != 0 || deployee_nonce != 0` guard. [4](#0-3) 

### Proof of Concept

**Setup (pre-state):**
- Deployer address: `0xDEAD...`
- Salt: `0x0000...0001`
- Initcode: minimal contract that writes `0xDEAD...` to slot 0 and calls `SELFDESTRUCT`
- Computed CREATE2 address: `X`

**Step 1 — Attacker transaction:**
```
CREATE2(value=0, offset, size, salt=0x01)
  → deploys to X
  → constructor: SSTORE(0, 0xDEAD...) ; SELFDESTRUCT(attacker)
  → result: X has nonce=0, code_len=0, storage[0]=0xDEAD...
```

**Step 2 — Victim transaction (e.g., factory deploying an upgradeable proxy):**
```
CREATE2(value=0, offset, size, salt=0x01)
  → ZKsync OS check: code_len==0 && nonce==0 → PASSES (storage not checked)
  → constructor executes, does NOT write slot 0
  → contract deployed at X with storage[0] = 0xDEAD... (attacker's value)
```

**Expected behavior (Ethereum):** Step 2 reverts with `CreateCollision` because `storage[0] != 0`.

**Actual behavior (ZKsync OS):** Step 2 succeeds. The victim's contract is live at `X` with `storage[0]` pre-set to the attacker's address — silently bypassing any ownership or access-control initialization that relied on slot 0 being zero.
