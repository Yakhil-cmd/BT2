### Title
Destroying a `CadenceOwnedAccount` with Remaining EVM Balance Permanently Locks FLOW Tokens - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` has no `destroy` function that enforces balance withdrawal before destruction. Because COAs are not key-controlled — the Cadence resource is the *sole* means of controlling the EVM address — destroying the resource while the EVM address holds a balance permanently locks those FLOW tokens. The EVM address continues to exist and accept deposits after the COA resource is gone, making any subsequent deposits also permanently irrecoverable.

---

### Finding Description

A `CadenceOwnedAccount` (COA) is a Cadence resource that owns and controls a deterministic EVM address. The EVM address is derived from the resource's UUID at creation time and is registered in the EVM state. Critically, COAs are **not key-controlled**:

> *"Because of this, users are never able to access a key for their account, meaning that they cannot control their COA's address on other EVM blockchains."* [1](#0-0) 

The `CadenceOwnedAccount` resource definition contains no `destroy` function: [2](#0-1) 

In Cadence, a resource without a custom `destroy` function can be destroyed unconditionally with the `destroy` keyword. When this happens, the Cadence resource is removed from storage, but the EVM-side account state (balance, nonce, code) persists in the EVM trie. Since there is no private key and no surviving Cadence resource, the EVM address becomes permanently inaccessible.

The `deposit` function on `CadenceOwnedAccount` is `access(all)`: [3](#0-2) 

This delegates to `EVMAddress.deposit`, which is also publicly callable. After the COA resource is destroyed, any caller can still invoke `EVM.EVMAddress(bytes: destroyedAddressBytes).deposit(from: <-vault)` and those tokens are permanently locked with no recovery path.

The `withdraw` function, which is the only way to recover EVM-side FLOW back to Cadence, requires `Owner | Withdraw` entitlement and operates on the live resource: [4](#0-3) 

Once the resource is destroyed, this function is unreachable. Additionally, `withdraw` enforces `!EVM.isPaused()`, but `destroy` has no such guard — a user can destroy a COA with a non-zero balance even when EVM operations are paused, bypassing the deliberate restriction on fund movement.

The pattern of destroying a funded COA is already present in the test suite, confirming it is accepted by the runtime without error: [5](#0-4) 

---

### Impact Explanation

Any FLOW tokens held at the EVM address of a destroyed COA are permanently locked. No key exists to sign EVM transactions from that address, and no Cadence resource exists to call `withdraw`. The EVM address remains a valid deposit target, so third parties who send FLOW to the address (e.g., based on a published COA address) after the COA is destroyed will also lose their funds with no recourse. This is a direct cross-VM asset loss: FLOW tokens are removed from the Cadence supply accounting but are unspendable on the EVM side.

---

### Likelihood Explanation

Any unprivileged user who creates a COA can trigger this. The entry path is a standard Cadence transaction: `destroy cadenceOwnedAccount`. No special privileges, leaked keys, or node compromise are required. The pattern appears in integration tests without any runtime rejection, meaning the FVM does not prevent it. Users who mismanage their COA lifecycle (e.g., destroy before withdrawing, or destroy a COA whose address was shared publicly) will silently lose funds.

---

### Recommendation

Add a `destroy` function to `CadenceOwnedAccount` that either:

1. **Panics if the EVM balance is non-zero**, forcing the caller to withdraw all funds before destruction, analogous to requiring `withdraw` before `destroy`; or
2. **Automatically withdraws the remaining EVM balance** to a caller-specified Cadence vault before the resource is destroyed.

This mirrors the recommendation in the external report: do not allow destruction of a payment-receiving resource without first draining its balance.

---

### Proof of Concept

The following Cadence transaction, submittable by any unprivileged account, permanently locks FLOW tokens:

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>

transaction(amount: UFix64) {
    prepare(signer: auth(Storage) &Account) {
        // Obtain FLOW tokens (e.g., from signer's vault)
        let vault <- signer.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(from: /storage/flowTokenVault)!
            .withdraw(amount: amount) as! @FlowToken.Vault

        // Create a COA — this registers a new EVM address
        let coa <- EVM.createCadenceOwnedAccount()
        let evmAddr = coa.address() // record the address

        // Deposit FLOW into the COA's EVM address
        coa.deposit(from: <-vault)
        // coa.balance() > 0 at this point

        // Destroy the COA resource — no balance check, no panic
        // The EVM address still exists with a non-zero balance
        destroy coa

        // `evmAddr` still exists in EVM state with `amount` FLOW locked forever.
        // No key exists. No Cadence resource exists. Funds are irrecoverable.
        // Any future deposit to evmAddr is also permanently lost.
    }
}
```

The `createCadenceOwnedAccount` call registers the EVM address via `InternalEVM.createCadenceOwnedAccount`: [6](#0-5) 

After `destroy coa`, the EVM state retains the address and its balance, but the controlling resource is gone. The `withdraw` path is permanently closed.
