### Title
Gateway Stateful Validator Skips Minimum Gas Price Threshold for Legacy `L1Gas` Transactions, Allowing Underpriced Admission — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`validate_tx_l2_gas_price_within_threshold` unconditionally returns `Ok(())` for any transaction carrying `ValidResourceBounds::L1Gas`, bypassing the `min_gas_price_percentage` admission guard entirely. An unprivileged sender can submit a V3 `L1Gas`-bounded transaction with an arbitrarily low (including zero) gas price and have it accepted by the gateway and admitted to the mempool, while an equivalent `AllResources` transaction at the same price would be rejected.

### Finding Description

The stateful gateway validator enforces a minimum gas price threshold to prevent underpriced transactions from entering the mempool. The check is implemented in `validate_tx_l2_gas_price_within_threshold`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
fn validate_tx_l2_gas_price_within_threshold(
    &self,
    tx_resource_bounds: ValidResourceBounds,
    previous_block_l2_gas_price: NonzeroGasPrice,
) -> StatefulTransactionValidatorResult<()> {
    match tx_resource_bounds {
        ValidResourceBounds::AllResources(tx_resource_bounds) => {
            let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
            let gas_price_threshold_multiplier =
                Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
            let threshold = (gas_price_threshold_multiplier
                * previous_block_l2_gas_price.get().0)
                .to_integer();
            if tx_l2_gas_price.0 < threshold {
                return Err(...);
            }
        }
        ValidResourceBounds::L1Gas(_) => {
            // No validation required for legacy transactions.   // <-- SKIPPED
        }
    }
    Ok(())
}
``` [1](#0-0) 

The `ValidResourceBounds::L1Gas` arm silently returns `Ok(())` with no price check. The production default for `min_gas_price_percentage` is `100`, meaning `AllResources` transactions must offer at least 100% of the previous block's L2 gas price. [2](#0-1)

`ValidResourceBounds::L1Gas` is the pre-0.13.3 resource bounds variant, still accepted by the gateway for V3 transactions. It is produced whenever a transaction's `l2_gas` and `l1_data_gas` bounds are both zero during protobuf deserialization:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(...)
})
``` [4](#0-3) 

The call chain that reaches the skipped check is:

`extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `validate_state_preconditions` → `validate_resource_bounds` → `validate_tx_l2_gas_price_within_threshold` [5](#0-4) 

### Impact Explanation

Any unprivileged sender can craft a `ValidResourceBounds::L1Gas` transaction with `max_price_per_unit = 0` (or any value below the configured threshold) and have it admitted by the gateway and inserted into the mempool. The gateway's admission decision is the wrong value: **accepted** when the invariant requires **rejected**. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The downstream blockifier `check_fee_bounds` will eventually reject execution if the price is below the current block's actual gas price, but that rejection happens only at sequencing time, after the transaction has already consumed mempool capacity and gateway resources. [6](#0-5) 

### Likelihood Explanation

- The `L1Gas` variant is still a valid, accepted transaction type in the current codebase.
- No privilege is required; any wallet or script can construct such a transaction.
- The bypass is unconditional — the `match` arm always returns `Ok(())` regardless of the configured `min_gas_price_percentage`.
- The stateless validator has no equivalent L1 gas price floor check either, so no earlier guard catches this. [7](#0-6) 

### Recommendation

Apply the same percentage-threshold check to `ValidResourceBounds::L1Gas` transactions, comparing `l1_gas.max_price_per_unit` against `min_gas_price_percentage`% of the previous block's L1 gas price (analogous to how `AllResources` checks the L2 gas price). The existing TODO comment already flags this gap; it should be resolved rather than deferred.

### Proof of Concept

1. Observe that `min_gas_price_percentage = 100` in the production config schema. [8](#0-7) 

2. Construct a V3 Invoke transaction with `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}`, and `l1_gas.max_price_per_unit = 0`. The protobuf deserializer will classify this as `ValidResourceBounds::L1Gas`.

3. Submit to the gateway. `validate_resource_bounds` calls `validate_tx_l2_gas_price_within_threshold` with this `L1Gas` variant. The `L1Gas` arm executes and returns `Ok(())` immediately — no price comparison is performed.

4. The transaction passes `run_pre_validation_checks` and proceeds to `run_validate_entry_point`. The blockifier's `check_fee_bounds` will reject it only if `max_price_per_unit < actual_block_l1_gas_price`, but the gateway has already admitted it to the mempool.

5. An `AllResources` transaction with the same zero L2 gas price would have been rejected at step 3 with `StarknetErrorCode.GAS_PRICE_TOO_LOW`. [9](#0-8)

        &self,

        let resource_bounds = *tx.resource_bounds();

        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
