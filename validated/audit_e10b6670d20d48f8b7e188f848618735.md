### Title
Gateway Stateful Validator Unconditionally Skips `__validate__` (Signature Verification) for Invoke Transactions with Nonce=1 When Any Account Transaction Is in Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway stateful validator unconditionally skips the `__validate__` entry-point call — the sole on-chain signature-verification step — for any Invoke V3 transaction whose nonce is exactly 1 and whose sender account has nonce 0 in state, provided `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for any account that has *any* transaction in the pool (not only a deploy-account), an attacker who first submits a valid deploy-account for a fresh address can immediately submit a second invoke transaction with nonce=1 carrying an **arbitrary, invalid signature** and have it admitted to the mempool without any signature check.

### Finding Description

The gateway stateful-validation path is:

```
extract_state_nonce_and_run_validations
  → run_pre_validation_checks
      → validate_state_preconditions   (nonce range, resource bounds)
      → validate_by_mempool            (duplicate/nonce-gap check)
      → skip_stateful_validations      ← decides whether to skip __validate__
  → run_validate_entry_point(skip_validate)
```

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

`perform_pre_validation_stage` only checks nonce, fee bounds, and proof facts — it does **not** call the account's `__validate__` entry point, which is where ECDSA signature verification happens: [4](#0-3) 

The third condition — `account_tx_in_pool_or_recent_block` — is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

This returns `true` for any account that has *any* transaction in the pool — including the deploy-account the attacker just submitted. It does **not** verify that the pending transaction is specifically a deploy-account, nor does it verify that the deploy-account itself is valid.

### Impact Explanation

An attacker can:

1. Compute the deterministic address `A` that a deploy-account with chosen parameters would produce.
2. Submit a valid `DeployAccount` for address `A` (passes full validation because the account doesn't exist yet and `__validate_deploy__` runs the constructor).
3. Immediately submit an `Invoke` with `nonce=1`, `sender_address=A`, and a **completely invalid signature** (e.g., all zeros).

At step 3, the gateway checks:
- `tx.nonce() == 1` ✓
- `account_nonce == 0` ✓ (account not yet deployed)
- `account_tx_in_pool_or_recent_block(A) == true` ✓ (deploy-account is in pool)

`skip_validate = true` → `__validate__` is never called → the invalid-signature invoke is admitted to the mempool.

The broken invariant is: *every transaction admitted to the mempool has a valid account signature*. The corrupted admission decision is the `Ok(account_nonce)` returned by `extract_state_nonce_and_run_validations` for a transaction whose signature was never verified.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The trigger is fully unprivileged and requires only two sequential RPC calls (`starknet_addDeployAccountTransaction` + `starknet_addInvokeTransaction`). No special knowledge, privileged access, or race condition is required. The attacker pays only for the deploy-account submission; the invalid invoke costs nothing because it will be rejected at execution time without fee deduction (the account has no balance and `__validate__` will panic). This makes the attack essentially free to repeat at scale, enabling mempool pollution and potential DoS.

### Recommendation

The `skip_stateful_validations` bypass should be narrowed so that it only activates when the pending mempool transaction for the account is **specifically a deploy-account**, not any arbitrary transaction. Additionally, the bypass should be removed entirely from the gateway path and instead handled at the batcher level where the deploy-account execution can be confirmed before the invoke is processed. At minimum, the condition should verify the transaction type of the pending mempool entry:

```rust
// Verify the pending transaction is a DeployAccount, not just any transaction.
let pending_tx = mempool_client.get_account_pending_tx(tx.sender_address()).await?;
let is_deploy_account = matches!(pending_tx, Some(InternalRpcTransactionWithoutTxHash::DeployAccount(_)));
return Ok(is_deploy_account);
```

### Proof of Concept

```

2. Compute address A = calculate_contract_address(S, C, [], 0x0).
3. POST /gateway/add_transaction:

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
