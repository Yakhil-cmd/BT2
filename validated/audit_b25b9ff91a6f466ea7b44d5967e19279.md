### Title
Missing Length Validation on Participant-Provided Vectors Causes Panic in Triple Generation — (`File: src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

In `do_generation_many`, participant-provided deserialized vectors are indexed directly with `[i]` (where `i` iterates `0..N`, a compile-time constant) without first asserting that the received vector has at least `N` elements. A single malicious participant can send a short vector, triggering an index-out-of-bounds panic in every honest party's process, aborting the triple generation session and permanently blocking OT-based ECDSA signing.

---

### Finding Description

`generate_triple_many` calls `do_generation_many<const N: usize>`, which runs a multi-round protocol collecting vectors from each peer. At three distinct points the code blindly indexes into a peer-supplied `Vec` using the compile-time constant `N` as the upper bound, with no prior length check:

**Point 1 — Round 1 commitment collection** [1](#0-0) 

```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]); // panics if commitments.len() < N
}
```

**Point 2 — Round 3 polynomial commitment message** [2](#0-1) 

```rust
for i in 0..N {
    let their_big_e = &their.big_e_v[i];   // panics if big_e_v.len() < N
    let their_big_f = &their.big_f_v[i];   // panics if big_f_v.len() < N
    let their_big_l = &their.big_l_v[i];   // panics if big_l_v.len() < N
    let their_randomizer = &their.randomizer_v[i];
    let their_phi_proof0 = &their.phi_proof0_v[i];
    let their_phi_proof1 = &their.phi_proof1_v[i];
```

**Point 3 — Round 3 private share collection** [3](#0-2) 

```rust
for (_, (a_j_i_v, b_j_i_v)) in recv_from_others::<(Vec<_>, Vec<_>)>(...).await? {
    for i in 0..N {
        let a_j_i = &a_j_i_v[i]; // panics if a_j_i_v.len() < N
        let b_j_i = &b_j_i_v[i]; // panics if b_j_i_v.len() < N
```

The message deserialization (`rmp_serde`) will happily decode a `Vec` of any length. The compile-time constant `N` is local to the honest party; the sender is free to serialize a vector of length 0 or any value less than `N`. No guard exists between deserialization and the first `[i]` access.

The `PolynomialCommitmentsMessageMany` struct that carries the Round-3 vectors is defined with plain `Vec` fields, imposing no length invariant at the type level: [4](#0-3) 

---

### Impact Explanation

`generate_triple_many` is the batch path for producing Beaver triples, which are a prerequisite for every OT-based ECDSA presignature and therefore every OT-based ECDSA signature. [5](#0-4) 

A Rust index-out-of-bounds panic in an async future propagates as an unrecoverable abort of that protocol instance. Every honest party running the same session receives the malicious message and panics independently. The attacker can repeat this against every new session attempt, permanently denying triple generation and therefore signing for the honest set.

**Impact classification: High — Permanent denial of signing for honest parties.**

---

### Likelihood Explanation

Any participant in a `generate_triple_many` session can craft the malicious message. No privileged access, leaked key, or cryptographic break is required — only the ability to send a protocol message with a shorter-than-expected vector, which is trivially achievable by any participant who controls their own serialization. The attacker needs only to be one of the `N` participants invited to the session.

---

### Recommendation

Before the first `[i]` access, assert that every received vector has exactly `N` elements and return a `ProtocolError` on mismatch instead of panicking:

```rust
// Point 1
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(
        format!("commitments from {from:?} has wrong length: expected {N}, got {}", commitments.len())
    ));
}

// Point 2
if their.big_e_v.len() != N || their.big_f_v.len() != N
    || their.big_l_v.len() != N || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(
        format!("polynomial commitment message from {from:?} has wrong vector lengths")
    ));
}

// Point 3
if a_j_i_v.len() != N || b_j_i_v.len() != N {
    return Err(ProtocolError::AssertionFailed(
        format!("private share vectors have wrong length: expected {N}")
    ));
}
```

Alternatively, use `.get(i).ok_or(ProtocolError::...)` at each access site to convert the potential panic into a recoverable error.

---

### Proof of Concept

A malicious participant `P_bad` in a `generate_triple_many::<4>` session (N=4) sends the Round-1 commitment message as an empty `Vec` (`[]`) instead of a `Vec` of length 4. Every honest party deserializes this successfully (it is valid MessagePack), then executes:

```rust
for i in 0..4 {
    all_commitments_vec[i].put(from, commitments[i]); // i=0: index 0 on empty Vec → panic
}
```

The process panics at `i=0` with `index out of bounds: the len is 0 but the index is 0`. The same attack applies at Round 3 by sending `PolynomialCommitmentsMessageMany` with any of its six `Vec` fields having fewer than `N` elements, or at the private-share round by sending `(Vec::new(), Vec::new())`.

/// As [`generate_triple`] but for many triples at once
pub fn generate_triple_many<const N: usize>(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    rng: impl CryptoRngCore + Send + 'static,

    let (participants, threshold) = validate_triple_inputs(participants, threshold)?;
    let ctx = Comms::new();
    let fut = do_generation_many::<N>(ctx.clone(), participants, me, threshold, rng);
    Ok(make_protocol(ctx, fut))
}
