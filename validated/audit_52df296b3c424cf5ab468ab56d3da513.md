### Title
Unprotected Pre-EIP-155 Legacy Transactions Accepted Without Chain ID Binding Enable Cross-Chain Signature Replay - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS explicitly accepts pre-EIP-155 legacy transactions (those with `v == 27` or `v == 28`). For these transactions, the signing hash is computed over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` — with no chain ID included. This makes the signature chain-agnostic, allowing any pre-EIP-155 transaction signed for Ethereum mainnet (or any other EVM chain) to be replayed on ZKsync OS, and vice versa, as long as the sender's nonce matches.

---

The `is_eip155()` check is:

```rust
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
``` [2](#0-1) 

When `v == 27` or `v == 28`, the signing hash covers only `[nonce, gasPrice, gasLimit, to, value, data]` — no chain ID. The transaction is then stored as `RlpEncodedTxInner::Legacy` (unprotected variant) and proceeds through full signature validation and execution:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    Self::Legacy(tx, sig_data)
};
``` [3](#0-2) 

All typed transactions (EIP-2930, EIP-1559, EIP-4844, EIP-7702) correctly enforce chain ID:

```rust
if tx.chain_id != expected_chain_id {
    return Err(InvalidTransaction::InvalidChainId.into());
}
``` [4](#0-3) 

But the unprotected legacy path has no such check. The `BasicBootloaderProvingExecutionConfig` (the production proving path) has `VALIDATE_EOA_SIGNATURE: true`, so the signature is verified — but it is verified against a hash that contains no chain ID, making the verification chain-agnostic. [5](#0-4) 

---

### Impact Explanation

An attacker who observes a pre-EIP-155 legacy transaction broadcast on Ethereum mainnet (or any other EVM chain) can replay it on ZKsync OS if:

1. The sender's nonce on ZKsync OS equals the transaction's nonce field.
2. The sender holds sufficient balance on ZKsync OS.

The result is an unauthorized fund transfer or contract call executed on ZKsync OS without the sender's knowledge or consent. Because ZKsync OS is a new chain, users who signed transactions on Ethereum years ago (before EIP-155 was universally enforced) may have matching nonces and balances on ZKsync OS, making this a realistic loss-of-funds path.

---

### Likelihood Explanation

Pre-EIP-155 transactions (v=27/28) are still produced by:
- Older hardware wallets in legacy mode
- Some DeFi tooling and scripts that do not enforce EIP-155
- Users who manually construct transactions

ZKsync OS is a new chain. A user whose Ethereum nonce is, say, 3 may also have nonce 3 on ZKsync OS (if they have not yet transacted there). Any pre-EIP-155 Ethereum transaction at nonce 3 from that user is immediately replayable on ZKsync OS. The attacker only needs to monitor the Ethereum mempool or historical transactions for pre-EIP-155 signed data.

---

### Recommendation

Reject legacy transactions with `v == 27` or `v == 28` at the parsing stage. ZKsync OS is a new chain with no legacy pre-EIP-155 user base to support. Only EIP-155 protected legacy transactions and typed transactions (EIP-2930, EIP-1559, EIP-4844, EIP-7702) — all of which bind the signature to the chain ID — should be accepted. The fix is a single additional check in `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`:

```rust
if legacy_signature.is_eip155() == false {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

---

### Proof of Concept

1. Alice signs a legacy ETH transfer on Ethereum mainnet with `v = 27` (no EIP-155): `[nonce=5, gasPrice=X, gasLimit=21000, to=Bob, value=1 ETH, data=0x]`.
2. Alice has nonce `5` on ZKsync OS (she has never transacted there) and holds ETH on ZKsync OS.
3. Attacker copies the raw RLP-encoded transaction bytes from the Ethereum mempool/history.
4. Attacker submits the identical bytes to ZKsync OS.
5. ZKsync OS parses the transaction: `v == 27` → `is_eip155() == false` → signing hash computed without chain ID → signature verifies correctly against Alice's public key.
6. Nonce check passes (Alice's ZKsync OS nonce is 5).
7. Transaction executes: 1 ETH is transferred from Alice's ZKsync OS balance to Bob without Alice's consent. [6](#0-5) [7](#0-6)
