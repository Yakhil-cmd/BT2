### Title
`_afterTimelock` uint32 truncation allows pool admin to bypass OracleValueStopLoss timelocks — (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

---

### Summary

The `_afterTimelock` helper in `OracleValueStopLossExtension` computes the execution deadline for every pending admin change by casting `block.timestamp + timelock` to `uint32`. Because `timelock` is a `uint32` with no upper-bound validation, a pool admin can set it to `type(uint32).max`, causing the sum to silently truncate to a timestamp already in the past. Every subsequent proposal (drawdown, decay, watermarks) then has an `executeAfter` that is already elapsed, so the admin can execute all changes instantly — completely bypassing the timelock that is the sole LP-protection mechanism.

---

### Finding Description

`_afterTimelock` is the single function that computes the execution deadline for all four pending-change flows:

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}

`oracleStopLossConfig[pool_].timelock` is a `uint32` field. Neither `initialize` nor `proposeOracleStopLossTimelock` validates its upper bound:

In Solidity ≥ 0.8, an explicit narrowing cast (`uint32(...)`) silently truncates — it does **not** revert. When `timelock = type(uint32).max = 4,294,967,295`:

```
block.timestamp (≈ 1,700,000,000)
+ type(uint32).max  (  4,294,967,295)
= 5,994,967,295

uint32(5,994,967,295) = 5,994,967,295 mod 4,294,967,296
                      = 1,699,999,999   ← 1 second in the past
```

`_requireElapsed` then passes immediately because `block.timestamp (1,700,000,000) < 1,699,999,999` is `false`:

```solidity
// line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [4](#0-3) 

Once `timelock = type(uint32).max` is stored, every call to `_afterTimelock` returns a past timestamp, so **all four proposal/execute pairs** (timelock, drawdown, decay, watermarks) can be executed in the same block with zero waiting.

---

### Impact Explanation

The OracleValueStopLossExtension is the mechanism that protects LPs by blocking swaps when per-share value drops below a watermark. The timelock is the only constraint on the pool admin's ability to weaken or disable that protection. Once bypassed, the admin can:

- Set `drawdownE6 = E6` (100 % drawdown) — stop-loss never triggers.
- Set `decayPerSecondE8 = E8` — watermarks decay to zero in one second, disabling all future checks.
- Set per-bin watermarks to zero — stop-loss disabled for targeted bins.

With stop-loss disabled, swaps that would have been reverted (e.g., during a price-manipulation attack or toxic-flow episode) proceed normally, draining LP value. This is a direct admin-boundary break: the pool admin exceeds the trust boundary defined by the timelock.

---

### Likelihood Explanation

The attack requires the admin to:
1. Propose `timelock = type(uint32).max` and wait for the **current** timelock to elapse.
2. Execute the new timelock.
3. From that point on, all proposals execute instantly.

If the initial timelock is short (or zero, which is allowed — no minimum is enforced), step 1 costs nothing. Even with a 7-day initial timelock, the admin can execute the bypass after one week and then act instantly on all subsequent changes. The overflow arithmetic is deterministic and requires no special on-chain conditions.

---

### Recommendation

Add an explicit upper-bound check on `timelock` in both `initialize` and `proposeOracleStopLossTimelock`, for example:

```solidity
uint256 private constant MAX_TIMELOCK = 365 days;

function _validateTimelock(uint256 t) private pure {
    if (t > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(t);
}
```

Alternatively, rewrite `_afterTimelock` to revert on overflow rather than truncate:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    require(result <= type(uint32).max, "Timelock overflow");
    return uint32(result);
}
```
