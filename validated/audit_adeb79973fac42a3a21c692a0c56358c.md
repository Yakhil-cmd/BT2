### Title
Inconsistent L2 Gas Price Lower Bounds Between Stateless and Stateful Gateway Admission — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs` and `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway enforces two separate, structurally inconsistent lower bounds on the L2 gas price of an incoming `AllResources` transaction. The stateless validator applies a fixed absolute floor (`min_gas_price`, default 8 Gwei), while the stateful validator applies a dynamic percentage-of-previous-block-price floor (`min_gas_price_percentage`, default 100%). These two checks are independent, use different reference values, and cover different transaction types. Critically, the stateless check applies only to `l2_gas.max_price_per_unit` and only for `AllResources` transactions, while the stateful check also only covers `AllResources` and silently skips `L1Gas` legacy transactions entirely. A transaction carrying `ValidResourceBounds::L1Gas` bypasses both the absolute floor and the dynamic threshold, and is admitted to the mempool with no L2 gas price validation at all.

### Finding Description

**Path 1 — Stateless validator (`StatelessTransactionValidator::validate_resource_bounds`)**

The stateless validator checks `resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price` against a static configured floor (default `8_000_000_000` fri). [1](#0-0) 

This check operates on the raw `AllResourceBounds` struct extracted from the RPC transaction. It does **not** branch on `ValidResourceBounds::L1Gas` vs `ValidResourceBounds::AllResources`; it reads `tx.resource_bounds()` which always returns `AllResourceBounds`. However, the `min_gas_price` check only tests `l2_gas.max_price_per_unit`. A transaction whose `l2_gas` fields are all zero but whose `l1_gas` fields are nonzero passes the zero-fee guard (because `max_possible_fee > 0`) and also passes the `min_gas_price` check (because `l2_gas.max_price_per_unit == 0 < min_gas_price` would fail — **but** the `min_gas_price` default is `8_000_000_000` and `0 < 8_000_000_000` is true, so this **does** reject zero L2 price when `min_gas_price > 0`). [2](#0-1) 

**Path 2 — Stateful validator (`StatefulTransactionValidator::validate_tx_l2_gas_price_within_threshold`)**

The stateful validator applies a dynamic threshold: `threshold = (min_gas_price_percentage / 100) * previous_block_l2_gas_price`. It explicitly skips `ValidResourceBounds::L1Gas` with the comment "No validation required for legacy transactions." [3](#0-2) 

**The core discrepancy — two independent, non-coordinated bounds:**

The stateless `min_gas_price` is a static absolute value (8 Gwei by default), while the stateful `min_gas_price_percentage` is a fraction of the *previous block's* L2 gas price. These are configured independently with no cross-validation. If the network's actual L2 gas price rises above 8 Gwei (which it will under normal EIP-1559 dynamics), the stateless floor becomes irrelevant and the stateful floor becomes the only guard. Conversely, if the stateful `validate_resource_bounds` flag is set to `false` (e.g., during bootstrap), the stateful check is entirely skipped, leaving only the static 8 Gwei floor. [4](#0-3) 

**The `L1Gas` legacy transaction bypass:**

A transaction submitted as `ValidResourceBounds::L1Gas` (pre-0.13.3 format) is converted from the RPC `AllResourceBounds` struct only when `l2_gas` and `l1_data_gas` are both zero. In that case:
- The stateless `min_gas_price` check fires on `l2_gas.max_price_per_unit == 0`, which is `< min_gas_price` (8 Gwei), so it **would** be rejected — **unless** `min_gas_price` is configured to `0` (which is the test default and a valid operator configuration).
- The stateful check explicitly skips `L1Gas` with no validation. [5](#0-4) 

**The `validate_resource_bounds` shared flag:**

Both validators share a single `validate_resource_bounds` pointer in the node config. When set to `false` (bootstrap mode), **both** checks are disabled simultaneously, meaning any gas price is accepted. [6](#0-5) 

**The structural inconsistency (direct analog to the reported bug):**

- Stateless: enforces `l2_gas.max_price_per_unit >= min_gas_price` (absolute, static, 8 Gwei default).
- Stateful: enforces `l2_gas.max_price_per_unit >= (min_gas_price_percentage/100) * prev_block_price` (relative, dynamic).

These two bounds are not guaranteed to be consistent. An operator can set `min_gas_price_percentage = 50` in the stateful config while `min_gas_price = 8_000_000_000` in the stateless config. If the previous block price is 20 Gwei, the stateful threshold is 10 Gwei, but the stateless threshold is 8 Gwei — the stateless check is the binding constraint. If the previous block price drops to 10 Gwei, the stateful threshold is 5 Gwei, below the stateless floor. There is no enforcement that these two independently configured values remain consistent, and no documentation that they must be. [7](#0-6) [8](#0-7) 

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

A transaction with an L2 gas price below the intended network floor can be admitted to the mempool if:
1. The stateless `min_gas_price` is misconfigured to a value lower than the current network price (e.g., left at 8 Gwei while the network price is 100 Gwei), and
2. The stateful `min_gas_price_percentage` is set below 100% (e.g., 50%), allowing transactions at 50% of the previous block price.

Such a transaction passes both gateway checks but will fail blockifier pre-validation (`check_fee_bounds`) when the batcher attempts to execute it, because `max_price_per_unit < actual_gas_price`. This causes the transaction to be included in a block proposal, consume batcher resources, and then revert — producing a wrong admission decision at the gateway/mempool layer.

Additionally, the `L1Gas` legacy path bypasses the stateful L2 gas price check entirely, meaning legacy transactions face only the static stateless floor, not the dynamic market-based floor.

### Likelihood Explanation

The likelihood is moderate. The default production configuration sets `min_gas_price_percentage = 100` and `min_gas_price = 8_000_000_000`, which are consistent at current network prices. However:
- The two values are independently configurable with no cross-validation or documented relationship.
- During bootstrap (`validate_resource_bounds = false`), both checks are disabled simultaneously.
- The `L1Gas` bypass is unconditional and permanent.
- The stateless `min_gas_price` is noted with a TODO to be removed in favor of versioned constants, suggesting it is a transitional mechanism that may be misconfigured. [9](#0-8) 

### Recommendation

1. **Short term:** Add a startup assertion or config validation that `min_gas_price` (stateless) is consistent with the expected range of `(min_gas_price_percentage / 100) * current_l2_gas_price`. Document the intended relationship between the two bounds explicitly.

2. **Short term:** In `validate_tx_l2_gas_price_within_threshold`, consider whether `L1Gas` legacy transactions should also be subject to a minimum L1 gas price check against the block's L1 gas price, analogous to the L2 check.

3. **Long term:** Consolidate the two gas price lower bounds into a single check with a single source of truth, as noted in the existing TODO comment. Remove `min_gas_price` from `StatelessTransactionValidatorConfig` and derive the floor from versioned constants or the previous block price alone.

### Proof of Concept

1. Configure the gateway with `min_gas_price = 1` (stateless) and `min_gas_price_percentage = 10` (stateful, 10% of previous block price).
2. Assume the previous block L2 gas price is 100 Gwei. The stateful threshold is 10 Gwei.
3. Submit an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = 5 Gwei`.
4. **Stateless check:** `5 Gwei >= 1` → passes.
5. **Stateful check:** `5 Gwei >= 10 Gwei` → fails. Transaction is correctly rejected.

Now change `min_gas_price_percentage = 0`:
4. **Stateless check:** `5 Gwei >= 1` → passes.
5. **Stateful check:** threshold = 0, `5 Gwei >= 0` → passes.
6. Transaction is admitted to the mempool with a gas price far below the network floor.
7. When the batcher executes the transaction, `check_fee_bounds` in `AccountTransaction::perform_pre_validation_stage` computes `actual_gas_price = 100 Gwei > max_price_per_unit = 5 Gwei` and returns `ResourceBoundsError::MaxGasPriceTooLow`, causing the transaction to fail at execution time rather than at admission — a wrong admission decision at the gateway layer. [10](#0-9)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
