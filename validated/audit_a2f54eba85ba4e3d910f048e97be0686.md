### Title
COA Ownership Proof Signature Replay Across Chains and Contexts Due to Missing Chain ID Binding - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies signatures over a caller-supplied `signedData` blob using the fixed domain tag `"FLOW-V0.0-user"`, which does not encode the chain ID or the target EVM address. Any signature produced with that tag for any purpose on any Flow network is a valid COA ownership proof on every other Flow network where the same key exists, and can be replayed against any COA owned by the same Cadence account. The function is `access(all)` and is directly reachable by any unprivileged Cadence transaction or EVM call.

### Finding Description
`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts an arbitrary `signedData: [UInt8]` byte array from the caller and verifies the provided signatures against it using the hardcoded domain separation tag `"FLOW-V0.0-user"`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Two structural defects exist:

1. **No chain ID in the signed message.** The tag `"FLOW-V0.0-user"` contains no network identifier. Flow defines distinct chain IDs (`flow-mainnet`, `flow-testnet`, `flow-previewnet`, etc.), but none of them are mixed into the signed payload or the domain tag. A signature produced on testnet is cryptographically identical to one produced on mainnet for the same key and the same `signedData` bytes.

2. **`signedData` is not required to encode `evmAddress`.** The function accepts the `evmAddress` to prove ownership of as a separate parameter and only checks it against the on-chain COA resource *after* the signature passes. The signature itself is never bound to the EVM address being claimed. The code comment at line 1003–1009 explicitly acknowledges this:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

The function is declared `access(all)`, so any unprivileged Cadence transaction or script can call it directly. It is also reachable via the EVM Cadence Arch precompile `verifyCOAOwnershipProof(address,bytes32,bytes)`, which any EVM transaction can call. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
An attacker who obtains a victim's valid `"FLOW-V0.0-user"` signature over any byte string (e.g., a dApp login challenge, a prior COA proof for a different COA, or a testnet proof) can replay it as a COA ownership proof on mainnet or against a different COA owned by the same Cadence account. A successful replay causes `validateCOAOwnershipProof` to return `ValidationResult(isValid: true, ...)`, falsely asserting that the victim's Cadence account controls the targeted EVM address.

This false assertion is consumed by:
- The EVM Cadence Arch precompile, which returns `true` to the calling EVM contract.
- The COA contract's ERC-1271 `isValidSignature` implementation, which returns `ValidERC1271Signature` to any EVM contract that calls it.

Any EVM protocol that relies on ERC-1271 COA ownership proofs for authorization (e.g., token claims, governance votes, bridge operations) can be bypassed. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
The attack requires the adversary to obtain a victim's signature over an arbitrary byte string using the `"FLOW-V0.0-user"` domain tag. This tag is the standard user-facing signing tag used by Flow wallets for off-chain authentication flows (e.g., dApp logins, WalletConnect challenges). Such signatures are routinely produced and transmitted over the network. An attacker who intercepts or is given such a signature (e.g., from a phishing dApp, a testnet interaction, or a prior COA proof for a different COA) can immediately replay it. No privileged access, staked node, or key compromise is required. [6](#0-5) [7](#0-6) 

### Recommendation
1. **Bind the chain ID into the signed payload.** The `validateCOAOwnershipProof` function (or its callers) should prepend the current chain ID to `signedData` before verification, or require that callers include it. Alternatively, extend the domain separation tag to include the chain ID (e.g., `"FLOW-V0.0-user-mainnet"`).

2. **Bind the EVM address into the signed payload.** Enforce that `signedData` encodes `evmAddress` (e.g., as a structured prefix), and verify this encoding inside `validateCOAOwnershipProof` rather than leaving it as a caller convention. The current comment acknowledges this gap but does not enforce it.

3. **Bind the Cadence account address into the signed payload.** To prevent cross-account replay, the Cadence `address` should also be part of the committed signed data.

A minimal structured signed payload would be: `chainID || cadenceAddress || evmAddress || applicationData`. [8](#0-7) [9](#0-8) 

### Proof of Concept

**Cross-chain replay (testnet → mainnet):**

1. Victim signs `signedData = <any 32 bytes>` on Flow Testnet using their wallet with the `"FLOW-V0.0-user"` domain tag. This is a normal dApp login or COA proof.
2. Victim has the same account key on Flow Mainnet (common for users who import their seed phrase to both networks).
3. Attacker submits the following Cadence script on **mainnet**:

```cadence
import EVM from 0xe467b9dd11fa00df  // mainnet EVM contract

access(all)
fun main(
    address: Address,          // victim's mainnet Cadence address
    path: PublicPath,          // /public/coa
    signedData: [UInt8],       // the bytes victim signed on testnet
    keyIndices: [UInt64],      // [0]
    signatures: [[UInt8]],     // victim's testnet signature (same key, same bytes)
    evmAddress: [UInt8; 20]    // victim's actual mainnet COA address
): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: address,
        path: path,
        signedData: signedData,
        keyIndices: keyIndices,
        signatures: signatures,
        evmAddress: evmAddress
    )
}
```

4. Because `"FLOW-V0.0-user"` contains no chain ID, the signature verifies successfully on mainnet. The function returns `isValid: true`.
5. The attacker can now pass this proof to any EVM contract on mainnet that calls `verifyCOAOwnershipProof` or checks ERC-1271 via the COA contract, falsely proving the victim's COA ownership. [2](#0-1) [10](#0-9) [11](#0-10)

```go
// DomainTagLength is set to 32 bytes.
//
// Signatures on Flow that needs to be scoped to a certain domain need to
// have the same length in order to avoid tag collision issues, when prefixing the
// message to sign.
const DomainTagLength = 32

const TransactionTagString = "FLOW-V0.0-transaction"

// TransactionDomainTag is the prefix of all signed transaction payloads.
//
// The tag is the string `TransactionTagString` encoded as UTF-8 bytes,
// right padded to a total length of 32 bytes.
var TransactionDomainTag = paddedDomainTag(TransactionTagString)

// paddedDomainTag padds string tags to form the actuatl domain separation tag used for signing
// and verifiying.
//
// A domain tag is encoded as UTF-8 bytes, right padded to a total length of 32 bytes.
func paddedDomainTag(s string) [DomainTagLength]byte {
	var tag [DomainTagLength]byte

	if len(s) > DomainTagLength {
		panic(fmt.Sprintf("domain tag %s cannot be longer than %d characters", s, DomainTagLength))
	}

	copy(tag[:], s)

	return tag
}
```
