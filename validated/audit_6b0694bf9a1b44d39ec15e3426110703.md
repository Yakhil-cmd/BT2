### Title
Wrong Receiver Context Used in `validate_delegate_action` for Inner `DeterministicStateInitAction` — (`File: runtime/runtime/src/action_validation.rs`)

### Summary

In `validate_delegate_action`, before the `FixDelegatedDeterministicStateInit` protocol feature, the inner actions of a `DelegateAction` are validated against the **outer transaction's receiver** (`receiver`, which equals the delegate's `sender_id`) rather than the **delegate action's own `receiver_id`**. This is a direct analog to the external bug: just as `AdapterBase._verifyAndSetupStrategy` called strategy functions in the wrong execution context (strategy context instead of adapter context), nearcore's `validate_delegate_action` validates inner actions against the wrong account identity (outer tx receiver instead of the delegate's inner receiver). The result is that any `DeterministicStateInitAction` embedded in a `DelegateAction` is always rejected at transaction admission, completely breaking the feature of creating deterministic accounts via meta-transactions.

### Finding Description

`validate_delegate_action` is called with `receiver` = the outer transaction's `receiver_id`, which by the meta-transaction protocol invariant equals `delegate_action.sender_id`. It then passes this `receiver` as the `inner_receiver` to `validate_actions_with_mode`, which in turn calls `validate_deterministic_state_init(limit_config, a, receiver)`.

`validate_deterministic_state_init` enforces:

```rust
let derived_id = derive_near_deterministic_account_id(&action.state_init);
if derived_id != *receiver_id {
    return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver { ... });
}
```

The correct `receiver_id` to check against is `delegate_action.receiver_id()` — the actual target account that the inner action will be applied to. Instead, the old code supplies the outer tx receiver (the sender's own account), which is never equal to the derived deterministic account ID. The validation therefore always fails for legitimate use.

The fix, gated behind `ProtocolFeature::FixDelegatedDeterministicStateInit`, corrects this by using `delegate_action.receiver_id()`:

```rust
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        delegate_action.receiver_id()   // correct
    } else {
        receiver                        // wrong: outer tx receiver = sender_id
    };
```

The old code path remains in the production binary for backward compatibility with protocol versions < 85.

### Impact Explanation

**High.** For any protocol version below 85, every `DelegateAction` carrying a `DeterministicStateInitAction` is unconditionally rejected at transaction admission with `InvalidDeterministicStateInitReceiver`. The feature of creating deterministic accounts via meta-transactions (relayer-sponsored transactions) is completely non-functional. An unprivileged user who crafts a valid meta-transaction for this purpose will always receive a rejection, with no workaround other than submitting the action directly (without a relayer).

Additionally, an attacker can craft an exploit transaction: embed `state_init_b` (whose derived ID is `det_account_b`) inside a `DelegateAction` whose `receiver_id` is `det_account_a` (a different deterministic account). The old validation checks `state_init_b` against the outer tx receiver `det_account_b`, which matches, so the transaction passes admission. The receipt validation later catches the mismatch and rejects it, but the relayer's gas is consumed.

### Likelihood Explanation

Any unprivileged user can trigger the functional breakage by submitting a `DelegateAction` containing a `DeterministicStateInitAction` on a node running protocol version < 85. No special privileges are required. The input is fully user-controlled: `sender_id`, `receiver_id`, and the inner `DeterministicStateInitAction` payload are all attacker-supplied fields in the signed transaction.

### Recommendation

The fix is already present in the codebase gated behind `ProtocolFeature::FixDelegatedDeterministicStateInit` (protocol version 85). Once all nodes have upgraded past version 85, the old code branch can be removed. Until then, the old branch remains reachable for backward-compatible receipt processing.
