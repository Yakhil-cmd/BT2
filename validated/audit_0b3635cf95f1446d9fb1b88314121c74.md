### Title
Fee-on-Transfer Token Accounting Bug in Two-Step Delegation Transfer — (`src/pool/pool.cairo`)

### Summary

`Pool::enter_delegation_pool` and `Pool::add_to_delegation_pool` perform a two-step token transfer using the same nominal `amount` for both legs. When the pool's configured token is fee-on-transfer, the pool receives `amount - fee` from the delegator but immediately attempts to forward the full `amount` to the staking contract. This either causes a permanent DoS on all delegation into that pool, or silently drains existing token balance held by the pool (e.g., returned principal from a prior staker exit), constituting theft of other pool members' funds.

---

### Finding Description

In `Pool::enter_delegation_pool` (and identically in `add_to_delegation_pool`), the delegation flow is:

**Step 1** — `transfer_from_delegator` pulls `amount` from the delegator into the pool contract:

```cairo
// src/pool/utils.cairo:20-28
pub(crate) fn transfer_from_delegator(
    pool_member: ContractAddress, amount: Amount, token_dispatcher: IERC20Dispatcher,
) {
    let self_contract = get_contract_address();
    token_dispatcher
        .checked_transfer_from(
            sender: pool_member, recipient: self_contract, amount: amount.into(),
        );
}
```

If the token charges a transfer fee, the pool actually receives `amount - fee`, not `amount`.

**Step 2** — `transfer_to_staking_contract` approves and forwards the original `amount` (not the actually received amount) to the staking contract:

```cairo
// src/pool/pool.cairo:668-682
fn transfer_to_staking_contract(
    self: @ContractState,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
    staker_address: ContractAddress,
) {
    let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
    token_dispatcher
        .approve(spender: staking_pool_dispatcher.contract_address, amount: amount.into());
    staking_pool_dispatcher.add_stake_from_pool(:staker_address, :amount);
}
```

Inside `Staking::add_stake_from_pool`, the staking contract pulls the full `amount` from the pool:

```cairo
// src/staking/staking.cairo:1031-1034
token_dispatcher
    .checked_transfer_from(
        sender: pool_contract, recipient: get_contract_address(), amount: amount.into(),
    );
```

The pool only holds `amount - fee` at this point. The `checked_transfer_from` will either:

1. **Revert** — if the pool has no pre-existing token balance, causing a DoS on all delegation into that pool.
2. **Succeed by consuming existing pool balance** — if the pool holds residual token balance (e.g., BTC returned to the pool after a staker calls `unstake_action`, which triggers `transfer_to_pools_when_unstake`), the `fee` is silently paid from those existing funds, stealing from other pool members.

The same two-step pattern is present in both entry points: [1](#0-0) [2](#0-1) 

The `transfer_from_delegator` helper always uses the caller-supplied `amount` verbatim, with no balance-before/balance-after check: [3](#0-2) 

And `add_stake_from_pool` in the staking contract pulls the same `amount` from the pool without any adjustment: [4](#0-3) 

The protocol supports arbitrary non-STRK tokens ("BTC tokens"). The only validation in `add_token` is that the token's `decimals()` return value is between 5 and 18 — there is no check that the token is not fee-on-transfer: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Scenario A — DoS (Temporary Freezing of Funds):** If the pool contract holds no pre-existing token balance, every call to `enter_delegation_pool` or `add_to_delegation_pool` reverts at the `checked_transfer_from` in `add_stake_from_pool`. No delegator can enter or increase their position in that pool. Existing delegators with a pending `exit_delegation_pool_intent` cannot complete their exit either, since the staking contract's accounting is blocked. This constitutes **temporary freezing of funds**.

**Scenario B — Theft of Existing Pool Balance (Theft of Unclaimed Yield/Principal):** When a staker calls `unstake_action`, the staking contract returns BTC principal to the pool via `transfer_to_pools_when_unstake`. The pool then holds BTC on behalf of pool members awaiting `remove_from_delegation_pool_action`. If a new delegator calls `enter_delegation_pool` with a fee-on-transfer token, the `fee` is silently deducted from this existing pool balance. Repeated calls drain the pool's BTC balance, stealing funds that belong to other pool members. This constitutes **theft of unclaimed yield / permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The `token_admin` role can add any ERC-20 token with decimals in [5, 18] as a BTC staking token. The protocol is explicitly designed to support multiple BTC-like tokens. A fee-on-transfer BTC-pegged token (e.g., a rebasing or tax token) added to the whitelist — even legitimately, without any compromise — immediately activates this vulnerability for every pool configured with that token. The THORChain precedent (PAXG in their whitelist) demonstrates this is a realistic scenario for production multi-token staking systems.

---

### Recommendation

Replace the two-step transfer with a balance-delta pattern. Measure the pool's token balance before and after `transfer_from_delegator`, and forward only the actually received amount to the staking contract:

```cairo
fn enter_delegation_pool(
    ref self: ContractState, reward_address: ContractAddress, amount: Amount,
) {
    // ... existing asserts ...
    let token_dispatcher = self.token_dispatcher.read();
    let staker_address = self.staker_address.read();

    // Measure balance before transfer
    let balance_before: u256 = token_dispatcher.balance_of(get_contract_address());
    transfer_from_delegator(:pool_member, :amount, :token_dispatcher);
    // Compute actually received amount
    let balance_after: u256 = token_dispatcher.balance_of(get_contract_address());
    let received_amount: Amount = (balance_after - balance_before)
        .try_into()
        .expect("amount overflow");

    // Forward only what was actually received
    self.transfer_to_staking_contract(amount: received_amount, :token_dispatcher, :staker_address);
    self.set_member_balance(:pool_member, amount: received_amount);
    // ... emit events with received_amount ...
}
```

Apply the same fix to `add_to_delegation_pool`.

---

### Proof of Concept

1. Token admin calls `add_token(fee_on_transfer_btc_token)` and `enable_token(fee_on_transfer_btc_token)`.
2. A staker calls `set_open_for_delegation(token_address: fee_on_transfer_btc_token)`, deploying a pool.
3. A delegator approves `amount` of the fee-on-transfer token to the pool contract and calls `pool.enter_delegation_pool(reward_address, amount)`.
4. Inside `enter_delegation_pool`:
   - `transfer_from_delegator` pulls `amount` from delegator; pool receives `amount - fee`.
   - `transfer_to_staking_contract(amount, ...)` approves `amount` and calls `add_stake_from_pool(staker, amount)`.
   - `add_stake_from_pool` calls `checked_transfer_from(pool, staking, amount)`.
   - Pool balance is only `amount - fee` → transfer reverts (DoS), **or** if pool has residual balance ≥ fee, the fee is consumed from existing pool funds (theft). [7](#0-6) [8](#0-7) [9](#0-8)

        fn add_stake_from_pool(
            ref self: ContractState, staker_address: ContractAddress, amount: Amount,
        ) {
            // Prerequisites and asserts.
            self.general_prerequisites();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let pool_contract = get_caller_address();
            let token_address = self
                .staker_pool_info
                .entry(staker_address)
                .get_pool_token(:pool_contract)
                .expect_with_err(Error::CALLER_IS_NOT_POOL_CONTRACT);
            let decimals = self.get_token_decimals(:token_address);
            let normalized_amount = NormalizedAmountTrait::from_native_amount(:amount, :decimals);

            // Update the staker's staked amount, and add to total_stake.
            let old_delegated_stake = self.get_delegated_balance(:staker_address, :pool_contract);
            let new_delegated_stake = old_delegated_stake + normalized_amount;
            self
                .insert_staker_delegated_balance(
                    :staker_address, :pool_contract, delegated_balance: new_delegated_stake,
                );
            self.add_to_total_stake(:token_address, amount: normalized_amount);

            // Transfer funds from the pool contract to the staking contract.
            // Sufficient approval is a pre-condition.
            let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
            token_dispatcher
                .checked_transfer_from(
                    sender: pool_contract, recipient: get_contract_address(), amount: amount.into(),
                );

            // Emit event.
            self
                .emit(
                    Events::StakeDelegatedBalanceChanged {
                        staker_address,
                        token_address,
                        old_delegated_stake: old_delegated_stake.to_native_amount(:decimals),
                        new_delegated_stake: new_delegated_stake.to_native_amount(:decimals),
                    },
                );
        }
