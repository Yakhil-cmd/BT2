Audit Report

## Title
Unbounded `artifact_processor_tasks` Spawning via `SLOT_TABLE_NO_LIMIT` Enables Memory/CPU Exhaustion - (File: `rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

Five of six artifact broadcast channels (consensus, certifier, DKG, iDKG, HTTPS-outcalls) are configured with `SLOT_TABLE_NO_LIMIT = usize::MAX`, meaning a malicious subnet peer can insert an unbounded number of entries into the per-peer slot table. Each unique `(slot_number, artifact_id)` pair spawns a new `process_slot_update` Tokio task that waits indefinitely for the advertising peer to retract the slot, causing unbounded growth of `artifact_processor_tasks` and `active_assembles` on victim nodes.

## Finding Description

In `rs/replica/setup_ic_network/src/lib.rs`, the constant `SLOT_TABLE_NO_LIMIT = usize::MAX` is defined alongside `SLOT_TABLE_LIMIT_INGRESS = 50_000`. The ingress channel uses the bounded limit, but consensus, certifier, DKG, iDKG, and HTTPS-outcalls all pass `SLOT_TABLE_NO_LIMIT` to `abortable_broadcast_channel`. [1](#0-0) [2](#0-1) 

In `ConsensusManagerReceiver::handle_slot_update_receive`, the guard `Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit` is always satisfied when `slot_limit = usize::MAX`, so every new unique slot number is inserted unconditionally. [3](#0-2) 

When a new artifact ID is seen (not already in `active_assembles`), a new task is spawned into the unbounded `artifact_processor_tasks` JoinSet: [4](#0-3) 

Inside `process_slot_update`, after `assemble_message` returns `AssembleResult::Unwanted` (which it will for any fake artifact ID), the task blocks on `peer_rx.wait_for(|p| p.is_empty()).await` with no timeout and no cancellation token check at this point: [5](#0-4) 

The `handle_topology_update` method does remove a peer's entries when it leaves the subnet topology, which would unblock waiting tasks. However, a malicious peer that remains in the subnet (below the Byzantine fault threshold) is never removed from the topology, so its spawned tasks remain alive indefinitely. [6](#0-5) 

The `update_handler` has no per-peer rate limiting, and the router uses `DefaultBodyLimit::disable()`, removing any payload size guard: [7](#0-6) 

The QUIC transport allows up to `MAX_CONCURRENT_BIDI_STREAMS = 1000` concurrent streams per connection, and `start_stream_acceptor` spawns a task per accepted stream with no per-peer rate limiting: [8](#0-7) [9](#0-8) 

## Impact Explanation

A malicious subnet peer can continuously send `SlotUpdate` messages for consensus/certifier/DKG/iDKG/HTTPS-outcalls artifact types, each with a unique `slot_number` and unique fake `artifact_id`, never retracting them. This causes unbounded growth of `active_assembles` (HashMap) and `artifact_processor_tasks` (JoinSet) on victim nodes. Each live task holds a `watch::Receiver<PeerCounter>`, a cloned `Assembler`, a `Sender<UnvalidatedArtifactMutation>`, and Tokio task overhead. At scale, this exhausts heap memory and saturates the Tokio thread pool, causing victim nodes to fall behind consensus, miss rounds, or crash — directly disrupting subnet liveness.

This matches the **High** impact category: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation

The attacker must be a legitimate subnet node (authenticated via mutual TLS using registry-keyed certificates). This is within the IC's explicit threat model for Byzantine peers below the fault threshold. No special cryptographic material beyond the node's own TLS key is required. The attack is low-cost and sustained: small protobuf-encoded `SlotUpdate` messages sent continuously. The existence of `SLOT_TABLE_LIMIT_INGRESS = 50_000` with the comment "used to protect against a malicious peer advertising many ingress messages" confirms the developers are aware of this attack class but have not applied the same protection to the other five artifact types.

## Recommendation

1. Apply a per-peer slot table limit to all artifact types. Derive appropriate per-type bounds from protocol maximums (e.g., consensus artifacts per round, DKG transcripts per epoch), mirroring the existing `SLOT_TABLE_LIMIT_INGRESS` pattern.
2. Add per-peer rate limiting on `SlotUpdate` message processing in `update_handler`, keyed by `NodeId`.
3. Add a global cap on `artifact_processor_tasks.len()` to bound total memory usage regardless of per-peer limits.
4. Add a timeout to `peer_rx.wait_for(|p| p.is_empty()).await` (tracked as `TODO: NET-1774`) so tasks do not wait indefinitely for a peer that never retracts.

## Proof of Concept

A malicious subnet node executes the following loop using its legitimate TLS credentials:

```rust
fn attack(transport: Arc<dyn Transport>, target: NodeId) {
    let mut slot_number: u64 = 0;
    loop {
        let slot_update = pb::SlotUpdate {
            commit_id: 1,
            slot_id: slot_number,           // unique slot number each iteration
            update: Some(pb::slot_update::Update::Id(
                encode_fake_consensus_id(slot_number), // unique fake artifact ID
            )),
        };
        let request = Request::builder()
            .uri("/consensus/update")
            .body(Bytes::from(slot_update.encode_to_vec()))
            .unwrap();
        let _ = transport.rpc(&target, request);
        slot_number += 1;
        // Never retract: peer_rx.wait_for(|p| p.is_empty()) never resolves
        // active_assembles and artifact_processor_tasks grow without bound
    }
}
```

Each iteration triggers `handle_slot_update_receive` to insert a new entry into `peer_slot_table` (unbounded due to `SLOT_TABLE_NO_LIMIT`), insert a new entry into `active_assembles`, and spawn a new task into `artifact_processor_tasks`. The bouncer returns `Unwanted` for the fake ID, but the task then blocks on `peer_rx.wait_for(|p| p.is_empty()).await` indefinitely. Memory and task count grow without bound until the victim node is resource-exhausted.

    fn handle_topology_update(&mut self) {
        self.metrics.topology_updates_total.inc();
        let new_topology = self.topology_watcher.borrow().clone();
        let mut nodes_leaving_topology = HashSet::new();

        self.slot_table.retain(|node_id, _| {
            if !new_topology.is_member(node_id) {
                nodes_leaving_topology.insert(*node_id);
                let _ = self
                    .metrics
                    .slot_table_new_entry_total
                    .remove_label_values(&[node_id.to_string().as_str()]);
                false
            } else {
                true
            }
        });

        for peers_sender in self.active_assembles.values() {
            peers_sender
                .send_if_modified(|set| nodes_leaving_topology.iter().any(|n| set.remove(*n)));
        }
        debug_assert!(
            self.slot_table.len() <= self.topology_watcher.borrow().iter().count(),
            "Slot table contains more nodes than nodes in subnet after pruning"
        );
    }
