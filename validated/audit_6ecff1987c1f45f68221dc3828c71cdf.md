### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` caps on the pool admin's fee authority. The sibling function `setPoolBinAdditionalFees` imposes no analogous cap on `addFeeBuyE6` / `addFeeSellE6`, allowing the pool admin to charge traders an additional uncapped spread on every bin, bypassing the fee-cap invariant the protocol relies on to bound semi-trusted admins.

### Finding Description

`setPoolAdminFees` explicitly guards both fee components:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`setPoolBinAdditionalFees` passes values straight through with no cap check:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the pool admin can set them to any value up to 65 535 (≈ 6.55 % in E6 units). These values are added directly to `baseFeeX64` in every swap path:

```solidity
// MetricOmmPool.sol L910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

```solidity
// MetricOmmPool.sol L1088
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
```

The same uncapped addition appears in `getSellAndBuyPrices` (L540-541). There is no guard anywhere in the pool or factory that limits the sum of `baseFee + addFee`.

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = addFeeSellE6 = 65535` on every bin. Every swap through those bins pays an extra ≈ 6.55 % on top of the already-capped base spread, regardless of what `maxAdminSpreadFeeE6` is set to. The excess fee accrues inside the bin as LP balance; a pool admin who is also an LP extracts it on `removeLiquidity`. Traders suffer a direct, unannounced loss of principal beyond the cap the protocol guarantees.

### Likelihood Explanation

The pool admin is semi-trusted and the call is a single transaction with no timelock. Any pool admin who turns adversarial (or whose key is compromised) can immediately impose the maximum bin fee on all bins. Users have no on-chain protection once the fee is set.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool) analogous to the one in `setPoolAdminFees`:

```solidity
if (uint256(addFeeBuyE6) + uint256(addFeeSellE6) > maxAdminSpreadFeeE6)
    revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` constant and enforce it in the factory before forwarding to the pool.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 1000` (0.1 %).
2. Pool admin calls `setPoolAdminFees(pool, 1000, 0)` — accepted, at cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no revert**.
4. A trader swaps through bin 0; the effective spread fee is `baseFee + 65535/1e6 ≈ baseFee + 6.55 %`, far above the 0.1 % cap.
5. The excess LP fee stays in the bin; the pool admin (as LP) withdraws it via `removeLiquidity`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
