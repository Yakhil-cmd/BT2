### Title
JIT Liquidity Sandwich Attack Steals LP Spread Fees from Existing Liquidity Providers - (File: metric-core/contracts/libraries/LiquidityLib.sol, metric-core/contracts/libraries/SwapMath.sol)

### Summary
The Metric OMM pool immediately credits LP spread fees into each bin's token balance (`token0BalanceScaled` / `token1BalanceScaled`) during a swap, and there is no locking period, withdrawal fee, or minimum holding time for liquidity positions. An attacker can front-run any large swap by adding a dominant share position to the active bin, capturing the majority of the LP fee, then immediately removing liquidity in the same block — a classic JIT (just-in-time) liquidity sandwich. Unlike JIT in a DEX, the oracle-anchored pricing means the attacker's liquidity provides zero benefit to the trader (price is fixed by the oracle), making the attack purely extractive.

### Finding Description

**Fee accrual mechanism:** In `SwapMath.buyToken1InBinSpecifiedIn` and `buyToken0InBinSpecifiedIn`, the LP fee portion of each swap is added directly to the bin's token balance:

```solidity
// buyToken1InBinSpecifiedIn (token0 in, token1 out)
binState.token0BalanceScaled =
    (uint256(binState.token0BalanceScaled) + totalIn0Scaled - protocolFeeAmountScaled).toUint104();
```

The LP fee (`totalIn0Scaled - protocolFeeAmountScaled`) is immediately reflected in `binState.token0BalanceScaled`. [1](#0-0) 

**Proportional withdrawal:** When `removeLiquidity` is called, each LP receives tokens proportional to their share of the bin's current balance:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [2](#0-1) 

**No locking or withdrawal fee:** `addLiquidity` and `removeLiquidity` impose no time lock, no withdrawal fee, and no minimum holding period. The only guard is `MINIMAL_MINTABLE_LIQUIDITY`, which only prevents dust positions. [3](#0-2) 

**Attack sequence:**

1. Attacker monitors the mempool for a large pending `swap()` call targeting the active bin.
2. In a front-running transaction, attacker calls `addLiquidity` on the active bin with a large share count (e.g., 9× the existing total shares), depositing proportional tokens.
3. The victim's `swap()` executes: LP fee enters the bin's token balance, now shared across all shares including the attacker's dominant position.
4. In a back-running transaction, attacker calls `removeLiquidity`, withdrawing their shares plus their proportional slice of the LP fee.

**Concrete numbers:**
- Existing LP: 10,000 shares, bin has 1,000,000 scaled token0
- Attacker adds 90,000 shares, depositing 9,000,000 scaled token0 → total shares = 100,000, bin balance = 10,000,000
- Swap generates 100,000 scaled token0 LP fee → bin balance = 10,100,000
- Attacker removes 90,000 shares: receives `90,000/100,000 × 10,100,000 = 9,090,000` → profit = 90,000 (90% of the LP fee)
- Original LP receives only `10,000/100,000 × 10,100,000 = 1,010,000` → earns only 10,000 instead of 100,000

**Oracle pricing eliminates any benefit:** In a standard DEX, JIT liquidity at least improves price execution for the trader. In Metric OMM, the swap price is fully determined by the oracle (`_getBidAndAskPriceX64()`), so the attacker's liquidity provides zero benefit to the trader or protocol. [4](#0-3) 

### Impact Explanation

Existing LPs lose a proportional share of their earned spread fees to JIT attackers. The LP fee is an owed LP asset — it is the compensation for providing liquidity and bearing inventory risk. When a JIT attacker captures the majority of this fee without bearing any meaningful risk (they add and remove in the same block), the protocol's LP incentive model is broken. LPing becomes economically unattractive, which will drain liquidity from the pool over time. This is a direct loss of owed LP assets.

### Likelihood Explanation

MEV bots routinely perform JIT attacks on-chain. The attack is:
- **Permissionless**: any address can call `addLiquidity` and `removeLiquidity`
- **Profitable**: profitable whenever LP fee from the swap exceeds gas cost of two transactions
- **Risk-free**: the attacker deposits and withdraws at the same bin balance ratio (modulo the fee), with no price exposure
- **Detectable**: large pending swaps are visible in the mempool

The attack is especially attractive on L2s (low gas) where even small swaps become profitable targets.

### Recommendation

Apply one or more of the following mitigations:

1. **Withdrawal delay / locking**: Track the block number when shares were minted and prevent removal within the same block (or N blocks). This is the most direct fix.
2. **Withdrawal fee**: Charge a small fee on `removeLiquidity` (e.g., 0.01–0.05% of withdrawn value) that makes same-block round-trips unprofitable.
3. **Fee vesting**: Instead of immediately crediting LP fees into bin balances, accumulate them in a separate vesting buffer and release them linearly over multiple blocks.
4. **Extension guard**: Implement a `beforeRemoveLiquidity` extension that enforces a minimum holding period per position, keyed by `(owner, salt, bin)` and the block of last deposit.

### Proof of Concept

```
State: bin 0 has 1,000,000 scaled token0, 10,000 total shares (existing LP holds all)

Block N, tx 1 (attacker front-runs):
  addLiquidity(attacker, salt=1, bin=0, shares=90,000)
  → attacker deposits 9,000,000 scaled token0
  → binState.token0BalanceScaled = 10,000,000
  → binTotalShares[0] = 100,000

Block N, tx 2 (victim swap):
  swap(zeroForOne=false, amountSpecified=X)
  → SwapMath.buyToken1InBinSpecifiedIn executes
  → LP fee = 100,000 scaled token0 enters bin
  → binState.token0BalanceScaled = 10,100,000

Block N, tx 3 (attacker back-runs):
  removeLiquidity(attacker, salt=1, bin=0, shares=90,000)
  → amount0Scaled = 90,000/100,000 × 10,100,000 = 9,090,000
  → attacker profit = 9,090,000 - 9,000,000 = 90,000 (90% of LP fee)
  → existing LP earns only 10,000 instead of 100,000
```

The root cause is in `LiquidityLib.removeLiquidity` at the proportional withdrawal calculation [2](#0-1)  combined with the immediate fee accrual in `SwapMath` [1](#0-0)  and the absence of any withdrawal delay or fee in `MetricOmmPool.removeLiquidity`. [5](#0-4)
