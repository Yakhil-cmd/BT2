Audit Report

## Title
Missing Post-Handshake Topology Membership Guard Allows Removed Peer to Persist in `peer_map` — (`rs/p2p/quic_transport/src/connection_manager.rs`)

## Summary

`handle_established_connection` unconditionally inserts a completed connection into `peer_map` without re-checking `self.topology.is_member(&peer_id)`. When a topology-change watcher fires in the same `select!` poll cycle as a handshake completion, `handle_topology_change` scans `peer_map` to close removed-peer connections — but the peer is not yet in `peer_map`, so no close is issued. On the next iteration, `handle_established_connection` inserts the peer unconditionally. The stale entry persists indefinitely because `PeerManager` uses `send_if_modified` and will not re-fire the watcher if topology is stable, while QUIC keep-alives prevent idle-timeout eviction.

## Finding Description

**Root cause — `handle_established_connection` lacks a topology guard:**

`handle_established_connection` (lines 495–535) writes directly to `peer_map` with no call to `self.topology.is_member`. By contrast, `handle_outbound_conn_attemp` (lines 439–442) calls `self.can_i_dial_to`, which internally calls `self.topology.is_member(dst)` (line 279). The asymmetry means the inbound path (and the outbound completion path at line 298) has no equivalent post-handshake membership check.

**Race window in the `select!` loop:**

The `run` loop (lines 290–381) uses an unbiased `select!`. Both `self.inbound_connecting.join_next()` (line 316) and `self.watcher.changed()` (line 365) can be simultaneously ready. Tokio's unbiased `select!` picks among ready branches pseudo-randomly, so either branch can win.

**`handle_topology_change` only closes connections already in `peer_map`:**

Lines 422–433 iterate `self.peer_map.read()` and call `conn_handle.conn().close(...)` for peers no longer in topology. If the peer's handshake completed but `handle_established_connection` has not yet run, the peer is absent from `peer_map` and the close is silently skipped.

**No follow-up topology event:**

`PeerManager::run` (lines 93–101 of `lib.rs`) calls `send_if_modified`, which only notifies watchers when topology actually changes. With `TOPOLOGY_UPDATE_INTERVAL = 3s` and no further membership changes, the watcher never fires again, so `handle_topology_change` never gets a second chance to close the stale connection.

**QUIC keep-alives prevent self-expiry:**

`KEEP_ALIVE_INTERVAL = 1s` and `IDLE_TIMEOUT = 5s` (lines 78–81) mean an active peer keeps the connection alive past the idle timeout. The `active_connections` task only removes the peer from `peer_map` when the stream acceptor terminates (lines 334–342), which does not happen while the connection is alive.

**Exploit flow:**

1. Peer P is a legitimate subnet member; its TLS handshake completes and the result sits in `inbound_connecting`.
2. A registry update removes P; `PeerManager` fires `send_if_modified`, making `watcher.changed()` ready.
3. In the same `select!` poll, both `inbound_connecting.join_next()` and `watcher.changed()` are ready.
4. `watcher.changed()` wins → `handle_topology_change` runs → P is not in `peer_map` → no close issued → `self.topology` now excludes P.
5. Next iteration: `inbound_connecting.join_next()` fires → `handle_established_connection` inserts P into `peer_map` without checking `self.topology`.
6. P's `ConnectionHandle` is now in `peer_map` and its stream-acceptor task is in `active_connections`. Keep-alives keep the connection alive. No further topology event fires.

## Impact Explanation

The removed node's `ConnectionHandle` remains in `peer_map`. Higher-layer P2P components (consensus, DKG, catch-up package distribution) use `peer_map` lookups to route artifacts; they will continue sending block proposals, notarizations, and DKG dealings to the removed node. The removed node can also push inbound artifacts through the still-open QUIC streams, consuming resources on the local replica and potentially injecting malformed artifacts into the P2P layer. This constitutes a subnet availability and P2P integrity impact: a removed Byzantine node retains an active, authenticated channel into the subnet's P2P fabric beyond its authorized membership window. This matches the **Medium** impact class: difficult attack requiring node-level control and strict timing conditions, but with meaningful security impact on subnet P2P availability and artifact routing integrity.

## Likelihood Explanation

The attacker must be a legitimate subnet member (governance-controlled entry — high bar for a single node, but not a majority). A Byzantine operator who controls their own node can time a connection attempt to complete just as a self-removal governance proposal propagates to the local registry. The outbound path (line 298) extends the exploitable window to up to `CONNECT_TIMEOUT = 10s`, making the race substantially easier to hit than the narrow inbound window. The condition is deterministically reproducible in a test harness and does not require mainnet DoS or volumetric attack.

## Recommendation

Add a topology membership guard at the top of `handle_established_connection`, mirroring the guard already present in `handle_outbound_conn_attemp`:

```rust
fn handle_established_connection(&mut self, connection: Connection, peer_id: NodeId) {
    if !self.topology.is_member(&peer_id) {
        connection.close(VarInt::from_u32(0), b"peer not in current topology");
        return;
    }
    // ... existing logic
}
```

This closes the race for both the inbound path (line 318) and the outbound completion path (line 298), since both call `handle_established_connection`.

Deterministic unit test sketch using `tokio::test`:

1. Construct a `ConnectionManager` with peer P in topology and a mock QUIC endpoint.
2. Spawn an inbound handshake task for P; drive it to completion but do not yet poll `inbound_connecting` (park the executor).
3. Update the topology watcher to remove P (simulating a registry update), making `watcher.changed()` ready.
4. Drive the `select!` loop once with `watcher.changed()` forced to win (use `biased` ordering in the test harness or manually call `handle_topology_change` then `handle_established_connection` in sequence).
5. Assert `peer_map.read().unwrap().contains_key(&peer_id)` — passes, demonstrating the stale entry.
6. Assert `self.topology.is_member(&peer_id)` returns `false` — confirms P is in `peer_map` but not in topology.
