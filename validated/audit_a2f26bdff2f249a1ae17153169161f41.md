### Title
Staker Can Increase Commission Mid-Epoch Within Commitment Bounds, Stealing Delegators' Unclaimed Yield — (File: `src/staking/staking.cairo`)

---

### Summary

A staker holding an active `CommissionCommitment` can call `set_commission` to raise their commission rate at any point during an ongoing epoch. Because the change is written to storage immediately — with no epoch-delay mechanism — all reward calculations that occur after the change (but within the same epoch) use the new, higher commission. Delegators who joined the pool under the original lower rate have their unclaimed yield silently redirected to the staker.

---

### Finding Description

`set_commission` (line 725) delegates to `update_commission` (line 1573) when a commission already exists. Inside `update_commission`, when a `CommissionCommitment` is active, the only constraint is that the new value must be `<= max_commission` and `!= old_commission`:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
}
```

The new value is then written directly to storage with no epoch delay:

```cairo
staker_pool_info.commission.write(Option::Some(commission));
``` [1](#0-0) 

Compare this to `set_public_key` and `set_peer_id`, which both defer their changes to `get_epoch_plus_k()` — a future epoch — precisely to avoid mid-epoch inconsistencies:

```cairo
let new_activation_epoch = self.get_epoch_plus_k();
self.public_key.write(staker_address, (new_activation_epoch, prev_public_key, public_key));
``` [2](#0-1) 

No equivalent deferral exists for commission. The commission stored in `staker_pool_info.commission` is read live at reward-calculation time in both the pre-consensus path (`update_rewards_from_attestation_contract`) and the consensus path (`update_rewards`), so any mid-epoch increase is immediately reflected in the split applied to delegators' rewards. [3](#0-2) [4](#0-3) 

The developers themselves acknowledge the gap in a code comment on `set_commission_commitment`:

```cairo
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
``` [5](#0-4) 

Additionally, `set_commission_commitment` imposes no lower bound on `max_commission` relative to the current commission — only that `max_commission >= current_commission`. A staker with commission = 100 (1 %) can immediately set `max_commission = 9900` (99 %):

```cairo
let current_commission = staker_pool_info.commission();
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
``` [6](#0-5) 

---

### Impact Explanation

Delegators' unclaimed yield for the current epoch is reduced in proportion to the commission increase. In the extreme case (1 % → 99 %), delegators receive roughly 1 % of the rewards they were entitled to; the remaining 98 % flows to the staker. This is a direct, in-epoch theft of unclaimed yield, matching the **High: Theft of unclaimed yield** impact category.

---

### Likelihood Explanation

The attack requires no privileged protocol role. Any staker who:
1. Has an open delegation pool, and
2. Holds (or can set) a `CommissionCommitment` with a high `max_commission`

can execute this at will. Setting a commitment with a wide range is a one-time, permissionless action available to every staker. The attack is repeatable every epoch as long as the commitment remains active (up to one year).

---

### Recommendation

Apply the same epoch-deferral pattern used for `public_key` and `peer_id` to commission changes. Specifically, when `set_commission` is called, record the pending new commission alongside an `activation_epoch = get_epoch_plus_k()`, and only apply it when `current_epoch >= activation_epoch`. This ensures delegators' rewards for the current epoch are always calculated with the commission rate that was in effect when they joined.

---

### Proof of Concept

1. Staker calls `set_commission(commission: 100)` — sets 1 % commission.
2. Staker calls `set_commission_commitment(max_commission: 9900, expiration_epoch: current_epoch + 10)` — commitment allows up to 99 %.
3. Delegators observe the 1 % commission and join the pool mid-epoch.
4. Staker calls `set_commission(commission: 9900)` — immediately raises to 99 %; written directly to `staker_pool_info.commission`.
5. The attestation contract (pre-consensus) or a block proposer (consensus) triggers reward calculation via `update_rewards_from_attestation_contract` / `update_rewards`.
6. `split_rewards_with_commission` reads the live commission (99 %) and routes 99 % of the delegated-pool rewards to the staker.
7. Delegators receive ≈ 1 % of the yield they expected for the epoch — the remaining 98 % is stolen by the staker.

    impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
