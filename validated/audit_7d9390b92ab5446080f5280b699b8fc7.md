### Title
Malicious Participant Can Corrupt CKD Output Without Verification — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly aggregates every participant's `(big_y, big_c)` contribution with no cryptographic verification. A single malicious participant can substitute arbitrary group elements, silently corrupting the derived confidential key that the TEE application ultimately receives. This is the direct analog of the Morpho isolation-mode bug: one actor's unverified input poisons a shared aggregated state that all honest parties depend on.

---

### Finding Description

`do_ckd_coordinator` (lines 35–57) collects each participant's `CKDOutput` and accumulates it into a running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

There is no proof that the received `big_y` equals `y·G` for the `y` used to form `big_c`, and no proof that `big_c` encodes the participant's actual secret share `x_i`. The coordinator has no way to distinguish a correctly formed share from an arbitrary pair of group elements.

Each honest participant computes:

```
big_y  = y · G
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + y · app_pk
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
``` [2](#0-1) 

A malicious participant instead sends `(big_y′, big_c′)` of their choice. The coordinator's final sums become:

```
Y_total = Σ_{honest} λ_i·y_i·G  +  big_y′
C_total = msk·H(pk‖app_id) + (Σ_{honest} λ_i·y_i)·app_pk  +  big_c′
```

When the TEE unmasks with `app_sk`:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [3](#0-2) 

the result is `msk·H(pk‖app_id) + (big_c′ − app_sk·big_y′)`, which is not the intended confidential key unless the malicious participant chose values that cancel perfectly. The attacker can steer the output to any target element in G1.

Contrast this with the DKG protocol, which enforces a proof of knowledge for every commitment:

```rust
verify_proof_of_knowledge(
    &session_id,
    &mut proof_domain_separator.clone(),
    threshold,
    p,
    old_participants.clone(),
    commitment_i,
    proof_i.as_ref(),
)?;
``` [4](#0-3) 

No equivalent check exists in the CKD aggregation path.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept an unusable or attacker-chosen cryptographic output.**

The TEE application calls `unmask` on the coordinator's `CKDOutput` and uses the result as a deterministic secret. If that value is wrong, the application either derives a key it cannot reproduce (denial of derived-key service) or, if the attacker can choose `big_y′` and `big_c′` such that `big_c′ − app_sk·big_y′` equals a target, the attacker steers the derived key to a value they know. The entire confidentiality guarantee of the CKD protocol collapses for that invocation.

---

### Likelihood Explanation

**High.** Any single participant in the CKD session — not a threshold, not the coordinator — can mount this attack. The participant simply sends a malformed `CKDOutput` over the private channel to the coordinator. No special privilege, no leaked key, and no cryptographic break is required. The attack is undetectable by the coordinator because no verification step exists.

---

### Recommendation

Add a non-interactive zero-knowledge proof (e.g., a Sigma protocol / Chaum–Pedersen proof) that each participant attaches to their `(norm_big_y, norm_big_c)` message, proving:

1. `norm_big_y = λ_i · y · G` for some `y` (discrete-log consistency).
2. `norm_big_c = λ_i · (x_i · H(pk‖app_id) + y · app_pk)` using the same `y` and the participant's committed share `x_i` (ElGamal well-formedness).

The coordinator must verify all proofs before accumulating any contribution, mirroring the `verify_proof_of_knowledge` pattern already used in `do_keyshare`.

---

### Proof of Concept

1. Honest participants P1, P2, P3 run `ckd(...)` with coordinator P1.
2. Malicious P2 intercepts the `send_private` call and instead sends `CKDOutput::new(G1::generator(), G1::generator())` to P1.
3. P1's coordinator loop adds `G1::generator()` to both `norm_big_y` and `norm_big_c` with no error.
4. The returned `CKDOutput` is `(Y_honest + G, C_honest + G)`.
5. The TEE calls `unmask(app_sk)` and receives `(C_honest + G) − app_sk·(Y_honest + G) = msk·H(pk‖app_id) + G·(1 − app_sk)`, which is not the correct confidential key.
6. The TEE silently uses the wrong key; no error is surfaced anywhere in the protocol.
