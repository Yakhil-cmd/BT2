### Title
Unrestricted `disable_rewards` Parameter in `update_rewards` Enables Griefing of Consensus Reward Distribution - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function is callable by any unprivileged address and accepts a `disable_rewards` boolean that is never validated. When set to `true`, the function updates the global `last_reward_block` without distributing rewards, permanently blocking all stakers from receiving consensus rewards for that block. This is the direct analog to the external report's missing identifier validation: a routing/control parameter goes unvalidated, causing incorrect accounting.

---

### Finding Description

`update_rewards` in `src/staking/staking.cairo` is a public function gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero. No caller identity check exists. [1](#0-0) 

The function accepts a `disable_rewards: bool` parameter. The critical ordering flaw is that `last_reward_block` is written **before** the `disable_rewards` branch is evaluated: [2](#0-1) 

Because `last_reward_block` is a **global** variable (not per-staker), once it is written to the current block number, the assertion `current_block_number > self.last_reward_block.read()` will fail for every subsequent call in the same block: [3](#0-2) 

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` at the start of a block therefore:
1. Advances `last_reward_block` to the current block.
2. Distributes zero rewards (early return at line 1487).
3. Causes every subsequent legitimate `update_rewards` call in that block to revert with `REWARDS_ALREADY_UPDATED`.

The `disable_rewards` parameter is the analog to the unvalidated nested `exec_id` in the external report: a control parameter that determines whether accounting is performed is accepted from any caller without restriction, leading to incorrect state (no rewards accrued instead of correct rewards). [4](#0-3) 

---

### Impact Explanation

If an attacker front-runs every legitimate `update_rewards` call with `disable_rewards: true`, no staker ever accumulates consensus block rewards. This constitutes **permanent freezing of unclaimed yield** for all stakers and pool members across the protocol (High impact). Even a partial, intermittent attack constitutes griefing with measurable damage to users (Medium impact).

---

### Likelihood Explanation

- The function is fully public with no role check.
- Valid staker addresses are discoverable from on-chain `NewStaker` events.
- The attacker needs only to submit a transaction with higher gas priority than the legitimate caller each block — a standard front-running pattern on Starknet.
- The attacker bears gas costs but gains nothing, making this a pure griefing vector consistent with the allowed impact scope.

---

### Recommendation

1. **Short-term**: Move `self.last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard, so a call with `disable_rewards: true` does not consume the block's reward slot.
2. **Long-term**: Restrict who may call `update_rewards` with `disable_rewards: true` (e.g., only the staker themselves, their operational address, or a trusted keeper role), or remove the parameter entirely and handle the edge cases through a separate privileged function.

---

### Proof of Concept

```
// Attacker script, repeated every block:
staking_contract.update_rewards(

//   last_reward_block = current_block_number  (written unconditionally)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
    }
