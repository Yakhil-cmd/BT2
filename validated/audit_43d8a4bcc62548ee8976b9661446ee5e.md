### Title
Destroying a `CadenceOwnedAccount` Resource With a Non-Zero EVM Balance Permanently Locks Funds - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` is the sole and exclusive controller of its associated EVM address. COA addresses are keyless by design — no private key ever exists for them. The resource defines no destructor that checks or rescues the EVM balance before the resource is destroyed. Any unprivileged transaction sender who owns a COA can call `destroy` on it while the EVM address still holds a FLOW balance, permanently and irrecoverably locking those funds.

### Finding Description
The `CadenceOwnedAccount` resource is defined at line 518 of `fvm/evm/stdlib/contract.cdc`. Its design is explicitly documented:

> "COAs are not controlled by a key. Instead, every COA account has a unique resource accessible on the Cadence side, and anyone who owns that resource submits transactions on behalf of this address." [1](#0-0) 

The resource stores only `addressBytes: [UInt8; 20]` and defines no Cadence `destroy` function (destructor). The resource definition closes at line 803 with no balance-guard destructor present. [2](#0-1) [3](#0-2) 

The EVM balance of a COA is stored in the EVM state trie, not as a Cadence resource. Cadence's resource linearity rules therefore impose no compile-time or runtime constraint preventing `destroy` from being called on a COA that holds a non-zero EVM balance. The `withdraw` function, which is the only mechanism to move funds from the EVM address back to Cadence, requires the `Owner | Withdraw` entitlement — an entitlement that is only accessible through the live COA resource itself. [4](#0-3) 

Once the COA resource is destroyed, the EVM address it controlled is permanently orphaned: no key exists to sign transactions from it, and no Cadence resource exists to authorize `withdraw`, `call`, or bridge operations on it.

The pattern is confirmed throughout the test suite, where COAs are routinely destroyed after use — including cases where a non-zero balance was deposited and only partially withdrawn, leaving a residual EVM balance: [5](#0-4) 

### Impact Explanation
Any FLOW tokens (or ERC-20/ERC-721 assets held at the COA's EVM address) present at the time of COA destruction are permanently inaccessible. The EVM address continues to exist in the EVM state trie with its balance intact, but there is no mechanism — on either the Cadence or EVM side — to authorize any outgoing transaction from it. This constitutes irreversible cross-VM asset loss for the COA owner.

### Likelihood Explanation
The likelihood is realistic. COA destruction is a normal, documented Cadence operation. A developer building a Cadence contract or transaction that creates a temporary COA for a bridging or EVM interaction step may destroy the COA at the end of the transaction without first verifying that the EVM balance is zero. Partial withdrawals (e.g., amounts below the 1e10 attoFLOW minimum precision threshold) can leave a residual balance that is silently abandoned on destruction. The pattern `destroy cadenceOwnedAccount` appears throughout the codebase without any balance pre-check. [6](#0-5) 

### Recommendation
Add a Cadence `destroy` function (destructor) to the `CadenceOwnedAccount` resource that panics if the EVM balance is non-zero at the time of destruction:

```cadence
// Inside resource CadenceOwnedAccount
destroy() {
    pre {
        self.balance().isZero():
            "EVM.CadenceOwnedAccount: cannot destroy a COA with a non-zero EVM balance. Withdraw all funds first."
    }
}
```

This mirrors the recommendation in the source report: override the renunciation path to always revert when funds are at risk. Alternatively, the destructor could automatically withdraw the remaining balance to the owning Cadence account, though this requires access to a recipient vault and is architecturally more complex.

### Proof of Concept
The following Cadence transaction demonstrates the issue. It deposits 1.0 FLOW into a COA, withdraws only 0.5 FLOW, and then destroys the COA — permanently locking the remaining 0.5 FLOW in the EVM address with no recovery path:

```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>
import FungibleToken from <FungibleTokenAddress>

transaction() {
    prepare(signer: auth(Storage) &Account) {
        let vault = signer.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(from: /storage/flowTokenVault)!

        // Create COA and deposit 1.0 FLOW into its EVM address
        let coa <- EVM.createCadenceOwnedAccount()
        let deposit <- vault.withdraw(amount: 1.0) as! @FlowToken.Vault
        coa.deposit(from: <-deposit)

        // Withdraw only 0.5 FLOW — 0.5 FLOW remains in the EVM address
        let partial <- coa.withdraw(balance: EVM.Balance(attoflow: 500000000000000000))
        vault.deposit(from: <-partial)

        // Destroy the COA — the remaining 0.5 FLOW in the EVM address
        // is now permanently inaccessible. No key exists, no resource exists.
        destroy coa
        // EVM address still exists in state trie with 0.5 FLOW, forever locked.
    }
}
