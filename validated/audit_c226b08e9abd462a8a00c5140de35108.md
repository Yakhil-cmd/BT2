### Title
Gateway Admission Uses Stale Previous-Block L2 Gas Price, Causing Valid Transactions to Be Rejected and Invalid Transactions to Be Admitted - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` gates mempool admission by comparing a transaction's `max_price_per_unit` against the **previous block's** L2 gas price. The production config sets `min_gas_price_percentage: 100`, so the threshold is exactly the previous block's price. Because the L2 gas price changes every block (driven by the L1 gas oracle), the gateway's admission decision is systematically wrong in both directions: it rejects transactions that are valid for the next block (when the price falls) and admits transactions that will fail `check_fee_bounds` during batcher execution (when the price rises). The code itself carries a developer TODO acknowledging the wrong price source.

### Finding Description

In `validate_resource_bounds`, the gateway fetches the previous block's info and extracts its STRK L2 gas price:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold computation is:

```rust
let threshold = (gas_price_threshold_multiplier * previous_block_l2_gas_price.get().0).to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(StarknetError { ... "GAS_PRICE_TOO_LOW" ... });
}
``` [2](#0-1) 

The production deployment config sets `min_gas_price_percentage: 100`, making the threshold exactly equal to the previous block's L2 gas price: [3](#0-2) 

The `run_validate_entry_point` function also builds its `BlockContext` from the same previous-block info, only incrementing the block number but **not** updating gas prices:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
// gas_prices remain from the previous block
let block_context = BlockContext::new(block_info, ...);
``` [4](#0-3) 

When the batcher later executes the transaction, it uses the **actual current block's** gas prices. The blockifier's `check_fee_bounds` inside `perform_pre_validation_stage` enforces:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [5](#0-4) 

This creates a permanent mismatch between the price used at admission and the price used at execution.

The `GatewayFixedBlockSyncStateClient` caches the block info in a `OnceCell`, so the same stale price is reused for every transaction validated against a given validator instance: [6](#0-5) 

The `StatefulTransactionValidatorFactory::instantiate_validator` creates a new validator per transaction from the latest block at that moment, but the validator's `gateway_fixed_block_state_reader` is fixed to that snapshot for the lifetime of the validator: [7](#0-6) 

### Impact Explanation

Two concrete failure modes arise, both unprivileged and triggered by normal gas price volatility:

**Mode 1 — Valid transactions rejected (price falls):**  
If the L2 gas price for block N+1 is lower than block N (e.g., L1 congestion eases), a user who sets `max_price_per_unit = current_block_price` has their transaction rejected with `GAS_PRICE_TOO_LOW` even though it would pass `check_fee_bounds` during execution. The user cannot transact despite offering a price the network would accept.

**Mode 2 — Invalid transactions admitted (price rises):**  
If the L2 gas price for block N+1 is higher than block N, a transaction with `max_price_per_unit` between `price_N` and `price_N+1` passes gateway admission and enters the mempool. When the batcher executes it, `perform_pre_validation_stage` → `check_fee_bounds` fails with `InsufficientResourceBounds`, producing a `TransactionPreValidationError`. The transaction is included in the block as a failed transaction, consuming block bouncer capacity and wasting sequencer resources.

Both modes match the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The L2 gas price is recalculated every block from the L1 gas oracle. Any non-trivial L1 gas price movement between consecutive blocks triggers one of the two failure modes. With `min_gas_price_percentage: 100` in production, there is zero tolerance buffer — even a 1-unit change in the L2 gas price causes a mismatch. This is a continuous, passive condition requiring no attacker action; ordinary network volatility is sufficient.

### Recommendation

Replace the previous-block L2 gas price with the **next block's** projected L2 gas price in `validate_resource_bounds`. The TODO comment at line 229 already identifies this fix. The batcher computes the next block's gas prices from the L1 gas oracle before building the block; the gateway should query the same source (or a dedicated endpoint) to obtain the pending block's gas prices. Alternatively, apply a conservative downward buffer via `min_gas_price_percentage` (e.g., 90%) so that normal inter-block price fluctuations do not cause false rejections, while still filtering clearly under-priced transactions.

### Proof of Concept

1. Observe the L2 gas price in block N: `price_N = 1000 fri`.
2. The L1 gas oracle reports a lower price; the batcher sets block N+1's L2 gas price to `price_N+1 = 900 fri`.
3. A user submits an invoke transaction with `max_price_per_unit = 950 fri` (above `price_N+1`, so valid for execution).
4. The gateway calls `validate_resource_bounds`:
   - `threshold = 1000 * 100 / 100 = 1000`
   - `950 < 1000` → `GAS_PRICE_TOO_LOW` error returned.
5. The transaction is rejected at the gateway even though `check_fee_bounds` in the batcher would have accepted it (`950 >= 900`).

Conversely, with a rising price (`price_N+1 = 1100`):
1. User submits with `max_price_per_unit = 1050`.
2. Gateway threshold = 1000; `1050 >= 1000` → admitted to mempool.
3. Batcher executes: `check_fee_bounds` compares `1050 < 1100` → `MaxGasPriceTooLow` → `TransactionPreValidationError` → transaction fails in block.
