### Title
Unvalidated `app_pk` (ElGamal Public Key) in CKD Protocol Allows Coordinator to Extract Confidential Derived Secret — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts a caller-supplied `app_pk: PublicKey` (the ElGamal public key `A`) with no validation that it is a non-identity group element. A malicious app that submits `A = G1::identity()` (the point at infinity) causes the ElGamal blinding term `y_i * A` to vanish from every node's contribution, so the coordinator aggregates the raw BLS signature `msk * H(pk, app_id)` in the clear — the exact confidential derived secret that the protocol is designed to keep hidden from every single node.

---

### Finding Description

The CKD protocol's confidentiality guarantee rests on ElGamal encryption. Each node `i` computes:

```
C_i = S_i + y_i * A      where S_i = x_i * H(pk, app_id)
```

The coordinator aggregates:

```
C = msk * H(pk, app_id) + (Σ λ_i * y_i) * A
```

The app recovers the secret as `sig = C − a * Y = msk * H(pk, app_id)`.

The blinding term `(Σ λ_i * y_i) * A` is what prevents the coordinator from learning `sig`. If `A` is the identity element, the blinding term is zero for every node, and the coordinator receives `C = msk * H(pk, app_id)` directly.

In `compute_signature_share`, `app_pk` is used without any validity check:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;   // if app_pk == identity, big_c == big_s
``` [1](#0-0) 

The `ckd()` entry point performs only participant-list checks; it never validates `app_pk`: [2](#0-1) 

`PublicKey` is `blstrs::G1Projective`, which can legally represent the identity element. No downstream code rejects it. [3](#0-2) 

---

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

When `app_pk = G1Projective::identity()`:

1. Every node computes `big_c = big_s` (the blinding vanishes).
2. The coordinator aggregates `C = msk * H(pk, app_id)` — the raw BLS signature that is the confidential key.
3. The coordinator (a single MPC node) now holds the secret `s`, directly violating the core security requirement: *"No single node in the MPC network should be capable of computing s."*
4. The malicious app also receives `(Y, C)` and trivially recovers `sig = C − 0 * Y = C`, obtaining the key without any TEE attestation being meaningful at the library level. [4](#0-3) 

---

### Likelihood Explanation

The `app_pk` value originates from an on-chain transaction submitted by the TEE app. A malicious or misconfigured app can submit `A = identity` trivially — it requires no cryptographic capability, no key material, and no brute-force search. The MPC nodes accept whatever `(app_id, A)` arrives from the chain and pass it directly into `ckd()`. The attack is a single malformed transaction. [5](#0-4) 

---

### Recommendation

Add an explicit validity check for `app_pk` at the start of `compute_signature_share` (or inside `ckd()` before the protocol starts):

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Reject the identity element — it removes ElGamal blinding entirely.
    if bool::from(app_pk.is_identity()) {
        return Err(ProtocolError::InvalidInput(
            "app_pk must not be the identity element".to_string(),
        ));
    }
    // ... rest of function
```

Additionally, consider requiring a zero-knowledge proof of knowledge of `a` (i.e., a Schnorr proof that the app knows the discrete log of `A`) so that the app cannot submit an arbitrary group element it did not generate honestly. [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
