### Title
Unverified Participant Contributions in CKD Coordinator Allow Malicious Participant to Corrupt Confidential Key Derivation Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator blindly accumulates `(norm_big_y, norm_big_c)` values received from each participant with no proof that those values were honestly computed from the participant's actual key share. A single malicious participant can submit arbitrary group elements, causing the coordinator to assemble and return a structurally valid but cryptographically incorrect `CKDOutput`. Every honest party that consumes this output will derive a wrong confidential key, permanently breaking the CKD result for that invocation.

---

### Finding Description

The CKD protocol is split into two roles. Each participant (including the coordinator) runs `compute_signature_share`, which produces:

```
norm_big_y = λ_i · y_i · G
norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

where `x_i` is the participant's secret signing share and `y_i` is a fresh random scalar.

The coordinator then collects every other participant's pair and sums them: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

There is **no proof of correctness** attached to the received values. Nothing binds `participant_output.big_c()` to the participant's actual public key share or to the `big_y` they sent. A malicious participant can transmit any pair `(Y', C')` of group elements.

The final unmasked confidential key is:

```
C - app_sk · Y  =  msk · H(pk ‖ app_id)
```

If participant `i` substitutes `norm_big_c' = λ_i · (x_i' · H(pk ‖ app_id) + y_i · app_pk)` for an arbitrary `x_i' ≠ x_i`, the coordinator computes:

```
C_final = (msk + λ_i·(x_i' - x_i)) · H(pk ‖ app_id) + app_sk · Y_final
```

The unmasked result is `(msk + λ_i·(x_i' - x_i)) · H(pk ‖ app_id)`, a completely different key. The attacker can also send `(0, 0)` to zero out their contribution, or send a value that shifts the output by any chosen multiple of `H(pk ‖ app_id)`.

The root cause is the absence of any zero-knowledge proof (e.g., a Schnorr-style DLEQ proof) that ties `norm_big_c` to the participant's committed public key share. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

A single malicious participant causes the coordinator to output a `CKDOutput` that is structurally well-formed (two valid G1 elements) but encodes a wrong confidential key. Every honest consumer of that output — including the application that calls `unmask(app_sk)` — will silently derive an incorrect key. This is a **corruption of CKD output so honest parties accept an unusable cryptographic output**, matching the allowed High impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

Any single participant in the CKD session is an unprivileged library caller who controls the values they send to the coordinator. The attack requires only that the participant deviate from the protocol by sending a crafted `(norm_big_y, norm_big_c)` pair. No cryptographic break, no key leakage, and no external assumption is needed. The coordinator has no mechanism to detect the deviation. Likelihood is **High**.

---

### Recommendation

Each participant should attach a zero-knowledge proof of correct formation alongside their `(norm_big_y, norm_big_c)` contribution. Concretely, a DLEQ (Discrete Log Equality) proof can demonstrate that the discrete log of `norm_big_c - λ_i · x_i · H(pk ‖ app_id)` with respect to `app_pk` equals the discrete log of `norm_big_y` with respect to `G`, without revealing `y_i`. The coordinator must verify this proof for every received contribution before accumulating it. Contributions that fail verification should be rejected and the protocol aborted, identifying the malicious participant.

---

### Proof of Concept

1. Honest setup: 3 participants run DKG to obtain shares `x_1, x_2, x_3` of master secret `msk`.
2. CKD is invoked with `app_id` and `app_pk = app_sk · G`.
3. Participant 2 (malicious) computes `compute_signature_share` normally to obtain `(norm_big_y_2, norm_big_c_2)`, but instead sends `(norm_big_y_2, G1::identity())` to the coordinator — zeroing out their `norm_big_c` contribution.
4. The coordinator sums: `C_final = norm_big_c_1 + G1::identity() + norm_big_c_3`, missing participant 2's share of `msk · H(pk ‖ app_id)`.
5. The app calls `ckd_output.unmask(app_sk)` and obtains `(msk - λ_2 · x_2) · H(pk ‖ app_id)` — a wrong key — with no error returned.
6. Any downstream use of this derived key (e.g., encryption, authentication) silently fails or produces attacker-influenced output. [4](#0-3) [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
