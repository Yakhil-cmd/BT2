### Title
`EvmErc20V2.withdrawToNear` Encodes Sender Address Into Precompile Input, Causing Silent Token Burn Without NEAR Transfer When `error_refund` Feature Is Disabled - (File: `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.withdrawToNear` prepends the caller's 20-byte address between the flag byte and the amount field when calling the `ExitToNear` precompile. The `ExitToNear` precompile (compiled without the `error_refund` feature) expects the layout `[flag(1)][amount(32)][recipient(N)]`, but `EvmErc20V2` sends `[flag(1)][sender(20)][amount(32)][recipient(N)]`. This causes the precompile to parse a garbage 32-byte value (sender address bytes concatenated with the high 12 bytes of the real amount) as the transfer amount. Because this value almost always exceeds `u128::MAX`, `parse_amount` returns `ERR_INVALID_AMOUNT` and the precompile sub-call fails. Since `withdrawToNear` does not check the assembly `call` return value, the outer transaction succeeds while the user's ERC-20 tokens are permanently burned with no corresponding NEP-141 transfer on NEAR.

---

### Finding Description

**`EvmErc20V2.withdrawToNear` (lines 53–63)** encodes the precompile input as:

```solidity
bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
uint input_size = 1 + 20 + 32 + recipient.length;
```

Layout sent: `[0x01][sender: 20 bytes][amount: 32 bytes][recipient: N bytes]` [1](#0-0) 

**`EvmErc20` (V1) `withdrawToNear` (lines 53–63)** encodes without the sender:

```solidity
bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
uint input_size = 1 + 32 + recipient.length;
``` [2](#0-1) 

**`ExitToNear` precompile `TryFrom` (lines 727–775)** — without the `error_refund` feature — strips only the 1-byte flag and then immediately reads bytes `[0..32]` as the amount:

```rust
#[cfg(not(feature = "error_refund"))]
let input = parse_input(input)?;   // strips flag byte only → remaining = [sender(20)][amount(32)][recipient]

match flag {
    0x1 => {
        let amount = parse_amount(&input[..32])?;  // reads [sender(20)][first 12 bytes of amount]
        let Recipient { .. } = parse_recipient(&input[32..])?;  // reads [last 20 bytes of amount][recipient]
    }
}
``` [3](#0-2) 

**`parse_input` without `error_refund`** strips only the flag byte:

```rust
#[cfg(not(feature = "error_refund"))]
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
``` [4](#0-3) 

**`parse_amount`** rejects values exceeding `u128::MAX`:

```rust
fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    Ok(amount)
}
``` [5](#0-4) 

For any sender address with non-zero bytes in its high 12 bytes (virtually all real addresses), the 32-byte value `[sender(20)][high_12_of_amount]` will exceed `u128::MAX`, causing `parse_amount` to return `ERR_INVALID_AMOUNT`. The precompile sub-call fails (returns 0 in the assembly). Because `withdrawToNear` does not check `res`, the outer call succeeds. The `_burn` on line 55 has already executed, permanently destroying the user's ERC-20 tokens. [6](#0-5) 

With the `error_refund` feature enabled, `parse_input` reads 20 bytes as a `refund_address` after the flag, which aligns with the sender field in V2 — so V2 only works correctly when `error_refund` is compiled in. Without it, the encoding is incompatible. [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `withdrawToNear` on an `EvmErc20V2` instance when the Aurora Engine is compiled without the `error_refund` feature will have their ERC-20 tokens permanently burned with no corresponding NEP-141 transfer to NEAR. The tokens are irrecoverably lost: they no longer exist on the EVM side and were never credited on the NEAR side.

---

### Likelihood Explanation

`EvmErc20V2` is a production contract in the repository's `etc/eth-contracts/contracts/` directory. Any user interacting with a deployed `EvmErc20V2` instance and calling `withdrawToNear` triggers the bug. No special privileges are required — any token holder can call `withdrawToNear`. The only condition is that the Aurora Engine binary was compiled without `error_refund`.

---

### Recommendation

1. **In `EvmErc20V2.withdrawToNear`**: Guard the function with a compile-time or runtime check, or align the encoding with what the precompile expects. If `error_refund` is not guaranteed to be enabled, `EvmErc20V2` must not prepend the sender address.
2. **In the `ExitToNear` precompile**: Add explicit validation that the input length matches the expected format for the active feature set, and reject inputs that do not conform before any state changes occur.
3. **In `withdrawToNear`**: Check the return value of the assembly `call` and revert if it fails, so that a precompile failure does not silently succeed after burning tokens.

---

### Proof of Concept

1. Deploy `EvmErc20V2` on an Aurora instance compiled **without** the `error_refund` feature.
2. Mint 1000 tokens to `alice`.
3. `alice` calls `withdrawToNear("alice.near", 1000)`.
4. `_burn(alice, 1000)` executes — alice's ERC-20 balance drops to 0.
5. The precompile receives `[0x01][alice_address(20)][1000_as_bytes32(32)]["alice.near"]`.
6. `parse_amount` reads `[alice_address(20)][high_12_bytes_of_1000]` — a value far exceeding `u128::MAX` — and returns `ERR_INVALID_AMOUNT`.
7. The assembly `call` returns 0; `withdrawToNear` does not revert.
8. Alice's 1000 ERC-20 tokens are permanently destroyed; her NEAR account receives nothing. [1](#0-0) [8](#0-7)

            .copied()
            .ok_or_else(|| ExitError::Other(Cow::from("ERR_MISSING_FLAG")))?;

        let (refund_address, input) = parse_input(input)?;
        #[cfg(not(feature = "error_refund"))]
        let input = parse_input(input)?;

        match flag {
            0x0 => {
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(input)?;

                Ok(Self::BaseToken(BaseTokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    message,
                }))

                    receiver_account_id,
                    amount,
                    message,
                }))
