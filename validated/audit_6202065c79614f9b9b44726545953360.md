### Title
Reward address funds permanently lost when `add_to_delegation_pool` is called by the reward address — (File: `src/pool/pool.cairo`)

### Summary

`add_to_delegation_pool` permits the pool member's `reward_address` to act as the caller and pulls tokens directly from that caller. However, `exit_delegation_pool_action` always returns the principal to `pool_member`, never to whoever actually supplied the tokens. When `reward_address != pool_member` and `reward_address` funds the top-up, those tokens are permanently redirected to `pool_member`.

The same structural flaw exists in `increase_stake` / `unstake_action` in `src/staking/staking.cairo`.

### Finding Description

**`add_to_delegation_pool` (pool.cairo)**

`add_to_delegation_pool` accepts calls from either `pool_member` or `pool_member_info.reward_address`: [1](#0-0) 

When the caller is `reward_address`, `transfer_from_delegator` pulls tokens from `caller_address` (i.e., `reward_address`): [2](#0-1) 

`transfer_from_delegator` does a `checked_transfer_from(sender: pool_member, ...)` where `pool_member` is the passed-in `caller_address`: [3](#0-2) 

On exit, `exit_delegation_pool_action` unconditionally returns the principal to `pool_member`, ignoring who actually deposited the tokens: [4](#0-3) 

**`increase_stake` / `unstake_action` (staking.cairo)**

`increase_stake` similarly allows `reward_address` to call it and transfers from `caller_address`: [5](#0-4) 

But `unstake_action` returns the full stake to `staker_address`, not to whoever funded the increase: [6](#0-5) 

### Impact Explanation

Direct theft of user funds. If `reward_address` calls `add_to_delegation_pool` (or `increase_stake`) and `reward_address != pool_member` (or `staker_address`), the deposited tokens are credited to `pool_member`'s balance and returned to `pool_member` on exit. `reward_address` receives nothing back. The loss is permanent and proportional to the amount deposited.

This matches the **Critical** impact category: *Direct theft of user funds, whether at-rest or in-motion*.

### Likelihood Explanation

The `reward_address` role is explicitly designed to be a separate, independent account (e.g., a cold wallet or treasury). The protocol actively encourages this separation. Any `reward_address` that calls `add_to_delegation_pool` or `increase_stake` to top up a position — a natural action given the function signatures permit it — will permanently lose those funds. No privileged access, no external dependency, and no special conditions are required beyond having a `reward_address` that differs from `pool_member`/`staker_address`.

### Recommendation

In `exit_delegation_pool_action`, check whether a `reward_address` is set and return the principal there instead of (or in addition to) `pool_member`, mirroring how `claim_rewards` already correctly routes to `reward_address`:

```cairo
// Analogous fix for exit_delegation_pool_action:
let recipient = if pool_member_info.reward_address != pool_member {
    pool_member_info.reward_address
} else {
    pool_member
};
token_dispatcher.checked_transfer(recipient: recipient, amount: unpool_amount.into());
```

Apply the same fix to `unstake_action` in `staking.cairo` for the `increase_stake` path.

### Proof of Concept

1. Alice (`pool_member`) enters the delegation pool with 100 STRK. Her `reward_address` is Bob.
2. Bob (acting as `reward_address`) calls `add_to_delegation_pool(pool_member: Alice, amount: 50)`. Bob approves the pool contract and 50 STRK is transferred **from Bob** via `transfer_from_delegator(pool_member: caller_address=Bob, ...)`. [7](#0-6) 
3. Alice's on-chain balance is now 150 STRK.
4. Alice calls `exit_delegation_pool_intent(150)`, waits for the window, then calls `exit_delegation_pool_action`.
5. `exit_delegation_pool_action` transfers 150 STRK to `pool_member` = Alice. [8](#0-7) 
6. Bob has lost 50 STRK permanently. Alice received 50 STRK she did not deposit.
