Audit Report

## Title
`sender_info.sig` Included in `MessageId` Hash Enables Ingress Deduplication Bypass via Non-Deterministic Canister Signatures ‚ÄĒ (`rs/types/types/src/messages/http.rs`, `rs/messaging/src/scheduling/valid_set_rule.rs`)

## Summary

`representation_independent_hash_call_or_query` includes the `sig` field of `sender_info` verbatim in the hash that defines `MessageId`. Because IC canister signatures are non-deterministic ‚ÄĒ the embedded BLS certificate changes with every new certified-state height ‚ÄĒ an attacker who controls any canister can produce two byte-distinct but cryptographically valid signatures over the same `info` blob. Submitting two otherwise-identical update calls carrying `sig1` and `sig2` yields two distinct `MessageId`s, both of which are `Unknown` in ingress history, causing both to be inducted and executed.

## Finding Description

**Root cause ‚ÄĒ `sig` is hashed into `MessageId`**

`representation_independent_hash_call_or_query` builds the map that is hashed to produce `MessageId`. When `sender_info` is present, all three sub-fields ‚ÄĒ `info`, `signer`, and `sig` ‚ÄĒ are inserted:

```rust

    map.insert("sender_info", Map(btreemap! {
        "info"   => Bytes(info),
        "signer" => Bytes(signer),
        "sig"    => Bytes(sig),   // ‚Üź non-deterministic canister signature bytes
    }));
}
hash_of_map(&map, |key, value| hash_key_val(key, value))
``` [1](#0-0)

**Root cause ‚ÄĒ canister signatures are non-deterministic**

A canister signature encodes a `certificate` (the subnet's BLS threshold signature over the state tree at a specific certified height) and a `tree` (Merkle witness). The certificate changes with every new certified state, so signing the same `info` blob at two different block heights produces two byte-distinct but cryptographically valid signatures. `verify_sender_info_canister_sig` only checks cryptographic validity; it has no notion of "same logical message": [3](#0-2) 

**Deduplication is keyed solely on `MessageId`**

`ValidSetRuleImpl::is_duplicate` queries ingress history by `msg.content().id()`, which includes `sig`: [4](#0-3) 

`validate_ingress_pool_object` in the ingress handler also checks by `message_id` (which is the `MessageId` derived from the same hash): [5](#0-4) 

`IngressHistorySet::contains` converts `IngressMessageId ‚Üí MessageId` and checks status: [6](#0-5) 

`IngressMessageId` wraps `MessageId` directly: [7](#0-6) 

Because `sig1 ‚Č† sig2`, `MessageId1 ‚Č† MessageId2`. Both messages are `Unknown` in ingress history, so both pass the duplicate check and are inducted.

**Exploit flow**

1. Attacker deploys any canister (no special privilege required).
2. Attacker signs the same `info` blob at two different certified-state heights ‚Üí `sig1 ‚Č† sig2`, both cryptographically valid.
3. Attacker submits two update calls with identical `sender`, `canister_id`, `method_name`, `arg`, `ingress_expiry`, `nonce`, but `sender_info.sig = sig1` and `sender_info.sig = sig2`.
4. Both pass `validate_request` (valid signatures, valid `sender_info`).
5. Both receive distinct `MessageId`s; both are `Unknown` in ingress history.
6. Both are inducted into the ingress pool and selected into a block payload.
7. `ValidSetRuleImpl::induct_messages` inductes both; both are executed.

## Impact Explanation

Any canister method reachable via ingress and lacking its own idempotency guard can be double-executed. Concrete targets include custom token canisters, DeFi protocols, cycles-minting flows, and governance submission paths. For canisters that do not implement application-level deduplication (e.g., `created_at_time`-based checks), a single attacker-controlled canister can cause double-crediting or double-execution of state-mutating operations. This matches the allowed impact: **unauthorized funds manipulation / double-execution of ledger or canister-controlled assets** at High‚ÄďCritical severity.

## Likelihood Explanation

- No privileged access required; any principal can deploy a canister on the IC.
- Obtaining two valid `sig` values for the same `info` blob requires only two calls to the signing canister separated by one certified-state advancement (~2 seconds).
- The `sender_info` feature is production-deployed: the integration test `requests_with_valid_sender_info` confirms acceptance of valid `sender_info` on live nodes. [8](#0-7) 
- The attack is fully local-testable with a state-machine test using a mock canister signer.

## Recommendation

Exclude `sig` from the `MessageId` hash in `representation_independent_hash_call_or_query`. The `sig` field is a proof-of-knowledge artifact whose byte representation is non-deterministic; it carries no semantic content beyond "this `info` was attested by `signer`". The `MessageId` should be computed from semantically meaningful fields only:

```rust
if let Some(RawSignedSenderInfoSlices { info, signer, .. }) = sender_info {
    map.insert("sender_info", Map(btreemap! {
        "info"   => Bytes(info),
        "signer" => Bytes(signer),
        // sig excluded from MessageId
    }));
}
```

The `sig` field is still validated cryptographically by `validate_sender_info` before induction; removing it from the hash does not weaken authentication. Two requests with the same `info`/`signer` but different `sig` bytes will then produce the same `MessageId`, restoring the at-most-once execution invariant. [9](#0-8) 

## Proof of Concept

```rust
// State-machine test sketch
let canister = deploy_signing_canister(&env);

let info_blob = b"user-attributes";
let sig1 = canister.sign_at_height(info_blob, height_1); // certified state S1
let sig2 = canister.sign_at_height(info_blob, height_2); // certified state S2 ‚Č† S1
assert_ne!(sig1, sig2); // different certificates ‚Üí different bytes

let base = UpdateCall {
    sender, canister_id: target, method: "execute",
    arg: call_args, ingress_expiry, nonce: None
};

let r1 = base.with_sender_info(info_blob, signer, sig1);
let r2 = base.with_sender_info(info_blob, signer, sig2);

let id1 = r1.message_id(); // includes sig1
let id2 = r2.message_id(); // includes sig2
assert_ne!(id1, id2);      // distinct MessageIds

env.submit(r1); // accepted, inducted
env.submit(r2); // accepted, inducted (id2 not in ingress history)
env.execute_round();
env.execute_round();

// Assert: target canister method executed TWICE ‚ÄĒ at-most-once violated
```

fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
