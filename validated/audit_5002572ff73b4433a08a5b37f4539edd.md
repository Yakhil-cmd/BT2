### Title
Unbounded `MessageBuffer` Growth via Malicious Message Injection Causes Memory Exhaustion in All Protocols — (File: `src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` used by every protocol instance (`DKG`, `reshare`, `refresh`, `presign`, `sign`, `CKD`) has no bound on the number of distinct `MessageHeader` keys it will store. Any participant in a session can call `Protocol::message()` with crafted raw bytes that parse to arbitrary `(channel_tag, waitpoint)` headers, causing the internal `HashMap` to grow without limit and exhaust the honest party's memory.

---

### Finding Description

`MessageBuffer` is a `HashMap<MessageHeader, SubMessageQueue>` that backs every protocol's incoming message queue. [1](#0-0) 

The public entry point `Protocol::message()` calls `push_message`, which parses a `MessageHeader` from the raw bytes and unconditionally inserts a new queue entry for every distinct header seen: [2](#0-1) 

The insertion path: [3](#0-2) 

A `MessageHeader` is a 32-byte `ChannelTag` plus an 8-byte `Waitpoint` (`u64`). An attacker can craft messages with any of the 2^64 possible waitpoint values (or arbitrary channel tags), each producing a distinct key and a new `SubMessageQueue` allocation. There is no capacity cap, no eviction, and no per-sender rate limit anywhere in this path.

The repository's own test suite confirms this behavior and explicitly names it as an attack: [4](#0-3) 

The test asserts `messages.len() == 512` after 512 injected messages — demonstrating the HashMap grows proportionally to attacker-controlled input with no resistance.

---

### Impact Explanation

Every honest party running `keygen`, `reshare`, `refresh`, `presign`, `sign`, or `ckd` holds a live `Comms`/`MessageBuffer` for the duration of the protocol. A single malicious co-participant can stream messages with distinct fabricated headers at any time during the protocol. Each message allocates a new `SubMessageQueue` (an `Arc<Mutex<UnboundedReceiver>>` + `UnboundedSender` pair) and a new `HashMap` entry. Sending millions of such messages causes the honest party's process to exhaust heap memory and be killed by the OS, permanently denying it the ability to complete DKG, signing, or any other operation.

This matches: **Medium — Griefing or resource-exhaustion by a malicious participant causing unbounded memory beyond documented behavior.**

---

### Likelihood Explanation

- The `Protocol::message()` method is the standard API for delivering network messages; every integration must call it for every received message.
- A malicious participant is explicitly within the threat model (the library tolerates up to `floor((n-1)/3)` malicious parties in DKG/broadcast).
- No authentication, rate-limiting, or header validation is applied before insertion.
- The attack requires only the ability to send raw bytes to an honest party's protocol instance — a capability every co-participant has by definition.
- The repository's own test (`attacker_can_fill_message_buffer_with_unused_waitpoints`) demonstrates the attack works and makes no attempt to mitigate it.

---

### Recommendation

1. **Cap the `MessageBuffer`**: Enforce a maximum number of distinct `MessageHeader` keys (e.g., `O(n × rounds)` where `n` is the participant count and `rounds` is the protocol's round count). Reject or drop messages that would exceed this cap.
2. **Per-sender accounting**: Track how many distinct headers each `Participant` has contributed and drop messages from senders that exceed a reasonable per-sender limit.
3. **Validate headers eagerly**: Before inserting, check that the `(channel_tag, waitpoint)` pair is one the honest party actually expects at the current protocol stage, and discard anything outside that set.

---

### Proof of Concept

The existing test in `src/protocol/internal.rs` already serves as a proof of concept: [4](#0-3) 

To escalate to OOM, replace `attack_count = 512` with `attack_count = u64::MAX` (or any large value) and observe heap exhaustion. In a real protocol session, the attacker simply sends messages with sequentially incrementing waitpoint values (e.g., `1_000_000`, `1_000_001`, …) to the honest party's `Protocol::message()` handler. Each call inserts a new `SubMessageQueue` into the `HashMap` with no eviction, growing memory without bound until the process is killed.

    fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
        let comms = Comms::new();
        let attacker = Participant::from(99_u32);
        let attack_count = 512_u64;

        for i in 0..attack_count {
            let header =
                MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
            let mut message = header.to_bytes().to_vec();
            message.extend_from_slice(&i.to_le_bytes());

            // Attacker injects messages for waitpoints the honest code never polls.
            comms.push_message(attacker, message);
        }

        let messages = comms
            .incoming
            .messages
            .lock()
            .expect("lock should not fail");

        assert!(messages.len() == usize::try_from(attack_count).unwrap());
    }
