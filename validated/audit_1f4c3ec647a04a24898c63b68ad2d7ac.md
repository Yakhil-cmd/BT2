Audit Report

## Title
Unbounded `inflight_requests` JoinSet + `read_to_end(128 MB)` per stream enables per-connection OOM by a single Byzantine peer — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary

`start_stream_acceptor` spawns one unbounded Tokio task per accepted QUIC bidi stream into an uncapped `JoinSet`. Each task calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`, which accumulates all received bytes into a heap-allocated `Vec<u8>`. A single Byzantine subnet node holding a valid TLS certificate can open `MAX_CONCURRENT_BIDI_STREAMS = 1000` streams and drip data on each, forcing up to 1000 × 128 MB ≈ 128 GB of heap allocation and OOM-killing the victim replica.

## Finding Description

**Unbounded task spawning.** `start_stream_acceptor` runs an event loop that calls `conn_handle.conn().accept_bi()` and immediately spawns a task for every accepted stream with no cap check: [1](#0-0) [2](#0-1) 

The comment at lines 52–56 explicitly acknowledges the risk of buffering up to the stream limit but does not account for the per-stream memory cost.

**`read_to_end` accumulates all bytes into a heap `Vec`.** Each spawned task calls: [3](#0-2) 

Quinn's `read_to_end(limit)` allocates a `Vec<u8>` that grows incrementally as bytes arrive. As the receiver reads, QUIC flow-control window updates are sent back to the sender, allowing it to push more data. The `Vec` accumulates every byte read — it is not freed until the task completes.

**Flow-control windows limit rate, not total allocation.** `STREAM_RECEIVE_WINDOW = 4 MB` and `RECEIVE_WINDOW = 200 MB` bound how much unread data can be in-flight at any instant: [4](#0-3) 

Because `read_to_end` continuously reads and acknowledges data, the windows slide forward. The `Vec` accumulates every byte that has been read — the windows do not cap total heap allocation.

**Production constants enabling the attack:** [5](#0-4) [6](#0-5) 

Their product (128 GB) is the theoretical per-connection memory ceiling.

## Impact Explanation

An OOM-kill of the replica process terminates its consensus participation. The attack is repeatable: after the replica restarts, the attacker can immediately reconnect (TLS authentication only requires a valid subnet certificate) and repeat. A single Byzantine node can execute this against every peer it is connected to simultaneously, potentially degrading subnet throughput and finality latency. This matches the allowed impact: **High — Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation

The precondition is possession of one valid subnet TLS certificate, corresponding to a single Byzantine subnet node — explicitly within the IC's Byzantine fault model. No governance majority, no threshold key, and no DDoS volume is required. The attack is rate-limited by the 200 MB connection window but fully deterministic: the attacker sends 4 MB chunks per stream, waits for window updates, and repeats. No timing luck is needed. The TLS configuration confirms only subnet members can connect: [7](#0-6) 

## Recommendation

1. **Cap `inflight_requests` before spawning**: Check `inflight_requests.len() >= MAX_CONCURRENT_BIDI_STREAMS` before calling `inflight_requests.spawn(...)`. If the cap is reached, stop calling `accept_bi` until a slot is free (back-pressure) or drop the stream. The existing comment at lines 52–56 already identifies this as the correct fix direction.
2. **Bound per-connection memory**: Track bytes-in-flight per connection and close the connection if the aggregate exceeds a configured limit (e.g., `MAX_CONCURRENT_BIDI_STREAMS × per_message_budget`).
3. **Consider streaming deserialization**: Instead of `read_to_end` into a full `Vec`, read a length prefix first and reject streams whose declared size exceeds a per-connection budget before allocating.

## Proof of Concept

```
1. Attacker (Byzantine subnet node with valid TLS cert) connects to victim replica.
2. Attacker opens 1000 bidi streams simultaneously (within MAX_CONCURRENT_BIDI_STREAMS=1000).
3. On each stream, attacker sends data in 4 MB chunks (respecting STREAM_RECEIVE_WINDOW),
   waiting for window updates before sending the next chunk.
4. start_stream_acceptor spawns 1000 tasks into inflight_requests (no cap check).
5. Each task blocks in recv_stream.read_to_end(128*1024*1024), accumulating bytes into a Vec.
6. As each Vec grows toward 128 MB, total heap pressure approaches 1000 × 128 MB = 128 GB.
7. The OS OOM-killer terminates the replica process before all streams complete.
```

**Deterministic test plan**: Mock a `quinn::Connection` that accepts 1000 streams each yielding `MAX_MESSAGE_SIZE_BYTES - 1` bytes; assert that `start_stream_acceptor` either rejects streams beyond a cap or that process RSS stays below a configured bound. Alternatively, run a local two-node IC replica setup where one node's transport layer is instrumented to open 1000 streams with slow drip writes and observe RSS growth on the victim node.
