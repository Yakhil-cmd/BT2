### Title
Missing Proof-of-Correct-Computation for Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates `(big_y, big_c)` contributions from every participant without any cryptographic proof that each contribution was honestly computed from the participant's actual private share. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a structurally valid but cryptographically incorrect `CKDOutput`. Honest parties that rely on this output will derive a wrong confidential key with no indication of failure.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–57), the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and unconditionally adds them into the running totals:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

Each honest participant is supposed to compute:

```
big_y  = y_i · G
big_c  = x_i · H(pk ‖ app_id) + y_i · app_pk
norm_* = λ_i · *
``` [2](#0-1) 

No ZK proof (e.g., a DLOG-equality proof showing that the same witness `x_i` was used in both `x_i · H(pk ‖ app_id)` and the participant's known public share `x_i · G`) is attached to the message, and none is verified by the coordinator. The coordinator has no way to distinguish a correctly formed contribution from an arbitrary group element.

Compare this with the OT-based triple generation, where every analogous contribution is accompanied by a `dlogeq` proof that is explicitly verified before the value is accumulated:

```rust
if !dlogeq::verify(
    &mut transcript.fork(b"dlogeq0", &from.bytes()),
    statement,
    their_phi_proof,
)? {
    return Err(ProtocolError::AssertionFailed(...));
}
big_c_v[i] += big_c_j;
``` [3](#0-2) 

The CKD protocol performs no equivalent check.

---

### Impact Explanation

The final `CKDOutput` is the sum of all participant contributions. If participant `P_m` replaces its honest `norm_big_c` with an arbitrary element `δ`, the coordinator produces:

```
total_big_c = msk · H(pk ‖ app_id) + (Σ λ_i · y_i) · app_pk + δ
```

When the application unmasks with `app_sk`:

```
total_big_c − app_sk · total_big_y = msk · H(pk ‖ app_id) + δ
```

The derived confidential key is shifted by `δ`, which is attacker-controlled. The output is structurally valid (non-identity group elements, correct serialization) so no downstream error is raised. Honest parties silently accept a wrong key.

**Impact category:** High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.

---

### Likelihood Explanation

Any participant in the CKD session is an attacker-controlled entry point. The participant sends a single private message to the coordinator; no additional privilege, leaked key, or cryptographic break is required. The attack requires only that the participant deviate from the protocol by sending an arbitrary `CKDOutput`, which is trivially achievable by any library caller who controls their own process.

---

### Recommendation

Require each participant to attach a proof of correct computation alongside `(norm_big_y, norm_big_c)`. Concretely, a DLOG-equality proof demonstrating that the same scalar `x_i` was used to compute both the participant's public verification share (known from DKG output) and the `x_i · H(pk ‖ app_id)` component of `big_c` would close this gap. The coordinator must verify this proof before accumulating the contribution, mirroring the pattern already used in triple generation.

---

### Proof of Concept

1. Run a CKD session with participants `[P_1, P_2, P_3]` and coordinator `P_1`.
2. `P_2` (malicious) intercepts the send step in `do_ckd_participant` and replaces `norm_big_c` with the identity element (or any chosen group element `δ`).
3. `P_1` (coordinator) executes `do_ckd_coordinator`, receives the tampered `CKDOutput` from `P_2`, and adds it without error.
4. The returned `CKDOutput` satisfies `ckd_output.unmask(app_sk) ≠ msk · H(pk ‖ app_id)`.
5. Any TEE or application that uses this output to derive a deterministic secret obtains a wrong value with no indication of failure, permanently corrupting the derived key for that `(pk, app_id)` pair. [4](#0-3)
