### Title
`increased_enough` Premature Overflow Rejects Valid `max_l2_gas_price` Fee-Escalation Replacements — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

`Mempool::increased_enough` multiplies `existing_value * percentage` before dividing by 100. When `existing_max_l2_gas_price > u128::MAX / percentage`, the intermediate product overflows u128 and `checked_mul` returns `None`, causing the function to return `false` and reject the replacement — even when the mathematically correct threshold `existing_value + ⌈existing_value × percentage / 100⌉` fits comfortably within u128 and the incoming value satisfies it.

---

### Finding Description

The computation in `increased_enough` is:

```
existing_value × percentage   ← overflows here
    ÷ 100
    + existing_value          ← final result would NOT overflow
``` [1](#0-0) 

When `checked_mul` returns `None`, the `else` branch unconditionally returns `false`, rejecting the replacement. [2](#0-1) 

**Type analysis — tip is immune, `max_l2_gas_price` is not:**

`tip` is `Tip(u64)`, widened to `u128` via `u128::from(tx.tip.0)`. Its maximum is `u64::MAX ≈ 1.8 × 10^19`. Since `u64::MAX × 100 ≈ 1.8 × 10^21 ≪ u128::MAX ≈ 3.4 × 10^38`, the multiply can never overflow for tip. The question's proof idea of `existing_tip = u128::MAX / 2 + 1` is **impossible** — tip is u64. [3](#0-2) 

`max_l2_gas_price` is `GasPrice(u128)`, used directly as u128. It can be up to `u128::MAX`. [4](#0-3) 

**Concrete arithmetic (percentage = 10, the production default):**

| | Value |
|---|---|
| `existing_max_l2_gas_price` | `u128::MAX / 2 + 1 = 170141183460469231731687303715884105728` |
| `existing_value × 10` | `1701411834604692317316873037158841057280` → **overflows u128** |
| Correct required increase | `⌈170141183460469231731687303715884105728 / 10⌉ = 17014118346046923173168730371588410573` |
| Correct threshold | `187155301806516154904856034087472516301` |
| `u128::MAX` | `340282366920938463463374607431768211455` |
| `u128::MAX ≥ threshold?` | **YES** — a valid replacement exists |

The code returns `false`; the correct answer is `true`.

**Gateway does not bound `max_l2_gas_price` from above.** The stateless validator only enforces a *lower* bound (`max_l2_gas_price >= min_gas_price`). No upper bound exists. [5](#0-4) 

**The existing overflow test covers a different (correct) case.** `test_fee_escalation_invalid_replacement_overflow_gracefully_handled` tests `existing_max_l2_gas_price ∈ {u128::MAX - 10, u128::MAX}`. For those values, the correct threshold itself overflows u128 (no valid replacement can exist), so rejection is correct. The test does not cover the range `(u128::MAX / percentage, u128::MAX / (1 + percentage/100)]` where the intermediate overflows but the final result does not. [6](#0-5) 

The production default `fee_escalation_percentage = 10` is confirmed in both deployment configs. [7](#0-6) 

---

### Impact Explanation

A transaction with `max_l2_gas_price` in the range `(u128::MAX / 10, u128::MAX / 1.1]` (≈ `3.4 × 10^37` to `3.09 × 10^38`) sitting in the mempool causes `validate_fee_escalation` to return `Err(MempoolError::DuplicateNonce)` for any replacement attempt, even one that genuinely satisfies the 10% escalation requirement. The existing transaction cannot be replaced via fee escalation. [8](#0-7) 

Impact category: **High — valid transactions rejected before sequencing.**

---

### Likelihood Explanation

**Low.** The trigger requires `max_l2_gas_price > u128::MAX / 10 ≈ 3.4 × 10^37`. Real-world gas prices are many orders of magnitude smaller. There is no upper-bound gateway check preventing such a transaction from being submitted, but no legitimate wallet or user would produce such a value under normal conditions. The scenario is theoretically reachable but practically improbable.

---

### Recommendation

Reorder the arithmetic to divide before multiplying, eliminating the premature overflow:

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);
    // Divide first to avoid intermediate overflow.
    // required_increase = ⌈existing_value * percentage / 100⌉
    let required_increase = existing_value / 100 * percentage
        + existing_value.rem_euclid(100) * percentage / 100
        + if (existing_value % 100) * percentage % 100 != 0 { 1 } else { 0 };
    let Some(threshold) = existing_value.checked_add(required_increase) else {
        return false; // threshold genuinely overflows u128 — no valid replacement possible
    };
    incoming_value >= threshold
}
```

Alternatively, use 256-bit arithmetic (e.g., via `u256` or `checked_mul` on a wider type) for the intermediate product, then truncate.

### Proof of Concept

The question's proof idea contains a factual error: `existing_tip = u128::MAX / 2 + 1` is impossible because `Tip` is `u64`. The correct PoC targets `max_l2_gas_price`:

```rust
#[test]
fn test_increased_enough_premature_overflow() {
    // existing_max_l2_gas_price * 10 overflows u128,
    // but the correct threshold (≈1.87e38) is well below u128::MAX (≈3.4e38).
    let existing_gas_price: u128 = u128::MAX / 2 + 1;
    let incoming_gas_price: u128 = u128::MAX;

    // Manually replicate increased_enough logic:
    let percentage: u128 = 10;
    let result = existing_gas_price
        .checked_mul(percentage)          // returns None — overflows
        .map(|s| s.div_ceil(100))
        .and_then(|inc| existing_gas_price.checked_add(inc));

    assert!(result.is_none(), "intermediate overflow confirmed");

    // But the correct threshold does NOT overflow:
    let correct_increase = (existing_gas_price / 100) * percentage
        + ((existing_gas_price % 100) * percentage).div_ceil(100);
    let correct_threshold = existing_gas_price + correct_increase;
    assert!(
        incoming_gas_price >= correct_threshold,
        "u128::MAX satisfies the 10% escalation requirement — replacement should be admitted"
    );
    // The production code returns false here, incorrectly rejecting the replacement.
}
```
