### Title
Gateway L2 Gas Price Admission Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` compares a transaction's `max_price_per_unit` against the **current block's** `l2_gas_price` (the price that was already used in the last committed block) rather than the **next block's** `next_l2_gas_price` (the EIP-1559-derived price that will actually govern the block being built). The code even carries a self-documenting TODO acknowledging the wrong field is used. This causes the gateway to admit transactions that will fail at blockifier pre-validation, and to reject transactions that are valid for the next block, whenever the EIP-1559 price adjustment is non-trivial.

---

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;          // ← price used IN the previous block
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,   // ← wrong reference
)?;
```

`BlockInfo::gas_prices.strk_gas_prices.l2_gas_price` is the price that was applied to transactions **inside** the latest committed block. The price that will be applied to the **next** block is `BlockHeaderWithoutHash::next_l2_gas_price`, which is computed via EIP-1559 from the previous block's gas consumption and stored separately in the block header. These two values diverge whenever the block is above or below the gas target.

The same stale `block_info` is forwarded into `run_validate_entry_point`, which builds a `BlockContext` from it (with only `block_number` bumped). So the blockifier's `check_fee_bounds` inside the gateway also uses the wrong price, making the gateway's full validation pipeline internally consistent but systematically misaligned with the price the batcher will use.

The `next_l2_gas_price` is a first-class field in `BlockHeaderWithoutHash` and `StorageBlockHeader`, is propagated through P2P (`SignedBlockHeader.next_l2_gas_price`), and is the value the consensus orchestrator reads back via `try_sync` to seed `context.l2_gas_price`. It is the authoritative price for the next block.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

**Case 1 — High congestion (`next_l2_gas_price > l2_gas_price`):**
The admission threshold is computed as `min_gas_price_percentage × l2_gas_price`, which is lower than the correct `min_gas_price_percentage × next_l2_gas_price`. Transactions whose `max_price_per_unit` falls in the gap between the two thresholds pass gateway validation and enter the mempool, but will be rejected by the batcher's blockifier `check_fee_bounds` with `MaxGasPriceTooLow`. These transactions pollute the mempool and waste batcher execution slots.

**Case 2 — Low congestion (`next_l2_gas_price < l2_gas_price`):**
The threshold is higher than it should be. Transactions that are perfectly valid for the next block (their `max_price_per_unit ≥ next_l2_gas_price`) are rejected at the gateway with `GAS_PRICE_TOO_LOW`. This is a systematic denial of service for users during periods of falling gas prices.

The EIP-1559 formula can move the price by up to `price / gas_price_max_change_denominator` per block. Under sustained high load the gap compounds across multiple blocks, widening the set of incorrectly admitted or rejected transactions.

---

### Likelihood Explanation

The EIP-1559 mechanism is always active. Any block that is above or below the gas target produces a `next_l2_gas_price` that differs from `l2_gas_price`. This is the normal operating condition of the network, not an edge case. The bug is therefore triggered on every block where gas usage deviates from the target, which is the common case.

---

### Recommendation

Replace the `l2_gas_price` read with `next_l2_gas_price` from the block header. The `GatewayFixedBlockStateReader` trait should expose the full `BlockHeaderWithoutHash` (or at minimum `next_l2_gas_price`), and `validate_resource_bounds` should use that value:

```rust
// In validate_resource_bounds:
let next_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_header()   // new method returning BlockHeaderWithoutHash
    .await?
    .next_l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    next_l2_gas_price,
)?;
```

The same fix must be applied to `run_validate_entry_point`: the `BlockInfo` passed to `BlockContext::new` must have its `gas_prices.strk_gas_prices.l2_gas_price` overwritten with `next_l2_gas_price` so that the blockifier's `check_fee_bounds` inside the gateway uses the same price the batcher will use.

---

### Proof of Concept

1. Observe a block N with `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (block was above gas target).
2. Submit an invoke V3 transaction with `l2_gas.max_price_per_unit = P` (equal to the stale price, below the correct next price `P'`).
3. Gateway `validate_resource_bounds` computes threshold as `min_gas_price_percentage × P`. With `min_gas_price_percentage = 100`, threshold = `P`. The transaction passes (`P >= P`).
4. `run_validate_entry_point` builds a `BlockContext` with `block_info.gas_prices.strk_gas_prices.l2_gas_price = P`. Blockifier `check_fee_bounds` checks `max_price_per_unit (P) >= actual_gas_price (P)` — passes.
5. Transaction enters the mempool.
6. Batcher builds block N+1 with `l2_gas_price = P'` (from `next_l2_gas_price` of block N).
7. Blockifier `check_fee_bounds` checks `max_price_per_unit (P) >= actual_gas_price (P')` — **fails** with `MaxGasPriceTooLow`.
8. Transaction is dropped from the block despite having passed full gateway validation.

The reverse (valid transaction rejected) follows symmetrically when `P' < P`.

```rust
    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
```
