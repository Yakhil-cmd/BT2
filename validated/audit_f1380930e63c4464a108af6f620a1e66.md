### Title
`execute_call` Hardcodes `TransactionInfo::Deprecated(DeprecatedTransactionInfo::default())` for All `starknet_call` Requests, Returning Wrong `get_execution_info` Values — (`crates/apollo_rpc_execution/src/lib.rs`)

---

### Summary

`execute_call` unconditionally constructs a zeroed-out deprecated transaction context for every `starknet_call` RPC invocation, regardless of the target contract's version. Any contract — Cairo 0 or Sierra — that reads `get_execution_info` / `get_tx_info` during a `starknet_call` receives fabricated values: `version=0`, `max_fee=0`, `nonce=0`, `sender_address=0`, `transaction_hash=0`. This is not attacker-forced; it is the universal, hardcoded behavior for the entire `starknet_call` path.

---

### Finding Description

In `execute_call` at line 278:

```rust
// TODO(yair): fix when supporting v3 transactions
let tx_info = TransactionInfo::Deprecated(DeprecatedTransactionInfo::default());
``` [1](#0-0) 

This single line is the entire transaction context for every `starknet_call` execution. The `DeprecatedTransactionInfo::default()` produces:
- `version = TransactionVersion::ZERO`
- `max_fee = Fee(0)`
- `nonce = Nonce(0)`
- `sender_address = ContractAddress::default()` (zero)
- `transaction_hash = TransactionHash(0)`

This `tx_info` is wrapped in a `TransactionContext` and passed to `EntryPointExecutionContext::new_invoke`: [2](#0-1) 

The blockifier's syscall handler reads `tx_info` directly from `self.base.context.tx_context.tx_info` when serving `get_execution_info` / `get_execution_info_v2` syscalls: [3](#0-2) 

For Sierra contracts using `get_execution_info_v2`, the handler branches on `TransactionInfo::Deprecated(_)` and returns all V3 fields (`resource_bounds`, `tip`, `paymaster_data`, etc.) as their defaults: [4](#0-3) 

By contrast, the correct path for a real V3 invoke transaction uses `TransactionInfo::Current(CurrentTransactionInfo { ... })` with actual fields populated from the transaction: [5](#0-4) 

---

### Impact Explanation

Every `starknet_call` RPC response for a contract that reads `get_execution_info` contains authoritative-looking but entirely fabricated values. Concretely:

| Field | Expected | Actual |
|---|---|---|
| `tx.version` | caller-supplied or 3 | `0` |
| `tx.max_fee` | caller-supplied | `0` |
| `tx.nonce` | caller-supplied | `0` |
| `tx.account_contract_address` | caller-supplied | `0` |
| `tx.transaction_hash` | computed | `0` |

This falls squarely under: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

Contracts that gate logic on `tx.version` (e.g., to distinguish V1 from V3 fee modes), `tx.nonce` (replay protection checks in view functions), or `tx.account_contract_address` (access control) will return wrong results when called via `starknet_call`. Callers relying on these results for off-chain decisions (e.g., dApp UIs, indexers, multicall aggregators) receive incorrect data.

---

### Likelihood Explanation

The bug is triggered by any `starknet_call` to any contract that reads `get_execution_info`. No special attacker capability is required — this is the universal behavior of the code path. The TODO comment confirms the developers are aware but have not yet fixed it. [1](#0-0) 

---

### Recommendation

Replace the hardcoded deprecated default with a properly constructed `TransactionInfo` that reflects the caller-supplied parameters (version, max_fee, nonce, sender address, resource bounds). For `starknet_call`, a synthetic `TransactionInfo::Current` with caller-provided fields (or sensible simulation defaults) should be constructed, matching the same logic used in `execute_transactions` → `to_blockifier_tx` → `create_tx_info`.

---

### Proof of Concept

A Rust integration test calling `execute_call` against a Sierra contract whose entry point executes `get_execution_info` and returns `tx.version` would observe `Felt::ZERO` in the return data instead of the expected version (e.g., `Felt::THREE`). The existing blockifier test infrastructure at `crates/blockifier/src/execution/syscalls/syscall_tests/get_execution_info.rs` demonstrates the correct `TransactionInfo::Current` construction needed for V3 contracts.
