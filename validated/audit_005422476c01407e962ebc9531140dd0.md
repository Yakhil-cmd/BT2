### Title
Greedy Regex in `parseOfferData` Silently Substitutes Last Offer Blob for First, Enabling Malicious Offer Acceptance — (`packages/gui/src/components/offers/OfferImport.tsx` and `packages/gui/src/components/offers2/OfferBuilderImport.tsx`)

### Summary

The `parseOfferData` function in both `OfferImport.tsx` and `OfferBuilderImport.tsx` uses a greedy dotAll regex to extract a bech32 offer blob from raw file content. Because the leading capture group `(?<leading>.*)` is greedy, the regex engine always resolves to the **last** `offer1…` blob in the file, not the first. An attacker who can deliver a crafted `.offer` file containing two valid offer blobs — a decoy first blob and a malicious second blob — will cause the GUI to silently process and display the malicious offer to the user.

### Finding Description

Both files contain an identical `parseOfferData` function:

```ts
const matches = data.match(
  /(?<leading>.*)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
);
``` [1](#0-0) [2](#0-1) 

With the `s` (dotAll) flag active, `.` matches newlines. The `(?<leading>.*)` group is greedy: the engine first consumes the entire input, then backtracks minimally to satisfy `offer1[…]+`. Minimal backtracking means it stops at the **rightmost** (last) occurrence of `offer1`. For input:

```
offer1<LEGITIMATE_BLOB> offer1<MALICIOUS_BLOB>
```

the engine sets `leading = "offer1<LEGITIMATE_BLOB> "` and `offer = "offer1<MALICIOUS_BLOB>"`. The legitimate blob is never seen.

The extracted blob is then passed directly to `getOfferSummary` and placed in navigation state:

```ts
const { data: response } = await getOfferSummary({ offerData });
...
navigate(navigationPath, {
  state: { offerData, offerSummary, offerFilePath, imported: true },
});
``` [3](#0-2) 

There is no post-extraction validation that the blob is the one the user intended to import, no count check, and no warning when multiple `offer1` patterns are present.

### Impact Explanation

A malicious counterparty sends a crafted `.offer` file. The file's first blob is a legitimate offer matching the verbally agreed terms (e.g., 10 XCH for a specific NFT). The second blob is a malicious offer (e.g., 10 XCH for a worthless CAT, or the same NFT for 0.001 XCH). The user opens the file through the normal import workflow. The GUI silently displays the malicious offer's summary. If the user does not carefully compare the displayed terms against the agreed terms — a realistic failure mode when the attacker crafts the malicious offer to look superficially similar — the user accepts the malicious offer, resulting in an unauthorized transfer of XCH, CAT, or NFT.

This satisfies: **High — Corruption or spoofing of offer state that causes a user to approve the wrong asset, amount, or status.**

### Likelihood Explanation

The attack requires only that the attacker deliver a crafted file to the user, which is the normal channel for offer exchange in Chia (email, messaging, marketplace). No special privileges, no key material, and no host compromise are needed. The regex behavior is deterministic and 100% reproducible. The only friction is that the user must not carefully read the displayed offer summary, which is a realistic failure mode in high-volume or time-pressured trading.

### Recommendation

Replace the greedy `(?<leading>.*)` with a non-greedy alternative, or use a first-match approach:

```ts
// Option 1: non-greedy leading group
const matches = data.match(
  /(?<leading>.*?)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
);

// Option 2: explicit first-match with exec
const re = /offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+/g;
const first = re.exec(data);
```

Additionally, if more than one `offer1` blob is detected in the file, the import should be rejected with an explicit error rather than silently picking one.
