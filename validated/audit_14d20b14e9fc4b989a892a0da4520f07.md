### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Unchecked `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` to the pool with no validation against `maxAdminSpreadFeeE6`, while the parallel `setPoolAdminFees` path enforces that cap. A pool admin can set per-bin additional fees up to `uint16` max (65535 = 6.5535%) regardless of what the factory owner has configured as the admin fee ceiling.

### Finding Description
`setPoolAdminFees` checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` and reverts with `AdminFeeTooHigh`. `setPoolBinAdditionalFees` performs no equivalent check. Both paths are callable by the pool admin. The bin additional fees are applied additively to `baseFeeX64` in every swap iteration across all four swap directions (`buyToken0SpecifiedIn`, `buyToken0SpecifiedOut`, `buyToken1SpecifiedIn`, `buyToken1SpecifiedOut`). The stored values are written directly to `BinState.addFeeBuyE6` / `addFeeSellE6` with no post-write validation. [8](#0-7) 

### Impact Explanation
Every swap through the affected bin pays `baseFee + addFee` instead of `baseFee`. With `addFeeBuyE6 = 65535`, traders pay an additional 6.5535% per bin on top of the oracle spread. If the factory owner has lowered `maxAdminSpreadFeeE6` (e.g., to 1000 = 0.1%), the pool admin can still impose 6.5535% — a 65× excess over the intended cap. The surplus accrues as LP-side spread surplus and is collectible by the admin via `collectPoolFees`, constituting direct extraction of trader principal.

### Likelihood Explanation
The pool admin is a semi-trusted role. The factory owner's ability to cap admin fees via `maxAdminSpreadFeeE6` is the primary protection for traders against admin extraction. The missing check is a single-transaction exploit requiring only pool admin access to a registered pool.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, define a separate `maxAdminBinFeeE6` cap if the protocol intends bin additional fees to have a different ceiling than the global admin spread fee.

### Proof of Concept
1. Factory owner calls `setFeeCaps(..., newMaxAdminSpreadFeeE6 = 1000, ...)` — limiting admin spread to 0.1%.
2. Pool admin calls `setPoolAdminFees(pool, 1001, 0)` → reverts `AdminFeeTooHigh`. Cap is enforced.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` → succeeds. No revert.
4. Trader executes a swap through bin 0. The effective fee is `baseFee + 6.5535%`, far exceeding the 0.1% cap the factory owner configured.
5. The excess fee accrues as spread surplus and is swept to the admin fee destination via `collectPoolFees`.
