### Title
Unprivileged caller can permanently suppress block rewards for all stakers via `update_rewards` with `disable_rewards=true` — (File: src/staking/staking.cairo)

---

### Summary

The spec mandates "Only starkware sequencer" access control for `update_rewards`, but the production implementation contains **no caller check**. Any address can call `update_rewards(staker_address, disable_rewards=true)` first in any block, consuming the global `last_reward_block` gate and permanently discarding that block's rewards for all stakers.

---

### Finding Description

The `update_rewards` function in `StakingRewardsManagerImpl` performs no `get_caller_address()` validation: [1](#0-0) 

The only guard is a global, single-slot `last_reward_block` check:

```
current_block_number > self.last_reward_block.read()
```

If this passes, the function immediately writes the current block number to `last_reward_block` **before** checking `disable_rewards`: [2](#0-1) 

When `disable_rewards == true`, the function returns after writing `last_reward_block` but **without distributing any rewards**. Because `last_reward_block` is a single global slot (not per-staker), every subsequent call to `update_rewards` in the same block — for any staker — fails with `REWARDS_ALREADY_UPDATED`.

The spec explicitly states the intended access control: [3](#0-2) 

The interface definition confirms the function is publicly exposed with no restriction: [4](#0-3) 

---

### Impact Explanation

An attacker who calls `update_rewards(any_valid_active_staker, disable_rewards=true)` first in a block:

1. Passes the `last_reward_block` gate (block is new).
2. Writes `last_reward_block = current_block`.
3. Returns early — zero rewards distributed to anyone.
4. The legitimate sequencer's subsequent call fails with `REWARDS_ALREADY_UPDATED`.

That block's rewards are **permanently lost** — there is no catch-up or retroactive distribution mechanism. Repeating this every block starves all active stakers of all consensus-era block rewards indefinitely.

Impact: **High — Permanent freezing of unclaimed yield** (matches allowed scope).

---

### Likelihood Explanation

- Requires no privilege, no leaked key, no bridge interaction.
- The attacker only needs to submit a transaction before the sequencer's `update_rewards` call in each block.
- On Starknet, transaction ordering within a block is sequencer-controlled, but the sequencer itself is the intended caller — an attacker who submits a transaction with a higher fee or exploits mempool ordering can front-run it.
- The attack is cheap (one contract call per block) and fully repeatable.

---

### Recommendation

Add a caller check at the top of `update_rewards` that restricts invocation to the authorized sequencer address (or a designated rewards-manager role), consistent with the spec's stated access control:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    let caller = starknet::get_caller_address();
    assert!(caller == self.sequencer_address.read(), "{}", Error::UNAUTHORIZED);
    ...
}
```

Alternatively, gate the function behind the existing roles/access-control component already present in the contract.

---

### Proof of Concept

1. Deploy the staking contract with at least one active staker past the K-epoch activation window and with consensus rewards enabled.
2. From an arbitrary EOA, call `update_rewards(staker_address, disable_rewards=true)` at the start of a block (before the sequencer's call).
3. Observe `last_reward_block` is now set to the current block.
4. The sequencer's `update_rewards(staker_address, disable_rewards=false)` call reverts with `REWARDS_ALREADY_UPDATED`.
5. Claim rewards — balance is unchanged; the block's rewards are gone.
6. Repeat for every block. Cumulative staker rewards remain at zero indefinitely.

The existing test suite already demonstrates the `REWARDS_ALREADY_UPDATED` revert behavior when a second call is made in the same block: [5](#0-4) 

The only missing piece in the current tests is that the *first* call is always made by a trusted party — the tests never verify that an untrusted caller is rejected, because no such check exists.
