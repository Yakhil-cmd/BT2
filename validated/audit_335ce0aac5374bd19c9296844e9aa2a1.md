### Title
Pool Admin Bypasses Fee Cap via Uncapped Per-Bin Additional Fees in `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

The Metric OMM fee cap system enforces per-component upper bounds on `adminSpreadFeeE6` and `protocolSpreadFeeE6`, but the per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) set via `setPoolBinAdditionalFees` are completely uncapped. These per-bin fees are added directly on top of the base spread fee in every swap, allowing the pool admin to charge traders more than the protocol's intended maximum fee.

### Finding Description

The factory enforces fee caps through two checks in `setPoolAdminFees`:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, `setPoolBinAdditionalFees` passes `addFeeBuyE6` and `addFeeSellE6` directly to the pool with **no cap validation**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

On the pool side, `setBinAdditionalFees` only validates the bin index range, not the fee values:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

These per-bin fees are then added directly on top of the oracle-derived base spread fee in every swap path:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [5](#0-4) 

The `uint16` type allows values up to 65,535, which in E6 units equals **6.5535%** of additional fee per bin, completely outside the fee cap system's control.

The analog to the external bug is exact: just as `_requireEnoughDepositCap` checked only the current deposit amount without accounting for the cumulative total already deposited, the fee cap system here checks only the base spread fee components (`adminSpreadFeeE6`, `protocolSpreadFeeE6`) without accounting for the per-bin additional fees that are added on top during every swap.

### Impact Explanation

The hard cap `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) is enforced per component. [6](#0-5) 

A pool admin can:
1. Set `adminSpreadFeeE6` to `maxAdminSpreadFeeE6` = 200,000 (20%) — passes the cap check.
2. Call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` (6.5535%) for any active bin — no cap check.

The effective fee paid by a trader in that bin becomes: oracle spread + 6.5535% (per-bin) + up to 40% (base spread) + notional fee — far exceeding the intended 20% admin cap. The excess fee is extracted from traders as a direct loss of principal, accruing to the pool's LP surplus (spread fee) rather than being tracked as a collectable admin fee, so it cannot be reversed.

### Likelihood Explanation

The pool admin is a semi-trusted role that can call `setPoolBinAdditionalFees` at any time with no timelock and no additional approval. The call path is a single transaction: `poolAdmin → factory.setPoolBinAdditionalFees → pool.setBinAdditionalFees`. Any pool admin who is malicious or compromised can immediately set per-bin fees to `type(uint16).max` for the active bin, affecting all subsequent swaps.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and symmetrically in `_unpackAndValidateBinStates` at pool creation) bounding `addFeeBuyE6` and `addFeeSellE6` to `maxAdminSpreadFeeE6` (or a dedicated hard cap constant):

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The same bound should be applied when validating bin data arrays in `_unpackAndValidateBinStates` during `createPool`. [7](#0-6) 

### Proof of Concept

```solidity
// Setup: pool with maxAdminSpreadFeeE6 = 200_000 (20%)
// Admin sets base fee to cap
factory.setPoolAdminFees(pool, 200_000, 0); // passes: 200_000 <= maxAdminSpreadFeeE6

// Admin sets per-bin additional fee to uint16 max — no cap check, passes silently
vm.prank(admin);
factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
// addFeeBuyE6 = 65535 = 6.5535% additional fee, unchecked

// Now any trader swapping through bin 0 pays:
// oracle_spread + 6.5535% (addFeeBuyE6) + 20% (adminSpreadFee) + protocol_fee
// Total far exceeds the intended 20% admin cap
``` [2](#0-1) [8](#0-7)
