### Title
Gateway L2 Gas Price Admission Uses Stale Previous-Block Price Instead of Next-Block Price - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` and `run_validate_entry_point` both derive their gas-price reference from `gateway_fixed_block_state_reader.get_block_info()`, which returns the **last committed block's** prices. The next block's L2 gas price is computed by the EIP-1559 fee market (`calculate_next_base_gas_price`) and can differ from the previous block's price by up to ~1–2% per block. With the production default of `min_gas_price_percentage = 100`, the gateway's admission threshold is exactly the previous block's price, not the price that will actually govern execution. This is the direct sequencer analog of the Derby `exchangeRateStored` bug: a cached value is used where the live/current value is required.

### Finding Description

**Root cause — `validate_resource_bounds`:** [1](#0-0) 

The function reads `strk_gas_prices.l2_gas_price` from the previous block and compares it against the transaction's `max_price_per_unit`. The inline TODO confirms the intent to fix this: `// TODO(Arni): getnext_l2_gas_price from the block header.`

**Root cause — `run_validate_entry_point`:** [2](#0-1) 

The `block_info` taken from `gateway_fixed_block_state_reader` has only its `block_number` incremented; the gas prices remain those of the previous block. The `BlockContext` built from this stale `block_info` is then passed to `StatefulValidator::perform_validations`, which calls `perform_pre_validation_stage` → `check_fee_bounds`. Both the gateway-level price check and the blockifier-level price check therefore use the same stale reference.

**How the next-block price is actually computed:** [3](#0-2) 

`calculate_next_base_gas_price` applies EIP-1559 logic: when a block is congested (`gas_used > gas_target`) the price rises; when underutilised it falls. The denominator (`gas_price_max_change_denominator`) bounds the per-block change to roughly 1–2%.

**Production configuration:** [4](#0-3) 

`min_gas_price_percentage` defaults to **100** in both the schema and the gateway replacer config, meaning the admission threshold equals the previous block's price exactly — no slack is given.

**The two broken invariants:**

1. **Rising price (congested blocks):** `next_price > prev_price`. A transaction with `prev_price ≤ max_price_per_unit < next_price` passes both the gateway check and the blockifier pre-validation at the gateway (both use `prev_price`), is admitted to the mempool, but fails `check_fee_bounds` at actual execution time in the batcher (which uses `next_price`). The gateway **admits an invalid transaction**.

2. **Falling price (underutilised blocks):** `next_price < prev_price`. A transaction with `next_price ≤ max_price_per_unit < prev_price` would succeed at execution time, but the gateway rejects it with `GAS_PRICE_TOO_LOW` because the threshold is `prev_price`. The gateway **rejects a valid transaction**.

### Impact Explanation

Matching impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

- In the rising-price direction, transactions that cannot pay the actual next-block price are admitted to the mempool and consume sequencer resources before being discarded at execution.
- In the falling-price direction, transactions that are economically valid for the next block are incorrectly rejected at the gateway, degrading user experience and liveness precisely when the network is underloaded.

### Likelihood Explanation

The EIP-1559 fee market is always active. Any block that is not exactly at the gas target causes the next-block price to diverge from the current block's price. With `min_gas_price_percentage = 100` in production, the discrepancy window opens on every non-stable block. Users who set `max_price_per_unit` to exactly the current block's price (a natural choice) will be affected whenever the price moves.

### Recommendation

Replace the stale `get_block_info()` call in `validate_resource_bounds` with the computed next-block L2 gas price. The TODO comment already identifies this:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
```

The next-block price is available from `FeeMarketInfo::next_l2_gas_price` stored in the committed block's metadata, or can be recomputed from the previous block's `l2_gas_consumed` and `l2_gas_price` fields using `calculate_next_base_gas_price`. The same fix should be applied to the `block_info` used in `run_validate_entry_point` so that the blockifier pre-validation also uses the correct price.

### Proof of Concept

1. Read the current committed block: `prev_l2_gas_price = P`, `l2_gas_consumed = C > gas_target` (congested block).
2. Compute `next_l2_gas_price = P' > P` via `calculate_next_base_gas_price(P, C, gas_target, min_price)`.
3. Submit a V3 invoke transaction with `l2_gas.max_price_per_unit = P` (exactly the previous block price).
4. Gateway `validate_resource_bounds` checks `P >= P * 100% = P` → **passes**.
5. Gateway `run_validate_entry_point` builds `BlockContext` with gas price `P`; blockifier `check_fee_bounds` checks `P >= P` → **passes**. Transaction is admitted to the mempool.
6. Batcher builds the next block with `BlockContext` gas price `P'`; blockifier `check_fee_bounds` checks `P >= P'` → **fails** (`MaxGasPriceTooLow`). Transaction is discarded after admission. [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
```
