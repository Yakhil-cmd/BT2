### Title
Pool Admin Bypasses Fee Cap via Uncapped `setPoolBinAdditionalFees` — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces hard fee caps on the base spread and notional fees for every admin-controlled path (`setPoolAdminFees`, `createPool`, `setPoolProtocolFee`). However, `setPoolBinAdditionalFees` — the pool-admin function that sets per-bin additional buy/sell spread fees — contains **no cap check whatsoever**. A pool admin can set `addFeeBuyE6` or `addFeeSellE6` to `type(uint16).max` (65 535, i.e. 6.5535% in E6 units) on any bin, silently exceeding the protocol-wide hard cap of 20% (`HARD_MAX_SPREAD_FEE_E6 = 200_000`) that governs every other admin fee path.

---

### Finding Description

The factory declares two hard constants that bound all admin-controlled spread fees:

```
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;   // 20 %
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000; // 1 %
``` [1](#0-0) 

Every admin-facing fee setter enforces these caps:

- `setPoolAdminFees` — `if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();`
- `setPoolProtocolFee` — `if (newProtocolSpreadFeeE6 > maxProtocolSpreadFeeE6) revert ProtocolFeeTooHigh();`
- `createPool → _validatePoolParameters` — `if (params.adminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();` [2](#0-1) [3](#0-2) 

`setPoolBinAdditionalFees`, however, passes the caller-supplied values straight through with **no validation**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [4](#0-3) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index range: [5](#0-4) 

The bin additional fees are then added directly to `baseFeeX64` in every swap iteration:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [6](#0-5) [7](#0-6) 

The `BinState` struct stores these as `uint16`: [8](#0-7) 

`uint16` max = 65 535, which in E6 units = **6.5535%**. Combined with a base spread fee already at the 20% hard cap, the effective per-bin fee reaches **~26.5%**, far above the protocol-enforced ceiling.

---

### Impact Explanation

Traders swapping through a bin whose additional fee has been set to `uint16.max` pay up to 6.5535% more than the protocol's hard cap permits. The excess fee accrues to LPs in that bin, constituting a direct loss of trader principal beyond the documented and enforced fee ceiling. Because the bin additional fee is applied multiplicatively inside `buyToken0InBinSpecifiedOut` / `buyToken0InBinSpecifiedIn` / `buyToken1InBinSpecifiedIn` / `buyToken1InBinSpecifiedOut`, the overcharge is proportional to swap size and can be material for large trades.

---

### Likelihood Explanation

The pool admin is semi-trusted and has a legitimate reason to call `setPoolBinAdditionalFees` (fine-grained bin incentives). The absence of any cap check means a pool admin — whether acting maliciously or by mistake — can set the value to `type(uint16).max` in a single transaction with no timelock. There is no on-chain guard that prevents it.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and symmetrically in `setBinAdditionalFees` on the pool, or at minimum in the factory gateway) that limits `addFeeBuyE6` and `addFeeSellE6` to a defined maximum — for example, `maxAdminSpreadFeeE6` or a dedicated `HARD_MAX_BIN_ADDITIONAL_FEE_E6` constant:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

---

### Proof of Concept

1. Deploy a pool with `adminSpreadFeeE6 = maxAdminSpreadFeeE6` (e.g. 200 000 = 20%).
2. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, type(uint16).max, type(uint16).max)` — this succeeds with no revert.
3. A trader swaps through bin 0. The effective buy fee is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately 26.5% total, exceeding the 20% hard cap.
4. The trader receives fewer tokens than the cap system permits; the excess is retained by LPs in that bin.

The factory's own test suite confirms `setPoolBinAdditionalFees` accepts arbitrary `uint16` values without any cap revert: [9](#0-8) 

No test asserts that values above `maxAdminSpreadFeeE6` are rejected, because no such guard exists.
