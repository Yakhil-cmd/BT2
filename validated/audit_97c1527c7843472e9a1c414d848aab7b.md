### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` to the pool with no validation against `maxAdminSpreadFeeE6`. The sibling function `setPoolAdminFees` enforces the cap; `setPoolBinAdditionalFees` does not. Because per-bin additional fees are added directly to the oracle spread fee in every swap through the affected bin, a pool admin can charge traders fees far above the factory-configured cap.

---

### Finding Description

`setPoolAdminFees` enforces the cap on both components before writing to `poolFeeConfig`:

```solidity
// MetricOmmPoolFactory.sol lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` performs no such check — it passes the caller-supplied values straight through to the pool:

```solidity
// MetricOmmPoolFactory.sol lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` also performs no cap check — only a bin-range check — and writes the raw value into `BinState.addFeeBuyE6`/`addFeeSellE6`: [3](#0-2) 

During every swap through the affected bin, the stored `addFeeBuyE6` is added directly to `baseFeeX64` (the oracle-derived spread fee) before the swap math runs:

```solidity
// MetricOmmPool.sol line 910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern applies to the `buyToken0InBinSpecifiedIn` path: [5](#0-4) 

The resulting surplus is split between protocol and admin via `poolFeeConfig` ratios, so the admin captures a share of the inflated fee.

---

### Impact Explanation

A pool admin sets `addFeeBuyE6 = 65535` (uint16 max; 6.5535 % in E6 units). Every trader buying token0 through that bin pays 6.5535 % on top of the oracle spread and the global spread fee, regardless of what `maxAdminSpreadFeeE6` is configured to (e.g. 0.1 %). The excess is real token principal extracted from traders on every swap. The admin receives a proportional share via the spread-fee split. This is a direct, repeatable loss of trader principal that violates the admin-fee-cap invariant the factory is designed to enforce.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that can call `setPoolBinAdditionalFees` at any time with no timelock, no co-signer, and no on-chain constraint on the fee values. The attack requires a single transaction and is immediately effective on the next swap.

---

### Recommendation

Add cap enforcement in `setPoolBinAdditionalFees` before forwarding to the pool, mirroring `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

---

### Proof of Concept

1. Deploy factory with `maxAdminSpreadFeeE6 = 1000` (0.1 %).
2. Create a pool; pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — succeeds with no revert.
3. Execute a swap through bin 0.
4. Observe that the effective per-bin fee applied is 6.5535 %, far exceeding the 0.1 % cap, and that the admin's fee destination receives a share of the inflated surplus.
