### Title
Blocklist Destination Check Bypassed via Internal EVM Contract Calls — (File: `app/block_address.go`)

### Summary
The Cronos blocklist enforces a destination-address restriction only at the transaction level (the outer EVM `to` field), not at the EVM execution level. An unprivileged attacker can route native CRO or CRC20/CRC21 tokens to a blocked address through an intermediary contract, bypassing the admin/governance-controlled blocklist entirely.

### Finding Description
Both enforcement points for the Cronos blocklist inspect only the direct, outer EVM transaction destination:

**`BlockAddressesDecorator.AnteHandle`** (`app/block_address.go`, lines 32–83) runs only during `CheckTx` and checks:
- Cosmos tx signers
- `ethTx.To()` — the direct EVM `to` field
- EIP-7702 authorization-list addresses [1](#0-0) 

**`ProposalHandler.ValidateTransaction`** (`app/proposal.go`, lines 262–335) runs during `ProcessProposal` and applies the same three checks: [2](#0-1) 

Neither enforcement point inspects **internal EVM call destinations** — calls made by one contract to another during EVM execution. Because the blocklist is applied only at the mempool (`CheckTx`) and proposal (`ProcessProposal`) layers, any transaction whose outer `to` field is an unblocked contract address passes both checks, even if that contract subsequently forwards value or tokens to a blocked address via an internal call.

The analog to the external report is direct:

| External report | Cronos |
|---|---|
| `maxWallet` checked in `buy()` only | Blocklist checked for direct EVM `to` only |
| Bypass via `transfer()` to a second wallet | Bypass via internal contract call to blocked address |

### Impact Explanation
**High** — Bypass of the Cronos admin/governance-controlled block-list authorization check. A blocked address can receive CRO (native token) or CRC20/CRC21 tokens through an intermediary contract, defeating the purpose of the blocklist. This falls squarely within the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation
**High** — Any unprivileged user can deploy a trivial forwarding contract and call it to route funds to a blocked address. No special privileges, leaked keys, governance access, or cryptographic breaks are required. The attacker only needs gas.

### Recommendation
Enforce the blocklist at the EVM execution level, not just at the transaction level. Options include:

1. Add an EVM state-transition hook (e.g., in `PostTxProcessing` or a custom `vm.EVMInterpreter` wrapper) that checks all internal call destinations against the blocklist before value or token transfers are committed.
2. Alternatively, check the blocklist inside the EVM `Call` opcode handler so that any internal call targeting a blocked address is reverted.

### Proof of Concept

```
1. Admin blocks address A via MsgStoreBlockList.

2. Attacker deploys contract C:
   function forward(address dest) external payable {
       (bool ok,) = dest.call{value: msg.value}("");
       require(ok);
   }

3. Attacker submits EVM tx:  to=C (not blocked), calling C.forward{value: X}(A).

4. BlockAddressesDecorator (CheckTx): to=C → not in blocklist → passes.
   ProposalHandler.ValidateTransaction (ProcessProposal): ethTx.To()=C → not in blocklist → passes.

5. During FinalizeBlock EVM execution, C internally calls A with value X.
   No blocklist check exists at the EVM execution layer.

6. A receives X CRO despite being on the blocklist.
   The same path works for CRC20/CRC21 tokens by having C call token.transfer(A, amount).
``` [3](#0-2) [4](#0-3)
