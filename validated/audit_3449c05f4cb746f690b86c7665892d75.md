### Title
Malicious Coordinator Can Force Presignature Reuse to Extract Aggregate Private Key — (File: `src/ecdsa/ot_based_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary
Both OT-based and robust ECDSA signing protocols allow a malicious coordinator to silently collect signature shares from multiple signing sessions that reuse the same presignature. Non-coordinator participants send their shares and immediately return without receiving any confirmation that signing completed. Because `PresignOutput` carries no consumed flag and the library provides no invalidation mechanism, a malicious coordinator can force presignature reuse across two sessions and algebraically extract the aggregate private key.

---

### Finding Description

In `do_sign_participant` in both implementations, non-coordinator participants send their signature share to the coordinator and immediately return `Ok(None)`: [1](#0-0) [2](#0-1) 

Participants receive **no acknowledgment** that the coordinator successfully produced a signature. The `PresignOutput` struct carries no "consumed" marker: [3](#0-2) 

The library warns in both presign modules that reuse is catastrophic, but provides zero enforcement: [4](#0-3) [5](#0-4) 

The robust signing module even documents the split-view attack surface that arises from this exact scenario: [6](#0-5) 

**Attack sequence:**

1. Coordinator initiates signing session 1 with message `h1` and presignature `P = (R, k, σ)`.
2. All participants send shares `s_i(h1)` privately to the coordinator; coordinator computes `s1 = h1·k + r·σ`.
3. Coordinator silently discards the result — participants have no way to detect this.
4. Coordinator initiates signing session 2 with the **same** presignature `P` but message `h2`.
5. Participants comply (they have no record that session 1 consumed the presignature).
6. Coordinator computes `s2 = h2·k + r·σ`.
7. Coordinator solves: `k = (s1 − s2)/(h1 − h2)`, `σ = (s1 − h1·k)/r`, `x = σ/k` — the full aggregate private key.

---

### Impact Explanation

The coordinator recovers the aggregate private key `x`, enabling unauthorized creation of valid ECDSA signatures for any message under the shared key. This matches the **Critical** impact tier: extraction of aggregate secret material and unauthorized threshold signature creation for attacker-chosen inputs.

---

### Likelihood Explanation

The coordinator role is a per-session parameter assignable to any participant. A single malicious participant who acts as coordinator needs only two consecutive signing sessions with the same presignature — no leaked keys, no cryptographic breaks, and no external capabilities are required. The `PresignOutput` struct is `Clone`-able and nothing in the API prevents passing it to `sign()` twice.

---

### Recommendation

- Add a consumed flag (e.g., a `Cell<bool>` or `AtomicBool`) to `PresignOutput` and `RerandomizedPresignOutput` that is set when `sign()` is first called, causing any subsequent call with the same struct to return an `InitializationError`.
- Alternatively, have the coordinator broadcast the final signature back to all participants so they can verify completion; participants should refuse to participate in a second session with the same presignature unless they have confirmed the first session failed **before** any shares were transmitted.
- Bind each presignature to a single `(msg_hash, participant_set)` tuple at the protocol level to address the split-view warning already present in `robust_ecdsa/sign.rs`.

---

### Proof of Concept

```rust
// Session 1 — coordinator collects shares for h1, computes s1, discards result silently
let presign_out = run_presign(...);
let rerandomized = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args)?;
// coordinator receives all s_i(h1), computes s1 = h1·k + r·σ, does NOT broadcast

// Session 2 — same presignature, different message h2
// Nothing in the API prevents this clone and reuse:
let rerandomized_clone = rerandomized.clone();
// coordinator receives all s_i(h2), computes s2 = h2·k + r·σ

// Key extraction (coordinator-side arithmetic):
// k   = (s1 - s2) * (h1 - h2)^{-1}
// σ   = (s1 - h1 * k) * r^{-1}
// x   = σ * k^{-1}   ← full aggregate private key recovered
```
