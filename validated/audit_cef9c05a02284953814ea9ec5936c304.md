### Title
Missing Share Validity Check in CKD Coordinator Aggregation Allows Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function aggregates `(big_y, big_c)` shares from all participants by direct addition with no cryptographic verification. Analogous to the reported `merge` calling `_updateCollateralShares` directly and bypassing the cap check, `do_ckd_coordinator` combines participant-supplied values without any check that each share is correctly formed. A single malicious participant can inject arbitrary group elements, causing the coordinator to accept and output a corrupted CKD result.

### Finding Description

`do_ckd_coordinator` in `src/confidential_key_derivation/protocol.rs` receives each participant's `(norm_big_y, norm_big_c)` pair and unconditionally accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

Each participant's share is computed in `compute_signature_share` as:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` [2](#0-1) 

For the final `unmask` to yield `msk · H(pk, app_id)`, every participant must contribute a correctly formed pair. There are **no zero-knowledge proofs**, **no commitments**, and **no range checks** on the received values. The coordinator has no mechanism to distinguish a valid share from an arbitrary group element injected by a malicious participant.

The public entry point `ckd()` validates only participant-list invariants (count ≥ 2, no duplicates, self-presence, coordinator-presence): [3](#0-2) 

It performs **no validation of the cryptographic content** of the shares that will later be aggregated. `do_ckd_coordinator` is then called directly and accumulates whatever values arrive over the channel. [4](#0-3) 

This mirrors the reported pattern exactly: the public function (`ckd`) performs structural checks, but the combining step (`do_ckd_coordinator`) calls the raw accumulation directly, bypassing any share-validity check.

### Impact Explanation

A malicious participant sends `(big_y', big_c')` instead of their correct `(λ_i · y_i · G, λ_i · C_i)`. The coordinator accumulates the tampered values. The final `unmask` computes:

```
(Σ big_c + δ_c) − app_sk · (Σ big_y + δ_y)
= msk · H(pk, app_id) + δ_c − app_sk · δ_y
```

which equals `msk · H(pk, app_id)` only if `δ_c = app_sk · δ_y`. Since the attacker does not know `app_sk`, they cannot satisfy this, so the derived confidential key is wrong. The honest coordinator accepts and outputs a corrupted `CKDOutput`. This matches the allowed impact:

> **High: Corruption of CKD outputs so honest parties accept inconsistent… unusable cryptographic outputs.**

### Likelihood Explanation

Any single participant in the protocol can mount this attack. No special privilege, leaked key, or external assumption is required — the attacker only needs to deviate from the protocol by sending arbitrary `ElementG1` values. The attack is trivially reachable through the public `ckd()` API by any participant who controls their own process.

### Recommendation

Add a zero-knowledge proof of correct share formation to each participant's message. Specifically, each participant should prove in zero knowledge that their submitted `(norm_big_y, norm_big_c)` is consistent with their public key share (the corresponding coefficient commitment from DKG). The coordinator must verify all proofs before accumulating any share. A standard approach is a Chaum–Pedersen DLEQ proof showing that `norm_big_c − norm_big_y · app_pk` lies on the same discrete-log relation as the participant's public key share.

### Proof of Concept

1. Run `ckd()` with `n` participants, one of which is malicious.
2. The malicious participant, instead of calling `compute_signature_share`, sends `(ElementG1::identity(), ElementG1::generator())` — arbitrary values — to the coordinator.
3. The coordinator's loop at lines 50–55 adds these values unconditionally.
4. `CKDOutput::new(norm_big_y, norm_big_c)` is constructed from the corrupted aggregate.
5. Any downstream call to `ckd_output.unmask(app_sk)` returns a value that is not `msk · H(pk, app_id)`, silently producing an incorrect confidential key with no error returned to the caller. [4](#0-3)
