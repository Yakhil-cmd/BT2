Audit Report

## Title
Ingress Cycle Pre-Admission Check Uses Raw Balance Instead of Debited Balance, Enabling Block-Space DoS During DTS Execution - (File: `rs/cycles_account_manager/src/cycles_account_manager.rs`)

## Summary
`can_withdraw_cycles_with_threshold` compares against `system_state.balance()` (raw `cycles_balance`) rather than `system_state.debited_balance()` (`cycles_balance - ingress_induction_cycles_debit`). Both the boundary-node pre-filter (`should_accept_ingress_message`) and the consensus ingress selector (`validate_ingress`) rely on this function. When a canister has accumulated a large `ingress_induction_cycles_debit` during DTS execution, ingress messages pass pre-admission but are rejected during actual induction, consuming block slots without inducting any messages.

## Finding Description
`SystemState` exposes two balance views: `balance()` returns the raw `cycles_balance` field, while `debited_balance()` returns `cycles_balance - ingress_induction_cycles_debit` (saturating). During DTS execution, new ingress induction costs cannot be immediately deducted from `cycles_balance`; they accumulate in `ingress_induction_cycles_debit` and are applied when execution completes via `apply_ingress_induction_cycles_debit`.

The actual induction path (`charge_ingress_induction_cost`) correctly branches on `has_paused_execution_or_install_code()` and checks `debited_balance()` at line 336. However, `can_withdraw_cycles_with_threshold` at line 903 unconditionally checks `system_state.balance()`, ignoring the pending debit entirely.

This function is called at two pre-admission sites:
1. `execution_environment.rs` line 3358 — boundary-node pre-filter
2. `ingress_selector.rs` line 570 — consensus ingress selector (which also accumulates a `cycles_needed` map starting from zero, never seeded with the existing `ingress_induction_cycles_debit`)

The result is a split: pre-admission sees `balance()` (inflated), induction sees `debited_balance()` (deflated). Messages admitted by the former are rejected by the latter.

A secondary consequence: if cycles burned during execution push `cycles_balance` below `ingress_induction_cycles_debit`, `apply_ingress_induction_cycles_debit` silently drops the excess debit (the `[EXC-BUG]` path at line 1021–1032), making some induction charges free. This is an acknowledged known edge case in the code.

## Impact Explanation
The primary impact is application/platform-level DoS on subnet block capacity. An attacker targeting a canister in DTS can flood the ingress pool with messages that pass both the boundary-node check and the consensus ingress selector (both using raw `balance()`), filling up to `max_ingress_messages_per_block` slots per block with messages that are then rejected during induction. These wasted slots reduce effective block throughput for all canisters on the subnet for the duration of the DTS execution. This matches the allowed High impact: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
Any canister running a long DTS execution (large `install_code`, compute-heavy update) naturally accumulates `ingress_induction_cycles_debit`. An unprivileged sender can observe delayed responses as a signal that a canister is in DTS, then submit many ingress messages targeting it. No special privileges, no governance majority, no node control, and no victim mistakes are required. The attacker pays only standard ingress submission costs. The condition (canister in DTS with large debit) is a normal operational state, not a contrived one.

## Recommendation
Replace `system_state.balance()` with `system_state.debited_balance()` in `can_withdraw_cycles_with_threshold` at line 903:

```rust
// Before:
if threshold + requested > system_state.balance() {
    Err(CanisterOutOfCyclesError {
        available: system_state.balance(),
        ...
    })
// After:
if threshold + requested > system_state.debited_balance() {
    Err(CanisterOutOfCyclesError {
        available: system_state.debited_balance(),
        ...
    })
```

Additionally, the `cycles_needed` accumulator in `ingress_selector.rs` should be seeded with the canister's existing `ingress_induction_cycles_debit()` so that cumulative per-block accounting also reflects already-pending debits. The `charge_ingress_induction_cost` path is already correct and does not need changes.

## Proof of Concept
1. Deploy canister `C` with `cycles_balance = 1_000_000` and a long-running update method that loops for enough instructions to trigger DTS.
2. Invoke the long-running method; confirm `C` enters DTS (`has_paused_execution_or_install_code() == true`), causing subsequent ingress costs to accumulate in `ingress_induction_cycles_debit` (e.g., reaching `900_000`).
3. From an unprivileged identity, submit many ingress messages to `C`, each costing ~`10_000` cycles.
4. Observe that the boundary node accepts all messages: `can_withdraw_cycles_with_threshold` checks `balance() = 1_000_000`, which passes.
5. Observe that the ingress selector includes up to `max_ingress_messages_per_block` of them in a block, using the same raw-balance check.
6. Observe that during induction, `charge_ingress_induction_cost` checks `debited_balance() = 100_000` and rejects all but the first ~`50_000`-cycle message with `CanisterOutOfCycles`.
7. Confirm that block slots were consumed by messages that were never inducted, demonstrating the over-admission and block-space waste.

A deterministic integration test or PocketIC test can reproduce this by directly setting `cycles_balance` and `ingress_induction_cycles_debit` on a canister state and asserting the divergence between `can_withdraw_cycles_with_threshold` and `charge_ingress_induction_cost` outcomes.
