### Title
Signature Verification Bypassed for Invoke Transactions via Overly Broad `account_tx_in_pool_or_recent_block` Check in `skip_stateful_validations` - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry-point (signature verification) for an invoke transaction when `account_tx_in_pool_or_recent_block` returns `true`. That helper checks whether **any** transaction from the sender address is in the mempool pool or recent-block state — it does **not** verify that the transaction is specifically a `deploy_account`. An attacker who observes a legitimate `deploy_account` in the mempool for address X can immediately submit an invoke with `nonce=1` and an **arbitrary signature** for X. The gateway accepts it without ever calling `__validate__`, injecting an unauthenticated transaction into the mempool.

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have passed signature verification, or must be provably the first post-deploy invoke for an account whose `deploy_account` is pending.

**Root cause — `skip_stateful_validations`:**

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ← checks ANY tx from this address, not specifically deploy_account
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}

The mempool implementation of `account_tx_in_pool_or_recent_block` is:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700

This returns `true` for **any** transaction type from that address — not only `deploy_account`.

**How the skip propagates to execution:**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`:

```rust

    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

Inside `StatefulValidator::perform_validations`, when `validate` is `false`, the function returns immediately after `perform_pre_validation_stage` — the `__validate__` call is never reached:

```rust

    return Ok(());

``` [4](#0-3) 

**Nonce check does not block the attack:**

`validate_nonce` for invoke transactions allows any nonce in `[account_nonce, account_nonce + max_allowed_nonce_gap]`. With `account_nonce=0` and the default `max_allowed_nonce_gap ≥ 1`, `tx_nonce=1` passes:

```rust
// lines 287-296

        return Err(create_error(...));

}

**`validate_by_mempool` does not check signatures** — it only validates nonce ordering and duplicate hashes: [7](#0-6) 

### Impact Explanation

An attacker who observes a legitimate `deploy_account` for address X in the mempool can:

1. Submit `Invoke(sender=X, nonce=1, signature=<arbitrary>)`.
2. The gateway passes nonce validation, mempool validation, and skips `__validate__` because `account_tx_in_pool_or_recent_block(X)` is `true`.
3. The attacker's invoke is admitted to the mempool **without signature verification**.
4. The attacker's invoke occupies the `(X, nonce=1)` slot. The legitimate user's own `nonce=1` invoke is rejected as a duplicate unless they pay a higher fee (fee escalation).
5. When the batcher executes the block, the attacker's invoke reaches `__validate__` with an invalid signature and is rejected — but the legitimate user's `nonce=1` invoke was already displaced from the block.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- Requires only observing the public mempool for a pending `deploy_account` (trivially observable).
- No privileged access needed; any unprivileged actor can submit an `RpcTransaction`.
- The `deploy_account + invoke` UX flow is explicitly supported and documented, making it a predictable target.
- The default `max_allowed_nonce_gap ≥ 1` ensures the nonce check never blocks the attack.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. Add a dedicated mempool query such as `deploy_account_in_pool_or_recent_block(address) -> bool` that inspects the transaction type, and use that in `skip_stateful_validations` instead of the current type-agnostic check.

```rust
// Proposed fix in skip_stateful_validations:
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .deploy_account_in_pool_or_recent_block(tx.sender_address())  // type-specific check
        .await
        ...;
}
```

### Proof of Concept

1. Legitimate user submits `deploy_account` for address `X` (valid signature, nonce=0). It enters the mempool.
2. Attacker calls the gateway `add_tx` endpoint with:
   - `RpcTransaction::Invoke(V3 { sender_address: X, nonce: 1, signature: [0xdead, 0xbeef], ... })`
3. Gateway flow:
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → **passes**
   - `validate_by_mempool`: no duplicate hash, nonce not too old → **passes**
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns **`true`**
   - `run_validate_entry_point(skip_validate=true)`: `perform_pre_validation_stage` passes (non-strict nonce, balance pre-funded), `__validate__` is **never called**
4. Attacker's invoke with `signature=[0xdead, 0xbeef]` is added to the mempool at slot `(X, nonce=1)`.
5. Legitimate user's own `nonce=1` invoke is rejected with `DuplicateNonce` unless they escalate fees.
6. In the block: `deploy_account` executes (account deployed, nonce→1), then attacker's invoke executes → `__validate__` fails → rejected. The legitimate user's invoke never ran.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
