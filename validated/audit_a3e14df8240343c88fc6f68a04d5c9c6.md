### Title
Attacker-Controlled `receiver_id` in `GetExecutionOutcome` Produces Internally Inconsistent Light-Client Proof Bundle — (`chain/client/src/view_client_actor.rs`)

---

### Summary

The `GetExecutionOutcome` handler accepts a caller-supplied `receiver_id` and uses it — without any on-chain validation — to select which shard's outcome root to include in the returned proof. In a multi-shard setup with skipped chunks, supplying a `receiver_id` that maps to a different shard than the one that actually executed the receipt causes the handler to advance `outcome_proof.block_hash` to a later block (where the wrong shard has its next new chunk) while leaving `outcome_proof.proof` anchored to the original execution block. The resulting `RpcLightClientExecutionProofResponse` is internally inconsistent: the intra-shard Merkle path and the block-level Merkle path refer to different blocks and different shards.

---

### Finding Description

**Call path (concrete):**

1. Attacker calls `EXPERIMENTAL_light_client_proof` (or `GetExecutionOutcome` directly) with:
   ```
   TransactionOrReceiptId::Receipt {
       receipt_id: <real receipt executed at block H, shard A>,
       receiver_id: <attacker account that maps to shard B>,
   }
   ```

2. `ViewClientActor::handle(GetExecutionOutcome)` extracts `account_id = receiver_id` (attacker-controlled) without verifying it against the on-chain receipt. [1](#0-0) 

3. `chain.get_execution_outcome(&receipt_id)` returns the real stored outcome with `block_hash = H` and `proof` = the intra-shard Merkle path for shard A at block H. [2](#0-1) 

4. `account_id_to_shard_id(receiver_id)` resolves to `shard_B` (attacker-chosen). [3](#0-2) 

5. `get_next_block_hash_with_new_chunk(H, shard_B)` walks forward from H and returns `(H+5, shard_B)` — the first block where shard B has a new chunk — which may be far later than the block where shard A's next new chunk appears. [4](#0-3) 

6. `outcome_proof.block_hash` is overwritten to `H+5`; `outcome_proof.proof` is **not** updated — it still encodes the intra-shard Merkle path for shard A at block H. [5](#0-4) 

7. `outcome_root_proof` is built from block H+5's chunk headers using `target_shard_index` for shard B. [6](#0-5) 

8. `light_client_proof_local` assembles the final bundle using `outcome_proof.block_hash` (now H+5) to fetch `block_header_lite` and `block_proof`. [7](#0-6) 

**Resulting inconsistency in the returned bundle:**

| Field | Actual content | Should be |
|---|---|---|
| `outcome_proof.proof` | Intra-shard Merkle path for shard A at block H | Same |
| `outcome_proof.block_hash` | H+5 (shard B's next new chunk) | H+1 (shard A's next new chunk) |
| `outcome_root_proof` | Merkle path for shard B's `prev_outcome_root` in block H+5 | Merkle path for shard A's `prev_outcome_root` in block H+1 |
| `block_header_lite` | Header of block H+5 | Header of block H+1 |

### Impact Explanation

The two-step verification defined in the light-client spec fails:

```python
# Step 1 — gives shard A's outcome root at H (correct)
shard_outcome_root = compute_root(sha256(borsh(execution_outcome)), outcome_proof.proof)

# Step 2 — tries to lift shard A's root into block H+5 using shard B's Merkle path → FAILS
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
# block_outcome_root ≠ block_header_lite.inner_lite.outcome_root
``` [8](#0-7) 

A correctly-implemented light client that performs both steps will reject the bundle. However, the nearcore RPC is contractually required to return a *valid* proof bundle; returning an inconsistent one is a proof-generation invariant violation. A partial verifier that only checks `block_merkle_root` membership (step 2 of the block-level check) will see a real block H+5 in the merkle tree and may not detect that the intra-shard proof is for a completely different block and shard.

---

### Likelihood Explanation

The trigger requires:
- A multi-shard deployment (≥2 shards) — standard on mainnet.
- Skipped chunks for shard B between H and H+5 — common under load.
- The attacker must know a real `receipt_id` and supply a `receiver_id` that maps to a different shard. The `receipt_id` is public (visible in transaction outcomes); the `receiver_id` is a free RPC parameter with no on-chain validation.

No validator or operator privilege is required. Any unprivileged RPC caller can trigger this.

---

### Recommendation

Before using `account_id` to select the target shard, look up the actual receipt from the chain store and verify that the provided `receiver_id` matches the stored receipt's `receiver_id`. Alternatively, derive the target shard directly from the stored outcome's `executor_id` (which is the account that actually executed the receipt) rather than from the caller-supplied `receiver_id`. The `Transaction` variant already uses `sender_id` for routing, which is similarly unvalidated, and should receive the same fix.

### Proof of Concept

```
Setup:
  - 2 shards: shard 0 (alice.near), shard 1 (bob.near)
  - alice.near sends a cross-shard transfer to bob.near
  - Receipt R is executed at block H, shard 1 (bob.near's shard)
  - Shard 0 skips chunks at H+1..H+4; shard 0's next new chunk is at H+5

Attack:
  curl -X POST ... -d '{
    "method": "EXPERIMENTAL_light_client_proof",
    "params": [{
      "type": "receipt",
      "receipt_id": "<R>",
      "receiver_id": "alice.near",   // maps to shard 0, NOT shard 1
      "light_client_head": "<head>"
    }]
  }'

Expected (correct): outcome_proof.block_hash = H+1 (shard 1's next new chunk)
Actual (buggy):     outcome_proof.block_hash = H+5 (shard 0's next new chunk)
                    outcome_root_proof = Merkle path for shard 0 at H+5
                    outcome_proof.proof = intra-shard path for shard 1 at H  ← mismatch

Verification: compute_root(shard_outcome_root, outcome_root_proof) ≠ block_header_lite.inner_lite.outcome_root
```

    pub fn get_next_block_hash_with_new_chunk(
        &self,
        block_hash: &CryptoHash,
        shard_id: ShardId,
    ) -> Result<Option<(CryptoHash, ShardId)>, Error> {
        let mut block_hash = *block_hash;
        let mut epoch_id = *self.get_block_header(&block_hash)?.epoch_id();
        let mut shard_layout = self.epoch_manager.get_shard_layout(&epoch_id)?;
        // this corrects all the shard where the original shard will split to if sharding changes
        let mut shard_ids = vec![shard_id];

        while let Ok(next_block_hash) = self.chain_store.get_next_block_hash(&block_hash) {
            let next_epoch_id = *self.get_block_header(&next_block_hash)?.epoch_id();
            if next_epoch_id != epoch_id {
                let next_shard_layout = self.epoch_manager.get_shard_layout(&next_epoch_id)?;
                if next_shard_layout != shard_layout {
                    shard_ids = shard_ids
                        .into_iter()
                        .flat_map(|id| {
                            next_shard_layout.get_children_shards_ids(id).unwrap_or_else(|| {
                                panic!("invalid shard layout {:?} because it does not contain children shards for parent shard {}", next_shard_layout, id)
                            })
                        })
                        .collect();

                    shard_layout = next_shard_layout;
                }
                epoch_id = next_epoch_id;
            }
            block_hash = next_block_hash;

            let block = self.get_block(&block_hash)?;
            let chunks = block.chunks();
            for &shard_id in &shard_ids {
                let shard_index = shard_layout.get_shard_index(shard_id)?;
                let chunk_header =
                    &chunks.get(shard_index).ok_or(Error::InvalidShardId(shard_id))?;
                if chunk_header.height_included() == block.header().height() {
                    return Ok(Some((block_hash, shard_id)));
                }
            }
