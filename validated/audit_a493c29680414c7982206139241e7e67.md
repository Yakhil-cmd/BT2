### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions With Arbitrary Signatures When a Pending `deploy_account` Exists in the Mempool - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the blockifier `__validate__` entry-point call (i.e., signature verification) for any invoke transaction with `nonce = 1` whose sender address appears in the mempool or a recent committed block. Because the mempool's `account_tx_in_pool_or_recent_block` check is satisfied the moment a legitimate user's `deploy_account` transaction enters the mempool, any third party can immediately submit a second invoke transaction for that same address with a completely arbitrary signature and have it admitted to the mempool without any cryptographic verification.

---

**What `account_tx_in_pool_or_recent_block` actually checks**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

`tx_pool.contains_account` returns `true` for **any** transaction type from that address currently in the pool:

```rust
pub fn contains_account(&self, address: ContractAddress) -> bool {
    self.txs_by_account.contains(address)
}
``` [6](#0-5) 

The code comment claims this is safe because the only way an undeployed account can have a transaction in the pool is via a `deploy_account`. That is true for `tx_pool.contains_account`. However, `state.contains_account` checks the mempool's internal committed/staged maps:

```rust
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [7](#0-6) 

The `staged` map is populated when a transaction is sent to the batcher (`get_txs`), before the block is committed. If a `deploy_account` for address X is staged (sent to batcher) but the block has not yet been committed, `state.contains_account(X)` returns `true` while the gateway's `gateway_fixed_block_state_reader` still returns `account_nonce = 0` for X (the block is not yet committed on-chain). This creates a reliable window during which the skip condition is satisfied.

**The attack**

1. Victim Alice submits a `deploy_account` for address X. The transaction enters the mempool and is eventually staged (sent to the batcher). At this point `account_tx_in_pool_or_recent_block(X)` returns `true` via either `tx_pool.contains_account` (while in pool) or `state.contains_account` (while staged).
2. The gateway's `gateway_fixed_block_state_reader` still reads `account_nonce(X) = 0` because the block has not been committed yet.
3. Attacker Bob submits an invoke transaction for address X with `nonce = 1` and a completely arbitrary/invalid signature.
4. The gateway evaluates: `tx_nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(X) == true` → `skip_stateful_validations` returns `true`.
5. `run_validate_entry_point` is called with `skip_validate = true` → `execution_flags.validate = false` → `__validate__` is never invoked → Bob's signature is never checked.
6. Bob's malicious invoke passes `validate_by_mempool` (no duplicate hash, nonce not too old) and is admitted to the mempool.
7. If fee escalation is enabled and Bob's fee is higher than Alice's legitimate invoke, Bob's tx replaces Alice's. Alice's deploy succeeds but her first invoke is gone from the mempool.

---

### Impact Explanation

The gateway admits an invoke transaction with an arbitrary (invalid) signature to the mempool without any cryptographic verification. This directly satisfies the impact criterion:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Concretely:
- Any account that has a pending `deploy_account` in the mempool (or staged to the batcher) can be targeted.
- The attacker can front-run or, if fee escalation is enabled, replace the victim's legitimate first invoke transaction with a malicious one that will fail during execution.
- The victim's deploy succeeds but their first post-deploy invoke is silently dropped, requiring them to resubmit.
- This is a targeted, low-cost, zero-knowledge-of-private-key DoS attack against every new account deployment.

---

### Likelihood Explanation

- All `deploy_account` transactions are broadcast over P2P and are publicly observable in the mempool.
- The attacker needs only to know the victim's address (derivable from the `deploy_account` transaction itself) and submit an invoke with `nonce = 1`.
- No cryptographic material belonging to the victim is required.
- The attack window lasts from the moment the `deploy_account` enters the mempool until the block containing it is committed and the gateway's state reader is updated — typically several seconds to minutes.
- The attack is cheap: the attacker pays only the gas for a failed invoke transaction (or nothing if the tx is evicted before execution).

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address. Add a dedicated mempool query such as `deploy_account_in_pool(address: ContractAddress) -> bool` that inspects transaction types in the pool, and use that in `skip_stateful_validations` instead of the generic presence check.

Alternatively, restrict the skip to only the `tx_pool.contains_account` path (not `state.contains_account`) and add a transaction-type filter so that only a `deploy_account` transaction triggers the skip.

---

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
