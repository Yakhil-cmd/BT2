### Title
Unverified Sender Puzzle Hash in Blockchain Offer Notifications Redirects Counter-Offer to Attacker - (File: packages/gui/src/hooks/useBlockchainNotifications.tsx)

### Summary
The blockchain notification system trusts the `ph` (puzzle hash) field embedded in an attacker-crafted on-chain notification message without verifying it belongs to the actual coin sender. When a victim clicks "Counter Offer," the pre-populated destination address is the attacker-controlled puzzle hash, causing the victim's counter-offer notification (containing the offer URL) to be delivered to the attacker, who can then accept the victim's counter-offer and drain the offered assets.

### Finding Description

The Chia notification protocol embeds a JSON payload in on-chain notification coins. The payload format is:

```json
{ "v": 1, "t": 1, "d": { "u": "<offer URL>", "ph": "<sender puzzle hash>" } }
```

The `ph` field is documented as "puzzlehash of the notification sender, for sending a response (counter offer)."

In `useBlockchainNotifications.tsx`, the GUI parses this payload and blindly trusts the `ph` value:

```js
const { u: url, ph: puzzleHash } = data;
if (puzzleHash) {
  return {
    type: NotificationType.OFFER,
    id,
    source: 'BLOCKCHAIN',
    timestamp: timestampData.timestamp,
    offerURL: url,
    puzzleHash,   // ← taken verbatim from attacker-controlled payload
  };
}
``` [1](#0-0) 

There is no check that `puzzleHash` matches the puzzle hash of the coin that actually sent the notification. Any unprivileged actor can send a notification coin to any wallet address and embed an arbitrary `ph` value.

This `puzzleHash` is then forwarded as the `address` prop when the victim clicks "Counter Offer":

```js
const address = currencyCode && puzzleHash
  ? toBech32m(puzzleHash, currencyCode.toLowerCase())
  : '';
navigate('/dashboard/offers/builder', {
  state: { isCounterOffer: true, address, offer },
});
``` [2](#0-1) 

The same pattern appears in `NotificationOffer.tsx` when the notification is clicked from the notification panel:

```js
address: 'puzzleHash' in notification ? notification.puzzleHash : undefined,
``` [3](#0-2) 

When the victim then sends the counter-offer notification, `NotificationSendDialog` uses this attacker-controlled address as the `target`:

```js
const targetPuzzleHash = fromBech32m(address);   // attacker-controlled
await sendNotification({
  target: targetPuzzleHash,
  ...
}).unwrap();
``` [4](#0-3) 

The address field is rendered as a disabled (read-only) text field, so the victim sees an address but has no way to know it is attacker-controlled rather than the legitimate offer sender's address.

### Impact Explanation

The attacker receives the victim's counter-offer notification, which contains the offer URL. Because Chia offers are public spend bundles — anyone holding the offer file can accept them — the attacker can immediately call `take_offer` on the victim's counter-offer and claim whatever assets the victim offered (XCH, CAT, NFT, etc.). This constitutes an unauthorized offer acceptance triggered by spoofed notification state, causing direct asset loss from the victim's wallet.

### Likelihood Explanation

Sending a notification coin costs a small, fixed XCH fee (the `amount` field in `sendNotification`). The attacker needs no special privileges, no leaked keys, and no host access. The only prerequisite is knowing the victim's wallet address (publicly derivable from any prior on-chain transaction). The victim must click "Counter Offer" and confirm the send dialog, but the UI gives no indication that the pre-populated destination is untrustworthy.

### Recommendation

Verify that the `ph` field in the notification payload matches the puzzle hash of the coin that actually delivered the notification. The blockchain notification record already contains the sender coin's puzzle hash (available via the `senderPuzzleHash` field on the coin metadata). The GUI should cross-check `notification.puzzleHash` against the on-chain sender puzzle hash before using it as the counter-offer destination. If they do not match, the counter-offer button should be disabled or the address field should be left blank.

### Proof of Concept

1. Attacker derives victim's wallet receive address (e.g., from a prior transaction).
2. Attacker crafts a notification payload:
   ```json
   {"v":1,"t":1,"d":{"u":"https://raw.dexie.space/<attacker-offer-id>","ph":"<attacker-puzzle-hash>"}}
   ```
3. Attacker calls `send_notification` targeting the victim's puzzle hash with the above hex-encoded message.
4. Victim's GUI displays "You have a new offer." Victim views the offer (attacker's offer, crafted to look attractive).
5. Victim clicks **Counter Offer**. The offer builder opens with `address` pre-populated from `ph` = attacker's address.
6. Victim creates a counter-offer (e.g., 5 XCH for the NFT) and clicks **Send Notification**.
7. `NotificationSendDialog` sends the counter-offer notification to `target` = attacker's puzzle hash.
8. Attacker receives the notification, extracts the counter-offer URL, and calls `take_offer` on the victim's counter-offer.
9. Victim's 5 XCH is transferred to the attacker.
