### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Consensus Block Rewards — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable by any non-zero address and accepts a `disable_rewards: bool` parameter. Because `last_reward_block` is written to storage **before** the `disable_rewards` guard is evaluated, any caller can invoke `update_rewards(valid_staker, disable_rewards: true)` every block to consume the one-call-per-block slot without distributing any rewards, permanently freezing consensus yield for all stakers.

---

### Finding Description

`update_rewards` is exposed via `IStakingRewardsManager` with only `general_prerequisites()` as a gate, which checks that the contract is unpaused and the caller is non-zero — no role restriction. [1](#0-0) 

The function writes `last_reward_block` to the current block number unconditionally, **before** the `disable_rewards` branch: [2](#0-1) 

Because the per-block guard is:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

once `last_reward_block` is set to block N, every other `update_rewards` call in block N reverts. An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` first in every block therefore:

1. Consumes the single allowed slot for that block.
2. Causes the function to return early at line 1487 without distributing any rewards.
3. Blocks every legitimate consensus-reward call for that block.

The `staker_address` argument only needs to resolve to an active staker with non-zero balance — the caller itself need not be a staker. [3](#0-2) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

All consensus block rewards (`strk_block_rewards`, `btc_block_rewards`) computed in `calculate_block_rewards` / `update_current_epoch_block_rewards` are never credited to any staker or pool. Rewards are silently discarded every block the attacker acts first. Because the `RewardSupplier` only tracks `unclaimed_rewards` after `update_unclaimed_rewards_from_staking_contract` is called (which is inside the skipped code path), the rewards are never even requested from L1, so no recovery is possible for the missed blocks. [4](#0-3) 

---

### Likelihood Explanation

**High.** Any non-zero address can call `update_rewards`. The attacker only needs to know one valid, active staker address (publicly observable on-chain). The cost is one cheap Starknet transaction per block. No capital is at risk for the attacker; the attack is pure griefing. The attack is effective from the moment `consensus_rewards_first_epoch` is set.

---

### Recommendation

1. **Short term**: Restrict `update_rewards` to a designated caller (e.g., a `REWARDS_MANAGER` role or the attestation contract) so that `disable_rewards: true` cannot be injected by an arbitrary address.
2. **Long term**: Move the `last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard, so that a no-op call does not consume the per-block slot. Alternatively, remove the `disable_rewards` parameter from the public interface entirely and handle the pre-consensus short-circuit via `is_pre_consensus()` alone.

---

### Proof of Concept

```
// Precondition: consensus_rewards_first_epoch has been set.
// Attacker is any EOA (does not need to be a staker).
// `victim_staker` is any currently active staker address (public info).

loop every block:
    staking_contract.update_rewards(
        staker_address = victim_staker,
        disable_rewards = true
    )
    // Effect: last_reward_block = current_block, no rewards distributed.
    // All subsequent update_rewards calls in this block revert with REWARDS_ALREADY_UPDATED.
```

The attacker pays only gas. All stakers and delegators receive zero consensus block rewards for every block the attacker acts first. The `block_snapshot` in `RewardSupplier` is also never updated (since `set_avg_block_duration` is inside the skipped path), compounding the disruption to the average block duration calculation used for future reward sizing.         fn update_current_epoch_block_rewards(ref self: ContractState) -> (Amount, Amount) {
            let staking_contract = self.staking_contract.read();
            assert!(
                get_caller_address() == staking_contract,
                "{}",
                GenericError::CALLER_IS_NOT_STAKING_CONTRACT,
            );
            self.set_avg_block_duration();
            // Calculate block rewards for the current epoch.
            let minting_curve_dispatcher = self.minting_curve_dispatcher.read();
            let yearly_mint = minting_curve_dispatcher.yearly_mint();
            let avg_block_duration = self.avg_block_duration.read();
            let total_rewards = mul_wide_and_div(
                lhs: yearly_mint,
                rhs: avg_block_duration.into(),
                div: BLOCK_DURATION_SCALE.into() * SECONDS_IN_YEAR.into(),
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW);
            let btc_rewards = calculate_btc_rewards(:total_rewards);
            let strk_rewards = total_rewards - btc_rewards;
            (strk_rewards, btc_rewards)
        }
