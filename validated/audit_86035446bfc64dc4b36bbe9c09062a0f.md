### Title
Signature Validation Bypass via Overly Broad `skip_stateful_validations` Check Admits Forged Invoke Transactions — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function is intended to skip `__validate__` only when a `deploy_account` transaction is pending in the mempool for a not-yet-deployed account. However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from that address, not exclusively a `deploy_account`. An unprivileged attacker who knows that a deployed account (on-chain nonce = 0) has any pending nonce-0 invoke in the mempool can submit a forged invoke with nonce = 1 and bypass `__validate__` (signature verification) entirely at the gateway admission layer.

---

### Finding Description

The `skip_stateful_validations` function in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements a UX shortcut: when an invoke transaction has `tx_nonce == 1` and the on-chain `account_nonce == 0`, it queries the mempool to decide whether to skip the blockifier `__validate__` call. [1](#0-0) 

The guard is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
```

`account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` whenever the address has **any** transaction in the pool — including a regular invoke with nonce = 0 — not only a `deploy_account`. The comment in the code acknowledges the assumption:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This assumption is false for a deployed account (nonce = 0 in state) that has a pending regular invoke with nonce = 0.

**Attack path:**

1. Account A is deployed on-chain (has contract code, state nonce = 0).
2. Account A submits a valid invoke with nonce = 0 → passes `__validate__` → enters mempool.
3. Attacker submits a forged invoke with nonce = 1 from account A (arbitrary calldata, wrong/empty signature).
4. Gateway `validate_nonce` passes: `0 ≤ 1 ≤ 200` (default `max_allowed_nonce_gap = 200`). [3](#0-2) 

5. `validate_by_mempool` passes: nonce = 1 is not a duplicate of the existing nonce = 0 tx.
6. `skip_stateful_validations` returns `true` (nonce == 1, account_nonce == 0, account in pool).
7. `run_validate_entry_point` is called with `skip_validate = true`, so `execution_flags.validate = false`. [4](#0-3) 

8. The forged transaction is admitted to the mempool **without any signature check**.

---

### Impact Explanation

The broken invariant is: *every invoke transaction admitted to the mempool must have passed `__validate__` (account signature verification)*. An attacker can inject a transaction with an arbitrary, invalid signature from any account that currently has a nonce-0 pending transaction in the mempool. This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

When the batcher later executes the forged transaction, `__validate__` will be called and will fail, causing a revert. For a reverted invoke, the nonce is incremented and a fee is charged to the victim account. [5](#0-4) 

---

### Likelihood Explanation

Moderate. The attacker needs to know the address of a deployed account (nonce = 0 in state) that currently has a pending transaction in the mempool. Mempool contents are observable via the gateway API. The attack requires no privileged access and no special resources beyond submitting a single HTTP request to the gateway.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a specific check that verifies a **`deploy_account` transaction** (not just any transaction) is pending for the sender address. Add a dedicated mempool query such as `has_pending_deploy_account(address)` that inspects the transaction type of the pooled entry, or alternatively check that the pooled transaction for that address has nonce = 0 and is of type `DeployAccount`. [6](#0-5) 

---

### Proof of Concept

1. Deploy account A on-chain (state nonce = 0, contract code present).
2. Submit a valid invoke from A with nonce = 0 (passes `__validate__`).
3. Immediately submit a second invoke from A with nonce = 1, an empty/wrong signature, and arbitrary calldata.
4. Observe that the gateway accepts the second transaction (HTTP 200, returns a tx hash) without rejecting it for signature failure.
5. Confirm via mempool snapshot that both transactions are present.
6. When the batcher executes the forged nonce-1 transaction, it reverts with a validate failure, A's nonce is incremented to 2, and a fee is deducted from A's balance — all without A's authorization. [1](#0-0) [2](#0-1)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
