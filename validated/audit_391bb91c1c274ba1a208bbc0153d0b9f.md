### Title
NEP-141 Fee-on-Transfer Tokens Cause ERC-20 Over-Minting and Bridge Insolvency - (File: `engine/src/engine.rs`)

---

### Summary

The Aurora Engine bridge mints ERC-20 tokens based on the `amount` field supplied in the `ft_on_transfer` callback arguments, without verifying the actual NEP-141 balance change received by the contract. For NEP-141 tokens that charge a fee on transfer, the stated `amount` exceeds the tokens actually credited to Aurora, causing the bridge to mint more ERC-20 tokens than it holds NEP-141 backing, leading to insolvency.

---

### Finding Description

When a NEP-141 token is bridged into Aurora via `ft_transfer_call`, the NEP-141 contract calls Aurora's `ft_on_transfer` entry point. Aurora reads the `amount` field from the callback arguments and uses it directly to mint ERC-20 tokens to the recipient.

The entry point is `ft_on_transfer` in `engine/src/contract_methods/connector.rs`: [1](#0-0) 

This dispatches to either `receive_base_tokens` or `receive_erc20_tokens` in `engine/src/engine.rs`. In `receive_base_tokens`: [2](#0-1) 

The amount is taken directly from `args.amount` and credited to the recipient's ETH balance without any balance-before/balance-after check.

In `receive_erc20_tokens`: [3](#0-2) 

`args.amount.as_u128()` is passed directly to `setup_receive_erc20_tokens_input`, which encodes it as the mint amount for the ERC-20 `mint(address, uint256)` call: [4](#0-3) 

Neither function checks the actual NEP-141 balance held by Aurora before and after the transfer. The NEP-141 standard does not prohibit fee-on-transfer implementations. A conforming NEP-141 token can deduct a fee from the receiver's credited balance while still passing the original pre-fee `amount` to `ft_on_transfer`. Aurora would then mint `amount` ERC-20 tokens while only holding `amount - fee` NEP-141 tokens.

---

### Impact Explanation

**Critical — Insolvency.**

For every `ft_transfer_call` of a fee-charging NEP-141 token, Aurora mints `fee` more ERC-20 tokens than it holds NEP-141 backing. The discrepancy accumulates with each bridge-in. When users later exit (burn ERC-20 via `withdrawToNear` / `exit_to_near`), the last users to exit will find Aurora's NEP-141 balance insufficient to cover their ERC-20 holdings. The bridge becomes insolvent for that token.

---

### Likelihood Explanation

**Medium.** Fee-on-transfer NEP-141 tokens exist and can be registered with Aurora via `deploy_erc20_token`. Any token holder can trigger the over-minting by calling `ft_transfer_call` on such a token with Aurora as the receiver. No privileged access is required. The attacker-controlled entry path is:

1. Deploy or use an existing fee-on-transfer NEP-141 token.
2. Call `ft_transfer_call` on the NEP-141 contract targeting Aurora.
3. The NEP-141 contract calls `ft_on_transfer` on Aurora with `amount = X` but credits Aurora with `X - fee`.
4. Aurora mints `X` ERC-20 tokens to the caller.
5. Repeat to accumulate unbacked ERC-20 tokens.
6. Exit ERC-20 tokens to drain Aurora's NEP-141 reserves.

---

### Recommendation

In both `receive_base_tokens` and `receive_erc20_tokens`, record the Aurora contract's NEP-141 balance before the transfer is accepted and use the actual balance delta (post-transfer balance minus pre-transfer balance) as the mint amount, rather than trusting `args.amount`. Since `ft_on_transfer` is called after the NEP-141 transfer has already occurred, the actual received amount can be determined by querying the NEP-141 contract's balance for Aurora's account ID. Alternatively, document and enforce that only non-fee-charging NEP-141 tokens may be registered, and add an on-chain invariant check.

---

### Proof of Concept

1. Deploy a NEP-141 token that charges a 10% fee on every transfer (credits receiver with 90% of stated amount but passes the full 100% as `amount` to `ft_on_transfer`).
2. Register it with Aurora via `deploy_erc20_token`.
3. Call `ft_transfer_call` with `amount = 1000` targeting Aurora.
4. Aurora's NEP-141 balance increases by 900, but `receive_erc20_tokens` is called with `args.amount = 1000`.
5. `setup_receive_erc20_tokens_input(&recipient, 1000)` mints 1000 ERC-20 tokens.
6. Repeat 10 times: Aurora holds 9000 NEP-141 but 10000 ERC-20 tokens are outstanding.
7. Exit all 10000 ERC-20 tokens — Aurora can only fulfill 9000, leaving the last 1000 ERC-20 holders unable to redeem.
