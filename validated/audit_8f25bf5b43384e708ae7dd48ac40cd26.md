### Title
Exact-Output Swap Silently Delivers Less Than Requested When Pool Liquidity Is Insufficient — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

Both exact-output swap paths in `MetricOmmPool` silently cap the requested output amount to the pool's available balance using an unchecked `min()` pattern, then proceed without reverting. The pool returns the reduced actual deltas to the caller with no error signal. Any integrator calling `pool.swap()` directly with a negative `amountSpecified` (exact-output mode) will silently receive fewer tokens than requested.

---

### Finding Description

In `MetricOmmPool._swapToken0ForToken1SpecifiedOutput` and `_swapToken1ForToken0SpecifiedOutput`, the very first action is to silently clamp the caller's requested output to whatever the pool currently holds:

```solidity
// _swapToken0ForToken1SpecifiedOutput (zeroForOne, exact-out token1)
uint256 totalAvailableToken1Scaled = binTotals.scaledToken1;
if (amountOutScaled > totalAvailableToken1Scaled) {
    amountOutScaled = totalAvailableToken1Scaled;   // silent cap — no revert
}
```

```solidity
// _swapToken1ForToken0SpecifiedOutput (!zeroForOne, exact-out token0)
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    amountOutScaled = totalAvailableToken0Scaled;   // silent cap — no revert
}
``` [1](#0-0) [2](#0-1) 

After the cap, the swap loop executes against the reduced target. `_executeSwap` then propagates the reduced `amountOutScaled` back as `amount0Delta` / `amount1Delta` and the outer `swap()` function returns those reduced deltas to the caller — no revert, no error flag. [3](#0-2) 

The `MetricOmmSimpleRouter` does add a post-swap equality check:

```solidity
if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);
``` [4](#0-3) [5](#0-4) 

However, this guard lives entirely in the periphery router, not in the pool itself. The pool is a standalone production contract whose interface (`IMetricOmmPoolActions.swap`) is explicitly designed for direct integration. Any caller that does not replicate the router's equality check — including multi-hop aggregators, lending protocols using the pool as a liquidation venue, or any bespoke integrator — will silently receive a partial fill.

---

### Impact Explanation

A direct caller requesting exact-output `X` tokens when the pool holds fewer than `X` will:

1. Receive `Y < X` tokens (the capped amount).
2. Pay only the token-in corresponding to `Y` (so no direct overpayment), but the transaction's stated intent — deliver exactly `X` — is violated without any on-chain signal.
3. Downstream logic that assumes the returned delta equals the requested amount (e.g., repaying a flash loan, settling a limit order, or chaining into a second swap) will operate on a wrong value, potentially causing insolvency or failed settlement in the calling contract.

This constitutes **broken core pool functionality** under the allowed impact gate: the pool's exact-output swap mode does not enforce its own invariant, and callers that trust the pool's interface semantics suffer a silent shortfall.

---

### Likelihood Explanation

The condition is reachable whenever a pool's bin liquidity for the output token is partially or fully depleted — a normal operational state after heavy one-sided trading or LP withdrawals. No privileged action is required; any user can trigger it by submitting an exact-output swap for an amount exceeding current pool depth. The `priceLimitX64` parameter does not protect against this: it bounds price movement, not output quantity.

---

### Recommendation

Enforce the exact-output invariant inside the pool itself. After the swap loop completes, verify that the actual output equals the (possibly capped) target, or — preferably — revert immediately when the requested output exceeds available liquidity:

```solidity
// Option A: revert up-front
if (amountOutScaled > totalAvailableToken1Scaled) {
    revert InsufficientLiquidity(amountOutScaled, totalAvailableToken1Scaled);
}

// Option B: revert after the loop if remaining > 0
if (state.amountSpecifiedRemainingScaled > 0) {
    revert InsufficientLiquidity(...);
}
```

This makes the pool's exact-output guarantee self-contained and removes the dependency on every downstream caller replicating the router's equality check.

---

### Proof of Concept

1. Deploy a pool with 100 token0 and 100 token1 in bins.
2. LP withdraws all token1, leaving `binTotals.scaledToken1 = 0`.
3. Integrator (not using the router) calls:
   ```solidity
   pool.swap(recipient, true, -50e18, 0, "", "");
   // amountSpecified = -50e18 → exact-output 50 token1
   ```
4. Inside `_swapToken0ForToken1SpecifiedOutput`:
   - `totalAvailableToken1Scaled = 0`
   - `amountOutScaled` is silently capped to `0`
   - Swap loop exits immediately; `amountCalculatedScaled = 0`
5. `swap()` returns `(amount0Delta=0, amount1Delta=0)` — no revert.
6. Integrator's downstream logic receives `0` token1 and `0` token0 charged, but interprets the zero delta as a successful no-op rather than a failed exact-output swap. [6](#0-5) [7](#0-6) [8](#0-7)
