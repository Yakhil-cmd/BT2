### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Over-Minting and Bridge Insolvency - (File: engine/src/engine.rs)

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` blindly trusts the `amount` field from `ft_on_transfer` arguments to mint ERC-20 tokens, without verifying the actual NEP-141 balance received by Aurora Engine. When a fee-on-transfer NEP-141 token is bridged, Aurora Engine mints more ERC-20 tokens than the NEP-141 tokens it actually holds, creating an insolvency condition that allows an attacker to drain other depositors' NEP-141 tokens.

### Finding Description

The NEP-141 `ft_transfer_call` flow calls `ft_on_transfer` on the receiver (Aurora Engine) with an `amount` parameter. A non-standard fee-on-transfer NEP-141 can deduct a fee from the transferred amount while still passing the original `amount` to `ft_on_transfer`. Aurora Engine's `receive_erc20_tokens` function takes this parameter at face value:

```rust
// engine/src/engine.rs:803
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount),  // line 831
``` [1](#0-0) [2](#0-1) 

The engine mints `args.amount` ERC-20 tokens even though it only received `args.amount - fee` NEP-141 tokens. There is no balance-before/balance-after check to verify the actual deposit.

Compounding this, `deploy_erc20_token` — the function that registers a NEP-141 token with Aurora Engine — has **no access control** beyond `require_running`. Any NEAR account can register any NEP-141 token:

```rust

    require_running(&state::get_state(&io)?)?;
    // No owner/admin check — anyone can call this
``` [3](#0-2)

### Impact Explanation

**Critical — Insolvency / Direct theft of user funds.**

Aurora Engine holds fewer NEP-141 tokens than the total ERC-20 supply it has minted. An attacker who exploits this can call `withdrawToNear` on the ERC-20 contract (which burns ERC-20 and triggers `ft_transfer` of NEP-141 from Aurora's balance) for the inflated amount, draining NEP-141 tokens that belong to other depositors. [5](#0-4) 

### Likelihood Explanation

**Medium.** The attack requires a fee-on-transfer NEP-141 token to be registered with Aurora Engine. Since `deploy_erc20_token` is permissionless, an attacker can register any NEP-141 they control. For profit, the attacker needs other users to have deposited the same fee-on-transfer NEP-141 token. This is realistic if a legitimate token with a fee mechanism (analogous to USDT with fees enabled) is bridged to Aurora, or if the attacker can attract deposits to a fee-on-transfer token they control.

### Recommendation

1. **In `receive_erc20_tokens`**: Record the NEP-141 balance of Aurora Engine before and after the `ft_on_transfer` callback, and use the actual delta (not `args.amount`) as the mint amount. Since NEAR's cross-contract call model makes synchronous balance checks impossible within `ft_on_transfer`, the correct fix is to use `ft_transfer_call`'s `ft_resolve_transfer` callback pattern — Aurora Engine should only mint ERC-20 tokens equal to the amount confirmed by the NEP-141 contract's `ft_resolve_transfer` result.

2. **In `deploy_erc20_token`**: Add an owner/admin access control check (`require_owner_only`) to prevent arbitrary NEP-141 tokens from being registered without governance approval. [6](#0-5) 

### Proof of Concept

1. Attacker deploys a fee-on-transfer NEP-141 token `fee_token.near` with a 10% transfer fee. Total supply: 1000 tokens. Legitimate user Alice has deposited 500 tokens into Aurora Engine (500 ERC-20 minted, Aurora holds 500 NEP-141).

2. Attacker calls `deploy_erc20_token` on Aurora Engine with `fee_token.near` — succeeds with no access control check. [3](#0-2) 

3. Attacker calls `ft_transfer_call(aurora, 1000, attacker_evm_address)` on `fee_token.near`. The NEP-141 deducts 1000 from attacker, transfers only 900 to Aurora Engine (10% fee), then calls `ft_on_transfer(attacker, 1000, attacker_evm_address)` on Aurora.

4. Aurora's `receive_erc20_tokens` reads `args.amount = 1000` and mints 1000 ERC-20 tokens for the attacker. [1](#0-0) 

5. Aurora Engine now holds 900 NEP-141 tokens (from attacker) + 500 (from Alice) = 1400 NEP-141, but has minted 1000 (attacker) + 500 (Alice) = 1500 ERC-20 tokens. **Insolvency: 100 token deficit.**

6. Attacker calls `withdrawToNear(attacker_near_account, 1000)` on the ERC-20. Aurora burns 1000 ERC-20 and calls `ft_transfer(attacker_near_account, 1000)` on `fee_token.near`. Aurora's balance drops to 400 NEP-141.

7. Alice tries to exit her 500 ERC-20 tokens. Aurora only holds 400 NEP-141 — Alice's exit fails or is partially fulfilled. **Alice loses 100 NEP-141 tokens.** [7](#0-6)

        };
