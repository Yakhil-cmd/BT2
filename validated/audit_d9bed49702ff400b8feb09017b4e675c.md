### Title
Pool `claim_rewards` Bypasses Staking Contract Pause — (`src/pool/pool.cairo`)

---

### Summary

The `Pool` contract's `claim_rewards` function contains no check against the staking contract's paused state. When the security agent pauses the staking contract, pool members can still call `Pool::claim_rewards` to extract accumulated STRK rewards held in the pool contract, defeating the purpose of the emergency pause.

---

### Finding Description

The staking contract enforces a pause guard via `general_prerequisites()` → `assert_is_unpaused()` on every state-changing function, including its own `claim_rewards`: [1](#0-0) 

The internal helper that enforces this is: [2](#0-1) 

The `Pool` contract, however, has **no pause component and no pause storage field**. Its `claim_rewards` implementation performs only two checks — pool member existence and caller authorization — and then immediately transfers STRK tokens out of the pool contract: [3](#0-2) 

There is no call to `assert_is_unpaused`, no query to the staking contract's `is_paused()` view, and no equivalent guard anywhere in the pool's `claim_rewards` path.

The STRK reward tokens are physically held by the pool contract (transferred there by `send_rewards_to_delegation_pool` in the staking contract): [4](#0-3) 

Because those tokens already reside in the pool contract, pausing the staking contract does not prevent the pool from transferring them out.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When the security agent triggers `pause()` on the staking contract (e.g., in response to a discovered exploit or accounting inconsistency), the intended effect is to freeze all reward flows. Pool members can circumvent this freeze by calling `Pool::claim_rewards` directly, draining all STRK rewards that have already been pushed into the pool contract. This constitutes theft of unclaimed yield that the protocol operator intended to freeze.

---

### Likelihood Explanation

Any pool member or their registered reward address can call `Pool::claim_rewards` at any time — it is a fully public, permissionless entry point requiring no special role. The only precondition is that the caller is the pool member or their reward address, both of which are unprivileged. The attack requires no front-running, no flash loan, and no privileged access: a pool member simply calls the function after the staking contract is paused.

---

### Recommendation

Add a check in `Pool::claim_rewards` that queries the staking contract's pause state before proceeding. The pool already holds a dispatcher to the staking contract (`staking_pool_dispatcher`): [5](#0-4) 

A minimal fix is to add at the top of `claim_rewards`:

```rust
fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
    // Pause guard: mirror the staking contract's pause state.
    let staking_dispatcher = IStakingDispatcher {
        contract_address: self.staking_pool_dispatcher.contract_address.read()
    };
    assert!(!staking_dispatcher.is_paused(), "{}", Error::CONTRACT_IS_PAUSED);
    // ... existing logic
}
```

Alternatively, introduce a dedicated `is_paused` storage field in the pool contract that is set/cleared by the staking contract when it deploys or upgrades the pool, keeping the pause state in sync without an extra cross-contract call.

---

### Proof of Concept

1. Deploy the full system (staking + pool contracts).
2. Have a staker open a delegation pool and a delegator enter it.
3. Advance epochs so rewards accumulate; the staking contract pushes STRK into the pool via `update_rewards`.
4. The security agent calls `staking.pause()` — `staking.is_paused()` returns `true`.
5. The delegator calls `pool.claim_rewards(delegator_address)` directly.
6. Observe: the call succeeds and STRK is transferred to the reward address, despite the staking contract being paused.
7. Calling `staking.claim_rewards(staker_address)` in the same state correctly reverts with `"Contract is paused"` (confirmed by the existing test at line 149 of `src/staking/tests/pause_test.cairo`), demonstrating the inconsistency. [6](#0-5)

```text
        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
```
