### Title
EIP-7702 Authorization List Bypasses `TxValidator` Access-Control Check — (`basic_bootloader/src/bootloader/transaction/authorization_list.rs`, `zk_ee/src/system/validator/mod.rs`)

---

### Summary

The `TxValidator::begin_tx` hook receives a `BeginTxContext` that contains only the EIP-7702 transaction's `from` address. The authority accounts listed in the EIP-7702 `authorization_list` are never presented to the validator. An unprivileged-but-authorized user can therefore submit an EIP-7702 transaction that installs a delegation on a *restricted* account (one that would be rejected if it sent its own transaction), then call that account within the same transaction to execute arbitrary code in its context — including draining its ETH balance — without the validator ever seeing the restricted account's address.

---

### Finding Description

**Step 1 — Validator is called with only `from`.**

`process_l2_transaction` calls `validator.begin_tx` before any other processing:

```rust
let ctx = BeginTxContext {
    from: *transaction.from(),   // authorized user only
    to: transaction.to(),
    value: *transaction.value(),
    calldata: transaction.calldata(),
    gas_limit: transaction.gas_limit(),
};
let pre_validation = validator.begin_tx(&ctx);
``` [1](#0-0) 

The `BeginTxContext` struct has no field for the authorization list:

```rust
pub struct BeginTxContext<'a> {
    pub from: B160,
    pub to: Option<B160>,
    pub value: U256,
    pub calldata: &'a [u8],
    pub gas_limit: u64,
}
``` [2](#0-1) 

**Step 2 — Authorization list is applied after `begin_tx` passes.**

Inside `validate_and_compute_fee_for_transaction` (called from `validate_and_prepare_context`, which runs after `begin_tx`), the authorization list is processed and delegations are written to state:

```rust
#[cfg(feature = "eip-7702")]
{
    if let Some(authorization_list) = transaction.authorization_list() {
        intrinsic_resources.with_infinite_ergs(|inf_resources| {
            parse_authorization_list_and_apply_delegations(system, inf_resources, authorization_list)
        })?;
    }
}
``` [3](#0-2) 

**Step 3 — `validate_and_apply_delegation` has no validator hook.**

The delegation logic verifies the authority's signature, nonce, and that it is not a deployed contract — but it never calls any validator or sanctions check on the `authority` address:

```rust
let Some(authority) = resources
    .with_infinite_ergs(|inf_ergs| recover_authority(system, inf_ergs, auth_sig_data, &msg))?
else { return Ok(false); };
// ... nonce / contract checks ...
system.io.set_delegation(inf_ergs, &authority, &delegation_address)?;
``` [4](#0-3) 

**Step 4 — Delegation is resolved at call time.**

When the authorized user's transaction body calls the restricted account, the runner resolves the delegation and executes the delegate contract's code in the restricted account's context (with access to its ETH balance and storage):

```rust
if cfg!(feature = "eip-7702") && account_properties.is_delegated.0
    && call_request.modifier != CallModifier::Constructor
{
    let address = parse_delegation(delegation)?;
    let delegate_properties = system.io.read_account_properties(..., &address, ...)?;
    (account_properties, Some(delegate_properties))
}
``` [5](#0-4) 

**Step 5 — `finish_tx` cannot undo the damage.**

`validator.finish_tx()` is called after execution completes. It receives no parameters and cannot inspect which accounts were affected, so it cannot retroactively block the delegation or the fund transfer. [6](#0-5) 

---

### Impact Explanation

Any operator deploying a `TxValidator` to enforce an address-level access-control policy (e.g., a sanctions list, a KYC allowlist, or a rate-limit per account) is silently bypassed. A restricted account can pre-sign an EIP-7702 authorization off-chain, delegating its code to a contract that transfers its ETH to an arbitrary recipient. An authorized user then submits an EIP-7702 transaction whose `from` passes the validator, the delegation is applied during validation, and the transaction body calls the restricted account — executing the delegate and draining its balance. The restricted account's funds are moved without it ever appearing as the `from` of any transaction. This is a direct loss of funds from the restricted account and a complete bypass of the operator's access-control policy.

---

### Likelihood Explanation

EIP-7702 support is gated behind the `eip-7702` feature flag, so the issue is only active when that feature is compiled in. The restricted account must willingly pre-sign an authorization (analogous to the sanctioned user creating a limit order in the external report). An authorized user must then submit the EIP-7702 transaction. Both parties must coordinate, but the authorized user can receive a commission from the proceeds, providing economic incentive. The attack requires no privileged role, no leaked key, and no oracle manipulation — only a valid EIP-7702 transaction from any non-restricted sender.

---

### Recommendation

1. **Extend `BeginTxContext`** to include the authorization list entries (authority addresses and delegation targets), so that `TxValidator` implementations can inspect and reject transactions whose authorization list contains restricted accounts.

2. **Add a per-authority validator hook** inside `parse_authorization_list_and_apply_delegations` that calls the validator for each recovered `authority` address before `set_delegation` is invoked, allowing the validator to veto individual delegations.

3. **Document the invariant gap** in the `TxValidator` trait: until the above is fixed, any validator that relies solely on `from` for access control is bypassable via EIP-7702 authorization lists.

---

### Proof of Concept

**Actors:**
- `restricted`: an account that would be rejected by the `TxValidator` if it sent its own transaction (e.g., on a sanctions list).
- `authorized`: a normal account that passes the validator.
- `DrainContract`: a contract whose code, when called, transfers `address(this).balance` to a specified recipient.

**Off-chain (by `restricted`):**
```
authorization = sign(EIP7702_MAGIC || rlp([chain_id, DrainContract_address, nonce]))
```

**On-chain (by `authorized`):**
Submit EIP-7702 transaction:
- `from`: `authorized`
- `authorization_list`: `[authorization]`  ← sets `restricted`'s code to `DrainContract`
- `to`: `restricted`                        ← calls `restricted`, which now runs `DrainContract`
- `data`: `drain(recipient=restricted_owner)`

**Execution trace:**
1. `validator.begin_tx({ from: authorized, ... })` → **passes** (authorized is not restricted).
2. `parse_authorization_list_and_apply_delegations` → sets `restricted`'s code to `0xef0100 || DrainContract`.
3. Transaction body calls `restricted` → runner resolves delegation → `DrainContract` runs in `restricted`'s context → transfers `restricted`'s ETH balance.
4. `validator.finish_tx()` → called after funds are already moved; no parameters to detect the violation.

**Result:** `restricted`'s ETH is transferred without `restricted` ever appearing as `from` in any transaction, and without the `TxValidator` ever seeing `restricted`'s address.
