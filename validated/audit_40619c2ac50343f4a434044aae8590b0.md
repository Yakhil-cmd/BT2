### Title
Unbounded Loop in `calculate_rewards()` Can Permanently Freeze Pool Member's Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards()` function in the `Pool` contract contains an unbounded `while` loop that iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace since their last claim. Because there is no cap on how many checkpoints can accumulate, a pool member who changes their delegated balance across many epochs without claiming rewards will eventually be unable to call `claim_rewards()` — permanently freezing their unclaimed yield.

---

### Finding Description

`claim_rewards()` delegates reward computation to `calculate_rewards()`, which loops over the pool member's balance trace: [1](#0-0) 

The code itself acknowledges the risk with the comment:

> **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.

The loop runs from the stored cursor `entry_to_claim_from` up to `pool_member_trace_length`. The cursor is only advanced on a **successful** `claim_rewards` call: [2](#0-1) 

Each time a pool member calls `add_to_delegation_pool`, `exit_delegation_pool_intent`, `enter_delegation_pool`, or `switch_delegation_pool` in a **different epoch**, a new checkpoint is appended to their trace via `set_member_balance`: [3](#0-2) 

The key is `get_epoch_plus_k()` (current epoch + K). Because the `PoolMemberBalanceTrace` insert logic only merges entries with the **same** key, operations in distinct epochs each produce a distinct checkpoint: [4](#0-3) 

There is no limit on how many epochs a pool member may participate in, and no mechanism to partially process the trace across multiple transactions. If a pool member changes their balance every epoch for N epochs without claiming, `claim_rewards()` must process all N entries atomically. Once N is large enough to exhaust the Starknet transaction gas/step limit, the call reverts and the rewards are permanently inaccessible.

The same unbounded loop is also exercised by the view function `pool_member_info_v1()`, which calls `calculate_rewards()` without advancing the cursor: [5](#0-4) 

---

### Impact Explanation

A pool member whose `pool_member_epoch_balance` trace has grown large enough will find every call to `claim_rewards()` reverting with an out-of-resources error. Because the cursor `entry_to_claim_from` is only written on success, there is no way to make incremental progress. The pool member's accrued yield is permanently frozen in the contract with no recovery path.

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The Starknet staking protocol is designed for long-term participation. A pool member who:
1. Delegates stake and adjusts it (add/remove) across many epochs, **and**
2. Does not call `claim_rewards()` regularly

will accumulate one new checkpoint per epoch in which they change their balance. After hundreds or thousands of epochs of normal protocol operation, the trace length for active delegators who defer claiming will grow large enough to trigger the gas limit. No privileged access is required — any unprivileged pool member can reach this state through ordinary protocol interactions.

---

### Recommendation

1. **Paginated claiming**: Allow `claim_rewards()` to accept a `max_entries` parameter and process only a bounded batch per call, writing the updated `entry_to_claim_from` cursor back to storage after each partial execution.
2. **Eager cursor advancement**: Advance `entry_to_claim_from` at the time each new balance checkpoint is written (i.e., inside `set_member_balance`), so that the loop in `calculate_rewards` never needs to revisit old entries.
3. **Cap balance-change frequency**: Enforce a maximum number of unprocessed balance checkpoints before a new one can be inserted, forcing the pool member to claim first.

---

### Proof of Concept

1. Pool member Alice calls `enter_delegation_pool` in epoch 1.
2. Alice calls `add_to_delegation_pool` in epochs 2, 3, 4, … N (one call per epoch), never calling `claim_rewards`. Each call appends a new checkpoint to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: epoch + K, ...)`.
3. After N epochs, Alice's trace has N checkpoints and `entry_to_claim_from == 0`.
4. Alice calls `claim_rewards()`. `calculate_rewards()` enters the `while` loop and must iterate N times, reading a storage slot on each iteration.
5. For sufficiently large N, the transaction exceeds Starknet's step/resource limit and reverts.
6. Because `entry_to_claim_from` is only written on success, every subsequent `claim_rewards()` attempt also reverts. Alice's yield is permanently frozen.
