### Title
Missing EIP-684 Storage Non-Emptiness Check in CREATE/CREATE2 Deployment Collision Guard — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM deployment collision guard checks only `code_len != 0 || nonce != 0` before allowing a CREATE/CREATE2 to proceed. It explicitly omits the EIP-684 requirement that the target address must also have **empty storage**. This is the direct analog of the "Consensus Key Re-utilization" class: a previously-used identifier (a contract address with residual storage) is not checked for historical use before a new entity is created at that address.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` function performs the CREATE/CREATE2 collision check:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // ... burn gas, return CreateCollision
}
``` [1](#0-0) 

The comment explicitly acknowledges the omission. The official divergence document confirms it:

> "Deployment doesn't fail if the storage for the deployed address is already used (when nonce is 0 and code is empty)." [2](#0-1) 

A concrete EVM compatibility test is disabled precisely because of this gap:

```yaml
dynamicAccountOverwriteEmpty_Paris.json:
  enabled: false
  comment: Expects revert due to storage collision
``` [3](#0-2) 

Both storage models (`ethereum_storage_model` and `flat_storage_model`) do clear storage when SELFDESTRUCT fires **in the same transaction as deployment** (EIP-6780 path), via `clear_state_impl`: [4](#0-3) 

However, the `mark_for_deconstruction` logic only triggers full account erasure when `deployed_in_tx == Some(cur_tx)`: [5](#0-4) 

This means that if a contract address accumulates storage through any path that leaves `nonce = 0` and `code = empty` (e.g., a pre-existing storage slot written by a parent contract via DELEGATECALL, or a cross-transaction scenario where EIP-6780 semantics diverge from mainnet), a subsequent CREATE2 to that address will succeed in ZKsync OS where mainnet EVM would revert.

---

### Impact Explanation

A new contract deployed at an address with pre-existing storage **inherits that storage silently**. Any logic in the new contract that assumes its storage slots are zero-initialized at construction will be violated. This can lead to:

- Incorrect initial state (e.g., inherited ownership, inherited balances in a token contract, inherited access-control flags)
- State-transition divergence between ZKsync OS and mainnet EVM, breaking cross-chain equivalence guarantees

The impact class is **EVM semantic mismatch / state-transition bug**.

---

### Likelihood Explanation

Exploitability in the Cancun context is **constrained but non-zero**:

- EIP-6780 prevents the classic "deploy → write storage → SELFDESTRUCT in different tx → redeploy" path because SELFDESTRUCT in a non-creation transaction no longer clears code/nonce, so the existing `code_len/nonce` check would catch it.
- Same-transaction SELFDESTRUCT does clear storage in both storage models, closing that path.
- However, the disabled test (`dynamicAccountOverwriteEmpty_Paris.json`) confirms a reachable scenario exists where ZKsync OS allows deployment where mainnet would revert. Any contract pattern that relies on EIP-684 for security (e.g., factory contracts that assume a freshly-deployed address is truly clean) is affected.
- An attacker who can pre-seed storage at a deterministic CREATE2 address (e.g., via a DELEGATECALL from a contract whose address is the target) before the victim deploys there can exploit this.

---

### Recommendation

Add a storage non-emptiness check to the CREATE/CREATE2 collision guard in `before_executing_frame`. Before allowing the constructor to execute, verify that the target address has no non-zero storage slots. If a full storage scan is not feasible, at minimum track whether any storage was ever written to the address (e.g., via a per-address "ever-written" flag in the account metadata) and use that as a proxy for the EIP-684 check. [6](#0-5) 

---

### Proof of Concept

1. Deploy `Factory` contract at address `F` via CREATE2 with salt `S`.
2. `Factory` writes a non-zero value to storage slot 0 (e.g., `SSTORE(0, 1)`).
3. `Factory` calls SELFDESTRUCT targeting itself in the **same transaction** as deployment.
   - In ZKsync OS: `should_be_deconstructed = true` → `clear_state_impl` clears storage → no residual storage. *(This path is safe.)*
4. **Alternative path**: Use DELEGATECALL from a contract at address `F` to write to `F`'s storage, then have `F`'s code removed via a governance force-deploy that sets empty bytecode. Now `F` has `nonce=0`, `code=empty`, `storage[0]=1`.
5. Attacker calls CREATE2 with the same salt `S` and new initcode targeting address `F`.
6. ZKsync OS: deployment succeeds (only `code_len` and `nonce` are checked, both zero). New contract starts with `storage[0] = 1` inherited from the old contract.
7. Mainnet EVM: deployment reverts with `CreateCollision` (EIP-684 storage check).

The divergence is confirmed by the disabled test `dynamicAccountOverwriteEmpty_Paris.json` which expects a revert that ZKsync OS does not produce. [7](#0-6) [8](#0-7)
