### Title
Unchecked `transferFrom` Return Value in XCC Precompile Allows Fee-Free Cross-Contract Calls - (File: `engine-precompiles/src/xcc.rs`)

### Summary

The `CrossContractCall` precompile calls `transferFrom` on the wNEAR ERC-20 token to collect payment from the caller before executing a cross-contract call. The EVM call's exit reason is checked, but the boolean return value of `transferFrom` is captured and never validated. If wNEAR's `transferFrom` returns `false` without reverting, the precompile silently proceeds as if payment succeeded, allowing the caller to execute cross-contract calls without paying the required wNEAR.

### Finding Description

In `engine-precompiles/src/xcc.rs`, the `run_with_handle` function of `CrossContractCall` computes a `required_near` amount and, when non-zero, issues an internal EVM call to wNEAR's `transferFrom` to move that amount from the caller to the engine's implicit address: [1](#0-0) 

The call returns `(exit_reason, return_value)`. The `match` on `exit_reason` handles `Revert`, `Error`, and `Fatal` correctly, but on `Succeed` it does nothing — `return_value` (the ABI-encoded `bool` from `transferFrom`) is never decoded or checked.

The ERC-20 standard allows a compliant token to signal failure by returning `false` rather than reverting. If wNEAR's `transferFrom` ever returns `false` without reverting (e.g., due to an upgrade, a replacement token at the configured address, or any non-standard implementation), the precompile falls through to the `Succeed` arm and continues to schedule the cross-contract call without the payment having been collected.

The `transfer_from_args` helper correctly encodes the `transferFrom(address,address,uint256)` selector and arguments: [2](#0-1) 

But there is no corresponding decode-and-assert step for the returned `bool`.

### Impact Explanation

`required_near` covers both the NEAR attached to the user's cross-contract promises and, for first-time callers, the storage-staking deposit (`STORAGE_AMOUNT = 2 × 10²⁴ yoctoNEAR`): [3](#0-2) 

If the transfer silently fails, the engine's NEAR balance funds the XCC call while the caller retains their wNEAR. Repeated exploitation drains the engine's NEAR reserve, leading to insolvency of the Aurora engine account and inability to fund legitimate operations — a **Critical** impact (insolvency / permanent fund freeze).

### Likelihood Explanation

The wNEAR address is a configurable storage value (`get_wnear_address`): [4](#0-3) 

Any wNEAR contract that does not revert on a failed `transferFrom` (e.g., insufficient allowance or balance) but instead returns `false` — a known non-standard ERC-20 pattern — triggers this path. The vulnerability is latent and becomes exploitable the moment such a token is set as wNEAR or the existing wNEAR contract is upgraded with non-reverting failure semantics. An unprivileged EVM user with zero wNEAR balance can trigger it immediately if the condition holds.

### Recommendation

After the `Succeed` arm, decode `return_value` and revert if the decoded boolean is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool return value of transferFrom
    let success = return_value
        .get(31)
        .map(|&b| b == 1)
        .unwrap_or(false);
    if !success {
        return Err(revert_with_message("ERR_WNEAR_TRANSFER_FAILED"));
    }
}
```

This is the Rust/EVM equivalent of using `safeTransferFrom` from `SafeERC20`: it checks the return value and reverts on failure regardless of whether the underlying token reverts or returns `false`.

### Proof of Concept

1. Deploy or configure a wNEAR-compatible ERC-20 at the engine's wNEAR address whose `transferFrom` returns `false` (without reverting) when the caller has insufficient allowance.
2. As an unprivileged EVM user with zero wNEAR allowance granted to the XCC precompile, call the `CrossContractCall` precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a valid `CrossContractCallArgs::Eager` payload.
3. The precompile computes `required_near > 0`, calls `transferFrom` on wNEAR, receives `ExitReason::Succeed` with `return_value = abi.encode(false)`.
4. The `match` arm for `Succeed` does nothing; execution continues.
5. The cross-contract call promise is scheduled and emitted in the log at lines 219–231, funded by the engine's own NEAR balance.
6. The caller paid nothing; the engine's NEAR reserve is decremented by `required_near` per call. [5](#0-4)
