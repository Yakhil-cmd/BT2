### Title
Missing Failure Callback on NEAR Transfer Batch Promise in wNEAR Unwrap Path Causes Permanent Loss of User Funds — (`engine/src/contract_methods/connector.rs`)

### Summary

When a user calls `withdrawToNear` on the wNEAR ERC-20 contract with a `:unwrap` suffix, the engine executes a two-step NEAR promise chain: (1) `near_withdraw` on the wNEAR NEP-141 contract, then (2) a `PromiseBatchAction::Transfer` to the user-specified `target_account_id`. If `near_withdraw` succeeds but the subsequent NEAR transfer fails (e.g., `target_account_id` is a non-existent named account), the user's wNEAR ERC-20 tokens are permanently burned and the equivalent NEAR is silently retained in Aurora's contract balance with no refund path.

### Finding Description

**Step 1 — EVM burn (irreversible):**
In `EvmErc20.sol`, `withdrawToNear` calls `_burn(_msgSender(), amount)` before invoking the precompile. The ERC-20 balance is destroyed at EVM execution time. [1](#0-0) 

**Step 2 — Precompile constructs the wNEAR unwrap path:**
In `exit_erc20_token_to_near`, when the caller is the wNEAR ERC-20 address and the message is `UnwrapWnear`, the method is set to `"near_withdraw"` and `transfer_near_args` is populated with the user-supplied `receiver_account_id` and `amount`. No existence check is performed on `receiver_account_id`. [2](#0-1) 

**Step 3 — Promise chain is constructed:**
The precompile emits a `PromiseWithCallbackArgs` where the base is `near_withdraw` (burns wNEAR NEP-141, releases NEAR to Aurora's balance) and the callback is `exit_to_near_precompile_callback`. [3](#0-2) 

**Step 4 — Callback creates the NEAR transfer with no further callback:**
Inside `exit_to_near_precompile_callback`, when `near_withdraw` succeeds (`PromiseResult::Successful`), a `PromiseBatchAction::Transfer` is created targeting `args.target_account_id`. The promise is returned via `handler.promise_return(promise_id)` — **no callback is attached to handle failure of this transfer**. [4](#0-3) 

**Step 5 — The refund branch is unreachable for this failure mode:**
The `else if let Some(args) = args.refund` branch at line 231 only executes when `near_withdraw` itself fails (promise result is not `Successful`). It cannot handle the case where `near_withdraw` succeeded but the subsequent NEAR transfer failed, because by the time the transfer fails, `exit_to_near_precompile_callback` has already returned. [5](#0-4) 

**The `refund` field is also `None` for the wNEAR path when `error_refund` is disabled:** [6](#0-5) 

**The `TransferNearArgs` struct carries no on-chain existence guarantee:** [7](#0-6) 

### Impact Explanation

In NEAR protocol, a `Transfer` action targeting a non-existent named account (e.g., `nonexistent.near`) fails with `AccountDoesNotExist`. When this happens:

- The user's wNEAR ERC-20 balance is already burned (EVM state, irreversible).
- The wNEAR NEP-141 balance held by Aurora is already burned via `near_withdraw` (irreversible).
- The NEAR that was released by `near_withdraw` stays in Aurora's contract NEAR balance.
- The user receives 0 NEAR.
- There is no callback, no refund, and no recovery path.

The invariant `wNEAR_burned == NEAR_received_by_target_account_id` is permanently violated. The NEAR is effectively transferred from the user to Aurora's unaccounted balance.

### Likelihood Explanation

The `target_account_id` is fully user-controlled (passed as the `recipient` bytes argument to `withdrawToNear`). Any syntactically valid NEAR account ID that does not exist on-chain triggers this path. This includes: typos in account names, accounts that have been deleted, or accounts that have never been created. The existing test suite (`test_exit_to_near_wnear_unwrapped`) only covers the happy path where the target account exists and is registered. [8](#0-7) 

Notably, the analogous failure case for regular ERC-20 `ft_transfer` is already tested and handled via the `error_refund` feature, but no equivalent test or handler exists for the wNEAR NEAR transfer failure. [9](#0-8) 

### Recommendation

After `promise_create_batch` for the NEAR transfer, attach a second callback (e.g., a new `exit_to_near_transfer_callback` method on Aurora) using `handler.promise_attach_callback`. That callback should check the transfer promise result; if it failed, re-mint the equivalent wNEAR ERC-20 tokens to the original sender's EVM address (mirroring the `refund_on_error` pattern already used for `ft_transfer` failures). Alternatively, validate that `target_account_id` exists on-chain before proceeding, though this requires an additional cross-contract view call and is more complex.

### Proof of Concept

```
1. Deploy Aurora with wNEAR configured (factory_set_wnear_address).
2. Bridge wNEAR into Aurora EVM: user holds N wNEAR ERC-20 tokens.
3. User calls withdrawToNear(bytes("nonexistent_named_account.near:unwrap"), N)
   on the wNEAR ERC-20 contract.
4. EVM burns N wNEAR ERC-20 tokens from user.
5. ExitToNear precompile schedules:
     near_withdraw({"amount": N}) → exit_to_near_precompile_callback
6. near_withdraw succeeds: Aurora's wNEAR NEP-141 balance decreases by N,
   Aurora's NEAR balance increases by N yoctoNEAR.
7. exit_to_near_precompile_callback runs, creates Transfer(N) to
   "nonexistent_named_account.near", calls promise_return — no callback.
8. Transfer fails: AccountDoesNotExist.
9. Assert: user's wNEAR ERC-20 balance = 0 (burned).
10. Assert: "nonexistent_named_account.near" NEAR balance = 0 (never received).
11. Assert: Aurora's NEAR balance increased by N (retained).
```

```rust
        let result = exit_to_near(
            &ft_owner,
            &format!("{}:unwrap", ft_owner.id().as_ref()),
            FT_EXIT_AMOUNT,
            &wnear_erc20,
            &aurora,
        )
        .await;
        let total_tokens_burnt: u128 = result
            .outcomes()
            .iter()
            .map(|o| o.tokens_burnt.as_yoctonear())
            .sum();

        // We need to skip at least 1 block before checking ft_owner's balance, because the refund
        // receipt is executed in the next(skipped) block, because the transaction broadcasts with
        // default `wait_until` parameter, which is [ExecutedOptimistic](https://github.com/near/nearcore/blob/master/core/primitives/src/views.rs#L1743)
        // since nearcore 1.39.0.
        aurora.node.skip_blocks(1).await.unwrap();

        // Check that the wnear tokens are properly unwrapped and transferred to `ft_owner`
        assert_eq!(
            aurora.node.get_balance(&ft_owner.id()).await.unwrap(),
            ft_owner_balance - total_tokens_burnt + FT_EXIT_AMOUNT
        );

        // Check wnear balances
        assert_eq!(
            nep_141_balance_of(&wnear, &ft_owner.id()).await,
            FT_TOTAL_SUPPLY - FT_TRANSFER_AMOUNT
        );
        assert_eq!(
            nep_141_balance_of(&wnear, &aurora.id()).await,
            FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT
        );
        assert_eq!(
            erc20_balance(&wnear_erc20, ft_owner_address, &aurora).await,
            (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into()
        );
```
