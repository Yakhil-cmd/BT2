### Title
Malicious Participant Can Corrupt CKD Output by Sending Shares Computed for a Different `app_id` Without Detection - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD protocol's coordinator blindly aggregates participant shares with no proof of correct computation. A malicious participant can send a share computed for a different `app_id` (or `app_pk`) than the one agreed upon. Because the coordinator performs no verification, the aggregated `(Y, C)` output is silently corrupted, causing honest parties to accept an unusable CKD output.

### Finding Description
In `do_ckd_participant`, each participant computes `(norm_big_y, norm_big_c)` locally using their own `app_id` and `app_pk`, then sends the result privately to the coordinator with no attached proof of correct computation:

```rust
fn do_ckd_participant(..., app_id: &AppId, app_pk: PublicKey, ...) {
    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
    Ok(None)
}
```

In `do_ckd_coordinator`, the coordinator sums every received pair unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

There is no NIZK proof, commitment, or cross-participant consistency check that binds each `(norm_big_y, norm_big_c)` to the agreed-upon `(app_id, app_pk)`. The coordinator cannot distinguish a correctly computed share from one computed for an entirely different application identifier.

This is structurally analogous to the reported `init_if_needed` bug: just as that bug allowed a malicious actor to overwrite a shared `ForeignContract` account by creating a new bridge instance that reused the same underlying state, here a malicious participant can "overwrite" their contribution to the shared aggregated output by substituting a share computed for a different `app_id`, corrupting the result for all honest parties. ### Impact Explanation
The correct CKD output satisfies:

```
C = msk · H(pk ‖ app_id) + y · A
```

If malicious participant `P_m` substitutes `app_id'` for `app_id`, the coordinator computes:

```
C_corrupted = (msk − λ_m · x_m) · H(pk ‖ app_id)
            + λ_m · x_m · H(pk ‖ app_id')
            + y · A
```

This is not equal to `msk · H(pk ‖ app_id) + y · A`. When the TEE application unmasks the output with its secret scalar `a`:

```
sig = C_corrupted − a · Y  ≠  msk · H(pk ‖ app_id)
```

The resulting BLS signature is invalid. The application cannot derive the correct deterministic key `s`. All honest parties — the coordinator and the requesting TEE app — accept this corrupted, unusable output without any error signal from the protocol itself.

This maps directly to the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.** [3](#0-2) [4](#0-3) 

### Likelihood Explanation
The CKD protocol is a single-round protocol with no interactive verification. Any one of the `n` participants can mount this attack unilaterally. The attacker only needs to call `ckd()` with the correct participant list and key material but substitute a different `app_id` when computing their local share. No cryptographic capability beyond normal participation is required. The coordinator has no observable signal that anything is wrong — the protocol completes normally and returns `Some(CKDOutput)`. [5](#0-4) 

### Recommendation
Add a non-interactive zero-knowledge proof of correct computation to each participant's message. Specifically, each participant should prove in zero knowledge that `(norm_big_y, norm_big_c)` was computed as `(λ_i · y_i · G, λ_i · (x_i · H(pk ‖ app_id) + y_i · A))` for the public values `(pk, app_id, A, participants)`. The coordinator must verify all proofs before aggregating. This is the standard approach used in threshold BLS and ElGamal-based protocols to prevent malicious share substitution. [6](#0-5) 

### Proof of Concept
1. Honest participants `P1`, `P2`, `P3` agree to run CKD for `app_id = "app1"` and `app_pk = A`.
2. Malicious participant `P3` calls `ckd(participants, coordinator, P3, key_pair, "app2", A, rng)` — using `"app2"` instead of `"app1"`.
3. `P3`'s `compute_signature_share` computes `S_3 = x_3 · H(pk ‖ "app2")` and returns `(λ_3 · Y_3, λ_3 · C_3')`.
4. `P3` sends this corrupted share privately to the coordinator.
5. The coordinator sums `P1`'s, `P2`'s, and `P3`'s shares. The resulting `C` is not `msk · H(pk ‖ "app1") + y · A`.
6. The coordinator returns `Some(CKDOutput { big_y: Y, big_c: C_corrupted })` with no error.
7. The TEE application computes `sig = C_corrupted − a · Y`, which fails BLS signature verification against `pk` for `"app1"`.
8. The application cannot derive its deterministic key `s`. The CKD output is permanently unusable for the intended `app_id`. [1](#0-0) [3](#0-2)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
