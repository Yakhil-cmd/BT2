### Title
Unprivileged Caller Can Permanently Block Consensus Reward Distribution via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract unconditionally writes the global `last_reward_block` state variable **before** checking the `disable_rewards` flag. Because `update_rewards` is publicly callable with no access control beyond a non-zero-caller check, any attacker can call it with `disable_rewards: true` every block, consuming the block's single reward slot without distributing any rewards to stakers.

---

### Finding Description

`update_rewards` is exposed as part of `IStakingRewardsManager` with no privileged-role guard — `general_prerequisites()` only asserts the contract is unpaused and the caller is non-zero. [1](#0-0) 

The function enforces a global, single-use-per-block invariant via `last_reward_block`: [2](#0-1) 

After validating the staker, it **unconditionally commits** `last_reward_block = current_block_number`: [3](#0-2) 

Only **after** that write does it check whether rewards should actually be distributed: [4](#0-3) 

Because `last_reward_block` is written first, a caller who passes `disable_rewards: true` silently consumes the block's reward slot and returns without distributing anything. Every subsequent call in the same block fails with `REWARDS_ALREADY_UPDATED`. The attacker repeats this every block.

This is structurally identical to the reported flash-loan bug: in that case `handleRepayment` updated `_underlyingAmount` without calling `_rebalance`; here `update_rewards` updates `last_reward_block` without distributing rewards — the "return path" performs the state mutation but skips the essential operation.

---

### Impact Explanation

An attacker who front-runs the first `update_rewards` call every block permanently prevents all stakers from accruing consensus rewards. No staker's `unclaimed_rewards_own` is ever incremented, and no pool ever receives rewards via `update_pool_rewards`. This constitutes **permanent freezing of unclaimed yield** for the entire protocol. [5](#0-4) 

---

### Likelihood Explanation

- `update_rewards` is a public, permissionless entry point.
- The attacker needs only a valid (active, non-zero-balance) staker address to pass the staker-existence checks — any staker address visible on-chain suffices.
- The cost is one transaction per block (gas only); no capital is at risk for the attacker.
- The attack is sustainable indefinitely.

---

### Recommendation

Move `last_reward_block.write(current_block_number)` to **after** the `disable_rewards` / pre-consensus guard, so the slot is only consumed when rewards are actually distributed:

```cairo
// Update last block rewards ONLY when rewards will be distributed.
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number);
// ... rest of reward distribution logic
```

Alternatively, add access control so that only the attestation contract or the staker's operational address may call `update_rewards` with `disable_rewards: true`.

---

### Proof of Concept

1. Attacker observes any active staker address `S` on-chain.
2. At the start of every block, attacker calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
3. Inside the call:
   - `current_block_number > last_reward_block` → passes (new block).
   - Staker `S` exists and has non-zero balance → passes.
   - `last_reward_block` is written to `current_block_number`. [3](#0-2) 
   - `disable_rewards == true` → early return, zero rewards distributed. [4](#0-3) 
4. Any legitimate call to `update_rewards` in the same block fails: `current_block_number > last_reward_block` is now `false`. [6](#0-5) 
5. Repeated every block: no staker ever accumulates `unclaimed_rewards_own`; no pool ever receives rewards. The entire consensus reward system is permanently frozen.
