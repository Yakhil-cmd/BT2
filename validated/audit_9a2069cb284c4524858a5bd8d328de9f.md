Audit Report

## Title
Missing `http2_max_header_list_size` on SOCKS Proxy Client Enables Unbounded Header Allocation in Adapter Process — (`rs/https_outcalls/adapter/src/rpc_server.rs`)

## Summary

`create_socks_proxy_client` builds a hyper `Client` without `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)`, while the direct client in `CanisterHttp::new` applies the 52 KB cap. Because the SOCKS path is the production route for IPv4-only destinations on IPv6-only replica nodes, an attacker-controlled HTTP/2 server can send arbitrarily large response header blocks through the uncapped SOCKS client, driving the adapter process toward OOM and denying canister HTTP outcall service on affected nodes.

## Finding Description

`MAX_HEADER_LIST_SIZE` is defined as 52 KB and applied to the direct client: [1](#0-0) [2](#0-1) 

`create_socks_proxy_client` omits this call entirely: [3](#0-2) 

When `http2_max_header_list_size` is not set, the h2 crate defaults to `u32::MAX` — no limit. The `SETTINGS_MAX_HEADER_LIST_SIZE` frame sent to the server will not advertise any cap, so a cooperative server may legally send arbitrarily large HPACK-decoded header blocks; h2 allocates memory for all of them before returning control to the caller.

The SOCKS fallback is triggered automatically when the direct connection fails, which is the documented production path for IPv4-only destinations on IPv6-only replicas: [4](#0-3) 

The SOCKS proxy addresses are sourced from the registry (boundary node IPs at port 1080) and are not attacker-controlled. The boundary node SOCKS proxy is a transparent TCP relay — the attacker's server is the HTTP/2 peer, not the proxy itself.

Post-response header accounting occurs after h2 has already allocated the full header list: [5](#0-4) 

This post-hoc size check does not prevent the allocation from occurring inside h2's frame parser.

## Impact Explanation

An attacker-controlled HTTP/2 server can send a response with a header list of arbitrary size (bounded only by h2's internal frame-level limits, which are orders of magnitude larger than 52 KB). Each such request causes the adapter process to allocate memory proportional to the header list size. With multiple concurrent canister HTTP outcalls routed through SOCKS — the normal production path for IPv4 destinations on IPv6-only replicas — an attacker can drive the adapter process toward OOM, crashing or severely degrading canister HTTP outcall service on affected replica nodes.

This matches: **High ($2,000–$10,000) — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS.** The impact is scoped to the adapter process on nodes that process the malicious outcalls; it does not affect consensus integrity.

## Likelihood Explanation

Any canister developer can deploy a canister that issues HTTP outcalls to an IPv4-only address they control. On IPv6-only replica nodes (the standard IC production deployment), the direct connection to an IPv4 address fails immediately, and the SOCKS path is taken automatically. No privileged access, no key material, no governance majority, and no network-level attack is required. The attacker's server is then the HTTP/2 peer and can send any response headers it chooses. The attack is repeatable with many concurrent outcalls to amplify memory pressure.

## Recommendation

Add `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)` to the `Client::builder` call inside `create_socks_proxy_client`, mirroring the direct client construction in `CanisterHttp::new`:

```rust
// rs/https_outcalls/adapter/src/rpc_server.rs, create_socks_proxy_client
Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)   // add this
    .build::<_, Full<Bytes>>(
        builder
            .enable_all_versions()
            .wrap_connector(SocksConnector { ... }),
    )
```

## Proof of Concept

1. Deploy a canister that issues an HTTP outcall to an IPv4-only address you control (e.g., `https://203.0.113.1/`).
2. On that server, run an HTTP/2 responder that, upon any request, sends a `200 OK` with a `HEADERS` frame whose decoded header list totals ~10 MB (e.g., 1,000 headers each with a 10 KB value).
3. On an IPv6-only replica node, submit the canister HTTP outcall. The direct connection to the IPv4 address fails; the adapter falls back to `do_https_outcall_socks_proxy`.
4. The SOCKS client (built without `http2_max_header_list_size`) forwards the request through the boundary node SOCKS proxy to your server.
5. Your server sends the oversized `HEADERS` frame. h2 allocates ~10 MB for that single response before the adapter's post-hoc size check runs.
6. Repeat with many concurrent outcalls. Observe adapter process RSS growing without the 52 KB-per-response bound that the direct client enforces.
7. **Differential confirmation**: send the same oversized response to the direct client (by making the server reachable over IPv6). The direct client returns an h2 protocol error and allocates nothing beyond the frame header. The SOCKS client accepts and allocates the full header list.
