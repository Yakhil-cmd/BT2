### Title
Unchecked Exit Precompile Call Return Value Causes Permanent Token Burn Without NEP-141 Transfer - (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear` and `withdrawToEthereum` burn the caller's ERC-20 tokens first, then invoke the exit precompile via inline assembly. The return value `res` of the `call` opcode is assigned but **never checked**. If the precompile call fails for any reason, execution continues without reverting: the ERC-20 tokens are permanently destroyed while no NEAR-side NEP-141 transfer promise is ever scheduled, permanently locking the corresponding NEP-141 tokens inside the Aurora contract.

### Finding Description

In `EvmErc20.sol` lines 53–63 and `EvmErc20V2.sol` lines 53–63, the withdraw functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ERC-20 tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert if call returns 0
    }
}
```

The same pattern appears in `withdrawToEthereum` in both contracts (lines 65–76 / 66–77), calling `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `ExitError` — causing the low-level `call` to return 0 — in multiple reachable conditions:

- `ERR_INVALID_IN_STATIC` / `ERR_INVALID_IN_DELEGATE` if called from a non-standard context
- Parsing failure in `ExitToNearParams::try_from(input)` if the recipient bytes do not form a valid NEAR account ID
- `ERR_TARGET_TOKEN_NOT_FOUND` if the ERC-20 address has no registered NEP-141 mapping
- `ExitError::OutOfGas` if gas is exhausted before the precompile completes

Because `res` is never inspected and there is no `require(res != 0)` guard, the EVM transaction succeeds, the burn is committed, but no `PromiseArgs` log is emitted and no NEAR-side `ft_transfer` is ever scheduled.

### Impact Explanation

When the precompile call silently fails:

1. The caller's ERC-20 balance is reduced by `amount` (burned, irreversible).
2. No NEAR promise is created, so the NEP-141 tokens that back those ERC-20 tokens remain permanently locked inside the Aurora contract.
3. There is no recovery path: the ERC-20 tokens no longer exist to trigger a future burn-and-exit, and the NEP-141 tokens cannot be reclaimed.

This constitutes **permanent freezing of funds** (Critical per the allowed impact scope). Over repeated occurrences the Aurora contract accumulates unbacked NEP-141 reserves, creating an insolvency risk for other users who hold ERC-20 tokens backed by the same NEP-141.

### Likelihood Explanation

The trigger is reachable by any unprivileged ERC-20 token holder:

- A user calling `withdrawToNear` with a malformed or invalid NEAR account ID as `recipient` causes `ExitToNearParams::try_from` to return an `ExitError`, making `call` return 0.
- The precompile is also reachable from any contract that calls `withdrawToNear` on behalf of a user, where the recipient string is attacker-controlled.
- The `error_refund` feature flag, which adds a callback to refund on NEAR-side failure, does **not** protect against this: it only handles asynchronous NEAR promise failures, not synchronous precompile call failures where no promise is ever created.

Likelihood is **medium**: it requires either a user mistake (invalid recipient) or a specific failure condition in the precompile, but the code path is fully reachable without any privileged access.

### Recommendation

Check the return value of the precompile `call` and revert if it fails, so that the `_burn` is also reverted:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;
    bool success;

    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ERR_EXIT_TO_NEAR_FAILED");
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

### Proof of Concept

1. Deploy `EvmErc20` with a valid NEP-141 mapping.
2. Mint `N` ERC-20 tokens to address `A`.
3. From address `A`, call `withdrawToNear(invalidRecipient, N)` where `invalidRecipient` is a byte string that fails NEAR account ID validation (e.g., `bytes("!!invalid!!")`).
4. The `ExitToNear` precompile returns `ExitError` → `call` returns 0 → `res` is ignored → transaction succeeds.
5. Observe: `balanceOf(A) == 0` (tokens burned), but no NEP-141 `ft_transfer` was ever scheduled on NEAR. The NEP-141 tokens remain locked in the Aurora contract with no mechanism to recover them. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
