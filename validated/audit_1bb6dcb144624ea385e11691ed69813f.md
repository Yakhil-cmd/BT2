### Title
Missing Validation of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function accepts `CKDOutput` values (group elements `big_y`, `big_c`) from all participants and sums them without any cryptographic proof that the contributions are correctly formed. A malicious participant can send arbitrary group elements, causing the coordinator to compute and return an incorrect CKD output that all honest parties will accept as valid.

### Finding Description
In `do_ckd_coordinator`, the coordinator collects each participant's `(big_y, big_c)` pair and accumulates them unconditionally: [1](#0-0) 

The correct contribution from participant `i` is:
- `big_y = lambda_i * y_i * G` (Lagrange-weighted nonce commitment)
- `big_c = lambda_i * (x_i * H(pk || app_id) + y_i * app_pk)` (Lagrange-weighted ElGamal ciphertext component) [2](#0-1) 

There is no zero-knowledge proof or any other mechanism to verify that the received `big_y` and `big_c` are actually derived from the participant's legitimate secret share `x_i` and a valid nonce `y_i`. The coordinator blindly adds whatever group elements it receives.

Contrast this with the DKG protocol, which validates every participant contribution via proof-of-knowledge and commitment-hash checks before incorporating them: [3](#0-2) 

No equivalent validation exists in the CKD protocol.

### Impact Explanation
The final CKD output `(Y, C)` is the sum of all participant contributions. When `unmask(app_sk)` is called, it computes `C - app_sk * Y`, which should equal `msk * H(pk || app_id)`. If any participant sends arbitrary `(big_y', big_c')`, the sum is corrupted and `unmask` produces an incorrect, unusable confidential key. The coordinator returns this corrupted output and honest parties accept it, as there is no per-contribution verification step.

**Impact class:** High — Corruption of CKD output so honest parties accept an unusable cryptographic output.

### Likelihood Explanation
Any participant in the CKD protocol can trivially exploit this. The attacker only needs to be a legitimate protocol participant (no key leakage or external compromise required). They simply send arbitrary group elements instead of their correct contribution. The protocol provides no detection mechanism.

### Recommendation
Each participant should accompany their `(big_y, big_c)` contribution with a zero-knowledge proof of correct formation, proving:
1. Knowledge of a nonce `y_i` such that `big_y = lambda_i * y_i * G`.
2. That `big_c = lambda_i * (x_i * H(pk || app_id) + y_i * app_pk)` is consistent with the participant's committed public share `x_i * G2` (established during DKG).

The coordinator must verify these proofs before accumulating each contribution, analogous to how `verify_proof_of_knowledge` and `validate_received_share` gate every contribution in `do_keyshare`. [4](#0-3) 

### Proof of Concept
1. Participants `{P_1, P_2, P_m}` run the CKD protocol; `P_m` is malicious.
2. `P_m` computes the correct `(norm_big_y, norm_big_c)` per `compute_signature_share` but instead sends `(G1::generator(), G1::generator())` — arbitrary group elements — to the coordinator.
3. The coordinator executes the loop at lines 50–55 and adds `P_m`'s arbitrary values to the running sum without any check.
4. The resulting `(Y, C)` is `(Y_honest + G, C_honest + G)` rather than the correct aggregate.
5. `ckd_output.unmask(app_sk)` returns `C_honest + G - app_sk * (Y_honest + G)`, which is not equal to `msk * H(pk || app_id)`.
6. The coordinator returns this corrupted `CKDOutput` as `Some(ckd_output)` and the caller accepts it as the legitimate confidential key. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
