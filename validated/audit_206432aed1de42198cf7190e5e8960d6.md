### Title
Missing Identity-Element Check on `app_pk` Allows Coordinator to Directly Extract Confidential Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` entry point in the Confidential Key Derivation (CKD) protocol accepts the application public key `app_pk` (of type `PublicKey = ElementG1`) without validating that it is not the group identity element. A malicious coordinator who supplies `app_pk = G1::identity()` to honest participants causes the ElGamal masking term to vanish, so each participant's share `C_i` collapses to the unmasked secret share `S_i = x_i · H(pk, app_id)`. The coordinator then directly reconstructs the confidential derived key `msk · H(pk, app_id)` from the protocol messages, without ever needing `app_sk`.

### Finding Description
In `src/confidential_key_derivation/protocol.rs`, the public entry point `ckd()` performs several input validations (participant count, duplicate check, self-membership, coordinator membership) but performs **no check** that `app_pk` is a non-identity element of G1. [1](#0-0) 

The masking computation inside `compute_signature_share` is:

```
C_i = S_i + app_pk * y_i
```

where `S_i = H(pk, app_id) * x_i` is the participant's secret contribution and `y_i` is a fresh random scalar. [2](#0-1) 

When `app_pk = G1::identity()`, the masking term `app_pk * y_i` evaluates to the identity element, so `C_i = S_i`. The coordinator then sums the Lagrange-weighted shares:

```
C = Σ λ_i · C_i = Σ λ_i · S_i = msk · H(pk, app_id)
```

This is the confidential derived key itself, delivered in plaintext to the coordinator.

The `CKDOutput::unmask` function confirms the intended design: the coordinator is supposed to need `app_sk` to recover the key via `C − app_sk · Y`. With `app_pk = identity`, `Y` is still a random non-zero point, but `C` already equals the target key, so the coordinator can read it directly without `app_sk`. [3](#0-2) 

By contrast, the robust ECDSA presign protocol correctly validates that the interpolated nonce `R` is not the identity and that the scalar `w` is not zero before proceeding. [4](#0-3) 

No equivalent guard exists for `app_pk` in the CKD path.

### Impact Explanation
When `app_pk = G1::identity()` is supplied, the ElGamal masking that protects the confidential derived key is completely bypassed. The coordinator receives `C = msk · H(pk, app_id)` in the clear — the exact value that the CKD protocol is designed to keep secret from the coordinator. This constitutes **disclosure of a confidential derived secret**, matching the Critical impact tier: *"Extraction, reconstruction, or disclosure of … confidential derived secrets."*

### Likelihood Explanation
The `app_pk` parameter is caller-supplied with no library-level validation. In a TEE integration scenario, the coordinator orchestrates the protocol and distributes parameters (including `app_pk`) to participants. A malicious coordinator can trivially pass `G1::identity()` as `app_pk`. Honest participants have no in-library mechanism to detect or reject this. The attack requires only the ability to call `ckd()` with a chosen `app_pk`, which is the normal coordinator role.

### Recommendation
Add an explicit identity-element check on `app_pk` at the start of `ckd()`, before the protocol is launched:

```rust
use elliptic_curve::Group;

if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern already used in the dlog-equality proof, which rejects an identity `generator1`: [5](#0-4) 

### Proof of Concept

```
1. Coordinator calls ckd() for all participants with app_pk = G1::identity().
2. Each honest participant executes compute_signature_share():
     y_i  <- random scalar
     Y_i  = G * y_i
     S_i  = H(pk, app_id) * x_i
     C_i  = S_i + identity * y_i  =  S_i          ← masking vanishes
     norm_C_i = λ_i · C_i
3. Coordinator sums received shares:
     C = Σ norm_C_i = Σ λ_i · S_i = msk · H(pk, app_id)
4. Coordinator reads C directly — this IS the confidential derived key.
   No app_sk is needed; unmask(0) = C − 0·Y = C = msk·H(pk,app_id).
``` [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
