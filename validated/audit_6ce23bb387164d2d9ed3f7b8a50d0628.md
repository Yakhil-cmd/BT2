### Title
Declare Transactions Bypass `max_l2_gas_amount` Admission Check in `StatelessTransactionValidator::validate_resource_bounds` — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `max_l2_gas_amount` upper-bound check for `RpcTransaction::Declare` while enforcing it for `Invoke` and `DeployAccount`. No compensating check exists anywhere in the stateful path for gas *amount* (only gas *price* is checked there). Any unprivileged user can submit a Declare transaction with `l2_gas.max_amount = u64::MAX` and have it pass the gateway's stateless admission gate.

### Finding Description

In `validate_resource_bounds`, the check that rejects transactions whose `l2_gas.max_amount` exceeds `config.max_l2_gas_amount` is guarded by an explicit Declare-type bypass:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1,210,000,000`. [2](#0-1) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas **price** against the previous block's price via `validate_tx_l2_gas_price_within_threshold`; it performs no check on the gas **amount** for any transaction type. [3](#0-2) [4](#0-3) 

The only downstream catch is the blockifier's `verify_can_pay_committed_bounds`, which checks whether the account balance covers `max_amount × max_price_per_unit`. [5](#0-4) 

This is called inside `perform_pre_validation_stage` only when `charge_fee` is true. [6](#0-5) 

The existing test suite **explicitly asserts** this bypass as expected behavior — `valid_l2_gas_amount_on_declare` passes a Declare with `max_amount: 200` against a limit of `100` and asserts `Ok(())`. The `test_invalid_max_l2_gas_amount` test only covers `DeployAccount` and `Invoke`, deliberately excluding `Declare`. [7](#0-6) [8](#0-7) 

### Impact Explanation

The `max_l2_gas_amount` limit is the gateway's primary admission-time DoS and resource-bound policy control. For Declare transactions it is entirely unenforced at the stateless layer, and the stateful layer has no compensating amount check. A Declare transaction with `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit = min_gas_price` passes `StatelessTransactionValidator::validate` and proceeds through stateful pre-checks (gas price check passes, nonce check passes) all the way to blockifier execution. The blockifier only rejects it if the account cannot cover `u64::MAX × min_gas_price` in STRK balance — a check that is irrelevant for an attacker who controls an account with sufficient balance, or who simply wants to flood the stateful validation path with expensive-to-process Declare transactions that the stateless gate was designed to stop.

The concrete corrupted admission value: `validate_resource_bounds` returns `Ok(())` for a Declare transaction with `l2_gas.max_amount > config.max_l2_gas_amount` (up to `u64::MAX`), when the gateway policy requires `Err(MaxGasAmountTooHigh)`.

**Impact: High — Mempool/gateway admission accepts Declare transactions that violate the gateway's resource bound policy, bypassing the stateless admission gate entirely.**

### Likelihood Explanation

Trivially exploitable. Any user can craft a V3 Declare transaction with an arbitrary `l2_gas.max_amount` field. No privileged access, no special state, no signature bypass required. The bypass is unconditional for all Declare transactions.

### Recommendation

Remove the Declare-type exemption at lines 79–85 and apply the same `max_l2_gas_amount` check uniformly across all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher L2 gas ceiling (e.g., due to compilation overhead), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.
