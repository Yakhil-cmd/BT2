### Title
Floor Division in `native_per_pubdata` Computation Allows Zero-Cost Pubdata Generation - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` value — which controls how many native (proving) resources a user must pay per byte of pubdata — is computed using floor division (`wrapping_div`), while the analogous `native_per_gas` value uses ceiling division (`div_ceil`). When `pubdata_price < native_price` (a realistic production configuration), `native_per_pubdata` truncates to zero. As a result, users pay **zero** native resources for any amount of pubdata they generate, directly analogous to the ERC4626 rounding bug where `convertToShares` returns 0 shares for non-zero assets.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two resource-rate computations are:

```rust
// native_per_gas: uses ceiling division — always ≥ 1 when gas_price > 0
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata: uses floor division — can be 0
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [1](#0-0) 

The same asymmetry is present in `api/src/helpers.rs`:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price  ← floor
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

When `native_per_pubdata == 0`, the downstream function `get_resources_to_charge_for_pubdata` computes:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // = 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

This means `check_enough_resources_for_pubdata` always returns `true` (0 native required ≤ any remaining balance), and the intrinsic pubdata overhead deducted from the native limit is also zero:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // = 0
``` [4](#0-3) 

The condition `pubdata_price < native_price` is not exotic. The existing test suite exercises exactly this scenario (`pubdata_price = 2`, `native_price = 100`): [5](#0-4) 

---

### Impact Explanation

When `native_per_pubdata == 0`, a user can craft transactions that generate the maximum allowed pubdata per block (storage writes, deployments, etc.) and pay **zero** native resources for it. The operator/sequencer bears the full RISC-V proving cost of that pubdata without receiving compensation. This is a resource accounting bug: the protocol's double-resource model is bypassed for pubdata costs, allowing users to underpay for a significant portion of the proving work their transactions impose.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a valid and realistic operator configuration — it occurs whenever the per-byte pubdata cost is less than the native cycle price. The existing test at line 1276–1278 demonstrates this with `pubdata_price = 2, native_price = 100`. Any unprivileged user transacting under such a block context can exploit this by maximizing pubdata-generating operations (SSTORE, contract deployments) within the block's pubdata limit.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — can produce 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — always ≥ 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [6](#0-5) [7](#0-6)

### Proof of Concept

**Setup**: Deploy a block with `pubdata_price = 50`, `native_price = 100`.

**Computation**:
```
native_per_pubdata = floor(50 / 100) = 0
```

**Attacker action**: Send a transaction that performs 20 SSTORE operations (each writing a new slot), generating ~20 × 65 = 1300 bytes of pubdata.

**Result**:
```
native_charged_for_pubdata = 1300 * 0 = 0
```

The attacker pays zero native resources for 1300 bytes of pubdata. `check_enough_resources_for_pubdata` returns `true` unconditionally. The transaction succeeds and the operator proves the pubdata at their own cost.

**Contrast with `native_per_gas`**: Under the same block context, `native_per_gas = ceil(gas_price / 100) ≥ 1`, so EVM computation is always charged. Only pubdata is free due to the floor division asymmetry. [1](#0-0) [8](#0-7)

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
