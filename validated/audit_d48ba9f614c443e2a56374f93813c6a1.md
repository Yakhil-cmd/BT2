### Title
Malicious CKD Participant Can Corrupt Derived Key by Injecting Arbitrary Contributions Without Proof of Private Share Usage - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` aggregates participant-supplied `(big_y, big_c)` values by simple addition with no cryptographic verification that each participant actually used their legitimate private share. Any single malicious participant can substitute arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that does not correspond to the true master secret key applied to the `app_id`. Honest parties will accept this corrupted output as valid.

### Finding Description
The CKD protocol is split into two roles. Each participant (including the coordinator) calls `compute_signature_share`, which correctly computes:

```
big_s  = hash_point * private_share      // x_i · H(pk ‖ app_id)
big_c  = big_s + app_pk * y              // x_i·H + y_i·A
big_y  = G * y                           // y_i·G
norm_big_y = lambda_i * big_y
norm_big_c = lambda_i * big_c
```

The coordinator then collects every participant's `(norm_big_y, norm_big_c)` and sums them:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no zero-knowledge proof, commitment binding, or any other check** that the received `big_c` encodes the participant's actual `private_share`. A malicious participant can send any pair `(big_y', big_c')` — for example, `big_c' = 0` (the identity element) — and the coordinator will silently incorporate it into the aggregate. The final `CKDOutput` will then be:

```
C_corrupt = Σ_{honest i} λ_i·(x_i·H + y_i·A)  +  big_c'_malicious
Y_corrupt = Σ_{honest i} λ_i·y_i·G             +  big_y'_malicious
```

When the application unmasks with `app_sk`: `C_corrupt − app_sk·Y_corrupt ≠ msk·H(pk, app_id)`. The derived confidential key is wrong and the output is accepted by all honest parties as legitimate.

Contrast this with the DKG protocol (`src/dkg.rs`), which enforces a Schnorr proof of knowledge for every participant's polynomial constant term (lines 452–460) and verifies every received secret share against a published commitment (lines 520–522). The CKD protocol has no equivalent safeguard.

### Impact Explanation
A single malicious participant can make the coordinator output a `CKDOutput` that does not encode `msk · H(pk ‖ app_id)`. Every honest party that relies on this output — including the application that calls `ckd_output.unmask(app_sk)` — will obtain a wrong derived key. This is a **corruption of CKD output so honest parties accept an unusable cryptographic output**, matching the defined **High** impact tier.

### Likelihood Explanation
Any participant in a CKD session can exploit this with no special privileges. The attacker only needs to be a legitimate member of the `participants` list passed to `ckd()`. No leaked keys, no external compromise, and no cryptographic break are required — only the ability to send a crafted message over the existing `SharedChannel`.

### Recommendation
Add a non-interactive zero-knowledge proof (e.g., a Schnorr-style DLEQ proof) that binds each participant's `big_c` to their publicly committed verification share. Specifically, each participant should prove in zero knowledge that the discrete-log relationship between `big_s` and `H(pk ‖ app_id)` equals the discrete-log relationship between their public verification share and the group generator. The coordinator must verify all such proofs before aggregating contributions, mirroring the proof-of-knowledge checks already present in `do_keyshare`.

### Proof of Concept
1. Run a CKD session with participants `[P1, P2, P3]` and coordinator `P1`.
2. `P2` (malicious) intercepts the protocol at the `waitpoint` send step in `do_ckd_participant` (line 30) and instead sends `(ElementG1::identity(), ElementG1::identity())` as its `(norm_big_y, norm_big_c)`.
3. `P1` (coordinator) receives the tampered values and sums them with the honest contributions from `P1` and `P3` — no error is raised.
4. The resulting `CKDOutput` satisfies `ckd_output.unmask(app_sk) ≠ msk · H(pk, app_id)`.
5. All honest parties accept this corrupted output as the legitimate derived key. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
