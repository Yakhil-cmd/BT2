### Title
Unprivileged Caller Can Permanently Freeze Consensus Rewards via `disable_rewards=true` in `update_rewards` — (File: `src/staking/staking.cairo`)

---

### Summary

The public `update_rewards` function accepts a caller-controlled `disable_rewards: bool` parameter with no access control. When set to `true`, the function writes the current block number to the global `last_reward_block` slot **before** the early-return, permanently consuming that block's reward opportunity without distributing any yield. Any unprivileged actor can call this every block to freeze all consensus-phase staking rewards.

---

### Finding Description

`update_rewards` is exposed via `#[abi(embed_v0)]` with no role guard — `general_prerequisites()` only asserts the contract is unpaused and the caller is non-zero. [1](#0-0) 

The function writes `last_reward_block` to the current block number unconditionally, **before** the `disable_rewards` branch: [2](#0-1) 

`last_reward_block` is a single global slot. Once written for block `N`, every subsequent call in block `N` reverts: [3](#0-2) 

When `disable_rewards=true` the function returns immediately after the write, distributing nothing: [4](#0-3) 

The structural analog to the Merk bug is exact: in Merk, phantom KV nodes are inserted into the lookup map but excluded from the root hash, so the map records an inclusion that was never committed. Here, a block is inserted into `last_reward_block` (the "processed" map) but excluded from reward distribution — the slot is consumed without the corresponding yield ever being committed to stakers.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

After `consensus_rewards_first_epoch` is activated, every block produces STRK (and BTC) block rewards distributed through `_update_rewards`. An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` every block prevents any legitimate call from succeeding in that block. Because `last_reward_block` is never reset, each wasted slot is permanently lost. Sustained over time this drains the entire consensus reward stream for all stakers and delegators.

---

### Likelihood Explanation

**High.** The attack requires:
- No privileged role.
- Only gas.
- Any valid active staker address (trivially obtained from on-chain `NewStaker` events).

It can be fully automated with a bot that fires one transaction per block. The attacker has no financial incentive to stop.

---

### Recommendation

Remove `disable_rewards` from the public ABI entirely, or gate it behind a role check (e.g., `only_app_governor` or `only_security_agent`). If the parameter is needed for internal pool-removal flows, refactor it into a separate internal function that is not externally callable.

---

### Proof of Concept

1. Consensus rewards are activated (`consensus_rewards_first_epoch` is set).
2. Attacker reads any `staker_address` from a past `NewStaker` event.
3. Each block, attacker submits:
   ```
   staking.update_rewards(staker_address=<valid_staker>, disable_rewards=true)
   ```
4. Inside the call:
   - `general_prerequisites()` passes (contract unpaused, caller non-zero). [5](#0-4) 
   - `current_block_number > last_reward_block` passes (new block). [3](#0-2) 
   - Staker validity checks pass. [6](#0-5) 
   - `last_reward_block` is written to current block. [7](#0-6) 
   - `disable_rewards=true` → early return, zero rewards distributed. [4](#0-3) 
5. Any legitimate `update_rewards` call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeated every block → all consensus yield is permanently frozen.
