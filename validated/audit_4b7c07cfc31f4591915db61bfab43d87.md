### Title
`skip_stateful_validations` Uses Overly Broad Mempool-Presence Check to Proxy for Deploy-Account Existence, Allowing Signature-Validation Bypass for Nonce-1 Invoke Transactions — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function is intended to skip blockifier `__validate__` (signature) checking for a nonce-1 invoke transaction only when a matching `deploy_account` transaction is already pending in the mempool — a UX convenience for simultaneous `deploy_account + invoke` submission. The proxy check it uses, `account_tx_in_pool_or_recent_block`, returns `true` for **any** transaction type in the pool, not exclusively `deploy_account`. Because a deployed account with on-chain nonce 0 can legitimately have a nonce-0 invoke in the mempool, an attacker who observes that account can immediately submit a nonce-1 invoke carrying an **invalid signature** and have it admitted to the mempool without any signature check.

---

### Finding Description

`skip_stateful_validations` fires when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke`.
2. Its nonce equals `Felt::ONE`.
3. The account's on-chain nonce equals `Felt::ZERO`.

When all three hold, the function calls `account_tx_in_pool_or_recent_block(sender_address)`. If that returns `true`, `skip_validate = true` is propagated back to `run_validate_entry_point`, which sets `ExecutionFlags { validate: false, … }` and therefore never calls the blockifier's `StatefulValidator::validate` — the path that invokes the account's `__validate__` entry point and verifies the signature. [1](#0-0) 

The mempool-side implementation of the proxy check is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` is satisfied by **any** transaction type — `Invoke`, `Declare`, or `DeployAccount`. The inline comment claims this is safe because the account "either has a deploy_account transaction or transactions with future nonces that passed validations," but the second branch of that disjunction is circular: those future-nonce transactions may themselves have been admitted via this same skip path. [3](#0-2) 

The concrete broken invariant: a deployed account whose on-chain nonce is 0 (just deployed, never transacted) can have a valid nonce-0 invoke in the mempool. That invoke passed full blockifier validation. An attacker who observes this account's address can then submit a nonce-1 invoke with an **arbitrary, invalid signature**. The gateway's `validate_nonce` check passes because `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap`: [4](#0-3) 

`validate_by_mempool` passes because the mempool sees a valid nonce-0 tx already queued for this account and accepts nonce-1 as a valid future nonce. Then `skip_stateful_validations` returns `true` because `tx_pool.contains_account` is satisfied by the nonce-0 invoke. The blockifier `__validate__` call is never made, and the invalid-signature transaction is forwarded to `mempool.add_tx`.

The test suite confirms the skip fires for any mempool presence, not specifically for a `deploy_account`: [5](#0-4) 

The test parameter `contains_tx = true` is satisfied by an ordinary invoke added to the mock mempool; there is no assertion that the pooled transaction is a `DeployAccount`.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway accepts a nonce-1 invoke with an invalid (or empty) signature into the mempool without running the account's `__validate__` entry point. The transaction will fail when the batcher later attempts execution (blockifier runs `__validate__` unconditionally during block building), but by then it occupies a mempool slot, consumes batcher CPU, and is counted against the account's pending-nonce window. An attacker who monitors the public mempool for accounts with a pending nonce-0 transaction can continuously inject invalid nonce-1 invokes for those accounts, polluting the mempool and degrading sequencer throughput without spending any fees (failed `__validate__` does not charge fees in Starknet).

---

### Likelihood Explanation

**Medium.** The precondition — a deployed account with on-chain nonce 0 that has a pending nonce-0 transaction in the mempool — is a normal, transient state that occurs for every newly deployed account the first time it transacts. Mempool contents are observable via the public RPC. No privileged access is required; the attacker only needs to submit a well-formed `InvokeV3` transaction with a garbage signature to the gateway's `add_transaction` endpoint.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` proxy with a check that specifically queries whether a `DeployAccount` transaction for the given address is present in the mempool. Concretely:

1. Add a new mempool API method, e.g. `deploy_account_tx_in_pool(address) -> bool`, that inspects `tx_pool` for a `DeployAccount` variant at the given address.
2. In `skip_stateful_validations`, call this narrower method instead of `account_tx_in_pool_or_recent_block`.

Alternatively, if the broader check is intentional for performance reasons, add an explicit type guard inside `skip_stateful_validations` before returning `true`:

```rust
// Only skip if the pooled transaction is a DeployAccount.
let is_deploy_account_pending = mempool_client
    .deploy_account_tx_in_pool(tx.sender_address())
    .await?;
return Ok(is_deploy_account_pending);
```

### Proof of Concept

**Setup**: Account `A` is deployed (class hash and code exist on-chain), on-chain nonce = 0.

**Step 1 — Seed the mempool (legitimate action):**
The account owner submits `invoke_v3(sender=A, nonce=0, signature=<valid>)`. The gateway runs full blockifier validation; `__validate__` succeeds; the transaction enters the mempool. Now `tx_pool.contains_account(A) == true`.

**Step 2 — Attacker submits invalid-signature invoke:**
The attacker submits `invoke_v3(sender=A, nonce=1, signature=[0xdead, 0xbeef])` to `POST /add_transaction`.

**Step 3 — Gateway validation trace:**
- `validate_nonce`: `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap` → **passes**.
- `validate_by_mempool`: mempool sees nonce-0 tx queued for `A`; nonce-1 is within gap → **passes**.
- `skip_stateful_validations`: `nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` → returns `true`.
- `run_validate_entry_point(skip_validate=true)`: `ExecutionFlags { validate: false }` → blockifier `__validate__` is **never called**.
- `mempool.add_tx(…)` → **invalid-signature transaction admitted**.

**Step 4 — Batcher execution:**
Batcher pulls both transactions. Executes nonce-0 tx (succeeds). Attempts nonce-1 tx: blockifier calls `__validate__`, signature check fails, transaction is reverted and marked rejected. `commit_block` removes it from the mempool.

**Step 5 — Repeat:**
Attacker repeats Step 2 indefinitely (each time the nonce-0 tx is committed, the account's on-chain nonce becomes 1, so the skip condition no longer fires — but the attacker can target a different newly-deployed account, or wait for the account to reset to nonce 0 via a fresh deploy).

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
