### Title
`PresignOutput` Not Consumed on Rerandomization Enables Presignature Reuse and Private Key Extraction вЂ” (`src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both `robust_ecdsa::RerandomizedPresignOutput::rerandomize_presign` and `ot_based_ecdsa::RerandomizedPresignOutput::rerandomize_presign` accept a shared reference `&PresignOutput` rather than consuming the value. Because `PresignOutput` also derives `Clone` and `Serialize`/`Deserialize`, the same presignature can be rerandomized an arbitrary number of times with different `RerandomizationArguments`. A malicious coordinator can exploit this to run two signing sessions from the same nonce material, recovering the aggregate private key via standard ECDSA nonce-reuse algebra.

---

### Finding Description

The `rerandomize_presign` function signature in both ECDSA variants is:

```rust
// src/ecdsa/robust_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // shared reference вЂ” NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // shared reference вЂ” NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

`PresignOutput` derives `Clone`, `Serialize`, and `Deserialize` in both modules, so callers can trivially clone or store the value before passing it to `rerandomize_presign`, then pass it again with different `RerandomizationArguments` (different `msg_hash`, `tweak`, or `entropy`).

The rerandomization step produces a `RerandomizedPresignOutput` whose secret shares are deterministic linear functions of the original presignature shares and the chosen `delta = HKDF(...)`. Two calls with different `args` yield two `delta` values `Оґв‚Ѓ в‰  Оґв‚‚`, but both are derived from the **same underlying nonce** `R`. The resulting two signatures therefore satisfy the standard ECDSA nonce-reuse relation, from which the aggregate secret key `x` can be recovered algebraically.

The library's own security documentation acknowledges this exact attack:

> "If different subsets of size at least 2t+1 sign different (h, Оµ) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."
> вЂ” `docs/ecdsa/robust_ecdsa/signing.md`

The documentation mandates "Never reuse a presignature", but the type system provides no enforcement: the API accepts `&PresignOutput` instead of `PresignOutput`, so Rust's move semantics cannot prevent reuse.

This is the direct analog of the `LenderCommitmentForwarder#updateCommitment` vulnerability: just as that function overwrites `maxPrincipal` on every update (resetting the "used" counter), `rerandomize_presign` never marks the presignature as consumed, allowing it to be rerandomized and signed against multiple times.

### Impact Explanation

**Critical вЂ” Extraction of the aggregate private signing key.**

Two completed signing sessions using the same `PresignOutput` but different `(msg_hash, tweak)` pairs expose the private key via the standard ECDSA nonce-reuse equation:

```
sв‚Ѓ = kВ·(hв‚Ѓ + rВ·x)   (after rerandomization with Оґв‚Ѓ)
sв‚‚ = kВ·(hв‚‚ + rВ·x)   (after rerandomization with Оґв‚‚)
```

Given `(r, sв‚Ѓ, hв‚Ѓ)` and `(r, sв‚‚, hв‚‚)` with the same effective nonce `k`, `x` is recovered as:

```
x = (sв‚ЃВ·hв‚‚ - sв‚‚В·hв‚Ѓ) / (rВ·(sв‚‚ - sв‚Ѓ))
```

This falls squarely within the allowed Critical impact: **Extraction, reconstruction, or disclosure of private signing shares or aggregate secret material.**

### Likelihood Explanation

A malicious coordinator is an explicitly modeled threat in this library. The coordinator role is required for both OT-based and robust ECDSA signing. A coordinator who has received a `PresignOutput` from honest participants can:

1. Call `rerandomize_presign(presig, args1)` with one `msg_hash`.
2. Call `rerandomize_presign(presig, args2)` with a different `msg_hash`.
3. Run two signing sessions concurrently or sequentially.
4. Recover the private key from the two resulting signatures.

No external assumptions, leaked keys, or cryptographic breaks are required. The attack is entirely within the library's own API surface.

---

### Recommendation

Change `rerandomize_presign` to consume the `PresignOutput` by value, so Rust's move semantics enforce single-use at compile time:

```rust
// Before (both modules):
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// After:
pub fn rerandomize_presign(
    presignature: PresignOutput,   // consumed вЂ” cannot be reused
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Additionally, remove the `Clone` derive from `PresignOutput` in both modules to prevent callers from cloning the value before passing it. If serialization is required for storage, consider a separate "sealed" wrapper type that can be deserialized exactly once (e.g., using a `Cell<Option<PresignOutput>>`).
