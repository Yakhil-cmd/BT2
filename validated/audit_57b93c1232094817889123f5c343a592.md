### Title
Unbounded `apply_to_array_vec` Push into Fixed-Capacity `ArrayVec` Panics on Overflow in Multi-Block Batch Mode - (File: `zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

In the multi-block batch proving path, `LogsStorage::apply_to_array_vec` unconditionally pushes every accumulated L2→L1 log hash into a fixed-capacity `ArrayVec<Bytes32, 16384>`. If the total number of logs across all blocks in a batch exceeds 16,384, the `ArrayVec::push` call panics (capacity overflow), halting the prover. An unprivileged user can trigger this by sending enough L1-messenger messages across multiple blocks in a batch to exceed the per-batch tree capacity.

---

### Finding Description

`LogsStorage::apply_to_array_vec` iterates over every log in the block's `HistoryList` and pushes each one into the caller-supplied `ArrayVec`:

```rust
// zk_ee/src/common_structs/logs_storage.rs:311-315
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.push(log.hash())   // panics if len == 16384
    });
}
```

This function is called from the multi-block batch post-tx-op path:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/post_tx_op_proving_multiblock_batch.rs:109-110
io.logs_storage
    .apply_to_array_vec(&mut batch_data.logs_storage);
```

`batch_data.logs_storage` is declared as `ArrayVec<Bytes32, 16384>` in `ZKBatchDataKeeper`:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs:27
pub logs_storage: ArrayVec<Bytes32, 16384>,
```

The per-block limit `MAX_NUMBER_OF_LOGS = 16_384` is enforced at the transaction level in `check_for_block_limits`:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/mod.rs:84-90
} else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
    Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
```

However, this check is **per-block**, not per-batch. In multi-block batch mode, logs from multiple blocks are accumulated into the single `ArrayVec<Bytes32, 16384>`. If block 1 contributes N₁ logs and block 2 contributes N₂ logs, and N₁ + N₂ > 16,384, the second call to `apply_to_array_vec` will overflow the `ArrayVec` and panic.

`ArrayVec::push` panics (does not return an error) when the array is full. There is no bounds check before the push, and no error propagation path — the panic is unrecoverable.

Logs are pushed into `LogsStorage` via two paths:
1. `emit_l1_message` (user-triggered via the L1 Messenger system hook) — `push_message`
2. `emit_l1_l2_tx_log` (one per L1→L2 priority transaction) — `push_l1_l2_tx_log`

Both are reachable by unprivileged callers.

---

### Impact Explanation

**Proving/forward divergence and valid-execution unprovability.** In the multi-block batch proving mode (`ZKHeaderStructurePostTxOpProvingMultiblockBatch`), the prover panics unconditionally when the accumulated log count across blocks in a batch exceeds 16,384. This:

- Halts the prover process entirely (Rust panic in `no_std`/RISC-V context = abort).
- Makes the batch unprovable, permanently blocking finalization of all blocks in that batch on L1.
- Constitutes a liveness failure: funds deposited via L1→L2 transactions in the affected batch cannot be finalized.

The single-block batch path (`ZKHeaderStructurePostTxOpProvingSingleblockBatch`) calls `io.logs_storage.tree_root()` instead, which allocates a `Vec` dynamically and does not panic, so it is not affected.

---

### Likelihood Explanation

The per-block log limit is 16,384. In a multi-block batch with even 2 blocks, a coordinated attacker needs only to fill each block to ~8,193 logs. Each L1 Messenger call emits one log. At standard EVM gas costs, filling a block with ~8,000 L1 messenger calls is feasible within a single block's gas limit. The attacker pays gas but causes permanent batch-level liveness failure. L1→L2 priority transactions also each emit one log automatically (via `emit_l1_l2_tx_log`), so a spam of priority transactions across two blocks in the same batch is sufficient.

---

### Recommendation

1. **Add a batch-level log count check** before calling `apply_to_array_vec`. Before pushing, verify `array_vec.len() + self.list.len() <= 16384` and return an error (or truncate/reject the block) rather than panicking.
2. **Replace `ArrayVec::push` with `try_push`** and propagate the error gracefully:
   ```rust
   self.list.iter().for_each(|el| {
       let log: L2ToL1Log = el.into();
       array_vec.try_push(log.hash()).expect("batch log limit exceeded");
   });
   ```
   Better: return a `Result` and handle the overflow at the call site.
3. **Enforce a per-batch log limit** in the block loop of the multi-block batch path, rejecting blocks that would push the batch over 16,384 total logs.

---

### Proof of Concept

1. Operator configures a multi-block batch (uses `ZKHeaderStructurePostTxOpProvingMultiblockBatch`).
2. Attacker submits transactions in block 1 that call the L1 Messenger system hook 9,000 times, each emitting one L2→L1 log. Block-level check passes (9,000 < 16,384).
3. Attacker submits transactions in block 2 that call the L1 Messenger system hook 8,000 times. Block-level check passes (8,000 < 16,384).
4. Both blocks are included in the same batch.
5. During batch proving, `post_op` for block 1 calls `apply_to_array_vec`, filling `batch_data.logs_storage` to 9,000 entries.
6. `post_op` for block 2 calls `apply_to_array_vec` again; at entry 7,385 (9,000 + 7,385 = 16,385 > 16,384), `ArrayVec::push` panics.
7. The prover aborts. The batch is never finalized on L1.

        &mut self,
        _ee_type: ExecutionEnvironmentType,
        resources: &mut Self::Resources,
        address: &<Self::IOTypes as SystemIOTypesConfig>::Address,
        data: &[u8],
    ) -> Result<Bytes32, SystemError> {
        // TODO(EVM-1077): consider adding COMPUTATIONAL_PRICE_FOR_PUBDATA as in Era

        // We need to charge cost of hashing:
        // - keccak256_native_cost(L2_TO_L1_LOG_SERIALIZE_SIZE) and
        //   keccak256_native_cost(64) when reconstructing L2ToL1Log
        // - keccak256_native_cost(64) + keccak256_native_cost(data.len())
        //   when reconstructing Messages
        // - at most 1 time keccak256_native_cost(64) when building the
        //   Merkle tree (as merkle tree can contain ~2*N nodes, where the
        //   first N nodes are leaves the hash of which is calculated on the
        //   previous step).

        let hashing_native_cost =
            keccak256_native_cost::<Self::Resources>(L2_TO_L1_LOG_SERIALIZE_SIZE).as_u64()
                + 3 * keccak256_native_cost::<Self::Resources>(64).as_u64()
                + keccak256_native_cost::<Self::Resources>(data.len()).as_u64();

        // We also charge some native resource for storing the log
        let native = hashing_native_cost
            + EVENT_STORAGE_BASE_NATIVE_COST
            + EVENT_DATA_PER_BYTE_COST * (data.len() as u64);

        resources.charge(&R::from_native(R::Native::from_computational(native)))?;

        // TODO(EVM-1078): for Era backward compatibility we may need to add events for l2 to l1 log and l1 message

        // Compute data hash directly: the native cost for this keccak is already
        // pre-charged above (included in `hashing_native_cost`), and this function
        // must not charge ergs — EVM gas accounting is the caller's responsibility
        // (the L1Messenger system contract charges it before invoking the hook).
        use crypto::MiniDigest;
        let data_hash = Bytes32::from_array(crypto::sha3::Keccak256::digest(data));
        let data = UsizeAlignedByteBox::from_slice_in(data, self.allocator.clone());
        self.logs_storage
            .push_message(self.tx_number, address, data, data_hash)?;
        Ok(data_hash)
