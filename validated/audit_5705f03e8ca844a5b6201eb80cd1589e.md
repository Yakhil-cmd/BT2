### Title
Cross-Network and Cross-Context COA Ownership Proof Signature Replay — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof()` verifies Cadence account signatures over an arbitrary `signedData` blob using the fixed domain separation tag `"FLOW-V0.0-user"`. Neither the chain ID nor the EVM address is enforced as part of the signed data. Because the domain tag is identical across all Flow networks (mainnet, testnet, previewnet, etc.), a COA ownership proof signature produced on one network is cryptographically valid on any other Flow network. This is the direct analog of the ERC20Permit missing `chainID` vulnerability: the signed message does not bind to the network on which it was produced.

---

### Finding Description

`EVM.validateCOAOwnershipProof()` in `fvm/evm/stdlib/contract.cdc` (lines 1082–1086) verifies signatures using a hardcoded domain separation tag:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The tag `"FLOW-V0.0-user"` is a static string with no network component. The `signedData` parameter is fully attacker-controlled (arbitrary bytes supplied by the caller). The function explicitly documents that it does not enforce that `signedData` encodes `evmAddress`:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

What the comment does **not** address is cross-network replay: the same domain tag `"FLOW-V0.0-user"` is used on every Flow network. A signature produced on testnet over `signedData = X` is indistinguishable from a signature produced on mainnet over the same `signedData = X`, because the signed message is `"FLOW-V0.0-user" || signedData` with no chain ID component.

The ERC-1271 entry point in `coa.sol` (`isValidSignature`) calls `verifyCOAOwnershipProof` via the Cadence Arch precompile, which in turn calls `validateCOAOwnershipProof`. Any EVM smart contract using ERC-1271 with COA ownership proofs for asset authorization is exposed.

---

### Impact Explanation

An attacker who observes a valid COA ownership proof signature produced by a victim on Flow testnet (e.g., for a testnet dApp login or authorization flow) can replay that exact `(signedData, signatures, keyIndices)` tuple on Flow mainnet. If an EVM smart contract on mainnet uses `isValidSignature` (ERC-1271) backed by `verifyCOAOwnershipProof` to gate asset transfers or privileged actions, the attacker can pass the replayed proof and the contract will return `ValidERC1271Signature` (`0x1626ba7e`), granting unauthorized access to the victim's EVM-side assets or contract permissions.

The same replay applies across application contexts on the same network: a signature produced for one dApp's `signedData` can be replayed in another dApp's `validateCOAOwnershipProof` call if the `signedData` bytes happen to match or are attacker-chosen.

---

### Likelihood Explanation

**Moderate.** The preconditions are:
1. The victim's Cadence account exists on multiple Flow networks with the same key material (common for developers and early adopters who use the same wallet on testnet and mainnet).
2. The victim has signed a COA ownership proof on a non-mainnet network (e.g., for a testnet dApp).
3. An EVM smart contract on mainnet uses ERC-1271 via `isValidSignature` / `verifyCOAOwnershipProof` to authorize asset operations.

All three conditions are realistic. The `coa.sol` ERC-1271 implementation is the deployed COA contract and is the primary attack surface. The attacker needs no privileged access — they only need to observe a proof signature (which is transmitted on-chain or via dApp APIs) and submit an EVM transaction.

---

### Recommendation

**Short term:** Enforce that `signedData` encodes both the `evmAddress` and the Flow chain ID at the protocol level inside `validateCOAOwnershipProof`, rather than relying on callers to do so. The function should hash or prepend `chainID || evmAddress` to the `signedData` before verification, making cross-network and cross-address replay impossible.

**Long term:** Introduce a structured, versioned proof schema (analogous to EIP-712 typed structured data) for COA ownership proofs that mandates inclusion of `chainID`, `evmAddress`, and a nonce or expiry. Document the replay risks explicitly in the public API and enforce them at the protocol layer, not as a caller convention.

---

### Proof of Concept

**Setup:** Alice holds a Cadence account `0xALICE` on both Flow testnet and mainnet, with the same key at index 0. She signs a COA ownership proof on testnet for a testnet dApp with `signedData = bytes("login:testnet-dapp")`.

**Attack:**
1. Attacker Bob observes the proof `(address=0xALICE, path=/public/coa, signedData=bytes("login:testnet-dapp"), keyIndices=[0], signatures=[SIG], evmAddress=ALICE_COA_EVM)` on testnet.
2. Bob submits an EVM transaction on **mainnet** to a smart contract that calls `isValidSignature(hash, encodedProof)` on Alice's mainnet COA address.
3. The COA's `isValidSignature` calls `verifyCOAOwnershipProof(address(this), hash, encodedProof)` via the Cadence Arch precompile.
4. The precompile calls `EVM.validateCOAOwnershipProof(0xALICE, /public/coa, signedData, [0], [SIG], ALICE_COA_EVM)` on mainnet.
5. `keyList.verify(signatureSet, signedData, "FLOW-V0.0-user")` succeeds — the signature is valid because the domain tag and signed bytes are identical on both networks.
6. The smart contract receives `ValidERC1271Signature` and grants Bob's transaction the authorization it sought on Alice's behalf.

**Root cause line:** `fvm/evm/stdlib/contract.cdc:1085` — `domainSeparationTag: "FLOW-V0.0-user"` with no chain ID binding, combined with the absence of any enforcement that `signedData` encodes `evmAddress` or chain ID. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
