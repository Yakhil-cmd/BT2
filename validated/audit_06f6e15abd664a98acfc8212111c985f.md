### Title
Malicious Participant Can Panic Triple Generation via Undersized Received Vectors — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

In `do_generation_many`, the code receives `PolynomialCommitmentsMessageMany` (and other `Vec`-typed messages) from remote participants and directly indexes into those vectors using `[i]` for `i in 0..N` without first verifying that the received vectors contain at least `N` elements. A malicious participant can send a crafted message with fewer than `N` elements, triggering an out-of-bounds index panic that aborts the triple generation protocol for all honest parties.

---

### Finding Description

`do_generation_many<const N: usize>` is the core function for batch triple generation in the OT-based ECDSA scheme. It receives messages from remote participants at three distinct waitpoints and indexes into the received `Vec` fields using the compile-time constant `N` — but never validates that the received vectors actually contain `N` elements.

**Instance 1 — Round 1 commitment receipt (line 180–183):** [1](#0-0) 

```rust
while all_commitments_vec.iter().any(|all_commitments| !all_commitments.full()) {
    let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
    for i in 0..N {
        all_commitments_vec[i].put(from, commitments[i]); // ← panics if commitments.len() < N
    }
}
```

`commitments` is a `Vec<_>` deserialized from a remote participant's message. No length check is performed before `commitments[i]`.

Serde will happily deserialize a message where any of these `Vec` fields has fewer than `N` elements. No length validation is performed after deserialization.

---

### Impact Explanation

The triple generation protocol (`generate_triple_many`) is a prerequisite for OT-based ECDSA presigning and signing. A panic in `do_generation_many` aborts the async task executing the protocol. Because the panic is not wrapped in a `Result`, it cannot be caught by the `?` operator and propagates out of the task, causing the protocol to fail for all honest participants. A single malicious participant can repeatedly trigger this at any of the three waitpoints, permanently denying triple generation — and therefore signing — for all honest parties.

This matches the allowed impact: **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

Any participant in the triple generation session can craft and send a `PolynomialCommitmentsMessageMany` (or a `Vec<Commitment>` / `Vec<SerializableScalar>`) with fewer than `N` elements. Serde deserialization succeeds unconditionally for `Vec` fields of any length. The attack requires no cryptographic capability — only the ability to send a malformed message, which is within the documented threat model (malicious participant).

---

### Recommendation

After receiving any `Vec`-typed message from a remote participant, validate that its length equals `N` before indexing. Return a `ProtocolError` on mismatch instead of panicking:

```rust
if their.big_e_v.len() != N
    || their.big_f_v.len() != N
    || their.big_l_v.len() != N
    || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N
    || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(
        format!("message from {from:?} has wrong vector length, expected {N}")
    ));
}
```

Apply the same guard to `commitments` at the Round 1 waitpoint and to `a_j_i_v`/`b_j_i_v` at the Round 3 waitpoint. Replace all `[i]` indexing on externally received slices with `.get(i).ok_or(...)` to align with the crate's `indexing-slicing = "deny"` clippy policy. [5](#0-4) 

---

### Proof of Concept

1. A set of honest participants initiates `generate_triple_many` with `N = 2`.
2. A malicious participant constructs a `PolynomialCommitmentsMessageMany` where `big_e_v` contains only 1 element instead of 2, and sends it at `wait2`.
3. An honest participant receives and deserializes the message successfully (Serde imposes no length constraint on `Vec`).
4. The honest participant enters `for i in 0..2` and evaluates `their.big_e_v[1]` — out of bounds, causing a panic.
5. The async task running `do_generation_many` is aborted; the triple generation protocol fails for all honest parties.
6. The malicious participant can repeat this at every protocol invocation, permanently blocking OT-based ECDSA signing.
