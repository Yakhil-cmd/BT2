### Title
`MsgTurnBridge` Handler Is a No-Op — Permission Check and Bridge State Update Never Executed - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` gRPC message handler, which is the emergency circuit breaker for the Gravity Bridge, is implemented as a stub that unconditionally returns `nil, nil` without performing any permission check or state mutation. Any unprivileged user can submit a `MsgTurnBridge` transaction and receive a success response, while the bridge state is never actually changed. The admin's ability to disable the bridge in an emergency is permanently broken.

### Finding Description

`MsgTurnBridge` is the on-chain mechanism designed to allow the Cronos admin or a permissioned address (holding `CanTurnBridge`) to disable the Gravity Bridge as an emergency circuit breaker (per ADR-009). The permission constant `CanTurnBridge` is defined, the permission system is wired up, and the CLI command exists — but the server-side handler is a stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly calls `k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` before acting: [2](#0-1) 

The permission bit `CanTurnBridge` is defined alongside `CanChangeTokenMapping` but is never consulted in the handler: [3](#0-2) 

ADR-009 explicitly states that `MsgTurnBridge` must check for the `CanTurnBridge` permission before processing: [4](#0-3) 

### Impact Explanation

Two distinct impacts arise:

1. **Permission bypass (High):** Any unprivileged address can submit `MsgTurnBridge` and receive a `code: 0` success response. The permission guard that should restrict this to the admin or a `CanTurnBridge`-permissioned address is entirely absent. This is a direct bypass of the Cronos admin/permission authorization check.

2. **Permanent inability to disable the bridge (High):** Because the handler never writes any bridge-enabled/disabled state, the Gravity Bridge emergency circuit breaker is permanently non-functional. In an active bridge exploit scenario, the admin cannot halt bridge flows via `MsgTurnBridge`. The only remaining path is a slow governance vote, which cannot stop an in-progress exploit.

Both impacts fall within the allowed High scope: bypass of Cronos admin/permission authorization checks, and permanent inability for honest users/validators to exercise bridge control flows.

### Likelihood Explanation

The entry path is a standard signed Cosmos SDK transaction — no special privileges, no leaked keys, no cryptographic assumptions. Any address with gas can submit `MsgTurnBridge`. The broken state is permanent until a code fix is deployed via a chain upgrade.

### Recommendation

Implement `TurnBridge` to:
1. Verify the sender holds `CanTurnBridge` permission via `k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`, mirroring the pattern in `UpdateTokenMapping`.
2. Persist the bridge enabled/disabled state to the KV store and enforce it in the relevant bridge hooks/handlers.
3. Add a unit test analogous to the existing `TestHasPermissions` that asserts an unpermissioned sender is rejected and that the bridge state is correctly toggled.

### Proof of Concept

```
# Any unprivileged address can call TurnBridge and receive code 0:
cronosd tx cronos turn-bridge false --from <any_address> --gas-prices ... -y

# Expected: code != 0 (unauthorized)
# Actual:   code == 0, bridge state unchanged, circuit breaker silently fails
```

The integration test `test_gravity_turn_bridge` asserts `rsp["code"] != 0` for an unpermissioned sender and `rsp["code"] == 0` for the validator — but because the handler is a stub, the first assertion would also pass with `code == 0` (both succeed), and the bridge would never actually be disabled after the second call.
