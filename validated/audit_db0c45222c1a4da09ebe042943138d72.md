### Title
Zero Native Resource Charge for All EIP-2537 BLS12-381 Precompiles Allows Unbounded Proving-Cost Consumption - (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

Every EIP-2537 BLS12-381 precompile in ZKsync OS hardcodes `cost_native = 0`, charging no native (RISC-V proving-cycle) resource for operations that are among the most computationally expensive in the EVM. Because the native resource is the mechanism that bounds the actual off-chain proving cost per transaction, an unprivileged caller can craft a transaction that stays within the EVM gas limit while consuming an unbounded amount of RISC-V proving cycles, breaking the double-resource-accounting invariant that the system relies on.

---

### Finding Description

ZKsync OS implements a **double resource accounting** model: every operation charges both EVM gas (ergs) and a "native" resource that models RISC-V proving cycles. The native limit per transaction is derived from `gasLimit × nativePerGas`, and if a transaction exhausts its native budget the entire transaction is reverted.

All seven EIP-2537 BLS12-381 precompile implementations set `cost_native = 0`:

**`pairing.rs`** — BLS12-381 pairing check: [1](#0-0) 

**`addition.rs`** — G1 and G2 point addition: [2](#0-1) 

**`msm.rs`** — G1 and G2 multi-scalar multiplication: [3](#0-2) 

**`mappings.rs`** — field-to-curve mappings: [4](#0-3) 

Compare this with the analogous BN254 pairing precompile, which correctly charges a large native cost proportional to the number of pairs: [5](#0-4) 

The BN254 native constants are: [6](#0-5) 

`BN254_PAIRING_BASE_NATIVE_COST` expands to `native_with_delegations!(13_000_000, 500_000, 0)` — roughly 13 million native units per pair. BLS12-381 operates over a 381-bit field (vs BN254's 254-bit field) and is computationally heavier, yet charges **zero** native.

The EVM gas for BLS12-381 pairing is 37,700 (fixed) + 32,600 per pair. With a 30 M gas transaction limit, a caller can submit ≈ 920 pairs. At the BN254-equivalent native cost, the correct charge would be ≈ 920 × 13 M ≈ **12 billion native units** — far exceeding any per-transaction native budget — yet the actual charge is 0.

The native resource documentation confirms the invariant that is broken:

> "If a transaction execution runs out of native resources, the entire transaction is reverted." [7](#0-6) 

Because native is never charged for BLS12-381 operations, the transaction never runs out of native, and the prover is forced to execute an unbounded amount of work that was not paid for.

---

### Impact Explanation

**Resource accounting bug / valid-execution unprovability path.**

1. A caller submits a transaction calling the BLS12-381 pairing precompile (address `0x0f`) with the maximum number of pairs that fit within the EVM gas limit.
2. EVM gas is correctly deducted; native resource is not touched (charged 0).
3. The transaction succeeds from the sequencer's perspective.
4. The prover must execute the full BLS12-381 Miller loop and final exponentiation for every pair — work that is orders of magnitude more expensive than what the native budget covers.
5. If the block contains multiple such transactions, the aggregate RISC-V cycle count can exceed the block's native computational limit, making the block unprovable or causing the prover to stall.
6. The same applies to G1/G2 MSM with large point sets, and to the field-to-curve mappings.

---

### Likelihood Explanation

The BLS12-381 precompiles are live EIP-2537 Prague precompiles reachable by any unprivileged EVM transaction. No special role, governance action, or oracle manipulation is required. The attacker only needs to send a standard EVM transaction calling address `0x0f` (BLS12_PAIRING_CHECK) with a large input. The cost to the attacker is only EVM gas; the cost to the prover is unbounded native computation.

---

### Recommendation

Assign non-zero native costs to all EIP-2537 precompiles, benchmarked against actual RISC-V cycle counts, following the same pattern used for BN254 operations in `basic_system/src/cost_constants.rs`. At minimum, the BLS12-381 pairing precompile should charge a per-pair native cost at least as large as `BN254_PAIRING_PER_PAIR_NATIVE_COST` (and likely larger given the wider field). Add corresponding constants to `cost_constants.rs` and wire them into each EIP-2537 precompile's `cost_native` computation.

---

### Proof of Concept

```
Attacker constructs a transaction:
  to:    0x000000000000000000000000000000000000000f  (BLS12_PAIRING_CHECK)
  gas:   30_000_000
  input: 920 valid (G1, G2) pairs  (920 × (128 + 256) = 349,440 bytes)

EVM gas charged:  37,700 + 920 × 32,600 = 37,700 + 29,992,000 = 30,029,700  ≈ 30 M  ✓ (fits)
Native charged:   0  (cost_native = 0 in pairing.rs line 36)

Actual RISC-V work per pair ≈ BN254 equivalent ≈ 13,000,000 native units
Total actual native work:    920 × 13,000,000 ≈ 12,000,000,000 native units

Per-transaction native budget (typical): MAX_NATIVE_COMPUTATIONAL << 12 billion
→ Prover must execute ~12 billion native units of work for a transaction
  that paid for 0 native units, breaking the resource accounting invariant.
```

The attacker-controlled entry path is: unprivileged EVM `CALL` → `BLS12_PAIRING_CHECK` hook → `Bls12381PairingCheckPrecompile::execute` → `cost_native = 0` charge → `<Bls12_381 as Pairing>::multi_pairing` with 920 pairs. [8](#0-7)
