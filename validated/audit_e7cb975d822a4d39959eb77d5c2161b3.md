Audit Report

## Title
Stale `checked_utxos` Cache Bypasses Re-Querying Bitcoin Checker After Transient Mint Failure — (File: `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary

When `mint_ckbtc` returns a transient error (not a panic), the UTXO remains in `checked_utxos` with `UtxoCheckStatus::Clean` and the scopeguard is defused, leaving no `CleanButMintUnknown` marker. On the next `update_balance` call, `check_utxo` finds the cached `Clean` entry and returns immediately without calling the Bitcoin checker canister. If the Bitcoin checker canister was upgraded between the failed mint and the retry — adding the depositor's address to the OFAC SDN list — ckBTC is minted for a now-sanctioned UTXO using stale compliance state.

## Finding Description

**Root cause — unconditional cache short-circuit in `check_utxo`:**

```rust
// rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs L397-399
if let Some(checked_utxo) = read_state(|s| s.checked_utxos.get(utxo).cloned()) {
    return Ok(checked_utxo.status);
}
```

`mark_utxo_checked` is called with `UtxoCheckStatus::Clean` **before** `mint_ckbtc` is attempted (L306-310). The scopeguard installed at L333-337 only fires on a **panic**; in the normal `Err` branch (L365-373) the guard is defused at L378, so the status stays `Clean` — not `CleanButMintUnknown`. The UTXO is not added to `minted_outpoints` or `utxos_state_addresses` because `add_utxos` is never called.

**Retry path:**

`processable_utxos_for_account` (L1360-1410) includes the UTXO again because `is_known` checks only `utxos_state_addresses` and `finalized_utxos` (L1368-1378), neither of which contains the UTXO. The `minted_outpoints` partition at L189-193 also passes. `check_utxo` is called, hits the `checked_utxos` cache, and returns `Ok(UtxoCheckStatus::Clean)` without an inter-canister call to the Bitcoin checker. Minting proceeds.

**Persistence across upgrades:**

`checked_utxos` is a field of `CkBtcMinterState` (L554) and is replayed from the event log on upgrade; it is not cleared. A `CheckedUtxoV2` event is recorded by `mark_utxo_checked` (audit.rs L123-131) and replayed at eventlog.rs L567-569, so the stale `Clean` entry survives a minter upgrade.

## Impact Explanation

The Bitcoin checker canister is the sole enforcement point for OFAC SDN compliance in the ckBTC deposit path. Bypassing it allows minting ckBTC backed by BTC from a sanctioned source. This constitutes a concrete, demonstrable compliance failure in the ckBTC chain-fusion system — a significant ck-token security impact with direct protocol harm. This maps to the **High** impact tier: "Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."

## Likelihood Explanation

Two independent events must overlap: (1) a transient `mint_ckbtc` failure (ledger temporarily unavailable — a realistic operational condition documented by the `UtxoStatus::Checked` return path and its comment "The caller should retry the update_balance call"); and (2) a Bitcoin checker upgrade adding the depositor's address to the OFAC list within that window. NNS upgrade proposals are public and have a multi-day voting period, making the timing observable. A sophisticated actor aware of an impending sanctioning proposal can deposit BTC and call `update_balance` speculatively before the proposal executes, then retry after. The transient failure is not attacker-controlled, making this opportunistic rather than deterministic. Likelihood is **low** but the conditions are individually plausible and documented operational events.

## Recommendation

1. **Re-check on retry**: In `check_utxo`, only return the cached result if the UTXO's outpoint is already in `minted_outpoints`. UTXOs that are cached as `Clean` but not yet minted should always be re-queried against the Bitcoin checker canister.
2. **Use `CleanButMintUnknown` for transient failures**: In the `Err` branch of `mint_ckbtc` (L365-373), call `mark_utxo_checked_mint_unknown` instead of defusing the guard, so the UTXO is treated as requiring manual review rather than silently retried with a stale cache hit.
3. **Cache invalidation on checker upgrade**: Track the Bitcoin checker canister version in minter state; on version change, clear or invalidate `checked_utxos` entries whose outpoints are not yet in `minted_outpoints`.

## Proof of Concept

1. Initialize a test minter with a mock Bitcoin checker and a mock ledger.
2. Deposit a UTXO from address `A` (not on the OFAC list). Call `update_balance`. The mock checker returns `CheckTransactionResponse::Passed`. `mark_utxo_checked` is called; `checked_utxos` now contains the UTXO with `Clean` status.
3. Configure the mock ledger to return a transient error on `mint`. Confirm `update_balance` returns `Ok(vec![UtxoStatus::Checked(utxo)])`. Confirm the UTXO is absent from `minted_outpoints` and `utxos_state_addresses`. Confirm the UTXO is present in `checked_utxos` with `Clean` status.
4. Upgrade the mock Bitcoin checker to return `CheckTransactionResponse::Failed` for address `A` (simulating an OFAC list update).
5. Restore the mock ledger to succeed. Call `update_balance` again. Assert that the mock checker's `check_transaction` is **not** called (cache hit). Assert `update_balance` returns `Ok(vec![UtxoStatus::Minted{…}])`. Assert the UTXO is now in `minted_outpoints`.
6. This demonstrates ckBTC minted for a UTXO that would have been rejected by the updated Bitcoin checker, with zero calls to the checker on the retry path.
