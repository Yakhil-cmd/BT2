### Title
Presignature One-Time-Use Invariant Unenforced by API — `rerandomize_presign` Accepts `&PresignOutput` by Reference, Enabling Nonce Reuse and Private Key Extraction - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

The `rerandomize_presign` function in `src/ecdsa/ot_based_ecdsa/mod.rs` accepts `presignature: &PresignOutput` by shared reference rather than consuming it by value. Additionally, `PresignOutput` derives `Clone`. Together, these API choices mean the library provides no enforcement of the documented one-time-use invariant for presignatures. A malicious coordinator can initiate two signing sessions referencing the same presignature, causing participants to produce signature shares with multiplicatively related nonces. The coordinator can then recover the aggregate private key using standard ECDSA nonce-reuse techniques.

---

### Finding Description

**Root cause — reference-taking API and `Clone` derive:**

`PresignOutput` is declared with `#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]`: [1](#0-0) 

`rerandomize_presign` takes the presignature by shared reference: [2](#0-1) 

Because the function signature is `presignature: &PresignOutput`, Rust's ownership system does **not** consume the presignature. The caller retains full ownership and can call `rerandomize_presign` again with the same `PresignOutput` for a different `(h, ε, ρ)` tuple. The `Clone` derive makes this even more explicit — any caller can `.clone()` the output before passing it anywhere.

**Why this is catastrophic — multiplicatively related nonces:**

Each call to `rerandomize_presign` with the same `PresignOutput` but different `RerandomizationArguments` produces a different `delta` via HKDF, yielding:

- Session 1: effective nonce = `k · δ₁⁻¹`, effective R = `R · δ₁`
- Session 2: effective nonce = `k · δ₂⁻¹`, effective R = `R · δ₂`

The two nonces are multiplicatively related by the known ratio `δ₂ / δ₁`. The library's own security documentation confirms this leads directly to key extraction: [3](#0-2) 

The same concern applies to the OT-based scheme, which shares the same `rerandomize_presign` API and the same nonce structure.

**Attack path (malicious coordinator):**

1. Participants complete presigning; each holds a `PresignOutput` containing secret nonce share `k_i`.
2. The malicious coordinator initiates **signing session A** with message hash `h₁` and tweak `ε₁`. Each participant calls `rerandomize_presign(&presign_output, args_A)` — the presignature is not consumed.
3. The coordinator immediately initiates **signing session B** with `h₂`, `ε₂` referencing the **same presignature**. Each participant calls `rerandomize_presign(&presign_output, args_B)` again.
4. The coordinator collects `s_i` shares from both sessions. With two signatures sharing multiplicatively related nonces and known `δ₁`, `δ₂`, the coordinator solves for the aggregate secret key `x` using standard linear algebra over the scalar field.

The coordinator role is reachable without privileged assumptions — any participant can act as coordinator, and the library's protocol does not authenticate or rate-limit coordinator signing requests.

**FROST analog:**

The same structural issue exists in `src/frost/mod.rs`. `PresignOutput<C>` derives `Clone` and contains `nonces: SigningNonces<C>` — the secret one-time nonces required by FROST (RFC 9591). Cloning and reusing this output across two signing sessions with different messages directly leaks the participant's private signing share via the standard FROST nonce-reuse attack. [4](#0-3) 

---

### Impact Explanation

**Critical — Extraction of private signing shares / aggregate secret key.**

Presignature reuse with a known nonce relationship allows the coordinator to recover the aggregate ECDSA private key from as few as two signing sessions. This falls squarely within:

> *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

The library's own documentation acknowledges the severity: [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

**High.** The attack requires only a malicious coordinator — a role reachable by any participant without privileged access. The API actively facilitates the attack: `rerandomize_presign` takes a shared reference, so participants have no Rust-level signal that the presignature has been consumed. In any system where the coordinator drives signing requests (the common deployment model), a single malicious coordinator can silently trigger two sessions against the same presignature before any participant detects the reuse.

---

### Recommendation

1. **Consume `PresignOutput` by value** in `rerandomize_presign` (change `presignature: &PresignOutput` to `presignature: PresignOutput`). Rust's move semantics then statically prevent reuse without an explicit `.clone()`.
2. **Remove or gate the `Clone` derive** on `PresignOutput` (both OT-based and FROST variants). If serialization round-trips require it, use a newtype wrapper that does not implement `Clone`, or add a `#[deprecated]` marker with a safety note.
3. **Add a consumed/invalidated flag** at the library level (analogous to the NFT fix of tracking `privateSaleInvalidated`) so that any attempt to rerandomize an already-used presignature returns an error.
4. For FROST, align with RFC 9591 §5.4 which mandates that `SigningNonces` are deleted immediately after use and must never be serialized for reuse.

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
