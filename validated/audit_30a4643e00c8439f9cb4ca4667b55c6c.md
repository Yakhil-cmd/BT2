### Title
Sensitive Secret Material Exposed via `Debug` Trait on `KeygenOutput` and Presign Output Types — (File: `src/lib.rs`, `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Multiple production types holding private signing shares and presign nonce secrets unconditionally derive `Debug`, causing their secret scalar contents to be printed in plaintext whenever a caller formats them with `{:?}`. This is the Rust-library analog of the bash-history issue: instead of a CLI argument being recorded in `.bash_history`, the secret material is emitted into logs, panic messages, test output, or any diagnostic sink that formats these types.

---

---

### Impact Explanation

- **`KeygenOutput.private_share`**: Leaking this scalar directly discloses the participant's secret signing share. An adversary who collects `t+1` such shares (e.g., from log aggregation across nodes) can reconstruct the aggregate private key and produce arbitrary valid signatures — **Critical: disclosure of private signing shares**.
- **Presign nonce secrets (`k`, `sigma`, `c`, `e`, `alpha`, `beta`)**: Leaking a participant's nonce share alongside a completed signature allows an attacker to solve for the private signing share algebraically, achieving the same outcome — **Critical: disclosure of nonce material enabling key extraction**.

---

### Likelihood Explanation

Likelihood is **Medium**. The `Debug` derive is unconditional and inherited by all downstream integrators. Common triggering scenarios include:

1. A caller wraps a protocol step in `unwrap()` or `expect()` — a panic will print the `Debug` representation of the value, including all secret scalars.
2. A logging or tracing framework (e.g., `tracing`, `log`) is configured to log at `DEBUG` level and a developer adds `tracing::debug!("{:?}", keygen_output)` during integration.
3. Test harnesses that print values on assertion failure will emit the full secret.

None of these require attacker control of the library itself; they are normal development and operational patterns.

---

### Recommendation

1. **Remove `Debug` from all types containing secret scalars.** Replace with a redacting implementation that prints only non-secret fields:

```rust
impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        f.debug_struct("KeygenOutput")
            .field("public_key", &self.public_key)
            .field("private_share", &"[REDACTED]")
            .finish()
    }
}
```

Apply the same pattern to `PresignOutput` and `RerandomizedPresignOutput` in both ECDSA modules, redacting all secret scalar fields (`c`, `e`, `alpha`, `beta`, `k`, `sigma`).

2. **Add a compile-time or lint guard** (e.g., a custom `clippy` lint or `deny(clippy::derive_debug_with_secret)`) to prevent future types holding `SigningShare` or `Scalar` from accidentally deriving `Debug`.

3. **Audit `ZeroizeOnDrop` coverage**: note that `public_key` is already marked `#[zeroize(skip)]` in `KeygenOutput`, confirming the authors are aware of zeroization hygiene — the same care should be applied to `Debug` output.

---

### Proof of Concept

```rust
use threshold_signatures::{KeygenOutput, keygen};
// After running DKG...
let output: KeygenOutput<_> = /* result of keygen protocol */;

// Any of the following will print the raw private signing share scalar:
println!("{:?}", output);
// or in a panic:
let _ = some_result.expect(&format!("failed with output {:?}", output));
// or via tracing:
tracing::debug!(keygen_output = ?output, "DKG complete");
```

The `Debug` output will contain the hex-encoded private scalar of `private_share`, directly readable from stdout, log files, or crash reports — exactly mirroring how bash history exposes CLI-passed private keys.
