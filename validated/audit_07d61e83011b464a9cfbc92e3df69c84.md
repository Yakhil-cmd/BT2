### Title
Unvalidated `app_pk` Identity Element Exposes Confidential Derived Key in CKD Protocol - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` entry point and `compute_signature_share()` accept the application's ElGamal public key `app_pk` without validating that it is a non-identity point on G1. A malicious requester that submits `app_pk = G1::identity()` causes every MPC node to compute `C_i = x_i · H(pk, app_id)` with the blinding term `y_i · A` reduced to zero, so the aggregated ciphertext `C` equals the raw BLS signature `msk · H(pk, app_id)` — the confidential derived secret — in the clear.

### Finding Description

The CKD ElGamal encryption step is:

```
C_i = S_i + y_i · A     where A = app_pk
```

When `A = G1::identity()` (the point at infinity), `y_i · A = identity`, so:

```
C_i = x_i · H(pk, app_id)
```

After Lagrange aggregation by the coordinator:

```
C = Σ λ_i · C_i = msk · H(pk, app_id)
```

The `CKDOutput { big_y, big_c }` is published on-chain. Normally `big_c` is computationally hiding because it contains the random mask `a · Y`. With `A = identity`, `big_c` is the raw BLS signature and is readable by any observer.

The `ckd()` initialisation function performs four checks (participant count, duplicates, self-presence, coordinator-presence) but performs **no check** on `app_pk`:

```rust

`compute_signature_share()` then uses `app_pk` directly in the encryption step:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;
``` [2](#0-1) 

The `verify_signature()` helper in `ciphersuite.rs` does check for identity elements, but it is only called by the *app* to verify the output after the fact — it is never invoked by the MPC nodes during protocol execution. [3](#0-2) 

### Impact Explanation

With `app_pk = G1::identity()`, the `CKDOutput.big_c` field published on-chain equals `msk · H(pk, app_id)` — the confidential derived secret — without any encryption. Any party observing the on-chain response can derive `s = HKDF(msk · H(pk, app_id))` without possessing the application's private scalar `a`. This constitutes full disclosure of the confidential derived secret, matching the Critical impact tier: *"Extraction, reconstruction, or disclosure of … confidential derived secrets."* [4](#0-3) 

### Likelihood Explanation

The `app_pk` value originates from the requesting application and is routed through the MPC contract to the MPC nodes. The MPC contract is explicitly documented as not performing attestation verification itself — that responsibility is delegated to the Developer contract. A malicious or compromised app can submit `A = G1::identity()` (the all-zero compressed encoding) as its public key. The MPC nodes accept it unconditionally and proceed, producing a `CKDOutput` that leaks the secret. [5](#0-4) 

### Recommendation

Add an explicit identity-point and subgroup check on `app_pk` inside `ckd()` before the protocol is started, mirroring the checks already present in `verify_signature()`:

```rust
// Reject the identity element and points not in the prime-order subgroup
let affine: blstrs::G1Affine = app_pk.into();
if (!affine.is_on_curve() | !affine.is_torsion_free() | affine.is_identity()).into() {
    return Err(InitializationError::BadParameters(
        "app_pk must be a non-identity point in the G1 prime-order subgroup".to_string(),
    ));
}
```

This mirrors the guard already applied to the signature and verifying key in `verify_signature()`. [6](#0-5) 

### Proof of Concept

1. App generates `app_sk = Scalar::ZERO` → `app_pk = G1Projective::identity()`.
2. App submits `app_pk` to the Developer contract, which forwards it to the MPC contract.
3. Each MPC node calls `ckd(..., app_pk = identity, ...)`.
4. Inside `compute_signature_share()`: `big_c = big_s + identity * y = big_s = x_i · H(pk, app_id)`.
5. Coordinator aggregates: `norm_big_c = Σ λ_i · x_i · H(pk, app_id) = msk · H(pk, app_id)`.
6. `CKDOutput { big_y: Y, big_c: msk·H(pk,app_id) }` is published on-chain.
7. Any observer reads `big_c` directly as the confidential derived key material; `s = HKDF(big_c)` is now public. [7](#0-6)

    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
