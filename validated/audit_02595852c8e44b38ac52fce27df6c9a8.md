### Title
`factory_set_wnear_address` updates Engine's wNEAR reference but cannot propagate to existing XCC Router `wnear_account`, causing DoS for XCC calls with attached NEAR - (`etc/xcc-router/src/lib.rs`)

---

### Summary

The Aurora Engine stores the wNEAR ERC-20 address in its own trie storage (updatable by the owner via `factory_set_wnear_address`), while each deployed XCC Router contract independently stores the corresponding wNEAR NEP-141 account ID (`wnear_account`) in its own persistent NEAR contract state. This `wnear_account` is set once at router initialization and is never automatically updated when the Engine's wNEAR address changes. When `factory_set_wnear_address` is called with a new wNEAR ERC-20 address that maps to a different NEP-141 contract, all existing routers remain registered only with the old wNEAR NEP-141 contract. Any subsequent XCC call requiring wNEAR unwrapping (i.e., any call with attached NEAR) will fail because the router is not registered with the new wNEAR NEP-141 contract, permanently breaking XCC functionality for all existing router sub-accounts.

---

### Finding Description

**Two independent stores of the same wNEAR reference:**

The Engine stores the wNEAR ERC-20 address in its own trie under `WNEAR_KEY`:

```rust
// engine/src/xcc.rs
pub fn set_wnear_address<I: IO>(io: &mut I, address: &Address) {
    let key = storage::bytes_to_key(KeyPrefix::CrossContractCall, WNEAR_KEY);
    io.write_storage(&key, address.as_bytes());
}
```

This is updated by the owner via `factory_set_wnear_address`:

```rust
// engine/src/contract_methods/xcc.rs
pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        ...
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));
        Ok(())
    })
}
```

Each XCC Router independently stores the wNEAR NEP-141 account ID in its own NEAR contract state:

```rust
// etc/xcc-router/src/lib.rs
pub struct Router {
    ...
    wnear_account: AccountId,  // set at initialize(), never updated automatically
}
```

**Router initialization registers with the wNEAR contract only once:**

When a router is first created (`create_needed = true`), `must_register: true` is passed, causing the router to call `storage_deposit` on the wNEAR NEP-141 contract:

```rust
// etc/xcc-router/src/lib.rs
if must_register {
    env::promise_create(
        wnear_account.clone(),
        "storage_deposit",
        b"{}",
        WNEAR_REGISTER_AMOUNT,
        WNEAR_REGISTER_GAS,
    );
}
```

**Upgrades do not re-register with the new wNEAR contract:**

When a router is upgraded (code version outdated), `create_needed = false` is passed, so `must_register: false`:

```rust
// engine/src/xcc.rs (handle_precompile_promise)
let init_args = format!(
    r#"{{"wnear_account": "{}", "must_register": {}}}"#,
    wnear_account.0.as_ref(),
    create_needed,  // false for upgrades
);
```

This means that even after `factory_update` triggers a router upgrade, the router is **not** registered with the new wNEAR NEP-141 contract.

**The broken execution path:**

When an XCC call with attached NEAR is made, the Engine calls `withdraw_wnear_to_router`, which uses the Engine's current (new) wNEAR ERC-20 address:

```rust
// engine/src/contract_methods/xcc.rs
let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
let (result, ids) = xcc::withdraw_wnear_to_router(&recipient, args.amount, wnear_address, ...)?;
```

This calls `withdrawToNear` on the new wNEAR ERC-20 contract, which triggers an `ft_transfer` from the new wNEAR NEP-141 contract to the router. Since the router is not registered with the new NEP-141 contract, the `ft_transfer` fails, causing the entire XCC call to revert.

---

### Impact Explanation

When `factory_set_wnear_address` is called with a new wNEAR ERC-20 address that maps to a different NEP-141 contract (e.g., due to a wNEAR contract migration or replacement):

1. All existing deployed router sub-accounts remain registered only with the old wNEAR NEP-141 contract.
2. Any XCC call with attached NEAR from an existing router will fail at the `ft_transfer` step because the router has no storage registration with the new wNEAR NEP-141 contract.
3. There is no automatic mechanism to re-register existing routers with the new wNEAR contract. Even calling `factory_update` (which triggers router code upgrades) does not fix this, because upgrades pass `must_register: false`.
4. The only remediation is for the operator to manually call `storage_deposit` on the new wNEAR contract for every existing router sub-account address — an operationally complex and error-prone process with no on-chain enumeration of all router addresses.

This constitutes a **High — Temporary (potentially permanent) freezing of XCC functionality** for all existing router sub-accounts. Users' wNEAR ERC-20 balances in the EVM cannot be used to fund cross-contract calls, breaking a core protocol feature.

---

### Likelihood Explanation

`factory_set_wnear_address` is a legitimate, documented owner-callable function designed to be used when the wNEAR address needs to change. Realistic triggers include:

- Migration of the wNEAR contract to a new address (e.g., a security upgrade or contract replacement).
- Deployment of a new wNEAR ERC-20 bridge token for a different NEP-141 wNEAR contract.
- Operator error setting a different wNEAR address.

The function is explicitly exposed as a public entrypoint in `engine/src/lib.rs` and is exercised in integration tests. Any legitimate use of this function with a new NEP-141 contract triggers the inconsistency.

---

### Recommendation

When `factory_set_wnear_address` is called with a new wNEAR address that maps to a different NEP-141 account, the Engine should either:

1. **Force a router code version bump** (equivalent to calling `factory_update`) so that all existing routers are lazily upgraded on next use, AND pass `must_register: true` during upgrades when the `wnear_account` has changed (not just when `create_needed` is true).
2. **Remove `wnear_account` from the Router's persistent state** entirely and have the Engine pass the current wNEAR NEP-141 account ID at call time (analogous to the recommended fix in the original report: remove the duplicated reference from the dependent component).
3. At minimum, document that `factory_set_wnear_address` must always be paired with `factory_update` and a manual re-registration sweep of all router sub-accounts.

---

### Proof of Concept

```
1. Deploy Aurora Engine and set up wNEAR (wrap.near) → ERC-20 bridge.
2. Call factory_set_wnear_address(wnear_erc20_v1.address).
3. User Alice makes an XCC call with attached NEAR → router is created,
   registered with wrap.near via storage_deposit (must_register: true).
4. Admin deploys a new wNEAR contract (wrap_v2.near) and a new ERC-20 bridge.
5. Admin calls factory_set_wnear_address(wnear_erc20_v2.address).
   → Engine's WNEAR_KEY now points to wnear_erc20_v2.
   → Alice's router still has wnear_account = "wrap.near" and is registered
     only with wrap.near, NOT with wrap_v2.near.
6. Alice makes another XCC call with attached NEAR:
   → Engine calls withdraw_wnear_to_router using wnear_erc20_v2.
   → wnear_erc20_v2 calls ft_transfer on wrap_v2.near to Alice's router.
   → wrap_v2.near rejects: Alice's router has no storage registration.
   → XCC call fails. Alice's wNEAR_v2 ERC-20 tokens cannot fund XCC calls.
7. Admin calls factory_update (new router bytecode) to trigger upgrades.
8. Alice makes another XCC call → router is upgraded, initialize called with
   wnear_account = "wrap_v2.near", must_register = false.
   → Router's wnear_account is updated but storage_deposit is NOT called.
   → ft_transfer from wrap_v2.near to Alice's router still fails.
   → DoS persists even after upgrade.

    #[init(ignore_state)]
    #[must_use]
    pub fn initialize(wnear_account: AccountId, must_register: bool) -> Self {
        // The first time this function is called there is no state and the parent is set to be
        // the predecessor account id. In subsequent calls, only the original parent is allowed to
        // call this function. The idea is that the Create, Deploy and Initialize actions are done in a single
        // NEAR batch when a new router is deployed by the engine, so the caller will be the Aurora
        // engine instance that the user's address belongs to. If we update this contract and deploy
        // a new version of it, again the Deploy and Initialize actions will be done in a single batch
        // by the engine.
        let caller = env::predecessor_account_id();
        let mut parent = LazyOption::new(StorageKey::Parent, None);
        match parent.get() {
            None => {
                parent.set(&caller);
            }
            Some(parent) => {
                // Allow self-calls to `initialize` also.
                // This happens during the upgrade flow.
                if (caller != parent) && (caller != env::current_account_id()) {
                    env::panic_str(ERR_ILLEGAL_CALLER);
                }
            }
        }

        if must_register {
            env::promise_create(
                wnear_account.clone(),
                "storage_deposit",
                b"{}",
                WNEAR_REGISTER_AMOUNT,
                WNEAR_REGISTER_GAS,
            );
        }

        let mut version = LazyOption::new(StorageKey::Version, None);
        if version.get().unwrap_or_default() != CURRENT_VERSION {
            // Future migrations would go here

            version.set(&CURRENT_VERSION);
        }

        let nonce = LazyOption::new(StorageKey::Nonce, None);
        let scheduled_promises = LookupMap::new(StorageKey::Map);
        Self {
            parent,
            version,
            nonce,
            scheduled_promises,
            wnear_account,
        }
    }
