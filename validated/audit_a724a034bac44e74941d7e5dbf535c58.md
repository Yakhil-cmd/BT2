### Title
Echo broadcast provides zero Byzantine fault tolerance for n≤3, enabling commitment equivocation that corrupts DKG public keys — (`src/dkg.rs`, `src/protocol/echo_broadcast.rs`)

### Summary
A malicious participant in a DKG session with n=2 or n=3 total participants can send different `commitment_hash` values to different honest parties in round 2.9 (which uses unprotected `send_many`) and simultaneously equivocate in the round 3 echo broadcast (which degenerates to zero-threshold delivery for n≤3), causing each honest party to pass all local verification checks while deriving a different master public key.

### Finding Description
`assert_key_invariants` permits DKG sessions with as few as 2 participants. For n≤3, `echo_ready_thresholds` returns `(echo_t=0, ready_t=0)`. With these thresholds, the delivery condition in `reliable_broadcast_receive_all` is `count > 0`, which is satisfied immediately by each party's own simulated self-echo/self-ready, before any cross-party echoes arrive. Once `finish_echo = true` is set, subsequent echoes carrying a different value are silently discarded. This allows a malicious sender to deliver different values to different honest parties through the echo broadcast.

Separately, round 2.9 sends `commitment_hash` via plain `send_many` with no consistency guarantee. A malicious party can replace this with targeted `send_private` calls to deliver different hashes to different honest parties.

The combination is lethal: M crafts two distinct commitments, computes matching hashes for each, delivers each hash to the corresponding honest party in round 2.9, and delivers each commitment to the corresponding honest party via the degenerate echo broadcast in round 3. Each honest party's round 4 check (`verify_commitment_hash`) passes locally, but the parties hold different commitments for M and therefore compute different master public keys.

### Impact Explanation
Honest parties complete the DKG protocol without error but hold inconsistent `KeygenOutput.public_key` values. Any subsequent threshold signing attempt will fail or produce invalid signatures, as parties disagree on the public key against which shares are verified. This matches the High impact: "Corruption of DKG outputs so honest parties accept inconsistent public keys."

### Likelihood Explanation
The minimum participant count (n=2 or n=3) is a common deployment for two-of-two or two-of-three multisig setups. The attack requires only that one participant be malicious and control their own protocol implementation — a standard adversarial assumption. No cryptographic assumptions need to be broken. The attack is fully local and testable.

### Recommendation
1. **Enforce a minimum participant count** of n≥4 in `assert_key_invariants` to ensure `echo_ready_thresholds` always returns non-zero thresholds, or document that n≤3 provides no Byzantine fault tolerance and must only be used in fully-trusted settings.
2. **Protect round 2.9 with echo broadcast** instead of plain `send_many`, so that `commitment_hash` delivery has the same consistency guarantees as the commitment itself.
3. Alternatively, have honest parties cross-check each other's received `commitment_hash` values before proceeding to round 3, so equivocation is detectable.

### Proof of Concept
Simulate n=3 DKG where M implements a custom `Protocol` that:
- In round 2.9: calls `send_private(A, hash_A)` and `send_private(B, hash_B)` where `hash_A = H(M, C_M_A, sid)` and `hash_B = H(M, C_M_B, sid)`
- In round 3 SEND phase: calls `send_private(A, SEND(C_M_A))` and `send_private(B, SEND(C_M_B))`

Assert that after protocol completion, `output_A.public_key != output_B.public_key`.

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
