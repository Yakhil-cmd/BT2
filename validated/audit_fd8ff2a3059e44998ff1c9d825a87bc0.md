### Title
Native Resource Exhaustion via Malicious Callee Bypasses 63/64 Rule, Causing Full Transaction Revert - (`evm_interpreter/src/ee_trait_impl.rs`, `basic_bootloader/src/bootloader/runner.rs`)

---

### Summary

ZKsync OS implements a dual resource model: **Ergs** (EVM gas) and **Native resources** (proving cost). While the 63/64 rule is correctly applied to Ergs when passing resources to a subcall, **native resources are transferred in full to the callee with no cap**. A malicious callee can deliberately exhaust all native resources, causing the **entire transaction** to revert — not just the subcall. This is an EVM semantic mismatch: in standard EVM, a caller can limit gas forwarded to a callee (via the 63/64 rule or explicit gas cap), protecting itself from a malicious receiver. In ZKsync OS, no such protection exists for native resources.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, `calculate_resources_passed_in_external_call` applies the 63/64 rule only to Ergs:

```rust
// we just need to apply 63/64 rule, as System/IO is responsible for the rest
let max_passable_ergs =
    gas_utils::apply_63_64_rule(resources_available_in_caller_frame.ergs());
let ergs_to_pass = core::cmp::min(call_request.ergs_to_pass, max_passable_ergs);
let mut resources_to_pass = S::Resources::from_ergs(ergs_to_pass);
resources_available_in_caller_frame.charge(&resources_to_pass).unwrap();
``` [1](#0-0) 

Immediately after, in `basic_bootloader/src/bootloader/runner.rs`, **all** remaining native resources are unconditionally transferred to the callee:

```rust
// Give native resource to the callee.
resources_in_caller_frame.give_native_to(&mut callee_resources);
``` [2](#0-1) 

This is explicitly documented as a design property:

> "The native resources are passed fully from frame to frame, a call cannot set a limit on how much of it the callee can spend." [3](#0-2) 

When native resources are exhausted inside a callee, the entire transaction is reverted and all ergs are exhausted:

```rust
Err(e) => match e.root_cause() {
    RootCause::Runtime(e @ RuntimeError::FatalRuntimeError(_)) => {
        context.resources.main_resources.exhaust_ergs();
        system.finish_global_frame(Some(&main_body_rollback_handle))?;
        (ExecutionResult::Revert { output: &[] }, None)
    }
``` [4](#0-3) 

The native resource budget is capped at `MAX_NATIVE_COMPUTATIONAL = 1 << 35` per transaction: [5](#0-4) 

A malicious callee that performs many storage writes, hash computations, or other native-expensive operations can exhaust this budget entirely within a single subcall, reverting the caller's transaction.

---

### Impact Explanation

Any on-chain contract that sends ETH (or calls) to an attacker-controlled address — such as a marketplace paying a seller, a DEX routing through a user-supplied hook, or any protocol that forwards value to an arbitrary recipient — is vulnerable. The attacker's contract, upon receiving the call, performs operations that are cheap in Ergs but expensive in native resources (e.g., repeated SSTORE to new slots). This exhausts the native budget, causing the **entire parent transaction** to revert. The caller's state changes are rolled back, the user loses all gas, and the protocol operation fails.

In standard EVM, the caller could protect itself by capping gas forwarded to the callee (e.g., `call{gas: 2300}(...)`). In ZKsync OS, this protection is ineffective because native resources are always passed in full regardless of the Ergs cap.

---

### Likelihood Explanation

The attack requires only:
1. Deploying a contract with a `receive()` or `fallback()` that performs native-expensive operations (e.g., writing to many new storage slots via SSTORE).
2. Positioning that contract as a recipient in any protocol that sends ETH to user-supplied addresses.

This is a low-cost, permissionless attack. Any marketplace, payment splitter, DEX, or lending protocol deployed on ZKsync OS that follows standard EVM patterns for ETH distribution is susceptible. The attacker does not need privileged access.

---

### Recommendation

1. **Document the semantic difference prominently** in developer-facing materials: unlike standard EVM, limiting gas forwarded to a callee does NOT protect the caller from native resource exhaustion.
2. **Consider applying a native resource cap per subcall**, analogous to the 63/64 rule for Ergs. This would allow callers to retain a portion of native resources after a subcall, preventing a malicious callee from exhausting the full budget.
3. **Alternatively**, treat native resource exhaustion in a subcall as a subcall failure (returning 0 on the stack) rather than a full transaction revert, consistent with how EVM handles out-of-gas in subcalls.

---

### Proof of Concept

**Attack contract (Solidity):**
```solidity
contract MaliciousReceiver {
    mapping(uint256 => uint256) private s;
    uint256 private counter;

    receive() external payable {
        // Exhaust native resources via many SSTORE to new slots
        for (uint256 i = 0; i < 10000; i++) {
            s[counter++] = i + 1;
        }
    }
}
```

**Victim contract (e.g., a marketplace):**
```solidity
contract Marketplace {
    function buy(address seller) external payable {
        // Sends ETH to seller — triggers MaliciousReceiver.receive()
        (bool ok,) = seller.call{value: msg.value}("");
        require(ok, "payment failed");
        // State updates below are never reached; entire tx reverts
        emit Sale(seller, msg.value);
    }
}
```

**Steps:**
1. Attacker deploys `MaliciousReceiver` and registers it as a seller in `Marketplace`.
2. A legitimate buyer calls `Marketplace.buy{value: 1 ether}(attackerAddress)`.
3. The bootloader passes all native resources to the `MaliciousReceiver.receive()` frame via `give_native_to`.
4. `MaliciousReceiver` exhausts native resources through repeated SSTORE operations.
5. `FatalRuntimeError::OutOfNativeResources` propagates up; the bootloader exhausts all Ergs and reverts the entire transaction.
6. The buyer loses all gas; the sale never completes; the marketplace is effectively DoS'd for any buyer targeting this seller. [6](#0-5) [7](#0-6) [8](#0-7)

        let mut stipend = None;

        // Additional cost for non-zero value for general calls
        if call_request.modifier != CallModifier::Constructor {
            // Gas stipend calculation
            let is_delegate = call_request.is_delegate();
            let is_callcode = call_request.is_callcode();

            // Positive value cost and stipend
            stipend = if !is_delegate && !call_request.nominal_token_value.is_zero() {
                let positive_value_cost = S::Resources::from_ergs(Ergs(CALLVALUE * ERGS_PER_GAS));
                resources_available_in_caller_frame.charge(&positive_value_cost)?;
                Some(Ergs(CALL_STIPEND * ERGS_PER_GAS))
            } else {
                None
            };

            // Account creation cost
            let callee_is_empty = callee_parameters.nonce == 0
                && callee_parameters.unpadded_code_len == 0
                && callee_parameters.nominal_token_balance.is_zero();
            if !is_callcode_or_delegate
                && !call_request.nominal_token_value.is_zero()
                && callee_is_empty
            {
                let callee_creation_cost = S::Resources::from_ergs(Ergs(NEWACCOUNT * ERGS_PER_GAS));
                resources_available_in_caller_frame.charge(&callee_creation_cost)?

        // we just need to apply 63/64 rule, as System/IO is responsible for the rest

        let mut resources_to_pass = S::Resources::from_ergs(ergs_to_pass);

        // This never panics because max_passable_ergs <= resources_available_in_caller_frame
        resources_available_in_caller_frame

            .unwrap();

        // Add stipend
        if let Some(stipend) = stipend {
            resources_to_pass.add_ergs(stipend);
        }

    }

The native resource models the offchain cost of processing a transaction. Currently, this is dominated by proving and publishing data. A good intuition for it is "how many RISC-V cycles it takes to prove a given computation".
