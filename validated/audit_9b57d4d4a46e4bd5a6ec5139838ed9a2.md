### Title
Immutable Extension Revert Permanently Locks LP Funds in `removeLiquidity` - (File: `metric-core/contracts/ExtensionCalling.sol`, `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

Extensions are stored as immutable variables in `ExtensionCalling`. `removeLiquidity` unconditionally calls `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` through `CallExtension.callExtension`, which bubbles up all reverts. If any registered extension begins reverting — due to a bug, an upstream dependency change, or a governance action in a dependency — every call to `removeLiquidity` reverts permanently. There is no mechanism to replace or bypass a broken extension, and pausing the pool does not protect `removeLiquidity`. LP principal is locked with no escape hatch.

---

### Finding Description

**Extension addresses are immutable.** `ExtensionCalling` stores up to seven extension addresses and six order bitmasks as Solidity `immutable` variables set at construction time: [1](#0-0) 

Once deployed, these values cannot be changed. There is no factory function, admin function, or emergency path to replace or nullify a broken extension.

**`removeLiquidity` calls extensions unconditionally.** Both the before- and after-hooks are invoked on every call: [2](#0-1) 

**`CallExtension.callExtension` propagates all reverts.** There is no `try/catch` or silent-failure path. A revert from the extension is re-thrown directly to the caller: [3](#0-2) 

**`_callExtensionsInOrder` iterates the packed bitmask and calls each registered extension.** A single reverting extension in the chain aborts the entire operation: [4](#0-3) 

**Pausing does not protect `removeLiquidity`.** The `whenNotPaused` modifier is applied only to `swap`. `removeLiquidity` carries no such guard, so pausing the pool does not unblock withdrawals when an extension is broken: [2](#0-1) [5](#0-4) 

**A real production extension (`OracleValueStopLossExtension`) is registered for `afterSwap` and reads external oracle state.** If the oracle feed it depends on becomes unavailable or its internal state becomes inconsistent, the extension can revert, bricking all swaps. The same pattern applies to any extension registered for remove-liquidity hooks: [6](#0-5) 

---

### Impact Explanation

If any extension registered for `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` begins reverting, every call to `removeLiquidity` reverts. LP principal is permanently locked in the pool. `addLiquidity` and `swap` are similarly bricked if the extension is registered for those hooks. There is no admin escape hatch, no factory override, and no pause-based bypass. This matches the "broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" impact gate.

---

### Likelihood Explanation

Extensions are explicitly designed to gate and modify pool behavior (e.g., `OracleValueStopLossExtension`, `GateExtension`). Any extension that reads external state — an oracle, a price feed, an allowlist contract — can begin reverting if that dependency changes. This is not a hypothetical: the Loihi audit identified the exact same class of failure in assimilators. The trigger requires no privileged actor; any condition that causes the extension contract to revert (including a bug in the extension itself, an upstream governance change, or a dependency upgrade) is sufficient.

---

### Recommendation

1. **Add a `try/catch` wrapper in `_callExtensionsInOrder`** (or in `CallExtension.callExtension`) so that extension failures can be handled gracefully rather than propagating as hard reverts. At minimum, allow `removeLiquidity` to proceed even if an extension reverts, emitting an event for the failure.

2. **Alternatively, add a factory-controlled emergency bypass flag** that, when set, skips extension calls in `removeLiquidity` only, allowing LPs to withdraw their principal even when an extension is broken.

3. **Or implement a timelocked extension-replacement mechanism** analogous to `proposePoolPriceProvider` / `setPriceProvider`, allowing a broken extension to be swapped out after a delay.

---

### Proof of Concept

1. Deploy a pool with an extension registered for `beforeRemoveLiquidity` (e.g., `BEFORE_REMOVE_LIQUIDITY_ORDER = 1`).
2. LP calls `addLiquidity` successfully; funds are deposited.
3. The extension contract begins reverting on `beforeRemoveLiquidity` (e.g., due to a dependency change or a bug).
4. LP calls `removeLiquidity`. Execution reaches `_beforeRemoveLiquidity` → `_callExtensionsInOrder` → `CallExtension.callExtension` → extension reverts → revert propagates → `removeLiquidity` reverts.
5. Because `EXTENSION_1` through `EXTENSION_7` are `immutable`, no admin or factory function can replace the broken extension.
6. Because `removeLiquidity` has no `whenNotPaused` guard and no bypass path, pausing the pool does not help.
7. LP funds are permanently locked.

The `GateExtension` mock already demonstrates this pattern: `beforeRemoveLiquidity` can revert (it does so when `allowDeposit` is false in the deposit gate path), confirming the revert-propagation chain is live in the codebase. [7](#0-6) [8](#0-7)

```text
  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }
```

```text
  function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
      if (result.length > 0) {
        assembly ("memory-safe") {
          revert(add(result, 32), mload(result))
        }
      }
      revert ExtensionCallFailed();
    }
    if (result.length < 32) {
      revert InvalidExtensionResponse();
    }
    bytes4 returnedSelector;
    assembly ("memory-safe") {
      returnedSelector := mload(add(result, 32))
    }
    bytes4 expectedSelector;
    assembly ("memory-safe") {
      expectedSelector := mload(add(data, 32))
    }
    if (returnedSelector != expectedSelector) {
      revert InvalidExtensionResponse();
    }
  }
```
