### Title
Pool Admin Can Instantly Set Bin Additional Fees to Maximum Without Cap or Timelock, Enabling Swap Frontrunning - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` allows the pool admin to instantly set per-bin additional spread fees (`addFeeBuyE6`, `addFeeSellE6`) to any value up to `uint16` max (65,535 = 6.5535% in E6 units) with no cap validation and no timelock. This is structurally inconsistent with the main pool fee path, which enforces explicit caps and collects accrued fees before any change. A pool admin can frontrun a pending user swap by spiking the bin additional fee, causing the user to pay a materially higher fee than expected.

---

### Finding Description

The factory exposes two distinct fee-change paths for the pool admin:

**Path 1 — Main pool fees (`setPoolAdminFees`):** [1](#0-0) 

This path enforces explicit cap checks before applying any change: [2](#0-1) 

And it collects all accrued fees at the old rate before the new rate takes effect: [3](#0-2) 

**Path 2 — Bin additional fees (`setPoolBinAdditionalFees`):** [4](#0-3) 

This path has **no cap check** and **no fee collection** before the change. The only guard is `onlyPoolAdmin(pool)`. The call passes directly to the pool: [5](#0-4) 

The pool itself also performs no cap validation — only a bin-index range check.

The factory's cap registry (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`, `maxProtocolSpreadFeeE6`, `maxProtocolNotionalFeeE8`) covers only the main spread and notional fees: [6](#0-5) 

There is no `maxAdminBinFeeE6` or equivalent. The `setFeeCaps` owner function also does not govern bin additional fees: [7](#0-6) 

During a swap, the bin additional fee is added directly on top of the oracle-derived base fee: [8](#0-7) 

---

### Impact Explanation

A pool admin who observes a pending user swap in the mempool can:

1. Call `setPoolBinAdditionalFees(pool, curBinIdx, 65535, 65535)` — raising the per-bin additional fee to 6.5535% instantly, with no timelock.
2. The user's swap executes at the inflated fee rate; the excess spread stays in the bin as LP revenue.
3. The admin calls `setPoolBinAdditionalFees` again to restore the original fee.

The user pays up to 6.5535% more than the fee they observed before submitting. Because the bin additional fee is additive on top of the base oracle spread fee, the total effective fee seen by the user can spike significantly above any rate they could have anticipated. The excess accrues to LP providers; if the pool admin is also an LP, they capture the extracted value directly.

This breaks the invariant that the pool admin is "semi-trusted only inside caps and timelocks." The main fee path enforces both; the bin additional fee path enforces neither.

---

### Likelihood Explanation

- The pool admin is a live, on-chain role with no timelock on this specific action.
- Mempool visibility of pending swaps is standard on non-private mempools.
- The attack requires only a single transaction from the pool admin, executable atomically before the user's swap lands.
- No special token behavior or external oracle manipulation is required.

---

### Recommendation

1. **Add a cap**: Introduce a `maxAdminBinAdditionalFeeE6` state variable (governed by `setFeeCaps` or a dedicated owner function) and enforce it inside `setPoolBinAdditionalFees`:
   ```solidity
   if (addFeeBuyE6 > maxAdminBinAdditionalFeeE6) revert BinFeeTooHigh();
   if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert BinFeeTooHigh();
   ```

2. **Add a timelock**: Mirror the price-provider rotation pattern — require a proposal + delay before bin additional fee changes take effect, so users can observe and react to pending changes before they are applied.

---

### Proof of Concept

```
Setup:
  - Pool deployed with addFeeBuyE6 = 0 on bin 0.
  - User submits swap(zeroForOne=false, ...) targeting bin 0; tx is in mempool.

Attack:
  1. Pool admin calls setPoolBinAdditionalFees(pool, 0, 65535, 65535).
     → No cap check, no timelock, executes immediately.
     → Pool's _binStates[0].addFeeBuyE6 = 65535 (6.5535%).

  2. User's swap executes. SwapMath applies:
       totalFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
     → User pays ~6.5535% more than expected on the swap output.

  3. Pool admin calls setPoolBinAdditionalFees(pool, 0, 0, 0) to restore.

Result: User's effective fee is inflated by up to 6.5535% with no on-chain
        protection or advance notice.
```
