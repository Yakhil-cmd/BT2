### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Bridge Exit - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens **before** calling the Aurora bridge precompile via inline assembly, and then **never check the return value** of that `call`. If the precompile call fails for any reason, the tokens are permanently destroyed with no corresponding NEP-141 or Ethereum withdrawal, resulting in an irreversible loss of user funds.

---

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
// EvmErc20.sol lines 53–63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // <-- tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no require(res), no revert
    }
}
```

`EvmErc20V2.sol` has the identical structural flaw in both `withdrawToNear` (lines 53–64) and `withdrawToEthereum` (lines 66–77).

The `res` variable captures the EVM `call` success flag (1 = success, 0 = failure), but it is **never inspected**. Solidity assembly `call` does not automatically revert the caller's state on failure — the caller must explicitly check `res` and revert. Because `_burn` executes unconditionally before the precompile call, a failed precompile call leaves the ERC-20 supply permanently reduced with no corresponding bridge action.

The ExitToNear precompile (`engine-precompiles/src/native.rs`) can and does return `ExitError` in multiple code paths — for example, when the recipient account ID is invalid, when the amount is zero, or when the input is malformed. When the precompile returns an error, the EVM `call` opcode returns 0, but the `EvmErc20` contracts proceed as if the withdrawal succeeded.

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` with any input that causes the precompile to fail (e.g., an overly long or malformed recipient string, an out-of-gas condition, or any internal precompile error) will have their ERC-20 tokens burned with no corresponding NEP-141 or Ethereum-side credit. The tokens are gone from the EVM state and cannot be recovered. There is no refund path because the burn is unconditional and the failure is silent.

---

### Likelihood Explanation

**Medium.** The ExitToNear precompile validates its input and returns errors in multiple branches. A user who passes an invalid NEAR account ID (e.g., one that is too long, contains invalid characters, or is otherwise rejected by the precompile's parsing logic) will silently lose their tokens. This is reachable by any unprivileged EVM user who calls `withdrawToNear` directly on a deployed `EvmErc20` or `EvmErc20V2` contract. No admin access or special privilege is required.

---

### Recommendation

Check the return value of the assembly `call` and revert if it indicates failure. The fix mirrors the standard pattern for safe precompile interaction:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to **both** `withdrawToNear` and `withdrawToEthereum` in **both** `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures that if the precompile call fails, the entire transaction (including the `_burn`) is reverted, preserving the user's token balance.

---

### Proof of Concept

1. Deploy an `EvmErc20` or `EvmErc20V2` contract on Aurora (standard bridge deployment).
2. Mint tokens to an attacker-controlled address via `mint`.
3. Call `withdrawToNear` with a recipient string that is syntactically invalid or too long to be accepted by the ExitToNear precompile (e.g., a 200-byte garbage string).
4. The ExitToNear precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` returns an error; the EVM `call` returns `res = 0`.
5. Because `res` is never checked, the function returns normally.
6. The user's ERC-20 balance is zero (burned at step 3), no NEP-141 tokens are credited on NEAR, and no refund occurs.
7. Funds are permanently lost.
