### Title
Missing Validation of Participant CKD Contributions Allows Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator aggregates `CKDOutput` contributions from all participants by blindly adding received group elements together with no proof of correctness or consistency check. A single malicious participant can send arbitrary elliptic-curve points, causing the coordinator to produce a corrupted `CKDOutput`. Any downstream consumer that unmasks this output will derive an incorrect confidential key. This is the direct analog of the Balancer report's pattern: data from an external source is consumed without calling the required guard/validation function.

### Finding Description
In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55), the coordinator receives a `CKDOutput` — a pair of G1 points `(norm_big_y, norm_big_c)` — from every other participant and accumulates them with unchecked addition:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The honest computation in `compute_signature_share` (`src/confidential_key_derivation/protocol.rs`, lines 148–182) produces:

```
norm_big_y_i = λ_i · (y_i · G)
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

so that after aggregation, `norm_big_c − app_sk · norm_big_y = msk · H(pk ‖ app_id)`.

No zero-knowledge proof, commitment, or any other consistency check is attached to the values a participant sends. A malicious participant is free to substitute any pair of G1 points for their honest contribution. There is no call to any validation routine before the coordinator incorporates the received values — the library provides no guard function here, and none is invoked.

The root cause is structurally identical to the Balancer finding: a critical validation step (proving that `norm_big_c_i − app_sk · norm_big_y_i = λ_i · x_i · H(pk ‖ app_id)`) is entirely absent, so the coordinator's aggregation silently incorporates attacker-controlled material.

### Impact Explanation
**High — Corruption of CKD outputs so honest parties accept an incorrect confidential derived key.**

After the malicious participant injects an arbitrary offset `Δ` into `norm_big_c`, the coordinator's `CKDOutput` satisfies:

```
unmask(app_sk) = msk · H(pk ‖ app_id) + Δ
```

Every honest party that relies on this output derives a wrong confidential key. Because the protocol produces a single aggregated output with no per-participant accountability, honest parties have no way to detect the corruption after the fact. The derived key is silently wrong, which can cause downstream cryptographic operations (e.g., decryption, re-encryption, or further derivation) to fail or produce attacker-influenced results.

### Likelihood Explanation
Any participant in a CKD session — a role that requires no special privilege beyond being listed in `participants` — can mount this attack. The attack requires only that the malicious party deviate from the protocol by sending a crafted pair of G1 points. No cryptographic assumption needs to be broken, no key material needs to be leaked, and no external dependency needs to be compromised. The attack is deterministic and requires a single protocol execution.

### Recommendation
Require each participant to accompany their `(norm_big_y_i, norm_big_c_i)` with a zero-knowledge proof of correct formation — specifically, a proof that the discrete-log relationship between `norm_big_y_i`, `norm_big_c_i`, and the participant's public verification share holds. A Chaum–Pedersen / `dlogeq`-style proof (already present in the codebase at `src/crypto/proofs/dlogeq.rs`) is the natural tool: it proves knowledge of `(y_i, x_i)` such that `norm_big_y_i = λ_i · y_i · G` and `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` without revealing the secrets. The coordinator must verify this proof before adding the contribution to the running sum.

### Proof of Concept
1. Run a CKD session with participants `[P1, P2, P3]` where `P2` is malicious.
2. `P2` computes the honest `(norm_big_y_2, norm_big_c_2)` but instead sends `(G1::generator(), G1::generator())` — the generator point — to the coordinator.
3. The coordinator executes the loop at lines 50–55 of `protocol.rs`, adding the injected generator to both accumulators.
4. The resulting `CKDOutput` satisfies:
   ```
   norm_big_c − app_sk · norm_big_y
     = msk · H(pk ‖ app_id)
       + (1 − app_sk) · G1::generator()   ← attacker-injected term
   ```
5. The application calls `ckd_output.unmask(app_sk)` and obtains a point that differs from the correct confidential key by `(1 − app_sk) · G1::generator()`.
6. All honest parties accept this corrupted output because the protocol provides no mechanism to detect the deviation. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
