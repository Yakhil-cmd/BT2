### Title
Missing Validation of `app_pk` Identity Element Breaks CKD Blinding, Disclosing Confidential Derived Key to Coordinator - (File: src/confidential_key_derivation/protocol.rs)

### Summary

The `ckd()` entry point and the internal `compute_signature_share()` function accept an `app_pk: PublicKey` (an `ElementG1`) without checking whether it is the group identity element. The entire confidentiality guarantee of the CKD protocol rests on the blinding term `app_pk * y`; if `app_pk` is the identity, that term vanishes, and the coordinator receives the unblinded confidential key directly.

### Finding Description

The CKD protocol is designed so that each participant computes:

```
big_s  = x_i · H(pk ‖ app_id)          // secret share contribution
big_c  = big_s + app_pk · y            // blinded with random y and client public key
big_y  = y · G
```

The coordinator sums the Lagrange-weighted shares and obtains `(Y, C)`. The client then recovers the confidential key as `C − app_sk · Y = msk · H(pk ‖ app_id)`. The blinding term `app_pk · y` is the only thing preventing the coordinator from reading the confidential key directly from `C`.

In `compute_signature_share()`:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;
``` [1](#0-0) 

If `app_pk` is the identity element of G1, then `app_pk * y.0 = identity`, so `big_c = big_s = x_i · H(pk ‖ app_id)`. After the coordinator sums all Lagrange-weighted contributions it holds:

```
C = Σ λ_i · x_i · H(pk ‖ app_id) = msk · H(pk ‖ app_id)
```

which is the confidential key itself, in the clear.

The `ckd()` public entry point performs several structural checks (participant count, duplicates, self-presence, coordinator presence) but performs **no validation whatsoever on `app_pk`**:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,          // ← accepted without any check
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
``` [2](#0-1) 

In a real deployment the coordinator receives the client's `app_pk` and distributes it to every participant before each participant calls `ckd()`. A malicious coordinator can substitute `app_pk = G1::identity()` for the value distributed to all participants, causing every participant to send an unblinded share. The coordinator then aggregates those shares and directly reads `msk · H(pk ‖ app_id)`.

### Impact Explanation

The confidential key `msk · H(pk ‖ app_id)` is the secret output that the CKD protocol is specifically designed to hide from the MPC participants, including the coordinator. Disclosure of this value to the coordinator is a **critical** impact matching the allowed scope: *"Extraction, reconstruction, or disclosure of … confidential derived secrets."*

The coordinator learns the full confidential key for any `app_id` it chooses to target, for the lifetime of the current master key, without the client's knowledge.

### Likelihood Explanation

The coordinator is a participant in the MPC network and is the natural entity that relays `app_pk` from the client to all other participants. A single malicious coordinator can mount this attack unilaterally, with no additional privileges, no cryptographic breaks, and no cooperation from other participants. The attack requires only substituting one field in the parameters passed to `ckd()`.

### Recommendation

Reject the identity element for `app_pk` at the `ckd()` entry point before any protocol state is created:

```rust
pub fn ckd(
    ...
    app_pk: PublicKey,
    ...
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // Reject the identity element: a zero app_pk destroys the blinding guarantee.
    if bool::from(app_pk.is_identity()) {
        return Err(InitializationError::BadParameters(
            "app_pk must not be the identity element".to_string(),
        ));
    }
    ...
}
```

Additionally, add a corresponding check inside `compute_signature_share()` as a defence-in-depth assertion, mirroring the pattern already used in `BLS12381G1Group::serialize()`: [3](#0-2) 

### Proof of Concept

```rust
#[test]
fn test_ckd_identity_app_pk_leaks_confidential_key() {
    use crate::confidential_key_derivation::{
        ciphersuite::{BLS12381SHA256, G1Projective},
        hash_app_id_with_pk, AppId, ElementG1, KeygenOutput, SigningShare, VerifyingKey,
    };
    use crate::participants::{Participant, ParticipantList};
    use crate::test_utils::{check_one_coordinator_output, generate_participants,
                            run_protocol, GenProtocol, MockCryptoRng};
    use blstrs::Scalar;
    use elliptic_curve::{Field, Group};
    use rand::{RngCore, SeedableRng};

    let mut rng = MockCryptoRng::seed_from_u64(42);
    let app_id = AppId::try_from(b"Near App").unwrap();

    // Malicious coordinator supplies the identity as app_pk
    let malicious_app_pk = ElementG1::identity(); // <-- attack input

    let participants = generate_participants(3);
    let coordinator = participants[0];
    let participant_list = ParticipantList::new(&participants).unwrap();

    // Build honest key material (same as the normal test)
    let mut private_shares = Vec::new();
    let mut msk = Scalar::ZERO;
    for (i, _) in participants.iter().enumerate() {
        let mut rng_p = MockCryptoRng::seed_from_u64(rng.next_u64());
        let share = SigningShare::new(Scalar::random(&mut rng_p));
        let lambda = participant_list
            .lagrange::<BLS12381SHA256>(participant_list.get_participant(i).unwrap())
            .unwrap();
        msk += lambda * share.to_scalar();
        private_shares.push(share);
    }
    let pk = VerifyingKey::new(blstrs::G2Projective::generator() * msk);

    let mut protocols: GenProtocol<_> = Vec::new();
    for (i, p) in participants.iter().enumerate() {
        let rng_p = MockCryptoRng::seed_from_u64(rng.next_u64());
        let key_pair = KeygenOutput { public_key: pk, private_share: private_shares[i] };
        // Every participant receives app_pk = identity from the malicious coordinator
        let protocol = crate::confidential_key_derivation::protocol::ckd(
            &participants, coordinator, *p, key_pair,
            app_id.clone(), malicious_app_pk, rng_p,
        ).unwrap();
        protocols.push((*p, Box::new(protocol)));
    }

    let result = run_protocol(protocols).unwrap();
    let ckd_output = check_one_coordinator_output(result, coordinator).unwrap();

    // With app_pk = identity, big_c = msk * H(pk || app_id) directly.
    // The coordinator can read the confidential key from big_c without app_sk.
    let expected_confidential_key = hash_app_id_with_pk(&pk, &app_id) * msk;
    // big_c IS the confidential key; no unmask needed.
    assert_eq!(ckd_output.big_c(), expected_confidential_key,
        "Coordinator obtained the confidential key directly due to missing app_pk validation");
}
```

The test demonstrates that when `app_pk = G1::identity()` is accepted without error, the coordinator's `CKDOutput::big_c` field equals `msk · H(pk ‖ app_id)` — the confidential key — without requiring `app_sk` to unmask it.
