Audit Report

## Title
Missing Caller Authorization in `icrc103_get_allowances` Allows Unrestricted Allowance Enumeration for Any Account — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary
The `icrc103_get_allowances` query endpoint accepts an arbitrary `from_account` argument and returns all allowances for that account without verifying that the caller owns the account. The `GetAllowancesError::AccessDenied` error variant and the `icrc103:public_allowances` metadata flag are both defined in the codebase but are never consulted at runtime, rendering the ICRC-103 access-control mechanism entirely inoperative.

## Finding Description
At `rs/ledger_suite/icrc1/ledger/src/main.rs` lines 1215–1231, the handler resolves `from_account` to either the caller's default account or the explicitly supplied account, then unconditionally calls `get_allowances` and wraps the result in `Ok(...)`:

```rust
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    // ... max_results computation ...
    Ok(get_allowances(from_account, arg.prev_spender, max_results, ic_cdk::api::time()))
}
```

There is no guard of the form `if caller != from_account.owner && !public_allowances { return Err(AccessDenied) }`. The `AccessDenied` variant defined in `packages/icrc-ledger-types/src/icrc103/get_allowances.rs` (line 17) is never instantiated anywhere in the ledger. The constant `ICRC103_PUBLIC_ALLOWANCES` defined in `packages/icrc-ledger-types/src/icrc/metadata_key.rs` (line 92) appears only in that one definition site — no ledger code reads or enforces it. A `grep` for `public_allowances` across the entire repository returns only that single constant definition.

Exploit path: any IC principal P supplies `from_account: Some(victim_account)` in a query call to `icrc103_get_allowances`. The ledger returns the full paginated list of `(from_account, to_spender, allowance, expires_at)` tuples for the victim. Repeated calls with `prev_spender` set to the last returned spender enumerate the complete allowance table.

## Impact Explanation
This is a significant ledger-level privacy/information-disclosure impact with concrete user harm. Any party can learn the complete allowance graph for any account on the ledger — every approved spender, approved amount, and expiry time — without the account owner's knowledge or consent. The ICRC-103 standard explicitly provides the `icrc103:public_allowances` flag so ledger operators can restrict this data to account owners; the implementation ignores it entirely. This affects production ICRC-1 ledger deployments (ckBTC, ckETH, ckERC-20, ICP ledger) and constitutes a significant ledger security impact with concrete user harm, fitting the **High ($2,000–$10,000)** bounty tier for significant ck-token/ledger security impact.

## Likelihood Explanation
The endpoint is a public query call requiring no cycles, no authentication, and no special privileges. Any IC principal can call it immediately and repeatedly. Pagination via `prev_spender` makes full enumeration of any account's allowance table trivial. No rate limiting beyond `max_take_allowances` per page exists.

## Recommendation
Add a caller-identity guard immediately after resolving `from_account`, consulting the `public_allowances` ledger configuration:

```rust
let caller = ic_cdk::api::msg_caller();
let public = Access::with_ledger(|l| l.public_allowances());
if !public && caller != from_account.owner {
    return Err(GetAllowancesError::AccessDenied {
        reason: "caller is not the account owner".to_string(),
    });
}
```

Additionally, ensure `icrc1_metadata` correctly exposes the `icrc103:public_allowances` value so clients can reason about the ledger's policy.

## Proof of Concept
1. Deploy the ICRC-1 ledger canister locally (or use an existing test environment).
2. Account A (principal `A_owner`) calls `icrc2_approve` to grant spender S an allowance of 1000 tokens.
3. From any unrelated principal P (P ≠ A_owner), call:
   ```
   icrc103_get_allowances({ from_account: Some(A), prev_spender: None, take: None })
   ```
4. Observe the response is `Ok([{from_account: A, to_spender: S, allowance: 1000, expires_at: ...}])` — no `AccessDenied` error is returned regardless of the `icrc103:public_allowances` metadata value.
5. This is directly confirmed by the existing integration tests in `rs/ledger_suite/tests/sm-tests/src/lib.rs` (lines 3017–3079), which call `list_allowances` with `from_account` set to the approver's account from the approver's own principal — but no test ever attempts the call from a *different* principal to verify rejection.
