Audit Report

## Title
Cross-Subaccount Allowance Enumeration via Incomplete Break Condition in `get_allowances` - (File: `rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function iterates a `StableBTreeMap` keyed by `(account, spender)` and breaks only when `account_spender.account.owner != from.owner`. Because `Account` ordering is `(owner, effective_subaccount)`, a query for `{owner:X, subaccount:S0}` will continue past the subaccount boundary and return allowances stored under any subaccount of X that sorts after S0. Any unauthenticated caller can trigger this via the public `icrc103_get_allowances` query endpoint to enumerate allowance metadata ‚Äî spender identities, amounts, and expiry timestamps ‚Äî across all subaccounts of a target principal.

## Finding Description
**Ordering chain.** `AccountSpender::Ord` orders by `account` first, then `spender` (`rs/ledger_suite/icrc1/ledger/src/lib.rs`, lines 372‚Äì378). `Account::Ord` orders by `owner` first, then `effective_subaccount()` (`packages/icrc-ledger-types/src/icrc1/account.rs`, lines 54‚Äì60). The BTreeMap therefore stores entries in the order:
```
{owner:X, subaccount:[0;32]} -> spender_A
{owner:X, subaccount:[1;32]} -> spender_B   ‚Üê different subaccount, same owner
{owner:X, subaccount:[2;32]} -> spender_C   ‚Üê different subaccount, same owner
{owner:Y, ...}                               ‚Üê different owner
```

**Defective loop.** `get_allowances` (lines 1174‚Äì1222) starts a range scan at `start_account_spender` built from the exact queried `from` account, then breaks only at line 1204:
```rust
if account_spender.account.owner != from.owner {
    break;
}
```
This condition is `false` for every subaccount of the same owner, so the loop continues past the queried subaccount boundary and collects allowances belonging to other subaccounts of X. The correct guard ‚Äî `account_spender.account != from` ‚Äî is never applied. `Account::PartialEq` already compares both `owner` and `effective_subaccount()` (lines 40‚Äì44 of `account.rs`), so the fix is a one-field change.

**No access control.** `icrc103_get_allowances` (`rs/ledger_suite/icrc1/ledger/src/main.rs`, lines 1214‚Äì1232) is a `#[query]` callable by any principal, including anonymous. It accepts an arbitrary `from_account` with no ownership check.

## Impact Explanation
An unprivileged attacker can enumerate, for any principal X: which subaccounts have active allowances, the identity of every approved spender, and exact allowance amounts and expiration timestamps. This is a concrete, unauthenticated information-disclosure vulnerability in the ICRC ledger ‚Äî an in-scope financial integration component. It violates the ICRC-103 invariant that results must be scoped to the exact queried account (owner + subaccount). No funds can be moved, placing this in the **Medium ($200‚Äì$2,000)** band: the attack is trivially easy and requires no special privileges, but the impact is limited to allowance metadata disclosure with no direct path to fund theft or protocol disruption.

## Likelihood Explanation
Exploitation requires a single unauthenticated query call with a crafted `from_account`. No key material, no privileged access, no social engineering. The attacker needs only a target principal X, which is observable from on-chain transaction history via ICRC-3. The attack is repeatable, cheap, and leaves no on-chain trace.

## Recommendation
Replace the owner-only break condition at line 1204 with a full `Account` comparison:
```rust
// Before:
if account_spender.account.owner != from.owner {
    break;
}

// After:
if account_spender.account != from {
    break;
}
```
`Account::PartialEq` already compares both `owner` and `effective_subaccount()`, so this single-line change is sufficient. The BTreeMap range already starts at the correct `from` account, so the first entry differing in subaccount will immediately trigger the break.

## Proof of Concept
```
Setup (as owner X):
  icrc2_approve(from_subaccount=[1u8;32], spender=S1, amount=100)
  icrc2_approve(from_subaccount=[2u8;32], spender=S2, amount=200)

Attack (as anonymous or any principal):
  icrc103_get_allowances({
    from_account: Some({owner: X, subaccount: Some([0u8;32])}),
    prev_spender: None,
    take: None
  })

Expected: []   (no allowances exist for subaccount [0;32])
Actual:
  [
    { from_account: {owner:X, subaccount:[1u8;32]}, to_spender: S1, allowance: 100 },
    { from_account: {owner:X, subaccount:[2u8;32]}, to_spender: S2, allowance: 200 }
  ]
```
The range scan starts at `{account:{owner:X, subaccount:[0;32]}, spender:‚ä•}`. Since `[0u8;32] < [1u8;32] < [2u8;32]` lexicographically, both other subaccounts' entries fall within the range. The break condition `owner != from.owner` is never satisfied for them, so they are returned. This is reproducible as a deterministic PocketIC integration test or a unit test over a mock `StableBTreeMap`.
