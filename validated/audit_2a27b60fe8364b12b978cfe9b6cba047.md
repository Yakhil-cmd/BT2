### Title
`Declare` Transactions Bypass the `max_l2_gas_amount` Stateless Admission Check - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces an upper bound on `l2_gas.max_amount` for every transaction type **except** `Declare`. An explicit type-guard silently skips the check for Declare transactions, and the code itself acknowledges the gap with a TODO comment. Any caller can submit a Declare transaction with `l2_gas.max_amount` set to `u64::MAX` and the gateway's stateless path will accept it, violating the admission invariant that all transactions must have their declared L2 gas amount bounded by `max_l2_gas_amount`.

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← empty arm: Declare transactions skip the check entirely
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1,210,000,000`: [2](#0-1) 

For `Invoke` and `DeployAccount` transactions the check fires correctly. For `Declare` it is a no-op. The same function does enforce the `min_gas_price` check on all three types, so the asymmetry is specific to the amount upper-bound. [3](#0-2) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas **price** against the previous block price; it never checks the declared **amount**: [4](#0-3) 

The blockifier's `perform_pre_validation_stage` calls `check_fee_bounds`, which only verifies `max_amount >= minimal_gas_amount` (a lower bound), and `verify_can_pay_committed_bounds`, which checks account balance against `max_amount × max_price_per_unit`. For Declare transactions the stateful validator runs a full execution path (`self.execute(tx)`) rather than the pre-validation-only path used for Invoke: [5](#0-4) [6](#0-5) 

The balance check (`verify_can_pay_committed_bounds`) is the only downstream guard, and it is only effective when `max_price_per_unit > 0` and the account balance is insufficient. An account with sufficient balance (or one that sets `max_price_per_unit` at exactly `min_gas_price` with a moderate `max_amount` that still exceeds `max_l2_gas_amount`) passes all downstream checks and is admitted to the mempool.

The test suite explicitly documents that a Declare transaction with `l2_gas.max_amount` far above the configured limit is accepted as valid: [7](#0-6) 

### Impact Explanation

The gateway's stateless admission path accepts `Declare` transactions whose declared `l2_gas.max_amount` exceeds the operator-configured ceiling (`max_l2_gas_amount`). This breaks the invariant that all admitted transactions are bounded by the same resource-amount policy. Concretely:

- A well-funded account can submit a Declare transaction with `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit = min_gas_price`. The stateless validator passes it; the stateful validator's balance check passes because `max_possible_fee` uses saturating arithmetic and the account balance covers the saturated value; the transaction enters the mempool and is executed.
- The `max_l2_gas_amount` ceiling exists to prevent individual transactions from monopolising the block's L2 gas budget at the admission stage. Bypassing it for Declare transactions undermines that control.

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

### Likelihood Explanation

The bypass requires only a well-formed Declare transaction with an oversized `l2_gas.max_amount` field. No privileged access, no special contract, and no race condition is needed. Any user who can submit a Declare transaction can trigger this path. The TODO comment in the source confirms the gap is known but unresolved.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx {}` arm and apply the same `max_l2_gas_amount` upper-bound check to Declare transactions:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas-amount ceiling (e.g., because compilation is more expensive), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

### Proof of Concept

The existing test `valid_l2_gas_amount_on_declare` already demonstrates the bypass:

```rust
// config: max_l2_gas_amount = 100
// tx:     l2_gas.max_amount = 200  (exceeds limit)
// tx_type: Declare
// result: Ok(())   ← accepted despite exceeding the limit
``` [7](#0-6) 

A production exploit would be:
1. Construct an `RpcDeclareTransactionV3` with `l2_gas.max_amount = GasAmount(u64::MAX)` and `l2_gas.max_price_per_unit = min_gas_price`.
2. Submit via the gateway HTTP endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` despite `u64::MAX >> max_l2_gas_amount`.
4. The transaction proceeds through stateful validation (balance check passes if the account holds sufficient STRK) and enters the mempool, violating the admission invariant.

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();

            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });

            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {

                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
