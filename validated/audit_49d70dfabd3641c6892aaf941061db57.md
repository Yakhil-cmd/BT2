### Title
Unrefunded ETH Permanently Locked in `tick()` When No Minting Is Required — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is `external payable` with no access control and no refund logic. When called with `msg.value > 0` but no pending mint requests (`amountToMint == 0`), the entire ETH value is silently absorbed into the contract's balance and permanently locked. There is no `receive()` fallback, no withdrawal function, and no refund path anywhere in the contract or its inherited storage/proxy contracts.

---

### Finding Description

`tick()` is the sole entry point for triggering reward minting on L1:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // No else branch — msg.value is silently kept when amountToMint == 0
}
``` [1](#0-0) 

When `amountToMint == 0`, the function returns without using or refunding `msg.value`. The ETH is absorbed into the contract's balance. Neither `RewardSupplierStorage` nor `ProxySupportImpl` define any ETH withdrawal or recovery mechanism. [2](#0-1) [3](#0-2) 

A secondary issue: when `amountToMint > 0` and `msg.value = 0`, both `depositWithMessage{value: 0}` and `sendMessageToL2{value: 0}` are called with zero fees. The Starknet messaging contract charges a fee for L1→L2 messages (`getMaxL1MsgFee()` is defined in `IStarknetMessaging`), so a zero-fee call may either revert or produce an unprocessed L2 message, breaking the total-supply update to `MintingCurve`. [4](#0-3) 

---

### Impact Explanation

ETH sent to `tick()` when `amountToMint == 0` is permanently frozen in the contract with no recovery path. This matches **High: Permanent freezing of funds**. The locked ETH is also never usable for future `tick()` calls because the fee-forwarding uses `msg.value` directly (`{value: msg.value / 2}`), not the contract's accumulated balance — so the locked ETH provides no benefit to anyone.

---

### Likelihood Explanation

`tick()` has no access control — any unprivileged caller can invoke it. A realistic scenario:

1. A keeper/bot monitors pending mint requests and calls `tick()` with ETH attached.
2. Due to a race condition (another keeper already processed the mint request in the same block), `amountToMint` evaluates to `0` when the second call executes.
3. The second caller's ETH is permanently locked.

This is not a contrived edge case: competitive keeper networks routinely race to call permissionless trigger functions.

---

### Recommendation

1. **Refund unused ETH**: Add a refund at the end of `tick()` when `amountToMint == 0`:
   ```solidity
   if (amountToMint == 0) {
       if (msg.value > 0) {
           (bool ok,) = msg.sender.call{value: msg.value}("");
           require(ok, "REFUND_FAILED");
       }
       return;
   }
   ```
2. **Validate `msg.value` sufficiency**: When `amountToMint > 0`, require `msg.value` to be at least the minimum fee required by both the bridge and the messaging contract before proceeding.
3. **Refund excess**: After the two `{value: ...}` calls, refund any remaining ETH to `msg.sender`.

---

### Proof of Concept

1. L2 `RewardSupplier` sends a mint request; `requiredMinting()` returns `amountToMint > 0`.
2. Keeper A calls `tick{value: 1 ether}()` and successfully processes the mint.
3. Keeper B's transaction (also `tick{value: 1 ether}()`) was submitted simultaneously and lands in the next block. Now `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` branch is skipped. Keeper B's 1 ETH is absorbed into `RewardSupplier`'s balance.
5. There is no withdrawal function. The 1 ETH is permanently locked.
6. Repeat across many keepers/epochs: the contract accumulates ETH that is irrecoverable, matching the "permanent freezing of funds" impact.

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
