### Title
Uninitialized `ONE` Static in `delegated_u256` Causes `is_one()` / `one()` to Return Wrong Values in Proving Path — (`supporting_crates/delegated_u256/src/arithmetic.rs`)

---

### Summary

`delegated_u256::init()` is never called in the production bootloader execution path. This leaves the `static mut ONE: MaybeUninit<DelegatedU256>` global zero-initialized by BSS instead of being written with `[1, 0, 0, 0]`. Every call to `DelegatedU256::one()`, `is_one()`, and `write_one()` on the riscv32 proving target silently operates against an all-zero value, producing wrong arithmetic results and a sequencer/prover divergence.

---

### Finding Description

`supporting_crates/delegated_u256/src/arithmetic.rs` declares two `static mut MaybeUninit<DelegatedU256>` globals that must be explicitly initialized before use:

```rust
pub static mut ZERO: MaybeUninit<DelegatedU256> = MaybeUninit::uninit();
pub static mut ONE:  MaybeUninit<DelegatedU256> = MaybeUninit::uninit();

pub(super) fn init() {
    unsafe {
        ZERO.write(DelegatedU256::ZERO);   // [0,0,0,0]
        ONE.write(DelegatedU256::ONE);     // [1,0,0,0]
    }
}
``` [1](#0-0) 

The public entry point is `delegated_u256::init()` in `lib.rs`:

```rust
pub fn init() {
    arithmetic::init();
}
``` [2](#0-1) 

A grep across the entire production bootloader path (`basic_system/`, `basic_bootloader/`, `proof_running_system/`) finds **zero calls** to `delegated_u256::init()`. The only call site is inside a `#[test]` block in `supporting_crates/u256/src/lib.rs`: [3](#0-2) 

The modexp delegation path does call its own *local* `self::u256::init()`: [4](#0-3) 

But that initializes a completely separate set of statics inside `basic_system/src/system_functions/modexp/delegation/u256.rs` — it does **not** initialize the `ONE` static in `supporting_crates/delegated_u256/src/arithmetic.rs`.

Because `static mut MaybeUninit<…>` variables with no explicit initializer land in BSS, the runtime zero-fills them. The result:

| Static | Expected after `init()` | Actual (BSS, no `init()`) | Correct? |
|--------|------------------------|--------------------------|----------|
| `ZERO` | `[0,0,0,0]` | `[0,0,0,0]` | ✓ |
| `ONE`  | `[1,0,0,0]` | `[0,0,0,0]` | ✗ |

Every method that references `ONE` is therefore broken on riscv32 with the `delegation` feature:

```rust
pub fn is_one(&self) -> bool {
    let eq = unsafe {
        let src = copy_if_needed(self as *const Self);
        bigint_op_delegation::<EQ_OP_BIT_IDX>(src.cast_mut(), ONE.as_ptr()) // ← compares against 0
    };
    eq != 0
}

pub fn one() -> Self {
    unsafe { copy_from_operand(ONE.as_ptr()) }  // ← returns 0
}

pub fn write_one(&mut self) {
    unsafe { bigint_op_delegation::<MEMCOPY_BIT_IDX>(self as *mut Self, ONE.as_ptr()) } // ← writes 0
}
``` [5](#0-4) [6](#0-5) 

The `U256` wrapper on riscv32 delegates directly to these methods: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

On the riscv32 proving target:

- `U256::one()` returns `U256::ZERO` — any code constructing the value 1 via this function gets 0.
- `U256::is_one(x)` returns `true` when `x == 0` and `false` when `x == 1` — inverted semantics.
- `U256::write_one(dst)` writes 0 into `dst`.

The sequencer runs on x86 and uses the `naive` U256 implementation, which has no such statics and always returns correct values. This creates a **sequencer/prover divergence**: the sequencer produces a valid state root, but the prover computes different arithmetic results, either generating a proof for a wrong state or failing to generate a proof at all. Either outcome breaks the integrity of the ZK rollup.

---

### Likelihood Explanation

The bug is triggered by any transaction that causes the proving path to call `U256::one()` or `U256::is_one()` on riscv32. These are general-purpose arithmetic helpers used throughout the EVM interpreter and system layer. No special privileges or unusual inputs are required — ordinary EVM execution is sufficient to reach the affected code.

---

### Recommendation

Add a single call to `delegated_u256::init()` at the very start of the proving bootloader, before any arithmetic is performed. The natural location is `proof_running_system/src/system/bootloader.rs` inside `run_proving`, immediately after `init_allocator`: [9](#0-8) 

```rust
unsafe { init_allocator(heap_start, heap_end); }
delegated_u256::init();   // ← add this
```

Alternatively, guard every use of `ONE` and `ZERO` with a `Once`-based lazy initializer, or replace the `MaybeUninit` statics with directly initialized `static` values where the target architecture permits it.

---

### Proof of Concept

1. Deploy any contract that performs EVM arithmetic relying on the value `1` constructed via `U256::one()` (e.g., an increment loop).
2. Submit a transaction calling that contract.
3. The sequencer executes correctly and records the expected state root.
4. The prover (riscv32, `delegation` feature) executes the same transaction; `U256::one()` returns `0` because `ONE` was never initialized, producing a different intermediate value.
5. The prover either panics on an internal assertion or produces a proof attesting to a wrong state root, breaking the soundness guarantee of the rollup. [1](#0-0) [2](#0-1) [9](#0-8)
