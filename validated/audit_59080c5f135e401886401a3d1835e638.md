### Title
Presign Participant Set Not Validated Against Triple Participants, Allowing Corrupted Presignature Outputs - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary
The OT-based ECDSA `presign()` function enforces threshold consistency against the Beaver triples but explicitly omits verifying that the caller-supplied `participants` set matches the participant set embedded in those triples (`TriplePub.participants`). A malicious coordinator or library caller can supply a mismatched participant list, causing all honest parties to compute Lagrange coefficients over the wrong set and produce corrupted, inconsistent presignature outputs that honest parties accept as protocol-complete.

### Finding Description
The vulnerability class from the external report is **inconsistent invariant enforcement across code paths**: one code path enforces a critical state invariant while a parallel path silently omits it, allowing the invariant to be violated. In MEME404, token-transfer enforced the "no dual-tier" invariant while NFT-transfer did not. The analog here is:

- **Triple generation path** (`generate_triple` → `validate_triple_inputs`): creates a `TriplePub` that permanently records the exact `participants` set and `threshold` used during generation.
- **Presign path** (`presign` → `do_presign`): checks that the caller-supplied `threshold` matches `triple.threshold`, but **explicitly omits** checking that the caller-supplied `participants` matches `triple.participants`.

The omission is acknowledged in a code comment:

```
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [1](#0-0) 

The threshold check that IS performed:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [2](#0-1) 

The participant set stored in `TriplePub` during generation:

```rust
ret.push((
    TripleShare { a: *a_i, b: *b_i, c: *c_i },
    TriplePub { big_a, big_b, big_c, participants: participants.clone().into(), threshold },
));
``` [3](#0-2) 

The caller-supplied `participants` list is passed directly into `do_presign` without any cross-check against `args.triple0.1.participants` or `args.triple1.1.participants`:

```rust
let participants = ParticipantList::new(participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
// ...
let fut = do_presign(ctx.shared_channel(), participants, me, args);
``` [4](#0-3) 

Inside `do_presign`, Lagrange coefficients are computed over the caller-supplied `participants` list. If this list differs from the triple's participant set, every party computes incorrect Lagrange weights, producing presignature components that are algebraically inconsistent with the secret shares derived from the triples.

### Impact Explanation
**High — Corruption of presign outputs so honest parties accept inconsistent cryptographic outputs.**

When a malicious coordinator supplies a `participants` list that differs from the triple's embedded participant set (e.g., substituting one participant for another while keeping the count and threshold equal), all honest parties silently proceed through the full presign protocol using wrong Lagrange coefficients. The resulting `PresignOutput` values (`big_r`, `c`, `e`, `alpha`, `beta`) are algebraically inconsistent: they do not correspond to a valid threshold sharing of the nonce. Honest parties have no way to detect this at presign time because no cross-check against `TriplePub.participants` is performed. The corrupted presignature is then passed to `sign()`, which produces an invalid ECDSA signature, permanently denying signing for that presignature. Since presignatures are one-time-use, the consumed triple and presignature are irrecoverably wasted.

In a more targeted split-view scenario (coordinator presents participant list `{A,B,C}` to party A and `{A,B,D}` to party B), the parties compute different Lagrange coefficients over the same underlying triple shares, producing signature shares with algebraic relationships that can be exploited analogously to the nonce-reuse attacks documented for the robust ECDSA scheme. [5](#0-4) 

### Likelihood Explanation
**Medium.** The attacker must be a malicious coordinator or a library caller who controls the `participants` argument passed to each party's `presign()` invocation. No privileged key material is required. In a realistic MPC deployment, the coordinator role is a documented trust boundary and is reachable by a malicious participant who volunteers to coordinate. The attack requires only crafting a `participants` slice that differs from the triple's embedded participant set while keeping the threshold equal — a trivially constructable input.

### Recommendation
Add an explicit check in `presign()` that the caller-supplied `participants` set equals `args.triple0.1.participants` and `args.triple1.1.participants` (as sets, not just by length/threshold). Remove the comment that rationalizes the omission. The invariant "presign participants must exactly match triple participants" is already enforced structurally in triple generation via `TriplePub.participants`; it must be re-enforced at the presign entry point to close the inconsistent-path gap.

```rust
// After building the ParticipantList from the caller-supplied slice:
if participants != args.triple0.1.participants || participants != args.triple1.1.participants {
    return Err(InitializationError::BadParameters(
        "Presign participants must exactly match the participants of both triples".to_string(),
    ));
}
```

### Proof of Concept

1. Party set `{A, B, C}` runs `generate_triple` with threshold 3. Each party receives a `TripleShare`; the `TriplePub` records `participants = {A, B, C}`.
2. A malicious coordinator instructs party A to call `presign(participants=[A,B,D], me=A, args=...)` and party B to call `presign(participants=[A,B,D], me=B, args=...)`, substituting D for C.
3. The threshold check `args.threshold == args.triple0.1.threshold` passes (both are 3).
4. The participant-set check against `args.triple0.1.participants` is absent; `do_presign` proceeds with `{A,B,D}`.
5. Lagrange coefficients are computed for `{A,B,D}` instead of `{A,B,C}`. Party C's triple share contribution is excluded; party D (who has no triple share) contributes nothing or garbage.
6. The resulting `PresignOutput` is algebraically invalid. Honest parties A and B accept it as protocol-complete.
7. When `sign()` is called with this presignature, the produced ECDSA signature fails verification — permanently denying signing for this presignature and wasting the consumed triple.
