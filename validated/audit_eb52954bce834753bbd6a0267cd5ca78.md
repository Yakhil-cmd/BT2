### Title
`gas_per_pubdata_limit` Not Enforced for ZK L2 Transactions — Users Pay More Pubdata Fees Than Stated Maximum - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

ZK (ABI-encoded) L2 transactions carry a `gas_per_pubdata_limit` field explicitly documented as "the maximum gas the user is willing to pay for a byte of pubdata." This field is parsed but **never validated** against the actual block-level `pubdata_price` during L2 ZK transaction validation. If the operator increases `pubdata_price` between when a user signs a transaction and when it is included in a block, the user is charged more for pubdata than their stated maximum, with no revert or protection.

### Finding Description

In `validate_and_compute_fee_for_transaction` (the ZK L2 transaction validation path), `native_per_pubdata` is derived directly from block-level oracle variables:

```rust
let pubdata_price = system.get_pubdata_price();
let native_price = system.get_native_price();
// ...
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The transaction's `gas_per_pubdata_limit` field is accessible via `transaction.gas_per_pubdata_limit()` and is documented as the user's maximum: [2](#0-1) [3](#0-2) 

However, a `grep` of all bootloader Rust files confirms `gas_per_pubdata_limit` is only consumed in `process_l1_transaction.rs` (for L1→L2 transactions). It is **never read or checked** inside `validate_and_compute_fee_for_transaction` for L2 ZK transactions. [4](#0-3) 

The effective gas cost per pubdata byte is `pubdata_price / gas_price`. The missing check is:

```
pubdata_price / gas_price  <=  gas_per_pubdata_limit
```

Without this check, the user's signed cap is silently ignored.

The pubdata fee flows into the refund calculation via `native_per_pubdata` in `before_refund`: [5](#0-4) 

A higher `native_per_pubdata` consumes more native resources for pubdata, reducing the gas refund, so the user pays more ETH than their `gas_per_pubdata_limit` implied.

### Impact Explanation

A ZK L2 transaction sender is charged more ETH than their signed `gas_per_pubdata_limit` permits. The user pre-pays `gas_price × gas_limit` ETH; the refund is reduced by the pubdata cost. If `pubdata_price` rises after signing, the user's refund shrinks below what `gas_per_pubdata_limit` would have guaranteed, constituting a direct, quantifiable financial loss. The user cannot protect themselves because the field they signed to cap their exposure is not enforced.

### Likelihood Explanation

The operator/sequencer sets `pubdata_price` as a block-level variable (derived from L1 gas prices). L1 gas prices are volatile and can spike significantly between the time a user signs a ZK L2 transaction and when it is included. Any pending ZK L2 transaction in the mempool is exposed. No attacker capability beyond submitting a normal ZK L2 transaction is required — the loss occurs passively whenever `pubdata_price` increases.

### Recommendation

In `validate_and_compute_fee_for_transaction`, after computing `native_per_pubdata` and `native_per_gas`, add a check that the effective pubdata gas price does not exceed the user's stated limit:

```rust
// Enforce gas_per_pubdata_limit for ZK L2 transactions
let gas_per_pubdata_limit = transaction.gas_per_pubdata_limit();
if !gas_per_pubdata_limit.is_zero() && !Config::SIMULATION {
    // Effective gas per pubdata = native_per_pubdata / native_per_gas
    // Equivalently: pubdata_price / gas_price > gas_per_pubdata_limit
    let effective_gas_per_pubdata = if native_per_gas == 0 {
        U256::ZERO
    } else {
        U256::from(native_per_pubdata) / U256::from(native_per_gas)
    };
    require!(
        effective_gas_per_pubdata <= gas_per_pubdata_limit,
        TxError::Validation(InvalidTransaction::PubdataPriceTooHigh),
        system
    )?;
}
```

This mirrors the existing enforcement for L1 transactions and gives the `gas_per_pubdata_limit` field its documented semantics.

### Proof of Concept

1. User signs a ZK L2 transaction with `gas_per_pubdata_limit = 100`, `gas_limit = 1_000_000`, `max_fee_per_gas = 1000`. User expects to pay at most 100 gas per pubdata byte.
2. At signing time, `pubdata_price / gas_price = 50` — within the user's limit.
3. Before the transaction is included, the operator updates the block's `pubdata_price` such that `pubdata_price / gas_price = 500`.
4. `validate_and_compute_fee_for_transaction` computes `native_per_pubdata` from the new `pubdata_price` with no check against `gas_per_pubdata_limit`.
5. The transaction executes, consuming e.g. 1000 bytes of pubdata. The user is charged 500 × 1000 = 500,000 gas-equivalent for pubdata instead of the 100 × 1000 = 100,000 they expected — a 5× overcharge — with no revert. [1](#0-0) [6](#0-5)
