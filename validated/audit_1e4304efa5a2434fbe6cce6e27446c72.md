### Title
Incomplete `REWARD_ADDRESS_IS_TOKEN` Guard in BTC Pool `change_reward_address` Allows Permanent Freezing of Unclaimed STRK Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `change_reward_address` function in `pool.cairo` only validates that the new reward address is not the pool's own delegation token. For BTC delegation pools, this means the STRK token address — the token in which all pool rewards are paid — is **not blocked**. A pool member can set their reward address to the STRK token contract, causing all future STRK reward claims to be permanently locked inside the STRK token contract.

---

### Finding Description

**Root cause — asymmetric validation between staking and pool contracts:**

In `staking.cairo`, `change_reward_address` uses a comprehensive check:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

`does_token_exist` returns `true` for STRK **and** every registered BTC token, so no registered token address can become a staker's reward address. [1](#0-0) [2](#0-1) 

In `pool.cairo`, `change_reward_address` uses a narrower check:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

`token_dispatcher` holds the pool's **delegation token** (e.g., BTC for a BTC pool). For a BTC pool this check only blocks the BTC token address; it does **not** block the STRK token address. [3](#0-2) 

**Reward payment is always in STRK, regardless of pool type:**

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [4](#0-3) 

Because `claim_rewards` always transfers STRK to `reward_address`, setting `reward_address` to the STRK token contract causes the STRK to be sent into the token contract itself, where it is permanently inaccessible.

**The spec explicitly requires this to be blocked:**

The spec lists `REWARD_ADDRESS_IS_TOKEN` as a mandatory error for `change_reward_address` in the pool contract, confirming the protocol's intent to prevent this state. [5](#0-4) 

---

### Impact Explanation

A pool member in a BTC delegation pool can permanently lock their own unclaimed STRK yield by setting their reward address to the STRK token contract. Once `claim_rewards` executes, the STRK is transferred into the token contract and is unrecoverable. This constitutes **permanent freezing of unclaimed yield**, which is within the allowed High impact scope.

---

### Likelihood Explanation

**Low.** The attack requires:
1. A BTC delegation pool to be deployed and active.
2. A pool member to call `change_reward_address` with the STRK token address.

This is most likely to occur accidentally (a user confuses the reward token address with a valid recipient), but a griefing actor could also deliberately trigger it to permanently destroy their own yield. No privileged access is required; any pool member can execute this.

---

### Recommendation

In `pool.cairo` `change_reward_address`, mirror the staking contract's comprehensive check. The simplest fix is to also assert that the new reward address is not the STRK token address:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
assert!(
    reward_address != STRK_TOKEN_ADDRESS,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

A more robust fix would be to expose a `does_token_exist` query on the staking contract and call it from the pool, so that any future token additions are automatically covered.

---

### Proof of Concept

1. A BTC delegation pool exists; `pool.token_dispatcher` points to the BTC token contract.
2. A pool member calls `pool.change_reward_address(STRK_TOKEN_ADDRESS)`.
3. The guard `self.token_dispatcher.contract_address.read() != reward_address` evaluates to `BTC_ADDRESS != STRK_TOKEN_ADDRESS` → **passes**.
4. `pool_member_info.reward_address` is now set to `STRK_TOKEN_ADDRESS`.
5. The pool member (or anyone) calls `pool.claim_rewards(pool_member)`.
6. `claim_rewards` executes `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
7. The STRK rewards are transferred into the STRK token contract and are permanently locked — the pool member's unclaimed yield is frozen forever.

        fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
            assert!(
                self.token_dispatcher.contract_address.read() != reward_address,
                "{}",
                GenericError::REWARD_ADDRESS_IS_TOKEN,
            );
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let old_address = pool_member_info.reward_address;

            // Update reward_address and commit to storage.
            pool_member_info.reward_address = reward_address;
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardAddressChanged {
                        pool_member, new_address: reward_address, old_address,
                    },
                );
        }
