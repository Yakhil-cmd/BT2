Audit Report

## Title
Byzantine Peer Memory Exhaustion via Unbounded Concurrent Stream Allocation in `handle_bi_stream` — (File: `rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
A Byzantine subnet peer can open up to 1,000 concurrent bidirectional QUIC streams and send valid artifact-fetch requests on each. After each handler completes, `to_response_bytes` fully materializes the response body into a heap allocation bounded only by `MAX_MESSAGE_SIZE_BYTES = 128 MiB`, with no cancellation guard and no per-connection concurrency limit. If the peer withholds flow-control credits, all 1,000 allocations remain live simultaneously, producing up to 128 GiB of heap pressure and causing an OOM crash of the replica.

## Finding Description
In `handle_bi_stream` (`rs/p2p/quic_transport/src/request_handler.rs`, L128–165), `stopped_fut` is created from `send_stream.stopped()` and polled exclusively inside the `tokio::select!` at L129–136. Once the handler arm (`svc`) wins, `stopped_fut` is dropped. No cancellation check exists between L136 and L155:

```rust
// L128-136: stopped_fut only lives inside this select!
let stopped_fut = send_stream.stopped();
let response = tokio::select! {
    response = svc => response.expect("Infallible"),
    stopped = stopped_fut => { return Ok(...); }
};
// stopped_fut is gone here
let response_bytes = to_response_bytes(response).await?;   // L149: allocates up to 128 MiB
send_stream.write_all(&response_bytes).await?;             // L150-155: blocks if peer stops reading
```

`to_response_bytes` (L210–214) calls `axum::body::to_bytes(body, MAX_MESSAGE_SIZE_BYTES)`, which collects the entire streaming body into a contiguous `Vec<u8>` bounded by `MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024` (`rs/p2p/quic_transport/src/lib.rs`, L74). This allocation is unconditional and has no interleaved cancellation point.

`start_stream_acceptor` (L43–110) spawns each accepted stream as an independent tokio task into a `JoinSet` with no semaphore or per-connection concurrency cap (L79–90). The QUIC configuration sets `MAX_CONCURRENT_BIDI_STREAMS = 1_000` (`rs/p2p/quic_transport/src/connection_manager.rs`, L74), so up to 1,000 tasks can be live simultaneously per connection. The developer comment at L52–56 acknowledges this: *"The extreme result of a slow handler is that the stream limit will be reached, hence having buffered up to the stream limit number of messages/requests."*

Real handlers produce large responses: `rpc_handler` in `rs/p2p/artifact_downloader/src/fetch_artifact/download.rs` serializes full artifacts into `Bytes`; `state_sync_chunk_handler` in `rs/p2p/state_sync_manager/src/routes/chunk.rs` returns chunks up to `MAX_CHUNK_SIZE = 8 MiB`. With 1,000 concurrent streams at 8 MiB each, simultaneous heap usage reaches 8 GiB before any byte is written to the wire.

The `STREAM_RECEIVE_WINDOW = 4 MiB` (L73) does not prevent the allocation — it only blocks `write_all` after the full buffer is already allocated. The `IDLE_TIMEOUT = 5 s` (L81) does not help because the attacker keeps streams active by sending valid requests; the timeout only fires on a fully idle connection.

## Impact Explanation
A single Byzantine subnet peer (below the consensus fault threshold) can crash a replica node by exhausting its heap memory. This constitutes an **application/platform-level DoS and replica crash** — a High severity impact per the allowed ICP bounty impact classes ($2,000–$10,000). The crash disrupts the affected node's participation in consensus and subnet operation without requiring any threshold majority.

## Likelihood Explanation
The attacker must be an authenticated subnet peer (Byzantine node below the fault threshold), which is explicitly within the stated attack surface. No timing precision is required: the attacker opens 1,000 streams, sends valid requests, and stops reading. Standard QUIC client libraries support opening many streams concurrently. The attack is repeatable and deterministic — it does not depend on race conditions or external state.

## Recommendation
1. **Re-arm cancellation around `to_response_bytes`**: Wrap the buffering call in a new `select!` that re-polls `send_stream.stopped()`, aborting if the peer sends STOP_SENDING before the allocation completes.
2. **Stream the response incrementally**: Replace the full-body buffer with chunked writes so QUIC flow control (`STREAM_RECEIVE_WINDOW = 4 MiB`) naturally caps per-stream in-flight memory.
3. **Bound concurrent tasks per connection**: Introduce a `tokio::sync::Semaphore` in `start_stream_acceptor` to cap the number of simultaneously executing handler tasks per peer connection, decoupling the QUIC stream limit from the application-level memory budget.

## Proof of Concept
1. Byzantine peer connects (authenticated via TLS as a subnet node).
2. Opens 1,000 bidirectional QUIC streams simultaneously.
3. On each stream, sends a valid serialized `/{artifact}/rpc` request for a known large artifact (e.g., a state sync chunk).
4. Does **not** send STOP_SENDING during handler execution, so `stopped_fut` does not fire inside the `select!`.
5. Handler completes → `to_response_bytes` allocates up to 128 MiB (or ~8 MiB for state sync chunks) per task.
6. Peer stops reading (exhausts `STREAM_RECEIVE_WINDOW = 4 MiB`) → `write_all` blocks for all 1,000 tasks.
7. All 1,000 allocations are live simultaneously → 8–128 GiB heap usage → replica OOM or OS kill.

A deterministic integration test can reproduce this by spawning a mock QUIC peer that opens 1,000 streams, sends valid requests, and never reads responses, then asserting that the serving replica process terminates with an OOM signal or that memory usage exceeds a threshold.
