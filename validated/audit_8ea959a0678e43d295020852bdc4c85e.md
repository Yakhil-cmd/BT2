### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Bin Additional Fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary
The factory enforces `maxAdminSpreadFeeE6` as a ceiling on pool-level admin spread fees, but `setPoolBinAdditionalFees` applies no analogous cap on per-bin additional fees (`addFeeBuyE6` / `addFeeSellE6`). A pool admin can set bin fees up to `uint16.max` (65 535 = 6.5535 %) on every bin, silently exceeding the factory-owner-intended fee ceiling and extracting excess fees from traders.

---

### Finding Description

`setPoolAdminFees` correctly enforces the factory cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` passes values straight through to the pool with **no cap check**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

Inside the pool, `setBinAdditionalFees` only validates the bin index, not the fee magnitude:

```solidity
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap, the bin additional fee is added directly on top of the base spread fee:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears in `getSellAndBuyPrices`:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [5](#0-4) 

The spread surplus (which includes the bin additional fee component) is split between protocol and admin in `collectFees`:

```solidity
uint256 surplus0Scaled =
    balance0() * TOKEN_0_SCALE_MULTIPLIER - uint256(binTotals.scaledToken0) - notionalFee0AmountScaled;
...
uint256 spreadFee0ToAdminScaled = spreadSumE6 == 0 ? 0 : (surplus0Scaled * adminSpreadFeeE6_) / spreadSumE6;
``` [6](#0-5) 

The bin additional fee accumulates in the surplus and is distributed to the admin at collection time, bypassing the factory-level cap entirely.

---

### Impact Explanation

Traders pay fees above the factory-owner-intended ceiling. With `maxAdminSpreadFeeE6 = 10 000` (1 %) and `addFeeBuyE6 = 65 535` (6.5535 %), the effective per-swap fee reaches 7.5535 %, a 7.5× overcharge. The excess accrues to the pool admin via the spread surplus in `collectFees`. This is a direct loss of trader principal with no on-chain recourse once the swap executes.

---

### Likelihood Explanation

The pool admin is semi-trusted. No timelock, no factory approval, and no additional privilege is required beyond the `onlyPoolAdmin` check. The admin can call `setPoolBinAdditionalFees` immediately after pool creation and apply the maximum `uint16` value to every bin. The factory owner has no mechanism to detect or revert this without pausing the pool.

---

### Recommendation

**Short term:** Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

**Long term:** Introduce a dedicated `maxAdminBinFeeE6` state variable (settable by the factory owner via `setFeeCaps`) so the factory owner can independently cap bin-level fees, consistent with the existing cap architecture for pool-level fees.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps` with `newMaxAdminSpreadFeeE6 = 10_000` (1 %).
2. Pool admin calls `setPoolAdminFees(pool, 10_000, 0)` — correctly capped, succeeds.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — no cap check, succeeds.
4. Trader calls `swap` through bin 0. Effective buy fee = 1 % (spread) + 6.5535 % (bin additional) = **7.5535 %**, far exceeding the 1 % cap the factory owner intended.
5. Factory owner calls `collectPoolFees(pool)`. The excess spread surplus is distributed to the pool admin's `adminFeeDestination`, extracting funds that should never have been charged.
