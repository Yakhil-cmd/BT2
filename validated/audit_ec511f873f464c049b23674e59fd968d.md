### Title
Unvalidated attacker-controlled `ph` field in blockchain notification causes counter-offer to be directed to attacker's address — (`packages/gui/src/hooks/useBlockchainNotifications.tsx`, `packages/gui/src/components/offers2/OfferIncomingTable.tsx`)

### Summary
An on-chain attacker can craft a blockchain notification whose JSON payload sets `t=1` and `ph` to the attacker's own puzzle hash (as a 64-char hex string or a bech32m address). `prepareNotifications` stores this value without any validation. When the victim clicks "Counter" on the notification, `handleCounterOffer` calls `toBech32m(attackerPuzzleHash, currencyCode)`, producing a valid bech32m address for the attacker's wallet, which is then used as the counter-offer destination.

### Finding Description
In `useBlockchainNotifications.tsx` lines 99–110, the `ph` field from the on-chain notification payload is extracted and stored as `puzzleHash` with no validation: [5](#0-4) 

In `OfferIncomingTable.tsx` line 190, this value is passed directly to `toBech32m`: [6](#0-5) 

`toBech32m` accepts any valid hex string or any string already prefixed with the currency code, so an attacker-supplied puzzle hash passes through cleanly and produces a valid destination address.

### Impact Explanation
The counter-offer destination address is derived entirely from attacker-supplied on-chain data. The victim's counter-offer is directed to the attacker's wallet. If the victim creates and submits the counter-offer, funds flow to the attacker. This meets the High impact criterion: "unsafe trust of notification state that causes a user to send to the wrong destination."

### Likelihood Explanation
Sending an on-chain notification to any puzzle hash is a standard, low-cost Chia operation. No special privileges are required. The victim only needs to have a synced wallet and click "Counter" on the notification — a natural action when receiving an offer notification.

### Recommendation
Validate `puzzleHash` in `prepareNotifications` before storing it:
1. Confirm it is exactly 64 hex characters (32 bytes) after stripping any `0x` prefix.
2. Reject (return `null`) if the value fails this check.
3. Optionally, verify the puzzle hash belongs to the current wallet's known addresses before enabling the counter-offer button.

### Proof of Concept
1. Attacker sends an on-chain notification to the victim's puzzle hash with payload (hex-encoded): `{"t":1,"d":{"u":"<valid_offer_url>","ph":"<attacker_64_hex_puzzle_hash>"}}`
2. Victim's wallet receives and displays the notification.
3. Victim clicks "Counter" in `OfferIncomingTable`.
4. `handleCounterOffer` calls `toBech32m("<attacker_64_hex_puzzle_hash>", "xch")` → produces `xch1<attacker_address>`.
5. Offer builder opens with `address = xch1<attacker_address>` pre-filled as the counter-offer destination.
6. Victim fills in counter-offer terms and submits; funds are directed to the attacker.

```typescript
export default function toBech32m(value: string, prefix: string): string {
  if (value.startsWith(prefix)) {
    return value;
  }

  const pureHash = removePrefix(value, '0x');
  const hexBytes = Buffer.from(pureHash, 'hex');

  if (pureHash.length / 2 !== hexBytes.length) {
    throw new Error('Invalid hex string');
  }

  const words = bech32m.toWords(hexBytes);

  return bech32m.encode(prefix, words);
}
```
