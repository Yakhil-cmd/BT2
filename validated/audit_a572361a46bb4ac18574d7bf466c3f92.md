### Title
Cross-COA ERC-1271 Signature Replay via Missing `evmAddress` Binding in `validateCOAOwnershipProof` — (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

`validateCOAOwnershipProof` in the EVM system contract does not enforce that `signedData` encodes the target `evmAddress`. Because the `COAOwnershipProof`'s `CapabilityPath` field is not covered by the cryptographic signatures (signatures are only over `signedData`), an attacker who observes a valid proof for COA-A can silently swap the `CapabilityPath` to point to COA-B — another COA owned by the same Cadence account — and replay the proof against COA-B's ERC-1271 `isValidSignature` endpoint. The result is that EVM contracts are made to believe COA-B has authorized a message it never signed.

---

### Finding Description

`validateCOAOwnershipProof` performs two checks:

1. The cryptographic signatures in the proof are valid over `signedData`.
2. The COA resource at the given `CapabilityPath` has an EVM address matching the `evmAddress` argument. [1](#0-0) 

The function explicitly documents the gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [2](#0-1) 

The `COAOwnershipProof` struct carries a `CapabilityPath` field that is **not** part of the signed payload — signatures are only over `signedData`: [3](#0-2) 

Because `CapabilityPath` is unsigned, an attacker can take a proof `{Address: A, CapabilityPath: /public/coaA, Signatures: [sig_over_H]}` and produce `{Address: A, CapabilityPath: /public/coaB, Signatures: [sig_over_H]}`. The signatures remain valid over `H`, and the `evmAddress` check passes because the COA at `/public/coaB` legitimately has COA-B's address — which is exactly `address(this)` when `isValidSignature` is called on COA-B: [4](#0-3) 

The precompile decodes the proof and passes `address(this)` as `evmAddress`, so the path-swap is invisible to the validator: [5](#0-4) 

The `evmAddress` check at lines 1095–1109 only verifies that the COA stored at the (attacker-controlled) `CapabilityPath` matches the calling contract's address — it never verifies that the signer intended to authorize *that* COA: [6](#0-5) 

---

### Impact Explanation

Any EVM contract that relies on ERC-1271 (`isValidSignature`) for authorization of financial operations — token transfers, NFT approvals, DeFi permit flows — can be deceived into accepting COA-B's "signature" on a message that was actually signed for COA-A. The attacker does not need the victim's private key; they only need a previously broadcast proof. This constitutes **direct loss of funds** for any victim whose COA-B is used as an authorization gate in such a contract.

---

### Likelihood Explanation

Two preconditions must hold:

1. The victim Cadence account owns **two or more COAs** at distinct public paths. The code comment acknowledges this is currently uncommon ("Flow wallets historically create at most one COA per account"), but it is not prevented by the protocol, and multi-COA patterns are expected to grow as the ecosystem matures.
2. The attacker must have observed a prior on-chain transaction in which the victim supplied a `COAOwnershipProof` for COA-A (e.g., an ERC-1271 interaction with a DeFi protocol). All proof fields — including the raw signatures — are visible in calldata.

No privileged access, leaked keys, or social engineering is required. The attacker's entry point is a standard EVM transaction calling `isValidSignature` on COA-B with the modified proof.

---

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. Concretely, before calling `keyList.verify`, assert that the first 20 bytes of `signedData` equal `evmAddress`. Alternatively, include `CapabilityPath` in the signed payload so that swapping the path invalidates the signatures. The existing comment already identifies the correct fix direction; it should be promoted from advisory to enforced.

---

### Proof of Concept

```
Setup:
  Victim Cadence account V owns:
    COA-A  →  EVM address 0xAAAA  stored at /public/coaA
    COA-B  →  EVM address 0xBBBB  stored at /public/coaB

Step 1 – Victim legitimately authorizes hash H for COA-A:
  Victim submits EVM tx to DeFi contract D.
  D calls: COA-A.isValidSignature(H, proof_A)
    proof_A = { Address: V, CapabilityPath: /public/coaA,
                KeyIndices: [0], Signatures: [sig_V_over_H] }
  validateCOAOwnershipProof passes → D records "COA-A approved H".

Step 2 – Attacker extracts proof_A from on-chain calldata.

Step 3 – Attacker constructs proof_B (only CapabilityPath changed):
  proof_B = { Address: V, CapabilityPath: /public/coaB,
              KeyIndices: [0], Signatures: [sig_V_over_H] }

Step 4 – Attacker calls D with proof_B, claiming COA-B approved H:
  D calls: COA-B.isValidSignature(H, proof_B)
    → verifyCOAOwnershipProof(0xBBBB, H, proof_B)
    → validateCOAOwnershipProof(V, /public/coaB, H, [0], [sig_V_over_H], 0xBBBB)
      ✓ sig_V_over_H is valid over H
      ✓ COA at /public/coaB has address 0xBBBB == evmAddress
    → returns isValid: true

Step 5 – D executes the authorized action on behalf of COA-B (e.g., token transfer),
         draining COA-B's assets without COA-B's owner ever signing for COA-B.
```
