Looking at the codebase, I need to find an analog to the external bug: a state check enforced in one code path but missing in another, allowing bypass of a restriction with economic impact.

The key finding is in `MetricOmmPool.sol`: [1](#0-0) [2](#0-1) [3](#0-2) 

`swap()` carries `whenNotPaused`; `addLiquidity()` does not.

---

### Title
Paused pool still accepts `addLiquidity`, allowing new LP principal to enter a compromised pool — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary
The `whenNotPaused` guard is applied to `swap()` but is absent from `addLiquidity()`. When a pool is paused at any level (including the protocol-level pause, level 2), users can still deposit tokens into the pool. Those tokens are immediately exposed to swaps the moment the pool is unpaused, even if the underlying issue that triggered the pause has not been fully resolved.

### Finding Description
`MetricOmmPool` defines three pause levels stored in `pauseLevel`:
- `0` — active
- `1` — admin-paused (admin can pause/unpause)
- `2` — protocol-paused (only protocol owner can unpause) [4](#0-3) 

The `whenNotPaused` modifier enforces `pauseLevel == 0`: [5](#0-4) 

`swap()` is correctly gated: [2](#0-1) 

But `addLiquidity()` has no such gate: [3](#0-2) 

`addLiquidity` uses a pull-callback pattern that transfers real tokens from the caller into the pool. Any user can call it on a paused pool and have their principal locked inside a pool that was halted precisely because it was deemed unsafe.

The factory's `protocolPausePool` is the most severe form of pause — it can only be reversed by the protocol owner — and is the expected response to a critical incident such as a price-provider failure or an active exploit: [6](#0-5) 

Despite this, `addLiquidity` remains callable throughout the pause, depositing new LP principal into the compromised pool.

### Impact Explanation
When the pool is unpaused, all liquidity added during the pause is immediately available for swaps. If the root cause of the pause (e.g., a stale or manipulated price provider) has not been fully remediated, swaps at bad prices can drain the newly deposited LP funds. This is a direct loss of user principal — the exact impact class the pause mechanism is designed to prevent.

### Likelihood Explanation
Medium. A protocol-level pause is an emergency action. During an incident, unsophisticated LPs or automated liquidity managers may not monitor pause state and could add liquidity while the pool is halted. The window between pause and unpause can be hours or days, giving ample time for deposits to accumulate.

### Recommendation
Add `whenNotPaused` to `addLiquidity()`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

`removeLiquidity` should remain ungated so existing LPs can always exit.

### Proof of Concept
1. Pool is operating normally (`pauseLevel = 0`).
2. A price-provider issue is detected; protocol calls `protocolPausePool(pool)` → `pauseLevel = 2`. Swaps revert.
3. Alice calls `addLiquidity(...)` on the paused pool. No revert — `addLiquidity` has no pause check. Alice's tokens are transferred into the pool and she receives shares.
4. Protocol investigates, believes the issue is resolved, calls `protocolUnpausePool(pool)` → `pauseLevel = 1` (admin-paused). Admin then calls `unpausePool(pool)` → `pauseLevel = 0`.
5. If the price-provider issue persists or recurs, the first swap after unpause executes at a bad price against Alice's freshly deposited liquidity, draining her principal. [3](#0-2) [7](#0-6)
