### Title
Silo Address Whitelist Bypassed in `receive_erc20_tokens` When No Fallback Address Is Configured - (File: `engine/src/engine.rs`)

### Summary

Aurora Engine's Silo mode enforces an Address whitelist (`WhitelistKind::Address`) on all EVM transaction submissions via `assert_access` → `is_allow_submit`. However, the analogous check in `receive_erc20_tokens` — the ERC-20 bridge deposit path triggered by `ft_on_transfer` — is gated behind the existence of a fallback address. When no fallback address is configured, the whitelist check is entirely skipped, allowing any EVM address (including explicitly non-whitelisted ones) to receive bridged ERC-20 tokens. Because the recipient address is then unable to submit transactions (blocked by the same whitelist), the deposited tokens become frozen.

### Finding Description

**Root cause — inconsistent whitelist enforcement across two code paths:**

**Path 1 — `submit` (always enforced):**

In `engine/src/engine.rs`, `assert_access` is called unconditionally during every transaction submission:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}

`is_allow_receive_erc20_tokens` only checks the Address whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [4](#0-3) 

**The bug:** When `get_erc20_fallback_address` returns `None` (no fallback configured), the Rust `if let Some(...) && ...` short-circuits at the `Some` pattern match. The call to `is_allow_receive_erc20_tokens` is never reached. The `recipient` variable is left as the caller-supplied address, regardless of whether it is in the Address whitelist. Tokens are then minted to that address unconditionally. [5](#0-4) 

This is called from `ft_on_transfer` in `engine/src/contract_methods/connector.rs`: [6](#0-5) 

### Impact Explanation

**Impact: High — Temporary (potentially permanent) freezing of funds.**

When the Address whitelist is enabled in a Silo deployment and no fallback address is set:

1. A user sends NEP-141 tokens to Aurora via `ft_transfer_call`, specifying a non-whitelisted EVM address as the recipient in the message field.
2. `ft_on_transfer` → `receive_erc20_tokens` mints ERC-20 tokens to the non-whitelisted address (whitelist check bypassed).
3. The non-whitelisted address cannot submit any EVM transactions — `assert_access` → `is_allow_submit` blocks it.
4. The minted ERC-20 tokens are frozen at the non-whitelisted address with no self-service recovery path.

The freeze persists until the Silo admin either adds the address to the whitelist or implements a manual rescue. If the admin does not act, the freeze is permanent.

### Likelihood Explanation

The preconditions are realistic for any production Silo deployment:
- The Address whitelist is a core Silo feature, enabled by operators to restrict participation.
- A fallback address is an optional configuration. Operators may enable the whitelist without configuring a fallback (the two are set independently via `set_erc20_fallback_address` and `set_whitelist_status`).
- The attacker entry point is the standard NEP-141 `ft_transfer_call` interface — no special privileges are required. Any NEAR account holding a bridged NEP-141 token can trigger this path by specifying an arbitrary EVM address in the message.
- The attacker does not need to control the recipient address; they only need to specify one that is not whitelisted.

### Recommendation

Decouple the whitelist enforcement from the fallback address existence. The whitelist check should always be applied when the Address whitelist is enabled, and the fallback redirect should be a separate, independent action:

```rust
// Always enforce the whitelist check
if !silo::is_allow_receive_erc20_tokens(&self.io, &recipient) {
    // Only redirect if a fallback is configured; otherwise reject
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io) {
        recipient = fallback_address;
    } else {
        return Err(/* ERR_NOT_ALLOWED or similar */);
    }
}
```

This mirrors the unconditional enforcement in `assert_access` and ensures the whitelist is always respected regardless of fallback configuration.

### Proof of Concept

**Setup:**
1. Deploy Aurora Engine in Silo mode.
2. Enable the Address whitelist: `set_whitelist_status(WhitelistKind::Address, active: true)`.
3. Do **not** configure a fallback address (leave `erc20_fallback_address` unset).
4. Deploy a NEP-141 token and register its ERC-20 mirror on Aurora.

**Attack:**
1. Choose a victim EVM address `V` that is **not** in the Address whitelist.
2. Call `ft_transfer_call` on the NEP-141 contract with `receiver_id = aurora`, `amount = X`, `msg = V.encode()`.
3. Aurora's `ft_on_transfer` → `receive_erc20_tokens` is invoked.
4. Because `get_erc20_fallback_address` returns `None`, the `if let Some(...)` block is skipped entirely.
5. ERC-20 tokens of amount `X` are minted to address `V`.

**Verification:**
- Query the ERC-20 balance of `V`: it equals `X` (tokens received despite not being whitelisted).
- Attempt to submit any EVM transaction from `V`: it fails with `NotAllowed` (whitelist enforced on submit).
- The tokens at `V` are frozen. [3](#0-2) [4](#0-3) [1](#0-0)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.

    }

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
