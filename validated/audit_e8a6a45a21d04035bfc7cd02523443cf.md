### Title
Unbounded Memory Growth via Unsanitized Message Injection into `MessageBuffer` - (File: `src/protocol/internal.rs`)

### Summary
The `MessageBuffer` inside `Comms` accepts and stores every inbound message keyed by its `MessageHeader` (channel tag + waitpoint), with no cap on the number of distinct headers or the number of messages per header. A malicious participant can craft messages with arbitrary, never-polled `MessageHeader` values and deliver them via the public `Protocol::message` API, causing the honest party's process to accumulate unbounded heap memory for the duration of any protocol session (DKG, reshare, presign, sign, CKD).

### Finding Description

`MessageBuffer` is defined as:

```rust
struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}
``` [1](#0-0) 

Each `SubMessageQueue` is backed by a `futures::channel::mpsc::unbounded()` channel:

```rust
let (sender, receiver) = futures::channel::mpsc::unbounded();
``` [2](#0-1) 

The `push` method on `MessageBuffer` unconditionally inserts a new `HashMap` entry (and thus a new unbounded MPSC channel) for every distinct `MessageHeader` it receives:

```rust
fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
    let mut messages_lock = self.messages.lock().expect("lock should not fail");
    messages_lock.entry(header).or_default().send(from, message);
}
``` [3](#0-2) 

`Comms::push_message`, the only validation gate before `push`, performs only a length check and a header parse — it applies no allowlist of expected headers, no per-sender rate limit, and no total-entry cap:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);
}
``` [4](#0-3) 

The public `Protocol::message` entry point calls `push_message` directly with no further filtering:

```rust
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
``` [5](#0-4) 

A `MessageHeader` is 40 bytes (32-byte channel tag + 8-byte waitpoint), giving a space of 2^320 distinct values. An attacker can therefore generate an effectively unlimited number of unique headers and inject one message per header, each creating a new `HashMap` entry and a new heap-allocated MPSC channel object. None of these entries are ever evicted; the `HashMap` only grows.

The codebase itself contains a test that explicitly confirms this attack surface:

```rust
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    // Attacker injects messages for waitpoints the honest code never polls.
    comms.push_message(attacker, message);
    ...
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [6](#0-5) 

The test is named and commented to acknowledge the attack, yet no fix is present in the production path.

### Impact Explanation

Every honest participant running any protocol (DKG, reshare, refresh, presign, sign, CKD) exposes a `Protocol::message` endpoint to the application layer, which routes network messages from peers. A malicious participant can flood this endpoint with messages carrying novel `MessageHeader` values. Because each unique header allocates a new `HashMap` slot and an unbounded MPSC channel, heap memory grows without bound for the lifetime of the session. At sufficient volume this exhausts process memory, crashing the honest node and permanently denying it the ability to complete key generation, signing, or any other protocol operation.

This matches the allowed Medium impact: **Griefing or resource-exhaustion by a malicious participant causing unbounded memory beyond documented behavior.**

### Likelihood Explanation

Any participant in a threshold session can send arbitrary byte payloads to peers. The application layer is explicitly responsible for routing these payloads to `Protocol::message`. No authentication of the `MessageHeader` content is performed before buffering. The attack requires only the ability to send network messages — a capability every protocol participant already possesses — and no cryptographic material or privileged access. The `MessageHeader` space is astronomically large, so the attacker is never constrained by header collisions.

### Recommendation

1. **Cap the `HashMap` size.** Reject `push_message` calls once the number of distinct headers in `MessageBuffer` exceeds a protocol-specific bound (e.g., `num_participants × num_expected_waitpoints`).
2. **Cap per-header queue depth.** Replace `futures::channel::mpsc::unbounded()` with a bounded channel (e.g., `futures::channel::mpsc::channel(MAX_MSGS_PER_SLOT)`) and drop or error on overflow.
3. **Allowlist expected headers.** Derive the set of valid `(channel_tag, waitpoint)` pairs from the protocol state machine at construction time and silently discard messages whose headers are not in that set.
4. **Per-sender rate limiting.** Track the number of messages buffered per `Participant` and drop excess messages from any single sender.

### Proof of Concept

The repository's own test at `src/protocol/internal.rs:530–554` is a direct proof of concept. Extending it to an adversarial scenario:

```rust
// Malicious participant floods an honest node during a DKG session.
let honest_protocol = keygen(&participants, honest_me, threshold, rng).unwrap();

// Attacker sends 1,000,000 messages with unique waitpoints.
for i in 0u64..1_000_000 {
    let header = MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
    let mut msg = header.to_bytes().to_vec();
    msg.extend_from_slice(&i.to_le_bytes());
    // Delivered via the normal application routing layer:
    honest_protocol.message(attacker_participant, msg);
}
// honest_protocol's MessageBuffer now holds 1,000,000 HashMap entries,
// each with a live heap-allocated MPSC channel, consuming hundreds of MB.
// The honest node OOMs before DKG completes.
