### Title
NEP-141 Fee-on-Transfer Token Causes ERC-20 Mirror Over-Minting and Insolvency - (File: engine/src/engine.rs)

### Summary
Aurora Engine's `ft_on_transfer` handler blindly trusts the `amount` field supplied by the calling NEP-141 contract without verifying the actual balance change credited to Aurora. If a NEP-141 token charges a transfer fee (passing the gross amount to `ft_on_transfer` but crediting only the net amount to Aurora's account), the engine mints more ERC-20 mirror tokens than the NEP-141 tokens it actually holds, creating a permanent insolvency for the last users to exit.

### Finding Description
When a user bridges a NEP-141 token into Aurora via `ft_transfer_call`, the NEP-141 contract calls `ft_on_transfer` on Aurora with an `amount` argument. Aurora's handler in `engine/src/contract_methods/connector.rs` dispatches to `engine.receive_erc20_tokens`, which reads the amount directly from `args.amount`:

```rust
// engine/src/engine.rs line 803
let amount = args.amount.as_u128();
```

This value is then passed verbatim to `setup_receive_erc20_tokens_input`, which encodes a `mint(recipient, amount)` call on the ERC-20 mirror contract:

```rust
// engine/src/engine.rs lines 826-837
let result = self
    .call(
        &erc20_admin_address,
        &erc20_token,
        Wei::zero(),
        setup_receive_erc20_tokens_input(&recipient, amount),
        ...
    )
    .and_then(submit_result_or_err)?;
```

At no point does Aurora query its own NEP-141 balance before and after the transfer to confirm the actual amount received. The engine unconditionally trusts the caller-supplied `amount`.

Any NEP-141 token whose `ft_transfer_call` implementation deducts a fee from the credited amount but still invokes `ft_on_transfer` with the gross (pre-fee) amount will cause Aurora to mint more ERC-20 tokens than the NEP-141 tokens it holds. Because `deploy_erc20_token` is callable by any NEAR account to register any NEP-141, this path is fully reachable by an unprivileged actor.

### Impact Explanation
**Critical — Insolvency.**

After each fee-bearing deposit, Aurora's ERC-20 total supply exceeds its NEP-141 custodial balance by the fee amount. Over many deposits the gap grows. When users attempt to exit (burn ERC-20 → receive NEP-141 via `exit_to_near`), the NEP-141 `ft_transfer` promise will fail for the last users because Aurora's NEP-141 balance is insufficient to cover all outstanding ERC-20 balances. Those users permanently lose their funds.

### Likelihood Explanation
**Medium.** The NEP-141 standard does not prohibit fee-on-transfer implementations. A token deployer (unprivileged) can register any NEP-141 with Aurora via `deploy_erc20_token`. Once registered and used, the accounting divergence is irreversible. No admin action is required to trigger the bug; ordinary user deposits are sufficient.

### Recommendation
In `receive_erc20_tokens`, record Aurora's NEP-141 balance before the `ft_transfer_call` completes and use the actual balance delta (post-transfer balance minus pre-transfer balance) as the mint amount, rather than trusting `args.amount`. Alternatively, document that only NEP-141 tokens with no fee-on-transfer and no rebasing mechanism are supported, and enforce this at registration time.

### Proof of Concept
1. Deploy a NEP-141 token whose `ft_transfer_call` charges a 1% fee: it credits `0.99 * amount` to the receiver's balance but calls `ft_on_transfer(sender, amount, msg)` with the full `amount`.
2. Call `deploy_erc20_token` on Aurora to register this NEP-141 (permissionless).
3. Alice calls `ft_transfer_call` with `amount = 1000` on the NEP-141, targeting Aurora with her EVM address as `msg`.
4. Aurora receives 990 NEP-141 tokens but `ft_on_transfer` is called with `amount = 1000`.
5. `receive_erc20_tokens` mints 1000 ERC-20 tokens to Alice's EVM address.
6. Repeat N times. Aurora's NEP-141 balance = `990 * N`; ERC-20 total supply = `1000 * N`.
7. The first `990 * N / 1000` users to call `exit_to_near` succeed; the remaining users find Aurora's NEP-141 balance exhausted and their ERC-20 tokens are permanently frozen.
