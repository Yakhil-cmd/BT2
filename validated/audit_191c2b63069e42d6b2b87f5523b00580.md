### Title
WalletConnect `chia_setNFTDID` Approval Dialog Displays Raw DID Identifier Without Profile Resolution, Enabling Attacker-Controlled NFT Transfer — (`packages/gui/src/electron/commands/Commands.ts`, `packages/gui/src/electron/commands/humanizeParamValue.ts`)

### Summary
The WalletConnect approval dialog for `chia_wallet.nft_set_nft_did` / `chia_setNFTDID` renders the `did` parameter as a raw string (hex or bech32m DID identifier) with no resolution to a human-readable profile name from the user's DID wallet list. A connected dApp can supply any attacker-controlled DID as the destination. The user sees an opaque identifier and cannot distinguish it from one of their own profiles. If they approve, the NFT is transferred to the attacker's DID.

### Finding Description

**Step 1 — Command schema defines `did` as a plain `'string'`:**

In `Commands.ts`, the `chia_wallet.nft_set_nft_did` entry declares:

```ts
{ name: 'did', label: () => i18n._(/* i18n */ { id: 'DID' }), type: 'string' },
```

No `humanize` property is set, and no lookup against the user's DID wallet list is performed. [1](#0-0) 

**Step 2 — `humanizeParamValue` renders `type: 'string'` verbatim:**

```ts
case 'string':
case 'number':
  return String(value);
```

The raw DID value supplied by the dApp is passed through unchanged. [2](#0-1) 

**Step 3 — `humanizeParams` builds the `rows` array from the schema:**

Each visible param becomes a `{ field, label, value }` row; the `did` field's value is the raw string from the dApp payload. [3](#0-2) 

**Step 4 — `Confirm.tsx` renders each row's `value` verbatim:**

```tsx
<div className="mt-0.5 text-sm font-medium break-all whitespace-pre-wrap text-chia-text">{value}</div>
```

The user sees the raw DID string (e.g. `did:chia:1abc…` or a hex blob) with no indication of whether it belongs to their own wallet. [4](#0-3) 

**Contrast with the normal UI flow:**

`NFTMoveToProfileDialog` uses `DIDProfileDropdown` and `useGetDIDsQuery` to restrict destination selection to the user's own DID wallets and resolves the chosen wallet to a human-readable `name`. The WalletConnect path has no equivalent guard. [5](#0-4) 

### Impact Explanation
An unprivileged WalletConnect dApp can call `chia_setNFTDID` with an arbitrary attacker-controlled DID. The approval dialog presents the raw DID string under the label "DID" with no profile name, no ownership indicator, and no warning that the destination is not one of the user's own profiles. A user who approves transfers their NFT to the attacker's DID — a permanent, on-chain asset loss.

This satisfies **High** impact: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation
Any WalletConnect-paired dApp can issue this request. The user must click "Move" to confirm, but the approval dialog gives them no signal that the destination DID is external/attacker-controlled. Users accustomed to the normal UI (which only shows their own profiles by name) are likely to assume the displayed identifier is one of their own.

### Recommendation
In `humanizeParamValue` (or a dedicated enrichment step called from `humanizeDappCommand`), resolve the `did` parameter for `chia_wallet.nft_set_nft_did` against the user's DID wallet list via `getDIDs`. If the DID matches a known wallet, display its name. If it does not match any wallet owned by the user, display a prominent warning (e.g. "⚠ External / unknown DID — not one of your profiles") alongside the raw identifier. This mirrors the pattern already used in `NFTMoveToProfileDialog`. [6](#0-5) [7](#0-6) 

### Proof of Concept
1. Establish a WalletConnect session with the Chia GUI (any paired dApp).
2. Send a `chia_setNFTDID` request:
   ```json
   {
     "wallet_id": <victim_nft_wallet_id>,
     "nft_coin_ids": ["<victim_nft_coin_id>"],
     "did": "did:chia:1<attacker_did_launcher_id_bech32m>",
     "fee": 0
   }
   ```
3. Observe the approval dialog: the "DID" row shows the raw bech32m string with no profile name and no ownership warning.
4. Click "Move". The `nft_set_nft_did` RPC is called with the attacker's DID as `didId`.
5. Confirm on-chain that the NFT's `owner_did` is now the attacker's DID launcher ID.

export async function humanizeParamValue(
  param: ParamSchema,
  value: unknown,
  data: Record<string, unknown>,
  networkPrefix?: string,
): Promise<string> {
  if (value === null || value === undefined) {
    return i18n._(/* i18n */ { id: 'Not provided' });
  }

  const { type, humanize } = param;

  if (humanize) {
    switch (humanize) {
      case 'mojo-to-xch':
        return formatMojoXch(value, networkPrefix);
      case 'mojo-to-cat':
        return formatMojoCat(value, data);
      default:
        throw new Error('Unhandled humanize type');
    }
  }

  switch (type) {
    case 'string':
    case 'number':
      return String(value);
    case 'bigint':
      return BigInt(value as string | number | bigint).toString();
    case 'bool':
      return value === true ? i18n._(/* i18n */ { id: 'Yes' }) : i18n._(/* i18n */ { id: 'No' });
    case 'json':
      try {
        return JSONBig.stringify(value, null, 2);
      } catch {
        return String(value);
      }
    default: {
      throw new Error('Unhandled param type');
    }
  }
}

import { getDappCommandSchema } from './getDappCommandSchema';
import { humanizeParams } from './humanizeParams';

export async function humanizeDappCommand(dappCommand: string, data: Record<string, unknown>, networkPrefix?: string) {
  const dappCommandSchema = getDappCommandSchema(dappCommand);

  const rows = await humanizeParams(dappCommandSchema.params, data, networkPrefix);

  return {
    destructive: dappCommandSchema.destructive === true,
    title: dappCommandSchema.title(),
    message: dappCommandSchema.message(),
    confirmLabel: dappCommandSchema.confirmLabel(),
    rows,
  };
}
