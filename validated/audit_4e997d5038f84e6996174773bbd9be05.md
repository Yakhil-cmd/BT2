### Title
Unprotected Legacy Transaction (Pre-EIP-155) Accepted Without Chain ID Binding Enables Cross-Chain Replay - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS accepts pre-EIP-155 legacy transactions (v=27 or v=28) whose signing hash is computed **without** the chain ID. Because the signing hash is chain-agnostic, a transaction signed for any other EVM chain (e.g., Ethereum mainnet) is cryptographically valid on ZKsync OS. An attacker who observes such a transaction on another chain can replay it on ZKsync OS, executing it against the victim's account if the nonce matches.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155-protected:

```rust
// basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs

    hasher.update(inner_slice);

        return Err(InvalidTransaction::InvalidEncoding.into());

```

`is_eip155()` returns `false` when `v == 27 || v == 28`:

```rust
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
```

When the unprotected branch is taken, the signing hash is `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical on every EVM chain. The transaction is then stored as `RlpEncodedTxInner::Legacy` (not `LegacyWithEIP155`) and proceeds through the full validation pipeline.

In the proving execution config (`BasicBootloaderProvingExecutionConfig`, `VALIDATE_EOA_SIGNATURE = true`), the signature is verified via `secp256k1_ec_recover` against this chain-agnostic hash. Because the hash is the same on all chains, a signature produced on Ethereum mainnet passes verification on ZKsync OS without modification.

The EIP-155-protected path correctly rejects mismatched chain IDs (typed transactions EIP-2930/1559/7702 also enforce `tx.chain_id != expected_chain_id`), but the unprotected legacy path has no such guard.

---

### Impact Explanation

An attacker who observes a pre-EIP-155 legacy transaction broadcast on Ethereum mainnet (or any other EVM chain) can submit the identical raw bytes to ZKsync OS. If the victim's ZKsync OS account nonce equals the nonce in the replayed transaction, the bootloader will:

1. Parse the transaction as a valid unprotected legacy transaction.
2. Compute the same signing hash (no chain ID).
3. Recover the same `from` address via `ecrecover`.
4. Increment the nonce and execute the transaction.

The attacker can thereby transfer the victim's ZKsync OS ETH/tokens to an arbitrary address, or execute arbitrary calldata on behalf of the victim — direct loss of user funds.

---

### Likelihood Explanation

Pre-EIP-155 transactions (v=27/28) are uncommon in modern wallets but are still produced by older tooling, hardware wallets in legacy mode, and some programmatic signers. The nonce must match the victim's current ZKsync OS nonce, which is most easily satisfied when the victim is a new ZKsync OS user (nonce=0) who previously signed a nonce-0 transaction on Ethereum without EIP-155. The attacker needs only to monitor public mempools or historical chain data for such transactions. Likelihood is **low-to-moderate** but the impact when triggered is **high** (direct fund loss).

---

### Recommendation

Reject pre-EIP-155 legacy transactions at the bootloader level by returning `InvalidTransaction::InvalidChainId` (or a new `UnprotectedLegacyNotAllowed` variant) when `is_eip155()` is `false`. This mirrors the protection already applied to all typed transactions (EIP-2930, EIP-1559, EIP-7702) and to EIP-155-protected legacy transactions. The change is a one-line guard in `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`:

```rust
if legacy_signature.is_eip155() == false {
    return Err(InvalidTransaction::InvalidChainId.into()); // reject unprotected legacy
}
```

If backward compatibility with pre-EIP-155 tooling is required, the chain ID should at minimum be mixed into the signing hash for ZKsync OS (deviating from strict EVM equivalence for this edge case), analogous to how `Claimable::signatureClaim` in the PhiFactory report substituted `block.chainid` before forwarding.

---

### Proof of Concept

```
1. On Ethereum mainnet, sign a legacy transfer:
      nonce=0, gasPrice=X, gasLimit=21000, to=attacker, value=V, data=[]
      Sign with v=27 (no EIP-155).

2. Broadcast on Ethereum; observe the raw RLP bytes.

3. Submit the identical raw RLP bytes as an L2 transaction to ZKsync OS
   (the victim must have nonce=0 and sufficient balance on ZKsync OS).

4. ZKsync OS bootloader:
   - Parses as RlpEncodedTxInner::Legacy (unprotected).
   - Computes sig_hash = keccak256(rlp([0, X, 21000, attacker, V, []])) — same as Ethereum.
   - ecrecover(sig_hash, v=27, r, s) → victim address ✓
   - Nonce check: 0 == 0 ✓
   - Executes: transfers V ETH from victim to attacker on ZKsync OS.
