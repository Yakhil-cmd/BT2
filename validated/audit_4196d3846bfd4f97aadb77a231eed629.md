### Title
`MsgTurnBridge` Handler Is a No-Op — Bridge Disable Circuit-Breaker Completely Bypassed - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` gRPC message handler in the Cronos keeper is implemented as a literal no-op, returning `nil, nil` with no permission check and no state mutation. The bridge-disable circuit-breaker intended to halt gravity bridge and IBC bridge operations in an emergency is entirely non-functional. Any unprivileged user can continue to burn CRC21/CRC20 tokens and initiate bridge transfers via `send_to_evm_chain` or `send_to_ibc` even after an admin has called `MsgTurnBridge{Enable: false}`.

### Finding Description

The `TurnBridge` handler in `x/cronos/keeper/msg_server.go` is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

It performs two critical omissions simultaneously:

1. **No permission check** — unlike `UpdateTokenMapping` which gates on `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`, `TurnBridge` checks nothing. Any sender address is accepted.
2. **No state mutation** — the bridge-enabled flag is never written to the KV store. The `CanTurnBridge` permission bit exists in the permission system but the handler that is supposed to use it does nothing. [2](#0-1) 

The proto definition and the `CanTurnBridge` permission constant both confirm the intended design: [3](#0-2) [4](#0-3) 

The integration test `test_gravity_turn_bridge` explicitly expects that after `turn_bridge("false")` succeeds, a subsequent `send_to_evm_chain` call on a CRC21 contract **reverts** (`txreceipt.status == 0`): [5](#0-4) 

Because `TurnBridge` never changes any state, the bridge is permanently enabled. The `ModuleCRC21.send_to_evm_chain` and `send_to_ibc` functions have no on-chain guard that could be set by the keeper: [6](#0-5) 

The `DSToken` base contract (which `ModuleCRC21` inherits) does expose a `stop()`/`stopped` mechanism, but `TurnBridge` never invokes it. The `ModuleCRC20Proxy` bridge functions are equally unguarded: [7](#0-6) 

### Impact Explanation

This is a **High** impact finding matching: *Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks.*

The bridge circuit-breaker is the primary emergency mechanism to halt token outflows during a security incident (e.g., a gravity bridge exploit or a CRC21 accounting bug). Because `TurnBridge` is a no-op:

- An unprivileged user can burn CRC21 tokens and emit `__CronosSendToEvmChain` / `__CronosSendToIbc` events that the gravity bridge and IBC EVM hooks will process and execute, moving assets off-chain.
- The admin's `MsgTurnBridge{Enable: false}` call succeeds (returns code 0) but has zero effect, giving a false sense of security that the bridge has been halted.
- There is no way to stop bridge outflows through the intended protocol mechanism.

### Likelihood Explanation

The entry path is fully unprivileged — any user holding CRC21 tokens can call `send_to_evm_chain` or `send_to_ibc` at any time. The admin's `TurnBridge` call is silently ignored. The likelihood is high whenever an emergency bridge-disable is attempted.

### Recommendation

Implement `TurnBridge` to:
1. Verify the sender holds `CanTurnBridge` permission via `k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`.
2. Persist the bridge-enabled flag to the KV store (e.g., a dedicated `BridgeEnabled` key in module params or a separate store key).
3. Have the `SendToEvmChainHandler` and `SendToIbcHandler` EVM hook handlers read this flag and return an error when the bridge is disabled, preventing the token burn/transfer from being committed.
4. Optionally propagate the `stop()`/`start()` call to deployed CRC21 contracts so the EVM-level transaction itself reverts.

### Proof of Concept

1. Admin calls `MsgTurnBridge{Sender: admin, Enable: false}` — tx succeeds (code 0), but `TurnBridge` returns `nil, nil` with no state change.
2. Bob (any unprivileged address holding CRC21 tokens) calls `ModuleCRC21.send_to_evm_chain(recipient, amount, chain_id, 0, "")`.
3. `send_to_evm_chain` executes: burns Bob's tokens via `unsafe_burn` and emits `__CronosSendToEvmChain`.
4. The gravity bridge EVM hook processes the event and queues a batch transfer to the Ethereum-side recipient.
5. The bridge transfer completes — tokens leave Cronos despite the admin's intent to halt the bridge. [8](#0-7) [1](#0-0)
