### Title
`protocolUnpausePool` Passes Wrong Pause Level, Leaving Pool Permanently Paused After Protocol Unpause - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`, so a pool that the protocol pauses can never be restored to the active state by the protocol alone. After the "unpause," the pool remains paused at admin-level (1), and all swaps continue to revert.

### Finding Description

The factory defines three pause levels:

| Level | Meaning |
|---|---|
| 0 | Active (swaps allowed) |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolUnpausePool` is the protocol owner's mechanism to lift a protocol-level pause (level 2) and restore the pool to active (level 0). However, the wrong target level is hard-coded:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← BUG: should be setPause(0)
}
``` [1](#0-0) 

The call `setPause(1)` transitions the pool from protocol-paused (2) to admin-paused (1), not to active (0). The pool's `swap` function enforces:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [2](#0-1) 

Any non-zero `pauseLevel` blocks swaps. After `protocolUnpausePool` executes, `pauseLevel == 1`, so every swap still reverts with `PoolPaused`.

For comparison, the admin-side `unpausePool` correctly targets level 0:

```solidity
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // ← correct
}
``` [3](#0-2) 

The protocol's unpause path is therefore broken by a wrong literal value — the direct analog of the external report's wrong token-order literal.

### Impact Explanation

Every swap call checks `pauseLevel != 0` and reverts when it is non-zero. [4](#0-3) 

After `protocolUnpausePool`, `pauseLevel` is 1, so:

- All swaps remain blocked — the pool is completely unusable for trading.
- LPs cannot earn fees; traders cannot execute.
- The protocol owner has no further lever to fix this: `protocolUnpausePool` can only be called when `cur == 2`, and after the call `cur == 1`, so calling it again reverts.
- Recovery requires the pool admin to call `unpausePool` (which transitions 1 → 0). If the admin key is lost, compromised, or uncooperative, the pool is permanently frozen.

This is broken core pool functionality with direct loss of protocol fees and LP yield.

### Likelihood Explanation

The bug fires unconditionally every time the protocol owner calls `protocolUnpausePool`. There is no code path through this function that produces the correct outcome. Any emergency pause/unpause cycle by the protocol triggers it.

### Recommendation

```diff
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-   if (cur != 2) revert InvalidPauseTransition(cur, 1);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   if (cur != 2) revert InvalidPauseTransition(cur, 0);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
``` [1](#0-0) 

### Proof of Concept

1. Factory owner calls `protocolPausePool(pool)` — pool transitions 0 → 2 (protocol-paused). All swaps revert.
2. Factory owner calls `protocolUnpausePool(pool)` — guard passes (`cur == 2`), `setPause(1)` executes, pool transitions 2 → 1 (admin-paused).
3. Any user calls `pool.swap(...)` — `_checkNotPaused()` reads `pauseLevel == 1 != 0` → reverts `PoolPaused`. Pool is still frozen.
4. Factory owner calls `protocolUnpausePool(pool)` again — guard reads `cur == 1`, reverts `InvalidPauseTransition(1, 1)`. Protocol is stuck.
5. Only the pool admin calling `unpausePool(pool)` (1 → 0) can restore the pool, meaning the protocol's emergency unpause capability is entirely dependent on admin cooperation.
