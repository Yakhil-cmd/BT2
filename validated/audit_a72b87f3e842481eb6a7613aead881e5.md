### Title
Unchecked `uint64` Multiplication Overflow in `MeterComputation` / `ComputationAvailable` Enables EVM Computation Limit Bypass — (`fvm/meter/computation_meter.go`)

---

### Summary

`MeterComputation` and `ComputationAvailable` in `fvm/meter/computation_meter.go` perform unchecked `uint64` multiplications (`w * intensity`) that silently wrap in Go. An unprivileged EVM transaction sender can craft a transaction whose gas limit causes this product to overflow to zero, making `ComputationAvailable` return `true` unconditionally and `MeterComputation` record zero usage — allowing the user to execute a massive amount of EVM computation while paying fees for none of it.

---

### Finding Description

`MeterComputation` accumulates weighted computation cost:

```go
// fvm/meter/computation_meter.go line 100
m.computationUsed += w * intensity
```

`ComputationAvailable` performs a pre-flight check:

```go
// fvm/meter/computation_meter.go line 117
potentialComputationUsage := m.computationUsed + w*usage.Intensity
return potentialComputationUsage <= m.params.computationLimit
```

Neither multiplication is guarded against overflow. Go's `uint64` arithmetic wraps silently.

For EVM transactions, `checkGasLimit` in `fvm/evm/handler/handler.go` calls `ComputationAvailable` with `Intensity = tx.Gas()` (the user-supplied EVM gas limit):

```go
// fvm/evm/handler/handler.go line 683-692
func (h *ContractHandler) checkGasLimit(limit types.GasLimit) error {
    usage := common.ComputationUsage{
        Kind:      environment.ComputationKindEVMGasUsage,
        Intensity: uint64(limit),
    }
    if !h.backend.ComputationAvailable(usage) {
        return types.ErrInsufficientComputation
    }
    return nil
}
```

After execution, `meterGasUsage` calls `MeterComputation` with `Intensity = res.GasConsumed`:

```go
// fvm/evm/handler/handler.go line 714-719
func (h *ContractHandler) meterGasUsage(res *types.Result) error {
    usage := common.ComputationUsage{
        Kind:      environment.ComputationKindEVMGasUsage,
        Intensity: res.GasConsumed,
    }
    return h.backend.MeterComputation(usage)
}
```

The internal weight for `ComputationKindEVMGasUsage` is stored in units of `1 << MeterExecutionInternalPrecisionBytes` (i.e., `1 << 16 = 65536`). Choosing gas limit `L = 2^48` (281,474,976,710,656):

```
w * L = 65536 * 2^48 = 2^64  →  mod 2^64  =  0
```

**Step-by-step exploit:**

1. User submits EVM transaction with `gasLimit = 2^48`.
2. `checkGasLimit` → `ComputationAvailable`: `potentialComputationUsage = computationUsed + 0 = computationUsed ≤ computationLimit` → returns `true`.
3. EVM executes with gas limit `2^48`; actual gas consumed up to `2^48`.
4. `meterGasUsage` → `MeterComputation`: `computationUsed += 65536 * gasConsumed`. If `gasConsumed = 2^48`, this is again `0` mod `2^64`.
5. `deductTransactionFees` reads `ComputationUsed() = computationUsed >> 16 = 0`.
6. User is charged fees for **0** computation units despite executing `2^48` EVM gas.

The same overflow path exists in `ComputationFromIntensities` (line 36), used for fee calculation from accumulated intensities.

---

### Impact Explanation

A user can execute an arbitrarily large amount of EVM computation — up to `2^48` gas units per transaction — while paying zero execution fees. This constitutes unauthorized extraction of network computation resources and direct fee evasion. Repeated exploitation drains execution node resources without compensation, and can be used to execute expensive on-chain operations (complex contract calls, storage writes) for free.

---

### Likelihood Explanation

Any unprivileged EVM transaction sender can trigger this. The weight for `ComputationKindEVMGasUsage` is a public on-chain parameter. The overflow target gas limit (`2^48` or any multiple of `2^64 / w`) is trivially computable. No special role, staked node, or leaked key is required. The `run` and `batchRun` EVM entry points are both reachable from a standard Cadence transaction calling `EVM.run` or `EVM.batchRun`.

---

### Recommendation

Replace the bare multiplications with overflow-safe variants. Go's `math/bits` package provides `bits.Mul64`:

```go
// In MeterComputation (line 100):
hi, lo := bits.Mul64(w, intensity)
if hi > 0 {
    // overflow: treat as exceeding limit
    return errors.NewComputationLimitExceededError(m.params.TotalComputationLimit())
}
m.computationUsed += lo
```

Apply the same pattern to `ComputationAvailable` (line 117) and `ComputationFromIntensities` (line 36).
