### Title
Missing Zero-Address Validation in `change_reward_address` Enables Permanent Burning of Unclaimed Yield — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `Staking::change_reward_address` and `Pool::change_reward_address` accept a caller-supplied `reward_address` without asserting it is non-zero. If a staker or pool member accidentally (or via a buggy front-end) passes `0x0`, all subsequent reward transfers are sent to the zero address and permanently lost.

---

### Finding Description

`Staking::change_reward_address` updates the stored `reward_address` for a staker with no zero-address guard:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;
    staker_info.reward_address = reward_address;   // ← written without zero check
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

The only guard present is `does_token_exist`, which rejects token contract addresses but does **not** reject `0x0`. [2](#0-1) 

The same omission exists in `Pool::change_reward_address`:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
    pool_member_info.reward_address = reward_address;   // ← written without zero check
    ...
}
``` [3](#0-2) 

Once `reward_address` is set to `0x0`, every subsequent call to `Staking::claim_rewards` transfers STRK to the zero address:

```cairo
let reward_address = staker_info.reward_address;
...
self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
``` [4](#0-3) 

And `Pool::claim_rewards` does the same:

```cairo
let reward_address = pool_member_info.reward_address;
...
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [5](#0-4) 

The same zero-address gap exists at initial entry points `Staking::stake` and `Pool::enter_delegation_pool`, where `reward_address` is first recorded. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

If `reward_address` is set to `0x0` (accidentally or via a buggy integration), all STRK rewards accrued by the staker or pool member are irrecoverably transferred to the zero address on every `claim_rewards` call. There is no recovery path: the staker cannot reclaim already-transferred rewards, and the zero address is uncontrolled on Starknet. The principal stake itself is unaffected, but the entire yield stream is permanently destroyed.

---

### Likelihood Explanation

**Medium.** The functions are callable by any unprivileged staker or pool member at any time. A zero address is a natural accidental input (e.g., uninitialized variable in a script, a front-end bug, or a copy-paste error). The Sandclock judge confirmed an identical pattern: *"In this case assets are at risk due to external factors. A zero address check makes sense."* The protocol has many active stakers and delegators, making accidental misuse realistic.

---

### Recommendation

Add an explicit non-zero assertion at the top of both `change_reward_address` functions and at the `reward_address` parameter of `stake` / `enter_delegation_pool`:

```cairo
assert!(reward_address.is_non_zero(), "reward address cannot be zero");
```

This mirrors the existing `is_non_zero` pattern already used elsewhere in the codebase (e.g., `assert!(amount.is_non_zero(), ...)`) and is consistent with the `does_token_exist` guard already present. [8](#0-7) 

---

### Proof of Concept

1. Staker Alice calls `Staking::stake(reward_address: 0x0, operational_address: ..., amount: min_stake)`. No revert — `0x0` passes the `does_token_exist` check.
2. Rewards accrue over several epochs via the attestation/consensus flow.
3. Anyone calls `Staking::claim_rewards(staker_address: Alice)`.
4. `send_rewards_to_staker` executes `token_dispatcher.checked_transfer(recipient: 0x0, amount: rewards)`.
5. STRK is transferred to the zero address and permanently lost. Alice receives nothing.

Alternatively, an existing staker with accrued rewards calls `change_reward_address(0x0)` — same outcome on the next `claim_rewards` invocation.
