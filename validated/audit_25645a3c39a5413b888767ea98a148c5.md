Audit Report

## Title
Unbounded Sequential Cross-Canister Calls in SNS Governance `maybe_finalize_disburse_maturity` Heartbeat — (`rs/sns/governance/src/governance.rs`)

## Summary
`maybe_finalize_disburse_maturity` in the SNS governance canister collects all neurons with matured disbursements into an unbounded `Vec` and then issues a sequential cross-canister `transfer_funds` `.await` call for each entry within a single heartbeat invocation. An unprivileged attacker who controls N SNS neurons with matured disbursements forces the governance canister's heartbeat to make N sequential ledger round-trips, stalling `run_periodic_tasks` for N consensus rounds and degrading SNS governance liveness. The NNS governance canister has already fixed this exact pattern by processing exactly one disbursement per timer invocation.

## Finding Description
In `rs/sns/governance/src/governance.rs` at L4938–4975, `maybe_finalize_disburse_maturity` collects every neuron whose first pending disbursement has passed its `finalize_disbursement_timestamp_seconds` into an unbounded `Vec<(NeuronId, DisburseMaturityInProgress)>`:

```rust
let neuron_id_and_disbursements: Vec<(NeuronId, DisburseMaturityInProgress)> = self
    .proto
    .neurons
    .values()
    .filter_map(|neuron| {
        ...
        if now_seconds >= finalize_disbursement_timestamp_seconds {
            Some((id.clone(), first_disbursement.clone()))
        } else {
            None
        }
    })
    .collect();
```

At L4976–5081, it then iterates over the entire collected set and issues a blocking cross-canister call for each entry:

```rust
for (neuron_id, disbursement) in neuron_id_and_disbursements.into_iter() {
    ...
    let transfer_result = self
        .ledger
        .transfer_funds(...)
        .await;   // sequential cross-canister call inside unbounded loop
    ...
}
self.proto.is_finalizing_disburse_maturity = None;  // L5082
```

Each `.await` suspends the heartbeat execution and waits for a full ledger round-trip before the next iteration begins. The `is_finalizing_disburse_maturity` flag set at L4935 prevents re-entry but does nothing to bound the work within a single invocation. This function is called unconditionally from `run_periodic_tasks` at L5527.

On the IC, a canister's heartbeat is not re-invoked until the current heartbeat execution completes. Therefore, all tasks that follow `maybe_finalize_disburse_maturity` in `run_periodic_tasks` — `maybe_move_staked_maturity`, `compute_cached_metrics`, `maybe_gc` — are blocked for the entire duration of the N-round loop.

The NNS governance canister has already addressed this exact pattern. `try_finalize_maturity_disbursement` in `rs/nns/governance/src/governance/disburse_maturity.rs` at L462–469 processes exactly one disbursement per invocation using `.find()` instead of `.collect()`, and `FinalizeMaturityDisbursementsTask` in `rs/nns/governance/src/timer_tasks/finalize_maturity_disbursements.rs` uses `RecurringAsyncTask` to self-schedule based on the next pending disbursement timestamp. The SNS canister retains the old unbounded design.

The `MAX_NUMBER_OF_NEURONS_CEILING` for SNS is 200,000 (`rs/sns/governance/src/types.rs` L386), meaning the loop can in principle iterate over up to 200,000 neurons, each requiring a ledger round-trip.

## Impact Explanation
This is a concrete application-level DoS against SNS governance liveness. With N neurons each holding a matured disbursement, the attacker forces `run_periodic_tasks` to occupy N consensus rounds in a single heartbeat invocation. During this period:

1. **Heartbeat starvation**: All subsequent periodic tasks in `run_periodic_tasks` (staked maturity movement, cached metrics computation, GC) are delayed proportionally to N. The next heartbeat invocation cannot begin until the current one completes.
2. **Cycles drain**: The governance canister pays cycles for every ledger call. With large N, this constitutes an attacker-directed cycles drain on the SNS treasury.
3. **Disbursement finalization lock**: The `is_finalizing_disburse_maturity` flag is held for the entire N-round duration, blocking any further disbursement finalization.

This matches the allowed impact: **High — Application/platform-level DoS and significant SNS security impact with concrete user or protocol harm**.

## Likelihood Explanation
The attack is permissionless for any party holding sufficient SNS tokens. The steps are standard user operations: acquire SNS tokens, create N neurons (each above `neuron_minimum_stake_e8s`), call `DisburseMaturity` on each, and wait 7 days (`MATURITY_DISBURSEMENT_DELAY_SECONDS`) for disbursements to mature. The incubation period blends with normal user behavior. The attacker's capital (staked tokens) is recoverable after dissolving neurons. The attack is repeatable: after the loop completes, the attacker can re-stake and repeat. The severity scales linearly with N and is bounded only by the SNS-configured `max_number_of_neurons` (ceiling 200,000).

## Recommendation
Apply the same fix already used in the NNS governance canister:

1. Replace the unbounded `for` loop in `maybe_finalize_disburse_maturity` with a single-disbursement pattern analogous to `try_finalize_maturity_disbursement` in `rs/nns/governance/src/governance/disburse_maturity.rs` — find only the first eligible, unlocked neuron and process exactly one disbursement per invocation.
2. Convert the SNS periodic task to a `RecurringAsyncTask` (as in `rs/nns/governance/src/timer_tasks/finalize_maturity_disbursements.rs`) that self-schedules based on the next pending disbursement timestamp, rather than calling it unconditionally from `run_periodic_tasks`.

## Proof of Concept
1. Deploy an SNS instance on a local PocketIC or test network.
2. Create N neurons (e.g., N = 500) each holding the minimum stake, all controlled by the attacker principal.
3. Call `ManageNeuron { command: DisburseMaturity { percentage_to_disburse: 100 } }` on each neuron.
4. Advance the test clock by `MATURITY_DISBURSEMENT_DELAY_SECONDS + 1` so all disbursements mature simultaneously.
5. Trigger a heartbeat. Observe via mock ledger instrumentation that `transfer_funds` is called N times sequentially within a single `run_periodic_tasks` invocation.
6. Verify that `maybe_move_staked_maturity`, `compute_cached_metrics`, and `maybe_gc` are not reached until all N ledger calls complete.
7. Measure the wall-clock delay in processing the next heartbeat invocation; confirm it scales linearly with N.

The unbounded loop is at `rs/sns/governance/src/governance.rs` L4976–5081; the unconditional call site is at L5527.
