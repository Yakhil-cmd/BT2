### Title
`RewardSupplier.tick()` Permanently Traps `msg.value` When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary

The `tick()` function in `RewardSupplier.sol` is `payable` and accepts ETH as fees for L1→L2 messaging. However, when `amountToMint == 0` (no pending mint requests), the entire fee-spending block is skipped and any `msg.value` sent by the caller is permanently locked in the contract with no refund path.

### Finding Description

`tick()` is declared `external payable` and uses `msg.value` to fund two cross-chain calls inside an `if (amountToMint > 0)` guard:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // consumes half
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // consumes other half
    }
    // ← if amountToMint == 0, msg.value is never spent and never returned
}
```

When `requiredMinting()` returns `(0, 0)` — which happens whenever there are no pending L2→L1 mint-request messages — the `if` block is entirely skipped. The contract has no `receive()`, no `fallback()`, and no ETH-withdrawal function anywhere in `RewardSupplier.sol` or `RewardSupplierStorage.sol`. The sent ETH is permanently frozen inside the contract. [1](#0-0) 

### Impact Explanation

Any caller who sends ETH with `tick()` when there are no pending mint requests permanently loses that ETH. The contract provides no mechanism to recover it. This is a direct, irreversible financial loss for the caller. It maps to **Medium: Griefing with no profit motive but damage to users or protocol** (the ETH is destroyed from the caller's perspective with no beneficiary).

### Likelihood Explanation

`tick()` has no access control — any address can call it. Automated keeper bots or integrators are the expected callers. A caller who sends ETH speculatively (e.g., to ensure the call succeeds if minting is needed) will silently lose funds when `amountToMint == 0`. This condition is routine: it occurs every time the L2 reward supplier has not yet emitted a new mint request.

### Recommendation

Mirror the fix applied to `Gateway.sol` in the referenced report — revert if `msg.value` is non-zero but no minting work will be done:

```diff
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

+   if (amountToMint == 0) {
+       require(msg.value == 0, "NO_MINT_REQUIRED_NO_FEE_NEEDED");
+       return;
+   }

    if (amountToMint > 0) {
        // ... existing logic
    }
}
```

Alternatively, refund `msg.value` to `msg.sender` before returning when `amountToMint == 0`.

### Proof of Concept

1. L2 reward supplier has no pending mint requests → `requiredMinting()` returns `(0, 0)`.
2. A keeper bot calls `tick{value: 0.01 ether}()` to pre-fund the fee in case minting is needed.
3. `amountToMint == 0` → the `if` block is skipped entirely.
4. The 0.01 ETH is now held by `RewardSupplier` with no withdrawal path.
5. The keeper's ETH is permanently lost. [2](#0-1) [3](#0-2)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
