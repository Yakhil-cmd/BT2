### Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp` for Price-Velocity Enforcement on Berachain — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` measures how fast the oracle mid-price moves by counting **blocks** elapsed between swaps. On Berachain, block time is not constant. When blocks are produced faster than the rate assumed when `maxChangePerBlockE18` was calibrated, the allowed price movement per unit of real time shrinks, causing legitimate swaps to revert. When blocks are slower, the guard becomes too permissive, weakening the manipulation protection it was designed to provide.

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap` records the current block number each time a swap is processed and computes the allowed price change as:

```
allowedSq = maxChange² × (1 + blockDiff)
```

where `blockDiff = block.number - prevBlock`. [1](#0-0) 

The pool admin calibrates `maxChangePerBlockE18` against a **current average block time** `T`. If Berachain later produces blocks at `T/k` (faster), then for the same wall-clock interval the guard sees `blockDiff` shrink by factor `k`, so `allowedSq` shrinks by `~k`, while the oracle price can still move at its normal real-world velocity. The check

```
if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
``` [2](#0-1) 

fires on every swap until the admin manually intervenes. The same field is written in `setLastMidPrice`, which also stamps `block.number`: [3](#0-2) 

The contract NatSpec explicitly frames the invariant in block-time terms ("Caps how fast the provided price can move **between blocks**"), confirming the design assumption of a constant block interval. [4](#0-3) 

---

### Impact Explanation

**Direction 1 — Faster blocks → swap DoS.**  
If Berachain's block time halves, `blockDiff` for a given real-world interval halves, `allowedSq` halves, but the oracle price can still move at its normal rate. Every swap where the oracle has moved by a legitimate amount reverts with `PriceVelocityExceeded`. The pool becomes completely unusable for swaps until the admin calls `setMaxChangePerBlock` or `setLastMidPrice`. This is broken core pool functionality.

**Direction 2 — Slower blocks → guard bypass.**  
If block time doubles, `blockDiff` doubles, `allowedSq` doubles, and a price manipulator can move the oracle price twice as fast (in real time) before the guard triggers. The manipulation protection the extension was deployed to provide is silently halved.

Both directions are triggered by any unprivileged user attempting a swap; no malicious setup is required.

---

### Likelihood Explanation

Berachain's block time is documented to vary with network congestion. The protocol is explicitly targeting Berachain deployment. The admin sets `maxChangePerBlockE18` once at configuration time; any sustained shift in block cadence (which Berachain's own documentation acknowledges as expected) produces the miscalibration without any further action by any party.

---

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`. Rename the stored field from `lastUpdateBlock` to `lastUpdateTimestamp` and express `maxChangePerBlockE18` as `maxChangePerSecondE18`. The velocity formula becomes:

```solidity
uint256 timeDiff = block.timestamp - prevTimestamp;   // seconds elapsed
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

This makes the guard invariant to block-time fluctuations, matching the fix applied to the analogous `StreamingNFT` vesting issue.

---

### Proof of Concept

1. Pool is deployed with `PriceVelocityGuardExtension` attached. Admin calls `setMaxChangePerBlock(pool, C)` calibrated for a 2-second average block time, intending to allow at most `C` price change per 2 seconds.
2. Berachain network congestion drops and blocks are now produced every 0.5 seconds (4× faster).
3. Oracle price moves by a legitimate 0.8 % over 4 seconds (2 blocks at old cadence → `blockDiff = 2`, `allowedSq = C² × 3`). At new cadence, 4 seconds = 8 blocks → `blockDiff = 8`, `allowedSq = C² × 9` — this direction is fine.
4. **Reverse scenario**: oracle price moves 0.8 % over 1 second. At old cadence that is 0.5 blocks → `blockDiff = 0`, `allowedSq = C²`. At new cadence (0.5 s/block) that is 2 blocks → `blockDiff = 2`, `allowedSq = C² × 3` — also fine in this direction.
5. **Correct harmful scenario**: oracle price moves 0.8 % over 2 seconds. At old cadence `blockDiff = 1`, `allowedSq = C² × 2`. At new 0.5 s/block cadence, 2 seconds = 4 blocks, `blockDiff = 4`, `allowedSq = C² × 5` — guard is more permissive.

Corrected harmful scenario for DoS: blocks are now produced every 8 seconds (4× slower than assumed 2 s). Oracle price moves 0.8 % over 2 seconds (real time). At old cadence that is 1 block → `blockDiff = 1`. At new cadence (8 s/block), 2 seconds = 0 blocks → `blockDiff = 0`, `allowedSq = C²`. If `actualSq > C²`, the swap reverts. The pool is stuck until admin intervention. [5](#0-4)
