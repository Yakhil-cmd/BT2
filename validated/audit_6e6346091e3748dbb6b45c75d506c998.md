### Title
Unrestricted `update_rewards` Allows Any Caller to Redirect Consensus Block Rewards to an Arbitrary Staker — (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` has no caller-identity access control. Any unprivileged address can invoke it with an arbitrary `staker_address` and `disable_rewards` flag. Because the function writes to the **global** `last_reward_block` on every call, a single invocation per block either (a) redirects that block's consensus reward to a staker who did not propose the block, or (b) consumes the per-block slot with `disable_rewards: true`, permanently denying the legitimate block proposer their yield.

---

### Finding Description

`update_rewards` is gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero. [1](#0-0) 

No check verifies that the caller is the actual block proposer, or that `staker_address` is the staker entitled to the reward for the current block. The function then unconditionally writes the current block number to the **global** `last_reward_block`: [2](#0-1) 

After that write, the guard at the top of the function will reject every subsequent call in the same block: [3](#0-2) 

If `disable_rewards` is `false`, rewards are distributed to the caller-supplied `staker_address`: [4](#0-3) 

If `disable_rewards` is `true`, the slot is consumed with no reward distributed: [5](#0-4) 

The analog to the original report is exact: `commitReaction` checked that the user joined `_gameId` but did not verify that `_questionId` belonged to `_gameId`. Here, `update_rewards` checks that `staker_address` is a valid active staker but does not verify that the caller is the block proposer entitled to the reward for this block.

---

### Impact Explanation

**Attack path A — reward theft (High):**
An attacker calls `update_rewards(victim_staker, disable_rewards: false)` in the block where `victim_staker` is the legitimate proposer. The attacker-chosen staker receives the full block reward; `last_reward_block` is set; the legitimate proposer's subsequent call reverts with `REWARDS_ALREADY_UPDATED`. The proposer's unclaimed yield for that block is permanently lost and misrouted.

**Attack path B — yield freeze (High / Medium):**
An attacker calls `update_rewards(any_valid_staker, disable_rewards: true)` every block. `last_reward_block` advances each block, no rewards are ever distributed, and all stakers are permanently frozen out of consensus block rewards. This matches "Permanent freezing of unclaimed yield."

---

### Likelihood Explanation

The function is fully public; no role, signature, or stake ownership is required. Any EOA or contract with a non-zero address can call it. The only cost is the gas for one transaction per block. On a low-fee L2 like Starknet this is economically trivial. As Starknet moves toward decentralized sequencing, front-running the legitimate proposer's `update_rewards` call becomes straightforward.

---

### Recommendation

1. **Restrict the caller**: Add a role check (e.g., `only_block_proposer` or `only_app_governor`) so that only the authorized entity can call `update_rewards`.
2. **Derive staker from caller**: Instead of accepting `staker_address` as a parameter, derive it from `get_caller_address()` via the `operational_address_to_staker_address` mapping, eliminating the ability to redirect rewards to an arbitrary staker.
3. **Separate the `disable_rewards` path**: If disabling rewards for an ineligible proposer is a legitimate use case, gate that path separately so it cannot be abused to consume the global `last_reward_block` slot.

---

### Proof of Concept

```
Block N:
  1. Staker A is the legitimate block proposer.
  2. Attacker submits tx: update_rewards(staker_B, disable_rewards=false)
     → last_reward_block = N
     → staker_B accrues block reward (unclaimed_rewards_own += block_reward)
  3. Staker A submits tx: update_rewards(staker_A, disable_rewards=false)
     → assert!(N > last_reward_block)  // N > N is false → REWARDS_ALREADY_UPDATED
     → staker_A receives nothing for block N

Repeat every block → staker_A's yield is permanently frozen;
staker_B (or any attacker-chosen staker) accumulates all block rewards.
```
