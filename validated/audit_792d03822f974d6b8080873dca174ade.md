Audit Report

## Title
`u64` Multiplication Overflow in `disburse_maturity` Causes Canister Trap - (File: `rs/sns/governance/src/governance.rs`)

## Summary
`Governance::disburse_maturity` in the SNS governance canister multiplies two `u64` values without widening to `u128`, then calls `.expect()` on the result. When a neuron's `maturity_e8s_equivalent` is sufficiently large, `checked_mul` returns `None` and `.expect()` panics, trapping the update call. The NNS governance already fixed the identical pattern by widening to `u128` first, but the SNS governance was never updated.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 1643–1649:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent                                        // u64
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)  // u64 × u64
    .expect("Overflow while processing maturity to disburse.")      // panics on None
    .checked_div(100)
    .expect("Error when processing maturity to disburse.")
    as u128;
``` [1](#0-0) 

Both operands are `u64`. `checked_mul` returns `None` when the product exceeds `u64::MAX ≈ 1.844 × 10¹⁹`. The `.expect()` call then panics, which in the IC execution model causes the update call to **trap** and roll back all state changes. The only input validation performed before this point is a range check that `percentage_to_disburse` is between 1 and 100 inclusive — there is no guard on `maturity_e8s_equivalent`. [2](#0-1) 

The overflow threshold for `percentage_to_disburse = 100` is `u64::MAX / 100 ≈ 1.844 × 10¹⁷ e8s ≈ 1.844 billion tokens` of maturity. The NNS governance already fixed the identical pattern in `rs/nns/governance/src/governance/disburse_maturity.rs` by widening both operands to `u128` before multiplying:

```rust
(total_maturity_e8s as u128)
    .checked_mul(percentage_to_disburse as u128)
    ...
``` [3](#0-2) 

The SNS governance `disburse_maturity` was never updated to apply the same widening.

## Impact Explanation
Any neuron controller whose neuron's `maturity_e8s_equivalent` meets or exceeds `u64::MAX / percentage_to_disburse` is permanently unable to disburse maturity via any `percentage_to_disburse` value. Every call to `manage_neuron { DisburseMaturity { ... } }` traps the SNS governance canister update call. The maturity remains locked in the neuron but is inaccessible via this path, causing concrete financial harm (inability to convert accumulated maturity into liquid SNS tokens). This matches the allowed High impact: **"Significant SNS security impact with concrete user or protocol harm."**

## Likelihood Explanation
SNS tokens can have very large total supplies. A neuron holding a large stake that has accumulated maturity over time can reach the overflow threshold (~1.844 billion tokens of maturity at 100% disbursement). The exploit path requires no special privileges beyond controlling the affected neuron — it is a standard unprivileged `manage_neuron` ingress call. The condition is self-inflicted only in the sense that the neuron must be large, but large neurons are the primary participants in SNS governance and the most likely to attempt maturity disbursement.

## Recommendation
Widen `maturity_e8s_equivalent` to `u128` before multiplying, matching the NNS governance fix in `rs/nns/governance/src/governance/disburse_maturity.rs`:

```rust
// After (safe):
let maturity_to_deduct = (neuron.maturity_e8s_equivalent as u128)
    .checked_mul(disburse_maturity.percentage_to_disburse as u128)
    .and_then(|v| v.checked_div(100))
    .expect("Overflow while processing maturity to disburse.");
``` [4](#0-3) 

## Proof of Concept
1. Deploy an SNS governance canister.
2. Create a neuron and set `maturity_e8s_equivalent` to a value ≥ `u64::MAX / 100 ≈ 1.844 × 10¹⁷` (achievable via a unit test by directly constructing the neuron state).
3. Call `manage_neuron` with `DisburseMaturity { percentage_to_disburse: 100, to_account: None }` as the neuron controller.
4. Observe the canister traps with `"Overflow while processing maturity to disburse."` — the call returns an error and no maturity is disbursed.
5. Confirm the same trap occurs for any `percentage_to_disburse` value P when `maturity_e8s_equivalent * P > u64::MAX`.

A minimal unit test can reproduce this deterministically by constructing a `Neuron` with `maturity_e8s_equivalent = u64::MAX / 50 + 1` and calling `disburse_maturity` with `percentage_to_disburse = 50`, which will panic at line 1646. [1](#0-0)

fn percentage_of_maturity(
    total_maturity_e8s: u64,
    percentage_to_disburse: u32,
) -> Result<u64, InitiateMaturityDisbursementError> {
    (total_maturity_e8s as u128)
        .checked_mul(percentage_to_disburse as u128)
        .and_then(|result| result.checked_div(100))
        .and_then(|result| {
            // This should be impossible as long as `percentage_to_disburse` is between 0 and 100.
            if result > u64::MAX as u128 {
                None
            } else {
                Some(result as u64)
            }
        })
        .ok_or_else(|| InitiateMaturityDisbursementError::Unknown {
            reason: format!(
                "Failed to calculate percentage of maturity: {percentage_to_disburse}% of {total_maturity_e8s} e8s"
            ),
        })
