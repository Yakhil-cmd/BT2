### Title
Missing Identity-Point Validation in CKD Coordinator Aggregation Allows Malicious Participant to Corrupt Derived Key — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

`do_ckd_coordinator` aggregates per-participant `(big_y, big_c)` shares with no validation that the received G1 points are non-identity. A malicious participant can send `(G1::identity(), G1::identity())`, causing the coordinator to silently produce a `CKDOutput` that is missing that participant's Lagrange-weighted contribution, yielding a cryptographically wrong derived key that fails `verify_signature`.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `CKDOutput` via `recv_from_others` and unconditionally adds the two G1 fields into running accumulators: [1](#0-0) 

`recv_from_others` performs only deserialization and deduplication — no point validation: [2](#0-1) 

`CKDOutput` derives `serde::Deserialize` directly over `blstrs::G1Projective`, which accepts the identity point (the point-at-infinity has a valid compressed encoding): [3](#0-2) 

There is no call to `is_identity()` anywhere in the aggregation path. By contrast, the `frost_core::Group` implementation for `BLS12381G1Group` **does** reject identity during deserialization — but that path is never invoked here: [4](#0-3) 

The CKD protocol requires every participant's Lagrange-weighted share to reconstruct the master secret:

```
Y  = Σ_i  λ_i · y_i · G
C  = Σ_i  λ_i · (x_i · H(app_id) + y_i · A)
```

`unmask(a)` recovers `C − a·Y = msk · H(app_id)` only when all shares are present. If participant `j` sends `(0, 0)`:

- `norm_big_y += identity` → no-op (λ_j · y_j · G is absent)
- `norm_big_c += identity` → no-op (λ_j · x_j · H(app_id) is absent)

The unmasked result becomes `(msk − λ_j · x_j) · H(app_id)`, which is not a valid BLS signature under the public key. [5](#0-4) 

---

### Impact Explanation

The coordinator returns a `CKDOutput` that appears structurally valid but encodes a wrong key. The TEE application calls `unmask`, obtains a corrupted G1 point, and `verify_signature` fails: [6](#0-5) 

This matches **High: Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs**. The coordinator has no way to detect the corruption; it silently propagates the bad output.

---

### Likelihood Explanation

Any single protocol participant can trigger this. The participant controls their own protocol binary and can substitute `(G1::identity(), G1::identity())` for their honest share. No cryptographic assumption needs to be broken; it is a pure input-validation gap. In a deployment where even one MPC node is compromised or malicious, this is trivially exploitable.

---

### Recommendation

In `do_ckd_coordinator`, after deserializing each `CKDOutput`, reject any share whose `big_y` or `big_c` is the identity point:

```rust
for (from, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    if participant_output.big_y().is_identity().into()
        || participant_output.big_c().is_identity().into()
    {
        return Err(ProtocolError::IdentityElement);
    }
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Additionally, consider adding a zero-knowledge proof of correct share formation (e.g., a Schnorr proof that `big_c` is consistent with the participant's public key share), which would prevent any malformed — not just identity — contribution.

    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,

    T: serde::de::DeserializeOwned,

    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    msg: &[u8],
    signature: &Signature,
) -> Result<(), frost_core::Error<BLS12381SHA256>> {
    let element1: G1Affine = signature.into();
    if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
        return Err(frost_core::Error::InvalidSignature);
    }
    let element2: G2Affine = verifying_key.to_element().into();
    if (!element2.is_on_curve() | !element2.is_torsion_free() | element2.is_identity()).into() {
        return Err(frost_core::Error::MalformedVerifyingKey);
    }

    // Concatenate the master public key (96 bytes) in the hash computation
    // H(pk || app_id) when H is a random oracle

            .into();

        Ok(())
