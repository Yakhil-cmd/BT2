### Title
Permanently Unwithdrawable Sub-Threshold EVM Balance in CadenceOwnedAccount — (File: fvm/evm/emulator/emulator.go)

---

### Summary

Any EVM-native balance below `1e10 attoFlow` (the minimum UFix64-representable unit) that accumulates in a Cadence Owned Account (COA) via EVM-side transfers becomes permanently unwithdrawable to Cadence, causing irreversible cross-VM asset loss with no recovery path.

---

### Finding Description

The `withdrawFrom` function enforces that the requested withdrawal amount must satisfy `AttoFlowBalanceIsValidForFlowVault`, which requires the value to be `>= UFixToAttoConversionMultiplier` (i.e., `1e10 attoFlow`). [1](#0-0) 

This guard exists because Cadence's `FlowToken.Vault` uses `UFix64` (8 decimal places), whose smallest representable unit is `1e-8 FLOW = 1e10 attoFlow`. [2](#0-1) 

The `CadenceOwnedAccount.withdraw` in the EVM contract passes the caller-supplied `balance.attoflow` directly to `InternalEVM.withdraw`, which routes to `withdrawFrom`. [3](#0-2) 

**The gap:** The `mintTo` (deposit) path has **no symmetric lower-bound check**. It accepts any amount from the EVM side. [4](#0-3) 

This means any EVM-native transfer (from a Solidity contract or EOA) can credit a COA with an amount `< 1e10 attoFlow`. Once the COA's total EVM balance is entirely below `1e10 attoFlow`, the user faces an impossible situation:

- Withdrawing the exact balance → rejected by `AttoFlowBalanceIsValidForFlowVault` (amount `< 1e10`)
- Withdrawing `1e10 attoFlow` → rejected by the EVM state machine (insufficient balance)

There is no "close account", "sweep dust", or "burn remainder" function anywhere in the contract or handler. [5](#0-4) 

The `ErrWithdrawBalanceRounding` error is explicitly defined as a non-fatal, non-recoverable rejection: [6](#0-5) 

---

### Impact Explanation

Any COA whose EVM balance is entirely below `1e10 attoFlow` has its funds permanently locked in the EVM environment. There is no protocol-level path to return those funds to Cadence. This is a direct cross-VM asset loss: FLOW tokens are provably held in the EVM state but are inaccessible to their Cadence owner. The `ErrInsufficientTotalSupply` safety guard confirms the protocol tracks this invariant but provides no recovery mechanism. [7](#0-6) 

---

### Likelihood Explanation

Moderate. The scenario does not require a privileged actor. Any unprivileged EVM contract author or EOA can trigger it by sending `< 1e10 attoFlow` to a COA's EVM address. Realistic vectors include:

1. **EVM gas refunds**: If a COA submits an EVM transaction with `gasPrice > 0` and the refund `(gasLimit − gasUsed) × gasPrice < 1e10 attoFlow`, the refund is credited to the COA but can never be withdrawn.
2. **EVM contract dust transfers**: Any Solidity contract can call `payable(coaAddress).transfer(1)`, crediting 1 attoFlow.
3. **Griefing**: An adversary deliberately sends dust to a COA whose balance is already `< 1e10 attoFlow`, permanently locking whatever balance exists.

The `AttoFlowBalanceIsValidForFlowVault` check confirms the protocol is aware of the precision gap but only guards the withdrawal direction, not the deposit direction. [8](#0-7) 

---

### Recommendation

Add a recovery mechanism analogous to the "close account" recommendation in the original report:

1. **Sweep/burn function**: Expose a privileged or owner-callable function on `CadenceOwnedAccount` that burns (or donates to a protocol fee sink) any sub-threshold remainder, allowing the account to reach a zero balance cleanly.
2. **Floor-and-burn on withdraw**: When a user requests withdrawal of their full balance, automatically floor the amount to the nearest `1e10 attoFlow` multiple, execute the withdrawal for the floored amount, and burn the remainder in the EVM state, keeping the Cadence-side vault and EVM-side balance in sync.
3. **Deposit guard**: Mirror the withdrawal guard on the `mintTo` path to reject EVM-native deposits that would leave a sub-threshold remainder, preventing the stuck-balance state from arising.

---

### Proof of Concept

```
// Step 1: Deploy a Solidity contract that sends dust to a COA address
// (any EVM EOA or contract can do this)
coaEVMAddress.transfer(5_000_000_000);  // 5e9 attoFlow < 1e10 threshold

// Step 2: COA now holds 5e9 attoFlow in EVM state.

// Step 3: Owner attempts to withdraw exact balance — FAILS
coa.withdraw(balance: EVM.Balance(attoflow: 5_000_000_000))
// → ErrWithdrawBalanceRounding: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"

// Step 4: Owner attempts to withdraw minimum threshold — FAILS
coa.withdraw(balance: EVM.Balance(attoflow: 10_000_000_000))
// → EVM execution fails: "insufficient funds for gas * price + value"

// Result: 5e9 attoFlow permanently locked in EVM, no recovery path.
```

func (proc *procedure) mintTo(
	call *types.DirectCall,
) (*types.Result, error) {
	// check and convert value
	value, isValid := checkAndConvertValue(call.Value)
	if !isValid {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrInvalidBalance,
		), nil
	}

	// create bridge account if not exist
	bridge := call.From.ToCommon()
	if !proc.state.Exist(bridge) {
		proc.state.CreateAccount(bridge)
	}

	// add balance to the bridge account before transfer
	proc.state.AddBalance(bridge, value, gethTracing.BalanceIncreaseWithdrawal)
	// check state errors
	if err := proc.state.Error(); err != nil {
		return nil, err
	}

	// withdraw the amount and move it to the bridge account
	res, err := proc.run(call.Message(), call.Hash(), types.DirectCallTxType)
	if err != nil {
		return res, err
	}

	// if any error (invalid or vm) on the internal call, revert and don't commit any change
	// this prevents having cases that we add balance to the bridge but the transfer
	// fails due to gas, etc.
	if res.Invalid() || res.Failed() {
		// reset the state to revert the add balances
		proc.state.Reset()
		return res, nil
	}

	// commit and finalize the state and return any stateDB error
	res.StateChangeCommitment, err = proc.commit(true)
	return res, err
}

func (proc *procedure) withdrawFrom(
	call *types.DirectCall,
) (*types.Result, error) {
	// check and convert value
	value, isValid := checkAndConvertValue(call.Value)
	if !isValid {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrInvalidBalance,
		), nil
	}

	// check balance is not prone to rounding error
	if !types.AttoFlowBalanceIsValidForFlowVault(call.Value) {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrWithdrawBalanceRounding,
		), nil
	}

	// create bridge account if not exist
	bridge := call.To.ToCommon()
	if !proc.state.Exist(bridge) {
		proc.state.CreateAccount(bridge)
	}

	// withdraw the amount and move it to the bridge account
	res, err := proc.run(call.Message(), call.Hash(), types.DirectCallTxType)
	if err != nil {
		return res, err
	}

	// if any error (invalid or vm) on the internal call, revert and don't commit any change
	// this prevents having cases that we deduct the balance from the account
	// but doesn't return it as a vault.
	if res.Invalid() || res.Failed() {
		// reset the state to revert the add balances
		proc.state.Reset()
		return res, nil
	}

	// now deduct the balance from the bridge
	proc.state.SubBalance(bridge, value, gethTracing.BalanceIncreaseWithdrawal)

	// commit and finalize the state and return any stateDB error
	res.StateChangeCommitment, err = proc.commit(true)
	return res, err
}

// BalanceConversionToUFix64ProneToRoundingError returns true
// if casting to UFix64 could result in rounding error
func BalanceConversionToUFix64ProneToRoundingError(bal Balance) bool {
	return new(big.Int).Mod(bal, UFixToAttoConversionMultiplier).BitLen() != 0
}

// AttoFlowBalanceIsValidForFlowVault returns true if the given balance,
// represented as atto-flow, can be stored in a Flow Vault, without loss
// in precision.
//
// Warning! The smallest unit of Flow token that a Flow Vault (Cadence)
// can store, is 1e-8 .
// This means the minimum balance, in atto-flow, that can be stored in a
// Flow Vault, is 1e10 .
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
	return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
