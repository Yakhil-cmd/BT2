## Title
SNS Governance `distribute_rewards` Settles Proposals Belonging to the Next Reward Round — (File: rs/sns/governance/src/governance.rs)

## Summary
`distribute_rewards` in SNS governance collects proposals to settle by calling `ready_to_be_settled_proposal_ids()`, which evaluates reward status against `now` (wall-clock time). The reward event it creates only covers the period up to `reward_event_end_timestamp_seconds`, which is strictly ≤ `now`. Any proposal whose voting deadline falls in the gap `(reward_event_end_timestamp_seconds, now]` is `ReadyToSettle` as of `now` but belongs to the next reward round; it is incorrectly swept into the current event, its ballots are permanently cleared, and maturity is distributed early. NNS governance avoids this by passing a round-boundary-truncated timestamp to its equivalent function.

## Finding Description
`ready_to_be_settled_proposal_ids` captures `now` internally and filters on `reward_status(now) == ReadyToSettle`: [1](#0-0) 

`distribute_rewards` calls it with no timestamp argument: [2](#0-1) 

`reward_event_end_timestamp_seconds` is then computed as the floor of `now` to the nearest completed round boundary: [3](#0-2) 

The gap `now − reward_event_end_timestamp_seconds` equals `(now − reward_start) mod round_duration_seconds`, ranging from 0 to `round_duration_seconds − 1` seconds. A proposal whose `get_deadline_timestamp_seconds()` falls inside this gap satisfies `accepts_vote(now) == false` (deadline < now) and therefore returns `ReadyToSettle`: [4](#0-3) 

Such a proposal is included in `considered_proposals` and settled under the current reward event even though its deadline postdates `reward_event_end_timestamp_seconds`. Its ballots are then permanently cleared: [5](#0-4) 

NNS governance avoids this by accepting an explicit `as_of_timestamp_seconds` parameter: [6](#0-5) 

And passing the truncated round-end timestamp computed by `most_recent_fully_elapsed_reward_round_end_timestamp_seconds`: [7](#0-6) [8](#0-7) 

SNS governance has no equivalent guard.

## Impact Explanation
This is a concrete SNS governance accounting integrity failure with three distinct harms:

1. **Premature maturity distribution.** Neurons that voted on a next-round proposal receive maturity one full round early. The maturity is credited immediately and irreversibly.
2. **Reward dilution for current-round voters.** The rewards purse is sized for `new_rounds_count` complete rounds. Settling additional next-round proposals spreads that fixed purse across more voting-power shares, reducing per-neuron maturity for all legitimate current-round voters.
3. **Permanent ballot data loss.** `p.ballots.clear()` is called on the prematurely settled proposal. Ballot data is gone with no recourse; the proposal can never be re-settled in its correct round.

This matches the allowed impact: **High — Significant SNS security impact with concrete user or protocol harm**, specifically unauthorized/incorrect distribution of governance maturity and permanent destruction of governance state.

## Likelihood Explanation
The bug fires deterministically on every `distribute_rewards` invocation where at least one proposal's deadline falls in the gap window. Because `should_distribute_rewards` triggers as soon as `seconds_since_last_reward_event > round_duration_seconds`, and the timer fires at the first opportunity after a full round elapses, the gap window is non-zero in virtually every real distribution cycle. No special privilege is required beyond the ability to submit SNS proposals (standard SNS participant capability). An attacker can time a proposal's `initial_voting_period_seconds` so its deadline lands just after the next round boundary, guaranteeing it is swept into the current event when the timer fires. [9](#0-8) 

## Recommendation
Mirror the NNS approach in SNS governance:

1. Add an `as_of_timestamp_seconds: u64` parameter to `ready_to_be_settled_proposal_ids` in `rs/sns/governance/src/governance.rs`.
2. In `distribute_rewards`, compute `reward_event_end_timestamp_seconds` first (already at line 5837), then pass it to `ready_to_be_settled_proposal_ids` instead of letting the function capture `now` internally.
3. This ensures only proposals whose voting deadline is ≤ `reward_event_end_timestamp_seconds` are settled in the current event, exactly mirroring the NNS guard.

## Proof of Concept
**Setup:**
- SNS with `round_duration_seconds = 604800` (7 days).
- Last reward event `end_timestamp_seconds = T₀`.
- Neuron A submits Proposal P with `initial_voting_period_seconds = 604801` at time `T₀ − 1`.
- Proposal P deadline = `T₀ − 1 + 604801 = T₀ + 604800`.

**Execution:**
1. At `T₀ + 604800 + 2`, the timer fires `should_distribute_rewards` → `true` (elapsed = 604801 > 604800).
2. `distribute_rewards` runs: `new_rounds_count = floor(604801 / 604800) = 1`; `reward_event_end_timestamp_seconds = T₀ + 604800`.
3. `ready_to_be_settled_proposal_ids()` evaluates `reward_status(now = T₀ + 604800 + 2)`: Proposal P deadline = `T₀ + 604800` < `now` → `accepts_vote = false` → `ReadyToSettle`.
4. Proposal P is included in `considered_proposals` and settled under the current event with `end_timestamp_seconds = T₀ + 604800`, even though P's deadline equals that boundary (i.e., P belongs to round 2).
5. Neuron A receives full proportional maturity for P in round 1. P's ballots are cleared permanently.
6. In round 2, P is already `Settled`; no correction is possible.

A deterministic integration test using PocketIC can reproduce this by setting mock time to `T₀ + 604800 + 2` and asserting that the resulting `RewardEvent.settled_proposals` contains P and that P's `reward_event_end_timestamp_seconds` equals `T₀ + 604800` despite P's deadline being `T₀ + 604800` (i.e., one second into the next round).

            .proposals

                let topic = proposal.topic();
                let voting_period_seconds = self.voting_period_seconds()(topic);
                let reward_status =
                    proposal.reward_status(as_of_timestamp_seconds, voting_period_seconds);

                reward_status == ProposalRewardStatus::ReadyToSettle
            })
            .map(|(k, _)| ProposalId { id: *k })
    }

    fn most_recent_fully_elapsed_reward_round_end_timestamp_seconds(&self) -> u64 {
        let now = self.env.now();
        let genesis_timestamp_seconds = self.heap_data.genesis_timestamp_seconds;

        if genesis_timestamp_seconds > now {
            println!(
                "{}WARNING: genesis is in the future: {} vs. now = {})",
                LOG_PREFIX, genesis_timestamp_seconds, now,
            );
            return 0;

            * REWARD_DISTRIBUTION_PERIOD_SECONDS // Convert back into seconds.

    }
