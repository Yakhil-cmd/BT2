Audit Report

## Title
Fee-on-Transfer ERC-20 Deposit Overmints ckERC20 Tokens - (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol, rs/ethereum/cketh/minter/src/deposit.rs)

## Summary
The ckERC20 deposit helper contracts emit the caller-supplied `amount` in the `ReceivedEthOrErc20` / `ReceivedErc20` log event rather than the actual tokens received by the minter's Ethereum address. The IC minter mints ckERC20 directly from the log's `amount` field without any balance reconciliation. For fee-on-transfer ERC-20 tokens, this causes the minter to mint more ckERC20 than the ERC-20 backing it holds, breaking the 1:1 peg and enabling an unprivileged user to drain other depositors' funds.

## Finding Description

**Helper contract — pre-fee amount emitted:**

In `DepositHelperWithSubaccount.sol` lines 519–531, `safeTransferFrom` is called with the caller-supplied `amount`, and the event is immediately emitted with that same `amount`:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```

For a fee-on-transfer token the minter's address receives `amount - fee`, but the event records `amount`. The identical pattern exists in `ERC20DepositHelper.sol` lines 498–503.

**Minter — mints directly from log value:**

`deposit.rs` `mint()` function (lines 73–81) calls the ICRC-1 ledger with `amount: event.value()`, where `event.value()` (defined in `eth_logs/mod.rs` lines 205–210) returns the raw `value` field parsed from the Ethereum log — the pre-fee amount. There is no balance-before/balance-after check anywhere in the deposit path.

**No existing guard:** The minter's `events_to_mint()` queue, the double-minting guard (`prevent_double_minting_guard`), and the `QuarantinedDeposit` path all operate on the log-derived value; none cross-check the minter's actual on-chain ERC-20 balance.

**Exploit flow:**
1. A fee-on-transfer ERC-20 token is added as a supported ckERC20 token via NNS governance proposal (the system is explicitly designed to accept arbitrary ERC-20 tokens).
2. Attacker calls `depositErc20(feeToken, 1000, principal, subaccount)`.
3. `safeTransferFrom` executes; token deducts fee; minter receives 990.
4. Event emitted: `ReceivedEthOrErc20(..., 1000, ...)`.
5. IC minter scrapes log, parses `value = 1000`, calls `client.transfer(TransferArg { amount: 1000, ... })`.
6. Minter mints 1000 ckERC20; minter's Ethereum address holds only 990 ERC-20 tokens.
7. Attacker repeats; excess ckERC20 accumulates. Attacker withdraws excess, draining ERC-20 tokens deposited by legitimate users.

## Impact Explanation

**High — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.** Each deposit of a fee-on-transfer token creates unbacked ckERC20 tokens. An attacker accumulates the excess and redeems it via `withdraw_erc20`, which burns ckERC20 and sends real ERC-20 tokens from the minter's Ethereum address. Once the minter's ERC-20 balance is exhausted, legitimate holders' withdrawals fail — direct, permanent loss of funds for other ckERC20 holders. The precondition (governance adding a fee-on-transfer token) constrains severity below Critical, but the exploit itself is fully unprivileged once the token is listed.

## Likelihood Explanation

**Medium.** The ckERC20 system is explicitly documented as supporting "any ERC-20 token on Ethereum" via NNS governance proposals, and multiple tokens have already been added (ckUSDC, ckUSDT, ckWBTC, ckXAUT, etc.). The governance participants reviewing a proposal have no on-chain mechanism to detect fee-on-transfer behavior. Once a fee-on-transfer token is listed, any unprivileged user triggers the overmint by calling `depositErc20` — no special access, no front-running, no victim interaction required. The attack is repeatable and cumulative.

## Recommendation

1. **Measure actual received amount on-chain:** In `depositErc20` / `deposit`, record the minter's ERC-20 balance before and after `safeTransferFrom` and emit `balanceAfter - balanceBefore` as the `amount` in the event instead of the caller-supplied parameter.

2. **Alternatively, enforce an explicit exclusion:** Document and enforce that fee-on-transfer tokens are not supported. Add an on-chain check in the helper contract (e.g., verify `balanceOf(minterAddress)` increased by exactly `amount`) or add a governance-enforced validation step that rejects such tokens before they are added to the supported list.

## Proof of Concept

**Minimal deterministic test plan (local fork / PocketIC):**

1. Deploy a mock ERC-20 token that deducts a 1% fee on every `transferFrom` (override `_transfer` to send `amount * 99 / 100` to recipient and burn the rest).
2. Register this token as a supported ckERC20 token in the minter state.
3. Call `depositErc20(mockFeeToken, 1_000_000, principal, subaccount)` from a test account.
4. Assert: Ethereum mock balance of minter address = 990_000.
5. Assert: ckERC20 minted to `principal` = 1_000_000 (overmint of 10_000).
6. Repeat N times; assert total ckERC20 supply > total ERC-20 held by minter.
7. Withdraw accumulated excess ckERC20; assert a legitimate depositor's subsequent withdrawal fails with insufficient balance.
