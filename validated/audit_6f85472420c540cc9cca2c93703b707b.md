### Title
Signature Bypass via `skip_stateful_validations` Allows Injection of Unauthorized Invoke Transactions for Undeployed Accounts — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (which performs cryptographic signature verification) for any invoke transaction with `nonce=1` targeting an account whose on-chain nonce is `0`, provided that **any** transaction for that address exists in the mempool. The check used — `account_tx_in_pool_or_recent_block` — does not verify that the incoming transaction is signed by the account owner. An attacker who observes a victim's `deploy_account` transaction in the mempool can inject an invoke transaction with `nonce=1` for the victim's address, carrying an arbitrary garbage signature, and have it admitted to the mempool without any signature check. When the batcher later executes this transaction, it reverts (signature check fails in `__validate__`), but the victim's nonce slot is consumed and a fee is charged.

---

**The authorization gap — `account_tx_in_pool_or_recent_block`:**

The mempool check used to decide whether to skip validation does not distinguish between a `deploy_account` transaction and any other transaction type. It returns `true` if **any** transaction for the address is present:

```rust

    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

**The stateless validator does not check signature cryptography:**

The stateless validator only checks signature *length*, not cryptographic validity:

```rust

**The fee check does not block the attack:**

`perform_pre_validation_stage` checks `verify_can_pay_committed_bounds` only when `charge_fee` is true. Since a `deploy_account` transaction requires the account address to be pre-funded before submission, the victim's address has a non-zero STRK balance. The fee check therefore passes for the attacker's transaction. [5](#0-4) 

**Batcher execution consumes the nonce:**

When the batcher later executes the attacker's transaction, `handle_nonce` increments the nonce **before** `__validate__` runs. The `__validate__` call then fails (wrong signature), the transaction reverts, but the nonce increment is permanent:

```rust
pub fn perform_pre_validation_stage<S: State + StateReader>(...) {
    Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
    // fee checks ...
    // validate_proof_facts ...
}
``` [5](#0-4) 

The batcher uses `new_for_sequencing` which always sets `validate: true`, so `__validate__` is called during execution and the bad signature causes a revert:

```rust

---

### Impact Explanation

An attacker can inject an invoke transaction carrying an arbitrary signature for any victim account that:
1. Has submitted a `deploy_account` transaction (nonce=0) currently in the mempool, and
2. Has pre-funded their address (required for `deploy_account` to pass fee checks).

The injected transaction is admitted to the mempool without signature verification. When the batcher executes it, the transaction reverts, but:
- The victim's nonce=1 slot is permanently consumed.
- The victim is charged a fee for the reverted transaction.
- The victim's own legitimate invoke with nonce=1 (the one the UX feature was designed to support) is blocked — it will be rejected as a duplicate nonce or displaced.

This directly breaks the deploy_account + invoke UX flow that `skip_stateful_validations` was designed to enable, and constitutes unauthorized admission of an invalid transaction.

**Impact category:** High — Mempool/gateway admission accepts invalid transactions (wrong signature) before sequencing.

---

### Likelihood Explanation

The attack requires:
1. Monitoring the public mempool for `deploy_account` transactions (trivially observable).
2. Submitting an invoke with `nonce=1` for the victim's address before the `deploy_account` is included in a block.

The window is the time between the victim's `deploy_account` entering the mempool and being committed to a block. This is a realistic, low-effort attack against any new account deployment.

---

### Recommendation

The `skip_stateful_validations` check must verify that the transaction in the mempool for the sender address is specifically a `deploy_account` transaction, not just any transaction. The `account_tx_in_pool_or_recent_block` API should be replaced with a type-specific check, e.g. `deploy_account_tx_in_pool_or_recent_block`, that only returns `true` when a `DeployAccount` transaction for the address is present.

Alternatively, the `skip_stateful_validations` path should be removed and replaced with a mechanism that defers signature validation to execution time while still protecting the nonce slot (e.g., by requiring the deploy_account and invoke to be submitted as an atomic bundle).

---

### Proof of Concept

1. Victim pre-funds address `0xVICTIM` and submits `deploy_account` (nonce=0) to the gateway. It enters the mempool.
2. Attacker calls `account_tx_in_pool_or_recent_block(0xVICTIM)` → returns `true`.
3. Attacker constructs an `RpcInvokeTransactionV3` with:
   - `sender_address = 0xVICTIM`
   - `nonce = 1`
   - `signature = [0x1, 0x2]` (arbitrary garbage, within `max_signature_length=4000`)
   - Valid resource bounds (sufficient to pass fee check against victim's pre-funded balance)
4. Attacker submits to gateway. Stateless validation passes (signature length OK). Stateful validation:
   - `account_nonce = 0`, `tx.nonce() = 1` → enters `skip_stateful_validations`
   - `account_tx_in_pool_or_recent_block(0xVICTIM)` → `true` (victim's deploy_account is in pool)
   - `skip_validate = true` → `__validate__` is skipped
   - Fee check passes (victim's address is pre-funded)
   - Transaction admitted to mempool.
5. Batcher executes: `deploy_account` runs first (nonce=0), then attacker's invoke (nonce=1). `handle_nonce` increments nonce to 2. `__validate__` runs, fails (wrong signature). Transaction reverts. Nonce=1 is consumed.
6. Victim's own invoke with nonce=1 is now rejected as `NonceTooOld` or displaced. Victim is charged a fee for the reverted transaction.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
