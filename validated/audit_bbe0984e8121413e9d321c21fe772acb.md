### Title
`_afterTimelock` uint32 truncation allows pool admin to bypass all timelocked parameter changes — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` computes `uint32(block.timestamp + oracleStopLossConfig[pool_].timelock)`. In Solidity ≥0.8 the addition is performed in `uint256` space (no revert), but the explicit `uint32(...)` cast silently truncates. When the sum exceeds `type(uint32).max` the stored `executeAfter` wraps to a value **smaller than the current `block.timestamp`**, so `_requireElapsed` passes immediately in the same block. There is no cap or validation on the `timelock` field anywhere in the contract.

---

### Finding Description

`_afterTimelock` is the sole source of every `executeAfter` timestamp written to `pendingTimelockExecuteAfter`, `pendingDrawdownExecuteAfter`, `pendingDecayExecuteAfter`, and `pendingHighWatermark.executeAfter`. [1](#0-0) 

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is safe in `uint256`, but the outer `uint32(...)` cast truncates mod 2³². No `_validateTimelock` function exists and no upper bound is enforced at `initialize` or `proposeOracleStopLossTimelock`. [2](#0-1) 

`_requireElapsed` compares `uint256 block.timestamp` against `uint32 executeAfter`; the `uint32` is promoted to `uint256` for the comparison: [3](#0-2) 

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

If `executeAfter` wrapped to a past value (e.g. `block.timestamp − 1`), the condition is `false` and execution proceeds without waiting.

**Concrete arithmetic (current mainnet):**

| Variable | Value |
|---|---|
| `block.timestamp` | ≈ 1,750,000,000 |
| `timelock` | `type(uint32).max` = 4,294,967,295 |
| Sum (uint256) | 6,044,967,295 |
| `uint32(sum)` | 6,044,967,295 mod 4,294,967,296 = **1,749,999,999** |
| `block.timestamp < 1,749,999,999` | **false → passes immediately** |

Any `timelock` value above `type(uint32).max − block.timestamp` (≈ 2.54 billion seconds, well within `uint32` range) triggers the wrap.

---

### Impact Explanation

The timelock is the sole LP-protection mechanism against pool-admin parameter changes. Bypassing it means:

- `drawdownE6` can be set to `0` (stop-loss disabled) in the same block as the proposal, with no LP reaction window.
- `decayPerSecondE8` can be set to `type(uint32).max` (watermarks decay to zero instantly), removing all value-loss protection.
- High watermarks can be reset to arbitrarily high values, causing every subsequent swap to revert (`OracleStopLossTriggered`), effectively freezing the pool.

The first two scenarios directly remove the fund-protection guarantees the extension is designed to enforce, satisfying the "admin-boundary break: pool admin bypasses timelocks" impact gate.

---

### Likelihood Explanation

The pool admin controls the `timelock` field with no upper-bound validation. A pool initialized with `timelock = 0` allows the admin to immediately execute `proposeOracleStopLossTimelock(pool, type(uint32).max)` → `executeOracleStopLossTimelock` in the same block (since `_afterTimelock` with `timelock=0` returns `uint32(block.timestamp)`, and `block.timestamp < block.timestamp` is false). After that, every subsequent proposal bypasses the delay. Alternatively, a pool initialized directly with a large `timelock` value is vulnerable from block 1.

---

### Recommendation

1. Validate the timelock at both `initialize` and `proposeOracleStopLossTimelock` against a safe maximum (e.g. 365 days = 31,536,000 seconds, well below the overflow threshold).
2. Replace the bare cast with a checked helper:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    require(result <= type(uint32).max, "timelock overflow");
    return uint32(result);
}
```

```solidity
pragma solidity ^0.8.35;

// Foundry test sketch
function test_timelockOverflowBypass() public {
    // 1. Deploy pool with timelock = 0
    address pool = factory.createPool(..., abi.encode(500_000, 58, 0));

    // 2. Admin immediately sets timelock to type(uint32).max
    ext.proposeOracleStopLossTimelock(pool, type(uint32).max);
    // executeAfter = uint32(block.timestamp + 0) = block.timestamp (same block passes)
    ext.executeOracleStopLossTimelock(pool);

    // 3. Admin proposes drawdown = 0 (disable stop-loss)
    ext.proposeOracleStopLossDrawdown(pool, 0);
    // executeAfter = uint32(block.timestamp + type(uint32).max) wraps to past timestamp

    // 4. Execute in the SAME block — no warp needed
    ext.executeOracleStopLossDrawdown(pool);

    // 5. Assert drawdown is now 0 (stop-loss disabled, no timelock waited)
    assertEq(ext.oracleStopLossConfig(pool).drawdownE6, 0);
}
```
