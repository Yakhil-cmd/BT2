### Title
WalletConnect Offer Confirmation Displays Understated Royalty Total for Multi-NFT Offers - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly calculates the total fungible amount (including royalties) shown to users in the WalletConnect confirmation dialog when an offer involves multiple NFTs. It divides the fungible amount by the number of NFTs before computing each NFT's royalty, causing the displayed total to be systematically lower than the amount actually spent on-chain.

### Finding Description
The `formatAmountWithRoyalties` function computes the royalty-inclusive total as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The `splitAmount` divides the full fungible amount by the number of NFTs before multiplying by each royalty percentage. This is structurally identical to the external bug: a loop over multiple entities (NFTs) that applies a calculation to a single divided value instead of the full value per entity.

The Chia protocol calculates each NFT's royalty on the **full** fungible amount. The correct formula is:

```
totalRoyalty = sum(fullAmount × royaltyPercentage_i / 10_000)
```

The current code computes instead:

```
totalRoyalty = sum((fullAmount / n) × royaltyPercentage_i / 10_000)
             = (1/n) × sum(fullAmount × royaltyPercentage_i / 10_000)
```

This is confirmed by `OfferBuilderProvider.tsx`, which passes the **full** fungible amounts to the `calculateRoyalties` RPC for each NFT: [2](#0-1) 

The `parseCommandDisplay` function is the WalletConnect confirmation path — it is called for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` commands and its output is what the user sees before approving: [3](#0-2) 

The `withRoyaltyTotals` function collects all royalty percentages from the opposite trade side and passes them to `formatAmountWithRoyalties`: [4](#0-3) 

### Impact Explanation
A user approving a WalletConnect offer that involves multiple NFTs (each with royalties) will see a displayed spending total that is **lower than the amount actually deducted from their wallet**. The on-chain transaction uses the correct (higher) royalty amounts; only the confirmation UI is wrong.

Concrete example — 2 NFTs with royalty percentages 500 (5%) and 10 (0.1%), fungible side 1 XCH:
- **Displayed** (current code): 1 + (0.5 × 0.05) + (0.5 × 0.001) = **1.0255 XCH**
- **Actual on-chain**: 1 + (1 × 0.05) + (1 × 0.001) = **1.051 XCH**
- **Hidden cost**: 0.0255 XCH per transaction

With higher royalty percentages (up to 100% per NFT is protocol-valid) and more NFTs, the gap grows proportionally. An attacker who minted NFTs with high royalty percentages and set the royalty address to their own wallet can systematically extract more XCH from victims than the confirmation dialog discloses.

This meets the **High** impact criterion: *"WalletConnect state that causes a user to approve the wrong amount."*

### Likelihood Explanation
- WalletConnect is an active, production-enabled feature.
- Any dApp or peer can submit a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command containing multiple NFTs.
- The attacker only needs to have minted NFTs with a high `royaltyPercentage` and set the royalty puzzle hash to their own address — both are standard, permissionless on-chain operations.
- No leaked keys, host compromise, or cryptographic break is required.
- The existing test at line 251 of `parseCommandDisplay.test.ts` encodes the incorrect value (`0.00010255`) as the expected output, confirming the bug is present and untested-against-correct-behavior. [5](#0-4) 

### Recommendation
Replace the `splitAmount` division with the full `amount` for each royalty calculation:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test expectation to reflect the correct value (1.051 XCH in the multi-NFT test case).

### Proof of Concept
1. Attacker mints two NFTs on mainnet with `royalty_percentage = 5000` (50%) each, with `royalty_puzzle_hash` pointing to attacker's wallet.
2. Attacker constructs a WalletConnect `chia_wallet.take_offer` payload offering both NFTs in exchange for 1 XCH.
3. Victim connects their wallet via WalletConnect and receives the offer confirmation.
4. The confirmation dialog calls `formatAmountWithRoyalties(line, 1_000_000_000_000n, [5000, 5000])`:
   - `splitAmount = 500_000_000_000n`
   - `royaltyAmount = (500_000_000_000 × 5000 / 10_000) × 2 = 500_000_000_000`
   - Displayed total: **2 XCH**
5. Correct on-chain total: `1 + (1 × 0.5) + (1 × 0.5)` = **2 XCH** — in this symmetric case the numbers happen to match, but with asymmetric percentages (e.g., 5000 and 10) the gap is clear:
   - Displayed: `1 + (0.5 × 0.5) + (0.5 × 0.001)` = **1.2505 XCH**
   - Actual: `1 + (1 × 0.5) + (1 × 0.001)` = **1.501 XCH**
   - Victim approves believing they spend 1.2505 XCH; wallet deducts 1.501 XCH; attacker receives 0.501 XCH in royalties vs. the 0.2505 XCH the victim expected to pay.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

      throw new Error('Offer is not valid');

    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
