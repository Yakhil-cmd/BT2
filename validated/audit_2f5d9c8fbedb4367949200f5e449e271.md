### Title
Unrestricted `execute_scheduled` Allows Premature Promise Execution, Permanently Freezing User NEAR - (File: `etc/xcc-router/src/lib.rs`)

### Summary
The `execute_scheduled` function in the XCC router contract has no caller restriction. Any external account can trigger execution of a user's scheduled promise before the user intends it to run. If the promise fails at that premature moment, the NEAR that was pre-funded into the router for that call is refunded back to the router contract and becomes permanently irrecoverable, because the router exposes no general NEAR withdrawal mechanism.

### Finding Description
When a user invokes the XCC precompile with `CrossContractCallArgs::Delayed`, the Aurora Engine orchestrates two sequential NEAR actions:

1. `withdraw_wnear_to_router` — unwraps the user's wNEAR and deposits the resulting NEAR into the router sub-account (`{evm_address}.aurora`).
2. `schedule` — stores the `PromiseArgs` (including `attached_balance`) in the router's `scheduled_promises` map under an auto-incremented nonce, and emits a log: `"Promise scheduled at nonce {nonce}"`. [1](#0-0) 

The stored promise is later meant to be executed by calling `execute_scheduled`. The function is intentionally open to any caller: [2](#0-1) 

The developer comment asserts "there is no security risk" because the function only acts on promises created via `schedule`. This reasoning is incomplete: it ignores the timing dimension. Executing the promise before the target contract is ready (or before the user intends) can cause the promise to fail. When it fails, NEAR is refunded to the router (the promise creator), but the router has no general withdrawal path for arbitrary NEAR balances. The only NEAR-transfer method is `send_refund`, which is parent-only and hardcoded to exactly `REFUND_AMOUNT` (2 NEAR): [3](#0-2) 

The nonce is publicly broadcast in a NEAR log at schedule time, so any observer can identify and race to execute any pending promise. [4](#0-3) 

The NEAR pre-funding happens unconditionally before `schedule` is called as a callback: [5](#0-4) 

### Impact Explanation
**Permanent freezing of funds.** If an attacker calls `execute_scheduled(nonce)` before the target contract is ready to accept the call, the promise fails. NEAR that was pre-deposited into the router (potentially many NEAR, depending on the user's XCC call) is refunded to the router's own account balance. Because the router exposes no mechanism to withdraw an arbitrary NEAR balance, those funds are permanently locked. The user cannot re-schedule the same promise (it was removed from storage by `execute_scheduled`), and cannot recover the NEAR through any router interface.

### Likelihood Explanation
**Medium.** The attack requires no special privilege. The nonce of every scheduled promise is emitted as a public NEAR log. An attacker monitoring the chain can observe any `schedule` call, read the nonce, and immediately submit `execute_scheduled` targeting that nonce. On NEAR, transaction finality is fast (~1–2 seconds), making a race-condition or frontrun straightforward. The attack is cheap (only gas cost) and can be repeated for every promise any user schedules. The scenario where a promise fails on premature execution is realistic: the user may be scheduling a call that depends on a prior on-chain state change, a time condition, or a contract that has not yet been initialized.

### Recommendation
Remove the open-access design of `execute_scheduled`. Add a caller check so only the parent Aurora Engine account (or the user's own EVM-derived account) can trigger execution:

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    let parent = self.get_parent().unwrap_or_else(env_panic);
    require_caller(&parent).unwrap_or_else(env_panic);
    // ... rest of function
}
```

Alternatively, if open execution is desired for composability, add a failure-recovery path: attach a callback to the created promise that, on failure, re-inserts the promise back into `scheduled_promises` so the user can retry.

### Proof of Concept

1. User (EVM address `0xABCD`) calls the XCC precompile with `CrossContractCallArgs::Delayed`, scheduling a call to `target.near` with 10 NEAR attached.
2. Aurora Engine calls `withdraw_wnear_to_router` → 10 NEAR lands in the router account `abcd...{40hex}.aurora`.
3. Aurora Engine calls `schedule(promise)` on the router → promise stored at nonce `0`; log emitted: `"Promise scheduled at nonce 0"`.
4. Attacker reads the log and immediately calls `execute_scheduled({"nonce": "0"})` on the router from any NEAR account.
5. The router removes the promise from storage and fires the cross-contract call to `target.near` with 10 NEAR attached.
6. `target.near` rejects the call (e.g., it requires a prior setup step the user has not yet performed).
7. NEAR runtime refunds 10 NEAR to the router account.
8. The promise entry is gone from `scheduled_promises`; the user cannot re-execute it.
9. The router has no `withdraw` or general refund function; the 10 NEAR is permanently locked. [6](#0-5) [7](#0-6)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };

        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
