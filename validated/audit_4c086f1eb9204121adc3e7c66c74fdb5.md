### Title
DKG Protocol Permanently Stalls Honest Parties When One Participant Aborts Due to Unimplemented Failure-Signaling in `broadcast_success` - (File: src/dkg.rs)

### Summary

The `broadcast_success` function in `src/dkg.rs` is documented to broadcast a failure signal when a participant detects a local error, but the implementation always broadcasts `(true, session_id)` unconditionally and does not even accept an error parameter. When a malicious participant sends a targeted invalid share to one honest party, that honest party aborts before reaching `broadcast_success` and never sends its final-round broadcast message. All remaining honest parties then stall indefinitely inside `reliable_broadcast_receive_all`, which requires all `n` sessions to complete before returning. This permanently denies DKG completion for all honest parties.

### Finding Description

**Root cause — docstring-implementation mismatch in `broadcast_success`:**

The function's own docstring states:

> "This function takes err as input. If err is None then broadcast success, otherwise, broadcast failure."

But the actual signature and body are:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
``` [1](#0-0) 

There is no `err` parameter. The function unconditionally broadcasts `(true, session_id)`. The failure-signaling path described in the docstring was never implemented.

**Attack path — malicious participant sends a targeted bad share:**

In `do_keyshare`, Round 5 receives private shares from every other participant and verifies each one:

```rust
for (from, signing_share_from) in
    recv_from_others(&chan, wait_round_3, &participants, me).await?
{
    let full_commitment_from = all_full_commitments.index(from)?;
    validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
    my_signing_share = my_signing_share + signing_share_from.to_scalar();
}
broadcast_success(&mut chan, &participants, me, session_id).await?;
``` [2](#0-1) 

A malicious participant B can:
1. Publish a valid polynomial commitment (passes the Round 3 proof-of-knowledge check for everyone).
2. Send the correct share evaluation to participants C, D, E (their `validate_received_share` passes).
3. Send a deliberately wrong share to honest participant A (A's `validate_received_share` returns `ProtocolError::InvalidSecretShare(B)` and the function returns early).

Because A returns an error at `validate_received_share`, A **never reaches** `broadcast_success` and **never sends** its final-round broadcast message.

**Why the remaining honest parties stall permanently:**

`broadcast_success` calls `do_broadcast`, which calls `reliable_broadcast_receive_all`. That function runs `n` independent echo-broadcast sessions (one per participant) and terminates only when every session has `finish_ready = true`:

```rust
if state.iter().all(|x| x.finish_ready) {
    return Ok(vote_output);
}
``` [3](#0-2) 

A session's `finish_ready` is set only after accumulating more than `2 * ready_t` READY votes, which requires the session's sender to have first sent a SEND message. Since A never sends its SEND message, no participant echoes it, no READY votes accumulate for A's session, and `finish_ready` for A's slot is never set. The loop in `reliable_broadcast_receive_all` has no timeout and no early-exit path for a missing sender:

```rust
loop {
    if !is_simulated_vote {
        match chan.recv(wait).await {
            Ok(value) => (from, (sid, vote)) = value,
            _ => continue,
        };
    }
    ...
}
``` [4](#0-3) 

The early-termination check for a malicious sender (the `is_enough` block) is only triggered when ECHO messages arrive for a session. Because A never sent a SEND message, no ECHO messages for A's session ever arrive, so this check is never evaluated for A's session. C, D, E block forever. [5](#0-4) 

### Impact Explanation

A single malicious DKG participant can permanently prevent all honest parties from completing key generation, resharing, or refresh. The honest parties that received valid shares from the malicious participant reach `broadcast_success` and block indefinitely waiting for the aborted party's broadcast message. No key is ever produced. This matches the **High** impact category: *Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs and documented trust assumptions.*

### Likelihood Explanation

**Moderate.** Any participant in a DKG, reshare, or refresh session can execute this attack. The attacker only needs to:
- Participate honestly through Round 3 (publish a valid commitment and proof of knowledge).
- Send a single deliberately wrong share to one chosen honest party in Round 4.

No cryptographic break, no external compromise, and no elevated privilege are required. The shares are sent privately (point-to-point), so the malicious participant can trivially target exactly one victim while remaining undetected by all others.

### Recommendation

Implement the failure-signaling path that the docstring already describes but the code omits:

1. Add an `err: Option<ProtocolError>` parameter to `broadcast_success`.
2. Broadcast `(false, session_id)` when `err` is `Some(_)`, and `(true, session_id)` when `err` is `None`.
3. In `do_keyshare`, catch local verification errors (e.g., from `validate_received_share`) instead of propagating them immediately, pass the captured error to `broadcast_success`, and only then return the error after the broadcast completes.
4. After the broadcast, if any participant's vote is `false`, all participants return an error identifying the failed round.

This ensures that even when a participant detects a local failure, it still participates in the final broadcast round, allowing all honest parties to learn about the failure and abort cleanly rather than stalling indefinitely.

### Proof of Concept

**Setup:** 4 participants — A (honest victim), B (malicious), C, D (honest bystanders). Threshold = 2.

1. **Round 1–3 (honest):** B participates honestly. B publishes a valid polynomial commitment and proof of knowledge. All participants accept B's commitment.

2. **Round 4 (targeted attack):** B sends the correct share `f_B(C)`, `f_B(D)` to C and D. B sends a deliberately wrong value `f_B(A) + 1` to A.

3. **Round 5 — A aborts:** A calls `validate_received_share(me=A, from=B, share=f_B(A)+1, commitment_B)`. The share does not satisfy the commitment polynomial, so `secret_share.verify()` returns `Error::InvalidSecretShare`. `validate_received_share` returns `ProtocolError::InvalidSecretShare(B)`. `do_keyshare` propagates this error and returns. **A never calls `broadcast_success`.**

4. **Round 5 — C, D stall:** C and D pass all share verifications and call `broadcast_success`. Inside `reliable_broadcast_receive_all`, they process their own simulated SEND messages and wait for SEND messages from A and B. B sends its SEND message. A never does. The termination condition `state.iter().all(|x| x.finish_ready)` is never satisfied. C and D block indefinitely. [2](#0-1) [6](#0-5) [7](#0-6)
