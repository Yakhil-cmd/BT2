### Title
`uint32` Truncation of `block.timestamp + timelock` in `_afterTimelock` Allows Pool Admin to Bypass OracleValueStopLoss Timelocks Instantly — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` computes `uint32(block.timestamp + oracleStopLossConfig[pool_].timelock)`. Because `block.timestamp` is `uint256` and `timelock` is `uint32`, the addition is performed in `uint256` without overflow, but the final cast to `uint32` silently truncates the high bits whenever the sum exceeds `type(uint32).max` (4,294,967,295). The truncated result is a value far in the past, so `_requireElapsed` passes immediately. No validation on the `timelock` parameter exists in either `initialize` or `proposeOracleStopLossTimelock`, making the overflow reachable with any `uint32` value above `type(uint32).max - block.timestamp`.

---

### Finding Description

`_afterTimelock` is the single function that computes the `executeAfter` timestamp for every timelocked admin action in `OracleValueStopLossExtension`: drawdown changes, decay changes, watermark overrides, and timelock changes themselves. [1](#0-0) 

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` (~1,753,000,000 at the time of writing) plus any `timelock` value greater than ~2,541,967,295 seconds (~80.6 years) produces a `uint256` sum that exceeds `type(uint32).max`. The cast to `uint32` wraps the result to a small number (e.g., `1,753,000,000 + 2,542,000,000 = 4,295,000,000 → uint32 → 32,704`).

`_requireElapsed` then compares `block.timestamp` (uint256, ~1.75 B) against this wrapped `uint32` value: [2](#0-1) 

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

`1,753,000,000 < 32,704` is false, so the revert never fires and the timelock is bypassed in the same block as the proposal.

The `timelock` parameter is accepted without any upper-bound validation in `initialize`: [3](#0-2) 

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// no _validateTimelock call
```

And equally unvalidated in `proposeOracleStopLossTimelock`: [4](#0-3)

### Impact Explanation

The `OracleValueStopLossExtension` is the LP-facing safety guard that blocks swaps when per-bin value per share falls below a drawdown floor. Its timelocks exist specifically to give LPs a reaction window before the pool admin can weaken or disable that guard. With the overflow, the pool admin can:

1. Set `drawdownE6 = 0` (disabling the stop-loss entirely) in a single block — `_afterSwapOracleStopLoss` returns early when `drawdown == 0`.
2. Set `decayPerSecondE8` to `E8` (maximum, 100%/second) so watermarks decay to zero within one second, making the guard permanently ineffective.
3. Override per-bin watermarks to arbitrarily low values, allowing immediate extraction without triggering the stop-loss.

All three actions remove the only on-chain protection against LP value extraction via price manipulation, constituting broken core pool functionality and a direct admin-boundary break (bypassing timelocks).

---

### Likelihood Explanation

The pool admin must supply an overflowing `timelock` value at pool creation or propose one after the fact. An overflowing value (~2.54 B seconds, ~80 years) appears to LPs as an extremely protective, near-permanent timelock, making it a plausible deception vector. The factory imposes no minimum or maximum on the `timelock` field, and no off-chain tooling can distinguish an overflowing value from a legitimate long timelock without computing the truncation manually.

---

### Recommendation

Cap `timelock` to a safe maximum (e.g., `type(uint32).max / 2` or a protocol-defined constant such as 365 days) and perform the arithmetic in `uint256` before casting:

```solidity
uint256 private constant MAX_TIMELOCK = 365 days;

function _validateTimelock(uint256 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}

function _afterTimelock(address pool_) private view returns (uint256) {
    return block.timestamp + oracleStopLossConfig[pool_].timelock;
}
```

Store and compare `executeAfter` as `uint256` throughout `PoolStopLossSchedule` to eliminate the truncation entirely.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Assume pool is already deployed with OracleValueStopLossExtension initialized
// with timelock = 2_542_000_000 (fits in uint32, looks like ~80 years to LPs).

// Step 1: Pool admin proposes drawdown = 0 (disable stop-loss).
//   _afterTimelock returns uint32(block.timestamp + 2_542_000_000)
//   = uint32(1_753_000_000 + 2_542_000_000)
//   = uint32(4_295_000_000)
//   = 32_704   ← wrapped past timestamp
extension.proposeOracleStopLossDrawdown(pool, 0);

// Step 2: Pool admin immediately executes in the same block.
//   _requireElapsed(32_704): block.timestamp (1.75B) < 32_704 → false → no revert
extension.executeOracleStopLossDrawdown(pool);

// Result: drawdownE6 == 0; _afterSwapOracleStopLoss returns early on every swap.
// Stop-loss is permanently disabled. Pool admin can now manipulate prices
// and extract LP value without the guard triggering.
```
