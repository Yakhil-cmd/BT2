### Title
Truncating Integer Division in `native_per_pubdata` Computation Allows Pubdata Cost Bypass — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The ZKsync OS bootloader computes `native_per_pubdata` — the native resource cost per byte of pubdata — using truncating integer division (`wrapping_div`). When `pubdata_price < native_price`, this division rounds down to zero, making pubdata effectively free for any transaction sender. The same block-level parameters that cause this condition are valid in production. The analogous `native_per_gas` computation correctly uses ceiling division (`div_ceil`) to prevent the same class of bug, but `native_per_pubdata` does not.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (the L2 ZK transaction validation path), the per-pubdata-byte native cost is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs truncating (floor) integer division. When `pubdata_price < native_price`, the result is `0`. [1](#0-0) 

By contrast, `native_per_gas` uses ceiling division to guarantee a non-zero result whenever `gas_price > 0`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

When `native_per_pubdata == 0`, the downstream pubdata charging functions compute a zero native cost for any amount of pubdata:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // = 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

This zero propagates through `get_resources_to_charge_for_pubdata` and `check_enough_resources_for_pubdata`, meaning the pubdata sufficiency check always passes and no native resources are deducted for pubdata, regardless of how many bytes the transaction writes to state. [4](#0-3) 

The same truncating division is also present in the external API helper `validate_l2_tx_intrinsic_native_resources`:

```rust
// api/src/helpers.rs:427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [5](#0-4) 

---

### Impact Explanation

When `pubdata_price < native_price` (a valid block configuration), any unprivileged transaction sender can submit a transaction that generates an arbitrarily large amount of pubdata (e.g., many storage slot writes) and pay **zero native resources** for it. The pubdata is still published to L1 — the real settlement-layer cost is borne by the protocol/operator, not the sender. This is a resource accounting bug: the sender's fee does not reflect the actual L1 data-availability cost their transaction imposes.

The `create_resources_for_tx` function also uses `native_per_pubdata` to pre-charge intrinsic pubdata overhead; when it is zero, no native budget is withheld for pubdata at all:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs:352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
``` [6](#0-5) 

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a normal operational state. The default test metadata already sets `pubdata_price = 0` and `native_price = 10`, producing `native_per_pubdata = 0`: [7](#0-6) 

Any block where the operator sets `pubdata_price` to a value smaller than `native_price` (e.g., during low L1 congestion) triggers this condition. An attacker monitoring block parameters can time their pubdata-heavy transactions to coincide with such blocks.

---

### Recommendation

Replace the truncating division with ceiling division for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, ensuring non-zero cost when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. If `pubdata_price == 0` is an intentional "free pubdata" configuration, add an explicit guard (`if pubdata_price.is_zero() { 0 } else { div_ceil(...) }`) rather than relying on silent truncation.

---

### Proof of Concept

1. Operator sets block with `pubdata_price = 5`, `native_price = 10`.
2. `native_per_pubdata = 5 / 10 = 0` (truncating division).
3. Attacker submits a ZK L2 transaction that writes to 1000 distinct storage slots (generating ~66 KB of pubdata).
4. `get_resources_to_charge_for_pubdata` returns `(66000, Resources::from_native(0))`.
5. `check_enough_resources_for_pubdata` returns `(true, ...)` — the transaction passes the pubdata check with zero native cost.
6. The transaction executes successfully; the attacker pays only gas fees, not pubdata fees.
7. The operator must pay the L1 data-availability cost for 66 KB of pubdata out of their own pocket.

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
