### Title
Unbounded Loop in `calculate_rewards` Over Attacker-Grown Balance Trace Can Permanently Freeze Pool Member's Unclaimed Yield - (File: `src/pool/pool.cairo`)

### Summary

The `calculate_rewards` internal function in the `Pool` contract iterates over a pool member's entire `pool_member_epoch_balance` trace in a single unbounded loop. A pool member (unprivileged delegator) can grow this trace by making balance changes across many distinct epochs. If the trace grows large enough before `claim_rewards` is called, the transaction will exceed Starknet's block gas limit, permanently freezing the pool member's unclaimed yield with no recovery path.

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` (lines 837–888) iterates over every entry in the `pool_member_epoch_balance` trace from `entry_to_claim_from` up to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

The developers themselves acknowledge the loop is unbounded (lines 857–858). Each iteration performs multiple storage reads via `pool_member_trace.at(...)` and `find_sigma(...)`, making each iteration expensive in Starknet gas.

The trace grows via `set_member_balance` (line 728), which inserts a new checkpoint at `current_epoch + K` whenever the key differs from the last stored key:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(
        balance: amount,
        cumulative_rewards_trace_idx: self.cumulative_rewards_trace_length() + 1,
    );
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
```

`set_member_balance` is called by:
- `enter_delegation_pool` (line 201) — initial entry
- `add_to_delegation_pool` (line 242) — adding stake
- `exit_delegation_pool_intent` (line 278) — initiating exit

Each call at a different epoch appends a new entry. The `insert` function in `trace.cairo` (lines 166–172) only overwrites the last entry if the epoch key matches; otherwise it pushes a new checkpoint. So one new entry is added per epoch in which a balance change occurs.

The `entry_to_claim_from` pointer (stored in `pool_member_info`) is only advanced on a **successful** `claim_rewards` call (line 358: `pool_member_info.entry_to_claim_from = updated_entry_to_claim_from`). There is no partial-batch mechanism. If `claim_rewards` reverts due to gas exhaustion, `entry_to_claim_from` is not updated, and every subsequent call will attempt the same full iteration and also revert.

### Impact Explanation

A pool member who makes balance changes across N distinct epochs without claiming rewards will have N entries in their trace. When they eventually call `claim_rewards`, the loop must process all N entries in a single transaction. If N is large enough to exceed Starknet's block gas limit, `claim_rewards` reverts. Because there is no partial-claim mechanism and `entry_to_claim_from` is only updated on success, the pool member's accumulated unclaimed yield is **permanently frozen** — they can never recover it.

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

Any pool member (unprivileged delegator) who:
1. Actively adjusts their delegation (add/exit-intent) across many epochs, AND
2. Does not call `claim_rewards` frequently

will accumulate a large trace. This is a realistic pattern for delegators who manage their position over months or years. The growth rate is bounded at one entry per epoch, but over a long protocol lifetime (hundreds of epochs) the trace can grow to a size that makes `claim_rewards` prohibitively expensive. The developers' own comment acknowledges the loop is unbounded and relies on the assumption that gas limits will not be hit — an assumption that weakens over time as the protocol matures.

### Recommendation

Introduce a `max_entries` parameter to `claim_rewards` (or a separate `claim_rewards_partial` entrypoint) that limits the number of trace entries processed per call and advances `entry_to_claim_from` incrementally. This allows pool members to drain their trace across multiple transactions, preventing permanent reward lockup.

Alternatively, cap the number of balance-change entries that can accumulate without a claim (e.g., enforce a maximum trace length per pool member and require a claim before further balance changes are allowed).

### Proof of Concept

1. Pool member `A` enters a delegation pool in epoch 0.
2. In each subsequent epoch `i` (for `i = 1 … N`), `A` calls `add_to_delegation_pool` with a minimal amount. Each call at a new epoch appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: epoch_i + K, ...)`.
3. `A` never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After N epochs, `A` calls `claim_rewards`. The loop in `calculate_rewards` iterates from index 0 to N, performing storage reads and `find_sigma` calls on each iteration.
5. For sufficiently large N, the transaction exceeds the Starknet block gas limit and reverts.
6. Every subsequent `claim_rewards` call also reverts (same starting index, same N iterations). `A`'s accumulated rewards are permanently inaccessible.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
