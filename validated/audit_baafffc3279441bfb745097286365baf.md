### Title
Unverified Participant Contributions Included in CKD Aggregation Corrupt the Derived Confidential Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator blindly sums `(norm_big_y, norm_big_c)` shares received from every participant without any proof of correctness. A single malicious participant can send arbitrary group elements that are unconditionally added into the aggregation, producing a corrupted `CKDOutput` that decrypts to a wrong confidential key. This is a direct analog of the external report's pattern: unverifiable state is included in a total calculation, corrupting the output for all honest parties.

### Finding Description

In `do_ckd_coordinator` the coordinator collects one `CKDOutput` per participant and adds each field directly into the running sum:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to compute, in `compute_signature_share`:

- `norm_big_y = lambda_i * y_i * G`
- `norm_big_c = lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)` [2](#0-1) 

There is **no zero-knowledge proof, commitment, or any other verification** that the received `(norm_big_y, norm_big_c)` pair was honestly computed from the participant's actual signing share `x_i` and a fresh random `y_i`. The coordinator has no mechanism to detect a deviation.

The correct final output satisfies:

```
big_C - app_sk * big_Y  =  msk * H(pk, app_id)
```

If participant `j` sends an arbitrary pair `(big_y', big_c')` instead of its honest contribution, the coordinator computes:

```
big_Y_corrupt = honest_sum_Y + big_y'
big_C_corrupt = honest_sum_C + big_c'
```

`unmask(app_sk)` then returns `msk * H(pk, app_id) + (big_c' - app_sk * big_y')`, which is an attacker-controlled offset from the correct key. The output is accepted by the coordinator as a valid `CKDOutput` with no error. [3](#0-2) 

### Impact Explanation

The derived confidential key is silently wrong. Any TEE application or downstream consumer that uses the `CKDOutput` will operate on an incorrect secret. Because the coordinator returns the corrupted value as a successful result, honest parties have no indication that the output is invalid. This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs**.

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker only needs to be a registered participant — no special privilege is required. `recv_from_others` waits for **all** other participants, so one malicious actor is sufficient to corrupt the result. The attack requires no cryptographic break; the attacker simply sends two arbitrary `ElementG1` values.

### Recommendation

Each participant should accompany its `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that:

1. `norm_big_y` is `lambda_i * y_i * G` for some `y_i` known to the sender, and
2. `norm_big_c` is `lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)` for the same `y_i` and the participant's committed `x_i`.

A standard sigma protocol (e.g., a Chaum-Pedersen proof of discrete-log equality) over the BLS12-381 group suffices. The coordinator must verify each proof before adding the contribution to the running sum, rejecting and aborting if any proof fails.

### Proof of Concept

1. Run a CKD session with `N = 3` participants where participant `P_2` is malicious.
2. `P_2` intercepts the normal protocol flow and, instead of calling `compute_signature_share`, sends `(big_y' = G, big_c' = G)` (the generator point) to the coordinator.
3. The coordinator's loop at lines 50-55 adds these values unconditionally.
4. The coordinator returns a `CKDOutput` with `big_Y_corrupt` and `big_C_corrupt`.
5. Calling `ckd_output.unmask(app_sk)` yields `msk * H(pk, app_id) + (G - app_sk * G)` — a value that differs from the correct confidential key by an attacker-controlled offset.
6. No error is raised anywhere in the protocol; the corrupted output is indistinguishable from a correct one to the coordinator and any downstream consumer.

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
