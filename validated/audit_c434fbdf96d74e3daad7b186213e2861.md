### Title
`exit_to_near` Precompile Permanently Loses User Funds When NEAR Transfer Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The `exit_to_near` precompile burns/deducts user tokens from the EVM side and schedules a NEAR `ft_transfer` promise. If that promise fails (e.g., recipient not registered with the NEP-141 token), the refund path is gated entirely behind the `error_refund` compile-time feature flag. When that feature is absent, no `RefundCallArgs` is ever populated, the callback has nothing to refund, and the user's funds are permanently destroyed.

---

### Finding Description

When a user calls the `exit_to_near` precompile (either for ETH or ERC-20 tokens), the following sequence occurs:

1. Tokens are burned/deducted from the user's EVM balance.
2. A NEAR `ft_transfer` promise is scheduled.
3. A callback (`exit_to_near_precompile_callback`) is registered to handle the result.

The refund logic inside that callback is:

```rust
// engine/src/contract_methods/connector.rs, lines 231–239
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None
};
```

`args.refund` is only ever `Some(...)` if the `error_refund` feature is compiled in. The function that populates it is:

```rust
// engine-precompiles/src/native.rs, lines 699–725
#[cfg(feature = "error_refund")]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs { ... })
}
```

Without `error_refund`, this function does not exist, `args.refund` is always `None`, and the `else { None }` branch is taken — silently discarding the failure with no refund.

The engine test suite explicitly documents this behavior:

```rust
// engine-tests/src/tests/erc20_connector.rs, lines 656–660
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

And for the ETH case:

```rust
// engine-tests/src/tests/erc20_connector.rs, lines 771–775
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

---

### Impact Explanation

**Critical — Permanent loss of user funds.**

Any user who calls `exit_to_near` and whose NEAR-side `ft_transfer` fails (e.g., unregistered recipient, insufficient storage deposit, paused token contract) permanently loses the full exited amount. The tokens are burned from the EVM ledger and never arrive on NEAR. There is no recovery path.

This is structurally identical to the Fontaine bug: a resource is consumed/reserved at the start of an operation, and the cleanup/reclaim step is simply absent.

---

### Likelihood Explanation

**High.** The failure condition (`ft_transfer` to an unregistered NEAR account) is trivially reachable by any unprivileged user — no special permissions or admin access required. A user simply needs to specify a NEAR recipient that is not registered with the NEP-141 token contract. This is a common mistake. Any deployment of Aurora Engine compiled without the `error_refund` feature is permanently vulnerable to this loss for every failed exit.

---

### Recommendation

1. Enable the `error_refund` feature unconditionally in all production builds, or promote the refund logic out of the feature flag entirely so it is always compiled in.
2. Alternatively, if the feature flag must remain, add a compile-time assertion or runtime guard that prevents `exit_to_near` from executing at all when `error_refund` is absent, rather than silently discarding failures.
3. Ensure the `ExitToNearPrecompileCallbackArgs::refund` field is always populated on the failure path regardless of feature flags.

#[cfg(feature = "error_refund")]
#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}
