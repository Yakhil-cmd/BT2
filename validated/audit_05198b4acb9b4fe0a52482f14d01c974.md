### Title
Gateway L2 Gas Price Admission Uses Stale Previous-Block Price Instead of Next-Block Price — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` gates mempool admission by comparing a transaction's `max_price_per_unit` against the **previous block's** L2 gas price. The transaction will actually execute in the **next** block, whose price is computed by the EIP-1559 formula in `calculate_next_base_gas_price`. Because the two prices diverge every time gas usage deviates from the target, the gateway systematically admits transactions that will fail at execution (rising price) and rejects transactions that would succeed (falling price). The code itself acknowledges the defect with an unfixed TODO.

---

### Finding Description

`validate_resource_bounds` reads the previous block's L2 gas price from `gateway_fixed_block_state_reader.get_block_info()` and passes it to `validate_tx_l2_gas_price_within_threshold`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold check then enforces `tx_l2_gas_price >= (min_gas_price_percentage / 100) * previous_block_l2_gas_price`:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold { ... }
``` [2](#0-1) 

The default `min_gas_price_percentage` is `100`, so the threshold equals the previous block's price exactly. [3](#0-2) 

The actual price for the next block is computed by `calculate_next_base_gas_price` using the EIP-1559 formula with `gas_price_max_change_denominator = 48` (≈ 2.08 % max change per block):

```rust
let price_change = (price_u256 * gas_delta) / denominator;
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [4](#0-3) [5](#0-4) 

The gateway's `run_validate_entry_point` also builds a `BlockContext` from the previous block's info (only incrementing the block number, not the gas price), so the blockifier's own `check_fee_bounds` inside gateway validation also uses the stale price:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
// gas_prices are NOT updated — they remain from the previous block
let block_context = BlockContext::new(block_info, ...);
``` [6](#0-5) 

At actual execution time, the blockifier enforces `max_price_per_unit >= actual_block_price`:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [7](#0-6) 

This creates two divergent admission paths:

**Path A — price rising (congested blocks, `gas_used > gas_target`):**
`next_block_price > prev_block_price`. A transaction with `max_price = prev_block_price` passes the gateway threshold (= 100 % of `prev_block_price`) but fails at execution because `max_price < next_block_price`. The gateway admits a transaction that will be rejected by the blockifier.

**Path B — price falling (under-utilized blocks, `gas_used < gas_target`):**
`next_block_price < prev_block_price`. A transaction with `max_price` in `[next_block_price, prev_block_price)` is rejected by the gateway even though it would succeed at execution. Valid transactions are incorrectly excluded from the mempool.

The analog to the LP token bug is exact: Napier used a static multiplier of 3 instead of the current virtual price; here the gateway uses the previous block's price instead of the current (next) block's price. Both substitute a stale scalar for a dynamically evolving value, causing systematic mispricing of admission decisions.

---

### Impact Explanation

**False admission (rising price):** Transactions admitted to the mempool will fail at blockifier pre-validation with `MaxGasPriceTooLow`, wasting mempool slots and batcher processing cycles. Under sustained congestion the price rises every block, so this is a continuous stream of admitted-but-doomed transactions.

**False rejection (falling price):** Legitimate transactions whose `max_price` is between `next_block_price` and `prev_block_price` are permanently rejected at the gateway. Users receive a `GAS_PRICE_TOO_LOW` error for a transaction that would have been accepted and executed correctly. This is a denial-of-service on valid transactions during any period of decreasing gas demand.

Both effects are triggered by any unprivileged user submitting a transaction during normal network operation. The magnitude is bounded by the EIP-1559 max change per block (~2.08 % with denominator 48), but the condition is permanent and self-reinforcing under sustained congestion or sustained low usage.

Impact category: **High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The condition fires whenever `gas_used ≠ gas_target`, which is the normal operating state of any non-perfectly-balanced block. No special privileges, no adversarial setup, and no race condition are required. Any user submitting a transaction at the boundary of the previous block's price triggers either path. The TODO comment in the source confirms the developers are aware the reference price is wrong.

---

### Recommendation

Replace the reference price in `validate_resource_bounds` with the **next block's** L2 gas price. The next block's price is already computed by `calculate_next_base_gas_price` / `calculate_next_l2_gas_price_for_fin` in the consensus orchestrator and stored in the block header as `next_l2_gas_price`. The gateway should read this field from the latest committed block header rather than re-reading `l2_gas_price` from the current block's gas price vector.

Concretely, `GatewayFixedBlockStateReader::get_block_info` (or a new method) should expose `next_l2_gas_price` from the block header, and `validate_resource_bounds` should use that value as the threshold reference instead of `strk_gas_prices.l2_gas_price`.

The same fix should be applied to `run_validate_entry_point`: the `BlockContext` built for gateway-side blockifier validation should carry the **next** block's gas prices, not the previous block's prices.

---

### Proof of Concept

1. Observe the current block's L2 gas price: `P_prev`.
2. The current block is congested (`gas_used > gas_target`), so `P_next = P_prev + P_prev / 48 ≈ 1.0208 * P_prev`.
3. Submit an `InvokeV3` transaction with `l2_gas.max_price_per_unit = P_prev` (exactly the previous block's price).
4. Gateway `validate_resource_bounds` computes `threshold = 100% * P_prev = P_prev`. Check passes: `P_prev >= P_prev`. Transaction is admitted to the mempool.
5. When the batcher picks the transaction for the next block, the blockifier's `perform_pre_validation_stage` computes `actual_gas_price = P_next ≈ 1.0208 * P_prev`. Check fails: `P_prev < P_next`. Transaction is rejected with `MaxGasPriceTooLow`.
6. The gateway has admitted a transaction that the blockifier will always reject, wasting mempool resources and batcher cycles.

For the false-rejection path, reverse the scenario: submit with `max_price = P_prev - 1` during a low-usage block where `P_next = P_prev - P_prev/48`. The gateway rejects it (`P_prev - 1 < threshold = P_prev`) even though the blockifier would accept it (`P_prev - 1 > P_next`).
