### Title
User-Controlled Coinbase Address in `EVM.run()` / `EVM.batchRun()` Enables Gas Fee Self-Rebate - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.run()`, `EVM.mustRun()`, and `EVM.batchRun()` accept a freely user-supplied `coinbase: EVMAddress` parameter that determines who receives the EVM gas fees. Any unprivileged Cadence transaction sender can specify their own EVM address as the coinbase, causing all gas fees from their own EVM transactions to be refunded back to themselves, effectively executing EVM transactions at zero EVM gas cost.

### Finding Description
`EVM.run()` and `EVM.batchRun()` are `access(all)` functions in the `EVM` system contract. Both accept a `coinbase: EVMAddress` argument that is passed directly to `InternalEVM.run()` / `InternalEVM.batchRun()` without any validation or restriction on who the coinbase may be. [1](#0-0) [2](#0-1) 

At the Go handler layer, `runWithGasFeeRefund()` measures the balance increase on the protocol's internal `CoinbaseAddress` after EVM execution, then transfers that entire delta to the caller-supplied `gasFeeCollector` (the coinbase): [3](#0-2) 

Because the coinbase is a free parameter with no allowlist or identity check, a user who is also the signer of the inner EVM transaction can set `coinbase` to their own EVM address. The gas fees deducted from their EVM transaction are immediately re-deposited into that same address, netting to zero EVM gas cost.

### Impact Explanation
- **Gas fee bypass / self-rebate**: An EVM transaction sender who calls `EVM.run(tx: myTx, coinbase: myOwnEVMAddress)` pays EVM gas but receives it back as coinbase rewards. Net EVM execution cost = 0 (only the outer Cadence transaction fee applies, which is separate and much smaller).
- **Relayer fee diversion**: Any protocol, dApp, or relayer that expects to receive gas fees via the coinbase parameter receives nothing when a user self-routes fees.
- **Analogous to the referral self-dealing bug**: Just as a buyer in `mintWithReferralFor()` could name themselves as `referrer` to pocket the referral payout, an EVM transaction sender here names themselves as `coinbase` to pocket the gas fee payout.

This is a direct on-chain asset impact: gas fees (FLOW tokens in attoflow denomination) that should flow to a designated recipient are instead redirected to the attacker's own EVM balance.

### Likelihood Explanation
- `EVM.run()` and `EVM.batchRun()` are `access(all)` — callable by any Cadence transaction with no role or entitlement requirement.
- The attacker only needs to: (1) own an EVM address (trivially created via `EVM.createCadenceOwnedAccount()` or any EOA), (2) sign an EVM transaction from that address, and (3) submit a Cadence transaction calling `EVM.run(tx: signedEvmTx, coinbase: ownAddress)`.
- No privileged access, leaked keys, or node compromise is required.
- The attack is repeatable on every EVM transaction.

### Recommendation
Restrict the `coinbase` parameter so it cannot be set to an address controlled by the EVM transaction signer. Options include:
1. **Remove the `coinbase` parameter entirely** and hard-code the coinbase to a protocol-controlled address (e.g., the fee vault or a designated relayer registry).
2. **Derive the coinbase from the Cadence transaction's payer/authorizer** rather than accepting it as a free argument, so the fee recipient is always the entity paying the Cadence-layer cost.
3. **Maintain an allowlist** of valid coinbase addresses that only privileged operators can update, and validate the supplied coinbase against it before execution.

### Proof of Concept
Any unprivileged user can submit the following Cadence transaction to execute an EVM transaction and receive all gas fees back:

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>

// Attacker submits this Cadence transaction.
// They are also the signer of `innerEvmTxBytes` (same EVM address as coinbase).
transaction(innerEvmTxBytes: [UInt8], myEVMAddressBytes: [UInt8; 20]) {
    prepare(account: auth(BorrowValue) &Account) {
        // Step 1: set coinbase = own EVM address
        let myCoinbase = EVM.EVMAddress(bytes: myEVMAddressBytes)

        // Step 2: run the EVM transaction — gas fees are deducted from the EVM tx signer
        let res = EVM.run(tx: innerEvmTxBytes, coinbase: myCoinbase)

        // Step 3: handler.runWithGasFeeRefund() transfers the coinbase balance delta
        //         to myCoinbase — which is the attacker's own address.
        // Net EVM gas cost to attacker = 0.
        assert(res.status == EVM.Status.successful, message: "tx failed")
    }
}
```

The `runWithGasFeeRefund` call at `fvm/evm/handler/handler.go:282` will transfer the full gas fee amount to `gasFeeCollector` (the attacker-supplied coinbase), completing the self-rebate. [4](#0-3)

// runWithGasFeeRefund runs a method and transfers the balance changes of the
// coinbase address to the provided gas fee collector
func (h *ContractHandler) runWithGasFeeRefund(gasFeeCollector types.Address, f func()) {
	// capture coinbase init balance
	cb := h.AccountByAddress(types.CoinbaseAddress, true)
	initCoinbaseBalance := cb.Balance()
	f()
	// transfer the gas fees collected to the gas fee collector address
	afterBalance := cb.Balance()
	diff := new(big.Int).Sub(afterBalance, initCoinbaseBalance)
	if diff.Sign() > 0 {
		cb.Transfer(gasFeeCollector, diff)
	}
	if diff.Sign() < 0 { // this should never happen but in case
		panic(fvmErrors.NewEVMError(fmt.Errorf("negative balance change on coinbase")))
	}
}
