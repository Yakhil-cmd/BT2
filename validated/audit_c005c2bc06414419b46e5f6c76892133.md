### Title
Cross-COA Signature Replay via Missing `evmAddress` Binding in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over caller-supplied `signedData` but does not enforce that `signedData` encodes the target `evmAddress`. Because the `CapabilityPath` (which selects which COA to prove) is a separate, unsigned parameter, an attacker who observes a valid proof for COA-A can reconstruct a proof pointing to COA-B — reusing the same signatures — and pass `validateCOAOwnershipProof` for COA-B. This is acknowledged in a code comment but left unmitigated. The COA's ERC-1271 `isValidSignature` implementation in `coa.sol` is the primary consumer of this primitive, making the replay directly exploitable by any unprivileged EVM transaction sender.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts:

- `address` — the Cadence account address
- `path` — the `PublicPath` at which the COA capability is published
- `signedData` — the bytes the Cadence keys signed
- `keyIndices` / `signatures` — the multi-sig proof
- `evmAddress` — the EVM address being proven [1](#0-0) 

The function verifies:
1. The signatures over `signedData` are valid for the Cadence account keys.
2. The Cadence account has a COA capability at `path`.
3. That COA's EVM address equals `evmAddress`. [2](#0-1) 

**The gap**: `signedData` is never required to contain `evmAddress`. The code itself acknowledges this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [3](#0-2) 

Because `path` is an unsigned parameter supplied by the caller at proof-verification time, an attacker can swap it from `"coa1"` to `"coa2"` while keeping the original signatures intact. The signature check still passes (same Cadence keys, same `signedData`), and the COA address check passes because the attacker also supplies COA-B's address as `evmAddress`.

The COA's ERC-1271 implementation in `coa.sol` calls the precompile with `address(this)` as the EVM address:

```solidity
cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig
    )
);
``` [4](#0-3) 

The precompile routes to `coaOwnershipProofValidator`, which calls `validateCOAOwnershipProof`: [5](#0-4) 

An attacker who intercepts a valid proof for COA-A can construct a new `COAOwnershipProof` struct with the same `Address`, `KeyIndices`, and `Signatures` but with `CapabilityPath` changed to point to COA-B, then submit it to any EVM contract that calls `isValidSignature` on COA-B.

### Impact Explanation

Any EVM DeFi protocol that uses ERC-1271 (`isValidSignature`) to authorize privileged actions — token approvals, permit-style withdrawals, governance votes, NFT transfers — on a COA will accept the replayed proof as valid. The attacker gains the ability to impersonate the owner of COA-B without ever controlling the Cadence account's private keys. This constitutes unauthorized account mutation and cross-VM asset loss: assets held by COA-B can be moved by an attacker using a signature the victim produced for a completely different COA.

### Likelihood Explanation

The precondition is a Cadence account with two or more COAs published at different public paths. The protocol explicitly supports this (the `path` parameter exists precisely to allow multiple COAs per account). As the Flow EVM ecosystem grows and more sophisticated users or protocols create multi-COA setups, the attack surface expands. The attacker's only requirement is to observe a single on-chain proof for COA-A — all Flow transactions are public — and then submit a modified proof to any EVM contract that calls `isValidSignature` on COA-B. No privileged access, no key material, and no brute force is required.

### Recommendation

Enforce that `signedData` commits to `evmAddress`. The simplest fix is to require, inside `validateCOAOwnershipProof`, that `signedData` contains the 20-byte `evmAddress` as a prefix or suffix, and reject proofs where it does not. Alternatively, the precompile layer (`coaOwnershipProofValidator`) can prepend `evmAddress` to `signedData` before passing it to `validateCOAOwnershipProof`, so callers never need to do this manually. Either approach closes the cross-COA replay channel.

### Proof of Concept

**Setup**: Alice's Cadence account owns two COAs:
- COA-A at `/public/coa1` with EVM address `0xAAAA…`
- COA-B at `/public/coa2` with EVM address `0xBBBB…`

**Step 1**: Alice legitimately signs `D = keccak256("authorize trade on ProtocolX")` with her Cadence key 0 to prove ownership of COA-A. The on-chain proof is:

```
proof_A = {
  Address:        Alice's Cadence address,
  CapabilityPath: "coa1",
  KeyIndices:     [0],
  Signatures:     [sig_over_D],
}
```

**Step 2**: Attacker observes `proof_A` and `D` on-chain. Attacker constructs:

```
proof_B = {
  Address:        Alice's Cadence address,
  CapabilityPath: "coa2",   // changed — unsigned
  KeyIndices:     [0],
  Signatures:     [sig_over_D],  // reused unchanged
}
```

**Step 3**: Attacker calls `COA-B.isValidSignature(D, encode(proof_B))` from any EVM contract.

**Step 4**: `isValidSignature` calls `verifyCOAOwnershipProof(0xBBBB…, D, encode(proof_B))`.

**Step 5**: `validateCOAOwnershipProof` is invoked with `evmAddress = 0xBBBB…`, `path = "coa2"`, `signedData = D`, `signatures = [sig_over_D]`.

- Signature check: `sig_over_D` is valid over `D` with Alice's key 0. ✅
- COA capability check: Alice's account has a COA at `"coa2"`. ✅
- Address match: COA at `"coa2"` has address `0xBBBB…`. ✅

**Step 6**: `validateCOAOwnershipProof` returns `isValid: true`. `isValidSignature` returns `0x1626ba7e` (valid). Any EVM protocol that trusted this result now authorizes the attacker to act as COA-B's owner. [6](#0-5) [7](#0-6) [8](#0-7)

func coaOwnershipProofValidator(contractAddress flow.Address, backend backends.Backend) func(proof *types.COAOwnershipProofInContext) (bool, error) {
	return func(proof *types.COAOwnershipProofInContext) (bool, error) {
		value, err := backend.Invoke(
			environment.ContractFunctionSpec{
				AddressFromChain: func(_ flow.Chain) flow.Address {
					return contractAddress
				},
				LocationName: "EVM",
				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}
		data, ok := value.(cadence.Struct)
		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		isValidValue := cadence.SearchFieldByName(data, ValidationResultTypeIsValidFieldName)
		if isValidValue == nil {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
}
