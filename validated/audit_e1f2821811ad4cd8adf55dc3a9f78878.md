### Title
Stale Hardcoded Nonce in `skip_stateful_validations` Ignores Configurable `max_nonce_for_validation_skip` - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`StatefulTransactionValidatorConfig` exposes a `max_nonce_for_validation_skip` field that operators can configure to control the maximum transaction nonce for which gateway-level `__validate__` is skipped (the deploy-account + invoke UX feature). However, the `skip_stateful_validations` free function hardcodes `Nonce(Felt::ONE)` instead of reading the config value. The config field is fully serialized, appears in deployment config files, and is documented — but is silently ignored in the new gateway. Any operator who sets `max_nonce_for_validation_skip` to `0` (to disable the skip) will find the skip still active for nonce-1 invoke transactions; any operator who sets it above `1` will find the skip absent for nonces `2..max_nonce_for_validation_skip`.

### Finding Description

`StatefulTransactionValidatorConfig` declares and serializes `max_nonce_for_validation_skip`: [1](#0-0) [2](#0-1) [3](#0-2) 

The field is described as "Maximum nonce for which the validation is skipped." Its default is `Nonce(Felt::ONE)`.

The `skip_stateful_validations` function that implements this policy is a free function — it does not take `self` and has no access to the config: [4](#0-3) 

Line 437 hardcodes `Nonce(Felt::ONE)` unconditionally. The config field `max_nonce_for_validation_skip` is never read here.

Compare with the legacy Python-binding path (`PyValidator`), which correctly reads the field: [5](#0-4) 

The `skip_stateful_validations` result flows directly into `run_validate_entry_point` as `skip_validate`, which sets `execution_flags.validate = !skip_validate`: [6](#0-5) [7](#0-6) 

When `skip_validate = true`, the account's `__validate__` entry point is not called at the gateway level.

A grep across the entire repository confirms `max_nonce_for_validation_skip` is referenced only in `py_validator.rs`, `config.rs`, the JSON schema, and two deployment config files — never in `stateful_transaction_validator.rs`.

### Impact Explanation

Two distinct broken invariants:

**Case A — operator sets `max_nonce_for_validation_skip = 0` to disable the skip feature:**
The hardcoded `Nonce(Felt::ONE)` still triggers the skip for any invoke transaction with nonce=1 from an undeployed account that has a deploy-account in the mempool. The gateway admits the transaction without calling `__validate__`, contrary to the operator's explicit configuration. This is a gateway admission path that accepts a transaction whose account-level signature has not been verified at the gateway stage.

**Case B — operator sets `max_nonce_for_validation_skip > 1` (e.g., `5`) to allow multiple pre-deploy invokes:**
Invoke transactions with nonces `2..5` from undeployed accounts are not skipped; they reach `run_validate_entry_point` with `skip_validate = false`, causing the blockifier to call `__validate__` on a non-existent account, which fails. The gateway rejects transactions that the operator's policy explicitly marks as valid.

Both cases match the allowed High impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded value, so the bug is invisible with default configuration. However:

- The field is fully serialized and documented.
- It appears in production deployment config files (`gateway_config.json`, `replacer_gateway_config.json`).
- Any operator who reads the documentation and attempts to tune the field (either to `0` to harden the gateway, or to `>1` to support multi-tx pre-deploy flows) will silently get the wrong behavior with no error or warning.

### Recommendation

Convert `skip_stateful_validations` from a free function to a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as a parameter), and replace the hardcoded `Nonce(Felt::ONE)` with the config value, mirroring the `PyValidator` logic:

```rust
// In skip_stateful_validations, replace:
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {

// With:
let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx.nonce();
let nonce_small_enough = tx.nonce() <= max_nonce_for_validation_skip;
if is_post_deploy_nonce && nonce_small_enough && account_nonce == Nonce(Felt::ZERO) {
```

### Proof of Concept

1. Deploy the gateway with `max_nonce_for_validation_skip` set to `Nonce(Felt::ZERO)` in `StatefulTransactionValidatorConfig`.
2. Submit a `DeployAccount` transaction for a fresh account (nonce=0). The mempool accepts it.
3. Submit an `Invoke` transaction from the same account with nonce=1, carrying an **invalid signature**.
4. `skip_stateful_validations` evaluates `tx.nonce() == Nonce(Felt::ONE)` → `true`, ignoring the configured `max_nonce_for_validation_skip = 0`.
5. `run_validate_entry_point` is called with `skip_validate = true`; `__validate__` is never invoked.
6. The gateway returns success and forwards the transaction to the mempool — despite the operator having configured the skip feature off and despite the invalid signature. [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
