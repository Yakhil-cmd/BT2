### Title
Delegation Signature Replay in `allowPushers()` Bypasses Pusher's Unilateral Revocation Within the Deadline Window — (`File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers()` in `CompressedOracle.sol` verifies a pusher's EIP-191 consent signature but includes no nonce in the signed hash. The only replay guard is a `deadline`. After a pusher calls `revokePusher()`, the creator can immediately replay the original signature (while the deadline has not yet expired) to re-establish the delegation without any fresh consent from the pusher. This directly mirrors the Nexus `_getEnableModeDataHash()` bug: a one-time authorization action is replayable because the hash commits to no monotonically-advancing state.

---

### Finding Description

`allowPushers()` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The hash commits to: chain id, contract address, deadline, pusher address, and creator address. There is no nonce, no revocation counter, and no per-pusher sequence number. The code comment itself acknowledges the risk:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [2](#0-1) 

The comment treats the deadline as the complete fix, but the deadline only prevents replay **after** it expires. Within the deadline window, the exact same `(deadline, pusher, creator)` tuple is still a valid, accepted signature. `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

But nothing invalidates the old signature. The creator can immediately call `allowPushers()` again with the same `deadline` and `signature` bytes, passing `_ensureDeadline(deadline)` (which only checks `block.timestamp <= deadline`) and re-writing `namespaceRemapping[pusher] = msg.sender`. [4](#0-3) 

---

### Impact Explanation

The `fallback()` push path resolves the namespace from `namespaceRemapping[msg.sender]`:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [5](#0-4) 

If a pusher's key is compromised and the pusher calls `revokePusher()` to stop the attacker from writing to the creator's namespace, the creator can replay the old signature to re-establish the delegation. The attacker (holding the compromised pusher key) can then continue pushing arbitrary slot values — including manipulated prices and spreads — into the creator's namespace. Any `MetricOmmPool` whose `IPriceProvider` reads from this oracle namespace will receive corrupted bid/ask prices, enabling bad-price swap execution and direct loss of trader principal.

---

### Likelihood Explanation

- The creator has the old `(deadline, signature)` tuple from the original `allowPushers()` call (it is calldata on a public transaction, trivially recoverable from chain history).
- The creator may replay it inadvertently (e.g., re-running a deployment script) or deliberately.
- The deadline window is creator-chosen and can be set to days or weeks, giving a large replay surface.
- No on-chain state change is required from the pusher to enable the replay; the creator acts alone.

---

### Recommendation

Include a per-pusher nonce in the signed hash and increment it on every successful delegation or revocation:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers():
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);
```

Alternatively, maintain a `mapping(bytes32 => bool) usedSignatures` and mark each digest as consumed on first use, rejecting any replay regardless of deadline.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = block.timestamp + 30 days
bytes memory sig = sign(PUSHER_KEY, keccak256(abi.encode(
    block.chainid, address(oracle), deadline, pusher, creator
)));

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes (e.g., key compromised)
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0));

// 4. Creator replays the SAME signature — still within deadline
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // succeeds, no revert
assertEq(oracle.namespaceRemapping(pusher), creator);   // delegation restored

// 5. Attacker (holding compromised pusher key) pushes manipulated price
vm.prank(attacker); // attacker == pusher key holder
(bool ok,) = address(oracle).call(maliciousSlotWord);
assertTrue(ok); // writes bad price into creator's namespace
// → pools reading this feed execute swaps at attacker-controlled prices
```
