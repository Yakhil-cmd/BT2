### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions With Arbitrary Signatures When Account Nonce Is Zero - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway stateful validator bypasses the `__validate__` entry-point call (signature verification) for any invoke transaction with `nonce=1` when the sender account has on-chain nonce `0` and the address appears in the mempool. An unprivileged attacker who observes a pending `deploy_account` transaction for address X can submit invoke transactions for X with completely arbitrary signatures that are admitted to the mempool without any signature check.

### Finding Description

The gateway stateful path calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

The function returns `true` (skip `__validate__`) when three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate = false`: [3](#0-2) 

This propagates into `StatefulValidator::perform_validations`, which returns `Ok(())` immediately without calling `validate_tx`: [4](#0-3) 

The `__validate__` entry point — the only place where the account contract verifies the transaction signature — is never executed.

The `account_tx_in_pool_or_recent_block` check is satisfied by the presence of **any** transaction for the address in the pool or a recent committed block: [5](#0-4) 

The comment in `skip_stateful_validations` claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, this reasoning is flawed: the check does not distinguish between a legitimate `deploy_account` and an invoke with `nonce=0` that was admitted via the non-strict nonce path. More critically, once the address is in the mempool for **any** reason, the attacker's subsequent invoke with `nonce=1` bypasses signature verification entirely.

The prior `validate_nonce` check allows `tx_nonce=1` when `account_nonce=0` as long as `max_allowed_nonce_gap >= 1`: [6](#0-5) 

The mempool's `validate_tx` only checks nonce ordering and fee escalation — it does not verify signatures: [7](#0-6) 

### Impact Explanation

An attacker can submit invoke transactions with garbage signatures for any address that has a pending `deploy_account` in the mempool. These transactions pass all gateway checks and are admitted to the mempool without the account contract ever verifying the signature. This matches the allowed High impact: **"Mempool/gateway/RPC admission accepts invalid transactions."**

The admitted transactions will fail during block execution (since `new_for_sequencing` sets `validate=true`), but they consume mempool capacity, waste batcher/blockifier resources, and can be used to grief legitimate users by filling the mempool with invalid transactions targeting any address with a pending deployment.

### Likelihood Explanation

The mempool is observable. Any attacker can watch for `deploy_account` transactions and immediately submit fake invokes. The attack requires no privileged access, no special knowledge of the private key, and no on-chain funds. The only prerequisite is that the target address has a pending `deploy_account` in the mempool — a condition that is publicly visible and common during normal UX flows.

### Recommendation

The `skip_stateful_validations` check must verify that the address in the mempool specifically has a **`deploy_account` transaction**, not just any transaction. The current `account_tx_in_pool_or_recent_block` API returns a boolean without distinguishing transaction types. Either:

1. Add a dedicated `has_deploy_account_in_pool(address)` query to the mempool that checks the transaction type, and use it instead of `account_tx_in_pool_or_recent_block`.
2. Alternatively, do not skip `__validate__` entirely; instead, run validation against a synthetic state where the account is assumed to be deployed (using the class hash from the pending `deploy_account`).

### Proof of Concept

```
1. Legitimate user submits deploy_account for address X (nonce=0).
   → Gateway validates it fully (execute path), admits to mempool.
   → account_tx_in_pool_or_recent_block(X) now returns true.

2. Attacker submits:
   InvokeTransaction {
       sender_address: X,
       nonce: 1,
       calldata: [arbitrary],
       signature: [0xdeadbeef],  // garbage
       resource_bounds: valid,
   }

3. Gateway stateful path:
   a. get_nonce_from_state(X) → Nonce(0)  (account not deployed)
   b. validate_nonce: 0 <= 1 <= max_allowed_nonce_gap → PASS
   c. validate_by_mempool: nonce gap accepted → PASS
   d. skip_stateful_validations:
      - tx is Invoke ✓
      - tx.nonce() == 1 ✓
      - account_nonce == 0 ✓
      - account_tx_in_pool_or_recent_block(X) == true ✓
      → returns true (skip __validate__)
   e. run_validate_entry_point(skip_validate=true):
      - execution_flags.validate = false
      - __validate__ NOT called
      → PASS

4. Attacker's invoke with garbage signature is admitted to the mempool.
   The account contract never verified the signature.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
