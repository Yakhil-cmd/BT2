### Title
Missing STRK Token Address Validation for `reward_address` in Pool Contract Allows Permanent Yield Freeze — (`src/pool/pool.cairo`)

### Summary

The `Pool` contract's `enter_delegation_pool()` and `change_reward_address()` functions only validate that `reward_address` is not the pool's own staking token. They do not validate that `reward_address` is not the STRK token address — the token used to pay rewards. Since `claim_rewards()` always transfers STRK to `reward_address`, a pool member in a BTC pool can set `reward_address = STRK_TOKEN_ADDRESS`, causing all future reward claims to transfer STRK into the STRK contract itself, permanently destroying the yield.

### Finding Description

The `Staking` contract's `stake()` and `change_reward_address()` functions use `does_token_exist()` to guard against setting `reward_address` to any registered token address (STRK or BTC): [1](#0-0) [2](#0-1) 

`does_token_exist()` returns `true` for STRK and all registered BTC tokens: [3](#0-2) 

However, the `Pool` contract's `enter_delegation_pool()` only checks that `reward_address` differs from the pool's own staking token: [4](#0-3) 

Similarly, `Pool::change_reward_address()` only checks against the pool's own token dispatcher: [5](#0-4) 

For a BTC pool, the pool's `token_dispatcher` holds the BTC token address. Neither check prevents `reward_address` from being set to `STRK_TOKEN_ADDRESS`.

When `claim_rewards()` is called, it always pays out in STRK, transferring to `reward_address`: [6](#0-5) 

If `reward_address == STRK_TOKEN_ADDRESS`, the STRK transfer succeeds (the ERC20 contract accepts it), but the tokens are credited to the STRK contract's own balance — permanently inaccessible to the pool member.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A pool member in a BTC delegation pool can set `reward_address = STRK_TOKEN_ADDRESS`. Every subsequent `claim_rewards()` call will silently transfer STRK rewards into the STRK token contract, permanently destroying the pool member's accumulated yield. The pool member's principal (BTC) is unaffected, but all STRK rewards are irrecoverably lost.

### Likelihood Explanation

**Medium.** The attack requires a BTC delegation pool to exist (a protocol-supported feature). Any unprivileged pool member can trigger this on themselves either at entry (`enter_delegation_pool`) or at any time afterward (`change_reward_address`). No privileged access is required. The scenario is reachable as soon as BTC pools are live.

### Recommendation

In `Pool::enter_delegation_pool()` and `Pool::change_reward_address()`, add a check that `reward_address` is not the STRK token address (and ideally not any registered token address). The pool contract should call the staking contract to query `does_token_exist()`, or hardcode a check against `STRK_TOKEN_ADDRESS`:

```cairo
assert!(
    reward_address != STRK_TOKEN_ADDRESS,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
assert!(
    token_dispatcher.contract_address != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

This mirrors the protection already present in `Staking::stake()` and `Staking::change_reward_address()`.

### Proof of Concept

1. A staker opens a BTC delegation pool via `set_open_for_delegation(btc_token_address)`.
2. An attacker (pool member) calls `pool.enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)`.
   - The check `btc_token_address != STRK_TOKEN_ADDRESS` passes.
   - The pool member record is created with `reward_address = STRK_TOKEN_ADDRESS`.
3. Epochs pass; the pool accumulates STRK rewards.
4. Anyone calls `pool.claim_rewards(pool_member: attacker_address)`.
   - `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)` executes successfully.
   - STRK rewards are credited to the STRK contract's own balance.
   - The pool member's `reward_address` receives nothing; the yield is permanently frozen.
