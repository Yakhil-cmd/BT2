### Title
`PresignOutput` Derives `Clone` and `rerandomize_presign` Accepts It by Shared Reference, Enabling Presignature Reuse and Private Key Recovery — (`src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

The OT-based ECDSA `PresignOutput` type derives `Clone` while simultaneously deriving `ZeroizeOnDrop`, and the `rerandomize_presign` function accepts it as `&PresignOutput` (shared reference). This combination means the same presignature can be rerandomized and consumed in multiple independent signing sessions without any library-level enforcement of single-use. This is the direct structural analog of the reported "modify cancelled order" bug: just as `modify()` failed to check whether an order was already cancelled before allowing a second withdrawal, `rerandomize_presign()` fails to enforce that a presignature has not already been consumed. The library's own documentation explicitly states that presignature reuse enables full private key recovery via ECDSA nonce-reuse attacks.

---

### Finding Description

**Vulnerability class:** Threshold-state bug / missing one-time-use enforcement.

`PresignOutput` in `src/ecdsa/ot_based_ecdsa/mod.rs` is declared as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    #[zeroize[skip]]
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
``` [1](#0-0) 

The `ZeroizeOnDrop` derive signals that the library intends for the secret nonce material (`k`, `sigma`) to be destroyed after use. However, the simultaneous `Clone` derive directly contradicts this intent: a caller can clone the `PresignOutput` before it is dropped, producing an identical copy with the same nonce material that is never zeroed.

The rerandomization entry point compounds this:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // shared reference — not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
``` [2](#0-1) 

Because `presignature` is taken by shared reference, the same `PresignOutput` value can be passed to `rerandomize_presign` an arbitrary number of times with different `RerandomizationArguments` (different tweaks `ε` and/or different message hashes `h`). Each call produces a distinct `RerandomizedPresignOutput` that can then be forwarded to a separate signing session. No state flag, no ownership transfer, and no type-system constraint prevents this.

`RerandomizedPresignOutput` itself also derives `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct RerandomizedPresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
``` [3](#0-2) 

The presign function itself also documents the single-use requirement but provides no enforcement:

> "it's crucial that a presignature is never reused" [4](#0-3) 

The orchestration documentation makes the security consequence explicit:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [5](#0-4) 

And the signing security documentation states the exact attack:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."
> "**Never reuse a presignature**, even across failed, aborted, or partially completed signing sessions." [6](#0-5) 

The library simultaneously warns that reuse is catastrophic and provides the `Clone` derive that makes reuse trivial. This is the same structural gap as the original report: the "cancelled" state existed but was never checked.

---

### Impact Explanation

**Critical — Extraction of private signing key material.**

When the same `PresignOutput` (with the same underlying nonce `k`) is used to produce two ECDSA signatures `(R, s₁)` and `(R, s₂)` over different message hashes `h₁` and `h₂` (or different tweaks `ε₁`, `ε₂`), the shared nonce `k` can be algebraically recovered from the two public signatures using the standard nonce-reuse equation:

```
k = (h₁ - h₂) / (s₁ - s₂)   (mod q)
```

Once `k` is known, the private key `x` is immediately recoverable from either signature. This constitutes full extraction of the aggregate secret signing key, matching the Critical impact tier: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

**High.** The attack requires no cryptographic break and no external assumption. Any participant who holds a `PresignOutput` can:

1. Call `rerandomize_presign(&presign_out, &args1)` for signing session 1.
2. Call `rerandomize_presign(&presign_out, &args2)` for signing session 2 (different `h` or `ε`).
3. Observe both resulting signatures and solve for `k` and then `x`.

Alternatively, a malicious coordinator can initiate two concurrent signing sessions referencing the same presignature batch entry before participants have locally invalidated it, since the library provides no invalidation mechanism. The `Clone` derive also means serialization/deserialization round-trips (e.g., for storage or backup) silently produce reusable copies.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`.** Rust's ownership system then enforces single-use at compile time: the value is moved into `rerandomize_presign` or `sign` and cannot be used again without an explicit (and auditable) clone.

2. **Change `rerandomize_presign` to take `PresignOutput` by value** (consuming it):
   ```rust
   pub fn rerandomize_presign(
       presignature: PresignOutput,   // consumed, not cloneable
       args: &RerandomizationArguments,
   ) -> Result<Self, ProtocolError>
   ```

3. **Apply the same fix to `PresignOutput<C>` in `src/frost/mod.rs`**, which also derives `Clone` and contains `SigningNonces` that FROST requires to be used exactly once. [7](#0-6) 

---

### Proof of Concept

```rust
// Attacker holds a PresignOutput after presigning
let presign_out: PresignOutput = /* result of presign protocol */;

// Session 1: rerandomize with message hash h1 and tweak eps1
let args1 = RerandomizationArguments::new(pk, tweak1, h1_bytes, presign_out.big_r, participants.clone(), entropy1);
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args1).unwrap();
// → produces signature (R, s1) over h1

// Session 2: rerandomize the SAME presign_out with different h2/eps2
// presign_out is still alive — Clone or shared reference allows this
let args2 = RerandomizationArguments::new(pk, tweak2, h2_bytes, presign_out.big_r, participants.clone(), entropy2);
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args2).unwrap();
// → produces signature (R, s2) over h2

// Both signatures share the same underlying nonce k.
// Attacker computes: k = (h1 - h2) / (s1 - s2) mod q
// Then: x = (s1 * k - h1) / r mod q  → full private key recovered.
```

The `&presign_out` shared reference on lines 5 and 11 is accepted by the current API with no error, no warning, and no state check — exactly as `modify()` accepted a cancelled order without checking its state.

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
