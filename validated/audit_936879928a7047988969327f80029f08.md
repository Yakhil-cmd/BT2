### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly aggregates each participant's `(norm_big_y, norm_big_c)` contribution with no cryptographic verification. A single malicious participant can send arbitrary group elements, causing every honest party to accept a silently corrupted confidential derived key.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives one `CKDOutput` from every other participant and unconditionally adds the two group elements together:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The correct per-participant computation is:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)
```

where `x_i` is the participant's secret signing share and `y_i` is a fresh random scalar. [2](#0-1) 

No commitment, range check, or zero-knowledge proof is attached to the message. The coordinator has no means to distinguish a correctly formed contribution from an arbitrary pair of G1 points. The final `CKDOutput` is assembled and returned directly:

```rust
// lines 56-57
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [3](#0-2) 

The `unmask` function then computes `C − app_sk · Y`, which equals `msk · H(pk, app_id)` only when every contribution is honest: [4](#0-3) 

If any single participant injects a wrong `(big_y, big_c)`, the unmasked result is an arbitrary G1 point unrelated to the master secret key.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable or attacker-influenced cryptographic outputs.**

The coordinator is an honest party. It receives a structurally valid `CKDOutput` (two G1 points), passes all type-level checks, and returns it as the protocol result. The client calling `unmask` obtains a value that is not `msk · H(pk, app_id)`. Any downstream operation that depends on the correct derived key (decryption, authentication, further derivation) silently fails or produces attacker-influenced material. Because the corruption is undetectable at the protocol layer, honest parties have no signal that the output is wrong.

---

### Likelihood Explanation

**High.** The attacker needs only to be one of the `n` participants in a CKD session. No special privilege beyond participation is required. The malicious participant sends a single crafted private message to the coordinator on the single waitpoint used by the protocol. The `recv_from_others` helper accepts the first message from each sender and does not re-request or cross-check it. [5](#0-4) 

---

### Recommendation

Attach a non-interactive zero-knowledge proof (e.g., a Schnorr-style proof of correct ElGamal encryption) to each participant's `(norm_big_y, norm_big_c)` message, proving that:

1. `norm_big_y = λ_i · y_i · G` for some `y_i` known to the sender.
2. `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` for the same `y_i` and the sender's committed signing share `x_i`.

The coordinator must verify all proofs before summing contributions and must abort if any proof fails, attributing the failure to the offending participant.

---

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` compute correct `(norm_big_y_i, norm_big_c_i)`.
2. Malicious participant `P_n` sends `(G1::generator(), G1::generator())` instead of its correct share.
3. The coordinator sums all contributions:
   - `Y = Σ_{i<n} norm_big_y_i + G`
   - `C = Σ_{i<n} norm_big_c_i + G`
4. The coordinator returns `CKDOutput { big_y: Y, big_c: C }` with no error.
5. The client calls `unmask(app_sk)`:
   - Result = `C − app_sk · Y`
   - = `(msk · H(pk, app_id) + G) − app_sk · (Σ_{i<n} norm_big_y_i + G)`
   - ≠ `msk · H(pk, app_id)`
6. The derived confidential key is silently wrong. No error is raised anywhere in the protocol.

The root cause is exclusively in `do_ckd_coordinator` at lines 50–55 of `src/confidential_key_derivation/protocol.rs`, which is reachable by any participant in a live CKD session. [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
