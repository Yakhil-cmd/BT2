### Title
Unbounded EIP-6110 Deposit Event Iteration During Block Finalization Without Native Resource Accounting — (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

`eip6110_events_parser` iterates over **every event emitted in the entire block** during block finalization and performs SHA-256 hashing for each matching deposit event, with **zero native resource charge** for any of this work. Unlike the analogous EIP-7002 and EIP-7251 system parts — which are hard-capped at `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16` and `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2` respectively — the EIP-6110 path has no iteration cap. An unprivileged transaction sender can force the prover to consume unbounded RISC-V cycles during finalization, making a validly-executed block unprovable.

---

### Finding Description

`eip6110_events_parser` is called unconditionally during block finalization in both the sequencing and proving post-op paths: [1](#0-0) [2](#0-1) 

Inside the function, the loop body is:

```rust
for event in system.io.events_iterator() {   // iterates ALL block events
    if event.address != &DEPOSIT_CONTRACT_ADDRESS { continue; }
    if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
        ...
        validate_and_write_event_data(event, requests_hasher, &mut logger)
        // SHA-256 hashes 192 bytes per deposit event
    }
}
``` [3](#0-2) 

No native resource is charged anywhere in this function — neither for iterating over non-deposit events nor for the SHA-256 hashing of deposit events. [4](#0-3) 

Compare this with `eip7002_system_part`, which explicitly caps its loop at `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16`: [5](#0-4) 

And `eip7251_system_part`, capped at `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2`: [6](#0-5) 

The EIP-6110 path has no equivalent cap.

The block log limit (`MAX_NUMBER_OF_LOGS`) checked in `check_for_block_limits` applies only to L2-to-L1 signals, not to regular EVM `LOG` events: [7](#0-6) 

Regular EVM events are bounded only by the block gas limit. With a ~10–15 M gas block limit and `LOG0` costing 375 gas, a block can contain up to ~40,000 events. The `eip6110_events_parser` must scan all of them.

---

### Impact Explanation

**Vulnerability class: resource accounting bug / valid-execution unprovability.**

The proving system executes in a RISC-V ZKVM (Airbender) with a fixed cycle budget. Block finalization work — including `eip6110_events_parser` — runs entirely outside the per-transaction native resource accounting framework. If a block contains enough events (especially deposit events requiring SHA-256 hashing), the finalization phase can consume RISC-V cycles that were never budgeted, causing the prover to exceed its cycle limit and fail to generate a proof for an otherwise valid block.

This is the direct analog of the reported `frxETHMinter.depositEther` issue: a loop that always processes all accumulated items without a cap, which can exhaust the available resource budget and permanently block progress.

---

### Likelihood Explanation

Two attack surfaces exist:

1. **Deposit events (higher cost, higher impact per event):** Each deposit requires calling the deposit contract with 32 ETH. A whale attacker or a coordinated set of depositors filling a block with deposit events causes `validate_and_write_event_data` to SHA-256 hash 192 bytes per event, all uncharged.

2. **Non-deposit events (low cost, lower impact per event):** Any contract can emit `LOG` events cheaply. The outer loop must scan every event in the block regardless of origin. With ~40,000 events possible per block, the address-comparison scan itself becomes non-trivial unaccounted work in the RISC-V prover.

The deposit-event path is the higher-severity variant because each iteration does meaningful cryptographic work (SHA-256) with no native charge.

---

### Recommendation

1. **Add a hard cap on processed deposit events**, analogous to `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK` and `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK`. The Ethereum EIP-6110 spec does not impose a per-block limit, but ZKsync OS must enforce one to bound prover work.

2. **Charge native resources** for each iteration of the event scan and for each SHA-256 hashing operation inside `validate_and_write_event_data`, so the cost is reflected in the block's native resource budget.

3. **Consider pre-indexing deposit events** during transaction execution (at the point of emission) rather than re-scanning all block events at finalization, eliminating the O(total block events) scan entirely.

---

### Proof of Concept

1. Attacker (or a set of depositors) submits N transactions in a single block, each calling `DEPOSIT_CONTRACT_ADDRESS` (`0x00000000219ab540356cbb839cbe05303d7705fa`) with 32 ETH and valid deposit data.
2. Each transaction emits one deposit event with topic `0x649bbc62...`.
3. All N transactions fit within the block gas limit (each deposit call costs ~O(50k) gas; at 10M gas limit, N ≈ 200 deposits).
4. At block finalization, `eip6110_events_parser` is called. It scans all events in the block and calls `validate_and_write_event_data` for each of the N deposit events, SHA-256 hashing 192 bytes per event — with zero native resource charge.
5. The prover attempts to prove the block. The finalization phase consumes RISC-V cycles proportional to N that were never budgeted. For sufficiently large N (or combined with a block full of non-deposit events to maximize the outer scan), the prover exceeds its cycle budget and fails to produce a proof.
6. The block is valid (all transactions executed correctly within their gas limits) but is permanently unprovable, halting the chain's proving pipeline. [3](#0-2) [1](#0-0)

```rust
pub fn eip6110_events_parser<S: EthereumLikeTypes>(
    system: &System<S>,
    requests_hasher: &mut impl crypto::sha256::Digest,
) -> Result<bool, SystemError>
where
    S::IO: IOSubsystemExt + IOTeardown<S::IOTypes>,
{
    // we can not easily get the number from one scan, so we will accumulate into hasher directly

    let mut event_encountered = false;
    let mut logger = system.get_logger();
    for event in system.io.events_iterator() {
        if event.address != &DEPOSIT_CONTRACT_ADDRESS {
            continue;
        }
        if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
            if event_encountered == false {
                event_encountered = true;
                requests_hasher.update(&[DEPOSIT_REQUEST_EIP_7685_TYPE]);
            }
            let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
                panic!("invalid deposit event structure");
            };
        }
    }

    Ok(event_encountered)
}
