### Title
Silent Message Drop in `push_message` Causes Permanent Protocol Hang — (File: `src/protocol/internal.rs`)

---

### Summary

The `push_message` function in `src/protocol/internal.rs` silently discards any incoming message that fails header validation (too short, or unparseable header bytes) with a bare `return`, no error propagation, no bookkeeping, and no recovery path. This is structurally identical to the `RewardDistributor` pattern: a resource/validity check silently truncates the output instead of surfacing an error or deferring the work. Because the async `pop` side of the message buffer blocks indefinitely waiting for a message that was already silently discarded, a single malicious participant can permanently stall any protocol phase — DKG, reshare, refresh, presign, or sign — for every honest party.

---

### Finding Description

In `src/protocol/internal.rs`, the `push_message` function is the sole entry point through which the protocol executor delivers network messages into the internal buffer:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {
        return;                          // ← silent drop, no error
    }

    let Some(header) = MessageHeader::from_bytes(&message) else {
        return;                          // ← silent drop, no error
    };

    self.incoming.push(header, from, message);
}
``` [1](#0-0) 

When either guard fires, the message is permanently gone — no counter is incremented, no error is returned to the caller, and no notification is sent to the waiting protocol future.

The consumer side, `pop`, blocks forever on the channel until a message for the expected `MessageHeader` arrives:

```rust
async fn pop(&self, header: MessageHeader) -> (Participant, MessageData) {
    ...
    receiver_lock.next().await.expect("Reference to sender held")
}
``` [2](#0-1) 

Every protocol phase uses `recv_from_others`, which loops calling `chan.recv(waitpoint)` until it has heard from **every** other participant:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;
    if seen.put(from) {
        messages.push((from, msg));
    }
}
``` [3](#0-2) 

If a malicious participant sends a message whose byte length is less than `MessageHeader::LEN` (40 bytes: 32-byte `ChannelTag` + 8-byte `Waitpoint`), `push_message` silently discards it. `recv_from_others` never marks that participant as `seen`, so `seen.full()` never becomes true, and the honest party's protocol future is permanently suspended — no timeout, no error, no abort signal.

This affects every protocol that calls `recv_from_others`:

- DKG / keygen (`do_keyshare`, `do_keygen`) [4](#0-3) 
- Reshare / refresh (`do_reshare`) [5](#0-4) 
- OT-based ECDSA presign (`do_presign`) [6](#0-5) 
- Robust ECDSA presign and sign [7](#0-6) 

The structural parallel to the `RewardDistributor` bug is exact:

| `RewardDistributor.sol` | `push_message` |
|---|---|
| `if (amount > balance) { amount = balance; }` — silent cap | `if message.len() < LEN { return; }` — silent drop |
| Accrued rewards permanently lost | Incoming message permanently lost |
| No bookkeeping of the deficit | No counter, no error, no notification |
| Contract must be re-funded externally | Protocol must be restarted externally |

---

### Impact Explanation

**High — Permanent denial of signing, key generation, reshare, and refresh for honest parties.**

A single malicious participant (below the Byzantine threshold) can permanently freeze every honest party's protocol instance by sending one crafted short message per round. Because the drop is silent and the `pop` future never resolves, the honest party's thread/task is stuck indefinitely. There is no internal timeout, no abort path, and no way for the application to distinguish "waiting for a slow peer" from "waiting for a message that was already silently discarded." The presigned material and key shares held by honest parties become permanently unusable for that session.

---

### Likelihood Explanation

**High.** The attack requires no cryptographic capability, no key material, and no privileged position. Any participant admitted to a protocol session can send a 1-byte (or otherwise short) UDP/TCP payload addressed to any honest peer. The malicious participant does not need to be above the Byzantine threshold — even a single corrupted party out of N can freeze all N honest parties simultaneously by targeting each one in the same round.

---

### Recommendation

1. **Return an error from `push_message`** (or its callers) when a message fails header validation, so the protocol executor can surface a `ProtocolError::MaliciousParticipant` and abort cleanly rather than hanging.
2. **Add a per-round timeout** in `recv_from_others` so that a missing or malformed message from one participant causes a bounded abort rather than an infinite wait.
3. **Track dropped messages** (analogous to the bookkeeping recommendation in the original report): log the sender and the reason for the drop so operators can identify and exclude the offending participant.

---

### Proof of Concept

```
Participants: [Alice (honest), Bob (honest), Mallory (malicious)]
Protocol: DKG keygen, Round 1 (do_keyshare → recv_from_others)

1. Mallory crafts a 5-byte message (< MessageHeader::LEN = 40).
2. Mallory sends this 5-byte payload to Alice via the network layer.
3. Alice's protocol executor calls push_message(Mallory, [5 bytes]).
4. push_message fires: `if message.len() < MessageHeader::LEN { return; }`
   → message is silently discarded; no error, no counter update.
5. Alice's recv_from_others loop calls chan.recv(waitpoint).await.
   → pop() blocks forever; seen.full() is never true because Mallory
     is never marked as seen.
6. Alice's DKG future is permanently suspended.
   Alice's key share is never produced.
   Alice cannot participate in any subsequent signing session.
```

The same attack applies identically to every round of every protocol that uses `recv_from_others`, including reshare, refresh, OT-based presign, robust ECDSA presign, and FROST sign.

pub async fn do_reshare<C: Ciphersuite>(
    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
    // either extract the share and linearize it or set it to zero
    let secret = old_signing_key
        .map(|x_i| {
            intersection
                .lagrange::<C>(me)
                .map(|lambda| lambda * x_i.to_scalar())
        })
        .transpose()?
        .unwrap_or_else(<C::Group as Group>::Field::zero);

    let old_reshare_package = Some((old_public_key, old_participants));
    let keygen_output = do_keyshare::<C>(
        chan,
        participants,
        me,
        threshold,
        secret,
        old_reshare_package,
        &mut rng,
    )
    .await?;

    Ok(keygen_output)
