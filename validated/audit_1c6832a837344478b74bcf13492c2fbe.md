### Title
`SwapAllowlistExtension` gates on the router address instead of the actual user, allowing any unprivileged user to bypass the swap allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

When `MetricOmmSimpleRouter` is used to swap on a pool protected by `SwapAllowlistExtension`, the extension checks the **router's address** against the allowlist instead of the actual end user's address. This is because the pool passes `msg.sender` (the router) as the `sender` argument to the extension hook. The result is a two-sided failure: if the router is not allowlisted, all router-mediated swaps are permanently blocked (DoS); if the router is allowlisted to fix that, any unprivileged user can bypass the allowlist entirely by routing through the public router.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to the extension:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- immediate caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the **router**, not the end user. So the extension receives `sender = router` and checks `allowedSwapper[pool][router]` — the actual user's address is never consulted.

---

### Impact Explanation

Two mutually exclusive failure modes, both breaking the allowlist invariant:

1. **DoS of router swaps**: If the pool admin allowlists specific users (e.g., `alice`, `bob`) but not the router, every swap through `MetricOmmSimpleRouter` reverts with `NotAllowedToSwap`, regardless of whether the actual user is allowlisted. All router-mediated swap functionality is permanently broken for that pool.

2. **Allowlist bypass**: If the admin allowlists the router address to restore router functionality, any unprivileged user — including addresses explicitly excluded from the allowlist — can bypass the restriction by calling `router.exactInputSingle()` or `router.exactInput()`. The allowlist provides zero protection against router-mediated swaps.

The `SwapAllowlistExtension` is a production periphery contract explicitly designed to restrict swap access to specific addresses. Both failure modes completely defeat its security guarantee.

---

### Likelihood Explanation

The `SwapAllowlistExtension` and `MetricOmmSimpleRouter` are both production contracts in the contest scope. Any pool that deploys with `SwapAllowlistExtension` and expects users to interact via the router (the standard user-facing path) will encounter this issue. The trigger requires only a normal user action — calling the public router — with no special privileges or setup.

---

### Recommendation

The pool must pass the **original end-user address** to the extension, not the immediate `msg.sender`. Two approaches:

1. **Router-level**: Have the router pass the original `msg.sender` as a verified parameter through `callbackData` or `extensionData`, and update the extension to decode and verify it. This requires trust that the router correctly reports the user.

2. **Pool-level**: Introduce a separate `originator` field (analogous to ERC-4337's `userOp.sender`) that the pool tracks via transient storage when a trusted router is the caller, and pass that to extensions instead of `msg.sender`.

The simplest safe fix is to have `SwapAllowlistExtension` check the `sender` argument only when `msg.sender` (the pool's caller) is not a known router, and check the actual user when it is — but this requires the extension to be aware of trusted routers, which is a design change.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as `beforeSwap` hook.
2. Admin calls `swapExtension.setAllowedToSwap(pool, alice, true)` — only `alice` is allowlisted.
3. `alice` calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `false` → reverts `NotAllowedToSwap`.
7. Alice, a legitimately allowlisted user, cannot swap through the router.

**Bypass variant:**
1. Admin adds `router` to the allowlist: `swapExtension.setAllowedToSwap(pool, router, true)`.
2. `mallory` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
3. Extension checks `allowedSwapper[pool][router]` → `true` → swap succeeds.
4. Mallory bypasses the allowlist entirely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
