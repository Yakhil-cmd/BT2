### Title
Missing Self-Membership Check in `validate_triple_inputs` Allows Protocol Hang for Non-Listed Participant - (File: `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

`validate_triple_inputs`, the shared input-validation helper for both `generate_triple` and `generate_triple_many`, omits the check `participants.contains(me)` that every other protocol entry point in the library enforces. When `me` is absent from the supplied participant list, the protocol silently proceeds, `me`'s own commitment is dropped by `ParticipantMap`, and the protocol hangs indefinitely waiting for messages that the other parties never send. Because Beaver triples are the mandatory prerequisite for OT-based ECDSA presigning and signing, this permanently denies signing for the affected honest party.

---

### Finding Description

Every other public protocol entry point in the library validates that the calling party is a member of the supplied participant list before starting the protocol:

- `assert_key_invariants` (DKG / keygen) [1](#0-0) 
- `presign` (OT-based ECDSA) [2](#0-1) 
- `presign` (robust ECDSA) [3](#0-2) 
- `sign` (OT-based ECDSA) [4](#0-3) 
- `sign` (robust ECDSA) [5](#0-4) 
- `ckd` [6](#0-5) 
- `assert_sign_inputs` (FROST) [7](#0-6) 

`validate_triple_inputs`, however, only checks participant count, threshold bounds, and duplicate-freedom. It never checks whether `me` is present:

```rust
fn validate_triple_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    let threshold = threshold.into();
    let threshold_value = threshold.value();
    if participants.len() < 2 { ... }
    if threshold_value > participants.len() { ... }
    if threshold_value < 2 { ... }
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;
    Ok((participants, threshold))   // ← no participants.contains(me) check
}
``` [8](#0-7) 

Both public entry points delegate to this helper without adding the check themselves:

```rust
pub fn generate_triple(participants, me, threshold, rng) -> ... {
    let (participants, threshold) = validate_triple_inputs(participants, threshold)?;
    // me is never verified to be in participants
    let fut = do_generation(ctx.clone(), participants, me, threshold, rng);
    ...
}
pub fn generate_triple_many<const N: usize>(participants, me, threshold, rng) -> ... {
    let (participants, threshold) = validate_triple_inputs(participants, threshold)?;
    let fut = do_generation_many::<N>(ctx.clone(), participants, me, threshold, rng);
    ...
}
``` [9](#0-8) 

Inside `do_generation_many`, the first thing that goes wrong is the commitment map initialisation:

```rust
for comi in my_commitments.iter().take(N) {
    let mut m = ParticipantMap::new(&participants);
    m.put(me, *comi);   // ← silently a no-op: me is not in the index map
    all_commitments_vec.push(m);
}
while all_commitments_vec.iter().any(|m| !m.full()) {
    let (from, commitments) = chan.recv(wait0).await?;  // ← hangs here
    ...
}
``` [10](#0-9) 

`ParticipantMap::put` silently ignores writes for participants not in its index:

```rust
pub fn put(&mut self, participant: Participant, data: T) {
    if let Some(&i) = self.participants.indices.get(&participant) { ... }
    // else: silently dropped
}
``` [11](#0-10) 

Because `me` is absent from `participants`, `m.put(me, *comi)` is a no-op. The map therefore requires N messages from the other parties to become `full()`. But the other parties use their own `participants` list (which also does not contain `me`) and call `send_many`, which routes only to listed participants — so `me` receives zero messages. The `while !full()` loop blocks forever.

---

### Impact Explanation

Beaver triples are the mandatory prerequisite for OT-based ECDSA presigning:

> "This requires a setup phase to have been conducted with these parties previously." [12](#0-11) 

Without a valid triple, `presign` cannot be called, and without a presignature, `sign` cannot be called. A permanent hang in `generate_triple` / `generate_triple_many` therefore permanently denies OT-based ECDSA signing for the affected honest party. This matches the allowed High impact: **Permanent denial of signing for honest parties under valid protocol inputs**.

---

### Likelihood Explanation

The entry path is reachable by any caller of the public API:

1. A **malicious coordinator** orchestrating the MPC session can supply `me` with a participant list that omits `me`. The function accepts it without error, `me` starts the protocol, and hangs.
2. A **misconfiguration** (e.g., off-by-one in participant list construction) produces the same outcome. Because every other entry point would immediately return `MissingParticipant`, a developer relying on the library's uniform validation behaviour would not anticipate that triple generation silently proceeds and hangs.

The inconsistency is directly observable by comparing `validate_triple_inputs` with `assert_key_invariants` — the missing line is a single `participants.contains(me)` guard present in all sibling functions.

---

### Recommendation

Add the self-membership check to `validate_triple_inputs`, consistent with every other protocol entry point:

```rust
fn validate_triple_inputs(
    participants: &[Participant],
    me: Participant,          // ← add me parameter
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    ...
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;
    // Add this guard:
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok((participants, threshold))
}
```

Pass `me` through from `generate_triple` and `generate_triple_many` to `validate_triple_inputs`.

```rust
fn validate_triple_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    let threshold = threshold.into();
    let threshold_value = threshold.value();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if threshold_value > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold_value,
            max: participants.len(),
        });
    }
    if threshold_value < 2 {
        return Err(InitializationError::ThresholdTooSmall {
            threshold: threshold_value,
            min: 2,
        });
    }
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;
    Ok((participants, threshold))
}
