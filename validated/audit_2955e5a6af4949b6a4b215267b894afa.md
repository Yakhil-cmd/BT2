### Title
Rebasing NEP-141 Token Deposits Cause ERC-20 Mirror Insolvency and Permanent Fund Freeze - (File: `engine/src/engine.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

The Aurora Engine's ERC-20 bridge mints mirror ERC-20 tokens 1:1 against the NEP-141 amount reported in `ft_on_transfer`. If the bridged NEP-141 token is a rebasing token (one whose supply adjusts without explicit transfers), the Aurora Engine's actual NEP-141 custody balance can silently diverge from the ERC-20 mirror's total supply. A negative rebase causes the bridge to become insolvent: the ERC-20 mirror supply exceeds the NEP-141 Aurora actually holds, and the last users to withdraw have their funds permanently frozen.

---

### Finding Description

When a NEP-141 token is bridged into Aurora, `ft_on_transfer` in `engine/src/contract_methods/connector.rs` routes to `receive_erc20_tokens` in `engine/src/engine.rs`. That function reads the transfer amount directly from the callback argument and mints exactly that many ERC-20 mirror tokens:

```rust
// engine/src/engine.rs:803
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount)  // calls mint(recipient, amount)
```

The ERC-20 mirror contract (`etc/eth-contracts/contracts/EvmErc20.sol`) tracks its own `totalSupply` in EVM storage, completely independent of the actual NEP-141 balance held by the Aurora Engine NEAR account.

When a user withdraws, `withdrawToNear` in `EvmErc20.sol` burns the ERC-20 tokens and calls the `ExitToNear` precompile with the burned amount. The precompile (`engine-precompiles/src/native.rs`, `exit_erc20_token_to_near`) then schedules an `ft_transfer` on the NEP-141 contract for that same amount:

```rust
// engine-precompiles/src/native.rs:632-636
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()
),
```

If the NEP-141 token is a rebasing token and its supply decreases after deposit, Aurora's actual NEP-141 balance shrinks without any corresponding reduction in the ERC-20 mirror's `totalSupply`. The invariant `ERC-20 totalSupply == NEP-141 balance held by Aurora` is broken. Subsequent `ft_transfer` calls will fail once Aurora's NEP-141 balance is exhausted.

The `exit_to_near_precompile_callback` in `connector.rs` handles the failure case only when the `error_refund` feature is compiled in (lines 231–239). Even with refunds, the ERC-20 tokens are re-minted to the user but the underlying NEP-141 is gone — the user is permanently unable to exit. Without `error_refund`, the ERC-20 tokens are burned with no recovery path at all.

There is no restriction or documentation preventing rebasing NEP-141 tokens from being registered via `deploy_erc20_token`.

---

### Impact Explanation

**Critical — Insolvency and Permanent Fund Freeze.**

After a negative rebase, the ERC-20 mirror's `totalSupply` exceeds the NEP-141 tokens Aurora actually holds. Users who withdraw first drain the available NEP-141 balance. Users who withdraw later find that `ft_transfer` fails on the NEP-141 contract because Aurora's balance is insufficient. Their ERC-20 tokens are either burned with no refund (if `error_refund` is disabled) or refunded but permanently unexitable (if `error_refund` is enabled). In both cases, a subset of users permanently loses access to the underlying NEP-141 value they deposited.

A positive rebase creates the symmetric problem: extra NEP-141 accumulates in Aurora with no corresponding ERC-20 tokens, making those tokens permanently unclaimable (locked yield / insolvency in the other direction).

---

### Likelihood Explanation

**Medium.** Any NEP-141 token with a rebasing mechanism (elastic supply, fee-on-transfer that adjusts balances, or supply-adjustment governance) can trigger this. The Aurora bridge accepts any NEP-141 token via `deploy_erc20_token` without filtering for rebasing behavior. A single bridged rebasing token is sufficient to trigger the insolvency for all depositors of that token pair. No privileged access is required; any token holder can initiate the deposit.

---

### Recommendation

1. **Short term**: Document that rebasing NEP-141 tokens are incompatible with the ERC-20 bridge and will cause accounting divergence. Consider adding a registry of approved tokens or an explicit blocklist for known rebasing tokens.
2. **Long term**: Implement balance-based accounting for ERC-20 minting. Instead of minting `args.amount`, query Aurora's actual NEP-141 balance before and after the `ft_transfer_call` and mint only the net difference. This is the standard "balance snapshot" pattern used to handle fee-on-transfer and rebasing tokens safely.

---

### Proof of Concept

1. A rebasing NEP-141 token `rebase.near` is registered via `deploy_erc20_token`, creating ERC-20 mirror at address `0xMIRROR`.
2. Alice calls `ft_transfer_call` on `rebase.near` transferring 100 tokens to Aurora with msg `<alice_evm_address>`. Aurora's `ft_on_transfer` → `receive_erc20_tokens` mints 100 `0xMIRROR` tokens to Alice.
3. Bob does the same, minting 100 `0xMIRROR` tokens. ERC-20 `totalSupply` = 200; Aurora holds 200 `rebase.near` tokens.
4. `rebase.near` governance executes a 50% negative rebase. Aurora's `rebase.near` balance drops to 100 without any NEAR transaction touching Aurora's state. ERC-20 `totalSupply` remains 200.
5. Alice calls `0xMIRROR.withdrawToNear(100, "alice.near")`. `EvmErc20.sol:54` burns 100 ERC-20. The `ExitToNear` precompile schedules `ft_transfer({receiver_id: "alice.near", amount: "100"})` on `rebase.near`. Aurora holds 100 tokens — transfer succeeds. Alice recovers her 100 tokens.
6. Bob calls `0xMIRROR.withdrawToNear(100, "bob.near")`. `EvmErc20.sol:54` burns 100 ERC-20. The precompile schedules `ft_transfer({receiver_id: "bob.near", amount: "100"})` on `rebase.near`. Aurora now holds 0 tokens — `ft_transfer` fails. Bob's 100 ERC-20 are already burned. With `error_refund` disabled, Bob's funds are permanently lost.     pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
