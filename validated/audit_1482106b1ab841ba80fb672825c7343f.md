### Title
Missing Access Control in `update_rewards` Allows Any Caller to Permanently Deny Block Rewards to All Stakers - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is specified to be callable only by the Starknet sequencer, but the implementation contains no caller check. Any unprivileged address can call it once per block with `disable_rewards: true`, consuming the global `last_reward_block` slot and permanently preventing the sequencer from distributing block rewards to any staker for that block.

### Finding Description
The protocol spec explicitly states the access control for `update_rewards`:

> **access control:** Only starkware sequencer.

However, the implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1449–1507) performs no caller identity check whatsoever. The only guards present are:

1. `self.general_prerequisites()` — checks if the contract is paused.
2. `current_block_number > self.last_reward_block.read()` — prevents calling twice in the same block.
3. Staker existence and active-status checks. [1](#0-0) 

The function then unconditionally writes the current block number to the global `last_reward_block` storage variable: [2](#0-1) 

Because `last_reward_block` is a single contract-wide value (not per-staker), writing it once blocks the sequencer from calling `update_rewards` for **any** staker in that block. The spec confirms this is the intended per-block gate: [3](#0-2) 

### Impact Explanation
An attacker calls `update_rewards(any_valid_staker_address, disable_rewards: true)` at the start of every block. This:

1. Sets `last_reward_block` to the current block number.
2. Returns early (no rewards distributed) because `disable_rewards` is `true`.
3. When the sequencer subsequently attempts to call `update_rewards` for any staker in the same block, it reverts with `REWARDS_ALREADY_UPDATED`.

The block rewards for that block are permanently lost — they are never credited to `unclaimed_rewards_own` of any staker, and never transferred to any delegation pool. This constitutes **theft of unclaimed yield** (High severity): stakers and delegators are permanently denied the block rewards they earned. [4](#0-3) 

### Likelihood Explanation
Likelihood is **High**:
- No special role, token balance, or prior interaction is required — any EOA can call the function.
- The attack costs only gas (one transaction per block).
- It can be fully automated with a bot that fires once per block.
- The attacker needs only to supply any currently-active staker address (publicly readable from on-chain state).
- There is no economic disincentive; the attacker loses only gas fees while all stakers lose their block rewards indefinitely.

### Recommendation
Add a caller check at the top of `update_rewards` that restricts execution to the authorized sequencer address, mirroring the pattern already used for `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    ...
}
```

The sequencer address should be stored in contract configuration and validated here, consistent with how `assert_caller_is_attestation_contract` is implemented for the attestation path. [5](#0-4) 

### Proof of Concept

```
1. Consensus rewards are active (post-consensus epoch).
2. Staker S is active with non-zero balance.
3. Attacker A (any address) calls:
       staking.update_rewards(staker_address: S, disable_rewards: true)
   at block N.
   → last_reward_block is written to N.
   → No rewards distributed (disable_rewards=true causes early return).
4. Sequencer calls:
       staking.update_rewards(staker_address: S, disable_rewards: false)
   at block N.
   → Panics: REWARDS_ALREADY_UPDATED (current_block_number == last_reward_block).
5. Staker S receives zero rewards for block N.
6. Attacker repeats step 3 every block → stakers receive zero rewards indefinitely.
```

The `REWARDS_ALREADY_UPDATED` guard in the test suite confirms this is the exact mechanism that prevents double-calling: [6](#0-5) [7](#0-6)
