### Title
Missing Caller Validation in `exit_delegation_pool_action` Allows Anyone to Force Pool Member Exit - (File: `src/pool/pool.cairo`)

---

### Summary

`Pool::exit_delegation_pool_action` accepts a `pool_member: ContractAddress` parameter but never validates that `get_caller_address() == pool_member`. Any unprivileged caller can trigger the exit action for any pool member whose exit window has elapsed, forcibly sending their funds to the pool member's address and destroying their pending exit intent — permanently blocking their ability to switch delegation pools without re-entering.

---

### Finding Description

`exit_delegation_pool_action` is a public function in the `Pool` contract that finalizes a pool member's pending withdrawal. It takes a `pool_member` address as a parameter, but unlike every other sensitive pool function, it contains **no access-control check** on the caller.

Compare the pattern used in `claim_rewards` and `add_to_delegation_pool`:

```cairo
// claim_rewards — has caller check
let caller_address = get_caller_address();
assert!(
    caller_address == pool_member || caller_address == reward_address,
    "{}",
    Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
);
``` [1](#0-0) 

```cairo
// add_to_delegation_pool — has caller check
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
``` [2](#0-1) 

Now the vulnerable function:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    // Asserts.
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    let unpool_time = pool_member_info
        .unpool_time
        .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
    assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);
    // ← NO caller check here
``` [3](#0-2) 

The function then clears `unpool_amount` and `unpool_time` and transfers the funds to `pool_member`:

```cairo
pool_member_info.unpool_amount = Zero::zero();
pool_member_info.unpool_time = Option::None;
// ...
token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());
``` [4](#0-3) 

The interface documentation for `exit_delegation_pool_action` lists no access-control restriction, while `exit_delegation_pool_intent` (the intent step) correctly derives the pool member from `get_caller_address()`:

```cairo
fn exit_delegation_pool_intent(ref self: ContractState, amount: Amount) {
    let pool_member = get_caller_address();
``` [5](#0-4) 

The asymmetry is the root cause: intent is caller-gated, action is not.

---

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users.**

The funds are transferred to the correct `pool_member` address, so there is no direct theft. However, the attack causes concrete damage:

1. **Blocks pool switching**: A pool member who has signalled exit intent can use `switch_delegation_pool` to redirect their funds to a different staker's pool without withdrawing. This requires `unpool_time.is_some()`:

```cairo
assert!(
    pool_member_info.unpool_time.is_some(),
    "{}",
    GenericError::MISSING_UNDELEGATE_INTENT,
);
``` [6](#0-5) 

An attacker who front-runs `switch_delegation_pool` with `exit_delegation_pool_action` clears `unpool_time`, making the switch impossible. The pool member's funds are forcibly returned to their wallet instead of being redirected, and they must re-enter the pool from scratch (paying gas, losing any accrued rewards during the re-entry epoch gap).

2. **Forced state transition**: The pool member's carefully timed exit is executed at the attacker's discretion, not their own. This can disrupt strategies that depend on timing (e.g., re-delegating at a specific epoch boundary).

---

### Likelihood Explanation

**High likelihood.** The function is fully public with no role or caller restriction. Any address on Starknet can call it. The only precondition is that the target pool member has a pending exit intent and the exit window has elapsed — both are observable on-chain. A griefing attacker or a competing staker who wants to prevent a delegator from switching to a rival pool has a clear, cheap, and reliable path to exploit this.

---

### Recommendation

Add a caller check identical to the pattern used in `claim_rewards` and `add_to_delegation_pool`. The simplest fix:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let caller_address = get_caller_address();
    assert!(
        caller_address == pool_member,
        "{}",
        Error::CALLER_IS_NOT_POOL_MEMBER, // or reuse an existing error
    );
    // ... rest of function unchanged
```

This mirrors the access-control pattern already established for `exit_delegation_pool_intent`, which derives the pool member exclusively from `get_caller_address()`. [7](#0-6) 

---

### Proof of Concept

1. Pool member `Alice` calls `exit_delegation_pool_intent(amount)` — sets `unpool_time` and `unpool_amount`.
2. Alice waits for the exit window to elapse, intending to call `switch_delegation_pool(to_staker, to_pool, amount)` to redirect her stake.
3. Attacker `Eve` observes Alice's pending intent on-chain and calls `exit_delegation_pool_action(pool_member: Alice)` once `Time::now() >= unpool_time`.
4. The function executes without any caller check:
   - `unpool_time` is set to `None` [8](#0-7) 
   - `unpool_amount` is zeroed [9](#0-8) 
   - Funds are transferred to Alice's address [10](#0-9) 
5. Alice's subsequent `switch_delegation_pool` call reverts with `MISSING_UNDELEGATE_INTENT` because `unpool_time` is now `None`. [6](#0-5) 
6. Alice must re-enter the pool, losing rewards for the missed epoch(s) and paying additional gas.
