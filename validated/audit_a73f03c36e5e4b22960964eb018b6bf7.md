### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Protocol's Hard Fee Cap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` / `addFeeSellE6` (both `uint16`, max 65 535 E6 = 6.5535%) directly to the pool with **no cap check**, while the analogous `setPoolAdminFees` enforces `maxAdminSpreadFeeE6`. A pool admin can therefore push the effective per-bin spread fee above the protocol's hard ceiling of 20%, causing traders to pay more than the protocol's intended maximum and constituting an admin-boundary break.

### Finding Description

`setPoolAdminFees` correctly guards the base admin spread and notional components:

```solidity
// MetricOmmPoolFactory.sol  lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, however, performs **no such check** and passes the caller-supplied values straight through:

```solidity
// MetricOmmPoolFactory.sol  lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index: [3](#0-2) 

During a swap, the per-bin additional fee is added directly to `baseFeeX64`:

```solidity
// MetricOmmPool.sol  line 999
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

`type(uint16).max` = 65 535 E6 ≈ 6.55%. When the base spread fee is already at the hard cap of 20% (200 000 E6), the effective per-bin fee becomes ≈ 26.55%, exceeding the hard ceiling enforced everywhere else in the system. [5](#0-4) 

### Impact Explanation

Traders swapping through the affected bin pay a spread fee up to 26.55% instead of the protocol-guaranteed maximum of 20%. The excess fee accrues to LPs in that bin. This is a direct loss of trader principal above the protocol's own hard cap, and a clear admin-boundary break: the pool admin exceeds the cap that the factory is supposed to enforce.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly constrained to act "inside caps." The uncapped path is reachable by any current pool admin with a single transaction and no preconditions beyond holding the role. No timelock or additional approval is required.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Define a factory-level constant or configurable cap (e.g. `maxAdminBinAdditionalFeeE6`) and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it:

```solidity
if (addFeeBuyE6  > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
```

### Proof of Concept

1. Pool is deployed with `adminSpreadFeeE6 = 200_000` (20%, the hard cap) and `protocolSpreadFeeE6 = 0`.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. A trader executes a swap through bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ 26.55%, exceeding the 20% hard cap.
4. The trader's output is reduced by the uncapped additional fee; the excess goes to LPs in that bin.
5. Calling `factory.setPoolAdminFees(pool, 200_001, 0)` for comparison reverts with `AdminFeeTooHigh` — confirming the asymmetry.
