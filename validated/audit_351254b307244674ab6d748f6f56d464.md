### Title
Missing Failure Broadcast Mechanism in DKG/Reshare/Refresh Causes Permanent Denial for Honest Parties - (File: src/dkg.rs)

### Summary
`broadcast_success` in `src/dkg.rs` is the only final-round notification mechanism and it unconditionally broadcasts `(true, session_id)`. There is no corresponding failure broadcast path. When a malicious participant causes an honest participant to abort before reaching `broadcast_success`, all other honest participants that successfully complete their checks enter the echo broadcast and wait indefinitely for the aborted participant's SEND message, which never arrives. The protocol provides no way to terminate this wait.

### Finding Description
`do_keyshare` (lines 342–538, `src/dkg.rs`) implements DKG, reshare, and refresh. Its final step at line 531 is:

```rust
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

`broadcast_success` (lines 307–338) always broadcasts `(true, session_id)`:

```rust
let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
```

There is no code path that broadcasts `(false, session_id)` or any failure token. The function is only ever called when all prior checks have passed.

If participant A encounters an error before line 531 — for example, `validate_received_share` at line 522 returns `Err(ProtocolError::InvalidSecretShare(M))` because malicious participant M sent A a bad share — A returns an error from `do_keyshare` and never calls `broadcast_success`. A therefore never sends its SEND message in the echo broadcast.

The echo broadcast protocol (`do_broadcast` / `reliable_broadcast_receive_all`) requires all N sessions to complete before returning:

```rust
if state.iter().all(|x| x.finish_ready) {
    return Ok(vote_output);
}
```

Honest participants B, C, D that complete all checks and enter `broadcast_success` call `do_broadcast`, send their own SEND messages, and then wait for A's SEND message. Since A never sends it, B, C, D are permanently blocked. There is no timeout, no fallback, and no mechanism for A to signal its failure to the waiting parties.

**Concrete attack path (N = 4, threshold = 2, M malicious):**
1. M sends valid shares to B, C, D but a crafted invalid share to A during Round 4 (`send_private` at line 505).
2. A calls `validate_received_share` at line 522, which fails; A returns `Err(InvalidSecretShare(M))` and exits `do_keyshare`.
3. B, C, D receive valid shares from all parties, pass all checks, and reach line 531.
4. B, C, D each call `broadcast_success`, which calls `do_broadcast` and sends their SEND messages.
5. The echo broadcast loop waits for all four SEND messages. A's never arrives.
6. B, C, D are permanently stuck; DKG/reshare/refresh never completes for any honest party.

The same liveness failure applies to reshare and refresh because both call `do_keyshare` and terminate with the same `broadcast_success` call.

### Impact Explanation
**High — Permanent denial of key generation, reshare, and refresh for all honest parties.** A single malicious participant (within the documented `MaxFaulty = floor((N-1)/3)` bound) can permanently prevent honest parties from completing any DKG, reshare, or refresh session. Because the library provides no timeout and no failure-notification path, the honest parties have no way to detect or recover from the stall without external intervention. All key management operations that depend on these protocols are permanently blocked.

### Likelihood Explanation
**Medium.** The attack requires one malicious participant, which is explicitly within the documented trust model. The technique — sending a single invalid share to one honest participant — is trivial to execute and requires no special capability beyond participation in the protocol. Any adversary who is a legitimate protocol participant can trigger this deterministically.

### Recommendation
Implement a failure broadcast path. Replace the unconditional `(true, session_id)` in `broadcast_success` with a result parameter so that a participant that detects an error can still enter the final broadcast round and send `(false, session_id)`. All participants — whether they succeeded or failed — must call this final broadcast before exiting, so that every honest party receives a complete set of votes and can terminate cleanly. This mirrors the standard "abort with identification" pattern used in robust MPC protocols.

    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
