### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `change_reward_address` functions in `staking.cairo` and `pool.cairo` accept an arbitrary `ContractAddress` for the new reward destination without checking whether it is the zero address (`0x0`). If a staker or pool member sets their `reward_address` to zero, all future reward claims are transferred to the zero address and permanently lost.

---

### Finding Description

In `staking.cairo`, `change_reward_address` performs only one validation: that the supplied address is not a registered token address. [1](#0-0) 

In `pool.cairo`, `change_reward_address` performs only one validation: that the supplied address is not the pool's token address. [2](#0-1) 

The zero address (`0x0`) is not a registered token, so both checks pass silently. The updated `reward_address` is written to storage with no further guard.

When rewards are subsequently claimed, `send_rewards_to_staker` in `staking.cairo` executes: [3](#0-2) 

And `claim_rewards` in `pool.cairo` executes: [4](#0-3) 

Both call `checked_transfer(recipient: reward_address, ...)` with the zero address as recipient. On Starknet, ERC-20 transfers to the zero address succeed without reverting, so the STRK tokens are permanently burned.

The same loss occurs during `unstake_action`, which calls `send_rewards_to_staker` before returning the principal, meaning accumulated rewards are silently destroyed at exit time as well. [5](#0-4) 

---

### Impact Explanation

A staker or pool member who sets `reward_address` to `0x0` — whether by mistake or through a front-end bug — will have all accrued STRK rewards permanently transferred to the zero address on every subsequent `claim_rewards` or `unstake_action` call. The tokens are irrecoverable. This maps directly to **High: Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The entry path is fully unprivileged: any active staker calls `Staking::change_reward_address(0x0)`, and any pool member calls `Pool::change_reward_address(0x0)`. No special role or key is required. The most realistic trigger is a UI/integration bug that passes an uninitialized or empty address field, which is a common class of user-facing error. The absence of any zero-address guard means the contract provides no safety net.

---

### Recommendation

Add an explicit zero-address check at the top of both `change_reward_address` implementations, mirroring the pattern already used for caller validation elsewhere in the codebase:

```cairo
assert!(reward_address.is_non_zero(), "{}", Error::CALLER_IS_ZERO_ADDRESS);
```

Apply the same guard to the `reward_address` parameter of `enter_delegation_pool` in `pool.cairo` for consistency. [6](#0-5) 

---

### Proof of Concept

1. Staker calls `Staking::change_reward_address(reward_address: 0x0)`.
   - Check `!does_token_exist(0x0)` → passes (zero is not a token).
   - `staker_info.reward_address` is written as `0x0`.
2. Protocol epoch advances; attestation occurs; rewards accrue to `staker_info.unclaimed_rewards_own`.
3. Staker calls `Staking::claim_rewards(staker_address)`.
   - Authorization: `caller == staker_address` → passes.
   - `send_rewards_to_staker` calls `checked_transfer(recipient: 0x0, amount: rewards)`.
   - STRK tokens are transferred to the zero address and permanently lost.
4. Same sequence applies to a pool member via `Pool::change_reward_address(0x0)` followed by `Pool::claim_rewards`. [7](#0-6) [8](#0-7)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
