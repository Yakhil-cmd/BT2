### Title
`DepositAllowlistExtension` gates by `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument (the actual caller who pays tokens) and only checks the `owner` argument (the position holder) against the allowlist. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint — it is explicitly an operator-pattern function — any non-allowlisted address can call `addLiquidity` with an allowlisted address as `owner`, pass the extension gate, and add liquidity to the pool while the shares are credited to the allowlisted address.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` with no ownership check:

```solidity
// MetricOmmPool.sol L182-196
function addLiquidity(
    address owner,   // ← any address; no msg.sender == owner guard
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    ...
}
``` [1](#0-0) 

The pool passes both `sender` (= `msg.sender`) and `owner` to every extension. `DepositAllowlistExtension.beforeAddLiquidity` discards `sender` entirely and only checks `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The same bypass is reachable through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)`, whose `_validateOwner` only rejects `address(0)` — it does not require `owner == msg.sender`:

```solidity
// MetricOmmPoolLiquidityAdder.sol L247-249
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
``` [3](#0-2) 

**Attack path:**

1. Pool is deployed with `DepositAllowlistExtension` active and `allowAllDepositors = false`.
2. Alice is allowlisted (`allowedDepositor[pool][alice] = true`). Bob is not.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")` directly (or routes through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, ...)`).
4. `_beforeAddLiquidity` fires; the extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
5. Bob's callback pays the required tokens; shares are minted to `(alice, salt)`.
6. Bob has added liquidity to the pool despite being non-allowlisted.

The protocol's own research file explicitly flags this attack surface:

> *"The attacker can separate payer from owner and can route through the liquidity adder, so the checked identity has to be exactly the one the pool intends to gate."* [4](#0-3) 

---

### Impact Explanation

The `DepositAllowlistExtension` is the pool admin's primary mechanism for restricting who may add liquidity (e.g., for KYC/AML compliance, permissioned market-making, or regulatory gating). By routing through any allowlisted `owner` address, a non-allowlisted caller completely bypasses this gate and injects liquidity into the pool. This is an admin-boundary break: a pool admin-configured access control is circumvented by an unprivileged path, violating the invariant that only allowlisted depositors can add liquidity.

Secondary effects include: Alice receives shares she did not initiate, which may interfere with her position accounting or downstream integrations that track her LP balance; and the pool's bin state is modified by an actor the admin explicitly excluded.

---

### Likelihood Explanation

The exploit requires no special privilege, no flash loan, and no front-running. Any EOA or contract that holds the pool's tokens can execute it in a single transaction. The only precondition is knowing one allowlisted address, which is observable on-chain from past `addLiquidity` calls or from the extension's `allowedDepositor` mapping.

---

### Recommendation

Gate by `sender` (the economically active depositor who pays tokens), not by `owner` (the passive position holder):

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

If the intent is to gate both who pays **and** who holds the position, check both `sender` and `owner`.

### Proof of Concept

```solidity
// Assume: pool has DepositAllowlistExtension; alice is allowlisted; bob is not.

// Bob calls directly on the pool (no periphery needed):
vm.startPrank(bob);
token0.approve(address(pool), type(uint256).max);
token1.approve(address(pool), type(uint256).max);

// Pass alice as owner — extension checks allowedDepositor[pool][alice] == true → passes
pool.addLiquidity(
    alice,          // owner: allowlisted address
    uint80(0),      // salt
    deltas,         // shares in target bins
    callbackData,   // bob pays via callback
    ""
);
vm.stopPrank();

// Result: bob added liquidity to the pool despite not being allowlisted.
// Shares are credited to (alice, 0, bin), which alice can remove.
// The allowlist invariant is broken.
```
