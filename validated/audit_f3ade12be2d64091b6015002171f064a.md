### Title
Unbounded `stakers` Vector Growth Enables Permanent Griefing of `get_stakers` - (File: src/staking/staking.cairo)

### Summary
An unprivileged attacker can permanently inflate the `stakers` vector by staking with many addresses at minimum stake and then unstaking. Because stakers are never removed from the vector and staker addresses cannot be reused after unstaking, the vector grows monotonically. This causes `get_stakers` to iterate over an ever-growing set of dead entries, producing unbounded gas consumption that can eventually make the function uncallable.

### Finding Description
In `staking.cairo`, every call to `stake()` appends the caller's address to the `stakers` vector:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
```

The storage comment explicitly documents that removal never happens:

```cairo
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
```

After `unstake_action` is called, `remove_staker` erases the staker's info from the protocol, but the address entry in `stakers` persists. Furthermore, `assert_staker_address_not_reused` permanently blocks any address that has ever staked from staking again, because `staker_own_balance_trace` is non-empty after the first use:

```cairo
fn assert_staker_address_not_reused(self: @ContractState, staker_address: ContractAddress) {
    assert!(
        self.staker_own_balance_trace.entry(key: staker_address).is_empty(),
        "{}",
        Error::STAKER_ADDRESS_ALREADY_USED,
    );
}
```

The `get_stakers` function (in `IStakingConsensus`) iterates over the entire `stakers` vector to compute staking power for each entry. Dead entries (addresses that have completed `unstake_action`) are iterated but contribute zero staking power. As the vector fills with dead entries, each call to `get_stakers` consumes proportionally more gas.

### Impact Explanation
**Medium**: Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption. `get_stakers` is the consensus-layer function that returns staking power for all active stakers. As the `stakers` vector grows with permanently dead entries, the cost of calling `get_stakers` grows without bound. On Starknet, where transactions have finite gas budgets, a sufficiently large vector will cause `get_stakers` to exceed the block gas limit and revert, making staking-power queries uncallable on-chain and severely degrading off-chain consensus tooling.

### Likelihood Explanation
**Low**: The attacker recovers their staked STRK after the exit wait window, so the only permanent cost is gas fees for N stake/unstake cycles. The minimum stake threshold raises the capital requirement, but the attacker suffers no permanent capital loss. The attack is therefore economically feasible for a well-funded griever.

### Recommendation
Remove stakers from the `stakers` vector when `unstake_action` completes, or replace the append-only `Vec` with a data structure that supports O(1) deletion (e.g., a swap-and-pop pattern). Alternatively, track active staker count separately and skip zero-power entries without iterating them, or impose a hard cap on the number of registered stakers.

### Proof of Concept
1. Attacker creates N addresses and funds each with `min_stake` STRK.
2. Each address calls `stake(reward_address, operational_address, min_stake)` — each address is pushed to `stakers`.
3. Each address calls `unstake_intent()` — staking power drops to zero, but the address remains in `stakers`.
4. After `exit_wait_window` elapses, each address calls `unstake_action()` — STRK is returned, staker info is erased, but the `stakers` vector entry is permanent.
5. `assert_staker_address_not_reused` now blocks these addresses from ever staking again, so the dead entries are irremovable.
6. Every subsequent call to `get_stakers` must iterate all N dead entries plus any live stakers.
7. Repeating steps 1–4 with fresh addresses grows the vector further. At sufficient N, `get_stakers` exceeds the Starknet gas limit and reverts, permanently breaking consensus-layer staking-power queries.
