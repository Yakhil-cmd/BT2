### Title
WalletConnect Offer Approval Displays Understated "Total Amount with Royalties" When Multiple NFTs Have Royalties - (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` divides the fungible amount by the number of NFTs before applying each royalty percentage, producing a "Total Amount with Royalties" that is systematically lower than the actual on-chain deduction. This figure is the primary amount shown to the user in the WalletConnect `Confirm` dialog. A malicious dApp can craft a multi-NFT offer so the displayed total is far below what the wallet will actually spend, causing the user to approve a transaction under false pretenses.

### Finding Description

`formatAmountWithRoyalties` computes the royalty-inclusive total as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

With N NFTs each carrying royalty percentage `r_i`, the formula computes:

```
royaltyAmount = Σ ( (amount / N) × r_i / 10000 )
              = (amount / N) × Σ r_i / 10000
```

The correct formula is:

```
royaltyAmount = Σ ( amount × r_i / 10000 )
              = amount × Σ r_i / 10000
```

The displayed total is therefore **N times smaller** in the royalty component than the actual on-chain cost. The existing test fixture confirms this:

- 2 NFTs with royalties 500 (5%) and 10 (0.1%), amount = 100 000 000 mojos
- Displayed `amountWithRoyalties`: `0.00010255` XCH
- Correct value: `0.0001051` XCH (≈ 2.4% understatement here; grows with royalty magnitude) [2](#0-1) 

The `amountWithRoyalties` field is passed directly into the WalletConnect `Confirm` dialog, where it is rendered as **"Total Amount with Royalties"** — the primary spend figure shown to the user before they click Approve. [3](#0-2) 

In `OfferBuilderValue`, when `amountWithRoyalties` is present it **replaces** the base amount as the displayed value:

```typescript
const displayValue = React.useMemo(
  () =>
    amountWithRoyalties ||
    (!value ? <Trans>Not Available</Trans> : ...),
  [amountWithRoyalties, type, value],
);
``` [4](#0-3) 

### Impact Explanation

A user approving a WalletConnect `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command that involves multiple royalty-bearing NFTs will see a "Total Amount with Royalties" that is materially lower than the amount the blockchain will actually deduct. The user's approval decision is based on a false total. The actual spend is determined by the backend and will be higher than what was displayed. This constitutes WalletConnect state spoofing that causes a user to approve the wrong amount.

### Likelihood Explanation

Any dApp connected via WalletConnect can send a `take_offer` command containing an offer with two or more NFTs that carry royalties. No special privileges are required. The discrepancy scales with both the number of NFTs and the royalty percentages, making it straightforward for a malicious dApp to engineer a scenario where the displayed total is significantly lower than the actual deduction.

### Recommendation

Remove the `splitAmount` division. Each royalty percentage should be applied to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [5](#0-4) 

### Proof of Concept

Using the existing test setup in `parseCommandDisplay.test.ts`:

1. Two NFTs offered: `firstNftLauncherId` (royalty 500 = 5%) and `secondNftLauncherId` (royalty 10 = 0.1%)
2. Requested: 100 000 000 mojos (0.0001 XCH)
3. Current code produces `amountWithRoyalties = '0.00010255'` XCH
4. Correct value: `0.0001 + (0.0001 × 5%) + (0.0001 × 0.1%) = 0.0001051` XCH
5. The Confirm dialog shows "Total Amount with Royalties: 0.00010255 XCH" — the user approves believing they spend ~0.00010255 XCH, but the wallet deducts ~0.0001051 XCH [1](#0-0) [6](#0-5)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
