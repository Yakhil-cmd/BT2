### Title
Malicious Participant Can Abort Triple Generation via Crafted `dlogeq` Proof Triggering Fatal `IdentityElement` Error — (`src/crypto/proofs/dlogeq.rs`, `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

A malicious participant can craft a `dlogeq` proof `(e, s)` paired with a chosen `big_c_j` such that the reconstructed commitment point `big_k1` in `verify` equals the group identity. This causes `encode_two_points` to return `Err(ProtocolError::IdentityElement)`, which is propagated as a fatal error via `?` in `do_generation_many`, permanently aborting triple generation for all honest parties.

---

### Finding Description

**Root cause — `verify` returns `Err` on identity commitment point:**

In `dlogeq::verify`, the reconstructed commitment is:

```
big_k1 = generator1 * s - public1 * e
``` [1](#0-0) 

If `big_k1` equals the group identity, `encode_two_points` returns `Err(ProtocolError::IdentityElement)` via the `?` on line 157: [2](#0-1) 

This means `verify` returns `Err(...)` rather than `Ok(false)` — a structurally well-formed proof can trigger a fatal error path instead of a clean rejection.

**Propagation — `do_generation_many` treats this `Err` as fatal:**

At Spec 4.1–4.3, the verifier calls `dlogeq::verify` with `?`: [3](#0-2) 

The `?` propagates `Err(ProtocolError::IdentityElement)` directly out of `do_generation_many`, aborting the entire protocol for all honest parties.

**Attacker control — `public1 = big_c_j` is uncommitted at the time of the attack:**

The malicious party sends `(big_c_j_v, their_phi_proofs)` at `wait4`: [4](#0-3) 

`big_c_j` is **not** included in the hash commitment sent at `wait0` (which only covers `big_E_i, big_F_i, big_L_i`). The malicious party therefore has full freedom to choose `big_c_j` after observing all other parties' polynomial commitments at `wait2`.

**Concrete construction:**

By `wait2`, the malicious party knows `generator1 = big_F(0)` (the aggregated sum of all parties' `big_F_i(0)` values): [5](#0-4) 

The attacker picks any non-zero scalars `s` and `e`, then sets:

```
public1 = big_c_j = generator1 * s * e⁻¹
```

This gives `big_k1 = generator1 * s − (generator1 * s * e⁻¹) * e = identity`.

`public1` is a valid non-identity point, so `statement.encode()` succeeds. Only `encode_two_points(&big_k0, &big_k1)` fails, returning `Err(IdentityElement)`.

---

### Impact Explanation

Any single malicious participant in a triple generation session can unconditionally abort the session for all honest parties by sending one crafted message at `wait4`. Since `do_generation_many` propagates the error fatally, honest parties cannot complete triple generation, which is a prerequisite for ECDSA presigning and signing. This matches the **High** impact: *Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions*.

---

### Likelihood Explanation

The attack requires only that the adversary be a legitimate participant in the triple generation protocol — no cryptographic assumptions need to be broken. The crafted `big_c_j` and proof `(e, s)` are trivially computable from public information available after `wait2`. The attack is deterministic and repeatable on every session attempt.

---

### Recommendation

In `do_generation_many`, treat `Err` from `dlogeq::verify` as a verification failure rather than a fatal protocol error:

```rust
let ok = dlogeq::verify(
    &mut transcript.fork(b"dlogeq0", &from.bytes()),
    statement,
    their_phi_proof,
).unwrap_or(false);  // treat Err as false
if !ok {
    return Err(ProtocolError::AssertionFailed(format!(
        "dlogeq proof from {from:?} failed to verify"
    )));
}
```

Alternatively, `dlogeq::verify` itself should be changed to return `Ok(false)` instead of `Err(IdentityElement)` when the reconstructed commitment is the identity, since that is a well-defined proof-rejection condition, not an internal protocol error.

---

### Proof of Concept

```rust
// Attacker knows generator1 = big_F(0) after wait2.
// Choose arbitrary non-zero s and e:
let s = Scalar::from(42u64);
let e = Scalar::from(7u64);
let e_inv = e.invert().unwrap();

// Craft public1 so that generator1 * s - public1 * e = identity:
let public1 = generator1 * s * e_inv;

// Build the crafted proof:
let crafted_proof = dlogeq::Proof { e, s };

// Verify returns Err, not Ok(false):
let result = dlogeq::verify(&mut transcript, statement_with_public1, &crafted_proof);
assert!(matches!(result, Err(ProtocolError::IdentityElement)));

// The ? in do_generation_many propagates this, aborting all honest parties.
```
