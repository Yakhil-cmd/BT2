### Title
Unverified `transferFrom` Return Value in XCC Precompile Allows Payment Bypass - (`engine-precompiles/src/xcc.rs`)

### Summary

The XCC precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect payment from the caller before creating a cross-contract call promise. The `exit_reason` of the inner EVM call is checked for revert/error/fatal, but the ABI-encoded `bool` return value is never decoded. If the wNEAR contract returns `false` (transfer failed) without reverting, the precompile treats the call as successful and proceeds to schedule the NEAR promise without the required payment having been made.

### Finding Description

In `engine-precompiles/src/xcc.rs`, `CrossContractCall::run_with_handle` collects the required NEAR payment by calling `transferFrom` on the wNEAR ERC-20 contract via `handle.call()`:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e) => { ... }
    aurora_evm::ExitReason::Fatal(f) => { ... }
}
``` [1](#0-0) 

The `return_value` variable is captured but is only forwarded as the revert output in the `Revert` arm. In the `Succeed` arm it is silently discarded. The ERC-20 `transferFrom` function returns an ABI-encoded `bool`; a conforming but non-reverting implementation that returns `false` on failure (e.g., insufficient allowance or balance) will produce `ExitReason::Succeed` with `return_value = abi.encode(false)`. The match arm treats this identically to a successful transfer.

The `transfer_from_args` helper constructs a standard ERC-20 `transferFrom(address,address,uint256)` call: [2](#0-1) 

After the unchecked call, the precompile unconditionally emits the promise log that causes the engine to schedule the NEAR cross-contract call: [3](#0-2) 

The `required_near` amount is computed from the user-supplied promise arguments and the storage staking requirement: [4](#0-3) 

### Impact Explanation

If the wNEAR ERC-20 contract returns `false` instead of reverting on a failed transfer:

1. The caller's wNEAR EVM balance is **not reduced** (the EVM state change inside `transferFrom` did not occur).
2. The engine's implicit address does **not receive** the wNEAR.
3. The XCC promise log is still emitted and the engine schedules the NEAR promise chain (`withdraw_wnear_to_router` → router `execute`).
4. The `withdraw_wnear_to_router` step will fail at the NEAR level because the engine's implicit address has no wNEAR to unwrap, consuming the engine's prepaid NEAR gas without the caller having paid for it.

An attacker can repeatedly trigger this path to drain the engine's NEAR gas budget at no wNEAR cost to themselves, causing a **temporary freeze** of XCC functionality for legitimate users and constituting **theft of the engine's NEAR gas reserves** (unclaimed yield / operational funds).

### Likelihood Explanation

The wNEAR contract address is stored in engine state and is set by the owner. The currently deployed wNEAR contract is expected to revert on failure, making this low-probability under normal conditions. However:

- The wNEAR address can be updated (`factory_set_wnear_address`), and a future or alternative wNEAR implementation could follow the older ERC-20 pattern of returning `false`.
- The code pattern is unconditionally unsafe regardless of the current wNEAR behavior, matching the exact vulnerability class from the reference report (assuming all money markets revert on error). [5](#0-4) 

### Recommendation

After the `Succeed` arm, decode `return_value` as an ABI-encoded `bool` and revert if it is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // ERC-20 transferFrom returns abi.encode(bool)
    let success = return_value.last().copied().unwrap_or(0) != 0;
    if !success {
        return Err(revert_with_message("ERR_WNEAR_TRANSFER_FAILED"));
    }
}
```

This mirrors the pattern already used for `exit_reason` checking and ensures that a non-reverting-but-failing wNEAR implementation cannot bypass the payment gate.

### Proof of Concept

1. Deploy or configure a wNEAR ERC-20 contract whose `transferFrom` returns `false` (instead of reverting) when the caller has insufficient allowance.
2. Call the XCC precompile from an EVM account with zero wNEAR allowance granted to the precompile address.
3. Observe that `exit_reason` is `Succeed` (the call did not revert), `return_value` is `abi.encode(false)`, and the precompile emits the promise log anyway.
4. The engine schedules the NEAR promise chain; `withdraw_wnear_to_router` fails consuming NEAR gas, while the attacker's wNEAR balance is unchanged. [6](#0-5)
