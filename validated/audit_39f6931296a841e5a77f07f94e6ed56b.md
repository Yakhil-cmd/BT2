### Title
`skip_stateful_validations` bypasses `__validate__` for invoke transactions when no deploy_account exists in the pool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point call (and therefore signature verification) for an invoke with nonce=1 **only** when the account is undeployed and has a pending `deploy_account` transaction. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` if **any** transaction exists for the account in the pool, not specifically a `deploy_account`. After a `deploy_account` fails and is removed from the pool as a rejected transaction, the paired invoke(nonce=1) remains in the pool. A subsequent invoke(nonce=1) submitted for the same address then triggers the skip again, bypassing signature verification at the gateway, even though no `deploy_account` is pending.

---

`skip_stateful_validations` at line 429 of `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when two conditions hold:

```rust

    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())

``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` if the pool holds **any** transaction for the address, regardless of type: [3](#0-2) 

The code comment claims this is safe because the pool entry "means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is circular and breaks after a `deploy_account` failure: [4](#0-3) 

**Reachable failure path:**

1. Account A (nonce=0 on-chain, pre-funded) submits `deploy_account(nonce=0)` + `invoke(nonce=1)` together.
2. The gateway accepts both. The invoke skips `__validate__` because `deploy_account` is in the pool. The invoke is inserted into the pool but **not** the priority queue (nonce 1 ≠ account nonce 0).
3. The batcher calls `get_txs`, receives only `deploy_account(nonce=0)` (the only queued tx), and executes it. The constructor fails.
4. `commit_block` is called with `rejected_tx_hashes = {deploy_account_hash}` and an empty `address_to_nonce`.

Inside `commit_block`:

```rust
let addresses_to_rewind = self.state.commit(address_to_nonce.clone());
let rewound_tx_hashes =
    self.rewind_txs(addresses_to_rewind, &address_to_nonce, &rejected_tx_hashes);
let mut account_nonce_updates =
    self.remove_rejected_txs(rejected_tx_hashes, &rewound_tx_hashes);
``` [5](#0-4) 

`remove_rejected_txs` removes `deploy_account` from the pool. `MempoolState::commit` clears `staged` and does **not** add account A to `committed` (it was not in `address_to_nonce`). [6](#0-5) 

After `commit_block`:
- On-chain nonce for A: **0**
- `state.contains_account(A)` = **false** (not in staged or committed)
- Pool: **invoke(nonce=1)** still present (was never staged, never rejected)
- `tx_pool.contains_account(A)` = **true**
- `account_tx_in_pool_or_recent_block(A)` = **true**

5. An attacker now submits a new `invoke(nonce=1)` for account A with an **arbitrary/invalid signature** and a higher tip (fee escalation).
6. `validate_nonce` passes: `0 ≤ 1 ≤ max_allowed_nonce_gap`. [7](#0-6) 

7. `skip_stateful_validations` returns **true** — `account_tx_in_pool_or_recent_block` is true because the original invoke is still in the pool.
8. `run_validate_entry_point` sets `execution_flags.validate = false`, so `__validate__` is **never called** and the signature is **never verified**. [8](#0-7) 

9. The attacker's invoke replaces the user's invoke via fee escalation and is admitted to the mempool.

The broken invariant is identical in structure to the AeraVault bug: a function that should only be callable in state X (`deploy_account` pending) does not check for state X, allowing an unauthorized actor to trigger the privileged path (skip signature verification) when the precondition no longer holds.

---

### Impact Explanation

The gateway admits an invoke transaction from an undeployed account without verifying the signature. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete harm:
- An attacker can replace a legitimate user invoke with an arbitrary-calldata/invalid-signature invoke via fee escalation.
- When the user later successfully deploys their account (nonce becomes 1), the attacker's invalid invoke is the first transaction executed. `__validate__` fails, the transaction is rejected, and the account pays gas for the failed validation.
- The user's original invoke is permanently gone from the pool.

---

### Likelihood Explanation

- A `deploy_account` constructor or `__validate_deploy__` failure is a realistic event (buggy constructor, insufficient gas, wrong class hash).
- The attacker only needs to observe the mempool for a failed `deploy_account` and the surviving invoke(nonce=1).
- Fee escalation is a standard mempool feature; the attacker pays a marginally higher tip to replace the victim's transaction.
- No privileged access is required; the attack is fully unprivileged.

---

### Recommendation

`skip_stateful_validations` must verify that the account has a **`deploy_account` transaction specifically** in the pool, not just any transaction. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)`, or the gateway should inspect the transaction type of the pool entry before deciding to skip validation.

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // Must check for a deploy_account specifically, not any transaction.
            return mempool_client
                .has_pending_deploy_account(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

### Proof of Concept

```
Step 1: Account A (pre-funded, nonce=0 on-chain) submits:
        - deploy_account(nonce=0, class_hash=C, constructor_calldata=[...fails...])
        - invoke(nonce=1, calldata=<user_calldata>, sig=<valid_sig>)

Step 2: Gateway accepts both.
        - deploy_account → pool + queue
        - invoke(nonce=1) → pool only (nonce 1 ≠ account nonce 0, not queued)
          skip_stateful_validations = true (deploy_account in pool)

Step 3: Batcher gets deploy_account from queue, executes it → constructor fails.
        commit_block(address_to_nonce={}, rejected={deploy_account_hash})

Step 4: Post-commit state:
        - deploy_account removed from pool (rejected)
        - invoke(nonce=1) still in pool (never staged, never rejected)
        - state.contains_account(A) = false
        - tx_pool.contains_account(A) = true
        - account_tx_in_pool_or_recent_block(A) = true

Step 5: Attacker submits:
        invoke(nonce=1, calldata=<attacker_calldata>, sig=<garbage>, tip=higher)

Step 6: Gateway validation:
        validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap → PASS
        validate_by_mempool: fee escalation → PASS (higher tip)
        skip_stateful_validations: nonce=1, account_nonce=0,
            account_tx_in_pool_or_recent_block(A)=true → returns true
        __validate__ NOT called, signature NOT verified → ADMITTED

Step 7: User retries deploy_account with fixed constructor → succeeds.
        Account A nonce becomes 1.
        Attacker's invoke(nonce=1) is now eligible for execution.
        Batcher executes it → __validate__ fails (invalid signature) → rejected.
        User's original invoke is gone; account paid gas for attacker's failed tx.
```
