### Title
`protocolUnpausePool` Sets Wrong Pause Level (1 Instead of 0), Leaving Pool Swap-Frozen After Protocol Unpause ÔÇö (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` is supposed to lift a protocol-level pause and restore the pool to active. Instead it writes `pauseLevel = 1` (admin-paused), so the pool remains swap-frozen after the owner believes it has been unpaused. Every swap continues to revert with `PoolPaused` until the pool admin separately calls `unpausePool`.

---

### Finding Description

The pool's `pauseLevel` encodes three states:

| Value | Meaning |
|---|---|
| 0 | Active ÔÇö swaps allowed |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`_checkNotPaused` rejects any non-zero level:

```solidity
// MetricOmmPool.sol
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

`swap` is gated by `whenNotPaused`, so level 1 is just as blocking as level 2.

The protocol-unpause path:

```solidity
// MetricOmmPoolFactory.sol  lines 399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ÔćÉ writes 1, not 0
}
```

The call `setPause(1)` leaves the pool in admin-paused state. The pool owner has no further lever ÔÇö only the pool admin can now call `unpausePool` (which transitions 1 Ôćĺ 0). If the admin is a slow multisig, a DAO, or an unresponsive key, the pool stays frozen indefinitely.

Compare with the correct admin-unpause path, which writes 0:

```solidity
// MetricOmmPoolFactory.sol  lines 467-471
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // ÔćÉ correct: writes 0
}
```

The wrong value written by `protocolUnpausePool` is the direct analog of the external report's `allStakesAreMigrated` being left at `false` after initialization: a gating state variable is set to the wrong value, causing the guarded core function (`swap`) to remain permanently blocked from the perspective of the actor who performed the "fix."

---

### Impact Explanation

After the factory owner calls `protocolUnpausePool`, every `swap` call on the pool reverts with `PoolPaused`. The pool is effectively still frozen. LPs cannot trade against the pool, arbitrageurs cannot rebalance it, and any protocol or user relying on the pool for settlement is broken. The owner has no direct path to fix this ÔÇö they must coordinate with the pool admin, who may be a separate entity or slow-moving governance.

---

### Likelihood Explanation

This triggers every single time `protocolUnpausePool` is called. There is no code path through that function that produces the correct outcome (level 0). The bug is unconditional.

---

### Recommendation

Change the target pause level in `protocolUnpausePool` from `1` to `0`:

```solidity
// MetricOmmPoolFactory.sol
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);   // target is 0
    IMetricOmmPoolFactoryActions(pool).setPause(0);         // restore active
}
```

If the design intent is a two-step handoff (protocol Ôćĺ admin Ôćĺ active), the function must be renamed and documented accordingly, and a separate `protocolFullyUnpausePool` that writes 0 should be provided.

---

### Proof of Concept

1. Pool is active (`pauseLevel == 0`).
2. Factory owner calls `protocolPausePool(pool)` Ôćĺ `pauseLevel` becomes `2`.
3. Factory owner resolves the emergency and calls `protocolUnpausePool(pool)`.
4. `setPause(1)` is executed Ôćĺ `pauseLevel` becomes `1`.
5. Any user calls `pool.swap(...)` Ôćĺ `_checkNotPaused()` fires Ôćĺ `revert PoolPaused()`.
6. The factory owner has no further action available; only the pool admin can call `unpausePool` to reach level 0.
7. If the pool admin is a multisig that is slow or unresponsive, the pool remains swap-frozen indefinitely despite the owner believing it was unpaused.
