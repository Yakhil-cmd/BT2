### Title
Malicious Participant Can Abort ECDSA Triple Generation via Crafted dlogeq Proof Causing Identity-Element Error - (`src/crypto/proofs/dlogeq.rs`, `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

A malicious participant in the ECDSA triple generation protocol can craft a `dlogeq` proof `(e, s)` such that the verifier's reconstructed commitment `big_k0 = s·G − public0·e` evaluates to the group identity. Because `encode_two_points` calls `C::Group::serialize` on the result and maps any serialization failure to `Err(ProtocolError::IdentityElement)`, the verifier returns an `Err` rather than `Ok(false)`. The `?` operator at the call site propagates this error, causing every honest party to abort the protocol permanently.

### Finding Description

**Root cause in `dlogeq::verify`:**

In `src/crypto/proofs/dlogeq.rs`, the verifier reconstructs the commitment points and immediately tries to serialize them:

```rust
let (phi0, phi1) = statement.phi(&proof.s.0);
let big_k0 = phi0 - *statement.public0 * proof.e.0;
let big_k1 = phi1 - *statement.public1 * proof.e.0;

let enc = encode_two_points::<C>(&big_k0, &big_k1)?;   // ← aborts on identity
``` [1](#0-0) 

`encode_two_points` maps any serialization failure (which includes the identity element) to `Err(ProtocolError::IdentityElement)`:

```rust
let mut ser1 = C::Group::serialize(point_1)
    .map_err(|_| ProtocolError::IdentityElement)?;
``` [2](#0-1) 

The only identity-element guard in `verify` checks `generator1`, not the reconstructed commitment points: [3](#0-2) 

**How the attacker controls `public0`:**

In `generation.rs`, the `dlogeq` statement is constructed as:

```rust
let statement = dlogeq::Statement::<C> {
    public0: &big_e_j_zero.index(from)?.value(),   // attacker's own E_j(0)
    generator1: &big_f.eval_at_zero()?.value(),
    public1: &big_c_j,
};
``` [4](#0-3) 

`public0` is the attacker's own `big_e` polynomial commitment evaluated at zero, stored from the attacker's round-2 message: [5](#0-4) 

Before being stored, a `dlog` proof is verified for this value, meaning the attacker must know its discrete log `x` — but since this is the attacker's own polynomial, they trivially know `x = e_attacker(0)`. [6](#0-5) 

**The crafted proof:**

The attacker sets `proof.s = x · proof.e` for any chosen non-zero scalar `proof.e`. Then:

```
big_k0 = proof.s·G − public0·proof.e
       = (x·proof.e)·G − (x·G)·proof.e
       = x·proof.e·G − x·proof.e·G
       = identity
```

`encode_two_points` immediately returns `Err(ProtocolError::IdentityElement)`.

**Error propagation to the call site:**

```rust
if !dlogeq::verify(
    &mut transcript.fork(b"dlogeq0", &from.bytes()),
    statement,
    their_phi_proof,
)?  {   // ← ? propagates Err, not just Ok(false)
``` [7](#0-6) 

The `?` operator propagates the `Err` directly, bypassing the `AssertionFailed` branch entirely. Every honest party running `do_generation_many` aborts.

### Impact Explanation

Triple generation (`generate_triple` / `generate_triple_many`) is the prerequisite for OT-based ECDSA signing. A single malicious participant can abort every honest party's protocol instance on every run, permanently preventing ECDSA signing for the group. This matches the **High: Permanent denial of signing** impact category.

### Likelihood Explanation

The attack requires only that the malicious party be a legitimate protocol participant (no key leakage, no cryptographic break). The crafted proof is trivial to compute given knowledge of one's own polynomial constant term. The attack is repeatable on every protocol invocation.

### Recommendation

In `dlogeq::verify`, add an explicit identity check on the reconstructed commitment points before calling `encode_two_points`, and return `Ok(false)` (invalid proof) rather than `Err`:

```rust
if big_k0.ct_eq(&C::Group::identity()).into()
    || big_k1.ct_eq(&C::Group::identity()).into()
{
    return Ok(false);
}
```

Alternatively, `encode_two_points` should be changed to handle the identity element gracefully (e.g., using a fixed encoding for the identity) so that a malformed proof results in a challenge mismatch (`Ok(false)`) rather than an error propagation.

### Proof of Concept

```rust
// Attacker knows x = e_attacker(0), the dlog of public0 = x*G
// Choose any non-zero e_attack
let e_attack = Scalar::from(42u64);
let s_attack = x * e_attack;  // ensures s*G - public0*e = identity

let crafted_proof = dlogeq::Proof { e: e_attack, s: s_attack };

// Verifier computes:
//   big_k0 = s_attack*G - public0*e_attack
//          = (x*e_attack)*G - (x*G)*e_attack = identity
// encode_two_points(identity, ...) -> Err(ProtocolError::IdentityElement)
// ? propagates -> honest party aborts do_generation_many
```
