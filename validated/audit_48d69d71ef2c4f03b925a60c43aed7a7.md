### Title
Stakers Permanently Lose Epoch Rewards When Staking Contract Is Paused During Their Attestation Window - (File: `src/attestation/attestation.cairo`)

---

### Summary

When the staking contract is paused, stakers cannot complete attestation for the current epoch. Because the attestation window is strictly enforced by block numbers with no grace period, any staker whose window expires during the pause permanently loses that epoch's rewards upon unpause. This is a direct analog to the reported "borrowers immediately liquidated once repayment resumes" pattern: the pause prevents a time-sensitive action, and when the pause lifts, the window has already closed with no recovery path.

---

### Finding Description

The `attest()` function in `src/attestation/attestation.cairo` calls `update_rewards_from_attestation_contract()` on the staking contract as its final step:

```
staking_dispatcher.update_rewards_from_attestation_contract(
    staker_address: staking_attestation_info.staker_address(),
);
```

`update_rewards_from_attestation_contract()` calls `general_prerequisites()` at the top of its body, which calls `assert_is_unpaused()`:

```cairo
fn assert_is_unpaused(self: @ContractState) {
    assert!(!self.is_paused(), "{}", Error::CONTRACT_IS_PAUSED);
}
```

Because the entire `attest()` call is a single transaction, the revert from the pause check rolls back everything, including the `_mark_attestation_is_done` write. The staker's attestation is not recorded.

The attestation window is strictly enforced by `_assert_attest_in_window()`:

```cairo
fn _assert_attest_in_window(self: @ContractState, target_attestation_block: BlockNumber) {
    let attestation_window = self.attestation_window.read();
    let current_block_number = get_block_number();
    let min_block = target_attestation_block + MIN_ATTESTATION_WINDOW.into();
    let max_block = target_attestation_block + attestation_window.into();
    assert!(
        min_block <= current_block_number && current_block_number <= max_block,
        "{}",
        Error::ATTEST_OUT_OF_WINDOW,
    );
}
```

The valid window is `[target_attestation_block + 11, target_attestation_block + attestation_window]`. Once `current_block_number > max_block`, the window is permanently closed for that epoch. There is no mechanism to extend the window, retroactively attest, or compensate stakers for epochs missed due to a pause.

---

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

Every staker whose per-epoch attestation window falls entirely within a pause period permanently loses that epoch's staking rewards. The rewards are never minted or credited; they are simply never generated. This is not a temporary delay — the epoch passes, the window closes, and the reward is gone forever. For a protocol with many stakers and a pause lasting even a single epoch, the aggregate lost yield can be substantial.

---

### Likelihood Explanation

**Likelihood: Medium.**

The security agent can pause the staking contract at any time for legitimate security reasons (e.g., a discovered exploit, an upgrade, an emergency). Epochs are finite-length windows measured in blocks. A pause of even moderate duration (covering one epoch's worth of blocks) will cause every active staker to miss their attestation window for that epoch. The attestation window per staker is a small fraction of the epoch length (bounded by `attestation_window` blocks), so a pause lasting longer than `attestation_window` blocks guarantees that every staker misses their window. This is a realistic scenario during any security incident requiring a pause.

---

### Recommendation

After the staking contract is unpaused, introduce a grace period during which stakers who missed their attestation window due to the pause can still attest. Concretely:

1. Record the block number at which the contract was unpaused.
2. In `_assert_attest_in_window()`, extend `max_block` by the duration of the pause (or a fixed grace period) if the staker's window fell during the pause.
3. Alternatively, track the total blocks the contract was paused per epoch and add that offset to each staker's valid window bounds.

---

### Proof of Concept

**Setup:** Staking contract is deployed. Staker S stakes and advances K epochs so their balance is active. Their `target_attestation_block` for epoch E falls at block X. Their valid window is `[X+11, X+20]` (assuming `attestation_window = 20`).

**Attack sequence:**

1. At block `X+10` (just before the window opens), the security agent calls `pause()` on the staking contract.
2. Blocks advance. The window `[X+11, X+20]` passes entirely while the contract is paused.
3. Staker S calls `attest(block_hash)` during blocks `X+11` to `X+20`. Each call reaches `update_rewards_from_attestation_contract()` which reverts with `CONTRACT_IS_PAUSED`. The transaction rolls back; `staker_last_attested_epoch` is not updated.
4. At block `X+21` (after the window closes), the security admin calls `unpause()`.
5. Staker S attempts `attest(block_hash)`. `_assert_attest_in_window()` checks `current_block_number <= X+20` — this is now false. The call reverts with `ATTEST_OUT_OF_WINDOW`.
6. Epoch E ends. `update_rewards_from_attestation_contract()` is never successfully called for staker S in epoch E. Staker S receives zero rewards for epoch E, permanently.
