### Title
Unbounded Loop Over Pool Member Balance Trace in `calculate_rewards` Can Permanently Freeze Delegator Rewards - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `Pool` iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because a delegator can accumulate an unbounded number of trace entries by repeatedly calling `add_to_delegation_pool` across different epochs without claiming, the loop can grow large enough to exceed Starknet's per-transaction gas limit, permanently preventing the delegator from claiming their accrued rewards.

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function contains a `while` loop that iterates over all entries in `pool_member_epoch_balance` from `entry_to_claim_from` to the current trace length:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The developers themselves acknowledge the risk in the comment at line 858. Each call to `add_to_delegation_pool` (or `exit_delegation_pool_intent` for partial amounts) in a new epoch appends one entry to the `pool_member_epoch_balance` trace. The `entry_to_claim_from` cursor stored in `pool_member_info` is only advanced when `claim_rewards` succeeds:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [2](#0-1) 

If a delegator accumulates many balance-change entries without claiming, the next `claim_rewards` call must iterate over all of them. Each iteration performs multiple storage reads via `pool_member_trace.at(...)` and `find_sigma(...)`, making the per-iteration gas cost non-trivial. [3](#0-2) 

The `pool_member_epoch_balance` storage map is per-member and grows monotonically: [4](#0-3) 

### Impact Explanation

A delegator who accumulates enough balance-change entries without claiming will find that their `claim_rewards` transaction exceeds the Starknet per-transaction gas limit and reverts. Because `entry_to_claim_from` is only updated on a successful claim, the delegator is permanently stuck: every future `claim_rewards` attempt starts from the same `entry_to_claim_from` and iterates the same (or larger) set of entries, always reverting. This permanently freezes the delegator's unclaimed yield.

This matches the allowed impact: **Permanent freezing of unclaimed yield** (High) and **Unbounded gas consumption** (Medium).

### Likelihood Explanation

The attack is self-inflicted or can be griefed:

- **Self-inflicted**: A delegator who calls `add_to_delegation_pool` many times across different epochs (e.g., dollar-cost averaging their stake) without regularly claiming rewards will naturally accumulate a large trace. There is no protocol-enforced upper bound on the number of balance changes.
- **Griefed**: Any unprivileged address can call `claim_rewards(pool_member)` on behalf of a victim (the function only requires the caller to be the pool member or reward address — but the victim themselves can be forced into this state by their own legitimate usage pattern).

The entry path is fully permissionless: `Pool::claim_rewards` → `calculate_rewards` → unbounded `while` loop. [5](#0-4) 

### Recommendation

1. **Paginated claiming**: Allow `claim_rewards` to accept a `max_iterations` parameter and process only that many balance-change entries per call, advancing `entry_to_claim_from` incrementally. Callers can invoke the function multiple times until all entries are processed.
2. **Pruning on claim**: After processing entries up to `until_epoch`, remove (or mark as consumed) the processed entries from `pool_member_epoch_balance` so the trace does not grow without bound.
3. **Enforce a maximum balance-change frequency**: Reject `add_to_delegation_pool` calls that would add a new epoch entry if the trace already exceeds a configurable cap, forcing the delegator to claim first.

### Proof of Concept

1. Staker stakes and enables a pool.
2. Delegator calls `enter_delegation_pool` with a small amount.
3. Delegator calls `add_to_delegation_pool` with a small amount in each of N consecutive epochs (e.g., N = 10,000), never calling `claim_rewards`.
4. Each call appends one entry to `pool_member_epoch_balance`; `entry_to_claim_from` remains 0.
5. Delegator calls `claim_rewards`. The `calculate_rewards` loop iterates N times, each time reading `pool_member_trace.at(i)` and calling `find_sigma` (which reads from `cumulative_rewards_trace`).
6. At sufficiently large N, the transaction exceeds the Starknet gas limit and reverts.
7. All subsequent `claim_rewards` calls also revert for the same reason. The delegator's accrued rewards are permanently frozen. [6](#0-5)

```text
        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
