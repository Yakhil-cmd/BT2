### Title
EVM Balance Permanently Locked on `CadenceOwnedAccount` Destruction Without Prior Withdrawal - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `EVM.CadenceOwnedAccount` resource in `fvm/evm/stdlib/contract.cdc` has no `destroy` function that enforces a zero balance or transfers the EVM balance before the Cadence resource is destroyed. When a COA is destroyed while holding a non-zero EVM balance, those FLOW tokens are permanently locked in the EVM state with no recovery path, because the EVM address is exclusively controlled by the now-destroyed Cadence resource and is not key-controlled.

---

### Finding Description

`EVM.CadenceOwnedAccount` is a Cadence resource that owns and controls an EVM address. Its EVM balance is tracked in the EVM state (not as a Cadence `@FlowToken.Vault`), so Cadence's normal resource-safety guarantees — which prevent `@` resources from being silently dropped — do not protect the EVM-side balance.

The resource definition at line 518 of `fvm/evm/stdlib/contract.cdc` contains no `destroy` function: [1](#0-0) 

The only way to recover EVM balance to Cadence is `withdraw`, which requires the `Owner | Withdraw` entitlement and explicitly panics for amounts below 1e10 attoFlow: [2](#0-1) 

The documentation for `withdraw` acknowledges the sub-1e10-attoFlow floor but does not warn that destroying the COA with any remaining balance permanently locks those tokens: [3](#0-2) 

COAs are explicitly documented as not key-controlled — the Cadence resource is the sole controller of the EVM address: [4](#0-3) 

Once `destroy cadenceOwnedAccount` executes, the EVM address still holds its balance in the EVM state, but no Cadence resource and no private key can ever issue a transaction from that address again. The balance is permanently inaccessible.

Test code confirms this pattern is exercised without any balance-drain guard: [5](#0-4) 

---

### Impact Explanation

Any FLOW tokens held in the EVM balance of a `CadenceOwnedAccount` at the time of its destruction are permanently locked in the EVM state. Because the EVM address is derived from the COA's UUID and is not key-controlled, no future transaction can spend or recover those tokens. This is a cross-VM asset loss: tokens exist in the EVM ledger but are forever inaccessible from both Cadence and EVM sides.

Two concrete sub-cases:

1. **Full balance destruction**: A user destroys their COA without calling `withdraw` first. The entire EVM balance is locked.
2. **Sub-precision remainder**: A user calls `withdraw` for the maximum UFix64-representable amount, leaving a remainder of up to 9,999,999,999 attoFlow (< 1e10). That remainder cannot be withdrawn (the call panics) and is locked when the COA is destroyed. [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged user who owns a `CadenceOwnedAccount` can trigger this by issuing a transaction that calls `destroy` on their COA resource. The entry path is a standard Cadence transaction requiring only the account's own authorization. No special privileges, leaked keys, or node compromise are needed. The sub-precision remainder case is structurally guaranteed to occur whenever a COA receives a deposit that is not a multiple of 1e10 attoFlow (e.g., from EVM-side gas refunds or direct EVM transfers of arbitrary attoFlow amounts), making the remainder-locking path reachable without any user error.

---

### Recommendation

Add a `destroy` function to `EVM.CadenceOwnedAccount` that either:

1. **Panics if balance > 0**, forcing callers to drain the EVM balance before destruction (analogous to Cadence's own resource-safety model), or
2. **Automatically transfers the remaining EVM balance** to a designated Cadence vault or burn address before the resource is destroyed.

Additionally, document explicitly that destroying a COA with a non-zero EVM balance permanently locks those tokens, and that sub-1e10-attoFlow remainders cannot be recovered via `withdraw`.

---

### Proof of Concept

```cadence
import EVM from <EVM_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>
import FungibleToken from <FUNGIBLE_TOKEN_ADDRESS>

transaction {
    prepare(account: auth(BorrowValue, Storage) &Account) {
        // 1. Create a COA and fund it with 1.0 FLOW
        let coa <- EVM.createCadenceOwnedAccount()
        let vault <- account.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(from: /storage/flowTokenVault)!
            .withdraw(amount: 1.0) as! @FlowToken.Vault
        coa.deposit(from: <-vault)

        // 2. Destroy the COA without withdrawing the EVM balance.
        //    The 1.0 FLOW (1e18 attoFlow) is now permanently locked at
        //    the COA's EVM address with no key and no Cadence resource
        //    to ever recover it.
        destroy coa

        // 3. No panic, no error — the transaction succeeds.
        //    The EVM state still records 1e18 attoFlow at the former COA address,
        //    but it is inaccessible forever.
    }
}
```

        /// Withdraws the balance from the cadence owned account's balance.
        /// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
        /// given that Flow Token Vaults use UFix64 to store balances.
        /// In other words, the smallest withdrawable amount is 1e10 attoFlow.
        /// Amounts smaller than 1e10 attoFlow, will cause the function to panic
        /// with: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".
        /// If the given balance conversion to UFix64 results in rounding loss,
        /// the withdrawal amount will be truncated to the maximum precision for UFix64.
        ///
        /// @param balance: The EVM balance to withdraw
        ///
        /// @return A FlowToken Vault with the requested balance
        access(Owner | Withdraw)
        fun withdraw(balance: Balance): @FlowToken.Vault {
            pre {
                !EVM.isPaused(): "EVM operations are temporarily paused"
            }

            if balance.isZero() {
                return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
            }
            let vault <- InternalEVM.withdraw(
                from: self.addressBytes,
                amount: balance.attoflow
            ) as! @FlowToken.Vault
            emit FLOWTokensWithdrawn(
                address: self.address().toString(),
                amount: balance.inFLOW(),
                withdrawnUUID: vault.uuid,
                balanceAfterInAttoFlow: self.balance().attoflow
            )
            return <-vault
        }
