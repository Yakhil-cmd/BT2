### Title
Unchecked Caller Authorization in Bank Precompile `transfer` Allows Unauthorized Native Token Transfers - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The bank precompile's `transfer` method accepts an arbitrary `sender` address from ABI-decoded input and passes it directly to `bankKeeper.SendCoins` without verifying that the EVM caller controls that address. Any EVM contract can drain `evm/<contractAddress>`-denominated native bank tokens from any holder to any recipient.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case of `BankContract.Run` unpacks three arguments from the call input: `sender`, `recipient`, and `amount`. The denom is correctly scoped to the calling contract via `EVMDenom(contract.Caller())`, but the `from` address for the actual `SendCoins` call is taken directly from `args[0]` — a fully attacker-controlled value — with no check that `contract.Caller() == sender`: [1](#0-0) 

```go
sender := args[0].(common.Address)   // caller-supplied, never validated
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())  // only denom is caller-scoped
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The same pattern exists for `BurnMethodName`: `args[0]` is used as the address to burn from, again without verifying the caller controls that address. [2](#0-1) 

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

Any EVM contract at address `C` can call `bankPrecompile.transfer(victim, attacker, amount)`. The bank precompile will execute `SendCoins(victim → attacker, "evm/C", amount)` without any consent from `victim`. This allows:

1. A malicious contract to issue `evm/<C>` tokens to users (e.g., as DeFi receipts or wrapped assets), then call `transfer(user, attacker, balance)` to drain every holder.
2. Any contract with a re-entrancy or logic flaw that reaches the bank precompile to have all its token holders drained by an attacker who triggers that path.

The `burn` variant allows any contract to destroy `evm/<C>` tokens held by any address, which is a High-severity accounting corruption.

### Likelihood Explanation

The bank precompile is reachable by any unprivileged EVM transaction. No special permissions, keys, or governance actions are required. The attacker only needs to deploy a contract and call the precompile at address `0x0000...0064`. [3](#0-2) 

### Recommendation

Replace the three-argument `transfer(sender, recipient, amount)` signature with a two-argument form `transfer(recipient, amount)` where the `from` address is implicitly `contract.Caller()`:

```go
// Correct: from is always the calling contract
from := sdk.AccAddress(contract.Caller().Bytes())
```

If delegated transfers are required, implement an on-chain allowance mapping (analogous to ERC-20 `approve`/`transferFrom`) so that `sender` can only be used when an explicit approval exists.

Apply the same fix to `BurnMethodName`: burn should only be permitted from `contract.Caller()`'s own balance, not from an arbitrary `args[0]` address.

### Proof of Concept

1. Attacker deploys contract `M` on Cronos EVM.
2. Legitimate users interact with `M` and receive `evm/M` tokens (e.g., as staking receipts). Their balances are recorded in the Cosmos bank module under denom `evm/<M>`.
3. Attacker calls the bank precompile (`0x0000...0064`) from contract `M` with calldata encoding `transfer(victim, attacker, victimBalance)`.
4. Inside `BankContract.Run`, `sender = victim`, `denom = "evm/M"`, and `bankKeeper.SendCoins(victim, attacker, coins)` executes unconditionally.
5. All `evm/M` tokens are transferred from `victim` to `attacker` with no authorization check. [4](#0-3)
