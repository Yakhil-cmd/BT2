### Title
Unbounded `pool_member_epoch_balance` Trace Growth Causes Permanent Freezing of Unclaimed Yield - (`src/pool/pool.cairo`)

### Summary

The `calculate_rewards` function in `pool.cairo` iterates over a pool member's entire unclaimed balance-change trace in an unbounded loop. Because every balance-change operation (`add_to_delegation_pool`, `exit_delegation_pool_intent`, `enter_delegation_pool`) appends a new checkpoint to the per-member `pool_member_epoch_balance` trace whenever it is called in a new epoch, a pool member who accumulates many balance changes without claiming rewards will eventually cause `claim_rewards` (and `pool_member_info_v1`) to exceed Starknet's per-transaction gas limit, permanently freezing their unclaimed yield.

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:** [1](#0-0) 

The code itself acknowledges the risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

**How the trace grows:**

Every call to `set_member_balance` or `increase_member_balance` delegates to `PoolMemberBalanceTrace::insert` with key `current_epoch + K`. [2](#0-1) 

The `insert` implementation appends a **new** checkpoint whenever the new key differs from the last stored key — i.e., whenever the call happens in a different epoch than the previous balance change. [3](#0-2) 

**Attacker-controlled entry paths that grow the trace:**

1. `add_to_delegation_pool` — callable by the pool member or their reward address, requires only `amount > 0`.
2. `exit_delegation_pool_intent` — callable only by the pool member; calling with `amount = 0` cancels any pending intent but still calls `set_member_balance` with the unchanged balance, appending a new checkpoint. [4](#0-3) 

Each call in a distinct epoch appends one new entry. The trace is **never pruned or bounded**.

**Why `entry_to_claim_from` does not fully mitigate this:**

`entry_to_claim_from` is stored in `InternalPoolMemberInfoV1` and advanced only inside `claim_rewards`. [5](#0-4) 

A pool member who makes N balance changes across N epochs without calling `claim_rewards` accumulates N unclaimed trace entries. The next `claim_rewards` call must iterate all N entries in a single transaction. Once N is large enough to exceed the Starknet gas limit, `claim_rewards` reverts permanently.

**`pool_member_info_v1` is also affected:** [6](#0-5) 

This view function calls `calculate_rewards` with the same unbounded loop, so it too becomes uncallable once the trace is large enough.

### Impact Explanation

Once the trace is large enough to exceed the per-transaction gas limit, `claim_rewards` reverts on every call. The pool member's accumulated STRK rewards are permanently locked in the pool contract with no recovery path — there is no administrative function to prune the trace or force-pay rewards. This constitutes **permanent freezing of unclaimed yield**, which is an explicitly allowed High impact in the bounty scope.

### Likelihood Explanation

The trace grows by at most one entry per epoch. For a human operator who claims rewards regularly, the loop stays short. However:

- A **smart-contract delegator** (e.g., a vault or auto-compounder) that rebalances delegation every epoch without claiming rewards will accumulate entries at the maximum rate.
- Even a human delegator who is inactive for a long period while having made prior balance changes will accumulate entries.
- The minimum cost per epoch is a single `add_to_delegation_pool` transaction with 1 token unit — the token is not lost, only temporarily locked.
- The code comment explicitly acknowledges the loop is unbounded, confirming the developers are aware of the risk but have not bounded it.

Likelihood is **medium**: it requires sustained operation over many epochs, but is a realistic scenario for any automated on-chain integrator.

### Recommendation

1. **Bound the loop**: Introduce a maximum number of iterations per `claim_rewards` call (e.g., process at most `MAX_ENTRIES_PER_CLAIM` checkpoints and store the updated `entry_to_claim_from` so the next call continues from where the last left off). This already partially exists via `entry_to_claim_from` — the missing piece is a per-call cap.
2. **Prune stale entries**: After `entry_to_claim_from` advances past old checkpoints, those entries are no longer needed. Consider compacting the trace by removing entries below `entry_to_claim_from`.
3. **Rate-limit balance changes**: Enforce that at most one new checkpoint can be created per epoch per pool member (the `insert` deduplication already does this for same-epoch calls, but does not prevent one-per-epoch accumulation over time).

### Proof of Concept

**Setup:** Pool member `A` enters a delegation pool with minimum stake.

**Attack loop (repeat for N epochs):**
```
epoch 1:  A calls add_to_delegation_pool(amount=1)  → trace length = 2
epoch 2:  A calls add_to_delegation_pool(amount=1)  → trace length = 3
epoch 3:  A calls add_to_delegation_pool(amount=1)  → trace length = 4
...
epoch N:  A calls add_to_delegation_pool(amount=1)  → trace length = N+1
```

A never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.

**Trigger:** After N epochs, A (or anyone) calls `claim_rewards(A)`. The `calculate_rewards` loop iterates from index 0 to N, reading a storage slot and calling `find_sigma` on each iteration. Once N exceeds the gas-per-iteration budget of the Starknet sequencer, the transaction reverts. All of A's accumulated STRK rewards are permanently inaccessible.

The same outcome is reached by calling `exit_delegation_pool_intent(0)` once per epoch (no tokens required beyond the initial stake), since that path also calls `set_member_balance` unconditionally. [7](#0-6) [8](#0-7) [9](#0-8)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
