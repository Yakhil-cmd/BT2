### Title
Malicious Non-Participant Aborts Echo Broadcast via Unguarded `?` on `participants.index(from)` in `Send` Phase - (File: src/protocol/echo_broadcast.rs)

### Summary
In `reliable_broadcast_receive_all`, the `MessageType::Send` arm uses `participants.index(from)?` to verify the sender belongs to the participant list. If `from` is not a known participant, `participants.index` returns `Err(ProtocolError::InvalidIndex)` and the `?` operator propagates that error, immediately aborting the entire echo-broadcast session. The `Echo` and `Ready` arms handle the same situation gracefully by calling `seen_echo.put(from)` / `seen_ready.put(from)`, which silently return `false` for unknown senders. The inconsistency means any party outside the participant list can permanently abort DKG, reshare, or refresh for all honest parties by injecting a single well-formed `Send` message.

### Finding Description
`reliable_broadcast_receive_all` runs a loop that receives `(sid, vote)` tuples. For `MessageType::Send`, the guard is:

```rust
if state_sid.finish_send || sid != participants.index(from)? {
    continue;
}
```

`participants.index(from)` returns `Err(ProtocolError::InvalidIndex)` when `from` is absent from the list. The `?` propagates that error out of the loop, out of `reliable_broadcast_receive_all`, out of `do_broadcast`, and ultimately out of `do_keyshare`, aborting the DKG/reshare/refresh protocol.

By contrast, the `Echo` and `Ready` arms call `ParticipantCounter::put`, which returns `false` for unknown participants and causes a `continue` — exactly the safe behaviour the `Send` arm should also exhibit.

The `MessageBuffer` in `internal.rs` accepts messages from any `Participant` without validating membership:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);
}
```

The channel tag for the shared channel (`ChannelTag::root_shared()`) is a deterministic public constant, and waitpoints are sequential counters starting at 0, so an attacker can craft a correctly-routed message without any secret knowledge.

### Impact Explanation
`do_broadcast` is called twice inside `do_keyshare` (once for session-ID exchange, once for commitment/proof broadcast) and once more in `broadcast_success`. A single injected `Send` message from an unknown participant aborts any of these calls, causing the entire DKG, reshare, or refresh to fail for every honest party. Because the attacker can repeat the injection on every retry, honest parties can be permanently denied key generation.

This matches: **High — Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs and documented trust assumptions.**
