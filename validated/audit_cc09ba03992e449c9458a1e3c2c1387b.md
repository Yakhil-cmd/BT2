### Title
`PresignOutput` Not Consumed After Rerandomization Enables Nonce Reuse and Secret Key Extraction - (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

### Summary

Both `ot_based_ecdsa::RerandomizedPresignOutput::rerandomize_presign` and `robust_ecdsa::RerandomizedPresignOutput::rerandomize_presign` accept the `PresignOutput` by shared reference (`&PresignOutput`) rather than by value. Because the presignature is not consumed, the caller retains the original `PresignOutput` after rerandomization and can call `rerandomize_presign` again with different `RerandomizationArguments` (different message hash, tweak, or entropy). Additionally, `PresignOutput` derives `Clone` in both modules, so even a by-value API could be trivially bypassed. The result is that a single presignature nonce can silently underlie multiple completed signing sessions, enabling classical ECDSA nonce-reuse secret-key recovery.

### Finding Description

`rerandomize_presign` in both ECDSA variants takes the presignature by reference:

```rust

    presignature: &PresignOutput,   // ← reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// src/ecdsa/robust_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) [2](#0-1) 

Both `PresignOutput` types also derive `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }
``` [3](#0-2) [4](#0-3) 

The documentation explicitly states that each presignature must be consumed exactly once and never reused:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [5](#0-4) 

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [6](#0-5) 

The library enforces this requirement only in documentation, not in the type system. The `sign` function for robust ECDSA does consume `RerandomizedPresignOutput` by value, but the upstream `rerandomize_presign` step — where the nonce material is first exposed — does not consume the `PresignOutput`. [7](#0-6) 

### Impact Explanation

ECDSA nonce reuse is a well-known catastrophic failure mode. If the same underlying nonce `k` (embedded in `PresignOutput.k` / `PresignOutput.c`) is used in two completed signing sessions producing signatures `(R, s1)` for hash `h1` and `(R, s2)` for hash `h2`, the secret key `x` is recoverable with simple algebra: `x = (s1·h2 - s2·h1) / (s2 - s1) · R_x^{-1}`. The robust ECDSA documentation explicitly acknowledges this: "the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [8](#0-7) 

This maps directly to **Critical: Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, or nonce material**.

### Likelihood Explanation

Any orchestration layer that manages a pool of presignatures and calls `rerandomize_presign` is exposed. A malicious coordinator can trigger two signing sessions with the same `PresignOutput` by simply calling `rerandomize_presign` twice before either signing session completes. A non-malicious but "lazy" caller might also clone or cache `PresignOutput` values for retry logic on network failure, inadvertently reusing the same nonce. The `Clone` derive makes this trivially easy. No cryptographic break or external compromise is required — only a single API call with the same `&PresignOutput`.

### Recommendation

1. **Change `rerandomize_presign` to consume `PresignOutput` by value** in both `ot_based_ecdsa` and `robust_ecdsa`:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // consumed, not a reference
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

2. **Remove `Clone` from `PresignOutput`** in both modules so that callers cannot duplicate the nonce material before passing it in. `ZeroizeOnDrop` is already derived, which is correct, but `Clone` undermines it.

3. For `robust_ecdsa::PresignOutput`, the same applies: remove `Clone` and take by value. [4](#0-3) 

### Proof of Concept

```
1. Orchestrator generates PresignOutput P for a set of participants.
2. Orchestrator calls rerandomize_presign(&P, args1) → R1  (for message h1)
3. Orchestrator calls rerandomize_presign(&P, args2) → R2  (for message h2, different entropy/tweak)
   -- P is still alive; Rust does not prevent this --
4. Both signing sessions complete, producing (R, s1) and (R, s2).
   (Both share the same underlying nonce k from P.)
5. Attacker observes both signatures and recovers x = (s1·h2 - s2·h1) / (s2 - s1) · Rx^{-1}.
```

The `Clone` derive makes step 2–3 even simpler: `let p2 = P.clone(); rerandomize_presign(&p2, args2)`. [9](#0-8)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
