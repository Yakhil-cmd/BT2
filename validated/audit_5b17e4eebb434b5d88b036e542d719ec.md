Audit Report

## Title
Unsanitized Memo Interpolation Enables Markdown Field Injection in ICRC-21 Consent Messages — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

`ConsentMessage::add_memo` interpolates raw, attacker-controlled UTF-8 memo bytes directly into a Markdown-formatted `GenericDisplayMessage` string without escaping backtick characters or newlines. A crafted memo can close the surrounding inline-code span and inject structurally identical Markdown paragraphs — including a fake `**To:**` field — that are visually indistinguishable from fields produced by `add_account`. This defeats the core security guarantee of ICRC-21: that the consent message faithfully represents the on-chain action to the user.

## Finding Description

`add_memo` in `responses.rs` at line 294 formats the memo as:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

`add_account` at line 90 uses the identical structural pattern:

```rust
message.push_str(&format!("\n\n**{name}:**\n`{account}`"))
```

`memo_str` is derived at lines 288–291 by attempting UTF-8 decoding of the raw memo bytes; if valid UTF-8, the string is used verbatim with no further sanitization. A memo value of `foo`\n\n**To:**\n`evil_address` (27 bytes, within the ICRC-1 32-byte limit) produces the following raw string appended to the message:

```
\n\n**Memo:**\n`foo`\n\n**To:**\n`evil_address`
```

When rendered as Markdown this is two separate paragraphs: `**Memo:** foo` and `**To:** evil_address`. The second is byte-for-byte identical to what `add_account("To", ...)` emits.

The build order in `lib.rs` is deterministic: `add_account("To", ...)` is called at line 206, and `add_memo` is called last at lines 290–292. On a hardware wallet that paginates `GenericDisplayMessage` screen-by-screen, the injected `**To:**` field appears on a later page near the confirmation button, while the real `To` field is on an earlier page the user has already scrolled past.

No escaping, sanitization, or allowlist validation of memo content exists anywhere in the pipeline before interpolation.

## Impact Explanation

A malicious dapp constructs a `TransferArg` with `to` set to the attacker's address and a crafted memo that injects a fake `**To:**` field showing a trusted or victim address. The ICRC-21 consent message — which is explicitly designed to give hardware wallet users a trustworthy view of what they are signing — displays the fabricated recipient. The user approves the transaction believing they are sending to the displayed address; the actual on-chain transfer goes to the attacker's address. This constitutes theft of user funds enabled by a broken ICRC-21 security guarantee, matching the allowed impact: **High ($2,000–$10,000) — significant ledger/ICRC security impact with concrete user funds harm**.

## Likelihood Explanation

The `icrc21_canister_call_consent_message` endpoint is fully unprivileged; any caller (dapp) can submit a `ConsentMessageRequest` with an arbitrary `arg` blob. The memo field in ICRC-1 `TransferArg` accepts arbitrary bytes up to 32 bytes; the injection payload fits within that limit. `GenericDisplayMessage` is explicitly designed for Markdown-capable wallets; hardware wallets implementing ICRC-21 are the primary consumers. The injected field is structurally identical to a real field with no visual indicator distinguishing it. The attack requires the user to interact with a malicious dapp, but ICRC-21 exists precisely to protect users in that scenario — this vulnerability nullifies that protection.

## Recommendation

Escape all Markdown special characters — at minimum backticks, asterisks, underscores, `#`, and newlines — in any user-controlled string before interpolating it into a `GenericDisplayMessage`. For memo content specifically, the safest approach is to hex-encode the bytes unconditionally (removing the UTF-8 display path entirely), or to apply a strict allowlist of printable ASCII characters that excludes all Markdown metacharacters before displaying as text. The `FieldsDisplayMessage` variant is not affected because it stores values in a structured `Vec<(String, Value)>` rather than a raw Markdown string.

## Proof of Concept

```rust
// memo bytes: foo`\n\n**To:**\n`evil_address (27 bytes, within ICRC-1 32-byte limit)
let memo_bytes = b"foo`\n\n**To:**\n`evil_address";
let memo = GenericMemo::Icrc1Memo(ByteBuf::from(memo_bytes.to_vec()));
let mut msg = ConsentMessage::GenericDisplayMessage(String::new());
msg.add_account("To", "real_attacker_address".to_string());
msg.add_memo(memo);

let rendered = match &msg {
    ConsentMessage::GenericDisplayMessage(s) => s.clone(),
    _ => panic!(),
};

// Fails: count is 2, not 1
assert!(
    rendered.matches("**To:**").count() == 1,
    "FAIL: injected To field found — rendered:\n{}", rendered
);
```
