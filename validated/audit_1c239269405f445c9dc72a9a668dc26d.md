### Title
Attacker Can Donate Tokens to Suppress L1 Mint Requests, Eventually Freezing Unclaimed Yield - (File: src/reward_supplier/reward_supplier.cairo)

### Summary

`RewardSupplier.request_funds` reads the contract's raw on-chain token balance via `balance_of(get_contract_address())` to decide whether to send a mint request to L1. Because any unprivileged address can transfer STRK tokens directly to the `RewardSupplier` contract (bypassing `on_receive`), an attacker can inflate the observed balance, suppress all L1 mint requests, and eventually cause `claim_rewards` to fail when the actual token balance is exhausted while `unclaimed_rewards` (internal accounting) continues to grow.

---

### Finding Description

`request_funds` is the internal function that decides whether to send an L1 mint message:

```cairo
// src/reward_supplier/reward_supplier.cairo  lines 301-331
fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())   // <-- raw on-chain balance
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    let credit = balance + l1_pending_requested_amount;
    let debit = unclaimed_rewards;

    let base_mint_amount = self.base_mint_amount.read();
    let threshold = compute_threshold(base_mint_amount);
    if credit < debit + threshold {          // <-- gate on raw balance
        // ... send L1 mint messages
    }
    self.l1_pending_requested_amount.write(l1_pending_requested_amount);
}
```

`request_funds` is called every time `update_unclaimed_rewards_from_staking_contract` is invoked (once per epoch):

```cairo
// lines 198-201
let unclaimed_rewards = self.unclaimed_rewards.read() + rewards;
self.unclaimed_rewards.write(unclaimed_rewards);
self.request_funds(:unclaimed_rewards);
```

The `on_receive` callback (the only legitimate deposit path) is restricted to the StarkGate bridge address and updates `l1_pending_requested_amount`. A plain ERC20 `transfer` to the contract address bypasses `on_receive` entirely, so `unclaimed_rewards` and `l1_pending_requested_amount` are **not** updated, but `balance_of` is.

An attacker calls `STRK.transfer(reward_supplier_address, D)` for a large `D`. From that point on, `credit = (B + D) + l1_pending_requested_amount` is artificially inflated. The condition `credit < debit + threshold` stays false for many epochs, so no L1 mint messages are sent.

Meanwhile, `unclaimed_rewards` keeps growing each epoch. When the staking contract calls `claim_rewards(amount)`:

```cairo
// lines 213-219
let unclaimed_rewards = self.unclaimed_rewards.read();
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);  // passes
self.unclaimed_rewards.write(unclaimed_rewards - amount);
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into()); // panics if balance < amount
```

The internal-accounting assert passes (because `unclaimed_rewards` is large), but `checked_transfer` panics when the actual token balance is insufficient. This freezes all reward distribution for every staker and delegator.

---

### Impact Explanation

- `claim_rewards` is called by the staking contract whenever a staker or delegator claims rewards.
- If `checked_transfer` panics, the entire reward-claim transaction reverts.
- All stakers and delegators are unable to receive their accrued yield.
- This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- The attack requires only a plain ERC20 `transfer` — no privileged role, no bridge access, no governance key.
- The attacker must spend capital (donated tokens), but those tokens are consumed paying out rewards, so the attacker loses them. The attack is therefore a griefing attack with a cost proportional to the total rewards accrued during the suppression window.
- The suppression is self-reinforcing: as long as the attacker keeps donating each epoch, L1 minting never resumes and the balance never recovers.
- A one-time large donation suppresses minting for a bounded window; repeated small donations make it permanent.

---

### Recommendation

Replace the raw `balance_of` call in `request_funds` with an internally tracked balance variable (e.g., `self.tracked_balance`), updated only through `on_receive` (increment) and `claim_rewards` (decrement). This mirrors the fix recommended in the original report: track the token balance internally and add sweep logic for any surplus tokens that arrive outside the normal deposit path.

---

### Proof of Concept

1. Deploy the system normally; `RewardSupplier` has balance `B` and `unclaimed_rewards = U`.
2. Attacker calls `STRK.transfer(reward_supplier_address, D)` where `D` is large enough that `B + D + l1_pending_requested_amount >= U + threshold`.
3. Each epoch, the staking contract calls `update_unclaimed_rewards_from_staking_contract(epoch_rewards)` → `request_funds` is called → `credit = (B + D - claimed_so_far) + l1_pending` still exceeds `debit + threshold` → **no L1 mint message sent**.
4. After enough epochs, `unclaimed_rewards` has grown to `U'` while the actual balance has shrunk to `B + D - claimed_so_far < U'`.
5. Staker calls `claim_rewards` on the staking contract → staking contract calls `RewardSupplier.claim_rewards(amount)` → `assert!(amount <= unclaimed_rewards)` passes → `checked_transfer` panics (insufficient balance) → **all reward claims revert permanently**.         fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
