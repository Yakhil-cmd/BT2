### Title
Permanent Loss of ERC-20 Tokens on Failed `ExitToNear` Bridge Exit When `error_refund` Feature Is Disabled ‚Äî (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens on the Aurora side and schedules a `ft_transfer` on the NEAR NEP-141 contract. When the compile-time `error_refund` feature is disabled (the non-default state), no refund callback is attached to the promise chain. If the NEAR-side transfer fails, the burned tokens are permanently unrecoverable ‚Äî they are destroyed on Aurora and never arrive on NEAR.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` constructs `ExitToNearPrecompileCallbackArgs` with the `refund` field unconditionally set to `None` when the `error_refund` feature is absent:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For ERC-20 exits (the common case), `transfer_near` is also `None`. This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, so the promise is emitted as a bare `PromiseArgs::Create` with **no callback attached**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The callback `exit_to_near_precompile_callback` ‚Äî which is the only place `refund_on_error` is invoked ‚Äî is never scheduled:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [3](#0-2) 

The `error_refund` feature is **not** in the `default` feature set of either `aurora-engine` or `aurora-engine-precompiles`: [4](#0-3) [5](#0-4) 

The `refund_on_error` function itself is unconditional ‚Äî it re-mints burned ERC-20 tokens or returns ETH from the precompile address ‚Äî but it is only reachable through the callback that is never scheduled when `error_refund` is off: [6](#0-5) 

---

### Impact Explanation

**Critical ‚Äî Permanent freezing of funds.**

When a user calls the `ExitToNear` precompile to bridge ERC-20 tokens to NEAR and the NEAR-side `ft_transfer` fails (e.g., recipient account not registered with the NEP-141 token, insufficient storage deposit), the ERC-20 tokens are already burned on the Aurora side. Without the refund callback, there is no mechanism to re-mint them. The tokens are permanently destroyed. This is confirmed by the integration test, which explicitly documents the loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

The same permanent loss applies to ETH exits via `withdrawEthToNear`: [8](#0-7) 

---

### Likelihood Explanation

**Medium.** The `ft_transfer` on the NEAR side fails whenever the recipient account is not registered with the NEP-141 token (a common user error), or when the NEP-141 contract rejects the transfer for any reason. Any EVM user who makes this mistake while the engine is compiled without `error_refund` permanently loses their tokens. The feature is not in the default build, making this the default behavior for any deployment that does not explicitly opt in.

---

### Recommendation

1. Add `error_refund` to the `default` features of `aurora-engine` and `aurora-engine-precompiles`, or make the refund mechanism unconditional (remove the feature gate entirely).
2. Alternatively, restructure the exit flow so that ERC-20 tokens are only burned **after** the NEAR-side transfer is confirmed successful, analogous to the external report's recommendation to allocate at action time rather than at a later privileged step.

---

### Proof of Concept

1. User holds 100 units of a bridged ERC-20 token on Aurora (backed by a NEP-141 on NEAR).
2. User submits an EVM transaction calling the `ExitToNear` precompile with flag `0x01` (ERC-20 exit), specifying a NEAR recipient account that is **not registered** with the NEP-141 contract.
3. The ERC-20 contract burns 100 tokens on the Aurora side (EVM state change is committed).
4. The precompile emits a `PromiseArgs::Create(ft_transfer_promise)` log ‚Äî no callback is attached because `error_refund` is disabled and `transfer_near` is `None`.
5. The NEAR runtime executes `ft_transfer`; it fails because the recipient is unregistered.
6. There is no callback to invoke `refund_on_error`; the 100 ERC-20 tokens are permanently gone.

Entry path: `submit` (EVM tx) ‚Üí `ExitToNear::run` ‚Üí `PromiseArgs::Create` (no callback) ‚Üí NEAR `ft_transfer` fails ‚Üí no recovery. [9](#0-8) [10](#0-9)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
