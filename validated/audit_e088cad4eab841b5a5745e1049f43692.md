### Title
Greedy Regex in `parseOfferData` Extracts Last Offer String Instead of First, Enabling Wrong-Offer Import — (`packages/gui/src/components/offers2/OfferBuilderImport.tsx`)

### Summary

The `parseOfferData` function uses a greedy `.*` for the `leading` capture group. When a file contains two valid bech32m offer strings, the regex engine backtracks to maximize `leading`, consuming the first offer string and leaving the second for the `offer` group. The second (attacker-controlled) offer is then passed to `getOfferSummary` and navigated to the offer viewer.

### Finding Description

In `parseOfferData` at line 30:

```ts
const matches = data.match(
  /(?<leading>.*)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
);
``` [1](#0-0) 

The `s` flag makes `.` match newlines. The `(?<leading>.*)` group is greedy and will consume as much as possible while still allowing the overall pattern to match. Given input `offer1AAAA...offer1BBBB...`:

1. `leading` greedily matches the entire string, then backtracks.
2. The engine finds the **rightmost** position where `offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+` can match — which is `offer1BBBB`.
3. Result: `leading = "...offer1AAAA..."`, `offer = "offer1BBBB"`, `trailing = ""`.

The extracted `offerData` is then passed directly to `getOfferSummary` and the offer viewer: [2](#0-1) 

There is no validation that the extracted offer is the first (or only) offer string in the file.

### Impact Explanation

An attacker who sends a crafted offer file containing two offer strings can cause the GUI to import and present the **second** offer for user acceptance, not the first. The user, believing they opened a specific offer, may accept a different one — transferring different assets, amounts, or to a different counterparty than intended. This falls under **High: Unsafe handling of external files that causes a user to approve the wrong asset/amount/destination**.

### Likelihood Explanation

Offer files are routinely exchanged between counterparties. A malicious counterparty can trivially prepend a decoy offer string before their actual (unfavorable) offer in the file. The user has no indication from the UI that the imported offer differs from the first string in the file. The user still sees the offer summary before accepting, which is a partial mitigation — but if the attacker crafts offers with similar-looking terms (e.g., same XCH amount but different CAT asset), the user may not notice the substitution.

### Recommendation

Change `(?<leading>.*)` to a non-greedy `(?<leading>.*?)` so the regex matches the **first** occurrence of the offer pattern:

```ts
/(?<leading>.*?)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
```

The fix:
```ts

console.log(matches.groups.offer);
// Outputs: "offer1qpzry9x8gf2tvdw0s3jn54khce6mua7lmqqqxw"  ← the FIRST offer, correct
```
