### Title
Single Malicious Participant Can Permanently Block DKG, Reshare, and Refresh by Withholding Broadcast Message - (File: src/protocol/echo_broadcast.rs)

### Summary
The `reliable_broadcast_receive_all` function requires ALL N participants to complete their broadcast session before returning. A single malicious participant that never sends their initial `Send` message causes the function to block indefinitely, permanently denying DKG, reshare, and refresh for all honest parties. There is no timeout or threshold-based early exit.

### Finding Description
`do_broadcast` (echo_broadcast.rs line 334) runs N simultaneous instances of the echo-broadcast protocol, one per participant. `reliable_broadcast_receive_all` (echo_broadcast.rs line 140) tracks a `BroadcastProtocolState` per session and exits only when every session has `finish_ready = true`:

```rust
// line 323
if state.iter().all(|x| x.finish_ready) {
    return Ok(vote_output);
}
```

`finish_ready` for session `sid` is set only when `> 2 * ready_t` ready votes accumulate for that session's value (line 295–300). If participant P never sends their `MessageType::Send` message:

1. No participant receives a `Send` from P → no one echoes P's value.
2. No echo votes → no ready votes → `state[P.index].finish_ready` is never set.
3. The loop blocks at `chan.recv(wait).await` (line 179) indefinitely.

The underlying `MessageBuffer::pop` (internal.rs line 245–255) has no timeout:

```rust
receiver_lock.next().await.expect("Reference to sender held")
```

The existing early-abort check (echo_broadcast.rs lines 241–263) only fires when an echo vote is actually received for a session. If P never sends `Send`, no echo is ever received for P's session, so the check is never evaluated and no error is returned.

`do_broadcast` is called in every critical DKG phase:
- `do_keyshare` session-ID broadcast (dkg.rs line 362)
- `do_keyshare` commitment + proof broadcast (dkg.rs line 435)
- `broadcast_success` final success vote (dkg.rs line 531)

A single withholding participant at any of these three points permanently stalls the entire DKG, reshare, or refresh run for all honest parties.

### Impact Explanation
This is a **High** impact finding: permanent denial of key generation, reshare, and refresh for honest parties. The protocol's echo-broadcast layer is parameterized with Byzantine fault thresholds (`echo_ready_thresholds`, line 67) implying tolerance for up to `t < n/3` malicious parties, yet the completion gate unconditionally requires all N sessions to finish. A single Byzantine participant exploiting this mismatch can prevent any honest quorum from ever completing DKG or reshare, rendering the key material permanently inaccessible.

### Likelihood Explanation
The attack requires no cryptographic capability, no key material, and no coordination. Any one of the N participants simply omits their `reliable_broadcast_send` call. Because the protocol provides no timeout and no threshold-based completion path, the hang is deterministic and immediate. The attacker can be a newly-joined participant with no prior trust.

### Recommendation
Two complementary mitigations:

1. **Threshold-based completion**: Replace the `state.iter().all(|x| x.finish_ready)` gate with a check that `n - missing_sessions <= t` (i.e., enough sessions have completed that the remaining absent parties are within the Byzantine tolerance). Sessions that never receive a `Send` after all other sessions complete should be treated as a detected malicious absence and the absent participant identified.

2. **Timeout / liveness deadline**: Introduce a per-session deadline after which an absent `Send` is treated as a protocol abort with the absent participant flagged as malicious, rather than blocking indefinitely.

### Proof of Concept

```
Setup: N=4 participants (P0, P1, P2, P3), threshold=2.
       Byzantine fault tolerance: t = (4-1)/3 = 1.

Step 1: All four participants enter do_keyshare → do_broadcast (session-ID round).

Step 2: P3 (attacker) calls reliable_broadcast_send for its own session
        but immediately drops the connection / never sends the MessageType::Send frame.

Step 3: P0, P1, P2 each send their Send messages, echo each other's values,
        and accumulate ready votes for sessions 0, 1, 2.
        state[0].finish_ready = true
        state[1].finish_ready = true
        state[2].finish_ready = true
        state[3].finish_ready = false  ← never set

Step 4: The loop at line 323 evaluates state.iter().all(|x| x.finish_ready) → false.
        chan.recv(wait).await blocks indefinitely.
        P0, P1, P2 are permanently stalled; DKG never completes.
```

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
