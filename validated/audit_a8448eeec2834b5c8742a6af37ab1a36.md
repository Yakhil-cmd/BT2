### Title
Cross-Address COA Ownership Proof Replay via Missing `evmAddress` Binding in `validateCOAOwnershipProof` — (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

`EVM.validateCOAOwnershipProof` performs two independent checks — it verifies signatures over `signedData` are valid for a Cadence account, and separately verifies the account's COA at a given path matches `evmAddress` — but never enforces that `signedData` encodes `evmAddress`. A signature produced to prove ownership of one COA (`COA_A`) can be replayed against a different COA (`COA_B`) owned by the same Cadence account, causing the function to return `isValid: true` for `COA_B` without the account holder ever authorizing it. The missing linkage is explicitly acknowledged in the source code comment but left unmitigated.

---

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, `validateCOAOwnershipProof` (lines 1011–1116) executes two structurally independent verification steps:

**Step 1 — Signature check** (lines 1082–1093): Verifies that the provided signatures over the caller-supplied `signedData` are cryptographically valid for the account at `address`.

**Step 2 — COA address check** (lines 1095–1110): Borrows the COA resource at `path` and verifies its EVM address matches the caller-supplied `evmAddress`.

The two steps share no binding: `signedData` is never required to encode `evmAddress`. The code comment at lines 1002–1009 explicitly acknowledges this gap:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

**Attacker-controlled entry path:**

Any unprivileged Cadence script or transaction can call `EVM.validateCOAOwnershipProof` with arbitrary parameters. The function is `access(all)` and imposes no caller restrictions. [2](#0-1) 

**Concrete replay scenario:**

1. Alice owns a Cadence account with two COAs: `COA_A` (at `/public/coa_a`, EVM address `addr_A`) and `COA_B` (at `/public/coa_b`, EVM address `addr_B`).
2. Alice signs `signedData = "authenticate_for_app_X"` to prove ownership of `COA_A` to some application.
3. An attacker calls:
   ```cadence
   EVM.validateCOAOwnershipProof(
       address: alice_address,
       path: /public/coa_b,           // COA_B, not COA_A
       signedData: "authenticate_for_app_X".utf8,
       keyIndices: [0],
       signatures: [alice_sig_for_coa_a],
       evmAddress: addr_B             // COA_B's EVM address
   )
   ```
4. The function returns `ValidationResult(isValid: true)` because:
   - Alice's signature over `"authenticate_for_app_X"` is cryptographically valid (Step 1 passes).
   - Alice's COA at `/public/coa_b` has EVM address `addr_B` (Step 2 passes).
5. The attacker has successfully proven ownership of `COA_B` using a signature Alice produced for `COA_A`.

The EVM-side precompile `verifyCOAOwnershipProof` is not vulnerable because it always passes the calling contract's own address as `evmAddress`: [3](#0-2) 

However, the Cadence function is directly callable by any script or transaction, bypassing the precompile's safe binding.

---

### Impact Explanation

Any application that uses `EVM.validateCOAOwnershipProof` directly to gate access to on-chain assets or privileged operations is vulnerable to unauthorized authentication. An attacker who obtains a valid COA ownership proof signature from a victim — from a prior authentication session, a cross-application signature, or any context where the victim signed an arbitrary message — can replay it to authenticate as a different COA owned by the same Cadence account. This constitutes unauthorized access to accounts or on-chain assets protected by COA ownership proof, matching the target impact class. [4](#0-3) 

---

### Likelihood Explanation

The EVM precompile path is safe. The vulnerable path requires: (1) the victim to own multiple COAs under one Cadence account, (2) an application to use `validateCOAOwnershipProof` directly for authentication rather than through the precompile, and (3) the attacker to obtain a valid signature from the victim. As the Flow EVM ecosystem grows and more off-chain authentication flows are built on top of this function — exactly the use case the code comment warns about — the likelihood increases. The missing check is not a theoretical edge case; it is explicitly flagged in the production source as an unresolved risk. [1](#0-0) 

---

### Recommendation

Enforce domain separation inside `validateCOAOwnershipProof` by requiring that `signedData` encodes `evmAddress`. Concretely, before calling `keyList.verify`, assert that the 20-byte `evmAddress` is present in `signedData` (e.g., as a fixed prefix or suffix), or define a canonical signed message format that commits to `evmAddress`. This mirrors the fix applied in the referenced zkSync commit: the linkage between the committed object and the proof must be enforced inside the verification function itself, not delegated to callers.

### Proof of Concept

```cadence
import EVM from 0x1

// Alice previously signed signedData = "login:app_X" to prove ownership of COA_A.
// Attacker replays that exact signature against COA_B (same Cadence account, different EVM address).

access(all) fun main(
    aliceAddress: Address,
    aliceSignature: [UInt8],
    coaBEvmAddress: [UInt8; 20]
): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: aliceAddress,
        path: /public/coa_b,                  // COA_B path
        signedData: "login:app_X".utf8,        // original signedData for COA_A
        keyIndices: [0],
        signatures: [aliceSignature],          // Alice's signature for COA_A
        evmAddress: coaBEvmAddress             // COA_B's EVM address
    )
    // Returns ValidationResult(isValid: true, problem: nil)
    // even though Alice never authorized COA_B for this signed data.
}
