### Title
Malicious CKD Participant Can Send Arbitrary Group Elements to Corrupt the Coordinator's Derived Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates `(big_y, big_c)` contributions from all participants with no cryptographic verification of their correctness. A single malicious participant can send arbitrary G1 elements, causing the coordinator to produce a `CKDOutput` that does not equal `msk · H(pk ‖ app_id)`. The honest app will then unmask an incorrect confidential key, permanently corrupting the CKD result for that session.

---

### Finding Description

The CKD protocol is a one-round aggregation. Each participant is supposed to compute and send:

- `big_y_i = λ_i · y_i · G₁` (blinding factor commitment)
- `big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · A)` (masked secret-share contribution)

where `x_i` is the participant's private share, `y_i` is a fresh random scalar, and `A = a · G₁` is the app's public key.

The coordinator in `do_ckd_coordinator` receives these values and sums them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

There is **no proof of knowledge** and **no consistency check** verifying that:

1. `big_y_i` and `big_c_i` use the same scalar `y_i`.
2. The correct private share `x_i` was used in `big_c_i`.
3. The received elements are even on the curve (beyond deserialization).

A malicious participant `P_m` can send any pair `(big_y_m', big_c_m')` of arbitrary G1 elements. The coordinator has no mechanism to detect or reject them.

The final output is used by the app to recover the confidential key:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [2](#0-1) 

With a corrupted `(Y, C)`, the app computes `C − a·Y ≠ msk · H(pk ‖ app_id)`, receiving a wrong key.

The `compute_signature_share` function shows the correct construction that each honest participant performs, but nothing enforces this on the wire: [3](#0-2) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A single malicious participant can cause the coordinator to output a `CKDOutput` that is cryptographically inconsistent with `msk · H(pk ‖ app_id)`. The app, which has no way to verify the output independently (it does not know `msk`), will silently accept and use the wrong key. This permanently corrupts the CKD result for that session. Because the protocol is designed to provide *deterministic* secrets to TEE applications, a corrupted output cannot be distinguished from a correct one by the app without an out-of-band oracle.

---

### Likelihood Explanation

**High.** Any single participant in the CKD session can mount this attack. The attacker needs only to be a valid protocol participant (i.e., hold a legitimate key share from DKG). No special cryptographic capability is required — the attacker simply sends two arbitrary G1 points instead of their honest contribution. The coordinator performs no verification before accepting and aggregating the values.

---

### Recommendation

Add a zero-knowledge proof of correct formation alongside each `(big_y_i, big_c_i)` contribution. Concretely, each participant should prove knowledge of `y_i` such that:

- `big_y_i = λ_i · y_i · G₁`
- `big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · A)`

A standard Chaum–Pedersen / sigma-protocol proof of discrete-log equality between `big_y_i / (λ_i · G₁)` and `(big_c_i − λ_i · x_i · H(pk ‖ app_id)) / A` would suffice, binding `y_i` across both components. The coordinator must verify all such proofs before aggregating.

---

### Proof of Concept

**Setup:** 3 participants `{P₁, P₂, P₃}`, threshold 2, `P₁` is coordinator, `P₃` is malicious.

**Honest execution:**
- `P₂` computes `(λ₂·y₂·G, λ₂·(x₂·H + y₂·A))` and sends to `P₁`.
- `P₃` should compute `(λ₃·y₃·G, λ₃·(x₃·H + y₃·A))` but instead sends `(G, G)` (arbitrary points).

**Coordinator aggregation** (`do_ckd_coordinator`, lines 50–57):
```
norm_big_y = λ₁·y₁·G + λ₂·y₂·G + G          ← corrupted
norm_big_c = λ₁·(x₁·H+y₁·A) + λ₂·(x₂·H+y₂·A) + G  ← corrupted
```

**App unmask:**
```
C − a·Y = msk·H + G − a·G  ≠  msk·H
```

The app receives `msk·H + (1−a)·G` — a wrong key — with no error raised anywhere in the protocol. The `recv_from_others` call at line 51 accepts the message purely on the basis of the sender's identity, not the content's validity. [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
