### Title
`MetricOmmPoolFactory::setPoolAdminFeeDestination()` does not flush accrued fees to the old destination before redirecting, causing the old admin fee recipient to lose all earned fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees`. All spread and notional fees that accrued under the old destination are subsequently swept to the new destination on the next collection, permanently depriving the old recipient of earned fees.

---

### Finding Description

Every other fee-configuration mutator in `MetricOmmPoolFactory` flushes accrued fees at the current rates before applying the change:

- `setPoolAdminFees` (lines 408–435): calls `collectFees(…, poolAdminFeeDestination[pool])` before writing new fee components.
- `setPoolProtocolFee` (lines 318–360): calls `collectFees(…, poolAdminFeeDestination[pool])` before writing new protocol components.

`setPoolAdminFeeDestination` (lines 438–447) breaks this pattern:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

After this call, `poolAdminFeeDestination[pool]` points to the new address. The next invocation of `collectFees` — whether via `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee` — passes the **new** destination to the pool:

```solidity
// MetricOmmPool.collectFees (lines 416–421)
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);
}
```

Two categories of accrued admin fees are misdirected:

1. **Spread fees** — the surplus balance (`balance * scaleMultiplier − binTotals.scaledToken* − notionalFee*`) that has built up from swap spread income since the last collection.
2. **Notional fees** — the explicit `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` accumulators that grow with every swap.

Both are computed and transferred in a single `collectFees` call using whichever destination is stored at that moment. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

The `adminFeeDestination` is a distinct address from `poolAdmin` — it is set independently at pool creation via `PoolParameters.adminFeeDestination` and stored separately in `poolAdminFeeDestination[pool]`. It is the intended recipient of the admin share of all swap fees. When the pool admin rotates this address, all fees that accrued under the old destination are silently redirected to the new one. The old recipient (e.g., a DAO treasury, a multisig, or a revenue-sharing contract) receives nothing for the period it was the designated beneficiary. This is a direct, unrecoverable loss of protocol fee revenue for the old destination. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The pool admin is a semi-trusted role that legitimately calls `setPoolAdminFeeDestination` during normal operations (treasury rotation, multisig migration, admin handover). No special precondition is required beyond the pool having accrued any non-zero fees since the last collection — which is the normal operating state of any active pool. The trigger is a routine, valid admin action. [1](#0-0) 

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: flush accrued fees to the **current** destination before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush all accrued fees to the current destination before changing it.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
``` [6](#0-5) 

---

### Proof of Concept

1. Pool is deployed with `adminFeeDestination = treasuryA`, `adminSpreadFeeE6 = 100_000` (10%).
2. Swaps occur; spread surplus and notional accumulators grow — e.g., 1 000 token0 and 500 token1 owed to the admin.
3. Pool admin calls `setPoolAdminFeeDestination(pool, treasuryB)`.
   - `poolAdminFeeDestination[pool]` is overwritten to `treasuryB` with no `collectFees` call.
4. Anyone calls `collectPoolFees(pool)`.
   - `collectFees` is invoked with `adminFeeDestination_ = treasuryB`.
   - `transferToken0(treasuryB, adminShare0)` and `transferToken1(treasuryB, adminShare1)` execute.
5. `treasuryA` receives 0 tokens despite being the designated recipient for the entire accrual period.
   `treasuryB` receives fees it did not earn. [7](#0-6) [8](#0-7)
