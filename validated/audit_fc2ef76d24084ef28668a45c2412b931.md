### Title
`max_nonce_for_validation_skip` config parameter ignored ŌĆö hardcoded `Nonce(Felt::ONE)` in `skip_stateful_validations` bypasses account `__validate__` regardless of operator config ŌĆö (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidatorConfig` declares and serializes a field `max_nonce_for_validation_skip` described as "Maximum nonce for which the validation is skipped." The gateway's `skip_stateful_validations` function never reads this field; it hardcodes `Nonce(Felt::ONE)` directly. When an operator sets `max_nonce_for_validation_skip = 0` to disable the deploy-account + invoke validation-skip UX feature, the gateway ignores the setting and still skips the account `__validate__` entry point for every nonce-1 invoke transaction whose sender has a pending deploy-account, accepting those transactions without signature verification.

---

### Finding Description

`StatefulTransactionValidatorConfig` in `crates/apollo_gateway_config/src/config.rs` declares:

```rust
pub max_nonce_for_validation_skip: Nonce,
```

with default `Nonce(Felt::ONE)`, serialized to the production config schema at key `gateway_config.static_config.stateful_tx_validator_config.max_nonce_for_validation_skip` (value `"0x1"`). [1](#0-0) 

The standalone function `skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` is called from `run_pre_validation_checks` and decides whether to set `validate = false` (skipping the account's `__validate__` entry point). It receives no config argument and hardcodes the threshold:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [2](#0-1) 

The caller passes only `executable_tx`, `account_nonce`, and `mempool_client` ŌĆö `self.config.max_nonce_for_validation_skip` is never forwarded:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [3](#0-2) 

The `skip_validate` result is then used to set `validate: !skip_validate` in `ExecutionFlags`, which controls whether the blockifier calls the account's `__validate__` entry point: [4](#0-3) 

By contrast, the `native_blockifier` path (`PyValidator::should_run_stateful_validations`) correctly reads `self.max_nonce_for_validation_skip` and applies it as a range bound:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
``` [5](#0-4) 

The config field is therefore **dead code** in the gateway path. The production config schema exposes it as a tunable parameter, but changing it has no effect. [6](#0-5) 

---

### Impact Explanation

When an operator sets `max_nonce_for_validation_skip = 0` (intending to disable the UX skip entirely), the gateway still returns `skip_validate = true` for any invoke transaction with `tx.nonce() == 1` and `account_nonce == 0` whose sender has a pending deploy-account in the mempool or a recent block. The blockifier then executes that invoke with `validate = false`, meaning the account contract's `__validate__` entry point ŌĆö where ECDSA signature verification occurs ŌĆö is never called. The gateway admits an invoke transaction without verifying its signature, satisfying the Critical impact criterion: **invalid or unauthorized Starknet transaction accepted through account validation logic**.

---

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded constant, so the bug is invisible under default configuration. An operator must explicitly set the field to `0` to expose the discrepancy. The field is publicly documented and serialized in the production config schema, making operator misconfiguration a realistic trigger. Likelihood is **Medium**.

---

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` and replace the hardcoded equality check with the same range logic used in `PyValidator`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,   // ŌåÉ add parameter
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        let tx_nonce = tx.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let within_skip_range = tx_nonce <= max_nonce_for_validation_skip;
        if is_post_deploy_nonce && within_skip_range && account_nonce == Nonce(Felt::ZERO) {
            // existing mempool check ŌĆ”
        }
    }
    Ok(false)
}
```

Update the call site in `run_pre_validation_checks` to forward `self.config.max_nonce_for_validation_skip`.

---

### Proof of Concept

1. Deploy a sequencer node with `max_nonce_for_validation_skip = 0` in the gateway config (operator intends to disable the skip feature).
2. Submit a `DeployAccount` transaction for a fresh address `A`; it enters the mempool.
3. Submit an `Invoke` transaction from `A` with `nonce = 1` and a **garbage signature** (e.g., all-zero felts).
4. The gateway calls `skip_stateful_validations`. Because `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)` and `account_tx_in_pool_or_recent_block` returns `true`, the function returns `true` ŌĆö ignoring `max_nonce_for_validation_skip = 0`.
5. `run_validate_entry_point` is called with `skip_validate = true`, setting `validate = false` in `ExecutionFlags`.
6. The blockifier executes the invoke without calling `__validate__`; the garbage signature is never checked.
7. The gateway returns success and the transaction enters the mempool, despite the operator having configured the skip feature off.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
