### Title
Missing Sequencer Access Control in `update_rewards` Allows Any Caller to Permanently Freeze Staker Yield — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function is specified to be callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged caller can invoke it with `disable_rewards: true`, consuming the single per-block reward slot and permanently preventing legitimate reward distribution for that block.

---

### Finding Description

The protocol specification explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation of `update_rewards` in `StakingRewardsManagerImpl` only calls `general_prerequisites()`: [2](#0-1) 

`general_prerequisites()` only checks that the contract is unpaused and the caller is non-zero — there is no sequencer identity check: [3](#0-2) 

Critically, `last_reward_block` is written **before** the `disable_rewards` guard: [4](#0-3) 

This means any caller can:
1. Call `update_rewards(staker_address: <any_valid_staker>, disable_rewards: true)` in block N.
2. `last_reward_block` is set to N and the function returns early — no rewards are distributed.
3. The legitimate sequencer call for block N fails with `REWARDS_ALREADY_UPDATED` because `current_block_number > last_reward_block` is now false.

`last_reward_block` is a single global variable shared across all stakers: [5](#0-4) 

A single attacker call per block is sufficient to deny rewards to **all** stakers and delegators for that block.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(..., disable_rewards: true)` for every block permanently prevents all stakers and all delegation pool members from accruing consensus-era block rewards. The rewards for each skipped block are irrecoverably lost — there is no catch-up mechanism. Pool rewards are transferred to pool contracts only inside `_update_rewards`: [6](#0-5) 

Staker `unclaimed_rewards_own` is incremented only there: [7](#0-6) 

Neither is ever back-filled for a skipped block.

---

### Likelihood Explanation

**High.** The attack requires:
- No capital.
- No privileged role.
- Only a valid (non-zero) staker address to pass the `STAKER_NOT_EXISTS` / `INVALID_STAKER` checks.

The attacker simply needs to submit a transaction per block before the sequencer's own `update_rewards` call. Because the function is publicly callable with no caller restriction, this is trivially automatable. The test suite itself calls `update_rewards` without any `cheat_caller_address` setup, confirming the absence of any on-chain caller check: [8](#0-7) 

---

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, analogous to how `update_rewards_from_attestation_contract` enforces `assert_caller_is_attestation_contract`: [9](#0-8) 

Introduce a stored `sequencer_address` (or use an existing role such as `OPERATOR`) and assert `get_caller_address() == sequencer_address` before any state mutation in `update_rewards`.

---

### Proof of Concept

1. Deploy the staking system and advance past the consensus rewards start epoch.
2. As an unprivileged address (any non-zero address), call:
   ```
   staking.update_rewards(staker_address: <any_active_staker>, disable_rewards: true)
   ```
3. Observe that `last_reward_block` is now set to the current block and no rewards were distributed.
4. The sequencer's legitimate `update_rewards(..., disable_rewards: false)` call for the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat for every block — stakers and delegators accumulate zero rewards indefinitely.

The `REWARDS_ALREADY_UPDATED` revert path is confirmed by the existing test suite: [10](#0-9)
