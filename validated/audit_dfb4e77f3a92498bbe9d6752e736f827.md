### Title
Owner Can Change ETH Connector Contract Account to Arbitrary Address Without Timelock, Enabling Theft of Bridged ETH - (File: `engine/src/contract_methods/connector.rs`)

### Summary

`set_eth_connector_contract_account` allows the Aurora Engine owner to instantly redirect the ETH connector to any arbitrary NEAR account with no timelock, delay, or interface validation. The stored connector account is the target of all ETH withdrawal cross-contract calls. A malicious owner can point it to a contract that silently discards `engine_withdraw` calls, causing users who invoke `exitToEthereum` to permanently lose their EVM ETH balance with no corresponding Ethereum-side release.

### Finding Description

`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` accepts any arbitrary NEAR `AccountId` and writes it directly to storage: [1](#0-0) 

The only access control is `require_owner_only` (or a private self-call). There is no minimum delay, no timelock, and no validation that the new account implements the ETH connector interface. The stored account ID is then consumed in two critical paths:

1. **`get_connector_account_id`** (line 550) — used by `ft_on_transfer` to decide whether incoming NEP-141 tokens are base ETH or ERC-20 tokens: [2](#0-1) 

2. **`get_eth_connector_contract_account`** in the precompile layer — used by `ExitToNear`/`ExitToEthereum` to resolve the target of the `engine_withdraw` cross-contract promise: [3](#0-2) 

The `withdraw` function creates the outbound promise targeting this stored account: [4](#0-3) 

`set_connector_account_id` writes the new account unconditionally: [5](#0-4) 

The public NEAR entrypoint is exposed with no additional guard: [6](#0-5) 

### Impact Explanation

**Critical — direct theft of user funds.**

When a user calls the `exitToEthereum` precompile from within the EVM:

1. The EVM execution reduces the caller's EVM ETH balance by the specified amount (this is irreversible within the EVM state).
2. The precompile reads the stored connector account ID from storage.
3. A cross-contract promise is created to call `engine_withdraw` on that account.
4. If the account has been changed to a malicious contract, the malicious contract receives the call, does not burn any NEP-141 tokens, and does not initiate the Ethereum-side release.
5. The user's EVM ETH balance is permanently reduced with no ETH received on Ethereum.

All users who attempt to bridge ETH out of Aurora after the connector account is changed lose their funds. The attack is silent — no on-chain event distinguishes a legitimate connector change from a malicious one, and there is no delay for users to detect and react.

### Likelihood Explanation

The attack requires the owner account to act maliciously or be compromised. However, the structural vulnerability is identical to the confirmed Fraxlend M-08 finding: a critical security address (the ETH connector, analogous to the timelock contract) can be changed to an arbitrary value by the owner with no delay, giving users zero opportunity to detect and exit before funds are at risk. The `EngineState` stores `owner_id` as a mutable field: [7](#0-6) 

The owner is a single NEAR account with no on-chain governance delay on this specific parameter, making the attack surface realistic for any deployment where the owner key is not perfectly secured.

### Recommendation

Implement a two-step commit-reveal pattern or a block-height-based timelock on `set_eth_connector_contract_account` changes, analogous to the `stage_upgrade`/`deploy_upgrade` pattern already used for contract upgrades: [8](#0-7) 

The new connector account should be staged with a minimum delay (at least matching `upgrade_delay_blocks`) before it becomes active, giving users time to observe the pending change and exit if desired. Additionally, consider emitting a hashchain event on connector account changes so the change is auditable.

#[named]
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

#[named]
pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}
