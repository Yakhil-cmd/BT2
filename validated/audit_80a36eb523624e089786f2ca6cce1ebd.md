Looking at the CKD protocol in `src/confidential_key_derivation/protocol.rs` and the proof primitives in `src/crypto/proofs/`, I need to find an analog to "accepting return data without validity checks."

The direct analog is: the CKD coordinator receives participant contributions and sums them with zero cryptographic validation — exactly like accepting Chainlink data without checking roundID or staleness.

### Title
Missing Cryptographic Validation of Participant Contributions in CKD Coordinator Allows Malicious Participant to Corrupt Confidential Derived Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary
The CKD coordinator blindly accumulates `(big_y, big_c)` shares from every participant with no proof-of-correctness check. A single malicious participant can inject arbitrary group elements, causing the coordinator to output a structurally valid but cryptographically wrong `CKDOutput`, permanently corrupting the confidential derived key for all honest parties.

---

### Finding Description

`do_ckd_coordinator` collects one `CKDOutput` per participant and unconditionally adds each contribution into the running sums:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();   // no check
    norm_big_c += participant_output.big_c();   // no check
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each participant is supposed to send:

```
norm_big_y = λᵢ · (yᵢ · G₁)
norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)
``` [2](#0-1) 

where `xᵢ` is the participant's private signing share. There is no zero-knowledge proof (e.g., a `dlogeq`-style proof) that `big_c` was formed using the participant's actual `xᵢ`, and no proof that `big_y` and `big_c` are consistent with each other. The coordinator has no way to distinguish a correctly-formed share from an arbitrary group element.

Contrast this with the DKG protocol, which enforces a proof-of-knowledge for every polynomial commitment before accepting it:

```rust
verify_proof_of_knowledge(
    &session_id,
    &mut proof_domain_separator.clone(),
    threshold,
    p,
    old_participants.clone(),
    commitment_i,
    proof_i.as_ref(),
)?;
``` [3](#0-2) 

No equivalent guard exists in the CKD path.

The library already defines a `dlogeq` proof primitive that proves knowledge of a scalar `x` satisfying `public0 = x·G` and `public1 = x·generator1` — precisely the relation needed to bind a participant's public key share to their CKD contribution — but it is never invoked from the CKD protocol: [4](#0-3) 

---

### Impact Explanation

The coordinator's final output is:

```
Y  = Σ λᵢ · yᵢ · G₁
C  = Σ λᵢ · (xᵢ · H(pk‖app_id) + yᵢ · app_pk)
```

The app recovers the confidential key as `C − app_sk · Y = msk · H(pk‖app_id)`. If any single participant substitutes an arbitrary `(big_y′, big_c′)`, the sum shifts by `(big_y′ − big_y_correct, big_c′ − big_c_correct)`, and the recovered key becomes `msk · H(pk‖app_id) + noise`. The output is structurally indistinguishable from a correct one; honest parties have no way to detect the corruption. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

---

### Likelihood Explanation

The attack requires only that one participant deviate from the protocol — sending `(identity, identity)` or any other pair of group elements. No cryptographic assumption needs to be broken. The malicious participant is an explicitly listed attacker in the threat model, and the entry path (sending a message to the coordinator) is the normal protocol flow.

---

### Recommendation

Before summing contributions, the coordinator should require each participant to attach a zero-knowledge proof that their `(big_y, big_c)` was formed correctly from their actual private share. Concretely:

1. Each participant generates a `dlogeq`-style proof (adapted to BLS12-381 G₁) showing that the same scalar `xᵢ` satisfies both `xᵢ · G₂ = pk_share_i` (the public key share from DKG) and `xᵢ · H(pk‖app_id) = big_s_i`.
2. The coordinator verifies every proof before adding the contribution, returning an error (and identifying the culprit) on failure — mirroring the pattern already used in `do_keyshare`.

---

### Proof of Concept

1. Run a 3-of-3 CKD session. Participants are `P1`, `P2`, `P3`; `P3` is malicious.
2. `P3` sends `CKDOutput { big_y: G₁::identity(), big_c: G₁::identity() }` instead of its correct share.
3. The coordinator executes:
   ```
   norm_big_y += identity  →  Y is missing P3's contribution
   norm_big_c += identity  →  C is missing P3's contribution
   ```
4. The coordinator returns `CKDOutput::new(corrupted_Y, corrupted_C)` with no error.
5. The app computes `corrupted_C − app_sk · corrupted_Y ≠ msk · H(pk‖app_id)`.
6. The derived confidential key is permanently wrong; no honest party can detect the corruption from the output alone.

pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    Ok(e == proof.e.0)
