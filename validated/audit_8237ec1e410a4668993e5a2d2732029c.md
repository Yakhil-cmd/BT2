Audit Report

## Title
DoS via `update_balance` Guard Slot Exhaustion in ckBTC Minter - (File: `rs/bitcoin/ckbtc/minter/src/guard.rs`)

## Summary
The ckBTC minter enforces a global cap of `MAX_CONCURRENT = 100` concurrent `update_balance` calls via a `Guard<PendingBalanceUpdates>` keyed on `Account`. Because the guard is held across an async `get_utxos` await and there is no per-principal quota or slot timeout, an unprivileged attacker can fill all 100 slots using 100 distinct subaccounts of a single principal, blocking every legitimate `update_balance` call for the duration of the attack. The attacker pays only IC cycles — no BTC or ckBTC is required.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/guard.rs` (L6, L51–54), `MAX_CONCURRENT` is set to 100 and the guard rejects any new call once the global set reaches that size:

```rust
const MAX_CONCURRENT: usize = 100;
// ...
if accounts.len() >= MAX_CONCURRENT {
    return Err(GuardError::TooManyConcurrentRequests);
}
accounts.insert(account);
```

The guard is acquired in `update_balance` (L168 of `update_balance.rs`) before the async `get_utxos` call (L175–183), and is only released when the `Guard` object is dropped at function return:

```rust
let _guard = balance_update_guard(caller_account)?;
// ...
let utxos = get_utxos(btc_network, &address, min_confirmations, CallSource::Client, runtime).await?...;
```

The `caller_account` is constructed from `args.owner.unwrap_or(caller)` and `args.subaccount` (L164–167), meaning any caller can freely specify 100 distinct `Account` values (e.g., `subaccount: Some([i; 32])` for `i` in `0..100`) without owning any BTC. There is no per-principal quota in the guard, no minimum-UTXO pre-check before slot acquisition, and no timeout on held slots. The `Drop` implementation (L63–66 of `guard.rs`) only removes the account on normal completion, so slots are held for the full Bitcoin canister round-trip.

## Impact Explanation
`update_balance` is the sole mechanism by which users mint ckBTC after depositing BTC on-chain. Exhausting all 100 guard slots prevents any new ckBTC from being minted for the duration of the attack. Legitimate depositors receive `UpdateBalanceError::TemporarilyUnavailable("too many concurrent requests")` and cannot advance their UTXO state. This constitutes a sustained, application-level DoS on a core ck-token financial integration with concrete user harm, matching the **High ($2,000–$10,000)** impact category: *"Application/platform-level DoS … not based on raw volumetric DDoS"* and *"Significant … ck-token … security impact with concrete user or protocol harm."*

## Likelihood Explanation
The attack requires no privileges, no BTC, and no ckBTC. Any IC principal can call `update_balance` with arbitrary `args.owner` and `args.subaccount`. Generating 100 distinct accounts costs nothing beyond the cycles for 100 concurrent canister calls. The Bitcoin canister's `get_utxos` round-trip takes several seconds, giving the attacker a comfortable window to re-submit calls as slots free up. The blockade can be sustained indefinitely at low, predictable cycle cost.

## Recommendation
- **Short term**: Before acquiring the guard, perform a cheap pre-check (e.g., verify the account has at least one known UTXO, or enforce a per-principal slot quota) so that accounts with no UTXOs are rejected without holding a slot.
- **Long term**: Introduce a guard-slot timeout so that slots held beyond a configurable threshold are automatically released. Alternatively, increase `MAX_CONCURRENT` and add per-principal quotas to raise the cost of exhaustion proportionally.

## Proof of Concept
1. Attacker generates 100 accounts: `Account { owner: attacker_principal, subaccount: Some([i; 32]) }` for `i` in `0..100`.
2. Attacker sends 100 concurrent `update_balance` calls, one per account, each with `args.owner = Some(attacker_principal)` and `args.subaccount = Some([i; 32])`.
3. Each call acquires a guard slot at `guard.rs:54` and suspends at `update_balance.rs:175` awaiting `get_utxos`.
4. `update_balance_accounts.len()` reaches 100.
5. Any new `update_balance` call from a legitimate user hits `guard.rs:51–52` and returns `GuardError::TooManyConcurrentRequests` → `UpdateBalanceError::TemporarilyUnavailable`.
6. As old calls complete and slots free, the attacker immediately re-submits to refill them, sustaining the blockade at cycles cost only.

A deterministic unit test confirming the exhaustion path already exists in `guard.rs` (L141–162): `guard_prevents_more_than_max_concurrent_accounts` demonstrates that exactly `MAX_CONCURRENT` guards can be held simultaneously and the 101st call returns `TooManyConcurrentRequests`. Extending this test with mixed-subaccount accounts from a single principal directly reproduces the attack precondition.
