### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Unclaimed Pool Yield via Trace Bloat - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in the `Pool` contract iterates over a pool member's entire `pool_member_epoch_balance` trace in an unbounded loop. Because any delegator can grow this trace by one entry per epoch (by calling `add_to_delegation_pool` or `exit_delegation_pool_intent` each epoch), a delegator who accumulates many balance-change entries without claiming rewards will eventually cause `claim_rewards` to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

### Finding Description

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each iteration performs at least two storage reads: one via `pool_member_trace.at(entry_to_claim_from)` and one via `find_sigma`. Storage reads are the most expensive operations on Starknet.

The trace grows via `set_member_balance`, which inserts at key `current_epoch + K`:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(...);
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

The `insert` function only deduplicates within the same epoch key — calls in different epochs each append a new checkpoint:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

Therefore, a delegator who calls `add_to_delegation_pool` (or `exit_delegation_pool_intent`) once per epoch accumulates exactly one new trace entry per epoch. After N epochs without claiming rewards, the loop in `calculate_rewards` must iterate over all N entries.

The `claim_rewards` function passes `pool_member_info.entry_to_claim_from` as the loop start index, which is only advanced upon a *successful* claim. If the transaction reverts due to gas exhaustion, `entry_to_claim_from` is never updated, making every subsequent `claim_rewards` attempt equally expensive and equally likely to revert. [4](#0-3) 

### Impact Explanation

Once the trace is large enough that `claim_rewards` exceeds the gas limit, the pool member's accumulated yield is permanently frozen: no successful call to `claim_rewards` can ever complete. The funds themselves (principal) remain accessible via `exit_delegation_pool_intent` + `exit_delegation_pool_action`, but the unclaimed STRK rewards are irrecoverable. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

Additionally, even before the hard gas-limit failure, an attacker can force any caller of `pool_member_info_v1` (which also calls `calculate_rewards` without updating `entry_to_claim_from`) to consume unbounded gas on every invocation. [5](#0-4) 

### Likelihood Explanation

The trigger requires no special privileges — any delegator can call `add_to_delegation_pool` with the minimum non-zero amount each epoch. The minimum delegation amount is `STRK_CONFIG.min_for_rewards = 10^18` (1 STRK). [6](#0-5) 

This can also occur non-maliciously: a legitimate delegator who frequently adjusts their stake (e.g., dollar-cost averaging) but infrequently claims rewards will naturally accumulate a large trace. The protocol provides no warning or mechanism to prevent this. Likelihood is **Medium** (requires sustained activity over many epochs, but is reachable by any unprivileged delegator with no special access).

### Recommendation

1. **Enforce a claim before balance changes**: Require `claim_rewards` to be called (or auto-call it internally) before any operation that appends to the trace (`add_to_delegation_pool`, `exit_delegation_pool_intent`). This keeps `entry_to_claim_from` at the tail of the trace, bounding each `claim_rewards` call to at most a constant number of iterations (≤ K+1 entries ahead of the last claim).

2. **Alternatively, bound the loop**: Introduce a maximum iteration count per `claim_rewards` call and allow partial claims, storing the updated `entry_to_claim_from` so subsequent calls resume from where the previous one stopped.

3. **Add a trace length guard**: Reject balance-change operations if the unclaimed trace segment (`pool_member_trace_length - entry_to_claim_from`) exceeds a safe threshold, forcing the member to claim first.

### Proof of Concept

```
// Epoch 0: Delegator enters pool with 1 STRK
enter_delegation_pool(reward_address, amount: 1e18)
// Trace length = 1

// Epochs 1..N: Delegator adds 1 STRK each epoch, never claims
for epoch in 1..N:
    advance_epoch()
    add_to_delegation_pool(pool_member, amount: 1e18)
    // Each call appends one new entry (different epoch key = current_epoch + K)
// Trace length = N

// After N epochs, call claim_rewards:
claim_rewards(pool_member)
// calculate_rewards loops N times, each iteration doing 2+ storage reads
// At N ≈ several hundred epochs, transaction exceeds Starknet gas limit
// Transaction reverts → entry_to_claim_from unchanged → rewards permanently frozen
// All subsequent claim_rewards calls also revert
```

The `add_to_delegation_pool` entry point is open to any existing pool member (caller must be `pool_member` or `reward_address`): [7](#0-6)
