### Title
Unbounded Iteration Over All Historical Stakers in `get_stakers` Causes Unbounded Gas Consumption - (File: src/staking/staking.cairo)

### Summary

`get_stakers` in `StakingConsensusImpl` iterates over every address ever added to the `stakers` storage `Vec`, including stakers who have fully exited the protocol. Because the `stakers` vector is append-only and never pruned, the gas cost of this function grows without bound as the protocol accumulates historical participants.

### Finding Description

The `stakers` storage field is declared as:

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

`get_stakers` iterates over the entire vector with no pagination or upper bound:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    let staking_power = self
        .get_staker_staking_power_at_epoch(...);
    ...
    let public_key = self.get_public_key_at_epoch(...);
    let peer_id   = self.get_peer_id_at_epoch(...);
    stakers.append(...);
}
``` [2](#0-1) 

For every active staker the loop also calls `get_staker_staking_power_at_epoch`, which internally calls `get_staker_total_strk_btc_balance_at_epoch`, which itself iterates over all pools of that staker:

```cairo
for (pool_contract, token_address) in staker_pool_info.pools {
    ...
}
``` [3](#0-2) 

Additionally, `get_total_staking_power_at_epoch` (called once per `get_stakers` invocation) iterates over all registered BTC tokens:

```cairo
for (token_address, active_status) in self.btc_tokens {
    ...
}
``` [4](#0-3) 

The combined complexity is **O(N × P + T)** where N = total stakers ever registered (monotonically increasing), P = average number of pools per active staker, and T = total BTC token types. There is no pagination, no cap, and no pruning mechanism.

### Impact Explanation

As the protocol matures and accumulates thousands of historical stakers (including fully-exited ones that remain in the `stakers` Vec), every call to `get_stakers` consumes proportionally more gas. On Starknet, even view functions executed on-chain (e.g., called from another contract) consume gas and are subject to block gas limits. If the vector grows large enough, the function will exceed the Starknet transaction gas limit and revert, permanently breaking the consensus layer's ability to query the staker set on-chain. Off-chain callers (RPC nodes) will also experience increasing latency and eventual timeouts.

This matches the allowed impact: **Medium — Unbounded gas consumption / griefing with damage to the protocol**.

### Likelihood Explanation

Every call to `stake` appends to the `stakers` Vec. The minimum stake requirement creates a cost barrier, but the protocol is designed to onboard a large and growing validator set. Stakers who exit are never removed. No governance action or upgrade is needed to trigger the issue — normal protocol growth is sufficient. The function is callable by any address with no access restriction. [5](#0-4) 

### Recommendation

1. **Maintain a separate active-stakers set** (e.g., an `IterableMap` keyed by staker address) that entries are removed from on `unstake_action`. Iterate only over this set in `get_stakers`.
2. Alternatively, **add pagination** to `get_stakers` (cursor + limit parameters) so callers can retrieve the staker list in bounded chunks.
3. If the append-only `stakers` Vec must be kept for other purposes, at minimum add a `max_stakers` guard inside the loop to prevent unbounded execution.

### Proof of Concept

1. Register N stakers (each calling `stake` with `min_stake`).
2. Have all N stakers call `unstake_intent` followed by `unstake_action` to fully exit.
3. Call `get_stakers(epoch_id)` from any address.
4. Observe that the function iterates over all N entries in `self.stakers`, performing multiple storage reads per entry, even though zero stakers are active. Gas cost scales linearly with N with no bound. [6](#0-5)

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
