### Title
L1 Priority Transactions Excluded by Block Limits Without Exemption, Enabling Priority Queue Stall — (`basic_bootloader/src/bootloader/block_flow/zk/tx_loop.rs`)

---

### Summary

The ZKsync OS transaction loop applies `check_for_block_limits` uniformly to every transaction, including L1→L2 priority transactions. There is no exemption for priority transactions from block-level resource enforcement. An unprivileged attacker who submits an L1 priority transaction whose execution would push any block resource counter (gas, native, pubdata, logs) past its limit causes that transaction to be silently excluded from the block — its hash is never added to `enforced_transaction_hashes_accumulator` or `batch_data`. Because the L1 priority queue requires every enqueued transaction to be processed, a permanently-unprocessable priority transaction can stall the chain.

---

### Finding Description

The ZK transaction loop in `basic_bootloader/src/bootloader/block_flow/zk/tx_loop.rs` processes every transaction — L2 and L1 priority alike — through the same post-execution gate:

```
// lines 150-162
if let Err(err) = check_for_block_limits(
    system,
    next_block_gas_used,
    next_block_computational_native_used,
    next_block_pubdata_used,
    block_logs_used,
    next_block_blob_gas_used,
) {
    // Revert to state before transaction
    system.finish_global_frame(Some(&pre_tx_rollback_handle))?;
    result_keeper.tx_processed(Err(err));   // ← excluded, no hash recorded
``` [1](#0-0) 

`check_for_block_limits` in `basic_bootloader/src/bootloader/block_flow/zk/mod.rs` enforces five hard limits with no priority-transaction bypass:

```rust
fn check_for_block_limits<S: EthereumLikeTypes>(
    system: &mut System<S>,
    gas_used: u64, computational_native_used: u64,
    pubdata_used: u64, logs_used: u64, blob_gas_used: u64,

``` [2](#0-1) 

The `is_priority_tx` flag is only consulted **after** the block-limits gate passes (line 201), so a transaction that fails the gate is never recorded:

```rust
if tx_processing_result.is_priority_tx {
    block_data.enforced_transaction_hashes_accumulator.add_tx_hash(...);
    batch_data.add_tx_hash(...);
}
``` [3](#0-2) 

The code itself acknowledges the danger of excluding L1 transactions in `process_l1_transaction.rs`:

> *"This function handles cases that for L2 transactions would be validation errors, as **invalidating** an L1 transaction can halt the chain (due to the priority queue)."* [4](#0-3) 

Yet the block-limits check is the one place where an L1 priority transaction **is** effectively invalidated, with no fallback.

The `priority_operations_hash` committed to the batch public input is derived exclusively from transactions that pass the gate:

```rust
let (priority_operations_hash, number_of_layer_1_txs) =
    block_data.enforced_transaction_hashes_accumulator.finish();
``` [5](#0-4) 

A priority transaction excluded by block limits is absent from this hash, so the L1 contract's priority-queue drain check will never see it processed.

---

### Impact Explanation

An L1 priority transaction that permanently exceeds any block resource limit can never be included in any block. Because the L1 `DiamondProxy` / `Mailbox` contract requires the priority queue to be drained in order (the `priorityOperationsHash` must match), a single stuck transaction halts all subsequent L2 state progression. This is a **chain-halting denial-of-service** reachable by any unprivileged L1 caller.

Secondary impact: even if the chain does not halt immediately (e.g., the operator skips the transaction in some off-chain workaround), the `enforced_transaction_hashes_accumulator` diverges from the L1 queue, making the batch proof invalid and preventing settlement.

---

### Likelihood Explanation

**Medium.** Two concrete trigger paths exist:

1. **Gas-limit overflow of block gas limit.** An attacker submits an L1→L2 priority transaction with `gas_limit` larger than the current L2 block gas limit. The L1 `Mailbox` contract validates `gas_limit` against a stored `l2GasLimit` cap, but that cap is set at deployment time and may not track dynamic changes to the L2 block gas limit. The code comment explicitly notes this gap: *"the validation errors are practically unreachable, as gas_limit … are either checked or set by the L1 contracts … We decide to handle these cases as a fallback in case the L1 contracts aren't properly updated."* [4](#0-3)  If the L2 block gas limit is ever reduced (e.g., via a system upgrade), previously-valid `gas_limit` values become permanently unprocessable.

2. **Pubdata exhaustion.** An attacker submits an L1 priority transaction that calls a contract performing many `SSTORE` operations. The resulting pubdata can exceed `system.get_pubdata_limit()`. The pubdata limit is not validated on L1 at submission time, so there is no L1-side guard. [6](#0-5) 

Both paths require only an L1 transaction submission — permissionless and low-cost.

---

### Recommendation

1. **Short term:** In `check_for_block_limits` (or at the call site in `tx_loop.rs`), add an exemption for transactions where `tx_processing_result.is_priority_tx` is `true`. Priority transactions must be committed regardless of block resource usage; the block should be sealed after them if limits are exceeded, but the transaction itself must be recorded.

2. **Long term:** Enforce on L1 that `gas_limit` submitted in a priority transaction cannot exceed the current L2 block gas limit, and add an L1-side pubdata budget cap per priority transaction. Ensure these caps are updated atomically with any L2 block-limit changes.

---

### Proof of Concept

**Scenario — pubdata exhaustion:**

1. Attacker deploys a contract on L2 that writes to `N` storage slots in a single call, where `N` is chosen so that `N * 64 bytes > block_pubdata_limit`.
2. Attacker submits an L1→L2 priority transaction (via `Mailbox.requestL2Transaction`) calling that contract with sufficient `gas_limit`.
3. The sequencer places the priority transaction in a block (possibly first).
4. `process_l1_transaction` executes the call; the contract writes `N` slots.
5. Back in `tx_loop.rs`, `check_for_block_limits` evaluates `next_block_pubdata_used > system.get_pubdata_limit()` → `Err(BlockPubdataLimitReached)`.
6. `system.finish_global_frame(Some(&pre_tx_rollback_handle))` reverts all state changes.
7. `result_keeper.tx_processed(Err(err))` is called; the transaction hash is **never** added to `enforced_transaction_hashes_accumulator`.
8. The priority queue on L1 still contains the transaction. Every subsequent block that attempts to process it hits the same limit. The `priorityOperationsHash` in every batch proof diverges from the L1 queue state.
9. The L1 contract rejects all batch commitments → chain halted. [1](#0-0) [6](#0-5) [3](#0-2)

fn check_for_block_limits<S: EthereumLikeTypes>(
    system: &mut System<S>,
    gas_used: u64,
    computational_native_used: u64,
    pubdata_used: u64,
    logs_used: u64,
    blob_gas_used: u64,
) -> Result<(), InvalidTransaction>
where
    S::IO: IOSubsystemExt,
    <S as SystemTypes>::Metadata: ZkSpecificPricingMetadata,
{
    if gas_used > system.get_gas_limit() {
        system_log!(
            system,
            "Block gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockGasLimitReached)
    } else if blob_gas_used > system.get_blob_gas_limit() {
        system_log!(
            system,
            "Block blob gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockBlobGasLimitReached)
    } else if !cfg!(feature = "resources_for_tester")
        && computational_native_used > MAX_NATIVE_COMPUTATIONAL
    {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block native limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockNativeLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && pubdata_used > system.get_pubdata_limit() {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block pubdata limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockPubdataLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block logs limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
    } else {
        Ok(())
    }
}
