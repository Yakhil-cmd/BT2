## Analysis

The question identifies a concrete index-mismatch bug in `GetExecutionOutcome` across a resharding epoch boundary. Let me trace the exact code path.

### Code Path

In `view_client_actor.rs`, the handler computes `target_shard_index` **once** from the old epoch's shard layout, then calls `get_next_block_hash_with_new_chunk` which may return a block in the new epoch with a different shard layout: [1](#0-0) 

The critical sequence:
1. `epoch_id` ŌåÉ old epoch (execution block's epoch)
2. `shard_layout` ŌåÉ **old** shard layout
3. `target_shard_index` ŌåÉ `shard_layout.get_shard_index(target_shard_id)` ŌĆö **old index, never updated**
4. `get_next_block_hash_with_new_chunk` returns `(h, new_child_shard_id)` ŌĆö block `h` is in the **new epoch**
5. `outcome_roots` ŌåÉ chunks of block `h` ŌĆö indexed by **new layout's shard indices**
6. `merklize(&outcome_roots).1[target_shard_index]` ŌĆö **old index applied to new layout's array** [2](#0-1) 

`get_next_block_hash_with_new_chunk` correctly updates `shard_ids` to the children when crossing an epoch boundary, and returns the first child with a new chunk. But the caller never recomputes `target_shard_index` for the new layout.

### Concrete Mismatch Scenario

Suppose old layout (V1): shard_ids = `[0, 1, 2, 3]`, indices = `[0, 1, 2, 3]`. Shard **1** is split into children **4** and **5** (new IDs = max+1, max+2).

New layout (V3): shard_ids = `[0, 2, 3, 4, 5]` (sorted), indices = `0ŌåÆ0, 2ŌåÆ1, 3ŌåÆ2, 4ŌåÆ3, 5ŌåÆ4`.

- `target_shard_index = 1` (old layout, for old shard 1)
- `get_next_block_hash_with_new_chunk` returns `(h, 4)` (child shard 4, index **3** in new layout)
- `outcome_roots` has 5 entries; `outcome_roots[3]` = child shard 4's `prev_outcome_root` = parent shard 1's outcome root
- But code uses `outcome_roots[1]` = **shard 2's** `prev_outcome_root` (a completely different shard)
- `merklize(&outcome_roots).1[1]` is the Merkle path for shard 2's outcome root

The bounds check at line 1174 does **not** catch this: `1 < 5` passes, but the wrong slot is used. [3](#0-2) 

### Proof Verification Failure

The light client verifies:
```
shard_outcome_root = compute_root(hash(execution_outcome), outcome_proof.proof)
block_outcome_root = compute_root(hash(shard_outcome_root), outcome_root_proof)
assert block_outcome_root == block_header_lite.inner_lite.outcome_root
```

The `outcome_proof.proof` (inner proof) correctly proves the execution outcome is in **shard 4's** outcome root (`outcome_roots[3]`). But `outcome_root_proof` is the Merkle path for **shard 2's** outcome root (`outcome_roots[1]`). These don't chain: the inner proof resolves to `outcome_roots[3]`, but the outer proof is anchored at `outcome_roots[1]`. Verification fails.

### Trigger Condition

This only fires when a **non-last** shard is split (so the parent's old index doesn't coincide with a child's new index). The dynamic resharding algorithm picks the shard with highest memory usage ŌĆö any shard, not necessarily the last. [4](#0-3) 

An unprivileged user triggers this by:
1. Submitting a transaction/receipt that executes in the shard selected for splitting
2. Waiting for the epoch boundary
3. Calling `EXPERIMENTAL_light_client_proof` (which calls `GetExecutionOutcome` internally) [5](#0-4)

### Title
Stale `target_shard_index` from old shard layout produces unverifiable `outcome_root_proof` across resharding epoch boundary ŌĆö (`chain/client/src/view_client_actor.rs`)

### Summary
`GetExecutionOutcome` computes `target_shard_index` from the old epoch's shard layout before calling `get_next_block_hash_with_new_chunk`. When that function returns a block in the new epoch (after a shard split), the `outcome_roots` vector is indexed by the new layout's shard indices. Using the stale old index to select the Merkle path produces an `outcome_root_proof` that does not verify against the new block's outcome root.

### Finding Description
In `view_client_actor.rs` lines 1155ŌĆō1158, `target_shard_index` is computed once from the old epoch's `shard_layout`. After `get_next_block_hash_with_new_chunk` returns `(h, new_child_shard_id)` at line 1163, the code updates `outcome_proof.block_hash` to `h` (new epoch block) and collects `outcome_roots` from that block's chunks (new layout, more shards). The stale `target_shard_index` is then used at line 1182 to select the Merkle path from the new layout's `outcome_roots`. When the parent shard's old index does not coincide with the child shard's new index (which occurs whenever a non-last shard is split), the selected path is for the wrong shard's outcome root.

### Impact Explanation
The returned `outcome_root_proof` (`MerklePath`) does not verify against the block's `outcome_root`. Any light client or bridge contract that calls `EXPERIMENTAL_light_client_proof` and verifies the proof will reject it. This breaks the client-proof invariant: the RPC is supposed to return a self-consistent, verifiable proof bundle.

### Likelihood Explanation
Requires dynamic resharding to be active and a non-last shard to be selected for splitting. The resharding algorithm picks the highest-memory shard, which can be any shard. Once resharding is live on mainnet, any user with a receipt in the split shard can trigger this by querying the proof after the epoch boundary.

### Recommendation
After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` using the **new epoch's** shard layout:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```

### Proof of Concept
A test-loop test with resharding enabled should:
1. Configure a 4-shard layout where shard 1 (not the last) is forced to split.
2. Submit a transaction whose receipt executes in shard 1.
3. Advance past the epoch boundary.
4. Call `GetExecutionOutcome` for the receipt.
5. Assert `verify_path(block_outcome_root, &outcome_root_proof, &chunk_outcome_root)` ŌĆö this assertion will **fail** with the current code, demonstrating the bug.

The existing `check_outcome_proofs` helper in `test-loop-tests/src/tests/light_client.rs` provides the verification pattern; it just needs to be run across a resharding boundary where a non-last shard is split. [6](#0-5)

    pub fn get_next_block_hash_with_new_chunk(
        &self,
        block_hash: &CryptoHash,
        shard_id: ShardId,
    ) -> Result<Option<(CryptoHash, ShardId)>, Error> {
        let mut block_hash = *block_hash;
        let mut epoch_id = *self.get_block_header(&block_hash)?.epoch_id();
        let mut shard_layout = self.epoch_manager.get_shard_layout(&epoch_id)?;
        // this corrects all the shard where the original shard will split to if sharding changes
        let mut shard_ids = vec![shard_id];

        while let Ok(next_block_hash) = self.chain_store.get_next_block_hash(&block_hash) {
            let next_epoch_id = *self.get_block_header(&next_block_hash)?.epoch_id();
            if next_epoch_id != epoch_id {
                let next_shard_layout = self.epoch_manager.get_shard_layout(&next_epoch_id)?;
                if next_shard_layout != shard_layout {
                    shard_ids = shard_ids
                        .into_iter()
                        .flat_map(|id| {
                            next_shard_layout.get_children_shards_ids(id).unwrap_or_else(|| {
                                panic!("invalid shard layout {:?} because it does not contain children shards for parent shard {}", next_shard_layout, id)
                            })
                        })
                        .collect();

                    shard_layout = next_shard_layout;
                }
                epoch_id = next_epoch_id;
            }
            block_hash = next_block_hash;

            let block = self.get_block(&block_hash)?;
            let chunks = block.chunks();
            for &shard_id in &shard_ids {
                let shard_index = shard_layout.get_shard_index(shard_id)?;
                let chunk_header =
                    &chunks.get(shard_index).ok_or(Error::InvalidShardId(shard_id))?;
                if chunk_header.height_included() == block.header().height() {
                    return Ok(Some((block_hash, shard_id)));
                }

        &self,
        protocol_version: ProtocolVersion,
        parent_hash: &CryptoHash,
        chunk_headers: &[ShardChunkHeader],
    ) -> Result<Option<(ShardId, AccountId)>, EpochError> {
        // Check if dynamic resharding is enabled
        let epoch_config = self.get_epoch_config(protocol_version);
        let dynamic_resharding_config = match &epoch_config.shard_layout_config {
            ShardLayoutConfig::Static { .. } => return Ok(None),
            ShardLayoutConfig::Dynamic { dynamic_resharding_config } => dynamic_resharding_config,
        };

        // Check if resharding is allowed based on epoch constraints
        let can_reshard = self
            .can_reshard(&parent_hash, dynamic_resharding_config.min_epochs_between_resharding)?;
        if !can_reshard {
            return Ok(None);
        }

        // Collect proposed splits from chunk headers
        let mut proposed_splits = HashMap::new();
        for chunk_header in chunk_headers {
            if let Some(split) = chunk_header.proposed_split() {
                proposed_splits.insert(chunk_header.shard_id(), split.clone());
            }
        }

        // Pick the shard to split
        let Some((shard_id, split)) =
            pick_shard_to_split(&proposed_splits, dynamic_resharding_config)
        else {
            return Ok(None);
        };

        Ok(Some((shard_id, split.boundary_account)))
    }

fn check_outcome_proofs(
    env: &mut TestLoopEnv,
    account_id: &AccountId,
    seed_hash: CryptoHash,
    tx: SignedTransaction,
) {
    let outcome = env.rpc_runner().execute_tx(tx, Duration::seconds(5)).unwrap();
    // Advance so the outcome's block is final and included in later blocks' merkle roots.
    env.rpc_runner().run_for_number_of_blocks(4);

    let mut ids = vec![TransactionOrReceiptId::Transaction {
        transaction_hash: outcome.transaction_outcome.id,
        sender_id: account_id.clone(),
    }];
    for receipt_outcome in &outcome.receipts_outcome {
        ids.push(TransactionOrReceiptId::Receipt {
            receipt_id: receipt_outcome.id,
            receiver_id: account_id.clone(),
        });
    }

    let mut rpc = env.rpc_node_mut();
    let view_client = rpc.view_client_actor();

    // The light client head and the block merkle root all proofs are checked against.
    let light_client_block = view_client
        .handle(GetNextLightClientBlock { last_block_hash: seed_hash })
        .unwrap()
        .unwrap();
    let light_client_head = light_client_block_hash(&light_client_block);
    let block_merkle_root = light_client_block.inner_lite.block_merkle_root;

    for id in ids {
        let execution_outcome = view_client.handle(GetExecutionOutcome { id }).unwrap();
        let outcome_proof = execution_outcome.outcome_proof;
        let block_proof = view_client
            .handle(GetBlockProof {
                block_hash: outcome_proof.block_hash,
                head_block_hash: light_client_head,
            })
            .unwrap();

        // outcome -> chunk outcome root -> block outcome root.
        let chunk_outcome_root =
            compute_root_from_path_and_item(&outcome_proof.proof, &outcome_proof.to_hashes());
        assert!(verify_path(
            block_proof.block_header_lite.inner_lite.outcome_root,
            &execution_outcome.outcome_root_proof,
            &chunk_outcome_root,
        ));

        // The light block header recomputes to the proof's block hash.
        assert_eq!(block_proof.block_header_lite.hash(), outcome_proof.block_hash);

        // block hash -> light client head's block merkle root.
        assert!(verify_hash(block_merkle_root, &block_proof.proof, outcome_proof.block_hash));
    }
