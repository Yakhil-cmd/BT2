### Title
Malicious CKD Participant Can Corrupt Derived Key Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates participant share contributions `(norm_big_y, norm_big_c)` with no proof of correct computation and no threshold subset selection. Any single malicious participant can send arbitrary group elements, causing the coordinator to produce a CKD output that does not equal `msk · H(pk ∥ app_id)`, permanently denying the client a usable confidential derived key.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and unconditionally adds them together: [1](#0-0) 

There is no zero-knowledge proof, no commitment-then-reveal, and no consistency check that verifies a participant actually computed:

```
big_s  = H(pk ∥ app_id) · x_i
big_c  = big_s + app_pk · y_i
big_y  = G · y_i
```

using their real secret share `x_i` and a self-chosen ephemeral `y_i`.

The correct aggregated output relies on Lagrange interpolation over **all** participants: [2](#0-1) 

Because every participant's Lagrange-weighted contribution is required for the sum to equal `msk · H(pk ∥ app_id) + app_pk · Y`, a single participant sending `(0, 0)` (identity elements) or any other arbitrary pair breaks the interpolation. The resulting `C − app_sk · Y` will not equal `msk · H(pk ∥ app_id)`.

Unlike the DKG protocol, which uses echo broadcast and proofs of knowledge to detect misbehaving parties: [3](#0-2) 

the CKD protocol has no analogous verification step. The `ckd()` entry point also accepts no threshold parameter and requires all `n` participants to contribute correctly: [4](#0-3) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A single malicious participant causes the coordinator to return a `CKDOutput` whose `unmask(app_sk)` value is not `msk · H(pk ∥ app_id)`. The client receives a wrong derived key. Because the protocol requires all `n` participants and provides no way to identify or exclude the cheating party, honest parties cannot recover a correct output without restarting with a different participant set — and even then, the malicious participant can repeat the attack.

---

### Likelihood Explanation

Any participant in the CKD protocol can perform this attack with zero cryptographic effort: they simply send `(G1::identity(), G1::identity())` or any other arbitrary pair instead of their correctly computed share. No special knowledge, no key material, and no coordination with other parties is required. The attack is silent — the coordinator has no way to distinguish a correct share from a forged one.

---

### Recommendation

1. **Add a proof of correct share computation.** Each participant should accompany `(norm_big_y, norm_big_c)` with a Schnorr-style NIZK proving knowledge of `(x_i, y_i)` such that `big_s = H(pk ∥ app_id) · x_i` (consistent with the public commitment to `x_i` from DKG) and `big_c = big_s + app_pk · y_i`, `big_y = G · y_i`.

2. **Alternatively, adopt a threshold aggregation model** (collect `t`-of-`n` shares, verify each against the public commitment from DKG, and interpolate), mirroring the robustness design of the DKG and ECDSA presign protocols.

---

### Proof of Concept

```
Setup:
  - 3 participants: P1 (honest), P2 (honest), P3 (malicious), coordinator = P1
  - All run ckd() with the same app_id, app_pk, and key_pair from a prior DKG

Attack:
  P3 overrides compute_signature_share() to return (G1::identity(), G1::identity())
  and sends (identity, identity) to the coordinator.

Coordinator aggregation (do_ckd_coordinator, lines 50-55):
  norm_big_y = λ1·y1·G  +  λ2·y2·G  +  0          ← missing λ3·y3·G
  norm_big_c = λ1·big_c1 + λ2·big_c2 + 0           ← missing λ3·big_c3

Expected:
  C - app_sk · Y  =  msk · H(pk ∥ app_id)

Actual:
  C' - app_sk · Y'  ≠  msk · H(pk ∥ app_id)
  (Lagrange interpolation is broken; the derived key is garbage)

Result:
  Client receives a wrong confidential key with no error or indication of failure.
```
