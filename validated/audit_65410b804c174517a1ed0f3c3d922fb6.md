### Title
Malicious Participant Causes Inconsistent Public Keys via Echo-Broadcast Threshold Collapse in 3-Party DKG — (`src/dkg.rs::public_key_from_commitments`)

---

### Summary

In a 3-party DKG/reshare/refresh run (`n=3, threshold=2`), a single malicious participant M can cause the two honest parties A and B to each compute a **different** verifying key from `public_key_from_commitments`, both passing all local validation checks. The protocol then completes successfully for both honest parties, leaving them with permanently inconsistent key material that can never produce a valid threshold signature.

---

### Finding Description

The attack exploits three independent weaknesses that compose into a complete break:

**Weakness 1 — Echo-broadcast thresholds collapse to zero for n ≤ 3.**

`echo_ready_thresholds` in `src/protocol/echo_broadcast.rs` returns `(0, 0)` for any `n ≤ 3`: [1](#0-0) 

With `echo_t = 0` and `ready_t = 0`, the delivery condition `count > 0` is satisfied by a single echo from the participant itself. A malicious sender M can transmit `Send(commitment_M_A)` to A and `Send(commitment_M_B)` to B. A echoes `commitment_M_A` (count 1 > 0 → delivers), B echoes `commitment_M_B` (count 1 > 0 → delivers). No error is raised; A and B deliver different values for M's broadcast session.

**Weakness 2 — Commitment hash is sent via plain `send_many`, not echo-broadcast.**

The commitment hash round uses `chan.send_many` (a simple, non-consistent broadcast): [2](#0-1) 

M can therefore send `hash_A = H(M, commitment_M_A, session_id)` to A and `hash_B = H(M, commitment_M_B, session_id)` to B with no consistency requirement. Each honest party stores a different hash for M.

**Weakness 3 — `broadcast_success` checks only `session_id`, not the derived public key.** [3](#0-2) 

If M sends the **same** session-ID nonce to A and B in Round 1 (so both compute the same `session_id`), then `broadcast_success` sees `(true, session_id)` from every participant and passes for both A and B, even though they hold different public keys.

**`public_key_from_commitments` is context-free and sums whatever it receives.** [4](#0-3) 

It takes no session_id, no participant list, no transcript. It simply sums the commitments it is given. Because A and B were given different commitments for M (via the two weaknesses above), they compute different verifying keys.

---

### Impact Explanation

Both honest parties complete `do_keyshare` successfully and return a `KeygenOutput`. Their public keys differ:

```
pk_A = commitment_A + commitment_B + commitment_M_A
pk_B = commitment_A + commitment_B + commitment_M_B
```

Their private shares are each valid against their respective (different) public keys. Any subsequent signing attempt between A and B will fail because they disagree on the group public key. The corruption is permanent and silent — neither party receives an error.

This matches: **High — Corruption of DKG outputs so honest parties accept inconsistent public keys.**

---

### Likelihood Explanation

- Requires exactly `n = 3` participants (the minimum that allows 2 honest parties with 1 malicious).
- The malicious party needs only to send two different commitments and two different commitment hashes — straightforward message-level manipulation, no cryptographic break required.
- The `assert_key_invariants` function permits `n = 3, threshold = 2` without restriction. [5](#0-4) 

---

### Recommendation

1. **Raise echo-broadcast thresholds for small n**: The comment "no malicious parties are assumed: when n ≤ 3" is not enforced at the DKG layer. Either enforce `n ≥ 4` in `assert_key_invariants`, or use a different consistency mechanism for small groups.
2. **Send commitment hashes via echo-broadcast**, not plain `send_many`, so M cannot equivocate on the hash.
3. **Add a public-key consistency check in `broadcast_success`**: broadcast the derived `verifying_key` alongside `session_id` and abort if any honest party reports a different key.

---

### Proof of Concept

**Setup**: `n=3`, `threshold=2`, participants `{A, B, M}` where M is malicious.

**Round 1 (session IDs via `do_broadcast`)**: M sends the **same** nonce to A and B → both compute identical `session_id`.

**Round 1b (commitment hash via `send_many`)**: M computes two polynomials `f_A`, `f_B` with different constant terms. M sends:
- `hash_A = H(M, commitment(f_A), session_id)` to A
- `hash_B = H(M, commitment(f_B), session_id)` to B

**Round 2 (commitment + PoK via `do_broadcast`)**: M sends `Send(commitment(f_A), PoK_A)` to A and `Send(commitment(f_B), PoK_B)` to B. With `echo_t = 0`, A delivers `commitment(f_A)` and B delivers `commitment(f_B)`.

**Verification (lines 452–469)**: Both A and B pass `verify_proof_of_knowledge` and `verify_commitment_hash` — each commitment matches its respective hash, and each PoK is valid for its commitment under the shared `session_id`.

**`public_key_from_commitments` (line 485)**:
- A: `pk_A = commitment_A + commitment_B + commitment(f_A)`
- B: `pk_B = commitment_A + commitment_B + commitment(f_B)`

**Share round**: M sends `f_A(id_A)` to A and `f_B(id_B)` to B. Both pass `validate_received_share`.

**`broadcast_success` (line 531)**: M broadcasts `(true, session_id)`. Since `session_id` is identical for A and B, all checks pass. Both parties return `Ok(KeygenOutput { public_key: pk_A })` and `Ok(KeygenOutput { public_key: pk_B })` respectively — different keys, no error.

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

```rust
fn public_key_from_commitments<C: Ciphersuite>(
    commitments: Vec<&VerifiableSecretSharingCommitment<C>>,
) -> Result<VerifyingKey<C>, ProtocolError> {
    let commitment = frost_core::keys::sum_commitments(&commitments)
        .map_err(|_| ProtocolError::IncorrectNumberOfCommitments)?;

    let vk = VerifyingKey::from_commitment(&commitment)
        .map_err(|_| ProtocolError::ErrorExtractVerificationKey)?;
    Ok(vk)
}
```

```rust
pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
```
