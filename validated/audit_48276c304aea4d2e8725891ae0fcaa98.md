### Title
Wrong `BehaviourPenaltyThreshold` Value in GossipSub Scoring Configuration - (`config/default-config.yml`)

### Summary

The `behaviour.penalty-threshold` in `config/default-config.yml` is set to `1000`, but every piece of documentation, code comment, integration test, and the README all state the intended value is `10`. This is a direct analog to the LybraGovernance wrong-threshold bug: the configured numeric value is 100× larger than the intended/documented value, causing the iHave broken-promise and GRAFT-flood protection to be effectively disabled for any realistic attacker.

### Finding Description

In `config/default-config.yml`, the `behaviour.penalty-threshold` field under `scoring-parameters.peer-scoring.internal` is set to `1000`:

```yaml
behaviour:
  # We set it to 10, meaning that we at most tolerate 10 of such RPCs...
  penalty-threshold: 1000   # <-- BUG: should be 10
  penalty-weight: -0.01
  penalty-decay: 0.5
```

The inline comment on the same block explicitly states `"We set it to 10"`. Every other authoritative source in the codebase agrees the intended value is `10`:

- `network/p2p/scoring/README.md` line 193: `BehaviourPenaltyThreshold is set to defaultBehaviourPenaltyThreshold, i.e., 10`
- `network/p2p/scoring/README.md` line 199: `This threshold is set to 10, meaning that we at most tolerate 10 such RPCs`
- `insecure/integration/functional/test/gossipsub/scoring/ihave_spam_test.go` line 74: `conf.NetworkConfig.GossipSub.ScoringParameters.PeerScoring.Internal.Behaviour.PenaltyThreshold = 10`
- `network/p2p/pubsub.go` line 256: `Setting it to 20 to reduce the noise; 20 is twice the threshold (defaultBehaviourPenaltyThreshold)`

The `penalty-threshold` value is loaded from `default-config.yml` at startup via mapstructure into `InternalScoringBehavioural.PenaltyThreshold` and then passed directly to GossipSub as `BehaviourPenaltyThreshold` in `NewScoreOption`.

The penalty formula is: `-|w| * (misbehavior_counter - threshold)^2`. With `threshold = 1000`, a peer must send **1000 iHave broken-promise RPCs** before any penalty is applied at all. With the correct value of `10`, only 10 such RPCs are tolerated. The entire iHave broken-promise and GRAFT-flood mitigation system is therefore inoperative under the default configuration.

Additionally, `penalty-decay` is set to `0.5` in the config, while the inline comment and README both state the intended value is `0.99`. With `decay = 0.5`, the misbehavior counter decays by 50% per minute instead of 1%, meaning even the 1000-count threshold is reset extremely fast, further weakening the protection.

### Impact Explanation

An unprivileged, unstaked Access/Observer node or any peer reachable via the GossipSub layer can mount an iHave broken-promise attack or GRAFT-flood attack without being penalized or graylisted. The GossipSub-level protection that is supposed to disconnect such peers after ~10 misbehaving RPCs is completely bypassed because the threshold is 100× too high. The attacker can sustain the attack indefinitely, causing resource exhaustion (redundant iWant requests) and degrading message propagation across the network.

### Likelihood Explanation

Any peer connected to the Flow GossipSub network can trivially trigger this path by advertising iHave messages and not responding to iWant requests. No privileged access is required. The default configuration is loaded by every node at startup, so all deployed nodes are affected.

### Recommendation

In `config/default-config.yml`, correct the two wrong values under `behaviour`:

```yaml
behaviour:
  penalty-threshold: 10     # was 1000
  penalty-weight: -0.01
  penalty-decay: 0.99       # was 0.5
```

These values match the documented intent in `network/p2p/scoring/README.md`, the inline comments, and the integration test overrides.

#### 1. `defaultBehaviourPenaltyThreshold`
This parameter sets the threshold for when the behavior of a peer is considered bad. Misbehavior is defined as advertising an iHave without responding to the iWants (iHave broken promises), and attempting on GRAFT when the peer is considered for a PRUNE backoff.
If a remote peer sends an RPC that advertises at least one iHave for a message but doesn't respond to the iWant requests for that message within the next `3 seconds`, the peer misbehavior counter is incremented by `1`. This threshold is set to `10`, meaning that we at most tolerate 10 such RPCs containing iHave broken promises. After this, the peer is penalized for every excess RPC containing iHave broken promises. The counter decays by (`0.99`) every decay interval (defaultDecayInterval) i.e., every minute.
