### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` — Ownership Proof Provides No Address-Binding Guarantee - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's keys signed some `signedData`, then separately checks that a COA at a caller-supplied `path` has the claimed `evmAddress`. Because the function never enforces that `signedData` encodes `evmAddress`, a signature produced to prove ownership of COA_A can be replayed verbatim — with only the `CapabilityPath` field changed — to pass ownership validation for a different COA_B owned by the same Cadence account. The function's own inline comment acknowledges this gap. The ERC-1271 entry point in `coa.sol` exposes this to any EVM contract that calls `isValidSignature`.

---

### Finding Description

**Root cause — `fvm/evm/stdlib/contract.cdc` lines 1001–1116**

`validateCOAOwnershipProof` performs two independent checks:

1. Verify `signatures` over `signedData` using the Cadence account's keys (lines 1082–1093).
2. Borrow the COA capability at the caller-supplied `path` and compare its EVM address to `evmAddress` (lines 1095–1110).

There is no step that binds `signedData` to `evmAddress`. The function itself documents this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The `COAOwnershipProof` struct (encoded in the `_sig` bytes passed to the precompile) contains `CapabilityPath` as an attacker-controlled field: [2](#0-1) 

The precompile decodes the proof and passes `CapabilityPath` directly to `validateCOAOwnershipProof` without any binding to the `evmAddress` argument: [3](#0-2) 

The ERC-1271 entry point in `coa.sol` calls the precompile with `address(this)` as the EVM address and the caller-supplied `_sig` as the proof blob: [4](#0-3) 

**Exploit path:**

Suppose a Cadence account owns two COAs: COA_A published at `/public/coaA` and COA_B published at `/public/coaB`.

1. A legitimate interaction causes the account holder to produce a valid proof for COA_A: `{KeyIndices:[0], Address:cadenceAcct, CapabilityPath:"coaA", Signatures:[sig_D]}` over `hash_D`.
2. An attacker observes this proof (e.g., from an on-chain transaction or an ERC-1271 call).
3. The attacker constructs a modified proof: `{KeyIndices:[0], Address:cadenceAcct, CapabilityPath:"coaB", Signatures:[sig_D]}` — identical except `CapabilityPath` now points to COA_B.
4. The attacker calls `verifyCOAOwnershipProof(COA_B_address, hash_D, modified_proof)` from any EVM contract (or via `isValidSignature` on COA_B).
5. `validateCOAOwnershipProof` is invoked with `path="coaB"` and `evmAddress=COA_B_address`.
6. Step 1 (signature check): `sig_D` is a valid signature over `hash_D` under the Cadence account's key 0 → **passes**.
7. Step 2 (address check): the COA at `/public/coaB` has EVM address `COA_B_address` → **passes**.
8. The function returns `ValidationResult(isValid: true)`.

The attacker has forged a COA ownership proof for COA_B using a signature the victim produced for COA_A. [5](#0-4) 

---

### Impact Explanation

Any EVM contract that uses ERC-1271 (`isValidSignature`) on a COA, or directly calls `verifyCOAOwnershipProof`, to gate privileged actions (token transfers, multisig approvals, DAO votes, bridge withdrawals) can be bypassed. An attacker who observes a single valid proof for any COA under a target Cadence account can replay it against every other COA that account owns, impersonating the account holder's authorization for those addresses without possessing the private key. The impact is unauthorized EVM-side authorization — equivalent to forging a signature — for all COAs beyond the first one under a given Cadence account.

---

### Likelihood Explanation

The precondition is that the victim Cadence account holds more than one COA. The comment in the source code dismisses this as "low-risk" because "Flow wallets historically create at most one COA per account," but this is a social/historical observation, not a protocol invariant. `createCadenceOwnedAccount()` is unrestricted; any dapp, bridge, or smart-contract wallet that creates multiple COAs per account (e.g., one per EVM chain role, one per asset class) is immediately vulnerable. The attacker needs only to observe one valid proof — which is public on-chain data — and modify the `CapabilityPath` field in the RLP-encoded proof blob before replaying it.

---

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` commits to `evmAddress`. The simplest fix is to require that `signedData` is the ABI- or RLP-encoding of `evmAddress` (or a hash thereof), and reject the proof if it does not. Alternatively, the function can hash `signedData || evmAddress` before passing it to `keyList.verify`, so that a signature produced for one EVM address is cryptographically unusable for any other. The ERC-1271 wrapper in `coa.sol` should pass `keccak256(abi.encode(_hash, address(this)))` as the signed data rather than `_hash` alone.

### Proof of Concept

```cadence
// Attacker script — no privileged access required
// Precondition: cadenceAcct owns COA_A at /public/coaA and COA_B at /public/coaB
// Attacker has observed a valid proof for COA_A: (hash_D, proof_A)

import EVM from 0x...

access(all) fun main() {
    // proof_A = {KeyIndices:[0], Address:cadenceAcct, CapabilityPath:"coaA", Signatures:[sig_D]}
    // Attacker replaces CapabilityPath with "coaB" — everything else identical
    let proof_B = EVM.COAOwnershipProof(
        keyIndices: [0],
        address: cadenceAcct,
        capabilityPath: "coaB",   // <-- only change
        signatures: [sig_D]       // same signature as proof_A
    )

    // validateCOAOwnershipProof passes:
    // - sig_D is valid over hash_D under cadenceAcct key 0  ✓
    // - COA at /public/coaB has EVM address COA_B_address   ✓
    let result = EVM.validateCOAOwnershipProof(
        address: cadenceAcct,
        path: /public/coaB,
        signedData: hash_D,
        keyIndices: proof_B.keyIndices,
        signatures: proof_B.signatures,
        evmAddress: COA_B_address
    )
    assert(result.isValid == true)  // succeeds — ownership of COA_B forged
}
```
