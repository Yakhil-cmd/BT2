### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` applies no cap check against `maxAdminSpreadFeeE6`, while the parallel function `setPoolAdminFees` does. A pool admin can set `addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` on every bin, bypassing whatever cap the factory owner has configured, and making the pool economically unusable for traders.

---

### Finding Description

`setPoolAdminFees` enforces the factory cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no such check — it passes values straight through to the pool:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index range, not the fee values:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6) external onlyFactory ... {
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

In swap math, the per-bin fee is **additive** on top of the oracle-derived base fee:

```solidity
// MetricOmmPool.sol:999
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so the maximum settable value is **65535** (≈ 6.5535% additional per bin). The factory owner can lower `maxAdminSpreadFeeE6` to any value (e.g. 1000 = 0.1%) via `setFeeCaps`, but the pool admin can still set per-bin fees to 65535 via `setPoolBinAdditionalFees`, completely ignoring the cap. [5](#0-4) 

---

### Impact Explanation

The factory's `maxAdminSpreadFeeE6` cap is the mechanism that bounds how much a semi-trusted pool admin can charge traders. Per-bin fees bypass this cap entirely. A pool admin can set all bins to `addFeeBuyE6 = addFeeSellE6 = 65535`, making effective swap costs far exceed what the factory owner intended to allow. Traders executing swaps receive significantly less than oracle-permitted output. This is a confirmed admin-boundary break: the pool admin exceeds the factory-enforced cap through an unchecked code path.

---

### Likelihood Explanation

Any pool admin — including one who created a pool specifically to exploit this — can call `setPoolBinAdditionalFees` at any time with no timelock, no prior collection step, and no cap guard. The call is immediate and affects all bins in a single loop of transactions.

---

### Recommendation

Add the same cap guard in `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`. Since `addFeeBuyE6` and `addFeeSellE6` are `uint16` and `maxAdminSpreadFeeE6` is `uint24`, the comparison is straightforward:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 1000` (0.1%) via `setFeeCaps`.
2. Pool admin calls `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` for every bin — no revert.
3. A trader executes a swap; the effective fee applied is `baseFee + 65535/1e6` (≈ 6.5535% additional), far exceeding the 0.1% cap.
4. Assert: trader receives significantly less than oracle-permitted output; `maxAdminSpreadFeeE6` invariant is violated.
