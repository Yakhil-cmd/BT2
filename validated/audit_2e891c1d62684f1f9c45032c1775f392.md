### Title
Unauthenticated Sender Causes Protocol Abort via Unguarded `?` in `reliable_broadcast_receive_all` — (`src/protocol/echo_broadcast.rs`)

---

### Summary

In `reliable_broadcast_receive_all`, a `MessageType::Send` message received from a participant **not in the participant list** causes the function to propagate `ProtocolError::InvalidIndex` via the `?` operator, immediately aborting the entire DKG, reshare, or refresh protocol for all honest parties. Because `Protocol::message()` accepts messages from any `Participant` without validating membership, any external party that can deliver a single crafted message can permanently abort key generation.

---

### Finding Description

`reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs` is the core broadcast primitive used by `do_broadcast`, which is called multiple times inside `do_keyshare` (the shared implementation of DKG, reshare, and refresh).

When processing a `MessageType::Send` message, the function checks whether the sender matches the session ID:

```rust
// src/protocol/echo_broadcast.rs, lines 193–205
MessageType::Send(data) => {
    // The second condition prevents a malicious party starting the protocol
    // on behalf on somebody else
    if state_sid.finish_send || sid != participants.index(from)? {
        continue;
    }
    ...
}
``` [1](#0-0) 

`participants.index(from)` returns `Err(ProtocolError::InvalidIndex)` when `from` is absent from the participant list:

```rust
// src/participants.rs, lines 135–140
pub fn index(&self, participant: Participant) -> Result<usize, ProtocolError> {
    self.indices
        .get(&participant)
        .copied()
        .ok_or(ProtocolError::InvalidIndex)
}
``` [2](#0-1) 

The `?` operator propagates this error out of `reliable_broadcast_receive_all`, aborting the protocol. The comment directly above the `recv` call states the intent is to **ignore** bad messages ("The recv should be failure-free / This translates to ignoring the received message when deemed wrong"), but this intent is not carried through to the `participants.index(from)?` call:

```rust
// src/protocol/echo_broadcast.rs, lines 175–183
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // recv errors are swallowed
};
``` [3](#0-2) 

By contrast, the `Echo` and `Ready` branches use `ParticipantCounter::put`, which silently returns `false` for unknown participants — so only the `Send` branch is vulnerable.

The entry point for injecting messages is `Protocol::message()`, which accepts any `(Participant, MessageData)` pair without membership validation:

```rust
// src/protocol/internal.rs, lines 512–514
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
``` [4](#0-3) 

The library's own test explicitly acknowledges that an attacker outside the participant set can inject messages into the buffer:

```rust
// src/protocol/internal.rs, lines 532–554
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    ...
    comms.push_message(attacker, message);
    ...
}
``` [5](#0-4) 

`do_broadcast` (and thus `reliable_broadcast_receive_all`) is called three times inside `do_keyshare`, which underlies `do_keygen`, `do_reshare`, and `do_refresh`:

```rust
// src/dkg.rs, lines 362, 435, 531
let session_ids = do_broadcast(&mut chan, &participants, me, my_session_id).await?;
...
let commitments_and_proofs_map = do_broadcast(&mut chan, &participants, me, ...).await?;
...
broadcast_success(&mut chan, &participants, me, session_id).await?;
``` [6](#0-5) [7](#0-6) [8](#0-7)

### Impact Explanation

Any party that can deliver a single message to a participant's `Protocol::message()` endpoint can permanently abort DKG, reshare, or refresh for that participant. Because the error propagates through `do_broadcast → do_keyshare → do_keygen / do_reshare`, the honest participant's protocol instance terminates with `ProtocolError::InvalidIndex` and cannot produce a `KeygenOutput`. If the attacker targets enough participants, the entire key generation or resharing ceremony fails and must be restarted — and can be aborted again immediately.

This matches: **High — Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs.**

---

### Likelihood Explanation

The `Protocol` trait is the public interface of the library. Any deployment that routes network messages to `Protocol::message()` without first filtering out non-participant senders is vulnerable. The library itself imposes no such filter. A single malicious message with a valid `sid` value (any integer in `[0, n)`) and a `MessageType::Send` payload from an unknown `Participant` ID is sufficient to trigger the abort.

---

### Recommendation

Replace the `?` operator with a graceful fallback that treats an unknown sender as a mismatch and continues:

```rust
// src/protocol/echo_broadcast.rs
MessageType::Send(data) => {
    let sender_idx = match participants.index(from) {
        Ok(idx) => idx,
        Err(_) => continue,   // unknown sender — ignore
    };
    if state_sid.finish_send || sid != sender_idx {
        continue;
    }
    ...
}
```

This aligns the `Send` branch with the existing behavior of the `Echo` and `Ready` branches, which already silently discard messages from unknown participants via `ParticipantCounter::put`.

---

### Proof of Concept

1. Start a DKG protocol with participants `[0, 1, 2]` (n = 3).
2. An attacker (e.g., `Participant(99)`, not in the list) crafts a message:
   - Header: valid `root_shared` channel tag, waitpoint matching the broadcast round.
   - Payload: `(sid=0, MessageType::Send(arbitrary_data))` — `sid=0` is in-bounds.
3. The attacker delivers this message to participant `0`'s protocol instance via `Protocol::message(Participant(99), crafted_bytes)`.
4. Inside `reliable_broadcast_receive_all`:
   - `chan.recv(wait)` succeeds, yielding `(from=Participant(99), (sid=0, MessageType::Send(...)))`.
   - `state.get_mut(0)` succeeds (sid 0 is valid).
   - `participants.index(Participant(99))` returns `Err(ProtocolError::InvalidIndex)`.
   - The `?` propagates the error; `reliable_broadcast_receive_all` returns `Err`.
5. `do_broadcast` returns `Err`, `do_keyshare` returns `Err`, and participant `0`'s DKG instance is permanently aborted. [9](#0-8) [6](#0-5)
