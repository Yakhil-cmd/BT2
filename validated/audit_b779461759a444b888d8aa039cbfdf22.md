### Title
ETH Sent to `tick()` Is Permanently Stuck When No Minting Is Required — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The L1 `RewardSupplier` contract exposes a `payable` `tick()` function intended to be called with ETH to cover StarkGate bridge message fees. When `requiredMinting()` returns zero (no pending mint requests), the entire `if (amountToMint > 0)` block is skipped and `msg.value` is never forwarded or refunded. Because the contract has no `receive()` fallback, no `withdrawETH()` function, and no refund path, any ETH sent in this scenario is permanently locked in the contract.

---

### Finding Description

`RewardSupplier.sol` declares `tick()` as `external payable`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        // ...
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // ← if amountToMint == 0, msg.value is silently consumed
}
```

When `amountToMint == 0`, the function returns without touching `msg.value`. The contract has no `receive()` or `fallback()` function, and no ETH-rescue function anywhere in the contract or its visible parent chain (`RewardSupplierStorage`, `Identity`, `ProxySupportImpl`). ETH sent in this call path is permanently irrecoverable.

This is the direct analog of the `HibernationDen` bug: a contract that can receive ETH (here via a `payable` entry point rather than a bare `receive()`) but provides no mechanism to withdraw it.

---

### Impact Explanation

Any caller who sends ETH to `tick()` when there are no pending L2 mint requests loses that ETH permanently. The contract accumulates ETH with no path to recovery. This constitutes permanent freezing of caller funds and griefing with direct financial damage to users of the protocol's operational infrastructure.

**Impact: Medium** — Griefing with no profit motive but damage to users or protocol (stuck ETH, value bounded by gas-fee amounts per call).

---

### Likelihood Explanation

`tick()` is a public, permissionless function. Any keeper, bot, or user can call it. The condition `amountToMint == 0` is the normal steady-state between reward epochs when all pending L2 mint requests have already been consumed. A caller who does not first query `requiredMinting()` off-chain, or who races against another `tick()` call that drains the pending queue, will send ETH into the void. This is a routine operational scenario.

**Likelihood: Medium** — Occurs whenever `tick()` is called with `msg.value > 0` and no mint requests are pending, which is the common case between epochs.

---

### Recommendation

Add a refund path for unused ETH inside `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund unused ETH to caller
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, add a privileged `rescueETH(address recipient)` function, or revert when `msg.value > 0` and `amountToMint == 0`.

---

### Proof of Concept

1. Observe that `requiredMinting()` returns `(0, 0)` when `messagingContract().l2ToL1Messages(msgHash) == 0` (no pending L2 mint requests).
2. Call `tick{value: 1 ether}()` in this state.
3. The `if (amountToMint > 0)` branch is skipped entirely.
4. `msg.value` (1 ETH) is now held by the contract.
5. No function exists to withdraw it — `address(rewardSupplier).balance` increases permanently. [1](#0-0) [2](#0-1)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
