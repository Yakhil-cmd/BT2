### Title
`RewardSupplier.tick()` Sends Two L1→L2 Messages When One Would Suffice, Forcing Double Message Fees - (File: L1/starkware/solidity/stake/RewardSupplier.sol)

### Summary
The `tick()` function in `RewardSupplier.sol` sends two separate L1→L2 messages in a single call: one via `bridge().depositWithMessage` (token transfer to the L2 RewardSupplier) and one via `messagingContract().sendMessageToL2` (total supply update to the L2 MintingCurve). The caller must fund both messages from `msg.value`. Since `depositWithMessage` already supports an arbitrary `message` payload delivered to the L2 recipient's `on_receive` handler, the total supply update could be piggybacked, eliminating the second fee. Every invocation of `tick()` — a mandatory protocol operation — therefore costs twice the necessary L1→L2 message fee.

### Finding Description
Inside `tick()`, after minting tokens, two outbound L1→L2 messages are created:

**Message 1** — token deposit via StarkGate bridge with an **empty** message payload:
```solidity
uint256 msgFee = msg.value / 2;
bridge().depositWithMessage{value: msgFee}(
    token(),
    amountToMint,
    mintDestination(),
    new uint256[](0)   // <-- empty, could carry totalSupply
);
```

**Message 2** — total supply update to the L2 MintingCurve:
```solidity
msgFee = msg.value - msgFee;
messagePayload[0] = IERC20(token()).totalSupply();
messagingContract().sendMessageToL2{value: msgFee}(
    mintingCurve(),
    UPDATE_TOTAL_SUPPLY_SELECTOR,
    messagePayload
);
```

The fee is split `msg.value / 2` each. The caller must supply enough ETH to cover both fees or the transaction reverts. The `depositWithMessage` interface already accepts a `uint256[] calldata message` parameter that is forwarded to the L2 recipient's `on_receive` handler. The total supply value could be included there, and the L2 `RewardSupplier.on_receive` could then propagate it to the MintingCurve on L2 (an intra-L2 call, no additional L1 fee). The second `sendMessageToL2` call would then be unnecessary. [1](#0-0) 

The L2 `on_receive` handler currently does nothing with the `message` span — it only adjusts `l1_pending_requested_amount`: [2](#0-1) 

The MintingCurve's `update_total_supply` is an `#[l1_handler]` (L1-message-only entry point), but a companion L2-callable setter could be added with equivalent access control (only the L2 RewardSupplier may call it), removing the need for the second L1 message entirely: [3](#0-2) 

### Impact Explanation
Every call to `tick()` that processes pending mint requests pays two L1→L2 message fees instead of one. `tick()` is a permissionless, mandatory protocol operation — it must be called regularly to keep the reward pipeline funded. The extra fee is a direct, recurring ETH cost borne by whoever operates the protocol (or any public caller). This constitutes concrete financial damage to the protocol's operational costs with no compensating benefit.

**Allowed impact matched:** Medium — Griefing with no profit motive but damage to users or protocol.

### Likelihood Explanation
`tick()` is `external payable` with no access control. It must be called whenever the L2 RewardSupplier has pending mint requests (i.e., after every epoch where rewards are distributed). This is a routine, high-frequency operation. The double fee is incurred on every successful invocation. [4](#0-3) 

### Recommendation
Pass the current STRK total supply as part of the `depositWithMessage` message payload instead of sending a separate `sendMessageToL2` call:

```solidity
uint256[] memory depositMsg = new uint256[](1);
depositMsg[0] = IERC20(token()).totalSupply();
bridge().depositWithMessage{value: msg.value}(
    token(),
    amountToMint,
    mintDestination(),
    depositMsg
);
// Remove the separate sendMessageToL2 call entirely.
```

On the L2 side, extend `RewardSupplier.on_receive` to decode the total supply from `message` and call a new L2-accessible setter on `MintingCurve` (guarded so only the L2 RewardSupplier may call it). This eliminates the second L1→L2 message and halves the fee burden on every `tick()` invocation.

### Proof of Concept
1. L2 RewardSupplier accumulates pending mint requests (via `send_mint_request_to_l1_reward_supplier`).
2. Anyone calls `tick{value: X}()` on L1 `RewardSupplier`.
3. `tick()` calls `bridge().depositWithMessage{value: X/2}(...)` — fee 1 paid.
4. `tick()` calls `messagingContract().sendMessageToL2{value: X - X/2}(...)` — fee 2 paid.
5. Both messages are necessary for correct protocol operation; the caller cannot omit either.
6. Result: caller pays `X` ETH in message fees where `X/2` would suffice if the total supply were piggybacked onto the bridge deposit message. [5](#0-4)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }

                .expect_with_err(GenericError::AMOUNT_TOO_HIGH);
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();

                self.l1_pending_requested_amount.write(l1_pending_requested_amount);

    #[l1_handler]
    fn update_total_supply(ref self: ContractState, from_address: felt252, total_supply: Amount) {
        assert!(
            from_address == self.l1_reward_supplier.read(),
            "{}",
            Error::UNAUTHORIZED_MESSAGE_SENDER,
        );
        let old_total_supply = self.total_supply.read();
        // Note that the total supply may only increase.
        // Check that total_supply > old_total_supply to handle possible message reordering.
        if total_supply > old_total_supply {
            self.total_supply.write(total_supply);
            self
                .emit(
                    Events::TotalSupplyChanged { old_total_supply, new_total_supply: total_supply },
                );
        }
    }
