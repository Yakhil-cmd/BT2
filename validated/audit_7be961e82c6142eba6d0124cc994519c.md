### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in `Pool` iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last reward checkpoint. The developers explicitly acknowledge this loop is unbounded. A pool member who makes balance changes across many epochs without claiming rewards accumulates an ever-growing trace. Once the trace is large enough, the `claim_rewards` transaction will exceed Starknet's gas limit and revert, with no partial-claim escape hatch, permanently freezing the member's unclaimed yield.

### Finding Description

The loop in `calculate_rewards` is explicitly flagged by the developers as unbounded: [1](#0-0) 

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
```

Each call to `set_member_balance` inserts a new entry keyed at `epoch + K` into `pool_member_epoch_balance`: [2](#0-1) 

`set_member_balance` is called by every balance-mutating entry point: `enter_delegation_pool`, `exit_delegation_pool_intent`, `add_to_delegation_pool`, and `enter_delegation_pool_from_staking_contract`. [3](#0-2) [4](#0-3) [5](#0-4) 

Because the key is `epoch + K`, multiple calls within the same epoch overwrite the same slot, so the trace grows by **at most one entry per epoch**. Over hundreds or thousands of epochs, a long-term delegator who makes at least one balance change per epoch accumulates a proportionally large trace.

The `entry_to_claim_from` cursor is only advanced on a **successful** `claim_rewards` call: [6](#0-5) 

There is no mechanism to claim rewards in partial batches. If the loop exceeds the block gas limit, the entire `claim_rewards` transaction reverts and the cursor is never advanced, making the situation permanent.

The same unbounded `calculate_rewards` call is also made inside the view function `pool_member_info_v1`, which is callable by anyone: [7](#0-6) 

### Impact Explanation

A pool member who has been active for many epochs without claiming rewards will eventually be unable to call `claim_rewards` because the loop over their trace exceeds the gas limit. Since there is no partial-claim path, their accumulated unclaimed yield is **permanently frozen**. This matches the allowed impact: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation

Long-term passive delegators who do not actively manage their position (e.g., they add stake once per epoch but rarely claim) are the natural victims. Starknet epochs are short (on the order of hours), so after a year of activity a member could accumulate thousands of trace entries. The cost to reach this state is only the gas for periodic `add_to_delegation_pool` calls, which is low.

### Recommendation

1. **Add a `max_entries` parameter** to `claim_rewards` so the loop can be bounded per call, allowing incremental reward collection.
2. **Alternatively**, enforce a maximum trace length per pool member by merging or pruning old entries when a new balance change is recorded in the same or adjacent epoch.
3. **At minimum**, document the maximum safe trace length and add an on-chain guard that reverts with a clear error if `pool_member_trace_length - entry_to_claim_from` exceeds a safe threshold, rather than silently running out of gas.

### Proof of Concept

1. Alice enters a delegation pool and makes one `add_to_delegation_pool` call per epoch for N epochs (N ≈ 1 000–10 000 depending on per-iteration gas cost).
2. Alice never calls `claim_rewards` during this period, so `entry_to_claim_from` remains at 0.
3. Alice (or her reward address) calls `claim_rewards`. The loop at line 859 iterates over all N entries, each requiring multiple storage reads via `find_sigma`.
4. The transaction runs out of gas and reverts.
5. Because `entry_to_claim_from` was never updated, every subsequent `claim_rewards` attempt also reverts. Alice's accumulated yield is permanently inaccessible.

        fn exit_delegation_pool_intent(ref self: ContractState, amount: Amount) {
            // Asserts.
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let old_delegated_stake = self.get_last_member_balance(:pool_member);
            let total_amount = old_delegated_stake + pool_member_info.unpool_amount;
            assert!(amount <= total_amount, "{}", GenericError::AMOUNT_TOO_HIGH);

            // Notify the staking contract of the removal intent.
            let unpool_time = self.undelegate_from_staking_contract_intent(:pool_member, :amount);

            // Edit the pool member to reflect the removal intent, and write to storage.
            if amount.is_zero() {
                pool_member_info.unpool_time = Option::None;
            } else {
                pool_member_info.unpool_time = Option::Some(unpool_time);
            }
            pool_member_info.unpool_amount = amount;
            let new_delegated_stake = total_amount - amount;
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Update the pool member's balance checkpoint.
            self.set_member_balance(:pool_member, amount: new_delegated_stake);

            // Emit events.
            self
                .emit(
                    Events::PoolMemberExitIntent {
                        pool_member, exit_timestamp: unpool_time, amount,
                    },
                );
            self
                .emit(
                    Events::PoolMemberBalanceChanged {
                        pool_member, old_delegated_stake, new_delegated_stake,
                    },
                );
        }
