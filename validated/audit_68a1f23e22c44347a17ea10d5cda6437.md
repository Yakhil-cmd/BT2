This fires whenever `token == WETH`, `nativeBalance > 0`, and `nativeBalance < value`. It wraps **all** of the router's current native ETH balance — regardless of who deposited it — and sends it to the pool as part of the current user's payment.

**How ETH gets stranded on the router between transactions:**

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses. ETH sent via `msg.value` on a `payable` function bypasses `receive()` entirely. The intended pattern (per the test file comment at line 8) is `multicall{value}(exactInput*) + refundETH`. But:

- `exactOutputSingle{value: X}` — user doesn't know the exact input in advance, so they send `X > amountIn`. The swap callback calls `pay()` which uses exactly `amountIn` ETH. The remaining `X - amountIn` ETH stays on the router if the user omits `refundETH`.
- Any `exactInput*{value: X}` call where `X > amountIn` without a `refundETH` in the same multicall leaves the excess on the router.

**The cross-user theft path:**

1. User A calls `exactOutputSingle{value: X}` (or any payable swap) where `X > amountIn`, without `refundETH`. Leftover `delta = X - amountIn` ETH sits on the router.
2. User B calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = V`, where `0 < delta < V`.
3. Inside the swap callback, `pay(WETH, userB, pool, V)` is called. `nativeBalance = delta > 0` and `delta < V`, so the hybrid branch fires.
4. Router wraps `delta` ETH (User A's funds) and transfers it to the pool, then pulls only `V - delta` WETH from User B.
5. User A's `delta` ETH is permanently consumed. User B's WETH cost is reduced by `delta`.

**Why existing guards do not prevent this:**

- `receive()` only blocks direct ETH pushes, not `msg.value` on payable calls. [1](#0-0) 
- `pay()` reads `address(this).balance` with no accounting of which caller deposited which ETH — it treats the entire native balance as available for the current payment. [2](#0-1) 
- There is no per-call ETH escrow, no lock preventing cross-transaction ETH consumption, and no enforcement that `refundETH` must be called.

### Title
Prior user's stranded ETH consumed by subsequent WETH swap via `pay()` hybrid branch — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay()` treats the router's entire native ETH balance as freely available for any WETH payment. ETH left on the router from a prior transaction (e.g., excess `msg.value` from an `exactOutputSingle` call without `refundETH`) is silently consumed to partially fund a subsequent user's WETH swap, causing the prior user to lose their ETH.

### Finding Description
`pay()` contains a hybrid branch for `token == WETH` that fires when `0 < address(this).balance < value`. It wraps the full native balance and sends it to the pool, then pulls the remainder from the current payer via `safeTransferFrom`. The router holds no per-user ETH accounting; any ETH present — regardless of origin — is consumed.

ETH arrives on the router via `msg.value` on payable swap functions (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, `multicall`). The intended usage pattern requires the caller to include `refundETH()` in the same multicall to recover excess ETH. If the caller omits `refundETH` (common with `exactOutput*` where the exact input is unknown), the excess ETH persists on the router across transaction boundaries.

A subsequent WETH swap by any user — including a griefing attacker who monitors the mempool — triggers the hybrid branch and drains the stranded ETH. [3](#0-2) 

### Impact Explanation
Direct loss of user principal. The prior user loses their stranded ETH; the subsequent user receives a discount on their WETH swap funded by the victim's ETH. The loss is bounded by the amount of ETH left on the router per transaction, but is repeatable and griefable. Meets Sherlock Medium threshold (loss of user funds, no trusted-party assumption required).

### Likelihood Explanation
`exactOutput*` callers routinely over-send ETH because the exact input is unknown before execution. The multicall + `refundETH` pattern is not enforced by the contract. Any user who calls a payable swap function directly (not via multicall) or forgets `refundETH` is vulnerable. An attacker needs only to observe a non-zero `address(router).balance` on-chain and submit a WETH swap before the victim's `refundETH`.

### Recommendation
Track per-call deposited ETH in transient storage (EIP-1153) and restrict `pay()` to consume only the ETH deposited in the current call. Alternatively, at the end of each top-level swap entry point, assert `address(this).balance == 0` (or auto-refund to `msg.sender`) so no ETH can persist across transaction boundaries.
