### Title
COA EVM Balance Below Minimum Withdrawable Unit Permanently Stranded After Partial Withdrawal - (File: `fvm/evm/impl/impl.go`)

---

### Summary

The `withdraw` function of `CadenceOwnedAccount` (COA) in Flow EVM only validates that the **withdrawal amount** is at or above the minimum representable unit in a Cadence `UFix64` Flow Vault (1e10 attoFlow = 1e-8 FLOW), but never validates that the **remaining balance after withdrawal** is either zero or also at or above that minimum. A COA can accumulate sub-1e10 attoFlow balances through EVM-native transfers, and a subsequent partial withdrawal can leave a remainder below 1e10 attoFlow that is permanently stranded — it can never be withdrawn to Cadence.

---

### Finding Description

The minimum unit that a Cadence `FlowToken.Vault` (UFix64) can represent is 1e-8 FLOW = 1e10 attoFlow. This is enforced by `AttoFlowBalanceIsValidForFlowVault`: [1](#0-0) 

The check is applied to the **withdrawal amount** in `newInternalEVMTypeWithdrawFunction`: [2](#0-1) 

And identically in the emulator layer `withdrawFrom`: [3](#0-2) 

Neither location checks whether `COA.balance - withdrawalAmount` is either zero or >= 1e10 attoFlow after the operation completes.

A COA's EVM balance can contain sub-1e10 attoFlow precision because EVM-native transactions (e.g., a Solidity contract calling `payable(coaAddress).transfer(1)`) can send arbitrary wei (attoFlow) amounts to any EVM address, including COAs. Once such a sub-1e10 attoFlow remainder exists in the COA's EVM balance, the `withdraw` function in `CadenceOwnedAccount.cdc` will reject any attempt to withdraw it: [4](#0-3) 

The `ErrWithdrawBalanceRounding` error is returned for any amount below 1e10 attoFlow: [5](#0-4) 

The stranded balance cannot be recovered to Cadence by any other mechanism on the `CadenceOwnedAccount` resource.

---

### Impact Explanation

**Impact: Medium** — A sub-1e10 attoFlow balance (maximum ~9,999,999,999 attoFlow ≈ 1e-8 FLOW) is permanently stranded in the COA's EVM balance and can never be bridged back to Cadence. This is a cross-VM asset loss: the balance exists on the EVM side but is inaccessible from Cadence. While the per-incident monetary value is small (< 1e-8 FLOW), it is a real and irreversible loss of on-chain assets for any COA that receives EVM-native transfers with sub-1e10 attoFlow precision. The loss compounds across all affected COAs.

---

### Likelihood Explanation

**Likelihood: Medium** — Any EVM smart contract or EOA can send an arbitrary attoFlow amount to a COA's EVM address. This is a standard EVM operation requiring no special privileges. Any COA that interacts with EVM contracts (e.g., DeFi protocols, token contracts, or any contract that sends ETH/FLOW as part of its logic) is susceptible. The only requirement is that the COA receives a transfer with sub-1e10 attoFlow precision and then performs a partial withdrawal.

---

### Recommendation

After computing the withdrawal amount, check that the remaining balance (`COA.balance - withdrawalAmount`) is either exactly zero or >= 1e10 attoFlow. If the remainder would fall below 1e10 attoFlow, either:
1. Reject the withdrawal with a descriptive error, or
2. Automatically include the sub-1e10 attoFlow remainder in the withdrawal by truncating it (since it cannot be represented in a Cadence vault, it would be burned/donated to the protocol).

The check should be added in `newInternalEVMTypeWithdrawFunction` in `fvm/evm/impl/impl.go` after line 785, before calling `account.Withdraw(amount)`.

---

### Proof of Concept

**Setup**: A COA receives 1_000_000_000_001 attoFlow total — 1_000_000_000_000 attoFlow from a Cadence deposit (valid UFix64 multiple) plus 1 attoFlow from an EVM-native transfer (e.g., a Solidity contract sends 1 wei to the COA address).

**Step 1**: User calls `cadenceOwnedAccount.withdraw(balance: EVM.Balance(attoflow: 1_000_000_000_000))`.
- Withdrawal amount: 1_000_000_000_000 attoFlow >= 1e10 → passes `AttoFlowBalanceIsValidForFlowVault` check.
- Withdrawal succeeds. Remaining COA balance: **1 attoFlow**.

**Step 2**: User attempts to withdraw the remaining 1 attoFlow: `cadenceOwnedAccount.withdraw(balance: EVM.Balance(attoflow: 1))`.
- `AttoFlowBalanceIsValidForFlowVault(1)` → `1 < 1e10` → returns `false`.
- Panics with `ErrWithdrawBalanceRounding`: `"withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"`. [2](#0-1) 

**Result**: The 1 attoFlow is permanently stranded. No function on `CadenceOwnedAccount` can recover it to Cadence. The attacker-controlled entry path is any EVM transaction sender who sends a sub-1e10 attoFlow amount to a COA address, combined with the COA owner performing a partial withdrawal — both are unprivileged, externally reachable operations. [6](#0-5) [7](#0-6)

// AttoFlowBalanceIsValidForFlowVault returns true if the given balance,
// represented as atto-flow, can be stored in a Flow Vault, without loss
// in precision.
//
// Warning! The smallest unit of Flow token that a Flow Vault (Cadence)
// can store, is 1e-8 .
// This means the minimum balance, in atto-flow, that can be stored in a
// Flow Vault, is 1e10 .
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
	return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}

        /// Withdraws the balance from the cadence owned account's balance.
        /// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
        /// given that Flow Token Vaults use UFix64 to store balances.
        /// In other words, the smallest withdrawable amount is 1e10 attoFlow.
        /// Amounts smaller than 1e10 attoFlow, will cause the function to panic
        /// with: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".
        /// If the given balance conversion to UFix64 results in rounding loss,
        /// the withdrawal amount will be truncated to the maximum precision for UFix64.
        ///
        /// @param balance: The EVM balance to withdraw
        ///
        /// @return A FlowToken Vault with the requested balance
        access(Owner | Withdraw)
        fun withdraw(balance: Balance): @FlowToken.Vault {
            pre {
                !EVM.isPaused(): "EVM operations are temporarily paused"
            }

            if balance.isZero() {
                return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
            }
            let vault <- InternalEVM.withdraw(
                from: self.addressBytes,
                amount: balance.attoflow
            ) as! @FlowToken.Vault
            emit FLOWTokensWithdrawn(
                address: self.address().toString(),
                amount: balance.inFLOW(),
                withdrawnUUID: vault.uuid,
                balanceAfterInAttoFlow: self.balance().attoflow
            )
            return <-vault
        }
