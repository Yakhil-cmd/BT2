### Title
Unchecked Supply Inflation via Direct `ft_on_transfer` Call on Permissionlessly Registered NEP-141 Token - (File: engine/src/contract_methods/connector.rs)

### Summary
`ft_on_transfer` mints ERC-20 tokens on Aurora based solely on a caller-supplied `amount` field, with no verification that any actual NEP-141 token transfer occurred. Because `deploy_erc20_token` has no access control, any NEAR account can register itself as a NEP-141 token and then call `ft_on_transfer` directly to mint an unbounded quantity of the corresponding ERC-20 token on Aurora without depositing any real assets.

### Finding Description

**Step 1 — Permissionless NEP-141 registration.**

Any NEAR account can therefore call `deploy_erc20_token` and have its account ID registered as a NEP-141 token with a freshly deployed ERC-20 mirror on Aurora.

**Step 2 — Unverified mint in `ft_on_transfer`.**

`ft_on_transfer` dispatches to `receive_erc20_tokens` for every predecessor that is not the ETH-connector account:

```rust

`args.amount` comes directly from the JSON input supplied by the caller. There is no check that the Aurora Engine contract's NEP-141 balance actually increased by that amount, and there is no check that the call arrived through `ft_transfer_call` rather than being a direct invocation.

The `EvmErc20.mint` function is `onlyAdmin`, and the admin is the Aurora Engine contract address itself, so the internal EVM call succeeds unconditionally: [5](#0-4) 

### Impact Explanation

An attacker who controls a NEAR account registered as a NEP-141 token can mint an arbitrary quantity of the corresponding ERC-20 token on Aurora with zero collateral. Any Aurora DeFi protocol (AMM, lending market, etc.) that holds a liquidity position denominated in that ERC-20 token can be drained: the attacker mints the token in unlimited supply, swaps it for real assets (ETH, USDC, etc.), and withdraws. This constitutes direct theft of user funds and/or insolvency of affected protocols.

**Impact: Critical — direct theft of user funds / insolvency.**

### Likelihood Explanation

The attack requires only two permissionless NEAR transactions: one `deploy_erc20_token` call and one direct `ft_on_transfer` call. No privileged key, governance vote, or external oracle is needed. Any NEAR account holder can execute this immediately. Likelihood is high.

### Recommendation

1. **Verify balance increase.** Inside `receive_erc20_tokens`, read the Aurora Engine's NEP-141 balance of `token` before and after the expected transfer and assert that the increase equals `args.amount`. This mirrors the recommendation in the referenced report.
2. **Restrict direct `ft_on_transfer` calls.** Require that `ft_on_transfer` is only reachable as a callback from `ft_transfer_call` (e.g., by checking that the call is a promise callback via `env.assert_private_call()` or an equivalent predecessor-whitelist check).
3. **Add access control to `deploy_erc20_token`.** Restrict registration of new NEP-141 tokens to the contract owner or a governance-approved list to prevent arbitrary accounts from creating mintable ERC-20 mirrors.

### Proof of Concept

```
# 1. Attacker registers their own NEAR account as a NEP-141 token (no permission check)
near call aurora.mainnet deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# 2. Attacker calls ft_on_transfer directly (not through ft_transfer_call)
#    with an arbitrary amount and their own EVM address as recipient
near call aurora.mainnet ft_on_transfer \
  '{"sender_id":"attacker.near","amount":"340282366920938463463374607431768211455","msg":"<attacker_evm_address_hex>"}' \
  --accountId attacker.near

# Result: Aurora Engine mints U128::MAX units of the ERC-20 mirror of attacker.near
# to the attacker's EVM address with zero NEP-141 tokens ever deposited.
# The attacker can now use these tokens in any Aurora DeFi protocol.
```

The root cause is that `ft_on_transfer` trusts the caller-supplied `amount` field without verifying any actual balance change, and `deploy_erc20_token` allows any account to become a mintable NEP-141 mirror, together forming an uncontrolled supply inflation path identical in class to the referenced `supplyReward()` vulnerability.
