The vulnerability is real and concrete. Here is the complete analysis:

**Tracing the math:**

In `compute_signature_share`, line 174:
```rust
let big_c = big_s + app_pk * y.0;
```
If `app_pk = G1Projective::identity()`, then `app_pk * y.0 = identity` (scalar multiplication by the additive identity is always identity), so `big_c = big_s = x_i * H(pk || app_id)`. The blinding term vanishes entirely.

The coordinator then aggregates (lines 53–54):
```rust
norm_big_y += participant_output.big_y();
norm_big_c += participant_output.big_c();
```
With all `big_c_i = big_s_i`, the sum is `Σ λ_i * x_i * H(pk || app_id) = msk * H(pk || app_id)` — the derived key `s` in the clear.

**No guard exists:** The `ckd()` initialization block (lines 76–101) only checks participant list validity. There is zero validation that `app_pk` is not the identity.

**Security model explicitly requires this to be prevented:** The spec states:

> "Liveness and correctness depend on this assumption, while the secrecy does not. Example values that should not be leaked even if a node is malicious are `s`, `msk` and private shares of other nodes"

The ElGamal blinding exists precisely to protect `s` from a malicious coordinator. Passing `app_pk = identity` defeats it entirely.

### Title
ElGamal Blinding Bypass via Identity `app_pk` Discloses Derived Key `s` to Coordinator — (`src/confidential_key_derivation/protocol.rs`)

### Summary
`ckd()` accepts `app_pk = G1Projective::identity()` without rejection. This causes the ElGamal blinding term `y * A` to collapse to the group identity in every participant's `compute_signature_share`, so the coordinator's aggregated `big_c` equals `msk * H(pk || app_id)` — the confidential derived key — in the clear.

### Finding Description
`compute_signature_share` computes the blinded ciphertext as:

```
C_i = S_i + y_i * A    (A = app_pk)
```

When `A = G1Projective::identity()`, scalar multiplication `y_i * identity = identity`, so `C_i = S_i = x_i * H(pk || app_id)`. After Lagrange-weighted aggregation by the coordinator:

```
C = Σ λ_i * C_i = Σ λ_i * x_i * H(pk || app_id) = msk * H(pk || app_id)
```

The coordinator's `CKDOutput.big_c` is now exactly `s` with no masking. The `ckd()` entry point performs no identity-point check on `app_pk`: [1](#0-0) 

The blinding computation that is bypassed: [2](#0-1) 

The coordinator aggregation that then exposes `s`: [3](#0-2) 

### Impact Explanation
The coordinator directly observes `s = msk * H(pk || app_id)` in `CKDOutput.big_c` without needing `app_sk`. This violates the core CKD security requirement:

> "Private means that the secret itself is never revealed to any entity other than the application itself, not even to individual MPC nodes" [4](#0-3) 

Impact: **Critical — unauthorized disclosure of the confidential derived key to the coordinator.**

### Likelihood Explanation
`app_pk` originates from an on-chain CKD request submitted by the TEE app. The MPC contract explicitly does not verify remote attestation:

> "Notice the MPC contract does not verify remote attestation, nor does it impose how the authentication of the TEE app is enforced." [5](#0-4) 

Any entity that can submit a CKD request on-chain (a malicious or compromised TEE app, or a developer contract that omits the `A ∈ report_data` check) can supply `A = identity`. The library itself provides no defense-in-depth check. The attack requires no cryptographic break — only passing a specific constant.

### Recommendation
Add an identity-point rejection in `ckd()` immediately after the participant list checks:

```rust
if app_pk.is_identity().into() {
    return Err(InitializationError::InvalidPublicKey);
}
```

This mirrors the pattern already used in `BLS12381G1Group::serialize` and `BLS12381G2Group::serialize`: [6](#0-5)
