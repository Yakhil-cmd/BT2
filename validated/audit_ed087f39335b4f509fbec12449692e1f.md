### Title
Non-Monotonic EVM Block Timestamps Due to Cadence Timestamp Truncation - (File: `fvm/environment/evm_block_store.go`)

### Summary

`constructBlockProposal()` in `fvm/environment/evm_block_store.go` derives the EVM block timestamp by truncating the Cadence block timestamp (nanoseconds) to seconds. No check is performed to ensure the resulting EVM block timestamp is strictly greater than the previous EVM block's timestamp. Consecutive Flow blocks that fall within the same wall-clock second produce EVM blocks with identical timestamps, violating the Ethereum specification requirement that `block.timestamp > parent.timestamp`.

### Finding Description

In `constructBlockProposal()`, the EVM block timestamp is computed as:

```go
// cadence block timestamp is unix nanoseconds but evm blocks
// expect timestamps in unix seconds so we convert here
timestamp := uint64(cadenceBlock.Timestamp / int64(time.Second))
``` [1](#0-0) 

`time.Second` is `1,000,000,000` nanoseconds. Integer division truncates sub-second precision. The Flow protocol's `BlockTimer` enforces that consecutive Flow block timestamps are at least `minInterval` milliseconds apart (e.g., 100 ms), but this minimum gap is far smaller than one second. Two consecutive Flow blocks with timestamps `T` ns and `T + 100,000,000` ns (100 ms apart) both truncate to the same second value.

After `CommitBlockProposal` finalizes EVM block N, it immediately calls `constructBlockProposal()` again to eagerly build the proposal for EVM block N+1, using the **same** current Cadence block's timestamp:

```go
newBP, err := bs.constructBlockProposal()
``` [2](#0-1) 

Neither `constructBlockProposal()` nor `CommitBlockProposal()` compares the new timestamp against `lastExecutedBlock.Timestamp`: [3](#0-2) 

The `types.Block` struct documents that `Timestamp` must come from the FVM block, but imposes no monotonicity invariant: [4](#0-3) 

By contrast, the Flow consensus layer does enforce monotonicity for Flow block headers via `BlockTimer.Validate()`: [5](#0-4) 

But this protection does not propagate to the EVM layer because the seconds-truncation collapses the guaranteed millisecond-level ordering.

### Impact Explanation

Any EVM smart contract that relies on `block.timestamp` for strict temporal ordering is affected. Concrete examples:

- **Time-locked vaults**: A contract enforcing `require(block.timestamp > unlockTime)` may be bypassed or permanently stuck if consecutive EVM blocks share the same timestamp.
- **Auction / vesting contracts**: Logic of the form `require(block.timestamp > lastActionTimestamp)` can be re-entered within the same second window.
- **Randomness seeds**: Contracts using `block.timestamp` as part of a seed get identical inputs across multiple blocks.

The Ethereum Yellow Paper requires `block.timestamp > parent.timestamp` (strictly greater). Violating this invariant means any EVM contract that assumes this property holds is vulnerable to unexpected behavior, including unauthorized asset access in time-gated contracts.

### Likelihood Explanation

Flow's `BlockTimer` enforces a `minInterval` of 100 ms between consecutive blocks on mainnet. Flow blocks are produced roughly every second, but bursts of blocks within the same second are routine. Any two consecutive Flow blocks whose nanosecond timestamps share the same integer second produce EVM blocks with equal timestamps. This is a near-certain occurrence during normal network operation, not a theoretical edge case.

### Recommendation

In `constructBlockProposal()`, after computing `timestamp`, enforce strict monotonicity against the last executed EVM block:

```go
timestamp := uint64(cadenceBlock.Timestamp / int64(time.Second))
if timestamp <= lastExecutedBlock.Timestamp {
    timestamp = lastExecutedBlock.Timestamp + 1
}
```

This mirrors the fix applied in Etherlink MR 13832 ("Forced blueprint checks always use a valid timestamp") and ensures the EVM chain always satisfies the Ethereum spec invariant regardless of how quickly Flow blocks are produced.

### Proof of Concept

1. Flow block K−1 has Cadence timestamp `1_700_000_000_100_000_000` ns → EVM block N gets timestamp `1_700_000_000` s.
2. Flow block K has Cadence timestamp `1_700_000_000_200_000_000` ns (100 ms later, valid per `BlockTimer`) → `CommitBlockProposal` for EVM block N eagerly constructs EVM block N+1 with timestamp `1_700_000_000` s (same second).
3. EVM block N and EVM block N+1 both carry `Timestamp = 1_700_000_000`.
4. A Solidity contract deployed on Flow EVM with `require(block.timestamp > lastTimestamp, "too soon")` will revert on the second call within the same second, or a contract using `>=` will allow re-entry within the same second window, depending on the guard used.

The root cause is the absence of a `timestamp > lastExecutedBlock.Timestamp` guard in `constructBlockProposal()` at `fvm/environment/evm_block_store.go:163`. [3](#0-2)

func (bs *BlockStore) constructBlockProposal() (*types.BlockProposal, error) {
	// if available construct a new one
	cadenceHeight, err := bs.blockInfo.GetCurrentBlockHeight()
	if err != nil {
		return nil, err
	}

	cadenceBlock, found, err := bs.blockInfo.GetBlockAtHeight(cadenceHeight)
	if err != nil {
		return nil, err
	}
	if !found {
		return nil, fmt.Errorf("cadence block not found")
	}

	lastExecutedBlock, err := bs.LatestBlock()
	if err != nil {
		return nil, err
	}

	parentHash, err := lastExecutedBlock.Hash()
	if err != nil {
		return nil, err
	}

	// cadence block timestamp is unix nanoseconds but evm blocks
	// expect timestamps in unix seconds so we convert here
	timestamp := uint64(cadenceBlock.Timestamp / int64(time.Second))

	// read a random value for block proposal
	prevrandao := gethCommon.Hash{}
	err = bs.randGen.ReadRandom(prevrandao[:])
	if err != nil {
		return nil, err
	}

	blockProposal := types.NewBlockProposal(
		parentHash,
		lastExecutedBlock.Height+1,
		timestamp,
		lastExecutedBlock.TotalSupply,
		prevrandao,
	)

	return blockProposal, nil
}
