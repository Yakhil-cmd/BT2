### Title
Missing Zero-Address Validation for `reward_address` Allows Permanent Loss of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
The `stake()` and `change_reward_address()` functions in `staking.cairo`, and `enter_delegation_pool()` and `change_reward_address()` in `pool.cairo`, accept a user-controlled `reward_address` parameter without validating that it is non-zero. If a staker or pool member sets their reward address to the zero address, all accumulated STRK rewards are permanently transferred to address zero when `claim_rewards()` is called, resulting in irreversible loss of unclaimed yield.

### Finding Description
Four public entry points accept `reward_address: ContractAddress` from an unprivileged caller without a zero-address guard:

1. **`Staking::stake()`** — `reward_address` is stored directly into `staker_info` with no `is_non_zero()` check. The only validation is `!does_token_exist(reward_address)`, which passes for the zero address because zero is not a registered token.

2. **`Staking::change_reward_address()`** — Same pattern: only `!does_token_exist(reward_address)` is checked; no zero-address guard.

3. **`Pool::enter_delegation_pool()`** — Only checks `token_address != reward_address`; no zero-address guard.

4. **`Pool::change_reward_address()`** — Only checks `token_dispatcher.contract_address.read() != reward_address`; no zero-address guard.

When `claim_rewards()` is subsequently called (by the staker/pool member or their reward address), the contract executes:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

If `reward_address` is zero, the STRK transfer succeeds (ERC-20 does not block transfers to address zero on Starknet) and the tokens are permanently burned.

### Impact Explanation
Any staker or pool member who — through a front-end bug, scripting error, or direct contract call mistake — passes `reward_address = 0` will have all their accumulated STRK staking rewards permanently sent to the zero address upon the next `claim_rewards()` call. The funds are irrecoverable. This constitutes **permanent freezing of unclaimed yield**.

### Likelihood Explanation
Medium. The zero address is the default/uninitialized value in many programming environments and tooling. Direct contract interactions, deployment scripts, or front-end bugs that omit the reward address field will silently pass zero. The protocol already guards the caller address against zero via `assert_caller_is_not_zero()` inside `general_prerequisites()`, but this protection does not extend to the `reward_address` parameter. The absence of a symmetric guard on `reward_address` is an inconsistency that makes accidental misuse realistic.

### Recommendation
Add an explicit non-zero assertion for `reward_address` in all four entry points, consistent with the existing `GenericError::ZERO_ADDRESS` error already defined in the codebase:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply this check in:
- `Staking::stake()` before storing `reward_address`
- `Staking::change_reward_address()` before updating `staker_info`
- `Pool::enter_delegation_pool()` before creating the pool member record
- `Pool::change_reward_address()` before updating `pool_member_info`

### Proof of Concept

**Staking path:**

1. Attacker/user calls `staking.stake(reward_address: 0, operational_address: X, amount: MIN_STAKE)`.
2. The call passes all existing checks: `!does_token_exist(0)` is `true` (zero is not a registered token), `operational_address_to_staker_address.read(0).is_zero()` is `true` (default storage value).
3. Staker accumulates rewards over epochs via attestation.
4. Anyone calls `staking.claim_rewards(staker_address)`.
5. `send_rewards_to_staker` executes `token_dispatcher.checked_transfer(recipient: 0, amount: rewards)` — rewards are permanently burned.

**Pool path:**

1. Pool member calls `pool.enter_delegation_pool(reward_address: 0, amount: X)`.
2. The call passes: `token_address != 0` (STRK address ≠ zero), `amount.is_non_zero()`.
3. Pool member accumulates rewards.
4. Anyone calls `pool.claim_rewards(pool_member)`.
5. `reward_token.checked_transfer(recipient: 0, amount: rewards)` — rewards permanently burned.
