### Title
Incomplete Abort Signaling in `broadcast_success` Leaves Honest DKG Participants Waiting Indefinitely — (`File: src/dkg.rs`)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` is documented to broadcast either success or failure depending on whether an error occurred, but the implementation always broadcasts success and does not accept an error parameter. When a participant encounters a local validation failure during DKG (e.g., an invalid secret share sent by a malicious party), it returns an error without ever broadcasting a failure signal. All other honest participants remain blocked waiting for that participant's broadcast message, which never arrives, permanently denying DKG completion.

---

### Finding Description

The `broadcast_success` function carries the following doc comment:

```
/// This function takes err as input.
/// If err is None then broadcast success
/// otherwise, broadcast failure
/// If during broadcast it receives an error then propagates it
/// This function is used in the final round of DKG
```

However, the actual function signature and body are:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    ...
    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant seems to have failed its checks. Aborting Protocol!"
        ));
    }
    Ok(())
}
```

There is **no `err` parameter**. The function unconditionally broadcasts `(true, session_id)`. The guard that checks for `false` votes from other participants (`!vote_list.iter().all(|&(boolean, _)| boolean)`) can therefore **never trigger** based on a local failure, because no participant ever broadcasts `false`.

In `do_keyshare`, if a participant detects an invalid share via `validate_received_share` (which returns `ProtocolError::InvalidSecretShare(from)`), the `?` operator propagates the error and the function returns **before** calling `broadcast_success`. The failing participant therefore never sends any broadcast message for the final round. All other honest participants are blocked inside `do_broadcast` waiting for every participant's message, which never arrives.

---

### Impact Explanation

**High — Permanent denial of key generation for honest parties.**

A malicious participant can send a deliberately invalid secret share to one or more honest participants. Each honest recipient will detect the invalid share, return an error, and silently drop out of the final broadcast round without notifying anyone. The remaining honest participants will block indefinitely in `do_broadcast` waiting for the missing broadcast messages. Because the library provides no network-layer timeout mechanism, DKG is permanently stalled for all honest parties. This applies equally to the reshare and refresh protocols, which share the same `do_keyshare` core.

---

### Likelihood Explanation

Any single malicious participant (one of the `MaxFaulty` Byzantine nodes) can trigger this by sending a syntactically valid but cryptographically invalid share to any honest participant. The `validate_received_share` function will reject it, the honest participant will abort without broadcasting, and the protocol will hang. The attacker needs no special privileges beyond being a registered participant.

---

### Recommendation

Extend `broadcast_success` to accept an optional error parameter as its documentation describes. When the caller has encountered a local failure, it should call `broadcast_success` with the failure indication **before** returning the error, so that other participants receive a `(false, session_id)` broadcast and can abort cleanly rather than waiting indefinitely. The existing check `!vote_list.iter().all(|&(boolean, _)| boolean)` is already in place to handle the `false` case — it simply needs to be reachable.

---

### Proof of Concept

1. Participants `P1 … P5` run DKG with `MaxFaulty = 1`.
2. Malicious participant `M` sends a share to honest participant `P1` that fails `validate_received_share` (e.g., the share does not lie on the committed polynomial).
3. `P1` hits `ProtocolError::InvalidSecretShare(M)` and returns the error via `?` before reaching `broadcast_success`. `P1` never sends a broadcast message for the final round.
4. `P2 … P5` are blocked inside `do_broadcast` waiting for `P1`'s message. Because the library has no timeout, they wait forever.
5. DKG never completes. No party obtains a `KeygenOutput`. Signing is permanently unavailable. [1](#0-0) [2](#0-1) [3](#0-2)
