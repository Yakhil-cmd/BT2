### Title
Missing Exit-Window Guard in `enter_delegation_pool_from_staking_contract` Allows Switched Principal to Be Temporarily Frozen — (File: `src/pool/pool.cairo`)

---

### Summary

The `enter_delegation_pool_from_staking_contract` function in `pool.cairo` does not enforce the spec's precondition that the receiving pool member must **not** be in an exit window. When a pool member switches principal into a destination pool where they already hold an active undelegate intent, the switched amount is credited to their balance trace but **not** to `unpool_amount`. A subsequent `exit_delegation_pool_action` call returns only the original `unpool_amount`; the switched principal remains locked in the staking contract under the destination pool's delegated balance and requires a full additional exit-window cycle to recover.

---

### Finding Description

The spec for `enter_delegation_pool_from_staking_contract` explicitly lists:

> **pre-condition 2**: `pool_member` is not in an exit window. [1](#0-0) 

The implementation, however, only asserts `amount > 0` and that the caller is the staking contract: [2](#0-1) 

There is no `assert!(pool_member_info.unpool_time.is_none(), ...)` guard. The staking-side entry point `switch_staking_delegation_pool` likewise has no such check before it calls `enter_delegation_pool_from_staking_contract`: [3](#0-2) 

**Vulnerable state-transition path:**

1. Pool member holds `X` tokens in Pool B and calls `exit_delegation_pool_intent(X)` in Pool B.
   - Pool B stores: `unpool_amount = X`, `unpool_time = T`, balance trace = 0.
   - Staking contract records undelegate intent for Pool B = `X`.
2. Pool member holds `Y` tokens in Pool A and calls `exit_delegation_pool_intent(Y)` in Pool A.
3. Pool member calls `switch_delegation_pool(to_staker_B, pool_B, Y)` in Pool A.
   - Pool A clears its intent; staking contract increases Pool B's delegated balance by `Y`.
   - Staking contract calls `pool_B.enter_delegation_pool_from_staking_contract(Y, data)`.
   - Pool B executes `increase_member_balance(pool_member, Y)` — balance trace now = `Y`.
   - Pool B's `unpool_amount` and `unpool_time` are **unchanged** (`X` and `T`). [4](#0-3) 

4. Pool member calls `exit_delegation_pool_action` in Pool B after time `T`:
   - `remove_from_delegation_pool_action` transfers `X` tokens from staking contract → Pool B.
   - Pool B transfers `unpool_amount = X` to pool member.
   - `unpool_amount` is zeroed; `unpool_time` is cleared. [5](#0-4) 

5. The `Y` tokens remain in the staking contract under Pool B's delegated balance. The pool member must now call `exit_delegation_pool_intent(Y)` again, wait a full exit-window period, and call `exit_delegation_pool_action` a second time.

---

### Impact Explanation

The switched principal `Y` is **temporarily frozen** inside the staking contract for one additional exit-window period (default: 1 week, maximum: 12 weeks). [6](#0-5) 

This matches the allowed impact: **Temporary freezing of funds**. The funds are not permanently lost — the pool member can recover them by repeating the intent/action cycle — but they are inaccessible for the duration of an extra exit window, which can be up to 12 weeks.

---

### Likelihood Explanation

The trigger requires a pool member to:
1. Hold positions in two separate pools simultaneously.
2. Have an active undelegate intent in the destination pool at the time of the switch.

Both conditions are reachable by any unprivileged delegator without any privileged access. A user who is exiting Pool B while simultaneously trying to consolidate their Pool A position into Pool B will hit this path naturally. The missing guard is not documented anywhere visible to callers, making accidental triggering plausible.

---

### Recommendation

Add an exit-window guard inside `enter_delegation_pool_from_staking_contract` when the pool member already exists:

```cairo
// In the Option::Some branch of enter_delegation_pool_from_staking_contract:
assert!(
    pool_member_info.unpool_time.is_none(),
    "{}",
    Error::UNDELEGATE_IN_PROGRESS,
);
``` [4](#0-3) 

Alternatively, add the check in `switch_staking_delegation_pool` in `staking.cairo` before the cross-contract call, querying the destination pool's member state.

---

### Proof of Concept

```
1. pool_member delegates 100 STRK to pool_A (staker_A) and 100 STRK to pool_B (staker_B).

2. pool_member calls pool_B.exit_delegation_pool_intent(100).
   → pool_B: unpool_amount=100, unpool_time=T, balance=0
   → staking: intent[pool_B][pool_member] = 100

3. pool_member calls pool_A.exit_delegation_pool_intent(100).
   → pool_A: unpool_amount=100, unpool_time=T, balance=0
   → staking: intent[pool_A][pool_member] = 100

4. pool_member calls pool_A.switch_delegation_pool(staker_B, pool_B, 100).
   → pool_A: unpool_amount=0, unpool_time=None  (intent cleared)
   → staking: intent[pool_A][pool_member] cleared; pool_B delegated_balance += 100
   → pool_B.enter_delegation_pool_from_staking_contract(100, data) called

            let unpool_time = pool_member_info

                .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
            assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Emit event.

                    Events::PoolMemberExitAction {
                        pool_member, unpool_amount: pool_member_info.unpool_amount,
                    },
                );
