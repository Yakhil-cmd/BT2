### Title
Missing Refund on Failed `ExitToNear` Promise When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled (which is the default), the `ExitToNear` precompile schedules a bare `ft_transfer` or `ft_transfer_call` NEAR promise with **no callback** and **no refund path**. If that promise fails, the user's ERC-20 tokens (already burned) or ETH (already deducted from the EVM balance) are permanently lost with no recovery mechanism.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` constructs a `callback_args` struct that conditionally includes a `refund` field:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in default build
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent (the default, since it is not listed in `[features] default`), `refund` is always `None`. [2](#0-1) 

For the two most common exit paths — ERC-20 legacy `ft_transfer` and ETH base-token `ft_transfer` — `transfer_near` is also `None`. This makes `callback_args` equal to its `Default` value, so the code takes the branch that schedules only a bare `PromiseArgs::Create` with **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no refund
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

Even in the wNEAR-unwrap path where a callback (`exit_to_near_precompile_callback`) **is** scheduled, the callback's `refund` field is still `None`, so the failure branch is a no-op:

```rust
} else if let Some(args) = args.refund {
    // refund tokens
} else {
    None   // ← silent no-op when error_refund is disabled
}
``` [4](#0-3) 

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

The same acknowledgment exists for the ETH exit path: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent loss of user funds.**

When a user calls the `ExitToNear` precompile (flag `0x00` for ETH, flag `0x01` for ERC-20):

- **ERC-20 path**: The ERC-20 contract burns the user's tokens before calling the precompile. If the subsequent `ft_transfer` NEAR promise fails (e.g., recipient account not registered with the NEP-141 contract, NEP-141 contract paused, insufficient storage deposit, or any other NEAR-side failure), the burned ERC-20 tokens are gone and no NEP-141 tokens are received. The user suffers a total loss of the exited amount.
- **ETH path**: ETH is deducted from the caller's EVM balance before the `ft_transfer` promise is dispatched. A failed promise leaves the ETH balance reduced with no recovery.

In both cases, the funds are permanently destroyed with no on-chain recovery path available to the user or the protocol.

---

### Likelihood Explanation

**High.** The failure condition is reachable by any unprivileged EVM user in realistic scenarios:

1. Exiting to a NEAR account that is not registered with the NEP-141 token contract (a common user mistake).
2. Exiting to a NEAR account with insufficient storage deposit.
3. Any transient or permanent failure of the NEP-141 `ft_transfer` call.

The `error_refund` feature is **not** in the `default` feature set of either `aurora-engine` or `aurora-engine-precompiles`, meaning any build that does not explicitly pass `--features error_refund` is vulnerable. [7](#0-6) 

---

### Recommendation

Enable the `error_refund` feature in the production build, or make it part of the `default` feature set so that the refund callback is always included. The refund mechanism (`refund_call_args`, `refund_on_error`) already exists and is correct — it is simply gated behind a compile-time flag that is off by default.

Alternatively, remove the `#[cfg(not(feature = "error_refund"))]` guard and unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs`, making the safe behavior the only behavior.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the default build).
2. Bridge a NEP-141 token to Aurora as an ERC-20.
3. Call the ERC-20's `withdrawToNear` function targeting a NEAR account that is **not registered** with the NEP-141 contract (e.g., `"unregistered.near"`).
4. The ERC-20 tokens are burned in the EVM state.
5. The `ft_transfer` NEAR promise is dispatched and fails because the recipient is unregistered.
6. No callback is scheduled; no refund occurs.
7. Observe: the user's ERC-20 balance is reduced by the exit amount, the NEP-141 balance of `"unregistered.near"` is 0, and the NEP-141 balance held by Aurora is unchanged — the tokens are permanently lost.

This exact scenario is confirmed by the test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs`, which asserts the reduced balance when `error_refund` is disabled: [8](#0-7)
