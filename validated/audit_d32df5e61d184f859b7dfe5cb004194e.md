### Title
Missing Caller Restriction on `update_rewards` Allows Anyone to Block Reward Distribution - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in the Staking contract is intended to be callable only by the Starkware sequencer, but no caller check is enforced in the implementation. Any unprivileged actor can call it first in any block with `disable_rewards: true`, which writes the current block number to `last_reward_block` without distributing rewards. This permanently blocks the sequencer from distributing block rewards for that block, causing permanent freezing of unclaimed yield.

### Finding Description
The spec explicitly states that `update_rewards` has access control: **"Only starkware sequencer"**. [1](#0-0) 

However, the actual implementation in `staking.cairo` contains no caller restriction whatsoever:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker checks ...
    self.last_reward_block.write(current_block_number);  // <-- state written unconditionally
    if disable_rewards || self.is_pre_consensus() {
        return;  // returns WITHOUT distributing rewards
    }
    // ... reward distribution ...
}
``` [2](#0-1) 

A grep for any sequencer-check helper (`assert_caller_is_sequencer`, `only_sequencer`, `CALLER_IS_NOT_SEQUENCER`) returns zero matches across all production Cairo files, confirming the check is entirely absent.

The critical state mutation is at line 1485: `self.last_reward_block.write(current_block_number)`. [3](#0-2) 

This write happens **before** the `disable_rewards` branch, meaning even a call with `disable_rewards: true` (which distributes nothing) still advances `last_reward_block` to the current block. The guard at lines 1454–1458 then prevents any subsequent call in the same block. [4](#0-3) 

### Impact Explanation
An attacker front-runs the sequencer's `update_rewards` call every block by calling it first with `disable_rewards: true` and any valid active staker address. The result:

- `last_reward_block` is set to the current block number with no rewards distributed.
- The sequencer's subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
- Block rewards for that block are permanently lost — there is no catch-up mechanism; each block's reward window is one block wide.

Sustained over time, this causes **permanent freezing of unclaimed yield** for all stakers and delegators. This matches the High-severity allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

### Likelihood Explanation
The entry point is fully public with no authentication. Any address can call `update_rewards` with any known staker address (staker addresses are public on-chain). The attacker only needs to submit a transaction before the sequencer in each block, which is trivially achievable via standard mempool front-running or by simply submitting the transaction at the start of every block. No special privileges, leaked keys, or external dependencies are required.

### Recommendation
Enforce the access control that the spec mandates. Add a caller check at the top of `update_rewards` that asserts the caller is the designated Starkware sequencer address (stored in contract storage or a constant), analogous to the pattern already used in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // add this
    ...
}
``` [5](#0-4) 

### Proof of Concept

1. Staker `S` is active and has been staked for `K` epochs (so `is_staker_active` and `staker_total_strk_balance.is_non_zero()` both pass).
2. Consensus rewards are active (`is_pre_consensus()` returns false).
3. Attacker `A` (any address) calls `update_rewards(staker_address: S, disable_rewards: true)` at block `N`.
   - `current_block_number (N) > last_reward_block` → passes.
   - `last_reward_block` is written to `N`.
   - `disable_rewards == true` → function returns early, **no rewards distributed**.
4. Sequencer calls `update_rewards(staker_address: S, disable_rewards: false)` in the same block `N`.
   - `current_block_number (N) > last_reward_block (N)` → **false** → reverts with `REWARDS_ALREADY_UPDATED`.
5. Block `N`'s rewards are permanently lost. Repeating steps 3–4 every block freezes all yield indefinitely.
