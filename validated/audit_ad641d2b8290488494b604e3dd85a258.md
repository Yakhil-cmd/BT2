### Title
Missing Validation of Participant-Supplied CKD Shares Allows Malicious Participant to Corrupt Confidential Key Derivation Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` receives elliptic-curve share contributions `(big_y, big_c)` from every participant and blindly accumulates them with no proof-of-correctness check. A single malicious participant can substitute arbitrary curve points, causing the coordinator to output a corrupted `CKDOutput` that does not correspond to the intended confidential derived key.

### Finding Description
The `do_ckd_coordinator` function collects each participant's `CKDOutput` — a pair of BLS12-381 G1 points `(big_y, big_c)` — and adds them directly into the running sum: [1](#0-0) 

The intended invariant is that each honest participant $P_i$ sends:

- `big_y_i = λ_i · y_i · G`
- `big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

so that the coordinator's sum satisfies `big_c − app_sk · big_y = msk · H(pk ‖ app_id)`.

No zero-knowledge proof, consistency check, or any other validation is performed on the received points before they are accumulated. The protocol simply deserializes the two group elements and adds them: [2](#0-1) 

Compare this with the DKG, where every received commitment is verified against a hash pre-commitment and a proof of knowledge before use: [3](#0-2) 

No equivalent verification exists in the CKD path.

### Impact Explanation
A malicious participant can send an arbitrary pair `(big_y', big_c')` to the coordinator. The coordinator will compute:

```
final_big_c = honest_sum_big_c + big_c'   (wrong)
final_big_y = honest_sum_big_y + big_y'   (wrong)
```

The resulting `CKDOutput` is silently corrupted. Any caller who invokes `unmask(app_sk)` on this output will obtain a point that is not `msk · H(pk ‖ app_id)`, i.e., the confidential derived key is wrong. The coordinator and all downstream consumers accept this output as valid because no final consistency check is performed.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept an incorrect derived confidential key.**

### Likelihood Explanation
Any single participant in the CKD session is sufficient to trigger the corruption. The participant's message is a private send to the coordinator, so no other honest party can detect the substitution. The attacker needs only to be a registered participant — no special privilege is required beyond that.

### Recommendation
Each participant should accompany their `(big_y, big_c)` contribution with a non-interactive zero-knowledge proof demonstrating:

1. Knowledge of the discrete log `y_i` such that `big_y_i = λ_i · y_i · G`.
2. That `big_c_i` was formed as `λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` for the participant's committed public share `x_i · G2` (already known from DKG output).

A Chaum–Pedersen / Sigma-protocol proof over the two-base relation `(G, app_pk)` with witness `y_i`, combined with a check that `big_c_i − λ_i · x_i · H(pk ‖ app_id) = y_i · app_pk`, would be sufficient. The coordinator must verify all proofs before accumulating any contribution, mirroring the pattern already used in DKG: [4](#0-3) 

### Proof of Concept

1. Honest participants $P_1, \ldots, P_{n-1}$ run `compute_signature_share` correctly and send their outputs to the coordinator.
2. Malicious participant $P_n$ instead sends `(big_y' = identity, big_c' = arbitrary_point)` to the coordinator.
3. `do_ckd_coordinator` accumulates without any check: [5](#0-4) 

4. The coordinator returns `CKDOutput::new(norm_big_y + identity, norm_big_c + arbitrary_point)`.
5. The caller invokes `ckd_output.unmask(app_sk)` and receives a point that is not `msk · H(pk ‖ app_id)` — the confidential derived key is silently corrupted with no error surfaced to any honest party.

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
