### Title
WalletConnect Take-Offer Royalty Amount Understated for Multi-NFT Offers Due to Inconsistent Calculation - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` uses a **split-then-multiply** formula to compute the total amount including royalties when a user is asked to take an offer containing multiple NFTs via WalletConnect. This formula systematically underestimates the true royalty cost by a factor of `1/N` (where N is the number of NFTs), causing the WalletConnect approval dialog to display a lower spending total than what is actually deducted from the user's wallet.

### Finding Description

In `parseCommandDisplay.ts`, the function `formatAmountWithRoyalties` computes the royalty-inclusive total shown to the user before they approve a WalletConnect `take_offer` command:

```ts
// packages/gui/src/electron/commands/parseCommandDisplay.ts:363-368
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The formula divides the fungible amount equally among all NFTs before multiplying by each royalty percentage. The correct calculation must multiply the **full** fungible amount by each NFT's royalty percentage independently, because each NFT's royalty is assessed against the entire payment, not a fraction of it.

**Correct formula:**
```
royaltyTotal = amount × r₁ + amount × r₂ + … = amount × (r₁ + r₂ + …)
```

**Actual formula used:**
```
royaltyTotal = (amount/N) × r₁ + (amount/N) × r₂ + … = (amount/N) × (r₁ + r₂ + …)
```

The result is always `1/N` of the correct royalty total, growing worse with each additional NFT.

The existing test in `parseCommandDisplay.test.ts` encodes and confirms this incorrect behavior: for 2 NFTs with royalties of 5% (500 basis points) and 0.1% (10 basis points) and a payment of 100,000,000 mojos, the test asserts `amountWithRoyalties: '0.00010255'`: [2](#0-1) 

The correct value is `0.0001051` (105,100,000 mojos), a ~2.5% understatement in this example. With higher royalties or more NFTs the gap widens proportionally.

The `withRoyaltyTotals` function feeds this incorrect `amountWithRoyalties` value into the `DisplayWalletDelta` object that is rendered in the WalletConnect approval dialog: [3](#0-2) 

### Impact Explanation

A user approving a WalletConnect `take_offer` request for a multi-NFT offer is shown a spending total that is lower than the amount that will actually leave their wallet. They consent to paying X XCH/CAT but the on-chain transaction deducts more. This is a direct, concrete asset loss caused by a spoofed/incorrect amount in the WalletConnect approval state — matching the **High** impact category: *"WalletConnect state that causes a user to approve the wrong amount."*

### Likelihood Explanation

Any party who can craft a WalletConnect `take_offer` payload referencing an offer with two or more NFTs that carry royalties can trigger this. Minting NFTs with royalties on Chia is permissionless and cheap. The attacker creates the NFTs (setting themselves as royalty recipient), builds a multi-NFT offer, and delivers the offer string to the victim's dApp session. The victim sees a lower-than-real total and approves.

### Recommendation

Replace the split-then-multiply formula with a full-amount-per-NFT formula:

```ts
// Correct: each NFT's royalty is assessed on the full fungible amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [4](#0-3) 

Update the corresponding test assertion in `parseCommandDisplay.test.ts` to reflect the corrected value (`0.0001051` for the existing two-NFT scenario). [5](#0-4) 

### Proof of Concept

**Setup:** Attacker mints two NFTs with royalty percentages of 500 (5%) and 10 (0.1%) and sets themselves as royalty recipient. They create an offer selling both NFTs for 100,000,000 mojos (0.0001 XCH) and deliver the offer string to a victim via a WalletConnect-connected dApp.

**Observed behavior (current code):**
- `splitAmount = 100,000,000 / 2 = 50,000,000`
- royalty₁ = `50,000,000 × 500 / 10,000 = 2,500,000`
- royalty₂ = `50,000,000 × 10 / 10,000 = 50,000`
- `amountWithRoyalties` displayed = `102,550,000 mojos = 0.00010255 XCH`

**Actual on-chain deduction:**
- royalty₁ = `100,000,000 × 500 / 10,000 = 5,000,000`
- royalty₂ = `100,000,000 × 10 / 10,000 = 100,000`
- Actual total = `105,100,000 mojos = 0.0001051 XCH`

The victim approves based on `0.00010255 XCH` but `0.0001051 XCH` is deducted — a ~2.5% overcharge in this example, scaling with royalty rates and NFT count.

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
