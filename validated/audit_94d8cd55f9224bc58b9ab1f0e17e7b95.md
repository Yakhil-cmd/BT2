### Title
Gateway `skip_stateful_validations` admits invoke transactions with arbitrary signatures for accounts with a pending `deploy_account` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point — the only place where the account's signature is verified — for any invoke transaction with `nonce == 1` when the account has any transaction in the mempool or a recent block. Because the mempool's `validate_tx` path checks only nonce ordering and fee escalation (no signature field exists in `ValidationArgs`), an unprivileged attacker can submit an invoke with a completely arbitrary signature for any third-party account that has a pending `deploy_account`, and the gateway will admit it.

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by that flag and is entirely skipped: [3](#0-2) 

**The mempool does not fill the gap**

`validate_by_mempool` is called before `skip_stateful_validations` and delegates to `Mempool::validate_tx`: [4](#0-3) 

`Mempool::validate_tx` calls only `validate_incoming_tx` (duplicate hash / nonce-too-old) and `validate_fee_escalation` (fee bump rules): [5](#0-4) 

`ValidationArgs` carries no signature field — it contains only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price`: [6](#0-5) 

Therefore, a transaction with a completely forged signature passes every check in the admission pipeline when the skip condition is met.

**`account_tx_in_pool_or_recent_block` is trivially satisfiable by an attacker**

The predicate returns `true` as soon as the target account appears in the pool or the committed-state map: [7](#0-6) 

Any observer of the public mempool can see a victim's `deploy_account` the moment it is submitted, satisfying condition 4 without any privileged access.

### Impact Explanation

An attacker who observes a victim's `deploy_account` in the mempool can immediately submit an invoke with `nonce = 1` for the victim's address carrying an arbitrary (invalid) signature. The gateway admits it. If fee escalation is enabled (`enable_fee_escalation = true`), the attacker can outbid the victim's legitimate `nonce = 1` invoke and replace it: [8](#0-7) 

When the batcher later executes the block, the `deploy_account` succeeds, then the attacker's invalid invoke is executed — the `__validate__` entry point now runs (the batcher does not set `validate = false`) and fails, consuming the victim's fee budget and leaving the victim's legitimate invoke unexecuted. This matches:

> **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The attack requires no privileged access. The only precondition is that the victim has a `deploy_account` pending in the public mempool, which is observable by anyone. The window between `deploy_account` submission and block inclusion is typically multiple seconds to minutes, giving the attacker ample time to front-run. The attack is repeatable for every new account deployment.

### Recommendation

Do not skip signature verification based solely on mempool membership. The intended UX goal (allowing `deploy_account + invoke` to be submitted atomically) can be preserved while still verifying the signature by one of the following approaches:

1. **Verify the signature before skipping `__validate__`**: Run a lightweight ECDSA/Stark-curve check on the raw signature bytes against the transaction hash before granting the skip. This does not require executing the account contract.
2. **Restrict the skip to the account's own submission**: Require that the `deploy_account` in the mempool was submitted in the same gateway request (e.g., via a session token or by checking that the `deploy_account` tx hash matches a field in the invoke).
3. **Remove the skip entirely and rely on the batcher's re-execution tolerance**: Accept that the first invoke after deployment will be validated in the next block rather than the same block as the deployment.

### Proof of Concept

```
1. Alice submits deploy_account for address A (nonce 0).
   → Mempool now contains Alice's deploy_account; account_tx_in_pool_or_recent_block(A) == true.

2. Bob (attacker) observes Alice's deploy_account in the public mempool.

3. Bob crafts an invoke:
     sender_address = A
     nonce          = 1
     signature      = [0xdeadbeef, ...]   ← arbitrary, invalid

4. Bob submits the invoke to the gateway.

5. Gateway stateful validation:
   a. validate_state_preconditions → nonce 1 >= account_nonce 0 ✓
   b. validate_by_mempool          → nonce ordering OK, fee OK ✓
   c. skip_stateful_validations:
        tx.nonce() == 1  ✓
        account_nonce == 0  ✓
        account_tx_in_pool_or_recent_block(A) == true  ✓
      → returns skip_validate = true
   d. run_validate_entry_point with execution_flags.validate = false
      → __validate__ is NOT called; signature is NOT checked ✓

6. Bob's invalid invoke is admitted to the mempool.
   If fee escalation is enabled and Bob's tip > Alice's tip,
   Alice's valid invoke (nonce 1) is evicted.

7. Batcher builds block:
   - Executes Alice's deploy_account (success, account A deployed).
   - Executes Bob's invalid invoke:
       __validate__ now runs (validate = true in batcher path).
       Signature check fails → transaction reverted.
       Fee is charged from A's balance.

8. Alice's legitimate invoke is gone; her account has been charged a fee
   for a transaction she never signed.
``` [9](#0-8) [10](#0-9) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
