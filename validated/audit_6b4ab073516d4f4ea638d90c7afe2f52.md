### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Any Account With a Pending Deploy-Account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX shortcut for the deploy-account + invoke flow unconditionally skips `__validate__` (signature verification) for **any** invoke transaction with nonce=1 targeting an account whose on-chain nonce is 0 and that has *any* transaction in the mempool. An attacker who observes a victim's `deploy_account` in the mempool can immediately submit a forged invoke transaction with nonce=1 for the victim's address, carrying arbitrary calldata and an invalid/empty signature, and the gateway will admit it without ever calling the account's `__validate__` entry point.

---

### Finding Description

`skip_stateful_validations` (lines 429–461) returns `true` — meaning "skip the validate entry point" — when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`.
2. The account's current on-chain nonce is `0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_validate` is `true`, `run_validate_entry_point` constructs `ExecutionFlags` with `validate: false`: [2](#0-1) 

This means `StatefulValidator::validate` is called with `validate=false`, so the account's `__validate__` entry point — the only place the signature is checked — is never invoked. The three earlier checks (`validate_state_preconditions`, `validate_by_mempool`) verify nonce range and resource bounds but perform no cryptographic signature verification. [3](#0-2) 

The critical flaw is that condition 3 is checked against the **victim's** address, not the attacker's. The function comment acknowledges this is a trust-by-presence check:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [4](#0-3) 

There is no binding between the deploy_account submitter and the invoke submitter. Any third party who observes a `deploy_account` for address `A` in the mempool can submit `Invoke { sender: A, nonce: 1, calldata: <anything>, signature: <garbage> }` and the gateway will accept it.

When the batcher later sequences the block, it uses `new_for_sequencing`, which sets `validate: true`: [5](#0-4) 

So the forged invoke will fail at execution time. However, by then it has already been admitted to the mempool. If the mempool enforces one transaction per `(account, nonce)` slot (standard behavior), the attacker's forged tx can **replace** the victim's legitimate nonce-1 invoke, permanently breaking the victim's deploy-account + invoke UX flow.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway admits a transaction whose signature has never been verified. The forged transaction occupies the victim's nonce-1 slot in the mempool. When executed it reverts (blockifier re-validates), but the victim's legitimate invoke is displaced and lost. Every user performing a deploy_account + invoke in a single round-trip is vulnerable during the window between mempool admission of their deploy_account and its inclusion in a block.

---

### Likelihood Explanation

Any party monitoring the public mempool can observe pending `deploy_account` transactions. The attack requires only constructing a well-formed invoke transaction (valid resource bounds, nonce=1, any calldata, any signature) for the victim's address and submitting it to the gateway. No privileged access, no special tooling, and no on-chain funds beyond the gas fee for the forged tx are required. The attack window is the latency between deploy_account mempool admission and block inclusion — typically tens of seconds on a live sequencer.

---

### Recommendation

1. **Bind the skip to the same sender.** Before returning `true`, verify that the mempool entry for the account is specifically a `DeployAccount` transaction (not just any transaction), and that the invoke's sender matches the deploy_account's contract address derivation. This prevents a third party from exploiting the skip for a victim's address.

2. **Alternatively, remove the skip entirely.** The UX benefit (submitting deploy_account + invoke atomically) can be preserved by having the wallet sign the invoke with a pre-image that does not require the account to exist yet, or by having the sequencer batch them internally without skipping validation.

3. **At minimum, add a check** that the account does not already have a nonce-1 transaction in the mempool before admitting a second one under the skip path.

---

### Proof of Concept

```
1. Alice submits DeployAccount { address: 0xALICE, nonce: 0, ... } to the gateway.
   → Mempool now contains a tx for 0xALICE.

2. Attacker observes 0xALICE in the mempool.

3. Attacker submits Invoke { sender: 0xALICE, nonce: 1,
       calldata: [drain_alice_funds], signature: [0x0] }

4. Gateway stateful validator:
   - account_nonce(0xALICE) == 0  ✓
   - tx.nonce() == 1              ✓
   - account_tx_in_pool_or_recent_block(0xALICE) == true  ✓
   → skip_validate = true
   → __validate__ NOT called; signature NOT checked.
   → Tx admitted to mempool at nonce-1 slot for 0xALICE.

5. Alice's legitimate Invoke { nonce: 1, calldata: [...], signature: <valid> }
   is now displaced (or rejected as duplicate nonce).

6. Block is produced:
   - DeployAccount executes → 0xALICE deployed.
   - Attacker's Invoke executes → __validate__ called → signature invalid → REVERT.
   - Alice's legitimate invoke is gone from the mempool.
```

Alice's deploy_account + invoke UX is permanently broken for this deployment attempt. The attacker pays only the gas for the forged invoke (cheap on L2).
