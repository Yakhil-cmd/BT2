### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Systematic User Overcharging - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` applies a `delta_gas` correction only in one direction: it increases `gas_used` when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), but silently ignores the symmetric case (`delta_gas < 0`) where EVM gas consumption exceeds native resource consumption. A `TODO` comment in the code explicitly acknowledges this gap. The result is that users are systematically overcharged whenever their transaction consumes proportionally more EVM gas than native resources, with the excess fee flowing to the operator.

---

### Finding Description

In `compute_gas_refund`, the double-resource accounting reconciliation step computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
``` [1](#0-0) 

When `delta_gas > 0`, `gas_used` is increased so the user pays for the extra native cost. When `delta_gas < 0` (native resource consumption is *lower* than EVM gas consumption), the code does nothing — `gas_used` is left at its higher EVM-derived value. The comment `// TODO: return delta_gas to gas_used?` at line 80 is an explicit, unresolved acknowledgment of this asymmetry. [2](#0-1) 

The documentation in `double_resource_accounting.md` only describes the `delta_gas > 0` path:

> If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. [3](#0-2) 

The corrected `gas_used` is then used to compute both the operator payment and the user refund in `refund_and_commit_fee`:

```
token_to_refund   = gas_price * (gas_limit - gas_used)   // user gets less
token_to_operator = gas_used  * gas_price                // operator gets more
``` [4](#0-3) [5](#0-4) 

The same `compute_gas_refund` function is called for both L2 transactions (via `ZkTransactionFlow::before_refund`) and L1→L2 priority transactions (via `process_l1_transaction`), so both paths are affected. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

When `delta_gas < 0`, the user is charged `gas_used` gas but the actual native-resource cost is only `native_used / native_per_gas` gas. The overcharge per transaction is:

```
overcharge = (gas_used - native_used / native_per_gas) * gas_price
```

This is a direct, quantifiable loss of ETH/base-token for the transaction sender. The excess is transferred to the operator's coinbase address. There is no cap on the magnitude of the overcharge — it scales with the gap between EVM gas consumption and native resource consumption.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever a transaction consumes proportionally more EVM gas than native (proving) resources. This is a realistic scenario for:

- Transactions dominated by EVM arithmetic or memory operations (cheap in native cycles, expensive in EVM gas).
- Transactions with large EVM gas refunds (e.g., SSTORE clearing) that reduce `gas_used` after the EIP-3529 cap but leave native consumption unchanged.
- Any transaction where the `native_per_gas` ratio is set high relative to actual proving cost.

The `TODO` comment and the asymmetric design confirm this is a known, unresolved code path that is reachable in normal operation.

---

### Recommendation

Apply the symmetric correction: when `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (floored at `minimal_gas_used`):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This ensures `gas_used` accurately reflects the maximum of EVM gas consumed and native resource cost in gas units, preventing both undercharging (already handled) and overcharging (currently unhandled).

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `native_per_gas = 10`
- `full_native_limit = 1_000_000`
- `gas_price = 1_000 wei/gas`

**Execution result:**
- `remaining_ergs = 50_000 * ERGS_PER_GAS` → `gas_used_from_ergs = 50_000`
- `remaining_native = 600_000` → `native_used = 400_000`

**delta_gas calculation:**
```
delta_gas = (400_000 / 10) - 50_000 = 40_000 - 50_000 = -10_000
```

**Current behavior (buggy):** `delta_gas < 0` → no adjustment → `gas_used = 50_000`
- User refund: `(100_000 - 50_000) * 1_000 = 50_000_000 wei`
- Operator fee: `50_000 * 1_000 = 50_000_000 wei`

**Correct behavior:** `gas_used` should be reduced to `40_000`
- User refund: `(100_000 - 40_000) * 1_000 = 60_000_000 wei`
- Operator fee: `40_000 * 1_000 = 40_000_000 wei`

**User loss per transaction: `10_000 * 1_000 = 10_000_000 wei = 0.00001 ETH`** at this gas price. At mainnet gas prices the loss scales proportionally. Any unprivileged sender submitting a standard L2 or L1→L2 transaction triggers this path with no special preconditions.
