### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Verification for Deployed Accounts with Any Pending Nonce-0 Transaction — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call (which performs cryptographic signature verification) for invoke transactions with nonce=1 when the account has not yet been deployed on-chain. The guard that is supposed to confirm a `deploy_account` is pending uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction type in the mempool — not exclusively `deploy_account`. An attacker who controls a deployed account with on-chain nonce=0 can place a valid nonce=0 invoke in the mempool, then submit a nonce=1 invoke with an **arbitrary/invalid signature** that bypasses `__validate__` entirely and is admitted to the mempool.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (from committed state).

When all three hold, the function calls `account_tx_in_pool_or_recent_block(sender_address)` and, if it returns `true`, returns `skip_validate = true`. [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` whenever the address has **any** transaction in the pool — `Invoke`, `Declare`, or `DeployAccount`: [3](#0-2) 

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, `perform_pre_validation_stage` (nonce/fee checks) still runs, but `validate_tx` — the call that executes `__validate__` and verifies the signature — is **skipped entirely**: [5](#0-4) 

`validate_tx` itself confirms: if `!self.execution_flags.validate`, it returns `Ok(None)` immediately without calling the account's `__validate__` entry point: [6](#0-5) 

The code comment in `skip_stateful_validations` asserts the check is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of this reasoning is the flaw: a nonce=0 `Invoke` that passed its own `__validate__` at nonce=0 provides no guarantee that a nonce=1 invoke from the same sender is validly signed. [7](#0-6) 

**Concrete attack path:**

1. Account A is deployed via `deploy` syscall (not `deploy_account`); on-chain nonce = 0.
2. Attacker submits a valid nonce=0 `Invoke` from A. It passes full validation including `__validate__` (account exists, signature valid). It enters the mempool pool.
3. Attacker submits a nonce=1 `Invoke` from A with an **invalid/arbitrary signature**.
4. Gateway evaluates `skip_stateful_validations`: tx is Invoke ✓, `tx.nonce()=1` ✓, `account_nonce=0` ✓, `tx_pool.contains_account(A)=true` (step 2) ✓ → `skip_validate = true`.
5. `run_validate_entry_point` runs with `validate=false`; `__validate__` is never called.
6. The invalid-signature transaction is admitted to the mempool.

The `validate_nonce` check in `validate_state_preconditions` allows nonce=1 when account_nonce=0 because `max_allowed_nonce_gap` defaults to 200: [8](#0-7) 

The mempool's `validate_tx` only checks nonce ordering and fee escalation — it does not verify signatures: [9](#0-8) 

### Impact Explanation

An attacker can inject invoke transactions with arbitrary/invalid signatures into the mempool, bypassing the account's `__validate__` entry point. These transactions will fail during batcher execution (when `__validate__` is called again), but they consume mempool capacity and batcher execution resources. No fees are charged for transactions that fail `__validate__`. This enables free mempool spam and batcher resource exhaustion.

**Impact category**: High — Mempool/gateway admission accepts invalid transactions before sequencing.

### Likelihood Explanation

The precondition (a deployed account with on-chain nonce=0 and a pending nonce=0 invoke in the mempool) is reachable by any unprivileged user who controls a contract deployed via `deploy` syscall. The attack requires no special privileges, no privileged keys, and no protocol-level access. The window is open for as long as the nonce=0 invoke remains unconfirmed in the mempool.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, not just the address presence. Alternatively, `skip_stateful_validations` should only return `true` when the account does not yet have a deployed class hash in state (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`), which is the true invariant the UX feature depends on.

### Proof of Concept

```

// Step 1: Submit valid nonce=0 invoke (passes __validate__).

  "type": "INVOKE",

  "nonce": "0x0",
  "signature": ["<valid_r>", "<valid_s>"],
  ...
}
// → Accepted. A is now in tx_pool.

// Step 2: Submit nonce=1 invoke with invalid signature.
POST /add_transaction
{
  "type": "INVOKE",
  "sender_address": "<A>",
  "nonce": "0x1",
  "signature": ["0xdeadbeef", "0xdeadbeef"],  // invalid
  ...
}
// Gateway: skip_stateful_validations returns true because
//   tx.nonce()==1, account_nonce==0, tx_pool.contains_account(A)==true.
// __validate__ is never called.
// → Accepted into mempool with invalid signature.
// When batcher executes: __validate__ fails, tx rejected, no fee charged.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
