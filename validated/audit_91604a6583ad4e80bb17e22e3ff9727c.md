### Title
`max_nonce_for_validation_skip` Config Field Silently Ignored in Gateway Stateful Validation Skip — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `StatefulTransactionValidatorConfig::max_nonce_for_validation_skip` field is declared, serialized, documented in the config schema, and correctly used in the native-blockifier Python path (`PyValidator`), but is **never read** by the Rust gateway's `skip_stateful_validations` free function. That function hardcodes `Nonce(Felt::ONE)` unconditionally. An operator who sets `max_nonce_for_validation_skip = 0` to disable the validation-skip UX feature gets no effect: the gateway continues to admit nonce-1 invoke transactions without running `__validate__`, bypassing account signature verification at the admission gate.

### Finding Description

`StatefulTransactionValidatorConfig` declares the field:

```rust
pub max_nonce_for_validation_skip: Nonce,
```

with default `Nonce(Felt::ONE)`, serialized and exposed in `config_schema.json` and deployment configs. [1](#0-0) 

The native-blockifier path (`PyValidator::should_run_stateful_validations`) correctly consults this field:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
``` [2](#0-1) 

The Rust gateway path calls `skip_stateful_validations`, a **free function** that receives no config argument and hardcodes the bound:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [3](#0-2) 

`run_pre_validation_checks` (a `&self` method with access to `self.config`) calls this free function but never passes `self.config.max_nonce_for_validation_skip`: [4](#0-3) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, causing `StatefulValidator::perform_validations` to return after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point: [5](#0-4) [6](#0-5) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

If an operator sets `max_nonce_for_validation_skip = 0` to disable the deploy-account UX bypass (e.g., for a stricter sequencer deployment), the Rust gateway ignores the setting. Any invoke transaction with `nonce = 1` submitted alongside a deploy-account transaction in the mempool will have its `__validate__` entry point skipped, meaning the account's signature verification is not performed at the gateway. The transaction is admitted to the mempool and forwarded to the batcher without the account contract ever having validated the caller's authorization. The batcher will execute `__execute__`, but accounts that do not re-verify the signature there (or that have a bug) will process the call.

Conversely, if `max_nonce_for_validation_skip > 1` is configured to extend the UX window, the Rust gateway still only skips for `nonce == 1`, silently rejecting valid transactions that the operator intended to admit.

### Likelihood Explanation

**Low** — the default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded constant, so the bug is invisible with default configuration. It surfaces only when an operator deliberately changes the field. However, the field is fully documented and exposed in production deployment configs, making operator misconfiguration a realistic scenario. [7](#0-6) [8](#0-7) 

### Recommendation

Convert `skip_stateful_validations` from a free function to a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as an explicit argument) so the configured bound is enforced:

```rust
// In run_pre_validation_checks:
let skip_validate = self.skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
).await?;

// In skip_stateful_validations (now a &self method):
if tx.nonce() <= self.config.max_nonce_for_validation_skip
    && account_nonce == Nonce(Felt::ZERO) { ... }
```

Add an integration test that sets `max_nonce_for_validation_skip = 0` and verifies that a nonce-1 invoke transaction is **not** admitted without a valid `__validate__` result.

### Proof of Concept

1. Deploy a gateway with `max_nonce_for_validation_skip = 0` in `StatefulTransactionValidatorConfig`.
2. Pre-fund account address `A` (e.g., via an L1 deposit so it has non-zero balance).
3. Submit a `DeployAccount` transaction for `A` → it enters the mempool.
4. Submit an `Invoke` transaction from `A` with `nonce = 1` and a **deliberately invalid signature**.
5. Observe: `skip_stateful_validations` returns `true` (hardcoded `Nonce(Felt::ONE)` matches), `validate` flag is set to `false`, `__validate__` is never called, and the transaction is admitted to the mempool — contradicting the operator's explicit `max_nonce_for_validation_skip = 0` configuration.

The root cause is the same structural pattern as the external report: a value (`max_nonce_for_validation_skip`) is computed/stored in the configuration object but never written into the critical execution path, leaving the invariant it was meant to enforce permanently unenforced.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
