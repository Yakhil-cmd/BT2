### Title
Unprivileged Caller Can Permanently Freeze Block Rewards via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

### Summary
The public `update_rewards` function in the `Staking` contract accepts a `disable_rewards: bool` parameter with no access control beyond `general_prerequisites()`. When called with `disable_rewards: true`, it consumes the global `last_reward_block` slot for the current block without distributing any rewards, permanently freezing that block's unclaimed yield. Any unprivileged caller can repeat this every block to halt all reward distribution indefinitely.

### Finding Description

`update_rewards` is exposed as a public external function under `StakingRewardsManagerImpl`: [1](#0-0) 

Its only gate is `general_prerequisites()`, which checks only that the contract is not paused and the caller is not the zero address: [2](#0-1) 

Inside the function, the global `last_reward_block` is written unconditionally before the `disable_rewards` branch: [3](#0-2) 

When `disable_rewards` is `true`, the function returns immediately after writing `last_reward_block`, distributing nothing. Any subsequent call to `update_rewards` in the same block then fails: [4](#0-3) 

Because `last_reward_block` is a single global slot shared across all stakers, one call with `disable_rewards: true` in block N prevents every staker from receiving rewards for block N. There is no mechanism to retroactively recover a missed block's rewards — `calculate_block_rewards` computes rewards for the current epoch only and does not account for skipped blocks: [5](#0-4) 

The attacker only needs to supply any valid, active staker address (observable from `NewStaker` events) and call the function once per block. The gas cost is the only barrier.

**Analogy to the CDP report:** In the CDP system, redemptions bypass the recovery-mode check that all other state-modifying operations must pass, allowing an operation to push the system into an undercollateralized state. Here, `update_rewards` with `disable_rewards: true` bypasses the reward-distribution logic that the normal call path enforces, allowing an unprivileged caller to consume the block's reward slot without the expected side effect (reward distribution). Both are state-transition inconsistencies where one code path skips a critical invariant that all other paths enforce.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Each block's rewards are computed and credited only when `_update_rewards` is reached. If the block slot is consumed by a `disable_rewards: true` call, those rewards are never credited to any staker or pool and cannot be recovered. An attacker calling once per block permanently halts all reward accrual for every staker and every delegation pool in the protocol.

### Likelihood Explanation

**High.** The attack requires no privileged role, no capital, and no special setup — only knowledge of one active staker address (trivially obtained from on-chain events) and the ability to submit a transaction before the legitimate caller each block. The cost is gas only. A motivated griever can sustain this indefinitely.

### Recommendation

Restrict who may pass `disable_rewards: true`. Options include:

1. **Remove the parameter entirely** from the public interface and handle the disable case through a separate privileged function (e.g., callable only by a `SECURITY_AGENT` role).
2. **Add a caller check**: only the staker themselves, their reward address, or a designated operator role should be permitted to call `update_rewards` with `disable_rewards: true`.
3. **Decouple slot consumption from reward skipping**: write `last_reward_block` only when rewards are actually distributed, so a `disable_rewards: true` call does not block legitimate callers in the same block.

### Proof of Concept

```
// Attacker (any address) calls once per block:
staking.update_rewards(
    staker_address: any_valid_active_staker,  // observable from NewStaker events
    disable_rewards: true,
);
// Result:
// - last_reward_block = current_block_number
// - No rewards distributed to any staker
// - Any subsequent update_rewards call in this block reverts with REWARDS_ALREADY_UPDATED
// - Block N's rewards are permanently lost
// Repeat every block → all reward distribution halted indefinitely
```
