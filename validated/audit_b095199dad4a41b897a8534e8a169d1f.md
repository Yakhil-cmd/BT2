### Title
Unsigned Invoke Transaction Admitted to Mempool via `skip_stateful_validations` Signature-Check Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry point (the account's on-chain signature verification) for any invoke transaction with `nonce == 1` when the sender's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** address that has **any** transaction in the mempool pool or committed-block state — not specifically a deploy-account owned by the same submitter — an unprivileged attacker can submit an invoke with an arbitrary or empty signature from a victim address that has a pending `deploy_account`, and the gateway will admit it to the mempool without ever calling `__validate__`.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip `__validate__`) when all three conditions hold:

1. The transaction is `Invoke` with `nonce == 1`.
2. The account's committed-state nonce is `0` (account not yet deployed on-chain).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

**What `account_tx_in_pool_or_recent_block` actually checks** [2](#0-1) 

It returns `true` if the address appears in the mempool's internal state **or** the tx-pool — regardless of whether the matching transaction is a `deploy_account` or whether the submitter of the new invoke is the same party who submitted the earlier transaction.

**What is skipped**

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by `execution_flags.validate`: [4](#0-3) 

`perform_pre_validation_stage` (nonce increment, fee-bound check, balance check) still runs, but the account's `__validate__` entry point — which is the sole on-chain mechanism for verifying the transaction signature — is never called.

**Checks that do pass for the attacker's transaction**

`run_pre_validation_checks` calls three things before deciding to skip: [5](#0-4) 

- `validate_state_preconditions` → `validate_nonce`: for account_nonce=0 and tx_nonce=1, the condition `0 ≤ 1 ≤ max_allowed_nonce_gap` passes as long as `max_allowed_nonce_gap ≥ 1`. [6](#0-5) 

- `validate_by_mempool`: the mempool's `validate_tx` checks nonce validity against the mempool's internal account nonce (0 for an undeployed account), and a tx_nonce of 1 is within the allowed gap. [7](#0-6) 

None of these checks verify the transaction signature.

---

### Impact Explanation

An attacker who observes that address `V` has a pending `deploy_account` in the mempool can submit an `Invoke(sender=V, nonce=1, calldata=<arbitrary>, signature=<fake>)`. The gateway admits this transaction to the mempool without ever running `V`'s `__validate__` entry point. The mempool now contains an invalid, unsigned transaction attributed to `V`.

- **Mempool pollution / DoS**: The attacker can flood the mempool with invalid transactions from every address that has a pending `deploy_account`, consuming mempool capacity and preventing legitimate transactions from being admitted.
- **Batcher resource waste**: The batcher pulls these transactions, runs full execution (which includes `__validate__` with `strict_nonce_check=true`), fails validation, and discards them — wasting sequencing resources.
- **Admission of invalid transactions**: The invariant "every invoke in the mempool has passed signature verification" is broken.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The trigger is entirely unprivileged. Any observer of the public mempool can identify addresses with pending `deploy_account` transactions (these are visible in the mempool snapshot or via P2P gossip). The attacker only needs to craft an `Invoke` with the victim's sender address and nonce=1; no key material, no privileged access, and no on-chain funds are required from the attacker. The `deploy_account + invoke` UX pattern is explicitly supported and documented, making victim addresses easy to find.

---

### Recommendation

Replace the weak `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction (not just any transaction) is pending for the sender address. Alternatively, require that the invoke's transaction hash is co-signed with the deploy_account (i.e., the two transactions are submitted together as an atomic bundle), so the gateway can verify the same key controls both. A minimal fix is to expose a `deploy_account_in_pool(address)` query on the mempool that returns `true` only when the pending transaction for that address at nonce 0 is specifically a `DeployAccount` type, and use that in `skip_stateful_validations` instead of the generic `account_tx_in_pool_or_recent_block`.

---

### Proof of Concept

**Setup**: Account `V` at address `0xABCD` has been funded with STRK. `V`'s owner submits `deploy_account(nonce=0, class_hash=..., salt=..., signature=<valid>)`. This transaction is accepted by the gateway and sits in the mempool. `V`'s on-chain nonce remains `0`.

**Attack**:

```
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "version": "0x3",
  "sender_address": "0xABCD",   // victim address
  "nonce": "0x1",               // exactly 1
  "calldata": ["0xDEAD", ...],  // arbitrary malicious calldata
  "signature": [],              // empty / fake signature
  "resource_bounds": { "l2_gas": { "max_amount": "0x1000", "max_price_per_unit": "0x1" }, ... },
  ...
}
```

**Gateway trace**:

1. `StatelessTransactionValidator::validate` — passes (signature size 0 ≤ max, calldata within bounds, DA modes L1).
2. `convert_rpc_tx_to_internal` — computes tx hash with `chain_id`; no signature check here.
3. `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(0xABCD)` → `0` (not deployed).
   - `validate_state_preconditions`: nonce `0 ≤ 1 ≤ max_allowed_nonce_gap` → **passes**.
   - `validate_by_mempool`: mempool account nonce for `0xABCD` is `0`; tx_nonce `1` within gap → **passes**.
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0` → calls `account_tx_in_pool_or_recent_block(0xABCD)` → `true` (deploy_account is in pool) → **returns `true`**.
   - `run_validate_entry_point(skip_validate=true)`: sets `execution_flags.validate=false`; `perform_pre_validation_stage` runs (nonce/fee/balance OK); `__validate__` is **never called**.
4. Transaction is forwarded to `mempool.add_tx` → **admitted**.

**Result**: An invoke with an empty signature from `0xABCD` is now in the mempool. When the batcher eventually pulls it and runs full execution with `validate=true`, it will fail `__validate__` and be discarded — but the mempool slot was consumed and batcher resources were wasted. Repeating this for every address with a pending `deploy_account` constitutes a practical DoS.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
