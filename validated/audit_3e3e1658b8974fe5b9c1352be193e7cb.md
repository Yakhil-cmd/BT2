### Title
COA Ownership Proof Signature Replay Across Multiple COAs — (`fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies that provided signatures are valid over `signedData` using a Cadence account's keys, but does not enforce that `signedData` encodes the target `evmAddress`. A valid proof obtained for one COA owned by a Cadence account can be replayed verbatim to prove ownership of any other COA owned by the same account, bypassing authentication in any Cadence-based flow that calls this function directly.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies that `signatures` are valid over `signedData` using the Cadence account's keys (line 1082–1086).
2. Verifies that the COA resource at `path` has an EVM address matching the caller-supplied `evmAddress` (lines 1095–1109). [1](#0-0) 

The critical gap is that `signedData` is never checked to contain `evmAddress`. The two checks are entirely decoupled. The code itself documents this at lines 1003–1009:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [2](#0-1) 

The function signature accepts all parameters from the caller with no binding between them: [3](#0-2) 

The function is `access(all)` — any unprivileged Cadence script or transaction can call it with attacker-chosen `path` and `evmAddress` while reusing a previously observed `signedData`/`signatures` pair.

**Exploit path:**

1. Victim's Cadence account owns two COAs: COA1 at `/public/coa1` (EVM address `0xAAA`) and COA2 at `/public/coa2` (EVM address `0xBBB`). The protocol supports multiple COAs per account at different storage paths.
2. Victim produces a valid proof for COA1: signs arbitrary `signedData` with their Cadence key.
3. Attacker observes this proof (e.g., from an on-chain transaction or off-chain authentication exchange).
4. Attacker calls `EVM.validateCOAOwnershipProof(address: victim, path: /public/coa2, signedData: <same>, keyIndices: <same>, signatures: <same>, evmAddress: 0xBBB)` from a Cadence script.
5. Step 1 passes: the signatures are valid over `signedData` (unchanged).
6. Step 2 passes: the COA at `/public/coa2` does have EVM address `0xBBB`.
7. The function returns `isValid: true` for COA2 — the attacker has proven ownership of a COA they do not control.

The Go-layer entry point that decodes the ABI-encoded proof and dispatches to `validateCOAOwnershipProof` is `coaOwnershipProofValidator` in `fvm/evm/handler/precompiles.go`: [4](#0-3) 

The `COAOwnershipProofInContext` struct confirms that `SignedData` and `EVMAddress` are independent fields with no enforced relationship: [5](#0-4) 

### Impact Explanation

Any Cadence contract or off-chain authentication flow that calls `EVM.validateCOAOwnershipProof` directly and uses the result to gate access to EVM assets or privileges is vulnerable to cross-COA impersonation. An attacker who observes a single valid proof for any COA owned by a Cadence account can replay it to impersonate that account as the owner of every other COA it controls. This constitutes unauthorized authorization bypass — the attacker gains authenticated access to EVM addresses they do not own.

The EVM-side path via the `verifyCOAOwnershipProof` precompile (called from `coa.sol`'s `isValidSignature`) is **not** affected because it always passes `address(this)` as the EVM address argument, which the attacker cannot substitute. [6](#0-5) 

### Likelihood Explanation

**Medium-low.** Two preconditions must hold:

1. The victim's Cadence account owns more than one COA (stored at different public paths). The protocol supports this; it is not the common case today but is architecturally permitted and will become more common as EVM usage grows.
2. A Cadence contract or off-chain flow calls `EVM.validateCOAOwnershipProof` directly for authentication decisions. The function is `access(all)` and is the documented API for Cadence-side COA ownership verification.

The attacker needs no privileged access — only the ability to submit a Cadence script and observe a previously used proof.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes the target `evmAddress`. Concretely, before verifying signatures, check that the first 20 bytes of `signedData` equal `evmAddress`. Alternatively, reconstruct the expected signed payload as `hash(evmAddress || applicationData)` and verify signatures over that, analogous to EIP-712 domain separation. This binds each proof to exactly one COA address and eliminates cross-COA replay.

### Proof of Concept

```cadence
// Attacker Cadence script — no privileges required
import EVM from <EVM_CONTRACT_ADDRESS>

access(all)
fun main(
    victimAddress: Address,
    // proof originally produced for COA1 at /public/coa1
    originalSignedData: [UInt8],
    originalKeyIndices: [UInt64],
    originalSignatures: [[UInt8]],
    // COA2's EVM address — different from COA1
    coa2EVMAddress: [UInt8; 20]
): EVM.ValidationResult {
    // Replay the same signedData+signatures against COA2
    return EVM.validateCOAOwnershipProof(
        address: victimAddress,
        path: /public/coa2,          // different path → different COA
        signedData: originalSignedData,
        keyIndices: originalKeyIndices,
        signatures: originalSignatures,
        evmAddress: coa2EVMAddress   // COA2's address
    )
    // Returns isValid: true — attacker is now authenticated as owner of COA2
}
```

The signature verification at line 1082 passes because `signedData` is unchanged and the Cadence account's keys are valid. The EVM address check at line 1095 passes because the attacker supplied COA2's actual address and path. No check binds the two together. [7](#0-6)
