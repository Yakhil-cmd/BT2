### Title
Unauthorized Caller Can Permanently Deny Block Rewards via `update_rewards` - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the `Staking` contract has no caller authorization check, despite the protocol specification explicitly requiring "Only starkware sequencer" access. Any unprivileged caller can invoke `update_rewards` with `disable_rewards: true` to consume the per-block reward slot without distributing rewards, permanently preventing the legitimate sequencer from distributing rewards for that block.

---

### Finding Description

`update_rewards` is defined in `StakingRewardsManagerImpl` and is the sole mechanism for distributing per-block consensus rewards to stakers and their delegation pools. [1](#0-0) 

The function begins with only `self.general_prerequisites()`, which checks that the contract is not paused and the caller is not the zero address — there is no check that the caller is the authorized sequencer. [2](#0-1) 

Critically, `last_reward_block` — a **global** storage variable — is written unconditionally before the `disable_rewards` branch: [3](#0-2) 

Once `last_reward_block` is set to the current block number, any subsequent call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The protocol specification explicitly states the intended access control for this function: [5](#0-4) 

The storage field is global (not per-staker), meaning a single call affects reward eligibility for **all** stakers in that block: [6](#0-5) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker calling `update_rewards(any_valid_staker, disable_rewards: true)` every block permanently prevents the sequencer from distributing consensus rewards to any staker. Because `last_reward_block` is a single global value, one call per block is sufficient to deny rewards to the entire protocol. Stakers and delegators accumulate zero consensus rewards indefinitely. This matches the allowed impact: *"Permanent freezing of unclaimed yield."*

---

### Likelihood Explanation

**High.** The function is publicly callable by any non-zero address. No special role, token balance, or prior state is required beyond knowing a valid `staker_address` (trivially observable on-chain via `NewStaker` events). The attack costs only gas per block and can be automated. There is no economic barrier.

---

### Recommendation

Add a sequencer address to contract storage and assert the caller matches it at the start of `update_rewards`, consistent with the specification:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
}
```

Alternatively, use the existing `RolesComponent` to define a `SEQUENCER_ROLE` and gate the function with `self.roles.only_sequencer()`.

---

### Proof of Concept

1. Alice stakes STRK and, after `K` epochs, becomes eligible for consensus block rewards.
2. Eve (any unprivileged address) monitors the chain. At each new block `B`:
   - Eve calls `update_rewards(alice_address, disable_rewards: true)`.
   - This passes all checks (contract not paused, Eve ≠ zero, Alice is a valid active staker, `B > last_reward_block`).
   - `last_reward_block` is written to `B` at line 1485.
   - The function returns early at line 1487 without distributing any rewards.
3. When the legitimate sequencer subsequently calls `update_rewards(alice_address, disable_rewards: false)` in the same block `B`, it reverts with `REWARDS_ALREADY_UPDATED` because `B > last_reward_block` is now false.
4. Alice (and every other staker) receives zero consensus rewards for block `B`.
5. Eve repeats this every block. The attack is confirmed by the existing test suite: [7](#0-6) 

This test demonstrates that calling `update_rewards` with `disable_rewards: true` blocks all subsequent calls for that block — including calls with `disable_rewards: false` — confirming the permanent denial-of-yield impact.
