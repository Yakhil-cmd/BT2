### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions When Any Sender Transaction Exists in Pool - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point call—and therefore all account signature verification—for any invoke transaction with `nonce=1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because that helper returns `true` for **any** transaction from the sender (not specifically a `deploy_account`), an attacker who has placed one valid `deploy_account` in the pool can immediately submit arbitrarily many invoke transactions carrying completely invalid signatures. All of them pass gateway admission and enter the mempool without any signature check.

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

When the three conditions are met (`Invoke`, `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true`), the function returns `true`. The caller `run_pre_validation_checks` passes this flag to `run_validate_entry_point`: [2](#0-1) 

`run_validate_entry_point` then constructs `ExecutionFlags { validate: !skip_validate, … }` and calls `blockifier_validator.validate(account_tx)`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `Invoke` branch checks `if !tx.execution_flags.validate { return Ok(()); }` and exits immediately—`__validate__` is never called: [4](#0-3) 

**The over-broad pool check:**

`account_tx_in_pool_or_recent_block` checks for **any** transaction from the sender, not specifically a `deploy_account`: [5](#0-4) 

The comment in `skip_stateful_validations` says "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning holds for the legitimate UX case, but it also means that once a valid `deploy_account` is in the pool, the attacker can submit invoke transactions with completely invalid signatures and they will all be admitted.

**Batcher always re-enables `validate`:**

When the batcher later executes these transactions, `AccountTransaction::new_for_sequencing` always sets `validate: true`: [6](#0-5) 

So `validate_tx` does call `__validate__`, which fails on the invalid signature. The transaction is **rejected** (not reverted), and **no fee is charged**. The attacker therefore pays only the cost of the initial `deploy_account` and can flood the mempool with signature-invalid invoke transactions at zero marginal cost.

**`validate_tx` respects the flag:** [7](#0-6) 

### Impact Explanation
This matches the allowed High impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."** The gateway admits invoke transactions whose signatures have never been verified. The mempool is polluted with transactions that will always be rejected by the batcher, consuming mempool capacity and batcher execution resources with no fee penalty to the attacker.

### Likelihood Explanation
The attacker needs only to:
1. Pre-fund a fresh address with enough STRK to cover one `deploy_account` fee.
2. Submit a valid `deploy_account` transaction (which passes full gateway validation including `__validate_deploy__`).
3. Immediately submit any number of invoke transactions with `nonce=1` and arbitrary/invalid signatures.

No privileged access is required. The cost is a single `deploy_account` fee; all subsequent invalid invokes are free.

### Recommendation
`skip_stateful_validations` should verify that the transaction in the pool is specifically a `deploy_account` transaction, not just any transaction. The mempool API should expose a `deploy_account_in_pool(sender_address) -> bool` query that only returns `true` when a `deploy_account` transaction is present for that sender. Alternatively, the gateway should still run a lightweight signature check even when skipping the full entry-point execution.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
