### Title
Changing ETH Connector Contract Account Mid-Flight Causes In-Flight Deposits to Fail - (`engine/src/contract_methods/connector.rs`)

### Summary

The `set_eth_connector_contract_account` function allows the owner to swap the ETH connector account at any time with no guard against in-flight NEAR receipts. Because NEAR processes cross-contract calls asynchronously, a `ft_transfer_call` receipt already queued from the **old** connector will call `ft_on_transfer` on the engine after the connector pointer has been updated. The engine's routing check then misidentifies the deposit as an ERC-20 transfer, causing it to fail and temporarily freezing the user's bridged funds.

---

### Finding Description

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` routes incoming token transfers by comparing the NEAR predecessor account to the stored connector account ID:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

The stored connector account is written by `set_eth_connector_contract_account`:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

There is **no check** for in-flight receipts before overwriting the connector pointer.

In NEAR Protocol, `ft_transfer_call` on the old connector creates an asynchronous receipt that calls `ft_on_transfer` on the engine in a subsequent block. If the owner calls `set_eth_connector_contract_account` between the creation of that receipt and its execution, the engine will evaluate:

```
predecessor_account_id (old connector) != get_connector_account_id(&io) (new connector)
```

and fall into the `receive_erc20_tokens` branch. Because the old ETH connector is not registered as a NEP-141 ERC-20 token in the engine, `receive_erc20_tokens` returns an error. The error-handling path then returns the full deposited amount to the old connector:

```rust
let amount_to_return = if let Err(_err) = &result {
    sdk::log!("Error in ft_on_transfer: {_err:?}");
    args.amount.as_u128()
} else {
    0
};
```

The deposit fails and the user must retry after the connector switch is complete.

The same class of issue applies to `factory_set_wnear_address` (`engine/src/contract_methods/xcc.rs`): `withdraw_wnear_to_router` reads the wNEAR address from storage **at callback execution time**, not at promise-creation time:

```rust
let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
```

If `factory_set_wnear_address` is called between when the XCC precompile schedules the `withdraw_wnear_to_router` self-call and when that self-call executes, the engine will attempt to withdraw from the **new** wNEAR ERC-20 contract. The user's balance is in the **old** wNEAR contract, so the EVM sub-call fails, the XCC operation aborts, and the user's cross-contract call is dropped.

---

### Impact Explanation

**High — Temporary freezing of funds.**

Any ETH deposit initiated via `ft_transfer_call` on the old connector that is still in the NEAR receipt queue at the moment the connector pointer is updated will fail. The tokens are refunded to the old connector (and ultimately to the user), but the deposit is lost and must be retried. Similarly, any XCC operation whose `withdraw_wnear_to_router` callback has not yet executed will fail silently, dropping the user's scheduled cross-contract call. In both cases funds are not permanently destroyed, but they are temporarily inaccessible and the user-visible operation is lost.

---

### Likelihood Explanation

**Low–Medium.** The owner must change the connector account (or wNEAR address) while at least one deposit or XCC receipt is still pending. NEAR block times are ~1 second, so the window is narrow but real, especially during urgent migrations (e.g., patching a critical vulnerability in the old connector). The owner has no on-chain signal that the queue is empty, so the unsafe change can happen inadvertently.

---

### Recommendation

Before calling `set_eth_connector_contract_account` or `factory_set_wnear_address`:

1. **Pause the contract** (`pause_contract`) to stop new deposits and XCC calls from entering the queue.
2. **Wait** until all in-flight NEAR receipts from the old connector/wNEAR contract have been processed (observable off-chain via the NEAR indexer).
3. **Perform the pointer update.**
4. **Unpause the contract** (`resume_contract`).

Alternatively, add an on-chain guard that rejects the update unless the contract is already paused, mirroring the pattern recommended for the Dispenser/VoteWeighting migration in the referenced report.

---

### Proof of Concept

**ETH deposit scenario:**

1. Alice calls `ft_transfer_call(receiver_id=aurora, amount=1 ETH, msg=...)` on the old ETH connector. The old connector queues a NEAR receipt to call `ft_on_transfer` on the Aurora engine.
2. Before that receipt is processed, the owner calls `set_eth_connector_contract_account(new_connector, ...)`.
3. The queued receipt executes. `ft_on_transfer` is called with `predecessor_account_id = old_connector`.
4. `get_connector_account_id(&io)` now returns `new_connector`. The equality check fails.
5. `receive_erc20_tokens(old_connector, ...)` is called. The old connector is not registered as an ERC-20 token → error.
6. `amount_to_return = args.amount` → full refund returned to old connector → old connector refunds Alice.
7. Alice's deposit is lost; she must retry after the migration.

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;
        if matches!(handler.promise_result_check(), Some(false)) {
            return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
        }
        let args: WithdrawWnearToRouterArgs = io.read_input_borsh()?;
        let current_account_id = env.current_account_id();
        let recipient = AccountId::try_from(format!(
            "{}.{}",
            args.target.encode(),
            current_account_id.as_ref()
        ))?;
        let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&current_account_id),
            current_account_id,
            io,
            env,
        );
        let (result, ids) = xcc::withdraw_wnear_to_router(
            &recipient,
            args.amount,
            wnear_address,
            &mut engine,
            handler,
        )?;
        if !result.status.is_ok() {
            return Err(b"ERR_WITHDRAW_FAILED".into());
        }
        let id = ids.last().ok_or(b"ERR_NO_PROMISE_CREATED")?;
        handler.promise_return(*id);
        Ok(result)
    })
}
