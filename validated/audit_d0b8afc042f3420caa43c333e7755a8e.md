### Title
Unvalidated `fd` Polynomial Evaluations Allow Malicious Participant to Permanently Corrupt Presignature `alpha`, Denying Signing — (`src/ecdsa/robust_ecdsa/presign.rs`)

---

### Summary

A malicious participant in the robust ECDSA presign protocol can send `fd` polynomial evaluations computed from a polynomial whose constant term is nonzero. Because the protocol performs no commitment, proof, or zero-constant-term check on received `fd` shares, the corrupted evaluations are silently accumulated into every honest participant's `d_me` share. This makes `alpha_me = c_me + d_me` wrong for all honest parties, causing `compute_signature_share` to produce an invalid `s`, and `sig.verify()` to fail with `AssertionFailed`. The attack is repeatable for every presignature session, permanently denying signing.

---

### Finding Description

**Protocol requirement (from spec):**

The spec mandates that each participant's `fd_i` polynomial must have its constant term set to zero:

> "Each Pi generates three random degree 2t polynomials fb_i, fd_i, and fe_i and **set their constant terms to zero**." [1](#0-0) 

This is enforced for honest participants via `zero_secret_polynomial`: [2](#0-1) 

**Why the zero constant term matters mathematically:**

The signing correctness requires that `sum_i alpha_i * lambda_i = k^{-1}`. Expanding:

```
sum_i alpha_i * lambda_i
  = sum_i (c_i + d_i) * lambda_i
  = c(0) + d(0)
  = w^{-1} * a(0) + d(0)
  = k^{-1} + d(0)
```

This equals `k^{-1}` only if `d(0) = sum_i fd_i(0) = 0`. If any `fd_M` has a nonzero constant term `c_M`, then `d(0) = c_M ≠ 0`, and the aggregate `alpha` is wrong.

**The attack path:**

In Round 1, each participant privately sends five polynomial evaluations `(fk(j), fa(j), fb(j), fd(j), fe(j))` to every other participant `j`: [3](#0-2) 

In Round 2, these are received and accumulated with no validation: [4](#0-3) 

`add_shares` is a plain scalar addition with no checks: [5](#0-4) 

The corrupted `d` share is then used directly to compute `alpha_me`: [6](#0-5) 

**Round 3 checks do not cover `d`:**

The three checks performed in Round 3 are:
1. Exponent interpolation consistency of `R_i` values — validates `fk` only
2. Interpolation of `w` and check `w ≠ 0` — validates `fa * fk + fb` only
3. Exponent interpolation consistency of `W_i = R^{a_i}` and check `W = g^w` — validates `fa` only [7](#0-6) 

None of these checks involve `fd` or `fe`. There is no commitment to `fd`, no zero-constant-term proof, and no way for honest participants to detect the malicious evaluation.

**Signing failure:**

With a corrupted `alpha_me`, `compute_signature_share` produces a wrong `s_me`: [8](#0-7) 

The coordinator sums the wrong shares, and `sig.verify()` fails: [9](#0-8) 

---

### Impact Explanation

Every honest participant's `PresignOutput.alpha` is permanently corrupted for the duration of that presignature. Since presignatures must never be reused (per the documented security requirement), the corrupted presignature is discarded. The attacker can repeat this for every presignature session, permanently denying signing for all honest parties. The attack is undetectable — no honest participant can identify which party sent the malicious `fd` evaluation, and no error is raised during presigning itself.

---

### Likelihood Explanation

Any single malicious participant within the threshold can execute this attack. It requires no cryptographic break, no leaked keys, and no external compromise — only the ability to send a crafted scalar value in Round 1. The attack is trivially repeatable and undetectable.

---

### Recommendation

Bind `fd` (and `fe`) evaluations to a commitment with a zero-constant-term proof. Concretely:

1. In Round 1, each participant should broadcast a `PolynomialCommitment` to their `fd_i` and `fe_i` polynomials (i.e., the vector of `fd_i(0)*G, fd_i(1)*G, ...` coefficient commitments).
2. Each participant should verify that the committed constant term `fd_i(0)*G` equals the group identity before accepting the evaluation.
3. Each participant should verify that the received scalar evaluation `fd_i(j)` is consistent with the broadcast commitment: `fd_i(j)*G == PolynomialCommitment_fd_i.eval_at_participant(j)`.

This is the standard VSS (Verifiable Secret Sharing) approach already used elsewhere in the codebase (e.g., in DKG). The same fix applies to `fe`.

// compute_signature_share produces wrong s_me values.

/// Generates a secret polynomial where the constant term is zero
fn zero_secret_polynomial(
    degree: usize,
    rng: &mut impl CryptoRngCore,
) -> Result<Polynomial, ProtocolError> {
    let secret = Secp256K1ScalarField::zero();
    Polynomial::generate_polynomial(Some(secret), degree, rng)
}

fn compute_signature_share(
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
    participants: &ParticipantList,
    me: Participant,
) -> Result<SerializableScalar<C>, ProtocolError> {
    // (beta_i + tweak * k_i) * delta^{-1}
    let big_r = presignature.big_r;
    let big_r_x_coordinate = x_coordinate(&big_r);
    // beta * Rx + e
    let beta = presignature.beta * big_r_x_coordinate + presignature.e;

    let s_me = msg_hash * presignature.alpha + beta;
    // lambda_i * s_i
    let linearized_s_me = s_me * participants.lagrange::<C>(me)?;
    Ok(SerializableScalar::<C>(linearized_s_me))
