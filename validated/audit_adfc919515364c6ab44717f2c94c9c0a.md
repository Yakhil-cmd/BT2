### Title
WalletConnect Multi-NFT Royalty Amount Understated in Approval Dialog — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible trade amount by the number of NFTs before computing each NFT's royalty contribution. When a WalletConnect `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` request involves two or more NFTs with royalties, the "Total Amount with Royalties" shown in the Electron approval dialog is materially lower than the amount the blockchain will actually deduct. A user who relies on this figure to decide whether to approve the offer will consent to a spend that is larger than what was disclosed.

### Finding Description

`formatAmountWithRoyalties` is called from `withRoyaltyTotals` to compute the `amountWithRoyalties` field that is rendered in the WalletConnect confirmation dialog under the label **"Total Amount with Royalties"**.

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts  lines 363-368
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

In Chia's royalty protocol each NFT's royalty is a percentage of the **full** fungible amount, not a per-NFT share of it. The correct formula is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The code instead computes:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
```

which is smaller by a factor of N (the number of NFTs) when all percentages are equal, and is wrong in general for any N ≥ 2.

**Concrete example** (taken directly from the existing test at line 251–335):

| | Displayed (buggy) | Actual on-chain |
|---|---|---|
| Base amount | 0.0001 XCH | 0.0001 XCH |
| NFT1 royalty (5 %) | 0.0000025 XCH | 0.000005 XCH |
| NFT2 royalty (0.1 %) | 0.000000005 XCH | 0.0000001 XCH |
| **Total shown** | **0.00010255 XCH** | **0.0001051 XCH** |

The discrepancy scales with N and with royalty percentages. For three NFTs each carrying a 10 % royalty on a 1 XCH offer, the dialog shows 1.1 XCH while the blockchain deducts 1.3 XCH — a 0.2 XCH gap.

The `amountWithRoyalties` value is rendered directly in the sandboxed approval iframe:

```tsx
// packages/gui/src/electron/dialogs/Confirm/Confirm.tsx  lines 109-114
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties}{' '}
    {networkPrefix ? networkPrefix.toUpperCase() : 'XCH'}
  </div>
)}
```

The correct server-side path (`OfferBuilderXCHSection.tsx`, lines 44–56) sums each NFT's royalty payment on the full amount without any division, confirming the intended semantics.

### Impact Explanation

A user approving a WalletConnect `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command that involves multiple royalty-bearing NFTs is shown a "Total Amount with Royalties" that is lower than the amount the blockchain will actually spend from their wallet. The user's informed-consent decision is based on a materially incorrect figure. This constitutes display of the wrong spend amount in the WalletConnect approval flow, directly matching the High-impact criterion: *"WalletConnect state that causes a user to approve… the wrong… amount."*

### Likelihood Explanation

Any WalletConnect-connected dApp can craft a `take_offer` or `create_offer_for_ids` request referencing two or more NFTs with royalties. No special privilege is required. Multi-NFT offers are a normal, documented use-case of the Chia offer protocol, so the condition is routinely reachable.

### Recommendation

Remove the `/ BigInt(royaltyPercentages.length)` division. Each NFT's royalty must be applied to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This matches the server-side calculation in `OfferBuilderXCHSection.tsx` and the `nft_calculate_royalties` RPC.

### Proof of Concept

The existing unit test at `parseCommandDisplay.test.ts` line 251 already encodes the buggy expected value (`'0.00010255'`). Changing the assertion to the correct value (`'0.0001051'`) and applying the fix above makes the test pass with the correct figure, while the current code produces the understated amount.

**Step-by-step:**
1. dApp sends `chia_wallet.take_offer` via WalletConnect with an offer string encoding two NFTs: NFT-A (5 % royalty) and NFT-B (0.1 % royalty), requested price 0.0001 XCH.
2. `parseCommandDisplay` calls `formatAmountWithRoyalties` with `amount = 100_000_000n` and `royaltyPercentages = [500, 10]`.
3. `splitAmount = 100_000_000n / 2n = 50_000_000n`.
4. Displayed total: `102_550_000` mojos → **0.00010255 XCH**.
5. Blockchain enforces: `105_100_000` mojos → **0.0001051 XCH**.
6. User approves at the lower displayed figure; wallet spends the higher actual figure. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
