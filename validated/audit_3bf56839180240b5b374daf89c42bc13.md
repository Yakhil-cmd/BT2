### Title
Missing Identity-Point Validation on `app_pk` Allows Coordinator to Extract Confidential Derived Key Directly - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry-point in `src/confidential_key_derivation/protocol.rs` accepts the application's ElGamal public key (`app_pk`) without checking whether it is the group identity element. When `app_pk = ElementG1::identity()`, the ElGamal encryption term vanishes and every participant's share of the confidential key is transmitted in the clear to the coordinator, allowing the coordinator to reconstruct the confidential derived key `msk · H(pk, app_id)` without possessing the application's secret key `app_sk`.

---

### Finding Description

`ckd()` validates participant counts, duplicate participants, and coordinator membership, but performs **no validation on `app_pk`**: [1](#0-0) 

The `app_pk` value flows directly into `compute_signature_share`, where it is used as the ElGamal encryption base: [2](#0-1) 

The critical line is:

```rust
let big_c = big_s + app_pk * y.0;   // line 174
```

When `app_pk = ElementG1::identity()`:

- `app_pk * y.0 = identity` (scalar multiplication of the identity is always the identity)
- Therefore `big_c = big_s = hash_point * private_share.to_scalar()`
  where `hash_point = H(pk ‖ app_id)`

Each participant sends `(norm_big_y, norm_big_c)` to the coordinator, where `norm_big_c = lambda_i · big_s_i`. The coordinator aggregates:

```
sum_norm_big_c = Σ lambda_i · x_i · H(pk, app_id) = msk · H(pk, app_id)
```

This is the confidential derived key itself. The coordinator does not need `app_sk` to unmask it; `sum_norm_big_c` is already the secret output.

Compare this to the identity-point checks that exist elsewhere in the codebase — for example, the `dlogeq` proof module explicitly rejects identity-point inputs: [3](#0-2) 

And the robust ECDSA presign protocol explicitly checks that the reconstructed `R` is not the identity: [4](#0-3) 

No equivalent guard exists for `app_pk` in `ckd()`.

---

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

A coordinator who passes `app_pk = ElementG1::identity()` to `ckd()` receives `sum_norm_big_c = msk · H(pk, app_id)` directly from the aggregation step. This is the exact value that the CKD protocol is designed to keep secret from the coordinator. The entire ElGamal masking layer is silently bypassed, and the confidential derived key is disclosed without any cryptographic break.

---

### Likelihood Explanation

The `ckd()` function is a public library API. The `app_pk` parameter is supplied by the caller at protocol initialization time. A malicious coordinator who also controls the protocol invocation (a documented trust assumption in the CKD design, where the coordinator drives the session) can trivially supply `ElementG1::identity()`. There is no in-protocol mechanism for participants to verify that the `app_pk` they received is the same one the coordinator used, and no library-level guard rejects the identity point. The attack requires only a single malformed function call with no cryptographic capability.

---

### Recommendation

Add an explicit identity-point check for `app_pk` inside `ckd()` before the protocol is started, mirroring the pattern already used in `dlogeq::prove_with_nonce` and the robust ECDSA presign:

```rust
// In ckd(), after the coordinator membership check:
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Additionally, consider adding the same check inside `compute_signature_share` as a defence-in-depth assertion, consistent with the library's existing pattern of validating cryptographic inputs at every layer.

---

    coordinator: Participant,
    me: Participant,

    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,

    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
