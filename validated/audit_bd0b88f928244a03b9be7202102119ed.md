### Title
Missing Cryptographic Verification of Participant Shares in CKD Coordinator Aggregation - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` aggregates participant-supplied `(norm_big_y, norm_big_c)` shares without any cryptographic verification that those shares are correctly formed. A single malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted CKD result that honest parties will accept as valid.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects its own share and then receives shares from every other participant, summing them directly:

```rust
// src/confidential_key_derivation/protocol.rs:44-57
let (mut norm_big_y, mut norm_big_c) =
    compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

let waitpoint = chan.next_waitpoint();

for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

An honest participant's share is computed by `compute_signature_share` as:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` [2](#0-1) 

There is **no zero-knowledge proof, no consistency check, and no commitment scheme** attached to the participant's message. The coordinator performs no verification before adding the received values into the aggregate. Compare this to the DKG protocol, which attaches a `proof_of_knowledge` to every commitment and verifies it before accepting any share: [3](#0-2) 

The CKD protocol has no equivalent guard. A malicious participant `P_j` can send any pair `(big_y_j', big_c_j')` instead of the correctly-formed share. The coordinator will silently incorporate the malicious values into the final output.

---

### Impact Explanation

The final CKD output `(Y, C)` is intended to satisfy:

```
C − app_sk · Y  =  msk · H(pk ‖ app_id)
```

If a malicious participant injects `(big_y_j', big_c_j')` such that `big_c_j' − app_sk · big_y_j' ≠ λ_j · x_j · H(pk ‖ app_id)`, the unmasked confidential key will be wrong. The honest coordinator and all downstream consumers will accept this corrupted output as the legitimate CKD result, because there is no mechanism to detect the deviation.

This matches the **High** allowed impact: *Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. No privileged access, no leaked keys, and no external dependency is required. The attacker only needs to be a valid participant (the role is reachable without privilege) and to deviate from the protocol by sending arbitrary group elements. The attack is deterministic and requires a single malformed message.

---

### Recommendation

Require each participant to attach a zero-knowledge proof of correct share formation alongside `(norm_big_y, norm_big_c)`. Concretely, each participant should prove:

1. Knowledge of scalar `y_i` such that `norm_big_y = λ_i · y_i · G` (a standard Schnorr PoK).
2. That `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`, where `x_i` is the participant's DKG share (verifiable against the public verifying key via a DLEQ proof).

The coordinator must verify both proofs before incorporating any participant's contribution into the aggregate, mirroring the `verify_proof_of_knowledge` pattern already used in `do_keyshare`. [4](#0-3) 

---

### Proof of Concept

1. Run the CKD protocol with `n ≥ 2` participants and one designated coordinator.
2. One participant (the attacker) deviates: instead of calling `compute_signature_share`, it sends `(G, G)` (the generator point for both fields) to the coordinator.
3. The coordinator's loop at lines 50–55 adds `G` to both `norm_big_y` and `norm_big_c` without error.
4. The coordinator outputs `CKDOutput::new(norm_big_y + G, norm_big_c + G)`.
5. When the client calls `ckd_output.unmask(app_sk)`, the result is `msk · H(pk ‖ app_id) + G − app_sk · G`, which is not the correct confidential key.
6. No error is raised at any point; the corrupted output is silently accepted. [5](#0-4)
