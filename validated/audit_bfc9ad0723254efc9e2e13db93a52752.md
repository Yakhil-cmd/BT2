### Title
Declare Transactions Bypass the `max_l2_gas_amount` Gateway Admission Cap - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The stateless gateway validator enforces a hard cap (`max_l2_gas_amount`, default 1,210,000,000) on the `l2_gas.max_amount` resource bound for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. Any submitter can craft a Declare transaction with an arbitrarily large `l2_gas.max_amount` and have it admitted through the gateway and into the mempool, bypassing the cap that protects all other transaction types.

### Finding Description

In `validate_resource_bounds` inside `StatelessTransactionValidator`, the `max_l2_gas_amount` upper-bound check is guarded by a type branch that short-circuits for `RpcTransaction::Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The remaining stateless checks (`ZeroResourceBounds`, `min_gas_price`) still apply to Declare, so the price floor is enforced. But the amount ceiling is not. [2](#0-1) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* threshold against the previous block price; it does not check the *amount* for any transaction type. [3](#0-2) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms that a Declare transaction with `max_amount = 200` passes when `max_l2_gas_amount = 100` — a value that would be rejected for Invoke or DeployAccount. [4](#0-3) 

The default cap is 1,210,000,000 gas units: [5](#0-4) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount = u64::MAX` (or any value above `max_l2_gas_amount`) passes all gateway stateless and stateful checks and is admitted to the mempool. This violates the gateway's stated admission invariant that no transaction may declare more than `max_l2_gas_amount` L2 gas. The impact category is **High — Mempool/gateway admission accepts invalid transactions before sequencing**.

Downstream, the blockifier's `verify_can_pay_committed_bounds` (called from `perform_pre_validation_stage`) computes `max_possible_fee = max_amount × max_price_per_unit` using saturating arithmetic and checks the sender's balance. This limits the economic harm: an attacker needs a balance proportional to the oversized claim to survive blockifier validation. However, the gateway admission invariant is still broken regardless of whether the blockifier later rejects the transaction. [6](#0-5) [7](#0-6) 

### Likelihood Explanation

Exploitability is straightforward and requires no privilege: any user can submit a well-formed Declare transaction (valid Sierra class, valid signature, valid nonce, price above threshold) with `l2_gas.max_amount` set to an arbitrarily large value. The gateway will admit it. The TODO comment in the source confirms the developers are aware of the gap but have not yet closed it.

### Recommendation

Apply the same `max_l2_gas_amount` upper-bound check to Declare transactions, removing the type-branch exemption:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas budget (e.g., for large Sierra programs), introduce a separate `max_l2_gas_amount_declare` configuration parameter rather than removing the check entirely.

### Proof of Concept

1. Construct an `RpcDeclareTransactionV3` with a valid Sierra class, valid signature, nonce matching the account, `l2_gas.max_price_per_unit >= min_gas_price` (e.g., 8,000,000,000), and `l2_gas.max_amount = 10_000_000_000_000` (far above the 1,210,000,000 cap).
2. Submit via the gateway's `add_transaction` RPC endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for Declare.
4. The transaction proceeds to stateful validation and mempool admission.
5. For comparison, submit an identical `RpcInvokeTransactionV3` with the same `l2_gas.max_amount`; it is rejected with `MaxGasAmountTooHigh`.

The existing test `valid_l2_gas_amount_on_declare` already demonstrates step 3 in the test suite, confirming the bypass is the intended (but unguarded) behavior. [4](#0-3) [8](#0-7)

/// Verifies that, given the current state, the account can cover the resource upper bounds.
/// Error may indicate insufficient balance, or some other error.
pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}
