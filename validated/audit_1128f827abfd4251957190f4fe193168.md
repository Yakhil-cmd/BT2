### Title
Multi-NFT Royalty Amount Underestimated in WalletConnect Offer Confirmation Dialog — (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible spend amount by the number of NFTs before computing each NFT's royalty contribution. When a WalletConnect `take_offer` or `create_offer_for_ids` request involves multiple NFTs with royalties, the "Total Amount with Royalties" shown in the confirmation dialog is systematically lower than the amount the blockchain will actually deduct. A user who approves based on the displayed total will pay more than they consented to.

---

### Finding Description

`formatAmountWithRoyalties` is the function that computes the `amountWithRoyalties` field shown in the WalletConnect confirmation dialog under **"Total Amount with Royalties"**:

```ts
// packages/gui/src/electron/commands/parseCommandDisplay.ts  lines 354-375
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  ...
  const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides amount by NFT count
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
``` [1](#0-0) 

The function first computes `splitAmount = amount / N` (where N is the number of NFTs), then applies each NFT's royalty percentage to that split amount. The correct formula applies each royalty to the **full** amount:

| Scenario | Displayed total | Actual total charged |
|---|---|---|
| 1 NFT, 5% royalty | correct | correct |
| 2 NFTs, 5% + 0.1% royalty | `amount + (amount/2)×5.1%` | `amount + amount×5.1%` |
| 2 NFTs, 10% + 10% royalty | `amount + (amount/2)×20%` | `amount + amount×20%` |

The existing test at line 251–334 of `parseCommandDisplay.test.ts` confirms the bug is baked in as expected behavior: with `amount = 100,000,000 mojos` and two NFTs (royalties 500 and 10), the test asserts `amountWithRoyalties: '0.00010255'`, but the correct value is `0.0001051` — a 2× underestimate of the royalty portion. [2](#0-1) 

The `amountWithRoyalties` value is rendered in the WalletConnect confirmation dialog (`Confirm.tsx`) as the authoritative "Total Amount with Royalties" line that the user reads before clicking **Confirm**:

```tsx
// packages/gui/src/electron/dialogs/Confirm/Confirm.tsx  lines 109-114
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties}{' '}
    ...
  </div>
)}
``` [3](#0-2) 

The `parseCommandDisplay` function is invoked for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands: [4](#0-3) 

---

### Impact Explanation

**High** — Corruption of WalletConnect offer state that causes a user to approve the wrong amount.

The blockchain enforces the full royalties regardless of what the GUI displays. A user who reads the confirmation dialog and approves based on the displayed "Total Amount with Royalties" will have more XCH (or CAT) deducted than they consented to. The discrepancy scales with both the number of NFTs and the royalty percentages. For two NFTs each carrying a 10% royalty and a 1,000 XCH offer, the dialog shows 1,100 XCH but the blockchain charges 1,200 XCH — a 100 XCH silent overpayment.

---

### Likelihood Explanation

**Medium.** The attacker must already hold a WalletConnect session with the victim (a prerequisite the user explicitly granted). Once connected, the attacker can craft or relay any offer string containing multiple NFTs with royalties and trigger the `take_offer` command. No additional privileges, key material, or social engineering beyond the existing WalletConnect pairing are required. Multi-NFT offers are a normal, documented use case.

---

### Recommendation

Replace the split-amount logic with per-NFT royalties applied to the full amount:

```ts
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test assertions in `parseCommandDisplay.test.ts` to reflect the corrected values.

---

### Proof of Concept

Given a WalletConnect `chia_wallet.take_offer` request where the offer summary contains:
- `requested.xch = "100000000"` (0.0001 XCH the user must spend)
- `offered` = two NFTs, royalty percentages 500 (5%) and 10 (0.1%)

**Current behavior** (`formatAmountWithRoyalties`):
```
splitAmount = 100000000 / 2 = 50000000
royaltyAmount = (50000000 × 500)/10000 + (50000000 × 10)/10000
             = 2500000 + 50000 = 2550000
displayed total = 102550000 mojos = 0.00010255 XCH
```

**Actual blockchain charge**:
```
royaltyAmount = (100000000 × 500)/10000 + (100000000 × 10)/10000
             = 5000000 + 100000 = 5100000
actual total = 105100000 mojos = 0.0001051 XCH
```

The user sees **0.00010255 XCH** in the "Total Amount with Royalties" line and clicks Confirm. The blockchain deducts **0.0001051 XCH** — twice the royalty portion the dialog indicated. [5](#0-4) [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }
