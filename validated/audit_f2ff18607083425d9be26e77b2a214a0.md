### Title
Rewards Transferred to Pool Contract Are Permanently Frozen When Pool Balance Is Below `min_delegation_for_rewards` - (File: src/pool/pool.cairo)

### Summary
When a delegation pool's total balance is below `min_delegation_for_rewards`, the staking contract still transfers STRK reward tokens to the pool contract, but the pool contract records zero in its `cumulative_rewards_trace`. These tokens are permanently frozen in the pool contract with no recovery mechanism, because pool members can only claim rewards via the cumulative trace.

### Finding Description
The reward distribution flow in `_update_rewards` (`src/staking/staking.cairo`) unconditionally:
1. Calls `update_unclaimed_rewards_from_staking_contract` on the reward supplier, increasing `unclaimed_rewards` by the full `staker_rewards + total_pools_rewards`.
2. Calls `claim_from_reward_supplier` to pull `total_pools_rewards` STRK tokens into the staking contract.
3. Calls `update_pool_rewards`, which for each pool calls `send_rewards_to_delegation_pool` (transferring STRK to the pool contract) and then `pool_dispatcher.update_rewards_from_staking_contract(rewards: pool_rewards, pool_balance: ...)`. [1](#0-0) 

Inside `update_rewards_from_staking_contract` in the pool contract, the cumulative trace is updated by calling `compute_rewards_per_unit`: [2](#0-1) 

`compute_rewards_per_unit` returns **zero** when `total_stake < min_delegation_for_rewards`: [3](#0-2) 

The code comment itself acknowledges this: *"Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case."*

The result is that real STRK tokens are transferred into the pool contract (via `send_rewards_to_delegation_pool`), but the `cumulative_rewards_trace` is incremented by zero. Since `claim_rewards` in the pool contract computes payouts exclusively from the cumulative trace, these tokens can never be claimed by any pool member and are permanently frozen. [4](#0-3) [5](#0-4) 

### Impact Explanation
STRK reward tokens transferred to a pool contract whose total delegated balance is below `min_delegation_for_rewards` are permanently frozen. There is no admin sweep function, no fallback distribution path, and no way for pool members to recover these tokens. Every reward cycle where the pool balance remains below the threshold compounds the frozen amount. This constitutes **permanent freezing of unclaimed yield**.

### Likelihood Explanation
This is reachable by any unprivileged delegator. A delegator who enters a pool with an amount below `min_delegation_for_rewards` (e.g., `STRK_CONFIG.min_for_rewards - 1` for STRK, or any amount below `BTC_CONFIG.min_for_rewards` for BTC pools) causes the entire pool's balance to be below the threshold. The flow test `delegate_min_strk_for_rewards_flow_test` explicitly demonstrates this scenario is reachable and results in zero rewards for the delegator, while the staking contract still forwards STRK to the pool. [6](#0-5) 

For BTC pools, the minimum threshold may be harder to exceed, making this scenario more likely for small or newly opened pools.

### Recommendation
Before transferring STRK tokens to the pool contract, check whether `pool_balance >= min_delegation_for_rewards`. If the pool balance is below the threshold, either:
- Skip the transfer entirely and do not count those rewards in `unclaimed_rewards` (i.e., do not mint them), or
- Redirect the frozen rewards to the staker or back to the reward supplier rather than sending them to the pool contract.

The root fix is to ensure the token transfer and the accounting update are always consistent: tokens should only be sent to the pool if they will be trackable and claimable by pool members.

### Proof of Concept
1. Staker stakes and opens a STRK delegation pool.
2. Delegator enters the pool with `amount = STRK_CONFIG.min_for_rewards - 1` (one unit below the threshold).
3. Staker attests and triggers `update_rewards_from_attestation_contract` (or `update_rewards` in V3).
4. In `_update_rewards`: `total_pools_rewards > 0` is calculated, `claim_from_reward_supplier` pulls STRK into the staking contract, and `send_rewards_to_delegation_pool` transfers STRK to the pool contract.
5. In `update_rewards_from_staking_contract`: `compute_rewards_per_unit` returns `0` because `pool_balance < min_delegation_for_rewards`, so `cumulative_rewards_trace` is incremented by zero.
6. Delegator calls `claim_rewards` → receives 0 STRK.
7. The STRK tokens transferred in step 4 remain in the pool contract with no mechanism to recover them.
8. Each subsequent reward cycle repeats steps 3–7, accumulating more frozen STRK in the pool contract.

        fn update_rewards_from_staking_contract(
            ref self: ContractState, rewards: Amount, pool_balance: Amount,
        ) {
            self.assert_caller_is_staking_contract();

            // `rewards_info` is initialized in the constructor or in the upgrade proccess,
            // so unwrapping should be safe.
            let (_, last) = self.cumulative_rewards_trace.last().unwrap();
            self
                .cumulative_rewards_trace
                .insert(
                    key: self.get_current_epoch(),
                    value: last
                        + self
                            .compute_rewards_per_unit(
                                staking_rewards: rewards, total_stake: pool_balance,
                            ),
                );
        }

        /// Compute the rewards for the pool trace.
        ///
        /// `staking_rewards` is in `STRK_DECIMALS` decimals.
        /// `total_stake` is in the contract's token decimals.
        /// **Note**: Delegation rewards lost when pool balance is less than
        /// `min_delegation_for_rewards`. The staking contract continues to forward
        /// `pool_rewards` to the pool contract even in this case.
        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }

fn delegate_min_strk_for_rewards_flow_test() {
    let cfg: StakingInitConfig = Default::default();
    let mut system = SystemConfigTrait::basic_stake_flow_cfg(:cfg).deploy();
    let stake_amount = system.staking.get_min_stake();
    let staker = system.new_staker(amount: stake_amount);
    system.stake(:staker, amount: stake_amount, pool_enabled: true, commission: Zero::zero());
    let pool = system.staking.get_pool(:staker);

    // Setup delegator.
    let delegate_amount = Pool::STRK_CONFIG.min_for_rewards;
    let delegator = system.new_delegator(amount: delegate_amount);

    // Enter pool with less than min STRK for rewards.
    system.delegate(:delegator, :pool, amount: delegate_amount - 1);

    // Attest.
    system.advance_k_epochs_and_attest(:staker);
    system.advance_epoch();

    // Check rewards.
    system.advance_epoch();
    let rewards = system.delegator_claim_rewards(:delegator, :pool);
    assert!(rewards == Zero::zero());

    // Add to delegation.
    system.add_to_delegation_pool(:delegator, :pool, amount: 1);

    // Attest.
    system.advance_k_epochs_and_attest(:staker);

    // Check rewards.
    system.advance_epoch();
    let rewards = system.delegator_claim_rewards(:delegator, :pool);
    assert!(rewards > Zero::zero());
