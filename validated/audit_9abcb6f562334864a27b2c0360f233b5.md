### Title
Single-Step Owner Transfer in `set_owner` Enables Permanent Loss of Admin Access and Permanent Fund Freeze - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The `set_owner` function in the Aurora Engine contract immediately and irrevocably overwrites `state.owner_id` with the caller-supplied `new_owner` in a single atomic step. There is no two-step confirmation requiring the new owner to accept the role. If the current owner supplies an incorrect NEAR account ID (e.g., a typo, a non-existent account, or a burned address), the owner role is permanently lost. Because `resume_contract`, `upgrade`, `attach_full_access_key`, and all other critical administrative functions are gated exclusively on `require_owner_only`, this mistake results in permanent, irrecoverable loss of all administrative control — and, critically, permanent freezing of all user funds if the contract is ever paused.

---

### Finding Description

`set_owner` in `engine/src/contract_methods/admin.rs` reads the new owner from Borsh-encoded input and immediately commits it to persistent state:

```rust
// engine/src/contract_methods/admin.rs, lines 103–121
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;   // ← immediate, irrevocable write
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [1](#0-0) 

The only validation performed is that the new owner is not identical to the current owner (`ERR_SAME_OWNER`). There is no check that the new account exists on NEAR, no pending-proposal state, and no requirement that the new owner confirm acceptance. The transfer is final the moment the transaction lands.

The `SetOwnerArgs` struct carries a single `new_owner: AccountId` field — a plain string — with no further constraints: [2](#0-1) 

The `require_owner_only` guard used by every privileged function performs a strict equality check against `state.owner_id`: [3](#0-2) 

Once `state.owner_id` is set to an address no one controls, this guard permanently blocks every owner-gated function.

---

### Impact Explanation

The owner exclusively controls the following critical functions (all gated by `require_owner_only`):

| Function | Consequence if owner is lost |
|---|---|
| `resume_contract` | Contract can never be unpaused |
| `pause_contract` | Contract can never be paused for emergency response |
| `upgrade` / `stage_upgrade` | Contract can never be upgraded to fix bugs |
| `attach_full_access_key` | No full-access key can ever be added |
| `resume_precompiles` | Paused precompiles (bridge exits) can never be re-enabled |
| `set_key_manager` | Relayer key management permanently broken |
| `factory_update` | XCC router bytecode can never be updated | [4](#0-3) [5](#0-4) [6](#0-5) 

**Critical path to permanent fund freeze:**

1. The current owner calls `pause_contract` (a routine emergency action).
2. The owner then calls `set_owner` with a mistyped NEAR account ID.
3. `state.owner_id` is immediately set to the wrong address.
4. `resume_contract` requires `require_owner_only` — it now permanently reverts with `ERR_NOT_ALLOWED`.
5. All user ETH, ERC-20 tokens, and bridged assets held in the Aurora Engine contract are permanently frozen with no recovery path.

Even without a prior pause, the inability to ever call `upgrade` means any subsequently discovered critical vulnerability in the contract becomes permanently unresolvable, which can lead to permanent fund loss.

---

### Likelihood Explanation

NEAR account IDs are arbitrary UTF-8 strings (e.g., `aurora`, `multisig.aurora-dao.near`). Typos during governance transitions, multi-sig coordination errors, or copy-paste mistakes from a wrong environment (testnet vs. mainnet account) are realistic operational risks. The `ERR_SAME_OWNER` guard demonstrates the function already anticipates input validation concerns, yet the most dangerous case — an incorrect but syntactically valid account ID — is not caught. The Aurora Engine is a production mainnet contract holding significant user funds, making this a high-consequence operational risk.

---

### Recommendation

Implement a two-step ownership transfer pattern:

1. **`propose_owner(new_owner: AccountId)`** — callable only by the current owner; stores `new_owner` as a `pending_owner` field in `EngineState`. Does not change `owner_id`.
2. **`accept_owner()`** — callable only by the account matching `pending_owner`; finalizes the transfer by setting `owner_id = pending_owner` and clearing `pending_owner`.

This ensures the new owner demonstrably controls the account before the transfer is finalized, making typos and non-existent accounts harmless.

---

### Proof of Concept

```
1. Current owner_id = "aurora-dao.near"
2. Owner calls set_owner({ new_owner: "aurora-doa.near" })  // typo
3. state.owner_id is immediately written as "aurora-doa.near"
4. No account "aurora-doa.near" exists or is controlled by anyone
5. Owner calls resume_contract() → ERR_NOT_ALLOWED (owner_id mismatch)
6. Owner calls upgrade()         → ERR_NOT_ALLOWED
7. Owner calls attach_full_access_key() → ERR_NOT_ALLOWED
8. If contract was paused: all user funds are permanently frozen
9. No on-chain recovery mechanism exists
``` [7](#0-6) [8](#0-7)

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
}
