### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Deployed Accounts with Any Pending Transaction — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is intended to improve UX for the `deploy_account + invoke` pattern by skipping `__validate__` (signature verification) for a nonce-1 invoke when the account is not yet deployed. However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** pending transaction from that address — not only a `deploy_account`. This allows an attacker to submit an invoke transaction with an **invalid signature** for a deployed account, bypassing the gateway's stateful validation and getting the transaction admitted to the mempool.

---

### Finding Description

`skip_stateful_validations` fires when three conditions are simultaneously true:

1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

Condition 3 is evaluated as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` whenever **any** transaction from the address is in the pool — including a valid `invoke(nonce=0)` submitted by the legitimate account owner. It does not verify that the pooled transaction is specifically a `deploy_account`.

When `skip_stateful_validations` returns `true`, the gateway sets `execution_flags.validate = false` and calls `perform_validations`, which short-circuits before the `__validate__` entry point:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

The signature is never verified. The transaction is then forwarded to the mempool via `add_tx`, which accepts it because `validate_nonce` passes (`1` is within `[0, max_allowed_nonce_gap=200]`) and no duplicate nonce exists yet. [4](#0-3) 

The code comment acknowledges the check is intentionally broad:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is incorrect for the case of a **deployed** account (nonce=0 in state) that has a pending `invoke(nonce=0)` in the pool. That pending invoke passed `__validate__` because the account is deployed — but it does not imply that a subsequent `invoke(nonce=1)` should skip signature verification.

---

### Impact Explanation

An attacker can get a transaction with an **invalid or forged signature** admitted to the mempool for any deployed account that currently has a pending transaction. Once admitted:

- The invalid transaction occupies the nonce-1 slot for the victim account.
- The legitimate owner's `invoke(nonce=1)` is rejected with `MempoolError::DuplicateNonce`. [5](#0-4) 

- The owner must pay a ≥10% fee escalation to replace the attacker's transaction. [6](#0-5) 

The invalid transaction will eventually fail at blockifier execution time (the blockifier always runs `__validate__` during block building), but the mempool has already accepted it, violating the admission invariant.

**Impact category**: High — "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

---

### Likelihood Explanation

- The attacker only needs to monitor the public mempool for any account with a pending `invoke(nonce=0)` and `account_nonce=0` in state.
- The attack requires submitting a single transaction and is cheap.
- The window is open for the entire duration the victim's `nonce=0` transaction remains in the mempool (up to TTL, default 60–300 seconds).
- No privileged access is required.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address. Alternatively, check that the account contract does **not** yet exist in state (i.e., the account is truly undeployed), which is the actual precondition that justifies skipping `__validate__`.

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use something like:
mempool_client.deploy_account_tx_in_pool(tx.sender_address())
// where deploy_account_tx_in_pool checks specifically for a DeployAccount tx
```

---

### Proof of Concept

1. Account `A` is deployed; on-chain nonce = 0.
2. `A`'s owner submits a valid `invoke(nonce=0, valid_sig)` → admitted to mempool. `tx_pool.contains_account(A)` is now `true`.
3. Attacker submits `invoke(nonce=1, sender=A, invalid_sig)` to the gateway.
4. Gateway stateless validation passes (no signature check at this layer). [7](#0-6) 
5. `extract_state_nonce_and_run_validations`: `account_nonce = 0`.
6. `validate_nonce`: `0 ≤ 1 ≤ 200` → passes. [4](#0-3) 
7. `validate_by_mempool`: no duplicate nonce=1 exists → passes.
8. `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
9. `run_validate_entry_point(skip=true)`: `__validate__` is **not called**; invalid signature is never checked.
10. `mempool.add_tx(...)` succeeds. Invalid-signature `invoke(nonce=1)` is now in the mempool.
11. `A`'s owner submits valid `invoke(nonce=1, valid_sig)` → rejected with `DuplicateNonce`.
12. Owner must pay ≥10% higher fee to replace the attacker's transaction via fee escalation.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
