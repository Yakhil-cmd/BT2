### Title
`overrideNFTSellerAmount` Applied Uniformly to All Fungible Token Rows, Enabling Display-Amount Spoofing in Imported NFT Offers — (`packages/gui/src/components/offers/NFTOfferViewer.tsx`, `OfferSummaryRow.tsx`)

---

### Summary

A single scalar `overrideNFTSellerAmount`, derived from only the *first* fungible token found in the offer, is passed unchanged to every `OfferSummaryTokenRow` rendered inside `NFTOfferSummaryRow`. When an imported offer contains more than one fungible asset on the same side (e.g., XCH **and** a CAT), every row overwrites its actual on-chain amount with this single value, causing the viewer to see fabricated amounts for all but the token that happened to be selected by `getNFTPriceWithoutRoyalties`.

---

### Finding Description

**Step 1 — `overrideNFTSellerAmount` is a single scalar derived from one token.**

In `NFTOfferDetails` (`NFTOfferViewer.tsx` lines 367–402):

```typescript
const { amount, assetId, assetType } = getNFTPriceWithoutRoyalties(summary) ?? {};
// ...
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
```

`getNFTPriceWithoutRoyalties` iterates `[OfferAsset.TOKEN, OfferAsset.CHIA]` and returns the **first** match — the first CAT if any CAT exists, otherwise XCH. [1](#0-0) 

**Step 2 — The scalar is broadcast to every token row without per-asset filtering.**

`NFTOfferSummary` passes the same `overrideNFTSellerAmount` to both `makerSummary` and `takerSummary` `NFTOfferSummaryRow` instances. [2](#0-1) 

Inside `NFTOfferSummaryRow`, every asset classified as `OfferAsset.CHIA` or `OfferAsset.TOKEN` receives the same prop:

```tsx
<OfferSummaryTokenRow
  assetId={assetId}
  amount={summaryData[assetId]}          // actual on-chain amount — ignored
  overrideNFTSellerAmount={overrideNFTSellerAmount}  // same value for every row
/>
``` [3](#0-2) 

**Step 3 — `OfferSummaryTokenRow` unconditionally replaces the actual amount.**

```typescript
const amount = overrideNFTSellerAmount ?? originalAmount;
```

Because `??` only guards against `null`/`undefined`, any numeric value (including `0`) fully replaces `originalAmount`. [4](#0-3)

### Impact Explanation

**Concrete attack scenario (TokenForNFT — victim is the NFT seller):**

Attacker crafts an offer file with:
- `summary.offered` = `{ "<cat_id>": 100_000 (100 CAT in mojos), "xch": 1_000 (0.000000001 XCH in mojos) }`
- `summary.requested` = `{ "<victim_nft_launcher_id>": 1 }`

When the victim (NFT owner) imports and views the offer:

1. `determineNFTOfferExchangeType` → `TokenForNFT` (NFT is on the requested side). [5](#0-4) 
2. `getNFTPriceWithoutRoyalties` iterates TOKEN first, finds the CAT, returns `{ amount: 100, assetType: TOKEN }`. [6](#0-5) 
3. `nftSellerNetAmount = amount = 100` (CAT units). `overrideNFTSellerAmount = catToMojo(100) = 100_000` mojos. [7](#0-6) 
4. Both the XCH row and the CAT row receive `overrideNFTSellerAmount = 100_000`.
   - **CAT row**: `catToMojo(100) = 100_000` → displays **100 CAT** ✓ (coincidentally correct)
   - **XCH row**: `formatAmountForWalletType(100_000, STANDARD_WALLET)` = `mojoToChia(100_000)` = **0.0000001 XCH** displayed, but actual on-chain commitment is **0.000000001 XCH** (1_000 mojos)

The attacker can amplify this by choosing a large CAT amount: with `catToMojo(100_000) = 100_000_000_000` mojos as `overrideNFTSellerAmount`, the XCH row displays **100 XCH** while the actual committed XCH is only 1_000 mojos (0.000000001 XCH). The victim accepts the offer believing they receive 100 XCH + 100,000 CAT for their NFT, but the on-chain spend only delivers a negligible XCH amount.

---

### Likelihood Explanation

- The Chia offer protocol supports arbitrary multi-asset combinations on each side; the GUI's own editor restricts to one token, but any offer file can be crafted manually or programmatically.
- The victim only needs to import the crafted offer file (via file picker, QR, or URL share) — a standard, documented workflow.
- No wallet compromise, leaked keys, or social engineering beyond sharing an offer file is required.
- The `checkOfferValidity` RPC call only checks cryptographic validity, not whether the displayed amounts match the on-chain amounts. [8](#0-7) 

---

### Recommendation

In `NFTOfferSummaryRow`, pass `overrideNFTSellerAmount` only to the row whose `assetId` matches the token for which the override was computed (i.e., the `assetId` returned by `getNFTPriceWithoutRoyalties`). All other token rows should render their actual `summaryData[assetId]` amount. The prop should carry both the value and the target asset ID, or the filtering should happen at the `NFTOfferSummaryRow` level before forwarding to `OfferSummaryTokenRow`.

### Proof of Concept

Render `NFTOfferDetails` (or `NFTOfferSummary`) with a crafted `offerSummary`:

```json
{
  "offered": {
    "<cat_asset_id>": 100000,
    "xch": 1000
  },
  "requested": {
    "<nft_launcher_id>": 1
  },
  "infos": {
    "<cat_asset_id>": { "type": "cat" },
    "<nft_launcher_id>": { "type": "singleton", "launcherId": "<nft_launcher_id>" }
  },
  "fees": 0
}
```

Assert that the XCH row's displayed amount equals `mojoToChia(1000)` (0.000000001 XCH). Observe instead that it displays `mojoToChia(catToMojo(100))` = 0.0000001 XCH — a 100× inflation of the actual XCH amount. Swap the CAT amount to `100_000_000` (100,000 CAT) and the XCH row will display **100 XCH** while the on-chain commitment remains 1,000 mojos.

export function determineNFTOfferExchangeType(summary: OfferSummaryRecord): NFTOfferExchangeType | undefined {
  const nftOffered = Object.keys(summary.offered).find(
    (assetId) => offerAssetTypeForAssetId(assetId, summary) === OfferAsset.NFT,
  );
  const nftRequested = Object.keys(summary.requested).find(

}

export function getNFTPriceWithoutRoyalties(
  summary: OfferSummaryRecord,
): GetNFTPriceWithoutRoyaltiesResult | undefined {
  for (const assetType of [OfferAsset.TOKEN, OfferAsset.CHIA]) {
    const assetId = offerAssetIdForAssetType(assetType, summary);
    if (assetId) {
      const amountInMojos = offerAssetAmountForAssetId(assetId, summary);
      if (amountInMojos) {
        const amountInTokens = assetType === OfferAsset.CHIA ? mojoToChia(amountInMojos) : mojoToCAT(amountInMojos);
        return { amount: amountInTokens.toNumber(), assetId, assetType };
      }
    }
  }

  return undefined;
}
