### Title
Pool Admin Front-Runs `setFeeCaps()` to Lock In Admin Fees Above the Incoming Cap, Overcharging Traders - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

When the factory owner lowers `maxAdminSpreadFeeE6` (or `maxAdminNotionalFeeE8`) via `setFeeCaps()`, the function auto-clamps only the global default protocol fee but leaves every pool's stored `poolFeeConfig.adminSpreadFeeE6` untouched. A pool admin who monitors the mempool can front-run the owner's `setFeeCaps()` call with `setPoolAdminFees()` at the old (higher) cap, locking in an admin fee that permanently exceeds the new cap until the owner separately calls `setPoolProtocolFee()` for that specific pool.

### Finding Description

`setFeeCaps()` updates the four cap storage variables and auto-clamps the factory-level default protocol fee, but it performs no retroactive clamping of per-pool admin fees stored in `poolFeeConfig`: [1](#0-0) 

`setPoolAdminFees()` validates the new value against `maxAdminSpreadFeeE6` **at the moment of the call**: [2](#0-1) 

The only place where a pool's stored admin fee is retroactively clamped to the current cap is inside `setPoolProtocolFee()`: [3](#0-2) 

That call is not automatic; it requires the owner to explicitly target each affected pool.

**Attack path:**

1. `maxAdminSpreadFeeE6` is currently 200,000 (20%). Pool admin's fee is 50,000.
2. Factory owner broadcasts `setFeeCaps(200_000, 50_000, ...)` to lower the admin cap to 50,000.
3. Pool admin sees the pending transaction and front-runs it with `setPoolAdminFees(pool, 200_000, 0)` — valid because the cap is still 200,000 at that moment.
4. `setFeeCaps()` executes: `maxAdminSpreadFeeE6` becomes 50,000, but `poolFeeConfig[pool].adminSpreadFeeE6` remains 200,000.
5. The pool's live `spreadFeeE6` (sum of protocol + admin) is immediately pushed to the pool via `setPoolFees()` at the inflated total: [4](#0-3) 

6. Every subsequent swap charges the 200,000 E6 admin component — 4× the new intended cap — until the owner calls `setPoolProtocolFee()` for this pool.

### Impact Explanation

Traders executing swaps against the pool pay a spread fee that exceeds the protocol-enforced cap. The excess fee accrues to the pool admin's fee destination. This is a direct, quantifiable loss of swap input tokens for every trader during the window between the front-run and the owner's remediation call. With `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) and a new cap of, say, 50,000 (5%), the overcharge is 15% of every swap's notional — a material loss of user principal on active pools. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

The pool admin is a semi-trusted role that controls a specific pool. On any EVM chain with a public mempool (Ethereum mainnet, most L2s), a pool admin can trivially observe a pending `setFeeCaps()` transaction and submit `setPoolAdminFees()` with a higher gas price. No special capability beyond mempool visibility is required. The incentive is direct: locking in the maximum fee before the cap drops maximises admin fee revenue.

### Recommendation

`setFeeCaps()` should iterate over all registered pools and clamp each pool's stored admin fee components to the new caps, or alternatively enforce the new caps lazily at the point of fee collection rather than at the point of `setPoolAdminFees()`. A simpler mitigation is to have `setFeeCaps()` call `setPoolProtocolFee()` (or an equivalent internal helper) for every pool in `idxToPool`, clamping admin fees atomically in the same transaction that lowers the cap.
