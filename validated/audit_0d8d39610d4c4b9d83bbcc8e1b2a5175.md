### Title
`PriceVelocityGuardExtension` velocity cap is silently nullified after a long pool pause — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` scales the allowed price-change budget by `(1 + blockDiff)`, where `blockDiff = block.number - lastUpdateBlock`. When a pool is paused for many blocks and then unpaused, `blockDiff` becomes arbitrarily large, making `allowedSq` effectively unbounded. The first swap after a long pause bypasses the velocity cap entirely, exposing LPs to the full oracle price movement in a single block — the exact risk the extension was deployed to prevent.

---

### Finding Description

`PriceVelocityGuardExtension` stores `lastUpdateBlock` at every swap and enforces:

```
allowedSq = maxChange^2 * (1 + blockDiff)
``` [1](#0-0) 

`lastUpdateBlock` is only written during `beforeSwap` or via the admin-only `setLastMidPrice`. When the pool is paused, no swaps execute, so `lastUpdateBlock` freezes at the last pre-pause block. [2](#0-1) 

After unpause, the first `beforeSwap` call computes `blockDiff` as the full pause duration (potentially thousands of blocks). `allowedSq` grows linearly with `blockDiff`, so any oracle price change — no matter how large — satisfies the check. The guard is silently disabled for that swap.

The pool pause/unpause path is: [3](#0-2) 

Neither `pausePool`, `unpausePool`, `protocolPausePool`, nor `protocolUnpausePool` resets `lastUpdateBlock` in the extension, and there is no enforcement requiring the admin to call `setLastMidPrice` before unpausing. [4](#0-3) 

---

### Impact Explanation

LPs who deploy a pool with `PriceVelocityGuardExtension` accept that large oracle price jumps will be spread across multiple blocks, giving them time to remove liquidity. After a long pause the guard is nullified: a trader can execute a swap at the full new oracle price in a single block, causing LP impermanent loss that the extension was specifically designed to bound. This matches the "bad-price execution: unbounded bid/ask quote reaches a pool swap" gate criterion.

---

### Likelihood Explanation

Any legitimate pause event (admin maintenance, protocol emergency) followed by an unpause triggers the condition. No adversarial setup is required — the pool admin performing a routine pause/unpause is sufficient. The longer the pause, the more completely the guard is bypassed.

---

### Recommendation

Mirror the TokenFaucet fix: require (or strongly document) that the pool admin calls `setLastMidPrice` with the current oracle mid-price **before** calling `unpausePool`, so `lastUpdateBlock` is reset to the current block and `blockDiff` starts at zero on the first post-unpause swap.

Alternatively, add a hook in `unpausePool` / `protocolUnpausePool` that calls `setLastMidPrice` automatically, or cap `blockDiff` to a protocol-defined maximum (e.g., 1 block) so the guard cannot be silently widened by inactivity.

---

### Proof of Concept

1. Pool is deployed with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 1e16` (1% per block).
2. At block 1000, last swap executes; `lastUpdateBlock = 1000`, `lastMidPriceX64 = P`.
3. Admin calls `pausePool` at block 1001.
4. 10 000 blocks pass; oracle price moves from `P` to `2P` (100% change).
5. Admin calls `unpausePool` at block 11001.
6. Trader calls `swap`; `beforeSwap` computes:
   - `blockDiff = 11001 - 1000 = 10000`
   - `changeE18 = 1e18` (100%)
   - `actualSq = 1e36`
   - `allowedSq = (1e16)^2 * (1 + 10000) = 1e32 * 10001 ≈ 1e36`
   - `actualSq <= allowedSq` → guard passes, swap executes at `2P`.
7. LPs suffer full 100% price-movement impermanent loss in one block with no protection. [5](#0-4)
