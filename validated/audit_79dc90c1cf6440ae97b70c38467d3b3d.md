Audit Report

## Title
Missing Minimum Deposit Enforcement Allows Dust UTXO State Bloat in ckBTC Minter - (File: rs/bitcoin/ckbtc/minter/src/state.rs)

## Summary
The ckBTC minter initializes `deposit_btc_min_amount` to `0` by default and `validate_config` enforces no lower bound on it. When both `deposit_btc_min_amount` and `check_fee` are zero, the effective deposit floor is 1 satoshi, allowing an unprivileged attacker to spam dust UTXOs that are permanently stored in the minter's heap state with no per-account or global cap, potentially exhausting canister memory and halting all ckBTC operations.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/state.rs` at line 2030, `deposit_btc_min_amount` is set via `deposit_btc_min_amount.unwrap_or_default()`, resolving to `0` when the `InitArgs` field is absent. The `validate_config` method (lines 759–769) only traps if `check_fee > retrieve_btc_min_amount`; it imposes no lower bound on `deposit_btc_min_amount` itself. A helper `effective_deposit_min_btc_amount()` (lines 1833–1836) computes `max(deposit_btc_min_amount, check_fee.saturating_add(1))` but is never called in `update_balance.rs`. Instead, `update_balance.rs` (lines 272–300) reads `s.deposit_btc_min_amount` and `s.check_fee` directly: when both are `0`, the first branch (`utxo.value < 0`) is never true for a `u64`, and the second branch (`utxo.value <= 0`) only rejects zero-value UTXOs, so 1-satoshi UTXOs pass both filters. Every passing UTXO is unconditionally inserted into `available_utxos`, `outpoint_account`, and `utxos_state_addresses` by `add_utxos()` (lines 776–796) with no cap on entries per account or globally. The mainnet upgrade `minter_upgrade_2026_01_23.md` (lines 42–48) confirms the minter ran with `deposit_btc_min_amount = 0` until January 2026, when it was patched via a governance proposal setting the value to 300 sats — a configuration fix, not a code-level fix. The code default and the absence of validation in `validate_config` remain unchanged.

## Impact Explanation
An attacker packs hundreds of dust outputs (≥1 satoshi each under the vulnerable configuration) into Bitcoin transactions targeting publicly derivable ckBTC deposit addresses. Each UTXO that clears the filters is permanently written to the minter's heap. The IC canister heap limit is approximately 4 GiB; with no UTXO count cap, repeated spam fills this space, causing the minter to trap on upgrade (serialization of oversized state) or exhaust available memory at runtime, halting deposits, withdrawals, and consolidations for all ckBTC users. This is a concrete application/platform-level DoS against a production Chain Fusion component, matching the **High ($2,000–$10,000) — Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS** impact class.

## Likelihood Explanation
No privileged access is required. ckBTC deposit addresses are deterministically derivable by any observer. Bitcoin dust outputs cost only the prevailing transaction fee (a few hundred satoshis per transaction carrying many outputs). The attack is trivially scriptable. The mainnet ran in the vulnerable state (`deposit_btc_min_amount = 0`) from launch until January 2026, confirming the configuration is not guaranteed to be set correctly. The current mainnet mitigation (300 sats via upgrade args) is a runtime configuration value, not enforced by code; any future re-initialization or misconfigured deployment reverts to the 0 default. On Regtest and testnet deployments the attack is immediately executable at near-zero cost.

## Recommendation
1. In `validate_config` (`state.rs` lines 759–769), add a trap when `deposit_btc_min_amount < check_fee.saturating_add(1)` and when `deposit_btc_min_amount` is below the P2WPKH dust relay threshold (294 satoshis).
2. Replace the two-field read in `update_balance.rs` (lines 272–273) with a single call to `effective_deposit_min_btc_amount()` to eliminate the divergence and make the intent explicit.
3. Add a per-account or global cap on entries in `available_utxos` / `utxos_state_addresses` inside `add_utxos()` to bound heap growth regardless of configuration.

## Proof of Concept
1. Deploy a ckBTC minter canister via `MinterArg::Init` omitting `deposit_btc_min_amount` (defaults to `0`) and setting `check_fee = 0`.
2. Derive a ckBTC deposit address for any principal.
3. Broadcast a Bitcoin transaction to Regtest containing 500 outputs of 1 satoshi each to that address.
4. Call `update_balance` for the account.
5. Observe via `available_utxos` and `utxos_state_addresses` that all 500 UTXOs are stored permanently; heap usage grows proportionally.
6. Repeat across multiple accounts until the canister approaches its memory limit; confirm that a subsequent upgrade attempt traps due to state serialization size.
