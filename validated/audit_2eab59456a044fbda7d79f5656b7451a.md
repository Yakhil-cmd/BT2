### Title
`setPoolAdminFeeDestination` Does Not Collect Accrued Fees Before Redirecting Admin Fee Destination — (`MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` changes `poolAdminFeeDestination[pool]` without first flushing accumulated spread fees to the old destination. Every other fee-parameter mutation (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` with the **old** destination before updating state. The omission in `setPoolAdminFeeDestination` means all spread-fee surplus that accrued under the old destination is silently redirected to the new one on the next `collectFees` call.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the correct pattern: call `collectFees(…, poolAdminFeeDestination[pool])` with the **current** destination, then update the config. [1](#0-0) 

`setPoolAdminFeeDestination` skips this step entirely: [2](#0-1) 

After the destination is overwritten, the next call to `collectFees` (whether via `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee`) distributes the entire accumulated spread surplus — including amounts earned before the destination change — to the **new** address: [3](#0-2) 

The spread surplus is computed as `balance * scaleMultiplier − binTotals − notionalFeeAccumulator`, which includes all swap-spread fees earned since the last `collectFees` call. None of that is ring-fenced per-destination; it is paid out in one lump to whoever `poolAdminFeeDestination[pool]` points to at collection time.

---

### Impact Explanation

The old admin fee destination (which may be a DAO treasury, revenue-sharing contract, or any address distinct from the pool admin) permanently loses all spread fees that accrued before the destination change. Those tokens are transferred to the new destination instead. There is no recovery path: once `collectFees` runs with the new destination, `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` are zeroed and the surplus is gone.

This is a direct, irreversible loss of owed protocol/admin fees — matching the "direct loss of protocol fees or owed LP assets" impact gate.

---

### Likelihood Explanation

The pool admin is semi-trusted and has the unilateral ability to call `setPoolAdminFeeDestination` at any time. No timelock or cap guards this function. A pool admin who controls a separate fee destination (e.g., a multisig treasury) can front-run or simply time a destination change to capture fees that were owed to the old address. The trigger is a single, permissioned transaction with no additional preconditions.

---

### Recommendation

Mirror the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before redirecting
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is live with `adminFeeDestination = Treasury`.
2. Many swaps occur; spread surplus accumulates (e.g., 10 000 token0 owed to `Treasury`).
3. Pool admin calls `setPoolAdminFeeDestination(pool, Attacker)` — no `collectFees` is triggered.
4. Anyone calls `collectPoolFees(pool)`.
5. `collectFees` reads `poolAdminFeeDestination[pool] == Attacker` and transfers the full 10 000 token0 to `Attacker`.
6. `Treasury` receives nothing; the 10 000 token0 are permanently lost to it. [4](#0-3) [5](#0-4)
