### Title
Receipt Size Invariant Bypassed via `promise_return` `output_data_receivers` Extension After Validation - (`runtime/runtime/src/lib.rs`)

---

### Summary

A contract can craft a receipt whose serialized size exactly equals `max_receipt_size`, pass the `NewReceipt` size gate, and then have the runtime silently extend that receipt's `output_data_receivers` list (via the `promise_return` path) after validation is complete. The resulting receipt exceeds `max_receipt_size` and is forwarded cross-shard without being re-validated, permanently breaking the receipt-size invariant. The bug is acknowledged in production code and tracked as nearcore issue #12606.

---

### Finding Description

**Invariant:** Every newly-created receipt must satisfy `borsh_size(receipt) ≤ max_receipt_size` before it is forwarded to another shard.

**Root cause — two-phase mutation after validation:**

`validate_receipt` enforces the size limit only when called with `ValidateReceiptMode::NewReceipt`: [1](#0-0) 

After that check passes, `apply_action_receipt` in `lib.rs` handles the `promise_return` case. When the parent receipt has non-empty `output_data_receivers` and the child contract returns a `ReceiptIndex`, the runtime **mutates** the already-validated child receipt by appending the parent's receivers to its `output_data_receivers`: [2](#0-1) 

No size re-check is performed after this mutation. The receipt is then forwarded via `receipt_sink.forward_or_buffer_receipt` without re-validation.

When the oversized receipt arrives at the destination shard, `process_incoming_receipts` calls `validate_receipt` with `ValidateReceiptMode::ExistingReceipt`, which **skips** the size check entirely: [3](#0-2) 

The `ExistingReceipt` mode was introduced precisely to tolerate receipts that already escaped the limit due to this bug: [4](#0-3) 

**Attack path (unprivileged user):**

1. Attacker deploys a contract.
2. Contract creates a two-promise DAG: `A -then-> B`.
3. When `A` executes, it creates a new receipt `C` whose `FunctionCall.args` are sized to exactly `max_receipt_size − sizeof(base_receipt_fields)` bytes, so `C` passes the `NewReceipt` size gate.
4. `A` calls `promise_return(C)`, causing the runtime to redirect `B`'s `output_data_receivers` into `C`.
5. `C`'s serialized size now exceeds `max_receipt_size` by the size of the appended `DataReceiver` entries.
6. `C` is forwarded cross-shard; the destination shard processes it under `ExistingReceipt` mode, skipping the size check.

This is demonstrated end-to-end in the production test suite: [5](#0-4) [6](#0-5) 

A second variant exists via `value_return`: returning a value of exactly `max_receipt_size` bytes causes the wrapping `DataReceipt` to exceed the limit: [7](#0-6) 

---

### Impact Explanation

An unprivileged contract can produce receipts that exceed `max_receipt_size` and have them accepted and stored in the trie of any destination shard. This breaks the receipt-size invariant that bounds the per-chunk state-proof size and the borsh-deserialization cost at every node. Nodes must deserialize and process these oversized receipts, and the oversized receipt is stored in the delayed-receipt queue if the shard is congested, persisting the violation across multiple chunks.

---

### Likelihood Explanation

Any account that can deploy a contract and pay for gas can trigger this. The construction requires only two chained promises and a `promise_return` call — standard NEAR contract patterns. No validator or operator privilege is needed. The attack is reachable on mainnet today.

---

### Recommendation

Re-validate the receipt size **after** `output_data_receivers` is extended in `apply_action_receipt`. Specifically, after lines 1029–1035 in `runtime/runtime/src/lib.rs`, compute `borsh::object_length(&new_receipt)` and return an `ActionError::NewReceiptValidationError(ReceiptSizeExceeded{…})` if the post-mutation size exceeds `limit_config.max_receipt_size`. This mirrors the existing guard in `validate_receipt` and closes the window between validation and mutation.

---

### Proof of Concept

```
1. Deploy contract with method max_receipt_size_promise_return_method1(args_size)
   which creates DAG: [A(method2) -then-> B(mark_test_completed)]

2. method2 creates receipt C with:
     FunctionCall { args: vec![0u8; max_receipt_size - base_receipt_size] }
   → borsh_size(C) == max_receipt_size  ← passes NewReceipt gate

3. method2 calls promise_return(C)
   → runtime appends B's DataReceiver to C.output_data_receivers
   → borsh_size(C) > max_receipt_size  ← no re-check

4. C is forwarded cross-shard; destination validates with ExistingReceipt
   → size check skipped; oversized receipt stored and executed
```

Confirmed by `test_max_receipt_size_promise_return` and `test_max_receipt_size_value_return` in `test-loop-tests/src/tests/max_receipt_size.rs`, both of which assert that an oversized receipt does reach the chain (`assert_oversized_receipt_occurred`). [8](#0-7)

/// Assert that there was an incoming receipt with size above max_receipt_size
fn assert_oversized_receipt_occurred(node: &TestLoopNode<'_>) {
    let client = node.client();
    let chain = &client.chain;
    let epoch_manager = &*client.epoch_manager;

    let tip = chain.head().unwrap();
    let epoch_id = epoch_manager.get_epoch_id(&tip.last_block_hash).unwrap();
    let protocol_version = epoch_manager.get_epoch_protocol_version(&epoch_id).unwrap();
    let runtime_config = client.runtime_adapter.get_runtime_config(protocol_version);
    let max_receipt_size = runtime_config.wasm_config.limit_config.max_receipt_size;

    let mut block = chain.get_block(&tip.last_block_hash).unwrap();

    // Go over all blocks down to genesis looking for a receipt above max_receipt_size.
    loop {
        if block.header().is_genesis() {
            panic!("Didn't find receipt with size above max_receipt_size!");
        }
        let prev_block = chain.get_block(block.header().prev_hash()).unwrap();

        let shard_layout = epoch_manager
            .get_shard_layout(&epoch_manager.get_epoch_id(block.hash()).unwrap())
            .unwrap();

        let oversized = if ProtocolFeature::Spice.enabled(protocol_version) {
            // With spice chunks are executed asynchronously and their produced receipts are
            // persisted as receipt proofs keyed by the block in which the chunk was applied,
            // rather than as incoming receipts on the following block.
            shard_layout.shard_ids().any(|shard_id| {
                chain
                    .chain_store()
                    .iter_receipt_proofs_for_shard(block.hash(), shard_id)
                    .iter()
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        } else {
            block.chunks().iter_new().any(|new_chunk| {
                let shard_id = new_chunk.shard_id();
                let prev_shard_index = epoch_manager
                    .get_prev_shard_id_from_prev_hash(block.header().prev_hash(), shard_id)
                    .unwrap()
                    .2;
                let prev_height_included =
                    prev_block.chunks().get(prev_shard_index).unwrap().height_included();
                let incoming_receipts_proofs = get_incoming_receipts_for_shard(
                    &chain.chain_store,
                    epoch_manager,
                    shard_id,
                    &shard_layout,
                    *block.hash(),
                    prev_height_included,
                    ReceiptFilter::TargetShard,
                )
                .unwrap();
                incoming_receipts_proofs
                    .iter()
                    .flat_map(|response| response.1.iter())
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        };

        if oversized {
            return;
        }

        block = prev_block;
    }
}
