### Title
Missing Participant-Set Consistency Check Between Triple Generation and Presigning After Reshare — (File: src/ecdsa/ot_based_ecdsa/presign.rs)

---

### Summary

The OT-based ECDSA `presign` function validates that the threshold of the supplied Beaver triples matches the presigning threshold, but **explicitly omits** any check that the participant set used to generate those triples matches the participant set currently performing presigning. After a reshare that changes the participant set, stale triples (whose embedded Lagrange coefficients are relative to the old participant set) can be silently combined with new key shares (whose Lagrange coefficients are relative to the new participant set), producing cryptographically invalid presignatures that honest parties will accept as legitimate protocol output.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs` lines 38–47, the `presign` entry-point performs the following validation:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
```

The threshold consistency check is present, but the participant-set consistency check is **deliberately absent**, with the rationale that "presumably they need to have been present in order to have shares." This assumption is violated after a reshare.

The reshare protocol (`src/lib.rs` lines 106–141, `src/dkg.rs` lines 600–635) produces a new `KeygenOutput` with fresh signing shares whose Lagrange coefficients are computed relative to the **new** participant set. Beaver triples generated before the reshare, however, encode secret shares whose Lagrange coefficients are relative to the **old** participant set. The two sets of Lagrange coefficients are in general different scalars.

During `do_presign`, the signing share and the triple shares are combined algebraically. When the participant sets differ, the combined value is not the intended masked key-share product; it is a garbage scalar that does not correspond to any consistent nonce or key relationship. The resulting `PresignOutput` is structurally well-formed (it passes all local checks) but is cryptographically invalid: the implicit nonce `R` and the masked key material are inconsistent with the actual signing key.

The library provides no mechanism to bind a triple to the participant set that generated it, and no documentation warning that triples must be discarded and regenerated after every reshare.

---

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs.**

Every honest participant completes `do_presign` without error and stores the resulting `PresignOutput`. When the subsequent signing phase consumes this output, it produces a signature whose `s` component is computed from the corrupted nonce material. The signature fails external verification against the public key. Because the presignature is consumed (never reused by design), the signing session is permanently wasted. If the coordinator repeats this pattern across all pre-generated triples, honest parties are permanently denied the ability to produce valid signatures until new triples are generated — matching the "Permanent denial of signing for honest parties" High-impact criterion.

Additionally, because the corrupted presign output mixes old triple shares with new key shares in a known algebraic structure, a malicious coordinator who controls the old triple secrets may be able to solve for the new signing shares, potentially elevating impact to Critical (key-share extraction). This cannot be confirmed without full access to the `do_presign` internals, so the conservative rating is High.

---

### Likelihood Explanation

The reshare workflow is a documented, first-class feature of the library (`src/lib.rs` lines 104–141). Any integrator who:

1. Runs DKG and generates triples,
2. Performs a reshare (e.g., to add/remove participants or change threshold), and
3. Reuses pre-generated triples in the next presigning session

will silently trigger this bug. The library emits no error, no warning, and no documentation note at the presign call site. The comment in the source code ("presumably they need to have been present") actively misleads integrators into believing the omission is safe. Likelihood is **Medium** for honest integrators and **High** for a malicious coordinator who deliberately triggers a reshare to invalidate honest parties' signing capability.

---

### Recommendation

1. **Bind triples to a participant set at generation time.** Store the sorted participant list (or a hash thereof) inside the triple's public metadata alongside the threshold. In `presign`, compare the stored participant set against the current `participants` argument and return `InitializationError::BadParameters` on mismatch.

2. **Document the invariant explicitly.** Add a `# Panics` / `# Errors` note to `presign` stating that triples must be regenerated after any reshare or refresh that changes the participant set or threshold.

3. **Consider a session-binding token.** Tie triple generation to the DKG session identifier so that triples generated under one key epoch cannot be presented to a presign session operating under a different epoch.

### Proof of Concept

**Setup:**

```
Participants (old): {P1, P2, P3}, threshold = 2
```

1. Run `keygen` → each party holds `KeygenOutput` with signing shares `x_i` relative to `{P1,P2,P3}`.
2. Run triple generation (`deal`) for `{P1,P2,P3}`, threshold 2 → triples `T0, T1` with shares `(a_i, b_i, c_i)` whose Lagrange basis is `{P1,P2,P3}`.
3. Run `reshare` to new set `{P1,P2,P4}`, threshold 2 → each party now holds new `KeygenOutput` with signing shares `x'_i` relative to `{P1,P2,P4}`.
4. **Do not regenerate triples.** Call `presign` with participants `{P1,P2,P4}`, new key shares `x'_i`, and old triples `T0, T1`.

**Expected behavior:** `presign` should reject with a participant-set mismatch error.

**Actual behavior:** `presign` accepts the call (threshold 2 == 2 passes the only check). `do_presign` combines `x'_i` (Lagrange basis `{P1,P2,P4}`) with `a_i, b_i` (Lagrange basis `{P1,P2,P3}`). The resulting `PresignOutput` is structurally valid but cryptographically garbage. The subsequent `sign` call produces a signature `(r, s)` where `s` is computed from the corrupted nonce; the signature fails `secp256k1` verification against the public key.

**Root cause line:** `src/ecdsa/ot_based_ecdsa/presign.rs` lines 38–40 (explicit omission of participant-set check).
