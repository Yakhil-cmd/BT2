### Title
Unbenchmarked `POINT_EVALUATION_NATIVE_COST` Constant Enables Native Resource Under-Charging via EIP-4844 Precompile — (File: `basic_system/src/cost_constants.rs`)

### Summary
The `POINT_EVALUATION_NATIVE_COST` constant used to charge native (proving) resources for the EIP-4844 KZG point evaluation precompile carries an explicit developer TODO acknowledging it has not been properly benchmarked. If the actual RISC-V proving cost exceeds the charged amount, any unprivileged caller can consume more proving resources per block than the system accounts for, potentially causing valid blocks to become unprovable.

### Finding Description

In `basic_system/src/cost_constants.rs`, the native cost for the point evaluation precompile is defined as:

```rust
// TODO(EVM-1178) Add more vectors and benchmark cost better
pub const POINT_EVALUATION_NATIVE_COST: u64 = native_with_delegations!(49_900_000, 3_301_000, 0);
``` [1](#0-0) 

This constant is consumed directly in `basic_system/src/system_functions/point_evaluation.rs` inside `point_evaluation_as_system_function_inner`, which charges both ergs and native resources upfront before performing the actual BLS12-381 KZG proof verification:

```rust
resources.charge(&R::from_ergs_and_native(
    POINT_EVALUATION_COST_ERGS,
    <R::Native as zk_ee::system::Computational>::from_computational(
        POINT_EVALUATION_NATIVE_COST,
    ),
))?;
``` [2](#0-1) 

The native resource is ZKsync OS's model for RISC-V proving cost. As documented in `docs/double_resource_accounting.md`, native resource consumption directly determines how much proving work a transaction is allowed to perform:

> "If a transaction execution runs out of native resources, the entire transaction is reverted." [3](#0-2) 

The only test for this precompile (`basic_test` in `point_evaluation.rs`) verifies the EVM gas cost (50,000 gas) and the output bytes, but **does not verify the native cost against any actual RISC-V cycle measurement**:

```rust
let gas_used = gas_before - resources.ergs().0 / ERGS_PER_GAS;
assert_eq!(gas_used, gas);
assert_eq!(output[..], expected_output);
``` [4](#0-3) 

This is the direct analog to the external report: a hardcoded magic constant used without derivation from actual measurements, with the test suite only checking the EVM-visible output rather than verifying the native resource accounting is correct. The TODO comment is the developer's own acknowledgment that the value is a placeholder.

A secondary instance of the same pattern exists for `L1_TX_NATIVE_PRICE`:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [5](#0-4) 

This constant is used in `prepare_and_check_resources` to compute `native_per_gas = gas_price / L1_TX_NATIVE_PRICE` for every L1→L2 priority transaction: [6](#0-5) 

If `L1_TX_NATIVE_PRICE = 10` is too low, every L1→L2 transaction receives a native budget far larger than what was economically paid for, bounded only by `MAX_NATIVE_COMPUTATIONAL`. Since L1→L2 transactions cannot be invalidated (the code explicitly handles arithmetic overflows with saturation to avoid halting the chain), a miscalibrated value here is a systemic, non-rejectable resource accounting error.

### Impact Explanation

**`POINT_EVALUATION_NATIVE_COST` path:** If the actual RISC-V cycle cost of `verify_kzg_proof` (BLS12-381 pairing + scalar operations) exceeds `POINT_EVALUATION_NATIVE_COST`, then each call to precompile `0x0a` consumes more proving capacity than the system charges. An attacker can pack many such calls into a block, causing the block's actual proving workload to exceed what the native resource budget was designed to permit. The result is a valid-execution block that the prover cannot prove within expected bounds — a forward/proving divergence.

**`L1_TX_NATIVE_PRICE` path:** A miscalibrated value (too low) means L1→L2 transactions receive a native budget proportionally larger than their gas payment justifies. Because these transactions are processed unconditionally (saturation arithmetic is used to avoid chain halts), the miscalibration cannot be corrected at runtime.

### Likelihood Explanation

The point evaluation precompile is reachable by any unprivileged EVM contract via a `CALL` to address `0x0a`. No special role or governance access is required. The likelihood that `POINT_EVALUATION_NATIVE_COST` is wrong is directly evidenced by the developer TODO — the value was set without sufficient benchmark vectors. The `tests/instances/precompiles/gas_model.py` infrastructure exists precisely to measure cycle/gas ratios, but the TODO indicates it has not been applied to this precompile with adequate coverage. [7](#0-6) 

### Recommendation

1. Run the existing benchmarking pipeline (`tests/instances/precompiles/`) against the point evaluation precompile with a broad set of valid KZG proof inputs to measure actual RISC-V cycle counts.
2. Recompute `POINT_EVALUATION_NATIVE_COST` using `native_with_delegations!` with values derived from measured worst-case cycle counts, adding a safety margin.
3. Add a test that asserts `POINT_EVALUATION_NATIVE_COST` is at least as large as the measured cycle count for the worst-case input, analogous to the `verify_intrinsic_native` check used for transaction intrinsic costs.
4. Resolve `TODO (EVM-1157)` for `L1_TX_NATIVE_PRICE` by benchmarking the actual native cost of a representative L1→L2 transaction and deriving the price from the relationship `nativePrice = gasPrice / nativePerGas`. [8](#0-7) 

### Proof of Concept

**For `POINT_EVALUATION_NATIVE_COST`:**

1. Deploy a contract containing a loop that calls precompile `0x0a` with a valid 192-byte KZG proof input (e.g., the test vector from `basic_test`).
2. Submit a transaction with `gas_limit` set high enough to execute N iterations, where `N * POINT_EVALUATION_COST_ERGS ≤ gas_limit * ERGS_PER_GAS`.
3. The system charges `N * POINT_EVALUATION_NATIVE_COST` native units and accepts the transaction.
4. If the actual RISC-V proving cost per call is `C > POINT_EVALUATION_NATIVE_COST`, the prover must handle `N * C` cycles for this transaction alone, exceeding the native budget the system believed was consumed.
5. Fill a block with such transactions; the aggregate actual proving cost exceeds the block's modeled native budget, causing a forward/proving divergence.

**For `L1_TX_NATIVE_PRICE`:**

1. Submit an L1→L2 priority transaction with `gas_price = 10` (the value of `L1_TX_NATIVE_PRICE`).
2. `native_per_gas = ceil(10 / 10) = 1`; `native_prepaid = 1 * gas_limit`.
3. With `gas_limit = 72_000_000`, `native_prepaid = 72_000_000` — potentially reaching `MAX_NATIVE_COMPUTATIONAL` for a payment of only 10 wei/gas.
4. If the correct `L1_TX_NATIVE_PRICE` should be orders of magnitude larger, the transaction receives a native budget far exceeding what was economically justified, consuming prover capacity without proportional payment. [1](#0-0) [5](#0-4)

def compute_ratios(all_results):
    ratio_map = defaultdict(list)

    for run in all_results:
        for marker in run:
            name = marker['name']
            cycles = marker['cycles']
            delegations = marker['delegations']
            ergs = marker['ergs']

            # Weighted delegation sum
            weighted_deleg_sum = 0
            for k, v in delegations.items():
                if k == 1994:
                    weighted_deleg_sum += v * U256BIGINTOPS_RATIO
                elif k == 1991:
                    weighted_deleg_sum += v * BLAKE2ROUNDEXTENDED_RATIO
                else:
                    weighted_deleg_sum += v  # Default weight is 1

            total_cycles = cycles + weighted_deleg_sum
            gas = ergs / ERGS_PER_GAS

            if gas > 0:
                ratio = total_cycles / gas
                ratio_map[name].append(ratio)
