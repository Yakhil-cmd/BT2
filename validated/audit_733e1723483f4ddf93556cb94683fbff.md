### Title
Gateway Skips `__validate__` Entry Point for Invoke Transactions When Account Has Any Pending Transaction, Allowing Unsigned Transaction Admission - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function bypasses the account's `__validate__` entry point for any Invoke transaction with nonce=1 when the sender address has *any* transaction in the mempool or a recent block — not exclusively a `deploy_account`. An attacker who observes a victim's pending nonce=1 invoke (submitted alongside a `deploy_account`) can submit a replacement invoke with arbitrary calldata and no valid signature, which the gateway admits without signature verification.

---

### Finding Description

`skip_stateful_validations` is designed to improve UX for the `deploy_account + invoke` pattern: when a user sends both transactions simultaneously, the invoke (nonce=1) arrives before the deploy_account is on-chain, so the gateway cannot call `__validate__` (the account doesn't exist yet). The function skips validation when three conditions hold: [1](#0-0) 

```
1. tx is Invoke
2. tx.nonce() == Nonce(Felt::ONE)
3. account_nonce == Nonce(Felt::ZERO)
4. account_tx_in_pool_or_recent_block(sender_address) == true
```

The code comment explicitly acknowledges the over-broad check:

> *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."* [2](#0-1) 

The second case — "transactions with future nonces that passed validations" — is the root cause. Once a victim's valid nonce=1 invoke is in the mempool (admitted because the victim's own `deploy_account` was in the pool), `account_tx_in_pool_or_recent_block` returns `true` for the victim's address. Any subsequent nonce=1 invoke for that address — regardless of signature — will have `skip_validate = true`, causing the gateway to set `validate: false` in `ExecutionFlags`: [3](#0-2) 

The resulting `AccountTransaction` is passed to `blockifier_validator.validate(account_tx)` with `validate: false`, meaning the account's `__validate__` entry point is never called during gateway admission. [4](#0-3) 

---

### Impact Explanation

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

An attacker can inject an unsigned (or maliciously-signed) invoke transaction into the mempool for any account that is in the `deploy_account + invoke` UX window. If the mempool permits fee-based replacement (a standard mempool feature), the attacker's higher-fee invalid transaction displaces the victim's valid transaction. The batcher, which always uses `validate: true` via `new_for_sequencing`, will reject the attacker's transaction — but the victim's original transaction has already been evicted, causing transaction censorship. [5](#0-4) 

---

### Likelihood Explanation

The `deploy_account + invoke` pattern is the standard onboarding flow for new Starknet accounts. Any user going through this flow creates a window during which their pending nonce=1 invoke can be replaced. The attacker only needs to monitor the public mempool for pending nonce=1 invokes from accounts with on-chain nonce=0, which requires no special privilege.

---

### Recommendation

Replace the over-broad `account_tx_in_pool_or_recent_block` check with a specific `deploy_account_in_pool_or_recent_block` check that returns `true` only when a `deploy_account` transaction (not any transaction) exists for the sender address. This preserves the intended UX while closing the bypass window.

---

### Proof of Concept

1. Victim submits `deploy_account` (nonce=0) and `invoke` (nonce=1) simultaneously.
2. Gateway admits `deploy_account` to mempool normally.
3. Gateway admits victim's `invoke` (nonce=1): `account_nonce=0`, `tx_nonce=1`, `deploy_account` in pool → `skip_stateful_validations` returns `true` → `validate: false` → admitted.
4. Attacker observes victim's pending nonce=1 invoke in the public mempool.
5. Attacker crafts a replacement invoke: same sender address, nonce=1, arbitrary calldata, no valid signature, higher `max_price_per_unit`.
6. Gateway processes attacker's invoke: `account_nonce=0`, `tx_nonce=1`, victim's invoke is in pool → `account_tx_in_pool_or_recent_block` returns `true` → `skip_stateful_validations` returns `true` → `validate: false` → **admitted without signature check**.
7. Mempool accepts the replacement (higher fee), evicting victim's valid transaction.
8. Batcher executes with `validate: true` (`new_for_sequencing`): `__validate__` fails (invalid signature) → transaction rejected.
9. Victim's `deploy_account` executes successfully, but the invoke is gone from the mempool — the victim's intended action is censored. [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
