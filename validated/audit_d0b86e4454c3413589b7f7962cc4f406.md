### Title
`setPoolAdminFeeDestination` Redirects Uncollected Admin Fees Without Prior Collection, Causing Loss of Earned Fees for the Previous Destination — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination()` changes the admin fee destination immediately without first flushing accumulated fees to the old destination. Every other fee-mutating function in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees()` before applying the change. The omission in `setPoolAdminFeeDestination` means all fees accrued since the last collection are silently redirected to the new address, causing a direct loss for the previous fee destination.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both call `collectFees()` with the current `poolAdminFeeDestination` before updating any state: [1](#0-0) 

This ensures that fees earned under the old rate/destination are settled before the change takes effect.

`setPoolAdminFeeDestination` does not follow this pattern: [2](#0-1) 

It overwrites `poolAdminFeeDestination[pool]` immediately. The next call to `collectPoolFees()` (permissionless, callable by anyone) will then send all previously accumulated admin fees — both spread surplus and notional fees stored in `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` — to the **new** destination: [3](#0-2) 

The pool's `collectFees` distributes to `adminFeeDestination_` (the value passed from the factory's `poolAdminFeeDestination[pool]` at call time): [4](#0-3) 

---

### Impact Explanation

The admin fee destination is a distinct address from the pool admin — it can be a DAO treasury, a revenue-sharing contract, or a third-party LP fee recipient. When the pool admin calls `setPoolAdminFeeDestination`, all fees that accrued since the last `collectPoolFees` call are permanently redirected to the new address. The previous destination receives nothing for the period it was entitled to. This is a direct, quantifiable loss of owed protocol/LP assets.

---

### Likelihood Explanation

The pool admin is semi-trusted and has a legitimate reason to change the fee destination (e.g., rotating a treasury address). The loss occurs even without malicious intent: a routine destination update after a period of high trading volume silently transfers all accumulated fees to the new address. A malicious pool admin can also exploit this deliberately by first accumulating fees and then atomically changing the destination to their own address before triggering collection.

---

### Recommendation

Mirror the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** destination before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accumulated fees to the current destination before rotating.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed; `adminFeeDestination` = `Treasury`.
2. Trading occurs; spread surplus and notional fees accumulate in the pool.
3. Pool admin calls `setPoolAdminFeeDestination(pool, AttackerWallet)`. No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is now `AttackerWallet`.
4. Anyone calls `collectPoolFees(pool)`. The factory passes `AttackerWallet` as `adminFeeDestination_`.
5. All accumulated admin fees — earned during the period `Treasury` was the destination — are transferred to `AttackerWallet`. `Treasury` receives zero. [2](#0-1) [3](#0-2)
