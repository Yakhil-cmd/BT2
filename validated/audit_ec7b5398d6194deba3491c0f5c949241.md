### Title
Staged Engine Upgrade Has No Expiration and Is Executable by Anyone - (File: engine/src/lib.rs)

### Summary
The Aurora Engine's two-step upgrade mechanism (`stage_upgrade` / `deploy_upgrade`) stores WASM code and a minimum deployment block height in contract storage, but imposes no upper-bound expiration. Because `deploy_upgrade` is intentionally callable by any account, a stale or abandoned staged upgrade can be deployed by an unprivileged user at any future block, potentially replacing the live engine with outdated or vulnerable code.

### Finding Description
`stage_upgrade` (owner-only) writes the new WASM binary to `CODE_KEY` and records `delay_block_height = block_height + upgrade_delay_blocks` in `CODE_STAGE_KEY`. [1](#0-0) 

`deploy_upgrade` enforces only a **lower-bound** check (`block_height <= index`). There is no upper-bound / expiration check, no caller restriction, and no cancellation mechanism: [2](#0-1) 

The open-caller design is intentional and documented in `CHANGES.md`: [3](#0-2) 

The `EngineState` struct stores no expiration field for a staged upgrade: [4](#0-3) 

Once `stage_upgrade` is called, the binary persists in storage indefinitely under `CODE_KEY`. The owner has no on-chain mechanism to cancel or expire it; the only mitigation would be to overwrite it with a new `stage_upgrade` call. If the owner stages an upgrade, then discovers a critical flaw in the new binary and stops monitoring, the stale binary remains deployable by any NEAR account forever.

### Impact Explanation
If the owner stages a WASM binary that contains a regression — for example, a version that breaks bridge accounting, removes a security check, or reintroduces a previously patched vulnerability — and then abandons it without overwriting, any unprivileged user can call `deploy_upgrade` at any future block to atomically replace the live engine contract with that binary. Depending on the content of the stale binary, this can result in:

- **Permanent freezing of funds**: if the deployed code breaks `ft_on_transfer`, `withdraw`, or the ETH connector logic, bridged assets become permanently inaccessible.
- **Direct theft of funds**: if the deployed code reintroduces a known exploit (e.g., a broken nonce check, a missing balance guard, or a broken exit-precompile accounting path), an attacker can drain bridged ETH or NEP-141 tokens.

The impact is **Critical** (permanent fund freeze / direct theft) because the engine contract controls all bridged assets and EVM state.

### Likelihood Explanation
The scenario requires: (1) the owner stages an upgrade, (2) the owner decides not to deploy it (e.g., a bug is found post-staging), and (3) the owner does not overwrite the staged binary. This is a realistic operational scenario — staging and then abandoning an upgrade is a normal governance outcome. Once the binary is staged and the delay passes, the window for exploitation is **unbounded in time** and requires only a single permissionless call from any NEAR account. Likelihood is **Medium**.

### Recommendation
Add an expiration block height to the staged upgrade. In `stage_upgrade`, also store `expiry_block_height = block_height + upgrade_delay_blocks + upgrade_expiry_blocks`. In `deploy_upgrade`, add a check:

```rust
if io.block_height() > expiry_index {
    sdk::panic_utf8(errors::ERR_UPGRADE_EXPIRED);
}
```

Additionally, expose a `cancel_upgrade` owner-only function that removes `CODE_KEY` and `CODE_STAGE_KEY` from storage, allowing the owner to explicitly invalidate a staged binary.

### Proof of Concept

1. Owner calls `stage_upgrade` with WASM binary `V_old` (e.g., a version that has a known bridge accounting bug). The engine stores `V_old` under `CODE_KEY` and records `delay_block_height` in `CODE_STAGE_KEY`. [5](#0-4) 

2. Owner discovers `V_old` has a critical flaw and stops the deployment process. Owner does **not** call `stage_upgrade` again to overwrite it.

3. `delay_block_height` passes. The staged binary is now deployable.

4. Attacker (any NEAR account) calls `deploy_upgrade`. The only checks performed are `require_running` and `block_height > index`: [6](#0-5) 

5. `Runtime::self_deploy` atomically replaces the live engine contract with `V_old`. The engine now runs the vulnerable binary. The attacker exploits the known flaw in `V_old` to steal or freeze bridged funds.

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
```

    pub extern "C" fn deploy_upgrade() {
        // This function is intentionally not implemented in `contract_methods`
        // because it only makes sense in the context of the NEAR runtime.
        let mut io = Runtime;
        let state = state::get_state(&io).sdk_unwrap();
        require_running(&state)
            .map_err(ContractError::msg)
            .sdk_unwrap();
        let index = internal_get_upgrade_index();
        if io.block_height() <= index {
            sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
        }
        Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
    }
```
