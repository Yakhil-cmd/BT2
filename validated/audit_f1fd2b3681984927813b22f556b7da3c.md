### Title
Missing In-Protocol Consistency Check Between Rerandomization Context and Signing Inputs Enables Split-View Secret Key Extraction - (`src/ecdsa/robust_ecdsa/sign.rs`, `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The `RerandomizedPresignOutput` struct does not record the `(msg_hash, tweak/ε, participants)` context that was used to compute it. The `sign()` entry-point for both the OT-based and Robust ECDSA schemes accepts a `RerandomizedPresignOutput` together with a fresh `participants` list and `msg_hash`, but never cross-checks that these values match what was embedded in the rerandomization. A malicious coordinator can therefore present different `(h, ε, participants)` tuples to different honest signers during the rerandomization step, causing each signer to hold an inconsistently rerandomized presignature share. Collecting signature shares from two such sessions yields two ECDSA signatures with multiplicatively related nonces, from which the aggregate secret key can be recovered by a standard nonce-reuse attack.

---

### Finding Description

**Vulnerability class:** Transcript/state confusion — stale or inconsistent cryptographic context accepted without deviation check.

**Analog to the external report:** The Usual protocol lacked an on-chain deviation check between the oracle exchange rate and the market price; the off-chain monitoring reaction could always be front-run. Here, the library documents that all participants must agree on `(h, ε, participants)` when rerandomizing, but provides no in-protocol enforcement of that agreement. A malicious coordinator is the exact analogue of the front-running attacker: it can act before any external check can intervene.

**Root cause — `RerandomizedPresignOutput` carries no binding context.**

Both variants of the struct store only the rerandomized cryptographic material:

```rust

If two participants call `rerandomize_presign()` with different `RerandomizationArguments`, they obtain different `delta` values and thus different rerandomized shares — yet the library has no way to detect this.

The library does enforce `N = 2t+1` and `msg_hash ≠ 0` to mitigate some split-view scenarios, but the security documentation explicitly identifies a second, independent requirement that is **not** enforced:

> "2. **Ensure all participants agree on (h, ε) and the signing set.** The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers." [6](#0-5) 

---

### Impact Explanation

The security documentation states:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [7](#0-6) 

Because `RerandomizedPresignOutput` carries no binding context, a malicious coordinator can silently violate the agreement requirement. Two signing sessions that use the same presignature but different `(h, ε)` assignments to different participants produce two signatures with multiplicatively related nonces. The aggregate secret key — and therefore all private signing shares — can be reconstructed from those two signatures.

**Matched allowed impact:** *Critical — Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets.*

---

### Likelihood Explanation

The coordinator role is explicitly part of the threat model (the docstring on `sign()` warns about it). A malicious coordinator needs only to:

1. Obtain a valid `PresignOutput` for each participant (normal protocol execution).
2. Instruct participant A to call `rerandomize_presign()` with `(h1, ε1)` and participant B with `(h2, ε2)` — both calls succeed without error.
3. Collect the resulting inconsistent signature shares across two sessions.
4. Apply a standard ECDSA nonce-reuse computation to recover the secret key.

No cryptographic primitive break, no leaked key, and no external dependency is required. The attack is entirely within the library's documented threat model and requires only the coordinator's ability to supply different `RerandomizationArguments` to different participants, which the library API makes trivially possible.

---

### Recommendation

Bind the rerandomization context to the `RerandomizedPresignOutput` so that `sign()` can enforce consistency:

1. **Store the binding context in `RerandomizedPresignOutput`:** Add fields `msg_hash: [u8; 32]`, `tweak: Scalar`, and `participants: ParticipantList` (or a hash thereof) to both `RerandomizedPresignOutput` structs.

2. **Enforce consistency in `sign()`:** At the start of `sign()`, verify that `presignature.msg_hash == msg_hash`, `presignature.tweak == tweak`, and `presignature.participants == participants`. Return `InitializationError::BadParameters` on mismatch.

3. **Alternative — single-call API:** Merge rerandomization and signing into one function that accepts `PresignOutput` + `RerandomizationArguments` + `participants`, performs the consistency check internally, and never exposes an intermediate `RerandomizedPresignOutput` that could be misused.

### Proof of Concept

```
Setup: N=5 participants {A,B,C,D,E}, t=2 (max_malicious=2), shared PresignOutput from presigning.

Session 1 (coordinator instructs):
  A,B,C,D,E each call rerandomize_presign() with RerandomizationArguments{msg_hash=h1, tweak=ε1, participants={A,B,C,D,E}}
  All call sign(participants={A,B,C,D,E}, msg_hash=h1, presignature=<their rerandomized output>)
  Coordinator collects shares → valid signature sig1=(R1, s1) for h1 under derived key Y1=X+ε1·G

Session 2 (coordinator instructs):
  A,B,C,D,E each call rerandomize_presign() with RerandomizationArguments{msg_hash=h2, tweak=ε2, participants={A,B,C,D,E}}
  All call sign(participants={A,B,C,D,E}, msg_hash=h2, presignature=<their rerandomized output>)
  Coordinator collects shares → valid signature sig2=(R2, s2) for h2 under derived key Y2=X+ε2·G

Both sessions used the same underlying PresignOutput (same nonce k).
R1 = delta1·R_orig, R2 = delta2·R_orig → nonces are multiplicatively related by delta1/delta2.
Standard ECDSA nonce-reuse equations applied to (sig1, sig2) recover the aggregate secret key x.

No error is returned by the library at any step; all N=2t+1 constraints pass.
```

    pub fn derive_randomness(&self) -> Result<Scalar, ProtocolError> {
        // create a string containing (pk, msg_hash, big_r, sorted(participants))
        let pk_encoded_point = self.pk.to_encoded_point(true);
        let encoded_pk: &[u8] = pk_encoded_point.as_bytes();
        let encoded_tweak: &[u8] = &<Secp256K1ScalarField as Field>::serialize(&self.tweak.value());
        let encoded_msg_hash: &[u8] = &self.msg_hash;
        let big_r_encoded_point = self.big_r.to_encoded_point(true);
        let encoded_big_r: &[u8] = big_r_encoded_point.as_bytes();

        // concatenate all the bytes
        let mut concatenation = Vec::new();
        // 1 byte counter, used in the unlikely case that the hash result is 0
        concatenation.extend_from_slice(&[0u8, 1]);
        concatenation.extend_from_slice(encoded_pk);
        concatenation.extend_from_slice(encoded_tweak);
        concatenation.extend_from_slice(encoded_msg_hash);
        concatenation.extend_from_slice(encoded_big_r);
        // Append each ParticipantId's
        for participant in self.participants.participants() {
            concatenation.extend_from_slice(&participant.bytes());
        }

        // initiate hkdf with the salt and with some `good' entropy
        let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);

        let mut delta = Scalar::ZERO;
        // If the randomness created is 0 then we want to generate a new randomness
        while bool::from(delta.is_zero()) {
            // Generate randomization out of HKDF(counter, entropy, pk, msg_hash, big_r, participants, )
            // where entropy is a public but unpredictable random string.
            // The counter depends on the number of times we enter into this loop
            let mut okm = [0u8; 32];

            hk.expand(&concatenation, &mut okm)
                .map_err(|_| ProtocolError::HashingError)?;

            // derive the randomness delta
            delta = Scalar::from_repr(okm.into()).unwrap_or(
                // if delta falls outside the field
                // probability is negligible: in the order of 1/2^224
                Scalar::ZERO,
            );
            // Increment the counter, the probability that this overflows is astronomically low
            let concatenation_0 = concatenation
                .first_mut()
                .ok_or(ProtocolError::InvalidIndex)?;
            *concatenation_0 += 1;
        }
        Ok(delta)
    }

            participants: participants.len(),

    if !participants.contains(me) {

            role: "self",
            participant: me,

            role: "coordinator",
            participant: coordinator,
