### Title
Signature Validation Bypassed for Invoke Transactions via `skip_stateful_validations` Frontrunning — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point call (signature verification) for any invoke transaction with `nonce == 1` sent to an undeployed account that has **any** transaction in the mempool. An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an invoke with `nonce = 1` carrying an **arbitrary/invalid signature** for the victim's address. The gateway accepts it without ever calling `__validate__`, inserting an unauthenticated transaction into the mempool. Because the failed execution rolls back before fee transfer, the attacker pays nothing and can repeat the attack indefinitely.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements a UX shortcut: when an invoke arrives with `tx.nonce == 1` and the on-chain account nonce is `0`, it calls `account_tx_in_pool_or_recent_block(sender_address)` and, if that returns `true`, returns `skip_validate = true`. [1](#0-0) 

`run_validate_entry_point` then sets `execution_flags.validate = !skip_validate`, so when `skip_validate = true` the blockifier's `StatefulValidator::perform_validations` reaches the early-return branch at line 79 without ever calling `__validate__`: [2](#0-1) [3](#0-2) 

The guard `account_tx_in_pool_or_recent_block` checks for **any** transaction in the pool or any recently committed account — it does not filter for `DeployAccount` specifically: [4](#0-3) 

In practice, the only transaction that can reach the pool for an undeployed account (on-chain nonce = 0) is a `deploy_account`, because any invoke/declare with nonce = 0 would require a passing `__validate__` call against a non-existent contract and would be rejected. So the guard does correctly identify the presence of a `deploy_account`. The vulnerability is not in the guard's logic but in the consequence: **once the guard passes, the attacker's invoke with a completely invalid signature is admitted with no cryptographic check whatsoever**.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and fee-escalation rules — it does not verify signatures: [5](#0-4) 

The stateless validator checks signature **length** but not validity: [6](#0-5) 

### Impact Explanation

The gateway admits an invoke transaction whose signature has never been verified. This violates the invariant that every transaction in the mempool has passed account-level authentication. Concretely:

1. The attacker's invalid invoke occupies the `(victim_address, nonce=1)` slot in the mempool, potentially displacing the victim's legitimate invoke (same nonce) via fee-escalation replacement.
2. When the batcher executes the block, `__validate__` is called during actual execution (`execution_flags.validate` is set independently by the batcher). It fails, the transaction is rejected, and the state — including the nonce increment from `perform_pre_validation_stage` — is rolled back: [7](#0-6) 

3. Because execution fails before `handle_fee`, **no fee is charged to the attacker**. The attacker can repeat the attack at zero cost, indefinitely delaying the victim's first post-deploy invoke.

Matching impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- Requires only that the attacker observe a `deploy_account` transaction in the public mempool (trivially visible).
- Requires crafting an invoke with `nonce = 1`, `sender_address = victim`, and any signature bytes that pass the length check (`≤ 4000` felts).
- No privileged access, no special knowledge of the victim's private key, no on-chain funds required.
- The attack is free (failed transactions are not charged), so it can be sustained indefinitely.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a **`DeployAccount`** transaction exists for the address. Alternatively, require that the invoke's signature be verifiable against the class hash embedded in the pending `deploy_account` transaction (pre-validate the signature off-chain using the declared class). At minimum, document that the skip path intentionally admits unauthenticated invokes and add a rate-limit or fee-deposit mechanism to prevent free griefing.

### Proof of Concept

```
1. Victim broadcasts:
     deploy_account { sender_address: V, nonce: 0, class_hash: C, signature: <valid> }
   → Gateway accepts (deploy_account is fully executed including __validate_deploy__).
   → Mempool now contains a tx for address V.

2. Attacker observes V's deploy_account in the mempool and broadcasts:
     invoke { sender_address: V, nonce: 1, calldata: [...], signature: [0xdead, 0xbeef] }
   → Gateway stateful path:
       account_nonce = get_nonce_from_state(V) = 0          // V not yet deployed
       skip_stateful_validations:
         tx.nonce() == 1  ✓
         account_nonce == 0  ✓
         account_tx_in_pool_or_recent_block(V) == true  ✓  // V's deploy_account is pooled
         → returns skip_validate = true
       run_validate_entry_point(skip_validate=true):
         execution_flags.validate = false
         StatefulValidator::perform_validations → early return at line 79
         __validate__ is NEVER called
   → Attacker's invoke with invalid signature is ACCEPTED into the mempool.

3. Victim submits their legitimate invoke { sender_address: V, nonce: 1, signature: <valid> }.
   → Mempool rejects it (duplicate nonce) or requires a higher fee to replace the attacker's tx.

4. Block N is produced:
     - deploy_account(V) executes → V is deployed, on-chain nonce becomes 1.
     - attacker's invoke executes:
         perform_pre_validation_stage: nonce 1 >= 1 ✓, nonce incremented to 2
         validate_tx: __validate__ called → FAILS (invalid signature)
         execute_raw returns Err → TransactionalState rolled back (nonce reverts to 1)
         Transaction rejected; no fee charged to attacker.

5. Attacker repeats from step 2 at zero cost, indefinitely blocking V's nonce-1 invoke.
```
