### Title
FLOW Tokens Locked in COA When EVM Is Paused — (`File: fvm/evm/stdlib/contract.cdc`)

### Summary
The `EVM` contract on Flow exposes a governance-controlled pause mechanism (`isPaused()`). When EVM is paused, the `CadenceOwnedAccount.withdraw()`, `withdrawNFT()`, and `withdrawTokens()` functions all revert due to a `pre { !EVM.isPaused() }` guard. Any FLOW tokens or bridged assets already deposited into a COA's EVM address become inaccessible for the entire duration of the pause, with no alternative recovery path.

### Finding Description
`EVM.isPaused()` reads a boolean from `/storage/evmOperationsPaused` on the EVM contract account: [1](#0-0) 

The Governance Committee can set this flag at any time via a multi-sig Cadence transaction. Once set, every state-mutating EVM operation is blocked. Critically, the fund-recovery functions on `CadenceOwnedAccount` carry the same guard:

**`CadenceOwnedAccount.withdraw()`** — the only way to move FLOW tokens from an EVM address back to Cadence: [2](#0-1) 

**`CadenceOwnedAccount.withdrawNFT()`** — the only way to recover a bridged NFT from EVM: [3](#0-2) 

**`CadenceOwnedAccount.withdrawTokens()`** — the only way to recover bridged fungible tokens from EVM: [4](#0-3) 

The test suite explicitly confirms that `withdraw` reverts with `"EVM operations are temporarily paused"` when the flag is set: [5](#0-4) 

There is no bypass, no emergency-withdrawal path, and no time-limit on the pause. The pause flag is simply a `Bool` stored at `/storage/evmOperationsPaused`; it persists indefinitely until explicitly cleared.

### Impact Explanation
**Impact: High.**
Any user who has deposited FLOW tokens into a COA EVM address (via `EVMAddress.deposit()` or `CadenceOwnedAccount.deposit()`) loses access to those tokens for the full duration of the pause. The same applies to NFTs bridged via `depositNFT()` and fungible tokens bridged via `depositTokens()`. Because the COA's EVM balance is tracked inside the EVM state trie (not in a Cadence vault), there is no alternative Cadence-level mechanism to retrieve the funds. The assets are effectively frozen — not lost permanently, but inaccessible for an unbounded period.

### Likelihood Explanation
**Likelihood: Low.**
Triggering the pause requires the Governance Committee to execute a multi-sig Cadence transaction. This is an intentional, privileged operation. However, the pause is explicitly designed for "maintenance or any situation that requires special governance handling" — meaning it is a realistic, planned operational event, not a hypothetical one. The duration of a pause is unspecified and could extend for days or weeks during a serious incident, making the fund-lock impact non-trivial.

### Recommendation
Remove the `!EVM.isPaused()` pre-condition from the three withdrawal/recovery functions:
- `CadenceOwnedAccount.withdraw()`
- `CadenceOwnedAccount.withdrawNFT()`
- `CadenceOwnedAccount.withdrawTokens()`

These functions move assets *out* of the EVM environment back to Cadence. Blocking them during a pause does not protect EVM state integrity (the EVM trie is not mutated in a way that requires consistency with new EVM transactions); it only harms users who need to recover their funds. Deposit and execution operations (`deposit`, `run`, `batchRun`, `call`, `deploy`) should remain gated by the pause check, as those introduce new state changes into EVM.

### Proof of Concept

1. User calls `EVM.createCadenceOwnedAccount()` and deposits 100 FLOW via `coa.deposit(from: <-vault)`.
2. Governance Committee executes a multi-sig transaction saving `true` to `/storage/evmOperationsPaused` on the EVM contract account.
3. User attempts to recover funds:
   ```cadence
   let bal = EVM.Balance(attoflow: 0)
   bal.setFLOW(flow: 100.0)
   let vault <- coa.withdraw(balance: bal)  // PANICS: "EVM operations are temporarily paused"
   ```
4. The pre-condition at `contract.cdc:589` fires, the transaction aborts, and the 100 FLOW remains locked in the EVM trie with no recovery path available to the user.
5. The lock persists until the Governance Committee explicitly clears the flag — with no protocol-enforced time limit. [6](#0-5) [7](#0-6)

                withdrawnUUID: vault.uuid,
                balanceAfterInAttoFlow: self.balance().attoflow
            )
            return <-vault
        }
