Audit Report

## Title
Stale `API_BOUNDARY_NODE_PRINCIPALS` Heap Cache Allows Removed Boundary Nodes Unauthorized Access and Causes Post-Upgrade Access Denial - (`rs/boundary_node/rate_limits/canister/canister.rs`, `rs/boundary_node/salt_sharing/canister/canister.rs`)

## Summary

Both the rate-limit canister and salt-sharing canister store authorized API boundary node principals in a non-stable `thread_local! RefCell<HashSet<Principal>>` that is reset to empty on every canister upgrade and is only refreshed by a periodic timer. This creates two concrete security windows: (1) a removed/compromised boundary node retains access for up to `registry_polling_period_secs` seconds after registry removal, and (2) after every canister upgrade, all legitimate boundary nodes are locked out for the same interval until the first timer tick repopulates the cache.

## Finding Description

**Root cause — non-stable heap storage:**

In the rate-limit canister, `API_BOUNDARY_NODE_PRINCIPALS` is declared as a plain `thread_local!` `RefCell<HashSet<Principal>>` initialized to `HashSet::new()`: [1](#0-0) 

The same pattern exists in the salt-sharing canister: [2](#0-1) 

Neither is backed by `ic_stable_structures`, so both are wiped to empty on every Wasm upgrade.

**Rate-limit canister — no immediate poll on init/post_upgrade:**

`post_upgrade` calls `init`, which calls `periodically_poll_api_boundary_nodes` using `set_timer_interval`: [3](#0-2) [4](#0-3) 

`set_timer_interval` fires only after the first full interval elapses (e.g., 60 s on mainnet). There is no `set_timer(Duration::ZERO, ...)` immediate poll. The cache therefore remains empty for the entire first interval after every upgrade.

**Salt-sharing canister — same gap:**

`init` schedules `init_async` via `set_timer(Duration::ZERO, ...)`, which runs at the next round. However, `init_async` itself only calls `set_timer_interval(period, ...)` for periodic polling — it does not call `poll_api_boundary_nodes()` immediately: [5](#0-4) 

The first actual poll therefore fires after the full `registry_polling_interval_secs` interval, leaving the cache empty post-upgrade.

**Authorization checks read from the stale/empty cache:**

Rate-limit canister `inspect_message` and `get_config` both call `is_api_boundary_node_principal`: [6](#0-5) 

Salt-sharing canister `inspect_message` and `get_salt` both call `is_api_boundary_node_principal`: [7](#0-6) [8](#0-7) 

`is_api_boundary_node_principal` reads directly from the non-stable cache: [9](#0-8) 

**Instance 1 — stale access after node removal:**
When a boundary node is removed from the registry, the registry is updated immediately, but neither canister's cache is invalidated. The removed node's principal remains in `API_BOUNDARY_NODE_PRINCIPALS` until the next timer tick. During this window the removed node passes `is_api_boundary_node_principal()` and can call `get_config` (reading confidential, not-yet-disclosed rate-limit rules) and `get_salt` (reading the shared privacy salt).

**Instance 2 — post-upgrade lockout:**
After every canister upgrade, `API_BOUNDARY_NODE_PRINCIPALS` is empty. Legitimate boundary nodes calling `get_config` as an update call hit `inspect_message`, which returns `false` and traps with `"message_inspection_failed: method call is prohibited in the current context"`. This persists for the full polling interval.

## Impact Explanation

**Instance 1** maps to: *Medium — Difficult attack requiring node/boundary-node control, but with meaningful security impact.* A removed (potentially compromised) boundary node can read confidential rate-limit rules (incident details kept private until disclosure) and the shared privacy salt (enabling cross-node user deanonymization) during a bounded ~60-second window. The attacker requires control of a previously registered boundary node principal, which constrains the attack but does not eliminate it — node removal via NNS governance is a normal operational event triggered precisely because a node is compromised or decommissioned.

**Instance 2** maps to: *Medium — Application/platform-level DoS with limited duration.* Every canister upgrade (a routine governance operation) causes a ~60-second window during which all legitimate boundary nodes are denied certified rate-limit rule retrieval, disrupting rate-limit enforcement immediately post-upgrade.

## Likelihood Explanation

**Instance 1:** Realistic. API boundary nodes are operated by independent parties. NNS removal proposals for decommissioned or compromised nodes are normal operational events. The 60-second window is sufficient for targeted calls. No subnet majority or threshold key is required — only possession of the removed node's principal/key.

**Instance 2:** Certain. Every canister upgrade resets the heap cache. Upgrades are routine governance operations. The lockout window is deterministic and repeatable.

## Recommendation

1. **Immediate poll on init/post_upgrade (rate-limit canister):** Replace the sole `set_timer_interval` call in `periodically_poll_api_boundary_nodes` with an initial `set_timer(Duration::ZERO, ...)` that fires the poll immediately, followed by the recurring interval — mirroring the salt-sharing canister's `set_timer(Duration::ZERO, async { init_async(...).await })` pattern. Additionally, ensure `init_async` itself calls `poll_api_boundary_nodes()` once before setting up the interval.

2. **Persist the principal set to stable memory:** Store `API_BOUNDARY_NODE_PRINCIPALS` in `ic_stable_structures` (e.g., a `StableBTreeMap` under a dedicated `MemoryId`) so it survives upgrades without requiring a fresh poll, eliminating the post-upgrade lockout entirely.

3. **Immediate cache invalidation on removal:** Expose an authenticated endpoint callable only by the registry canister or NNS governance that pushes removal notifications directly to both canisters, eliminating the polling-window stale-access problem.

## Proof of Concept

**Instance 1 (stale access):**
1. Register boundary node `BN_A`; both canisters poll and add `BN_A` to `API_BOUNDARY_NODE_PRINCIPALS`.
2. Submit NNS proposal calling `remove_api_boundary_nodes({node_ids: [BN_A]})` on the registry. Registry is updated immediately.
3. Within 60 seconds (before next timer tick), `BN_A` calls `get_config` as an update call on the rate-limit canister. `inspect_message` calls `is_api_boundary_node_principal(BN_A)` → `true` (stale cache). Call is accepted; `BN_A` reads confidential rate-limit rules.
4. `BN_A` calls `get_salt` on the salt-sharing canister. `is_api_boundary_node_principal(BN_A)` → `true` (stale cache). `BN_A` reads the shared privacy salt.

**Instance 2 (post-upgrade lockout):**
1. Upgrade the rate-limit canister via NNS proposal. `post_upgrade` → `init` → `periodically_poll_api_boundary_nodes(60, ...)` registers a 60-second interval timer. `API_BOUNDARY_NODE_PRINCIPALS` is now empty.
2. Within the next 60 seconds, legitimate boundary node `BN_B` calls `get_config` as an update call. `inspect_message` calls `is_api_boundary_node_principal(BN_B)` → `false` (empty cache). Call is rejected with `"message_inspection_failed: method call is prohibited in the current context"`.
3. A PocketIC integration test can reproduce this deterministically: install canister, upgrade it, immediately call `get_config` from a known boundary node principal, assert rejection, advance time by `registry_polling_period_secs`, assert success.
