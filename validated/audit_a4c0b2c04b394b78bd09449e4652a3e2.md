### Title
`collectFees` double-counts notional fees, draining LP principal — (`MetricOmmPool.sol`)

### Summary

The `collectFees` function in `MetricOmmPool.sol` uses the total pool surplus — which already includes accumulated notional fees — as the base for the spread-fee split, then distributes the notional fees a second time from the same balance. The double-counted amount is taken from LP principal. The protocol's own test explicitly asserts this over-allocation.

### Finding Description

**Root cause — wrong base for spread-fee split**

Inside `collectFees`, the surplus is computed as:

```
surplus0Scaled = balance0() * TOKEN_0_SCALE_MULTIPLIER − binTotals.scaledToken0
```

This surplus contains **both** the spread-fee accumulation and the notional-fee accumulation (`notionalFeeToken0Scaled`), because notional fees are withheld from traders but never added to bin balances.

The spread-fee split then distributes the **entire** surplus:

```solidity
// MetricOmmPool.sol lines 391-395
uint256 spreadFee0ToAdminScaled    = (surplus0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
```

Because `adminSpreadFeeE6_ + protocolSpreadFeeE6_ == spreadSumE6`, the two lines together pay out exactly `surplus0Scaled` — the full surplus including the notional portion.

Immediately after, the notional fees are distributed **again** from the same pool balance:

```solidity
// MetricOmmPool.sol lines 397-403
uint256 notionalFee0ToAdminScaled    = (notionalFee0AmountScaled * adminNotionalFeeE8_) / notionalSumE8;
uint256 notionalFee0ToProtocolScaled = notionalFee0AmountScaled − notionalFee0ToAdminScaled;
```

**Total payout** = `surplus0Scaled + notionalFee0AmountScaled`
**Available balance** = `surplus0Scaled` (which already contains `notionalFee0AmountScaled`)
**Over-allocation** = `notionalFee0AmountScaled` — taken from LP principal (`binTotals.scaledToken0`)

The protocol's own test documents this invariant break:

```solidity
// metric-core/test/MetricOmmPool.notionalFee.t.sol lines 264-265
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
``` [1](#0-0) [2](#0-1) 

**Trigger path (permissionless)**

`MetricOmmPoolFactory.collectPoolFees(pool)` is callable by anyone and directly invokes `pool.collectFees(...)`:

```solidity
// MetricOmmPoolFactory.sol lines 379-389
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
}
``` [3](#0-2) 

No access control, no guard against the over-allocation.

### Impact Explanation

Every call to `collectPoolFees` (or any internal `collectFees` trigger such as `setPoolAdminFees` / `setPoolProtocolFee`) when both spread and notional fees are non-zero drains LP principal by exactly `notionalFeeToken0Scaled + notionalFeeToken1Scaled`. For a pool with 1 % notional fee and $1 M cumulative volume between collections, LPs lose $10 000 per collection cycle. The pool's bin balances fall short of LP share claims, causing partial insolvency that worsens with each collection. [4](#0-3) 

### Likelihood Explanation

High. The condition requires only that both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` on the same pool — a normal production configuration. The trigger is permissionless and can be called by any address at any time, including bots or adversaries who want to maximise LP drain by calling frequently. [5](#0-4) 

### Recommendation

Subtract the notional accumulator from the surplus before computing the spread-fee split:

```solidity
uint256 spreadSurplus0Scaled = surplus0Scaled - notionalFee0AmountScaled;
uint256 spreadSurplus1Scaled = surplus1Scaled - notionalFee1AmountScaled;

uint256 spreadFee0ToAdminScaled    = spreadSumE6 == 0 ? 0 : (spreadSurplus0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = spreadSumE6 == 0 ? 0 : (spreadSurplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
// ... same for token1
```

This ensures the spread-fee split covers only the spread portion of the surplus, and the notional fees are distributed exactly once.

### Proof of Concept

1. Deploy a pool with `adminSpreadFeeE6 = 100_000` (10 %) and `adminNotionalFeeE8 = 1_000_000` (1 %).
2. Add liquidity; perform several swaps to accumulate both fee types.
3. Read `notionalFeeToken0Scaled` (slot 2) — call it `N`.
4. Read `balance0() * TOKEN_0_SCALE_MULTIPLIER − binTotals.scaledToken0` — call it `S`.
5. Call `factory.collectPoolFees(pool)` (permissionless).
6. Observe that `binTotals.scaledToken0` decreases by `N` — LP principal is drained.
7. The protocol's existing test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive` in `metric-core/test/MetricOmmPool.notionalFee.t.sol` already asserts `total0Attempted > surplus0Scaled`, confirming the over-allocation. [6](#0-5) [7](#0-6)
