### Title
Gateway Signature Validation Bypass via `skip_stateful_validations` Admits Unsigned Invoke Transactions into Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX feature, designed to allow a `deploy_account + invoke` pair to be submitted atomically, uses an overly broad mempool check (`account_tx_in_pool_or_recent_block`) that returns `true` for **any** transaction in the pool for a given address — not only a `deploy_account`. When the condition fires, `run_validate_entry_point` sets `validate: false`, completely skipping the `__validate__` entry-point call at the gateway. An unprivileged attacker who observes that address X has **any** pending transaction in the mempool can submit an invoke with nonce=1 and an arbitrary (invalid) signature for X, and the gateway will admit it without signature verification.

---

### Finding Description

**Step 1 — UX skip path in `skip_stateful_validations`** [1](#0-0) 

The function returns `Ok(true)` (skip validation) when all three conditions hold:
1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == 1`.
3. `account_nonce == 0` (account not yet deployed in committed state).
4. `account_tx_in_pool_or_recent_block(sender)` returns `true`.

**Step 2 — `account_tx_in_pool_or_recent_block` is not deploy-account-specific** [2](#0-1) 

The function returns `true` if the account has **any** transaction in the pool (`tx_pool.contains_account`) or is known from a recent committed block (`state.contains_account`). It does not distinguish between a `deploy_account` and a regular `invoke` with nonce=0.

The inline comment claims:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is incorrect: an account with a valid nonce=0 invoke already in the pool satisfies the check, yet no `deploy_account` is present and the account may already be deployed.

**Step 3 — `validate: false` is set, skipping `__validate__`** [3](#0-2) 

When `skip_validate=true`, `ExecutionFlags { validate: false, … }` is constructed. Inside `StatefulValidator::perform_validations`: [4](#0-3) 

The `__validate__` entry point — which is the account contract's signature-verification function — is never called. The transaction is admitted to the mempool with no cryptographic check on its signature.

**Step 4 — Batcher re-validates with `validate: true`** [5](#0-4) 

`AccountTransaction::new_for_sequencing` always sets `validate: true`. So during block execution the `__validate__` entry point **is** called, the invalid signature is rejected, and the transaction reverts. The invalid transaction does not alter state, but it was admitted to the mempool and consumed batcher resources.

---

### Impact Explanation

An unprivileged attacker can inject invoke transactions with arbitrary (invalid) signatures into the mempool for any account that currently has a pending transaction. The gateway's only cryptographic admission guard — the `__validate__` entry-point call — is bypassed. This matches the **High** impact scope:

> *Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

Concrete consequences:
- **Mempool pollution**: the attacker can flood the mempool with signature-invalid invokes for any active account, displacing legitimate transactions.
- **Batcher resource waste**: each invalid transaction is fetched, converted, and executed (up to the `__validate__` revert) before being discarded.
- **Targeted DoS**: by continuously submitting invalid nonce=1 invokes for a victim account that has a nonce=0 tx in the pool, the attacker can keep the victim's slot occupied with junk.

---

### Likelihood Explanation

The mempool is publicly observable (P2P propagation, RPC). Any attacker can watch for accounts with pending transactions, then immediately submit a nonce=1 invoke with a garbage signature. No privileged access, no special key material, and no race condition is required — the only prerequisite is that the target account has **any** transaction in the pool, which is the normal operating state for active accounts.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the account. For example, expose a `deploy_account_in_pool(address)` query from the mempool that inspects the transaction type, and only skip `__validate__` when that specific condition holds. Alternatively, restrict the skip to cases where the account's class hash is not yet set in state (i.e., the account truly does not exist), which is the only scenario where skipping is semantically safe.

---

### Proof of Concept

```
1. Alice submits a valid Invoke V3 (nonce=0) for address X.
   → mempool admits it; tx_pool.contains_account(X) = true.

2. Attacker submits Invoke V3 for address X:
     nonce      = 1
     signature  = [0xdeadbeef, 0xdeadbeef]   // invalid
     calldata   = [arbitrary]

3. Gateway stateful validation:
   - get_nonce_from_state(X) → 0              (Alice's tx not yet executed)
   - validate_nonce: 0 ≤ 1 ≤ 0+gap → OK
   - skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → true
       account_tx_in_pool_or_recent_block(X) → true  (Alice's nonce=0 tx)
       returns Ok(true)
   - run_validate_entry_point(skip_validate=true):
       ExecutionFlags { validate: false, … }
       __validate__ is NOT called
   → transaction admitted to mempool ✓

4. Batcher fetches attacker's tx, calls AccountTransaction::new_for_sequencing:
     ExecutionFlags { validate: true, … }
   → __validate__ called → INVALID_SIGNATURE → tx reverts.
   → State unchanged, but mempool slot consumed and batcher CPU wasted.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
