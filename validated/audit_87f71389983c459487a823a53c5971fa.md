### Title
Missing Participant-Set Consistency Check in OT-Based ECDSA Presigning Enables Triple Reuse and Private Key Extraction — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in the OT-based ECDSA module explicitly omits the check that the presigning participant set is a subset of the triple generation participant set. The `TriplePub` struct records which participants generated each triple, but `presign()` never validates that the current signing participants are a subset of those recorded participants. A malicious coordinator can exploit this to run two presigning sessions with different valid subsets of the same triple's participants, producing two presignatures that share the same nonce `R`. Signing two different messages with these presignatures enables full private key extraction via standard ECDSA nonce-reuse algebra.

---

### Finding Description

The protocol specification in `docs/ecdsa/ot_based_ecdsa/signing.md` line 18 states that presigning requires:

> Two multiplicative (Beaver) triples private shares **generated by the same set of parties P₀**

And `docs/ecdsa/ot_based_ecdsa/orchestration.md` lines 40–56 establish the invariant:

```
P (keygen) ⊇ P₀ (triples) ⊇ P₁ (presign) ⊇ P₂ (sign)
```

The `TriplePub` struct in `src/ecdsa/ot_based_ecdsa/triples/mod.rs` stores the participants who generated the triple: [1](#0-0) 

However, the `presign()` function in `src/ecdsa/ot_based_ecdsa/presign.rs` contains an explicit comment acknowledging the omission of this check: [2](#0-1) 

The only threshold-related check performed is that the presigning threshold matches the triple's threshold — it does **not** verify that `participants ⊆ args.triple0.1.participants` or `participants ⊆ args.triple1.1.participants`.

The nonce `R` is computed in `do_presign` as: [3](#0-2) 

`R = D · (1/e) = (d·G) / (k·d) = (1/k)·G`

where `k` is the Lagrange-reconstructed secret from the triple. Because any threshold-sized subset of the triple generation participants reconstructs the **same** `k` (they all hold shares of the same polynomial), two presigning sessions using the same triple but different valid participant subsets produce presignatures with **identical** `R`.

---

### Impact Explanation

**Critical — Private key extraction.**

Let triple T be generated by `{A, B, C}` with threshold 2. A malicious coordinator runs:

1. **Presign₁** with participants `{A, B}` using triple T → `PresignOutput` with `big_r = R`, shares `(k_A, σ_A)` and `(k_B, σ_B)`
2. **Presign₂** with participants `{A, C}` using the **same** triple T → `PresignOutput` with `big_r = R` (identical), shares `(k_A, σ_A)` and `(k_C, σ_C)`

Both reconstruct the same `k` and `σ = k·x` via Lagrange interpolation. The signing phase in `src/ecdsa/ot_based_ecdsa/sign.rs` computes: [4](#0-3) 

Aggregated signatures over two different messages `m₁`, `m₂`:
- `s₁ = k·h₁ + r·σ`
- `s₂ = k·h₂ + r·σ`

Private key recovery:
```
k = (s₁ - s₂) / (h₁ - h₂)
σ = (s₁ - k·h₁) / r
x = σ / k
```

This is a complete extraction of the aggregate private key `x`.

---

### Likelihood Explanation

The malicious coordinator is an explicitly documented attacker profile in `RESEARCHER.md` lines 36–42: [5](#0-4) 

The coordinator orchestrates which participants use which triples. Honest participants A, B, C have no mechanism to detect that their triple share is being consumed in two separate presigning sessions — the protocol provides no per-triple usage tracking. The `TriplePub.participants` field is present and accessible inside `presign()` but is never consulted. The attack requires only that the coordinator issue two presigning requests to overlapping participant subsets with the same triple, which is a normal coordinator capability.

---

### Recommendation

In `presign()` (`src/ecdsa/ot_based_ecdsa/presign.rs`), after constructing the `ParticipantList`, add an explicit subset check against both triples' recorded participant sets:

```rust
// Enforce P₁ ⊆ P₀ for both triples (spec invariant)
let triple0_participants = ParticipantList::new(&args.triple0.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants = ParticipantList::new(&args.triple1.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;

for p in participants.iter() {
    if !triple0_participants.contains(p) || !triple1_participants.contains(p) {
        return Err(InitializationError::BadParameters(
            "Presigning participants must be a subset of triple generation participants".to_string(),
        ));
    }
}
```

This enforces the documented invariant `P₁ ⊆ P₀` and prevents a malicious coordinator from reusing the same triple across different participant subsets.

### Proof of Concept

```
Setup:
  Participants: A (index 1), B (index 2), C (index 3), threshold t = 2
  Triple T generated by {A, B, C}: polynomial f_k with f_k(0) = k
    k_A = f_k(1), k_B = f_k(2), k_C = f_k(3)
    similarly for σ: σ_A = f_σ(1), σ_B = f_σ(2), σ_C = f_σ(3)
    where σ = k·x (x = private key)

Attack:
  1. Coordinator calls presign(participants=[A,B], triple=T, ...)
       → Lagrange over {A,B}: λ_A = 2, λ_B = -1
       → k_reconstructed = 2·k_A + (-1)·k_B = k  ✓
       → R = (1/k)·G

  2. Coordinator calls presign(participants=[A,C], triple=T, ...)  ← SAME triple T, no check!
       → Lagrange over {A,C}: λ_A = 3/2, λ_C = -1/2
       → k_reconstructed = (3/2)·k_A + (-1/2)·k_C = k  ✓ (same k)
       → R = (1/k)·G  ← IDENTICAL nonce

  3. Sign m₁ with Presign₁ → (R, s₁) where s₁ = k·h₁ + r·σ
  4. Sign m₂ with Presign₂ → (R, s₂) where s₂ = k·h₂ + r·σ

  5. Extract: k = (s₁-s₂)/(h₁-h₂), x = (s₁ - k·h₁)/(r·k)
```

The root cause is the explicitly omitted check at `src/ecdsa/ot_based_ecdsa/presign.rs` lines 38–40, combined with the `participants` field stored in `TriplePub` (`src/ecdsa/ot_based_ecdsa/triples/mod.rs` line 62) that is never consulted during presigning initialization. [6](#0-5) [1](#0-0)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
