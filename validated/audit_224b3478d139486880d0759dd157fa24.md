### Title
Blacklisted `adminFeeDestination` Bricks Fee Collection and Fee-Config Updates — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.collectFees()` atomically transfers accrued fees to both `adminFeeDestination_` and `FACTORY` in a single transaction. If the pool trades USDC or USDT and the `adminFeeDestination` address is later blacklisted by the token issuer, every call path that invokes `collectFees` reverts permanently, freezing all protocol fee revenue inside the pool and blocking fee-configuration updates for both the pool admin and the factory owner.

### Finding Description

`collectFees` in `MetricOmmPool.sol` performs four sequential `safeTransfer` calls inside one atomic block:

```
transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // line 417
transferToken1(adminFeeDestination_, totalFee1ToAdmin);   // line 420
transferToken0(FACTORY, totalFee0ToProtocol);             // line 423
transferToken1(FACTORY, totalFee1ToProtocol);             // line 426
notionalFeeToken0Scaled = 0;                              // line 429
notionalFeeToken1Scaled = 0;                              // line 430
``` [1](#0-0) 

`transferToken0` and `transferToken1` are thin wrappers around `IERC20.safeTransfer`: [2](#0-1) 

USDC and USDT implement a blacklist: `transfer` to a blacklisted recipient reverts. If `adminFeeDestination_` is blacklisted, line 417 (or 420) reverts, rolling back the entire transaction. The protocol-leg transfers (lines 423–426) never execute, and `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` are never reset to zero.

Three distinct callers all route through `collectFees` with the stored `poolAdminFeeDestination[pool]`:

1. **`collectPoolFees(pool)`** — permissionless; anyone can call it. [3](#0-2) 

2. **`setPoolAdminFees(pool, ...)`** — pool admin; calls `collectFees` *before* writing the new fee config. [4](#0-3) 

3. **`setPoolProtocolFee(pool, ...)`** — factory owner; also calls `collectFees` *before* writing the new protocol fee config. [5](#0-4) 

All three revert while `adminFeeDestination` is blacklisted.

The `adminFeeDestination` is set at pool creation and can be updated by the pool admin via `setPoolAdminFeeDestination`, which does **not** call `collectFees` first and therefore remains callable: [6](#0-5) 

This means a recovery path exists — but only if the pool admin is responsive. If the pool admin is a multisig that has been compromised, is unresponsive, or is itself the blacklisted address, the DoS becomes permanent.

### Impact Explanation

- **Protocol fee revenue frozen**: All spread and notional fees that have accrued in the pool (tracked in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` and the surplus balance) cannot be extracted by the protocol or the admin for as long as the destination is blacklisted.
- **Fee-configuration updates blocked**: Neither the pool admin (`setPoolAdminFees`) nor the factory owner (`setPoolProtocolFee`) can change fee parameters for the affected pool, because both functions flush fees first.
- **Permanent if admin is unresponsive**: If the pool admin cannot or does not call `setPoolAdminFeeDestination` to rotate to a clean address, the freeze is permanent and all accumulated protocol fees are stranded inside the pool contract.

Severity: **Medium** — direct loss of protocol fee revenue above dust thresholds; core fee-management functionality broken. Funds are not stolen but are inaccessible to the protocol.

### Likelihood Explanation

- The protocol explicitly targets USDC and USDT pools on Ethereum, Base, and HyperEVM.
- USDC/USDT blacklisting of addresses associated with sanctions or exploits is a documented, recurring real-world event.
- `adminFeeDestination` is a user-supplied address (set at pool creation or updated by the pool admin) and is not restricted to a contract-controlled address.
- The trigger requires no privileged attacker action — the blacklisting is performed by the token issuer, not by any protocol participant.

### Recommendation

Separate the admin-leg and protocol-leg transfers so that a failure on one does not block the other, mirroring the fix applied in the Axis Finance case. Two concrete options:

1. **Split into two independent functions**: expose `collectProtocolFees` (transfers only to `FACTORY`) and `collectAdminFees` (transfers only to `adminFeeDestination_`), so a blacklisted admin destination never blocks protocol fee extraction.

2. **Use try/catch or a pull model for the admin leg**: record the admin's owed amount in storage and let the admin pull it separately, while the protocol leg always executes unconditionally.

Either approach ensures that a blacklisted `adminFeeDestination` cannot freeze protocol fee collection or block fee-configuration updates.

### Proof of Concept

1. Deploy a USDC/USDT pool with `adminFeeDestination = 0xAlice`.
2. Trades occur; `notionalFeeToken0Scaled` and surplus accumulate.
3. USDC issuer blacklists `0xAlice` (e.g., OFAC sanction).
4. Anyone calls `collectPoolFees(pool)` → `collectFees` is called → `safeTransfer(USDC, 0xAlice, amount)` reverts → entire tx reverts.
5. Factory owner calls `setPoolProtocolFee(pool, newFee, newFee)` → same revert path → protocol cannot update fees.
6. Pool admin calls `setPoolAdminFees(pool, newFee, newFee)` → same revert path → admin cannot update fees.
7. All protocol fees remain stranded in the pool contract indefinitely until the pool admin calls `setPoolAdminFeeDestination` to rotate to a non-blacklisted address.
