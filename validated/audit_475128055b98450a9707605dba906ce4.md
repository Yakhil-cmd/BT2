### Title
Random Beacon Private Key Retained in Memory Indefinitely Across Epoch Transitions - (File: `consensus/hotstuff/signature/randombeacon_signer_store.go`)

### Summary
`EpochAwareRandomBeaconKeyStore` caches random beacon BLS private keys in a plain Go map keyed by epoch counter. The map grows without bound and is never pruned. Once a private key is loaded for an epoch, it remains in the process heap for the entire lifetime of the consensus node, including epochs that have long since ended and whose keys are no longer needed for signing.

### Finding Description
`EpochAwareRandomBeaconKeyStore` holds a `privateKeys map[uint64]crypto.PrivateKey` field. [1](#0-0) 

On the first `ByView` call for a given epoch, the private key is fetched from the secrets database and inserted into the map: [2](#0-1) 

There is no eviction, pruning, or zeroing of entries. The map accumulates one entry per epoch for the entire uptime of the node. Contrast this with `consensus/hotstuff/committees/Consensus`, which explicitly prunes its epoch cache to the most recent three epochs: [3](#0-2) 

No equivalent pruning exists in `EpochAwareRandomBeaconKeyStore`. The struct is instantiated once at node startup and lives for the full process lifetime: [4](#0-3) 

### Impact Explanation
A consensus node's random beacon BLS private key is the most sensitive cryptographic material it holds. It is the key share used to produce threshold signatures that drive HotStuff finality. If the process crashes or is killed while the map holds keys from multiple past epochs, any crash-reporting or core-dump mechanism (e.g., `pprof` heap dumps, OS core files, container runtime snapshots) will capture the plaintext private key material for every epoch the node has participated in since startup. An attacker who obtains such a dump gains the private key shares for all those epochs, which can be used to forge threshold signatures retroactively or, if enough shares are collected from multiple nodes, to break the random beacon for those epochs.

### Likelihood Explanation
Consensus nodes are long-running processes. Over many epoch transitions (each ~week on mainnet), the map accumulates keys. Crash dumps, heap profiling endpoints, and memory-forensics tools are realistic vectors on infrastructure that has been compromised at the OS/container level. The root cause is entirely within Flow's own production code and requires no attacker control over consensus logic — only access to a post-crash memory artifact.

### Recommendation
Apply the same epoch-pruning discipline used in `Consensus.pruneEpochInfo()` to `EpochAwareRandomBeaconKeyStore`. After caching a key for a new epoch, delete map entries for epochs older than the current epoch minus a small retention window (e.g., 2). Before deleting, zero out the key bytes if the `crypto.PrivateKey` interface exposes a `Zeroize`/`Destroy` method, or replace the map value with `nil` so the GC can collect the underlying key object. The store should also implement an `EpochCommittedPhaseStarted`-style notification hook so it can proactively evict stale keys when epoch transitions are observed, mirroring the pattern in `Consensus`.

### Proof of Concept
1. A consensus node starts at epoch N and participates through epochs N, N+1, N+2, …, N+k.
2. On the first `ByView` call in each epoch, `ByView` calls `s.keys.RetrieveMyBeaconPrivateKey(epoch)` and stores the result: `s.privateKeys[epoch] = privKey`.
3. After k epochs, `s.privateKeys` contains k live `crypto.PrivateKey` objects, one per epoch, none ever removed.
4. The node crashes (OOM, panic, SIGABRT). A core dump or heap snapshot is captured.
5. Scanning the dump for BLS12-381 private key material (32-byte scalars in the expected range) recovers all k private keys.
6. With a threshold number of key shares from different nodes for the same epoch, an attacker can reconstruct the epoch's random beacon private key and forge threshold signatures for that epoch. [5](#0-4)

func (s *EpochAwareRandomBeaconKeyStore) ByView(view uint64) (crypto.PrivateKey, error) {
	epoch, err := s.epochLookup.EpochForView(view)
	if err != nil {
		return nil, fmt.Errorf("could not get epoch by view %v: %w", view, err)
	}

	// When DKG has completed,
	//   - if a node successfully generated the Random Beacon key, the valid private key will be stored in database.
	//   - if a node failed to generate the Random Beacon key, we will save a record in database to indicate this
	//     node has no private key for this epoch.
	// Within the epoch, we can look up my random beacon private key for the epoch. There are 3 cases:
	//   1. DKG has completed, and the private key is stored in database, and we can retrieve it (happy path)
	//   2. DKG has completed, but we failed to generate a private key (unhappy path)
	//   3. DKG has not completed locally yet
	key, found := s.privateKeys[epoch]
	if found {
		// a nil key means that we don't (and will never) have a beacon key for this epoch
		if key == nil {
			// case 2:
			return nil, fmt.Errorf("beacon key for epoch %d (queried view: %d) never available: %w", epoch, view, module.ErrNoBeaconKeyForEpoch)
		}
		// case 1:
		return key, nil
	}

	privKey, safe, err := s.keys.RetrieveMyBeaconPrivateKey(epoch)
	if err != nil {
		if errors.Is(err, storage.ErrNotFound) {
			// case 3:
			return nil, fmt.Errorf("beacon key for epoch %d (queried view: %d) not available yet: %w", epoch, view, module.ErrNoBeaconKeyForEpoch)
		}
		return nil, fmt.Errorf("[unexpected] could not retrieve beacon key for epoch %d (queried view: %d): %w", epoch, view, err)
	}

	// If DKG ended without a safe end state, there will never be a valid beacon key for this epoch.
	// Since this fact never changes, we cache a nil signer for this epoch, so that when this function
	// is called again for the same epoch, we don't need to query the database.
	if !safe {
		// case 2:
		s.privateKeys[epoch] = nil
		return nil, fmt.Errorf("DKG for epoch %d ended without safe beacon key (queried view: %d): %w", epoch, view, module.ErrNoBeaconKeyForEpoch)
	}

	// case 1: DKG succeeded and a beacon key is available -> cache the key for future queries
	s.privateKeys[epoch] = privKey

	return privKey, nil
}

// pruneEpochInfo removes any epochs older than the most recent 3.
// CAUTION: Not safe for concurrent use - the caller must first acquire the lock.
func (c *Consensus) pruneEpochInfo() {
	// find the maximum counter, including the epoch we just computed
	maxCounter := uint64(0)
	for counter := range c.epochs {
		if counter > maxCounter {
			maxCounter = counter
		}
	}

	// remove any epochs which aren't within the most recent 3
	for counter := range c.epochs {
		if counter+3 <= maxCounter {
			delete(c.epochs, counter)
		}
	}
