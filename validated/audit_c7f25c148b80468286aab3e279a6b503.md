### Title
Missing Minimum Threshold Enforcement in FROST Presign and Sign Inputs Allows Threshold Invariant Bypass — (`src/frost/mod.rs`)

### Summary

`assert_key_invariants` enforces a minimum threshold of 2 at DKG time, but the analogous signing-time validators `assert_sign_inputs` and `presign` omit this lower-bound check entirely. A malicious or buggy caller can invoke the FROST presign/sign path with `threshold = 1`, bypassing the invariant established during key generation and causing the signing protocol to produce a cryptographically invalid signature, denying honest parties a usable output.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces three threshold invariants: [1](#0-0) 

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

Both the upper-bound check (`threshold > n`) **and** the lower-bound check (`threshold < 2`) are present.

The signing-time validator `assert_sign_inputs` in `src/frost/mod.rs` only replicates the upper-bound check: [2](#0-1) 

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

The `threshold < 2` guard is entirely absent. The same omission exists in `presign`: [3](#0-2) 

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

Neither function rejects `threshold = 1` (or `threshold = 0`), even though the DKG that produced the key shares was required to use `threshold >= 2`.

### Impact Explanation

In FROST, signing shares are generated from a degree-`(t-1)` polynomial where `t` is the DKG threshold. Reconstructing the aggregate secret at evaluation point 0 requires exactly `t` shares with correct Lagrange coefficients. If the signing protocol is invoked with `threshold = 1`, only one participant's share is used, and the Lagrange coefficient collapses to 1 (a single-point interpolation). The resulting partial signature aggregates to a value that does not equal the true secret key scalar, producing a cryptographically invalid signature. Every honest participant has completed their round of the protocol, but the output is unusable — a denial of the signing operation under valid protocol inputs.

This matches: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation

The `threshold` parameter is caller-supplied at every invocation of `presign` and `assert_sign_inputs`. Any application layer that passes the wrong threshold value — whether through a bug, a misconfiguration, or a malicious coordinator — triggers the condition. Because the library exports `assert_sign_inputs` as a public validation helper, callers are expected to rely on it as the authoritative gate; the missing check is not compensated elsewhere in the visible protocol flow.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces, in both `assert_sign_inputs` and `presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the `participants.len() < 2` check in both functions, mirroring the pattern in `assert_key_invariants`.

### Proof of Concept

1. Run DKG with `n = 3` participants and `threshold = 2` (accepted by `assert_key_invariants`).
2. Call `presign` with the same 3 participants but `args.threshold = ReconstructionLowerBound(1)`.
   - `participants.len() < 2` → false (3 ≥ 2), passes.
   - `args.threshold.value() > participants.len()` → false (1 ≤ 3), passes.
   - No `threshold < 2` check exists; the protocol proceeds.
3. Call the signing function with `threshold = 1`; `assert_sign_inputs` performs the same incomplete validation and returns `Ok`.
4. The signing protocol runs with a single participant's Lagrange coefficient equal to 1, producing a partial-signature aggregate that does not reconstruct the correct secret. The final signature fails external verification, and honest parties receive no valid output despite completing all protocol rounds. [4](#0-3) [5](#0-4) [6](#0-5)

pub fn assert_key_invariants(
    participants: &[Participant],

    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {

        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
