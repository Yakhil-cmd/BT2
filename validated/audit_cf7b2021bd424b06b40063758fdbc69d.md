### Title
Duplicate Account Key Indexes Allow Double-Weight Accumulation to Bypass Transaction Authorization Threshold — (`fvm/transactionVerifier.go`, `fvm/environment/accounts.go`)

### Summary

An attacker who controls their own account can register the same cryptographic key twice (at two different key indexes, each with weight 500), then sign a transaction with both key indexes using identical signature bytes. The transaction verifier aggregates weights by key index without checking whether two indexes resolve to the same underlying stored key, so `payloadWeights[addr]` reaches 1000 and authorization succeeds — despite the attacker possessing only one key whose individual weight (500) is below the threshold.

---

### Finding Description

**Key addition path** (`fvm/environment/accounts_status.go`, `AppendAccountPublicKeyMetadata`):

When key K is added at index 0, it is stored in the `apk_0` register and the key count is incremented to 1. No metadata entry is written for index 0. [1](#0-0) 

When the same key K is added again at index 1, `FindDuplicateKey` correctly detects the duplicate (digest match + byte-level equality), and `AppendDuplicateKeyMetadata` is called. This writes weight=500 and revoked=false into the `weightAndRevokedStatusBytes` section for index 1, and records a mapping `keyIndex 1 → storedKeyIndex 0`. The key count becomes 2, the deduplication flag is set, but **the key bytes are not stored again** (`saveKey=false`). [2](#0-1) [3](#0-2) 

**Transaction verification path** (`fvm/transactionVerifier.go`):

`newSignatureEntries` deduplicates signatures by `(address, keyIndex)` pair. Since keyIndex=0 and keyIndex=1 are distinct, both signatures pass through. [4](#0-3) 

`getAccountKeysAndAggregateWeights` then calls `GetRuntimeAccountPublicKey` for each signature entry independently and adds the returned weight to `aggregateWeights[addr]`: [5](#0-4) 

For keyIndex=0, `GetRuntimeAccountPublicKey` reads the `apk_0` register directly and returns weight=500. [6](#0-5) 

For keyIndex=1, `GetRuntimeAccountPublicKey` reads the metadata, resolves `storedKeyIndex=0` via the deduplication mapping, reads the same `apk_0` register, and returns weight=500 again. [7](#0-6) 

The result: `aggregateWeights[addr] = 500 + 500 = 1000`. The weight check passes. [8](#0-7) 

**Signature verification** (`fvm/transactionVerifier.go`, `verify`):

`VerifySignatureFromTransaction` is called with the same public key K for both entries. Since the transaction payload message does not include the key index, the same signature bytes `sig_K` are valid for both entries. [9](#0-8) 

All checks pass. The transaction is authorized.

---

### Impact Explanation

An attacker with a single key of weight 500 (below the 1000 threshold) can authorize any transaction on their own account that requires weight ≥ 1000, including draining funds, deploying/updating contracts, or revoking other keys. The deduplication metadata stored during key addition is never consulted during transaction verification, so the invariant "the same cryptographic key material must not contribute its weight more than once" is violated.

---

### Likelihood Explanation

The attack requires only standard account key management operations (adding a key twice) and standard transaction submission. No privileged access, staked node control, or external dependency is needed. It is fully reproducible on the Flow emulator without any code modification.

---

### Recommendation

In `getAccountKeysAndAggregateWeights`, after resolving each signature's key via `GetRuntimeAccountPublicKey`, track the `storedKeyIndex` (already returned by `AccountPublicKeyMetadata`) and reject or skip any signature whose `storedKeyIndex` has already contributed weight for that address. Alternatively, perform deduplication by `(address, storedKeyIndex)` rather than `(address, keyIndex)` in `newSignatureEntries` or in the weight aggregation loop.

---

### Proof of Concept

```
1. Create account A.
2. Add key K (weight 500, ECDSA P-256 / SHA3-256) → keyIndex 0.
3. Add key K again (weight 500) → keyIndex 1.
   - FindDuplicateKey detects duplicate; AppendDuplicateKeyMetadata stores
     weight=500 for index 1, mapping index 1 → storedKeyIndex 0.
4. Build transaction: payer=A, proposer=(A, keyIndex=1, seqNum=0), authorizer=A.
5. Sign payload with K at keyIndex=0 → sig_K.
6. Sign payload with K at keyIndex=1 → sig_K (identical bytes, same key/message).
7. Submit transaction.
8. newSignatureEntries: (A,0) and (A,1) are distinct → both pass.
9. getAccountKeysAndAggregateWeights:
     GetRuntimeAccountPublicKey(A, 0) → weight=500
     GetRuntimeAccountPublicKey(A, 1) → storedKeyIndex=0, weight=500
     payloadWeights[A] = 1000
10. hasSufficientKeyWeight: 1000 >= 1000 → passes.
11. verifySignatures: both verify against K → valid.
12. Transaction executes. Authorization threshold bypassed.
```

// AppendDuplicateKeyMetadata appends duplicate key metadata.
func (m *KeyMetadataAppender) AppendDuplicateKeyMetadata(
	keyIndex uint32,
	duplicateStoredKeyIndex uint32,
	revoked bool,
	weight uint16,
) (err error) {

	// Append revoked status and weight
	m.weightAndRevokedStatusBytes, err = appendWeightAndRevokedStatus(m.weightAndRevokedStatusBytes, revoked, weight)
	if err != nil {
		return err
	}

	if !m.deduplicated {
		// Set deduplication flag
		m.deduplicated = true

		// Save mapping start key index.
		m.startIndexForMapping = keyIndex
	}

	// Append duplicate stored key index to mapping
	m.mappingBytes, err = appendStoredKeyIndexToMappings(m.mappingBytes, duplicateStoredKeyIndex)
	if err != nil {
		return err
	}

	return nil
}
