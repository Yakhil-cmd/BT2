### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Over-Minting and Bridge Insolvency - (`engine/src/engine.rs`)

### Summary

`receive_erc20_tokens` unconditionally trusts the `amount` field supplied by the calling NEP-141 token contract in `ft_on_transfer`. If the NEP-141 token deducts a transfer fee before crediting Aurora Engine, the engine mints more ERC-20 mirror tokens than the NEP-141 tokens it actually holds, creating a permanent insolvency in the bridge.

### Finding Description

When a NEP-141 token is bridged into Aurora, the flow is:

1. User calls `ft_transfer_call(receiver_id: "aurora", amount: N, msg: "<evm_address>")` on the NEP-141 contract.
2. The NEP-141 contract transfers tokens to Aurora Engine (crediting it with `N - fee` tokens if a fee applies), then calls `ft_on_transfer(sender_id, amount: N, msg)` on Aurora Engine.
3. Aurora Engine's `ft_on_transfer` dispatches to `receive_erc20_tokens`.

Inside `receive_erc20_tokens`:

```rust
// engine/src/engine.rs:803
let amount = args.amount.as_u128();
```

This takes the nominal `amount` from the callback argument — the value the NEP-141 contract reported, **not** the actual balance change in Aurora Engine's NEP-141 account. The engine then mints exactly `amount` ERC-20 tokens:

```rust
// engine/src/engine.rs:831
setup_receive_erc20_tokens_input(&recipient, amount),
```

If the NEP-141 token charges a 1% fee, Aurora Engine receives `N * 0.99` NEP-141 tokens but mints `N` ERC-20 tokens. Every deposit inflates the ERC-20 supply beyond the backing NEP-141 reserve. The same pattern exists in `receive_base_tokens` for the ETH base token path:

```rust
// engine/src/engine.rs:778
let amount = Wei::new_u128(args.amount.as_u128());
// ...
set_balance(&mut self.io, &receipient, &new_balance);
```

The `ft_on_transfer` entry point in `engine/src/contract_methods/connector.rs` performs no balance-before/after check and passes `args` directly to both handlers.

### Impact Explanation

**Critical — Insolvency / Permanent Freezing of Funds.**

Each deposit with a fee-bearing NEP-141 token inflates the ERC-20 mirror supply relative to the actual NEP-141 reserve held by Aurora Engine. When users later exit (burn ERC-20 → receive NEP-141 via `ExitToNear`), the last users to exit will find Aurora Engine holds insufficient NEP-141 tokens to honor the redemption. Their funds are permanently frozen inside the bridge with no recovery path.

### Likelihood Explanation

**Medium.** The NEP-141 standard does not prohibit fee-on-transfer tokens. Any token registered via `deploy_erc20_token` that implements a transfer fee (e.g., a deflationary token, a tax token, or a token that later upgrades to add fees) triggers this path. The `ft_on_transfer` entrypoint is publicly reachable by any NEP-141 token contract that has been registered as a mirrored ERC-20. No privileged access is required beyond the token being registered.

### Recommendation

Apply the balance-before/after pattern. In `receive_erc20_tokens` (and `receive_base_tokens`), query Aurora Engine's NEP-141 balance before and after the transfer is credited by the NEAR runtime, and use the delta as the actual minted amount. Since `ft_on_transfer` is a callback (the transfer has already occurred by the time it is called), the correct approach is to compare the engine's NEP-141 balance at the start of `ft_on_transfer` against the balance after the call, or to use a cross-contract call to `ft_balance_of` on the NEP-141 contract before and after. Alternatively, return `args.amount - actual_received` as the refund amount so the NEP-141 contract recoups the fee portion.

### Proof of Concept

1. A fee-on-transfer NEP-141 token `fee_token.near` (1% fee) is registered on Aurora via `deploy_erc20_token`.
2. Alice calls `ft_transfer_call(receiver_id: "aurora", amount: 1000, msg: "<alice_evm_addr>")` on `fee_token.near`.
3. `fee_token.near` transfers 990 tokens to Aurora Engine (deducts 10 as fee), then calls `ft_on_transfer(alice, 1000, "<alice_evm_addr>")`.
4. Aurora Engine executes `receive_erc20_tokens` with `amount = 1000` and mints 1000 ERC-20 mirror tokens to Alice's EVM address.
5. Aurora Engine's actual NEP-141 balance: 990. ERC-20 mirror total supply: 1000. Deficit: 10.
6. After many such deposits the deficit grows proportionally.
7. Bob, the last depositor, tries to exit via `ExitToNear` (burns ERC-20 → Aurora attempts `ft_transfer` of NEP-141 back to Bob). The transfer fails because Aurora Engine's NEP-141 balance is insufficient. Bob's funds are permanently frozen.

```rust
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
```
