### Title
Missing Access Control on `update_rewards` Allows Any Caller to Block Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in the Staking contract lacks the access control check specified in the protocol spec ("Only starkware sequencer"). Any unprivileged caller can invoke `update_rewards` with `disable_rewards: true` for any active staker, which updates the global `last_reward_block` without distributing rewards. This prevents the legitimate sequencer from distributing rewards in the same block, permanently blocking consensus reward distribution — an analog to the external report's pattern where a partial operation updates shared state and causes subsequent legitimate calls to revert.

---

### Finding Description

The `update_rewards` function is specified as callable only by the Starknet sequencer:

> **access control**: Only starkware sequencer.

However, the implementation only calls `general_prerequisites()`, which checks for unpaused state and a non-zero caller — no sequencer identity check is performed:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();  // only checks: unpaused + caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards — happens BEFORE the disable_rewards check.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;  // returns early, no rewards distributed
    }
    ...
}
``` [1](#0-0) 

The critical flaw is that `last_reward_block` — a **single global storage variable** (not per-staker) — is written unconditionally before the `disable_rewards` guard. Any call to `update_rewards` in block N sets `last_reward_block = N`. A subsequent call in the same block N reverts with `REWARDS_ALREADY_UPDATED`. [2](#0-1) [3](#0-2) 

The `general_prerequisites` function confirms there is no sequencer check:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [4](#0-3) 

**Attack sequence (analog to the external report's frontrun pattern):**

1. Attacker identifies any currently active staker with non-zero balance (publicly readable on-chain).
2. In block N, attacker calls `update_rewards(active_staker_address, disable_rewards: true)`.
3. The function passes all checks (staker is valid, block is new), writes `last_reward_block = N`, then returns early — **zero rewards distributed**.
4. The legitimate sequencer's call to `update_rewards(staker_address, disable_rewards: false)` in the same block N hits the assertion `current_block_number > last_reward_block` → **reverts with `REWARDS_ALREADY_UPDATED`**.
5. All stakers miss consensus rewards for block N.
6. Repeating this every block permanently freezes all consensus reward accrual.

This mirrors the external report exactly: a "partial" operation (calling with `disable_rewards: true`) updates shared state (`last_reward_block`) without performing the intended effect (reward distribution), causing the legitimate full operation to revert.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

`last_reward_block` is a single contract-wide variable. One attacker transaction per block is sufficient to block reward distribution for **all stakers and all delegators** in the protocol. Stakers and pool members accumulate zero consensus rewards for every blocked block. Since the attack requires no special role and costs only one transaction per block, a motivated attacker can sustain it indefinitely. [5](#0-4) 

---

### Likelihood Explanation

**High.** The entry point is a public function with no role check. The only prerequisite is knowing any active staker address, which is trivially discoverable from on-chain events. The cost is one Starknet transaction per block. No profit motive is required — a competing protocol or griefing actor can execute this continuously.

---

### Recommendation

Add an access control check at the top of `update_rewards` to enforce that only the authorized sequencer address may call it. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
}
```

Alternatively, use the existing `RolesComponent` to assign and check a `SEQUENCER_ROLE`.

---

### Proof of Concept

```
1. Deploy staking contract with consensus rewards active.
2. Staker A stakes and becomes active (epoch + K passes).
3. Attacker (any address) calls:
       staking.update_rewards(staker_A_address, disable_rewards: true)
   in block N.
   → last_reward_block is written to N.
   → No rewards distributed.
4. Sequencer calls:
       staking.update_rewards(staker_A_address, disable_rewards: false)
   in the same block N.
   → Assertion `current_block_number > last_reward_block` fails (N > N is false).
   → Reverts with REWARDS_ALREADY_UPDATED.
5. Staker A and all pool delegators receive zero rewards for block N.
6. Attacker repeats step 3 in every subsequent block.
   → All consensus rewards are permanently frozen.
```
