### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Invalid Invoke Transactions into Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function skips the account's `__validate__` entry point (signature verification) for invoke transactions with nonce=1 from undeployed accounts when `account_tx_in_pool_or_recent_block` returns `true`. However, that check returns `true` for **any** account with **any** pending transaction in the mempool — not specifically for accounts with a pending `deploy_account` transaction. An attacker can exploit this by submitting an invoke transaction with nonce=1 and an invalid signature for any address that has a pending `deploy_account` transaction in the mempool, bypassing signature validation and getting the invalid transaction admitted.

### Finding Description

`skip_stateful_validations` is a UX feature: when a user submits `deploy_account(nonce=0)` + `invoke(nonce=1)` simultaneously, the invoke tx can skip the `__validate__` entry point because the account contract does not yet exist on-chain. The guard used to authorize this skip is `account_tx_in_pool_or_recent_block(sender_address)`. [1](#0-0) 

The guard delegates to: [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool (`tx_pool.contains_account`) or has been seen in any committed block (`state.contains_account`). It does **not** verify that the pending transaction is a `deploy_account` transaction.

The comment in the code acknowledges this ambiguity but incorrectly treats it as safe:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of that disjunction is the flaw: any valid invoke tx with nonce ≥ 1 submitted by the legitimate owner makes the check return `true`, after which an attacker can submit a second invoke(nonce=1) with an **invalid signature** that skips `__validate__`.

**Attack path:**

1. Legitimate user submits `deploy_account(nonce=0)` for fresh address X. It passes all validations and enters the mempool.
2. `account_tx_in_pool_or_recent_block(X)` now returns `true`.
3. Attacker submits `invoke(nonce=1)` for X with an **arbitrary/invalid signature**.
4. Gateway `validate_nonce`: passes — nonce=1 ≥ account\_nonce=0, within `max_allowed_nonce_gap`. [3](#0-2) 

5. Gateway `validate_by_mempool`: passes — nonce=1 is a valid future nonce for account\_nonce=0, no existing tx at nonce=1. [4](#0-3) 

6. Gateway `skip_stateful_validations`: returns `true` — X is in the pool, tx nonce=1, account\_nonce=0. [5](#0-4) 

7. `run_validate_entry_point` is called with `skip_validate=true` → `execution_flags.validate = false` → `__validate__` is **not called**. [6](#0-5) 

8. The invalid tx is admitted to the mempool.

When the legitimate user subsequently submits their own `invoke(nonce=1)`, the mempool rejects it as `DuplicateNonce`. The attacker's invalid tx will fail during batcher execution (the blockifier runs `__validate__` with fresh `ExecutionFlags`), but the user's `deploy_account + invoke` flow is permanently disrupted until they resubmit.

The test suite confirms the skip fires exactly when `contains_tx = true`: [7](#0-6) 

And the integration test confirms the intended (but insufficiently guarded) UX flow: [8](#0-7) 

### Impact Explanation

Invalid transactions — carrying arbitrary signatures that would fail the account's `__validate__` entry point — are admitted to the mempool, bypassing the gateway's stateful signature verification. This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Secondary effect: the attacker can systematically front-run every `deploy_account` transaction observed on the P2P network and inject a competing invalid `invoke(nonce=1)`, rendering the deploy-account-plus-invoke UX feature unusable for all users — analogous to the Winnables raffle bug where anyone could cancel a raffle in the `PRIZE_LOCKED` transitional state before the admin could start it.

### Likelihood Explanation

- `deploy_account` transactions are broadcast over the P2P network and are trivially observable.
- The attacker needs only to submit a single invoke tx per target address; no fee escalation or mempool manipulation is required.
- The attack is cheap: the attacker pays only the gas for the invalid invoke submission; the invalid tx is rejected during execution so no fee is charged.
- No privileged access is required.

### Recommendation

`skip_stateful_validations` must verify that the account has a **pending `deploy_account` transaction** specifically, not merely any transaction. Options:

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

/// Generates a deploy account transaction followed by an invoke transaction from the same account.
/// The first invoke_tx can be inserted to the first block right after the deploy_tx due to
/// the skip_validate feature. This feature allows the gateway to accept this transaction although
/// the account does not exist yet.
pub fn create_deploy_account_tx_and_invoke_tx(
    tx_generator: &mut MultiAccountTransactionGenerator,
    account_id: AccountId,
) -> Vec<RpcTransaction> {
    let undeployed_account_tx_generator = tx_generator.account_with_id_mut(account_id);
    assert!(!undeployed_account_tx_generator.is_deployed());
    let deploy_tx = undeployed_account_tx_generator.generate_deploy_account();
    let invoke_tx = undeployed_account_tx_generator.generate_trivial_rpc_invoke_tx(1);
    vec![deploy_tx, invoke_tx]
}
