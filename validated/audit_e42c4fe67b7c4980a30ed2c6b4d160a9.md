### Title
Unauthorized Invoke Transaction Bypasses Signature Verification via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry-point call (the only on-chain signature check) for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that helper returns `true` for **any** transaction from the address — not specifically a `deploy_account` — an attacker who observes a victim's pending `deploy_account` can inject an invoke transaction with an arbitrary signature for the victim's undeployed address and have it admitted to the mempool without any authorization check.

---

### Finding Description

`skip_stateful_validations` is a UX feature: when a user sends `deploy_account` (nonce 0) and `invoke` (nonce 1) simultaneously, the account does not yet exist on-chain, so the blockifier cannot call `__validate__`. The gateway therefore skips that call when the conditions below are met:

```
tx.nonce() == Nonce(Felt::ONE)
  && account_nonce == Nonce(Felt::ZERO)
  && account_tx_in_pool_or_recent_block(sender_address) == true
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the function returns `Ok(())` immediately, before the `__validate__` call:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

The guard that triggers the skip is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

This returns `true` for **any** transaction from the address — invoke, declare, or deploy_account — not exclusively for a `deploy_account` submitted by the legitimate owner. The code comment acknowledges the assumption but does not enforce it:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [5](#0-4) 

The assumption is broken: an attacker can cause `account_tx_in_pool_or_recent_block` to return `true` for a victim's address by observing the victim's pending `deploy_account` — which was submitted by the victim, not the attacker.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid (unauthorized) transactions.**

An attacker can inject an invoke transaction for any address that has a pending `deploy_account`, with a completely arbitrary signature, and have it admitted to the mempool without any signature check. Concrete consequences:

1. **Nonce-slot squatting**: The attacker's transaction occupies the nonce=1 slot for the victim's address. The victim must perform fee escalation (pay a strictly higher tip) to replace it, or wait for the attacker's transaction to be rejected during execution and the slot to free up.
2. **Block-space waste**: The injected transaction will be included in a block and rejected during execution (because `__validate__` runs with `validate=true` during batcher execution and will fail on the invalid signature), wasting block capacity.
3. **Mempool pollution at scale**: An attacker can monitor the mempool for all pending `deploy_account` transactions and flood the mempool with garbage invokes for every undeployed address, degrading the deploy+invoke UX for all users simultaneously.

---

### Likelihood Explanation

**Medium.** The attack requires:
- Observing a pending `deploy_account` transaction (trivially possible by monitoring the public mempool/P2P layer).
- Submitting a crafted invoke before the `deploy_account` is committed (a short window, but the mempool retains transactions for the configured TTL).

No privileged access is required. Any unprivileged network participant can execute this.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. Add a dedicated mempool API, e.g.:

```rust
fn deploy_account_tx_in_pool(account_address: ContractAddress) -> MempoolResult<bool>;
```

that inspects `tx_pool` for a `DeployAccount` variant at nonce 0 for the given address. Only when that specific transaction is present should signature validation be skipped for the paired invoke.

Alternatively, require the client to supply the `deploy_account` transaction hash alongside the invoke (as the `native_blockifier` `PyValidator` already does via `deploy_account_tx_hash`), and verify that hash is present in the mempool as a `DeployAccount` transaction. [6](#0-5) 

---

### Proof of Concept

1. Alice submits `deploy_account(class_hash=C, salt=S)` for address X → transaction enters the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.

2. Eve observes Alice's pending `deploy_account` in the mempool.

3. Eve submits `invoke(sender_address=X, nonce=1, signature=[0xdeadbeef, 0xdeadbeef], calldata=[...])` to the gateway.

4. Gateway stateless validation passes (signature length ≤ max, address valid, resource bounds non-zero). [7](#0-6) 

5. Gateway stateful validation: `account_nonce = get_nonce(X) = 0`, `tx.nonce = 1`. Nonce check passes (within `max_allowed_nonce_gap`). [8](#0-7) 

6. `skip_stateful_validations`: `tx.nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(X) == true` → returns `true`. [9](#0-8) 

7. `run_validate_entry_point(skip_validate=true)` → `execution_flags.validate = false` → `__validate__` is **never called** → Eve's transaction is admitted to the mempool. [10](#0-9) 

8. Eve's transaction now occupies the `(X, nonce=1)` slot. When Alice submits her legitimate invoke with nonce=1, the mempool sees a duplicate nonce and requires fee escalation to displace Eve's transaction. [4](#0-3)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
