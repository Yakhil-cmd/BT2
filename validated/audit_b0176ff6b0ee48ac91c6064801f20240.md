### Title
FROST `PresignOutput` Derives `Clone` Without Invalidation, Enabling One-Time Nonce Reuse ŌĆö (File: `src/frost/mod.rs`)

### Summary

The `PresignOutput<C>` struct in `src/frost/mod.rs` contains secret one-time-use FROST nonces (`SigningNonces<C>`) but derives `Clone` and `Serialize`/`Deserialize` with no invalidation mechanism. The `sign_v2` function takes the presignature by value (move semantics) as the sole enforcement of single-use, but `Clone` completely bypasses this protection. Any caller that clones or deserializes a stored `PresignOutput` and passes it to `sign_v2` twice with different messages causes nonce reuse, from which the private signing share can be algebraically extracted.

---

### Finding Description

`PresignOutput<C>` is declared in `src/frost/mod.rs`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

The `nonces` field holds the secret binding and hiding nonces generated during the presign phase. These nonces are cryptographically one-time-use: reusing the same nonce pair across two distinct messages allows any observer of both signature shares to recover the signer's private key share via standard linear algebra over the scalar field.

The library's own documentation states this constraint explicitly:

> Each output is consumed **exactly once** (one-time use). [2](#0-1) 

The `sign_v2` function enforces this by taking `presignature: PresignOutput` by value (move semantics):

```rust
pub fn sign_v2(
    ...
    presignature: PresignOutput,
    ...
)
``` [3](#0-2) 

However, because `PresignOutput` derives `Clone`, a caller can trivially clone the value before passing it to `sign_v2`, completely defeating the move-based single-use enforcement. The library's own test suite demonstrates this pattern:

```rust
sign_v2(
    ...
    presign_output.clone(),   // ŌåÉ clone bypasses single-use enforcement
    ...
)
``` [4](#0-3) 

Additionally, `PresignOutput` derives `Serialize`/`Deserialize` but **not** `ZeroizeOnDrop`, meaning the secret nonces are not cleared from memory after use and can be persisted to storage and reloaded for a second signing session. [1](#0-0) 

Compare this to the OT-based ECDSA `PresignOutput`, which at least derives `ZeroizeOnDrop`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }
``` [5](#0-4) 

The FROST `PresignOutput` has neither `ZeroizeOnDrop` nor any consumed/invalidated flag.

The signing logic in `do_sign_coordinator_v2` and `do_sign_participant_v2` directly uses `presignature.nonces` to compute signature shares:

```rust
let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)
``` [6](#0-5) 

If the same `nonces` are used in two calls to `round2::sign` with different `signing_package` values (different messages), the private key share is recoverable.

---

### Impact Explanation

In FROST/EdDSA, each participant's signature share is:

```
z_i = nonce_i + (binding_factor * hiding_nonce_i) + lambda_i * x_i * challenge
```

If the same `nonces` (binding nonce `a_i`, hiding nonce `b_i`) are used for two messages `m1` and `m2` producing challenges `c1` and `c2`, an observer of both signature shares `z_i^(1)` and `z_i^(2)` can compute:

```
z_i^(1) - z_i^(2) = lambda_i * x_i * (c1 - c2)
```

Since `lambda_i`, `c1`, and `c2` are all public, `x_i` (the private signing share) is directly recoverable. This is a **Critical** impact: extraction of a private signing share, which falls squarely within the allowed impact scope ("Extraction, reconstruction, or disclosure of private signing shares").

---

### Likelihood Explanation

The likelihood is **Medium-High** for the following reasons:

1. **`Serialize`/`Deserialize` encourages persistence**: Production systems routinely serialize presignatures to a database for later use. A system that stores a `PresignOutput`, uses it for signing, and then retries on failure (a common pattern) will reuse the same nonces.

2. **`Clone` is explicitly used in the library's own tests**: The test at line 595 of `src/frost/eddsa/sign.rs` calls `.clone()` on a `PresignOutput` before passing it to `sign_v2`. This normalizes the pattern for library consumers.

3. **Malicious coordinator abort-and-retry**: A malicious coordinator can abort a signing session after collecting signature shares (without broadcasting the result), then initiate a new signing session with a different message. If participants stored their presignatures (serialized), they may reuse them in the retry, giving the coordinator two signature shares under the same nonces for different messages.

4. **No library-level protection**: The library provides no nonce-used flag, no `ZeroizeOnDrop` to clear nonces after use, and no API-level guard against reuse.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput`**: The move-based single-use enforcement only works if `Clone` is absent. Removing it makes accidental reuse a compile-time error.

2. **Add `ZeroizeOnDrop`**: Ensure the secret nonces are cleared from memory immediately after use, preventing recovery from memory or serialized state.

3. **Add a consumed/used marker**: If `Clone` must be retained for architectural reasons, introduce a `Cell<bool>` or `AtomicBool` used flag that panics or errors on a second call to `sign_v2` with the same instance.

4. **Document the danger of serialization**: Explicitly warn that deserializing a `PresignOutput` that was already used for signing and using it again is a critical security violation.

---

### Proof of Concept

```rust
// Participant generates a presignature
let presig: PresignOutput<Ed25519Sha512> = run_presign(...);

// Clone the presignature BEFORE passing to sign_v2 (bypasses move enforcement)
let presig_clone = presig.clone();

// First signing session with message "transfer $100"
sign_v2(participants, threshold, me, coordinator,
        keygen_output.clone(), presig, b"transfer $100".to_vec());

// Second signing session with message "transfer $1000000" ŌĆö SAME NONCES
sign_v2(participants, threshold, me, coordinator,
        keygen_output.clone(), presig_clone, b"transfer $1000000".to_vec());

// A malicious coordinator observing both z_i^(1) and z_i^(2) computes:
// x_i = (z_i^(1) - z_i^(2)) / (lambda_i * (c1 - c2))
// recovering the participant's private signing share.
```

The root cause is at `src/frost/mod.rs` line 36 (`#[derive(... Clone ...)]` on `PresignOutput`) combined with the absence of `ZeroizeOnDrop` and any invalidation mechanism. [1](#0-0)
