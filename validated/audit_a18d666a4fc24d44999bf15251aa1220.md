### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` to cap the pool admin's global spread fee component, but `setPoolBinAdditionalFees` applies **no analogous cap** on per-bin `addFeeBuyE6`/`addFeeSellE6`. A pool admin can set per-bin additional fees up to `uint16.max = 65 535` (6.5535 % in E6 units) regardless of what `maxAdminSpreadFeeE6` is set to, bypassing the factory owner's intended fee ceiling and causing traders to pay more than the protocol-sanctioned maximum.

---

### Finding Description

The factory stores two separate admin-controlled fee channels:

**Channel 1 — global admin spread fee** (`adminSpreadFeeE6` in `poolFeeConfig`): [1](#0-0) 

This channel is validated against `maxAdminSpreadFeeE6` before being applied: [2](#0-1) 

**Channel 2 — per-bin additional fees** (`addFeeBuyE6`, `addFeeSellE6` in `BinState`): [3](#0-2) 

No cap check exists. The call passes directly to the pool with no validation: [4](#0-3) 

Both channels are in E6 units and both add to the fee charged to traders during a swap. In the swap loop, the per-bin fee is added directly to `baseFeeX64` before computing the trader's cost: [5](#0-4) 

The `BinState` type stores `addFeeBuyE6` and `addFeeSellE6` as `uint16`, whose maximum value is 65 535 — equivalent to 6.5535 % in E6 units: [6](#0-5) 

The factory owner's hard cap `HARD_MAX_SPREAD_FEE_E6 = 200 000` (20 %) governs only the global fee caps set via `setFeeCaps`. The per-bin channel is entirely outside this governance: [7](#0-6) 

**The mismatch (analog to the external bug):** The factory owner sets `maxAdminSpreadFeeE6` believing it caps all admin-controlled spread fees. The global channel is measured against this cap. The per-bin channel is measured against nothing — it is bounded only by `uint16.max`. These are two different metrics for the same conceptual limit, and they can diverge arbitrarily whenever `maxAdminSpreadFeeE6 < 65 535`.

---

### Impact Explanation

A pool admin can charge traders up to 6.5535 % additional fee per bin regardless of the factory-owner-configured `maxAdminSpreadFeeE6`. If the factory owner sets `maxAdminSpreadFeeE6 = 0` (intending zero admin spread), the admin can still extract 6.5535 % per swap on every bin. Traders receive less output than the protocol-sanctioned price permits — a direct loss of user principal on every swap routed through the affected bin. The factory owner's cap is silently bypassed with no on-chain revert.

---

### Likelihood Explanation

The pool admin is a semi-trusted role, explicitly scoped to operate only within factory-owner-set caps. The bypass requires only a single call to `setPoolBinAdditionalFees` with `addFeeBuyE6 = type(uint16).max`. No timelock, no special condition, no privileged escalation is needed beyond holding the pool admin role. Any pool admin — including one that was granted the role under the assumption that `maxAdminSpreadFeeE6` would constrain their fee power — can trigger this immediately.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (factory) that validates the new per-bin fees against `maxAdminSpreadFeeE6`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, if per-bin fees are intentionally uncapped, document this explicitly and rename `maxAdminSpreadFeeE6` to reflect that it only governs the global component, so factory owners are not misled about the scope of the cap.

---

### Proof of Concept

1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 0` (zero admin spread fee allowed).
2. Pool is created with `adminSpreadFeeE6 = 0` — consistent with the cap.
3. Pool admin calls:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   ```
   No revert occurs. `_binStates[0].addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535`.
4. A trader swaps token1 → token0 through bin 0. The effective fee applied is:
   ```
   baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)   // ≈ baseFee + 6.5535%
   ``` [5](#0-4) 
5. The trader pays 6.5535 % more than the oracle-derived price permits, despite the factory owner having set `maxAdminSpreadFeeE6 = 0`. The factory owner's cap is bypassed with no revert and no event distinguishing this from a legitimate fee update.
