### Title
Signature Validation Bypassed for Invoke Transactions via Overly Broad `skip_stateful_validations` Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator uses `account_tx_in_pool_or_recent_block` to decide whether to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 from an account whose on-chain nonce is 0. The check returns `true` for **any** transaction type in the mempool from that address, not exclusively a `deploy_account` transaction. An unprivileged attacker who observes a deployed account with nonce=0 in state and any pending transaction in the mempool can submit an invoke with nonce=1 carrying an **arbitrary/invalid signature** from that account. The gateway accepts it without ever calling `__validate__`, admitting an unauthorized transaction into the mempool.

---

### Finding Description

`skip_stateful_validations` is the UX feature that allows a user to submit `deploy_account + invoke` simultaneously. The intent is: if the account does not exist on-chain yet (nonce=0) but a `deploy_account` is already queued, the immediately following invoke (nonce=1) should be admitted without running `__validate__` (which would fail because the account contract does not exist yet).

The guard condition is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` whenever the address has **any** transaction in the pool — including a regular invoke with nonce=0 from a fully deployed account. It does not distinguish between a `deploy_account` transaction and any other type.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate=false` the function returns immediately after `perform_pre_validation_stage`, never calling `validate_tx` (the `__validate__` entry point):

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:78-81
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

`perform_pre_validation_stage` checks nonce, fee bounds, and balance — but **not the signature**. The signature is only verified inside `validate_tx` via the `__validate__` entry point call, which is skipped entirely. [5](#0-4) 

---

### Impact Explanation

An attacker can submit an invoke transaction with nonce=1 and a completely invalid (or attacker-chosen) signature from any victim account that satisfies:
1. On-chain nonce = 0 (account deployed but no executed transactions yet).
2. Any transaction from that address is currently in the mempool.

The gateway's stateful validator accepts this transaction without verifying the signature. The transaction enters the mempool. This constitutes admission of an **invalid/unauthorized transaction through signature validation logic**, matching the Critical impact tier. Additionally, the attacker's nonce=1 transaction occupies the mempool slot, causing the legitimate owner's nonce=1 transaction to be rejected as a duplicate nonce — a targeted DoS.

---

### Likelihood Explanation

The preconditions are observable from public mempool state. Any account that has just been deployed and has a pending nonce=0 transaction is a valid target. This is a common pattern (e.g., a user deploys an account and immediately submits their first invoke). No privileged access is required; any network participant can submit transactions to the gateway.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction exists for the address. The mempool should expose a dedicated `has_deploy_account_in_pool(address)` query, or the existing check should be narrowed to only return `true` when the queued transaction for that address is of type `DeployAccount`. The comment in the code acknowledges the assumption but does not enforce it:

```
// "It is sufficient to check if the account exists in the mempool since it means
// that either it has a deploy_account transaction or transactions with future nonces
// that passed validations."
``` [6](#0-5) 

This assumption is false for deployed accounts with pending regular invoke transactions.

---

### Proof of Concept

1. Alice deploys account at address `0xALICE` (nonce=0 in state). She submits invoke tx with nonce=0 (valid signature). This tx is accepted into the mempool.
2. Attacker observes the mempool: `0xALICE` has nonce=0 in state and a pending tx.
3. Attacker crafts an invoke tx: `sender=0xALICE`, `nonce=1`, `signature=[0xDEAD, 0xBEEF]` (invalid).
4. Attacker submits to gateway. `validate_nonce` passes (0 ≤ 1 ≤ max_gap). `validate_by_mempool` passes (nonce gap allowed). `skip_stateful_validations` calls `account_tx_in_pool_or_recent_block(0xALICE)` → `true` (Alice's nonce=0 tx is in pool). Returns `true`.
5. `run_validate_entry_point` is called with `skip_validate=true` → `validate=false`. `__validate__` is never called. Gateway returns success.
6. Attacker's transaction with invalid signature is now in the mempool at nonce=1 for `0xALICE`.
7. Alice's legitimate nonce=1 transaction is rejected by the mempool as a duplicate nonce.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        &self,
        state: &mut S,
