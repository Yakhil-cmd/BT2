### Title
Pausing the Contract After `exit_to_near` Precompile Execution Permanently Freezes User Funds - (`engine/src/contract_methods/connector.rs`)

### Summary

The `exit_to_near_precompile_callback` function enforces a `require_running` check at the start of its execution. This callback is the mandatory second phase of the wNEAR-unwrap exit flow: the first phase burns the user's ERC-20 wNEAR tokens and calls `near_withdraw` on the NEP-141 contract (moving real NEAR onto the Aurora account), while the callback is supposed to forward that NEAR to the user's target account. If an authorized account pauses the Aurora Engine contract between these two phases, the callback permanently fails, leaving the NEAR stranded on the Aurora contract account with no recovery path for the user.

### Finding Description

The `exit_to_near` precompile supports a wNEAR-unwrap flow. When a user calls `withdrawToNear` on the `EvmErc20` contract with the unwrap message, the following two-phase sequence is committed atomically in the first NEAR receipt but resolved across two separate receipts:

**Phase 1 — committed in the `submit` receipt:**
1. The ERC-20 wNEAR tokens are burned from the user's EVM balance (`_burn` in `EvmErc20.sol`).
2. The `exit_to_near` precompile schedules a `near_withdraw` call on the wNEAR NEP-141 contract, which burns wNEAR and deposits real NEAR onto the Aurora contract account.
3. A callback receipt `exit_to_near_precompile_callback` is chained to that promise, carrying `transfer_near` args that specify the target NEAR account and amount.

**Phase 2 — the callback receipt:**

```rust
// engine/src/contract_methods/connector.rs, line 196-245
pub fn exit_to_near_precompile_callback<...>(...) -> Result<...> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // ← checked at callback time
        env.assert_private_call()?;
        ...
        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                // Transfer the unwrapped NEAR to the user's target account
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }
            None
        } else if let Some(args) = args.refund {
            // Refund ERC-20 tokens on failure
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
            ...
        }
        ...
    })
}
```

`require_running` is defined as:

```rust
// engine/src/contract_methods/mod.rs, line 65-70
pub fn require_running(state: &state::EngineState) -> Result<(), ContractError> {
    if state.is_paused {
        return Err(errors::ERR_PAUSED.into());
    }
    Ok(())
}
```

If the contract is paused between Phase 1 and Phase 2, `require_running` returns an error and the entire callback body is skipped. The NEAR that was already deposited onto the Aurora account by `near_withdraw` is never forwarded to the user. In NEAR Protocol, a failed callback does **not** revert the base promise (`near_withdraw`), so the NEAR remains stranded on the Aurora account with no user-accessible recovery path.

The same issue applies to the `error_refund` branch: if the NEP-141 `ft_transfer` in Phase 1 fails and the contract is paused before the callback runs, the refund of burned ERC-20 tokens never executes.

The precompile pause flags (`PrecompileFlags::EXIT_TO_NEAR`, `EXIT_TO_ETHEREUM`) only gate the precompile call itself (Phase 1). The contract-level pause (`is_paused`) gates the callback (Phase 2). These are independent controls, creating the same asymmetry as M-04's independent `onStart`/`onStop` hook status.

### Impact Explanation

**Critical — Permanent freezing of funds.**

- The user's ERC-20 wNEAR tokens are irreversibly burned in Phase 1.
- The real NEAR deposited onto the Aurora account by `near_withdraw` is permanently inaccessible to the user.
- There is no retry mechanism: the callback receipt is a one-shot NEAR receipt; once it fails, it cannot be re-submitted.
- Resuming the contract does not replay the failed callback.

### Likelihood Explanation

**Medium.** The scenario requires two independent events:
1. A user initiates a wNEAR unwrap exit (a normal, documented user action).
2. An authorized account calls `pause_contract` between the `submit` receipt and the callback receipt (a legitimate security response, e.g., to an ongoing exploit).

Both events are individually routine. The window between Phase 1 and Phase 2 is a single NEAR block boundary, which is short but non-zero. In a security incident where the contract is paused urgently, any in-flight exit callbacks will be silently broken. The impact per affected user can be arbitrarily large (the full amount of wNEAR being unwrapped).

### Recommendation

Remove the `require_running` guard from `exit_to_near_precompile_callback`. This callback is not a user-initiated entry point; it is a system-level receipt that must complete to preserve fund integrity. The pause mechanism should only block new user-initiated operations (Phase 1), not the completion of already-committed operations (Phase 2). A targeted fix:

```rust
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        // Do NOT call require_running here — this callback must complete
        // regardless of pause state to avoid permanent fund loss.
        env.assert_private_call()?;
        ...
    })
}
```

### Proof of Concept

1. User holds wNEAR ERC-20 tokens on Aurora and calls `EvmErc20.withdrawToNear("target.near", amount)` with the unwrap message.
2. The `exit_to_near` precompile burns the ERC-20 tokens and schedules `near_withdraw` on the wNEAR NEP-141 account, chaining `exit_to_near_precompile_callback` as the callback.
3. An authorized account calls `pause_contract` on the Aurora engine (e.g., in response to a security incident).
4. The `near_withdraw` receipt executes successfully — NEAR is deposited onto the Aurora contract account.
5. The `exit_to_near_precompile_callback` receipt executes, hits `require_running(&state)?` at line 203, and returns `Err(ERR_PAUSED)`.
6. The NEAR transfer to `target.near` never occurs. The user's ERC-20 tokens are burned. The NEAR is stranded on the Aurora account.
7. Resuming the contract does not replay the failed callback. Funds are permanently frozen.

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
