### Title
Gateway Signature Verification Bypass via `skip_stateful_validations` Allows Invalid Invoke Transactions into Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point (which performs signature verification) for any invoke transaction with `nonce = 1` targeting an account whose on-chain nonce is `0`, provided that `account_tx_in_pool_or_recent_block` returns `true` for that sender address. An unprivileged attacker can exploit this to inject invoke transactions with invalid/arbitrary signatures into the mempool for any victim account that has a pending `deploy_account` transaction, bypassing the only signature check in the gateway admission path.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls three sub-checks in sequence:

1. `validate_state_preconditions` — checks resource bounds and nonce range
2. `validate_by_mempool` — checks nonce validity in the mempool (no signature check)
3. `skip_stateful_validations` — decides whether to skip the `__validate__` entry point
4. `run_validate_entry_point` — runs `__validate__` (or skips it entirely) [1](#0-0) 

The `skip_stateful_validations` function returns `true` (skip validation) when all three conditions hold:

```
tx.nonce() == Nonce(Felt::ONE)   &&
account_nonce == Nonce(Felt::ZERO) &&
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()) == true
``` [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, causing the blockifier's `StatefulValidator` to skip the `__validate__` entry point entirely: [3](#0-2) 

The `account_tx_in_pool_or_recent_block` check is supposed to confirm that a `deploy_account` transaction exists for the sender. However, it actually checks whether the address appears in the mempool pool **or** in the mempool's committed/staged state — it does not verify that the existing transaction is specifically a `deploy_account`: [4](#0-3) 

`MempoolState::contains_account` returns `true` for any address that has staged or committed nonces: [5](#0-4) 

### Impact Explanation

An attacker who observes that victim `V` has submitted a `deploy_account` transaction (which is now in the mempool, so `V`'s on-chain nonce is still `0`) can:

1. Craft an `InvokeTransaction` with `sender_address = V`, `nonce = 1`, arbitrary calldata, and an **invalid or empty signature**.
2. Submit it to the gateway.
3. The gateway's `validate_nonce` passes (nonce 1 is within the allowed gap from account nonce 0).
4. `validate_by_mempool` passes (no signature check, only nonce range).
5. `skip_stateful_validations` returns `true` because `V`'s deploy_account is in the pool.
6. `run_validate_entry_point` skips `__validate__` entirely.
7. The attacker's transaction is admitted to the mempool **without any signature verification**.

If the attacker submits with a higher tip/fee than `V`'s legitimate nonce-1 invoke, the mempool's replacement logic will evict `V`'s valid transaction and keep the attacker's invalid one. When the batcher executes the block, the attacker's transaction will revert (because `__validate__` is called with `validate = true` during actual execution), but `V`'s legitimate transaction has already been displaced from the mempool.

The broken invariant is: **the gateway must not admit transactions whose signatures have not been verified**. This invariant is violated for the specific nonce-1 / account-nonce-0 window.

### Likelihood Explanation

The attack window is the period between a user submitting `deploy_account` and it being included in a block. This window is observable on-chain (the deploy_account appears in the mempool). Any unprivileged attacker monitoring the mempool can trigger this for any new account deployment. The conditions are:
- `tx.nonce == 1` — trivially set by the attacker
- `account_nonce == 0` — true for any not-yet-deployed account
- `account_tx_in_pool_or_recent_block == true` — true as soon as the victim's deploy_account enters the mempool

No privileged access is required.

### Recommendation

The `skip_stateful_validations` check must verify that the transaction in the mempool for the sender address is specifically a `deploy_account` transaction, not just any transaction. The `account_tx_in_pool_or_recent_block` API should be replaced with a more specific `deploy_account_tx_in_pool(sender_address)` check, or the mempool should expose the transaction type so the gateway can confirm the pending transaction is a `deploy_account`.

Alternatively, the skip logic should be moved out of the gateway entirely and into the blockifier's execution path, where the account's class hash can be verified to be zero (confirming the account is not yet deployed) before skipping `__validate__`.

### Proof of Concept

```
1. Victim V submits RpcDeployAccountTransactionV3 → gateway accepts → mempool contains V's deploy_account
   (V's on-chain nonce = 0, account_tx_in_pool_or_recent_block(V) = true)

2. Attacker submits RpcInvokeTransactionV3 {
       sender_address: V,
       nonce: 1,
       calldata: [arbitrary],
       signature: [] // empty / invalid
   }

3. Gateway stateful validation:
   - validate_nonce: nonce 1 >= account_nonce 0, within max_allowed_nonce_gap → PASS
   - validate_by_mempool: nonce range check only, no signature → PASS
   - skip_stateful_validations:
       tx.nonce() == 1 ✓
       account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(V) == true ✓
       → returns true (SKIP __validate__)
   - run_validate_entry_point: validate=false → __validate__ NOT called

4. Attacker's invalid-signature invoke is admitted to the mempool.

5. If attacker's tip > V's legitimate nonce-1 invoke tip:
   - Mempool replaces V's valid invoke with attacker's invalid invoke
   - V's legitimate transaction is lost

6. Batcher executes block:
   - V's deploy_account executes (nonce 0 → 1)
   - Attacker's invoke executes with validate=true → __validate__ called → REVERT
   - V's legitimate invoke is gone; V must resubmit
``` [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
