### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no validation against `maxAdminSpreadFeeE6`, allowing a pool admin to impose per-bin fees that exceed the factory-enforced cap and extract excess fees from traders.

---

### Finding Description

The factory enforces a hard cap on pool-level admin spread fees inside `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, the sibling function `setPoolBinAdditionalFees` performs **no such validation**:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

These bin-level fees are **additive** to the pool-level spread fee in every swap through that bin. In `MetricOmmPool.swap` → `_swapToken1ForToken0SpecifiedOutput`, the effective buy fee is computed as:

```solidity
// MetricOmmPool.sol L910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [3](#0-2) 

And in `getSellAndBuyPrices`:

```solidity
// MetricOmmPool.sol L540-541
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16` (max 65 535, i.e. 6.5535 % in E6 units). The factory owner can lower `maxAdminSpreadFeeE6` to any value (e.g. 1 000 = 0.1 %), but the pool admin can still set bin additional fees up to 65 535 via `setPoolBinAdditionalFees` with no revert, silently charging traders far beyond the cap. [5](#0-4) 

---

### Impact Explanation

Every swap routed through the manipulated bin pays `poolSpreadFee + addFeeBuyE6` (or `addFeeSellE6`). The excess above `maxAdminSpreadFeeE6` is a direct, per-swap loss of trader principal that accrues to the pool's surplus balance and is ultimately collectible by the admin via `collectPoolFees`. This is an admin-boundary break with direct fund loss to traders.

---

### Likelihood Explanation

High. Any pool admin can call `setPoolBinAdditionalFees` at any time with no timelock, no governance delay, and no on-chain cap check. The current bin (the one most swaps traverse) is the natural target.

---

### Recommendation

Mirror the cap check already present in `setPoolAdminFees`. Before forwarding to the pool, validate:

```solidity
if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap and enforce it here.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(5_000, 5_000, ...)` — capping admin spread fee at 0.5 %.
2. Pool admin calls `setPoolAdminFees(pool, 5_000, 0)` — accepted, pool-level fee = 0.5 %.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**, bin 0 additional fee = 6.5535 %.
4. Any trader swapping through bin 0 pays 0.5 % + 6.5535 % = 7.0535 % total spread fee — 14× the factory-enforced cap — with the excess flowing to the admin fee destination. [6](#0-5)
