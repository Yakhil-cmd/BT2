Audit Report

## Title
Missing Caller Ownership Check in `refresh_buyer_tokens` Allows Unauthorized Swap Participation Registration — (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` endpoint in the SNS swap canister accepts an arbitrary `buyer` principal without verifying that the caller is that buyer. Any unprivileged ingress sender can register another user's ICP participation in an SNS swap — including supplying the mandatory `confirmation_text` consent gate on the victim's behalf — as long as the victim has already transferred ICP to the swap canister's subaccount. If the swap commits, the victim's ICP is swept to the SNS treasury and they receive SNS tokens without having explicitly consented.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 130–134, the buyer principal is resolved from the caller-supplied `arg.buyer` string with no ownership check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no check that caller == p
};
```

This `p` is passed directly to `swap_mut().refresh_buyer_token_e8s(p, arg.confirmation_text, ...)` in `rs/sns/swap/src/swap.rs` at line 1134. Inside that function:

1. `validate_confirmation_text` (lines 363–384) only checks that the supplied text matches the SNS-configured text — it does not verify that the entity supplying the text is the buyer whose funds are being committed.
2. The ICP balance is read from the **buyer's** subaccount (lines 1153–1163), keyed on the attacker-supplied `buyer` principal.
3. The buyer's participation is written into `self.buyers` (lines 1285–1288) under the victim's principal.

The grep search across all swap Rust sources confirms there is no `caller == buyer` check anywhere in this path. The proto comment at line 844 of `swap.proto` confirms the field is optional and defaults to the caller only when omitted — there is no enforcement that a non-empty value must equal the caller.

The ticket system (lines 1248–1272) does not block this attack: if no open ticket exists for the victim, the check is skipped entirely and the flow proceeds to register participation.

**Exploit path:**
1. Victim transfers ICP to `swap_canister_subaccount(victim_principal)` but has not yet called `refresh_buyer_tokens`.
2. Attacker reads `confirmation_text` via the public `get_init` query endpoint.
3. Attacker calls `refresh_buyer_tokens({ buyer: "<victim_principal>", confirmation_text: opt "<correct_text>" })`.
4. Swap canister reads victim's ICP balance, validates the attacker-supplied consent text, and writes victim's participation into `self.buyers`.
5. If the swap commits, victim's ICP is swept to the SNS treasury; victim receives SNS tokens without having agreed to the terms.

Without the attacker's call, the victim's ICP would remain in the subaccount and be reclaimable via `error_refund_icp` after the swap closes. The attacker's call forecloses that option.

## Impact Explanation
This is a **High** severity finding. An unprivileged attacker can force a victim's ICP funds into an SNS swap without the victim's consent, bypassing the explicit `confirmation_text` consent gate that exists precisely to record user agreement to swap terms. The victim suffers a concrete financial outcome — loss of ICP in exchange for SNS tokens — that they never authorized. This matches the allowed ICP bounty impact: *"Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds"* with significant user and protocol harm under the SNS framework.

## Likelihood Explanation
The attack requires no privileged access. The victim's principal is public on-chain information. Their ICP subaccount balance on the swap canister is queryable via ledger calls. The `confirmation_text` is set at SNS initialization and is publicly readable from `get_init`. The scenario where a user transfers ICP but has not yet called `refresh_buyer_tokens` (multi-step UI flow, deliberate delay to review terms, or accidental transfer) is realistic and common. The attack is repeatable across any SNS swap that uses `confirmation_text`.

## Recommendation
Add a caller ownership check in `refresh_buyer_tokens` before passing the principal to `refresh_buyer_token_e8s`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() {
        ic_cdk::trap("caller must match the specified buyer");
    }
    specified
};
```

This ensures that only the buyer themselves can register their own participation and supply their own consent text.

## Proof of Concept
1. Deploy a local SNS swap canister with a `confirmation_text` set in `SnsInitPayload`.
2. As identity A (victim), transfer 100 ICP to `swap_canister_subaccount(A)` on the ICP ledger. Do not call `refresh_buyer_tokens`.
3. As identity B (attacker), query `get_init` to retrieve the `confirmation_text`.
4. As identity B, call `refresh_buyer_tokens({ buyer: "<A_principal>", confirmation_text: opt "<correct_text>" })`.
5. Assert: `swap.buyers` now contains an entry for A with the transferred amount.
6. Advance the swap to committed state; assert A's ICP is swept and A receives SNS tokens — without A ever having called `refresh_buyer_tokens` or agreed to the terms.

This can be implemented as a deterministic PocketIC integration test using the existing SNS swap test infrastructure in `rs/sns/swap/tests/`.
