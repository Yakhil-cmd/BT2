### Title
Gateway Admits Unsigned Invoke Transactions for Undeployed Accounts via `skip_stateful_validations` Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the Apollo gateway's stateful validation path unconditionally skips the `__validate__` entry point (the account's signature-verification function) for any Invoke transaction with `nonce = 1` targeting an account address that has **any** transaction in the mempool or a recent block. Because the check is purely address-based and not cryptographically bound to the submitter, any unprivileged attacker can submit a forged Invoke transaction for a victim account that has a pending `deploy_account` transaction, bypassing signature verification entirely and having the transaction admitted to the mempool.

---

### Finding Description

The `skip_stateful_validations` function implements a UX feature: when a user submits a `deploy_account` + `invoke` pair simultaneously, the gateway cannot yet run `__validate__` on the invoke because the account contract does not exist on-chain yet. The function therefore skips blockifier validation when:

1. The transaction is an `Invoke`
2. The transaction nonce is `Nonce(Felt::ONE)` (i.e., `1`)
3. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns early **before** calling `validate_tx` (which invokes `__validate__`): [3](#0-2) 

The `__validate__` entry point is the account contract's signature-verification function. Skipping it means the gateway never verifies that the submitter controls the private key for `sender_address`.

The condition `account_tx_in_pool_or_recent_block(sender_address)` only checks whether the address appears in the mempool pool or committed state: [4](#0-3) 

This check is purely address-based. It does **not** verify that the entity submitting the current Invoke is the same entity that submitted the `deploy_account`. Any observer can see that address `V` has a pending `deploy_account` in the mempool and submit a forged Invoke for `V`.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker who observes a victim's `deploy_account` transaction in the mempool can:

1. Submit `Invoke(sender_address=V, nonce=1, calldata=arbitrary, signature=garbage)`.
2. The gateway checks: `tx.nonce() == 1`, `account_nonce == 0`, `V` is in mempool → `skip_validate = true` → transaction admitted without signature verification.
3. The attacker's forged Invoke occupies nonce slot `1` for victim `V` in the mempool.
4. The victim's legitimate nonce-1 Invoke is rejected by the mempool as a duplicate nonce, or must perform fee escalation to replace the attacker's transaction.

The attacker can submit with a high tip/fee to make replacement expensive or impossible. The victim's account is effectively griefed: their first post-deployment transaction is blocked until they out-bid the attacker.

When the attacker's transaction is eventually executed by the blockifier (with `validate = true` and `strict_nonce_check = true` via `new_for_sequencing`), it will fail `__validate__` and be reverted. However, the damage to the victim's mempool slot has already occurred. [5](#0-4) 

---

### Likelihood Explanation

**High.** The attack requires only:
- Observing the mempool for `deploy_account` transactions (public information in a distributed sequencer).
- Submitting a single Invoke transaction with the victim's address, nonce=1, and any payload.

No privileged access, special keys, or complex setup is required. The attack is trivially automatable against every new account deployment.

---

### Recommendation

The skip-validation path must be restricted to the legitimate account owner. Two complementary fixes:

1. **Verify the signature statically before skipping `__validate__`**: Even without the account contract deployed, the transaction hash and signature can be verified against the `sender_address` using the expected account class's verification logic (e.g., ECDSA over the Starknet transaction hash). This is how the original Starknet gateway handled this case.

2. **Bind the skip to the deploy_account transaction hash**: Instead of checking only that the address exists in the mempool, verify that the `deploy_account` transaction hash in the mempool was submitted with the same key that signed the Invoke. The `deploy_account` transaction's `contract_address` is deterministically derived from `(class_hash, salt, constructor_calldata)`, so the Invoke's signature can be verified against the expected public key embedded in the constructor calldata.

The current comment in the code acknowledges the weakness: [6](#0-5) 

The comment's reasoning ("it means that either it has a deploy_account transaction or transactions with future nonces that passed validations") is circular for the nonce-1 case: the nonce-1 transaction is the one currently being validated, so no prior nonce-1 transaction could have "passed validations."

---

### Proof of Concept

```
1. Alice submits deploy_account for address A (class_hash=C, salt=S, constructor_calldata=[alice_pubkey]).
   → Mempool now contains A's deploy_account. account_tx_in_pool_or_recent_block(A) == true.

2. Bob (attacker) observes A in the mempool.

3. Bob submits:
   Invoke {
     sender_address: A,
     nonce: 1,
     calldata: [<arbitrary malicious calldata>],
     signature: [0x1337, 0xdead],  // garbage
     resource_bounds: <valid>,
   }

4. Gateway stateful validation:
   - account_nonce = get_nonce(A) = 0  (A not deployed yet)
   - validate_nonce: 0 <= 1 <= max_allowed → OK
   - skip_stateful_validations: tx.nonce()==1 && account_nonce==0 && A in mempool → returns true
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       StatefulValidator::perform_validations → returns Ok(()) without calling __validate__
   → Transaction ADMITTED to mempool with garbage signature.

5. Alice submits her legitimate Invoke(sender=A, nonce=1, calldata=[real_call], signature=[valid]).
   → Mempool rejects: DuplicateNonce { address: A, nonce: 1 }
   → Alice must fee-escalate to replace Bob's transaction.

6. Bob's transaction is eventually executed by the blockifier:
   - perform_pre_validation_stage: nonce check passes (strict=true, nonce=1==account_nonce after deploy)
   - validate_tx: __validate__ called → signature invalid → transaction reverts
   → Bob's nonce-1 slot is consumed (reverted), Alice's nonce-1 slot was blocked.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
