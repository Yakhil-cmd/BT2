### Title
Missing Caller Restriction on `update_rewards` Allows Anyone to Permanently Suppress Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is documented to be callable only by the Starkware sequencer, but the implementation contains no such access control check. Any unprivileged address can call it with `disable_rewards: true`, consuming the per-block reward slot and preventing the legitimate sequencer from distributing rewards for that block. Repeated across every block, this permanently freezes all unclaimed yield for every staker and delegator.

### Finding Description
The spec at `docs/spec.md` line 1644–1645 states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1449–1507 only calls `self.general_prerequisites()`, which checks two things: the contract is not paused, and the caller is not the zero address. There is no sequencer identity check.

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);  // consumes the slot

    if disable_rewards || self.is_pre_consensus() {
        return;   // exits without distributing any rewards
    }
    ...
```

`last_reward_block` is a single global value shared across all stakers. Once it is written to the current block number, the `REWARDS_ALREADY_UPDATED` assertion prevents any further call to `update_rewards` in the same block. An attacker who calls `update_rewards(valid_staker, disable_rewards: true)` before the sequencer in any block:

1. Writes `last_reward_block = current_block_number`.
2. Returns immediately without distributing rewards.
3. Causes every subsequent call in that block (including the sequencer's) to revert with `REWARDS_ALREADY_UPDATED`.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

If the attacker front-runs the sequencer every block, no rewards are ever distributed to any staker or delegator. Stakers' `unclaimed_rewards_own` fields are never incremented; pool contracts never receive their reward transfers. The yield is not stolen — it is simply never credited, constituting a permanent freeze of all unclaimed yield for the entire protocol.

### Likelihood Explanation
**High.** The attack requires no funds, no privileged keys, and no special setup. The attacker only needs to submit a transaction calling `update_rewards(any_valid_staker_address, disable_rewards: true)` at the start of each block. Staker addresses are public (emitted in `NewStaker` events). The cost is one cheap transaction per block. The attack is sustainable indefinitely and can be automated.

### Recommendation
Add an explicit caller check inside `update_rewards` to restrict it to the authorized sequencer address, consistent with the spec. For example, store the sequencer address in contract storage and assert `get_caller_address() == self.sequencer_address.read()` at the top of the function, analogous to how `update_rewards_from_attestation_contract` asserts `self.assert_caller_is_attestation_contract()`.

### Proof of Concept

1. Staker Alice stakes and waits K epochs for her balance to become effective.
2. Consensus rewards are activated (`consensus_rewards_first_epoch` is set).
3. Attacker Bob monitors the chain. At the start of every new block, Bob submits:
   ```
   staking_rewards_dispatcher.update_rewards(
       staker_address: alice_address,
       disable_rewards: true
   )
   ```
4. Bob's transaction is included first. `last_reward_block` is set to the current block number. No rewards are distributed.
5. The sequencer's own `update_rewards` call reverts with `REWARDS_ALREADY_UPDATED`.
6. Alice's `unclaimed_rewards_own` is never incremented. Pool contracts never receive STRK.
7. Bob repeats this every block. All staker and delegator yield is permanently frozen.
