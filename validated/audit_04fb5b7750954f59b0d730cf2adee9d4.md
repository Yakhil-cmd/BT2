### Title
Unbounded `pool_member_epoch_balance` Trace Growth Causes Permanent Freezing of Unclaimed Yield via OOG in `claim_rewards` - (File: src/pool/pool.cairo)

### Summary

A pool member can inflate their own `pool_member_epoch_balance` trace without bound by calling `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch. Because `claim_rewards` iterates over every trace entry since the last claim in a single unbounded loop, a sufficiently large trace will cause the call to run out of gas, permanently freezing the pool member's unclaimed yield.

### Finding Description

`claim_rewards` in `src/pool/pool.cairo` calls `calculate_rewards`, which contains an explicitly unbounded loop over the pool member's balance-change trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
``` [1](#0-0) 

Each call to `set_member_balance` (invoked by both `add_to_delegation_pool` and `exit_delegation_pool_intent`) inserts a new checkpoint into the trace at key `current_epoch + K`:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

The underlying `insert` implementation only deduplicates when the key (epoch) is identical to the last entry; otherwise it appends a new checkpoint:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

Neither `enter_delegation_pool` nor `add_to_delegation_pool` enforces a minimum amount beyond `amount > 0`:

```cairo
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
``` [4](#0-3) [5](#0-4) 

The `entry_to_claim_from` cursor is advanced and persisted on each successful `claim_rewards` call, so the loop only re-processes entries since the last claim. However, if the pool member accumulates N balance-change entries across N epochs without claiming, the next `claim_rewards` call must iterate all N entries in one transaction. [6](#0-5) 

There is no mechanism to prune or compact the trace. Once the trace is large enough to exceed the Starknet transaction gas limit, `claim_rewards` will always revert with OOG, and the pool member's accumulated rewards are permanently inaccessible.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the trace length exceeds the gas-per-iteration threshold of a single transaction, `claim_rewards` becomes permanently uncallable for that pool member. The pool member's STRK rewards are locked in the contract forever with no recovery path, because:

1. The trace cannot be shortened — there is no compaction or pruning function.
2. `claim_rewards` processes all pending entries in a single loop with no pagination.
3. `pool_member_info_v1` also calls `calculate_rewards` and would similarly OOG. [7](#0-6) 

### Likelihood Explanation

**Medium.** The attack requires the pool member to call `add_to_delegation_pool(1)` once per epoch over an extended period without claiming rewards. On Starknet, epochs are time-bounded, so the attacker must sustain the inflation over many epochs. However:

- The minimum deposit is 1 wei — there is no economic barrier.
- A motivated attacker (e.g., one who wishes to permanently destroy their own rewards as a griefing act, or a delegator who simply makes frequent small top-ups over months without claiming) can reach the OOG threshold organically.
- The developers themselves acknowledge the loop is unbounded in a code comment, indicating awareness of the risk. [8](#0-7) 

### Recommendation

1. **Enforce a minimum delegation amount** per `add_to_delegation_pool` call (analogous to the FactoryDAO fix) to raise the economic cost of trace inflation.
2. **Paginate `calculate_rewards`**: accept a `max_entries` parameter and allow partial reward claims, storing the updated `entry_to_claim_from` cursor so subsequent calls continue from where the previous left off.
3. **Compact the trace on claim**: after processing entries up to `until_epoch`, remove or overwrite consumed checkpoints so the trace does not grow monotonically.

### Proof of Concept

```
Epoch 0: attacker calls enter_delegation_pool(amount=1)
         → trace: [(epoch=K, balance=1)]

Epoch 1: attacker calls add_to_delegation_pool(pool_member, amount=1)
         → trace: [(epoch=K, balance=1), (epoch=1+K, balance=2)]

Epoch 2: attacker calls add_to_delegation_pool(pool_member, amount=1)
         → trace: [(epoch=K, balance=1), (epoch=1+K, balance=2), (epoch=2+K, balance=3)]

... repeat for N epochs without calling claim_rewards ...

Epoch N: attacker calls claim_rewards(pool_member)
         → calculate_rewards loops over all N entries
         → if N is large enough, transaction runs out of gas
         → rewards permanently frozen
```

Each `add_to_delegation_pool(1)` costs only 1 wei of token plus gas. The trace grows by one entry per epoch. On Starknet, with a finite per-transaction gas limit, there exists a concrete N beyond which `claim_rewards` always OOGs.
