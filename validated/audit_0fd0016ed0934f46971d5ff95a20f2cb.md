### Title
Bootstrap `fee_proposal_fri` Window Seeding Allows Unconstrained Gas Price Manipulation — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During the first `fee_proposal_window_size` (10) blocks, the `fee_proposal_fri` field in `ProposalInit` carries **no bounds enforcement** because `fee_actual` is `None` (the sliding window is incomplete). A malicious proposer can set `fee_proposal_fri` to any value — including `u128::MAX` — during this bootstrap phase. Those values are recorded verbatim into `fee_proposals_window`, become the median `fee_actual` once the window fills, and are then used as the floor for `effective_min` in `calculate_next_l2_gas_price_for_fin`, permanently inflating `next_l2_gas_price` for all subsequent blocks until the manipulated entries age out of the window.

---

### Finding Description

**Step 1 — `fee_actual` is `None` for the first `window_size` blocks.**

`compute_fee_actual` performs a `checked_sub(window_size)` on `height`. When `height < window_size` the subtraction underflows and the function returns `None` with a warning. [1](#0-0) 

**Step 2 — Validator skips the bounds check when `fee_actual` is `None`.**

`is_proposal_init_valid` enforces that `fee_proposal_fri` is `Some` for V0_14_3+ blocks, but the only value-range guard is wrapped in `if let (Some(fee_actual), Some(fee_proposal)) = ...`. When `fee_actual` is `None` the entire bounds check is bypassed. [2](#0-1) 

The code comment explicitly acknowledges this: *"During initiation (fee_actual is None, <window_size blocks), bounds are not enforced."*

**Step 3 — Manipulated values are recorded into the window.**

After a decision is reached the proposer's `fee_proposal_fri` is stored via `record_fee_proposal`, which inserts it unconditionally into `fee_proposals_window`. [3](#0-2) 

**Step 4 — `fee_actual` becomes the floor for `effective_min`.**

Once the window is full, `compute_fee_actual` returns the median of the stored values. `calculate_next_l2_gas_price_for_fin` then sets `effective_min = max(config_min, fee_actual)`, making `fee_actual` a hard floor for the EIP-1559 price adjustment. [4](#0-3) 

**Step 5 — `next_l2_gas_price` is driven toward the inflated floor.**

`calculate_next_base_gas_price` detects `price < min_gas_price` and ratchets the price upward by at most `1/MIN_GAS_PRICE_INCREASE_DENOMINATOR` per block until it reaches `effective_min`. [5](#0-4) 

**Versioned constants confirm `window_size = 10` and `margin_ppt = 2`.** [6](#0-5) 

---

### Impact Explanation

A proposer who controls any of the first 10 blocks can set `fee_proposal_fri` to an arbitrarily large value (e.g., `u128::MAX`). After block 10, `fee_actual` equals the median of those 10 values. `effective_min` is then set to that inflated value, and `next_l2_gas_price` ratchets upward toward it every block. Every user transaction pays fees computed against this inflated price until the manipulated entries age out of the 10-block window — but by then the EIP-1559 price may already be far above the honest equilibrium, and the slow-decay path (`gas_used < gas_target`) can take hundreds of blocks to return to normal.

This matches: **Critical — Incorrect fee/gas accounting with economic impact.**

---

### Likelihood Explanation

In the current single-sequencer deployment this requires the sequencer operator to act maliciously (privileged). In the decentralized validator model the code is designed for, any validator who wins the proposer role during the first 10 blocks can trigger this without any special permission beyond normal consensus participation. The bootstrap phase occurs at genesis and after every chain restart or revert, so the window is not a one-time event.

---

### Recommendation

Enforce a value-range bound on `fee_proposal_fri` even when `fee_actual` is `None`, anchoring it to `l2_gas_price` (the current block's gas price) instead of `fee_actual`. For example, apply the same `fee_proposal_margin_ppt` geometric band around `l2_gas_price` during bootstrap:

```rust
// During bootstrap, bound fee_proposal relative to l2_gas_price.
let reference = proposal_init_validation.fee_actual
    .unwrap_or(proposal_init_validation.l2_gas_price_fri);
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    let (lower, upper) = fee_proposal_bounds(
        reference,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower || fee_proposal.0 > upper {
        return Err(...);
    }
}
```

This mirrors the mitigation suggested for the AMM bug: seed the accumulator with a trusted initial value rather than leaving it unbounded at inception.

### Proof of Concept

```
Block 0..9  (bootstrap, fee_actual = None):
  Malicious proposer sets fee_proposal_fri = u128::MAX in ProposalInit.
  Validator: fee_actual is None → bounds check skipped → proposal accepted.
  record_fee_proposal(height, Some(u128::MAX)) → fee_proposals_window[h] = u128::MAX.

Block 10 (first post-bootstrap block):
  compute_fee_actual(window, 10, 10) → median([MAX; 10]) = u128::MAX.
  calculate_next_l2_gas_price_for_fin(..., fee_actual=Some(u128::MAX)):
    effective_min = max(config_min, u128::MAX) = u128::MAX.
  calculate_next_base_gas_price(current_price, gas_used, gas_target, u128::MAX):
    current_price < u128::MAX → ratchet up by current_price / 333 per block.

Blocks 11..N:
  next_l2_gas_price increases ~0.3% per block toward u128::MAX.
  All user transactions pay fees against this inflated price.
  Effect persists until the 10 manipulated entries age out AND the EIP-1559
  price decays back to equilibrium (hundreds of additional blocks).
```

The honest proposer path (`compute_proposer_fee_proposal` freezes at `l2_gas_price` when `fee_actual` is `None`) is only advisory — nothing in the validator enforces it during bootstrap. [7](#0-6)
