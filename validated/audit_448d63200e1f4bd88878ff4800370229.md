### Title
TOCTOU Race: `prepareSubmitData` Makes Independent `getPoolInfo` Fetch Allowing Malicious Pool to Substitute `targetPuzzleHash` After Display — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary

The pool-join flow uses two completely independent HTTP fetches to the same `/pool_info` endpoint: one in `usePoolInfo` (for display only) and a second in `prepareSubmitData` (for the actual transaction). The submit button is not disabled while `usePoolInfo` is loading. A malicious pool server can delay the display fetch, serve a different `targetPuzzleHash` to the submit fetch, and cause the user to join a pool with a payout address they never saw or approved.

### Finding Description

**Two independent fetches, no shared state:**

`usePoolInfo` in `PlotNFTSelectBase` fetches pool info for display: [1](#0-0) 

Its result is rendered in the "Verify Pool Details" card: [2](#0-1) 

`prepareSubmitData` in `PlotNFTSelectPool` makes a completely separate, independent `getPoolInfo` call at submit time: [3](#0-2) 

The `targetPuzzleHash` extracted from this second fetch is what gets written into the on-chain transaction — it is never cross-checked against the value shown in the UI.

**Submit button is not gated on pool info loading:**

The `ButtonLoading` submit button is only disabled when `exceededNFTLimit` (≥50 NFTs). There is no guard on `poolInfo.loading`: [4](#0-3) 

**`fetchJSON` has a 10-minute default timeout and no request deduplication:** [5](#0-4) 

The two requests are independent `net.request` instances with no shared cancellation or result-sharing.

### Impact Explanation

`targetPuzzleHash` is the pool's payout puzzle hash — it determines where block rewards are sent when the farmer wins a block. If the attacker substitutes a different `targetPuzzleHash` in the submit fetch, the farmer's rewards are directed to the attacker's address for the lifetime of the pool membership (until the farmer notices and switches pools, subject to `relativeLockHeight` delay).

This is a direct, concrete financial impact on farming rewards, matching: *"causes a user to approve, import, sign, send, revoke, burn, join, or display the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation

The attacker must operate a pool server (a realistic precondition — the user chose to connect to this URL). The attack requires:

1. Delay the first `/pool_info` response (display fetch) by several seconds — trivially implemented server-side.
2. Respond immediately to the second `/pool_info` request (submit fetch) with a different `targetPuzzleHash`.
3. The user clicks submit while the spinner is showing — plausible for any user who types a URL and immediately clicks "Create."

The server can distinguish request order by connection arrival time. No client-side secrets, no local compromise, and no cryptographic break are required.

### Recommendation

- **Disable the submit button while `usePoolInfo` is loading or errored.** Pass `poolInfo.loading` and `poolInfo.error` up from `PlotNFTSelectBase` to `PlotNFTSelectPool` and add them to the `disabled` condition on `ButtonLoading`.
- **Eliminate the second fetch.** `prepareSubmitData` should consume the already-resolved `poolInfo` value from `usePoolInfo` rather than calling `getPoolInfo` again. This removes the TOCTOU window entirely.
- **Require pool info to be fully loaded and displayed before enabling submit.** The "Verify Pool Details" step exists precisely to let the user inspect `targetPuzzleHash` — the submit button should enforce that this step has completed successfully.

### Proof of Concept

1. Stand up a local HTTP server at `http://localhost:8080`:
   - On the **first** `GET /pool_info` request: wait 8 seconds, then respond with `{"target_puzzle_hash": "0xAAAA...AAAA", ...}`.
   - On the **second** `GET /pool_info` request: respond immediately with `{"target_puzzle_hash": "0xBBBB...BBBB", ...}` (attacker's address).
2. Open the Chia GUI → Pool → Add a Plot NFT.
3. Enter `http://localhost:8080` as the pool URL.
4. While the "Verify Pool Details" card shows a loading spinner (display fetch still pending), click **Create**.
5. Observe: `prepareSubmitData` fires a second fetch, receives `0xBBBB...BBBB` immediately, and calls `createNewPoolWallet` with `targetPuzzleHash = 0xBBBB...BBBB`.
6. The user never saw `0xBBBB...BBBB` in the UI; the display eventually resolves to `0xAAAA...AAAA`.

The transaction is submitted with the attacker's puzzle hash as the pool payout destination.
