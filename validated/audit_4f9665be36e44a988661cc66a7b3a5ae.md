### Title
Cross-COA Signature Replay via Unbound `signedData` in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` includes the `evmAddress` of the specific COA being authenticated. A signature produced for one COA can be replayed against a different COA owned by the same Cadence account, allowing an attacker to forge ERC1271 approval for any COA whose owner also owns another COA for which a valid proof has been observed.

---

### Finding Description

The `COA` contract implements ERC1271 `isValidSignature()` by calling the Cadence Arch precompile with `address(this)` as the `evmAddress` argument:

```solidity
(bool ok, bytes memory data) = cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig
    )
);
``` [1](#0-0) 

The `_sig` bytes are decoded as a `COAOwnershipProof` containing four attacker-controlled fields: `address` (Cadence Flow account), `path` (PublicPath), `keyIndices`, and `signatures`. The Cadence-side `validateCOAOwnershipProof` function then:

1. Verifies the signatures are valid for keys on the supplied `address` account.
2. Borrows the COA capability at the supplied `path` from `address`.
3. Checks that the borrowed COA's EVM address matches `evmAddress` (i.e., `address(this)`). [2](#0-1) 

The critical gap is that `signedData` (the `_hash` passed to `isValidSignature`) is verified against the account's keys but is **never required to commit to `evmAddress`**. The code itself acknowledges this:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [3](#0-2) 

Because `path` is fully attacker-controlled inside the encoded `_sig` bytes, an attacker who observes a valid proof for COA_1 (at `/public/coa1`) can substitute the `path` field with `/public/coa2` and replay the same signature against COA_2. The signature check still passes (same Cadence account, same keys, same hash), and the EVM address check passes because the capability at `/public/coa2` resolves to COA_2, whose EVM address matches `address(this)` when `isValidSignature` is called on COA_2.

This is the direct analog of the reported Alligator vulnerability: in that case the authority chain was attacker-supplied and unverified; here the `path` binding the proof to a specific COA is attacker-supplied and the signed data is not bound to the target COA's EVM address.

---

### Impact Explanation

Any EVM protocol that uses ERC1271 to verify that a specific COA authorized an action (e.g., off-chain authentication, permit-style approvals, NFT marketplace listings, DAO voting) can be bypassed. An attacker who observes a single valid `COAOwnershipProof` for COA_1 can forge an ERC1271 approval from COA_2 for the same hash, without the Cadence account owner ever intending to authorize COA_2. This constitutes unauthorized account authorization — the COA_2 contract returns `0x1626ba7e` (valid signature) for data it never signed.

---

### Likelihood Explanation

The prerequisite is that the target Cadence account has **two or more COAs published at different public paths**. The comment dismisses this as low-risk because "Flow wallets historically create at most one COA per account," but this is not enforced by the protocol. As the Flow EVM ecosystem matures and multi-COA patterns emerge (e.g., one COA per application, treasury COAs, bridge COAs), this attack surface grows. The attacker needs only to observe one legitimate proof for any COA on the target account — a proof that may have been broadcast publicly as part of a prior authentication flow.

---

### Recommendation

Enforce that `signedData` commits to the specific `evmAddress` being authenticated. The simplest fix is to have `COA.isValidSignature` hash `_hash || address(this)` before passing it to the precompile, or to have `validateCOAOwnershipProof` require that `signedData` encodes `evmAddress` and verify this binding explicitly. This mirrors the fix in the referenced report: binding the proof to the specific proxy address prevents cross-proxy replay.

---

### Proof of Concept

1. Alice has Cadence account `A` with two COAs: COA_1 published at `/public/coa1` and COA_2 published at `/public/coa2`.
2. Alice legitimately signs hash `H` to prove ownership of COA_1 for some EVM protocol. The proof is:
   ```
   {address: A, path: /public/coa1, keyIndices: [0], signatures: [sig_H]}
   ```
3. An attacker observes this proof (e.g., from a public transaction or API call) and constructs a modified proof:
   ```
   {address: A, path: /public/coa2, keyIndices: [0], signatures: [sig_H]}
   ```
4. The attacker calls `COA_2.isValidSignature(H, encode(modified_proof))`.
5. Inside `validateCOAOwnershipProof`, called with `evmAddress = COA_2_evm_address`:
   - `keyList.verify(signatureSet, signedData: H, ...)` → **passes** (Alice's key signed `H`)
   - `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coa2)` → **succeeds** (COA_2 is there)
   - `coaRef.address().bytes == evmAddress` → **passes** (COA_2's EVM address matches `address(this)`)
6. `validateCOAOwnershipProof` returns `ValidationResult(isValid: true, ...)`.
7. `COA_2.isValidSignature` returns `0x1626ba7e` — COA_2 has been made to appear to have signed `H` without Alice's consent for COA_2. [4](#0-3) [5](#0-4)
