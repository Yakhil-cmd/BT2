### Title
Signature Bypass via `skip_stateful_validations` Allows Unauthorized Invoke Transactions for Accounts with Pending Deploy — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (the only on-chain signature check) for any invoke transaction with `nonce = 1` when the sender account has *any* transaction in the mempool and its on-chain nonce is still `0`. An unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an invoke with `nonce = 1` from the victim's address carrying an arbitrary/invalid signature. Because `__validate__` is skipped, the gateway accepts the malicious transaction into the mempool without verifying the signature, occupying the victim's nonce-1 slot and blocking the victim's legitimate first invoke.

---

### Finding Description

**Vulnerable code path** (`crates/apollo_gateway/src/stateful_transaction_validator.rs`):

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range check only)
       ├─ validate_by_mempool            (duplicate / fee-escalation check)
       └─ skip_stateful_validations  ←── returns true → __validate__ is SKIPPED
  └─ run_validate_entry_point(skip_validate = true)
       └─ StatefulValidator::validate()  ←── never reached
```

`skip_stateful_validations` returns `true` when all three conditions hold:

---

### Impact Explanation

**Impact: High** — Mempool/gateway/RPC admission accepts an invalid transaction (one with an arbitrary/forged signature) before sequencing.

Concrete consequences:
1. The attacker's transaction occupies the victim's `nonce = 1` slot in the mempool. The victim's legitimate invoke is rejected with `DuplicateNonce`.
2. The attacker's transaction will revert during block execution (because `__validate__` *is* called during execution), but the victim's transaction has already been blocked.
3. The victim must perform a fee-escalation replacement to recover, incurring extra cost and latency.
4. The attack can be repeated for every new account deployment observed in the mempool, creating a systematic DoS against the deploy-account + invoke UX flow.

---

### Likelihood Explanation

**Likelihood: Low**

- The attacker must observe a victim's `deploy_account` transaction in the mempool (possible via P2P propagation) and race to submit the malicious invoke before the victim does.
- The window is narrow: it closes once the victim's own invoke with `nonce = 1` is accepted (which would then block the attacker via `DuplicateNonce`).
- No privileged access is required; any network participant can submit transactions.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **`deploy_account` transaction** is pending for the sender address. Alternatively, expose a dedicated `has_pending_deploy_account(address)` query on the mempool that inspects transaction types, so that the skip is only granted when the precondition it was designed for actually holds.

---

### Proof of Concept

1. Victim generates a new account address `A` (deterministic from class hash + salt).
2. Victim submits `deploy_account` for `A` (nonce=0). It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits `invoke` from `A` with `nonce = 1` and a random 2-felt signature `[0x1, 0x2]`.
4. Gateway stateful validation:
   - `account_nonce = 0` (A not yet on-chain)
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → **passes**
   - `validate_by_mempool`: no existing tx at `(A, 1)` → **passes**
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns **`true`**
   - `run_validate_entry_point(skip_validate=true)`: `__validate__` is **not called**
5. Attacker's transaction is forwarded to the mempool and accepted.
6. Victim submits their legitimate invoke from `A` with `nonce = 1` and a valid signature.
7. Mempool rejects it: `MempoolError::DuplicateNonce { address: A, nonce: 1 }`. [7](#0-6) [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
