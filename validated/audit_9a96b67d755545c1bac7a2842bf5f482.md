### Title
Non-Consensus Nodes Silently Start with Unencrypted Secrets Database When Encryption Key File Is Absent - (File: cmd/scaffold.go)

### Summary
`initSecretsDB()` in `cmd/scaffold.go` enforces secrets database encryption only for Consensus nodes. For all other node roles (Collection, Execution, Verification, Access), if the AES-256 encryption key file is absent from the bootstrap directory, the node silently falls back to starting with a fully unencrypted Badger secrets database, emitting only a warning log. This is the direct structural analog to the Uniswap iCloud backup vulnerability: sensitive private key material is stored without encryption when the encryption key is absent, rather than failing safely.

### Finding Description
In `cmd/scaffold.go`, `initSecretsDB()` calls `loadSecretsEncryptionKey()` to load the AES-256 key for the Badger secrets database. The key-missing branch at lines 1156–1161 applies asymmetric enforcement:

```go
if errors.Is(err, os.ErrNotExist) {
    if fnb.NodeRole == flow.RoleConsensus.String() {
        // missing key is a fatal error for SN nodes
        return fmt.Errorf("secrets db encryption key not found: %w", err)
    }
    fnb.Logger.Warn().Msg("starting with secrets database encryption disabled")
    // falls through — opts has NO encryption key set
}
``` [1](#0-0) 

When the key file is absent for a non-Consensus node, `opts` is never updated with `WithEncryptionKey`, so `bstorage.InitSecret(opts)` at line 1169 opens the Badger database in plaintext mode. [2](#0-1) 

The secrets database is the designated store for all sensitive cryptographic material. `RecoverablePrivateBeaconKeyStateMachine` enforces at construction time that it must only be instantiated against a secrets database (`operation.EnsureSecretDB`), and `InsertMyBeaconPrivateKey` / `UpsertMyBeaconPrivateKey` write the random beacon private key into it. [3](#0-2) [4](#0-3) 

The `--insecure-secrets-db` flag provides a second, explicit opt-out path for Consensus nodes: [5](#0-4) [6](#0-5) 

The flag is registered for all node types, meaning any node role can pass `--insecure-secrets-db=true` and the Consensus-role guard at line 1152 is the only thing preventing non-Consensus nodes from also using it explicitly — but the silent fallback on missing key file already achieves the same unencrypted state without any flag.

The `NodePrivateKeys` struct, which wraps both `StakingKey` and `NetworkKey`, is explicitly documented as containing all sensitive node information: [7](#0-6) 

### Impact Explanation
When a non-Consensus node (Collection, Execution, Verification, or Access) starts without the encryption key file, the Badger secrets database is written to disk in plaintext. Any party with read access to the node's secrets directory — via a cloud storage misconfiguration, container escape, snapshot/backup exposure, or compromised volume mount — can open the Badger files directly and extract:

- The random beacon private key (for nodes that participate in DKG), enabling forgery of threshold signatures and manipulation of protocol randomness.
- Staking and networking private keys, enabling impersonation of the node on the network.

For Collection nodes, staking key exposure allows an attacker to sign fraudulent collection guarantees. For Execution nodes, it allows signing fraudulent execution receipts. These are direct protocol-integrity impacts, not merely operational ones.

### Likelihood Explanation
The missing-key-file scenario is realistic in production deployments:

1. The bootstrap directory is commonly distributed via cloud object storage (S3, GCS). A misconfigured bucket policy or accidental omission of the `secretsdb-key` file during node provisioning leaves the key absent.
2. The `bootstrap key` command generates the encryption key automatically, but the separate `db-encryption-key` subcommand exists precisely because operators may run them independently — creating a window where the key file is absent.
3. The node starts successfully with only a `Warn`-level log line. Operators monitoring at `Info` or `Error` level will not observe the degraded security state.
4. The `NodeInfo` struct itself carries a documented deprecation warning about accidental private-key exposure in shared contexts, indicating this class of mistake is recognized as realistic. [8](#0-7) 

### Recommendation
- **Short term**: Promote the missing-encryption-key condition to a fatal error for **all** node roles that use the secrets database, not only Consensus nodes. Remove the silent fallback at lines 1160–1161.
- **Long term**: Require the encryption key to be present and valid before `InitSecret` is called for any node role. Consider removing the `--insecure-secrets-db` flag entirely from production builds, or gating it behind a compile-time flag.

### Proof of Concept
1. Bootstrap a Collection, Execution, Verification, or Access node normally using `bootstrap key`, which generates the `secretsdb-key` file.
2. Delete or omit the `secretsdb-key` file from the bootstrap directory before starting the node.
3. Start the node. It starts successfully; the only observable signal is a single `WARN`-level log: `"starting with secrets database encryption disabled"`.
4. The Badger secrets database is created at `--secretsdir` without any encryption key (`WithEncryptionKey` is never called on `opts`).
5. An attacker with read access to the secrets directory opens the Badger `.vlog` and SST files directly (using any Badger reader or raw file inspection) and reads the plaintext-encoded random beacon private key and staking/networking keys stored therein.
