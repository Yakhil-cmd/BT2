### Title
Unguarded `execute_scheduled` Allows Anyone to Force Execution of a User's Scheduled XCC Promise - (File: `etc/xcc-router/src/lib.rs`)

### Summary
The `execute_scheduled` function in the XCC router contract has no access control. Any NEAR account can call it with any nonce to immediately execute a user's scheduled cross-contract call promise. There is also no cancel mechanism, so once a promise is scheduled, the user cannot revoke it. This is the direct analog of the `Timelock.executeTransaction` missing-access-control vulnerability: a privileged operation (executing a user's scheduled NEAR promise) can be triggered by an unprivileged third party at any time.

### Finding Description
The `Router` contract deployed per-user as a NEAR sub-account exposes three methods relevant to promise execution:

- `execute` — parent-only (Aurora Engine), executes a promise immediately.
- `schedule` — parent-only (Aurora Engine), stores a promise in `scheduled_promises` keyed by a monotonically incrementing nonce.
- `execute_scheduled` — **no access control**, callable by anyone. [1](#0-0) 

The comment at line 146 explicitly acknowledges this design choice:

> "It is intentional that this function can be called by anyone (not just the parent). There is no security risk to allowing this function to be open because it can only act on promises that were created via `schedule`."

This reasoning is flawed for the same reason as the Timelock bug: the fact that a promise was previously authorized does not mean executing it at an arbitrary time chosen by a third party is safe. The `schedule` function, which is parent-only, stores arbitrary `PromiseArgs` including `PromiseCreateArgs` with `attached_balance` (NEAR tokens) and arbitrary calldata. [2](#0-1) 

There is no `cancel_scheduled` function anywhere in the `Router` contract. Once a promise is stored, the only way to remove it from storage is to execute it via `execute_scheduled`. [3](#0-2) 

The nonce is a sequential `u64` starting at 0, making all pending scheduled promises trivially enumerable by any observer. [4](#0-3) 

The `CrossContractCallArgs::Delayed` path in the XCC precompile is the entry point that causes promises to be scheduled. When a user submits an EVM transaction targeting the XCC precompile with `Delayed` args, the engine burns the user's wNEAR from their EVM balance and calls `schedule` on the router sub-account. [5](#0-4) 

The `attached_balance` field in `PromiseCreateArgs` represents NEAR tokens that will be sent from the router's balance to the target contract when the promise executes. [6](#0-5) 

### Impact Explanation
An attacker can force execution of any user's scheduled promise at any time by calling `execute_scheduled` with the target nonce. Because:

1. The user's wNEAR is already burned from their EVM balance at schedule time (irreversible on the EVM side).
2. There is no cancel mechanism.
3. The promise may carry `attached_balance` (NEAR tokens from the router's balance).

A forced execution at an unfavorable time — e.g., when a target DeFi contract has bad liquidity, is paused, or has an unfavorable exchange rate — causes the user to lose funds they would have recovered by cancelling or waiting. If the forced execution causes the target call to fail, the NEAR attached to the promise may be returned to the router, but the EVM-side accounting (wNEAR burned) is already finalized, leaving the user's EVM balance permanently reduced with no corresponding NEAR benefit. This constitutes **High: Theft of user funds / Temporary freezing of funds**.

### Likelihood Explanation
- Router sub-account IDs are deterministic and public: `{hex(evm_address)}.{aurora_engine_account}`.
- Scheduled promise nonces start at 0 and increment by 1, so all pending nonces are trivially enumerable.
- Any NEAR account can call `execute_scheduled` with zero special privileges.
- The `CrossContractCallArgs::Delayed` path is a documented, production-supported feature used in integration tests. [7](#0-6) 

Likelihood is **High** for any user of the `Delayed` XCC path.

### Recommendation
- **Short term:** Add a caller check to `execute_scheduled` so that only the parent (Aurora Engine) or the router account itself can trigger execution. This mirrors the access control already applied to `execute` and `schedule`.
- **Long term:** Add a `cancel_scheduled(nonce: U64)` function callable only by the parent, so the Aurora Engine can expose a cancellation path to EVM users. Consider adding a time-lock or expiry to scheduled promises.

### Proof of Concept
1. Alice submits an EVM transaction to the XCC precompile with `CrossContractCallArgs::Delayed(promise_args)` where `promise_args` is an `ft_transfer` of 1000 tokens to address X with `attached_balance: Yocto::new(1)`.
2. The Aurora Engine burns Alice's wNEAR from her EVM balance and calls `schedule` on Alice's router sub-account (`{alice_addr}.aurora`), storing the promise at nonce 0.
3. Alice realizes the market conditions are unfavorable and wants to cancel. There is no cancel function.
4. Eve observes the router sub-account on-chain, sees nonce 0 is pending, and calls:
   ```
   near call {alice_addr}.aurora execute_scheduled '{"nonce": "0"}' --accountId eve.near --gas 300000000000000
   ```
5. The `ft_transfer` executes immediately, sending Alice's 1000 tokens to address X at unfavorable conditions.
6. Alice's EVM wNEAR balance was already burned at step 2 and is not restored. Alice suffers permanent loss of funds. [8](#0-7) [5](#0-4)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
