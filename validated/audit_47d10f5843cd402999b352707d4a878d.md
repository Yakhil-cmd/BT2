Audit Report

## Title
Provider-Level Rate Limit Starvation: One Node Operator Can Exhaust Shared Provider Bucket, Blocking All Sibling Operators — (`rs/registry/canister/src/rate_limits.rs`)

## Summary
`NODE_PROVIDER_MAX_SPIKE` and `NODE_OPERATOR_MAX_SPIKE` are both set to 140. Because `try_reserve_capacity_for_node_operator_operation` debits both the caller's per-operator bucket and the shared per-provider bucket on every operation, a single operator O1 can fully drain the provider bucket within their own operator budget. After 140 committed operations by O1, every sibling operator O2 under the same provider is blocked from all four rate-limited registry operations for at least one 72-minute refill cycle, and indefinitely if O1 re-exhausts the one unit that refills each cycle.

## Finding Description
`rs/registry/canister/src/rate_limits.rs` defines both spike constants identically:

```rust
const NODE_PROVIDER_MAX_SPIKE: u64 = NODE_PROVIDER_MAX_AVG_OPERATIONS_PER_DAY * 7; // = 140
const NODE_OPERATOR_MAX_SPIKE: u64 = NODE_OPERATOR_MAX_AVG_OPERATIONS_PER_DAY * 7; // = 140
```

Both `NODE_PROVIDER_RATE_LIMITER` and `NODE_OPERATOR_RATE_LIMITER` are configured with `max_capacity: 140`.

`try_reserve_capacity_for_node_operator_operation` performs two sequential reservations:
1. One unit from the per-operator bucket keyed by `node_operator_key(node_operator_id)`.
2. One unit from the per-provider bucket keyed by `node_provider_key(node_provider_id)`.

The provider bucket is shared across all operators under the same provider. Because its capacity equals one operator's full spike budget, O1 can drain it entirely within their own operator budget.

The attack vector is `do_update_node_ipv4_config_directly_` with `ipv4_config: None`. In `validate_update_node_ipv4_config_directly_payload`, the IPv4 validity and uniqueness checks are inside `if let Some(ipv4_config) = &payload.ipv4_config`, so they are skipped when the config is absent. The mutation (setting `public_ipv4_config = None`) is idempotent and always succeeds. `commit_used_capacity_for_node_operator_operation` is called unconditionally after `maybe_apply_mutation_internal`, permanently consuming 1 unit from both buckets per call.

After 140 such calls by O1, the provider bucket is at zero. Any subsequent call by O2 to `try_reserve_capacity_for_node_operator_operation` succeeds at the operator-bucket step but fails at the provider-bucket step with `RateLimiterError::NotEnoughCapacity`. The operator reservation is dropped (rolled back via `Drop`), and O2 receives `"Rate Limit Capacity exceeded. Please wait and try again later."` The refill rate is 1 unit per 4320 seconds (72 minutes). O1 can re-exhaust the one refilled unit every 72 minutes, maintaining the DoS indefinitely at a cost of one ingress call per cycle.

All four operations that route through `try_reserve_capacity_for_node_operator_operation` are affected: `do_add_node_`, `do_remove_node_directly_`, `do_update_node_domain`, and `do_update_node_ipv4_config_directly_`.

The existing `test_combined_rate_limiting` test only verifies proportional depletion with uncommitted (dropped) reservations and does not assert that full exhaustion by one operator blocks a sibling.

## Impact Explanation
This is an application/platform-level DoS on the IC registry canister's node management operations. A legitimate node operator registered under the same provider as the victim can permanently block the victim from adding nodes, removing nodes, and updating node domain or IPv4 configuration. This maps to the High impact category: "Application/platform-level DoS... not based on raw volumetric DDoS." The impact is concrete: O2 cannot perform any of the four rate-limited registry operations for as long as O1 maintains the exhaustion, which requires only one ingress call per 72 minutes.

## Likelihood Explanation
The attacker must be a registered node operator under the same node provider as the victim. This is a realistic relationship in the IC ecosystem — a single node provider can and does have multiple operators. The attack requires no privileged access, no governance majority, no key material, and no special technical skill beyond sending ingress calls to the registry canister. The cost is negligible: 140 initial calls plus one call per 72 minutes to maintain the DoS.

## Recommendation
Size the provider-level bucket to accommodate all operators under it fairly. The most direct fix is to increase `NODE_PROVIDER_MAX_SPIKE` to a multiple of the operator spike (e.g., `N × 140` where N is the expected maximum number of operators per provider), so no single operator can exhaust the full provider budget within their own operator budget. Alternatively, add a per-operator cap on provider-bucket consumption so that each operator can consume at most `provider_max / max_operators_per_provider` units from the shared bucket. A cross-operator starvation regression test should also be added to `rs/registry/canister/src/rate_limits.rs` that sets up two operators under one provider, exhausts the provider bucket via O1 with committed reservations, and asserts O2's subsequent call returns a rate-limit error.

## Proof of Concept
```
State: O1 and O2 both registered under provider P.
       Provider bucket: 140/140. O1 operator bucket: 140/140. O2 operator bucket: 140/140.

Step 1: O1 calls do_update_node_ipv4_config_directly_(node_id=N1, ipv4_config=None) × 140
        → each call: operator[O1] -= 1 (committed), provider[P] -= 1 (committed)
        → after 140 calls: operator[O1] = 0, provider[P] = 0

Step 2: O2 calls do_update_node_ipv4_config_directly_(node_id=N2, ipv4_config=None)
        → try_reserve: operator[O2] -= 1 (succeeds, 139 left, not yet committed)
        → try_reserve: provider[P] -= 1 (FAILS, 0 capacity)
        → operator reservation is dropped (rolled back)
        → returns Err("Rate Limit Capacity exceeded. Please wait and try again later.")

Step 3 (maintenance): O1 waits 72 minutes, provider[P] refills to 1.
        O1 calls once more → provider[P] = 0 again.
        O2 remains blocked indefinitely.
```

A deterministic unit test can reproduce this by calling `try_reserve_capacity_for_node_operator_operation` + `commit_used_capacity_for_node_operator_operation` 140 times for O1 at a fixed `now`, then asserting that O2's `try_reserve_capacity_for_node_operator_operation` at the same `now` returns `Err(RateLimiterError::NotEnoughCapacity)`.
