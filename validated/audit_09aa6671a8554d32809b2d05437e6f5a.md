### Title
Missing Declaration-Time Guard for Cairo V0 Classes in `ExecutionStateReader::get_compiled_class` When `class_manager_handle` Is Active — (`crates/apollo_rpc_execution/src/state_reader.rs`)

### Summary

When `class_manager_handle` is `Some`, `ExecutionStateReader::get_compiled_class` enforces a temporal declaration check for `ContractClass::V1` (Sierra) classes but **unconditionally skips it for `ContractClass::V0`** (deprecated/Cairo 0) classes. An unprivileged caller can supply a V0 `class_hash` that was declared in a block *after* the simulation's `state_number`, and the class manager will return the class body while the missing guard allows it to be used as if it were already declared. The simulation, fee estimate, or trace then executes with a class that did not exist at the queried state, producing an authoritative-looking wrong result.

### Finding Description

In `get_compiled_class`, the `class_manager_handle` branch handles two variants: [1](#0-0) 

For `ContractClass::V1`, `is_contract_class_declared` is called, which queries `get_class_definition_block_number` (the Sierra/V1 table) and verifies the declaration block is ≤ `state_number`: [2](#0-1) 

For `ContractClass::V0`, the check is **entirely absent**, with an explicit TODO acknowledging the gap: [3](#0-2) 

The reason the TODO exists is that `is_contract_class_declared` calls `get_class_definition_block_number`, which only reads `declared_classes_block_table` (Sierra classes): [4](#0-3) 

The equivalent for deprecated classes is `get_deprecated_class_definition_block_number`, which reads `deprecated_declared_classes_block_table`: [5](#0-4) 

The storage layer *does* enforce the temporal invariant for V0 classes when accessed directly via `get_deprecated_class_definition_at` (it checks `state_number.is_before(value.block_number)`): [6](#0-5) 

But this storage path is only reached when `class_manager_handle` is `None` (the fallback `get_contract_class` path at lines 144–160). When the class manager is active, the storage fallback is never reached for V0 classes, so the temporal guard is never applied.

### Impact Explanation

The class manager is populated with V0 classes from all synced blocks. When a node has synced to block N, the class manager holds V0 classes declared in blocks 0–N. If a caller requests simulation against block M < N, V0 classes declared in blocks M+1 through N are present in the class manager but should not be visible at `state_number = right_after_block(M)`. Because the declaration check is skipped, `get_compiled_class` returns the class body unconditionally, and the simulation executes with a class that did not exist at the queried state. The returned trace, fee estimate, or simulation result is wrong — it reflects a state that never existed at block M.

This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

- The `class_manager_handle` is `Some` in the production RPC execution path; `simulate_transactions`, `estimate_fee`, and `trace_transaction` all pass a `class_manager_client`: [7](#0-6) 

- Any unprivileged caller can invoke `starknet_simulateTransactions` with a historical `block_id` and a transaction that references a V0 class hash declared after that block.
- No special knowledge is required beyond knowing a V0 class hash that exists on the network — all declared class hashes are publicly visible on-chain.

### Recommendation

Implement the V0 declaration check using `get_deprecated_class_definition_block_number` (analogous to how `is_contract_class_declared` uses `get_class_definition_block_number` for V1). Specifically, in the `ContractClass::V0` arm, query `deprecated_declared_classes_block_table` for the class's declaration block and return `StateError::UndeclaredClassHash` if `state_number.is_before(declaration_block)`. This mirrors the existing guard for V1 and resolves the TODO comment.

### Proof of Concept

**Setup:**
1. Node is synced to block 100. V0 class `C` was declared in block 75. Class manager holds `C`.
2. Caller sends `starknet_simulateTransactions` with `block_id = block 50` and an `Invoke` transaction targeting a contract whose class hash is `C`.

**Execution trace:**
1. `simulate_transactions` builds `ExecutionStateReader` with `state_number = right_after_block(50)` and `class_manager_handle = Some(...)`.
2. During execution, blockifier calls `get_compiled_class(C)`.
3. `class_manager_handle` is `Some`, so `get_executable(C)` is called → returns `ContractClass::V0(body_of_C)`.
4. The `ContractClass::V0` arm is entered. No `is_declared` check is performed.
5. `RunnableCompiledClass::V0(...)` is returned successfully.
6. The simulation executes using class `C` as if it were declared at block 50, when it was not declared until block 75.
7. The RPC response contains a simulation trace and fee estimate computed against a state that never existed at block 50.

**Expected behavior:** `StateError::UndeclaredClassHash(C)` should be returned, causing the simulation to fail or revert, consistent with the actual state at block 50.
