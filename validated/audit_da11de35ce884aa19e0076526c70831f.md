### Title
Unbounded Loop in `calculate_rewards` Over `pool_member_epoch_balance` Trace Enables Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop over a pool member's `pool_member_epoch_balance` trace. A delegator who makes repeated balance changes across epochs without claiming rewards grows this trace without bound. Once the trace is large enough, the `claim_rewards` transaction exceeds the gas limit, permanently freezing the delegator's accumulated unclaimed yield.

### Finding Description
`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) iterates over every entry in `pool_member_epoch_balance` starting from `entry_to_claim_from`:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each iteration calls `find_sigma`, which performs multiple storage reads from `cumulative_rewards_trace_vec`. [2](#0-1) 

The trace grows via `set_member_balance`, which inserts a new checkpoint at `current_epoch + K` (K = 2):

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [3](#0-2) 

The `insert` logic in `PoolMemberBalanceTrace` only updates the last entry if the key matches; otherwise it appends a new checkpoint: [4](#0-3) 

With K = 2, a delegator can append one new trace entry every 2 epochs by calling `add_to_delegation_pool` or `exit_delegation_pool_intent` in alternating epochs. [5](#0-4) 

The `entry_to_claim_from` cursor is only advanced when `claim_rewards` succeeds: [6](#0-5) 

If a delegator never calls `claim_rewards` (or calls it infrequently), the loop must traverse all accumulated entries since the last claim. Both `claim_rewards` and the read-only `pool_member_info_v1` call `calculate_rewards`: [7](#0-6) 

### Impact Explanation
A delegator who makes frequent balance changes across epochs without claiming rewards accumulates an unbounded number of entries in `pool_member_epoch_balance`. Once the trace is large enough, every call to `claim_rewards` runs out of gas. Because `entry_to_claim_from` is only updated on a successful claim, and a successful claim is impossible once the loop is too large, the delegator's accumulated STRK rewards are permanently unclaimable — **permanent freezing of unclaimed yield**.

### Likelihood Explanation
The trace grows at a rate of one entry per 2 epochs. A long-lived delegator who adjusts their delegation frequently (e.g., via `add_to_delegation_pool` or `exit_delegation_pool_intent`) without regularly claiming rewards will organically accumulate entries over time. Additionally, the `add_to_delegation_pool` function permits the pool member's `reward_address` to call it on behalf of the pool member: [8](#0-7) 

If the reward address is a separate party or a contract, it can deliberately grow the trace. The minimum amount is 1 (non-zero check only), so the cost per trace entry is minimal. [9](#0-8) 

The developers themselves acknowledge the risk in a code comment: *"This loop is unbounded but unlikely to exceed gas limits."* [10](#0-9) 

### Recommendation
1. **Enforce a maximum number of unclaimed balance-change entries** before allowing further `add_to_delegation_pool` or `exit_delegation_pool_intent` calls, or require a `claim_rewards` call first.
2. **Alternatively**, restructure reward accounting so that `claim_rewards` does not need to iterate over all historical balance changes — for example, by eagerly settling rewards at each balance change and storing only the running total.
3. **At minimum**, document a hard cap on the number of balance changes between claims and enforce it on-chain.

### Proof of Concept
1. Delegator calls `enter_delegation_pool` with a valid amount.
2. Every 2 epochs, delegator calls `add_to_delegation_pool` with amount = 1 (minimum non-zero). Each call in a new epoch window appends a new entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert`.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N such operations, `pool_member_trace_length` = N.
5. Calling `claim_rewards` triggers `calculate_rewards`, which loops N times, each iteration reading storage via `find_sigma` (up to 3 storage reads per iteration from `cumulative_rewards_trace_vec`).
6. For sufficiently large N, the transaction exceeds the Starknet gas limit and reverts.
7. Since `entry_to_claim_from` is never updated (the transaction always reverts), the delegator's accumulated rewards are permanently frozen.
