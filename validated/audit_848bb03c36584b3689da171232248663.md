### Title
Unprivileged Caller Can Freeze All Stakers' Consensus Block Rewards via `update_rewards(disable_rewards: true)` — (File: src/staking/staking.cairo)

---

### Summary

`IStakingRewardsManager::update_rewards` carries no role-based access control. Any non-zero address may call it with `disable_rewards: true`, which advances the global `last_reward_block` counter without distributing any rewards. Because the contract enforces a strict one-call-per-block invariant on that single global variable, an attacker who front-runs every block with `disable_rewards: true` permanently prevents every staker from receiving consensus block rewards.

---

### Finding Description

`update_rewards` is gated only by `general_prerequisites()`:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [1](#0-0) 

No role check exists. The function then enforces a global, single-slot one-call-per-block guard:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [2](#0-1) 

`last_reward_block` is a single `BlockNumber` field in contract storage — not a per-staker map:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [3](#0-2) 

Immediately after the guard, the block is unconditionally committed:

```cairo
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [4](#0-3) 

When `disable_rewards` is `true`, the function writes the current block number to `last_reward_block` and returns **before** calling `_update_rewards`. Because the slot is now consumed, every subsequent call in the same block — including a legitimate staker call with `disable_rewards: false` — reverts with `REWARDS_ALREADY_UPDATED`.

The `disable_rewards` parameter is part of the public `IStakingRewardsManager` interface with no caller restriction:

```cairo
fn update_rewards(
    ref self: TContractState, staker_address: ContractAddress, disable_rewards: bool,
);
``` [5](#0-4) 

The attacker only needs to supply any currently-active staker address (publicly readable from the `stakers` vector or emitted events). The validity checks — staker existence, non-zero balance — are trivially satisfied with any live staker.

---

### Impact Explanation

Every staker's consensus block rewards are frozen for as long as the attacker sustains the attack. Because `last_reward_block` is global, a single attacker call per block silences reward distribution for the **entire protocol**. Stakers accumulate zero `unclaimed_rewards_own` and pools receive zero `update_rewards_from_staking_contract` calls, freezing delegator yield as well.

This matches the allowed impact: **High — Temporary (or permanent) freezing of unclaimed yield**.

---

### Likelihood Explanation

**Medium.** The attacker requires no special role, no token balance, and no prior relationship with any staker. The only cost is one Starknet transaction per block. Starknet transaction fees are low, making sustained griefing economically feasible. The attacker gains no direct profit but inflicts protocol-wide yield denial.

---

### Recommendation

1. **Restrict the caller.** Add an access-control check so only the staker themselves, their operational address, or a designated consensus contract may call `update_rewards`.
2. **Per-staker tracking.** Replace the global `last_reward_block: BlockNumber` with a `Map<ContractAddress, BlockNumber>` keyed by staker address, so one staker's call cannot block another's.
3. **Remove `disable_rewards` from the public interface.** If the flag is needed only during the pre-consensus transition, handle it internally or restrict it to a privileged role.

---

### Proof of Concept

1. Consensus rewards are active (`current_epoch >= consensus_rewards_first_epoch`).
2. Attacker observes any valid staker address `S` from on-chain events.
3. At the start of block `N`, attacker submits `update_rewards(S, disable_rewards: true)`.
4. Contract writes `last_reward_block = N` and returns without distributing rewards.
5. Staker `S` (or any other staker) submits `update_rewards(their_address, disable_rewards: false)` in the same block → reverts with `REWARDS_ALREADY_UPDATED`.
6. All stakers miss block `N` rewards.
7. Attacker repeats at block `N+1`, `N+2`, … indefinitely.
8. `unclaimed_rewards_own` for every staker remains zero; pool `cumulative_rewards_trace` is never updated; delegators cannot claim yield.
