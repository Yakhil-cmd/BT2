### Title
Missing Lower-Bound Threshold Validation in FROST Signing Allows Single-Party Signature Forgery - (`src/frost/mod.rs`)

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that `threshold` does not exceed the participant count but omits the lower-bound check (`threshold >= 2`) that is present in the DKG path. A caller can pass `threshold = 1` to `sign_v1`, `sign_v2` (EdDSA), or `sign` (RedJubjub), causing the protocol to proceed with a single-party threshold, allowing the coordinator alone to produce a valid threshold signature without the required quorum.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on `threshold`:

```rust
// src/dkg.rs lines 572-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs`, which is the sole validation gate for all FROST signing entry points, only checks the upper bound:

```rust
// src/frost/mod.rs lines 144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check for threshold < 2
``` [2](#0-1) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no minimum enforced at the type level: [3](#0-2) 

Both `sign_v1` and `sign_v2` in the EdDSA path delegate entirely to `assert_sign_inputs` for threshold validation: [4](#0-3) 

The RedJubjub `sign` function does the same: [5](#0-4) 

The FROST `presign` function in `src/frost/mod.rs` also only checks the upper bound, meaning a `PresignArguments` with `threshold = 1` is accepted without error: [6](#0-5) 

---

### Impact Explanation

With `threshold = 1`, the coordinator constructs a `KeyPackage` with `min_signers = 1`. The FROST `aggregate` call then accepts a single signature share as sufficient to produce a valid, verifiable threshold signature. The coordinator — acting alone — can produce a valid EdDSA or RedJubjub threshold signature over any message without the participation of any other honest party. This is an **unauthorized creation of a valid threshold signature for attacker-chosen inputs**, matching the Critical impact tier.

---

### Likelihood Explanation

The entry path is a direct library API call. Any caller (including a malicious coordinator) who controls the `threshold` argument to `sign_v1`, `sign_v2`, or `redjubjub::sign` can pass `threshold = 1`. No special privileges, leaked keys, or cryptographic breaks are required. The missing check is a single missing `if threshold < 2` guard, identical in structure to the check already present in `assert_key_invariants`. The inconsistency between the DKG and signing validation paths makes accidental misuse by an integrator equally likely.

---

### Recommendation

Add the same lower-bound guard to `assert_sign_inputs` in `src/frost/mod.rs` that already exists in `assert_key_invariants` in `src/dkg.rs`:

```rust
// src/frost/mod.rs — inside assert_sign_inputs, after the upper-bound check
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same guard to the `presign` function in `src/frost/mod.rs` where `args.threshold` is validated. [7](#0-6) 

---

### Proof of Concept

1. A malicious coordinator calls `frost::eddsa::sign_v1` with `threshold = 1` and a participant list of two honest parties.
2. `assert_sign_inputs` checks `1 > 2` (false) and `1 > participants.len()` (false) — both pass. No error is returned.
3. The coordinator constructs a `KeyPackage` with `min_signers = 1`.
4. The coordinator sends a signing package to participants but ignores their responses.
5. The coordinator calls FROST `aggregate` with only its own signature share.
6. FROST produces a valid, verifiable EdDSA signature over the coordinator's chosen message without any other participant contributing.
7. The resulting signature is indistinguishable from a legitimately threshold-produced signature.

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
