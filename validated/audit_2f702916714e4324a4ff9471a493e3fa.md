### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `src/pool/pool.cairo` contains an unbounded loop that iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace since their last reward claim. Because each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends a new checkpoint, a delegator who makes many balance changes across many epochs without claiming rewards will eventually cause `claim_rewards` (and `pool_member_info_v1`) to exceed Starknet's per-transaction gas limit, permanently freezing their unclaimed yield.

### Finding Description
`calculate_rewards` iterates from `entry_to_claim_from` to the end of the `pool_member_epoch_balance` trace: [1](#0-0) 

The code itself acknowledges the risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

Each call to `set_member_balance` (invoked by both `add_to_delegation_pool` and `exit_delegation_pool_intent`) inserts a new checkpoint keyed at `current_epoch + K`: [2](#0-1) 

The underlying `PoolMemberBalanceTrace::insert` only updates the last entry in-place when the key matches; otherwise it **appends** a new checkpoint: [3](#0-2) 

Because `entry_to_claim_from` is only advanced when `claim_rewards` is called: [4](#0-3) 

a delegator who makes one balance change per epoch and defers claiming will accumulate one new trace entry per epoch. After N such epochs, the next `claim_rewards` call must iterate N times, each iteration performing storage reads via `find_sigma`. Once N is large enough to exhaust the block gas limit, the transaction always reverts, and the delegator's accumulated yield is permanently inaccessible.

The same loop is also executed inside the view function `pool_member_info_v1`: [5](#0-4) 

meaning even querying the member's state becomes impossible once the trace is large enough.

### Impact Explanation
When the trace grows beyond the gas-limit threshold, every call to `claim_rewards` reverts with an out-of-gas error. The delegator's unclaimed STRK rewards are permanently locked in the pool contract with no recovery path, because there is no mechanism to claim rewards in batches or to reset `entry_to_claim_from` without claiming. This satisfies the **High** impact criterion: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation
The entry path requires only the delegator themselves (an unprivileged actor) to call `add_to_delegation_pool` or `exit_delegation_pool_intent` in distinct epochs without claiming rewards. Active delegators who frequently adjust their stake — a normal and encouraged usage pattern — will naturally accumulate trace entries over time. The protocol has no minimum claim frequency requirement and no cap on trace length. The risk grows monotonically with protocol age and delegator activity. Likelihood is **Medium**: it requires sustained activity over many epochs, but no special privilege or external dependency.

### Recommendation
1. **Short term:** Enforce a maximum number of unclaimed balance-change epochs before a new balance change is allowed, or require `claim_rewards` to be called before `add_to_delegation_pool` / `exit_delegation_pool_intent` if the trace exceeds a safe threshold. Emit a warning event when the trace length approaches a danger zone.
2. **Long term:** Refactor `calculate_rewards` to accept a `max_iterations` parameter so rewards can be claimed in bounded batches, preventing any single transaction from iterating an unbounded number of checkpoints.

### Proof of Concept
1. Delegator calls `enter_delegation_pool` with a small amount.
2. For each of N consecutive epochs, the delegator calls `add_to_delegation_pool(amount: 1)` without ever calling `claim_rewards`. Each call appends one new entry to `pool_member_epoch_balance` (since `get_epoch_plus_k()` returns a strictly increasing key each epoch).
3. After N epochs, the delegator calls `claim_rewards`. The `calculate_rewards` loop iterates N times, each time reading storage via `pool_member_trace.at(entry_to_claim_from)` and `find_sigma`. For sufficiently large N (bounded by Starknet's per-transaction gas limit), the transaction reverts with out-of-gas.
4. Every subsequent `claim_rewards` attempt also reverts. The delegator's accumulated yield is permanently frozen.

The minimum cost to the attacker is N × (gas cost of `add_to_delegation_pool` + 1 token), which is affordable over a long time horizon. No privileged access is required.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
