### Title
Malicious Participant Permanently Terminates Honest Party's Protocol via Malformed Private-Channel Message — (File: `src/protocol/internal.rs`)

### Summary

`PrivateChannel::recv` delegates deserialization to `Comms::recv`, which propagates any `rmp_serde` decode failure as a `ProtocolError` before the sender-identity check is performed. A malicious participant can craft a message whose header matches the deterministic private-channel tag for any two honest parties, inject it with a garbage payload, and cause the victim's `PrivateChannel::recv` to return a permanent error — aborting the entire protocol instance for that honest party. Because the private-channel tags are computed deterministically from public participant identities, no privileged information is required.

### Finding Description

**Root cause — `Comms::recv` propagates deserialization errors unconditionally** [1](#0-0) 

`Comms::recv` pops the next raw message from the `SubMessageQueue` for the requested `MessageHeader`, attempts to decode it with `rmp_serde::decode::from_slice`, and immediately propagates any failure via `?`. The decoded `(Participant, T)` pair is returned only on success.

**`PrivateChannel::recv` propagates the error before checking the sender** [2](#0-1) 

The loop calls `self.comms.recv(...).await?`. The `?` operator exits the loop and returns the error to the caller *before* the `if from != self.to` guard is ever evaluated. Therefore, a message from any participant — including one who is not `self.to` — can terminate the loop with a permanent error if its payload fails to deserialize.

**Private-channel tags are deterministic and publicly computable** [3](#0-2) 

`ChannelTag::root_private(p0, p1)` is a SHA-256 hash of a fixed domain separator and the two sorted participant byte representations. Any participant who knows the participant set can compute the exact 40-byte `MessageHeader` (32-byte channel tag + 8-byte waitpoint) that will be used for the private channel between any two honest parties at any protocol round.

**`push_message` routes solely on the header bytes embedded in the message, not on the authenticated `from` field** [4](#0-3) 

The `from: Participant` argument (authenticated by the network layer) is stored alongside the message but plays no role in routing. A malicious participant C can therefore send a message to honest party A with header bytes equal to `root_private(A, B) || waitpoint_W` and a garbage payload; `push_message` will route it into the `SubMessageQueue` for the A–B private channel at waitpoint W.

**`SubMessageQueue` is an unbounded MPSC channel with no sender-identity filter** [5](#0-4) 

`SubMessageQueue::send` calls `unbounded_send` without any validation. The queue for a given `MessageHeader` accepts messages from any participant.

**The existing test documents the attack surface but does not fix it** [6](#0-5) 

The test `attacker_can_fill_message_buffer_with_unused_waitpoints` explicitly asserts that 512 attacker-injected messages are stored in the buffer, confirming the absence of any injection guard.

**End-to-end exploit path**

1. Participants A, B, C run OT-based ECDSA triple generation (or any protocol using `PrivateChannel`). C is malicious.
2. C computes `header = root_private(A, B) || waitpoint_W` for the round where A waits for a private message from B.
3. C sends a message to A with those header bytes prepended to a garbage payload (e.g., a single `0xFF` byte).
4. A's `push_message` routes the message into the `SubMessageQueue` for `(root_private(A, B), W)`.
5. A's `PrivateChannel::recv` (expecting B's message) calls `comms.recv`, which pops C's message, fails to deserialize the payload, and returns `Err(ProtocolError::DeserializationError(...))`.
6. The `?` in `PrivateChannel::recv` propagates the error; A's protocol future resolves to `Err(...)`.
7. A's `ProtocolExecutor::poke` returns `Action::Return(Err(...))`, permanently terminating A's participation.
8. Without A, the signing/keygen/reshare/CKD session cannot complete for any honest party.

### Impact Explanation

A single malicious participant — requiring no privileged keys, no leaked secrets, and no cryptographic breaks — can permanently abort any protocol instance that uses `PrivateChannel` (OT-based ECDSA triple generation, and any other two-party sub-protocol). The abort is immediate and unrecoverable within the session: the honest party's `ProtocolExecutor` has already resolved to an error and will return `Action::Wait` on all subsequent `poke()` calls. This constitutes **permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties**, matching the High impact tier.

### Likelihood Explanation

The attack requires only that the adversary be a registered participant (a normal precondition for any threshold protocol invocation) and that they know the participant set (public information). The exact headers are deterministically computable. No timing precision or side-channel access is needed. The attack succeeds on the first injected message.

### Recommendation

1. **Catch deserialization errors in `PrivateChannel::recv` and retry.** When `comms.recv` returns a deserialization error, log or count the failure and `continue` the loop rather than propagating the error. Only propagate errors that indicate a broken invariant (e.g., the sender channel is closed).

2. **Filter by authenticated sender before deserializing.** Restructure `MessageBuffer`/`SubMessageQueue` to store raw bytes alongside the authenticated `from` field, and expose a `pop_from(header, expected_sender)` API so that `PrivateChannel::recv` can discard messages from unexpected senders without attempting deserialization.

3. **Apply the same fix to `SharedChannel::recv` / `recv_from_others`.** The shared-channel path has the same deserialization-before-validation ordering and is equally exploitable.

### Proof of Concept

```rust
// Attacker C computes the private-channel header for the A–B channel at waitpoint 0.
let tag = ChannelTag::root_private(participant_a, participant_b);
let header = MessageHeader::new(tag); // waitpoint = 0

// Craft a message: header bytes || garbage payload
let mut malicious_msg = header.to_bytes().to_vec();
malicious_msg.push(0xFF); // invalid msgpack

// C delivers this message to A via the authenticated network layer.
// The `from` field is C (authenticated), but routing is based on header bytes.
protocol_a.message(participant_c, malicious_msg);

// On the next poke(), A's PrivateChannel::recv pops C's message,
// rmp_serde fails to decode 0xFF as the expected type T,
// the error propagates through `?`, and A's protocol returns permanently.
let action = protocol_a.poke(); // => Ok(Action::Return(Err(DeserializationError(...))))
```

```rust
        // Sort participants, for uniqueness.
        let (p0, p1) = (p0.min(p1), p0.max(p1));

        let mut hasher = Sha256::new();
        hasher.update(NEAR_CHANNEL_TAGS_DOMAIN);
        hasher.update(b"root private");
        hasher.update(b"p0");
        hasher.update(p0.bytes());
        hasher.update(b"p1");
        hasher.update(p1.bytes());

        let out = hasher.finalize().into();
        Self(out)
    }
```

```rust
struct SubMessageQueue {
    sender: futures::channel::mpsc::UnboundedSender<(Participant, MessageData)>,
    receiver: Arc<Mutex<futures::channel::mpsc::UnboundedReceiver<(Participant, MessageData)>>>,
}

impl SubMessageQueue {
    pub fn send(&self, from: Participant, message: MessageData) {
        // This cannot fail because the receiver is also alive.
        self.sender
            .unbounded_send((from, message))
            .expect("unbound_send should not fail");
    }
}

impl Default for SubMessageQueue {
    fn default() -> Self {
        let (sender, receiver) = futures::channel::mpsc::unbounded();
        Self {
            sender,
            receiver: Arc::new(Mutex::new(receiver)),
        }
    }
}
```
