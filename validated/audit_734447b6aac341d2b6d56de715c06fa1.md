### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on the pool admin's fee components. The parallel function `setPoolBinAdditionalFees` applies no such cap, allowing the pool admin to set per-bin additional spread fees up to `uint16.max = 65535` (≈6.55% in E6 units) on any bin, bypassing the admin fee cap entirely.

### Finding Description

`MetricOmmPoolFactory` exposes two pool-admin fee-setting paths:

**Path A — `setPoolAdminFees`** enforces caps:
```solidity
// MetricOmmPoolFactory.sol lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

**Path B — `setPoolBinAdditionalFees`** has no cap check:
```solidity
// MetricOmmPoolFactory.sol lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The per-bin additional fees are directly added to the effective swap fee in every swap path:
```solidity
// e.g. line 999 (buy token0 specified-in path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [3](#0-2) [4](#0-3) 

The `BinState` struct stores `addFeeBuyE6` and `addFeeSellE6` as `uint16`, so the maximum settable value is 65535 (≈6.5535% in E6 units). [5](#0-4) 

The `maxAdminSpreadFeeE6` cap is set by the factory owner via `setFeeCaps`, bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%). If the owner sets `maxAdminSpreadFeeE6` to a low value (e.g., 1000 = 0.1%), the pool admin is blocked from raising the global admin spread fee above 0.1% via `setPoolAdminFees`, but can freely set `addFeeBuyE6 = 65535` (6.5535%) on every bin via `setPoolBinAdditionalFees`, achieving a 65× bypass of the intended cap. [6](#0-5) [7](#0-6) 

### Impact Explanation

Traders swapping in affected bins pay effective spread fees of up to `spreadFeeE6 + 65535` instead of the capped `spreadFeeE6 + maxAdminSpreadFeeE6`. The excess fee is extracted directly from the trader's swap output (or added to their input), constituting a direct loss of user principal on every swap through those bins. The pool admin collects this excess via the admin fee destination.

### Likelihood Explanation

The pool admin is semi-trusted only within the cap system. Any pool admin who wishes to extract excess fees from traders can do so immediately after pool creation by calling `setPoolBinAdditionalFees` on all active bins. No timelock, no protocol approval, and no on-chain guard prevents this. The call is a single transaction.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the same cap at pool creation time inside `_unpackAndValidateBinStates` for the initial bin data. [8](#0-7) 

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 1000` (0.1%) via `setFeeCaps`.
2. Pool admin calls `setPoolAdminFees(pool, 2000, 0)` → reverts with `AdminFeeTooHigh`. Cap is enforced.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` → succeeds. No cap check.
4. A trader swaps through bin 0. The effective buy fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately 6.5535% additional spread on top of the base fee — far exceeding the 0.1% cap the factory owner intended.
5. The excess fee accrues to the pool's LP surplus and is collected by the admin fee destination on the next `collectPoolFees` call. [2](#0-1) [9](#0-8)
