Audit Report

## Title
Unbounded Frame Size Allocation and Deserialize Panic in `FrameDecoder::decode()` Allow Compromised Sandbox to Crash Replica — (`File: rs/canister_sandbox/src/frame_decoder.rs`)

## Summary
`FrameDecoder::decode()` reads a raw `u64` frame-size field from the IPC socket and calls `data.reserve(size)` with no upper-bound check. A compromised sandbox process can send an 8-byte header containing `u64::MAX`, causing the replica's `CanisterSandboxIPC` thread to attempt an allocation of ~18 EiB, triggering an OOM kill or allocator panic. A second crash vector exists on the same code path: `bincode::deserialize(&frame).unwrap()` panics on any malformed payload. Neither vector is guarded by `SocketReaderConfig`, which carries no `max_message_size` field.

## Finding Description
`FrameDecoder<Message>::decode()` in `rs/canister_sandbox/src/frame_decoder.rs` (L40–52):

1. Reads 8 bytes and interprets them as a `u64` frame size with no validation (`data.get_u64()`).
2. Casts the `u64` directly to `usize` and calls `data.reserve(size)` — no upper bound enforced.
3. Calls `bincode::deserialize(&frame).unwrap()` — panics on any deserialization error.

`socket_read_messages()` in `rs/canister_sandbox/src/transport.rs` (L442–450) drives this decoder in a tight loop. `SocketReaderConfig` (L289–323) controls only idle-trim and malloc-trim behaviour; it has no `max_message_size` field. The `CanisterSandboxIPC` thread in `rs/canister_sandbox/src/replica_controller/launch_as_process.rs` (L115–131) calls `socket_read_messages` with `SocketReaderConfig::default()`, adding no protection. The same pattern applies to the `SandboxLauncherIPC` thread (L47–69) which also uses `SocketReaderConfig::default()`.

Existing checks are absent: there is no size guard anywhere between the raw socket read and the `reserve()` call.

## Impact Explanation
A compromised sandbox process sends a single 8-byte frame header containing `0xFFFFFFFFFFFFFFFF`. The replica's `FrameDecoder` reads the size, casts it to `usize`, and calls `data.reserve(18_446_744_073_709_551_615)`. On a 64-bit Linux host this triggers an immediate OOM kill of the replica process or a Rust allocator panic, both of which terminate the process. The replica process is the consensus and state-management engine for the node; its crash halts that node's participation in block production. This maps to **High — Application/platform-level DoS, crash, or subnet availability impact** (single-node crash degrading subnet liveness). The `bincode::deserialize(...).unwrap()` path provides a second, equally reliable crash vector requiring only a syntactically valid length followed by garbage bytes.

## Likelihood Explanation
The IC security model explicitly treats the sandbox process as an untrusted boundary: canister Wasm runs inside the sandbox, and any Wasmtime or instrumentation bug granting native code execution inside the sandbox immediately enables this attack. The attack requires only writing 8 bytes to the Unix-domain socket the sandbox already holds open toward the replica. No network access, no privileged credentials, and no consensus-level corruption are needed — only native code execution inside one sandbox process, which is the exact threat the sandbox boundary is designed to contain. The sandbox is the execution environment for all user-deployed canisters, making this reachable from an unprivileged canister owner who exploits a Wasm-escape bug.

## Recommendation
1. Add a `max_frame_size: usize` field to `SocketReaderConfig` and enforce it in `FrameDecoder::decode()` before calling `data.reserve()`. A reasonable limit is the maximum inter-canister payload size (currently 2 MiB for responses, with a small overhead for framing). Return an error or close the socket if the limit is exceeded.
2. Replace `bincode::deserialize(&frame).unwrap()` with a `match` or `?`-propagation that returns `None` (or closes the socket) on deserialization failure instead of panicking.
3. Propagate the limit through all call sites: `SocketReaderConfig::default()`, `SocketReaderConfig::for_sandbox()`, and `SocketReaderConfig::for_testing()`, and update both `spawn_canister_sandbox_process` and `spawn_launcher_process` to use the bounded config.

## Proof of Concept
From inside a compromised sandbox process (one that has escaped Wasm isolation), write the following 8 bytes to file descriptor 3 (the Unix-domain socket to the replica controller):

```rust
// Inside compromised sandbox process — fd 3 is the controller socket
use std::os::unix::io::FromRawFd;
use std::io::Write;
let mut sock = unsafe { std::os::unix::net::UnixStream::from_raw_fd(3) };
// Frame header: u64 big-endian value of u64::MAX
sock.write_all(&u64::MAX.to_be_bytes()).unwrap();
```

The replica's `CanisterSandboxIPC` thread calls `FrameDecoder::decode()`, reads the size `18446744073709551615`, casts it to `usize`, and calls `data.reserve(18446744073709551615)`. The allocator immediately fails, the thread panics, and the replica process terminates.

A deterministic unit test can reproduce this without a real sandbox escape:

```rust
#[test]
fn test_oversized_frame_crashes() {
    let mut buf = BytesMut::new();
    buf.put_u64(u64::MAX); // craft malicious header
    let mut decoder = FrameDecoder::<SomeMessage>::new();
    decoder.decode(&mut buf); // triggers data.reserve(u64::MAX) → panic/OOM
}
```
