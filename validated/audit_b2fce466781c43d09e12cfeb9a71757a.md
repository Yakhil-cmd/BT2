### Title
COA Ownership Proof Signatures Do Not Commit to the Target EVM Address, Enabling Cross-COA Replay - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account key signatures are valid over an arbitrary `signedData` blob, then separately checks that the COA stored at the given `path` matches the supplied `evmAddress`. Because the protocol never enforces that `signedData` encodes `evmAddress`, a valid proof produced for one COA (COA_A) owned by a Cadence account can be replayed verbatim against a different COA (COA_B) owned by the same account. The code itself acknowledges this gap in a comment but leaves the function publicly callable without any mitigation. Any Cadence script, transaction, or off-chain authentication flow that calls `validateCOAOwnershipProof` directly is therefore vulnerable to cross-COA signature replay.

### Finding Description

`EVM.validateCOAOwnershipProof` is declared `access(all)` and accepts six caller-controlled parameters:

```
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
```

The function performs two independent checks:

1. It verifies that the supplied `signatures` are cryptographically valid over `signedData` using the account keys at `keyIndices` on the Cadence account `address`.
2. It borrows the `CadenceOwnedAccount` resource at `path` and checks that its `.address().bytes` equals `evmAddress`.

Crucially, `signedData` is never required to contain `evmAddress`. The two checks are structurally decoupled. [1](#0-0) 

The signature verification step: [2](#0-1) 

The EVM address matching step (separate, after signature check): [3](#0-2) 

Because `path` and `evmAddress` are both attacker-supplied when calling the function directly, an attacker who observes a valid proof `(signedData=D, signatures=S, keyIndices=K)` for COA_A can construct a new call:

```
EVM.validateCOAOwnershipProof(
    address: victim_cadence_address,
    path: /public/coaB,          // different COA
    signedData: D,               // same signed data
    keyIndices: K,               // same key indices
    signatures: S,               // same signatures
    evmAddress: COA_B_address    // different EVM address
)
```

The function returns `isValid: true` for COA_B even though the victim only ever signed `D` in the context of COA_A.

The `COAOwnershipProof` struct that is RLP-encoded and passed through the EVM precompile path does not include `SignedData` or `EVMAddress` — those are supplied externally: [4](#0-3) 

The `COAOwnershipProofInContext` wrapper that carries `SignedData` and `EVMAddress` is assembled by the caller, not by the signer: [5](#0-4) 

### Impact Explanation

Any Cadence-side or off-chain authentication protocol that calls `EVM.validateCOAOwnershipProof` directly (rather than through the EVM precompile) and uses the result to authorize privileged actions on behalf of a specific COA is vulnerable. An attacker who obtains a valid proof for COA_A can replay it to impersonate COA_B (a different COA owned by the same Cadence account), bypassing ownership verification. Depending on what the relying protocol does with a `true` result — e.g., authorizing a bridge withdrawal, an ERC-1271 approval, or an off-chain login — this can lead to unauthorized asset movement or unauthorized account access.

The EVM-side precompile path (`verifyCOAOwnershipProof` called from Solidity) is not directly exploitable because `coa.sol` hardcodes `address(this)` as the EVM address argument: [6](#0-5) 

However, `validateCOAOwnershipProof` is `access(all)` and is callable by any unprivileged Cadence script or transaction with fully attacker-controlled arguments, making the Cadence-direct call path the exploitable surface.

### Likelihood Explanation

The preconditions are:
1. The victim Cadence account holds more than one COA (technically unrestricted by the protocol).
2. A valid proof for one of those COAs has been observed (e.g., from a prior on-chain transaction or off-chain authentication flow).
3. A relying protocol calls `validateCOAOwnershipProof` directly and acts on the result.

Condition 1 is uncommon today ("Flow wallets historically create at most one COA per account" per the code comment), but is not prevented by the protocol. Conditions 2 and 3 are realistic for any DeFi or bridge protocol built on top of COA ownership proofs. Likelihood is **low-medium**: the attack surface is real and reachable by an unprivileged transaction sender, but requires a multi-COA account and a direct-call relying party.

### Recommendation

Enforce that `signedData` commits to the target `evmAddress`. The simplest fix is to require callers to include the EVM address bytes in `signedData` and add an on-chain check inside `validateCOAOwnershipProof`:

```cadence
// Inside validateCOAOwnershipProof, after signature verification:
// Require that signedData encodes evmAddress to prevent cross-COA replay.
let addrBytes = evmAddress
var addrFound = false
if signedData.length >= 20 {
    addrFound = true
    for i, b in addrBytes {
        if signedData[signedData.length - 20 + i] != b {
            addrFound = false
            break
        }
    }
}
if !addrFound {
    return ValidationResult(isValid: false, problem: "signedData must commit to evmAddress")
}
```

Alternatively, the protocol can construct the message to sign as `keccak256(abi.encode(evmAddress, userProvidedData))` and verify this inside the function, mirroring the recommendation in the original report.

### Proof of Concept

**Setup**: Alice's Cadence account `0xALICE` holds two COAs:
- COA_A at `/public/coaA` with EVM address `0xAAAA...`
- COA_B at `/public/coaB` with EVM address `0xBBBB...`

**Step 1 – Legitimate proof for COA_A is produced** (e.g., for an off-chain login):
```
signedData = randomChallenge   // does NOT include 0xAAAA...
signatures = sign(signedData, aliceKey0)
proof_A = validateCOAOwnershipProof(
    address: 0xALICE,
    path: /public/coaA,
    signedData: randomChallenge,
    keyIndices: [0],
    signatures: [sig],
    evmAddress: 0xAAAA...
) // → isValid: true
```

**Step 2 – Attacker replays the same signature against COA_B**:
```cadence
// Any unprivileged Cadence script:
import EVM from 0xEVM
let result = EVM.validateCOAOwnershipProof(
    address: 0xALICE,
    path: /public/coaB,          // ← different path
    signedData: randomChallenge, // ← same bytes
    keyIndices: [0],             // ← same key
    signatures: [sig],           // ← same signature
    evmAddress: 0xBBBB...        // ← different COA address
)
// result.isValid == true  ← COA_B ownership falsely proven
```

The signature check passes because `sig` is valid over `randomChallenge` with Alice's key. The address check passes because COA_B is at `/public/coaB` and its address is `0xBBBB...`. The function has no mechanism to detect that the signature was produced for a different COA. [7](#0-6) [2](#0-1) [8](#0-7)
