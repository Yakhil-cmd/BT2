### Title
Malicious Participant Can Equivocate on `commitment_hash` via `send_many` to Permanently Abort Honest Parties' DKG — (`src/dkg.rs`)

---

### Summary

The commitment hash round (Step 2.9) in `do_keyshare` uses `chan.send_many` — a plain peer-to-peer send with no consistency guarantee — rather than `do_broadcast` (echo broadcast). A malicious participant can exploit this to send a different `commitment_hash` value to different subsets of honest parties. Because the actual commitment is later delivered via echo broadcast (Step 3.2), all honest parties receive the same commitment `C`, but their locally stored hashes for the malicious sender differ. The `verify_commitment_hash` check in Step 4.4 then causes a subset of honest parties to abort with `InvalidCommitmentHash`, permanently denying them key generation.

---

### Finding Description

In `src/dkg.rs`, `do_keyshare` computes a `commitment_hash` and sends it to all other participants: [1](#0-0) 

This uses `chan.send_many` — documented explicitly as "peer-to-peer sending with no security guarantees" (`docs/network-layer.md`): [2](#0-1) 

Later, the actual commitment is distributed via `do_broadcast` (echo broadcast), which guarantees all honest parties receive the same value: [3](#0-2) 

In Step 4.4, each honest party verifies that the commitment received via broadcast hashes to the value received in the hash round: [4](#0-3) 

The `verify_commitment_hash` function computes `H(commitment)` and compares it against `all_hash_commitments[p]` — the value stored from the unprotected `send_many` round: [5](#0-4) 

Because the `Protocol` trait exposes `Action::SendMany` and `Action::SendPrivate` as separate primitives driven by the external runner: [6](#0-5) 

A malicious participant is not constrained to emit a single `Action::SendMany`. They can instead emit `Action::SendPrivate(P1, hash_A)` and `Action::SendPrivate(P2, hash_B)` — sending different hashes to different honest parties. The library has no mechanism to enforce that the same hash was sent to all parties in this round.

---

### Impact Explanation

- Honest party P1 receives `hash_A = H(C)` → `verify_commitment_hash` passes → P1 continues.
- Honest party P2 receives `hash_B ≠ H(C)` → `verify_commitment_hash` returns `Err(ProtocolError::InvalidCommitmentHash)` → P2 permanently aborts.

The DKG session is permanently broken for P2 (and any other party that received a wrong hash). Since the commitment itself arrives consistently via echo broadcast, there is no way for P2 to distinguish a network fault from a malicious hash — it simply aborts. This matches the **High** impact: *Permanent denial of key generation for honest parties under valid protocol inputs and documented trust assumptions*.

---

### Likelihood Explanation

Any participant in a valid DKG session can execute this attack. No special privilege, leaked key, or external compromise is required. The attacker only needs to control their own protocol instance and route their `send_many` messages as targeted `send_private` messages to different parties. The attack is deterministic and requires no cryptographic break.

---

### Recommendation

Replace the `send_many` / `recv_from_others` pattern for the commitment hash round with `do_broadcast`, matching the treatment of the commitment itself in Step 3.2. This ensures all honest parties agree on the same `commitment_hash` for each sender before the commitment is revealed, closing the equivocation window.

Concretely, Step 2.9 should become:

```rust
// Step 2.9 — use echo broadcast instead of send_many
let all_hash_commitments = do_broadcast(
    &mut chan,
    &participants,
    me,
    commitment_hash,
).await?;
```

This mirrors the existing pattern used for the commitment and proof of knowledge in Step 3.2 and is consistent with the protocol's stated reliance on reliable broadcast for security.

---

### Proof of Concept

Simulate a 3-honest + 1-malicious DKG session. The malicious participant M:

1. Computes `commitment_hash = H(C)` honestly.
2. Sends `commitment_hash` (correct) to P1 via `send_private`.
3. Sends a random/wrong `bad_hash` to P2 via `send_private`.
4. Broadcasts `C` honestly via `do_broadcast` (Step 3.2).

Expected result:
- P1 completes Step 4.4 successfully.
- P2 returns `Err(ProtocolError::InvalidCommitmentHash)` at `verify_commitment_hash`.

This is directly analogous to the existing test `do_broadcast_dishonest_consume_version_1` in `src/protocol/echo_broadcast.rs` (lines 412–437), which already demonstrates that a malicious participant can send different values to different parties when echo broadcast is absent — and that honest parties diverge as a result.     async fn do_broadcast_dishonest_consume_version_1(
        mut chan: SharedChannel,
        participants: ParticipantList,
        me: Participant,
    ) -> Result<Vec<bool>, ProtocolError> {
        let wait_broadcast = chan.next_waitpoint();
        let sid = participants.index(me)?;

        // malicious reliable broadcast send
        let vote_true = MessageType::Send(true);
        let vote_false = MessageType::Send(false);

        for (cnt, p) in participants.others(me).enumerate() {
            if cnt >= participants.len() / 2 {
                chan.send_private(wait_broadcast, p, &(&sid, &vote_false))?;
            } else {
                chan.send_private(wait_broadcast, p, &(&sid, &vote_true))?;
            }
        }

        let vote_list =
            reliable_broadcast_receive_all(&chan, wait_broadcast, &participants, me, vote_false)
                .await?;
        let vote_list = vote_list.into_vec_or_none().unwrap();
        Ok(vote_list)
    }
