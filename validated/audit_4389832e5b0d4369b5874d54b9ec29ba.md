### Title
Deployed Account with Nonce=0 Allows Signature-Bypass via `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` uses `account_nonce == Nonce(Felt::ZERO)` as a proxy for "account not yet deployed," but this condition is also true for any **already-deployed** account that has never sent a transaction. When such an account has any pending transaction in the mempool, an attacker can submit a nonce-1 invoke with an **invalid signature** that completely bypasses the `__validate__` entry-point check and is admitted to the mempool.

---

### Finding Description

`extract_state_nonce_and_run_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` orchestrates the gateway stateful path:

```
account_nonce = get_nonce_from_state(sender)
skip_validate = run_pre_validation_checks(tx, account_nonce, mempool)
run_validate_entry_point(tx, skip_validate)   // __validate__ skipped when skip_validate=true
``` [1](#0-0) 

`run_pre_validation_checks` calls `skip_stateful_validations`, which returns `true` (skip `__validate__`) when all three conditions hold:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
``` [2](#0-1) 

The intent (per the comment) is to support the deploy-account + invoke UX: the account does not exist yet, a `deploy_account` is pending in the mempool, and the first post-deploy invoke should not fail solely because the deploy has not been processed. The comment explicitly acknowledges a second case: *"transactions with future nonces that passed validations."*

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in either the mempool's `tx_pool` (any pending transaction) or its `state` (staged/committed nonces):

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`MempoolState::contains_account` checks the `staged` and `committed` maps:

```rust
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [4](#0-3) 

**The broken invariant:** `account_nonce == Nonce(Felt::ZERO)` does **not** distinguish between:

- A **non-existent** account (not deployed) — the intended case.
- A **deployed** account with nonce=0 (deployed, but no transaction ever committed).

For a deployed account with nonce=0, any valid invoke (e.g., nonce=2 with a correct signature) passes `__validate__` and enters the mempool. Once `tx_pool.contains_account` returns `true` for that address, the three conditions in `skip_stateful_validations` are all satisfied for any subsequent nonce=1 invoke from the same address — regardless of its signature.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is gated on this flag:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [6](#0-5) 

The transaction is therefore admitted to the mempool with no signature verification.

---

### Impact Explanation

An attacker can submit a nonce=1 invoke for any deployed account with nonce=0 that already has a pending transaction in the mempool, using an **arbitrary or invalid signature**, and have it accepted by the gateway and inserted into the mempool. Two concrete sub-impacts:

1. **Nonce-slot hijacking via fee escalation:** If the account owner has a valid nonce=1 invoke pending, the attacker submits a nonce=1 invoke with a higher fee and an invalid signature. `skip_stateful_validations` returns `true` (the owner's tx is already in the pool), `__validate__` is skipped, and the attacker's invalid transaction replaces the owner's valid one via the fee-escalation path. The owner's transaction is evicted; the attacker's invalid transaction occupies the slot.

2. **Nonce-gap injection:** If the owner has a nonce=2 invoke pending (creating a gap), the attacker fills nonce=1 with an invalid transaction. When the batcher executes it, `__validate__` runs and fails; the transaction reverts but the nonce is incremented and a fee is charged from the account's balance.

Both sub-impacts match: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The precondition — a deployed account with nonce=0 that has any pending transaction in the mempool — is common: every freshly deployed account that submits its first transaction is in this state for the duration of the mempool's processing window. The mempool is public, so the attacker can observe the pending transaction. No privileged access is required; the attacker only needs to craft an invoke transaction with the victim's address as `sender_address`.

---

### Recommendation

Replace the coarse `account_nonce == Nonce(Felt::ZERO)` proxy with an explicit check that the account does **not** exist in the blockchain state (i.e., has no deployed class hash). If the account is already deployed, the deploy-account UX skip does not apply and `__validate__` must always run. Alternatively, restrict `account_tx_in_pool_or_recent_block` to only return `true` when a `deploy_account` transaction (not just any transaction) is present in the pool for the address, so the proxy accurately reflects the intended invariant.

---

### Proof of Concept

1. Deploy account `A` (class hash set, nonce=0 in blockchain state).
2. Account owner submits `invoke_v3` for `A` with `nonce=2` and a valid signature. It passes `__validate__` and enters the mempool. Now `tx_pool.contains_account(A) == true`.
3. Attacker submits `invoke_v3` for `A` with `nonce=1` and `signature = [0x0, 0x0]` (invalid).
   - Gateway reads `account_nonce = 0` from state.
   - `validate_nonce`: `0 <= 1 <= max_gap` — passes.
   - `validate_by_mempool`: no duplicate hash, nonce in range — passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
   - `run_validate_entry_point` called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` not called.
   - Transaction admitted to mempool with invalid signature.
4. Batcher picks up nonce=1 invoke, runs `__validate__` during execution — it fails, transaction reverts, fee charged from `A`'s balance.
5. Attacker repeats step 3 with escalating fees to continuously drain `A`'s balance or to displace the owner's nonce=1 transaction via fee escalation.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
