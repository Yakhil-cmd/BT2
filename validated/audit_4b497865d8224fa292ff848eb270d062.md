### Title
Unverified Participant Contributions in CKD Protocol Allow Malicious Participant to Corrupt Confidential Key Derivation Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` aggregates elliptic-curve contributions `(big_y, big_c)` from every participant with no cryptographic proof of correctness. A single malicious participant can send arbitrary curve points; the coordinator blindly adds them into the running sum and outputs a permanently corrupted `CKDOutput`. The resulting confidential key is wrong for every honest caller, matching the external report's class of *trusting a heuristic that is safe under honest behaviour but offers no guarantee under adversarial input*.

---

### Finding Description

`do_ckd_coordinator` collects each participant's `CKDOutput` and accumulates it unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation each participant is supposed to perform is:

```
big_s  = x_i · H(pk ‖ app_id)          // partial BLS signature
big_c  = big_s + app_pk · y_i           // ElGamal ciphertext share
big_y  = y_i · G                        // ephemeral public key share
``` [2](#0-1) 

There is no NIZK proof (e.g. a discrete-log-equality proof) that `big_s` was derived from the participant's actual private share `x_i`, and no proof that `big_c` is a well-formed ElGamal ciphertext. The coordinator has no way to distinguish a correct contribution from an arbitrary pair of group elements.

The participant side simply sends whatever `compute_signature_share` returns with no binding commitment:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

A malicious participant running a modified protocol instance can substitute any `(Y', C')` before the `send_private` call.

---

### Impact Explanation

Let `(Y_h, C_h)` be the sum of all honest contributions. A malicious participant injects `(Y', C')`. The coordinator computes:

```
Y_total = Y_h + Y'
C_total = C_h + C'
```

The `unmask` call by the application then yields:

```
C_total − app_sk · Y_total
  = msk · H(pk ‖ app_id)  +  (C' − app_sk · Y')
```

The additive term `(C' − app_sk · Y')` is an attacker-controlled, unknown corruption. The output is not the correct confidential key and cannot be corrected without re-running the protocol. This directly matches:

> **High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session is a sufficient attacker. They need only substitute arbitrary `G1` points for their honest contribution before sending to the coordinator. No special knowledge (private keys, `app_sk`, etc.) is required to corrupt the output. The attack is one-shot and requires no coordination with other parties.

---

### Recommendation

Require each participant to attach a NIZK proof of correct formation alongside their `(big_y, big_c)` share. Concretely, a Chaum–Pedersen / discrete-log-equality proof demonstrating:

```
DLEQ( big_s / H(pk ‖ app_id)  ==  vk_share_i / G )
```

where `vk_share_i = x_i · G` is the participant's public verification share (already derivable from the DKG output commitments). The coordinator must verify this proof before accumulating the contribution, rejecting and aborting on failure. This replaces the implicit "trust the participant" assumption with a cryptographic guarantee, directly analogous to replacing `CONFIRMATION_BLOCKS=30` with a finalized-block tag.

---
