Audit Report

## Title
Floating-Point Precision Loss in `convert_tokens_to_string_representation` Produces Incorrect Consent Message Amounts — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

`convert_tokens_to_string_representation` converts a `Nat` token amount to a display string via `BigUint::to_f64()`, which silently rounds any integer above 2^53 due to IEEE 754 f64's 53-bit mantissa. The rounded value is then formatted into the ICRC-21 `GenericDisplayMessage` consent message. Because the consent message is the sole human-readable authorization surface shown to a user before signing, displaying an amount that differs from the actual on-chain debit violates the core security invariant of the ICRC-21 standard and enables a malicious dApp to obtain user approval for a transfer amount the user did not explicitly authorize.

## Finding Description

The vulnerable function at [1](#0-0)  calls `tokens.0.to_f64()` on a `num_bigint::BigUint`. Per the `num-traits` crate, `to_f64()` returns `None` only when the value exceeds `f64::MAX` (~1.8×10^308); for all practical token amounts it returns `Some(f64)` — but silently rounds any integer above 2^53 (9,007,199,254,740,992) to the nearest representable f64. The rounded value is divided by `10_f64.pow(decimals)` and formatted into the consent message string.

This function is called from four public methods on `ConsentMessage` in the `GenericDisplayMessage` branch:
- `add_amount` at [2](#0-1) 
- `add_fee` at [3](#0-2) 
- `add_allowance` at [4](#0-3) 
- `add_existing_allowance` at [5](#0-4) 

The `FieldsDisplayMessage` branch uses `nat_to_u64` at [6](#0-5)  which returns an error for values above `u64::MAX` and is exact within range — so only the `GenericDisplayMessage` path is affected.

The `icrc21_canister_call_consent_message` endpoint is publicly exposed on the ICRC-1 ledger, ICP ledger, ckBTC minter, and ckDOGE minter, as confirmed by [7](#0-6)  and [8](#0-7) .

Exploit flow:
1. Attacker (malicious dApp) constructs a `TransferArg` with `amount = 2^53 + K`, where K is chosen so `to_f64()` rounds down.
2. The wallet calls `icrc21_canister_call_consent_message` to obtain the human-readable consent message.
3. The ledger generates a `GenericDisplayMessage` showing the rounded-down f64 value.
4. The user sees and approves the consent message, believing they are authorizing the displayed (smaller) amount.
5. The wallet signs and submits the original `TransferArg` with the exact `Nat` value.
6. The ledger debits the exact `Nat` value — which differs from what the user approved.

The relative error is bounded by machine epsilon (~2^-52), so the absolute discrepancy in raw token units grows with the magnitude of the amount. For values near 2^63 (~9.2×10^18 raw units), the ULP is 1024 raw units. For a token with 8 decimals, that is 0.00001024 TOKEN per transaction. While small per transaction, the invariant violation is structural and affects every `GenericDisplayMessage` consent message for amounts above 2^53.

## Impact Explanation

This is a concrete violation of the ICRC-21 consent message security invariant on in-scope production ledger and minter canisters (ICRC-1 ledger, ICP ledger, ckBTC, ckDOGE). The consent message is the sole human-readable authorization surface; displaying an amount that differs from the actual debit means users can be made to authorize transfers for amounts they did not explicitly consent to. This constitutes a significant ledger security impact with concrete user harm, fitting the High impact class: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The attack requires no special privileges. Any unprivileged caller can submit a `TransferArg` with an amount above 2^53 via the public `icrc21_canister_call_consent_message` endpoint. No key material, governance role, or subnet access is needed. The condition is trivially satisfiable on any ICRC-1 ledger or ckBTC/ckDOGE minter deployed on ICP mainnet. The attack is repeatable and deterministic.

## Recommendation

Replace the lossy `f64` conversion with exact arbitrary-precision integer arithmetic using `BigUint` division and modulo:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let divisor = num_bigint::BigUint::from(10u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let frac  = &tokens.0 % &divisor;
    if decimals == 0 {
        Ok(whole.to_string())
    } else {
        Ok(format!("{}.{:0>width$}", whole, frac, width = decimals as usize))
    }
}
```

This produces an exact decimal string for any `Nat` value with no floating-point involvement.

## Proof of Concept

```rust
#[test]
fn precision_loss_above_2_pow_53() {
    use candid::Nat;
    use num_bigint::BigUint;

    // 2^53 + 1 cannot be represented exactly as f64
    let amount = Nat(BigUint::from(2u64.pow(53)) + BigUint::from(1u64));
    let displayed = convert_tokens_to_string_representation(amount.clone(), 8).unwrap();

    // Parse the displayed string back to raw units and compare
    let displayed_units: u64 = displayed.replace('.', "").parse().unwrap();
    let actual_units: u64 = amount.0.try_into().unwrap();

    assert_eq!(displayed_units, actual_units,
        "Consent message shows {} but actual amount is {}", displayed_units, actual_units);
    // FAILS: displayed_units == 9007199254740992, actual == 9007199254740993
}
```
