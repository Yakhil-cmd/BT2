### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` as a hard ceiling on pool-admin-controlled spread fees, but `setPoolBinAdditionalFees` applies no analogous cap to per-bin additional fees (`addFeeBuyE6` / `addFeeSellE6`). A pool admin can therefore charge effective swap fees that exceed the factory-enforced cap, breaking the admin-boundary invariant.

---

### Finding Description

The factory exposes two pool-admin fee-setting paths:

**Path 1 — pool-level spread fee (capped):** [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

**Path 2 — per-bin additional fee (no cap):** [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

There is no check of `addFeeBuyE6` or `addFeeSellE6` against `maxAdminSpreadFeeE6`. The pool-level function accepts `uint16` values, so the pool admin can set bin fees up to `65 535 / 1e6 ≈ 6.5 %` per bin regardless of what the factory owner has set as the cap.

Inside the pool, the per-bin additional fee is added directly on top of the base spread fee during every swap: [3](#0-2) 

```solidity
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

The pool-level `setBinAdditionalFees` enforces only a bin-index range check, not a fee magnitude check: [4](#0-3) 

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
```

**Analog to the external report:** The staking bug checks `rd > 0` instead of `rd > minDuration_`, allowing arbitrarily short lock periods. Here, `setPoolAdminFees` checks `newAdminSpreadFeeE6 <= maxAdminSpreadFeeE6`, but `setPoolBinAdditionalFees` performs no analogous check, allowing the pool admin to charge fees that exceed the cap through the bin-fee path.

---

### Impact Explanation

The factory owner uses `maxAdminSpreadFeeE6` (and `setFeeCaps`) to bound how much a pool admin can extract from traders. A pool admin can trivially circumvent this bound by calling `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` on every active bin. Every swap through those bins pays the uncapped bin fee on top of the pool-level spread fee, directly reducing the amount of output token received by the trader. This constitutes a direct loss of trader principal and a broken admin-boundary invariant.

---

### Likelihood Explanation

Any pool admin of any pool (regardless of the factory-owner-set cap) can call `setPoolBinAdditionalFees` at any time. No special precondition is required beyond being the registered pool admin. The trigger is a single transaction.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's `maxAdminSpreadFeeE6` via a factory call, or store a per-pool bin-fee cap as an immutable set at deployment time.

---

### Proof of Concept

1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 1_000` (0.1 %).
2. Pool admin calls `setPoolAdminFees(pool, 1_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; bin 0 now carries an additional ~6.5 % fee on both buy and sell sides.
4. Every swap routed through bin 0 pays ~6.6 % total fee instead of the factory-capped 0.1 %, with the excess accruing as surplus that the pool admin collects via `collectPoolFees`. [2](#0-1) [5](#0-4) [4](#0-3) [3](#0-2)
