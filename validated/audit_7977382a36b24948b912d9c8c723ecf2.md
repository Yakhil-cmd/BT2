### Title
`skip_stateful_validations` Bypasses `__validate__` for Nonce-1 Invoke When Any Future-Nonce Invoke Is in Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip `__validate__` only when a deploy-account transaction is pending for an undeployed account (UX feature). However, the implementation checks `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction in the pool for that address — including a future-nonce invoke. An attacker with a deployed account at nonce=0 can submit a valid nonce=2 invoke (which passes `__validate__`), then submit a nonce=1 invoke with an **invalid signature**. The gateway skips `__validate__` for the nonce=1 invoke and admits it to the mempool.

### Finding Description

The `skip_stateful_validations` function in `crates/apollo_gateway/src/stateful_transaction_validator.rs` checks two conditions before skipping `__validate__`:

1. The incoming invoke has `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)`
2. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The code's own comment acknowledges the flaw:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**." [2](#0-1) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

This returns `true` for **any** transaction type in the pool — not just deploy-account. A nonce=2 invoke from the same address satisfies this check.

**Attack path:**

1. Attacker controls address A, which is a deployed contract with on-chain nonce=0 (e.g., pre-deployed via genesis, or deployed via a non-deploy-account mechanism).
2. Attacker submits a nonce=2 invoke with a **valid** signature. `skip_stateful_validations` returns `false` (nonce ≠ 1), so `__validate__` runs and passes. The nonce=2 invoke is admitted to the mempool. `tx_pool.contains_account(A)` is now `true`.
3. Attacker submits a nonce=1 invoke with an **invalid** signature. The gateway checks:
   - `tx.nonce() == Nonce(Felt::ONE)` ✓
   - `account_nonce == Nonce(Felt::ZERO)` ✓ (on-chain state unchanged)
   - `account_tx_in_pool_or_recent_block(A)` → `true` (nonce=2 invoke is in pool) ✓
4. `skip_stateful_validations` returns `true`. `run_validate_entry_point` is called with `skip_validate=true`, so `execution_flags.validate = false` and `__validate__` is **not executed**. [4](#0-3) [5](#0-4) 

The nonce=1 invoke with an invalid signature is admitted to the mempool without signature verification.

**Why the "undeployed account + nonce=2 invoke" variant in the question does not work:**

If the account is not deployed (no contract at address A), the nonce=2 invoke submitted in step 2 would fail `__validate__` (no contract code to call), be rejected by `run_validate_entry_point`, and never reach the mempool. `account_tx_in_pool_or_recent_block` would return `false`, and `skip_stateful_validations` would return `false` for the nonce=1 invoke. The exploit requires a **deployed** account with nonce=0.

### Impact Explanation

An invoke transaction with an invalid signature is admitted to the mempool without `__validate__` execution. This is a concrete corrupted admission value: the gateway's stateful validation decision is wrong. The transaction will fail during sequencing when the blockifier calls `__validate__`, but it has already been admitted, consuming mempool capacity and potentially causing downstream issues (stuck nonce gaps, wasted sequencer resources).

Impact: **High** — "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

### Likelihood Explanation

Requires a deployed account with nonce=0 (achievable via pre-deployed contracts or genesis accounts). The attacker must spend gas on a valid nonce=2 invoke to seed the mempool. The attack is fully unprivileged and requires only public RPC access.

### Recommendation

Change `skip_stateful_validations` to verify that the account's mempool entry is specifically a **deploy-account transaction**, not just any transaction. Query the mempool for the transaction type at the account's address, or add a dedicated `has_deploy_account_in_pool(address)` API to the mempool client that only returns `true` when a deploy-account transaction (nonce=0) is present.

### Proof of Concept

```rust
// Pseudocode for the exploit:
// 1. Deploy contract at address A (nonce becomes 0 after deploy via non-deploy-account path,
//    or use a genesis-deployed account).
// 2. Submit invoke(nonce=2, valid_signature) for address A → admitted (skip_validate=false,
//    __validate__ runs and passes).
// 3. Submit invoke(nonce=1, INVALID_signature) for address A:
//    - account_nonce = 0 (on-chain), tx.nonce() = 1 → conditions met
//    - account_tx_in_pool_or_recent_block(A) = true (nonce=2 invoke is in pool)
//    - skip_stateful_validations returns true
//    - run_validate_entry_point called with skip_validate=true → __validate__ NOT called
//    - Transaction admitted to mempool with invalid signature
```

The test case `should_not_skip_validation_nonce_over_one` in the test suite confirms that nonce=2 invokes do NOT trigger the skip — but it does not test the scenario where a nonce=2 invoke is already in the mempool when a nonce=1 invoke arrives. [6](#0-5)
