### Title
Secret FROST Nonce Material Exposed in Serializable `PresignOutput` Without Zeroization - (File: src/frost/mod.rs)

### Summary
The FROST `PresignOutput<C>` struct bundles secret per-session nonces (`SigningNonces<C>`) together with public commitment data in a single `Serialize`/`Deserialize`-derived type, without `ZeroizeOnDrop`. Any caller who serializes this struct for persistence (the natural use case given the derives) exposes the secret nonces in plaintext. An attacker who reads the serialized output can recover the participant's long-term signing key share from the nonces and the resulting public signature share.

### Finding Description
`PresignOutput<C>` in `src/frost/mod.rs` is the library's public output type for the FROST presign phase. It is defined as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

`nonces: SigningNonces<C>` holds the secret hiding nonce `a_i` and binding nonce `b_i` generated during presigning. `commitments_map` holds the corresponding public points `A_j = a_j·G` and `B_j = b_j·G` for all participants. These two fields — one secret, one public — are co-located in the same serializable struct.

The struct derives `Serialize` and `Deserialize` but **not** `ZeroizeOnDrop`. This is inconsistent with both ECDSA presign output types in the same codebase, which both derive `ZeroizeOnDrop`:

- `src/ecdsa/ot_based_ecdsa/mod.rs` `PresignOutput`: `#[derive(..., ZeroizeOnDrop)]` [2](#0-1) 
- `src/ecdsa/robust_ecdsa/mod.rs` `PresignOutput`: `#[derive(..., ZeroizeOnDrop)]` [3](#0-2) 

The nonces are produced inside `do_presign` and returned directly in `PresignOutput`:

```rust
let (nonces, commitments) = commit(&signing_share, &mut rng);
...
Ok(PresignOutput { nonces, commitments_map })
``` [4](#0-3) 

The signing phase then consumes both fields:

```rust
let signing_package = SigningPackage::new(presignature.commitments_map, message);
let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)...;
``` [5](#0-4) 

Because the struct is `Serialize`/`Deserialize`, callers are naturally expected to persist it between the presign and sign phases. When they do, the secret nonces are written to disk (or transmitted) in plaintext alongside the public commitments.

### Impact Explanation
**Critical — Extraction of private signing shares.**

The FROST signing share equation (RFC 9591) is:

```
z_i = a_i + b_i * ρ_i + λ_i * x_i * c
```

where `a_i` and `b_i` are the secret nonces, `ρ_i` is the binding factor (public, derived from `commitments_map`), `c` is the challenge (public), `λ_i` is the Lagrange coefficient (public), and `z_i` is the signature share sent to the coordinator (observable on the network).

If an attacker reads the serialized `PresignOutput<C>` and obtains `a_i` and `b_i`, they can solve for the long-term signing key share:

```
x_i = (z_i - a_i - b_i * ρ_i) / (λ_i * c)
```

This directly yields the participant's secret signing share `x_i`, which is the output of DKG and the most sensitive long-term secret in the system. With `threshold` such shares, the full private key can be reconstructed.

### Likelihood Explanation
**Medium-High.** The `Serialize`/`Deserialize` derives on `PresignOutput<C>` are a strong signal that the library intends callers to persist this struct. Any MPC node that stores presign outputs to disk (e.g., for crash recovery, batching, or offline signing) will write the secret nonces in plaintext. An attacker with read access to the node's storage — or who can observe a memory dump, a log, or a serialized message — obtains the nonces. The subsequent signature share `z_i` is observable on the network. The key recovery computation is trivial algebra.

### Recommendation
1. **Remove `Serialize`/`Deserialize` from `PresignOutput<C>`**, or split the struct into a secret part (nonces only, never serialized) and a public part (commitments only, freely serializable). The nonces must never be written to persistent storage.
2. **Add `ZeroizeOnDrop`** to `PresignOutput<C>` to match the ECDSA presign outputs and ensure secret nonces are cleared from memory immediately after use.
3. **Consume the nonces by value** in the signing function so the compiler enforces single-use and the memory is released after signing.

### Proof of Concept
1. Caller runs `presign(...)` and receives `PresignOutput<C> { nonces, commitments_map }`.
2. Caller serializes the output: `serde_json::to_string(&presign_output)` — the JSON contains the plaintext secret nonces `a_i` and `b_i`.
3. Attacker reads the serialized file and extracts `a_i`, `b_i`.
4. Caller runs `sign(...)`, producing signature share `z_i` (observable by the attacker on the network or as coordinator output).
5. Attacker computes `ρ_i` from the public `commitments_map`, computes `c` from the public signing package, and solves: `x_i = (z_i - a_i - b_i * ρ_i) / (λ_i * c)`.
6. Attacker now holds participant `i`'s long-term signing key share. Repeating for `threshold` participants reconstructs the full private key. [1](#0-0) [6](#0-5)
