### Title
wNEAR Tokens Permanently Frozen in Engine's Implicit Address When XCC Router Deployment Fails - (File: `engine/src/xcc.rs`)

---

### Summary

When a user invokes the XCC precompile for the first time (requiring a new router sub-account), wNEAR tokens are transferred from the user's EVM balance to the engine's implicit address as part of the committed EVM state. If the subsequent asynchronous NEAR promise chain fails (e.g., router deployment batch fails), there is no refund callback to return the wNEAR to the user. The tokens are permanently frozen in the engine's implicit address.

---

### Finding Description

**Step 1 — wNEAR transfer is committed atomically with the EVM transaction.**

In `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the XCC precompile calls `transferFrom` on the wNEAR ERC-20 contract inside the EVM execution, moving `required_near` worth of wNEAR from the caller's EVM balance to the engine's implicit address: [1](#0-0) 

This EVM state change (the `transferFrom`) is committed as part of the `submit` call's state. It is **not** rolled back if subsequent NEAR promises fail.

**Step 2 — Promise chain is built without a user-refund error handler.**

After EVM execution, `handle_precompile_promise` in `engine/src/xcc.rs` constructs the following asynchronous NEAR promise chain when a new router must be created (`create_needed = true`):

1. **Batch**: `CreateAccount` + `Transfer(STORAGE_AMOUNT)` from engine's NEAR balance + `DeployContract` + `FunctionCall(initialize)`
2. **Callback**: `factory_update_address_version` — updates the engine's record of the router version
3. **Callback**: `withdraw_wnear_to_router` — converts the wNEAR in the engine's implicit address to NEAR and sends it to the router
4. **Callback**: `send_refund` — sends `REFUND_AMOUNT` (2 NEAR) back from the router to the engine
5. **Callback**: user's actual cross-contract call (`execute`/`schedule`) [2](#0-1) 

**Step 3 — Failure propagates silently, leaving wNEAR stranded.**

If the router deployment batch (step 1) fails for any reason (e.g., engine's NEAR balance is below `STORAGE_AMOUNT = 2_000_000_000_000_000_000_000_000` yoctoNEAR), `factory_update_address_version` panics because it detects a failed prior promise: [3](#0-2) 

`withdraw_wnear_to_router` is then called as a callback to `factory_update_address_version`. It checks `promise_result_check()` and returns an error when it sees the prior promise failed: [4](#0-3) 

`send_refund` on the router also panics because `require_no_failed_promises()` detects the chain failure: [5](#0-4) 

The wNEAR tokens transferred in Step 1 remain in the engine's implicit EVM address. **There is no error-handling callback anywhere in the chain that returns the wNEAR to the user.** Compare this to the `exit_to_near_precompile_callback`, which explicitly handles the failure case with `refund_on_error`: [6](#0-5) 

No equivalent refund path exists for the XCC precompile flow.

---

### Impact Explanation

The user's wNEAR tokens are frozen in the engine's implicit EVM address (`near_account_to_evm_address(engine_account_id.as_bytes())`). This address has no associated private key. The user has no on-chain mechanism to recover the tokens. Recovery requires the engine owner to manually execute an EVM call from the engine's implicit address to transfer the wNEAR back — an out-of-band admin action with no guarantee of occurrence.

**Impact: Critical — Permanent freezing of user funds.**

---

### Likelihood Explanation

The failure trigger is the router deployment batch failing. The most realistic causes are:

- The engine's NEAR balance falls below `STORAGE_AMOUNT` (2 NEAR). This can happen organically if many new routers are deployed in rapid succession before refunds complete, or if the engine's balance is otherwise low.
- The `initialize` call on the router fails (e.g., wNEAR account not registered or misconfigured).

Any unprivileged user who calls the XCC precompile for the first time while the engine's NEAR balance is insufficient will trigger this path. The `STORAGE_AMOUNT` constant is hardcoded at 2 NEAR: [7](#0-6) 

**Likelihood: Low-to-Medium** — the engine is typically well-funded, but the condition is reachable without any privileged access.

---

### Recommendation

Add an error-handling callback at the end of the XCC promise chain (analogous to `exit_to_near_precompile_callback`) that:

1. Checks if any prior promise in the chain failed.
2. If so, re-mints or transfers the wNEAR back to the original caller's EVM address using `refund_on_error` or an equivalent mechanism.

The `ExitToNearPrecompileCallbackArgs` pattern with a `refund` field already demonstrates the correct approach: [6](#0-5) 

A similar `XccPrecompileCallbackArgs` struct carrying the original sender address and wNEAR amount should be introduced, and a callback should be appended to the promise chain in `handle_precompile_promise`.

---

### Proof of Concept

1. User holds wNEAR in their EVM balance and calls the XCC precompile for the first time (no router deployed for their address).
2. XCC precompile executes `transferFrom(user, engine_implicit_address, required_near)` on the wNEAR ERC-20 — this is committed to EVM state.
3. Engine builds promise chain: `[deploy_router_batch]` → `[factory_update_address_version]` → `[withdraw_wnear_to_router]` → `[send_refund]` → `[user_call]`.
4. The engine's NEAR balance is below `STORAGE_AMOUNT` (2 NEAR); the deployment batch fails.
5. `factory_update_address_version` panics (`ERR_ROUTER_DEPLOY_FAILED`).
6. `withdraw_wnear_to_router` panics (`ERR_CALLBACK_OF_FAILED_PROMISE`).
7. `send_refund` panics (`ERR_CALLBACK_OF_FAILED_PROMISE`).
8. User's wNEAR tokens remain in the engine's implicit address. User's EVM wNEAR balance is zero. No refund is issued.

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
