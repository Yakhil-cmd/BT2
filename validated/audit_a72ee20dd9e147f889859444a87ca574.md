### Title
Incomplete L1 Deposit Validation Allows `TopLevelInsufficientBalance` to Halt Block Processing — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`process_l1_transaction` validates that `total_deposited >= gas_price * gas_limit` but never checks that `total_deposited >= gas_price * gas_limit + value`. When `to_transfer = total_deposited - max_fee_commitment` equals zero while `transaction.value > 0`, the sender is minted no balance yet the runner attempts a value transfer. The resulting `TopLevelInsufficientBalance` error is not classified as a `FatalRuntimeError` and therefore escapes the graceful-revert handler, propagating as a fatal block-level error that halts chain processing and permanently locks the deposited L1 funds.

---

### Finding Description

**Step 1 — Incomplete deposit check.**

`process_l1_transaction` performs a single guard:

```rust
let tx_internal_cost = gas_price
    .checked_mul(U256::from(gas_limit))
    .ok_or(internal_error!("gp*gl"))?;
let total_deposited = transaction.reserved[0].read();
require_internal!(
    total_deposited >= tx_internal_cost,
    "Deposited amount too low",
    system
)?;
```

This only verifies `total_deposited >= gas_price * gas_limit`. It does **not** verify `total_deposited >= gas_price * gas_limit + value`. The code comment acknowledges the intent to double-check L1 invariants, but the check is incomplete. [1](#0-0) 

**Step 2 — Zero-balance mint when `to_transfer == 0`.**

Inside `execute_l1_transaction_and_notify_result`, the amount minted to the sender is:

```rust
let to_transfer = total_deposited
    .checked_sub(max_fee_commitment)
    .ok_or(internal_error!("td-mfc"))?;
// ...
if to_transfer > U256::ZERO || Config::SIMULATION {
    // mint to sender
}
```

When `total_deposited == gas_price * gas_limit`, `to_transfer == 0` and the mint is skipped entirely. The sender's balance remains zero. [2](#0-1) 

**Step 3 — Value transfer attempted against zero balance.**

Immediately after, `run_single_interaction` is called with `&value` as `nominal_token_value`:

```rust
BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
    system, system_functions, memories.reborrow(),
    calldata, &from, &to,
    resources_for_tx,
    &value,   // ← positive value, sender has 0 balance
    false,
    tracer, validator,
)
```

The runner's `perform_transfer_if_required` detects the balance shortfall and, because the caller is `ExecutionEnvironmentType::NoEE` (top-level), returns:

```rust
Err(interface_error!(BootloaderInterfaceError::TopLevelInsufficientBalance))
``` [3](#0-2) [4](#0-3) 

**Step 4 — Error escapes the graceful-revert handler.**

Back in `process_l1_transaction`, the only errors converted to a graceful revert are `FatalRuntimeError` (out-of-native / OOM). `TopLevelInsufficientBalance` is a `LeafUsage` interface error, not a `RuntimeError::FatalRuntimeError`, so it falls into the catch-all branch:

```rust
_ => {
    system.finish_global_frame(Some(&rollback_handle))?;
    return Err(e);   // ← propagates, halts block processing
}
``` [5](#0-4) 

---

### Impact Explanation

An L1→L2 priority transaction with `value > 0` and `total_deposited == gas_price * gas_limit` causes `process_l1_transaction` to return a fatal error. Because L1 priority transactions cannot be skipped (skipping one would halt the chain by design), the block cannot be sealed. The deposited base tokens are locked in the L1 bridge with no refund path: the L2 transaction never commits, no `refund_recipient` mint is issued, and the priority queue is permanently blocked. This is the direct analog of the original report's "funds trapped indefinitely" — a deposit enters a state from which there is no exit.

---

### Likelihood Explanation

The L1 bridge contract is expected to enforce `total_deposited >= gas_price * gas_limit + value`. However:

1. The ZKsync OS code explicitly states it double-checks L1 invariants as a safety net against L1 contract bugs or version skew.
2. The double-check at line 133–137 is incomplete — it omits the `value` component.
3. Any future L1 contract upgrade, misconfiguration, or cross-chain bridge variant that does not enforce the full invariant would expose this path to any unprivileged user who submits an L1→L2 transaction with `value > 0` and `total_deposited == gas_price * gas_limit`.

---

### Recommendation

1. **Extend the deposit guard** in `process_l1_transaction` to include the transaction value:

```rust
require_internal!(
    total_deposited >= tx_internal_cost.saturating_add(value),
    "Deposited amount too low to cover fee and value",
    system
)?;
```

2. **Alternatively, treat `TopLevelInsufficientBalance` as a graceful revert** in the error handler (alongside `FatalRuntimeError`), so a malformed L1 transaction is recorded as failed rather than halting the block.

---

### Proof of Concept

```
gas_price      = 1_000
gas_limit      = 100_000
total_deposited = 100_000_000   (= gas_price * gas_limit, exactly the fee)
value           = 1             (any positive amount)

In process_l1_transaction:
  tx_internal_cost = 100_000_000  ✓  (passes the existing check)
  to_transfer      = 100_000_000 - 100_000_000 = 0

In execute_l1_transaction_and_notify_result:
  to_transfer == 0  →  mint to sender is skipped
  run_single_interaction called with nominal_token_value = 1
  sender balance = 0  →  TopLevelInsufficientBalance

Error root_cause ≠ FatalRuntimeError
  →  return Err(e)  →  block processing halts
  →  deposited funds locked in L1 bridge indefinitely
```

    let to_transfer = total_deposited

    // Transfer value from treasury to sender (the deposit minus max fee).
    // We want to ensure that the simulation of a transaction
    // never underestimates gas/pubdata compared to the actual execution
    // of said transaction.
    // During simulation the gas price is typically set to 0. So we need
    // to be conservative about operations that incur in gas/pubdata depending
    // on the value of the fee. For that reason, we always perform the
    // following transfer on simulation, and avoid compressing the pubdata
    // for the balance changes resulting from it.
    //
    // Mint the value portion of the deposit (total deposited minus max fee)
    // to the sender inside the execution frame, it does not
    // persist if the main tx body reverts.
    //
    // Use with_infinite_ergs so the call cannot fail due to out-of-gas,
    // but native consumption is still tracked against the user's resources.
    if to_transfer > U256::ZERO || Config::SIMULATION {
        resources
            .with_infinite_ergs(|inf_resources| {
                mint_base_token::<S, Config>(
                    system,
                    system_functions,
                    memories.reborrow(),
                    &to_transfer,
                    &from,
                    l1_chain_id,
                    inf_resources,
                    tracer,
                    validator,
                )
            })
            .map_err(|e| match e.root_cause() {
                RootCause::Runtime(RuntimeError::OutOfErgs(_)) => {
                    system_log!(
                        system,
                        "Out of ergs on infinite ergs: inner error was {e:?}"
                    );
                    BootloaderSubsystemError::LeafDefect(internal_error!(
                        "Out of ergs on infinite ergs"
                    ))
                }
                _ => e,
            })?;
