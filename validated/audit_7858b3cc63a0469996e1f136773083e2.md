### Title
`setPoolAdminFeeDestination` Redirects Already-Accrued Admin Fees to New Destination Without Prior Collection - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-parameter setter in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before mutating state, establishing a clear protocol invariant: *fees earned under old parameters must be settled before those parameters change*. `setPoolAdminFeeDestination` silently breaks that invariant, causing all previously accrued admin fees to be paid to the new destination when collection eventually occurs.

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the collect-then-update pattern:

```solidity
// setPoolAdminFees (lines 418-425) — correct
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination used before update
);
c.adminSpreadFeeE6 = newAdminSpreadFeeE6;
...
```

`setPoolAdminFeeDestination` does not:

```solidity
// setPoolAdminFeeDestination (lines 438-447) — missing collect
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

The pool holds two categories of accrued admin fees at any moment:

1. **Spread fees** — implicit surplus: `balance * scaleMultiplier − binTotals.scaledToken − notionalFeeScaled` (lines 385-388 of `MetricOmmPool.sol`). This surplus grows with every swap and is only realized at `collectFees` time.
2. **Notional fees** — explicit accumulators `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` (lines 382-383), zeroed only inside `collectFees` (lines 429-430).

Both are paid to whichever address `adminFeeDestination_` is at the moment `collectFees` executes. After `setPoolAdminFeeDestination` runs, the next `collectFees` call (whether triggered by `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee`) will send the entire accumulated admin share — including everything earned before the destination change — to the new address.

### Impact Explanation

The old admin fee destination permanently loses all admin fees that accrued before the destination was changed. These are real token balances already held by the pool contract. The loss is proportional to the volume of swaps since the last `collectFees` call and the configured admin fee rates. There is no on-chain mechanism for the old destination to recover these funds once the new destination has been paid.

### Likelihood Explanation

The pool admin is a semi-trusted role. The trigger is a single valid admin transaction (`setPoolAdminFeeDestination`) with no cap or timelock guard. The scenario arises naturally during:
- Routine treasury rotation (admin changes the fee destination to a new wallet/contract).
- Pool admin transfer: the outgoing admin changes the fee destination before handing over control, redirecting fees owed to the old treasury.
- Adversarial ordering: any party aware of a pending `setPoolAdminFeeDestination` transaction can back-run it with `collectPoolFees` (which is permissionless) to protect the old destination — but the old destination has no guarantee of winning that race.

### Recommendation

Apply the same collect-before-update pattern used by `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before rotating.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool accumulates spread and notional fees over N swaps; `poolAdminFeeDestination[pool] = oldTreasury`.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)`. No `collectFees` is triggered. `poolAdminFeeDestination[pool]` is now `newTreasury`.
3. Anyone calls `collectPoolFees(pool)`. Inside `collectFees`, `adminFeeDestination_` is `newTreasury` (line 387 of factory reads the already-updated mapping). All admin fees — including those earned before step 2 — are transferred to `newTreasury`.
4. `oldTreasury` receives zero tokens despite having been the designated recipient for the entire accrual period.

Contrast with the protected path: if `setPoolAdminFees` had been called instead, `collectFees` would have been invoked with `poolAdminFeeDestination[pool]` still pointing to `oldTreasury`, correctly settling all prior fees there before any state mutation.
