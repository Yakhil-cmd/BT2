### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Verification for Invoke Transactions with Nonce=1 from Undeployed Accounts - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point — the account contract's signature verification — for any invoke transaction with `nonce=1` when the account is undeployed (`account_nonce=0`) and the mempool reports any transaction for that address. An attacker who first submits a valid `deploy_account` transaction can then submit a second invoke transaction with a completely forged or invalid signature that is admitted to the mempool without any signature check.

---

### Finding Description

`skip_stateful_validations` at lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip) when all four conditions hold:

1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate = true` is returned, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }` and passes it to the blockifier's `StatefulValidator::validate`: [2](#0-1) 

Inside the blockifier, `AccountTransaction::validate_tx` immediately returns `Ok(None)` when `execution_flags.validate == false`, so the `__validate__` entry point — the only place where the account contract verifies the caller's signature — is never invoked: [3](#0-2) 

`StatefulValidator::perform_validations` also short-circuits before calling `__validate__` when the flag is false: [4](#0-3) 

The mempool-level check `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or any committed block entry — it does not verify that the pooled transaction is specifically a `deploy_account`: [5](#0-4) 

The stateless validator checks only signature **length**, not cryptographic validity: [6](#0-5) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` performs only nonce/pool-level checks and has no access to account contract code: [7](#0-6) 

Note also that `StatefulTransactionValidatorConfig` defines a `max_nonce_for_validation_skip` field (default `Nonce(Felt::ONE)`) intended to bound the skip window, but this field is **never read** inside `skip_stateful_validations`; the function hardcodes `Nonce(Felt::ONE)` directly. The field is only consumed by the legacy `PyValidator` path: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

The gateway is the admission gate for the mempool. Its stateful validation is the only point at which the account contract's `__validate__` entry point is executed before sequencing. By skipping it, the gateway admits invoke transactions whose signatures have never been cryptographically verified. The mempool therefore contains transactions that are invalid by the protocol's own account-abstraction rules. This matches the allowed High impact: **Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The attacker needs only to:
1. Generate a fresh key pair and compute the deterministic contract address for a `deploy_account` transaction (public information from the class hash and salt).
2. Submit a valid `deploy_account` (signed correctly) — this is the only privileged step.
3. Immediately submit an invoke with `nonce=1` carrying an arbitrary/forged signature.

No privileged access, no special network position, and no race condition is required. The four conditions are trivially satisfiable by any external user.

---

### Recommendation

Remove the unconditional signature-verification bypass. If the deploy-account + invoke UX improvement must be preserved, the gateway should still execute `__validate__` against the **expected** account class (obtained from the pending `deploy_account` transaction's `class_hash`) so that the signature is verified against the correct contract logic before admission. Alternatively, enforce that the `max_nonce_for_validation_skip` config field is actually read and applied inside `skip_stateful_validations`, and add a hard requirement that the pooled transaction for the account is specifically a `deploy_account` type, not any arbitrary transaction.

---

### Proof of Concept

```
1. Attacker generates key pair (K_valid, K_forged).
2. Computes address A = hash(class_hash, salt, constructor_calldata, deployer=0).
3. Submits deploy_account(A, class_hash, salt, sig=sign(K_valid, deploy_hash))
   → Gateway validates: account_nonce=0, tx_nonce=0, __validate_deploy__ runs → ACCEPTED.
4. Submits invoke(sender=A, nonce=1, calldata=..., sig=random_bytes)
   → Gateway stateful check:
       validate_nonce: account_nonce=0, tx_nonce=1, within gap → PASS
       validate_by_mempool: nonce gap OK → PASS
       skip_stateful_validations:
           tx is Invoke ✓
           tx.nonce() == 1 ✓
           account_nonce == 0 ✓
           account_tx_in_pool_or_recent_block(A) == true (deploy_account is in pool) ✓
       → returns skip=true
       run_validate_entry_point(skip_validate=true):
           execution_flags.validate = false
           StatefulValidator::validate_tx → returns Ok(None) without calling __validate__
   → ACCEPTED into mempool with forged signature, no signature verification performed.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
