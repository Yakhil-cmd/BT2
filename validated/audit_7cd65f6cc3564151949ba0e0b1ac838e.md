### Title
`meta_tx_v0` Syscall Permits Self-Call with Forged `tx_info`, Bypassing `__validate__` and Nonce for Inner Execution — (`crates/blockifier/src/execution/syscalls/syscall_base.rs`)

---

### Summary

The `meta_tx_v0` syscall implementation in the blockifier does not check that the caller-supplied `contract_address` differs from the currently-executing contract's own storage address. A contract can therefore invoke `meta_tx_v0` with itself as the target, causing its own `__execute__` entry point to be re-entered with a completely forged `tx_info` (version = 0, nonce = 0, caller_address = 0, attacker-controlled signature). Because V0 transactions skip `__validate__`, the inner call is never signature-validated by the account's own validation logic.

---

### Finding Description

`SyscallHandlerBase::meta_tx_v0` in `syscall_base.rs` performs two guards before executing:

1. Reject the syscall in `Validate` execution mode.
2. Reject any selector other than `__execute__`.

It then unconditionally builds a `CallEntryPoint` targeting the caller-supplied `contract_address` with `caller_address = ContractAddress::default()` (address 0), replaces the live `tx_context` with a synthetic `DeprecatedTransactionInfo` carrying `version = 0`, `nonce = 0`, and the attacker-supplied `signature`, and dispatches the call. [1](#0-0) 

There is **no guard** of the form:

```rust
if contract_address == self.call.storage_address {
    return Err(SyscallExecutionError::Revert { ... });
}
```

The same omission exists in the OS Cairo implementation: [2](#0-1) 

When `contract_address == self.call.storage_address`, the sequence is:

1. The outer transaction is valid and its `__execute__` is running.
2. Inside `__execute__`, the contract calls `meta_tx_v0(address = self, calldata = [...], signature = [...])`.
3. The blockifier builds a new `CallEntryPoint` targeting the same contract with `caller_address = 0`.
4. The `tx_context` is replaced: `version = 0`, `nonce = 0`, `sender_address = contract_address`, `signature = attacker-supplied`.
5. The contract's `__execute__` is invoked again. Because `caller_address == 0`, the standard "only the OS may call `__execute__`" assertion passes.
6. Because `version = 0`, the Starknet protocol specifies that `__validate__` is **skipped** for this inner call.
7. The inner `__execute__` sees `tx_info.nonce = 0` and `tx_info.signature = attacker-supplied`, regardless of the outer transaction's real nonce and signature. [3](#0-2) 

The `account_with_real_validate` test contract demonstrates the standard mitigation pattern — an explicit `assert(tx_info.version != 0, 'INVALID_TX_VERSION')` in `__execute__`: [4](#0-3) 

Contracts that omit this version check (a common pattern in custom account contracts and DeFi protocols that implement `__execute__`) are fully exposed.

---

### Impact Explanation

**Matching impact**: *Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

A contract whose `__execute__` uses `tx_info.nonce` for replay protection, `tx_info.version` for authorization gating, or any field of `tx_info` for accounting will produce **wrong storage state** when the self-referential `meta_tx_v0` call succeeds. Specifically:

- `tx_info.nonce` is forced to `0`, bypassing any nonce-monotonicity check inside `__execute__`.
- `tx_info.version` is forced to `0`, bypassing any version-gated logic.
- `tx_info.signature` is attacker-controlled, allowing forged authorization signals that the contract reads from `tx_info` rather than from `__validate__`.
- The inner call's state changes (storage writes, events, L2→L1 messages) are committed as part of the outer transaction's receipt, producing a receipt that does not correspond to any legitimately signed operation.

---

### Likelihood Explanation

Any deployed account contract or protocol contract that:
- Implements `__execute__` (required for account contracts),
- Checks `caller_address == 0` (standard guard), but
- Does **not** additionally assert `tx_info.version != 0`,

is reachable by any user who can submit a valid outer invoke transaction. The attacker controls the `calldata` and `signature` passed to `meta_tx_v0`, so the inner call's arguments are fully attacker-chosen. No privileged access is required beyond the ability to submit a normal transaction.

---

### Recommendation

Add a self-call guard in `SyscallHandlerBase::meta_tx_v0` immediately after the selector check:

```rust
if contract_address == self.call.storage_address {
    return Err(SyscallExecutionError::Revert {
        error_data: vec![INVALID_ARGUMENT_FELT],
    });
}
```

Apply the equivalent check in the OS Cairo implementation (`execute_meta_tx_v0` in `syscall_impls.cairo`):

```cairo
if (contract_address == caller_execution_info.contract_address) {
    write_failure_response(remaining_gas=remaining_gas, failure_felt=ERROR_INVALID_ARGUMENT);
    return ();
}
```

Additionally, document that all account contracts **must** assert `tx_info.version != 0` inside `__execute__` as a defence-in-depth measure against any future path that sets version to 0.

### Proof of Concept

```cairo
// Vulnerable account contract (no version check in __execute__)
#[starknet::contract(account)]
mod VulnerableAccount {
    #[storage]
    struct Storage { privileged_op_count: u64 }

    extern fn meta_tx_v0_syscall(
        address: ContractAddress,
        entry_point_selector: felt252,
        calldata: Span<felt252>,
        signature: Span<felt252>,
    ) -> starknet::SyscallResult<Span<felt252>> implicits(GasBuiltin, System) nopanic;

    #[external(v0)]
    fn __validate__(ref self: ContractState, calls: Array<Call>) -> felt252 {
        // Real ECDSA check here — attacker cannot pass this for arbitrary calldata
        starknet::VALIDATED
    }

    #[external(v0)]
    fn __execute__(ref self: ContractState, calls: Array<Call>) -> Array<Span<felt252>> {
        // Standard guard — passes because meta_tx_v0 sets caller_address = 0
        assert(starknet::get_caller_address().is_zero(), 'INVALID_CALLER');
        // NO version check — vulnerability exposed here
        let tx_info = starknet::get_tx_info().unbox();
        if tx_info.nonce == 0 {
            // Attacker forces nonce=0 via meta_tx_v0 self-call,
            // triggering privileged path without going through __validate__
            self.privileged_op_count.write(self.privileged_op_count.read() + 1);
        }
        // ... execute calls
        array![]
    }

    #[external(v0)]
    fn trigger_exploit(ref self: ContractState) {
        // Called from within a legitimate __execute__ invocation.
        // Passes self as the target — no guard prevents this.
        meta_tx_v0_syscall(
            starknet::get_contract_address(),   // address = self
            selector!("__execute__"),
            array![/* attacker calldata */].span(),
            array![/* forged signature */].span(),
        ).unwrap_syscall();
        // privileged_op_count is now incremented without a valid outer nonce=0 tx
    }
}
```

Step-by-step:

1. Attacker submits a valid invoke transaction to `VulnerableAccount.__execute__` (passes `__validate__`).
2. Inside `__execute__`, the attacker's calldata routes to `trigger_exploit`.
3. `trigger_exploit` calls `meta_tx_v0_syscall(address=self, ...)`.
4. The blockifier executes `SyscallHandlerBase::meta_tx_v0` — no self-call guard fires.
5. A new `CallEntryPoint` is dispatched to `VulnerableAccount.__execute__` with `caller_address=0`, `tx_info.version=0`, `tx_info.nonce=0`.
6. The inner `__execute__` passes the `caller_address==0` check and enters the `nonce==0` privileged branch.
7. `privileged_op_count` is incremented — wrong state committed to the receipt without a legitimately signed nonce-0 transaction. [5](#0-4) [6](#0-5)
