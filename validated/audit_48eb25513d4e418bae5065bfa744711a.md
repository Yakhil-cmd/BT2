### Title
Unused Gas Fee Refund Minted to Zero Address When `refund_recipient` Is Unset in L1ŌåÆL2 Transactions ŌĆö (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

For L1ŌåÆL2 priority transactions, the bootloader reads the refund recipient from `transaction.reserved[1]`. There is no validation that this field is non-zero. When `reserved[1]` is `address(0)` ŌĆö the default when a user or bridge does not explicitly set it ŌĆö any unused gas fee refund is minted to the zero address and permanently lost, rather than being returned to the transaction sender.

---

### Finding Description

In `process_l1_transaction`, after computing the unused gas refund amount (`to_refund_recipient`), the bootloader unconditionally reads the refund recipient from `transaction.reserved[1]`:

```rust
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
    mint_base_token::<S, Config>(
        system, system_functions, memories.reborrow(),
        &to_refund_recipient,
        &refund_recipient,   // ŌåÉ zero address if reserved[1] was not set
        ...
    )
}
``` [1](#0-0) 

The `validate_structure` function explicitly defers this check with a `TODO`:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
``` [2](#0-1) 

The `L1TxBuilder` test helper confirms that `refund_recipient` defaults to `Address::default()` (zero address) when not explicitly provided:

```rust
refund_recipient: self.refund_recipient.unwrap_or_default(),
``` [3](#0-2) 

This default-to-zero pattern mirrors real-world L1 bridge usage where callers omit the field. The `ZKsyncL1Tx` struct documents `refund_recipient` as the address that receives the refund, but provides no enforcement: [4](#0-3) 

---

### Impact Explanation

When `reserved[1] == address(0)` and `to_refund_recipient > 0`, the bootloader calls `mint_base_token` targeting the zero address. The refund tokens are minted to `0x0000...0000` and are permanently unrecoverable. The transaction sender ŌĆö who overpaid gas ŌĆö receives nothing back. This is a direct, irreversible loss of user funds proportional to `(gas_limit - gas_used) * gas_price`.

---

### Likelihood Explanation

Any L1ŌåÆL2 transaction submitted without an explicit `refund_recipient` (a common omission, as shown by the builder default) triggers this path. The `TODO: validate address?` comment in `validate_structure` confirms the gap is known but unresolved. Since L1ŌåÆL2 transactions always set a gas limit higher than actual usage (to avoid failure), `to_refund_recipient` is almost always `> 0`, making the loss near-certain whenever `reserved[1]` is zero.

---

### Recommendation

In `validate_structure`, reject L1ŌåÆL2 and upgrade transactions where `reserved[1]` decodes to `address(0)`:

```rust
Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
    if u256_to_b160_checked(self.reserved[1].read()) == B160::ZERO {
        return Err(());
    }
}
```

Alternatively, fall back to `transaction.from()` when `reserved[1]` is zero, mirroring the L2 transaction refund logic: [5](#0-4) 

---

### Proof of Concept

1. Submit an L1ŌåÆL2 transaction with `refund_recipient = address(0)` (the default when omitted), `gas_limit = 100_000`, `gas_price = 1000`, and a simple ETH transfer that uses ~21,000 gas.
2. The bootloader computes `to_refund_recipient = (100_000 - 21_000) * 1000 = 79_000_000` wei.
3. Since `to_refund_recipient > U256::ZERO`, it reads `refund_recipient = u256_to_b160_checked(reserved[1]) = address(0)`.
4. `mint_base_token` is called with `recipient = address(0)`.
5. 79,000,000 wei is minted to the zero address and permanently lost.
6. The transaction sender's balance is not restored for the unused gas.

The existing test `test_treasury_based_token_distribution_regression` already demonstrates this path ŌĆö it uses `refund_recipient = address(0)` and asserts the refund goes to the zero address, inadvertently confirming the bug rather than catching it: [6](#0-5) [7](#0-6)
