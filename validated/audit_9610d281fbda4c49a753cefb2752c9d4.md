Audit Report

## Title
Unbounded Loop with No Input Validation on `ingress_start`/`ingress_end` Causes OOM/Infinite-Loop DoS — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `construction_payloads` handler accepts attacker-controlled `ingress_start` and `ingress_end` values from the request metadata with no bounds validation, then iterates a `while` loop over the range in 120-second steps. With `ingress_end = u64::MAX`, the loop executes ~153 billion iterations before `Time::add`'s truncating `as u64` cast wraps `now` back to a small value, making the loop infinite. Even without wrapping, the loop allocates ~1.2 TB of heap before OOM-killing the process. A single unauthenticated HTTP POST is sufficient to trigger this.

## Finding Description

**The loop** at lines 99–107 of `construction_payloads.rs` has no pre-loop guard:

```rust
let mut ingress_expiries = vec![];
let mut now = ingress_start;
while now < ingress_end {
    ingress_expiries.push(...);
    now += interval;   // interval = 120_000_000_000 ns
}
``` [1](#0-0) 

`ingress_start` and `ingress_end` are taken directly from the client-supplied JSON metadata with no range checks: [2](#0-1) 

The metadata struct accepts raw `u64` nanosecond values with no validation: [3](#0-2) 

`interval` is `MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120_000_000_000 ns`: [4](#0-3) [5](#0-4) 

**Root cause — `Time::add` truncating cast:** `Time::from_duration` casts a `u128` nanosecond value to `u64` with a silent truncating `as` cast:

```rust
fn from_duration(t: Duration) -> Self {
    Time(t.as_nanos() as u64)   // truncating cast, wraps on overflow
}
``` [6](#0-5) [7](#0-6) 

When `now` reaches near `u64::MAX`, `Duration::from_nanos(u64::MAX) + Duration::from_secs(120)` produces a `Duration` whose `.as_nanos()` returns a `u128` value of `18_446_744_193_709_551_615`, which exceeds `u64::MAX`. The `as u64` cast truncates this to `119_999_999_999` (≈120 ns). Since `119_999_999_999 < u64::MAX`, the loop condition remains true and the loop never terminates.

**Contrast with ICRC1 Rosetta**, which has explicit guards before its equivalent loop: [8](#0-7) 

The ICP Rosetta handler has none of these guards.

**Two concrete attack paths:**
1. **OOM path**: With `ingress_end = u64::MAX`, the loop runs ~153 billion iterations before the first wrap, each pushing a `u64` (8 bytes) to `ingress_expiries`, allocating ~1.2 TB of heap → OOM kills the process.
2. **Infinite loop path**: After the first wrap, `now` resets to ~120s and the cycle repeats indefinitely, consuming 100% CPU on the handler thread.

## Impact Explanation

This is a complete availability loss for the ICP Rosetta service. A single crafted request permanently hangs or crashes the Rosetta node process, dropping all subsequent API requests. The ICP Rosetta API is explicitly in-scope under "Financial integrations: ICP/ICRC ledgers, Rosetta API." This matches the **High** bounty impact: *"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS."* The attack requires no credentials, no rate limiting bypass, and no special privileges — a single HTTP POST suffices.

## Likelihood Explanation

The `/construction/payloads` endpoint is a public, unauthenticated HTTP POST endpoint. The exploit requires setting two integer fields (`ingress_start`, `ingress_end`) to extreme values in the JSON metadata. No keys, credentials, or privileged access are needed. The attack is trivially reproducible locally and can be repeated indefinitely to prevent recovery.

## Recommendation

1. **Add input validation** before the loop, mirroring the ICRC1 Rosetta guards:
   - Reject if `ingress_start >= ingress_end`
   - Reject if `ingress_end - ingress_start > 86_400_000_000_000` ns (24 hours), yielding at most 720 iterations
2. **Use `checked_add`** (already available on `Time`) instead of the unchecked `+=` in the loop to detect and break on overflow.
3. **Cap `ingress_expiries`** to a maximum length (e.g., 1000 entries) as defense-in-depth.

## Proof of Concept

```
POST /construction/payloads HTTP/1.1
Content-Type: application/json

{
  "network_identifier": { "blockchain": "Internet Computer", "network": "<network_id>" },
  "operations": [<valid_transfer_op>],
  "public_keys": [<valid_pubkey>],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

The server enters the `while now < ingress_end` loop at line 101 of `construction_payloads.rs`. After ~153 billion iterations (OOM path) or after the first wrap via the truncating cast at line 104 of `time.rs` (infinite loop path), the process is rendered permanently unavailable. A minimal integration test can confirm this by asserting that the handler returns an error (rather than hanging) when `ingress_end - ingress_start` exceeds a reasonable bound.
