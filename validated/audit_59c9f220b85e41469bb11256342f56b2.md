### Title
Off-by-One Rounding Error in EVM 63/64 Gas Rule Gives Callee Excess Gas - (File: `evm_interpreter/src/gas.rs`)

---

### Summary

The `apply_63_64_rule` function in `evm_interpreter/src/gas.rs` contains a rounding error structurally identical to the external report's class: it computes `gas - floor(gas/64)` instead of the EVM-spec-correct `floor(gas * 63/64)`. These two expressions diverge by exactly 1 gas unit whenever `gas % 64 != 0`, which is the common case. The callee receives 1 extra gas unit and the caller retains 1 fewer gas unit than the Yellow Paper requires, producing an EVM semantic mismatch reachable by any unprivileged transaction sender.

---

### Finding Description

The function is:

```rust
// evm_interpreter/src/gas.rs  line 152-156
pub(crate) fn apply_63_64_rule(ergs: Ergs) -> Ergs {
    // We need to apply the rule over gas, not ergs
    let gas = ergs.0 / ERGS_PER_GAS;
    Ergs(ergs.0 - (gas / 64) * ERGS_PER_GAS)
}
``` [1](#0-0) 

It is called unconditionally for every external EVM call:

```rust

``` [2](#0-1) 

**Mathematical root cause (the rounding-order error):**

The EVM Yellow Paper (EIP-150) specifies the maximum gas passable to a callee as:

```
max_passable = floor(gas * 63 / 64)
```

For integer `gas`, this equals `gas - ceil(gas / 64)`.

The code computes:

```
max_passable_code = gas - floor(gas / 64)
```

When `gas % 64 != 0`:
- `ceil(gas / 64) = floor(gas / 64) + 1`
- Therefore `max_passable_code = max_passable_spec + 1`

**Concrete example — gas = 65:**

| | Callee receives | Caller retains |
|---|---|---|
| EVM spec | `floor(65 × 63/64) = 63` gas | 2 gas |
| ZKsync OS code | `65 − floor(65/64) = 64` gas | 1 gas |

The caller retains **half** the gas the spec guarantees it. Any post-call logic that requires 2 gas will OOG on ZKsync OS but succeed on Ethereum mainnet.

This is the same class as the external report: `f(a − b)` is used where `f(a) − f(b)` is required. Here `f(x) = x / 64` (floor division), `a = gas`, `b = gas * 63/64`.

---

### Impact Explanation

1. **EVM semantic mismatch / state-transition divergence**: Transactions that revert on Ethereum mainnet due to OOG in a subcall may succeed on ZKsync OS (callee gets 1 extra gas unit). Conversely, transactions that succeed on mainnet may OOG on ZKsync OS in the caller's post-call code (caller retains 1 fewer gas unit). Both directions break EVM equivalence.

2. **Caller OOG on valid transactions**: Any contract that passes the maximum allowed gas to a callee and then performs post-call work requiring exactly `ceil(gas/64)` gas will fail with OOG on ZKsync OS when it should succeed. This is the direct analog of the external report's "user prevented from locking assets even when all conditions are met."

3. **Scope**: The impact class is **EVM semantic mismatch** and **valid-execution unprovability** (a transaction provably valid under the Yellow Paper is rejected by ZKsync OS's gas accounting).

---

### Likelihood Explanation

`gas % 64 != 0` is true for 63 out of every 64 possible gas values. Every CALL, DELEGATECALL, STATICCALL, and CALLCODE opcode goes through `apply_63_64_rule`. Any unprivileged user submitting an EVM transaction that makes a subcall triggers this path. No special privileges, oracle manipulation, or governance access is required.

---

### Recommendation

Replace the approximate formula with the exact EVM-spec formula:

```rust
pub(crate) fn apply_63_64_rule(ergs: Ergs) -> Ergs {
    // EVM spec: callee gets floor(gas * 63 / 64) gas
    let gas = ergs.0 / ERGS_PER_GAS;
    let gas_to_pass = gas * 63 / 64; // integer floor(gas * 63/64)
    Ergs(gas_to_pass * ERGS_PER_GAS)
}
```

This matches the Yellow Paper exactly and eliminates the off-by-one for all `gas % 64 != 0` cases.

### Proof of Concept

Deploy the following contract on ZKsync OS and Ethereum mainnet and compare outcomes:

```solidity
contract Caller {
    // gas_limit set to 65 (65 % 64 = 1, so rounding error applies)
    function test() external {
        // Pass all available gas to callee
        (bool ok,) = address(callee).call{gas: gasleft()}("");
        // Post-call: needs 2 gas to complete (SSTORE costs 2 gas minimum)
        // On mainnet: caller retains ceil(65/64)=2 gas → succeeds
        // On ZKsync OS: caller retains floor(65/64)=1 gas → OOGs here
        assembly { sstore(0, 1) }
    }
}
```

On Ethereum mainnet with `gas = 65`:
- Callee gets `floor(65 × 63/64) = 63` gas; caller retains 2 gas → post-call SSTORE succeeds.

On ZKsync OS with `gas = 65`:
- `apply_63_64_rule` returns `65 − 1 = 64` gas to callee; caller retains 1 gas → post-call SSTORE OOGs.
