### Title
Gateway Skips Signature Verification for Invoke Transactions with Nonce=1 on Undeployed Accounts, Allowing Admission of Unsigned Transactions - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally bypasses the `__validate__` entry-point call (which performs signature verification) for any invoke transaction with `nonce == 1` targeting an account whose on-chain nonce is `0`, provided that *any* transaction for that account address exists in the mempool or a recent block. Because the check only verifies that the account address appears in the mempool — not that the submitter of the invoke transaction is the same party who submitted the deploy-account transaction — an unprivileged attacker can submit an invoke transaction with an arbitrary payload and a forged/empty signature for any victim account that has a pending `deploy_account` in the mempool, and the gateway will admit it without verifying the signature.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, which causes the blockifier's `StatefulValidator::perform_validations` to return `Ok(())` immediately for invoke transactions without calling `__validate__`: [2](#0-1) [3](#0-2) 

The condition for skipping is:
1. Transaction type is `Invoke`
2. Transaction nonce is exactly `1`
3. On-chain account nonce is `0` (account not yet deployed)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

Condition 4 only checks whether *any* transaction for the sender address exists in the mempool — it does not verify that the party submitting the invoke transaction is the same party who submitted the `deploy_account`. This is the broken invariant: the authorization check is on the account address's presence in the mempool, not on the cryptographic binding between the deploy-account submitter and the invoke submitter.

The `account_tx_in_pool_or_recent_block` check is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

And `contains_account` simply checks whether the address appears in the mempool state or pool: [5](#0-4) 

### Impact Explanation

An attacker who observes a `deploy_account` transaction for victim address `X` in the mempool can immediately submit an invoke transaction with `nonce=1`, `sender_address=X`, arbitrary calldata, and a forged/empty signature. The gateway will:

1. Confirm `tx.nonce() == 1` and `account_nonce == 0` ✓
2. Call `account_tx_in_pool_or_recent_block(X)` → `true` (victim's deploy_account is present) ✓
3. Set `skip_validate = true`, bypassing `__validate__` entirely
4. Admit the transaction to the mempool without any signature verification

The admitted transaction carries an invalid signature. When the batcher later executes it (with `validate: true` via `new_for_sequencing`), the `__validate__` entry point will be called and will reject the invalid signature, causing the transaction to revert. However, the transaction has already been admitted to the mempool, occupying the victim's `nonce=1` slot. If the mempool supports fee escalation (replacing a lower-fee tx with a higher-fee tx at the same nonce), the attacker can use a high-fee invalid transaction to displace the victim's legitimate invoke transaction, permanently disrupting the victim's deploy-account + invoke UX flow.

**Impact**: High — Mempool/gateway admission accepts invalid (unsigned) transactions before sequencing.

### Likelihood Explanation

The attack requires only:
- Monitoring the public mempool for `deploy_account` transactions (observable by any node)
- Submitting a crafted invoke transaction with nonce=1 for the target address

No privileged access is required. The window of opportunity exists from the moment the victim's `deploy_account` enters the mempool until it is committed to a block.

### Recommendation

The `skip_stateful_validations` check should be strengthened so that the invoke transaction's admission is cryptographically tied to the deploy-account transaction. Options include:

1. **Require the invoke transaction to reference the deploy-account transaction hash** (e.g., in a dedicated field), and verify that the referenced deploy-account transaction is in the mempool for the same sender address.
2. **Do not skip `__validate__` entirely**; instead, run a lightweight signature check against the class hash that will be deployed (if known from the deploy-account transaction in the mempool).
3. **Accept the UX trade-off** and remove the skip entirely, requiring users to wait for the deploy-account to be confirmed before submitting invoke transactions.

### Proof of Concept

1. Victim submits `deploy_account` for address `X` (class_hash `C`, salt `S`). The transaction enters the mempool.
2. Attacker observes the mempool and sees the deploy_account for `X`.
3. Attacker constructs an invoke transaction:
   - `sender_address = X`
   - `nonce = 1`
   - `calldata = [arbitrary malicious calldata]`
   - `signature = [0x0, 0x0]` (invalid/empty)
4. Attacker submits this invoke transaction to the gateway.
5. Gateway evaluates `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` (X not yet deployed) ✓
   - `account_tx_in_pool_or_recent_block(X)` → `true` (victim's deploy_account is present) ✓
   - Returns `true` → `validate = false`
6. Gateway calls `blockifier_validator.validate(account_tx)` with `validate=false`. The `__validate__` entry point is skipped. The transaction is admitted to the mempool.
7. The attacker's invalid transaction now occupies the `nonce=1` slot for address `X` in the mempool. If the attacker used a higher fee than the victim's legitimate invoke tx, the victim's tx is displaced and will not be executed. [1](#0-0) [6](#0-5) [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
