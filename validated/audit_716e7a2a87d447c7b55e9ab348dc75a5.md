### Title
Unbounded Bin Additional Fees Allow Pool Admin to Bypass Protocol-Enforced Spread Fee Caps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory.sol` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no cap validation, while the parallel `setPoolAdminFees` path enforces `maxAdminSpreadFeeE6`. A pool admin can therefore charge traders additional spread fees on every bin beyond what the protocol cap permits.

### Finding Description

The factory enforces a hard cap on the admin's base spread fee component via `maxAdminSpreadFeeE6` (itself bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%). [1](#0-0) 

`setPoolAdminFees` checks this cap before writing: [2](#0-1) 

But `setPoolBinAdditionalFees` performs **no analogous check** — it passes the caller-supplied `uint16` values straight through to the pool: [3](#0-2) 

The pool's `setBinAdditionalFees` likewise applies no cap — it only validates the bin index: [4](#0-3) 

These per-bin fees are added directly on top of the oracle-derived base spread fee in every swap path: [5](#0-4) 

and in the quoted price surface exposed to traders: [6](#0-5) 

`uint16` allows values up to 65 535, which in E6 notation is ≈ 6.55 %. This is a separate fee layer that is **additive** to the pool's `spreadFeeE6` (protocol + admin) and to the oracle spread, and it is never counted against `maxAdminSpreadFeeE6`.

### Impact Explanation

A pool admin who is constrained to, say, `maxAdminSpreadFeeE6 = 10 000` (1 %) can simultaneously set `addFeeBuyE6 = addFeeSellE6 = 65 535` (≈ 6.55 %) on every bin. Every swap in those bins pays ≈ 6.55 % extra on top of the capped pool spread and the oracle spread. The excess is collected as LP fee and flows to the pool's LP shares — effectively extracting value from traders beyond the protocol-sanctioned limit. This is a direct loss of trader principal and constitutes an admin-boundary break: the pool admin exceeds the fee cap the factory owner set.

### Likelihood Explanation

`setPoolBinAdditionalFees` requires only `onlyPoolAdmin(pool)` and has no timelock. Any pool admin can call it immediately after pool creation or at any time thereafter. The pool admin role is semi-trusted only within the caps the protocol enforces; this path bypasses those caps entirely.

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool) that ensures `addFeeBuyE6` and `addFeeSellE6` do not exceed a factory-configured maximum (e.g. `maxAdminSpreadFeeE6` expressed in E6, or a dedicated `maxBinAdditionalFeeE6` constant). For example:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 10_000` (1 %) via `setFeeCaps`.
2. Pool admin calls `setPoolAdminFees(pool, 10_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**.
4. A trader swaps through bin 0. The effective fee applied is:

   ```
   currBinBuyFeeX64 = baseFeeX64 + mulDiv(65_535, ONE_X64, 1e6)
                    ≈ oracleSpread + 6.55%
   ```

   on top of the 1 % pool spread fee, totalling ≈ 7.55 % + oracle spread — far above the 1 % admin cap the factory owner intended.

5. The excess fee accrues as LP fee inside the bin, draining trader input beyond the protocol-sanctioned limit with no on-chain guard to prevent it. [7](#0-6) [8](#0-7)
