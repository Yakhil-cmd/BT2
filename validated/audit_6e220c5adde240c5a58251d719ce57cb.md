### Title
`setPoolAdminFeeDestination` Does Not Collect Accrued Fees Before Updating Destination, Causing Misdirected Admin Fee Loss - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. All spread-surplus and notional fees earned before the destination change are subsequently sent to the new address, permanently depriving the old destination of fees it was owed.

### Finding Description

`MetricOmmPool.collectFees` distributes two categories of admin fees using the `adminFeeDestination_` address supplied at call time:

1. **Spread fees** — the token surplus above `binTotals` (LP-fee residual): [1](#0-0) 

2. **Notional fees** — per-swap accumulators `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`: [2](#0-1) 

Both are transferred to `adminFeeDestination_` and the accumulators are zeroed: [3](#0-2) 

The factory's `collectPoolFees` passes `poolAdminFeeDestination[pool]` as that address: [4](#0-3) 

`setPoolAdminFeeDestination` simply overwrites the stored destination with no prior flush: [5](#0-4) 

This is structurally inconsistent with every other fee-configuration mutator. Both `setPoolAdminFees` and `setPoolProtocolFee` call `collectFees` with the **current** destination before writing any new value: [6](#0-5) [7](#0-6) 

The omission in `setPoolAdminFeeDestination` means that the next `collectPoolFees` call (or the implicit collection inside `setPoolAdminFees`) will send **all previously accrued fees** — including those earned entirely under the old destination — to the new address.

### Impact Explanation
Direct loss of owed admin fees. The old fee destination (e.g., a DAO treasury or prior operator) permanently loses all spread-surplus and notional fees that accrued before the destination change. The magnitude equals the full uncollected admin share at the time of the call, which can be arbitrarily large depending on trading volume and collection frequency.

### Likelihood Explanation
The trigger is a routine, valid pool-admin action: rotating a treasury address. No malicious setup is required. The bug fires on every destination change where `collectPoolFees` was not manually called immediately beforehand. Because `collectPoolFees` is permissionless and asynchronous, there is no on-chain enforcement that it precedes a destination update.

### Recommendation
Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before rotating.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old address
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool accumulates 1 000 token0 in spread surplus and 500 token0 in `notionalFeeToken0Scaled` for the admin share, all earned while `poolAdminFeeDestination[pool] == oldTreasury`.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)`. The function writes `poolAdminFeeDestination[pool] = newTreasury` with no prior `collectFees` call.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` → `newTreasury` and passes it to `collectFees`.
4. `collectFees` transfers the full 1 500 token0 admin share to `newTreasury`.
5. `oldTreasury` receives nothing despite having earned those fees; the loss is permanent.
