### Title
Unchecked Precompile Call Return Value After Irreversible Token Burn Leads to Permanent Fund Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

The `withdrawToNear` and `withdrawToEthereum` functions in `EvmErc20.sol` (and `EvmErc20V2.sol`) permanently burn the caller's ERC-20 tokens **before** invoking the exit precompile, and **never check the return value** of that precompile call. If the precompile call fails for any reason — including an invalid or oversized `recipient` input, insufficient gas forwarded, or any internal precompile error — the tokens are irreversibly destroyed with no recovery path. This is the direct Aurora analog of the Winnables `propagateRaffleWinner` bug: state is changed irreversibly before the cross-chain routing step is validated, and the routing step's failure is silently ignored.

---

### Finding Description

In `EvmErc20.sol`, the `withdrawToNear` function executes as follows:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // (1) tokens destroyed — irreversible

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // (2) res is captured but NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum`: [2](#0-1) 

And is replicated verbatim in `EvmErc20V2.sol`: [3](#0-2) 

The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` (`ExitToNear`) enforces a strict input size window and validates the recipient account ID: [4](#0-3) [5](#0-4) 

The precompile enforces `MAX_INPUT_SIZE = 1_024` bytes: [6](#0-5) 

If the caller passes a `recipient` whose encoded length causes the total input to exceed `MAX_INPUT_SIZE`, or passes bytes that are not valid UTF-8, or passes a syntactically invalid NEAR account ID, the precompile returns `ExitError`. In the EVM, this causes the inner `call` to return `res = 0`. Because the Solidity code never checks `res` and never reverts, the `_burn` that already executed is **not rolled back**. The tokens are gone.

The `error_refund` feature in the precompile is a compile-time flag that only handles NEAR-side promise failures (i.e., the `ft_transfer` on NEAR failing after the precompile itself succeeded). It does **not** protect against the EVM-side precompile call returning failure: [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

When the precompile call fails silently:
- The user's ERC-20 tokens are burned and cannot be re-minted (no admin path exists to recover them).
- No NEAR-side `ft_transfer` promise is ever created, so no tokens arrive on NEAR.
- The user has permanently lost their funds with no recourse.

This matches the "Permanent freezing of funds" impact class.

---

### Likelihood Explanation

**Medium-High.** Any unprivileged EVM user can trigger this by:

1. Calling `withdrawToNear` with a `recipient` byte array whose length, when combined with the flag byte and 32-byte amount, exceeds `MAX_INPUT_SIZE` (1,024 bytes). The Solidity function accepts `bytes memory recipient` with no length bound.
2. Calling `withdrawToNear` with a `recipient` that contains non-UTF-8 bytes.
3. Calling `withdrawToNear` with a `recipient` that is syntactically invalid as a NEAR account ID (e.g., contains uppercase letters, is empty, or exceeds 64 characters).
4. Forwarding insufficient gas to the precompile call (the assembly passes `gas()` which is the remaining gas, but if the caller's transaction has very little gas left at that point, the inner call can fail).

Cases 1–3 are trivially reachable by any user who makes a mistake or deliberately crafts bad input. Case 1 is particularly easy: a `recipient` of 992+ bytes passes Solidity's ABI layer without issue but will be rejected by the precompile's size check.

---

### Recommendation

Check the return value of the precompile call and revert if it fails, **before** burning tokens — or burn tokens only after confirming the precompile accepted the input:

**Option A** — Check `res` and revert on failure:
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }
    }
}
```

**Option B** — Validate input before burning (preferred, avoids gas waste on revert):
Add a Solidity-level check that `recipient.length <= 991` (to stay within `MAX_INPUT_SIZE` after prepending the flag and amount), and that `recipient` is valid UTF-8 and a valid NEAR account ID format, before calling `_burn`.

---

### Proof of Concept

1. User holds 1,000 bridged ERC-20 tokens on Aurora.
2. User calls `withdrawToNear(recipient, 1000e18)` where `recipient` is a 1,000-byte byte array (e.g., `bytes(new uint8[](1000))`).
3. `_burn(msg.sender, 1000e18)` executes — tokens are destroyed.
4. The precompile is called with total input size `1 + 32 + 1000 = 1033 > 1024 = MAX_INPUT_SIZE`.
5. The precompile returns `ExitError::Other("ERR_INVALID_INPUT")` — the inner `call` returns `res = 0`.
6. The Solidity function does not check `res`, does not revert, and returns normally.
7. The user's 1,000 tokens are permanently lost. No NEAR-side transfer occurs. [1](#0-0) [6](#0-5) [4](#0-3)
