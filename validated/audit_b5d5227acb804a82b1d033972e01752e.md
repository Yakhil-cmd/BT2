### Title
Hardcoded Mainnet-Only `CUSTODIAN_ADDRESS` Leaves `return_output` Security Guard Bypassed on Non-Mainnet Deployments - (File: `engine-sdk/src/near_runtime.rs`)

---

### Summary

The `return_output` function in `engine-sdk/src/near_runtime.rs` contains a security guard that prevents the engine from returning output bytes that structurally match a forged `WithdrawResult` (the GHSA-3p69-m8gg-fwmf exploit fix). However, the custodian address used in this guard is hardcoded to the Ethereum mainnet `eth_custodian` address `0x6BFaD42cFC4EfC96f529D786D643Ff4A8B89FA52`. The code comment explicitly acknowledges this limitation. On any non-mainnet Aurora deployment (testnet, or any future deployment with a different custodian address), the guard does not match the actual custodian address in use, leaving the original exploit vector fully open.

---

### Finding Description

In `engine-sdk/src/near_runtime.rs`, a compile-time constant holds the mainnet custodian address: [1](#0-0) 

This constant is the sole input to the security assertion inside `return_output`: [2](#0-1) 

The check asserts that no output value of length ≥ 56 bytes has the custodian address occupying bytes `[36..56]`. This byte range corresponds exactly to the Borsh-serialized layout of `WithdrawResult`: [3](#0-2) 

Borsh encodes this as: 16 bytes (`amount: u128`) + 20 bytes (`recipient_id: Address`) + 20 bytes (`eth_custodian_address: Address`) = 56 bytes total, with the custodian address at `[36..56]`.

The test `engine-tests/src/tests/ghsa_3p69_m8gg_fwmf.rs` documents the exact exploit: a malicious EVM contract echoes back a crafted 56-byte payload containing the custodian address at the correct offset, and the `view` call returns it as if it were a legitimate `WithdrawResult`: [4](#0-3) 

The test hardcodes `"6bfad42cfc4efc96f529d786d643ff4a8b89fa52"` — the mainnet address — and confirms the guard fires. On testnet or any other deployment where the actual `eth_custodian` address differs, an attacker substitutes the deployment-specific custodian address in the payload. The guard compares only against the mainnet constant and does **not** fire, so the forged `WithdrawResult` is returned without error.

The code comment itself acknowledges this design flaw explicitly: [5](#0-4) 

---

### Impact Explanation

The bypassed guard re-enables the GHSA-3p69-m8gg-fwmf exploit on non-mainnet deployments. An attacker can cause the Aurora engine's `view` entrypoint to return bytes that Borsh-decode as a valid `WithdrawResult` carrying an arbitrary `amount` and `recipient_id` alongside the real deployment's custodian address. Any off-chain relayer or on-chain consumer that trusts the `view` return value as a genuine engine withdrawal result can be tricked into initiating or crediting an unauthorized withdrawal of bridged ETH from the eth-connector, constituting **direct theft of user funds**.

**Impact: Critical — direct theft of bridged user funds.**

---

### Likelihood Explanation

Aurora testnet is a live, publicly accessible deployment. The custodian address for testnet differs from the mainnet constant. Any unprivileged EVM user can deploy a contract and invoke `view` with a crafted payload. No special privileges, key compromise, or governance action is required. The exploit technique is already publicly documented via the GHSA advisory and the in-repo test.

---

### Recommendation

Replace the compile-time constant with a runtime lookup of the actual custodian address stored in the engine's connector state (via `get_connector_account_id` or equivalent storage read), or extend the guard to cover all configured custodian addresses. At minimum, the guard must use the custodian address that is actually active for the current deployment rather than a hardcoded mainnet value.

---

### Proof of Concept

1. On Aurora testnet, deploy a Solidity contract that echoes its `bytes` argument (identical to `echo.sol` used in the existing test).
2. Determine the testnet `eth_custodian` address (call it `TESTNET_CUSTODIAN`).
3. Construct the 56-byte payload:
   - Bytes `[0..16]`: any `u128` amount in little-endian (e.g., `1_000_000u128`)
   - Bytes `[16..36]`: any 20-byte Ethereum recipient address
   - Bytes `[36..56]`: `TESTNET_CUSTODIAN` (the real testnet custodian address, **not** the mainnet one)
4. Call `view` on the Aurora testnet engine, targeting the echo contract with this payload.
5. Observe that **no** `ERR_ILLEGAL_RETURN` panic is raised and the forged `WithdrawResult` bytes are returned successfully — because `CUSTODIAN_ADDRESS` in `return_output` only matches the mainnet address `0x6BFaD42cFC4EfC96f529D786D643Ff4A8B89FA52`, not the testnet custodian.
6. The returned bytes Borsh-decode as a `WithdrawResult` with the attacker-chosen `amount` and `recipient_id`, enabling downstream abuse by any relayer or contract that consumes the `view` result as a trusted withdrawal proof.
