### Title
Signature Validation Bypass via `skip_stateful_validations` — Invoke Transactions with Invalid Signatures Admitted to Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (which is the only place the account's signature is verified) for any `Invoke` transaction whose nonce equals 1 and whose sender address appears in the mempool or a recent committed block. The membership check (`account_tx_in_pool_or_recent_block`) does **not** verify that the pooled transaction is a `DeployAccount`; it returns `true` for any transaction type. An unprivileged attacker can therefore submit an `Invoke` with nonce=1 and an **invalid signature** from any address that has a pending `DeployAccount` in the mempool, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

**Broken invariant**: The gateway must only admit transactions whose `__validate__` entry point would succeed. For `Invoke` transactions, `__validate__` is the sole mechanism that verifies the cryptographic signature.

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when:
1. The transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`

The code comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

**Root cause — `account_tx_in_pool_or_recent_block` does not filter by transaction type** [2](#0-1) 

The function checks `state.contains_account || tx_pool.contains_account` — both are address-keyed maps that hold **any** transaction type. There is no assertion that the pooled transaction is a `DeployAccount`.

**Root cause — `run_validate_entry_point` with `skip_validate=true` omits `__validate__`** [3](#0-2) 

When `skip_validate=true`, `execution_flags.validate` is set to `false`, and the blockifier's `StatefulValidator::perform_validations` returns immediately after `perform_pre_validation_stage` without ever calling `validate_tx` (the `__validate__` entry point). [4](#0-3) 

**No other guard checks the signature**: `validate_by_mempool` only checks nonce ordering and duplicate hashes; the stateless validator only checks signature *length*, not validity. [5](#0-4) 

### Impact Explanation

An attacker can submit an `Invoke` transaction carrying an **invalid signature** from any address that has a `DeployAccount` pending in the mempool. The gateway admits the transaction without verifying the signature. The transaction is then forwarded to the mempool and eventually handed to the batcher. During actual block execution the blockifier **does** call `__validate__`, so the transaction will be rejected at that stage (nonce not consumed), but the gateway has already accepted and propagated an invalid transaction.

Impact category matched: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack is fully unprivileged and requires only two sequential RPC calls:
1. Submit (or observe) a `DeployAccount` for target address X.
2. Submit `Invoke(sender=X, nonce=1, signature=<garbage>)`.

No special knowledge, privileged access, or on-chain funds are required beyond the standard fee deposit for the `DeployAccount`. The `skip_stateful_validations` path is exercised in the normal UX flow (deploy + invoke in one batch), so the code path is always active in production.

### Recommendation

Replace the address-only membership check with a check that specifically verifies a **`DeployAccount` transaction** for the sender is present in the pool. One approach:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use a new query:
mempool_client.deploy_account_tx_in_pool(tx.sender_address())
```

The mempool should expose a method that inspects the transaction type stored for the address, returning `true` only when the pooled transaction is a `DeployAccount`. Alternatively, the gateway can re-run a lightweight signature check (e.g., ECDSA verification against the sender's public key derived from the pending `DeployAccount`'s constructor calldata) before skipping the full `__validate__` call.

  "sender_address": "<X>",

}
# Gateway flow:
#   stateless_tx_validator.validate()  → passes (signature length OK)
#   convert_rpc_tx_to_internal()       → passes
#   extract_state_nonce_and_run_validations():
#     get_nonce_from_state(X)          → Nonce(0)
#     validate_nonce()                 → passes (0 ≤ 1 ≤ 200)
#     validate_by_mempool()            → passes (no dup, nonce ordering OK)
#     skip_stateful_validations():
#       tx.nonce()==1 && account_nonce==0 → true

The invalid `Invoke` is now in the mempool. When the batcher later executes it, `__validate__` is called and the transaction is rejected, but the gateway has already accepted and propagated an unauthenticated transaction, violating the admission invariant.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
