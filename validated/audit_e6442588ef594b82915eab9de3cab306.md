### Title
`RewardSupplier.claim_rewards()` assumes actual token balance covers `unclaimed_rewards`, causing temporary freeze of staker yield when L1 funds are in transit — (`src/reward_supplier/reward_supplier.cairo`)

---

### Summary

`RewardSupplier.claim_rewards()` validates only the accounting invariant (`amount <= unclaimed_rewards`) before attempting a token transfer, but never checks whether the contract's actual STRK balance is sufficient. Because `unclaimed_rewards` is credited immediately when rewards are registered while the corresponding L1 mint funds may still be in transit, the actual balance can be less than `unclaimed_rewards`. Any staker who calls `staking.claim_rewards()` during this window will receive a revert, temporarily freezing their unclaimed yield.

---

### Finding Description

**Step 1 — Rewards are registered, L1 funds requested**

When a staker attests, `_update_rewards` calls `update_unclaimed_rewards_from_staking_contract(rewards)`:

```cairo
// src/reward_supplier/reward_supplier.cairo:198-201
let unclaimed_rewards = self.unclaimed_rewards.read() + rewards;
self.unclaimed_rewards.write(unclaimed_rewards);
// Request funds from L1 if needed.
self.request_funds(:unclaimed_rewards);
```

`request_funds` computes `credit = balance + l1_pending_requested_amount`. If `credit < unclaimed_rewards + threshold`, it sends L1 mint messages and increases `l1_pending_requested_amount`:

```cairo
// src/reward_supplier/reward_supplier.cairo:311-330
let credit = balance + l1_pending_requested_amount;
let debit = unclaimed_rewards;
if credit < debit + threshold {
    // ... send L1 mint requests ...
    l1_pending_requested_amount += total_amount;
}
self.l1_pending_requested_amount.write(l1_pending_requested_amount);
```

After this call, `unclaimed_rewards` has grown but the actual token balance has **not** — the gap is covered only by `l1_pending_requested_amount`, which represents funds not yet received from L1.

**Step 2 — Staker claims rewards before L1 funds arrive**

`staking.claim_rewards` → `send_rewards_to_staker` → `claim_from_reward_supplier`:

```cairo
// src/staking/utils.cairo:50-60
pub(crate) fn claim_from_reward_supplier(...) {
    let balance_before = token_dispatcher.balance_of(account: staking_contract);
    reward_supplier_dispatcher.claim_rewards(:amount);
    let balance_after = token_dispatcher.balance_of(account: staking_contract);
    assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
}
```

Inside `reward_supplier.claim_rewards`:

```cairo
// src/reward_supplier/reward_supplier.cairo:213-219
let unclaimed_rewards = self.unclaimed_rewards.read();
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH); // ← passes
self.unclaimed_rewards.write(unclaimed_rewards - amount);
let token_dispatcher = self.token_dispatcher.read();
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into()); // ← REVERTS
```

The accounting check at line 214 passes because `amount <= unclaimed_rewards`. However, the actual STRK balance of the reward supplier is less than `amount` (the difference is still pending from L1). The `checked_transfer` reverts, rolling back the entire transaction including the `unclaimed_rewards` decrement.

**Concrete example**:
- Reward supplier STRK balance: 500
- `unclaimed_rewards`: 1000 (500 is pending from L1)
- Staker calls `claim_rewards` for 800
- `800 <= 1000` passes, but `checked_transfer(800)` fails → revert
- Staker's yield is frozen until L1 funds arrive

The same revert path also blocks `_update_rewards` at attestation time, since it immediately calls `claim_from_reward_supplier` for `total_pools_rewards` right after updating `unclaimed_rewards`:

```cairo
// src/staking/staking.cairo:2351-2360
reward_supplier_dispatcher
    .update_unclaimed_rewards_from_staking_contract(
        rewards: staker_rewards + total_pools_rewards,
    );
// Claim pools rewards.
claim_from_reward_supplier(
    :reward_supplier_dispatcher,
    amount: total_pools_rewards,
    ...
);
```

This means attestation itself can revert when pool rewards are non-zero and the balance is insufficient.

---

### Impact Explanation

Stakers and pool members cannot claim their accrued STRK yield during the window between reward registration and L1 fund arrival. Every `staking.claim_rewards()` call reverts. Additionally, attestation transactions that include pool reward distribution also revert, blocking the entire reward update cycle. This matches **Temporary freezing of unclaimed yield / funds** (High impact).

---

### Likelihood Explanation

The L1→L2 bridge delay is structural and unavoidable. Every epoch in which the reward supplier's on-hand balance is below `unclaimed_rewards` — which is the normal operating state when the protocol is growing — creates this window. Any staker or reward address can trigger the revert simply by calling `claim_rewards` at the wrong moment. No special privileges or attack setup are required.

---

### Recommendation

Before executing the transfer in `claim_rewards`, verify the actual token balance is sufficient:

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    ...
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    // Add: check actual balance
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())
        .try_into()
        .expect_with_err(...);
    assert!(amount <= balance, "{}", GenericError::INSUFFICIENT_BALANCE);
    ...
}
```

Alternatively, `claim_rewards` should return the minimum of `amount` and the actual balance, and the staking contract should handle partial claims gracefully, deferring the remainder until L1 funds arrive.

---

### Proof of Concept

1. Deploy the system with a reward supplier whose STRK balance is 500 and `unclaimed_rewards` is 0.
2. Advance one epoch and have a staker attest. `_update_rewards` calls `update_unclaimed_rewards_from_staking_contract(800)`, setting `unclaimed_rewards = 800`. `request_funds` fires an L1 mint request for the shortfall; `l1_pending_requested_amount` increases but the actual balance remains 500.
3. Before `on_receive` is called (L1 funds not yet bridged), the staker calls `staking.claim_rewards(staker_address)`.
4. `send_rewards_to_staker` → `claim_from_reward_supplier(800)` → `reward_supplier.claim_rewards(800)`.
5. `assert!(800 <= 800)` passes. `checked_transfer(800)` fails because balance is 500. Transaction reverts.
6. Staker's `unclaimed_rewards_own` remains non-zero; yield is frozen until `on_receive` is processed.

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
