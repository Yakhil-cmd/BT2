### Title
Unchecked Exit-Precompile Return Value Burns ERC-20 Tokens Without Transferring Them to NEAR — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's tokens via `_burn`, then invoking the Aurora exit precompile through an inline assembly `call`. The assembly block captures the call's success flag in a local variable `res`, but `res` is never inspected. If the precompile call fails for any reason — including an input-format mismatch between the contract and the engine version — the function returns normally while the user's tokens have already been permanently destroyed.

---

### Finding Description

In `EvmErc20.sol` (and identically in `EvmErc20V2.sol`), `withdrawToNear` is implemented as:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert on failure
    }
}
```

The same pattern appears in `withdrawToEthereum`.

The exit precompile (`ExitToNear`) in `engine-precompiles/src/native.rs` documents two distinct input formats depending on whether the `error_refund` feature is compiled in:

```
// ERC-20 transfer input format:
//  - flag (1 byte)
//  - refund_address (20 bytes), present if the feature "error_refund" is enabled.
//  - amount (32 bytes)
//  - recipient_account_id (variable)
```

`EvmErc20.sol` (V1) encodes: `flag | amount_b | recipient` — the **old** format without a refund address.  
`EvmErc20V2.sol` encodes: `flag | sender | amount_b | recipient` — the **new** format with a refund address.

When the engine is compiled with `error_refund` enabled, V1 contracts send the wrong format: the precompile reads the first 20 bytes of `amount_b` as the refund address and the remaining bytes as the amount, producing a completely wrong transfer or an outright failure. Conversely, when the feature is **not** enabled, V2 contracts send the wrong format: the precompile reads `sender` (20 bytes) as the amount and `amount_b` as the recipient, again producing garbage or failure.

In either mismatch scenario the precompile call returns `0` (failure), but because `res` is never tested and no `require(res != 0)` guard exists, the Solidity function returns successfully. The `_burn` that preceded the call is irreversible.

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

Any user who calls `withdrawToNear` or `withdrawToEthereum` on a contract whose input format does not match the currently deployed engine version will have their ERC-20 tokens burned with no corresponding NEP-141 credit on NEAR and no refund. The tokens are gone forever. The same outcome occurs for any other precompile failure (e.g., an invalid NEAR account-ID supplied as `recipient`).

---

### Likelihood Explanation

**High.** The two contract versions (`EvmErc20` / `EvmErc20V2`) exist precisely because the precompile input format changed. Old V1 contracts remain deployed on Aurora mainnet alongside the updated engine. Any user of a V1 contract calling `withdrawToNear` after the engine was updated to expect the `error_refund` format will silently lose funds. Additionally, any user who passes a malformed `recipient` string (e.g., an account ID that exceeds NEAR's length limit) triggers the same silent-burn path regardless of contract version.

---

### Recommendation

Add a `require` check on the assembly `call` result in both functions, and in both contract versions:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile rejects the call for any reason, the entire transaction reverts before `_burn` takes effect — or, since `_burn` is called first, the ordering should be reversed so the precompile is called (and checked) before tokens are destroyed.

---

### Proof of Concept

1. Deploy `EvmErc20` (V1) on Aurora with the engine compiled with `error_refund` enabled.
2. Mint tokens to `user`.
3. `user` calls `withdrawToNear("user.near", 1e18)`.
4. `_burn(user, 1e18)` executes — balance drops to 0.
5. The assembly `call` to the precompile sends `\x01 | amount_b | recipient`; the precompile (expecting `\x01 | refund_addr | amount_b | recipient`) misparses the input and returns `0`.
6. `res == 0` is never checked; the function returns normally.
7. `user` has lost `1e18` tokens with no NEAR credit and no revert. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
