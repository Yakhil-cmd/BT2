### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Contributions Without Restriction — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator accumulates participant contributions without any cryptographic restriction or validation. A malicious participant can send arbitrary `(norm_big_y, norm_big_c)` values, causing the coordinator to compute an incorrect CKD output and derive an incorrect confidential key, which honest parties will accept as valid.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs` lines 44–57), the coordinator receives a `CKDOutput` from every other participant and blindly sums the two group elements:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

The correct per-participant computation is defined in `compute_signature_share` (lines 148–182):

```
norm_big_y  =  λ_i · y_i · G
norm_big_c  =  λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

where `x_i` is the participant's private share and `y_i` is a fresh random nonce. There is **no zero-knowledge proof, no commitment-then-reveal, and no other cryptographic mechanism** that restricts a participant to sending values that are consistent with their actual private share. The protocol imposes no lock on what a participant may contribute.

A malicious participant runs a modified client that, instead of executing `compute_signature_share`, sends arbitrary group elements `(norm_big_y′, norm_big_c′)` to the coordinator. Because `recv_from_others` only filters by participant identity (the sender must be in the participant list) and not by value correctness, the coordinator accepts and accumulates the forged contribution without any error.

The root cause is structurally identical to the external report: a restriction that *should* exist (participants must contribute correctly computed values) is simply not enforced in the code.

---

### Impact Explanation

The final CKD output satisfies:

```
confidential_key = big_c − big_y · app_sk
                 = msk · H(pk ‖ app_id)   (when all contributions are honest)
```

When a malicious participant injects an arbitrary `norm_big_c′`, the coordinator's `big_c` is shifted by an attacker-controlled offset, so the derived confidential key is **incorrect**. Honest parties (coordinator and TEE application) accept this corrupted output as the legitimate result, because there is no post-hoc verification step.

This matches the allowed impact:
> **High: Corruption of … CKD outputs so honest parties accept inconsistent … unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant who is legitimately enrolled in the CKD session can execute this attack. No cryptographic break, no leaked key, and no external dependency is required. The attacker only needs to replace the `compute_signature_share` call with a send of arbitrary group elements. The attack is trivially reachable by any malicious library caller who controls one participant's execution environment.

---

### Recommendation

Add a cryptographic restriction on participant contributions before the coordinator accumulates them. Two standard approaches:

1. **Commit-then-reveal for the nonce `y_i`**: each participant first broadcasts a hash commitment to `y_i`; after all commitments are collected, they reveal `y_i` and prove consistency of `norm_big_c` with the committed nonce and the public share `x_i · G2`.
2. **Zero-knowledge proof of correct encryption**: each participant attaches a Sigma-protocol proof (or a Pedersen-style proof) showing that `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` for the same `y_i` used in `norm_big_y = λ_i · y_i · G1`, without revealing `x_i` or `y_i`.
