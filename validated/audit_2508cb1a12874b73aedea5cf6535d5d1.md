### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

The factory enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on the global admin fee components, but `setPoolBinAdditionalFees` passes per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) directly to the pool with no cap validation. Because these per-bin fees are added on top of the base spread fee during every swap, the pool admin can impose effective swap fees that exceed the documented hard cap on any individual bin.

### Finding Description

`setPoolAdminFees` correctly enforces the cap: [1](#0-0) 

But `setPoolBinAdditionalFees` passes values straight through with no cap check: [2](#0-1) 

And `setBinAdditionalFees` on the pool also performs no cap validation: [3](#0-2) 

During every swap, the per-bin additional fee is added directly on top of the base spread fee: [4](#0-3) [5](#0-4) 

The `addFeeBuyE6` and `addFeeSellE6` fields are `uint16`, so the pool admin can set them to any value up to 65,535 (≈ 6.55% in E6 units) per bin, with no relationship to `maxAdminSpreadFeeE6`: [6](#0-5) 

The hard cap `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) is the ceiling the owner may set for `maxAdminSpreadFeeE6`: [7](#0-6) 

A pool admin who has already set `adminSpreadFeeE6 = maxAdminSpreadFeeE6` can additionally set `addFeeBuyE6 = 65535` on every bin, making the effective per-bin fee `spreadFeeE6 + 65535/1e6` — exceeding the cap the factory is supposed to enforce.

### Impact Explanation

Traders swapping through bins with elevated additional fees pay more than the protocol-enforced maximum. The excess fee accrues to the pool's LP balance (not split as protocol/admin fee), so it is a direct, unrecoverable loss of user principal on every swap routed through the affected bin. The pool admin can apply this to all bins simultaneously, making the pool effectively unusable at fair prices.

### Likelihood Explanation

The pool admin is semi-trusted and can call `setPoolBinAdditionalFees` at any time with no timelock. The call requires only `onlyPoolAdmin(pool)` access, which is a single address that can act unilaterally. No governance delay or owner approval is required.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees`) that bounds `addFeeBuyE6` and `addFeeSellE6` against `maxAdminSpreadFeeE6` (or a dedicated per-bin cap). For example:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept

```solidity
// Pool admin sets global admin spread fee at the cap (200_000 = 20%)
vm.prank(admin);
factory.setPoolAdminFees(pool, 200_000, 0);

// Pool admin then sets bin additional fee to uint16 max (65_535 ≈ 6.55%) — no revert
vm.prank(admin);
factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535);

// A trader swapping through bin 0 now pays ~26.55% effective spread fee,
// exceeding the 20% hard cap the factory is supposed to enforce.
