### Title
Staker Can Atomically Increase Commission to 100% After Delegators Are Committed, Stealing Their Unclaimed Yield - (File: `src/staking/staking.cairo`)

---

### Summary

A staker can call `set_commission_commitment` followed immediately by `set_commission` in consecutive transactions to raise their commission from near-zero to 100% with no enforced delay. Delegators who have already committed their stake cannot exit immediately — they must wait through the `exit_wait_window` (1 week). During that window, all pool rewards flow to the staker as commission, stealing delegators' unclaimed yield.

---

### Finding Description

The `set_commission_commitment` function allows a staker to register a `CommissionCommitment` that permits raising commission up to `max_commission`. Once the commitment is active, `set_commission` enforces only `commission <= commitment.max_commission` and `commission != old_commission` — there is no minimum delay between setting the commitment and using it to raise commission.

The code itself acknowledges this gap:

> **Note**: Current commission increase safeguards still allow for sudden commission changes. [1](#0-0) 

The `set_commission_commitment` function sets the commitment immediately in storage with no timelock: [2](#0-1) 

The `update_commission` internal function, when a commitment is active, only checks `commission <= commitment.max_commission` — it does not require any elapsed time since the commitment was set: [3](#0-2) 

This means a staker can atomically:
1. Call `set_commission_commitment(max_commission=10000, expiration_epoch=current_epoch+1)`
2. Call `set_commission(10000)` in the very next transaction (or same block)

Commission is now 100%. Delegators receive zero rewards from that point forward.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Delegators who have already committed their stake cannot exit immediately. They must call `undelegate_intent` and then wait for the `exit_wait_window` (1 week) before calling `undelegate_action`. During that entire window, every reward epoch distributes 100% of pool rewards to the staker as commission, and 0% to delegators. All yield accrued during the exit window is permanently redirected to the staker. [4](#0-3) 

---

### Likelihood Explanation

**Medium-High.** Any staker who has opened a delegation pool can execute this attack in two sequential transactions with no special access beyond their own staker key. The attack requires no external conditions, no oracle manipulation, and no governance approval. The only prerequisite is that delegators have already committed funds — which is the normal operating state of any active pool.

---

### Recommendation

Enforce a minimum delay between when a `CommissionCommitment` is set and when it can be used to increase commission. For example, require that `set_commission` with an upward change can only be called at least `N` epochs after the commitment was registered. This gives delegators time to observe the pending increase and exit before it takes effect — analogous to requiring `maxRewardPerVote` to only increase (never decrease) in the external report's recommendation.

Alternatively, mirror the pattern already used for `numberOfPeriods`/`totalRewardAmount` in the external report: require that commission changes via commitment are only effective starting from the next epoch, not the current one.

---

### Proof of Concept

1. Staker stakes with `commission = 100` (1%) and opens a delegation pool via `set_open_for_delegation`.
2. Delegators delegate large amounts, attracted by the low commission. Their funds are locked; exiting requires calling `undelegate_intent` and waiting `exit_wait_window` (~1 week).
3. Staker calls `set_commission_commitment(max_commission=10000, expiration_epoch=current_epoch+1)`. The commitment is written to storage immediately with no delay. [5](#0-4) 

4. In the next transaction (same block is possible), staker calls `set_commission(10000)`. The `update_commission` check passes because `10000 <= commitment.max_commission` and `10000 != 100`. [6](#0-5) 

5. Commission is now 100%. Every subsequent `update_rewards` call computes `commission_rewards = pool_rewards_including_commission` and `pool_rewards = 0`. [7](#0-6) 

6. Delegators call `undelegate_intent` but must wait the full `exit_wait_window`. All yield earned during that window is captured by the staker. After the window, delegators exit with their principal intact but having lost all unclaimed yield from the period.

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );
        }
