### Title
Unconditional EIP-2537/EIP-152 Initialization Blocked by Feature-Gated Sanity Check in `add_precompile` - (`system_hooks/src/lib.rs`, `basic_bootloader/src/bootloader/block_flow/ethereum/post_init_op.rs`)

---

### Summary

`post_init_op` unconditionally calls `initialize_eip_2537` and `initialize_eip_152` during bootloader startup. Both functions internally call `add_precompile`, which enforces a sanity check against `PRECOMPILE_ADDRESSES_LOWS`. That list is compile-time feature-gated: BLS12-381 addresses are only present when the `eip-2537` feature is active, and the Blake2F address is only present when `eip-152` or `mock-unsupported-precompiles` is active. `basic_bootloader/Cargo.toml` declares neither `eip-2537` nor `eip-152` as its own features, and depends on `evm_interpreter` with `default-features = false`. If those features are absent from the unified feature set at build time, the sanity check returns a hard `InternalError`, aborting bootloader initialization before any transaction is processed.

---

### Finding Description

**`post_init_op` (unconditional calls):**

```rust
// basic_bootloader/src/bootloader/block_flow/ethereum/post_init_op.rs
fn post_init_op<Config: BasicBootloaderExecutionConfig>(
    _system: &mut System<S>,
    system_functions: &mut HooksStorage<S, <S as SystemTypes>::Allocator>,
) -> Result<(), InternalError> {
    add_precompiles(system_functions)?;
    hooks::eip_2537::initialize_eip_2537(system_functions)?;  // no #[cfg]
    hooks::eip_152::initialize_eip_152(system_functions)?;    // no #[cfg]
    Ok(())
}
``` [1](#0-0) 

Both calls are unconditional — no `#[cfg(feature = "eip-2537")]` or `#[cfg(feature = "eip-152")]` guard.

```rust

        "Attempted to add a precompile that is not in the precompile addresses list"

**`PRECOMPILE_ADDRESSES_LOWS` is feature-gated:**

```rust
// evm_interpreter/src/precompile_addresses.rs
pub const PRECOMPILE_ADDRESSES_LOWS: &[u16] = &[
    ...
    #[cfg(feature = "eip-2537")]
    BLS12_G1ADD_ADDRESS_LOW,   // 0x0b
    ...
    #[cfg(any(feature = "eip-152", feature = "mock-unsupported-precompiles"))]
    BLAKE2F_HOOK_ADDRESS_LOW,  // 0x09
    ...
];
``` [3](#0-2) 

**`basic_bootloader` does not declare these features:**

```toml
# basic_bootloader/Cargo.toml
evm_interpreter = { path = "../evm_interpreter", default-features = false }
# No eip-2537, no eip-152 in [features]
``` [4](#0-3) 

**Execution path to failure:**

1. Bootloader starts → `post_init_op` is called.
2. `initialize_eip_2537` calls `add_precompile(..., BLS12_G1ADD /* 0x0b */)`.
3. `add_precompile` checks `PRECOMPILE_ADDRESSES_LOWS.contains(&0x0b)`.
4. Without `eip-2537` feature, `0x0b` is absent → `Err(internal_error!(...))`.
5. Error propagates through `initialize_eip_2537` → `post_init_op` → bootloader init → entire block execution aborts. [5](#0-4) 

The same failure path applies to `initialize_eip_152` / `BLAKE_HOOK_ADDRESS_LOW = 0x0009`. [6](#0-5) 

---

### Impact Explanation

If `basic_bootloader` is compiled in a configuration where neither `eip-2537` nor `eip-152` is propagated to `evm_interpreter` (e.g., a stripped deployment, a custom integration, or a future build where EIP-2537 support is removed), the bootloader cannot complete `post_init_op`. No block can be executed. All transactions are rejected before validation. This is a complete denial-of-service at the state-transition level — equivalent to the Notional analog where the constructor check blocked all new vault deployments.

---

### Likelihood Explanation

`basic_bootloader/Cargo.toml` explicitly sets `evm_interpreter = { ..., default-features = false }` and does not declare `eip-2537` or `eip-152` as its own features. The features are only present at runtime if some other crate higher in the dependency graph re-enables them. This is an implicit, undeclared dependency. Any future refactor, new integration target, or stripped build that omits those upstream feature activations will silently break bootloader initialization. The pattern is directly analogous to the deprecated Balancer Oracle check: a guard that was valid at one point becomes a deployment blocker when the underlying feature is absent.

---

### Recommendation

Mirror the Notional fix: remove the unnecessary `PRECOMPILE_ADDRESSES_LOWS` guard from `add_precompile`, or feature-gate the calls in `post_init_op`:

```rust
// Option A: remove the guard in add_precompile (system_hooks/src/lib.rs)
// Delete lines 278-283 (the PRECOMPILE_ADDRESSES_LOWS.contains check)

// Option B: feature-gate in post_init_op
#[cfg(feature = "eip-2537")]
hooks::eip_2537::initialize_eip_2537(system_functions)?;
#[cfg(any(feature = "eip-152", feature = "mock-unsupported-precompiles"))]
hooks::eip_152::initialize_eip_152(system_functions)?;
```

Option B also requires `basic_bootloader/Cargo.toml` to declare `eip-2537` and `eip-152` as explicit features that propagate to `evm_interpreter` and `system_hooks`.

---

### Proof of Concept

Build `basic_bootloader` in isolation with `default-features = false` and without any upstream crate enabling `eip-2537` for `evm_interpreter`. Invoke any code path that triggers `post_init_op` (e.g., `run_forward`). The call chain:

```
post_init_op
  → initialize_eip_2537
    → add_precompile(hooks, BLS12_G1ADD /* 0x0b */)
      → PRECOMPILE_ADDRESSES_LOWS.contains(&0x0b) == false
        → Err(internal_error!("Attempted to add a precompile that is not in the precompile addresses list"))
```

The `InternalError` propagates up and aborts block execution before any transaction is processed. [7](#0-6) [8](#0-7)

pub fn add_precompile<S: EthereumLikeTypes, A: Allocator + Clone, P, E>(
    hooks: &mut HooksStorage<S, A>,
    address_low: u16,
) -> Result<(), InternalError>
where
    S::IO: IOSubsystemExt,
    P: SystemFunction<S::Resources, E>,
    E: Subsystem,
{
    // Sanity check to ensure that the address being added is indeed in the precompile addresses list
    if !PRECOMPILE_ADDRESSES_LOWS.contains(&address_low) {
        return Err(internal_error!(
            "Attempted to add a precompile that is not in the precompile addresses list"
        ));
    }
    hooks.add_call_hook(
        address_low,
        SystemCallHook::new(
            pure_system_function_hook_impl::<SystemFunctionInvocationUser<S, E, P>, E, S>,
        ),
    )
}
```
