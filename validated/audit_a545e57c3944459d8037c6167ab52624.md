### Title
Missing Version Check in Blockchain Notification Parsing Enables Counter-Offer Destination Spoofing - (File: packages/gui/src/hooks/useBlockchainNotifications.tsx)

### Summary
`useBlockchainNotifications.tsx` parses on-chain notification messages with a raw `JSON.parse` that skips the version field (`v`) check present in the shared `parseNotificationPayload` utility. The unvalidated `ph` (puzzleHash) field is then trusted directly as the counter-offer destination address, allowing any unprivileged sender to redirect a victim's counter-offer notification to an attacker-controlled address.

### Finding Description
The project defines a canonical notification parser in `packages/gui/src/components/notification/utils.ts`:

```ts
export function parseNotificationPayload(payload: string) {
  const { v, t: notificationType, d } = JSON.parse(payload);
  if (v !== NOTIFICATION_MESSAGE_VERSION) {   // ← version guard
    return null;
  }
  return { type: notificationType, data: d };
}
``` [1](#0-0) 

`useBlockchainNotifications.tsx` does **not** use this utility. Instead it performs its own `JSON.parse` and immediately destructures `t` and `d` without ever checking `v`:

```ts
const { t: type, d: data } = JSON.parse(message);   // no version check
// ...
if (type === 1) {
  const { u: url, ph: puzzleHash } = data;           // ph used without validation
``` [2](#0-1) 

The raw `puzzleHash` value is stored directly on the `Notification` object and later consumed by `OfferIncomingTable.handleCounterOffer`:

```ts
const address = currencyCode && puzzleHash
  ? toBech32m(puzzleHash, currencyCode.toLowerCase())
  : '';
// ...
navigate('/dashboard/offers/builder', { state: { address, offer, isCounterOffer: true } });
``` [3](#0-2) 

`address` is then used as the destination when the victim sends their counter-offer notification via `NotificationSendDialog`:

```ts
const targetPuzzleHash = fromBech32m(address);
// ...
await sendNotification({ target: targetPuzzleHash, ... });
``` [4](#0-3) 

### Impact Explanation
An attacker who sends a blockchain notification containing `"ph": "<attacker_puzzle_hash_hex>"` causes the victim's counter-offer notification — including the victim's offer URL and terms — to be delivered to the attacker's wallet instead of the legitimate offer creator's wallet. The attacker can then inspect the victim's counter-offer terms and act on the original offer before the victim's counter-offer is processed, constituting spoofing of notification state that causes the user to send to the wrong destination.

This satisfies the **High** criterion: *"Corruption, spoofing, or unsafe trust of… notification… state that causes a user to… send… the wrong… destination."*

### Likelihood Explanation
Sending a blockchain notification to any wallet address is an unprivileged, permissionless on-chain action (it requires only a small XCH fee). The attacker needs no access to the victim's keys or machine. Any user who receives such a notification and clicks "Counter" is affected.

### Recommendation
Replace the inline `JSON.parse` in `useBlockchainNotifications.tsx` with the existing `parseNotificationPayload` utility, which already enforces the version check:

```ts
// Before (no version check):
const { t: type, d: data } = JSON.parse(message);

// After (use canonical parser):
const parsed = parseNotificationPayload(message);
if (!parsed) throw new Error('Invalid or unsupported notification version');
const { type, data } = parsed;
```

Additionally, validate `ph` as a 64-character lowercase hex string before storing it as `puzzleHash`, rejecting notifications where `ph` does not conform to the expected puzzle-hash format.

### Proof of Concept
1. Attacker calls `send_notification` RPC targeting the victim's puzzle hash with the hex-encoded payload:
   ```json
   {"v":1,"t":1,"d":{"u":"https://dexie.space/offers/legit-offer","ph":"<attacker_32_byte_hex>"}}
   ```
   (The `v` field is present but is never checked by `useBlockchainNotifications.tsx`; omitting it entirely also works.)
2. Victim's GUI decodes the hex, calls `JSON.parse`, and stores `puzzleHash = <attacker_32_byte_hex>` on the `Notification` object without any format or version validation. [5](#0-4) 
3. Victim sees the offer notification and clicks **Counter**.
4. `handleCounterOffer` converts the attacker's puzzle hash to a bech32m address and navigates to the offer builder with `address` pre-set to the attacker's address. [6](#0-5) 
5. Victim creates and submits the counter-offer; `sendNotification` delivers it to the attacker's wallet. [7](#0-6) 
6. Attacker receives the victim's counter-offer URL and terms, and can act on the original offer before the victim's counter-offer reaches the legitimate party.
