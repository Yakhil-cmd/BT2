### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Over-Minting and Bridge Insolvency - (`engine/src/engine.rs`)

### Summary
The Aurora Engine's NEP-141→ERC-20 bridge unconditionally trusts the `amount` field supplied in the `ft_on_transfer` callback to determine how many ERC-20 tokens to mint. If the bridged NEP-141 token deducts a transfer fee before crediting Aurora's account, Aurora mints more ERC-20 tokens than the NEP-141 tokens it actually holds, making the bridge insolvent.

### Finding Description
When a user bridges a NEP-141 token into Aurora, they call `ft_transfer_call` on the NEP-141 contract. The NEP-141 contract then calls `ft_on_transfer` on Aurora, passing the `amount` that was requested for transfer. Aurora's handler reads this value directly:

In `engine/src/contract_methods/connector.rs`, `ft_on_transfer` reads `args.amount` from the JSON input and dispatches to `receive_erc20_tokens`: [1](#0-0) 

In `engine/src/engine.rs`, `receive_erc20_tokens` uses `args.amount.as_u128()` verbatim as the mint quantity: [2](#0-1) 

The ERC-20 mint call is then constructed with this unverified amount: [3](#0-2) 

There is no balance-before/balance-after check. If the NEP-141 token charges a fee on transfer (e.g., 1%), Aurora's actual NEP-141 balance increases by `amount * 0.99`, but it mints `amount` ERC-20 tokens — a 1% over-issuance per deposit.

The same pattern exists for base-token deposits in `receive_base_tokens`: [4](#0-3) 

The `FtOnTransferArgs.amount` field is defined as a plain `Balance` with no verification against actual on-chain balance change: [5](#0-4) 

### Impact Explanation
**Critical — Insolvency.**

The total ERC-20 supply on Aurora for a fee-bearing NEP-141 token will exceed the NEP-141 tokens Aurora actually holds. Every deposit inflates the ERC-20 supply by the fee amount. Users who withdraw later will find Aurora cannot fulfill their redemption because its NEP-141 reserve is insufficient. This is a direct, permanent insolvency of the bridge for that token, and it allows early depositors to drain NEP-141 tokens that were deposited by other users (theft of funds at rest).

### Likelihood Explanation
**Medium.** Any NEP-141 token with a configurable or built-in transfer fee can trigger this. The `deploy_erc20_token` entry point is permissionless — any account can register a NEP-141 token on Aurora. A malicious or fee-bearing token can be registered and exploited by an unprivileged user with no special access. The NEP-141 standard does not prohibit fee-on-transfer tokens. [6](#0-5) 

### Recommendation
Replace the direct use of `args.amount` with a balance-difference check: record Aurora's NEP-141 balance before the transfer completes and compute the actual credited amount. Because `ft_on_transfer` is a callback (the transfer has already occurred when it is called), the actual credited amount equals `current_nep141_balance_of(aurora) - balance_before`. Mint ERC-20 tokens equal to this verified delta, not the caller-supplied `args.amount`.

Alternatively, maintain an explicit whitelist of NEP-141 tokens that are known to be standard-compliant (no fees, no rebasing) and reject `ft_on_transfer` calls from tokens not on the whitelist.

### Proof of Concept
1. Deploy a NEP-141 token `fee_token.near` that charges a 10% fee on every transfer (i.e., receiver gets 90% of the requested amount).
2. Call `deploy_erc20_token` on Aurora to register `fee_token.near` → ERC-20 mirror.
3. Call `ft_transfer_call` on `fee_token.near` with `amount = 1000`, `receiver_id = aurora`, `msg = <attacker_evm_address>`.
4. `fee_token.near` credits Aurora with 900 tokens, then calls `ft_on_transfer(sender, amount=1000, msg=...)` on Aurora.
5. Aurora executes `receive_erc20_tokens` → mints **1000** ERC-20 tokens to the attacker's EVM address.
6. Attacker calls `ExitToNear` precompile to burn 1000 ERC-20 tokens and withdraw 1000 NEP-141 tokens.
7. Aurora's `ft_transfer` call to `fee_token.near` sends 1000 tokens, but Aurora only holds 900 — the call either fails (freezing other users' funds) or, if other users' deposits cover the shortfall, drains 100 tokens from the collective reserve.
8. Repeating this drains the entire NEP-141 reserve held by Aurora, leaving all other depositors unable to withdraw. [7](#0-6) [8](#0-7)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
