### Title
Unbounded NFT Royalty Percentage from Attacker-Controlled Driver Dict Spoofs WalletConnect Offer Confirmation Amount - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`parseRoyaltyPercentage` in `parseCommandDisplay.ts` accepts any positive integer from the offer's driver dict without an upper bound or cross-validation against the NFT's on-chain royalty state. For `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands, this value is used to compute `amountWithRoyalties` displayed in the signing confirmation dialog. A malicious dApp or offer creator can embed an arbitrarily small royalty percentage in the driver dict, causing the confirmation dialog to display a lower total cost than the user will actually pay (since the daemon enforces the correct on-chain royalty at execution time). The user approves based on the understated amount and is charged more than they consented to.

### Finding Description

`parseRoyaltyPercentage` only checks that the value is a positive integer â€” there is no upper bound and no cross-validation with the NFT's on-chain royalty:

```typescript
function parseRoyaltyPercentage(value: unknown): number | undefined {
  if (typeof value === 'number' && Number.isInteger(value) && value > 0) {
    return value;
  }
  if (typeof value === 'string' && /^\d+$/.test(value)) {
    return Number(value);
  }
  return undefined;
}

### Impact Explanation

For `chia_wallet.take_offer`: A malicious dApp sends an offer whose embedded driver dict sets `royalty_percentage` to `1` (0.01%) instead of the NFT's actual on-chain royalty (e.g., 250 = 2.5%). The confirmation dialog shows `amountWithRoyalties: 1.0001 XCH` instead of the correct `1.025 XCH`. The user approves. The daemon executes the offer using the NFT's actual on-chain royalty, charging the user `1.025 XCH` â€” 0.0249 XCH more than they consented to. This is a direct, unprivileged-attacker-reachable financial loss via WalletConnect state spoofing.

For `chia_wallet.create_offer_for_ids`: The dApp directly controls `driver_dict`, so it can set any royalty percentage. An overstated value causes the user to see an inflated total and reject a legitimate offer creation; an understated value causes the user to approve an offer creation without understanding the true royalty burden on the taker.

### Likelihood Explanation

The attacker must be a dApp connected to the user's wallet via WalletConnect. WalletConnect is an opt-in feature, but once connected, any dApp can send `take_offer` or `create_offer_for_ids` requests. The user must have WalletConnect enabled and must approve the request. The attack requires no privileged keys, no host compromise, and no cryptographic break â€” only a malicious connected dApp or a crafted offer blob.

### Recommendation

1. In `parseRoyaltyPercentage`, enforce an upper bound matching the protocol maximum (10000 = 100%): reject any value `> 10000`.
2. For `take_offer`, cross-validate the driver dict's royalty percentage against the NFT's on-chain royalty (fetched via `nftGetInfo`) before displaying `amountWithRoyalties`. If they differ, display the on-chain value or show a warning.
3. For `create_offer_for_ids`, similarly validate the `driver_dict` royalty against on-chain data when the NFT is locally known.

### Proof of Concept

1. Attacker dApp is connected via WalletConnect.
2. Attacker crafts an offer blob where the NFT's driver dict sets `royalty_percentage: 1` (0.01%), while the actual NFT on-chain royalty is 250 (2.5%).
3. Attacker sends `chia_wallet.take_offer` via WalletConnect with this offer.
4. `parseCommandDisplay` calls `getOfferSummary`, which returns `infos` with `royalty_percentage: 1`.
5. `parseRoyaltyPercentage(1)` returns `1` â€” no upper bound check, no on-chain validation.
6. `formatAmountWithRoyalties` computes `amountWithRoyalties` as `1.0001 XCH` (for a 1 XCH offer).
7. The confirmation dialog shows: `Total Amount with Royalties: 1.0001 XCH`.
8. User approves.
9. Daemon enforces the NFT's actual on-chain royalty of 2.5%, charging `1.025 XCH`.
10. User pays 0.0249 XCH more than the displayed and consented-to amount.

function createOfferRoyaltyPercentages(
  walletDelta: WalletDelta,
  driverDict: Record<string, unknown>,
): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},

    royaltyPercentages.receiving[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

}
