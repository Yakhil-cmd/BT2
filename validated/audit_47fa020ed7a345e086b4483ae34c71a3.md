### Title
Unverified Single-Participant Contributions in CKD Protocol Allow Any Malicious Participant to Corrupt the CKD Output - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The Confidential Key Derivation (CKD) protocol's coordinator blindly sums each participant's `(norm_big_y, norm_big_c)` contribution without any proof of correctness. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a corrupted CKD output that honest parties accept as valid.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, the `do_ckd_coordinator` function collects contributions from all participants and aggregates them:

```rust
// lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to compute their share as:

```
norm_big_y_i = y_i * G * λ_i
norm_big_c_i = (x_i * H(pk, app_id) + y_i * app_pk) * λ_i
``` [2](#0-1) 

The coordinator sums these to form an ElGamal ciphertext `(Y, C)` where `C - app_sk * Y = msk * H(pk, app_id)`. However, there is **no zero-knowledge proof, commitment, or any other verification** that each participant's submitted `(norm_big_y_i, norm_big_c_i)` was honestly computed from their actual key share. Participants send their contribution **privately** to the coordinator only:

```rust
// do_ckd_participant, line 30
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

This is the direct analog to the PriceProvider trusting a single DEX: here, the coordinator trusts each participant's contribution as a single unverified source. Because contributions are sent privately (not broadcast), no other participant can cross-check them either.

Contrast this with the DKG protocol, which uses proofs of knowledge and commitment hashes to verify every participant's polynomial commitment before accepting it: [4](#0-3) 

No equivalent verification exists in the CKD protocol.

---

### Impact Explanation

A single malicious participant submits arbitrary group elements `(R, S)` instead of their honest `(norm_big_y_i, norm_big_c_i)`. The coordinator computes:

```
Y_corrupt  = Σ norm_big_y_j  (j ≠ m)  +  R
C_corrupt  = Σ norm_big_c_j  (j ≠ m)  +  S
```

The unmask operation then yields:

```
C_corrupt - app_sk * Y_corrupt  ≠  msk * H(pk, app_id)
```

The coordinator outputs a `CKDOutput` that is cryptographically well-formed (two valid group elements) but encodes the wrong confidential key. Honest parties have no mechanism to detect the corruption. This matches the allowed impact:

> **High: Corruption of CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs.** [5](#0-4) 

---

### Likelihood Explanation

Any single participant in the CKD session is a sufficient attacker. The participant need only deviate from the protocol by sending `(ElementG1::identity(), ElementG1::identity())` or any other arbitrary pair. There is no detection mechanism, no abort, and no retry. The attack requires no special knowledge beyond participation in the protocol.

---

### Recommendation

Add a non-interactive zero-knowledge proof of correct computation alongside each participant's `(norm_big_y_i, norm_big_c_i)` contribution. Specifically, each participant should prove in zero knowledge that:

1. `norm_big_y_i` is a scalar multiple of the generator consistent with their claimed `y_i * λ_i`.
2. `norm_big_c_i` encodes `(x_i * H(pk, app_id) + y_i * app_pk) * λ_i` for the same `y_i` and their committed public share `x_i * G2`.

This is analogous to the DKG fix (proofs of knowledge on polynomial commitments) and to the TWAP oracle fix in the original report: replacing blind trust in a single unverified source with cryptographic cross-validation.

---

### Proof of Concept

```
Setup:
  participants = [P1, P2, P3], coordinator = P1
  threshold = 3 (all-threshold for simplicity)
  msk = λ1*x1 + λ2*x2 + λ3*x3

Honest execution:
  P2 sends (norm_big_y_2, norm_big_c_2) to P1 privately
  P3 sends (norm_big_y_3, norm_big_c_3) to P1 privately
  P1 computes Y = Σ norm_big_y_i, C = Σ norm_big_c_i
  unmask(app_sk) = C - app_sk*Y = msk * H(pk, app_id)  ✓

Attack (P3 is malicious):
  P3 sends (ElementG1::identity(), ElementG1::identity()) to P1
  P1 computes Y' = norm_big_y_1 + norm_big_y_2 + 0
                 = Y - norm_big_y_3
              C' = norm_big_c_1 + norm_big_c_2 + 0
                 = C - norm_big_c_3
  unmask(app_sk) = C' - app_sk*Y'
                 = msk*H(pk,app_id) - norm_big_c_3 + app_sk*norm_big_y_3
                 ≠ msk * H(pk, app_id)

  The coordinator outputs a corrupted CKDOutput with no error.
  No participant detects the deviation.
``` [6](#0-5)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
