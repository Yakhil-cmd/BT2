### Title
Hardcoded `FT_TRANSFER_GAS` in `ExitToNear` Precompile Causes Permanent Fund Freeze for Non-Standard NEP-141 Tokens - (File: engine-precompiles/src/native.rs)

### Summary
The `ExitToNear` precompile hardcodes `FT_TRANSFER_GAS = 10 TGas` for every NEP-141 `ft_transfer` promise it creates during ERC-20 token exits. Because `EvmErc20.sol` burns the caller's ERC-20 tokens **before** the precompile is invoked, a NEP-141 token whose `ft_transfer` implementation requires more than 10 TGas will cause the asynchronous promise to fail after the burn is already committed. Without the compile-time `error_refund` feature, there is no recovery path and the user's funds are permanently destroyed.

### Finding Description
In `engine-precompiles/src/native.rs`, the `costs` module fixes a single gas budget for every outbound `ft_transfer` call:

```rust
// engine-precompiles/src/native.rs  lines 51-53
/// Value determined experimentally based on tests and mainnet data.
pub(super) const FT_TRANSFER_GAS: NearGas = NearGas::new(10_000_000_000_000);
``` [1](#0-0) 

This constant is used when the `ExitToNear` precompile constructs the `PromiseCreateArgs` that the engine later dispatches as a NEAR cross-contract call to the bridged NEP-141 token's `ft_transfer` method.

The burn-then-promise ordering is enforced by `EvmErc20.sol`:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol  lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, unconditionally

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
}
``` [2](#0-1) 

The precompile call succeeds (it only emits a log encoding the promise), so the EVM transaction commits with the burn applied. The actual `ft_transfer` on the NEP-141 contract is executed asynchronously by the NEAR runtime with exactly `FT_TRANSFER_GAS` (10 TGas) attached. If the NEP-141 token's `ft_transfer` implementation consumes more than 10 TGas — for example, a token with fee-on-transfer logic, storage-deposit checks, or additional hooks — the promise fails after the ERC-20 tokens are already gone.

The `error_refund` feature is the only mitigation, and it is a compile-time opt-in:

```rust
// engine-precompiles/src/native.rs  lines 36-39
#[cfg(not(feature = "error_refund"))]
const MIN_INPUT_SIZE: usize = 3;
#[cfg(feature = "error_refund")]
const MIN_INPUT_SIZE: usize = 21;
``` [3](#0-2) 

The test suite explicitly documents the no-refund outcome when the feature is absent:

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 656-660
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

### Impact Explanation
When `error_refund` is not compiled in, a failed `ft_transfer` promise leaves the NEP-141 tokens held by the Aurora engine account while the corresponding ERC-20 tokens are permanently burned. The user has no on-chain mechanism to recover the value. This constitutes **permanent freezing of funds** (Critical).

### Likelihood Explanation
The standard NEAR reference FT implementation consumes roughly 5–10 TGas for `ft_transfer`, placing the hardcoded limit right at the boundary. Any NEP-141 token with non-trivial transfer logic (fee-on-transfer, allowlist checks, storage-deposit enforcement, or cross-contract callbacks inside `ft_transfer`) can exceed 10 TGas. Aurora supports arbitrary NEP-141 tokens via `deploy_erc20_token`, so the affected token space is unbounded. An unprivileged user triggers the path simply by calling `withdrawToNear` on any such bridged ERC-20.

### Recommendation
Mirror the fix proposed in the reference report: make the gas budget a caller-supplied parameter rather than a compile-time constant. Concretely, extend the `ExitToNear` precompile input format to accept an optional `ft_transfer_gas` field, defaulting to the current value for backward compatibility. Alternatively, adopt a sufficiently conservative upper bound (e.g., 30 TGas) that covers realistic custom NEP-141 implementations, and ensure `error_refund` is always enabled in production builds as a safety net.

### Proof of Concept
1. Deploy a custom NEP-141 token whose `ft_transfer` implementation performs additional on-chain work (e.g., a storage-deposit check or a secondary cross-contract call) that pushes its gas consumption above 10 TGas.
2. Register the token with Aurora via `deploy_erc20_token`; bridge tokens to an ERC-20 balance.
3. Call `withdrawToNear` on the `EvmErc20` contract from a user account.
4. The EVM transaction succeeds and the ERC-20 balance is zeroed (`_burn` committed).
5. The NEAR runtime dispatches the `ft_transfer` promise with 10 TGas; the NEP-141 contract runs out of gas and the promise fails.
6. Without `error_refund`, no callback re-mints the ERC-20 tokens. The user's funds are permanently lost.
