### Title
Dysfunctional `MsgTurnBridge` Handler — No-Op Stub Bypasses `CanTurnBridge` Permission and Permanently Disables Bridge Circuit Breaker - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` message handler in Cronos is a no-op stub that silently succeeds without performing any state change and without enforcing the `CanTurnBridge` permission. This is the direct analog to the external report: a fully-defined, proto-registered message surface exists (`MsgTurnBridge`) with a dedicated permission bit (`CanTurnBridge`), a CLI command, and integration-test coverage — but the server-side implementation is permanently broken, making the bridge circuit breaker permanently non-functional.

### Finding Description

The `TurnBridge` gRPC handler in `x/cronos/keeper/msg_server.go` is implemented as a stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly enforces its permission before acting:

```go
func (k msgServer) UpdateTokenMapping(...) (...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    ...
}
``` [2](#0-1) 

The permission system defines `CanTurnBridge` as a distinct bit alongside `CanChangeTokenMapping`:

```go
const (
    CanChangeTokenMapping uint64 = 1 << iota // 1
    CanTurnBridge                             // 2
    All = CanChangeTokenMapping | CanTurnBridge // 3
)
``` [3](#0-2) 

`CanTurnBridge` is never passed to `HasPermission` anywhere in the production keeper code. The `TurnBridge` handler:
1. Performs **no permission check** — any unprivileged sender can call it and receive a success response.
2. Performs **no state mutation** — the `enable` field of `MsgTurnBridge` is decoded and discarded.
3. Returns `nil, nil` — the Cosmos SDK interprets this as a successful transaction.

The proto definition, CLI command (`CmdTurnBridge`), and integration tests all treat `TurnBridge` as a real, functional message: [4](#0-3) [5](#0-4) 

### Impact Explanation

**High — Bypass of Cronos admin/governance bridge authorization checks and permanent inability to disable bridge flows.**

Two distinct impacts:

1. **Permission bypass**: The `CanTurnBridge` permission bit is defined and can be granted via `MsgUpdatePermissions`, but it is never enforced in the `TurnBridge` handler. Any unprivileged address can submit `MsgTurnBridge` and receive a success response, completely bypassing the intended authorization model.

2. **Permanent bridge circuit-breaker failure**: The `TurnBridge` message is the designated emergency mechanism to disable the Gravity Bridge (as confirmed by ADR-001 and integration tests in `test_gravity.py`). Because the handler is a no-op, the bridge can never be disabled by any actor — admin, governance, or permissioned address. In a bridge exploit scenario, the circuit breaker is permanently inoperative, leaving all bridge/conversion flows exposed indefinitely. [6](#0-5) 

### Likelihood Explanation

The entry path is fully reachable: `MsgTurnBridge` is a registered proto message, exposed via gRPC and CLI, with no ante-handler or module-level guard blocking it. Any unprivileged user on the network can submit the transaction. The no-op behavior is deterministic and unconditional — it applies on every call regardless of the `enable` field value or the sender's identity.

### Recommendation

Implement the `TurnBridge` handler body to:
1. Enforce `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` before proceeding.
2. Persist the `enable` flag to module state (e.g., a `bridge_active` key in the KV store or a module parameter).
3. Have the Gravity Bridge hooks and end-blocker read this flag and gate all bridge operations accordingly.

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    // persist bridge active state
    k.SetBridgeActive(ctx, msg.Enable)
    return &types.MsgTurnBridgeResponse{}, nil
}
```

### Proof of Concept

1. Any address submits: `cronosd tx cronos turn-bridge false --from <any_unprivileged_key>`
2. Transaction is accepted and returns code 0 (success) — confirmed by the `nil, nil` return path.
3. Query bridge state: bridge remains active regardless of the `enable` value sent.
4. Admin attempts to disable bridge during an exploit: transaction succeeds on-chain but bridge is never disabled, all bridge/conversion flows continue unimpeded. [1](#0-0)
