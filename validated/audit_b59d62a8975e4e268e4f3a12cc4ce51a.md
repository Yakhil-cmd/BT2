### Title
Unauthenticated `ft_on_transfer` Allows Minting Unbacked ERC-20 Tokens — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`ft_on_transfer` in Aurora Engine does not verify that the calling account actually transferred NEP-141 tokens before minting ERC-20 tokens. Because `deploy_erc20_token` is also unrestricted, any unprivileged NEAR account can register a NEP-141, then call `ft_on_transfer` directly to mint an arbitrary quantity of the corresponding ERC-20 on Aurora without depositing any real tokens. This is the direct analog of the external report's `parseResponse` bug: a single entry-point branches on a field (caller identity) and, when the "authenticated" branch is not taken, processes the request without enforcing the invariant that a real transfer occurred.

---

### Finding Description

**Vulnerable entry-point — `ft_on_transfer`**

`ft_on_transfer` is a public NEAR contract method. Its only caller-identity check is whether `predecessor_account_id` equals the stored ETH-connector account. If it does not, the function unconditionally calls `receive_erc20_tokens`, using the predecessor as the NEP-141 token identifier:

```rust

        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,

};

There is no check that the predecessor is a legitimate NEP-141 contract that has already transferred tokens to Aurora (the invariant the NEP-141 `ft_transfer_call` flow guarantees). Any NEAR account whose account ID is registered as a NEP-141 on Aurora can call `ft_on_transfer` directly and claim any `amount`.

**`receive_erc20_tokens` mints without transfer verification**

Inside `receive_erc20_tokens`, the engine looks up the ERC-20 for the predecessor and calls the ERC-20 `mint` selector with the attacker-supplied `amount`:

```rust

No on-chain proof, receipt, or balance check is performed. The only gate is that the predecessor must have a registered ERC-20.

---

### Impact Explanation

An attacker can mint an unbounded quantity of ERC-20 tokens on Aurora that are not backed by any real NEP-141 deposit. These tokens are indistinguishable on-chain from legitimately bridged tokens. The attacker can:

1. Deposit the unbacked ERC-20 as collateral in a lending protocol deployed on Aurora, borrow real bridged assets (e.g., `usdc-erc20`), exit those assets via `ExitToNear`, and default — directly stealing real NEP-141 tokens held by Aurora on behalf of legitimate users.
2. Provide one-sided liquidity in a DEX pool pairing the unbacked ERC-20 against a real bridged asset, drain the real side, and exit.

Either path causes **bridge insolvency**: Aurora holds fewer NEP-141 tokens than the total ERC-20 supply it has minted, so legitimate users cannot fully exit. This meets the Critical threshold (direct theft of user funds / insolvency).

---

### Likelihood Explanation

The prerequisite steps are all trivially reachable by an unprivileged EVM/NEAR user:

- Deploying a NEAR NEP-141 contract: standard NEAR account operation.
- Calling `deploy_erc20_token` on Aurora: no access control, open to any caller.
- Calling `ft_on_transfer` directly: public NEAR method, no deposit or key required.
- Monetising via DeFi: Aurora mainnet hosts active DEXes and lending protocols; listing a new ERC-20 and providing initial liquidity is routine.

No privileged keys, governance capture, or oracle compromise is required. Likelihood is **High**.

---

### Recommendation

**Short term**: In `ft_on_transfer`, add a whitelist check that the `predecessor_account_id` is a known, trusted NEP-141 contract before calling `receive_erc20_tokens`. Alternatively, split the function into two: one that is callable only by registered, trusted token contracts (enforced via an allowlist stored in contract state), and one for the ETH-connector path. This mirrors the external report's recommendation to split `parseResponse` into a decryption function and a parsing function so that encryption is always enforced.

**Long term**: Treat the invariant "ERC-20 tokens on Aurora are always backed 1:1 by NEP-141 tokens held by Aurora" as a protocol invariant. Enforce it at the entry point rather than relying on callers to follow the NEP-141 `ft_transfer_call` convention. Consider requiring a verifiable NEAR receipt or cross-contract proof before minting.
