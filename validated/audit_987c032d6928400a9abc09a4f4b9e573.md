### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function aggregates participant contributions to the Confidential Key Derivation (CKD) protocol without any verification that participants computed their shares correctly. A single malicious participant can send arbitrary group elements to the coordinator, silently corrupting the final `CKDOutput` that all honest parties will accept as valid.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives `(norm_big_y, norm_big_c)` from every other participant and unconditionally adds them into the running aggregate: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The correct computation each participant is supposed to perform is: [2](#0-1) 

```rust
let big_y = ElementG1::generator() * y.0;
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
let big_s = hash_point * private_share.to_scalar();
let big_c = big_s + app_pk * y.0;
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
```

There is no zero-knowledge proof, commitment, or any other mechanism attached to the message that would allow the coordinator to verify that a participant used their actual private key share `private_share` and a valid ephemeral scalar `y`. The coordinator blindly trusts whatever `(norm_big_y, norm_big_c)` pair arrives over the channel.

This is directly analogous to the reported pattern: a function (`do_ckd_coordinator`) performs a consequential aggregation operation without first validating that the inputs satisfy the invariants required for the output to be correct. Just as `GetRestorationProof` generates a valid-looking proof for an invalid target, `do_ckd_coordinator` produces a valid-looking `CKDOutput` from maliciously crafted inputs.

---

### Impact Explanation

The `CKDOutput` is the sole product of the protocol. It is used by the application to unmask the confidential derived key: [3](#0-2) 

A malicious participant who sends an arbitrary `(norm_big_y, norm_big_c)` pair causes the summed `CKDOutput` to be cryptographically incorrect. The coordinator and the application have no way to detect this — the output is a valid pair of `G1` curve points regardless of whether it was computed honestly. The application will call `unmask(app_sk)` on the corrupted output and receive a wrong derived key, silently. This constitutes **corruption of CKD outputs so honest parties accept unusable cryptographic outputs** — a **High** severity impact under the allowed scope.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker-controlled entry path is straightforward:

1. A malicious party participates in a legitimate CKD session.
2. Instead of calling `compute_signature_share` honestly, it sends arbitrary `G1` elements `(norm_big_y', norm_big_c')` to the coordinator via the private channel.
3. The coordinator aggregates these without complaint.
4. The final `CKDOutput` is corrupted.

No special privileges, leaked keys, or cryptographic breaks are required. The attack is reachable by any participant in any CKD session.

---

### Recommendation

Each participant should attach a zero-knowledge proof of correct computation alongside their `(norm_big_y, norm_big_c)` contribution. Concretely, a participant must prove:

- Knowledge of `y` such that `norm_big_y = lambda_i * (y * G)`.
- Knowledge of `private_share` such that `norm_big_c - norm_big_y * (app_pk / G) = lambda_i * (private_share * hash_point)`, i.e., a proof of discrete-log equality between the participant's public key share and the `big_s` component embedded in `norm_big_c`.

The coordinator should verify all such proofs before aggregating any contribution, analogous to how `verify_proof_of_knowledge` is called in `do_keyshare` before accepting DKG commitments: [4](#0-3) 

---

### Proof of Concept

1. Run a 3-of-3 CKD session with participants `[A, B, C]` where `C` is the coordinator.
2. Participant `A` (malicious) computes `norm_big_y' = ElementG1::identity()` and `norm_big_c' = ElementG1::identity()` and sends these to `C` instead of the honest values.
3. `do_ckd_coordinator` adds `identity` elements into the aggregate — effectively subtracting `A`'s honest contribution.
4. The coordinator returns a `CKDOutput` that does not correspond to `msk * H(pk, app_id)`.
5. The application calls `ckd_output.unmask(app_sk)` and receives a wrong key with no error or indication of failure. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
