Audit Report

## Title
Unbounded `disburse_maturity_in_progress` Array Growth Causes Instruction-Limit DoS on SNS Neuron - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance `disburse_maturity` function appends to a neuron's `disburse_maturity_in_progress` vector without any cap on its length. A neuron controller can call `DisburseMaturity` repeatedly with small percentages, growing this array unboundedly until subsequent operations on the neuron — including the periodic finalization task — trap on the IC instruction limit. The NNS governance already has an identical fix (`MAX_NUM_DISBURSEMENTS = 10`) that the SNS governance is missing, confirming this is a known class of issue with a known remediation.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 1696–1698, the `disburse_maturity` function unconditionally pushes a new `DisburseMaturityInProgress` entry onto the neuron's vector:

```rust
neuron
    .disburse_maturity_in_progress
    .push(disbursement_in_progress);
```

The only guards present (lines 1633–1678) are: `percentage_to_disburse` must be 1–100, the worst-case modulated amount must exceed `transaction_fee_e8s`, and the caller must hold `DisburseMaturity` permission. There is no check on the current length of `disburse_maturity_in_progress`.

The SNS minimum disbursement is bounded only by the transaction fee (default 10,000 e8s), far lower than the NNS minimum of `MINIMUM_DISBURSEMENT_E8S = E8 = 100,000,000 e8s`. This means an SNS neuron with 1 ICP of maturity can create ~9,500 entries (vs. 1 entry in NNS), and a neuron with 100 ICP can create ~950,000 entries.

The SNS `run_periodic_tasks` (line 5527) calls `maybe_finalize_disburse_maturity()`, which iterates over `neuron_id_and_disbursements` (lines 4976–5035) to process pending disbursements. When the vector is very large, deserializing the neuron, iterating over the vector, and re-serializing it all consume instructions proportional to the vector length, eventually trapping on the 40B/50B instruction limit.

By contrast, the NNS governance enforces `MAX_NUM_DISBURSEMENTS = 10` (line 40 of `rs/nns/governance/src/governance/disburse_maturity.rs`) and checks it at lines 306–308 before any push, returning `TooManyDisbursements`. No equivalent constant or check exists anywhere in the SNS governance codebase.

## Impact Explanation
This matches the allowed High impact: "Significant SNS security impact with concrete user or protocol harm." Specifically:

1. **Permanent maturity loss**: Maturity deducted from `maturity_e8s_equivalent` is irrecoverably stuck in the queue once the neuron becomes unprocessable. The neuron is permanently bricked — no `manage_neuron` call, no `get_neuron` query, and no periodic finalization can succeed on it.
2. **SNS governance periodic task DoS**: The `run_periodic_tasks` heartbeat calls `maybe_finalize_disburse_maturity()` synchronously. A single bloated neuron can cause this task to trap on every invocation, delaying finalization of all other neurons' disbursements across the entire SNS instance and degrading SNS governance availability for all participants.

## Likelihood Explanation
Any SNS neuron controller with `DisburseMaturity` permission (the default for the neuron controller) and sufficient accumulated maturity can trigger this. No special privilege is required. SNS neurons accumulate maturity through voting rewards over time; active participants with long-locked neurons naturally accumulate large balances. The attack requires only repeated calls to a standard governance endpoint. The attacker primarily harms themselves (their own neuron is bricked and maturity is lost), but the periodic-task disruption is an externality affecting all SNS participants.

## Recommendation
Add a cap on `disburse_maturity_in_progress.len()` in `rs/sns/governance/src/governance.rs` inside `disburse_maturity()`, mirroring the NNS fix:

```rust
const MAX_DISBURSE_MATURITY_IN_PROGRESS: usize = 10;

if neuron.disburse_maturity_in_progress.len() >= MAX_DISBURSE_MATURITY_IN_PROGRESS {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        format!(
            "Too many disbursements in progress (max {MAX_DISBURSE_MATURITY_IN_PROGRESS}). \
             Wait for existing disbursements to finalize before initiating new ones."
        ),
    ));
}
```

This check should be inserted after the authorization and amount checks (after line 1678) and before the push at line 1696.

## Proof of Concept
1. Deploy an SNS with a neuron holding large maturity (e.g., 100 ICP equivalent in `maturity_e8s_equivalent`).
2. As the neuron controller, call `manage_neuron { DisburseMaturity { percentage_to_disburse: 1 } }` in a loop. Each call succeeds, deducting ~1% of remaining maturity and appending one entry to `disburse_maturity_in_progress`.
3. After ~100+ calls (or more with a low SNS transaction fee), attempt `get_neuron` or another `manage_neuron` call on the same neuron.
4. Observe the call traps due to instruction limit exhaustion during neuron deserialization/processing.
5. Observe that `run_periodic_tasks` also traps when `maybe_finalize_disburse_maturity()` processes this neuron, delaying finalization for all other neurons in the SNS.
6. A deterministic integration test can be written using the existing `setup_test_environment_with_one_neuron_with_maturity` fixture (as used in `rs/sns/governance/tests/governance.rs`) by calling `DisburseMaturity` with `percentage_to_disburse: 1` in a loop until maturity is exhausted, then asserting that `run_periodic_tasks_now()` traps or that `disburse_maturity_in_progress.len()` grows without bound.
