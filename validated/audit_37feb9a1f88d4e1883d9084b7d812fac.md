### Title
Permanent Fund Loss in `ExitToEthereum` Precompile When NEP-141 `withdraw` Is Paused — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns ERC-20 tokens inside the EVM and schedules an asynchronous NEAR promise to call `withdraw` on the corresponding NEP-141 contract. There is no callback or refund mechanism attached to this promise. If the NEP-141 contract's `withdraw` method is paused or temporarily blocked when the promise executes, the ERC-20 tokens are permanently destroyed with no corresponding asset release, resulting in a permanent freeze of user funds.

---

### Finding Description

When a user exits ERC-20 tokens from Aurora to Ethereum via the `ExitToEthereum` precompile (flag `0x01`), the flow is:

1. The ERC-20 contract burns the user's tokens and calls the precompile.
2. The precompile constructs a `PromiseArgs::Create` targeting the NEP-141 contract's `withdraw` method.
3. The EVM transaction succeeds — tokens are burned.
4. The NEAR promise executes asynchronously.

The critical issue is in `ExitToEthereum::run()`:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

Unlike `ExitToNear`, which conditionally attaches a `PromiseArgs::Callback` with an `exit_to_near_precompile_callback` refund handler, `ExitToEthereum` always uses `PromiseArgs::Create` — a fire-and-forget promise with no error handling path. [2](#0-1) 

The eth-connector (and any NEP-141 contract) has its own independent pause mechanism (`pa_pause_feature` / `engine_withdraw` key), separate from the Aurora Engine's `pause_contract` / `pause_precompiles` controls: [3](#0-2) 

The Aurora Engine's precompile-level pause only prevents the EVM call from starting. It does not protect against the NEAR-side promise failing after the EVM tokens are already burned. [4](#0-3) 

The `ExitToNear` callback refund path (when `error_refund` feature is enabled) demonstrates the engine is aware of this failure mode for one precompile but left `ExitToEthereum` without any equivalent protection: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the NEP-141 contract's `withdraw` is paused at the time the NEAR promise executes:
- The user's ERC-20 tokens are already burned in the EVM (irreversible).
- The NEAR promise fails silently — no refund, no retry, no callback.
- The user permanently loses the full token amount.
- If the pause is later lifted, no mechanism exists to recover the burned tokens.

This matches the Illuminate pattern exactly: assets are consumed on one side before the other side confirms receipt, and there is no accounting for the intermediate failure state.

---

### Likelihood Explanation

**Medium.** The eth-connector's `engine_withdraw` feature can be paused by the eth-connector admin for legitimate maintenance or security reasons. This is a documented, exercised capability (tested in `test_withdraw_from_near_pausability`). Any user who calls `ExitToEthereum` while this pause is active — or whose promise executes during a pause window — permanently loses funds. The attacker-controlled entry path is simply calling the standard ERC-20 bridge-out flow; no special privileges are required. [6](#0-5) 

---

### Recommendation

Attach a callback promise to `ExitToEthereum` analogous to the `exit_to_near_precompile_callback` pattern already used in `ExitToNear`. The callback should:
1. Check the promise result.
2. On failure, re-mint the burned ERC-20 tokens (or restore the ETH balance) to the original caller's EVM address.

The `ExitToNear` refund path in `engine/src/contract_methods/connector.rs` (`refund_on_error`) provides the template for this fix. [5](#0-4) 

---

### Proof of Concept

1. User holds 1000 units of an ERC-20 token on Aurora (backed by a NEP-141).
2. The eth-connector admin pauses `engine_withdraw` (e.g., for a security incident).
3. User calls the ERC-20's `withdrawToEthereum` function, which burns 1000 tokens and triggers `ExitToEthereum`.
4. The EVM transaction succeeds — 1000 tokens are burned.
5. The NEAR promise to call `withdraw` on the eth-connector executes and fails with `"Pausable: Method is paused"`.
6. No callback fires. No refund occurs.
7. The eth-connector admin later unpauses. The user's 1000 tokens are gone with no recovery path.
