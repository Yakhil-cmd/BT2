### Title
Missing Proof-of-Correct-Computation for Participant Share Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates share contributions from all participants without any cryptographic verification that each contribution was honestly computed. A single malicious participant can send arbitrary group elements, causing the coordinator to output a silently corrupted derived key that honest parties will accept as valid.

---

### Finding Description

The external report's vulnerability class is **missing prerequisite authorization before a consuming operation**: `UniV3PoolHelper` calls `UniV3TokenizedLp.deposit()` which internally calls `transferFrom()`, but the required `approve()` was never issued, so the operation silently fails or reverts. The analog here is: the CKD coordinator calls into the aggregation step which requires each participant's contribution to be cryptographically "authorized" (proven correct), but no such proof is ever issued or checked, so the aggregation silently accepts malicious inputs.

In `do_ckd_coordinator`, the coordinator receives a `CKDOutput` from every other participant and unconditionally adds the group elements together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation each participant is supposed to perform is:

```
big_s_i  = x_i · H(pk ‖ app_id)          // BLS signature share
big_c_i  = big_s_i + y_i · app_pk         // ElGamal encryption
``` [2](#0-1) 

The coordinator has no way to verify that the received `(norm_big_y, norm_big_c)` was produced using the participant's actual private share `x_i`. There is no zero-knowledge proof, no commitment, and no consistency check between `big_y` and `big_c`. The coordinator simply trusts whatever group elements arrive over the channel.

The prerequisite that is missing — analogous to the missing `approve()` — is a **proof of correct computation**: before the coordinator may consume a participant's contribution, that participant must prove that `big_c_i - y_i · app_pk = x_i · H(pk ‖ app_id)` for their actual registered share `x_i`. Without this proof the aggregation step has no authorization to treat the received value as a valid share.

---

### Impact Explanation

A malicious participant sends arbitrary `(big_y', big_c')` instead of the honestly computed values. The coordinator adds these into the running sum:

```
norm_big_c_total = Σ honest_i(big_c_i) + big_c'_malicious
```

The resulting `CKDOutput` is silently incorrect. When the requester calls `ckd_output.unmask(app_sk)` they obtain a group element that does not equal `msk · H(pk ‖ app_id)`. The honest coordinator and all honest participants have no mechanism to detect this; they return `Ok(Some(ckd_output))` with the corrupted value. [3](#0-2) 

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session can exploit this. No special capability is required beyond being a registered participant. The attacker only needs to deviate from the protocol by sending arbitrary `ElementG1` values instead of the honestly computed ones. The coordinator has no defense. Likelihood is **High**.

---

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` with a Schnorr-style zero-knowledge proof of correct computation demonstrating that:

- `big_c_i - y_i · app_pk = x_i · H(pk ‖ app_id)`
- `big_y_i = y_i · G`

where `x_i` is the participant's private share whose corresponding public share is known to all parties from the DKG output. The coordinator must verify all proofs before aggregating contributions, rejecting any participant whose proof fails.

---

### Proof of Concept

1. Honest participants P_1 … P_{n-1} each compute `(norm_big_y_i, norm_big_c_i)` correctly using their private share `x_i` and send to coordinator.
2. Malicious participant P_m picks arbitrary scalars `r, s` and sends `(r·G, s·G)` as their `CKDOutput` to the coordinator.
3. The coordinator executes the loop at lines 50–55 of `protocol.rs`, adding P_m's arbitrary elements into `norm_big_y` and `norm_big_c` without any check. [4](#0-3) 
4. The coordinator returns `CKDOutput::new(norm_big_y, norm_big_c)` where `norm_big_c ≠ msk · H(pk ‖ app_id) + Y · app_pk`.
5. The requester calls `ckd_output.unmask(app_sk)` and obtains a corrupted derived key with no error or warning. All honest parties accept this output as the protocol result.
