### Title
Malicious CKD Coordinator Can Omit or Substitute Participant Shares During Aggregation, Corrupting the Derived Confidential Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the CKD protocol, non-coordinator participants send their cryptographic shares **privately** to the coordinator and immediately return `None` with no further interaction. A malicious coordinator can omit, substitute, or partially aggregate participant shares, producing an incorrect `CKDOutput` that honest participants have no mechanism to detect or verify. This is the direct structural analog to the `ValidateVoteExtensions` bug: a single privileged party (proposer/coordinator) controls the aggregation of externally-provided data, and the result is accepted without independent cross-verification.

---

### Finding Description

In `do_ckd_participant` (lines 17–33 of `src/confidential_key_derivation/protocol.rs`), each non-coordinator participant:

1. Computes their Lagrange-weighted share `(norm_big_y, norm_big_c)`.
2. Sends it **privately** to the coordinator via `chan.send_private`.
3. Returns `Ok(None)` immediately — receiving no response, no broadcast, and no confirmation. [1](#0-0) 

In `do_ckd_coordinator` (lines 35–58), the coordinator:

1. Computes its own share.
2. Calls `recv_from_others` to collect all other participants' shares.
3. Sums them into `norm_big_y` and `norm_big_c`.
4. Returns `Some(CKDOutput::new(norm_big_y, norm_big_c))` — the sole holder of the result. [2](#0-1) 

The aggregation loop at lines 50–55 is entirely under the coordinator's control:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [3](#0-2) 

There is no broadcast of the aggregated result back to participants, no commitment scheme, and no mechanism for any non-coordinator participant to verify that their share was included or used correctly. The coordinator is the only party that ever sees the `CKDOutput`.

The correct CKD output requires summing **all** Lagrange-weighted shares:

```
C = Σ_i λ_i · (x_i · H(pk, app_id) + y_i · app_pk)
  = msk · H(pk, app_id) + y · app_pk
``` [4](#0-3) 

If the coordinator omits participant `j`'s share, the output becomes:

```
C' = Σ_{i≠j} λ_i · (x_i · H(pk, app_id) + y_i · app_pk)
```

which does **not** equal `msk · H(pk, app_id) + y · app_pk`. The derived confidential key is therefore wrong, and no honest party can detect this.

This is structurally identical to the `ValidateVoteExtensions` bug: the proposer/coordinator injects or controls the data used to compute a total (`totalVP` / `CKDOutput`), and the result is accepted without independent verification against a trusted source.

---

### Impact Explanation

A malicious coordinator produces a `CKDOutput` that does not correctly encode `msk · H(pk, app_id)`. Any application that uses this output to derive a confidential key will obtain an incorrect key — one that cannot be reproduced by honest re-execution of the protocol. This constitutes **corruption of CKD output**: honest parties accept an unusable or attacker-controlled cryptographic output. This matches the allowed High impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

The coordinator is simply one of the `participants` passed to `ckd()` — there is no elevated trust assumption documented in the code or enforced by the protocol. [5](#0-4) 

Any participant who is selected as coordinator can trivially exploit this: they need only skip adding one or more received shares in the aggregation loop, or substitute arbitrary group elements. The attack requires no cryptographic break, no external compromise, and no special capability beyond being the coordinator.

---

### Recommendation

Replace the private-send-to-coordinator pattern with a verifiable aggregation scheme:

1. **Commit-then-reveal**: Each participant broadcasts a commitment to their share before revealing it. After the coordinator publishes the aggregated output, every participant can verify their committed share was included.
2. **Broadcast aggregation**: The coordinator broadcasts the final `CKDOutput` to all participants. Each participant verifies that `norm_big_y` and `norm_big_c` are consistent with their own contribution (e.g., by checking `C - λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` lies in the expected subspace).
3. **Peer-to-peer cross-check**: Participants exchange shares with each other (not only with the coordinator), enabling independent reconstruction and verification of the aggregate.

---

### Proof of Concept

**Setup**: 3 participants `A` (coordinator), `B`, `C` with valid key shares `x_A, x_B, x_C` and Lagrange coefficients `λ_A, λ_B, λ_C`.

**Honest execution**: Coordinator sums all three shares → `C = msk · H(pk, app_id) + y · app_pk`.

**Attack**:
1. `B` sends `(λ_B · y_B · G, λ_B · (x_B · H + y_B · app_pk))` privately to `A`.
2. `C` sends `(λ_C · y_C · G, λ_C · (x_C · H + y_C · app_pk))` privately to `A`.
3. `A` receives both shares but **does not add them** — it returns `CKDOutput::new(norm_big_y_A, norm_big_c_A)` using only its own contribution.
4. `B` and `C` both return `Ok(None)` and have no further protocol interaction.
5. The resulting `CKDOutput` encodes only `λ_A · x_A · H(pk, app_id) + ...`, not `msk · H(pk, app_id)`.
6. The application unmasks an incorrect confidential key. `B` and `C` cannot detect the manipulation. [6](#0-5)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

        return Err(InitializationError::DuplicateParticipants);

        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });

    // ensure the coordinator is a participant

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
