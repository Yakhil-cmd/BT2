### Title
ERC-20 Tokens Permanently Burned With No Refund When `ft_transfer` Fails in `exitToNear` Flow (Without `error_refund` Feature) - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When a user calls the `ExitToNear` precompile to bridge ERC-20 tokens back to NEAR, the ERC-20 tokens are burned first, then a NEAR `ft_transfer` promise is scheduled. If that promise fails (e.g., unregistered recipient), the callback `exit_to_near_precompile_callback` is supposed to re-mint the burned tokens. However, because the `error_refund` Cargo feature is **not enabled in the production build**, the callback is compiled with `refund: None`, and the failure branch silently does nothing. The burned ERC-20 tokens are permanently lost and the corresponding NEP-141 tokens remain stuck in Aurora's contract with no recovery path.

---

### Finding Description

**Step 1 — ERC-20 tokens are burned and a promise is scheduled.**

In `engine-precompiles/src/native.rs`, the `ExitToNear` precompile constructs the callback arguments:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,                          // ← always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent (the production case), `refund` is unconditionally `None`. The promise is then structured as a `PromiseArgs::Callback` only when `callback_args != Default::default()`. Because `transfer_near` can also be `None` for a plain ERC-20 exit, the callback may be omitted entirely; but even when it is included (e.g., for wNEAR unwrap), the `refund` field carries no recovery information.

**Step 2 — The callback silently ignores the failure.**

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` handles the promise result:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← taken in production when ft_transfer fails
};
``` [2](#0-1) 

When `args.refund` is `None` and the promise failed, the `else { None }` branch executes. No re-mint of the burned ERC-20 tokens occurs, and no error is returned. The transaction succeeds from NEAR's perspective while the user's funds are gone.

**Step 3 — `error_refund` is not part of the production feature set.**

`engine/Cargo.toml` defines the features:

```toml
[features]
default = ["std"]
contract = ["log", "aurora-engine-sdk/contract", ...]
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [3](#0-2) 

`error_refund` is not included in `default`, `contract`, or `integration-test`. The production WASM is compiled with `contract`, so `error_refund` is never active on-chain.

**Step 4 — The test explicitly documents the loss.**

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The test confirms that without `error_refund`, `FT_EXIT_AMOUNT` tokens are permanently missing from the user's ERC-20 balance after a failed exit.

---

### Impact Explanation

- The ERC-20 tokens are **burned** inside the EVM at the start of the exit flow.
- The corresponding NEP-141 tokens remain in Aurora's contract account (the `ft_transfer` failed, so Aurora still holds them).
- With `error_refund` disabled, `refund_on_error` is never called, so the ERC-20 tokens are never re-minted.
- Result: the user loses their ERC-20 tokens permanently, and the equivalent NEP-141 balance is frozen inside Aurora's account with no on-chain recovery mechanism.
- **Impact: Permanent freezing / direct loss of user funds (Critical).**

---

### Likelihood Explanation

Any unprivileged EVM user can trigger this by calling `exitToNear` (directly or via an ERC-20 contract's burn/exit function) with a NEAR recipient account that is not registered with the target NEP-141 token. This is a realistic, low-effort mistake (typo in account ID, account not yet registered for storage). The `ft_transfer` call will fail, and the loss is immediate and irreversible. No special privileges or external conditions are required beyond a normal EVM transaction.

---

### Recommendation

Enable the `error_refund` feature in the production `contract` feature set, or unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` regardless of the feature flag. The `refund_call_args` helper already exists and correctly encodes the re-mint arguments; the only change needed is to remove the `#[cfg(not(feature = "error_refund"))] refund: None` branch and always pass the refund data to the callback. The callback logic in `exit_to_near_precompile_callback` already handles the refund correctly when `args.refund` is `Some`.

---

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature (the default production configuration).
2. Bridge a NEP-141 token into Aurora, receiving ERC-20 tokens.
3. Call the ERC-20 contract's exit function targeting an **unregistered** NEAR account (e.g., `"unregistered.near"`).
4. The `ExitToNear` precompile burns the ERC-20 tokens and schedules `ft_transfer` on the NEP-141 contract with `refund: None` in the callback.
5. `ft_transfer` fails because the recipient is not registered.
6. `exit_to_near_precompile_callback` executes, finds `args.refund == None`, and returns `Ok(None)` without re-minting.
7. Observe: the user's ERC-20 balance is reduced by `FT_EXIT_AMOUNT`, the NEP-141 balance of the recipient is 0, and Aurora's NEP-141 balance is unchanged — the tokens are permanently lost.

This is directly confirmed by the existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` which asserts `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into()` when `error_refund` is not enabled. [5](#0-4)
