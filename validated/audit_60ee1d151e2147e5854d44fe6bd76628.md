### Title
Missing Minimum Threshold Lower-Bound Check in FROST Signing and Presigning Initialization — (`src/frost/mod.rs`)

### Summary

`assert_key_invariants` (DKG) and `validate_triple_inputs` (triple generation) both enforce `threshold >= 2`. The FROST signing entry-points `assert_sign_inputs` and `presign` in `src/frost/mod.rs` enforce only the upper bound (`threshold <= participants.len()`) but **never enforce the lower bound** (`threshold >= 2`). A caller can therefore invoke `sign_v1`, `sign_v2`, or `redjubjub::sign` with `threshold = 1` (or `0`), bypassing the minimum security requirement of the threshold scheme and corrupting the signing output.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
// src/dkg.rs:573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` also enforces both bounds:

```rust
// src/ecdsa/ot_based_ecdsa/triples/generation.rs:693-704
if threshold_value > participants.len() { ... }
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
}
``` [2](#0-1) 

However, `assert_sign_inputs` — the shared validation function called by `sign_v1`, `sign_v2`, and `redjubjub::sign` — only checks the upper bound:

```rust
// src/frost/mod.rs:144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// NO lower-bound check: threshold < 2 is never rejected
``` [3](#0-2) 

Similarly, the FROST `presign` function only checks the upper bound:

```rust
// src/frost/mod.rs:71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// NO lower-bound check
``` [4](#0-3) 

When `threshold = 1` is accepted, `construct_key_package` creates a FROST `KeyPackage` with `min_signers = 1`:

```rust
// src/frost/eddsa/sign.rs:360-368
Ok(KeyPackage::new(
    identifier,
    signing_share,
    verifying_share,
    *verifying_key,
    u16::try_from(threshold.value())...,
))
``` [5](#0-4) 

With `min_signers = 1`, the FROST library's internal validation in `round2::sign` and `aggregate` is weakened to accept a signing package containing only a single commitment. The keys were generated under a polynomial of degree `threshold - 1 >= 1` (since DKG enforces `threshold >= 2`), so a single share does not reconstruct the secret. The aggregated signature is cryptographically invalid and unusable.

### Impact Explanation

Passing `threshold = 1` to `sign_v1`, `sign_v2`, or `redjubjub::sign` causes the FROST signing protocol to proceed with `min_signers = 1` in every participant's `KeyPackage`. The FROST `aggregate` call then produces a signature that fails external verification, because the secret was shared under a degree-≥1 polynomial but the aggregation treats it as a degree-0 (single-share) scheme. Honest participants complete the protocol and receive a `SignatureOption::Some(signature)` that is cryptographically invalid — a corrupted, unusable signing output. This matches the allowed High impact: **Corruption of sign outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

The entry-points `sign_v1`, `sign_v2`, and `redjubjub::sign` are public library API functions. Any unprivileged library caller controls the `threshold` argument directly. No special privilege, leaked key, or cryptographic break is required. The inconsistency is a straightforward missing bounds check, reachable on the first call.

### Recommendation

Add the same lower-bound check present in `assert_key_invariants` and `validate_triple_inputs` to `assert_sign_inputs` and the FROST `presign` function:

```rust
// In assert_sign_inputs (src/frost/mod.rs)
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same fix to the `presign` function in `src/frost/mod.rs` for the `args.threshold` field.

### Proof of Concept

1. Run DKG with `participants = [P1, P2, P3]`, `threshold = 2` — succeeds (lower-bound enforced).
2. Call `sign_v1(participants, threshold = 1, me = P1, coordinator = P1, keygen_output, message, rng)`.
3. `assert_sign_inputs` accepts `threshold = 1` (only upper-bound checked: `1 <= 3`).
4. `construct_key_package` creates `KeyPackage { min_signers: 1, ... }` for every participant.
5. The FROST `aggregate` call completes and returns `Some(signature)`.
6. External verification of `signature` against the public key fails — the output is a corrupted, unusable signature.
7. Contrast: calling `keygen(participants, threshold = 1, ...)` correctly returns `InitializationError::ThresholdTooSmall { threshold: 1, min: 2 }`. [6](#0-5) [7](#0-6)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
