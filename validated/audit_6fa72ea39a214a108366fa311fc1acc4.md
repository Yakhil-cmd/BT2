Looking at the full attack chain across the relevant files:

**Step 1 — Entry point**: `IBCConversionModule.OnRecvPacket` passes `data.Receiver` (attacker-controlled) directly to `ConvertVouchersToEvmCoins` with no validation that the receiver is not a contract/proxy address. [1](#0-0) 

**Step 2 — Address derivation**: `ConvertVouchersToEvmCoins` decodes the bech32 receiver into a cosmos `AccAddress`, then passes `common.BytesToAddress(acc.Bytes())` as the EVM `sender` to `ConvertCoinFromNativeToCRC21`. No blocked-address check, no contract-address check. [2](#0-1) 

**Step 3 — Self-transfer + mint**: For non-source IBC denoms, `ConvertCoinFromNativeToCRC21` does `SendCoins(sender_cosmos, contract_cosmos, coins)` then calls `mint_by_cronos_module(sender_evm, amount)`. When `sender_evm == contract_evm` (i.e., the receiver IS the proxy contract), this is a self-transfer followed by minting CRC20 tokens to the proxy's own EVM address. [3](#0-2) 

**Step 4 — Permanent lock**: The proxy contract's `mint_by_cronos_module` mints into the underlying CRC20 contract at `addr = address(this)`. The proxy has no function to transfer its own CRC20 balance out — `transfer_from_cronos_module` and `burn_by_cronos_module` are gated to `module_address` only, and `send_to_ibc`/`send_to_evm_chain` operate on `msg.sender`'s balance, which cannot be the proxy itself without a self-call mechanism. [4](#0-3) 

**Precondition check**: `canBeConverted` only triggers conversion for denoms with a registered contract mapping — a realistic precondition for any live IBC token registered in Cronos. [5](#0-4) 

**Blocked-address check**: The proxy contract's cosmos address is NOT a module account and is NOT in the bank module's blocked-address set, so the IBC transfer module's underlying `OnRecvPacket` successfully credits the vouchers to the proxy's cosmos address before the middleware runs. [6](#0-5)

### Title
IBC Receiver Set to Proxy Contract Address Causes Permanent CRC20 Token Lock — (`x/cronos/middleware/conversion_middleware.go`, `x/cronos/keeper/ibc.go`, `x/cronos/keeper/evm.go`)

### Summary
An unprivileged IBC sender can craft a packet with `receiver = bech32(proxy_evm_address)`. The `IBCConversionModule` middleware passes this address unchecked into `ConvertVouchersToEvmCoins`, which derives the EVM address and calls `mint_by_cronos_module(proxy_evm_address, amount)` on the proxy contract. CRC20 tokens are minted to the proxy's own EVM address, where they are permanently locked with no user-accessible release path.

### Finding Description
`IBCConversionModule.OnRecvPacket` extracts `data.Receiver` from the attacker-controlled IBC packet and passes it directly to `ConvertVouchersToEvmCoins`:

```go
// conversion_middleware.go:135
if err := im.cronoskeeper.ConvertVouchersToEvmCoins(cacheCtx, data.Receiver, sdk.NewCoins(token)); err != nil {
```

`ConvertVouchersToEvmCoins` decodes the bech32 string to `acc`, then calls:

```go
// ibc.go:60
err := k.ConvertCoinFromNativeToCRC21(ctx, common.BytesToAddress(acc.Bytes()), c, params.EnableAutoDeployment)
```

For non-source IBC denoms, `ConvertCoinFromNativeToCRC21` does:

```go
// evm.go:132-137
err = k.bankKeeper.SendCoins(ctx, sdk.AccAddress(sender.Bytes()), sdk.AccAddress(contract.Bytes()), coins)
...
_, err = k.CallModuleCRC21(ctx, contract, "mint_by_cronos_module", sender, coin.Amount.BigInt())
```

When `sender == contract` (the receiver IS the proxy contract), this is a self-transfer (no-op on balance) followed by `mint_by_cronos_module(proxy_evm_address, amount)`. The proxy's `mint_by_cronos_module` mints CRC20 tokens to `proxy_evm_address` in the underlying CRC20 contract:

```solidity
// ModuleCRC20Proxy.sol:41-44
function mint_by_cronos_module(address addr, uint amount) public {
    require(msg.sender == module_address);
    crc20Contract.mint(addr, amount);
}
```

The proxy contract has no function callable by an unprivileged user to transfer its own CRC20 balance. `transfer_from_cronos_module` and `burn_by_cronos_module` require `msg.sender == module_address`. `send_to_ibc` and `send_to_evm_chain` operate on `msg.sender`'s balance, which cannot be the proxy itself without a self-call mechanism that does not exist.

### Impact Explanation
- CRC21 tokens are minted to the proxy contract's own EVM address and are permanently inaccessible to any user.
- The IBC escrow on the counterparty chain holds tokens that can never be released (the corresponding CRC21 tokens cannot be burned/transferred back through the IBC module).
- The CRC21 total supply is inflated by the locked amount, breaking the supply invariant.
- Recovery requires explicit admin/governance intervention to call `transfer_from_cronos_module` or `burn_by_cronos_module` via the cronos module address — there is no automatic or user-accessible path.

This fits: **Critical — unauthorized accounting change for CRC21/IBC voucher assets** and **High — corruption of IBC channel/accounting state with direct security impact**.

### Likelihood Explanation
Any user on a counterparty IBC chain can initiate a transfer to `bech32(proxy_evm_address)`. No special privileges are required. The attack is repeatable and affects any denom that has a registered contract mapping in Cronos (`canBeConverted` returns `true`). The proxy contract's EVM address is publicly visible on-chain.

### Recommendation
1. In `ConvertVouchersToEvmCoins` (or in `OnRecvPacket` before calling it), validate that the receiver address is not a known contract address — specifically, reject any receiver whose EVM address matches a registered contract in the denom-to-contract mapping.
2. Alternatively, add a check in `ConvertCoinFromNativeToCRC21` to reject calls where `sender == contract`.
3. Consider adding the proxy contract's cosmos address to the bank module's blocked-address set, or implementing a general "no IBC receive to contract addresses" guard in the middleware.

### Proof of Concept
1. Register denom `ibc/XXXX` with a `ModuleCRC20Proxy` contract at EVM address `0xABCD...`.
2. Compute the proxy's cosmos address: `bech32_encode("crc", 0xABCD...)` → `crc1...`.
3. From a counterparty chain, send an IBC transfer: `receiver = "crc1..."`, `denom = "ibc/XXXX"`, `amount = 100`.
4. Observe `OnRecvPacket` succeeds (proxy cosmos address is not blocked).
5. Observe middleware calls `ConvertVouchersToEvmCoins("crc1...", [ibc/XXXX: 100])`.
6. Observe `mint_by_cronos_module(0xABCD..., 100)` is called on the proxy contract.
7. Query `crc20Contract.balanceOf(0xABCD...)` — returns 100.
8. Attempt any user-callable function to recover the tokens — none exist.
9. Verify the IBC escrow on the counterparty chain still holds the original tokens, permanently locked.

	acc, err := sdk.AccAddressFromBech32(from)
	if err != nil {
		return err
	}

	params := k.GetParams(ctx)
	evmParams := k.GetEvmParams(ctx)
	for _, c := range coins {
		switch c.Denom {
		case params.IbcCroDenom:
			if params.IbcCroDenom == "" {
				return errorsmod.Wrap(types.ErrIbcCroDenomEmpty, "ibc is disabled")
			}

			// Send ibc tokens to escrow address
			err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, acc, types.ModuleName, sdk.NewCoins(c))
			if err != nil {
				return err
			}

			// a 18 decimals token
			amount18dec := sdk.NewCoin(evmParams.EvmDenom, c.Amount.Mul(sdkmath.NewIntFromBigInt(types.TenPowTen)))

			); err != nil {
				return err
			}

			if err := k.bankKeeper.SendCoinsFromModuleToAccount(
				ctx, types.ModuleName, acc, sdk.NewCoins(amount18dec),
			); err != nil {
				return err
			}

		default:
			err := k.ConvertCoinFromNativeToCRC21(ctx, common.BytesToAddress(acc.Bytes()), c, params.EnableAutoDeployment)
			if err != nil {
				return err
			}
		}
