### Title
`DeltaView.CreateAccount` Overwrites Post-Destruction Balance with Stale Pre-Destruction Balance, Enabling EVM ETH Inflation — (`fvm/evm/emulator/state/delta.go`)

### Summary

`DeltaView.CreateAccount` contains a balance-accounting bug in the branch that handles recreation of a previously self-destructed account. When `HasSelfDestructed` returns `true`, the function assigns the **pre-destruction balance** (captured at destruction time in `toBeDestructed`) directly to `d.balances[addr]`, silently discarding any ETH that was added to the address between destruction and recreation. This allows an attacker to inflate the EVM total supply by the difference `B − X` in a single transaction.

---

### Finding Description

**Exact state machine trace:**

**Step 1 — `SelfDestruct(A)`** (reached via `SelfDestruct6780`, which requires `IsNewContract(A) == true`): [1](#0-0) 

- `d.toBeDestructed[A]` ← B (pre-destruction balance, snapshot)
- `d.balances[A]` ← 0 (zeroed)

**Step 2 — `AddBalance(A, X)`:** [2](#0-1) 

- `GetBalance(A)` reads `d.balances[A]` = 0
- `d.balances[A]` ← X

**Step 3 — `CreateAccount(A)`:** [3](#0-2) 

- `HasSelfDestructed(A)` returns `(true, B)` — the stale snapshot
- The `if !destructed` branch is **skipped entirely**, so `GetBalance(A)` (which would return X) is never called
- `d.balances[addr] = balance` assigns **B**, overwriting X

**Step 4 — `GetBalance(A)`** returns **B**, not X.

The root cause is that `HasSelfDestructed` returns the balance frozen at destruction time: [4](#0-3) 

…and `CreateAccount` uses that frozen value unconditionally when `destructed == true`, instead of reading the live `d.balances[addr]` which already reflects the post-destruction `AddBalance` call.

---

### Impact Explanation

For any `B > X`:

| Account | Expected | Actual (buggy) |
|---|---|---|
| Beneficiary | +B | +B (correct) |
| Address A | X | B |
| Sender of X | −X | −X (correct) |
| **Net supply delta** | **0** | **+(B − X)** |

ETH equal to `B − X` is created from nothing within the EVM. The X ETH subtracted from the sender is permanently lost (not reflected in A's balance), while A holds B ETH that was already transferred to the beneficiary. Both the beneficiary and A hold B ETH simultaneously.

---

### Likelihood Explanation

The attack is reachable through a standard EVM transaction submitted to Flow EVM. The only precondition is `IsNewContract(addr) == true`, which is satisfied for any contract created within the same transaction. The "phoenix" pattern (CREATE2 → selfdestruct → CREATE2 to same address) is a well-known EVM pattern. The attacker needs only to deploy a factory contract that orchestrates the four-step sequence entirely within one transaction. No privileged access, operator control, or key compromise is required.

The entry path is:

`EVM transaction → SelfDestruct6780 (IsNewContract check) → DeltaView.SelfDestruct → AddBalance → CreateAccount` [5](#0-4) 

---

### Recommendation

In `DeltaView.CreateAccount`, when `HasSelfDestructed` returns `true`, do **not** use the stale snapshot balance. Instead, read the current live balance via `d.GetBalance(addr)`, which correctly reflects any `AddBalance` calls made after destruction:

```go
destructed, _ := d.HasSelfDestructed(addr)
if !destructed {
    err = d.SelfDestruct(addr)
    if err != nil {
        return err
    }
}
// Always read the current balance, not the pre-destruction snapshot
balance, err := d.GetBalance(addr)
if err != nil {
    return err
}
d.balances[addr] = balance
``` [6](#0-5)

func (db *StateDB) SelfDestruct6780(addr gethCommon.Address) (uint256.Int, bool) {
	balance, err := db.latestView().GetBalance(addr)
	db.handleError(err)

	if db.IsNewContract(addr) {
		err := db.latestView().SelfDestruct(addr)
		db.handleError(err)
		return *balance, true
	}

	return *balance, false
