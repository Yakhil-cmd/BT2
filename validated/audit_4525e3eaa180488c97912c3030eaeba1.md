### Title
Gateway `skip_stateful_validations` Bypasses Signature Verification for Nonce-1 Invokes When Any Prior Tx Exists in Mempool, Admitting Unsigned Transactions from `deploy`-syscall Accounts - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to skip `__validate__` for an invoke with nonce=1 when a `deploy_account` is pending in the mempool (UX feature). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from that address, not only `deploy_account` transactions. Because the production `max_allowed_nonce_gap` is 200, any account with on-chain nonce=0 (including accounts deployed via the `deploy` syscall, which start at nonce=0) that has **any** pending transaction in the mempool becomes a target: an unprivileged attacker can submit a nonce-1 invoke with a completely invalid signature and the gateway will accept it into the mempool without ever calling `__validate__`.

### Finding Description

**Trigger condition in `skip_stateful_validations`:**

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`:

```rust

    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations`, when `validate = false`, execution returns immediately after `perform_pre_validation_stage` — no `__validate__` call is made:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 78-81
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
```

`perform_pre_validation_stage` checks nonce range, fee bounds, and balance — but **not the signature**.

**The flawed guard:**

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` for **any** transaction type from that address. The code comment claims:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or **transactions with future nonces that passed validations**."

The second branch is the flaw. A nonce-0 invoke from an account deployed via `deploy` syscall passes full validation (the account exists on-chain with a class hash and a working `__validate__`). Its presence in the pool then satisfies the guard for a subsequent nonce-1 invoke — even one with a garbage signature submitted by a third party.

**Production configuration confirms exploitability:**

```json
// crates/apollo_deployments/resources/app_configs/gateway_config.json  line 17
"gateway_config.static_config.stateful_tx_validator_config.max_allowed_nonce_gap": 200,
```

With `max_allowed_nonce_gap = 200`, the gateway's `validate_nonce` accepts tx_nonce=1 when account_nonce=0 (range `[0, 200]`). The `max_nonce_for_validation_skip` field in `StatefulTransactionValidatorConfig` is set to `0x1` but is **not read** by `skip_stateful_validations`; the function hardcodes `tx.nonce() == Nonce(Felt::ONE)`.

### Impact Explanation

The gateway admits an invoke transaction bearing an invalid (or empty) signature into the mempool without any cryptographic check. This violates the admission invariant: **every accepted transaction must have passed account-level signature validation before entering the mempool**. The corrupted value is the admission decision itself — the gateway returns `Ok(account_nonce)` for a transaction whose signer was never verified.

Downstream consequences:
- The mempool is polluted with signature-invalid transactions that the batcher must attempt to execute and then discard, wasting sequencer resources.
- An attacker can target any account with on-chain nonce=0 and a pending mempool transaction (both are publicly observable), submitting up to `max_allowed_nonce_gap` (200) unsigned nonce-1 invokes per such account per mempool cycle, creating a sustained DoS vector against the sequencer's execution pipeline.
- The nonce-1 slot for the victim account is occupied by the attacker's invalid transaction; the legitimate owner's nonce-1 invoke must win a fee-escalation race to displace it.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- Accounts deployed via the `deploy` syscall are common (any contract can call `deploy_syscall`); they start with nonce=0 and are fully functional account contracts.
- The mempool is public; an attacker can trivially observe which addresses have nonce=0 on-chain and a pending transaction.
- No privileged access is required. The attacker only needs to craft an `InvokeV3` transaction with `nonce=1`, any `sender_address` matching the target, and an arbitrary (invalid) signature.
- The production `max_allowed_nonce_gap=200` ensures the nonce check passes.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. Alternatively, before skipping `__validate__`, confirm that the sender address has no deployed class hash on-chain (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`), which is the true precondition the UX feature intends to handle. The `max_nonce_for_validation_skip` config field that already exists in `StatefulTransactionValidatorConfig` should also be wired into `skip_stateful_validations` instead of the hardcoded `Nonce(Felt::ONE)`.

### Proof of Concept

```
1. Deploy account contract C (with __validate__ / __execute__) via deploy_syscall
   from an existing funded account. C's on-chain nonce = 0.

2. Legitimate owner of C submits InvokeV3(sender=C, nonce=0, valid_sig).
   → Gateway: validate_nonce passes (0 ∈ [0,200]).
   → Gateway: run_validate_entry_point called, __validate__ runs, sig accepted.
   → Mempool: tx added. account_tx_in_pool_or_recent_block(C) now returns true.

3. Attacker submits InvokeV3(sender=C, nonce=1, sig=[0xdead, 0xbeef]).
   → Gateway validate_nonce: account_nonce=0, tx_nonce=1, max=200 → passes.
   → Gateway validate_by_mempool: no duplicate, no fee escalation → passes.
   → skip_stateful_validations: tx_nonce==1 && account_nonce==0 → checks
     account_tx_in_pool_or_recent_block(C) → true (step 2 tx is in pool).
     Returns true (skip validation).
   → run_validate_entry_point: execution_flags.validate = false.
     StatefulValidator::perform_validations returns Ok(()) after
     perform_pre_validation_stage without calling __validate__.
   → Gateway returns Ok(account_nonce=0). Invalid tx admitted to mempool.

4. Batcher dequeues the attacker's tx, blockifier calls __validate__,
   signature check fails → tx rejected, fee not charged, but sequencer
   resources consumed. Attacker repeats at scale across all nonce-0 accounts
   with pending mempool transactions.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
