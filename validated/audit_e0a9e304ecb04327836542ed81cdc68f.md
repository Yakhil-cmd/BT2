### Title
Malicious Participant Equivocates Coefficient Commitments via Degenerate Echo-Broadcast, Producing Inconsistent DKG Outputs — (`src/dkg.rs::broadcast_success`, `src/protocol/echo_broadcast.rs::echo_ready_thresholds`)

---

### Summary

A single malicious participant in a 3-party DKG/reshare/refresh session can send different `VerifiableSecretSharingCommitment` values to each honest party. The echo-broadcast protocol explicitly sets both thresholds to zero for `n ≤ 3`, meaning each honest party delivers whatever value it first echoed — with no cross-party consistency check. Because `broadcast_success` only verifies a shared `session_id` and a boolean success flag (not the derived `verifying_key`), both honest parties complete the protocol successfully while holding different public keys and mutually inconsistent signing shares.

---

### Finding Description

**Root cause 1 — degenerate echo-broadcast thresholds for n ≤ 3** [1](#0-0) 

For `n = 3`, `echo_ready_thresholds` returns `(echo_t=0, ready_t=0)`. The delivery condition is `data_ready.get(&data) > 2 * ready_t`, i.e., `> 0`, meaning a single ready vote (the party's own simulated vote) is sufficient to deliver. A malicious sender M can send `commitment_A` to honest party A and `commitment_B` to honest party B:

- A echoes `commitment_A`, immediately accumulates 1 echo (`> 0 = echo_t`), sends `Ready(commitment_A)`, accumulates 1 ready vote (`> 0 = 2*ready_t`), and **delivers `commitment_A`**.
- B echoes `commitment_B`, follows the same path, and **delivers `commitment_B`**.

The cross-echoes (A's echo of `commitment_A` arriving at B, and vice versa) are ignored because `finish_echo = true` is set before they are processed. [2](#0-1) 

**Root cause 2 — commitment hash is sent point-to-point, not via echo-broadcast**

In Round 2, M sends `commitment_hash_A = H(M, commitment_A, session_id)` to A and `commitment_hash_B = H(M, commitment_B, session_id)` to B via `chan.send_many` (plain unicast, no consistency guarantee). [3](#0-2) 

M then broadcasts `(commitment_A, proof_A)` to A and `(commitment_B, proof_B)` to B in Round 3. Each honest party's `verify_commitment_hash` check passes because M crafted the hashes to match the respective commitments. [4](#0-3) 

**Root cause 3 — `broadcast_success` does not bind the derived `verifying_key`**

After all share verifications pass locally, `broadcast_success` is called. It only broadcasts and checks `(true, session_id)`: [5](#0-4) 

The `verifying_key` computed by each honest party is never included in this final consistency broadcast. Both A and B broadcast `(true, session_id)` — the same `session_id` because it was derived from the Round 1 echo-broadcast of random nonces, which is unaffected by the equivocation — and both return `Ok(KeygenOutput { ... })` with different `public_key` values.

**Root cause 4 — no API-level enforcement of `n ≥ 3f+1`**

`assert_key_invariants` permits `n = 3, threshold = 2` without any warning or rejection, even though the echo-broadcast protocol explicitly documents that it assumes zero malicious parties for `n ≤ 3`. [6](#0-5) 

---

### Impact Explanation

After the attack completes:

- Honest party A holds `KeygenOutput { private_share: f_A(A) + p_A(A), public_key: f_A(0)·G + a_0·G }`.
- Honest party B holds `KeygenOutput { private_share: f_A(B) + p_B(B), public_key: f_A(0)·G + b_0·G }`.

If M chooses `a_0 ≠ b_0`, A and B hold **different public keys** — a direct corruption of the DKG transcript. If M chooses `a_0 = b_0` but different higher-degree coefficients, A and B hold the **same public key** but shares that are inconsistent with any single underlying secret, permanently preventing valid threshold signatures. Both outcomes match the allowed High impact: *"Corruption of DKG, reshare, or refresh outputs so honest parties accept inconsistent public keys or unusable cryptographic outputs."*

---

### Likelihood Explanation

The attack requires only `n = 3` participants (the smallest realistic group), a threshold of 2, and one malicious participant who controls their own protocol messages. No cryptographic assumptions are broken. The malicious party simply sends two different but individually valid `(commitment, proof, commitment_hash)` tuples — one to each honest party. The library's public API (`keygen`, `reshare`, `refresh`) accepts this configuration without restriction.

---

### Recommendation

1. **Enforce `n ≥ 4` at the API level**, or document explicitly that `n = 3` provides no security against a single malicious participant and reject it in `assert_key_invariants`.
2. **Include the `verifying_key` in `broadcast_success`**: replace the `(bool, session_id)` payload with `(bool, session_id, verifying_key_hash)` so that honest parties abort if their derived public keys differ.
3. **Send the commitment hash via echo-broadcast** (not plain `send_many`) so that M cannot give A and B different pre-images for the same commitment slot.

### Proof of Concept

```
Participants: A (honest), B (honest), M (malicious), n=3, threshold=2

Setup:
  M generates two polynomials:
    p_A(x) = a_0 + a_1·x   (commitment_A = [a_0·G, a_1·G])
    p_B(x) = b_0 + b_1·x   (commitment_B = [b_0·G, b_1·G])  where a_0 ≠ b_0

Round 1 (session ID broadcast — honest):
  All parties agree on session_id.

Round 2 (commitment hash — point-to-point):
  M → A: commitment_hash_A = H(M, commitment_A, session_id)
  M → B: commitment_hash_B = H(M, commitment_B, session_id)

Round 3 (echo-broadcast of commitments):
  M → A: Send(commitment_A, proof_A)   [valid PoK for commitment_A]
  M → B: Send(commitment_B, proof_B)   [valid PoK for commitment_B]

  echo_ready_thresholds(3) = (0, 0)
  A: echoes commitment_A → 1 echo > 0 → Ready(commitment_A) → 1 ready > 0 → delivers commitment_A ✓
  B: echoes commitment_B → 1 echo > 0 → Ready(commitment_B) → 1 ready > 0 → delivers commitment_B ✓

Round 4 (verification — each passes locally):
  A: verify_commitment_hash(commitment_A, commitment_hash_A) → OK
  A: verify_proof_of_knowledge(commitment_A, proof_A)        → OK
  B: verify_commitment_hash(commitment_B, commitment_hash_B) → OK
  B: verify_proof_of_knowledge(commitment_B, proof_B)        → OK

  A computes verifying_key_A = f_A(0)·G + a_0·G
  B computes verifying_key_B = f_A(0)·G + b_0·G   ← DIFFERENT

Round 5 (share exchange):
  M → A: p_A(A)  [validates against commitment_A → OK]
  M → B: p_B(B)  [validates against commitment_B → OK]

broadcast_success:
  A broadcasts (true, session_id) → all agree → OK
  B broadcasts (true, session_id) → all agree → OK
  verifying_key is never checked.

Result:
  A returns KeygenOutput { public_key: verifying_key_A, ... }
  B returns KeygenOutput { public_key: verifying_key_B, ... }
  verifying_key_A ≠ verifying_key_B  →  DKG output is corrupted.
```

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}

async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                broadcast the wrong session id. Aborting Protocol!"
                .to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
}

pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
