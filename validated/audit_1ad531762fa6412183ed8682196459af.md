### Title
Stale Staker Entries Never Removed from `stakers` Vec Causes Unbounded Gas Growth in `get_stakers` - (File: `src/staking/staking.cairo`)

### Summary
`unstake_action` in `staking.cairo` never removes the staker's address from the `stakers: Vec<ContractAddress>` storage vector. `get_stakers()` iterates over the full range of this vector on every call. An unprivileged attacker can inflate the vector without bound by repeatedly staking with fresh addresses and then unstaking (recovering capital each cycle), making `get_stakers()` progressively more expensive until it exceeds the block gas limit and permanently breaks the consensus validator-set query.

### Finding Description

`stake()` appends every new staker address to the `stakers` vector: [1](#0-0) 

The storage field is explicitly annotated as never shrinking: [2](#0-1) 

`unstake_action()` calls `remove_staker` and clears pool data, but never touches the `stakers` vector: [3](#0-2) 

`get_stakers()` iterates over `self.stakers.into_iter_full_range()` — the complete, ever-growing vector — performing at least one storage read per entry plus an `is_staker_active` check: [4](#0-3) 

The `DiverseStakerVecFlow` integration test confirms that a staker who has fully completed `unstake_action` remains in the vector at index 3 alongside active stakers: [5](#0-4) 

### Impact Explanation

`get_stakers` is the consensus-layer entry point that returns the validator set for a given epoch: [6](#0-5) 

As the `stakers` vector grows, every call to `get_stakers` must iterate over all historical staker addresses — including those whose `staker_info` was deleted — performing storage reads for each. On Starknet, view functions are subject to gas limits. Once the vector is large enough, `get_stakers` will exceed the gas limit and revert on every call, permanently preventing the consensus layer from reading the validator set. This constitutes **unbounded gas consumption** leading to **griefing with damage to the protocol** (Medium impact).

### Likelihood Explanation

The attack requires no sustained capital lock-up. An attacker:
1. Stakes with address A (min stake, currently 20,000 STRK).
2. Calls `unstake_intent()`, waits for the exit window.
3. Calls `unstake_action()` — full capital returned, address A permanently in `stakers`.
4. Transfers funds to fresh address B and repeats.

`assert_staker_address_not_reused` prevents re-staking from the same address, so each cycle requires a fresh address, but the capital is recycled. The only cost is gas per cycle. The attack is permissionless and callable by any unprivileged address. [7](#0-6) 

### Recommendation

Remove the staker address from the `stakers` vector inside `remove_staker` (or `unstake_action`) when the staker record is deleted. Because `Vec` on Starknet does not support O(1) removal, a common pattern is to swap the target entry with the last element and then pop, or to maintain a separate `Map<ContractAddress, bool>` of active stakers and skip deleted entries during iteration. Either approach bounds the effective iteration length to the number of currently active stakers.

### Proof of Concept

```
// Attacker script (pseudocode):
for i in 0..N:
    addr = fresh_address(i)
    fund(addr, min_stake)
    staking.stake(from=addr, ...)          // stakers.len() += 1
    staking.unstake_intent(from=addr)
    advance_time(exit_wait_window)
    staking.unstake_action(addr)           // funds returned; addr stays in stakers
    transfer_funds(addr -> next_addr)

// After N iterations:
// stakers.len() == N (all stale)
// get_stakers(epoch_id) iterates N entries → OOG revert
staking.get_stakers(epoch_id)              // reverts: out of gas
```

The `stakers` vector length is observable on-chain. Each completed `unstake_action` leaves a permanent stale entry. Once the vector length exceeds the gas budget for `get_stakers`, the consensus layer cannot retrieve the validator set.
