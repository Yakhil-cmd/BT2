### Title
Pool admin collects spread fees at a rate exceeding `maxAdminSpreadFeeE6` after owner reduces the cap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolAdminFees` collects accrued fees using the **currently stored** `poolFeeConfig[pool].adminSpreadFeeE6` before updating it. `setFeeCaps` never retroactively clamps per-pool admin fee configs. A pool admin can therefore trigger a fee collection at the old (higher) rate by calling `setPoolAdminFees(pool, 0, 0)` after the owner has reduced `maxAdminSpreadFeeE6`, bypassing the cap entirely.

### Finding Description
`setFeeCaps` updates `maxAdminSpreadFeeE6` globally but only auto-clamps the protocol fee **defaults** (`spreadProtocolFeeE6`, `protocolNotionalFeeE8`). Per-pool `poolFeeConfig[pool].adminSpreadFeeE6` is left unchanged.

`setPoolAdminFees` validates only the **new** value against the cap (`newAdminSpreadFeeE6 > maxAdminSpreadFeeE6`), then immediately calls `collectFees` with the **old stored** `c.adminSpreadFeeE6` — which may now exceed `maxAdminSpreadFeeE6`. The stored value is updated to the new (compliant) value only after collection.

The owner's only retroactive remedy is `setPoolProtocolFee`, which does clamp the admin fee before collecting. But the admin can race ahead of that call.

### Impact Explanation
The admin receives a share of the pool surplus proportional to the old `adminSpreadFeeE6` (e.g. 200000 = 20%) instead of the cap-reduced value (e.g. 0). This is a direct transfer of pool tokens to the admin's fee destination that exceeds what the current cap permits — a concrete admin-boundary break with direct fund loss from the pool.

### Likelihood Explanation
Requires: (1) owner reduces `maxAdminSpreadFeeE6` via `setFeeCaps`, (2) pool has nonzero surplus, (3) admin calls `setPoolAdminFees` before the owner calls `setPoolProtocolFee` to retroactively clamp. Steps 1 and 3 are both single transactions available to their respective actors; the window is real and exploitable in the same block if the admin front-runs the owner's follow-up.

### Recommendation
In `setPoolAdminFees`, clamp `c.adminSpreadFeeE6` to `maxAdminSpreadFeeE6` (and `c.adminNotionalFeeE8` to `maxAdminNotionalFeeE8`) **before** passing them to `collectFees`, mirroring the logic already present in `setPoolProtocolFee` (lines 337–346).

// Foundry test sketch
function test_adminCollectsAboveCap() public {
    // 1. Create pool with adminSpreadFeeE6 = 200_000
    PoolParameters memory params = _defaultPoolParams();
    params.adminSpreadFeeE6 = 200_000;
    address pool = factory.createPool(params);

    // 2. Add surplus (e.g. donate tokens directly to pool)
    token0.mint(pool, 1e18);

    // 3. Owner reduces cap to 0
    factory.setFeeCaps(200_000, 0, 1_000_000, 1_000_000);

    // 4. Admin calls setPoolAdminFees(pool, 0, 0)
    //    Cap check: 0 > 0 = false → passes
    //    collectFees called with adminSpreadFeeE6 = 200_000 (old stored value)
    uint256 adminBalBefore = token0.balanceOf(adminFeeDestination);
    vm.prank(admin);
    factory.setPoolAdminFees(pool, 0, 0);
    uint256 adminBalAfter = token0.balanceOf(adminFeeDestination);

    // Admin received tokens at 200_000 rate despite cap = 0
    assertGt(adminBalAfter, adminBalBefore);

    // Stored config is now 0 — but the damage is done
    (, uint24 storedAdmin,,) = factory.poolFeeConfig(pool);
    assertEq(storedAdmin, 0);
}
