### Title
`new_settlement_layer_chain_id_storage` Not Rolled Back on Frame Revert — (`basic_system/src/system_implementation/system/io_subsystem.rs`)

### Summary

`FullIOStateSnapshot` omits a snapshot field for `new_settlement_layer_chain_id_storage`. When `finish_io_frame` is called with a rollback handle, every other sub-storage is reverted, but `new_settlement_layer_chain_id_storage` is silently left in its post-write state. This is a direct analog of the external report's pattern: one state component is modified during an operation, but the cancellation/rollback path fails to restore it.

### Finding Description

`FullIO` holds six rollbackable sub-storages:

| Sub-storage | Snapshotted in `FullIOStateSnapshot`? | Rolled back in `finish_io_frame`? |
|---|---|---|
| `storage` | ✅ (`io` field) | ✅ |
| `transient_storage` | ✅ (`transient` field) | ✅ |
| `logs_storage` | ✅ (`messages` field) | ✅ |
| `events_storage` | ✅ (`events` field) | ✅ |
| `interop_root_storage` | ✅ (`interop_roots` field) | ✅ |
| **`new_settlement_layer_chain_id_storage`** | ❌ **missing** | ❌ **missing** |

`FullIOStateSnapshot` is defined as:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    // ← no field for new_settlement_layer_chain_id_storage
}
``` [1](#0-0) 

`start_io_frame` takes a snapshot of every sub-storage **except** `new_settlement_layer_chain_id_storage`:

```rust

`finish_io_frame` rolls back every sub-storage **except** `new_settlement_layer_chain_id_storage`:

```rust

``` [3](#0-2) 

`NewSettlementLayerChainIdStorage` **does** implement `start_frame` / `finish_frame` with full rollback support via `HistoryCounter`: [4](#0-3) 

The write path is triggered by the `system_context_event_hook` whenever a `SettlementLayerChainIdUpdated` event is emitted from the `SystemContext` contract: [5](#0-4) 

This calls `update_settlement_layer_chain_id` on `FullIO`, which writes to `new_settlement_layer_chain_id_storage`: [6](#0-5) 

`NewSettlementLayerChainIdStorage::update` also enforces a **once-per-block** invariant — it returns an internal error if called a second time: [7](#0-6) 

### Impact Explanation

**Scenario:** A service transaction emits `SettlementLayerChainIdUpdated`, which triggers the hook and writes to `new_settlement_layer_chain_id_storage`. The transaction then reverts (e.g., runs out of native resources, or the block-limit check in the ZK tx loop forces a rollback via `finish_global_frame(Some(&pre_tx_rollback_handle))`). [8](#0-7) 

After the rollback, all other IO state is correctly restored, but `new_settlement_layer_chain_id_storage` still holds the value written by the reverted transaction. This causes two concrete problems:

1. **Stale chain-ID value persists into block finalization.** `read_batch_context_inputs` reads `new_settlement_layer_chain_id_storage.value()` and asserts it matches the on-chain storage slot. If the transaction that wrote it was reverted, the on-chain slot was also rolled back, so the assertion `assert_eq!(new_settlement_layer_chain_id, &settlement_layer_chain_id)` will **panic**, halting block processing. [9](#0-8) 

2. **Once-per-block guard is permanently consumed.** Because the write was not rolled back, `value()` returns `Some(...)` for the rest of the block. Any subsequent legitimate service transaction that tries to update the settlement layer chain ID will hit the "Tried to update settlement layer chain id more than once in a block" internal error, making the update impossible for the remainder of the block. [10](#0-9) 

The net result is either a block-processing panic (forward/proving divergence) or a denial of the settlement-layer chain-ID update mechanism — both are state-transition correctness failures reachable by an unprivileged transaction sender who can cause a transaction revert after the hook fires.

### Likelihood Explanation

The trigger requires a transaction that (a) emits `SettlementLayerChainIdUpdated` from `SystemContext` and (b) subsequently reverts. The revert can be forced by any of the existing revert paths: out-of-native, out-of-ergs, or the block-limit check. A user who can submit a service transaction (or influence one) and craft it to run out of resources after the event emission can trigger this. The block-limit revert path in the ZK tx loop is particularly relevant because it fires after a fully-processed transaction result, meaning the hook has already run.

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and include it in both `start_io_frame` and `finish_io_frame`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    new_sl_chain_id: NewSettlementLayerChainIdSnapshotId, // add this
}
```

In `start_io_frame`:
```rust
let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();
Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots, new_sl_chain_id })
```

In `finish_io_frame`:
```rust
self.new_settlement_layer_chain_id_storage
    .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id));
```

### Proof of Concept

1. Deploy a `SystemContext`-like contract at the expected address that emits `SettlementLayerChainIdUpdated(newId)` and then consumes all remaining gas (e.g., via an infinite loop or a very expensive operation).
2. Submit a service transaction calling this contract with a gas limit just sufficient to emit the event but not to complete execution.
3. The transaction reverts due to out-of-native/ergs.
4. Observe that `new_settlement_layer_chain_id_storage.value()` is `Some(newId)` despite the revert.
5. Submit a second legitimate service transaction attempting to update the settlement layer chain ID — it fails with the "more than once" internal error, or block finalization panics at the `assert_eq!` in `read_batch_context_inputs`.
