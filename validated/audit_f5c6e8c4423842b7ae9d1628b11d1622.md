### Title
Gateway Admits Transactions Signed for a Foreign Chain ID When `skip_stateful_validations` Bypasses `__validate__` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `StatelessTransactionValidator` performs no chain ID check whatsoever. Chain ID correctness is enforced entirely by the account contract's `__validate__` entry point during stateful validation. However, `skip_stateful_validations` can suppress that entry-point call for invoke transactions with `nonce == 1` when the account's on-chain nonce is still `0` and a `deploy_account` transaction exists in the mempool. In that window, a transaction whose signature was produced over a different chain's hash is admitted to the mempool without any chain-of-origin verification — a direct structural analog to the missing `_fromChain` validation in the external report.

---

### Finding Description

**Stateless path — no chain ID field exists or is checked.**

`StatelessTransactionValidator::validate` checks contract address, empty `account_deployment_data`/`paymaster_data`, resource bounds, calldata/signature/proof sizes, DA modes, Sierra version, and class size. It holds no reference to the node's `chain_id` and performs no chain-of-origin check. [1](#0-0) 

**Conversion path — hash is recomputed with the node's chain ID, but the signature is never verified here.**

`TransactionConverter::convert_rpc_tx_to_internal` computes `tx_hash` using `self.chain_id`. It stores the user-supplied `signature` verbatim without checking it against the hash. [2](#0-1) 

**Stateful path — chain ID is validated only implicitly, via `__validate__`.**

`run_validate_entry_point` calls the blockifier's `StatefulValidator::validate`, which invokes the account contract's `__validate__` entry point. That entry point verifies the ECDSA signature over the Poseidon hash that includes `chain_id`. This is the sole chain-of-origin guard. [3](#0-2) 

**The bypass — `skip_stateful_validations` suppresses `__validate__` entirely.**

When an invoke transaction has `nonce == 1`, the account's on-chain nonce is `0`, and `account_tx_in_pool_or_recent_block` returns `true`, `skip_stateful_validations` returns `true`. [4](#0-3) 

`run_pre_validation_checks` propagates this flag: [5](#0-4) 

`run_validate_entry_point` then sets `execution_flags.validate = false`: [6](#0-5) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns `Ok(())` before calling `__validate__`: [7](#0-6) 

`perform_pre_validation_stage` (which still runs) only checks nonce, fee bounds, and proof facts — it does not verify the signature or the chain ID.

**Result:** a transaction whose signature was produced over `H(tx_fields, chain_id_A)` is accepted by a node running `chain_id_B`, because the only check that would catch the mismatch (`__validate__`) is skipped.

---

### Impact Explanation

An attacker who observes a `deploy_account` transaction in the mempool for address `A` can replay any invoke transaction (nonce=1) that was originally signed for a different Starknet network (e.g., Sepolia → Mainnet) targeting the same address. The gateway admits it to the mempool without signature or chain-of-origin verification. The transaction will revert during batcher execution (because the batcher runs `__validate__` with `validate = true`), but it has already been admitted, consuming mempool capacity and potentially displacing the legitimate post-deploy invoke.

This matches the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."**

---

### Likelihood Explanation

The preconditions are:
1. An account with a pending `deploy_account` in the mempool (nonce=0 on-chain) — observable by any mempool participant.
2. An invoke transaction for that account with nonce=1 signed for a different chain ID — trivially constructable by replaying a transaction from Sepolia or any other Starknet network.

Both conditions are reachable by an unprivileged external actor with read access to the mempool P2P layer.

---

### Recommendation

Add an explicit chain ID check in the stateless or early stateful path so that it cannot be bypassed by the `skip_validate` UX shortcut. One approach: in `StatelessTransactionValidator`, accept the node's `chain_id` and verify that the transaction hash recomputed with that chain ID matches the hash the user would have signed (i.e., reject if the signature is structurally inconsistent with the local chain). Alternatively, ensure that `skip_stateful_validations` only suppresses the nonce/fee re-check, not the signature/chain-ID check, by separating the two concerns in `run_validate_entry_point`.

---

### Proof of Concept

```
Precondition:
  - Starknet Mainnet node (chain_id = SN_MAIN) is running.
  - Account address 0xABCD has a pending deploy_account tx in the Mainnet mempool
    (on-chain nonce = 0).

Attack:
  1. Attacker takes any valid invoke tx for 0xABCD with nonce=1 that was signed
     for Sepolia (chain_id = SN_SEPOLIA).
  2. Attacker submits it to the Mainnet gateway via POST /add_transaction.

Gateway path:
  - StatelessTransactionValidator::validate() → passes (no chain_id field checked).
  - TransactionConverter::convert_rpc_tx_to_internal() → computes tx_hash with
    SN_MAIN; stores the Sepolia signature verbatim.
  - StatefulTransactionValidator:
      account_nonce = 0, tx.nonce() = 1
      → skip_stateful_validations() returns true
      → run_validate_entry_point() sets execution_flags.validate = false
      → StatefulValidator::perform_validations() returns Ok(()) without calling
        __validate__
  - mempool_client.add_tx() is called → tx admitted to Mainnet mempool.

Outcome:
  The Sepolia-signed transaction is now in the Mainnet mempool.
  It will revert during batcher execution (wrong signature), but it has
  consumed a mempool slot and may displace the legitimate nonce-1 invoke
  that the account owner intended to send.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
