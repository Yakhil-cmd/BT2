### Title
Hardcoded `PERIOD_MINT_CAP` With No Admin Override Can Permanently Freeze Reward Minting - (File: `L1/starkware/solidity/stake/PeriodMintLimit.sol`)

---

### Summary

`PERIOD_MINT_CAP` is a file-level Solidity `constant` set to 6.5 M STRK per week with no admin setter. As the protocol grows and weekly reward demand exceeds this cap, `tick()` in `RewardSupplier.sol` will revert with `"EXCEED_PERIOD_MINTING"`, permanently freezing a growing portion of unclaimed staker yield with no recovery path short of a full L1 contract upgrade.

---

### Finding Description

In `PeriodMintLimit.sol`, `PERIOD_MINT_CAP` and `MINTING_PERIOD_DURATION` are declared as immutable file-level constants:

```solidity
uint256 constant PERIOD_MINT_CAP = 6_500_000 * 10**18;
uint256 constant MINTING_PERIOD_DURATION = 1 weeks;
```

`checkAndUpdatePeriodicalQuota` enforces the cap with a hard revert:

```solidity
require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
``` [1](#0-0) 

`MintManager.mintRequest()` calls `checkAndUpdatePeriodicalQuota` before minting:

```solidity
checkAndUpdatePeriodicalQuota(token, amount);
mintingAllowance(token)[requester] -= amount;
mintableToken(token).mint(requester, amount);
``` [2](#0-1) 

`RewardSupplier.tick()` (callable by any public address, no access control) calls `mintManager().mintRequest(token(), amountToMint)`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        ...
        mintManager().mintRequest(token(), amountToMint);
``` [3](#0-2) 

The system's three hardcoded parameters are arithmetically coupled:

| Constant | Value |
|---|---|
| `TOKENS_PER_MINT_REQUEST` | 1,300,000 STRK |
| `MAX_MESSAGES_TO_PROCESS_PER_TICK` | 5 |
| `PERIOD_MINT_CAP` | 6,500,000 STRK (= 5 × 1.3 M) | [4](#0-3) 

This means a single `tick()` call can exhaust the entire weekly cap. If the L2 reward supplier has queued **more than 5** pending mint-request messages in a given week (because weekly rewards exceed 6.5 M STRK), the first `tick()` consumes all 5 messages and hits the cap. Every subsequent `tick()` call in that same week reverts. The excess messages cannot be processed until the next weekly period resets, and the backlog grows indefinitely.

The L2 side has no setter for `base_mint_amount` either — it is written once in the constructor and never updated:

```cairo
self.base_mint_amount.write(base_mint_amount);
``` [5](#0-4) 

There is no `set_base_mint_amount` function anywhere in the L2 reward supplier interface, confirming the parameter is frozen post-deployment. [6](#0-5) 

---

### Impact Explanation

Once weekly reward demand exceeds 6.5 M STRK, every `tick()` call beyond the first in a given week reverts. The L2 `unclaimed_rewards` counter keeps growing (stakers are owed rewards), but the L1 cannot mint and bridge the tokens to cover them. The shortfall compounds week over week — this is a **permanent, growing freeze of unclaimed staker and delegator yield** with no on-chain recovery path. The only fix is a full L1 proxy upgrade to redeploy `MintManager` and `PeriodMintLimit` with a higher cap.

Impact classification: **High — Permanent freezing of unclaimed yield.**

---

### Likelihood Explanation

With current parameters (C_NUM = 213/10,000, ~10 B STRK total supply, ~1 B STRK staked), weekly rewards are well below 6.5 M STRK. However:

1. The minting curve scales with `sqrt(total_stake × total_supply)`. As both grow, weekly rewards grow super-linearly.
2. `PERIOD_MINT_CAP` is a compile-time constant — it cannot be raised without a contract upgrade, which requires governance coordination and delay.
3. There is no monitoring or early-warning mechanism in the contracts; the failure mode is a silent revert of `tick()`.

Likelihood: **Low today, but structurally certain as the protocol scales** — identical to the original report's scenario where a large BOO accumulation or sandwich attack could exceed `MAX_SLIPPAGE`.

---

### Recommendation

Mirror the accepted fix from the original report: convert `PERIOD_MINT_CAP` from a file-level `constant` to a named-storage variable in `PeriodMintLimit`, and expose a privileged setter (e.g., `onlyAppGovernor`) in `MintManager`. Similarly, expose a `set_base_mint_amount` function in the L2 `RewardSupplier` (gated to `app_governor`) so both sides can be updated atomically when the protocol's reward volume grows.

---

### Proof of Concept

1. Protocol matures: weekly STRK rewards grow beyond 6.5 M STRK.
2. L2 `request_funds` sends **N > 5** L2→L1 messages, each with payload `base_mint_amount = 1.3 M STRK`. [7](#0-6) 
3. Any public caller invokes `tick()` on L1. `requiredMinting()` finds `min(N, 5) = 5` matching messages; `amountToMint = 6.5 M STRK`. [8](#0-7) 
4. `mintManager().mintRequest(token, 6_500_000e18)` calls `checkAndUpdatePeriodicalQuota`, which sets `mintedThisPeriod = 6.5 M STRK` — exactly at the cap. Minting succeeds.
5. `N - 5` messages remain unconsumed. Any caller invokes `tick()` again in the same week. `amountToMint = min(N-5, 5) × 1.3 M STRK > 0`. `checkAndUpdatePeriodicalQuota` computes `mintedThisPeriodAfter = 6.5 M + amount > PERIOD_MINT_CAP` → **reverts with `"EXCEED_PERIOD_MINTING"`**. [9](#0-8) 
6. The remaining messages cannot be consumed until the next weekly period. Meanwhile, L2 `unclaimed_rewards` keeps growing each epoch, and new mint-request messages are queued. The backlog grows without bound, permanently freezing the excess unclaimed yield.

abstract contract PeriodMintLimit {
    // Named storage slot tags.
    string internal constant PERIODICAL_MINT_ACCOUNTING_TAG = "PERIODICAL_MINT_ACCOUNTING_SLOT_TAG";

    function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
        bytes32 periodSlot = periodAccountingSlot(token);
        uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
        uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
        require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
        periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;

    function requiredMinting() public view returns (uint256, uint256) {
        uint256[] memory messagePayload = new uint256[](1);
        messagePayload[0] = TOKENS_PER_MINT_REQUEST;

        bytes32 msgHash = l2ToL1MsgHash(mintRequestSource(), address(this), messagePayload);
        // Limit the number of msgs to consume to limit.
        uint256 numMsgsToConsume = Math.min(
            messagingContract().l2ToL1Messages(msgHash),
            MAX_MESSAGES_TO_PROCESS_PER_TICK
        );

        return (TOKENS_PER_MINT_REQUEST * numMsgsToConsume, numMsgsToConsume);
    }
