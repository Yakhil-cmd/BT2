### Title
Stale `eligible_operational_addresses` Entry Allows Staker to Reclaim Operational Address Without Fresh Consent - (File: src/staking/staking.cairo)

---

### Summary

The `eligible_operational_addresses` mapping is keyed only by `operational_address` and is never cleared when a staker changes their operational address. This is the direct analog to the external report's stale `clientCache` keyed only by `chainId`: a mapping that does not account for a changing identity (the staker). A staker can exploit the stale entry to reclaim a previously-used operational address without the operational address re-declaring consent, bypassing the two-step authorization mechanism.

---

### Finding Description

The protocol enforces a two-step process for changing an operational address:
1. The operational address calls `declare_operational_address(staker_address)`, writing `eligible_operational_addresses[operational_address] = staker_address`.
2. The staker calls `change_operational_address(operational_address)`, which checks `eligible_operational_addresses[operational_address] == staker_address` before proceeding.

This two-step design exists to ensure the operational address **consents** to being bound to a staker.

However, `change_operational_address` only clears `operational_address_to_staker_address[old_address]` — it never clears `eligible_operational_addresses[old_address]`:

```cairo
// src/staking/staking.cairo ~L689-694
let old_address = staker_info.operational_address;
self.operational_address_to_staker_address.write(old_address, Zero::zero()); // cleared
staker_info.operational_address = operational_address;
self.write_staker_info(:staker_address, :staker_info);
self.operational_address_to_staker_address.write(operational_address, staker_address);
// eligible_operational_addresses[old_address] is NEVER cleared ← stale entry
```

Similarly, `remove_staker` clears `operational_address_to_staker_address` but not `eligible_operational_addresses`:

```cairo
// src/staking/staking.cairo ~L1694-1695
let operational_address = staker_info.operational_address;
self.operational_address_to_staker_address.write(operational_address, Zero::zero()); // cleared
// eligible_operational_addresses[operational_address] is NEVER cleared ← stale entry
```

This is structurally identical to the external report's bug: a mapping keyed by one identifier (`operational_address`) that does not account for the changing identity (`staker_address`), leaving a stale authorization entry that can be replayed.

---

### Impact Explanation

**Medium — Griefing with damage to users.**

A staker can lock an operational address to themselves without that address's fresh consent:

- Operational address X is forced back into service for staker A without re-declaring.
- X cannot call `declare_operational_address` for a different staker while it is in use (`OPERATIONAL_IN_USE` check at line 709).
- X is effectively held hostage: it must attest for staker A or staker A loses rewards, and X cannot freely associate with another staker until staker A voluntarily releases it.
- No financial profit accrues to the attacker; the harm is the unauthorized lockup of X's operational identity and denial of X's ability to serve another staker.

---

### Likelihood Explanation

**Low-Medium.** The conditions required are:
1. Staker A previously used operational address X (via `declare_operational_address` + `change_operational_address`).
2. Staker A later changed to a different operational address Y, freeing X.
3. X has not re-declared for any other staker (which would overwrite the stale entry).
4. Staker A calls `change_operational_address(X)` again.

All four conditions are reachable by an unprivileged staker with no special access. The scenario is realistic in a live protocol where stakers rotate operational keys.

---

### Recommendation

In `change_operational_address`, clear the stale `eligible_operational_addresses` entry for the old address:

```cairo
// After freeing old_address:
self.operational_address_to_staker_address.write(old_address, Zero::zero());
self.eligible_operational_addresses.write(old_address, Zero::zero()); // add this
```

Similarly, clear it in `remove_staker`:

```cairo
self.operational_address_to_staker_address.write(operational_address, Zero::zero());
self.eligible_operational_addresses.write(operational_address, Zero::zero()); // add this
```

### Proof of Concept

```
1. X calls declare_operational_address(staker_A)
   → eligible_operational_addresses[X] = staker_A

2. Staker A calls change_operational_address(X)
   → operational_address_to_staker_address[X] = staker_A
   → eligible_operational_addresses[X] = staker_A  (unchanged)

3. Y calls declare_operational_address(staker_A)
   → eligible_operational_addresses[Y] = staker_A

4. Staker A calls change_operational_address(Y)
   → operational_address_to_staker_address[X] = 0  (X freed)
   → operational_address_to_staker_address[Y] = staker_A
   → eligible_operational_addresses[X] = staker_A  (STALE — never cleared)

5. X has NOT re-declared for anyone.

6. Staker A calls change_operational_address(X):
   - operational_address_to_staker_address[X].is_zero() → true  ✓
   - eligible_operational_addresses[X] == staker_A → true (stale!) ✓
   → SUCCEEDS: X is now staker A's operational address again
     without X's fresh consent.

7. X is now locked to staker A. X cannot call declare_operational_address
   for staker B because OPERATIONAL_IN_USE check fires (line 709).
   X is held until staker A voluntarily releases it.
```

        fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
            self.general_prerequisites();
            let operational_address = get_caller_address();
            assert!(
                self.operational_address_to_staker_address.read(operational_address).is_zero(),
                "{}",
                Error::OPERATIONAL_IN_USE,
            );
            assert!(
                !self.does_token_exist(token_address: operational_address),
                "{}",
                Error::OPERATIONAL_IS_TOKEN,
            );
            if self.eligible_operational_addresses.read(operational_address) == staker_address {
                return;
            }
            self.eligible_operational_addresses.write(operational_address, staker_address);
            self.emit(Events::OperationalAddressDeclared { operational_address, staker_address });
        }
