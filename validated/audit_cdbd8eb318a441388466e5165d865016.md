### Title
Circular `account_tx_in_pool_or_recent_block` check in `skip_stateful_validations` allows perpetual `__validate__` bypass for invoke(nonce=1) after deploy_account eviction - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 when the account is not yet deployed, as a UX convenience for the deploy_account + invoke flow. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the account in the pool, not specifically a deploy_account. Once an invoke(nonce=1) is admitted with skipped validation (because a deploy_account was in the pool), it can itself serve as the basis for admitting further invoke(nonce=1) transactions with skipped validation, even after the original deploy_account has been evicted. This breaks the invariant the check is meant to enforce and allows an attacker to perpetually bypass `__validate__` (signature verification) for invoke(nonce=1) transactions targeting an undeployed account.

### Finding Description

`skip_stateful_validations` returns `true` (skip `__validate__`) when:

1. The transaction is an Invoke with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` if the account has **any** transaction in the pool — including a previously-admitted invoke(nonce=1) whose own `__validate__` was skipped. The code comment acknowledges this: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* However, this reasoning is circular: those future-nonce transactions "passed validations" only because a deploy_account was in the pool at the time of their admission. If the deploy_account is subsequently evicted, the future-nonce transactions are orphaned, yet they continue to satisfy the `account_tx_in_pool_or_recent_block` check for new submissions. [3](#0-2) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without calling `__validate__`: [4](#0-3) [5](#0-4) 

The mempool does not track the relationship between a deploy_account and subsequent invoke transactions; evicting the deploy_account does not evict the associated invoke(nonce=1).

### Impact Explanation

An attacker can admit invoke(nonce=1) transactions with **arbitrary or invalid signatures** for an undeployed account into the mempool, bypassing the account's `__validate__` entry point at the gateway layer. These transactions will fail (revert) during block execution when the blockifier calls `__validate__` with `execution_flags.validate = true`, but they:

- Pollute the mempool with invalid transactions
- Consume sequencer resources (execution, state reads, fee accounting for reverted transactions)
- Can be sustained indefinitely via fee escalation (if enabled) or TTL cycling

This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The attack requires:
1. An initial valid deploy_account submission (to bootstrap the pool presence)
2. The deploy_account to be evicted (via TTL expiry, capacity overflow, or fee-escalation replacement)
3. Fee escalation enabled (or waiting for TTL) to replace the orphaned invoke(nonce=1) with a new one

These conditions are reachable by any unprivileged user. The `transaction_ttl` is configurable (default 60 seconds in tests), and capacity overflow is achievable on a loaded network.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists in the pool for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects only `DeployAccount`-typed transactions. This eliminates the circular dependency and restores the intended invariant.

### Proof of Concept

```
1. Attacker generates a fresh keypair → address A (not deployed, on-chain nonce = 0).

2. Attacker submits a valid deploy_account tx for A (signed correctly).
   → mempool: {deploy_account(A, nonce=0)}
   → account_tx_in_pool_or_recent_block(A) = true

3. Attacker submits invoke(A, nonce=1, signature=INVALID).
   Gateway stateful path:
     - account_nonce = 0  ✓
     - tx.nonce() == 1    ✓
     - account_tx_in_pool_or_recent_block(A) = true  ✓
     → skip_stateful_validations returns true
     → __validate__ NOT called
     → invoke admitted to mempool
   → mempool: {deploy_account(A, nonce=0), invoke(A, nonce=1, sig=INVALID)}

4. Attacker waits for deploy_account TTL to expire (or triggers capacity eviction).
   → mempool: {invoke(A, nonce=1, sig=INVALID)}
   → account_tx_in_pool_or_recent_block(A) = true  (invoke still in pool)
   → on-chain nonce of A still = 0

5. Attacker submits invoke(A, nonce=1, tip=higher, signature=INVALID_2).
   Gateway stateful path:
     - account_nonce = 0  ✓
     - tx.nonce() == 1    ✓
     - account_tx_in_pool_or_recent_block(A) = true  ✓  ← satisfied by orphaned invoke
     → skip_stateful_validations returns true
     → __validate__ NOT called
     → new invoke admitted; old invoke replaced via fee escalation
   → mempool: {invoke(A, nonce=1, sig=INVALID_2)}

6. Repeat step 5 indefinitely with increasing tips.
   The attacker can submit arbitrary-content invoke(nonce=1) transactions for
   undeployed account A, all bypassing __validate__, with no deploy_account
   ever present in the pool.

7. When the batcher executes any of these transactions, __validate__ is called
   with execution_flags.validate=true, the invalid signature causes a revert,
   and the transaction is included as a failed transaction.
``` [6](#0-5) [2](#0-1)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
