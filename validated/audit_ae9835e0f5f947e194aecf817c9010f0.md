### Title
Staker Can Immediately Increase Commission to 100% via `set_commission_commitment` + `set_commission`, Stealing All Delegator Yield During Mandatory Exit Window - (File: src/staking/staking.cairo)

---

### Summary

A staker can use `set_commission_commitment` followed immediately by `set_commission` to raise their pool commission from any value to 100% in two consecutive transactions (or even the same block). Because the commission change takes effect instantly and delegators must wait through a mandatory exit window (minimum 1 week) before recovering their funds, all rewards earned during that window are diverted entirely to the staker. The developers themselves acknowledge this gap with an inline comment: *"Current commission increase safeguards still allow for sudden commission changes."*

---

### Finding Description

Without a `CommissionCommitment`, `set_commission` only allows commission to be **decreased**. However, `set_commission_commitment` lets a staker set a `max_commission` ceiling (up to 10 000 = 100%) that, while active, permits commission to be raised to any value at or below that ceiling.

**`set_commission_commitment` validation** (`src/staking/staking.cairo`):

```cairo
let current_commission = staker_pool_info.commission();
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
```

The only constraints are that `max_commission >= current_commission` and `expiration_epoch > current_epoch`. There is **no minimum delay** before the commitment can be used. The commitment becomes active the moment it is written.

**`is_commission_commitment_active`** (`src/staking/staking.cairo`):

```cairo
fn is_commission_commitment_active(
    self: @ContractState, commission_commitment: CommissionCommitment,
) -> bool {
    self.get_current_epoch() < commission_commitment.expiration_epoch
}
```

Because `expiration_epoch > current_epoch` is required, the commitment is **immediately active** after `set_commission_commitment` returns.

**`update_commission` path when commitment is active** (`src/staking/staking.cairo`):

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission,
        "{}",
        Error::INVALID_COMMISSION_WITH_COMMITMENT,
    );
    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
}
```

With an active commitment, the only checks are that the new commission does not exceed `max_commission` and is not identical to the old one. A staker with commission = 500 (5%) can therefore:

1. Call `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)` — commitment is immediately active.
2. Call `set_commission(10000)` — commission jumps to 100% in the same block.

The commission is written directly to storage with no epoch-delay:

```cairo
// Update commission in storage.
staker_pool_info.commission.write(Option::Some(commission));
```

The pool contract reads commission live from the staking contract on every reward distribution:

```cairo
fn get_commission_from_staking_contract(self: @ContractState) -> Commission {
    ...
    staking_dispatcher
        .staker_pool_info(staker_address: self.staker_address.read())
        .commission
        .unwrap()
}
```

So the very next reward distribution after the change sends 100% of pool rewards to the staker and 0% to delegators.

Delegators who wish to exit must first call `exit_delegation_pool_intent` and then wait the full `exit_wait_window` (default 1 week, up to 12 weeks) before calling `exit_delegation_pool_action`. During this entire window, every reward epoch yields nothing for them.

---

### Impact Explanation

**Theft of unclaimed yield.** Delegators lose all rewards accrued from the moment the commission is raised to 100% until they complete the mandatory exit process. With a default exit window of one week and typical staking yields, this represents a material, quantifiable loss of yield that flows directly to the malicious staker. The delegators' principal is eventually recoverable, but the yield earned during the exit window is permanently diverted.

---

### Likelihood Explanation

Any staker who controls a delegation pool can execute this attack unilaterally. No external dependency, privileged role beyond being a registered staker, or bridge compromise is required. The attack is two transactions and is profitable whenever the pool holds meaningful delegated stake. On Starknet, both transactions can be included in the same block, making mempool-based front-running of a delegator's `enter_delegation_pool` call straightforward. The developers' own inline note — *"Current commission increase safeguards still allow for sudden commission changes"* — confirms the path is reachable and unmitigated.

---

### Recommendation

1. **Enforce a delay before a new commitment can be used to increase commission.** For example, require that `set_commission` with a value above the old commission can only be called at least `K` epochs after `set_commission_commitment` was set, giving delegators time to observe and exit.
2. **Allow delegators to specify a maximum acceptable commission** when calling `enter_delegation_pool`, analogous to slippage protection in AMMs.
3. **Emit a prominent on-chain event and enforce a notice period** (e.g., one epoch) before a commission increase takes effect, so delegators can react before rewards are affected.

---

### Proof of Concept

```
State: Staker S has pool P with commission = 500 (5%).
       Delegator D has 1 000 000 STRK delegated to P.

Block N:
  Tx 1 (Staker S): set_commission_commitment(max_commission=10000, expiration_epoch=current_epoch+1)
       → CommissionCommitment { max_commission: 10000, expiration_epoch: E+1 } stored.
       → Commitment is immediately active (current_epoch E < E+1).

  Tx 2 (Staker S): set_commission(10000)
       → update_commission: 10000 <= 10000 ✓, 10000 != 500 ✓
       → commission written as 10000 (100%) immediately.

Block N+1 onward (reward distributions):
  → get_commission_from_staking_contract() returns 10000.
  → split_rewards_with_commission: pool_rewards = total_rewards * (10000 - 10000) / 10000 = 0.
  → Delegator D receives 0 rewards every epoch.

Delegator D calls exit_delegation_pool_intent(amount = 1_000_000).
  → Must wait exit_wait_window (≥ 1 week = ~52 epochs at default settings).
  → During all 52 epochs, D earns 0 STRK in rewards.
  → Staker S captures 100% of pool rewards for the entire exit window.

After exit_wait_window:
  Delegator D calls exit_delegation_pool_action → recovers principal only.
  All yield earned during the exit window is permanently lost to D and captured by S.
```

```text
        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );
```
