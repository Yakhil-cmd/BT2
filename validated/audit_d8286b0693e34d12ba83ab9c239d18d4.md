### Title
Missing Zero Address Validation on `reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `change_reward_address` and `enter_delegation_pool` functions in `pool.cairo`, and `change_reward_address` and `stake` in `staking.cairo`, accept a `reward_address` parameter without validating it against the zero address. If a pool member or staker sets their `reward_address` to `0`, all subsequent reward claims are transferred to the zero address and permanently lost.

### Finding Description
In `pool.cairo`, `change_reward_address` (line 505) only asserts that `reward_address` is not the token contract address, but performs no zero-address check: [1](#0-0) 

Similarly, `enter_delegation_pool` (line 182) only checks `token_address != reward_address`: [2](#0-1) 

In `staking.cairo`, `change_reward_address` (line 517) and `stake` (line 288) both only check that `reward_address` is not a registered token address, with no zero-address guard: [3](#0-2) [4](#0-3) 

When `claim_rewards` is called in `pool.cairo`, the reward transfer goes directly to whatever `reward_address` is stored, including zero: [5](#0-4) 

The codebase already defines `GenericError::ZERO_ADDRESS` and `InternalError::TOKEN_IS_ZERO_ADDRESS`, confirming developer awareness of zero-address risks, but these are not applied to `reward_address` inputs: [6](#0-5) [7](#0-6) 

### Impact Explanation
A pool member who calls `change_reward_address(0)` (or enters with `reward_address = 0`) will have all future reward claims permanently transferred to the zero address. The tokens are irrecoverably burned. This matches the **"Permanent freezing of unclaimed yield"** impact category. The same applies to a staker calling `staking.cairo`'s `change_reward_address(0)`.

### Likelihood Explanation
Any unprivileged pool member or staker can trigger this by passing zero as `reward_address`. While the most common scenario is accidental user error (e.g., passing an uninitialized address), the lack of a guard means the protocol offers no protection. The call path is fully permissionless and requires no special role.

### Recommendation
Add a zero-address assertion in all functions that accept a `reward_address` parameter before writing it to storage. The `GenericError::ZERO_ADDRESS` error variant already exists and should be used:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This check should be added to:
- `pool.cairo`: `enter_delegation_pool` and `change_reward_address`
- `staking.cairo`: `stake` and `change_reward_address`

### Proof of Concept
1. A pool member calls `pool.change_reward_address(reward_address: 0)`.
2. No assertion fires; `pool_member_info.reward_address` is written as `0`.
3. The pool member (or anyone) calls `pool.claim_rewards(pool_member)`.
4. `claim_rewards` computes accrued rewards and executes:
   ```cairo
   reward_token.checked_transfer(recipient: Zero::zero(), amount: rewards.into());
   ``` [8](#0-7) 
5. All accrued STRK rewards are transferred to the zero address and permanently lost. The pool member's `reward_checkpoint` is updated, so the rewards cannot be reclaimed.
