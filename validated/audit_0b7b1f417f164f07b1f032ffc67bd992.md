### Title
Missing Minimum Threshold Lower-Bound in FROST Signing Validation Allows Malicious Coordinator to Corrupt Sign Outputs — (File: `src/frost/mod.rs`)

### Summary
`assert_sign_inputs` and `presign` in `src/frost/mod.rs` do not enforce `threshold >= 2`, unlike `assert_key_invariants` in `src/dkg.rs`. A malicious coordinator can invoke the signing protocol with `threshold=1`, causing Lagrange interpolation to use a coefficient of 1 for the sole signing participant. The resulting aggregate signature uses the raw Shamir share `f(x_i)` instead of the reconstructed secret key `f(0)`, producing a cryptographically invalid and unusable signature that honest parties cannot distinguish from a legitimate protocol output until external verification fails.

### Finding Description
`assert_key_invariants` in `src/dkg.rs` enforces a hard lower bound:

```rust
// src/dkg.rs lines 580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The signing-side validator `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound and participant presence — it never checks `threshold >= 2`:

```rust
// src/frost/mod.rs lines 144-149
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [4](#0-3) 

Likewise, `presign` only checks the upper bound: [3](#0-2) 

A malicious coordinator therefore passes `threshold=1` to the signing protocol. With a signing set of size 1 (participant `i`), the Lagrange coefficient at 0 collapses to the empty product:

```
L_i(0) = ∏_{j ≠ i} (0 - x_j)/(x_i - x_j)  →  1   (empty product)
```

The partial signature becomes:

```
s_i = nonce_i + 1 · share_i · challenge
```

instead of the correct:

```
s_i = nonce_i + L_i(0) · share_i · challenge
```

The aggregate signature therefore encodes `f(x_i)` (the raw Shamir share) rather than `f(0)` (the secret key). Because DKG required `threshold >= 2`, the polynomial has degree ≥ 1 and `f(x_i) ≠ f(0)` in general, so the signature is cryptographically invalid. The protocol completes without returning an error, and honest participants have no in-protocol mechanism to detect the corruption before the signature reaches an external verifier.

### Impact Explanation
**High — Corruption of sign outputs.** The signing protocol terminates successfully but emits an unusable cryptographic output (an invalid signature). This matches the allowed impact: *"Corruption of … sign … outputs so honest parties accept … unusable cryptographic outputs."* Every honest participant who contributed a partial signature believes the protocol succeeded; the corruption is silent and only discovered at external verification time, potentially after the signing session's nonces have been consumed and cannot be reused.

### Likelihood Explanation
**Medium.** The coordinator role is a documented participant in the FROST protocol. A malicious coordinator — or a library caller who deliberately or accidentally supplies `threshold=1` — can trigger this path with a single API call. No cryptographic capability or key material is required beyond participation in the signing session.

### Recommendation
Add the same lower-bound guard that `assert_key_invariants` already applies, in both `assert_sign_inputs` and `presign`:

```rust
// src/frost/mod.rs — assert_sign_inputs
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the identical check inside `presign` immediately after the upper-bound check. This mirrors the existing protection in `src/dkg.rs` and closes the asymmetry between key-generation and signing validation. [5](#0-4) 

### Proof of Concept
1. Run DKG with participants `[P1, P2, P3]` and `threshold=2` — succeeds, polynomial has degree 1.
2. Malicious coordinator calls `assert_sign_inputs(&[P1, P2], threshold=1, me=P1, coordinator=P1)`.
3. All checks pass: `participants.len()=2 >= 2`, `threshold=1 <= 2`, `P1` present. Returns `Ok`.
4. Coordinator initiates signing with `threshold=1`, selecting only `P1`.
5. `P1` computes `s_1 = nonce_1 + 1 · share_1 · challenge` (Lagrange coefficient = 1).
6. Coordinator aggregates: `s = s_1`.
7. Verification against the public key `f(0)·G` fails because `s` encodes `f(x_1)`, not `f(0)`.
8. The signing session is silently corrupted; nonces are spent and cannot be reused.

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
