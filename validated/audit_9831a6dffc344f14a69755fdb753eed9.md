### Title
Missing `collectFees()` call in `setPoolAdminFeeDestination` causes accrued admin fees to be misdirected to new destination — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees()` correctly calls `collectFees()` before updating fee parameters to settle accrued fees at the old destination. `setPoolAdminFeeDestination()` changes the admin fee destination without first calling `collectFees()`, causing all previously accrued fees to be sent to the new destination instead of the old one when fees are eventually collected.

---

### Finding Description

In `MetricOmmPoolFactory.sol`, two pool-admin functions manage fee configuration asymmetrically:

**`setPoolAdminFees()`** (lines 408–435): calls `collectFees()` with the current `poolAdminFeeDestination[pool]` **before** updating fee rates, ensuring fees accrued under old rates are settled to the correct destination. [1](#0-0) 

**`setPoolAdminFeeDestination()`** (lines 437–447): updates `poolAdminFeeDestination[pool]` to `newAdminFeeDestination` **without** calling `collectFees()` first. [2](#0-1) 

After `setPoolAdminFeeDestination()` executes, the next invocation of `collectFees()` — whether via `collectPoolFees()`, `setPoolAdminFees()`, or `setPoolProtocolFee()` — passes `poolAdminFeeDestination[pool]` (now the new address) as the `adminFeeDestination` argument. This causes **all** accrued fees, including those that accumulated before the destination change, to be sent to the new destination. [3](#0-2) 

The two categories of accrued fees that are misdirected:

- **Spread fees**: computed at collection time as the surplus of tokens in the pool above what is tracked in bins (`balance × scaleMultiplier − totalScaledTokenInBins`). There is no per-destination snapshot; the entire surplus is split at collection time using the destination stored in `poolAdminFeeDestination[pool]`.
- **Notional fees**: tracked in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` and zeroed only inside `collectFees()`. [4](#0-3) 

The `collectFees()` interface confirms that `adminFeeDestination` is a caller-supplied parameter, not a pool-stored value, so the factory's `poolAdminFeeDestination[pool]` mapping is the sole source of truth at collection time. [5](#0-4) 

---

### Impact Explanation

**Medium.** The old admin fee destination loses all fees that accrued before the destination change. These fees are instead sent to the new destination. For a pool with significant trading volume, this represents a direct, permanent, and irreversible loss of owed protocol/admin fee assets. The loss is proportional to the volume processed since the last `collectFees()` call and can be material for active pools.

---

### Likelihood Explanation

**Medium.** Pool admins legitimately change fee destinations (treasury migrations, multisig upgrades, DAO address rotations). The missing `collectFees()` call is non-obvious, and the asymmetry with `setPoolAdminFees()` — which does collect first — makes this a natural oversight. Any pool admin who changes their fee destination without manually calling `collectPoolFees()` first will trigger the loss.

---

### Recommendation

Add a `collectFees()` call at the start of `setPoolAdminFeeDestination()`, mirroring the pattern in `setPoolAdminFees()`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Settle accrued fees to the OLD destination before overwriting it
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool has been running with `poolAdminFeeDestination[pool] = oldTreasury` for some time, accumulating spread-fee surplus tokens and notional-fee accumulators.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)`.
3. `poolAdminFeeDestination[pool]` is overwritten to `newTreasury`; no `collectFees()` is triggered.
4. Any caller invokes `collectPoolFees(pool)` (permissionless).
5. `collectFees()` is called with `adminFeeDestination = newTreasury`.
6. All accrued fees — including those from before step 2 — are transferred to `newTreasury`.
7. `oldTreasury` receives nothing; the fees owed to it are permanently lost.
