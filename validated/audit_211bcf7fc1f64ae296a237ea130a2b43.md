### Title
Gateway `skip_stateful_validations` Bypasses Signature Verification for Nonce-1 Invoke When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip the blockifier `__validate__` entry-point call for the deploy-account + invoke UX pattern. Its guard condition calls `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from the sender address — not only a `DeployAccount` transaction. As a result, any already-deployed account that has a valid nonce-0 invoke sitting in the mempool can have a subsequent nonce-1 invoke with a completely invalid signature admitted to the mempool without signature verification.

---

### Finding Description

`skip_stateful_validations` (lines 429–461 of `stateful_transaction_validator.rs`) fires when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain nonce is 0).

When all three hold, the function calls `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` and, if that returns `true`, returns `true` itself — causing `run_validate_entry_point` to set `validate = false` and skip the blockifier `__validate__` call entirely. [1](#0-0) 

The mempool implementation of `account_tx_in_pool_or_recent_block` checks whether the account has **any** transaction in the pool or any committed block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It does **not** filter for `DeployAccount` transactions. The inline comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but the second branch of that disjunction is the flaw: a nonce-0 `Invoke` that passed full validation (including `__validate__`) is sufficient to make the check return `true`, even though it provides no assurance that a subsequent nonce-1 `Invoke` from the same sender carries a valid signature.

When `skip_validate = true` is returned, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }` and calls `blockifier_validator.validate(account_tx)` with validation disabled, so the account's `__validate__` entry point — and therefore its signature check — is never executed at the gateway. ```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
