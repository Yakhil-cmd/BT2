### Title
Stale Deploy-Account Presence in Mempool Bypasses `__validate__` for Invoke Transactions — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the account's `__validate__` entry-point call for an invoke transaction (nonce = 1) whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender. The check only confirms that *some* transaction from that address is currently in the mempool or was recently committed — it does not confirm that the pending deploy-account will actually succeed. An attacker can therefore submit a deploy-account that is destined to revert (e.g., a constructor that always reverts) together with an invoke carrying an invalid signature; the gateway admits the invoke without ever calling `__validate__`, and the invalid transaction enters the mempool.

### Finding Description

**Invariant:** Every invoke transaction admitted to the mempool must have passed the account's `__validate__` entry point, which is the sole on-chain mechanism for signature verification.

**Where the invariant breaks:**

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–460):

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `validate: false` in `ExecutionFlags`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations` (lines 79–81 of `crates/blockifier/src/blockifier/stateful_validator.rs`):

```rust
if !tx.execution_flags.validate {
    return Ok(());   // __validate__ is never called
}
```

The gateway therefore admits the invoke without any signature check.

**`account_tx_in_pool_or_recent_block` does not distinguish a healthy deploy-account from one that will revert:**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)   // staged or committed nonces
        || self.tx_pool.contains_account(account_address)  // any tx in pool
}
```

The pool check (`tx_pool.contains_account`) returns `true` as long as the deploy-account is present in the pool — regardless of whether it will succeed or revert during batcher execution.

**Attack flow:**

1. Attacker pre-computes a contract address using a class whose constructor always reverts.
2. Attacker submits a `deploy_account` (nonce = 0) with valid resource bounds and a valid signature — it passes all gateway checks and enters the mempool.
3. Attacker immediately submits an `invoke` (nonce = 1) with an **invalid signature** from the same address.
4. Gateway: `account_nonce = 0`, `tx_nonce = 1`, `account_tx_in_pool_or_recent_block = true` → `skip_stateful_validations` returns `true` → `__validate__` is **not called** → invoke is admitted to the mempool.
5. Batcher executes the deploy-account → constructor reverts → account is never deployed.
6. Batcher attempts to execute the invoke → fails (account does not exist / invalid signature) → no fee charged because the account has no balance.
7. The attacker repeats with a fresh address, paying only the (minimal) fee for the reverted deploy-account each time.

### Impact Explanation

The gateway's admission invariant — that every invoke must pass `__validate__` — is broken. Invoke transactions with arbitrary invalid signatures are admitted to the mempool. The batcher must process and reject them, wasting execution resources. Because the account is never deployed, no fee is charged for the invalid invoke, making this a low-cost mempool-pollution vector. This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack requires only two concurrent RPC calls (one `deploy_account`, one `invoke`) and a contract class whose constructor reverts. No privileged access is needed. The attacker controls the contract class, the salt, and the invoke payload. The window between the deploy-account entering the pool and the invoke arriving at the gateway is not a race condition — the attacker controls both submissions and can sequence them deterministically.

### Recommendation

Replace the heuristic presence check with a check that is robust to deploy-account failure:

1. **Preferred:** Do not skip `__validate__` at the gateway. Instead, run `__validate__` against a speculative state that includes the pending deploy-account's constructor side-effects. This is the only way to guarantee the signature is valid before admission.

2. **Minimal fix:** After the deploy-account is rejected by the batcher (reported via `commit_block` with `rejected_tx_hashes`), immediately evict all pending invoke transactions from the same address that were admitted under the `skip_validate` path. This mirrors the oDAO recommendation: remove the stale contribution when the underlying participant is removed.

3. **Partial mitigation:** Track, per mempool entry, whether the invoke was admitted via the `skip_validate` path. On `commit_block`, if the deploy-account hash appears in `rejected_tx_hashes`, purge the associated invoke.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
