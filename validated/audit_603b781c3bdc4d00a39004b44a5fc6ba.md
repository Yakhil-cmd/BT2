### Title
`unchecked_next()` Block Number Advance Creates One-Block Window Where Valid Proof Transactions Are Rejected — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, `crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

`run_validate_entry_point` advances the block number by one (`unchecked_next()`) before building the `BlockContext` used for validation. This shifts the `max_allowed` proof block number forward by one relative to the actual committed state. The state passed to `validate_proof_block_hash` is still rooted at the latest committed block N, which does not yet contain the hash for block `N - STORED_BLOCK_HASH_BUFFER + 1`. Any legitimate invoke V3 transaction carrying a proof for exactly that block number passes the block-number range check but then fails the hash check with "Block hash mismatch" because `get_storage_at` returns `Felt::ZERO`.

---

### Finding Description

**Step 1 — Block number inflation in `run_validate_entry_point`.** [1](#0-0) 

The gateway reads the latest committed block info and immediately calls `.unchecked_next()`, so the `BlockContext` used for the entire pre-validation stage (including `validate_proof_facts`) has `block_number = N + 1`, where N is the latest finalized block.

**Step 2 — `validate_proof_block_number` computes `max_allowed` against the inflated number.** [2](#0-1) 

With `current_block_number = N + 1` and `STORED_BLOCK_HASH_BUFFER = 10`:

```
max_allowed = (N + 1) - 10 = N - 9
```

A proof with `proof_block_number = N - 9` satisfies `proof_block_number <= max_allowed` and passes this check.

**Step 3 — `validate_proof_block_hash` reads from state at block N.** [3](#0-2) 

The state object is the `CachedState` wrapping the state reader obtained from the latest committed block N. It has never had `pre_process_block` called for block `N + 1`.

**Step 4 — `pre_process_block` stores the hash for `next_block_number - STORED_BLOCK_HASH_BUFFER`.** [4](#0-3) 

When block `N + 1` is eventually processed, `pre_process_block` will write the hash for block `(N + 1) - 10 = N - 9` into the block hash contract. But at gateway validation time the state is at block N, so that write has not happened. The block hash contract only contains entries for blocks `0 … N - 10`.

**Step 5 — The mismatch.**

`get_storage_at(block_hash_contract, N - 9)` returns `Felt::ZERO` (uninitialized storage). The user's proof carries the real, non-zero hash for block `N - 9`. The comparison `Felt::ZERO != proof_block_hash` triggers the "Block hash mismatch" error, and the transaction is rejected at the gateway before it ever reaches the mempool.

The comment in `validate_proof_block_number` itself states the invariant that is violated:

> *"Stored block hashes are guaranteed only up to: current − STORED\_BLOCK\_HASH\_BUFFER."*

With `current = N + 1` the guarantee claims coverage up to `N - 9`, but the actual state only covers up to `N - 10`.

---

### Impact Explanation

Any user who generates a valid SNOS proof for block `N - STORED_BLOCK_HASH_BUFFER + 1` (= `N - 9`) and submits an invoke V3 transaction to the gateway will have that transaction rejected with a spurious "Block hash mismatch" error. The transaction is valid and would be accepted by the batcher once block `N + 1` is processed, but the gateway permanently rejects it. The user must regenerate a proof for an older block or wait until the chain advances, at which point the window shifts and the same problem recurs for the new `N - 9`.

Impact category: **High — Mempool/gateway admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The off-by-one window is always open: for every value of N there is exactly one block number (`N - 9`) that passes the range check but whose hash is absent from the gateway state. Any client that generates a proof for the most-recently-eligible block (a natural choice to minimize proof staleness) will hit this rejection deterministically.

---

### Recommendation

Align the block number used in `validate_proof_block_number` with the actual committed state. Either:

1. **Do not advance the block number for proof-facts validation.** Pass `block_info.block_number` (= N) to `validate_proof_block_number` so `max_allowed = N - 10`, which matches the hashes actually present in state.
2. **Or pre-populate the state.** Before running `validate_proof_facts`, call `pre_process_block` on the cached state with the `N - 9` hash so the state is consistent with the inflated block number. This mirrors what the batcher does before executing block `N + 1`.

Option 1 is simpler and lower-risk.

// Block pre-processing.
// Writes the hash of the (current_block_number - N) block under its block number in the dedicated
// contract state, where N=STORED_BLOCK_HASH_BUFFER.
// NOTE: This function must remain idempotent since full nodes can call it for an already updated
// block hash table.
pub fn pre_process_block(
    state: &mut dyn State,
    old_block_number_and_hash: Option<BlockHashAndNumber>,
    next_block_number: BlockNumber,
    os_constants: &OsConstants,
) -> StateResult<()> {
    let should_block_hash_be_provided =
        next_block_number >= BlockNumber(constants::STORED_BLOCK_HASH_BUFFER);
    if let Some(BlockHashAndNumber { number, hash }) = old_block_number_and_hash {
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();
        let block_number_as_storage_key = StorageKey::from(number.0);
        state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
    } else if should_block_hash_be_provided {
        return Err(StateError::OldBlockHashNotProvided);
    }
