### Title
Missing Caller Access Control on `update_rewards` Allows Any Address to Suppress Block Rewards — (File: `src/staking/staking.cairo`)

---

### Summary

The spec mandates that `update_rewards` is callable by **"Only starkware sequencer"**, but the production implementation enforces no such check. Any unprivileged address can call it with `disable_rewards = true`, consuming the global `last_reward_block` slot for that block and permanently discarding that block's rewards for the targeted staker.

---

### Finding Description

The spec at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The actual implementation in `StakingRewardsManagerImpl::update_rewards` calls only `general_prerequisites()`, which checks: [2](#0-1) 

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

There is **no check that `get_caller_address() == sequencer`**. Any non-zero address passes.

The function then checks a **single global** `last_reward_block` (not per-staker): [3](#0-2) 

If the block number is new, it immediately writes the current block to `last_reward_block`: [4](#0-3) 

```cairo
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
```

Once `last_reward_block` is set to the current block, **every subsequent call in that block** — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`.

---

### Impact Explanation

An attacker who calls `update_rewards(victim_staker, disable_rewards: true)` in a block:

1. Passes all guards (non-zero caller, staker active, staker has balance).
2. Writes `last_reward_block = current_block`.
3. Returns early — **zero rewards distributed**.
4. The sequencer's legitimate call later in the same block fails with `REWARDS_ALREADY_UPDATED`.

That block's rewards are **permanently lost** — they are never credited to `unclaimed_rewards_own` and never transferred to pools. Repeating this every block starves the staker indefinitely.

This matches: **High — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

On Starknet the sequencer controls intra-block transaction ordering, so it can in principle always place its own `update_rewards` call first. However:

- The missing access control is a direct, concrete deviation from the spec.
- Any block in which the sequencer does **not** proactively call `update_rewards` (e.g., no-op blocks, sequencer downtime, or future protocol changes) is exploitable.
- The attack requires no funds, no privileged role, and no special setup — only a valid `staker_address` and a single public call.

---

### Recommendation

Add a sequencer-only guard at the top of `update_rewards`, consistent with the spec:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
}
```

Alternatively, restrict via an existing role (e.g., `OPERATOR_ROLE`) if a sequencer address registry is not yet present.

---

### Proof of Concept

```
Block N:
  Attacker tx:   update_rewards(victim_staker, disable_rewards=true)
                 → last_reward_block = N, returns early, 0 rewards
  Sequencer tx:  update_rewards(victim_staker, disable_rewards=false)
                 → assert!(N > N) fails → REWARDS_ALREADY_UPDATED

Repeat every block → victim accumulates zero rewards indefinitely.
```

The flow test at `src/flow_test/test.cairo` lines 2785–2788 already demonstrates that `disable_rewards=true` produces zero rewards: [5](#0-4) 

and the unit test at `src/staking/tests/test.cairo` lines 3879–3884 confirms that after any successful `update_rewards` call, all further calls in the same block revert: [6](#0-5)
