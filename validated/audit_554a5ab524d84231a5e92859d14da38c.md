### Title
Unattributable Deserialization Failure Enables Permanent Protocol Denial-of-Service - (File: src/protocol/internal.rs)

### Summary

A malicious participant can send a structurally valid (correctly routed) but semantically malformed MessagePack payload during any protocol round. When `rmp_serde` fails to deserialize the payload, the resulting `ProtocolError` is propagated **without the sender's identity**, making it impossible for honest parties to identify and exclude the culprit. This mirrors the Solidity `try-catch` / ABI-decode split: the routing layer succeeds (analogous to the external call succeeding), but the decode step fails outside the attribution boundary.

---

### Finding Description

In `src/protocol/internal.rs`, the `Comms::recv` function retrieves a message and its sender, then attempts deserialization:

```rust
// src/protocol/internal.rs lines 330-341
async fn recv<T: DeserializeOwned>(
    &self,
    header: MessageHeader,
) -> Result<(Participant, T), ProtocolError> {
    let (from, data) = self.incoming.pop(header).await;   // sender identity known here
    let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| {
        ProtocolError::DeserializationError("Failed to deserialize message data".to_string())
        // ^^^ no `from` in error
    })?;
    let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
        rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
    Ok((from, decoded?))   // deserialization error propagated WITHOUT `from`
}
```

`from` is bound on line 334 but is **never embedded in either error path** (lines 335–340). Both the short-buffer branch and the `rmp_serde` decode failure branch emit errors that contain only a generic string or the serde error message — no participant identity.

This error propagates through the call stack:

1. `SharedChannel::recv` (lines 392–397) — passes the error up with `?`
2. `recv_from_others` (helpers.rs lines 19–23) — `chan.recv(waitpoint).await?` propagates without capturing `from`
3. Protocol entry points such as `do_keyshare` in `dkg.rs` (lines 422–426 and 514–515) — receive only an opaque `ProtocolError::Other(...)` or `ProtocolError::DeserializationError(...)`

A malicious participant needs only to:
1. Compute a valid `MessageHeader` for the current round (the header format is deterministic and public — SHA-256 of domain tags and participant IDs).
2. Append garbage bytes after the header.
3. Deliver this message via the `Protocol::message(from, data)` entry point (lines 512–514 of `internal.rs`).

The message passes `push_message`'s only guard (header length check, lines 287–295), is buffered, and is later popped by `recv`. Deserialization then fails, and the returned `ProtocolError` carries no indication of which participant sent the malformed payload.

---

### Impact Explanation

**High — Permanent denial of DKG, reshare, refresh, presign, and sign for honest parties.**

Because the error is unattributable:
- Honest parties cannot identify the malicious participant from the error alone.
- The malicious participant cannot be excluded from the next attempt.
- The attacker can repeat the attack in every subsequent session, permanently blocking all threshold operations that rely on `recv_from_others` or `SharedChannel::recv` — which includes every round of DKG (`do_keyshare`), reshare (`do_reshare`), and any signing protocol that uses the shared channel infrastructure.

---

### Likelihood Explanation

**High.** Any participant admitted to the protocol (a role with no cryptographic barrier to message injection) can execute this attack with trivial effort: send a valid header followed by random bytes. No key material, no cryptographic capability, and no privileged access is required. The attack is repeatable across sessions because the victim cannot attribute the failure.

---

### Recommendation

Embed the sender identity in every deserialization error path inside `Comms::recv`. For example:

```rust
async fn recv<T: DeserializeOwned>(
    &self,
    header: MessageHeader,
) -> Result<(Participant, T), ProtocolError> {
    let (from, data) = self.incoming.pop(header).await;
    let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| {
        ProtocolError::MaliciousParticipant(from)  // attribute to sender
    })?;
    let decoded: T = rmp_serde::decode::from_slice(message_data)
        .map_err(|_| ProtocolError::MaliciousParticipant(from))?;  // attribute to sender
    Ok((from, decoded))
}
```

Alternatively, add a dedicated `MalformedMessage(Participant)` variant to `ProtocolError` to distinguish malformed-message attribution from other malicious-participant errors. Callers such as `recv_from_others` can then surface the offending participant identity to the application layer.

### Proof of Concept

**Setup:** 3-party DKG with participants A (honest), B (honest), C (malicious).

**Attack steps:**

1. C participates normally through round 1 (commitment hash broadcast).
2. In round 2, when the protocol expects `(VerifiableSecretSharingCommitment, Option<Signature>)` via `recv_from_others` at `wait_round_1` / the broadcast waitpoint, C instead sends:
   - A correctly formed `MessageHeader` (deterministic, computable from public inputs).
   - Followed by `[0xFF, 0xFE]` — invalid MessagePack.
3. A and B each call `recv_from_others`, which calls `chan.recv(waitpoint).await?`.
4. `Comms::recv` pops C's message, successfully extracts `from = C`, then calls `rmp_serde::decode::from_slice` on the garbage payload.
5. Deserialization fails; `decoded?` returns `Err(ProtocolError::Other("...rmp_serde error..."))` — no mention of C.
6. `recv_from_others` propagates this error via `?` on line 20 of `helpers.rs`.
7. `do_keyshare` in `dkg.rs` receives the error and returns it to the caller.
8. The caller sees only `ProtocolError::Other("invalid type: ...")` — C is not identified.
9. C repeats this in every subsequent DKG attempt, permanently blocking key generation.
