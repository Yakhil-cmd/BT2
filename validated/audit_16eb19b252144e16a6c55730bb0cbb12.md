### Title
Nonce-Free COA Ownership Proof Enables Signature Replay for Repeated ERC-1271 Authorization — (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies a Cadence account's signatures over an arbitrary `signedData` blob but imposes no nonce, timestamp, or proof-consumption tracking. A valid proof captured from any on-chain transaction can be replayed indefinitely against the same COA. Because the COA's ERC-1271 `isValidSignature` implementation in `coa.sol` delegates directly to this stateless precompile, any EVM contract that uses `isValidSignature` to gate a one-time privileged action (token mint, permit, etc.) without its own replay guard is exploitable by an unprivileged attacker who observed the original proof.

### Finding Description
`EVM.validateCOAOwnershipProof` is declared `access(all)` and accepts caller-supplied `signedData`, `keyIndices`, `signatures`, and `evmAddress`. It verifies the Cadence key-list signature over `signedData` and then checks that the COA stored at `path` matches `evmAddress`. No nonce field exists in the proof struct, and no consumed-proof registry is maintained. [1](#0-0) 

The code's own comment acknowledges cross-address replay but explicitly does **not** address same-address replay: the identical `(signedData, keyIndices, signatures)` tuple passes verification every time it is presented for the same COA, because the function is purely stateless. [2](#0-1) 

The EVM-side entry point is `coa.sol`'s `isValidSignature`, which calls the `verifyCOAOwnershipProof` CadenceArch precompile with `address(this)` and the caller-supplied `_hash`/`_sig`: [3](#0-2) 

The Go precompile layer reconstructs a `COAOwnershipProofInContext` and invokes `validateCOAOwnershipProof` via `backend.Invoke`: [4](#0-3) 

The `COAOwnershipProofInContext` struct carries `SignedData` as a plain byte slice with no nonce field: [5](#0-4) 

**Exploit path (no privilege required):**
1. Alice owns a COA and signs hash `H` (e.g., a permit to mint one NFT) with her Flow account key. She submits the encoded proof in a Cadence transaction or EVM call — the proof is now public on-chain.
2. An EVM contract (e.g., a lazy-mint NFT contract) calls `isValidSignature(H, proof)` on Alice's COA, receives `ValidERC1271Signature`, and mints one NFT.
3. An attacker re-submits the identical `(H, proof)` to the same EVM contract. `validateCOAOwnershipProof` returns `isValid: true` again because nothing has changed — no nonce was consumed, no proof was marked used.
4. The EVM contract mints a second NFT (or executes any other privileged action) without Alice's fresh consent.

This is structurally identical to the reported ERC-1155 voucher replay: a signed authorization with no nonce that can be presented multiple times to exceed the intended single-use limit.

### Impact Explanation
Any EVM contract deployed on Flow EVM that relies on `isValidSignature` (ERC-1271) against a COA to gate a one-time or quantity-limited action — token minting, permit-based transfers, governance votes, etc. — is vulnerable to unlimited replay by any party who observed the original proof. The attacker needs no privileged access; the proof is public once submitted. The direct impact is unauthorized minting or asset movement beyond what the COA owner authorized.

### Likelihood Explanation
ERC-1271 is the standard mechanism for smart-contract wallets to sign off-chain messages, and COAs are Flow's native smart-contract wallet type. Lazy-mint NFT patterns, EIP-2612-style permits, and meta-transaction relayers all commonly use `isValidSignature` to authorize one-time actions. Because `validateCOAOwnershipProof` provides no nonce primitive, every such integration must independently implement replay protection — a requirement that is not documented in the function signature or enforced by the protocol. The likelihood that at least one deployed EVM contract on Flow EVM omits this guard is high.

### Recommendation
1. **Add a nonce field to `COAOwnershipProof`** (and to `COAOwnershipProofInContext`) and maintain a per-COA consumed-nonce registry inside `validateCOAOwnershipProof`, analogous to how EIP-2612 tracks permit nonces per account.
2. Alternatively, **require that `signedData` encodes both `evmAddress` and a monotonically increasing nonce**, and enforce this structurally inside `validateCOAOwnershipProof` rather than leaving it as a caller advisory.
3. Update `coa.sol`'s `isValidSignature` to include the COA's current nonce in the signed digest so that each authorization is cryptographically unique.

// COAOwnershipProofInContext contains all the data
// needed to verify a COAOwnership proof.
// The proof is verified by checking the signatures over the
// input signed data (SignedData), then loading the resource
// capability from the provided path in the proof, and
// at last checking if the EVMAddress of the resource matches
// the provided one.
type COAOwnershipProofInContext struct {
	COAOwnershipProof
	SignedData SignedData
	EVMAddress Address
}

func NewCOAOwnershipProofInContext(sd []byte, addr Address, encodedProof []byte) (*COAOwnershipProofInContext, error) {
	proof, err := COAOwnershipProofFromEncoded(encodedProof)
	if err != nil {
		return nil, err
	}
	return &COAOwnershipProofInContext{
		COAOwnershipProof: *proof,
		SignedData:        sd,
		EVMAddress:        addr,
	}, nil
}
