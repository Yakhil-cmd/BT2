### Title
Receipt Size Invariant Bypass via Post-Validation `output_data_receivers` Mutation in `promise_return` Path — (File: `runtime/runtime/src/lib.rs`)

---

### Summary

A contract can craft a new action receipt whose serialized size is exactly at `max_receipt_size` (4 MiB), pass the `NewReceipt` size check, and then use `promise_return` to cause the runtime to append `output_data_receivers` to that already-validated receipt. The mutation happens after validation, producing an oversized receipt that is forwarded cross-shard in violation of the protocol's receipt size invariant. The bug is acknowledged in the codebase (issue #12606) and is confirmed reachable by an existing integration test.

---

### Finding Description

**Invariant broken**: Every receipt emitted by the runtime must satisfy `borsh_size(receipt) ≤ max_receipt_size` (4,194,304 bytes). This invariant underpins the state witness size budget (≤ 17 MiB) and cross-shard receipt routing limits.

**Root cause — two-phase mutation after single-phase validation**

`validate_receipt` in `runtime/runtime/src/verifier.rs` enforces the size check only when `mode == ValidateReceiptMode::NewReceipt`:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

This check is called from `apply_action_receipt` immediately after each action produces new receipts:

```rust
if let Err(e) = new_result.new_receipts.iter().try_for_each(|receipt| {
    validate_receipt(..., ValidateReceiptMode::NewReceipt)
}) {
    new_result.result = Err(ActionErrorKind::NewReceiptValidationError(e).into());
}
``` [2](#0-1) 

However, **after** this validation loop, the runtime handles the `promise_return` case. When a function call returns `ReturnData::ReceiptIndex`, the runtime mutates the already-validated receipt by appending the parent receipt's `output_data_receivers` to it:

```rust
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
            ReceiptEnum::Action(new_action_receipt) | ... => new_action_receipt
                .output_data_receivers
                .extend_from_slice(&action_receipt.output_data_receivers()),
            ...
        }
    }
``` [3](#0-2) 

No re-validation of the receipt size occurs after this mutation. The receipt is then forwarded as an outgoing receipt with its new, larger size.

**Attacker-controlled field**: The `args` field of a `FunctionCallAction` inside the new receipt (set via `promise_create` host function). The user controls `args_size` to size the receipt exactly at `max_receipt_size - base_receipt_size`, so it passes validation. After `output_data_receivers` are appended, the receipt exceeds `max_receipt_size`.

**Exact corrupted value**: The borsh-serialized size of the forwarded receipt. For example, with `args_size = max_receipt_size - base_receipt_size` and one `DataReceiver` appended, the receipt size becomes `4,194,504` bytes — 200 bytes over the 4,194,304-byte limit (as confirmed by the integration test). [4](#0-3) 

**Why existing guards fail**: The `ExistingReceipt` mode explicitly skips the size check to handle these oversized receipts gracefully, as documented in the codebase:

> "There is a bug which allows to create receipts that are above the size limit. Runtime has to handle them gracefully until the receipt size limit bug is fixed. See https://github.com/near/nearcore/issues/12606 for details." [5](#0-4) 

So when the oversized receipt arrives at the destination shard, it is processed without triggering any error.

---

### Impact Explanation

**Scope**: Pre-inclusion transaction validation / receipt causality.

The `max_receipt_size` invariant is a hard protocol limit introduced to keep `ChunkStateWitness` under 17 MiB. An oversized receipt forwarded cross-shard:

1. Inflates the `source_receipt_proofs` in the state witness beyond the intended budget.
2. Can cause chunk validators to observe a state witness that violates the size constraints they are supposed to enforce, potentially leading to disagreement on chunk application results.
3. Persists in the chain permanently — the oversized receipt is stored and processed, not rejected.

**Severity**: High. The invariant is broken at the protocol level, not merely at the application level. The impact scales with the number of `output_data_receivers` appended (each adds ~40 bytes), and a user can chain multiple `promise_then` calls to maximize the overflow.

---

### Likelihood Explanation

Any unprivileged user can trigger this by:
1. Deploying a contract (standard operation).
2. Calling a method that uses `promise_create` with large args and `promise_return`.
3. Wrapping that call in a `promise_then` chain so the parent receipt has non-empty `output_data_receivers`.

No special privileges are required. The attack is deterministic and reproducible. The integration test `test_max_receipt_size_promise_return` confirms it is reachable in the current production codebase. [6](#0-5) 

---

### Recommendation

After the `promise_return` mutation at lines 1019–1037 of `runtime/runtime/src/lib.rs`, re-validate the size of the mutated receipt before forwarding it. If the receipt now exceeds `max_receipt_size`, return a `NewReceiptValidationError(ReceiptSizeExceeded { ... })` for the action, consistent with how other receipt validation errors are handled at line 864.

Alternatively, enforce the size check at the point of appending `output_data_receivers` in `create_action_receipt` inside `receipt_manager.rs`, so the invariant is maintained at the source rather than relying on post-hoc validation. [7](#0-6) 

---

### Proof of Concept

The existing integration test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete proof of concept:

1. Deploy `rs_contract` to `account0`.
2. Call `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`. This creates the DAG: `A[method2(args)] -then-> B[mark_test_completed()]`.
3. When `A` executes, it calls `max_receipt_size_promise_return_method2`, which creates receipt `C` with `args` sized exactly at the limit and calls `promise_return(C)`.
4. The runtime appends `B`'s `DataReceiver` to `C`'s `output_data_receivers`, making `C` exceed `max_receipt_size`.
5. `assert_oversized_receipt_occurred` confirms the oversized receipt (size `4,194,504 > 4,194,304`) appears in the chain.
6. `assert_test_completed` succeeds, confirming the oversized receipt was processed without error. [8](#0-7)

// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}

    pub(super) fn create_action_receipt(
        &mut self,
        input_data_ids: Vec<CryptoHash>,
        receipt_indices: Vec<ReceiptIndex>,
        receiver_id: AccountId,
    ) -> Result<ReceiptIndex, VMLogicError> {
        assert_eq!(input_data_ids.len(), receipt_indices.len());
        for (data_id, receipt_index) in input_data_ids.iter().zip(receipt_indices.into_iter()) {
            self.action_receipts
                .get_mut(receipt_index as usize)
                .ok_or(HostError::InvalidReceiptIndex { receipt_index })?
                .output_data_receivers
                .push(DataReceiver { data_id: *data_id, receiver_id: receiver_id.clone() });
        }

        let new_receipt = ActionReceiptMetadata {
            receiver_id,
            refund_to: None,
            output_data_receivers: vec![],
            input_data_ids,
            actions: vec![],
            is_promise_yield: false,
        };
        let new_receipt_index = self.action_receipts.len() as ReceiptIndex;
        self.action_receipts.push(new_receipt);
        Ok(new_receipt_index)
    }
