### Title
Missing `transferFrom` Return Value Check in XCC Precompile Enables Unpaid Cross-Contract Calls — (File: `engine-precompiles/src/xcc.rs`)

---

### Summary

The Cross-Contract Call (XCC) precompile calls `transferFrom` on the configured wNEAR ERC-20 contract to collect payment from the EVM caller before scheduling a NEAR cross-contract call. The EVM subcall result is destructured into `(exit_reason, return_value)`, but only `exit_reason` is inspected. The ABI-encoded boolean `return_value` — the actual ERC-20 success indicator — is silently discarded. Any ERC-20 implementation that signals failure by returning `false` rather than reverting (which is explicitly permitted by EIP-20) will cause the precompile to proceed as if payment succeeded, scheduling the NEAR promise without transferring any wNEAR.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, `CrossContractCall::run_with_handle` constructs `transferFrom` calldata via `transfer_from_args` and dispatches it as an EVM subcall:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),   // ← return_value never decoded
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e)  => { ... }
    aurora_evm::ExitReason::Fatal(f)  => { ... }
}
``` [1](#0-0) 

`return_value` is bound but never used in the `Succeed` arm. The comment "Transfer successful, nothing to do" is incorrect: a `Succeed` exit only means the EVM call did not revert or error; it says nothing about the boolean payload the ERC-20 returned. EIP-20 explicitly allows a conforming `transferFrom` to return `false` on failure without reverting.

The `transfer_from_args` helper itself correctly encodes the `transferFrom(address,address,uint256)` selector and arguments:

```rust
pub(super) const TRANSFER_FROM_SELECTOR: [u8; 4] = [0x23, 0xb8, 0x72, 0xdd];
``` [2](#0-1) 

The test suite even documents the expected `bool` output type for `transferFrom`:

```rust
outputs: vec![ethabi::Param {
    name: String::new(),
    kind: ethabi::ParamType::Bool,
    ...
}],
``` [3](#0-2) 

Yet the production path never decodes that `Bool` from `return_value`.

After the `Succeed` arm, execution falls through unconditionally to emit the promise log and return `Ok(PrecompileOutput { ... })`: [4](#0-3) 

The wNEAR address is set by the owner via `factory_set_wnear_address`, which stores any 20-byte address without validating that the target contract is a reverting ERC-20: [5](#0-4) 

The `EvmErc20` contract (the canonical wNEAR mirror) inherits OpenZeppelin ERC-20, which does revert on failure. However, the precompile is written to accept any address as wNEAR, and the ERC-20 standard does not require reverting.

---

### Impact Explanation

**Critical — Insolvency / Direct theft of funds.**

If `transferFrom` returns `false` without reverting, the XCC precompile:

1. Skips the wNEAR debit from the caller.
2. Emits the promise log that the Aurora engine interprets as authorization to attach real NEAR to the outgoing cross-contract call.
3. The engine's own NEAR balance is consumed to fund the call.

Repeated exploitation drains the engine's NEAR balance, making it unable to pay for future operations (insolvency) and effectively stealing the NEAR held by the engine on behalf of all users.

---

### Likelihood Explanation

**Medium.** The currently deployed wNEAR (`EvmErc20` / OpenZeppelin ERC-20) reverts on insufficient allowance or balance, so the bug is not immediately triggerable on a standard deployment. However:

- The wNEAR address is a mutable, owner-controlled parameter with no on-chain validation of the target contract's revert behavior.
- EIP-20 explicitly permits returning `false` instead of reverting; many real-world tokens (e.g., USDT, BNB) follow this pattern.
- If the wNEAR contract is ever upgraded or replaced with a non-reverting ERC-20, every XCC caller can immediately exploit the missing check with zero additional privileges.

---

### Recommendation

After the `Succeed` arm, ABI-decode `return_value` as a `bool` and revert if it is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // ERC-20 transferFrom returns a bool; false means failure without revert.
    let success = return_value
        .get(31)
        .map_or(false, |&b| b == 1);
    if !success {
        return Err(PrecompileFailure::Revert {
            exit_status: aurora_evm::ExitRevert::Reverted,
            output: b"ERR_WNEAR_TRANSFER_FAILED".to_vec(),
        });
    }
}
```

This mirrors the pattern already used for the `Revert` arm and makes the precompile robust against any ERC-20 implementation.

---

### Proof of Concept

1. Deploy a minimal ERC-20 whose `transferFrom` always returns `false` (no revert).
2. Call `factory_set_wnear_address` with that contract's address (owner action, or triggered if wNEAR is ever rotated).
3. From any EVM account, call the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with valid `CrossContractCallArgs`.
4. The `handle.call` to wNEAR returns `(Succeed, [0x00…00])` — exit succeeds, boolean is `false`.
5. The `Succeed` arm does nothing; execution continues to `Ok(PrecompileOutput { logs: [promise_log], … })`.
6. The Aurora engine processes the promise log and attaches real NEAR to the outgoing call — no wNEAR was debited from the caller.
7. Repeat to drain the engine's NEAR balance. [6](#0-5)
