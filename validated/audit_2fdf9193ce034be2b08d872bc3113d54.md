### Title
Gateway Signature-Validation Bypass via `skip_stateful_validations` UX Shortcut â€” (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point (signature verification) for any `Invoke` transaction whose nonce is `1` and whose sender address appears in the mempool â€” regardless of whether the matching mempool entry is a `DeployAccount` or any other transaction type. An unprivileged attacker who observes a pending `DeployAccount` for address `V` can immediately submit a crafted `Invoke(nonce=1, sender=V)` with an arbitrary/invalid signature and have it admitted to the mempool without any cryptographic check.

The check does **not** verify that the matching mempool entry is specifically a `DeployAccount`. Any transaction from that address â€” including one submitted by a third party â€” satisfies the predicate.

**Attack path:**

1. Alice broadcasts `DeployAccount(class_hash, salt, ctor_calldata)` â†’ address `V` enters the mempool's `tx_pool`.
2. Bob observes `V` in the public mempool.
3. Bob submits `Invoke(sender=V, nonce=1, calldata=<drain V's balance>, signature=<garbage>)`.
4. Gateway stateless checks pass (signature length is within limit).
5. `validate_nonce`: `account_nonce=0`, `tx_nonce=1`, within `max_allowed_nonce_gap` â€” passes.
6. `validate_by_mempool`: nonce ordering check passes.
7. `skip_stateful_validations`: `tx.nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(V)==true` â†’ returns `true`.
8. `run_validate_entry_point` is called with `skip_validate=true` â†’ `__validate__` is **never called**.
9. Bob's `Invoke` with invalid signature is forwarded to the mempool via `add_tx`. [5](#0-4) [6](#0-5) 

### Impact Explanation

The gateway admits an `Invoke` transaction that carries an invalid (attacker-controlled) signature. This violates the invariant that every admitted transaction must have passed account-level signature verification. Concretely:

- Bob's invalid `Invoke(nonce=1)` occupies the `(V, nonce=1)` slot in the mempool.
- Alice's legitimate `Invoke(nonce=1)` is either rejected as a duplicate or must pay a higher tip to displace Bob's transaction via fee escalation.
- If Bob's transaction reaches the batcher before Alice's, it will be executed: the batcher creates a fresh `AccountTransaction` with `execution_flags.validate=true`, so `__validate__` is called on the now-deployed account and the transaction reverts â€” but it consumes block space and delays Alice's transaction.
- The attack is repeatable for every new `DeployAccount` that appears in the mempool, enabling systematic griefing of the deploy-account + invoke UX flow.

This matches: **High â€” Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- The mempool is public; any observer can detect a pending `DeployAccount`.
- No privileged access is required; a standard RPC `starknet_addInvokeTransaction` call suffices.
- The condition (`nonce==1`, `account_nonce==0`, address in pool) is deterministically satisfied the moment a `DeployAccount` is submitted.
- The attack is cheap: the attacker pays only the gas for submitting the invalid invoke (which will revert).

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` predicate with a check that specifically verifies a **`DeployAccount`** transaction for the sender address is present in the mempool. Alternatively, require that the `Invoke(nonce=1)` transaction carries a valid signature even when the account is not yet deployed, by running `__validate__` against the class hash declared in the pending `DeployAccount` (pre-fetched from the class manager). At minimum, the comment's stated invariant â€” "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations" â€” should be enforced in code, not just in prose.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
