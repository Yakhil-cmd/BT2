### Title
Missing Caller Access Control on `update_rewards` Allows Any Address to Prevent or Redirect Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in `StakingRewardsManagerImpl` is specified to be callable by "Only starkware sequencer" (per `docs/spec.md`), but the implementation contains no such caller check. Any non-zero address can invoke it. Because `last_reward_block` is a **global** state variable, a single unauthorized call per block either silently suppresses reward distribution (via `disable_rewards: true`) or redirects block rewards to an attacker-chosen staker (via `disable_rewards: false`), preventing the legitimate sequencer call from succeeding in that block.

### Finding Description

The spec at `docs/spec.md:1645` states:

> **Access control:** Only starkware sequencer.

The implementation at `src/staking/staking.cairo:1449–1507` is:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... no assert_caller_is_sequencer() or equivalent
    self.last_reward_block.write(current_block_number);   // global write
    if disable_rewards || self.is_pre_consensus() {
        return;   // exits without distributing rewards
    }
    // ... distributes rewards to staker_address
```

`general_prerequisites` (line 1794–1797) only calls `assert_is_unpaused()` and `assert_caller_is_not_zero()`. There is no `assert_caller_is_sequencer()` anywhere in the codebase (confirmed by grep: zero matches for `sequencer_address`, `CALLER_IS_NOT_SEQUENCER`, `assert_caller_is_sequencer`).

`last_reward_block` is a single global slot — not per-staker. Once written in a block, the check `current_block_number > self.last_reward_block.read()` causes every subsequent call in the same block to revert with `REWARDS_ALREADY_UPDATED`.

**Attack path A – griefing / yield suppression:**
1. Attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)`.
2. `last_reward_block` is set to the current block number (line 1485).
3. Function returns early at line 1487–1489 — no rewards distributed, no `unclaimed_rewards_own` updated, no pool transfers.
4. The sequencer's legitimate call for the actual block producer reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers and delegators lose their block rewards for that block permanently.

**Attack path B – reward misrouting / theft:**
1. Attacker (who is a valid staker) calls `update_rewards(attacker_staker_address, disable_rewards: false)` before the sequencer.
2. Block rewards are calculated and credited to the attacker's staker instead of the intended block producer.
3. The sequencer's call for the actual block producer reverts.
4. The attacker's staker receives rewards it did not earn; the legitimate block producer receives nothing.

### Impact Explanation

- **Path A:** Block rewards for the targeted block are permanently lost — they are never minted/allocated to any staker or delegator. This constitutes **permanent freezing of unclaimed yield** (High) or at minimum **griefing with damage to users and protocol** (Medium).
- **Path B:** Block rewards are redirected from the legitimate block producer to the attacker's staker. This constitutes **theft of unclaimed yield** (High).

### Likelihood Explanation

On Starknet, the sequencer controls transaction ordering and can in principle always place its own `update_rewards` call first. This reduces exploitability compared to Ethereum. However:
- The spec explicitly mandates "Only starkware sequencer" access control, which is entirely absent from the code.
- Any non-zero address can call the function at any time; no special role, key, or privilege is required.
- If the sequencer ever fails to call `update_rewards` first in a block (e.g., due to a sequencer bug, operational gap, or a block where the sequencer does not include its own transaction), an attacker can exploit this immediately.
- The attack is cheap (a single transaction) and repeatable every block.

### Recommendation

Add an explicit caller check at the top of `update_rewards` to enforce the spec's "Only starkware sequencer" access control. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ...
}
```

Alternatively, if the sequencer address is not stored on-chain, use Starknet's `get_execution_info` to verify the transaction originates from the expected sequencer.

### Proof of Concept

1. Deploy the staking contract and advance to the consensus rewards epoch.
2. Stake as `staker_A` (the legitimate block producer) and `staker_B` (the attacker).
3. From `staker_B`'s address, call `update_rewards(staker_B_address, disable_rewards: false)`.
4. Observe that `staker_B.unclaimed_rewards_own` increases by the full block reward.
5. Attempt to call `update_rewards(staker_A_address, disable_rewards: false)` from the sequencer — it reverts with `REWARDS_ALREADY_UPDATED`.
6. `staker_A` receives zero rewards for that block.

The `last_reward_block` global guard (line 1455) and the early-return path (line 1487) are the two code paths that make this exploitable.
