### Title
FROST Presign Commitments Lack Reliable Broadcast, Enabling Equivocation That Corrupts Presign Transcripts - (File: src/frost/mod.rs)

### Summary
The FROST presign protocol distributes round-1 signing commitments using a simple `chan.send_many` call rather than a reliable broadcast (`do_broadcast`). A malicious participant can exploit this to send different commitments to different honest participants, causing each honest party to build a different `commitments_map`. Honest parties accept these inconsistent presign transcripts without detecting the equivocation — a direct analog to the chain-reorganization bug, where state is accepted without verifying consistency with what other observers recorded.

### Finding Description
In `src/frost/mod.rs`, the `do_presign` function collects FROST round-1 signing commitments from all participants using a plain `chan.send_many` / `recv_from_others` exchange:

```rust
let commit_waitpoint = chan.next_waitpoint();
chan.send_many(commit_waitpoint, &commitments)?;          // ← simple broadcast, no consistency guarantee

for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
    commitments_map.insert(from.to_identifier()?, commitment);
}

Ok(PresignOutput { nonces, commitments_map })
``` [1](#0-0) 

Unlike the DKG protocol, which uses `do_broadcast` (an echo-broadcast protocol) for every round that carries security-critical state, the FROST presign protocol has no mechanism to ensure that all participants receive the same commitment from each sender. [2](#0-1) 

A malicious participant P_m can send commitment `C_m^A` to participant A and a different commitment `C_m^B` to participant B. Both A and B will accept these commitments and insert them into their `commitments_map`. The presign protocol completes successfully for both, but their `PresignOutput.commitments_map` fields are inconsistent — they record different "histories" of what P_m committed to, exactly as a chain reorganization produces two different histories of what was deposited.

When signing is subsequently attempted using these outputs:

- A uses `C_m^A` to compute binding factor `ρ_i^A` and group commitment `R_A`
- B uses `C_m^B` to compute binding factor `ρ_i^B` and group commitment `R_B`
- A and B compute different Fiat-Shamir challenges `c_A ≠ c_B`
- A and B produce signature shares `s_i^A ≠ s_i^B` computed under different challenges
- Aggregation fails; no valid signature is produced [3](#0-2) 

The DKG protocol explicitly avoids this class of bug by using `do_broadcast` for commitment hashes and for the commitments themselves, so that any equivocation is detected before state is accepted: [4](#0-3) [5](#0-4) 

The FROST presign protocol has no equivalent protection.

### Impact Explanation
Honest parties accept inconsistent presign transcripts — different `commitments_map` values — without detecting the equivocation. This is a direct instance of the allowed High impact: **"Corruption of presign outputs so honest parties accept inconsistent transcripts."**

The corrupted transcripts render the presign output unusable: subsequent signing fails because signature shares are computed under different challenges. Additionally, if the application layer retries signing with the same `PresignOutput` (nonce reuse), the two signing sessions with different challenges over the same nonces expose the secret key via standard FROST nonce-reuse analysis.

### Likelihood Explanation
The attack requires exactly one malicious participant — any member of the presign group. The attacker needs only to send two different `SigningCommitments` values to two different recipients during the single presign round. No cryptographic capability is required; the attack is a straightforward message-level equivocation. The presign protocol provides no detection mechanism, so the attack succeeds silently every time it is attempted.

```rust
async fn do_presign<C: Ciphersuite + Send>(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    signing_share: SigningShare<C>,
    mut rng: impl CryptoRngCore,
) -> Result<PresignOutput<C>, ProtocolError> {
    // Round 1
    let mut commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>> = BTreeMap::new();

    // Creating two commitments and corresponding nonces
    let (nonces, commitments) = commit(&signing_share, &mut rng);
    commitments_map.insert(me.to_identifier()?, commitments);

    let commit_waitpoint = chan.next_waitpoint();
    // Sending the commitments to all
    chan.send_many(commit_waitpoint, &commitments)?;

    // Collecting the commitments
    for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
        commitments_map.insert(from.to_identifier()?, commitment);
    }

    Ok(PresignOutput {
        nonces,
        commitments_map,
    })
}
