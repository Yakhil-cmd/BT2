### Title
Single Malicious Participant Permanently Aborts DKG for All Honest Parties via Invalid Proof of Knowledge — (`src/dkg.rs`)

### Summary

In `do_keyshare`, after the echo-broadcast round delivers each participant's `(commitment, proof_of_knowledge)`, the code iterates over all other participants and calls `verify_proof_of_knowledge(...)?`. The `?` operator causes an immediate, unrecoverable abort of the entire DKG function the moment any single participant's PoK fails verification. Because the echo-broadcast layer guarantees that all honest parties receive the same (invalid) PoK from the malicious sender, every honest party independently hits the same error and aborts. There is no mechanism to exclude the offending participant and continue.

### Finding Description

The abort path is:

**`do_keyshare` (src/dkg.rs, lines 446–460):** [1](#0-0) 

The loop iterates over `participants.others(me)`. For each participant `p`, it calls:

```rust
verify_proof_of_knowledge(...)?;
```

`verify_proof_of_knowledge` delegates to `internal_verify_proof_of_knowledge`, which performs the check:

**`internal_verify_proof_of_knowledge` (src/dkg.rs, lines 162–164):** [2](#0-1) 

```rust
if *big_r != <C::Group>::generator() * *z - vk_share.value() * c.to_scalar() {
    return Err(ProtocolError::InvalidProofOfKnowledge(participant));
}
```

If the malicious participant broadcasts a random `(R, mu)` pair, this check fails and `Err(InvalidProofOfKnowledge)` is returned. The `?` at the call site in `do_keyshare` propagates this error immediately, aborting `do_keyshare` for the honest party — before any share exchange or key derivation occurs.

**Why all honest parties abort:** The commitment and PoK are distributed via `do_broadcast` (echo-broadcast): [3](#0-2) 

The echo-broadcast protocol guarantees consistency — all honest parties receive the same value from each sender. For n≤3, `echo_ready_thresholds` returns `(0, 0)`: [4](#0-3) 

This means there is zero Byzantine tolerance at the broadcast layer for the minimum viable configuration (n=3). The malicious party's invalid PoK is reliably and consistently delivered to all honest parties. Each honest party then independently calls `verify_proof_of_knowledge(...)?`, hits the same `Err(InvalidProofOfKnowledge)`, and aborts.

**No recovery path exists.** There is no logic to skip a participant with an invalid PoK, remove them from the participant set, or continue with the remaining honest parties. The error is fatal and unhandled.

### Impact Explanation

With n=3, threshold=2, and 1 malicious participant (f=1 < threshold=2), the malicious party can abort every DKG attempt indefinitely. As long as the malicious party is included in the participant list, honest parties can never complete key generation. The same attack applies to `do_reshare` and `do_refresh`, since they all call `do_keyshare` internally. [5](#0-4) 

### Likelihood Explanation

The attack requires only that the malicious participant send a random `(R, mu)` as their PoK — a trivial deviation from the honest protocol. No cryptographic assumptions need to be broken. The attacker needs only to be a registered participant in the DKG session.

### Recommendation

After `verify_proof_of_knowledge` fails, the honest party should record the offending participant as malicious and continue processing the remaining participants, rather than aborting immediately. At the end of the loop, if any malicious participants were detected, the protocol should abort with a list of identified culprits — but only if the number of remaining honest participants is insufficient to meet the threshold. If enough honest participants remain (i.e., `n - f >= threshold`), the protocol should be able to proceed without the malicious party's contribution.

### Proof of Concept

```
Setup: participants = [P1 (honest), P2 (honest), P3 (malicious)], threshold = 2

1. All parties run do_keyshare.
2. P3 generates a random (R, mu) as its proof_of_knowledge instead of a valid one.
3. P3 broadcasts (commitment, random_proof) via do_broadcast.
4. Echo-broadcast delivers P3's (commitment, random_proof) consistently to P1 and P2.
5. P1 iterates participants.others(P1) = [P2, P3].
   - verify_proof_of_knowledge for P2: OK
   - verify_proof_of_knowledge for P3: Err(InvalidProofOfKnowledge(P3)) → ? → do_keyshare returns Err
6. P2 iterates participants.others(P2) = [P1, P3].
   - verify_proof_of_knowledge for P1: OK
   - verify_proof_of_knowledge for P3: Err(InvalidProofOfKnowledge(P3)) → ? → do_keyshare returns Err
7. Both P1 and P2 have aborted. DKG fails permanently as long as P3 is included.
```
