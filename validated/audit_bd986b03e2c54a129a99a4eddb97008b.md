### Title
Admin HTTP Server Executes Privileged Commands Without Mandatory Caller Authentication - (File: `admin/command_runner.go`)

### Summary

The `CommandRunner` in `admin/command_runner.go` exposes an HTTP server that dispatches privileged admin commands with no mandatory caller authentication. Mutual TLS is entirely optional and is only applied when the operator explicitly provides cert/key flags. When TLS is absent — the default — any network-reachable caller can POST to `/admin/run_command` and invoke any registered command, including `set-config` with `consensus-required-approvals-for-sealing: 0`, which disables verification requirements for block sealing on a consensus node.

### Impact Explanation

An attacker who can reach the admin HTTP port can:

1. **Disable consensus sealing verification**: POST `{"commandName":"set-config","data":{"consensus-required-approvals-for-sealing":0}}` to a consensus node's admin port. This sets `requiredApprovalsForSealConstruction` to 0, causing the consensus node to construct and include seals without requiring any verification approvals from verification nodes. This is a critical integrity failure in the sealing pipeline — the consensus node will accept execution results as sealed without independent verification.

2. **Crash execution nodes on demand**: POST `{"commandName":"stop-at-height","data":{"height":<N>,"crash":true}}` to halt an execution node at any chosen block height, causing targeted availability disruption of the execution layer.

3. **Manipulate peer blocklists**: Add or remove arbitrary node IDs from the network blocklist, disrupting peer connectivity.

### Likelihood Explanation

The admin server has no mandatory authentication in code. The only protection is network-level access control (binding to localhost or firewall rules), which is entirely operator-dependent. The `--admin-addr` flag accepts any address including `0.0.0.0`. In containerized deployments (Docker, Kubernetes), the admin port may be reachable from co-located containers or pods without any special privileges. The `cmd/ledger` README explicitly shows `0.0.0.0:9003` as a documented example. Any attacker with network access to the admin port — no credentials, no keys, no staked node identity required — can invoke any command.

### Recommendation

Make caller authentication mandatory at the code level, not optional via configuration:

1. **Require mTLS unconditionally**: Remove the `if node.AdminCert != NotSet` branch. The admin server should refuse to start without a valid TLS configuration. Operators must supply cert/key/CA files.
2. **Add an authentication middleware**: Wrap the HTTP mux with a middleware that verifies the client certificate or a bearer token before dispatching to `gwmux`.
3. **Bind to localhost by default**: Change the default value of `--admin-addr` to `localhost:9002` and document that binding to `0.0.0.0` requires mTLS to be configured.

### Proof of Concept

Against a consensus node with `--admin-addr 0.0.0.0:9002` and no TLS configured:

```bash
# Step 1: Disable verification requirements for sealing
curl -s -X POST http://<consensus-node-ip>:9002/admin/run_command \
  -H 'Content-Type: application/json' \
  -d '{"commandName":"set-config","data":{"consensus-required-approvals-for-sealing":0}}'
# Response: {"output":{"newValue":0,"oldValue":1}}

# Step 2: Confirm the change took effect
curl -s -X POST http://<consensus-node-ip>:9002/admin/run_command \
  -H 'Content-Type: application/json' \
  -d '{"commandName":"get-config","data":"consensus-required-approvals-for-sealing"}'
# Response: {"output":0}
```

After step 1, the consensus node constructs seals with zero required verification approvals, bypassing the verification node pipeline entirely. No credentials, no staked identity, and no privileged access are required — only network reachability to the admin port.
