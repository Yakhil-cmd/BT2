### Title
Production Sequencer Uses Hardcoded Testing Private Key for Precommit Vote Signing — (`crates/apollo_signature_manager/src/lib.rs`)

---

### Summary

`create_signature_manager()` — the function called by the production node component factory — unconditionally instantiates `LocalKeyStore::new_for_testing()`, which embeds a publicly known private key. Any party who knows this key (it is in the public source) can forge valid precommit vote signatures for arbitrary block hashes, and `verify_precommit_vote_signature` will return `Ok(true)` for those forgeries.

---

### Finding Description

`LocalKeyStore::new_for_testing()` hard-codes both the private and public keys:

```
PRIVATE_KEY = 0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133
PUBLIC_KEY  = 0x125d56b1fbba593f1dd215b7c55e384acd838cad549c4a2b9c6d32d264f4e2a
``` [1](#0-0) 

The method is `pub(crate)`, but `lib.rs` exposes it through the public `create_signature_manager()` function:

```rust
pub fn create_signature_manager() -> SignatureManager {
    SignatureManager::new()   // calls LocalKeyStore::new_for_testing() internally
}
``` [2](#0-1) 

`LocalKeyStoreSignatureManager::new()` is the only constructor and it always calls `new_for_testing()`: [3](#0-2) 

The production node component factory calls `create_signature_manager()` unconditionally whenever the signature manager component is enabled:

```rust
let signature_manager = match config.components.signature_manager.execution_mode {
    ReactiveComponentExecutionMode::LocalExecutionWithRemoteDisabled
    | ReactiveComponentExecutionMode::LocalExecutionWithRemoteEnabled => {
        Some(create_signature_manager())   // ← hardcoded key in production
    }
    ...
};
``` [4](#0-3) 

There is no configuration path, environment variable, or key-loading mechanism that substitutes a real key. The `_new(private_key: PrivateKey)` constructor that would accept an external key is private and unused in any non-test path. [5](#0-4) 

---

### Impact Explanation

`sign_precommit_vote` signs `PRECOMMIT_VOTE || block_hash` with the hardcoded private key. [6](#0-5) 

`verify_precommit_vote_signature` verifies against the corresponding public key. [7](#0-6) 

Because the private key is public knowledge, an attacker can:

1. Compute `digest = blake2s_to_felt(b"PRECOMMIT_VOTE" || attacker_chosen_block_hash.to_bytes_be())`.
2. Sign `digest` with the known private key.
3. Submit the forged `(block_hash, signature)` pair to any peer that calls `verify_precommit_vote_signature`.
4. The call returns `Ok(true)`, making the forged precommit vote indistinguishable from a legitimate one.

A consensus peer that accepts this forged vote can be made to treat an attacker-chosen block hash as finalized, causing RPC endpoints to serve wrong block state, storage values, and class hashes for that height.

---

### Likelihood Explanation

The private key is embedded in the public source repository. No special access is required to obtain it. The only precondition is network reachability to a consensus peer — not operator or validator privilege. Any deployment that enables the signature manager component (the default production path) is affected.

---

### Recommendation

1. Remove `new_for_testing()` from production-reachable code. Gate it with `#[cfg(test)]`.
2. Implement a real key-loading path in `LocalKeyStore::_new` (rename it `new`) that reads the private key from a secrets manager, HSM, or environment-injected secret at startup.
3. Make `create_signature_manager()` accept a `PrivateKey` parameter (or a config struct) so the node config layer is forced to supply a key explicitly.
4. Add a startup assertion that panics if the key matches the known testing value in a non-test build.

---

### Proof of Concept

```rust
#[test]
fn attacker_forges_precommit_vote_with_known_key() {
    use starknet_api::block::BlockHash;
    use starknet_api::felt;
    use crate::signature_manager::{
        LocalKeyStore, SignatureManager, verify_precommit_vote_signature,
    };

    // Step 1: attacker uses the publicly known private key (same as new_for_testing)
    let key_store = LocalKeyStore::new_for_testing();
    let sm = SignatureManager::new(key_store);

    // Step 2: attacker chooses an arbitrary block hash
    let evil_block_hash = BlockHash(felt!("0xdeadbeef"));

    // Step 3: attacker signs it
    let rt = tokio::runtime::Runtime::new().unwrap();
    let forged_sig = rt.block_on(sm.sign_precommit_vote(evil_block_hash)).unwrap();

    // Step 4: any verifier using the known public key accepts it
    let public_key = key_store.public_key;
    assert!(
        verify_precommit_vote_signature(evil_block_hash, forged_sig, public_key).unwrap(),
        "Forged signature accepted — hardcoded key is exploitable in production"
    );
}
```

The test passes because `create_signature_manager()` in production uses the identical key. [2](#0-1) [4](#0-3)
