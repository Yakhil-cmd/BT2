### Title
`MsgTurnBridge` Has No Permission Check and No Implementation, Permanently Breaking the Bridge Circuit Breaker - (File: `x/cronos/keeper/msg_server.go`)

### Summary
`MsgTurnBridge` is the designated emergency circuit breaker for the Cronos gravity bridge. Its keeper implementation is a stub that returns `nil, nil` with no permission check and no state change. Any unprivileged account can submit the message and receive a success response, while the bridge state is never actually modified — making it impossible for any authorized party to disable the bridge.

### Finding Description

The `TurnBridge` handler in `x/cronos/keeper/msg_server.go` is a two-line stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, the sibling permissioned message, which correctly gates execution behind `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`: [2](#0-1) 

The permission system defines `CanTurnBridge` as a distinct bit-flag and the `HasPermission` helper enforces it for other messages, but `TurnBridge` never calls it: [3](#0-2) 

The integration test for `test_gravity_turn_bridge` explicitly asserts that an unprivileged `community` account must be rejected (`code != 0`) and that a privileged `validator` account must succeed and actually stop bridge batches from being created: [4](#0-3) 

With the current stub, both assertions are violated: the `community` call returns success (code 0) instead of an authorization error, and the bridge is never disabled regardless of who calls the message.

### Impact Explanation

Two distinct High-severity impacts are present:

1. **Auth bypass**: The `CanTurnBridge` permission check is entirely absent. Any unprivileged account can submit `MsgTurnBridge` and receive a success response, bypassing the Cronos admin/governance authorization model.

2. **Permanent inability to disable the bridge**: Because the function performs no state change, the bridge circuit breaker is permanently non-functional. In an emergency (e.g., an active exploit draining bridge funds), no authorized party — admin, permissioned address, or governance — can halt bridge operations through this message.

### Likelihood Explanation

The stub is reachable by any account that can submit a Cosmos SDK transaction. No special privileges, leaked keys, or cryptographic breaks are required. The `CmdTurnBridge` CLI command is publicly documented and wired to the handler: [5](#0-4) 

### Recommendation

Restore the full `TurnBridge` implementation:
1. Add a permission check: `if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) { return nil, ErrUnauthorized }`.
2. Persist the `Enable` flag to the gravity module's `bridge_active` parameter (or the equivalent Cronos-side state) so that EVM hooks and the gravity end-blocker respect the disabled state.

// CmdTurnBridge returns a CLI command handler for enable or disable the bridge
func CmdTurnBridge() *cobra.Command {
	cmd := &cobra.Command{
		Use:   "turn-bridge [true/false]",
		Short: "Turn Bridge",
		Args:  cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			clientCtx, err := client.GetClientTxContext(cmd)
			if err != nil {
				return err
			}

			enable, err := strconv.ParseBool(args[0])
			if err != nil {
				return err
			}
			msg := types.NewMsgTurnBridge(clientCtx.GetFromAddress().String(), enable)
			if err := msg.ValidateBasic(); err != nil {
				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	flags.AddTxFlagsToCmd(cmd)
	return cmd
}
