### Title
Malicious CKD Participant Can Corrupt Aggregate Output Without Verification — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly aggregates every participant's `(norm_big_y, norm_big_c)` contribution with no zero-knowledge proof, commitment binding, or consistency check. A single malicious participant can send arbitrary group elements, causing the coordinator to output a silently corrupted `CKDOutput` that all honest parties accept as valid.

---

### Finding Description

In `do_ckd_coordinator`, after computing its own share, the coordinator loops over every other participant's response and unconditionally adds the received values:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

The honest computation each participant is supposed to perform is:

```
big_y  = y_i · G
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + y_i · app_pk
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
``` [2](#0-1) 

There is **no ZK proof of correct formation**, no Pedersen commitment binding `big_y` to `big_c`, and no consistency check that the received `big_y` and `big_c` are related to the participant's actual signing share. The coordinator has no mechanism to distinguish a correctly formed share from an arbitrary pair of group elements.

A malicious participant `j` can instead send:

```
norm_big_y' = norm_big_y_honest + Δ_Y
norm_big_c' = norm_big_c_honest + Δ_C
```

for any attacker-chosen group elements `Δ_Y`, `Δ_C`. The coordinator then computes:

```
Y_agg = Y_honest + Δ_Y
C_agg = C_honest + Δ_C
```

The application decrypts the confidential key as `C_agg − app_sk · Y_agg`, which equals:

```
msk · H(pk, app_id) + (Δ_C − app_sk · Δ_Y)
```

This is not the intended confidential key. The output is silently wrong; no error is raised.

The analog to the BondVault sandwich attack is direct: just as an attacker manipulates the pool spot price so the DAO commits BASE at a bad rate, a malicious CKD participant manipulates their share contribution so the coordinator aggregates a corrupted output — and in both cases the honest protocol machinery accepts the manipulated state without complaint.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The coordinator returns a `CKDOutput` that is structurally valid (two non-zero group elements) but cryptographically wrong. Any TEE application that calls `unmask(app_sk)` on this output will derive an incorrect, unpredictable secret. Because the corruption is silent, the honest coordinator and all honest participants have no indication that the output is invalid. The master secret key is never reconstructed, so this does not rise to secret extraction, but the derived confidential key is permanently corrupted for that invocation.

---

### Likelihood Explanation

**High.** Any single participant in the CKD protocol can trigger this. The attacker needs only to be a legitimate participant (no key leakage, no external compromise required). The attack requires sending two arbitrary group elements instead of the honest computation — a trivial modification for a participant who controls their own protocol implementation. There is no retry or detection mechanism; the coordinator outputs the corrupted value and terminates normally.

---

### Recommendation

Add a **Chaum-Pedersen / sigma-protocol ZK proof** that binds `big_y` and `big_c` to the same secret randomness `y_i` and to the participant's committed signing share `x_i`. Concretely, each participant should prove in zero knowledge:

- `big_y = y_i · G` (knowledge of `y_i`)
- `big_c = x_i · H(pk ‖ app_id) + y_i · app_pk` (correct ElGamal formation using the same `y_i` and the committed `x_i`)

The coordinator must verify all proofs before aggregating. This is the standard fix for unverified ElGamal share aggregation (see e.g. Pedersen's VSS or the PVSS literature). Alternatively, use a commit-then-reveal round so participants cannot adaptively choose their contribution after seeing others'.

---

### Proof of Concept

**Setup:** 3 participants, threshold 3, coordinator = participant 0. Participant 1 is malicious.

**Honest flow** (`compute_signature_share` for participant 1): [3](#0-2) 

**Malicious flow:** Participant 1 replaces `do_ckd_participant` with a version that sends `(G, G)` (the generator point) instead of its honest `(norm_big_y, norm_big_c)`:

```rust
// Malicious participant sends arbitrary points
chan.send_private(waitpoint, coordinator, &(ElementG1::generator(), ElementG1::generator()))?;
```

**Coordinator aggregation** (no verification): [1](#0-0) 

**Result:** The coordinator outputs `CKDOutput::new(Y_honest + G, C_honest + G)`. The application decrypts `(C_honest + G) − app_sk · (Y_honest + G) = msk · H(pk, app_id) + G(1 − app_sk)`, which is not the intended confidential key. The protocol returns `Ok(Some(ckd_output))` with no error, and all honest parties accept the corrupted output.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
