### Title
Unbounded `pool_member_epoch_balance` Trace Causes Gas Exhaustion in `claim_rewards` - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over every entry in a delegator's `pool_member_epoch_balance` trace without any bound. Because every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a distinct epoch appends a new checkpoint to this trace, a delegator who changes their balance once per epoch over many epochs will eventually make their own `claim_rewards` (and `pool_member_info_v1`) calls fail due to gas exhaustion, permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`: [2](#0-1) 

The underlying `insert` implementation appends a **new** checkpoint whenever the key (epoch+K) differs from the last stored key, and only overwrites when the key matches: [3](#0-2) 

`set_member_balance` / `increase_member_balance` is called from two unprivileged entry points:

- `add_to_delegation_pool` (line 242): calls `increase_member_balance` on every top-up.
- `exit_delegation_pool_intent` (line 278): calls `set_member_balance` on every partial-exit intent. [4](#0-3) [5](#0-4) 

If a delegator performs one balance-changing operation per epoch, each operation in a new epoch appends a distinct checkpoint. After N such epochs, the trace has N entries. When `claim_rewards` is later called, the loop must iterate all N entries before returning, consuming O(N) gas.

`claim_rewards` calls `calculate_rewards` directly: [6](#0-5) 

There is no cap on the trace length anywhere in the codebase.

---

### Impact Explanation

Once the trace is large enough that iterating it exceeds the Starknet transaction gas limit, `claim_rewards` reverts on every call. The delegator's accumulated yield becomes permanently unclaimable â€” a **permanent freezing of unclaimed yield** (High severity). The delegator's principal is also locked until `exit_delegation_pool_action` can be called, but that path does not iterate the trace, so only rewards are permanently frozen.

---

### Likelihood Explanation

Any delegator who actively manages their position â€” topping up or partially exiting once per epoch â€” naturally grows the trace by one entry per epoch. At typical Starknet epoch lengths, a delegator active for hundreds of epochs accumulates hundreds of trace entries. No special adversarial intent is required; the condition arises from normal long-term usage. The minimum cost per entry is one on-chain transaction plus the minimum delegation amount, making deliberate self-griefing cheap.

---

### Recommendation

Cap the maximum number of entries in `pool_member_epoch_balance` per pool member, analogous to the `MAX_NFT_APPROVALS` fix in the reference report. A constant such as `MAX_BALANCE_TRACE_ENTRIES` should be checked inside `set_member_balance` before appending a new checkpoint:

```cairo
fn set_member_balance(
    ref self: ContractState, pool_member: ContractAddress, amount: Amount,
) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
+   assert!(trace.length() < MAX_BALANCE_TRACE_ENTRIES, "trace cap exceeded");
    ...
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
```

Alternatively, require delegators to call `claim_rewards` before any balance-changing operation, resetting `entry_to_claim_from` and pruning already-processed entries, so the effective working set of the loop stays bounded.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with the minimum amount.
2. Each epoch, delegator calls `add_to_delegation_pool` with 1 token. Because `get_epoch_plus_k()` returns a strictly larger epoch key each time, `trace.insert` appends a new `PoolMemberBalanceCheckpoint` rather than overwriting the last one.
3. After N epochs, `pool_member_epoch_balance.entry(delegator).length() == N`.
4. Delegator calls `claim_rewards`. Inside `calculate_rewards`, the `while` loop at line 859 iterates all N entries. For sufficiently large N (bounded only by the Starknet block gas limit), the transaction reverts.
5. Every subsequent `claim_rewards` call also reverts. The delegator's yield is permanently frozen. [7](#0-6) [8](#0-7)

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
