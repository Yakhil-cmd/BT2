### Title
Unauthenticated Invoke Bypasses `__validate__` via `skip_stateful_validations` Deploy-Account UX Shortcut — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function skips the account's `__validate__` entry-point for any Invoke transaction whose nonce is `1` and whose sender address appears in `account_tx_in_pool_or_recent_block`. The check is purely address-based: it does not verify that the pending mempool entry is a deploy-account submitted by the same user. Any observer can therefore inject an Invoke with an arbitrary (invalid) signature from a victim address the moment that victim's deploy-account lands in the mempool, and the gateway will admit it without ever running signature verification.

### Finding Description

`skip_stateful_validations` implements a UX shortcut: when a user sends `deploy_account` (nonce 0) and `invoke` (nonce 1) together, the gateway cannot yet run `__validate__` on the invoke because the account does not exist on-chain yet. The shortcut is:

```
tx_nonce == 1  AND  account_nonce == 0  AND  account_tx_in_pool_or_recent_block(sender) == true
  ⟹  skip_validate = true  ⟹  __validate__ is never called
``` [1](#0-0) 

The guard `account_tx_in_pool_or_recent_block` returns `true` whenever **any** transaction from that address is in the pool or the mempool's recent-block state — it does not check that the pooled transaction is a deploy-account, nor that it was submitted by the same party as the incoming invoke. [2](#0-1) 

The stateless validator only checks signature **size**, not validity: [3](#0-2) 

`validate_by_mempool` checks nonce ordering and fee-escalation rules, but not the signature: [4](#0-3) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false` and the blockifier's `StatefulValidator::perform_validations` returns immediately after `perform_pre_validation_stage` without ever calling `__validate__`: [5](#0-4) [6](#0-5) 

### Impact Explanation

An attacker can inject an Invoke transaction carrying an invalid (or empty) signature into the mempool for any victim address that has a pending deploy-account. Because the mempool accepts the transaction (no signature check at that layer), the invalid transaction:

1. Occupies a mempool slot.
2. Can displace the victim's legitimate Invoke via fee-escalation replacement (attacker sets a higher tip/gas price).
3. Forces the batcher to waste execution resources when it eventually runs `__validate__` and reverts the transaction.
4. Denies the victim's first post-deploy invoke from being sequenced in the same block as the deploy-account, breaking the UX guarantee the shortcut was designed to provide.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The mempool is observable. Any time a user broadcasts a deploy-account transaction, an attacker can immediately craft and submit a nonce-1 Invoke from the same address with a garbage signature and a higher fee. No privileged access is required; the only prerequisite is that the victim's deploy-account is in the mempool, which is the exact scenario the shortcut is designed for.

### Recommendation

Mirror the approach used in the Python-blockifier path (`PyValidator::should_run_stateful_validations`), which requires the caller to supply the concrete `deploy_account_tx_hash` and verifies it is present and unprocessed: [7](#0-6) 

Concretely:

1. Extend the RPC/internal invoke representation to carry an optional `deploy_account_tx_hash` field (already present in the Starknet spec).
2. In `skip_stateful_validations`, replace the address-only `account_tx_in_pool_or_recent_block` check with a lookup that confirms the mempool contains a **deploy-account** transaction with that exact hash from the same sender.
3. If the hash is absent or does not match a deploy-account in the pool, do not skip `__validate__`.

### Proof of Concept

```
// Precondition: victim Alice has submitted deploy_account (nonce 0).
// Alice's address is now visible in the mempool.

// Attacker constructs an Invoke with:
//   sender_address = Alice's address
//   nonce          = 1
//   signature      = [0x0]   // invalid / garbage
//   resource_bounds = valid non-zero bounds (passes stateless check)

// Gateway flow:
// 1. StatelessTransactionValidator::validate  → passes (signature size ≤ limit)
// 2. convert_rpc_tx_to_internal               → succeeds
// 3. extract_state_nonce_and_run_validations:
//      get_nonce_from_state(Alice)             → Nonce(0)   (not yet on-chain)
//      validate_state_preconditions            → passes (0 ≤ 1)
//      validate_by_mempool                     → passes (no dup hash, fee ok)
//      skip_stateful_validations:
//        tx.nonce() == 1  ✓
//        account_nonce == 0  ✓
//        account_tx_in_pool_or_recent_block(Alice) == true  ✓  (deploy_account is pooled)
//        → returns true  (skip __validate__)
//      run_validate_entry_point(skip=true)     → __validate__ NOT called
// 4. mempool_client.add_tx                    → attacker's invalid Invoke admitted

// Result: attacker's unsigned Invoke is now in the mempool.
// If attacker's fee > Alice's legitimate Invoke fee, Alice's Invoke is evicted.
// Batcher later executes the invalid Invoke → __validate__ fails → revert.
// Alice's deploy_account succeeds but her Invoke is lost.
``` [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
