### Title
`MsgTurnBridge` Has No Access Control and No Implementation — Any Caller Succeeds, Bridge Circuit Breaker Is Permanently Broken - (File: x/cronos/keeper/msg_server.go)

### Summary
`MsgTurnBridge` is a privileged Cosmos SDK message intended to enable or disable the Cronos bridge, callable only by addresses holding the `CanTurnBridge` permission (or the `CronosAdmin`). Its server-side handler contains neither an authorization check nor any state-modifying logic — it unconditionally returns `nil, nil`. Any unprivileged address can submit the message and receive a success response, while the bridge's emergency-stop capability is permanently non-functional.

### Finding Description
The permissions system in `x/cronos/keeper/permissions.go` defines `CanTurnBridge` as a discrete permission bit, and ADR-009 explicitly lists `MsgTurnBridge` as one of the two restricted messages that must be gated behind this permission. `UpdateTokenMapping` — the other restricted message — correctly enforces the check:

```go
// UpdateTokenMapping — correct
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
```

`TurnBridge`, however, is implemented as:

```go
// TurnBridge — no check, no logic
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

Two distinct defects are present simultaneously:
1. **Missing authorization guard** — `HasPermission(ctx, ..., CanTurnBridge)` is never called. Any signer can submit `MsgTurnBridge` and the chain accepts it.
2. **Missing implementation** — no bridge-enabled flag is read or written. The message is a no-op regardless of who sends it.

### Impact Explanation
- **Auth bypass (High):** The `CanTurnBridge` permission check — a core part of Cronos's bridge authorization model — is completely absent. Any unprivileged address can call the message that is explicitly documented as restricted to permissioned operators.
- **Permanent loss of emergency-stop capability (High):** The bridge circuit breaker is the only on-chain mechanism for halting bridge flows in an emergency (e.g., an active exploit draining bridged assets). Because the handler is a no-op, neither the admin nor any permissioned operator can ever disable the bridge through the intended path. This constitutes a permanent inability to halt bridge/conversion flows under normal network assumptions.

### Likelihood Explanation
The entry path is a standard, publicly routable Cosmos SDK `Msg` transaction. No special privileges, leaked keys, or off-chain conditions are required to reach the handler. Any wallet can construct and broadcast `MsgTurnBridge`. The defect is present in every block since the message type was introduced.

### Recommendation
1. Add the `CanTurnBridge` permission guard at the top of `TurnBridge`, mirroring `UpdateTokenMapping`.
2. Implement the actual bridge-enable/disable logic (read/write a `BridgeEnabled` parameter or equivalent state key).
3. Add a test case that verifies an unpermissioned sender is rejected and a permissioned sender successfully toggles bridge state.

### Proof of Concept
```
// Attacker (any address) submits:
MsgTurnBridge { sender: <attacker_address>, enable: false }

// Chain response: success (nil error, nil response body)
// Bridge state: unchanged — bridge remains enabled
// Admin attempting the same call: also a no-op
```

// UpdateTokenMapping implements the grpc method
func (k msgServer) UpdateTokenMapping(goCtx context.Context, msg *types.MsgUpdateTokenMapping) (*types.MsgUpdateTokenMappingResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)

	// check permission
	if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
		return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
	}

	// msg is already validated
	if err := k.RegisterOrUpdateTokenMapping(ctx, msg); err != nil {
		return nil, err
	}
	return &types.MsgUpdateTokenMappingResponse{}, nil
}
