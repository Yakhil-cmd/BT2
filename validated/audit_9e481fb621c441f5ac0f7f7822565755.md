### Title
Unchecked Return Value of Exit Precompile `call()` in `EvmErc20`/`EvmErc20V2` Enables Permanent Token Burn Without Withdrawal - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear()` and `withdrawToEthereum()` by first burning the caller's ERC-20 tokens via `_burn()`, then invoking the exit precompile via a low-level assembly `call()`. The return value `res` of that `call()` is captured but **never checked**. If the precompile call fails for any reason, the burn is irreversible but no NEAR-side withdrawal promise is scheduled, permanently destroying the user's bridged tokens.

---

### Finding Description

In `EvmErc20.sol`, both exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert on failure
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum()`: [2](#0-1) 

And in both functions of `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` can return failure (EVM `call()` returns 0) in several conditions visible in its `run()` implementation: [5](#0-4) 

It also fails if the precompile is paused: [6](#0-5) 

And it fails if the NEP-141 mapping lookup fails: [7](#0-6) 

In all these failure cases, the EVM `call()` returns `0` into `res`, but because `res` is never inspected, the Solidity function returns successfully. The `_burn()` that already executed is not rolled back.

---

### Impact Explanation

When the exit precompile call silently fails:

1. The user's ERC-20 balance on Aurora is permanently reduced (burned).
2. No NEAR-side `ft_transfer` or `ft_transfer_call` promise is scheduled.
3. The NEP-141 tokens remain locked in the Aurora engine account forever.
4. There is no recovery path — the `error_refund` feature only handles failures in the NEAR-side promise callback, not failures in the EVM-side precompile call itself.

This constitutes **permanent freezing of funds** (bridged NEP-141 tokens become unrecoverable) and is equivalent to direct theft from the user's perspective. [8](#0-7) 

---

### Likelihood Explanation

Concrete failure scenarios reachable without admin compromise:

1. **Precompile paused by governance**: The `ExitToNear` precompile can be paused. If a user calls `withdrawToNear()` while the precompile is paused, the burn succeeds but the withdrawal does not. The user loses funds.
2. **Delegate-call context**: If `withdrawToNear` is called via a proxy that changes `context.address`, the precompile rejects with `ERR_INVALID_IN_DELEGATE` — the burn already happened.
3. **Out-of-gas within precompile**: If the caller provides insufficient gas for the precompile's internal operations, the sub-call fails silently.

The `error_refund` feature in `ExitToNearPrecompileCallbackArgs` only covers NEAR-side promise failures, not EVM-side precompile call failures: [9](#0-8) 

---

### Recommendation

Check the return value of every low-level `call()` to the exit precompiles and revert on failure. This ensures the `_burn()` is rolled back atomically if the precompile cannot schedule the withdrawal:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum()` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` with a registered NEP-141 mapping.
2. Bridge NEP-141 tokens into Aurora so a user holds ERC-20 balance.
3. Pause the `ExitToNear` precompile (or arrange for the precompile call to fail via any of the conditions above).
4. User calls `withdrawToNear(recipient, amount)`.
5. `_burn(user, amount)` executes — ERC-20 balance decremented.
6. Assembly `call()` to precompile returns `0` (failure) — `res` is ignored.
7. Function returns without reverting.
8. User's ERC-20 tokens are gone; no NEP-141 transfer was scheduled; funds are permanently frozen. [1](#0-0) [6](#0-5) [5](#0-4)
