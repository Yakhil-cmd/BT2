### Title
Signature Verification Bypass via `skip_stateful_validations` Allows Unsigned Invoke Transactions to Enter Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (the account contract's signature check) for any Invoke transaction with nonce=1 when the sender address appears in the mempool. An unprivileged attacker who observes a pending `deploy_account` transaction in the public P2P mempool can submit an Invoke with nonce=1 and a completely invalid signature for the victim's account. The gateway admits this transaction to the mempool without ever verifying the signature, violating the invariant that every admitted transaction must carry a valid account signature.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` (lines 429–461) returns `true` — meaning "skip `__validate__`" — whenever all three conditions hold simultaneously:

1. The transaction is an `ExecutableTransaction::Invoke` with `tx.nonce() == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [1](#0-0) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

This flag is consumed by `StatefulValidator::perform_validations` in the blockifier: [3](#0-2) 

When `execution_flags.validate == false`, the `__validate__` entry point — the only place the account contract verifies the caller's signature — is never invoked. `perform_pre_validation_stage` still runs (nonce increment, fee-bound check, balance check), but none of those steps inspect the cryptographic signature. [4](#0-3) 

The stateless validator only checks signature *length*, not validity: [5](#0-4) 

The mempool's `validate_tx` path checks nonce gaps and duplicates but never verifies the cryptographic signature. Therefore, an attacker can:

1. Observe a pending `deploy_account` for victim account A in the public P2P mempool (the mempool broadcasts transactions via Gossipsub).
2. Construct an Invoke transaction: `sender_address = A`, `nonce = 1`, `signature = [0xDEAD, 0xBEEF]` (arbitrary garbage).
3. Submit it to the gateway's `/gateway/add_transaction` or `/gateway/add_rpc_transaction` endpoint — both are unauthenticated and have no Host-header validation: [6](#0-5) 

4. The gateway calls `skip_stateful_validations` → returns `true` → `__validate__` is skipped → transaction is forwarded to the mempool. [7](#0-6) 

5. The mempool accepts the transaction (nonce=1 is within the allowed gap from account nonce=0).
6. The batcher later executes `deploy_account` (nonce=0) → account A is deployed. Then it executes the attacker's Invoke (nonce=1) → the blockifier calls `__validate__` → signature verification fails → transaction reverts → **fees are charged to account A**.

The comment in the code acknowledges the assumption but does not guard against third-party injection: [8](#0-7) 

### Impact Explanation

**Broken invariant:** Every transaction admitted to the mempool must carry a signature that satisfies the account contract's `__validate__` entry point. The `skip_stateful_validations` path breaks this invariant for nonce=1 Invoke transactions targeting any account with a pending `deploy_account`.

**Concrete harm:**
- **Fee drain:** The reverted Invoke is included in the block and fees are deducted from the victim's pre-funded balance (the balance must exist for the `deploy_account` to succeed).
- **Nonce-slot hijacking:** If the attacker's invalid transaction reaches the mempool before the victim's legitimate nonce=1 Invoke, the victim's transaction is rejected as a duplicate nonce (or must win a fee-escalation race). This matches the "High" impact category: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
