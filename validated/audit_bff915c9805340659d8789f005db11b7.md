### Title
`calculateNFTRoyalties`: `nftSellerNetAmount` Ignores Royalty Deduction, Causing Inflated "Net Proceeds" Display in TokenForNFT Offer Acceptance - (File: `packages/gui/src/components/offers/utils.ts`)

### Summary

In `calculateNFTRoyalties` (`packages/gui/src/components/offers/utils.ts`), `nftSellerNetAmount` is unconditionally set to the raw `amount` with the royalty subtraction commented out. In the `TokenForNFT` exchange flow (buyer creates offer, NFT seller accepts), the GUI displays this value as both "Net Proceeds" and "You will receive" to the NFT seller at the moment of offer acceptance. Because the royalty deduction is missing, the NFT seller is shown an inflated amount — equal to the full offered price — when the actual on-chain settlement will deliver `amount − royaltyAmount` to them.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The correct subtraction is commented out, leaving `nftSellerNetAmount = amount` for all exchange types. [1](#0-0) 

In `NFTOfferViewer.tsx`, `nftSaleInfo` is computed by calling `calculateNFTRoyalties` and then used in two ways that are both visible to the NFT seller at acceptance time:

**1. `overrideNFTSellerAmount`** — passed to `NFTOfferSummary` as the "You will receive" figure:

```typescript
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

**2. "Net Proceeds" label** — rendered only in the `TokenForNFT` case with the tooltip "The net proceeds include the asking price, minus any associated creator fees":

```tsx
{exchangeType === NFTOfferExchangeType.TokenForNFT && (
  ...
  <FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} />
  ...
)}
``` [3](#0-2) 

The tooltip explicitly promises the royalty is subtracted, but the code does not subtract it. [4](#0-3) 

The same inflated value is also shown in the NFT offer editor (`NFTOfferEditor.tsx`) when the offer creator previews the deal: [5](#0-4) 

### Impact Explanation

When a buyer creates a `TokenForNFT` offer for an NFT that carries a non-zero royalty, the NFT seller who opens the offer to accept it sees:

- **"You will receive: X tokens"** (inflated — should be `X − royaltyAmount`)
- **"Net Proceeds: X tokens"** (inflated — tooltip says royalty is subtracted, but it is not)

The royalty line item is shown separately and correctly, but the headline figures the seller uses to decide whether to accept are wrong. For a 20 % royalty on a 100 XCH offer, the seller sees "Net Proceeds: 100 XCH" and "You will receive: 100 XCH" but on-chain settlement delivers only 80 XCH. The seller is induced to accept an offer they might otherwise reject.

This maps to the **High** allowed impact: *"Corruption … of … offer … state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation

- Any `TokenForNFT` offer involving an NFT with a non-zero `royaltyPercentage` triggers the bug — no special attacker action is required.
- NFT royalties are common (the GUI even warns when royalty ≥ 20 %).
- The NFT seller has no reason to distrust the GUI's own "Net Proceeds" label, especially given the tooltip that explicitly states the royalty is deducted.

### Recommendation

Restore the royalty subtraction in `calculateNFTRoyalties` for the `TokenForNFT` case:

```typescript
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? parseFloat((amount - royaltyAmount).toFixed(12))
    : amount;
```

The `makerFee` subtraction in the original commented-out code should be evaluated separately; the maker fee is paid by the offer creator (the buyer), not deducted from the NFT seller's proceeds, so it should not be subtracted from `nftSellerNetAmount`. [6](#0-5) 

### Proof of Concept

1. An NFT exists with `royaltyPercentage = 2000` (20 %).
2. A buyer creates a `TokenForNFT` offer: "I offer 100 XCH for NFT X."
3. The NFT seller opens the offer in the GUI to review it.
4. `calculateNFTRoyalties(100, makerFee, 20, TokenForNFT)` is called:
   - `royaltyAmount = 20 % × 100 = 20`
   - `nftSellerNetAmount = 100` ← **bug: should be 80**
5. The GUI displays:
   - "You will receive: **100 XCH**" (via `overrideNFTSellerAmount`)
   - "Net Proceeds: **100 XCH**" (via `nftSaleInfo.nftSellerNetAmount`)
   - "Creator Fee (20%): 20 XCH" (correct)
6. The seller accepts, believing they will receive 100 XCH. On-chain, they receive 80 XCH; the creator receives 20 XCH.

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
