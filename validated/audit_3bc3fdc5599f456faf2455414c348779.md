### Title
Bank Precompile `transfer` and `burn` Methods Lack Caller-Authorization Check, Enabling Unauthorized Native Token Theft — (`x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile at `0x0000000000000000000000000000000000000064` exposes `transfer(address sender, address recipient, uint256 amount)` and `burn(address recipient, uint256 amount)` methods that accept an arbitrary `sender`/`recipient` address as a call argument but never verify that `contract.Caller() == sender`. Any unprivileged EVM contract can therefore drain `evm/<contract_address>`-denominated native tokens from any victim address without the victim's consent.

### Finding Description
`BankContract.Run` in `x/cronos/keeper/precompiles/bank.go` handles three mutating methods. For `transfer`, the sender is read directly from ABI-decoded call arguments:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(to); err != nil { ... }   // only checks SDK module-account list
denom := EVMDenom(contract.Caller())                    // denom tied to calling contract
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no assertion that `contract.Caller() == sender`. The same pattern applies to `burn`, where `recipient` (the address whose tokens are burned) is taken from arguments with no caller check:

```go
recipient := args[0].(common.Address)
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
``` [2](#0-1) 

The only guard present, `checkBlockedAddr`, checks the Cosmos SDK bank module's own blocked-address list (module accounts), not the Cronos blocklist and not whether the caller owns the funds: [3](#0-2) 

The intended usage pattern (shown in `TestBank.sol`) always passes `msg.sender` as the `sender` argument, but this is a convention enforced nowhere in the precompile itself: [4](#0-3) 

### Impact Explanation
The denom operated on is `evm/<calling_contract_address>`. Any user who holds native `evm/X` tokens (obtained by calling `bank.mint` through contract X, i.e., converting ERC20 balance to native form) is vulnerable to having those tokens stolen or burned by the owner of contract X — or by any other contract that calls the precompile with a spoofed `sender`. This constitutes an **unauthorized transfer and burn of precompile-controlled assets**, which is a Critical impact under the allowed scope.

### Likelihood Explanation
The bank precompile is a live, permissionlessly callable precompile at a fixed address. Any deployed EVM contract can call it. A malicious contract author who deploys a token contract and attracts users to convert balances to native form can immediately drain all converted balances. No privileged access, leaked keys, or governance action is required.

### Recommendation
Add a caller-authorization check at the top of the `TransferMethodName` and `BurnMethodName` cases:

```go
// For transfer:
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer on behalf of sender")
}

// For burn:
if contract.Caller() != recipient {
    return nil, errors.New("caller is not authorized to burn on behalf of recipient")
}
```

This mirrors the fix applied in the referenced ERC777 report: override the relevant entry points to enforce the missing authorization check at the function level rather than relying on callers to pass the correct address.

### Proof of Concept

1. Attacker deploys `MaliciousToken` at address `0xDEAD`. The bank precompile denom for this contract is `evm/0xDEAD`.
2. `MaliciousToken` exposes a public `deposit()` that calls `bank.mint(msg.sender, amount)` after burning the caller's ERC-20 balance — a standard "move to native" pattern. Users deposit, receiving `evm/0xDEAD` native tokens.
3. Attacker calls a `steal(address victim)` function on `MaliciousToken`:

```solidity
function steal(address victim, uint256 amount) external {
    // calls bank precompile at 0x64
    // sender = victim, recipient = attacker — no check in precompile
    IBankModule(0x64).transfer(victim, msg.sender, amount);
}
```

4. The bank precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, coins)` with no authorization check. The victim's `evm/0xDEAD` native balance is transferred to the attacker.
5. The attacker calls `bank.burn(victim, remaining)` to destroy any remaining victim balance, again with no check.

Neither the `BlockAddressesDecorator` (which only runs during `ctx.IsCheckTx()`) nor the `ProposalHandler.ValidateTransaction` (which only checks tx signers and EVM `To` addresses) intercepts this intra-execution precompile call: [5](#0-4) [6](#0-5)

func (h *ProposalHandler) ValidateTransaction(tx sdk.Tx, txBz []byte) error {
	if len(h.blocklist) == 0 {
		// fast path, accept all txs
		return nil
	}

	var err error
	if tx == nil {
		tx, err = h.TxDecoder(txBz)
		if err != nil {
			return err
		}
	}

	sigTx, ok := tx.(signing.SigVerifiableTx)
	if !ok {
		return fmt.Errorf("tx of type %T does not implement SigVerifiableTx", tx)
	}

	signers, err := sigTx.GetSigners()
	if err != nil {
		return err
	}
	for _, signer := range signers {
		encoded, err := h.addressCodec.BytesToString(signer)
		if err != nil {
			return fmt.Errorf("invalid bech32 address: %s, err: %w", signer, err)
		}
		if _, ok := h.blocklist[encoded]; ok {
			return fmt.Errorf("signer is blocked: %s", encoded)
		}
	}
