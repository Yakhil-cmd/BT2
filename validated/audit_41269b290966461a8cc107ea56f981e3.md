### Title
Unchecked Precompile Return Value After ERC-20 Burn Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn ERC-20 tokens **before** calling the `ExitToNear` precompile, and neither contract checks the return value of that precompile call. If the precompile fails for any reason, the ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer on the NEAR side and no refund path, resulting in permanent loss of user funds.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear` and `withdrawToEthereum` functions follow this pattern:

**`EvmErc20.sol` (lines 53–63):**
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure
    }
}
```

**`EvmErc20V2.sol` (lines 53–63):**
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                                // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)

    }
}

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can fail and return an error in several reachable paths:

1. **ERC-20 not registered with a NEP-141**: `get_nep141_from_erc20` returns an error, causing the precompile to return `Err(ExitError::...)`.
2. **Input format mismatch**: `EvmErc20.sol` (old version) sends `\x01 | amount_b (32 bytes) | recipient`, but when the engine is compiled with the `error_refund` feature, the precompile's `parse_input` expects `\x01 | refund_address (20 bytes) | amount_b (32 bytes) | recipient`. The precompile misparses the input — the first 20 bytes of `amount_b` are consumed as the refund address, leaving only 12 bytes of `amount_b` plus the start of `recipient` to be parsed as the 32-byte amount field. The subsequent recipient parsing then receives malformed bytes and fails with an invalid NEAR account ID error.
3. **Any other precompile error path** (e.g., `ERR_ETH_ATTACHED_FOR_ERC20_EXIT`, `ERR_INVALID_IN_STATIC`). [3](#0-2) [4](#0-3) 

When the precompile call fails (returns 0), the EVM sub-call reverts its own state changes, but the outer call's `_burn` is **not** reverted. The Solidity code does not check `res` and does not revert. The transaction succeeds, the tokens are gone, and no NEAR promise is ever scheduled.

The `error_refund` callback path (`exit_to_near_precompile_callback` → `refund_on_error`) only fires when the precompile **succeeds** in scheduling a NEAR promise that subsequently fails. It provides zero protection when the precompile itself fails before any promise is created. [5](#0-4) [6](#0-5) 

The test suite explicitly acknowledges that without `error_refund` tokens are lost, but even with `error_refund` enabled the precompile-failure path has no recovery: [7](#0-6) 

---

### Impact Explanation

When the precompile call silently fails:
- The user's ERC-20 balance is permanently reduced to zero.
- No NEP-141 tokens are transferred on the NEAR side.
- No refund callback is ever triggered.
- There is no on-chain recovery mechanism.

The result is **permanent, irrecoverable destruction of user funds** — a critical permanent fund freeze. The total supply of the ERC-20 token decreases while the NEP-141 supply held by the Aurora contract remains unchanged, breaking the 1:1 accounting invariant of the bridge.

---

### Likelihood Explanation

The scenario is reachable by any unprivileged EVM user who:

1. Holds tokens in an `EvmErc20` contract that was deployed with the old `EvmErc20.sol` bytecode while the engine runs with `error_refund` enabled (input format mismatch → precompile parse failure → silent burn).
2. Calls `withdrawToNear` or `withdrawToEthereum` on any `EvmErc20` contract whose ERC-20 address is not registered in the engine's NEP-141 mapping (e.g., a contract deployed by a third party that inherits `EvmErc20` but was never registered via `deploy_erc20_token`).

Both conditions are reachable without any admin compromise. The first condition is particularly relevant during any engine upgrade that enables `error_refund` without simultaneously upgrading all deployed `EvmErc20` contracts to `EvmErc20V2`.

---

### Recommendation

Add a revert on precompile call failure in both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile fails for any reason, the entire transaction reverts (including the `_burn`), so no tokens are lost. The same fix applies to `withdrawToEthereum` in both contracts. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

1. Deploy Aurora engine with `error_refund` feature enabled.
2. Deploy an `EvmErc20` contract (old version, without sender in input) mapped to a NEP-141.
3. Mint 1000 tokens to `alice`.
4. `alice` calls `withdrawToNear("alice.near", 1000)`.
5. `_burn(alice, 1000)` exec
