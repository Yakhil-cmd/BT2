### Title
Missing Access Control on `update_rewards` Allows Any Caller to Block Consensus Reward Distribution — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` to mark the current block as reward-processed without distributing any rewards, permanently blocking all stakers from receiving consensus block rewards for that block.

---

### Finding Description

`docs/spec.md` line 1645 explicitly states the access control for `update_rewards`:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo:1449–1507` enforces no such restriction:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);   // global write, no auth

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // exits without distributing
    }
    ...
}
```

The critical sequence is:

1. `last_reward_block` is a **single global** storage variable — not per-staker.
2. It is written **before** the `disable_rewards` branch is evaluated.
3. Any caller can pass `disable_rewards: true`, causing the function to write the current block number and return immediately without distributing rewards.
4. All subsequent calls in the same block — including the legitimate sequencer call — fail with `REWARDS_ALREADY_UPDATED`. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once consensus rewards are active (`is_pre_consensus()` returns `false`), an attacker calling `update_rewards(valid_staker, disable_rewards: true)` once per block:

- Writes `last_reward_block = current_block`, consuming the single allowed call for that block.
- Returns without invoking `_update_rewards`, so no staker's `unclaimed_rewards_own` is incremented and no pool receives rewards.
- Causes the sequencer's legitimate call to revert with `REWARDS_ALREADY_UPDATED`.

Because `last_reward_block` is global, **one transaction per block permanently denies block rewards to every staker and every delegator** for that block. Repeated every block, this freezes all consensus-era yield indefinitely. [4](#0-3) 

---

### Likelihood Explanation

**Medium.**

- The attack requires one cheap L2 transaction per block — no capital at risk.
- Valid active staker addresses are publicly discoverable from `NewStaker` events.
- The attack is only effective after `set_consensus_rewards_first_epoch` is called and K epochs have elapsed, but once active it can be executed continuously.
- On Starknet, the sequencer controls transaction ordering, which partially mitigates the risk if `update_rewards` is issued as a privileged system transaction. However, the code provides **no on-chain enforcement** of this assumption, making the protocol's security entirely dependent on an off-chain operational guarantee that is not expressed in the contract. [5](#0-4) [6](#0-5) 

---

### Recommendation

Add an explicit caller check at the top of `update_rewards` to enforce the spec's stated access control. For example, store the authorized sequencer address in contract storage during initialization and assert `get_caller_address() == sequencer_address` before any state mutation. Alternatively, use the existing roles component (already present in the contract) to gate this function to a dedicated `SEQUENCER_ROLE`. [7](#0-6) 

---

### Proof of Concept

```
// Attacker script — runs once per block after consensus rewards activate:

1. Monitor the chain for new blocks.
2. On each new block, read any valid active staker address S from on-chain events.
3. Submit: staking_contract.update_rewards(staker_address: S, disable_rewards: true)
4. This call succeeds (no auth check), writes last_reward_block = current_block,
   and returns without distributing rewards.
5. The sequencer's subsequent update_rewards call reverts with REWARDS_ALREADY_UPDATED.
6. All stakers and delegators receive zero block rewards for this block.
7. Repeat every block.
```

The attacker spends only gas. Every staker and delegator permanently loses the yield they would have earned for each attacked block. [8](#0-7) [2](#0-1)
