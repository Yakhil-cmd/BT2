### Title
Malicious Coordinator Can Distribute Split-Entropy `RerandomizationArguments` to Cause Undetectable Signing Failure - (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/mod.rs`)

---

### Summary

A malicious coordinator can construct two `RerandomizationArguments` structs that are identical in every field except `entropy`, distribute one to a subset of participants and the other to the rest, and cause the signing protocol to produce an invalid signature that fails the coordinator's own verification step. No honest participant can detect the inconsistency before the final verification, and the presignature is consumed.

---

### Finding Description

`RerandomizationArguments` is a plain public struct with all fields public, including `entropy: [u8; 32]`. [1](#0-0) 

`derive_randomness()` uses `entropy` as the HKDF key material: [2](#0-1) 

So two `RerandomizationArguments` that differ only in `entropy` produce two completely different scalars `delta_A ≠ delta_B`.

`rerandomize_presign()` is called **locally by each participant** with whatever args they received. It computes:

- `big_r' = delta * R`
- `k_i' = k_i / delta`
- `sigma_i' = (sigma_i + tweak * k_i) / delta` [3](#0-2) 

Each participant then computes their signature share using their own local `big_r'`:

```
s_i = h * λ_i * k_i' + r' * λ_i * σ_i'
```

where `r' = x_coord(big_r')`. [4](#0-3) 

Participants in subset A use `r_A = x_coord(delta_A * R)`, while participants in subset B use `r_B = x_coord(delta_B * R)`. These are different values (with overwhelming probability), so the shares are computed in two incompatible contexts.

The coordinator sums all shares and verifies using its own `presignature.big_r`: [5](#0-4) 

The sum `Σ s_i` is cryptographically invalid because it mixes shares from two different rerandomization contexts. The `sig.verify()` call at line 129 fails, and the coordinator returns `ProtocolError::AssertionFailed("signature failed to verify")`.

**There is no cross-participant consistency check on `entropy`, `delta`, or `big_r'` at any point in the signing protocol.** Non-coordinator participants simply compute their share and send it privately to the coordinator: [6](#0-5) 

They return `None` and have no way to detect that they were given inconsistent args.

---

### Impact Explanation

The signing protocol fails at the coordinator's verify step. The presignature `(big_r, k_i, sigma_i)` is consumed — reusing it would be cryptographically unsafe (nonce reuse). Honest participants cannot distinguish this failure from a legitimate abort. The coordinator can repeat this attack on every signing attempt, permanently denying signing for any presignature they are asked to coordinate.

This maps to: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

The coordinator role is chosen from the participant set and is not assumed to be fully trusted (the protocol is designed to tolerate malicious participants). The attack requires only that the coordinator call `RerandomizationArguments::new()` twice with different `entropy` bytes and route the two structs to different participants before calling `sign()`. This is entirely within the coordinator's control via the public API. No cryptographic assumption needs to be broken.

---

### Recommendation

Before participants send their signature shares, they should reach consensus on the `RerandomizationArguments` (or at minimum the `entropy` and derived `delta`). Concretely:

1. **Broadcast `entropy` via echo-broadcast** (the infrastructure already exists in `src/protocol/echo_broadcast.rs`) so all participants verify they hold the same value before proceeding.
2. Alternatively, derive `entropy` deterministically from a value all participants already agree on (e.g., a hash of the presignature `big_r` combined with a shared session nonce), removing the coordinator's ability to supply it unilaterally.

---

### Proof of Concept

```rust
// Same presign output for all participants
let presign = PresignOutput { big_r, k: k_i, sigma: sigma_i };

// Coordinator constructs two args with different entropy
let args_a = RerandomizationArguments::new(pk, tweak, msg_hash, big_r, participants.clone(), entropy_a);
let args_b = RerandomizationArguments::new(pk, tweak, msg_hash, big_r, participants.clone(), entropy_b);

let delta_a = args_a.derive_randomness().unwrap();
let delta_b = args_b.derive_randomness().unwrap();
assert_ne!(delta_a, delta_b); // guaranteed with overwhelming probability

// Subset A rerandomizes with args_a, subset B with args_b
let rerand_a = RerandomizedPresignOutput::rerandomize_presign(&presign, &args_a).unwrap();
let rerand_b = RerandomizedPresignOutput::rerandomize_presign(&presign, &args_b).unwrap();

// big_r values differ → r values differ → shares are incompatible
assert_ne!(rerand_a.big_r, rerand_b.big_r);

// Running sign() with mixed outputs produces a share sum that fails verify()
// → coordinator returns ProtocolError::AssertionFailed("signature failed to verify")
```

The `sig.verify()` call at [7](#0-6)  will fail because the aggregated `s` was computed with two different `r` values, while the coordinator's `big_r` (and thus `r`) corresponds to only one of the two deltas.
