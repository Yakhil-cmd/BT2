### Title
FROST `PresignOutput` Derives `Clone` With No Single-Use Enforcement, Enabling Nonce Reuse and Signing Share Extraction — (`src/frost/mod.rs`, `src/frost/redjubjub/sign.rs`)

---

### Summary

`PresignOutput<C>` unconditionally derives `Clone`. The `sign()` API takes the presignature by value (a move), which is the idiomatic Rust mechanism for enforcing single use, but deriving `Clone` completely defeats it. A caller can trivially call `presign.clone()` before each `sign()` invocation, reusing the same nonces for two different messages. The coordinator then holds two signature shares per participant that were produced with identical nonces but different challenge hashes, which is sufficient to algebraically extract each participant's private signing share via the standard FROST nonce-reuse equation.

---

### Finding Description

`PresignOutput<C>` is defined in `src/frost/mod.rs` with `#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]`: [1](#0-0) 

The struct holds `nonces: SigningNonces<C>` — the per-participant secret nonce scalars that FROST requires to be used exactly once — alongside the shared `commitments_map`. [2](#0-1) 

The public `sign()` entry point in `src/frost/redjubjub/sign.rs` takes `presignature: PresignOutput` by value: [3](#0-2) 

This move-by-value is the only mechanism intended to enforce single use. Because `Clone` is derived, any caller can write:

```rust
let presig = run_presign(...)?;
sign(..., presig.clone(), msg1, ...)?;  // first use
sign(..., presig.clone(), msg2, ...)?;  // reuse — identical nonces, different message
```

Both calls succeed. The library contains no atomic consumed-flag, no zeroize-on-use, and no documentation warning against cloning. The existing test helper in `src/frost/redjubjub/test.rs` already calls `presignature.clone()` when constructing signing protocols: [4](#0-3) 

Inside `do_sign_participant`, the nonces are consumed only locally via `Zeroizing::new(presignature.nonces)`, but this does not prevent a second invocation from using a previously cloned copy: [5](#0-4) 

---

### Impact Explanation

In FROST, each participant's signature share satisfies:

```
z_i = r_i + λ_i · x_i · c
```

where `r_i` is the nonce scalar, `λ_i` is the Lagrange coefficient, `x_i` is the private signing share, and `c` is the challenge hash (which binds the message). If the same `r_i` is used for two messages with challenges `c1 ≠ c2`:

```
z_i1 - z_i2 = λ_i · x_i · (c1 - c2)
x_i = (z_i1 - z_i2) / (λ_i · (c1 - c2))
```

A malicious coordinator who orchestrates two signing sessions using the same `commitments_map` (i.e., the same nonce commitments) receives both `z_i1` and `z_i2` from each honest participant and can recover every participant's `x_i`. This is a full extraction of all private signing shares from a single presignature reuse.

**Impact: Critical** — extraction of private signing shares (`x_i`) for all participants in the signing quorum.

---

### Likelihood Explanation

The `Clone` derive is public and unconditional. The existing test infrastructure already calls `.clone()` on presignatures as a matter of course. There is no API-level barrier, no documentation warning, and no runtime check. Any caller who stores a `PresignOutput` and passes it to two signing sessions — whether accidentally or under coordinator coercion — triggers the attack. The coordinator role is trusted in this library's threat model, but the library provides no defense against a coordinator that replays the same presignature.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput<C>`** in `src/frost/mod.rs`. The move-by-value API in `sign()` is the correct design; `Clone` defeats it entirely.
2. Add `ZeroizeOnDrop` to `PresignOutput<C>` (as is already done for the ECDSA `PresignOutput` in `src/ecdsa/robust_ecdsa/mod.rs`) so that nonce material is erased after use.
3. If serialization/deserialization of `PresignOutput` is required for persistence, document explicitly that a deserialized presignature must never be used more than once and that the stored copy must be deleted immediately after the first signing session. [6](#0-5)

### Proof of Concept

```rust
// Pseudocode — all types are from src/frost/redjubjub
let presig_outputs = run_presign(&participants, threshold, n, rng)?;

// Clone the presignature before the first sign call
let presig_clone: Vec<(Participant, PresignOutput)> = presig_outputs
    .iter()
    .map(|(p, o)| (*p, o.clone()))
    .collect();

// First signing session — msg1
let sigs1 = run_sign(&participants, coordinator, threshold, presig_outputs, msg1)?;

// Second signing session — same nonces, different message
let sigs2 = run_sign(&participants, coordinator, threshold, presig_clone, msg2)?;

// For each participant i, recover x_i:
// z_i1 and z_i2 are the signature shares from sigs1 and sigs2
// c1, c2 are the FROST challenge hashes for msg1 and msg2
// lambda_i is the Lagrange coefficient for participant i
//
// x_i = (z_i1 - z_i2) / (lambda_i * (c1 - c2))
//
// assert_eq!(x_i, keygen_output_i.private_share);
```

Both `run_sign` calls complete successfully because the library performs no nonce-reuse check. The coordinator, holding both sets of signature shares, recovers every participant's signing share with a single field division.
