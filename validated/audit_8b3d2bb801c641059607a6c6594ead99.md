### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
The factory enforces `maxAdminSpreadFeeE6` on `setPoolAdminFees` but applies **no equivalent cap** in `setPoolBinAdditionalFees`. A pool admin can set `addFeeBuyE6` or `addFeeSellE6` to `uint16.max` (65535 = 6.5535%) on any bin, imposing fees on traders that far exceed the factory-enforced ceiling.

### Finding Description
`MetricOmmPoolFactory.setPoolAdminFees` correctly guards the base admin spread fee: [1](#0-0) 

But `setPoolBinAdditionalFees` passes values straight through with no validation: [2](#0-1) 

The pool's `setBinAdditionalFees` only checks the bin index range: [3](#0-2) 

During every swap, the per-bin additional fee is added on top of the oracle-derived base spread fee: [4](#0-3) 

This means the effective fee charged to a trader in a bin is `baseFeeX64 + addFeeBuyE6/1e6`, with no factory-enforced ceiling on the second term.

### Impact Explanation
A pool admin can call `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` to set per-bin fees to 6.5535% on top of the base spread. Traders swapping through that bin pay the full inflated fee. The `maxAdminSpreadFeeE6` cap — the protocol's intended protection against excessive admin fees — is completely bypassed. This is a direct admin-boundary break: the pool admin exceeds the factory-enforced fee cap, causing traders to pay more than the protocol guarantees.

The extra fee accrues to LPs (not the admin directly), but the trader's principal loss is real and unbounded by the cap. The admin can effectively price-out or grief traders on specific bins without triggering any on-chain guard.

### Likelihood Explanation
Any pool admin (a semi-trusted role, not the factory owner) can execute this in a single transaction with no preconditions beyond holding the `poolAdmin[pool]` role. The call path is fully permissioned to the pool admin and requires no special setup.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

(Or introduce a dedicated `maxAdminBinAdditionalFeeE6` cap if per-bin fees are intended to have a different ceiling than the base admin spread fee.)

### Proof of Concept
1. Deploy a pool with `maxAdminSpreadFeeE6 = 10_000` (1%).
2. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Read `_binState(pool, 0)` — `addFeeBuyE6 == 65535`, `addFeeSellE6 == 65535`. No revert.
4. Execute a swap through bin 0. The effective fee is `baseFeeX64 + 65535/1e6` — far above the 1% cap.
5. Assert the trader paid more than `maxAdminSpreadFeeE6` total spread fee, confirming the cap is bypassed. [5](#0-4) [6](#0-5)
