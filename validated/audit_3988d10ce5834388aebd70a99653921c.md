### Title
Broken extension permanently blocks `removeLiquidity`, trapping user LP funds - (File: `metric-core/contracts/MetricOmmPool.sol`)

### Summary

The `removeLiquidity` function unconditionally invokes `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` extension hooks on every call. All extension addresses and call-order bitmasks are stored as `immutable` variables and cannot be changed after pool deployment. There is no emergency exit, no admin override, and no factory mechanism to disable extensions. If any extension in the remove-liquidity hook order becomes permanently broken, every user's `removeLiquidity` call reverts and their deposited funds are permanently locked in the pool.

### Finding Description

`removeLiquidity` in `MetricOmmPool.sol` executes the following sequence:

```
_beforeRemoveLiquidity(...)   // line 207
LiquidityLib.removeLiquidity(...)  // line 208-210
_afterRemoveLiquidity(...)    // line 211
```

Both hook calls route through `_callExtensionsInOrder` in `ExtensionCalling.sol`:

```solidity
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);   // propagates revert
        order >>= 3;
    }
}
```

`CallExtension.callExtension` bubbles up any revert from the extension contract verbatim:

```solidity
(bool success, bytes memory result) = extension.call(data);
if (!success) {
    if (result.length > 0) {
        assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
    }
    revert ExtensionCallFailed();
}
```

All seven extension slots and all six order bitmasks (including `BEFORE_REMOVE_LIQUIDITY_ORDER` and `AFTER_REMOVE_LIQUIDITY_ORDER`) are declared `internal immutable` in `ExtensionCalling.sol` and set once in the constructor. They cannot be updated by any admin, factory owner, or pool admin after deployment. There is no `emergencyRemoveLiquidity`, no factory-level extension-bypass flag, and no separate withdrawal path that skips hooks.

The `BaseMetricExtension` base class ships with `beforeRemoveLiquidity` and `afterRemoveLiquidity` that unconditionally `revert ExtensionNotImplemented()` unless overridden. Any extension that inherits from `BaseMetricExtension` and is wired into the remove-liquidity order without overriding those methods will immediately and permanently block all withdrawals from the pool.

Beyond the base-class default, any extension that later becomes broken — due to a bug in its own logic, a failed external call it depends on, or a self-destructed dependency — will produce the same permanent DoS on `removeLiquidity`.

### Impact Explanation

All LP principal deposited into the pool is permanently inaccessible. Users cannot call `removeLiquidity` because every attempt reverts at the broken extension hook. Since extensions are immutable and there is no alternative withdrawal path, the funds are locked for the lifetime of the pool contract. This is a direct, total loss of user principal for all LPs in the affected pool.

### Likelihood Explanation

The trigger requires a pool to be deployed with an extension wired into `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER`, and for that extension to subsequently break. This is not a privileged or malicious-setup scenario: `createPool` is permissionless, extensions are a documented first-class feature, and the `BaseMetricExtension` default already provides a concrete path to an immediately broken remove-liquidity hook if a developer forgets to override the relevant methods. Any extension that relies on an external oracle, allowlist, or state that can become unavailable also creates this risk. The combination of immutable extension wiring and no emergency exit makes the impact permanent once triggered.

### Recommendation

Add an emergency liquidity-removal path that bypasses extension hooks. Options include:

1. A factory-controlled `emergencyRemoveLiquidity` function on the pool that skips all extension calls and transfers funds directly to the position owner.
2. A factory-level flag (e.g., `extensionsBypassEnabled`) that, when set by the factory owner, causes `_callExtensionsInOrder` to return early, restoring the ability to withdraw.
3. Require that extensions deployed into remove-liquidity hook slots implement a circuit-breaker or fallback that always succeeds, enforced at `ValidateExtensionsConfig` time.

### Proof of Concept

1. Deploy a pool via `createPool` with one extension whose `beforeRemoveLiquidity` is wired into `BEFORE_REMOVE_LIQUIDITY_ORDER`. The extension inherits `BaseMetricExtension` but does not override `beforeRemoveLiquidity` (or its external dependency later fails).
2. User calls `addLiquidity` and deposits funds.
3. User calls `removeLiquidity`. Execution reaches `_beforeRemoveLiquidity` → `_callExtensionsInOrder` → `CallExtension.callExtension(extension, data)`.
4. The extension call reverts (`ExtensionNotImplemented` or any other revert).
5. `CallExtension.callExtension` re-reverts with the extension's revert data.
6. `removeLiquidity` reverts. No state change occurs. User funds remain locked.
7. Because `BEFORE_REMOVE_LIQUIDITY_ORDER` is `immutable`, no admin action can change the wired extension. There is no alternative withdrawal function. Funds are permanently locked.
