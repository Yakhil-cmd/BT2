### Title
Commented-Out `nftSellerNetAmount` Calculation Displays Incorrect Net Proceeds to NFT Sellers Before Offer Acceptance - (File: `packages/gui/src/components/offers/utils.ts`)

---

### Summary

The `calculateNFTRoyalties` function in `utils.ts` contains a commented-out net-proceeds calculation. As a result, `nftSellerNetAmount` is always set equal to the gross `amount` instead of `amount − royaltyAmount`. This incorrect value propagates into the NFT offer viewer's "Net Proceeds" display and into the `overrideNFTSellerAmount` that overrides the offer summary row shown to the NFT seller before they accept a `TokenForNFT` offer.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes the amount an NFT seller will net after royalties:

```typescript
// packages/gui/src/components/offers/utils.ts, lines 314–317
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The correct formula (`amount − royaltyAmount − makerFee`) is commented out. `nftSellerNetAmount` is unconditionally set to the full `amount`. [1](#0-0) 

This value is consumed in two security-relevant display paths in `NFTOfferViewer.tsx`:

**Path 1 — `overrideNFTSellerAmount`** (lines 397–402): For `TokenForNFT` offers, the viewer converts `nftSellerNetAmount` to mojos and passes it as `overrideNFTSellerAmount` to `NFTOfferSummary`. This prop overrides the amount shown in the main offer summary row ("You will receive"). [2](#0-1) 

**Path 2 — "Net Proceeds" label** (lines 580–598): The viewer renders a dedicated "Net Proceeds" section using `nftSaleInfo?.nftSellerNetAmount`, which is the same incorrect value. [3](#0-2) 

The same incorrect value is also shown in `NFTOfferEditor.tsx` under "They will receive" for the `TokenForNFT` tab. [4](#0-3) 

---

### Impact Explanation

When a buyer creates a `TokenForNFT` offer (offering XCH/CAT for an NFT), the NFT seller views the offer and sees:

- **"Net Proceeds"**: displayed as the full offered `amount` (e.g., 10 XCH)
- **Offer summary row ("You will receive")**: overridden with the same inflated value

The NFT seller's actual on-chain receipt is `amount − royaltyAmount` (e.g., 9.5 XCH for a 5% royalty NFT). The GUI shows 10 XCH; the seller receives 9.5 XCH. The seller accepts the offer based on a materially incorrect amount display. This is a concrete accounting spoofing issue in the offer acceptance flow that causes a user to approve a transaction displaying the wrong amount they will receive.

---

### Likelihood Explanation

Any NFT with a non-zero `royaltyPercentage` triggers this bug. The `TokenForNFT` exchange path is a standard user flow (buyer makes offer for seller's NFT). No special attacker capability is required — the bug is triggered by the normal offer-viewing flow for any royalty-bearing NFT.

---

### Recommendation

Restore the commented-out net-proceeds calculation in `calculateNFTRoyalties`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

Additionally, add unit tests for `calculateNFTRoyalties` covering both `NFTForToken` and `TokenForNFT` exchange types with non-zero royalty percentages and maker fees to prevent regression.

---

### Proof of Concept

1. Mint or obtain an NFT with `royaltyPercentage = 500` (5%).
2. As a buyer, create a `TokenForNFT` offer: offer 10 XCH for the NFT.
3. As the NFT seller, open the offer in the GUI via `NFTOfferViewer`.
4. Observe: "Net Proceeds" displays **10 XCH**; "You will receive" in the summary also shows **10 XCH**.
5. Accept the offer. On-chain settlement pays the 5% royalty (0.5 XCH) to the creator, so the seller actually receives **9.5 XCH**.
6. The GUI showed 10 XCH; the seller received 9.5 XCH — a 5% discrepancy driven entirely by the commented-out calculation at `utils.ts:314–317`.
