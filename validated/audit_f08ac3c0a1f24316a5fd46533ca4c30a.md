### Title
Unbounded Loop Over User-Controlled `pool_member_epoch_balance` Trace Causes Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `Pool` iterates over the entire `pool_member_epoch_balance` trace for a pool member without any bound. Because a delegator can grow this trace by making one balance change per epoch (via `add_to_delegation_pool` or `exit_delegation_pool_intent`) across arbitrarily many epochs without claiming rewards, the loop can eventually exceed Starknet's gas limit, permanently freezing the delegator's unclaimed yield.

### Finding Description

`Pool::calculate_rewards` contains an explicit unbounded `while` loop over the `pool_member_epoch_balance` trace:

```rust
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which inserts a new entry keyed at `current_epoch + K` each time a balance change occurs in a new epoch:

```rust
fn set_member_balance(...) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

`set_member_balance` is called by:
- `enter_delegation_pool` (line 201)
- `add_to_delegation_pool` via `increase_member_balance` (line 242)
- `exit_delegation_pool_intent` (line 278) [3](#0-2) [4](#0-3) 

Because the key is `current_epoch + K`, each call in a distinct epoch appends a new checkpoint. The `entry_to_claim_from` cursor stored in `pool_member_info` advances only after a successful `claim_rewards`, so all unclaimed balance-change entries must be iterated on the next claim.

`claim_rewards` calls `calculate_rewards` directly:

```rust
let (mut rewards, updated_entry_to_claim_from) = self
    .calculate_rewards(
        :pool_member,
        from_checkpoint: pool_member_info.reward_checkpoint,
        :until_checkpoint,
        entry_to_claim_from: pool_member_info.entry_to_claim_from,
    );
``` [5](#0-4) 

Inside the loop, `find_sigma` is also called per iteration, adding further gas cost per entry. [6](#0-5) 

### Impact Explanation

A delegator who makes one balance change per epoch (e.g., calling `add_to_delegation_pool` with the minimum non-zero amount of 1 FRI) across a large number of epochs without claiming rewards will accumulate a trace of length proportional to the number of epochs elapsed. When `claim_rewards` is eventually called — either by the delegator or their reward address — the `calculate_rewards` loop must iterate over all accumulated entries. Once the trace is large enough, the transaction will exceed Starknet's gas limit and revert. Because the `entry_to_claim_from` cursor is only updated on a *successful* `claim_rewards`, and there is no partial-claim mechanism, the delegator's unclaimed yield is **permanently frozen**: every future `claim_rewards` call will also revert.

This matches the allowed impact: **Permanent freezing of unclaimed yield** (High severity).

### Likelihood Explanation

The minimum cost per epoch to grow the trace is 1 FRI (the minimum non-zero `add_to_delegation_pool` amount). Starknet epochs are short (on the order of hours to days). A patient attacker can self-inflict this condition over weeks or months, or a griefing attacker can target another delegator's pool member account if they control the reward address (which is allowed to call `add_to_delegation_pool`). The developers themselves acknowledge the risk in the code comment: *"This loop is unbounded but unlikely to exceed gas limits."* [7](#0-6) 

### Recommendation

1. **Checkpoint-based pagination**: Store `entry_to_claim_from` as the cursor (already done) and enforce a maximum number of iterations per `claim_rewards` call, returning partial rewards and updating the cursor so subsequent calls can continue.
2. **Limit trace growth**: Deduplicate trace entries within the same epoch at write time (if `trace.insert` does not already do so for the same key), so that multiple balance changes in the same epoch produce only one entry.
3. **Alternatively**, adopt a cumulative-sigma approach that avoids iterating over individual balance-change entries entirely, computing rewards in O(1) per claim regardless of how many balance changes occurred.

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with a valid amount. Trace length = 1.
2. For each of N epochs, delegator calls `add_to_delegation_pool(amount: 1)` without ever calling `claim_rewards`. Each call in a new epoch appends one entry to `pool_member_epoch_balance`. Trace length grows to N+1.
3. After N epochs, delegator (or reward address) calls `claim_rewards`.
4. `calculate_rewards` enters the `while` loop and iterates N+1 times, calling `find_sigma` on each iteration.
5. For sufficiently large N, the transaction exceeds the Starknet block gas limit and reverts.
6. All subsequent `claim_rewards` calls also revert for the same reason. The delegator's accumulated yield is permanently inaccessible.

            // Asserts.
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let old_delegated_stake = self.get_last_member_balance(:pool_member);
            let total_amount = old_delegated_stake + pool_member_info.unpool_amount;

            // Notify the staking contract of the removal intent.
            let unpool_time = self.undelegate_from_staking_contract_intent(:pool_member, :amount);

            // Edit the pool member to reflect the removal intent, and write to storage.
            if amount.is_zero() {
                pool_member_info.unpool_time = Option::None;
            } else {
                pool_member_info.unpool_time = Option::Some(unpool_time);
            }
            pool_member_info.unpool_amount = amount;
            let new_delegated_stake = total_amount - amount;
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Update the pool member's balance checkpoint.
            self.set_member_balance(:pool_member, amount: new_delegated_stake);

                        pool_member, exit_timestamp: unpool_time, amount,
                    },
                );

                    Events::PoolMemberBalanceChanged {
