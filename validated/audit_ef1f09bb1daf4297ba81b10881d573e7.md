### Title
Unbounded Loop Over Pool Member Balance Trace Causes Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `DelegationPool` iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace without any upper bound. A pool member who makes one balance change per epoch across many epochs without claiming rewards accumulates an arbitrarily large trace. When `claim_rewards` or `pool_member_info_v1` is eventually called, the loop exceeds Starknet's gas limit, permanently freezing the pool member's unclaimed yield.

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:**

`calculate_rewards` iterates from `entry_to_claim_from` to `pool_member_trace_length`, reading and processing every checkpoint in the pool member's balance trace. The developers themselves acknowledge this with an inline comment:

> "This loop is unbounded but unlikely to exceed gas limits." [1](#0-0) 

**How the trace grows — one new entry per epoch per balance change:**

`set_member_balance` inserts a checkpoint keyed by `current_epoch + K` (K = 2). The underlying `insert` in `trace.cairo` only merges into the last entry when the key is identical; otherwise it appends a new entry. [2](#0-1) [3](#0-2) 

Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a distinct epoch therefore appends a new checkpoint. There is no cap on how many checkpoints a single pool member may accumulate, and the trace is never pruned. [4](#0-3) [5](#0-4) 

**`entry_to_claim_from` does not prevent the problem:**

`entry_to_claim_from` is advanced and saved only after a successful `claim_rewards`. If the pool member never claims (or claims infrequently while continuing to make balance changes), the number of unprocessed entries grows without bound. [6](#0-5) 

**Two public entry points trigger the loop:**

1. `claim_rewards` — called by the pool member or their reward address.
2. `pool_member_info_v1` — a read-only view callable by anyone. [7](#0-6) [8](#0-7) 

### Impact Explanation

Once the trace is large enough that iterating it exceeds Starknet's per-transaction gas limit, every call to `claim_rewards` reverts. The pool member's accrued yield is permanently frozen inside the contract with no recovery path, because:
- There is no partial-claim mechanism that would let the member advance `entry_to_claim_from` in smaller batches.
- The trace is append-only and never pruned.

This matches the allowed impact: **Permanent freezing of unclaimed yield** (High).

### Likelihood Explanation

The attack is self-inflicted but realistic:
- A delegator who makes one balance change per epoch (e.g., periodic top-ups or partial exits) and defers claiming rewards for an extended period will naturally accumulate a large trace.
- Starknet epochs are short; hundreds of epochs can pass in weeks.
- No privileged access is required — any pool member can reach this state through normal protocol interactions.
- The code comment explicitly acknowledges the loop is unbounded, confirming the developers are aware of the risk but have not mitigated it.

### Recommendation

1. **Introduce a batch-claim mechanism**: Allow `claim_rewards` to accept a `max_entries` parameter so the loop processes at most N checkpoints per call, advancing `entry_to_claim_from` incrementally.
2. **Prune processed checkpoints**: After `entry_to_claim_from` advances past old entries, compact or discard them so the effective trace length stays bounded.
3. **Enforce a maximum trace length**: Reject balance changes that would push the trace beyond a safe limit (e.g., 500 entries), forcing the member to claim first.

### Proof of Concept

1. Pool member `A` joins a pool with the minimum delegation amount.
2. Each epoch, `A` calls `add_to_delegation_pool` with 1 token — this appends one new checkpoint to `pool_member_epoch_balance` because `current_epoch + K` advances each epoch.
3. `A` never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After `N` epochs (e.g., N = 1 000), the trace has 1 000 entries.
5. `A` (or anyone) calls `claim_rewards(pool_member: A)`. `calculate_rewards` enters the `while` loop and iterates all 1 000 entries, each requiring a storage read (`pool_member_trace.at(entry_to_claim_from)`) and a `find_sigma` call.
6. At a sufficient N the transaction exceeds the Starknet gas limit and reverts.
7. Every subsequent `claim_rewards` call also reverts. `A`'s accumulated yield is permanently frozen. [1](#0-0)

            // Asserts.
            let pool_member = get_caller_address();

                pool_member_info.unpool_time = Option::Some(unpool_time);

            self.write_pool_member_info(:pool_member, :pool_member_info);

            self.set_member_balance(:pool_member, amount: new_delegated_stake);

            self
                .emit(
                    Events::PoolMemberExitIntent {
                        pool_member, exit_timestamp: unpool_time, amount,
                    },
                );
            self
                .emit(
                    Events::PoolMemberBalanceChanged {
                        pool_member, old_delegated_stake, new_delegated_stake,

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
