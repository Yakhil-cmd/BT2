### Title
Pool admin can silently overwrite a pending price provider proposal to reset the timelock — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`proposePoolPriceProvider` contains no guard against an already-pending proposal. A pool admin can call it a second time at any point during the timelock window, atomically replacing both `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]` with new values, resetting the countdown to `block.timestamp + timelock`. The invariant that a pending oracle rotation cannot be silently replaced mid-timelock is violated.

### Finding Description
`proposePoolPriceProvider` in `MetricOmmPoolFactory.sol` unconditionally writes:

```solidity
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;   // block.timestamp + timelock
``` [1](#0-0) 

There is no `if (pendingPriceProvider[pool] != address(0)) revert ...` guard before these writes. The function only checks that the pool is in mutable-oracle mode and that the new provider passes token-pair validation. [2](#0-1) 

`executePoolPriceProviderUpdate` enforces the timelock only against the stored `pendingPriceProviderExecuteAfter[pool]`, so resetting that value via a second `proposePoolPriceProvider` call directly resets the enforcement point. [3](#0-2) 

There is no `cancelPoolPriceProvider` function in `IMetricOmmPoolFactoryPoolAdmin`; the only way to clear a pending proposal is to execute it after the timelock elapses. [4](#0-3) 

### Impact Explanation
The timelock exists specifically to constrain the pool admin: users who observe a pending oracle rotation can evaluate the incoming provider and exit the pool before the change takes effect. Allowing the pool admin to overwrite the pending proposal mid-timelock breaks this protection in two ways:

1. **Timelock reset / indefinite deferral.** The admin can keep re-proposing (even the same address) to prevent `executePoolPriceProviderUpdate` from ever succeeding, permanently blocking an oracle rotation.
2. **Silent provider substitution.** The admin proposes a benign provider A (users observe it and decide to stay), then — one block before execution — proposes malicious provider B, resetting `executeAfter` to `now + timelock`. After the new timelock elapses the admin executes with provider B. Users who made their exit decision based on provider A are now exposed to provider B's prices. If provider B returns manipulated bid/ask quotes, every subsequent swap executes at bad prices, causing direct loss of user principal or protocol fees.

This is an admin-boundary break: the pool admin bypasses the timelock that is the sole on-chain constraint on their oracle-rotation power.

### Likelihood Explanation
The pool admin is a semi-trusted role. The attack requires no external conditions beyond the admin calling `proposePoolPriceProvider` twice. The second call is indistinguishable from a legitimate re-proposal (e.g., "correcting a mistake"), so off-chain monitoring may not flag it in time. The `PoolPriceProviderChangeProposed` event is emitted on both calls, but the first proposal's `executeAfter` is silently discarded with no on-chain record.

### Recommendation
Add a guard at the top of `proposePoolPriceProvider` that reverts if a pending proposal already exists:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();
```

Alternatively, add a `cancelPoolPriceProvider` function (analogous to `cancelPoolAdminTransfer`) that explicitly clears the pending state, making the replacement visible and auditable as a two-step action.

### Proof of Concept
```
t=0:   admin calls proposePoolPriceProvider(pool, providerA)
       → pendingPriceProvider[pool] = providerA
       → pendingPriceProviderExecuteAfter[pool] = t0 + 7 days

t=6d:  admin calls proposePoolPriceProvider(pool, providerB)   // 1 day before execution
       → pendingPriceProvider[pool] = providerB                // providerA silently dropped
       → pendingPriceProviderExecuteAfter[pool] = t6d + 7 days // timelock reset

t=13d: admin calls executePoolPriceProviderUpdate(pool)
       → pool's active oracle is now providerB
       → users who stayed based on providerA are now priced by providerB
```

Assert: after the second `proposePoolPriceProvider`, `pendingPriceProvider[pool] == providerB` and `pendingPriceProviderExecuteAfter[pool] == block.timestamp + 7 days` (not the original `t0 + 7 days`). The original proposal for `providerA` is irrecoverably lost with no on-chain cancellation record.
