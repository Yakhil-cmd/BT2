### Title
Gateway Admits Invoke Transactions with Unverified Signatures via `skip_stateful_validations` Deploy-Account UX Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point (signature verification) for any invoke transaction with `nonce == 1` when the sender's on-chain nonce is `0` and the sender address appears in the mempool or a recent block. Because the mempool check is a simple address-presence test, an unprivileged attacker can submit an invoke with an arbitrary (invalid) signature from any victim address that currently has a `deploy_account` transaction in the mempool, and the gateway will admit it without ever verifying the signature.

### Finding Description

`skip_stateful_validations` returns `true` (skip `__validate__`) when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` **and** `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

`account_tx_in_pool_or_recent_block` is a pure address-presence check — it returns `true` if the address has **any** transaction in the pool or was seen in a recent committed block, regardless of transaction type or whether the caller controls the account. [2](#0-1) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, so `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` — the `__validate__` entry point is never called. [3](#0-2) 

`perform_pre_validation_stage` still runs `handle_nonce`, `check_fee_bounds`, and `verify_can_pay_committed_bounds`, but these checks use the **victim's** on-chain state (nonce and balance), not the attacker's. [4](#0-3) 

**Attack path:**

1. Alice broadcasts a `deploy_account` (nonce 0) for her new account address `A`. The transaction enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker observes Alice's pending `deploy_account` (mempool is public).
3. Attacker crafts an `Invoke` with `sender_address = A`, `nonce = 1`, and an **arbitrary/invalid signature**, with resource bounds set above the minimum and Alice's balance sufficient to pass `verify_can_pay_committed_bounds`.
4. Gateway stateless validation passes (resource bounds are syntactically valid).
5. Gateway stateful validation: nonce check passes (`0 ≤ 1 ≤ max_allowed_nonce_gap`), resource-bounds check passes, balance check passes against Alice's balance, `skip_stateful_validations` returns `true`, `__validate__` is **skipped**.
6. The attacker's transaction is admitted to the mempool occupying Alice's nonce-1 slot. [5](#0-4) 

### Impact Explanation

The broken invariant is: *every transaction admitted to the mempool must have passed signature verification*. The gateway violates this invariant for invoke transactions with `nonce == 1` from any address visible in the mempool.

Concrete consequences:

- **Nonce-slot griefing**: The attacker's invalid transaction occupies Alice's nonce-1 slot. Alice's legitimate invoke must either wait for the batcher to execute and reject the fake transaction, or pay a fee-escalation premium to replace it.
- **Mempool/batcher resource waste**: The batcher fetches and executes the invalid transaction; `__validate__` fails, the transaction is rejected without fee collection, and the batcher's execution budget is consumed.
- **Scalable attack**: Any address with a pending `deploy_account` is a target. An attacker can simultaneously grief every new account deployment visible in the mempool.

This matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

### Likelihood Explanation

- `deploy_account` transactions are publicly visible in the mempool.
- The attacker needs no funds, no private key, and no privileged access — only the victim's address and the ability to submit an RPC transaction.
- The condition (`nonce == 1`, `account_nonce == 0`, address in mempool) is reliably satisfied during the standard deploy-account + invoke UX flow that the feature is designed to support.
- The attack window is the time between Alice's `deploy_account` entering the mempool and the batcher committing it.

### Recommendation

Replace the coarse address-presence check with a check that verifies a **`deploy_account` transaction specifically** is pending for the sender address, rather than any transaction. Alternatively, record the `tx_hash` of the pending `deploy_account` and require the incoming invoke to reference it, or restrict the skip to cases where the mempool can confirm the pending transaction is a `deploy_account` type. [6](#0-5)

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    // Performs static checks before executing validation entry point.
    // Note that nonce is incremented during these checks.
    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
