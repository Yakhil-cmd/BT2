### Title
`MsgTransferTokens` Permanently Broken for Source (CRC21) Tokens Due to Missing `channel_id` Field - (File: `x/cronos/keeper/msg_server.go`, `x/cronos/keeper/ibc.go`)

---

### Summary

`MsgTransferTokens` is a proto-defined, unprivileged transaction surface for IBC-transferring Cronos tokens. It always passes an empty string `""` as `channelId` to `IbcTransferCoins`. For source tokens (denom prefix `cronos0x…`), `ibcSendTransfer` unconditionally requires a valid channel ID and returns an error when it is absent. As a result, any user attempting to IBC-transfer a source (CRC21-native) token via `MsgTransferTokens` is permanently blocked, with no workaround through this message path.

---

### Finding Description

`MsgTransferTokens` in `msg_server.go` hardcodes `channelId = ""`:

```go
// x/cronos/keeper/msg_server.go:47-51
func (k msgServer) TransferTokens(...) {
    // TODO change the msg to be able to specify the channel id
    // Only sending non source token is supported at the moment
    err := k.IbcTransferCoins(ctx, msg.From, msg.To, msg.Coins, "")
``` [1](#0-0) 

Inside `IbcTransferCoins`, source-coin denoms (`cronos0x…`) pass the denom-validity check and reach `ibcSendTransfer` with the empty channel ID:

```go
// x/cronos/keeper/ibc.go:133-141
if !types.IsValidIBCDenom(c.Denom) && !types.IsValidCronosDenom(c.Denom) {
    return fmt.Errorf(...)
}
_, found := k.GetContractByDenom(ctx, c.Denom)
if !found { ... }
err = k.ibcSendTransfer(ctx, acc, destination, c, channelId)
``` [2](#0-1) 

`ibcSendTransfer` then unconditionally rejects an empty channel ID for source coins:

```go
// x/cronos/keeper/ibc.go:161-165
func (k Keeper) ibcSendTransfer(...) error {
    if types.IsSourceCoin(coin.Denom) {
        if !channeltypes.IsValidChannelID(channelId) {
            return errors.New("invalid channel id for ibc transfer of source token")
        }
``` [3](#0-2) 

The proto definition of `MsgTransferTokens` has no `channel_id` field, so there is no way for a caller to supply one:

```proto
// proto/cronos/tx.proto:53-59
message MsgTransferTokens {
  string from  = 1;
  string to    = 2;
  repeated cosmos.base.v1beta1.Coin coins = 3;
}
``` [4](#0-3) 

The `ModuleCRC21` contract's `send_to_ibc` function (the EVM-hook path) does carry a `channel_id` and works correctly via `SendToIbcV2Handler`. The Cosmos-message path (`MsgTransferTokens`) has no equivalent parameter and is therefore permanently broken for source tokens. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Any unprivileged user who holds a source token (`cronos0x…` denom) and submits `MsgTransferTokens` to IBC-transfer it will always receive `"invalid channel id for ibc transfer of source token"`. The message is accepted by the mempool, consumes gas, and deterministically fails. There is no way to supply a channel ID through this message path. This is a **permanent inability for honest users to use the `MsgTransferTokens` bridge/conversion flow for source tokens**, matching the allowed High impact category.

---

### Likelihood Explanation

Any user who has received or minted a CRC21-source token and attempts to use the documented `MsgTransferTokens` Cosmos message to send it back via IBC will trigger this failure. The path is reachable by any unprivileged account with no special preconditions beyond holding a source token.

---

### Recommendation

Add a `channel_id` field to `MsgTransferTokens` in `proto/cronos/tx.proto` and thread it through `TransferTokens` → `IbcTransferCoins` → `ibcSendTransfer`. Validate that `channel_id` is non-empty when the submitted coins include any source denom, mirroring the validation already present in `ibcSendTransfer`. This mirrors the fix applied in the referenced Livepeer report: expose the missing parameter at the entry-point interface so the full call chain can succeed.

---

### Proof of Concept

1. Deploy or obtain a CRC21 source token (denom `cronos0xABCD…`).
2. Submit `MsgTransferTokens` with `coins = [{"denom":"cronos0xABCD…","amount":"1"}]`, any valid `to` bech32 address.
3. Observe the transaction is accepted into the mempool, executes, and returns `"invalid channel id for ibc transfer of source token"` — deterministically, every time, with no way to avoid it through this message path.
4. Contrast with the EVM-hook path: calling `send_to_ibc(recipient, amount, channel_id, extraData)` on the CRC21 contract succeeds because `channel_id` is carried in the event and processed by `SendToIbcV2Handler`.
