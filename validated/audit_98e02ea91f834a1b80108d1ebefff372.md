### Title
Missing EVM Contract Existence Check on `wnear_address` Allows Fee-Free XCC Calls, Draining Engine NEAR Balance - (File: `engine-precompiles/src/xcc.rs`)

---

### Summary

The `CrossContractCall` precompile makes a low-level EVM sub-call to the stored `wnear_address` to collect the wNEAR fee required for cross-contract calls. It does not verify that a contract actually exists at `wnear_address` before calling it. Per EVM semantics, a call to an address with no deployed code returns `ExitReason::Succeed` with empty output. The precompile treats this as a successful `transferFrom`, proceeds to emit the XCC promise log, and the engine creates real NEAR promises — consuming the engine's NEAR balance — without the caller having paid any wNEAR.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, `CrossContractCall::run_with_handle` computes `required_near` (line 177–182) and, when nonzero, attempts to collect it by calling `transferFrom` on the wNEAR ERC-20:

```rust
let wnear_address = state::get_wnear_address(&self.io);   // line 193
// ...
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context); // line 200
match exit_reason {
    aurora_evm::ExitReason::Succeed(_) => (),   // treated as successful transfer
    // ...
}
``` [1](#0-0) 

There is no `extcodesize` check or equivalent existence guard before this call. The EVM specification guarantees that a `CALL` to an address with no deployed bytecode returns success with empty return data. If `wnear_address` has no code, `exit_reason` is `ExitReason::Succeed`, the match arm at line 203 silently passes, and execution continues to emit the XCC promise log:

```rust
let promise_log = Log {
    address: cross_contract_call::ADDRESS.raw(),
    topics,
    data: borsh::to_vec(&promise)...?,
};
Ok(PrecompileOutput { logs: vec![promise_log], ... })
``` [2](#0-1) 

This log is consumed by `filter_promises_from_logs` in `engine/src/engine.rs`, which calls `xcc::handle_precompile_promise` to create real NEAR protocol promises: [3](#0-2) 

`handle_precompile_promise` then spends the engine's own NEAR balance (e.g., `STORAGE_AMOUNT = 2 NEAR` for new router accounts) to fund the XCC sub-account and execute the cross-contract call: [4](#0-3) 

The `wnear_address` is stored in engine state and retrieved via `get_wnear_address`, which panics only if the key is entirely absent — it does not validate that a contract exists at the stored address: [5](#0-4) 

---

### Impact Explanation

**High — Temporary Freezing of Funds / Theft of Engine NEAR Balance.**

If `wnear_address` resolves to an address with no EVM bytecode, any EVM user can invoke the XCC precompile and make cross-contract calls without paying the wNEAR fee. Each such call causes the engine to spend its own NEAR balance (up to `STORAGE_AMOUNT` = 2 NEAR per new router account, plus gas costs for each call). Repeated exploitation drains the engine's NEAR balance, preventing all future XCC operations that require NEAR funding — a temporary freeze of the XCC subsystem — and constitutes direct theft of the engine's NEAR reserves.

---

### Likelihood Explanation

**Medium.** The `wnear_address` is set by the owner via `factory_set_wnear_address` (owner-only): [6](#0-5) 

The condition is triggered if:
1. The address is misconfigured to a non-contract address (operator error).
2. The wNEAR ERC-20 contract is self-destructed. Aurora Engine uses the Berlin EVM config (which retains `SELFDESTRUCT`), so if the wNEAR ERC-20 exposes any self-destruct path, an attacker could destroy it and then exploit this gap.

Once the condition holds, exploitation requires no special privilege — any EVM user or contract can call the XCC precompile address `0x516cded1d16af10cad47d6d49128e2eb7d27b372` directly.

---

### Recommendation

Before executing `handle.call(wnear_address.raw(), ...)`, verify that the target address has deployed bytecode. In the EVM backend this is equivalent to checking `extcodesize(wnear_address) > 0`. In Aurora's Rust layer, this can be done via the `Backend::code` method already implemented on `Engine`: [7](#0-6) 

Add a guard such as:

```rust
// Reject if wNEAR contract does not exist
if handle.code(wnear_address.raw()).is_empty() {
    return Err(PrecompileFailure::Error {
        exit_status: ExitError::Other(Cow::from("ERR_WNEAR_CONTRACT_NOT_FOUND")),
    });
}
```

Additionally, validate that the `transferFrom` return value is a valid ABI-encoded `true` (ERC-20 standard), rather than accepting any `Succeed` exit — an empty return from a nonexistent contract would be rejected by this check as well.

---

### Proof of Concept

1. Deploy Aurora Engine with XCC enabled and `factory_set_wnear_address` called with an address that has no deployed ERC-20 bytecode (e.g., a fresh EOA address, or after self-destructing the wNEAR ERC-20).
2. From any EVM account, construct a `CrossContractCallArgs::Eager` payload targeting any NEAR account method and submit it as a transaction to `cross_contract_call::ADDRESS` (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`).
3. Observe: `handle.call(wnear_address, ...)` returns `ExitReason::Succeed` with empty data (no contract at target). The precompile does not revert. The XCC promise log is emitted.
4. `filter_promises_from_logs` picks up the log and calls `handle_precompile_promise`, which creates a NEAR batch promise spending `STORAGE_AMOUNT` (2 NEAR) from the engine's balance.
5. Repeat to drain the engine's NEAR balance without paying any wNEAR. [8](#0-7) [9](#0-8)
