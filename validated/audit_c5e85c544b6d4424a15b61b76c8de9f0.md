### Title
Unprivileged Caller Can Permanently Suppress Block Rewards via `update_rewards` with `disable_rewards=true` ‚Äî (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` is documented as "Only starkware sequencer" but the implementation enforces no caller restriction. Any unprivileged address can call it with `disable_rewards=true`, which advances the global `last_reward_block` gate without distributing rewards, permanently blocking the legitimate sequencer call for that block.

---

### Finding Description

The `update_rewards` entry point is implemented in `StakingRewardsManagerImpl`: [1](#0-0) 

The only guard at the top is `self.general_prerequisites()`. Because `general_prerequisites` is called by ~40 functions including fully public ones like `stake`, `claim_rewards`, and `increase_stake`, it cannot enforce sequencer-only access ‚Äî it only checks the contract's paused state. There is no `sequencer_address` variable, no `assert_caller_is_sequencer`, and no sequencer-related keyword anywhere in the production code.

After passing the `REWARDS_ALREADY_UPDATED` check, the function unconditionally writes the current block number to the global `last_reward_block` **before** checking `disable_rewards`: [2](#0-1) 

When `disable_rewards=true`, the function returns immediately after updating `last_reward_block`, distributing nothing. Any subsequent call in the same block ‚Äî including the legitimate sequencer call ‚Äî hits: [3](#0-2) 

and reverts with `REWARDS_ALREADY_UPDATED`. The block's reward opportunity is permanently lost.

The spec acknowledges the intended access control: [4](#0-3) 

but this is not enforced in the implementation.

---

### Impact Explanation

An attacker who front-runs the sequencer every block with `update_rewards(any_valid_staker, disable_rewards=true)` causes:

- `last_reward_block` advances to the current block with zero rewards distributed.
- The sequencer's legitimate call reverts.
- All stakers and pool delegators receive zero yield for every attacked block.
- Because `last_reward_block` is a **global** gate (not per-staker), a single attacker call suppresses rewards for **all** stakers simultaneously.
- Repeated indefinitely ‚Üí **permanent freezing of unclaimed yield** for the entire protocol.

Impact: **High ‚Äî Permanent freezing of unclaimed yield.**

---

### Likelihood Explanation

- No privileged role, leaked key, or external dependency is required.
- The attacker only needs to submit a transaction before the sequencer's `update_rewards` call each block.
- On Starknet, transaction ordering within a block is controlled by the sequencer, which limits the attacker's ability to reliably front-run. However, the sequencer itself is not the only entity that can submit transactions, and the window exists at every block boundary.
- The attack is cheap (one contract call per block) and requires no capital.

---

### Recommendation

Add an explicit caller check inside `update_rewards` (or in a dedicated modifier) that asserts `get_caller_address() == sequencer_address` (or an equivalent role stored in contract state). Do not rely on `general_prerequisites` for this, as it is shared with public user-facing functions.

---

### Proof of Concept

1. Deploy the staking system with two active stakers past the consensus rewards epoch.
2. At the start of block N, submit `update_rewards(staker_A, disable_rewards=true)` from an arbitrary EOA.
3. Observe `last_reward_block` is now N; staker_A's `unclaimed_rewards_own` is unchanged.
4. The sequencer's `update_rewards(staker_A, disable_rewards=false)` reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat for every block. After M blocks, both stakers have accumulated zero rewards despite being fully active and eligible.

The existing test suite confirms the `REWARDS_ALREADY_UPDATED` revert behavior: [5](#0-4) 

and confirms that `disable_rewards=true` skips distribution while still consuming the block slot: [6](#0-5)
