Audit Report

## Title
Asymmetric API Boundary Node Coverage Between Pre-Guard and Post-Invariant Permanently Blocks Version Unelection â€” (`rs/registry/canister/src/mutations/do_revise_elected_replica_versions.rs`)

## Summary
`remove_blessed_versions_or_panic` checks only subnets and unassigned nodes before preparing unelect mutations, but `check_replica_version_invariants` (invoked post-mutation via `maybe_apply_mutation_internal`) also checks API boundary nodes. When any API boundary node uses a version being unelected, the pre-guard passes, the `ReplicaVersionRecord` delete mutation is prepared, and the post-invariant panics â€” rolling back all state. The governance proposal is consumed with no registry effect, and the version cannot be unelected while any API BN holds it.

## Finding Description
In `do_revise_elected_guestos_versions`, the mutations batch includes a `Type::Delete` for each `ReplicaVersionRecord` of versions to unelect (lines 46â€“53). `remove_blessed_versions_or_panic` (line 55) then iterates subnets (lines 121â€“150) and unassigned nodes (lines 152â€“167) but contains no iteration over API boundary node records. After the guard returns, `maybe_apply_mutation_internal` (line 98) calls `check_global_state_invariants`, which calls `check_replica_version_invariants`. That function builds `versions_in_use` from subnets, unassigned nodes, **and** API boundary nodes via `get_all_api_boundary_node_versions` (line 43 of `replica_version.rs`). The snapshot passed to the invariant already reflects the pending `Delete` mutations, so the `ReplicaVersionRecord` for the unelected version is absent. The superset assertion at lines 55â€“58 of `replica_version.rs` then fires: the version is in `versions_in_use` (from the API BN record) but absent from `elected_set` (record deleted). The canister traps, all mutations are rolled back, and the registry is unchanged.

## Impact Explanation
Any version deployed to API boundary nodes via `do_deploy_guestos_to_some_api_boundary_nodes` becomes permanently unelectable through the normal `ReviseElectedGuestosVersions` NNS governance flow until a separate proposal first migrates all API BNs away from that version. If the version carries a security vulnerability, the community cannot retire it through the standard lifecycle path. NNS governance proposals are silently consumed with no registry effect, breaking the version lifecycle management for API boundary nodes. This matches **High ($2,000â€“$10,000): Significant boundary/API node and NNS infrastructure security impact with concrete user or protocol harm**.

## Likelihood Explanation
No adversarial intent is required. The sequence is: (1) elect version V, (2) deploy V to one or more API boundary nodes via governance, (3) attempt to unelect V via `ReviseElectedGuestosVersions`. Step 3 silently fails. Any NNS neuron holder can submit both proposals. The bug manifests in routine version lifecycle operations and is repeatable on every unelect attempt while any API BN holds the version.

## Recommendation
Add API boundary node version checking to `remove_blessed_versions_or_panic`, mirroring the logic already present in `check_replica_version_invariants`. Iterate over all `ApiBoundaryNodeRecord` entries using `get_api_boundary_node_records_from_snapshot` (or the equivalent registry lookup), check whether any use a version in `versions_to_remove`, and panic with an appropriate message (e.g., `"Cannot retire version {v}, because it is currently deployed to an API boundary node!"`) if so. This makes the pre-guard and post-invariant consistent in their coverage.

## Proof of Concept
State-machine test:
1. Initialize registry with version V elected and one `ApiBoundaryNodeRecord` with `version = V`.
2. Call `do_revise_elected_guestos_versions` with `replica_versions_to_unelect = [V]`.
3. Observe `remove_blessed_versions_or_panic` does **not** panic (API BNs not checked).
4. Observe `maybe_apply_mutation_internal` panics with `"Using a version that isn't elected. Elected versions: {}, in use: {"V"}"`.
5. Confirm registry state is unchanged (V still elected).

The existing test `panic_when_retiring_a_version_in_use` at line 203 of `replica_version.rs` demonstrates the post-invariant fires for subnet versions; an analogous test inserting an `ApiBoundaryNodeRecord` with version V before calling `do_revise_elected_guestos_versions` would reproduce this bug directly.
