### Title
Censorable Pool Token (USDC/USDT) Blacklisting `adminFeeDestination` or `FACTORY` Permanently Bricks `collectFees`, Locking All Accrued Protocol and Admin Fees — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.collectFees` performs unconditional `safeTransfer` calls to both `adminFeeDestination_` and the immutable `FACTORY` address. If either recipient is blacklisted by a censorable pool token (USDC or USDT), every call to `collectFees` reverts. Because `setPoolAdminFees` and `setPoolProtocolFee` in the factory both call `collectFees` as a mandatory first step before updating fee configuration, those admin paths are also permanently blocked. All accrued spread and notional fees remain locked inside the pool with no alternative withdrawal route.

---

### Finding Description

`MetricOmmPool.collectFees` (called exclusively by the factory) computes the admin and protocol shares of accrued spread and notional fees, then transfers them out:

```solidity
// MetricOmmPool.sol lines 416-427
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // safeTransfer
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);   // safeTransfer
}
if (totalFee0ToProtocol > 0) {
    transferToken0(FACTORY, totalFee0ToProtocol);             // safeTransfer
}
if (totalFee1ToProtocol > 0) {
    transferToken1(FACTORY, totalFee1ToProtocol);             // safeTransfer
}
notionalFeeToken0Scaled = 0;   // only reached if all transfers succeed
notionalFeeToken1Scaled = 0;
``` [1](#0-0) 

`FACTORY` is an immutable set once at construction and can never be changed: [2](#0-1) [3](#0-2) 

Three factory entry points all call `collectFees` as a mandatory prerequisite before updating state:

**`collectPoolFees`** (permissionless): [4](#0-3) 

**`setPoolAdminFees`** (pool admin): [5](#0-4) 

**`setPoolProtocolFee`** (factory owner): [6](#0-5) 

If any single `safeTransfer` inside `collectFees` reverts, the entire call reverts, `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` are never zeroed, and all three factory entry points above become permanently unusable for that pool.

**Scenario A — `adminFeeDestination` blacklisted**: USDC/USDT blacklists the admin fee destination address. `collectFees` reverts on the admin leg transfer. The pool admin can call `setPoolAdminFeeDestination` (which does not call `collectFees`) to rotate to a clean address and then retry. This provides a recovery path, but all fees accrued during the blacklist window are locked until recovery, and `setPoolProtocolFee` is also blocked for the protocol owner during that window. [7](#0-6) 

**Scenario B — `FACTORY` blacklisted**: USDC/USDT blacklists the factory contract address. `collectFees` reverts on the protocol leg transfer. Because `FACTORY` is an immutable in the pool bytecode, there is no recovery path. `collectPoolFees`, `setPoolAdminFees`, and `setPoolProtocolFee` are all permanently broken for that pool. All accrued protocol and admin fees are permanently locked inside the pool with no extraction mechanism.

---

### Impact Explanation

- **Direct loss of protocol fees**: `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` are only cleared after successful transfers. Fees continue to accrue but can never be extracted (Scenario B: permanent; Scenario A: until destination is rotated).
- **Broken admin functionality**: `setPoolAdminFees` and `setPoolProtocolFee` are both gated behind `collectFees`. A blacklisted recipient makes it impossible to adjust fee rates for the affected pool, removing the protocol's ability to respond to market conditions.
- **`collectPoolFees` is permissionless** but equally blocked — any keeper or bot attempting to sweep fees will revert.

---

### Likelihood Explanation

USDC and USDT both implement address blacklisting and are the most common stablecoin pool tokens. The `adminFeeDestination` is a user-controlled address (e.g., a multisig or EOA) that can be blacklisted by Circle or Tether for regulatory reasons. The factory contract itself can also be blacklisted. Neither action requires any malicious intent from the pool admin or factory owner — it is an external, third-party action by the token issuer. Pools with USDC or USDT as either token are directly exposed.

---

### Recommendation

Decouple fee collection from fee configuration updates. Two options:

1. **Separate the transfer from the accounting**: Record fees owed to each recipient in storage and let recipients pull their own fees independently. This way a blacklisted recipient only blocks their own withdrawal, not the entire fee system or admin configuration.

2. **Wrap transfers in try/catch**: Catch transfer failures and leave the uncollected amount in a per-recipient claimable balance rather than reverting the entire `collectFees` call. Ensure `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` are still cleared so the accounting does not double-count.

3. **Remove the mandatory `collectFees` call from `setPoolAdminFees` and `setPoolProtocolFee`**: Allow fee rate changes to proceed independently of fee collection. Collect fees separately before or after the rate change.

---

### Proof of Concept

1. Deploy a pool with USDC as `token0` and any token as `token1`, with non-zero `adminSpreadFeeE6` and `protocolSpreadFeeE6`.
2. Execute swaps to accrue spread fees (surplus above `binTotals.scaledToken0`).
3. USDC issuer blacklists `adminFeeDestination` (or `FACTORY`).
4. Call `collectPoolFees(pool)` → reverts because `IERC20(TOKEN0).safeTransfer(adminFeeDestination_, amount)` reverts.
5. Call `setPoolAdminFees(pool, newFee, newFee)` → reverts at the same internal `collectFees` call.
6. Call `setPoolProtocolFee(pool, newFee, newFee)` → reverts at the same internal `collectFees` call.
7. All accrued fees remain locked; fee rate changes are impossible.
