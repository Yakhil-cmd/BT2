### Title
Unbounded Loop in `calculate_rewards` Over Uncapped `pool_member_epoch_balance` Trace Causes Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` internal function in `Pool` iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. The trace grows by one entry per epoch in which the member changes their balance. Because there is no cap on trace length and no partial-claim mechanism, a pool member who accumulates enough balance-change epochs without claiming will eventually be unable to call `claim_rewards` at all — permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
``` [1](#0-0) 

The loop iterates from `pool_member_info.entry_to_claim_from` to the current length of `pool_member_epoch_balance` for that member. Each iteration reads a storage slot and calls `find_sigma`, which itself reads from `cumulative_rewards_trace`.

The trace grows by one entry each time `set_member_balance` is called with a new epoch key:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

`set_member_balance` is invoked by:
- `enter_delegation_pool` (via `set_member_balance` directly) [3](#0-2) 
- `add_to_delegation_pool` (via `increase_member_balance`) [4](#0-3) 
- `exit_delegation_pool_intent` (via `set_member_balance` directly) [5](#0-4) 

Because the insert key is `current_epoch + K`, multiple calls within the same epoch overwrite the same slot (no new entry). But one call per epoch in a distinct epoch appends a new entry. A pool member who makes one balance change per epoch across N epochs without claiming accumulates N trace entries. When `claim_rewards` is finally called, the loop must process all N entries in a single transaction.

`entry_to_claim_from` is only advanced inside `claim_rewards`:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [6](#0-5) 

There is no partial-claim path and no cap on trace length.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.**

Once the trace is long enough that a single `claim_rewards` call exceeds the Starknet transaction gas limit, the pool member's accumulated rewards become permanently unclaimable. The funds remain locked in the pool contract with no recovery path, because:

1. `claim_rewards` is the only function that transfers rewards to the reward address.
2. `exit_delegation_pool_action` does not claim rewards — it only returns the principal.
3. There is no administrative sweep or rescue function.

---

### Likelihood Explanation

Starknet epochs are short (roughly one week based on the `exit_wait_window` default). A pool member who calls `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch and defers claiming for a long period (e.g., a year ≈ 52 epochs, or longer) will accumulate a proportionally large trace. The Starknet block gas limit is finite; at sufficient trace depth the transaction will revert. The code itself acknowledges this risk with the comment "This loop is unbounded but unlikely to exceed gas limits," confirming the developers are aware but have not mitigated it.

---

### Recommendation

1. **Cap the loop**: Introduce a `max_entries_per_claim` parameter and allow partial reward claims, advancing `entry_to_claim_from` by at most that many entries per call.
2. **Lazy compaction**: After each `claim_rewards`, truncate already-processed entries from the trace (or maintain a separate start pointer).
3. **Encourage frequent claiming**: Document clearly that users who make frequent balance changes must also claim frequently, or enforce a maximum unclaimed-epoch window.

---

### Proof of Concept

1. Pool member calls `enter_delegation_pool` at epoch E₀.
2. For each subsequent epoch E₁, E₂, …, Eₙ, the pool member calls `add_to_delegation_pool` with a minimal amount (1 token). Each call in a new epoch appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`.
3. The member never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs, `pool_member_trace_length == N`.
5. The member calls `claim_rewards`. `calculate_rewards` enters the while-loop and iterates N times, each iteration reading two storage slots and calling `find_sigma` (which reads from `cumulative_rewards_trace`).
6. At sufficiently large N, the transaction exceeds the Starknet gas limit and reverts.
7. All subsequent `claim_rewards` calls also revert. The member's accumulated yield is permanently frozen. [7](#0-6) [8](#0-7)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
