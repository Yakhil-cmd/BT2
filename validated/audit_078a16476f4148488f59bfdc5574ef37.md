Audit Report

## Title
Unbounded Per-Round Loop in SNS Governance `distribute_rewards` Can Exhaust Instruction Limit - (File: rs/sns/governance/src/governance.rs)

## Summary
The `distribute_rewards` function in SNS governance iterates once per missed reward round with no upper bound on `new_rounds_count`. When `round_duration_seconds` is set to its minimum allowed value of 1 second and the canister experiences any downtime (e.g., a routine subnet upgrade), the loop can grow to millions of iterations of `rust_decimal` 128-bit arithmetic, exhausting the IC's per-message instruction limit. Because `latest_reward_event` is only updated on successful completion, every subsequent timer invocation recomputes the same unbounded `new_rounds_count` and traps again, permanently disabling SNS reward distribution.

## Finding Description
In `rs/sns/governance/src/governance.rs`, `distribute_rewards` computes:

```rust
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
``` [1](#0-0) 

It then iterates over every missed round with no guard:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = ...;
    let current_reward_rate = voting_rewards_parameters.reward_rate_at(...);
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
``` [2](#0-1) 

The `round_duration_seconds` field is validated to be in the range `[1, MAX_REWARD_ROUND_DURATION_SECONDS]` (i.e., 1 second to 365.25 days): [3](#0-2) 

`MAX_REWARD_ROUND_DURATION_SECONDS` is computed as exactly 31,557,600 seconds (one nominal year): [4](#0-3) 

With `round_duration_seconds = 1`, one day of downtime yields `new_rounds_count = 86,400`; one year yields `new_rounds_count = 31,536,000`. Each iteration calls `reward_rate_at` (which performs multiple `rust_decimal` 128-bit operations) plus two Decimal multiplications and an addition. There is no `MAX_ROUNDS_PER_DISTRIBUTION` cap or any other guard anywhere in the function. The `latest_reward_event` is only written at the end of a successful execution, so a trap leaves it unchanged and the next timer invocation recomputes the same (or larger) `new_rounds_count`, creating a permanent livelock.

## Impact Explanation
This is a concrete application/platform-level DoS against the SNS governance canister's reward subsystem. Once triggered, neuron maturity stops accruing for all participants, and the SNS reward mechanism is permanently disabled with no on-chain recovery path (since the governance canister itself is the one that is stuck). This matches the allowed High impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact"* and *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation
Two conditions must coincide: (1) an SNS initialized with a small `round_duration_seconds` (the minimum of 1 second is explicitly permitted by the schema and passes validation), and (2) any period of canister downtime. Subnet upgrades are routine IC operations that cause brief canister unavailability. An SNS deployer who sets `round_duration_seconds` to even a few seconds (not just the minimum) combined with a subnet upgrade lasting minutes could accumulate enough missed rounds to exhaust the instruction budget. The SNS framework itself permits this configuration without warning, making accidental misconfiguration realistic.

## Recommendation
Add an upper bound on `new_rounds_count` before entering the loop:

```rust
const MAX_ROUNDS_PER_DISTRIBUTION: u64 = 10_000;
let new_rounds_count = new_rounds_count.min(MAX_ROUNDS_PER_DISTRIBUTION);
```

This ensures that even with arbitrarily long downtime or a very small `round_duration_seconds`, a single `distribute_rewards` invocation processes at most a bounded number of rounds, advances `latest_reward_event` by that amount, and allows subsequent invocations to catch up incrementally. Alternatively, replace the O(n) loop with a closed-form integral of the piecewise-quadratic `reward_rate_at` function to compute the total purse in O(1).

## Proof of Concept
1. Deploy an SNS with `round_duration_seconds = 1` (passes `VotingRewardsParameters::validate()` per `rs/sns/governance/src/reward.rs` lines 254–259).
2. Stop the SNS governance canister (or simulate subnet downtime) for 24 hours.
3. Restart the canister. The next timer tick calls `run_periodic_tasks` → `distribute_rewards`.
4. `new_rounds_count = 86,400`. The `for i in 1..=86400` loop with Decimal arithmetic executes, exhausting the 5-billion-instruction limit and causing a trap.
5. `latest_reward_event` is never updated. Every subsequent timer invocation recomputes `new_rounds_count ≥ 86,400` and traps again.
6. With 1 year of downtime, `new_rounds_count = 31,536,000`, which is orders of magnitude beyond the instruction limit.

A deterministic reproduction can be written as a PocketIC or state-machine integration test: initialize an SNS with `round_duration_seconds = Some(1)`, advance the mock clock by 86,400 seconds without calling `run_periodic_tasks`, then call `run_periodic_tasks` and assert that the canister traps or that `latest_reward_event` remains unchanged across repeated invocations.
