### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Unverified Signatures for Accounts with Pending Deploy-Account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally bypasses the `__validate__` entry point (the account's signature-verification logic) for any invoke transaction with nonce=1 when the sender account has a pending deploy-account in the mempool. An attacker who observes a deploy-account transaction in the mempool can front-run the legitimate user's nonce-1 invoke by submitting a forged invoke with an arbitrary/invalid signature. The gateway admits the forged transaction without ever calling `__validate__`, satisfying the "High – Mempool/gateway admission accepts invalid transactions before sequencing" impact criterion.

---

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must carry a signature that passes the account's `__validate__` entry point. The `skip_stateful_validations` UX shortcut creates a window where this invariant is silently dropped.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold simultaneously:
1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender)` returns `true`

**Condition 3 is too broad.** The mempool check returns `true` whenever the address appears in the pool for *any* reason: [2](#0-1) 

An attacker who submits a deploy-account for address X (or who simply observes Alice's deploy-account in the pool) can immediately submit a forged invoke for X with nonce=1 and an invalid signature. Condition 3 is satisfied by Alice's deploy-account, so the attacker's forged invoke also gets `skip_validate = true`.

**How the skip propagates to zero signature verification:**

`run_pre_validation_checks` calls `skip_stateful_validations` and returns the boolean: [3](#0-2) 

`extract_state_nonce_and_run_validations` then passes the flag directly to `run_validate_entry_point`: [4](#0-3) 

Inside `run_validate_entry_point`, `skip_validate = true` sets `execution_flags.validate = false`: [5](#0-4) 

`StatefulValidator::perform_validations` then returns `Ok(())` immediately for invoke transactions when `validate == false`, without ever calling `validate_tx` (the `__validate__` entry point): [6](#0-5) 

The `validate_tx` implementation confirms: when `execution_flags.validate == false`, the function returns `Ok(None)` before touching the account's class or signature: [7](#0-6) 

**Pre-validation checks do not catch the forged transaction.** `validate_by_mempool` (called before `skip_stateful_validations`) only checks for duplicate hashes and nonce range — not signature validity: [8](#0-7) 

`perform_pre_validation_stage` (called inside `run_validate_entry_point`) checks nonce, fee bounds, and balance — but not the signature: [9](#0-8) 

The balance check (`verify_can_pay_committed_bounds`) passes because Alice must pre-fund her account address before deploying it — a standard Starknet requirement.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

1. A forged invoke with an arbitrary/invalid signature is admitted to the mempool without any signature check.
2. The forged invoke occupies Alice's nonce-1 slot. Alice's legitimate invoke is either rejected as a duplicate nonce or forced to pay a higher fee via fee escalation to displace the forged transaction.
3. The forged invoke fails at execution time (when `__validate__` *is* called with `validate=true`), but the damage to admission integrity and user experience has already occurred.

---

### Likelihood Explanation

**Moderate.** The attack requires:
- Monitoring the public mempool for deploy-account transactions (trivial with a full node or gateway subscription).
- Submitting a forged invoke before the legitimate user's nonce-1 invoke arrives (a standard front-running window).
- The target account must be pre-funded (required for any deploy-account to succeed, so this is always true in practice).

No privileged access, no special tooling, and no cooperation from the victim is required.

---

### Recommendation

The `skip_stateful_validations` bypass should not completely suppress signature verification. Concrete options:

1. **Verify against the declared class.** The deploy-account transaction in the pool carries the `class_hash`. The gateway can instantiate the account class from the class manager and call `__validate__` against it even before the account is deployed on-chain.

2. **Require atomic submission.** Accept the deploy-account + invoke pair only as a single gateway call, so the gateway can enforce that both transactions originate from the same submitter and carry consistent signatures.

3. **Narrow the nonce window.** Restrict the skip to `nonce == 1` *and* require that the deploy-account transaction in the pool was submitted in the same gateway request (e.g., carry a session token or batch ID), preventing third-party exploitation.

---

### Proof of Concept

```
1. Alice submits deploy_account for address X (nonce=0).
   → Mempool: account X added to tx_pool.

2. Attacker observes Alice's deploy_account in the mempool.

3. Attacker submits forged_invoke for address X:
     nonce = 1
     signature = [0xdeadbeef, ...]  // arbitrary invalid bytes

4. Gateway stateful validation:
     account_nonce = get_nonce(X) = 0          ✓ (not deployed yet)
     tx.nonce() == 1                            ✓
     account_tx_in_pool_or_recent_block(X) = true  ✓ (Alice's deploy_account)
     → skip_validate = true
     → execution_flags.validate = false
     → __validate__ NOT called
     → forged_invoke ADMITTED to mempool

5. Alice submits her legitimate invoke (nonce=1).
   → Mempool: duplicate nonce for X; rejected or requires fee escalation.

6. Forged invoke reaches the batcher:
   → perform_pre_validation_stage succeeds (nonce=1 >= on-chain nonce=1 after deploy)
   → __validate__ IS called (validate=true in sequencing path)
   → Signature check fails → transaction reverted
   → Alice's nonce slot was occupied; she must resubmit with higher fee.
``` [10](#0-9) [2](#0-1)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
