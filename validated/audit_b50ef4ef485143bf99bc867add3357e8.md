### Title
EVM Block Time Advances During Pause, Accruing Interest While Users Cannot Protect Positions - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
The `Heartbeat` resource's `heartbeat()` function unconditionally calls `InternalEVM.commitBlockProposal()` on every Flow block, advancing EVM block height and timestamp even when EVM operations are paused. All state-mutating EVM entry points guard on `!EVM.isPaused()`, so users with open positions in EVM-based lending/DeFi protocols cannot repay, add collateral, or close positions during the pause. Because `block.timestamp` and `block.number` continue to advance, those contracts keep accruing interest and updating liquidation thresholds throughout the pause period.

### Finding Description
`EVM.isPaused()` reads a boolean from `/storage/evmOperationsPaused` and is checked in every user-facing state-mutating function: `EVM.run()`, `EVM.batchRun()`, `EVM.createCadenceOwnedAccount()`, `CadenceOwnedAccount.call()`, `CadenceOwnedAccount.deploy()`, `CadenceOwnedAccount.deposit()`, `CadenceOwnedAccount.withdraw()`, `CadenceOwnedAccount.depositTokens()`, `CadenceOwnedAccount.withdrawTokens()`, and `CadenceOwnedAccount.withdrawNFT()`. [1](#0-0) [2](#0-1) 

However, the `Heartbeat` resource — invoked as a system call once per Flow block — contains no such guard:

```cadence
access(all) resource Heartbeat {
    access(all)
    fun heartbeat() {
        InternalEVM.commitBlockProposal()   // no isPaused() check
    }
}
``` [3](#0-2) 

`commitBlockProposal()` constructs a new EVM block whose timestamp is derived directly from the Cadence block's wall-clock timestamp:

```go
timestamp := uint64(cadenceBlock.Timestamp / int64(time.Second))
``` [4](#0-3) 

This means every Flow block that is finalized during a pause period produces a new EVM block with a monotonically increasing `block.timestamp` and `block.number`, even though no EVM transactions can be submitted. [5](#0-4) 

### Impact Explanation
Any EVM lending or DeFi protocol deployed on Flow EVM that calculates interest, funding rates, or liquidation thresholds using `block.timestamp` or `block.number` will continue to accrue those charges during the pause. Users with open borrow positions or leveraged positions cannot call `repay`, `addCollateral`, or `close` because every EVM state-mutating path is blocked by the `isPaused()` guard. When the pause is lifted, users may find their health factor has dropped below the liquidation threshold due to interest that accrued while they were locked out, resulting in immediate liquidation of their collateral by third-party liquidators.

### Likelihood Explanation
The EVM pause mechanism is explicitly designed for governance-triggered maintenance windows. The comment in the contract states: *"Only the Governance Committee can pause the EVM transactions, with a multi-sig Cadence transaction."* Any pause of non-trivial duration (hours to days) on a network hosting active lending protocols will cause measurable interest accrual. The root cause is a missing guard in a single function that is called unconditionally on every block. [6](#0-5) 

### Recommendation
Add an `isPaused()` check inside `Heartbeat.heartbeat()` so that EVM block proposals are not committed during a pause period:

```cadence
access(all)
fun heartbeat() {
    if !EVM.isPaused() {
        InternalEVM.commitBlockProposal()
    }
}
```

This mirrors the fix applied in the Isomorph protocol: freeze the time-based accrual mechanism when the system is paused, and resume it from the correct point when unpaused. Alternatively, record the EVM block height at pause time and replay the correct timestamp on unpause so that no phantom time elapses from the perspective of EVM contracts.

### Proof of Concept

1. Governance sets `/storage/evmOperationsPaused = true` on the EVM contract account.
2. A user (`Alice`) has an open borrow position in an EVM lending protocol (e.g., an Aave-style contract) deployed on Flow EVM. Her health factor is 1.05 (5% above liquidation threshold).
3. Every Flow block, `Heartbeat.heartbeat()` fires as a system call and calls `InternalEVM.commitBlockProposal()`.
4. Each committed EVM block carries a real wall-clock timestamp derived from the Cadence block header.
5. The lending protocol's interest index updates on each interaction using `block.timestamp`; because EVM blocks keep being produced, the next interaction after unpause will compute accrued interest for the entire paused duration.
6. Alice attempts to call `EVM.run()` with a `repay` transaction — it reverts with `"EVM operations are temporarily paused"`.
7. After 48 hours of pause, the governance unpauses. Alice's health factor has dropped below 1.0 due to accrued interest. A liquidator immediately liquidates her collateral before she can repay. [7](#0-6) [8](#0-7)
