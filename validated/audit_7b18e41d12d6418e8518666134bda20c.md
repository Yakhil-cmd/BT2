Based on my analysis of the codebase, I found a valid analog. The external report's vulnerability class is **incomplete state validation during a completion/finalization step** — a protocol trusts a sub-system has satisfied all obligations without actually verifying them. In the CKD protocol, the coordinator trusts that participants correctly computed their shares without any cryptographic verification.

### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (`src/confidential_key_derivation/protocol.rs`)

### Summary
In `do_ckd_coordinator`, the coordinator collects `(norm_big_y, norm_big_c)` shares from all participants and sums them to produce the final `CKDOutput`, without any verification that the received values were honestly computed. A single malicious participant can send arbitrary group elements, silently corrupting the derived confidential key delivered to the application.

### Finding Description
The coordinator's aggregation loop in `do_ckd_coordinator` (lines 50–55) performs no validation of participant-supplied values:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each honest participant is supposed to compute, per `compute_signature_share` (lines 148–182):

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` [2](#0-1) 

The coordinator checks only that a message arrived from every participant (via `recv_from_others`), which is the **primary check**. The **secondary check** — that the received `(big_y, big_c)` pair is a correctly formed ElGamal encryption of the participant's key share — is entirely absent. There is no zero-knowledge proof, no consistency check against the participant's public verification key, and no post-aggregation sanity check.

This is structurally identical to the reported Notional bug: the protocol checks that all parties *participated* (analogous to checking `vaultShares == 0`) but does not check that each party *fulfilled its cryptographic obligation* (analogous to checking secondary debt is zero).

### Impact Explanation
The final `CKDOutput` satisfies `big_c − app_sk · big_y = msk · H(pk ‖ app_id)` only when all participants are honest. A malicious participant who sends `(big_y', big_c')` of their choosing causes the coordinator to output a corrupted `CKDOutput`. When the application unmasks it with `app_sk`, it obtains a wrong confidential key — an **unusable cryptographic output** — with no indication that corruption occurred. This matches the allowed High impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation
Any single participant in the protocol can trigger this. No special privilege, colluding parties, or leaked secrets are required. The attacker only needs to be an enrolled participant and deviate from the protocol by sending arbitrary group elements in place of their honest share.

### Recommendation
Add a zero-knowledge proof of correct share formation alongside each `(norm_big_y, norm_big_c)` message. Concretely, each participant should prove in zero knowledge that:

> `norm_big_c − λ_i · x_i · H(pk ‖ app_id)` is a scalar multiple of `app_pk` with the **same** scalar used to produce `norm_big_y` (i.e., a proof of equality of discrete logarithms across two bases: `G` and `app_pk`).

The coordinator must verify all such proofs before summing the shares. This is a standard Chaum–Pedersen proof and adds only a constant-size proof per participant.

### Proof of Concept
1. Honest participants `P_1, …, P_{n-1}` compute correct `(norm_big_y_i, norm_big_c_i)` and send them to the coordinator.
2. Malicious participant `P_n` sends `(0_G1, 0_G1)` (identity elements) instead of their honest share.
3. The coordinator's loop at line 50–55 sums all values, silently dropping `P_n`'s contribution.
4. The resulting `CKDOutput` is `(Σ_{i<n} norm_big_y_i, Σ_{i<n} norm_big_c_i)`, which is missing `P_n`'s share.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `Σ_{i<n} λ_i · x_i · H(pk ‖ app_id)` — a wrong key — with no error or warning. [3](#0-2)
