### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Delegator's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in the Pool contract contains an explicitly acknowledged unbounded loop over a delegator's `pool_member_epoch_balance` trace. A delegator who repeatedly changes their delegated balance across many epochs without claiming rewards will grow this trace without bound. Once the trace is large enough, every call to `claim_rewards` will run out of gas, permanently freezing the delegator's unclaimed yield.

### Finding Description

`claim_rewards` in `src/pool/pool.cairo` calls `calculate_rewards`, which iterates over every entry in the caller's `pool_member_epoch_balance` trace that falls between the last claimed checkpoint and the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` (with a partial amount) invokes `set_member_balance` / `increase_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`. The `insert` implementation appends a **new** checkpoint whenever the target epoch key differs from the last stored key:

```cairo
} else {
    // Checkpoint keys must be non-decreasing.
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

Because the key is always `current_epoch + K`, one balance-change operation per epoch appends one new checkpoint. A delegator who performs one balance change per epoch for N epochs without claiming accumulates N unclaimed checkpoints. When `claim_rewards` is finally called, the loop must iterate all N entries. If N is large enough to exhaust the Starknet transaction gas limit, the call reverts and the delegator's accumulated yield is permanently inaccessible.

The `entry_to_claim_from` cursor is only advanced and persisted on a **successful** `claim_rewards` execution:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
self.write_pool_member_info(:pool_member, :pool_member_info);
``` [3](#0-2) 

Because the cursor is never advanced on a failed (OOG) call, every subsequent `claim_rewards` attempt restarts the loop from the same position and hits the same OOG condition, making the freeze permanent.

### Impact Explanation

A delegator's entire accumulated unclaimed yield becomes permanently inaccessible. No partial-claim mechanism exists; the only path to rewards is through `claim_rewards`, which unconditionally iterates the full unclaimed trace segment. This matches **"Permanent freezing of unclaimed yield"** (High severity).

### Likelihood Explanation

**Medium.** The attack is self-inflicted: only the pool member or their reward address can call `add_to_delegation_pool`, so a third party cannot grow another user's trace. However:
- A delegator who actively manages their position (e.g., dollar-cost averaging, partial exits) across many epochs without claiming will naturally accumulate trace entries.
- The codebase itself flags the risk with the comment *"This loop is unbounded but unlikely to exceed gas limits"*, acknowledging the issue exists but relying on practical limits rather than a hard cap.
- Starknet's per-transaction resource limits are finite; a sufficiently long history of balance changes will eventually breach them.

### Recommendation

1. **Enforce a maximum trace length** by capping the number of balance changes a pool member may have between consecutive `claim_rewards` calls (e.g., require a claim before the N-th balance change is allowed).
2. **Alternatively**, split `calculate_rewards` into a paginated form that processes at most `MAX_ENTRIES` checkpoints per call and persists the cursor, allowing incremental reward collection.
3. **At minimum**, add a hard assertion or early-return guard in `calculate_rewards` that reverts with a descriptive error (rather than OOG) when the trace segment exceeds a safe bound, so the failure mode is explicit and recoverable.

### Proof of Concept

1. Delegator calls `add_to_delegation_pool` once per epoch for K×N epochs (e.g., 1 000 epochs), never calling `claim_rewards`.
2. Each call appends one entry to `pool_member_epoch_balance` trace via `set_member_balance` → `trace.insert`.
3. After 1 000 epochs the trace contains 1 000 unclaimed checkpoints.
4. Delegator calls `claim_rewards`. `calculate_rewards` enters the `while` loop and iterates all 1 000 entries, each performing a storage read (`pool_member_trace.at(entry_to_claim_from)`) and a `find_sigma` lookup.
5. The transaction exhausts its gas budget and reverts.
6. Because `entry_to_claim_from` was never written back, every future `claim_rewards` call repeats step 4 and reverts identically — the yield is permanently frozen. [4](#0-3) [5](#0-4)
