### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures for Accounts with Pending Deploy-Account - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (which performs signature verification) for any Invoke transaction with nonce=1 when the target account has **any** transaction present in the mempool or a recent block. An unprivileged attacker who observes a legitimate user's `deploy_account` transaction in the mempool can immediately submit an Invoke transaction for that same account address with an arbitrary/invalid signature, and the gateway will admit it to the mempool without ever calling `__validate__`.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

The function returns `true` (skip `__validate__`) when all three conditions hold:
1. The transaction is an `Invoke` type
2. `tx.nonce() == Nonce(Felt::ONE)` — the transaction nonce is exactly 1
3. `account_nonce == Nonce(Felt::ZERO)` — the on-chain account nonce is 0 (account not yet deployed)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

The fourth check is implemented as: [2](#0-1) 

This returns `true` if **any** transaction from the account address exists in the pool or was seen in a recent block — it does **not** verify that the existing transaction is specifically a `deploy_account`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is circular: the very transaction being admitted is the one that would need to pass validation.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false, ... }`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate=false`, the function returns `Ok(())` immediately without calling `__validate__`: [4](#0-3) 

The production default configuration sets `max_allowed_nonce_gap = 200`, so an Invoke with nonce=1 passes the nonce range check when account_nonce=0: [5](#0-4) 

The `max_nonce_for_validation_skip` is `0x1`, confirming nonce=1 is the exact target window: [6](#0-5) 

### Impact Explanation

An attacker can inject an Invoke transaction with an **invalid/arbitrary signature** into the mempool for any account that has a pending `deploy_account` transaction, bypassing the `__validate__` entry-point entirely at the gateway admission stage. This breaks the invariant that all transactions admitted to the mempool have passed account-level signature verification.

The transaction will fail at batcher execution time (since `new_for_sequencing` always sets `validate: true`), so no unauthorized state change occurs. However:

- The invalid transaction occupies a mempool slot for the account at nonce=1.
- If fee escalation is disabled (the default), the legitimate user's own Invoke at nonce=1 is rejected with `DuplicateNonce` until the attacker's transaction is processed and rejected by the batcher, forcing the user to resubmit.
- The attacker can repeat this for every new `deploy_account` observed in the mempool, causing systematic griefing of new account deployments.

This matches the **High** impact scope: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

### Likelihood Explanation

- **Unprivileged trigger**: The attacker only needs to observe the mempool for `deploy_account` transactions (publicly visible) and know the target account address.
- **No special access required**: Any user can submit an Invoke transaction to the gateway.
- **Reliable**: The `skip_stateful_validations` condition is deterministic and always fires when nonce=1, account_nonce=0, and any transaction for the account is in the mempool.
- **Economically cheap**: The attacker only needs to pay the gateway's minimum resource bounds; the transaction is never executed.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** exists for the sender address in the mempool. Add a dedicated mempool API method such as `deploy_account_tx_in_pool(address)` that inspects transaction type, rather than checking for any transaction. This closes the window where an attacker can exploit a legitimate user's pending `deploy_account` to inject an unsigned Invoke.

Alternatively, still run `__validate__` but catch the specific failure case where the account contract does not yet exist (class hash is zero), treating that as a provisional pass only when a `deploy_account` is confirmed in the mempool.

### Proof of Concept

1. Legitimate user submits `deploy_account` for address `A` (nonce=0). It enters the mempool.
2. Attacker calls `mempool.account_tx_in_pool_or_recent_block(A)` → `true`.
3. Attacker constructs an Invoke transaction: `sender_address=A`, `nonce=1`, `signature=[0x1337, 0xdead]` (invalid).
4. Attacker submits to the gateway's `add_tx` endpoint.
5. Gateway stateful path:
   - `get_nonce_from_state(A)` → `Nonce(0)` (not deployed)
   - `validate_nonce`: `0 <= 1 <= 200` → passes
   - `validate_by_mempool`: checks duplicate hash/nonce only → passes
   - `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)=true` → returns `true`
   - `run_validate_entry_point(skip_validate=true)`: sets `validate=false`, `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`
6. Gateway calls `mempool.add_tx(attacker_invoke)` → attacker's transaction with invalid signature is now in the mempool at `(A, nonce=1)`.
7. Legitimate user submits their own Invoke at nonce=1 → rejected with `DuplicateNonce` (fee escalation disabled by default).
8. Batcher eventually executes attacker's transaction → `__validate__` fails → transaction rejected; but the legitimate user's transaction has already been displaced.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
