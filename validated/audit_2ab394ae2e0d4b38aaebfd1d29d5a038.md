### Title
Missing Cross-Phase Participant-Set Consistency Check in OT-Based ECDSA Presigning Allows Protocol State Confusion — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in `src/ecdsa/ot_based_ecdsa/presign.rs` explicitly omits the check that the presigning participant set (P1) is a subset of the triple-generation participant set (P0), and also never verifies that both supplied triples were generated by the **same** participant set. The `TriplePub` struct carries exactly this metadata (`participants` and `threshold`), but only the threshold is compared. A malicious coordinator can supply two triples produced by disjoint participant sets while running presigning with a third set that is a subset of both, causing the protocol to accept and output a presignature whose security proof is broken.

---

### Finding Description

The OT-based ECDSA pipeline is a three-phase process:

```
Triple Generation (P0)  →  Presigning (P1 ⊆ P0)  →  Signing (P2 ⊆ P1)
```

> Two multiplicative (Beaver) triples private shares … **generated by the same set of parties P₀**

And the orchestration document (`docs/ecdsa/ot_based_ecdsa/orchestration.md`, line 40) requires:

> P ⊇ P₀ ⊇ P₁ ⊇ P₂
