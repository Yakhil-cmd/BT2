### Title
ZK Transaction Flow Missing Intrinsic Gas Floor Allows EVM Refunds to Under-Compensate Operator - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the `eip_7623` feature is disabled, the ZK transaction flow sets `minimal_gas_used = 0` as the floor for gas accounting. The Ethereum transaction flow correctly sets this floor to `TX_INTRINSIC_GAS = 21_000`. Because EVM refunds (from SSTORE clears) are applied after computing `gas_used`, they can reduce the final `gas_used` below the intrinsic transaction cost of 21,000 gas. The operator is then paid less than the cost of processing the transaction, and the user receives a larger refund than they are entitled to.

---

### Finding Description

In `compute_gas_refund`, the `minimal_gas_used` parameter acts as a floor on the final `gas_used` after EVM refunds are applied:

```rust
// refund_calculation.rs line 56
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);
```

For the **Ethereum transaction flow**, `minimal_gas_used` is always at least `TX_INTRINSIC_GAS = 21_000`:

```rust
// ethereum/validation_impl.rs line 159-162
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, TX_INTRINSIC_GAS)
}
```

For the **ZK transaction flow**, `minimal_gas_used` is `0` when `eip_7623` is disabled:

```rust
// zk/validation_impl.rs line 538-541
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, 0)
}
```

This `0` propagates through `TxContextForPreAndPostProcessing::minimal_ergs_to_charge` and is used in `before_refund`:

```rust
// zk/mod.rs line 426
let min_gas_used = context.minimal_ergs_to_charge.0 / ERGS_PER_GAS; // = 0
```

The EVM refund counter (from SSTORE clears) is capped at `gas_used / 5` per EIP-3529. When execution gas is small relative to the refund, the final `gas_used` can fall below `TX_INTRINSIC_GAS`. With no floor enforced, the operator is under-compensated.

**Concrete example:**
- `TX_INTRINSIC_GAS = 21_000`, one warm SSTORE clear costs 2,900 gas, yields 4,800 refund
- `gas_used_before_refund = 21_000 + 2_900 = 23_900`
- `max_refund = 23_900 / 5 = 4_780`
- `actual_refund = min(4_800, 4_780) = 4_780`
- `gas_used_after_refund = 23_900 − 4_780 = 19_120 < 21_000`

The operator receives fees for 19,120 gas instead of the minimum 21,000.

---

### Impact Explanation

The operator (sequencer/prover) is under-compensated for the fixed intrinsic cost of processing a transaction. The user receives a refund that partially covers the intrinsic gas cost, which they should not be entitled to. The maximum loss per transaction is bounded by `TX_INTRINSIC_GAS × gas_price`. An attacker can systematically exploit this by submitting ZK transactions that clear pre-set storage slots with minimal additional execution, extracting a small but consistent subsidy from the operator on every such transaction.

---

### Likelihood Explanation

Any unprivileged user can submit a ZK transaction that clears storage slots (e.g., interacting with a contract that has pre-set storage). The pattern requires only that execution gas is small relative to the EVM refund, which is achievable with a single warm SSTORE clear. No special privileges, oracle manipulation, or governance access is required. The condition is triggered whenever `eip_7623` is not compiled in, which is the default non-feature-gated path.

---

### Recommendation

In `compute_calldata_tokens` for the ZK transaction flow, return `TX_INTRINSIC_GAS` as the minimum `minimal_gas_used` when `eip_7623` is disabled, mirroring the Ethereum transaction flow:

```rust
// zk/validation_impl.rs
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, TX_INTRINSIC_GAS)  // enforce intrinsic gas floor
}
```

This ensures the operator is always compensated for at least the base 21,000 gas intrinsic cost, regardless of EVM refunds.

---

### Proof of Concept

1. Deploy a contract that pre-sets a storage slot (e.g., `SSTORE(0, 1)` in constructor).
2. Submit a ZK transaction calling a function that clears the slot (`SSTORE(0, 0)`).
3. Execution gas ≈ 2,900 (warm SSTORE clear). EVM refund counter = 4,800.
4. `gas_used_before_refund = 21,000 + 2,900 = 23,900`.
5. `max_refund = 23,900 / 5 = 4,780`. `actual_refund = 4,780`.
6. `gas_used_after_refund = 19,120`.
7. `min_gas_used = 0` (ZK flow, no `eip_7623`), so floor does not apply.
8. Operator receives `19,120 × gas_price` instead of `≥ 21,000 × gas_price`.
