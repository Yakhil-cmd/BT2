### Title
Stale Pre-Resharding `target_shard_index` Corrupts `outcome_root_proof` Merkle Path After Epoch Split — (`chain/client/src/view_client_actor.rs`)

---

### Summary

`ViewClientActor::handle(GetExecutionOutcome)` computes `target_shard_index` once from the **pre-resharding epoch's shard layout**, then uses that index to select a leaf from `outcome_roots` that is built from the **post-resharding block's chunks** (which has a different number of shards and different index assignments). When `get_next_block_hash_with_new_chunk` crosses an epoch boundary and returns the **second child shard** of the split, the index is off by at least one, the wrong Merkle path is returned, and every light-client that calls `verify_path(block_header_lite.inner_lite.outcome_root, outcome_root_proof, chunk_outcome_root)` rejects the proof.

---

### Finding Description

The handler at lines 1148–1182 of `chain/client/src/view_client_actor.rs` executes the following sequence:

**Step 1 — index computed from old layout:**
```rust
let epoch_id = *self.chain.get_block(&outcome_proof.block_hash)?.header().epoch_id();
let shard_layout = self.epoch_manager.get_shard_layout(&epoch_id).into_chain_error()?;
let target_shard_id =
    account_id_to_shard_id(self.epoch_manager.as_ref(), &account_id, &epoch_id)
        .into_chain_error()?;
let target_shard_index = shard_layout
    .get_shard_index(target_shard_id)   // ← index in PRE-resharding layout
    .map_err(Into::into)
    .into_chain_error()?;
``` [1](#0-0) 

**Step 2 — `get_next_block_hash_with_new_chunk` may cross the epoch boundary:**

Inside `Chain::get_next_block_hash_with_new_chunk`, when the next block is in a new epoch with a different shard layout, `shard_ids` is expanded to the children of the parent shard and `shard_layout` is updated to the new layout. The function then returns `(block_hash, child_shard_id)` — a child shard in the **new** layout. [2](#0-1) 

**Step 3 — `outcome_roots` built from the new block, `target_shard_index` never updated:**
```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let outcome_roots = self.chain.get_block(&h)?.chunks().iter()
        .map(|header| *header.prev_outcome_root())
        .collect::<Vec<_>>();          // ← N+1 entries, new layout
    if target_shard_index >= outcome_roots.len() { ... }   // ← only catches OOB
    Ok(GetExecutionOutcomeResponse {
        outcome_proof: outcome_proof.into(),
        outcome_root_proof: merklize(&outcome_roots).1[target_shard_index].clone(),
        //                                              ^^^^^^^^^^^^^^^^^^^
        //                  still the OLD layout index — never recomputed
    })
``` [3](#0-2) 

**Why the bounds check does not save it:**

In V2/V3 layouts, `get_shard_index` uses an explicit `id_to_index_map` — shard ID and shard index are **not** the same. [4](#0-3) 

When parent shard S1 (at index `j`) splits into children `[max_id+1, max_id+2]` via `derive_impl`, the splice inserts two shards at position `j`, giving child1 index `j` and child2 index `j+1`. Every shard after the split position also shifts up by one. [5](#0-4) 

The old layout had N shards → `target_shard_index` ∈ [0, N−1]. The new layout has N+1 shards → `outcome_roots.len()` = N+1. So `target_shard_index < N < N+1` always, the `InconsistentState` guard never fires, and the wrong Merkle path is silently returned.

**When does `get_next_block_hash_with_new_chunk` return child2 instead of child1?**

The function iterates `shard_ids = [child1, child2]` and returns the first child with a new chunk (`height_included == block.header().height()`). If child1's chunk is missing (chunk producer offline, chunk not included) in the first post-resharding block where child2 has a new chunk, the function returns `(h, child2)`. `target_shard_index` is still `j` (child1's index), but the correct index for child2 is `j+1`. [6](#0-5) 

---

### Impact Explanation

The exact corrupted value is `outcome_root_proof`: it is the Merkle path for leaf `j` (child1's `prev_outcome_root`) instead of leaf `j+1` (child2's `prev_outcome_root`). The light-client proof verification step:

```python
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
# must equal block_header_lite.inner_lite.outcome_root
```

fails because the sibling hashes and direction bits in the path correspond to the wrong leaf position. Any light client or bridge relying on `RpcLightClientExecutionProofResponse` will reject the proof as invalid, causing valid finalized transactions to appear unconfirmed. [7](#0-6) 

No funds are stolen and no on-chain state is corrupted. The impact is confined to the RPC/light-client proof layer: the client-proof invariant (`outcome_root_proof` verifies against `block_header_lite.inner_lite.outcome_root`) is broken for outcomes whose next-new-chunk block crosses a resharding epoch boundary and whose child shard is the second child.

---

### Likelihood Explanation

Three conditions must coincide:

1. A resharding epoch boundary exists between the outcome block and the next block that has a new chunk for the outcome's shard (realistic: the outcome is in the last new-chunk block of the epoch, and the epoch ends before the shard produces another new chunk).
2. The account's `receiver_id` maps to the **second** child shard in the new layout (roughly 50% of accounts in the split shard).
3. The first child shard's chunk is **missing** in the first post-resharding block where the second child has a new chunk (possible whenever a chunk producer is offline).

Conditions 1 and 2 are structurally guaranteed to occur for some transactions during any resharding event. Condition 3 requires a chunk to be missing, which is an operational edge case but not rare in production. The combination makes this a low-to-medium likelihood event that is guaranteed to surface eventually across any resharding.

---

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` from the **new block's epoch layout** using the returned child `target_shard_id`:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    // Recompute index in the layout that actually governs block `h`.
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout =
        self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```

This mirrors the pattern already used correctly inside `get_next_block_hash_with_new_chunk` itself, where `shard_layout.get_shard_index(shard_id)` is always called against the layout that matches the current block's epoch. [8](#0-7)

### Proof of Concept

A test-loop reproducer using the existing `DynamicResharding` infrastructure:

1. Configure a two-shard genesis with `DynamicResharding` enabled and a low `memory_usage_threshold` so resharding triggers after a few blocks.
2. Submit a cross-shard receipt whose `receiver_id` maps to the **second** child shard in the post-split layout (deterministic from the boundary account chosen by `find_trie_split`).
3. Produce blocks until the outcome is finalized in epoch E (parent shard S1).
4. Simulate a missing chunk for child1 in the first block of epoch E+2 (the post-resharding epoch) by dropping the chunk producer's endorsement for child1 while child2 produces normally.
5. Call `view_client.handle(GetExecutionOutcome { id: receipt_id, receiver_id })`.
6. Assert that `verify_path(block_proof.block_header_lite.inner_lite.outcome_root, execution_outcome.outcome_root_proof, chunk_outcome_root)` returns **true** — this assertion will **fail** without the fix, demonstrating the corrupted Merkle path.

The existing proof-verification harness in `test-loop-tests/src/tests/light_client.rs` (`check_outcome_proofs`) provides the exact verification logic needed. [9](#0-8)
