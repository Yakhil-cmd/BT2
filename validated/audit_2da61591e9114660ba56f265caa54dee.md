### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `Staking::change_reward_address` and `Pool::change_reward_address` accept any `ContractAddress` as the new reward destination. Neither function checks whether the supplied address is the zero address (`0x0`). A staker or pool member can call either function with `reward_address = 0`, after which every subsequent `claim_rewards` call will attempt to transfer accumulated STRK rewards to address zero, permanently destroying or freezing the unclaimed yield.

---

### Finding Description

**Root cause — `staking.cairo` `change_reward_address`:** [1](#0-0) 

The only guard present is a check that the address is not a registered token contract: [2](#0-1) 

There is no `assert!(reward_address.is_non_zero(), ...)` guard. The zero address passes this check because it is not a token contract, so the write at line 530 succeeds unconditionally.

**Root cause — `pool.cairo` `change_reward_address`:** [3](#0-2) 

The same pattern: only a token-address check is present (per the interface spec), no zero-address guard.

**Downstream effect — `staking.cairo` `send_rewards_to_staker`:** [4](#0-3) 

`reward_address` is read directly from storage and passed to `checked_transfer`. If it is zero, the ERC-20 transfer either reverts (freezing the rewards in the contract forever) or succeeds and burns the tokens, depending on the STRK token implementation.

**Downstream effect — `pool.cairo` `claim_rewards`:** [5](#0-4) 

Same pattern: `reward_address` is read from storage and used as the transfer recipient with no zero-address guard.

The same missing check also exists in the initial `stake` call: [6](#0-5) 

---

### Impact Explanation

A staker or pool member who calls `change_reward_address(0)` will have all future reward claims routed to address zero. Because `unclaimed_rewards_own` is zeroed out after the transfer attempt, the rewards are either burned (if the ERC-20 allows transfers to zero) or the claim reverts every time it is attempted, permanently freezing the accumulated yield. This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The call is permissionless for any active staker or pool member — no privileged role is required. A user could trigger this accidentally (e.g., passing a default/uninitialized address) or a malicious actor could grief a staker by social-engineering them into making the call. The function is a standard lifecycle operation with no cooldown or time-lock, so the window for exploitation is always open while the staker is active.

---

### Recommendation

Add a zero-address guard at the top of both `change_reward_address` implementations, and also in `stake` / `enter_delegation_pool` where `reward_address` is first recorded:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

Apply the same guard to `operational_address` in `stake` and `declare_operational_address` for defence-in-depth.

---

### Proof of Concept

1. Staker calls `staking.stake(amount, reward_address: 0, operational_address: X)` — accepted because `0` is not a token address.
2. Alternatively, an existing staker calls `staking.change_reward_address(reward_address: 0)` — accepted for the same reason.
3. After rewards accrue, anyone calls `staking.claim_rewards(staker_address)`.
4. `send_rewards_to_staker` reads `reward_address = 0` from storage and calls `checked_transfer(recipient: 0, amount: rewards)`.
5. Depending on the STRK ERC-20 implementation: the transfer reverts (rewards permanently frozen in the contract) or succeeds and burns the tokens (rewards permanently destroyed).
6. The identical path exists for pool members via `Pool::change_reward_address` → `Pool::claim_rewards` → `checked_transfer(recipient: 0, ...)`.
