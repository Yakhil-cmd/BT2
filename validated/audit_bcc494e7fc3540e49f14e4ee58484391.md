### Title
Gateway Stateless Validator Skips `max_l2_gas_amount` Check for Declare Transactions, Admitting Over-Bounded Declares — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `max_l2_gas_amount` cap for `RpcTransaction::Declare`. Any unprivileged user can submit a Declare V3 with `l2_gas.max_amount` exceeding the configured `max_l2_gas_amount` (default: 1,210,000,000) and the gateway will admit it. The blockifier then uses `l2_gas.max_amount` directly as the initial Sierra gas limit for execution. The premise that the blockifier will *reject* such a transaction is incorrect — if the account balance covers `max_amount × max_price_per_unit`, the Declare executes successfully with a higher-than-intended gas limit, bypassing the gateway's admission control entirely.

### Finding Description

In `validate_resource_bounds`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The `if let RpcTransaction::Declare(_) = tx {}` branch is an intentional no-op that causes the `else if` branch to be skipped entirely for Declare transactions. The TODO comment confirms the developers are aware this check is absent.

The test suite explicitly asserts this bypass is the expected behavior:

```rust
fn valid_l2_gas_amount_on_declare(...)  // l2_gas.max_amount = 200, max_l2_gas_amount = 100
    assert_matches!(tx_validator.validate(&tx), Ok(()));
``` [2](#0-1) 

And `test_invalid_max_l2_gas_amount` only parameterizes over `DeployAccount` and `Invoke`, explicitly excluding `Declare`: [3](#0-2) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* against the previous block's threshold — it performs no `max_l2_gas_amount` cap check for any transaction type: [4](#0-3) 

Critically, `l2_gas.max_amount` is used directly as the **initial Sierra gas limit** for execution in `TransactionContext::initial_sierra_gas`:

```rust
TransactionInfo::Current(CurrentTransactionInfo {
    resource_bounds: ValidResourceBounds::AllResources(AllResourceBounds { l2_gas, .. }),
    ..
}) => {
    l2_gas.max_amount  // <-- execution gas limit set from user-supplied field
}
``` [5](#0-4) 

The only downstream backstop is `verify_can_pay_committed_bounds`, which checks `account_balance >= max_amount × max_price_per_unit`: [6](#0-5) 

### Impact Explanation

An attacker with sufficient STRK balance can submit a Declare with `l2_gas.max_amount` set to any value above `max_l2_gas_amount` (up to the value their balance can cover at `min_gas_price`). The gateway admits it. The blockifier executes it with a gas limit equal to the attacker-supplied `max_amount`, not the gateway-enforced cap. The bouncer accounts for *actual* gas used (not `max_amount`), so the Declare executes and lands in a block. The gateway's admission control for Declare transactions is entirely ineffective for the `max_l2_gas_amount` dimension.

Concrete corrupted admission value: a Declare with `l2_gas.max_amount = max_l2_gas_amount + N` (for any N ≥ 1) is admitted and executed when the gateway's own policy requires rejection.

### Likelihood Explanation

Trivially exploitable by any user who can craft a Declare transaction. No special privileges required. The bypass is unconditional for all Declare transactions — the check is structurally absent, not gated on any condition.

### Recommendation

Remove the `if let RpcTransaction::Declare(_) = tx {}` no-op branch and apply the same `max_l2_gas_amount` check to Declare transactions as is applied to Invoke and DeployAccount. The TODO comment should be resolved by adding the check, not by leaving it absent.

### Proof of Concept

The existing test `valid_l2_gas_amount_on_declare` already serves as the proof of concept — it asserts `Ok(())` for a Declare with `l2_gas.max_amount = 200` when `max_l2_gas_amount = 100`: [2](#0-1) 

To confirm end-to-end admission: submit a Declare V3 with `l2_gas.max_amount = max_l2_gas_amount + 1` and `l2_gas.max_price_per_unit = min_gas_price` from an account with balance ≥ `(max_l2_gas_amount + 1) × min_gas_price`. Both `StatelessTransactionValidator::validate` and the stateful path will return `Ok`, and the transaction will be sequenced with an execution gas limit of `max_l2_gas_amount + 1`.
