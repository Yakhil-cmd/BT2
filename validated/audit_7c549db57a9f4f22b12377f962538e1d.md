### Title
Unprivileged caller can permanently suppress per-block consensus rewards by calling `update_rewards` with `disable_rewards: true` — (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is a public, permissionless function that accepts a caller-controlled `disable_rewards` boolean. Because the function writes `last_reward_block` to the current block **before** branching on `disable_rewards`, any unprivileged attacker can call it once per block with `disable_rewards: true` to permanently skip reward distribution for that block. No legitimate caller can recover the missed rewards for that block.

---

### Finding Description

`update_rewards` in `StakingRewardsManagerImpl` has no access-control guard beyond `general_prerequisites()` (which only checks the pause flag):

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // ← last_reward_block is written HERE, before the disable_rewards branch
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // exits without distributing
    }
    ...
    self._update_rewards(...);
}
``` [1](#0-0) 

`last_reward_block` is a **global** storage slot, not per-staker. Once it is set to the current block number, the guard `current_block_number > self.last_reward_block.read()` prevents any further call to `update_rewards` in the same block: [2](#0-1) 

The attacker's call sequence per block:
1. Call `update_rewards(any_valid_active_staker, disable_rewards: true)`.
2. `last_reward_block` is set to the current block.
3. The function returns early — no rewards are credited to any staker or pool.
4. Every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

The test suite itself confirms this behaviour: [3](#0-2) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When `disable_rewards: true` is used by an attacker, the block rewards that would have been credited to stakers and pool members via `_update_rewards` are never distributed. The `reward_supplier` is never notified (`update_unclaimed_rewards_from_staking_contract` is never called), so the rewards are permanently unaccounted for. Stakers and delegators lose the yield they were entitled to for every block the attacker front-runs. An attacker running a bot can suppress rewards for every single block at negligible cost (one transaction per block). [4](#0-3) 

---

### Likelihood Explanation

**High.** The function is fully public with no role check. The only precondition is that a valid, active staker address is supplied — trivially satisfied by reading any staker from the public `stakers` vector. The attack requires one cheap transaction per block and is profitable for any party that benefits from suppressing staking rewards (e.g., a competing protocol, or a staker who wants to reduce dilution from other stakers' rewards). [5](#0-4) 

---

### Recommendation

Restrict `update_rewards` to a trusted caller. Based on the consensus-rewards design, the intended caller is the block proposer or an authorized consensus/attestation contract. Add an access-control assertion at the top of the function, for example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.assert_caller_is_rewards_manager();   // new guard
    self.general_prerequisites();
    ...
}
```

Alternatively, remove the `disable_rewards` parameter from the public interface entirely and handle the "no rewards" case internally based on protocol state (e.g., `is_pre_consensus()`), so no external caller can influence whether rewards are distributed.

---

### Proof of Concept

The following pseudocode (mirroring the existing test style) demonstrates the attack:

```cairo
// Precondition: consensus rewards are active, staker has been staked for K+ epochs.
// Attacker is any unprivileged address.

// Block N: attacker front-runs the legitimate update_rewards call.
cheat_caller_address_once(staking_contract, attacker_address);
staking_rewards_dispatcher.update_rewards(
    staker_address: any_valid_staker,
    disable_rewards: true,   // ← attacker-controlled
);
// last_reward_block is now set to block N.
// No rewards were distributed.

// Any legitimate call in the same block now reverts:
let result = staking_rewards_safe_dispatcher
    .update_rewards(staker_address: any_valid_staker, disable_rewards: false);
// → panics with REWARDS_ALREADY_UPDATED

// Staker claims rewards → zero for block N.
let rewards = staking_dispatcher.claim_rewards(staker_address: any_valid_staker);
assert!(rewards == 0);
```

The existing flow test `update_rewards_disable_rewards_consensus_rewards_flow_test` already validates that `disable_rewards: true` with consensus active yields zero rewards and blocks subsequent calls in the same block, confirming the mechanism is reachable and effective: [6](#0-5)
