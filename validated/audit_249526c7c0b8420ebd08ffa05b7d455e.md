Audit Report

## Title
ckBTC Minter Address Self-Send Guard Bypassed via Uppercase Bech32 Address - (File: `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

Both `retrieve_btc` and `retrieve_btc_with_approval` guard against withdrawals to the minter's own Bitcoin address using a case-sensitive string comparison against a lowercase-only string. Because `BitcoinAddress::parse` explicitly accepts uppercase bech32 input and normalises it to the same `BitcoinAddress` value, an attacker or deceived user can supply the minter's address in uppercase, bypass the guard, burn their ckBTC, and receive no BTC in return.

## Finding Description

`derive_minter_address_str` produces the minter's address by calling `display()`, which calls `encode_bech32`, which in turn calls `bech32::encode` — a function that always emits a **lowercase** string. [1](#0-0) [2](#0-1) 

The guard in both `retrieve_btc` and `retrieve_btc_with_approval` compares the caller-supplied string directly against this lowercase result:

```rust
if args.address == main_address_str {   // case-sensitive ==
    ic_cdk::trap("illegal retrieve_btc target");
}
``` [3](#0-2) [4](#0-3) 

`BitcoinAddress::parse` explicitly routes uppercase-prefixed input (`'B'`, `'T'`) to `parse_bip173_address`: [5](#0-4) 

Inside `parse_bip173_address`, the HRP comparison uses `.to_lowercase()`, so `"BC1..."` is accepted identically to `"bc1..."`: [6](#0-5) 

An in-tree test explicitly confirms that the uppercase form `"BC1QW508D6QEJXTDG4Y5R3ZARVARY0C5XW7KV8F3T4"` parses to the **same** `BitcoinAddress::P2wpkhV0` byte array as the lowercase form: [7](#0-6) 

Exploit flow:
1. Caller queries the minter's address (publicly available) — always returned in lowercase.
2. Caller converts it to uppercase and calls `retrieve_btc` with the uppercase string.
3. The string guard (`args.address == main_address_str`) does **not** fire — strings differ.
4. `BitcoinAddress::parse` succeeds and returns `P2wpkhV0(minter_pkhash)`.
5. The caller's ckBTC is burned on the ledger.
6. A Bitcoin transaction is constructed targeting the minter's own address.
7. The minter later re-absorbs that UTXO as change; the caller receives no BTC.

The TLA+ specification explicitly models this as a forbidden state: [8](#0-7) 

## Impact Explanation

A caller permanently loses their ckBTC: it is burned on the ledger while the corresponding BTC is returned to the minter's own UTXO pool. This is a concrete, irreversible user-funds loss on a production in-scope Chain Fusion / ck-token component. This matches the **High** impact class: *Significant Chain Fusion, ck-token, ledger security impact with concrete user or protocol harm*.

## Likelihood Explanation

The minter's Bitcoin address is publicly queryable. Bech32 addresses are routinely displayed in uppercase by QR-code generators, hardware wallets, and block explorers. A user copying the minter's address from such a source and calling `retrieve_btc` would silently lose funds. An adversarial or misconfigured frontend could also deliberately inject the uppercase form. The exploit requires only a standard unprivileged ingress call with no special privileges or prior conditions.

## Recommendation

Replace the string comparison with a structural comparison of the parsed `BitcoinAddress`, which is representation-independent:

```rust
let main_address = state::read_state(|s| runtime.derive_minter_address(s));
let parsed_address = BitcoinAddress::parse(&args.address, btc_network)?;
if parsed_address == main_address {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

This eliminates the case-sensitivity gap and aligns with how the rest of the minter compares addresses structurally.

## Proof of Concept

1. Query the minter's address: `dfx canister call ckbtc_minter get_btc_address '(record { owner = opt principal "minter-id"; subaccount = null })'` → returns e.g. `bc1q...xyz`.
2. Convert to uppercase: `BC1Q...XYZ`.
3. Ensure the caller holds sufficient ckBTC.
4. Call `retrieve_btc '(record { address = "BC1Q...XYZ"; amount = <amount> })'`.
5. Observe: the call succeeds (no trap), ckBTC is burned on the ledger, and a Bitcoin transaction is submitted sending BTC to the minter's own address. The caller receives no BTC.

A deterministic unit test can be written by mocking `derive_minter_address_str` to return a known lowercase bech32 address, then calling `retrieve_btc` with its uppercase equivalent and asserting that the guard does **not** trap but the resulting `RetrieveBtcRequest.address` equals the minter's own `BitcoinAddress`.
