### Title
Unauthenticated Invoke Injection via `skip_stateful_validations` Bypasses Signature Verification for Accounts with Pending Deploy — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path unconditionally skips the `__validate__` entry-point call (the only place where the account contract cryptographically verifies the transaction signature) for any invoke transaction whose nonce is `1` and whose sender account has a transaction already in the mempool. Because the check is keyed only on the sender address taken from the attacker-controlled transaction fields — not on any proof of ownership — any third party can inject an invoke transaction carrying an arbitrary/invalid signature into the victim's nonce-1 slot immediately after the victim's `deploy_account` transaction appears in the mempool.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function fires when all three conditions hold:

1. The incoming transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`.

When it fires it returns `true`, which propagates back through `run_pre_validation_checks`: [2](#0-1) 

…and into `run_validate_entry_point`, where `validate: !skip_validate` becomes `validate: false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false` the function returns `Ok(())` before ever calling `validate_tx` (the `__validate__` entry point): [4](#0-3) 

`__validate__` is the **only** place where the account contract checks the ECDSA/Stark signature. The stateless validator checks only signature *length*, not cryptographic validity: [5](#0-4) 

**Condition 3 is attacker-controllable**

`account_tx_in_pool_or_recent_block` returns `true` as soon as the victim's `deploy_account` transaction enters the mempool: [6](#0-5) 

The victim's `deploy_account` is public the moment it is broadcast. The attacker reads the `contract_address` field (deterministically computed and stored in `InternalRpcDeployAccountTransaction`) and immediately submits an invoke with `sender_address = victim`, `nonce = 1`, and a fabricated signature.

**Nonce and mempool checks do not block the attack**

`validate_nonce` accepts `tx_nonce = 1` when `account_nonce = 0` (within the allowed gap): [7](#0-6) 

`validate_by_mempool` / `validate_tx` only checks for duplicate hashes and fee-escalation rules — no signature check: [8](#0-7) 

The attacker's transaction therefore passes every gateway check and is forwarded to the mempool via `add_tx`: [9](#0-8) 

---

### Impact Explanation

An attacker can inject an invoke transaction carrying an invalid signature into the mempool for **any account whose `deploy_account` is pending**, without possessing the account's private key. This matches the allowed High impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:

- The victim's nonce-1 slot is occupied by the attacker's transaction. The victim's own legitimate invoke at nonce=1 is rejected with `DuplicateNonce` unless the victim can fee-escalate above the attacker's chosen tip.
- The batcher does call `__validate__` (`validate: true` in `new_for_sequencing`), so the attacker's transaction will revert on execution — but it still consumes block space and disrupts the victim's UX flow (deploy succeeds, first invoke fails).
- With fee escalation disabled or a sufficiently high attacker tip, the victim's first post-deploy invoke is permanently blocked at that nonce.

---

### Likelihood Explanation

- **Trigger**: Entirely unprivileged. Any external actor watching the public mempool can execute this.
- **Precondition**: The victim must have a `deploy_account` transaction in the mempool (a common, observable event for new accounts).
- **Effort**: Trivial — craft an invoke with the victim's address, nonce=1, any signature of valid length, and submit it.
- **Detection**: None at the gateway layer; the invalid signature is never checked before mempool admission.

---

### Recommendation

The `skip_stateful_validations` UX shortcut must not be the sole gate for signature verification. Two complementary fixes:

1. **Require a valid deploy-account transaction hash in the invoke itself** (as the `native_blockifier` path already does via `deploy_account_tx_hash`): [10](#0-9) 

   The gateway should require the caller to supply the hash of the pending `deploy_account` transaction and verify it exists in the mempool before skipping `__validate__`.

2. **Alternatively, perform a lightweight signature pre-check** (e.g., verify the Stark signature against the computed transaction hash) in the stateless validator before the skip path is reached, so that a cryptographically invalid signature is rejected regardless of the skip flag.

---

### Proof of Concept

```
1. Alice submits deploy_account(class_hash=C, salt=S, ...) → tx_hash_A.
   Gateway accepts it; mempool now has Alice's address in tx_pool.

2. Attacker observes tx_hash_A in the mempool, extracts Alice's contract_address.

3. Attacker submits:
     Invoke {
       sender_address: Alice,
       nonce: 1,
       signature: [0xdead, 0xbeef],   // arbitrary, cryptographically invalid
       resource_bounds: <valid>,
       calldata: <anything>,
     }

4. Gateway stateless check: signature length is valid → passes.

5. Gateway stateful check:
   - account_nonce = 0  (Alice not deployed yet)
   - validate_nonce: 0 <= 1 <= max_gap → passes
   - validate_by_mempool: no duplicate hash, no nonce conflict → passes
   - skip_stateful_validations:
       tx.nonce() == 1 ✓  &&  account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(Alice) == true ✓  (deploy_account is pooled)
       → returns true
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false → returns Ok(()) without calling __validate__

6. Attacker's invoke is forwarded to mempool and accepted at Alice's nonce-1 slot.

7. Alice tries to submit her own invoke(nonce=1) → rejected: DuplicateNonce.

8. Batcher executes: deploy_account succeeds, attacker's invoke calls __validate__
   on Alice's newly deployed contract → signature check fails → transaction reverts.
   Alice's first post-deploy invoke never executes.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
