### Title
Threshold Rounded Down in `validate_tx_l2_gas_price_within_threshold` Admits Transactions Below the Intended Minimum Gas Price — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`validate_tx_l2_gas_price_within_threshold` computes the minimum acceptable L2 gas price by multiplying `previous_block_l2_gas_price` by `min_gas_price_percentage / 100` and calling `.to_integer()`, which truncates (floors) the result. Because the threshold is rounded **down** instead of **up**, a transaction whose `max_price_per_unit` equals the floored value — but is strictly below the true fractional threshold — passes the gateway admission check when it should be rejected.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs` lines 367–371:

```rust
let gas_price_threshold_multiplier =
    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();          // ← floor division
if tx_l2_gas_price.0 < threshold { ... }
```

`Ratio::to_integer()` performs floor (truncating-toward-zero) division. The invariant the check is meant to enforce is:

```
tx_l2_gas_price  ≥  ceil( min_gas_price_percentage × previous_block_l2_gas_price / 100 )
```

Because integer gas prices cannot be fractional, the conservative (protective) direction is **ceiling**. Using floor produces a threshold that is 1 unit too low whenever `min_gas_price_percentage × previous_block_l2_gas_price` is not exactly divisible by 100.

**Concrete example** (`min_gas_price_percentage = 80`, `previous_block_l2_gas_price = 101`):

| | Value |
|---|---|
| True threshold (80 % of 101) | 80.8 → must require ≥ 81 |
| Computed threshold (`floor(80.8)`) | **80** |
| Transaction with `tx_l2_gas_price = 80` | **admitted** (should be rejected) |

The `min_gas_price_percentage` field is explicitly documented as configurable to values other than 100 (the comment reads *"E.g., 80 to require 80% of threshold"*), so the rounding error is reachable in any production deployment that sets a sub-100 percentage. [1](#0-0) [2](#0-1) 

### Impact Explanation

The gateway stateful path is the last admission gate before a transaction enters the mempool. A transaction that passes `validate_tx_l2_gas_price_within_threshold` with a gas price 1 unit below the operator-configured minimum threshold violates the admission invariant. Under the allowed impact scope this maps to:

> **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The transaction is accepted into the mempool and forwarded to the batcher, bypassing the intended economic floor that protects the network from spam at sub-threshold prices.

### Likelihood Explanation

- Requires `validate_resource_bounds = true` (the default) and `min_gas_price_percentage ≠ 100`.
- Any operator deploying with a sub-100 percentage (the documented use-case) is affected.
- An attacker only needs to observe the previous block's L2 gas price (public on-chain data) and submit a transaction with `max_price_per_unit = floor(percentage × price / 100)`.
- No privileged access is required; any unprivileged user can craft such a transaction.

### Recommendation

Replace `.to_integer()` with `.ceil().to_integer()` so the threshold is rounded **up**, matching the conservative direction required by the invariant:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .ceil()          // ← round up
    .to_integer();
```

This mirrors the pattern already used correctly elsewhere in the codebase, e.g. `convert_l1_to_l2_gas_price_round_up` and `l1_gas_to_sierra_gas_amount_round_up`, both of which explicitly call `.ceil()` before `.to_integer()` for the same reason. [3](#0-2) 

### Proof of Concept

1. Deploy the sequencer with `min_gas_price_percentage = 80` (or any value where `80 × P` is not divisible by 100 for the current `previous_block_l2_gas_price = P`).
2. Read the previous block's L2 gas price `P` from the chain (e.g., `P = 101`).
3. Compute `floor(80 × 101 / 100) = 80`.
4. Submit an `AllResources` V3 invoke transaction with `l2_gas.max_price_per_unit = 80`.
5. `validate_tx_l2_gas_price_within_threshold` computes `threshold = 80`, evaluates `80 < 80 = false`, and **admits** the transaction.
6. The true threshold is `ceil(80.8) = 81`; the transaction should have been rejected with `"Transaction L2 gas price 80 is below the required threshold 81."` but instead enters the mempool.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
