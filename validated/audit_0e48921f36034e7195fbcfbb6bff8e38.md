### Title
Missing Minimum Threshold Enforcement in FROST Presign/Sign Allows Malicious Coordinator to Corrupt Signing Output — (`src/frost/mod.rs`)

### Summary
The `presign` function and `assert_sign_inputs` function in `src/frost/mod.rs` do not enforce the minimum threshold of 2, while `assert_key_invariants` in `src/dkg.rs` explicitly does. A malicious coordinator can supply `threshold = 1` to the FROST presign/sign protocol, causing Lagrange interpolation to use incorrect coefficients that do not reconstruct the shared secret, producing an invalid and unusable signature for all honest participants.

### Finding Description

**Root cause — inconsistent guard enforcement:**

`assert_key_invariants` in `src/dkg.rs` enforces a hard lower bound:

```rust
// src/dkg.rs lines 579-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The analogous validation functions for FROST presign and sign omit this check entirely. `presign` only validates the upper bound:

```rust
// src/frost/mod.rs lines 72-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: args.threshold.into(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

`assert_sign_inputs` has the same gap — it checks only the upper bound:

```rust
// src/frost/mod.rs lines 144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [3](#0-2) 

Neither function rejects `threshold = 1`.

**Exploit path:**

1. A malicious coordinator constructs `PresignArguments` with `threshold = ReconstructionLowerBound::from(1u8)` and a valid `keygen_out` produced by a legitimate DKG (which enforced `threshold ≥ 2`).
2. `presign` passes all validation checks and the presign protocol executes normally, collecting nonces and commitments from all participants.
3. The coordinator then drives the sign protocol via `assert_sign_inputs` with `threshold = 1`. Again, no rejection occurs.
4. In FROST signing, each participant computes a signature share `z_i = nonce_i + λ_i · x_i · c`, where `λ_i` is the Lagrange coefficient for the signing set at the chosen threshold. With `threshold = 1`, the Lagrange coefficient for a single participant evaluating at 0 is 1, so `z = nonce_i + x_i · c`.
5. The combined signature satisfies `g^z = g^{nonce_i} · (g^{x_i})^c`, but the public key is `g^x` (the full reconstructed secret), not `g^{x_i}`. Verification fails unconditionally. [4](#0-3) 

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

Every honest participant expends their nonce material and signature share in a signing session that the coordinator has deliberately poisoned. The resulting signature is cryptographically invalid and cannot be used. Because nonces must never be reused in FROST (reuse leaks the signing key), the consumed nonces are permanently spent. The coordinator can repeat this attack across sessions, permanently denying honest parties the ability to produce a valid signature under valid protocol inputs.

### Likelihood Explanation

The coordinator role is explicitly modeled as potentially adversarial in this library. The coordinator controls the `threshold` parameter passed to `presign` and `assert_sign_inputs`. No privilege beyond being the session coordinator is required. The attack requires only constructing a `PresignArguments` struct with `threshold = 1`, which is a trivial, zero-cost operation.

### Recommendation

Add the same lower-bound check present in `assert_key_invariants` to both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
// In presign(), after the upper-bound check:
if args.threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: args.threshold.into(),
        min: 2,
    });
}

// In assert_sign_inputs(), after the upper-bound check:
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the guard already present in `assert_key_invariants` and ensures consistent enforcement of the minimum threshold invariant across all threshold-signature operations.

### Proof of Concept

```rust
// Assume a legitimate 2-of-3 keygen has been completed.
// A malicious coordinator constructs presign args with threshold=1:
let args = PresignArguments {
    keygen_out: legitimate_keygen_output,
    threshold: ReconstructionLowerBound::from(1u8), // passes validation, should not
};

// presign() accepts threshold=1 — no ThresholdTooSmall error is returned.
let protocol = presign(&participants, me, &args, rng).unwrap();

// The sign protocol proceeds with threshold=1.
// assert_sign_inputs also accepts threshold=1.
// The resulting signature z satisfies g^z = g^{nonce_i} * (g^{x_i})^c,
// which does not equal the expected g^{nonce} * (g^x)^c.
// Signature verification fails; nonces are permanently consumed.
```
