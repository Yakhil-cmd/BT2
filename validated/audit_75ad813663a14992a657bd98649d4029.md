### Title
Unauthorized Token Transfer via Missing Caller-Sender Authorization Check in Bank Precompile `transfer` Method - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract` precompile's `transfer` method accepts an arbitrary `sender` address from ABI-decoded input without verifying that `contract.Caller()` equals `sender`. Any EVM contract can invoke this method to drain `evm/<contract_address>`-denom tokens from any holder without their consent.

### Finding Description

The `BankContract` precompile at `x/cronos/keeper/precompiles/bank.go` exposes a `transfer` method. The denom is correctly scoped to `EVMDenom(contract.Caller())` (i.e., `evm/<calling_contract_address>`), but the `from` address used in `bankKeeper.SendCoins` is taken directly from the ABI-decoded argument `args[0]`, not from `contract.Caller()`:

There is no guard of the form `if sender != contract.Caller() { return error }`. The same missing check exists in the `burn` case, where `addr` (the address whose tokens are burned) is also taken from ABI input without verifying the caller is authorized to burn from that address. [1](#0-0) 

### Impact Explanation

An unprivileged attacker who deploys a contract at address `X` can call the bank precompile's `transfer` method from that contract, specifying any victim address as `sender` and themselves as `recipient`. The precompile will execute `bankKeeper.SendCoins(ctx, victim, attacker, coins)` for the denom `evm/X`, transferring the victim's entire `evm/X` balance without any approval or consent. This is an **unauthorized transfer of precompile-controlled assets**, matching the Critical impact tier.

The same root cause in the `burn` branch allows the contract to destroy any holder's `evm/X` tokens without authorization — an **unauthorized burn of precompile-controlled assets**. [2](#0-1) 

### Likelihood Explanation

The entry path is fully unprivileged: any EOA can deploy an EVM contract and call the bank precompile. No admin keys, governance votes, or validator access are required. The only precondition is that victims hold tokens of the `evm/<attacker_contract>` denom, which is realistic for any token issued via this precompile.

### Recommendation

Add an explicit caller-authorization check before executing the transfer and burn operations:

- In `TransferMethodName`: verify `contract.Caller() == sender`; reject otherwise.
- In `BurnMethodName`: verify `contract.Caller() == recipient` (the address being burned from); reject otherwise.

Alternatively, redesign the `transfer` method to always use `contract.Caller()` as the `from` address, consistent with how `denom` is already scoped to the caller. [3](#0-2) 

### Proof of Concept

1. Attacker deploys a contract `AttackerContract` at EVM address `0xATK`.
2. A legitimate user (`Victim`) receives `evm/0xATK` tokens (e.g., minted by `AttackerContract` as part of a token launch).
3. Attacker calls the bank precompile's `transfer(victim_evm_address, attacker_evm_address, victim_balance)` from `AttackerContract`.
4. The precompile computes `denom = "evm/0xATK"`, `from = victim`, `to = attacker`, and executes `bankKeeper.SendCoins(ctx, victim, attacker, coins)`.
5. No authorization check is performed — the victim's entire `evm/0xATK` balance is transferred to the attacker in a single transaction.
6. The same call pattern with `burn(victim_evm_address, victim_balance)` destroys the victim's tokens without consent. [4](#0-3)

				return err
