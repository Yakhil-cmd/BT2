### Title
Missing Vector Length Validation Before Index Access on Participant-Supplied Data Causes Panic in Triple Generation — (File: `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

`do_generation_many<const N: usize>` processes a `PolynomialCommitmentsMessageMany` message received from each remote participant. It accesses six parallel `Vec` fields of that message by index `[i]` for `i in 0..N` without first asserting that each vector has at least `N` elements. A malicious participant can send a well-formed but short message, triggering an index-out-of-bounds **panic** in every honest participant's process, permanently aborting triple generation and therefore signing.

### Finding Description

`generate_triple_many::<N>` is a public API entry point that calls `do_generation_many::<N>`. Inside that async function, after receiving a broadcast `PolynomialCommitmentsMessageMany` from each peer, the code iterates `for i in 0..N` and directly indexes into six `Vec` fields of the received struct:

```rust
// src/ecdsa/ot_based_ecdsa/triples/generation.rs  ~lines 331-342
for (from, their) in
    recv_from_others::<PolynomialCommitmentsMessageMany>(&chan, wait2, &participants, me)
        .await?
{
    for i in 0..N {
        let their_big_e   = &their.big_e_v[i];      // panics if len < N
        let their_big_f   = &their.big_f_v[i];      // panics if len < N
        let their_big_l   = &their.big_l_v[i];      // panics if len < N
        let their_randomizer = &their.randomizer_v[i]; // panics if len < N
        let their_phi_proof0 = &their.phi_proof0_v[i]; // panics if len < N
        let their_phi_proof1 = &their.phi_proof1_v[i]; // panics if len < N
``` [1](#0-0) 

`PolynomialCommitmentsMessageMany` is a plain `#[derive(Serialize, Deserialize)]` struct whose fields are `Vec<_>` with no length invariant enforced during deserialization: [2](#0-1) 

A malicious participant can therefore serialize and broadcast a `PolynomialCommitmentsMessageMany` whose vectors contain fewer than `N` elements. The first honest participant to reach `their.big_e_v[i]` (or any of the other five fields) for `i >= actual_len` will **panic**. In Rust, an unhandled panic in an `async` future that is not spawned as an isolated task propagates to the caller and terminates the protocol execution.

The public entry point that exposes this path is: [3](#0-2) 

No analogous length guard exists before the indexed access, unlike the explicit length checks present in `eval_interpolation` and `eval_exponent_interpolation`: [4](#0-3) 

### Impact Explanation

Triple generation is a prerequisite for OT-based ECDSA presigning and signing. A malicious participant who joins any `generate_triple_many` session can abort that session for all honest participants by sending a single short message. Because the malicious participant can repeat this attack every time honest parties attempt to generate triples, honest parties are permanently denied the ability to produce presignatures and therefore signatures.

**Impact: High — Permanent denial of signing for honest parties.**

### Likelihood Explanation

Any participant admitted to the triple-generation session can exploit this. The attacker only needs to craft a serialized `PolynomialCommitmentsMessageMany` with one or more vectors shorter than `N`. No cryptographic material, key leakage, or external assumption is required. The attack is trivially repeatable.

### Recommendation

Add an explicit length guard immediately after deserializing each peer's `PolynomialCommitmentsMessageMany`, before the `for i in 0..N` loop:

```rust
if their.big_e_v.len() != N
    || their.big_f_v.len() != N
    || their.big_l_v.len() != N
    || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N
    || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(format!(
        "PolynomialCommitmentsMessageMany from {from:?} has wrong vector length (expected {N})"
    )));
}
```

Apply the same pattern to any other received `Vec` that is subsequently indexed by `i in 0..N` (e.g., the private `(a_i_j_v, b_i_j_v)` tuples received on `wait3`).

### Proof of Concept

1. Honest parties call `generate_triple_many::<2>(participants, me, threshold, rng)`.
2. The malicious participant constructs a `PolynomialCommitmentsMessageMany` where `big_e_v` contains only 1 element instead of 2, serializes it, and broadcasts it at `wait2`.
3. Each honest participant deserializes the message successfully (no length check in `Deserialize`).
4. On the second iteration (`i = 1`), `their.big_e_v[1]` panics with an index-out-of-bounds error.
5. The panic propagates through the async future, aborting triple generation for all honest participants.
6. The malicious participant repeats this every session, permanently preventing honest parties from generating triples and thus from signing.
