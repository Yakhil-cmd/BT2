### Title
Signature-bypass via `skip_stateful_validations` lets an attacker admit an invoke transaction with an arbitrary signature into the mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX feature, designed to let a user submit a `deploy_account` + `invoke(nonce=1)` pair simultaneously, unconditionally suppresses the `__validate__` entry-point call for any invoke transaction whose nonce is 1 and whose sender address already has any transaction in the mempool. Because `__validate__` is the only place the account's signature is checked at the gateway, an attacker who observes a victim's `deploy_account` in the mempool can submit a second invoke transaction for the victim's address with a completely arbitrary signature, and the gateway will admit it without any signature verification.

---

### Finding Description

**Trigger path:**

`GenericGateway::add_tx_inner` → `extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `skip_stateful_validations` → `run_validate_entry_point(skip_validate=true)`.

`skip_stateful_validations` returns `true` when all three conditions hold:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false, … }`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by that flag:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:79-81
if !tx.execution_flags.validate {
    return Ok(());   // ← returns without calling __validate__
}
``` [3](#0-2) 

`account_tx_in_pool_or_recent_block` returns `true` for any address that has **any** transaction in the pool or was seen in a recent committed block — it does not verify that the existing transaction is specifically a `deploy_account`:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

**Concrete attack sequence:**

1. Victim submits `deploy_account` for address `V`. The mempool now contains a transaction for `V`, so `account_tx_in_pool_or_recent_block(V)` returns `true`.
2. Attacker crafts an `InvokeV3` transaction with `sender_address = V`, `nonce = 1`, and an arbitrary/wrong signature. The attacker sets `tip` and `max_l2_gas_price` to values that will win any subsequent fee-escalation race.
3. Gateway processes the attacker's transaction:
   - **Stateless validation** passes (signature length is within bounds, DA modes are L1, etc.).
   - **`validate_nonce`** passes: `0 ≤ 1 ≤ max_allowed_nonce_gap`.
   - **`validate_by_mempool`** passes: no existing transaction at `(V, nonce=1)` yet, so `validate_fee_escalation` returns `Ok(None)`.
   - **`skip_stateful_validations`** returns `true`: nonce=1, account_nonce=0, and `account_tx_in_pool_or_recent_block(V)` is `true`.
   - **`run_validate_entry_point`** is called with `validate=false` → `__validate__` is **never invoked** → the wrong signature is never checked.
   - **`mempool.add_tx`** admits the attacker's invalid transaction. [5](#0-4) 

4. Victim later submits their valid `invoke(nonce=1)`. The mempool now has an existing transaction at `(V, nonce=1)` (the attacker's). `validate_fee_escalation` requires the victim's fee to exceed the attacker's by `fee_escalation_percentage` (default 10 %). If the victim's fee is lower, the gateway returns `DuplicateNonce` and the victim's valid transaction is **rejected at the gateway**. [6](#0-5) 

The attacker's invalid transaction remains in the mempool. When the batcher eventually executes it, `__validate__` will fail and the transaction will be reverted — but the victim's valid transaction was never admitted.

---

### Impact Explanation

**Broken invariant:** The gateway must not admit a transaction whose signature has not been verified. Here, an invoke transaction with a completely arbitrary signature is admitted to the mempool because the `skip_stateful_validations` path suppresses the only signature-checking step (`__validate__`) without any alternative verification.

**Concrete wrong value:** The mempool contains an `InternalRpcTransaction` for address `V` at nonce 1 whose `signature` field was never validated. This is an invalid transaction accepted through account validation logic — matching the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Secondary effect: the victim's valid invoke is rejected by the gateway with `DuplicateNonce` if the attacker's fee is higher, constituting *"rejects valid transactions before sequencing"* as well.

---

### Likelihood Explanation

- **No privileged access required.** The attacker only needs to observe the public mempool to learn that a `deploy_account` for address `V` is pending.
- **No private key required.** Because `__validate__` is skipped, the attacker does not need to know `V`'s private key.
- **Cheap to execute.** The attacker submits one transaction with a slightly higher fee than the victim's expected invoke. The default `fee_escalation_percentage` is 10 %, so a 10 % fee premium is sufficient.
- **Timing window is wide.** The window is open from the moment the victim's `deploy_account` enters the mempool until the block containing it is committed and the account nonce advances to 1.

---

### Recommendation

The `skip_stateful_validations` guard must be tightened so that it only fires when the **specific** `deploy_account` transaction for address `V` is in the mempool, not merely any transaction. Concretely:

1. Expose a `deploy_account_tx_in_pool(address)` query on the mempool that returns `true` only when the pending transaction for `(address, nonce=0)` is a `DeployAccount` type.
2. Replace the `account_tx_in_pool_or_recent_block` call in `skip_stateful_validations` with this stricter check.

Alternatively, mirror the approach used in `native_blockifier/src/py_validator.rs`, where the caller passes the explicit `deploy_account_tx_hash` and the validator checks `deploy_account_tx_hash.is_some()` — binding the skip to a specific, caller-supplied hash rather than a generic address-presence check. [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {

            return Ok(None);
        }

        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
