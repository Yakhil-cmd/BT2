### Title
Pool Admin Bypasses Fee Cap via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on the pool admin's base spread and notional fees. However, the parallel admin path `setPoolBinAdditionalFees` applies no cap whatsoever on `addFeeBuyE6` / `addFeeSellE6`, allowing the pool admin to set per-bin additional fees that, when added to the base spread, push the effective swap fee above the factory-enforced hard cap.

---

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` guards both admin fee components against the factory caps before writing them:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`MetricOmmPoolFactory.setPoolBinAdditionalFees`, the other admin fee-setting path, performs no analogous check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

Per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) are stored as `uint16` (max 65 535, i.e. 6.5535 % in E6 units) and are applied **on top of** the base `spreadFeeE6` during swap execution. The factory hard cap `HARD_MAX_SPREAD_FEE_E6 = 200 000` (20 %) is enforced only on the base spread. A pool admin can therefore set `addFeeBuyE6 = 65535` on every bin, making the effective per-bin fee up to 26.5535 % — 6.5535 percentage points above the hard cap — without any revert.

The asymmetry is exact: `setPoolAdminFees` enforces caps; `setPoolBinAdditionalFees` does not. This mirrors the external report's pattern where one code path enforces a limit and the parallel path silently omits it.

---

### Impact Explanation

Traders swapping through bins with inflated additional fees pay more than the protocol's hard-capped maximum. The excess fee accrues as spread surplus and is split between protocol and admin at collection time, meaning the pool admin extracts value from traders beyond the boundary the factory owner intended to enforce. This is a direct loss of user principal (overpayment of swap fees) and an admin-boundary break: the pool admin exceeds the fee cap the factory owner set.

---

### Likelihood Explanation

The pool admin is semi-trusted only within caps. Any pool admin — including one who deployed a pool permissionlessly via `createPool` — can call `setPoolBinAdditionalFees` at any time with `addFeeBuyE6 = type(uint16).max` on every bin. No timelock, no escalation, and no factory-owner approval is required. The trigger is a single transaction by the pool admin.

---

### Recommendation

Add a cap check in `MetricOmmPoolFactory.setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    // addFeeBuyE6 and addFeeSellE6 are additive to spreadFeeE6; cap them so the
    // total effective fee cannot exceed the admin spread cap.
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap at the pool level inside `setBinAdditionalFees` by reading the stored `spreadFeeE6` and rejecting values that would push the total above the hard limit.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20 %) and `maxProtocolSpreadFeeE6 = 0`.
2. Pool admin creates a pool with `adminSpreadFeeE6 = 200_000` (at the cap). `setPoolAdminFees` enforces this correctly.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`. No revert occurs.
4. A trader swaps through bin 0. The effective spread fee is `200_000 + 65_535 = 265_535` (26.5535 %), exceeding the 20 % hard cap by 6.5535 percentage points.
