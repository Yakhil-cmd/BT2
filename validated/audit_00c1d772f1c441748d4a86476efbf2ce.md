### Title
WalletConnect `chia_sendTransaction` Accepts Unbounded `puzzle_decorator` Clawback Timelock Without Humanized Display, Causing Recipient Fund Lockout — (File: `packages/gui/src/electron/commands/Commands.ts`)

### Summary
A malicious dapp connected via WalletConnect can call `chia_sendTransaction` with a `puzzle_decorator` parameter containing an arbitrarily large `clawbackTimelock` value (e.g., `9999999999` seconds ≈ 317 years). The WalletConnect confirmation dialog renders `puzzle_decorator` as raw JSON with no humanization or upper-bound validation of the timelock. A user who approves the transaction sends XCH locked under a centuries-long clawback window, permanently denying the recipient access to the funds for a practically infinite period.

### Finding Description

The `chia_wallet.send_transaction` command schema in `Commands.ts` declares `puzzle_decorator` as an optional `json`-typed parameter:

```typescript
{
  name: 'puzzle_decorator',
  label: () => i18n._(/* i18n */ { id: 'Puzzle Decorator' }),
  type: 'json',
  isOptional: true,
},
``` [1](#0-0) 

This parameter is forwarded verbatim to the wallet RPC `send_transaction` call, which accepts `puzzleDecorator?: PuzzleDecorator[]`: [2](#0-1) 

The `PuzzleDecorator` type places no upper bound on `clawbackTimelock`: [3](#0-2) 

In the native GUI send flow (`WalletSend.tsx`), the clawback timelock is assembled from bounded day/hour/minute fields (`max: 365`, `max: 24`, `max: 60`), but these are HTML `max` hints only — `getClawbackTimeInSeconds` performs no server-side or JavaScript upper-bound check: [4](#0-3) 

The WalletConnect path bypasses even these UI hints entirely. A dapp can supply `puzzle_decorator: [{"decorator":"CLAWBACK","clawbackTimelock":9999999999}]` directly. The confirmation dialog shows this as a raw JSON blob labeled "Puzzle Decorator" — no seconds-to-human conversion, no warning, no cap.

### Impact Explanation

When the user approves the WalletConnect transaction:

1. XCH is sent on-chain with a `clawbackTimelock` of `9999999999` seconds (≈317 years).
2. The recipient's wallet shows the funds as `INCOMING_CLAWBACK_RECEIVE` with a countdown of centuries — they cannot claim the coins.
3. The sender (user) retains the ability to claw back, but only if they discover the manipulation. If they do not, the funds sit in limbo indefinitely.

The `WalletHistoryClawbackChip` component computes `timeLeft` from `transactionRow.metadata.timeLock` with no cap, so the recipient's UI will display an astronomically large "Can be claimed in…" value: [5](#0-4) 

This matches the **High** impact class: *"Corruption, spoofing, or unsafe trust of WalletConnect state that causes a user to approve… the wrong… status"* — specifically, the wrong custody/clawback restriction with direct asset-access impact on the recipient.

### Likelihood Explanation

Any dapp granted `chia_sendTransaction` permission via WalletConnect can trigger this. The `parseDappParams` allowlist confirms `puzzle_decorator` is a permitted parameter for `chia_sendTransaction`: [6](#0-5) 

The user sees `Puzzle Decorator: [{"decorator":"CLAWBACK","clawbackTimelock":9999999999}]` — a raw JSON blob that provides no human-readable indication of a 317-year lock. Most users will not recognize the significance of the numeric value and will approve.

### Recommendation

1. **Validate `clawbackTimelock` in `puzzle_decorator`** at the WalletConnect command-processing layer: reject or cap values exceeding a reasonable maximum (e.g., 365 days = 31,536,000 seconds), consistent with the UI field limits in `WalletSend.tsx` and `SettingsCustodyClawbackOutgoing.tsx`.
2. **Humanize the `puzzle_decorator` display** in the WalletConnect confirmation dialog: parse the `clawbackTimelock` field and render it as days/hours/minutes so the user can make an informed decision.
3. Consider adding a dedicated `humanize` handler for `puzzle_decorator` in the `Commands.ts` param schema, analogous to the `mojo-to-xch` humanizer used for `amount` and `fee`.

### Proof of Concept

A malicious dapp with `chia_sendTransaction` permission sends the following WalletConnect request:

```json
{
  "method": "chia_sendTransaction",
  "params": {
    "amount": "1000000000",
    "fee": "0",
    "address": "xch1victim...",
    "puzzle_decorator": [{"decorator": "CLAWBACK", "clawbackTimelock": 9999999999}]
  }
}
```

The GUI confirmation dialog renders:
- **Amount**: 0.001 XCH
- **Fee**: 0 XCH
- **Address**: xch1victim…
- **Puzzle Decorator**: `[{"decorator":"CLAWBACK","clawbackTimelock":9999999999}]`

The user approves. The recipient's wallet shows the transaction as claimable in ~317 years. The `getClawbackTimeInSeconds` path in the native UI would have capped this at 365 days via HTML `max`, but the WalletConnect path has no equivalent guard. [7](#0-6)
