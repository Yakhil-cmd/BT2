### Title
Unchecked `app_pk` in CKD Protocol Allows Malicious Coordinator to Extract Confidential Derived Key — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function accepts an `app_pk` (ElGamal encryption key) parameter with no validation. A malicious coordinator who orchestrates session setup can supply `app_pk = G1Projective::identity()` to all participants. When the identity element is used as the encryption key, the blinding term vanishes and the coordinator's output `CKDOutput.big_c` directly equals `msk · H(pk ∥ app_id)` — the confidential derived key — in plaintext.

---

### Finding Description

The CKD protocol encrypts the derived key under `app_pk` using an ElGamal-style construction inside `compute_signature_share()`: [1](#0-0) 

```rust
let big_y = ElementG1::generator() * y.0;
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
let big_s = hash_point * private_share.to_scalar();
// C <- S + y . A
let big_c = big_s + app_pk * y.0;
```

If `app_pk` is the group identity element (`G1Projective::identity()`), then `app_pk * y.0 = identity`, so `big_c = big_s` — the raw secret share contribution with no blinding. After the coordinator sums all Lagrange-weighted shares:

```
C = Σ λᵢ · Cᵢ = Σ λᵢ · (xᵢ · H(pk ∥ app_id)) = msk · H(pk ∥ app_id)
```

The confidential derived key is directly exposed in `CKDOutput.big_c`.

Neither `ckd()` nor `compute_signature_share()` validates that `app_pk` is a non-identity point or that it corresponds to a legitimate requester: [2](#0-1) 

The `PublicKey` type is `blstrs::G1Projective`, which can represent the identity element. The identity check that exists in `BLS12381G1Group::deserialize()` is only triggered during deserialization, not when a `G1Projective` value is passed directly: [3](#0-2) 

In a real deployment the coordinator orchestrates session setup and distributes session parameters — including `app_pk` — to all participants before they call `ckd()`. Participants have no in-protocol mechanism to independently verify that `app_pk` is the legitimate requester's key. This mirrors the external report exactly: the coordinator is the privileged role, and `app_pk` is the unchecked receiver parameter.

---

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

When `app_pk = identity`, `CKDOutput.big_c` equals `msk · H(pk ∥ app_id)` in plaintext. Any party that receives the coordinator's output (including the coordinator itself) can read the confidential derived key directly without knowing any secret scalar. The `unmask()` method confirms this: `C − a·Y` with `a = 0` and `Y = anything` still yields `C = msk · H(pk ∥ app_id)`: [4](#0-3) 

---

### Likelihood Explanation

The coordinator is a trusted-but-verify role that initiates CKD sessions and distributes session parameters. In any realistic deployment the coordinator tells participants which `app_id` and `app_pk` to use. A malicious coordinator can trivially pass `G1Projective::identity()` as `app_pk` to all participants. Because the library performs no validation, all participants will silently compute shares under the identity key, and the coordinator will receive the derived key in plaintext.

---

### Recommendation

Add an explicit non-identity check for `app_pk` at the start of `ckd()` (or inside `compute_signature_share()`):

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Additionally, consider requiring a proof of knowledge (proof of discrete log) for `app_pk` so that participants can verify the requester actually controls the corresponding secret key before contributing their shares.

---

### Proof of Concept

```rust
// Attacker (malicious coordinator) sets app_pk = identity
let attacker_app_pk = G1Projective::identity(); // no secret key needed

// All participants are told to use this app_pk (coordinator controls session setup)
// Each participant computes:
//   big_c = big_s + identity * y = big_s   (blinding vanishes)

// Coordinator sums Lagrange-weighted shares:
//   C = Σ λᵢ · Cᵢ = msk · H(pk ∥ app_id)

// Attacker reads CKDOutput.big_c directly — no secret key required
let stolen_derived_key = ckd_output.big_c(); // = msk · H(pk ∥ app_id)

// Verify: unmask with a=0 gives the same value
assert_eq!(ckd_output.unmask(Scalar::ZERO), stolen_derived_key);
```

The `compute_signature_share` function at line 174 performs `big_s + app_pk * y.0` with no guard, making this attack unconditionally reachable by any caller who controls the `app_pk` argument passed to `ckd()`. [5](#0-4)

        me,
        participants,
        key_pair,

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
