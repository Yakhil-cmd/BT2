### Title
Missing Sequencer Access Control on `update_rewards` Enables Permanent Yield Freeze — (`src/staking/staking.cairo`)

---

### Summary

The spec mandates "Only starkware sequencer" for `update_rewards`, but the implementation enforces no such check. Any unprivileged caller can invoke `update_rewards` with `disable_rewards: true` once per block, consuming the global `last_reward_block` slot and permanently preventing the legitimate sequencer from distributing rewards for that block. Repeated across every block, this freezes all staker and pool yield indefinitely.

---

### Finding Description

The spec entry for `update_rewards` states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The actual implementation delegates all access control to `general_prerequisites()`: [2](#0-1) 

`general_prerequisites` only checks:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [3](#0-2) 

There is no sequencer role check. The per-block guard is a **global** `last_reward_block` storage variable:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [4](#0-3) 

When `disable_rewards` is `true`, the function writes `last_reward_block = current_block_number` and returns immediately without distributing any rewards:

```cairo
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [5](#0-4) 

Because `last_reward_block` is global (not per-staker), a single attacker call per block with `disable_rewards: true` exhausts the slot for every staker in that block. The sequencer's subsequent call reverts with `REWARDS_ALREADY_UPDATED`. The missed block rewards are never recoverable.

---

### Impact Explanation

- **High — Permanent freezing of unclaimed yield**: An attacker calling `update_rewards(any_valid_staker, disable_rewards: true)` once per block permanently prevents all stakers and all BTC/STRK delegation pools from accruing rewards for every blocked block. There is no catch-up mechanism; missed block rewards are gone.

---

### Likelihood Explanation

- The function is publicly callable (no role gate in code).
- The attacker only needs to submit one transaction per block, which is cheap on Starknet.
- The attacker needs no stake, no privileged key, and no special setup — only a valid `staker_address` that passes the `is_staker_active` check.
- The attack is sustainable indefinitely.

---

### Note on the Normalization / BTC-18-Decimal Angle

The specific precision-drift hypothesis in the question (repeated tiny top-ups causing `from_native_amount`/`to_native_amount` drift for 18-decimal BTC wrappers) does **not** produce a real vulnerability. For a token with `decimals = 18`, the normalization multiplier is `10^(18-18) = 1`, so there is zero rounding loss in either direction:

```cairo
fn from_native_amount(amount: Amount, decimals: u8) -> NormalizedAmount {
    NormalizedAmount { amount_18_decimals: amount * 10_u128.pow(18 - decimals.into()) }
}
``` [6](#0-5) 

The reward calculation uses `to_amount_18_decimals()` consistently on both numerator and denominator, so no drift accumulates regardless of call ordering or top-up size.

---

### Recommendation

Add a sequencer/role check at the top of `update_rewards`, consistent with the spec. For example, gate the function with the existing roles infrastructure (e.g., `self.roles.only_sequencer()` or an equivalent `assert_caller_is_sequencer()`) before any other logic.

---

### Proof of Concept

```
Block N:
  Attacker → staking.update_rewards(valid_staker, disable_rewards=true)
    → general_prerequisites() passes (not paused, caller != 0)
    → last_reward_block := N
    → returns (no rewards distributed)

  Sequencer → staking.update_rewards(staker, disable_rewards=false)
    → assert!(N > N)  ← FAILS: REWARDS_ALREADY_UPDATED
    → reverts

Block N+1: repeat indefinitely
```

All stakers' `unclaimed_rewards_own` and all pool balances remain frozen at zero for every attacked block.
