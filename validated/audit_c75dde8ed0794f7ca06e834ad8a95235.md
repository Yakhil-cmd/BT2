### Title
Reliable Broadcast Agreement Failure for n=3: Malicious Participant Can Equivocate Commitments, Causing Split-View on Aggregate Public Key — (`src/protocol/echo_broadcast.rs`)

### Summary

`echo_ready_thresholds(3)` returns `(0, 0)`, reducing all threshold checks to `count > 0`. Combined with the simulated-vote mechanism, each honest party delivers a broadcast value based solely on its own echo/ready — without corroboration from any other party. A malicious participant can send different commitments to different honest parties in the DKG SEND phase, and both honest parties will independently deliver their respective (different) values. This breaks the **Agreement** property of reliable broadcast and causes honest parties to compute different aggregate public keys.

---

### Finding Description

**Root cause — `echo_ready_thresholds`:** [1](#0-0) 

When `n <= 3`, both `echo_t` and `ready_t` are `0`. All three threshold comparisons in `reliable_broadcast_receive_all` then reduce to `count > 0`:

- **Echo → Ready trigger** (line 225): `count > echo_t` → `count > 0`
- **Ready amplification** (line 282): `count > ready_t` → `count > 0`
- **Ready delivery** (line 295): `count > 2 * ready_t` → `count > 0` [2](#0-1) [3](#0-2) 

**Root cause — simulated vote mechanism:**

When an honest party receives a SEND message, it immediately echoes and then simulates its own ECHO as a local vote (lines 207–209). That simulated ECHO has count=1, which is `> 0`, so it immediately triggers a READY send and another simulated READY. That simulated READY also has count=1 `> 0`, triggering amplification and then delivery — all before any real network message from another party arrives. [4](#0-3) [5](#0-4) 

**Exploit trace (n=3, attacker A, honest B and C):**

1. A sends `SEND(commitment_X)` to B and `SEND(commitment_Y)` to C (different values, possible via `send_private` or by manipulating the network layer).
2. B processes `SEND(commitment_X)` → echoes → simulates own ECHO (count=1 > 0) → sends `READY(commitment_X)` → simulates own READY (count=1 > 0) → **delivers commitment_X**. `finish_ready=true` is set before any message from C arrives.
3. C processes `SEND(commitment_Y)` → echoes → simulates own ECHO (count=1 > 0) → sends `READY(commitment_Y)` → simulates own READY (count=1 > 0) → **delivers commitment_Y**. `finish_ready=true` is set before any message from B arrives.
4. When B later receives `READY(commitment_Y)` from C, `finish_ready=true` causes it to be skipped (line 268). B's delivered value remains `commitment_X`. C's delivered value remains `commitment_Y`.

**DKG impact path:**

`do_broadcast` is called in `do_keyshare` at line 435 to broadcast each participant's `(commitment, proof_of_knowledge)`: [6](#0-5) 

A can send `(commitment_X, proof_X)` to B and `(commitment_Y, proof_Y)` to C. Since A controls its own secret, it can generate valid Schnorr proofs for both commitments. A also sends matching commitment hashes in Round 2 (via `send_many`, which has no consistency guarantee), so `verify_commitment_hash` passes for both B and C independently. [7](#0-6) 

At step 4.5, each honest party computes the aggregate public key by summing all commitments: [8](#0-7) 

B computes `pk_B = C_A_X(0) + C_B(0) + C_C(0)` and C computes `pk_C = C_A_Y(0) + C_B(0) + C_C(0)`. These are different group elements, so B and C hold different aggregate public keys after DKG completes.

---

### Impact Explanation

Honest parties B and C complete DKG successfully (no error is returned) but hold **different verifying keys**. Any threshold signature produced by B's view is unverifiable under C's key and vice versa. The DKG output is permanently corrupted. This matches the allowed High impact: *"Corruption of DKG outputs so honest parties accept inconsistent public keys."*

---

### Likelihood Explanation

- n=3 is the minimum supported DKG configuration and is explicitly handled by the `n <= 3` branch.
- The attacker is a standard protocol participant — no elevated privilege required.
- The attack requires only sending different SEND-phase messages to different recipients, which is trivially achievable by any participant who controls their own message delivery.
- No cryptographic assumption needs to be broken.

---

### Recommendation

The `n <= 3` special case in `echo_ready_thresholds` is incorrect. For n=3, the BFT model with `MaxFaulty = floor((3-1)/3) = 0` means **zero** Byzantine faults are tolerated, so the protocol should require all n=3 parties to agree — not deliver on a single self-echo. The correct fix is to require `count >= n` (all parties) for delivery when `MaxFaulty = 0`, or to enforce a minimum participant count of n=4 for DKG (which gives `MaxFaulty >= 1`). The `(0, 0)` threshold effectively removes all cross-party corroboration and must not be used when a Byzantine participant is possible.

```rust
// B: verifying_key = commitment_X.at_zero() + C_B.at_zero() + C_C.at_zero()
```

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```
