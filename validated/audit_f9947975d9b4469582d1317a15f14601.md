### Title
Synthetic Ratio Truncates to Zero for Low-Price Base Assets, Permanently Bricking Pool Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider._getBidAndAskPrice` computes the synthetic cross-rate as `mid1 * ORACLE_DECIMALS / mid2` (integer division). When the base asset's USD price is much lower than the quote asset's USD price — specifically when `mid1 * 1e8 < mid2` — the division truncates to zero. The zero `mid` propagates into `_computeBidAsk`, which returns the `(0, type(uint128).max)` sentinel, causing every call to `getBidAndAskPrice` to revert with `FeedStalled`. All swaps on the pool are permanently blocked.

---

### Finding Description

In `_getBidAndAskPrice`, when `quoteFeedId` is set (synthetic mode), the cross-rate is computed as:

```solidity
// AnchoredPriceProvider.sol line 267
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
```

`ORACLE_DECIMALS = 1e8`. Both `mid` (base feed) and `mid2` (quote feed) are 8-decimal oracle prices (i.e., `price_USD * 1e8`). The formula is therefore:

```
synthetic_mid = price_base_USD * 1e8 * 1e8 / (price_quote_USD * 1e8)
              = price_base_USD * 1e8 / price_quote_USD
```

Integer division truncates to zero whenever `price_base_USD * 1e8 < price_quote_USD`, i.e., whenever the base asset is more than 8 decimal orders of magnitude cheaper than the quote asset.

Concrete example — SHIB/ETH pool:
- SHIB/USD: `mid1 = 1_000` (= 0.00001 × 1e8)
- ETH/USD: `mid2 = 300_000_000_000` (= 3000 × 1e8)
- `Math.mulDiv(1_000, 1e8, 300_000_000_000)` = `1e11 / 3e11` = **0**

`_computeBidAsk(0, spreadBps)` then computes `refBid = mulDiv(0, …) = 0`, hits the `refBid == 0` guard, and returns `(0, type(uint128).max)`. `getBidAndAskPrice` then reverts with `FeedStalled`. [1](#0-0) 

The same pattern appears in the `FaithfulAnchor` mock used in core E2E tests: [2](#0-1) 

---

### Impact Explanation

Every `swap` call on the affected pool invokes `getBidAndAskPrice`, which permanently reverts with `FeedStalled`. The pool is completely unusable for its primary function. LPs cannot trade; they can only withdraw, but the pool's economic purpose is destroyed. This matches the allowed impact gate: **broken core pool functionality causing unusable swap flows**. [3](#0-2) 

---

### Likelihood Explanation

The `AnchoredProviderFactory.createAnchoredProvider` accepts any `baseFeedId` / `quoteFeedId` pair without validating that the resulting ratio will be non-zero at current prices. [4](#0-3) 

Any pool pairing a micro-cap token (SHIB, PEPE, many L2 tokens priced below ~0.00003 USD) against ETH or BTC as the quote leg will silently produce `mid = 0` from day one. The factory does not prevent this configuration, and the condition is not transient — it persists as long as the price relationship holds.

---

### Recommendation

Scale the numerator before dividing to preserve precision. The result should remain 8-decimal so downstream `_bandEdge` / `_computeBidAsk` math is unchanged:

```solidity
// Before (truncates when mid * 1e8 < mid2):
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);

// After: scale to 16-decimal intermediate, then renormalize to 8-decimal
// mid_16dec = mid * 1e16 / mid2  (preserves 8 more decimal places)
// then divide by 1e8 to return to 8-decimal domain
mid = Math.mulDiv(mid, ORACLE_DECIMALS * ORACLE_DECIMALS, mid2) / ORACLE_DECIMALS;
// equivalently: Math.mulDiv(mid, ORACLE_DECIMALS, mid2) with a pre-scaled numerator
```

Or, more robustly, add a post-division zero check and revert with `FeedStalled` immediately rather than propagating a silent zero into `_computeBidAsk`:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max); // explicit guard
```

The second option at minimum makes the failure mode explicit; the first option actually fixes the precision loss for valid low-price pairs.
