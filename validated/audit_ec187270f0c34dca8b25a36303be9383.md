### Title
Blind Trust of Pool-Supplied `target_puzzle_hash` Redirects All Farming Rewards to Attacker — (`packages/gui/src/util/getPoolInfo.ts`, `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`, `packages/gui/src/components/plotNFT/PlotNFTChangePool.tsx`)

### Summary

The GUI fetches pool info from an attacker-controlled HTTP endpoint, applies `toCamelCase` to the raw JSON, and passes the resulting `targetPuzzleHash` directly to `pwJoinPool` with only a JavaScript truthiness check. No format validation, no length check, and no cross-reference against the pool's on-chain singleton is performed. A malicious or compromised pool server can return an arbitrary 32-byte hex string as `target_puzzle_hash`, causing the user's PlotNFT singleton to be updated on-chain to point to the attacker's address, permanently redirecting all future farming block rewards.

### Finding Description

**Step 1 — Electron main fetches raw pool JSON with no validation:**

`FETCH_POOL_INFO` in `main.tsx` fetches `${poolUrl}/pool_info` and returns the raw parsed JSON object to the renderer process. [1](#0-0) 

**Step 2 — `getPoolInfo` applies `toCamelCase` and casts blindly:**

`getPoolInfo` calls `toCamelCase(data)` on the untrusted pool server response and casts it directly to `PoolInfo`. No field validation occurs. [2](#0-1) 

`toCamelCase` mechanically renames `target_puzzle_hash` → `targetPuzzleHash` and passes the value through unchanged. [3](#0-2) 

**Step 3 — `prepareSubmitData` checks only truthiness:**

The only guard on `targetPuzzleHash` is `if (!targetPuzzleHash)`. Any non-empty string — including an attacker-controlled 64-char hex address — passes this check and is stored directly into `initialTargetState.targetPuzzleHash`. [4](#0-3) 

**Step 4 — `pwJoinPool` is called with the attacker's puzzle hash:**

`PlotNFTChangePool.handleSubmit` passes `targetPuzzleHash` verbatim as `targetPuzzlehash` to `pwJoinPool`, which sends it to the wallet daemon via the `pw_join_pool` RPC. [5](#0-4) 

The wallet daemon's `pwJoinPool` takes `targetPuzzlehash` as a caller-supplied parameter and uses it to construct the on-chain singleton spend — it does not independently fetch or verify the pool's puzzle hash. [6](#0-5) 

### Impact Explanation

Once `pw_join_pool` is submitted with the attacker's `targetPuzzlehash`, the user's PlotNFT singleton is updated on-chain to point to the attacker's puzzle hash. All block rewards won by the user's plots are sent to the attacker's address. The user's `payout_instructions` are irrelevant — the block reward goes to the singleton's `target_puzzle_hash` first. The user must notice the change and submit another pool-change transaction (paying fees again) to recover. This is a direct, permanent pooled farming reward loss.

### Likelihood Explanation

The attack does not require local access or key compromise. It requires only that the user enter a pool URL controlled or compromised by the attacker. This is a realistic scenario: a legitimate pool's `/pool_info` endpoint could be MITM'd (HTTP on testnet is explicitly allowed by the URL validator), the pool server could be compromised, or the attacker could advertise a pool with a plausible name. The GUI displays pool details (`PoolInfo.tsx`) including the `targetPuzzleHash` value, but users have no way to know what a legitimate puzzle hash should look like, so the display provides no effective warning. [7](#0-6) 

### Recommendation

1. **Format validation**: Before using `targetPuzzleHash`, validate it is a 64-character lowercase hex string (optionally `0x`-prefixed), rejecting anything else.
2. **On-chain cross-reference**: The wallet daemon or GUI should fetch the pool's registered singleton from the blockchain and compare its `target_puzzle_hash` against the HTTP API response before submitting `pw_join_pool`.
3. **User confirmation**: Display the full `targetPuzzleHash` prominently in the confirmation dialog and warn users that this is the address that will receive their block rewards.

### Proof of Concept

1. Run a local HTTP server at `http://localhost:8080` serving `/pool_info`:
   ```json
   {
     "name": "Legit Pool",
     "description": "Totally legit",
     "pool_url": "http://localhost:8080",
     "fee": "0.01",
     "logo_url": "",
     "minimum_difficulty": 1,
     "protocol_version": "1.0",
     "relative_lock_height": 32,
     "target_puzzle_hash": "0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"
   }
   ```
2. In the GUI, navigate to Pool → Change Pool (or Add Plot NFT), enter `http://localhost:8080` as the pool URL.
3. Submit the form. Observe that `prepareSubmitData` calls `getPoolInfo`, receives `targetPuzzleHash = "0xdeadbeef..."`, passes the truthiness check, and calls `pwJoinPool` with the attacker's hash.
4. Inspect the resulting `pw_join_pool` RPC message: `targetPuzzlehash` will be `"0xdeadbeef..."`.
5. On-chain, the PlotNFT singleton's `target_puzzle_hash` is now the attacker's address.

export default function toCamelCase(object: Object): { [key: string]: unknown } {
  return transform(object, (acc, value, key, target) => {
    const newKey = isArray(target) || key.indexOf('_') === -1 ? key : camelCase(key);

    acc[newKey] = isObject(value) ? toCamelCase(value) : value;
  });
