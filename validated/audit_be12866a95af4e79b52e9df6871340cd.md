### Title
Unvalidated Pool API Response Allows Malicious Pool to Redirect Farming Rewards - (File: `packages/gui/src/util/getPoolInfo.ts`)

### Summary

`getPoolInfo` fetches the `/pool_info` endpoint of an external pool server and casts the raw JSON response directly to `PoolInfo` using a TypeScript type assertion (`toCamelCase(data) as PoolInfo`) with no runtime field-level validation. The `targetPuzzleHash` field — which determines the on-chain address where pooled farming rewards are sent — is only checked for truthiness before being forwarded to the `pwJoinPool` RPC call. A malicious pool operator can return a crafted `targetPuzzleHash` (their own address) that passes every GUI-side guard, causing the farmer's XCH rewards to be permanently redirected to the attacker for the duration of the lock height.

### Finding Description

The pool-join flow in `PlotNFTSelectPool.tsx` calls `getPoolInfo(normalizedPoolUrl)` to retrieve pool parameters from an attacker-controlled HTTPS endpoint:

```
packages/gui/src/util/getPoolInfo.ts
  const data = await window.appAPI.fetchPoolInfo(poolUrl);
  return toCamelCase(data) as PoolInfo;          // ← compile-time cast only
``` [1](#0-0) 

The IPC handler in the main process calls `fetchJSON`, which validates the URL, enforces a timeout, and enforces a size limit, but returns the raw parsed JSON object with no schema or field validation:

```
packages/gui/src/electron/main.tsx
  ipcMainHandle(AppAPI.FETCH_POOL_INFO, async (poolUrl: string) => {
    const poolInfoUrl = `${poolUrl}/pool_info`;
    return fetchJSON(poolInfoUrl);               // ← no response schema check
  });
``` [2](#0-1) 

`fetchJSON` itself only parses JSON and enforces transport-level limits; it performs no field-level validation on the returned object: [3](#0-2) 

Back in `PlotNFTSelectPool.tsx`, the only guards applied to the response are:

```typescript
const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
if (!targetPuzzleHash) {
  throw new Error(t`Pool does not provide targetPuzzleHash.`);
}
if (relativeLockHeight === undefined) {
  throw new Error(t`Pool does not provide relativeLockHeight.`);
}
initialTargetState.targetPuzzleHash = targetPuzzleHash;
initialTargetState.relativeLockHeight = relativeLockHeight;
``` [4](#0-3) 

Neither field is validated for format or range:

- `targetPuzzleHash` is not checked to be a valid 32-byte hex string (64 lowercase hex characters). Any non-empty string passes.
- `relativeLockHeight` is not checked to be a safe positive integer within a reasonable range. `0`, negative values, or `Number.MAX_SAFE_INTEGER` all pass.

The `PoolInfo` type definition confirms these fields are expected to be a hex string and a number respectively, but the type is only enforced at compile time: [5](#0-4) 

### Impact Explanation

`targetPuzzleHash` is the on-chain puzzle hash that the farmer's Plot NFT singleton is configured to pay rewards to when a block is won. If a malicious pool returns an attacker-controlled puzzle hash that is a valid non-empty string, it passes the `!targetPuzzleHash` guard and is forwarded verbatim to `pwJoinPool`. The farmer's XCH pooling rewards are then sent to the attacker's address for the entire duration of `relativeLockHeight` blocks. This is a direct, irreversible payout change affecting pooled farming rewards — Critical impact per the allowed scope.

Additionally, a malicious pool can return `relativeLockHeight: 0` (passes the `=== undefined` check) or an arbitrarily large integer to manipulate the lock-in period without the user being warned.

### Likelihood Explanation

The attacker only needs to operate an HTTPS server that returns a crafted JSON response at `/pool_info`. No access to the user's machine, keys, or wallet is required. The user must enter the attacker's pool URL, which is a realistic scenario (phishing links, typosquatting of popular pool domains, or a compromised legitimate pool). The `targetPuzzleHash` is displayed in `PoolInfo.tsx` before confirmation, but users have no practical way to verify whether a 64-character hex string belongs to the pool or to an attacker. [6](#0-5) 

### Recommendation

- Add runtime validation of `targetPuzzleHash` as a valid 64-character lowercase hex string (matching the 32-byte puzzle hash format used throughout the Chia protocol) before it is accepted.
- Add runtime validation of `relativeLockHeight` as a safe positive integer within a protocol-defined maximum.
- Validate the overall shape of the `/pool_info` response (e.g., using Zod) rather than relying on `toCamelCase(data) as PoolInfo`.

### Proof of Concept

1. Attacker runs an HTTPS server at `https://evil-pool.example/pool_info` returning:
   ```json
   {
     "name": "Legitimate Pool",
     "target_puzzle_hash": "<attacker_32byte_hex>",
     "relative_lock_height": 2000,
     "protocol_version": "1.0",
     "fee": "0.01",
     "description": "...",
     "minimum_difficulty": 1
   }
   ```
2. User enters `https://evil-pool.example` as the pool URL in the Chia GUI "Join Pool" dialog.
3. `getPoolInfo` fetches the endpoint; `toCamelCase(data) as PoolInfo` casts the response with no validation.
4. `targetPuzzleHash` is a non-empty string → passes `!targetPuzzleHash` check.
5. `relativeLockHeight` is `2000` → passes `=== undefined` check.
6. `pwJoinPool` is called with the attacker's puzzle hash as `targetPuzzleHash`.
7. All subsequent farming rewards from the user's plots are sent to the attacker's address for 2000 blocks. [7](#0-6)
