Audit Report

## Title
Unenforced Access Control in `icrc103_get_allowances` Allows Any Caller to Enumerate Any Account's Full Allowance Table — (File: `rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary
The `icrc103_get_allowances` query endpoint accepts an arbitrary `from_account` argument and returns all allowances for that account without verifying that the caller is the account owner. The `AccessDenied` error variant and the `icrc103:public_allowances` metadata flag are both defined in the codebase but are never consulted at runtime, making the access-control mechanism defined by the ICRC-103 standard entirely inoperative.

## Finding Description
At `rs/ledger_suite/icrc1/ledger/src/main.rs` lines 1215–1231, the handler resolves `from_account` to either the caller's account (if `None`) or the explicitly supplied account, then unconditionally calls `get_allowances(...)` and wraps the result in `Ok(...)`:

```rust
fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    // ...
    Ok(get_allowances(from_account, arg.prev_spender, max_results, ic_cdk::api::time()))
}
```

There is no guard of the form:
```rust
if caller != from_account.owner && !public_allowances {
    return Err(GetAllowancesError::AccessDenied { ... });
}
```

Three independent code artifacts confirm the access-control mechanism was intended but never wired up:

1. `GetAllowancesError::AccessDenied { reason: String }` is defined in `packages/icrc-ledger-types/src/icrc103/get_allowances.rs` lines 16–18 but is never returned anywhere in the codebase.
2. `MetadataKey::ICRC103_PUBLIC_ALLOWANCES` (`"icrc103:public_allowances"`) is defined as a constant in `packages/icrc-ledger-types/src/icrc/metadata_key.rs` line 92 but is the only reference to `public_allowances` in the entire repository — no ledger state field, no reader, and no enforcement logic exists.
3. The integration test in `rs/ledger_suite/tests/sm-tests/src/lib.rs` line 2956–2958 explicitly exercises listing allowances of a *different* principal and expects success, confirming the behavior is present in the deployed code.

Any IC principal can supply `from_account: Some(victim_account)` and receive the complete paginated allowance list for that account. Repeated calls with `prev_spender` set to the last returned spender enumerate the full table.

## Impact Explanation
The concrete impact is **unauthorized disclosure of the full allowance graph** for any account on any ICRC-1 ledger that implements ICRC-103 (including ckBTC, ckETH, and ckERC20 ledgers). An attacker learns every `(spender, amount, expiry)` tuple for any target account without restriction. This enables identification of accounts with large outstanding approvals to specific spenders, facilitating targeted exploitation of compromised spender keys or front-running of pending `transfer_from` operations. This matches the allowed ICP bounty impact: **High ($2,000–$10,000) — Significant ledger security impact with concrete user harm**, as it constitutes unauthorized access to sensitive ledger state data affecting all ICRC-103-enabled ledgers in production.

## Likelihood Explanation
The endpoint is a public query call — no cycles cost, no authentication, no rate limit beyond `max_take_allowances`. Any IC principal can call it immediately from any wallet or script. Pagination via `prev_spender` makes full enumeration of an account's allowance table trivial. No special privileges, no victim interaction, and no preconditions beyond knowing a target account principal are required.

## Recommendation
Add a caller-identity guard before invoking `get_allowances`, reading the `public_allowances` ledger configuration (which must also be implemented in ledger state):

```rust
let caller = ic_cdk::api::msg_caller();
let public = Access::with_ledger(|l| l.public_allowances());
if !public && caller != from_account.owner {
    return Err(GetAllowancesError::AccessDenied {
        reason: "caller is not the account owner".to_string(),
    });
}
```

Additionally, implement `public_allowances()` as a configurable ledger field (defaulting to `false` per the ICRC-103 standard's privacy-first intent), and ensure `icrc1_metadata` correctly reflects the `icrc103:public_allowances` value.

## Proof of Concept
1. Deploy the ICRC-1 ledger canister (any ckBTC/ckETH/ckERC20 instance or local deployment).
2. Account A (principal `P_A`) approves spender S for 1000 tokens via `icrc2_approve`.
3. From any unrelated principal `P` (`P ≠ P_A`), call:
   ```
   icrc103_get_allowances({ from_account: Some(A), prev_spender: None, take: None })
   ```
4. Observe the response is `Ok([{from_account: A, to_spender: S, allowance: 1000, ...}])` — no `AccessDenied` error is returned regardless of the caller identity.
5. Repeat with `prev_spender` set to the last returned spender to paginate through the full allowance table.

This is directly confirmed by the existing integration test at `rs/ledger_suite/tests/sm-tests/src/lib.rs` lines 2956–2958, which explicitly tests and expects success when listing another principal's allowances.
