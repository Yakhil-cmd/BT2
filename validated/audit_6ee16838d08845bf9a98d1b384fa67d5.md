### Title
Unbounded `MessageBuffer` Growth via Arbitrary Waitpoint Injection — (`File: src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` inside `Comms` accumulates one `HashMap` entry and one `UnboundedSender` queue per unique `(ChannelTag, Waitpoint)` pair. Because `Comms::push_message` performs no validation on the `Waitpoint` value embedded in an incoming message header, a malicious participant can inject messages for arbitrarily many distinct waitpoints. The honest protocol only ever polls a small, fixed set of waitpoints, so the injected entries are never drained. Memory grows without bound until the process is killed by the OS, permanently denying DKG, reshare, refresh, presign, or sign for all honest parties.

---

### Finding Description

Every participant's protocol state is backed by a `Comms` object whose `incoming` field is a `MessageBuffer`:

```rust
struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}
``` [1](#0-0) 

Each `SubMessageQueue` is created on demand via `entry(header).or_default()` and uses an **explicitly unbounded** MPSC channel:

```rust
impl Default for SubMessageQueue {
    fn default() -> Self {
        let (sender, receiver) = futures::channel::mpsc::unbounded();
        ...
    }
}
``` [2](#0-1) 

The `push` path that stores every incoming message is:

```rust
fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
    let mut messages_lock = self.messages.lock().expect("lock should not fail");
    messages_lock.entry(header).or_default().send(from, message);
}
``` [3](#0-2) 

`Comms::push_message`, the only gate before `push`, performs exactly two checks: the raw byte slice is at least `MessageHeader::LEN` (40) bytes long, and those bytes parse as a valid `MessageHeader`. It performs **no validation** of the `Waitpoint` field:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {
        return;
    }
    let Some(header) = MessageHeader::from_bytes(&message) else {
        return;
    };
    self.incoming.push(header, from, message);
}
``` [4](#0-3) 

A `MessageHeader` is a `(ChannelTag, Waitpoint)` pair where `Waitpoint = u64`, giving 2⁶⁴ possible values:

```rust
struct MessageHeader {
    channel: ChannelTag,
    waitpoint: Waitpoint,
}
pub type Waitpoint = u64;
``` [5](#0-4) 

The honest protocol increments waitpoints sequentially from 0 and only ever polls the small set it actually uses. Injected entries for waitpoints such as `1_000_000`, `1_000_001`, … are never consumed and never evicted.

The codebase itself contains a test that explicitly demonstrates this attack surface:

```rust
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;
    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());
        comms.push_message(attacker, message);
    }
    // asserts all 512 entries are present — none were rejected
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [6](#0-5)

### Impact Explanation

Every protocol operation (DKG via `do_keyshare`, reshare, refresh, presign, sign) creates a `Comms` instance and drives it through `Protocol::message`. A malicious participant who can deliver messages to an honest party's `Protocol::message` endpoint can continuously inject 40-byte payloads with distinct high-valued waitpoints. Each injection allocates a new `HashMap` entry plus an `UnboundedSender`/`UnboundedReceiver` pair. With no eviction and no cap, the process heap grows without bound, eventually triggering an OOM kill. This permanently denies DKG, reshare, refresh, presign, and sign for all honest parties sharing that session.

---

### Likelihood Explanation

The `Protocol::message` method is the standard delivery interface for all network messages. Any participant in a session — including a single malicious one — can call it with crafted payloads. The attack requires only the ability to send 40-byte messages, which is the minimum message size already accepted by the protocol. No cryptographic material, no key compromise, and no coordination with other attackers is needed.

---

### Recommendation

1. **Cap the `HashMap` size**: Reject incoming messages whose `MessageHeader` key is not in a pre-computed allowlist of expected `(channel, waitpoint)` pairs, or enforce a hard upper bound (e.g., `N_participants × max_rounds`) on the number of distinct keys in `MessageBuffer`.
2. **Cap per-slot queue depth**: Replace `futures::channel::mpsc::unbounded()` with a bounded channel (e.g., `futures::channel::mpsc::channel(N_participants)`) so that even for legitimate waitpoints a single sender cannot queue more than `N` messages.
3. **Validate waitpoints at ingress**: In `Comms::push_message`, check that the parsed `waitpoint` falls within the range `[0, max_expected_waitpoint)` before inserting into the buffer.

---

### Proof of Concept

The existing test `attacker_can_fill_message_buffer_with_unused_waitpoints` in `src/protocol/internal.rs` (lines 530–554) is a direct proof of concept. It shows that 512 messages with distinct high-valued waitpoints all accumulate in the buffer with zero rejections. Scaling `attack_count` to millions and running it against a live DKG session would exhaust process memory before the honest protocol completes Round 1. [7](#0-6) [3](#0-2) [4](#0-3)
