### Title
Missing Cryptographic Validation of CKD Share Contributions Allows Malicious Participant to Corrupt Derived Key Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function accepts `(norm_big_y, norm_big_c)` contributions from participants and accumulates them without any cryptographic proof of correctness. A malicious participant can substitute arbitrary curve points, causing the coordinator to assemble a corrupted `CKDOutput` that delivers a wrong confidential key to the TEE.

---

### Finding Description

`compute_signature_share` (lines 148–181) produces two values per participant:

- `norm_big_y = λᵢ · yᵢ · G` — a Lagrange-weighted random blinding point
- `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)` — the Lagrange-weighted encrypted share

No NIZK proof is attached to these values. The participant simply sends them privately to the coordinator: [1](#0-0) 

The coordinator then blindly accumulates every received pair: [2](#0-1) 

There is no check that `norm_big_c` was formed from the participant's actual signing share `xᵢ`, nor that `norm_big_y` and `norm_big_c` share the same ephemeral scalar `yᵢ`. Any participant can replace both values with arbitrary group elements before calling `chan.send_private`.

Compare this with the robust ECDSA presign, which performs exponent-interpolation consistency checks on every received share before accepting it: [3](#0-2) 

No equivalent guard exists in the CKD path.

---

### Impact Explanation

The final coordinator output is:

```
Y = Σ norm_big_y_i
C = Σ norm_big_c_i
```

The TEE recovers the confidential key as `C − Y · app_sk`. If any participant injects wrong points, both `Y` and `C` are shifted by attacker-chosen offsets, so the recovered key is wrong. The TEE silently accepts the corrupted value because no verification step follows the accumulation.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session can mount this attack. No special cryptographic capability is required — the attacker only needs to deviate from `compute_signature_share` and call `chan.send_private` with hand-crafted `ElementG1` values. The attack is undetectable by the coordinator under the current protocol.

---

### Recommendation

Require each participant to attach a Chaum-Pedersen-style NIZK proof demonstrating that `norm_big_y` and `norm_big_c` share the same ephemeral scalar `yᵢ` and that `norm_big_c` encodes the correct signing share `xᵢ`. Concretely, the participant should prove knowledge of `(xᵢ, yᵢ)` such that:

```
norm_big_y = λᵢ · yᵢ · G
norm_big_c = λᵢ · xᵢ · H(pk ‖ app_id) + λᵢ · yᵢ · app_pk
```

The coordinator must verify each proof before adding the contribution to the running sum, analogous to how `validate_received_share` is used in DKG: [4](#0-3) 

---

### Proof of Concept

1. A threshold CKD session is initiated with `n` participants, one of which is malicious.
2. The malicious participant bypasses `compute_signature_share` and calls `chan.send_private(waitpoint, coordinator, &(ElementG1::identity(), ElementG1::generator()))` — two arbitrary points.
3. The coordinator executes lines 50–55 of `protocol.rs`, adding the injected points to the running sums without error.
4. The coordinator returns `CKDOutput::new(norm_big_y, norm_big_c)` where both components are shifted by the injected offsets.
5. The TEE calls `ckd_output.unmask(app_sk)` and obtains `C − Y · app_sk ≠ msk · H(pk ‖ app_id)`.
6. The TEE silently uses the wrong derived key; no protocol-level error is raised. [5](#0-4)

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}
