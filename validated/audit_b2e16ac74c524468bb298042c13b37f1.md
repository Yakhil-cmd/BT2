### Title
Gateway `skip_stateful_validations` admits invoke transactions with invalid signatures by checking any mempool presence instead of a deploy-account - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (i.e., signature verification) for any Invoke V3 transaction whose nonce equals 1 and whose sender's on-chain nonce is 0, provided `account_tx_in_pool_or_recent_block` returns `true`. The check is too broad: it returns `true` whenever the address has **any** transaction in the pool or in a recently committed block, not only a `deploy_account`. An unprivileged attacker who observes a pending `deploy_account` for address X can immediately submit an Invoke with nonce=1 carrying an **invalid signature** for that same address. The gateway passes all pre-validation checks, skips the `__validate__` call, and forwards the malformed transaction to the mempool.

### Finding Description

**Root cause — `skip_stateful_validations`**

```
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461

                .await ...

    Ok(false)

``` [1](#0-0) 

The comment says "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is the problem: any invoke with nonce ≥ 2 that the attacker themselves submitted for address X (with a valid signature, since nonce ≠ 1 so skip does not apply) causes `account_tx_in_pool_or_recent_block` to return `true`, after which the attacker's second invoke with nonce=1 and an **invalid** signature bypasses `__validate__`.

**`account_tx_in_pool_or_recent_block` implementation**

```
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`contains_account` on `MempoolState` checks staged/committed nonces; `contains_account` on `TransactionPool` checks whether any transaction from that address is in the pool. A `deploy_account` submitted by the legitimate user is sufficient to trigger `true`. [3](#0-2) 

**Nonce pre-check passes**

The gateway's `validate_nonce` for non-declare, non-deploy-account transactions allows any nonce in `[account_nonce, account_nonce + max_allowed_nonce_gap]`. With `account_nonce = 0` and the production default `max_allowed_nonce_gap = 200`, nonce=1 is accepted. [4](#0-3) [5](#0-4) 

**Mempool `validate_tx` does not check signatures**

`Mempool::validate_tx` only checks for duplicate hashes and stale nonces; it has no knowledge of cryptographic validity. [6](#0-5) 

**`run_validate_entry_point` is skipped**

When `skip_validate = true`, `run_validate_entry_point` is called with `validate: false`, so the blockifier's `StatefulValidator::perform_validations` returns early without calling `__validate__`. [7](#0-6) [8](#0-7) 

### Impact Explanation

The gateway admits an Invoke transaction with an invalid (or empty) signature to the mempool. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Downstream consequences:
- The invalid transaction occupies a mempool slot and may displace legitimate transactions.
- When the batcher includes it in a block, the blockifier calls `__validate__` with `validate=true` (the skip flag is a gateway-only concept). The transaction reverts, but the account is still charged fees, causing economic harm to the victim.
- The attack is repeatable for every new `deploy_account` observed in the mempool.

### Likelihood Explanation

- The Starknet mempool is public and transactions are broadcast over P2P, so any observer can detect a pending `deploy_account`.
- The attacker needs only to craft an Invoke with nonce=1 and any (invalid) signature bytes; no private key knowledge is required.
- The production configuration (`max_allowed_nonce_gap = 200`, `max_nonce_for_validation_skip = 0x1`) is exactly the setting that enables the attack.
- No privileged access, no special tooling, and no race condition beyond observing the mempool before the `deploy_account` is committed.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. Alternatively, pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (the field already exists in `StatefulTransactionValidatorConfig` but is unused in this code path) and additionally require that the matching mempool entry is a `deploy_account` type, not an arbitrary transaction. [9](#0-8) 

### Proof of Concept

1. **Setup**: Victim submits `deploy_account` for address `X` (nonce=0). The transaction enters the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.

2. **Attack**: Attacker calls `gateway.add_tx(invoke_v3 { sender=X, nonce=1, signature=[] })`.

3. **Gateway path**:
   - `StatelessTransactionValidator::validate` — passes (empty signature is within `max_signature_length`).
   - `convert_rpc_tx_to_internal` — computes tx hash, no class compilation needed.
   - `extract_state_nonce_and_run_validations`:
     - `get_nonce_from_state(X)` → `Nonce(0)` (account not yet deployed).
     - `validate_nonce`: `0 ≤ 1 ≤ 200` → OK.
     - `validate_by_mempool`: no duplicate hash, nonce 1 ≥ 0 → OK.
     - `skip_stateful_validations`: nonce==1 && account_nonce==0 && `account_tx_in_pool_or_recent_block(X)` == `true` → returns `true`.
     - `run_validate_entry_point(skip_validate=true)` → `validate=false` → returns immediately without calling `__validate__`.
   - Transaction forwarded to mempool. **Invalid invoke with empty signature is now queued.**

4. **Block execution**: Batcher pulls both transactions. `deploy_account` executes (nonce 0→1). Blockifier then executes the attacker's invoke with `validate=true`; `__validate__` fails (invalid signature); transaction reverts; victim's account is charged fees.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
