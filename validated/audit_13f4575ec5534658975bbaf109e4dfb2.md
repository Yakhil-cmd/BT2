### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Corrupted Signature Output - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate the `threshold` parameter only with an upper-bound check (`threshold > participants.len()`), but omit the lower-bound check (`threshold < 2`) that is enforced in every other protocol entry point in the codebase. A malicious coordinator or library caller can pass `threshold = 0` or `threshold = 1` to the FROST signing path, causing incorrect Lagrange interpolation and producing an invalid, unusable signature — permanently denying signing for honest parties in that session.

### Finding Description

`ReconstructionLowerBound` is a plain `usize` newtype with no built-in constraints: [1](#0-0) 

Every other protocol entry point that accepts a threshold enforces a minimum of 2. For example, `assert_key_invariants` in `src/dkg.rs`: [2](#0-1) 

And `validate_triple_inputs` in the OT-based triple generation: [3](#0-2) 

However, `assert_sign_inputs` — the shared validation gate for all FROST EdDSA and RedJubjub signing — only checks the upper bound: [4](#0-3) 

The lower-bound check (`threshold < 2`) is entirely absent. The same omission exists in the FROST `presign` function: [5](#0-4) 

### Impact Explanation

In FROST, the `threshold` value directly controls Lagrange interpolation during signature aggregation. Keys are always generated with `threshold >= 2` (enforced by `assert_key_invariants`). If a caller passes `threshold = 1` to the signing path, the Lagrange coefficients are computed for a degree-0 polynomial, which is inconsistent with the degree-(threshold-1) polynomial used during key generation. The resulting signature shares are computed with wrong coefficients, and the aggregated signature fails verification — producing an unusable cryptographic output. With `threshold = 0`, Lagrange interpolation over an empty participant set is undefined and may panic, crashing the signing session entirely.

**Impact: High** — Corruption of FROST sign outputs so honest parties accept an unusable cryptographic output (invalid signature), or permanent denial of signing for honest parties in the affected session.

### Likelihood Explanation

The entry point is the public `assert_sign_inputs` function and the `presign` function, both callable by any library user or coordinator. No privileged access is required — the caller simply passes an out-of-range `threshold` value (e.g., `1usize.into()`). The inconsistency with every other validated entry point makes accidental misuse equally likely alongside deliberate abuse.

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` and `validate_triple_inputs` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This check should be inserted before the upper-bound check in both functions, mirroring the pattern at `src/dkg.rs:580-582`.

### Proof of Concept

1. Run DKG with `threshold = 2`, `participants = [A, B, C]` — succeeds normally (lower-bound enforced).
2. Call `assert_sign_inputs(&[A, B, C], 1usize, A, A)` — **returns `Ok`** because only the upper-bound check fires.
3. Proceed to FROST signing with `threshold = 1`; Lagrange coefficients are computed for a degree-0 polynomial inconsistent with the degree-1 shares from step 1.
4. The aggregated signature fails `verify()` — honest parties cannot produce a usable signature for this session.

The root cause is the missing `threshold < 2` guard in `src/frost/mod.rs` at lines 71–77 (`presign`) and lines 144–150 (`assert_sign_inputs`), in direct contrast to the validated path at `src/dkg.rs:580–582`.
