### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions When Any Account Tx Exists in Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 when a `deploy_account` transaction is pending in the mempool (UX improvement for simultaneous deploy+invoke submission). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the account, not exclusively a `deploy_account` transaction. An attacker who observes a victim's `deploy_account` tx in the mempool can immediately submit a forged invoke tx (nonce=1, invalid signature) from the victim's address. The gateway skips `__validate__` for this forged tx, admits it to the mempool, and the victim's own legitimate nonce=1 invoke tx is subsequently rejected as a duplicate nonce — permanently blocking the account's first post-deployment invoke.

### Finding Description

**Path A — normal invoke validation (full check):**

`extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `run_validate_entry_point` with `validate: true` → blockifier calls `__validate__` (signature verification). [1](#0-0) 

**Path B — skipped validation (the bypass):**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: !skip_validate = false`, so `__validate__` is never invoked. [2](#0-1) 

**The flawed guard:**

`skip_stateful_validations` triggers the bypass whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. The comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is wrong: the check is satisfied by **any** pending tx from the account, including a `deploy_account` tx that the attacker observed but did not submit. [3](#0-2) 

**The mempool check is type-blind:**

`account_tx_in_pool_or_recent_block` checks only whether the account address appears in the pool or recent-block state — it does not distinguish `deploy_account` from `invoke`. [4](#0-3) 

### Impact Explanation

An attacker can submit a structurally valid but cryptographically forged invoke transaction (nonce=1, arbitrary/invalid signature) from a victim's address immediately after observing the victim's `deploy_account` tx in the mempool. The gateway admits the forged tx without calling `__validate__`. The victim's own legitimate nonce=1 invoke tx is then rejected by the mempool as a duplicate nonce. This constitutes:

- **Gateway/mempool admission of an invalid (unauthorized) transaction** — the forged tx carries an invalid signature yet passes all gateway checks.
- **Denial-of-service on the victim's account** — the victim cannot submit their first post-deployment invoke tx; their account is stuck at nonce=1 in the mempool with an unexecutable forged tx occupying the slot.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The attack requires only:
1. Observing a `deploy_account` tx in the public mempool (trivially possible).
2. Constructing a forged invoke tx with nonce=1 from the victim's address (any bytes as signature).
3. Submitting it before the victim's legitimate invoke tx.

No privileged access, no special account state, and no cryptographic capability is required. The window is the time between the victim's `deploy_account` submission and their invoke submission — a common simultaneous-submission UX pattern explicitly supported by the codebase.

### Recommendation

Replace the type-blind `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** exists for the sender address in the mempool. Add a new mempool API method such as `deploy_account_tx_in_pool(address)` that inspects transaction type, and use it in `skip_stateful_validations` instead of the current broad check. [5](#0-4) 

### Proof of Concept

1. **Setup**: Victim Alice generates a fresh keypair; her account address is `A`, account nonce in committed state = 0.
2. **Alice submits**: `deploy_account` tx (nonce=0, valid signature) → admitted to mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. **Attacker observes** Alice's `deploy_account` tx in the mempool.
4. **Attacker submits**: invoke tx from address `A`, nonce=1, `signature = [0x1337]` (invalid).
5. **Gateway evaluation** of attacker's tx:
   - `validate_nonce`: account_nonce=0, tx_nonce=1, gap=1 ≤ `max_allowed_nonce_gap` → **passes**.
   - `validate_by_mempool`: no duplicate nonce=1 tx yet → **passes**.
   - `skip_stateful_validations`: invoke? ✓ nonce=1? ✓ account_nonce=0? ✓ `account_tx_in_pool_or_recent_block(A)`? ✓ → returns `true`.
   - `run_validate_entry_point` called with `validate: false` → **`__validate__` never called**.
6. **Forged tx admitted** to mempool with nonce=1.
7. **Alice submits** her legitimate invoke tx (nonce=1, valid signature) → mempool rejects with `DuplicateNonce`.
8. **Result**: Alice's account is stuck; the forged tx will fail at `__validate__` during block execution and be reverted, but Alice cannot replace it without fee-escalation support, and the attacker can repeat step 4 indefinitely. [6](#0-5) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
