### Title
RCAT Wallet Mojo Conversion Uses `chiaToMojo` Instead of `catToMojo` in NFT Offer Builder, Causing 1e9x Token Overcounting — (`packages/gui/src/components/offers/NFTOfferEditor.tsx`)

### Summary

`NFTOfferTokenSelector` exposes `WalletType.RCAT` as a selectable token option, but `buildOfferRequest` in `NFTOfferEditor.tsx` omits `WalletType.RCAT` from its CAT-family mojo conversion check. When a user selects an RCAT wallet and submits an NFT offer, the amount is converted using `chiaToMojo` (1 unit = 1e12 mojos) instead of `catToMojo` (1 unit = 1e3 mojos), encoding an amount 1,000,000,000× larger than intended. A counterparty who accepts the offer drains the user's RCAT balance at that inflated rate.

### Finding Description

**Step 1 — RCAT is offered as a valid selection.**

`NFTOfferTokenSelector.tsx` line 59 filters wallets to include `WalletType.RCAT`:

```ts
.filter((wallet: Wallet) => [WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(wallet.type))
``` [1](#0-0) 

**Step 2 — `buildOfferRequest` omits RCAT from the CAT-family branch.**

`NFTOfferEditor.tsx` line 493 checks only `[WalletType.CAT, WalletType.CRCAT]`:

```ts
const baseMojoAmount: BigNumber = [WalletType.CAT, WalletType.CRCAT].includes(tokenWalletInfo.walletType)
  ? catToMojo(tokenAmount)
  : chiaToMojo(tokenAmount);
``` [2](#0-1) 

`WalletType.RCAT = 132` is not in that array, so it falls through to `chiaToMojo`.

**Step 3 — Conversion factor mismatch.**

- `chiaToMojo("1")` → 1,000,000,000,000 mojos (1e12)
- `catToMojo("1")` → 1,000 mojos (1e3)
- Ratio: **1,000,000,000× (1e9)** [3](#0-2) [4](#0-3) 

**Contrast with display logic** — `formatAmountForWalletType` in `utils.ts` correctly includes RCAT in the CAT group, confirming the omission in `buildOfferRequest` is an inconsistency, not intentional design:

```ts
if ([WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(walletType)) {
  return mojoToCATLocaleString(amount, locale);
}
``` [5](#0-4) 

### Impact Explanation

A user with an RCAT wallet who creates a TokenForNFT offer and enters e.g. `"1"` RCAT will have their offer encoded as 1,000,000,000,000 mojos instead of 1,000 mojos. If they hold sufficient RCAT balance, the offer is created and broadcast. A counterparty who accepts it receives 1e9× the intended RCAT amount, constituting direct, irreversible asset loss. The user sees no warning — the UI input and display path treat RCAT correctly; only the offer-building conversion is wrong.

### Likelihood Explanation

RCAT (`WalletType = 132`) is a defined, real wallet type in the Chia API. [6](#0-5) 

The attacker's role is minimal: they only need to be a willing counterparty who accepts the malformed offer. No privileged access, key compromise, or social engineering beyond proposing an NFT trade is required. The user performs all actions themselves through the normal NFTOfferEditor UI flow.

### Recommendation

Add `WalletType.RCAT` to the CAT-family check in `buildOfferRequest`:

```ts
const baseMojoAmount: BigNumber = [WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(tokenWalletInfo.walletType)
  ? catToMojo(tokenAmount)
  : chiaToMojo(tokenAmount);
``` [2](#0-1) 

### Proof of Concept

```ts
// Unit test
import { WalletType } from '@chia-network/api';
import { catToMojo, chiaToMojo } from '@chia-network/core';
import buildOfferRequest from './buildOfferRequest'; // extracted or tested inline

const params = {
  exchangeType: NFTOfferExchangeType.TokenForNFT,
  nft: mockNFT,
  nftLauncherId: 'abc123',
  tokenWalletInfo: { walletId: 5, walletType: WalletType.RCAT },
  tokenAmount: '1',
  fee: '0',
};

const [offerDict] = buildOfferRequest(params);
const mojoAmount = offerDict[5]; // walletId 5

// BUG: mojoAmount === chiaToMojo('1') === 1_000_000_000_000n
// EXPECTED: mojoAmount === catToMojo('1') === 1_000n
assert.deepEqual(mojoAmount, catToMojo('1')); // fails without the fix
```
