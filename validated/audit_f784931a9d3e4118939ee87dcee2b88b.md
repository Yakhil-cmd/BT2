### Title
Sliding-Window Digest Eviction Allows Duplicate Key Re-Registration, Bypassing Key Revocation — (`fvm/environment/account-key-metadata/metadata.go`)

### Summary

The `FindDuplicateKey` function uses a sliding window of only `MaxStoredDigests=2` digests to detect duplicate public keys. An attacker who controls an account can deliberately evict a key's digest from this window by adding two unique keys, then re-register the same public key material as a new "unique" key at a new key index. Revoking the original key index does not affect the re-registered key index, allowing the attacker to continue signing transactions with the "revoked" key material indefinitely.

### Finding Description

`MaxStoredDigests` is set to 2 in `fvm/environment/accounts_status.go`: [1](#0-0) 

The `appendDigest` function in `fvm/environment/account-key-metadata/metadata.go` evicts the oldest digest when the window is full: [2](#0-1) 

`FindDuplicateKey` only searches the current in-window digests: [3](#0-2) 

`findDuplicateDigest` only scans `m.digestBytes` (the current window): [4](#0-3) 

When `FindDuplicateKey` returns `found=false`, `AppendUniqueKeyMetadata` is called, which assigns a new `storedKeyIndex` and sets `saveKey=true`, causing the key bytes to be written to storage again: [5](#0-4) 

Revoking key index 0 only modifies the `apk_0` register: [6](#0-5) 

The transaction verifier checks `accountKey.Revoked` for the *presented* key index (3), not for the original key index (0): [7](#0-6) 

The sequence number checker also checks revoked status by the presented key index: [8](#0-7) 

### Impact Explanation

An attacker who controls an account can permanently bypass key revocation. After revoking key index 0, the same private key material remains usable via key index 3 (or any subsequent re-registered index). This violates the core security invariant that `RevokeAccountPublicKey` permanently disables a public key's signing authority. The attacker can sign arbitrary transactions — including those authorizing resource transfers, contract updates, or capability grants — using the supposedly-revoked key.

### Likelihood Explanation

The attack requires only 3 `addKey` Cadence transactions (to add key1, key2, and re-add K) plus 1 `revokeKey` transaction. All operations are available to any account owner via standard Cadence `AuthAccount.keys` API. No privileged access, staked node control, or external dependency is required. The window size of 2 makes the eviction trivially achievable with exactly 2 intermediate keys.

### Recommendation

1. **Increase `MaxStoredDigests`** to cover the full key history, or store digests for all keys ever added (not just the last N). This eliminates the eviction window entirely.
2. **Alternatively**, when a key is added as "unique" (digest not found in window), perform a full linear scan of all stored keys to confirm it is truly unique before assigning a new `storedKeyIndex`. This is more expensive but closes the gap.
3. **At minimum**, document that the current design intentionally permits duplicate key re-registration after window eviction, and assess whether the security model of key revocation is still sound under this constraint.

### Proof of Concept

```
1. Create account A with key K (index 0, weight 1000).
2. Add unique key1 to A → key index 1, storedKeyIndex 1.
   Digest window: [digest(K), digest(key1)]
3. Add unique key2 to A → key index 2, storedKeyIndex 2.
   appendDigest evicts digest(K): window = [digest(key1), digest(key2)]
4. Add K again to A → FindDuplicateKey misses it (digest(K) not in window).
   K is stored as storedKeyIndex 3, key index 3, revoked=false.
5. Revoke key index 0 → only apk_0 register is marked revoked.
   Key index 3 remains revoked=false.
6. Submit transaction signed with K's private key, proposalKey.KeyIndex=3.
   TransactionVerifier: GetRuntimeAccountPublicKey(A, 3) → revoked=false, pubkey=K → PASS.
   Signature verification: valid (K's private key matches K's public key) → PASS.
   Transaction executes successfully despite K being "revoked".
```

func (m *KeyMetadataAppender) appendDigest(digest uint64) {
	digestCount := 1 + len(m.digestBytes)/digestSize

	if digestCount > int(m.maxStoredDigests) {
		// Remove digest from front
		removeCount := digestCount - int(m.maxStoredDigests)
		m.digestBytes = slices.Delete(m.digestBytes, 0, removeCount*digestSize)

		// Adjust digest start index
		m.startIndexForDigests += uint32(removeCount)
	}

	var digestBytes [digestSize]byte
	binary.BigEndian.PutUint64(digestBytes[:], digest)

	m.digestBytes = append(m.digestBytes, digestBytes[:]...)
}

func (m *KeyMetadataAppender) findDuplicateDigest(digest uint64) (found bool, duplicateStoredKeyIndex uint32) {
	if len(m.digestBytes) == 0 {
		return false, 0
	}

	var digestBytes [digestSize]byte
	binary.BigEndian.PutUint64(digestBytes[:], digest)

	for off, i := 0, uint32(0); off < len(m.digestBytes); off, i = off+digestSize, i+1 {
		if bytes.Equal(digestBytes[:], m.digestBytes[off:off+digestSize]) {
			return true, m.startIndexForDigests + i
		}
	}

	return false, 0
}
