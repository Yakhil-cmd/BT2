Audit Report

## Title
SNS Treasury 7-Day Transfer Cap Bypassed via Stale Genesis-Price Oracle for SNS Token Valuation — (`rs/sns/governance/token_valuation/src/lib.rs`)

## Summary
The SNS treasury protection system enforces a 7-day transfer cap by converting a 300,000 XDR hard limit into a token count using a price derived solely from the genesis swap finalization price, adjusted only for token supply inflation. When an SNS token has appreciated significantly in market value since its genesis swap, the stale price undervalues the tokens, causing the computed token allowance to represent far more XDR value than the protocol intends to permit. A governance-approved `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal can therefore drain treasury value orders of magnitude above the intended hard cap.

## Finding Description
`IcpsPerSnsTokenClient::fetch_icps_per_sns_token()` in `rs/sns/governance/token_valuation/src/lib.rs` (lines 314–415) computes the SNS token price as:

```
current_icps_per_token = (1 / genesis_sns_tokens_per_icp) / (current_supply / initial_supply)
```

The `genesis_sns_tokens_per_icp` is obtained by calling `get_derived_state` on the swap canister. In `rs/sns/swap/src/swap.rs` (lines 2992–2995), `derived_state()` computes `sns_tokens_per_icp` from `tokens_available_for_swap` and `participant_total_icp_e8s`. After the SNS initialization swap finalizes, both of these fields are frozen — the swap canister never updates them to reflect post-swap market activity. Consequently, `genesis_sns_tokens_per_icp` is permanently fixed at the swap finalization price.

The only adjustment applied is a dilution factor for token supply inflation (`current_supply / initial_supply`). There is no secondary price source, no market price feed, no deviation threshold, and no staleness guard.

This stale price feeds directly into `ProposalsAmountTotalUpperBound::in_tokens()` in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` (lines 66–114). For large treasuries (>1,200,000 XDR), the hard cap is `MAX_XDR = 300,000 XDR`. The function computes:

```
allowance_tokens = 300_000 XDR * (1 / stale_xdrs_per_token)
```

When the SNS token has appreciated 100× since genesis, `stale_xdrs_per_token` is 100× lower than the actual market price, so `allowance_tokens` is 100× larger than intended. The existing `clamp_xdrs_per_icp` guard (lines 137–140) only clamps the ICP/XDR rate to a minimum of 1 XDR/ICP; it does not clamp `icps_per_token` and provides no protection against a stale SNS token price.

The valuation is snapshotted at proposal submission time (`rs/sns/governance/src/proposal.rs`, lines 570–578) and stored as `ActionAuxiliary::TransferSnsTreasuryFunds(valuation)`. At execution time (`rs/sns/governance/src/governance.rs`, lines 3000–3005), `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` reuses this stored valuation, so the price is doubly stale: fixed at genesis and again at submission time.

## Impact Explanation
The treasury protection mechanism — the primary on-chain safeguard against large SNS treasury drains — is rendered ineffective for any SNS whose token has appreciated since its genesis swap. A governance-approved proposal can transfer SNS tokens whose actual market value is orders of magnitude above the intended 300,000 XDR hard cap. This constitutes a significant SNS security impact with concrete, quantifiable harm to token holders and the SNS treasury, matching the High ($2,000–$10,000) impact category: "Significant SNS or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
Any SNS token that has appreciated since its genesis swap is affected. Token appreciation is a normal and expected outcome for successful SNS projects. The exploit requires only that a `TransferSnsTreasuryFunds` proposal be submitted and adopted through normal SNS governance — no privileged access, no key compromise, and no threshold attack is required. The proposal submitter is an ordinary neuron holder. The treasury protection cap is specifically designed to be a hard protocol-level limit that blocks even governance-approved proposals; the vulnerability defeats this cap for any SNS with a post-genesis appreciating token.

## Recommendation
1. Replace the single genesis-price oracle with a dual-source approach: retain the inflation-adjusted genesis price as a reference, and integrate a current on-chain price source (e.g., a DEX TWAP or exchange-rate canister query for the SNS token).
2. When computing the treasury cap, use the **higher** of the two prices (i.e., assume the token is worth at least as much as the market says), so the XDR cap is not bypassed when the token appreciates.
3. Add a `MAX_ICPS_PER_TOKEN` clamp analogous to the existing `MIN_XDRS_PER_ICP` to bound how far the stale genesis price can diverge from a reasonable current estimate.
4. Add a staleness guard that rejects the genesis price if it deviates beyond a configurable threshold from the secondary source.

## Proof of Concept
**Setup**: Genesis swap finalized at 10 SNS tokens/ICP. Token supply doubled since genesis (2× inflation). Current market price: 10 ICP/SNS token (100× appreciation). ICP/XDR rate: 10 XDR/ICP. Treasury holds 1,000,000 SNS tokens.

**Stale valuation computed by protocol**:
- `icps_per_token` = (1/10) / 2 = 0.05 ICP/token
- `xdrs_per_token` = 0.05 × 10 = 0.5 XDR/token
- Treasury XDR value = 1,000,000 × 0.5 = 500,000 XDR → "large" regime
- `allowance_tokens` = 300,000 / 0.5 = **600,000 tokens**

**Actual market value of allowance**:
- 600,000 tokens × 10 ICP/token × 10 XDR/ICP = **60,000,000 XDR**

**Intended cap**: 300,000 XDR. **Actual cap enforced**: 60,000,000 XDR — a **200× bypass**.

A unit test can be written against `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` by constructing a `Valuation` with `icps_per_token = 0.05` and `xdrs_per_icp = 10`, verifying the returned token allowance is 600,000, and then asserting that 600,000 tokens × actual market price (10 ICP × 10 XDR) = 60,000,000 XDR >> 300,000 XDR.
