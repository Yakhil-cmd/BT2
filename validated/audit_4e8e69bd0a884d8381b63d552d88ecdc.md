### Title
Missing G2 Subgroup Check in BN254 ECPAIRING Precompile — (`basic_system/src/system_functions/bn254_pairing_check.rs`)

### Summary

`bn254_pairing_check_inner` validates G2 points only for curve membership (`g2_point.check()`) but never calls `is_in_correct_subgroup_assuming_on_curve`. The method is fully implemented for BN254 G2 but is simply never invoked. The BLS12-381 pairing path in the same codebase correctly performs the subgroup check. This asymmetry is the root cause.

### Finding Description

In `bn254_pairing_check_inner`, after constructing a G2 point from raw bytes, the code does:

```rust
let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
g2_point.check().map_err(|_| ())?;   // only verifies point is on E'(Fq2)
// ← is_in_correct_subgroup_assuming_on_curve is NEVER called
``` [1](#0-0) 

`g2_point.check()` (from `ark_serialize::Valid`) only verifies the Weierstrass equation holds over Fq2. It does **not** verify the point is in the r-torsion subgroup G2 ⊂ E'(Fq2).

The subgroup check IS implemented for BN254 G2 using the efficient method from §4.3 of https://eprint.iacr.org/2022/352.pdf:

```rust
fn is_in_correct_subgroup_assuming_on_curve(point: &G2Affine) -> bool {
    let x_times_point = point.mul_bigint(SIX_X_SQUARED);
    let p_times_point = p_power_endomorphism(point);
    x_times_point.eq(&p_times_point)
}
``` [2](#0-1) 

By contrast, the BLS12-381 pairing path explicitly calls `is_in_correct_subgroup_assuming_on_curve` for both G1 and G2: [3](#0-2) 

The BN254 G2 cofactor `h ≈ 2^254` is large: [4](#0-3) 

This means E'(Fq2) contains a large cofactor subgroup H of order h, entirely disjoint from G2. Points in H satisfy `g2_point.check()` (they are on the curve) but fail `is_in_correct_subgroup_assuming_on_curve`.

### Impact Explanation

EIP-197 requires that G2 inputs to ECPAIRING be in the r-torsion subgroup; non-subgroup points must cause the precompile to return failure. ZKsync OS instead accepts them and feeds them into `Bn254::multi_pairing`. [5](#0-4) 

For a point Q' in the cofactor subgroup H (order dividing h, not in G2), the Ate pairing's Miller loop still executes and the final exponentiation to `(p^12-1)/r` is applied. Because the Ate pairing is non-degenerate only on G1 × G2, points in H lie in the kernel of the pairing map — `e(P, Q') = 1` for all P ∈ G1. An attacker can therefore:

1. Construct Q' ∈ H (a point on E'(Fq2) with order dividing h, not in G2).
2. Submit `(P, Q')` to the ECPAIRING system function.
3. `bn254_pairing_check_inner` accepts Q' (passes `check()`), computes `Bn254::multi_pairing`, and returns `true` because `result.0.is_one()` holds.
4. Any smart contract using ECPAIRING for Groth16 proof verification, BLS signature verification, or similar receives a false `1` (success) for an invalid input.

### Likelihood Explanation

- The attacker is unprivileged; the ECPAIRING precompile is callable by any contract.
- Constructing a point in the cofactor subgroup H is straightforward: take any point T on E'(Fq2) and compute Q' = [r]T. Since |E'(Fq2)| = r·h, [r]T has order dividing h and is in H (unless T ∈ G2, in which case [r]T = O).
- No special privileges, leaked keys, or external oracle cooperation is required.
- The BLS12-381 path in the same codebase demonstrates the developer knows the check is needed — its omission from the BN254 path is an oversight, not a design choice.

### Recommendation

After `g2_point.check()` passes, add:

```rust
if !g2_point.is_in_correct_subgroup_assuming_on_curve() {
    return Err(());
}
```

This mirrors the BLS12-381 path (`parse_g2_with_subgroup_check`) and uses the already-implemented efficient check in `crypto/src/bn254/curves/g2.rs`.

### Proof of Concept

```
1. Pick any generator T of E'(Fq2) that is NOT in G2.
2. Compute Q' = [r]T  (this has order dividing h, so Q' ∈ H ⊂ E'(Fq2) \ G2).
3. Verify Q' is on the curve: Q'.check() == Ok(()).
4. Verify Q' is NOT in G2: Q'.is_in_correct_subgroup_assuming_on_curve() == false.
5. Pick any non-zero P ∈ G1.
6. Encode (P, Q') as 192 bytes per EIP-197 encoding.
7. Call Bn254PairingCheckImpl::execute with this input.
8. Assert the output is 0x00…01 (true).
9. Compare against go-ethereum's bn256.RunPair: it returns an error / false for the same input.
```

The divergence between step 8 (ZKsync OS returns true) and step 9 (Ethereum returns false/error) is the concrete, locally-testable proof of the vulnerability.
