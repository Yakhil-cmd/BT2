### Title
Unauthorized Drain of IBC Escrow Accounts via Bank Precompile `transfer` — Missing `from == caller` Authorization Check - (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `TransferMethodName` handler in `BankContract.Run` accepts an arbitrary `from` address supplied by the caller and passes it directly to `bankKeeper.SendCoins` without verifying that `from` equals `contract.Caller()`. The denom is correctly scoped to `evm/<callerContract>`, but any account holding that denom — including IBC escrow accounts — can be drained by the contract that owns the denom.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case:

```go
sender := args[0].(common.Address)   // attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(to); err != nil { return nil, err }
denom := EVMDenom(contract.Caller())  // "evm/0xCallerContract"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is **no check that `from == contract.Caller()`**. The only guard on the sender side is that the denom is `evm/<callerContract>`. Any account holding that denom — regardless of who owns it — can be used as the `from` address.

The `checkBlockedAddr` guard only protects the **recipient** from being a module account; it does not protect the **sender**: [2](#0-1) 

IBC escrow accounts are **not** module accounts. `BlockedAddrs()` is built exclusively from `maccPerms`: [3](#0-2) 

The `maccPerms` map contains only named module accounts (`ibctransfer`, `evm`, `cronos`, etc.), not the per-channel escrow accounts derived by `ibctransfertypes.GetEscrowAddress(portID, channelID)`: [4](#0-3) 

Therefore `bc.bankKeeper.BlockedAddr(escrowAddr)` returns `false` for any IBC escrow account, and `SendCoins` proceeds.

### Impact Explanation

When a user initiates an IBC transfer of `evm/0xA` tokens (a source-chain token), the IBC transfer module escrows those tokens in the per-channel escrow account (a regular, non-module account). Before the packet acknowledgement or timeout arrives, contract `A` can call:

```
bank.transfer(escrowAccount, attacker, amount)
```

The precompile computes `denom = "evm/0xA"` (matching the escrowed tokens), skips all authorization checks, and calls `bankKeeper.SendCoins(escrowAccount → attacker)`. The escrow account is drained. When the IBC ack arrives, the IBC module attempts to release tokens from the now-empty escrow account, causing an accounting failure. The attacker has received tokens that are simultaneously "in flight" on the counterparty chain — a double-spend of IBC-escrowed `evm/` tokens.

### Likelihood Explanation

The attack requires only:
1. Deploying a contract on Cronos EVM (permissionless).
2. Having some `evm/<contract>` tokens escrowed via an IBC transfer (initiatable by any user).
3. Calling the bank precompile `transfer` method with the escrow account as `from`.

No admin keys, governance, or special permissions are needed. The escrow account address is publicly queryable via the IBC transfer query API.

### Recommendation

Add an authorization check in the `TransferMethodName` case to enforce that the `from` address equals the calling contract:

```go
case TransferMethodName:
    ...
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
```

Alternatively, remove the `from` argument entirely and derive it from `contract.Caller()`, consistent with how `mint` and `burn` operate.

### Proof of Concept

1. Deploy contract `A` at address `0xA` on Cronos.
2. Mint `evm/0xA` tokens to a user via `bank.mint`.
3. User calls `MsgTransfer` to IBC-transfer `evm/0xA` tokens to chain B. The IBC transfer module escrows the tokens at `escrowAddr = GetEscrowAddress("transfer", "channel-0")`.
4. Before the relayer submits the ack, contract `A` calls:
   ```solidity
   IBankModule(0x64).transfer(escrowAddr, attacker, amount);
   ```
5. The precompile computes `denom = "evm/0xA"`, checks `BlockedAddr(attacker) == false`, and executes `SendCoins(escrowAddr → attacker, evm/0xA)`.
6. The escrow account is drained. The attacker holds the tokens. When the ack arrives, the IBC module's unescrow step fails due to insufficient balance, breaking IBC accounting. [5](#0-4)
