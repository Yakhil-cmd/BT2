### Title
Unchecked Exit Precompile Return Value After Irreversible Token Burn Causes Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary
Both `EvmErc20.sol` and `EvmErc20V2.sol` — the production ERC-20 mirror contracts deployed by the Aurora Engine for every bridged NEP-141 token — burn the caller's tokens **before** calling the exit precompile via inline assembly, and **never check the return value** of that call. If the precompile call fails for any reason, the tokens are irreversibly destroyed on the EVM side while no corresponding transfer is ever issued on the NEAR side, permanently freezing the user's funds.

---

### Finding Description

In `EvmErc20.sol`, both exit functions follow the same pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, irreversibly

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is captured but NEVER checked — no require(res != 0)
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum`: [2](#0-1) 

And in both functions of `EvmErc20V2.sol`: [3](#0-2) 

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return `ExitError` in multiple reachable paths:

- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — if the caller-supplied `recipient` bytes are not valid UTF-8 or not a valid NEAR account ID.
- `ERR_TARGET_TOKEN_NOT_FOUND` — if the ERC-20 contract address has no NEP-141 mapping in storage.
- `ERR_INVALID_INPUT` — if the input length is outside `[MIN_INPUT_SIZE, MAX_INPUT_SIZE]`.
- `ERR_ETH_ATTACHED_FOR_ERC20_EXIT` — if `apparent_value != 0`. [4](#0-3) [5](#0-4) 

When any of these errors occur, the EVM `call` opcode returns `0` into `res`. Because the Solidity assembly block does not contain `if iszero(res) { revert(0,0) }` or any equivalent check, execution continues normally and the function returns successfully — **after the `_burn` has already committed**.

The engine deploys `EvmErc20.bin` / `EvmErc20V2.bin` (compiled from these exact sources) for every registered NEP-141 token: [6](#0-5) 

---

### Impact Explanation

A user who calls `withdrawToNear` or `withdrawToEthereum` with a recipient that fails precompile validation (e.g., an invalid NEAR account ID string, or a recipient whose byte length causes the total input to exceed `MAX_INPUT_SIZE = 1024`) will have their ERC-20 tokens permanently burned. No NEP-141 `ft_transfer` or `withdraw` promise is ever scheduled. The tokens cannot be recovered: the EVM balance is zero and no NEAR-side credit was issued. This constitutes **permanent freezing of user funds**.

---

### Likelihood Explanation

The `recipient` parameter in `withdrawToNear` is a raw `bytes` argument supplied directly by the caller with no on-chain validation before the burn. Any caller who passes:
- Non-UTF-8 bytes,
- A string that is not a valid NEAR account ID (e.g., uppercase letters, special characters, length > 64),
- Or a recipient string long enough to push total input past 1 024 bytes,

will silently lose their tokens. This is reachable by any unprivileged EVM user holding a balance of any bridged ERC-20 token on Aurora. No admin access or special privilege is required.

---

### Recommendation

Check the return value of the assembly `call` and revert on failure in both functions of both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the exit precompile rejects the call for any reason, the entire transaction — including the `_burn` — is rolled back, preserving the user's balance.

---

### Proof of Concept

1. Deploy an `EvmErc20` instance (as the Aurora Engine does for any bridged NEP-141 token).
2. Mint tokens to address `A`.
3. From address `A`, call `withdrawToNear(bytes("\xff\xfe"), 100)` — `\xff\xfe` is not valid UTF-8.
4. The `_burn` executes: `A`'s balance drops by 100.
5. The assembly `call` to the exit precompile returns `0` (precompile returns `ERR_INVALID_RECEIVER_ACCOUNT_ID`).
6. `res` is never checked; the function returns without reverting.
7. `A`'s ERC-20 balance is now 100 less; no NEP-141 transfer was issued; the tokens are permanently lost.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
