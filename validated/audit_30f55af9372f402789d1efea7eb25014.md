### Title
`action_delete_account` Overstates Storage Usage for Global-Contract Accounts, Permanently Blocking Deletion - (File: `runtime/runtime/src/actions.rs`)

### Summary

In `action_delete_account()`, the legacy code path (active before `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage`, protocol version 85) computes the account's non-contract storage by subtracting only the **local** contract code length. For accounts that reference a global contract (`AccountContract::Global` or `AccountContract::GlobalByAccount`), `local_contract_hash()` returns `None`, so `code_len` is 0 and the global contract identifier bytes are never subtracted. The resulting `account_storage_usage` is inflated by exactly `identifier_storage_usage()` bytes (32 for hash-based, or `account_id.len()` for account-id-based), causing the `> MAX_ACCOUNT_DELETION_STORAGE_USAGE` guard to fire incorrectly and permanently block deletion of accounts that are legitimately small enough to delete.

### Finding Description

`action_use_global_contract()` / `use_global_contract()` correctly adds the identifier length to `account.storage_usage()` when an account adopts a global contract:

```rust
// runtime/runtime/src/global_contracts.rs L97-104
account.set_storage_usage(
    account.storage_usage().checked_add(contract_identifier.len() as u64)…
);
``` [1](#0-0) 

So `account.storage_usage()` correctly includes those identifier bytes. The deletion guard is supposed to measure only the **non-contract** portion of storage. The fixed path does this correctly via `get_contract_storage_usage()`:

```rust
// Fixed path (protocol version >= 85)
let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
account_ref.storage_usage().saturating_sub(contract_storage)
``` [2](#0-1) 

But the **legacy path** (protocol version < 85) only subtracts local code:

```rust
// Legacy behavior: only subtracts local contract code, misses the
// global contract identifier overhead.
let account_storage_usage = account_ref.storage_usage();
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(),  // returns None → default hash → 0
)?;
account_storage_usage.saturating_sub(code_len)  // subtracts 0 for global contracts
``` [3](#0-2) 

`local_contract_hash()` returns `None` for `AccountContract::Global` and `AccountContract::GlobalByAccount`, so `code_len` is always 0 for these variants. The identifier bytes (32 for `Global`, `account_id.len()` for `GlobalByAccount`) remain in `account_storage_usage`, inflating it past `MAX_ACCOUNT_DELETION_STORAGE_USAGE`:

```rust
if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
    result.result = Err(ActionErrorKind::DeleteAccountWithLargeState { … }.into());
    return Ok(());
}
``` [4](#0-3) 

`MAX_ACCOUNT_DELETION_STORAGE_USAGE` is 10,000 bytes: [5](#0-4) 

`identifier_storage_usage()` returns 32 for `Global` and `account_id.len()` for `GlobalByAccount`: [6](#0-5) 

The protocol feature comment confirms this is a known bug in the legacy path: [7](#0-6) 

The fix is gated at protocol version 85: [8](#0-7) 

### Impact Explanation

Any account that:
1. Called `UseGlobalContract` (adding identifier bytes to its `storage_usage`), and
2. Has total `storage_usage` in the range `(MAX_ACCOUNT_DELETION_STORAGE_USAGE, MAX_ACCOUNT_DELETION_STORAGE_USAGE + identifier_len]`

will have its `DeleteAccount` action permanently rejected with `DeleteAccountWithLargeState`, even though its actual non-contract state is within the allowed limit. The account's balance cannot be swept to a beneficiary via deletion; the account is stuck until a protocol upgrade to version 85 activates the fix. This is a broken admission decision in the pre-inclusion transaction validation / accounting scope.

### Likelihood Explanation

The trigger is fully unprivileged: any user can submit a `UseGlobalContract` action followed by a `DeleteAccount` action. The vulnerable window is protocol versions 77 (when global contracts were introduced) through 84. The identifier overhead is small (32 bytes for hash-based), so only accounts whose non-contract storage is in the narrow band `(9968, 10000]` bytes are affected by the hash variant; the `GlobalByAccount` variant widens the band by `account_id.len()` bytes. The condition is reachable without any special privilege.

### Recommendation

The fix already exists as `get_contract_storage_usage()` and is activated by `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage`. Ensure this feature is stabilized and deployed. No additional code change is needed beyond the existing fix path.

### Proof of Concept

The existing unit test `test_delete_account_global_contract_protocol_transition` in `runtime/runtime/src/actions.rs` directly demonstrates the bug:

```rust
// Before the fix: the identifier is not subtracted, so `MAX + 32 > MAX`.
let before = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    storage,   // = MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32
    enabled - 1,  // protocol version before fix
);
expect_delete_account_too_large(&before);  // deletion incorrectly blocked
``` [9](#0-8) 

An unprivileged user reproduces this by:
1. Calling `UseGlobalContract` with a hash-based identifier → `account.storage_usage` increases by 32 bytes.
2. Ensuring the account's total `storage_usage` is in `(10000, 10032]` bytes.
3. Submitting `DeleteAccount` → the legacy path computes `account_storage_usage = storage_usage - 0 > 10000` and returns `DeleteAccountWithLargeState`, even though the true non-contract storage is ≤ 10,000 bytes.
