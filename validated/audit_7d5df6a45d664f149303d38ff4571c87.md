### Title
Omni-path `withdrawToNear` burns ERC-20 tokens with no refund when NEP-141 lacks `ft_transfer_call` and `error_refund` is disabled — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

When a user calls `withdrawToNear` with a recipient string containing a colon (e.g., `"some.near:msg"`), the ExitToNear precompile silently upgrades the NEAR-side call from `ft_transfer` to `ft_transfer_call`. If the target NEP-141 contract does not implement `ft_transfer_call`, the promise fails. Because the production build does not enable the `error_refund` feature, no callback is scheduled and no refund is issued. The ERC-20 tokens are already burned; the user loses them with no recovery path.

---

### Finding Description

**Step 1 — Burn before validation (`EvmErc20V2.sol` lines 53–63)**

`withdrawToNear` accepts arbitrary `bytes memory recipient`, burns the caller's tokens unconditionally, then forwards the raw bytes to the ExitToNear precompile. There is no validation of the recipient format before the burn. [1](#0-0) 

**Step 2 — Colon triggers Omni path (`native.rs` lines 359–379)**

`parse_recipient` splits the recipient bytes on the first `:`. Any suffix that is not `"unwrap"` is classified as `Message::Omni(msg)`. [2](#0-1) 

**Step 3 — Omni path selects `ft_transfer_call` (`native.rs` lines 611–623)**

`exit_erc20_token_to_near` matches `Some(Message::Omni(msg))` and sets `method = "ft_transfer_call"` with `transfer_near_args = None`. [3](#0-2) 

**Step 4 — No callback scheduled without `error_refund` (`native.rs` lines 449–483)**

Without the `error_refund` feature, `refund` is hardcoded to `None`. For the Omni path, `transfer_near` is also `None`. This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, so the branch at line 470 selects `PromiseArgs::Create` — a fire-and-forget promise with **no callback**. [4](#0-3) 

**Step 5 — Production build does not include `error_refund`**

`Makefile.toml` sets `CARGO_FEATURES_BUILD = "contract"` — `error_refund` is absent from the production build features. [5](#0-4) 

`engine/Cargo.toml` confirms `error_refund` is not a default feature. [6](#0-5) 

**Step 6 — Existing tests confirm the no-refund behavior**

The test `test_exit_to_near_refund` explicitly documents: *"If the refund feature is not enabled then there is no refund in the EVM."* [7](#0-6) 

---

### Impact Explanation

The user's ERC-20 tokens are burned on the Aurora EVM side. The corresponding NEP-141 tokens remain locked in Aurora's account on NEAR with no automatic recovery path. The user suffers a net loss of their bridged tokens. An operator could manually recover the NEP-141 balance, making this a **temporary** freeze rather than permanent destruction — but no on-chain mechanism exists to trigger that recovery automatically.

---

### Likelihood Explanation

- The production binary is built without `error_refund` (confirmed by `CARGO_FEATURES_BUILD = "contract"`).
- Any unprivileged token holder can call `withdrawToNear` with a colon-containing recipient — no special access is required.
- The Omni bridge flow is an advertised feature; users following Omni documentation will naturally pass colon-separated recipients.
- A NEP-141 contract that implements only `ft_transfer` (not `ft_transfer_call`) is a valid, if non-standard, deployment. The Aurora Engine makes no on-chain check that the target NEP-141 supports `ft_transfer_call` before burning.

---

### Recommendation

1. **Validate recipient format before burning.** In `withdrawToNear`, check whether the recipient contains a colon and, if so, verify the target NEP-141 supports `ft_transfer_call` before calling `_burn`. Alternatively, reject colon-containing recipients at the Solidity level unless the caller explicitly opts into the Omni path.
2. **Enable `error_refund` in the production build**, or add it to `CARGO_FEATURES_BUILD` in `Makefile.toml`, so that a failed NEAR-side promise always triggers a callback that re-mints the burned tokens.
3. **Schedule a callback unconditionally for `ft_transfer_call` promises.** The Omni path sets `transfer_near_args = None` and relies on `error_refund` for safety. A dedicated callback that handles `ft_transfer_call` failure should be scheduled regardless of the `error_refund` feature flag.

---

### Proof of Concept

```
1. Deploy a NEP-141 contract that implements ft_transfer but panics on ft_transfer_call.
2. Bridge the NEP-141 to an ERC-20 on Aurora (standard factory flow).
3. Mint ERC-20 tokens to address A.
4. From address A, call:
       erc20.withdrawToNear("some.near:omni_msg", amount)
   where "some.near" is a registered account and "omni_msg" is any non-empty string.
5. Observe:
   a. _burn(A, amount) succeeds — ERC-20 balance of A drops to zero.
   b. ExitToNear precompile parses "some.near:omni_msg" → Message::Omni("omni_msg").
   c. exit_erc20_token_to_near selects method = "ft_transfer_call".
   d. PromiseArgs::Create is emitted (no callback, because error_refund is disabled
      and transfer_near is None).
   e. The NEAR runtime executes ft_transfer_call on the NEP-141 → panic/failure.
   f. No callback fires; no refund is issued.
6. Assert: ERC-20 balance of A == 0, NEP-141 balance of A == 0,
   NEP-141 balance of Aurora unchanged (tokens stuck).
```

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
