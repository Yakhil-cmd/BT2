Audit Report

## Title
Unbounded Event Log Replay on `post_upgrade` Enables Upgrade-Blocking DoS — (`File: rs/ethereum/cketh/minter/src/lifecycle/upgrade.rs`)

## Summary
The ckETH minter (and ckBTC minter) reconstructs its entire in-memory state by replaying every event ever recorded in its stable-memory event log on every `post_upgrade`. The event log is append-only with no pruning path, and grows autonomously via timer-driven scraping regardless of user activity. Once the log is large enough that replaying it exceeds `MAX_INSTRUCTIONS_PER_INSTALL_CODE` (300 billion instructions), every future `install_code` call will trap, permanently preventing the canister from being upgraded or patched.

## Finding Description
`post_upgrade` in `rs/ethereum/cketh/minter/src/lifecycle/upgrade.rs` (L35–54) unconditionally calls `replay_events()` with no offset, snapshot, or checkpoint:

```rust
STATE.with(|cell| {
    *cell.borrow_mut() = Some(replay_events());
});
```

`replay_events()` delegates to `replay_events_internal` in `rs/ethereum/cketh/minter/src/state/audit.rs` (L185–204), which iterates from event index 0 through every event in the log:

```rust
for event in events_iter {
    apply_state_transition(&mut state, &event.payload);
}
```

The event log is a `StableLog` stored in `rs/ethereum/cketh/minter/src/storage.rs` (L37–59). `record_event` only appends; there is no trim, prune, or compaction path anywhere in the codebase. The log therefore grows monotonically for the lifetime of the canister.

The log grows autonomously: every scraping timer tick emits at least one `SyncedToBlock` event (and optionally `SyncedErc20ToBlock` / `SyncedDepositWithSubaccountToBlock`) via `apply_state_transition` (L61–71), independent of any user action. The existing benchmark fixture in `rs/ethereum/cketh/minter/src/storage.rs` (L95–100) already records 49,263 events in stable memory and measures `post_upgrade` against them, confirming the team is aware of the growth trajectory.

The ckBTC minter has an identical pattern in `rs/bitcoin/ckbtc/minter/src/lifecycle/upgrade.rs` (L91–97), with the additional overhead of running `CheckInvariantsImpl` after each replayed event, making per-event cost higher.

The hard ceiling is `MAX_INSTRUCTIONS_PER_INSTALL_CODE = NumInstructions::new(300 * B)` in `rs/config/src/subnet_config.rs` (L85). DTS allows the upgrade to span multiple rounds, but the aggregate instruction budget across all rounds is still bounded by this constant. Once the replay cost exceeds 300 billion instructions, every upgrade attempt traps unconditionally.

## Impact Explanation
If the event log grows to the point where full replay exceeds 300 billion instructions, the minter canister is permanently frozen at its current Wasm version. No security patches, parameter changes, or bug fixes can be applied. Because the ckETH and ckBTC minters are the sole custodians of bridged funds (they hold the threshold signing keys and control minting/burning), a frozen minter constitutes an application-level DoS on the entire chain-fusion bridge — matching the **High** impact category: *Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS*.

## Likelihood Explanation
Growth occurs on two independent axes. First, autonomous growth: the scraping timer emits `SyncedToBlock` (and analogous) events on every tick regardless of user activity; over years of production operation this alone produces millions of events. Second, user-accelerated growth: any non-anonymous principal can call `withdraw_eth` (ckETH) or `update_balance` (ckBTC) to append additional events; the cost to the caller is only the on-chain transaction fee. The risk materialises gradually and irreversibly — there is no mechanism to reverse log growth once it begins. The existing benchmark at 49,263 events confirms the trajectory is already being tracked internally.

## Recommendation
**Short term:** Introduce a persisted replay checkpoint. After each successful `post_upgrade`, serialize the fully-replayed state into a dedicated stable-memory region (e.g., a `StableCell`) and record the event-log offset at which the snapshot was taken. On the next upgrade, deserialize the snapshot and replay only events appended after the snapshot offset. The snapshot and offset must be written atomically to avoid partial-state corruption on a failed upgrade.

**Long term:** Add an integration test that loads a large synthetic event log (e.g., 10 M events) and asserts that `post_upgrade` completes within the instruction budget. Add a canister metric tracking the current event count and emitting an alert when it approaches a configurable threshold relative to the instruction budget.
