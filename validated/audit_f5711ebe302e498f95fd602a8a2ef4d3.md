### Title
`meta_tx_v0` Syscall Hash Omits Nonce, Enabling Unlimited Signature Replay — (`crates/blockifier/src/execution/syscalls/syscall_base.rs`, `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that does not include a nonce or any other replay-prevention field. Because the hash is a pure deterministic function of `(contract_address, entry_point_selector, calldata, chain_id)`, any signature a user produces over a `meta_tx_v0` hash remains valid forever and can be replayed an unlimited number of times by any party who captures it.

---

### Finding Description

`compute_meta_tx_v0_hash` in the OS Cairo program calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`, explicitly passing no nonce:

```cairo
// transaction_hash.cairo lines 294-315
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

The Rust blockifier mirrors this exactly: `meta_tx_v0` in `syscall_base.rs` constructs an `InvokeTransactionV0` with `max_fee: Fee(0)` and no nonce field, then calls `calculate_transaction_hash`. The resulting `new_tx_info` hardcodes `nonce: Nonce(0.into())`:

```rust
// syscall_base.rs lines 317-343
let transaction_hash = InvokeTransactionV0 {

        transaction_hash,
        version: TransactionVersion::ZERO,
        signature,
        nonce: Nonce(0.into()),   // ← always zero, never incremented
        sender_address: contract_address,
        only_query,
    },
    max_fee: Fee(0),
});
``` [2](#0-1) 

The OS `execute_meta_tx_v0` function also sets `nonce=0` in the new `TxInfo` and never reads or increments any per-account nonce for the meta-transaction: [3](#0-2) 

Compare this with how regular account transactions are protected: `handle_nonce` in `account_transaction.rs` reads the on-chain nonce, checks it, and increments it — but it explicitly skips this for v0 (`if tx_info.is_v0() { return Ok(()); }`), and `meta_tx_v0` is always v0: [4](#0-3) 

The `meta_tx_v0` syscall is callable from any contract's `__execute__` function during normal execution (it is explicitly blocked only in `validate` mode): [5](#0-4) 

---

### Impact Explanation

The `meta_tx_v0` syscall is designed to let a contract call another account's `__execute__` entry point with a caller-supplied signature, presenting a synthetic v0 transaction context to the callee. Because the hash commits only to `(contract_address, selector, calldata, chain_id)` with no nonce, any signature Alice produces over such a hash is valid for all time and for all future blocks. An attacker who captures one valid `meta_tx_v0` signature can replay it in arbitrarily many outer transactions (each with the attacker's own valid nonce), causing Alice's `__execute__` to run repeatedly without her consent. This constitutes unauthorized transaction acceptance through signature/hash logic — matching the **Critical** impact tier: *Invalid or unauthorized Starknet transaction accepted through account validation, signature, nonce, chain id, fee/resource bound, paymaster, or account-deployment logic.*

---

### Likelihood Explanation

**Medium.** The attacker must first observe a valid `meta_tx_v0` signature in the wild (e.g., from a mempool broadcast, a public transaction, or a front-running scenario). Once obtained, replay requires only submitting standard outer transactions — no privileged access, no special tooling. The `meta_tx_v0` syscall is a production feature callable by any deployed contract.

---

### Recommendation

Include a per-account, per-meta-tx nonce in the hash preimage. Concretely:

1. Add a `meta_tx_nonce` field to the `MetaTxV0Request` struct.
2. In `compute_meta_tx_v0_hash` (and its Rust counterpart), pass this nonce as `additional_data` so the hash becomes `H(prefix, version, contract_address, selector, calldata_hash, max_fee, chain_id, meta_tx_nonce)`.
3. Track and increment a per-account meta-tx nonce in `contract_state_changes` inside `execute_meta_tx_v0`, analogous to `check_and_increment_nonce` for regular transactions.
4. Alternatively, allow the account contract itself to enforce replay protection inside `__execute__` by exposing the meta-tx nonce via `get_execution_info`, but this places the burden on every account implementor and is error-prone.

---

### Proof of Concept

1. Alice deploys an account contract whose `__execute__` transfers 100 STRK to a recipient when called with `calldata = [recipient, amount]`.
2. A legitimate outer transaction calls a relay contract, which invokes `meta_tx_v0(alice, __execute__, [bob, 100], alice_sig)`. Alice signs the hash `H(INVOKE, 0, alice, __execute__, H([bob,100]), 0, chain_id)` — no nonce.
3. The transfer succeeds. The attacker observes `alice_sig` on-chain.
4. The attacker submits a new outer transaction (attacker's nonce = N+1) calling the same relay with the same `alice_sig`.
5. `compute_meta_tx_v0_hash` produces the identical hash (same inputs, no nonce), `alice_sig` verifies, and Alice's `__execute__` runs again — transferring another 100 STRK without Alice's knowledge.
6. Steps 4–5 repeat until Alice's balance is drained. Alice has no mechanism to revoke the signature.

    fn handle_nonce(
        state: &mut dyn State,
        tx_info: &TransactionInfo,
        strict: bool,
    ) -> TransactionPreValidationResult<()> {
        if tx_info.is_v0() {
            return Ok(());
        }

        let address = tx_info.sender_address();
        let account_nonce = state.get_nonce_at(address)?;
        let incoming_tx_nonce = tx_info.nonce();
        let valid_nonce = if strict {
            account_nonce == incoming_tx_nonce
        } else {
            account_nonce <= incoming_tx_nonce
        };
        if valid_nonce {
            return Ok(state.increment_nonce(address)?);
        }
        Err(TransactionPreValidationError::InvalidNonce {
            address,
            account_nonce,
            incoming_tx_nonce,
        })
    }
