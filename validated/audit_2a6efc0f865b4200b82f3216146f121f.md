### Title
Underestimated Royalty Total in WalletConnect Multi-NFT Offer Confirmation Dialog - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each NFT's royalty, causing the "Total Amount with Royalties" shown in the WalletConnect confirmation dialog to be systematically lower than the amount the wallet daemon will actually deduct. A dApp operator can exploit this to make a user approve a multi-NFT `take_offer` transaction while believing they are spending significantly less XCH/CAT than they will actually lose.

### Finding Description

`formatAmountWithRoyalties` is the function that populates `amountWithRoyalties` on each spending line of the `DisplayWalletDelta` shown in the WalletConnect `Confirm` dialog under the label **"Total Amount with Royalties"**. [1](#0-0) 

The critical lines:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

In the Chia NFT protocol, each NFT's royalty is computed on the **full** fungible amount exchanged in the offer — not on a per-NFT share of it. The wallet daemon's authoritative `nft_calculate_royalties` RPC (called via `useCalculateRoyaltiesForNFTsQuery` in the offer builder) applies each royalty percentage to the full amount. [2](#0-1) 

The GUI-side approximation in `parseCommandDisplay.ts` instead divides `amount` by `royaltyPercentages.length` before multiplying, producing a result that is `1/N` of the correct royalty total (where N = number of NFTs with royalties).

**Concrete example** (from the existing test):

| | Displayed (split) | Actual (full) |
|---|---|---|
| Amount | 0.0001 XCH | 0.0001 XCH |
| NFT1 royalty (5%) | 0.0001/2 × 5% = 0.0000025 | 0.0001 × 5% = 0.000005 |
| NFT2 royalty (0.1%) | 0.0001/2 × 0.1% = 0.00000005 | 0.0001 × 0.1% = 0.0000001 |
| **Total shown** | **0.00010255** | **0.0001051** | [3](#0-2) 

The test itself encodes the wrong expected value (`0.00010255`), confirming the bug is baked into the test suite.

The `amountWithRoyalties` value flows directly into the `Confirm` dialog's **"Total Amount with Royalties"** label: [4](#0-3) 

### Impact Explanation

A user accepting a WalletConnect `chia_wallet.take_offer` request for a bundle of N NFTs with royalties will see a "Total Amount with Royalties" that is up to N× lower than what the wallet daemon actually deducts. With high-royalty NFTs (Chia allows up to 100 % = 10 000 basis points) and a bundle of several NFTs, the gap between displayed and actual spend can be arbitrarily large. The user approves based on the displayed figure; the on-chain spend is the correct (higher) figure. This constitutes a user approving the wrong amount — a direct wallet accounting impact.

### Likelihood Explanation

Any dApp connected via WalletConnect can issue a `take_offer` call. The attacker only needs to:
1. Mint NFTs with elevated royalty percentages (permissionless on Chia).
2. Create an offer bundling ≥ 2 such NFTs against XCH/CAT.
3. Deliver the offer string to the victim's WalletConnect-connected dApp.

No leaked keys, host compromise, or cryptographic break is required.

### Recommendation

Remove the `/ BigInt(royaltyPercentages.length)` division. Each royalty percentage should be applied to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test expectation to reflect the corrected value.

### Proof of Concept

Using the existing test fixture (two NFTs, royalties 500 bp and 10 bp, amount = 100 000 000 mojos):

- **Current (buggy) display**: `0.00010255 XCH`
- **Correct display**: `0.0001051 XCH`
- **Discrepancy**: ~2.4 % understatement with only 0.1 % + 5 % royalties and 2 NFTs; scales linearly with N and royalty magnitude. [5](#0-4)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
