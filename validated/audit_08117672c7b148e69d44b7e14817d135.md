### Title
Silo Whitelist Bypass via Unrestricted `call` and `deploy_code` Methods Allows Unauthorized EVM Execution — (File: `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

The public NEAR contract methods `call` and `deploy_code` execute EVM transactions without enforcing the Silo whitelist checks that `submit` and `submit_with_args` enforce. Any unprivileged NEAR account can invoke `call` (or `deploy_code`) to execute arbitrary EVM calls or deploy contracts as their derived EVM address, completely bypassing the access-control mechanism that Silo mode is designed to enforce.

---

### Finding Description

Aurora Engine exposes four EVM-execution entry points as public NEAR contract methods:

| Method | Silo whitelist enforced? | Gas charged? | Nonce checked? | Signature verified? |
|---|---|---|---|---|
| `submit` | **Yes** | **Yes** | **Yes** | **Yes** |
| `submit_with_args` | **Yes** | **Yes** | **Yes** | **Yes** |
| `call` | **No** | **No** | **No** | **No** |
| `deploy_code` | **No** | **No** | **No** | **No** |

`submit` and `submit_with_args` both delegate to `engine::submit_with_alt_modexp`, which calls `assert_access` before any EVM execution: [1](#0-0) 

`assert_access` checks both the NEAR predecessor account ID and the EVM sender address against the Silo whitelist: [2](#0-1) 

`submit_with_alt_modexp` also enforces nonce correctness, gas payment, chain-ID validation, and EIP-3607: [3](#0-2) 

By contrast, `call` in `evm_transactions.rs` constructs an `Engine` directly and invokes `call_with_args` with **none** of those checks: [4](#0-3) 

`deploy_code` has the identical omission: [5](#0-4) 

Both are exposed as unrestricted public `extern "C"` NEAR contract methods with no caller restriction: [6](#0-5) 

---

### Impact Explanation

In Silo mode, the whitelist (`WhitelistKind::Account` for NEAR callers, `WhitelistKind::Address` for EVM senders) is the sole mechanism restricting which accounts may interact with the EVM. By calling `call` instead of `submit`, any NEAR account that is **not** in the whitelist can:

1. Execute arbitrary EVM calls as their derived EVM address (`predecessor_address`) against any deployed contract.
2. Deploy arbitrary EVM contracts, bypassing `is_allow_deploy`.
3. Do both without paying any ETH gas fees (no `charge_gas` call), breaking gas-accounting invariants.

If a Silo deployment holds user funds in EVM contracts that rely on the Silo whitelist as their access-control boundary (rather than implementing independent on-chain ACL), an attacker can call withdrawal or transfer functions on those contracts without authorization, constituting **direct theft of user funds**.

---

### Likelihood Explanation

- Silo mode is a production feature of Aurora Engine, actively used for permissioned deployments.
- The `call` NEAR method is publicly callable by any NEAR account with no prerequisite.
- No special knowledge, leaked keys, or governance capture is required — only a standard NEAR account and knowledge of the target contract's ABI.
- The bypass is trivially discoverable by inspecting the on-chain contract ABI.

---

### Recommendation

Add a Silo whitelist check inside `call` and `deploy_code` in `engine/src/contract_methods/evm_transactions.rs`, mirroring the `assert_access` call that `submit_with_alt_modexp` performs. Specifically, after deriving `predecessor_account_id` and the EVM origin address, call `silo::is_allow_submit` / `silo::is_allow_deploy` and return `EngineErrorKind::NotAllowed` if the check fails — exactly as `assert_access` does in `engine/src/engine.rs` lines 1756–1775.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode. Enable the `Account` and `Address` whitelists. Add only `alice.near` / Alice's EVM address to the whitelist.
2. Deploy a simple vault EVM contract (holds ETH, exposes `withdraw(address recipient)` callable by anyone).
3. Fund the vault with ETH via Alice's whitelisted `submit` call.
4. As `bob.near` (not whitelisted), attempt `submit` with a signed transaction targeting `withdraw` → rejected with `NotAllowed`.
5. As `bob.near`, call the NEAR method `call` with `CallArgs::V2 { contract: vault_address, value: 0, input: abi_encode("withdraw", bob_evm_address) }`.
6. The `call` method skips `assert_access`, executes the EVM call as Bob's derived EVM address, and the vault transfers ETH to Bob — **whitelist bypassed, funds stolen**.

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

#[named]
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}
