### Title
Stale `eligible_operational_addresses` Entry Persists After `change_operational_address` — (File: `src/staking/staking.cairo`)

---

### Summary

The `eligible_operational_addresses` map, which records an operational address's consent to be associated with a specific staker, is **never cleared** after `change_operational_address` is successfully executed. This leaves a persistent "pre-authorization" that the staker can exploit at any future time to re-associate with the operational address without requiring fresh consent, analogous to the whitelisting-persistence bug in the referenced report.

---

### Finding Description

The protocol implements a two-step process for changing a staker's operational address:

1. **Step 1 – Consent**: The operational address calls `declare_operational_address(staker_address)`, which writes `eligible_operational_addresses[operational_address] = staker_address`.
2. **Step 2 – Confirmation**: The staker calls `change_operational_address(operational_address)`, which reads and validates the consent entry, then updates the staker's operational address.

In `change_operational_address`, after the consent is consumed and the operational address is activated, the entry in `eligible_operational_addresses` is **never zeroed out**:

```cairo
// src/staking/staking.cairo lines 683–694
assert!(
    self.eligible_operational_addresses.read(operational_address) == staker_address,
    "{}",
    Error::OPERATIONAL_NOT_ELIGIBLE,
);

let old_address = staker_info.operational_address;
self.operational_address_to_staker_address.write(old_address, Zero::zero());
staker_info.operational_address = operational_address;
self.write_staker_info(:staker_address, :staker_info);
self.operational_address_to_staker_address.write(operational_address, staker_address);
// ← eligible_operational_addresses[operational_address] is NEVER cleared
``` [1](#0-0) 

The `declare_operational_address` function writes the consent entry:

```cairo
// src/staking/staking.cairo lines 705–722
fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
    ...
    self.eligible_operational_addresses.write(operational_address, staker_address);
    ...
}
``` [2](#0-1) 

The storage field is declared as a persistent map with no TTL or cleanup mechanism:

```cairo
// src/staking/staking.cairo line 126
eligible_operational_addresses: Map<ContractAddress, ContractAddress>,
``` [3](#0-2) 

---

### Impact Explanation

**Medium — Griefing with damage to users.**

After a staker uses `change_operational_address(OA2)` and later switches away to `OA3` (freeing `OA2`), the stale entry `eligible_operational_addresses[OA2] = staker_A` remains. The staker can immediately call `change_operational_address(OA2)` again — bypassing the requirement for `OA2` to issue fresh consent — and re-occupy `OA2` indefinitely.

While `OA2` is occupied by staker A, `declare_operational_address` reverts with `OPERATIONAL_IN_USE`:

```cairo
// src/staking/staking.cairo lines 708–711
assert!(
    self.operational_address_to_staker_address.read(operational_address).is_zero(),
    "{}",
    Error::OPERATIONAL_IN_USE,
);
``` [4](#0-3) 

This means `OA2` **cannot revoke its stale consent** while it is occupied, and cannot be claimed by any other staker. A malicious staker can cycle between two operational addresses to permanently block a target operational address from being used by any other staker, with no cost beyond gas.

---

### Likelihood Explanation

**Medium.** The attack requires only that a staker previously used an operational address and then changed away from it. No privileged access, leaked keys, or external dependencies are needed. Any staker who has ever called `change_operational_address` is in a position to exploit the stale entry. The operational address has no on-chain mechanism to force revocation while it is occupied.

---

### Recommendation

Clear the `eligible_operational_addresses` entry inside `change_operational_address` immediately after the consent check passes:

```cairo
// After the assert, add:
self.eligible_operational_addresses.write(operational_address, Zero::zero());
```

This ensures the authorization is single-use (transient), directly mirroring the fix applied in the referenced report (replacing persistent permissions with transient ones that expire after each use).

---

### Proof of Concept

1. `staker_A` stakes and has operational address `OA1`.
2. `OA2` calls `declare_operational_address(staker_A)` → `eligible_operational_addresses[OA2] = staker_A`.
3. `staker_A` calls `change_operational_address(OA2)` → `OA2` is now active; `eligible_operational_addresses[OA2]` remains `staker_A`.
4. `OA3` calls `declare_operational_address(staker_A)`.
5. `staker_A` calls `change_operational_address(OA3)` → `OA2` is freed; `eligible_operational_addresses[OA2]` still = `staker_A`.
6. `staker_A` immediately calls `change_operational_address(OA2)` — the stale entry satisfies the `OPERATIONAL_NOT_ELIGIBLE` check, so it succeeds **without `OA2` issuing any new consent**.
7. `OA2` is now occupied again. Any attempt by `OA2` to call `declare_operational_address` for a different staker reverts with `OPERATIONAL_IN_USE`.
8. Repeat steps 5–7 indefinitely to permanently deny `OA2` to any other staker.
