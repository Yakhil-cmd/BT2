The vulnerability is real. Here is the analysis:

---

### Title
Residual ETH from over-funded `exactInputSingle` call is stealable by any caller via `refundETH()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`refundETH()` sends the router's **entire** native ETH balance to `msg.sender` with no per-user accounting. The `pay()` helper deposits only the exact `value` needed (not the full `msg.value`), leaving residual ETH on the router after the swap transaction ends. Any subsequent caller can drain that residual.

### Finding Description

`exactInputSingle` is `external payable`. [1](#0-0) 

Inside the swap callback, `pay()` is called with `value = amountIn`. When `nativeBalance >= value`, it deposits only `value` as WETH — **not** the full native balance — leaving `msg.value - amountIn` as residual ETH on the router after the transaction completes: [2](#0-1) 

`refundETH()` then sends **all** of `address(this).balance` to whoever calls it, with no check that the caller is the one who deposited the ETH: [3](#0-2) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does **not** prevent ETH from arriving via `msg.value` on payable functions, so residual ETH can accumulate between transactions: [4](#0-3) 

The intended safe pattern (shown in the test suite) is to bundle `exactInputSingle` + `refundETH()` in a single `multicall`: [5](#0-4) 

But `exactInputSingle` is independently callable with `msg.value > amountIn`, and nothing forces the caller to include `refundETH()` in the same atomic multicall.

### Impact Explanation

Direct loss of user principal (ETH). User A sends `msg.value = X` with `amountIn = Y < X`; after the swap, `X - Y` ETH sits on the router. User B calls `refundETH()` and receives `X - Y` ETH. User A's subsequent `refundETH()` call returns 0. This is a straightforward theft of funds with no trust assumptions required.

### Likelihood Explanation

Any user who calls `exactInputSingle` directly (not via multicall) with excess ETH — a natural pattern for users who want to ensure their swap succeeds — is vulnerable. A MEV bot or attacker monitoring the mempool can back-run the swap transaction with a `refundETH()` call.

### Recommendation

Either:
1. Track per-caller ETH deposits in transient storage and restrict `refundETH()` to return only the caller's own deposited amount, or
2. Enforce that `refundETH()` can only be called from within a `multicall` context (e.g., via a reentrancy-style flag set in `multicall`), or
3. In `pay()`, deposit the **full** `nativeBalance` (not just `value`) when `nativeBalance >= value`, so no residual ETH ever remains on the router after a swap.

### Proof of Concept

```
1. User A calls router.exactInputSingle{value: 2 ether}(..., amountIn: 1 ether, ...)
   → pay() deposits 1 ether as WETH; 1 ether residual stays on router
2. User B calls router.refundETH()
   → receives 1 ether (User A's residual)
3. User A calls router.refundETH()
   → receives 0 (balance already drained)
```

The test `test_refundETH_sendsBalanceToCaller` already demonstrates that any caller receives the full router balance: [6](#0-5)
