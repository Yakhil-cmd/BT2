### Title
Malicious Coordinator Extracts Secret Key via Split-View Attack: `presign()` Does Not Enforce N1 = 2t+1, Allowing Presignature Reuse Across Signing Sessions — (File: `src/ecdsa/robust_ecdsa/presign.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The robust ECDSA `presign()` function places no upper-bound constraint on the number of presigning participants (N1). The `sign()` function enforces N2 = 2·max\_malicious+1 but cannot verify N1 = N2 because `RerandomizedPresignOutput` carries no record of N1. A malicious coordinator can therefore run presigning with N1 > 2t+1 participants and then drive two independent signing sessions over different subsets of exactly 2t+1 participants, each signing a different message hash, using shares derived from the same presignature. The resulting pair of signatures exposes multiplicatively related nonces, enabling full secret-key extraction via standard ECDSA nonce-reuse algebra.

---

### Finding Description

**Root cause — `presign()` does not enforce N1 = 2t+1.**

`src/ecdsa/robust_ecdsa/presign.rs` accepts any participant list whose size is at least `threshold`: [1](#0-0) 

The function signature receives `participants` and a threshold but never receives `max_malicious`, so it cannot enforce `participants.len() == 2*max_malicious+1`. No upper-bound check exists.

**Partial mitigation in `sign()` is insufficient.**

`src/ecdsa/robust_ecdsa/sign.rs` does enforce N2 = 2t+1 and rejects `msg_hash == 0`: [2](#0-1) 

The inline comment reads *"The next two conditions prevent split-view attacks"*, but this claim is only true when N1 = N2. When N1 > 2t+1, the coordinator can select two distinct subsets of size 2t+1 from the N1 presigning participants, run a separate signing session on each subset with a different message hash, and the two conditions in `sign()` are satisfied by both sessions individually.

**`RerandomizedPresignOutput` carries no N1 record.**

Because the rerandomized presignature struct does not store the presigning participant count, `sign()` has no way to retroactively verify N1 = N2, even if it wanted to.

**Documentation acknowledges the requirement but the code does not enforce it.** [3](#0-2) 

The documentation explicitly states that N1 = N2 = 2t+1 is required to prevent split-view attacks and that the novel linearization-based variant needs only 2t+2 presigning participants and two signing sessions. The code enforces only half of this invariant.

---

### Impact Explanation

When the coordinator runs presigning with N1 = 2t+2 participants and then drives two signing sessions:

- Session 1 uses subset S1 (size 2t+1), message h1 → rerandomization delta1 = HKDF(X, ε1, h1, R\_raw, ρ1), effective nonce k1 = k\_raw · delta1⁻¹.
- Session 2 uses subset S2 (size 2t+1), message h2 → rerandomization delta2 = HKDF(X, ε2, h2, R\_raw, ρ2), effective nonce k2 = k\_raw · delta2⁻¹.

Because delta1 and delta2 are derived from public inputs, the coordinator knows the ratio k1/k2 = delta2/delta1. Combined with the two ECDSA equations, the secret key x is algebraically recoverable. This constitutes **Critical** impact: extraction and reconstruction of the aggregate secret signing key.

---

### Likelihood Explanation

The coordinator is a standard participant role, reachable without any privileged external assumption. The attack requires only that the coordinator initiates presigning with one extra participant (N1 = 2t+2 instead of 2t+1) and then runs two normal signing sessions. Both signing sessions individually pass all validation checks in `sign()`. No cryptographic primitive break, no leaked key, and no external dependency is required. Likelihood is **Medium**: the coordinator must be malicious, but the exploit path is straightforward and requires no special capability beyond controlling protocol initiation.

---

### Recommendation

1. **Enforce N1 = 2t+1 in `presign()`**: Accept `max_malicious` as a parameter and return `InitializationError` if `participants.len() != 2*max_malicious+1`, mirroring the check already present in `sign()`.

2. **Bind N1 into `PresignOutput` / `RerandomizedPresignOutput`**: Store the presigning participant count (or the full participant list hash) in the output struct so that `sign()` can assert N1 = N2 at signing time.

3. **Correct the misleading comment**: The comment *"The next two conditions prevent split-view attacks"* in `sign()` should be updated to clarify that the conditions are necessary but not sufficient without the corresponding N1 = 2t+1 enforcement in `presign()`.

### Proof of Concept

```
Setup: threshold t=1, max_malicious=1, so 2t+1 = 3.
Participants: {P1, P2, P3, P4}  (N1 = 4 > 3)

Step 1: Coordinator calls presign({P1,P2,P3,P4}, ...) — accepted, no upper-bound check.

Step 2: Coordinator calls sign({P1,P2,P3}, coordinator=P1, max_malicious=1, msg_hash=h1)
        → N2=3 == 2*1+1 ✓, h1≠0 ✓ → session proceeds, produces (R1, s1).

Step 3: Coordinator calls sign({P2,P3,P4}, coordinator=P2, max_malicious=1, msg_hash=h2)
        → N2=3 == 2*1+1 ✓, h2≠0 ✓ → session proceeds, produces (R2, s2).

Step 4: Coordinator computes delta1, delta2 from public HKDF inputs.
        Derives k1/k2 = delta2/delta1.
        Solves the two ECDSA equations for secret key x.
```

Both signing calls satisfy every check in `sign()`. The presigning call satisfies every check in `presign()`. No honest participant can detect the attack. [4](#0-3) [5](#0-4) [6](#0-5)

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

4. **Do not sign with $h = 0$** (the zero message hash).
   This input enables a related algebraic split-view attack in the modified scheme when
   $N_1 > 2t + 1$.
