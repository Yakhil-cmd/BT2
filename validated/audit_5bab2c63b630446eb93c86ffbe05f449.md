### Title
COA Ownership Proof Signature Replay Across Multiple COAs of the Same Flow Account - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof()` in `contract.cdc` verifies that a Flow account's keys signed some `signedData`, then checks that the account owns a COA at the given `path` whose EVM address matches `evmAddress`. However, the function never enforces that `signedData` commits to `evmAddress`. A signature produced to prove ownership of COA-A can be replayed verbatim to prove ownership of COA-B on the same Flow account, because the signed bytes do not bind to any specific COA address. The code itself acknowledges this gap in a comment but dismisses it as "low-risk" based on a historical assumption that is not a protocol invariant.

### Finding Description

`EVM.validateCOAOwnershipProof()` accepts `signedData`, `keyIndices`, `signatures`, and `evmAddress` as separate parameters. The cryptographic check verifies the signatures only over `signedData`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

After this passes, the function separately checks that the COA stored at `path` has an address matching `evmAddress`. The `evmAddress` is never part of what was signed. The function's own comment explicitly states:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The comment dismisses this as low-risk because "Flow wallets historically create at most one COA per account." This is a social/historical assumption, not a protocol-enforced invariant. The Flow protocol permits any account to create and store multiple COAs at different public paths.

The `COA.sol` ERC-1271 implementation calls the precompile directly:

```solidity
cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig
    )
);
```

Here `address(this)` is the calling COA's EVM address, and `_hash` is the message hash. The `_hash` does not include `address(this)`. An attacker who observes a valid `(_hash, _sig)` pair used to authorize an action on COA-A can replay it against COA-B of the same Flow account, because:

1. The signature over `_hash` is cryptographically valid (same keys, same data).
2. COA-B exists at its own path and its address matches the `evmAddress` argument the attacker supplies.
3. `validateCOAOwnershipProof` returns `isValid: true`.

**Exploit path:**

1. Victim Flow account `0xALICE` has two COAs: COA-A (`0xEVM_A`) at `/public/coaA` and COA-B (`0xEVM_B`) at `/public/coaB`.
2. An EVM DeFi protocol calls `COA-A.isValidSignature(_hash, _sig)` to verify Alice authorized an action on COA-A. Alice signs `_hash` and the call succeeds.
3. Attacker observes `_hash` and `_sig` on-chain (they are public inputs to the EVM transaction).
4. Attacker calls `COA-B.isValidSignature(_hash, _sig)` (or directly calls `verifyCOAOwnershipProof(0xEVM_B, _hash, proof_with_sig)` with a crafted proof pointing to `/public/coaB`).
5. `validateCOAOwnershipProof` verifies the signature over `_hash` (valid), finds COA-B at `/public/coaB`, confirms its address is `0xEVM_B`, and returns `isValid: true`.
6. The DeFi protocol now believes Alice authorized the same action on COA-B, which she never intended.

### Impact Explanation

An unprivileged attacker can forge a valid ERC-1271 ownership proof for any COA belonging to a Flow account that has ever produced a COA ownership proof for any other COA on the same account. This enables unauthorized authorization of EVM actions — including token approvals, trades, governance votes, or any protocol action gated by ERC-1271 — on a COA the victim never intended to authorize. The impact is unauthorized account mutation and potential asset loss on any EVM protocol that relies on `verifyCOAOwnershipProof` / `isValidSignature` for access control.

### Likelihood Explanation

The attack requires a victim Flow account to have at least two COAs. While uncommon today, the protocol places no restriction on this. As Flow EVM adoption grows and protocols encourage users to create multiple COAs (e.g., one per application), the population of vulnerable accounts increases. The attacker needs only to observe a prior valid proof on-chain, which is trivially achievable since all EVM calldata is public. No privileged access is required.

### Recommendation

`validateCOAOwnershipProof` should enforce that `signedData` cryptographically commits to `evmAddress`. The simplest fix is to require callers to include `evmAddress` in the signed payload and verify this inside the function, or to hash `evmAddress` into the domain separation tag before calling `keyList.verify`. The EVM-side precompile path via `coa.sol` should pass `abi.encode(address(this), _hash)` as the signed data rather than `_hash` alone.

### Proof of Concept

**Setup:** Alice's Flow account `0xALICE` has two COAs:
- COA-A at EVM address `0xEVM_A`, capability published at `/public/coaA`
- COA-B at EVM address `0xEVM_B`, capability published at `/public/coaB`

**Step 1 — Legitimate proof for COA-A:**
Alice signs `signedData = keccak256("authorize_action")` with her Flow key. A valid `COAOwnershipProof` is constructed pointing to `/public/coaA` with Alice's signature. `validateCOAOwnershipProof(0xALICE, /public/coaA, signedData, [0], [sig], 0xEVM_A)` returns `isValid: true`.

**Step 2 — Replay against COA-B:**
Attacker constructs a new `COAOwnershipProof` with the same `sig` but pointing to `/public/coaB`. Calls `validateCOAOwnershipProof(0xALICE, /public/coaB, signedData, [0], [sig], 0xEVM_B)`.

- `keyList.verify(signatureSet, signedData, "FLOW-V0.0-user")` → `true` (same key, same data)
- `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coaB)` → succeeds
- `coaRef.address().bytes == 0xEVM_B` → `true`
- Returns `ValidationResult(isValid: true, problem: nil)` ✓

The attacker has successfully replayed Alice's signature to prove ownership of COA-B, which Alice never authorized.
