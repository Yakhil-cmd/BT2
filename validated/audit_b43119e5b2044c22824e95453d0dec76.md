### Title
Single Private Key Bypasses Weight Threshold via Duplicate Key Registration — (`fvm/environment/account_public_key_util.go`, `fvm/transactionVerifier.go`)

### Summary
An account owner can register the same ECDSA public key material under multiple key indices with sub-threshold weights. Because `getAccountKeysAndAggregateWeights` aggregates weights by `(address, keyIndex)` without checking whether two key indices resolve to the same underlying stored key, a single private key can satisfy the 1000-weight threshold by signing with multiple indices that share the same key bytes.

### Finding Description
The deduplication system in `AppendAccountPublicKeyMetadata` (`fvm/environment/accounts_status.go`) is a **storage optimization**, not a security control. When key1 is added with the same bytes as key0, `FindDuplicateKey` detects the duplicate and calls `AppendDuplicateKeyMetadata`, which:
- Stores key1's weight (e.g., 600) independently in the weight-and-revoked list
- Maps key1's `storedKeyIndex` to 0 (key0's stored bytes)
- **Does not reject the key or return an error**

When `GetRuntimeAccountPublicKey` is later called for key1, it returns `Weight=600` with key0's public key bytes. `getAccountKeysAndAggregateWeights` then adds both weights (600+600=1200) without any check that the two key indices resolve to identical stored key material. `verifySignatures` verifies both signatures against the same public key — both pass since the same private key produced them.

`newSignatureEntries` only deduplicates by `(address, keyIndex)` pair, so signatures for indices 0 and 1 are treated as independent.

### Impact Explanation
A single private key holder can satisfy the 1000-weight authorization threshold on an account by registering the same key material under multiple indices with sub-threshold weights. This violates the core invariant that weight=1000 requires independent key holders. An attacker can create an account that appears to be a multi-sig (multiple keys, each weight < 1000) but is fully controlled by one private key, then authorize arbitrary transactions on that account — including deploying contracts, draining resources, or modifying capabilities — using a single key.

### Likelihood Explanation
The attack requires only standard transaction submission (`auth(AddKey)` or account creation with duplicate keys in the initial key list). No admin access, leaked keys, or staked node control is needed. The attacker only needs to own the account being attacked (or create a new one). The path is directly reachable on unmodified emulator/localnet.

### Recommendation
In `getAccountKeysAndAggregateWeights`, after resolving each signature's account key, check whether the resolved `storedKeyIndex` (or the public key bytes themselves) has already been counted for that address. If so, skip aggregating its weight. Alternatively, `AppendAccountPublicKeyMetadata` should reject duplicate key material outright (return an error) rather than silently accepting it as a storage-deduplicated entry.

### Proof of Concept
```
// 1. Create account A with key0 (ECDSA_P256/SHA3_256, weight=600)
// 2. From account A (using key0 + any co-signer to reach 1000), add key1:
//    account.keys.add(publicKey: key0.publicKey, hashAlgorithm: HashAlgorithm.SHA3_256, weight: 600.0)
//    -- OR create account A directly with both keys in the initial key list --
// 3. Build a transaction with authorizer=A, proposer=(A, keyIndex=0)
// 4. Sign payload with (A, keyIndex=0) using privateKey
// 5. Sign payload with (A, keyIndex=1) using the SAME privateKey
// 6. Sign envelope with (A, keyIndex=0) using privateKey
// 7. Submit transaction
// Result: getAccountKeysAndAggregateWeights sums 600+600=1200 >= 1000,
//         verifySignatures passes both (same public key bytes),
//         transaction executes despite threshold requiring independent keys.
```
