### Title
Staker Can Increase Commission Without Checkpointing Rewards, Stealing Delegator Yield - (File: src/staking/staking.cairo)

### Summary
The `set_commission` function updates the staker's commission rate without first distributing pending rewards. When a staker holds an active commission commitment, they can increase their commission to `max_commission` immediately before reward distribution. Because the reward-split logic reads the commission from storage at distribution time, the new higher commission retroactively applies to rewards accrued during the period when the old lower commission was in effect, stealing yield from delegators.

### Finding Description
`set_commission` (staking.cairo:725–743) writes the new commission directly to storage with no reward checkpoint:

```cairo
fn set_commission(ref self: ContractState, commission: Commission) {
    self.general_prerequisites();
    assert!(commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
    let staker_address = get_caller_address();
    let staker_info = self.internal_staker_info(:staker_address);
    assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);

    let staker_pool_info = self.staker_pool_info.entry(staker_address);
    if let Option::Some(old_commission) = staker_pool_info.commission.read() {
        self.update_commission(:staker_address, :staker_pool_info, :old_commission, :commission);
    } else {
        staker_pool_info.commission.write(Option::Some(commission));
        self.emit(Events::CommissionInitialized { staker_address, commission });
    }
}
``` [1](#0-0) 

The internal `update_commission` helper also only writes to storage and emits an event — no reward distribution occurs: [2](#0-1) 

When rewards are later distributed (either via `update_rewards_from_attestation_contract` in V2 or `update_rewards` in V3), `calculate_staker_pools_rewards` reads the commission from storage **at that moment**:

```cairo
let commission = staker_pool_info.commission();
``` [3](#0-2) 

This commission is then passed directly to `split_rewards_with_commission`, which determines how much of the pool's accrued rewards go to the staker vs. delegators: [4](#0-3) 

The commission commitment mechanism (`set_commission_commitment`) explicitly allows a staker to change commission to **any value up to `max_commission`**, including values **higher** than the current commission, as long as a commitment is active:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission,
        "{}",
        Error::INVALID_COMMISSION_WITH_COMMITMENT,
    );
    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
``` [5](#0-4) 

The commitment itself only requires `max_commission >= current_commission`, so a staker can set `max_commission = 10000` (100%) while currently at 0%: [6](#0-5) 

### Impact Explanation
**High — Theft of unclaimed yield from delegators.**

Delegators who joined a pool at commission 0% have their accrued epoch/block rewards retroactively taxed at the new higher commission rate. In the worst case (commission raised to 100% just before attestation), delegators receive **zero** rewards for the entire epoch despite having staked throughout it. The staker captures all pool rewards as commission. This is repeatable: the staker can lower commission again after claiming, then repeat the attack in the next epoch.

### Likelihood Explanation
**Medium.**

The attack requires the staker to have an active commission commitment. However, the staker fully controls:
1. When to call `set_commission_commitment` (setting a high `max_commission`).
2. In V2 (pre-consensus, attestation-based rewards): **when to call `attest`**, which triggers `update_rewards_from_attestation_contract`. The staker can delay attestation to accumulate a larger reward pool, then increase commission immediately before attesting.
3. In V3 (consensus rewards): the window is per-block (sequencer calls `update_rewards`), but the staker can still submit `set_commission` in the same block before the sequencer's call is processed.

The V2 path is the higher-risk scenario because the staker controls the attestation timing entirely.

### Recommendation
Before writing the new commission to storage in `set_commission`, trigger a reward distribution for the current period so that rewards accrued at the old commission rate are settled first. Analogous to the Munchables fix:

```diff
fn set_commission(ref self: ContractState, commission: Commission) {
    self.general_prerequisites();
    assert!(commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
    let staker_address = get_caller_address();
    let staker_info = self.internal_staker_info(:staker_address);
    assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
+   // Distribute pending rewards at the current commission before changing it.
+   self._distribute_pending_rewards(staker_address);
    let staker_pool_info = self.staker_pool_info.entry(staker_address);
    ...
}
```

Alternatively, record the commission at the time rewards are accrued (e.g., store it alongside each reward epoch checkpoint) rather than reading the live commission at distribution time.

### Proof of Concept
1. Staker stakes with `commission = 0` and calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
2. Delegators enter the pool, expecting 0% commission.
3. Rewards accrue over the epoch (pool balance grows).
4. Just before calling `attest` (V2), staker calls `set_commission(10000)` — this succeeds because `10000 <= max_commission` and `10000 != 0`.
5. Staker calls `attest`; the attestation contract calls `update_rewards_from_attestation_contract`.
6. Inside `calculate_staker_pools_rewards`, `commission = 10000` is read from storage.
7. `split_rewards_with_commission` assigns 100% of pool rewards to the staker as commission; delegators receive 0.
8. Staker calls `set_commission(0)` to reset for the next epoch and repeats.

        fn update_commission(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
            old_commission: Commission,
            commission: Commission,
        ) {
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                if self.is_commission_commitment_active(:commission_commitment) {
                    assert!(
                        commission <= commission_commitment.max_commission,
                        "{}",
                        Error::INVALID_COMMISSION_WITH_COMMITMENT,
                    );
                    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
                } else {
                    assert!(
                        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
                    );
                }
            } else {
                assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
            }

            // Update commission in storage.
            staker_pool_info.commission.write(Option::Some(commission));

            // Emit event.
            self
                .emit(
                    Events::CommissionChanged {
                        staker_address, old_commission, new_commission: commission,
                    },
                );
        }
