### Title
Gateway Skips Signature Validation for Invoke Transactions When Any Account Transaction Exists in Mempool, Not Just `deploy_account` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations()` in the gateway's stateful validator skips the blockifier's `__validate__` entry point (the only place where the account contract verifies the transaction signature) for any invoke transaction with `nonce=1` when `account_nonce=0` and `account_tx_in_pool_or_recent_block()` returns `true`. The guard `account_tx_in_pool_or_recent_block()` returns `true` for **any** transaction type in the mempool or recent block, not exclusively for a `deploy_account` transaction. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can submit a malicious invoke with `nonce=1` for the same address, bypass signature verification entirely, and have that transaction admitted to the mempool and subsequently executed from the victim's account.

### Finding Description

The UX feature for simultaneous `deploy_account + invoke` submission is implemented in `skip_stateful_validations()`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}

The `__validate__` entry point is the **only** place where the account contract verifies the transaction signature. Skipping it means the signature is never checked.

The guard `account_tx_in_pool_or_recent_block()` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

`MempoolState::contains_account` checks only whether the address appears in `staged` or `committed` maps, with no filter on transaction type:

```rust

``` [5](#0-4) 

`TransactionPool::contains_account` similarly checks only address presence, not transaction type: [6](#0-5) 

The code comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect: the check returns `true` for **any** transaction type (invoke, declare, deploy_account) in the pool or recent block.

**Attack path:**

1. Victim submits a `deploy_account` for address `X` (nonce=0). This enters the mempool; `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. Attacker observes the mempool (via RPC), sees the `deploy_account` for `X`.
3. Attacker crafts an invoke transaction for `X` with `nonce=1`, arbitrary `calldata`, and an invalid/attacker-chosen signature.
4. Gateway stateful validation:
   - `validate_state_preconditions`: passes — `nonce=1` is within `[0, 200]`.
   - `validate_by_mempool`: passes — `nonce=1 >= account_nonce=0`.
   - `skip_stateful_validations`: returns `true` — `tx.nonce()==1`, `account_nonce==0`, and `account_tx_in_pool_or_recent_block(X)==true`.
   - `run_validate_entry_point` with `skip_validate=true`: `__validate__` is **never called**; signature is **never verified**.
5. Attacker's invoke is admitted to the mempool.
6. Attacker sets a higher tip than the victim's invoke (if any) to ensure priority.
7. Block execution: `deploy_account` runs (nonce=0, deploys `X`), then attacker's invoke runs (nonce=1) — executing arbitrary calls from `X`'s account without the victim's private key.

The stateless validator only checks signature **size**, not validity: [7](#0-6) 

No other production path verifies the signature before execution.

### Impact Explanation

An unprivileged attacker can submit an invoke transaction with an invalid or attacker-chosen signature for any account that has a `deploy_account` pending in the mempool. The transaction bypasses the only signature-verification step (`__validate__`) and is admitted to the mempool. Upon execution, the attacker's `calldata` runs through the victim's `__execute__` function, enabling arbitrary calls from the victim's account — including token transfers, approvals, or any other privileged action — without knowledge of the victim's private key.

This matches: **Critical. Invalid or unauthorized Starknet transaction accepted through account validation, signature, nonce, chain id, fee/resource bound, paymaster, or account-deployment logic.**

### Likelihood Explanation

The attack window is the period between a victim's `deploy_account` entering the mempool and being executed. Mempool contents are observable via RPC. The attacker needs no privileged access, only the ability to submit transactions. New account deployments are a routine operation. The attack is straightforward to automate.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address)` that filters by transaction type, or `skip_stateful_validations` should inspect the transaction type of the existing mempool entry before deciding to skip validation.

### Proof of Concept

```
1. Victim submits DeployAccount for address X (nonce=0, valid signature).
   → Mempool accepts it; account_tx_in_pool_or_recent_block(X) == true.

2. Attacker submits Invoke for address X:
     nonce = 1
     calldata = [transfer_all_tokens_to_attacker]
     signature = [0x0, 0x0]  // invalid / arbitrary

3. Gateway stateful validation for attacker's invoke:
   a. validate_state_preconditions:
      - validate_nonce: account_nonce=0, tx_nonce=1 → 0 <= 1 <= 200 → PASS
   b. validate_by_mempool:
      - MempoolState::validate_incoming_tx: tx_nonce(1) >= account_nonce(0) → PASS
   c. skip_stateful_validations:
      - tx.nonce() == Nonce(1) ✓
      - account_nonce == Nonce(0) ✓
      - account_tx_in_pool_or_recent_block(X) == true ✓ (victim's deploy_account is there)
      → returns true (skip validation)
   d. run_validate_entry_point(skip_validate=true):
      - execution_flags.validate = false
      - StatefulValidator::perform_validations returns Ok(()) without calling __validate__
      → Signature NEVER VERIFIED

4. Attacker's invoke admitted to mempool.

5. Attacker sets tip higher than victim's invoke (if any).

6. Block execution order:
   - DeployAccount(X, nonce=0): deploys X with victim's public key
   - Attacker's Invoke(X, nonce=1): executes transfer_all_tokens_to_attacker
     from X's __execute__ without signature check

Result: Attacker drains victim's account at deployment time.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

```rust
    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
