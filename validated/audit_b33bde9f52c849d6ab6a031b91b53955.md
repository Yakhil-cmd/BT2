### Title
Presignature Reuse Enabled by `Clone` on `PresignOutput` — No One-Time-Use Enforcement - (File: `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/frost/mod.rs`)

### Summary

Both ECDSA `PresignOutput` types and the FROST `PresignOutput` type publicly derive `Clone` and `Serialize/Deserialize`, making presignature material trivially copyable and storable. The library documents that presignatures must be used exactly once, but provides no type-system or runtime enforcement of this invariant. A malicious coordinator can clone a `PresignOutput` before rerandomization, drive two independent signing sessions against the same underlying nonce with different message hashes, and use the resulting signature pair to extract the full private key via standard ECDSA nonce-reuse algebra.

The `sign()` function for robust ECDSA takes `RerandomizedPresignOutput` by value (moves it, consuming it), which appears to enforce one-time use: [4](#0-3) 

However, because `PresignOutput` is `Clone`, a caller can clone it **before** calling `rerandomize_presign`, producing two independent `RerandomizedPresignOutput` values that share the same underlying nonce scalar `k`. The library's own security documentation explicitly identifies this as the dangerous path:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [5](#0-4) 

The orchestration documentation reinforces this:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [6](#0-5) 

Yet the library provides no mechanism to enforce destruction or single-use. `PresignOutput` is also `Serialize/Deserialize`, so it can be persisted to disk and replayed across process restarts.

**Exploit flow (malicious coordinator, robust ECDSA):**

1. Coordinator participates in presigning → receives `PresignOutput { big_r, alpha, beta, c, e }`.
2. Coordinator clones the output: `let presign2 = presign1.clone()`.
3. Coordinator rerandomizes `presign1` with `(h1, ε1)` → `RerandomizedPresignOutput1`, then initiates signing session 1 with all `2t+1` participants. Participants use their own (uncloned) shares for session 1. Session 1 produces signature `(R, s1)` for `h1`.
4. Coordinator claims session 1 failed (network error, timeout, etc.) and initiates session 2 using the same participant set and the same presignature pool entry (participants have no tracking of used presignatures). Coordinator rerandomizes `presign2` with `(h2, ε2)` → `RerandomizedPresignOutput2`. Session 2 produces signature `(R', s2)` for `h2`.
5. Because both sessions derive from the same nonce `k`, the coordinator can solve for the private key `x` using the two signature equations.

For FROST (`src/frost/mod.rs`), the same `Clone` on `PresignOutput<C>` allows a participant to reuse `SigningNonces`, leaking their individual signing share via the same nonce-reuse algebra. [3](#0-2) 

### Impact Explanation

**Critical — private key extraction.** Reusing a presignature across two signing sessions with different `(h, ε)` pairs yields two ECDSA equations sharing the same nonce. Standard nonce-reuse algebra recovers the aggregate secret key `x`. For FROST, reuse of `SigningNonces` by a single participant recovers that participant's signing share. Both outcomes fall under the allowed Critical impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

### Likelihood Explanation

A malicious coordinator is an explicitly modeled attacker in this library (the coordinator role requires no privileged access beyond being a signing participant). The attack requires only that the coordinator orchestrate two signing sessions against the same presignature pool entry, which is straightforward in any system where the coordinator manages presignature distribution. The `Clone` and `Serialize` derives make the reuse trivial to implement programmatically. Honest participants have no in-library mechanism to detect or refuse presignature reuse.

### Recommendation

1. **Remove `Clone` from `PresignOutput`** in all three modules (`src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/frost/mod.rs`). Removing `Clone` forces callers to move the value, making accidental cloning a compile-time error.
2. **Consider removing or gating `Serialize/Deserialize`** on `PresignOutput` behind a feature flag with a prominent warning, since serialization enables cross-process replay.
3. **Add a consumed/used flag or wrapper type** (e.g., `OneShot<PresignOutput>`) that panics or errors on second use, providing a runtime safety net even if serialization is retained.
4. **Document in the `presign()` function signature** (not just in docs/) that the returned value must be used exactly once, and consider a `#[must_use]` attribute.

### Proof of Concept

```rust
// Attacker (malicious coordinator) holds presign_output: PresignOutput
// from a completed presigning session.

// Step 1: Clone before rerandomization — library permits this
let presign_copy = presign_output.clone(); // compiles fine; Clone is derived

// Step 2: Rerandomize with message 1
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output, &args_for_h1_epsilon1
).unwrap();

// Step 3: Rerandomize the clone with message 2
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_copy, &args_for_h2_epsilon2
).unwrap();

// Step 4: Run two signing sessions — both succeed because the library
// has no presignature-used tracking.
// The two resulting signatures (R1,s1) and (R2,s2) share the same
// underlying nonce k, enabling private key extraction:
//   s1 = h1*k_inv + Rx*x*k_inv  (after rerandomization algebra)
//   s2 = h2*k_inv + Rx*x*k_inv
//   => x = (s1*h2 - s2*h1) / (Rx*(s2 - s1))  [standard nonce-reuse recovery]
```

The `Clone` derive on `PresignOutput` at `src/ecdsa/robust_ecdsa/mod.rs:26` and `src/ecdsa/ot_based_ecdsa/mod.rs:40` is the necessary and sufficient library-level root cause. No external assumption, leaked key, or cryptographic break is required.
