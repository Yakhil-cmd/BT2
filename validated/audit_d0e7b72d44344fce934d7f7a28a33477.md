### Title
`_validatePriceProvider` accepts any arbitrary price-provider address without factory-registry membership check — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`MetricOmmPoolFactory._validatePriceProvider` only verifies that the candidate price provider returns the correct `token0()`/`token1()` pair. It never checks whether the provider was deployed and registered by the trusted `PriceProviderFactory`. A pool admin can therefore propose a hand-crafted contract that passes the token-pair check but returns fully manipulated bid/ask prices, causing bad-price execution against every swap in the pool.

---

### Finding Description

`_validatePriceProvider` is the single guard used in both the initial `createPool` path and the mutable-provider update path (`proposePoolPriceProvider` → `executePoolPriceProviderUpdate`):

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 541-546
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
      revert PriceProviderTokenMismatch();
    }
}
``` [1](#0-0) 

The two checks are trivially satisfied by any contract that hard-codes the correct token addresses in `token0()`/`token1()` while returning arbitrary values from `getBidAndAskPrice()`.

The `PriceProviderFactory` in the oracle sub-repo already maintains an on-chain registry of approved providers:

```solidity
// smart-contracts-poc/contracts/PriceProviderFactory.sol  lines 148-150
function isProvider(address provider) external view returns (bool) {
    return _providers.contains(provider);
}
``` [2](#0-1) 

`MetricOmmPoolFactory` holds no reference to `PriceProviderFactory` and never calls `isProvider`. The gap exists in both the initial pool-creation path and the post-deployment provider-change path:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 474-491
function proposePoolPriceProvider(address pool, address newPriceProvider)
    external override nonReentrant onlyPoolAdmin(pool)
{
    ...
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);   // ← only token-pair check
    ...
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    ...
}
``` [3](#0-2) 

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 494-507
function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    ...
    _validatePriceProvider(p.token0, p.token1, pending);   // ← same weak check at execution time
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    ...
}
``` [4](#0-3) 

The pool's swap loop reads bid/ask exclusively from the active price provider:

```solidity
// metric-core/contracts/MetricOmmPool.sol  lines 59-60
address internal immutable IMMUTABLE_PRICE_PROVIDER;
...
address internal priceProvider;   // mutable slot, written by setPriceProvider
``` [5](#0-4) 

Once the malicious provider is active, every swap in the pool prices against attacker-controlled bid/ask values.

---

### Impact Explanation

A pool admin who controls a pool with a mutable price provider (`immutablePriceProvider == address(0)`) can install a price provider that returns an arbitrarily low bid or arbitrarily high ask. Traders executing swaps against the pool receive prices that deviate from the true oracle price by an unbounded amount, resulting in direct loss of user principal. The pool admin can front-run the activation to extract the maximum value from pending swaps or LP positions.

This matches the allowed impact gate: **bad-price execution** (unbounded bid/ask quote reaches a pool swap) and **direct loss of user principal**.

---

### Likelihood Explanation

- The pool admin role is semi-trusted and can be any address set at pool creation — it is not restricted to the factory owner.
- The `priceProviderTimelock` is set by the pool creator and can be **zero**, allowing the provider change to be proposed and executed in the same block with no window for LPs or traders to exit.
- Constructing a malicious price provider that passes the token-pair check requires only a few lines of Solidity.
- No privileged factory-owner action is needed after pool creation.

---

### Recommendation

Store the canonical `PriceProviderFactory` address in `MetricOmmPoolFactory` and extend `_validatePriceProvider` to require registry membership:

```solidity
// In MetricOmmPoolFactory constructor / storage:
address public priceProviderFactory;

// In _validatePriceProvider:
if (!IPriceProviderFactory(priceProviderFactory).isProvider(priceProvider)) {
    revert PriceProviderNotRegistered();
}
```

This mirrors the pattern already present in `PriceProviderFactory.isProvider` and closes the gap identified in the external report: the factory must bind each pool to a predefined, registry-verified set of price-provider addresses.

---

### Proof of Concept

```solidity
// Attacker deploys a malicious price provider
contract MaliciousPriceProvider {
    address public token0;
    address public token1;
    constructor(address _t0, address _t1) { token0 = _t0; token1 = _t1; }

    // Returns correct tokens → passes _validatePriceProvider
    // Returns manipulated prices → bad-price execution
    function getBidAndAskPrice() external returns (uint128 bid, uint128 ask) {
        bid = 1;                      // near-zero bid: sell token0 for almost nothing
        ask = type(uint128).max - 1;  // near-max ask: buy token0 at infinite cost
    }
}

// Pool admin flow (timelock == 0 for instant execution):
factory.proposePoolPriceProvider(pool, address(malicious));
factory.executePoolPriceProviderUpdate(pool);   // executes in same block

// Any swap now executes at attacker-controlled prices
pool.swap(recipient, true, amountIn, ...);  // trader loses principal
```

The `_validatePriceProvider` call at line 483 and line 502 both pass because `malicious.token0() == pool.token0` and `malicious.token1() == pool.token1`. No further check prevents the malicious provider from being installed. [6](#0-5) [7](#0-6)

    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.

    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();

    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
