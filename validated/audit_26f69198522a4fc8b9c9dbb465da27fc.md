### Title
L1→L2 Block Processing Permanently Halted by `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Revert — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary
The `notify_l2_asset_tracker` function pushes a notification to the `L2AssetTracker` contract during every L1→L2 transaction with a non-zero deposit. If this external call reverts for any reason, the bootloader treats it as a fatal internal error and permanently halts block processing. This is a direct "push over pull" analog: a push to an external contract is embedded in the critical path of every L1→L2 deposit, and any revert in that contract makes the entire block unprocessable.

### Finding Description
In `process_l1_transaction.rs`, `notify_l2_asset_tracker` (lines 855–915) calls `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)` via `run_single_interaction`. If the call returns a failed result, the function immediately returns a fatal `InternalError`:

```rust
if failed {
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    ).into());
}
```

The code comment at line 848 explicitly acknowledges the consequence: *"Failure halts block processing — if the asset tracker reverts, the chain's token accounting would be inconsistent, so we treat it as fatal rather than silently continuing with incorrect bookkeeping."*

`notify_l2_asset_tracker` is called by `mint_base_token` (lines 741–769), which is invoked up to **three times** per L1→L2 transaction with a non-zero deposit:

1. **Value mint to sender** — line 634, inside `execute_l1_transaction_and_notify_result`
2. **Operator fee payment** — line 290, post-execution
3. **Refund to refund recipient** — line 338, post-execution

The `_fromChainId` argument is read from storage slot 154 of `L2AssetTracker` via `read_l1_chain_id` (lines 921–943). This is a live storage read from a real deployed contract whose state is accessible to L2 callers.

The `L2AssetTracker` contract is a real EVM smart contract at `L2_ASSET_TRACKER_ADDRESS`. Its `handleFinalizeBaseTokenBridgingOnL2` function contains internal logic (access control, accounting, state transitions) that can revert under conditions that may be reachable by an unprivileged L2 caller who manipulates the contract's state before an L1→L2 deposit is processed. For example, if the contract tracks cumulative deposit amounts and enforces an upper bound, an attacker who pre-fills that counter via legitimate L2 calls could cause the next L1→L2 deposit notification to revert, halting the block.

### Impact Explanation
A single revert of `handleFinalizeBaseTokenBridgingOnL2` during any of the three `mint_base_token` calls causes `process_l1_transaction` to return a fatal `BootloaderSubsystemError`. This propagates up through the block execution loop and halts all block processing. Every subsequent L1→L2 transaction with a non-zero deposit will trigger the same fatal path, permanently blocking the deposit pipeline. The impact is equivalent to the AeraVault scenario: a push to an external entity is on the critical path of all deposit operations, and a single failure makes those operations inaccessible.

### Likelihood Explanation
The `L2AssetTracker` is a real deployed Solidity contract whose internal state is readable and potentially writable by unprivileged L2 callers. The `_fromChainId` value is read live from storage at transaction processing time. Any reachable code path inside `handleFinalizeBaseTokenBridgingOnL2` that reverts — whether due to overflow in accounting mappings, an uninitialized `L1_CHAIN_ID` (slot 154 returning 0), or a state invariant that an attacker can violate via prior L2 calls — is sufficient to trigger the halt. The bootloader provides no fallback, retry, or graceful degradation for this call.

### Recommendation
Replace the fatal-error treatment of `L2AssetTracker` revert with a pull pattern or a non-fatal fallback:

- **Pull pattern**: Instead of pushing a notification synchronously during block execution, record the pending notification in a queue and let the `L2AssetTracker` pull it in a separate, non-critical call.
- **Non-fatal fallback**: Wrap the `notify_l2_asset_tracker` call in a try/catch equivalent. If the call fails, emit an event or store the failed notification for later reconciliation, but do not halt block processing. The accounting inconsistency is less severe than a permanently halted chain.
- **Invariant pre-check**: Before calling `handleFinalizeBaseTokenBridgingOnL2`, verify that the call will succeed (e.g., check that `L1_CHAIN_ID != 0` and that no accounting overflow will occur) and handle the failure case gracefully.

### Proof of Concept

1. An unprivileged L2 caller interacts with `L2AssetTracker` to manipulate its internal state (e.g., filling an accounting counter to its maximum value, or triggering a state transition that causes `handleFinalizeBaseTokenBridgingOnL2` to revert on the next call).
2. An L1→L2 transaction with `total_deposited > 0` is submitted and reaches the sequencer.
3. `process_l1_transaction` calls `mint_base_token` → `notify_l2_asset_tracker`.
4. `notify_l2_asset_tracker` calls `handleFinalizeBaseTokenBridgingOnL2` on `L2AssetTracker`.
5. The call reverts due to the manipulated state.
6. `notify_l2_asset_tracker` returns `internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted")`.

/// chain's token accounting would be inconsistent, so we treat it as
/// fatal rather than silently continuing with incorrect bookkeeping.
///
/// If no contract is deployed at L2AssetTracker, the call succeeds silently
/// (a call to an empty address returns success with no returndata in EVM).
/// However, we are certain that L2AssetTracker is available after the upgrade.
fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())

            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
