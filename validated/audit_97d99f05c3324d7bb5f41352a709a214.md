### Title
One-Step `transferProviderOwnership` Allows Malicious Curator to Silently Persist Updater Grants, Enabling Post-Transfer Oracle Price Manipulation - (File: smart-contracts-poc/contracts/PriceProviderFactory.sol)

---

### Summary

`PriceProviderFactory`, `PriceProviderFactoryL2`, and `AnchoredProviderFactory` implement `transferProviderOwnership` as an immediate, one-step transfer with no accept step. A malicious outgoing owner can front-run the transfer transaction with `grantUpdater(provider, attacker_controlled_address)`, silently persisting an updater that survives the ownership change. Because the `isUpdater` mapping is non-enumerable, the incoming owner has no on-chain mechanism to discover or sweep all granted updaters. The persistent updater can then call `setConfidence` to push `confidenceParam` to extreme values, corrupting the bid/ask prices fed to any pool that uses the affected provider.

---

### Finding Description

`PriceProviderFactory.transferProviderOwnership` (lines 92–102) and its identical copies in `PriceProviderFactoryL2` (lines 95–105) and `AnchoredProviderFactory` (lines 230–240) execute the ownership change atomically in a single call:

```solidity
// PriceProviderFactory.sol L92-102
function transferProviderOwnership(address provider, address newOwner)
    external override onlyProviderOwner(provider)
{
    require(_providers.contains(provider), ProviderNotTracked());
    require(newOwner != address(0));
    address previousOwner = providerOwner[provider];
    providerOwner[provider] = newOwner;          // immediate, no accept
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);
    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
```

The outgoing owner retains full `onlyProviderOwner` rights until the transfer transaction is mined. In the same block (or with a higher-gas front-run), they can call:

```solidity
// PriceProviderFactory.sol L80-84
function grantUpdater(address provider, address updater)
    external override onlyProviderOwner(provider)
{
    require(_providers.contains(provider), ProviderNotTracked());
    isUpdater[provider][updater] = true;
    emit UpdaterGranted(provider, updater);
}
```

`isUpdater` is a plain `mapping(address provider => mapping(address updater => bool))` — there is no enumerable set, no event index that the new owner can query to reconstruct the full updater list, and no automatic clearing on transfer. After the transfer completes, the new owner holds `providerOwner[provider]` but the attacker-controlled address remains `isUpdater[provider][attacker] == true`.

The persistent updater can then call `setConfidence` (via the factory) at any time (subject only to `CONFIDENCE_COOLDOWN`):

```solidity
// PriceProvider.sol L92-104
function setConfidenceParam(uint256 newValue) external {
    require(msg.sender == factory, OnlyFactory());
    if (newValue > CONFIDENCE_MAX) revert ConfidenceParamOutOfBounds();
    if (block.timestamp < lastConfidenceUpdate + CONFIDENCE_COOLDOWN)
        revert CooldownNotElapsed();
    confidenceParam = newValue;
    lastConfidenceUpdate = block.timestamp;
    emit ConfidenceParamSet(newValue);
}
```

`confidenceParam` directly scales the oracle spread applied to the mid-price before the bid/ask is returned to the pool. Setting it to `CONFIDENCE_MAX` (1 000 000, confirmed by `testConfidenceMaxValue`) can drive `bid` to zero, causing `FeedStalled()` to revert every swap on any pool bound to this provider. Setting it to 0 collapses the confidence band to zero, leaving only the `marginStep` — effectively narrowing the spread and allowing traders to extract value at the pool's expense.

**Contrast with the pool-admin transfer**, which correctly uses a two-step pattern:

```solidity
// MetricOmmPoolFactory.sol L510-526
function proposePoolAdminTransfer(...) { pendingPoolAdmin[pool] = newAdmin; }
function acceptPoolAdmin(address pool) {
    if (msg.sender != pending) revert NotPendingPoolAdmin(...);
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
}
```

The oracle factory ownership transfer has no equivalent accept step and no state-clearing on handover.

---

### Impact Explanation

A pool bound to a manipulated provider will receive corrupted bid/ask prices on every swap call. At `confidenceParam = CONFIDENCE_MAX`, `bid` can reach zero, causing `FeedStalled()` to revert all swaps — a complete denial of the pool's core swap functionality. At intermediate values, the spread is widened or narrowed beyond what the pool admin or protocol intended, constituting bad-price execution: traders receive prices outside the oracle/bin curve bounds, and the pool either over-pays or under-collects on every swap until the new owner discovers and revokes the hidden updater.

Because `isUpdater` is non-enumerable, the new owner cannot programmatically sweep all granted updaters; they must know the attacker's address to call `revokeUpdater`. Until they do, every swap on every pool using the provider is exposed.

---

### Likelihood Explanation

`createPriceProvider` is permissionless — any address can become a provider owner. Ownership transfers are a normal operational event (e.g., handing a provider to a DAO, a new team, or a pool admin). The front-run requires only that the outgoing owner submit `grantUpdater` with a higher gas price than the `transferProviderOwnership` transaction, which is trivially achievable. The attack is silent: `UpdaterGranted` is emitted before the transfer, but the new owner has no reason to scan historical events for a provider they did not yet own at the time of the grant.

---

### Recommendation

1. **Adopt a two-step transfer** (propose + accept) identical to `MetricOmmPoolFactory.proposePoolAdminTransfer` / `acceptPoolAdmin`, so the incoming owner can inspect on-chain state before committing.
2. **Clear all updaters on transfer**, or replace `mapping(address => bool)` with an `EnumerableSet.AddressSet` so the new owner can enumerate and revoke all updaters atomically upon acceptance.
3. **Alternatively**, emit a `UpdaterGranted` event indexed on the provider and document that new owners must audit the event log before accepting, and provide a helper view that returns the full updater set.

---

### Proof of Concept

```
1. Attacker (owner A) deploys a PriceProvider via createPriceProvider(oracle, feedId, ...).
   → providerOwner[provider] = A

2. A negotiates to sell/transfer the provider to party B.
   B prepares a tx to call acceptPoolAdmin (or simply waits for the transfer).

3. A submits two transactions in the same block, grantUpdater first (higher gas):
     factory.grantUpdater(provider, A_bot);          // isUpdater[provider][A_bot] = true
     factory.transferProviderOwnership(provider, B); // providerOwner[provider] = B

4. Both mine. B now owns the provider. isUpdater[provider][A_bot] is still true.
   B has no enumerable list of updaters and does not know A_bot's address.

5. A_bot calls factory.setConfidence([provider], [CONFIDENCE_MAX]).
   → confidenceParam = 1_000_000 on the PriceProvider.

6. Any pool bound to this provider calls getBidAndAskPrice():
   → bid computed as 0 (delta ≥ mid) → FeedStalled() reverts.
   → All swaps on the pool revert until B discovers A_bot and calls revokeUpdater.
