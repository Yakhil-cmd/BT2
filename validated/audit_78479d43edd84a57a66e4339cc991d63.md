### Title
Missing Access Control on `update_rewards` Allows Any Caller to Suppress Staker Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the staking contract is specified to be callable only by the Starknet sequencer, but the implementation contains no such access control. Any unprivileged caller can invoke it with `disable_rewards: true`, consuming the per-block reward slot and preventing legitimate reward distribution for that block.

### Finding Description
The protocol specification (`docs/spec.md`) explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` only calls `general_prerequisites()`, which checks that the contract is not paused and the caller is not the zero address — no sequencer-only guard exists:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();  // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ...
    self.last_reward_block.write(current_block_number);  // consumes the slot

    if disable_rewards || self.is_pre_consensus() {
        return;  // exits without distributing rewards
    }
    // ... reward distribution
}
```

The function enforces a one-call-per-block invariant via `last_reward_block`. Once any caller invokes it for block N, no further call can succeed for block N. An attacker who calls `update_rewards(staker_address, disable_rewards: true)` before the sequencer does so:

1. Writes `current_block_number` into `last_reward_block`, consuming the slot.
2. Returns immediately without distributing any rewards to the staker or its delegation pools.

The legitimate sequencer call for that same block will then revert with `REWARDS_ALREADY_UPDATED`.

This is the direct analog of the reference report's pattern: a caller-controlled parameter (`disable_rewards`) that the spec restricts to a trusted party but the code leaves open to any address, allowing an unprivileged actor to alter the outcome of a privileged operation.

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users or protocol.**

For every block where the attacker front-runs the sequencer, the staker and all its pool delegators receive zero block rewards. Repeated across many blocks this constitutes sustained, targeted suppression of yield. The attacker gains nothing financially; the damage is entirely borne by stakers and delegators.

If the attacker can reliably act before the sequencer on a given block (e.g., during sequencer downtime or in a future decentralized sequencer model), the impact escalates to **High — Permanent freezing of unclaimed yield**, since rewards for those blocks are irrecoverably lost (the slot cannot be reclaimed).

### Likelihood Explanation
On the current centralized Starknet sequencer, the sequencer controls transaction ordering and can always place its own `update_rewards` call ahead of any user transaction. This limits exploitability to windows where the sequencer has not yet submitted its call for a given block (e.g., sequencer lag, downtime, or missed blocks). As Starknet moves toward decentralized sequencing, the attack surface widens significantly. The missing guard is a latent vulnerability whose exploitability increases with protocol maturity.

### Recommendation
Add an access control check that restricts `update_rewards` to the authorized sequencer address, consistent with the specification. For example, store the sequencer address in contract storage and assert it at the top of the function:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ...
}
```

Alternatively, use the existing `RolesComponent` to define a `SEQUENCER` role and gate the function with `self.roles.only_sequencer()`.

### Proof of Concept
1. Staker stakes and waits K epochs for balance to become effective.
2. Consensus rewards are activated.
3. At block N, before the sequencer submits its `update_rewards` call, attacker calls:
   ```
   staking_contract.update_rewards(staker_address, disable_rewards: true)
   ```
4. `last_reward_block` is set to N; no rewards are distributed.
5. The sequencer's subsequent call for block N reverts with `REWARDS_ALREADY_UPDATED`.
6. The staker's `unclaimed_rewards_own` and all pool balances remain unchanged — block N rewards are permanently lost.

            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,

            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`

            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self

                );
            // Assert staker has non-zero balance.
