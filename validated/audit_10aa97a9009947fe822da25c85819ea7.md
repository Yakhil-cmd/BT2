### Title
Gateway Admits Unsigned Invoke Transactions for Pre-Funded Undeployed Accounts via `skip_stateful_validations` Bypass - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point call (i.e., signature verification) for any Invoke transaction with `nonce == 1` submitted against an account whose on-chain nonce is still `0` and which has **any** transaction present in the mempool pool or recent-block state. Because the only way to place a transaction in the pool for a non-existent account is via a `DeployAccount`, the check is semantically equivalent to "is there a pending deploy for this address?" — but it never verifies the **signature** of the incoming nonce-1 Invoke. An unprivileged attacker who observes a victim's `DeployAccount` in the mempool can inject a nonce-1 Invoke with an arbitrary or invalid signature, and the gateway will admit it to the mempool without any account-level signature check.

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

When `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)`, the function queries `account_tx_in_pool_or_recent_block` and, if it returns `true`, returns `true` itself — meaning "skip the validate entry point."

**How `skip_validate` propagates** [2](#0-1) 

`skip_validate = true` sets `execution_flags.validate = false`. Inside `StatefulValidator::perform_validations`: [3](#0-2) 

`perform_pre_validation_stage` (nonce increment + fee-bound check) still runs, but the `__validate__` Cairo entry point — which is the only place the account contract verifies the ECDSA/Stark signature — is **never called**.

**What `account_tx_in_pool_or_recent_block` actually checks** [4](#0-3) 

It returns `true` if the address appears in `tx_pool` (any pending transaction) or in `state` (any committed block). It does **not** verify that the pooled transaction is a `DeployAccount`, nor does it inspect the signature of the incoming Invoke.

**Attack path**

1. Victim Alice pre-funds address `A` with STRK (required before deployment) and submits a `DeployAccount` for `A`. The `DeployAccount` enters the mempool; `account_nonce` for `A` in state is still `0`.
2. Attacker Mallory observes Alice's `DeployAccount` in the public mempool.
3. Mallory crafts a nonce-1 `InvokeTransaction` for address `A` with an **invalid or forged signature** and arbitrary calldata.
4. Gateway stateless validation passes (resource bounds, calldata size, etc. are all attacker-controlled).
5. `convert_rpc_tx_to_internal` computes the tx hash and stores the transaction.
6. `extract_state_nonce_and_run_validations` fetches `account_nonce = 0`.
7. `run_pre_validation_checks` → `validate_state_preconditions` passes (nonce `1` is within `[0, 200]`; balance check passes because Alice pre-funded `A`).
8. `validate_by_mempool` passes (no duplicate nonce, fee escalation rules satisfied).
9. `skip_stateful_validations` returns `true` because `tx.nonce() == 1`, `account_nonce == 0`, and `account_tx_in_pool_or_recent_block(A) == true`.
10. `run_validate_entry_point` is called with `validate: false`; `__validate__` is **never invoked**.
11. Mallory's unsigned/invalidly-signed Invoke is **admitted to the mempool**.

### Impact Explanation

This directly satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

- An invalid transaction (one that would fail `__validate__` during execution) is admitted to the mempool, violating the invariant that all pooled transactions carry a valid account signature.
- If Mallory's transaction reaches a block before Alice's `DeployAccount` is committed, it fails immediately (no code at address). If it reaches the block after Alice's `DeployAccount`, the account's `__validate__` is called during execution, rejects the invalid signature, the transaction reverts, and Alice's pre-funded balance is charged the validation fee — a griefing/balance-drain attack.
- Mallory can also use fee escalation to **replace** Alice's legitimate nonce-1 Invoke with Mallory's invalid one, causing Alice's real transaction to be evicted from the mempool.

### Likelihood Explanation

- `DeployAccount` transactions are publicly visible in the mempool.
- Pre-funding an account address before deployment is the standard Starknet UX flow; virtually every new account deployment involves a pre-funded address.
- The attack requires no privileged access, no special contract, and no on-chain state beyond observing the mempool.
- The `max_nonce_for_validation_skip` config is hardcoded to `Nonce(Felt::ONE)` in production, so the window is exactly one transaction per account.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **`DeployAccount`** transaction for the sender address is present in the pool. Alternatively, do not skip `__validate__` entirely; instead, run it with a relaxed nonce check (the account contract can be designed to tolerate nonce-0 state during the deploy window). At minimum, add a type assertion:

```rust
// Only skip validation if the pooled transaction is specifically a DeployAccount.
let has_pending_deploy = mempool_client
    .has_pending_deploy_account(tx.sender_address())
    .await?;
return Ok(has_pending_deploy);
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

```rust
    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
