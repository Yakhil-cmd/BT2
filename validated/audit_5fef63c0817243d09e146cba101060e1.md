### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Is Called With `msg.value` and No Pending Mint Requests — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is declared `external payable` and accepts ETH as L1→L2 message fees. When there are no pending L2 mint requests (`amountToMint == 0`), the entire fee-spending block is skipped and any ETH sent with the call is permanently locked in the contract. The contract has no `receive`/`fallback` function and no ETH-withdrawal mechanism, so the funds are irrecoverable.

---

### Finding Description

`RewardSupplier.tick()` is the public entry point for processing L2 mint requests on L1:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... consume messages, mint tokens ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // spends half
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // spends rest
    }
    // ← when amountToMint == 0, msg.value is never spent and never returned
}
``` [1](#0-0) 

When `amountToMint == 0` the `if` block is entirely skipped. `msg.value` is neither forwarded to any sub-call nor refunded to the caller. Because `RewardSupplier` inherits only `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl` and defines no `receive()` or `fallback()` function, the ETH has no exit path and is permanently trapped in the contract. [2](#0-1) 

The same structural gap exists in the PausableZone analogy: a contract that participates in a value-bearing protocol flow but lacks any endpoint to handle ETH that is not consumed.

---

### Impact Explanation

Any ETH sent to `tick()` when no mint requests are pending is permanently frozen inside `RewardSupplier`. There is no admin withdrawal function, no `receive`/`fallback`, and no upgrade path that automatically recovers stranded ETH. The caller's funds are unrecoverable without a contract upgrade.

**Allowed impact matched**: *Temporary/Permanent freezing of funds* (High) and *Griefing with no profit motive but damage to users or protocol* (Medium).

---

### Likelihood Explanation

`tick()` carries no access control — any address may call it. [3](#0-2) 

The triggering condition (`amountToMint == 0`) is routine: it occurs whenever the L2 reward supplier has not yet emitted new mint-request messages, or when all pending messages have already been consumed in a prior `tick()` call. A caller who does not pre-check `requiredMinting()` — or who is front-run between the check and the call — will silently lose their ETH. A griefer can also deliberately call `tick{value: X}()` at any time when no messages are pending to permanently lock `X` wei. [4](#0-3) 

---

### Recommendation

Add an explicit refund of unused ETH at the end of `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund unused fee to caller
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, add a `receive()` function and an owner-controlled ETH-sweep function, or revert at the top of `tick()` when `msg.value > 0` and `amountToMint == 0`.

---

### Proof of Concept

1. Deploy `RewardSupplier` with no pending L2→L1 mint-request messages (i.e., `messagingContract().l2ToL1Messages(msgHash) == 0`).
2. Call `tick{value: 1 ether}()` from any EOA.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. The transaction succeeds (no revert), but `address(rewardSupplier).balance` increases by `1 ether`.
6. No function exists to withdraw that ETH; it is permanently locked. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
