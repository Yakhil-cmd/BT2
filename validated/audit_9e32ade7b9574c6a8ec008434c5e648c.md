### Title
Zero `reward_address` Accepted Without Validation, Causing Permanent Loss of Unclaimed Yield - (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Neither `Staking` nor `Pool` validates that a supplied `reward_address` is non-zero. A staker or pool member can register or update their reward address to `0x0`, after which every subsequent reward transfer is sent to the zero address and permanently lost.

### Finding Description
The analog to the `GaugeProxy::distributeToken` bug is exact: a reward-routing address is accepted without a non-zero guard, so rewards are silently burned when the transfer executes.

**Staking contract – `stake()` and `change_reward_address()`**

`stake()` validates `reward_address` only against the token-address registry: [1](#0-0) 

`does_token_exist` returns `false` for the zero address because it only checks `STRK_TOKEN_ADDRESS` and the BTC token map: [2](#0-1) 

`change_reward_address()` applies the identical (and only) guard: [3](#0-2) 

When rewards are later claimed, `send_rewards_to_staker` transfers directly to whatever `reward_address` is stored, with no zero-check: [4](#0-3) 

**Pool contract – `enter_delegation_pool()` and `change_reward_address()`**

`enter_delegation_pool()` only checks that `reward_address` is not the pool's token address: [5](#0-4) 

`change_reward_address()` in the pool applies the same single guard: [6](#0-5) 

`claim_rewards` then transfers to the stored `reward_address` unconditionally: [7](#0-6) 

### Impact Explanation
Any accumulated STRK rewards are transferred to `0x0` and are unrecoverable. This constitutes **permanent freezing of unclaimed yield** for the affected staker or pool member. The staked principal is unaffected, but all future reward claims are burned for as long as the zero address remains set.

### Likelihood Explanation
Low. The zero address must be explicitly passed by the staker or pool member. This can occur through a front-end bug, a scripting error, or a deliberate self-griefing action. No privileged role is required; any registered staker or pool member can trigger it via the public `change_reward_address` entry point.

### Recommendation
Add a non-zero assertion in every function that accepts or stores a `reward_address`:

```cairo
// In staking.cairo – stake() and change_reward_address()
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);

// In pool.cairo – enter_delegation_pool() and change_reward_address()
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);
```

### Proof of Concept

1. A staker calls `Staking::change_reward_address(reward_address: 0)`.
   - The only guard is `!does_token_exist(0)` → `false` (zero is not a registered token) → assertion passes.
   - `staker_info.reward_address` is written as `0`.
2. The staker accrues rewards over subsequent epochs.
3. Anyone calls `Staking::claim_rewards(staker_address)`.
   - `send_rewards_to_staker` reads `reward_address = 0` and executes `token_dispatcher.checked_transfer(recipient: 0, amount: rewards)`.
   - All accrued STRK rewards are sent to the zero address and permanently lost.

The same sequence applies to a pool member via `Pool::change_reward_address(0)` followed by `Pool::claim_rewards`.
