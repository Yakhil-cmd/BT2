### Title
Unchecked NEAR Transfer Promise Result in wNEAR Unwrap Exit Path Causes Permanent Fund Loss - (`engine/src/contract_methods/connector.rs`)

### Summary

In `exit_to_near_precompile_callback`, when handling the wNEAR-unwrap exit path, a `PromiseAction::Transfer` is dispatched to send unwrapped NEAR to a user-controlled `target_account_id` with no callback to verify the transfer succeeded. If the transfer fails (e.g., the target is a non-existent named NEAR account), the wNEAR has already been burned in the EVM but the NEAR is silently refunded to the engine contract with no recovery path for the user.

### Finding Description

The `ExitToNear` precompile handles wNEAR unwrapping via a two-step promise chain:

1. `near_withdraw` is called on the wNEAR NEP-141 contract (burning the wNEAR and releasing NEAR to the engine).
2. `exit_to_near_precompile_callback` is called as the callback. If step 1 succeeded and `transfer_near` args are present, it dispatches a `PromiseAction::Transfer` batch to send the NEAR to the user-specified `target_account_id`. [1](#0-0) 

The critical issue is at lines 226–227: `handler.promise_create_batch(&promise)` fires the NEAR transfer and `handler.promise_return(promise_id)` returns it, but **no callback is attached** to detect whether the transfer succeeded or failed. [2](#0-1) 

The `target_account_id` originates from `exit_params.receiver_account_id`, which is parsed directly from user-controlled calldata in the EVM transaction. [3](#0-2) 

`parse_recipient` validates only the syntactic format of the NEAR account ID, not whether the account actually exists on-chain. [4](#0-3) 

In NEAR, a `Transfer` action to a non-existent named account (e.g., `nonexistent.near`) fails at the receipt level. The NEAR is refunded to the predecessor (the engine contract), but the engine has no mechanism to track or redistribute these failed-transfer refunds. The wNEAR was already burned in the EVM in step 1, so the user's funds are permanently frozen in the engine contract.

Compare this to the `error_refund` feature path, which does attach a callback (`exit_to_near_precompile_callback`) to handle failures in the `ft_transfer` step. The wNEAR-unwrap transfer step has no equivalent safety net. [5](#0-4) 

### Impact Explanation

**Permanent freezing of funds.** The user's wNEAR is burned (EVM state is finalized), the NEAR is released from the wNEAR contract to the engine, but the final `Transfer` to the user's specified account fails silently. The NEAR is refunded to the engine contract's balance with no on-chain record linking it to the user. There is no recovery function exposed to retrieve these orphaned funds.

### Likelihood Explanation

Any EVM user who calls the wNEAR ERC-20 `withdrawToNear` with the `:unwrap` suffix and specifies a syntactically valid but non-existent named NEAR account as the recipient triggers this path. This is a realistic user mistake (e.g., a typo in a NEAR account name, or specifying an account that has been deleted). No special privileges are required.

### Recommendation

Attach a callback after the `PromiseAction::Transfer` batch in `exit_to_near_precompile_callback` to verify the transfer succeeded. If it failed, re-credit the equivalent amount of wNEAR to the user's EVM address (analogous to the `refund_on_error` path used for `ft_transfer` failures when `error_refund` is enabled).

```rust
// After promise_create_batch, attach a callback:
let callback = PromiseCreateArgs {
    target_account_id: current_engine_account_id,
    method: "near_transfer_callback".to_string(),
    args: borsh::to_vec(&refund_args).unwrap(),
    attached_balance: Yocto::new(0),
    attached_gas: NEAR_TRANSFER_CALLBACK_GAS,
};
handler.promise_attach_callback(promise_id, &callback);
```

The callback should check `handler.promise_result(0)` and, on failure, re-mint wNEAR or credit ETH back to the original sender.

### Proof of Concept

1. Deploy wNEAR ERC-20 on Aurora and register it via `factory_set_wnear_address`.
2. Acquire wNEAR ERC-20 tokens in an EVM account.
3. Call the wNEAR ERC-20 `withdrawToNear` with the `:unwrap` suffix, specifying `nonexistent-account.near` (a valid-format but non-existent NEAR account) as the recipient.
4. The `ExitToNear` precompile fires `near_withdraw` on the wNEAR NEP-141 contract — wNEAR is burned.
5. `exit_to_near_precompile_callback` is invoked, sees `PromiseResult::Successful`, and dispatches `PromiseAction::Transfer` to `nonexistent-account.near`.
6. The transfer receipt fails (account does not exist); NEAR is refunded to the engine contract.
7. The user's wNEAR is gone, the NEAR is in the engine with no recovery path. Funds are permanently frozen. [6](#0-5)
