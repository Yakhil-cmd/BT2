### Title
TOCTOU in Pool Join/Change Flow Allows Malicious Pool to Substitute `relativeLockHeight` and `targetPuzzleHash` Between Display and Submission — (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

The pool-joining flow in `PlotNFTSelectPool.tsx` fetches pool parameters (`relativeLockHeight`, `targetPuzzleHash`) from the pool server **twice**: once for display and once at submission time. A malicious pool operator can serve different values between the two fetches. The user approves the displayed values but the transaction is signed and submitted with the substituted values — a direct analog to the external report's cooldown-period manipulation that locks users' assets.

---

### Finding Description

`PlotNFTSelectBase.tsx` renders pool details to the user via the `usePoolInfo` hook:

```
// PlotNFTSelectBase.tsx line 39
const poolInfo = usePoolInfo(poolUrl);
// ...
{poolInfo.poolInfo && <PoolInfo poolInfo={poolInfo.poolInfo} />}
```

`usePoolInfo` calls `getPoolInfo(normalizedUrl)` — an HTTP GET to `<poolUrl>/pool_info` — and renders the returned `relativeLockHeight` and `targetPuzzleHash` in the "Verify Pool Details" card.

When the user clicks **Create / Change**, `handleSubmit` calls `prepareSubmitData`, which issues a **second, independent** HTTP fetch to the same endpoint:

```
// PlotNFTSelectPool.tsx lines 28-39
const normalizedPoolUrl = normalizeUrl(poolUrl);
const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
// ...
initialTargetState.targetPuzzleHash = targetPuzzleHash;
initialTargetState.relativeLockHeight = relativeLockHeight;
```

The values from this second fetch — not the displayed ones — are passed directly to `pw_join_pool`:

```
// PlotNFTChangePool.tsx lines 54-60
await pwJoinPool({
  walletId,
  poolUrl,
  relativeLockHeight,
  targetPuzzlehash: targetPuzzleHash,
  fee,
}).unwrap();
```

There is no check that the submission-time values match the display-time values. The two fetches are completely independent HTTP calls with no binding between them.

---

### Impact Explanation

**`relativeLockHeight` substitution (direct analog to the external report):**
A malicious pool serves `relativeLockHeight: 32` (≈10 minutes) during the display fetch, then serves `relativeLockHeight: 4608000` (≈1 year) during the submission fetch. The user's PlotNFT singleton is committed on-chain with the inflated lock height. The user cannot leave the pool until that many blocks pass. During that window the pool operator can stop paying rewards, raise fees, or simply abandon the pool — all while the user's farming rewards continue to flow to the pool's `targetPuzzleHash`. This is structurally identical to the external report's `cooldown_period_sec` manipulation that prevents unstaking.

**`targetPuzzleHash` substitution:**
The pool serves a legitimate-looking puzzle hash during display, then substitutes an attacker-controlled puzzle hash at submission time. All pooled farming rewards are permanently redirected to the attacker's address for the lifetime of the PlotNFT's pool membership.

Both attacks are silent: the user sees correct values in the UI and has no way to detect the substitution before or after clicking confirm.

---

### Likelihood Explanation

Any operator of a pool server reachable by a Chia GUI user can execute this attack. No leaked keys, host compromise, or cryptographic break is required — only the ability to serve different JSON responses to the two sequential HTTP requests. The attack window is the few milliseconds between the display fetch and the submit fetch, which is trivially controllable server-side. The flow is reachable through both the initial "Join a Pool" wizard and the "Change Pool" action on an existing PlotNFT.

---

### Recommendation

Cache the pool-info response from the display fetch and reuse it at submission time instead of re-fetching. Concretely, `prepareSubmitData` should accept the already-fetched `PoolInfo` object (obtained from `usePoolInfo`) as a parameter rather than calling `getPoolInfo` again. This eliminates the second network round-trip and ensures the values the user reviewed are exactly the values submitted to `pw_join_pool`.

Optionally, add a final confirmation dialog that explicitly shows the `relativeLockHeight` and `targetPuzzleHash` values that will be committed on-chain, sourced from the cached (display-time) fetch result.

---

### Proof of Concept

1. Attacker operates a pool server at `https://evil-pool.example`.
2. The server tracks request count per client IP. On the **first** request to `/pool_info` it returns `{"relative_lock_height": 32, "target_puzzle_hash": "0xlegit...", ...}`.
3. On the **second** request (within the same session) it returns `{"relative_lock_height": 4608000, "target_puzzle_hash": "0xlegit...", ...}` (or substitutes `target_puzzle_hash` with an attacker-controlled address).
4. User opens Chia GUI → Pool → Join a Pool → enters `https://evil-pool.example`.
5. `usePoolInfo` fires the first fetch; the "Verify Pool Details" card shows `Relative Lock Height: 32`.
6. User clicks **Create**.
7. `prepareSubmitData` fires the second fetch; receives `relativeLockHeight: 4608000`.
8. `pw_join_pool` is called with `relativeLockHeight: 4608000` and the transaction is broadcast.
9. The user's PlotNFT is now locked to the pool for ≈1 year. The displayed value (32 blocks) was never what was submitted.
