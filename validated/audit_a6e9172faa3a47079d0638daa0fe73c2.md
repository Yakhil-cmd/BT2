### Title
No Cancel Mechanism for Pending Price Provider Proposal Lets Pool Admin Permanently Break Swap Functionality Despite Factory Owner Pause - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory` exposes a two-step oracle rotation (`proposePoolPriceProvider` → `executePoolPriceProviderUpdate`) protected by a configurable timelock, but provides **no cancel path** for the factory owner. The admin-transfer analog (`cancelPoolAdminTransfer`) exists; the price-provider analog does not. Because `executePoolPriceProviderUpdate` contains no pause guard, a compromised pool admin can execute the rotation even after the factory owner has protocol-paused the pool, installing an arbitrary oracle and permanently disabling swap functionality.

### Finding Description
`IMetricOmmPoolFactoryPoolAdmin` defines three price-provider functions:

```
proposePoolPriceProvider(pool, newPriceProvider)   // pool admin only
executePoolPriceProviderUpdate(pool)               // pool admin only, after timelock
// ← no cancelPoolPriceProviderProposal exists
```

The factory owner's only defensive tool is `protocolPausePool`, which sets `pauseLevel = 2`. The `swap` function is gated by `whenNotPaused`, so swaps halt. However, `executePoolPriceProviderUpdate` carries no equivalent guard:

```solidity
// MetricOmmPoolFactory.sol lines 494-507
function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    // ← no pause check
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);   // writes priceProvider slot
    ...
}
```

`setPriceProvider` on the pool is equally unchecked:

```solidity
// MetricOmmPool.sol lines 477-480
function setPriceProvider(address newPriceProvider) external onlyFactory {
    priceProvider = newPriceProvider;   // no pause guard
    emit PriceProviderUpdated(newPriceProvider);
}
```

After execution the pool holds a malicious oracle. The factory owner cannot revert this: only the pool admin can propose and execute oracle changes. The factory owner can keep the pool at `pauseLevel = 2` (the pool admin cannot unpause from level 2 — `unpausePool` requires `cur == 1`), but that permanently disables swaps. The pool is irrecoverable without pool-admin cooperation.

The asymmetry is explicit in the interface: admin-transfer proposals have `cancelPoolAdminTransfer`; price-provider proposals have no equivalent.

### Impact Explanation
A compromised pool admin can install an arbitrary oracle (e.g., one returning `bid == ask`, zero prices, or extreme prices) and leave the factory owner with no recourse other than permanent protocol-pause. Swaps become permanently unusable for all LPs and traders on that pool. LPs can still call `removeLiquidity` (pause only blocks `swap`), so principal is not directly lost, but the pool's core swap functionality is irreversibly broken.

### Likelihood Explanation
Requires the pool admin key to be compromised or the pool admin to act maliciously. Pool admins are semi-trusted; the timelock mechanism exists precisely to give the factory owner reaction time. The absence of a cancel function nullifies that protection: the factory owner can pause but cannot stop the rotation from landing.

### Recommendation
Add a `cancelPoolPriceProviderProposal` callable by the factory owner (and optionally the pool admin) that clears `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]`. Mirror the pattern already used for admin transfers:

```solidity
function cancelPoolPriceProviderProposal(address pool) external onlyOwner {
    if (pendingPriceProvider[pool] == address(0)) revert NoPriceProviderChangeProposed();
    address cancelled = pendingPriceProvider[pool];
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderProposalCancelled(pool, cancelled);
}
```

Alternatively, add a pause check inside `executePoolPriceProviderUpdate` so a protocol-paused pool cannot have its oracle rotated.

### Proof of Concept

```
1. Pool deployed with priceProviderTimelock = 1 days (mutable oracle).
2. Pool admin key is compromised.
3. Attacker calls factory.proposePoolPriceProvider(pool, maliciousOracle).
   → pendingPriceProvider[pool] = maliciousOracle
   → pendingPriceProviderExecuteAfter[pool] = block.timestamp + 1 days
4. Factory owner detects the proposal and calls factory.protocolPausePool(pool).
   → pauseLevel = 2; swaps revert PoolPaused.
5. 1 day elapses.
6. Attacker calls factory.executePoolPriceProviderUpdate(pool).
   → No pause check; executes successfully.
   → pool.priceProvider = maliciousOracle.
7. Factory owner cannot call proposePoolPriceProvider (not pool admin).
   Factory owner cannot call executePoolPriceProviderUpdate (not pool admin).
   Factory owner can only keep pauseLevel = 2.
8. Pool is permanently paused with a malicious oracle installed.
   All swap functionality is permanently disabled.
