### Title
`isProtocolParticipant` Passes Leaving/Joining Nodes as Authorized Protocol Participants - (`File: network/underlay/network.go`)

### Summary

`isProtocolParticipant()` in `network/underlay/network.go` and `notEjectedPeerFilter()` in `network/p2p/builder/utils.go` both gate network-layer access using only `id.IsEjected()`. This is the direct analog of the `_hasFundableKeys` bug: a weaker condition (`NotEjected`) is used instead of the correct condition (`Active`), causing nodes with `EpochParticipationStatusLeaving` or `EpochParticipationStatusJoining` to pass the eligibility check when they should be rejected.

### Finding Description

`isProtocolParticipant()` is the peer filter applied to every non-public pubsub channel subscription. It is supposed to admit only authorized, staked protocol participants:

```go
// isProtocolParticipant returns a PeerFilter that allows if a peer is a staked (i.e., authorized) node.
func (n *Network) isProtocolParticipant() p2p.PeerFilter {
    return func(p peer.ID) error {
        id, ok := n.Identity(p)
        if !ok { ... }
        if id.IsEjected() {
            return fmt.Errorf("peer with peer id %s is ejected", ...)
        }
        return nil   // ← Leaving and Joining nodes reach here and are ALLOWED
    }
}
``` [1](#0-0) 

`id.IsEjected()` is equivalent to `HasParticipationStatus(EpochParticipationStatusEjected)`. The four possible statuses are:

```
EpochParticipationStatusJoining  (0)
EpochParticipationStatusActive   (1)
EpochParticipationStatusLeaving  (2)
EpochParticipationStatusEjected  (3)
``` [2](#0-1) 

So `isProtocolParticipant()` returns `nil` (allows) for `Joining`, `Active`, **and `Leaving`** nodes. The correct filter for "authorized current-epoch participant" is `IsValidCurrentEpochParticipant`, which is defined as:

```go
var IsValidCurrentEpochParticipant = HasParticipationStatus(flow.EpochParticipationStatusActive)
``` [3](#0-2) 

The same flaw exists in `notEjectedPeerFilter()`, which is used as the connection gater filter for both `InterceptPeerDial` and `InterceptSecured`:

```go
func notEjectedPeerFilter(idProvider module.IdentityProvider) p2p.PeerFilter {
    return func(p peer.ID) error {
        if id, found := idProvider.ByPeerID(p); !found {
            return fmt.Errorf(...)
        } else if id.IsEjected() {
            return fmt.Errorf(...)
        }
        return nil  // ← Leaving and Joining nodes pass
    }
}
``` [4](#0-3) 

This filter is wired into the connection gater for both outbound and inbound connections:

```go
peerFilter := notEjectedPeerFilter(idProvider)
peerFilters := []p2p.PeerFilter{peerFilter}
connGater := connection.NewConnGater(
    logger, idProvider,
    connection.WithOnInterceptPeerDialFilters(append(peerFilters, ...)),
    connection.WithOnInterceptSecuredFilters(append(peerFilters, ...)))
``` [5](#0-4) 

`isProtocolParticipant()` is then used as the peer filter for all non-public channel subscriptions:

```go
} else {
    validators = append(validators, n.authorizedSenderValidator.PubSubMessageValidator(channel))
    peerFilter = n.isProtocolParticipant()
}
``` [6](#0-5) 

The `EpochParticipationStatusLeaving` status is explicitly defined as nodes that **were active in the previous epoch but are NOT active in the current epoch**:

```
// leaving - the node was active in the previous epoch but is not active in the current epoch.
``` [7](#0-6) 

These nodes appear in the identity table during the staking phase of the new epoch with `Leaving` status. They have valid, known networking keys. They are not ejected. They pass both `notEjectedPeerFilter` (connection gating) and `isProtocolParticipant` (pubsub peer filter).

The correct filter — `IsValidCurrentEpochParticipant` — is used in higher-level protocol logic (e.g., `authorizedVerifiersAtBlock`, `ensureNodeHasWeightAndRole`, `IsNodeAuthorizedAt`) but is **not** used at the network layer peer filters where it is the first and intended line of defense. [8](#0-7) [9](#0-8) 

### Impact Explanation

A node operator whose node transitions to `EpochParticipationStatusLeaving` (i.e., it was active in epoch N but did not re-register for epoch N+1) retains valid networking keys and a known identity. Because `notEjectedPeerFilter` only rejects `Ejected` nodes, a `Leaving` node can:

1. **Establish inbound and outbound connections** to active staked nodes (connection gater passes it).
2. **Publish messages on all non-public protocol channels** (pubsub peer filter `isProtocolParticipant` passes it).
3. **Have those messages reach the engine-level validators** of active nodes.

This is unauthorized network-layer access: a node that is no longer a valid current-epoch participant is treated as one by the peer filter layer. The `authorizedSenderValidator` provides a second layer, but the peer filter — the intended first gate — is incorrect, mirroring exactly the `_hasFundableKeys` pattern where the eligibility check returns "eligible" for entities that are at their limit.

### Likelihood Explanation

This is reachable by any node operator whose node transitions to `Leaving` status at an epoch boundary — a normal, expected protocol event requiring no special privileges, no key compromise, and no admin access. The node operator simply does not re-register for the next epoch. The `Leaving` status is a designed part of the identity lifecycle and the node's keys remain valid.

### Recommendation

Replace `id.IsEjected()` with `filter.IsValidCurrentEpochParticipant(id)` (i.e., check for `EpochParticipationStatusActive`) in both `isProtocolParticipant()` and `notEjectedPeerFilter()`. This aligns the network-layer peer filters with the higher-level protocol authorization checks already in use throughout the codebase.

```go
// Corrected isProtocolParticipant
func (n *Network) isProtocolParticipant() p2p.PeerFilter {
    return func(p peer.ID) error {
        id, ok := n.Identity(p)
        if !ok { ... }
        if !filter.IsValidCurrentEpochParticipant(id) {
            return fmt.Errorf("peer %s is not an active current-epoch participant (status: %s)", ...)
        }
        return nil
    }
}
```

### Proof of Concept

1. Node A is active in epoch N with `EpochParticipationStatusActive`.
2. Node A does not re-register for epoch N+1. At the epoch boundary, its status becomes `EpochParticipationStatusLeaving`.
3. Node A's networking key is still valid and known to the identity provider.
4. Node A calls `notEjectedPeerFilter`: `id.IsEjected()` returns `false` → connection allowed.
5. Node A publishes a message on a non-public channel (e.g., `channels.ConsensusCommittee`).
6. Active nodes receive the message; `isProtocolParticipant()` is called: `id.IsEjected()` returns `false` → peer filter passes.
7. The message proceeds to `authorizedSenderValidator` — Node A has bypassed the intended first authorization gate. [10](#0-9) [4](#0-3) [11](#0-10) [3](#0-2)

// EpochParticipationStatus represents the status of a node's participation. Depending on what
// changes were applied to the protocol state, a node may be in one of four states:
// /   - joining - the node is not active in the current epoch and will be active in the next epoch.
// /   - active - the node was included in the EpochSetup event for the current epoch and is actively participating in the current epoch.
// /   - leaving - the node was active in the previous epoch but is not active in the current epoch.
// /   - ejected - the node has been permanently removed from the network.
//
// /            EpochSetup
// /	      ┌────────────⬤ unregistered ◯◄───────────┐

// /                         └───────────┘

func notEjectedPeerFilter(idProvider module.IdentityProvider) p2p.PeerFilter {
	return func(p peer.ID) error {
		if id, found := idProvider.ByPeerID(p); !found {
			return fmt.Errorf("failed to get identity of unknown peer with peer id %s", p2plogging.PeerId(p))
		} else if id.IsEjected() {
			return fmt.Errorf("peer %s with node id %s is ejected", p2plogging.PeerId(p), id.NodeID.String())
		}

		return nil
	}
}

func ensureNodeHasWeightAndRole(identity *flow.Identity, expectedRole flow.Role) error {
