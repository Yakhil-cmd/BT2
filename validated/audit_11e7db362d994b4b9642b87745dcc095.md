### Title
Malicious Coordinator Can Substitute Commitments in FROST EdDSA v1 Signing Package Without Participant Verification — (File: `src/frost/eddsa/sign.rs`)

---

### Summary

In the FROST EdDSA v1 signing protocol (`sign_v1`), participants send their nonce commitments to the coordinator in Round 1, then receive a `SigningPackage` from the coordinator in Round 2. The participant verifies only that the message in the signing package matches the expected message, but does **not** verify that the commitments in the signing package match what was actually sent in Round 1. A malicious coordinator can substitute any participant's commitment in the signing package, causing all participants to compute signature shares using a different commitments set than what they generated — bypassing the FROST binding factor protection.

---

### Finding Description

In `do_sign_participant_v1` (`src/frost/eddsa/sign.rs`, lines 234–302), the protocol proceeds as follows:

**Round 1** — the participant generates fresh nonces and commitments, then sends the commitments to the coordinator:

```rust
let (nonces, commitments) = round1::commit(&signing_share, rng);
let nonces = Zeroizing::new(nonces);
let commit_waitpoint = chan.next_waitpoint();
chan.send_private(commit_waitpoint, coordinator, &commitments)?;
``` [1](#0-0) 

**Round 2** — the participant receives a `SigningPackage` from the coordinator and validates only the message field, not the commitments:

```rust
let signing_package = loop {
    let (from, signing_package): (_, frost_ed25519::SigningPackage) =
        chan.recv(r2_wait_point).await?;
    if from != coordinator { continue; }
    break signing_package;
};

if signing_package.message() != message.as_slice() {
    return Err(ProtocolError::AssertionFailed(...));
}
``` [2](#0-1) 

The participant then immediately signs using the coordinator-supplied (potentially manipulated) commitments map:

```rust
let signature_share = round2::sign(&signing_package, &nonces, &key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
``` [3](#0-2) 

There is no check that the commitments in `signing_package` match what the participant sent in Round 1. The FROST binding factor ρ_i = H1(X, h, commits, i) is computed inside `round2::sign` from the full commitments map embedded in the signing package. If the coordinator substitutes any commitment, all binding factors change, and every participant computes signature shares with binding factors derived from fake commitments — not the ones they generated.

The coordinator's own path (`do_sign_coordinator_v1`) constructs the signing package from the commitments it received and broadcasts it without any tamper-evident mechanism:

```rust
let signing_package = frost_ed25519::SigningPackage::new(commitments_map, message.as_slice());
// ...
chan.send_many(r2_wait_point, &signing_package)?;
``` [4](#0-3) 

The coordinator is free to insert arbitrary commitment values before broadcasting.

---

### Impact Explanation

The FROST binding factor exists precisely to prevent the coordinator from manipulating the signing session. RFC 9591 (Section 5.2) requires each participant to verify that their own identifier and commitment appear correctly in the signing package before proceeding. This codebase omits that check.

By substituting commitments in the signing package, a malicious coordinator can:

1. **Break nonce binding**: Participants' nonces (a_i, b_i) are bound to fake commitments, not the real ones. The binding factor ρ_i is computed from the manipulated commitments map, so the nonce-to-session binding that FROST's security proof relies on is severed.

2. **Enable Wagner-style multi-session attacks**: By running multiple signing sessions with different fake commitment maps, the coordinator collects signature shares computed under different binding factors for the same nonces. This is exactly the attack scenario that the FROST binding factor is designed to prevent. Across enough sessions, the coordinator can potentially forge signatures for attacker-chosen messages.

3. **Corrupt the signing transcript**: Honest participants accept and sign an inconsistent transcript — the commitments they signed with differ from what they generated and sent. The resulting signature is valid on-chain but was produced outside the security model.

This maps to: **High — Corruption of sign outputs so honest parties accept inconsistent transcripts.**

---

### Likelihood Explanation

The attack requires no cryptographic breaks. A malicious coordinator simply replaces one or more commitment values in the `commitments_map` before constructing the `SigningPackage` and broadcasting it. The coordinator is a designated, reachable role in every `sign_v1` invocation. No special access beyond being the coordinator is needed. The missing check is a single absent validation step that any coordinator-role attacker can exploit in every signing session.

---

### Recommendation

After receiving the signing package from the coordinator in `do_sign_participant_v1`, verify that the participant's own commitment in the signing package matches what was sent in Round 1:

```rust
let my_id = me.to_identifier()?;
if signing_package.signing_commitments().get(&my_id) != Some(&commitments) {
    return Err(ProtocolError::AssertionFailed(
        "Commitment in signing package does not match sent commitment".to_string(),
    ));
}
```

Additionally, verify that the commitments map contains exactly the expected participant set (matching the `participants` list) and no extra or missing entries. This mirrors the validation required by RFC 9591 Section 5.2 and restores the binding guarantee that FROST's security proof depends on.

---

### Proof of Concept

1. Set up a `sign_v1` session with participants P1 (coordinator), P2, P3.
2. P2 and P3 send their real commitments (A2, B2) and (A3, B3) to P1 in Round 1.
3. P1 (malicious coordinator) constructs a `SigningPackage` with fake commitments (A2′, B2′) substituted for P2's real (A2, B2).
4. P1 broadcasts this manipulated signing package to all participants via `chan.send_many`.
5. P2 receives the signing package, checks only `signing_package.message() == message` (passes), and calls `round2::sign` with the fake commitments map — computing its share under ρ2′ = H1(X, h, {(A2′, B2′), (A3, B3)}, 2) instead of the real ρ2.
6. P3 similarly computes its share under the same fake binding factors.
7. P1 aggregates all shares into a valid signature.
8. The signature verifies correctly, but was produced with fake commitments for P2, breaking the FROST binding factor protection.

The coordinator can repeat this across multiple sessions with different fake commitment maps, collecting signature shares under different binding factors — exactly the multi-session attack that FROST's binding factor is designed to prevent.
