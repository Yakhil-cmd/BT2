### Title
Hardcoded 2,300-gas cap on `DepositCallGasLimit` prevents EVM smart contracts with complex `receive`/`fallback` logic from receiving FLOW tokens via Cadence deposit - (`File: fvm/evm/types/call.go`)

### Summary

`NewDepositCall` in Flow EVM hardcodes a gas limit of `21,000 + 2,300 = 23,300` for all Cadence-initiated FLOW token deposits to EVM addresses. Because 21,000 is consumed as intrinsic transaction cost, only 2,300 gas remains for the target contract's `receive()` or `fallback()` function — the same stipend as Solidity's deprecated `transfer()`. Any EVM smart contract whose `receive`/`fallback` requires more than 2,300 gas will have every Cadence-side deposit revert, permanently breaking that integration path.

### Finding Description

In `fvm/evm/types/call.go`, the constants are defined as:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

`NewDepositCall` unconditionally stamps this limit onto every deposit `DirectCall`:

```go
func NewDepositCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: DepositCallGasLimit,   // always 23_300
    }
}
``` [2](#0-1) 

This `DirectCall` is constructed inside `Account.Deposit()` in the handler: [3](#0-2) 

Which is invoked from the Cadence-level `InternalEVM.deposit` host function: [4](#0-3) 

Which is the implementation behind the public Cadence `EVMAddress.deposit(from:)` and `CadenceOwnedAccount.deposit(from:)` entry points: [5](#0-4) [6](#0-5) 

The existing test already documents the revert outcome for a contract that does not implement a payable receive: [7](#0-6) 

### Impact Explanation

Any EVM smart contract deployed on Flow EVM whose `receive()` or `fallback()` function consumes more than 2,300 gas (e.g., emitting an event, writing to storage, calling another contract) will have every Cadence-initiated deposit revert. The FLOW tokens are returned to the sender, but the integration is permanently broken: there is no way for the contract author or the depositor to override the gas limit because `DepositCallGasLimit` is a compile-time constant applied unconditionally. This silently breaks any protocol or application that relies on deposit-triggered on-chain logic (e.g., vault accounting, liquidity pool crediting, event-driven automation).

### Likelihood Explanation

The entry path is fully unprivileged: any Cadence transaction or script can call `EVMAddress.deposit(from:)`. The affected class of contracts — those with non-trivial `receive`/`fallback` — is common in DeFi (ERC-4626 vaults, liquidity pools, yield aggregators). The 2,300-gas stipend has been a known footgun in Ethereum since Istanbul repriced `SSTORE`, and the EVM ecosystem has largely moved away from it. Any developer porting such a contract to Flow EVM and expecting Cadence-side deposits to trigger their receive logic will encounter silent reverts.

### Recommendation

Allow the caller to pass an explicit `gasLimit` parameter to `EVMAddress.deposit(from:gasLimit:)` and propagate it through `NewDepositCall`. Where backward compatibility requires a default, use a higher value (e.g., the full transaction gas budget minus intrinsic cost) rather than the 2,300-stipend pattern. The `NewTransferCall` path has the same issue via `DefaultGasLimitForTokenTransfer`. [8](#0-7) 

### Proof of Concept

1. Deploy an EVM contract on Flow EVM with a `receive()` that writes to storage (costs >2,300 gas).
2. From a Cadence transaction, call `evmAddress.deposit(from: <-vault)` targeting that contract.
3. Observe `res.VMError == gethVM.ErrExecutionReverted` — the deposit reverts despite the contract being designed to accept FLOW tokens.
4. The root cause is `GasLimit: DepositCallGasLimit` (23,300) in `NewDepositCall`, leaving only 2,300 gas after the 21,000 intrinsic cost for the contract's receive logic. [9](#0-8)

func newInternalEVMTypeDepositFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeDepositFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			// Get from vault

			fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amountValue, ok := fromValue.GetField(
				context,
				fungibleTokenVaultTypeBalanceFieldName,
			).(interpreter.UFix64Value)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))

			// Get to address

			toAddressValue, ok := invocation.Arguments[1].(*interpreter.ArrayValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			toAddress, err := AddressBytesArrayValueToEVMAddress(context, toAddressValue)
			if err != nil {
				panic(err)
			}

			// NOTE: We're intentionally not destroying the vault here,
			// because the value of it is supposed to be "kept alive".
			// Destroying would incorrectly be equivalent to a burn and decrease the total supply,
			// and a withdrawal would then have to perform an actual mint of new tokens.

			// Deposit

			const isAuthorized = false
			account := handler.AccountByAddress(toAddress, isAuthorized)
			account.Deposit(types.NewFlowTokenVault(amount))

			return interpreter.Void
		},
	)
