### Title
Missing Failure Broadcast Implementation in `broadcast_success` Enables Malicious Participant to Permanently Stall DKG/Reshare/Refresh for All Honest Parties - (File: src/dkg.rs)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` is explicitly documented to accept an error parameter and broadcast failure when an error is detected, but this parameter was never implemented. The function always broadcasts success. When a malicious participant causes any honest party to abort before reaching `broadcast_success`, the remaining honest participants are left waiting indefinitely inside `do_broadcast`, permanently denying key generation.

---

### Finding Description

The `broadcast_success` function carries the following documentation: [1](#0-0) 

```
/// This function takes err as input.
/// If err is None then broadcast success
/// otherwise, broadcast failure
/// If during broadcast it receives an error then propagates it
/// This function is used in the final round of DKG
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
```

The documented `err` parameter is entirely absent from the function signature. The body unconditionally broadcasts `(true, session_id)` — the failure broadcast path is never implemented: [2](#0-1) 

This is the direct analog to the external report: a mechanism is documented and its checks exist, but the function to set the failure state was never added.

`broadcast_success` is called as the final step of `do_keyshare` only after all verification passes: [3](#0-2) 

Before this call, several verification steps can abort a participant early:

- `verify_proof_of_knowledge` at line 452 — returns `ProtocolError::InvalidProofOfKnowledge` or `ProtocolError::MaliciousParticipant`
- `verify_commitment_hash` at line 463 — returns `ProtocolError::InvalidCommitmentHash`
- `validate_received_share` at line 522 — returns `ProtocolError::InvalidSecretShare` [4](#0-3) 

When any of these return an error, the participant exits `do_keyshare` immediately without ever calling `broadcast_success`. The remaining honest participants are blocked inside `do_broadcast` waiting for that participant's final-round message, which never arrives.

`reliable_broadcast_receive_all` loops indefinitely until all `n` participants have delivered their broadcast: [5](#0-4) 

There is no timeout or partial-completion path. If one participant never sends, the loop never exits.

---

### Impact Explanation

**High — Permanent denial of key generation, resharing, and refresh for honest parties.**

A malicious participant can send an invalid secret share to exactly one honest participant. That honest participant detects `InvalidSecretShare`, aborts before `broadcast_success`, and never sends its final-round broadcast. All remaining honest participants are permanently blocked inside `reliable_broadcast_receive_all`. Because the failure broadcast path was never implemented, there is no mechanism for the aborting participant to signal failure to others, and no mechanism for others to detect the abort and exit cleanly.

This affects `do_keygen`, `do_reshare`, and the key-refresh path, all of which call `do_keyshare`. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

**High.** Any single malicious participant in the protocol can trigger this with a single crafted message (an invalid share sent to one honest party). No special privileges, leaked keys, or external assumptions are required. The attacker only needs to be a registered participant in the DKG/reshare/refresh session.

---

### Recommendation

Implement the documented failure broadcast path. Add an `err: Option<ProtocolError>` parameter to `broadcast_success`. When `err` is `Some`, broadcast `(false, session_id)` before returning the error. Honest participants receiving any `false` vote should abort immediately rather than waiting. This mirrors the design already described in the function's own documentation and closes the permanent-stall vector.

---

### Proof of Concept

1. Participants `{A, B, C}` run `do_keygen`.
2. Malicious participant `M` sends a correctly-formed commitment and proof of knowledge (passes rounds 2–4), but sends an invalid secret share to participant `A` in round 4 (`wait_round_3`).
3. `A` calls `validate_received_share` at line 522, receives `ProtocolError::InvalidSecretShare(M)`, and returns immediately — never reaching `broadcast_success` at line 531.
4. `B` and `C` complete all verifications successfully and enter `broadcast_success` → `do_broadcast` → `reliable_broadcast_receive_all`.
5. `B` and `C` loop forever waiting for `A`'s final-round broadcast, which never arrives.
6. Key generation is permanently denied for all honest parties. [8](#0-7) [9](#0-8)

pub async fn do_reshare<C: Ciphersuite>(
    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
    // either extract the share and linearize it or set it to zero
    let secret = old_signing_key
        .map(|x_i| {
            intersection
                .lagrange::<C>(me)
                .map(|lambda| lambda * x_i.to_scalar())
        })
        .transpose()?
        .unwrap_or_else(<C::Group as Group>::Field::zero);

    let old_reshare_package = Some((old_public_key, old_participants));
    let keygen_output = do_keyshare::<C>(
        chan,
        participants,
        me,
        threshold,
        secret,
        old_reshare_package,
        &mut rng,
    )
    .await?;

    Ok(keygen_output)
