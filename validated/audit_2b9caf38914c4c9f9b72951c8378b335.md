### Title
`MsgTurnBridge` Handler Is a No-Op — Permission Check Completely Bypassed and Bridge Circuit Breaker Non-Functional - (File: x/cronos/keeper/msg_server.go)

### Summary

The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is an empty stub that returns `nil, nil` unconditionally. It never checks the `CanTurnBridge` permission, never reads `msg.Enable`, and never writes any state. Any unprivileged user can call `MsgTurnBridge` and receive a success response (code 0), bypassing the `CanTurnBridge` authorization check entirely. Simultaneously, the bridge circuit breaker is permanently non-functional: no authorized party can ever disable the bridge.

### Finding Description

`MsgTurnBridge` is defined in the proto service as the mechanism to "disable or enable the gravity bridge." A dedicated permission bit `CanTurnBridge` exists in the permission system, and the CLI command `turn-bridge` is wired to it. However, the server-side handler is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this with `UpdateTokenMapping`, which correctly gates on `CanChangeTokenMapping`:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

And `UpdatePermissions`, which gates on admin:

```go
if admin != msg.From {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [3](#0-2) 

The `CanTurnBridge` constant is defined and the permission infrastructure is fully in place:

```go
const (
    CanChangeTokenMapping uint64 = 1 << iota // 1
    CanTurnBridge                             // 2
    All = CanChangeTokenMapping | CanTurnBridge // 3
)
``` [4](#0-3) 

The proto definition confirms `MsgTurnBridge` is a live, registered message surface: [5](#0-4) 

### Impact Explanation

Two distinct impacts arise from the same root cause:

1. **Permission bypass (High):** Any unprivileged user can submit `MsgTurnBridge` with any `enable` value and receive a success response. The `CanTurnBridge` authorization check — which exists in the permission system and is queryable on-chain — is never enforced. This directly satisfies: *"High: Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

2. **Permanent bridge circuit-breaker failure (High):** Because the handler writes no state, the bridge can never be disabled. If a vulnerability in the gravity bridge or CRC21 bridge path is exploited (e.g., unauthorized minting or draining), the intended emergency stop mechanism is silently inoperative. This satisfies: *"High: Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows … under normal network assumptions."*

### Likelihood Explanation

The entry path is a standard, publicly documented Cosmos SDK message (`/cronos.Msg/TurnBridge`) reachable by any account with gas. No special privilege, leaked key, or cryptographic break is required. The no-op behavior is silent — callers receive `code: 0` and have no indication the bridge state was not changed.

### Recommendation

Implement the `TurnBridge` handler body:

1. Verify the sender holds `CanTurnBridge` permission (mirroring the pattern in `UpdateTokenMapping`).
2. Read `msg.Enable` and persist the bridge-enabled flag to the module's KV store (e.g., as a new `Params` field or a dedicated key).
3. In all EVM log handlers that trigger bridge operations (`SendToIbcHandler`, `SendToIbcV2Handler`, gravity bridge handler), read the persisted flag and return an error if the bridge is disabled.
