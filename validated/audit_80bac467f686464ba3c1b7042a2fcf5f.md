### Title
Missing Zero-Address Check in Bank Precompile `mint` and `transfer` Methods Allows Permanent Destruction of Precompile-Controlled Assets - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The bank precompile's `mint` and `transfer` methods accept a caller-supplied `recipient` address without validating it is non-zero. The only guard, `checkBlockedAddr`, only rejects Cosmos module-account addresses — not `address(0)`. Any unprivileged EVM contract can call these methods with `address(0)` as recipient, permanently destroying `evm/<caller>` precompile-controlled assets.

### Finding Description
In `bank.go`, both the `MintMethodName` branch (lines 113–156) and the `TransferMethodName` branch (lines 167–200) extract a `recipient` (or `to`) address from caller-supplied ABI arguments and pass it directly to the Cosmos bank keeper without a zero-address guard. [1](#0-0) 

The only validation applied to the destination address is `checkBlockedAddr`: [2](#0-1) 

`checkBlockedAddr` calls `bc.bankKeeper.BlockedAddr(to)`, which only returns `true` for registered Cosmos module accounts. The zero address (`0x000…000`) is not a module account and is not blocked, so `checkBlockedAddr(zero_address)` returns `nil` and execution continues.

For `mint`: any EVM contract can call `bank.mint(address(0), amount)`, causing `MintCoins` + `SendCoinsFromModuleToAccount` to credit `evm/<caller>` tokens to the zero address. [3](#0-2) 

For `transfer`: any EVM contract can call `bank.transfer(victim, address(0), amount)`, causing `SendCoins` to move `evm/<caller>` tokens from `victim` to the zero address. [4](#0-3) 

Note that in `transfer`, `sender` is taken from `args[0]` (not `contract.Caller()`), so the calling contract can specify any victim as the source. [5](#0-4) 

### Impact Explanation
Tokens of denom `evm/<caller>` are precompile-controlled assets. Once sent to the zero address via `SendCoins` or `SendCoinsFromModuleToAccount`, they are permanently inaccessible — no private key controls the zero address on Cosmos. This constitutes an irreversible accounting change (permanent burn without authorization) for precompile-controlled assets, matching the Critical/High impact scope.

### Likelihood Explanation
The entry path is fully unprivileged: any EVM contract deployed on Cronos can call the bank precompile at address `0x0000…0064`. No admin role, governance vote, or token mapping is required. The attacker only needs to deploy a contract that calls `bank.mint(address(0), N)` or `bank.transfer(victim, address(0), N)`.

### Recommendation
Add an explicit zero-address guard in both the `mint` and `transfer` branches before any bank keeper call:

```go
if recipient == (common.Address{}) {
    return nil, errors.New("recipient cannot be zero address")
}
```

This mirrors the pattern already used in `validateIsEvmAddresses` in `x/cronos/types/params.go`: [6](#0-5) 

### Proof of Concept
1. Deploy attacker contract `X` on Cronos EVM.
2. From `X`, call the bank precompile at `0x0000…0064` with selector for `mint(address,uint256)` and arguments `(address(0), 1e18)`.
3. `checkBlockedAddr(address(0))` returns `nil` (zero address is not a module account).
4. `MintCoins` + `SendCoinsFromModuleToAccount` credit `1e18 evm/X` tokens to the zero address — permanently inaccessible.
5. Alternatively, if users hold `evm/X` balances, call `transfer(victim, address(0), balance)` to drain and permanently destroy their holdings. [7](#0-6) [8](#0-7)

func (bc *BankContract) checkBlockedAddr(addr sdk.AccAddress) error {
	to, err := sdk.AccAddressFromBech32(addr.String())
	if err != nil {
		return err
	}
	if bc.bankKeeper.BlockedAddr(to) {
		return errorsmod.Wrapf(errortypes.ErrUnauthorized, "%s is not allowed to receive funds", to.String())
	}
	return nil
}
