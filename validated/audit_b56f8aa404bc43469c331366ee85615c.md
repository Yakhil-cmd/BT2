### Title
Incomplete `REWARD_ADDRESS_IS_TOKEN` Validation in BTC Pool Allows Permanent Freezing of STRK Rewards — (`src/pool/pool.cairo`)

---

### Summary

The `Pool` contract's `enter_delegation_pool` and `change_reward_address` functions validate that the `reward_address` is not a token contract, but only check against the pool's own staking token. For BTC pools, this means the STRK reward token address is never checked. Because `claim_rewards` always pays out in STRK regardless of pool type, a BTC pool member can set `reward_address = STRK_TOKEN_ADDRESS`, causing all future STRK reward claims to be transferred into the STRK token contract itself, permanently freezing them.

---

### Finding Description

In `pool.cairo`, `enter_delegation_pool` performs the following guard:

```cairo
// pool.cairo:193-195
let token_address = token_dispatcher.contract_address;
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [1](#0-0) 

`token_address` here is the pool's staking token — BTC for a BTC pool. The check `token_address != reward_address` therefore only prevents `reward_address` from equalling the BTC token address. It does **not** prevent `reward_address` from being set to `STRK_TOKEN_ADDRESS`.

The same incomplete check exists in `change_reward_address`:

```cairo
// pool.cairo:507-510
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

However, `claim_rewards` always pays out in STRK, regardless of the pool's staking token:

```cairo
// pool.cairo:365-366
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [3](#0-2) 

By contrast, the staking contract's `stake` function uses `does_token_exist`, which checks against **all** registered tokens (STRK and BTC), not just one:

```cairo
// staking.cairo:307-311
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [4](#0-3) 

The pool contract's check is weaker: it only guards against the pool's own token, not the STRK reward token.

---

### Impact Explanation

A BTC pool member who sets `reward_address = STRK_TOKEN_ADDRESS` (either at entry or via `change_reward_address`) will cause every subsequent `claim_rewards` call to transfer STRK into the STRK token contract itself. The STRK ERC20 contract has no recovery mechanism for tokens sent to its own address. All accumulated STRK rewards for that pool member are permanently frozen.

This matches the allowed impact: **Permanent freezing of unclaimed yield** (High).

---

### Likelihood Explanation

The entry path is fully unprivileged. Any BTC pool member can call `enter_delegation_pool` or `change_reward_address` with `reward_address = STRK_TOKEN_ADDRESS`. The check `token_address != reward_address` passes because the pool's token is BTC, not STRK. No privileged role, leaked key, or external dependency is required. The scenario is reachable by any delegator in any BTC pool.

---

### Recommendation

In both `enter_delegation_pool` and `change_reward_address` in `pool.cairo`, add an explicit check that `reward_address != STRK_TOKEN_ADDRESS` (or, better, mirror the staking contract's approach and check against all registered tokens). For example:

```cairo
assert!(
    reward_address != STRK_TOKEN_ADDRESS,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

---

### Proof of Concept

1. A staker opens a BTC pool via `set_open_for_delegation(token_address: BTC_TOKEN_ADDRESS)`.
2. A delegator calls `pool.enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)`.
   - Check `token_address != reward_address` → `BTC_TOKEN_ADDRESS != STRK_TOKEN_ADDRESS` → **passes**.
3. Rewards are distributed via `update_rewards_from_attestation_contract` or `update_rewards`, which calls `send_rewards_to_delegation_pool` and then `update_rewards_from_staking_contract` on the pool.
4. Delegator calls `pool.claim_rewards(pool_member: delegator)`.
   - `reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS }`.
   - `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
   - STRK rewards are transferred **into** the STRK token contract itself and are permanently unrecoverable.
