### Title
Signature Verification Bypass via Overly Broad `skip_stateful_validations` Check Admits Unsigned Invoke Transactions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is designed to skip the `__validate__` entry point (signature verification) for an invoke transaction with nonce=1 when the account's on-chain nonce is 0 and a `deploy_account` is pending — a UX feature allowing users to submit `deploy_account + invoke` simultaneously. However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from the address in the mempool or recent block, not specifically a `deploy_account`. An unprivileged attacker who observes a pending `deploy_account` in the mempool can submit a forged invoke (nonce=1, victim's address, arbitrary calldata, invalid signature) that the gateway admits without signature verification.

The guard `account_tx_in_pool_or_recent_block` checks:

`contains_account` in `MempoolState` returns `true` for any address in `staged` or `committed` — it does not distinguish `deploy_account` from invoke transactions:

```rust
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [5](#0-4) 

The code comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is flawed: an account deployed by a factory contract (not via `deploy_account`) can have on-chain nonce=0 with a pending nonce=0 invoke in the pool. An attacker can then submit a nonce=1 invoke for that address and bypass `__validate__`. [6](#0-5) 

**Attack path (most impactful variant):**

1. Attacker monitors the public mempool and finds address `X` with a pending `deploy_account` (nonce=0).
2. Attacker crafts an invoke transaction: `sender_address=X`, `nonce=1`, arbitrary calldata, invalid/attacker-controlled signature, with fees higher than any legitimate nonce=1 invoke already in the pool.
3. Gateway checks: `tx.nonce()==1` ✓, `account_nonce==0` ✓, `account_tx_in_pool_or_recent_block(X)==true` ✓ → `skip_validate=true`.
4. `run_validate_entry_point` runs with `validate=false` — `__validate__` is never called. Transaction is admitted to the mempool.
5. Mempool's fee-escalation logic (`validate_fee_escalation`) replaces the legitimate user's nonce=1 invoke with the attacker's forged one.
6. At execution time, the batcher creates `AccountTransaction::new_for_sequencing` with `validate: true`, so `__validate__` runs and the forged transaction reverts — but the legitimate user's invoke has already been evicted from the mempool. [7](#0-6) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway admits an invoke transaction with an invalid (attacker-controlled) signature without running `__validate__`. The forged transaction occupies the nonce=1 slot in the mempool. Via fee escalation, it can displace the legitimate user's nonce=1 invoke. The legitimate user's `deploy_account` succeeds but their intended first invoke is silently dropped. The forged transaction reverts at execution time (the batcher re-enables `validate: true`), but the damage — eviction of the legitimate transaction — is already done.

Additionally, if the account has sufficient balance after deployment, the reverted forged transaction charges fees from the victim's account for a transaction they never signed.

### Likelihood Explanation

**Medium.** The mempool is observable by any network participant. An attacker needs only to:
1. Watch for `deploy_account` transactions (trivially detectable by transaction type).
2. Submit a forged nonce=1 invoke with higher fees before the `deploy_account` is committed.

The window is the time between the `deploy_account` entering the mempool and being included in a block. No privileged access, no special knowledge, and no cooperation from the victim is required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction (not just any transaction) is pending for the address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type. Alternatively, store the transaction type alongside the address in `MempoolState` so the gateway can distinguish the intended UX case from an arbitrary pending transaction.

### Proof of Concept

```
1. Deploy a contract factory that deploys account contract at address X (on-chain nonce of X = 0).
2. Submit a legitimate invoke from X with nonce=0 to the mempool (passes __validate__).
   → mempool.tx_pool.contains_account(X) == true
3. Attacker submits:
     InvokeV3 {
       sender_address: X,
       nonce: 1,
       calldata: [<drain_funds_calldata>],
       signature: [0xdeadbeef],   // invalid
       resource_bounds: { l2_gas: { max_amount: HIGH, max_price: HIGH } }
     }
4. Gateway: skip_stateful_validations returns true (nonce==1, account_nonce==0, in_pool==true).
5. run_validate_entry_point called with validate=false → __validate__ NOT called.
6. Transaction admitted to mempool.
7. If attacker's fees > legitimate nonce=1 tx fees: fee escalation replaces legitimate tx.
8. Batcher executes: forged tx runs with validate=true → __validate__ fails → revert.
   Legitimate nonce=1 tx is gone from mempool.
``` [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
