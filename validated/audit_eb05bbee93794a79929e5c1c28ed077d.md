### Title
`MsgTurnBridge` Handler Is a No-Op Stub — Bridge Can Never Be Disabled - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` message handler is registered in the gRPC service descriptor and is reachable on-chain, but its implementation unconditionally returns `nil, nil` without performing any state change. The bridge active/inactive flag is never written, making it permanently impossible to disable the Gravity bridge through the intended control path.

### Finding Description
`MsgTurnBridge` is fully declared in the proto service, wired into the gRPC service descriptor, and exposed via the CLI. However, the server-side implementation in `x/cronos/keeper/msg_server.go` is an empty stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

The handler has two simultaneous defects:

1. **No state write.** It never calls `k.SetParams` or any equivalent to persist `msg.Enable` into the `bridge_active` parameter. The bridge enable/disable flag is therefore immutable at runtime.

2. **No permission check.** Every other privileged handler checks either `k.HasPermission(ctx, ..., CanTurnBridge)` or compares `msg.From` against `CronosAdmin`. This handler skips both. The dedicated `CanTurnBridge` permission bit defined in `permissions.go` is never consulted. [2](#0-1) 

The message is registered and dispatched correctly at the gRPC layer: [3](#0-2) 

So the transaction is accepted, fees are charged, and a success response is returned — but the bridge state is never changed.

### Impact Explanation
The Gravity bridge `bridge_active` flag is the primary emergency circuit-breaker for bridge flows. The integration test `test_gravity_turn_bridge` confirms the intended behavior: after a successful `TurnBridge(false)`, `send_to_ethereum` calls must fail. [4](#0-3) 

Because the handler is a no-op, the bridge can never be disabled regardless of who sends the message. Any attempt by the admin or a `CanTurnBridge`-permissioned account to halt bridge flows silently succeeds on-chain while leaving the bridge fully operational. This is a permanent inability to stop bridge/conversion flows — a **High** impact finding under the allowed scope.

### Likelihood Explanation
The message is fully reachable by any sender with a valid bech32 address and sufficient gas. No special preconditions are required. The admin would discover the failure only after an emergency attempt to disable the bridge produces no effect.

### Recommendation
Implement `TurnBridge` to:
1. Check that the sender holds `CanTurnBridge` permission (mirroring the pattern in `UpdateTokenMapping`).
2. Read the current params, set `params.EnableGravity = msg.Enable`, and persist via `k.SetParams(ctx, params)`.

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    params := k.GetParams(ctx)
    params.EnableGravity = msg.Enable
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &types.MsgTurnBridgeResponse{}, nil
}
```

### Proof of Concept
1. Any address broadcasts `MsgTurnBridge{Sender: <any valid address>, Enable: false}`.
2. The transaction is accepted and returns code 0.
3. Query `cronos params` — `enable_gravity` is unchanged.
4. `send_to_ethereum` / `send_to_evm_chain` continue to succeed, confirming the bridge was never disabled.
