### Title
Missing Signer-vs-Address Authorization in `RegisterEncryptionKey` Allows Any Account to Overwrite Any Victim's E2EE Key — (`x/e2ee/keeper/keeper.go`)

---

### Summary

`Keeper.RegisterEncryptionKey` writes `req.Key` to the store slot for `req.Address` with no check that the transaction signer equals `req.Address`. Any unprivileged account can overwrite any other account's on-chain X25519 encryption key.

---

### Finding Description

`Keeper.RegisterEncryptionKey` passes `req.Address` and `req.Key` directly to the internal `registerEncryptionKey` helper, which resolves the address bytes and writes to the KV store unconditionally: [1](#0-0) 

The only validation that exists is in `ValidateBasic`, which checks bech32 format and that the key is a parseable X25519 recipient — it never compares the address to the transaction signer: [2](#0-1) 

There is no `GetSigners()` method anywhere in the `x/e2ee/types` package, and no `cosmos.msg.v1.signer` proto annotation is present in the generated `tx.pb.go` that would cause the Cosmos SDK ante-handler to enforce signer == address: [3](#0-2) 

The CLI client sets `Address: clientCtx.GetFromAddress().String()` as a convention, but this is purely client-side and is not enforced on-chain: [4](#0-3) 

---

### Impact Explanation

The e2ee module stores public X25519 keys on-chain so that senders can encrypt messages to recipients. If an attacker replaces victim B's stored key with an attacker-controlled key, all future messages encrypted to B using the on-chain key are encrypted to the attacker instead. B can no longer decrypt them. This is permanent until B re-registers their own key — but the attacker can immediately overwrite it again in the same block or subsequent blocks, creating a persistent denial of correct decryption. This constitutes **corruption of e2ee key/message state with direct security impact** (High impact per scope rules).

---

### Likelihood Explanation

The attack requires only a valid bech32 address (publicly visible) and a valid X25519 key (trivially generated). No special privileges, leaked keys, or admin access are needed. The transaction is a standard `MsgRegisterEncryptionKey` submitted via any RPC endpoint. Likelihood is high.

---

### Recommendation

In `Keeper.RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    signers := sdkCtx.TxSigner() // or extract from msg signers
    // assert signers[0] == req.Address
    ...
}
```

Alternatively, add `option (cosmos.msg.v1.signer) = "address";` to the proto definition so the SDK ante-handler enforces it automatically, **and** add an explicit keeper-level check as defense-in-depth.

---

### Proof of Concept

```go
// Unit test sketch
func TestRegisterEncryptionKeySpoof(t *testing.T) {
    ctx, keeper := setupKeeper(t)

    victim  := sdk.AccAddress([]byte("victim______________"))
    attacker := sdk.AccAddress([]byte("attacker____________"))
    attackerKey := generateX25519Key() // valid age X25519 recipient string

    // Tx signed by attacker, but Address field set to victim
    msg := &types.MsgRegisterEncryptionKey{
        Address: victim.String(),
        Key:     attackerKey,
    }

    // Context signer is attacker, not victim
    ctx = setTxSigner(ctx, attacker)
    _, err := keeper.RegisterEncryptionKey(ctx, msg)
    require.NoError(t, err)

    // Victim's stored key is now the attacker's key
    resp, _ := keeper.Key(ctx, &types.KeyRequest{Address: victim.String()})
    require.Equal(t, attackerKey, resp.Key) // PASSES — victim's key is poisoned
}
