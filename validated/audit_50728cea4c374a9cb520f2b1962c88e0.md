### Title
Unbounded `MessageBuffer` Growth via Attacker-Injected Arbitrary Waitpoints Enables Memory Exhaustion DoS - (File: src/protocol/internal.rs)

### Summary
The `MessageBuffer` in `src/protocol/internal.rs` accepts messages with arbitrary `MessageHeader` values (any channel tag + any waitpoint) through the public `Protocol::message()` API with no bound on distinct headers or per-slot message count. A malicious participant can inject messages for waitpoints the honest protocol never polls, causing the internal `HashMap<MessageHeader, SubMessageQueue>` to grow without limit, exhausting process memory and permanently denying DKG, reshare, refresh, presign, or signing for honest parties. The codebase itself contains a test that explicitly demonstrates this attack succeeds.

### Finding Description

The `MessageBuffer` struct holds a `HashMap<MessageHeader, SubMessageQueue>`: [1](#0-0) 

Each `SubMessageQueue` wraps a `futures::channel::mpsc::unbounded()` channel: [2](#0-1) 

The `push` method on `MessageBuffer` calls `or_default()` on the HashMap entry, unconditionally creating a new `SubMessageQueue` for every distinct `MessageHeader` it receives: [3](#0-2) 

The public `Protocol::message()` implementation on `ProtocolExecutor` routes every inbound message directly into `push_message` with no validation of the sender identity, channel tag, or waitpoint value: [4](#0-3) 

`push_message` itself only checks that the raw byte slice is long enough to parse a header, then inserts unconditionally: [5](#0-4) 

A `MessageHeader` is a 32-byte channel tag plus an 8-byte waitpoint, giving an attacker 2^64 distinct waitpoints on the shared channel alone, each creating a new HashMap entry and a new OS-level allocation for the unbounded MPSC channel pair.

The codebase itself contains a test that explicitly proves the attack works: [6](#0-5) 

The test is named `attacker_can_fill_message_buffer_with_unused_waitpoints` and asserts that after injecting 512 messages at distinct unused waitpoints, the HashMap contains exactly 512 entries — confirming unbounded growth with no mitigation.

### Impact Explanation

Every protocol that uses `ProtocolExecutor` — DKG (`do_keygen`), reshare (`do_reshare`), refresh, FROST signing, OT-based ECDSA presign/sign, and CKD — is exposed through the same `Protocol::message()` entry point. A single malicious participant (or any network-level message sender, since `from` is not validated against the participant list before buffering) can stream messages with monotonically increasing waitpoints. Each message allocates a new HashMap entry and a new unbounded channel pair. When process memory is exhausted, the honest party's async runtime crashes or the OS kills the process, permanently terminating the in-progress protocol session. Honest parties cannot complete DKG, reshare, or signing.

**Impact: Medium — Griefing/resource-exhaustion by a malicious participant causing unbounded memory growth beyond documented behavior, with secondary High impact of permanent denial of DKG/signing for honest parties.**

### Likelihood Explanation

The attack requires only the ability to deliver raw bytes to a protocol participant's `message()` method, which is the normal message-delivery API. Any participant in a DKG or signing session, or any network-adjacent attacker who can inject bytes into the transport layer, can trigger this. No cryptographic material, no key knowledge, and no coordination is required. The attack is trivially scriptable and the codebase's own test confirms it succeeds.

### Recommendation

1. **Cap the HashMap size**: Reject messages whose `MessageHeader` does not correspond to a waitpoint that the current protocol state has already registered as expected. Maintain a whitelist of valid `(channel, waitpoint)` pairs and drop messages for unknown headers.
2. **Bound per-slot queue depth**: Replace `mpsc::unbounded()` with `mpsc::channel(N)` for a small fixed `N` (e.g., the participant count), so that a single slot cannot accumulate unlimited messages.
3. **Validate sender before buffering**: Check that `from` is a member of the known participant list before calling `push_message`, discarding messages from unknown senders entirely.

### Proof of Concept

The existing test in `src/protocol/internal.rs` at lines 532–554 is a self-contained proof of concept. It constructs a `Comms` instance, injects 512 messages with waitpoints `1_000_000` through `1_000_511` (values the honest protocol never polls), and asserts the internal HashMap has grown to 512 entries. Scaling this to millions of messages with a real network transport will exhaust process memory proportionally, terminating any in-progress DKG or signing session. [6](#0-5)

    fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
        let comms = Comms::new();
        let attacker = Participant::from(99_u32);
        let attack_count = 512_u64;

        for i in 0..attack_count {
            let header =
                MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);

            message.extend_from_slice(&i.to_le_bytes());

            .messages
            .lock()
            .expect("lock should not fail");

        assert!(messages.len() == usize::try_from(attack_count).unwrap());
