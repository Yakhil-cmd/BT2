### Title
Unprivileged Caller Can Invoke `update_rewards` with `disable_rewards=true` to Freeze All Stakers' Block Rewards — (File: `src/staking/staking.cairo`)

---

### Summary

The public `update_rewards` function accepts a caller-controlled `disable_rewards` boolean. When set to `true`, the function writes the current block number to the **global** `last_reward_block` storage slot and returns early — distributing zero rewards. Because `last_reward_block` is shared across all stakers, a single call per block by any unprivileged address is sufficient to permanently prevent every staker from receiving consensus-era block rewards.

---

### Finding Description

`update_rewards` is declared in the `IStakingRewardsManager` ABI and is callable by any non-zero address: [1](#0-0) 

The function first asserts the block has not already been processed: [2](#0-1) 

It then unconditionally writes the current block number to the **global** `last_reward_block`: [3](#0-2) 

Immediately after, if `disable_rewards` is `true`, the function returns without computing or distributing any rewards: [4](#0-3) 

`last_reward_block` is a single global slot, not per-staker: [5](#0-4) 

Because the `REWARDS_ALREADY_UPDATED` guard fires on any subsequent call in the same block, one attacker call per block with `disable_rewards=true` (and any valid active staker address) permanently blocks every staker's reward update for that block. The attacker needs no funds, no privileged role, and no approval — only a valid active staker address (trivially obtained from on-chain `NewStaker` events).

---

### Impact Explanation

In consensus-rewards mode (V3), block rewards are the sole mechanism for distributing staking yield. By calling `update_rewards(any_active_staker, disable_rewards: true)` once per block, an attacker:

1. Marks the block as "already processed" globally.
2. Distributes zero rewards to any staker or pool.
3. Prevents any legitimate call to `update_rewards` for that block from succeeding.

Sustained over time this constitutes **permanent freezing of unclaimed yield** for all stakers and all delegation pools — matching the High impact tier in the allowed scope.

---

### Likelihood Explanation

- No funds required; only gas per block.
- No privileged role required; `general_prerequisites` only checks the contract is unpaused and the caller is non-zero.
- Active staker addresses are publicly observable from `NewStaker` events.
- The attack is fully automatable: one transaction per block, targeting any single active staker, suffices to grief the entire protocol.

Likelihood is **High**.

---

### Recommendation

1. **Remove the `disable_rewards` parameter from the public interface.** If the parameter is needed for an internal migration path, expose it only through a role-gated function (e.g., `only_app_governor`).
2. **Alternatively**, restrict `update_rewards` so it can only be called by the staker's registered operational address or the staker address itself, consistent with how `update_rewards_from_attestation_contract` is restricted to the attestation contract. [6](#0-5) 

---

### Proof of Concept

```
// Attacker script — runs once per block
loop {
    wait_for_new_block();
    let active_staker = fetch_any_active_staker_from_events();
    staking_contract.update_rewards(
        staker_address: active_staker,
        disable_rewards: true   // ← caller-controlled, no access check
    );
    // last_reward_block is now set to current block;
    // no staker can receive rewards for this block.
}
```

Step-by-step:

1. Attacker calls `update_rewards(active_staker, true)` at block N.
2. `last_reward_block` is written to N; function returns early — no rewards distributed.
3. Any legitimate call to `update_rewards(any_staker, false)` at block N reverts with `REWARDS_ALREADY_UPDATED`.
4. Attacker repeats at block N+1, N+2, … permanently freezing all staker and pool rewards.
