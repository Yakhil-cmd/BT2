### Title
`use_global_contract` understates `account.storage_usage` by 5 bytes for `AccountId`-mode global contracts — (`runtime/runtime/src/global_contracts.rs`)

### Summary
`use_global_contract` increments `account.storage_usage` by `GlobalContractIdentifier::len()`, which for the `AccountId` variant returns only `account_id.len()` (= L). The actual borsh-encoded size of the `AccountContract::GlobalByAccount(id)` reference stored in the account record is `1 + 4 + L` bytes (discriminant + u32 length prefix + account_id bytes). The account's `storage_usage` is therefore understated by exactly **5 bytes** every time an account adopts a global contract by AccountId.

### Finding Description
`GlobalContractIdentifier::len()` at [1](#0-0)  returns bare `account_id.len()` for the `AccountId` arm — no discriminant byte, no borsh `u32` string-length prefix.

`GlobalContractCodeIdentifier::len()` at [2](#0-1)  correctly accounts for both: `1 + size_of::<u32>() + account_id.len()`.

`use_global_contract` at [3](#0-2)  calls the wrong one (`GlobalContractIdentifier::len()`), so the `storage_usage` delta is `L` instead of `1 + 4 + L`.

The same function also has a 1-byte undercharge for `CodeHash` mode (`32` vs `33`), but the `AccountId` case is the larger gap and the one an attacker can amplify by choosing a long account ID.

### Impact Explanation
`account.storage_usage` is the exact value used to compute the storage staking requirement. An understated `storage_usage` means the account's locked balance requirement is lower than the bytes it actually occupies in the trie. An attacker can repeatedly call `UseGlobalContractAction` (switching between global contracts) to keep `storage_usage` artificially low, allowing the account to hold more state than its staked balance covers — a direct violation of the storage-staking ledger invariant.

### Likelihood Explanation
Any unprivileged user can submit a `UseGlobalContractAction` referencing any already-deployed `AccountId`-mode global contract. No special privileges are required. The undercharge is deterministic and repeatable.

### Recommendation
Replace `contract_identifier.len()` in `use_global_contract` with a call to `GlobalContractCodeIdentifier::from(contract_identifier.clone()).len()` (or an equivalent that includes the discriminant byte and the borsh u32 string-length prefix), so the charged delta matches the actual bytes stored in the account record.

### Proof of Concept
1. Deploy a global contract with `GlobalContractDeployMode::AccountId` from account `deployer.near` (length 12).
2. From a second account, submit `UseGlobalContractAction { contract_identifier: GlobalContractIdentifier::AccountId("deployer.near") }`.
3. Record `account.storage_usage` before and after.
4. Assert the delta equals `1 + 4 + 12 = 17`. Actual delta will be `12` — off by 5.
5. Confirm the account can now hold 5 extra bytes of state without additional staking balance.
