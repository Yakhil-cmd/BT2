### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Yield via Trace Bloat - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `Pool` iterates over every entry in a delegator's `pool_member_epoch_balance` trace without any bound. A delegator who makes one balance change per epoch (e.g., calling `add_to_delegation_pool` with a minimal amount each epoch) will grow this trace indefinitely. After enough epochs, any call to `claim_rewards` (or `pool_member_info_v1`) will exceed Starknet's gas limit, permanently freezing the delegator's unclaimed yield. The code itself acknowledges this with an explicit comment.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains an unbounded `while` loop that iterates over every checkpoint in `pool_member_epoch_balance` for the given pool member:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `pool_member_epoch_balance.entry(pool_member).insert(key: self.get_epoch_plus_k(), ...)`. The `insert` implementation only **updates** the last entry when the key matches; otherwise it **appends** a new checkpoint:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

Since the key is always `current_epoch + K` (with `K = 2`), any balance-changing call made in a **different epoch** than the previous one appends a new entry. [3](#0-2) [4](#0-3) 

The operations that trigger a new trace entry are:
- `add_to_delegation_pool` → `increase_member_balance` → `set_member_balance`
- `exit_delegation_pool_intent` → `set_member_balance`
- `enter_delegation_pool_from_staking_contract` (pool switch) → `set_member_balance` [5](#0-4) [6](#0-5) 

`claim_rewards` calls `calculate_rewards` directly, passing `pool_member_info.entry_to_claim_from` as the starting index. Crucially, `entry_to_claim_from` is **saved** after each successful claim, so it advances past already-processed entries. However, if the delegator never claims (or claims infrequently while continuing to add balance changes), the number of unprocessed entries in the loop grows without bound. [7](#0-6) 

`pool_member_info_v1` also calls `calculate_rewards` with `entry_to_claim_from` from storage, making even read-only queries vulnerable once the trace is large enough. [8](#0-7) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace for a delegator grows large enough, every call to `claim_rewards` will exceed Starknet's gas limit and revert. The delegator's accrued yield becomes permanently unclaimable. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- Any delegator can trigger this by calling `add_to_delegation_pool` with the minimum non-zero amount once per epoch.
- No privileged access is required; `add_to_delegation_pool` is callable by the pool member or their reward address.
- The minimum amount is `1` (just non-zero), so the cost per epoch is negligible.
- Over months or years of normal protocol operation, a long-term delegator who adjusts their stake frequently will naturally accumulate a large trace, eventually hitting the gas limit without any malicious intent.
- A malicious actor could deliberately bloat their own trace to grief themselves (e.g., to avoid a slashing mechanism) or could bloat another delegator's trace if they control the reward address (which can call `add_to_delegation_pool`). [9](#0-8) 

---

### Recommendation

Track `entry_to_claim_from` as a persistent cursor and process only a bounded number of trace entries per `claim_rewards` call (pagination). Alternatively, cap the number of checkpoints that can be appended to `pool_member_epoch_balance` per pool member, or compact old entries when they are fully processed.

---

### Proof of Concept

```
1. Delegator D enters pool with minimum stake.
2. Each epoch E (for E = 1, 2, ..., N):
   a. Advance to epoch E.
   b. D calls add_to_delegation_pool(pool_member: D, amount: 1).
      → set_member_balance inserts a new checkpoint at epoch E + K.
      → pool_member_epoch_balance trace length grows by 1.
3. After N epochs without claiming, D calls claim_rewards(pool_member: D).
   → calculate_rewards loops over all N entries.
   → For large N (e.g., N > gas_limit / cost_per_iteration), the transaction reverts.
   → D's accumulated yield is permanently frozen.
```

The loop body performs a storage read (`pool_member_trace.at(entry_to_claim_from)`) and a `find_sigma` call (which itself reads from `cumulative_rewards_trace`) on every iteration, making each iteration relatively expensive in gas. [10](#0-9)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
