### Title
Unvalidated Oracle-Supplied `native_price` and `pubdata_price` Not Committed to ZK Proof Public Input — (File: `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

---

### Summary

`BlockMetadataFromOracle` fields `native_price` and `pubdata_price` are read from the oracle without bounds validation and are never committed to the ZK proof public input. A malicious prover can supply arbitrary values for these fields, manipulating native resource accounting to prove state transitions that would be invalid under the correct fee model.

---

### Finding Description

In `metadata_op.rs`, `BlockMetadataFromOracle` is deserialized from the oracle via `BLOCK_METADATA_QUERY_ID`. The only validation performed is a bounds check on `block_gas_limit` and `individual_tx_gas_limit`:

```rust
let block_level_metadata: BlockMetadataFromOracle =
    oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)?;
// ...
if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT
    || metadata.individual_tx_gas_limit() > MAX_TX_GAS_LIMIT
{
    return Err(internal_error!("block or tx gas limit is too high"));
}
```

`native_price` and `pubdata_price` pass through with no validation whatsoever. [1](#0-0) 

These values are then used directly in `validation_impl.rs` to compute the native resource budget for every L2 transaction:

```rust
let pubdata_price = system.get_pubdata_price();
let native_price = system.get_native_price();
// ...
let native_per_gas = gas_price.div_ceil(native_price);
let native_per_pubdata = pubdata_price.wrapping_div(native_price);
let native_prepaid_from_gas = native_per_gas.saturating_mul(tx_gas_limit);
``` [2](#0-1) 

The ZK proof public input — computed in `post_tx_op_proving_singleblock_batch.rs` — commits only to `chain_state_commitment_before`, `chain_state_commitment_after`, and `blocks_output_hash`. The `ChainStateCommitment` struct hashes `state_root`, `next_free_slot`, `block_number`, `last_256_block_hashes_blake`, and `last_block_timestamp`. Neither `native_price` nor `pubdata_price` appear anywhere in this commitment. [3](#0-2) [4](#0-3) 

The project's own documentation explicitly acknowledges this gap:

> "The block header should determine the block fully, i.e. include all the inputs needed to execute the block. **Currently it misses `gas_per_pubdata` and `native_price`**, but we already working on design and implementation to solve this issue." [5](#0-4) 

The `BlockMetadataFromOracle` struct confirms both fields are oracle-supplied with no on-chain anchor: [6](#0-5) 

---

### Impact Explanation

Because `native_price` and `pubdata_price` are unconstrained by the ZK proof, a malicious prover can freely choose any values when generating a proof, and the settlement layer will accept the proof as valid. Concrete scenarios:

1. **Set `native_price = 1`**: `native_per_gas = gas_price / 1 = gas_price` (maximally large). Every transaction receives an effectively unlimited native resource budget. Transactions that should have been reverted due to native resource exhaustion (e.g., computationally expensive contracts) are proven as successful. The resulting state transition — including storage writes and fee payments — is accepted by L1 as valid.

2. **Set `pubdata_price = 0`**: `native_per_pubdata = 0`. All pubdata charges are eliminated. Transactions that write large amounts of pubdata and should have been reverted post-execution succeed, allowing the prover to prove state transitions with unbounded pubdata at zero cost.

3. **Set `native_price = U256::MAX`**: `native_per_gas` rounds to 0, triggering the `free_native` path (`native_per_gas == 0` → `native_limit = u64::MAX - 1`), again giving unlimited native resources. [7](#0-6) 

In all cases, the ZK proof verifies correctly on the settlement layer because the manipulated values are not part of the public input. This breaks the core ZK rollup security guarantee: the proof no longer certifies that the fee model was applied correctly.

---

### Likelihood Explanation

This requires a malicious prover/sequencer, which is a privileged role. However, the ZKsync OS Immunefi scope explicitly lists **"prover/forward execution input"** as a valid attacker. In a ZK rollup, the ZK proof is the mechanism that removes the need to trust the prover — if `native_price` is unconstrained, the proof fails to provide this guarantee for the fee model. The attack is trivially executable: the prover simply returns a crafted value from the `BLOCK_METADATA_QUERY_ID` oracle handler. [8](#0-7) 

---

### Recommendation

Commit `native_price` and `pubdata_price` to the ZK proof public input, or include them in the block header (which is already identified as a TODO). One approach is to add them to `ChainStateCommitment` or to `blocks_output_hash` so that the settlement layer can verify they match the values used during execution. For L1→L2 transactions, a hardcoded constant (`L1_TX_NATIVE_PRICE`) is already used to avoid this problem — the same principle should be applied to L2 blocks. [3](#0-2) 

---

### Proof of Concept

1. Prover crafts a `BlockMetadataResponder` that returns `native_price = 1` and `pubdata_price = 0` for `BLOCK_METADATA_QUERY_ID`.
2. `metadata_op` deserializes these values; only gas limit bounds are checked — `native_price` and `pubdata_price` pass through unchecked. [9](#0-8) 
3. `validation_impl` computes `native_per_gas = gas_price / 1 = gas_price` and `native_per_pubdata = 0`. [10](#0-9) 
4. `create_resources_for_tx` assigns `native_limit = gas_limit * native_per_gas` (very large) and `intrinsic_pubdata_overhead = 0`. [11](#0-10) 
5. Computationally expensive transactions that would normally exhaust native resources now succeed; pubdata-heavy transactions incur no pubdata charge.
6. The prover runs the RISC-V proving binary with this oracle, producing a valid proof.
7. The proof public input is computed from `ChainStateCommitment` (which does not include `native_price` or `pubdata_price`) and is accepted by the settlement layer. [12](#0-11)

#[derive(Debug)]
pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }
