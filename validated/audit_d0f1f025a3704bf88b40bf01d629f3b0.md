### Title
Gateway Signature Verification Bypass via Overly Broad `skip_stateful_validations` Condition — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry point (which performs signature verification) for invoke transactions with nonce=1 when the account has **any** transaction in the mempool pool — not specifically a `deploy_account` transaction. An attacker controlling a deployed account whose on-chain nonce is still 0 (e.g., a genesis-deployed account) can submit a valid nonce=0 invoke to populate the pool, then submit a nonce=1 invoke with an **invalid or forged signature** that bypasses gateway signature verification and is admitted to the mempool.

---

### Finding Description

The `skip_stateful_validations` function is designed to improve UX for the `deploy_account + invoke` simultaneous submission pattern: when a user sends both transactions at once, the invoke (nonce=1) should be admitted even though the account doesn't exist yet in committed state.

The skip condition is:

```
tx_nonce == 1  AND  account_nonce == 0  AND  account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or any recently committed transaction: [2](#0-1) 

The code comment acknowledges the second case: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or **transactions with future nonces that passed validations**."*

This reasoning is flawed. A deployed account with on-chain nonce=0 (e.g., a genesis account) can have a nonce=0 invoke in the pool that passed validation. This satisfies `account_tx_in_pool_or_recent_block`, triggering the skip for a subsequent nonce=1 invoke — even though no `deploy_account` is pending and the account's `__validate__` has never been called for the nonce=1 transaction.

When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate=false`, execution returns after `perform_pre_validation_stage` without ever calling `__validate__`: [4](#0-3) 

`perform_pre_validation_stage` checks nonce, fee bounds, and balance — but **not** the account signature: [5](#0-4) 

The nonce=1 invoke with an invalid signature therefore passes all gateway checks and is admitted to the mempool.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A transaction carrying an invalid or forged signature for an account's nonce=1 is admitted to the mempool without signature verification. The invariant "every transaction admitted to the mempool must have passed `__validate__`" is broken for the specific case of deployed accounts with on-chain nonce=0. The batcher will later call `__validate__` during execution and the transaction will revert, but the admission itself is unauthorized and consumes mempool resources without the attacker having proven authorization.

---

### Likelihood Explanation

Requires a deployed account whose on-chain nonce is 0 — i.e., an account that exists in state but has never had a committed transaction. This applies to genesis-deployed accounts (sequencer system accounts, pre-funded test accounts, etc.). The attacker only needs to submit one valid nonce=0 invoke to arm the condition, then submit the invalid nonce=1 invoke. No privileged access is required.

---

### Recommendation

The `skip_stateful_validations` check must verify that the transaction in the pool is specifically a `deploy_account` transaction for the sender address, not just any transaction. The mempool's `account_tx_in_pool_or_recent_block` API does not distinguish transaction types; a new query (e.g., `has_pending_deploy_account(address)`) should be introduced, or the skip should be restricted to cases where the account contract does not yet exist in state (i.e., `get_class_hash_at(sender) == ClassHash::default()`). [6](#0-5) 

---

### Proof of Concept

**Setup:** Account `A` is deployed at genesis (class hash set, on-chain nonce = 0, no committed transactions).

**Step 1 — Arm the pool:**
```
POST /add_transaction
{
  type: INVOKE_V3,
  sender_address: A,
  nonce: 0,
  signature: <valid_sig_for_nonce_0>,
  ...
}
```
Gateway runs `__validate__` (nonce=0, account_nonce=0 → no skip). Transaction passes and enters the pool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

**Step 2 — Submit invalid nonce=1 invoke:**
```
POST /add_transaction
{
  type: INVOKE_V3,
  sender_address: A,
  nonce: 1,
  signature: [0xDEAD, 0xBEEF],   // invalid / forged
  ...
}
```

Gateway evaluation:
- `validate_nonce`: nonce=1 ≥ account_nonce=0, within gap → **passes**
- `validate_by_mempool`: nonce/duplicate check only → **passes**
- `skip_stateful_validations`: tx_nonce=1 ✓, account_nonce=0 ✓, pool contains A ✓ → returns **`true`**
- `run_validate_entry_point` with `validate=false` → `__validate__` **not called**

**Result:** The nonce=1 invoke with forged signature is admitted to the mempool. When the batcher executes it, `__validate__` is called, the signature check fails, and the transaction reverts — but the gateway's admission invariant has been violated.

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        state: &mut S,
