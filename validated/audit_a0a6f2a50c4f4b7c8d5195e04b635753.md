### Title
Malicious CKD Participant Can Corrupt Derived Key Output Without Verification - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator blindly sums participant-provided `(big_y, big_c)` values without any cryptographic proof that each participant computed their share correctly from their actual key share. A single malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that honest parties will accept as valid.

### Finding Description
In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–57), the coordinator receives `CKDOutput` from every other participant and unconditionally accumulates the values:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each honest participant is supposed to compute:
- `big_y = λ_i · y_i · G` (commitment to fresh randomness)
- `big_c = λ_i · (x_i · H(pk, app_id) + y_i · A)` (ElGamal encryption share)

where `x_i` is their secret key share and `y_i` is fresh randomness. There is **no zero-knowledge proof** or any other check that the received `(big_y, big_c)` pair was actually derived from the participant's legitimate key share. The coordinator simply trusts the return values from each participant.

This is structurally identical to the external report's root cause: the rebalancing logic trusted the return value of `_swapper.swap()` rather than independently verifying the outcome. Here, the CKD coordinator trusts participant-provided group elements rather than verifying them against committed key material.

By contrast, both ECDSA signing protocols perform an independent final verification. In `src/ecdsa/ot_based_ecdsa/sign.rs` the coordinator calls `sig.verify(&public_key, &msg_hash)` after aggregating shares, catching any corruption: [2](#0-1) 

The robust ECDSA coordinator does the same: [3](#0-2) 

No equivalent check exists in the CKD coordinator path. [4](#0-3) 

### Impact Explanation
A malicious participant sends `(big_y', big_c')` where `big_c' = big_c_correct + δ` for an arbitrary group element `δ`. The coordinator accumulates this into the final `CKDOutput`. The resulting `(Y, C)` pair no longer encodes the correct threshold-derived secret: when the application unmasks with `app_sk`, it recovers a wrong value. Honest parties receive and accept this corrupted output with no indication of failure, permanently breaking the CKD result for that invocation.

This matches the **High** allowed impact: *Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

### Likelihood Explanation
Any single participant in the CKD protocol can trigger this. The attacker-controlled entry path is direct: a participant calls `ckd(...)`, constructs a protocol instance, and during execution sends an arbitrary `CKDOutput` to the coordinator instead of the correctly computed one. No special privilege, leaked key, or external dependency is required — only participation in the protocol. [5](#0-4) 

### Recommendation
Require each participant to accompany their `(big_y, big_c)` with a non-interactive zero-knowledge proof of correct ElGamal encryption — specifically, a proof that `big_c` was formed as `λ_i · (x_i · H(pk, app_id) + y_i · A)` for the same `x_i` committed to in the DKG output and the same `y_i` used for `big_y`. The coordinator must verify all proofs before accumulating shares, analogous to how the ECDSA coordinators verify the final signature before returning it.

### Proof of Concept
1. Honest participants `P_1, …, P_{n-1}` and malicious `P_n` run `ckd(...)`.
2. `P_n`'s protocol instance, instead of calling `compute_signature_share`, sends `(ElementG1::identity(), arbitrary_delta)` to the coordinator.
3. The coordinator loop at lines 50–55 adds `arbitrary_delta` to `norm_big_c` without any check.
4. The returned `CKDOutput` encodes `C_correct + arbitrary_delta` instead of `C_correct`.
5. The application calls `ckd_output.unmask(app_sk)` and obtains a wrong derived key, silently accepting the corrupted result. [6](#0-5)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
