### Title
Zero Base-Fee Causes Unlimited Native Resources and Zero Operator Payment for ZK Transactions - (`basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs`)

---

### Summary

The `get_gas_price` function in `gas_helpers.rs` unconditionally returns `U256::ZERO` when `base_fee == 0`. This cascades through the ZK transaction validation path to set `native_per_gas = 0`, which triggers the `free_native = true` branch in `create_resources_for_tx`, granting the transaction `native_limit = u64::MAX - 1` (effectively unlimited native computational resources) at zero cost. The operator also receives zero payment. This behavior is absent from the Ethereum transaction flow, which correctly honors the priority fee even when `base_fee == 0`, creating a state-transition and resource-accounting divergence between the two flows.

---

### Finding Description

**Root cause — `get_gas_price` in `gas_helpers.rs`:** [1](#0-0) 

When `base_fee == 0`, the function immediately returns `U256::ZERO` without inspecting `max_fee_per_gas` or `max_priority_fee_per_gas`.

**Cascade 1 — `native_per_gas` becomes 0 in ZK validation:** [2](#0-1) 

`gas_price.div_ceil(native_price)` with `gas_price = 0` yields `native_per_gas = 0`. The simulation guard on line 128 does not fire in production execution.

**Cascade 2 — `free_native = true` is passed to resource creation:** [3](#0-2) 

`native_per_gas == 0` evaluates to `true`, so `free_native = true`.

**Cascade 3 — `native_limit = u64::MAX - 1`:** [4](#0-3) 

The `free_native` flag bypasses the normal `native_prepaid_from_gas` budget and sets the native limit to `u64::MAX - 1`.

**Cascade 4 — zero fee charged:** [5](#0-4) 

`gas_price * tx_gas_limit = 0`, so `fee_to_prepay = 0` and the operator receives nothing.

**Divergence — the Ethereum transaction flow does NOT have this shortcut:** [6](#0-5) 

When `base_fee == 0`, `max_fee_minus_base_fee = max_fee_per_gas`, so `effective_gas_price = priority_fee_per_gas`. ETH transactions correctly pay the priority fee and receive a bounded native limit.

---

### Impact Explanation

1. **Resource accounting bug**: Any ZK transaction submitted when `base_fee == 0` receives `native_limit = u64::MAX - 1`. Native resources gate proving complexity (storage I/O, precompile calls, etc.). A transaction can exhaust the prover's budget for a block without paying for it, enabling a low-cost DoS against the sequencer/prover.

2. **Fee loss**: The operator receives zero payment for every ZK transaction in a zero-base-fee block, even when users set `max_fee_per_gas > 0`. This is a direct loss of operator revenue.

3. **State-transition divergence**: ETH-type and ZK-type transactions with identical fee parameters produce different state transitions (different native resource budgets, different fee deductions) when `base_fee == 0`. This breaks the invariant that both flows are equivalent for equivalent inputs.

---

### Likelihood Explanation

ZKsync OS is designed to run on multiple chains. A chain operator may legitimately configure `base_fee = 0` (e.g., to subsidize users or during bootstrapping). The block re-executor test harness already exercises `eip1559_basefee = U256::ZERO` as a valid block context. Once `base_fee = 0` is set, every unprivileged user submitting a ZK transaction automatically receives unlimited native resources and pays nothing — no special knowledge or privilege is required.

---

### Recommendation

In `get_gas_price`, do not short-circuit to `U256::ZERO` when `base_fee == 0`. Instead, apply the same EIP-1559 formula used by the Ethereum flow:

```rust
// When base_fee == 0, effective_gas_price = min(max_priority_fee_per_gas, max_fee_per_gas)
let priority_fee = max_priority_fee_per_gas.min(*max_fee_per_gas);
Ok(base_fee.saturating_add(priority_fee))
```

This aligns the ZK flow with the Ethereum flow and ensures `native_per_gas > 0` whenever the user specifies a non-zero fee, preventing the `free_native` path from being triggered unintentionally.

---

### Proof of Concept

**Setup**: Deploy ZKsync OS with `eip1559_basefee = 0`, `native_price = 100`, `max_fee_per_gas = 10_000`, `max_priority_fee_per_gas = 10_000`.

**Step 1**: `get_gas_price` is called.
- `base_fee = 0` → returns `U256::ZERO` immediately. [7](#0-6) 

**Step 2**: `native_per_gas = 0 / 100 = 0`. [8](#0-7) 

**Step 3**: `create_resources_for_tx` called with `free_native = true`. [9](#0-8) 

**Result**: Transaction gets `native_limit = u64::MAX - 1` and pays `fee = 0`, regardless of the user's stated `max_fee_per_gas = 10_000`. An ETH-type transaction with identical parameters would pay `10_000 * gas_used` and receive `native_limit = (10_000 / 100) * gas_limit = 100 * gas_limit`.
