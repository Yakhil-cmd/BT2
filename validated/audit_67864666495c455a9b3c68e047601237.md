### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` / `addFeeSellE6` to the pool with **no validation against `maxAdminSpreadFeeE6`**, while the parallel path `setPoolAdminFees` enforces that cap. A pool admin can therefore charge traders per-bin fees that exceed the factory-owner-imposed ceiling, breaking the admin-boundary invariant.

---

### Finding Description

The factory enforces a fee cap on the pool admin's base spread fee:

```solidity
// setPoolAdminFees — cap enforced
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

The parallel function for per-bin fees performs **no such check**:

```solidity
// setPoolBinAdditionalFees — no cap check
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index:

```solidity
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During a swap, the per-bin additional fee is **added on top of** the base spread fee:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so their maximum value is **65 535 E6 units ≈ 6.55 %** per bin. This is additive to the base spread fee, meaning the effective admin-controlled fee can far exceed `maxAdminSpreadFeeE6`.

The same gap exists at pool creation: `_unpackAndValidateBinStates` stores initial per-bin fees from caller-supplied packed data without checking them against `maxAdminSpreadFeeE6`:

```solidity
nonNegativeBinStates[k] = BinState({
    token0BalanceScaled: 0, token1BalanceScaled: 0,
    lengthE6: length, addFeeBuyE6: buyFee, addFeeSellE6: sellFee
});
``` [5](#0-4) 

---

### Impact Explanation

The factory owner's `maxAdminSpreadFeeE6` cap is the protocol's mechanism to bound how much a pool admin can extract from traders. Because `setPoolBinAdditionalFees` bypasses this cap, a pool admin can:

1. Set `addFeeBuyE6 = addFeeSellE6 = 65535` on the active bin, charging traders an extra ≈ 6.55 % on every swap through that bin.
2. Do so even when the factory owner has set `maxAdminSpreadFeeE6 = 0` (intending zero admin fees).

Traders lose the overcharged amount on every affected swap; the excess accrues as surplus inside the pool and is collected by the admin via `collectPoolFees`. This is a direct, repeatable loss of trader principal that violates the admin-boundary invariant.

---

### Likelihood Explanation

Medium. Requires a pool admin who is willing to act adversarially. Because pool admins are semi-trusted (not fully trusted), this is a realistic scenario — especially for permissionlessly created pools where the admin is an unknown third party. The bypass requires only a single transaction with no special preconditions.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
+   if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
+   if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Apply the same check to the initial bin fees unpacked in `_unpackAndValidateBinStates` during `createPool`.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(0, 0, 0, 0)` — intending zero admin fees for all pools.
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
   - No revert: `setPoolBinAdditionalFees` has no cap check.
   - Pool stores `addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` for bin 0.
3. A trader swaps through bin 0. The swap math computes:
   ```
   effectiveFee = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
   ```
   The trader pays ≈ 6.55 % extra on the swap amount.
4. The surplus accumulates in the pool and is collected by the pool admin via `collectPoolFees`, bypassing the `maxAdminSpreadFeeE6 = 0` cap set by the factory owner.
