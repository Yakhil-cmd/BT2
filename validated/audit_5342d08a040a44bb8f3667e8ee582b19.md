### Title
Missing Lower-Bound Threshold Validation in FROST Presign and Sign Allows Signing Denial-of-Service - (`File: src/frost/mod.rs`)

---

### Summary

The `presign()` and `assert_sign_inputs()` functions in `src/frost/mod.rs` validate only the upper bound of the `threshold` parameter (`threshold > participants.len()`) but omit the lower-bound check (`threshold < 2`). This is inconsistent with every other threshold-accepting entry point in the library (`assert_key_invariants`, `validate_triple_inputs`), which enforce both bounds. Because `ReconstructionLowerBound` is a plain `usize` wrapper with no type-level minimum, a caller can pass `threshold = 0` or `threshold = 1` into the FROST presign/sign pipeline without triggering any error at initialization time, causing the signing protocol to produce an invalid (unverifiable) signature or panic at runtime, permanently denying signing for honest parties.

---

### Finding Description

`ReconstructionLowerBound` in `src/thresholds.rs` is a transparent `usize` newtype with no enforced minimum: [1](#0-0) 

Every DKG and triple-generation entry point enforces `threshold >= 2`: [2](#0-1) [3](#0-2) 

However, the FROST `presign()` function only checks the upper bound: [4](#0-3) 

And `assert_sign_inputs()` — the shared validation gate for EdDSA and RedJubjub signing — likewise only checks the upper bound: [5](#0-4) 

Neither function contains a `threshold < 2` guard. A caller supplying `threshold = 1` (or `threshold = 0`) passes both checks and the protocol is launched.

---

### Impact Explanation

**Impact: High — Permanent denial of signing for honest parties.**

In FROST, the signing threshold controls how many participants' signature shares are combined via Lagrange interpolation to reconstruct the group signature. The key shares were generated by DKG with `threshold >= 2` (a degree-`(threshold-1)` polynomial). If the signing threshold is set to `1`:

- Only one participant's share is used in aggregation; the Lagrange coefficient for a singleton set is `1`, so the "reconstructed" secret is just that participant's raw share, not the actual group secret.
- The resulting signature fails verification against the public key.
- Every signing attempt with this misconfigured threshold produces an invalid signature, permanently blocking signing for all honest parties who rely on the output.

If `threshold = 0` is passed, the Lagrange interpolation over an empty set will panic or return an arithmetic error, crashing the signing protocol outright.

---

### Likelihood Explanation

The `ReconstructionLowerBound` type accepts any `usize` value. A library caller — including an application developer making an off-by-one error, a misconfigured frontend, or a malicious coordinator constructing `PresignArguments` — can trivially supply `threshold = 1`. The DKG path would have rejected this value, but the FROST sign path silently accepts it. The inconsistency between the two paths makes this easy to trigger accidentally or intentionally.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already use, in both `presign()` and `assert_sign_inputs()` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be placed immediately after the upper-bound check in both functions, mirroring the pattern at: [2](#0-1) 

---

### Proof of Concept

1. Run DKG correctly with `participants = [P1, P2, P3]`, `threshold = 2`. All parties receive valid key shares.
2. Call `frost::presign(participants, me, &PresignArguments { keygen_out, threshold: ReconstructionLowerBound(1) }, rng)`. This succeeds — the upper-bound check passes (`1 <= 3`), the missing lower-bound check does not fire.
3. Call `assert_sign_inputs(participants, ReconstructionLowerBound(1), me, coordinator)`. This also succeeds for the same reason.
4. The signing protocol proceeds with `threshold = 1`. Only one participant's share enters Lagrange interpolation; the reconstructed scalar is that participant's raw share, not the group secret.
5. The produced signature fails `verify(public_key, message, signature)`. Every subsequent signing attempt with this threshold configuration fails identically — permanent denial of signing. [6](#0-5) [7](#0-6)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
