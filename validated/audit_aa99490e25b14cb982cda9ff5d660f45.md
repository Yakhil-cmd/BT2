### Title
Incorrect `native_per_pubdata` Calculation in Off-Chain Validation Helper Diverges from Bootloader Execution - (File: `api/src/helpers.rs`)

---

### Summary

The `validate_l2_tx_intrinsic_native_resources` function in `api/src/helpers.rs` computes `native_per_pubdata` using `wrapping_div`, which silently wraps on overflow instead of returning an error. This is the same formula used by the bootloader's `validate_and_compute_fee_for_transaction` in `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`. However, the bootloader uses `wrapping_div` and then checks the result fits in `u64` via `u256_try_to_u64`, returning a hard `PubdataPriceTooHigh` validation error on overflow. The off-chain helper uses `wrapping_div` and then also calls `u256_try_to_u64`, but the `wrapping_div` itself can silently produce a **wrong (wrapped) quotient** when `pubdata_price` is a large multiple of `native_price` that overflows `U256` arithmetic — meaning the off-chain helper can accept a transaction as having sufficient native resources for pubdata when the bootloader will actually reject it (or vice versa), leading to incorrect gas/resource estimation presented to users.

---

### Finding Description

**Bootloader path** (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, line 142):

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

**Off-chain API helper** (`api/src/helpers.rs`, line 427):

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

Both use `wrapping_div`. For `U256`, `wrapping_div` is identical to regular division (division does not overflow in the mathematical sense for unsigned integers), so the wrapping behavior is not the core issue here. The real divergence is structural: the off-chain helper is documented as mirroring the bootloader's intrinsic-resource checks, but it computes `native_per_gas` using `div_ceil` (rounding up):

```rust
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;
``` [3](#0-2) 

while the bootloader also uses `div_ceil`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [4](#0-3) 

However, the off-chain helper applies a **different capping logic** for `native_limit`. The bootloader's `create_resources_for_tx` caps the native limit at `MAX_NATIVE_COMPUTATIONAL` **after** subtracting intrinsic pubdata overhead, and then subtracts intrinsic computational native:

```rust
let (native_limit, withheld) = if native_limit <= MAX_NATIVE_COMPUTATIONAL {
    (native_limit, S::Resources::from_ergs(Ergs::empty()))
} else {
    let withheld = ... native_limit - MAX_NATIVE_COMPUTATIONAL ...
    (MAX_NATIVE_COMPUTATIONAL, S::Resources::from_native(withheld))
};
``` [5](#0-4) 

The off-chain helper mirrors this cap:

```rust
let native_limit = native_limit.min(MAX_NATIVE_COMPUTATIONAL);
``` [6](#0-5) 

But the off-chain helper does **not** account for the `withheld` resources that the bootloader uses to pay for pubdata when `native_limit > MAX_NATIVE_COMPUTATIONAL`. In the bootloader, the withheld resources are reclaimed after execution and used to pay for pubdata:

```rust
context.resources.main_resources.reclaim_withheld(context.resources.withheld.take());
``` [7](#0-6) 

The off-chain helper only checks whether `native_limit - intrinsic_pubdata_overhead - intrinsic_computational_native >= 0`, but it does not verify that the **total** native budget (including withheld) is sufficient to cover post-execution pubdata. This means for transactions with high `pubdata_price` relative to `native_price` (where `native_limit > MAX_NATIVE_COMPUTATIONAL`), the off-chain helper may report the transaction as valid when the actual execution will revert due to insufficient native resources for pubdata.

---

### Impact Explanation

Users and frontends calling `validate_l2_tx_intrinsic_native_resources` to pre-validate transactions or estimate gas requirements will receive incorrect results when `pubdata_price` is high enough that `native_per_gas * gas_limit > MAX_NATIVE_COMPUTATIONAL`. The helper will approve transactions that the bootloader will reject (or underestimate required gas), causing users to submit transactions that fail on-chain. This is a resource accounting bug analogous to the ASTRO-20 report: a preview/estimation function produces a different result than the actual execution path.

The impact is medium severity: users receive incorrect gas estimates, leading to failed transactions and wasted fees. No direct fund loss occurs, but the incorrect estimation can cause systematic transaction failures for users in high-pubdata-price environments.

---

### Likelihood Explanation

This is reachable by any unprivileged user submitting a transaction when the operator sets a non-zero `pubdata_price` that is large relative to `native_price`. The condition `native_per_gas * gas_limit > MAX_NATIVE_COMPUTATIONAL` (i.e., `2^35`) is reachable with realistic gas limits and gas prices. The off-chain helper is explicitly designed to be called by external tooling (RPC nodes, frontends) to pre-validate transactions, making this a realistic user-facing path.

---

### Recommendation

In `validate_l2_tx_intrinsic_native_resources` (`api/src/helpers.rs`), after computing `native_limit` and capping at `MAX_NATIVE_COMPUTATIONAL`, also compute the withheld amount and verify that the total budget (main + withheld) is sufficient to cover both intrinsic pubdata overhead and the intrinsic computational native. The check should mirror the full `create_resources_for_tx` logic including the withheld resources path, so that the off-chain helper accurately reflects what the bootloader will accept.

---

### Proof of Concept

Consider:
- `native_price = 1`
- `pubdata_price = 1`
- `gas_price = MAX_NATIVE_COMPUTATIONAL + 1` (e.g., `2^35 + 1`)
- `gas_limit = 1`

In the bootloader:
- `native_per_gas = gas_price / native_price = 2^35 + 1`
- `native_prepaid_from_gas = native_per_gas * gas_limit = 2^35 + 1`
- After subtracting intrinsic pubdata overhead (say 0 for simplicity): `native_limit = 2^35 + 1`
- Since `native_limit > MAX_NATIVE_COMPUTATIONAL`, `withheld = 1`, `main_resources.native = MAX_NATIVE_COMPUTATIONAL`
- Intrinsic computational native is subtracted from `main_resources.native`

In the off-chain helper:
- Same `native_per_gas`, same `native_limit = 2^35 + 1`
- After cap: `native_limit = MAX_NATIVE_COMPUTATIONAL`
- The helper checks `native_limit - intrinsic_computational_native >= 0` — passes
- But the helper does **not** verify that the withheld `1` unit is sufficient to cover any post-execution pubdata, nor does it correctly model the split between main and withheld resources [8](#0-7) [9](#0-8)
