### Title
COA Ownership Proof Signature Replay Across Multiple COAs Owned by the Same Cadence Account - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. Because a single Cadence account can own multiple `CadenceOwnedAccount` (COA) resources stored at different capability paths, a signature produced to prove ownership of COA₁ can be replayed — with only the `CapabilityPath` field of the proof changed — to pass ownership verification for COA₂ owned by the same Cadence account. The `isValidSignature` ERC-1271 entry point in `coa.sol` is the concrete on-chain trigger, and any EVM contract that relies on it for access control is directly affected.

### Finding Description

**Root cause — `validateCOAOwnershipProof` does not bind `signedData` to `evmAddress`**

`fvm/evm/stdlib/contract.cdc` lines 1001–1116 implement `validateCOAOwnershipProof`. The function:

1. Verifies that the Cadence account's keys produced valid signatures over the caller-supplied `signedData`.
2. Borrows the COA capability from the caller-supplied `path` and checks that its EVM address equals the caller-supplied `evmAddress`.

The code itself documents the gap:

```
/// Note: this function does not enforce that `signedData` includes `evmAddress`.
/// In principle, a signature produced for one purpose could be replayed here against
/// a different COA owned by the same Cadence account.
``` [1](#0-0) 

The signature verification step uses only `signedData` and the account's public keys — `evmAddress` is never mixed into the signed message:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
``` [2](#0-1) 

After signature verification passes, the function only checks that the COA at the given `path` matches `evmAddress` — it does not verify that `signedData` commits to that `evmAddress`: [3](#0-2) 

**The proof structure is fully attacker-controlled**

`COAOwnershipProof` is an RLP-encoded struct containing `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures`. It is passed as the `_sig` bytes argument to `isValidSignature` and decoded by the precompile: [4](#0-3) 

The precompile decodes the proof from the raw bytes supplied by the EVM caller: [5](#0-4) 

**ERC-1271 entry point in `coa.sol`**

`isValidSignature` calls the precompile with `address(this)` (the calling COA's EVM address) as the `evmAddress` argument:

```solidity
(bool ok, bytes memory data) = cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig
    )
);
``` [6](#0-5) 

**End-to-end exploit path**

Precondition: a single Cadence account owns two COAs — COA₁ published at `/public/coa1` and COA₂ published at `/public/coa2`.

1. A legitimate user signs hash `H` to prove ownership of COA₁. The resulting proof is:
   ```
   { Address: cadenceAddr, CapabilityPath: /public/coa1,
     Signatures: [sig], KeyIndices: [0] }
   ```
   This proof is submitted on-chain (e.g., to an NFT marketplace) and becomes observable.

2. The attacker constructs a modified proof by changing only `CapabilityPath`:
   ```
   { Address: cadenceAddr, CapabilityPath: /public/coa2,
     Signatures: [sig], KeyIndices: [0] }
   ```

3. The attacker calls `COA2.isValidSignature(H, encodedModifiedProof)`.

4. Inside `validateCOAOwnershipProof`:
   - Signature verification: `sig` is valid over `H` using `cadenceAddr`'s key 0 → **passes**.
   - COA address check: the COA at `/public/coa2` has EVM address = COA₂'s address, which equals `address(this)` = COA₂'s address → **passes**.
   - Returns `isValid: true`.

5. `isValidSignature` returns `0x1626ba7e` (valid), impersonating COA₂ to any EVM contract that relies on ERC-1271.

No nonce or replay-prevention mechanism exists in `validateCOAOwnershipProof`; the same signature is valid indefinitely across all COAs owned by the same Cadence account.

### Impact Explanation

Any EVM contract that uses ERC-1271 (`isValidSignature`) to gate privileged actions — NFT marketplace order approval, DAO vote submission, multi-sig wallet authorization, token permit schemes — can be deceived into accepting a signature intended for COA₁ as a valid authorization from COA₂. The attacker does not need to control either COA; they only need to observe a valid proof for COA₁ and re-encode it with a different `CapabilityPath`. The result is unauthorized authorization of on-chain actions on behalf of COA₂, which can lead to direct asset loss (e.g., unauthorized NFT sale orders executed against COA₂'s holdings).

### Likelihood Explanation

The Cadence runtime imposes no limit on the number of COAs a single account may own. As the Flow EVM ecosystem matures, multi-COA patterns (e.g., one COA per application, one COA for DeFi, one COA for NFTs) are a natural usage pattern. The attack requires only: (a) observing a valid proof for COA₁ (proofs are submitted as transaction arguments and are public), and (b) knowing that the same Cadence account owns COA₂ (discoverable from on-chain events). No privileged access, key material, or cryptographic break is required.

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. The simplest fix is to require that `signedData` equals `keccak256(abi.encode(evmAddress, applicationData))` or, equivalently, to hash `evmAddress || signedData` before verifying signatures:

```cadence
// Bind the proof to the specific COA address
let boundData = evmAddress.concat(signedData)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: boundData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Alternatively, enforce at the Cadence level that `signedData` must contain the 20-byte `evmAddress` as a prefix, and reject proofs where it does not. This mirrors the mitigation applied in the Ambire QuickAccManager fix: the identity address must be part of the signed hash.

### Proof of Concept

```
Cadence account A owns:
  COA₁ at /storage/coa1, published at /public/coa1  → EVM address 0xAAAA...
  COA₂ at /storage/coa2, published at /public/coa2  → EVM address 0xBBBB...

Step 1 — Victim signs H for COA₁:
  proof₁ = RLP({ KeyIndices:[0], Address:A, CapabilityPath:"coa1", Signatures:[sig] })
  Submitted to marketplace: COA1.isValidSignature(H, proof₁) → 0x1626ba7e ✓

Step 2 — Attacker re-encodes proof with coa2:
  proof₂ = RLP({ KeyIndices:[0], Address:A, CapabilityPath:"coa2", Signatures:[sig] })

Step 3 — Attacker calls:
  COA2.isValidSignature(H, proof₂)
  → verifyCOAOwnershipProof(0xBBBB, H, proof₂)
  → validateCOAOwnershipProof(A, /public/coa2, H, [0], [sig], 0xBBBB)
     • keyList.verify([sig], H, "FLOW-V0.0-user") → true  (sig is over H, not H||evmAddress)
     • COA at /public/coa2 has address 0xBBBB == evmAddress → true
  → returns 0x1626ba7e ✓  (COA₂ appears to have signed H)

Result: any EVM contract that calls COA2.isValidSignature(H, proof₂) grants COA₂-level
authorization to the attacker, using only a signature the victim produced for COA₁.
```
