### Title
`protocolUnpausePool` Sets Wrong Pause Level, Leaving Pool Permanently Paused Without Admin Cooperation - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`, transitioning a protocol-paused pool (level 2) to admin-paused (level 1) rather than to active (level 0). The pool remains paused after the protocol owner's "unpause" action.

### Finding Description

The pool's pause system uses three levels:

- `0` = active
- `1` = paused by admin
- `2` = paused by protocol [1](#0-0) 

Any non-zero level causes `_checkNotPaused` to revert: [2](#0-1) 

`protocolPausePool` correctly transitions from level 0 or 1 to level 2: [3](#0-2) 

But `protocolUnpausePool` calls `setPause(1)` — admin-paused — instead of `setPause(0)` — active: [4](#0-3) 

The pool remains paused (`pauseLevel != 0`) after the protocol owner's unpause call. The only path to level 0 after this is the pool admin calling `unpausePool`, which requires `cur == 1`: [5](#0-4) 

This is the direct analog to the external bug: the wrong value is written in a critical state-changing operation — wrong pause level instead of wrong `transferFrom` address — producing a broken invariant with the same structural root cause.

### Impact Explanation

After `protocolUnpausePool` executes, `pauseLevel == 1`, so every call to `swap`, `addLiquidity`, and `removeLiquidity` reverts with `PoolPaused`. User funds deposited as LP positions are inaccessible until the pool admin separately calls `unpausePool`. If the protocol paused the pool precisely because the admin was acting maliciously or is unresponsive, the protocol owner has no unilateral path to restore the pool — the malicious/absent admin retains a veto over the unpause. This breaks the core invariant that the protocol owner can independently manage protocol-level pauses, and it locks LP assets and swap flows for an indefinite period.

### Likelihood Explanation

The trigger is the factory owner (a valid, non-malicious actor) calling `protocolUnpausePool` in the normal course of operations — e.g., after a security incident is resolved. No special setup, malicious input, or non-standard token is required. The bug fires on every invocation of `protocolUnpausePool`.

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the intended design is a two-step unpause (protocol → admin), the function must be renamed and documented accordingly, and a separate `protocolForceUnpause` path to level 0 must be provided so the protocol owner retains unilateral control.

### Proof of Concept

1. Pool is active: `pauseLevel == 0`.
2. Factory owner calls `protocolPausePool(pool)` → `pauseLevel == 2`. All swaps/liquidity calls revert.
3. Incident resolved; factory owner calls `protocolUnpausePool(pool)`.
4. `setPause(1)` executes → `pauseLevel == 1`.
5. `swap(...)` is called → `_checkNotPaused()` fires → `revert PoolPaused()`. Pool is still paused.
6. Factory owner has no further lever; only the pool admin calling `unpausePool` can reach level 0.
7. If the admin is absent or adversarial, LP positions and swap flows remain permanently locked.
