Audit Report

## Title
Permanently Stuck ckERC20 Withdrawal Requests with No Reimbursement Path When Gas Prices Exceed Burned Fee - (File: `rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

The `withdraw_erc20` endpoint burns both ckETH (for gas) and ckERC20 tokens at acceptance time using a point-in-time gas estimate. If Ethereum gas prices subsequently spike above the stored `max_transaction_fee`, `create_transaction` returns `InsufficientTransactionFee` and the request is silently rescheduled indefinitely. No reimbursement path, timeout, or cancellation mechanism exists for this state, leaving the user's already-burned tokens permanently unrecoverable unless gas prices fall back below the original estimate.

## Finding Description

**Step 1 — Tokens burned at acceptance time.**
In `rs/ethereum/cketh/minter/src/main.rs` lines 430–492, `withdraw_erc20` calls `estimate_erc20_transaction_fee()`, burns `erc20_tx_fee` ckETH from the caller, burns `ckerc20_withdrawal_amount` ckERC20 from the caller, then records an `Erc20WithdrawalRequest` with `max_transaction_fee: erc20_tx_fee`. [1](#0-0) [2](#0-1) 

**Step 2 — Gas price check at transaction creation time.**
In `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` lines 1159–1168, `create_transaction` computes `actual_min_max_fee_per_gas` from the current gas oracle and compares it against `request_max_fee_per_gas` derived from the stored `max_transaction_fee`. If the current gas price exceeds the stored cap, it returns `Err(CreateTransactionError::InsufficientTransactionFee { ... })`. [3](#0-2) 

**Step 3 — Error handler only reschedules; no reimbursement.**
In `rs/ethereum/cketh/minter/src/withdraw.rs` lines 281–291, the `InsufficientTransactionFee` arm of `create_transactions_batch` logs the event and calls `reschedule_withdrawal_request(request)`. It does not add the request to any reimbursement queue. [4](#0-3) 

**Step 4 — `maybe_reimburse` is never populated for rescheduled requests.**
`maybe_reimburse.insert(withdrawal_id)` is called exclusively inside `record_created_transaction` (line 552 of `transactions/mod.rs`), which is only reached when `create_transaction` succeeds. A request that repeatedly fails with `InsufficientTransactionFee` never reaches `record_created_transaction` and therefore never enters the reimbursement flow. [5](#0-4) 

**Step 5 — No timeout or cancellation mechanism.**
A search for `cancel`, `timeout`, `max_age`, and `expir` across all minter source files returns zero matches. There is no mechanism to bound how long a request can remain in `pending_withdrawal_requests`.

**Step 6 — Asymmetry with `withdraw_eth`.**
`withdraw_eth` has an upfront `minimum_withdrawal_amount` guard (lines 291–296 of `main.rs`) that rejects requests that cannot cover fees before any burn occurs. No equivalent guard exists in `withdraw_erc20` to prevent accepting a request whose fee estimate may become stale. [6](#0-5) 

## Impact Explanation

Any unprivileged user calling `withdraw_erc20` during a period of low gas prices that is followed by a significant spike suffers permanent, unrecoverable loss of both ckETH and ckERC20 tokens. The tokens are burned on-chain; the minter holds no escrowed balance to return. This constitutes a concrete, user-triggered permanent loss of ck-token assets with no recovery path, matching the High impact class: **"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."** The loss is per-user rather than protocol-wide insolvency, and requires a market condition (gas spike), placing it at High rather than Critical.

## Likelihood Explanation

The condition requires a gas price spike after a withdrawal is accepted. Ethereum gas prices are historically volatile; spikes of 5–20× baseline during NFT mints, major protocol launches, or network congestion events are well-documented. The ckBTC minter experienced a directly analogous stuck-transaction incident in June 2025 requiring an emergency upgrade (`rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`), confirming that this class of bug has real-world precedent in the same codebase family. Any user who withdraws ckERC20 during a low-gas window is at risk. No special privileges or attacker coordination are required — the condition is triggered by normal market behavior. [7](#0-6) 

## Recommendation

1. **Add a reimbursement path for rescheduled ckERC20 requests**: After N consecutive `InsufficientTransactionFee` failures (or after a configurable wall-clock threshold tracked in `Erc20WithdrawalRequest.created_at`), emit a `ReimbursedErc20Withdrawal` event and mint back both the ckETH gas fee and the ckERC20 withdrawal amount via the existing `process_reimbursement` infrastructure.
2. **Add a maximum queue age**: Use the existing `created_at` field on `Erc20WithdrawalRequest` to trigger automatic reimbursement after a bounded period (e.g., 24 hours) regardless of retry count.
3. **Add an upfront fee adequacy guard**: Before burning tokens in `withdraw_erc20`, check that `erc20_tx_fee` meets the current `min_max_fee_per_gas` threshold, mirroring the `minimum_withdrawal_amount` guard in `withdraw_eth`.

## Proof of Concept

1. Call `withdraw_erc20` when Ethereum base fee is ~10 gwei. Minter estimates `erc20_tx_fee ≈ 650,000 gwei` (10 gwei × 65,000 gas) and burns ckETH + ckERC20.
2. Ethereum base fee spikes to 200 gwei (e.g., during a major network event).
3. On the next timer tick, `create_transactions_batch` calls `create_transaction` for the queued request.
4. `actual_min_max_fee_per_gas` (≥200 gwei) > `request_max_fee_per_gas` (~10 gwei) → `InsufficientTransactionFee` returned.
5. `reschedule_withdrawal_request` moves the request to the back of the queue. `maybe_reimburse` is never updated.
6. Steps 3–5 repeat on every timer tick indefinitely.
7. The user's ckETH and ckERC20 remain burned with no reimbursement ever triggered.

A deterministic integration test can reproduce this by: (a) initializing the minter state with a low gas fee estimate, (b) calling `withdraw_erc20`, (c) updating the gas fee estimate to a value exceeding `max_transaction_fee / 65_000`, (d) invoking `create_transactions_batch`, and (e) asserting that `maybe_reimburse` remains empty and `pending_withdrawal_requests` still contains the original request after arbitrarily many iterations.
