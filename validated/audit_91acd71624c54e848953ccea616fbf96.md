Based on my thorough investigation of the codebase, I have identified a valid analog to the external report's missing-validation pattern.

---

### Title
Gateway Admits Invoke Transaction with Unverified Signature via `skip_stateful_validations` Deploy-Account UX Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the blockifier's `__validate__` entry-point call — which is the only place the account's signature is checked at admission time — for any invoke transaction whose nonce is `1` and whose sender address appears in `account_tx_in_pool_or_recent_block`. An unprivileged attacker who observes a `deploy_account` transaction in the mempool for address `X` can immediately submit a second invoke transaction for `X` carrying an arbitrary (invalid) signature. The gateway accepts it without verifying the signature, admitting an invalid transaction into the mempool.

### Finding Description

`extract_state_nonce_and_run_validations` is the gateway's stateful admission gate:

```
account_nonce = get_nonce_from_state(tx.contract_address())
skip_validate = run_pre_validation_checks(tx, account_nonce, mempool_client)
run_validate_entry_point(tx, skip_validate)   // __validate__ is skipped when skip_validate=true
``` [1](#0-0) 

`run_pre_validation_checks` calls `skip_stateful_validations`, which returns `true` — meaning "skip `__validate__`" — whenever:

1. The transaction is an `Invoke`,
2. `tx.nonce() == 1`, and
3. `account_nonce == 0` (account not yet deployed), and
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

When `skip_validate` is `true`, `run_validate_entry_point` constructs the `AccountTransaction` with `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`validate_tx` in the blockifier immediately returns `Ok(None)` when `execution_flags.validate` is `false`, so the account's `__validate__` entry point — and therefore the ECDSA signature check — is never executed:

```rust
fn validate_tx(...) -> TransactionExecutionResult<Option<CallInfo>> {
    if !self.execution_flags.validate {
        return Ok(None);
    }
    ...
}
``` [4](#0-3) 

`account_tx_in_pool_or_recent_block` returns `true` as soon as **any** transaction for the address is present in the pool — including the `deploy_account` transaction itself:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

The `validate_contract_address` check in the stateless validator explicitly skips address validation for `DeployAccount` transactions and does not validate the sender address of the attacker's invoke transaction beyond a basic range check: [6](#0-5) 

At block-execution time, `AccountTransaction::new_for_sequencing` always sets `validate: true`, so `__validate__` **is** called then:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        ...
        validate: true,
        ...
    };
    ...
}
``` [7](#0-6) 

The transaction therefore reverts during block execution. Crucially, `handle_nonce` increments the nonce **before** `run_or_revert` is called, and the increment is committed even on revert: [8](#0-7) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker who front-runs the legitimate user's nonce-1 invoke transaction can:

1. Have an invalid (arbitrary-signature) invoke transaction admitted to the mempool without any signature check.
2. Cause that transaction to be included in a block, where it reverts but still increments the account nonce from 1 to 2.
3. Invalidate the legitimate user's nonce-1 invoke transaction (nonce mismatch), breaking the intended `deploy_account + invoke` UX flow.
4. Drain a portion of the pre-funded account's balance through the revert fee.

### Likelihood Explanation

**Medium.** The mempool is observable by all network participants. Any node can watch for incoming `deploy_account` transactions and immediately race-submit a crafted nonce-1 invoke transaction. No privileged access is required; the only constraint is submitting before the legitimate user's invoke transaction occupies the nonce-1 slot.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a narrower predicate that verifies a **`deploy_account` transaction specifically** is present in the mempool for the sender address (e.g., `deploy_account_tx_in_pool(sender_address)`). This preserves the UX intent while preventing an attacker from triggering the skip via an unrelated transaction. Alternatively, add a lightweight signature-format sanity check (e.g., non-empty, correct length) even when the full `__validate__` call is skipped, so that obviously invalid signatures are rejected at the gateway boundary.

### Proof of Concept

```
1. Legitimate user submits:
     deploy_account_tx { class_hash: C, salt: S, constructor_calldata: D, nonce: 0, signature: <valid> }
   → address X = calculate_contract_address(S, C, D, deployer=0)
   → tx accepted; mempool now has X in tx_pool

2. Attacker observes X in the mempool and submits:
     invoke_tx { sender_address: X, nonce: 1, signature: [0xdead, 0xbeef], calldata: [...] }

3. Gateway stateless checks pass (valid address, valid resource bounds, etc.).

4. Gateway stateful checks:
     account_nonce = get_nonce_from_state(X) → 0   (X not yet deployed)
     validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap  → OK
     validate_by_mempool                             → OK (no duplicate hash)
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true
       → returns true
     run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       __validate__ NOT called → signature NOT checked
   → attacker's invalid invoke_tx admitted to mempool

5. Block is built:
     deploy_account_tx executes → X deployed, nonce incremented to 1
     attacker's invoke_tx executes with new_for_sequencing (validate=true):
       __validate__ called → signature [0xdead,0xbeef] fails → tx reverts
       handle_nonce increments nonce to 2 (committed even on revert)

6. Legitimate user's invoke_tx { nonce: 1 } is now invalid (account nonce is 2).
   The deploy_account + invoke UX flow is broken.
```

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(

        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();

            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
