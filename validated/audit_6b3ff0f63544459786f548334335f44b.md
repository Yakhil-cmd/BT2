### Title
Floor Division in `native_per_pubdata` Calculation Causes Systematic Pubdata Undercharging — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` rate — the number of native resource units charged per byte of pubdata — is computed using **floor (integer) division** (`wrapping_div`), while the analogous `native_per_gas` rate uses **ceiling division** (`div_ceil`). When `pubdata_price < native_price`, `native_per_pubdata` truncates to zero, making pubdata completely free for any transaction sender even when the operator intends to charge for it. This is a resource accounting bug directly analogous to the Omni `FeeOracleV1` report: a data-cost component is systematically underestimated due to an incorrect arithmetic operation.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (the L2 transaction validation path), the two key rates are derived as follows:

**`native_per_gas`** — uses ceiling division:
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

**`native_per_pubdata`** — uses floor division (`wrapping_div` = integer division):
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same asymmetry exists in the public API helper `validate_l2_tx_intrinsic_native_resources`:
```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price  ← floor, not ceil
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

`pubdata_price` and `native_price` are independent block-level parameters supplied by the operator oracle: [4](#0-3) 

They represent fundamentally different costs — L1 data-posting cost vs. RISC-V proving cost — and their ratio can easily be less than 1 in normal operation (e.g., when proving is expensive relative to blob fees).

When `pubdata_price < native_price` (e.g., `pubdata_price = 9`, `native_price = 10`):

```
native_per_pubdata = floor(9 / 10) = 0
```

With `native_per_pubdata = 0`, the downstream effects cascade through the entire pubdata charging pipeline:

1. **Intrinsic pubdata overhead becomes zero** — `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0`: [5](#0-4) 

2. **Post-execution pubdata check always passes** — `get_resources_to_charge_for_pubdata` returns 0 native resources regardless of how many bytes were written: [6](#0-5) 

3. **`check_enough_resources_for_pubdata` always returns `true`** — no pubdata-induced revert is possible: [7](#0-6) 

4. **`compute_gas_refund` delta-gas adjustment is suppressed** — since `native_used` does not include pubdata cost, `delta_gas` is lower and the user is charged less gas overall: [8](#0-7) 

---

### Impact Explanation

Any unprivileged transaction sender can generate pubdata (via SSTORE, contract deployments, etc.) up to the block's `pubdata_limit` at zero cost whenever `pubdata_price < native_price`. The operator must still post this pubdata to L1 and bears the full cost. This is a direct resource accounting bug: the fee charged to the user does not cover the operator's actual data-posting expenditure, leading to operator fund loss. The severity scales with the amount of pubdata generated per block.

---

### Likelihood Explanation

`pubdata_price` reflects L1 data availability costs (blob fees, calldata costs), while `native_price` reflects ZK proving costs (RISC-V cycles). These are independent market prices. During periods of low L1 congestion (low blob fees) combined with high proving demand (high `native_price`), `pubdata_price < native_price` is a realistic steady-state condition, not an edge case. The operator may also intentionally set `pubdata_price` to a small non-zero value to charge *something* for pubdata, unaware that floor division silently rounds it to zero.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — undercharges when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` `validate_l2_tx_intrinsic_native_resources`. [9](#0-8) 

This ensures that whenever `pubdata_price > 0`, at least 1 native unit is charged per pubdata byte, preventing free pubdata generation.

---

### Proof of Concept

**Setup:**
- `native_price = 10` (operator-set, reflects proving cost)
- `pubdata_price = 9` (operator-set, reflects L1 data cost — slightly cheaper than proving)
- `eip1559_basefee = 10`, `gas_price = 10`

**Computation:**
```
native_per_gas    = ceil(10 / 10) = 1   ✓ (correct, non-zero)
native_per_pubdata = floor(9 / 10) = 0  ✗ (should be 1, is 0)
```

**Attack:**
1. Attacker submits a transaction calling a contract that performs 100 SSTORE operations, each writing a new storage slot (generating ~6,500 bytes of pubdata).
2. `check_enough_resources_for_pubdata(system, 0, resources, ...)` returns `(true, Resources::empty(), 6500)` — no native resources charged.
3. `compute_gas_refund` sees `native_used ≈ 0` (no pubdata native cost), so `delta_gas = 0`, and the user is refunded all unused gas.
4. The attacker pays only EVM execution gas for the SSTOREs, zero for pubdata.
5. The operator posts 6,500 bytes to L1 at their own expense.

Repeating this across many transactions fills the block's pubdata budget at no cost to attackers, draining operator funds proportional to L1 data prices.

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
```

pub fn check_enough_resources_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    resources: &S::Resources,
    base_pubdata: Option<u64>,
) -> Result<(bool, S::Resources, u64), SystemError> {
    let (pubdata_used, resources_for_pubdata) =
        get_resources_to_charge_for_pubdata(system, native_per_pubdata, base_pubdata)?;
    system_log!(system, "Checking gas for pubdata, resources_for_pubdata: {resources_for_pubdata:?}, resources: {resources:?}\n");
    let enough = resources.has_enough(&resources_for_pubdata);
    Ok((enough, resources_for_pubdata, pubdata_used))
}
