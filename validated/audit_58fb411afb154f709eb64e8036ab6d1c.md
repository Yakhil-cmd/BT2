### Title
Unvalidated Identity-Element `app_pk` Strips Masking in CKD, Exposing Confidential Derived Key to Coordinator — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry-point in `src/confidential_key_derivation/protocol.rs` accepts the requester's public key `app_pk: PublicKey` (`blstrs::G1Projective`) without checking whether it is the group identity element. When `app_pk = G1Projective::identity()`, the ElGamal masking term `y * app_pk` collapses to the identity (zero), causing every participant to transmit their unmasked Lagrange-weighted secret share directly to the coordinator. The coordinator can then aggregate these shares to reconstruct the confidential derived key `msk · H(pk ‖ app_id)` in the clear — the exact secret the protocol is designed to keep hidden.

---

### Finding Description

The CKD protocol encrypts the confidential key under the requester's public key `app_pk` using a one-time ElGamal blinding term. In `compute_signature_share()`:

```
big_s  = private_share · H(pk ‖ app_id)   // participant's secret contribution
big_c  = big_s + app_pk · y               // masked ciphertext component
big_y  = y · G                            // blinding base
```

Each participant sends `(λᵢ · big_y, λᵢ · big_c)` to the coordinator. The coordinator aggregates:

```
C = Σ λᵢ · big_cᵢ = msk · H(pk ‖ app_id) + app_pk · Σ(λᵢ · yᵢ)
```

The requester recovers the confidential key as `C − app_sk · Y`. The coordinator cannot do this because they do not know `app_sk`.

**The flaw:** if `app_pk = G1Projective::identity()`, then `app_pk · y = identity` for any scalar `y`, so:

```
big_c = big_s + identity = big_s = private_share · H(pk ‖ app_id)
```

Each participant's message becomes `(λᵢ · y · G, λᵢ · private_shareᵢ · H(pk ‖ app_id))`. The coordinator aggregates the second component to obtain `msk · H(pk ‖ app_id)` directly — no `app_sk` needed.

The `ckd()` function performs four input checks (participant count, duplicates, self-membership, coordinator-membership) but **no check that `app_pk` is a non-identity group element**: [1](#0-0) 

The masking computation that silently degenerates when `app_pk = identity`: [2](#0-1) 

The coordinator aggregation that then directly accumulates the unmasked secret: [3](#0-2) 

The `blstrs::G1Projective` type freely represents the identity element; the `BLS12381G1Group` serialization layer does reject it on deserialization, but `app_pk` is passed as a live Rust value — never serialized/deserialized through that guard — so the check is never reached. [4](#0-3) 

---

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

The confidential derived key `msk · H(pk ‖ app_id)` is the core secret output of the CKD protocol. Its disclosure to the coordinator completely breaks the protocol's confidentiality guarantee: the coordinator learns the exact key that was supposed to be delivered only to the requester (after decryption with `app_sk`). Because the coordinator receives individual per-participant shares `λᵢ · private_shareᵢ · H(pk ‖ app_id)`, they also learn each participant's Lagrange-weighted secret contribution, which is a partial disclosure of the underlying signing shares.

---

### Likelihood Explanation

**High.** The coordinator role is a privileged but not fully trusted party in the protocol. A malicious coordinator controls the orchestration layer and can supply any `app_pk` value to all participants. Passing `G1Projective::identity()` requires no cryptographic capability — it is a single-line Rust call (`G1Projective::identity()` or `G1Projective::default()`). There is no protocol-level mechanism for honest participants to detect or reject an identity `app_pk`, since they each independently compute their share and send it to the coordinator without cross-checking `app_pk` with peers.

---

### Recommendation

Add an explicit identity-element check at the start of `ckd()`, immediately after the existing participant-list validations:

```rust
// Reject the identity element: app_pk = identity strips the ElGamal mask,
// exposing the confidential derived key to the coordinator.
if app_pk.is_identity().into() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the guard already present in `BLS12381G1Group::deserialize()` and `BLS12381G2Group::deserialize()` for serialized group elements, and closes the gap for live values passed directly through the API. [5](#0-4)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
