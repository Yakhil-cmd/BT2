### Title
Gateway Signature Verification Bypass via `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo Gateway unconditionally skips the `__validate__` entry-point call (the account's on-chain signature check) for any Invoke transaction with nonce=1 whose sender address appears in the mempool pool or recent-block state. An unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a forged Invoke with an arbitrary (invalid) signature for the same address and have it admitted to the mempool without any signature verification.

### Finding Description

**Root cause — `skip_stateful_validations`**

`crates/apollo_gateway/src/stateful_transaction_validator.rs` lines 429–460:


**The `account_tx_in_pool_or_recent_block` check is too broad**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`tx_pool.contains_account` returns `true` for **any** transaction from the address — not specifically a `deploy_account`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is only valid if the only way an undeployed account (on-chain nonce = 0) can appear in the pool is via a `deploy_account`. That invariant is not enforced anywhere.

**End-to-end exploit path**

1. Victim pre-funds address `A` and submits a `deploy_account` transaction. The mempool now has `tx_pool.contains_account(A) == true`.
2. Attacker (no privileged access) submits a forged `Invoke` transaction:
   - `sender_address = A`
   - `nonce = 1` (Felt::ONE)
   - `signature = [0x41, 0x41]` (arbitrary garbage)
   - valid resource bounds (passes stateless check)
3. Gateway stateless validation passes — `validate_tx_signature_size` only checks length, not content.
4. `convert_rpc_tx_to_internal_and_executable_txs` succeeds — no signature verification here.
5. `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(A)` → `Nonce(0)` (account not yet on-chain)
   - `validate_state_preconditions`: nonce `0 ≤ 1` passes; resource bounds pass
   - `validate_by_mempool`: passes (no duplicate, nonce in range)
   - `skip_stateful_validations`: `nonce==1 ∧ account_nonce==0 ∧ account_tx_in_pool_or_recent_block(A)==true` → returns `true`
   - `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate=false` → `__validate__` is **never called**
6. Forged Invoke is forwarded to the mempool via `mempool_client.add_tx(...)` and accepted.

**What happens at execution time**

The batcher creates `AccountTransaction` via `new_for_sequencing`, which sets `execution_flags.validate = true` (the default). When the batcher executes the forged Invoke, `__validate__` is called and fails (invalid signature), causing the transaction to be rejected. The account is not charged fees. However, the forged transaction **was admitted to the mempool** and consumed a mempool slot, wasted batcher resources, and occupied a nonce slot that blocks the victim's legitimate nonce-1 transaction.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An unprivileged attacker can flood the mempool with signature-invalid Invoke transactions targeting any address that has a pending `deploy_account`. Each forged transaction:
- Occupies a mempool slot for address `A` at nonce 1, potentially displacing or delaying the victim's legitimate nonce-1 transaction (depending on mempool replacement policy).
- Forces the batcher to attempt execution and discard the transaction, wasting sequencer resources.
- Can be repeated at scale for every observed `deploy_account` in the mempool, constituting a sustained DoS against the deploy-account UX flow.

The invariant violated: **every transaction admitted to the mempool must have passed account signature verification**. This invariant is deliberately broken for the UX convenience path without a compensating control.

### Likelihood Explanation

**Medium-High.** The mempool is publicly observable. Any attacker watching the RPC/P2P layer can detect `deploy_account` transactions and immediately submit forged Invokes. No privileged keys, no internal access, and no cryptographic break are required. The only cost is the gas resource bounds specified in the forged transaction (which are never actually paid since the transaction is rejected at execution).

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a **`deploy_account` transaction** (not just any transaction) exists in the pool for the sender address. Concretely:

1. Add a `deploy_account_in_pool(address: ContractAddress) -> bool` method to the mempool that inspects the transaction type of the pooled entry for the address.
2. In `skip_stateful_validations`, replace the call to `account_tx_in_pool_or_recent_block` with `deploy_account_in_pool`.
3. Alternatively, enforce the skip only when the mempool explicitly tracks a pending `deploy_account` hash for the address (analogous to the `deploy_account_tx_hash` parameter in `PyValidator::should_run_stateful_validations`).

```rust
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,

    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.

                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
```
