### Title
`protocolUnpausePool` Always Transitions to Admin-Paused (Level 1) Instead of Active (Level 0), Leaving Pool Permanently Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` unconditionally calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). After the protocol owner "unpauses" a pool that was previously active, the pool remains paused at level 1, and every subsequent `swap` call reverts with `PoolPaused`. The pool admin must issue a separate `unpausePool` transaction to restore swap functionality — a step that is not required by the function's name or semantics, and that may be delayed or impossible depending on the admin setup.

---

### Finding Description

The pause-level state machine in `MetricOmmPool` uses three levels:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`_checkNotPaused` reverts on **any** non-zero level:

```solidity
// MetricOmmPool.sol
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

`swap` is gated by `whenNotPaused`, so levels 1 and 2 both block swaps identically.

The factory's protocol-pause path correctly transitions 0→2 or 1→2:

```solidity
// MetricOmmPoolFactory.sol L392-396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
```

The protocol-unpause path, however, always targets level **1**, not level **0**:

```solidity
// MetricOmmPoolFactory.sol L399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← should be setPause(0)
}
```

When the pool was **active (level 0)** before the protocol pause, the round-trip is:

```
active (0) → protocolPausePool → protocol-paused (2)
             protocolUnpausePool → admin-paused (1)   ← pool still paused
```

The pool is now stuck at level 1. Every `swap` call reverts with `PoolPaused` even though the protocol owner has explicitly unpaused. The admin must separately call `unpausePool` to reach level 0.

---

### Impact Explanation

- **Broken core pool functionality**: swaps are blocked after the protocol owner explicitly unpauses. The pool is unusable for trading until the admin takes an additional, unannounced action.
- **State corruption**: a protocol pause+unpause cycle on an active pool silently changes the admin-pause state from 0 to 1 without the admin's consent, violating the invariant that admin-pause state is only changed by the pool admin.
- **Potential permanent DoS**: if the pool admin is a contract that cannot call `unpausePool` (e.g., a governance contract with no such proposal type), the pool is permanently stuck at level 1 after any protocol unpause.

---

### Likelihood Explanation

The trigger is a routine operational action: the protocol owner pauses a live (level-0) pool for any reason (oracle issue, exploit response, upgrade) and then unpauses it. This is an expected lifecycle event. Every such cycle on an active pool leaves the pool in the wrong state.

---

### Recommendation

Change `protocolUnpausePool` to target level 0 (active), not level 1:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // restore to active
}
```

If the intended design is that the protocol unpause only hands control back to the admin (i.e., the admin must still explicitly unpause), the function must be renamed and documented accordingly, and the admin-pause path must be adjusted so that a pool that was active before a protocol pause is not silently demoted to admin-paused after the protocol releases it.

---

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel` becomes 2.
3. Protocol owner calls `protocolUnpausePool(pool)` → `setPause(1)` is called → `pauseLevel` becomes 1.
4. Any user calls `pool.swap(...)` → `_checkNotPaused()` fires → `revert PoolPaused()`.
5. Pool admin must call `unpausePool(pool)` to reach level 0 — a step not implied by the protocol unpause action.
