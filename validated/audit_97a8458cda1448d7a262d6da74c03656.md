### Title
Missing Caller Restriction on `update_rewards` Allows Anyone to Block Per-Block Reward Distribution - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in the Staking contract is specified to be callable only by the Starknet sequencer, but the implementation contains no such access control check. Any unprivileged caller can invoke it with `disable_rewards: true`, consuming the single per-block reward slot without distributing rewards, causing the sequencer's legitimate call to revert with `REWARDS_ALREADY_UPDATED` and permanently denying stakers their block rewards for that block.

### Finding Description
The spec at `docs/spec.md` line 1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` performs no caller identity check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... no get_caller_address() == sequencer check ...
``` [1](#0-0) 

The function then unconditionally writes `current_block_number` to `last_reward_block`:

```cairo
self.last_reward_block.write(current_block_number);
if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [2](#0-1) 

This is the direct analog to the `burnFrom` bypass: `burn` has `onlyOwner` (the restricted path), while `burnFrom` is the unrestricted path that achieves the same token destruction. Here, the spec mandates sequencer-only access for `update_rewards`, but the code exposes it to any caller. The `disable_rewards` boolean is the bypass lever â€” an attacker passes `true` to consume the slot without distributing rewards, exactly as `burnFrom` bypasses `onlyOwner` to destroy tokens without permission.

The spec confirms the intent: [3](#0-2) 

### Impact Explanation
An attacker calls `update_rewards(victim_staker_address, disable_rewards: true)` before the sequencer does. This:
1. Sets `last_reward_block = current_block_number` with no rewards distributed.
2. The sequencer's subsequent call for the same block reverts with `REWARDS_ALREADY_UPDATED`.
3. Stakers and pool members receive zero block rewards for that block.

Repeated across every block, this constitutes sustained freezing of unclaimed yield for targeted stakers. This matches the allowed impact: **Temporary freezing of unclaimed yield / Griefing with damage to users or protocol**.

### Likelihood Explanation
The function is publicly callable with no authentication. The attacker only needs to know a valid active staker address (publicly readable from chain events) and submit a transaction per block. The cost is gas per block. On Starknet, the sequencer controls ordering, which limits opportunistic front-running, but a determined attacker running their own transactions can still race the sequencer's reward-distribution call, especially if the sequencer's call is not the first transaction in a block.

### Recommendation
Add a caller check enforcing that only the designated sequencer address (or a role-controlled address) can invoke `update_rewards`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   assert!(
+       get_caller_address() == self.sequencer_address.read(),
+       "{}",
+       Error::CALLER_IS_NOT_SEQUENCER,
+   );
    ...
```

Alternatively, restrict via the existing roles component (e.g., `OPERATOR_ROLE`) consistent with the rest of the contract's access control patterns. [4](#0-3) 

### Proof of Concept
1. Staker `S` is active with non-zero balance.
2. Attacker `A` (any address) calls `update_rewards(S, disable_rewards: true)` at block `N`.
3. `last_reward_block` is set to `N`; no rewards are distributed.
4. Sequencer calls `update_rewards(S, disable_rewards: false)` at block `N` â†’ reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker `S` receives no block rewards for block `N`.
6. Repeat for every block to permanently deny yield. [5](#0-4) [6](#0-5)
