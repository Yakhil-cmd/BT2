### Title
Missing Zero-Address Validation in `change_reward_address` Permanently Freezes Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
Both `change_reward_address` in the Staking contract and `change_reward_address` in the Pool contract accept `reward_address = 0x0` without rejection. Once stored, every subsequent attempt to claim rewards triggers a `checked_transfer` to the zero address, which reverts on Starknet's ERC20 implementation, permanently freezing the staker's or pool member's unclaimed yield with no recovery path.

### Finding Description
The `change_reward_address` function in `staking.cairo` validates only that the new address is not a registered token address (`REWARD_ADDRESS_IS_TOKEN`), but performs no zero-address check:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No assert against zero address
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
```

The same gap exists in `pool.cairo`'s `change_reward_address`, which only checks `token_dispatcher.contract_address.read() != reward_address`. The same omission is present at registration time in `stake` (staking.cairo) and `enter_delegation_pool` (pool.cairo).

When rewards are later claimed, `send_rewards_to_staker` unconditionally transfers to the stored `reward_address`:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

Starknet's OpenZeppelin ERC20 `transfer` panics on a zero recipient. The transaction reverts, the `unclaimed_rewards_own` field is never cleared, and no future call can succeed — the yield is permanently frozen.

### Impact Explanation
Any staker or pool member who sets (or initially registers with) `reward_address = 0x0` will have their accumulated STRK yield permanently frozen inside the contract. The `unclaimed_rewards_own` balance remains non-zero but is unreachable: every `claim_rewards` call reverts at the transfer step, and there is no admin override or recovery function. This matches the allowed High impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The entry path requires only a single call to `change_reward_address(0x0)` by the staker themselves, or passing `reward_address: 0x0` to `stake` / `enter_delegation_pool`. No privileged role is needed. Accidental zero-address submission is a realistic user error (e.g., a front-end bug, a scripting mistake, or a migration that clears the field). The protocol provides no safeguard.

### Recommendation
Add an explicit zero-address assertion in all four entry points:
- `stake` in `staking.cairo`
- `change_reward_address` in `staking.cairo`
- `enter_delegation_pool` in `pool.cairo`
- `change_reward_address` in `pool.cairo`

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

The error constant `ZERO_ADDRESS` already exists in the codebase.

### Proof of Concept

1. Staker calls `stake(reward_address: 0x0, operational_address: X, amount: MIN_STAKE)` — accepted, no revert.
2. Attestation occurs; `unclaimed_rewards_own` accumulates a non-zero value.
3. Staker calls `claim_rewards(staker_address)`.
4. `send_rewards_to_staker` executes `checked_transfer(recipient: 0x0, amount: rewards)`.
5. ERC20 panics: "ERC20: transfer to the zero address".
6. Transaction reverts; `unclaimed_rewards_own` remains non-zero forever.
7. All future `claim_rewards` calls repeat step 4–6 indefinitely.

Alternatively, an existing staker calls `change_reward_address(0x0)` at any point after staking, triggering the same freeze from that moment forward. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
