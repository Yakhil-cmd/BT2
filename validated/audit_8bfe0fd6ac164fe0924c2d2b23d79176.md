### Title
`calculateNFTRoyalties` Omits Royalty Deduction from `nftSellerNetAmount` in `TokenForNFT` Offers, Causing Seller to Approve Wrong Net Amount - (File: packages/gui/src/components/offers/utils.ts)

### Summary

In `calculateNFTRoyalties` (`packages/gui/src/components/offers/utils.ts`), `nftSellerNetAmount` is unconditionally set to the full offered `amount` regardless of exchange type. For `TokenForNFT` offers (buyer making an offer to purchase an NFT), Chia's protocol deducts the royalty from the buyer's locked tokens before paying the seller, so the seller's true net proceeds are `amount - royaltyAmount`. Because the GUI hardcodes `nftSellerNetAmount = amount`, the NFT Offer Viewer displays an inflated "Net Proceeds" figure to the seller. A buyer can craft a `TokenForNFT` offer for any royalty-bearing NFT, share it, and the seller will see a "Net Proceeds" value that is higher than what they will actually receive on-chain, causing them to approve a transfer at a worse effective price than displayed.

### Finding Description

**Root cause — `calculateNFTRoyalties` (utils.ts lines 306–329):**

```typescript
export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;          // ← always the full offered amount
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );                                              // ← correct deduction was commented out
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken
      ? amount + royaltyAmount
      : amount + makerFee + royaltyAmount;            // ← also wrong for TokenForNFT
  ...
}
```

The commented-out block shows the originally intended formula. For `TokenForNFT`, Chia's on-chain royalty mechanism deducts `royaltyAmount` from the buyer's locked coins before crediting the seller. The correct values are:

| Field | Correct | Actual |
|---|---|---|
| `nftSellerNetAmount` | `amount - royaltyAmount` | `amount` |
| `totalAmount` (TokenForNFT) | `amount + makerFee` | `amount + makerFee + royaltyAmount` |

**Propagation to the offer viewer (`NFTOfferViewer.tsx`):**

`overrideNFTSellerAmount` is computed from the wrong `nftSellerNetAmount`:

```typescript
// NFTOfferViewer.tsx lines 397–402
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)   // ← inflated
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
```

This value is passed as `overrideNFTSellerAmount` into `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it replaces the raw on-chain amount in the "You will receive" display:

```typescript
// OfferSummaryRow.tsx line 146
const amount = overrideNFTSellerAmount ?? originalAmount;
```

Additionally, the "Net Proceeds" section (lines 580–598) renders `nftSaleInfo?.nftSellerNetAmount` directly with the tooltip "The net proceeds include the asking price, **minus** any associated creator fees" — but the value shown is the full offered amount, not the deducted amount.

### Impact Explanation

A seller viewing an imported `TokenForNFT` offer for an NFT with a non-zero `royaltyPercentage` sees:

- **"You will receive"**: full offered amount (e.g., 1 XCH)
- **"Net Proceeds"**: full offered amount (e.g., 1 XCH), with a tooltip explicitly claiming royalties have been subtracted

The seller actually receives `amount - royaltyAmount` on-chain (e.g., 0.9 XCH for a 10% royalty). The seller approves the offer based on a displayed net proceeds figure that is `royaltyAmount` higher than reality. This is a direct case of offer state spoofing causing a user to approve the wrong amount — matching the **High** impact category: *"Corruption, spoofing, or unsafe trust of... offer... state that causes a user to approve... the wrong... amount."*

### Likelihood Explanation

- Any unprivileged user can create a `TokenForNFT` offer for any NFT that has a non-zero `royaltyPercentage` (a standard NFT attribute set at mint time).
- The seller only needs to open the offer file in the GUI's NFT Offer Viewer — a normal workflow for accepting offers received via messaging or marketplaces.
- The discrepancy scales with royalty percentage: a 10% royalty on a 10 XCH offer inflates the displayed net proceeds by 1 XCH.

### Recommendation

In `calculateNFTRoyalties`, restore the royalty deduction for `nftSellerNetAmount` when `exchangeType === NFTOfferExchangeType.TokenForNFT`:

```typescript
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? parseFloat((amount - royaltyAmount).toFixed(12))
    : amount;

const totalAmount: number =
  exchangeType === NFTOfferExchangeType.NFTForToken
    ? amount + royaltyAmount
    : amount + makerFee;   // royalty is paid from the offered amount, not added on top
```

Also update the tooltip in `NFTOfferEditor.tsx` for `TokenForNFT` to consistently state that royalties are deducted from the offered amount (not paid by the accepting party), aligning with the viewer's tooltip at lines 562–568.

### Proof of Concept

**Setup:** NFT with `royaltyPercentage = 1000` (10%). Buyer creates a `TokenForNFT` offer offering 1 XCH.

**Attacker (buyer) action:** Creates and shares the offer file.

**Victim (seller) view in `NFTOfferViewer`:**

| UI Label | Displayed Value | Actual On-Chain Value |
|---|---|---|
| You will receive | 1 XCH | 0.9 XCH |
| NFT Purchase Price | 1 XCH | 1 XCH (buyer's locked amount) |
| Creator Fee (10%) | 0.1 XCH | 0.1 XCH ✓ |
| **Net Proceeds** | **1 XCH** | **0.9 XCH** |
| Total Amount Offered | 1.1 XCH | 1 XCH |

The seller reads "Net Proceeds: 1 XCH" with the tooltip "The net proceeds include the asking price, minus any associated creator fees" and clicks **Accept Offer**. On-chain settlement delivers 0.9 XCH to the seller — 0.1 XCH less than the GUI stated.

**Code path:**
1. `getNFTPriceWithoutRoyalties(summary)` → `amount = 1` (XCH) [1](#0-0) 
2. `calculateNFTRoyalties(1, makerFee, 10, TokenForNFT)` → `nftSellerNetAmount = 1` (wrong) [2](#0-1) 
3. `overrideNFTSellerAmount = chiaToMojo(1)` → passed to `OfferSummaryTokenRow` [3](#0-2) 
4. `OfferSummaryTokenRow` displays `overrideNFTSellerAmount` (1 XCH) instead of true net (0.9 XCH) [4](#0-3) 
5. "Net Proceeds" renders `nftSaleInfo?.nftSellerNetAmount = 1` with misleading tooltip [5](#0-4)

    }
