### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions When Any Account Transaction Exists in Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator skips the `__validate__` entry point (the account's signature check) for any Invoke transaction with `nonce == 1` whenever the sender address has **any** transaction in the mempool or a recent block — not exclusively a `DeployAccount` transaction. An unprivileged attacker can exploit this to inject an Invoke transaction carrying an arbitrary (invalid) signature for a victim account into the mempool without the gateway ever calling `__validate__`, bypassing the admission-level authorization invariant.

---

### Finding Description

The UX feature is documented in the code:

> "Check if validation of an invoke transaction should be skipped due to deploy_account not being processed yet."

The intended two-transaction flow is:
- TX1: user submits `DeployAccount` (nonce 0) — account does not exist yet.
- TX2: user submits `Invoke` (nonce 1) — account still does not exist, so `__validate__` cannot run.

The gateway skips `__validate__` for TX2 only when it can confirm TX1 is pending. The guard is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437–456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697–700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type — `Invoke`, `Declare`, or `DeployAccount`. The code comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incorrect for the following case:

**Vulnerable scenario:**
1. Account `A` is already deployed; its on-chain nonce is `0` (it has never sent a transaction).
2. The legitimate owner submits a valid `Invoke` with `nonce = 0`; it passes `__validate__` and enters the mempool. Now `tx_pool.contains_account(A)` is `true`.
3. An attacker submits an `Invoke` for `A` with `nonce = 1` and an **arbitrary/invalid signature**.
4. The gateway evaluates: `tx.nonce() == 1` ✓, `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block(A)` ✓ → `skip_validate = true`.
5. `run_validate_entry_point` is called with `validate: !skip_validate = false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 311–312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

6. `StatefulValidator::validate` is called; because `execution_flags.validate == false`, `validate_tx` returns `Ok(None)` immediately without invoking `__validate__`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 999–1001
if !self.execution_flags.validate {
    return Ok(None);
}
``` [4](#0-3) 

7. The attacker's transaction — carrying an invalid signature — is forwarded to the mempool via `mempool_client.add_tx(add_tx_args)` and admitted. [5](#0-4) 

The broken invariant is: **every Invoke transaction must have its `__validate__` entry point executed before admission to the mempool**. The `skip_stateful_validations` path violates this invariant for any account whose address appears in the mempool for any reason, not only because a `DeployAccount` is pending.

---

### Impact Explanation

This matches the allowed High impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

An attacker can inject Invoke transactions with forged/invalid signatures for any account that satisfies `nonce_in_state == 0 && any_tx_in_mempool`. The gateway's signature-validation gate — the only admission-level authorization check — is bypassed. The injected transactions occupy mempool slots, can displace or delay legitimate nonce-1 transactions from the real owner, and are sequenced into a block before the batcher's execution-time `__validate__` call rejects them (causing a reverted transaction and wasted block space).

---

### Likelihood Explanation

The conditions are common: any freshly deployed account (on-chain nonce = 0) that has submitted even one transaction (nonce = 0) to the mempool is immediately vulnerable. An attacker monitoring the public mempool can identify such accounts and race to inject nonce-1 transactions before the legitimate owner's nonce-0 transaction is committed.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that is specific to `DeployAccount` transactions. The mempool should expose a method such as `has_pending_deploy_account(address) -> bool` that returns `true` only when a `DeployAccount` transaction for that address is in the pool. The `skip_stateful_validations` function should call this narrower predicate:

```rust
// Proposed fix
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .has_pending_deploy_account(tx.sender_address())  // NEW: type-specific check
                .await
                ...
        }
    }
    Ok(false)
}
```

This preserves the intended UX (deploy_account + invoke submitted together) while closing the bypass for accounts that have only non-deploy transactions in the mempool.

---

### Proof of Concept

```
1. Deploy account A on-chain (nonce becomes 0 in state).
2. Owner submits Invoke(sender=A, nonce=0, valid_sig) → passes __validate__ → enters mempool.
   Now: account_tx_in_pool_or_recent_block(A) == true.
3. Attacker submits Invoke(sender=A, nonce=1, sig=[0xdead, 0xbeef]) to the gateway.
4. Gateway stateful validator:
     account_nonce = get_nonce(A) = 0
     validate_nonce: 0 <= 1 <= 0+max_gap  ✓
     validate_by_mempool: no duplicate, nonce gap ok  ✓
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true
       → returns true (skip __validate__)
     run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       blockifier StatefulValidator.validate() → validate_tx returns Ok(None)  ← no sig check
5. Gateway calls mempool_client.add_tx(attacker_tx) → attacker tx admitted.
6. Batcher later executes: __validate__ called with sig=[0xdead,0xbeef] → INVALID_SIGNATURE → tx reverts.
   But the invalid tx was sequenced, consuming block resources and a nonce slot.
```
