### Title
Presign Protocol Accepts Triples From Mismatched Participant Sets Without Validation - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary
The OT-based ECDSA presign protocol accepts `triple0` and `triple1` as caller-supplied inputs but performs no validation that these triples were generated by the same participant set as the current presigning session. A malicious coordinator can supply triples from a different participant set, causing all honest parties to silently produce a cryptographically invalid presignature that passes internal consistency checks but fails at signing time.

### Finding Description
In `src/ecdsa/ot_based_ecdsa/presign.rs`, the `presign()` entry point accepts a `PresignArguments` struct containing two Beaver triples (`triple0`, `triple1`) and a `keygen_out`. The only cross-validation performed between the triples and the current session is a threshold equality check:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [1](#0-0) 

There is no check that the triples were generated by the same participant set as the current `participants` argument. The code explicitly acknowledges this omission:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [2](#0-1) 

Inside `do_presign`, the Lagrange coefficient is computed relative to the **current** participant list:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
``` [3](#0-2) 

If the triples were generated under a different participant set (e.g., `{P1, P2, P4}` instead of the current `{P1, P2, P3}`), the Lagrange linearization is applied with the wrong basis. The triple is internally self-consistent, so the two verification checks that follow:

```rust
if big_e != (ProjectivePoint::GENERATOR * e).to_affine() { ... }
if (ProjectivePoint::GENERATOR * alpha != big_k + big_a)
    || (ProjectivePoint::GENERATOR * beta != big_x + big_b) { ... }
``` [4](#0-3) 

…both **pass** (because the triple's internal algebraic relationships hold regardless of which participant set is used), but the resulting `PresignOutput` carries shares linearized against the wrong basis. The presignature is silently corrupted and will produce an invalid signature.

The `PresignArguments` struct exposes `triple0` and `triple1` as public fields with no participant-set binding:

```rust
pub struct PresignArguments {
    pub triple0: (TripleShare, TriplePub),
    pub triple1: (TripleShare, TriplePub),
    pub keygen_out: KeygenOutput,
    pub threshold: ReconstructionLowerBound,
}
``` [5](#0-4) 

### Impact Explanation
A malicious coordinator controlling the `PresignArguments` supplied to each honest participant can provide triples from a different participant set (or mix triples from two different sessions). All honest parties will complete the presign protocol without error, each producing a `PresignOutput` that is internally consistent but cryptographically invalid relative to the actual signing key and participant set. When these outputs are consumed in the signing phase, the resulting signature will fail verification. This constitutes **corruption of presign outputs so honest parties accept unusable cryptographic outputs**, matching the High impact tier.

### Likelihood Explanation
The coordinator role is a standard part of the OT-based ECDSA protocol. In any deployment where the coordinator manages a pool of pre-generated triples across multiple sessions or participant configurations, a malicious coordinator has direct, unauthenticated control over which triples are passed into `PresignArguments`. No out-of-band mechanism in the library prevents this substitution. Likelihood is **Medium**: the attack requires a malicious coordinator but no cryptographic break, no key leakage, and no external dependency.

### Recommendation
`TriplePub` should include a binding to the participant set that generated it (e.g., a hash of the sorted participant list committed during triple generation). The `presign()` entry point should verify that `triple0.1.participants == triple1.1.participants == hash(current participants)` before proceeding. This closes the same class of mismatch that the code comment already identifies as an omitted check.

### Proof of Concept
1. Run triple generation for participant set `S0 = {P1, P2, P4}` at threshold 2, producing `(triple0_pub_S0, triple0_shares_S0)`.
2. Run a presigning session for participant set `S1 = {P1, P2, P3}` at threshold 2, but supply each participant with `triple0 = (triple0_shares_S0[i], triple0_pub_S0)`.
3. Inside `do_presign`, `lambda_me` is computed for `S1`. The shares `k_i` from `S0` are linearized with `S1`'s Lagrange coefficients, producing `k_prime_i` that does not sum to the correct secret nonce.
4. The check `big_e == (G * e).to_affine()` passes because `e = sum(lambda_{S0,i} * e_i)` is internally consistent within the triple.
5. All participants output a `PresignOutput` with no error. The presignature is silently wrong.
6. Passing this presignature into `sign()` produces a signature `(r, s)` that fails `verify(public_key, msg, sig)`.
