### Title
Understated "Total Amount with Royalties" in WalletConnect Offer Confirmation Due to Incorrect Amount Splitting Across Multiple NFTs - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible payment amount equally among all NFTs before applying each NFT's royalty percentage. The Chia protocol charges each NFT's royalty on the **full** fungible amount, not a split portion. This causes the WalletConnect confirmation dialog to display a materially understated "Total Amount with Royalties," leading users to approve offers while believing they will spend less than the transaction actually deducts.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect command display for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. When multiple NFTs with royalties are on one side of a trade, the function computes:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides total by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The `splitAmount` division is wrong. Each NFT's royalty in the Chia protocol is applied to the **full** fungible amount, not to `amount / nftCount`. The correct calculation (used by the actual `nft_calculate_royalties` RPC, as consumed by `OfferBuilderXCHSection` and `OfferBuilderTokensSection`) is:

```
royaltyAmount = Σ (fullAmount × royaltyPercentage_i / 10000)
```

The test fixture itself encodes the buggy behavior as expected output: [2](#0-1) 

For the test case (0.0001 XCH, NFT1 royalty 5%, NFT2 royalty 0.1%):
- **Displayed** (split): `(50000000 × 500 + 50000000 × 10) / 10000 = 2,550,000 mojos` → total `0.00010255 XCH`
- **Actual on-chain**: `(100000000 × 500 + 100000000 × 10) / 10000 = 5,100,000 mojos` → total `0.0001051 XCH`

The `amountWithRoyalties` field is rendered directly in the WalletConnect confirmation dialog (`Confirm.tsx`) under "Total Amount with Royalties": [3](#0-2) 

The `withRoyaltyTotals` function feeds this display value into the spending summary shown before the user approves the WalletConnect request: [4](#0-3) 

### Impact Explanation

A user reviewing a WalletConnect `take_offer` or `create_offer_for_ids` request involving multiple royalty-bearing NFTs sees an understated total cost. The actual transaction, once signed and submitted, deducts the correct (higher) royalty amounts computed by the full node. The user approved based on a false lower figure. With two NFTs each carrying a 10% royalty and a 1 XCH payment, the dialog shows 1.1 XCH total but the wallet actually spends 1.2 XCH — a 9% understatement that scales with royalty rates and NFT count.

This matches the **High** impact category: WalletConnect state causes a user to approve the wrong amount.

### Likelihood Explanation

Any party who can present a WalletConnect offer to a victim — including a malicious dApp or a crafted offer link — can trigger this path. No special privileges are required. The attacker only needs to construct an offer containing two or more NFTs with non-zero royalties, which is a standard Chia offer format. The victim's confirmation dialog will silently understate the cost.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be applied to the full fungible `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This aligns `formatAmountWithRoyalties` with the actual on-chain calculation performed by `nft_calculate_royalties` and with the correct accumulation logic in `OfferBuilderXCHSection`: [5](#0-4) 

### Proof of Concept

1. Attacker creates an offer: sell 2 NFTs (NFT-A: 10% royalty, NFT-B: 10% royalty) for 1 XCH.
2. Attacker presents the offer to victim via WalletConnect (`chia_wallet.take_offer`).
3. Victim's confirmation dialog calls `formatAmountWithRoyalties` with `amount = 1_000_000_000_000n` and `royaltyPercentages = [1000, 1000]`.
4. `splitAmount = 500_000_000_000n`; displayed royalty = `50_000_000_000 + 50_000_000_000 = 100_000_000_000` mojos → dialog shows **1.1 XCH**.
5. Victim approves. Actual on-chain royalties = `100_000_000_000 + 100_000_000_000 = 200_000_000_000` mojos → wallet spends **1.2 XCH**.
6. Victim loses 0.1 XCH more than the approved amount indicated.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
