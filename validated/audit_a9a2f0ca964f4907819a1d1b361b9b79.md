### Title
`ft_on_transfer` Does Not Validate Caller Is a Legitimate NEP-141 Token Contract, Enabling Unbacked ERC-20 Token Minting — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The public NEAR entrypoint `ft_on_transfer` in the Aurora Engine does not validate that the caller is a legitimate NEP-141 token contract that has actually transferred tokens. Any NEAR account that is registered as a NEP-141 token in the engine can call `ft_on_transfer` directly with an arbitrary `amount`, causing the engine to mint the corresponding ERC-20 tokens on Aurora without any real token deposit. Because `deploy_erc20_token` also has no access control, an unprivileged attacker can register their own account and exploit this in a two-step attack.

---

### Finding Description

`ft_on_transfer` is the NEP-141 receiver callback, exposed as a public WASM entrypoint in `engine/src/lib.rs`:

```rust
// engine/src/lib.rs:603
pub extern "C" fn ft_on_transfer() { ... }
```

Inside `engine/src/contract_methods/connector.rs`, the function dispatches on `predecessor_account_id`:

```rust
// connector.rs:81-90
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)          // ETH path: caller IS validated
} else {
    engine.receive_erc20_tokens(               // ERC-20 path: NO caller validation
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

For the base-token (ETH) path the connector account is checked. For the ERC-20 path there is **no check** that the predecessor actually performed a real NEP-141 `ft_transfer_call`. The engine simply trusts `predecessor_account_id` as the token identity and `args.amount` as the amount to mint.

`receive_erc20_tokens` in `engine/src/engine.rs` then:
1. Resolves the ERC-20 contract address via `get_erc20_from_nep141(&self.io, token)` (line 824)
2. Calls `mint(recipient, amount)` on that ERC-20 contract **as the engine admin** (lines 826–837)

The ERC-20 token (`EvmErc20V2.sol`) restricts `mint` to `onlyAdmin`, where admin is the Aurora Engine contract address — so the engine's internal call succeeds unconditionally once the NEP-141 mapping exists.

`deploy_erc20_token` (connector.rs line 112) has **no access control** — only `require_running` is checked — so any NEAR account can register itself as a NEP-141 token and obtain a corresponding ERC-20 contract on Aurora.

---

### Impact Explanation

An attacker mints ERC-20 tokens on Aurora that have no NEP-141 backing on NEAR. These tokens are indistinguishable from legitimately bridged tokens. The attacker can sell them on Aurora DEXes. When victims later call `withdrawToNear` on the ERC-20 contract, they burn their ERC-20 tokens expecting to receive NEP-141 tokens on NEAR. Because the attacker never deposited any NEP-141 tokens, the bridge is insolvent for those victims — their funds are permanently lost.

**Impact class:** Critical — direct theft of user funds / insolvency of the ERC-20 bridge accounting.

---

### Likelihood Explanation

High. The attack requires only two permissionless NEAR transactions:
1. `deploy_erc20_token` — no access control, callable by anyone.
2. `ft_on_transfer` — no caller validation, callable by anyone.

No admin compromise, leaked keys, governance capture, or oracle manipulation is required.

---

### Recommendation

Add a caller whitelist check in the ERC-20 branch of `ft_on_transfer`. Only accounts that are already registered as NEP-141 tokens in the engine's mapping should be allowed to trigger minting. Alternatively, mirror the base-token pattern and maintain an explicit allowlist of trusted NEP-141 token contracts. A minimal fix:

```rust
// In receive_erc20_tokens, before minting:
let erc20_token = get_erc20_from_nep141(&self.io, token)
    .map_err(|_| errors::ERR_NOT_ALLOWED)?; // revert if token not registered
// Additionally, consider requiring the caller to be on an allowlist
// rather than any account that has ever been registered.
```

---

### Proof of Concept

1. Attacker creates NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token(DeployErc20TokenArgs::Legacy("attacker.near"))` on the Aurora Engine. No access control — succeeds. An ERC-20 token is deployed on Aurora and the mapping `attacker.near → ERC20_addr` is stored.
3. Attacker calls `ft_on_transfer` directly on the Aurora Engine with:
   - `predecessor_account_id = attacker.near` (set automatically by NEAR runtime)
   - `args = { sender_id: "attacker.near", amount: 1_000_000_000, msg: "<victim_eoa_hex>" }`
4. `ft_on_transfer` (connector.rs:84) routes to `receive_erc20_tokens("attacker.near", args, ...)`.
5. `receive_erc20_tokens` (engine.rs:824) resolves `ERC20_addr` from the mapping, then calls `mint(victim_eoa, 1_000_000_000)` as the engine admin (engine.rs:826–837). **No NEP-141 tokens were ever transferred.**
6. Attacker (or victim_eoa) now holds 1 000 000 000 unbacked ERC-20 tokens.
7. Attacker sells them on an Aurora DEX. Buyers who later call `withdrawToNear` burn their ERC-20 tokens but receive nothing on NEAR — the NEP-141 reserve is empty.
