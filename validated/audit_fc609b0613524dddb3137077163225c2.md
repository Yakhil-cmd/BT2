### Title
`MsgTurnBridge` Handler Is a No-Op Stub with No Permission Check, Permanently Bypassing Admin Bridge-Disable Authority - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` gRPC handler, which is the sole mechanism for an authorized admin to disable the Gravity bridge in an emergency, is implemented as a stub that returns `(nil, nil)` unconditionally. It performs no permission check and changes no state. Any unprivileged caller can invoke it without error, and — critically — even a legitimately authorized admin calling it cannot actually disable the bridge. The bridge remains permanently enabled regardless of admin intent.

### Finding Description

The `MsgTurnBridge` message is proto-defined as the privileged operation to enable or disable the Gravity bridge: [1](#0-0) 

ADR-009 explicitly designates `MsgTurnBridge` as one of only two restricted messages requiring the `CanTurnBridge` permission bit: [2](#0-1) 

The permission system defines `CanTurnBridge` as a distinct bit flag: [3](#0-2) 

However, the actual handler implementation is: [4](#0-3) 

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

Two distinct flaws exist simultaneously:

1. **Missing permission check**: Unlike `UpdateTokenMapping` which correctly calls `k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`, `TurnBridge` performs no authorization check. Any unprivileged address can submit a `MsgTurnBridge` transaction and receive a success response.

2. **No state mutation**: The `enable` field of `MsgTurnBridge` is never read. No bridge-enabled flag is written to state. The bridge cannot be disabled regardless of who calls the function or what `enable` value is passed. [5](#0-4) 

### Impact Explanation

**High — Bypass of Cronos admin bridge authorization check.**

When a dangerous condition arises (e.g., a Gravity bridge exploit, oracle manipulation, or malicious relayer), the admin or governance-authorized address is expected to call `TurnBridge(enable=false)` to halt bridge flows. Because the handler is a no-op:

- The admin's disable decision is silently accepted (no error) but never applied.
- The bridge continues processing deposits and withdrawals as if no disable was issued.
- Funds continue to flow through the bridge during an active emergency.
- This is a permanent condition — there is no fallback path to disable the bridge at the Cosmos message layer.

This directly matches the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation

The entry path is a standard Cosmos SDK `MsgTurnBridge` transaction, reachable by any unprivileged address with no preconditions. The broken disable path is triggered whenever an authorized admin attempts an emergency bridge halt. The stub has been present since the permission system was introduced per ADR-009.

### Recommendation

Implement `TurnBridge` with:
1. A permission check mirroring `UpdateTokenMapping`:
   ```go
   if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
       return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
   }
   ```
2. A persistent bridge-enabled flag written to the KV store (e.g., `SetBridgeEnabled(ctx, msg.Enable)`).
3. All bridge-entry paths (EVM hooks, IBC callbacks, Gravity event handlers) must gate on this flag before processing transfers.

### Proof of Concept

1. Admin observes a Gravity bridge exploit and submits `MsgTurnBridge{sender: adminAddr, enable: false}`.
2. The transaction is accepted on-chain with a success response (`nil` error).
3. No state is written; the bridge-enabled flag does not exist.
4. An unprivileged attacker simultaneously submits `MsgTurnBridge{sender: attackerAddr, enable: true}` — also accepted with no error and no permission check.
5. Bridge deposits and withdrawals continue uninterrupted; the admin's emergency halt is silently discarded. [4](#0-3) [3](#0-2) [1](#0-0) [6](#0-5)

// UpdateTokenMapping implements the grpc method
func (k msgServer) UpdateTokenMapping(goCtx context.Context, msg *types.MsgUpdateTokenMapping) (*types.MsgUpdateTokenMappingResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)

	// check permission
	if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
		return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
	}

	// msg is already validated
	if err := k.RegisterOrUpdateTokenMapping(ctx, msg); err != nil {
		return nil, err
	}
	return &types.MsgUpdateTokenMappingResponse{}, nil
}
