### Title
Gateway Skips `__validate__` Signature Check for Any Invoke with Nonce=1 When Account Has Any Pending Mempool Transaction — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful path returns `true` (skip the `__validate__` entry-point call) for any Invoke transaction with `tx_nonce == 1` and `account_nonce == 0`, as long as `account_tx_in_pool_or_recent_block` returns `true` for the sender address. The check is supposed to confirm that a `deploy_account` transaction is pending for the account, but `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type in the pool for that address. An attacker who observes a victim's `deploy_account` in the mempool can immediately submit an Invoke with `nonce=1` from the victim's address carrying an arbitrary (invalid) signature, and the gateway will admit it to the mempool without ever calling `__validate__`.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```
skip_stateful_validations(executable_tx, account_nonce, mempool_client)
``` [1](#0-0) 

The function returns `true` (skip) when all three conditions hold:

1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false`, so `StatefulValidator::perform_validations` returns `Ok(())` without ever calling `tx.validate_tx(...)` (the `__validate__` entry point that checks the ECDSA signature): [3](#0-2) [4](#0-3) 

The guard that is supposed to confirm a `deploy_account` exists is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

This returns `true` for **any** transaction type in the pool for that address — not specifically a `deploy_account`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is not enforced. Any transaction (e.g., a `declare`, or even a `deploy_account` submitted by the attacker themselves) satisfies the check.

The analog to `StandardToken::transferFrom` without approval check is exact: just as `transferFrom` allowed any caller to move tokens from any owner without checking `allowance[owner][caller]`, `skip_stateful_validations` allows any submitter to inject an Invoke from any address without checking that the submitter controls the private key for that address.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can submit an Invoke transaction from a victim's address (with an arbitrary/invalid signature) and have it admitted to the mempool without any signature verification. The transaction will be rejected by the batcher when `__validate__` is eventually called during block building, but:

- The invalid transaction occupies a mempool slot for the victim's address at nonce 1.
- The batcher wastes execution resources attempting to validate it.
- If the attacker submits many such transactions targeting many accounts that have pending `deploy_account` transactions, this constitutes a targeted DoS on the deploy-account + invoke UX flow and on batcher throughput.
- The gateway's invariant — "every admitted transaction must have passed its account's `__validate__` entry point or be provably safe to skip" — is broken: the safety proof relies on `account_tx_in_pool_or_recent_block` implying a `deploy_account` is pending, which is not guaranteed.

### Likelihood Explanation

**Medium.** The attack window is open whenever a victim's `deploy_account` transaction is visible in the public mempool (which is the normal state during the deploy-account + invoke UX flow). The attacker needs only the victim's contract address (derivable from the `deploy_account` transaction itself) and the ability to submit a transaction to the gateway. No privileged access is required.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type, rather than returning `true` for any transaction type. Alternatively, the gateway should only skip `__validate__` when it can confirm the pending transaction for that address is of type `DeployAccount`.

### Proof of Concept

1. Victim submits `deploy_account` (nonce 0) for address `V` to the gateway. The transaction enters the mempool pool for address `V`.

2. Attacker observes `V` in the mempool (public). Attacker constructs an Invoke transaction:
   - `sender_address = V`
   - `nonce = 1`
   - `signature = [attacker_felt_1, attacker_felt_2]` (arbitrary, not a valid ECDSA signature for `V`)
   - Valid `resource_bounds`, `tip`, etc.

3. Attacker submits the Invoke to the gateway. Gateway calls `extract_state_nonce_and_run_validations`:
   - `account_nonce = get_nonce_from_state(V) = 0` (account not yet deployed)
   - `validate_state_preconditions`: nonce check passes (`0 ≤ 1 ≤ max_allowed_nonce_gap`)
   - `validate_by_mempool`: passes (no duplicate, no fee escalation issue)
   - `skip_stateful_validations`: `tx.nonce() == 1` ✓, `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block(V) == true` ✓ → returns `true`
   - `run_validate_entry_point`: `validate = false` → `__validate__` is **never called**

4. Gateway returns `Ok`, attacker's Invoke is forwarded to the mempool via `mempool_client.add_tx(...)`. [6](#0-5) 

5. The attacker's Invoke now occupies nonce-1 slot for address `V` in the mempool, admitted without any signature verification — a direct analog to `transferFrom` without checking `allowance`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
