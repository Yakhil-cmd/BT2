### Title
`setPoolAdminFeeDestination` Redirects All Accrued Admin Fees to New Destination Without Prior Collection — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees` on the pool. Every other fee-configuration mutator in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) collects accrued fees at the old rates/destination before applying the change. The missing collection step means all spread-surplus and notional fees that accrued under the old destination are silently redirected to the new destination on the next `collectPoolFees` call, which is permissionless.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the invariant: **collect first, then change**. [1](#0-0) 

Both functions call `IMetricOmmPoolCollectFees(pool).collectFees(…, poolAdminFeeDestination[pool])` before updating any configuration, ensuring that fees earned under the old destination are paid to that destination.

`setPoolAdminFeeDestination` breaks this pattern entirely: [2](#0-1) 

It overwrites `poolAdminFeeDestination[pool]` with no prior collection. After the write, the pool still holds:

1. **Spread surplus** — `balance * scaleMultiplier − binTotals.scaledToken* − notionalFeeToken*Scaled` — accumulated since the last `collectFees` call.
2. **Notional fee accumulators** — `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`. [3](#0-2) 

`collectPoolFees` is permissionless: [4](#0-3) 

Any caller (including the pool admin themselves) can immediately invoke it after the destination change, and the pool will transfer the entire accrued admin share to the **new** destination, not the old one.

---

### Impact Explanation

The old `adminFeeDestination` — which may be a DAO treasury, a revenue-sharing contract, or any address distinct from the pool admin — permanently loses all admin fees that accrued before the destination change. The new destination receives fees it did not earn. The magnitude equals the full admin share of:

- All spread fees accumulated since the last collection (proportional to trading volume × `adminSpreadFeeE6 / totalSpreadFeeE6`).
- All notional fees accumulated since the last collection (proportional to volume × `adminNotionalFeeE8 / totalNotionalFeeE8`).

This is a direct loss of owed protocol/admin fees, matching the "direct loss of protocol fees" impact class.

---

### Likelihood Explanation

The pool admin is semi-trusted and can call `setPoolAdminFeeDestination` at any time with no timelock. `collectPoolFees` is permissionless, so the admin does not even need to be the one to trigger the collection — a keeper or any third party can do it immediately after the destination change. The scenario requires only one normal admin action followed by one permissionless call, making it straightforward to execute either accidentally or deliberately.

---

### Recommendation

Apply the same collect-first pattern used by `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Settle accrued fees to the OLD destination before switching.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool accumulates 1 000 USDC in admin spread fees and 500 USDC in admin notional fees (old destination = `treasury`).
2. Pool admin calls `setPoolAdminFeeDestination(pool, attacker)` — no collection occurs.
3. Anyone calls `collectPoolFees(pool)`.
4. `collectFees` reads `poolAdminFeeDestination[pool]` → `attacker`, transfers 1 500 USDC to `attacker`.
5. `treasury` receives 0; all accrued admin fees are permanently lost to it.

The corrupted value is `poolAdminFeeDestination[pool]` being updated before the accrued fee balance is settled, directly mirroring the Moonwell pattern where `accountTokens[liquidator]` was updated before `updateAndDistributeSupplierRewardsForToken` was called. [2](#0-1) [4](#0-3) [5](#0-4)
