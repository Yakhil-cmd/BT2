### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Contribution Values Without Proof of Correctness - (File: src/confidential_key_derivation/protocol.rs)

### Summary
In `do_ckd_coordinator`, the coordinator blindly aggregates `(big_y, big_c)` contributions from all participants without any cryptographic proof that each contribution is correctly formed. A single malicious participant can send arbitrary group elements, causing the coordinator to produce an incorrect CKD output and deliver a corrupted derived key to the application.

### Finding Description
The analog to the external report's "EXECUTOR can delegatecall to a specially crafted library to write arbitrary storage" is: **a malicious participant can inject arbitrary cryptographic material into the CKD aggregation step, which is processed without any validation**, just as the EXECUTOR's delegatecall target is not whitelisted.

In `do_ckd_coordinator` at lines 50–55 of `src/confidential_key_derivation/protocol.rs`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` and unconditionally adds them to the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is no zero-knowledge proof or commitment binding each participant's `(big_y, big_c)` to:
1. Their actual key share `x_i` (i.e., `big_c = lambda_i * (x_i * H(app_id) + y_i * app_pk)`)
2. A consistent ephemeral scalar `y_i` (i.e., `big_y = lambda_i * y_i * G`)

The correct contribution is computed in `compute_signature_share` (lines 148–181), but nothing prevents a malicious participant from sending different values than what that function would produce. [2](#0-1) 

In `do_ckd_participant`, the participant sends its contribution privately to the coordinator with no attached proof:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

The coordinator has no mechanism to reject a malformed contribution.

### Impact Explanation
A malicious participant sends `(big_y = G_identity, big_c = G_identity)` (or any arbitrary group elements). The coordinator computes:

- `Y_out = sum_{honest i} lambda_i * y_i * G` (missing the malicious participant's term)
- `C_out = sum_{honest i} lambda_i * (x_i * H(app_id) + y_i * app_pk)` (missing the malicious participant's term)

The application unmasks: `C_out - app_sk * Y_out = (sum_{honest i} lambda_i * x_i) * H(app_id)`, which is **not** `msk * H(app_id)`. The application receives a silently incorrect derived key with no indication of failure.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
Any single participant in the CKD protocol can trigger this. No special privilege is required beyond being a listed participant. The attack is trivially executable: the malicious participant simply sends zero or random group elements instead of their correct contribution. There is no detection mechanism in the protocol.

### Recommendation
Require each participant to attach a non-interactive zero-knowledge proof of correct formation alongside their `(big_y, big_c)` contribution. Specifically, each participant should prove in zero-knowledge that:
- `big_y = lambda_i * y_i * G` for the same `y_i` used in `big_c`
- `big_c = lambda_i * (x_i * H(app_id) + y_i * app_pk)` where `x_i` is consistent with the participant's committed public key share from DKG

The coordinator must verify these proofs before incorporating any contribution into the aggregation. This is the direct analog to the external report's recommendation of "implementing a whitelist of permitted libraries" — here, the fix is to cryptographically constrain what values a participant is permitted to contribute.

### Proof of Concept
1. Honest participants P_1, …, P_{n-1} and malicious participant P_m run the CKD protocol.
2. P_m, instead of calling `compute_signature_share`, sends `(big_y = 0·G, big_c = 0·G)` (identity elements) to the coordinator via `chan.send_private`.
3. The coordinator at lines 50–55 adds these zero contributions, effectively dropping P_m's share from the aggregation.
4. The coordinator outputs `CKDOutput::new(norm_big_y, norm_big_c)` where both values are missing P_m's Lagrange-weighted term.
5. The application calls `ckd_output.unmask(app_sk)` and receives `(sum_{i≠m} lambda_i * x_i) * H(app_id)` — a value that is cryptographically indistinguishable from a valid key to the application, but is not the correct master-secret-derived key.
6. Any subsequent use of this derived key (e.g., decryption, authentication) silently fails or produces incorrect results, with no error surfaced by the protocol.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
