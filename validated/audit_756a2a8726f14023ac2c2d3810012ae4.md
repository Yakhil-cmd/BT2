### Title
Signature Bypass via `skip_stateful_validations` Accepts Invoke Transactions with Invalid Signatures — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator bypasses the `__validate__` entry point (account signature verification) for any invoke transaction with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that check returns `true` for **any** transaction type in the pool — not exclusively a `deploy_account` — an unprivileged attacker who pre-funds a target address can submit an invoke with an arbitrary/invalid signature and have it admitted to the mempool without signature verification.

---

### Finding Description

`skip_stateful_validations` is triggered when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. The transaction nonce is exactly `Nonce(Felt::ONE)`.
3. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed). [1](#0-0) 

When all three conditions hold, the function calls `account_tx_in_pool_or_recent_block(sender_address)` and, if it returns `true`, returns `true` itself — causing `run_validate_entry_point` to set `execution_flags.validate = false`, which skips the `__validate__` call entirely. [2](#0-1) 

The code comment asserts this check is "sufficient" because the address in the pool "either has a deploy_account transaction or transactions with future nonces that passed validations." However, `account_tx_in_pool_or_recent_block` is implemented as: [3](#0-2) 

which delegates to: [4](#0-3) 

Neither check filters by transaction type. Any transaction from the address — including an invoke that the attacker themselves submitted — satisfies the condition.

The `perform_pre_validation_stage` is still executed (nonce and fee bounds are checked), but `__validate__` — the only step that verifies the cryptographic signature — is skipped. [5](#0-4)

### Impact Explanation

An attacker who:

1. Pre-funds a target address `A` with enough STRK to satisfy `verify_can_pay_committed_bounds`, and
2. Observes a legitimate `deploy_account` for address `A` in the public mempool,

can submit an `Invoke` with nonce=1 from address `A` carrying an arbitrary/invalid signature. The gateway accepts it without running `__validate__`. The transaction enters the mempool.

When the batcher executes the block, the `deploy_account` runs first (creating the account), then the attacker's invoke runs with `validate=true` and reverts due to the invalid signature. In Starknet, even reverted transactions increment the account nonce. The legitimate user's invoke with nonce=1 is now permanently invalid (nonce already consumed), forcing the user to resubmit with nonce=2 — which may itself be blocked by the same attack.

This matches **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The mempool is public. Any observer can detect a pending `deploy_account` for a fresh address. Pre-funding an address with STRK before it is deployed is a standard Starknet operation. The attacker pays fees only for the reverted invoke (which is bounded and predictable). No privileged access is required.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender address is present in the pool. Alternatively, add a transaction-type filter inside `skip_stateful_validations` so that the skip is only triggered when the pool entry for the address is confirmed to be a `DeployAccount` variant. [6](#0-5) 

---

### Proof of Concept

```
1. Attacker computes the deterministic address A for a known class_hash + salt.
2. Attacker sends STRK to address A (pre-fund, so verify_can_pay_committed_bounds passes).
3. Legitimate user submits deploy_account for address A (valid signature, nonce=0).
   → deploy_account enters the mempool; account_tx_in_pool_or_recent_block(A) = true.
4. Attacker submits Invoke(sender=A, nonce=1, signature=[0xdead, 0xbeef]).
   Gateway stateful validation:
     - validate_nonce: 0 <= 1 <= max_gap  ✓
     - validate_by_mempool: nonce not too old  ✓
     - skip_stateful_validations:
         tx.nonce()==1 ∧ account_nonce==0 ∧ account_tx_in_pool_or_recent_block(A)==true
         → returns true
     - run_validate_entry_point(validate=false):
         perform_pre_validation_stage runs (nonce/fee checks pass)
         __validate__ is NOT called  ✓ (skipped)
   → Invoke admitted to mempool with invalid signature.
5. Batcher builds block:
     - deploy_account executes → account A created, nonce becomes 1.
     - Attacker's invoke executes with validate=true → __validate__ fails → REVERT.
       Nonce incremented to 2 (Starknet increments nonce even on revert).
6. Legitimate user's invoke (nonce=1) is now rejected: nonce too old.
   User must resubmit with nonce=2; attacker repeats step 4 indefinitely.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
