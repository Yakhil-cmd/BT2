## Title
Unbounded ingress-expiry loop in ICP Rosetta `construction_payloads` causes memory exhaustion â€” (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `/construction/payloads` endpoint accepts attacker-controlled `ingress_start` and `ingress_end` metadata fields as raw `u64` nanosecond timestamps with no validation of the window size. The loop at lines 99â€“107 iterates from `ingress_start` to `ingress_end` in fixed 120-second steps, allocating one `u64` per step into `ingress_expiries`, then `add_payloads` allocates two `SigningPayload` structs per entry per transaction. A single unauthenticated request with `ingress_start=0` and `ingress_end=u64::MAX` causes ~153 million loop iterations, exhausting process memory and denying service to all Rosetta users.

## Impact Explanation
With `ingress_start=0` and `ingress_end=18446744073709551615` (u64::MAX), the loop executes â‰ˆ153 million iterations. `ingress_expiries` alone consumes ~1.2 GB (153M Ã— 8 bytes). `add_payloads` then allocates ~306 million `SigningPayload` structs per transaction in the request, each containing heap-allocated `String` fields (`hex_bytes`, `account_identifier`), multiplying total allocation well beyond available memory. The process terminates via OOM or becomes unresponsive, denying service to all concurrent Rosetta users. This matches the allowed impact: **Application/platform-level DoS of the Rosetta API** â€” a listed in-scope financial integration component â€” **not based on raw volumetric DDoS**, triggered by a single tiny HTTP request. Severity: **High**.

## Likelihood Explanation
The `/construction/payloads` endpoint requires no authentication. The `ingress_start` and `ingress_end` fields are documented optional metadata. Any unprivileged attacker with network access to the Rosetta HTTP port can trigger this with a single POST request containing two JSON integers. The attack is repeatable, requires no prior state, and the request body is negligibly small.

## Recommendation
Add a server-side cap on the computed number of expiries before the loop executes, mirroring the ICRC1 Rosetta pattern:

```rust
// rs/rosetta-api/icp/src/request_handler/construction_payloads.rs
// After computing ingress_start and ingress_end, before the loop:
const MAX_INGRESS_EXPIRIES: u64 = 10;
let window_ns = ingress_end
    .saturating_sub(ingress_start)
    .as_nanos_since_unix_epoch();
let interval_ns = interval.as_nanos() as u64;
if window_ns / interval_ns > MAX_INGRESS_EXPIRIES {
    return Err(ApiError::invalid_request("ingress window too large"));
}
```

Alternatively, clamp `ingress_end` to `ingress_start + MAX_INGRESS_EXPIRIES * interval`. The constant should be derived from `MAX_INGRESS_TTL / interval` (which equals 2 for the current values) or set to a small fixed bound (e.g., 5â€“10).
