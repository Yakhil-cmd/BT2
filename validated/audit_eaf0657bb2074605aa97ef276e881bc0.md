### Title
Pool admin bypasses all fee caps via uncapped `setPoolBinAdditionalFees`, imposing up to 6.55% additional per-bin spread on traders - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap validation. Every other admin fee setter enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`, but the per-bin path has no such guard. A pool admin can set either value to `uint16.max` (65 535 E6 ≈ 6.55%) on any bin, immediately and without a timelock, causing every swap that crosses that bin to pay fees far above what the factory's cap system permits.

### Finding Description

`setPoolAdminFees` enforces the factory-owner-controlled cap before writing:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`setPoolBinAdditionalFees` has no equivalent guard:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool stores the values verbatim:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
  if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
  BinState storage s = _binStates[bin];
  s.addFeeBuyE6 = addFeeBuyE6;
  s.addFeeSellE6 = addFeeSellE6;
  emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

During every swap the per-bin fee is added directly on top of the oracle-derived base fee before the swap math executes:

```solidity
// MetricOmmPool.sol:910 (buy token0 specified-out direction)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),

// MetricOmmPool.sol:1088 (buy token1 specified-out direction)
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6),

// MetricOmmPool.sol:1177 (buy token1 specified-in direction)
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6),
```

`uint16.max = 65 535` in E6 units equals ≈ 6.55%. The factory owner can set `maxAdminSpreadFeeE6` to any value from 0 to `HARD_MAX_SPREAD_FEE_E6 = 200 000` (20%). If the owner sets the cap to, for example, 5 000 (0.5%), the admin can still impose 65 535 (6.55%) per-bin fees — 13× the cap — on any bin, in any block, with no timelock.

The per-bin additional fee accrues to LPs in the affected bin (it widens the effective spread the trader pays). The admin, who can also be an LP, can therefore extract value from traders above the protocol-enforced ceiling.

### Impact Explanation

Every swap that crosses a bin with elevated `addFeeBuyE6` / `addFeeSellE6` pays a higher effective price than the oracle-anchored quote plus the capped global spread fee. The excess is retained in the pool as LP profit. Traders receive less output (or pay more input) than the cap system guarantees, constituting a direct loss of user principal. The maximum per-bin overcharge is bounded by `uint16.max` (≈ 6.55%), but this can be applied to every bin simultaneously and can be toggled on and off around individual swaps.

### Likelihood Explanation

The trigger is a valid semi-trusted pool admin action — no privileged factory-owner cooperation is required. The admin can observe a large pending swap in the mempool, call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` in the same block (no timelock), let the swap execute at the inflated fee, then reset the fee. This is the direct analog of the external report's "late participant observes price action and acts with higher certainty": the admin observes the pending swap and acts with certainty of extracting excess fees.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` consistent with `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap so the factory owner can tune per-bin limits independently of the global spread cap.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 5_000` (0.5%) via `setFeeCaps`.
2. Pool admin calls `setPoolAdminFees(pool, 5_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; per-bin fees are now ≈ 6.55% on bin 0.
4. A trader submits a swap that crosses bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65_535, ONE_X64, 1e6)` — approximately 6.55% above the oracle mid-price, far exceeding the 0.5% cap the factory owner intended.
5. The trader receives ≈ 6.55% less output than the capped fee would allow; the excess remains in the pool as LP profit.
6. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 0, 0)` to reset, leaving no trace beyond the `BinAdditionalFeesUpdated` event. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)
