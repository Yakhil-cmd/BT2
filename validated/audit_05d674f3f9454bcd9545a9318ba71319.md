### Title
Swap functions in `MetricOmmSimpleRouter` are unnecessarily `payable`, allowing any accidentally sent ETH to be silently consumed by a subsequent WETH swap caller - (File: metric-periphery/contracts/MetricOmmSimpleRouter.sol)

### Summary
`exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` in `MetricOmmSimpleRouter` are all marked `payable` despite the interface explicitly scoping the router to "ERC-20 routes only. No native ETH, WETH wrap/unwrap." Any ETH sent with a non-WETH swap call is silently accepted and retained by the contract. A subsequent caller who swaps with WETH as `tokenIn` will have their settlement paid using the stranded ETH, effectively stealing it from the original sender.

### Finding Description
`PeripheryPayments.pay()` contains a WETH-specific branch that first consumes any native ETH balance held by the contract before falling back to `transferFrom` on the payer:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

The `receive()` function in `PeripheryPayments` reverts for any sender other than WETH, but this guard only applies to plain ETH transfers (no calldata). A `payable` function call with `msg.value > 0` bypasses `receive()` entirely — the ETH is deposited into the contract regardless. If a user calls `exactInputSingle{value: X}(...)` with a non-WETH `tokenIn`, the ETH is accepted and stranded. The next caller who executes any WETH swap will have their callback settlement funded from the stranded ETH rather than their own wallet.

The same vulnerability exists in `MetricOmmPoolLiquidityAdder`, whose `addLiquidityExactShares` and `addLiquidityWeighted` overloads are also `payable` and share the same `pay()` logic.

### Impact Explanation
Direct loss of user principal. A victim who accidentally sends ETH with a non-WETH swap loses that ETH permanently to the next WETH swap caller. The attacker pays less (or nothing) from their own wallet for the WETH leg of their swap. The loss is bounded only by the victim's `msg.value`.

### Likelihood Explanation
Low. The user must accidentally attach ETH to a non-WETH swap call. This is plausible when integrators or scripts reuse a payable call pattern, or when a user mistakenly sends ETH expecting WETH wrapping that the router explicitly does not support. The attacker path requires no special privilege — any WETH swap call immediately drains the stranded balance.

### Recommendation
Remove the `payable` modifier from `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` in `MetricOmmSimpleRouter`, and from `addLiquidityExactShares` and `addLiquidityWeighted` in `MetricOmmPoolLiquidityAdder`. The `multicall` entry point can remain `payable` for the legitimate ETH→WETH multicall pattern (send ETH → WETH swap → `refundETH`). Individual swap and liquidity functions that are not intended to accept native ETH should not be `payable`.

### Proof of Concept
1. Alice calls `exactInputSingle{value: 1 ether}(params)` where `params.tokenIn` is USDC (not WETH). The function is `payable`, so 1 ETH is deposited into the router. The swap settles via `transferFrom(Alice, pool, usdcAmount)`. Alice's 1 ETH is now stranded in the router.
2. Bob calls `exactInputSingle(params)` where `params.tokenIn` is WETH and `params.amountIn` is `1e18`. Inside `_justPayCallback`, `pay(WETH, Bob, pool, 1e18)` is called. `PeripheryPayments.pay()` reads `address(this).balance == 1 ether >= 1e18`, wraps Alice's ETH into WETH, and transfers it to the pool. Bob's own WETH wallet is untouched.
