### Title
Pool Admin Can Arbitrarily Reset the Price-Provider Timelock by Re-Calling `proposePoolPriceProvider` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` contains no guard against overwriting an already-pending proposal. The pool admin can call it repeatedly, resetting `pendingPriceProviderExecuteAfter` to `block.timestamp + timelock` on every call. This is the direct structural analog of the external report: just as the Optimism Manager could push the vote deadline forward indefinitely, the Metric pool admin can push the oracle-rotation execution window forward indefinitely, defeating the timelock's LP-protection guarantee.

---

### Finding Description

`proposePoolPriceProvider` in `MetricOmmPoolFactory.sol` unconditionally overwrites both `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]`:

```solidity
// MetricOmmPoolFactory.sol lines 487-490
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
``` [1](#0-0) 

There is no check such as `if (pendingPriceProvider[pool] != address(0)) revert AlreadyPending()`. The function only validates that the pool is not immutable and that the new provider's token pair matches: [2](#0-1) 

`executePoolPriceProviderUpdate` enforces the timelock strictly — it reverts if `block.timestamp < execAfter`: [3](#0-2) 

But because `proposePoolPriceProvider` can be called again at any time, the admin can reset `pendingPriceProviderExecuteAfter` to a new future timestamp before the current one elapses, making the timelock window perpetually slide forward.

There is also no `cancelPriceProviderProposal` function — once a proposal is live, LPs have no on-chain mechanism to force resolution; only the admin can act.

---

### Impact Explanation

The timelock is the sole on-chain protection that gives LPs advance notice of an oracle change so they can exit positions before a new (potentially adversarial) price provider is activated. By re-proposing before the window closes, the admin can:

1. **Prevent execution indefinitely**: Keep resetting the timer so `executePoolPriceProviderUpdate` always reverts with `PriceProviderTimelockNotElapsed`, blocking any oracle rotation.
2. **Switch the pending oracle mid-window**: Replace the announced provider with a different address, invalidating LP monitoring. LPs who were watching for provider A to be applied will instead receive provider B.
3. **Fatigue-then-execute**: Repeatedly re-propose to exhaust LP vigilance, then let the final proposal expire and execute a malicious oracle when LPs have stopped monitoring. A malicious oracle can return arbitrary bid/ask prices, enabling the admin to drain LP value through bad-price swaps.

The third scenario leads to direct LP principal loss via bad-price execution — a swap against a manipulated oracle quote extracts token value from the pool's LP reserves.

---

### Likelihood Explanation

- The pool admin is explicitly classified as "semi-trusted only inside caps and timelocks." The timelock is the boundary; re-proposal bypasses it.
- The call requires only `onlyPoolAdmin(pool)` — no additional privilege, no cost beyond gas.
- Pools with a non-zero, non-`type(uint256).max` `priceProviderTimelock` are the affected surface. Immutable-oracle pools (`priceProviderTimelock == type(uint256).max`) are unaffected.
- No existing guard, event-based circuit-breaker, or cancel function prevents repeated re-proposal. [4](#0-3) 

---

### Recommendation

Add a guard that prevents overwriting an active pending proposal:

```diff
function proposePoolPriceProvider(address pool, address newPriceProvider)
    external override nonReentrant onlyPoolAdmin(pool)
{
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
+   if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);
    ...
}
```

Alternatively, add a `cancelPriceProviderProposal` function (callable by the pool admin) so that a pending proposal can be explicitly withdrawn before a new one is submitted, giving LPs a clear on-chain signal that the previous proposal was abandoned.

### Proof of Concept

```solidity
// Pool created with priceProviderTimelock = 7 days
address pool = factory.createPool(params); // mutable oracle

MockOracle maliciousOracle = new MockOracle();
maliciousOracle.setTokens(token0, token1);
maliciousOracle.setBidAndAskPrice(/* manipulated prices */);

// Step 1: Admin proposes malicious oracle. LPs see it and start monitoring.
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(maliciousOracle));
// pendingPriceProviderExecuteAfter[pool] = block.timestamp + 7 days

// Step 2: Just before 7 days elapse, admin re-proposes the same oracle.
vm.warp(block.timestamp + 7 days - 1);
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(maliciousOracle));
// pendingPriceProviderExecuteAfter[pool] = block.timestamp + 7 days (RESET)

// Step 3: Repeat until LPs stop monitoring. Then let the timer expire.
vm.warp(block.timestamp + 7 days);
vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool);
// Pool now uses maliciousOracle. Swaps execute at manipulated prices, draining LP value.

// Verify: executePoolPriceProviderUpdate would have reverted at step 2 without re-proposal
// because block.timestamp < original executeAfter.

  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
