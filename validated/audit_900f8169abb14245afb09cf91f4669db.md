### Title
USDC-Blacklisted LP Cannot Withdraw Principal via `removeLiquidity` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`LiquidityLib.removeLiquidity` pushes pool tokens directly to the position `owner` address. If either pool token is USDC (or another blacklist-capable token) and the LP's address is later blacklisted, every call to `removeLiquidity` reverts, permanently locking the LP's deposited principal inside the pool with no recovery path.

---

### Finding Description

In `LiquidityLib.removeLiquidity`, after all share accounting is updated, the withdrawn token amounts are transferred directly to `owner`:

```solidity
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
```

The `owner` is enforced to be `msg.sender` by the pool's `removeLiquidity` wrapper:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
```

There is no alternative withdrawal path, no pull-payment pattern, and no admin rescue function for LP positions. If `owner` is USDC-blacklisted after depositing liquidity, `safeTransfer` reverts on every withdrawal attempt, and the LP's funds are permanently stranded.

The same push-payment pattern exists in `MetricOmmPool.collectFees`, which pushes tokens directly to `adminFeeDestination_`:

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);
}
```

A blacklisted `adminFeeDestination_` causes `collectFees` to revert entirely, which also blocks `setPoolAdminFees` and `setPoolProtocolFee` in the factory (both call `collectFees` as a prerequisite). However, the pool admin can mitigate this by first calling `setPoolAdminFeeDestination` to update the destination before attempting fee changes. The LP withdrawal path has no equivalent escape hatch.

---

### Impact Explanation

**LP principal loss (primary):** An LP whose address is USDC-blacklisted after depositing cannot call `removeLiquidity` — the `safeTransfer` to their own address reverts. Their entire deposited principal (token0 and/or token1) is permanently locked in the pool. No admin, factory owner, or third party can withdraw on their behalf. This is a direct, irrecoverable loss of user principal.

**Fee collection DoS (secondary):** If `adminFeeDestination` is blacklisted, `collectFees` reverts, blocking both admin and protocol fee collection and preventing fee-rate updates via `setPoolAdminFees` and `setPoolProtocolFee`. This is partially mitigable by the pool admin updating the destination address first.

---

### Likelihood Explanation

USDC blacklisting is a real, documented on-chain event (e.g., Tornado Cash-linked addresses, sanctioned entities). Metric pools are explicitly designed to support USDC/USDT as pool tokens. An LP who deposits into a USDC pool and is subsequently sanctioned — or who uses a smart-contract wallet that becomes blacklisted — has no recourse. The condition requires only: (1) pool token is USDC, (2) LP address is blacklisted after deposit. Both are plausible in production.

---

### Recommendation

Replace the push-payment pattern in `removeLiquidity` with a pull-payment (credit) model:

1. Instead of calling `safeTransfer(owner, ...)` directly, credit the owed amounts to an internal `claimable[owner][token]` mapping.
2. Add a separate `claimTokens(address token)` function that lets any address withdraw its own credited balance.

For `collectFees`, apply the same pattern: credit admin fees to a claimable balance rather than pushing to `adminFeeDestination_` inline.

---

### Proof of Concept

1. Pool is deployed with USDC as `token0`.
2. Alice (LP) calls `addLiquidity` — her USDC is deposited into the pool.
3. Alice's address is added to the USDC blacklist (e.g., via OFAC sanction).
4. Alice calls `removeLiquidity` → `LiquidityLib.removeLiquidity` executes share accounting, then reaches:
   ```solidity
   IERC20(ctx.token0).safeTransfer(owner, amount0Removed); // reverts: Alice is blacklisted
   ```
5. The call reverts. Alice's shares are still decremented in the accounting (the revert rolls back the entire transaction), so her position is intact — but every future attempt also reverts.
6. Alice's USDC principal is permanently locked. No other address can call `removeLiquidity` on her behalf because `msg.sender != owner` is enforced. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
