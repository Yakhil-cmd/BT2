### Title
Unvalidated Zero `reward_address` Causes Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

Both the Staking contract and the Pool contract accept a `reward_address` parameter in multiple public entry points without validating that it is non-zero. Because all accrued rewards are unconditionally transferred to the stored `reward_address`, setting it to the zero address permanently freezes all future unclaimed yield for that staker or pool member.

---

### Finding Description

Four public entry points accept a `reward_address` without a zero-address guard:

**`stake()` — `src/staking/staking.cairo` lines 288–317**
The only address checks performed are that `reward_address` is not a registered token address. No non-zero check exists.

**`change_reward_address()` — `src/staking/staking.cairo` lines 517–540**
Same pattern: only the token-address guard is present.

**`enter_delegation_pool()` — `src/pool/pool.cairo` lines 182–219**
Only checks `token_address != reward_address`. No non-zero check.

**`change_reward_address()` — `src/pool/pool.cairo` lines 505–526**
Only checks the token address. No non-zero check.

Once a zero `reward_address` is stored, every subsequent reward disbursement targets address `0`:

- Staking contract `send_rewards_to_staker` (`src/staking/staking.cairo` line 1625):
  ```
  token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
  ```
- Pool contract `claim_rewards` (`src/pool/pool.cairo` line 366):
  ```
  reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
  ```

Depending on the STRK ERC-20 implementation, the transfer either reverts (permanently blocking all future `claim_rewards` calls) or silently burns the tokens. Either outcome permanently destroys the unclaimed yield.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A staker or pool member who sets `reward_address` to zero (by mistake or typo) will have every future reward transfer either revert or go to the zero address. The principal stake is unaffected, but all accrued and future rewards are irrecoverably lost. There is no recovery path: `change_reward_address` can be called again to fix the address, but rewards already sent to zero are gone, and any rewards that accrued while the address was zero and whose `claim_rewards` call reverted remain frozen in the contract with no mechanism to redirect them.

---

### Likelihood Explanation

**Medium.** The entry points are callable by any staker or pool member without privilege. A simple typo (e.g., forgetting to pass the address, passing a default-initialized variable, or a front-end bug that submits `0x0`) is a realistic user error. The original confirmed report (M-20) was triggered by exactly this class of mistake. The protocol provides no safety net.

---

### Recommendation

Add a non-zero guard to every entry point that stores or updates `reward_address`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This should be added in:
- `stake()` in `src/staking/staking.cairo`
- `change_reward_address()` in `src/staking/staking.cairo`
- `enter_delegation_pool()` in `src/pool/pool.cairo`
- `change_reward_address()` in `src/pool/pool.cairo`

---

### Proof of Concept

1. A staker calls `stake(reward_address: 0, operational_address: X, amount: MIN_STAKE)`.
2. The call passes all existing checks (zero is not a registered token address).
3. The staker accrues rewards over several epochs.
4. The staker or anyone calls `claim_rewards(staker_address)`.
5. `send_rewards_to_staker` executes `checked_transfer(recipient: 0, amount: rewards)`.
6. The transfer either reverts (rewards frozen in contract forever) or succeeds (rewards burned at address 0).
7. The staker's principal is unaffected but all yield is permanently lost.

The same scenario applies to a pool member calling `enter_delegation_pool(reward_address: 0, amount: X)` or either party calling `change_reward_address(0)`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
