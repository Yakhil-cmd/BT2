### Title
Gateway Skips `__validate__` Signature Check for Invoke Transactions Based on Unauthenticated Mempool Presence — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (which performs signature verification) for any invoke transaction with `nonce=1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The check does not verify that the submitter of the invoke is the same entity that submitted the deploy-account transaction. An attacker who observes a victim's pending deploy-account in the mempool can submit an invoke with a forged signature for the victim's address, have it admitted by the gateway without signature verification, and occupy the victim's nonce-1 slot in the mempool.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–461) returns `true` — causing `run_validate_entry_point` to set `execution_flags.validate = false` — when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`
2. The account's on-chain nonce is `0` (not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The third condition is satisfied by the presence of **any** transaction for that address in the mempool pool or the mempool's committed-block state — not specifically a deploy-account transaction: [2](#0-1) 

When `skip_validate=true`, `run_validate_entry_point` sets `validate: !skip_validate = false`: [3](#0-2) 

This means `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`: [4](#0-3) 

And `validate_tx` itself short-circuits when `execution_flags.validate == false`: [5](#0-4) 

The gateway then admits the transaction to the mempool via `add_tx`: [6](#0-5) 

### Impact Explanation

An attacker who monitors the public mempool for a victim's pending deploy-account transaction can submit an invoke transaction for the victim's address with `nonce=1` and a completely forged signature. The gateway accepts it without any signature check. The forged invoke occupies the victim's nonce-1 slot in the mempool. The victim's legitimate invoke with `nonce=1` is then rejected as a duplicate nonce (or must pay a fee-escalation premium to replace it). The attacker can repeat this indefinitely at near-zero cost because the batcher will reject the forged invoke at execution time (since `new_for_sequencing` sets `validate=true`), meaning the attacker pays no fees. [7](#0-6) 

**Impact:** High — Mempool/gateway admission accepts invalid transactions (forged-signature invoke) before sequencing.

### Likelihood Explanation

The mempool is observable by any network participant. Any account that submits a deploy-account transaction is immediately vulnerable during the window between submission and block inclusion. The attack requires no privileged access, no special funds, and no on-chain state — only the ability to submit an RPC transaction.

**Likelihood:** Medium

### Recommendation

The `skip_stateful_validations` check must verify that the pending transaction for the account is specifically a **deploy-account** transaction, not just any transaction. The current `account_tx_in_pool_or_recent_block` check is too broad. One approach: query the mempool for the specific deploy-account transaction hash for the sender address and verify it is present and is of type `DeployAccount`. Alternatively, maintain a separate set of pending deploy-account addresses in the mempool and only consult that set in `skip_stateful_validations`. [8](#0-7) 

### Proof of Concept

1. Alice submits `deploy_account` for address `X` (class_hash `C`, salt `S`). The deploy-account enters the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. On-chain nonce for `X` is still `0` (deploy-account not yet executed).
3. Attacker submits `invoke` for `sender_address=X`, `nonce=1`, `signature=[0x1, 0x2]` (forged).
4. Gateway stateless validator passes (signature length ≤ max, resource bounds valid).
5. `extract_state_nonce_and_run_validations` fetches on-chain nonce `0` for `X`.
6. `run_pre_validation_checks` → `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
7. `run_validate_entry_point` is called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` is never called → returns `Ok(())`.
8. Attacker's forged invoke is added to the mempool at nonce-1 for address `X`.
9. Alice's legitimate invoke with `nonce=1` is rejected as `DuplicateNonce` unless she pays a fee-escalation premium.
10. Attacker can repeat step 3 with a higher fee each time Alice replaces, forcing Alice into an unbounded fee auction. [9](#0-8) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
