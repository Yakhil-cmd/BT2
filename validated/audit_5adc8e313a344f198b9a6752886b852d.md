Audit Report

## Title
ckBTC Minter Finalizes `AmountTooLow` Withdrawal Requests Without Reimbursement, Causing Permanent ckBTC Loss - (File: rs/bitcoin/ckbtc/minter/src/lib.rs)

## Summary
When a user calls `retrieve_btc`, their ckBTC is burned atomically before the request is queued. If Bitcoin fees rise between request acceptance and batch processing, the minter finalizes the request as `FinalizedStatus::AmountTooLow` with no reimbursement, permanently destroying the user's ckBTC. The `InvalidTransaction` error arm in the same function does trigger reimbursement, confirming the omission in the `AmountTooLow` arm is inconsistent and harmful.

## Finding Description

**Burn before queue** (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, lines 209–232): `burn_ckbtcs` is called and completes before `accept_retrieve_btc_request` is called. The burn is irreversible. The `RetrieveBtcRequest` is constructed with `reimbursement_account: Some(Account { owner: caller, subaccount: None })`, confirming the infrastructure for reimbursement is present on the request object. [1](#0-0) 

**Min-amount guard is insufficient** (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, lines 169–171): The guard checks `args.amount < fee_based_retrieve_btc_min_amount` only at call time. The `fee_based_retrieve_btc_min_amount` is updated dynamically; already-queued requests are never re-validated against the new minimum. [2](#0-1) 

**`AmountTooLow` arm discards without reimbursement** (`rs/bitcoin/ckbtc/minter/src/lib.rs`, lines 412–434): When `build_tx` returns `BuildTxError::AmountTooLow`, every request in the batch is passed to `state::audit::remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. No call to `reimburse_canceled_requests` or `audit::reimburse_withdrawal` is made. [3](#0-2) 

**`remove_retrieve_btc_request` schedules no reimbursement** (`rs/bitcoin/ckbtc/minter/src/state/audit.rs`, lines 67–84): The function only records a `RemovedRetrieveBtcRequest` event and pushes to `finalized_requests`. It does not call `schedule_withdrawal_reimbursement` or any equivalent. [4](#0-3) 

**Contrast: `InvalidTransaction` arm does reimburse** (`rs/bitcoin/ckbtc/minter/src/lib.rs`, lines 400–411): The immediately preceding match arm calls `reimburse_canceled_requests`, which schedules a mint back to the user's `reimbursement_account`. The `AmountTooLow` arm has no equivalent call, making the omission a clear inconsistency. [5](#0-4) 

**`WithdrawalReimbursementReason` has no `AmountTooLow` variant** (`rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`, lines 39–43): The enum only contains `InvalidTransaction`, confirming no reimbursement path exists for fee-spike-induced failures. [6](#0-5) 

The `DustOutput` arm (`lib.rs`, lines 436–467) has the same defect for the individual request it finalizes. [7](#0-6) 

## Impact Explanation
A user who calls `retrieve_btc` with an amount that passes the minimum check at call time permanently loses their ckBTC if Bitcoin fees spike before the minter's timer processes the batch. The ckBTC ledger burn is irreversible; no BTC is sent; no reimbursement mint is ever scheduled. The user's `retrieve_btc_status_v2` returns `AmountTooLow` with no recovery path. This constitutes a concrete, permanent loss of ck-token assets — matching the allowed High impact: **"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."**

## Likelihood Explanation
Bitcoin fees are historically volatile (e.g., inscription/ordinal activity causes sharp spikes). The minter processes requests in batches on a timer, creating a guaranteed window between burn and transaction construction. The `fee_based_retrieve_btc_min_amount` is updated dynamically, so a request accepted just above the minimum during a low-fee period is at risk if fees rise before the batch fires. This is an unprivileged ingress path (`retrieve_btc` is a public update call) requiring no special access. The scenario is realistic and repeatable.

## Recommendation
Add a reimbursement path for `BuildTxError::AmountTooLow` mirroring the `InvalidTransaction` handler:
1. Add a `FeeTooHigh` (or equivalent) variant to `WithdrawalReimbursementReason` in `rs/bitcoin/ckbtc/minter/src/reimbursement/mod.rs`.
2. In the `AmountTooLow` arm of `submit_pending_requests` (`lib.rs`), replace the bare `remove_retrieve_btc_request` loop with a call to `reimburse_canceled_requests` using the new reason variant, deducting a small penalty fee to cover the failed attempt.
3. Apply the same fix to the `DustOutput` arm for the individual request that is finalized as `AmountTooLow`.

## Proof of Concept
1. Bitcoin fees are low; `fee_based_retrieve_btc_min_amount` = 5,000 sat.
2. User calls `retrieve_btc { amount: 5,500, address: "bc1q..." }`. Amount passes the guard. `burn_ckbtcs` executes; ckBTC is burned from the user's account. `RetrieveBtcOk { block_index: N }` is returned.
3. Before the minter timer fires, Bitcoin fees spike. The minter updates `fee_based_retrieve_btc_min_amount` to 10,000 sat.
4. The minter timer fires, calls `submit_pending_requests`, builds a batch containing the 5,500-sat request, then calls `build_tx`. `build_tx` returns `BuildTxError::AmountTooLow` because 5,500 < current fee.
5. The `AmountTooLow` arm executes: `remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)`. No reimbursement is scheduled.
6. User queries `retrieve_btc_status_v2(N)` → `AmountTooLow`. ckBTC is gone; no BTC was sent; no reimbursement mint will ever occur.

A deterministic integration test can reproduce this by: (a) setting a low fee estimate, (b) calling `retrieve_btc` just above the minimum, (c) raising the fee estimate above the request amount, (d) triggering the minter timer, and (e) asserting that `retrieve_btc_status_v2` returns `AmountTooLow` and that no `ScheduleWithdrawalReimbursement` event was recorded in the event log.

```rust
pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
```
