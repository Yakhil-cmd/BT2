### Title
Cross-Network Signature Replay in `validateCOAOwnershipProof` Due to Chain-ID-Agnostic Domain Separation Tag - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account ownership of a COA using the hardcoded domain separation tag `"FLOW-V0.0-user"`, which contains no chain ID or network identifier. Because the `signedData` payload is arbitrary caller-supplied bytes with no enforced chain-specific content, a valid COA ownership proof signature produced on one Flow network (e.g., testnet) is cryptographically indistinguishable from one produced on any other Flow network (e.g., mainnet) where the same Cadence account address and key exist. Any EVM contract on mainnet that relies on the `verifyCOAOwnershipProof` precompile or the COA's ERC-1271 `isValidSignature` for authorization can be fooled by a signature originally produced on a different network.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies signatures using a fixed domain separation tag:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"   // ← no chain ID
)
``` [1](#0-0) 

The tag `"FLOW-V0.0-user"` is a compile-time constant that is identical across all Flow networks (mainnet `flow-mainnet`, testnet `flow-testnet`, previewnet `flow-previewnet`, etc.). [2](#0-1) 

The `signedData` parameter is opaque bytes whose content is entirely controlled by the caller — the function imposes no requirement that it encode the chain ID, the EVM address, or any other network-specific value. The code comment itself acknowledges a related replay concern but only for cross-address replay, not cross-network replay: [3](#0-2) 

This precompile is exposed to unprivileged EVM callers via the Cadence Arch contract (`verifyCOAOwnershipProof(address,bytes32,bytes)`): [4](#0-3) 

The COA's ERC-1271 `isValidSignature` implementation calls this precompile directly, making it the primary on-chain consumer: [5](#0-4) 

The Go-layer validator (`coaOwnershipProofValidator`) passes the proof through to `validateCOAOwnershipProof` without injecting any chain-specific context: [6](#0-5) 

---

### Impact Explanation

An attacker who obtains a valid COA ownership proof signature from a user on testnet can replay it on mainnet, provided:

1. The same Cadence account address exists on both networks (Flow addresses are deterministically generated from account creation order, so this is common for early accounts and for accounts created via the same tooling).
2. The same public key is registered on both accounts.
3. A COA is published at the same capability path on both accounts.
4. The `signedData` (the `bytes32 _hash` passed to `isValidSignature`) is meaningful on mainnet — which is true whenever the same EVM contract is deployed at the same address on both networks (a standard practice for deterministic deployments).

Under these conditions, a user who signs a low-stakes testnet operation (e.g., authorizing a testnet multisig action, signing a testnet permit) produces a signature that is cryptographically valid on mainnet for the same `signedData`. Any mainnet EVM contract using `isValidSignature` for authorization — multisig wallets, ERC-4337 account abstraction, off-chain order protocols, permit-style approvals — would accept the replayed signature, potentially authorizing unauthorized asset transfers or contract interactions on behalf of the victim's COA.

---

### Likelihood Explanation

**Medium.** The preconditions are realistic:

- Flow address generation is deterministic; many users and protocols have matching addresses across mainnet and testnet.
- Deterministic EVM contract deployment (`CREATE2`) is standard practice, producing identical contract addresses across networks.
- The `signedData` for ERC-1271 is typically a structured hash (e.g., EIP-712) that is identical across networks when the same contract and action are involved.
- The attack requires no privileged access, no staked node compromise, and no admin key — only the ability to observe a testnet signature and submit a mainnet EVM transaction.

The attacker-controlled entry path is a standard unprivileged EVM transaction calling any contract that invokes `isValidSignature` on a COA.

---

### Recommendation

**Short term:** Enforce that `signedData` encodes the Flow chain ID. The simplest approach is to require callers to include the chain ID as a prefix in `signedData`, or to have `validateCOAOwnershipProof` prepend the chain ID to `signedData` before verification:

```cadence
// Prepend chain ID bytes to signedData before verification
let chainBytes = /* current chain ID as bytes */
let boundSignedData = chainBytes.concat(signedData)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: boundSignedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Alternatively, encode the chain ID into the domain separation tag itself (e.g., `"FLOW-V0.0-user-mainnet"`), analogous to EIP-712's `DOMAIN_SEPARATOR`.

**Long term:** Define a structured signed-data format for COA ownership proofs (analogous to EIP-712 typed structured data) that mandates inclusion of chain ID, contract address, and nonce, preventing replay across networks, contracts, and time.

---

### Proof of Concept

1. Alice has a Cadence account at address `0xABCD` on both testnet and mainnet, with the same ECDSA key at index 0, and a COA published at `/public/coa` on both.
2. A DeFi protocol `DeFiVault` is deployed at address `0xDEAD` on both testnet and mainnet (via `CREATE2`). It uses `isValidSignature` to authorize withdrawals: it calls `verifyCOAOwnershipProof(0xABCD, keccak256(withdrawalData), proof)`.
3. Alice signs a testnet withdrawal of 1 FLOW: `sig = sign("FLOW-V0.0-user" || keccak256(withdrawalData))`.
4. An attacker submits a mainnet transaction calling `DeFiVault.withdraw(withdrawalData, proof)` where `proof` encodes Alice's testnet signature.
5. `isValidSignature` calls `verifyCOAOwnershipProof(0xABCD, keccak256(withdrawalData), proof)` on mainnet.
6. `validateCOAOwnershipProof` verifies `sig` against Alice's mainnet key using tag `"FLOW-V0.0-user"` — the verification passes because the tag and `signedData` are identical on both networks.
7. `DeFiVault` authorizes the mainnet withdrawal, draining Alice's mainnet assets without her consent.
