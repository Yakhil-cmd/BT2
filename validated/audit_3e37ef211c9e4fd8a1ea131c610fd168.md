### Title
Hardcoded `CUSTODIAN_ADDRESS` Security Guard Cannot Be Updated Without Redeployment, Re-Enabling Bridge Fund-Theft Exploit on Custodian Migration - (File: engine-sdk/src/near_runtime.rs)

### Summary
The mainnet `eth_custodian` Ethereum address (`0x6BFaD42cFC4EfC96f529D786D643Ff4A8B89FA52`) is baked in as a compile-time constant `CUSTODIAN_ADDRESS` in `engine-sdk/src/near_runtime.rs`. It is the sole runtime guard inside `return_output` that blocks the GHSA-3p69-m8gg-fwmf exploit, where a malicious EVM contract crafts a 56-byte return value mimicking a `WithdrawResult` to fake a bridge withdrawal. Because the address is fixed at compile time, it cannot be updated if the eth_custodian contract is ever migrated to a new Ethereum address. Any such migration silently invalidates the guard without any on-chain signal, re-enabling the original exploit and allowing an unprivileged EVM attacker to steal bridged ETH.

### Finding Description
In `engine-sdk/src/near_runtime.rs`, the constant is defined:

```rust
/// The mainnet `eth_custodian` address 0x6BFaD42cFC4EfC96f529D786D643Ff4A8B89FA52. We use the
/// address for the mainnet only, since for the testnet it's not so critical.
const CUSTODIAN_ADDRESS: &[u8] = &[
    107, 250, 212, 44, 252, 78, 252, 150, 245, 41, 215, 134, 214, 67, 255, 74, 139, 137, 250, 82,
];
``` [1](#0-0) 

It is consumed exclusively in the `return_output` implementation of the `IO` trait for the production `Runtime`:

```rust
fn return_output(&mut self, value: &[u8]) {
    unsafe {
        assert!(
            !(value.len() >= 56 && &value[36..56] == CUSTODIAN_ADDRESS),
            "ERR_ILLEGAL_RETURN"
        );
        exports::value_return(value.len() as u64, value.as_ptr() as u64);
    }
}
``` [2](#0-1) 

The guard matches the binary layout of `WithdrawResult`:

```rust
pub struct WithdrawResult {
    pub amount: NEP141Wei,           // bytes  0–15  (u128, 16 bytes)
    pub recipient_id: Address,       // bytes 16–35  (20 bytes)
    pub eth_custodian_address: Address, // bytes 36–55 (20 bytes)
}
``` [3](#0-2) 

The check `&value[36..56] == CUSTODIAN_ADDRESS` is therefore specifically testing whether the EVM output is a valid `WithdrawResult` carrying the real custodian address — the exact payload a malicious EVM contract would forge to fake a bridge withdrawal. The code comment itself acknowledges the limitation: *"We use the address for the mainnet only, since for the testnet it's not so critical"* — confirming that the testnet custodian address is already a different value and the guard is already ineffective there.

The existing regression test `engine-tests/src/tests/ghsa_3p69_m8gg_fwmf.rs` confirms the exploit is real and the guard is the fix: [4](#0-3) 

Because `CUSTODIAN_ADDRESS` is a `const` embedded in the compiled WASM binary, it cannot be changed without a full engine contract redeployment. There is no admin function, storage key, or state variable that controls it at runtime.

### Impact Explanation
If the Aurora bridge team migrates the eth_custodian contract to a new Ethereum address (a normal operational event for bridge upgrades), the hardcoded guard immediately becomes stale. An unprivileged EVM user can then deploy a malicious contract on Aurora that returns a 56-byte payload with the new custodian address at bytes [36..56]. The guard passes, the NEAR runtime receives a payload indistinguishable from a legitimate `WithdrawResult`, and the attacker can initiate fraudulent bridge withdrawals, stealing bridged ETH from the connector. **Impact: Critical — direct theft of bridged user funds.**

### Likelihood Explanation
The eth_custodian address is stable today, but bridge migrations do occur in practice. The code comment explicitly acknowledges that the testnet custodian address is already different and unprotected, demonstrating that the guard is already non-functional in one deployed environment. Any future bridge upgrade that changes the custodian address would silently invalidate the guard with no on-chain signal and no mechanism for the engine to self-correct without a full redeployment. **Likelihood: Medium** — the address is stable on mainnet today, but the guard provides no mechanism to stay synchronized with the actual deployed custodian, and the testnet case proves the gap is already real.

### Recommendation
Store the eth_custodian address in the engine's on-chain state — it is already present as `eth_custodian_address` in `InitCallArgs` / `SetContractDataCallArgs` — and read it dynamically inside `return_output` via the `IO` storage layer rather than using a compile-time constant. This allows the guard to be updated through the existing `set_contract_data` admin path without requiring a full contract redeployment, eliminating the staleness risk. [5](#0-4) 

### Proof of Concept
1. The Aurora bridge team migrates the eth_custodian contract to a new Ethereum address `NEW_ADDR`. The Aurora Engine WASM is **not** redeployed; `CUSTODIAN_ADDRESS` still holds the old address.
2. An attacker deploys a malicious EVM contract on Aurora. When called, it returns a 56-byte payload: `[0u8; 16] ++ victim_eth_address_bytes ++ NEW_ADDR_bytes`.
3. The attacker calls the Aurora `view` entrypoint targeting the malicious contract.
4. `return_output` evaluates `&value[36..56] == CUSTODIAN_ADDRESS` — this compares against the **old** address. The payload carries `NEW_ADDR`, so the assertion passes silently.
5. The 56-byte output is returned to the NEAR runtime. It is structurally identical to a valid `WithdrawResult` for `NEW_ADDR`, enabling the attacker to trigger a fraudulent bridge withdrawal of bridged ETH to their Ethereum address.
