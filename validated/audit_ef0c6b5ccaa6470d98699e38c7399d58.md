### Title
Gateway Skips `__validate__` Signature Verification for Nonce-1 Invoke Transactions When Deploy-Account Is Pending — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX feature in the gateway's stateful validator unconditionally skips the blockifier `__validate__` entry-point call — the only place where the account's signature is verified — for any Invoke transaction with nonce=1 submitted while the account's on-chain nonce is still 0 and any transaction for that address exists in the mempool. An unprivileged attacker can exploit this to inject Invoke transactions carrying an arbitrary (invalid) signature into the mempool, bypassing the admission invariant that every accepted transaction must pass its account's `__validate__` check.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` and returns `true` (skip) when three conditions hold simultaneously:

1. The transaction is an `Invoke` with `nonce == 1`
2. The account's on-chain nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The boolean returned by `account_tx_in_pool_or_recent_block` is satisfied whenever the address appears in either the mempool's `tx_pool` (any pending transaction) or its `committed`/`staged` state maps: [2](#0-1) [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: !skip_validate` → `validate: false`: [4](#0-3) 

Inside the blockifier's `StatefulValidator::perform_validations`, when `execution_flags.validate == false` for an Invoke transaction, execution returns immediately after `perform_pre_validation_stage` — the `__validate__` entry-point (which runs the account's signature-verification logic) is never called: [5](#0-4) 

`perform_pre_validation_stage` only checks nonce ordering and fee bounds; it does not verify the cryptographic signature: [6](#0-5) 

The mempool's own `validate_tx` path also performs no signature check — it only validates nonce ordering and fee escalation: [7](#0-6) 

---

### Impact Explanation

An attacker can submit an Invoke transaction with nonce=1 and a completely invalid (e.g., empty) signature for any address that currently has a pending deploy-account transaction in the mempool. The gateway accepts the transaction without running `__validate__`, so the invalid signature is never detected at admission time. The transaction enters the mempool and is handed to the batcher. During block execution the batcher will call `__validate__` and the transaction will revert, but the admission invariant — *every transaction accepted by the gateway must pass its account's validation logic* — is broken. This enables:

- **Mempool pollution / resource exhaustion**: An attacker who controls (or observes) any address with a pending deploy-account can flood the mempool with signature-invalid Invoke transactions that consume pool capacity and batcher execution budget before being rejected.
- **Admission of unauthorized transactions**: The gateway's stateful validation path, which is the authoritative admission gate, certifies transactions it has not actually validated.

Impact category: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The trigger is straightforward and requires no privileged access:

1. The attacker submits their own deploy-account transaction (or observes any pending deploy-account in the public mempool).
2. They immediately submit an Invoke with nonce=1 and an empty/garbage signature from the same address.
3. All three skip conditions are satisfied automatically.

No special knowledge of the account's class or constructor is required. The attacker can use their own freshly-generated address. The window is open from the moment the deploy-account transaction enters the mempool until the account's on-chain nonce advances past 0.

---

### Recommendation

The skip should bypass only the *execution cost* of running `__validate__`, not the signature check itself. Two options:

1. **Perform a lightweight signature pre-check** before deciding to skip: verify that the Invoke transaction's signature is structurally valid (e.g., correct length, non-zero) even when the full entry-point call is skipped.
2. **Require the Invoke transaction to carry a proof-of-key-ownership** that ties it to the same key used in the deploy-account transaction (e.g., sign the Invoke hash with the same key and verify it statically at the gateway before skipping the entry-point call).

### Proof of Concept

```
1. Generate a fresh key-pair (sk, pk) and compute the counterfactual address A
   for a deploy-account transaction using class_hash C and salt S.

2. Submit deploy_account(A, C, S, valid_signature(sk)) → gateway accepts it,
   tx enters mempool, account_tx_in_pool_or_recent_block(A) == true.

3. Submit invoke(sender=A, nonce=1, calldata=<arbitrary>, signature=[]) →
   gateway stateful validator:
     - get_nonce_from_state(A) == 0  ✓
     - validate_nonce: 0 <= 1 <= max_gap  ✓
     - validate_by_mempool: nonce ordering OK  ✓
     - skip_stateful_validations: nonce==1 && account_nonce==0 &&
       account_tx_in_pool_or_recent_block(A)==true  → returns true
     - run_validate_entry_point called with validate=false
       → __validate__ never executed, empty signature never checked
     - gateway returns success, tx admitted to mempool

4. Batcher picks up both transactions, executes deploy_account (succeeds),
   then executes invoke with validate=true → __validate__ fails → tx reverted.
   Batcher calls commit_block with invoke hash in rejected_tx_hashes.

5. Repeat step 3 with different tx_hashes to exhaust mempool capacity.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
