### Title
Stale `eligible_operational_addresses` Entry Allows Staker to Re-use Operational Address Without Current Consent — (File: `src/staking/staking.cairo`)

### Summary
When a staker changes their operational address via `change_operational_address`, the `eligible_operational_addresses` mapping entry for the **old** operational address is never cleared. This stale consent entry allows the staker to silently re-associate the old operational address with their staking at any future time, without the old operational address re-declaring consent.

### Finding Description
The `Staking` contract maintains two mappings for operational addresses:

- `operational_address_to_staker_address`: active 1-to-1 mapping of operational address → staker [1](#0-0) 
- `eligible_operational_addresses`: consent mapping, written when an operational address calls `declare_operational_address` [2](#0-1) 

When `change_operational_address(new_op_addr)` is called, the function:
1. Clears `operational_address_to_staker_address[old_address]` [3](#0-2) 
2. Updates `staker_info.operational_address` and sets `operational_address_to_staker_address[new_op_addr]` [4](#0-3) 
3. **Never clears `eligible_operational_addresses[old_address]`** — the stale consent entry persists indefinitely [5](#0-4) 

The consent check in `change_operational_address` only verifies `eligible_operational_addresses[operational_address] == staker_address`: [6](#0-5) 

Because the stale entry is never cleared, the staker can pass this check again for the old address without the old operational address calling `declare_operational_address` again.

Meanwhile, `declare_operational_address` blocks overwriting the stale entry if the address is currently in use: [7](#0-6) 

This means once the staker re-uses the old operational address (O1), O1 is marked in-use and **cannot** call `declare_operational_address` to overwrite the stale entry — O1 is trapped.

### Impact Explanation
A malicious staker can permanently grief a former operational address operator:

1. Force O1 back into service as their operational address without O1's current consent.
2. Once re-associated, O1 is blocked from calling `declare_operational_address` (the `OPERATIONAL_IN_USE` assert fires).
3. O1 cannot associate itself with any other staker until the attacker voluntarily releases it.
4. The attacker can repeat this cycle indefinitely, permanently denying O1 the ability to freely choose which staker to serve.

This is **griefing with no profit motive but concrete damage to the operational address operator** — matching the Medium impact tier.

### Likelihood Explanation
The preconditions are realistic in normal protocol usage:
- Node operators routinely rotate operational addresses (key rotation, infrastructure changes).
- Any staker who previously used O1 retains a permanent stale consent entry.
- No special privilege is required — any staker can trigger this against any former operational address.

### Recommendation
In `change_operational_address`, clear the `eligible_operational_addresses` entry for the old address after the change:

```cairo
// After clearing operational_address_to_staker_address[old_address]:
self.eligible_operational_addresses.write(old_address, Zero::zero());
```

### Proof of Concept

```
1. O1 calls declare_operational_address(staker_A)
   → eligible_operational_addresses[O1] = staker_A

2. Staker A calls change_operational_address(O1)
   → operational_address_to_staker_address[O1] = staker_A
   → eligible_operational_addresses[O1] = staker_A  (unchanged)

3. O2 calls declare_operational_address(staker_A)
   → eligible_operational_addresses[O2] = staker_A

4. Staker A calls change_operational_address(O2)
   → operational_address_to_staker_address[O1] = 0  (cleared)
   → eligible_operational_addresses[O1] = staker_A  (STALE — never cleared)

5. Staker A calls change_operational_address(O1) again
   → Check: operational_address_to_staker_address[O1].is_zero() → TRUE ✓
   → Check: eligible_operational_addresses[O1] == staker_A → TRUE ✓ (stale entry)
   → Succeeds WITHOUT O1's current consent
   → operational_address_to_staker_address[O1] = staker_A

6. O1 tries to escape: calls declare_operational_address(staker_B)
   → Assert: operational_address_to_staker_address[O1].is_zero() → FALSE ✗
   → REVERTS with OPERATIONAL_IN_USE

7. O1 is permanently trapped as staker_A's operational address
   until staker_A voluntarily releases it (step 4 can repeat).
```

        fn change_operational_address(
            ref self: ContractState, operational_address: ContractAddress,
        ) {
            // Prerequisites and asserts.
            self.general_prerequisites();
            assert!(
                self.operational_address_to_staker_address.read(operational_address).is_zero(),
                "{}",
                Error::OPERATIONAL_EXISTS,
            );
            assert!(
                !self.does_token_exist(token_address: operational_address),
                "{}",
                Error::OPERATIONAL_IS_TOKEN,
            );
            let staker_address = get_caller_address();
            let mut staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            assert!(
                self.eligible_operational_addresses.read(operational_address) == staker_address,
                "{}",
                Error::OPERATIONAL_NOT_ELIGIBLE,
            );

            // Set operational address and write to storage.
            let old_address = staker_info.operational_address;
            self.operational_address_to_staker_address.write(old_address, Zero::zero());
            staker_info.operational_address = operational_address;
            self.write_staker_info(:staker_address, :staker_info);
            self.operational_address_to_staker_address.write(operational_address, staker_address);

            // Emit event.
            self
                .emit(
                    Events::OperationalAddressChanged {
                        staker_address, new_address: operational_address, old_address,
                    },
                );
        }
