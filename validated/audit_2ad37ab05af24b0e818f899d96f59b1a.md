### Title
Missing Maximum Fixed-Gas Guard in `submit`/`submit_with_args` Allows Owner-Triggered Overpayment — (File: engine/src/engine.rs)

---

### Summary

In Silo mode, the `fixed_gas` parameter is read from on-chain storage at the moment a user's transaction is executed, not at the moment it is signed. The owner can update `fixed_gas` at any time via `set_silo_params` or `set_fixed_gas`. If the owner raises `fixed_gas` between when a user signs their transaction and when it is processed, the user pays more ETH than they expected. The `SubmitArgs` struct exposes a `max_gas_price` cap but provides no analogous `max_fixed_gas` cap, leaving users with no slippage protection against a gas-amount change.

---

### Finding Description

Inside the transaction-execution path, `fixed_gas` is fetched live from storage:

```rust
let fixed_gas = silo::get_fixed_gas(&io);
``` [1](#0-0) 

That value is then used directly to compute the amount debited from the sender:

```rust
let prepaid_amount = fixed_gas
    .map_or(transaction.gas_limit, EthGas::as_u256)
    .checked_mul(effective_gas_price)
    .map(Wei::new)
    .ok_or(GasPaymentError::EthAmountOverflow)?;
``` [2](#0-1) 

When `fixed_gas` is active, `refund_unused_gas` also uses `fixed_gas` (not actual gas consumed) for both the `spent_amount` and `reward_amount` calculations, so the user receives **zero refund** and the relayer collects the full `fixed_gas × priority_fee_per_gas`:

```rust
let gas_to_wei = |price: U256| {
    fixed_gas
        .map_or_else(|| gas_used.into(), EthGas::as_u256)
        .checked_mul(price)
        ...
};
``` [3](#0-2) 

The owner can raise `fixed_gas` at any time through `set_fixed_gas` or `set_silo_params`, both of which are owner-only and take effect immediately:

```rust
pub fn set_fixed_gas<I: IO>(io: &mut I, gas: Option<EthGas>) { ... }
pub fn set_silo_params<I: IO>(io: &mut I, args: Option<SiloParamsArgs>) { ... }
``` [4](#0-3) 

The `SubmitArgs` struct offers `max_gas_price` to cap the per-unit price, but nothing to cap the gas *quantity*:

```rust
pub struct SubmitArgs {
    pub tx_data: Vec<u8>,
    pub max_gas_price: Option<u128>,   // caps price, not amount
    pub gas_token_address: Option<Address>,
}
``` [5](#0-4) 

There is also a guard that rejects transactions where `fixed_gas > gas_limit`, but this only prevents a hard failure — it does not protect the user from paying a higher-than-expected amount within the `[0, gas_limit]` range:

```rust
if fixed_gas.is_some_and(|gas| gas.as_u256() > transaction.gas_limit) {
    return Err(EngineErrorKind::FixedGasOverflow.into());
}
``` [6](#0-5) 

---

### Impact Explanation

A user who signs a transaction while `fixed_gas = X` will pay `X × gas_price` Wei. If the owner raises `fixed_gas` to `Y` (where `X < Y ≤ gas_limit`) before the transaction is processed, the user pays `Y × gas_price` Wei — the surplus `(Y − X) × gas_price` is transferred to the relayer without the user's consent. Because NEAR blocks finalize in ~1–2 seconds and the owner's `set_fixed_gas` call takes effect in the very next block, any transaction already signed but not yet submitted (or queued in a relayer's mempool) is exposed. The loss is bounded by `gas_limit × gas_price` but can be substantial for high-value Silo deployments with large gas limits and non-zero gas prices.

---

### Likelihood Explanation

The owner of a Silo deployment has a legitimate operational reason to adjust `fixed_gas` (e.g., to reflect updated computation costs). Any such adjustment races with in-flight user transactions. Because NEAR has no user-controlled nonce ordering or slippage guard on the gas-amount dimension, the race is unavoidable whenever `fixed_gas` is changed while users are actively submitting transactions. The scenario does not require the owner to act maliciously — a routine parameter update is sufficient.

---

### Recommendation

**Short term:** Add a `max_fixed_gas: Option<u64>` field to `SubmitArgs`. At the start of `submit_with_args`, compare the live `fixed_gas` against the user-supplied cap and return an error if it is exceeded, analogous to the existing `max_gas_price` check. This gives users a slippage guard on the gas-amount dimension.

**Long term:** Audit all other mutable Silo/engine parameters (e.g., `erc20_fallback_address`, whitelist entries) for similar races where a legitimate owner update can silently change the outcome of an in-flight user transaction.

---

### Proof of Concept

1. Owner deploys Aurora in Silo mode and sets `fixed_gas = 100_000` via `set_silo_params`.
2. Alice signs a legacy transaction with `gas_limit = 1_000_000` and `gas_price = 10`. She expects to pay `100_000 × 10 = 1_000_000 Wei`.
3. Before Alice's transaction is included, the owner calls `set_fixed_gas(Some(EthGas::new(800_000)))` — a legitimate operational adjustment.
4. Alice's transaction is processed in the next block. `fixed_gas` is now `800_000`.
5. `charge_gas` debits `800_000 × 10 = 8_000_000 Wei` from Alice.
6. `refund_unused_gas` computes `spent_amount = 800_000 × 10 = 8_000_000 Wei`; refund = 0.
7. Alice loses `7_000_000 Wei` more than she expected, with no recourse. The relayer collects the full `8_000_000 Wei`. [1](#0-0) [7](#0-6) [8](#0-7)

pub fn set_silo_params<I: IO>(io: &mut I, args: Option<SiloParamsArgs>) {
    let (cost, address) = args.map_or((None, None), |params| {
        (Some(params.fixed_gas), Some(params.erc20_fallback_address))
    });

    set_fixed_gas(io, cost);
    set_erc20_fallback_address(io, address);
}

/// Return gas amount per transaction.
pub fn get_fixed_gas<I: IO>(io: &I) -> Option<EthGas> {
    let key = fixed_gas_key();
    io.read_storage(&key)
        .and_then(|bytes| bytes.to_value().ok())
}

/// Set an amount of gas per transaction.
pub fn set_fixed_gas<I: IO>(io: &mut I, gas: Option<EthGas>) {
    let key = fixed_gas_key();

    if let Some(gas) = gas {
        io.write_borsh(&key, &gas);
    } else {
        io.remove_storage(&key);
    }
}
