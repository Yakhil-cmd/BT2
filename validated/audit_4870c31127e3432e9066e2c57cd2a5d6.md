### Title
`transfer()` with Fixed 2300 Gas Stipend in `WSEI.withdraw()` Permanently Freezes Funds for Smart Contract Callers — (`contracts/src/WSEI.sol`)

---

### Summary

The Sei chain ships a production `WSEI` (Wrapped SEI) contract with compiled artifacts embedded in the Go binary. Its `withdraw()` function uses `payable(msg.sender).transfer(wad)`, which forwards a hard-capped 2300 gas stipend. Any smart contract whose `receive()` or `fallback()` consumes more than 2300 gas — including multisig wallets, proxy contracts, and contracts that emit events on receipt — will have their withdrawal permanently revert, freezing their deposited SEI inside WSEI with no recovery path.

---

### Finding Description

`WSEI.sol` is not a user-deployed example contract. It has compiled artifacts embedded in the chain binary: [1](#0-0) 

and is deployable via the chain CLI: [2](#0-1) 

The vulnerable line is in `withdraw()`: [3](#0-2) 

```solidity
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    payable(msg.sender).transfer(wad);   // ← 2300 gas cap
    emit Withdrawal(msg.sender, wad);
}
```

`transfer()` reverts the entire transaction on failure. Because the balance decrement and the transfer are in the same transaction, the accounting reverts too — but the caller is left unable to ever successfully call `withdraw()`. Their WSEI balance is permanently unspendable as native SEI.

---

### Impact Explanation

Any smart contract that:
- deposits SEI via `deposit()` or the `receive()` fallback, and
- has a `receive()`/`fallback()` that uses >2300 gas (e.g., emits an event, writes storage, is a proxy, or is a multisig)

…will have its SEI permanently frozen inside the WSEI contract. The user cannot transfer WSEI to an EOA and withdraw from there because `transferFrom` requires the original depositor's approval and balance. The only exit is `withdraw()`, which is broken for them.

Severity maps to **Medium** (fund freeze; amount depends on depositor) or **Critical** if a large depositor is affected (freeze ≥ $5k).

---

### Likelihood Explanation

- Gnosis Safe and other multisig wallets are the most common smart contract users of WSEI-style contracts; their `receive()` hooks routinely exceed 2300 gas.
- Proxy-based DeFi contracts (e.g., upgradeable vaults) that hold WSEI on behalf of users are equally affected.
- No privileged access is required. Any unprivileged smart contract that calls `deposit()` triggers the condition.

---

### Recommendation

Replace `transfer()` with a low-level `call()` and add a reentrancy guard:

```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

function withdraw(uint wad) public nonReentrant {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    (bool success, ) = payable(msg.sender).call{value: wad}("");
    require(success, "WSEI: transfer failed");
    emit Withdrawal(msg.sender, wad);
}
```

The CEI (Checks-Effects-Interactions) pattern is already followed (balance decremented before the external call), so adding `nonReentrant` is sufficient protection.

---

### Proof of Concept

1. Deploy a contract `Vault` with a `receive()` that emits an event (>2300 gas).
2. `Vault` calls `WSEI.deposit{value: 1 ether}()`.
3. `Vault` calls `WSEI.withdraw(1 ether)`.
4. The `transfer()` call to `Vault` runs out of gas at the `emit` opcode; the transaction reverts.
5. `Vault`'s WSEI balance remains 1 ether but can never be redeemed as native SEI.
