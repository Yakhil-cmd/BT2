### Title
Wrong inequality in `apply_delegate_action` allows DelegateAction execution at the `max_block_height` boundary — (`runtime/runtime/src/actions.rs`)

---

### Summary

`apply_delegate_action` uses a strict `>` comparison to check DelegateAction expiry, but the protocol specification requires `>=`. When `block_height == max_block_height`, the code does **not** expire the action, while the spec and field documentation both state it must be expired at that exact height. An unprivileged user who sets `max_block_height = N` to bound their signed meta-transaction to blocks strictly below `N` will have that action executed at block `N` by a relayer, violating the signed intent.

---

### Finding Description

In `apply_delegate_action`:

```rust
if apply_state.block_height > delegate_action.max_block_height() {
    result.result = Err(ActionErrorKind::DelegateActionExpired.into());
    return Ok(());
}
``` [1](#0-0) 

The field `max_block_height` is documented as:

> "The maximal height of the block in the blockchain **below which** the given DelegateAction is valid." [2](#0-1) 

The runtime spec explicitly states the expiry condition as:

> "If the current block is **equal or greater than** `max_block_height`" → `DelegateActionExpired` [3](#0-2) 

The error variant's own doc comment reads:

> "Delegate action has expired. `max_block_height` is **less than** actual block height." [4](#0-3) 

All three authoritative sources agree: the action must be expired when `block_height >= max_block_height`. The production code uses `>` (strict), so the boundary case `block_height == max_block_height` passes the guard and the action executes.

The existing unit test `test_delegate_action_max_height` only tests `max_block_height + 1` (strictly above), never the equal case:

```rust
let apply_state =
    create_apply_state(signed_delegate_action.delegate_action.max_block_height + 1);
``` [5](#0-4) 

Meanwhile `test_delegate_action` passes `block_height == max_block_height` and asserts success — confirming the off-by-one is baked into the test suite and has gone undetected:

```rust
let apply_state =
    create_apply_state(signed_delegate_action.delegate_action.max_block_height);
``` [6](#0-5) 

---

### Impact Explanation

A `DelegateAction` is a signed meta-transaction: the **sender** (an unprivileged user) signs the payload including `max_block_height`, and a **relayer** submits it on-chain. The sender's cryptographic commitment is that the action is valid only for blocks strictly below `max_block_height`. Because the expiry guard uses `>` instead of `>=`, a relayer can submit the action at exactly block `max_block_height` and it will execute — creating a new `ActionReceipt` with the sender's funds and actions — one block beyond the sender's signed intent.

Concrete consequences:
- **Fund transfers**: A `Transfer` inside the DelegateAction executes at block `N` when the sender intended it to be invalid at `N`.
- **Key/account mutations**: `AddKey`, `DeleteKey`, `DeleteAccount` actions execute at the boundary block.
- **Nonce advancement**: The sender's access-key nonce is incremented at block `N`, consuming a nonce the sender believed was safe.

The corrupted value is the **new `ActionReceipt`** emitted by `apply_delegate_action` at line 488 — it should not exist when `block_height == max_block_height`. [7](#0-6) 

---

### Likelihood Explanation

The root cause is reachable by any unprivileged user who relies on `max_block_height` for time-bounding. A relayer (which is a separate, potentially adversarial party in the meta-transaction model) can observe the current chain height and deliberately submit the DelegateAction at exactly block `max_block_height`. No validator or node-operator privilege is required. The window is exactly one block, but block production is deterministic enough for a relayer to target it.

---

### Recommendation

Change the strict inequality to `>=` in `apply_delegate_action`:

```rust
// Before (wrong):
if apply_state.block_height > delegate_action.max_block_height() {

// After (correct):
if apply_state.block_height >= delegate_action.max_block_height() {
``` [8](#0-7) 

Also update `test_delegate_action` to assert that `block_height == max_block_height` produces `DelegateActionExpired`, and add a new passing test for `block_height == max_block_height - 1`.

---

### Proof of Concept

1. Sender Alice signs a `DelegateAction` with `max_block_height = 1000`, intending it to be valid only for blocks 0–999.
2. Alice sends the signed action to a relayer.
3. The relayer monitors the chain and waits until block 1000.
4. At block 1000, the relayer submits the enclosing transaction.
5. `apply_delegate_action` evaluates: `1000 > 1000` → **false** → no expiry error.
6. The DelegateAction executes: a new `ActionReceipt` is created and Alice's funds/actions are applied at block 1000.
7. Alice's intent — that the action be invalid at block 1000 — is violated.

The fix `>=` would evaluate `1000 >= 1000` → **true** → `DelegateActionExpired`, matching the spec. [9](#0-8) [10](#0-9)

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
