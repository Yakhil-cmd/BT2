### Title
Missing Zero-Address Validation in `change_reward_address` Causes `unstake_action` to Permanently Revert, Temporarily Freezing Staker Principal - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

Both `Staking.change_reward_address` and `Pool.change_reward_address` accept `address(0)` as a valid reward address because neither function validates the input for zero. Once a staker's `reward_address` is set to zero, `unstake_action` unconditionally calls `send_rewards_to_staker`, which calls `checked_transfer(recipient: 0x0, ...)` on the STRK ERC20 token. The OpenZeppelin ERC20 implementation reverts on any transfer to the zero address, causing `unstake_action` to revert and temporarily freezing the staker's principal in the contract.

---

### Finding Description

`Staking.change_reward_address` validates only that the new address is not a registered token address:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

There is no `reward_address.is_non_zero()` guard. [1](#0-0) 

The same omission exists in `Pool.change_reward_address`, which only checks that the new address is not the pool's token contract:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

The same omission is present in the initial `stake` call, which also only checks `!self.does_token_exist(token_address: reward_address)` for the reward address. [3](#0-2) 

Once `reward_address` is zero, `unstake_action` calls `send_rewards_to_staker` unconditionally before erasing the staker record: [4](#0-3) 

`send_rewards_to_staker` always calls `checked_transfer` to `reward_address`, even when `amount` is zero: [5](#0-4) 

The STRK ERC20 (OpenZeppelin Cairo) rejects any transfer to `address(0)`, causing the entire `unstake_action` transaction to revert.

The same pattern applies to `Pool.claim_rewards`, which calls `checked_transfer(recipient: reward_address, ...)` without a zero-address guard: [6](#0-5) 

---

### Impact Explanation

A staker who has set (or initially registered with) `reward_address = 0x0` cannot complete `unstake_action`. The call reverts at the `send_rewards_to_staker` step, before the staker record is erased and before the principal is returned. The staker's principal remains locked in the staking contract for as long as `reward_address` remains zero. This constitutes a **temporary freezing of funds** (principal, not just yield). The staker can recover by calling `change_reward_address` with a valid address and retrying `unstake_action`, but the protocol itself does not prevent the broken state from being entered.

For pool members, `exit_delegation_pool_action` transfers to `pool_member` (not `reward_address`), so the principal is not blocked — but `claim_rewards` reverts, causing **temporary freezing of unclaimed yield**.

---

### Likelihood Explanation

Any staker or pool member can call `change_reward_address` with `0x0` at any time, including after calling `unstake_intent`. A user who accidentally passes the wrong argument, or a script/UI that sends a malformed transaction, can trigger this state. The entry path requires no privilege — it is a standard user-facing function callable by any active staker or pool member.

---

### Recommendation

Add a non-zero check in both `change_reward_address` implementations and in the `stake` / `enter_delegation_pool` entry points:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing `GenericError::ZERO_ADDRESS` error already defined in the codebase. [7](#0-6) 

---

### Proof of Concept

1. Staker calls `staking.stake(reward_address: 0x0, ...)` — succeeds (no zero-address check). [8](#0-7) 
2. Alternatively, an existing staker calls `staking.change_reward_address(reward_address: 0x0)` — succeeds. [1](#0-0) 
3. Staker calls `staking.unstake_intent()` — succeeds, sets `unstake_time`.
4. After the `exit_wait_window` elapses, staker calls `staking.unstake_action(staker_address)`.
5. Inside `unstake_action`, `send_rewards_to_staker` is called, which executes `token_dispatcher.checked_transfer(recipient: 0x0, amount: ...)`. [9](#0-8) 
6. The STRK ERC20 reverts on transfer to `address(0)`. The entire `unstake_action` transaction reverts.
7. The staker's principal remains locked. The staker must call `change_reward_address` with a valid address before `unstake_action` can succeed.
