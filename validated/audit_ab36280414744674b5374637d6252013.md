### Title
Gateway `skip_stateful_validations` admits invoke transactions with invalid signatures by accepting any mempool presence as deploy-account proof — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator bypasses the `__validate__` entry-point call (the only place the account signature is cryptographically verified) for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true`. An unprivileged attacker can exploit this by first submitting a valid `deploy_account` transaction to seed the mempool, then immediately submitting an invoke transaction with nonce=1 and an **arbitrary/invalid signature**. The signature is never verified, and the transaction is admitted to the mempool.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function skips the `__validate__` entry point when all four conditions hold:

1. The transaction is an `Invoke` tx.
2. `tx.nonce() == Nonce(Felt::ONE)` (nonce is exactly 1).
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

The comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is **incorrect** — the check is not type-specific.

**`account_tx_in_pool_or_recent_block` is not type-specific** [3](#0-2) 

It returns `true` if **any** transaction for the address is in the pool or in the committed-nonce state. It does not distinguish between `deploy_account`, `invoke`, or `declare` transactions.

**`MempoolState::contains_account` persists after commit** [4](#0-3) 

The state retains the address even after a block is committed, as confirmed by the test: [5](#0-4) 

**`run_validate_entry_point` fully skips `__validate__` when `skip_validate=true`** [6](#0-5) 

This sets `execution_flags.validate = false`, which causes `StatefulValidator::perform_validations` to return immediately before calling `__validate__`: [7](#0-6) 

And `validate_tx` also short-circuits: [8](#0-7) 

No other path in the gateway verifies the signature cryptographically. The stateless validator only checks signature **length**, not validity: [9](#0-8) 

---

### Impact Explanation

**Broken invariant:** Every invoke transaction admitted to the mempool must have its `__validate__` entry point verified (the account's signature check). This invariant is violated.

**Exact corrupted value:** An `InternalRpcTransaction::Invoke` with an attacker-chosen, cryptographically invalid signature is inserted into the mempool pool (`tx_pool`) and forwarded to the batcher. The admission decision is wrong.

**Downstream effect:** The batcher picks up the transaction and executes it with `execution_flags.validate = true` (the default for block execution). The `__validate__` entry point then runs and fails, causing the transaction to revert. The attacker has:
- Bypassed gateway admission controls.
- Polluted the mempool with invalid transactions.
- Forced the batcher to waste execution resources on guaranteed-revert transactions.
- Achieved a repeatable DoS vector (each cycle costs only two cheap transactions).

**Matching impact:** High — *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The attack requires only two standard transaction submissions from any unprivileged account:

1. A valid `deploy_account` transaction (normal UX operation, costs gas).
2. An invoke transaction with nonce=1 and an arbitrary signature.

No privileged access, no special contract, no race condition beyond the deploy_account being in the pool at the moment the invoke is submitted. The attacker controls both transactions entirely. The attack is repeatable across different fresh addresses.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the address in the mempool pool. The mempool should expose a method such as `deploy_account_tx_in_pool(address)` that inspects `InternalRpcTransactionWithoutTxHash::DeployAccount` entries in `tx_pool` for the given address. Only if a `deploy_account` transaction is present should the `__validate__` skip be granted.

Alternatively, restrict the skip to only when the account's **only** pool transaction is a `deploy_account` at nonce 0, ensuring the UX improvement cannot be abused to bypass signature verification for arbitrary invoke transactions.

        block_info.block_number = block_info.block_number.unchecked_next();

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
