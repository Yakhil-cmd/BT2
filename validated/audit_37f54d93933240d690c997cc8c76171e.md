### Title
Declare Transaction `l2_gas.max_amount` Stateless Admission Check Unconditionally Skipped, Allowing Policy-Violating Declares Into the Mempool — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` contains an explicit empty branch that skips the `max_l2_gas_amount` check for every `RpcTransaction::Declare`. Any unprivileged user can submit a Declare transaction whose `l2_gas.max_amount` exceeds the configured gateway limit and have it admitted through the stateless gate into the mempool. The bypass is intentional-but-incomplete (a TODO acknowledges it), is confirmed by a dedicated passing test, and is not compensated by any downstream stateless or stateful amount check.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs` the relevant block is:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The `if let RpcTransaction::Declare(_) = tx { }` arm is an empty no-op. For every Declare the function falls through to `Ok(())` regardless of `l2_gas.max_amount`. The production default is `max_l2_gas_amount = 1_210_000_000`. [2](#0-1) 

The existing test `valid_l2_gas_amount_on_declare` explicitly asserts `Ok(())` for a Declare whose `max_amount = 200` when the configured limit is `100`: [3](#0-2) 

The parallel test `test_invalid_max_l2_gas_amount` only parameterises over `TransactionType::DeployAccount` and `TransactionType::Invoke`, confirming Declare is deliberately excluded from the rejection path: [4](#0-3) 

**Downstream checks do not close the gap for amounts near the limit.**

The stateful validator's `validate_resource_bounds` checks only `l2_gas_price` (price per unit), not `max_amount`: [5](#0-4) 

`verify_can_pay_committed_bounds` checks whether the sender's balance covers `max_amount × max_price_per_unit`. With production values (`min_gas_price = 8_000_000_000`, `max_l2_gas_amount = 1_210_000_000`), a Declare with `max_amount = 1_210_000_001` (one unit above the limit) produces a `max_possible_fee` of only ~9.68 STRK — well within a normal account balance — so the balance check passes and the transaction enters the mempool. [6](#0-5) 

The blockifier bouncer tracks `sierra_gas`, `proving_gas`, and `receipt_l2_gas` from **actual execution**, not from `max_amount`, so a Declare with an inflated `max_amount` does not affect block-capacity accounting: [7](#0-6) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` exceeding `max_l2_gas_amount` is admitted to the mempool when the gateway's own stateless policy requires it to be rejected. The corrupted value is the **admission decision**: the gateway returns `Ok(())` for a transaction it is configured to reject. This maps directly to the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Downstream execution, fees, and state remain correct because the bouncer and fee logic use actual gas consumed, not `max_amount`. The harm is confined to the admission layer: the gateway's resource-bound policy is unenforceable for Declare transactions.

### Likelihood Explanation

Exploitation requires no privileges. Any user who can submit an RPC Declare transaction can trigger this path. The bypass is unconditional — it applies to every Declare regardless of the configured limit. The TODO comment and the dedicated passing test confirm the team is aware the check is absent.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx { }` arm and apply the same `max_l2_gas_amount` guard uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher or separate limit, introduce a dedicated `max_l2_gas_amount_declare` config field rather than skipping the check entirely.

### Proof of Concept

The existing test already proves the bypass. To make it explicit as a regression test:

```rust
#[test]
fn declare_bypasses_max_l2_gas_amount() {
    let config = StatelessTransactionValidatorConfig {
        validate_resource_bounds: true,
        max_l2_gas_amount: 1_210_000_000,
        min_gas_price: 8_000_000_000,
        ..Default::default()
    };
    let validator = StatelessTransactionValidator { config };

    // Amount = limit + 1; same transaction type as Invoke would be rejected.
    let declare_tx = rpc_tx_for_testing(
        TransactionType::Declare,
        RpcTransactionArgs {
            resource_bounds: AllResourceBounds {
                l2_gas: ResourceBounds {
                    max_amount: GasAmount(1_210_000_001),
                    max_price_per_unit: GasPrice(8_000_000_000),
                },
                ..Default::default()
            },
            ..Default::default()
        },
    );
    // Passes — should fail.
    assert_matches!(validator.validate(&declare_tx), Ok(()));

    // Same amount on Invoke is correctly rejected.
    let invoke_tx = rpc_tx_for_testing(
        TransactionType::Invoke,
        RpcTransactionArgs {
            resource_bounds: AllResourceBounds {
                l2_gas: ResourceBounds {
                    max_amount: GasAmount(1_210_000_001),
                    max_price_per_unit: GasPrice(8_000_000_000),
                },
                ..Default::default()
            },
            ..Default::default()
        },
    );
    assert_matches!(
        validator.validate(&invoke_tx),
        Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { .. })
    );
}

```rust
pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}
