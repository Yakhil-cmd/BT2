### Title
Missing Storage Collision Check in CREATE/CREATE2 Allows Pre-Poisoning of Deterministic Deployment Addresses - (`evm_interpreter/src/ee_trait_impl.rs`)

### Summary

ZKsync OS's EVM interpreter does not check whether a target address has non-empty storage before allowing a `CREATE`/`CREATE2` deployment to proceed. An unprivileged attacker can pre-populate storage at a deterministic `CREATE2` address before the legitimate deployer acts, causing the deployed contract to inherit attacker-controlled storage state. This is the direct analog of the GithubHint race condition: in both cases, a "first-come-first-served" registration check silently omits a critical ownership/state guard, allowing an attacker to pre-claim the slot with malicious data.

---

### Finding Description

In `before_executing_frame` inside `evm_interpreter/src/ee_trait_impl.rs`, the CREATE collision guard checks only `deployee_code_len != 0 || deployee_nonce != 0`:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // ... burn gas, return CreateCollision error
    return Ok(false);
}
```

The comment explicitly acknowledges the missing storage check. [1](#0-0) 

EIP-7610 (active since Paris/Cancun) requires that `CREATE`/`CREATE2` also fail when the target address has non-empty storage. Because ZKsync OS skips this check, a deployment to an address that has pre-existing storage (but zero nonce and no code) **succeeds silently**, and the newly deployed contract inherits the attacker-supplied storage values.

This is confirmed as a known divergence in the project documentation:

> "Deployment doesn't fail if the storage for the deployed address is already used (when nonce is 0 and code is empty)." [2](#0-1) 

Multiple EVM compatibility test cases are explicitly disabled with comments such as `"We do not check for storage collisions"` and `"We do not check storage for collision"`. [3](#0-2) 

The `docs/not-a-bug.md` file does **not** list this behavior as intentionally accepted, distinguishing it from other documented non-bugs. [4](#0-3) 

---

### Impact Explanation

An attacker who knows a victim's upcoming `CREATE2` address (which is fully deterministic and publicly computable from deployer address, salt, and initcode hash) can:

1. Call `SSTORE` on the target address before the victim deploys, writing attacker-controlled values to storage slots used by the victim's contract (e.g., `initialized = true`, `owner = attacker`, `balance[attacker] = MAX_UINT`).
2. The victim's `CREATE2` deployment proceeds without error — the storage collision check is absent.
3. The deployed contract starts execution with the attacker's pre-set storage, bypassing any `require(!initialized)` guard, corrupting ownership state, or pre-crediting the attacker with tokens.

Concrete impact: **direct theft of funds or complete ownership takeover** of any contract that relies on storage being zero at initialization and is deployed via `CREATE2`.

---

### Likelihood Explanation

- `CREATE2` addresses are deterministic and computable by anyone from public parameters.
- Any unprivileged account can write to any storage slot on any address (including undeployed ones) via a normal EVM transaction.
- The attacker only needs to submit their `SSTORE` transaction before the victim's deployment transaction is included in a block — a standard front-running scenario on a sequencer-ordered chain.
- Contracts using `CREATE2` for factory patterns (proxy deployers, minimal proxies, counterfactual wallets) are extremely common and are the primary targets.

Likelihood: **Medium-High** for any protocol using `CREATE2` with initialization guards.

---

### Recommendation

Implement the EIP-7610 storage-emptiness check before allowing a `CREATE`/`CREATE2` frame to execute. In `before_executing_frame`, after the existing `deployee_code_len != 0 || deployee_nonce != 0` check, add a check that the target address has no non-zero storage slots. Because ZKsync OS uses a flat storage model, this requires reading the account's storage root or iterating warm storage entries for the target address. If a non-empty storage root is detected, the deployment should fail with `CreateCollision` (burning all gas), consistent with EIP-7610 semantics.

---

### Proof of Concept

**Setup:**
- Victim contract `V` uses `CREATE2` with a known salt to deploy a token contract `T` that has `mapping(address => uint256) balances` at slot 0 and `bool initialized` at slot 1.
- `T`'s constructor does `require(!initialized); initialized = true; owner = msg.sender;`

**Attack:**
1. Attacker computes `T_addr = keccak256(0xff ++ V_addr ++ salt ++ keccak256(T_initcode))[12:]`.
2. Attacker sends a transaction: `SSTORE(slot=1, value=1)` at address `T_addr` (sets `initialized = true`).
3. Attacker sends: `SSTORE(slot=keccak256(attacker_addr, 0), value=MAX_UINT)` at `T_addr` (pre-credits attacker with max tokens).
4. Victim calls `V.deploy(salt)` → `CREATE2` succeeds (nonce=0, code_len=0 → no collision detected).
5. `T`'s constructor runs: `require(!initialized)` reads storage slot 1 = `true` → **constructor reverts**, OR if the constructor doesn't check, the attacker already has `MAX_UINT` tokens.

The root cause is the absent storage check at: [1](#0-0)

```markdown
# Not a Bug

This page lists ZKsync OS behavior that is intentional, protocol-correct, or already documented, but is commonly misreported as a vulnerability.

Use this page as a triage aid, not as a substitute for checking the code. If an observed behavior differs from what is described here, investigate it normally.

## EVM / Cancun Behavior

### Cancun is the supported EVM hardfork

ZKsync OS currently targets Cancun EVM semantics. Reports that assume later hardfork behavior are not valid unless they identify an issue under the supported hardfork.

See [EVM Execution Environment](./execution_environments/evm.md).

### EIP-4844 transactions are disabled in production

EIP-4844 type `0x03` transactions are implemented behind the `basic_bootloader/eip-4844` feature, but this feature is not enabled by the production feature sets. It is enabled for test and Ethereum test-runner configurations.

See [Transaction formats](./bootloader/transaction_format.md).

### `BLOBHASH` returns `0`

This is expected in production. Since EIP-4844 transactions are disabled, transactions do not have blob versioned hashes. Cancun/EIP-4844 specifies that `BLOBHASH(index)` returns a zeroed `bytes32` when `index` is outside `tx.blob_versioned_hashes`.

### `BLOBBASEFEE` returns `1`

This is expected in production. There are no EIP-4844 transactions in production history, so `excess_blob_gas` remains `0`. Cancun/EIP-4844 defines the minimum blob base fee as `1`, and EIP-7516 makes `BLOBBASEFEE` return the current block's blob base fee.

### Opcode `0x44` returns `1`

Opcode `0x44` is `PREVRANDAO` under Cancun semantics, historically named `DIFFICULTY`. ZKsync OS uses the block `mix_hash` value for this opcode. In production, `mix_hash` is mocked to `1`.

This does not contradict the block header `difficulty` field being `0` after the Merge.

See [Bootloader block header](./bootloader/bootloader.md#block-header).

### `SUB` operand order

ZKsync OS follows the standard EVM operand order for `SUB`. The top stack item is the first operand, and the item below it is the second operand. For example, `PUSH1 0x03 PUSH1 0x05 SUB` leaves `0x02` on the stack.

This is not a reversed subtraction bug.
```
