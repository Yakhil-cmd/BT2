### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures via Non-Deploy-Account Mempool Presence — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway skips the `__validate__` entry-point check for invoke transactions with nonce=1 when the account has **any** transaction in the mempool, not exclusively a `deploy_account` transaction. An unprivileged attacker who observes a victim account's pending nonce=0 invoke in the mempool can submit a nonce=1 invoke for that account with an arbitrary (invalid) signature, bypassing gateway-level signature validation and injecting the transaction into the mempool.

### Finding Description

`skip_stateful_validations` is the UX feature that allows a user to submit `deploy_account + invoke` atomically. Its guard is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The code comment justifies the check as: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type in the pool — invoke, declare, or deploy_account — keyed only by `ContractAddress`. There is no check that the pooled transaction is a `deploy_account`.

**Attack path:**

1. Victim account `X` is already deployed (on-chain nonce = 0) and has a valid nonce=0 invoke pending in the mempool (it passed `__validate__` normally).
2. Attacker observes `X` in the public mempool.
3. Attacker crafts a nonce=1 invoke for `X` with an arbitrary invalid signature (passes stateless size checks).
4. Gateway evaluates `skip_stateful_validations`:
   - `tx.nonce() == Nonce(Felt::ONE)` ✓
   - `account_nonce == Nonce(Felt::ZERO)` ✓ (on-chain state not yet updated)
   - `account_tx_in_pool_or_recent_block(X)` → `true` ✓ (victim's nonce=0 invoke is pooled)
5. `validate: false` is set; `__validate__` is **not** called at the gateway. [3](#0-2) 

6. The invalid transaction is forwarded to the mempool and admitted. [4](#0-3) 

The broken invariant: the gateway's stateful validation path must call `__validate__` for every invoke transaction unless the account is provably undeployed and a `deploy_account` for it is pending. The current check conflates "account has any pending transaction" with "account has a pending deploy_account," allowing the skip to fire on already-deployed accounts with a pending nonce=0 invoke.

### Impact Explanation

The gateway admits an invoke transaction with an invalid (attacker-controlled) signature into the mempool without running `__validate__`. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The batcher will call `__validate__` during block execution and the transaction will revert/be rejected there, so state corruption does not occur. The concrete harm is mempool pollution: for every victim account with a pending nonce=0 invoke, an attacker can inject one invalid nonce=1 transaction, consuming mempool capacity and wasting batcher execution resources.

### Likelihood Explanation

The mempool is public. Any observer can enumerate accounts with pending nonce=0 invokes (account on-chain nonce = 0, tx in pool). No privileged access, no special key material, and no prior relationship with the victim is required. The attacker only needs to craft a well-formed but incorrectly-signed invoke transaction, which passes all stateless checks.

### Recommendation

Replace the address-only `account_tx_in_pool_or_recent_block` check with a check that verifies a `deploy_account` transaction specifically is pending for the sender address. The mempool should expose a `deploy_account_in_pool(address)` query, or the gateway should inspect the transaction type of the pooled entry before granting the skip. Alternatively, restrict the skip to the case where `account_nonce == Nonce(Felt::ZERO)` **and** the account contract class is not yet set in state (i.e., the account truly does not exist on-chain), which is the only scenario where skipping `__validate__` is semantically safe.

### Proof of Concept

```rust
// Precondition: account X is deployed on-chain (nonce=0 in state).
// Victim submits a valid nonce=0 invoke; it passes __validate__ and enters the mempool.

// Attacker step:
let attacker_tx = RpcInvokeTransactionV3 {
    sender_address: X,          // victim's address
    nonce: Nonce(Felt::ONE),    // nonce=1
    signature: TransactionSignature(vec![Felt::from(0xdeadbeef_u64)]), // invalid
    resource_bounds: valid_resource_bounds(),
    // ... other fields
};

// Gateway evaluation of skip_stateful_validations:
//   tx.nonce() == 1                                  → true
//   account_nonce (from state) == 0                  → true
//   account_tx_in_pool_or_recent_block(X)            → true (victim's nonce=0 invoke is pooled)
// Result: validate=false, __validate__ skipped.
// Transaction admitted to mempool with invalid signature.

// Batcher later calls __validate__ → fails → transaction rejected.
// Net effect: one invalid transaction injected per victim account with a pending nonce=0 invoke.
```
