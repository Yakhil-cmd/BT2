### Title
Unbounded Loop in `calculate_rewards` Over Pool Member Balance Trace Can Permanently Freeze Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` internal function in `Pool` iterates over every entry in a pool member's `pool_member_epoch_balance` trace without any upper bound. A delegator who repeatedly calls `add_to_delegation_pool` across many distinct epochs without ever claiming rewards will grow this trace unboundedly. Once the trace is large enough, every call to `claim_rewards` (and `pool_member_info_v1`) will exceed the Starknet transaction gas limit and revert, permanently freezing the delegator's unclaimed yield with no recovery path.

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains the following loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop variable `entry_to_claim_from` starts from `pool_member_info.entry_to_claim_from` (the last claimed index, stored in `InternalPoolMemberInfoV1`) and runs to `pool_member_trace_length` (the total number of entries in the member's `pool_member_epoch_balance` trace). [2](#0-1) 

Each call to `add_to_delegation_pool` (or `enter_delegation_pool`) invokes `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. Because the trace only deduplicates entries with the same epoch key, each call in a **different epoch** appends a new checkpoint: [3](#0-2) [4](#0-3) 

The `entry_to_claim_from` counter is only advanced when `claim_rewards` is successfully called and the updated value is written back to storage: [5](#0-4) 

If a delegator never calls `claim_rewards`, `entry_to_claim_from` stays at 0 and the loop must traverse the entire accumulated trace on every future claim attempt.

Inside each loop iteration, `find_sigma` is called, which itself reads up to 3 entries from the `cumulative_rewards_trace` storage vector: [6](#0-5) 

Each storage read on Starknet is expensive in terms of gas. With N balance trace entries, the loop performs O(N) storage reads plus O(N) `find_sigma` calls (each reading up to 3 more storage slots), making the total cost O(N) storage operations. Beyond a few thousand entries, this will exceed the Starknet per-transaction gas limit.

The same `calculate_rewards` call path is also triggered by the view function `pool_member_info_v1`, meaning even reading the member's state becomes impossible: [7](#0-6) 

### Impact Explanation

Once the balance trace grows beyond the gas-limit threshold, every call to `claim_rewards` and `pool_member_info_v1` for that pool member will always revert. The delegator's accumulated unclaimed yield is permanently frozen inside the pool contract with no recovery mechanism. There is no partial-claim parameter, no way to reset `entry_to_claim_from` externally, and no emergency withdrawal path for rewards. The principal (staked tokens) can still be recovered via `exit_delegation_pool_intent` / `exit_delegation_pool_action`, but all accrued STRK rewards are permanently lost.

**Impact**: Permanent freezing of unclaimed yield — matches the allowed High impact category.

### Likelihood Explanation

Any delegator who calls `add_to_delegation_pool` in a different epoch each time without claiming rewards will grow their trace. This is a realistic pattern for:

- Automated compounding bots that top up delegation each epoch
- Delegators who add small amounts frequently (e.g., daily) over months or years without claiming

The Starknet epoch length is configurable and can be short. The developers themselves acknowledge the risk in the code comment: *"This loop is unbounded but unlikely to exceed gas limits."* The dismissal mirrors the Aura Finance judge's reasoning that was ultimately accepted as Medium severity. On Starknet, where storage reads are particularly expensive in gas, the threshold for failure is lower than on EVM chains.

**Likelihood**: Medium — requires sustained use of `add_to_delegation_pool` across many epochs without claiming, which is a natural pattern for active delegators.

### Recommendation

1. **Introduce a `max_entries` parameter** to `claim_rewards` so a delegator can claim rewards for a bounded number of balance trace entries per transaction, advancing `entry_to_claim_from` incrementally.
2. **Alternatively**, enforce a maximum trace length by requiring delegators to claim rewards before adding to the pool if the trace exceeds a threshold.
3. **Document the risk** clearly and recommend that delegators claim rewards before the trace grows large.

### Proof of Concept

1. Delegator calls `enter_delegation_pool` at epoch E₀ → trace length = 1, `entry_to_claim_from = 0`.
2. Delegator calls `add_to_delegation_pool` at epochs E₁, E₂, …, Eₙ (one call per epoch, never claiming) → trace length = N+1, `entry_to_claim_from = 0`.
3. After N epochs, delegator calls `claim_rewards`. The loop in `calculate_rewards` must iterate from index 0 to N, performing O(N) storage reads plus O(N) `find_sigma` calls (each reading up to 3 storage slots from `cumulative_rewards_trace`).
4. For sufficiently large N (on the order of thousands of epochs, achievable in months with short epoch durations), the transaction exceeds the Starknet gas limit and reverts.
5. Every subsequent `claim_rewards` call also reverts. The delegator's unclaimed yield is permanently frozen. [1](#0-0) [8](#0-7) [9](#0-8)

pub(crate) fn find_sigma_standard_case(
    cumulative_rewards_trace_vec: StorageBase<Trace>,
    cumulative_rewards_trace_idx: VecIndex,
    target_epoch: Epoch,
) -> Amount {
    // Three entries in the cumulative rewards trace are relevant (idx, idx - 1, idx - 2).
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx);
    if epoch < target_epoch {
        return sigma;
    }
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx - 1);
    if epoch < target_epoch {
        return sigma;
    }
    // Note: When handling a checkpoint from the old version, it never reaches here.
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx - 2);
    assert!(epoch < target_epoch, "{}", InternalError::INVALID_EPOCH_IN_TRACE);
    sigma
}
