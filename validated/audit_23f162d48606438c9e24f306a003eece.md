### Title
`MsgUpdateTokenMapping.ValidateBasic()` Blocks the Only Direct Path to Delete a Token Mapping, Permanently Locking Incorrect Mappings - (File: `x/cronos/types/messages.go`)

### Summary

`MsgUpdateTokenMapping.ValidateBasic()` unconditionally rejects an empty `Contract` field, but `RegisterOrUpdateTokenMapping` in the keeper explicitly interprets an empty `Contract` as a **delete** instruction for non-source-coin mappings. This is the exact same class of inconsistency as the external report: a guard at the outer layer blocks a legitimate operation that the inner layer was designed to support. The result is that the delete-mapping code path in the keeper is permanently unreachable via the only direct privileged message path, leaving any incorrectly registered token mapping impossible to remove without governance.

### Finding Description

**Outer layer — `ValidateBasic()` rejects empty contract unconditionally:** [1](#0-0) 

`common.IsHexAddress("")` returns `false` (go-ethereum requires exactly 40 hex chars after an optional `0x` prefix), so any `MsgUpdateTokenMapping` with an empty `Contract` field is rejected at the SDK ante-handler stage, before the keeper is ever reached.

**Inner layer — `RegisterOrUpdateTokenMapping` treats empty contract as a delete:** [2](#0-1) 

For non-source-coin denoms (`gravity…`, `ibc/…`), the keeper explicitly branches on `len(msg.Contract) == 0` to call `DeleteExternalContractForDenom`. This is the **only** keeper-level path to remove an external contract mapping.

**Message server — no bypass exists:** [3](#0-2) 

The gRPC handler calls `RegisterOrUpdateTokenMapping` directly after the permission check. Because `ValidateBasic()` runs before the handler is invoked (Cosmos SDK framework), the delete branch in the keeper is dead code from the perspective of `MsgUpdateTokenMapping`.

**Governance proposal path — also blocked:**

The legacy proposal handler (`NewTokenMappingChangeProposalHandler`) calls `c.ValidateBasic()` on the proposal before constructing the `MsgUpdateTokenMapping` struct and calling `RegisterOrUpdateTokenMapping`. [4](#0-3) 

`TokenMappingChangeProposal.ValidateBasic()` validates the `Contract` field. If it applies the same hex-address check (which is the natural expectation given the shared type), the governance path is equally blocked.

### Impact Explanation

If a token mapping is registered with an incorrect contract address (e.g., a wrong CRC21 contract for a `gravity0x…` or `ibc/…` denom), the CronosAdmin and any address holding `CanChangeTokenMapping` permission cannot delete it via `MsgUpdateTokenMapping`. The incorrect mapping persists in the KV store, causing every subsequent `ConvertVouchers` or IBC-receive event for that denom to mint tokens into the wrong CRC21 contract — permanently corrupting the denom↔contract binding and breaking bridge/conversion flows for all users of that asset.

This matches: **High — Corruption of token mappings, denom/contract binding with direct security impact** and **High — Permanent inability for honest users to process valid bridge/conversion flows**.

### Likelihood Explanation

The CronosAdmin is a privileged role, but the bug is triggered by a routine administrative action (correcting a wrong mapping). Any operator who registers a mapping with a typo or a stale contract address and then attempts to delete it via the standard message path will hit this silently. The delete branch in the keeper exists and is documented, so operators have a reasonable expectation that it works.

### Recommendation

In `MsgUpdateTokenMapping.ValidateBasic()`, allow an empty `Contract` field for non-source-coin denoms, mirroring the keeper's own semantics:

```go
func (msg *MsgUpdateTokenMapping) ValidateBasic() error {
    _, err := sdk.AccAddressFromBech32(msg.Sender)
    if err != nil {
        return errors.Wrapf(sdkerrors.ErrInvalidAddress, "invalid sender address (%s)", err)
    }
    if !IsValidCoinDenom(msg.Denom) {
        return errors.Wrapf(sdkerrors.ErrInvalidRequest, "invalid denom format (%s)", msg.Denom)
    }
    // Allow empty contract for non-source coins: empty == delete mapping
    if len(msg.Contract) > 0 && !common.IsHexAddress(msg.Contract) {
        return errors.Wrapf(sdkerrors.ErrInvalidRequest, "invalid contract address (%s)", msg.Contract)
    }
    // Source coins always require a valid contract (no deletion path)
    if IsSourceCoin(msg.Denom) && len(msg.Contract) == 0 {
        return errors.Wrapf(sdkerrors.ErrInvalidRequest, "contract address required for source denom")
    }
    return nil
}
```

Apply the same fix to `TokenMappingChangeProposal.ValidateBasic()` so the governance path is consistent.
