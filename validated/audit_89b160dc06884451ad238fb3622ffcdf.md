### Title
Missing Zero-Address Validation on `reward_address` Causes Permanent Freezing of Unclaimed Yield and Principal - (File: src/staking/staking.cairo)

### Summary
Neither `staking.cairo::stake()`, `staking.cairo::change_reward_address()`, nor `pool.cairo::change_reward_address()` validate that the supplied `reward_address` is non-zero. A staker or pool member who registers or later sets their reward address to `0x0` will have their unclaimed yield permanently frozen. Because `unstake_action` unconditionally calls `send_rewards_to_staker` before returning principal, any accumulated rewards cause the entire unstake to revert, permanently freezing the staker's principal as well.

### Finding Description
The vulnerability class from the external report is **missing input validation before using an address in a call that transfers value**. The analog here is that three entry points accept `reward_address` without a zero-address guard:

**`staking.cairo::stake()`** — validates only that `reward_address` is not a registered token address: [1](#0-0) 

**`staking.cairo::change_reward_address()`** — same single check, no zero guard: [2](#0-1) 

**`pool.cairo::change_reward_address()`** — checks only against the pool's token address: [3](#0-2) 

When rewards are later claimed or the staker exits, `send_rewards_to_staker` unconditionally calls `claim_from_reward_supplier` and then `checked_transfer(recipient: reward_address, amount: rewards)`: [4](#0-3) 

`unstake_action` calls `send_rewards_to_staker` **before** returning principal, with no fallback path if the transfer fails: [5](#0-4) 

If `reward_address` is `0x0` and `unclaimed_rewards_own > 0`, the ERC20 `transfer` to the zero address reverts (standard OpenZeppelin Cairo ERC20 behaviour), causing the entire `unstake_action` transaction to revert. The staker's principal is then permanently locked in the contract.

The same pattern applies in `pool.cairo::claim_rewards()`: [6](#0-5) 

### Impact Explanation
**HIGH — Permanent freezing of unclaimed yield and principal.**

A staker who registers with or later changes to `reward_address = 0x0` will:
1. Be unable to claim rewards (`claim_rewards` reverts on the zero-address transfer).
2. Be unable to recover their staked principal: `unstake_action` is callable by anyone but always reverts because `send_rewards_to_staker` runs first and fails, leaving the principal permanently locked in the staking contract.

### Likelihood Explanation
**LOW–MEDIUM.** The zero address is a natural default/uninitialized value in many off-chain integrations and scripts. A programmatic staker that fails to populate `reward_address` before calling `stake()`, or a UI bug that passes `0x0`, silently succeeds at registration time and only surfaces the freeze when the user attempts to claim or exit. The missing guard is the sole root cause; no privileged role or external dependency is required.

### Recommendation
Add an explicit non-zero assertion for `reward_address` in all three entry points:

- `staking.cairo::stake()` — add `assert!(reward_address.is_non_zero(), ...)` alongside the existing token check.
- `staking.cairo::change_reward_address()` — same guard.
- `pool.cairo::change_reward_address()` — same guard.

Optionally, add the same guard to `pool.cairo::enter_delegation_pool()` if it also accepts a caller-supplied `reward_address`.

### Proof of Concept
1. Staker calls `stake(reward_address: 0x0, operational_address: X, amount: MIN_STAKE)` — **succeeds** (no zero-address check).
2. Staker attests for one or more epochs; `unclaimed_rewards_own` becomes non-zero.
3. Staker calls `unstake_intent()` — succeeds, sets exit timestamp.
4. After the exit window, anyone calls `unstake_action(staker_address)`.
5. `unstake_action` calls `send_rewards_to_staker`, which calls `claim_from_reward_supplier` (pulls rewards into the staking contract) and then `checked_transfer(recipient: 0x0, amount: rewards)`.
6. The ERC20 transfer to the zero address **reverts**; the entire `unstake_action` reverts.
7. The staker's principal remains locked in the staking contract with no recovery path, because `reward_address` cannot be changed after `unstake_intent` is in progress (the staker record still exists but the exit window is running, and `change_reward_address` requires the staker to exist — however, the staker is stuck in the exit window and cannot re-stake or otherwise escape the frozen state without first successfully executing `unstake_action`, which always reverts).
