Looking at the `skip_stateful_validations` function and the `account_tx_in_pool_or_recent_block` check, I've found a real analog to the external bug.

The external bug: `takeMultipleOneOrders` skips the per-order `canExecTakeOneOrder` authorization check. The sequencer analog: `skip_stateful_validations` skips the per-transaction `__validate__` entry-point (signature) check, and the guard condition it relies on is too broad.

---

### Title
`skip_stateful_validations` Admits Unsigned Invoke Transactions for Any Account with a Pending Deploy-Account in the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is designed to improve UX for the deploy-account + invoke flow. It skips the `__validate__` entry-point call for an invoke transaction with `nonce=1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. However, `account_tx_in_pool_or_recent_block` checks for **any** transaction in the pool for that address — not specifically a deploy-account transaction. An unprivileged attacker who observes a victim's deploy-account transaction in the mempool can immediately submit an invoke transaction with `nonce=1` for the victim's address carrying an **arbitrary (invalid) signature**, and the gateway will admit it to the mempool without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip) when all four conditions hold:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

Condition 4 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` if **any** transaction for the address is in the pool — including a deploy-account transaction submitted by the legitimate owner. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but it does not distinguish transaction types.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is then skipped:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [4](#0-3) 

`perform_pre_validation_stage` (nonce, fee-bounds, balance, proof-facts) still runs, but the account's `__validate__` entry point — which is the only place the signature is verified — is never called. [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and fee levels; it performs no signature verification. [6](#0-5) 

### Impact Explanation

An attacker can inject an invoke transaction carrying an **invalid signature** into the mempool for any victim account that has a pending deploy-account transaction. This transaction occupies the `nonce=1` slot for the victim's address. When the batcher later executes it, `AccountTransaction::new_for_sequencing` sets `validate: true`:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        ...
        validate: true,
        strict_nonce_check: true,
    };
``` [7](#0-6) 

The `__validate__` call then fails and the transaction is rejected — but without charging fees (validation failure is non-revertible). The victim's own valid `nonce=1` invoke is blocked from the mempool until the attacker's transaction is evicted or replaced via fee escalation. Because the attacker can set the fee up to the victim's pre-funded balance, replacement may be economically infeasible for the victim.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- The attacker only needs to watch the public mempool for deploy-account transactions (trivially observable).
- No privileged access is required; any unprivileged caller of the gateway's `add_tx` endpoint can trigger this.
- The victim must have pre-funded the account address (standard practice for the deploy-account + invoke UX flow).
- The attack is limited to one invalid transaction per victim account per nonce slot, but that is sufficient to block the victim's first post-deploy invoke.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a type-specific check. The mempool should expose a dedicated `deploy_account_in_pool(address)` query that returns `true` only when a deploy-account transaction for that address is currently staged. `skip_stateful_validations` should call this narrower predicate instead:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await

// Use:
mempool_client.deploy_account_in_pool(tx.sender_address()).await
```

This ensures that the `__validate__` skip is granted only when the legitimate account owner has actually submitted a deploy-account transaction, not whenever any transaction for the address happens to be present.

### Proof of Concept

1. Alice pre-funds address `A` with STRK and submits a valid `DeployAccount` transaction. The mempool now contains Alice's deploy-account; `account_tx_in_pool_or_recent_block(A)` returns `true`.

2. Attacker Bob constructs an `Invoke` transaction for address `A` with `nonce=1` and a random (invalid) signature, setting `max_l2_gas_price` high enough to pass `validate_resource_bounds` and `verify_can_pay_committed_bounds` against Alice's pre-funded balance.

3. Bob submits the transaction to the gateway. The gateway:
   - Passes `validate_state_preconditions` (nonce `1 >= 0`, balance sufficient).
   - Passes `validate_by_mempool` (no duplicate, nonce in range).
   - Calls `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
   - Calls `run_validate_entry_point` with `skip_validate=true` → `validate: false` → `__validate__` is **never called**.
   - Bob's transaction is admitted to the mempool.

4. Alice's deploy-account executes; her account is now live at nonce `1`.

5. Alice submits a valid `Invoke` with `nonce=1`. The mempool rejects it: Bob's transaction already occupies `(A, nonce=1)`. Alice must pay a higher fee than Bob's to replace it; if Bob set the fee near Alice's full balance, replacement is economically blocked.

6. When the batcher eventually processes Bob's transaction, `__validate__` is called with `validate: true`, the invalid signature causes a `ValidateTransactionError`, the transaction is rejected with no fee charged to Bob, and the slot is freed — but Alice's UX has been disrupted.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
