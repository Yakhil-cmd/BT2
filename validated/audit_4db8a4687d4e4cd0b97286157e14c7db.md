### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Callers to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the `sender` argument (the actual payer/caller) and gates only on `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` does not require `msg.sender == owner`, any non-allowlisted address can call the pool directly with an allowlisted `owner`, pass the extension check, and deposit into a restricted pool.

### Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` from any caller without enforcing `msg.sender == owner` (unlike `removeLiquidity`, which does enforce this): [1](#0-0) 

It passes both `msg.sender` (as `sender`) and the caller-supplied `owner` to `_beforeAddLiquidity`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, ...)` but explicitly discards `sender` (unnamed first parameter) and checks only `owner`: [3](#0-2) 

The contract's own NatSpec states it "Gates `addLiquidity` by **depositor** address" and the mapping is named `allowedDepositor` — both pointing to the payer/caller, not the position recipient. The implementation contradicts this intent. [4](#0-3) 

### Impact Explanation

Any non-allowlisted address can call `pool.addLiquidity(allowlistedOwner, ...)` directly (bypassing the periphery's `_validateOwner`, which only checks for `address(0)`): [5](#0-4) 

The extension checks `allowedDepositor[pool][allowlistedOwner]` → `true` → passes. The non-allowlisted caller pays the tokens; the allowlisted owner receives the LP position. The deposit allowlist is fully defeated as an access control. This is an admin-boundary break: a pool admin-configured access control is bypassed by an unprivileged path.

### Likelihood Explanation

The attack path is a direct public call to `MetricOmmPool.addLiquidity` with a chosen `owner`. No special privileges, no flash loans, no oracle manipulation. Any address can execute it at any time against any pool using `DepositAllowlistExtension`.

### Recommendation

`DepositAllowlistExtension.beforeAddLiquidity` should check `sender` (the actual payer/caller), not `owner`:

```solidity
// current (wrong):
function beforeAddLiquidity(address, address owner, ...)
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) revert ...

// fix:
function beforeAddLiquidity(address sender, address, ...)
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) revert ...
```

### Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension` configured as a `beforeAddLiquidity` hook.
2. Allowlist `alice` via `setAllowedToDeposit(pool, alice, true)`. Do not allowlist `bob`.
3. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")` directly.
4. Extension receives `(bob, alice, ...)`, checks `allowedDepositor[pool][alice]` → `true` → does not revert.
5. `bob` pays tokens; `alice` receives the LP position. The allowlist is bypassed. [3](#0-2) [6](#0-5)
