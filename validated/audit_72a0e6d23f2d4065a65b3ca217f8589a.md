### Title
Gateway Stateful Validator Uses Previous Block's `l2_gas_price` Instead of `next_l2_gas_price` for Resource Bounds Admission Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the **current committed block's** `l2_gas_price` to gate incoming transactions, but the price that will actually govern execution is `next_l2_gas_price` (the EIP-1559-style fee-market price stored in `BlockHeaderWithoutHash.next_l2_gas_price`). The code even carries a self-documenting TODO acknowledging the mistake. The mismatch causes the gateway to admit transactions that will fail blockifier fee-bound checks during block building, and to reject transactions that would be perfectly valid for the next block.

---

### Finding Description

In `validate_resource_bounds`, the gateway fetches block info and extracts the STRK L2 gas price:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

`get_block_info()` is implemented by `GatewayFixedBlockSyncStateClient`, which reads `block_header.l2_gas_price` — the price **of the already-committed block** — and places it into `BlockInfo.gas_prices.strk_gas_prices.l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

The Starknet protocol stores a separate, forward-looking field `next_l2_gas_price` in `BlockHeaderWithoutHash`:

```rust
pub struct BlockHeaderWithoutHash {
    pub l2_gas_price: GasPricePerToken,
    pub l2_gas_consumed: GasAmount,
    pub next_l2_gas_price: GasPrice,   // ← the price for the NEXT block
    ...
}
``` [3](#0-2) 

`next_l2_gas_price` is computed by the EIP-1559-style fee market (`calculate_next_base_gas_price`) and is the price that the blockifier will use when it executes transactions in the next block. The `BlockInfo` struct returned by `get_block_info()` does **not** carry `next_l2_gas_price`; it is silently dropped during conversion in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`.

The same stale price is also propagated into `run_validate_entry_point`, which builds a `BlockContext` from the same `get_block_info()` call (only bumping the block number, not the gas price):

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
// gas_prices are still from the previous block
let block_context = BlockContext::new(block_info, ...);
``` [4](#0-3) 

The blockifier's `check_fee_bounds` (called during `perform_pre_validation_stage`) compares `resource_bounds.max_price_per_unit` against `block_info.gas_prices`, so the same wrong price is used there too. [5](#0-4) 

---

### Impact Explanation

**Scenario A — price rising (`next_l2_gas_price > l2_gas_price`):**

The gateway threshold is `min_gas_price_percentage% × l2_gas_price` (too low). A transaction with `max_price_per_unit` in the range `[threshold_old, next_l2_gas_price)` passes gateway admission and enters the mempool. When the batcher pulls it and the blockifier runs `check_fee_bounds` against the actual next-block price, the transaction fails with `MaxGasPriceTooLow`. The transaction was incorrectly admitted — **invalid transaction accepted by gateway/mempool**.

**Scenario B — price falling (`next_l2_gas_price < l2_gas_price`):**

The gateway threshold is too high. A transaction with `max_price_per_unit` in the range `[threshold_new, threshold_old)` is rejected at the gateway even though it would satisfy the actual next-block price. **Valid transaction rejected by gateway**.

Both scenarios fall squarely within the "High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing" impact category.

---

### Likelihood Explanation

The EIP-1559 fee market (`calculate_next_base_gas_price`) adjusts the L2 gas price every block based on gas consumption. Any block that is not exactly at the gas target will produce a `next_l2_gas_price` that differs from `l2_gas_price`. Under normal network load this happens continuously. The bug is therefore triggered on every transaction submitted to a live sequencer whenever `validate_resource_bounds` is enabled (the default: `validate_resource_bounds: true`, `min_gas_price_percentage: 100`). [6](#0-5) 

No privilege is required; any user submitting a standard V3 (`AllResources`) transaction triggers the path.

---

### Recommendation

Replace the `l2_gas_price` read with `next_l2_gas_price` from the block header. Because `BlockInfo` does not carry this field, either:

1. Extend `GatewayFixedBlockStateReader::get_block_info` to return `next_l2_gas_price` alongside the existing `BlockInfo`, or
2. Add a dedicated `get_next_l2_gas_price()` method to `GatewayFixedBlockStateReader` that reads `block_header.next_l2_gas_price` directly.

Then update `validate_resource_bounds`:

```rust
- // TODO(Arni): getnext_l2_gas_price from the block header.
- let previous_block_l2_gas_price = self
-     .gateway_fixed_block_state_reader
-     .get_block_info()
-     .await?
-     .gas_prices
-     .strk_gas_prices
-     .l2_gas_price;
+ let next_l2_gas_price = self
+     .gateway_fixed_block_state_reader
+     .get_next_l2_gas_price()
+     .await?;
  self.validate_tx_l2_gas_price_within_threshold(
      executable_tx.resource_bounds(),
-     previous_block_l2_gas_price,
+     next_l2_gas_price,
  )?;
```

Apply the same fix to the `BlockContext` constructed in `run_validate_entry_point` so that `block_info.gas_prices` reflects the next block's prices.

---

### Proof of Concept

1. Observe the current L2 gas price from the latest committed block header: `P_current`.
2. Compute the expected next-block price via the fee market: `P_next = calculate_next_base_gas_price(P_current, gas_used, gas_target, min_price)`. Assume `P_next > P_current` (block was over-target).
3. Submit a V3 Invoke transaction with `l2_gas.max_price_per_unit = P_current` (satisfies the gateway's threshold of `100% × P_current` but is below `P_next`).
4. The gateway's `validate_tx_l2_gas_price_within_threshold` computes `threshold = 100% × P_current = P_current` and passes the transaction.
5. The transaction enters the mempool.
6. The batcher builds the next block with `l2_gas_price = P_next`. The blockifier's `check_fee_bounds` finds `max_price_per_unit (P_current) < actual_gas_price (P_next)` and emits `ResourceBoundsError::MaxGasPriceTooLow`.
7. The transaction is reverted or dropped during block building — it was incorrectly admitted by the gateway.
