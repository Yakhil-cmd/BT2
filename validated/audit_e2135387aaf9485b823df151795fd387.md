Audit Report

## Title
Async Reentrancy in `Gtc::claim_neurons` and `Gtc::donate_account` — Guard Flags Set After Inter-Canister Await - (File: `rs/nns/gtc/src/lib.rs`)

## Summary

The GTC `claim_neurons` and `donate_account` functions set their respective guard flags (`has_claimed`, `has_donated`) only after inter-canister `await` points. On the Internet Computer, any `await` suspends the canister and allows new ingress messages to be processed. A Genesis Token holder can submit concurrent `claim_neurons` and `donate_account` calls that both pass their mutual-exclusion checks before either sets its guard flag, resulting in both `has_claimed` and `has_donated` being set to `true` simultaneously — a direct violation of the GTC's invariant — and in the worst ordering, causing a neuron to be both claimed (controller transferred to caller) and donated (stake transferred to custodian, neuron deleted from Governance).

## Finding Description

**Root cause in `claim_neurons`** (`rs/nns/gtc/src/lib.rs`, lines 62–68):

```
if account.has_claimed { return Ok(...); }          // guard check
GovernanceCanister::claim_gtc_neurons(...).await?;  // SUSPEND — new messages processed here
account.has_claimed = true;                         // guard set AFTER await
```

**Root cause in `donate_account`** (`rs/nns/gtc/src/lib.rs`, lines 89–90):

```
account.transfer(custodian_neuron_id).await?;  // SUSPEND — no has_donated check before this
account.has_donated = true;                    // guard set AFTER await
```

`AccountState::transfer` (`rs/nns/gtc/src/lib.rs`, lines 174–183) does check `has_claimed`, `has_donated`, and `has_forwarded` synchronously before its own loop of per-neuron `transfer_gtc_neuron` awaits. However, these checks are only valid at the moment `transfer` is entered; they do not prevent a concurrent `claim_neurons` from passing its own guard and issuing `claim_gtc_neurons` to Governance while `transfer` is suspended mid-loop.

**Exploit flow for Scenario B (claim + donate simultaneously):**

1. Attacker holds a GTC account with all flags false.
2. M1 (`claim_neurons`): passes `has_donated`, `has_forwarded`, `has_claimed` checks → sends `claim_gtc_neurons` to Governance → GTC suspends.
3. M2 (`donate_account`): `account.transfer()` checks `has_claimed` (still false), `has_donated` (still false) → passes → sends `transfer_gtc_neuron(N1, custodian)` to Governance → GTC suspends.
4. **Ordering A in Governance** — `claim_gtc_neurons` processed first (synchronous, atomic): changes controller of all neurons from GTC to caller. Then `transfer_gtc_neuron(N1)` arrives: checks `is_donor_controlled_by_gtc` → fails (controller is now caller) → returns `PreconditionFailed`. GTC M1 callback: `has_claimed = true`. GTC M2 callback: records N1 as failed, loop continues (all remaining neurons also fail). `transfer()` returns `Ok(())` unconditionally. `has_donated = true`. **Result: both flags true; neurons are claimed but GTC records them as also donated.**
5. **Ordering B in Governance** — `transfer_gtc_neuron(N1)` processed first: checks controller (GTC) → passes → awaits ledger. During ledger await, Governance processes `claim_gtc_neurons`: changes controller of all neurons to caller. Ledger returns: N1 is deleted, custodian stake increased. GTC M1 callback: `has_claimed = true`. GTC M2 callback for N1: success, N1 removed from `neuron_ids`. Loop continues for N2…Nn: `transfer_gtc_neuron` fails (controller no longer GTC). `transfer()` returns `Ok(())`. `has_donated = true`. **Result: N1 is deleted (stake donated) while GTC records it as claimed; N2…Nn are claimed but GTC records them as also donated. Caller "claimed" a neuron that no longer exists in Governance.**

The `transfer()` function's unconditional `Ok(())` return (line 209) means `has_donated` is set regardless of how many individual neuron transfers failed, amplifying the state corruption.

## Impact Explanation

This falls under **High ($2,000–$10,000): Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints.**

Concrete harms:
- Both `has_claimed` and `has_donated` are set to `true` simultaneously, permanently corrupting the GTC's authoritative settlement record for the affected account.
- In Ordering B, one or more neurons are deleted from Governance (stake transferred to custodian) while the GTC records them as claimed. The caller loses those neurons permanently — they no longer exist in Governance — while the GTC state falsely indicates a successful claim.
- The GTC's `AccountState` proto explicitly models these flags as mutually exclusive terminal states; violating this invariant means the GTC can no longer be used as a reliable record of which accounts have been settled.

## Likelihood Explanation

The attacker must be a Genesis Token holder (owns a GTC account with all flags false). No other privilege is required. The attack requires only two concurrent ingress messages submitted before the first `await` returns. The GTC and Governance are on the same NNS subnet, so the inter-canister round-trip latency is short but non-zero and spans multiple IC rounds, providing a reliable window. The attack is repeatable for any account that has not yet been settled.

## Recommendation

Apply the check-effect-interaction pattern: set guard flags **before** the inter-canister call, and roll back on failure.

For `claim_neurons`:
```rust
account.has_claimed = true;  // set BEFORE await
let result = GovernanceCanister::claim_gtc_neurons(caller, account.neuron_ids.clone()).await;
if result.is_err() {
    account.has_claimed = false;  // roll back on failure
    return Err(...);
}
```

For `donate_account`, add a `has_donated = true` guard before calling `account.transfer(...)`, and roll back if `transfer` returns an error.

Alternatively, adopt the `LedgerUpdateLock` / `NeuronAsyncLock` pattern already used in NNS Governance (`rs/nns/governance/src/neuron_lock.rs`) to hold an in-flight lock per account for the duration of any async operation.

## Proof of Concept

A deterministic PocketIC or StateMachine integration test can reproduce this:

1. Initialize NNS with a GTC account holding neurons N1…Nn, all flags false.
2. Submit ingress M1 (`claim_neurons`) and M2 (`donate_account`) for the same account in the same round, before advancing the state machine past the first inter-canister response.
3. Advance the state machine to process both Governance responses.
4. Assert: `account.has_claimed == true && account.has_donated == true` (invariant violated).
5. For Ordering B: assert that N1 no longer exists in Governance (deleted by donation) while `has_claimed` is true.

The existing `rs/nns/integration_tests/src/gtc.rs` test infrastructure (`state_machine_builder_for_nns_tests`, `claim_neurons`, `donate_accounts`) provides all scaffolding needed; the test need only submit both calls before ticking the state machine past the first callback.
