### Title
`TurnBridge` Handler Is a No-Op Stub With No Permission Check, Permanently Breaking the Bridge Circuit Breaker - (File: `x/cronos/keeper/msg_server.go`)

### Summary
`MsgTurnBridge` is the designated emergency circuit breaker for the Cronos bridge. Its handler implementation returns `nil, nil` unconditionally — it performs no permission check and changes no state. Any unprivileged caller can submit the message and receive a success response, while the bridge state is never actually modified. The bridge can never be disabled.

### Finding Description

The `TurnBridge` message handler in `msg_server.go` is a stub:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

The permission system defines `CanTurnBridge` as a distinct permission bit, and ADR-009 explicitly designates `MsgTurnBridge` as a restricted message requiring that permission: [2](#0-1) 

The `HasPermission` guard is correctly used for `UpdateTokenMapping` (line 73), but is entirely absent from `TurnBridge`: [3](#0-2) 

ADR-009 explicitly states the bridge circuit breaker must be callable from the ABCI end-blocker path and that disabling it completely is a required emergency capability: [4](#0-3) 

### Impact Explanation

Two distinct High-severity impacts:

1. **Auth bypass**: The `CanTurnBridge` permission check is completely absent. Any unprivileged address can submit `MsgTurnBridge` and receive a success response (`nil` error), bypassing the entire bridge authorization model.

2. **Permanent inability to disable bridge flows**: Because the handler never writes any state, the bridge can never be turned off. In an emergency (e.g., an active bridge exploit draining IBC vouchers or CRC20/CRC21 assets), the designated circuit breaker is permanently non-functional. Authorized parties — including the `CronosAdmin` multi-sig — cannot stop ongoing unauthorized asset transfers.

Both map directly to the allowed High impact scope: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks"* and *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows."*

### Likelihood Explanation

The stub is reachable by any unprivileged user via a standard Cosmos SDK transaction (`MsgTurnBridge`). No special privileges, leaked keys, or cryptographic assumptions are required. The bridge circuit breaker is silently broken in production.

### Recommendation

Implement `TurnBridge` with the required permission check and state write, mirroring the pattern used in `UpdateTokenMapping`:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    // persist bridge enabled/disabled state
    params := k.GetParams(ctx)
    params.EnableAutoDeployment = msg.Enable  // or a dedicated BridgeEnabled field
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &types.MsgTurnBridgeResponse{}, nil
}
```

### Proof of Concept

1. Any unprivileged address constructs `MsgTurnBridge{From: <attacker>, Enable: false}`.
2. Submits the transaction to the Cronos node.
3. The handler at `msg_server.go:85–87` returns `nil, nil` — the transaction is accepted on-chain with a success code.
4. No bridge state is written; the bridge remains enabled.
5. Repeat with `Enable: true` — same result. The bridge state is immutable regardless of what any authorized party submits.
6. During an active bridge exploit, the `CronosAdmin` multi-sig submits `MsgTurnBridge{Enable: false}` — it succeeds silently, the bridge keeps processing packets, and asset loss continues. [1](#0-0)
