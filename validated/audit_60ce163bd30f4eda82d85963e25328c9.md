### Title
Pool admin can instantaneously raise fees to the cap with no timelock, front-running swappers for direct principal loss — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`createPool` is permissionless: any address can deploy a pool and designate itself as pool admin. The pool admin can call `setPoolAdminFees` to raise the admin spread fee to `maxAdminSpreadFeeE6` (up to 20 %) and the notional fee to `maxAdminNotionalFeeE8` (up to 1 %) with **no timelock and no delay**. Because the pool's `spreadFeeE6` and `notionalFeeE8` storage slots are read live during every `swap` call, a malicious admin can atomically front-run a pending swap transaction, forcing the swapper to pay the maximum fee instead of the fee they observed when constructing their transaction.

---

### Finding Description

**Root cause — `setPoolAdminFees` has no timelock** [1](#0-0) 

The function validates only that the new values are within the factory-level caps, then immediately calls `setPoolFees` on the pool, which overwrites `spreadFeeE6` and `notionalFeeE8` in pool storage: [2](#0-1) 

Those two storage variables are read at swap time to compute the effective buy/sell prices: [3](#0-2) 

There is **no pending/delay mechanism** analogous to the price-provider timelock that exists for oracle rotation: [4](#0-3) 

**Permissionless pool creation makes the pool admin untrusted**

`createPool` is open to any caller: [5](#0-4) 

The `admin` field in `PoolParameters` is set by the caller, so the pool admin is whoever the pool creator chooses — including themselves. The factory caps (`maxAdminSpreadFeeE6 = 200_000`, `maxAdminNotionalFeeE8 = 1_000_000`) are the only constraint: [6](#0-5) 

**Additional uncapped vector — `setPoolBinAdditionalFees`**

The pool admin can also raise per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`, each `uint16` ≤ 65 535 in E6 ≈ 6.55 %) with no cap check and no timelock: [7](#0-6) 

These are added on top of the base spread fee during every swap step: [8](#0-7) 

Combined, the pool admin can push the effective fee to ≈ 26.5 % instantaneously.

---

### Impact Explanation

A swapper who reads `spreadFeeE6` / `notionalFeeE8` before submitting a transaction can be front-run in the same block: the admin raises fees to the maximum, the swapper's transaction executes at the new rate, and the swapper receives far less output than expected. The `priceLimitX64` parameter does not protect against fee increases — it constrains the marginal price, not the fee multiplier applied on top of it. The loss is bounded by the fee cap (≈ 26.5 % of the swap notional) but is unbounded in absolute dollar terms for large swaps. LPs are not harmed (they receive the extra fee), but swappers suffer direct principal loss.

---

### Likelihood Explanation

- `createPool` is permissionless; any address can become a pool admin.
- The fee change requires a single transaction from the pool admin — no special setup.
- Front-running is straightforward on any chain with a public mempool; on L2s with private sequencers the admin can simply raise fees in the block immediately before a known large swap.
- The attack is profitable for the admin (they receive the extra fee via `adminFeeDestination`) and requires no capital.

---

### Recommendation

**Short term:** Document that pools with a non-renounced admin carry the risk of instantaneous fee changes; surface the current admin address and fee history prominently in any UI.

**Long term:** Introduce a `feeChangeTimelock` per pool (analogous to `priceProviderTimelock`) so that fee increases are subject to a propose-then-execute delay, giving swappers time to observe the pending change and avoid the pool. Alternatively, enforce a maximum rate-of-change per block.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// 1. Malicious admin creates a pool with 0 % admin fee to attract swappers.
PoolParameters memory params = PoolParameters({
    ...
    adminSpreadFeeE6: 0,          // advertised as zero-fee pool
    adminNotionalFeeE8: 0,
    admin: address(this),         // attacker is admin
    ...
});
address pool = factory.createPool(params);

// 2. Victim submits a large swap transaction (pending in mempool).

// 3. Attacker front-runs: raises admin spread fee to the maximum in the same block.
factory.setPoolAdminFees(pool, 200_000, 1_000_000); // 20 % spread + 1 % notional

// 4. Victim's swap executes at 20 %+ fee instead of 0 %.
//    Victim receives ~20 % less output than expected.
//    Attacker collects the excess via adminFeeDestination.
```

The `setPoolAdminFees` call writes directly to `spreadFeeE6` / `notionalFeeE8` on the pool with no delay: [9](#0-8) 

and those values are consumed immediately by the next `swap`: [10](#0-9)
