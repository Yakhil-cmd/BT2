### Title
Permanent ETH Freeze on `ft_transfer` Failure Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` Cargo feature is **not** compiled in, an unprivileged EVM user can permanently destroy their own ETH by calling the `ExitToNear` precompile with `flag=0x0` targeting any NEAR account that lacks a NEP-141 storage deposit. The ETH is debited from the caller's EVM balance, the `ft_transfer` NEAR promise is scheduled without a callback, the promise fails silently, and no refund path exists.

---

### Finding Description

**Entrypoint:** Any EVM user calls the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with `flag=0x0` (base ETH transfer), a non-zero `msg.value`, and a recipient NEAR account ID that has no storage deposit registered with the ETH connector NEP-141 contract.

**Code path:**

1. `ExitToNear::run` parses `flag=0x0` → `ExitToNearParams::BaseToken` branch. [1](#0-0) 

2. `exit_base_token_to_near` with `message=None` returns `method="ft_transfer"` and `transfer_near_args=None`. [2](#0-1) 

3. `callback_args` is constructed. **Without `error_refund`**, `refund` is hardcoded to `None`; `transfer_near` is also `None`. This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`. [3](#0-2) 

4. The branch at line 470 detects `callback_args == default()` and schedules only a bare `PromiseArgs::Create` — **no callback is attached**. [4](#0-3) 

5. On NEAR, `ft_transfer` panics because the recipient has no storage deposit. Since there is no `exit_to_near_precompile_callback` scheduled, `engine::refund_on_error` is never called, and the ETH debited from the caller's EVM balance is never restored. [5](#0-4) 

**The `error_refund` feature is optional and absent from the default feature set:** [6](#0-5) [7](#0-6) 

---

### Impact Explanation

ETH is debited from the caller's EVM balance at EVM execution time. If the downstream NEAR `ft_transfer` promise fails and no callback is scheduled, the ETH is gone permanently — it is neither credited to the recipient nor returned to the sender. This is **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

- No privilege is required; any EVM account with ETH can trigger this.
- The failure condition (unregistered NEAR account) is trivially reachable — any account that has never called `storage_deposit` on the ETH connector will cause `ft_transfer` to fail.
- The behavior is **explicitly confirmed by the existing test suite**:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [8](#0-7) 

The same acknowledgment exists for the ERC-20 exit path: [9](#0-8) 

---

### Recommendation

1. **Enable `error_refund` unconditionally** in the production WASM build. The feature already implements the correct refund path via `refund_call_args` and `engine::refund_on_error`. [10](#0-9) [11](#0-10) 

2. Alternatively, remove the `#[cfg(not(feature = "error_refund"))]` guard and always populate `refund` for the base-token ETH exit path, making the callback unconditional.

3. Consider adding a pre-flight check that the recipient account is registered before scheduling the promise, reverting the EVM transaction early if not.

---

### Proof of Concept

The existing test `test_exit_to_near_eth_refund` already proves the scenario:

1. Fund a signer with `INITIAL_ETH_BALANCE` ETH on Aurora.
2. Call `withdrawEthToNear` (which invokes `ExitToNear` with `flag=0x0`) targeting `"any.near"` — an account with no NEP-141 storage deposit — with value `ETH_EXIT_AMOUNT`.
3. The `ft_transfer` promise fails (Aurora's NEP-141 balance was drained to force failure in the test, but an unregistered recipient achieves the same result).
4. Without `error_refund`, the test asserts:

```rust
assert_eq!(
    eth_balance_of(signer_address, &aurora).await,
    Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT)  // ETH is gone
);
``` [12](#0-11) 

ETH is debited, the recipient receives nothing, and no refund occurs — confirming permanent fund destruction.

#[cfg(feature = "error_refund")]
#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
