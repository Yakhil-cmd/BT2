### Title
Pool Admin Can Silently Overwrite a Pending Price-Provider Proposal and Reset the Timelock, Undermining User-Protection Guarantees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`proposePoolPriceProvider` contains no guard against re-proposal while a pending proposal already exists. The function unconditionally overwrites both `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]`, allowing the pool admin to silently substitute a different (potentially malicious) oracle for the one users already evaluated, while simultaneously resetting the timelock clock. This is the direct Metric OMM analog of the `BadDebtManager` bug: a privileged actor can modify a critical parameter (the pending oracle identity) after users have already acted on the original value, with no restriction on timing or number of modifications.

---

### Finding Description

`proposePoolPriceProvider` in `MetricOmmPoolFactory` is the only path through which a mutable-oracle pool can schedule an oracle rotation. The function enforces a timelock (`priceProviderTimelock[pool]`) whose stated purpose is to give users advance notice of an oracle change so they can exit before a new, potentially unfavorable oracle takes effect.

The function, however, contains no check that a proposal is not already pending:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 474-491
function proposePoolPriceProvider(address pool, address newPriceProvider)
    external override nonReentrant onlyPoolAdmin(pool)
{
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;          // ← unconditional overwrite
    pendingPriceProviderExecuteAfter[pool] = executeAfter;  // ← unconditional overwrite
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
}
```

The attack sequence:

1. Admin proposes oracle **A** (a legitimate, publicly known provider). Users observe the `PoolPriceProviderChangeProposed` event, evaluate A, and decide to remain in the pool.
2. Users stop monitoring, believing the only pending change is to A.
3. Just before the timelock for A expires, the admin calls `proposePoolPriceProvider` again with oracle **B** (a malicious provider that returns manipulated bid/ask prices). Both `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]` are silently overwritten. A new `PoolPriceProviderChangeProposed` event is emitted, but users who stopped monitoring never see it.
4. After the new timelock for B elapses, the admin calls `executePoolPriceProviderUpdate`, installing B as the active oracle.
5. Subsequent swaps read manipulated bid/ask prices from B, causing LPs to receive less than the oracle-fair amount for their liquidity.

The `executePoolPriceProviderUpdate` path does re-validate the pending provider at execution time, but that validation only checks token-pair matching — it cannot detect that the provider returns manipulated prices. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Once a malicious oracle is installed, every swap in the pool reads bid/ask prices from it. The pool's swap math trusts the oracle output unconditionally:

```solidity
// MetricOmmPool.sol — buy-token0 path (line 910)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
```

A manipulated oracle can return an artificially low ask price (token0 cheaper than market) or an artificially high bid price (token1 cheaper than market), allowing the admin or a colluding trader to drain LP-owned token reserves at below-market rates. This is a direct loss of LP principal — the pool's token balances no longer cover LP claims at fair value. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

- **Trigger**: The pool admin — explicitly classified as semi-trusted, with oracle-proposal actions listed in scope — is the only required actor.
- **No external dependency**: The attack requires only two `proposePoolPriceProvider` calls and one `executePoolPriceProviderUpdate` call, all callable by the admin alone.
- **Stealth**: The substitution event is on-chain but easily missed by users who stopped monitoring after evaluating the first proposal. There is no on-chain mechanism that alerts users that the pending oracle identity changed.
- **Timelock does not protect**: The timelock bounds *when* execution can happen, not *how many times* or *to what oracle* the pending slot can be changed. The admin can keep re-proposing to extend the window of deception indefinitely. [5](#0-4) 

---

### Recommendation

Add a guard in `proposePoolPriceProvider` that prevents overwriting an existing pending proposal:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PendingPriceProviderExists();
```

Alternatively, require the admin to explicitly cancel the current proposal (emitting a cancellation event that users can monitor) before a new one can be submitted. This mirrors the pattern already used for admin transfers (`cancelPoolAdminTransfer`), which requires an explicit cancellation step before a new nominee can be set. [6](#0-5)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Scenario: pool with priceProviderTimelock = 7 days
// Step 1 – Admin proposes legitimate oracle A
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(oracleA));
// Users see PoolPriceProviderChangeProposed(pool, current, oracleA, t+7days)
// Users evaluate oracleA, decide to stay, stop monitoring.

// Step 2 – Just before t+7days, admin silently substitutes malicious oracle B
vm.warp(block.timestamp + 7 days - 1);
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(oracleB));
// pendingPriceProvider[pool] is now oracleB
// pendingPriceProviderExecuteAfter[pool] is now block.timestamp + 7 days (reset)
// Users who stopped monitoring never see the new PoolPriceProviderChangeProposed event.

// Step 3 – After new timelock elapses, admin installs malicious oracle
vm.warp(block.timestamp + 7 days);
vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool);
// oracleB is now the active price provider
assertEq(PoolStateLibrary._slot3(pool), address(oracleB));

// Step 4 – Swaps now use oracleB's manipulated prices, draining LP reserves

    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
