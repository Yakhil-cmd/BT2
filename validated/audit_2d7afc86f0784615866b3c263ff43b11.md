### Title
COA Ownership Proof Signature Not Bound to `evmAddress` Enables Cross-COA Identity Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account key signatures are valid over caller-supplied `signedData`, then checks that the COA resource at the supplied `path` matches the supplied `evmAddress`. However, the function never enforces that `signedData` encodes `evmAddress`. A signature produced to prove ownership of one COA (EVM address) on a Cadence account can be replayed verbatim to prove ownership of any other COA on the same Cadence account, because the signed bytes are identical and the path/evmAddress arguments are attacker-controlled.

### Finding Description

`EVM.validateCOAOwnershipProof` is an `access(all)` function on the system EVM contract. Its signature is:

```cadence
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
``` [1](#0-0) 

The function:
1. Verifies that `signatures` over `signedData` are valid for the keys of the Cadence account at `address`.
2. Borrows the COA resource at `path` and checks its `.address().bytes` against `evmAddress`. [2](#0-1) 

At no point does the function verify that `signedData` contains `evmAddress`. The code itself acknowledges this gap:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [3](#0-2) 

The `path` and `evmAddress` arguments are fully attacker-controlled. If a Cadence account holds two COA resources (e.g., at `/public/coa1` and `/public/coa2`), a signature over any `signedData` that was produced to prove ownership of COA1 can be submitted with `path=/public/coa2` and `evmAddress=<COA2 bytes>` to obtain a `ValidationResult{isValid: true}` for COA2.

The ERC-1271 `isValidSignature` implementation in the COA contract calls `verifyCOAOwnershipProof(address(this), _hash, _sig)` where `_hash` is the hash supplied by the verifier and `_sig` is the encoded proof:

```solidity
(bool ok, bytes memory data) = cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig));
``` [4](#0-3) 

The EVM precompile `proofVerifier.Run` decodes the ABI-encoded call and delegates to `coaOwnershipProofValidator`, which invokes `validateCOAOwnershipProof` on the Cadence side: [5](#0-4) [6](#0-5) 

The `COAOwnershipProof` struct embedded in the proof carries `CapabilityPath` (the storage path of the COA resource), which is also attacker-controlled: [7](#0-6) 

### Impact Explanation

An attacker who observes a valid `(signedData, signatures, keyIndices)` tuple that a victim used to prove ownership of COA1 can replay it with a different `path` and `evmAddress` to fraudulently prove ownership of COA2 on the same Cadence account. Any EVM-side protocol relying on ERC-1271 (`isValidSignature`) or on direct calls to `validateCOAOwnershipProof` for authentication or authorization (e.g., NFT marketplaces, DeFi protocols, off-chain authentication flows) would accept the attacker's claim as the legitimate owner of COA2. This constitutes unauthorized EVM identity impersonation and could lead to unauthorized access to EVM-side assets or privileged actions gated behind COA ownership checks.

### Likelihood Explanation

The attack requires a victim whose Cadence account holds at least two COA resources at distinct public paths, and the attacker must have observed a valid proof (e.g., from a public transaction, an API response, or a dApp authentication flow). While "Flow wallets historically create at most one COA per account" (per the code comment), the protocol imposes no such limit, and multi-COA accounts are a supported and documented pattern. As the EVM-on-Flow ecosystem matures and dApps begin using `isValidSignature` for authentication, the preconditions become increasingly realistic. The function is `access(all)` and callable from any Cadence script or transaction without any privilege.

### Recommendation

Enforce that `signedData` cryptographically commits to `evmAddress`. The simplest fix is to require callers to include the target `evmAddress` bytes within `signedData` and add an on-chain assertion:

```cadence
// Inside validateCOAOwnershipProof, after verifying signatures:
// Require that signedData encodes evmAddress to prevent cross-COA replay.
var evmAddressFound = false
// ... check that evmAddress bytes appear in signedData ...
assert(evmAddressFound, message: "signedData must commit to evmAddress")
```

Alternatively, the function can internally construct the message to sign as `hash(evmAddress || callerSuppliedData)` and verify signatures against that, so the `evmAddress` is always part of the signed payload regardless of what the caller passes as `signedData`.

### Proof of Concept

1. Victim account `A` holds two COAs: COA1 at `/public/coa1` (EVM address `0xAAA...`) and COA2 at `/public/coa2` (EVM address `0xBBB...`).
2. A dApp challenges the victim to sign `challenge_bytes` to prove ownership of COA1. Victim signs and submits a transaction; the proof `{Address: A, CapabilityPath: "coa1", KeyIndices: [0], Signatures: [sig]}` is visible on-chain.
3. Attacker constructs a new proof: `{Address: A, CapabilityPath: "coa2", KeyIndices: [0], Signatures: [sig]}` (identical signatures, different path).
4. Attacker calls (from any Cadence script):
   ```cadence
   EVM.validateCOAOwnershipProof(
       address: A,
       path: /public/coa2,
       signedData: challenge_bytes,   // same bytes victim signed
       keyIndices: [0],
       signatures: [sig],             // victim's signature, unchanged
       evmAddress: 0xBBB...           // COA2's address
   )
   ```
5. Step 1 of the function: `keyList.verify(signatureSet, signedData: challenge_bytes, ...)` returns `true` — the signature is valid over `challenge_bytes` regardless of which COA is being claimed.
6. Step 2: `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coa2)` succeeds; `coaRef.address().bytes == 0xBBB...` matches the supplied `evmAddress`.
7. Function returns `ValidationResult{isValid: true}` for COA2, even though the victim never authorized this claim. [8](#0-7)
