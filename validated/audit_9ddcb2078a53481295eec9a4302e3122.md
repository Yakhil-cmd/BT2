### Title
`skip_stateful_validations` admits unvalidated invoke transactions by front-running a pending `deploy_account` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips `__validate__` for any invoke with `nonce=1` when `account_nonce=0` and `account_tx_in_pool_or_recent_block` returns true. The check uses a mempool presence signal that does not distinguish between a legitimate `deploy_account` and any other transaction type. An unprivileged attacker who observes a pending `deploy_account` in the mempool can front-run it by submitting an invoke with `nonce=1` carrying an arbitrary (invalid) signature, which bypasses the `__validate__` entry point and is admitted to the mempool.

### Finding Description

`skip_stateful_validations` is designed to improve UX for the simultaneous `deploy_account + invoke` flow. Its logic is:

```
if tx.nonce() == 1 && account_nonce == 0 {
    return account_tx_in_pool_or_recent_block(sender_address)
}
``` [1](#0-0) 

The comment asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* [2](#0-1) 

This assumption is broken. `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type in the pool for that address:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

When `skip_validate=true` is returned, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags` and calls `StatefulValidator::perform_validations`. For invoke transactions this path calls `perform_pre_validation_stage` (nonce/fee checks only) and then returns `Ok(())` **without** calling the `__validate__` entry point:

```rust
ApiTransaction::Invoke(_) => {
    tx.perform_pre_validation_stage(self.state(), &tx_context)?;
    if !tx.execution_flags.validate {
        return Ok(());   // __validate__ is never called
    }
    ...
}
``` [4](#0-3) 

`perform_pre_validation_stage` with `strict_nonce_check=false` accepts `tx_nonce=1` when `account_nonce=0` (non-strict: `account_nonce <= incoming_tx_nonce`):

```rust
let valid_nonce = if strict {
    account_nonce == incoming_tx_nonce
} else {
    account_nonce <= incoming_tx_nonce
};
``` [5](#0-4) 

The gateway sets `strict_nonce_check = false`: [6](#0-5) 

**Attack sequence:**

1. Victim funds address `X` and submits a `deploy_account` (nonce=0). It passes full gateway validation and enters the mempool.
2. Attacker observes the `deploy_account` in the mempool → `account_tx_in_pool_or_recent_block(X)` now returns `true`.
3. Attacker submits an invoke for address `X` with `nonce=1` and an **arbitrary/invalid signature**.
4. Gateway `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
5. `validate_by_mempool`: no existing nonce=1 entry for `X` → passes.
6. `skip_stateful_validations`: `nonce==1 && account_nonce==0 && pool_contains(X)` → returns `true`.
7. `run_validate_entry_point` with `validate=false`: `perform_pre_validation_stage` passes (non-strict nonce, funded balance), `__validate__` is **never called**.
8. Attacker's invoke with invalid signature is **admitted to the mempool**.
9. Victim's legitimate invoke with nonce=1 is rejected by the mempool as `DuplicateNonce`.
10. At block execution: `deploy_account` executes (nonce→1), then attacker's invoke executes with `validate=true` (full execution path), `__validate__` fails, transaction reverts, nonce incremented to 2, fee charged to victim's account.
11. Victim must resubmit with nonce=2; attacker can repeat.

### Impact Explanation

An invoke transaction carrying an invalid (attacker-chosen) signature bypasses the `__validate__` entry point and is admitted to the mempool. This directly violates the admission invariant: **every accepted invoke must have passed account signature validation**. Concretely:

- The victim's first post-deploy invoke is displaced from the mempool.
- The victim's balance is charged the execution fee for the attacker's reverted transaction.
- The attack is repeatable at each subsequent nonce if the attacker can keep front-running.

This matches: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The mempool is publicly observable (P2P gossip or direct RPC). Any `deploy_account` transaction is visible before it is committed. The attacker needs only to submit a well-formed invoke (valid resource bounds, funded sender) with an arbitrary signature before the victim's own nonce=1 invoke arrives. No privileged access is required.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that returns `true` only when a `deploy_account` transaction is present in the pool for the given address. For example, expose `deploy_account_in_pool(address: ContractAddress) -> bool` from the mempool and use it in `skip_stateful_validations`. This preserves the UX intent (simultaneous `deploy_account + invoke`) while closing the front-running window.

```rust
// Proposed fix in skip_stateful_validations:
return mempool_client
    .deploy_account_in_pool(tx.sender_address())   // type-specific check
    .await
    ...
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
