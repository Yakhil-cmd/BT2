### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Silently Overcharges Users When Native Resource Consumption Is Below EVM Gas Consumption — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` applies a one-sided correction to `gas_used` based on the difference between native resource consumption and EVM gas consumption (`delta_gas`). When `delta_gas > 0` (native more expensive than gas), `gas_used` is increased and the user pays more. When `delta_gas < 0` (native cheaper than gas), `gas_used` is **not** decreased, so the user is overcharged. The excess fee is transferred to the operator rather than refunded to the sender. A `// TODO: return delta_gas to gas_used?` comment at the exact branch confirms this is an acknowledged but unresolved gap.

---

### Finding Description

In `compute_gas_refund`, after computing `gas_used` from remaining ergs and applying EIP-3529 refunds, the function computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
```

Only the positive branch is handled:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The negative branch — where native resource consumption is less than what EVM gas consumption implies — is silently ignored. The `gas_used` value returned from this function is then used directly in `refund_and_commit_fee` to compute:

- `token_to_refund = gas_price × (gas_limit − gas_used)` — the amount returned to the sender
- `token_to_pay_operator = gas_used × gas_price_for_operator` — the amount paid to the coinbase

When `delta_gas < 0`, `gas_used` is inflated by `|delta_gas|` relative to what it should be. The user receives `|delta_gas| × gas_price` fewer tokens in their refund, and the operator receives that same excess.

The fee pre-charge path confirms the full `gas_limit × gas_price` is deducted from the sender upfront in `precharge_fee`, so the user has already paid the maximum. The refund is the only mechanism to return the unused portion. When `gas_used` is not reduced for the negative `delta_gas` case, the excess is permanently lost to the user.

---

### Impact Explanation

Every L2 transaction (ZK flow) and every Ethereum-flow transaction processed through `compute_gas_refund` with `native_per_gas > 0` and `delta_gas < 0` results in the user being overcharged by:

```
overcharge = |delta_gas| × gas_price
           = (gas_used − native_used / native_per_gas) × gas_price
```

This amount is transferred to the operator (or burned under `burn_base_fee`). It is never returned to the sender. For a transaction with `gas_limit = 1,000,000`, `gas_price = 1000`, and `delta_gas = −50,000`, the overcharge is `50,000,000` native tokens per transaction. This is a direct, quantifiable financial loss to every affected user.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever native resource consumption per gas unit is less than `native_per_gas = ceil(gas_price / native_price)`. This is the **common case** for transactions that are EVM-gas-heavy but proving-light (e.g., large calldata, many storage reads, complex arithmetic). Any user submitting a standard EVM transaction at a gas price above the minimum required to cover native costs will trigger this path. No special attacker action is required — the overcharge is structural and affects every qualifying transaction automatically.

---

### Recommendation

Apply the `delta_gas` correction symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, subject to the `minimal_gas_used` floor:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction);
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

This ensures the user is charged only for the resources actually consumed, whichever dimension (EVM gas or native) is the binding constraint.

### Proof of Concept

**Setup:**
- `gas_limit = 200_000`
- `gas_price = 2000` (native_price = 100, so `native_per_gas = 20`)
- `native_limit = 200_000 × 20 = 4_000_000`
- Transaction executes and consumes `gas_used_from_ergs = 100_000` EVM gas
- Native resource consumed: `native_used = 1_000_000` (only 50,000 gas-equivalent)

**In `compute_gas_refund`:**

```
gas_used = 100_000  (from ergs)
native_used = 4_000_000 − remaining_native
delta_gas = (1_000_000 / 20) − 100_000 = 50_000 − 100_000 = −50_000
```

Since `delta_gas < 0`, the branch is skipped. `gas_used` remains `100_000`.

**In `refund_and_commit_fee`:**

```
token_to_refund   = 2000 × (200_000 − 100_000) = 200_000_000
token_to_operator = 100_000 × 2000             = 200_000_000
```

**Correct behavior** (if `delta_gas` were applied symmetrically):

```
gas_used_corrected = 100_000 − 50_000 = 50_000
token_to_refund   = 2000 × (200_000 − 50_000) = 300_000_000
token_to_operator = 50_000 × 2000             = 100_000_000
```
