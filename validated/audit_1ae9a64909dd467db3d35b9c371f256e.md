### Title
Unauthenticated Invoke Transaction Admitted to Mempool via Unverified `skip_stateful_validations` Condition — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry-point call for any Invoke transaction with `nonce == 1` when the sender address has *any* transaction present in the mempool or a recent block. Because the check uses the generic `account_tx_in_pool_or_recent_block` predicate rather than confirming a `DeployAccount` transaction specifically, an attacker who observes a legitimate `DeployAccount` in the mempool can immediately submit a crafted Invoke with `nonce = 1` carrying an arbitrary (invalid) signature and have it admitted to the mempool without any signature verification.

---

### Finding Description

**Invariant broken:** Every Invoke transaction submitted to the gateway must have its account signature verified via the `__validate__` entry point before being admitted to the mempool.

**Root cause — `skip_stateful_validations`:**

```rust

**The unverified predicate — `account_tx_in_pool_or_recent_block`:**

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

This returns `true` for **any** transaction from the address — it does not confirm that the transaction is specifically a `DeployAccount`. The code comment claims this is "sufficient," but it is not: it is the exact condition an attacker can satisfy by observing a legitimate `DeployAccount` in the mempool.

**Attack path:**

1. Legitimate user submits `DeployAccount` (nonce=0) for address `X`. It enters the mempool.
2. Attacker observes `X` in the mempool (public mempool state).
3. Attacker submits `Invoke` with `nonce=1` for address `X`, carrying an arbitrary/invalid signature and arbitrary calldata.
4. Gateway evaluates:
   - `account_nonce == 0` ✓ (account not yet deployed in state)
   - `tx.nonce() == 1` ✓
   - `account_tx_in_pool_or_recent_block(X)` → `true` ✓ (the legitimate `DeployAccount` is there)
5. `skip_stateful_validations` returns `true`; `validate: false` is set; `__validate__` is **never called**.
6. The attacker's Invoke is admitted to the mempool with no signature check.

**Effect on execution:** The batcher later executes the `DeployAccount` (nonce=0) first, deploying the account, then attempts the attacker's Invoke (nonce=1). At that point the blockifier runs `__validate__` normally and the transaction is rejected (invalid signature). However:

- If the attacker submitted with a higher tip than the legitimate user's Invoke, the mempool's fee-escalation logic replaces the legitimate Invoke with the attacker's. The legitimate user's Invoke is evicted from the mempool and must be resubmitted.
- The attacker can repeat this for every address that has a pending `DeployAccount`, flooding the mempool with unauthenticated transactions and disrupting the deploy-account + invoke UX at scale.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Any unprivileged attacker who can observe the public mempool can submit Invoke transactions with arbitrary signatures for any address that has a pending `DeployAccount`. These transactions bypass the gateway's signature-verification step and are admitted to the mempool. The attacker can use fee escalation to evict legitimate post-deploy Invokes, forcing users to resubmit and creating a reliable DoS vector against the deploy-account + invoke UX flow.

---

### Likelihood Explanation

The `DeployAccount` transactions are visible in the public mempool. The condition to trigger the bypass (`nonce == 1`, `account_nonce == 0`, address in mempool) is trivially detectable and reproducible by any network participant. No privileged access, special keys, or prior state is required.

---

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a predicate that specifically confirms a `DeployAccount` transaction exists in the mempool for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects transaction types, not just address presence. This mirrors the fix in the referenced EIP-1271 bug: the authority chain must be verified against the actual proxy owner, not just checked for existence.

Alternatively, always run `__validate__` at the gateway and rely on the blockifier's `strict_nonce_check = false` to handle the ordering race, accepting that the first Invoke may fail validation if the `DeployAccount` has not yet been processed — which is the safer invariant.

---

### Proof of Concept

```
1. Alice submits DeployAccount for address 0xALICE (nonce=0, valid signature).
   → Mempool accepts it. account_tx_in_pool_or_recent_block(0xALICE) == true.

2. Alice also submits Invoke(nonce=1, calldata=[transfer_to_alice], valid_sig).
   → Admitted normally (skip_validate=true, but valid sig anyway).

3. Attacker submits Invoke(nonce=1, calldata=[transfer_to_attacker], sig=0xDEADBEEF, tip=alice_tip+1).
   → Gateway: account_nonce=0, tx_nonce=1, account_tx_in_pool_or_recent_block(0xALICE)=true
   → skip_stateful_validations returns true
   → validate: false → __validate__ NOT called
   → Mempool fee-escalation: attacker's Invoke replaces Alice's Invoke (higher tip)
   → Alice's Invoke is evicted.

4. Batcher executes DeployAccount(0xALICE) → account deployed.
5. Batcher executes attacker's Invoke(nonce=1) → __validate__ runs → INVALID_SIGNATURE → rejected.
6. Alice's Invoke is gone. Alice must resubmit.
```

The attacker pays only the gas for the `DeployAccount` observation (zero cost) and the tip for the replacement Invoke (which is never executed, so no fee is charged on rejection). The attack is essentially free and repeatable.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
