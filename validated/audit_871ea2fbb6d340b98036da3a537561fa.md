### Title
Observer Node `--local-service-api-enabled` and `--event-query-mode` Can Be Set to Conflicting Modes Simultaneously — (`File: cmd/observer/node_builder/observer_builder.go`)

---

### Summary

The Observer node builder accepts two independent configuration flags — `--local-service-api-enabled` and `--event-query-mode` (or `--script-execution-mode` / `--tx-result-query-mode`) — that are mutually conflicting in their data-source semantics. When `--local-service-api-enabled=true` is set, it silently overrides the query mode flags at runtime to `IndexQueryModeLocalOnly`. However, there is no startup validation that prevents an operator from simultaneously setting `--event-query-mode=execution-nodes-only` (or any non-local mode) alongside `--local-service-api-enabled=true`. The two modes have incompatible data-source assumptions and the silent override means the node silently ignores the explicitly configured query mode, creating an inconsistent and unvalidated configuration state.

---

### Finding Description

In `cmd/observer/node_builder/observer_builder.go`, the `ValidateFlags` callback (lines 898–953) does **not** check for the conflict between `localServiceAPIEnabled` and the query mode flags (`EventQueryMode`, `ScriptExecutionMode`, `TxResultQueryMode`). [1](#0-0) 

Later, during RPC engine construction in `enqueueRPCServer()`, the `backendParams` struct is first populated with the user-configured query modes (e.g., `execution-nodes-only`), and then — if `localServiceAPIEnabled` is true — those modes are silently overwritten to `IndexQueryModeLocalOnly`: [2](#0-1) 

The two modes are fundamentally incompatible:
- `IndexQueryModeLocalOnly` reads from the local indexed register/event database, which is only populated when `--execution-data-indexing-enabled=true` and `--execution-data-sync-enabled=true` are active.
- `IndexQueryModeExecutionNodesOnly` (or `failover`) routes queries to remote Execution Nodes over gRPC.

The `--local-service-api-enabled` flag is registered as a plain boolean with no cross-validation against the query mode flags: [3](#0-2) 

By contrast, the Access node builder (`access_node_builder.go`) does validate some related combinations (e.g., `check-payer-balance` requires `execution-data-indexing-enabled`), but no equivalent guard exists for the observer's `localServiceAPIEnabled` vs. query mode conflict: [4](#0-3) 

---

### Impact Explanation

An operator (or a misconfigured deployment script) can start an Observer node with:
```
--local-service-api-enabled=true
--event-query-mode=execution-nodes-only
--execution-data-indexing-enabled=false
```

The node starts without error. The `localServiceAPIEnabled` override silently forces `EventQueryMode` to `IndexQueryModeLocalOnly` at runtime, but the local index is not populated (indexing is disabled). Every call to `GetEventsForBlockIDs`, `GetEventsForHeightRange`, `ExecuteScriptAt*`, or `GetAccount*` will fail with an out-of-range or not-initialized error for all block heights, because the `EventsIndex` and `ScriptExecutor` are uninitialized or empty.

The impact is **incorrect API responses served to all clients of the Observer node** — scripts return errors, event queries return empty or error results — without any indication at startup that the configuration is invalid. This is an API correctness failure reachable by any unprivileged Access API caller (gRPC or REST) who sends a valid `ExecuteScriptAtBlockHeight` or `GetEventsForHeightRange` request. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

Observer nodes are commonly deployed with `--local-service-api-enabled=true` to serve local indexed data. The `--event-query-mode` flag is independently configurable and defaults to `execution-nodes-only`. An operator who enables `localServiceAPIEnabled` without removing the default or explicit `event-query-mode=execution-nodes-only` flag will silently enter this broken state. The Access node builder already demonstrates awareness of similar dependency constraints (e.g., `check-payer-balance` requires indexing), confirming this is a known class of misconfiguration. The absence of the analogous guard for the observer is a concrete gap. [7](#0-6) 

---

### Recommendation

Add a cross-validation check in the Observer's `ValidateFlags` callback to enforce that when `--local-service-api-enabled=true`, the node must also have `--execution-data-indexing-enabled=true` and `--execution-data-sync-enabled=true`. Additionally, emit a warning or error if `--event-query-mode`, `--script-execution-mode`, or `--tx-result-query-mode` are explicitly set to a non-local mode while `--local-service-api-enabled=true`, since those flags will be silently overridden.

Example guard to add in `ValidateFlags`:
```go
if builder.localServiceAPIEnabled {
    if !builder.executionDataIndexingEnabled {
        return errors.New("execution-data-indexing-enabled must be set when local-service-api-enabled is true")
    }
    if !builder.executionDataSyncEnabled {
        return errors.New("execution-data-sync-enabled must be set when local-service-api-enabled is true")
    }
}
``` [1](#0-0) 

---

### Proof of Concept

1. Start an Observer node with:
   ```
   --local-service-api-enabled=true
   --event-query-mode=execution-nodes-only
   --execution-data-indexing-enabled=false
   --execution-data-sync-enabled=false
   ```
2. Node starts successfully with no error from `ValidateFlags`.
3. At runtime, `enqueueRPCServer` sets `backendParams.EventQueryMode = query_mode.IndexQueryModeLocalOnly` (line 2193), overriding the user-configured `execution-nodes-only`.
4. `NewEventsBackend` constructs a `LocalEventProvider` backed by an uninitialized `EventsIndex`.
5. Any unprivileged client calling `GetEventsForHeightRange` or `ExecuteScriptAtBlockHeight` via the Access API receives an error (index not initialized / out of range) for all block heights, despite the node appearing healthy. [8](#0-7) [5](#0-4)
