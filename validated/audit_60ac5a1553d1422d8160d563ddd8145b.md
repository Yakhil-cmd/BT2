### Title
Missing `Execute` Capability Grant for `ScheduledTransactionExecutor` on Non-Transient Networks Will Prevent Execution of All Scheduled Transactions - (`fvm/bootstrap.go`)

---

### Summary

The `deployFlowTransactionScheduler` function in `fvm/bootstrap.go` conditionally issues the `auth(FlowTransactionScheduler.Execute)` capability to the `ScheduledTransactionExecutor` account **only on transient (local/test) networks**. On non-transient networks (Mainnet, Testnet), this capability is never issued during bootstrap, meaning the `ScheduledTransactionExecutor` account has no way to borrow the `SharedScheduler` with `Execute` entitlement and all scheduled transaction execution will permanently fail with a panic.

---

### Finding Description

In `fvm/bootstrap.go`, the `deployFlowTransactionScheduler` function deploys the `FlowTransactionScheduler` and `FlowTransactionSchedulerUtils` contracts, then issues the `Execute`-entitled capability to the `ScheduledTransactionExecutor` account — but only when `b.ctx.Chain.ChainID().Transient()` is true:

```go
// fvm/bootstrap.go lines 837-850
// scheduled transactions wont work on newly bootstrapped non-transient networks
// TODO: JanezP this could be controlled by a flag instead of a hidden check here
if b.ctx.Chain.ChainID().Transient() {
    sc := systemcontracts.SystemContractsForChain(b.ctx.Chain.ChainID())
    txBody, err = blueprints.IssueScheduledTransactionExecutorTransaction(b.ctx.Chain, sc.ScheduledTransactionExecutor.Address)
    ...
    panicOnMetaInvokeErrf("failed to create scheduled transaction executor: %s", txError, err)
}
```

The `IssueScheduledTransactionExecutorTransaction` function runs the `issueScheduledTransactionExecutorTemplate.cdc` script, which issues the `auth(FlowTransactionScheduler.Execute) &FlowTransactionScheduler.SharedScheduler` capability from the service account's storage and saves it to `/storage/executeScheduledTransactionsCapability` in the executor account:

```cadence
// fvm/blueprints/scripts/issueScheduledTransactionExecutorTemplate.cdc
transaction {
    prepare(serviceAccount: auth(Capabilities) &Account, capabilityAccount: auth(Storage) &Account) {
        let capability = serviceAccount.capabilities.storage.issue<auth(FlowTransactionScheduler.Execute) &FlowTransactionScheduler.SharedScheduler>(/storage/sharedScheduler)
        capabilityAccount.storage.save(capability, to: /storage/executeScheduledTransactionsCapability)
    }
}
```

The v1 system collection's `ExecuteCallbacksTransactions` (in `fvm/blueprints/scheduled_callback.go`) generates execute-callback transactions authorized by `sc.ScheduledTransactionExecutor.Address`. The corresponding Cadence transaction script (`GenerateSchedulerExecutorTransactionScript`) borrows the `SharedScheduler` with `Execute` entitlement from the executor account's storage. If the capability was never saved there, the borrow returns `nil` and the transaction panics.

The comment in the code explicitly acknowledges this: *"scheduled transactions wont work on newly bootstrapped non-transient networks"* — but this is not a configuration choice; it is a functional breakage. The `ScheduledTransactionExecutor` account exists on Mainnet (`0x45df3724e7c13957`) and Testnet (`0x9275945e651650bd`) per `system_contracts.go`, but the capability grant is skipped for both.

---

### Impact Explanation

**High**: On any non-transient network (Mainnet, Testnet) where `FlowTransactionScheduler` is deployed and `ScheduledTransactionsEnabled` is set to `true` in the FVM context, every block's system collection will include execute-callback transactions authorized by `ScheduledTransactionExecutor`. Each such transaction will attempt to borrow `auth(FlowTransactionScheduler.Execute) &FlowTransactionScheduler.SharedScheduler` from `/storage/executeScheduledTransactionsCapability` in the executor account. Since the capability was never issued and saved there during bootstrap, the borrow returns `nil` and the transaction panics. All scheduled transactions submitted by users will permanently fail to execute, and any FLOW fees paid by users to schedule transactions are lost without the scheduled work being performed.

---

### Likelihood Explanation

**High**: The condition `b.ctx.Chain.ChainID().Transient()` is `false` for both `flow.Mainnet` and `flow.Testnet`. The code path that issues the capability is therefore never executed on production networks. The comment in the source code itself confirms this is a known gap. The `ScheduledTransactionExecutor` accounts are already registered with fixed addresses on Mainnet and Testnet, indicating the feature is intended for production use.

---

### Recommendation

Remove the `if b.ctx.Chain.ChainID().Transient()` guard, or replace it with a flag that defaults to `true` for all networks where `FlowTransactionScheduler` is deployed. The `IssueScheduledTransactionExecutorTransaction` should be called unconditionally during bootstrap for any network that deploys `FlowTransactionScheduler`, so that the `ScheduledTransactionExecutor` account always holds the required `Execute` capability before the first block is produced.

---

### Proof of Concept

1. Bootstrap a non-transient network (Mainnet or Testnet configuration).
2. `deployFlowTransactionScheduler` is called in `bootstrapExecutor.Execute()` at `fvm/bootstrap.go:469`.
3. The `if b.ctx.Chain.ChainID().Transient()` branch at line 839 is **not entered** — `IssueScheduledTransactionExecutorTransaction` is never called.
4. The `ScheduledTransactionExecutor` account at `scheduledTransactionExecutorAddressMainnet` (`0x45df3724e7c13957`) has no capability stored at `/storage/executeScheduledTransactionsCapability`.
5. When `ScheduledTransactionsEnabled` is `true`, the FVM's system collection builder (`builderV1`) calls `ExecuteCallbacksTransactions`, which generates transactions authorized by `sc.ScheduledTransactionExecutor.Address` using `GenerateSchedulerExecutorTransactionScript`.
6. That script borrows `auth(FlowTransactionScheduler.Execute) &FlowTransactionScheduler.SharedScheduler` from the executor account's storage — the borrow returns `nil` — and the transaction panics: `"Could not borrow FlowTransactionScheduler"`.
7. Every scheduled transaction submitted by any user on the network permanently fails. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

func generateExecuteCallbacksTransaction(
	sc *systemcontracts.SystemContracts,
	script []byte,
	id uint64,
	effort uint64,
) (*flow.TransactionBody, error) {
	encID, err := jsoncdc.Encode(cadence.UInt64(id))
	if err != nil {
		return nil, fmt.Errorf("failed to encode id: %w", err)
	}

	tx, err := flow.NewTransactionBodyBuilder().
		AddAuthorizer(sc.ScheduledTransactionExecutor.Address).
		SetScript(script).
		AddArgument(encID).
		SetComputeLimit(effort).
		Build()
	if err != nil {
		return nil, fmt.Errorf("failed to construct execute callback transactions: %w", err)
	}

	return tx, nil
}
