### Title
WalletConnect `take_offer` Approval Displays Understated "Total Amount with Royalties" for Multi-NFT Offers — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the payment amount by the number of NFTs before computing each NFT's royalty, producing a displayed total that is systematically lower than the actual on-chain deduction. This understated figure is shown as "Total Amount with Royalties" in the WalletConnect `chia_wallet.take_offer` confirmation dialog, causing a user to approve a transaction believing they will spend less than the protocol will actually deduct.

---

### Finding Description

When a WalletConnect-connected dApp sends a `chia_wallet.take_offer` command, `parseCommandDisplay` calls `formatAmountWithRoyalties` to compute the total the taker will spend including creator royalties. The function is:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts  lines 354-375
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  ...
  const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by NFT count
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
``` [1](#0-0) 

The correct formula is: each NFT's royalty is `amount × royaltyPercentage_i / 10_000`. The code instead computes `(amount / N) × royaltyPercentage_i / 10_000` for each NFT, where N is the number of NFTs. The total royalty is therefore understated by a factor of `(N−1)/N`.

The existing test for the multi-NFT case confirms the bug is baked into the expected output:

```
// 2 NFTs: royalties 500 bp (5%) and 10 bp (0.1%), amount = 100_000_000 mojos
// Test expects amountWithRoyalties: '0.00010255'   ← buggy value
// Correct value:                    '0.0001051'
``` [2](#0-1) 

The understated `amountWithRoyalties` string is then rendered in the WalletConnect confirmation dialog under the "You Spend" section as "Total Amount with Royalties":

```tsx
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties} XCH
  </div>
)}
``` [3](#0-2) 

The dialog is the sole approval gate for the `chia_wallet.take_offer` WalletConnect command: [4](#0-3) 

---

### Impact Explanation

A user approves the offer seeing a "Total Amount with Royalties" that is lower than what the Chia protocol will actually deduct. Royalties are enforced at the protocol level and are automatically taken from the taker's wallet upon offer acceptance. The discrepancy scales with the number of NFTs and their royalty percentages.

**Concrete example** — 10 NFTs each with 10% royalty (1000 bp), payment = 1 XCH:
- Displayed: `1 + (1/10) × 10 × 10% = 1.1 XCH`
- Actual deduction: `1 + 10 × (1 × 10%) = 2 XCH`
- User is deceived by 0.9 XCH (45% of actual cost)

This matches the allowed High impact: *"WalletConnect state that causes a user to approve … the wrong … amount."*

---

### Likelihood Explanation

Any dApp with an active WalletConnect session can send a `chia_wallet.take_offer` request. No privileged access is required. The attacker constructs or selects an offer containing multiple NFTs with royalties — a valid and reachable offer type — and submits it via WalletConnect. The user sees the understated total and approves. The bug is triggered by the normal multi-NFT royalty code path, not an edge case.

---

### Recommendation

Replace the split-amount calculation with per-NFT royalties computed on the full amount:

```typescript
// Correct: each royalty is calculated on the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [5](#0-4) 

Update the corresponding test expectation in `parseCommandDisplay.test.ts` to reflect the corrected value (`'0.0001051'` instead of `'0.00010255'`). [6](#0-5) 

---

### Proof of Concept

1. Attacker creates or obtains an offer: pay 1 XCH for 2 NFTs, NFT-A has 10% royalty (1000 bp), NFT-B has 10% royalty (1000 bp).
2. Attacker connects a dApp to the victim's wallet via WalletConnect.
3. dApp sends `chia_wallet.take_offer` with the crafted offer string.
4. GUI calls `parseCommandDisplay` → `formatAmountWithRoyalties([1000, 1000], 1_000_000_000_000n)`:
   - `splitAmount = 1_000_000_000_000 / 2 = 500_000_000_000`
   - `royaltyAmount = (500_000_000_000 × 1000)/10_000 + (500_000_000_000 × 1000)/10_000 = 100_000_000_000`
   - Displayed total: `1.1 XCH`
5. Correct total: `1 + 0.1 + 0.1 = 1.2 XCH`
6. User sees "Total Amount with Royalties: 1.1 XCH", approves, and 1.2 XCH is deducted from their wallet. [1](#0-0) [7](#0-6)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
