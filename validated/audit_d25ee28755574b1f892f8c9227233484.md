### Title
L1 Transaction Processing DoS via Fatal Treatment of `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Revert — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function in `process_l1_transaction.rs` unconditionally escalates any revert from `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` into a fatal `BootloaderSubsystemError` that permanently halts block processing. This is the direct ZKsync OS analog of the external report's push-payment DoS: a single failing push-call to one recipient blocks all subsequent execution. Here the "recipient" is the L2AssetTracker system contract, and the "execution" is the entire block.

---

### Finding Description

During L1→L2 transaction processing, `process_l1_transaction` calls `mint_base_token` up to three times — once for the value mint to the sender (inside the execution frame), once for the operator fee payment to coinbase, and once for the refund to the refund recipient. [1](#0-0) 

Each `mint_base_token` call first invokes `notify_l2_asset_tracker`, which calls `handleFinalizeBaseTokenBridgingOnL2(l1_chain_id, amount)` on the L2AssetTracker contract: [2](#0-1) 

If the call returns `failed = true` (i.e., the EVM call reverted), the function immediately returns a fatal internal error:

```rust
return Err(internal_error!(
    "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
).into());
```

This `internal_error!` produces a `LeafDefect` variant of `BootloaderSubsystemError`. The error-handling path in `process_l1_transaction` only converts `RuntimeError::FatalRuntimeError` into a graceful revert; all other errors — including `LeafDefect` — propagate upward as fatal, halting the entire block: [3](#0-2) 

The `l1_chain_id` passed to the L2AssetTracker is read once from its own storage slot 154 before execution begins: [4](#0-3) 

The L2AssetTracker's `handleFinalizeBaseTokenBridgingOnL2` has multiple internal conditions that can cause it to revert: the asset must be registered (`isAssetRegistered[assetId]`), the migration number must match, and the settlement layer chain ID must be consistent. These are verified by the test predeploy setup: [5](#0-4) 

The comment in the code itself acknowledges the fatal nature of this design:

> "Failure halts block processing — if the asset tracker reverts, the chain's token accounting would be inconsistent, so we treat it as fatal rather than silently continuing with incorrect bookkeeping." [6](#0-5) 

This is the push-pattern DoS: a single external call that, if it reverts, blocks all L1 transaction processing — with no recovery path.

---

### Impact Explanation

Any L1 priority transaction with `to_mint > 0` (i.e., `reserved[0] > 0`) triggers `notify_l2_asset_tracker`. If the L2AssetTracker reverts for any reason — a migration state mismatch, an unregistered asset, a storage inconsistency introduced by the main transaction body, or a future upgrade that changes the contract's revert conditions — the entire block processing halts with a fatal error. No further L1 or L2 transactions in the block can be processed. This is a complete DoS of the ZKsync OS state transition function for any block containing such a transaction.

The impact is more severe than the external report: the external report blocked a single strategy's automation; here, the entire chain's block production is halted.

---

### Likelihood Explanation

The entry path requires only an unprivileged L1 transaction sender submitting a deposit transaction (`to_mint > 0`). The L2AssetTracker is a complex upgradeable contract. Conditions under which it reverts include:

1. **Migration state mismatch**: If `assetMigrationNumber[chainId][assetId]` is not set or mismatched (e.g., during a protocol upgrade), the call reverts.
2. **Unregistered asset**: If `isAssetRegistered[assetId]` is false (e.g., after a re-deployment or storage wipe), the call reverts.
3. **Chain ID inconsistency**: If `l1_chain_id` (slot 154) is 0 or stale relative to what the contract expects, the call reverts.
4. **Main body modifies L2AssetTracker state**: The `l1_chain_id` is read before execution. If the main transaction body calls a function on the L2AssetTracker that modifies its state (e.g., slot 154), the post-execution `notify_l2_asset_tracker` calls use the pre-execution `l1_chain_id`, potentially causing a mismatch and revert.

Scenario 4 is directly attacker-controlled: an L1 transaction sender can deploy a contract that, when called, modifies the L2AssetTracker's storage, causing the post-execution `notify_l2_asset_tracker` calls to revert and halt block processing.

---

### Recommendation

1. **Wrap `notify_l2_asset_tracker` in a recoverable error path** rather than treating its revert as a fatal block-halting error. Log the failure and continue with a fallback accounting strategy (e.g., skip the notification but still perform the treasury transfer).
2. **Alternatively, use a pull pattern**: accumulate the amounts to notify and allow the L2AssetTracker to pull them, rather than pushing notifications inline during block processing.
3. **Validate `l1_chain_id` before use**: If slot 154 is zero or inconsistent, skip the notification rather than propagating a fatal error.
4. **Isolate the L2AssetTracker call in a separate frame** with a rollback handle, so a revert only affects the notification, not the entire block.

---

### Proof of Concept

**Attacker-controlled path (scenario 4):**

1. Attacker deploys a contract `C` on L2 that, when called, writes to L2AssetTracker storage slot 154 (e.g., sets `L1_CHAIN_ID` to a value that causes `handleFinalizeBaseTokenBridgingOnL2` to revert on subsequent calls).
2. Attacker submits an L1 priority transaction:
   - `from`: attacker address
   - `to`: contract `C`
   - `to_mint` (`reserved[0]`): `gas_limit * gas_price + value` (non-zero deposit)
   - `value`: any non-zero amount
3. The bootloader processes the transaction:
   - `read_l1_chain_id` reads the current value from slot 154 (e.g., `1`).
   - `execute_l1_transaction_and_notify_result` calls `mint_base_token` for the value mint → `notify_l2_asset_tracker(1, value_amount)` succeeds.
   - The main transaction body executes contract `C`, which writes `0` to L2AssetTracker slot 154.
   - Post-execution: `mint_base_token` is called for the operator fee → `notify_l2_asset_tracker(1, fee_amount)` is called. The L2AssetTracker now has `L1_CHAIN_ID = 0`, causing `handleFinalizeBaseTokenBridgingOnL2(1, fee_amount)` to revert (chain ID mismatch).
4. `notify_l2_asset_tracker` returns `Err(internal_error!(...))`.
5. Block processing halts. All subsequent transactions in the block are not processed. [7](#0-6) [8](#0-7)

```rust
fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
```
