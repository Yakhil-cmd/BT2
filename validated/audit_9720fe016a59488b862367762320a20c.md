### Title
Missing Access Control and No-Op Implementation in `MsgTurnBridge` Handler — (`x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` message handler in the Cronos module message server contains no permission check and no implementation — it unconditionally returns `nil, nil`. Any unprivileged user can submit a `MsgTurnBridge` transaction and receive a success response. More critically, because the function never mutates any state, the bridge circuit-breaker is permanently non-functional: even the authorized admin cannot disable the Gravity Bridge in an emergency.

### Finding Description

ADR-009 explicitly designates `MsgTurnBridge` as a permission-restricted message requiring the `CanTurnBridge` bit, exactly as `MsgUpdateTokenMapping` requires `CanChangeTokenMapping`. The permission constant is defined and the `HasPermission` helper exists:

```go
// x/cronos/keeper/permissions.go
const (
    CanChangeTokenMapping uint64 = 1 << iota // 1
    CanTurnBridge                            // 2
    All = CanChangeTokenMapping | CanTurnBridge // 3
)
```

`UpdateTokenMapping` correctly enforces this:

```go
// x/cronos/keeper/msg_server.go:69-81
func (k msgServer) UpdateTokenMapping(...) ... {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    ...
}
```

`TurnBridge`, however, is a stub:

```go
// x/cronos/keeper/msg_server.go:84-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

Two distinct defects are present simultaneously:

1. **No authorization check** — the `CanTurnBridge` permission is never consulted. Any address can submit `MsgTurnBridge` and receive a `code: 0` response.
2. **No state mutation** — the bridge-enabled flag is never written, so the Gravity Bridge remains permanently in its default (enabled) state regardless of what any caller submits.

The proto definition confirms the message is live and routable:

```proto
// proto/cronos/tx.proto:29-30
// TurnBridge defines a method to disable or enable the gravity bridge
rpc TurnBridge(MsgTurnBridge) returns (MsgTurnBridgeResponse);
```

The integration test `test_gravity_turn_bridge` demonstrates the intended contract — a non-admin call must fail and an admin call must actually disable bridge operations — but the current implementation satisfies neither invariant.

### Impact Explanation

**High — Bypass of Cronos admin/bridge authorization checks.**

The `CanTurnBridge` permission system is entirely bypassed: any unprivileged account can call `TurnBridge` and receive a success response, violating the authorization boundary documented in ADR-009.

**High — Permanent inability for the admin to halt bridge/conversion flows.**

Because the function never writes state, the Gravity Bridge circuit-breaker is permanently inoperative. In a bridge exploit scenario the admin has no on-chain mechanism to stop outbound `send_to_ethereum` flows or inbound voucher minting, leaving user funds exposed for the full duration of any incident.

### Likelihood Explanation

The entry point is a standard Cosmos SDK transaction (`/cronos.Msg/TurnBridge`) reachable by any account with gas. No privileged preconditions are required to trigger the auth bypass. The broken circuit-breaker is a persistent, always-present condition requiring no attacker action at all.

### Recommendation

Implement `TurnBridge` with the same pattern used by `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` on failure.
2. Persist the `enable` flag to the module KV store.
3. Have the Gravity Bridge end-blocker and EVM hook read that flag before processing outbound/inbound transfers.

### Proof of Concept

```
# Any unprivileged account
cronosd tx cronos turn-bridge false --from attacker --chain-id cronos_777-1
# → code: 0  (success, no permission check, no state change)

# Admin attempting emergency shutdown
cronosd tx cronos turn-bridge false --from admin --chain-id cronos_777-1
# → code: 0  (success, but bridge state unchanged — bridge remains enabled)
