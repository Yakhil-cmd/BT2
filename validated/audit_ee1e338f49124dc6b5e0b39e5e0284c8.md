### Title
`DelegateAction` Signed Payload Lacks Chain Binding, Enabling Cross-Chain Signature Replay - (File: `core/primitives/src/action/delegate.rs`, `core/primitives/src/signable_message.rs`)

---

### Summary

The `DelegateAction` (NEP-366 meta transaction) signed payload does not include a chain identifier or a block hash. Unlike regular `Transaction` objects, which include a `block_hash` that implicitly binds them to a specific chain, `DelegateAction` only includes a `max_block_height` (a plain integer). The `SignableMessage` discriminant is a fixed NEP number identical across all NEAR networks. A `SignedDelegateAction` created and broadcast on testnet can be extracted and replayed on mainnet (or any other NEAR network) if the same account exists on both chains with the same key and a valid nonce.

---

### Finding Description

**Regular transactions** include a `block_hash` field:

```rust
pub struct TransactionV0 {
    pub signer_id: AccountId,
    pub public_key: PublicKey,
    pub nonce: Nonce,
    pub receiver_id: AccountId,
    pub block_hash: CryptoHash,   // ← chain-specific anchor
    pub actions: Vec<Action>,
}
```

The `block_hash` must reference a block within `transaction_validity_period` blocks of the current chain tip. Because block hashes are chain-specific, a transaction signed on testnet will fail the validity-period check on mainnet.

**`DelegateAction`** has no such field:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,  // ← plain integer, not chain-specific
    pub public_key: PublicKey,
}
```

The hash signed by the user is computed as:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

`SignableMessage` prepends only a `MessageDiscriminant` (the constant `MIN_ON_CHAIN_DISCRIMINANT + 366`), which is identical on every NEAR network:

```rust
const MIN_ON_CHAIN_DISCRIMINANT: u32 = 1 << 30;
const NEP_366_META_TRANSACTIONS: u32 = 366;
```

No chain ID, genesis hash, or block hash is mixed into the signed bytes.

**`apply_delegate_action`** enforces only:
1. Signature validity against the chain-agnostic hash.
2. `block_height ≤ max_block_height` — a plain integer comparison, valid on any chain.
3. Nonce strictly greater than the on-chain nonce — prevents same-chain replay, not cross-chain replay.

```rust
if !signed_delegate_action.verify() { ... }
if apply_state.block_height > delegate_action.max_block_height() { ... }
validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
```

No check ever compares the signed payload against the current chain's identity.

**Attack path:**

1. Alice signs a `DelegateAction` on testnet (e.g., `ft_transfer` to Bob, nonce = 5, `max_block_height` = 200 000 000).
2. The relayer wraps it in a testnet transaction; it is included in a block and becomes public on-chain data.
3. An attacker extracts the raw `SignedDelegateAction` bytes from the testnet block.
4. The attacker constructs a mainnet transaction (paying their own gas) wrapping the same `SignedDelegateAction` and submits it to mainnet.
5. If Alice's mainnet access-key nonce is < 5 and mainnet block height is < 200 000 000, `apply_delegate_action` accepts the action and executes it on mainnet.

The same attack applies to `DelegateActionV2` / `VersionedSignedDelegateAction`, which uses `SignableMessageType::DelegateActionV2` (NEP 611) — also a fixed constant with no chain binding.

---

### Impact Explanation

An attacker who observes any `SignedDelegateAction` on one NEAR network can replay it on any other NEAR network where:
- the `sender_id` account exists with the same public key,
- the stored nonce is below the delegate action's nonce, and
- the current block height is below `max_block_height`.

Exploitable actions include token transfers, `AddKey` (granting the attacker a new access key), `DeployContract`, `DeleteAccount`, and any `FunctionCall`. The corrupted state values are the sender's token balance, access-key set, and contract code — all modified without the user's consent on the unintended chain.

---

### Likelihood Explanation

NEAR account IDs are human-readable strings (e.g., `alice.near`). The same account ID with the same ED25519 key pair routinely exists on both mainnet and testnet (wallets create identical accounts on both networks). Nonces on testnet are often higher than on mainnet (testnet is used for testing), making the reverse direction (testnet → mainnet) the most dangerous. `max_block_height` values are typically set far in the future (e.g., `current_height + 1 000 000`), so the window is wide. The `SignedDelegateAction` is embedded in a public on-chain transaction, so no privileged access is needed to extract it.

---

### Recommendation

Include a chain-specific binding in the `DelegateAction` signed payload. Two complementary approaches:

1. **Add `chain_id` to `DelegateAction`**: Include the genesis `chain_id` string (e.g., `"mainnet"`, `"testnet"`) in the struct and mix it into `get_nep461_hash()`. This is the direct analog of the EIP-712 `chainId` fix.

2. **Add a `block_hash` field** (analogous to regular `Transaction`): Require the signer to commit to a recent block hash, which is chain-specific and also provides a freshness bound, replacing the weaker `max_block_height` mechanism.

Either change is a protocol-breaking modification requiring a protocol version gate.

---

   - nonce(5) > ak_nonce(3) ✓

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }
