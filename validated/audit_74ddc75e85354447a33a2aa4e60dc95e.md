### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes User Overpayment When Native Resource Consumption Is Below EVM Gas Equivalent - (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` correction is only applied in one direction: when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), `gas_used` is increased and the user is charged more. When `delta_gas < 0` (native resource consumption is *below* what EVM gas implies), `gas_used` is never decreased. A developer-authored `TODO` comment in the code explicitly acknowledges this gap. As a result, users systematically overpay gas fees whenever their transaction consumes less native resource than the EVM gas used would imply, with the excess flowing to the operator rather than being refunded.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements ZKsync OS's double-resource accounting reconciliation step. After execution, it computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
```

The intent, as documented in `docs/double_resource_accounting.md`, is to reconcile EVM gas consumption with native resource consumption. The documentation states:

> "If `deltaGas > 0`, we add it to `gasUsed`…"

However, the symmetric case — when `delta_gas < 0` — is silently ignored. The code reads:

```rust
if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The `TODO` comment is a direct acknowledgment by the developers that the negative branch is missing. When `delta_gas < 0`, `gas_used` remains inflated relative to what native resource consumption justifies. Since the user's refund is computed as `(gas_limit - gas_used) * gas_price`, an inflated `gas_used` means the user receives a smaller refund than they are owed. The difference `|delta_gas| * gas_price` is effectively transferred to the operator.

This is structurally analogous to the external report: two independent adjustments (charge extra when native is high; refund extra when native is low) are conflated into a single one-sided `if` block, causing the second adjustment to be silently skipped.

---

### Impact Explanation

Every transaction where `native_used / native_per_gas < gas_used` results in the user paying more than they should. The overpayment per transaction is:

```
overpayment = |delta_gas| * gas_price
            = (gas_used - native_used / native_per_gas) * gas_price
```

This is a direct, irreversible loss of user funds. The excess is credited to the operator's coinbase address. The loss scales with gas price and the magnitude of the discrepancy between EVM gas and native resource consumption. For transactions with high EVM gas usage but low native resource consumption (e.g., many cheap EVM opcodes, or transactions that benefit from storage refunds), the discrepancy can be substantial.

---

### Likelihood Explanation

The condition `delta_gas < 0` is triggered whenever a transaction's native resource consumption is proportionally lower than its EVM gas consumption. This is a common scenario:

- Transactions that use many EVM opcodes that are cheap in native (RISC-V cycle) terms but expensive in EVM gas terms.
- Transactions that accumulate large EVM storage refunds (reducing `gas_used` via `evm_refund`) but whose native consumption is not similarly reduced.
- Any transaction where the EVM gas model overestimates the native proving cost.

The `#[cfg(not(feature = "unlimited_native"))]` gate means this code path is active in all production builds (the `unlimited_native` feature is a testing convenience). Any unprivileged user submitting a standard L2 transaction can trigger this path.

---

### Recommendation

When `delta_gas < 0`, decrease `gas_used` by `|delta_gas|`, clamped so that `gas_used` does not fall below `minimal_gas_used`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This makes the native-to-gas reconciliation symmetric, ensuring users are neither overcharged nor undercharged relative to their actual native resource consumption.

---

### Proof of Concept

1. Submit a transaction with `gas_limit = 200_000`, `gas_price = 1000`, `native_per_gas = 10`.
2. After execution, suppose `gas_used = 100_000` (from ergs) and `native_used = 500_000` (native units consumed).
3. `native_used / native_per_gas = 50_000`.
4. `delta_gas = 50_000 - 100_000 = -50_000`.
5. Because `delta_gas < 0`, the branch is skipped; `gas_used` remains `100_000`.
6. Refund = `(200_000 - 100_000) * 1000 = 100_000_000` tokens.
7. Correct refund should be `(200_000 - 50_000) * 1000 = 150_000_000` tokens.
8. User loses `50_000_000` tokens (50,000 gas × 1000 gas_price) that are instead paid to the operator.
