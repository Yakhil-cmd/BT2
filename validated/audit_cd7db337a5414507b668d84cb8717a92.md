### Title
Malicious Participant Can Corrupt CKD Output by Injecting Arbitrary Group Elements Without Proof of Correctness - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

In `do_ckd_coordinator`, the coordinator accumulates each participant's `(big_y, big_c)` contribution by simple addition, with no proof of correctness. A malicious participant can substitute arbitrary G1 elements for their correctly-computed share. Because the coordinator performs no validation before summing, the final `CKDOutput` is silently corrupted, and every honest party that calls `unmask` receives a wrong derived key.

---

### Finding Description

Each honest participant is supposed to compute, in `compute_signature_share`:

```
y  ← random scalar
Y  = y · G
S  = x_i · H(pk ‖ app_id)
C  = S + y · app_pk
norm_Y = λ_i · Y
norm_C = λ_i · C
```

These values are sent privately to the coordinator. The coordinator then accumulates them:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

There is no check that the received `(big_y, big_c)` satisfies the relation `big_c = x_i · H(pk ‖ app_id) + y · app_pk` for the same `y` used in `big_y = y · G`. No Schnorr proof, no DLEQ proof, no commitment-then-reveal — nothing. A malicious participant simply sends two arbitrary `ElementG1` values and the coordinator adds them verbatim into the aggregate.

This is the direct analog of the Timeswap bug: just as `zIncrease` was a free user-controlled input that updated `pool.state.z` without being constrained to the actual collateral deposited, here `(big_y, big_c)` is a free participant-controlled input that updates the aggregate CKD state without being constrained to the participant's actual key share. ### Impact Explanation

The final output returned to the coordinator is:

```
Y_final = Σ (λ_i · y_i · G)  +  attacker_Y_arb
C_final = Σ (λ_i · (x_i · H + y_i · A))  +  attacker_C_arb
```

When the client calls `unmask(a)`:

```rust
// src/confidential_key_derivation/mod.rs  lines 54-56
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
```

the result is `msk · H(pk ‖ app_id) + attacker_C_arb − a · attacker_Y_arb`, which is not the correct derived key unless the attacker chose `attacker_C_arb = a · attacker_Y_arb` (which would require knowing `a`). In the general case the output is a random, unusable G1 element. [3](#0-2) 

**Matched allowed impact:** *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

- Any **single** participant out of the full set suffices; no collusion is required.
- The attack requires only the ability to participate in the protocol (i.e., hold a valid key share and be included in the participant list).
- The malicious contribution is completely undetectable: the coordinator has no way to distinguish a correct `(norm_big_y, norm_big_c)` from an arbitrary pair of G1 points.
- There is no post-hoc verification of the CKD output (unlike the ECDSA signing path, which calls `sig.verify` before returning). [4](#0-3) 

---

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a non-interactive zero-knowledge proof (e.g., a DLEQ / Chaum-Pedersen proof) demonstrating that the same discrete-log witness `y` was used in both `Y = y · G` and the `y · app_pk` term inside `C`. Concretely:

- Prove `log_G(Y) = log_{app_pk}(C − x_i · H(pk ‖ app_id))` using a standard Chaum-Pedersen sigma protocol.
- The coordinator verifies every proof before adding the contribution to the aggregate.
- Participants whose proof fails are identified and excluded (or the protocol aborts).

This mirrors the fix recommended in the Timeswap report: make the state-updating value a *computed* quantity whose correctness is enforced, rather than a free caller-supplied input.

---

### Proof of Concept

1. Honest participants P1…P_{n-1} run `compute_signature_share` correctly and send their shares to the coordinator.
2. Malicious participant P_n sends `(big_y = G, big_c = G)` (or any two arbitrary G1 points) instead of the correctly computed values.
3. The coordinator's loop at lines 50-55 adds `G` to both `norm_big_y` and `norm_big_c` without complaint.
4. The returned `CKDOutput` contains `(Y_honest + G, C_honest + G)`.
5. The client calls `unmask(a)` and obtains `C_honest + G − a · (Y_honest + G) = msk · H(pk ‖ app_id) + G(1 − a)`, which is not the correct derived key.
6. The derived key is unusable; the attack is silent and leaves no detectable trace. [5](#0-4) [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
