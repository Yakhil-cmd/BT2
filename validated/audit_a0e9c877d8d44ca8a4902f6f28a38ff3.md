### Title
`verify_tx_weights_within_max_capacity` Uses Full `executed_class_hashes` Instead of Marginal Set, Causing False `TransactionTooLarge` Rejection During Sequencing — (`crates/blockifier/src/bouncer.rs`)

---

### Summary

`verify_tx_weights_within_max_capacity` passes the transaction's **full** `executed_class_hashes` to `get_tx_weights`, while `Bouncer::try_update` — the authoritative admission gate — passes only the **marginal** class hashes (those not already seen in the current block). When a transaction re-executes a class already present in the block, `verify_tx_weights_within_max_capacity` over-counts CASM hash computation gas and can return `TransactionTooLarge` for a transaction that `try_update` would accept.

---

### Finding Description

`verify_tx_weights_within_max_capacity` is called inside `Transaction::execute_raw` immediately after execution, before `Bouncer::try_update` is ever reached: [1](#0-0) 

It calls `get_tx_weights` with the **full** class hash set: [2](#0-1) 

`Bouncer::try_update`, by contrast, first subtracts the block's already-seen class hashes: [3](#0-2) 

and passes only `marginal_executed_class_hashes` to `get_tx_weights`: [4](#0-3) 

Inside `get_tx_weights`, every class hash in the supplied set is mapped to its CASM hash computation resources and those resources are converted to `sierra_gas` and `proving_gas`: [5](#0-4) [6](#0-5) 

`get_executed_class_hashes` — the set that `try_update` subtracts — is derived from the block's accumulated `casm_hash_computation_data_sierra_gas` map: [7](#0-6) 

So when class C is already in the block's bouncer, `try_update` charges **zero** CASM hash gas for C, while `verify_tx_weights_within_max_capacity` charges the full amount. The two functions compute structurally different weights for the same transaction.

---

### Impact Explanation

`verify_tx_weights_within_max_capacity` compares the inflated weight against `block_max_capacity` (the full block budget, not the remaining budget): [8](#0-7) 

If the inflated weight exceeds `block_max_capacity`, the function returns `TransactionTooLarge`. Because `verify_tx_weights_within_max_capacity` is called **before** `try_update` inside `execute_raw`, the transaction is permanently rejected — `try_update` is never reached. A transaction that `try_update` would have accepted (marginal weight fits in the remaining block capacity) is instead killed with `TransactionTooLarge`.

Concrete scenario:
- `block_max_capacity.proving_gas = P`
- Block already has class C; C's CASM hash proving gas = `G_C`
- Transaction T executes C; T's other proving gas = `G_T`
- `G_T + G_C > P` → `verify_tx_weights_within_max_capacity` returns `TransactionTooLarge`
- Accumulated block proving gas = `A`; `A + G_T ≤ P` → `try_update` would accept T

The admission decision is corrupted: a valid transaction is permanently rejected during sequencing.

---

### Likelihood Explanation

The condition requires `G_C` (CASM hash proving gas for a single class) to be large enough to push `G_T + G_C` over `block_max_capacity.proving_gas`. With the default config (`proving_gas = 5 × 10⁹`) this demands an unusually large class. However:

- The bouncer config is operator-tunable; tighter configs make the condition trivially reachable.
- Large Sierra classes (e.g., complex account contracts) can have non-trivial CASM hash computation resources.
- The scenario is deterministic and reproducible once the class and block state are fixed.

Likelihood is **low in production with default config**, but **high with tight configs or large classes**.

---

### Recommendation

Replace the full `executed_class_hashes` in `verify_tx_weights_within_max_capacity` with the marginal set (i.e., subtract the block's already-seen class hashes before calling `get_tx_weights`), or remove the function entirely and rely solely on `Bouncer::try_update` for the admission decision. If the "too large for any block" invariant must be preserved as a separate check, it should be performed once at gateway ingress (before any block state exists), not inside `execute_raw` where block state is already accumulated.

---

### Proof of Concept

```rust
// In bouncer_test.rs (illustrative)
#[test]
fn test_verify_vs_try_update_discrepancy() {
    // Tight config: proving_gas budget just above G_T but below G_T + G_C
    let bouncer_config = BouncerConfig {
        block_max_capacity: BouncerWeights {
            proving_gas: GasAmount(G_T + G_C - 1), // full weight exceeds limit
            ..BouncerWeights::empty()
        },
        ..BouncerConfig::empty()
    };

    // Block already has class C in its bouncer
    let mut bouncer = Bouncer::new(bouncer_config.clone());
    // ... seed bouncer with class C already accumulated ...

    // verify_tx_weights_within_max_capacity uses full class hashes → TransactionTooLarge
    let full_result = verify_tx_weights_within_max_capacity(
        &state, &tx_execution_summary, /* full hashes include C */ ...
    );
    assert!(matches!(full_result, Err(TransactionExecutionError::TransactionTooLarge { .. })));

    // try_update uses marginal hashes (C excluded) → fits
    let marginal_result = bouncer.try_update(
        &state, &tx_state_changes_keys, &tx_execution_summary, ...
    );
    assert!(marginal_result.is_ok()); // contradiction: same tx accepted here
}
```

The test demonstrates that for the same transaction and block state, `verify_tx_weights_within_max_capacity` returns `TransactionTooLarge` while `try_update` would return `Ok(())`, confirming the corrupted admission decision.
