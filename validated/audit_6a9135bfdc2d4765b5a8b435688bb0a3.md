### Title
Missing Participant Share Verification in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary

The Confidential Key Derivation (CKD) protocol coordinator aggregates `(norm_big_y, norm_big_c)` shares from all participants without performing any verification that those shares are correctly formed. A malicious participant can send arbitrary group elements, causing the coordinator to produce a silently corrupted `CKDOutput` that honest parties accept as valid. The verification step — which should prove each participant's `big_c` is correctly derived from their actual secret share — is entirely absent from the implementation.

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, the `do_ckd_coordinator` function receives each participant's `(norm_big_y, norm_big_c)` and blindly sums them: [1](#0-0) 

Each participant is supposed to compute:
- `big_y = y_i * G` (random blinding factor)
- `big_s = x_i * H(pk || app_id)` (secret share contribution)
- `big_c = big_s + y_i * app_pk` (ElGamal-style encryption of the share)
- Normalize both by the Lagrange coefficient `lambda_i` [2](#0-1) 

The coordinator then sums all `(norm_big_y, norm_big_c)` to produce the final `CKDOutput`: [3](#0-2) 

**No proof of correct computation is sent or verified.** There is no zero-knowledge proof, no consistency check, and no mechanism for the coordinator to confirm that a participant's `norm_big_c` was actually derived from their legitimate secret share `x_i` and the correct `app_pk`. The `do_ckd_participant` function simply sends the pair and returns `None`: [4](#0-3) 

This is the direct analog to the "functionality outlined in specification that is not implemented yet" class: the protocol's correctness guarantee — that each participant proves their contribution is well-formed — is documented by the mathematical structure (ElGamal encryption requires the discrete-log relationship between `big_y` and `big_c` to be provable) but is never implemented in the code.

### Impact Explanation

A single malicious participant can send any arbitrary `(norm_big_y', norm_big_c')` pair. The coordinator sums all contributions and produces a `CKDOutput` that is silently wrong. The honest coordinator and all honest participants accept this output as the legitimate derived key, but it will not decrypt to `msk * H(pk || app_id)` under `app_sk`. The derived key is permanently corrupted for that `(pk, app_id)` pair.

This maps to: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

The malicious participant can also bias the output toward a value of their choosing (by computing what offset they need to inject), potentially enabling them to influence the derived key to a known value if they can predict or control other participants' contributions.

### Likelihood Explanation

Any single participant in the CKD protocol is a sufficient attacker. No privileged access is required — the attacker only needs to be one of the `n` participants, which is the normal operating assumption. The attack requires no cryptographic breaks, only sending a crafted message in place of the honest protocol message. The coordinator has no way to detect the substitution.

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof of knowledge of `y_i` and `x_i` such that `norm_big_y = lambda_i * y_i * G` and `norm_big_c = lambda_i * (x_i * H(pk||app_id) + y_i * app_pk)`, consistent with the participant's committed public key share. A standard Chaum-Pedersen sigma proof over the pair `(G, app_pk)` with common exponent `y_i`, combined with a proof that `x_i` matches the public key share from DKG, would close this gap. The coordinator must verify all proofs before aggregating.

### Proof of Concept

1. Run CKD with 3 participants where participant `P2` is malicious.
2. `P2` replaces their honest `(norm_big_y, norm_big_c)` with `(ElementG1::identity(), ElementG1::identity())` (or any arbitrary pair) before sending to the coordinator.
3. The coordinator receives the tampered values at `recv_from_others` (line 51) and adds them into `norm_big_y` and `norm_big_c` without any check.
4. The resulting `CKDOutput` is `(sum_honest_Y + identity, sum_honest_C + identity)` — a value that does not correspond to `msk * H(pk||app_id)` encrypted under `app_pk`.
5. `ckd_output.unmask(app_sk)` returns a wrong group element, silently accepted as the derived key.

The test in `src/confidential_key_derivation/protocol.rs` at line 211 demonstrates the happy path but contains no adversarial participant test, confirming the missing verification is not caught by the existing test suite. [5](#0-4)

    fn test_ckd() {
        let mut rng = MockCryptoRng::seed_from_u64(42);

        // Create the app necessary items
        let app_id = AppId::try_from(b"Near App").unwrap();
        let app_sk = Scalar::random(&mut rng);
        let app_pk = ElementG1::generator() * app_sk;

        let participants = generate_participants(3);

        // choose a coordinator at random
        let coordinator = *participants
            .choose(&mut rng)
            .expect("participant list is not empty");
        let participant_list = ParticipantList::new(&participants).unwrap();

        // Manually compute signing keys
        let mut private_shares = Vec::new();
        let mut msk = Scalar::ZERO;
        for (i, _) in participants.iter().enumerate() {
            let mut rng_p = MockCryptoRng::seed_from_u64(rng.next_u64());
            let private_share = SigningShare::new(Scalar::random(&mut rng_p));
            // compute lambda(i)
            let lambda_i = participant_list
                .lagrange::<BLS12381SHA256>(participant_list.get_participant(i).unwrap())
                .unwrap();

            msk += lambda_i * private_share.to_scalar();
            private_shares.push(private_share);
        }

        // Manually compute master verification
        let pk = VerifyingKey::new(G2Projective::generator() * msk);

        let mut protocols: GenProtocol<CKDOutputOption> = Vec::with_capacity(participants.len());
        for (i, p) in participants.iter().enumerate() {
            let rng_p = MockCryptoRng::seed_from_u64(rng.next_u64());
            let key_pair = KeygenOutput {
                public_key: pk,
                private_share: private_shares[i],
            };

            let protocol = ckd(
                &participants,
                coordinator,
                *p,
                key_pair,
                app_id.clone(),
                app_pk,
                rng_p,
            )
            .unwrap();

            protocols.push((*p, Box::new(protocol)));
        }

        let result = run_protocol(protocols).unwrap();

        // test one single some for the coordinator
        let ckd_output = check_one_coordinator_output(result, coordinator).unwrap();

        // compute msk . H(pk, app_id)
        let confidential_key = ckd_output.unmask(app_sk);

        // H(pk || app_id) * msk
        let expected_confidential_key = hash_app_id_with_pk(&pk, &app_id) * msk;

        assert_eq!(
            confidential_key, expected_confidential_key,
            "Keys should be equal"
        );
        insta::assert_json_snapshot!(ckd_output);
    }
