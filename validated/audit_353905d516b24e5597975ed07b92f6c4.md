### Title
`evmAddress` and `CapabilityPath` Not Included in Signed Message Hash Enables Cross-COA Ownership Proof Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies Cadence account key signatures only over the caller-supplied `signedData` byte array. The two other user-controlled fields that determine *which* COA is being proven — `evmAddress` and `CapabilityPath` — are never included in the signed message. An attacker who obtains a valid signature from a Cadence account over any `signedData` can substitute a different `CapabilityPath`/`evmAddress` pair and pass the proof check for a different COA owned by the same account. The code itself acknowledges this gap in a comment but treats it as low-risk; the analysis below shows it is reachable and has concrete impact.

---

### Finding Description

`validateCOAOwnershipProof` accepts six parameters: `address`, `path` (a `PublicPath`), `signedData`, `keyIndices`, `signatures`, and `evmAddress`. [1](#0-0) 

Signature verification is performed exclusively over `signedData`: [2](#0-1) 

After the signature check passes, the function borrows the COA resource at the caller-supplied `path` and compares its EVM address to the caller-supplied `evmAddress`: [3](#0-2) 

Neither `path` nor `evmAddress` is ever hashed into the signed message. The code explicitly acknowledges this: [4](#0-3) 

The `CapabilityPath` field originates from the RLP-encoded `COAOwnershipProof` bytes that the caller supplies: [5](#0-4) 

These bytes are decoded and passed directly to `validateCOAOwnershipProof` without any binding to the signed content: [6](#0-5) 

**Attack path (direct Cadence call):**

1. Victim Cadence account `A` owns two COAs: `COA_1` at `/public/coa` (EVM address `0xAAA`) and `COA_2` at `/public/coa2` (EVM address `0xBBB`).
2. `A` legitimately signs some `signedData` to prove ownership of `COA_1`. The signature covers only `signedData`.
3. An attacker intercepts the proof bytes, replaces `CapabilityPath` with `"coa2"` and sets `evmAddress` to `0xBBB`.
4. The signature check passes (same `signedData`, same keys). The EVM address check passes (`COA_2.address() == 0xBBB`). The function returns `isValid: true` for `COA_2`, which the victim never authorized.

**Attack path (EVM precompile / ERC-1271):**

The EVM precompile `verifyCOAOwnershipProof` fixes `evmAddress` to `address(this)`, but the `CapabilityPath` still comes from the attacker-supplied proof bytes: [7](#0-6) 

If the victim account has two capability paths that both resolve to the same COA (e.g., a legacy path and a new path), a signature produced for one path is accepted for the other. More broadly, any off-chain or Cadence-native consumer of `validateCOAOwnershipProof` that does not enforce `evmAddress`-in-`signedData` is directly exploitable.

---

### Impact Explanation

An attacker can forge a passing COA ownership proof for a COA they do not control, using a signature the victim produced for a different purpose or a different COA. Concretely:

- **ERC-1271 impersonation**: A contract relying on `isValidSignature` (via the COA precompile) to gate asset transfers or authorizations can be tricked into accepting a proof for the wrong COA, enabling unauthorized EVM-side actions attributed to the victim's COA.
- **Cross-COA replay**: Any Cadence application that calls `EVM.validateCOAOwnershipProof` directly (e.g., a bridge, a multisig, an NFT gating contract) and does not independently enforce that `signedData` encodes `evmAddress` is vulnerable to cross-address impersonation.

The impact class is **signature/proof verification bypass leading to unauthorized asset access or identity impersonation**.

---

### Likelihood Explanation

- Entry point is fully unprivileged: any EVM transaction can call `verifyCOAOwnershipProof` via the Cadence Arch precompile; any Cadence script or transaction can call `EVM.validateCOAOwnershipProof` directly.
- The precondition (victim account with multiple COAs, or an application that calls the function directly) is realistic for any non-trivial Flow/EVM integration.
- The vulnerability is acknowledged in the production code comment, confirming the developers are aware of the gap but have not closed it at the protocol level.

---

### Recommendation

Include `evmAddress` (and optionally `CapabilityPath`) in the data that is cryptographically signed. The simplest fix is to require that `signedData` is a commitment over `(evmAddress || capabilityPath || applicationData)` and enforce this inside `validateCOAOwnershipProof` by reconstructing and verifying the expected message structure, analogous to how the original Switchboard report recommended incorporating the `offset` into the message hash.

Alternatively, enforce at the protocol level that `signedData` must encode `evmAddress` before the signature check is performed, removing the current advisory-only note from the comment.

---

### Proof of Concept

**Setup**: Cadence account `A` at address `0x01` has:
- `COA_1` stored at `/storage/coa`, published at `/public/coa` → EVM address `0xAAA`
- `COA_2` stored at `/storage/coa2`, published at `/public/coa2` → EVM address `0xBBB`

**Step 1**: `A` signs `signedData = [0xDE, 0xAD]` with key index 0 to prove ownership of `COA_1`. The resulting `COAOwnershipProof` has `CapabilityPath = "coa"`.

**Step 2**: Attacker constructs a new `COAOwnershipProof` with the same `Address`, `KeyIndices`, and `Signatures`, but sets `CapabilityPath = "coa2"`.

**Step 3**: Attacker calls (from any EVM transaction or Cadence script):

```cadence
EVM.validateCOAOwnershipProof(
    address: 0x01,
    path: /public/coa2,       // attacker-substituted
    signedData: [0xDE, 0xAD], // same as victim's
    keyIndices: [0],           // same as victim's
    signatures: [<victim_sig>],// same as victim's
    evmAddress: <0xBBB>        // attacker-substituted to match coa2
)
```

**Result**: `keyList.verify(signatureSet: ..., signedData: [0xDE, 0xAD], ...)` passes because the signature is valid over `signedData`. The EVM address check passes because `COA_2.address() == 0xBBB`. The function returns `ValidationResult(isValid: true)` for `COA_2`, which `A` never authorized. [2](#0-1) [8](#0-7) [9](#0-8)
