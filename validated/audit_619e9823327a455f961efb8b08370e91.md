### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Contribution Without Cryptographic Verification — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator accepts each participant's `(norm_big_y, norm_big_c)` contribution based solely on the check that the sender is a registered participant. There is no cryptographic proof that the participant used their actual private share. A single malicious participant can send an arbitrary value, causing the coordinator to assemble and return a corrupted confidential derived key that all honest parties accept as valid.

---

### Finding Description

The vulnerability class from the external report is **insufficient validation**: a check that only verifies "is this a registered entity" is treated as proof of legitimate behavior, but any registered entity can abuse that status to bypass the intended protection.

The direct analog lives in `do_ckd_coordinator` and `do_ckd_participant`.

**Participant side** — `do_ckd_participant` computes and sends its share to the coordinator with no attached proof:

```rust
// src/confidential_key_derivation/protocol.rs  lines 17-33
fn do_ckd_participant(...) -> Result<CKDOutputOption, ProtocolError> {
    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
    Ok(None)
}
``` [1](#0-0) 

**Coordinator side** — `do_ckd_coordinator` receives each participant's contribution via `recv_from_others` (which only checks sender identity) and blindly accumulates the values:

```rust
// src/confidential_key_derivation/protocol.rs  lines 35-57
async fn do_ckd_coordinator(...) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    let waitpoint = chan.next_waitpoint();
    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();   // no verification
        norm_big_c += participant_output.big_c();   // no verification
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
``` [2](#0-1) 

The correct value of `norm_big_c` for participant `i` is:

```
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

where `x_i` is the participant's private signing share. [3](#0-2) 

The coordinator's only gate is `recv_from_others`, which filters by registered participant identity — exactly the same pattern as `factory_.isSwapper(swapper_)` in the external report. [4](#0-3) 

The `ckd` entry-point validates that the coordinator and every participant are in the registered list, but performs no cryptographic binding between a participant's identity and the correctness of their contribution:

```rust
// src/confidential_key_derivation/protocol.rs  lines 95-101
if !participants.contains(coordinator) {
    return Err(InitializationError::MissingParticipant {
        role: "coordinator",
        participant: coordinator,
    });
}
``` [5](#0-4) 

---

### Impact Explanation

The final CKD output is:

```
big_C = Σ norm_big_c_i  →  unmask with app_sk  →  msk · H(pk ‖ app_id)
```

If participant `i` substitutes `norm_big_c_i` with any arbitrary group element `Z`, the coordinator computes:

```
big_C_corrupted = Z + Σ_{j≠i} norm_big_c_j
```

The coordinator returns this as a valid `CKDOutput`. Every honest party that consumes the output accepts an incorrect confidential derived key. There is no downstream check that would detect the corruption.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable or incorrect cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. No special privilege beyond being a registered participant is required. The attacker does not need to break any cryptographic primitive; they simply send a wrong group element. The attack is deterministic and silent — the coordinator produces output without error.

---

### Recommendation

Require each participant to attach a non-interactive zero-knowledge proof (e.g., a Schnorr proof of knowledge of `x_i` consistent with their public verification share, and a proof that `norm_big_c_i` was formed correctly) alongside their `(norm_big_y, norm_big_c)` message. The coordinator must verify all proofs before accumulating contributions. This is the standard approach used in verifiable secret sharing and threshold ElGamal encryption schemes.

---

### Proof of Concept

1. A legitimate participant `P_malicious` is registered in a CKD session with `N` participants.
2. Instead of calling `compute_signature_share` and sending the result, `P_malicious` sends `(ElementG1::identity(), ElementG1::identity())` — the group identity — as its contribution.
3. `do_ckd_coordinator` receives this via `recv_from_others`, passes the identity check (sender is a registered participant), and adds the identity elements to the running sums.
4. The coordinator assembles `CKDOutput { big_y, big_c }` where `big_c` is missing `P_malicious`'s share of `msk · H(pk ‖ app_id)`.
5. The coordinator returns `Some(ckd_output)` with no error. Any consumer that calls `ckd_output.unmask(app_sk)` obtains a key that is not `msk · H(pk ‖ app_id)`, silently breaking the confidential key derivation for the entire session.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
