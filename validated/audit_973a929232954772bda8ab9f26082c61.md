### Title
Operator-Controlled `native_price` / `pubdata_price` Oracle Input Bypasses Native Resource Accounting Integrity — (`File: zk_ee/src/system/metadata/zk_metadata.rs`, `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

---

### Summary

The ZKsync OS bootloader reads `native_price` and `pubdata_price` directly from the oracle-provided `BlockMetadataFromOracle` struct at block initialization. These two values are the sole divisors used to compute `native_per_gas` and `native_per_pubdata` — the ratios that govern how much native resource budget each transaction receives. Neither value is included in the ZK public input commitment, and neither is validated against any on-chain bound. A sequencer/operator who controls the oracle input can set these values to any arbitrary `U256` at block construction time, causing the STF to silently under-charge or over-charge every transaction in the block for native resources, with no cryptographic enforcement.

---

### Finding Description

**Root cause — oracle input accepted without commitment:**

In `metadata_op.rs`, the ZK-mode bootloader reads the entire `BlockMetadataFromOracle` struct from the oracle in a single call:

```rust
let block_level_metadata: BlockMetadataFromOracle =
    oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)?;
```

The only validation performed is a gas-limit upper-bound check:

```rust
if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT
    || metadata.individual_tx_gas_limit() > MAX_TX_GAS_LIMIT
{
    return Err(internal_error!("block or tx gas limit is too high"));
}
```

`native_price` and `pubdata_price` are accepted verbatim with no range check, no minimum, and no maximum. [1](#0-0) 

**Usage in resource accounting:**

During L2 transaction validation, `native_per_gas` is computed as:

```rust
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))
```

and `native_per_pubdata` as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
```

These two ratios directly set the native resource budget for every transaction in the block. [2](#0-1) 

**Not committed to public input:**

The `BatchOutput` and `ChainStateCommitment` structures that form the ZK public input contain: `chain_id`, `timestamp`, `state_root`, `block_number`, `block_hashes`, `da_commitment`, `priority_ops_hash`, `l2_logs_root`, `upgrade_tx_hash`. Neither `native_price` nor `pubdata_price` appear anywhere in the public input hash. [3](#0-2) 

The project's own documentation explicitly acknowledges this gap:

> "The block header should determine the block fully, i.e. include all the inputs needed to execute the block. Currently it misses `gas_per_pubdata` and `native_price`, but we are already working on design and implementation to solve this issue." [4](#0-3) 

The `form_block_header` function also carries a `// TODO: add pubdata price and native price` comment, confirming the omission is known but unresolved. [5](#0-4) 

**`BlockMetadataFromOracle` struct fields:**

`native_price` and `pubdata_price` are plain `U256` fields deserialized directly from the oracle byte stream with no post-deserialization validation. [6](#0-5) 

---

### Impact Explanation

**Resource accounting manipulation (oracle IO mismatch / resource accounting bug):**

By supplying an inflated `native_price` (e.g., `U256::MAX - 1`), the operator makes `native_per_gas = ceil(gas_price / native_price) = 0`, which causes `native_limit = gas_limit * 0 = 0`. The bootloader then treats `native_per_gas == 0` as a special case granting `u64::MAX - 1` native budget (see `api/src/helpers.rs` line 430–432), effectively giving every transaction unlimited native resources. This allows computationally unbounded transactions to pass native resource checks and be included in a provable block.

Conversely, by supplying a near-zero `native_price` (e.g., `1`), the operator makes `native_per_gas` enormous, causing every transaction to exhaust its native budget immediately and revert — a selective censorship vector.

By supplying `pubdata_price = 0`, `native_per_pubdata = 0`, removing all pubdata cost from native accounting, allowing storage-heavy transactions to bypass pubdata native charges entirely.

**Concrete impact:** A malicious or compromised sequencer can produce a valid ZK proof for a block where:
- Transactions that should have been rejected for insufficient native resources are included and succeed.
- The fee charged to users does not reflect actual proving cost, enabling the operator to extract value or subsidize specific transactions at the expense of protocol economics.
- The ZK proof verifier on L1 cannot detect this because `native_price`/`pubdata_price` are not part of the public input. [7](#0-6) 

---

### Likelihood Explanation

The attacker is the sequencer/operator — the entity that constructs the oracle input for the proving run. This is a **privileged role**, but the threat model for a ZK rollup is that the ZK proof should enforce correctness even against a malicious sequencer. The entire purpose of the public input commitment is to bind the sequencer to declared block parameters. Since `native_price` and `pubdata_price` are absent from the commitment, a sequencer can freely vary them between the forward run (used for user-facing gas estimation) and the proving run (used to generate the proof), with no on-chain enforcement. The project documentation confirms this is a known, unresolved gap. Likelihood is **medium** (requires a malicious or compromised sequencer, but no external oracle, no key leak, and no governance attack is needed beyond the sequencer's normal operational role).

---

### Recommendation

1. Include `native_price` and `pubdata_price` in the `ChainStateCommitment` or `BatchOutput` hash that forms the ZK public input, so the settlement layer can verify these values match what was declared.
2. Until (1) is implemented, add on-chain enforced bounds (minimum and maximum) for `native_price` and `pubdata_price` in the block header or a separate commitment, and validate them inside `metadata_op` before use.
3. Remove the `// TODO: add pubdata price and native price` deferral and treat this as a blocking correctness issue for mainnet.

---

### Proof of Concept

**Step 1:** Operator constructs a block oracle with `native_price = U256::MAX - 1`, `pubdata_price = U256::ZERO` in `BlockMetadataFromOracle`.

**Step 2:** The bootloader reads this via `oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)` — no validation rejects it. [8](#0-7) 

**Step 3:** For each L2 transaction, `native_per_gas = ceil(gas_price / (U256::MAX - 1)) = 0`. The bootloader enters the `native_per_gas == 0` branch and sets `native_limit = u64::MAX - 1`. [9](#0-8) 

**Step 4:** With `pubdata_price = 0`, `native_per_pubdata = 0`, so pubdata native charges are zero for all transactions.

**Step 5:** Computationally expensive transactions that would normally exhaust native resources now succeed. The block is proven. The public input hash does not include `native_price` or `pubdata_price`, so the L1 verifier accepts the proof. [10](#0-9) 

**Step 6:** The settlement layer verifies the proof successfully, finalizing a block where native resource accounting was silently bypassed.

        }

        Ok(metadata)
    }
