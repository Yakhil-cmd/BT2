### Title
Missing Authorization Check in `action_deploy_global_contract` Allows Unauthorized Overwrite of `AccountId`-Mode Global Contracts - (File: `runtime/runtime/src/global_contracts.rs`)

### Summary

`DeployGlobalContractAction` with `GlobalContractDeployMode::AccountId` stores a global contract under the **receiver account's** ID. Neither the action validation nor the execution function verifies that the transaction signer equals the receiver. Any unprivileged NEAR account can therefore overwrite another account's global contract with arbitrary WASM, affecting every account that references the victim's contract identifier.

### Finding Description

When `action_deploy_global_contract` executes with `deploy_mode = GlobalContractDeployMode::AccountId`, the contract code is stored under `TrieKey::GlobalContractCode { identifier: AccountId(receiver_id) }` and distributed to all shards. [1](#0-0) 

The design comment on `GlobalContractDeployMode::AccountId` explicitly states:

> "This allows the **owner** to update the contract for all its users." [2](#0-1) 

However, the validation function `validate_deploy_global_contract_action` only checks code size: [3](#0-2) 

And the execution function `action_deploy_global_contract` only checks that the receiver has sufficient balance for storage costs — it receives `account_id` (the receiver) but never the `signer_id`: [4](#0-3) 

In NEAR's transaction model, `signer_id` and `receiver_id` are independent fields. A full-access key holder can send any action to any `receiver_id`. The `check_account_existence` guard for `DeployGlobalContract` only verifies the receiver account exists: [5](#0-4) 

There is no protocol-level restriction that `signer_id == receiver_id` for `DeployGlobalContract`. The `signer_id` is present in `ActionReceipt` but is never threaded into `action_deploy_global_contract`: [6](#0-5) 

### Impact Explanation

**Global contract code corruption (Critical):** The attacker stores arbitrary malicious WASM under `TrieKey::GlobalContractCode { identifier: AccountId(victim) }`. Every account that previously called `UseGlobalContract(AccountId(victim))` stores `AccountContract::GlobalByAccount(victim)` as its contract reference: [7](#0-6) 

On any subsequent function call to those accounts, the runtime resolves the contract by looking up `GlobalContractCode { identifier: AccountId(victim) }` — which now contains the attacker's code. This enables theft of funds, unauthorized state mutation, and denial of service across all dependent accounts simultaneously.

**Balance drain:** The storage cost is deducted from the **receiver's** (victim's) balance, not the attacker's: [8](#0-7) 

The attacker pays only gas; the victim pays storage. Repeated deployments drain the victim's balance.

**Nonce mechanism does not prevent this:** The nonce in `GlobalContractDistributionReceipt` only prevents stale in-flight overwrites during legitimate updates; it does not prevent an unauthorized party from initiating a new distribution with a higher nonce: [9](#0-8) 

### Likelihood Explanation

Any NEAR account with a standard full-access key — the default for every account — can execute this attack. The attacker needs only the victim's account ID and enough NEAR to pay gas. No validator, operator, or privileged role is required. The attack is a single signed transaction.

### Recommendation

In `action_deploy_global_contract`, pass the `signer_id` from the enclosing `ActionReceipt` and assert `signer_id == account_id` when `deploy_mode == GlobalContractDeployMode::AccountId`. Alternatively, enforce this invariant in `validate_deploy_global_contract_action` by threading the signer and receiver IDs through the validation call chain. The `signer_id` is already present in `ActionReceipt.signer_id` and `ActionReceiptV2.signer_id`.

### Proof of Concept

1. **Setup:** Bob deploys a legitimate global contract:
   `signer_id = bob.near`, `receiver_id = bob.near`, `DeployGlobalContract(AccountId, legitimate_code)`.
   Many accounts call `UseGlobalContract(AccountId("bob.near"))`, storing `AccountContract::GlobalByAccount("bob.near")` in their state.

2. **Attack:** Alice (any unprivileged account) sends:
   `signer_id = alice.near`, `receiver_id = bob.near`, `DeployGlobalContract(AccountId, malicious_code)`.

3. **Execution:** The runtime applies the action to Bob's account:
   - Deducts storage cost from **Bob's** balance.
   - Stores `malicious_code` under `GlobalContractCode { identifier: AccountId("bob.near") }`.
   - Initiates cross-shard distribution via `GlobalContractDistributionReceipt` with a fresh nonce, overwriting all shards.

4. **Impact:** Every account holding `AccountContract::GlobalByAccount("bob.near")` now executes Alice's malicious WASM on any function call, enabling fund theft, unauthorized state changes, or complete denial of service across all dependent accounts.

pub(crate) fn action_deploy_global_contract(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    account_id: &AccountId,
    apply_state: &ApplyState,
    deploy_contract: &DeployGlobalContractAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let _span = tracing::debug_span!(target: "runtime", "action_deploy_global_contract").entered();

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {
        result.result = Err(ActionErrorKind::LackBalanceForState {
            account_id: account_id.clone(),
            amount: storage_cost,
        }
        .into());
        return Ok(());
    };
    result.tokens_burnt =
        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;
    account.set_amount(updated_balance);

    initiate_distribution(
        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}

fn initiate_distribution(
    state_update: &mut TrieUpdate,
    account_id: AccountId,
    contract_code: Arc<[u8]>,
    deploy_mode: &GlobalContractDeployMode,
    current_shard_id: ShardId,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let id = match deploy_mode {
        GlobalContractDeployMode::CodeHash => {
            GlobalContractIdentifier::CodeHash(hash(&contract_code))
        }
        GlobalContractDeployMode::AccountId => {
            GlobalContractIdentifier::AccountId(account_id.clone())
        }
    };
    // Increment the nonce and write it to state immediately to prevent multiple
    // distributions with the same nonce from being initiated. This requires
    // allowing the same nonce in the freshness check when applying the
    // distribution receipt.
    let nonce = increment_nonce(state_update, &id)?;
    let distribution_receipt =
        GlobalContractDistributionReceipt::new(id, current_shard_id, vec![], contract_code, nonce);
    let distribution_receipts =
        Receipt::new_global_contract_distribution(account_id, distribution_receipt);
    // No need to set receipt_id here, it will be generated as part of apply_action_receipt
    result.new_receipts.push(distribution_receipts);
    Ok(())
