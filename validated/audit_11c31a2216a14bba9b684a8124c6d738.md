### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Unauthorized Invoke Admission for Any Account with a Pending Deploy-Account Transaction - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point (signature check) for any invoke transaction with `nonce == 1` when the target account's on-chain nonce is `0` and the account address appears in the mempool. Because the check is purely address-based and not tied to the submitter's identity, any unprivileged attacker can submit an invoke transaction with an empty or forged signature on behalf of any victim account that has a pending `deploy_account` transaction, and the gateway will admit it to the mempool without ever verifying the signature.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}

### Impact Explanation

An attacker who observes that victim address `V` has a pending `deploy_account` transaction in the mempool can submit an invoke transaction with `sender_address = V`, `nonce = 1`, and an empty or arbitrary signature. The gateway will:

1. Pass stateless validation (empty signature satisfies the length check).
2. Retrieve `account_nonce = 0` for `V` from state.
3. Call `skip_stateful_validations` → `account_tx_in_pool_or_recent_block(V)` returns `true` → `skip_validate = true`.
4. Call `run_validate_entry_point` with `validate = false` → `__validate__` is never invoked → no signature check.
5. Admit the transaction to the mempool.

The attacker's transaction now occupies the `(V, nonce=1)` slot in the mempool. The victim's legitimate invoke (also `nonce = 1`) must either pay a higher fee to replace it (fee escalation) or be rejected as a duplicate nonce. If the attacker's transaction is executed first, it will fail at `__validate__` during batcher execution (the batcher sets `validate = true`), but the victim's legitimate invoke may have already been evicted from the mempool, breaking the deploy-account + invoke UX flow and wasting the victim's deploy-account fee.

This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack requires only that the victim has a pending `deploy_account` transaction, which is publicly observable from the mempool. No privileged access is needed. The attacker only needs to submit a single well-formed invoke transaction (with an empty signature) targeting the victim's address with `nonce = 1`. The window of vulnerability is the entire time the victim's `deploy_account` transaction remains unconfirmed.

### Recommendation

The `skip_stateful_validations` bypass should not suppress signature verification entirely. Instead, the gateway should still run the `__validate__` entry point even in the deploy-account + invoke UX flow. If the account does not yet exist on-chain (because the deploy-account is pending), the gateway can either:

1. **Defer validation**: Accept the invoke into a pending queue and re-validate once the deploy-account is confirmed, rather than skipping validation entirely.
2. **Require a proof of authorization**: Require the invoke transaction to include a signature that can be verified against the class hash declared in the pending deploy-account transaction (which is known from the mempool).
3. **Restrict the bypass to the same submitter**: Track the submitter identity of the deploy-account transaction and only skip validation for invokes from the same network peer/session.

At minimum, the `validate` flag should never be set to `false` for an invoke transaction that has not had its signature cryptographically verified.

### Proof of Concept

1. Victim submits `deploy_account` for address `V` (class hash `C`, salt `S`). The mempool now contains `V` with `nonce = 0`.
2. Attacker constructs an `RpcInvokeTransactionV3` with:
   - `sender_address = V`
   - `nonce = 1`
   - `signature = []` (empty)
   - `calldata = [<arbitrary malicious calldata>]`
   - Valid resource bounds (above the minimum gas price threshold)
3. Attacker submits this transaction to the gateway's `add_invoke_transaction` endpoint.
4. Gateway stateless validation passes (empty signature length `0 <= max_signature_length`).
5. Gateway stateful validation: `account_nonce = 0`, `tx_nonce = 1`, `account_tx_in_pool_or_recent_block(V) = true` → `skip_validate = true` → `validate = false` → `__validate__` is never called.
6. Transaction is admitted to the mempool at slot `(V, nonce=1)`.
7. Victim submits their legitimate invoke with `nonce = 1` and a valid signature. The mempool rejects it as a duplicate nonce (or requires fee escalation to replace the attacker's transaction).
8. The batcher eventually executes the attacker's transaction: `deploy_account` succeeds (nonce becomes 1), then the attacker's invoke fails at `__validate__` (invalid signature). The victim's legitimate invoke is no longer in the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
