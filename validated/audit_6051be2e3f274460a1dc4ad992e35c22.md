### Title
Unvalidated `app_pk` Identity Point in CKD Protocol Allows Confidential Derived Key Disclosure — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts the caller-supplied `app_pk: PublicKey` (the ElGamal encryption key `A`) without validating that it is a non-identity point. If a malicious app submits `A = G1::identity()` (the point at infinity), the ElGamal blinding term `y_i * A` collapses to zero for every participant, causing the aggregated ciphertext `C` to equal `msk * H(pk, app_id)` — the raw confidential derived key material — in plaintext. This is then broadcast on-chain as part of `CKDOutput`, making the confidential key `s` recoverable by any observer.

---

### Finding Description

The CKD protocol is designed so that each MPC node `i` computes:

```
C_i = S_i + y_i * A    where S_i = x_i * H(pk, app_id)
```

The random scalar `y_i` and the app's ElGamal public key `A` together provide the blinding that prevents any single node or the coordinator from learning `s = HKDF(msk * H(pk, app_id))`. The coordinator aggregates:

```
C = msk * H(pk, app_id) + a * Y
```

and the app recovers `s` by computing `C - a * Y` using its private scalar `a`.

The entire confidentiality guarantee rests on `A` being a valid non-identity point for which the app knows the discrete logarithm `a`. The `ckd()` entry point performs no such check:

```rust

    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,          // ← accepted without any validation
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // checks: participant count, duplicates, self-presence, coordinator-presence
    // NO check: app_pk.is_identity()
``` [1](#0-0) 

Inside `compute_signature_share`, the blinding term is computed unconditionally:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;   // if app_pk = 0, big_c = big_s = x_i * H(pk, app_id)
``` [2](#0-1) 

When `app_pk = G1::identity()`, `app_pk * y.0 = G1::identity()` for every participant, so `big_c = big_s`. After Lagrange normalization and aggregation by the coordinator:

```
C = Σ λ_i * x_i * H(pk, app_id) = msk * H(pk, app_id)
```

This is the raw BLS signature — the confidential key material — with no blinding whatsoever. The `CKDOutput { big_y, big_c }` is returned and published on-chain. [3](#0-2) 

The `unmask` method then trivially yields the key with `secret_scalar = 0`:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar   // unmask(0) = big_c = msk * H(pk, app_id)
}
``` [4](#0-3) 

The `BLS12381G1Group` serialization and `verify_signature` do reject identity points, but `app_pk` is never passed through either of those paths during the `ckd()` execution. [5](#0-4) 

---

### Impact Explanation

The security specification explicitly requires:

> *"s must be deterministic as a function of `app_id` and only known by app"*
> *"No single node in the MPC network should be capable of computing s"* [6](#0-5) 

When `app_pk = G1::identity()`, `C = msk * H(pk, app_id)` is published on-chain in the `CKDOutput`. Any observer — including the coordinator, any MPC node, or any blockchain participant — can compute `s = HKDF(C)` directly. This is a complete, public disclosure of the confidential derived secret, matching the Critical allowed impact: **extraction, reconstruction, or disclosure of confidential derived secrets**.

---

### Likelihood Explanation

The `app_pk` value originates from the app's on-chain request. The Developer contract is documented to verify that `A` appears in `report_data`, but there is no documented or enforced requirement that `A ≠ G1::identity()`. [7](#0-6) 

A malicious app (or a poorly implemented Developer contract) can submit `A = G1::identity()` with a valid attestation that commits to `A = 0` in `report_data`. The MPC library accepts this without error. The attack requires no cryptographic break, no key leakage, and no compromise of MPC nodes — only a single malformed on-chain request.

---

### Recommendation

Add an explicit identity-point check in `ckd()` before the protocol runs:

```rust
if app_pk.is_identity().into() {
    return Err(InitializationError::InvalidInput("app_pk must not be the identity point".into()));
}
```

Additionally, consider requiring a proof of knowledge of the discrete logarithm of `app_pk` (i.e., a Schnorr proof that the app knows `a` such that `A = a * G`) to prevent substitution attacks where the app commits to a public key it cannot decrypt.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
