### Title
Missing Participant Share Verification in CKD Coordinator Allows Undetectable Output Corruption — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator accumulates participant-contributed `(big_y, big_c)` pairs with no zero-knowledge proof or consistency check. A single malicious participant can send arbitrary group elements, silently corrupting the derived confidential key output with no detection mechanism.

---

### Finding Description

In `do_ckd_coordinator` (lines 44–57 of `src/confidential_key_derivation/protocol.rs`), the coordinator receives each participant's `(norm_big_y, norm_big_c)` pair and unconditionally adds them to its running totals:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is supposed to compute:
- `big_y_i = y_i · G` (random nonce times generator)
- `big_c_i = x_i · H(pk ‖ app_id) + y_i · app_pk` (ElGamal encryption of the key share)
- `norm_big_y_i = λ_i · big_y_i`, `norm_big_c_i = λ_i · big_c_i`

as implemented in `compute_signature_share` (lines 148–182). However, there is **no proof** that the received `big_c_i` is correctly formed relative to the claimed `big_y_i` and the participant's committed public key share. The coordinator has no way to verify that `big_c_i = x_i · H(pk ‖ app_id) + y_i · app_pk` holds for the `y_i` embedded in `big_y_i`.

Contrast this with the DKG protocol in `src/dkg.rs`, which enforces proofs of knowledge (`proof_of_knowledge`, lines 118–141) and commitment hash binding (`verify_commitment_hash`, lines 222–236) before accepting any participant contribution. The CKD protocol has no equivalent safeguard. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

The CKD output `(big_Y, big_C)` is used by the application to recover the confidential derived key via `unmask`:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [5](#0-4) 

When all participants are honest, `big_C − a · big_Y = msk · H(pk ‖ app_id)`. If a malicious participant injects an arbitrary additive offset `Δ_C` into `big_c` and `Δ_Y` into `big_y`, the coordinator outputs `(big_Y + Δ_Y, big_C + Δ_C)`, and `unmask` returns `msk · H(pk ‖ app_id) + Δ_C − a · Δ_Y` — a completely wrong value. The honest coordinator and all honest callers accept this corrupted output with no error, because there is no invariant check after aggregation.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The CKD protocol requires contributions from **all** participants (not just a threshold subset). A single malicious participant among the `N` participants is sufficient to corrupt the output. The attacker's entry path is direct: they participate in the protocol, call `ckd(...)` with their legitimate credentials, but return fabricated `(norm_big_y, norm_big_c)` values instead of the correctly computed ones. No special privilege, leaked key, or external assumption is required — only participation in the protocol. [6](#0-5) 

---

### Recommendation

1. **Add a Chaum-Pedersen / DLEQ proof** alongside each participant's `(norm_big_y, norm_big_c)` submission, proving that the discrete log of `big_y_i` relative to `G` equals the discrete log of `(big_c_i − x_i · H(pk ‖ app_id))` relative to `app_pk`. This requires the participant's public key share to be known to the coordinator.
2. Alternatively, use a **commitment-then-reveal** scheme (as done in DKG) so that participants commit to their values before seeing others', and the coordinator verifies the opening before aggregating.
3. At minimum, document the trust assumption that all CKD participants must be honest, and enforce this at the API level if the TEE deployment model guarantees it.

---

### Proof of Concept

A malicious participant `P_evil` participates in `ckd(...)` normally up to the point of sending its share. Instead of calling `compute_signature_share` honestly, it sends:

```
norm_big_y = 0·G  (identity element)
norm_big_c = arbitrary_point  (e.g., generator G)
```

The coordinator in `do_ckd_coordinator` adds these without complaint:

```rust
norm_big_y += participant_output.big_y();  // adds identity — no change to Y
norm_big_c += participant_output.big_c();  // adds G — shifts C by G
```

The resulting `CKDOutput` has `big_C` shifted by `G`. When the application calls `unmask(app_sk)`, it recovers `msk · H(pk ‖ app_id) + G` instead of `msk · H(pk ‖ app_id)`. The derived key is wrong, the application silently uses it, and no error is raised anywhere in the protocol. [1](#0-0)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

fn proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    me: Participant,
    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,
    rng: &mut impl CryptoRngCore,
) -> Result<Signature<C>, ProtocolError> {
    // creates an identifier for the participant
    let id = me.scalar::<C>();
    let vk_share = coefficient_commitment.eval_at_zero()?;

    // pick a random k_i and compute R_id = g^{k_id},
    // Step 2.5
    let (k, big_r) = <C>::generate_nonce(rng);

    // Step 2.6
    // compute H(domain_separator, id, me, g^{a_0}, R_id) as a scalar
    let hash = challenge::<C>(domain_separator, session_id, id, &vk_share, &big_r)?;
    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
    Ok(Signature::new(big_r, mu))
}

fn verify_commitment_hash<C: Ciphersuite>(
    session_id: &HashOutput,
    participant: Participant,
    domain_separator: &mut DomainSeparator,
    commitment: &VerifiableSecretSharingCommitment<C>,
    all_hash_commitments: &ParticipantMap<'_, HashOutput>,
) -> Result<(), ProtocolError> {
    let actual_commitment_hash = all_hash_commitments.index(participant)?;
    let commitment_hash =
        domain_separate_hash(domain_separator, &(&participant, &commitment, &session_id))?;
    if *actual_commitment_hash != commitment_hash {
        return Err(ProtocolError::InvalidCommitmentHash);
    }
    Ok(())
}
