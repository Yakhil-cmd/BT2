### Title
Hardcoded `PERIOD_MINT_CAP` Can Permanently Block Reward Minting as Protocol Grows — (File: `L1/starkware/solidity/stake/PeriodMintLimit.sol`)

---

### Summary

`PeriodMintLimit.sol` defines a hardcoded, immutable constant `PERIOD_MINT_CAP = 6_500_000 * 10**18` that caps the total STRK that can be minted per week across all calls to `MintManager.mintRequest`. This constant gates the entire L1→L2 reward-funding pipeline. As staking participation grows, weekly reward demand can exceed this cap, causing every `tick()` call after the quota is exhausted to revert with `"EXCEED_PERIOD_MINTING"`. The L2 `RewardSupplier` then runs out of token balance, and stakers/delegators can no longer claim their earned rewards.

---

### Finding Description

**Root cause — `PeriodMintLimit.sol`:**

```solidity
uint256 constant PERIOD_MINT_CAP = 6_500_000 * 10**18;   // hardcoded, file-level constant
uint256 constant MINTING_PERIOD_DURATION = 1 weeks;

function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
    bytes32 periodSlot = periodAccountingSlot(token);
    uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
    uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
    require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
    periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;
}
``` [1](#0-0) 

`MintManager` inherits `PeriodMintLimit` and calls `checkAndUpdatePeriodicalQuota` inside `mintRequest`:

```solidity
function mintRequest(address token, uint256 amount) external {
    ...
    checkAndUpdatePeriodicalQuota(token, amount);   // reverts if weekly cap exceeded
    mintingAllowance(token)[requester] -= amount;
    mintableToken(token).mint(requester, amount);
}
``` [2](#0-1) 

`RewardSupplier.tick()` calls `mintManager().mintRequest(token(), amountToMint)` where `amountToMint = TOKENS_PER_MINT_REQUEST * numMsgsToConsume` (up to `5 * 1_300_000e18 = 6_500_000e18` per tick): [3](#0-2) 

The cap is calibrated to exactly one maximum-capacity `tick()` per week (`5 msgs × 1.3M = 6.5M STRK`). If the protocol needs a second `tick()` in the same week — because reward demand has grown — the call reverts.

**L2 side — how the freeze propagates:**

The L2 `RewardSupplier` calls `request_funds` whenever `credit < debit + threshold`. If L1 minting is blocked, the L2 contract's token balance is never replenished. When stakers call `claim_rewards`, the `checked_transfer` to the staking contract fails because the balance is insufficient: [4](#0-3) 

The `request_funds` logic that triggers L1 minting messages: [5](#0-4) 

---

### Impact Explanation

When `PERIOD_MINT_CAP` is exhausted within a week, every subsequent `tick()` call reverts. The L2 `RewardSupplier` balance is not replenished. Stakers and delegators who have legitimately earned rewards cannot claim them until the next weekly period resets the quota. This constitutes **temporary freezing of unclaimed yield** for all protocol participants. If the protocol grows to a state where demand consistently exceeds the cap, the freeze becomes effectively permanent until a contract upgrade changes the constant.

---

### Likelihood Explanation

`PERIOD_MINT_CAP` is calibrated to the current protocol scale. The minting curve formula is:

```
yearly_mint = (C_NUM / C_DENOM) * sqrt(total_stake * total_supply)
``` [6](#0-5) 

At `MAX_C_NUM = 500` (5% inflation) and `total_supply ≈ 10B STRK`, weekly rewards exceed 6.5M STRK when `total_stake ≳ 4.5B STRK` (~45% participation). This is a realistic long-term scenario for a major L2 staking protocol. The constant is not adjustable without a full `MintManager` contract upgrade, and there is no on-chain mechanism to detect or warn before the cap is hit. [7](#0-6) 

---

### Recommendation

Replace the file-level `constant` with a governance-controlled storage variable in `MintManager`, allowing the `PERIOD_MINT_CAP` to be updated via a privileged role as protocol TVL grows. Alternatively, decouple the cap from the per-tick batch size so that multiple ticks per week are possible without hitting the quota. At minimum, add an on-chain view function that exposes remaining weekly quota so operators can monitor proximity to the cap.

---

### Proof of Concept

1. Protocol reaches ~45% staking participation; `yearly_mint` exceeds 338M STRK/year (~6.5M/week).
2. L2 `RewardSupplier.request_funds` sends 5+ L2→L1 messages requesting `base_mint_amount` each.
3. A public caller invokes `RewardSupplier.tick()` on L1; it processes 5 messages, minting exactly 6.5M STRK — the weekly quota is now exhausted.
4. Rewards continue to accrue on L2; `unclaimed_rewards` grows; `request_funds` sends more L2→L1 messages.
5. Any subsequent `tick()` call in the same week reaches `checkAndUpdatePeriodicalQuota`, which reverts with `"EXCEED_PERIOD_MINTING"`.
6. The L2 `RewardSupplier` token balance is never replenished.
7. Stakers call `claim_rewards` → `checked_transfer` fails → **unclaimed yield is frozen** until the next weekly period or a contract upgrade. [8](#0-7) [9](#0-8)

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
