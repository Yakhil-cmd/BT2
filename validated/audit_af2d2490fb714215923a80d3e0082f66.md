### Title
Unbounded `move_to_block` Loop in Hashchain Causes Out-of-Gas on All User Transactions After Inactivity - (File: `engine-hashchain/src/hashchain.rs`)

---

### Summary

`Hashchain::move_to_block` contains an unbounded `while` loop that iterates once per NEAR block height skipped since the last Aurora transaction. Because `load_hashchain` calls `move_to_block` unconditionally at the start of every user-facing method (`submit`, `call`, `deploy_code`, `submit_with_args`), a sufficiently long period of engine inactivity causes every subsequent transaction to exhaust NEAR's 300 TGas per-transaction gas limit before any EVM execution occurs. This temporarily freezes all user funds until an admin resets the hashchain via `start_hashchain`.

---

### Finding Description

**Root cause — `move_to_block` in `engine-hashchain/src/hashchain.rs`:**

```rust
// engine-hashchain/src/hashchain.rs  L75-85
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
```

Each iteration calls `compute_block_hashchain`, which concatenates ~370 bytes of data and computes a `keccak256` hash. There is no upper bound on the number of iterations; the loop runs exactly `(current_NEAR_block_height − stored_hashchain_block_height)` times.

**Call path into every user transaction:**

`load_hashchain` (called unconditionally before `require_running`) invokes `move_to_block` whenever the stored height lags behind the current block:

```rust
// engine/src/hashchain.rs  L88-96
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
```

`load_hashchain` is called from both `with_hashchain` and `with_logs_hashchain`, which wrap every externally reachable method:

| Wrapper | Methods wrapped |
|---|---|
| `with_logs_hashchain` | `submit`, `submit_with_args`, `call`, `deploy_code`, `withdraw_wnear_to_router` |
| `with_hashchain` | `register_relayer`, `set_upgrade_delay_blocks`, `pause_precompiles`, `resume_precompiles`, and 8+ connector/admin methods |

**Accumulation mechanism:**

NEAR produces approximately one block per second. The hashchain's `current_block_height` is only advanced when a transaction is processed. During any period of engine inactivity (no transactions submitted), the gap between the stored height and the live NEAR block height grows at ~1 block/second. When the first transaction arrives after the gap, `move_to_block` must iterate over every skipped block.

---

### Impact Explanation

**High — Temporary Freezing of Funds.**

Once the block-height gap exceeds the threshold at which the loop's cumulative keccak cost surpasses NEAR's 300 TGas per-transaction limit, every call to `submit`, `call`, or `deploy_code` reverts with out-of-gas before any EVM logic executes. Users cannot transfer, withdraw, or interact with any EVM contract. The freeze persists until an admin calls `start_hashchain` with a recent `args.block_height`, which resets the stored hashchain to a height close to the current block (making the loop trivially short). Until that admin action occurs, all user funds are inaccessible.

---

### Likelihood Explanation

NEAR produces ~1 block/second. A gap of roughly 150,000–200,000 blocks (~1.7–2.3 days of zero Aurora transactions) is sufficient to exhaust 300 TGas given the per-iteration keccak cost. Realistic triggers include:

- A scheduled maintenance pause (`pause_contract`) lasting more than ~2 days before `start_hashchain` is called.
- A prolonged period of organic inactivity on a low-traffic Aurora silo or testnet deployment.
- Any scenario where the engine is live but receives no transactions for an extended period.

No privileged keys, no external oracle, and no special attacker capability are required. Any ordinary user submitting the first transaction after the gap triggers the failure.

---

### Recommendation

Replace the per-block iteration with a single computation that jumps directly from the stored height to the target height, storing only the cumulative hash. For empty blocks (no transactions), the block hashchain reduces to a deterministic function of `(chain_id, account_id, block_height, previous_hashchain)`, so the intermediate hashes for all skipped empty blocks can be computed in O(log Δ) using repeated squaring / fast-forward hashing, or simply by storing the last-seen height and computing the hash lazily only for the current block. Alternatively, cap the number of iterations per call and require the caller to advance the hashchain incrementally over multiple transactions before submitting EVM work.

---

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled (`start_hashchain` called at block N).
2. Submit no transactions for K blocks (K ≈ 200,000, ~2.3 days on NEAR mainnet).
3. At block N+K, any user submits a `submit` transaction with a valid signed Ethereum transaction.
4. Execution enters `with_logs_hashchain` → `load_hashchain` → `move_to_block(N+K)`.
5. The loop runs K iterations, each computing `keccak256` over ~370 bytes.
6. Total gas consumed by the loop alone exceeds 300 TGas; the NEAR transaction aborts with out-of-gas.
7. Every subsequent `submit`, `call`, and `deploy_code` call fails identically.
8. All EVM user funds are inaccessible until an admin resets the hashchain.

    pub fn compute_block_hashchain(
        &self,
        chain_id: &[u8; 32],
        contract_account_id: &[u8],
        current_block_height: u64,
        previous_block_hashchain: RawH256,

        let txs_hash = self.txs_merkle_tree.compute_hash();

pub fn with_logs_hashchain<I, E, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<SubmitResult, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<SubmitResult, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let log_bloom = bloom::get_logs_bloom(&result.logs);
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &log_bloom,
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}
