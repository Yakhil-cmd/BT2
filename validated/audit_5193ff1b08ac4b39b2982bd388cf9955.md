### Title
Permanent ERC-20 Token Freeze When NEP-141 `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens before issuing a NEAR promise to call `ft_transfer` on the backing NEP-141 token. The refund mechanism that re-mints burned tokens on failure is gated behind the compile-time `error_refund` feature flag. When that feature is absent, any `ft_transfer` failure — including one caused by a non-standard NEP-141 token — permanently destroys the user's ERC-20 tokens with no recovery path. Because `deploy_erc20_token` is permissionless, an attacker can register a malicious NEP-141 token and lure users into a one-way bridge.

### Finding Description

**Step 1 — Permissionless ERC-20 deployment.**
`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` enforces only `require_running`; there is no owner or allowlist check. [1](#0-0) 

Any caller can register an arbitrary NEP-141 account ID and obtain a corresponding ERC-20 on Aurora.

**Step 2 — ERC-20 tokens are burned before the NEAR promise executes.**
The `ExitToNear` precompile (`ExitToNearParams::Erc20TokenParams` branch) resolves the NEP-141 account from the ERC-20 address, then schedules an `ft_transfer` promise. The ERC-20 burn happens inside the EVM call that invokes the precompile; by the time the NEAR promise is dispatched, the tokens are already gone. [2](#0-1) 

**Step 3 — Refund is unconditionally `None` without `error_refund`.**
The `callback_args.refund` field is populated only when the `error_refund` feature is compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [3](#0-2) 

**Step 4 — Callback silently skips refund when `refund` is `None`.**
In `exit_to_near_precompile_callback`, a failed promise falls into the `else if let Some(args) = args.refund` branch. When `refund` is `None`, the `else { None }` arm is taken and no re-mint occurs: [4](#0-3) 

The `refund_on_error` function that would re-mint the burned ERC-20 tokens is never called: [5](#0-4) 

**Step 5 — The existing test confirms permanent loss.**
The workspace test `test_exit_to_near_refund` explicitly documents that without `error_refund` the user's ERC-20 balance is reduced by the exited amount with no recovery:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

**Attack scenario (direct M-07 analog):**
1. Attacker deploys a NEP-141 token whose `ft_transfer_call` succeeds (so tokens can enter Aurora) but whose `ft_transfer` always panics or returns an error for any recipient (analogous to CryptoPunk lacking `safeTransferFrom`).
2. Attacker calls `deploy_erc20_token` — permissionless — to register the ERC-20 mirror on Aurora.
3. Victim bridges NEP-141 tokens into Aurora via `ft_transfer_call` → `ft_on_transfer`; ERC-20 tokens are minted.
4. Victim calls the `ExitToNear` precompile to exit. ERC-20 tokens are burned in the EVM.
5. The NEAR `ft_transfer` promise fails (non-compliant NEP-141).
6. Without `error_refund`, `exit_to_near_precompile_callback` takes the `else { None }` path. No re-mint. Tokens are permanently destroyed.

### Impact Explanation

**Critical — Permanent freezing of funds.** The burned ERC-20 tokens have no recovery path. The NEP-141 tokens remain held by the Aurora contract account (never transferred to the victim), and the ERC-20 tokens no longer exist. The loss is irreversible.

### Likelihood Explanation

**Medium.** The `deploy_erc20_token` entrypoint is permissionless, so any external actor can register a malicious NEP-141 token. The attack requires the production binary to be compiled without `error_refund`. The feature flag is explicitly present in `engine/Cargo.toml` and the codebase treats both compilation modes as valid, making it plausible that some deployments (e.g., Silo instances) ship without it. Users who bridge tokens into Aurora have no on-chain way to detect that exit will fail before their ERC-20 tokens are burned.

### Recommendation

1. **Always enable `error_refund` in production builds**, or remove the conditional compilation and make the refund path unconditional.
2. **Add an allowlist or owner-only gate to `deploy_erc20_token`** so that only vetted, fully NEP-141-compliant tokens can be registered as ERC-20 mirrors — directly mirroring the "whitelist NFTs in the order" recommendation from M-07.
3. **Validate NEP-141 compliance** (e.g., confirm `storage_deposit` is implemented) before completing ERC-20 deployment.

### Proof of Concept

```
1. Deploy malicious NEP-141 token `bad.near`:
   - ft_transfer_call: succeeds (mints tokens to caller)
   - ft_transfer: always panics

2. Call aurora.deploy_erc20_token(borsh("bad.near"))
   → ERC-20 deployed at address 0xBAD...

3. Call bad.near.ft_transfer_call(receiver_id=aurora, amount=1000, msg=<victim_evm_addr>)
   → ft_on_transfer mints 1000 ERC-20 to victim

4. Victim calls ExitToNear precompile (flag=0x01, amount=1000, receiver=victim.near)
   → ERC-20 tokens burned in EVM
   → NEAR promise: bad.near.ft_transfer(receiver_id=victim.near, amount=1000) → PANICS

5. exit_to_near_precompile_callback fires:
   - promise_result(0) = Failed
   - args.refund = None  (error_refund not compiled in)
   - → else { None }  — no re-mint

Result: victim has 0 ERC-20 tokens, 0 NEP-141 tokens. 1000 tokens permanently destroyed.
```
