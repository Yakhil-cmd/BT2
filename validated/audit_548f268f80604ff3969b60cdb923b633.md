### Title
Reverted Invoke Transaction Trace Reports Zero Execute-Phase Resources in `execution_resources` — (`crates/apollo_rpc/src/v0_8/execution.rs`)

### Summary

When a simulated or traced Invoke transaction reverts during the `__execute__` phase, the `From<(ExecutionTransactionTrace, ThinStateDiff)> for TransactionTrace` conversion hard-codes `ExecutionResources::default()` (all-zero) for the execute phase's contribution to `InvokeTransactionTrace.execution_resources`. The resulting total omits the actual gas/steps consumed by the reverted execute phase, even though the blockifier computed and charged for those resources. The `fee_estimation` field is correct (it reads from `receipt.gas`), but the trace-level `execution_resources` is structurally wrong for every reverting invoke.

---

### Finding Description

**Conversion code** (`crates/apollo_rpc/src/v0_8/execution.rs`, lines 147–180):

```rust
let (execute_invocation, execute_execution_resources) = match trace.execute_invocation {
    ExecutionFunctionInvocationResult::Ok(function_invocation) => {
        let execution_resources = function_invocation.execution_resources.clone();
        (FunctionInvocationResult::Ok(function_invocation.into()),
         ExecutionResources::from(execution_resources))
    }
    ExecutionFunctionInvocationResult::Err(revert_reason) => (
        FunctionInvocationResult::Err(revert_reason),
        ExecutionResources::default(),   // ← always zero on revert
    ),
};
// ...
execution_resources: validate_execution_resources.unwrap_or_default()
    + execute_execution_resources          // zero when reverted
    + fee_transfer_execution_resources.unwrap_or_default(),
``` [1](#0-0) [2](#0-1) 

**Why the data is lost upstream**: The execution-crate `InvokeTransactionTrace` (`crates/apollo_rpc_execution/src/objects.rs`) is built from `TransactionExecutionInfo`. When `revert_error` is `Some`, the conversion stores only the error string and discards `execute_call_info` entirely — no resource data is carried in the `Err` variant:

```rust
Some(revert_error) => {
    FunctionInvocationResult::Err(RevertReason::RevertReason(revert_error.to_string()))
}
``` [3](#0-2) 

The execution-crate `InvokeTransactionTrace` struct has no field for the receipt: [4](#0-3) 

**What the blockifier actually computes**: `run_revertible` in `account_transaction.rs` explicitly tracks `execution_steps_consumed` and `sierra_gas_revert_tracker.get_gas_consumed()` for the reverted execute phase and folds them into `get_revert_receipt()`: [5](#0-4) 

Those resources are used for fee charging and are reflected in `receipt.gas`, which is what `tx_execution_output_to_fee_estimation` reads: [6](#0-5) 

So the fee is correct, but the trace-level `execution_resources` is not.

---

### Impact Explanation

Any caller of `starknet_simulateTransactions`, `starknet_traceTransaction`, or `starknet_traceBlockTransactions` on a reverting Invoke transaction receives an `execution_resources` value that is structurally undercounted: it equals only `validate + fee_transfer` resources, omitting the (potentially large) execute-phase gas. The `fee_estimation` sibling field is correct, creating an internal contradiction in the same RPC response. Tools and wallets that consume `execution_resources` for gas budgeting, profiling, or fee estimation will silently underestimate costs for reverting transactions.

This fits the allowed impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a reverting Invoke transaction to the public `starknet_simulateTransactions` endpoint. No special privileges are required. Reverting transactions are common in practice (insufficient gas, failed assertions, etc.).

---

### Recommendation

The receipt-level resource data must be threaded through to the trace. Two approaches:

1. **Carry receipt resources in `InvokeTransactionTrace`** (execution crate): add an `execution_resources` field populated from `transaction_execution_info.receipt.gas` in `TryFrom<TransactionExecutionInfo>`, and use it in the RPC-crate `From` conversion instead of reconstructing from per-phase invocations.

2. **Pass `TransactionExecutionOutput` alongside the trace** to the `From` conversion so the RPC crate can read `receipt.gas` directly for the reverted case.

Either way, the `Err` branch in the `From` conversion must not use `ExecutionResources::default()` when the blockifier has computed non-zero resources for the reverted execute phase.

---

### Proof of Concept

Conceptual Rust unit test (using the existing `execute_simulate_transactions` test harness):

```rust
// Build a reverting invoke (e.g., calldata that triggers a panic in __execute__)
let txs = TxsScenarioBuilder::default()
    .invoke_reverting(ACCOUNT_ADDRESS, CONTRACT_ADDRESS)
    .collect();

let results = execute_simulate_transactions(storage_reader, None, txs, None, true, true);
let output = &results[0];

let TransactionTrace::Invoke(ref invoke_trace) =
    // (after RPC-crate From conversion)
    (output.transaction_trace.clone(), thin_state_diff).into()
else { panic!() };

// fee_estimation reflects actual gas (correct)
assert_ne!(output.fee_estimation.overall_fee, Fee(0));

// execution_resources should include validate + execute + fee_transfer
// but currently only includes validate + fee_transfer (execute = default/zero)
let exec_resources = &invoke_trace.execution_resources;

// This assertion FAILS today: execute-phase steps are missing
assert_ne!(exec_resources.steps, validate_steps + fee_transfer_steps);
// Specifically: exec_resources.steps == validate_steps + fee_transfer_steps
// (execute phase steps are zero, not the actual consumed steps)
```

The discrepancy is concrete and deterministic: every reverting Invoke simulation returns `execution_resources` that excludes the execute-phase gas, while `fee_estimation.overall_fee` correctly accounts for it.
