### Title
Presignature One-Time-Use Not Enforced at API Level Enables Secret Key Extraction via Nonce Reuse — (File: `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both the robust ECDSA and OT-based ECDSA schemes expose `PresignOutput` as a `Clone`-able, serializable type and accept it by shared reference (`&PresignOutput`) in `rerandomize_presign`. This means the library provides no API-level enforcement of the one-time-use property that is cryptographically mandatory for presignatures. A malicious coordinator can drive two signing sessions from the same presignature with different `(h, ε)` inputs (a split-view attack), producing signatures with multiplicatively related nonces from which the aggregate secret key is directly recoverable via standard ECDSA nonce-reuse algebra.

---

### Finding Description

**Root cause — `PresignOutput` is `Clone` and `rerandomize_presign` borrows it:**

**Exploit flow (robust ECDSA, split-view):**

1. Presigning completes; every participant holds a `PresignOutput` containing secret nonce shares `(c_i, e_i, alpha_i, beta_i)` all tied to the same public nonce `R`.
2. A malicious coordinator prepares two distinct `RerandomizationArguments` — `args_A = (h_A, ε_A)` and `args_B = (h_B, ε_B)` — both referencing the same `big_r`.
3. The coordinator presents `args_A` to subset `S_A` and `args_B` to subset `S_B` (both of size `2t+1`). Each participant calls `rerandomize_presign(&presign_output, &args_X)`. Because the function borrows `presign_output`, the underlying secret shares are not erased; the same participant can be asked to rerandomize again with `args_B`.
4. Each subset runs `sign()` and sends its linearized share `s_i` to the coordinator.
5. The coordinator now holds two valid ECDSA signatures `(R_A, s_A)` and `(R_B, s_B)` where `R_A = δ_A · R` and `R_B = δ_B · R`. The nonces are multiplicatively related: `k_A = k / δ_A`, `k_B = k / δ_B`, so `k_A / k_B = δ_B / δ_A` — a known ratio. Standard two-signature nonce-relation algebra recovers the secret key `x`.

The documentation itself confirms this exact attack path and its severity: [7](#0-6) 

The orchestration documentation for OT-based ECDSA similarly states that reuse "is bad" and that destruction is "critical," but provides no code-level enforcement: [8](#0-7) 

---

### Impact Explanation

**Critical — secret key extraction.**

Two signing sessions driven from the same `PresignOutput` with different `(h, ε)` inputs yield signatures whose nonces satisfy a known multiplicative relation. This is sufficient for full secret key recovery via standard ECDSA nonce-reuse algebra. The recovered key is the aggregate threshold secret; all future signatures can be forged unilaterally by the attacker without any participation from honest parties.

This matches the allowed critical impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

**High.** The malicious coordinator role is explicitly within the threat model of the robust ECDSA scheme (the scheme is named "robust" precisely because it is supposed to tolerate malicious participants). The attack requires only that the coordinator present different signing parameters to different participant subsets — a straightforward network-layer manipulation requiring no cryptographic capability. The `Clone` + `&PresignOutput` API makes this trivially reachable: no special tooling is needed beyond calling the public library API twice with the same presignature object.

---

### Recommendation

1. **Consume `PresignOutput` by value in `rerandomize_presign`**: Change the signature to `presignature: PresignOutput` (move semantics). This forces the caller to give up ownership, making accidental reuse a compile-time error.
2. **Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`**: Deriving `Clone` on secret nonce material directly contradicts the one-time-use requirement. Removing it prevents silent duplication.
3. **Remove `Clone` from `RerandomizedPresignOutput`**: Even after rerandomization, the type carries secret scalar shares; cloning it enables the same nonce to be submitted to two concurrent signing sessions.
4. **Retain `Serialize/Deserialize` only behind a feature flag with an explicit "I understand this is dangerous" marker**, or document that deserialized presignatures must be tracked and invalidated by the application layer with a persistent, crash-safe used-set.

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
