### Title
Fixed 2,300-Gas Stipend on `DepositCallGasLimit` Causes Permanent Revert When Depositing FLOW to EVM Smart Contracts with Non-Trivial `receive()` — (`File: fvm/evm/types/call.go`)

---

### Summary

`DepositCallGasLimit` is hardcoded to `21,000 + 2,300 = 23,300` gas, mirroring Solidity's deprecated `transfer()` pattern. Any Cadence transaction that calls `EVMAddress.deposit()` or `CadenceOwnedAccount.deposit()` targeting an EVM smart contract whose `receive()` or `fallback()` function consumes more than 2,300 gas will permanently revert. The depositor cannot bridge FLOW tokens to that contract.

---

### Finding Description

In `fvm/evm/types/call.go`, the gas limit for all deposit direct-calls is defined as:

```go
IntrinsicFeeForTokenTransfer = gethParams.TxGas          // 21,000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23,300
DepositCallGasLimit = DefaultGasLimitForTokenTransfer    // 23,300
``` [1](#0-0) 

`NewDepositCall` hard-wires this limit into every deposit `DirectCall`:

```go
func NewDepositCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: DepositCallGasLimit,   // always 23,300
    }
}
``` [2](#0-1) 

`Account.Deposit()` in the contract handler calls `NewDepositCall` unconditionally:

```go
func (a *Account) Deposit(v *types.FLOWTokenVault) {
    ...
    res, err := a.fch.executeAndHandleCall(
        types.NewDepositCall(bridge, a.address, v.Balance(), bridgeAccount.Nonce()),
        v.Balance(), false,
    )
    panicOnErrorOrInvalidOrFailedState(res, err)
}
``` [3](#0-2) 

`panicOnErrorOrInvalidOrFailedState` panics on any `Failed()` result (i.e., `VMError != nil`):

```go
func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {
    if res != nil && res.Failed() {
        panic(fvmErrors.NewEVMError(res.VMError))
    }
    ...
}
``` [4](#0-3) 

The public Cadence entry points `EVMAddress.deposit()` and `CadenceOwnedAccount.deposit()` in `contract.cdc` both funnel through this path:

```cadence
fun deposit(from: @FlowToken.Vault) {
    ...
    InternalEVM.deposit(from: <-from, to: self.bytes)
    ...
}
``` [5](#0-4) 

The emulator's `mintTo` procedure executes the deposit call via `proc.run()`. If the EVM contract's `receive()` function consumes more than 2,300 gas, `execResult.Failed()` is true, the result is returned as-is, and `panicOnErrorOrInvalidOrFailedState` panics:

```go
if res.Invalid() || res.Failed() {
    proc.state.Reset()
    return res, nil
}
``` [6](#0-5) 

The existing test suite already documents this failure mode explicitly:

```go
t.Run("tokens deposit to an smart contract that doesn't accept native token", func(t *testing.T) {
    ...
    require.Equal(t, res.VMError, gethVM.ErrExecutionReverted)
})
``` [7](#0-6) 

The same fixed limit applies to `NewTransferCall` (COA-to-contract transfers) and `WithdrawCallGasLimit`: [8](#0-7) 

---

### Impact Explanation

Any EVM smart contract whose `receive()` or `fallback()` function requires more than 2,300 gas — proxy contracts, contracts that emit events on receipt, contracts that update storage on receipt, multi-sig wallets, etc. — **permanently cannot receive FLOW tokens via the Cadence bridge deposit path**. The Cadence transaction panics, the deposit is rolled back, and no workaround exists at the caller level because the gas limit is not caller-configurable. This is a cross-VM asset-delivery failure: FLOW tokens held in Cadence cannot be moved into a large class of EVM contracts.

---

### Likelihood Explanation

The entry path is fully unprivileged: any Cadence transaction can call `EVMAddress.deposit()` with an arbitrary target address. Proxy-pattern contracts (e.g., OpenZeppelin `TransparentUpgradeableProxy`, UUPS), vault contracts, and any contract that logs or updates state in `receive()` all exceed the 2,300-gas stipend. This is a well-known class of contracts in EVM ecosystems, making the likelihood of encountering it in practice high.

---

### Recommendation

Allow the caller to supply a gas limit for deposit calls, or raise `DepositCallGasLimit` to a value sufficient for typical `receive()` implementations (e.g., the same ceiling used for `ContractCall` or `Deploy`). At minimum, expose a variant of `EVMAddress.deposit()` that accepts an explicit `gasLimit` parameter so callers can accommodate contracts with non-trivial `receive()` logic.

---

### Proof of Concept

1. Deploy an EVM contract whose `receive()` emits an event or writes to storage (consuming > 2,300 gas).
2. From any Cadence transaction, call:
   ```cadence
   let addr = EVM.EVMAddress(bytes: <contract_address_bytes>)
   addr.deposit(from: <-vault)   // vault holds any non-zero FLOW amount
   ```
3. Observe: the Cadence transaction panics with an EVM error wrapping `execution reverted`.
4. The FLOW tokens are returned to the caller (no loss), but the deposit to that contract is **permanently impossible** through the standard bridge path.

The test at `fvm/evm/emulator/emulator_test.go:83–111` already reproduces this exact failure for a contract with no `receive()` function; the same outcome occurs for any contract whose `receive()` exceeds 2,300 gas. [7](#0-6)

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
