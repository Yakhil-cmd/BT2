### Title
Paused `EXIT_TO_NEAR` Precompile Causes Permanent Freezing of User wNEAR During XCC Execution - (`engine-precompiles/src/xcc.rs`)

### Summary

The XCC precompile (`CrossContractCall`) transfers wNEAR from the user to the engine's implicit address **synchronously** during EVM execution, then schedules the actual NEAR unwrapping as an asynchronous NEAR callback (`withdraw_wnear_to_router`). If the `EXIT_TO_NEAR` precompile is paused between these two steps, the callback fails while the wNEAR transfer is already committed, permanently freezing the user's wNEAR in the engine's implicit address with no recovery path.

### Finding Description

When a user calls the XCC precompile with `required_near > ZERO_YOCTO`, the following two-phase flow occurs:

**Phase 1 (synchronous, within EVM execution):** The XCC precompile calls `transferFrom` on the wNEAR ERC-20 contract to move the user's wNEAR to the engine's implicit address. This is a synchronous EVM sub-call via `handle.call()`. [1](#0-0) 

**Phase 2 (asynchronous NEAR callback):** A promise is scheduled to call `withdraw_wnear_to_router` on the engine itself. This callback calls `engine.call_with_args()` to execute the wNEAR contract's `withdrawToNear` function, which internally calls the `EXIT_TO_NEAR` precompile to unwrap wNEAR into NEAR and route it to the XCC router sub-account. [2](#0-1) [3](#0-2) 

The `EXIT_TO_NEAR` precompile is independently pausable by any account in the `EngineAuthorizer` ACL, which is separate from the engine owner: [4](#0-3) [5](#0-4) 

When `EXIT_TO_NEAR` is paused, calling it returns a fatal EVM error (`ERR_PAUSED`), as confirmed by the existing test: [6](#0-5) 

Because NEAR transactions are independent, the wNEAR `transferFrom` committed in Phase 1 is **not rolled back** when the Phase 2 callback fails. The wNEAR ends up permanently locked in the engine's implicit EVM address (`near_account_to_evm_address(engine_account_id)`), which is not user-controlled and has no recovery function exposed in the contract.

The XCC precompile performs no pre-flight check of the `EXIT_TO_NEAR` pause state before executing the `transferFrom`: [7](#0-6) 

### Impact Explanation

Any user who calls the XCC precompile with a NEAR payment requirement while `EXIT_TO_NEAR` is paused (or becomes paused before the callback executes) loses their wNEAR permanently. The wNEAR is transferred out of the user's EVM balance in Phase 1 and is irrecoverable from the engine's implicit address without an admin-level intervention not present in the current codebase. This constitutes **permanent freezing of user funds**.

### Likelihood Explanation

The `EXIT_TO_NEAR` precompile can be paused by any account in the `EngineAuthorizer` ACL — a set of accounts distinct from the engine owner. A legitimate security pause of `EXIT_TO_NEAR` (e.g., in response to a bridge incident) would silently freeze wNEAR for all in-flight XCC calls that have already completed Phase 1. No attacker compromise is required; the condition arises from normal admin operation of two independent subsystems.

### Recommendation

Before executing the `transferFrom` in Phase 1, the XCC precompile should read the current `PrecompileFlags` from storage and abort with an error if `EXIT_TO_NEAR` is paused:

```rust
// In CrossContractCall::run_with_handle, before the transferFrom block:
let pauser = EnginePrecompilesPauser::from_io(&self.io);
if pauser.is_paused(PrecompileFlags::EXIT_TO_NEAR) {
    return Err(revert_with_message("ERR_EXIT_TO_NEAR_PAUSED"));
}
```

Alternatively, implement a recovery function that allows the engine owner to return stuck wNEAR from the engine's implicit address to the original sender.

### Proof of Concept

1. User holds wNEAR and calls the XCC precompile with `required_near > 0` (e.g., first-time XCC call requiring router deployment storage staking).
2. Phase 1 executes: `transferFrom` moves user's wNEAR to `engine_implicit_address`. EVM transaction commits.
3. An authorized pauser calls `pause_precompiles` with `EXIT_TO_NEAR` flag set (legitimate security action).
4. NEAR runtime executes the scheduled `withdraw_wnear_to_router` callback.
5. `engine.call_with_args()` → wNEAR `withdrawToNear` → `EXIT_TO_NEAR` precompile → `ERR_PAUSED` fatal error.
6. Callback fails; promise chain aborts. User's wNEAR remains in `engine_implicit_address` with no recovery path.

bitflags! {
    /// Wraps unsigned integer where each bit identifies a different precompile.
    #[derive(BorshSerialize, BorshDeserialize, Default)]
    #[borsh(crate = "aurora_engine_types::borsh")]
    pub struct PrecompileFlags: u32 {
        const EXIT_TO_NEAR        = 0b01;
        const EXIT_TO_ETHEREUM    = 0b10;
    }
}

impl PrecompileFlags {
    #[must_use]
    pub fn from_address(address: &Address) -> Option<Self> {
        Some(if address == &exit_to_ethereum::ADDRESS {
            Self::EXIT_TO_ETHEREUM
        } else if address == &exit_to_near::ADDRESS {
            Self::EXIT_TO_NEAR
        } else {
            return None;
        })
    }

    /// Checks if the precompile belonging to the `address` is marked as paused.
    #[must_use]
    pub fn is_paused_by_address(&self, address: &Address) -> bool {
        Self::from_address(address).is_some_and(|precompile_flag| self.contains(precompile_flag))
    }
}
