### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap and Can Front-Run Swaps via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolBinAdditionalFees` applies per-bin additional fees immediately with no cap validation and no timelock. Unlike `setPoolAdminFees`, which is bounded by `maxAdminSpreadFeeE6`, the per-bin path passes values directly to the pool without any upper-bound check. A pool admin can therefore exceed the fee cap enforced by the protocol owner and can front-run pending swaps by raising fees on the active bin in the same block.

---

### Finding Description

The factory exposes two distinct paths for the pool admin to charge spread fees:

**Path 1 — global spread fee (capped):**
`setPoolAdminFees` enforces `newAdminSpreadFeeE6 <= maxAdminSpreadFeeE6` before writing to `poolFeeConfig` and calling `setPoolFees` on the pool.

**Path 2 — per-bin additional fee (uncapped):**
`setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` directly to `setBinAdditionalFees` on the pool with no cap check whatsoever:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool-side handler only validates the bin index, not the fee values:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the maximum settable value is **65 535 E6 = 6.5535%** per bin per direction. These fees are added directly on top of the oracle spread fee in every swap:

```solidity
// MetricOmmPool.sol L999
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

The protocol owner controls `maxAdminSpreadFeeE6` via `setFeeCaps`, which is bounded by the hard ceiling `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%). However, this cap is **never consulted** in the per-bin path. If the owner sets `maxAdminSpreadFeeE6 = 0` to prevent the pool admin from charging any additional spread, the admin can still impose up to 6.5535% per bin through `setPoolBinAdditionalFees`, completely bypassing the owner's cap.

Furthermore, the change takes effect in the same transaction with no timelock, enabling the pool admin to observe a pending swap in the mempool, raise `addFeeBuyE6` or `addFeeSellE6` to `type(uint16).max` on the active bin, and have the swap execute at the inflated fee rate.

---

### Impact Explanation

- **Admin-boundary break**: The protocol owner's `maxAdminSpreadFeeE6` cap is rendered ineffective. Even with the cap set to zero, the pool admin can charge up to 6.5535% additional spread per bin.
- **Direct loss of user funds**: Traders pay more in fees than the protocol's cap system is supposed to allow. The excess fee accrues to LPs in the affected bin (not to the admin directly), but the trader's output is reduced beyond what the fee cap permits.
- **Front-running**: Because the change is immediate and requires no timelock, the pool admin can selectively inflate fees on the active bin for specific transactions, then reset them — a targeted extraction that users cannot anticipate or avoid.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is expected to operate only within the caps and timelocks set by the protocol owner. The `setPoolBinAdditionalFees` path is always available to the pool admin with no preconditions beyond holding the admin role. Any pool whose admin is an EOA or a compromised multisig is immediately exploitable. The attack requires a single transaction and leaves no on-chain evidence distinguishing it from a legitimate fee adjustment.

---

### Recommendation

1. **Add a cap check in `setPoolBinAdditionalFees`**: Validate that `addFeeBuyE6` and `addFeeSellE6` do not exceed `maxAdminSpreadFeeE6` (or a dedicated per-bin cap), consistent with how `setPoolAdminFees` is bounded.
2. **Consider a timelock**: Apply the same propose-then-execute pattern used for price provider rotations to per-bin fee changes, giving traders advance notice before the new rates take effect.
3. **Enforce a combined cap**: Ensure that `globalSpreadFeeE6 + addFeeBuyE6` (and the sell-side equivalent) cannot exceed `HARD_MAX_SPREAD_FEE_E6` at the point of the setter, not just at the point of `setPoolAdminFees`.

---

### Proof of Concept

```
Setup:
  - Protocol owner calls setFeeCaps(..., maxAdminSpreadFeeE6 = 0, ...)
    → pool admin is supposed to be unable to charge any spread fee
  - Pool is active; current bin is bin 0

Attack:
  1. Pool admin observes a pending swap (buy token0) targeting bin 0 in the mempool.
  2. Pool admin calls:
       factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)
     This sets addFeeBuyE6 = 65535 (6.5535%) on bin 0 with no revert.
  3. Swap executes. The swap math computes:
       feeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
     Trader receives ~6.5535% less token0 than they would at the capped fee rate.
  4. Pool admin resets:
       factory.setPoolBinAdditionalFees(pool, 0, 0, 0)

Result:
  - Trader paid 6.5535% additional fee despite maxAdminSpreadFeeE6 = 0.
  - The protocol owner's cap was bypassed via the unchecked per-bin path.
