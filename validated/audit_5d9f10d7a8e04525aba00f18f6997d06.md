### Title
Single Malformed Message from a Malicious Participant Permanently Aborts DKG/Reshare/Refresh for Honest Parties - (File: src/protocol/helpers.rs)

### Summary
`recv_from_others` in `src/protocol/helpers.rs` propagates a deserialization error immediately via `?` when any single received message fails to decode. A malicious participant who is part of the protocol can inject a message with a valid routing header but an invalid msgpack payload, causing the entire DKG, reshare, or refresh protocol to abort for every honest participant that receives it.

### Finding Description

`recv_from_others` is the central gather-all-messages primitive used in `do_keyshare` (the shared implementation of DKG, reshare, and refresh). It loops until it has received exactly one message from every other participant:

```rust
// src/protocol/helpers.rs lines 19-24
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // <-- ? propagates immediately
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

The underlying `Comms::recv` in `src/protocol/internal.rs` deserializes the payload with `rmp_serde`:

```rust
// src/protocol/internal.rs lines 338-340
let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
    rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
Ok((from, decoded?))   // <-- deserialization failure becomes a ProtocolError
```

The message ingestion path (`Comms::push_message`, lines 286-296) only validates that the raw bytes are long enough to contain a `MessageHeader` and that the header parses correctly. It does **not** validate the payload. Any message that carries a syntactically valid 40-byte header is accepted into the `MessageBuffer` unconditionally.

Therefore:
1. A malicious participant constructs a byte string whose first 40 bytes are a valid `MessageHeader` (correct `ChannelTag` + correct `Waitpoint`) and whose remaining bytes are arbitrary garbage.
2. The message is accepted by `push_message` and queued.
3. When an honest participant's `recv_from_others` loop dequeues this message, `Comms::recv` fails at `decoded?`.
4. The `?` in `recv_from_others` propagates the error to `do_keyshare`, which propagates it to the public `keygen`/`reshare`/`refresh` entry points, aborting the protocol permanently.

`recv_from_others` is called twice inside `do_keyshare`:
- Round 1 (commitment hash collection): `src/dkg.rs` lines 422-426
- Round 5 (secret share collection): `src/dkg.rs` lines 514-515

A single poisoned message at either call site terminates the protocol for the targeted honest participant.

### Impact Explanation

**High – Permanent denial of key generation, reshare, or refresh for honest parties.**

Every honest participant that receives the malformed message will abort its own protocol instance. Because the protocol has no retry or skip-bad-sender mechanism, the entire DKG/reshare/refresh session fails. The attacker needs to be a registered participant (which is the documented trust assumption for these protocols) but does not need any cryptographic capability beyond constructing a byte string.

### Likelihood Explanation

Any participant in the protocol can trivially craft the attack: the `ChannelTag` and `Waitpoint` values are deterministic and derivable from public information (the channel derivation uses SHA-256 over public constants; the waitpoint counter is sequential and predictable). The attacker sends the poisoned message once; no repeated interaction is required. The attack is therefore low-effort and high-reliability.

### Recommendation

Replace the hard-fail `?` in `recv_from_others` with a skip-and-continue strategy for deserialization errors, analogous to the pattern already used in `reliable_broadcast_receive_all` (`src/protocol/echo_broadcast.rs` line 179-182):

```rust
// echo_broadcast.rs already does this correctly:
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // bad message is skipped, not fatal
}
```

Apply the same pattern to `recv_from_others`:

```rust
while !seen.full() {
    let Ok((from, msg)) = chan.recv(waitpoint).await else {
        continue;   // skip malformed messages instead of aborting
    };
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

This ensures a single malformed message from a malicious participant cannot abort the protocol for honest parties.

### Proof of Concept

**Setup:** 3-of-3 DKG with participants P0 (honest), P1 (honest), P2 (malicious).

1. All three participants start `keygen`. Each calls `do_keyshare`, which calls `do_broadcast` for session IDs (round 1), then sends commitment hashes via `chan.send_many(wait_round_1, &commitment_hash)`.

2. P2, instead of sending a valid commitment hash, sends a raw byte string to P0 and P1 whose first 40 bytes encode the correct `MessageHeader` for `wait_round_1` on the shared channel, and whose remaining bytes are `[0xFF, 0xFF, 0xFF, ...]` (invalid msgpack).

3. P0's `recv_from_others` loop (dkg.rs line 422) dequeues P2's message. `Comms::recv` (internal.rs line 338-340) calls `rmp_serde::decode::from_slice` on the garbage payload, which returns `Err(...)`. `decoded?` converts this to `ProtocolError::DeserializationError` and returns it.

4. The `?` at helpers.rs line 20 propagates the error out of `recv_from_others`, out of `do_keyshare`, and out of `keygen` for P0. P1 experiences the same failure.

5. DKG is permanently aborted. No key material is produced. P2 has achieved a complete denial of key generation with a single crafted message. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
