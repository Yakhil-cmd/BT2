### Title
Zero `priceProviderTimelock` Bypasses Timelock Guard, Enabling Instant Oracle Swap and LP Drain — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`_validatePoolParameters` does not reject `priceProviderTimelock = 0` when the value is not `type(uint256).max`. Because `proposePoolPriceProvider` computes `executeAfter = block.timestamp + timelock`, a zero timelock sets `executeAfter = block.timestamp`. The guard in `executePoolPriceProviderUpdate` — `block.timestamp < execAfter` — is then immediately false, so propose and execute can be called in the same transaction. A pool admin who created the pool with a zero timelock can atomically rotate the oracle to a malicious price provider, execute a swap at a fabricated price, and drain LP principal.

---

### Finding Description

`createPool` stores `params.priceProviderTimelock` verbatim:

```solidity
// MetricOmmPoolFactory.sol line 213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`_validatePoolParameters` checks every other critical parameter for zero but has no guard on `priceProviderTimelock`:

```solidity
// MetricOmmPoolFactory.sol lines 548–563
function _validatePoolParameters(PoolParameters calldata params) internal view {
    if (params.token0 == address(0) || ...) revert InvalidTokenConfig();
    if (params.admin == address(0)) revert InvalidAdmin();
    _validatePriceProvider(...);
    if (params.adminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    if (params.initialAmount0PerShareE18 == 0 || ...) revert InvalidInitialAmount();
    if (params.minimalMintableLiquidity == 0) revert InvalidMinimalMintableLiquidity();
    // ← no check: priceProviderTimelock > 0 when != type(uint256).max
}
```

`proposePoolPriceProvider` computes the deadline:

```solidity
// MetricOmmPoolFactory.sol line 487
uint256 executeAfter = block.timestamp + timelock;  // = block.timestamp when timelock == 0
```

`executePoolPriceProviderUpdate` enforces the timelock with a strict-less-than comparison:

```solidity
// MetricOmmPoolFactory.sol line 499
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
```

When `timelock = 0`, `execAfter = block.timestamp`, so `block.timestamp < block.timestamp` is `false` — the revert never fires. Both calls succeed in the same transaction.

---

### Impact Explanation

An attacker who is the pool admin (or who controls it) can:

1. Deploy a pool with `priceProviderTimelock = 0` (permissionless via `createPool`).
2. Attract LP deposits.
3. In a single atomic transaction:
   - Call `proposePoolPriceProvider(pool, maliciousOracle)` — sets `pendingPriceProviderExecuteAfter[pool] = block.timestamp`.
   - Call `executePoolPriceProviderUpdate(pool)` — timelock check passes immediately; pool's `priceProvider` is replaced with `maliciousOracle`.
   - Call `pool.swap(...)` — `_getBidAndAskPriceX64()` reads from `maliciousOracle`, which returns an arbitrarily skewed bid/ask; the swap drains LP token balances at a fabricated price.
4. Optionally restore the original oracle to obscure the attack.

The pool's entire LP principal in both tokens is at risk. The `collectFees` path is unaffected — this is a direct swap-path drain.

---

### Likelihood Explanation

`createPool` is fully permissionless once `poolDeployer` is set. Any caller can supply `priceProviderTimelock = 0`. The pool admin role is assigned at creation time and can be the same address as the creator. No off-chain monitoring or LP action can prevent the attack because propose and execute happen atomically in one block. The only precondition is that the pool has LP deposits, which a malicious creator can solicit after deployment.

---

### Recommendation

Add a non-zero check for `priceProviderTimelock` in `_validatePoolParameters` when the value is not the immutable sentinel:

```solidity
// in MetricOmmPoolFactory._validatePoolParameters
if (params.priceProviderTimelock != type(uint256).max && params.priceProviderTimelock == 0) {
    revert InvalidPriceProviderTimelock();
}
```

This mirrors the pattern already applied to `minimalMintableLiquidity` and `initialAmount*PerShareE18`, and directly closes the zero-timelock bypass.

---

### Proof of Concept

```solidity
// Attacker contract
function attack(MetricOmmPoolFactory factory, address pool, address maliciousOracle) external {
    // Step 1: propose — executeAfter = block.timestamp (timelock == 0)
    factory.proposePoolPriceProvider(pool, maliciousOracle);

    // Step 2: execute — block.timestamp < block.timestamp is false → succeeds immediately
    factory.executePoolPriceProviderUpdate(pool);

    // Step 3: swap at fabricated price — drains LP funds
    IMetricOmmPool(pool).swap(
        address(this), true, type(int128).max, 0, "", ""
    );
}
```

The root cause is in `_validatePoolParameters` (no `priceProviderTimelock > 0` guard) and the strict-less-than check in `executePoolPriceProviderUpdate` (line 499), which together allow a zero-timelock pool to bypass the entire propose-wait-execute safety window.
