Audit Report

## Title
ckBTC Minter Withdrawal Processing Halts When Bitcoin Canister Returns Fewer Than 100 Fee Percentiles — (`rs/bitcoin/ckbtc/minter/src/fees/mod.rs`, `rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary

`estimate_nth_fee` unconditionally returns `None` when the Bitcoin canister supplies fewer than 100 fee-percentile entries. This `None` propagates through `estimate_fee_per_vbyte` to `submit_pending_requests`, which silently returns without processing any queued withdrawal. Because the ckBTC ledger burn is committed before the Bitcoin transaction is constructed, affected users have already surrendered their ckBTC with no BTC sent and no automatic reimbursement path. Recovery requires an NNS upgrade proposal, which takes days.

## Finding Description

**Root cause — hard length check with no fallback:**

In `rs/bitcoin/ckbtc/minter/src/fees/mod.rs` lines 103–107, `estimate_nth_fee` returns `None` for any fee-percentile vector shorter than 100 entries on Mainnet/Testnet:

```rust
Network::Mainnet | Network::Testnet => {
    if fee_percentiles.len() < 100 || nth >= 100 {
        return None;
    }
```

`estimate_median_fee` (line 17) calls `estimate_nth_fee(..., 50)`, so any sub-100-entry response from the Bitcoin canister yields `None`.

**Propagation — no fallback to cached state:**

`estimate_fee_per_vbyte` in `rs/bitcoin/ckbtc/minter/src/lib.rs` lines 235–253 receives `None` from `estimate_median_fee` and propagates it directly upward without consulting `s.last_fee_per_vbyte` (which is already stored in state and updated on every successful call):

```rust
match fee_estimator.estimate_median_fee(&fees) {
    Some(median_fee) => { ... Some(median_fee) }
    None => None,   // cached last_fee_per_vbyte is never consulted
}
```

**Silent abort in the timer loop:**

`submit_pending_requests` in `rs/bitcoin/ckbtc/minter/src/lib.rs` lines 358–361 returns unconditionally on `None`:

```rust
let fee_millisatoshi_per_vbyte = match estimate_fee_per_vbyte(runtime).await {
    Some(fee) => fee,
    None => return,   // entire pending queue is skipped, no log at Error/Warn level
};
```

Every timer tick executes this path, leaving every `RetrieveBtcRequest` in `Pending` state indefinitely.

**No on-chain bypass:** The `Mode` enum (`ReadOnly`, `RestrictedTo`, `DepositsRestrictedTo`, `GeneralAvailability`) controls access but provides no mechanism to force-process pending requests with a cached or operator-supplied fee rate. The only recovery path is an NNS upgrade proposal.

**Secondary DoS — `estimate_withdrawal_fee` query trap:**

`rs/bitcoin/ckbtc/minter/src/main.rs` lines 229–230 calls `.expect(...)` on `last_median_fee_per_vbyte`, which traps the canister on any query call while the field is `None`, making fee estimation unavailable to users during the degraded period.

## Impact Explanation

All pending ckBTC → BTC withdrawals are frozen for the duration of the degraded condition. Users have already irrevocably burned their ckBTC on the ledger; no BTC transaction is ever submitted; `retrieve_btc_status` remains `Pending` indefinitely. The freeze persists across every timer tick until the Bitcoin canister resumes returning ≥ 100 percentiles or an NNS upgrade is executed. This constitutes a significant Chain Fusion / ck-token security impact with concrete, demonstrable user-fund harm, matching the **High ($2,000–$10,000)** impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

The trigger condition — the Bitcoin canister returning fewer than 100 fee-percentile entries — is a realistic natural occurrence under low Bitcoin mempool activity and does not require any attacker action. The mainnet upgrade proposal `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` documents that on 2025-06-21 three withdrawal transactions became stuck and required an emergency NNS upgrade, confirming the real-world exploitability of this class of failure. The minimum-fee floor added in PR #5742 (1,500 ms/vbyte for Mainnet) addresses only the anomalously-low-fee sub-case; it does not protect against a short percentile vector, which is the structural root cause. No privileged access or attacker capability is required; the condition can arise from normal Bitcoin network dynamics.

## Recommendation

1. **Fallback to cached percentiles:** In `estimate_fee_per_vbyte`, when `estimate_median_fee` returns `None` due to a short vector, fall back to `s.last_median_fee_per_vbyte` (already stored in state) so pending withdrawals continue to be processed using the last known-good fee rate.
2. **Governance-accessible fee override:** Add an upgrade-arg or NNS-callable setter that allows operators to supply a manual fee rate, bypassing the Bitcoin canister oracle entirely when it is degraded.
3. **Explicit error surfacing:** Replace the silent `return` in `submit_pending_requests` with a logged, metered error at `Priority::Info` or higher so operators are alerted immediately when the fee oracle is degraded.
4. **Guard `estimate_withdrawal_fee`:** Replace the `.expect(...)` on `last_median_fee_per_vbyte` with a graceful error return to avoid trapping the query endpoint.

## Proof of Concept

1. User calls `retrieve_btc` / `retrieve_btc_with_approval`; ckBTC is burned on the ledger; `RetrieveBtcRequest` enters the `Pending` queue.
2. Bitcoin canister's `bitcoin_get_current_fee_percentiles` returns a vector with fewer than 100 entries (e.g., near-empty mempool or canister anomaly).
3. Every timer tick: `submit_pending_requests` → `estimate_fee_per_vbyte` → `estimate_median_fee` → `estimate_nth_fee` returns `None` → `submit_pending_requests` returns at `lib.rs` line 360 without touching the pending queue.
4. State: user's ckBTC is burned, no BTC transaction is ever submitted, `retrieve_btc_status` remains `Pending` indefinitely.
5. Reproducible as a deterministic integration test using PocketIC: configure a mock `CanisterRuntime` whose `get_current_fee_percentiles` returns a vector of length < 100, enqueue a `RetrieveBtcRequest`, advance the timer, and assert the request remains in `Pending` state with no Bitcoin transaction submitted.
