### Title
Gateway Stateless Validator Unconditionally Skips `max_l2_gas_amount` Enforcement for Declare Transactions, Admitting Unbounded L2 Gas Claims - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` contains a deliberate but unguarded branch that silently skips the `max_l2_gas_amount` upper-bound check whenever the incoming transaction is a `Declare`. Any user can submit a `RpcDeclareTransaction::V3` with `l2_gas.max_amount` set to `u64::MAX` (≈18.4 × 10¹⁸), which is ~15 billion times the production limit of 1,210,000,000. The gateway admits the transaction; no downstream component re-enforces the upper bound before the transaction enters the mempool.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs` lines 78–85:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The `if let RpcTransaction::Declare(_) = tx { }` arm is an empty body — it matches Declare and does nothing, falling through to `Ok(())`. The `else if` branch that enforces `max_l2_gas_amount` is therefore unreachable for Declare transactions. The TODO comment confirms this is a known, unresolved gap.

The production configuration sets `max_l2_gas_amount = 1_210_000_000` for all other transaction types. The blockifier's `check_fee_bounds` (in `crates/blockifier/src/transaction/account_transaction.rs`) only checks the **lower** bound (`minimal_gas_amount <= max_amount`); it never re-checks the upper bound. `verify_can_pay_committed_bounds` checks whether the account balance covers `max_amount × max_price_per_unit`, but with `max_amount = u64::MAX` and `max_price_per_unit = min_gas_price` (the minimum non-zero value enforced by the price check that does apply to Declare), the committed fee overflows practical balance limits and the transaction is rejected at execution time — but only after it has already been admitted to the mempool and consumed batcher resources.

The structural analog to the external report is exact:

| External bug | This finding |
|---|---|
| `IS_CURVE_V2` only initialized under `CHAIN_ID == MAINNET`, defaults `false` elsewhere | `max_l2_gas_amount` check only executed for non-Declare types; Declare silently defaults to "no upper bound" |
| Wrong `add_liquidity` branch taken on sidechains | Invalid Declare transactions admitted to mempool on every chain |
| Root cause: conditional initialization of a flag | Root cause: conditional execution of an admission check |

### Impact Explanation

The gateway's stateless validator is the first and only enforcement point for `max_l2_gas_amount`. No equivalent upper-bound check exists in the blockifier, the mempool's `validate_tx` path, or the batcher. A Declare transaction with `l2_gas.max_amount = u64::MAX` passes `StatelessTransactionValidator::validate`, passes `validate_by_mempool`, and enters the mempool. The batcher will attempt to execute it; `verify_can_pay_committed_bounds` will reject it only if the account balance is insufficient to cover the inflated committed fee. An attacker controlling an account with a large STRK balance can submit Declare transactions with arbitrarily large gas claims that execute successfully, bypassing the protocol's intended per-transaction gas ceiling. Even without a large balance, the attacker can flood the mempool with Declare transactions that pass gateway admission but fail at execution, wasting batcher and mempool resources.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The trigger requires only a valid Starknet account and the ability to craft a `RpcDeclareTransaction::V3` with an oversized `l2_gas.max_amount`. No privileged access, special role, or race condition is needed. The gateway is publicly reachable via HTTP. The existing test `valid_l2_gas_amount_on_declare` (lines 191–201 of the test file) explicitly confirms and documents that this bypass is reachable in production.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx { }` arm and apply the same `max_l2_gas_amount` upper-bound check uniformly to all transaction types, or introduce a separate (potentially higher) `max_l2_gas_amount_declare` config value if Declare transactions legitimately require a different ceiling. The TODO comment should be resolved rather than left open.

```rust
// Apply to all transaction types uniformly:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

### Proof of Concept

1. Construct a `RpcDeclareTransaction::V3` with:
   - `l2_gas.max_amount = GasAmount(u64::MAX)` (18,446,744,073,709,551,615)
   - `l2_gas.max_price_per_unit = GasPrice(min_gas_price)` (satisfies the price check)
   - All other fields valid (valid Sierra class, correct nonce, etc.)
2. Submit via `POST /add_transaction` to the gateway.
3. `StatelessTransactionValidator::validate` reaches `validate_resource_bounds`. The `if let RpcTransaction::Declare(_) = tx { }` arm matches and returns `Ok(())` without checking `max_l2_gas_amount`.
4. The transaction passes all remaining stateless checks and proceeds to stateful validation.
5. `validate_by_mempool` and `skip_stateful_validations` run; neither enforces `max_l2_gas_amount`.
6. The transaction is inserted into the mempool with `l2_gas.max_amount = u64::MAX`, violating the protocol's intended ceiling of 1,210,000,000.

The existing test `valid_l2_gas_amount_on_declare` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` lines 173–201 directly demonstrates this: it passes `max_amount = GasAmount(200)` against a config with `max_l2_gas_amount = 100` for a Declare transaction and asserts `Ok(())`.     fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
