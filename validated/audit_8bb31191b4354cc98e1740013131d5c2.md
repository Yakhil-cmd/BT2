### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Loss of Staker/Delegator Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

The `stake()` and `change_reward_address()` functions in `staking.cairo`, and `enter_delegation_pool()` and `change_reward_address()` in `pool.cairo`, accept a `reward_address` parameter without validating it is non-zero. A staker or delegator can register or update their reward address to the zero address. All subsequent reward transfers — including those triggered by `claim_rewards()` and `unstake_action()` — will silently send tokens to the zero address, permanently destroying the yield.

---

### Finding Description

In `staking.cairo::stake()`, the only validation applied to `reward_address` is that it is not a registered token address:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [1](#0-0) 

The zero address passes this check because it is not a registered token. The same pattern is repeated in `staking.cairo::change_reward_address()`: [2](#0-1) 

And in `pool.cairo::enter_delegation_pool()`:

```cairo
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [3](#0-2) 

Once a zero `reward_address` is stored, every reward transfer flows through `send_rewards_to_staker()`:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [4](#0-3) 

And through `pool.cairo::claim_rewards()`:

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [5](#0-4) 

Both transfer directly to the stored `reward_address` with no further validation. Tokens sent to the zero address on Starknet are unrecoverable.

Notably, the protocol's own error catalogue defines a `ZERO_ADDRESS` error: [6](#0-5) 

This confirms the intent to guard against zero addresses, but the guard is absent from all four `reward_address` acceptance points.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Any staker or delegator who registers or updates their `reward_address` to the zero address (accidentally, via a buggy front-end, or through a misconfigured script) will have all accrued rewards transferred to the zero address on every `claim_rewards()` or `unstake_action()` call. The tokens are permanently destroyed. The principal stake is unaffected, but the entire yield stream for that participant is irreversibly lost.

---

### Likelihood Explanation

**Medium.** The entry path requires only a standard user transaction — `stake()`, `change_reward_address()`, or `enter_delegation_pool()` — with a zero value for `reward_address`. No privileged role is needed. The risk is realistic in practice: front-end bugs, copy-paste errors, or scripted deployments that fail to populate the reward address field are common causes of zero-address submissions. The protocol's own spec defines a `ZERO_ADDRESS` error, indicating the authors anticipated this class of mistake.

---

### Recommendation

Add a non-zero assertion for `reward_address` in all four acceptance points:

1. `staking.cairo::stake()` — after existing token-address check
2. `staking.cairo::change_reward_address()` — after existing token-address check
3. `pool.cairo::enter_delegation_pool()` — after existing token-address check
4. `pool.cairo::change_reward_address()` — after existing token-address check

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing `ZERO_ADDRESS` error already defined in the error catalogue and aligns with the constructor-level zero-address checks applied to other critical addresses in the protocol.

---

### Proof of Concept

1. A staker calls `staking.stake(reward_address: 0, operational_address: <valid>, amount: <min_stake>)`.
2. The call succeeds — the zero address is not a registered token, so all assertions pass.
3. The staker accumulates rewards over multiple epochs via `update_global_index`.
4. Anyone calls `staking.claim_rewards(staker_address)` (or the staker calls `unstake_action`).
5. `send_rewards_to_staker` executes `checked_transfer(recipient: 0, amount: rewards)`.
6. All accrued yield is transferred to the zero address and permanently lost.

The same sequence applies to a delegator calling `pool.enter_delegation_pool(reward_address: 0, amount: ...)` followed by `pool.claim_rewards(pool_member)`. [7](#0-6) [8](#0-7)

                        staker_address, new_address: reward_address, old_address,

            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;

                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self

                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();

                        pool_member, reward_address, amount: rewards,
