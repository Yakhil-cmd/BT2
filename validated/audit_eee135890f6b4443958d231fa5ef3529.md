### Title
Missing `TriplePub.participants` Validation in OT-Based ECDSA Presigning Allows Cross-Session Triple Reuse, Corrupting Presign Outputs — (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA presigning function (`presign`) accepts `TriplePub` values that carry an embedded `participants` field recording which participant set generated the triple. This field is **never validated** against the presigning participant set. A malicious participant can supply a `TriplePub` from a different triple generation session (a different participant set), causing the protocol to proceed past initialization, broadcast private key-blinded shares to all participants, and then abort at a runtime consistency check — after sensitive material has already been exposed.

---

### Finding Description

`TriplePub` is the public output of triple generation and explicitly records the participant set that produced it:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    /// The threshold which will be able to reconstruct it.
    pub threshold: ReconstructionLowerBound,
}
``` [1](#0-0) 

This field is populated during triple generation:

```rust
TriplePub {
    big_a, big_b, big_c,
    participants: participants.clone().into(),
    threshold,
}
``` [2](#0-1) 

However, in `presign()`, the only cross-phase check performed is that the threshold values match. The participant set recorded in `TriplePub` is **explicitly skipped** with a comment acknowledging the omission:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [3](#0-2) 

The reasoning in the comment is flawed: a participant can possess a valid `TripleShare` from one session and pair it with a `TriplePub` from a completely different session (different participant set, same threshold). The library accepts this at initialization time without error.

Inside `do_presign`, the Lagrange coefficient is computed from the **presigning** participant set, not from `TriplePub.participants`:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
let x_prime_i = lambda_me * private_share;
``` [4](#0-3) 

Then, before the consistency checks fire, all participants broadcast their private-key-blinded shares to every other participant:

```rust
chan.send_many(wait0, &e_i)?;
// ...
chan.send_many(wait1, &(alpha_i, beta_i))?;
``` [5](#0-4) 

where `beta_i = lambda_me * (x_i + b_i)` — a linear combination of the participant's **private key share** `x_i` and their triple share `b_i`. Only after these values are broadcast does the protocol check:

```rust
if big_e != (ProjectivePoint::GENERATOR * e).to_affine() {
    return Err(ProtocolError::AssertionFailed("received incorrect shares of kd"...));
}
// ...
if (ProjectivePoint::GENERATOR * alpha != big_k + big_a)
    || (ProjectivePoint::GENERATOR * beta != big_x + big_b) {
    return Err(ProtocolError::AssertionFailed("received incorrect shares of additive triple phase."...));
}
``` [6](#0-5) 

When a mismatched `TriplePub` is used, these checks fail — but only after `beta_i` values have been sent to all participants.

---

### Impact Explanation

**High — Corruption of presign outputs and potential private key share leakage.**

1. **Corrupted presign output**: Honest participants run the full presigning protocol, broadcast their private-key-blinded shares, and then the protocol aborts at the consistency check. The presignature is unusable. Honest parties cannot distinguish this from a network fault.

2. **Private key share exposure window**: `beta_i = lambda_me * (x_i + b_i)` is broadcast to all participants before the abort. Any participant who knows `b_i` for a target participant (e.g., a malicious participant who was the triple generator and knows their own shares, or who colluded during triple generation) can compute `lambda_me * x_i`. By orchestrating multiple failed presigning sessions with different participant sets (yielding different Lagrange coefficients), they can solve for `x_i` directly — extracting the private key share.

3. **Denial of signing**: A malicious participant can repeatedly inject mismatched triples to prevent any presignature from completing, permanently blocking signing for honest parties.

---

### Likelihood Explanation

**Medium.** The attacker must be a participant in the presigning session and must be able to supply a `TriplePub` from a different triple generation session. This is a realistic scenario: a library caller controls their own `PresignArguments` struct (including `triple0` and `triple1`), and the library imposes no binding between the `TriplePub.participants` field and the presigning participant set. No privileged access is required beyond being a legitimate participant.

---

### Recommendation

In `presign()`, after constructing the `ParticipantList`, validate that `TriplePub.participants` for both triples is a superset of (or exactly matches) the presigning participant set:

```rust
let triple0_participants = ParticipantList::new(&args.triple0.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants = ParticipantList::new(&args.triple1.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;

for p in participants.participants() {
    if !triple0_participants.contains(*p) || !triple1_participants.contains(*p) {
        return Err(InitializationError::BadParameters(
            "Presigning participants must be a subset of triple generation participants".to_string(),
        ));
    }
}
```

This check should be added at initialization time, before any protocol messages are sent, consistent with how other participant-set invariants are enforced across the codebase. [7](#0-6) 

---

### Proof of Concept

1. Participants `{P1, P2, P3}` run `generate_triple` to produce triple `T_A` with `TriplePub.participants = [P1, P2, P3]`.
2. Participants `{P1, P2, P4}` run `generate_triple` to produce triple `T_B` with `TriplePub.participants = [P1, P2, P4]`.
3. A presigning session is initiated with participant set `{P1, P2, P3}`.
4. Malicious participant `P2` supplies `PresignArguments { triple0: (share_from_T_B, T_B.pub), ... }` — a share from `T_B` paired with `T_B`'s public values, which record `participants = [P1, P2, P4]`.
5. `presign()` accepts this without error (only threshold is checked, not `TriplePub.participants`).
6. `do_presign` runs: all participants broadcast `e_i` and `(alpha_i, beta_i)` — including `beta_i = lambda_me * (x_i + b_i)` — to all other participants.
7. The consistency check `big_e == e*G` fails because `P2`'s contribution to `e` is based on `T_B`'s shares while `big_e` is from `T_A`'s public values (used by `P1` and `P3`).
8. The protocol aborts. The presignature is corrupted. `beta_i` values have already been broadcast.
9. If `P2` knows `b_i` for any participant (e.g., their own), they have `lambda_me * x_i`. Repeating with different participant sets (different Lagrange coefficients) allows solving for `x_i`. [1](#0-0) [8](#0-7)

    participants: &[Participant],
    me: Participant,
    args: PresignArguments,

    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
