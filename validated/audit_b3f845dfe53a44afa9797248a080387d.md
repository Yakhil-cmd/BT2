### Title
Unauthenticated `X-Near-Pool-Coordinator-Query` Header Grants Coordinator Privilege to Any External Caller - (File: chain/jsonrpc/src/lib.rs)

### Summary

The sharded-RPC coordinator protocol uses a plain HTTP header (`X-Near-Pool-Coordinator-Query`) to signal that a request is an internal coordinator-forwarded sub-request. The `rpc_handler` function accepts this header from **any caller on the public RPC port** without any authentication, secret, or IP restriction. Any unprivileged user who knows the header name (which is a public constant) can set `RequestSource::Coordinator` on their own requests, bypassing the scatter-gather routing layer and forcing the node to process requests locally as if they were trusted internal sub-requests.

### Finding Description

In `chain/jsonrpc/src/lib.rs`, the `rpc_handler` function determines the `RequestSource` solely by checking for the presence of the `X-Near-Pool-Coordinator-Query` header:

```rust
let source = if headers.contains_key(SHARDED_RPC_COORDINATOR_HEADER) {
    RequestSource::Coordinator
} else {
    RequestSource::User
};
``` [1](#0-0) 

The header name is a public constant defined in the client library:

```rust
pub const SHARDED_RPC_COORDINATOR_HEADER: &'static str = "X-Near-Pool-Coordinator-Query";
``` [2](#0-1) 

There is no authentication, shared secret, IP allowlist, or any other validation of whether the caller is actually a legitimate coordinator node. The `RequestSource` value then flows into `process_sharded_method_call`, which dispatches to the **local handler** (bypassing scatter-gather) when `source == RequestSource::Coordinator`:

```rust
match source {
    RequestSource::User => sharded_handler(request).await,
    RequestSource::Coordinator => serialize_response(local_handler(request).await?),
}
``` [3](#0-2) 

The `Coordinator` source also gates additional logic in `changes_in_block` and `changes_in_block_by_type`, where coordinator requests trigger stricter `ensure_chunks_applied` checks that are skipped for user requests: [4](#0-3) 

### Impact Explanation

**RPC proof/query trust** is the exact target scope. The `RequestSource::Coordinator` privilege controls which execution path handles a query. An external attacker who sets the coordinator header can:

1. **Force local-only processing** on any sharded EXPERIMENTAL method (`EXPERIMENTAL_view_account`, `EXPERIMENTAL_view_state`, `EXPERIMENTAL_view_access_key`, `EXPERIMENTAL_view_access_key_list`, `EXPERIMENTAL_call_function`, `EXPERIMENTAL_view_code`, `EXPERIMENTAL_view_gas_key_nonces`, `EXPERIMENTAL_light_client_proof`, `EXPERIMENTAL_receipt`, `block_effects`, `changes`). When the node does not track the queried shard, the local handler returns `UNAVAILABLE_SHARD` or `SHARD_NOT_APPLIED` instead of forwarding — the attacker can use this to probe which shards a node tracks.

2. **Obtain different (coordinator-path) query results** than a normal user would receive. For `changes_in_block` and `changes_in_block_by_type`, the coordinator path enforces `ensure_chunks_applied` and returns `ShardNotApplied` errors that are not returned on the user path, leaking internal node state about chunk application status.

3. **Bypass scatter-gather aggregation**: For `block_effects` and `changes`, a coordinator-flagged request is answered with only the local node's partial data rather than the full merged result across all shards. An attacker can use this to obtain deliberately incomplete state-change sets that differ from what a legitimate user would receive, potentially misleading downstream consumers about the completeness of the response.

The corrupted invariant is **RPC query result identity**: the same query with the same parameters returns structurally different results (partial vs. full, error vs. success) depending solely on an unauthenticated header the attacker controls.

### Likelihood Explanation

The header name `X-Near-Pool-Coordinator-Query` is a public constant in the open-source client library. Any user of the public JSON-RPC port (port 3030, which is the default and is typically internet-facing) can include this header in any POST request. No special knowledge, credentials, or network position is required. The attack is trivially reproducible with a single `curl` command.

### Recommendation

1. **Authenticate the coordinator header**: Introduce a shared secret (e.g., a randomly generated token at node startup, or a configured pre-shared key) that must accompany the `X-Near-Pool-Coordinator-Query` header. The receiving node validates the secret before granting `RequestSource::Coordinator`.

2. **Alternatively, restrict by source IP**: Only accept the coordinator header from IP addresses listed in the `sharded_rpc` pool configuration. The pool configuration already knows the addresses of peer nodes.

3. **Separate the coordinator endpoint**: Expose the coordinator-only methods on a separate, non-public port (e.g., bound to localhost or an internal network interface only), so external callers cannot reach it at all.

### Proof of Concept

```bash
# Any external user can set RequestSource::Coordinator by adding the header.
# This forces local-only processing on a node that may not track the queried shard,
# returning SHARD_NOT_APPLIED instead of the full scatter-gathered result.

curl -s -X POST http://<rpc-node>:3030/ \
  -H "Content-Type: application/json" \
  -H "X-Near-Pool-Coordinator-Query: 1" \
  -d '{
    "jsonrpc": "2.0",
    "id": "dontcare",
    "method": "changes",
    "params": {
      "block_id": "<block_hash>",
      "changes_type": "account_changes",
      "account_ids": ["<account_on_untracked_shard>"]
    }
  }'
# Returns SHARD_NOT_APPLIED (coordinator path) instead of the full merged result (user path).
# Confirms the node does not track that shard, leaking internal topology.
# Also returns structurally different (partial) data for block_effects queries.
```

The root cause is the absence of any origin validation in `rpc_handler` before assigning `RequestSource::Coordinator`. [5](#0-4) [6](#0-5)
