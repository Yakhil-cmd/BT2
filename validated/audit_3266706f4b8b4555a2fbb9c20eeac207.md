### Title
Zero `priceProviderTimelock` Allows Pool Admin to Atomically Swap Oracle in One Block, Enabling Instant LP Fund Drain - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`createPool` accepts `priceProviderTimelock = 0` without validation. When a pool is created in mutable-oracle mode with a zero timelock, the pool admin can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in the same block (or same transaction via a contract), atomically replacing the oracle with a malicious one and immediately executing swaps at manipulated prices to drain LP principal.

### Finding Description

`_validatePoolParameters` validates fees, tokens, admin, and initial amounts, but imposes **no lower bound on `priceProviderTimelock`** when the value is not `type(uint256).max`. [1](#0-0) 

The only branch on `priceProviderTimelock` is the immutability check: [2](#0-1) 

So `priceProviderTimelock = 0` produces `immutablePriceProvider = false` — mutable mode — with a stored timelock of zero.

When `proposePoolPriceProvider` is called, `executeAfter` is computed as: [3](#0-2) 

`block.timestamp + 0 = block.timestamp`. The guard in `executePoolPriceProviderUpdate` is: [4](#0-3) 

`block.timestamp < block.timestamp` is always `false`, so the revert never fires. Both calls can be batched in a single transaction through a contract, atomically replacing the oracle before any LP can react.

### Impact Explanation

After the oracle is swapped to a malicious `IPriceProvider` that returns an extreme bid/ask, every subsequent `swap` call prices trades against the manipulated quote. The pool's swap math trusts the oracle unconditionally: [5](#0-4) 

An attacker who controls the oracle can set the ask price arbitrarily low (for `zeroForOne`) or the bid price arbitrarily high (for the reverse direction), extracting all token0 or token1 from the pool's `binTotals` in a single swap. LP principal is fully at risk.

### Likelihood Explanation

`createPool` is permissionless. Any caller can deploy a pool with `priceProviderTimelock = 0` and themselves as `admin`. The pool appears legitimate (real tokens, real initial oracle, real fee config). LPs who deposit without inspecting `factory.priceProviderTimelock(pool)` are exposed. The attack requires only two factory calls batched in one transaction — no flash loan, no governance delay, no special privilege beyond pool admin.

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters` for mutable-oracle pools:

```solidity
if (params.priceProviderTimelock != type(uint256).max && params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

`MIN_PRICE_PROVIDER_TIMELOCK` should be at least 24 hours (86400 seconds) to give LPs meaningful time to withdraw before an oracle change takes effect. Alternatively, expose the timelock value prominently in `PoolCreated` and require integrators to surface it to depositors.
