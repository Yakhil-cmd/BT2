### Title
Attacker Bypasses `__validate__` Signature Check to Inject Unsigned Invoke Transaction into Mempool, Blocking Victim's Post-Deploy Invoke — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function grants a signature-verification bypass to any invoke transaction with `nonce=1` sent to an undeployed account, provided `account_tx_in_pool_or_recent_block` returns `true` for that address. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type in the pool — not exclusively a `deploy_account` — an unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a second invoke transaction for the same address with an arbitrary (invalid) signature. The gateway accepts it without calling `__validate__`, and the attacker's transaction occupies nonce=1 in the mempool. The victim's legitimate invoke transaction is then rejected with `DuplicateNonce` (or forced to pay a fee-escalation premium), permanently blocking the intended post-deploy UX flow.

---

### Finding Description

**Broken invariant:** Every invoke transaction admitted to the mempool must have passed `__validate__` (signature verification) unless the account is provably the one that submitted the pending `deploy_account`.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip `__validate__`) when three conditions hold simultaneously:
1. The transaction is an `Invoke` with `nonce == 1`.
2. The on-chain `account_nonce == 0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The third condition is evaluated by: [2](#0-1) 

which returns `true` if **any** transaction — of any type, from any submitter — exists in the pool or recent-block state for that address. The code comment in `skip_stateful_validations` claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is circular: the attacker's invoke transaction passes validations *precisely because* the victim's `deploy_account` is already in the pool, which triggers the skip.

**Effect on `run_validate_entry_point`:**

When `skip_validate = true`, the execution flags are set as `validate: !skip_validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling `__validate__`: [4](#0-3) 

**Attack sequence:**

1. Victim submits `deploy_account` (nonce=0) to the gateway; it is accepted into the mempool.
2. Attacker observes the victim's `deploy_account` in the mempool.
3. Attacker crafts an invoke transaction: `sender_address = victim_address`, `nonce = 1`, arbitrary `calldata`, invalid/attacker-controlled `signature`.
4. Gateway stateful validation for the attacker's tx:
   - `validate_nonce`: `account_nonce=0 ≤ tx_nonce=1 ≤ max_gap` → passes.
   - `validate_by_mempool` (`validate_tx`): no existing tx at `(victim_address, nonce=1)` → passes.
   - `skip_stateful_validations`: `nonce=1`, `account_nonce=0`, `account_tx_in_pool_or_recent_block(victim_address) = true` (victim's `deploy_account` is in pool) → returns `true`.
   - `run_validate_entry_point` with `skip_validate=true` → `__validate__` **not called**.
5. Attacker's unsigned/invalid-signature invoke tx is admitted to the mempool at `(victim_address, nonce=1)`.
6. Victim submits their legitimate invoke tx (nonce=1):
   - `validate_by_mempool` → `validate_fee_escalation` → `DuplicateNonce` (fee escalation disabled) or requires a fee premium exceeding `fee_escalation_percentage` (fee escalation enabled).
   - Victim's transaction is **rejected** or **forced to pay higher fees**. [5](#0-4) 

---

### Impact Explanation

**Admission of invalid transaction (High):** An invoke transaction with an attacker-controlled, cryptographically invalid signature is accepted by the gateway and inserted into the mempool. The invariant that every admitted transaction has passed account-level signature verification is violated.

**Rejection of valid transaction (High):** The victim's correctly signed invoke transaction — the one the `skip_stateful_validations` UX feature was designed to serve — is rejected with `DuplicateNonce` or must pay an escalated fee. The intended deploy-account-plus-invoke UX flow is broken for the victim.

This matches the allowed impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The attack requires only that the attacker observe a `deploy_account` transaction in the public mempool and submit a competing invoke transaction before the victim's invoke transaction is processed. Both conditions are trivially achievable: the mempool is public, and the attacker's submission is a single RPC call. No privileged access, special keys, or on-chain funds are required. The attack is repeatable for every new account deployment.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the pool for the target address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type stored in `tx_pool` for the given address at nonce=0. Only when a `deploy_account` transaction is confirmed present should `skip_stateful_validations` return `true`.

Alternatively, the gateway can record the `deploy_account` transaction hash at submission time and pass it alongside the invoke transaction, verifying that the specific `deploy_account` hash is present in the pool before granting the skip — analogous to the `deploy_account_tx_hash` parameter already used in `native_blockifier`'s `should_run_stateful_validations`: [6](#0-5)

### Proof of Concept

```
// State: victim_address has account_nonce = 0 (not deployed).

// Step 1: Victim submits deploy_account (nonce=0).
gateway.add_tx(RpcTransaction::DeployAccount { sender: victim_address, nonce: 0, sig: valid_sig, ... });
// → Accepted. mempool.account_tx_in_pool_or_recent_block(victim_address) == true.

// Step 2: Attacker submits invoke (nonce=1) with invalid signature.
gateway.add_tx(RpcTransaction::Invoke { sender: victim_address, nonce: 1, sig: [0xDEAD, 0xBEEF], calldata: [...] });
// Gateway path:
//   validate_nonce: 0 <= 1 <= max_gap  → OK
//   validate_by_mempool: no tx at (victim_address, 1) yet → OK
//   skip_stateful_validations:
//     nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block==true → returns true
//   run_validate_entry_point(skip_validate=true):
//     execution_flags.validate = false → __validate__ NOT called → OK
// → Attacker's tx ACCEPTED into mempool at (victim_address, nonce=1).

// Step 3: Victim submits their legitimate invoke (nonce=1).
gateway.add_tx(RpcTransaction::Invoke { sender: victim_address, nonce: 1, sig: valid_sig, calldata: [...] });
// validate_by_mempool → validate_fee_escalation:
//   existing tx at (victim_address, 1) found → DuplicateNonce (or fee escalation required)
// → Victim's tx REJECTED.
```

The victim's post-deploy invoke is permanently blocked unless they resubmit with a fee exceeding the attacker's by the configured `fee_escalation_percentage`, and the attacker can repeat the attack indefinitely at negligible cost.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    /// transaction to be replaced when a valid replacement exists, `None` when there is nothing to
    /// replace, or an error when a replacement is present but not permitted.
    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
