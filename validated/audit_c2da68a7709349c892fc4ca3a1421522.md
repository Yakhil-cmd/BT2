### Title
Direct STRK Transfer to `RewardSupplier` Inflates `balance_of` Check, Suppressing L1 Mint Requests and Enabling Temporary Reward Claim Freeze - (File: src/reward_supplier/reward_supplier.cairo)

### Summary
The `request_funds` internal function in `RewardSupplier` uses a live `token_dispatcher.balance_of(get_contract_address())` call to decide whether to send L1 mint requests. Because STRK is a standard ERC-20, any address can transfer tokens directly to the `RewardSupplier` contract without going through the bridge. This inflates the `balance` term in the credit/debit comparison, suppresses L1 mint requests, and can eventually cause `claim_rewards` to revert when the actual balance is depleted — temporarily freezing reward distribution for all stakers and delegators.

### Finding Description

`request_funds` is called every time `update_unclaimed_rewards_from_staking_contract` is invoked (i.e., on every reward distribution event). Its logic is:

```
credit = balance_of(self) + l1_pending_requested_amount
debit  = unclaimed_rewards
if credit < debit + threshold  →  send L1 mint request(s)
``` [1](#0-0) 

The `balance` variable is read directly from the ERC-20 `balance_of` call: [2](#0-1) 

An attacker who directly transfers `D` STRK to the `RewardSupplier` address (bypassing the bridge) increases `balance` by `D` without triggering `on_receive`, so `l1_pending_requested_amount` is **not** reduced. The net effect is that `credit` rises by `D`, making the condition `credit < debit + threshold` false for longer than intended, and no new L1 mint requests are sent. [3](#0-2) 

As rewards continue to be distributed, the staking contract calls `claim_rewards` on the `RewardSupplier`: [4](#0-3) 

When the actual token balance falls below the amount being claimed, `checked_transfer` reverts. Because `claim_rewards` is called inside `_update_rewards` in the staking contract (via `claim_from_reward_supplier`), the entire reward distribution transaction reverts: [5](#0-4) 

No staker or delegator can receive rewards until new L1 funds arrive — a delay of hours to days on Ethereum.

The `on_receive` path (bridge deposits) correctly reduces `l1_pending_requested_amount`, keeping `credit` neutral. Only **direct** ERC-20 transfers bypass this accounting, creating the asymmetry: [6](#0-5)
