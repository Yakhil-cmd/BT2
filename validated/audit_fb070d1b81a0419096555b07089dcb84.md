### Title
Gateway skips `__validate__` for invoke nonce=1 when any nonce=0 transaction exists in mempool, admitting transactions with invalid signatures — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (signature verification) for an Invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. That helper returns `true` for **any** transaction type in the pool or any committed block entry — not exclusively for a `deploy_account`. For an account that already exists in state with nonce=0 (deployed but never transacted), an attacker can first submit a valid nonce=0 invoke (which passes full validation and enters the pool), then submit a nonce=1 invoke with an **invalid signature** — the gateway skips `__validate__` and admits the invalid transaction to the mempool.

---

### Finding Description

`skip_stateful_validations` is the analog of the external bug's "early-return when `totalStakedAmount == 0`". Both share the same structural flaw: a guard on a "first-state" condition (`account_nonce == 0` here; `totalStakedAmount == 0` there) causes a critical check to be bypassed.

The relevant code path in `extract_state_nonce_and_run_validations`:

```
account_nonce = get_nonce_from_state(address)   // returns 0
skip_validate = run_pre_validation_checks(...)   // calls skip_stateful_validations
run_validate_entry_point(tx, skip_validate)      // skips __validate__ if true
```

The code comment in `skip_stateful_validations` claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* [4](#0-3) 

This reasoning is incorrect. A nonce=0 **invoke** transaction for an already-deployed account (nonce=0 in state) passes full validation and enters the pool. Its presence makes `account_tx_in_pool_or_recent_block` return `true`, which then causes the nonce=1 invoke to skip `__validate__` — even though the nonce=0 invoke is not a `deploy_account` and provides no guarantee about the validity of the nonce=1 invoke's signature.

The `run_validate_entry_point` function sets `validate: !skip_validate`, so when `skip_validate = true`, the blockifier's `StatefulValidator::validate` is called with `execution_flags.validate = false`, and `validate_tx` returns `Ok(None)` immediately:

```rust

    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4)

    if !self.execution_flags.validate {

    ...
}

### Impact Explanation

The gateway admits an Invoke transaction with an invalid (or entirely absent) signature to the mempool. This violates the invariant that every transaction in the mempool has passed `__validate__`. The matching impact category is:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The admitted transaction will fail `__validate__` when the batcher executes it (the batcher sets its own `ExecutionFlags` with `validate: true`), resulting in a rejected transaction with no fee charged. The attacker can therefore spam the mempool with signature-invalid transactions at zero cost, degrading sequencer throughput and wasting batcher resources.

---

### Likelihood Explanation

The precondition — an account deployed in state with nonce=0 — is common. Any contract deployed via a factory or constructor call, or any pre-funded account that has not yet sent a transaction, satisfies it. The attack requires only two sequential RPC calls (`add_transaction` twice) and no special privileges.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the address. The mempool should expose a dedicated `deploy_account_in_pool_or_recent_block(address)` query that filters by transaction type, or the gateway should track deploy_account hashes separately (as the `PyValidator` path does via `deploy_account_tx_hash`). [8](#0-7)

### Proof of Concept

1. Let address `A` be a deployed account contract with nonce=0 in state (e.g., a factory-deployed wallet that has never sent a transaction).
2. Attacker submits `invoke_A_nonce0` with a valid signature.
   - `validate_nonce`: nonce 0 ∈ [0, 200] ✓
   - `skip_stateful_validations`: `tx.nonce() = 0 ≠ 1` → returns `false`
   - `run_validate_entry_point`: `__validate__` runs, valid signature ✓
   - Transaction enters mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits `invoke_A_nonce1` with an **invalid/garbage signature**.
   - `validate_nonce`: nonce 1 ∈ [0, 200] ✓
   - `validate_by_mempool`: nonce 1 ≥ resolved nonce 0 ✓
   - `skip_stateful_validations`: `tx.nonce() = 1`, `account_nonce = 0`, `account_tx_in_pool_or_recent_block(A) = true` → returns **`true`**
   - `run_validate_entry_point`: `validate = false`, `__validate__` **not called**
   - Transaction with invalid signature is admitted to the mempool.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
