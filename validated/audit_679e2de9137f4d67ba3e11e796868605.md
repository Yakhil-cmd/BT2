### Title
Unbounded Loop in `calculate_rewards` Over Growing `pool_member_epoch_balance` Trace Can Permanently Freeze Delegator's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in the delegation pool contract iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because each balance change in a new epoch appends a new checkpoint to this trace, a pool member who makes repeated balance changes across many epochs without claiming rewards will accumulate an unbounded number of trace entries. When the trace grows large enough, the loop exceeds Starknet's per-transaction resource limits, causing `claim_rewards` (and `pool_member_info_v1`) to revert permanently — freezing the delegator's unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace (`pool_member_epoch_balance`) is appended to by `set_member_balance`, which is called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`. The `insert` logic only deduplicates entries with the **same epoch key** (`current_epoch + K`); balance changes in **different epochs** each produce a distinct checkpoint:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(
        balance: amount,
        cumulative_rewards_trace_idx: self.cumulative_rewards_trace_length() + 1,
    );
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

The `entry_to_claim_from` cursor is only advanced when `claim_rewards` completes successfully:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [3](#0-2) 

If a pool member makes one balance change per epoch (e.g., small top-ups via `add_to_delegation_pool`) across N epochs without ever calling `claim_rewards`, the trace accumulates N entries. The next `claim_rewards` call must iterate all N entries, each requiring a storage read via `pool_member_trace.at(...)` and a `find_sigma` call (itself doing additional storage reads). Beyond a threshold of accumulated entries, the transaction exceeds Starknet's resource limits and reverts.

The same loop is also exercised by the view function `pool_member_info_v1`:

```cairo
let (rewards, _) = self.calculate_rewards(
    :pool_member,
    from_checkpoint: pool_member_info.reward_checkpoint,
    until_checkpoint: self.get_current_checkpoint(:pool_member),
    entry_to_claim_from: pool_member_info.entry_to_claim_from,
);
``` [4](#0-3) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace grows large enough that `calculate_rewards` exceeds the per-transaction resource cap, every call to `claim_rewards` will revert. Because `entry_to_claim_from` is only updated on a successful `claim_rewards`, there is no way to partially process the trace across multiple transactions. The pool member's accumulated unclaimed yield becomes permanently inaccessible — matching the **"Permanent freezing of unclaimed yield"** impact category.

---

### Likelihood Explanation

The scenario requires a pool member to:
1. Make balance changes (via `add_to_delegation_pool` or `exit_delegation_pool_intent`) in many distinct epochs.
2. Defer `claim_rewards` across those epochs.

This is a realistic pattern for delegators who use automated rebalancing bots or who regularly top up their delegation without claiming. The developers themselves acknowledge the loop is unbounded in the code comment. The number of epochs required to hit Starknet's resource ceiling depends on the per-iteration cost (multiple storage reads per loop iteration), but given that each `find_sigma` call reads from `cumulative_rewards_trace`, the cost per iteration is non-trivial. Over months of active delegation without claiming, the threshold can be reached.

---

### Recommendation

Track `entry_to_claim_from` as a persistent cursor that is advanced even across partial reward computations, or introduce a `max_iterations` parameter to `claim_rewards` so that rewards can be claimed in batches. Alternatively, enforce a maximum number of unclaimed balance-change epochs before new balance changes are accepted (requiring a claim first). The simplest mitigation is to allow `claim_rewards` to accept a `limit` parameter and process only up to `limit` trace entries per call, updating `entry_to_claim_from` after each partial run.

---

### Proof of Concept

1. Pool member calls `enter_delegation_pool` in epoch E.
2. In each subsequent epoch E+1, E+2, …, E+N, the pool member calls `add_to_delegation_pool` with a minimal amount (1 unit). Each call invokes `increase_member_balance` → `set_member_balance` → `trace.insert(key: epoch + K, ...)`. Because the key differs each epoch, a new checkpoint is appended. [5](#0-4) 
3. The pool member never calls `claim_rewards`, so `entry_to_claim_from` remains at its initial value and `pool_member_epoch_balance` trace length grows to N.
4. After N epochs, the pool member calls `claim_rewards`. The loop at line 859 iterates from `entry_to_claim_from` (0) to N, performing a storage read and a `find_sigma` call per iteration. [6](#0-5) 
5. When N is large enough to exhaust Starknet's per-transaction resource budget, the transaction reverts. Because `entry_to_claim_from` was never updated (the transaction failed), every subsequent `claim_rewards` attempt also reverts — permanently freezing the delegator's unclaimed yield.
