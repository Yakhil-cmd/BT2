### Title
ETH Bridge Deposit via `receive_base_tokens` Bypasses Recipient Contract's EVM `receive()` Hook, Permanently Freezing Funds - (File: `engine/src/engine.rs`)

### Summary

When ETH is bridged from NEAR into Aurora via the `ft_on_transfer` → `receive_base_tokens` path, the engine directly writes the recipient's balance in storage without executing any EVM code on the recipient address. If the recipient is a smart contract that relies on its Solidity `receive()` or `fallback()` function to register incoming ETH (e.g., for deposit accounting), that hook is never triggered. The ETH is credited to the contract's balance but the contract has no record of it, permanently freezing the funds.

### Finding Description

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to one of two paths depending on whether the predecessor is the ETH connector account: [1](#0-0) 

When the predecessor **is** the connector (i.e., ETH is being bridged in), `receive_base_tokens` is called: [2](#0-1) 

`receive_base_tokens` resolves the recipient address from the `msg` field of `FtTransferMessageData`, then calls `set_balance` directly: [3](#0-2) 

`set_balance` is a raw storage write — it does not invoke the EVM at all: [4](#0-3) 

**Contrast this with the ERC-20 bridging path**, `receive_erc20_tokens`, which correctly calls into the EVM via `self.call()` to execute the `mint` function on the ERC-20 contract: [5](#0-4) 

The ERC-20 path goes through the full EVM execution pipeline, so any EVM-level hooks in the recipient contract fire normally. The base-ETH path does not.

The `msg` field parsed by `FtTransferMessageData` accepts any 20-byte EVM address — including smart contract addresses — with no restriction: [6](#0-5) 

### Impact Explanation

**Permanent freezing of funds (Critical).**

Any Aurora smart contract that tracks ETH deposits inside its `receive()` function will have its accounting permanently desynchronized when ETH arrives via the bridge. A concrete example:

```solidity
contract Vault {
    mapping(address => uint256) public deposits;
    receive() external payable {
        deposits[msg.sender] += msg.value;   // never called via bridge
    }
    function withdraw() external {
        uint256 amt = deposits[msg.sender];
        deposits[msg.sender] = 0;
        payable(msg.sender).transfer(amt);   // always reverts: amt == 0
    }
}
```

ETH bridged to this contract is credited to its EVM balance (visible via `address(this).balance`) but `deposits[sender]` is never incremented. The `withdraw()` path is permanently blocked. The ETH cannot be recovered by any user-accessible path, constituting a permanent fund freeze.

### Likelihood Explanation

**High.** The bridge is a primary, documented entry point for ETH into Aurora. Any contract that follows the standard Solidity pattern of using `receive()` for deposit accounting — a common and recommended pattern — is affected whenever ETH arrives via the NEAR bridge rather than via an EVM transaction. No special attacker capability is required; any NEAR account can call `ft_transfer_call` on the ETH connector and specify an arbitrary EVM contract address as the recipient.

### Recommendation

Replace the direct `set_balance` call in `receive_base_tokens` with a proper EVM `call` to the recipient address (with the bridged ETH as `value` and empty calldata), mirroring how `receive_erc20_tokens` invokes the EVM. This will trigger the recipient's `receive()` or `fallback()` function as expected by the EVM standard. If the call reverts, the tokens should be returned to the sender (consistent with the existing error-return pattern already present in `ft_on_transfer`).

### Proof of Concept

1. Deploy a `Vault` contract on Aurora whose `withdraw()` depends on `deposits[msg.sender]` being set by `receive()`.
2. From a NEAR account, call `ft_transfer_call` on the ETH connector with `receiver_id = aurora` and `msg = <hex address of Vault>`.
3. The ETH connector calls `ft_on_transfer` on Aurora; `receive_base_tokens` fires and writes the balance via `set_balance` — no EVM execution occurs.
4. Observe: `address(vault).balance` reflects the bridged amount, but `vault.deposits(sender) == 0`.
5. Call `vault.withdraw()` — it reverts because `deposits[msg.sender]` is zero.
6. The bridged ETH is permanently inaccessible.

The root cause is confirmed at: [2](#0-1) 
with the missing EVM dispatch compared to the correct pattern at: [5](#0-4)

```rust
    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }
```

```rust
    type Error = errors::ParseOnTransferMessageError;

    fn try_from(message: &str) -> Result<Self, Self::Error> {
        if message.len() == 40 {
            // Parse message to determine recipient
            let recipient = {
                // Message format:
                // Recipient of the transaction - 40 characters (Address in hex)
                let mut address_bytes = [0; 20];
                hex::decode_to_slice(message, &mut address_bytes)
                    .map_err(|_| errors::ParseOnTransferMessageError::InvalidHexData)?;
                Address::from_array(address_bytes)
            };

            #[allow(deprecated)]
            return Ok(Self {
                recipient,
                fee: None,
            });
```
