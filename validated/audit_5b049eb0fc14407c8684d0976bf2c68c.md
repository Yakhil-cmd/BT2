### Title
Incorrect Multi-NFT Royalty Calculation in WalletConnect Confirmation Display Causes Underestimated Spend Amount - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly splits the fungible payment amount equally among all NFTs before applying each royalty percentage. The Chia blockchain applies each NFT's royalty to the **full** fungible amount. This causes the WalletConnect confirmation dialog to display a materially lower total spend than what is actually deducted from the user's wallet when an offer involving multiple royalty-bearing NFTs is accepted.

### Finding Description
In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` computes the royalty-inclusive total shown to the user in the WalletConnect signing confirmation:

```ts
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

For a single NFT this is equivalent to the correct formula. For multiple NFTs it is wrong: the amount is divided by the number of NFTs before each royalty is applied, so the displayed royalty is `(amount/N) * r1 + (amount/N) * r2 + …` instead of the correct `amount * r1 + amount * r2 + …`.

The Chia blockchain's `calculate_royalties` RPC applies each NFT's royalty percentage to the **full** fungible amount, not a per-NFT split. The discrepancy grows with the number of NFTs and the magnitude of their royalty percentages.

**Concrete example** (from the existing test at line 251–335):
- Offer: pay 100,000,000 mojos (0.0001 XCH) to receive 2 NFTs with royalties 5% and 0.1%
- **Displayed** (`amountWithRoyalties`): `(50M × 5%) + (50M × 0.1%) = 2,550,000` → total **102,550,000 mojos (0.00010255 XCH)**
- **Actual on-chain**: `(100M × 5%) + (100M × 0.1%) = 5,100,000` → total **105,100,000 mojos (0.0001051 XCH)**

The user is shown ~2.5% less than what is actually deducted. The gap scales linearly with the number of NFTs and their royalty rates. [2](#0-1) 

This display path is invoked for both the WalletConnect `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` commands, which are the two offer-related commands exposed to dApps. [3](#0-2) 

The regular (non-WalletConnect) offer acceptance flow uses `calculateNFTRoyalties` in `utils.ts`, which only handles a single royalty percentage and is not affected. [4](#0-3) 

### Impact Explanation
A user who receives a WalletConnect request to accept an offer containing multiple royalty-bearing NFTs will see a confirmation dialog showing a lower total XCH/CAT spend than what the blockchain will actually deduct. The user approves based on the underestimated figure. This is a direct instance of the allowed High impact: "WalletConnect state that causes a user to approve … the wrong … amount."

### Likelihood Explanation
Any WalletConnect-connected dApp can trigger this by sending a `take_offer` or `create_offer_for_ids` command for an offer that bundles two or more NFTs with non-zero royalties. No special privileges, leaked keys, or social engineering are required beyond a normal WalletConnect session. Multi-NFT bundle offers are a standard use case in NFT marketplaces.

### Recommendation
Replace the split-then-apply formula with the correct full-amount-per-NFT formula:

```ts
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

Add a regression test asserting that for two NFTs with royalties `r1` and `r2` and payment `P`, `amountWithRoyalties` equals `P * (1 + r1/10000 + r2/10000)`, not `P * (1 + (r1+r2)/(2*10000))`.

### Proof of Concept
The existing test at `parseCommandDisplay.test.ts:251–335` already encodes the incorrect expected value `'0.00010255'` (split formula). Changing the assertion to the correct value `'0.0001051'` and fixing the implementation would demonstrate the discrepancy. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,

  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;

  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,

  };
}
