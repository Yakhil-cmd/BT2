### Title
Stale `eligible_operational_addresses` Entry Never Cleared After `change_operational_address` Allows Consent Bypass - (File: `src/staking/staking.cairo`)

### Summary
After `change_operational_address` consumes an `eligible_operational_addresses` entry to verify consent, it never clears that entry. The stale mapping value persists indefinitely, allowing a staker to later reclaim a previously-used operational address without the operational address owner re-declaring consent, bypassing the two-step consent mechanism entirely.

### Finding Description
The `eligible_operational_addresses` storage mapping (`Map<ContractAddress, ContractAddress>`) is the protocol's mechanism for an operational address owner to grant consent to a specific staker before that staker can call `change_operational_address`. The intended flow is:

1. Operational address `O` calls `declare_operational_address(staker_A)` → writes `eligible_operational_addresses[O] = staker_A`
2. Staker A calls `change_operational_address(O)` → reads and verifies `eligible_operational_addresses[O] == staker_A`, then updates `operational_address_to_staker_address`

The bug: `change_operational_address` never clears `eligible_operational_addresses[O]` after consuming it. [1](#0-0) 

After step 2, `eligible_operational_addresses[O]` still equals `staker_A`. If staker A later changes to a different operational address `Z` (freeing `O` by zeroing `operational_address_to_staker_address[O]`), the stale entry `eligible_operational_addresses[O] = staker_A` remains. Staker A can then call `change_operational_address(O)` again — the `OPERATIONAL_EXISTS` check passes (O is free) and the `OPERATIONAL_NOT_ELIGIBLE` check passes (stale entry) — without `O`'s owner ever re-declaring consent. [2](#0-1) 

`remove_staker`, called during `unstake_action`, clears `operational_address_to_staker_address` but also never touches `eligible_operational_addresses`. [3](#0-2) 

The only place `eligible_operational_addresses` is written is inside `declare_operational_address` itself. [4](#0-3) 

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users.**

A staker can claim an operational address without the owner's current consent. The operational address owner is then blocked from using their address for any other staker (because `declare_operational_address` checks `operational_address_to_staker_address.read(operational_address).is_zero()` and would fail with `OPERATIONAL_IN_USE`). [5](#0-4) 

The owner's only recourse is to call `declare_operational_address(Zero::zero())` to overwrite the stale entry, but this requires them to be aware of the attack and act proactively.

### Likelihood Explanation
Any active staker who has previously used an operational address and then changed away from it can trigger this. No privileged role is required. The staker controls both the timing of `change_operational_address` calls and the choice of which previously-used address to reclaim. The stale entry persists across the full staker lifecycle including `unstake_action` and re-staking by a different address.

### Recommendation
Clear the `eligible_operational_addresses` entry for the new operational address at the end of `change_operational_address`:

```cairo
// After updating operational address in storage:
self.eligible_operational_addresses.write(operational_address, Zero::zero());
```

This mirrors how `operational_address_to_staker_address` is zeroed for the old address on the same lines. [6](#0-5) 

### Proof of Concept

```
1. O calls declare_operational_address(staker_A)
   → eligible_operational_addresses[O] = staker_A

2. staker_A calls change_operational_address(O)
   → operational_address_to_staker_address[O] = staker_A
   → eligible_operational_addresses[O] = staker_A  ← NOT cleared (bug)

3. O calls declare_operational_address(staker_B)
   → eligible_operational_addresses[O] = staker_B  ← O revokes consent from A

4. staker_A calls declare_operational_address(staker_A) from Z
   → eligible_operational_addresses[Z] = staker_A

5. staker_A calls change_operational_address(Z)
   → operational_address_to_staker_address[O] = zero  ← O is freed
   → eligible_operational_addresses[O] = staker_B     ← correctly updated in step 3

   BUT: if step 3 never happened (O's owner did nothing after step 2):
   → eligible_operational_addresses[O] = staker_A     ← stale entry persists

6. staker_A calls change_operational_address(O):
   - operational_address_to_staker_address[O].is_zero() → TRUE  ✓
   - eligible_operational_addresses[O] == staker_A    → TRUE (stale!) ✓
   → staker_A reclaims O without O's owner's consent
   → O's owner is now blocked from using address O for any other staker
```
