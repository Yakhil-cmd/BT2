### Title
Missing Identity-Point Check on `app_pk` Allows Direct Disclosure of Confidential Derived Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary

The `ckd()` entry-point in `src/confidential_key_derivation/protocol.rs` accepts a caller-supplied `app_pk: PublicKey` (a BLS12-381 G1 element) and passes it directly into the ElGamal encryption step inside `compute_signature_share()` without verifying that it is not the group identity (point at infinity). When `app_pk` is the identity, the ElGamal masking term vanishes and the coordinator's output `big_c` becomes the raw confidential key `msk · H(pk, app_id)`, fully exposing it to the requester.

### Finding Description

The CKD protocol is designed so that each participant computes an ElGamal-encrypted share of the confidential key:

```
big_c  = big_s + app_pk · y          (line 174, protocol.rs)
big_y  = y · G
```

where `big_s = x_i · H(pk, app_id)` is the participant's secret contribution and `y` is a fresh random scalar. The coordinator aggregates Lagrange-weighted shares:

```
C_out = Σ λ_i · big_c_i  =  msk · H(pk, app_id)  +  app_pk · Y
```

The requester recovers the confidential key as `C_out − app_sk · Y_out`.

**Root cause:** `ckd()` performs no check that `app_pk` is a valid, non-identity group element before the protocol runs. [1](#0-0) 

If a caller supplies `app_pk = O` (the G1 identity / point at infinity):

- `app_pk · y = O` for any scalar `y`
- Therefore `big_c = big_s` (the masking term disappears entirely)
- The coordinator's aggregated output becomes:

```
C_out = Σ λ_i · x_i · H(pk, app_id)  =  msk · H(pk, app_id)
``` [2](#0-1) 

The attacker reads `big_c` directly from the returned `CKDOutput` and obtains the confidential key without needing `app_sk`.

There is no guard anywhere in the call chain — not in `ckd()`, not in `run_ckd_protocol()`, and not in `compute_signature_share()` — that rejects or detects an identity `app_pk`. [3](#0-2) 

### Impact Explanation

**Critical — Extraction/disclosure of a confidential derived secret.**

The entire security guarantee of CKD is that the output `(big_y, big_c)` is an ElGamal ciphertext: only the holder of `app_sk` can recover `msk · H(pk, app_id)`. Supplying the identity as `app_pk` collapses the ciphertext to plaintext. The attacker directly reads the confidential key from the coordinator's output without performing any cryptographic inversion. This falls squarely under the allowed impact: *"Extraction, reconstruction, or disclosure of … confidential derived secrets."*

### Likelihood Explanation

The `ckd()` function is a public library API. The `app_pk` argument is entirely caller-controlled — it is the application's public key, supplied by the requester who initiates the derivation. A malicious requester can trivially construct the identity point in BLS12-381 G1 (it is the standard additive identity of the group) and pass it as `app_pk`. No privileged access, no key material, and no cryptographic break is required. The attack is a single-call, zero-precondition exploit.

### Recommendation

Add an explicit check in `ckd()` (or at the top of `compute_signature_share()`) that rejects the identity element:

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the existing pattern used elsewhere in the codebase (e.g., `msg_hash.is_zero()` check in robust ECDSA signing). [4](#0-3) 

### Proof of Concept

```rust
use bls12_381::G1Projective;  // identity = G1Projective::identity()

// Attacker supplies the G1 identity as their "public key"
let malicious_app_pk = G1Projective::identity();

// All participants run ckd() with this app_pk
let protocol = ckd(
    &participants,
    coordinator,
    me,
    key_pair,
    app_id,
    malicious_app_pk,   // <-- identity point
    rng,
).unwrap();

// Coordinator's output:
//   ckd_output.big_c  ==  msk * H(pk, app_id)   (the raw confidential key)
//   ckd_output.big_y  ==  some random Y
//
// Attacker reads big_c directly — no app_sk needed.
let stolen_confidential_key = ckd_output.big_c();
// stolen_confidential_key == msk * H(pk, app_id)  ✓
```

The masking term `app_pk · Y = O · Y = O` vanishes, so `big_c` is the unencrypted confidential key. The attacker recovers it by reading the coordinator output, bypassing the ElGamal confidentiality guarantee entirely.
