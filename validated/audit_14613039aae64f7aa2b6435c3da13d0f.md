### Title
Overly Broad `account_tx_in_pool_or_recent_block` Check in `skip_stateful_validations` Allows Admission of Unsigned Invoke Transactions for Any Account with a Pending Mempool Entry - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is intended to skip the `__validate__` entry-point call only when a deploy-account transaction for the sender is pending in the mempool (UX improvement for the deploy-account + invoke flow). However, the guard condition uses `account_tx_in_pool_or_recent_block(sender_address)`, which returns `true` for **any** transaction type in the mempool for that address — not specifically a deploy-account transaction. An attacker can exploit this by submitting a crafted invoke transaction (nonce=1, arbitrary calldata, garbage signature) for any victim account that currently has **any** pending transaction in the mempool with `account_nonce == 0` in committed state. The gateway admits the transaction without running `__validate__`, bypassing signature verification entirely at the admission layer.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` decides whether to skip the blockifier's `__validate__` call:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                // ...
        }
    }
    Ok(false)
}
```

The comment states: *"it is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."*

The second branch of that disjunction is the flaw. `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`contains_account` checks only whether the address appears in the pool or state maps — it does not distinguish deploy-account transactions from invoke transactions. Therefore, if a victim account (with `account_nonce == 0` in committed state) has **any** pending invoke transaction in the mempool (e.g., a nonce=0 invoke that passed its own `__validate__`), the condition fires and the gateway skips `__validate__` for a new nonce=1 invoke submitted by the attacker.

The result of `skip_stateful_validations` is passed directly into `run_validate_entry_point`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate = false`, `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
```

The transaction is then forwarded to the mempool and eventually to the batcher. The batcher re-runs `__validate__` during execution; the transaction reverts. However, the nonce is incremented and fees are charged even on revert, causing the victim to lose their nonce=1 slot and pay fees for a transaction they never authorized.

### Impact Explanation

An attacker can submit an invoke transaction with `sender_address = victim`, `nonce = 1`, arbitrary calldata, and a garbage signature. The gateway admits it to the mempool without signature verification. This is a direct violation of the invariant that every transaction in the mempool must have passed account-level signature validation. The concrete effects are:

- **Invalid transaction admitted**: A transaction with an invalid (attacker-crafted) signature enters the mempool and is sequenced.
- **Nonce slot consumed**: The victim's nonce=1 slot is permanently consumed by the reverted transaction, invalidating any legitimate nonce=1 transaction the victim intended to submit.
- **Fee loss**: The batcher charges fees from the victim's account for the reverted transaction.
- **Griefing of new accounts**: Any account with `account_nonce == 0` that has submitted any pending transaction is vulnerable.

### Likelihood Explanation

The preconditions are:
1. The victim account has `account_nonce == 0` in committed state (new account).
2. The victim has any pending transaction in the mempool (extremely common — the deploy-account + invoke UX pattern is the primary use case for this code path).

Both conditions are routinely satisfied by any new Starknet account using the standard deploy-account + invoke flow. The attacker needs no privileged access, only the ability to submit RPC transactions.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction exists for the sender address in the mempool. Add a new mempool API method such as `deploy_account_tx_in_pool(address)` that inspects transaction types, and use it in `skip_stateful_validations`. The current check conflates "account has any pending transaction" with "account has a pending deploy-account transaction," which is the root cause of the bypass.

### Proof of Concept

1. Victim V is a new account (`account_nonce == 0` in state). V submits `invoke(sender=V, nonce=0, calldata=X, sig=valid_sig)` — this passes `__validate__` normally and enters the mempool. `tx_pool.contains_account(V)` is now `true`.

2. Attacker A submits `invoke(sender=V, nonce=1, calldata=drain_funds, sig=0xdeadbeef)` to the gateway.

3. Gateway stateful validation:
   - `account_nonce = 0` (from state), `tx.nonce() = 1` → condition `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)` is **true**.
   - `account_tx_in_pool_or_recent_block(V)` → **true** (V's nonce=0 invoke is in the pool).
   - `skip_stateful_validations` returns **true**.
   - `run_validate_entry_point` sets `validate = false` → `__validate__` is **not called**.
   - Transaction is admitted to the mempool.

4. Batcher executes V's nonce=0 invoke (succeeds). Then executes A's nonce=1 invoke: `__validate__` runs, fails (bad signature), transaction reverts. V's nonce is now 2.

    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,

        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");

            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.

                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");

                        );
                    }
                });

    }
