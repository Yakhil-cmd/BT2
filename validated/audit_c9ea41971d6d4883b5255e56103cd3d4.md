### Title
Domain Allowlist Bypass via `endsWith` Without Dot-Boundary Check in `fetchOffer` — (`packages/gui/src/util/fetchOffer.ts`)

### Summary

The `fetchOffer` function validates offer URLs against a trusted-domain allowlist using `String.prototype.endsWith`. Because the check does not enforce a dot-boundary, an attacker who registers a domain whose name ends with an allowed domain string (e.g., `evildexie.space` for the allowed domain `dexie.space`) can bypass the allowlist entirely and have the GUI fetch and present attacker-controlled offer data to the victim.

---

### Finding Description

The guard in `fetchOffer.ts` is:

```typescript
const domain = new URL(offerUrl).hostname;
const service = OfferServices.find(({ domains }) =>
  domains.find((localDomain) => domain.endsWith(localDomain))
);
``` [1](#0-0) 

The allowed domains are `dexie.space`, `mintgarden.io`, `spacescan.io`, and `offerpool.io`. [2](#0-1) 

`String.endsWith` performs a pure suffix match with no awareness of DNS label boundaries. Therefore:

| Attacker-controlled hostname | `.endsWith('dexie.space')` | Result |
|---|---|---|
| `evil.com` | false | blocked |
| `evil.dexie.space` | true | allowed (legitimate subdomain — attacker can't register) |
| **`evildexie.space`** | **true** | **allowed — attacker CAN register this** |

An attacker who registers `evildexie.space` (or `evilmintgarden.io`, `evilspacescan.io`, `evilofferpool.io`) passes the check unconditionally.

---

### Impact Explanation

The attacker-controlled server at `evildexie.space` can serve any offer blob. That blob is fetched, parsed via `parseFileContent`, submitted to `getOfferSummary` and `checkOfferValidity`, and the resulting offer summary is rendered in the GUI and navigated to the offer-view page with `imported: true`. [3](#0-2) [4](#0-3) 

The user is presented with a spoofed offer (wrong asset, wrong amount, wrong counterparty) and may accept it, causing direct asset loss. This satisfies the **High** impact criterion: *"Corruption, spoofing, or unsafe trust of offer state that causes a user to approve the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation

The attack entry point is the Chia on-chain notification system. Any unprivileged actor can send a notification message to any wallet address on-chain, embedding an arbitrary `offerURL` in the payload's `u` field. [5](#0-4) 

The attacker only needs to:
1. Register a domain like `evildexie.space` (trivially available, ~$10/year).
2. Host a crafted offer file there.
3. Send an on-chain notification to the victim's address.

No local access, no leaked keys, and no social engineering beyond the notification itself is required.

---

### Recommendation

Replace `endsWith` with an exact match or a dot-anchored suffix check:

```typescript
// Exact match or legitimate subdomain only:
domain === localDomain || domain.endsWith(`.${localDomain}`)
```

This ensures `evildexie.space` is rejected while `api.dexie.space` is still accepted.

---

### Proof of Concept

```
1. Register domain: evildexie.space
2. Host at https://evildexie.space/offer a valid-looking but malicious Chia offer blob
   (e.g., offering 1 worthless CAT in exchange for 1 XCH from the victim)
3. Send an on-chain Chia notification to victim's wallet address with payload:
   { v:1, t:1, d:{ u:"https://evildexie.space/offer" } }
4. Victim's GUI receives the notification, calls fetchOffer({ offerUrl: "https://evildexie.space/offer", ... })
5. new URL("https://evildexie.space/offer").hostname  →  "evildexie.space"
6. "evildexie.space".endsWith("dexie.space")  →  true  ← allowlist bypassed
7. Offer content fetched from attacker server, parsed, presented to victim as a valid offer
8. Victim clicks "Accept" → asset loss
```

The specific example in the question (`https://evil.com/dexie.space/offer`) does **not** bypass the check because `new URL(...)` correctly returns `evil.com` as the hostname. The real bypass is the `evildexie.space` class of domains, which is concrete and immediately exploitable.
