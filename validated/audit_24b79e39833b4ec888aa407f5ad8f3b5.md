### Title
Non-Atomic Two-Step Balance Update in `transfer_from_treasury` Causes System Halt via `MintingBalanceOverflow` — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`transfer_from_treasury` performs two sequential, non-atomic balance mutations: it first decrements the treasury, then increments the recipient. If the recipient's balance would overflow `U256::MAX`, the function returns `MintingBalanceOverflow`. This error is not treated as a per-transaction failure — it propagates as a `BootloaderSubsystemError` all the way to the block-level transaction loop, where it is classified as `TxError::Internal` and causes the entire block to halt. An attacker who can arrange for a recipient's balance to be near `U256::MAX` (feasible for custom base tokens with very large supplies) can permanently stall block processing.

---

### Finding Description

`transfer_from_treasury` issues two independent calls to `update_account_nominal_token_balance`:

**Step 1 — treasury debit** (lines 793–811): [1](#0-0) 

**Step 2 — recipient credit** (lines 813–831): [2](#0-1) 

There is no snapshot/rollback wrapping these two calls inside `transfer_from_treasury` itself. If Step 1 succeeds but Step 2 fails with `BalanceError::Overflow`, the function returns `MintingBalanceOverflow` while the treasury has already been decremented and the recipient has not been credited.

The error then propagates through the call stack:

1. `mint_base_token` propagates via `?` at line 768: [3](#0-2) 

2. `execute_l1_transaction_and_notify_result` propagates via `?` at line 657: [4](#0-3) 

3. In `process_l1_transaction`, `MintingBalanceOverflow` is a `BootloaderInterfaceError` (leaf usage error). Its `root_cause()` does **not** match `RootCause::Runtime(FatalRuntimeError)`, so it falls into the `_ =>` branch which rolls back the outer frame and **returns `Err(e)`** — propagating the error out of `process_l1_transaction`: [5](#0-4) 

4. In the block-level transaction loop, any `Err` from `process_transaction` is classified as `TxError::Internal`, which causes `return Err(err)` — **halting the entire block**: [6](#0-5) 

The same overflow path exists for the **post-execution** operator-fee and refund minting calls (lines 290–309 and 338–359), which occur *outside* the outer frame and therefore cannot be rolled back if they fail: [7](#0-6) [8](#0-7) 

The `BalanceError` variants are defined as: [9](#0-8)

### Impact Explanation

When `MintingBalanceOverflow` is triggered:

- The block-level loop receives `TxError::Internal` and calls `return Err(err)`, halting all further block processing. The chain cannot seal the block.
- For the **post-execution** minting paths (operator fee, refund), the main execution frame has already been committed before the overflow occurs. The treasury has been decremented but the operator/refund-recipient has not been credited. Since the block halts, these partial state changes are never finalized — but the block itself is unrecoverable without operator intervention.
- This is a **system halt / denial-of-service** impact: an attacker can prevent the sequencer from producing valid blocks.

---

### Likelihood Explanation

The overflow requires a recipient's balance `B` and transfer amount `T` such that `B + T > U256::MAX`. For ETH (total supply ≈ 1.2 × 10²⁶ wei vs. `U256::MAX` ≈ 1.16 × 10⁷⁷), this is practically impossible. However:

- ZKsync OS supports **custom base tokens**. A token with a total supply near `U256::MAX` (e.g., a token with 18 decimals and a supply of `2^255`) makes this overflow reachable.
- The attacker controls `transaction.reserved[1]` (the refund recipient address) in L1→L2 transactions. By directing the refund to an address whose balance is near `U256::MAX - refund_amount`, the overflow is triggered in the post-execution refund path.
- The attacker also controls `total_deposited` (`transaction.reserved[0]`), which determines `to_transfer`. A large deposit to a near-full recipient triggers the overflow in the value-minting path.

For chains using custom base tokens with large supplies, this is a realistic, externally-reachable attack requiring only a crafted L1→L2 transaction.

---

### Recommendation

1. **Make `transfer_from_treasury` atomic**: Before mutating any balance, check that the recipient's balance will not overflow. Use `checked_add` on the current recipient balance and the transfer amount before issuing either `update_account_nominal_token_balance` call.

2. **Treat `MintingBalanceOverflow` as a per-transaction failure for L1→L2 transactions**, not a block-level fatal error. L1 transactions cannot be invalidated (they come from the priority queue), but the correct response to an overflow is to revert the transaction's state and continue block processing — not to halt the block. This mirrors how `TreasuryTransferFailed` should also be handled.

3. **Use the existing atomic `transfer_nominal_token_value`** (which wraps both debit and credit in a single function with consistent overflow checking) instead of two separate `update_account_nominal_token_balance` calls: [11](#0-10) 

---

### Proof of Concept

**Setup** (custom base token with large supply):

1. Deploy a ZKsync chain with a custom base token whose total supply is `2^255` (a valid `U256` value).
2. Pre-fund the treasury (`BASE_TOKEN_HOLDER_ADDRESS`) with `T = 1000` tokens.
3. Pre-fund address `VICTIM = 0xDEAD...` with balance `B = U256::MAX - 500` tokens (achievable by sending many L1→L2 deposits to `VICTIM`).

**Attack**:

4. Craft an L1→L2 transaction:
   - `from`: attacker address
   - `to`: `VICTIM`
   - `reserved[0]` (`total_deposited`): `600` (treasury has enough)
   - `gas_price = 0`, `gas_limit = 0` → `max_fee_commitment = 0`, so `to_transfer = 600`
   - `reserved[1]` (refund recipient): any address

5. The sequencer processes the transaction:
   - `transfer_from_treasury(600, VICTIM)` is called.
   - Step 1: treasury balance decremented from `1000` to `400`. ✓
   - Step 2: `VICTIM` balance = `(U256::MAX - 500) + 600 = U256::MAX + 100` → **overflow** → `MintingBalanceOverflow` returned.

6. Error propagates: `mint_base_token` → `execute_l1_transaction_and_notify_result` → `process_l1_transaction` (line 238: `return Err(e)`) → `tx_loop` (line 112: `return Err(err)`).

7. **Block halts.** The chain cannot produce further blocks until the operator manually intervenes.

    fn transfer_nominal_token_value_inner<const PROOF_ENV: bool>(
        &mut self,
        from_ee: ExecutionEnvironmentType,
        resources: &mut R,
        from: &B160,
        to: &B160,
        amount: &U256,
        oracle: &mut impl IOOracle,
        is_selfdestruct: bool,
    ) -> Result<(), BalanceSubsystemError> {
        use zk_ee::system::BalanceError;

        let mut f = |addr, op: fn(U256, U256) -> (U256, bool), err| {
            self.update_nominal_token_value_inner::<PROOF_ENV>(
                from_ee,
                resources,
                addr,
                move |old_balance: &U256| {
                    let (new_value, of) = op(*old_balance, *amount);
                    if of {
                        Err(err)
                    } else {
                        Ok(new_value)
                    }
                },
                oracle,
                is_selfdestruct,
            )
        };

        // can do update twice
        f(
            from,
            U256::overflowing_sub,
            interface_error!(BalanceError::InsufficientBalance),
        )?;
        f(
            to,
            U256::overflowing_add,
            interface_error!(BalanceError::Overflow),
        )?;

        Ok(())
    }
