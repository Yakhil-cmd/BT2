### Title
Missing Minimum Threshold Validation in `assert_sign_inputs` Allows Single-Party Signature Forgery - (File: src/frost/mod.rs)

### Summary
The `assert_sign_inputs` function in `src/frost/mod.rs` validates signing inputs for FROST EdDSA and RedJubjub signing protocols but omits the lower-bound check `threshold < 2`. A malicious coordinator can invoke `sign_v1`, `sign_v2`, or `sign` with `threshold = 1`, bypassing the threshold security requirement and producing a valid threshold signature using only their own share.

### Finding Description
`assert_sign_inputs` in `src/frost/mod.rs` performs the following checks:

```rust
// src/frost/mod.rs lines 127-150
if participants.len() < 2 { ... }          // lower bound on participants
if threshold.value() > participants.len() { ... }  // upper bound on threshold
```

It does **not** check `threshold.value() < 2`. This is the direct analog of the external report's inverted comparison: the guard that should prevent an unsafe threshold value is simply absent.

Compare to `assert_key_invariants` in `src/dkg.rs`, which correctly enforces both bounds:

```rust
// src/dkg.rs lines 573-582
if threshold > participants.len() { ... }  // upper bound
if threshold < 2 { ... }                   // lower bound — PRESENT here, ABSENT in assert_sign_inputs
```

`validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` also enforces both bounds. The minimum-threshold check is consistently applied everywhere except `assert_sign_inputs`.

`assert_sign_inputs` is the sole validation gate for:
- `sign_v1` / `sign_v2` in `src/frost/eddsa/sign.rs`
- `sign` in `src/frost/redjubjub/sign.rs`

Both call `assert_sign_inputs` and then pass the accepted `threshold` directly into `construct_key_package`, which embeds it into the FROST `KeyPackage`. The FROST `aggregate` function enforces that the number of collected signature shares meets the threshold stored in the `KeyPackage`. With `threshold = 1`, the coordinator's own share alone satisfies aggregation.

### Impact Explanation
With `threshold = 1` accepted by `assert_sign_inputs`, a malicious coordinator can:
1. Collect nonce commitments from all participants in the presign round (normal protocol flow).
2. Construct a `KeyPackage` with `threshold = 1`.
3. Aggregate only their own signature share — the FROST `aggregate` call succeeds because `1 >= 1`.
4. Produce a cryptographically valid threshold signature without any other participant's cooperation.

This constitutes **unauthorized creation of a valid threshold signature for attacker-chosen inputs**, which is a Critical impact under the allowed scope.

### Likelihood Explanation
The `threshold` parameter is caller-supplied with no type-level enforcement of a minimum value (`ReconstructionLowerBound` is a plain `usize` wrapper with no invariant). Any unprivileged library caller acting as coordinator can pass `threshold = 1` directly to `sign_v1`, `sign_v2`, or `sign`. No special access, leaked keys, or cryptographic breaks are required. The path is direct and reachable.

### Recommendation
Add the same lower-bound check present in `assert_key_invariants` and `validate_triple_inputs` to `assert_sign_inputs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Add a corresponding test analogous to `keygen__should_fail_if_threshold_is_below_limit` that verifies `sign_v1`, `sign_v2`, and `sign` reject `threshold = 1`.
