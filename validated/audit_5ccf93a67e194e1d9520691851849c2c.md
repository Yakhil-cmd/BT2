### Title
Authz `MsgExec` Bypasses `BlockAddressesDecorator` for `MsgConvertVouchers` — (`app/block_address.go`, `x/cronos/keeper/msg_server.go`)

### Summary
`BlockAddressesDecorator` only inspects tx-level signers and only during `CheckTx`. A `MsgExec` wrapping `MsgConvertVouchers` is signed by an unblocked grantee, so the decorator passes it. The `ConvertVouchers` msg server then executes the IBC-voucher-to-CRC20 conversion for the blocked address (`msg.Address`) with no secondary blocklist check.

### Finding Description
`BlockAddressesDecorator.AnteHandle` gates its entire enforcement block on `ctx.IsCheckTx()` and iterates only `sigTx.GetSigners()` — the outer tx signers. When the Cosmos SDK authz module processes `MsgExec`, the outer signer is the grantee; the granter (blocked address) is embedded in the inner `MsgConvertVouchers.Address` field. The decorator never inspects inner messages. The `ConvertVouchers` msg server calls `k.ConvertVouchersToEvmCoins(ctx, msg.Address, msg.Coins)` directly with no blocklist lookup. If `MsgConvertVouchers` is not in `DisabledAuthzMsgs`, the authz module will dispatch it, and the conversion executes for the blocked address.

### Impact Explanation
A blocked address's IBC vouchers are converted to CRC20 tokens, changing bank and EVM balances for an address the operator explicitly intended to freeze. This is an unauthorized balance/accounting change for IBC vouchers and CRC20 assets — Critical impact per scope rules.

### Likelihood Explanation
Requires a prior authz grant from the blocked address to an accomplice, which must have been created before the address was blocked. This is a realistic operational scenario (e.g., address blocked after a suspicious grant is detected). No validator compromise, no leaked keys, no external dependencies.

### Recommendation
1. Add `MsgConvertVouchers` and `MsgTransferTokens` to `DisabledAuthzMsgs` in `app/app.go`.
2. Add a blocklist check inside `ConvertVouchers` and `TransferTokens` msg server handlers against the dynamic blocklist.
3. Move the `BlockAddressesDecorator` signer check outside the `ctx.IsCheckTx()` gate so it also enforces during DeliverTx.

### Proof of Concept
1. Deploy a local Cronos node. Add address `A` to the blocklist via `MsgStoreBlockList`.
2. Before step 1 (or via a validator bypassing CheckTx), have `A` submit `MsgGrant` authorizing address `B` to execute `MsgConvertVouchers`.
3. Fund `A` with IBC vouchers.
4. From `B`, submit `MsgExec{msgs: [MsgConvertVouchers{Address: A, Coins: [...]}]}`.
5. Observe: `BlockAddressesDecorator` passes (B is not blocked), authz dispatches the inner msg, `ConvertVouchers` executes for `A`, CRC20 tokens are minted to `A`'s EVM address.
6. Assert conversion succeeded despite `A` being blocked.
