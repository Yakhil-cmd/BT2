### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Permanently Freeze Consensus-Era Yield — (`src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the staking contract is documented as callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` every block, consuming the single global `last_reward_block` slot without distributing rewards. This permanently prevents the sequencer from distributing consensus-era block rewards to any staker or delegator.

---

### Finding Description

The spec at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` lines 1447–1507 is a public `#[abi(embed_v0)]` function with no caller check:

```cairo
#[abi(embed_v0)]
impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
    fn update_rewards(
        ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
    ) {
        self.general_prerequisites();          // only checks pause
        let current_block_number = starknet::get_block_number();
        assert!(
            current_block_number > self.last_reward_block.read(),
            "{}",
            Error::REWARDS_ALREADY_UPDATED,
        );
        // ... no assert on get_caller_address() ...
        self.last_reward_block.write(current_block_number);

        if disable_rewards || self.is_pre_consensus() {
            return;
        }
``` [1](#0-0) 

The `last_reward_block` is a **single global variable** shared across all stakers. Once it is written to the current block number, any subsequent call in the same block — including the sequencer's legitimate call — reverts with `REWARDS_ALREADY_UPDATED`. [2](#0-1) 

The spec confirms this is the intended guard: [3](#0-2) 

**Attack path:**

1. Attacker monitors the mempool or simply calls `update_rewards(any_active_staker, disable_rewards: true)` at every new block.
2. `last_reward_block` is set to the current block number.
3. No rewards are distributed (`disable_rewards = true` causes an early return before `_update_rewards`).
4. The sequencer's legitimate `update_rewards(staker_B, disable_rewards: false)` call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Stakers and delegators accumulate zero `unclaimed_rewards_own` for every blocked block.

The attacker only needs one valid active staker address (publicly readable via `get_stakers`) and pays gas per block — a trivially cheap, sustained griefing attack.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

In the consensus-rewards era (after `consensus_rewards_first_epoch` is reached), all staker and delegator rewards flow exclusively through `update_rewards` → `_update_rewards` → `unclaimed_rewards_own`. If `last_reward_block` is consumed every block with `disable_rewards: true`, no rewards are ever credited. Stakers calling `claim_rewards` receive zero. Delegators calling `pool.claim_rewards` receive zero. The yield is not merely delayed — it is permanently lost for every blocked block, because `last_reward_block` is never retroactively replayed. [4](#0-3) 

---

### Likelihood Explanation

**High.** The function is publicly callable with no authentication. The attacker needs only: (a) knowledge that consensus rewards are active, (b) any valid active staker address (obtainable from `get_stakers`), and (c) the ability to submit one transaction per block. No privileged keys, no front-running precision, no capital at risk. The attack is cheap, deterministic, and sustainable indefinitely.

---

### Recommendation

Add a sequencer-only caller check at the top of `update_rewards`, analogous to the pattern used for `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // enforce spec: "Only starkware sequencer"
    ...
}
```

Alternatively, if the sequencer address is not stored on-chain, use a role-based check (e.g., `OPERATOR` role) consistent with the existing `Roles` infrastructure. The key invariant is that `last_reward_block` must only be advanceable by the trusted sequencer, not by any arbitrary caller. [5](#0-4)

### Proof of Concept

```
// Precondition: consensus rewards are active (post consensus_rewards_first_epoch + K epochs)
// Attacker has no privileged access.

loop every new block N:
    staking_contract.update_rewards(
        staker_address = any_active_staker,  // from get_stakers()
        disable_rewards = true               // no rewards distributed
    )
    // last_reward_block is now N
    // sequencer's update_rewards call at block N → REWARDS_ALREADY_UPDATED
    // all stakers earn 0 rewards for block N
end loop

// After T blocks: all stakers have unclaimed_rewards_own == 0
// staking_contract.claim_rewards(staker) → transfers 0 tokens
// pool.claim_rewards(pool_member) → transfers 0 tokens
``` [6](#0-5) [7](#0-6)
