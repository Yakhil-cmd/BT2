### Title
Unchecked Precompile Return Value in `withdrawToNear` Allows Silent Token Burn Without NEAR Credit — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` burns the caller's tokens before invoking the `ExitToNear` precompile, and never checks the precompile's return value. If the precompile rejects the call (e.g., because the recipient bytes exceed `MAX_INPUT_SIZE = 1024` or are not a valid NEAR account ID), the EVM `call` opcode silently returns `0`, the Solidity function returns normally, and the already-executed burn is never reverted. Tokens are permanently destroyed with no corresponding NEAR credit.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` follows this sequence:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // (1) burn first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // (2) res is captured but NEVER checked
    }
    // (3) function returns normally regardless of res
}
``` [1](#0-0) 

The precompile enforces a hard `MAX_INPUT_SIZE` of 1 024 bytes:

```rust
const MAX_INPUT_SIZE: usize = 1_024;

fn validate_input_size(input: &[u8], min: usize, max: usize) -> Result<(), ExitError> {
    if input.len() < min || input.len() > max {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_INPUT")));
    }
    Ok(())
}
``` [2](#0-1) [3](#0-2) 

For the ERC-20 path the input layout is `flag(1) + amount(32) + recipient(N)`, so any `recipient.length > 991` causes the total to exceed 1 024 bytes and the precompile returns `Err(ExitError::Other("ERR_INVALID_INPUT"))`. [4](#0-3) 

Additionally, even a shorter but syntactically invalid NEAR account ID (e.g., containing illegal characters or exceeding NEAR's 64-character account-ID limit) causes `parse_recipient` to return `Err(ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID"))`: [5](#0-4) 

In both cases the precompile call returns `0` at the EVM level. Standard EVM semantics: a failed sub-`call` returns `0` but does **not** revert the parent call frame's state changes. Because `res` is never inspected and there is no `require` after the assembly block, `withdrawToNear` returns successfully with the burn already committed.

`EvmErc20V2` has the identical pattern (threshold shifts to `recipient.length > 971` due to the extra 20-byte sender field): [6](#0-5) 

---

### Impact Explanation

Any token holder can permanently destroy their own tokens (or, via a malicious wrapper, another user's tokens) without receiving any NEAR-side credit. This decouples the EVM-side ERC-20 supply from the NEAR-side NEP-141 backing, producing **insolvency**: the EVM supply shrinks while the NEAR supply is unchanged, and the destroyed tokens can never be recovered.

---

### Likelihood Explanation

The attack requires no privilege. Any address that holds a non-zero balance of any `EvmErc20`-based token can trigger it by passing a `recipient` byte array longer than 991 bytes (or any bytes that are not a valid NEAR account ID). This is trivially constructable in a single EVM transaction. No admin key, governance action, or external dependency is needed.

---

### Recommendation

1. **Check the precompile return value and revert on failure:**

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Validate recipient length before burning**, e.g. `require(recipient.length > 0 && recipient.length <= 64)` to match NEAR account-ID constraints, so the revert happens before any state change.

3. Apply the same fix to `withdrawToEthereum` and both `EvmErc20V2` variants for consistency.

---

### Proof of Concept

```solidity
// Attacker holds `amount` tokens of a deployed EvmErc20 instance.
// Step 1: craft a recipient that exceeds MAX_INPUT_SIZE threshold.
bytes memory oversizedRecipient = new bytes(992); // 1 + 32 + 992 = 1025 > 1024
for (uint i = 0; i < 992; i++) oversizedRecipient[i] = 0x61; // 'a'

uint256 balanceBefore = token.balanceOf(attacker);

// Step 2: call withdrawToNear — no revert, succeeds silently.
token.withdrawToNear(oversizedRecipient, amount);

uint256 balanceAfter = token.balanceOf(attacker);

// Step 3: assert tokens are gone with no NEAR credit issued.
assert(balanceBefore - balanceAfter == amount);  // tokens burned
// NEAR-side NEP-141 balance of any account is unchanged — insolvency confirmed.
```

The fuzz variant: iterate `recipient.length` from 0 to 2 000; for each length where the precompile would reject (>991 bytes or invalid UTF-8/account-ID), assert that `withdrawToNear` reverts. On unpatched code, no revert occurs for any length, confirming the invariant is broken.
