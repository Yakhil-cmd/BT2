### Title
NFT Offer "Net Proceeds" Display Omits Royalty Deduction in TokenForNFT Offers, Causing Sellers to Accept Offers Based on Overstated Receive Amounts — (File: `packages/gui/src/components/offers/utils.ts`)

---

### Summary

In `calculateNFTRoyalties`, `nftSellerNetAmount` is unconditionally set to the full offered `amount` regardless of exchange type. For `TokenForNFT` offers (buyer-created), the NFT seller (taker) actually receives `amount − royaltyAmount` because royalties are deducted from the offered amount at settlement. The GUI displays the inflated pre-royalty figure as "Net Proceeds" and as the override amount in the offer summary, causing the NFT seller to approve an offer believing they will receive more tokens than they actually will.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as:

```ts
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The commented-out branch — which would have subtracted `royaltyAmount` and `makerFee` for the `TokenForNFT` case — was disabled, leaving `nftSellerNetAmount = amount` for all exchange types. [1](#0-0) 

This value propagates to two display surfaces in `NFTOfferViewer.tsx`:

1. **`overrideNFTSellerAmount`** — computed from `nftSaleInfo?.nftSellerNetAmount` and passed to `NFTOfferSummary` as the displayed receive amount in the "Purchase Summary" panel shown to the NFT seller when they import and review the offer before accepting. [2](#0-1) 

2. **"Net Proceeds" label** — rendered directly from `nftSaleInfo?.nftSellerNetAmount` with a tooltip that explicitly states *"The net proceeds include the asking price, minus any associated creator fees"*, yet the displayed value does **not** subtract creator fees. [3](#0-2) 

In Chia's offer protocol, royalties are paid by the **taker**. When a buyer creates a `TokenForNFT` offer offering `X` tokens, the NFT seller (taker) receives `X − royalties` at settlement. The GUI shows `X` as "Net Proceeds" and as the override amount in the summary — the full pre-royalty figure — before the seller clicks "Accept Offer". [4](#0-3) 

---

### Impact Explanation

An NFT seller imports a buyer's `TokenForNFT` offer. The GUI's "Purchase Summary" and "Net Proceeds" panels display the full offered token amount (e.g., 10 XCH) as what the seller will receive. The seller approves the offer on this basis. At settlement, the protocol deducts royalties (e.g., 5% = 0.5 XCH), so the seller actually receives 9.5 XCH — below the amount they believed they were agreeing to. The seller has approved a transaction displaying the wrong receive amount, resulting in a direct, unrecoverable asset shortfall.

This matches the High impact category: *"unsafe trust of offer state that causes a user to approve the wrong amount."*

---

### Likelihood Explanation

Any NFT with a non-zero `royaltyPercentage` triggers this path. A buyer can craft a `TokenForNFT` offer for any such NFT and share the offer file. The NFT seller, relying on the GUI's "Net Proceeds" display (which even has a tooltip promising the royalty-subtracted figure), accepts without independently computing the net amount. No special attacker capability is required beyond creating a standard offer for an NFT with royalties.

---

### Recommendation

Restore the commented-out ternary in `calculateNFTRoyalties` so that `nftSellerNetAmount` correctly reflects the post-royalty amount for `TokenForNFT` offers:

```ts
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.NFTForToken
    ? amount
    : parseFloat((amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12));
``` [5](#0-4) 

Ensure `overrideNFTSellerAmount` and the "Net Proceeds" label in `NFTOfferViewer` both derive from the corrected value, so the seller sees the actual post-royalty amount before accepting. [6](#0-5) 

---

### Proof of Concept

1. Mint an NFT with `royaltyPercentage = 500` (5%).
2. As a buyer, create a `TokenForNFT` offer offering 10 XCH for the NFT.
3. Share the offer file with the NFT seller.
4. Seller imports the offer. The GUI "Net Proceeds" panel displays **10 XCH** (the full offered amount). The tooltip reads "asking price, minus any associated creator fees" — but the value shown is not minus creator fees.
5. Seller accepts. At settlement, 0.5 XCH royalty is deducted; seller receives **9.5 XCH**.
6. The seller approved a transaction displaying 10 XCH but received 9.5 XCH — a 5% shortfall with no on-screen warning.

The root cause is `nftSellerNetAmount = amount` (line 314 of `utils.ts`) instead of `amount − royaltyAmount` for the `TokenForNFT` exchange type, directly analogous to Slingshot's premature `finalOutputAmount >= finalAmountMin` check before the fee-on-transfer deduction. [7](#0-6)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
