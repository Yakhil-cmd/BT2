### Title
`set_eth_connector_contract_account` Breaks Existing User Withdrawals Without State Migration — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` atomically replaces the engine's pointer to the external ETH-connector contract but performs no state migration. After the swap, the engine's `withdraw` path issues `engine_withdraw` calls against the **new** connector, which holds zero user balances, while all existing user funds remain locked in the **old** connector. Because `engine_withdraw` on the old connector is gated to the aurora engine account, users have no independent path to recover their tokens.

---

### Finding Description

`set_eth_connector_contract_account` is the owner-callable function that updates which external NEP-141 contract the Aurora Engine treats as the canonical ETH bridge connector. [1](#0-0) 

The function writes exactly two storage keys: [2](#0-1) 

It does **not**:
- Verify that the new connector has the engine's account ID registered as its authorized `aurora_engine_account_id`.
- Migrate any user NEP-141 balances from the old connector to the new one.
- Migrate storage registrations.

After the swap, every call to `withdraw` on the engine creates a cross-contract promise targeting the **new** connector via `engine_withdraw`: [3](#0-2) 

The new connector has no record of any user's balance, so every `engine_withdraw` call fails. Meanwhile, `ft_on_transfer` uses the same stored key to decide whether an incoming NEP-141 transfer is base-ETH or an ERC-20: [4](#0-3) 

After the swap, transfers from the **old** connector are no longer recognized as ETH, so the deposit path also breaks for any user whose tokens are still on the old connector.

The old connector's `engine_withdraw` is restricted to the aurora engine account: [5](#0-4) 

Users cannot call it directly, and the engine no longer routes to it. There is no recovery path for affected users unless the owner reverts the connector pointer.

---

### Impact Explanation

All users who held ETH-on-NEAR balances on the old connector at the time of the swap lose the ability to withdraw those tokens to Ethereum. The engine's `withdraw` entry point is the only authorized path to `engine_withdraw`; after the pointer change it targets the wrong contract. Funds are effectively frozen until the owner manually reverts the connector address. If the owner does not revert, the freeze is permanent.

**Impact class:** Permanent freezing of funds (Critical) / Temporary freezing of funds (High) depending on whether the owner reverts.

---

### Likelihood Explanation

The function is owner-only, so the trigger requires the engine owner to call `set_eth_connector_contract_account` with a new account — a realistic operational action (connector upgrade, migration, bug fix). The owner may not realize that no balance migration occurs. The scenario is analogous to the original report: a legitimate administrative action silently breaks the protocol for all existing users.

---

### Recommendation

Before accepting the new connector address, the function should verify that:
1. The new connector already recognizes the engine's account ID as its authorized caller (`aurora_engine_account_id`).
2. Either the old connector has zero outstanding balances, or a migration plan is in place.

A minimal guard:

```rust
// Pseudo-code addition inside set_eth_connector_contract_account
// 1. Assert new connector's aurora_engine_account_id == env.current_account_id()
//    (via a synchronous view call or a pre-check promise).
// 2. Assert old connector total_supply == 0, or require explicit migration flag.
set_connector_account_id(io, &args.account);
set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
```

At minimum, emit a warning or revert if the old connector still holds a non-zero total supply, preventing silent fund freezes.

---

### Proof of Concept

1. Deploy Aurora Engine and initialize with connector `A`. Users deposit ETH; connector `A` holds their balances.
2. Owner calls `set_eth_connector_contract_account` with connector `B` (a fresh, empty contract).
3. User calls `withdraw` on the engine. The engine issues `engine_withdraw` to connector `B`. Connector `B` has no balance for the user → call fails.
4. User attempts to call `engine_withdraw` directly on connector `A` → rejected: "Method can be called only by aurora engine."
5. User's funds are frozen on connector `A` with no recovery path through the engine. [6](#0-5) [3](#0-2) [7](#0-6)

#[named]
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_connector_account_id<I: IO>(io: &I) -> Result<AccountId, ContractError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(errors::ERR_CONNECTOR_STORAGE_KEY_NOT_FOUND)
    .and_then(|x| {
        x.to_value()
            .map_err(|_| errors::ERR_BORSH_DESERIALIZE.as_bytes())
    })
    .map_err(Into::into)
}

pub fn set_connector_account_id<I: IO + Copy>(mut io: I, account_id: &AccountId) {
    io.write_borsh(
        &construct_contract_key(EthConnectorStorageId::EthConnectorAccount),
        account_id,
    );
}
