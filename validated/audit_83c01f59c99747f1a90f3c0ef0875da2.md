### Title
FLOW Tokens Permanently Stranded in EVM State When `CadenceOwnedAccount` Resource Is Destroyed - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` has no destructor guard that checks for a non-zero EVM balance before the resource is destroyed. Because the COA resource is the **sole** Cadence-side mechanism for authorizing withdrawals from its paired EVM address, destroying the resource while the EVM address still holds FLOW permanently strands those tokens in the EVM state with no recovery path. Any FLOW deposited into the COA's EVM address by a third party — analogous to the "donated tokens" in M-08 — is irrecoverably lost the moment the COA owner calls `destroy`.

---

### Finding Description

`EVM.CadenceOwnedAccount` is a Cadence resource that owns and controls a deterministic EVM address. FLOW tokens are moved into the EVM environment by calling `deposit` on the resource or directly on the `EVMAddress` struct. The `deposit` function on the COA resource is declared `access(all)`:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    self.address().deposit(from: <-from)
}
```

This means **any caller** can deposit FLOW into a COA's EVM address — including third parties who have borrowed a public capability to the COA, or who call `EVMAddress.deposit` directly with the known EVM address bytes.

The only way to move FLOW back from the EVM address to Cadence is through `CadenceOwnedAccount.withdraw`, which requires holding the resource with the `Owner | Withdraw` entitlement:

```cadence
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault { ... }
```

Cadence's resource type system allows any resource to be destroyed with `destroy`. The `CadenceOwnedAccount` resource has **no destructor** (`destroy` hook) that would:
- check whether the paired EVM address still holds a non-zero balance, or
- automatically withdraw and return that balance before destruction.

Once the resource is destroyed, the EVM address persists in the EVM state with its balance intact, but there is no Cadence-side object that can authorize a withdrawal from it. The tokens are permanently stranded.

The existing test at `fvm/evm/evm_test.go` explicitly demonstrates this path succeeds without error:

```cadence
let cadenceOwnedAccount <- EVM.createCadenceOwnedAccount()
cadenceOwnedAccount.deposit(from: <-vault)   // 1.23 FLOW deposited
destroy cadenceOwnedAccount                  // resource destroyed, balance stranded
```

---

### Impact Explanation

FLOW tokens held at a COA's EVM address become permanently inaccessible once the controlling `CadenceOwnedAccount` resource is destroyed. Unlike a normal Cadence resource that holds a `FlowToken.Vault` (where destruction would require the vault to be empty or explicitly handled), the COA resource holds **no vault itself** — the balance lives in the EVM state. Cadence's resource linearity rules therefore do not protect the EVM-side balance. The tokens are not burned (they remain in the EVM state and inflate the EVM total supply relative to the Cadence-side accounting), but they can never be withdrawn. This is a cross-VM asset loss: tokens leave Cadence accounting permanently without a corresponding EVM-side owner who can spend them.

---

### Likelihood Explanation

The entry path is reachable by any unprivileged transaction sender:

1. **Accidental self-loss**: A user creates a COA, deposits FLOW, and later destroys the resource (e.g., during account cleanup or contract migration) without first withdrawing. No warning or error is produced.
2. **Third-party donation loss**: A COA owner publishes a public capability (as shown in `setupCOA` in `fvm/evm/evm_test.go`). A third party deposits FLOW into the COA's EVM address via the public capability or via `EVMAddress.deposit` with the known address bytes. The COA owner then destroys the resource, permanently stranding the third party's tokens — a direct analog to the M-08 "donated tokens" scenario.
3. **Contract-level misuse**: Any Cadence contract that programmatically creates and destroys COAs (e.g., a factory pattern) without draining the EVM balance first will silently strand tokens.

No privileged access, key compromise, or node-level control is required.

---

### Recommendation

Add a `destroy` hook (destructor) to the `CadenceOwnedAccount` resource in `fvm/evm/stdlib/contract.cdc` that asserts the EVM balance is zero before allowing destruction:

```cadence
destroy() {
    pre {
        self.balance().isZero():
            "EVM.CadenceOwnedAccount: cannot destroy a COA with a non-zero EVM balance; withdraw all tokens first"
    }
}
```

Alternatively, the destructor could automatically withdraw the remaining balance and deposit it into a designated Cadence vault, though the simpler guard (panic on non-zero balance) is the safer default and mirrors how Cadence `FungibleToken.Vault` resources are expected to be handled.
