### Title
Hardcoded `ROUTER_EXEC_BASE` / `ROUTER_EXEC_PER_CALLBACK` Gas Constants in XCC Precompile Can Cause Permanent wNEAR Fund Freeze - (`File: engine-precompiles/src/xcc.rs`)

---

### Summary

The `CrossContractCall` precompile attaches a hardcoded amount of NEAR gas to the router's `execute` promise. If those constants become insufficient (e.g., after a NEAR protocol gas-cost repricing), the router call fails on the NEAR side while the user's wNEAR has already been irrevocably transferred to the engine's implicit EVM address, freezing those funds with no automatic refund path.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, two constants govern how much NEAR gas is forwarded to the XCC router's `execute` method:

```rust
pub const ROUTER_EXEC_BASE: NearGas = NearGas::new(7_000_000_000_000);   // 7 Tgas
pub const ROUTER_EXEC_PER_CALLBACK: NearGas = NearGas::new(12_000_000_000_000); // 12 Tgas
``` [1](#0-0) 

For an `Eager` XCC call these constants are combined and forwarded as the gas budget for the router:

```rust
let router_exec_cost = costs::ROUTER_EXEC_BASE
    + NearGas::new(callback_count * costs::ROUTER_EXEC_PER_CALLBACK.as_u64());
let promise = PromiseCreateArgs {
    ...
    attached_gas: router_exec_cost.saturating_add(call_gas),
};
``` [2](#0-1) 

**Before** that promise is dispatched, the precompile executes an EVM sub-call that transfers the user's wNEAR to the engine's implicit address:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
``` [3](#0-2) 

This EVM state change is committed as part of the outer EVM transaction. The NEAR promise is only created afterward as a log entry. If the NEAR promise subsequently fails (because `router_exec_cost` is too low for the router's `execute` method to complete), the EVM state change is **not** rolled back. The user's wNEAR balance is permanently debited from their EVM account and credited to the engine's implicit address, with no automatic refund.

The router's `execute` method itself has no gas-refund or error-recovery path:

```rust
pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [4](#0-3) 

The only refund mechanism in the router (`send_refund`) returns the storage-staking deposit, not the user's wNEAR: [5](#0-4) 

A secondary hardcoded constant compounds the risk: the NEAR-to-EVM gas conversion ratio used to price the XCC precompile call in EVM gas units is also fixed:

```rust
pub const CROSS_CONTRACT_CALL_NEAR_GAS: u64 = 175_000_000;
``` [6](#0-5) 

If NEAR protocol reprices operations (as it has done historically), this ratio becomes stale. An underestimate means the EVM gas charged to the user is too low, so the engine's NEAR gas budget for the entire transaction shrinks, making promise failure even more likely.

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of user funds.**

When an Eager XCC call is submitted:
1. The user's wNEAR ERC-20 balance is reduced and moved to the engine's implicit EVM address (committed EVM state).
2. A NEAR promise is created to call `router.execute()` with the hardcoded gas budget.
3. If the NEAR runtime's actual gas cost for `execute` exceeds `ROUTER_EXEC_BASE`, the promise fails.
4. The EVM state change is not reverted; the user's wNEAR is stuck in the engine's implicit address.
5. Recovery requires manual admin intervention (no on-chain refund path exists for this scenario).

---

### Likelihood Explanation

**Medium.**

- NEAR protocol has repriced gas operations in the past (e.g., storage and trie-node costs).
- The constants are explicitly annotated as "estimated from simulation tests," acknowledging they are approximations.
- Any EVM user can trigger the vulnerable path by calling the XCC precompile with `CrossContractCallArgs::Eager` — no special privilege is required.
- The window of exposure is every block after a NEAR gas repricing until the Aurora Engine is upgraded with corrected constants.

---

### Recommendation

1. Make `ROUTER_EXEC_BASE`, `ROUTER_EXEC_PER_CALLBACK`, and `CROSS_CONTRACT_CALL_NEAR_GAS` updatable by the engine owner/admin via a dedicated setter, analogous to how `factory_set_wnear_address` allows updating the wNEAR address.
2. Add a NEAR-side callback to the `execute` promise that detects failure and refunds the wNEAR to the originating EVM address, so that a gas-budget shortfall does not permanently freeze user funds.
3. Emit an on-chain event when the gas budget is within a configurable threshold of the actual cost, to give operators early warning before funds are at risk.

---

### Proof of Concept

**Trigger path (no special privilege required):**

1. Any EVM user constructs a `CrossContractCallArgs::Eager` payload with a `PromiseArgs::Create` containing a non-zero `attached_gas`.
2. They send an EVM transaction to `cross_contract_call::ADDRESS` (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`).
3. The precompile executes the wNEAR `transferFrom` sub-call, moving the user's wNEAR to the engine's implicit address (EVM state committed).
4. The precompile emits a log containing a `PromiseCreateArgs` with `attached_gas = ROUTER_EXEC_BASE + call_gas`.
5. The engine dispatches the NEAR promise. If `ROUTER_EXEC_BASE` (7 Tgas) is insufficient for the router's `execute` overhead after a NEAR gas repricing, the promise fails.
6. The user's wNEAR is gone from their EVM balance; the engine's implicit address holds it with no automatic return.
