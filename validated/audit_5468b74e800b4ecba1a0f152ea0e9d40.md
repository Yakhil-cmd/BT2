### Title
`skip_stateful_validations` Bypasses Signature Verification for Invoke Transactions with Nonce=1 When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry point only when a `deploy_account` transaction is pending in the mempool (UX improvement for the deploy+invoke pattern). However, it uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from the account — not specifically a `deploy_account` transaction. This allows an attacker to submit an invoke transaction with nonce=1 and an invalid/arbitrary signature for any account that already has a nonce=0 transaction in the mempool, bypassing gateway signature verification and gaining admission to the mempool.

---

### Finding Description

`skip_stateful_validations` triggers when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When all three hold, `skip_validate = true` is returned, and `run_validate_entry_point` sets `execution_flags.validate = false`, causing `StatefulValidator::perform_validations` to return `Ok(())` without ever calling the account's `__validate__` entry point: [2](#0-1) [3](#0-2) 

The code comment justifies condition 3 as: *"either it has a deploy_account transaction or transactions with future nonces that passed validations."* Both halves of this reasoning are flawed:

- **Case A (deploy_account in pool):** Correct — this is the intended use case.
- **Case B (any other tx in pool):** Incorrect — if account A has a nonce=0 invoke tx in the pool, that tx's `__validate__` was called with nonce=0 calldata/signature. It does **not** validate the nonce=1 transaction's signature. Each nonce produces a distinct transaction hash and distinct signature.

`account_tx_in_pool_or_recent_block` does not distinguish transaction types: [4](#0-3) 

The mempool's own `validate_tx` path also does not check signatures — it only checks for duplicates and fee escalation: [5](#0-4) 

The nonce check in `validate_nonce` permits nonce=1 when account_nonce=0 (within the default gap of 200): [6](#0-5) 

So the full gateway admission path — stateless check, nonce check, resource bounds check, mempool validation, and `__validate__` — is traversed with `__validate__` silently skipped.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject invoke transactions carrying an arbitrary/invalid signature into the mempool for any account that currently has a nonce=0 transaction pending. The gateway returns a success response (`GatewayOutput`) and forwards the transaction to the mempool. The batcher will later attempt execution with `validate: true` (the default), at which point `__validate__` runs and the transaction is dropped — but only after consuming sequencer resources (CPU, mempool slots, batcher execution cycles). Because the condition is nonce=1 and account_nonce=0, every account in the "just submitted nonce=0 tx" state is a target, which is a large and observable set during normal operation.

---

### Likelihood Explanation

The mempool is observable. Any party watching the public transaction stream can identify accounts with a pending nonce=0 transaction and immediately submit a crafted nonce=1 invoke with a garbage signature. No privileged access, no special contract, and no prior relationship with the victim account is required. The default `max_allowed_nonce_gap = 200` and `max_nonce_for_validation_skip = Nonce(1)` are both satisfied by the nonce=1 case. [7](#0-6) 

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific check that confirms a **`deploy_account`** transaction is pending for the sender address. The mempool should expose a dedicated predicate such as `has_pending_deploy_account(address) -> bool`. Alternatively, check on-chain state directly: if `get_class_hash_at(sender_address)` returns the zero class hash, the account is not yet deployed and skipping `__validate__` is safe; if the account is already deployed, `__validate__` must always run.

---

### Proof of Concept

```
1. Account A is deployed on-chain; on-chain nonce = 0.

2. Legitimate owner submits InvokeV3(sender=A, nonce=0, sig=VALID_SIG_0).
   → Gateway runs __validate__ → passes → tx enters mempool.
   → account_tx_in_pool_or_recent_block(A) now returns true.

3. Attacker submits InvokeV3(sender=A, nonce=1, sig=0xdeadbeef_GARBAGE).
   → stateless checks pass (valid structure, resource bounds non-zero).
   → validate_nonce: 0 ≤ 1 ≤ 200 → passes.
   → validate_by_mempool: no duplicate, no fee escalation issue → passes.
   → skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(A) == true  ✓  (nonce=0 tx is pooled)
     → returns true (skip __validate__)
   → run_validate_entry_point called with validate=false → __validate__ NOT called.
   → Gateway returns Ok(tx_hash) and forwards to mempool.

4. Attacker's nonce=1 tx with garbage signature is now in the mempool.
   Batcher later executes nonce=0 tx (account nonce → 1), then attempts nonce=1 tx
   with validate=true → __validate__ fails → tx dropped.
   Sequencer resources wasted; mempool slot consumed without authorization.
``` [8](#0-7) [9](#0-8) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
