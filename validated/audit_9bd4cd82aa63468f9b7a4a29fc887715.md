### Title
Cross-Address Replay of COA Ownership Proof Enables Unauthorized EVM ERC-1271 Authorization - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` commits to the target `evmAddress` or `CapabilityPath`. A valid signature produced by a Cadence account for one of its COAs can be replayed by an unprivileged attacker against a different COA owned by the same account, causing `isValidSignature` (ERC-1271) to return `ValidERC1271Signature` for the wrong COA. This is the direct Flow analog of the external report's "stateless, nonce-free signed-operation" class: a single user signature authorizes repeated or cross-target EVM actions without any on-chain state tracking.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` (lines 1001–1116) verifies two things:

1. That the supplied `signatures` are cryptographically valid over `signedData` for the Cadence account at `address`.
2. That the COA resource stored at `path` in that account has EVM address `evmAddress`.

The code's own comment at lines 1003–1009 documents the root cause:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

Because `signedData` is never required to commit to `evmAddress` or `CapabilityPath`, the two checks are decoupled: check (1) passes for any COA owned by the same Cadence account, and check (2) is satisfied by simply supplying the correct `path`/`evmAddress` pair for the target COA.

The ERC-1271 entry point in `fvm/evm/handler/coa/coa.sol` (lines 114–125) calls the Cadence Arch precompile `verifyCOAOwnershipProof(address(this), _hash, _sig)`. The `_sig` bytes are an RLP-encoded `COAOwnershipProof` struct (`fvm/evm/types/proof.go` lines 139–144) that contains `Address`, `CapabilityPath`, `KeyIndices`, and `Signatures` — but **not** the EVM address. The precompile (`fvm/evm/handler/precompiles.go` lines 115–153) forwards the call to `validateCOAOwnershipProof`, supplying `address(this)` as `evmAddress`.

**Exploit path (no privileged access required):**

1. Victim owns Cadence account `A` with two COAs: `COA_X` at `/public/coaX` and `COA_Y` at `/public/coaY`.
2. Victim legitimately signs `_hash` (e.g., an ERC-2612 permit hash or an NFT marketplace order hash) to authorize an action for `COA_X`. The resulting `COAOwnershipProof` encodes `CapabilityPath = "coaX"`.
3. Attacker observes the on-chain or off-chain proof (it is transmitted in EVM calldata and is fully public).
4. Attacker constructs a new `COAOwnershipProof` with identical `Address`, `KeyIndices`, and `Signatures`, but sets `CapabilityPath = "coaY"`.
5. Attacker calls `COA_Y.isValidSignature(_hash, modifiedProof)` from any EVM contract or EOA.
6. The precompile invokes `validateCOAOwnershipProof(A, /public/coaY, _hash, keyIndices, signatures, address(COA_Y))`.
7. Check (1) passes — the signatures are valid over `_hash` for account `A`.
8. Check (2) passes — the COA at `/public/coaY` has EVM address `address(COA_Y)`.
9. `isValidSignature` returns `ValidERC1271Signature` (0x1626ba7e) for `COA_Y`, even though the victim never authorized any action for `COA_Y`.

Any EVM contract that gates asset transfers or state changes on `isValidSignature` (e.g., ERC-2612 `permit`, Seaport-style order validation, Safe-style multisig) will now accept the replayed proof as a valid authorization from `COA_Y`.

Additionally, because there is no nonce or consumed-proof registry anywhere in the stack, the **same** `(signedData, proof)` tuple can be submitted an unlimited number of times against the same or different COAs — a direct temporal replay matching the external report's root cause.

### Impact Explanation

**Medium.** An unprivileged attacker who observes a single valid COA ownership proof can:

- Impersonate `COA_Y` in any ERC-1271-gated EVM contract, causing that contract to treat `COA_Y` as having signed arbitrary data it never signed.
- Replay the same proof indefinitely (no consumed-proof tracking), enabling repeated unauthorized authorizations.

Concrete consequences include unauthorized ERC-20/ERC-721 approvals, permit-based token drains, and acceptance of forged orders in EVM marketplaces — all without the victim's knowledge or any additional on-chain action by the attacker. The impact is bounded to users who hold multiple COAs under the same Cadence account, which is an uncommon but protocol-supported configuration.

### Likelihood Explanation

**Low.** Exploitation requires:

1. The victim to own at least two COAs under the same Cadence account.
2. The attacker to observe a valid `COAOwnershipProof` (available in EVM calldata).
3. An EVM contract that uses ERC-1271 without its own nonce/replay guard.

All three conditions are realistic and require no privileged access. The vulnerability is self-documented in the production source, confirming the development team is aware of the gap but has not closed it.

### Recommendation

Enforce that `signedData` commits to the target EVM address. The simplest fix is to require callers to include `evmAddress` in the signed payload and verify this inside `validateCOAOwnershipProof`:

```cadence
// Inside validateCOAOwnershipProof, after signature verification:
let expectedPrefix = evmAddress.toConstantSized<[UInt8; 20]>()!
// Require that signedData starts with (or hashes over) evmAddress
// so a proof for COA_X cannot be replayed for COA_Y.
```

Alternatively, introduce a consumed-proof registry (mapping `hash(signedData || evmAddress)` → `Bool`) to prevent both cross-address and temporal replay, analogous to the nonce mapping recommended in the external report.

        bytes32 _hash,
        bytes memory _sig
    ) external view virtual returns (bytes4){
        (bool ok, bytes memory data) = cadenceArch.staticcall(abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig));

        if (output) {
            return ValidERC1271Signature;
        }
        return InvalidERC1271Signature;
