### Title
Silent ERC-20 Token Burn Without NEP-141 Transfer in `EvmErc20V2.withdrawToNear` Due to Sender-Prefixed Precompile Input Misparse — (File: `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.withdrawToNear` prepends the caller's 20-byte Ethereum address to the `ExitToNear` precompile input before the 32-byte amount field. When the Aurora Engine is compiled **without** the `error_refund` feature, the precompile's `parse_input` strips only the 1-byte flag, leaving the 20-byte sender address as the leading bytes of what the precompile reads as the amount. The resulting 32-byte value is always far greater than `u128::MAX`, so `parse_amount` returns `ERR_INVALID_AMOUNT`. The precompile call fails silently (the assembly return value is never checked), ERC-20 tokens are already burned, and no NEP-141 tokens are ever transferred. Users permanently lose their bridged funds.

---

### Finding Description

**`EvmErc20V2.withdrawToNear` input layout** (line 58):

```solidity
bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
// Layout: flag(1) | sender(20) | amount(32) | recipient(N)
```

**`ExitToNear` precompile `parse_input` without `error_refund`** (line 788–791):

```rust
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])   // strips only the flag byte
}
```

After stripping the flag, the remaining slice is `sender(20) | amount(32) | recipient`. For flag `0x1`, the precompile then reads:

```rust
let amount = parse_amount(&input[..32])?;   // reads sender(20) + amount[:12]
``` [1](#0-0) 

The 32-byte value fed to `parse_amount` has the full 20-byte sender address in its high bits. Any non-zero sender address makes this value exceed `u128::MAX`:

```rust
fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    Ok(amount)
}
``` [2](#0-1) 

The precompile returns `Err`, the low-level `call` returns 0, but the Solidity assembly never checks the return value:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
}
``` [3](#0-2) 

The `_burn` on line 55 has already executed and is not reverted. The user's ERC-20 tokens are gone; no NEP-141 tokens are ever sent. [3](#0-2) 

**Symmetric issue with `EvmErc20.withdrawToNear` when `error_refund` IS enabled**: `EvmErc20.sol` does not include the sender prefix, so with `error_refund` the precompile strips `flag(1) + refund_address(20)` = 21 bytes, leaving `amount[20..32] + recipient` as the first 32 bytes parsed as the amount — again producing a value >> `u128::MAX` for any non-empty recipient string. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Every call to `EvmErc20V2.withdrawToNear` (without `error_refund`) silently burns the caller's ERC-20 tokens and transfers zero NEP-141 tokens. The burned ERC-20 supply is destroyed; the NEP-141 tokens remain locked in the Aurora contract with no recovery path. The symmetric case applies to `EvmErc20.withdrawToNear` with `error_refund` enabled.

---

### Likelihood Explanation

**High.** The vulnerability is triggered by any ordinary user calling the standard `withdrawToNear` interface on a deployed `EvmErc20V2` token. No special privileges, no front-running, no governance action required. The only precondition is the compile-time feature flag state, which is fixed at deployment and not user-controllable.

---

### Recommendation

1. **`EvmErc20V2.withdrawToNear`**: Guard the precompile call result and revert if it fails, so the `_burn` is rolled back:
   ```solidity
   assembly {
       let res := call(gas(), EXIT_TO_NEAR, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Unify input formats**: Decide on a single canonical input layout (with or without sender prefix) and ensure the Solidity contracts and the Rust precompile parser are always consistent regardless of feature flags.
3. **Add integration tests** that exercise `withdrawToNear` for both `EvmErc20` and `EvmErc20V2` under both `error_refund` configurations.

---

### Proof of Concept

```
Precondition: Aurora Engine compiled without `error_refund`; EvmErc20V2 deployed.

1. Alice holds 1000 EvmErc20V2 tokens (bridged from some NEP-141).
2. Alice calls: erc20v2.withdrawToNear("alice.near", 1000)
3. Inside withdrawToNear:
   - sender = Alice's EVM address, e.g. 0xABCD...1234 (20 bytes, all non-zero)
   - amount_b = bytes32(1000)
   - input = 0x01 | 0xABCD...1234(20B) | 0x000...03E8(32B) | "alice.near"
4. ExitToNear.run() calls parse_input → strips only flag byte.
5. parse_amount(input[0..32]) = U256::from_big_endian(0xABCD...1234 | 0x000...000)
   → value has 20 non-zero bytes in high bits → >> u128::MAX
   → returns Err("ERR_INVALID_AMOUNT")
6. Precompile call returns 0; assembly ignores it.
7. Transaction succeeds. Alice's 1000 ERC-20 tokens are burned.
8. No ft_transfer promise is created. Alice receives 0 NEP-141 tokens.
9. Funds permanently lost.
```
