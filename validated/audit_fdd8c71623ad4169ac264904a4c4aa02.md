### Title
Fee-on-Transfer Token Accounting Bug in Pool Delegation Functions - (File: src/pool/pool.cairo)

### Summary
The `enter_delegation_pool` and `add_to_delegation_pool` functions in `pool.cairo` use the caller-supplied `amount` parameter for both the inbound transfer from the delegator and the outbound transfer to the staking contract. If the pool's token (e.g., a BTC-wrapped token) charges a fee on transfer, the pool receives `amount - fee` from the delegator but immediately attempts to forward the full `amount` to the staking contract. Because the pool's balance is insufficient, the `checked_transfer_from` inside `add_stake_from_pool` reverts, permanently blocking delegation for that token.

### Finding Description
In `enter_delegation_pool`, the two consecutive calls are:

```cairo
// pool.cairo:198-199
transfer_from_delegator(:pool_member, :amount, :token_dispatcher);
self.transfer_to_staking_contract(:amount, :token_dispatcher, :staker_address);
```

`transfer_from_delegator` (pool/utils.cairo:20-28) pulls exactly `amount` tokens from the delegator into the pool contract. If the token deducts a fee during transfer, the pool's actual balance increases by only `amount - fee`.

`transfer_to_staking_contract` (pool.cairo:668-682) then approves the full `amount` to the staking contract and calls `add_stake_from_pool(:staker_address, :amount)`.

Inside `add_stake_from_pool` (staking.cairo:1003-1046), the staking contract first records the full `amount` in its accounting (delegated balance and total stake traces), then attempts to pull `amount` from the pool:

```cairo
// staking.cairo:1031-1034
token_dispatcher
    .checked_transfer_from(
        sender: pool_contract, recipient: get_contract_address(), amount: amount.into(),
    );
```

Since the pool only holds `amount - fee`, this call reverts. The entire transaction is rolled back, so no funds are lost in this specific call — but the delegation is permanently impossible for any fee-bearing token.

The identical pattern exists in `add_to_delegation_pool` (pool.cairo:238-239).

Notably, the staking contract already applies the correct balance-before/after guard for reward claims in `claim_from_reward_supplier` (staking/utils.cairo:50-60):

```cairo
let balance_before = token_dispatcher.balance_of(account: staking_contract);
reward_supplier_dispatcher.claim_rewards(:amount);
let balance_after = token_dispatcher.balance_of(account: staking_contract);
assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
```

This protection is absent in the pool delegation path.

### Impact Explanation
Any BTC-variant token added to the protocol whitelist that charges a transfer fee (or is later upgraded to do so) will make `enter_delegation_pool` and `add_to_delegation_pool` permanently revert for all delegators. No delegation can succeed, and no pool member can increase their stake. This constitutes **griefing with damage to users and the protocol** — the Medium impact class "Griefing with no profit motive but damage to users or protocol."

### Likelihood Explanation
The protocol explicitly supports non-STRK tokens (BTC-wrapped tokens) via `set_open_for_delegation` and the `btc_tokens` map. Wrapped tokens are proxy-upgradeable by design; a fee switch is a one-line governance change on many such tokens. USDT already carries fees on some chains. The protocol's own token-addition logic (`add_token` / `btc_tokens`) anticipates an expanding whitelist, increasing the surface area over time. The absence of any fee-on-transfer guard in the delegation path — despite the pattern being correctly applied elsewhere in the same codebase — makes this a latent structural defect.

### Recommendation
Apply the balance-before/after pattern (already present in `claim_from_reward_supplier`) to the pool's inbound transfer, and use the measured delta as the forwarded amount:

```cairo
let balance_before = token_dispatcher.balance_of(account: get_contract_address());
transfer_from_delegator(:pool_member, :amount, :token_dispatcher);
let balance_after = token_dispatcher.balance_of(account: get_contract_address());
let actual_amount: Amount = (balance_after - balance_before)
    .try_into()
    .expect_with_err(GenericError::AMOUNT_TOO_HIGH);
self.transfer_to_staking_contract(:actual_amount, :token_dispatcher, :staker_address);
// use actual_amount for all subsequent accounting
```

Apply the same fix to `add_to_delegation_pool`.

### Proof of Concept
Assume a BTC-wrapped token with a 1 % transfer fee is whitelisted.

1. Alice calls `pool.enter_delegation_pool(reward_address, 100_BTC)`.
2. `transfer_from_delegator` executes `checked_transfer_from(Alice → pool, 100)`. The token deducts 1 % fee; pool receives **99 BTC**.
3. `transfer_to_staking_contract` calls `token.approve(staking, 100)` and then `add_stake_from_pool(staker, 100)`.
4. Inside `add_stake_from_pool`, the staking contract records `+100 BTC` in the delegated-balance trace and total-stake trace, then calls `checked_transfer_from(pool → staking, 100)`.
5. The pool balance is only 99 BTC → the transfer reverts.
6. The entire transaction is rolled back. Alice's 100 BTC is returned, but she cannot delegate at all.
7. Every subsequent attempt by any delegator produces the same revert. The pool is effectively bricked for that token. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
