### Title
Floor Division in `delta_gas` Computation Systematically Undercharges Users for Native Resource Consumption — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` converts native resource consumption back into EVM gas via integer floor division. Because `native_per_gas` is computed with `div_ceil` (rounds up) while the inverse conversion `native_used / native_per_gas` uses plain integer division (rounds down), `delta_gas` is systematically underestimated on every transaction that consumes native resources beyond what EVM gas alone accounts for. Users pay less gas — and therefore less ETH fee — than the protocol intends, causing the operator to receive less fee revenue per block.

---

### Finding Description

ZKsync OS implements a "double resource accounting" model: every transaction is charged both EVM gas (ergs) and a native resource that models off-chain proving cost. When native resource consumption exceeds what EVM gas alone would imply, the difference is converted back to gas via `delta_gas` and added to `gas_used` before the final fee is settled.

**Step 1 — `native_per_gas` is computed with ceiling division (rounds up, favors protocol):**

In `validation_impl.rs` line 135:
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

**Step 2 — `native_per_pubdata` is computed with floor division (`wrapping_div`, rounds down, favors users):**

In `validation_impl.rs` line 142:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
``` [2](#0-1) 

This asymmetry is confirmed in `api/src/helpers.rs` where the comment explicitly notes the difference:
```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
// native_per_pubdata = pubdata_price / native_price   ← no rounding direction noted
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [3](#0-2) 

**Step 3 — `delta_gas` is computed with floor division in `compute_gas_refund`:**

In `refund_calculation.rs` line 72:
```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
``` [4](#0-3) 

The documentation confirms the intended formula is `deltaGas := (nativeUsed / nativePerGas) - gasUsed`, but does not specify rounding direction: [5](#0-4) 

**Combined rounding error chain:**

- `native_per_gas = ceil(gas_price / native_price)` → larger than true ratio
- `native_per_pubdata = floor(pubdata_price / native_price)` → smaller than true ratio → `native_used` for pubdata is smaller
- `delta_gas = floor(native_used / native_per_gas)` → floor of (smaller numerator / larger denominator) → doubly underestimated

Concrete example with `gas_price = 100`, `native_price = 3`, `pubdata_price = 100`, `pubdata_bytes = 1000`:
- True `native_per_gas` = 33.33 → computed = `ceil(100/3)` = **34**
- True `native_per_pubdata` = 33.33 → computed = `floor(100/3)` = **33**
- `native_used` for pubdata = 1000 × 33 = **33,000** (true: 33,333)
- `delta_gas` = `floor(33000 / 34)` = **970** (true: `33333 / 33.33` = **1000**)
- User pays 30 fewer gas units than they should → underpays by `30 × gas_price` ETH per transaction

---

### Impact Explanation

Every transaction that writes pubdata (storage writes, events, L2→L1 messages) triggers this undercharge. The operator receives less fee revenue than the protocol intends. Over a high-throughput block with many pubdata-heavy transactions, the cumulative shortfall is proportional to `(pubdata_bytes_per_block × rounding_error_per_byte × gas_price)`. This is a resource accounting bug that systematically favors users over the protocol — the direct analog of the reported `utilizeIndex` rounding issue.

---

### Likelihood Explanation

Every L2 transaction that produces pubdata (any storage write) reaches `compute_gas_refund` and triggers the floor-division undercharge. No special attacker setup is required — a normal ERC-20 transfer or any contract interaction that writes storage is sufficient. The entry path is fully unprivileged and reachable on every block.

---

### Recommendation

1. **`native_per_pubdata`**: Use `div_ceil` instead of `wrapping_div` to match the rounding direction of `native_per_gas`:
   ```rust
   let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
   ```

2. **`delta_gas`**: Use ceiling division when converting `native_used` back to gas, so any fractional native consumption is rounded up to the next gas unit:
   ```rust
   (native_used.div_ceil(native_per_gas)) as i64 - (gas_used as i64)
   ```
   If `delta_gas` can become negative after this change (e.g., due to the larger `native_per_gas` denominator), add a saturating floor at zero analogous to the `_repay()` adjustment recommended in the source report.

---

### Proof of Concept

1. Deploy any contract that writes to storage (e.g., `SSTORE` in a loop).
2. Submit a transaction calling it with a non-zero `gas_price` and `pubdata_price` where neither divides evenly by `native_price` (e.g., `gas_price = 100`, `pubdata_price = 100`, `native_price = 3`).
3. Observe `gas_used` reported by the bootloader.
4. Compute the expected `gas_used` using exact arithmetic: `native_used_exact / (gas_price / native_price)`.
5. The reported `gas_used` will be strictly less than the exact value by up to `native_per_gas - 1` gas units per transaction, confirming the systematic undercharge.
