### Title
`Debug` Trait Exposes Secret Signing Shares and Presign Nonce Material in Public API Types - (File: `src/lib.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`, `src/frost/mod.rs`)

---

### Summary

Multiple public-API output types that hold secret cryptographic material unconditionally derive `Debug`, making the raw secret values available to any code that formats those types with `{:?}`. This is the direct structural analog of the clipboard vulnerability: a standard, widely-used mechanism silently exposes sensitive material to any consumer of the library (log aggregators, error-reporting frameworks, test harnesses, tracing spans, etc.).

---

### Finding Description

The following production types derive `Debug` while simultaneously carrying secret scalar material and implementing `ZeroizeOnDrop` (indicating the authors recognise the sensitivity of the data):

**1. `KeygenOutput<C>` — long-term private signing share**

```rust
// src/lib.rs line 48
#[derive(Debug, Clone, Deserialize, Serialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,   // ← secret scalar
    #[zeroize[skip]]
    pub public_key: VerifyingKey<C>,
}
```

`private_share` is the participant's long-term Shamir share of the aggregate secret key. Printing `{:?}` on any `KeygenOutput` value emits this scalar in full.

**2. `PresignOutput` (OT-based ECDSA) — nonce share `k` and sigma share**


---

### Impact Explanation

**`KeygenOutput<C>` — Critical (private signing share disclosure)**

`private_share` is the participant's Shamir share of the aggregate secret key. Any caller that logs or formats a `KeygenOutput` (e.g., `tracing::debug!("{:?}", keygen_output)`, `format!("{keygen_output:?}")`, `assert_eq!` failure output, `unwrap()` panic messages) will emit the raw scalar. An adversary who collects `threshold` such shares — from logs, error reports, or crash dumps — can reconstruct the full aggregate private key and forge arbitrary signatures.

**`PresignOutput` variants — Critical (nonce material disclosure)**

For ECDSA, exposing the nonce share `k` together with the resulting signature `(r, s)` allows direct extraction of the private key share via `x = (s·k − m) / r`. For FROST, exposing `SigningNonces` allows the same attack. A single leaked presign output per participant is sufficient to compromise that participant's long-term key share.

---

### Likelihood Explanation

Rust's `Debug` derive is the default mechanism for logging, panic messages, `assert_eq!` failures, and tracing instrumentation. Library callers routinely write `log::debug!("{:?}", output)` or rely on frameworks that automatically format all fields. The `ZeroizeOnDrop` already present on these types demonstrates the authors' intent to protect the secrets at rest, but `Debug` bypasses that protection entirely during the object's lifetime. The risk is elevated because `KeygenOutput` is the primary return type of the public `keygen`, `reshare`, and `refresh` APIs, making inadvertent exposure highly likely in any real integration.

---

### Recommendation

1. **Remove `Debug` from all types carrying secret scalars.** Replace with a redacting implementation:
   ```rust
   impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
       fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
           f.debug_struct("KeygenOutput")
            .field("private_share", &"[REDACTED]")
            .field("public_key", &self.public_key)
            .finish()
       }
   }
   ```
2. Apply the same pattern to all `PresignOutput` and `RerandomizedPresignOutput` types, redacting every secret scalar field.
3. For `PresignOutput<C>` (FROST), redact `nonces` entirely.
4. Add a `#[deny(clippy::derive_debug_with_sensitive_fields)]` or equivalent lint gate to prevent future regressions.

### Proof of Concept

```rust
use threshold_signatures::{keygen, KeygenOutput, participants::Participant, ReconstructionLowerBound};

// After running DKG, a caller logs the output — standard practice:
let output: KeygenOutput<_> = run_keygen(...);
println!("{:?}", output);
// → KeygenOutput { private_share: SigningShare(Scalar(0xdeadbeef...)), public_key: ... }
// The raw secret scalar is now in stdout / log files / crash reports.
```

The `Debug` derive on `KeygenOutput<C>` is at [1](#0-0)  and directly exposes `private_share`.

The OT-based `PresignOutput` with secret `k` and `sigma` fields derives `Debug` at [2](#0-1) , and `RerandomizedPresignOutput` at [3](#0-2) .

The Robust ECDSA `PresignOutput` with four secret scalars derives `Debug` at [4](#0-3) , and `RerandomizedPresignOutput` at [5](#0-4) .

The FROST `PresignOutput<C>` exposing `SigningNonces` derives `Debug` at [6](#0-5) .

The `ZeroizeOnDrop` on `KeygenOutput<C>` confirms the authors' intent to protect `private_share`, making the simultaneous `Debug` derive a direct contradiction of that security goal. [1](#0-0)
