### Title
Malicious Participant Can Equivocate on `(alpha_j, beta_j)` at `wait1` to Cause Split-View Aborts in Presign — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

In `do_presign`, Round 2 broadcasts `(alpha_i, beta_i)` via `send_many` with no broadcast-consistency guarantee. A malicious participant can deliver different `(alpha_j, beta_j)` values to different honest participants. Because each honest participant independently verifies `alpha*G == K + A` using only the values it received, different participants will compute different aggregates and reach inconsistent pass/fail outcomes. This permanently prevents a quorum of honest participants from completing presign simultaneously.

---

### Finding Description

In Round 2 of `do_presign`, each participant broadcasts its `(alpha_i, beta_i)` shares: [1](#0-0) 

The `send_many` call produces a single `Action::SendMany(data)` action returned to the caller: [2](#0-1) 

The library does not enforce that the same bytes are delivered to every participant — that is entirely the caller's responsibility. A malicious participant is not bound by the library's `send_many` path; they can call `protocol.message(from, data)` on each honest participant's instance with different payloads: [3](#0-2) 

`recv_from_others` collects exactly one message per participant with no consistency check: [4](#0-3) 

After aggregation, each honest participant independently checks: [5](#0-4) 

There is no echo-broadcast, commitment, or cross-participant consistency mechanism protecting this exchange. The `echo_broadcast` module exists in the codebase but is not used here: [6](#0-5) 

---

### Impact Explanation

A malicious participant M sends the correct `alpha_M` to P1 (so P1's aggregate satisfies `alpha*G == K + A` and P1 proceeds) and sends `alpha_M + delta` to P2 (so P2's aggregate fails the check and P2 aborts). This creates a permanent split-view: no quorum of honest participants ever simultaneously holds a valid, consistent presign output. M can repeat this on every presign attempt, indefinitely blocking signing.

**Impact: High — Permanent denial of signing for honest parties under valid protocol inputs.**

---

### Likelihood Explanation

Any participant in the protocol controls the bytes delivered to each other participant's `message()` entry point. No cryptographic binding (commitment, echo broadcast, or MAC) prevents equivocation on `(alpha_j, beta_j)`. The attack requires only one malicious participant and is repeatable at zero cost.

---

### Recommendation

Protect the `(alpha_j, beta_j)` broadcast at `wait1` with an echo-broadcast or commit-then-reveal scheme so that all honest participants are guaranteed to see the same value from each sender. The existing `echo_broadcast` module should be applied to this round, analogous to how other protocols in the codebase use it for broadcast-consistency. [6](#0-5) 

---

### Proof of Concept

```
Setup: 3 participants P1, P2, M (malicious). Round 1 completes honestly.

Round 2:
  M computes correct alpha_M = k'_M + a'_M.
  M delivers (alpha_M,       beta_M)       to P1 via P1.message(M, ...)
  M delivers (alpha_M + 1,   beta_M)       to P2 via P2.message(M, ...)

P1 aggregates: alpha_1 = alpha_P1 + alpha_P2 + alpha_M       → alpha_1*G == K+A  ✓ → P1 proceeds
P2 aggregates: alpha_2 = alpha_P1 + alpha_P2 + alpha_M + 1   → alpha_2*G != K+A  ✗ → P2 aborts

Result: P1 holds a presign output; P2 does not.
        No threshold quorum of honest participants has a consistent presign.
        M repeats this on every attempt → permanent denial of signing.
``` [7](#0-6)

    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
