Looking at the `pay()` function and the `multicall` flow carefully to trace the exact execution path.

### Title
Stale Native ETH Balance in `PeripheryPayments.pay()` Allows Cross-User ETH Theft — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` reads `address(this).balance` — the router's **total** native ETH balance — when deciding how to fund a WETH swap leg. Any ETH left on the router from a prior transaction (because the sender omitted `refundETH()`) is silently consumed to settle a subsequent, unrelated user's swap obligation. The prior user loses their ETH; the later user (or an attacker watching the mempool) receives a proportional discount on their WETH input.

---

### Finding Description

The vulnerable branch in `pay()` is:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← total router ETH, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`multicall` is `payable` and loops over delegate-calls, so `msg.value` is available for the entire batch: [2](#0-1) 

**Step-by-step exploit:**

1. **User A** calls `router.multicall{value: 1000}([exactInputSingle(WETH→token1, amountIn=600)])` — no `refundETH()` appended.
2. During the swap callback, `pay(WETH, userA, pool, 600)` runs: `nativeBalance = 1000 ≥ 600`, so 600 ETH is deposited and sent to the pool. **400 ETH remains on the router.**
3. **Attacker** calls `router.exactInputSingle(WETH→token1, amountIn=400)` with zero `msg.value`.
4. During the callback, `pay(WETH, attacker, pool, 400)` runs: `nativeBalance = 400 ≥ 400`, so the router deposits user A's 400 residual ETH and transfers WETH to the pool. **The attacker pays zero WETH from their own wallet.**

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH pushes; it does not prevent ETH from accumulating via `msg.value` in payable function calls. [3](#0-2) 

---

### Impact Explanation

- **User A suffers a direct loss of principal** — their residual ETH is consumed to fund a stranger's swap.
- **The attacker receives free WETH input** — they execute a valid swap without spending any of their own tokens.
- The pool itself receives the correct WETH amount and is not insolvent; the loss is entirely borne by the user who left ETH on the router.

---

### Likelihood Explanation

- Any user who sends excess ETH in a multicall and omits `refundETH()` is vulnerable. The existing test suite demonstrates this is the expected pattern (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth`), but omission is easy and the router does not enforce it. [4](#0-3) 

- An attacker can passively monitor the router's ETH balance on-chain and front-run or follow any transaction that leaves residual ETH, requiring no special permissions.
- The `multicall` pattern is the primary intended interface for ETH-input swaps, making this a realistic user-facing path.

---

### Recommendation

Track only the ETH that belongs to the **current call** rather than the router's total balance. One approach: snapshot `address(this).balance` before the outermost `multicall` entry and pass the per-call ETH budget through transient storage (similar to how the callback context is already managed). Alternatively, require callers to wrap ETH to WETH themselves before calling the router, removing the native-ETH shortcut entirely from `pay()`.

---

```solidity

// Foundry fork test outline (pseudo-code):

//    — deliberately omit refundETH().
// 3. Assert address(router).balance == 400.
// 4. attacker: router.exactInputSingle{value: 0}(WETH->token1, amountIn=400)
//    — attacker has WETH allowance = 0, but the call succeeds.
// 5. Assert attacker's WETH balance is unchanged (paid nothing).
// 6. Assert address(router).balance == 0 (userA's ETH consumed).
// 7. Assert userA's ETH balance decreased by 1000 (lost 400 extra).
```

The pool receives the correct WETH in step 4 (the invariant the pool cares about holds), but user A's 400 ETH is irrecoverably consumed to fund the attacker's swap — confirming the cross-user ETH theft.
