### Title
Gateway Signature Verification Bypass for Invoke Transactions via `skip_stateful_validations` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (which performs signature verification) for any Invoke transaction with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true`. An unprivileged attacker can first submit a valid `deploy_account` for a fresh address (which places the address in the mempool's account set), then submit a second Invoke transaction with nonce=1 carrying an **invalid or absent signature**. The gateway accepts this second transaction into the mempool without ever calling `__validate__`, violating the invariant that every admitted transaction must have passed account-level signature verification.

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

The comment claims this is safe because the address can only be in the pool if it has a `deploy_account` or a future-nonce transaction that already passed validation. That reasoning is correct for the *first* transaction, but it says nothing about the *signature* of the nonce=1 invoke being submitted right now.

**How `skip_validate=true` propagates:** [2](#0-1) 

When `skip_validate` is `true`, `execution_flags.validate` is set to `false`. [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the Invoke branch checks `if !tx.execution_flags.validate { return Ok(()); }` — the `__validate__` call is never reached. [4](#0-3) 

`validate_tx` also short-circuits immediately when `execution_flags.validate == false`.

**Full gateway admission path:** [5](#0-4) 

After `extract_state_nonce_and_run_validations` returns without error, the transaction is forwarded to `mempool_client.add_tx` unconditionally. No subsequent signature check exists.

**`account_tx_in_pool_or_recent_block` semantics:** [6](#0-5) 

This returns `true` for **any** transaction type in the pool for the address — not specifically a `deploy_account`. An attacker who submits a valid `deploy_account` (which requires only a valid constructor + `__validate_deploy__` call) immediately satisfies this predicate for any subsequent invoke.

**`validate_by_mempool` does not check signatures:** [7](#0-6) 

`validate_by_mempool` calls `mempool.validate_tx`, which only checks for duplicate tx hash and nonce-too-old: [8](#0-7) 

No signature content is inspected at any point in the admission path when `skip_validate=true`.

**Exploit flow:**

1. Attacker generates key pair `(K, P)` and computes the counterfactual address `A`.
2. Attacker submits a valid `deploy_account` for `A` (signed with `K`). It passes full stateful validation and enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits an `Invoke` for `A` with `nonce=1` and a **garbage/empty signature**.
4. Gateway stateless check passes (only checks signature *length*, not validity). [9](#0-8) 

5. `validate_nonce` passes (`0 ≤ 1 ≤ 0 + max_gap`).
6. `validate_by_mempool` passes (not a duplicate, nonce not too old).
7. `skip_stateful_validations` returns `true` → `__validate__` is skipped.
8. Gateway calls `mempool_client.add_tx` — the invalid invoke is now in the mempool.

### Impact Explanation

The gateway admits an Invoke transaction whose signature has never been verified into the mempool. This directly matches the High impact category: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

During batcher execution the transaction will eventually fail (the batcher uses `new_for_sequencing` which sets `validate=true`): [10](#0-9) 

So no funds are stolen and no state is corrupted. However:
- The mempool accepts and stores transactions that violate the fundamental admission invariant (every admitted transaction must have passed account signature verification).
- An attacker can repeat this at scale (one `deploy_account` per fresh address, then arbitrarily many invalid invokes per address until the mempool's per-address nonce slot is exhausted), polluting the mempool and wasting batcher execution resources.
- The invalid transactions consume mempool capacity, potentially displacing valid transactions.

### Likelihood Explanation

Exploitation requires no privileged access. Any external user can generate a fresh key pair, submit a `deploy_account`, and immediately follow with an invalid invoke. The `deploy_account` requires paying resource bounds, but the invalid invoke does not require any actual fee payment at admission time (fee is only charged during execution, which will revert). The attack is cheap, repeatable, and requires no special knowledge beyond the public API.

### Recommendation

The `skip_stateful_validations` function should not skip `__validate__` solely because the address appears in the mempool. Two concrete options:

1. **Verify the invoke signature against the deploy_account's public key at admission time.** When `skip_validate` would be `true`, look up the pending `deploy_account` transaction for the address, extract the account's public key from its constructor calldata, and verify the invoke's signature against it before admitting the transaction.

2. **Restrict the skip to a tighter predicate.** Instead of checking `account_tx_in_pool_or_recent_block` (which matches any transaction type), check that a `deploy_account` transaction specifically exists in the pool for the address. This does not fix the signature bypass but at least prevents the predicate from being satisfied by non-deploy-account transactions.

The long-term fix is option 1. Option 2 is a partial mitigation that narrows the attack surface but does not eliminate the signature bypass for the legitimate deploy+invoke UX flow.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
