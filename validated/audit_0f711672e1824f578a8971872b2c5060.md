### Title
Missing Old-Code Storage Refund on AccountId-Mode Global Contract Redeployment Permanently Burns Excess NEAR — (`runtime/runtime/src/global_contracts.rs`)

### Summary

`action_deploy_global_contract` always charges `global_contract_storage_amount_per_byte * new_code.len()` and burns that amount unconditionally on every invocation. When an account redeploys a global contract in `AccountId` mode (replacing an existing one), no refund for the old code's storage cost is issued. The deployer permanently loses `global_contract_storage_amount_per_byte * old_code.len()` NEAR on every redeployment, violating the ledger invariant that storage costs reflect actual net storage consumed.

### Finding Description

In `action_deploy_global_contract`: [1](#0-0) 

The function computes `storage_cost` as the **full** new code length multiplied by the per-byte rate, deducts it from `account.amount`, and adds it to `result.tokens_burnt` — permanently burning it. There is no read of any existing code stored under the `AccountId` trie key, no delta computation, and no refund path for the old code's previously paid storage cost.

When the distribution receipt is later applied in `apply_distribution_current_shard`, it simply overwrites the old code in the trie: [2](#0-1) 

No balance is returned to the deployer at this point either. The only place `global_contract_storage_amount_per_byte` is used in the entire runtime is the single charge site in `action_deploy_global_contract`.

**Concrete ledger corruption on redeployment:**

| Step | Operation | `account.amount` change | `tokens_burnt` change |
|---|---|---|---|
| 1 | Deploy 4 MB contract (AccountId mode) | `−4_194_304 * rate` | `+4_194_304 * rate` |
| 2 | Redeploy 1-byte contract (AccountId mode) | `−1 * rate` | `+1 * rate` |
| **Net** | Only 1 byte of global storage in use | `−(4_194_304 + 1) * rate` | `+(4_194_304 + 1) * rate` |

The deployer has been charged for 4 MB + 1 byte but only 1 byte of global contract storage is currently occupied. The excess `4_194_304 * rate` NEAR is permanently burned with no recovery path.

The documentation for this action states the error condition is `LackBalanceForState` "if the account does not hold enough NEAR to cover the **added** storage," implying the charge should be for the net addition, not the full new size. [3](#0-2) 

The existing test suite confirms `tokens_burnt` includes the full storage cost on first deploy but has no test for the redeployment case: [4](#0-3) 

### Impact Explanation

Any account that deploys a large global contract in `AccountId` mode and later redeploys a smaller one (or any different-sized one) permanently loses NEAR proportional to the **old** code size. With `global_contract_storage_amount_per_byte = 0.0001 N` and a 4 MB initial contract, a single redeployment burns an extra **419.43 NEAR** that should have been refunded. This loss compounds with every subsequent redeployment.

### Likelihood Explanation

`AccountId` mode is the explicitly supported upgrade path for global contracts (it is the only mode that allows updating the contract for all users). Any developer who deploys a large contract and later updates it — the primary intended use case — will silently lose funds. The operation requires only a standard signed transaction from the account owner.

### Recommendation

Before charging the new storage cost, read the existing code stored under `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(account_id) }` from `state_update`, compute its length, and refund `global_contract_storage_amount_per_byte * old_code.len()` to `account.amount` before deducting the new cost. Only the net difference `(new_len - old_len) * rate` should be charged (or refunded if the new code is smaller). The `tokens_burnt` field should be adjusted accordingly.

### Proof of Concept

A runtime `apply` test that:
1. Sets up an account with a known balance and `global_contract_storage_amount_per_byte = R`.
2. Applies a `DeployGlobalContractAction` with `code = [0u8; 4_194_304]` in `AccountId` mode; records `balance_after_first`.
3. Applies a second `DeployGlobalContractAction` with `code = [0u8; 1]` in `AccountId` mode; records `balance_after_second`.
4. Asserts `balance_after_first - balance_after_second == 1 * R` (only 1 byte net change).
5. The assertion will **fail** with the current code, showing `balance_after_first - balance_after_second == 1 * R` but the actual difference is `1 * R` while the total burned from initial is `(4_194_304 + 1) * R` instead of the correct `4_194_304 * R + 1 * R - 4_194_304 * R = 1 * R`.
