### Title
NFT Offer Net Proceeds Displayed as Gross Amount Due to Commented-Out Royalty Deduction — (File: `packages/gui/src/components/offers/utils.ts`)

### Summary

`calculateNFTRoyalties` in `utils.ts` sets `nftSellerNetAmount` to the raw asking `amount` instead of `amount - royaltyAmount`. The correct deduction is present in the codebase but has been commented out. This causes the offer editor and viewer to display an inflated "You will receive" / "Net Proceeds" figure to NFT sellers, mirroring the external report's pattern of omitting a component from a net-amount calculation.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The commented-out block is the correct formula. As written, `nftSellerNetAmount` equals the full asking price, not the seller's actual net proceeds after royalties are deducted.

This value propagates to two critical display paths:

1. **Offer editor** (`NFTOfferEditor.tsx` line 365): Under the "Sell an NFT" tab, the label "You will receive" renders `nftSellerNetAmount` — showing the full asking price instead of `amount - royaltyAmount`. [2](#0-1) 

2. **Offer viewer** (`NFTOfferViewer.tsx` lines 595–597): The "Net Proceeds" section (with tooltip "The net proceeds include the asking price, minus any associated creator fees") also renders the uncorrected `nftSellerNetAmount`. [3](#0-2) 

3. **`overrideNFTSellerAmount`** (`NFTOfferViewer.tsx` lines 397–402): This value is derived from `nftSellerNetAmount` and passed into `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it overrides the displayed token amount in the offer summary panel shown to the accepting party. [4](#0-3) 

### Impact Explanation

An NFT seller sets an asking price of 100 XCH for an NFT with a 10% creator royalty. The GUI displays "You will receive: 100 XCH" and "Net Proceeds: 100 XCH." The seller creates the offer believing they will net 100 XCH. Upon acceptance, the protocol correctly deducts the 10 XCH royalty, and the seller receives only 90 XCH — a 10% shortfall relative to what the GUI promised. The impact scales with royalty percentage and offer size. This matches the allowed High impact category: spoofing of offer state that causes a user to display the wrong amount, leading to a concrete financial discrepancy.

### Likelihood Explanation

Any NFT offer involving an NFT with a non-zero `royaltyPercentage` triggers this path. NFT royalties are a standard feature of the Chia NFT1 standard, so this affects a broad class of normal user interactions. No special attacker capability is required — the bug is triggered by the seller's own offer creation flow.

### Recommendation

Restore the commented-out net-proceeds calculation:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [1](#0-0) 

Add unit tests for `calculateNFTRoyalties` covering non-zero royalty percentages to verify that `nftSellerNetAmount` is always less than `amount` when royalties apply.

### Proof of Concept

1. Open the Chia GUI and navigate to the NFT offer editor.
2. Select "Sell an NFT" tab and choose an NFT with `royaltyPercentage > 0` (e.g., 10%).
3. Enter an asking price of 100 XCH.
4. Observe: "You will receive: 100 XCH" — the royalty deduction is absent.
5. Create the offer and have a counterparty accept it.
6. Observe: the seller's wallet receives 90 XCH, not 100 XCH, because the 10 XCH royalty is correctly enforced on-chain.
7. The discrepancy between the displayed "Net Proceeds" (100 XCH) and the actual received amount (90 XCH) confirms the accounting bug.
