### Title
Permanent ERC-20 Token Loss When NEAR-Side `ft_transfer` Fails Without `error_refund` Feature — (`File: etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`, `engine-precompiles/src/native.rs`)

---

### Summary

In `EvmErc20.withdrawToNear` and `EvmErc20V2.withdrawToNear`, ERC-20 tokens are burned on the EVM side **before** the exit precompile schedules the corresponding NEAR-side `ft_transfer`. When the `error_refund` feature is not compiled in (it is **not** a default feature), no callback is attached to the NEAR promise. If the NEAR-side `ft_transfer` fails (e.g., the recipient lacks storage deposit for the NEP-141 token), the burned ERC-20 tokens are permanently lost with no refund path.

---

### Finding Description

**Step 1 — Tokens burned before the NEAR promise is scheduled.**

In both `EvmErc20.sol` and `EvmErc20V2.sol`, `_burn` is called unconditionally before the exit precompile is invoked:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here
    // ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, ...);
        // res is never checked
    }
}
```

**Step 2 — The precompile schedules a bare `ft_transfer` promise with no error callback.**

In `engine-precompiles/src/native.rs`, the callback structure is:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,   // None for regular ERC-20
};

let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← NO callback attached
} else {
    PromiseArgs::Callback(...)
};
```

For a regular ERC-20 token with `error_refund` disabled, `callback_args` equals the default (both fields `None`), so the promise is a bare `Create` with **no error-handling callback**.

**Step 3 — `error_refund` is not a default feature.**

From `engine/Cargo.toml`:

```toml
[features]
default = ["std"]
error_refund = ["aurora-engine-precompiles/error_refund"]
```

`error_refund` must be explicitly opted into. In a standard production build it is absent, meaning the `exit_to_near_precompile_callback` is never scheduled for regular ERC-20 withdrawals.

**Step 4 — NEAR-side `ft_transfer` can fail for unprivileged reasons.**

On NEAR Protocol, `ft_transfer` reverts if the recipient account has not registered a storage deposit for the NEP-141 token. This is a routine, user-triggerable condition requiring no admin action.

---

### Impact Explanation

When `ft_transfer` fails on NEAR:
- The ERC-20 tokens are already burned (irreversible on the EVM side).
- No refund callback exists to re-mint them.
- The user permanently loses their bridged assets.

**Impact: Critical — Permanent freezing/loss of user funds.**

---

### Likelihood Explanation

Any user who calls `withdrawToNear` targeting a NEAR account that has not registered storage for the specific NEP-141 token will trigger this path. Storage registration is a prerequisite that many users overlook, and the ERC-20 contract provides no guard against it. The entry path is fully unprivileged and reachable from any EVM caller.

---

### Recommendation

1. **Enable `error_refund` by default** in production builds, or make it a hard requirement for the `contract` feature.
2. **Alternatively**, restructure `withdrawToNear` to call the exit precompile first and only burn tokens inside a success callback, mirroring the check-effects-interactions pattern.
3. At minimum, **check the return value `res`** of the assembly `call` and revert if it is zero, so the `_burn` is rolled back when the precompile itself fails.

---

### Proof of Concept

1. Deploy `EvmErc20` for a NEP-141 token (standard Aurora bridge setup, `error_refund` not enabled).
2. Bridge tokens into Aurora so a user holds ERC-20 balance.
3. Create a fresh NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
4. Call `EvmErc20.withdrawToNear(freshAccount, amount)` from the EVM.
5. Observe: ERC-20 tokens are burned; the NEAR `ft_transfer` receipt fails; no refund receipt is created; the user's funds are permanently gone.
