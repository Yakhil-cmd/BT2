### Title
`increased_enough` Zero-Value Bypass Allows Fee-Escalation Replacement Without Tip Increase — (`crates/apollo_mempool/src/mempool.rs`)

### Summary

`Mempool::increased_enough` returns `true` for any incoming tip when the existing queued transaction has `tip=0`, because `0 * percentage / 100 = 0` and `escalation_qualified_value = 0 + 0 = 0`, making `incoming_value >= 0` trivially true. An unprivileged attacker can replace a victim's `tip=0` queued transaction with their own `tip=0` transaction (while satisfying only the `max_l2_gas_price` escalation), changing the sequencing order and front-running the victim's execution without paying any tip premium.

### Finding Description

`Mempool::increased_enough` is called by `should_replace_tx` for both the tip and `max_l2_gas_price` dimensions: [1](#0-0) 

When `existing_value = 0`:

```
0.checked_mul(percentage) = 0
div_ceil(0, 100)           = 0
0.checked_add(0)           = 0   → escalation_qualified_value = 0
incoming_value >= 0        = true  (for all u128)
```

The function's own comment states the intent: *"Round up the required increase so any positive percentage always demands a strictly higher fee, even for small values."* [2](#0-1) 

This intent is violated when `existing_value = 0`: the required increase rounds to zero, so the threshold collapses to zero and any incoming value (including zero) satisfies it.

The gateway's stateless validator enforces `ZeroResourceBounds` (resource bounds without tip must be non-zero) and a `min_gas_price` floor on `max_price_per_unit`, but imposes **no minimum on `tip`**: [3](#0-2) 

A `tip=0` transaction is therefore a valid, normally-admitted transaction. This is confirmed by the existing test `test_fee_escalation_valid_replacement_minimum_values`, which uses `tip: 0` for the existing transaction: [4](#0-3) 

That test only verifies that a `tip=1` replacement is accepted; it does not test whether a `tip=0` replacement is (incorrectly) accepted.

### Impact Explanation

An attacker observes a victim's queued transaction with `tip=0` and `max_l2_gas_price=P`. The attacker submits a replacement transaction for the same `(address, nonce)` with:
- `tip = 0` (same as victim — tip escalation check bypassed via zero-value collapse)
- `max_l2_gas_price = P * 1.10 + 1` (satisfies the gas price escalation check legitimately)

`validate_fee_escalation` calls `should_replace_tx`, which calls `increased_enough(0, 0)` for the tip dimension — returning `true` — and `increased_enough(P, P*1.1+1)` for the gas price dimension — also returning `true`. The replacement is accepted: [5](#0-4) 

`remove_replaced_tx` then removes the victim's transaction from both the queue and the pool: [6](#0-5) 

The attacker's transaction (with different calldata/payload and a different `tx_hash`) is sequenced in place of the victim's, achieving front-running without paying any tip premium. The fee escalation invariant — that replacement always requires a strictly higher tip — is violated.

### Likelihood Explanation

- `tip=0` transactions are valid and accepted by the production gateway (no tip floor exists in `StatelessTransactionValidator`).
- The production mempool config has `enable_fee_escalation: true` and `fee_escalation_percentage: 10`, so the code path is active.
- No privileged access is required; any unprivileged user can submit a `tip=0` transaction and any other unprivileged user can exploit the bypass.
- The existing test suite does not cover the `tip=0` replacement case, so the bug is undetected.

### Recommendation

Add an explicit guard in `increased_enough` for the zero base-value case. The simplest correct fix is to require `incoming_value > existing_value` when `existing_value == 0` (i.e., always require a strictly positive incoming value):

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);

    let Some(escalation_qualified_value) = existing_value
        .checked_mul(percentage)
        .map(|scaled| scaled.div_ceil(100))
        // Enforce a minimum increase of 1 so that existing_value=0 always
        // requires incoming_value >= 1.
        .map(|required_increase| required_increase.max(1))
        .and_then(|required_increase| existing_value.checked_add(required_increase))
    else {
        return false;
    };

    incoming_value >= escalation_qualified_value
}
```

This ensures that for any `existing_value` (including 0), the required increase is at least 1, so `incoming_value` must be strictly greater than `existing_value`.

```rust
fn test_zero_tip_bypass_allows_same_tip_replacement() {
    // Victim submits tip=0 tx (valid, no tip floor in gateway).
    let existing_tx = tx!(tx_hash: 0, tip: 0, max_l2_gas_price: 100);
    let mempool = MempoolTestContentBuilder::new()
        .with_pool([existing_tx.clone()])
        .with_fee_escalation_percentage(10)
        .build_full_mempool();

    // Attacker submits tip=0 replacement with 10%+1 higher gas price.
    // increased_enough(0, 0) == true  ← tip check bypassed
    // increased_enough(100, 111) == true ← gas price check passes

    // Assert: replacement is accepted and victim's tx is removed.
    validate_and_add_tx_and_verify_replacement_in_pool(mempool, attacker_replacement);
    // Victim's tx_hash=0 is gone; attacker's tx_hash=1 is sequenced instead.
}
```

The test demonstrates that `increased_enough(0, 0)` returns `true`, the replacement is admitted, and the victim's transaction is evicted — all without the attacker paying any tip.
