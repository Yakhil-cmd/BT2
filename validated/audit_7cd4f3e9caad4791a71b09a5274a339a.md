### Title
WalletConnect Offer Approval Displays Understated Royalty Total for Multi-NFT Offers — (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

The `formatAmountWithRoyalties` function in the WalletConnect command display pipeline incorrectly divides the fungible amount equally among all NFTs before computing each royalty, producing a displayed total that is systematically lower than the amount the protocol will actually charge. A malicious WalletConnect dapp can exploit this to cause a user to approve a `take_offer` transaction believing they are paying significantly less than the real cost.

### Finding Description

`parseCommandDisplay` is the function that generates the human-readable approval dialog shown to users before they confirm a WalletConnect request. For `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`, it calls `formatAmountWithRoyalties` to compute the `amountWithRoyalties` field displayed in the confirmation UI.

The faulty logic is:

```typescript
// parseCommandDisplay.ts lines 363-367
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

When N NFTs are present, the code divides the fungible amount by N and computes each NFT's royalty on that reduced `splitAmount`. The correct behavior — as implemented in the regular offer builder via `nft_calculate_royalties` — is to compute each NFT's royalty on the **full** fungible amount, because each NFT creator is independently entitled to their royalty percentage of the entire sale price.

**Correct formula (protocol / `OfferBuilderProvider`):**
```
royaltyTotal = Σ (amount × royaltyPercentage_i / 10_000)
```

**Displayed formula (WalletConnect dialog):**
```
royaltyTotal = Σ ((amount / N) × royaltyPercentage_i / 10_000)
```

The display underestimates royalties by a factor of N. The existing test at line 251–334 of `parseCommandDisplay.test.ts` encodes and validates this incorrect behavior, confirming it is the live production path.

### Impact Explanation

A user who has an active WalletConnect session with a malicious dapp is shown a fabricated `amountWithRoyalties` total in the approval dialog. They approve the transaction believing they are spending, for example, 10.5 XCH, while the actual on-chain spend is 11 XCH (or more, depending on the number of NFTs and royalty percentages). The discrepancy scales linearly with the number of NFTs and their royalty rates. This constitutes a direct, concrete asset loss caused by the user approving the wrong amount — matching the High-impact criterion: *WalletConnect state that causes a user to approve the wrong amount*.

**Concrete example** (2 NFTs, each 5% royalty, 10 XCH offer):
| | Displayed | Actual |
|---|---|---|
| Base amount | 10 XCH | 10 XCH |
| NFT1 royalty (5%) | 0.25 XCH | 0.5 XCH |
| NFT2 royalty (5%) | 0.25 XCH | 0.5 XCH |
| **Total shown/charged** | **10.5 XCH** | **11 XCH** |

### Likelihood Explanation

The attacker must have an active WalletConnect session with the victim. This is a realistic precondition: WalletConnect is a supported, advertised feature, and users routinely connect dapps. The attacker controls the offer blob passed to `take_offer` and can freely craft an offer containing multiple NFTs (minted by the attacker with high royalty percentages set on-chain). No key compromise, phishing, or host access is required beyond the already-established WalletConnect session.

### Recommendation

Replace the split-amount approximation with the same per-NFT full-amount calculation used by the regular offer builder. Each royalty should be computed on the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
// Remove the splitAmount line entirely
```

Alternatively, call the `nft_calculate_royalties` RPC (already available via `nftGetInfo` infrastructure) to obtain the authoritative royalty amounts from the daemon, matching the regular offer builder path in `OfferBuilderProvider`.

### Proof of Concept

The existing test at `parseCommandDisplay.test.ts` lines 251–334 demonstrates the incorrect behavior. With 2 NFTs (royalties 500 and 10) and 100,000,000 mojos:

- **Displayed** `amountWithRoyalties`: `0.00010255` XCH (test asserts this value — `splitAmount = 50,000,000`, royalties computed on half the amount)
- **Correct** `amountWithRoyalties`: `0.0001051` XCH (each royalty computed on full 100,000,000 mojos)

The gap is 2,550,000 mojos on a 100,000,000 mojo offer. Scaled to a 10 XCH offer with two 5% royalty NFTs, the user is deceived by 0.5 XCH. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
