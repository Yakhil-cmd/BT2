### Title
Non-Atomic Two-Step Balance Update in `transfer_from_treasury` Causes Permanent Token Loss on `MintingBalanceOverflow` — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`transfer_from_treasury` performs a two-step balance update — first debiting the treasury, then crediting the recipient — with no rollback frame wrapping both operations. If the treasury deduction succeeds but the recipient credit fails with `MintingBalanceOverflow`, the treasury is permanently debited while the recipient receives nothing. Additionally, `notify_l2_asset_tracker` commits its state changes before `transfer_from_treasury` runs, so the L2AssetTracker records a deposit that never materialised. There is no recovery path.

---

### Finding Description

`mint_base_token` is called up to three times per L1→L2 transaction: once for the value mint to `from`, once for the operator fee, and once for the refund recipient. Each call first invokes `notify_l2_asset_tracker` (which runs inside its own isolated frame via `should_make_frame: true` and commits on success), then calls `transfer_from_treasury`. [1](#0-0) 

Inside `transfer_from_treasury`, the two `update_account_nominal_token_balance` calls are sequential and unguarded by any rollback frame: [2](#0-1) 

Step 1 (subtract from treasury) uses the `?` operator — if it fails, the function returns early with no partial state. Step 2 (add to recipient) also uses `?` — but if it fails, the treasury deduction from step 1 is **not rolled back**. There is no `start_global_frame` / `finish_global_frame` pair wrapping both operations.

The operator-fee and refund-recipient `mint_base_token` calls in `process_l1_transaction` are made **outside** any rollback frame: [3](#0-2) 

The inner execution frame (opened at line 597 of `execute_l1_transaction_and_notify_result`) has already been committed or rolled back before these calls execute: [4](#0-3) 

When the error propagates to the transaction loop, an `Internal` error causes `finish_global_frame(None)` — committing the partial state rather than rolling it back: [5](#0-4) 

The `notify_l2_asset_tracker` call uses `should_make_frame: true`, so its state changes are committed on success before `transfer_from_treasury` runs: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

If step 1 of `transfer_from_treasury` succeeds (treasury debited) and step 2 fails with `MintingBalanceOverflow` (recipient not credited):

- The treasury (`BASE_TOKEN_HOLDER_ADDRESS`) permanently loses `nominal_token_value` tokens.
- The intended recipient (operator coinbase or refund recipient) receives nothing.
- The L2AssetTracker has already recorded the deposit as successful (its frame was committed).
- There is no retry mechanism, no rescue function, and no rollback — the tokens are permanently destroyed.

This is a direct public funds-loss path: base tokens locked in the treasury on behalf of L1 depositors are destroyed without being delivered. [8](#0-7) 

---

### Likelihood Explanation

The trigger condition is `MintingBalanceOverflow`, which fires when `old_balance.checked_add(diff)` overflows U256: [9](#0-8) 

This requires the recipient address (operator coinbase or `reserved[1]` refund recipient) to hold a balance near `U256::MAX`. For ETH-denominated chains this is practically unreachable. However, ZKsync OS is designed to support arbitrary custom base tokens; a token with a very large total supply or very small denomination could place a recipient's balance near `U256::MAX`. An attacker who controls the refund recipient field (`transaction.reserved[1]`) and can pre-fund that address to near `U256::MAX` can deterministically trigger this path. Likelihood is **Low** but non-zero and attacker-controllable. [10](#0-9) 

---

### Recommendation

Wrap both `update_account_nominal_token_balance` calls in `transfer_from_treasury` inside a single `start_global_frame` / `finish_global_frame` pair so that a failure on the credit step atomically rolls back the debit step. Alternatively, use the existing `transfer_nominal_token_value` primitive (which already performs both operations atomically in `account_cache`): [11](#0-10) 

This atomic transfer primitive already handles the debit-then-credit sequence correctly and should be preferred over two separate `update_account_nominal_token_balance` calls.

---

### Proof of Concept

1. Deploy a custom base-token chain via ZKsync OS.
2. Pre-fund address `R` to `U256::MAX - refund_amount + 1` (one below overflow threshold for the expected refund).
3. Submit an L1→L2 priority transaction with `reserved[1] = R` (refund recipient) and a `gas_limit` / `gas_price` such that `to_refund_recipient = refund_amount`.
4. The main transaction body executes (succeeds or reverts — either path reaches the refund mint).
5. `mint_base_token` is called for the refund: `notify_l2_asset_tracker` succeeds and commits its state.
6. `transfer_from_treasury` step 1 subtracts `refund_amount` from the treasury — succeeds.
7. `transfer_from_treasury` step 2 attempts to add `refund_amount` to `R` — overflows, returns `MintingBalanceOverflow`.
8. Error propagates; `tx_loop` calls `finish_global_frame(None)` (commits partial state).
9. Treasury has lost `refund_amount`; `R` received nothing; L2AssetTracker records the deposit. [12](#0-11)

/// Transfers [value] from the treasury account to address [to].
///
/// Returns `TreasuryTransferFailed` if:
/// - Treasury has insufficient balance
/// - Balance overflow occurs
pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
{
    system_log!(
        system,
        "Transferring {nominal_token_value:?} tokens from treasury to {to:?}\n"
    );

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
            true, // true = subtract from balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Treasury transfer failed: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::TreasuryTransferFailed)
                }
                _ => wrap_error!(e),
            }
        })?;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            to,
            nominal_token_value,
            false, // false = add to balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Error while minting: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::MintingBalanceOverflow)
                }
                _ => wrap_error!(e),
            }
        })?;

    Ok(())
}
