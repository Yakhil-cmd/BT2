### Title
`skip_stateful_validations` bypasses `__validate__` signature check for invoke transactions with nonce=1 when **any** (not only `deploy_account`) transaction exists in the mempool for the account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX feature that allows a `deploy_account` + `invoke` pair to be submitted atomically uses `account_tx_in_pool_or_recent_block` as its guard. That helper returns `true` for **any** transaction in the pool or recent block for the sender address — not exclusively a `deploy_account`. As a result, an attacker can submit an invoke transaction with nonce=1 and an **invalid signature** for any account that (a) has on-chain nonce=0 and (b) already has any pending transaction in the mempool, and the gateway will admit it without running the `__validate__` entry point.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` is the function that decides whether to skip the blockifier's `__validate__` call for an incoming invoke transaction: [1](#0-0) 

The guard condition is:

```
tx is Invoke  AND  tx.nonce() == 1  AND  account_nonce == 0
    AND  account_tx_in_pool_or_recent_block(sender) == true
```

When all four conditions hold, `skip_validate = true` is returned, and `run_validate_entry_point` sets `validate: false`: [2](#0-1) 

With `validate = false`, `StatefulValidator::perform_validations` returns immediately after `perform_pre_validation_stage` without ever calling the `__validate__` entry point: [3](#0-2) 

The comment in `skip_stateful_validations` claims the mempool check is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is **incorrect**. `account_tx_in_pool_or_recent_block` checks: [4](#0-3) 

which in turn checks: [5](#0-4) 

This returns `true` for **any** account that has a transaction in the pool or appears in the mempool's committed/staged state — including accounts that were deployed via the `deploy` syscall (not `deploy_account`) and therefore have on-chain nonce=0 with a legitimate nonce=0 invoke already in the pool.

None of the earlier checks in `run_pre_validation_checks` catch an invalid signature:

- `validate_state_preconditions` only checks resource bounds and nonce range: [6](#0-5) 
- `validate_by_mempool` only checks for duplicate tx hash and nonce ordering: [7](#0-6) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An invoke transaction carrying an invalid (or entirely forged) signature is admitted to the mempool and forwarded to the batcher. The batcher will eventually attempt to execute it; the `__validate__` entry point will then fail, and the transaction will be dropped. However, the gateway's admission invariant — *only cryptographically valid transactions enter the mempool* — is broken. This enables:

- Mempool pollution: an attacker can flood the mempool with signature-invalid transactions for any account that has a pending nonce=0 transaction, consuming mempool capacity and batcher CPU.
- Ordering interference: the invalid nonce=1 transaction occupies the account's nonce=1 slot in the pool, potentially blocking the legitimate nonce=1 transaction from being admitted (duplicate-nonce rejection).

---

### Likelihood Explanation

**Medium.** The trigger requires:
1. A target account with on-chain nonce=0 (deployed via `deploy` syscall, or a genesis-deployed contract that has not yet sent any committed transaction).
2. That account having at least one transaction currently in the mempool pool or staged/committed state.

Both conditions are observable from public mempool state. No privileged access is required; any unprivileged user can submit the malformed invoke.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** exists in the pool for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, not merely the account's presence. Alternatively, the gateway can inspect the type of the pending nonce=0 transaction before deciding to skip validation.

---

### Proof of Concept

```
1. Deploy account A via the `deploy` syscall from another contract.
   → A has class_hash set, on-chain nonce = 0.

2. Account A submits a valid invoke tx (nonce=0, valid signature).
   → Gateway: stateless checks pass, stateful checks pass (signature verified),
     tx admitted to mempool. account_tx_in_pool_or_recent_block(A) = true.

3. Attacker submits invoke tx for A:
     nonce = 1, signature = [0xdead, 0xbeef]  (invalid)

4. Gateway stateful path:
   - get_nonce_from_state(A) → 0          (account_nonce == 0 ✓)
   - validate_state_preconditions:
       validate_resource_bounds → passes (valid gas price)
       validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap → passes
   - validate_by_mempool: no duplicate hash, nonce ≥ 0 → passes
   - skip_stateful_validations:
       tx is Invoke ✓, tx.nonce()==1 ✓, account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(A) = true  ← triggered by step 2
       → returns true (skip_validate = true)
   - run_validate_entry_point: validate = false → __validate__ NOT called

5. Invalid-signature invoke is admitted to the mempool.

6. Batcher executes nonce=0 invoke (succeeds, A's nonce → 1).
   Batcher executes nonce=1 invoke → __validate__ fails (invalid signature),
   transaction is dropped/reverted.
   Nonce=1 slot was occupied; any legitimate nonce=1 tx from A was blocked.
``` [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
