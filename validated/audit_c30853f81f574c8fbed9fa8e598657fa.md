### Title
Missing Zero Address Validation for `reward_address` Causes Permanent Freezing of Unclaimed Yield and Staked Funds - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary
Neither `stake()`, `enter_delegation_pool()`, nor either `change_reward_address()` function validates that `reward_address` is non-zero. A staker or pool member who supplies or later sets a zero `reward_address` will cause every subsequent call to `claim_rewards()` — and, for stakers, `unstake_action()` — to revert when the protocol attempts `checked_transfer` to address `0x0`. This permanently freezes the victim's accrued yield and, in the staker case, also blocks recovery of the principal stake.

---

### Finding Description

**Root cause — `stake()` in `src/staking/staking.cairo`:** [1](#0-0) 

The function validates that `reward_address` is not a registered token address, but performs no `is_non_zero()` guard. A caller may pass `reward_address = 0`.

**Root cause — `change_reward_address()` in `src/staking/staking.cairo`:** [2](#0-1) 

Same omission: only the token-address check is present; zero is accepted.

**Root cause — `enter_delegation_pool()` in `src/pool/pool.cairo`:** [3](#0-2) 

Only `token_address != reward_address` is checked; zero passes freely.

**Root cause — `change_reward_address()` in `src/pool/pool.cairo`:** [4](#0-3) 

Same omission.

**Consequence — `send_rewards_to_staker()` in `src/staking/staking.cairo`:** [5](#0-4) 

When `reward_address` is zero and `amount > 0`, `checked_transfer(recipient: 0x0, ...)` reverts (standard ERC-20 behaviour; the codebase already defines `Erc20Error::TRANSFER_FAILED` for this path). This causes both `claim_rewards()` and `unstake_action()` to revert. [6](#0-5) [7](#0-6) 

**Consequence — `claim_rewards()` in `src/pool/pool.cairo`:** [8](#0-7) 

Line 366 performs `reward_token.checked_transfer(recipient: reward_address, ...)`. With `reward_address = 0` and `rewards > 0`, this reverts, permanently blocking reward withdrawal for that pool member.

The protocol already defines `GenericError::ZERO_ADDRESS` and uses it in `add_token()`, confirming the intent to guard against zero addresses: [9](#0-8) [10](#0-9) 

---

### Impact Explanation

**For a staker with `reward_address = 0`:**
- `claim_rewards()` reverts every time rewards are non-zero → unclaimed yield is frozen.
- `unstake_action()` calls `send_rewards_to_staker()` before returning principal; it reverts → the staker's entire staked principal is also frozen until the reward address is corrected.
- Because `unstake_action()` is permissionless, an adversary can call it repeatedly to grief the staker while the staker races to call `change_reward_address()`.

**For a pool member with `reward_address = 0`:**
- `claim_rewards()` reverts whenever accrued rewards are non-zero → unclaimed yield is frozen.

Impact classification: **High — Permanent/temporary freezing of unclaimed yield; temporary freezing of staked funds.**

---

### Likelihood Explanation

Any unprivileged staker or delegator can trigger this by passing `reward_address = 0` to `stake()` or `enter_delegation_pool()`, or by calling `change_reward_address(0)` at any later time. No privileged access, external dependency, or bridge compromise is required. Accidental zero-address submission is a realistic user error; the protocol's own `GenericError::ZERO_ADDRESS` constant signals awareness of this class of mistake elsewhere in the codebase.

---

### Recommendation

Add a non-zero guard in all four entry points:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply this in:
- `stake()` — `src/staking/staking.cairo`
- `change_reward_address()` — `src/staking/staking.cairo`
- `enter_delegation_pool()` — `src/pool/pool.cairo`
- `change_reward_address()` — `src/pool/pool.cairo`

---

### Proof of Concept

1. Deploy the system normally.
2. Call `stake(reward_address: 0, operational_address: <valid>, amount: <min_stake>)` — succeeds; no revert.
3. Advance K epochs so rewards accrue.
4. Call `claim_rewards(staker_address)` — reverts with `TRANSFER_FAILED` because `checked_transfer(recipient: 0x0, amount: <rewards>)` fails.
5. Call `unstake_intent()` from the staker, advance past the exit window.
6. Call `unstake_action(staker_address)` — reverts for the same reason; the staker's principal is now inaccessible until `change_reward_address` is called with a valid address.

The same sequence applies to a pool member via `enter_delegation_pool(reward_address: 0, ...)` followed by `claim_rewards(pool_member)`.

            // Prerequisites and asserts.
            self.general_prerequisites();

            let unstake_time = staker_info

            assert!(Time::now() >= unstake_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Send rewards to staker's reward address.
            // It must be part of this function's flow because staker_info is about to be erased.
            let token_dispatcher = strk_token_dispatcher();
            self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
            // Update staker info to storage (it will be erased later).
            // This is done here to avoid re-entrancy.
            self.write_staker_info(:staker_address, :staker_info);

            let staker_amount = self.get_own_balance(:staker_address).to_strk_native_amount();
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            self.remove_staker(:staker_address, :staker_info, :staker_pool_info);

            // Return stake to staker.
            token_dispatcher
                .checked_transfer(recipient: staker_address, amount: staker_amount.into());
            // Return delegated stake to pools and zero their balances.
            self

                    :staker_address, staker_pool_info: staker_pool_info.as_non_mut(),
                );
            // Clear staker pools.
            staker_pool_info.pools.clear();
            staker_amount
        }

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
