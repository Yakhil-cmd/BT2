### Title
Switching ETH Connector Account Without State Migration Freezes All User NEP-141 ETH Balances - (File: `engine/src/contract_methods/connector.rs`)

### Summary
`set_eth_connector_contract_account` allows the owner to atomically redirect the engine's external ETH connector pointer to a new account. Because no state is migrated, every subsequent connector-facing call (`ft_balance_of`, `ft_transfer`, `ft_total_supply`, `storage_balance_of`, `storage_deposit`, `storage_withdraw`, `withdraw`) is forwarded to the new connector, which holds zero balances for all users. All NEP-141 ETH that users deposited into the old connector becomes inaccessible through the engine.

### Finding Description

`set_eth_connector_contract_account` (engine/src/contract_methods/connector.rs:418–438) is the direct analog of `Controller.switchLpTokenStaker()` in the reference report. It performs two writes and nothing else:

```rust
set_connector_account_id(io, &args.account);
set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
``` [1](#0-0) 

`set_connector_account_id` overwrites the single storage key `EthConnectorStorageId::EthConnectorAccount` with the new account ID: [2](#0-1) 

Every connector-facing method reads that same key via `get_connector_account_id` and forwards the call as a cross-contract promise to whatever account is stored there:

```rust
pub fn ft_balance_of<I: IO + Copy + PromiseHandler + Env>(io: I) -> Result<(), ContractError> {
    let args = io.read_input().to_vec();
    return_promise(io, &io, "ft_balance_of", args, ZERO_YOCTO)
}
``` [3](#0-2) 

The same pattern applies to `ft_total_eth_supply_on_near`, `storage_balance_of`, `storage_deposit`, `storage_withdraw`, and `withdraw`: [4](#0-3) 

The moment `set_eth_connector_contract_account` completes, the engine's connector pointer is updated. The new connector contract has no record of any user's NEP-141 ETH balance. Every call that previously returned a real balance now returns zero or reverts. The old connector still holds the actual ETH, but the engine no longer routes to it.

This is structurally identical to the reference vulnerability: a stored contract-address pointer is replaced, the new contract returns zero for all user-state queries, and the accumulated user state (here: deposited ETH balances rather than reward integrals) is silently made inaccessible.

The exposed entry point is the public WASM export: [5](#0-4) 

### Impact Explanation

All users who bridged ETH into Aurora (minting NEP-141 ETH in the old connector) lose access to those balances through the engine. Concretely:

- `ft_balance_of` returns 0 for every account.
- `ft_transfer` reverts (insufficient balance on the new connector).
- `withdraw` reverts (new connector has no NEP-141 ETH to burn and release underlying ETH).

The ETH is not destroyed — it remains locked in the old connector — but it is unreachable through the engine's interface. If the old connector is decommissioned or its access keys are removed, the freeze becomes permanent. Even if the owner reverts the pointer, any `withdraw` calls that executed between the two pointer changes may have burned EVM ETH without successfully releasing the underlying ETH, resulting in direct fund loss for those users.

**Impact class: High (temporary fund freeze) to Critical (permanent fund freeze / direct fund loss for in-flight withdrawals).**

### Likelihood Explanation

The function is owner-only, so the trigger is a legitimate administrative action — upgrading or replacing the connector contract — not a malicious one. The Conic Finance reference report describes the same class: a routine governance action (`switchLpTokenStaker`) that silently breaks accounting. An Aurora owner performing a connector upgrade would naturally call `set_eth_connector_contract_account` without realizing that no migration of NEP-141 balances occurs, because the function provides no warning and performs no migration. The likelihood of this occurring during a connector upgrade is realistic.

### Recommendation

Before updating the connector pointer, the engine should either:
1. Require that the new connector already holds a total supply equal to the old connector's total supply (verified via a cross-contract view call), or
2. Pause all connector-facing operations during the migration window and provide an explicit migration path that transfers NEP-141 balances from the old connector to the new one before the pointer is switched.

At minimum, the function should emit a log or require an explicit acknowledgement flag confirming the caller understands that no balance migration is performed.

### Proof of Concept

1. User calls `ft_on_transfer` (or `deposit`) to bridge 10 ETH into Aurora. The old connector mints 10 NEP-141 ETH for the user; the engine credits 10 EVM ETH to the user's address.
2. Owner calls `set_eth_connector_contract_account` with a freshly deployed, empty connector contract (e.g., as part of a connector upgrade).
3. User calls `ft_balance_of` on the engine → the engine forwards to the new connector → new connector returns `0`.
4. User calls `withdraw` on the engine → the engine attempts to burn NEP-141 ETH on the new connector → the new connector has no balance → the call reverts. The user's EVM ETH may or may not be refunded depending on the exact execution path.
5. The user's 10 NEP-141 ETH remains locked in the old connector with no engine-accessible path to retrieve it.
