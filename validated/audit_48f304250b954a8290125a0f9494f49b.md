### Title
Unchecked Precompile Return Value After Committed `_burn` Enables Permanent Token Destruction — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2::withdrawToNear` burns the caller's tokens unconditionally before invoking the `ExitToNear` precompile, and never checks the assembly `call` return value. Supplying any recipient bytes that fail NEAR `AccountId` validation causes the precompile to return `ExitError`, the EVM `call` opcode to return `0`, and the already-committed `_burn` to be unrecoverable — permanently destroying the caller's tokens with no corresponding NEAR-side transfer.

---

### Finding Description

**Step 1 — Burn before precompile call, return value ignored.**

In `EvmErc20V2.sol`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // ← tokens destroyed here, unconditionally

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no require(res != 0)
    }
}
``` [1](#0-0) 

**Step 2 — Precompile fails on invalid account ID.**

`ExitToNear::run` calls `ExitToNearParams::try_from(input)?` at the very first parsing step. For flag `0x1` (ERC-20 path), this calls `parse_recipient(&input[32..])`:

```rust

    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    // ...
    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        // ...
    })
}
``` [2](#0-1) 

`AccountId::validate` rejects any string containing uppercase letters, starting with a separator, or exceeding 64 characters:

```rust
for c in account_id.bytes() {
    let current_char_is_separator = match c {
        b'a'..=b'z' | b'0'..=b'9' => false,
        b'-' | b'_' | b'.' => true,
        _ => { return Err(ParseAccountError::Invalid); }  // uppercase hits here
    };
    if current_char_is_separator && last_char_is_separator {
        return Err(ParseAccountError::Invalid);            // leading hyphen hits here
    }
``` [3](#0-2) 

**Step 3 — `ExitError` propagates, EVM `call` returns 0, no revert.**

`ExitToNear::run` returns `Err(ExitError)` at line 419 before any promise is created: [4](#0-3) 

In the EVM, a failed `call` opcode returns `0` in `res` and execution of the calling frame continues. Because `EvmErc20V2` never checks `res`, the function returns normally. The `_burn` that already executed is not rolled back.

**Step 4 — `error_refund` feature does NOT protect this path.**

The refund callback is constructed only after `ExitToNearParams::try_from` succeeds (lines 449–483). Since the failure occurs at line 419, the callback is never created and no refund is possible regardless of whether `error_refund` is compiled in. [5](#0-4) 

---

### Impact Explanation

Any token holder can permanently destroy their own tokens (or be tricked into doing so) by calling `withdrawToNear` with a syntactically valid UTF-8 byte string that is not a valid NEAR account ID. The ERC-20 balance is reduced to zero; no NEP-141 tokens are minted on NEAR; no refund occurs. The funds are permanently frozen/destroyed. This matches the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The path requires no privilege, no admin key, and no external dependency. Any token holder can trigger it by accident (e.g., passing an uppercase account ID like `"Alice.near"`) or deliberately. The `recipient` parameter is a raw `bytes` argument with no on-chain format validation before the burn.

---

### Recommendation

Check the precompile call return value and revert on failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, validate the recipient format in Solidity before calling `_burn`, or restructure the function to call the precompile first and only burn on success. The same unchecked-return-value pattern exists in `EvmErc20.sol::withdrawToNear` and both contracts' `withdrawToEthereum` functions and should be fixed consistently. [6](#0-5)
