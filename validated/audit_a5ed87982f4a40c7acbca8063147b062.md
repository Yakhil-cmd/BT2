### Title
Malicious participant aborts presign for all honest parties via type-confused message at wait1 — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

### Summary
A single malicious participant can permanently prevent presign completion for all honest parties by broadcasting a single `Scalar` at `wait1` instead of the expected `(Scalar, Scalar)` pair. The deserialization failure propagates unconditionally via `?` through `recv_from_others` and `do_presign`, aborting every honest party's protocol instance.

### Finding Description
In `do_presign` (Round 2), each honest participant calls:

```rust
recv_from_others::<(Scalar, Scalar)>(&chan, wait1, &participants, me).await?
```

`recv_from_others` loops calling `chan.recv(waitpoint).await?`. `Comms::recv` deserializes the raw bytes via `rmp_serde::decode::from_slice`. If a malicious participant broadcasts a single `Scalar` (valid MessagePack, valid channel header, correct waitpoint) instead of a `(Scalar, Scalar)` tuple, deserialization fails and `Comms::recv` returns `Err(ProtocolError::DeserializationError(...))`. The `?` in `recv_from_others` (line 20 of `helpers.rs`) immediately propagates this error, which then propagates through the `?` at line 152 of `presign.rs`, aborting the presign future. Since the malicious message is broadcast (`send_many`), every honest participant receives it and aborts identically.

### Impact Explanation
Every honest participant's presign instance terminates with `ProtocolError::DeserializationError`. The consumed triples cannot be reused. The malicious participant can repeat this on every retry, permanently denying presign (and therefore signing) capability to all honest parties. Impact: **High — permanent denial of signing**.

### Likelihood Explanation
Any participant in the protocol can trivially craft and send a single-scalar message with the correct channel header and waitpoint. No cryptographic capability is required. The attack is repeatable at negligible cost.

### Recommendation
In `recv_from_others`, catch deserialization errors per-message and skip (or attribute) the offending sender rather than propagating the error immediately. Specifically, change the `?` on `chan.recv(waitpoint).await?` to handle `Err(ProtocolError::DeserializationError(_))` by logging/attributing the sender and continuing the loop. This is consistent with how `echo_broadcast.rs` handles recv errors (it uses `_ => continue` on recv failure). [4](#0-3) 

### Proof of Concept
1. Set up 3 participants completing Round 1 of `do_presign` honestly.
2. One malicious participant, instead of sending `(alpha_j, beta_j): (Scalar, Scalar)` at `wait1`, sends a single `Scalar` with the correct channel header and waitpoint.
3. Each honest participant's `recv_from_others::<(Scalar, Scalar)>` dequeues the malicious message, `rmp_serde` fails to decode a single scalar as a 2-tuple, `Comms::recv` returns `Err`, `?` propagates it, and `do_presign` returns `Err(ProtocolError::DeserializationError(...))`.
4. Assert: all honest protocol instances return an error. No presign output is produced.
5. Repeat with fresh triples — the malicious participant repeats the attack, confirming permanence.
