### Title
Gateway Skips `__validate__` Signature Check for Invoke(nonce=1) Transactions, Allowing Admission of Unauthorized Transactions and Griefing of Deploy+Invoke Pairs — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point execution (the only mechanism that verifies a transaction's signature) for any `Invoke` transaction with `nonce=1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because the mempool's `validate_tx` path checks only nonce ordering and not signatures, an attacker can submit an `Invoke(nonce=1)` with an arbitrary/invalid signature for any address that has a pending `DeployAccount` in the pool. The gateway admits it. Combined with fee-escalation replacement, the attacker can evict a legitimate user's valid `Invoke(nonce=1)` from the mempool and replace it with an invalid one, permanently dropping the user's transaction.

### Finding Description

**Step 1 – The skip condition.**

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three hold:

```
tx is Invoke  AND  tx.nonce() == 1  AND  account_nonce == 0
AND  account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

**Step 2 – `account_tx_in_pool_or_recent_block` is too broad.**

The check returns `true` if the address appears in the mempool pool *or* in the `staged`/`committed` maps — it does not verify that the matching transaction is specifically a `DeployAccount`. [2](#0-1) [3](#0-2) 

**Step 3 – When `skip_validate=true`, `__validate__` is never called at the gateway.**

`run_validate_entry_point` sets `validate: !skip_validate`, so the blockifier's `validate_tx` returns `Ok(None)` immediately without executing the account's signature-checking code. [4](#0-3) [5](#0-4) 

**Step 4 – No other path verifies the signature.**

`validate_by_mempool` calls `mempool.validate_tx`, which only checks nonce ordering, not the signature. [6](#0-5) [7](#0-6) 

**Step 5 – Fee-escalation replacement.**

The mempool allows a transaction at `(address, nonce)` to be replaced if the new transaction's fee is ≥ `fee_escalation_percentage` (default 10%) higher. No signature check is performed during replacement.

**Attack sequence:**

1. Victim submits `DeployAccount(A)` + `Invoke(A, nonce=1, valid_sig, fee=F)`.
2. Attacker observes `DeployAccount(A)` in the pool → `account_tx_in_pool_or_recent_block(A) = true`.
3. Attacker submits `Invoke(A, nonce=1, invalid_sig, fee=F*1.1)`.
4. Gateway: `account_nonce=0`, `tx.nonce()=1`, pool check passes → `skip_validate=true` → `__validate__` skipped → transaction admitted.
5. Mempool: fee escalation replaces victim's valid invoke with attacker's invalid one.
6. Batcher executes `DeployAccount(A)` (succeeds), then `Invoke(A, nonce=1, invalid_sig)` → `__validate__` runs in blockifier → fails/reverts.
7. Victim's original invoke is permanently gone from the mempool; victim must resubmit.

### Impact Explanation

The gateway admits an `Invoke` transaction whose signature has never been verified. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."* Combined with fee-escalation replacement, an unprivileged attacker can systematically drop any victim's `Invoke(nonce=1)` that accompanies a `DeployAccount`, forcing repeated resubmission and degrading the deploy+invoke UX feature to a griefing vector.

### Likelihood Explanation

The `deploy_account + invoke` UX pattern is explicitly supported and documented in the codebase and integration tests. [8](#0-7) 

Any observer of the mempool (or the gateway's public API) can detect a pending `DeployAccount` and immediately submit a competing `Invoke(nonce=1)` with a higher fee and an invalid signature. No privileged access is required.

### Recommendation

1. **Verify the signature even when skipping `__validate__`**: perform a lightweight ECDSA/Stark-curve check on the transaction hash against the declared public key before admitting the transaction, independent of the full `__validate__` entry-point execution.
2. **Narrow the `account_tx_in_pool_or_recent_block` check**: instead of checking for any transaction from the address, verify that a `DeployAccount` transaction specifically is present in the pool for that address.
3. **Alternatively, record the deploy_account tx hash** at admission time and bind the skip-validate permission to that specific hash, so that if the `DeployAccount` is evicted, the paired `Invoke` is also evicted.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

/// The first invoke_tx can be inserted to the first block right after the deploy_tx due to
/// the skip_validate feature. This feature allows the gateway to accept this transaction although
/// the account does not exist yet.
pub fn create_deploy_account_tx_and_invoke_tx(
    tx_generator: &mut MultiAccountTransactionGenerator,
    account_id: AccountId,

    let undeployed_account_tx_generator = tx_generator.account_with_id_mut(account_id);
    assert!(!undeployed_account_tx_generator.is_deployed());

    vec![deploy_tx, invoke_tx]
