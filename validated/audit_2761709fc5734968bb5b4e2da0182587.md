### Title
`execute_scheduled()` Permanently Locks Any Attached NEAR Sent by Callers - (File: `etc/xcc-router/src/lib.rs`)

### Summary

The XCC router contract's `execute_scheduled` function is marked `#[payable]` and is intentionally open to any caller, but it never reads or uses `env::attached_deposit()`. Any NEAR tokens attached to a call of this function are permanently locked in the router sub-account with no recovery mechanism.

### Finding Description

In `etc/xcc-router/src/lib.rs`, `execute_scheduled` is decorated with `#[payable]`, which in NEAR SDK means the function accepts an attached NEAR deposit without panicking: [1](#0-0) 

The function body removes a stored `PromiseArgs` from the `scheduled_promises` map and executes it. The promise's `attached_balance` field was fixed at schedule time (inside the `PromiseCreateArgs` struct), and is entirely independent of `env::attached_deposit()` at execution time. A `grep` for `env::attached_deposit` or `env::attached` across the entire file returns **zero matches**, confirming the attached deposit is never read, forwarded, or returned.

The only fund-recovery function in the router is `send_refund`, which transfers a hardcoded constant `REFUND_AMOUNT` (2 NEAR) to the parent (Aurora Engine): [2](#0-1) 

This is a fixed-amount refund for storage staking purposes and cannot recover arbitrary NEAR amounts sent by callers of `execute_scheduled`.

The `deploy_upgrade` function is also `#[payable]` and similarly never reads `env::attached_deposit()`: [3](#0-2) 

However, `deploy_upgrade` is restricted to the parent via `assert_preconditions`, so it is less reachable by unprivileged users. The primary issue is `execute_scheduled`.

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

Any NEAR attached to a call to `execute_scheduled` is irrecoverably locked in the router sub-account (e.g., `{evm_address}.aurora`). The router contract exposes no general withdrawal or sweep function. The `send_refund` path only sends a fixed 2 NEAR to the Aurora Engine parent, not to the original caller. The locked NEAR cannot be retrieved by the sender.

### Likelihood Explanation

**Medium.** The comment in the source code explicitly states that `execute_scheduled` is open to any caller by design: [4](#0-3) 

A user executing a scheduled promise that itself requires NEAR (e.g., a `near_deposit` or `ft_transfer` call with an attached balance) may reasonably assume they need to attach NEAR to `execute_scheduled` to fund the outgoing call. In reality, the outgoing promise's `attached_balance` was fixed at schedule time and is funded separately via the wNEAR unwrapping flow in the engine. The mismatch between the `#[payable]` annotation and the actual behavior creates a realistic user-error trap. Integration scripts or wallets that forward NEAR to this function would also trigger the loss.

### Recommendation

1. **Remove `#[payable]`** from `execute_scheduled` if no attached deposit is ever needed. The NEAR SDK will then panic if any deposit is attached, preventing accidental loss.
2. If there is a legitimate future use case for attaching NEAR to `execute_scheduled`, add an explicit assertion: `assert_eq!(env::attached_deposit(), NearToken::from_yoctonear(0), "ERR_UNEXPECTED_DEPOSIT");` until that use case is implemented.
3. Apply the same fix to `deploy_upgrade` if the attached NEAR is always intended to remain in the router's balance for storage staking (in which case `#[payable]` is correct but should be documented clearly).

### Proof of Concept

1. Alice's EVM address has a scheduled promise stored at nonce `0` in her router sub-account (`{alice_addr}.aurora`).
2. Alice (or any third party) calls `execute_scheduled` with `nonce: 0` and attaches 1 NEAR, believing it is needed to fund the outgoing cross-contract call.
3. `execute_scheduled` removes the promise from storage and calls `Self::promise_create(promise)`, which uses `promise.attached_balance` (set at schedule time, e.g., 1 yoctoNEAR) â€” not `env::attached_deposit()`.
4. The 1 NEAR attached to the call remains in the router sub-account's balance.
5. There is no function in the router to return this NEAR to Alice. `send_refund` only sends a fixed 2 NEAR to the Aurora Engine parent, not to Alice.
6. The 1 NEAR is permanently frozen.

```rust
    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }
```

```rust
    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
```
