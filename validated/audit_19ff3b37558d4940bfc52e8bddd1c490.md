### Title
Missing Threshold Validation in CKD Protocol Allows Malicious Coordinator to Corrupt Derived Key Output — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts any participant list with `>= 2` members but never validates that the participant count meets the DKG reconstruction threshold. Because the Lagrange coefficients used to normalize each participant's BLS signature share are computed over the caller-supplied participant set, invoking `ckd()` with fewer participants than the DKG threshold causes every share to be scaled by a wrong coefficient. The coordinator aggregates these malformed shares and produces a `CKDOutput` whose embedded `C` value does not equal `msk · H(pk, app_id) + a·Y`. The app therefore cannot recover the correct confidential key, and the output is silently accepted as valid by all honest parties.

### Finding Description

The `ckd()` entry point performs four checks before launching the protocol: [1](#0-0) 

It verifies `participants.len() >= 2`, absence of duplicates, presence of `me`, and presence of `coordinator`. It does **not** verify that `participants.len() >= threshold` (the reconstruction lower bound used during DKG).

Inside `compute_signature_share`, each participant computes its Lagrange coefficient over the supplied participant list: [2](#0-1) 

The Lagrange coefficient `λ_i` is evaluated at zero over the set `{P_1, …, P_k}` passed to `ckd()`. For the coordinator's aggregation to equal `msk · H(pk, app_id)`, the identity `∑ λ_i · x_i = msk` must hold, which requires Shamir reconstruction over at least `threshold` shares. If `k < threshold`, the polynomial cannot be reconstructed and the sum diverges from `msk`.

The coordinator aggregates without any correctness check: [3](#0-2) 

The resulting `CKDOutput` is returned to the app, which calls `unmask(app_sk)`: [4](#0-3) 

`unmask` performs no BLS signature verification. The app receives a structurally valid but cryptographically wrong `(Y, C)` pair and silently accepts it.

The `KeygenOutput` type passed to `ckd()` carries only `private_share` and `public_key`; the threshold is not stored in it and is not passed as a parameter, so the function has no way to enforce the invariant even if it tried. [5](#0-4) 

### Impact Explanation

A malicious coordinator instructs all honest participants to run `ckd()` with a participant list of size `k < threshold`. Each honest participant computes `λ_i` over the undersized set and sends a malformed share to the coordinator. The coordinator aggregates these shares into a `CKDOutput` where `C ≠ msk · H(pk, app_id) + a·Y`. The app unmasks it and obtains a value that is not the intended BLS signature over `app_id`. If the app verifies the BLS signature it detects the failure (permanent denial of CKD); if it does not verify, it silently uses a wrong key (corruption of CKD output). Either outcome maps to: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

The coordinator role is a documented participant in the CKD protocol. A single compromised or malicious MPC node that acts as coordinator can trigger this with a single API call, passing any participant list of size 2 (the only enforced lower bound). No cryptographic capability is required. The `ckd()` function is a public API with no external guard against undersized participant sets.

### Recommendation

Pass the DKG `ReconstructionLowerBound` (threshold) as an explicit parameter to `ckd()` and add a guard analogous to those in `assert_key_invariants` and `validate_triple_inputs`:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,  // add this
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < threshold.value() {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            participants: participants.len(),
            threshold: threshold.value(),
        });
    }
    // existing checks …
}
```

This mirrors the pattern used in `validate_triple_inputs`: [6](#0-5) 

and in `assert_key_invariants`: [7](#0-6) 

### Proof of Concept

```
Setup:
  N = 5 participants {P1, P2, P3, P4, P5}, DKG threshold t = 3
  Each P_i holds share x_i = f(i) for degree-2 polynomial f with f(0) = msk

Attack:
  Malicious coordinator instructs all participants to call ckd() with
  participants = {P1, P2}  (k = 2 < t = 3)

Per-participant computation (honest, but over wrong set):
  λ_1 = P2_scalar / (P2_scalar - P1_scalar)   [Lagrange over {P1,P2} at 0]
  λ_2 = P1_scalar / (P1_scalar - P2_scalar)
  norm_big_c_i = (x_i · H(pk,app_id) + y_i · A) · λ_i

Coordinator aggregation:
  C = λ_1·x_1·H + λ_2·x_2·H + (λ_1·y_1 + λ_2·y_2)·A
    = (λ_1·x_1 + λ_2·x_2)·H + y'·A

  But λ_1·x_1 + λ_2·x_2 ≠ msk  (degree-2 polynomial cannot be
  reconstructed from 2 points; result is the evaluation of the
  degree-1 interpolant through (P1,x_1),(P2,x_2) at 0, not f(0))

App receives wrong (Y, C):
  confidential_key = C - a·Y
                   ≠ msk · H(pk, app_id)
  BLS verification fails (or wrong key is silently used if skipped)
```

The root cause — no threshold guard in `ckd()` — is directly analogous to the H-04 pattern: a valid configuration of inputs (participant count = 2, which passes the `>= 2` check) produces an output that violates the protocol's correctness invariant, and the violation is not caught at the point of output acceptance.     me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,

    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });

    // ensure the coordinator is a participant

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
