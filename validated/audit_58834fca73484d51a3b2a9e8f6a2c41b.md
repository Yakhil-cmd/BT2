### Title
Deserialization-Before-Sender-Check in `do_sign_participant` Allows Any Non-Coordinator to Abort Honest Participant's Signing Session — (`src/frost/redjubjub/sign.rs`)

### Summary

In `do_sign_participant`, the participant loop at `wait_round_1` calls `chan.recv(wait_round_1).await?` and only checks `from != coordinator` **after** successful deserialization. `Comms::recv` deserializes the message body before returning the sender identity. A deserialization failure propagates via `?` and exits the function with an error before the sender check is ever evaluated. Any malicious non-coordinator participant can inject a single message with a valid header but invalid body to permanently abort the honest participant's signing session.

### Finding Description

In `src/frost/redjubjub/sign.rs`, `do_sign_participant` waits for the coordinator's `Randomizer` using this loop:

```rust
let randomizer = loop {
    let (from, randomizer): (_, Randomizer) = chan.recv(wait_round_1).await?;
    if from != coordinator {
        continue;
    }
    break randomizer;
};
``` [1](#0-0) 

`chan.recv` resolves to `Comms::recv` in `src/protocol/internal.rs`:

```rust
async fn recv<T: DeserializeOwned>(
    &self,
    header: MessageHeader,
) -> Result<(Participant, T), ProtocolError> {
    let (from, data) = self.incoming.pop(header).await;
    let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| { ... })?;
    let decoded: Result<T, ...> =
        rmp_serde::decode::from_slice(message_data).map_err(...);
    Ok((from, decoded?))   // <-- deserialization error propagates here
}
``` [2](#0-1) 

`decoded?` on line 340 propagates a `ProtocolError::DeserializationError` if the body cannot be decoded as `Randomizer`. This error is returned from `Comms::recv` before the `(from, ...)` tuple is ever constructed. Back in the signing loop, the `?` on line 218 propagates this error immediately, exiting `do_sign_participant` with a fatal error. The `from != coordinator` guard on line 219 is never reached.

The `MessageBuffer` stores messages keyed by `MessageHeader` (channel tag + waitpoint). The header for `wait_round_1` is deterministic — it is the first waitpoint allocated in the signing protocol. A malicious participant can craft a message with the correct header prefix but a garbage body, deliver it via the `Protocol::message` API before the coordinator's message arrives, and trigger the abort.

Note: `push_message` silently drops messages with a malformed or missing header, but it does **not** validate the body: [3](#0-2) 

The body validation only happens at deserialization time inside `recv`, after the message has already been queued and popped.

### Impact Explanation

A single malicious non-coordinator participant can permanently abort any honest participant's signing session by sending one message with a valid header and invalid body at `wait_round_1`. The honest participant's protocol future returns `Err(ProtocolError::DeserializationError(...))` and cannot be resumed. This is a **permanent denial of signing** for the targeted honest party under valid protocol inputs.

The same structural flaw exists in `src/frost/eddsa/sign.rs` `do_sign_participant_v1` at the `r2_wait_point` loop. [4](#0-3) 

### Likelihood Explanation

Any participant in the signing session can trigger this. No special privilege is required beyond being a protocol participant. The attack requires sending one crafted message at the correct waitpoint, which is deterministic and predictable. The attack is trivially reproducible.

### Recommendation

Handle deserialization errors gracefully inside the loop instead of propagating them. The existing `echo_broadcast.rs` already demonstrates the correct pattern:

```rust
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,  // skip malformed messages
};
``` [5](#0-4) 

Apply the same pattern to the `wait_round_1` loop in `do_sign_participant`:

```rust
let randomizer = loop {
    let Ok((from, randomizer)): Result<(_, Randomizer), _> =
        chan.recv(wait_round_1).await else { continue };
    if from != coordinator {
        continue;
    }
    break randomizer;
};
```

This ensures a deserialization failure from any non-coordinator message is silently skipped rather than aborting the session.

### Proof of Concept

1. Set up a signing session with participants `[coordinator, honest, malicious]`.
2. Before the coordinator sends its `Randomizer` at `wait_round_1`, have `malicious` call `Protocol::message` on `honest`'s protocol instance with a message whose header matches `wait_round_1` but whose body is random bytes (e.g., `[0xFF; 32]`).
3. Observe that `honest`'s next `poke()` call returns `Err(ProtocolError::DeserializationError(...))`.
4. The coordinator's subsequent legitimate `Randomizer` message is never processed; the signing session is permanently aborted for `honest`.
