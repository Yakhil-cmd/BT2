### Title
EVM Pause Guard Blocks User Withdrawals During Emergency — (`fvm/evm/stdlib/contract.cdc`)

### Summary
The `CadenceOwnedAccount.withdraw`, `withdrawNFT`, and `withdrawTokens` functions in the EVM Cadence contract each include a `!EVM.isPaused()` pre-condition. When the Governance Committee pauses EVM — precisely the scenario triggered by exploits, vulnerabilities, or protocol emergencies — every COA owner is simultaneously locked out of their only exit route back to Cadence. This directly mirrors the BellumNursery pattern: a function that must remain available during an emergency is incorrectly gated by the emergency-state guard itself.

### Finding Description
`EVM.isPaused()` reads a boolean from `/storage/evmOperationsPaused` and returns `true` when the Governance Committee has activated the pause. The contract comment explicitly states the pause is for "maintenance or any situation that requires special governance handling" and that "the EVM enters a read-only mode." [1](#0-0) 

The `withdraw` function — the sole mechanism for a COA owner to move FLOW tokens from EVM back to Cadence — enforces this guard as a hard pre-condition: [2](#0-1) 

The same guard is applied to `withdrawNFT` and `withdrawTokens`, blocking all asset-exit paths: [3](#0-2) 

The contract itself demonstrates the correct design intent for read-only operations: `dryRun` and `dryCallWithSigAndArgs` are explicitly **exempt** from the `isPaused()` guard with the comment "dry-run functions are intentionally exempt from the isPaused() guard — they perform no state mutations and remain available in read-only mode." [4](#0-3) 

A `withdraw` call moves tokens **out** of EVM state into a Cadence `FlowToken.Vault` — it reduces EVM state rather than introducing new risk. There is no technical reason to block it during a pause; the pause is intended to prevent new EVM state mutations (calls, deploys, deposits), not to prevent users from reclaiming their own assets.

The test suite confirms this behavior is enforced and expected, showing `withdraw` fails with "EVM operations are temporarily paused" when EVM is paused: [5](#0-4) 

### Impact Explanation
Any unprivileged Cadence transaction sender who owns a `CadenceOwnedAccount` with a FLOW balance cannot retrieve their tokens while EVM is paused. The pause is activated precisely during emergencies — EVM exploits, bridge vulnerabilities, or protocol-level incidents — which are the exact conditions under which users most urgently need to exit. All COA holders are affected simultaneously, with no alternative withdrawal path available. Asset availability is denied for the full duration of the pause.

### Likelihood Explanation
The EVM pause requires a Governance Committee multi-sig Cadence transaction, so it is not a routine event. However, the pause mechanism exists specifically for emergency scenarios. When triggered, the impact is protocol-wide and immediate: every COA holder loses access to their funds at the moment the emergency is declared. The likelihood of the pause being activated at least once during the protocol's lifetime is non-trivial given it is a designed safety mechanism.

### Recommendation
Remove the `!EVM.isPaused()` pre-condition from `withdraw`, `withdrawNFT`, and `withdrawTokens`. These functions reduce EVM state by moving assets back to Cadence and do not introduce the risks the pause is designed to contain. Only state-increasing or externally-triggered operations (`run`, `batchRun`, `call`, `deploy`, `deposit`, `depositNFT`, `depositTokens`) should be blocked during a pause. The corrected signature:

```cadence
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault {
    // Remove: pre { !EVM.isPaused(): "EVM operations are temporarily paused" }
    ...
}
```

### Proof of Concept

1. User deposits FLOW tokens into a `CadenceOwnedAccount` via `coa.deposit(from: <-vault)`.
2. An EVM exploit is discovered; the Governance Committee activates the pause via a multi-sig transaction writing `true` to `/storage/evmOperationsPaused`.
3. User submits a Cadence transaction:
   ```cadence
   let coa = account.storage.borrow<auth(EVM.Withdraw) &EVM.CadenceOwnedAccount>(
       from: /storage/coa
   )!
   let vault <- coa.withdraw(balance: bal)
   ```
4. The pre-condition `!EVM.isPaused()` evaluates to `false`; the transaction aborts with `"EVM operations are temporarily paused"`.
5. The user's FLOW tokens remain locked in EVM for the entire duration of the pause, with no alternative exit path. [6](#0-5)
