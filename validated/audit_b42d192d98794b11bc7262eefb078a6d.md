### Title
`uint32` Overflow in `_afterTimelock` Allows Pool Admin to Bypass Stop-Loss Timelocks Immediately — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` casts `block.timestamp + timelock` to `uint32` without an overflow guard, and the `timelock` field is never capped at initialization or update time. When `timelock` is large enough that the sum exceeds `type(uint32).max`, the result wraps to a small past timestamp. `_requireElapsed` then passes immediately, letting the pool admin execute any timelocked parameter change (drawdown, decay, watermarks) in the same transaction as the proposal — completely defeating the LP-protection timelock.

---

### Finding Description

`_afterTimelock` computes the deadline for every pending change:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256` (~1.753 × 10⁹ on mainnet today). `timelock` is `uint32` (max 4,294,967,295 ≈ 136 years). Their sum is computed in `uint256` — no overflow there — but the result is then **truncated to `uint32`**. If `block.timestamp + timelock > 4,294,967,295`, the high bits are silently discarded. [1](#0-0) 

The guard that enforces the delay:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

compares `uint256 block.timestamp` against the `uint32 executeAfter` (zero-extended to `uint256`). If `executeAfter` wrapped to a small past value, `block.timestamp >= executeAfter` is trivially true and the revert never fires. [2](#0-1) 

`_afterTimelock` is called by every propose function — drawdown, decay, and watermarks: [3](#0-2) [4](#0-3) [5](#0-4) 

The `timelock` field is stored as `uint32` in `PoolStopLossConfig`: [6](#0-5) 

`initialize` validates `drawdownE6` and `decayPerSecondE8` but **never validates `timelock`**:

```solidity
// L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// ← no _validateTimelock
``` [7](#0-6) 

`executeOracleStopLossTimelock` also applies the new timelock without any cap check: [8](#0-7)

### Impact Explanation

The `OracleValueStopLossExtension` is the sole mechanism protecting LPs from value drain. Its NatSpec explicitly states: *"Drawdown and decay changes are timelocked so LPs can react."* [9](#0-8) 

By bypassing the timelock, the pool admin can:

1. **Immediately set `drawdownE6 = 1e6` (100%)** — `floorMultiplier = E6 - drawdown = 0`, so `_applyWatermark` never triggers a breach regardless of how far the per-share value falls. Stop-loss is silently disabled.
2. **Immediately set `decayPerSecondE8 = 1e8`** — watermarks decay to zero in one second, again disabling all stop-loss protection.
3. **Immediately lower watermarks** — directly suppressing the breach threshold so swaps that drain LP value are no longer blocked.

LPs who deposited under the assumption that the timelock gives them a reaction window lose that protection with no on-chain warning period. This is a direct LP principal-loss risk: once stop-loss is disabled, a malicious or compromised admin can drain value from LP bins through adversarial swaps without the extension halting them.

**Impact: High** — broken core LP-protection invariant, direct path to LP fund loss.

---

### Likelihood Explanation

**Trigger 1 (pool creation):** The pool creator passes `timelock = type(uint32).max` in `extensionInitData`. No validation exists. This requires zero additional steps — the overflow is active from block 1.

**Trigger 2 (post-deployment):** Admin proposes a large timelock (e.g., `2,542,000,000` seconds, ~80.5 years, which is a plausible "permanent lock" intent), waits out the current timelock, executes it, then immediately executes any subsequent proposal.

Overflow threshold at current mainnet time (~1.753 × 10⁹): any `timelock > 4,294,967,295 − 1,753,000,000 ≈ 2,541,967,295` seconds (~80.6 years) triggers the wrap. `type(uint32).max` itself wraps to `block.timestamp − 1`, which is always in the past.

**Likelihood: Medium** — requires the pool admin to act maliciously or be compromised, but the trigger is a single parameter at pool creation with no guard.

---

### Recommendation

1. **Cap `timelock` to a safe maximum** (e.g., `365 days = 31,536,000`, well below the overflow threshold) in both `initialize` and `executeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 365 days;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

2. **Use `uint48` or `uint256` for `executeAfter`** throughout `PoolStopLossSchedule` and `PendingHighWatermarks` to eliminate the truncation entirely. `uint48` safely covers timestamps until year 8,921,556.

3. **Add an overflow-safe helper**:

```solidity
function _afterTimelock(address pool_) private view returns (uint256) {
    return block.timestamp + oracleStopLossConfig[pool_].timelock;
}
```

and store/compare as `uint256`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Foundry test demonstrating immediate timelock bypass via uint32 overflow

function test_timelockOverflowBypassesGuard() public {
    // 1. Deploy pool with timelock = type(uint32).max at creation
    uint32 overflowTimelock = type(uint32).max; // 4,294,967,295 seconds
    vm.prank(address(factoryStub));
    extension.initialize(
        address(mockPool),
        abi.encode(uint32(50_000), uint32(0), overflowTimelock)
    );

    // 2. Propose drawdown = 1e6 (disables stop-loss)
    vm.prank(admin);
    extension.proposeOracleStopLossDrawdown(address(mockPool), 1e6);

    // 3. executeAfter = uint32(block.timestamp + type(uint32).max)
    //    = uint32(1_753_000_000 + 4_294_967_295)
    //    = uint32(6_047_967_295)
    //    = 6_047_967_295 - 4_294_967_296 = 1_752_999_999  ← in the past
    // block.timestamp (1_753_000_000) >= 1_752_999_999 → _requireElapsed passes

    // 4. Execute immediately — no warp needed
    vm.prank(admin);
    extension.executeOracleStopLossDrawdown(address(mockPool)); // succeeds

    // 5. Stop-loss is now disabled
    (uint32 dd,,,) = extension.oracleStopLossConfig(address(mockPool));
    assertEq(dd, 1e6); // drawdown = 100%, floorMultiplier = 0
}
```

The `executeOracleStopLossDrawdown` call succeeds in the same block as the proposal, bypassing the intended timelock entirely.
