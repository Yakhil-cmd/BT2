### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Is Called With No Pending Mint Requests — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is `payable` and requires the caller to attach ETH to cover two L1→L2 message fees. However, the ETH-forwarding logic is gated inside an `if (amountToMint > 0)` block. When there are no pending L2 mint requests, the function returns silently without spending or refunding `msg.value`, permanently locking the caller's ETH in the contract.

---

### Finding Description

`tick()` is declared `external payable` and is designed to accept ETH from the caller to pay for two cross-chain messages:

1. `bridge().depositWithMessage{value: msgFee}(...)` — forwards minted STRK to L2
2. `messagingContract().sendMessageToL2{value: msgFee}(...)` — sends a `totalSupply` update to the L2 `MintingCurve`

Both calls are inside the `if (amountToMint > 0)` branch:

```solidity
// RewardSupplier.sol lines 107–143
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        // ...
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // ← if amountToMint == 0, msg.value is never used or refunded
}
```

When `requiredMinting()` returns `(0, 0)` — which happens whenever there are no pending L2→L1 mint-request messages — the function exits without touching `msg.value`. The ETH is absorbed by the contract with no withdrawal path defined anywhere in `RewardSupplier.sol` or its storage base `RewardSupplierStorage`. [1](#0-0) 

---

### Impact Explanation

Any ETH sent to `tick()` during a no-op call is permanently frozen inside the `RewardSupplier` contract. There is no `receive()`, `fallback()`, or withdrawal function in the contract. The funds cannot be recovered by the caller or the protocol.

This matches the allowed impact: **Permanent freezing of funds** (caller's ETH) and **Griefing** (a caller acting in good faith, e.g., a keeper bot, loses ETH on every no-op tick). [2](#0-1) 

---

### Likelihood Explanation

`tick()` is `external` with no access control — any address can call it. Keeper bots or protocol operators are expected to call it periodically. The condition `amountToMint == 0` is normal and expected: it occurs whenever the L2 reward supplier has not yet emitted a new mint-request message. A caller who attaches ETH (required for the message fees when minting *is* needed) and calls during a quiet period loses all attached ETH. This is a realistic, reachable scenario with no privileged access required. [3](#0-2) 

---

### Recommendation

Refund `msg.value` to the caller when `amountToMint == 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund unused ETH
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, document that callers must not attach ETH when there is nothing to mint, and enforce `require(msg.value == 0)` in the no-op branch.

---

### Proof of Concept

1. No L2 mint-request messages are pending (`messagingContract().l2ToL1Messages(msgHash) == 0`).
2. Caller invokes `tick{value: 1 ether}()`.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` branch is skipped.
5. Function returns. `address(rewardSupplier).balance` increases by `1 ether`.
6. No function exists to recover the ETH; it is permanently locked. [4](#0-3) [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
