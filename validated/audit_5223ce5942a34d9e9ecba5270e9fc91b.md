### Title
`claim_rewards` will self-DOS due to insufficient STRK balance when L1 funds are pending — (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

`RewardSupplier.claim_rewards` validates only the accounting counter `unclaimed_rewards` before calling `checked_transfer`, but never checks the contract's actual STRK token balance. `request_funds` counts `l1_pending_requested_amount` (tokens requested from L1 but not yet bridged) as available credit, so it may suppress further L1 requests even when the real on-chain balance is zero. Any subsequent call to `claim_rewards` will then revert on the transfer, DOSing all reward distribution.

---

### Finding Description

**`request_funds` (lines 301–331)** computes credit as:

```
credit = balance + l1_pending_requested_amount
``` [1](#0-0) 

If `credit >= unclaimed_rewards + threshold`, no new L1 mint request is sent. This means the function treats in-flight, not-yet-bridged tokens as if they were already in the contract.

**`claim_rewards` (lines 205–220)** then only checks the accounting counter:

```cairo
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
// ...
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [2](#0-1) 

There is no check that the actual STRK balance covers `amount`. If the real balance is less than `amount`, `checked_transfer` reverts, and the entire reward-distribution transaction is rolled back.

**Concrete scenario:**

| State | Value |
|---|---|
| Actual STRK balance | 0 |
| `l1_pending_requested_amount` | 1 000 STRK (requested, not yet bridged) |
| `unclaimed_rewards` | 800 STRK |
| `threshold` | 200 STRK |

`request_funds` sees `credit (0 + 1000) >= debit (800) + threshold (200)` → no new L1 request is sent. [3](#0-2) 

Staking contract then calls `claim_rewards(800)`:
- `800 <= 800` passes ✓
- `checked_transfer(800)` reverts because actual balance is 0 ✗

The L1→L2 bridge delay (hours to days) is the window during which this failure is guaranteed.

---

### Impact Explanation

Every reward-distribution path in the staking contract goes through `RewardSupplier.claim_rewards`. A revert here rolls back the entire attestation/reward-update transaction. No staker or delegator can receive rewards until the L1 bridge deposit is processed. This constitutes **temporary freezing of unclaimed yield** (High severity per the allowed impact scope). [4](#0-3) 

---

### Likelihood Explanation

The condition is reached whenever:
1. The RewardSupplier's actual STRK balance is below the amount being claimed, and
2. `l1_pending_requested_amount` is large enough that `request_funds` does not issue a new L1 request.

This is a normal operating state during every L1→L2 bridging cycle. The L1 mint request is sent proactively, but the tokens arrive only after the bridge delay. During that window, any attestation that triggers reward distribution will revert. No privileged access or special attacker action is required — a regular staker's operational address calling `attest()` is sufficient to hit this path.

---

### Recommendation

In `claim_rewards`, add a balance check before the transfer:

```cairo
let balance: Amount = token_dispatcher
    .balance_of(account: get_contract_address())
    .try_into()
    .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);
assert!(amount <= balance, "{}", Error::INSUFFICIENT_BALANCE);
```

Alternatively, implement a deferred/queued reward mechanism so that reward claims are retried once the bridged funds arrive, rather than reverting the entire transaction.

---

### Proof of Concept

1. RewardSupplier has 0 STRK balance; a prior epoch's `request_funds` sent an L1 mint request for 1 000 STRK → `l1_pending_requested_amount = 1000`.
2. Staking contract calls `update_unclaimed_rewards_from_staking_contract(800)` → `unclaimed_rewards = 800`. [5](#0-4) 
3. Inside that call, `request_funds(800)` computes `credit = 0 + 1000 = 1000`, `debit + threshold = 800 + 200 = 1000`; condition `credit < debit + threshold` is false → no new L1 request. [6](#0-5) 
4. Staking contract calls `claim_rewards(800)`.
5. `assert!(800 <= 800)` passes.
6. `token_dispatcher.checked_transfer(staking_contract, 800)` reverts — actual balance is 0. [7](#0-6) 
7. All reward distribution is DOSed until the L1 bridge deposit is processed.

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
