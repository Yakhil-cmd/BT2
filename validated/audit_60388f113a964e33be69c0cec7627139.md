### Title
Pool admin bypasses all future timelocks via `uint32` overflow in `_afterTimelock` — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` casts `block.timestamp + timelock` to `uint32` without overflow protection. A pool admin who sets `timelock = type(uint32).max` causes every subsequent `pendingXxxExecuteAfter` to wrap to a value below `block.timestamp`, making `_requireElapsed` pass immediately for all future proposals. This permanently removes the LP protection window for drawdown, decay, and watermark changes.

---

### Finding Description

`_afterTimelock` computes the execution deadline as:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

The result is truncated to `uint32`. If `timelock = type(uint32).max` (4 294 967 295), then at any realistic `block.timestamp` (e.g. 1 753 000 000 in July 2026):

```
block.timestamp + type(uint32).max  =  6 047 967 295
uint32(6 047 967 295)               =  1 752 999 999   ← one second in the past
```

`_requireElapsed` then compares this wrapped value against `block.timestamp`:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
``` [2](#0-1) 

`1 753 000 000 < 1 752 999 999` is false, so the check passes immediately for every proposal made after the overflow timelock is active.

There is no upper-bound validation on the timelock value anywhere — neither in `initialize` (which validates `drawdownE6` and `decayPerSecondE8` but not `timelock`) nor in `proposeOracleStopLossTimelock`:

```solidity
// L56-58 — no _validateTimelock call
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock accepted as-is
``` [3](#0-2)

    sched.pendingTimelock = newTimelock;
``` [4](#0-3)

### Impact Explanation

Once the overflow timelock is active, the admin can call `proposeOracleStopLossDrawdown` and `executeOracleStopLossDrawdown` in the same transaction (or block), bypassing the LP protection window entirely. The same applies to `proposeOracleStopLossDecay` / `executeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks`. LPs have no time to exit before the admin weakens stop-loss parameters (e.g. raising `drawdownE6` to `1e6` = 100%, effectively disabling the stop-loss). This is an admin-boundary break: the timelock is the only on-chain constraint on the pool admin for these parameters, and it is rendered permanently inoperative.

---

### Likelihood Explanation

The pool admin is a privileged but semi-trusted role. The attack requires one deliberate proposal of `type(uint32).max` as the new timelock, followed by waiting through the current timelock (which may be zero at initialization). No external conditions, oracle data, or token behavior are required. The initial timelock is not validated either, so a pool deployed with `timelock = type(uint32).max` from the start is immediately vulnerable. [3](#0-2) 

---

### Recommendation

Add a maximum timelock cap and validate it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply `_validateTimelock` alongside `_validateDrawdown` and `_validateDecay` in `initialize`, and at the top of `proposeOracleStopLossTimelock`. This prevents any timelock value that would cause `uint32(block.timestamp + timelock)` to wrap below `block.timestamp`.

---

### Proof of Concept

```solidity
// 1. Deploy pool with timelock = 0 (or wait through any existing timelock)
// 2. Admin proposes timelock = type(uint32).max
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);
// pendingTimelockExecuteAfter = uint32(block.timestamp + 0) = block.timestamp → passes immediately

extension.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock = type(uint32).max

// 3. Now propose a drawdown change (e.g. disable stop-loss entirely)
extension.proposeOracleStopLossDrawdown(pool, 1e6);
// _afterTimelock returns uint32(block.timestamp + 4294967295) ≈ block.timestamp - 1
// pendingDrawdownExecuteAfter < block.timestamp

// 4. Execute immediately in the same block — no waiting
extension.executeOracleStopLossDrawdown(pool);
// drawdownE6 = 1e6 (100%) — stop-loss disabled, LPs unprotected
```
