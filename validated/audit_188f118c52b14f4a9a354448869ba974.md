### Title
Missing Zero-Value Checks in `interop_root_reporter_event_hook` Allow Injection of Invalid Interop Roots into Batch Commitment - (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` processes any `InteropRootAdded` event matching the expected signature and injects the parsed `root` and `chain_id` into the system's interop root storage without validating that either value is non-zero. Because the hook fires for events emitted by **any** contract (no emitter-address restriction), an unprivileged attacker can deploy a contract that emits a crafted `InteropRootAdded` event with `root = 0` or `chain_id = 0`, causing an invalid entry to be committed into the `interop_roots_rolling_hash` that is sealed into the batch public input.

---

### Finding Description

`InteropRoot` documents the invariants clearly:

> "The merkle root hash (cannot be zero for valid roots)"
> "Source chain identifier (must be non-zero)" [1](#0-0) 

However, none of the three layers that handle an incoming interop root enforce these invariants in code:

**Layer 1 – the event hook** (`interop_root_reporter_event_hook`): validates the event signature, data length, ABI offset, array length, and topic count, but never checks `root != 0` or `chain_id != 0` before calling `add_interop_root`. [2](#0-1) 

**Layer 2 – the IO subsystem** (`add_interop_root`): charges resources and immediately delegates to `push_root` with no zero-value guard. [3](#0-2) 

**Layer 3 – the storage** (`push_root`): unconditionally appends the root to the history list. [4](#0-3) 

The hook also has **no emitter-address restriction** — `_caller_ee` is explicitly ignored and there is no check that the event originated from `L2_INTEROP_ROOT_STORAGE_ADDRESS`. Any EVM contract can emit `InteropRootAdded` with the correct 4-byte topic and trigger the hook. [5](#0-4) 

The only existing zero-root rejection is inside the Solidity bytecode of the `L2_INTEROP_ROOT_STORAGE` contract, which is bypassed entirely when the event is emitted by a different contract.

The zero root, once stored, is unconditionally folded into the rolling hash during batch finalization: [6](#0-5) 

This rolling hash is then sealed into `BatchOutput.interop_roots_rolling_hash` and committed to the settlement layer as part of the batch public input. [7](#0-6) 

---

### Impact Explanation

An attacker can corrupt the `interop_roots_rolling_hash` field of the batch public input by injecting zero or otherwise invalid interop roots. The settlement layer uses this hash to verify cross-chain messages. A corrupted hash can:

1. **Cause valid batches to be rejected** on the settlement layer if it enforces non-zero roots, creating a liveness/DoS vector against the rollup's settlement.
2. **Allow fake cross-chain messages to be committed** if the settlement layer does not independently validate root values — a zero root could be used to "prove" a non-existent cross-chain message, enabling unauthorized asset transfers or state manipulation across chains.

The `interop_roots_rolling_hash` is a security-critical field: it is the only on-chain commitment to which cross-chain roots were processed in a batch. [8](#0-7) 

---

### Likelihood Explanation

The attack requires only deploying an EVM contract and calling it within a block. No privileged access, leaked keys, or governance majority is needed. The attacker controls the event data entirely. The hook fires for any matching event signature regardless of emitter, making this trivially reachable by any unprivileged transaction sender.

---

### Recommendation

Add explicit zero-value guards in `interop_root_reporter_event_hook` before calling `add_interop_root`:

```rust
// After parsing root and chain_id:
if root.is_zero() {
    return Err(internal_error!("Interop root cannot be zero").into());
}
if chain_id.is_zero() {
    return Err(internal_error!("Interop root chain_id cannot be zero").into());
}
```

Additionally, enforce the same invariants defensively in `push_root` inside `InteropRootStorage`, and optionally in `add_interop_root` in the IO subsystem, so the documented invariants are enforced at every layer rather than relying solely on the Solidity contract. [2](#0-1) 

---

### Proof of Concept

1. Attacker deploys an EVM contract with the following logic:
   ```solidity
   // topic[0] = InteropRootAdded(uint256,uint256,bytes32[]) sig
   // topic[1] = chain_id = 0
   // topic[2] = block_or_batch_number = 0
   // data = abi.encode(32, 1, bytes32(0))  // offset=32, len=1, root=0
   emit InteropRootAdded(0, 0, new bytes32[](1)); // sides[0] = 0
   ```
2. Attacker submits a transaction calling this contract.
3. `interop_root_reporter_event_hook` fires, passes all structural checks (signature match, `data.len()==96`, `offset==32`, `len==1`, `topics.len()==3`), and calls `add_interop_root` with `root=Bytes32::ZERO, chain_id=U256::ZERO`.
4. The zero root is stored in `interop_root_storage`.
5. At batch finalization, `calculate_interop_roots_rolling_hash` folds the zero root into the rolling hash, producing a different `interop_roots_rolling_hash` than would be produced without the injected root.
6. The corrupted hash is committed to the settlement layer as part of `BatchOutput`. [9](#0-8) [10](#0-9)

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;
```

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
```
