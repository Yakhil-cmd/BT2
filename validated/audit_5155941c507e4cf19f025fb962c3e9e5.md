### Title
Integer Truncation in `native_per_pubdata` Calculation Causes Systematic Undercharging of Pubdata Fees - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the `native_per_pubdata` ratio — the number of native resource units charged per byte of pubdata — is computed using **floor (integer) division** via `wrapping_div`, while the analogous `native_per_gas` ratio correctly uses **ceiling division** (`div_ceil`). This asymmetry means that whenever `pubdata_price % native_price != 0`, every transaction that generates pubdata pays fewer native resources than the actual cost, systematically undercompensating the operator for L1 data-availability costs. An unprivileged sender can exploit this by submitting transactions that generate large amounts of pubdata, paying a discounted rate on every byte.

---

### Finding Description

In `validation_impl.rs`, the two key ratios are computed as:

```rust
// native_per_gas: ceiling division — correct, rounds UP to protect operator
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata: floor division — truncates remainder, undercharges user
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same floor-division pattern is replicated in the off-chain helper:

```rust
// native_per_pubdata = pubdata_price / native_price  (floor)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

`native_per_pubdata` is then used in two critical places:

1. **Intrinsic pubdata overhead deduction** — reducing the native limit available to the transaction:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   let native_limit = native_limit.checked_sub(intrinsic_pubdata_overhead)...
   ``` [3](#0-2) 

2. **Runtime pubdata charging** — the actual native cost charged for pubdata bytes consumed:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)...
   ``` [4](#0-3) 

Because `native_per_pubdata` is truncated, the per-byte charge is `floor(pubdata_price / native_price)` instead of the true ratio. The truncated remainder — `pubdata_price % native_price` native units per pubdata byte — is never collected.

The documentation confirms the intended formula is `nativePerGas := gasPrice / nativePrice` (exact ratio), and the design intent is that pubdata costs are fully covered: [5](#0-4) 

---

### Impact Explanation

**Operator/protocol revenue loss on every transaction with pubdata.** The operator sets `pubdata_price` to recover the cost of publishing state diffs to L1. When `pubdata_price % native_price != 0`, the truncation means the operator collects less than `pubdata_price` per native unit of pubdata cost.

Concrete example:
- `native_price = 10`, `pubdata_price = 15`
- `native_per_pubdata = floor(15/10) = 1` (true value: 1.5)
- For 1,000 pubdata bytes: user pays 1,000 native units; actual cost is 1,500 native units
- **Shortfall: 500 native units = 50 gas units per transaction**

Extreme case: if `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`):
- `native_per_pubdata = 0`
- Users are charged **zero** native resources for all pubdata
- A sender can generate arbitrarily large pubdata at no native cost, forcing the operator to absorb the full L1 publication cost

---

### Likelihood Explanation

The truncation fires on **every L2 transaction** whenever `pubdata_price % native_price != 0`. The operator-controlled `native_price` and `pubdata_price` are arbitrary U256 values read from oracle/block metadata. There is no enforcement that `pubdata_price` must be a multiple of `native_price`. Any unprivileged sender can trigger the undercharge simply by submitting a normal transaction that touches storage (generating pubdata). No special permissions, keys, or governance access are required. [6](#0-5) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — truncates remainder):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...

// After (ceiling division — rounds up, protects operator):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...
```

Apply the same fix in `api/src/helpers.rs` line 427 to keep the off-chain validation helper consistent with the on-chain bootloader logic.

---

### Proof of Concept

**Setup**: `native_price = 10`, `pubdata_price = 15`, transaction writes 1,000 storage slots (generating ~66,000 pubdata bytes).

**Step 1**: Sender submits a normal EIP-1559 transaction with `max_fee_per_gas` such that `gas_price = 15`.

**Step 2**: Bootloader computes:
```
native_per_gas     = ceil(15 / 10) = 2   ✓ (correct, rounds up)
native_per_pubdata = floor(15 / 10) = 1  ✗ (should be 2 to match the ratio)
```

**Step 3**: For 66,000 pubdata bytes:
- Charged: `66,000 × 1 = 66,000` native units
- Should charge: `66,000 × 1.5 = 99,000` native units
- **Shortfall: 33,000 native units = 3,300 gas units per transaction**

**Step 4**: Repeat across many transactions. The operator absorbs the L1 publication cost for the truncated remainder on every transaction, with no mechanism to recover it.

First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
