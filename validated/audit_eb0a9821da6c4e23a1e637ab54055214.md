### Title
Missing Access Control on `update_rewards` Allows Anyone to Permanently Suppress Block Rewards - (File: `src/staking/staking.cairo`)

### Summary
`IStakingRewardsManager::update_rewards` is a public function with no caller restriction. It accepts a `disable_rewards: bool` parameter and unconditionally writes the current block number to the global `last_reward_block` storage variable before checking `disable_rewards`. Any unprivileged caller can invoke it with `disable_rewards: true` to mark a block as "processed" while skipping reward distribution, permanently destroying that block's rewards for all stakers.

### Finding Description
The `update_rewards` function in `StakingRewardsManagerImpl` is exposed via `IStakingRewardsManager` with `#[abi(embed_v0)]` and has no access-control guard beyond `general_prerequisites()` (which only checks the pause flag). [1](#0-0) 

The critical sequence is:

1. `general_prerequisites()` — only checks `is_paused`. [2](#0-1) 
2. The block-uniqueness guard asserts `current_block_number > last_reward_block`. [3](#0-2) 
3. `last_reward_block` is **unconditionally** written to storage before the `disable_rewards` branch. [4](#0-3) 
4. If `disable_rewards` is `true`, the function returns immediately without distributing rewards. [5](#0-4) 

Because `last_reward_block` is a **global** (not per-staker) storage slot, once it is set to block N, no further call to `update_rewards` can succeed in block N — the `REWARDS_ALREADY_UPDATED` assertion will revert every subsequent attempt. [6](#0-5) 

The `disable_rewards` parameter is clearly intended for the consensus mechanism to signal that a staker did not fulfil its duties in a given block. The function is in a dedicated `IStakingRewardsManager` interface, which implies it was designed to be called by a specific privileged entity, but the access-control check was never added — directly analogous to `LSP8Burnable` inheriting from the "core" base and thereby missing the `supportsInterface` guard that all other extensions include.

### Impact Explanation
Every block's consensus rewards are a one-time event: `last_reward_block` is global and monotonically increasing. An attacker who calls `update_rewards(valid_staker, disable_rewards: true)` at the start of any block permanently destroys that block's reward allocation for **all** stakers and delegators. Repeating this every block eliminates all consensus-phase reward distribution. The lost rewards are never recoverable — they are not queued or deferred. This constitutes **permanent freezing of unclaimed yield** (High severity under the allowed impact scope).

### Likelihood Explanation
The attack requires only:
- Knowledge of any currently active staker address with non-zero balance (publicly readable from the `stakers` Vec and `staker_info` map). [7](#0-6) 
- Gas to submit one transaction per block.

No privileged key, bridge access, or token-admin role is needed. Any unprivileged address can execute this continuously.

### Recommendation
Add an explicit caller restriction to `update_rewards`. The function should only be callable by the designated consensus/sequencer contract (or a role such as `REWARDS_MANAGER`). For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_rewards_manager(); // <-- add this
    ...
}
```

Alternatively, remove the `disable_rewards` parameter from the public interface entirely and let the authorized caller invoke a separate, role-gated function.

### Proof of Concept
1. Identify any active staker address `S` with non-zero STRK balance (read from `stakers` Vec).
2. At the start of block `N` (before the legitimate consensus call), submit:
   ```
   staking_contract.update_rewards(S, disable_rewards: true)
   ```
3. `last_reward_block` is set to `N`; no rewards are distributed.
4. Any subsequent legitimate call to `update_rewards` in block `N` reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat every block. All stakers and delegators receive zero consensus rewards indefinitely. [8](#0-7) [9](#0-8)

            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
