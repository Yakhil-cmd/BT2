### Title
Paused Pool Still Accepts New Liquidity Deposits via `addLiquidity()` — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.addLiquidity()` lacks the `whenNotPaused` modifier. When a pool is paused at any level (admin level 1 or protocol level 2), new user funds can still be deposited into the pool. This directly mirrors the Maia M-05 class: a deactivated/toggled-off entity still allows a class of operations that the deactivation was meant to block.

---

### Finding Description

The pool tracks operational status in `pauseLevel`:

```
/// @dev 0 = active, 1 = paused by admin, 2 = paused by protocol.
uint8 internal pauseLevel;
``` [1](#0-0) 

The `_checkNotPaused()` guard reverts on any non-zero level:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [2](#0-1) 

`swap()` correctly applies `whenNotPaused`:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
``` [3](#0-2) 

But `addLiquidity()` only carries `nonReentrant` — **no `whenNotPaused`**:

```solidity
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
``` [4](#0-3) 

The pause transitions are enforced by the factory:

- `pausePool()` (pool admin) sets level 0 → 1
- `protocolPausePool()` (factory owner) sets level 0/1 → 2 [5](#0-4) [6](#0-5) 

Neither pause path prevents `addLiquidity()` from executing on the paused pool.

---

### Impact Explanation

The pause mechanism exists to halt all pool activity in emergencies — oracle failure, accounting bug, or an active exploit. When a pool is paused, `swap()` is correctly blocked. However, `addLiquidity()` remains fully callable, meaning:

1. New user principal enters a pool that was paused precisely because it is unsafe.
2. The deposited tokens are immediately subject to the same vulnerability that triggered the pause (e.g., a broken price provider, a bin accounting bug, or a reentrancy vector in an extension hook).
3. The depositor cannot be warned in-protocol; the transaction succeeds silently.

This is a **direct loss of user principal** — the exact impact class required by the allowed impact gate.

---

### Likelihood Explanation

- The trigger is **unprivileged**: any EOA or contract can call `addLiquidity()`.
- The condition is **reachable**: any pool that has been paused (level 1 or 2) is vulnerable.
- Pauses are an explicit, documented feature used in emergencies, so the scenario is not hypothetical.
- Users who are unaware of the pause reason, or who race the pause transaction, will deposit into a compromised pool.

---

### Recommendation

Add `whenNotPaused` to `addLiquidity()`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

`removeLiquidity()` may intentionally omit the check to allow LPs to exit a paused pool — that is a separate design decision. The critical fix is on the deposit side.

---

### Proof of Concept

1. Pool admin detects anomalous oracle behavior and calls `factory.pausePool(pool)` → `pauseLevel` set to `1`.
2. Any user calls `pool.addLiquidity(owner, salt, deltas, callbackData, extensionData)` on the now-paused pool.
3. The call succeeds: `nonReentrant` passes, no pause check fires, `LiquidityLib.addLiquidity` executes, tokens are transferred in via the swap callback.
4. The user's tokens are now locked in a pool that was paused due to a security issue.
5. If the underlying vulnerability (e.g., price provider returning manipulated prices, or a bin accounting invariant break) is exploited before the pool is fixed, the newly deposited funds are lost alongside pre-existing LP positions.
