### Title
`calculateNFTRoyalties` Returns Gross Amount as "Net Proceeds", Misleading NFT Sellers at Offer Acceptance — (File: `packages/gui/src/components/offers/utils.ts`)

### Summary
`calculateNFTRoyalties` in `utils.ts` sets `nftSellerNetAmount` to the full asking price (`amount`) instead of `amount − royaltyAmount`. Both the NFT offer viewer and editor surface this value under labels that explicitly promise a royalty-deducted figure ("Net Proceeds", "You will receive"). An NFT seller reviewing an imported buy-offer sees an inflated net-proceeds figure and may accept an offer believing they will receive the full price, while the on-chain settlement pays them only the price minus the creator royalty.

### Finding Description
In `packages/gui/src/components/offers/utils.ts` the `calculateNFTRoyalties` function computes `nftSellerNetAmount` as:

```typescript
// utils.ts line 314
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The correct subtraction is commented out. `nftSellerNetAmount` therefore equals the gross `amount` for every call. [1](#0-0) 

This value propagates to two user-facing surfaces:

**1. NFT Offer Viewer — "Net Proceeds" label**
`NFTOfferViewer.tsx` renders a "Net Proceeds" section (shown only for `TokenForNFT` offers, i.e. when an NFT owner is deciding whether to accept a buy offer). The tooltip reads *"The net proceeds include the asking price, minus any associated creator fees"*, but the displayed value is `nftSaleInfo?.nftSellerNetAmount` — the uncorrected gross amount. [2](#0-1) 

**2. NFT Offer Editor — "You will receive" label**
`NFTOfferEditor.tsx` shows `nftSellerNetAmount` under the label "You will receive" when the tab is `NFTForToken` (seller creating the offer). [3](#0-2) 

**3. `overrideNFTSellerAmount` in the viewer summary**
`NFTOfferViewer.tsx` also derives `overrideNFTSellerAmount` from `nftSellerNetAmount` and passes it into `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it replaces the raw on-chain amount in the "You will receive" row. [4](#0-3) [5](#0-4) 

### Impact Explanation
An NFT owner who receives or imports a `TokenForNFT` offer (e.g. via a dApp, QR code, or offer file) opens the viewer and sees:

| Label | Displayed | Correct |
|---|---|---|
| NFT Purchase Price | 100 XCH | 100 XCH |
| Creator Fee (10%) | 10 XCH | 10 XCH |
| **Net Proceeds** | **100 XCH** | **90 XCH** |

The seller reads the prominently displayed "Net Proceeds: 100 XCH" (bold, `h5` variant) and clicks Accept. The on-chain settlement pays them 90 XCH; 10 XCH goes to the royalty address. The discrepancy scales linearly with the royalty percentage — a 50 % royalty NFT would show 100 XCH net proceeds while delivering 50 XCH. This fits the allowed High impact: *"offer… state that causes a user to approve… the wrong… amount."*

### Likelihood Explanation
Any NFT with a non-zero `royaltyPercentage` triggers the bug. The attacker path requires no special privilege: an adversary who mints an NFT with a high royalty and then creates a buy offer (or simply shares a crafted offer file) can cause the victim to accept while seeing an inflated net-proceeds figure. The victim has no in-GUI signal that the displayed "Net Proceeds" is wrong; the royalty line and the net-proceeds line are both visible but their arithmetic is inconsistent, and most users will anchor on the large bold "Net Proceeds" figure.

### Recommendation
Restore the commented-out subtraction in `calculateNFTRoyalties`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = royaltyPercentage
  ? parseFloat((amount - parseFloat(royaltyAmountString)).toFixed(12))
  : amount;
``` [6](#0-5) 

The `makerFee` deduction in the original comment should be reviewed separately; for the viewer path the maker fee is already shown as a distinct line item and should not be double-counted.

### Proof of Concept
1. Mint an NFT with `royaltyPercentage = 1000` (10 %).
2. Create a `TokenForNFT` offer offering 100 XCH for the NFT; export the offer file.
3. Import the offer file into a second wallet that owns the NFT.
4. Open the NFT Offer Viewer. Observe:
   - "Creator Fee (10%): 10 XCH" — correct.
   - "**Net Proceeds: 100 XCH**" — incorrect; should be 90 XCH.
5. Accept the offer. The wallet receives 90 XCH, not 100 XCH as displayed.

The root cause is `nftSellerNetAmount = amount` at [7](#0-6) 
with the correct formula commented out immediately below. [8](#0-7)
