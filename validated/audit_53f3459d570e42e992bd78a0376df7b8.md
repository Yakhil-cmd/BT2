### Title
`max_receipt_size` Invariant Bypassed via `promise_return` Mutation of `output_data_receivers` — (`runtime/runtime/src/lib.rs`)

---

### Summary

The `max_receipt_size` hard limit (4 MiB) is enforced only at the moment a new receipt is first constructed and validated. When a contract uses `promise_return`, the runtime post-validation mutates the returned receipt by appending `output_data_receivers` from the parent receipt, silently pushing the receipt's serialized size above `max_receipt_size`. The resulting oversized receipt is stored in state and subsequently processed under `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check. Any unprivileged user who can deploy a contract can trigger this path.

---

### Finding Description

**Invariant**: Every receipt stored in state must satisfy `borsh::object_length(receipt) ≤ limit_config.max_receipt_size` (currently 4 MiB).

**Where the invariant is enforced**: `validate_receipt()` in `runtime/runtime/src/verifier.rs` checks the size only when `mode == ValidateReceiptMode::NewReceipt`:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

**Where the invariant is bypassed**: After a receipt passes `NewReceipt` validation, the runtime in `apply_action_receipt` (called from `process_receipt_and_instant_receipts`) directly mutates the receipt's `output_data_receivers` without re-validating size:

```rust
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        // Modifying a new receipt instead of sending data
        match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
            ReceiptEnum::Action(new_action_receipt) | ... => new_action_receipt
                .output_data_receivers
                .extend_from_slice(&action_receipt.output_data_receivers()),
            ...
        }
    }
``` [2](#0-1) 

No size re-check follows this mutation. The receipt is then assigned an ID and forwarded to the outgoing receipt sink.

**Why the downstream check fails**: When the oversized receipt is later processed (as a delayed or incoming receipt), it is validated with `ValidateReceiptMode::ExistingReceipt`, which skips the size check. The `ExistingReceipt` mode was explicitly introduced to tolerate these oversized receipts:

```rust
/// 2) There is a bug which allows to create receipts that are above the size limit. Runtime has
///    to handle them gracefully until the receipt size limit bug is fixed.
///    See https://github.com/near/nearcore/issues/12606 for details.
ExistingReceipt,
``` [3](#0-2) 

Both `process_delayed_receipts` and `process_incoming_receipts` use `ExistingReceipt` mode: [4](#0-3) [5](#0-4) 

**Attacker-controlled fields**: The attacker controls:
- The size of the FunctionCall `args` in the inner promise (to size receipt C to exactly `max_receipt_size`)
- The promise DAG structure (to ensure `output_data_receivers` is non-empty on the parent)
- The `promise_return` call (to trigger the mutation path)

The test in the codebase explicitly demonstrates this is reachable and produces an oversized receipt in state: [6](#0-5) 

---

### Impact Explanation

The `max_receipt_size` limit exists to keep `ChunkStateWitness` under 17 MiB. A receipt exceeding 4 MiB violates this bound. Concretely:

1. **State witness size violation**: An oversized receipt contributes more than its allowed share to the state witness, potentially pushing the total witness above the 17 MiB target and causing chunk validators to produce a different application result than the chunk producer, breaking consensus endorsement.
2. **Invariant corruption persisted in state**: The oversized receipt is stored in the delayed receipt queue or forwarded cross-shard, meaning the corrupted state persists across blocks until the receipt is eventually executed.
3. **Second variant — value return**: A contract returning a value of exactly `max_receipt_size` bytes causes the wrapping `DataReceipt` to exceed `max_receipt_size` (due to `data_id` overhead), via the same bypass. `validate_data_receipt` only checks `max_length_returned_data`, not the total receipt size. [7](#0-6) 

---

### Likelihood Explanation

- **Unprivileged user**: Any account that can deploy a contract and call it can trigger this. No validator, admin, or special privilege is required.
- **Precise sizing is feasible**: The test demonstrates that the exact `args_size` needed to hit `max_receipt_size` is computable from the base receipt template size, which is deterministic.
- **No rate limiting**: There is no per-account limit on how many such receipts can be created per chunk.
- **Acknowledged open bug**: The issue is tracked as `nearcore#12606` and the `ExistingReceipt` mode was added as a workaround, confirming the bug is reachable in production.

---

### Recommendation

After the `promise_return` mutation at `runtime/runtime/src/lib.rs` lines 1029–1035, add a post-mutation size check:

```rust
ReceiptEnum::Action(new_action_receipt) | ... => {
    new_action_receipt.output_data_receivers
        .extend_from_slice(&action_receipt.output_data_receivers());
    // Re-validate size after mutation
    let new_size = borsh::object_length(&*modified_receipt).unwrap() as u64;
    if new_size > limit_config.max_receipt_size {
        // Abort: treat as action error, do not forward the oversized receipt
        result.result = Err(ActionErrorKind::NewReceiptValidationError(
            ReceiptValidationError::ReceiptSizeExceeded { size: new_size, limit: limit_config.max_receipt_size }
        ).into());
        return Ok(());
    }
}
```

Similarly, for the value-return path, validate the total `DataReceipt` size (not just `data.len()`) against `max_receipt_size` before forwarding.

Once the fix is in place, `ValidateReceiptMode::ExistingReceipt` can be tightened to also enforce the size limit, and the workaround comment referencing issue #12606 can be removed.

---

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete, runnable proof of concept:

1. Deploy `near_test_contracts::rs_contract()`.
2. Compute `args_size = max_receipt_size - base_receipt_size` (deterministic).
3. Call `max_receipt_size_promise_return_method1` with `{"args_size": <args_size>}`.
   - Method 1 creates promise DAG `[A -then-> B]`.
   - When A executes, it creates receipt C (sized to exactly `max_receipt_size`) and calls `promise_return(C)`.
   - The runtime appends B's `output_data_receivers` to C, making C exceed `max_receipt_size`.
4. Call `assert_test_completed` — succeeds, proving the oversized receipt was processed.
5. `assert_oversized_receipt_occurred` scans the chain and confirms a receipt with `size > max_receipt_size` exists in the incoming receipts.

The test passes today, confirming the invariant is broken in production code. [8](#0-7)

// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
