### Title
Malicious Participant Can Permanently Deny Signing by Sending an Invalid Signature Share When Cheater Detection Is Disabled — (`src/frost/eddsa/sign.rs`)

### Summary
In `do_sign_coordinator_v1`, the coordinator collects `round2::SignatureShare` values from all participants via `recv_from_others` with no per-share validation. Because `PublicKeyPackage` is constructed with an empty `verifying_shares` map (cheater detection explicitly disabled), `aggregate` cannot verify individual shares and only checks the final aggregated signature. A single malicious participant who sends any invalid share (e.g., a zero scalar) causes `aggregate` to fail, returning `ProtocolError::AssertionFailed`. The coordinator has no mechanism to identify the culprit and must abort. Since the coordinator always waits for **all** participants in the list before aggregating, the malicious participant can repeat this every session, permanently blocking signing.

### Finding Description

In `do_sign_coordinator_v1`:

- Line 150–153: `recv_from_others` collects one `round2::SignatureShare` from every participant. The helper performs only deserialization — no cryptographic validation of the share value. [1](#0-0) 

- Lines 163–165: `PublicKeyPackage::new(BTreeMap::new(), vk_package)` passes an **empty** `verifying_shares` map. The `frost_ed25519::aggregate` function uses `verifying_shares` to verify each share individually (cheater detection). With an empty map, it skips per-share verification and only checks whether the final aggregated signature is valid against the group public key. [2](#0-1) 

- The code comment explicitly acknowledges this: *"We supply empty map as `verifying_shares` because we have disabled 'cheater-detection' feature flag."* [3](#0-2) 

- `recv_from_others` itself performs no validation beyond deserialization: [4](#0-3) 

The same pattern is present in `do_sign_coordinator_v2`: [5](#0-4) 

### Impact Explanation

Any participant who is a legitimate member of the signing group can send an arbitrary (invalid) `SignatureShare` — e.g., a zero scalar or a random scalar — to the coordinator. The coordinator inserts it into `signature_shares` without checking it, then calls `aggregate`. Because the sum of shares is wrong, the final EdDSA signature fails verification inside `aggregate`, which returns an error. The coordinator maps this to `ProtocolError::AssertionFailed` and aborts. Since cheater detection is disabled, the coordinator receives no information about which participant caused the failure. The malicious participant can repeat this on every signing session, indefinitely blocking signing for all honest parties.

This matches: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation

- The attacker only needs to be a legitimate participant in the signing group (no key leakage, no external compromise required).
- The attack requires sending one crafted message per signing session — trivially easy.
- The coordinator always waits for **all** participants before aggregating, so there is no way to proceed without the malicious participant.
- Without cheater detection, honest parties have no in-protocol mechanism to identify and exclude the culprit.

### Recommendation

Enable per-share verification before aggregation. The `frost_ed25519` library supports this when `verifying_shares` is populated. The `KeygenOutput` already contains the group public key; the per-participant verifying shares can be derived from the DKG output and stored in `KeygenOutput`. Populate `PublicKeyPackage` with the actual `verifying_shares` so that `aggregate` can identify and reject the bad share before the final signature check fails opaquely.

Alternatively, if cheater detection is intentionally omitted for performance reasons, the library must document that any participant can undetectably abort signing, and callers must implement an out-of-band accountability mechanism.

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
