### Title
WalletConnect Bypass-Approved Commands Execute Silently After Wallet Logout — (File: `packages/core/src/components/Auth/AuthProvider.tsx`, `packages/gui/src/hooks/useWalletConnectCommand.tsx`)

---

### Summary

The GUI's `logOut()` function is an explicit no-op. After a user clicks "Log Out" and is returned to the key-selection screen, the Chia daemon backend remains logged in with the previous fingerprint, and the `WalletConnectProvider` — mounted at the application root — continues processing inbound `session_request` events. A dApp that was previously granted **bypass** (no-confirmation-required) permissions for commands such as `chia_sendTransaction` can silently drain XCH or CAT from the wallet during this window, with no user prompt and no visible indication.

---

### Finding Description

**1. `logOut` is a deliberate no-op**

`AuthProvider.tsx` defines `handleLogOut` as an empty callback with an explicit comment acknowledging the backend stays logged in:

```typescript
// immutable
const handleLogOut = useCallback(async () => {
  // do nothing until backend change API,
  // user is still logged in and syncing with previously selected fingerprint
}, []);
``` [1](#0-0) 

`LogoutButton` calls this no-op and then navigates to `/`:

```typescript
const handleLogout = useCallback(async () => {
  await logOut();   // ← no-op
  navigate('/');    // ← only a client-side route change
}, [logOut, navigate]);
``` [2](#0-1) 

No RPC call to the daemon is made. The daemon's `get_logged_in_fingerprint` will continue to return the previous fingerprint.

**2. `WalletConnectProvider` is mounted at the application root, above all routes**

`AppProviders.tsx` wraps the entire app — including the key-selection screen — with `WalletConnectProvider`:

```tsx
<WalletConnectProvider projectId={WalletConnectChiaProjectId}>
  <AppState>{outlet ? <Outlet /> : children}</AppState>
  ...
</WalletConnectProvider>
``` [3](#0-2) 

The `session_request` listener registered inside `WalletConnectProvider` is never torn down on logout:

```typescript
client.on('session_request', handleSessionRequest);
``` [4](#0-3) 

**3. Fingerprint check passes because the backend is still logged in**

`useWalletConnectCommand` reads `currentFingerprint` from `useGetLoggedInFingerprintQuery()`. Because the daemon was never told to log out, this query still returns the old fingerprint, and the guard passes:

```typescript
const requestedFingerprint = fingerprint ?? currentFingerprint;
if (
  typeof requestedFingerprint !== 'number' ||
  !requestedFingerprint ||
  requestedFingerprint !== pair.fingerprint ||
  currentFingerprint !== pair.fingerprint   // ← still true after logout
) {
  throw new WcError(`Fingerprint not allowed for this command`, ...);
}
``` [5](#0-4) 

**4. Bypass commands execute without any user confirmation**

`dispatchPairRequest.ts` skips the `confirm()` dialog entirely for commands in the pair's `bypass` list:

```typescript
if (pair.bypass.includes(command)) {
  return process(context);   // ← no dialog, no user interaction
}
``` [6](#0-5) 

Commands such as `chia_sendTransaction`, `chia_createOfferForIds`, or `chia_cancelOffer` can be placed in the bypass list by the user at pairing time. After logout, these execute silently.

---

### Impact Explanation

An attacker who controls a dApp that was previously granted bypass permissions for `chia_sendTransaction` (or equivalent spend/offer commands) can:

1. Wait for the user to click "Log Out."
2. Send a `session_request` over the still-active WalletConnect session.
3. The request passes all guards (pair exists, command allowed, fingerprint matches, bypass set).
4. The transaction is submitted to the daemon and broadcast to the network — **without any user prompt, dialog, or visible indication** — while the user believes they have ended the session.

This results in unauthorized XCH/CAT transfer, offer creation/cancellation, or other spend-class actions against the wallet the user thought they had logged out of.

---

### Likelihood Explanation

- The user must have previously connected a dApp via WalletConnect and granted it bypass permissions for a spend-class command. This is a normal, documented user action.
- The attacker controls the dApp (or the dApp's backend is compromised).
- The attack window is open from the moment the user clicks "Log Out" until they either select a different key (which triggers a real `log_in` RPC to the daemon, changing the fingerprint) or closes the application.
- No cryptographic break, phishing, or host compromise is required beyond the already-established WalletConnect pairing.

---

### Recommendation

1. **Make `handleLogOut` call the daemon.** Issue a `log_in` RPC with a sentinel value, or add a dedicated `log_out` RPC, so the daemon's active fingerprint is cleared on logout.
2. **Disconnect or suspend WalletConnect sessions on logout.** When the user navigates to the key-selection screen, call `disconnectPair` for all active pairs, or at minimum reject all inbound `session_request` events until a new key is selected.
3. **Guard `processSessionRequest` against the unauthenticated state.** If `currentFingerprint` is `undefined` or does not match any logged-in key, reject the request immediately before any bypass or confirmation logic runs.

---

### Proof of Concept

1. Connect a dApp via WalletConnect and grant it bypass permission for `chia_sendTransaction` (checked during the pairing dialog).
2. Click the "Log Out" button in the GUI. The UI navigates to the key-selection screen.
3. From the dApp, send a `session_request` for `chia_sendTransaction` with the previously granted fingerprint and a destination address/amount controlled by the attacker.
4. Observe in the daemon logs or on-chain that the transaction is submitted and confirmed — **no dialog appeared in the GUI**.

The root cause is confirmed at:
- `packages/core/src/components/Auth/AuthProvider.tsx` lines 52–55 (no-op logout)
- `packages/gui/src/components/app/AppProviders.tsx` line 105 (provider mounted above all routes)
- `packages/gui/src/hooks/useWalletConnectCommand.tsx` lines 38–46 (fingerprint check passes because daemon is still logged in)
- `packages/gui/src/electron/utils/dispatchPairRequest.ts` lines 57–59 (bypass skips confirmation)
