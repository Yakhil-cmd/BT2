### Title
Uint64 Underflow in `ComputationRemaining` Returns Inflated Budget Instead of Zero - (`File: fvm/meter/computation_meter.go`)

### Summary

`ComputationRemaining` in the Flow FVM computation meter contains a dead-code guard (`uint64 <= 0`) that can never trigger, combined with an unsigned integer subtraction that underflows when `computationUsed > computationLimit`. The result is that the function returns a value near `math.MaxUint64` instead of `0`, misreporting available computation budget to callers — including the EVM gas-limit derivation path.

### Finding Description

In `fvm/meter/computation_meter.go`, `ComputationRemaining` computes remaining budget as:

```go
remainingComputationUsage := m.params.computationLimit - m.computationUsed
if remainingComputationUsage <= 0 {
    return 0
}
return remainingComputationUsage / w
```

Both `computationLimit` and `computationUsed` are `uint64`. When `computationUsed > computationLimit`, the subtraction wraps around to a value near `math.MaxUint64` (Go unsigned integer underflow). The guard `remainingComputationUsage <= 0` is **dead code**: a `uint64` is always `>= 0`, so this branch is unreachable. The function then returns `(math.MaxUint64 - delta) / w` — an astronomically large value — instead of `0`. [1](#0-0) 

`computationUsed` can legitimately exceed `computationLimit` in two ways:

1. **`MeterComputation`** adds usage first, then checks the limit and returns an error — leaving `computationUsed > computationLimit` in the meter state. [2](#0-1) 

2. **`Merge`** accumulates a child meter's `computationUsed` into the parent with no bounds check. [3](#0-2) 

### Impact Explanation

`ComputationRemaining` is consumed by the EVM integration layer (`fvm/evm/handler/handler.go`) to derive the gas limit passed to EVM transaction execution. When the function returns a near-`MaxUint64` value, the EVM handler believes an enormous gas budget remains and passes that inflated limit to the EVM, allowing EVM transactions to execute with effectively unbounded gas — bypassing the Flow computation limit that was already exceeded. This constitutes unauthorized resource consumption beyond the protocol-enforced limit. [4](#0-3) 

### Likelihood Explanation

Any transaction sender can craft a transaction that first exhausts the computation limit (causing `computationUsed > computationLimit` in the meter), then triggers an EVM call in the same execution context. The `Merge` path also makes this reachable without the sender explicitly exceeding the limit in a single metered call. No privileged access, staked node control, or key compromise is required.

### Recommendation

Replace the dead `uint64 <= 0` guard with an explicit signed comparison or a pre-check:

```go
// Before subtracting, check if limit is already exhausted
if m.computationUsed >= m.params.computationLimit {
    return 0
}
remainingComputationUsage := m.params.computationLimit - m.computationUsed
return remainingComputationUsage / w
```

This mirrors the fix pattern recommended in the external report: explicitly handle the edge case where the "reserve" (used computation) exceeds the "cash" (limit) before performing the subtraction.

### Proof of Concept

1. Submit a Flow transaction that meters computation until `MeterComputation` returns `ComputationLimitExceededError` — at this point `computationUsed > computationLimit` in the meter.
2. Within the same transaction execution context, invoke an EVM call that reads `ComputationRemaining` to derive its gas limit.
3. `ComputationRemaining` computes `computationLimit - computationUsed` as a `uint64`, underflows to ~`math.MaxUint64`, the `<= 0` guard is skipped (dead code), and the EVM receives a gas limit of `math.MaxUint64 / w`.
4. The EVM transaction executes with an effectively unbounded gas budget, consuming resources beyond the Flow computation limit. [5](#0-4)
