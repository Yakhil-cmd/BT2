### Title
Overly Broad `account_tx_in_pool_or_recent_block` Check Allows Signature Bypass for Invoke Transactions at Nonce 1 - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 when the account is not yet deployed, but only when a `deploy_account` transaction is already in the mempool. The guard used — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the pool for that address, not exclusively a `deploy_account`. Because `max_allowed_nonce_gap` is 200 in production, an attacker can first inject a future-nonce invoke transaction (nonce ≥ 2) for an undeployed account, causing the guard to return `true`, and then submit a nonce=1 invoke transaction with an **arbitrary/invalid signature**. The gateway skips `__validate__` and admits the invalid transaction into the mempool.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` decides whether to skip the blockifier's `__validate__` entry-point call:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The comment's reasoning is incorrect. `account_tx_in_pool_or_recent_block` checks:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type in the pool — including invoke transactions with future nonces. It does not verify that the pooled transaction is a `deploy_account`.

The production `max_allowed_nonce_gap` is 200:

```json
"gateway_config.static_config.stateful_tx_validator_config.max_allowed_nonce_gap": 200
``` [3](#0-2) 

This means an invoke transaction with nonce=2 (or up to nonce=200) can be submitted for an undeployed account (account_nonce=0) and will pass the gateway's nonce check:

```rust
let max_allowed_nonce =
    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
``` [4](#0-3) 

Once the nonce=2 invoke tx is in the pool, `account_tx_in_pool_or_recent_block` returns `true` for that address. A subsequent nonce=1 invoke tx with an invalid signature then satisfies all three conditions in `skip_stateful_validations` and `skip_validate=true` is returned.

In `run_validate_entry_point`, `skip_validate=true` sets `validate: false` in `ExecutionFlags`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4) 

And in `validate_tx`:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [6](#0-5) 

The `__validate__` entry point — which performs signature verification — is never called. The invalid-signature nonce=1 invoke transaction is admitted to the mempool.

### Impact Explanation

**High** — Mempool/gateway admission accepts invalid transactions before sequencing. An unprivileged attacker can inject an invoke transaction with an arbitrary/invalid signature into the mempool for any undeployed account. The transaction will eventually fail at execution time (the blockifier will call `__validate__` during block building and reject it), but it occupies mempool space, wastes batcher resources, and pollutes the transaction pool with unauthenticated transactions. This matches the impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

Any unprivileged user can trigger this with two sequential RPC calls:
1. Submit an invoke tx with nonce=2 (or any nonce 2–200) for a target undeployed account.
2. Submit an invoke tx with nonce=1 and an invalid signature for the same account.

No special privileges, no race conditions, and no configuration changes are required. The production `max_allowed_nonce_gap=200` makes step 1 trivially achievable.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the account. For example, add a dedicated mempool query `deploy_account_tx_in_pool(address)` that inspects the transaction type of the pooled entry at nonce=0, or filter `tx_pool.contains_account` to only match `DeployAccount` transaction types.

### Proof of Concept

```
// Step 1: Attacker submits a future-nonce invoke tx for an undeployed account.
// account_nonce = 0, tx_nonce = 2 (within max_allowed_nonce_gap=200).
// This passes validate_nonce (0 <= 2 <= 200) and is added to the mempool.
// Now account_tx_in_pool_or_recent_block(attacker_address) == true.

// Step 2: Attacker submits a nonce=1 invoke tx with an INVALID signature.
// In skip_stateful_validations:
//   tx.nonce() == Nonce(Felt::ONE)  ✓
//   account_nonce == Nonce(Felt::ZERO)  ✓
//   account_tx_in_pool_or_recent_block(attacker_address) == true  ✓  (due to nonce=2 tx)
// => skip_validate = true
// => run_validate_entry_point called with validate=false
// => __validate__ entry point NOT called
// => Invalid-signature tx admitted to mempool without signature verification.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
