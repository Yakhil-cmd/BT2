Audit Report

## Title
Unbounded Reward Rollover Allows First Voter to Claim Entire Accumulated SNS Rewards Purse - (File: rs/sns/governance/src/governance.rs)

## Summary
SNS governance `distribute_rewards` computes a rewards purse each round proportional to the **total token supply** regardless of whether any neurons are staked or any proposals exist. When no proposals are settled, the entire purse rolls over via `total_available_e8s_equivalent`. An attacker who stakes the first neuron after any period of no settled proposals can vote on a single proposal and receive the entire accumulated purse, which grows linearly with the number of empty rounds.

## Finding Description

**Reward accrual (no staked-supply guard):**
`distribute_rewards` computes the purse using the full circulating supply as principal with no check for whether any neurons are staked:

```rust
result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
``` [1](#0-0) 

**Unconditional rollover when no proposals settle:**
After computing the purse, if `considered_proposals` is empty, `total_reward_shares == 0` and no maturity is distributed. The function still writes a new `RewardEvent` with `settled_proposals: []` and `total_available_e8s_equivalent` set to the full purse: [2](#0-1) 

**Rollover predicate feeds back into next round:**
`rewards_rolled_over()` returns `true` when `settled_proposals.is_empty()`, and `e8s_equivalent_to_be_rolled_over()` returns the full `total_available_e8s_equivalent` in that case: [3](#0-2) 

That rolled-over amount seeds the next round's purse: [4](#0-3) 

**Integration test confirms the rollover accumulation:**
The SNS integration test explicitly asserts that `total_available_e8s_equivalent` grows each empty round and that the voter who settles the first proposal after a hiatus receives the entire accumulated amount: [5](#0-4) 

**Attack path:**
1. SNS is deployed with no initial neurons (or all neurons dissolve). Rewards begin accruing from `genesis_timestamp_seconds`.
2. Each round, `run_periodic_tasks` calls `distribute_rewards`; `considered_proposals` is empty; the purse rolls over.
3. After N rounds the purse ≈ `N × rate × round_duration × total_supply`.
4. Attacker stakes one neuron with `neuron_minimum_stake_e8s`, submits a `Motion` proposal (cost: `proposal_reject_cost_e8s`), and votes on it — the only neuron, so 100% of voting power.
5. At the next reward event, `total_reward_shares == attacker_shares`, so `neuron_reward_e8s = rewards_purse_e8s × 1.0` — the entire accumulated purse. [6](#0-5) 
6. Attacker calls `DisburseMaturity` to convert maturity to tokens.

**No existing guard prevents this:** The only early-return guards in `distribute_rewards` are for `voting_rewards_parameters` not set, `round_duration_seconds == 0`, and `new_rounds_count == 0`. There is no check for zero staked supply. [7](#0-6) 

## Impact Explanation
This is a **High** severity finding matching: *"Significant SNS or infrastructure security impact with concrete user or protocol harm."* An unprivileged attacker with only `neuron_minimum_stake_e8s` tokens can extract governance rewards that were funded by dilution of all token holders over an extended period. For a 30-day gap at 5% annual rate, the purse ≈ 0.41% of total supply — potentially millions of dollars for large SNS deployments. The attacker's cost is bounded by `neuron_minimum_stake_e8s + proposal_reject_cost_e8s`; the gain is unbounded and grows linearly with the gap duration.

## Likelihood Explanation
The attack is realistic in at least two concrete, documented SNS lifecycle scenarios:
1. **SNS genesis before swap completion**: The swap canister calls `claim_swap_neurons` only after the swap commits. During the swap window, no neurons exist in governance, but `distribute_rewards` runs every round.
2. **All neurons dissolved**: If all existing neurons dissolve and no one re-stakes, the purse accumulates until the first new staker.

No privileged access is required. The attack is executable by any principal via the public `manage_neuron` endpoint. [8](#0-7) 

## Recommendation
Add a guard in `distribute_rewards` before computing the purse to skip reward accrual (or skip the rollover) when total staked supply is zero:

```rust
let total_staked_e8s: u64 = self.proto.neurons.values()
    .map(|n| n.cached_neuron_stake_e8s)
    .sum();
if total_staked_e8s == 0 {
    // Advance the reward event round counter without rolling over the purse.
    // Update latest_reward_event with total_available_e8s_equivalent = 0.
    return;
}
```

Alternatively, discard (rather than roll over) the purse when `total_staked_e8s == 0`, so rewards only accumulate during periods when there is actual staked supply to distribute to. This mirrors the standard DeFi pattern of skipping reward accrual when `totalSupply == 0`.

## Proof of Concept
A deterministic unit/integration test using PocketIC or the existing SNS test harness:

1. Deploy SNS with `initial_reward_rate_basis_points = 500`, `round_duration_seconds = 86400`, and **no initial neurons**.
2. Advance time by 30 days, calling `run_periodic_tasks` each round. Assert `latest_reward_event.settled_proposals == []` and `total_available_e8s_equivalent` grows each round (mirrors the existing rollover test at `rs/sns/integration_tests/src/proposals.rs` lines 2120–2147).
3. Stake one neuron with `neuron_minimum_stake_e8s`. Submit and vote on a `Motion` proposal.
4. Advance time past the voting deadline and one reward round. Call `distribute_rewards`.
5. Assert `neuron.maturity_e8s_equivalent == total_available_e8s_equivalent` from step 2 (entire 30-day purse).
6. Call `DisburseMaturity` and assert the attacker receives far more than their initial stake.
