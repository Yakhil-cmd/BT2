### Title
Malicious Participant Can Inject Arbitrary Unverified Shares into CKD Aggregation, Corrupting the Derived Confidential Key — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator unconditionally aggregates `(big_y, big_c)` values received from every participant with no cryptographic verification that those values were honestly computed. Any single malicious participant can send arbitrary group elements, causing the coordinator to output a `CKDOutput` that does not correspond to the actual master secret key applied to the `app_id`. This is a direct analog of the GNTDeposit access-control bug: just as `withdraw` accepted any `_to` address without restriction, `do_ckd_coordinator` accepts any `CKDOutput` payload from any participant without restriction.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, `do_ckd_coordinator` collects one `CKDOutput` from every other participant and adds the components together:

```rust
// lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each honest participant is supposed to compute:

```
big_y  = λ_i · y_i · G
big_c  = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

where `x_i` is their private share and `y_i` is a fresh random scalar (see `compute_signature_share`, lines 148–181). The coordinator then sums these to obtain:

```
Y = Σ λ_i · y_i · G
C = Σ λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

so that `unmask(app_sk) = C − app_sk · Y = msk · H(pk ‖ app_id)`.

**There is no zero-knowledge proof, commitment, or any other check** that the received `(big_y, big_c)` pair was computed from the participant's actual key share and a valid random scalar. The coordinator has no mechanism to distinguish a correctly-formed share from an arbitrary pair of group elements.

A malicious participant `P_m` can send any `(big_y', big_c')` of their choosing. The coordinator will add these into the running sum, producing a `CKDOutput` whose `unmask` result is:

```
C' − app_sk · Y' ≠ msk · H(pk ‖ app_id)
```

The coordinator outputs this corrupted value as `Some(ckd_output)` and considers the protocol successful.

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an incorrect derived key.**

The coordinator is the sole party that receives `Some(CKDOutput)`. It has no way to detect the corruption. Any downstream consumer of the derived confidential key (e.g., a TEE that uses it to decrypt application secrets) will silently operate on a wrong key. Because the corruption is additive and attacker-controlled, the attacker can steer the output to any target value they choose in `G1`, including the identity element, making the derived key entirely predictable or unusable.

---

### Likelihood Explanation

**Medium.** The attacker must be an enrolled participant in the CKD session (a documented semi-honest-or-worse trust boundary). No external capability is required: the attack is a single-round deviation — send a crafted `(big_y, big_c)` instead of the honest computation. The protocol provides no detection mechanism, so the attack succeeds silently every time it is attempted.

---

### Recommendation

Require each participant to accompany their `(big_y, big_c)` with a non-interactive zero-knowledge proof of correct formation — specifically, a proof of knowledge of `(y_i, x_i)` such that:

```
big_y = λ_i · y_i · G
big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

and that `x_i` is consistent with the participant's public key share (which is publicly known from the DKG output). The coordinator must verify all proofs before aggregating. Alternatively, use a verifiable ElGamal encryption scheme where the ciphertext structure itself can be checked against the participant's public key share commitment.

---

### Proof of Concept

1. Run a CKD session with participants `[P_1, P_2, P_3]` and coordinator `P_1`.
2. `P_2` (malicious) intercepts the protocol at the point where it would call `compute_signature_share` and instead constructs:
   ```
   big_y' = identity (G1 zero element)
   big_c' = identity (G1 zero element)
   ```
3. `P_2` sends `(big_y', big_c')` to `P_1` via `chan.send_private`.
4. `P_1`'s `do_ckd_coordinator` (lines 50–55) adds these zero elements into `norm_big_y` and `norm_big_c` without complaint.
5. The final `CKDOutput` is missing `P_2`'s contribution entirely, so `unmask(app_sk)` returns a value that does not equal `msk · H(pk ‖ app_id)`.
6. The coordinator outputs `Some(corrupted_ckd_output)` and the protocol terminates successfully with no error.

The attacker can generalize step 2 to any chosen `(big_y', big_c')` pair, steering the output to any desired group element. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
