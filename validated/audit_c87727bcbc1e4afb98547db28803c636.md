### Title
`TransactionReceipt.fee` Computed from Full-Execution Gas While `.gas` Uses Validation-Only State — (`crates/blockifier/src/transaction/account_transaction.rs`)

### Summary

In `run_revertible`, when execution succeeds but `PostExecutionReport` detects a resource overdraft under `AllResources` bounds, the returned `TransactionReceipt` is assembled with a struct-update expression that sources `fee` and `gas` from two **different gas vectors**. The `fee` field is computed from the full-execution gas vector (capped at bounds), while the `gas` field is computed from validation-only state changes plus reverted steps/sierra gas. This violates the invariant `receipt.fee == receipt.gas.cost(gas_prices, tip)` and causes simulation/trace to return an authoritative-looking but internally inconsistent receipt.

### Finding Description

**Step 1 — Execution succeeds, full-execution receipt is built:**

At lines 770–786, after `run_execute` returns `Ok`, a `tx_receipt` is built from the squashed state diff of both `validate_state_cache` and `execution_state` (i.e., full execution state changes), plus both validate and execute call infos. [1](#0-0) 

**Step 2 — `PostExecutionReport` detects overdraft, `recommended_fee` is computed from full-execution gas:**

`PostExecutionReport::new` receives `tx_receipt` (the full-execution receipt). When `check_actual_cost_within_bounds` finds `MaxGasAmountExceeded`, `FeeCheckReport::from_fee_check_error` is called with `actual_gas = tx_receipt.gas` (the full-execution gas vector). For `AllResources`, `recommended_fee` is computed as:

```
fee = cost(min(l1_gas_bound, full_exec_l1_gas),
           min(l2_gas_bound, full_exec_l2_gas),
           min(l1_data_gas_bound, full_exec_l1_data_gas))
``` [2](#0-1) 

**Step 3 — `get_revert_receipt()` produces a different gas vector:**

The `get_revert_receipt` closure (lines 752–764) builds a receipt using only `validate_state_cache.to_state_diff()` (no execution state changes) and only `validate_call_info`. It passes `execution_steps_consumed` and `sierra_gas_revert_tracker.get_gas_consumed()` as `reverted_steps`/`reverted_sierra_gas`, which affect computation gas but **not** the L1 data gas from state changes. [3](#0-2) 

**Step 4 — The struct-update expression mixes the two:**

```rust
let tx_receipt = TransactionReceipt {
    fee: post_execution_report.recommended_fee(),  // from full-exec gas
    ..get_revert_receipt()                          // gas from validate-only state
};
``` [4](#0-3) 

The `gas` field in the returned receipt comes from `get_revert_receipt()`, which uses only validation state changes. The `fee` field was computed from `min(bounds, full_exec_gas)`, which includes execution state changes. These are structurally different gas vectors.

**The invariant broken:** `receipt.fee != receipt.gas.cost(gas_prices, tip)`.

`GasVector::cost` is the canonical fee-from-gas function: [5](#0-4) 

`from_params` in `receipt.rs` always sets `fee = get_fee_by_gas_vector(gas, ...)` so that `fee == gas.cost(...)` for any normally-constructed receipt: [6](#0-5) 

The struct-update at lines 801–804 is the only place in the codebase that breaks this invariant by overwriting `fee` with a value derived from a different gas vector than the one stored in `gas`.

### Impact Explanation

Any caller that receives this `TransactionReceipt` — including RPC simulation, fee estimation, and tracing endpoints — will observe `receipt.fee` and `receipt.gas` that are mutually inconsistent. A client computing `receipt.gas.cost(gas_prices, tip)` will get a different value than `receipt.fee`. The discrepancy is proportional to the L1 data gas from execution state changes (storage writes, new contracts, etc.) that were included in `recommended_fee` but excluded from `receipt.gas`. This is a High-impact wrong value returned from an authoritative-looking simulation/trace path.

### Likelihood Explanation

The trigger requires:
1. A V3 invoke transaction with `AllResources` bounds.
2. Execution that succeeds (no revert error) but whose full-execution gas vector exceeds at least one resource bound.
3. The overdrafted resource must have a state-related component (e.g., L1 data gas from storage writes during execution).

All three conditions are fully attacker-controllable with no privileged access. Any user can craft an invoke that writes storage during execution and set `l1_data_gas` bounds exactly at the validation-only state cost, causing the execution state changes to push the full-execution gas over the bound.

### Recommendation

Replace the struct-update with a fresh `TransactionReceipt::from_account_tx` call that uses the same gas vector as the one `recommended_fee` was computed from, or recompute `fee` directly from `get_revert_receipt().gas` so that `fee == gas.cost(gas_prices, tip)` is maintained. The simplest fix is:

```rust
let revert_receipt = get_revert_receipt();
let fee = tx_context.tx_info.get_fee_by_gas_vector(
    &tx_context.block_context.block_info,
    revert_receipt.gas,
    tx_context.effective_tip(),
);
let tx_receipt = TransactionReceipt { fee, ..revert_receipt };
```

This ensures the fee is always derived from the same gas vector that is reported in the receipt.

### Proof of Concept

Construct a Rust unit test in `crates/blockifier/src/transaction/account_transaction.rs`:

1. Deploy a contract that writes to storage (generating L1 data gas during execution).
2. Create a V3 invoke with `AllResources` bounds where `l1_data_gas.max_amount` equals the L1 data gas from validation-only state changes (zero, since `__validate__` cannot write storage), and `l2_gas.max_amount` is set high enough to not be the limiting factor.
3. Run the transaction through `run_revertible`.
4. Assert that `receipt.fee != receipt.gas.cost(gas_prices, tip)` on the returned reverted receipt.

The execution will succeed (storage write completes), `PostExecutionReport` will detect `MaxGasAmountExceeded` on `l1_data_gas`, `recommended_fee` will be computed from `min(0, exec_l1_data_gas) = 0` for data gas but the full execution L2/L1 gas, while `receipt.gas` will reflect validation-only state (also zero data gas but different computation gas), producing a concrete mismatch.
