Audit Report

## Title
Slot-pool exhaustion via public `canister_status` blocks VIP governance calls due to broken zero-check guard — (File: rs/nervous_system/clients/src/management_canister_client.rs, rs/nns/handlers/root/impl/canister/canister.rs)

## Summary

`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` evaluates the `available_slot_count == 0` guard unconditionally before the VIP path (`used_slot_count = 0`) takes effect, so NNS Governance and SNS-W callers are rejected with `SysTransient` whenever the shared 167-slot pool is empty. The pool can be emptied by any unprivileged principal sending 167 concurrent calls to the unauthenticated `canister_status` endpoint on NNS root, temporarily blocking `update_canister_settings`, `take_canister_snapshot`, `load_canister_snapshot`, and `change_canister_controllers`.

## Finding Description

**Root cause — `try_borrow_slot`** (`rs/nervous_system/clients/src/management_canister_client.rs`, lines 264–287):

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {   // ← fires for VIPs too
                return Err((SysTransient, ...));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The intent is that VIP callers (`is_caller_vip = true`) consume zero slots and should never be rejected. However, the zero-check is evaluated inside `with_borrow_mut` before the subtraction, so when `available_slot_count` is 0 the function returns `Err` regardless of VIP status.

**Pool exhaustion path** (`rs/nns/handlers/root/impl/canister/canister.rs`, lines 88–98):

```rust
#[update]   // no auth check
async fn canister_status(canister_id_record: CanisterIdRecord) -> CanisterStatusResult {
    let client = new_management_canister_client();
    client.canister_status(canister_id_record).await...unwrap()
}
```

`canister_status` is callable by any principal. Each call borrows one slot via `try_borrow_slot` and holds it for the full round-trip to the management canister (at least one consensus round). With 167 concurrent calls the shared `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` (initialized to 167, line 50) reaches 0.

**Affected governance paths** — all call `new_management_canister_client()` and therefore go through `try_borrow_slot`:
- `update_canister_settings` (lines 220–230) — caller: NNS Governance (`is_caller_vip = true`)
- `take_canister_snapshot` (lines 244–254) — caller: NNS Governance
- `load_canister_snapshot` (lines 258–268) — caller: NNS Governance
- `change_canister_controllers` (lines 206–216) — caller: SNS-W (`is_caller_vip = true`)

**Unaffected paths** — `change_nns_canister` calls `change_canister()` directly (line 151) and `stop_or_start_nns_canister` calls `stop_canister::<CdkRuntime>` directly (line 177), bypassing `new_management_canister_client()` entirely.

**Why existing checks fail**: `new_management_canister_client()` correctly sets `is_caller_vip = true` for NNS canisters (lines 58–60), but the downstream zero-check in `try_borrow_slot` ignores this flag, negating the protection.

## Impact Explanation

This is an application/platform-level DoS of NNS governance operations not based on raw volumetric DDoS — matching the High ($2,000–$10,000) allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

During the attack window, NNS Governance proposals that execute `update_canister_settings` or snapshot operations are rejected with a transient error. SNS-W's `change_canister_controllers` is similarly blocked. While the DoS is temporary and self-limiting, an attacker can sustain it continuously by refreshing the 167-call batch, indefinitely delaying time-sensitive governance actions.

## Likelihood Explanation

- Sending 167 concurrent ingress messages is straightforward; the IC per-canister queue limit is ~500.
- NNS root is a controller of all NNS canisters, so `canister_status` calls targeting any NNS canister ID will succeed at the management canister layer, keeping slots occupied for a full consensus round (~1–2 s).
- The attacker must continuously re-submit batches as old calls complete, which costs cycles but is not prohibitive.
- No special privileges, leaked keys, or social engineering are required — any principal can call `canister_status`.
- The attack is repeatable and can be sustained indefinitely.

## Recommendation

Short-circuit `try_borrow_slot` for VIP callers before entering the zero-check:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    if self.is_caller_vip {
        return Ok(SlotLoan {
            available_slot_count: self.available_slot_count,
            used_slot_count: 0,
        });
    }
    self.available_slot_count.with_borrow_mut(|available_slot_count| {
        if *available_slot_count == 0 {
            return Err((RejectCode::SysTransient as i32,
                        "Unavailable. Maybe, try again later?".to_string()));
        }
        *available_slot_count -= 1;
        Ok(())
    })?;
    Ok(SlotLoan {
        available_slot_count: self.available_slot_count,
        used_slot_count: 1,
    })
}
```

Additionally, consider giving `canister_status` its own isolated slot pool separate from lifecycle-management operations, so public read traffic cannot interfere with governance writes.

## Proof of Concept

State-machine test outline (PocketIC or `ic-state-machine-tests`):

1. Install NNS root canister with a target NNS canister it controls.
2. From an unprivileged principal, submit 167 concurrent `canister_status` ingress messages targeting the controlled canister. Each call will borrow one slot and await the management canister response.
3. Before those calls complete (within the same round or the next), submit a `update_canister_settings` call from the simulated NNS Governance principal.
4. Assert the `update_canister_settings` response is `Err` with reject code `SysTransient` and message `"Unavailable. Maybe, try again later?"`.
5. Advance rounds until all 167 `canister_status` calls complete (slots drain back to 167).
6. Retry `update_canister_settings` from Governance and assert it succeeds.
