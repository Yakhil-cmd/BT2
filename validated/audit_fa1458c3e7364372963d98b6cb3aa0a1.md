### Title
Gateway `validate_proof_facts` rejects valid invoke-V3 transactions whose proof references block `tip - STORED_BLOCK_HASH_BUFFER + 1` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`run_validate_entry_point` advances `block_info.block_number` to `N+1` (tip + 1) before building the `BlockContext` passed to `StatefulValidator`, but the underlying state object remains at tip `N`. `validate_proof_facts` then uses `N+1` to compute the maximum allowed proof block number (`N - STORED_BLOCK_HASH_BUFFER + 1`), while `validate_proof_block_hash` reads the stored block hash from the tip-`N` state — where that hash has not yet been written. The result is a guaranteed `InvalidProofFacts` / `ValidateFailure` rejection for any invoke-V3 transaction whose proof legitimately references block `N - STORED_BLOCK_HASH_BUFFER + 1`, even though the same transaction would pass during actual block `N+1` execution.

### Finding Description

**Step 1 — block context is advanced, state is not.**

In `run_validate_entry_point`:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // N → N+1
let block_context = BlockContext::new(block_info, ...);
// state is still at tip N
let state = CachedState::new(state_reader_and_contract_manager);
``` [1](#0-0) 

**Step 2 — `perform_pre_validation_stage` calls `validate_proof_facts` with the N+1 context and the N-state.** [2](#0-1) 

**Step 3 — range check uses N+1, so `max_allowed = N - STORED_BLOCK_HASH_BUFFER + 1`.**

```rust
let max_allowed =
    current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
// current_block_number = N+1  →  max_allowed = N - STORED_BLOCK_HASH_BUFFER + 1
if proof_block_number > max_allowed { return Err(...); }
```

A proof with `proof_block_number = N - STORED_BLOCK_HASH_BUFFER + 1` passes this check. [3](#0-2) 

**Step 4 — hash lookup uses the tip-N state, where that slot is zero.**

`pre_process_block` writes the hash for block `next_block_number - STORED_BLOCK_HASH_BUFFER` when processing each block. At tip N the stored range is `[0 … N - STORED_BLOCK_HASH_BUFFER]`. Block `N - STORED_BLOCK_HASH_BUFFER + 1` has not been written yet. [4](#0-3) 

```rust
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;
// stored_block_hash == Felt::ZERO  (slot not yet populated)
if stored_block_hash != proof_block_hash {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
``` [5](#0-4) 

**Step 5 — the same transaction is valid during actual block N+1 execution.**

When block N+1 is executed, `pre_process_block` is called first and writes the hash for block `N - STORED_BLOCK_HASH_BUFFER + 1`. The OS Cairo check (`assert_nn_le(base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER)` with `current_block_number = N+1`) also allows this proof block number. [6](#0-5) 

### Impact Explanation

Every invoke-V3 transaction whose proof legitimately references block `N - STORED_BLOCK_HASH_BUFFER + 1` (the newest block for which the OS would accept a proof when executing block N+1) is rejected at the gateway with `ValidateFailure / InvalidProofFacts`. The admission decision is wrong: the transaction is valid and would succeed in the actual block, but the gateway permanently refuses it for the current tip. The user must wait for the next block to be produced and then resubmit, which breaks the client-side proving UX and constitutes a systematic, deterministic valid-tx-rejection at the boundary block number.

### Likelihood Explanation

The condition is structural, not probabilistic. Any user who generates a proof for the most recent allowed block (`tip - STORED_BLOCK_HASH_BUFFER + 1`) and submits it while the gateway tip is still at `tip` will hit this rejection every time. No special privileges or race conditions are required — only a correctly formed invoke-V3 transaction with a valid proof.

### Recommendation

In `validate_proof_facts`, use the actual tip block number (before the `unchecked_next()` increment) as `current_block_number` for the proof block number range check, so that `max_allowed` is consistent with the hashes actually present in the gateway state:

```rust
// Use tip N, not N+1, so max_allowed = N - STORED_BLOCK_HASH_BUFFER
Self::validate_proof_block_number(
    proof_block_number,
    block_context.block_info.block_number.prev()
        .expect("block number must be > 0 at gateway"),
)?;
```

Alternatively, expose the pre-increment block number separately in the block context passed to `validate_proof_facts`, or perform the proof-facts check before the `unchecked_next()` increment.
