Audit Report

## Title
ICP Permanently Locked in CMC Subaccount When Refund Fails and Block Notification Is Purged - (File: rs/nns/cmc/src/main.rs)

## Summary
The Cycles Minting Canister clears a block's notification status when `refund_icp` fails, allowing retry. However, `purge_old_notifications` is called on every `notify_*` invocation and advances `last_purged_notification` based on ledger block count. If the user's block index is purged before they retry, the CMC returns the non-retriable `TransactionTooOld` error and the ICP remains permanently locked in the CMC subaccount with no recovery path.

## Finding Description
All three notify flows (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`) share the same pattern:

1. Block status is set to `Processing` after passing the `last_purged_notification` check at `main.rs:1172-1207`.
2. The primary action is attempted. On failure, `refund_icp` is called via `?` at `main.rs:2005`, `1949`, `1976`.
3. If `refund_icp` fails (e.g., ledger mid-upgrade), it returns `NotifyError::Other { error_code: RefundFailed }` at `main.rs:2090-2096`.
4. This error propagates up. Back in `notify_*`, `is_transient_error` calls `is_retriable()` (`lib.rs:220-222`), which returns `true` for `NotifyError::Other` (only `Refunded` is non-retriable). The block is removed from `blocks_notified` at `main.rs:1219-1221`, allowing retry.
5. On every subsequent `notify_*` call by any user, `purge_old_notifications` advances `last_purged_notification`. Once the victim's block index falls below this threshold, `main.rs:1175-1178` returns `TransactionTooOld` â€” a non-retriable error.
6. The ICP remains in the CMC subaccount (`Subaccount::from(&canister_id)` for top-up, `Subaccount::from(&controller)` for create canister) with no sweep or recovery endpoint.

Existing guards are insufficient: the retry mechanism correctly clears the block status, but it does not account for the finite notification history window. There is no persistent failed-refund queue and no user-callable recovery function.

## Impact Explanation
Permanent loss of user ICP funds held in an NNS canister (CMC). The CMC is an in-scope NNS financial integration canister. The ICP enters the CMC subaccount but can never exit once the notification window expires. This matches the allowed impact: "Significant NNS/SNS or infrastructure security impact with concrete user or protocol harm" (High), and potentially "Theft, permanent loss... of in-scope ledger assets" (Critical) if the amount is large. The loss is irreversible and requires no privileged access to trigger.

## Likelihood Explanation
The attack requires four conditions to coincide: (1) primary action fails (e.g., target canister does not exist, or cycles deposit fails), (2) ICP ledger is transiently unavailable during the refund attempt (e.g., during a ledger upgrade, which occurs regularly), (3) the user does not retry before their block is purged, and (4) sufficient ledger activity advances `last_purged_notification` past the victim's block. Each condition is individually plausible; their conjunction is low-probability but non-zero and reachable by any unprivileged user. No special privileges, social engineering, or governance majority are required.

## Recommendation
1. **Add a recovery endpoint**: Expose a function callable by the original sender that transfers any remaining balance from a CMC subaccount back to the owner, callable regardless of notification history status (analogous to `error_refund_icp` in the SNS swap canister).
2. **Decouple refund retry from notification expiry**: Store failed refund tasks in a persistent queue (analogous to `pending_reimbursements` in the ckBTC minter) and process them independently of the notification history window.
3. **Do not purge blocks with pending failed refunds**: If a block's last known status was a failed refund, exclude it from `purge_old_notifications` until the refund succeeds.

## Proof of Concept
1. Alice sends 10 ICP to `AccountIdentifier(CMC, Subaccount::from(&canister_id))`. Block index = `B`.
2. Alice calls `notify_top_up(block_index: B, canister_id: X)`.
3. CMC calls `deposit_cycles(X, ...)`. Canister `X` does not exist â†’ error.
4. CMC calls `refund_icp(...)` â†’ `send_pb` on the ICP ledger. Ledger is mid-upgrade â†’ call fails.
5. `refund_icp` returns `Err(NotifyError::Other { error_code: RefundFailed })`.
6. `is_transient_error` returns `true`; block `B` is removed from `blocks_notified`. Alice can retry.
7. Alice does not retry. High ledger activity advances `last_purged_notification` past `B`.
8. Alice calls `notify_top_up(block_index: B, canister_id: X)` again.
9. CMC returns `NotifyError::TransactionTooOld` â€” non-retriable.
10. Alice's 10 ICP remains in `Subaccount::from(&canister_id)` permanently with no recovery path.

A deterministic integration test can reproduce this by: (a) deploying a mock ledger that rejects the refund `send_pb` call, (b) calling `notify_top_up` with a non-existent canister, (c) advancing `last_purged_notification` past the block index by inserting synthetic entries, and (d) asserting that a second `notify_top_up` returns `TransactionTooOld` while the CMC subaccount balance remains non-zero.
