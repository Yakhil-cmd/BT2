### Title
Unprivileged `tick()` caller can exhaust period mint cap with a partial batch, blocking full-batch processing for up to one week — (`L1/starkware/solidity/stake/RewardSupplier.sol`, `L1/starkware/solidity/stake/PeriodMintLimit.sol`)

---

### Summary

`RewardSupplier.tick()` is an unguarded public function. Because `PERIOD_MINT_CAP` equals exactly `MAX_MESSAGES_TO_PROCESS_PER_TICK × TOKENS_PER_MINT_REQUEST`, any partial consumption of the cap in a period makes a subsequent full-batch `tick()` revert with `EXCEED_PERIOD_MINTING`, leaving up to 4 messages unprocessable until the next weekly period.

---

### Finding Description

Three constants define the system's minting budget:

- `TOKENS_PER_MINT_REQUEST = 1_300_000e18` [1](#0-0) 
- `MAX_MESSAGES_TO_PROCESS_PER_TICK = 5` [2](#0-1) 
- `PERIOD_MINT_CAP = 6_500_000e18` and `MINTING_PERIOD_DURATION = 1 weeks` [3](#0-2) 

The cap is set to exactly `5 × 1_300_000e18 = 6_500_000e18` — the maximum a single full-batch `tick()` can mint. This tight equality is the root cause.

`tick()` carries no access control:

```solidity
function tick() external payable {
``` [4](#0-3) 

`requiredMinting()` always greedily returns up to 5 messages with no awareness of remaining period quota:

```solidity
uint256 numMsgsToConsume = Math.min(
    messagingContract().l2ToL1Messages(msgHash),
    MAX_MESSAGES_TO_PROCESS_PER_TICK
);
return (TOKENS_PER_MINT_REQUEST * numMsgsToConsume, numMsgsToConsume);
``` [5](#0-4) 

`checkAndUpdatePeriodicalQuota` is all-or-nothing — it either accepts the full requested amount or reverts; there is no partial-batch path:

```solidity
uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
``` [6](#0-5) 

**Attack sequence:**

1. Attacker monitors the Starknet messaging contract for pending mint-request messages.
2. When exactly 1 message is pending, attacker calls `tick()` with sufficient `msg.value`. This mints `1_300_000e18`, consuming `1/5` of the weekly cap. Period accounting: `1_300_000e18` used, `5_200_000e18` remaining.
3. L2 sends 5 more mint-request messages (normal protocol operation).
4. Any caller (or the attacker again) calls `tick()`. `requiredMinting()` returns `5 × 1_300_000e18 = 6_500_000e18`.
5. `checkAndUpdatePeriodicalQuota` evaluates `1_300_000e18 + 6_500_000e18 = 7_800_000e18 > 6_500_000e18` → **REVERT**.
6. All 5 messages remain unconsumed. The period slot resets only after `block.timestamp / 1 weeks` increments — up to 7 days later.

The same outcome occurs if the attacker processes any N < 5 messages in step 2, because `N × 1_300_000e18 + 6_500_000e18 > 6_500_000e18` for any N ≥ 1.

---

### Impact Explanation

Reward distribution to L2 stakers is delayed by up to one week per attack. The messages are not lost — they remain in the messaging contract — but they cannot be processed until the period resets. This is a **Medium griefing** impact: no profit for the attacker, but measurable damage to stakers who do not receive their rewards on schedule.

---

### Likelihood Explanation

`tick()` is callable by any EOA. The attacker's only cost is the ETH bridge fee for the initial partial `tick()`. Monitoring the Starknet messaging contract for pending messages is straightforward on-chain. The attack can be repeated every week. Likelihood is **Medium**.

---

### Recommendation

Cap the batch size inside `requiredMinting()` (or `tick()`) to the remaining period quota before computing `numMsgsToConsume`:

```solidity
// pseudocode
uint256 remainingCap = PERIOD_MINT_CAP - mintedThisPeriod(token);
uint256 maxMsgsFromCap = remainingCap / TOKENS_PER_MINT_REQUEST;
uint256 numMsgsToConsume = Math.min(
    Math.min(messagingContract().l2ToL1Messages(msgHash), MAX_MESSAGES_TO_PROCESS_PER_TICK),
    maxMsgsFromCap
);
```

This ensures `tick()` always processes as many messages as the remaining cap allows, rather than reverting when the cap is partially consumed. Alternatively, raise `PERIOD_MINT_CAP` to a value strictly greater than `MAX_MESSAGES_TO_PROCESS_PER_TICK × TOKENS_PER_MINT_REQUEST` so a single partial tick can never prevent a full batch.

---

### Proof of Concept

State-level test (Foundry/Hardhat):

```solidity
// 1. Setup: 1 pending L2→L1 mint-request message in messagingContract.
// 2. Attacker calls tick() — succeeds, mints 1_300_000e18.
//    periodMintAccounting[slot] == 1_300_000e18.
// 3. Inject 5 pending L2→L1 mint-request messages.
// 4. Call tick() — requiredMinting() returns (6_500_000e18, 5).
//    checkAndUpdatePeriodicalQuota: 1_300_000e18 + 6_500_000e18 = 7_800_000e18 > 6_500_000e18
//    → revert("EXCEED_PERIOD_MINTING")  ✓
// 5. Assert messagingContract.l2ToL1Messages(msgHash) == 5 (messages still pending).
// 6. Warp block.timestamp by 1 week; call tick() again — succeeds.
```

The 5 messages remain stuck for up to one week, confirming the griefing impact.
