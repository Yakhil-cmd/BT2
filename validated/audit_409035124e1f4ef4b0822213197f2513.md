### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions With Unverified Signatures When a Deploy-Account Is in the Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's `skip_stateful_validations` function intentionally skips the `__validate__` entry-point call for an Invoke transaction whose nonce is 1 and whose account has nonce 0 in state, provided any transaction for that address exists in the mempool. Because `__validate__` is the only place the account's signature is checked, an attacker can submit an Invoke with a completely invalid signature that passes every gateway check and enters the mempool unchallenged. The signature is only evaluated at block-execution time, mirroring the yToken pattern where the true value (here: signature validity) is deferred to a later, costlier resolution point.

### Finding Description

**Admission path (gateway)**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        ...
}
```

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations` (called via the `StatefulValidatorTrait::validate` dispatch), the early-return guard fires before `__validate__` is ever invoked:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
```

`perform_pre_validation_stage` (nonce, fee bounds, balance) still runs, but it does **not** verify the cryptographic signature. The signature is only verified inside `__validate__`, which is now skipped.

**What the gateway does check**

`validate_by_mempool` calls `mempool_client.validate_tx`, which checks for duplicate hashes and nonce ordering — not signature validity. `validate_resource_bounds` checks the L2 gas price against the previous block's price — not the signature. No other path in the gateway verifies the signature for this code branch.

**Execution path (batcher)**

When the batcher later executes the block, the deploy-account runs first (deploying the contract), and then the Invoke runs with `validate = true` (the default for sequencing). `__validate__` is called, the invalid signature is detected, and the transaction reverts. No state is corrupted, but the mempool accepted and propagated an invalid transaction.

**Attacker-controlled field and exact corrupted value**

The attacker controls the `signature` field of the Invoke transaction. The corrupted value is the admission decision: the gateway returns `Ok(account_nonce)` for a transaction whose signature would fail `__validate__`, instead of returning a `ValidateFailure` error.

### Impact Explanation

Matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An unprivileged user can inject Invoke transactions with arbitrary (invalid) signatures into the mempool. Each injected transaction consumes mempool slots, propagates to peers, and forces the batcher to attempt execution before reverting. Because the account is freshly deployed and may have minimal funds, the fee-recovery path for the reverted `__validate__` may also be incomplete, allowing near-free mempool pollution.

### Likelihood Explanation

The trigger requires two sequential, unprivileged RPC calls:
1. A valid `deploy_account` for a fresh address (normal operation, one-time cost).
2. An Invoke with `nonce = 1` and an invalid signature for the same address.

`account_tx_in_pool_or_recent_block` returns `true` as soon as the deploy-account enters the pool, so the window is open immediately. An attacker can repeat this with freshly generated salts to create many address/deploy-account pairs and flood the mempool.

### Recommendation

- In `skip_stateful_validations`, do not set `validate = false` unconditionally. Instead, run `__validate__` even for the skip-validate UX path, but tolerate a specific "account not found" execution error (which is the only legitimate failure for a not-yet-deployed account). This preserves the UX benefit while still verifying the signature.
- Alternatively, add a lightweight signature-format check (e.g., ECDSA point validity) in the stateless validator so that structurally invalid signatures are rejected before reaching the stateful path.

### Proof of Concept

```
# Step 1 – attacker submits a valid deploy_account for address X
POST /gateway/add_transaction
{ "type": "DEPLOY_ACCOUNT", "sender_address": X, "nonce": 0, ... valid sig ... }
# → admitted; mempool now contains a tx for address X

# Step 2 – attacker submits an invoke with nonce=1 and a garbage signature
POST /gateway/add_transaction
{ "type": "INVOKE", "sender_address": X, "nonce": 1,
  "signature": ["0xdeadbeef", "0xdeadbeef"], ... }

# Gateway evaluation:
#   validate_nonce:  0 <= 1 <= 200  → pass
#   validate_resource_bounds: price check against prev block → pass
#   validate_by_mempool: nonce ordering → pass
#   skip_stateful_validations:
#     tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true
#     → returns true (skip __validate__)
#   run_validate_entry_point: execution_flags.validate = false
#     → perform_pre_validation_stage runs (nonce/fee/balance), __validate__ is NOT called
#   → gateway returns Ok, tx enters mempool

# Step 3 – batcher builds block
#   deploy_account executes → account X deployed
#   invoke executes → __validate__ called → signature invalid → REVERT
#   mempool accepted an invalid transaction; batcher wasted execution resources
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
