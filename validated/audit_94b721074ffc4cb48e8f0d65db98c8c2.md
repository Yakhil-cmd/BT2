### Title
Cross-Purpose Signature Replay in `EVM.validateCOAOwnershipProof` Enables Forged ERC-1271 Signatures on COA - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that caller-supplied signatures are cryptographically valid over caller-supplied `signedData`, but never enforces that `signedData` encodes the target `evmAddress`. Any Flow account key signature produced under the `"FLOW-V0.0-user"` domain tag for *any* purpose can be replayed as a COA ownership proof for the same data, causing the COA's ERC-1271 `isValidSignature` to return `true` for messages the account owner never intended to authorize on that COA.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two checks:

1. Verifies that the supplied `signatures` are cryptographically valid over `signedData` using the account's keys and the fixed domain tag `"FLOW-V0.0-user"`.
2. Verifies that the COA resource stored at the supplied `path` in the supplied Flow `address` has an EVM address matching the supplied `evmAddress`. [1](#0-0) 

The function's own comment explicitly acknowledges the gap: [2](#0-1) 

Because `signedData` is fully attacker-controlled and is never required to encode `evmAddress`, any signature a Flow account key has ever produced over any 32-byte value `H` under the `"FLOW-V0.0-user"` tag — for *any* purpose — is a valid COA ownership proof for `H`.

The COA Solidity contract implements ERC-1271 by forwarding directly to the `verifyCOAOwnershipProof` Cadence Arch precompile: [3](#0-2) 

The precompile decodes the ABI-encoded proof and delegates to `validateCOAOwnershipProof`: [4](#0-3) 

The `COAOwnershipProof` struct (Flow address, capability path, key indices, signatures) is entirely supplied by the caller and is not itself signed: [5](#0-4) 

This means an attacker who observes a victim's Flow key signature over any hash `H` (e.g., from an off-chain login flow, a Cadence contract interaction, or any other use of the `"FLOW-V0.0-user"` tag) can construct a proof `{victim_flow_addr, coa_path, key_indices, [sig_over_H]}` and submit it to `isValidSignature(H, proof)` on the victim's COA. The function returns `0x1626ba7e` (valid).

Additionally, if a Flow account owns more than one COA, a proof constructed for COA-A can be replayed against COA-B by simply substituting the `path` field in the unsigned proof struct, since the `evmAddress` check only verifies that the COA at the given path matches the calling contract's address — it does not verify that the path was part of the signed data. [6](#0-5) 

---

### Impact Explanation

An attacker can cause `isValidSignature(H, proof)` to return `true` on a victim's COA for any hash `H` the victim has previously signed with their Flow key under the `"FLOW-V0.0-user"` domain tag. ERC-1271 is used by EVM DeFi protocols for permit-style token approvals, DEX order signing, NFT marketplace listings, and DAO voting. A forged ERC-1271 response allows the attacker to impersonate the victim's COA in any of these contexts, potentially draining EVM-side assets or executing unauthorized on-chain actions on behalf of the victim's COA.

---

### Likelihood Explanation

The `"FLOW-V0.0-user"` domain tag is the standard user-facing signing tag in Flow. Any off-chain authentication flow, dApp login, or Cadence contract that asks a user to sign a 32-byte value with this tag produces a signature that is immediately reusable as a COA ownership proof. The attacker needs only to observe one such signature (e.g., from a public off-chain authentication endpoint) and know the victim's Flow address and COA capability path (both are public on-chain). No privileged access is required. The cross-COA variant additionally requires the victim to have two COAs, which the codebase notes is historically uncommon, but the cross-purpose replay applies to any COA owner.

---

### Recommendation

Enforce that `signedData` commits to the `evmAddress` being proven. The simplest fix is to require `signedData` to be a structured message that includes the `evmAddress` bytes, e.g., by prepending a fixed context tag and the 20-byte EVM address before hashing, and verifying this structure inside `validateCOAOwnershipProof` rather than leaving it as a caller convention. Alternatively, construct `signedData` inside the function from the `evmAddress` argument so callers cannot supply arbitrary pre-signed data.

---

### Proof of Concept

**Setup**: Victim has a Flow account at address `0xALICE` with a key at index 0 and a COA stored at `/public/coa` with EVM address `0xCOA`.

**Step 1 – Victim signs an unrelated message.**
Alice signs hash `H = keccak256("login:myapp.com")` with her Flow key using the `"FLOW-V0.0-user"` domain tag for an off-chain dApp login. The signature `sig` is publicly observable (e.g., submitted to the dApp's API).

**Step 2 – Attacker constructs a replay proof.**
Attacker builds:
```
proof = RLP.encode({
    KeyIndices:     [0],
    Address:        0xALICE,
    CapabilityPath: "coa",
    Signatures:     [sig],
})
```

**Step 3 – Attacker calls `isValidSignature` on Alice's COA.**
```solidity
bytes4 result = ICOA(0xCOA).isValidSignature(H, proof);
// result == 0x1626ba7e  (ERC-1271 "valid")
```

**Step 4 – Execution path.**
- `isValidSignature` calls `cadenceArch.verifyCOAOwnershipProof(0xCOA, H, proof)`.
- The precompile calls `EVM.validateCOAOwnershipProof(0xALICE, /public/coa, H, [0], [sig], 0xCOA)`.
- `keyList.verify(signatureSet, signedData: H, domainSeparationTag: "FLOW-V0.0-user")` → `true` (sig is valid over H).
- COA address check: COA at `/public/coa` has address `0xCOA` == `evmAddress` → passes.
- Returns `ValidationResult(isValid: true, problem: nil)`.

The attacker has successfully forged an ERC-1271 signature on Alice's COA for a message she never intended to authorize on that COA. [1](#0-0) [3](#0-2) [7](#0-6)
