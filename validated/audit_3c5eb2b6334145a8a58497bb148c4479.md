### Title
Commitment Hash Mismatch Lacks Participant Attribution, Enabling Persistent DKG Denial — (`src/dkg.rs`)

### Summary

`verify_commitment_hash` detects when a participant's Round-3 commitment does not match the hash they broadcast in Round-1, but returns the generic `ProtocolError::InvalidCommitmentHash` instead of `ProtocolError::MaliciousParticipant(participant)`. Because the culprit is never identified, honest parties cannot exclude the malicious participant and retry, allowing a single bad actor to abort every DKG/reshare session indefinitely.

### Finding Description

`verify_commitment_hash` receives the offending `participant: Participant` as an explicit argument, yet on mismatch it discards that information: [1](#0-0) 

```rust
if *actual_commitment_hash != commitment_hash {
    return Err(ProtocolError::InvalidCommitmentHash);   // participant is silently dropped
}
```

The error variant `InvalidCommitmentHash` carries no participant identity: [2](#0-1) 

Compare this to every other adversarial-input check in the same file, which all embed the culprit:

- `InvalidProofOfKnowledge(Participant)` — [3](#0-2) 
- `MaliciousParticipant(participant)` — [4](#0-3) 
- `InvalidSecretShare(from)` — [5](#0-4) 

The call site in `do_keyshare` propagates the error immediately with `?`: [6](#0-5) 

This aborts the entire DKG without telling callers which participant equivocated.

### Impact Explanation

A malicious participant sends a valid `commitment_hash_A` in Round 1 and a different commitment `commitment_B` (where `H(commitment_B) ≠ commitment_hash_A`) in Round 3. Every honest party's `do_keyshare` call terminates with `Err(InvalidCommitmentHash)`. Because the error carries no identity, honest parties cannot determine who to exclude and retry without. The malicious participant can repeat this attack in every subsequent attempt, permanently blocking DKG and reshare for all honest parties.

**Impact:** High — Permanent denial of key generation / reshare for honest parties.

### Likelihood Explanation

Any participant in the protocol can trivially trigger this: send an arbitrary hash in Round 1, then send the real commitment in Round 3. No cryptographic capability is required. The attacker only needs to be an enrolled participant.

### Recommendation

Change the error returned on mismatch to include the offending participant:

```rust
// src/dkg.rs, inside verify_commitment_hash
if *actual_commitment_hash != commitment_hash {
    return Err(ProtocolError::MaliciousParticipant(participant));
}
```

This is consistent with every other adversarial-input check in `do_keyshare` and allows callers to identify and exclude the malicious participant before retrying.

### Proof of Concept

1. Run DKG with participants `[A, B, C]`, threshold 2.
2. Participant `B` (malicious) sends `commitment_hash = H(fake_commitment)` in Round 1, but sends `real_commitment` (where `H(real_commitment) ≠ commitment_hash`) in Round 3.
3. Each honest party reaches the `verify_commitment_hash` call for `B` and receives `Err(InvalidCommitmentHash)`.
4. `do_keyshare` returns `Err(InvalidCommitmentHash)` — no participant identity in the error.
5. Honest parties restart DKG; `B` repeats the attack. DKG never completes.
6. Assert: the returned error should be `Err(MaliciousParticipant(B))` so honest parties can exclude `B` and proceed with `[A, C]`.

fn verify_commitment_hash<C: Ciphersuite>(
    session_id: &HashOutput,
    participant: Participant,
    domain_separator: &mut DomainSeparator,
    commitment: &VerifiableSecretSharingCommitment<C>,
    all_hash_commitments: &ParticipantMap<'_, HashOutput>,
) -> Result<(), ProtocolError> {
    let actual_commitment_hash = all_hash_commitments.index(participant)?;
    let commitment_hash =
        domain_separate_hash(domain_separator, &(&participant, &commitment, &session_id))?;
    if *actual_commitment_hash != commitment_hash {
        return Err(ProtocolError::InvalidCommitmentHash);
    }
    Ok(())
}
