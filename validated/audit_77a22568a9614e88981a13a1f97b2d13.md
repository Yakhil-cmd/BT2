### Title
Permanently Locked Sub-Minimum attoFlow Balances in Flow EVM Accounts with No Recovery Mechanism - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

The Flow EVM integration enforces a minimum withdrawal threshold of 1e10 attoFlow (= 0.00000001 FLOW) when bridging EVM balances back to Cadence, due to the precision mismatch between EVM's 18-decimal attoFlow and Cadence's 8-decimal UFix64. Any attoFlow balance below this threshold that accumulates in an EVM account — including truncation remainders from every non-aligned withdrawal — is permanently locked in EVM state with no sweep or recovery mechanism. This is the direct Flow analog of the Comet/Bulker locked-ETH issue.

---

### Finding Description

The EVM balance system uses 18-decimal attoFlow precision internally, while Cadence `FlowToken.Vault` uses UFix64 (8-decimal precision). The conversion multiplier `UFixToAttoConversionMultiplier = 1e10` means the minimum representable unit in a Cadence vault is 1e10 attoFlow.

The withdrawal path enforces this minimum in two places:

**1. `AttoFlowBalanceIsValidForFlowVault` in `fvm/evm/types/balance.go`:**
```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
```
Any amount strictly below 1e10 attoFlow is rejected outright.

**2. Truncation in `newInternalEVMTypeWithdrawFunction` in `fvm/evm/impl/impl.go`:**
```go
value := new(big.Int).Div(amountValue.BigInt, types.UFixToAttoConversionMultiplier)
amount := types.NewBalanceFromUFix64(cadence.UFix64(value.Uint64()))
```
When a user requests withdrawal of `X` attoFlow where `X mod 1e10 != 0`, the EVM account is debited only `floor(X / 1e10) * 1e10` attoFlow. The remainder `X mod 1e10` (which is < 1e10) stays in the EVM account and **can never be withdrawn**.

**Sources of sub-minimum accumulation:**

- **Withdrawal truncation remainder**: Every withdrawal of a non-aligned amount leaves a remainder < 1e10 attoFlow permanently in the EVM account. For example, withdrawing `1230000789912345678` attoFlow deducts only `1230000780000000000` attoFlow; the `9912345678` attoFlow remainder is locked.
- **Gas fee dust at coinbase**: EVM gas fees (e.g., 21000 attoFlow for a simple transfer at gas price 1) accumulate at the coinbase address via `runWithGasFeeRefund`. If the coinbase is a COA and the accumulated fee balance is < 1e10 attoFlow, it cannot be bridged back to Cadence.
- **EVM-to-EVM transfers**: Any EVM-level transfer that results in a sub-minimum balance at the recipient cannot be recovered to Cadence.

There is **no sweep, drain, or recovery function** anywhere in the EVM contract (`fvm/evm/stdlib/contract.cdc`) or the handler (`fvm/evm/handler/handler.go`) that would allow recovery of sub-minimum balances.

---

### Impact Explanation

This is a **cross-VM asset loss** vulnerability. FLOW tokens deposited into the EVM environment (via `EVMAddress.deposit` or `CadenceOwnedAccount.deposit`) can become permanently irrecoverable if the resulting balance falls below 1e10 attoFlow. The locked tokens exist in EVM state but cannot be bridged back to Cadence. There is no admin sweep function, no protocol-level recovery path, and no upgrade mechanism for the EVM state trie that would allow recovery.

Per-transaction the locked amount is small (< 0.00000001 FLOW), but the loss is **cumulative and permanent** across all users and all time. Every non-aligned withdrawal by every COA owner contributes dust that is locked forever. Additionally, any EVM account that receives only sub-minimum amounts (e.g., from gas fee distribution) has its entire balance permanently locked.

---

### Likelihood Explanation

**Certain** — the truncation remainder is a structural consequence of the precision mismatch. Any user who calls `CadenceOwnedAccount.withdraw(balance:)` with an attoFlow amount that is not a multiple of 1e10 will leave a locked remainder. This includes:
- Any EVM transaction sender whose gas fees are not a multiple of 1e10 attoFlow
- Any COA owner who withdraws a non-aligned amount (which is the common case when EVM balances are set by EVM-level operations)

The entry path requires only an unprivileged Cadence transaction calling `EVM.createCadenceOwnedAccount()`, depositing FLOW, and calling `withdraw(balance:)` with a non-aligned amount — all publicly accessible operations.

---

### Recommendation

1. **Add a `sweepDust` or `withdrawAll` function** to `CadenceOwnedAccount` that withdraws the maximum withdrawable amount (i.e., `floor(balance / 1e10) * 1e10` attoFlow) and discards or burns the sub-minimum remainder, analogous to the sweep functions added to the Bulker contract in the referenced fix.
2. **Track and aggregate sub-minimum remainders** at the protocol level (e.g., in a dedicated dust account) so they can be periodically swept by governance.
3. **Document the permanent loss** explicitly in the `withdraw` function's NatSpec so users understand that non-aligned withdrawals result in irrecoverable dust.

---

### Proof of Concept

The following Cadence transaction demonstrates the locked remainder:

```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin
        )!
        let minter <- admin.createNewMinter(allowedAmount: 2.34)
        let vault <- minter.mintTokens(amount: 2.34)
        destroy minter

        let coa <- EVM.createCadenceOwnedAccount()
        coa.deposit(from: <-vault)
        // EVM balance: 2340000000000000000 attoFlow

        // Withdraw a non-aligned amount
        let bal = EVM.Balance(attoflow: 1230000789912345678)
        let vault2 <- coa.withdraw(balance: bal)
        // Vault returned: 1230000780000000000 attoFlow (truncated)
        // EVM balance after: 2340000000000000000 - 1230000780000000000 = 1109999220000000000 attoFlow
        // BUT: 9912345678 attoFlow of the requested amount was NEVER deducted
        //      because the withdraw only deducted the truncated amount.
        // The 9912345678 attoFlow remainder from truncation is permanently locked
        // and cannot be withdrawn (below 1e10 minimum).

        destroy coa
        destroy vault2
    }
}
```

This is confirmed by the existing test `"test coa withdraw with remainder truncation"` in `fvm/evm/evm_test.go` (lines 2622–2692), which asserts that `balanceAfterInAttoFlow` is `1_109_999_220_000_000_000` — meaning the `9912345678` attoFlow sub-minimum remainder is silently left in the EVM account with no recovery path. // runWithGasFeeRefund runs a method and transfers the balance changes of the
// coinbase address to the provided gas fee collector
func (h *ContractHandler) runWithGasFeeRefund(gasFeeCollector types.Address, f func()) {
	// capture coinbase init balance
	cb := h.AccountByAddress(types.CoinbaseAddress, true)
	initCoinbaseBalance := cb.Balance()
	f()
	// transfer the gas fees collected to the gas fee collector address
	afterBalance := cb.Balance()
	diff := new(big.Int).Sub(afterBalance, initCoinbaseBalance)
	if diff.Sign() > 0 {
		cb.Transfer(gasFeeCollector, diff)
	}
	if diff.Sign() < 0 { // this should never happen but in case
		panic(fvmErrors.NewEVMError(fmt.Errorf("negative balance change on coinbase")))
	}
}
