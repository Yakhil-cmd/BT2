### Title
`#[payable]` `execute_scheduled` Accepts NEAR Deposits With No Withdrawal Path, Permanently Freezing Funds - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router contract's `execute_scheduled` function is marked `#[payable]`, allowing any caller to attach NEAR tokens to the call. The function never reads `env::attached_deposit()` and the router contract has no general withdrawal mechanism. Any NEAR attached to an `execute_scheduled` call is permanently locked in the router sub-account.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, the `execute_scheduled` function is annotated `#[payable]`:

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The function body never calls `env::attached_deposit()`. The attached NEAR is silently accepted and left in the router account's balance. The scheduled `PromiseArgs` deserialized from storage carries its own pre-set `attached_balance` field; the runtime deposit is entirely separate and is never forwarded.

The only fund-recovery path in the contract is `send_refund`, which transfers a hard-coded constant `REFUND_AMOUNT` (2 NEAR) back to the parent and is restricted to the parent caller:

```rust
const REFUND_AMOUNT: NearToken = NearToken::from_near(2);

pub fn send_refund(&self) -> Promise {
    let parent = self.get_parent().unwrap_or_else(env_panic);
    require_caller(&parent)
        .and_then(|_| require_no_failed_promises())
        .unwrap_or_else(env_panic);
    Promise::new(parent).transfer(REFUND_AMOUNT)
}
```

There is no function that transfers an arbitrary amount of NEAR out of the router, and no admin escape hatch for excess balance.

The code comment explicitly states `execute_scheduled` is open to any caller:

> "It is intentional that this function can be called by anyone (not just the parent). There is no security risk to allowing this function to be open because it can only act on promises that were created via `schedule`."

This is structurally identical to the reported `receive()` pattern: a publicly reachable entry point that accepts funds with no corresponding withdrawal path.

---

### Impact Explanation

Any NEAR attached to `execute_scheduled` is permanently frozen in the router sub-account (e.g., `<evm_address>.aurora`). There is no function in the router contract that can drain an arbitrary balance. The only recovery would require the Aurora engine admin to delete the router sub-account entirely, which destroys the XCC state for that EVM address and is a destructive, non-routine operation. The frozen funds cannot be recovered by the depositor through any normal protocol path.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

`execute_scheduled` is callable by any NEAR account. A user who schedules a cross-contract call that requires attached NEAR for the target call may reasonably (but incorrectly) attach NEAR to `execute_scheduled` rather than embedding it in the `PromiseArgs` at schedule time. The `#[payable]` annotation actively signals to callers that a deposit is accepted, making accidental loss realistic. A malicious actor could also deliberately send NEAR to any router sub-account to grief that address's router balance accounting.

---

### Recommendation

1. Remove the `#[payable]` attribute from `execute_scheduled` if no deposit is needed. In NEAR SDK, a non-`#[payable]` function panics if any deposit is attached, preventing accidental loss.
2. If a deposit is intentionally accepted (e.g., to forward NEAR to the scheduled promise), explicitly read `env::attached_deposit()` and incorporate it into the promise's `attached_balance`.
3. Add a general `withdraw_excess` function restricted to the parent that transfers any balance above the minimum storage requirement back to the engine.

---

### Proof of Concept

1. Alice's EVM address `0xABCD…` has a router deployed at `abcd….aurora`.
2. Alice schedules a promise via the XCC precompile (no NEAR required for the target call).
3. Bob (any NEAR account) calls `execute_scheduled` on `abcd….aurora` with `1 NEAR` attached.
4. The function succeeds: the scheduled promise is executed, and `1 NEAR` is silently absorbed into the router's balance.
5. No function in the router contract can return that `1 NEAR` to Bob or to Alice.
6. `send_refund` is restricted to the parent and only transfers the fixed `REFUND_AMOUNT` (2 NEAR), not the excess.
7. The `1 NEAR` is permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2)
