Audit Report

## Title
Arbitrary `buyer` Field in `refresh_buyer_tokens` Allows Third-Party Forced Participation and `confirmation_text` Consent Bypass — (File: `rs/sns/swap/canister/canister.rs`)

## Summary

The SNS Swap canister's `refresh_buyer_tokens` update endpoint resolves the effective buyer from a caller-controlled payload field rather than the authenticated ingress sender. Any unprivileged principal can therefore register any other principal as a swap participant — including satisfying the swap's mandatory `confirmation_text` consent gate on that principal's behalf — without that principal's knowledge or consent. A victim who has already transferred ICP to their swap subaccount (the normal first step of participation) can be forcibly registered before they choose to participate, with their ICP locked and the swap recording that they accepted the confirmation text.

## Finding Description

In `rs/sns/swap/canister/canister.rs` lines 128–143, the handler resolves the effective buyer from the caller-controlled `arg.buyer` field:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller_principal_id()
    } else {
        PrincipalId::from_str(&arg.buyer).unwrap()   // ← arbitrary, caller-controlled
    };
    ...
    swap_mut()
        .refresh_buyer_token_e8s(p, arg.confirmation_text, this_canister_id(), &icp_ledger)
        .await
}
```

No check is performed to verify that the supplied `buyer` matches the actual ingress sender. The `RefreshBuyerTokensRequest.buyer` field is documented as "If not specified, the caller is used" (`rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` lines 843–851), confirming the field is intentionally caller-supplied.

Inside `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` lines 1134–1310), the swap canister:

1. Calls `validate_confirmation_text` (lines 363–384), which only performs a string equality check against the swap's configured text — it does **not** verify that the caller is the same principal as `buyer`. The confirmation text is stored in the public SNS init payload and is readable by anyone.
2. Queries the ICP ledger balance of `Account { owner: swap_canister_id, subaccount: principal_to_subaccount(&buyer) }` — where `buyer` is the attacker-supplied value.
3. If the balance meets `min_participant_icp_e8s`, registers `buyer` as a participant in `self.buyers` (lines 1285–1288) and records acceptance of the confirmation text — attributed to `buyer`, not to the actual caller.

The `buyers` map is keyed by the supplied `buyer` string, and `min_participation_reached` counts `self.buyers.len()` (lines 2801–2821), meaning attacker-registered entries directly affect swap outcome thresholds.

**Attack Path A — zero-cost forced registration:**
1. Victim Bob transfers `min_participant_icp_e8s` ICP to the swap's subaccount derived from his own principal (the normal first step of participation).
2. Before Bob calls `refresh_buyer_tokens` himself, attacker Alice calls it with `buyer = Bob` and `confirmation_text = <swap's required text>`.
3. Bob is registered as a participant; his ICP is locked; the swap records that Bob accepted the confirmation text — without Bob ever sending a message to the swap canister.

**Attack Path B — attacker-funded forced registration:**
1. Alice transfers `min_participant_icp_e8s` ICP to the swap's subaccount for any target principal T.
2. Alice calls `refresh_buyer_tokens` with `buyer = T` and `confirmation_text = <swap's required text>`.
3. T is registered as a participant and recorded as having consented.

Once registered, a buyer's ICP is locked in escrow. `error_refund_icp` is only available after the swap reaches ABORTED or COMMITTED state (lines 1931–1936), and if the swap commits, the victim's ICP is swept to the SNS governance canister and they receive SNS neurons — with no path to recover their ICP.

## Impact Explanation

This is a **High** severity finding. The `confirmation_text` mechanism is the SNS's explicit informed-consent gate, ensuring each participant has read and agreed to the swap's terms before their ICP is committed. Any caller can satisfy this check on behalf of any other principal by supplying the publicly readable confirmation text string, rendering the consent gate ineffective. A victim who has pre-funded their swap subaccount (the standard first step of participation) can have their ICP locked and their consent recorded without ever sending a message to the swap canister. If the swap subsequently commits, the victim's ICP is permanently converted to SNS neurons without their explicit agreement. This constitutes a significant SNS security impact with concrete user harm — unauthorized commitment of user funds and bypass of the protocol's consent mechanism — matching the High impact class ($2,000–$10,000): "Significant SNS or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

`refresh_buyer_tokens` is a public, unauthenticated update call reachable by any ingress sender. No role, key, or privileged access is required. Attack Path A costs the attacker nothing beyond a single update call; it only requires that the victim has already transferred ICP to their subaccount, which is the standard and publicly observable first step of participation. The confirmation text is stored in the public SNS init payload and is readable by any observer. The attack is repeatable against any victim who has pre-funded their subaccount and has not yet called `refresh_buyer_tokens` themselves.

## Recommendation

Enforce that when `arg.buyer` is non-empty and specifies a principal different from the caller, the caller must not be permitted to supply `confirmation_text` on that principal's behalf:

```rust
if !arg.buyer.is_empty() {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id() && arg.confirmation_text.is_some() {
        ic_cdk::trap("confirmation_text may not be submitted on behalf of another principal");
    }
}
```

Alternatively, require that `buyer`, when non-empty, must equal the caller, eliminating the third-party notification path entirely and aligning with the pattern used by other IC canisters that explicitly verify caller-vs-beneficiary identity.

## Proof of Concept

A deterministic unit test can be added to `rs/sns/swap/tests/swap.rs` following the pattern of `test_swap_participation_confirmation`:

```rust
#[test]
fn test_third_party_can_bypass_confirmation_text_for_another_buyer() {
    let confirmation_text = "I confirm I have read the terms.".to_string();
    let victim = PrincipalId::new_user_test_id(1);
    let attacker = PrincipalId::new_user_test_id(2);
    let amount = 101 * E8;

    let mut swap = SwapBuilder::new()
        .with_lifecycle(Open)
        .with_confirmation_text(confirmation_text.clone())
        .build();

    // Attacker calls refresh_buyer_token_e8s with victim's principal and the correct confirmation text.
    // The ledger reports that victim's subaccount holds `amount`.
    let result = swap.refresh_buyer_token_e8s(
        victim,  // buyer = victim, but caller would be attacker in real ingress
        Some(confirmation_text.clone()),
        SWAP_CANISTER_ID,
        &mock_stub(vec![LedgerExpect::AccountBalance(
            Account {
                owner: SWAP_CANISTER_ID.get().into(),
                subaccount: Some(principal_to_subaccount(&victim)),
            },
            Ok(Tokens::from_e8s(amount)),
        )]),
    )
    .now_or_never()
    .unwrap();

    // Victim is registered as a participant with confirmation text accepted,
    // without victim ever sending a message to the swap canister.
    assert!(result.is_ok());
    assert!(swap.buyers.contains_key(&victim.to_string()));
}
```

This test will pass against the current code, demonstrating that the confirmation text consent check is not tied to the authenticated caller identity.
