### Title
`setPoolAdminFeeDestination` Does Not Collect Accrued Fees Before Updating Destination, Enabling Pool Admin to Redirect Earned Admin Fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` immediately without first flushing accrued admin fees to the old destination. Every other fee-state-mutating function in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `pool.collectFees(…)` with the old configuration before writing new values. `setPoolAdminFeeDestination` omits this step, so all spread-surplus and notional fees that accrued under the old destination are silently redirected to the new one on the next `collectPoolFees` call.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the collect-then-update pattern: [1](#0-0) [2](#0-1) 

`setPoolAdminFeeDestination` does not: [3](#0-2) 

`collectFees` on the pool uses the current `adminFeeDestination_` argument (supplied by the factory from `poolAdminFeeDestination[pool]`) to route both the spread-surplus admin share and the notional-fee admin share: [4](#0-3) 

Because `poolAdminFeeDestination[pool]` is overwritten before any flush, the next `collectPoolFees` call (permissionless, callable by anyone) delivers all previously accrued admin fees to the new address.

---

### Impact Explanation

The pool admin can atomically:
1. Observe that a large amount of admin fees has accrued (spread surplus + `notionalFeeToken0/1Scaled`).
2. Call `setPoolAdminFeeDestination(pool, attackerAddress)`.
3. Call `collectPoolFees(pool)` (or let a keeper do it).

All admin fees that accrued while the original `adminFeeDestination` was set are transferred to `attackerAddress`. The original fee destination — which may be a separate DAO treasury or LP-fee-sharing contract — receives nothing for that period. The loss is bounded by the total admin-fee share of all swap volume since the last collection, which in an active pool can be substantial.

---

### Likelihood Explanation

The pool admin is semi-trusted and has unilateral, timelock-free authority to call `setPoolAdminFeeDestination`. No two-step confirmation is required. `collectPoolFees` is permissionless, so the admin does not even need to be the one who triggers the final transfer — a keeper or bot will do it. The attack requires only one transaction from the pool admin and is trivially executable whenever fees have accrued.

---

### Recommendation

Mirror the collect-then-update pattern used by `setPoolAdminFees` and `setPoolProtocolFee`. Before overwriting `poolAdminFeeDestination[pool]`, flush accrued fees to the old destination:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before updating.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Deploy a pool with `admin = Alice`, `adminFeeDestination = Treasury`.
2. Execute swaps; spread surplus and notional fees accrue in the pool.
3. Alice calls `factory.setPoolAdminFeeDestination(pool, Alice)`.
   — `poolAdminFeeDestination[pool]` is now `Alice`; no fees collected yet.
4. Anyone calls `factory.collectPoolFees(pool)`.
   — `collectFees` is invoked with `adminFeeDestination_ = Alice`.
   — All admin fees (spread + notional) flow to `Alice`.
5. `Treasury` receives zero fees for the entire accrual period, despite being the designated recipient throughout that time.
