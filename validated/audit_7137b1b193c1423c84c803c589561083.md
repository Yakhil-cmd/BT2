### Title
Unchecked Precompile Call Return Value in `withdrawToNear` / `withdrawToEthereum` Burns ERC-20 Tokens Without Completing Withdrawal — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear()` and `EvmErc20.withdrawToEthereum()` burn the caller's ERC-20 tokens first, then invoke the exit precompile via a low-level assembly `call`. The return value of that `call` is captured in a local variable `res` but is **never inspected**. If the precompile call fails at the EVM level (returns `0`), the burn is already committed and the tokens are permanently destroyed with no withdrawal ever scheduled.

---

### Finding Description

In `etc/eth-contracts/contracts/EvmErc20.sol`, both withdrawal functions follow the same pattern:

1. `_burn(_msgSender(), amount)` — irreversibly destroys the caller's ERC-20 balance.
2. A low-level assembly `call` is made to the exit precompile address.
3. The `call` return value `res` is stored but **never checked**; there is no `if res == 0 { revert }` guard.

```solidity
// withdrawToNear (lines 53–63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}

// withdrawToEthereum (lines 65–76)
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

In EVM semantics, a failed `call` (including a failed precompile call) returns `0` to the caller but does **not** revert the caller's execution. Because neither function checks `res`, the `_burn` is finalized even when the precompile returns failure.

The exit precompile (`engine-precompiles/src/native.rs`) can fail for several reasons: the engine is paused, the input is rejected, gas is exhausted inside the precompile, or an internal error occurs. Any such failure causes the `call` opcode to return `0`, but the Solidity code proceeds as if the withdrawal succeeded. [2](#0-1) 

---

### Impact Explanation

**Critical — Permanent freezing / theft of funds.**

When the precompile call fails silently:
- The user's ERC-20 token balance is permanently reduced to zero (burned).
- No NEAR promise is scheduled, so no NEP-141 tokens are transferred to the NEAR recipient and no ETH is sent to the Ethereum recipient.
- The underlying bridged value is irrecoverably lost; there is no refund path because the `exit_to_near_precompile_callback` refund mechanism is only triggered when the *async promise* fails, not when the synchronous precompile `call` itself fails. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The precompile call can fail under realistic conditions:

- The engine or the exit precompile is paused (pausability is an explicit feature of the system).
- The precompile encounters an internal error (e.g., serialization failure, missing connector account ID, or gas exhaustion inside the precompile).
- Any ERC-20 token holder can trigger this path by calling `withdrawToNear` or `withdrawToEthereum` — no special privilege is required. [4](#0-3) 

---

### Recommendation

After the assembly `call`, check `res` and revert if it is `0`:

```solidity
assembly {
    let res := call(gas(), EXIT_TO_NEAR_PRECOMPILE, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile call fails, the entire transaction reverts (including the `_burn`), so the user's tokens are never destroyed without a corresponding withdrawal being initiated.

---

### Proof of Concept

1. Deploy `EvmErc20` and give `alice` a balance of `N` tokens.
2. Pause the exit precompile (or arrange for it to fail, e.g., by calling with a gas limit that causes the precompile to run out of gas internally).
3. `alice` calls `withdrawToNear(recipient, N)`.
4. `_burn(alice, N)` executes — alice's balance drops to `0`.
5. The assembly `call` to the precompile returns `0` (failure).
6. Because `res` is not checked, the function returns normally.
7. Alice's ERC-20 balance is `0`; no NEP-141 tokens arrive at `recipient`; the value is permanently lost. [1](#0-0)
