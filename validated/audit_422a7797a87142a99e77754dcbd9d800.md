### Title
Hardcoded `chain_id = 1u64` in Ethereum Execution Mode Enables Cross-Chain Transaction Replay and CHAINID Opcode Mismatch - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`)

---

### Summary

In the Ethereum execution mode of ZKsync OS, the `chain_id` used for EIP-155 transaction validation and the `CHAINID` opcode is hardcoded to `1u64` (Ethereum mainnet) inside `HeaderAndHistory::new()`, rather than being derived from the oracle-provided block header. This is a direct analog to the Linea `GENESIS_SHNARF` hardcoding issue: a critical initialization value is fixed at compile time instead of being parameterized, preventing correct operation on any non-mainnet EVM chain and enabling cross-chain transaction replay.

---

### Finding Description

In `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`, the `HeaderAndHistory::new()` constructor unconditionally sets:

```rust
let chain_id = 1u64;
``` [1](#0-0) 

This value is stored in the `HeaderAndHistory` struct and returned by its `BasicBlockMetadata::chain_id()` implementation: [2](#0-1) 

The `chain_id` flows into two critical paths:

**Path 1 — CHAINID opcode:** `system.get_chain_id()` delegates to `metadata.chain_id()`, which returns the hardcoded `1`. The EVM interpreter's `chainid()` instruction pushes this value onto the stack: [3](#0-2) 

**Path 2 — EIP-155 transaction validation:** `RlpEncodedTxInner::parse_and_compute_signed_hash()` receives `expected_chain_id` and enforces it against every typed transaction (EIP-2930, EIP-1559, EIP-4844, EIP-7702) and legacy EIP-155 transactions: [4](#0-3) [5](#0-4) 

For legacy EIP-155 transactions, the `expected_chain_id` is also used to compute the signing hash itself: [6](#0-5) 

The block header is fully parsed from the oracle (including all fields), but the `chain_id` is never extracted from it — it is simply overwritten with the literal `1u64`: [7](#0-6) 

---

### Impact Explanation

When the Ethereum execution mode is used on any EVM chain with `chain_id ≠ 1`:

1. **Transaction replay attack**: The system accepts any transaction signed for Ethereum mainnet (`chain_id = 1`). An attacker can take a valid Ethereum mainnet transaction and submit it to the non-mainnet ZKsync OS deployment. Because `expected_chain_id` is always `1`, the chain_id check passes and the transaction is executed — a classic EIP-155 replay.

2. **CHAINID opcode mismatch (EVM semantic mismatch)**: Smart contracts that use `CHAINID` for replay protection (e.g., EIP-712 domain separators, permit functions) will always observe `1` regardless of the actual network. This breaks all chain-specific signature verification logic inside contracts, allowing signatures crafted for Ethereum mainnet to be accepted on the target chain.

3. **Denial of service for legitimate users**: All EIP-155 protected transactions signed for the actual chain (`chain_id ≠ 1`) are rejected with `InvalidChainId`, making the chain unusable for its intended users.

---

### Likelihood Explanation

The Ethereum execution mode is designed to be a general-purpose ZK-EVM prover. The `HeaderAndHistory` struct already reads all other block parameters (gas limit, base fee, blob gas, block hashes) from the oracle — the `chain_id` is the sole field that is hardcoded. Any deployment of this mode on a non-mainnet EVM chain (testnet, L2, or alternative L1) immediately triggers the vulnerability. The attacker entry path requires only submitting a standard Ethereum mainnet transaction — no privileged access, no governance, no oracle manipulation.

---

### Recommendation

Replace the hardcoded literal with a value derived from the oracle-provided block header or a dedicated oracle query, consistent with how `BlockMetadataFromOracle` handles `chain_id` in the ZK execution mode: [8](#0-7) 

Concretely, in `HeaderAndHistory::new()`, remove `let chain_id = 1u64;` and instead read the chain ID from the oracle (analogous to `ETHEREUM_TARGET_HEADER_BUFFER_DATA_QUERY_ID`) or extract it from the parsed `PectraForkHeader` by adding a `chain_id` field to that struct and populating it from the oracle-provided block context.

---

### Proof of Concept

1. Deploy ZKsync OS Ethereum execution mode targeting a chain with `chain_id = 42` (e.g., a testnet).
2. Craft an EIP-1559 transaction signed for Ethereum mainnet (`chain_id = 1`): `tx.chain_id = 1`.
3. Submit the transaction to the ZKsync OS node.
4. In `RlpEncodedTxInner::parse_and_compute_signed_hash`, the check `if tx.chain_id != expected_chain_id` evaluates as `1 != 1` → **false** — the transaction passes chain ID validation.
5. The transaction is executed on the `chain_id = 42` network using a signature that was never authorized for that chain.
6. Simultaneously, any contract calling `CHAINID` receives `1` instead of `42`, breaking all EIP-712 domain separators and permit-style signatures that include the chain ID.
