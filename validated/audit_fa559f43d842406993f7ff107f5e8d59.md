### Title
Signature Verification Bypass via `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures into Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary
The `skip_stateful_validations` function unconditionally skips the `__validate__` entry point (the account's signature-verification step) for any Invoke transaction with nonce=1 when the account nonce is 0 and the account has any transaction in the mempool or a recently committed block. An unprivileged attacker who observes the public mempool can submit an Invoke transaction carrying an arbitrary/invalid signature from any account that has a pending `deploy_account`, and the gateway will admit it to the mempool without ever verifying the signature.

### Finding Description
In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` (lines 429–461) returns `true` when three conditions hold simultaneously:

1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)` (nonce = 1)
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in committed state)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = !skip_validate = false`: [2](#0-1) 

This `AccountTransaction` is then passed to `blockifier_validator.validate(account_tx)`, which calls `StatefulValidatorTrait::validate` → `perform_validations`. Inside `perform_validations`, the Invoke branch checks `if !tx.execution_flags.validate { return Ok(()); }` and exits **before** calling the `__validate__` entry point: [3](#0-2) [4](#0-3) 

The result is that `validate_tx` (which calls the account's `__validate__` Cairo entry point and verifies the signature) is never reached: [5](#0-4) 

The guard `account_tx_in_pool_or_recent_block` checks `self.state.contains_account || self.tx_pool.contains_account`. Crucially, `self.state` retains an account address **after** its `deploy_account` has been committed to a block (confirmed by the test `tx_from_address_exists` and `mempool_state_retains_address_across_api_calls`): [6](#0-5) [7](#0-6) 

This means the bypass window is not limited to the brief moment the `deploy_account` is in the pool; it persists for as long as the mempool's committed-nonce state retains the address.

The comment in the code acknowledges the intent ("It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations"), but this reasoning does not account for an adversary who deliberately submits an Invoke with a forged signature targeting a known-pending or recently-deployed account.

### Impact Explanation
An attacker can submit an Invoke transaction with a completely arbitrary signature from any account that has (or recently had) a `deploy_account` in the mempool. The gateway admits the transaction without invoking `__validate__`, so the account's signature-verification logic is never executed at the admission layer. The transaction enters the mempool as if it were legitimate. This directly matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The transaction will ultimately fail at batcher execution time (the batcher runs `__validate__` with `validate=true`), but the admission invariant—that every transaction in the mempool carries a signature that has been verified against the account's own `__validate__` logic—is broken.

### Likelihood Explanation
The mempool is public. Any observer can enumerate accounts with pending `deploy_account` transactions (nonce=0 in state, tx in pool). The attack requires no special privilege: submit an Invoke with nonce=1 and any signature bytes. The condition is met during every normal account-deployment flow, making the attack surface continuously available.

### Recommendation
Even when the UX skip is desired, the gateway should perform an independent, lightweight signature check before admitting the transaction. One approach: run a stateless ECDSA/Stark-curve verification against the transaction hash and the provided signature before calling `skip_stateful_validations`, so that at minimum the cryptographic signature is valid even if the full `__validate__` entry point is deferred. Alternatively, restrict the skip to only the blockifier's `perform_pre_validation_stage` (nonce/fee checks) while still invoking `__validate__` with a read-only state snapshot of the not-yet-deployed account class.

### Proof of Concept
1. Attacker observes the public mempool and identifies account `A` with a pending `deploy_account` transaction (account nonce = 0 in committed state, `deploy_account` in pool).
2. Attacker constructs an `RpcInvokeTransactionV3` from address `A` with `nonce = 1` and a completely arbitrary `signature` field (e.g., all zeros).
3. Attacker submits the transaction to the gateway's `add_tx` endpoint.
4. Gateway flow:
   - `StatelessTransactionValidator::validate` passes (signature length within limit, resource bounds non-zero, DA modes L1).
   - `convert_rpc_tx_to_internal_and_executable_txs` succeeds (no class compilation needed for Invoke).
   - `extract_state_nonce_and_run_validations` reads `account_nonce = 0` from state.
   - `validate_state_preconditions`: nonce check passes (`0 ≤ 1 ≤ 0 + max_gap`); resource bounds pass.
   - `validate_by_mempool`: mempool accepts nonce=1 with account_nonce=0 (gap allowed).
   - `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
   - `run_validate_entry_point` called with `skip_validate=true` → `execution_flags.validate=false` → `perform_validations` returns `Ok(())` at line 80 without calling `__validate__`.
5. Transaction is forwarded to the mempool via `mempool_client.add_tx(...)` and admitted.
6. The invalid-signature Invoke now sits in the mempool alongside the legitimate `deploy_account`. [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
