### Title
Gateway `skip_stateful_validations` bypasses `__validate__` for any account with a pending mempool transaction, not only deploy-account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry point for the deploy-account + invoke UX pattern (account not yet on-chain). Its guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** pending transaction, not only a `DeployAccount`. An attacker who observes that account A has any pending nonce-0 transaction in the mempool can submit a forged invoke (nonce=1, arbitrary signature) for account A; the gateway admits it without running `__validate__`, injecting an invalid transaction into the mempool.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The function's own doc-comment states the intent is to handle the case where a **deploy-account** has not been processed yet:

> "Check if validation of an invoke transaction should be skipped due to deploy_account not being processed yet." [2](#0-1) 

However, `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction associated with the address: [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

`perform_validations` then returns early after `perform_pre_validation_stage`, never calling the `__validate__` entry point: [5](#0-4) 

`perform_pre_validation_stage` checks nonce and balance but does **not** verify the account signature: [6](#0-5) 

The mempool's own `validate_tx` path also does not verify signatures — it only checks nonce gaps and duplicate tx hashes: [7](#0-6) 

**Attack path:**

1. Account A has on-chain nonce = 0.
2. Account A submits a legitimate invoke (nonce = 0) → admitted to mempool.
3. Attacker observes this via P2P propagation or RPC.
4. Attacker crafts `Invoke { sender = A, nonce = 1, calldata = <arbitrary>, signature = <garbage> }`.
5. Gateway stateful path:
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: no duplicate hash, nonce gap OK → passes.
   - `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block` = true (account A has the nonce-0 invoke in pool) → returns `true`.
   - `run_validate_entry_point`: `validate = false` → `__validate__` skipped entirely.
6. Forged transaction is admitted to the mempool without any signature check.

During actual block execution the batcher creates `AccountTransaction::new_for_sequencing` with `validate: true`, so `__validate__` is called and the forged transaction fails with no state change. However, the invalid transaction has already been admitted to the mempool and the sequencer has wasted resources attempting to execute it.

---

### Impact Explanation

An attacker can inject signature-invalid invoke transactions into the mempool for **any** account that has a pending nonce-0 transaction. This directly matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The forged transactions pollute the mempool, consume sequencer execution resources, and may delay or displace legitimate transactions from the same account.

---

### Likelihood Explanation

- Pending mempool transactions are observable via P2P gossip and RPC.
- The only prerequisite is that the target account has any pending transaction with nonce = 0 while its on-chain nonce is also 0 — a common state for new accounts.
- No privileged access is required; any unprivileged network participant can trigger this.

---

### Recommendation

Change the guard in `skip_stateful_validations` to verify that the transaction in the pool is specifically a `DeployAccount`, not just any transaction. For example, query the mempool for a deploy-account transaction for the given address rather than using the generic `account_tx_in_pool_or_recent_block` check. [8](#0-7) 

---

### Proof of Concept

```
// State: account 0xABCD has on-chain nonce = 0.
// Step 1: 0xABCD submits a valid invoke (nonce=0) → mempool accepts it.

// Step 2: Attacker submits:
RpcInvokeTransaction {
    sender_address: 0xABCD,
    nonce: 1,
    calldata: [<drain funds or arbitrary>],
    signature: [0x1, 0x2],   // garbage, never checked
    resource_bounds: <valid>,
    ...
}

// Gateway flow:
// validate_nonce(account_nonce=0, tx_nonce=1) → OK (within gap)
// validate_by_mempool → OK (no dup hash)
// skip_stateful_validations:
//   tx.nonce() == 1 ✓
//   account_nonce == 0 ✓
//   account_tx_in_pool_or_recent_block(0xABCD) == true ✓  ← triggered by nonce-0 invoke
//   → returns true (skip __validate__)
// run_validate_entry_point: validate=false → returns Ok without calling __validate__

// Result: forged transaction admitted to mempool.
// During execution: __validate__ called with validate=true → fails, tx rejected, no state change.
// Net effect: invalid tx in mempool, sequencer resources wasted.
```

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
