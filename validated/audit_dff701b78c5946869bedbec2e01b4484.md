### Title
Missing Validation of `app_pk` Identity Element in CKD Protocol Allows Coordinator to Extract Confidential Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd` function in `src/confidential_key_derivation/protocol.rs` accepts an `app_pk` (application public key) parameter without validating that it is not the group identity element. If `app_pk` is the identity, the ElGamal encryption layer in `compute_signature_share` degenerates: `big_c = big_s + identity * y = big_s`, exposing the raw BLS signature share. The coordinator then aggregates these unencrypted shares and directly obtains the confidential derived key `H(app_id) * msk` without needing the application's secret key. This is a direct analog to the external report's vulnerability class: a missing check on an input value that should never be a specific degenerate case (zero/identity).

### Finding Description
In `compute_signature_share`, each participant computes an ElGamal-style encryption of their BLS signature share:

```
big_s  = H(app_id || pk) * x_i          // BLS signature share (secret)
big_c  = big_s + app_pk * y_i           // ElGamal ciphertext
big_y  = y_i * G                        // ElGamal nonce commitment
```

The coordinator aggregates via Lagrange interpolation:

```
big_C = Σ λ_i · big_c_i = H(app_id) · msk + app_pk · Y
```

The application decrypts with its secret key `app_sk`: `big_C - app_sk · Y = H(app_id) · msk`.

The `ckd` entry-point performs no check that `app_pk` is a valid non-identity group element: [1](#0-0) 

The `compute_signature_share` function then directly uses the unchecked `app_pk`: [2](#0-1) 

Specifically, the vulnerable line is: [3](#0-2) 

If `app_pk` is the G1 identity element, `app_pk * y.0 = identity`, so `big_c = big_s`. Every participant's ciphertext collapses to their raw BLS signature share. The coordinator's aggregation then yields:

```
big_C = Σ λ_i · big_s_i = H(app_id || pk) · msk
```

This is the confidential derived key in plaintext, with no decryption step required.

### Impact Explanation
A malicious coordinator controls the `app_pk` value forwarded to all participants when initiating a CKD session. By substituting `app_pk = G1::identity()`, the coordinator causes all participants to skip the ElGamal masking step. The coordinator then receives `CKDOutput` containing the unmasked `H(app_id) · msk` directly — the confidential derived key that is supposed to be accessible only to the legitimate application holding `app_sk`. This matches the allowed impact: **Critical — Extraction or disclosure of confidential derived secrets**.

### Likelihood Explanation
The `ckd` function is a public library API. In any deployment where the coordinator is responsible for distributing the `app_pk` to participants (the standard deployment model described in the README), a malicious coordinator can trivially pass the identity element. No cryptographic break, leaked key, or external dependency failure is required — only a single malformed input parameter that the library does not reject.

### Recommendation
Add an explicit identity-element check in `ckd` before the protocol is started, analogous to the zero-scalar checks already present elsewhere in the codebase (e.g., `do_keyshare` rejects a zero secret):

```rust
// In ckd(), after building the participants list:
if app_pk == <ElementG1 as Group>::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern used in `do_presign` for the robust ECDSA scheme, which rejects identity-element outputs: [4](#0-3) 

### Proof of Concept
1. Coordinator initiates a CKD session and passes `app_pk = ElementG1::identity()` to all participants via the `ckd` API.
2. Each participant's `compute_signature_share` computes `big_c_i = big_s_i + identity * y_i = big_s_i`.
3. Each participant sends `(λ_i · big_y_i, λ_i · big_s_i)` to the coordinator.
4. The coordinator sums: `big_C = Σ λ_i · big_s_i = H(app_id || pk) · msk`.
5. The coordinator now holds the confidential derived key without ever possessing `app_sk`. The `CKDOutput::unmask(app_sk)` step is bypassed entirely.

    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
