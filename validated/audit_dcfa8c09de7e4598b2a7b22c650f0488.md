### Title
Stale `target_shard_index` from pre-reshard layout used to index post-reshard `outcome_roots`, producing a wrong `outcome_root_proof` for light clients — (`chain/client/src/view_client_actor.rs`)

### Summary
`ViewClientActor::handle(GetExecutionOutcome)` computes `target_shard_index` from the **execution block's (old) shard layout**, then calls `get_next_block_hash_with_new_chunk` which — when a resharding event has occurred — returns a block in the **new shard layout** with a child `shard_id`. The `outcome_roots` vector is built from that new block's chunks (new layout, different count and ordering), but `target_shard_index` is never recomputed for the new layout. The `outcome_root_proof` Merkle path is therefore extracted from the wrong position, producing an invalid proof that no correct light client can verify.

### Finding Description

In `get_next_block_hash_with_new_chunk`, when a new epoch with a different shard layout is encountered, `shard_ids` is replaced with the children shards of the original shard, and the returned `shard_id` is a child shard ID in the new layout: [3](#0-2) 

The child shard's index in the new layout is **not** the same as the parent's index in the old layout. For example, in V2/V3 layouts, shard IDs are not contiguous integers and shard index ≠ shard ID: [4](#0-3) 

The bounds guard at line 1174 only catches `target_shard_index >= outcome_roots.len()`, which will not fire when the old index is within the new (larger) layout's range — it silently returns the wrong Merkle path. [5](#0-4) 

### Impact Explanation
The `outcome_root_proof` field of `GetExecutionOutcomeResponse` (and the `RpcLightClientExecutionProofResponse` it feeds) is a Merkle path that a light client uses to verify:

```
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
```

When `target_shard_index` is wrong, the returned path proves a **different shard's** outcome root, not the one containing the actual execution outcome. Any correct light client will reject the proof. The RPC returns no error — the caller receives a structurally valid but semantically incorrect proof with no indication of failure.

### Likelihood Explanation
This is triggered by any `EXPERIMENTAL_light_client_proof` (or `GetExecutionOutcome`) query for a receipt whose execution block is in epoch N, while the next block with a new chunk for that shard falls in epoch N+1 (post-reshard). This is a normal operational condition during any resharding event. No special privileges are required — any user with a receipt executed just before a resharding boundary can trigger this via the public RPC.

### Recommendation
After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` from the **new block's epoch shard layout**:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```
