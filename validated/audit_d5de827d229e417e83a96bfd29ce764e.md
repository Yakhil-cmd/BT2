### Title
Commented-Out Royalty Deduction in `calculateNFTRoyalties` Causes NFT Seller to See Inflated "Net Proceeds" on Imported TokenForNFT Offers — (`packages/gui/src/components/offers/utils.ts`)

---

### Summary

The `calculateNFTRoyalties` function in `utils.ts` has its royalty-deduction logic commented out, causing `nftSellerNetAmount` to equal the full offer amount rather than `amount - royalty`. `NFTOfferDetails` uses this value to render both the "Net Proceeds" label and the `overrideNFTSellerAmount` passed to `OfferSummaryTokenRow`. An NFT seller who imports a `TokenForNFT` offer is shown they will receive the full offer price when they will actually receive less after royalties are paid to the creator.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The deduction of `royaltyAmountString` and `makerFee` is commented out, so `nftSellerNetAmount` always equals the raw offer `amount`.

In `NFTOfferDetails` (`NFTOfferViewer.tsx`), for `TokenForNFT` offers, `overrideNFTSellerAmount` is derived directly from this value:

```typescript
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

This value is passed down to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it overrides the displayed amount:

```typescript
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [3](#0-2) 

Additionally, the "Net Proceeds" section in the UI directly renders `nftSaleInfo?.nftSellerNetAmount`: [4](#0-3) 

The tooltip for "Net Proceeds" explicitly states it is "the asking price, minus any associated creator fees" — but the displayed value does not subtract creator fees. [5](#0-4) 

The `totalAmount` for `TokenForNFT` is also inflated: `amount + makerFee + royaltyAmount` instead of just `amount`: [6](#0-5) 

---

### Impact Explanation

For a `TokenForNFT` offer of 100 XCH on an NFT with 10% royalty:

| Field displayed | Shown value | Correct value |
|---|---|---|
| NFT Purchase Price | 100 XCH | 100 XCH |
| Creator Fee (10%) | 10 XCH | 10 XCH |
| **Net Proceeds** | **100 XCH** | **90 XCH** |
| Total Amount Offered | 110 XCH | 100 XCH |

The NFT seller sees "Net Proceeds: 100 XCH" — a label that explicitly implies royalties have been deducted — when they will actually receive 90 XCH. The seller may accept the offer under a false belief about their proceeds, resulting in an unintended financial loss. This fits the High impact category: display of the wrong amount causes a user to approve/accept an offer.

---

### Likelihood Explanation

The bug is triggered by any `TokenForNFT` offer where the NFT has a non-zero `royaltyPercentage`. No special crafting is required; a standard offer file is sufficient. The `nftSaleInfo` block is only rendered when `useNFT(launcherId)` resolves the NFT from the viewer's local wallet, which is the normal case for an NFT seller reviewing an imported offer. The attacker simply creates a legitimate-looking offer at a price the seller finds acceptable, knowing the seller will see inflated net proceeds.

---

### Recommendation

Restore the commented-out deduction in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? parseFloat((amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12))
    : amount;
```

Also correct `totalAmount` for `TokenForNFT` to not add `royaltyAmount` on top of `amount` (since royalty is paid from within the offered amount, not in addition to it).

### Proof of Concept

Unit test against `calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts`:

```typescript
import { calculateNFTRoyalties } from './utils';
import NFTOfferExchangeType from './NFTOfferExchangeType';

test('TokenForNFT: nftSellerNetAmount should be amount minus royalty', () => {
  const result = calculateNFTRoyalties(
    100,       // 100 XCH offered
    0,         // no maker fee
    10,        // 10% royalty
    NFTOfferExchangeType.TokenForNFT,
  );
  // Bug: result.nftSellerNetAmount === 100 (full amount, royalty not deducted)
  // Expected: result.nftSellerNetAmount === 90
  expect(result.nftSellerNetAmount).toBe(90);   // FAILS with current code
  expect(result.royaltyAmount).toBe(10);         // passes
});
```

With the current code, `nftSellerNetAmount` is `100` instead of `90`, confirming the "Net Proceeds" display is inflated by the full royalty amount.
