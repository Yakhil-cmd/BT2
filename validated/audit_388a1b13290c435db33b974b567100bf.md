### Title
Unchecked Precompile Call Return Value After Token Burn Causes Permanent Loss of Funds - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` and `EvmErc20.withdrawToEthereum` burn the caller's ERC-20 tokens before invoking the exit precompile via a low-level assembly `call`. The return value of that `call` is captured in a local variable `res` but is **never checked and never causes a revert**. If the precompile call fails for any reason, the tokens are permanently destroyed on the Aurora/EVM side while no corresponding transfer is initiated on the NEAR or Ethereum side, resulting in a permanent freeze of user funds.

---

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow the same unsafe pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}
``` [2](#0-1) 

The exit precompile (`ExitToNear` at `0xe9217bc7...`, `ExitToEthereum` at `0xb0bd02f6...`) can legitimately fail and return `0` from the `call` opcode in multiple situations documented in the precompile implementation:

- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — recipient bytes are not valid UTF-8 or not a valid NEAR account ID
- `ERR_TARGET_TOKEN_NOT_FOUND` — the ERC-20 is not registered in the NEP-141 map
- `ERR_KEY_NOT_FOUND` — the eth connector account is not set in storage
- `ERR_INVALID_AMOUNT` — amount exceeds `u128::MAX` [3](#0-2) 

In all these cases, the EVM `call` opcode returns `0` (failure). Because the assembly block does not check `res` and does not `revert`, the outer transaction continues normally. The `_burn` that already executed is committed to state. The user's tokens are gone from Aurora with no corresponding NEAR-side transfer ever scheduled.

This is structurally identical to the reported bug: a compound operation (burn + cross-system transfer) partially succeeds (burn commits), the failure of the second step is silently ignored, and the accounting invariant — *every burned token must produce a corresponding NEAR transfer* — is permanently broken.

---

### Impact Explanation

**Permanent freezing of user funds.** A user who calls `withdrawToNear` or `withdrawToEthereum` with any input that causes the precompile to fail will have their ERC-20 tokens burned with no recovery path. The tokens cease to exist on Aurora and are never credited on NEAR or Ethereum. There is no refund mechanism in the non-`error_refund` build path, and even in the `error_refund` path the refund callback is only scheduled when the precompile *succeeds* in returning a `PrecompileOutput` — a failing precompile call returns `0` from the assembly `call` and the callback is never set up. [4](#0-3) 

---

### Likelihood Explanation

The trigger conditions are reachable by any unprivileged EVM user:

1. **Invalid recipient bytes**: `withdrawToNear` accepts `bytes memory recipient` directly from the caller. Passing bytes that are not valid UTF-8 (e.g., `0xFF`) causes `parse_recipient` to return `ERR_INVALID_RECEIVER_ACCOUNT_ID`, the precompile call returns `0`, and the burn is committed.
2. **Unregistered ERC-20**: If an `EvmErc20` instance is deployed but its NEP-141 mapping is not yet written to storage, `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND`. Any user holding tokens in that contract who attempts withdrawal loses them permanently.
3. **Amount overflow**: An amount exceeding `u128::MAX` passes Solidity's `uint256` type but is rejected by `parse_amount` inside the precompile, again returning `0` from the assembly `call`. [5](#0-4) 

---

### Recommendation

Check the return value of the assembly `call` and revert the entire transaction if the precompile call fails. This ensures the `_burn` is also rolled back:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum`. This guarantees atomicity: either both the burn and the cross-chain transfer are initiated, or neither is.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IEvmErc20 {
    function mint(address account, uint256 amount) external;
    function withdrawToNear(bytes memory recipient, uint256 amount) external;
    function balanceOf(address account) external view returns (uint256);
}

contract BurnWithoutTransferPOC {
    IEvmErc20 public token;

    constructor(address _token) {
        token = IEvmErc20(_token);
    }

    function attack() external {
        // Admin mints 100 tokens to this contract (precondition)
        uint256 balanceBefore = token.balanceOf(address(this));
        require(balanceBefore == 100e18, "need tokens");

        // Pass invalid UTF-8 bytes as recipient — precompile will return ERR_INVALID_RECEIVER_ACCOUNT_ID
        // The call opcode returns 0 (failure) but EvmErc20 never checks it
        bytes memory invalidRecipient = hex"ff"; // not valid UTF-8
        token.withdrawToNear(invalidRecipient, 100e18);

        uint256 balanceAfter = token.balanceOf(address(this));

        // Tokens are burned (balanceAfter == 0) but no NEAR transfer was initiated
        // Funds are permanently lost
        require(balanceAfter == 0, "tokens should be burned");
        // No corresponding ft_transfer was scheduled on NEAR side
    }
}
```

The `_burn` at line 54 commits, reducing `balanceAfter` to zero. The precompile call at line 61 returns `0` because `0xFF` is not valid UTF-8, but `res` is never checked. The transaction succeeds. The 100 tokens are permanently destroyed with no NEAR-side credit. [1](#0-0)
