### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Pool Member Unclaimed Yield via Trace Bloat - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in the delegation pool contract contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace since their last reward claim. Because there is no cap on how many distinct-epoch balance changes a pool member can accumulate, a delegator who repeatedly modifies their balance across many epochs without claiming rewards will eventually cause `claim_rewards` to exceed the Starknet block gas limit, permanently freezing their unclaimed yield.

### Finding Description
`calculate_rewards` in `src/pool/pool.cairo` iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace is grown by `set_member_balance`, which is called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`. The underlying `trace.insert` appends a **new checkpoint** whenever the epoch key differs from the last stored key:

```cairo
} else {
    // Checkpoint keys must be non-decreasing.
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

`set_member_balance` always inserts at `get_epoch_plus_k()` (current epoch + K), so one new trace entry is appended per distinct epoch in which the pool member changes their balance: [3](#0-2) 

`add_to_delegation_pool` is callable by the pool member or their reward address with no per-epoch frequency cap: [4](#0-3) 

`exit_delegation_pool_intent` is callable by the pool member with no per-epoch frequency cap: [5](#0-4) 

The `entry_to_claim_from` cursor advances only when `claim_rewards` succeeds. If the delegator accumulates N distinct-epoch balance changes without claiming, the loop must iterate N times in a single `claim_rewards` call. Once N is large enough to exhaust the block gas limit, `claim_rewards` becomes permanently uncallable for that pool member. [6](#0-5) 

### Impact Explanation
When the `pool_member_epoch_balance` trace grows beyond the gas-iteration threshold, `claim_rewards` reverts on every call. The pool member's accumulated unclaimed yield is permanently frozen inside the pool contract with no recovery path — matching the **High: Permanent freezing of unclaimed yield** impact category.

### Likelihood Explanation
Each epoch contributes at most one new trace entry (same-epoch calls overwrite the last entry). The delegator must therefore sustain balance changes across many consecutive epochs without ever claiming rewards. In practice this requires sustained activity over a large number of epochs, making the likelihood **Low-Medium**: the scenario is reachable by any unprivileged delegator acting on their own account, but requires deliberate or negligent long-term non-claiming combined with frequent balance changes. The developers themselves flag the loop as "unbounded" in a code comment, confirming awareness of the risk without a mitigation.

### Recommendation
1. **Cap trace length per pool member**: Enforce a maximum number of unclaimed balance-change checkpoints (e.g., 1 000). Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if the cap would be exceeded without a prior `claim_rewards` call.
2. **Partial reward claiming**: Allow `claim_rewards` to accept a `max_iterations` parameter so a pool member can drain the trace in multiple transactions rather than one.
3. **Automatic checkpoint compaction**: When `claim_rewards` is called, compact all processed entries rather than relying solely on the `entry_to_claim_from` cursor.

### Proof of Concept
1. Delegator calls `enter_delegation_pool` with `amount = MIN_STAKE`.
2. Each epoch, delegator calls `add_to_delegation_pool(pool_member, amount: 1)`. Because `get_epoch_plus_k()` advances each epoch, `trace.insert` appends a new `PoolMemberBalanceCheckpoint` every epoch.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs, `pool_member_epoch_balance` trace length = N.
5. Delegator calls `claim_rewards`. `calculate_rewards` enters the `while` loop and iterates N times, each iteration performing multiple storage reads (`pool_member_trace.at(i)`, `find_sigma`, arithmetic). For sufficiently large N this exceeds the Starknet block gas limit.
6. Every subsequent `claim_rewards` call also reverts. The delegator's accumulated yield is permanently frozen. [7](#0-6) [8](#0-7)

                // If the balance change is after `until_epoch` (and therefore does not affect

                    break;

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
