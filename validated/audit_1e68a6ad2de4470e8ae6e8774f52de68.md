### Title
Missing Participant-Set Validation in OT-Based ECDSA Presign Allows Corrupted Presignature Acceptance - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary
The `presign` function in the OT-based ECDSA implementation explicitly omits the check that the presigning participant set matches the participant set used during triple generation. A malicious coordinator or library caller can supply triples generated by one participant set while invoking presign with a different participant set, causing all honest parties to accept a mathematically corrupted presignature.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the public `presign` entry point accepts a `participants` slice and a `PresignArguments` struct that embeds two Beaver triples (`triple0`, `triple1`) produced during a prior triple-generation phase. The function validates that the threshold matches the triples' recorded threshold:

```rust
// src/ecdsa/ot_based_ecdsa/presign.rs lines 43-47
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
```

However, the function **explicitly omits** the analogous check for the participant set:

```rust
// src/ecdsa/ot_based_ecdsa/presign.rs lines 38-40
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
```

Inside `do_presign`, the private triple shares are immediately linearized using Lagrange coefficients derived from the **presign** participant set — not the triple-generation participant set:

```rust
// src/ecdsa/ot_based_ecdsa/presign.rs lines 93-103
let lambda_me = participants.lagrange::<Secp256>(me)?;

let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;

let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
...
let x_prime_i = lambda_me * private_share;
```

The triple shares `k_i`, `e_i`, `a_i`, `b_i`, `c_i` encode secret-sharing polynomials evaluated at the **triple-generation** participant's identifier. Multiplying them by a Lagrange coefficient computed for a **different** participant set breaks the additive-sharing invariant that the presign protocol relies on. The subsequent consistency checks (e.g., `E =?= e·G`, `α·G =?= K + A`, `β·G =?= X + B`) operate on the corrupted linearized values and will either silently pass (if the attacker controls enough parties to make the sums consistent) or abort — but in either case honest parties have accepted or acted on a structurally invalid presignature.

The assumption in the comment — "presumably they need to have been present in order to have shares" — is not enforced anywhere in the code and is trivially violated by a malicious coordinator who mixes triples from one session with a different participant list.

---

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

When the presign participant set differs from the triple-generation participant set, the Lagrange linearization at lines 95–103 is mathematically wrong. The resulting `PresignOutput` is structurally invalid: the implicit Shamir-sharing invariant `Σ λᵢ · shareᵢ = secret` no longer holds. Any subsequent signing round that consumes this presignature will either produce an invalid ECDSA signature (undetectable until verification fails externally) or cause the signing protocol to abort, permanently denying signing for that presignature slot. Because the corruption is introduced silently at the presign stage, honest parties have no in-protocol signal that the presignature is unusable until the signing phase fails.

---

### Likelihood Explanation

**Medium.** The `presign` function is a public library API. Any caller — including a malicious coordinator orchestrating the multi-party protocol — controls both the `participants` argument and the `PresignArguments` (which contains the triples). No cryptographic barrier prevents supplying mismatched inputs; the only guard is the threshold equality check, which a mismatched-but-same-threshold participant set trivially satisfies. The codebase itself flags this gap with an explicit `NOTE` comment, confirming the omission is known and unmitigated.

---

### Recommendation

Store the participant set (or a collision-resistant hash of it) inside the triple output during generation, and add an explicit equality check in `presign` before proceeding:

```rust
// After the threshold check in presign():
if args.triple0.1.participants != participants_set_hash
    || args.triple1.1.participants != participants_set_hash
{
    return Err(InitializationError::BadParameters(
        "Participant set must match the set used during triple generation".to_string(),
    ));
}
```

Alternatively, at minimum, document that the caller is responsible for ensuring the participant sets match and add a debug-mode assertion. The current comment "presumably they need to have been present" is insufficient as a security guarantee.

---

### Proof of Concept

1. Party set **G₁ = {A, B, C}** runs triple generation; each party receives triple shares `(a_i, b_i, c_i)` and `(k_i, d_i, e_i)` evaluated at their identifier under G₁'s Lagrange basis.
2. A malicious coordinator constructs `PresignArguments` using the triples from step 1 but calls `presign` with participant set **G₂ = {A, B, D}** (same size and threshold, but D ≠ C).
3. The threshold check at line 43 passes (thresholds are equal).
4. The participant-set check is absent (lines 38–40), so `do_presign` proceeds.
5. Inside `do_presign`, `lambda_me` is computed for G₂ (line 93), but the triple shares encode the G₁ polynomial. The linearized values `k'_i`, `a'_i`, `b'_i`, `x'_i` are now wrong.
6. The consistency checks (`E =?= e·G`, etc.) may still pass if the attacker controls the broadcast values, or they abort — but in either case the presignature is corrupted.
7. Honest parties store and later attempt to use the corrupted `PresignOutput`, causing the signing phase to produce an invalid signature or abort, permanently consuming the presignature. [1](#0-0) [2](#0-1)
