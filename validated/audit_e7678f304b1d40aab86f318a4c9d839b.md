### Title
COA Ownership Proof Signature Does Not Bind to `evmAddress`, Enabling Cross-COA Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof()` verifies that a Cadence account's keys signed `signedData`, then separately checks that the COA resource at the supplied `path` matches the supplied `evmAddress`. Because the signature covers only `signedData` and never commits to `evmAddress`, a signature produced to prove ownership of one COA can be replayed by any unprivileged caller to prove ownership of a different COA belonging to the same Cadence account. The code itself acknowledges this gap in a comment but does not enforce a fix at the protocol level.

### Finding Description

`validateCOAOwnershipProof` is a public (`access(all)`) Cadence function in the deployed `EVM` system contract. Its signature is:

```cadence
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
```

The function performs two independent checks:

1. **Signature check** — verifies that the keys of `address` signed `signedData`:
   ```cadence
   let isValid = keyList.verify(
       signatureSet: signatureSet,
       signedData: signedData,
       domainSeparationTag: "FLOW-V0.0-user"
   )
   ``` [1](#0-0) 

2. **COA address check** — borrows the COA resource at `path` and compares its EVM address to the caller-supplied `evmAddress`:
   ```cadence
   if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
       let coaAddressBytes = coaRef.address().bytes
       for index, item in coaAddressBytes {
           if item != evmAddress[index] { ... }
       }
   ``` [2](#0-1) 

The two checks are **not linked**: the signature never commits to `evmAddress` or `path`. The code explicitly acknowledges this:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [3](#0-2) 

**Attack path:**

1. Alice owns a Cadence account with two COAs: `COA-A` published at `/public/coaA` and `COA-B` published at `/public/coaB`.
2. Alice signs `signedData` (e.g., a challenge nonce) to prove ownership of `COA-A` to some off-chain or on-chain authentication system.
3. An attacker intercepts Alice's `(signedData, keyIndices, signatures)` tuple.
4. The attacker calls `validateCOAOwnershipProof` with:
   - `address = Alice`
   - `path = /public/coaB`
   - `signedData = <Alice's original signed data>`
   - `keyIndices / signatures = <Alice's original proof>`
   - `evmAddress = COA-B's EVM address`
5. The signature check passes (Alice's keys did sign `signedData`). The COA address check passes (the resource at `/public/coaB` has address `COA-B`). The function returns `isValid: true`.
6. The attacker has forged a valid ownership proof for `COA-B` using a signature Alice only intended for `COA-A`.

The entry point is fully unprivileged: `validateCOAOwnershipProof` is `access(all)` and callable from any Cadence script or transaction. [4](#0-3) 

It is also reachable from EVM via the `verifyCOAOwnershipProof` precompile, which decodes the proof and calls the Cadence function: [5](#0-4) [6](#0-5) 

### Impact Explanation

Any off-chain authentication system or on-chain Cadence/EVM contract that calls `validateCOAOwnershipProof` directly (not via the `coa.sol` ERC-1271 path) and relies on the result to grant access or privileges tied to a specific COA is vulnerable to cross-COA impersonation. An attacker who obtains a valid proof for one COA can replay it to pass ownership checks for a different COA of the same Cadence account, bypassing authentication without possessing the account's private keys.

The `coa.sol` ERC-1271 path (`isValidSignature`) is **not** vulnerable because it always passes `address(this)` as the EVM address, binding the proof to the calling contract: [7](#0-6) 

However, any caller invoking `validateCOAOwnershipProof` directly from Cadence or from a custom EVM contract that does not enforce `address(this)` is exposed.

### Likelihood Explanation

The precondition is that a Cadence account holds more than one COA. The protocol does not prevent this; a user can call `EVM.createCadenceOwnedAccount()` multiple times and store COAs at different paths. The comment in the code acknowledges the risk but relies on the social assumption that "Flow wallets historically create at most one COA per account," which is not a protocol-enforced invariant. As COA usage grows and multi-COA patterns emerge (e.g., for organizational key management or protocol integrations), the exploitable population increases. The attacker needs only a previously captured valid proof tuple, which is public data in any authentication flow.

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof` by verifying that the 20-byte EVM address is present within `signedData`, or by constructing the signed payload as `hash(evmAddress || applicationData)` at the protocol level before passing it to `keyList.verify`. This mirrors the recommended mitigation in the original report: the signature must include the identifier it is intended for.

### Proof of Concept

```cadence
// Alice previously produced a valid proof for COA-A:
// signedData = someChallenge
// keyIndices = [0], signatures = [aliceSig]
// evmAddress = COA-A-address, path = /public/coaA

// Attacker replays it for COA-B (same Cadence account, different COA):
let result = EVM.validateCOAOwnershipProof(
    address: aliceCadenceAddress,
    path: /public/coaB,           // attacker changes path to COA-B
    signedData: someChallenge,    // same signedData Alice signed
    keyIndices: [0],              // same key indices
    signatures: [aliceSig],       // same signatures
    evmAddress: coaBEvmAddress    // attacker supplies COA-B's address
)
// result.isValid == true — COA-B ownership falsely proven
```

The signature verification at line 1082–1086 passes because Alice's key did sign `someChallenge`. The COA address check at lines 1095–1105 passes because the resource at `/public/coaB` has address `coaBEvmAddress`. No cryptographic primitive is broken; the attack exploits the missing binding between the signed payload and the target COA identity. [3](#0-2) [1](#0-0) [8](#0-7)
