### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator aggregates participant contributions `(norm_big_y, norm_big_c)` with no cryptographic verification of their correctness. Any single malicious participant can send arbitrary group elements, silently corrupting the final `CKDOutput` delivered to the application.

### Finding Description
In `do_ckd_coordinator`, the coordinator computes its own share and then receives each other participant's `(norm_big_y, norm_big_c)` pair, summing them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

There is no zero-knowledge proof, commitment binding, or any other check that the received `(norm_big_y, norm_big_c)` values were honestly computed from the participant's actual key share and the agreed `app_id`/`app_pk`. The correct computation each participant should perform is:

```rust
let norm_big_y = big_y * lambda_i;          // λ_i · y_i · G
let norm_big_c = big_c * lambda_i;          // λ_i · (x_i · H(pk,app_id) + y_i · A)
``` [2](#0-1) 

A malicious participant can substitute any pair of group elements for these values. Because the coordinator has no access to individual private shares and the protocol provides no public commitment to verify against, the substitution is undetectable at the protocol level.

Contrast this with the OT-based ECDSA signing coordinator, which at least performs a final signature verification to catch corrupted shares:

```rust
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed(
        "signature failed to verify".to_string(),
    ));
}
``` [3](#0-2) 

No equivalent final-output check exists in the CKD coordinator path. [4](#0-3) 

### Impact Explanation
**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The `CKDOutput` encodes `(Y, C)` where `C = msk · H(pk, app_id) + y · A`. If any participant substitutes a wrong `norm_big_c`, the aggregated `C` no longer encrypts the correct confidential key under the application's public key `A`. The application will decrypt a random, attacker-influenced value instead of the intended `msk · H(pk, app_id)`. The coordinator and all honest parties have no way to detect this before delivering the output.

### Likelihood Explanation
**High.** The attack requires only one malicious participant out of the threshold set. No special capability beyond participation in the protocol is needed — the attacker simply sends arbitrary group elements in place of their honest contribution. The attack is silent: the protocol completes normally and returns `Ok(Some(ckd_output))` with no error.

### Recommendation
Add a non-interactive zero-knowledge proof (e.g., a Schnorr-style proof of discrete-log equality) that binds each participant's `(norm_big_y, norm_big_c)` to their public key share. Specifically, each participant should prove:

- `norm_big_y = λ_i · y_i · G` for a known `y_i` (proof of knowledge of `y_i`)
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · A)` is consistent with the participant's public verification share `x_i · G2`

The coordinator must verify all proofs before aggregating. Alternatively, a commitment-then-reveal round can be added so that participants commit to their contributions before seeing others', preventing adaptive manipulation.

### Proof of Concept

1. Honest setup: `n` participants run CKD with `app_id`, `app_pk = A`, and their key shares `x_i`.
2. Malicious participant `P_m` computes the correct `(norm_big_y_m, norm_big_c_m)` but instead sends `(G, G)` (the generator point) to the coordinator.
3. The coordinator sums all contributions:
   - `Y_final = Σ_{i≠m} λ_i·y_i·G + G` (wrong)
   - `C_final = Σ_{i≠m} λ_i·(x_i·H + y_i·A) + G` (wrong)
4. `do_ckd_coordinator` returns `Ok(Some(CKDOutput::new(Y_final, C_final)))` with no error. [5](#0-4) 
5. The application calls `ckd_output.unmask(app_sk)` and recovers a value that is not `msk · H(pk, app_id)`, silently producing a wrong confidential key with no indication of failure.
