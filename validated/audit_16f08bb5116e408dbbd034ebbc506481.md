Audit Report

## Title
Unbounded `ingress_expiries` Loop Enables OOM DoS via Crafted `/construction/payloads` Metadata — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta API's `construction_payloads` handler accepts user-controlled `ingress_start` and `ingress_end` metadata fields as raw `u64` nanosecond values and feeds them directly into an unbounded `while` loop with no range validation. An unauthenticated attacker can supply `ingress_start=0` and `ingress_end=u64::MAX` in a single small HTTP POST, causing the loop to iterate approximately 154 million times, allocating over 1 GB of memory for `ingress_expiries` alone and crashing the Rosetta process. The ICRC1 Rosetta counterpart already contains the correct fix; the ICP Rosetta handler has no equivalent guard.

## Finding Description
In `rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`, the loop step is:

```
interval = MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120,000,000,000 ns
``` [1](#0-0) [2](#0-1) 

`ingress_start` and `ingress_end` are taken directly from user-supplied metadata with no validation: [3](#0-2) 

The loop then runs without any iteration cap or range guard: [4](#0-3) 

With `ingress_start=0` and `ingress_end=18446744073709551615` (`u64::MAX`):
- **Iterations**: `u64::MAX / 120_000_000_000 ≈ 154,000,000`
- **`ingress_expiries` Vec**: 154M × 8 bytes ≈ **1.23 GB**

The `add_payloads` function then pushes **2 `SigningPayload` structs per expiry per transaction**, each containing a hex-encoded hash, multiplying memory consumption further: [5](#0-4) 

The endpoint is publicly accessible with no authentication: [6](#0-5) 

The 4 MB JSON body limit is irrelevant — the malicious request body is two JSON integers: [7](#0-6) 

The ICRC1 Rosetta counterpart already validates both `ingress_start < ingress_end` and that `ingress_end` is within a reasonable future window before entering its equivalent loop: [8](#0-7) 

The ICP Rosetta handler has no equivalent guard, confirming the root cause is a missing validation that was already known to be necessary.

## Impact Explanation
This is a **High** severity finding matching the allowed ICP bounty impact: *"Application/platform-level DoS, crash... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant... Rosetta... security impact with concrete user or protocol harm."* A single unauthenticated request crashes the ICP Rosetta process, denying service to all users — including exchanges and custodians relying on it for ICP ledger transfers and neuron management. The process does not recover without a manual restart.

## Likelihood Explanation
The endpoint requires no credentials. The malicious request body is trivially small (two JSON integers). The attack is reproducible with a single HTTP POST. The contrast with the already-fixed ICRC1 version confirms the root cause is a known, missing validation. Any external attacker with network access to the Rosetta API can trigger this reliably and repeatedly.

## Recommendation
Apply the same validation already present in `rs/rosetta-api/icrc1/src/construction_api/services.rs` (lines 148–158) before the loop in `construction_payloads`:

1. Reject if `ingress_start >= ingress_end`.
2. Reject if `ingress_end - ingress_start` exceeds a reasonable maximum (e.g., 24 hours = `86_400_000_000_000` ns, yielding at most 720 iterations).
3. Alternatively, cap `ingress_expiries` to a hard maximum count (e.g., 1000) and return an error if the range would exceed it.

## Proof of Concept
Send the following HTTP request to a running ICP Rosetta instance:

```http
POST /construction/payloads HTTP/1.1
Content-Type: application/json

{
  "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
  "operations": [{"operation_identifier":{"index":0},"type":"TRANSACTION","account":{"address":"<valid_address>"},"amount":{"value":"-100","currency":{"symbol":"ICP","decimals":8}}}],
  "public_keys": [{"hex_bytes":"<valid_pubkey>","curve_type":"edwards25519"}],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

**Expected result**: The Rosetta process allocates ~1.23 GB for `ingress_expiries` alone, then OOMs and crashes.

**Deterministic test**: A unit test asserting `ingress_expiries.len() <= 1000` and `elapsed < Duration::from_secs(1)` will both fail for any `ingress_end - ingress_start > ~1000 * 120_000_000_000`.
