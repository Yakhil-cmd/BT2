### Title
Greedy-regex last-blob extraction in `parseOfferData` silently substitutes a malicious offer for the one the user intended to import — (`packages/gui/src/components/offers/OfferImport.tsx`, `packages/gui/src/components/offers2/OfferBuilderImport.tsx`)

---

### Summary

The `parseOfferData` function in both offer-import components uses a dotAll (`s`-flag) regex with a greedy `(?<leading>.*)` group. When a file contains two `offer1` bech32 blobs, the greedy group consumes the first blob as "leading text," and the `(?<offer>…)` group matches only the **last** blob. The extracted blob is passed directly to `getOfferSummary` and the offer viewer with no warning, no multi-blob rejection, and no check of the discarded leading content.

---

### Finding Description

The regex at the root of the issue is identical in both files:

```
/(?<leading>.*)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
``` [1](#0-0) [2](#0-1) 

With the `s` (dotAll) flag, `.` matches newlines. Because `(?<leading>.*)` is greedy, the engine maximally extends `leading`, then backtracks only far enough to satisfy `(?<offer>offer1…)` — which means it always lands on the **last** `offer1` token in the file. For a two-blob file:

```
offer1<benign_blob>
some separator text
offer1<malicious_blob>
```

`leading` captures `offer1<benign_blob>\nsome separator text\n`, and `offer` captures `offer1<malicious_blob>`.

The caller discards `leadingText` entirely (it is commented out):

```javascript
const [offerData /* , leadingText, trailingText */] = parseOfferData(rawOfferData);
``` [3](#0-2) 

There is no check that `leading` is free of additional `offer1` tokens, no rejection of multi-blob files, and no user-visible warning. The malicious blob is then sent to the daemon:

```javascript
const { data: response } = await getOfferSummary({ offerData });
``` [4](#0-3) 

and the viewer is populated with the resulting summary:

```javascript
navigate(navigationPath, {
  state: { offerData, offerSummary, offerFilePath, imported: true },
});
``` [5](#0-4) 

The same flaw exists verbatim in `OfferBuilderImport.tsx`. [6](#0-5) 

---

### Impact Explanation

A user who opens a crafted `.offer` file is silently routed to a viewer pre-loaded with a **different** offer than the one they believed they were reviewing. If the malicious offer is crafted to superficially resemble the intended one (same asset types, slightly different amounts, or a different CAT asset ID that looks similar in the UI), the user may click "Accept" and execute an unauthorized XCH/CAT/NFT spend. There is no UI indication that the displayed offer is not the one the file was supposed to contain.

This satisfies: **High — causes a user to approve or display the wrong asset/amount/status via unsafe handling of an external file.**

---

### Likelihood Explanation

The attacker only needs to deliver a file with two `offer1` blobs to the victim (e.g., via a marketplace link, email, chat, or a tampered download). No local compromise, leaked keys, or cryptographic break is required. The file size limit (1 MB) is generous enough to accommodate both blobs. The user must still click "Accept" in the viewer, but if the malicious offer is crafted to mimic the expected one, that click is the expected outcome of the deception.

---

### Recommendation

1. **Reject multi-blob files**: after extracting `offerData`, re-run a search on `leadingText` for any `offer1[…]` token; if found, surface an error ("File contains multiple offer blobs — import aborted").
2. **Alternatively, anchor to the first match**: replace the greedy `(?<leading>.*)` with a non-greedy `(?<leading>.*?)` so the regex matches the **first** `offer1` token, then check that `trailingText` contains no additional `offer1` tokens.
3. Apply the fix identically in both `OfferImport.tsx` and `OfferBuilderImport.tsx`.

---

### Proof of Concept

```javascript
// Minimal reproducible test (no daemon needed for the regex step)
const benign   = 'offer1' + 'q'.repeat(20);   // stand-in for a real bech32 blob
const malicious = 'offer1' + 'p'.repeat(20);

const fileContent = `${benign}\nsome separator text\n${malicious}`;

const matches = fileContent.match(
  /(?<leading>.*)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
);

console.assert(matches.groups.offer === malicious,
  'FAIL: expected last blob'); // passes — first blob is silently discarded
console.assert(matches.groups.leading.includes(benign),
  'FAIL: benign blob is in leading');  // passes — benign blob is in discarded leading text
```

Running this in any JS runtime confirms that `parseOfferData` returns the malicious blob and discards the benign one. Wiring this into a full integration test against a local Chia daemon would show `getOfferSummary` receiving and summarising the malicious offer, and the viewer being populated accordingly.
