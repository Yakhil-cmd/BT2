### Title
Missing EIP-2537 (BLS12-381) and EIP-152 (Blake2F) Precompile Initialization in ZK Block Flow `post_init_op` - (File: `basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs`)

---

### Summary

`ZKHeaderPostInitOp::post_init_op` does not call `initialize_eip_2537` or `initialize_eip_152`, leaving the BLS12-381 and Blake2F precompiles unregistered in the ZK execution path. The Ethereum block flow's `EthereumPostInitOp::post_init_op` correctly registers all three groups of precompiles. Any EVM transaction routed through the ZK flow that calls these precompile addresses will receive incorrect behavior (empty return or revert) instead of the correct cryptographic output.

---

### Finding Description

`EthereumPostInitOp::post_init_op` registers three groups of precompiles:

```rust
// basic_bootloader/src/bootloader/block_flow/ethereum/post_init_op.rs
add_precompiles(system_functions)?;
hooks::eip_2537::initialize_eip_2537(system_functions)?;
hooks::eip_152::initialize_eip_152(system_functions)?;
```

`ZKHeaderPostInitOp::post_init_op` only calls `add_precompiles` and ZK-specific system hooks:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs
system_hooks::add_precompiles(system_functions)?;
system_hooks::add_l1_messenger(system_functions)?;
system_hooks::add_set_bytecode_on_address_hook(system_functions)?;
system_hooks::add_contract_deployer(system_functions)?;
system_hooks::add_interop_root_reporter(system_functions)?;
system_hooks::add_system_context_reporter(system_functions)?;
system_hooks::add_base_token_mint(system_functions)?;
// initialize_eip_2537 and initialize_eip_152 are NEVER called
```

`initialize_eip_2537` registers seven BLS12-381 precompile hooks at addresses `BLS12_G1ADD`, `BLS12_G2ADD`, `BLS12_G1MSM`, `BLS12_G2MSM`, `BLS12_PAIRING_CHECK`, `BLS12_MAP_FP_TO_G1`, `BLS12_MAP_FP2_TO_G2`.

`initialize_eip_152` registers the Blake2F precompile at address `BLAKE_HOOK_ADDRESS_LOW` (0x0009).

The `add_precompiles` function in `system_hooks/src/lib.rs` only adds Blake2F under the `mock-unsupported-precompiles` feature flag (as a stub that returns an error), and never adds BLS12-381 precompiles at all. Without the explicit `initialize_eip_2537` and `initialize_eip_152` calls, these hooks are absent from `HooksStorage` in the ZK flow.

---

### Impact Explanation

Any EVM smart contract deployed on ZKsync OS that invokes BLS12-381 operations (EIP-2537) or Blake2F (EIP-152) via a low-level call to the respective precompile addresses will receive no registered hook in the ZK execution path. The call will not be dispatched to the correct cryptographic implementation. Contracts that use these precompiles for signature verification, zero-knowledge proof verification, or hash computation will silently fail or revert, breaking their intended security invariants. This constitutes an EVM semantic mismatch between the Ethereum and ZK execution paths and can lead to loss of funds for users whose contracts depend on these precompiles functioning correctly.

---

### Likelihood Explanation

The ZK block flow is the production proving path used in `proof_running_system`. Any user transaction submitted to ZKsync OS that calls a BLS12-381 or Blake2F precompile address triggers this path. No special privileges are required — any unprivileged transaction sender can reach this code path. As EIP-2537 and EIP-152 are standard EVM precompiles, contracts using them are expected to work on any EVM-compatible chain.

---

### Recommendation

Add the missing initialization calls to `ZKHeaderPostInitOp::post_init_op`:

```rust
// basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs
system_hooks::add_precompiles(system_functions)?;
hooks::eip_2537::initialize_eip_2537(system_functions)?;
hooks::eip_152::initialize_eip_152(system_functions)?;
// ... existing ZK-specific hooks
```

Ensure the `hooks` module is imported in the ZK flow's `mod.rs` analogously to the Ethereum flow.
