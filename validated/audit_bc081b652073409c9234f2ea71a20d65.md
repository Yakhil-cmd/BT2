### Title
`protocolUnpausePool` Sets Pause Level to 1 (Admin-Paused) Instead of 0 (Active), Leaving Pool Permanently Paused After Protocol Unpause - (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` is intended to undo a protocol-level pause (level 2) and restore the pool to active state. However, it calls `setPause(1)` instead of `setPause(0)`, transitioning the pool from protocol-paused (2) to admin-paused (1). The pool remains paused after the protocol owner's unpause action, making the protocol's emergency unpause mechanism non-functional.

---

### Finding Description

The pause level semantics are:
- `0` = active (swaps allowed)
- `1` = paused by admin (swaps blocked)
- `2` = paused by protocol (swaps blocked)

The factory defines four pause transitions:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol

function pausePool(address pool) ...          // admin:    0 → 1
function unpausePool(address pool) ...        // admin:    1 → 0
function protocolPausePool(address pool) ...  // owner: 0|1 → 2
function protocolUnpausePool(address pool) ... // owner:   2 → 1  ← BUG (should be 0)
```

The buggy function:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← sets to 1, not 0
}
``` [1](#0-0) 

After `protocolUnpausePool` executes, `pauseLevel == 1`. The pool's `whenNotPaused` modifier rejects any non-zero pause level:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [2](#0-1) 

So swaps remain blocked after the protocol owner's unpause call. The pool is stuck in admin-paused state (level 1) even though the admin never paused it.

The admin's `unpausePool` can transition from 1 → 0, but it requires the pool admin to separately act. If the admin is unresponsive, a multisig with a long delay, or deliberately refuses to unpause, the protocol owner has no path to restore swap functionality — despite having called `protocolUnpausePool`.

This is the direct analog of the external bug: just as the LoopFi check `assetIn == targetToken` made the swap feature completely unusable by enforcing the wrong equality, `protocolUnpausePool` enforces the wrong target pause level (1 instead of 0), making the protocol's emergency unpause mechanism non-functional.

---

### Impact Explanation

After a protocol pause, the protocol owner cannot restore swap functionality unilaterally. `protocolUnpausePool` leaves the pool in admin-paused state (level 1), so all swaps remain blocked via `whenNotPaused`. The protocol's emergency unpause path is broken. LPs can still remove liquidity (the `removeLiquidity` function does not carry `whenNotPaused`), so there is no direct principal loss, but the core swap flow is rendered unusable until the pool admin separately calls `unpausePool`.

---

### Likelihood Explanation

This triggers every time the protocol owner calls `protocolUnpausePool`. There are no conditions or edge cases — the wrong target value is hardcoded. Any protocol-level pause event followed by a protocol unpause will leave the pool stuck at level 1.

---

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);  // target is 0
    IMetricOmmPoolFactoryActions(pool).setPause(0);        // fully unpause
}
``` [1](#0-0) 

---

### Proof of Concept

1. Pool is active: `pauseLevel == 0`.
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel == 2`. Swaps revert with `PoolPaused`.
3. Protocol owner calls `protocolUnpausePool(pool)` → `setPause(1)` → `pauseLevel == 1`. Swaps still revert with `PoolPaused`.
4. Pool admin must separately call `unpausePool(pool)` → `setPause(0)` → `pauseLevel == 0`. Swaps now work.

If the pool admin is unavailable or refuses to act, the pool remains permanently paused at level 1 despite the protocol owner having called `protocolUnpausePool`. The protocol's emergency unpause provides no actual relief. [3](#0-2) [4](#0-3)
