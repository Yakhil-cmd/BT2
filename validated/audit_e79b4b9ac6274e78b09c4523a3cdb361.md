### Title
Gateway `skip_stateful_validations` admits invoke transactions with invalid signatures when a deploy_account is pending in the mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point — the only place where the account's signature is verified — for any invoke transaction with `nonce=1` submitted against an account whose on-chain nonce is still `0` and that has any transaction in the mempool. An unprivileged attacker who observes a pending `deploy_account` transaction can submit an invoke transaction with an **invalid signature** from that account and have it admitted to the mempool without any signature check.

---

### Finding Description

**Step 1 — The skip condition.**

`skip_stateful_validations` returns `true` (skip the `__validate__` entry point) when all of the following hold:

```
tx is Invoke
tx.nonce() == Nonce(Felt::ONE)          // nonce = 1
account_nonce == Nonce(Felt::ZERO)      // account not yet deployed on-chain
account_tx_in_pool_or_recent_block(sender_address) == true
``` [1](#0-0) 

**Step 2 — The skip propagates to the blockifier validator.**

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

**Step 3 — `validate = false` means `__validate__` is never called.**

Inside `StatefulValidator::perform_validations`, for an Invoke transaction:

```rust
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());   // ← returns here; __validate__ is never executed
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

**Step 4 — `perform_pre_validation_stage` does not check the signature.**

The only checks performed are nonce increment, fee bounds, and proof-facts validation — no signature verification: [4](#0-3) 

**Step 5 — The mempool's `validate_tx` also does not check the signature.**

`ValidationArgs` carries only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — the signature field is absent: [5](#0-4) 

The mempool's `validate_tx` only checks for duplicate hashes, duplicate nonces, and fee escalation: [6](#0-5) 

**Step 6 — `account_tx_in_pool_or_recent_block` checks for any account transaction, not specifically a `deploy_account`.**

The comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, the check is purely structural — it does not distinguish transaction types: [7](#0-6) 

---

### Impact Explanation

An attacker who observes that account `A` has a pending `deploy_account` in the mempool (on-chain nonce = 0) can submit an invoke transaction with `sender_address = A`, `nonce = 1`, and a **completely invalid signature**. The gateway will:

1. Pass all stateless checks (address, resource bounds, DA modes, size).
2. Pass `validate_state_preconditions` (nonce 1 is within the allowed gap from 0).
3. Pass `validate_by_mempool` (no duplicate hash or nonce yet).
4. Receive `skip_validate = true` from `skip_stateful_validations`.
5. Admit the transaction to the mempool **without ever calling `__validate__`**.

Consequences:
- **Invalid transactions are admitted to the mempool** — matching the "High. Mempool/gateway/RPC admission accepts invalid transactions" impact.
- The attacker occupies the `nonce = 1` slot for account `A`. The legitimate user's subsequent invoke with `nonce = 1` is rejected as a duplicate nonce (unless they pay a fee-escalation premium).
- An attacker can flood the mempool with signature-less invokes targeting every visible pending `deploy_account`, causing mempool pollution and degraded UX for all new account deployments.

---

### Likelihood Explanation

The deploy_account + invoke UX pattern is explicitly supported and documented in the codebase. Any account deploying via this flow is visible in the mempool. An attacker needs only to:
1. Watch the mempool for `deploy_account` transactions (publicly observable).
2. Craft an invoke with `nonce = 1` and any (invalid) signature.
3. Submit before the legitimate user's invoke arrives.

No privileged access, no special keys, and no on-chain funds are required.

---

### Recommendation

1. **Verify the signature even when skipping `__validate__`**: The skip is intended to avoid calling the account contract (which doesn't exist yet), but the signature can still be verified off-chain against the transaction hash and the public key embedded in the `constructor_calldata` of the pending `deploy_account`.

2. **Restrict the skip to accounts with a pending `deploy_account` specifically**: The mempool should expose a query that returns `true` only when the account's pending transaction is a `deploy_account`, not any transaction type.

3. **Alternatively, limit the skip to the exact transaction hash pair**: The gateway could require the user to supply the hash of the pending `deploy_account` (as the legacy `native_blockifier` `PyValidator` does via `deploy_account_tx_hash`), and only skip validation when that hash is present in the mempool. [8](#0-7) 

---

### Proof of Concept

```
1. Alice submits RpcDeployAccountTransactionV3 for address A.
   → Gateway admits it; mempool now contains deploy_account for A.
   → On-chain nonce for A = 0.

2. Attacker submits RpcInvokeTransactionV3:
     sender_address = A
     nonce          = 1
     signature      = [0xdeadbeef]   ← invalid
     resource_bounds, tip, calldata  ← any valid values

3. Gateway stateless checks: all pass.

4. extract_state_nonce_and_run_validations:
   a. get_nonce_from_state(A) → 0
   b. validate_state_preconditions: nonce 1 ∈ [0, max_gap] → OK
   c. validate_by_mempool: no duplicate hash/nonce → OK
   d. skip_stateful_validations:
        nonce == 1 ✓, account_nonce == 0 ✓,
        account_tx_in_pool_or_recent_block(A) == true ✓
        → returns true (skip)
   e. run_validate_entry_point(skip_validate=true):
        execution_flags.validate = false
        perform_pre_validation_stage → OK (no sig check)
        __validate__ NOT called
        → returns Ok(())

5. Gateway pushes attacker's tx to mempool.
   → Attacker's invalid-signature invoke is now queued at nonce=1 for A.

6. Alice's legitimate invoke with nonce=1 arrives:
   → mempool rejects it as DuplicateNonce (or requires fee escalation).

7. During block building, blockifier runs __validate__ with validate=true
   → attacker's tx fails; but Alice's tx was already blocked.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
