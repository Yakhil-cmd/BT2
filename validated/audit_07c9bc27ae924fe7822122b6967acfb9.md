### Title
`exit_to_near_precompile_callback` Ignores Partially Returned Tokens in `ft_transfer_call` Omni Path, Causing Permanent ERC-20 Token Loss - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

When a user exits ERC-20 tokens to NEAR via the Omni message path (which uses `ft_transfer_call`), the `exit_to_near_precompile_callback` only distinguishes between complete promise success and complete promise failure. It does not account for the NEP-141 standard's partial-acceptance mechanism: a receiver's `ft_on_transfer` may return a non-zero refund amount, causing `ft_resolve_transfer` to return those tokens to Aurora's NEP-141 balance. Because the callback treats any successful promise result as a full transfer, no ERC-20 refund is issued for the returned portion. The user permanently loses those ERC-20 tokens, and the corresponding NEP-141 tokens are permanently locked in Aurora's NEP-141 balance with no ERC-20 representation.

---

### Finding Description

**Step 1 – ERC-20 burn and promise creation.**

When a user calls `withdrawToNear` with an Omni message on an ERC-20 contract, the ERC-20 contract burns `X` tokens and invokes the `ExitToNear` precompile. Inside `exit_erc20_token_to_near`, the Omni branch selects `ft_transfer_call` as the method: [1](#0-0) 

The precompile then constructs `ExitToNearPrecompileCallbackArgs` with a `refund` field (when the `error_refund` feature is enabled) capturing the **full** burned amount `X`: [2](#0-1) 

A callback `exit_to_near_precompile_callback` is attached to the `ft_transfer_call` promise: [3](#0-2) 

**Step 2 – NEP-141 partial acceptance.**

In NEAR's NEP-141 standard, `ft_transfer_call` calls `ft_on_transfer` on the receiver, which may return `Y` tokens (0 < Y ≤ X). `ft_resolve_transfer` then returns those `Y` tokens to Aurora's NEP-141 balance. The `ft_transfer_call` promise resolves as **successful** regardless of whether a partial refund occurred; the promise result data contains only the amount actually transferred (`X - Y`).

**Step 3 – Callback ignores the partial return.**

`exit_to_near_precompile_callback` checks only for binary success or failure: [4](#0-3) 

The `_` in `PromiseResult::Successful(_)` discards the promise result data entirely. Because the promise is "successful," the `else if let Some(args) = args.refund` branch — which would re-mint ERC-20 tokens — is never reached. The `Y` NEP-141 tokens silently accumulate in Aurora's NEP-141 balance with no corresponding ERC-20 minting.

**Step 4 – Accounting divergence.**

`refund_on_error` (the function that would re-mint ERC-20 tokens) is only called on complete failure: [5](#0-4) 

There is no code path that reads the amount actually transferred from the promise result and issues a partial ERC-20 re-mint for the difference `Y = X - amount_transferred`.

---

### Impact Explanation

| Quantity | Value |
|---|---|
| ERC-20 tokens burned from user | X |
| NEP-141 tokens received by receiver | X − Y |
| NEP-141 tokens returned to Aurora | Y |
| ERC-20 tokens refunded to user | **0** |

The user permanently loses `Y` ERC-20 tokens. The `Y` NEP-141 tokens are permanently locked in Aurora's NEP-141 balance: they cannot be claimed via the ERC-20 mechanism because no corresponding ERC-20 tokens exist, and `ft_resolve_transfer` does not trigger `ft_on_transfer` on Aurora (so no minting occurs). This constitutes **permanent freezing of funds** (Critical).

---

### Likelihood Explanation

The Omni message path (`ft_transfer_call`) is a production-supported feature for interacting with NEAR DeFi protocols. Partial acceptance is a standard NEP-141 pattern: protocols routinely return excess tokens when slippage limits are exceeded or liquidity is insufficient. Any user who calls `withdrawToNear` with an Omni message targeting such a protocol will silently lose the returned portion. No privileged access is required; any EVM user holding ERC-20 tokens can trigger this path.

---

### Recommendation

In `exit_to_near_precompile_callback`, when the promise result is `Successful`, parse the returned bytes to extract the amount actually transferred. Compute the difference `Y = X - amount_transferred`. If `Y > 0`, call `refund_on_error` (or an equivalent re-mint path) for `Y` tokens to the original sender's ERC-20 address. The `refund` field in `ExitToNearPrecompileCallbackArgs` already carries the recipient address and ERC-20 address needed for this re-mint; only the amount needs to be adjusted from the full burned amount to the partial unaccepted amount.

---

### Proof of Concept

1. User holds `X` ERC-20 tokens (e.g., a bridged NEP-141 token).
2. User calls `withdrawToNear` with an Omni message targeting a NEAR DeFi contract that accepts only `X − Y` tokens and returns `Y` via `ft_on_transfer`.
3. ERC-20 contract burns `X` tokens and calls `ExitToNear` precompile.
4. Precompile creates `ft_transfer_call(receiver, X, msg)` promise with callback `exit_to_near_precompile_callback` carrying `refund.amount = X`.
5. NEP-141 `ft_transfer_call` executes: receiver gets `X − Y`, `ft_resolve_transfer` returns `Y` to Aurora's NEP-141 balance.
6. Promise resolves as `Successful` with result data `"X-Y"` (amount transferred).
7. Callback matches `PromiseResult::Successful(_)` — discards result data — and returns `None` with no refund.
8. User has `0` ERC-20 tokens and `0` NEP-141 tokens. Aurora's NEP-141 balance has `Y` extra tokens permanently inaccessible to the user.
