### Title
Silent Error Swallowing in Transaction Result Error Message Processing Causes Incorrect API Responses - (File: `engine/access/ingestion/engine.go`)

### Summary
In `engine/access/ingestion/engine.go`, the `processTransactionResultErrorMessagesByReceipts` worker function catches errors from `FetchErrorMessages` but only logs them, never escalating via `ctx.Throw()` and never retrying the failed block. The function has access to an `irrecoverable.SignalerContext` specifically designed for escalating unexpected failures, but does not use it. A developer TODO comment in the same block explicitly acknowledges that some errors "should cause an exception" — confirming the missing escalation is a known gap. As a result, the Access Node silently skips storing transaction result error messages for affected blocks and continues running in a potentially corrupted state.

### Finding Description
`processTransactionResultErrorMessagesByReceipts` is a long-running component worker that drains `txResultErrorMessagesChan` and calls `txErrorMessagesCore.FetchErrorMessages` for each block ID. When `FetchErrorMessages` returns an error — whether from a storage failure, an unexpected internal condition, or an unclassified EN response — the error is only logged at `Error` level and the loop immediately moves on to the next block:

```go
err := e.txErrorMessagesCore.FetchErrorMessages(ctx, blockID)
if err != nil {
    // TODO: we should revisit error handling here.
    // Errors that come from querying the EN and possibly ExecutionNodesForBlockID should be logged and
    // retried later, while others should cause an exception.
    e.log.Error().
        Err(err).
        Msg("error encountered while processing transaction result error messages by receipts")
}
```

The function signature receives `ctx irrecoverable.SignalerContext`, which is the Flow-prescribed mechanism for escalating irrecoverable errors. Per the project's own coding conventions, any error that is not a documented benign sentinel must be treated as a potential symptom of state corruption and escalated via `ctx.Throw(err)`. Neither escalation nor retry is implemented here.

The downstream consequence is that `TransactionResultErrorMessages` storage for the affected block is never populated. The `ByBlockID` storage method explicitly documents: *"CAUTION: This method will return an empty slice both if the block is not indexed yet and if the block does not have any errors."* — meaning callers cannot distinguish a silent storage failure from a block with no failed transactions.

When an Access API caller subsequently queries `GetTransactionResultErrorMessage` or `GetTransactionResultsByBlockID` for a transaction in the affected block, `ErrorMessageByTransactionID` first checks local storage (finds nothing), then falls back to querying ENs. If the EN query also fails, the function returns `DefaultFailedErrorMessage` ("failed") for all failed transactions — a static placeholder that provides no actionable diagnostic information to the caller.

### Impact Explanation
Access API callers querying transaction error messages for blocks where `FetchErrorMessages` silently failed receive either an empty result or the static string `"failed"` instead of the actual Cadence error message. This constitutes a private data exposure / API validation failure: the API contract promises the actual error message but silently delivers incorrect data. Additionally, the Access Node continues running without restarting from a known-good state, violating the Flow high-assurance principle that unexpected errors must not be silently absorbed.

### Likelihood Explanation
`FetchErrorMessages` can fail due to: storage write errors, lock contention, EN connectivity failures that exhaust all retries, or any other unexpected internal condition. These are realistic operational conditions on a live Access Node. The TODO comment in the code confirms the developers are aware the current handling is incomplete. Any Access API caller querying error messages for a block processed during such a failure window will receive incorrect data.

### Recommendation
- Classify errors from `FetchErrorMessages`: retryable EN/network errors (e.g., `codes.Unavailable`) should be retried with backoff and re-queued; all other unexpected errors should be escalated via `ctx.Throw(err)` to trigger a node restart from a known-good state, consistent with Flow's high-assurance error handling policy.
- Remove the TODO and implement the classification explicitly, documenting which sentinel errors are benign in this context per the project's coding conventions.

### Proof of Concept

1. An Access Node processes a finalized block and sends its ID to `txResultErrorMessagesChan`.
2. `processTransactionResultErrorMessagesByReceipts` dequeues the block ID and calls `FetchErrorMessages`.
3. `FetchErrorMessages` encounters a storage write error (e.g., disk full, lock failure) and returns a non-nil error.
4. The error is logged and the block is silently skipped — `ctx.Throw` is never called, no retry is scheduled.
5. An unprivileged Access API caller submits `GetTransactionResultErrorMessage(blockID, txID)` for a failed transaction in that block.
6. `ErrorMessageByTransactionID` finds nothing in local storage, queries ENs, and if ENs also fail, returns `DefaultFailedErrorMessage = "failed"` — masking the real error.

### Anti-Pattern

Continuing on a best-effort basis is not an option, i.e. the following is an anti-pattern in the context of Flow:

```golang
err := foo()
if err != nil {
   log.Error().Err(err).Msg("foo failed")
   return
}
```

There are _rare_ instances, where this might be acceptable. For example, when attempting to send a message via the
network to another node and the networking layer errored. In this case, please include a comment with an explanation,
why it is acceptable to keep going, even if the other component returned an error. For this example, it is acceptable to handle the error by logging because:
* we expect transient errors from this component (networking layer) during normal operation
* the networking layer uses a 3rd party library, which does not expose specific and exhaustive sentinel errors for expected failure conditions
* either it is not critical that the message is successfully sent at all, or it will be retried later on
