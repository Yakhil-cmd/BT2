### Title
Missing Participant-Set Binding Between Triple Generation and Presigning Allows Malicious Coordinator to Corrupt Presignature Outputs — (File: src/ecdsa/ot_based_ecdsa/presign.rs)

---

### Summary

The OT-based ECDSA `presign` function explicitly omits the check that the participant set used during presigning matches the participant set that was present during Beaver triple generation. Only the threshold *value* is verified, not the participant identity set. A malicious coordinator can initiate a presigning session with a participant set that diverges from the one used to generate the triples, causing honest participants to combine key shares with triple shares that were not generated for them, producing cryptographically invalid presignature outputs that honest parties accept as complete.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the `presign` initialization function performs the following check:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [1](#0-0) 

The comment is a self-documenting admission: the library **assumes** that any participant in the presigning set was also present during triple generation, but **does not enforce this invariant**. The only structural check performed is that the threshold integer matches between the presign arguments and the two triples. The participant identity set embedded in the triple generation output is never compared against the presigning participant list.

The `PresignArguments` struct carries `triple0` and `triple1` (each a `(TripleGenerationOutput, TripleShareOutput)` pair). The threshold is extracted from the share metadata, but the participant set from triple generation is not surfaced or compared. Because the `TripleGenerationOutput` type does not expose the original participant set to the presign caller, there is no mechanism — even for a well-intentioned integrator — to perform this check externally.

**Attack path (malicious coordinator):**

1. Coordinator runs triple generation with participant set `{A, B, C}` at threshold 2, obtaining triple shares for each.
2. Coordinator initiates a presigning session with participant set `{A, B, D}` at threshold 2, where `D` was absent from triple generation.
3. The `presign` initialization check at line 43 passes: both triples report threshold 2, matching the presign threshold 2.
4. `D` has no valid triple shares. `D` either provides zero/random values (corrupting the presignature) or the coordinator supplies fabricated shares on `D`'s behalf.
5. Honest participants `A` and `B` proceed through the presigning rounds, combining their legitimate key shares with triple material that is inconsistent across the participant set.
6. The resulting `PresignOutput` is mathematically invalid — it does not represent a coherent threshold sharing of the intended nonce — but the protocol completes without aborting, and honest parties store it as a valid presignature.

---

### Impact Explanation

**HIGH — Corruption of presign outputs so honest parties accept unusable cryptographic outputs.**

Honest participants `A` and `B` complete the presigning protocol and store a `PresignOutput` they believe is valid. When this presignature is subsequently consumed in the signing phase, the resulting ECDSA signature is invalid (the nonce commitment `R` does not correspond to the combined key shares). The signing session is permanently wasted; the presignature cannot be reused (one-time-use invariant), so the honest parties have permanently lost a presignature slot. If the coordinator repeats this attack across all available presignatures, it constitutes a permanent denial of signing for honest parties under valid protocol inputs.

This maps directly to the allowed HIGH impact: *"Corruption of presign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs"* and *"Permanent denial of signing for honest parties under valid protocol inputs."*

---

### Likelihood Explanation

The coordinator role is an explicitly documented attacker profile in the library's security model. The coordinator controls which participant list is passed to `presign` and which triple shares are supplied via `PresignArguments`. No out-of-band mechanism prevents the coordinator from mixing participant sets across triple generation and presigning. The missing check is acknowledged in the source code itself, confirming the gap is known but unguarded.

---

### Recommendation

1. **Embed the participant set in `TripleGenerationOutput`** (or its share metadata) so that the presigning participant set can be compared against it at initialization time.
2. **Add an explicit check in `presign`** that every participant in the presigning list is present in the participant set recorded in both `triple0` and `triple1`.
3. If the participant-set binding is intentionally left to the caller, **document the invariant as a safety precondition** and provide a helper function that callers can use to verify it before constructing `PresignArguments`.

### Proof of Concept

```
// Setup
triple_gen_participants = [A, B, C], threshold = 2
(triple0, triple1) = generate_triple(triple_gen_participants, threshold)

// Malicious coordinator constructs PresignArguments with a different participant set
presign_participants = [A, B, D]   // D was NOT in triple generation
args = PresignArguments {
    threshold: 2,                  // matches triple threshold → check at line 43 passes
    triple0: (triple0_output, triple0_share_for_D_fabricated),
    triple1: (triple1_output, triple1_share_for_D_fabricated),
    keygen_output: ...,
}

// presign() at line 43: args.threshold(2) == triple0.threshold(2) == triple1.threshold(2) → OK
// No participant-set check is performed (lines 38-40 explicitly omit it)
// Protocol proceeds; A and B produce partial presignatures against inconsistent triple material
// Resulting PresignOutput is stored by A and B as valid but is cryptographically unusable
```
