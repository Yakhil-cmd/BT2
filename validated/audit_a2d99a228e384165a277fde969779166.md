### Title
Deflationary NEP-141 Token Over-Minting in ERC-20 Bridge — (`engine/src/engine.rs`)

### Summary

When a NEP-141 token with a fee-on-transfer (deflationary) mechanic is bridged to Aurora via `ft_transfer_call`, the `ft_on_transfer` callback credits the ERC-20 mirror with the *claimed* transfer amount (`args.amount`) rather than the *actual* tokens received. This causes the ERC-20 mirror supply to exceed the NEP-141 tokens held by Aurora, making the bridge insolvent for that token.

### Finding Description

Aurora's NEP-141 → ERC-20 bridge flow works as follows:

1. A user calls `ft_transfer_call(aurora_engine, amount, msg)` on a NEP-141 token contract.
2. The NEP-141 contract transfers tokens to Aurora and then calls `ft_on_transfer` on Aurora with `args.amount = amount`.
3. Aurora's `ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to `engine.receive_erc20_tokens`.
4. `receive_erc20_tokens` in `engine/src/engine.rs` reads `args.amount` directly and mints exactly that many ERC-20 tokens to the recipient. [1](#0-0) 

```rust
let amount = args.amount.as_u128();
``` [2](#0-1) 

The `setup_receive_erc20_tokens_input` function then encodes a `mint(recipient, amount)` call to the ERC-20 contract: [3](#0-2) 

For a deflationary NEP-141 token that deducts a fee during transfer, Aurora actually receives `amount - fee` tokens, but mints `amount` ERC-20 tokens. There is no balance-before/balance-after check to verify the actual received amount.

The same pattern exists in `receive_base_tokens` for the ETH connector path: [4](#0-3) 

The `ft_on_transfer` dispatcher trusts `args.amount` unconditionally: [5](#0-4) 

### Impact Explanation

**Critical — Insolvency.** For any deflationary NEP-141 token bridged to Aurora:

- Each bridge-in mints more ERC-20 tokens than the NEP-141 tokens Aurora actually holds.
- The cumulative shortfall grows with every bridge-in transaction.
- When users attempt to bridge back (exit to NEAR), the last users in queue cannot redeem their ERC-20 tokens because Aurora holds insufficient NEP-141 backing. Their ERC-20 tokens become permanently unredeemable — a direct, permanent loss of user funds.

### Likelihood Explanation

**Medium.** Any token deployer can create a deflationary NEP-141 token and register it as an ERC-20 on Aurora via `deploy_erc20_token`. No admin compromise is required. The NEP-141 standard does not prohibit fee-on-transfer tokens, and Aurora imposes no restriction on which NEP-141 tokens can be bridged. A single malicious or inadvertently deflationary token deployment is sufficient to trigger the insolvency for that token's bridge pool.

### Recommendation

In `receive_erc20_tokens`, measure the actual NEP-141 balance of the Aurora contract before and after the transfer to determine the true received amount, and mint only that delta. Alternatively, explicitly reject NEP-141 tokens that implement fee-on-transfer mechanics by checking that the balance change equals `args.amount` and reverting (returning `args.amount` to the sender) if it does not.

### Proof of Concept

1. Deploy a deflationary NEP-141 token `deflation.near` that deducts a 10% fee on every transfer.
2. Call `deploy_erc20_token("deflation.near")` on Aurora to create the ERC-20 mirror.
3. Call `deflation.near::ft_transfer_call(aurora_engine, 1000, "<evm_address>")`.
   - The NEP-141 contract transfers 900 tokens to Aurora (deducts 100 as fee).
   - The NEP-141 contract calls `aurora_engine::ft_on_transfer(sender, 1000, "<evm_address>")`.
4. Aurora's `receive_erc20_tokens` reads `args.amount = 1000` and mints 1000 ERC-20 tokens to `<evm_address>`.
5. Aurora holds 900 NEP-141 tokens but has issued 1000 ERC-20 tokens — a 100-token shortfall.
6. Repeat step 3 many times; the shortfall compounds.
7. When the first user exits (burns ERC-20, receives NEP-141), it succeeds. Eventually, the pool is drained and subsequent exit attempts fail, permanently freezing the remaining ERC-20 holders' funds.
