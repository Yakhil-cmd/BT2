### Title
COA EVM Balance Below Minimum Withdrawal Threshold Permanently Inaccessible - (`fvm/evm/stdlib/contract.cdc`)

### Summary

The `CadenceOwnedAccount.withdraw()` function enforces a hard minimum of 1e10 attoFlow per withdrawal due to the precision gap between EVM balances (18 decimal places, attoFlow) and Cadence `UFix64` Flow Token Vaults (8 decimal places). EVM gas fees are deducted at full attoFlow precision, which can reduce a COA's EVM balance to a sub-minimum amount that can never be bridged back to Cadence — permanently locking those funds.

### Finding Description

Flow EVM stores account balances in attoFlow (1e-18 FLOW), while Cadence `FlowToken.Vault` uses `UFix64` (1e-8 FLOW precision). The conversion multiplier between the two is `UFixToAttoConversionMultiplier = 10^10`. [1](#0-0) 

The `AttoFlowBalanceIsValidForFlowVault` guard enforces that any withdrawal amount must be **at least** 1e10 attoFlow: [2](#0-1) 

This guard is applied in `newInternalEVMTypeWithdrawFunction` before any withdrawal is processed: [3](#0-2) 

The Cadence-level `CadenceOwnedAccount.withdraw()` documents this restriction explicitly: [4](#0-3) 

The error triggered is: [5](#0-4) 

When a COA executes EVM transactions (via `COA.call()`, `COA.deploy()`, or `Run()`), EVM gas fees are deducted from the COA's EVM balance at full attoFlow precision. If the remaining balance after gas deduction falls below 1e10 attoFlow (i.e., between 1 and 9,999,999,999 attoFlow), the user:

- **Cannot withdraw** the remaining balance (panics with `ErrWithdrawBalanceRounding`)
- **Cannot withdraw zero** (returns an empty vault, not the stuck funds)
- Has no other Cadence-side mechanism to recover those attoFlow units

The funds remain in the COA's EVM state indefinitely, inaccessible from Cadence.

### Impact Explanation

Any COA whose EVM balance is reduced to a value in the range `[1, 9_999_999_999]` attoFlow — which can happen naturally through EVM gas fee deductions — has permanently inaccessible FLOW. The maximum stuck amount per COA is 9,999,999,999 attoFlow (≈ 0.00000001 FLOW). While individually small, this is a systematic, irreversible loss of user funds with no recovery path on the Cadence side.

### Likelihood Explanation

This is a realistic scenario for any COA user who:
1. Deposits a UFix64-aligned amount of FLOW into a COA (e.g., exactly 0.00000001 FLOW = 1e10 attoFlow), then
2. Executes any EVM transaction that consumes gas (gas fees are charged in attoFlow, e.g., 21,000 attoFlow for a basic transfer at 1 attoFlow/gas), leaving a remainder of 9,999,979,000 attoFlow — below the 1e10 threshold.

The condition is reachable by any unprivileged user submitting a standard Cadence transaction that calls `COA.call()` or `COA.deploy()`.

### Recommendation

Add a "withdraw all" path that handles the sub-minimum case. When the requested withdrawal amount equals the full COA balance and that balance is below 1e10 attoFlow, the protocol should either:
- Round down to zero (effectively burning the dust, with an explicit event), or
- Allow the withdrawal to proceed by treating the sub-minimum remainder as 0 UFix64 and returning an empty vault, so the EVM balance is zeroed out.

This mirrors the mitigation suggested in M-03: allow users to fully exit their position even when the remaining amount falls below the minimum threshold.

### Proof of Concept

1. User creates a COA and deposits exactly `0.00000001` FLOW (= 1e10 attoFlow, the minimum UFix64 unit).
2. User calls `COA.call(...)` to execute any EVM transaction. Gas fees (e.g., 21,000 attoFlow at 1 attoFlow/gas) are deducted from the COA's EVM balance, leaving `9,999,979,000` attoFlow.
3. User attempts to withdraw the remaining balance:
   ```cadence
   let bal = EVM.Balance(attoflow: 9999979000)
   let vault <- coa.withdraw(balance: bal)
   ```
4. The transaction panics: `"withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"`.
5. The user attempts to withdraw zero: returns an empty vault, does not clear the stuck balance.
6. The `9,999,979,000` attoFlow is permanently inaccessible from Cadence.

This behavior is confirmed by the existing test: [6](#0-5) 

which explicitly verifies that any amount below 1e10 attoFlow causes a panic, with no alternative recovery path tested or provided.

// AttoFlowBalanceIsValidForFlowVault returns true if the given balance,
// represented as atto-flow, can be stored in a Flow Vault, without loss
// in precision.
//
// Warning! The smallest unit of Flow token that a Flow Vault (Cadence)
// can store, is 1e-8 .
// This means the minimum balance, in atto-flow, that can be stored in a
// Flow Vault, is 1e10 .
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
	return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}

        /// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
        /// given that Flow Token Vaults use UFix64 to store balances.
        /// In other words, the smallest withdrawable amount is 1e10 attoFlow.
        /// Amounts smaller than 1e10 attoFlow, will cause the function to panic
        /// with: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".
        /// If the given balance conversion to UFix64 results in rounding loss,
        /// the withdrawal amount will be truncated to the maximum precision for UFix64.
        ///
        /// @param balance: The EVM balance to withdraw
        ///
        /// @return A FlowToken Vault with the requested balance
        access(Owner | Withdraw)
        fun withdraw(balance: Balance): @FlowToken.Vault {
            pre {
                !EVM.isPaused(): "EVM operations are temporarily paused"
            }

            if balance.isZero() {
                return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
            }
            let vault <- InternalEVM.withdraw(
                from: self.addressBytes,
                amount: balance.attoflow
            ) as! @FlowToken.Vault
