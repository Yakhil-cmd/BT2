### Title
Unvalidated Attacker-Controlled `ph` Field in Blockchain Notification Redirects Counter-Offer Destination to Attacker Address — (`packages/gui/src/hooks/useBlockchainNotifications.tsx`, `packages/gui/src/components/offers2/OfferIncomingTable.tsx`)

### Summary

An unprivileged on-chain attacker can send a blockchain notification to the victim's puzzle hash with a crafted `ph` field containing the attacker's own puzzle hash. `prepareNotifications` stores this value without any validation. When the victim clicks "Counter" on the notification, the unvalidated puzzle hash is passed to `toBech32m()`, producing the attacker's bech32m address, which is then pre-filled as the counter-offer destination. The counter-offer share dialog is pre-populated with the attacker's address, enabling the attacker to accept the counter-offer and receive the victim's assets.

### Finding Description

**Step 1 — No validation of `ph` in `prepareNotifications`:**

In `useBlockchainNotifications.tsx`, the `ph` field from the attacker-controlled JSON payload is extracted and stored directly into the notification object with zero validation: [1](#0-0) 

At line 99, `ph: puzzleHash` is destructured from `data` and at line 109 it is stored as-is. There is no hex-format check, no length check, no network prefix check, and no cross-reference against the wallet's own puzzle hashes.

**Step 2 — `NotificationOffer.handleClick` passes raw `puzzleHash` into router state:** [2](#0-1) 

The unvalidated `notification.puzzleHash` is placed directly into `address` in the navigation state to `/dashboard/offers/view`.

**Step 3 — `OfferIncomingTable.handleCounterOffer` calls `toBech32m` on the unvalidated value:** [3](#0-2) 

At line 190, `toBech32m(puzzleHash, currencyCode.toLowerCase())` is called with no prior validation. If the attacker supplied a valid 32-byte hex string (their own puzzle hash), this call succeeds and produces the attacker's bech32m address. That address is then passed to `/dashboard/offers/builder` as `address` in router state.

**Step 4 — `OfferBuilderViewer.handleCounterOffer` propagates the address:** [4](#0-3) 

The `address` prop (the attacker's puzzle hash, raw) is forwarded to the builder route.

**Step 5 — `CreateOffer / handleOfferCreated` converts and passes to `OfferShareDialog`:** [5](#0-4) 

After the counter-offer is created, `handleOfferCreated` receives the `address` (attacker's puzzle hash), calls `toBech32m` on it, and passes the resulting attacker-controlled bech32m address to `OfferShareDialog`. The share dialog is pre-populated with the attacker's address, so the victim shares the counter-offer with the attacker rather than the legitimate original offerer.

**Step 6 — `CreateOfferBuilder` receives the attacker-controlled address:** [6](#0-5) [7](#0-6) 

The `address` from location state is passed directly to `CreateOfferBuilder` without sanitization.

### Impact Explanation

The attacker's puzzle hash is pre-filled as the counter-offer share destination. When the victim creates and shares the counter-offer, the attacker receives it and can accept it, causing the victim's offered assets (XCH, CAT, NFT) to be transferred to the attacker. This is a direct asset-loss path triggered by a single on-chain notification requiring no local access.

This matches the **High** impact category: *"Corruption, spoofing, or unsafe trust of RPC, event, offer, NFT metadata, DataLayer, notification, or WalletConnect state that causes a user to approve, import, sign, send, revoke, burn, join, or display the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation

- Sending an on-chain notification to any puzzle hash is permissionless and costs only a small on-chain fee.
- The attacker only needs to know the victim's puzzle hash (publicly derivable from any transaction the victim has made).
- The victim must click "Counter" on the notification — a plausible action when presented with an attractive offer.
- No local access, leaked keys, or social engineering beyond the notification itself is required.

### Recommendation

In `prepareNotifications` (`useBlockchainNotifications.tsx`), validate the `ph` field before storing it:
- Verify it is a 64-character lowercase hex string (32 bytes).
- Optionally, verify it decodes to a valid puzzle hash format before accepting it.
- Reject (return `null`) any notification whose `ph` fails validation.

Additionally, in `OfferIncomingTable.handleCounterOffer` and `OfferBuilderViewer.handleCounterOffer`, add a guard that validates `puzzleHash` is a well-formed 32-byte hex string before calling `toBech32m`.

### Proof of Concept

1. Attacker crafts a hex-encoded JSON payload:
   ```json
   {"t":1,"d":{"u":"<valid_offer_url>","ph":"<attacker_32byte_puzzle_hash_hex>"}}
   ```
2. Attacker sends an on-chain notification to the victim's puzzle hash with this payload (using `send_notification` RPC or equivalent).
3. Victim's wallet syncs and `prepareNotifications` stores `puzzleHash = "<attacker_32byte_puzzle_hash_hex>"` with no validation.
4. Victim sees the notification, clicks "Counter".
5. `toBech32m("<attacker_32byte_puzzle_hash_hex>", "xch")` → attacker's `xch1...` address.
6. Offer builder opens with `address = attacker_address` pre-filled.
7. Victim creates and shares the counter-offer; the share dialog targets the attacker's address.
8. Attacker accepts the counter-offer; victim's assets transfer to the attacker.

**Unit test plan**: Call `prepareNotifications` with `ph` set to a valid 32-byte attacker hex string, then assert that `notification.puzzleHash` is either absent or equal to a known-safe value — not the attacker's input.

  async function handleCounterOffer(notification: Notification) {
    try {
      const puzzleHash = 'puzzleHash' in notification ? notification.puzzleHash : undefined;
      const offerId =
        'offerURL' in notification
          ? notification.offerURL
          : 'offerData' in notification
            ? notification.offerData
            : undefined;
      const offerState = getOffer(offerId);

      if (!offerState || !puzzleHash || !currencyCode) {
        return;
      }

      const address = currencyCode && puzzleHash ? toBech32m(puzzleHash, currencyCode.toLowerCase()) : '';
      const offerSummary = offerState.offer?.summary;

      if (!offerSummary || isDataLayerOfferSummary(offerSummary)) {
        return;
      }

      const offer = offerToOfferBuilderData(offerSummary);

      navigate('/dashboard/offers/builder', {
        state: {
          referrerPath: location.pathname,
          isCounterOffer: true,
          address,
          offer,
        },
      });
    } catch (e) {
      showError(e);
    }
  }
