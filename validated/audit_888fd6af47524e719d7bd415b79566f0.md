### Title
Permanent Loss of User Funds When `ExitToNear` Promise Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is not enabled, a user calling `withdrawToNear` on a bridged ERC-20 token will have their ERC-20 tokens permanently burned with no refund if the downstream NEAR `ft_transfer` promise fails. The ERC-20 tokens are destroyed in the EVM, the NEP-141 tokens remain stranded in Aurora's account, and no recovery mechanism exists.

### Finding Description

The `ExitToNear` precompile handles ERC-20 bridge exits. When a user calls `withdrawToNear` on `EvmErc20.sol`, the contract first burns the user's ERC-20 tokens, then calls the precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`, which schedules an asynchronous NEAR `ft_transfer` promise to deliver the corresponding NEP-141 tokens to the recipient.

The critical accounting gap is in the callback setup logic: [1](#0-0) 

When `error_refund` is **not** enabled, `refund` is hardcoded to `None`. Combined with no `transfer_near` (the non-wNEAR path), `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the branch at line 470 fires: [2](#0-1) 

A bare `PromiseArgs::Create` is scheduled — **no callback is registered**. If the `ft_transfer` promise fails (e.g., recipient account not registered with the NEP-141 contract), there is no `exit_to_near_precompile_callback` invocation, no re-mint of ERC-20 tokens, and no recovery path.

The `EvmErc20.sol` contract used in the non-`error_refund` build does not include a refund address in the precompile input and ignores the assembly call return value: [3](#0-2) 

The `EvmErc20V2.sol` (used only with `error_refund`) encodes the `sender` as refund address: [4](#0-3) 

The engine's `setup_deploy_erc20_input` confirms which contract is deployed based on the feature flag: [5](#0-4) 

The `exit_to_near_precompile_callback` only refunds when `args.refund` is `Some`: [6](#0-5) 

The test suite explicitly acknowledges the permanent loss: [7](#0-6) [8](#0-7) 

### Impact Explanation

**Permanent freezing of funds.** When the exit promise fails:
- The user's ERC-20 tokens are permanently burned (supply reduced).
- The corresponding NEP-141 tokens remain stranded in Aurora's account with no ERC-20 representation and no recovery mechanism.
- The user receives nothing and cannot reclaim their assets.

This is a direct analog to the original report: just as staking DAIx drained DAI from the PolicyBook leaving a deficit, here burning ERC-20 tokens without a guaranteed NEP-141 delivery leaves the user's value permanently destroyed.

### Likelihood Explanation

**High.** Any user who provides an unregistered or mistyped NEAR account ID as the recipient triggers this path. This is a realistic user error (NEAR account IDs must be explicitly registered with each NEP-141 contract). No special attacker capability is required — any unprivileged EVM user calling `withdrawToNear` with an unregistered recipient loses their tokens permanently. The `test_exit_to_near_refund` test demonstrates this exact scenario is reachable.

### Recommendation

Enable the `error_refund` feature in all production builds, which deploys `EvmErc20V2.sol` (encoding the refund address) and populates `RefundCallArgs` so that `exit_to_near_precompile_callback` can re-mint tokens on promise failure. Alternatively, remove the feature flag and make the refund path unconditional, ensuring `EvmErc20V2.sol` is always the deployed ERC-20 implementation.

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature (default build using `EvmErc20.sol`).
2. Bridge a NEP-141 token to Aurora, receiving ERC-20 tokens.
3. Call `withdrawToNear(bytes("unregistered.near"), amount)` on the ERC-20 contract.
4. The ERC-20 contract calls `_burn(msg.sender, amount)` — tokens destroyed.
5. The `ExitToNear` precompile schedules `ft_transfer` to `unregistered.near` with no callback.
6. The `ft_transfer` promise fails (account not registered with NEP-141).
7. No callback fires; no re-mint occurs.
8. User's ERC-20 balance: 0. NEP-141 balance at `unregistered.near`: 0. NEP-141 tokens remain in Aurora's account. Funds permanently lost.
