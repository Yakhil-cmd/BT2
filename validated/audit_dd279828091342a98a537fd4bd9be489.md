### Title
Stale `MAX_BLOBS_PER_BLOCK = 6` in `zk_ee` Causes Incorrect Blob-Count Validation in Ethereum STF, Rejecting Valid Post-Pectra Transactions — (`zk_ee/src/system/constants.rs` + `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs`)

---

### Summary

`zk_ee/src/system/constants.rs` still carries the pre-Pectra value `MAX_BLOBS_PER_BLOCK = 6`. The Ethereum-STF block-header code in `basic_bootloader` was updated to the post-Pectra value of 9, but the shared constant in `zk_ee` was not. Because the transaction-validation path imports `MAX_BLOBS_PER_BLOCK` from `zk_ee`, it enforces the old limit of 6 blobs per transaction. Any valid post-Pectra Ethereum mainnet transaction carrying 7–9 blobs is therefore incorrectly rejected with `BlobListTooLong`, producing a state-transition divergence between ZKsync OS and actual Ethereum mainnet.

---

### Finding Description

**Constant mismatch across two modules**

`zk_ee/src/system/constants.rs` defines:

```rust
pub const MAX_BLOBS_PER_BLOCK: usize = 6;   // pre-Pectra value
pub const GAS_PER_BLOB: u64 = 1 << 17;
``` [1](#0-0) 

`basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs` defines its own **local** constant:

```rust
const MAX_BLOBS_PER_BLOCK: usize = 9;   // post-Pectra (EIP-7691) value
const TARGET_BLOBS_PER_BLOCK: u64 = 6;
const TARGET_BLOB_GAS_PER_BLOCK: u64 = GAS_PER_BLOB * TARGET_BLOBS_PER_BLOCK;
``` [2](#0-1) 

The `HeaderAndHistory` implementation of `BasicBlockMetadata` uses the **local** constant (9) for `max_blobs()` and `blobs_gas_limit()`:

```rust
fn max_blobs(&self) -> usize { MAX_BLOBS_PER_BLOCK }          // 9
fn blobs_gas_limit(&self) -> u64 { self.max_blobs() as u64 * GAS_PER_BLOB }
``` [3](#0-2) 

The `verify_chain` function also asserts that every historical block is post-Pectra (`block_number >= PECTRA_EL_FORK_BLOCK_NUMBER`), confirming the Ethereum STF is exclusively post-Pectra: [4](#0-3) [5](#0-4) 

**Transaction validation uses the stale constant**

`metadata_op.rs` explicitly imports `MAX_BLOBS_PER_BLOCK` from `zk_ee::system` (= 6) and uses it to size the `EthereumTransactionMetadata` array:

```rust
use zk_ee::system::{..., MAX_BLOBS_PER_BLOCK, ...};
pub type EthereumBlockMetadata = SystemMetadata<
    EthereumIOTypesConfig,
    HeaderAndHistory,
    EthereumTransactionMetadata<{ MAX_BLOBS_PER_BLOCK }>,  // 6
>;
``` [6](#0-5) 

`validation_impl.rs` inherits `MAX_BLOBS_PER_BLOCK` via `use super::*` → `use zk_ee::system::*` (= 6) and passes it as the const-generic limit to `parse_blobs_list`:

```rust
match parse_blobs_list::<MAX_BLOBS_PER_BLOCK>(blobs_list) { ... }
``` [7](#0-6) 

`parse_blobs_list` rejects any transaction whose blob count exceeds `MAX_BLOBS_IN_TX`:

```rust
if blobs_list.count > MAX_BLOBS_IN_TX {
    return Err(TxError::Validation(InvalidTransaction::BlobListTooLong));
}
``` [8](#0-7) 

**Result:** The block-level gas budget allows 9 blobs (`blobs_gas_limit = 9 × GAS_PER_BLOB`), but the per-transaction validator caps blobs at 6. A valid post-Pectra Ethereum transaction with 7, 8, or 9 blobs is rejected with `BlobListTooLong` instead of being executed, diverging from the true Ethereum state.

---

### Impact Explanation

The ZKsync OS Ethereum STF is used to prove Ethereum mainnet blocks. EIP-7691 (Pectra, active from block 22431084) raised `MAX_BLOBS_PER_BLOCK` from 6 to 9, meaning a single EIP-4844 transaction may now carry up to 9 blob hashes. When ZKsync OS processes a post-Pectra block containing such a transaction, it incorrectly rejects it, producing a different post-state root than Ethereum mainnet. This is a **state-transition divergence / EVM semantic mismatch** that breaks the correctness guarantee of the proving system for any block containing a 7–9 blob transaction.

---

### Likelihood Explanation

Post-Pectra Ethereum mainnet (block ≥ 22431084) permits up to 9 blobs per transaction. Any unprivileged user can submit a valid EIP-4844 transaction with 7–9 blobs to Ethereum mainnet. Once such a transaction is included in a block, ZKsync OS will fail to reproduce the correct state transition for that block. No special access or privileged role is required; the trigger is a standard user transaction.

---

### Recommendation

Update `zk_ee/src/system/constants.rs` to reflect the post-Pectra value:

```rust
pub const MAX_BLOBS_PER_BLOCK: usize = 9;   // EIP-7691 (Pectra)
pub const TARGET_BLOBS_PER_BLOCK: usize = 6;
```

Remove the duplicate local definitions in `basic_bootloader/src/bootloader/block_flow/ethereum/block_header.rs` and `mod.rs` and have them import from `zk_ee::system` to ensure a single source of truth. Verify that `EthereumTransactionMetadata<{ MAX_BLOBS_PER_BLOCK }>` and all `parse_blobs_list` call-sites are updated consistently.

---

### Proof of Concept

1. Construct a valid post-Pectra EIP-4844 transaction with 7 blob versioned hashes.
2. Submit it to Ethereum mainnet (block ≥ 22431084); it is accepted.
3. Feed the block to ZKsync OS Ethereum STF.
4. `validate_and_compute_fee_for_transaction` calls `parse_blobs_list::<6>(blobs_list)`.
5. `blobs_list.count = 7 > MAX_BLOBS_IN_TX = 6` → returns `Err(BlobListTooLong)`.
6. The transaction is rejected; ZKsync OS computes a different state root than Ethereum mainnet.
