Audit Report

## Title
Unprivileged Caller Can Frontrun Victim's `update_balance` to Cause Persistent `AlreadyProcessing` DoS - (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary

The `update_balance` endpoint in the ckBTC minter accepts an optional `owner` principal that any non-anonymous caller can set to an arbitrary victim's principal. The function acquires a per-account `balance_update_guard` keyed on the resolved account and holds it across multiple async inter-canister calls. An attacker can continuously call `update_balance(owner = victim)` to keep the guard occupied, causing every legitimate call from the victim to return `AlreadyProcessing`. The victim's confirmed BTC deposit cannot be converted to ckBTC for as long as the attacker sustains the attack.

## Finding Description

In `update_balance`, the target account is resolved from the caller-supplied `args.owner` with no authorization check:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),   // attacker sets this to victim's principal
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;  // guard keyed on victim's account
```

`balance_update_guard` calls `Guard::new`, which inserts the account into `state.update_balance_accounts` and returns `GuardError::AlreadyProcessing` if already present:

```rust
if accounts.contains(&account) {
    return Err(GuardError::AlreadyProcessing);
}
accounts.insert(account);
```

The guard is held for the entire duration of the async function. In the common case where no new UTXOs are found, the function makes **two** sequential async calls to the Bitcoin canister while holding the guard — first `get_utxos` with `min_confirmations` (L175–183), then `get_utxos` with 0 confirmations (L225–236). Each round-trip to the Bitcoin canister is a cross-canister async await point during which the guard remains held.

The `GuardError::AlreadyProcessing` error is directly converted and returned to the caller:

```rust
GuardError::AlreadyProcessing => Self::AlreadyProcessing,
```

The `Guard` is only released on `Drop` at function end. The `MAX_CONCURRENT = 100` limit provides no protection against a targeted single-account attack — it only limits the total number of concurrent guards across all accounts.

The intentional design of allowing third-party notification (confirmed by the integration test at `rs/tests/ckbtc/ckbtc_minter_update_balance.rs:284–306`, which explicitly tests a different caller calling `update_balance` with `owner: Some(other_principal)`) is what creates this attack surface: the guard is keyed on the **owner** (victim), not the **caller** (attacker), so the attacker's in-flight call directly blocks the victim's own call.

The same pattern exists in the ckDOGE minter.

## Impact Explanation

A targeted, sustained denial-of-service against a specific user's ckBTC minting flow. The victim's BTC is confirmed on-chain but ckBTC cannot be minted for as long as the attacker continues. This constitutes a significant ck-token security impact with concrete user harm: funds are effectively locked in limbo (BTC deposited, ckBTC not receivable). This matches the **High** bounty impact: "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm" and "Application/platform-level DoS... not based on raw volumetric DDoS."

## Likelihood Explanation

The attack requires only a valid non-anonymous IC principal and knowledge of the victim's principal ID (which is publicly observable from on-chain transactions). No BTC, ckBTC, or special permissions are needed. The attacker pays only the standard IC ingress fee per call (a few cycles), while the ckBTC minter canister bears the cost of Bitcoin canister calls. The attack is trivially scriptable: the attacker pre-queues multiple `update_balance(owner = victim)` calls so that as soon as one guard is released, the next call immediately re-acquires it. The IC message queue allows the attacker to maintain continuous guard occupation with minimal effort.

## Recommendation

Key the `balance_update_guard` on the **caller** rather than the resolved owner, so the attacker's guard does not block the victim's own call. Alternatively, restrict the `owner` field so that unprivileged ingress callers may only set `owner` to `None` or `Some(caller)`, mirroring the approach in `retrieve_btc_with_approval` which always uses `ic_cdk::api::msg_caller()` directly without an overridable owner field. If third-party notification must be preserved, a separate authorization mechanism (e.g., an allowlist of helper canisters) should gate the ability to set `owner` to a different principal.

## Proof of Concept

```
// Setup: Victim has confirmed BTC on-chain, waiting to mint ckBTC.

// Attacker (any non-anonymous principal) submits in a tight loop:
update_balance({ owner = opt principal "VICTIM_PRINCIPAL_ID"; subaccount = null })
// → Attacker's call acquires balance_update_guard for victim's account
// → Guard held across async get_utxos() Bitcoin canister call(s)

// Victim submits their legitimate call concurrently:
update_balance({ owner = null; subaccount = null })
// → Returns: Err(AlreadyProcessing)
// because attacker's in-flight call holds the guard for victim's account

// Attacker pre-queues N calls so the guard is re-acquired immediately on release.
// Victim's minting is blocked indefinitely at near-zero cost to attacker.
```

A deterministic integration test can reproduce this using `PocketIC` or the existing `StateMachine` test harness: install the minter, create two identities (attacker and victim), submit an attacker `update_balance(owner=victim)` call that is in-flight (paused at the Bitcoin canister mock), then submit the victim's own `update_balance` call and assert it returns `Err(AlreadyProcessing)`.
