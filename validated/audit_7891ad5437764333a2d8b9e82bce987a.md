### Title
Missing Identity-Element Check for `app_pk` in CKD Protocol Allows Direct Disclosure of Confidential Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts an `app_pk: PublicKey` (a BLS12-381 G1 element) without validating that it is not the group identity element. If a caller passes the identity element as `app_pk`, the blinding term `y * app_pk` collapses to zero, and the aggregated CKD output directly equals the unmasked confidential derived key `msk * H(pk, app_id)` — exposing it to any party who observes the protocol output without requiring knowledge of `app_sk`.

### Finding Description
The CKD protocol is designed to produce a masked output `C = msk * H(pk, app_id) + y_total * app_pk`, where `app_pk` is the application's public key used as a blinding base. Only the holder of `app_sk` (the corresponding secret key) can unmask the output to recover the confidential derived key.

In `compute_signature_share`, each participant computes:

```rust
let big_c = big_s + app_pk * y.0;   // big_s = x_i * H(pk, app_id)
let norm_big_c = big_c * lambda_i;
``` [1](#0-0) 

After the coordinator aggregates all shares:

```
norm_big_c_total = Σ lambda_i * (x_i * H(pk, app_id) + app_pk * y_i)
                 = msk * H(pk, app_id) + app_pk * Σ(lambda_i * y_i)
```

If `app_pk` is the identity element of G1, then `app_pk * y_i = identity` for all `y_i`, so:

```
norm_big_c_total = msk * H(pk, app_id)
```

The output is the raw confidential derived key with no blinding applied. The `ckd()` entry point performs no check for this case: [2](#0-1) 

Compare this to how other protocol entry points validate their critical parameters — for example, the robust ECDSA `sign` function explicitly rejects a zero `msg_hash`: [3](#0-2) 

No equivalent guard exists for `app_pk` in `ckd()`.

### Impact Explanation
**Critical — Disclosure of confidential derived secrets.**

When `app_pk` is the identity element, the CKD output `CKDOutput` returned to the coordinator directly encodes `msk * H(pk, app_id)` — the confidential derived key — in plaintext. Any party who observes the coordinator's output (including the coordinator itself, or any network observer if the output is transmitted) obtains the confidential derived key without possessing `app_sk`. This fully breaks the confidentiality guarantee of the CKD protocol.

### Likelihood Explanation
The `ckd()` function is a public API entry point callable by any library user. The `app_pk` parameter is caller-supplied with no type-level restriction preventing the identity element. A malicious application integrator, a compromised coordinator, or any participant who controls the `app_pk` argument can trivially trigger this by passing `ElementG1::identity()` (or the equivalent zero/identity point). No cryptographic capability is required.

### Recommendation
Add an explicit check in `ckd()` that `app_pk` is not the group identity element before proceeding:

```rust
if app_pk == PublicKey::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern already used elsewhere in the codebase (e.g., the `big_r` identity check in robust ECDSA presigning): [4](#0-3) 

### Proof of Concept
1. Caller invokes `ckd(participants, coordinator, me, key_pair, app_id, PublicKey::identity(), rng)`.
2. Each participant's `compute_signature_share` computes `big_c = big_s + identity * y = big_s = x_i * H(pk, app_id)`.
3. The coordinator aggregates: `norm_big_c_total = msk * H(pk, app_id)`.
4. The returned `CKDOutput` directly contains the confidential derived key `msk * H(pk, app_id)` with no blinding.
5. The caller reads the confidential derived key from the output without ever knowing `app_sk`. [5](#0-4)
