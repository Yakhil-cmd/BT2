Audit Report

## Title
Incomplete Confidentiality Redaction Exposes Rule Metadata (`rule_id`, `incident_id`, `added_in_version`, `removed_in_version`) to Unauthenticated Callers â€” (`rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`, `canister.rs`, `getter.rs`)

## Summary

The rate-limit canister's `inspect_message` hook is structurally inert for query calls: the IC runtime only invokes it for ingress (update) messages, never for non-replicated queries. Since `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` are all declared `#[query]`, any anonymous principal reaches them without triggering the guard. The formatter applied to unauthorized callers (`ConfigConfidentialityFormatter`, `RuleConfidentialityFormatter`) only nulls `rule_raw` and `description`, leaving `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` fully populated in every undisclosed rule returned, partially defeating the canister's confidentiality model.

## Finding Description

**Root cause 1 â€” `inspect_message` is never invoked for query calls.**

`get_config` is registered as `#[query]`: [1](#0-0) 

The `inspect_message` hook at lines 34â€“55 checks `called_method == REPLICATED_QUERY_METHOD` and traps for unauthorized callers: [2](#0-1) 

Per the IC protocol specification, `inspect_message` is only invoked for ingress (update) messages before consensus. Non-replicated query calls bypass it entirely. An anonymous caller sending a plain query to `get_config`, `get_rule_by_id`, or `get_rules_by_incident_id` never reaches lines 48â€“55. The same applies to the other two query endpoints: [3](#0-2) [4](#0-3) 

**Root cause 2 â€” `ConfigConfidentialityFormatter` only redacts two fields.**

For callers that are neither `FullAccess` nor `FullRead`, `ConfigGetter::get()` applies the formatter: [5](#0-4) 

`ConfigConfidentialityFormatter::format()` sets `is_redacted = true` and nulls `description` and `rule_raw` for undisclosed rules, but leaves `rule_id` and `incident_id` intact: [6](#0-5) 

`api::OutputRule` always serializes both fields on the wire: [7](#0-6) 

**Root cause 3 â€” `RuleConfidentialityFormatter` additionally leaks version metadata.**

`RuleConfidentialityFormatter::format()` also only nulls `rule_raw` and `description`: [8](#0-7) 

`added_in_version` and `removed_in_version` are left intact and confirmed present in the `RestrictedRead` test output: [9](#0-8) 

**Existing guards are insufficient.** The `inspect_message` guard is structurally bypassed for all query calls. The formatter is the sole protection, and it is incomplete by design â€” the unit tests explicitly assert that `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` are present in the redacted response.

## Impact Explanation

Any anonymous principal can call `get_config` as a plain query and enumerate `rule_id` and `incident_id` for every undisclosed rate-limit rule. Using the leaked `rule_id` values, the same principal can call `get_rule_by_id` to obtain `added_in_version` and `removed_in_version`, revealing exactly when each confidential rule was introduced and retired. Using leaked `incident_id` values, `get_rules_by_incident_id` groups all rules by incident including their version timeline. This allows correlation of rate-limit enforcement events with network incidents before public disclosure, partially defeating the confidentiality model of the boundary node rate-limit canister. This constitutes a moderate boundary/API security impact â€” the actual rule payload (`rule_raw`) remains hidden, but the structural metadata of every confidential rule is fully observable.

## Likelihood Explanation

The attack requires no privileges, no keys, and no social engineering. It is a single unauthenticated query call to a public canister endpoint reachable by any IC client library. The `inspect_message` guard provides zero protection for query calls by IC protocol design. The incomplete redaction is confirmed by existing unit tests that explicitly assert the presence of the leaked fields. The attack is trivially repeatable and requires no special conditions.

## Recommendation

1. **Move access control into the query handlers themselves.** In `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id`, check `access_resolver.get_access_level()` at the start of the handler and return an authorization error for callers below `RestrictedRead` threshold if full blocking is desired, rather than relying on `inspect_message`.
2. **Extend `ConfigConfidentialityFormatter::format()`** to also null out `rule_id` and `incident_id` for undisclosed rules if the design intent is that their existence must not be observable to unauthorized callers.
3. **Extend `RuleConfidentialityFormatter::format()`** to also null out `added_in_version` and `removed_in_version` for undisclosed rules.
4. Remove or document the `inspect_message` guard for `REPLICATED_QUERY_METHOD` to avoid misleading future maintainers about its scope.

## Proof of Concept

```bash
# 1. Add a confidential rule (authorized update call)
dfx canister call rate_limits add_config \
  '(record { schema_version=1; rules=vec { record { incident_id="<uuid>"; rule_raw=blob "{}"; description="secret" } } })'

# 2. Call get_config as anonymous principal via plain query â€” inspect_message is NOT invoked
dfx canister call --query rate_limits get_config '(null)' --no-wallet
# Response: is_redacted=true, rule_raw=null, description=null
# BUT rule_id and incident_id are fully populated for the undisclosed rule

# 3. Use the leaked rule_id to retrieve version metadata
dfx canister call --query rate_limits get_rule_by_id '("<leaked-rule-id>")' --no-wallet
# Response: added_in_version and removed_in_version are returned for the undisclosed rule

# 4. Use the leaked incident_id to enumerate all rules grouped by incident
dfx canister call --query rate_limits get_rules_by_incident_id '("<leaked-incident-id>")' --no-wallet
```

The existing unit test at `getter.rs` lines 358â€“384 already documents the exact redacted shape returned to `RestrictedRead` callers and explicitly asserts that `rule_id` and `incident_id` are present, confirming the leak is not an oversight in the test but the actual intended (incomplete) redaction behavior.             state.is_api_boundary_node_principal(&caller_id),
        )
