### Title
Signature Verification Bypassed for Invoke Transactions via Insufficient Mempool Presence Check in `skip_stateful_validations` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature verification) for invoke transactions with nonce=1 when the account has any transaction in the mempool. The check does not verify that the mempool entry is specifically a `deploy_account` transaction. An attacker who observes a victim's pending `deploy_account` transaction can inject an invoke transaction with nonce=1, arbitrary calldata, and an invalid/empty signature for the victim's address. The gateway admits this transaction without signature verification. At execution time the transaction reverts (invalid signature), but the fee is charged from the victim's account and the victim's nonce is disrupted.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` decides whether to skip the `__validate__` entry point for an invoke transaction:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The condition `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or has been seen in a recent block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

The code comment claims this is sufficient because it "means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, the check does not distinguish between a `deploy_account` transaction and any other transaction type. It only checks account presence.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: !skip_validate = false`, meaning the `__validate__` entry point (which performs signature verification) is entirely skipped:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering, not signatures: [4](#0-3) 

The `validate_nonce` check allows nonce=1 when account_nonce=0 (within the `max_allowed_nonce_gap` of 200): [5](#0-4) 

At batcher execution time, `AccountTransaction::new_for_sequencing` sets `validate: true`, so `__validate__` IS called during block production. For a standard account contract, an invalid signature causes the transaction to revert — but the fee is still charged and the nonce is incremented. [6](#0-5) 

The legacy Python-based validator (`native_blockifier/src/py_validator.rs`) used a more precise check: it required `deploy_account_tx_hash.is_some()` (an explicit deploy-account hash passed by the caller) AND bounded the nonce by `max_nonce_for_validation_skip`. The new Rust gateway replaced this with the weaker mempool-presence check and hardcoded `Nonce(Felt::ONE)`, leaving `max_nonce_for_validation_skip` in `StatefulTransactionValidatorConfig` as a dead field that is never read. [7](#0-6) [8](#0-7) 

### Impact Explanation

An attacker can inject an invoke transaction with nonce=1, arbitrary calldata, and an invalid/empty signature for any account whose `deploy_account` transaction is pending in the mempool. The gateway admits this transaction without signature verification. At execution time the transaction reverts (invalid signature from `__validate__`), but:

1. The fee is charged from the victim's newly deployed account.
2. The victim's nonce is incremented to 2, forcing them to resubmit their intended nonce=1 transaction with nonce=2.
3. The victim's intended nonce=1 transaction (already in the mempool) becomes stale and must be replaced.

This matches the **High** impact category: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

### Likelihood Explanation

- `deploy_account` transactions are publicly visible in the mempool and via P2P gossip.
- The attacker only needs to know the victim's contract address (deterministic from class hash, salt, and constructor calldata — all public in the `deploy_account` transaction).
- No special privileges are required; any unprivileged user can submit an invoke transaction.
- The attack window is the time between the victim's `deploy_account` entering the mempool and being committed to a block.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should pass the deploy-account transaction hash explicitly (as the legacy `py_validator.rs` did via `deploy_account_tx_hash`). Additionally, restore use of the `max_nonce_for_validation_skip` configuration field so the skip window is operator-configurable and bounded.

### Proof of Concept

1. Victim submits `deploy_account` for address `A` (valid signature, nonce=0). The transaction enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

2. Attacker submits an invoke transaction:
   - `sender_address = A`
   - `nonce = 1`
   - `calldata = [drain_all_tokens_calldata]`
   - `signature = []` (empty / invalid)
   - `resource_bounds` set to any valid non-zero value

3. Gateway stateful validation path (`extract_state_nonce_and_run_validations`):
   - `get_nonce_from_state(A)` → `Nonce(0)` (account not yet deployed)
   - `validate_nonce`: `0 <= 1 <= 200` → passes
   - `validate_by_mempool`: nonce ordering check → passes
   - `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)=true` → returns `true`
   - `run_validate_entry_point(skip_validate=true)`: `validate=false` → `__validate__` **not called**
   - Transaction admitted to mempool.

4. Batcher builds block: `deploy_account` (nonce=0) executes first → account `A` deployed. Attacker's invoke (nonce=1) executes next with `validate=true` → `__validate__` called on deployed account → invalid signature → transaction **reverts**. Fee charged from `A`'s balance. `A`'s nonce is now 2.

5. Victim's legitimate nonce=1 invoke transaction is now stale (nonce < committed nonce=2) and is rejected from the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
