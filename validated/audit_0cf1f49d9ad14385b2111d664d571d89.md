### Title
Missing Minimum Threshold Validation in FROST Signing Allows Malicious Coordinator to Corrupt Signing Outputs — (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` does not enforce the minimum threshold of 2 that `assert_key_invariants` in `src/dkg.rs` enforces during key generation. A malicious coordinator can invoke the FROST signing protocol with `threshold = 1`, causing the signing session to produce a cryptographically invalid, unusable signature — permanently corrupting that session's output for all honest participants.

---

### Finding Description

During DKG, `assert_key_invariants` explicitly rejects any threshold below 2:

```rust
// src/dkg.rs:580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

This guarantees that every key share is a point on a polynomial of degree ≥ 1, so no single share can reconstruct the secret.

However, `assert_sign_inputs` — the validation gate for FROST signing — only checks the upper bound:

```rust
// src/frost/mod.rs:144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

There is **no lower-bound check** (`threshold < 2`). A caller supplying `threshold = 1` passes all validation and the signing protocol proceeds. [3](#0-2) 

The same omission exists in the `presign` entry point: [4](#0-3) 

The `ReconstructionLowerBound` type is a plain `usize` wrapper with no built-in minimum enforcement: [5](#0-4) 

---

### Impact Explanation

When `threshold = 1` is passed to the FROST signing protocol:

- The coordinator collects only **one** participant's nonce commitment and signature share.
- The Lagrange coefficient for a single signer is trivially 1, so the aggregated signature share equals `nonce_i + secret_share_i · challenge`.
- `secret_share_i = f(i)` (a polynomial evaluation at index `i`), **not** `f(0)` (the actual secret key).
- The resulting `(R, z)` pair fails verification against the group public key, producing a **permanently unusable signature** for that session.

All honest participants have committed nonces and sent shares; those nonces are consumed and must never be reused. The session cannot be replayed with the same nonces, so the corruption is irreversible for that signing instance.

**Matched impact**: *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

The coordinator role is the natural entry point for this attack. The `assert_sign_inputs` function is a public API surface; any caller constructing the signing invocation controls the `threshold` argument directly. No special privilege beyond being the designated coordinator is required. The attack requires passing a single integer (`1`) where the correct value would be ≥ 2 — trivially achievable by a malicious or compromised coordinator.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` uses, inside `assert_sign_inputs` and `presign`:

```rust
// src/frost/mod.rs — assert_sign_inputs, after line 126
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the identical check inside the `presign` function's validation block (after line 72). This mirrors the invariant already enforced in `assert_key_invariants` and closes the gap between key-generation and signing validation.

---

### Proof of Concept

1. Run DKG with `participants = [P1, P2, P3]`, `threshold = 2`. All three parties receive valid key shares on a degree-1 polynomial `f`, with public key `f(0)·G`.
2. A malicious coordinator invokes FROST signing with the same participants but passes `threshold = 1`.
3. `assert_sign_inputs` accepts this: `1 > 3` is false, so no error is returned. [2](#0-1) 
4. The coordinator collects only P1's commitment and share `z_1 = nonce_1 + f(1)·challenge`.
5. The aggregated signature `(R, z_1)` is checked against public key `f(0)·G`; since `f(1) ≠ f(0)` for a non-constant polynomial, verification fails.
6. All honest parties' nonces are consumed; the session cannot be retried with the same nonce material, permanently corrupting this signing instance.
