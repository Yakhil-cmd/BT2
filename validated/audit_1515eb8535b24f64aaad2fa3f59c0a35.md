### Title
`native_per_pubdata` Truncates to Zero When `pubdata_price < native_price`, Allowing Pubdata to Be Generated for Free — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's L2 transaction validation, `native_per_pubdata` is computed using floor (integer) division. When the operator-set `pubdata_price` is less than `native_price`, the result truncates to zero. With `native_per_pubdata == 0`, the system charges zero native resources for any amount of pubdata, bypassing the pubdata cost check entirely. Any unprivileged transaction sender can then generate pubdata up to the block limit at no native-resource cost, causing the operator to bear L1 data-publishing costs without compensation.

---

### Finding Description

**Root cause — floor division producing zero:**

In `validation_impl.rs` line 142:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is floor (integer) division. When `pubdata_price < native_price` (e.g., `pubdata_price = 1`, `native_price = 2`), the result is `0`. There is no minimum-of-1 guard, and no check that `native_per_pubdata > 0` when `pubdata_price > 0`. [1](#0-0) 

Contrast this with `native_per_gas`, which uses `div_ceil` and therefore always yields at least 1 when `gas_price > 0`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The same floor-division pattern is replicated in the public API helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

**Downstream effect — pubdata cost check is nullified:**

`native_per_pubdata` flows into `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

With `native_per_pubdata == 0`, `to_charge_for_pubdata` is always zero native resources. The check in `check_enough_resources_for_pubdata` therefore always returns `has_enough = true`, regardless of how many pubdata bytes the transaction writes. [5](#0-4) 

The same zero flows into `create_resources_for_tx` as `native_per_pubdata_byte`, so the intrinsic pubdata overhead is also zero:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata); // 0
``` [6](#0-5) 

**Attacker-controlled entry path:**

1. The operator sets block metadata with `pubdata_price < native_price` (e.g., during a period of high proving costs or low L1 gas prices, or simply by misconfiguration — the system has no guard against this ratio).
2. An unprivileged user submits a transaction that writes to many storage slots, generating large amounts of pubdata.
3. Because `native_per_pubdata == 0`, `to_charge_for_pubdata` is zero, the pubdata check passes unconditionally, and `compute_gas_refund` charges zero native resources for pubdata.
4. The user's `gas_used` is not adjusted upward for pubdata cost (the `delta_gas` path only adjusts for computational native, not pubdata native).
5. The operator publishes the data to L1 at their own expense. [7](#0-6) 

---

### Impact Explanation

When `native_per_pubdata == 0`, every transaction in the block can generate pubdata up to the block's pubdata limit (`get_pubdata_limit()`) without paying any native-resource cost for it. The operator bears the full L1 data-publishing cost. At scale, an attacker can fill every block with maximum pubdata, causing sustained financial loss to the operator and degrading chain throughput. This is a **resource accounting bug** with direct financial impact on the operator.

---

### Likelihood Explanation

The precondition `pubdata_price < native_price` is operator-set. It can arise:
- During periods of high proving costs (large `native_price`) relative to low L1 gas prices (small `pubdata_price`).
- By operator misconfiguration — the operator may not realize that any ratio below 1:1 silently makes pubdata free due to truncation, since `native_per_gas` uses `div_ceil` (protected) while `native_per_pubdata` uses `wrapping_div` (unprotected).

The asymmetry between the two division modes is subtle and not documented as a known limitation.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (floor division — can produce 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — always >= 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. This ensures that any non-zero `pubdata_price` always results in at least 1 native unit charged per pubdata byte, eliminating the truncation-to-zero path. [1](#0-0) [3](#0-2)

### Proof of Concept

```
Block context:
  native_price  = 1_000_000  (high proving cost)
  pubdata_price =   999_999  (just below native_price)

Calculation:
  native_per_pubdata = floor(999_999 / 1_000_000) = 0

Effect:
  - intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0
  - to_charge_for_pubdata      = 0 * pubdata_bytes_used = 0
  - check_enough_resources_for_pubdata always returns true

Attacker action:
  Submit a transaction that writes to 1000 new storage slots
  (each slot write = 32 bytes pubdata = 32,000 bytes total).
  
  Expected native charge: 32,000 * 1 = 32,000 native (using ceil)
  Actual native charge:   32,000 * 0 = 0 native

  Operator publishes 32,000 bytes to L1 at their own cost.
  Repeat for every block.
```
