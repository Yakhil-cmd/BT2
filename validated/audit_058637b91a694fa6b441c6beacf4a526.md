Audit Report

## Title
Unbounded Per-Peer Heap Allocation via Concurrent QUIC Bidi Streams — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary
The `start_stream_acceptor` loop in `request_handler.rs` accepts every incoming bidi stream from a peer and immediately spawns a Tokio task without any backpressure or per-connection byte budget. Each spawned task calls `read_to_end(MAX_MESSAGE_SIZE_BYTES)`, allocating a `Vec<u8>` that grows up to 128 MiB. A Byzantine subnet peer can open up to `MAX_CONCURRENT_BIDI_STREAMS = 1000` concurrent streams and stream 128 MiB on each, causing up to ~128 GiB of simultaneous heap allocation and an OOM-induced replica process crash.

## Finding Description
**Root cause:** `start_stream_acceptor` (`request_handler.rs`, lines 57–90) calls `conn_handle.conn().accept_bi()` in a `tokio::select!` loop and unconditionally spawns a new task via `inflight_requests.spawn(...)` for every accepted stream. There is no check on the number of in-flight tasks, no per-connection byte budget, and no backpressure mechanism. The code comment at lines 52–56 explicitly acknowledges this: *"The extreme result of a slow handler is that the stream limit will be reached, hence having buffered up to the stream limit number of messages/requests."*

**Per-stream allocation:** Each spawned task calls `read_request`, which calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)` (`request_handler.rs`, lines 173–174). Quinn's `read_to_end` reads chunks from the QUIC receive buffer into a `Vec<u8>`, issuing `MAX_STREAM_DATA`/`MAX_DATA` window-update frames as each chunk is consumed. This means the QUIC receive buffer stays bounded by `STREAM_RECEIVE_WINDOW = 4 MB`, but the application-level `Vec<u8>` accumulates all data up to `MAX_MESSAGE_SIZE_BYTES = 128 MiB` per stream. The flow-control windows pace delivery; they do not cap total heap allocation.

**Exploit flow:**
1. Byzantine peer holds a valid TLS-authenticated QUIC connection to the victim replica.
2. Peer opens 1000 concurrent bidi streams (the QUIC-layer limit set by `MAX_CONCURRENT_BIDI_STREAMS`).
3. On each stream, peer writes exactly 128 MiB of data and sends FIN.
4. Victim's `start_stream_acceptor` spawns 1000 Tokio tasks, each calling `read_to_end(128 MiB)`.
5. Each task's `Vec<u8>` grows to 128 MiB as window-update frames allow the sender to refill the QUIC buffer.
6. Total heap pressure reaches ~128 GiB; the replica process is OOM-killed.

**Why existing guards fail:** `STREAM_RECEIVE_WINDOW = 4 MB` and `RECEIVE_WINDOW = 200 MB` are QUIC transport-layer flow-control parameters. They limit in-flight bytes in the QUIC layer, not the size of the application-level `Vec<u8>` that `read_to_end` builds. The `MAX_MESSAGE_SIZE_BYTES` cap only prevents any single stream from exceeding 128 MiB; it does not bound the aggregate allocation across 1000 concurrent streams.

## Impact Explanation
An OOM kill of the replica process causes that node to drop out of consensus participation, constituting an application/platform-level DoS and subnet availability impact. This matches the **High ($2,000–$10,000)** impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The attack is non-volumetric — it requires no flooding, only a single connection sending protocol-legal messages.

## Likelihood Explanation
The precondition is a valid TLS-authenticated QUIC connection, meaning the attacker must be a legitimate subnet member. This is within the IC Byzantine fault model (one Byzantine node out of `3f+1` is sufficient). The attack is deterministic and requires no timing, race conditions, or special privileges beyond subnet membership. It is fully repeatable.

## Recommendation
1. **Enforce a per-connection in-flight byte budget.** Track total bytes currently held in `read_to_end` `Vec`s across all active streams for a given peer. Reject or stall new streams once the budget is exceeded.
2. **Apply backpressure before accepting streams.** Gate `accept_bi` on router readiness (as the existing comment at lines 52–56 already identifies): accept a new stream only when a handler slot is available, bounding concurrently buffered messages by handler throughput rather than `MAX_CONCURRENT_BIDI_STREAMS`.
3. **Reduce the product `MAX_CONCURRENT_BIDI_STREAMS × MAX_MESSAGE_SIZE_BYTES`** to fit within the replica's memory budget. At present `1000 × 128 MiB = 128 GiB` far exceeds any realistic heap limit.
4. **Stream data incrementally.** Replace `read_to_end` with a chunked reader that enforces a hard limit on total bytes buffered across all concurrent streams from a single peer.

## Proof of Concept
```rust
// Byzantine peer pseudocode
let conn = quic_connect(victim_addr, tls_cert).await; // valid subnet TLS cert

let mut handles = vec![];
for _ in 0..1000 {
    let conn = conn.clone();
    handles.push(tokio::spawn(async move {
        let (mut tx, _rx) = conn.open_bi().await.unwrap();
        // Write exactly MAX_MESSAGE_SIZE_BYTES = 128 MiB
        tx.write_all(&vec![0u8; 128 * 1024 * 1024]).await.unwrap();
        tx.finish().unwrap();
        // Do NOT read the response — keep stream alive to hold the allocation
    }));
}
futures::future::join_all(handles).await;
// Victim replica OOM-killed; 1000 tasks each hold a Vec<u8> growing to 128 MiB
```

A deterministic integration test can be written using `PocketIC` or a local `quinn` endpoint pair: spawn a test peer that opens 1000 streams and writes 128 MiB on each, then assert that the victim's process RSS exceeds a threshold or that the `read_to_end` tasks are bounded by a configurable budget.
