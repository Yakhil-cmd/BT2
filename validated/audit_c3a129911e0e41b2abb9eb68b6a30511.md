Audit Report

## Title
Unbounded Loop Over Missed Reward Rounds in SNS Governance `distribute_rewards` - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS Governance canister's `distribute_rewards` function iterates once per missed reward round with no cap on `new_rounds_count`. An SNS configured with a short `round_duration_seconds` (minimum 1 second, as documented) that experiences any period of canister unavailability will accumulate a massive loop count. When the canister resumes, the unbounded loop exceeds the IC instruction limit, permanently trapping every subsequent heartbeat invocation and blocking all periodic governance tasks.

## Finding Description
At `rs/sns/governance/src/governance.rs` lines 5812–5814, `new_rounds_count` is derived directly from wall-clock time with no upper bound:

```rust
let new_rounds_count = now
    .saturating_sub(reward_start_timestamp_seconds)
    .saturating_div(round_duration_seconds);
```

At lines 5861–5872, the function iterates `new_rounds_count` times performing floating-point arithmetic per iteration:

```rust
for i in 1..=new_rounds_count {
    let seconds_since_genesis = ...;
    let current_reward_rate = voting_rewards_parameters.reward_rate_at(...);
    result += current_reward_rate * voting_rewards_parameters.round_duration() * supply;
}
```

The only guards present are a zero-check on `round_duration_seconds` (line 5796) and a zero-check on `new_rounds_count` (line 5815) — neither caps the loop. The `reward.rs` module explicitly documents the minimum `round_duration_seconds` as 1 second (`rs/sns/governance/src/reward.rs` line 43: `"The corresponding minimum is 1 s."`). There is no `is_message_over_threshold` or instruction-limit guard anywhere in the SNS governance reward path (confirmed by exhaustive search). By contrast, the NNS Governance canister's `continue_processing` in `rs/nns/governance/src/reward/distribution.rs` lines 154–187 checks `is_over_instructions_limit()` between iterations and breaks, enabling batched multi-message processing. The SNS canister has no equivalent mechanism.

`distribute_rewards` is called synchronously from `run_periodic_tasks` (line 5513), which is triggered by the canister heartbeat. Once the loop is large enough to trap, every subsequent heartbeat traps at the same point because `new_rounds_count` only grows with time.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

Concretely: the SNS governance canister's heartbeat permanently traps on every invocation of `run_periodic_tasks` → `distribute_rewards`. This blocks all periodic governance tasks (reward distribution, upgrade checks, maturity finalization), permanently freezing neuron maturity accrual and SNS governance liveness for all participants of the affected SNS.

## Likelihood Explanation
An SNS deployer (unprivileged relative to the IC protocol) sets `round_duration_seconds = 1` at deployment time or via a `ManageNervousSystemParameters` proposal. Any subsequent subnet upgrade, canister upgrade, or period of unavailability accumulates missed rounds. With `round_duration_seconds = 1`, even 1 hour of downtime yields 3,600 loop iterations; 1 day yields 86,400. The IC instruction limit per message is 40 billion instructions; at even a few hundred instructions per loop iteration, tens of thousands of rounds suffice to trap. This is a realistic scenario for any SNS with a short round duration.

## Recommendation
1. **Cap `new_rounds_count`** to a safe maximum (e.g., 1,000) per single invocation, emitting a log when the cap is applied.
2. **Adopt the NNS pattern**: Refactor the purse calculation to use a timer-driven state machine that checks `is_message_over_threshold` between iterations and resumes across multiple messages.
3. **Enforce a minimum `round_duration_seconds`** in `VotingRewardsParameters` validation (e.g., 1 day = 86,400 seconds) to prevent rapid accumulation of missed rounds.

## Proof of Concept
1. Deploy an SNS with `round_duration_seconds = 1`.
2. Stop the SNS governance canister (or trigger a subnet upgrade) for 24 hours.
3. Restart the canister. The next heartbeat calls `run_periodic_tasks` → `distribute_rewards`.
4. `new_rounds_count = (now - reward_start) / 1 = 86400`.
5. The loop `for i in 1..=86400` executes 86,400 iterations of `Decimal` floating-point arithmetic.
6. The message exceeds the IC instruction limit and traps.
7. Every subsequent heartbeat traps at the same point. The SNS governance canister is permanently unable to distribute rewards or run any other periodic task.

A deterministic integration test can reproduce this by mocking `env.now()` to return a timestamp far in the future relative to `latest_reward_event().end_timestamp_seconds`, then calling `distribute_rewards` and asserting it traps or exceeds a simulated instruction budget.
