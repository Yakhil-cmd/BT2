### Title
Unconditional `PromiseArgs::Create` in `ExitToEthereum::run` Permanently Burns ETH When NEAR-Side Withdraw Fails — (`engine-precompiles/src/native.rs`)

### Summary

`ExitToEthereum::run` always schedules the connector's `withdraw` call as a bare `PromiseArgs::Create` with no callback, regardless of build configuration. If the NEAR-side `withdraw` promise fails for any reason (connector paused, insufficient NEP-141 balance, gas exhaustion, etc.), the ETH already deducted from the sender's EVM balance is permanently unrecoverable, violating the invariant that ETH burned via `ExitToEthereum` must be refundable on NEAR-side failure.

### Finding Description

In `ExitToEthereum::run`, after constructing the `withdraw` promise, the code unconditionally wraps it in `PromiseArgs::Create`: [1](#0-0) 

```rust
let withdraw_promise = PromiseCreateArgs { ... };
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

There is no `#[cfg(feature = "error_refund")]` branch, no `PromiseArgs::Callback`, and no `exit_to_ethereum_precompile_callback` equivalent. This is in direct contrast to `ExitToNear::run`, which conditionally attaches a callback when `error_refund` is enabled: [2](#0-1) 

The `error_refund` feature is imported and used only for `ExitToNear`: [3](#0-2) 

For the ETH base-token path (flag `0x0`), the EVM deducts `context.apparent_value` from the caller's balance as part of the call. The precompile returns `Ok(...)`, committing the EVM state change. The NEAR promise is then scheduled asynchronously. If that promise fails, there is no registered callback to restore the ETH balance. [4](#0-3) 

### Impact Explanation

ETH is permanently burned from EVM state with no recovery path. The total ETH supply tracked by the Aurora EVM diverges from the NEP-141 balance held by the connector, causing insolvency. Any user who successfully submits an ETH exit transaction during a window where the NEAR-side withdraw fails loses their funds with no recourse.

### Likelihood Explanation

The trigger conditions are realistic and do not require admin compromise:

- The connector contract can be **paused** by its admin for routine maintenance; a user's exit transaction submitted in that window will burn ETH and receive no refund.
- The connector's NEP-141 balance can be **temporarily insufficient** due to accounting lag or a separate bug.
- The `withdraw` call can fail due to **gas exhaustion** if `WITHDRAWAL_GAS` (100 TGas) is insufficient for a given connector implementation. [5](#0-4) 

Any unprivileged EVM user can trigger this path by calling the `ExitToEthereum` precompile at address `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with flag `0x0` and a valid 20-byte Ethereum recipient. [6](#0-5) 

### Recommendation

Mirror the `error_refund` callback pattern from `ExitToNear` into `ExitToEthereum`. When `error_refund` is enabled, wrap the `withdraw_promise` in a `PromiseArgs::Callback` whose callback invokes an `exit_to_ethereum_precompile_callback` method on the engine. That callback should inspect the promise result and, on failure, re-credit the sender's EVM ETH balance. When `error_refund` is disabled, document explicitly that failed withdrawals are non-refundable and consider whether that is an acceptable deployment posture.

### Proof of Concept

1. Deploy Aurora Engine locally with `error_refund` **disabled** (the default build).
2. Pause the ETH connector contract on NEAR.
3. Submit an EVM transaction calling the `ExitToEthereum` precompile (`0xb0bd02f6...`) with `flag=0x0`, a valid 20-byte Ethereum recipient, and `value = 1 ETH`.
4. Observe: the EVM transaction succeeds; the sender's EVM ETH balance decreases by 1 ETH.
5. The scheduled `withdraw` NEAR promise executes and fails (connector is paused).
6. Assert: no `exit_to_ethereum_precompile_callback` is ever called; the sender's EVM balance is not restored; the 1 ETH is permanently lost.

The root cause is at: [7](#0-6) 

where `PromiseArgs::Create` is used unconditionally with no failure-handling callback, for both `error_refund`-enabled and `error_refund`-disabled builds.
