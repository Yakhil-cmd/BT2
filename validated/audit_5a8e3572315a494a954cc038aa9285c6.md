### Title
`skip_stateful_validations` Circular Dependency Allows Perpetual Admission of Unvalidated Invoke Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` skips the `__validate__` entry-point for an invoke transaction with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true`. The function's own output — the unvalidated nonce=1 invoke transaction it just admitted — is itself sufficient to make `account_tx_in_pool_or_recent_block` return `true` for every subsequent nonce=1 invoke transaction from the same address. Once bootstrapped with a single valid `deploy_account` transaction, an attacker can maintain an invalid-signature invoke transaction in the mempool indefinitely, with `__validate__` never running, by repeatedly replacing it via fee escalation.

---

### Finding Description

**The anchor and the mechanism that corrupts it**

`skip_stateful_validations` is designed to improve UX for the `deploy_account + invoke` pattern. It skips `__validate__` for an invoke transaction when three conditions hold simultaneously:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 434-458
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The anchor value is `account_tx_in_pool_or_recent_block`. The code comment explains the intended invariant:

> "it means that either it has a deploy_account transaction or transactions with future nonces that **passed validations**."

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  line 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` if **any** transaction from that address is in the pool — including a nonce=1 invoke transaction that was itself admitted by the skip mechanism without running `__validate__`. [3](#0-2) 

**The circular dependency**

When `skip_validate=true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, so the blockifier's `StatefulValidator` never calls the `__validate__` entry point:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 308-312
let only_query = false;
let charge_fee = enforce_fee(executable_tx, only_query);
let strict_nonce_check = false;
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

The admitted unvalidated invoke transaction is then stored in `tx_pool`. On the next call to `account_tx_in_pool_or_recent_block` for the same address, `tx_pool.contains_account` returns `true` — not because a `deploy_account` exists, but because the previously admitted unvalidated invoke transaction is still in the pool.

**Attack sequence**

1. Attacker pre-funds address `A` with fee tokens (tokens sent to an undeployed address are visible in state).
2. Attacker submits a valid `deploy_account` tx for `A` (nonce=0) → admitted to mempool. `account_tx_in_pool_or_recent_block(A)` = `true`.
3. Attacker submits an invoke tx for `A` with `nonce=1` and an **invalid signature** → `skip_stateful_validations` returns `true` (deploy_account is in pool) → `__validate__` is skipped → `verify_can_pay_committed_bounds` passes (pre-funded balance) → **admitted to mempool without signature verification**.
4. The deploy_account tx expires after TTL (default 60 s per `add_tx_old_transactions_cleanup`). Only the invalid-signature invoke tx remains.
5. `account_tx_in_pool_or_recent_block(A)` is still `true` — now because the invalid-signature invoke tx is in the pool.
6. Attacker submits a new invoke tx for `A` with `nonce=1`, invalid signature, and a higher fee (fee escalation). `skip_stateful_validations` returns `true` again (the existing invalid-signature invoke tx satisfies the anchor check). `__validate__` is skipped. The old tx is replaced. Loop continues indefinitely. [5](#0-4) 

The comment's invariant — "transactions with future nonces that **passed validations**" — is broken: the nonce=1 invoke tx that satisfies the anchor check is itself the product of the skip mechanism and never passed `__validate__`.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Invalid-signature invoke transactions are admitted to the mempool and persist indefinitely. When the batcher pulls them for execution, `__validate__` runs (with `validate=true` in the batcher's execution flags), fails because the account is undeployed or the signature is invalid, and the transaction is rejected without being included in a block. The attacker wastes batcher execution resources and occupies mempool capacity. Because `__validate__` failure in Starknet does not charge a fee, the pre-funded tokens at address `A` are never consumed, making the attack economically self-sustaining after the one-time bootstrap cost.

---

### Likelihood Explanation

**Likelihood: Medium.**

The attack requires:
- One valid `deploy_account` transaction to bootstrap (standard cost).
- Pre-funding the target address with fee tokens (one-time transfer).
- Periodic fee-escalation replacements to keep the invalid invoke tx alive past TTL.

No privileged access is required. Any unprivileged user can execute this against any Starknet sequencer running this gateway code with `skip_stateful_validations` enabled.

---

### Recommendation

The anchor check must be restricted to verify that a **`deploy_account` transaction specifically** is present in the pool for the address, not merely that any transaction from that address exists. Replace the broad `account_tx_in_pool_or_recent_block` check with a targeted query such as `deploy_account_tx_in_pool(address)` that only returns `true` when a `DeployAccount` transaction is present. Alternatively, document the known limitation that the skip mechanism can be bootstrapped by any pooled transaction from the address, and accept the resulting mempool-pollution risk.

   → __validate__ is NOT called
   → fee escalation replaces old tx
   → new invalid-sig tx admitted  ✓

   (failed __validate__ during batcher execution charges no fee in Starknet).

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
```
