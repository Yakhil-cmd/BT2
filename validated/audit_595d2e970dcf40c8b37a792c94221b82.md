### Title
Rewards Permanently Locked in Pool Contract When `pool_balance < min_delegation_for_rewards` - (File: `src/pool/pool.cairo`)

### Summary

When the staking contract calculates and transfers STRK rewards to a pool contract, but the pool's `pool_balance` is below `min_delegation_for_rewards`, the pool's `compute_rewards_per_unit` silently returns zero. The STRK tokens are already in the pool contract but the `cumulative_rewards_trace` is never updated, making those rewards permanently unclaimable by any pool member.

### Finding Description

The reward distribution flow involves two steps that are not atomically consistent:

**Step 1 — Staking contract calculates and transfers rewards** (`staking.cairo`, `calculate_staker_pools_rewards` + `update_pool_rewards`):

In `calculate_staker_pools_rewards`, the staking contract computes `pool_rewards` as:

```
pool_rewards_including_commission = total_rewards * pool_balance / total_stake
```

If `pool_balance` is non-zero and equals `total_stake` (e.g., the only BTC delegator), this equals `total_rewards`, which is non-zero. The pool is appended to `pool_rewards_array`. [1](#0-0) 

Then `update_pool_rewards` **unconditionally transfers** the computed STRK rewards to the pool contract before calling `update_rewards_from_staking_contract`: [2](#0-1) 

**Step 2 — Pool contract records zero interest** (`pool.cairo`, `compute_rewards_per_unit`):

Inside `update_rewards_from_staking_contract`, `compute_rewards_per_unit` is called. When `total_stake < min_delegation_for_rewards`, it returns **zero**: [3](#0-2) 

The `cumulative_rewards_trace` is updated with `last + 0`, so no sigma increase is recorded: [4](#0-3) 

For STRK pools, `min_delegation_for_rewards = 10^18` (1 STRK). For BTC pools with 8 decimals, `min_delegation_for_rewards = 10^3 = 1000 satoshis`. [5](#0-4) 

Since `cumulative_rewards_trace` never increases, `calculate_rewards` computes zero rewards for all pool members for that epoch. The STRK tokens sitting in the pool contract are permanently inaccessible — there is no admin withdrawal function in the pool contract. [6](#0-5) 

### Impact Explanation

STRK reward tokens transferred to the pool contract are permanently frozen. No pool member can ever claim them because the sigma-based accounting (`cumulative_rewards_trace`) was not updated. The pool contract has no recovery path — the only outflow of STRK is through `claim_rewards`, which is gated on the cumulative trace.

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

For **BTC pools** (8-decimal token), the threshold is only 1000 satoshis. Any unprivileged delegator can enter a BTC pool with 1–999 satoshis. If they are the sole BTC delegator for that staker, `btc_total_stake == pool_balance`, so `pool_rewards = btc_total_rewards` (non-zero), STRK is transferred to the pool, and `compute_rewards_per_unit` returns zero. This is directly reachable with no privileged access.

For **STRK pools**, the threshold is 1 STRK. A delegator can enter with any amount > 0 (no minimum enforced in `enter_delegation_pool`). If the pool balance is below 1 STRK and constitutes a meaningful fraction of `strk_total_stake`, the same lock occurs. [7](#0-6) 

### Recommendation

The transfer of rewards to the pool and the accounting update must be consistent. Two options:

1. **Guard the transfer**: In `calculate_staker_pools_rewards`, skip pools whose `pool_balance < min_delegation_for_rewards` (do not append to `pool_rewards_array`), so no STRK is transferred when the pool cannot account for it.

2. **Enforce a minimum delegation**: Add a `min_delegation_for_rewards` check in `enter_delegation_pool` and `add_to_delegation_pool` so that a pool's balance can never fall into the unaccountable range.

### Proof of Concept

1. Staker stakes STRK, opens a BTC pool (8-decimal token), sets commission = 0.
2. Delegator calls `enter_delegation_pool` with `amount = 999` satoshis (below `min_delegation_for_rewards = 1000`).
3. After K epochs, the delegation becomes effective. Delegator is the only BTC delegator.
4. Staker attests → `update_rewards_from_attestation_contract` is called.
5. In `calculate_staker_pools_rewards`: `btc_total_stake = 999`, `pool_balance = 999`, `pool_rewards = btc_total_rewards * 999 / 999 = btc_total_rewards` (non-zero). Pool appended to `pool_rewards_array`.
6. In `update_pool_rewards`: `btc_total_rewards` worth of STRK is transferred to the pool contract via `send_rewards_to_delegation_pool`.
7. `update_rewards_from_staking_contract(rewards: btc_total_rewards, pool_balance: 999)` is called on the pool.
8. `compute_rewards_per_unit(staking_rewards: btc_total_rewards, total_stake: 999)` → returns `0` because `999 < 1000`.
9. `cumulative_rewards_trace` is updated with `last + 0`.
10. Delegator calls `claim_rewards` → receives 0. STRK rewards are permanently locked in the pool contract. [3](#0-2) [8](#0-7) [9](#0-8)

        }

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }

        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
