### Title
Unverified Participant Contributions Allow Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator aggregates each participant's `(big_y, big_c)` contribution without any cryptographic proof of correct formation. A single malicious participant can substitute arbitrary group elements, causing the coordinator to produce a silently incorrect CKD output that all honest parties accept as valid.

---

### Finding Description

The CKD protocol computes a confidential key `msk · H(pk ∥ app_id)` by having each participant contribute a Lagrange-weighted ElGamal share:

- `big_y_i = y_i · G`
- `big_s_i = x_i · H(pk ∥ app_id)` (private share contribution)
- `big_c_i = big_s_i + y_i · app_pk`
- `norm_big_y_i = λ_i · big_y_i`, `norm_big_c_i = λ_i · big_c_i`

The coordinator sums all contributions and the client unmasks with `app_sk`:

```
sum_big_c − app_sk · sum_big_y = msk · H(pk ∥ app_id)
```

In `do_ckd_coordinator`, the coordinator receives each participant's `CKDOutput` and adds it directly with no verification: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no zero-knowledge proof, commitment binding, or any other check that the received `(big_y, big_c)` satisfies the relation `big_c = λ_i · (x_i · H(pk ∥ app_id) + y_i · app_pk)` for the participant's actual private share `x_i`. Compare this with the DKG protocol, which enforces a proof-of-knowledge and commitment hash before accepting any participant's polynomial: [2](#0-1) 

No equivalent verification exists in the CKD path.

The `compute_signature_share` function that produces the correct values is only called locally; the coordinator has no way to enforce that remote participants ran the same computation: [3](#0-2) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

If participant `P_m` sends `(big_y', big_c') = (G, G)` instead of its correct Lagrange-weighted share, the coordinator computes:

```
sum_big_c' − app_sk · sum_big_y'
  = [msk − λ_m · x_m] · H(pk ∥ app_id) + (1 − app_sk) · G
```

This is not `msk · H(pk ∥ app_id)`. The client's `unmask` call: [4](#0-3) 

produces a silently wrong confidential key. There is no integrity check on the coordinator's output, so neither the coordinator nor the client can detect the corruption. The derived key is permanently wrong for that `app_id`, constituting a full denial of the CKD service for honest parties.

---

### Likelihood Explanation

**Medium.** Any single participant in the threshold group — not a threshold-sized coalition — can trigger this. The attacker only needs to be enrolled as one of the `n` participants and send a malformed message in a single protocol round. No cryptographic break or external compromise is required. The attack is silent: the coordinator returns `Ok(Some(ckd_output))` with no error.

---

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — for example, a Schnorr proof demonstrating knowledge of `y_i` such that `big_y_i = y_i · G` and `big_c_i − big_s_i = y_i · app_pk`, where `big_s_i` is bound to the participant's committed public key share from DKG. The coordinator must verify all proofs before aggregating. This mirrors the proof-of-knowledge verification already present in `do_keyshare`: [5](#0-4) 

---

### Proof of Concept

1. All `n` participants run `ckd(...)`. Honest participants compute correct `(norm_big_y_i, norm_big_c_i)` via `compute_signature_share`.
2. Malicious participant `P_m` intercepts the send at line 30 and instead transmits `(G, G)` — the generator point for both fields — to the coordinator.
3. The coordinator's loop at lines 50–55 adds `(G, G)` to the running sum without rejection.
4. The coordinator returns `CKDOutput { big_y: Σ_honest(norm_big_y_i) + G, big_c: Σ_honest(norm_big_c_i) + G }`.
5. The client calls `ckd_output.unmask(app_sk)` and obtains `[msk − λ_m · x_m] · H(pk ∥ app_id) + (1 − app_sk) · G`, which is not the intended confidential key.
6. No error is raised anywhere in the protocol; the wrong key is silently accepted. [6](#0-5)

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
