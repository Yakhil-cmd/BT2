### Title
`starknet_estimateMessageFee` Always Returns `overall_fee = 0` Regardless of Actual Gas Consumed - (`crates/blockifier/src/transaction/l1_handler_transaction.rs`)

### Summary

`l1_handler_tx_execution_info` unconditionally zeroes `receipt.fee` before returning `TransactionExecutionInfo`. The fee estimation pipeline reads `overall_fee` directly from `receipt.fee`, so `starknet_estimateMessageFee` always returns `overall_fee = Fee(0)` even when `gas_consumed` is non-zero, violating the OpenRPC spec invariant that `overall_fee = gas_consumed * gas_price + data_gas_consumed * data_gas_price`.

### Finding Description

**Step 1 — RPC entry point.**
`estimate_message_fee` wraps the caller-supplied `MessageFromL1` as `ExecutableTransactionInput::L1Handler(message.into(), Fee(u128::MAX), false)` and dispatches to `exec_estimate_fee`. [1](#0-0) 

**Step 2 — Execution.**
`execute_raw` on `L1HandlerTransaction` runs the entry point, computes a real `TransactionReceipt` (with non-zero `gas` and `fee` fields), then calls `l1_handler_tx_execution_info`. [2](#0-1) 

**Step 3 — Unconditional zero.**
`l1_handler_tx_execution_info` overwrites `receipt.fee = Fee(0)` unconditionally before constructing `TransactionExecutionInfo`. This applies to every call path: success, fee-check revert, and execution error. [3](#0-2) 

**Step 4 — Fee estimation reads the zeroed field.**
`tx_execution_output_to_fee_estimation` populates `overall_fee` directly from `receipt.fee`. Because that field was zeroed in step 3, `overall_fee` is always `Fee(0)` regardless of `gas_consumed`. [4](#0-3) 

**Step 5 — Confirmed by existing test.**
The existing `call_estimate_message_fee` test explicitly asserts `gas_consumed: felt!("0x3933")` (non-zero) alongside `overall_fee: Fee(0)`, and even carries a `TODO` acknowledging the fee is not verified correctly. [5](#0-4) 

The OpenRPC spec definition of `FEE_ESTIMATE` states `overall_fee` equals `gas_consumed * gas_price + data_gas_consumed * data_gas_price`. With `gas_consumed = 0x3933` and a non-zero `l1_gas_price`, the correct `overall_fee` is `0x3933 * l1_gas_price`, not `0`. [6](#0-5) 

### Impact Explanation

Any client calling `starknet_estimateMessageFee` and reading `overall_fee` to determine the L1 fee to attach to a message will receive `0`. This is an authoritative-looking wrong value from a public, unprivileged RPC endpoint. Clients that trust `overall_fee` (rather than manually computing `gas_consumed * l1_gas_price`) will systematically underestimate the required fee. This maps directly to: **High — RPC fee estimation returns an authoritative-looking wrong value.**

### Likelihood Explanation

The trigger is trivially reachable: any unauthenticated caller can invoke `starknet_estimateMessageFee` with a valid `MessageFromL1` targeting any deployed contract with an `l1_handler` entry point. No special privileges are required. The zeroing is unconditional, so every call is affected.

### Recommendation

In `tx_execution_output_to_fee_estimation`, compute `overall_fee` from the gas vector and gas prices directly rather than reading `receipt.fee`:

```rust
overall_fee: Fee(
    gas_vector.l1_gas.0 * l1_gas_price.0
    + gas_vector.l1_data_gas.0 * l1_data_gas_price.0
),
```

Alternatively, populate `receipt.fee` correctly before calling `l1_handler_tx_execution_info` when the call originates from a fee-estimation context (e.g., via an `only_query` flag), and keep the zero only for actual sequenced execution where L1 handlers genuinely do not pay L2 fees.

### Proof of Concept

The existing test already demonstrates the divergence without any additional setup:

```

l1_gas_price  = GAS_PRICE.price_in_wei  (non-zero)
overall_fee   = Fee(0)  ← always zero due to unconditional receipt.fee = Fee(0)
```
