### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield — (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in the delegation pool contract contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace. A pool member who modifies their delegation balance once per epoch without claiming rewards grows this trace by one entry per epoch. After a sufficient number of epochs, the gas cost of iterating the full trace in a single `claim_rewards` call will exceed the Starknet transaction gas limit, permanently freezing the pool member's unclaimed yield with no recovery path.

---

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:**

In `src/pool/pool.cairo`, the internal `calculate_rewards` function (lines 837–888) contains a `while` loop that iterates over every entry in `pool_member_epoch_balance` from the stored `entry_to_claim_from` index up to the current epoch:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The developers themselves flag this as unbounded. Each loop iteration performs at least one storage read (`pool_member_trace.at(...)`) and calls `find_sigma`, which performs additional storage reads from `cumulative_rewards_trace`.

**How the trace grows:**

Every call to `set_member_balance` or `increase_member_balance` invokes `trace.insert(key: self.get_epoch_plus_k(), value: ...)`. The `insert` implementation only deduplicates entries sharing the same epoch key; a different epoch always appends a new checkpoint:

```
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

`set_member_balance` / `increase_member_balance` are called from every balance-mutating entry point:

- `enter_delegation_pool` → `set_member_balance` [3](#0-2) 
- `add_to_delegation_pool` → `increase_member_balance` [4](#0-3) 
- `exit_delegation_pool_intent` → `set_member_balance` [5](#0-4) 
- `enter_delegation_pool_from_staking_contract` → `increase_member_balance` / `set_member_balance` [6](#0-5) 

Therefore, one balance change per epoch appends exactly one new trace entry per epoch.

**Why `entry_to_claim_from` does not protect against this:**

The `entry_to_claim_from` pointer is only advanced inside a successful `claim_rewards` call:

```
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [7](#0-6) 

If `claim_rewards` reverts (out of gas), the pointer is never advanced. The next call starts from the same position and faces the same (or larger) trace, creating a permanent deadlock. There is no partial-claim mechanism.

**Attacker-controlled entry path:**

`add_to_delegation_pool` is callable by the pool member **or** their reward address:

```
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
``` [8](#0-7) 

The reward address is an unprivileged actor listed explicitly in the bounty scope. A reward address (e.g., an auto-compounding contract) that calls `add_to_delegation_pool` once per epoch on behalf of a pool member will grow the trace without the pool member's active participation. Once the trace is large enough, neither the pool member nor the reward address can successfully call `claim_rewards`.

---

### Impact Explanation

When `claim_rewards` consistently reverts due to gas exhaustion, the pool member's entire accumulated unclaimed yield is permanently frozen. There is no alternative withdrawal path, no partial-claim function, and no admin escape hatch to drain the rewards. This matches **"Permanent freezing of unclaimed yield"** (High).

---

### Likelihood Explanation

The likelihood is **medium**:

- Epochs are finite-duration; a pool member active for hundreds of epochs without claiming rewards (e.g., a smart-contract delegator that auto-compounds via `add_to_delegation_pool` but never calls `claim_rewards`) will accumulate a proportionally large trace.
- The reward address — an unprivileged actor — can independently trigger `add_to_delegation_pool` each epoch, growing the trace without the pool member's cooperation.
- The developers themselves acknowledge the loop is unbounded (line 858), confirming awareness of the gas risk.
- Starknet's per-transaction gas ceiling is finite; a trace of sufficient depth will breach it.

---

### Recommendation

1. **Cap unclaimed balance changes**: Enforce a maximum number of unprocessed trace entries (analogous to the 10-action-per-proposal cap in the referenced report). Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if `pool_member_trace_length - entry_to_claim_from` exceeds a safe threshold (e.g., 100).
2. **Partial claim**: Allow `claim_rewards` to accept an optional `max_entries` parameter so the caller can process the trace in bounded batches, advancing `entry_to_claim_from` incrementally.
3. **Claim-before-modify**: Require a successful `claim_rewards` (or an explicit checkpoint update) before any balance-mutating operation that would append a new trace entry.

---

### Proof of Concept

1. Pool member `P` enters a delegation pool with the minimum amount.
2. In each of epochs `E`, `E+1`, …, `E+N−1`, `P` (or `P`'s reward address) calls `add_to_delegation_pool` with 1 token. Each call appends one entry to `pool_member_epoch_balance[P]` because the epoch key differs each time.
3. `P` never calls `claim_rewards` during this period, so `entry_to_claim_from` remains at its initial value.
4. After `N` epochs, `P` calls `claim_rewards`. `calculate_rewards` enters the `while` loop and must iterate all `N` entries, each requiring storage reads and `find_sigma` computation.
5. For sufficiently large `N`, the transaction exceeds the Starknet gas limit and reverts.
6. Every subsequent `claim_rewards` attempt faces the same (or larger) trace and also reverts.
7. `P`'s entire accumulated yield is permanently frozen with no recovery path.
