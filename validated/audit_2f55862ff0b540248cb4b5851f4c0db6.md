### Title
Unvalidated `app_pk` Identity Element in CKD Protocol Discloses Confidential Derived Key to Coordinator - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` function and `compute_signature_share()` in the Confidential Key Derivation protocol accept `app_pk` (the TEE application's ElGamal public key `A`) without validating that it is a non-identity G1 point. If a caller supplies `app_pk = G1Projective::identity()`, the ElGamal blinding term vanishes, and the coordinator directly receives the unblinded BLS signature `msk · H(pk, app_id)` — the confidential derived secret — violating the core secrecy guarantee of the CKD protocol.

### Finding Description

The CKD protocol is designed so that each participant computes:

```
C_i = S_i + y_i · A    where S_i = x_i · H(pk, app_id)
```

and sends `(λ_i · Y_i, λ_i · C_i)` to the coordinator. The coordinator aggregates:

```
C = msk · H(pk, app_id) + (Σ λ_i · y_i) · A
```

The app then unmasks with its private scalar `a`: `C − a · Y = msk · H(pk, app_id)`.

The blinding term `y_i · A` is the only mechanism preventing the coordinator from seeing `msk · H(pk, app_id)` directly.

In `compute_signature_share()`:

```rust
let big_c = big_s + app_pk * y.0;
``` [1](#0-0) 

`app_pk` is used directly with no check that it is a non-identity element. If `app_pk = G1Projective::identity()`, then `app_pk * y.0 = identity`, so `big_c = big_s = x_i · H(pk, app_id)`. The coordinator aggregates to `C = msk · H(pk, app_id)` — the raw BLS signature — with no blinding at all.

Neither the public entry point `ckd()` nor any internal function validates `app_pk`:

```rust

    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,

    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
``` [2](#0-1) 

The G1 group deserialization in `BLS12381G1Group` does reject the identity element, but `app_pk` is passed as a native `G1Projective` value and never goes through that deserialization path: [3](#0-2) 

### Impact Explanation

The security specification explicitly states:

> "No single node in the MPC network should be capable of computing `s`. This avoids key leakage in the case a single TEE is compromised." [4](#0-3) 

And:

> "Example values that should not be leaked even if a node is malicious are `s`, `msk` and private shares of other nodes." [5](#0-4) 

When `app_pk = identity`, the coordinator receives `CKDOutput { big_y: Σ λ_i·Y_i, big_c: msk·H(pk, app_id) }`. The coordinator — a single MPC node — now holds the confidential derived secret `msk·H(pk, app_id)` in plaintext, from which `s = HKDF(msk·H(pk, app_id))` is trivially computable. This is a **Critical** impact: disclosure of a confidential derived secret.

### Likelihood Explanation

The `app_pk` parameter originates from the CKD request submitted on-chain by the Developer contract. A malicious developer contract, or any party who can craft a CKD request, can supply `G1Projective::identity()` as `app_pk`. The library imposes no on-chain or off-chain constraint preventing this. The attack requires no cryptographic break — only passing a single crafted group element. A malicious coordinator colluding with a malicious developer contract can silently extract the confidential derived key for any `app_id` without detection.

### Recommendation

Add an explicit identity-element check for `app_pk` at the entry point of `ckd()`, returning an `InitializationError` if the check fails:

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the existing identity check already applied during G1 group deserialization in `BLS12381G1Group::deserialize`. [6](#0-5) 

### Proof of Concept

```rust
// Attacker supplies app_pk = G1Projective::identity()
let malicious_app_pk = G1Projective::identity();

// All participants compute:
//   big_c = big_s + identity * y = big_s = x_i * H(pk, app_id)
// Coordinator aggregates:
//   C = sum(lambda_i * big_s_i) = msk * H(pk, app_id)
//
// CKDOutput { big_y: Y, big_c: msk * H(pk, app_id) }
// Coordinator directly holds the confidential derived key.

let protocol = ckd(
    &participants,
    coordinator,
    me,
    key_pair,
    app_id,
    malicious_app_pk,  // identity element — no error returned
    rng,
).unwrap();
// After protocol completes, coordinator's CKDOutput.big_c == msk * H(pk, app_id)
// s = HKDF(big_c) is now known to the coordinator.

    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });

    // ensure the coordinator is a participant

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
