### Title
Unauthorized Global Contract Namespace Overwrite via Cross-Account Promise Batch — (`runtime/runtime/src/global_contracts.rs`)

### Summary
Any contract executing on account A can register or overwrite a global contract keyed by an arbitrary account B's ID by creating a promise batch targeting B and appending a `DeployGlobalContract { deploy_mode: AccountId }` action. No authorization check enforces that the receipt receiver (whose ID becomes the trie key) must also be the receipt predecessor. This breaks the isolation invariant that `GlobalContractDeployMode::AccountId` is exclusively controlled by the named account.

### Finding Description

`promise_batch_action_deploy_global_contract_by_account_id` appends a `DeployGlobalContractAction { deploy_mode: AccountId }` to any promise batch, regardless of whether the batch targets the calling contract's own account or a third-party account. [5](#0-4) 

When the resulting receipt is executed, `action_deploy_global_contract` receives `account_id` = the receipt receiver (account B). It passes this directly to `initiate_distribution`, which constructs `GlobalContractIdentifier::AccountId(account_id.clone())` as the trie key — with no check that the receipt's predecessor matches the receiver. [6](#0-5) [3](#0-2) 

The trie write then stores attacker-supplied code at `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(B) }`, overwriting any previously legitimate global contract B had deployed under its own ID. [7](#0-6) 

### Impact Explanation

**Corrupted state value:** `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(B) }` — overwritten with attacker-controlled WASM bytes.

**Consequences:**
- Any account that previously called `UseGlobalContract { contract_identifier: AccountId(B) }` now executes attacker-supplied code on every function call, with full access to that account's storage and balance.
- Account B's ability to update its own global contract is not blocked, but the nonce-based idempotency mechanism means a subsequent legitimate deploy by B with a higher nonce will overwrite the attacker's version — however, the attacker can race again.
- Storage costs for the malicious deployment are deducted from account B's balance (not A's), constituting an unauthorized balance drain. [8](#0-7) 

### Likelihood Explanation

Reachable by any deployed contract on mainnet. The attacker only needs to:
1. Deploy a contract on account A (standard operation).
2. Call `promise_batch_create("victim.near")` then `promise_batch_action_deploy_global_contract_by_account_id(...)` with malicious WASM.
3. Pay gas and the storage cost (deducted from victim's balance, not attacker's).

No special privileges, validator access, or node-level access required.

### Recommendation

In `action_deploy_global_contract`, add an authorization check before calling `initiate_distribution` when `deploy_mode == AccountId`:

```rust
if deploy_contract.deploy_mode == GlobalContractDeployMode::AccountId {
    // The receipt predecessor must be the account itself (self-initiated action)
    // OR enforce via a separate validation step that predecessor_id == account_id.
    // Alternatively, reject DeployGlobalContract(AccountId) actions in receipts
    // whose predecessor_id != receiver_id.
}
```

The simplest correct fix is to reject `DeployGlobalContract { deploy_mode: AccountId }` in any receipt where `predecessor_id != receiver_id`, enforcing that only the account itself can register a global contract under its own ID.

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {

        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;

        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
