### Title
Missing Caller Access Control on `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo`)

### Summary

The `update_rewards` function is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for the current block and permanently discarding that block's rewards for all stakers.

### Finding Description

The spec explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation, however, only calls `self.general_prerequisites()` (a pause check) and performs no caller identity verification: [2](#0-1) 

The function uses a single **global** `last_reward_block` storage variable as a per-block gate: [3](#0-2) 

Once any call succeeds for block N, `last_reward_block` is written to N: [4](#0-3) 

When `disable_rewards` is `true`, the function returns immediately after writing `last_reward_block`, distributing nothing: [5](#0-4) 

Any subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`. The block's reward opportunity is permanently gone.

### Impact Explanation

An attacker can front-run the sequencer's `update_rewards` call every block with `disable_rewards: true`. Because `last_reward_block` is global (not per-staker), a single hostile call per block permanently suppresses rewards for **all** stakers for that block. Repeated across many blocks, this constitutes permanent freezing of unclaimed yield. The rewards are never minted/attributed; they cannot be recovered.

**Impact: High — Permanent freezing of unclaimed yield.**

### Likelihood Explanation

- No privilege required; any EOA or contract can call `update_rewards`.
- The attacker only needs to submit a transaction before the sequencer's reward update transaction in the same block. On Starknet, transaction ordering within a block is controlled by the sequencer, but the sequencer is the Starkware sequencer — not the staking contract caller. Any user transaction can be included in the same block before the sequencer's staking reward call.
- The attack is cheap (one transaction per block) and repeatable indefinitely.
- The attacker can target any valid `staker_address` (must be an active staker with non-zero balance to pass the guards), but the damage is global due to the shared `last_reward_block`.

### Recommendation

Add a caller check at the top of `update_rewards` to enforce the "Only starkware sequencer" access control stated in the spec. This could be implemented as a role check (e.g., a `SEQUENCER` role stored in contract state) or by asserting `get_caller_address() == sequencer_address`. The check should be placed before any state writes.

### Proof of Concept

1. Deploy the staking contract with two active stakers, A and B, both past the K-epoch activation window with consensus rewards enabled.
2. At block N, attacker (any address) calls:
   ```
   update_rewards(staker_address: A, disable_rewards: true)
   ```
3. `last_reward_block` is written to N. No rewards are distributed.
4. Sequencer attempts:
   ```
   update_rewards(staker_address: A, disable_rewards: false)
   ```
   → Reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat step 2–4 every block. Both stakers A and B accumulate zero rewards indefinitely, matching the test behavior already demonstrated in the test suite: [6](#0-5) [7](#0-6) 

The existing test at line 3887 already confirms that calling with `disable_rewards: true` followed by any second call in the same block yields `REWARDS_ALREADY_UPDATED` — the attacker simply needs to be first.
