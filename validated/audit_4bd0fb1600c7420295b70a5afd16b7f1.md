### Title
Cross-COA Ownership Proof Replay via Unsigned `CapabilityPath` — (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`validateCOAOwnershipProof` verifies that a Cadence account owns a specific COA by checking cryptographic signatures over caller-supplied `signedData`. However, the `CapabilityPath` field inside the `COAOwnershipProof` struct is **not covered by those signatures**. An attacker who obtains a legitimately produced proof for COA₁ can decode the RLP blob, swap `CapabilityPath` to point at COA₂ (a different COA owned by the same Cadence account), re-encode it, and submit it to `isValidSignature` on COA₂. The function returns `isValid: true`, impersonating COA₂ in ERC-1271 verification without the account holder's knowledge or consent.

---

### Finding Description

`COAOwnershipProof` is defined in `fvm/evm/types/proof.go` and contains four fields:

```
KeyIndices     KeyIndices
Address        FlowAddress
CapabilityPath PublicPath
Signatures     Signatures
``` [1](#0-0) 

The struct is RLP-encoded/decoded with no integrity protection over `CapabilityPath`: [2](#0-1) [3](#0-2) 

`validateCOAOwnershipProof` in the EVM system contract performs three checks:

1. Verify `signatures` over `signedData` using keys from the Cadence account at `address`.
2. Borrow the COA resource from `path` on that account.
3. Assert the COA's EVM address equals the supplied `evmAddress`. [4](#0-3) 

Because `CapabilityPath` is **not** part of `signedData`, an attacker can freely substitute it. The code itself acknowledges this gap: [5](#0-4) 

The EVM-side entry point is `isValidSignature` on the COA Solidity contract, which passes `address(this)` (the calling COA's EVM address) and the caller-supplied `_sig` blob directly to the `verifyCOAOwnershipProof` precompile: [6](#0-5) 

The precompile decodes the blob and forwards it to `validateCOAOwnershipProof`: [7](#0-6) [8](#0-7) 

**Attack path:**

1. Account A owns COA₁ (at `/public/coa1`, EVM address `0xAAA`) and COA₂ (at `/public/coa2`, EVM address `0xBBB`).
2. Account A produces a legitimate proof for COA₁ over some `signedData = H`.
3. Attacker intercepts the RLP-encoded proof, decodes it, changes `CapabilityPath` from `/public/coa1` to `/public/coa2`, and re-encodes.
4. Attacker calls `COA₂.isValidSignature(H, modifiedProof)`.
5. The precompile invokes `validateCOAOwnershipProof(address=A, path=/public/coa2, signedData=H, …, evmAddress=0xBBB)`.
6. Signatures verify (they were produced by Account A's keys over `H`); COA₂ exists at `/public/coa2` with EVM address `0xBBB` → function returns `isValid: true`.
7. Attacker has successfully impersonated COA₂ in ERC-1271 verification.

---

### Impact Explanation

ERC-1271 `isValidSignature` is the standard mechanism by which on-chain contracts (DEXes, lending protocols, multisigs, NFT marketplaces) verify that a smart-contract wallet has authorised an action. A successful replay allows the attacker to:

- Satisfy ERC-1271 checks on behalf of COA₂ without COA₂'s authorisation.
- Drain or transfer assets from contracts that gate withdrawals or approvals on COA₂'s ERC-1271 signature.
- Bypass multisig or governance thresholds that count COA₂ as a distinct signer.

This is a direct, on-chain asset-loss vector for any account that holds multiple COAs.

---

### Likelihood Explanation

**Preconditions:**
- The victim Cadence account must hold at least two COAs (non-default but supported by the protocol and increasingly common as EVM usage grows).
- The attacker must obtain a valid proof blob for COA₁. Proofs are transmitted off-chain (wallet ↔ dApp) and may also appear in on-chain calldata of past transactions, making interception realistic.

**No privileged access required.** The attacker only needs to call a public EVM function (`isValidSignature`) with a crafted `bytes` argument. The modification to the RLP blob is trivial (decode, swap one string field, re-encode).

---

### Recommendation

Enforce that `signedData` commits to the specific COA being proved. The simplest fix is to require `signedData` to encode `evmAddress` (the target COA's EVM address) before signature verification, so a proof produced for COA₁ cannot satisfy the check for COA₂. This should be enforced **inside `validateCOAOwnershipProof`** rather than left as a caller convention:

```cadence
// Enforce that signedData encodes evmAddress
let encodedEVMAddress: [UInt8] = evmAddress.toConstantSized()!.toVariableSized()
// require signedData to contain evmAddress bytes
pre {
    signedData.length >= 20 && signedData.slice(from: signedData.length - 20, upTo: signedData.length) == encodedEVMAddress:
        "signedData must commit to the target evmAddress"
}
```

Alternatively, sign over the full proof struct (including `CapabilityPath` and `evmAddress`) rather than over an opaque `signedData` blob.
