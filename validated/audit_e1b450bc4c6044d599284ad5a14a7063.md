### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolBinAdditionalFees` allows the pool admin to write per-bin additional spread fees (`addFeeBuyE6` / `addFeeSellE6`) with **no validation against `maxAdminSpreadFeeE6`**. Because these values are added directly on top of the base spread fee inside every swap path, the pool admin can impose an effective per-bin fee that exceeds the factory-enforced cap, extracting more value from traders than the cap permits.

---

### Finding Description

The factory enforces `maxAdminSpreadFeeE6` on the base admin spread fee through `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, the sibling function `setPoolBinAdditionalFees` passes the caller-supplied values straight through to the pool with **no cap check**:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

On the pool side, `setBinAdditionalFees` writes the values directly to `BinState` storage with only a bin-index range check: [3](#0-2) 

In every swap path, the per-bin additional fee is added **on top of** the oracle-derived base fee before the swap math executes:

```solidity
// MetricOmmPool.sol:910 (buy token0, specified-out)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the pool admin can set them to any value up to `65535` (≈ 6.5535 % in E6 scale) on any bin, regardless of what `maxAdminSpreadFeeE6` is set to.

The same gap exists at pool creation: `_unpackAndValidateBinStates` validates only `lengthE6` (bin distances) and never checks `addFeeBuyE6` / `addFeeSellE6` against the cap. [8](#0-7) 

---

### Impact Explanation

Every trader whose swap routes through an affected bin pays `spreadFeeE6 + addFeeBuyE6` (or `addFeeSellE6`) instead of the capped `spreadFeeE6`. The excess fee accrues as pool surplus and is swept to the admin and protocol via `collectFees`. This is a direct, quantifiable loss of trader principal on every swap through the manipulated bin, constituting an admin-boundary break where the semi-trusted pool admin exceeds the factory-enforced fee cap.

---

### Likelihood Explanation

The pool admin can call `setPoolBinAdditionalFees` at any time while the pool is live — there is no timelock, no pause requirement, and no factory-owner co-signature. The trigger is a single transaction from the pool admin address, which is a valid semi-trusted actor. The factory owner's ability to lower `maxAdminSpreadFeeE6` does not retroactively remove already-written bin additional fees.

---

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Apply the same validation to the bin data arrays unpacked in `_unpackAndValidateBinStates` at pool creation.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(…, maxAdminSpreadFeeE6 = 10_000, …)` — admin spread cap is 1 %.
2. Pool admin calls `setPoolAdminFees(pool, 10_000, 0)` — base admin spread fee is at the cap (1 %).
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert; bin 0 now carries an additional 6.5535 % fee in both directions.
4. A trader swaps through bin 0. The effective fee applied is `spreadFeeE6 (1 %) + addFeeBuyE6 (6.5535 %) = 7.5535 %`, far above the 1 % cap the factory owner intended.
5. The excess fee accrues as pool surplus and is collected by the admin via `collectPoolFees`. [9](#0-8) [2](#0-1) [3](#0-2)
