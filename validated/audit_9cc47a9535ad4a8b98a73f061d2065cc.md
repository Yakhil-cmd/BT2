### Title
Unbounded Loop in `calculate_rewards` Over `pool_member_epoch_balance` Trace Enables Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` internal function in the Pool contract iterates over an unbounded `pool_member_epoch_balance` trace. The developers explicitly acknowledge this in a code comment. A delegator who repeatedly calls `add_to_delegation_pool` across many epochs without claiming rewards will grow this trace without bound, eventually causing `claim_rewards` to revert due to gas exhaustion, permanently freezing their unclaimed yield.

### Finding Description

In `src/pool/pool.cairo`, the `calculate_rewards` function contains an unbounded `while` loop that iterates over every entry in the `pool_member_epoch_balance` trace for a given pool member: [1](#0-0) 

The developers themselves flag this in a comment:

> `// **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.`

The trace grows every time `set_member_balance` is called, which is invoked by `increase_member_balance`: [2](#0-1) 

`increase_member_balance` is called from `add_to_delegation_pool` on every invocation: [3](#0-2) 

Each call to `add_to_delegation_pool` in a new epoch inserts a new entry into the trace at key `current_epoch + K`: [4](#0-3) 

The `entry_to_claim_from` field in `InternalPoolMemberInfoV1` tracks the starting index for the loop and is only advanced when `claim_rewards` is successfully called: [5](#0-4) 

If a delegator never calls `claim_rewards` while repeatedly calling `add_to_delegation_pool` across epochs, `entry_to_claim_from` stays at 0 while `pool_member_trace_length` grows without bound. The loop in `calculate_rewards` must then iterate over the entire accumulated trace on the next `claim_rewards` call.

`calculate_rewards` is invoked both from `claim_rewards` (state-changing) and `pool_member_info_v1` (view): [6](#0-5) 

### Impact Explanation

Once the trace is large enough, every call to `claim_rewards` for that pool member will revert due to gas exhaustion. The delegator's accumulated yield becomes permanently unclaimable — a **permanent freezing of unclaimed yield** (High impact).

### Likelihood Explanation

The only precondition is that a delegator calls `add_to_delegation_pool` with any non-zero amount once per epoch without claiming rewards. The minimum amount is 1 wei (`amount.is_non_zero()` is the only check). Starknet epochs are finite in duration, so an attacker can systematically grow the trace over time. No privileged access is required — any pool member can trigger this against themselves, or a griefing actor could do so to a victim's address if they control the reward address (which is allowed to call `add_to_delegation_pool`): [7](#0-6) 

### Recommendation

1. **Enforce a maximum trace length** per pool member by capping the number of balance changes allowed without an intervening `claim_rewards` call.
2. **Batch/paginate `claim_rewards`**: Accept a `max_entries` parameter so the loop processes a bounded number of trace entries per call, storing the updated `entry_to_claim_from` for the next call.
3. **Deduplicate trace entries within the same epoch**: If `set_member_balance` is called multiple times in the same epoch, overwrite the existing entry rather than appending, reducing trace growth to at most one entry per epoch.

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with minimum stake.
2. Delegator calls `add_to_delegation_pool(pool_member, 1)` once per epoch for N epochs, never calling `claim_rewards`.
3. After N epochs, `pool_member_epoch_balance.entry(pool_member).length()` equals N.
4. Delegator calls `claim_rewards`. The loop at line 859 iterates N times. For sufficiently large N (e.g., thousands of epochs), the transaction exceeds the Starknet gas limit and reverts.
5. All subsequent `claim_rewards` calls also revert. The delegator's accumulated yield is permanently frozen.
