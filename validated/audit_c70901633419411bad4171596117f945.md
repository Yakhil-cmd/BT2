### Title
Unbounded `MessageBuffer` in `Protocol::message()` Allows Malicious Participant to Exhaust Honest Party Memory - (File: src/protocol/internal.rs)

---

### Summary

The `Protocol::message()` public API feeds incoming bytes directly into a `MessageBuffer` backed by an unbounded `HashMap` of unbounded MPSC channels. A malicious participant can call `message()` with crafted payloads targeting arbitrary `(channel, waitpoint)` header combinations that the honest protocol state machine never reads, causing those entries to accumulate in memory without bound. The codebase itself contains a test that explicitly confirms this attack succeeds.

---

### Finding Description

**Root cause — `src/protocol/internal.rs`**

The `Protocol` trait exposes a single public ingestion point:

```rust
fn message(&mut self, from: Participant, data: MessageData);
```

`ProtocolExecutor` implements this by forwarding unconditionally to `Comms::push_message`:

```rust
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);   // line 513
}
```

`push_message` performs only a *minimum*-length check (≥ 40 bytes) and then stores the full payload:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {   // only a floor, no ceiling
        return;
    }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);   // stored in full
}
``` [1](#0-0) 

`MessageBuffer` is a `HashMap<MessageHeader, SubMessageQueue>` with no cap on the number of distinct keys:

```rust
struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}
``` [2](#0-1) 

Each `SubMessageQueue` wraps a `futures::channel::mpsc::unbounded()` channel — no message-count limit and no per-message size limit:

```rust
let (sender, receiver) = futures::channel::mpsc::unbounded();
``` [3](#0-2) 

A `MessageHeader` is a 40-byte value: a 32-byte `ChannelTag` plus an 8-byte `Waitpoint` (`u64`). An attacker controls both fields freely by crafting the first 40 bytes of any `MessageData` they submit. The honest protocol only ever reads a small, fixed set of `(channel, waitpoint)` pairs; every other combination is buffered forever.

**The codebase itself documents this attack in a test:**

```rust
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;

    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());
        comms.push_message(attacker, message);
    }
    // asserts all 512 entries are present — no rejection, no eviction
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [4](#0-3) 

The test name and comment ("Attacker injects messages for waitpoints the honest code never polls") confirm this is a known, unmitigated path.

**Three compounding dimensions of the attack:**

| Dimension | Mechanism | Limit |
|---|---|---|
| Distinct headers | New `HashMap` entry per unique `(channel, waitpoint)` | None |
| Messages per header | Unbounded MPSC channel per entry | None |
| Per-message size | `MessageData = Vec<u8>`, no max check | None |

All three are independently exploitable and combine multiplicatively.

---

### Impact Explanation

Any honest party running any protocol (DKG, reshare, refresh, presign, sign, CKD) that accepts messages from a malicious participant will have its `MessageBuffer` grown without bound. Because `MessageData` is a `Vec<u8>` with no enforced maximum, a single attacker can send a small number of large messages or a large number of small messages to the same effect. With enough injected data the honest party's process is OOM-killed, permanently denying it the ability to complete the current protocol session.

This matches the allowed impact: **Medium — Griefing or resource-exhaustion by a malicious participant causing unbounded memory beyond documented behavior.**

The README explicitly documents that the library waits indefinitely for messages and that the caller is responsible for timeouts, but it does not document that the caller must also rate-limit or size-limit incoming messages to prevent memory exhaustion — making this an undocumented, library-level vulnerability rather than a caller misconfiguration.

---

### Likelihood Explanation

- **Attacker precondition**: Be any participant in a running protocol session. No privileged keys required.
- **Trigger**: Call `Protocol::message(from, crafted_data)` repeatedly with payloads whose first 40 bytes encode unused `(channel, waitpoint)` values.
- **Detection difficulty**: The messages are syntactically valid (correct header length, parseable header); no existing check rejects them.
- **Amplification**: Each concurrent protocol session (DKG, sign, etc.) has its own independent `Comms`/`MessageBuffer`, so a single attacker participating in multiple sessions can exhaust memory proportionally faster.

---

### Recommendation

1. **Cap the `HashMap` size**: Reject `push_message` calls once `messages.len()` exceeds a protocol-specific bound (e.g., `N * max_rounds * max_messages_per_round`).
2. **Cap per-queue depth**: Replace `futures::channel::mpsc::unbounded()` with `futures::channel::mpsc::channel(capacity)` using a bounded capacity per `(channel, waitpoint)` slot.
3. **Cap individual message size**: Add a `MAX_MESSAGE_SIZE` check in `push_message` before storing, analogous to the `MAX_APP_ID_LEN` guard already present in `AppId::try_new`. [5](#0-4)

### Proof of Concept

The existing test at `src/protocol/internal.rs:532–554` is a direct PoC. To scale it to an OOM condition:

```rust
// Malicious participant floods the buffer of an honest party's running protocol.
let huge_payload = vec![0u8; 64 * 1024 * 1024]; // 64 MiB per message

for i in 0u64.. {
    // Craft a header for a waitpoint the honest protocol never reads.
    let header = MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
    let mut message = header.to_bytes().to_vec();
    message.extend_from_slice(&huge_payload);

    // Protocol::message() is the public API — no authentication of content.
    honest_party_protocol.message(attacker_participant, message);
    // Each call adds a new HashMap entry holding 64 MiB; no eviction occurs.
}
// Process OOM-kills after ~(available_RAM / 64 MiB) iterations.
``` [6](#0-5) [7](#0-6)
