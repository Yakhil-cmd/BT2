### Title
Stale `target_shard_index` from epoch-N shard layout used to index epoch-N+1 `outcome_roots` in `GetExecutionOutcome`, producing an unverifiable `outcome_root_proof` — (`chain/client/src/view_client_actor.rs`)

### Summary

`ViewClientActor::handle(GetExecutionOutcome)` computes `target_shard_index` once from the **outcome block's** shard layout (epoch N / L1), then — when `get_next_block_hash_with_new_chunk` crosses a resharding boundary and returns a block in epoch N+1 (L2) — uses that stale index to select a leaf from `outcome_roots` built from the L2 block. Because shard IDs and shard indices are explicitly decoupled in `ShardLayoutV2`/`V3`, the index is wrong for every shard whose position in the ordered shard list changed due to the split, producing a Merkle proof that fails verification against `block_header_lite.inner_lite.outcome_root`.

### Finding Description

**Step 1 — index computed from L1.** [1](#0-0) 

`epoch_id` is taken from the outcome block (epoch N). `shard_layout` is L1. `target_shard_index` is the position of `target_shard_id` in L1's ordered shard list.

**Step 2 — `get_next_block_hash_with_new_chunk` may cross the epoch boundary.** [2](#0-1) 

When the function crosses into epoch N+1 it replaces `shard_ids` with the children shards from L2 and updates its local `shard_layout` to L2. It returns `(block_hash_in_N+1, child_shard_id_in_L2)`.

**Step 3 — `target_shard_id` is shadowed; `target_shard_index` is NOT recomputed.** [3](#0-2) 

`outcome_roots` is collected from the L2 block (one entry per L2 shard). The bounds check at line 1174 only catches an out-of-range index; it does not detect a wrong-but-in-range index. `merklize(&outcome_roots).1[target_shard_index]` then selects the leaf at the L1 position, which in L2 belongs to a **different** shard.

**Why the index diverges in V2/V3.** [4](#0-3) 

The comment is explicit: "In V2 & V3 the shard id and shard index are **not** the same." When shard `P` (index `k` in L1) is split into children `C1`, `C2`, every shard whose ID is numerically less than `C1`/`C2` but greater than `P` shifts its index. The V2 documentation gives a concrete example: [5](#0-4) 

L1 = `[0, 2, 5]` → indices `{0→0, 2→1, 5→2}`. L2 = `[0, 5, 6, 7]` (shard 2 split into 6, 7) → indices `{0→0, 5→1, 6→2, 7→3}`. Shard 5 was at index 2 in L1 but is at index 1 in L2. A transaction whose receiver maps to shard 5 gets `target_shard_index = 2`, which in L2 selects shard 6's outcome root — the wrong leaf.

### Impact Explanation

The `outcome_root_proof` returned by `EXPERIMENTAL_light_client_proof` / `GetExecutionOutcome` is a Merkle path whose root must equal `block_header_lite.inner_lite.outcome_root`. With the wrong leaf selected, the path is structurally invalid and verification fails. Any light client or bridge relying on this proof to confirm transaction finality will reject it, breaking the client-proof invariant for all transactions executed in the last block(s) of epoch N whose receiver's shard index shifts in L2.

### Likelihood Explanation

The condition requires: (a) a resharding event (static or dynamic), (b) a transaction submitted in the last block(s) of epoch N whose receiver maps to a shard whose index changes in L2, and (c) a call to `GetExecutionOutcome` after the first L2 block with a new chunk for that shard. Condition (b) is satisfied by any shard that is not the split shard itself but whose position in the ordered shard list shifts because a lower-ID shard was removed and replaced by higher-ID children. This is a deterministic, reproducible condition on every resharding event.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns a block hash `h`, recompute `target_shard_index` using the shard layout of block `h`:

### Proof of Concept

1. Configure a test-loop network with a static resharding boundary at protocol version N+1 (e.g., 3 → 4 shards, using `ShardLayoutV2` so shard IDs ≠ shard indices).
2. Submit a transaction in the last block of epoch N whose `receiver_id` maps to a shard whose index shifts in L2 (e.g., shard 5 in the `[0,2,5]→[0,5,6,7]` example).
3. Advance to epoch N+1 and wait for the first block with a new chunk for the relevant child shard.
4. Call `GetExecutionOutcome` (or `EXPERIMENTAL_light_client_proof`) for that transaction.
5. Verify the returned `outcome_root_proof` against the block's `outcome_root` using `merklize(&outcome_roots_from_L2_block).0`. The verification will fail because `outcome_root_proof` was built with the L1 index.
6. Recompute `target_shard_index` from L2's shard layout and verify again — it will pass.
