Audit Report

## Title
Fee-on-Transfer ERC-20 Token Causes ckERC20 Over-Minting (Chain-Fusion Ledger Conservation Bug) - (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`, `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary

Both ckERC20 deposit helper contracts emit a deposit event carrying the caller-supplied `amount` without verifying the minter's address actually received that amount. The IC minter canister scrapes these events and mints exactly `event.value()` ckERC20 tokens to the beneficiary. For any fee-on-transfer ERC-20 token, the minter receives `amount - fee` while the IC ledger mints `amount`, permanently breaking the 1:1 backing invariant and causing protocol insolvency. USDT — already a supported ckERC20 token — contains a dormant fee mechanism that its owner can activate at any time.

## Finding Description

**Ethereum side:** In `DepositHelperWithSubaccount.sol` at lines 519–531, `depositErc20` calls `erc20Token.safeTransferFrom(msg.sender, minterAddress, amount)` and then unconditionally emits `ReceivedEthOrErc20(..., amount, ...)` using the caller-supplied `amount` with no balance-before/balance-after check. The identical pattern exists in `ERC20DepositHelper.sol` at lines 498–503 (`CkErc20Deposit.deposit`), which emits `ReceivedErc20(erc20_address, msg.sender, amount, principal)` after `safeTransferFrom`.

**IC minter side:** In `rs/ethereum/cketh/minter/src/deposit.rs` at lines 73–81, the `mint()` function reads the scraped event and calls `ICRC1Client.transfer(TransferArg { amount: event.value(), ... })`, where `event.value()` is populated verbatim from the `amount` field of the `ReceivedErc20Event` struct (defined in `src/eth_logs/mod.rs` lines 57–75, field `value: Erc20Value`). There is no cross-check against the minter's actual on-chain ERC-20 balance.

**Exploit flow:**
1. A fee-on-transfer ERC-20 token is NNS-approved (or an existing supported token such as USDT activates its dormant fee).
2. An unprivileged user calls `depositErc20(tokenAddress, 1000e18, principal, 0x)`.
3. The ERC-20 contract deducts its fee: minter receives `990e18`, but the helper emits `ReceivedEthOrErc20(..., 1000e18, ...)`.
4. The IC minter scrapes the log, reads `value = 1000e18`, and mints `1000e18` ckERC20 to the user.
5. After repeated deposits, ckERC20 total supply exceeds the minter's ERC-20 balance; withdrawal requests eventually fail.

**Existing guards are insufficient:** The minter's token whitelist (enforced via `s.ckerc20_tokens.get_entry_alt`) only filters which ERC-20 contracts trigger minting — it does not validate that the emitted `amount` equals the actual received amount. The `safeTransferFrom` call in the helper contracts succeeds even when the token deducts a fee, because the ERC-20 standard permits fee-on-transfer behavior.

## Impact Explanation

This constitutes **illegal minting and protocol insolvency** of an in-scope chain-key/ledger asset (ckERC20). The ckERC20 total supply on the IC ledger exceeds the ERC-20 balance held by the minter's Ethereum address, permanently breaking the 1:1 peg. Withdrawal requests will fail once the minter's ERC-20 balance is exhausted, causing direct, irreversible loss of funds for ckERC20 holders. This matches the Critical allowed impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets."* Given the medium likelihood (precondition required), the effective severity is **High** ($2,000–$10,000).

## Likelihood Explanation

No privileged attacker capability is required — any unprivileged Ethereum user can call `depositErc20`. The precondition is that a fee-on-transfer ERC-20 token is NNS-approved. USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is already a supported ckERC20 token and contains a fee mechanism (currently set to zero) that the Tether issuer can activate unilaterally. This is not a "third-party compromise" — it is a legitimate, documented capability of the USDT contract. Any future NNS proposal adding a fee-on-transfer token would also trigger the vulnerability immediately upon approval. Likelihood is **medium**.

## Recommendation

In both `depositErc20` (`DepositHelperWithSubaccount.sol`) and `deposit` (`ERC20DepositHelper.sol`), record the minter's ERC-20 balance before and after `safeTransferFrom` and emit the actual received amount (the difference) rather than the caller-supplied `amount`:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
require(actualReceived > 0, "ERC20: zero amount received");
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

This ensures the IC minter mints only what was actually received, preserving the 1:1 backing invariant regardless of the underlying token's fee behavior.

## Proof of Concept

1. Deploy a mock ERC-20 token that deducts 1% on every `transferFrom` call.
2. Submit an NNS proposal (or use a local test environment) to add this token as a supported ckERC20 token.
3. Call `depositErc20(mockToken, 1000e18, alicePrincipal, 0x)` on the `CkDeposit` helper contract (`DepositHelperWithSubaccount.sol`).
4. Observe: the helper emits `ReceivedEthOrErc20(..., 1000e18, ...)` while the minter address only received `990e18`.
5. The IC minter scrapes the log and calls `ICRC1Client.transfer(amount: 1000e18)` on the ckERC20 ledger.
6. Assert: Alice holds `1000e18` ckERC20 but the minter holds only `990e18` ERC-20.
7. Repeat 100 times: minter holds `99_000e18` ERC-20, ckERC20 total supply is `100_000e18`; the last ~1% of ckERC20 holders cannot redeem.

A deterministic integration test can be written using the existing `CkErc20Setup` test harness in `rs/ethereum/cketh/minter/tests/ckerc20.rs` by substituting a fee-on-transfer mock ERC-20 and asserting the invariant `ckERC20_total_supply == minter_erc20_balance` after each deposit.
