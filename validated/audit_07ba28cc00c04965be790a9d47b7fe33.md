### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions Using a Global Mempool Presence Check Instead of Per-Transaction Authorization — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (the account contract's signature verification) for any invoke transaction with `nonce == 1` when the sender address has *any* transaction present in the mempool or a recent block. Because this check is a global, address-level presence query rather than a per-transaction authorization check, an attacker who observes a legitimate `deploy_account` transaction in the mempool can submit a crafted invoke transaction with an arbitrary/invalid signature for the same address and have it accepted into the mempool without signature verification.

---

### Finding Description

**The skip condition:**

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when:
- the incoming transaction is an `Invoke`,
- `tx.nonce() == Nonce(Felt::ONE)`, and
- `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain). [1](#0-0) 

When those conditions hold, the function calls `account_tx_in_pool_or_recent_block(tx.sender_address())` and returns `true` (skip) if the result is `true`.

**What `account_tx_in_pool_or_recent_block` actually checks:**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This is a **global, address-level** presence check. It returns `true` if *any* transaction from that address is in the pool or was committed in a recent block. It does **not** verify that the specific incoming invoke transaction carries a valid signature, nor does it verify that the pooled transaction is specifically a `deploy_account`.

**What skipping means:**

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `tx.execution_flags.validate == false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling the `__validate__` entry point: [4](#0-3) 

**No signature in mempool validation either:**

`ValidationArgs` — the struct passed to `validate_by_mempool` — contains no signature field: [5](#0-4) 

`Mempool::validate_tx` only checks for duplicate tx_hash, nonce validity, and fee escalation — no signature check: [6](#0-5) 

**The complete attack path:**

1. Legitimate user submits a `deploy_account` for address `A` (valid signature). It enters the mempool.
2. Attacker observes this and submits an `Invoke` with `nonce=1`, `sender_address=A`, and an **arbitrary/invalid signature** (e.g., empty `[]`).
3. Gateway stateful validation:
   - `validate_state_preconditions`: nonce check passes (`0 ≤ 1 ≤ 0 + max_gap`).
   - `validate_by_mempool`: passes (no signature check).
   - `skip_stateful_validations`: `account_tx_in_pool_or_recent_block(A)` returns `true` → returns `true` (skip).
4. `run_validate_entry_point` is called with `skip_validate=true` → `__validate__` is **never called**.
5. The attacker's invoke with invalid signature is accepted into the mempool.
6. The legitimate user's invoke with `nonce=1` is now rejected as `DuplicateNonce` (or `DuplicateTransaction` if same hash). [7](#0-6) 

---

### Impact Explanation

**Matching impact:** *High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

An attacker can inject an invoke transaction carrying an invalid/arbitrary signature into the mempool for any address that has a pending `deploy_account`. The mempool accepts it as a valid pending transaction. The legitimate user's first invoke (nonce=1) is then rejected as a duplicate nonce, blocking their account's first operation. The attacker can repeat this for every new account deployment they observe, creating a persistent griefing/DoS vector against new account onboarding.

---

### Likelihood Explanation

- `deploy_account` transactions are publicly visible in the mempool/P2P layer.
- The attacker needs no special privilege: they only need to submit a standard invoke RPC call with an arbitrary signature.
- The condition (`nonce=1`, `account_nonce=0`, any tx in pool) is trivially satisfiable whenever a `deploy_account` is pending.
- No funds are required at the attacker's address; the attacker submits from the victim's address.

---

### Recommendation

Replace the global address-presence check with a check that specifically verifies a `deploy_account` transaction for the exact sender address is pending, **and** do not skip `__validate__` entirely — instead, defer it or use a lightweight signature pre-check. At minimum, `skip_stateful_validations` should verify that the pooled transaction for the address is specifically a `deploy_account` type, not just any transaction. The comment in the code acknowledges the weaker reasoning:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of that disjunction is the exploitable case: an attacker's own previously accepted transaction (even a `deploy_account` they control) satisfies the check and unlocks signature-skip for any subsequent invoke they craft for that address.

---

### Proof of Concept

```
1. Attacker calls gateway `add_tx` with:
     RpcTransaction::DeployAccount { sender: A, nonce: 0, signature: <valid> }
   → Accepted. Mempool now contains deploy_account for A.

2. Attacker calls gateway `add_tx` with:
     RpcTransaction::Invoke { sender: A, nonce: 1, calldata: [<anything>], signature: [] }

3. Gateway flow:
   - stateless_tx_validator.validate() → passes (empty signature is within length bounds)
   - stateful_validator.extract_state_nonce_and_run_validations():
       account_nonce = get_nonce_from_state(A) = 0
       run_pre_validation_checks():
         validate_state_preconditions(): nonce 1 in [0, 0+max_gap] → OK
         validate_by_mempool(): no sig check → OK
         skip_stateful_validations():
           tx.nonce()==1 && account_nonce==0 → true
           account_tx_in_pool_or_recent_block(A) → true (deploy_account is there)
           returns true (SKIP)
       run_validate_entry_point(skip_validate=true):
         execution_flags.validate = false
         StatefulValidator::perform_validations():
           if !tx.execution_flags.validate { return Ok(()); }  ← exits here
   → Invoke with empty signature ACCEPTED into mempool.

4. Legitimate user submits:
     RpcTransaction::Invoke { sender: A, nonce: 1, calldata: [<real>], signature: <valid> }
   → validate_by_mempool() → MempoolError::DuplicateNonce → REJECTED.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
