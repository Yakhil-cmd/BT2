### Title
Silent VC Enumeration via Bypassed `chia_getVCList` Enables Targeted Revocation with Unidentifiable Confirmation Dialog — (`packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

`chia_wallet.vc_get_list` is marked `allowConfirmationBypass: true`, allowing a dApp to silently enumerate all of the user's VCs — including each VC's `coin.parentCoinInfo` — without any per-call confirmation. The `chia_wallet.vc_revoke` command (which consumes `vc_parent_id`) always requires confirmation, but its confirmation dialog displays only the raw hex `vc_parent_id` and a fee, with no launcher ID or human-readable VC identity. A dApp that has been granted both commands can silently identify a target VC's `parentCoinInfo` and then present a revoke dialog the user cannot meaningfully verify.

---

### Finding Description

**Step 1 — `vc_get_list` bypass is explicitly enabled.**

`chia_wallet.vc_get_list` carries `allowConfirmationBypass: true`: [1](#0-0) 

A dApp that has been granted this command can call `chia_requestPermissions` to add it to its bypass list. Once bypassed, every subsequent `chia_getVCList` call executes silently with no user prompt.

**Step 2 — `vc_get_list` returns `parentCoinInfo` for every VC.**

The `VCType` returned by `vc_get_list` includes the full coin record: [2](#0-1) 

`coin.parentCoinInfo` is exactly the value `vc_revoke` requires as `vc_parent_id`.

**Step 3 — `vc_revoke` confirmation dialog shows only the hex `vc_parent_id`.**

The `vc_revoke` command definition exposes only two fields to the user: [3](#0-2) 

The dialog title is "Confirm Revoke VC", the message is "Are you sure you want to revoke this verifiable credential?", and the only displayed parameter is `vc_parent_id` labeled "Parent Coin Id" — a raw hex string. The launcher ID, proof hash, proof provider, or any human-readable VC identity is **not shown**.

**Step 4 — `vc_revoke` is in `BlockedCommands` (cannot be bypassed), but the confirmation is uninformative.** [4](#0-3) 

The revoke always requires confirmation, but the dialog gives the user no way to identify *which* VC is being revoked. A user who has granted `vc_revoke` to a dApp and sees "Parent Coin Id: 0xabcd…" cannot determine whether this is their low-value test VC or a high-value credential.

**Step 5 — `addDappBypassPermissions` correctly gates bypass on `allowConfirmationBypass`.** [5](#0-4) 

`vc_revoke` cannot be bypassed. The attack does not bypass the revoke confirmation — it exploits the fact that the confirmation is insufficient to identify the target.

---

### Impact Explanation

A dApp granted both `chia_wallet.vc_get_list` (bypassed) and `chia_wallet.vc_revoke` can:

1. Call `chia_getVCList` silently → receive all VCs with `coin.parentCoinInfo`, `launcherId`, `proofHash`, `proofProvider`.
2. Select the `parentCoinInfo` of any specific VC.
3. Call `chia_revokeVC` with that `parentCoinInfo`.
4. The user sees: "Are you sure you want to revoke this verifiable credential? Parent Coin Id: 0x…" — with no launcher ID or name.
5. If the user approves (unable to distinguish which VC this is), the targeted VC is permanently revoked on-chain.

This fits the High impact category: *"WalletConnect state that causes a user to approve… revoke… the wrong asset, identity…"*

---

### Likelihood Explanation

- Requires the user to have granted both `chia_wallet.vc_get_list` and `chia_wallet.vc_revoke` to the dApp.
- The bypass for `vc_get_list` is user-approved via `chia_requestPermissions`, which itself requires confirmation.
- However, a dApp that presents itself as a VC management tool would naturally request both permissions, and users routinely approve such bundles without scrutinizing each command.
- The confirmation dialog's failure to display the launcher ID is a structural deficiency, not a race condition or edge case.

---

### Recommendation

1. **Add `launcherId` (and optionally `proofHash`) to the `vc_revoke` confirmation dialog params** so the user can identify which VC is being revoked. This requires either a pre-fetch in the handler or passing the launcher ID as an additional (hidden-from-RPC but shown-in-dialog) field.
2. **Consider whether `vc_get_list` should be bypassable at all**, given that it returns sensitive coin-level data (`parentCoinInfo`, `proofHash`, `proofProvider`) that directly enables targeted destructive operations.

---

### Proof of Concept

```
1. dApp calls chia_requestPermissions({ commands: ["chia_getVCList"] })
   → User approves once; chia_getVCList is now in bypass list.

2. dApp calls chia_getVCList() [no confirmation shown]
   → Response: { vcRecords: [{ vc: { coin: { parentCoinInfo: "0xdeadbeef..." }, launcherId: "0xabc..." }, ... }] }

3. dApp selects parentCoinInfo of the target VC.

4. dApp calls chia_revokeVC({ vcParentId: "0xdeadbeef...", fee: 0 })
   → Confirmation dialog: "Are you sure you want to revoke this verifiable credential?"
      Parent Coin Id: 0xdeadbeef...
      Fee: 0 XCH
   → User cannot identify which VC this is; approves.

5. Target VC is revoked on-chain.
```
