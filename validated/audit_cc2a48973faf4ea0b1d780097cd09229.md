### Title
Attached Deposit Permanently Locked in `rlp_execute` Early-Return Paths — (`File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary
`rlp_execute` is marked `#[payable]` and accepts an attached NEAR deposit from any external caller. In two distinct early-return paths the function returns `PromiseOrValue::Value(...)` — a successful execution result — without issuing any refund transfer. Because the NEAR runtime only auto-refunds deposits on receipt **failure** (panic/abort), a successful return with a value causes the deposit to be silently credited to the wallet contract's balance and irrecoverable by the caller.

### Finding Description

`rlp_execute` is the sole public entry point of the eth-wallet-contract deployed for every ETH-implicit account on NEAR. [1](#0-0) 

It is declared `#[payable]`, so any caller may attach an arbitrary NEAR deposit. Two code paths return a direct value instead of a promise, bypassing the `CallerDeposit` refund mechanism entirely:

**Path 1 — in-flight transaction guard (line 97–104):** [2](#0-1) 

When `has_in_flight_tx` is `true` (a normal operational state whenever a prior call is still resolving), the function returns immediately with `PromiseOrValue::Value(...)`. No promise is created, so `rlp_execute_callback` is never reached and no refund is issued.

**Path 2 — `inner_rlp_execute` error fallthrough (line 126):** [3](#0-2) 

Any error that is not a self-signed `Relayer` error (i.e., `Error::User`, `Error::AccountId`, `Error::AccountNonceExhausted`, or an unsigned-relayer `Error::Relayer`) falls through to `PromiseOrValue::Value(e.into())`. Again, no promise chain is created.

The `CallerDeposit` struct is designed to track and refund external deposits, but it is only consulted inside `rlp_execute_callback`: [4](#0-3) 

That callback is only scheduled when `inner_rlp_execute` returns `Ok(promise)`. In both early-return paths above, `inner_rlp_execute` is either never called (Path 1) or returns `Err` (Path 2), so the callback is never scheduled and the deposit is never refunded.

The NEAR runtime's deposit-refund rule confirms this: automatic refunds are generated only when a receipt **fails** (aborts/panics). A receipt that returns a value successfully — as both paths do — causes the deposit to be credited to the contract's balance with no protocol-level refund. [5](#0-4) 

### Impact Explanation

Any external caller who attaches a non-zero deposit to `rlp_execute` while `has_in_flight_tx == true`, or while the RLP transaction fails with a user/relayer/account-id error, permanently loses that deposit. The tokens are credited to the wallet contract's balance. The wallet owner (the Ethereum key holder) can subsequently spend those tokens via future `rlp_execute` calls, constituting an unintended and irrecoverable transfer of funds from the caller to the wallet owner. There is no withdrawal function and no recovery path for the caller.

**Scope:** Accounting — conservation of caller-attached value.
**Severity:** High — arbitrary NEAR amounts can be permanently lost by unprivileged callers.

### Likelihood Explanation

The `has_in_flight_tx == true` condition is a routine operational state: it is set to `true` every time a valid transaction is submitted and remains `true` until all promise callbacks resolve (potentially multiple blocks). Any relayer or external caller who races a second `rlp_execute` call with a deposit during this window loses their deposit. The user-error path (malformed RLP, wrong nonce, unsupported action) is also reachable by any caller. No special privilege is required.

### Recommendation

In every early-return path of `rlp_execute` that returns `PromiseOrValue::Value(...)`, explicitly refund any non-zero attached deposit to `env::predecessor_account_id()` before returning. Alternatively, remove the `#[payable]` attribute and instead require callers to attach zero deposit when the function cannot guarantee forwarding the value, or add a guard at the top of the function that panics (triggering the automatic runtime refund) if a deposit is attached but the function cannot proceed.

### Proof of Concept

1. Wallet account `W` has `has_in_flight_tx = true` (a prior valid transaction is in flight).
2. External caller `C` calls `rlp_execute` on `W` with `attached_deposit = 10 NEAR`.
3. The guard at line 97 fires; the function returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })`.
4. The receipt succeeds (no panic), so the NEAR runtime does **not** generate a deposit refund receipt.
5. `W`'s balance increases by 10 NEAR; `C`'s balance decreases by 10 NEAR.
6. `C` has no mechanism to recover the 10 NEAR; `W`'s owner can spend it freely via subsequent `rlp_execute` calls.
