### Title
Missing Minimum Threshold Enforcement in FROST Signing Allows Threshold-1 Protocol Execution - (File: src/frost/mod.rs)

---

### Summary

The FROST signing path (`assert_sign_inputs` and `presign` in `src/frost/mod.rs`) does not enforce a minimum threshold of 2, unlike the DKG path (`assert_key_invariants` in `src/dkg.rs`) which explicitly rejects `threshold < 2`. A malicious coordinator or misconfigured caller can invoke FROST signing with `threshold = 1`, causing honest participants to execute the signing protocol under a weaker-than-intended security parameter, producing a corrupted or invalid signature output that honest parties accept as protocol-complete.

---

### Finding Description

In `src/dkg.rs`, `assert_key_invariants` enforces a hard lower bound:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

This correctly prevents key generation with a trivially weak threshold.

However, the FROST signing entry points in `src/frost/mod.rs` — both `presign` (lines 44–88) and `assert_sign_inputs` (lines 120–159) — only check the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

There is no corresponding lower-bound check. A caller may pass `threshold = 1` (or even `threshold = 0`) and the function returns `Ok(...)`, proceeding into the live protocol.

The `ReconstructionLowerBound` type in `src/thresholds.rs` is a plain `usize` wrapper with no invariant enforcement — it accepts any value including 0 and 1.

---

### Impact Explanation

**Impact: High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

When `threshold = 1` is supplied to FROST signing while the key was generated with `threshold ≥ 2`:

- Lagrange interpolation uses only one participant's share with coefficient 1, reconstructing a value that is not the actual secret key.
- The resulting aggregate signature is cryptographically invalid against the public key established during DKG.
- All honest participants complete the protocol without error and accept the output as valid — there is no runtime rejection because the threshold mismatch is never detected after initialization.

This directly corrupts the signing output: honest parties believe they have produced a valid threshold signature, but the output is unusable and will fail external verification.

---

### Likelihood Explanation

**Likelihood: Medium.**

The entry path is direct: any library caller invoking `sign_v1`, `sign_v2`, or `presign` for EdDSA/RedJubjub controls the `threshold` argument. A malicious coordinator who orchestrates the signing session can supply `threshold = 1` to all participants. Participants have no independent way to verify that the threshold passed at signing time matches the threshold used during key generation, since `KeygenOutput` does not carry the threshold value. No cryptographic primitive break or key compromise is required.

---

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Additionally, consider embedding the threshold into `KeygenOutput` so that signing functions can cross-validate the runtime threshold against the one used at key generation time.

---

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
