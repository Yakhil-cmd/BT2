Audit Report

## Title
Unbounded `public_keys` Array Enables Synchronous CPU Exhaustion DoS in ICP Rosetta `/construction/payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary
The ICP Rosetta `/construction/payloads` endpoint accepts an unbounded `public_keys` array and performs a full cryptographic derivation (hex decode + DER encode + SHA-224 hash) for every entry with no count guard. The handler executes this work synchronously inside an `async` actix-web handler, blocking the worker thread for the entire O(n) loop. A small number of concurrent requests carrying ~40,000 keys each (permitted by the 4 MB body limit) can exhaust all actix-web worker threads, rendering the Rosetta process unresponsive to all other requests.

## Finding Description
**Handler registration** — `POST /construction/payloads` is registered with no authentication and no per-endpoint rate limiting: [1](#0-0) 

The only server-side guard is a 4 MB JSON body limit applied globally: [2](#0-1) 

After extracting `pks`, there is no check on `pks.len()` before the loop. The function immediately iterates every supplied key: [3](#0-2) 

Each iteration calls `principal_id_from_public_key`, which dispatches to `Ed25519KeyPair::get_principal_id` or `Secp256k1KeyPair::get_principal_id`: [4](#0-3) 

`Ed25519KeyPair::get_principal_id` performs hex decode + DER encode + SHA-224 hash on every call: [5](#0-4) 

`PrincipalId::new_self_authenticating` confirms the SHA-224 hash is performed per key: [6](#0-5) 

**Synchronous blocking in async context** — the handler is declared `async fn` but calls `construction_payloads` without `.await`, meaning the entire O(n) loop runs on the actix-web worker thread, starving other requests. The `HttpServer` is constructed without a `.workers()` override, defaulting to the number of logical CPUs: [7](#0-6) 

**Contrast with ICRC-1 Rosetta**, which correctly enforces a hard limit of exactly one public key before any processing: [8](#0-7) 

No equivalent guard exists in the ICP Rosetta path. Note that even with 40,000 identical keys, the `.map()` iterator calls `principal_id_from_public_key` 40,000 times before `.collect()` builds the `HashMap`, so deduplication provides no protection.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Rosetta, boundary/API security impact with concrete user or protocol harm."*

An attacker can render the ICP Rosetta API completely unresponsive — blocking all balance queries, block fetches, and transaction submissions — without crashing the process. This is an algorithmic complexity attack, not raw volumetric DDoS. The Rosetta API is an in-scope financial integration component used for ICP transfers and neuron management.

## Likelihood Explanation
The endpoint is publicly reachable on any deployed ICP Rosetta instance. No credentials, tokens, or prior state are required. The attack payload is trivially constructable (repeat a single valid public key ~40,000 times). The 4 MB body limit is the only barrier and it does not prevent the attack — it only caps the per-request key count. A single attacker with a standard HTTP client can sustain the attack indefinitely by sending a small number of concurrent requests.

## Recommendation
1. **Add a hard key-count limit** immediately after extracting `pks` in `construction_payloads.rs`, before the iteration loop, analogous to the ICRC-1 guard:
   ```rust
   const MAX_PUBLIC_KEYS: usize = 10;
   if pks.len() > MAX_PUBLIC_KEYS {
       return Err(ApiError::invalid_request(
           format!("Too many public_keys: max {MAX_PUBLIC_KEYS}")
       ));
   }
   ```
2. **Move CPU-bound work off the async executor** using `actix_web::web::block` or `tokio::task::spawn_blocking` to avoid blocking worker threads.
3. **Optionally add per-IP rate limiting** at the HTTP layer as defense-in-depth.

## Proof of Concept
```python
import requests, json, threading

KEY = {"hex_bytes": "a" * 64, "curve_type": "edwards25519"}
PAYLOAD = {
    "network_identifier": {"blockchain": "Internet Computer", "network": "<network_id>"},
    "operations": [],
    "public_keys": [KEY] * 40000,
}

def attack():
    requests.post("http://<rosetta-host>:8080/construction/payloads", json=PAYLOAD, timeout=120)

threads = [threading.Thread(target=attack) for _ in range(8)]
for t in threads: t.start()
for t in threads: t.join()
# All subsequent requests to the Rosetta API will time out during the attack window.
```

A deterministic unit test can confirm the O(n) behavior by benchmarking `construction_payloads` with `public_keys` of length 1, 100, 1,000, 10,000, and 40,000 and asserting that wall-clock time scales linearly with key count.

pub fn principal_id_from_public_key(pk: &PublicKey) -> anyhow::Result<PrincipalId> {
    match pk.curve_type {
        CurveType::Edwards25519 => Ed25519KeyPair::get_principal_id(&pk.hex_bytes),
        CurveType::Secp256K1 => Secp256k1KeyPair::get_principal_id(&pk.hex_bytes),
        _ => bail!("Curve Type {:?} is not supported", pk.curve_type),
    }
}

    fn get_principal_id(pk_encoded: &str) -> anyhow::Result<PrincipalId> {
        match Ed25519KeyPair::hex_decode_pk(pk_encoded) {
            Ok(pk_decoded) => {
                let pub_der = Ed25519KeyPair::der_encode_pk(pk_decoded)?;
                Ok(PrincipalId::new_self_authenticating(&pub_der))
            }
            Err(e) => Err(e.context(format!("Could not decode public key {pk_encoded}"))),
        }
    }
