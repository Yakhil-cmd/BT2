### Title
`EVM.validateCOAOwnershipProof` Signature Not Bound to Target EVM Address Enables Cross-COA Ownership Proof Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
The `validateCOAOwnershipProof` function in the EVM system contract performs signature verification over a caller-supplied `signedData` blob without requiring that blob to commit to the `evmAddress` being claimed. This is the direct Flow analog of the reported "event-based" authentication pattern: the cryptographic check (signature validity) is decoupled from the protected resource (the specific COA). An unprivileged attacker who observes a valid signature produced by a victim for one COA can replay it to forge a valid ownership proof for a different COA owned by the same Cadence account, bypassing the ERC-1271 `isValidSignature` guard on the second COA.

### Finding Description
**Root cause** — `fvm/evm/stdlib/contract.cdc` lines 1082–1086 verify that the provided signatures are valid over `signedData`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The subsequent check at lines 1095–1109 independently verifies that the COA resource at the caller-supplied `path` has an address matching `evmAddress`:

```cadence
if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
    let coaAddressBytes = coaRef.address().bytes
    for index, item in coaAddressBytes {
        if item != evmAddress[index] { ... }
    }
    return ValidationResult(isValid: true, problem: nil)
}
```

These two checks are not cryptographically bound to each other. The signature does not commit to `evmAddress`, and the COA address check does not depend on the signature content. The code comment at lines 1001–1009 explicitly acknowledges this gap:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

**Exploit path:**
1. Alice owns two COAs: COA_A (published at `/public/coa_a`) and COA_B (published at `/public/coa_b`), both under the same Cadence account.
2. Alice signs data `D` to authorize some action for COA_A (e.g., an ERC-1271 permit). This signature is observable on-chain.
3. An attacker constructs a `COAOwnershipProof` with:
   - `address` = Alice's Cadence address
   - `path` = `/public/coa_b` (pointing to COA_B, not COA_A)
   - `signedData` = `D`
   - `keyIndices` / `signatures` = Alice's key indices and signatures over `D`
   - `evmAddress` = COA_B's EVM address
4. The attacker calls `EVM.validateCOAOwnershipProof` directly from a Cadence script, or triggers it via the `verifyCOAOwnershipProof` Cadence Arch precompile from an EVM transaction.
5. Validation succeeds: Alice's signatures over `D` are valid (lines 1082–1086 pass), and the COA at `/public/coa_b` has address COA_B_address (lines 1095–1109 pass).
6. The attacker has forged a valid ownership proof for COA_B using a signature Alice produced for COA_A.

The ERC-1271 `isValidSignature` function in `coa.sol` (lines 114–125) calls `verifyCOAOwnershipProof(address(this), _hash, _sig)` where `address(this)` is the calling COA's EVM address. A malicious EVM contract can call `isValidSignature` on COA_B with Alice's signature, and the validation will succeed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

### Impact Explanation
An attacker can forge ERC-1271 signatures for a victim's COA_B by replaying signatures the victim produced for COA_A. Any EVM contract that uses `isValidSignature` for authorization — ERC-20 permit approvals, governance votes, NFT marketplace approvals, cross-chain bridge authorizations — can be tricked into accepting the forged authorization. This enables unauthorized asset transfers or privileged state changes on behalf of the victim's COA_B without the victim's consent. The impact is direct theft or unauthorized movement of on-chain assets.

### Likelihood Explanation
Medium. The attack requires the victim to have multiple COAs under the same Cadence account (not restricted by the protocol, and increasingly common as COA usage grows) and the attacker to have observed the victim's on-chain signatures (trivially achievable by monitoring the chain). The vulnerability is explicitly acknowledged in the production code comments, confirming it is a known design gap rather than a speculative issue. As multi-COA patterns emerge in DeFi integrations, the likelihood increases.

### Recommendation
Enforce that `signedData` encodes `evmAddress` as part of the proof. The `validateCOAOwnershipProof` function should verify that `signedData` contains the `evmAddress` bytes before accepting the proof as valid — for example, by requiring `signedData` to be structured as `evmAddress || application_data` and checking the prefix. Alternatively, the function should internally construct the signed payload as `hash(evmAddress || caller_data)` to cryptographically bind the signature to the specific COA being claimed, making cross-address replay impossible.

### Proof of Concept
```cadence
// Alice has COA_A at /public/coa_a and COA_B at /public/coa_b
// Alice previously signed data D for COA_A (signature observable on-chain)
// Attacker replays Alice's signature to forge ownership proof for COA_B

import EVM from 0x...

access(all) fun main(
    aliceAddress: Address,
    aliceSignature: [UInt8],
    aliceKeyIndex: UInt64,
    dataD: [UInt8],
    coaBAddress: [UInt8; 20]
): EVM.ValidationResult {
    // path points to COA_B, but signedData was produced by Alice for COA_A
    return EVM.validateCOAOwnershipProof(
        address: aliceAddress,
        path: /public/coa_b,       // COA_B's public path
        signedData: dataD,          // Data Alice signed for COA_A
        keyIndices: [aliceKeyIndex],
        signatures: [aliceSignature],
        evmAddress: coaBAddress     // COA_B's EVM address
    )
    // Returns ValidationResult(isValid: true) — forged proof for COA_B
    // Signature check passes (Alice's sig over D is valid)
    // Address check passes (COA at /public/coa_b has address coaBAddress)
}
```

A malicious EVM contract then calls `COA_B.isValidSignature(hash(D), encoded_proof)`, which calls `verifyCOAOwnershipProof(COA_B_address, hash(D), encoded_proof)` via the Cadence Arch precompile, returning `true` and granting the attacker ERC-1271 authorization over COA_B.
