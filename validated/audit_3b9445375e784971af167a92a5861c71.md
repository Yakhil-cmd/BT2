### Title
Missing Refund Mechanism for ERC-20 Tokens Burned During Failed `ExitToNear` Operations — (`engine-precompiles/src/native.rs`)

---

### Summary

When a user calls `withdrawToNear` on a bridged ERC-20 contract, the ERC-20 tokens are burned synchronously on Aurora before the asynchronous NEP-141 `ft_transfer` promise is dispatched. When the `error_refund` compile-time feature is absent, no callback is attached to the promise, so if the NEP-141 transfer fails the burned ERC-20 tokens are permanently unrecoverable. This is the direct analog of the JOJO "missing burn mechanism" class: instead of tokens being received without being burned (oversupply), tokens are burned without the corresponding transfer succeeding (permanent freeze / undersupply of user funds).

---

### Finding Description

**Step 1 — ERC-20 burn is synchronous and unconditional.**

`EvmErc20.withdrawToNear` (and `EvmErc20V2.withdrawToNear`) call `_burn` on the caller's balance *before* invoking the `ExitToNear` precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here
    // ... encodes input and calls precompile at 0xe9217bc7...
}
``` [1](#0-0) [2](#0-1) 

**Step 2 — The precompile builds `callback_args` that are empty when `error_refund` is absent.**

Inside `ExitToNear::run`, the refund field of `ExitToNearPrecompileCallbackArgs` is only populated when the `error_refund` Cargo feature is compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,   // None for non-wNEAR paths
};
``` [3](#0-2) 

For all non-wNEAR ERC-20 exits (`transfer_near_args = None`) without `error_refund`, `callback_args` equals the default value (`refund: None, transfer_near: None`).

**Step 3 — When `callback_args` is default, no callback is attached to the promise.**

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)          // ← no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [4](#0-3) 

**Step 4 — The callback is the only place where burned tokens are re-minted on failure.**

`exit_to_near_precompile_callback` calls `engine::refund_on_error`, which re-mints the ERC-20 tokens to the original sender:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [5](#0-4) 

`refund_on_error` calls the ERC-20 contract's `mint` selector to restore the burned balance: [6](#0-5) 

Without the callback, this re-mint never happens.

**Step 5 — The test suite explicitly confirms the fund loss.**

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

The test acknowledges that `FT_EXIT_AMOUNT` tokens are permanently lost when `error_refund` is absent.

---

### Impact Explanation

Any user who calls `withdrawToNear` on a bridged ERC-20 token with a recipient NEAR account that is not registered with the NEP-141 contract (or any other condition that causes `ft_transfer` to fail) will have their ERC-20 tokens permanently burned with no compensation. The NEP-141 tokens remain in Aurora's custody, but the corresponding ERC-20 supply has been reduced — a permanent, irrecoverable fund freeze for the affected user. This matches the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The trigger is fully user-controlled: any EVM address can call `withdrawToNear` with an arbitrary `recipient` byte string. A recipient that is not registered with the NEP-141 contract is sufficient to cause `ft_transfer` to fail on NEAR. This requires no special privilege, no admin key, and no external oracle. The likelihood is **High** because unregistered recipients are a common real-world condition (e.g., a user mistyping a NEAR account ID, or targeting an account that has not called `storage_deposit` on the NEP-141 contract).

---

### Recommendation

Ensure the `error_refund` feature is always compiled into the production binary, or unconditionally attach the `exit_to_near_precompile_callback` for all ERC-20 exit paths regardless of the feature flag. The callback should always be present when ERC-20 tokens have been burned, so that a failed NEP-141 transfer triggers a re-mint. The `error_refund` guard should be removed from the `refund` field population in `ExitToNear::run`, making the refund path unconditional for ERC-20 exits.

---

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora (deploy ERC-20 mirror).
2. Acquire ERC-20 tokens on Aurora.
3. Call `withdrawToNear(bytes("unregistered.near"), amount)` on the ERC-20 contract from an EVM address.
4. The ERC-20 contract calls `_burn(msg.sender, amount)` — tokens are destroyed.
5. The `ExitToNear` precompile schedules `ft_transfer` to `unregistered.near` with **no callback** (because `error_refund` is absent and `transfer_near_args` is `None`).
6. The NEAR `ft_transfer` promise fails because `unregistered.near` has no storage deposit on the NEP-141 contract.
7. No `exit_to_near_precompile_callback` fires; `refund_on_error` is never called.
8. The caller's ERC-20 balance remains at zero; the NEP-141 tokens remain in Aurora's custody. Funds are permanently frozen.
