### Title
Gateway `skip_stateful_validations` Ignores `max_nonce_for_validation_skip` Config — Hardcoded `== Nonce(Felt::ONE)` Instead of Configurable Range Check - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function hardcodes an exact equality check `tx.nonce() == Nonce(Felt::ONE)` instead of using the configurable `max_nonce_for_validation_skip` field from `StatefulTransactionValidatorConfig`. The `native_blockifier` path correctly implements a range check (`Nonce(Felt::ONE) <= tx_nonce && tx_nonce <= self.max_nonce_for_validation_skip`), but the gateway path ignores the config entirely. This is the direct Sequencer analog of the MinipoolManager `==` vs. `<=` boundary-check bug: an exact-match guard where a range check was intended.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the free function `skip_stateful_validations` decides whether to skip the `__validate__` entry-point call for an invoke transaction submitted alongside a pending `deploy_account`: [1](#0-0) 

The critical line is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [2](#0-1) 

The config struct defines `max_nonce_for_validation_skip` precisely to make this threshold configurable: [3](#0-2) 

The caller `run_pre_validation_checks` has full access to `self.config` but passes none of it to `skip_stateful_validations`: [4](#0-3) 

By contrast, the `native_blockifier` path correctly implements the range check using the same config field:

```rust
let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [5](#0-4) 

The gateway path never reads `max_nonce_for_validation_skip` at all — confirmed by grep: the field appears only in `native_blockifier/src/py_validator.rs` and config/schema files, never in `apollo_gateway`.

### Impact Explanation

Two exploitable divergences from the intended invariant:

**Scenario A — operator sets `max_nonce_for_validation_skip = 0` to disable the skip:**
The hardcoded `== Nonce(Felt::ONE)` still fires. Any attacker who has a `deploy_account` transaction in the mempool (or a recent block) for their address can submit an invoke with nonce 1 carrying an arbitrary/invalid signature. `run_validate_entry_point` is called with `validate: false`, so the `__validate__` entry point is never executed at the gateway admission stage. The transaction is admitted to the mempool without signature verification, violating the operator's explicit configuration to disable this bypass.

**Scenario B — operator sets `max_nonce_for_validation_skip = 2` (or higher) to extend the UX skip:**
The hardcoded `== Nonce(Felt::ONE)` only matches nonce 1. Invoke transactions with nonce 2 (or higher, up to the configured maximum) from accounts with a pending `deploy_account` are rejected at the gateway even though the config says they should be admitted. Valid transactions are incorrectly rejected before sequencing.

The `skip_validate` flag flows directly into the blockifier validation call: [6](#0-5) 

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded value, so the bug is invisible in default deployments. However, the field is fully serialized, documented, and exposed in the config schema: [7](#0-6) 

Any operator who reads the config documentation and attempts to tune or disable the validation-skip feature will silently get the wrong behavior. The `native_blockifier` path working correctly creates a false sense that the feature is properly parameterized end-to-end.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded equality check with the same range logic used in `native_blockifier`:

```rust
// In run_pre_validation_checks, pass the config value:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,  // <-- add this
).await?;

// In skip_stateful_validations, replace:
//   if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)
// with:
if account_nonce == Nonce(Felt::ZERO)
    && tx.nonce() >= Nonce(Felt::ONE)
    && tx.nonce() <= max_nonce_for_validation_skip
```

This mirrors the correct implementation in `native_blockifier/src/py_validator.rs` lines 112–118.

### Proof of Concept

1. Deploy a sequencer node with `max_nonce_for_validation_skip = 0` in `StatefulTransactionValidatorConfig` (operator intent: disable the UX skip entirely).
2. Submit a `deploy_account` transaction for address `A` — it enters the mempool.
3. Submit an `invoke` transaction from address `A` with `nonce = 1` and a completely invalid signature (e.g., all-zero).
4. Observe: `skip_stateful_validations` evaluates `tx.nonce() == Nonce(Felt::ONE)` → `true`, `account_nonce == Nonce(Felt::ZERO)` → `true`, `account_tx_in_pool_or_recent_block(A)` → `true` (the deploy_account is in the pool). Returns `skip_validate = true`.
5. `run_validate_entry_point` is called with `validate: false` — `__validate__` is never executed.
6. The invalid-signature invoke is admitted to the mempool, contradicting the operator's `max_nonce_for_validation_skip = 0` configuration.

The config field that should have prevented this is read at: [8](#0-7) 
but is never consumed by the gateway's `skip_stateful_validations` path.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
