### Title
Unauthenticated Invoke Transaction Bypasses `__validate__` Signature Check via `skip_stateful_validations` Race — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 when a deploy-account for the same address is already in the mempool (UX improvement). The guard that enables this skip is `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from that address in the pool — not specifically a deploy-account. An unprivileged attacker who observes a pending deploy-account for address X can front-run the legitimate user's first invoke by submitting an invoke(nonce=1) with an **invalid or arbitrary signature**. The gateway accepts it without calling `__validate__`, admitting an unauthenticated transaction to the mempool.

### Finding Description

The relevant code path in `crates/apollo_gateway/src/stateful_transaction_validator.rs`:

**`run_pre_validation_checks`** calls `skip_stateful_validations`, whose return value controls whether `__validate__` is called:

```
run_pre_validation_checks
  → validate_state_preconditions   (nonce range, resource bounds)
  → validate_by_mempool            (duplicate hash/nonce check only)
  → skip_stateful_validations      ← decides whether to skip __validate__
```

`skip_stateful_validations` returns `true` (skip) when:
1. Transaction is `Invoke`
2. `tx.nonce() == 1`
3. `account_nonce == 0` (account not yet deployed in committed state)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

Condition 4 is satisfied by **any** transaction from that address in the pool, not only a deploy-account: [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate = false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage` — the `__validate__` entry point (which verifies the signature) is never called: [4](#0-3) 

`perform_pre_validation_stage` with `strict_nonce_check = false` only checks nonce ordering (≥ account nonce), fee bounds, and proof facts — it does not verify the cryptographic signature: [5](#0-4) 

### Impact Explanation

An attacker can inject an invoke transaction carrying an **arbitrary/invalid signature** into the mempool for any account that is currently being deployed. This directly satisfies the allowed impact: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

Secondary effects:
- The attacker's invalid invoke occupies the `(address, nonce=1)` slot in the mempool.
- The legitimate user's invoke(nonce=1) is rejected by the mempool as `DuplicateNonce`.
- The attacker's invalid invoke eventually fails during batcher execution (the batcher sets `validate=true` independently), but by then the legitimate user's transaction has been displaced.
- The attacker can repeat this for every new deploy-account observed in the mempool, creating a persistent griefing vector against all new account deployments.

### Likelihood Explanation

- Mempool contents are observable via P2P propagation and RPC endpoints.
- The attacker needs only to detect a pending deploy-account and submit an invoke(nonce=1) before the legitimate user does — a standard front-running scenario.
- No privileged access, special tokens, or rebase tokens are required.
- The attack is cheap: the attacker pays only the gateway admission cost (no on-chain fee is charged for a transaction that fails `__validate__` during execution).

### Recommendation

`skip_stateful_validations` must verify that the transaction in the pool for the target address is specifically a **deploy-account** transaction, not just any transaction. The check should be:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use a new mempool query:
mempool_client.has_deploy_account_in_pool(tx.sender_address())
```

Alternatively, add a dedicated `deploy_account_in_pool` field to the mempool state, or query the transaction type of the lowest-nonce transaction for the address before deciding to skip validation.

### Proof of Concept

```
1. Legitimate user U prepares: deploy_account(X) + invoke(X, nonce=1, valid_sig)
2. U submits deploy_account(X) → accepted (nonce=0, constructor runs, no __validate__)
3. Attacker A observes deploy_account(X) in mempool via RPC/P2P
4. A submits invoke(X, nonce=1, sig=GARBAGE) before U submits their invoke
   Gateway checks:
     validate_nonce:          account_nonce=0, tx_nonce=1, 0 ≤ 1 ≤ max_gap → PASS
     validate_by_mempool:     no existing (X,1) in pool → PASS
     skip_stateful_validations:
       tx.nonce()==1 ✓, account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(X) → true (deploy_account is in pool) ✓
       → returns true (skip __validate__)
     run_validate_entry_point: validate=false → __validate__ NOT called → PASS
   Result: A's invalid invoke is admitted to the mempool.
5. U submits invoke(X, nonce=1, valid_sig)
   validate_by_mempool: DuplicateNonce → REJECTED
6. A's invalid invoke is eventually executed by batcher with validate=true → __validate__ fails → rejected
7. U must resubmit; A can repeat from step 4 indefinitely.
``` [6](#0-5) [1](#0-0) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
