### Title
Unbounded `stakers` Vec Growth Enables Permanent Griefing of `get_stakers` via Unbounded Gas Consumption — (File: `src/staking/staking.cairo`)

---

### Summary
The `stakers` storage `Vec` in the Staking contract grows monotonically with every new staker and is **never pruned**. The `get_stakers` function iterates over the entire Vec on every call. An unprivileged attacker can register many staker accounts (each requiring only `min_stake` STRK, fully recoverable after the exit-wait window) to bloat the Vec, eventually making `get_stakers` too expensive to execute and permanently disrupting the consensus layer's ability to retrieve the active validator set.

---

### Finding Description

In `src/staking/staking.cairo`, the `stake()` function unconditionally pushes every new staker address into the `stakers: Vec<ContractAddress>` storage vector: [1](#0-0) 

The contract explicitly documents that this vector is never cleaned up: [2](#0-1) 

The `get_stakers` function iterates over the **full range** of this vector on every invocation: [3](#0-2) 

Crucially, `unstake_action` deletes the staker's `staker_info` record but **does not remove the address from `stakers`**: [4](#0-3) 

Each iteration of `get_stakers` still performs a storage read for the address and a call to `is_staker_active`. With N dead entries, this is O(N) storage reads per call, with no upper bound enforced anywhere in the contract.

---

### Impact Explanation

`get_stakers` is the primary interface through which the consensus layer retrieves the active validator set for a given epoch: [5](#0-4) 

If the `stakers` Vec is bloated with enough dead entries, `get_stakers` exceeds the gas limit and becomes permanently uncallable. There is no administrative function to prune the Vec, no snapshot, and no rollback mechanism — exactly the class of issue described in the reference report. The only remediation would be a contract upgrade, which requires privileged governance action.

**Impact:** Unbounded gas consumption (Medium per scope), with secondary risk of consensus layer disruption.

---

### Likelihood Explanation

The attack requires `N × min_stake` STRK locked for at most `exit_wait_window` (default 1 week, max 12 weeks per `MAX_EXIT_WAIT_WINDOW`): [6](#0-5) 

After the exit-wait window, the attacker recovers all principal via `unstake_action`. The net cost is gas fees plus the opportunity cost of temporarily locked capital. There is no profit motive, but the damage to the protocol is permanent without a governance upgrade. The entry path (`stake()`) is fully permissionless and requires no privileged role.

---

### Recommendation

1. **Remove staker addresses from `stakers` during `unstake_action`**, or replace the `Vec` with a data structure that supports O(1) deletion (e.g., a swap-and-pop pattern or an `IterableMap`).
2. **Maintain a separate active-staker count** and cap iteration in `get_stakers` to active entries only.
3. **Enforce a minimum stake high enough** to make mass-registration economically infeasible, or add an explicit cap on the total number of registered stakers.

---

### Proof of Concept

1. Attacker creates N accounts, each funded with `min_stake` STRK.
2. Each account calls `stake()` → each call executes `self.stakers.push(staker_address)` (line 348), growing the Vec by 1.
3. Each account immediately calls `unstake_intent()` → staker is marked for exit; address **remains** in `stakers`.
4. After `exit_wait_window` elapses, each account calls `unstake_action()` → `staker_info` is deleted (line 1693), but the address **remains** in `stakers`.
5. Attacker recovers all `min_stake` STRK — net cost is only gas.
6. Any subsequent call to `get_stakers` must iterate over all N dead entries, performing N storage reads.
7. With sufficient N, `get_stakers` exceeds the Starknet gas limit and reverts on every call.
8. The consensus layer can no longer retrieve the validator set; no on-chain recovery path exists without a privileged contract upgrade.
