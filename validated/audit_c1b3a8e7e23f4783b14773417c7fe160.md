### Title
Gateway `skip_stateful_validations` Skips Signature Verification for Invoke Nonce=1 When Any Account Transaction Exists in Mempool, Not Only Deploy-Account - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call (signature verification) for an invoke transaction with nonce=1 when the account's on-chain nonce is 0 and a deploy-account transaction is believed to be pending. However, the predicate used — `account_tx_in_pool_or_recent_block` — returns `true` for **any** account that has **any** transaction in the mempool pool or recent committed block, not exclusively a deploy-account transaction. An attacker can exploit this by submitting an invoke transaction with nonce=1 and an **invalid signature** for any existing account (on-chain nonce=0) that happens to have a pending nonce=0 invoke in the mempool, causing the gateway to admit the invalid transaction without signature verification.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements the deploy-account + invoke UX shortcut:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            ...
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The comment claims the check is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of this claim is the flaw: a valid nonce=0 invoke from the legitimate account owner passes validation and lands in the pool, but this does **not** imply that a subsequent nonce=1 transaction from the same address carries a valid signature. The two transactions are independent.

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`state.contains_account` returns `true` if the address appears in `staged` or `committed` maps — both of which are populated by any transaction type (invoke, declare, deploy-account):

```rust
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `execution_flags.validate = false`, completely bypassing the `__validate__` entry-point call:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

The nonce check in `validate_nonce` does not block this path: for `account_nonce=0` and `tx_nonce=1`, the condition `0 <= 1 <= max_allowed_nonce_gap (200)` passes cleanly: [5](#0-4) 

### Impact Explanation

An attacker who observes a pending nonce=0 invoke for an existing account X (on-chain nonce=0) can submit a nonce=1 invoke for X with a **fabricated or zeroed signature**. The gateway will:

1. Pass stateless validation (no signature content check at that layer).
2. Pass `validate_nonce` (nonce=1 is within the allowed gap).
3. Pass `validate_by_mempool` (mempool state sees no prior nonce conflict).
4. Enter `skip_stateful_validations`, find `account_tx_in_pool_or_recent_block(X) = true` due to the pending nonce=0 invoke, and return `skip_validate = true`.
5. Skip `run_validate_entry_point` entirely.
6. Admit the transaction to the mempool.

The admitted transaction carries an invalid signature and will fail `__validate__` during block execution, wasting sequencer execution resources. Because the gateway is the admission gate, this breaks the invariant that every transaction in the mempool has passed signature verification. This matches the **High** impact tier: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

### Likelihood Explanation

The precondition — an account with on-chain nonce=0 and a pending nonce=0 invoke in the public mempool — is observable by any network participant. Mempool contents are visible via RPC. The attack requires no privileged access, no special account state beyond what is publicly observable, and no cryptographic capability. Any unprivileged actor can execute it.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` predicate with a check that specifically confirms a **deploy-account** transaction is pending for the address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that inspects the transaction type stored in the pool, rather than returning true for any transaction type. Alternatively, restrict the skip to only the case where the account does not yet exist on-chain (i.e., `get_class_hash_at(address) == ClassHash::default()`), which is the only scenario where skipping `__validate__` is semantically justified.

### Proof of Concept

1. Deploy account X on-chain (nonce becomes 0 after deployment).
2. Submit a valid invoke with nonce=0 for X (valid signature). It enters the mempool; `tx_pool.contains_account(X)` becomes `true`.
3. Craft an invoke transaction for X with nonce=1 and a zeroed/random signature.
4. Submit it to the gateway.
5. Gateway flow:
   - `stateless_tx_validator.validate` passes (no signature content check).
   - `validate_nonce`: `0 <= 1 <= 200` → passes.
   - `validate_by_mempool`: no nonce conflict → passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
   - `run_validate_entry_point` called with `validate=false` → `__validate__` is never invoked.
6. Transaction is admitted to the mempool with an invalid signature, violating the admission invariant. [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
