### Title
Inconsistent Proposer-as-Authorizer Weight Check Between Collection Validator and FVM Verifier - (File: fvm/transactionVerifier.go)

### Summary

The `access/validator/validator.go` `checkAccounts` function and the `fvm/transactionVerifier.go` `verifyTransaction` function apply inconsistent authorization rules for an account that is simultaneously the **proposer** and an **authorizer**. The Collection-node validator skips the authorizer signature requirement for the proposer, but the FVM verifier does not — causing a valid transaction pattern to be silently rejected at execution time.

### Finding Description

In `access/validator/validator.go`, `checkAccounts` skips the authorizer-signature check for any account that is also the proposer or the payer:

```go
for _, authorizer := range tx.Authorizers {
    if authorizer == tx.Payer || authorizer == tx.ProposalKey.Address {
        // at this point, payer and proposer are guaranteed to have signatures
        continue
    }
    if !observedEnvelopeSig[authorizer] && !observedPayloadSig[authorizer] {
        return MissingSignatureError{...}
    }
}
``` [1](#0-0) 

This means a transaction where the proposer is also an authorizer and signs **only the envelope** passes Collection-node validation.

However, in `fvm/transactionVerifier.go`, `verifyTransaction` only skips the authorizer weight check when the authorizer is the **payer** — it does **not** skip when the authorizer is the proposer:

```go
for _, addr := range tx.Authorizers {
    // Skip this authorizer if it is also the payer. ...
    if addr == tx.Payer {
        continue
    }
    if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
        return errors.NewAccountAuthorizationErrorf(...)
    }
}
``` [2](#0-1) 

The `payloadWeights` map is populated exclusively from `tx.PayloadSignatures`. If the proposer-as-authorizer signed only the envelope, their weight in `payloadWeights` is **0**, causing the check to fail with `"authorizer account does not have sufficient signatures"`. [3](#0-2) 

The comment in `verifyTransaction` even acknowledges the payer-as-authorizer/proposer case but the code only implements the payer half:

> `// Skip this authorizer if it is also the payer. In the case where an account is both a PAYER as well as an AUTHORIZER or PROPOSER, that account is required to sign only the envelope.` [4](#0-3) 

### Impact Explanation

A transaction where the proposer is also an authorizer (but not the payer) and signs only the envelope:
1. Passes `checkAccounts` at the Collection node — the transaction is accepted into the mempool and included in a block.
2. Fails `verifyTransaction` inside the FVM at execution time — the transaction is rejected with an authorization error.

This is broken functionality: a transaction pattern that the protocol's own validator explicitly permits is silently rejected by the execution layer. The user's transaction is consumed (sequence number incremented) but the execution fails, wasting fees and breaking the expected behavior.

### Likelihood Explanation

The scenario is realistic. Any user who wants to be both the proposer (to control nonce/sequence) and an authorizer (to access their own account resources) while using a separate payer account will hit this. This is a documented and common multi-party transaction pattern in Flow. The inconsistency is not guarded by any test covering proposer-as-authorizer signing only the envelope.

### Recommendation

In `fvm/transactionVerifier.go`, extend the skip condition to also exclude the proposer, mirroring the logic in `access/validator/validator.go`:

```go
for _, addr := range tx.Authorizers {
    if addr == tx.Payer || addr == tx.ProposalKey.Address {
        continue
    }
    if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
        return errors.NewAccountAuthorizationErrorf(...)
    }
}
```

### Proof of Concept

1. Construct a `TransactionBody` where `ProposalKey.Address == Authorizers[0]` and `Payer` is a different address.
2. Sign only the **envelope** with the proposer/authorizer account (valid per Flow signing rules for a proposer).
3. Sign the envelope with the payer account.
4. Submit the transaction.
5. `checkAccounts` in `access/validator/validator.go` passes (line 453 skips the proposer-as-authorizer).
6. The FVM's `verifyTransaction` fails at line 257: `payloadWeights[proposerAuthorizerAddr]` = 0 < `keyWeightThreshold`, returning `"authorizer account does not have sufficient signatures"`. [2](#0-1) [1](#0-0)
