### Title
Missing Payout Address Validation Allows Silent Corruption of Pool Farming Reward Destination - (`packages/gui/src/components/plotNFT/PlotNFTPayoutInstructionsDialog.tsx`)

### Summary
`PlotNFTPayoutInstructionsDialog` accepts and submits any string — including an empty string — as the pool payout address without validation. The underlying `usePayoutAddress` hook silently falls back to using the raw (potentially empty) input when bech32m decoding fails, permanently overwriting the pool payout instructions with an invalid value and redirecting all future pooled farming XCH rewards to an unrecoverable destination.

### Finding Description

`PlotNFTPayoutInstructionsDialog` renders a free-form `TextField` for the pool payout address with no `required` constraint and no `validate` rule registered via react-hook-form: [1](#0-0) 

Compare this to `FarmManageFarmingRewards`, which correctly registers both fields with `required: true` and `validate: checkAddress`: [2](#0-1) 

When the user clears the field and clicks Save, `handleSubmit` calls `setPayoutAddress("")` unconditionally: [3](#0-2) 

Inside `usePayoutAddress`, `handleSetPayoutAddress` attempts `fromBech32m(newPayoutAddress)`. When that throws (as it will for an empty string or any non-bech32m input), the catch block **silently assigns the raw input** — the empty string — as `newPayoutInstructions`, then immediately calls `setPayoutInstructions` with it: [4](#0-3) 

The RPC call `set_payout_instructions` is then dispatched to the farmer daemon with `payoutInstructions: ""`: [5](#0-4) 

### Impact Explanation

Once the pool payout instructions are overwritten with an empty or garbage string, the pool operator has no valid address to which it can send XCH rewards. All future pooled farming rewards for that PlotNFT are directed to an invalid destination. Because `set_payout_instructions` is a persistent daemon-level configuration, the corruption survives restarts. The user receives no warning — the dialog closes normally after a successful RPC call. This constitutes a **payout change affecting pooled farming rewards** (XCH), matching the Critical/High impact scope.

### Likelihood Explanation

The "Edit Payout Instructions" dialog is reachable from the PlotNFT card menu for any user running a pool-joined PlotNFT. The field defaults to the current payout address, but a user who accidentally clears it and clicks Save — or who pastes a non-bech32m string (e.g., a Bitcoin address, a URL, or whitespace) — will silently corrupt the payout destination. No confirmation, no format check, and no error is shown.

### Recommendation

1. Register the `payoutAddress` field with `required: true` and a `validate` function (mirroring `checkAddress` in `FarmManageFarmingRewards`) so the form rejects empty or malformed input before submission.
2. In `usePayoutAddress.handleSetPayoutAddress`, replace the silent catch-fallback with an explicit validation guard: if `fromBech32m` throws and the input is not a known-valid alternative format, throw an error rather than forwarding the raw string.
3. Add a confirmation dialog that displays the resolved address before committing the change, consistent with the pattern used for other destructive farming configuration changes.

### Proof of Concept

1. Open the Chia GUI with a pool-joined PlotNFT.
2. Click the "⋮" menu on the PlotNFT card → **Edit Payout Instructions**.
3. Clear the "Pool Payout Instructions" field entirely.
4. Click **Save**.
5. Observe: no validation error is shown; the dialog closes successfully.
6. The farmer daemon now has `payoutInstructions: ""` for that launcher ID.
7. All subsequent pool reward payouts target an empty/invalid address — XCH rewards are permanently unrecoverable.

The root cause is the missing `required`/`validate` registration on the `payoutAddress` field in `PlotNFTPayoutInstructionsDialog` (line 76–89) combined with the silent fallback in `usePayoutAddress.handleSetPayoutAddress` (lines 27–31), directly analogous to the missing zero-address guard on `feeTo` in the external report.
