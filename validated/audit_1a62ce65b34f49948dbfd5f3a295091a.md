### Title
Unbounded Loop Over Delegator Balance Trace Enables Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
A pool member (delegator) who repeatedly calls `add_to_delegation_pool` across many epochs without claiming rewards grows their `pool_member_epoch_balance` trace without bound. The `calculate_rewards` function iterates over every trace entry since the last claim in an explicitly acknowledged unbounded loop. Once the trace is large enough, `claim_rewards` exceeds the Starknet gas limit and permanently reverts, freezing the delegator's unclaimed yield forever.

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over the entire `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current trace length: [1](#0-0) 

The code itself acknowledges the risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

Each call to `add_to_delegation_pool` invokes `increase_member_balance` → `set_member_balance`, which inserts a new checkpoint keyed by `current_epoch + K`: [2](#0-1) 

The `insert` logic in the trace only deduplicates within the same epoch key; calls in different epochs always append a new entry: [3](#0-2) 

The `entry_to_claim_from` cursor that limits the loop is only advanced inside `claim_rewards`: [4](#0-3) 

If the delegator never calls `claim_rewards`, `entry_to_claim_from` stays at `0` and the loop iterates over every entry ever written. After N epochs of balance changes without claiming, the loop performs N iterations, each involving multiple storage reads and a `find_sigma` call: [5](#0-4) 

`claim_rewards` is also the only way to advance the cursor; there is no partial-claim or pagination mechanism. Once the gas limit is exceeded, every future `claim_rewards` call reverts and the rewards are permanently inaccessible.

The same unbounded loop is also executed inside the view function `pool_member_info_v1`, callable by anyone: [6](#0-5) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

A delegator who accumulates a sufficiently large balance trace (e.g., hundreds of epochs of balance changes without claiming) will find that every subsequent `claim_rewards` transaction reverts due to gas exhaustion. Because the cursor `entry_to_claim_from` is only updated on a successful claim, the delegator is permanently locked out of their accrued rewards with no recovery path.

### Likelihood Explanation
**Medium.**

Any delegator who:
1. Calls `add_to_delegation_pool` (or `exit_delegation_pool_intent`) in a different epoch each time, and
2. Defers claiming rewards for an extended period

will silently grow their trace. The minimum `amount` is `1`, so the cost per epoch is negligible. Long-term participants who compound delegations without regularly claiming are the natural victims. The protocol itself flags the loop as unbounded, confirming awareness of the risk.

### Recommendation
Introduce a maximum trace depth per pool member (e.g., cap the number of unclaimed balance-change entries) and/or implement a paginated `claim_rewards` that accepts a `max_entries` parameter so the delegator can drain the trace incrementally across multiple transactions. Alternatively, automatically checkpoint and consolidate the trace on each `add_to_delegation_pool` call to prevent unbounded growth.

### Proof of Concept

1. Staker stakes and opens a delegation pool.
2. Delegator calls `enter_delegation_pool` with `amount = min_delegation`.
3. For each of N epochs (e.g., N = 500):
   - Advance one epoch.
   - Delegator calls `add_to_delegation_pool` with `amount = 1` (minimum non-zero).
   - Delegator does **not** call `claim_rewards`.
4. After N epochs, `pool_member_epoch_balance` trace for the delegator has N entries; `entry_to_claim_from = 0`.
5. Delegator calls `claim_rewards`. `calculate_rewards` enters the `while` loop and iterates N times, each iteration reading from storage and calling `find_sigma`.
6. At a sufficiently large N the transaction exceeds the Starknet gas limit and reverts.
7. Every subsequent `claim_rewards` call also reverts — the delegator's accumulated yield is permanently frozen.
