### Title
Commitment-Hash Equivocation via Non-Echo-Broadcast Round Enables Split-View DKG Denial - (File: src/dkg.rs)

---

### Summary

In `do_keyshare`, the per-participant commitment hash is disseminated via a plain `send_many` / `recv_from_others` exchange (not echo broadcast), while the commitment itself is later disseminated via `do_broadcast` (echo broadcast). A single malicious participant can exploit this asymmetry to send a different commitment hash to each honest peer. Because the commitment is consistent (echo-broadcast) but the hash is not, some honest participants accept the commitment while others reject it, producing an irreconcilable split view that permanently stalls all honest parties.

---

### Finding Description

**Round 2 – commitment hash sent without echo broadcast** [1](#0-0) 

```
let wait_round_1 = chan.next_waitpoint();
chan.send_many(wait_round_1, &commitment_hash)?;   // ← plain broadcast, no consistency guarantee

let mut all_hash_commitments = ParticipantMap::new(&participants);
all_hash_commitments.put(me, commitment_hash);

for (from, their_commitment_hash) in
    recv_from_others(&chan, wait_round_1, &participants, me).await?
{
    all_hash_commitments.put(from, their_commitment_hash);
}
```

`chan.send_many` produces a single `Action::SendMany` that the application layer delivers to all peers. A malicious participant is not bound by the library's honest implementation; they can craft and deliver distinct byte payloads to each peer at the transport layer. `recv_from_others` stores the first value received from each sender with no consistency check.

**Round 3 – commitment sent *with* echo broadcast** [2](#0-1) 

```
let commitments_and_proofs_map = do_broadcast(
    &mut chan,
    &participants,
    me,
    (commitment, proof_of_knowledge),
)
.await?;
```

`do_broadcast` runs the full three-phase echo-broadcast protocol, guaranteeing that every honest participant receives the *same* commitment from each sender.

**Round 4 – hash verification against the non-echo-broadcast value** [3](#0-2) 

```
verify_commitment_hash(
    &session_id,
    p,
    &mut commit_domain_separator.clone(),
    commitment_i,
    &all_hash_commitments,
)?;
```

`verify_commitment_hash` computes `H(participant ‖ commitment ‖ session_id)` and compares it to the hash stored in `all_hash_commitments`: [4](#0-3) 

Because `all_hash_commitments` was populated from the non-echo-broadcast round, different honest participants may hold different hashes for the same malicious sender. The commitment (from echo broadcast) is identical for all, so the hash check passes for some and fails for others.

**Permanent stall in `broadcast_success`**

Participants that fail `verify_commitment_hash` return `Err(ProtocolError::InvalidCommitmentHash)` and never reach `broadcast_success`. Participants that pass continue and call: [5](#0-4) 

`broadcast_success` calls `do_broadcast`, which runs `reliable_broadcast_receive_all`. That function loops until every participant's session slot reaches `finish_ready`: [6](#0-5) 

```
if state.iter().all(|x| x.finish_ready) {
    return Ok(vote_output);
}
```

Participants that aborted never send their `Send` message for this round. The surviving participants wait indefinitely; there is no timeout in the library. The DKG is permanently stalled for every honest party.

---

### Impact Explanation

**High – Permanent denial of key generation for honest parties.**

A single malicious participant can, with one targeted equivocation, cause the entire DKG (or reshare/refresh) to fail permanently for all honest participants. No threshold of honest parties can recover without restarting the protocol from scratch, and the attack can be repeated on every restart. The same path applies to `do_reshare` and `do_refresh`, which both call `do_keyshare`.

---

### Likelihood Explanation

The attack requires only that a malicious participant deliver different byte payloads to different peers during Round 2. This is a standard capability of any network-level adversary or any participant who does not use the honest library implementation. No cryptographic break, no key leakage, and no external dependency is required. The attack is deterministic and repeatable.

---

### Recommendation

Replace the plain `send_many` / `recv_from_others` exchange for the commitment hash with `do_broadcast` (echo broadcast). This ensures every honest participant holds the same hash for each sender before the reveal round, closing the equivocation window.

```rust
// Round 2 – send commitment hash via echo broadcast
let all_hash_commitments = do_broadcast(
    &mut chan,
    &participants,
    me,
    commitment_hash,
)
.await?;
```

Alternatively, eliminate the hash-then-reveal round entirely and send the commitment directly via echo broadcast in a single round, relying on echo broadcast's own consistency guarantee to prevent adaptive commitment changes.

```rust
fn verify_commitment_hash<C: Ciphersuite>(
    session_id: &HashOutput,
    participant: Participant,
    domain_separator: &mut DomainSeparator,
    commitment: &VerifiableSecretSharingCommitment<C>,
    all_hash_commitments: &ParticipantMap<'_, HashOutput>,
) -> Result<(), ProtocolError> {
    let actual_commitment_hash = all_hash_commitments.index(participant)?;
    let commitment_hash =
        domain_separate_hash(domain_separator, &(&participant, &commitment, &session_id))?;
    if *actual_commitment_hash != commitment_hash {
        return Err(ProtocolError::InvalidCommitmentHash);
    }
    Ok(())
}
```
