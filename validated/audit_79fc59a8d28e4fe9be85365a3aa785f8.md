### Title
NFT Seller Net Proceeds Displayed as Pre-Royalty Amount Due to Commented-Out Deduction — (File: `packages/gui/src/components/offers/utils.ts`)

### Summary
In `calculateNFTRoyalties`, the `nftSellerNetAmount` field is assigned the raw pre-royalty `amount` instead of `amount - royaltyAmount`. The royalty deduction was explicitly commented out. This stale value propagates into the NFT offer viewer's "Net Proceeds" display and into the `overrideNFTSellerAmount` used to render "You will receive" in the offer summary, causing the NFT seller to see an inflated amount that does not reflect what they will actually receive after creator fees are deducted.

### Finding Description
In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `royaltyAmount` correctly but then assigns `nftSellerNetAmount` to the full `amount`:

```typescript
// utils.ts lines 312–317
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The commented-out block is the correct formula. The current code returns `amount` (pre-royalty) as `nftSellerNetAmount`.

This value flows into two display paths in `NFTOfferViewer.tsx`:

1. **`overrideNFTSellerAmount`** (lines 397–402) — computed from `nftSaleInfo?.nftSellerNetAmount` and passed to `NFTOfferSummary` to override the "You will receive" amount shown to the NFT seller when a `TokenForNFT` offer is viewed.

2. **"Net Proceeds" label** (line 596) — directly renders `nftSaleInfo?.nftSellerNetAmount` under the heading "Net Proceeds", whose own tooltip (lines 588–591) explicitly states: *"The net proceeds include the asking price, minus any associated creator fees."*

Both surfaces display `amount` (e.g., 10 XCH) when the correct value is `amount - royaltyAmount` (e.g., 9 XCH for a 10% royalty NFT).

### Impact Explanation
An NFT seller viewing a `TokenForNFT` offer is shown an inflated "Net Proceeds" and "You will receive" figure that does not subtract the creator royalty. The seller may accept the offer believing they will receive the full offered amount, when in reality the royalty is deducted from their proceeds. This constitutes displaying the wrong amount at the offer-approval decision point, directly matching the High impact class: *"causes a user to approve … the wrong … amount."*

### Likelihood Explanation
This triggers for every `TokenForNFT` offer involving an NFT that has a non-zero `royaltyPercentage`. No special attacker capability is required — any user can create such an offer. The victim is the NFT seller who views the offer in `NFTOfferDetails`. The royalty percentage is set by the NFT creator and is embedded in on-chain NFT metadata, so it is always present for royalty-bearing NFTs.

### Recommendation
Restore the royalty deduction in `calculateNFTRoyalties`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString)).toFixed(12),
);
```

The `makerFee` subtraction from the commented-out code should be evaluated separately, as the maker fee is a network fee paid from the seller's XCH balance, not from the token proceeds, and its inclusion depends on asset type.
