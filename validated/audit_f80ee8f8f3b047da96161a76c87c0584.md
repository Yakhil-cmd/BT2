### Title
Declare Transactions Bypass the `max_l2_gas_amount` Admission Cap in `StatelessTransactionValidator` - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces an upper bound (`max_l2_gas_amount`) on the `l2_gas.max_amount` field for `Invoke` and `DeployAccount` transactions, but explicitly skips that check for `Declare` transactions. An unprivileged user can submit a `Declare` transaction with an arbitrarily large `l2_gas.max_amount` (up to `u64::MAX`) and the gateway will admit it to the mempool, violating the sequencer's own admission invariant.

### Finding Description

In `validate_resource_bounds`, the check reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1,210,000,000` (1.21 billion gas units): [2](#0-1) 

For `Invoke` and `DeployAccount`, any transaction whose `l2_gas.max_amount` exceeds this limit is rejected with `MaxGasAmountTooHigh`. For `Declare`, the branch is an empty no-op and the check is never reached. The test suite explicitly encodes this asymmetry — `test_invalid_max_l2_gas_amount` only parameterises over `DeployAccount` and `Invoke`: [3](#0-2) 

And a dedicated test `valid_l2_gas_amount_on_declare` asserts that a `Declare` with `max_amount = 200` passes even when the configured limit is `100`: [4](#0-3) 

The `l2_gas.max_amount` field is the per-transaction Sierra gas budget. It is used directly as the initial gas counter for execution: [5](#0-4) 

A `Declare` transaction with `l2_gas.max_amount = u64::MAX` passes every gateway check, is forwarded to the mempool via `add_tx`, and enters the batcher's execution queue with an effectively unbounded per-transaction gas limit.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The gateway's stateless validator is the first and only place where the `max_l2_gas_amount` cap is enforced. There is no compensating upper-bound check in `check_fee_bounds` (which only checks that `max_amount ≥ minimal_gas_amount`, a lower-bound): [6](#0-5) 

A `Declare` transaction with `l2_gas.max_amount = u64::MAX` is admitted to the mempool and executed with a per-transaction gas ceiling of `u64::MAX`. Block-level bouncer limits (`sierra_gas: 5,000,000,000`) still apply, so a single transaction cannot exhaust the entire block, but the per-transaction cap — which exists precisely to prevent individual transactions from monopolising block gas — is completely absent for `Declare`. An attacker can flood the mempool with `Declare` transactions each claiming `u64::MAX` L2 gas, bypassing the admission policy that governs every other transaction type.

### Likelihood Explanation

Any unprivileged user who can submit a `Declare` transaction (subject only to the `block_declare` flag and the `authorized_declarer` allowlist, both of which are operator-controlled) can trigger this. No special account state, balance, or nonce is required beyond what is needed for a normal declare. The bypass requires only setting `l2_gas.max_amount` to a value above `max_l2_gas_amount` in the RPC payload.

### Recommendation

Apply the same upper-bound check to `Declare` transactions. Remove the type-based exemption and enforce `max_l2_gas_amount` uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher gas ceiling (e.g., due to compilation overhead), introduce a separate `max_l2_gas_amount_declare` configuration parameter rather than removing the check entirely.

### Proof of Concept

1. Construct an `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = u64::MAX` and a valid Sierra contract class.
2. Submit it to the gateway's `add_tx` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised.
4. The transaction proceeds through `convert_rpc_tx_to_internal`, `extract_state_nonce_and_run_validations`, and `mempool_client.add_tx` without rejection.
5. For comparison, submit an identical `RpcInvokeTransactionV3` with the same `l2_gas.max_amount`; it is rejected with `MaxGasAmountTooHigh`.

The existing test `valid_l2_gas_amount_on_declare` already demonstrates step 3 with `max_amount = 200` against a limit of `100`: [4](#0-3)
