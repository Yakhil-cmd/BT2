### Title
Router-Mediated Swap Bypasses `SwapAllowlistExtension` Per-Trader Gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the `msg.sender` of `MetricOmmPool.swap`. When a swap is routed through `MetricOmmSimpleRouter`, that `msg.sender` is the router address, not the original trader. Any trader can therefore bypass a per-trader allowlist on a curated pool by routing through the router, provided the router address is itself allowlisted.

### Finding Description
`MetricOmmPool.swap` unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // <-- always the direct caller, i.e. the router
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then gates on that value:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][trader]`.

A pool admin who wants to allow some traders to use the router must allowlist the router address. Once the router is allowlisted, every trader — including non-allowlisted ones — can execute swaps on the curated pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point). The extension has no way to recover the original trader's identity because the router does not forward it.

The dilemma is structural: if the router is not allowlisted, all router-mediated swaps revert even for allowlisted traders; if the router is allowlisted, the per-trader gate is entirely bypassed.

### Impact Explanation
Direct bypass of the curated-pool swap allowlist. A non-allowlisted EOA executes a real swap on a pool that was configured to restrict access to specific addresses, receiving token output they should not be entitled to. This breaks the core access-control invariant of `SwapAllowlistExtension` and constitutes broken core pool functionality under the contest rules.

### Likelihood Explanation
The `MetricOmmSimpleRouter` is the primary public swap interface. Any pool that (a) uses `SwapAllowlistExtension` and (b) needs to support router-mediated swaps for at least one allowlisted user must allowlist the router, at which point the bypass is available to every trader. The required state is a natural operational configuration, not an exotic edge case.

### Recommendation
The extension should receive and verify the original trader identity, not the intermediary. Two options:

1. **Pass original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Check `recipient` instead of `sender`**: For swap allowlists the economically relevant actor is the recipient of output tokens. Gating on `recipient` is bypass-resistant regardless of routing intermediaries.
3. **Allowlist-aware router**: Add a dedicated router entry point that verifies the caller is allowlisted before forwarding, so the pool admin allowlists individual traders at the router level rather than the extension level.

### Proof of Concept
