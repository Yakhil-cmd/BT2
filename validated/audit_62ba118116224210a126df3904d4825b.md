### Title
Unbounded `MessageBuffer` Allows Malicious Participant to Exhaust Memory and Permanently Deny DKG/Signing - (File: src/protocol/internal.rs)

### Summary
The `MessageBuffer` inside `src/protocol/internal.rs` is an unbounded `HashMap<MessageHeader, SubMessageQueue>`. Any participant who can call `Protocol::message()` — the standard public entry point for delivering messages — can inject messages with arbitrary channel tags and waitpoints that the honest protocol code never polls. Each such message creates a new entry in the HashMap with no eviction or size cap. The codebase itself contains a test (`attacker_can_fill_message_buffer_with_unused_waitpoints`) that explicitly confirms 512 such entries accumulate unchecked.

### Finding Description
`MessageBuffer` is defined as:

```rust
struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}
```

Its `push` method inserts unconditionally:

```rust
fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
    let mut messages_lock = self.messages.lock().expect("lock should not fail");
    messages_lock.entry(header).or_default().send(from, message);
}
``` [1](#0-0) 

The `push_message` function, called from the public `Protocol::message()` entry point, routes any inbound message into this buffer after only a length and header-parse check — no validation of whether the waitpoint or channel tag is one the honest protocol will ever consume:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);
}
``` [2](#0-1) 

Each `SubMessageQueue` is backed by `futures::channel::mpsc::unbounded()`, so both the HashMap key-space (one entry per unique `(channel_tag, waitpoint)` pair) and each queue's depth are unbounded: [3](#0-2) 

A `MessageHeader` is a 32-byte channel tag plus an 8-byte waitpoint, giving an attacker 2^(8×40) distinct keys. The attacker only needs to vary the `waitpoint` field (a `u64`) to generate an unlimited number of distinct HashMap entries.

The repository's own test confirms the attack surface:

```rust
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;
    for i in 0..attack_count {
        let header = MessageHeader::new(ChannelTag::root_shared())
            .with_waitpoint(1_000_000 + i);
        ...
        comms.push_message(attacker, message);
    }
    let messages = comms.incoming.messages.lock()...;
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [4](#0-3) 

The test asserts that all 512 injected entries are stored — it documents the vulnerability rather than guarding against it.

Every protocol that uses `SharedChannel` or `PrivateChannel` — including `do_keygen`, `do_reshare`, `do_keyshare`, `do_broadcast`, and all FROST/ECDSA signing flows — shares the same `Comms` / `MessageBuffer` instance for the duration of the protocol run. [5](#0-4) 

### Impact Explanation
A malicious participant (or any entity that can deliver messages to a protocol instance via `Protocol::message()`) can continuously inject messages with novel waitpoints. The `MessageBuffer` HashMap grows without bound, consuming heap memory. In a long-running DKG, reshare, or signing session with many participants, this causes:

1. **Memory exhaustion / OOM kill** of the honest node's process, permanently aborting the in-progress DKG or signing session.
2. **Lock contention**: every `push` and `pop` acquires the same `std::sync::Mutex` over the entire HashMap; a large HashMap degrades all honest protocol operations running on the same `Comms` instance.

This maps to: **High — Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions**, and at minimum **Medium — Griefing or resource-exhaustion by a malicious participant causing unbounded memory growth beyond documented behavior**.

### Likelihood Explanation
The `Protocol::message()` method is the standard, documented delivery path for all inbound messages. Any participant in a DKG or signing session is expected to call it. A single malicious participant in a threshold group (even one below the signing threshold) can execute this attack by sending a stream of messages with incrementing waitpoint values before or during any protocol round. No special privileges, leaked keys, or cryptographic breaks are required.

### Recommendation
1. **Cap the HashMap size**: Reject (or drop) any inbound message whose `(channel_tag, waitpoint)` pair would create a new HashMap entry beyond a configurable maximum (e.g., `MAX_PENDING_HEADERS`).
2. **Validate waitpoints against expected protocol state**: Only accept messages for waitpoints that the local protocol state machine has already registered as pending. Discard messages for unknown waitpoints.
3. **Per-sender quotas**: Track how many buffered-but-unconsumed messages each `Participant` has contributed and drop messages from participants that exceed a threshold.
4. **Replace `unbounded()` channels**: Use bounded MPSC channels (`futures::channel::mpsc::channel(N)`) for `SubMessageQueue` to bound per-slot depth independently.

### Proof of Concept
The existing test in the repository is itself a proof of concept:

```rust
// src/protocol/internal.rs, line 532
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    // Inject 512 messages for waitpoints [1_000_000 .. 1_000_512]
    // that no honest protocol code will ever poll.
    for i in 0..512_u64 {
        let header = MessageHeader::new(ChannelTag::root_shared())
            .with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());
        comms.push_message(attacker, message);
    }
    // All 512 entries are stored — no eviction, no cap.
    assert!(comms.incoming.messages.lock().unwrap().len() == 512);
}
``` [4](#0-3) 

To escalate to OOM: replace `512` with a loop that runs until the process is killed, injecting messages with `waitpoint = i` for `i in 0..u64::MAX`. Each iteration allocates one new `SubMessageQueue` (two heap allocations for the MPSC channel) plus one HashMap entry. On a 64-bit system with 8 GB RAM, roughly 50–100 million iterations suffice to exhaust memory, well within reach of a network-speed attacker during a multi-round DKG session.
