### Title
Observer Node Upstream gRPC Connection Silently Falls Back to Unauthenticated Transport on Key Parsing Failure — (File: `cmd/observer/node_builder/observer_builder.go`)

---

### Summary

The Observer node's `deriveUpstreamIdentities()` function silently discards upstream public-key parsing errors, leaving `NetworkPubKey = nil` for any malformed entry. `Manager.createConnection()` treats a nil key as a signal to use `insecure.NewCredentials()` — completely unauthenticated gRPC. A network-positioned attacker can MITM the Observer→upstream channel and inject arbitrary API responses (script results, transaction results, events, or silently dropped transactions) to every client of that Observer.

---

### Finding Description

**Step 1 — Silent key-parsing failure in `deriveUpstreamIdentities()`** [1](#0-0) 

`NetworkPubKey` is initialised to `nil`. The JSON unmarshal error is checked only to decide whether to overwrite the field; if it fails the function continues without returning an error and without logging a warning. The caller receives a `nil` error and a skeleton with `NetworkPubKey = nil`.

**Step 2 — `validateParams()` does not catch the silent failure** [2](#0-1) 

The guard only fires when `len(upstreamNodePublicKeys) > 0` **and** the counts differ. It never inspects whether any individual key was actually parseable, so a count-matching but malformed key passes validation without error.

**Step 3 — Nil key triggers insecure gRPC dial** [3](#0-2) 

When `networkPubKey == nil`, `grpc.WithTransportCredentials(insecure.NewCredentials())` is used. No TLS handshake occurs; the channel is plaintext and unauthenticated.

**Step 4 — Forwarder passes the nil key through** [4](#0-3) 

`reconnectingClient` calls `GetAccessAPIClientWithPort(identity.Address, identity.NetworkPubKey)` directly from the stored skeleton, propagating the nil key to the connection manager.

**Step 5 — All sensitive API calls are forwarded over the unauthenticated channel** [5](#0-4) [6](#0-5) 

`ExecuteScriptAtBlockID`, `ExecuteScriptAtBlockHeight`, `SendTransaction`, `GetTransactionResult`, `GetEventsForHeightRange`, and every other forwarded endpoint traverse the same unauthenticated connection.

---

### Impact Explanation

An attacker with network-level access between the Observer and its upstream (e.g., same cloud VPC, BGP hijack, or ARP poisoning on a shared segment) can:

1. **Inject false script execution results** — return fabricated account balances or contract state to any caller of `ExecuteScriptAt*`, causing DApps or wallets to display incorrect on-chain state.
2. **Forge transaction results** — return `SEALED/SUCCESS` for a transaction that was never submitted, causing users to believe a payment or contract call completed when it did not.
3. **Suppress or replay events** — return empty or replayed event sets for `GetEventsForHeightRange`, breaking event-driven DApp logic.
4. **Silently drop submitted transactions** — accept `SendTransaction` at the Observer, return a success response, and discard the payload before it reaches the real Access Node, causing time-sensitive or fund-moving transactions to be lost.

The Observer's own README acknowledges the absence of integrity guarantees: [7](#0-6) 

This design comment describes the intended read-only limitation, but the silent key-parsing failure means operators who believe they have configured TLS are silently running without it, removing even the transport-level protection they intended to deploy.

---

### Likelihood Explanation

* The `--upstream-node-public-keys` flag accepts keys in a specific JSON-encoded `encodable.NetworkPubKey` format. Operators who supply a hex string, a PEM block, or a base64 blob — all plausible mistakes — will trigger the silent failure.
* `validateParams()` performs only a count check; it never attempts a trial parse of the supplied keys.
* The default value for `upstreamNodePublicKeys` is `[]string{}` (empty), and the flag description does not mark it as mandatory, encouraging operators to omit it or supply it in the wrong format.
* Observer nodes are commonly deployed in cloud environments where intra-VPC traffic is not encrypted by default, making network-level interception realistic without any privileged access to the Flow protocol.

---

### Recommendation

1. **Return an error on key-parse failure** in `deriveUpstreamIdentities()`: replace the silent `if err == nil` pattern with an explicit error return so the Observer refuses to start with a malformed key.
2. **Enforce key presence when addresses are provided**: add a check in `validateParams()` (or `deriveUpstreamIdentities()`) that rejects a configuration where `len(upstreamNodeAddresses) > 0` but any resulting `NetworkPubKey` is nil.
3. **Log a fatal/error** if a connection is about to be created with `networkPubKey == nil` for an upstream that was configured with a key string, so operators are alerted even in library-initialization paths.
4. **Analog to the CSP kill-switch recommendation**: expose an admin command that can atomically replace the upstream identity list at runtime, allowing operators to revoke a compromised upstream without restarting the Observer.

---

func (f *Forwarder) reconnectingClient(i int) error {
	identity := f.upstream[i].id

	accessApiClient, closer, err := f.connFactory.GetAccessAPIClientWithPort(identity.Address, identity.NetworkPubKey)
	if err != nil {
		return fmt.Errorf("failed to connect to access node at %s: %w", accessApiClient, err)
	}
	// closer is not nil iff err is nil, should use to close the connection when done
	f.upstream[i].closer = closer
	f.upstream[i].client = accessApiClient
	return nil
}
