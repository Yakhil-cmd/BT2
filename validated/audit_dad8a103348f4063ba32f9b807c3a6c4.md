### Title
Zero `priceProviderTimelock` Allows Pool Admin to Atomically Swap Price Provider and Drain LP Funds in the Same Block — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory._validatePoolParameters` does not enforce a minimum value for `priceProviderTimelock` when the pool is created in mutable-provider mode. A pool admin who sets `priceProviderTimelock = 0` at creation can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in the same block (two sequential transactions), instantly replacing the live oracle with a malicious one, executing a swap at a manipulated price, and draining LP principal.

### Finding Description

`createPool` stores `params.priceProviderTimelock` verbatim into `priceProviderTimelock[pool]` without any lower-bound check: [1](#0-0) 

The only branch on `priceProviderTimelock` is whether it equals `type(uint256).max` (immutable mode). Any other value — including `0` — is accepted as a valid mutable-mode timelock: [2](#0-1) 

In `proposePoolPriceProvider`, the deadline is computed as:

```
executeAfter = block.timestamp + timelock;   // = block.timestamp when timelock == 0
``` [3](#0-2) 

In `executePoolPriceProviderUpdate`, the guard is a strict less-than:

```
if (block.timestamp < execAfter) revert ...
``` [4](#0-3) 

When `timelock == 0`, `execAfter == block.timestamp` at proposal time. Any transaction in the **same block** satisfies `block.timestamp >= execAfter`, so `executePoolPriceProviderUpdate` succeeds immediately. The `nonReentrant` guard on each function only prevents re-entrancy within a single call; it does not prevent two separate transactions in the same block. [5](#0-4) 

### Impact Explanation

A pool admin who deployed with `priceProviderTimelock = 0` can execute the following atomic (same-block) sequence:

1. Deploy a malicious `IPriceProvider` that returns an extreme bid/ask (e.g., bid ≈ 0 or ask ≈ ∞) while correctly reporting `token0()`/`token1()` to pass `_validatePriceProvider`.
2. Call `proposePoolPriceProvider(pool, maliciousProvider)` — sets `executeAfter = block.timestamp`.
3. In the next transaction of the same block: call `executePoolPriceProviderUpdate(pool)` — passes because `block.timestamp >= executeAfter`.
4. Call `swap(...)` — the pool reads the malicious bid/ask and settles at the manipulated price, transferring LP token balances to the attacker at near-zero cost.
5. Optionally restore the legitimate provider in subsequent blocks to obscure the attack.

This is a direct loss of LP principal with no on-chain recourse. The timelock mechanism exists precisely to give LPs time to exit before an oracle change takes effect; a zero timelock eliminates that window entirely.

### Likelihood Explanation

The trigger requires the pool admin to have set `priceProviderTimelock = 0` at creation. Because `createPool` is permissionless and the factory emits `priceProviderTimelock` in the `PoolCreated` event, a malicious deployer can create a pool that appears legitimate (valid tokens, valid initial oracle, reasonable fees) while silently setting a zero timelock. LPs who do not inspect the raw event or factory storage before depositing are exposed. The attack is fully executable by the pool admin alone — no miner collusion or external dependency is required.

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters`. For example:

```solidity
uint256 constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

// inside _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooShort();
```

This ensures that even in mutable-provider mode, LPs always have at least one day to observe a pending oracle change and exit before it takes effect — matching the intent of the timelock mechanism documented in `POOL_CONFIGURATION_AND_MANAGEMENT.md`. [6](#0-5) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Attacker deploys pool with priceProviderTimelock = 0
PoolParameters memory params = _defaultParams();
params.priceProviderTimelock = 0; // zero timelock — accepted by factory
address pool = factory.createPool(params);

// LPs deposit liquidity into the pool (trusting the legitimate oracle)

// --- Same block, two transactions ---

// Tx 1: propose malicious provider (executeAfter = block.timestamp)
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(maliciousOracle));
// maliciousOracle.getBidAndAskPrice() returns bid ≈ 0, ask ≈ type(uint128).max

// Tx 2: execute immediately (block.timestamp >= executeAfter)
vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool); // succeeds — no elapsed time required

// Tx 3: swap at manipulated price, draining LP balances
router.exactInputSingle(...); // receives far more token1 than the oracle price permits

// Result: LP principal drained; admin profits the difference
```

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
