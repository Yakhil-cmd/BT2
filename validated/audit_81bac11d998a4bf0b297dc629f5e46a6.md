### Title
Missing `eligible_operational_addresses` Validation in `stake()` Bypasses Operational Address Consent Mechanism — (File: `src/staking/staking.cairo`)

---

### Summary

The `stake()` function allows a caller to register any unused address as their `operational_address` without requiring that address to have pre-declared consent via `declare_operational_address()`. The `change_operational_address()` function enforces this two-step consent check, but `stake()` does not, creating an inconsistency that enables front-running attacks to permanently lock out a legitimate operational address from its intended staker.

---

### Finding Description

The protocol implements a two-step consent mechanism for operational address assignment:

1. The operational address calls `declare_operational_address(staker_address)`, which writes `eligible_operational_addresses[operational_address] = staker_address`.
2. The staker calls `change_operational_address(operational_address)`, which enforces:

```cairo
assert!(
    self.eligible_operational_addresses.read(operational_address) == staker_address,
    "{}",
    Error::OPERATIONAL_NOT_ELIGIBLE,
);
``` [1](#0-0) 

This ensures the operational address has explicitly consented to being used by a specific staker. However, the `stake()` function — which also permanently binds an `operational_address` to a staker — performs no such check:

```cairo
fn stake(ref self: ContractState, reward_address: ContractAddress, operational_address: ContractAddress, amount: Amount) {
    ...
    assert!(
        self.operational_address_to_staker_address.read(operational_address).is_zero(),
        "{}",
        Error::OPERATIONAL_EXISTS,
    );
    // ← No eligible_operational_addresses check here
    ...
    self.operational_address_to_staker_address.write(operational_address, staker_address);
``` [2](#0-1) 

Once `operational_address_to_staker_address[X]` is set by `stake()`, the address X is permanently locked to that staker until they call `unstake_intent()` + `unstake_action()`. Any subsequent call to `change_operational_address(X)` by a legitimate staker will revert with `OPERATIONAL_EXISTS`. [3](#0-2) 

---

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users or protocol.**

An attacker can permanently prevent a target operational address from being used by its intended staker. Concretely:

- Alice controls address X and calls `declare_operational_address(alice_staker)` from X, signaling intent.
- Mallory observes this on-chain and front-runs Alice's `stake()` call by calling `stake(any_reward_addr, X, min_stake)`.
- `stake()` succeeds because it only checks `operational_address_to_staker_address[X].is_zero()` — which is still true at this point.
- X is now permanently bound to Mallory's staker record.
- Alice's subsequent `stake()` or `change_operational_address(X)` both revert with `OPERATIONAL_EXISTS`.

Alice must abandon X and repeat the `declare_operational_address` flow with a new address. If Alice's current operational address is compromised or unavailable, this attack delays her ability to attest, causing loss of attestation rewards for the duration of the disruption. [4](#0-3) 

---

### Likelihood Explanation

The attack is executable by any unprivileged address that can call `stake()` with `min_stake` tokens. The attacker must monitor the mempool for `declare_operational_address` transactions and front-run the victim's `stake()` or `change_operational_address()` call. This is a realistic on-chain front-running scenario. The cost to the attacker is the `min_stake` amount (which remains staked and is not lost), making the attack economically viable as a targeted griefing tool.

---

### Recommendation

Add the same `eligible_operational_addresses` check to `stake()` that is already present in `change_operational_address()`:

```cairo
assert!(
    self.eligible_operational_addresses.read(operational_address) == staker_address,
    "{}",
    Error::OPERATIONAL_NOT_ELIGIBLE,
);
```

This enforces the two-step consent mechanism consistently across all code paths that bind an operational address to a staker. [1](#0-0) 

---

### Proof of Concept

1. Operational address `X` calls `declare_operational_address(alice)` — sets `eligible_operational_addresses[X] = alice`.
2. Mallory observes the transaction and front-runs with `stake(mallory_reward, X, min_stake)`.
   - Check at line 298–302 passes: `operational_address_to_staker_address[X]` is still zero.
   - No `eligible_operational_addresses` check exists in `stake()`.
   - Line 342 executes: `operational_address_to_staker_address[X] = mallory_staker`.
3. Alice calls `stake(alice_reward, X, alice_amount)` — reverts at line 298–302 with `OPERATIONAL_EXISTS`.
4. Alice calls `change_operational_address(X)` — reverts at line 670–674 with `OPERATIONAL_EXISTS`.
5. X is permanently locked to Mallory's staker until Mallory completes the full `unstake_intent()` + `unstake_action()` exit cycle, during which Alice cannot use X and loses attestation rewards if she has no alternative operational address available. [5](#0-4)

        fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
            self.general_prerequisites();
            let operational_address = get_caller_address();
            assert!(
                self.operational_address_to_staker_address.read(operational_address).is_zero(),
                "{}",
                Error::OPERATIONAL_IN_USE,
            );
            assert!(
                !self.does_token_exist(token_address: operational_address),
                "{}",
                Error::OPERATIONAL_IS_TOKEN,
            );
            if self.eligible_operational_addresses.read(operational_address) == staker_address {
                return;
            }
            self.eligible_operational_addresses.write(operational_address, staker_address);
            self.emit(Events::OperationalAddressDeclared { operational_address, staker_address });
