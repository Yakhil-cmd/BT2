### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Corrupted Signing Outputs — (`File: src/frost/mod.rs`)

---

### Summary

The FROST signing module (`src/frost/mod.rs`) validates only the upper bound of the `threshold` parameter in both `assert_sign_inputs` and `presign`, but omits the lower-bound check (`threshold >= 2`) that is enforced during key generation in `src/dkg.rs`. A caller can pass `threshold = 1` (or `threshold = 0`) to the FROST signing or presigning functions, causing the protocol to proceed with an insufficient number of partial signatures and producing a cryptographically invalid/unusable signature that honest parties accept as a completed protocol run.

---

### Finding Description

In `src/dkg.rs`, `assert_key_invariants` enforces a strict lower bound:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

This ensures that during DKG, the secret is shared across a polynomial of degree ≥ 1, requiring at least 2 participants to reconstruct.

However, in `src/frost/mod.rs`, both the `presign` function and the `assert_sign_inputs` function only check the upper bound:

```rust
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

There is no corresponding `ThresholdTooSmall` check in either function. The `ReconstructionLowerBound` type itself imposes no minimum:

```rust
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
``` [4](#0-3) 

A caller invoking `presign` or `assert_sign_inputs` with `threshold = 1` (or `threshold = 0`) passes all validation checks and the protocol proceeds.

---

### Impact Explanation

With `threshold = 1` passed to FROST signing:

- The Lagrange interpolation during partial signature aggregation uses only 1 point, with a trivial coefficient of 1.
- The aggregated signature becomes `s = nonce_i + challenge * share_i`, where `share_i` is a single participant's Shamir share — **not** the reconstructed secret key (which requires ≥ 2 shares per DKG invariant).
- The resulting signature is cryptographically invalid against the public key.
- Honest parties complete the protocol and receive a corrupted, unusable signature output, with no indication that the threshold parameter was invalid.

With `threshold = 0`:
- `0 > participants.len()` is always false, so validation passes.
- Lagrange interpolation over an empty set of points may panic or produce undefined behavior, causing a denial of signing for honest parties.

**Matched allowed impact:** *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

The `assert_sign_inputs` function is a public API entry point explicitly designed for callers to validate signing inputs before running the protocol. Any library caller — including a malicious coordinator orchestrating a signing session — can supply an arbitrary `threshold` value. Since `ReconstructionLowerBound` is constructed directly from `usize` with no internal guards, passing `threshold = 1` requires no special privilege. Misconfiguration or a malicious coordinator can trivially trigger this path.

---

### Recommendation

Add a lower-bound check in both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`, mirroring the check already present in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the upper-bound check in both functions, consistent with the pattern in `src/dkg.rs`. [5](#0-4) 

---

### Proof of Concept

1. Generate keys with a valid threshold (e.g., `threshold = 2`, 3 participants) using `keygen` — this succeeds because `assert_key_invariants` enforces `threshold >= 2`.
2. Call `assert_sign_inputs` with the same 3 participants but `threshold = 1`:
   - `participants.len() = 3`, `threshold.value() = 1`
   - Check `1 > 3` → false → **no error returned**
   - Function returns `Ok(participants)` with threshold=1 silently accepted.
3. Call `presign` with `args.threshold = ReconstructionLowerBound(1)`:
   - Check `1 > 3` → false → **no error returned**
   - Presigning proceeds normally.
4. Proceed to FROST signing with threshold=1: only 1 partial signature is collected and aggregated via Lagrange interpolation with a single point.
5. The resulting signature `(R, s)` fails verification against the public key because `s` was computed from a single share rather than the reconstructed secret — honest parties receive a completed but invalid signature with no protocol-level error.
