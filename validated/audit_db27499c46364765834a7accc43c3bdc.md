### Title
Staker Can Frontrun `switch_delegation_pool` to Impose Arbitrarily High Commission via `commission_commitment` - (File: src/pool/pool.cairo)

---

### Summary

`switch_delegation_pool` in `pool.cairo` accepts only `to_staker`, `to_pool`, and `amount` — no expected commission parameter. The commission applied to the delegator's future rewards is read dynamically from the staking contract's mutable storage. A staker who holds an active `commission_commitment` can call `set_commission` in the same block to raise their commission up to `max_commission` (up to 100%) immediately before the delegator's switch lands, permanently diverting the delegator's yield to the staker.

---

### Finding Description

When a pool member calls `switch_delegation_pool`, the function signature is:

```rust
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
) -> Amount
``` [1](#0-0) 

No commission value is passed or validated. The commission that will be deducted from the delegator's rewards is stored in the staking contract under `staker_pool_info[to_staker].commission` and is read only at reward-distribution time — not at switch time.

The staking contract's `update_commission` internal function enforces that commission can only decrease **unless** an active `commission_commitment` exists:

```rust
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
} else {
    assert!(commission < old_commission, ...);
}
``` [2](#0-1) 

When a commitment is active, `set_commission` permits setting commission to **any value up to `max_commission`**, including values higher than the current commission. `is_commission_commitment_active` simply checks `current_epoch < expiration_epoch`: [3](#0-2) 

A `commission_commitment` can be set with `max_commission` up to `COMMISSION_DENOMINATOR` (10000 = 100%): [4](#0-3) 

The staker operator therefore controls a two-step weapon: first publish a low commission to attract delegators, then raise it to near-100% the moment a switch transaction is observed.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

After the switch completes, the delegator's funds are locked in the destination pool. All future rewards accrued by those funds are split according to the staker's commission. At `commission = 10000` (100%), the entire pool reward share is forwarded to the staker's reward address and zero reaches the delegator. The delegator can exit again, but the exit wait window (default one week, up to 12 weeks) means yield continues to be stolen for the full unbonding period. The principal is not at risk, but all yield during that window is permanently diverted.

---

### Likelihood Explanation

**Medium-High.** The precondition — an active `commission_commitment` — is a publicly visible on-chain state that any staker can set at will. The staker only needs to:
1. Set `commission_commitment` with `max_commission = 10000` (one transaction, no privileged role).
2. Observe a pending `switch_delegation_pool` call targeting their pool.
3. Submit `set_commission(10000)` with higher priority before the delegator's transaction is sequenced.

On Starknet, where the sequencer currently processes transactions in arrival order and the mempool is observable, this ordering attack is realistic. The staker does not need any privileged role — only their own staker key.

The code comment at line 745 of `staking.cairo` itself acknowledges this gap:

```
/// **Note**: Current commission increase safeguards still allow for sudden commission changes.
``` [5](#0-4) 

---

### Recommendation

Add a `max_commission: Commission` parameter to `switch_delegation_pool` (and symmetrically to `enter_delegation_pool`). Before executing the switch, assert that the destination staker's current commission does not exceed the caller-supplied maximum:

```rust
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
    max_commission: Commission,   // <-- new
) -> Amount {
    // ... existing checks ...
    let actual_commission = self.get_commission_from_staking_contract();
    assert!(actual_commission <= max_commission, Error::COMMISSION_EXCEEDS_MAX);
    // ... proceed ...
}
```

This mirrors the fix applied in the MechMarketplace report: require the caller to supply the critical parameter so it can be matched against on-chain state atomically.

---

### Proof of Concept

**Setup:**
- Staker S deploys a STRK pool with commission = 100 (1%).
- S calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 52)`.
- Delegator D is in pool P1 (a different staker) with `unpool_amount > 0` (exit intent already filed).
- D observes S's pool advertising 1% commission and submits:
  ```
  pool_P1.switch_delegation_pool(to_staker: S, to_pool: pool_S, amount: X)
  ```

**Attack (same block, ordered before D's tx):**
- S submits `staking.set_commission(commission: 10000)`.
  - Allowed because `commission_commitment` is active and `10000 <= max_commission`. [6](#0-5) 

**Result:**
- D's `switch_delegation_pool` executes after S's `set_commission`. [7](#0-6) 
- D's funds land in pool_S. Commission stored in staking contract is now 10000.
- Every subsequent reward distribution calls `split_rewards_with_commission` using commission = 10000, sending 100% of pool rewards to S's reward address and 0 to D.
- D must file a new exit intent and wait the full `exit_wait_window` (up to 12 weeks) to recover principal, losing all yield during that period.

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
