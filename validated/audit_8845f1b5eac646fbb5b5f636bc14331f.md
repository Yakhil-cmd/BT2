### Title
Zero `priceProviderTimelock` Bypasses Oracle Rotation Delay, Enabling Atomic Oracle Manipulation — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`_validatePoolParameters` enforces no minimum on `params.priceProviderTimelock`. A value of `0` is accepted as valid "mutable mode," but it collapses the two-step propose/execute oracle rotation into a single atomic transaction, completely nullifying the timelock protection that was introduced to fix audit finding M01.

---

### Finding Description

`createPool` distinguishes two oracle modes solely by checking whether `priceProviderTimelock == type(uint256).max`: [1](#0-0) 

Any value below `type(uint256).max` is stored verbatim and treated as a finite delay: [2](#0-1) 

`_validatePoolParameters` checks every other critical parameter for a lower bound but has no guard on `priceProviderTimelock`: [3](#0-2) 

When `timelock == 0`, `proposePoolPriceProvider` sets:

```
executeAfter = block.timestamp + 0  →  block.timestamp
``` [4](#0-3) 

`executePoolPriceProviderUpdate` then checks:

```
if (block.timestamp < execAfter)  →  block.timestamp < block.timestamp  →  false  →  passes
``` [5](#0-4) 

Both calls can be batched in a single transaction, making the oracle swap atomic and unobservable by LPs before it executes.

The timelock mechanism was introduced specifically to address M01 ("Admin can arbitrarily change the pool priceProvider"), whose resolution states that mutable mode requires "delayed execution": [6](#0-5) 

A zero timelock re-opens the exact vulnerability M01 was meant to close.

---

### Impact Explanation

A pool admin controlling a pool with `priceProviderTimelock = 0` can, in a single transaction:

1. Call `proposePoolPriceProvider(pool, maliciousOracle)` — sets `pendingPriceProviderExecuteAfter = block.timestamp`.
2. Call `executePoolPriceProviderUpdate(pool)` — timelock check passes immediately; pool's active oracle is replaced.
3. Call `swap` — pool prices are now dictated by the attacker-controlled oracle; LPs receive far less than fair value.
4. Optionally restore the original oracle to obscure the attack.

LP principal is directly at risk. The pool's swap math trusts the oracle unconditionally: [7](#0-6) 

There is no secondary price guard in the core swap path that would reject a manipulated oracle quote.

---

### Likelihood Explanation

`createPool` is permissionless once the deployer is set. Any actor can deploy a pool with `priceProviderTimelock = 0` and become its admin. LPs who deposit into such a pool — whether because they did not inspect the timelock value or because the pool was presented as legitimate — are fully exposed. The scenario is directly analogous to the seeded finding: the creator sets a critical protection parameter to zero, either by mistake or to exploit it.

---

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters` (or in `createPool` before storing the value). For example:

```solidity
// In _validatePoolParameters or createPool, before storing priceProviderTimelock:
if (params.priceProviderTimelock != type(uint256).max && params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

`MIN_PRICE_PROVIDER_TIMELOCK` should be a protocol constant (e.g., 1 hour or 24 hours) that gives LPs sufficient time to observe a pending oracle change and exit before it takes effect.

---

### Proof of Concept

```solidity
// Pool created with priceProviderTimelock = 0 (mutable, zero delay)
PoolParameters memory params = _defaultPoolParams();
params.priceProviderTimelock = 0;          // <-- no validation rejects this
address pool = factory.createPool(params);

// Attacker-controlled oracle returns manipulated prices
MockOracle maliciousOracle = new MockOracle();
maliciousOracle.setTokens(token0, token1);
maliciousOracle.setBidAndAskPrice(MANIPULATED_BID, MANIPULATED_ASK);

// Step 1 + 2 in the same transaction — timelock = 0 means executeAfter = block.timestamp
vm.startPrank(admin);
factory.proposePoolPriceProvider(pool, address(maliciousOracle));
// block.timestamp < block.timestamp  →  false  →  no revert
factory.executePoolPriceProviderUpdate(pool);
vm.stopPrank();

// Pool now prices swaps via maliciousOracle; LPs are drained at manipulated rates
```
