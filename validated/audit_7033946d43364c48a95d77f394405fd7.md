Audit Report

## Title
Blocklist Bypass via Proxy Contract — `from_address` Reflects `msg.sender` (Immediate Caller), Not True Originator — (`rs/ethereum/cketh/minter/src/deposit.rs`, `rs/ethereum/cketh/minter/src/blocklist.rs`)

## Summary

Both ckETH deposit helper contracts emit `msg.sender` — the immediate EVM caller — as the `from_address` field in their deposit events. The Rust minter parses this field verbatim and checks only it against the blocklist. A blocked address can deploy an unlisted proxy contract, route deposits through it, and cause the helper to emit the proxy's address as `from_address`. The minter then evaluates `is_blocked(proxy)` → `false` and mints ckETH or ckERC20 to the sanctioned party, directly violating the stated invariant that blocked addresses cannot use the protocol.

## Finding Description

**Root cause — Solidity emits `msg.sender`, not `tx.origin`**

`EthDepositHelper.sol` line 33 emits `msg.sender` as the indexed `from` field:
```solidity
emit ReceivedEth(msg.sender, msg.value, _principal);
```
`DepositHelperWithSubaccount.sol` line 504 and lines 525–531 do the same for both ETH and ERC20 paths, emitting `msg.sender` as the second argument to `ReceivedEthOrErc20`. In Solidity, `msg.sender` is the immediate caller of the function. If a proxy contract `P` calls `deposit()`, then `msg.sender == address(P)`, regardless of who called `P`.

**Rust parser reads `from_address` verbatim**

`ReceivedEthLogParser` reads `entry.topics[1]` as `from_address` (parser.rs L45–46). `ReceivedErc20LogParser` reads `entry.topics[2]` (L83–84). `ReceivedEthOrErc20LogParser` reads `entry.topics[2]` (L124–126). No secondary check against `tx.origin` or any other field exists.

**Blocklist check operates only on the parsed `from_address`**

In `register_deposit_events` (deposit.rs L323):
```rust
if crate::blocklist::is_blocked(&event.from_address()) {
```
`is_blocked` performs a binary search on `ETH_ADDRESS_BLOCKLIST` (blocklist.rs L107–109). If the proxy address is absent from the list, the check passes and `process_event(s, event.into_deposit())` is called at L340, triggering minting.

**Existing guards are insufficient**

The only guard is the single `is_blocked` call on `from_address`. There is no check on `tx.origin`, no on-chain allowlist of callers, and no mechanism to verify that the immediate caller is not itself a proxy for a blocked address.

## Impact Explanation

A sanctioned/blocked Ethereum address can receive minted ckETH or ckERC20 tokens on the Internet Computer by routing deposits through a freshly deployed, unlisted proxy contract. This directly violates the stated invariant in `blocklist.rs` L15: *"ETH is not accepted from nor sent to addresses on this list."* The impact is a significant ck-token security failure with concrete protocol harm — illegal minting of in-scope chain-key assets to sanctioned parties — qualifying as **High** ($2,000–$10,000) under the "Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm" category. If exploited at scale (large ETH/ERC20 amounts), it could escalate toward the Critical illegal-minting threshold.

## Likelihood Explanation

The attack requires only: (1) deploying a proxy contract on Ethereum (permissionless, ~$5–$50 in gas), (2) transferring ETH or pre-approving ERC20 tokens to the proxy, and (3) calling `deposit()` / `depositErc20()` from the proxy. No privileged access, no key compromise, no governance majority, no victim interaction. Any blocked address can execute this immediately and repeatedly. The barrier is negligible.

## Recommendation

1. **Emit and check `tx.origin`**: Modify both Solidity helpers to emit `tx.origin` as an additional indexed field alongside `msg.sender`. Update the Rust parsers to extract it and check both addresses against the blocklist in `register_deposit_events`. Reject the deposit if either is blocked.
2. **Dual-field blocklist check**: If both fields are emitted, the minter should treat a deposit as blocked if `is_blocked(from_address) || is_blocked(tx_origin)`.
3. **Coordinated upgrade**: This requires a synchronized upgrade of both the Solidity helper contracts (redeployment) and the Rust minter parser/deposit logic.

Note: Using `tx.origin` in Solidity is generally discouraged for authorization, but for a blocklist/compliance check where the goal is to identify the true human originator of a transaction, it is appropriate and does not introduce the phishing risks associated with `tx.origin`-based auth.

## Proof of Concept

```solidity
// ProxyDeposit.sol — deployed by blocked address A
contract ProxyDeposit {
    CkEthDeposit helper = CkEthDeposit(HELPER_ADDRESS);

    function doDeposit(bytes32 principal) external payable {
        // msg.sender inside helper.deposit() == address(this), NOT address(A)
        helper.deposit{value: msg.value}(principal);
    }
}
```

Attack flow:
1. Blocked address `A` deploys `ProxyDeposit` → address `P` (not on blocklist).
2. `A` calls `P.doDeposit{value: 1 ether}(principal_of_A)`.
3. `P` calls `helper.deposit{value: 1 ether}(principal_of_A)`.
4. Helper emits `ReceivedEth(P, 1 ether, principal_of_A)` — `msg.sender == P`.
5. Minter scrapes log, `ReceivedEthLogParser` parses `from_address = P` (parser.rs L45).
6. `is_blocked(P)` → `false` (blocklist.rs L107–109).
7. `process_event(s, event.into_deposit())` executes (deposit.rs L340).
8. Minter mints 1 ckETH to `principal_of_A` — controlled by blocked address `A`.

The identical pattern applies to ERC20 via `depositErc20` in `DepositHelperWithSubaccount.sol` L511–532, using `safeTransferFrom(msg.sender, ...)` and emitting `msg.sender` as `from_address`.
