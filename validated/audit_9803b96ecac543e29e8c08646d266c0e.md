### Title
Gateway Stateful Validator Omits L1 Gas and L1 Data Gas Price Threshold Checks, Admitting Transactions Guaranteed to Fail at Execution — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The stateful gateway validator enforces a minimum price threshold only for the **L2 gas** component of `AllResourceBounds`. The L1 gas and L1 data gas `max_price_per_unit` fields are never checked against the previous block's prices. A transaction with `l1_gas.max_price_per_unit = 0` and `l1_data_gas.max_price_per_unit = 0` (but a valid `l2_gas` price) passes every gateway check and is admitted to the mempool, yet is guaranteed to be rejected by the blockifier at execution time with `InsufficientResourceBounds`. This is the direct sequencer analog of the sNOTE `minimumBPT = 0` bug: a minimum bound that should be configurable and enforced is instead silently skipped.

---

### Finding Description

`validate_tx_l2_gas_price_within_threshold` is the only price-floor check in the stateful path:

```rust

fn validate_tx_l2_gas_price_within_threshold(
    &self,
    tx_resource_bounds: ValidResourceBounds,
    previous_block_l2_gas_price: NonzeroGasPrice,
) -> StatefulTransactionValidatorResult<()> {
    match tx_resource_bounds {
        ValidResourceBounds::AllResources(tx_resource_bounds) => {
            let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
            // ... checks only l2_gas price ...
        }
        ValidResourceBounds::L1Gas(_) => {
            // No validation required for legacy transactions.
        }
    }
    Ok(())
}
``` [1](#0-0) 

The stateless validator similarly only enforces a `min_gas_price` floor on `l2_gas.max_price_per_unit`:

```rust
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(...MaxGasPriceTooLow...);
}
``` [2](#0-1) 

There is no equivalent check for `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit` at either the stateless or stateful gateway layer.

The blockifier's `check_fee_bounds`, called during `perform_pre_validation_stage`, **does** check all three prices against the actual block gas prices:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(
        ResourceBoundsError::MaxGasPriceTooLow { ... }
    );
}
``` [3](#0-2) 

This means the blockifier will reject any transaction where `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit` is below the actual block price — but only **after** the transaction has already been admitted to the mempool and pulled by the batcher.

The `StatefulTransactionValidatorConfig` holds `min_gas_price_percentage` (a `u8` percentage of the previous block's L2 gas price), but this is only applied to L2 gas. The analogous L1 and L1-data gas price percentages do not exist: [4](#0-3) 

The production `config_schema.json` confirms `min_gas_price_percentage` defaults to `100` (full enforcement) for L2 gas, but no equivalent field exists for L1 or L1-data gas: [5](#0-4) 

---

### Impact Explanation

An attacker submits `AllResourceBounds` transactions with:
- `l2_gas.max_price_per_unit` ≥ `min_gas_price` (passes stateless check)
- `l2_gas.max_price_per_unit` ≥ `min_gas_price_percentage` × previous-block L2 price (passes stateful check)
- `l1_gas.max_price_per_unit = 0` and `l1_data_gas.max_price_per_unit = 0`

These transactions pass all gateway admission checks and enter the mempool. When the batcher pulls them and the blockifier runs `perform_pre_validation_stage`, they fail with `InsufficientResourceBounds` because the actual L1 gas price is always > 0 in production. The transactions consume mempool slots and batcher execution cycles without ever producing a valid receipt.

**Impact category**: High — Mempool/gateway admission accepts invalid transactions before sequencing.

---

### Likelihood Explanation

The attack requires only a valid RPC endpoint and the ability to craft a V3 (`AllResourceBounds`) transaction. No privileged access is needed. The TODO comment at line 358 of `stateful_transaction_validator.rs` explicitly acknowledges the gap (`// TODO(Arni): Consider running this validation for all gas prices.`), confirming the omission is known but unresolved. Any actor who reads the source code can exploit it immediately.

---

### Recommendation

Extend `validate_tx_l2_gas_price_within_threshold` (or introduce a parallel function) to also check `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` against the corresponding previous-block prices, scaled by a configurable `min_gas_price_percentage`. Add `min_l1_gas_price_percentage` and `min_l1_data_gas_price_percentage` fields to `StatefulTransactionValidatorConfig` (analogous to the existing `min_gas_price_percentage`), and expose them in `config_schema.json`. The stateless validator should similarly add floor checks for `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` alongside the existing `l2_gas` check.

---

### Proof of Concept

1. Obtain the current L2 gas price from the previous block (e.g., via `starknet_getBlockWithTxHashes`).
2. Craft an `RpcInvokeTransactionV3` with:
   ```
   resource_bounds = AllResourceBounds {
       l2_gas: { max_amount: 1, max_price_per_unit: <valid_l2_price> },
       l1_gas: { max_amount: 1, max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 1, max_price_per_unit: 0 },
   }
   ```
3. Submit via `starknet_addInvokeTransaction`.
4. **Stateless check** (`validate_resource_bounds`): `max_possible_fee(Tip::ZERO)` is non-zero (due to `l2_gas`); `l2_gas.max_price_per_unit ≥ min_gas_price`. ✅ Passes.
5. **Stateful check** (`validate_tx_l2_gas_price_within_threshold`): only `l2_gas.max_price_per_unit` is compared to the threshold. ✅ Passes.
6. Transaction enters the mempool.
7. Batcher pulls the transaction; blockifier runs `check_fee_bounds`: `l1_gas.max_price_per_unit (0) < actual_l1_gas_price (> 0)` → `InsufficientResourceBounds`. ❌ Rejected at execution.
8. Repeat at scale to exhaust mempool capacity. [1](#0-0) [6](#0-5) [7](#0-6)
