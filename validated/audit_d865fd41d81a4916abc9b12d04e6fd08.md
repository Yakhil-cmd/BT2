Audit Report

## Title
SNS Swap Canister Locks SNS Tokens With No On-Chain Recovery Path on Aborted Swap ‚Äî (`rs/sns/swap/src/swap.rs`)

## Summary
When an SNS decentralization swap is aborted, `finalize_inner` returns early after restoring dapp controllers without calling `sweep_sns`, leaving all pre-loaded SNS tokens locked in the swap canister's SNS ledger account. No public canister endpoint exists to recover these tokens, and the SNS governance canister remains permanently stuck in `PreInitializationSwap` mode, rendering the entire SNS non-functional without NNS governance intervention.

## Finding Description
In `rs/sns/swap/src/swap.rs`, `finalize_inner` (line 1544) executes the following control flow on abort:

1. `sweep_icp` ‚Äî ICP returned to buyers ‚úì
2. `settle_neurons_fund_participation` ‚Äî NF maturity refunded ‚úì
3. `should_restore_dapp_control()` returns `true` (line 1348‚Äì1350) because `lifecycle == Lifecycle::Aborted`
4. `restore_dapp_controllers_for_finalize` is called, then **`return finalize_swap_response`** at line 1583 ‚Äî execution ends here

`sweep_sns` at line 1595 is never reached. Additionally, `sweep_sns` itself enforces `lifecycle == Lifecycle::Committed` at line 2170‚Äì2177, so it cannot be called on an aborted swap even if exposed. The canister's public API (`canister.rs`) exposes `error_refund_icp` for ICP recovery but has no analogous `error_refund_sns` or standalone `sweep_sns` endpoint. The integration test at `rs/nervous_system/integration_tests/tests/sns_lifecycle.rs` line 1336‚Äì1338 explicitly asserts `swap_canister_balance_sns_e8s == swap_distribution_sns_e8s` after abort, confirming the tokens are retained. The same test at line 1118‚Äì1127 confirms SNS governance remains in `PreInitializationSwap` mode, blocking all non-trivial governance operations. The protocol invariant ("The swap canister can be deleted when all tokens registered with the swap canister have been disbursed to their rightful owners") is violated.

## Impact Explanation
This is a **High** severity finding. Any SNS project whose decentralization swap fails has its entire governance token allocation (the `swap_distribution_sns_e8s` amount, potentially millions of dollars in governance tokens) locked in the swap canister with no on-chain recovery path. Compounding this, the SNS governance canister is stuck in `PreInitializationSwap` mode, which blocks most governance operations (dissolving neurons, submitting most proposal types), leaving the SNS in a permanently degraded state. Recovery requires an NNS governance proposal to upgrade the swap canister ‚Äî a process that requires community coordination and is not guaranteed. This constitutes significant SNS-level protocol harm with concrete, demonstrable user impact. It does not reach Critical because the tokens are not cryptographically destroyed and NNS governance retains an upgrade path.

## Likelihood Explanation
Any SNS swap that fails to meet minimum participation thresholds triggers this outcome. No privileged access is required ‚Äî the swap transitions to `Aborted` automatically when the due date passes without `sufficient_participation`, and `finalize_swap` is a permissionless public endpoint callable by any user. This is a realistic, documented scenario that has occurred on mainnet with real SNS launches.

## Recommendation
In `finalize_inner`, before the early return at line 1583, add a `sweep_sns`-equivalent call that transfers the SNS tokens back to the SNS governance treasury (or a designated recovery subaccount). This requires either: (a) modifying `sweep_sns` to accept an explicit destination and lifecycle override for the abort path, or (b) adding a new `sweep_sns_on_abort` function that transfers the swap canister's SNS ledger balance directly to the SNS governance treasury account. Additionally, `set_sns_governance_to_normal_mode` should be called on the abort path to prevent the SNS from being permanently stuck in `PreInitializationSwap` mode.

## Proof of Concept
The existing integration test at `rs/nervous_system/integration_tests/tests/sns_lifecycle.rs` already demonstrates the issue without recognizing it as a bug:

1. Deploy an SNS with `swap_distribution_sns_e8s` tokens loaded into the swap canister.
2. Allow the swap to expire without meeting minimum participation (set `min_participants > 0`, provide zero participants).
3. Call `finalize_swap` (permissionless).
4. Query SNS ledger balance of the swap canister's default account: equals `swap_distribution_sns_e8s` (asserted at line 1338).
5. Query SNS governance mode: `PreInitializationSwap` (asserted at line 1126).
6. Attempt any governance operation (e.g., `ManageNervousSystemParameters`): rejected with `PreInitializationSwap` error (asserted at line 1249).
7. Attempt to call any SNS token recovery endpoint on the swap canister: no such endpoint exists (confirmed by `canister.rs` full API scan).
