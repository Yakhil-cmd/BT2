### Title
Cross-COA Signature Replay via Unbound `signedData` in `validateCOAOwnershipProof` — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`validateCOAOwnershipProof` verifies that a set of signatures over `signedData` are valid for a Cadence account's keys, then separately checks that the COA at the supplied `path` has the expected `evmAddress`. Neither `path` nor `evmAddress` is included in the signed payload. Because the domain tag is the same hardcoded `"FLOW-V0.0-user"` string for every call, a valid `(signedData, signatures, keyIndices)` tuple produced for COA-A can be replayed verbatim against COA-B owned by the same Cadence account, causing `isValidSignature` on COA-B to return `ValidERC1271Signature` (0x1626ba7e).

The code itself documents this gap: [1](#0-0) 

---

### Finding Description

**Signature verification** (`contract.cdc` line 1082–1086) checks only that the supplied signatures are valid over `signedData` under the hardcoded tag `"FLOW-V0.0-user"`: [2](#0-1) 

**EVM-address check** (`contract.cdc` lines 1095–1105) then borrows the COA at the caller-supplied `path` and compares its address to `evmAddress`: [3](#0-2) 

Neither `path` nor `evmAddress` is part of the signed payload. The `COAOwnershipProof` struct that is RLP-encoded and passed as `_sig` contains only `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures`: [4](#0-3) 

`CapabilityPath` is not signed; it is freely mutable by the caller. The EVM-side `isValidSignature` always passes `address(this)` as `evmAddress`: [5](#0-4) 

So the attacker only needs to swap `CapabilityPath` from `"coaA"` to `"coaB"` in the encoded proof bytes; the signature bytes remain unchanged and still verify correctly.

---

### Impact Explanation

If an attacker can supply a `(signedData=H, proof_with_path_coaB)` pair to `isValidSignature` on COA-B, and the underlying signature over H is valid for the Cadence account that owns COA-B, the call returns `ValidERC1271Signature`. Any EVM contract that relies on ERC-1271 to gate asset transfers from COA-B (e.g., a third-party EVM bridge or escrow) would treat this as a legitimate authorization.

The main Flow Cadence bridge uses `auth(Call) &CadenceOwnedAccount` for authorization, not ERC-1271: [6](#0-5) 

So the official Flow bridge is **not** directly exploitable via this path. The impact is limited to EVM-side contracts that use `isValidSignature` on a COA for authorization.

---

### Likelihood Explanation

Three hard preconditions must all hold simultaneously:

1. **The victim Cadence account must own two or more COAs at distinct capability paths.** The code comment itself notes "Flow wallets historically create at most one COA per account." [7](#0-6) 

2. **The attacker must possess a valid signature from that account over the exact 32-byte hash `H` that the target EVM contract will use as the authorization hash for COA-B.** The attacker cannot choose `H` freely; the bridge/escrow contract computes it from the action being authorized.

3. **An EVM contract must use `isValidSignature` on COA-B to gate a withdrawal**, and the hash format must coincidentally match a hash the account previously signed for COA-A.

Because the official Flow bridge does not use ERC-1271 for withdrawal authorization, and because multi-COA accounts are rare, the realistic attack surface is narrow. The vulnerability is real and acknowledged in the source, but the combination of preconditions makes exploitation unlikely in the current ecosystem.

---

### Recommendation

Bind the signed payload to the specific COA being claimed. The simplest fix is to require that `signedData` encodes `evmAddress` (e.g., `signedData = abi.encode(evmAddress, nonce, chainID)`), and enforce this inside `validateCOAOwnershipProof` rather than leaving it as a caller convention. The existing comment already identifies this as the correct mitigation: [8](#0-7) 

---

### Proof of Concept

```
1. On emulator, create Cadence account X with one key (ECDSA/Keccak-256).
2. Create COA-A stored at /public/coaA and COA-B stored at /public/coaB under account X.
3. Choose arbitrary 32-byte hash H.
4. Sign H with account X's key under domain tag "FLOW-V0.0-user" → sig.
5. Encode proof_A = COAOwnershipProof{Address:X, CapabilityPath:"coaA", KeyIndices:[0], Signatures:[sig]}.
6. Encode proof_B = COAOwnershipProof{Address:X, CapabilityPath:"coaB", KeyIndices:[0], Signatures:[sig]}.
   (Only CapabilityPath differs; sig is identical.)
7. Call COA-B.isValidSignature(H, RLP(proof_B)) from any EVM address.
8. The precompile calls validateCOAOwnershipProof(X, /public/coaB, H, [0], [sig], COA-B.address).
9. Signature check passes (sig is valid over H for account X).
10. Address check passes (COA at /public/coaB has address == COA-B.address).
11. isValidSignature returns 0x1626ba7e (ValidERC1271Signature).
```

The replay succeeds. Any EVM contract that calls `COA-B.isValidSignature(H, proof_B)` to gate an asset transfer will grant the attacker authorization they never legitimately obtained for COA-B.
