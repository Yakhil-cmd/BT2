### Title
Missing Lower-Bound Check on `threshold` in FROST Signing Validation Allows `threshold = 1`, Enabling Single-Participant Signing — (File: `src/frost/mod.rs`)

---

### Summary

Both `assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate only the upper bound of the `threshold` parameter (`threshold <= participants.len()`) but omit the lower-bound check (`threshold >= 2`). This allows a malicious library caller or coordinator to pass `threshold = 1`, which in Shamir secret sharing means a single participant's share is sufficient to reconstruct the secret and produce a valid FROST signature — completely bypassing the multi-party threshold requirement.

---

### Finding Description

In `src/frost/mod.rs`, both the `presign` function and the `assert_sign_inputs` function perform only an upper-bound check on the threshold:

**`presign` (lines 71–77):**
```rust
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: args.threshold.into(),
        max: participants.len(),
    });
}
```

**`assert_sign_inputs` (lines 144–150):**
```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
```

Neither function checks `threshold < 2`. This means `threshold = 1` (or even `threshold = 0`) is accepted without error.

Contrast this with the analogous validation functions elsewhere in the same codebase, which correctly enforce both bounds:

**`assert_key_invariants` in `src/dkg.rs` (lines 573–582):**
```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// Step 1.1
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

**`validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` (lines 693–703):**
```rust
if threshold_value > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
}
```

The lower-bound guard is present in every other threshold-validating function in the codebase except the two FROST signing entry points.

---

### Impact Explanation

**Impact: Critical — Unauthorized creation of a valid threshold signature.**

In FROST (and Shamir secret sharing generally), a threshold of `t` means `t` out of `n` participants must cooperate to produce a valid signature. If `threshold = 1` is accepted:

- Only **one** participant's signature share is needed for aggregation.
- A single malicious coordinator can run the FROST signing protocol alone, contributing only their own share, and produce a cryptographically valid threshold signature over any attacker-chosen message.
- This completely defeats the multi-party security guarantee: the entire purpose of threshold signing is that no single party can sign unilaterally.

The `PresignArguments` struct stores the threshold value, which is then passed through to the signing phase via `assert_sign_inputs`. A coordinator who controls the threshold parameter at call time can set it to 1 and proceed to sign without any other participant's involvement.

---

### Likelihood Explanation

**Likelihood: Medium.**

This requires a malicious or compromised coordinator/library caller who controls the `threshold` argument passed to `presign` or `assert_sign_inputs`. In a correctly deployed system the threshold would be agreed upon during DKG. However, the signing phase accepts a freshly supplied threshold value with no cross-check against the DKG-established threshold, so a malicious coordinator can freely supply `threshold = 1` at signing time without any other participant being able to detect or prevent it.

---

### Recommendation

Add the missing lower-bound check in both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`, mirroring the pattern already used in `assert_key_invariants` and `validate_triple_inputs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be added immediately after the upper-bound check in both functions.

---

### Proof of Concept

The discrepancy is directly observable by comparing the two code paths:

**Vulnerable path — `assert_sign_inputs` (`src/frost/mod.rs`, lines 144–150):** [1](#0-0) 

Only the upper bound is checked; `threshold = 1` passes silently.

**Secure path — `assert_key_invariants` (`src/dkg.rs`, lines 573–582):** [2](#0-1) 

Both bounds are enforced; `threshold = 1` is rejected with `ThresholdTooSmall`.

**Secure path — `validate_triple_inputs` (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`, lines 693–703):** [3](#0-2) 

A caller invoking `assert_sign_inputs(participants, 1usize, me, coordinator)` or constructing `PresignArguments { threshold: ReconstructionLowerBound::from(1usize), ... }` and calling `presign(...)` will receive no error. The signing session then proceeds with a threshold of 1, requiring only the coordinator's own share to produce a valid aggregated FROST signature.
