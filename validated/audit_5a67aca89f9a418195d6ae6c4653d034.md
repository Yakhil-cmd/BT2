### Title
Pre-EIP-155 Legacy Transactions Accepted Without Chain ID Binding, Enabling Cross-Chain Signature Replay — (`basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

The ZKsync OS bootloader explicitly accepts pre-EIP-155 legacy transactions (those with `v=27` or `v=28`) and computes their signing hash **without including the chain ID**. This means any pre-EIP-155 transaction signed on Ethereum mainnet or any other EVM chain can be replayed on ZKsync OS if the sender's nonce matches, draining the victim's ZKsync OS funds.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the `is_eip155()` predicate returns `false` when `v == 27` or `v == 28`: [1](#0-0) 

When that branch is taken, the signing hash is computed over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` — the `expected_chain_id` parameter is **completely ignored**: [2](#0-1) 

The caller in `RlpEncodedTxInner::parse_and_compute_signed_hash` then classifies the transaction as `Self::Legacy` (unprotected) and returns the chain-ID-free hash as the authoritative signing hash: [3](#0-2) 

Downstream, both the ZK and Ethereum validation paths perform `ecrecover` against this chain-ID-free hash: [4](#0-3) 

Because the hash contains no chain identifier, a valid signature produced on Ethereum mainnet (or any other EVM chain) is indistinguishable from one produced for ZKsync OS. The `expected_chain_id` that was fetched from `system.get_chain_id()` at parse time: [5](#0-4) 

is silently discarded for the unprotected legacy path.

---

### Impact Explanation

An attacker who observes a pre-EIP-155 legacy transaction on Ethereum mainnet (or any EVM chain) can replay it verbatim on ZKsync OS. The conditions required are:

1. The victim's ZKsync OS nonce equals the nonce in the replayed transaction (always true for fresh accounts at nonce 0, and achievable by waiting or front-running).
2. The victim holds a balance on ZKsync OS at the same address.

The result is an unauthorized fund transfer from the victim's ZKsync OS account to the `to` address encoded in the replayed transaction. This is a direct, unprivileged loss of user funds.

---

### Likelihood Explanation

Pre-EIP-155 transactions (v=27/28) are still produced by some hardware wallets, older software wallets, and certain DeFi tooling. The nonce constraint limits the attack surface but does not eliminate it: fresh accounts (nonce=0) are trivially exploitable, and an attacker can monitor the mempool for matching nonces. The attack requires no privileged access, no oracle manipulation, and no governance majority — only a publicly visible pre-EIP-155 transaction on any EVM chain.

---

### Recommendation

**Short-term**: Reject pre-EIP-155 legacy transactions at the bootloader level by returning `Err(InvalidTransaction::InvalidChainId)` when `legacy_signature.is_eip155() == false`. ZKsync OS is a purpose-built L2 chain and has no obligation to maintain backward compatibility with the pre-EIP-155 signing scheme.

**Long-term**: Document the explicit decision to reject or accept unprotected legacy transactions, and add a regression test that verifies a pre-EIP-155 transaction signed for chain ID 1 is rejected when the bootloader's expected chain ID differs.

---

### Proof of Concept

1. Alice holds 10 ETH on ZKsync OS (chain ID 324) at address `0xAlice`, nonce = 0.
2. Alice (or any user) signs a pre-EIP-155 legacy transfer on Ethereum mainnet: `[nonce=0, gasPrice=X, gasLimit=21000, to=0xEve, value=10ETH, data=0x]`, producing `v=27, r, s`.
3. Eve observes this transaction on Ethereum.
4. Eve submits the identical RLP-encoded bytes to ZKsync OS.
5. `LegacyPayloadParser` detects `v=27`, takes the unprotected branch, and computes `sig_hash = keccak256(rlp([0, X, 21000, 0xEve, 10ETH, 0x]))` — identical to the Ethereum signing hash.
6. `ecrecover(sig_hash, 27, r, s)` recovers `0xAlice`.
7. The bootloader accepts the transaction; Alice's 10 ETH on ZKsync OS is transferred to Eve.
