### Title
Cross-COA Ownership Proof Replay via Missing `evmAddress` Binding in `signedData` - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` does not enforce that the caller-supplied `signedData` encodes the target `evmAddress`. Because the signature is verified only against the Cadence account's keys—not against the specific COA being claimed—a valid signature produced to prove ownership of one COA can be replayed verbatim to pass ownership validation for a *different* COA on the same Cadence account. The code itself documents this gap but leaves it unmitigated at the protocol level.

---

### Finding Description

`validateCOAOwnershipProof` performs two independent checks:

1. **Signature check** – verifies that `signedData` was signed by keys on the Cadence account at `address` with sufficient weight.
2. **COA address check** – borrows the `CadenceOwnedAccount` capability at `path` and compares its EVM address to the caller-supplied `evmAddress`. [1](#0-0) 

The two checks are entirely decoupled: the signature is never bound to `evmAddress`. The code comment at line 1003 explicitly acknowledges this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [2](#0-1) 

The function is `access(all)` and callable by any Cadence script or transaction without restriction. [3](#0-2) 

The EVM-side precompile (`verifyCOAOwnershipProof`) is safe because it always passes `address(this)` (the calling COA's own address) as `evmAddress`: [4](#0-3) 

However, any Cadence-level caller—including third-party contracts, scripts, or off-chain authentication flows—that invokes `validateCOAOwnershipProof` directly inherits the missing binding, with no protocol-level safeguard.

---

### Impact Explanation

An account that owns two or more COAs (e.g., at `/public/coaA` and `/public/coaB`) can have a signature it produced for COA_A replayed by any party to pass `validateCOAOwnershipProof` for COA_B. Any on-chain contract or off-chain relayer that uses this function as an authentication gate—without independently binding `signedData` to `evmAddress`—can be tricked into accepting proof of ownership for the wrong COA. This maps directly to the reported class: a constraint (COA identity derivation) is enforced in one context (the EVM precompile) but absent in the general-purpose Cadence entry point, allowing an incorrect account to be substituted.

---

### Likelihood Explanation

The precondition—a Cadence account owning multiple COAs—is uncommon today ("Flow wallets historically create at most one COA per account"), but the function is public and the gap is structural. Any future protocol, bridge, or wallet that creates multiple COAs per account and relies on `validateCOAOwnershipProof` for authentication is immediately vulnerable without any additional attacker capability beyond possessing a previously observed valid signature (e.g., extracted from a prior on-chain transaction).

---

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` contains the bytes of `evmAddress` before accepting the proof as valid. Concretely, add a pre-check that `signedData` includes the 20-byte `evmAddress` as a required prefix or suffix, or document and enforce a canonical encoding that callers must follow. This mirrors the fix pattern from the referenced audit: add the missing constraint check directly in the function rather than relying on callers to do it correctly.

---

### Proof of Concept

```
// Setup: Alice owns two COAs on the same Cadence account
// COA_A stored at /storage/coaA, published at /public/coaA  → EVM address 0xAAAA...
// COA_B stored at /storage/coaB, published at /public/coaB  → EVM address 0xBBBB...

// Step 1: Alice legitimately signs arbitrary `signedData` (e.g., a login nonce)
//         to prove ownership of COA_A. The signature does NOT encode 0xAAAA...
//         (nothing in the protocol forces it to).

// Step 2: Attacker (or Alice herself) calls the public Cadence function:
EVM.validateCOAOwnershipProof(
    address:     Alice,          // Cadence account — signature check passes ✓
    path:        /public/coaB,   // points to COA_B
    signedData:  <Alice's sig>,  // same bytes signed for COA_A
    keyIndices:  [0],
    signatures:  [<sig>],
    evmAddress:  0xBBBB...       // COA_B's address — address check passes ✓
)
// Returns: ValidationResult { isValid: true, problem: nil }
// COA_B ownership is "proven" using a signature intended for COA_A.
```

The two independent checks both pass because `signedData` is never required to commit to `evmAddress`. The result is a false positive ownership proof for COA_B, exploitable by any caller with access to a prior valid signature from Alice's account. [5](#0-4)
