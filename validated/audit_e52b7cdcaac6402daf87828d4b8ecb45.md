### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Unverified Shares to Coordinator — (File: src/confidential_key_derivation/protocol.rs)

### Summary
In `do_ckd_coordinator`, the coordinator aggregates `CKDOutput` shares received from all participants by simple addition, with no cryptographic verification that each share was honestly computed from the participant's actual private key share. A single malicious participant can inject arbitrary `(big_y, big_c)` group elements, causing the coordinator to output a corrupted confidential derived key that honest parties will accept as legitimate.

### Finding Description
The CKD protocol's coordinator path (`do_ckd_coordinator`, lines 50–55) collects each participant's `(norm_big_y, norm_big_c)` pair and accumulates them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

Each honest participant is supposed to compute, in `compute_signature_share` (lines 160–180):

- `norm_big_y = λ_i · y_i · G`  (blinding term)
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`  (encrypted secret share) [2](#0-1) 

The coordinator has no mechanism to verify that the received pair satisfies this relation. There is no zero-knowledge proof, no commitment-then-reveal, and no pairing check binding `big_c` to the participant's known public verification share. The coordinator simply trusts whatever group elements arrive over the channel.

This is structurally identical to the reported `CredibleAccountModule` bug: just as `_validateSingleCall` returned `true` for any `approve()` call without checking the spender, `do_ckd_coordinator` accepts any `(big_y, big_c)` pair without checking that it was derived from the participant's legitimate key share.

### Impact Explanation
A malicious participant sends fabricated `(big_y*, big_c*)` values. The coordinator accumulates them alongside honest contributions and returns `CKDOutput::new(norm_big_y, norm_big_c)` where both components are corrupted. When the application calls `unmask(app_sk)` it computes `big_c − app_sk · big_y`, which no longer equals `msk · H(pk ‖ app_id)`. The coordinator and all honest parties accept this output as the protocol result with no error signal.

**Impact class matched:** High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.

### Likelihood Explanation
Any single participant in the CKD session can mount this attack. No threshold collusion is required; the CKD aggregation is a simple linear sum over all participants, so one bad actor is sufficient. The attacker needs only to deviate from the protocol by sending arbitrary BLS12-381 G1 elements instead of honestly computed shares — a trivial modification for a library caller who controls their own node.

### Recommendation
Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct construction. Concretely, a Schnorr-style proof of knowledge of `y_i` such that `big_y = y_i · G` and `big_c − big_s = y_i · app_pk` (where `big_s` is verifiable via a pairing against the participant's public verification share from the DKG output) should be generated by the participant and verified by the coordinator before accumulation. Alternatively, a commit-then-reveal scheme can bind each participant to their blinding scalar before the coordinator aggregates.

### Proof of Concept
1. An honest CKD session is initiated among participants `{P_1, P_2, P_m}` with coordinator `P_1`.
2. Malicious participant `P_m` intercepts the protocol at `do_ckd_participant` and instead of calling `compute_signature_share`, sends `(big_y* = G, big_c* = G)` (arbitrary non-zero G1 elements) to the coordinator via `chan.send_private(waitpoint, coordinator, &(big_y*, big_c*))`.
3. The coordinator's loop at lines 50–55 adds `big_y*` and `big_c*` to its running sums without any check.
4. `CKDOutput::new(norm_big_y, norm_big_c)` is returned with corrupted values.
5. The application calls `ckd_output.unmask(app_sk)` and receives a value that is not `msk · H(pk ‖ app_id)`.
6. Any downstream use of the derived confidential key (e.g., decryption, authentication) silently fails or produces wrong results, with no protocol-level error raised.
