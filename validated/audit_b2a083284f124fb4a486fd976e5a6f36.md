### Title
`setPoolAdminFeeDestination` Missing Pre-Change Fee Collection Causes Accrued Admin Fees to Be Misdirected to New Destination - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolAdminFeeDestination` does not collect accrued fees before updating `poolAdminFeeDestination[pool]`, while both `setPoolAdminFees` and `setPoolProtocolFee` — the two analogous fee-config mutators — always collect first. Any fees accrued on the pool between the last collection and the destination change are permanently redirected to the new destination, causing the old destination to lose owed funds.

### Finding Description
Both `setPoolAdminFees` and `setPoolProtocolFee` follow the invariant: **collect all accrued fees at the current rates and to the current destination before mutating any fee-related state**.

`setPoolAdminFees` (lines 417–425): [1](#0-0) 

`setPoolProtocolFee` (lines 327–335): [2](#0-1) 

`setPoolAdminFeeDestination` (lines 438–447) performs no such collection: [3](#0-2) 

The pool stores accrued notional fees in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` until `collectFees` is called. `collectFees` receives `adminFeeDestination_` as a parameter and transfers the admin share directly to it: [4](#0-3) 

`collectPoolFees` always reads the current `poolAdminFeeDestination[pool]` at call time: [5](#0-4) 

Therefore, any fees that accrued while the old destination was active are swept to the new destination on the next `collectPoolFees` call.

### Impact Explanation
The old `adminFeeDestination` permanently loses all admin-share fees that accrued between the last `collectPoolFees` call and the `setPoolAdminFeeDestination` call. The pool admin and the admin fee destination are distinct addresses (`poolAdmin[pool]` vs `poolAdminFeeDestination[pool]`): [6](#0-5) 

The old destination (e.g., a DAO treasury or a separate fee-receiver contract) has no mechanism to recover these funds once the destination mapping is overwritten. The magnitude equals the full admin share of all spread and notional fees accrued since the last collection — directly proportional to swap volume and time elapsed.

### Likelihood Explanation
The pool admin is a semi-trusted role that legitimately calls `setPoolAdminFeeDestination` during treasury rotations, multisig migrations, or operational changes. The admin fee destination is routinely distinct from the pool admin address. Any such rotation while uncollected fees exist — a normal operational state — triggers the loss silently, with no revert or warning.

### Recommendation
Apply the same collect-before-mutate pattern used by `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Collect accrued fees to the OLD destination before switching
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept
1. Pool is deployed; `poolAdminFeeDestination[pool] = treasuryA`.
2. Swaps occur; the pool accumulates `notionalFeeToken0Scaled > 0`.
3. Pool admin calls `setPoolAdminFeeDestination(pool, treasuryB)`. No collection happens; `poolAdminFeeDestination[pool]` is now `treasuryB`.
4. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool] == treasuryB` and passes it to `collectFees`, which transfers the entire admin share — including fees accrued while `treasuryA` was the destination — to `treasuryB`.
5. `treasuryA` receives nothing. The loss equals the admin share of all fees accrued in step 2.
