### Title
Missing L2 Sequencer Uptime Check in L2 Price Providers Allows Stale-Price Swap Execution - (File: `smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are the L2-specific price providers deployed on Base and HyperEVM. Both rely solely on a time-delta staleness check (`_isStale`) to validate oracle data freshness. Neither contract queries a sequencer uptime feed before accepting and forwarding a price to the pool's swap engine. When the L2 sequencer goes down and restarts, a window exists where the last pre-downtime oracle price passes the staleness check but is economically stale, enabling bad-price swap execution against LPs.

---

### Finding Description

The protocol targets Base and HyperEVM (both L2s with sequencers). The L2 price providers use a push-based oracle model: off-chain keepers call `ChainlinkOracle.updateReport()` or `PythOracle` equivalents to store price data on-chain. The stored `refTime` is the oracle observation timestamp (from the DON/Pyth network), not `block.timestamp`.

Both L2 providers implement `_isStale` as follows:

```solidity
// PriceProviderL2.sol lines 135-150
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
```

And `_getBidAndAskPrice` calls it directly:

```solidity
// PriceProviderL2.sol lines 208-217
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spread, , uint256 refTime) =
        IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);
    if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
        return (0, type(uint128).max);
    }
    ...
}
```

`ProtectedPriceProviderL2._computeBidAsk` is identical in structure.

**Sequencer downtime scenario:**

1. The sequencer goes down at time `T`. The last oracle report stored has `refTime = T`.
2. Keepers cannot push new reports while the sequencer is down.
3. The sequencer restarts at time `T + D` (downtime duration `D`).
4. `block.timestamp` is now `T + D`. The stored price has `refTime = T`.
5. `_isStale` evaluates: `(T + D) - T = D`. If `D < MAX_TIME_DELTA`, the price **passes** the staleness check.
6. Before any keeper pushes a fresh report, any trader can call `swap()` on a pool using this stale price.

`MAX_TIME_DELTA` is bounded only by `0 < MAX_TIME_DELTA <= 7 days` at construction. Even a conservative 1-hour setting creates a 1-hour exploitation window after a 59-minute outage.

There is **no sequencer uptime feed check** anywhere in the production Solidity source:

```
grep "sequencerUptimeFeed" **/*.sol → No matches found
```

The `FUTURE_TOLERANCE` parameter (up to 1 hour) is only for clock skew (oracle `refTime` slightly ahead of `block.timestamp`) and does not address sequencer downtime at all.

---

### Impact Explanation

During the post-restart window, the pool's `getBidAndAskPrice()` returns a stale bid/ask derived from pre-downtime market conditions. If the real market price moved significantly during the outage (which is common — sequencer outages often coincide with high-volatility events), traders (or MEV bots monitoring the sequencer restart) can execute swaps at the stale price, extracting value directly from LP positions. This is a **bad-price execution** impact: stale bid/ask reaches a pool swap, causing LP principal loss.

---

### Likelihood Explanation

Base has experienced sequencer downtime historically. HyperEVM is a newer chain. The trigger is unprivileged — any trader can call `swap()` during the window. MEV bots routinely monitor sequencer restarts and are positioned to exploit the first available block. The window duration equals `min(MAX_TIME_DELTA, time-to-first-fresh-report)`, which can be minutes to hours.

---

### Recommendation

In `PriceProviderL2._getBidAndAskPrice()` and `ProtectedPriceProviderL2._getBidAndAskPrice()`, add a sequencer uptime check before using oracle data, following the Chainlink L2 Sequencer Uptime Feed pattern:

```solidity
// Example check to add at the top of _getBidAndAskPrice()
if (address(sequencerUptimeFeed) != address(0)) {
    (, int256 answer, uint256 startedAt, , ) = sequencerUptimeFeed.latestRoundData();
    // answer == 0 means sequencer is up; 1 means down
    if (answer != 0) revert SequencerDown();
    // Enforce grace period after restart (e.g., 1 hour)
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotElapsed();
}
```

Store `sequencerUptimeFeed` as an immutable set at construction in `PriceProviderFactoryL2.createPriceProvider()`. Set `GRACE_PERIOD` to at least 3600 seconds (1 hour) per Chainlink's recommendation.

### Proof of Concept

1. Deploy `PriceProviderL2` on Base with `MAX_TIME_DELTA = 3600` (1 hour).
2. At `T=0`, keeper pushes oracle report with `refTime = 0`. Price stored: `mid = 2000e8` (ETH = $2000).
3. At `T=1800`, Base sequencer goes down. Real ETH price drops to $1800 during outage.
4. At `T=3500` (58 minutes later), sequencer restarts. `block.timestamp = 3500`.
5. `_isStale(1800, 3500, 3600, futureTol)` → `3500 - 1800 = 1700 < 3600` → **not stale**.
6. Attacker calls `swap()` on the ETH/USDC pool, buying ETH at the stale $2000 ask price.
7. Pool delivers ETH at $2000 ask; real price is $1800. Attacker profits ~$200/ETH at LP expense.
