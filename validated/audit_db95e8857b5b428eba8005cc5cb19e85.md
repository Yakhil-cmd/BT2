### Title
Deployment Authorization Check Bypassed for Existing Contract Updates - (File: `fvm/environment/contract_updater.go`)

### Summary

The `SetContract` function in `ContractUpdaterImpl` enforces the deployment authorization check only for **new** contract deployments. The code explicitly exempts updates to existing contracts from any authorization check ("Contract updates are always allowed"). This mirrors the GovernorAlpha veto-bypass pattern: just as a malicious proposer could append an innocent self-call to make a proposal un-vetoable, a contract owner can exploit the "existing contract" path to push arbitrary new code past the governance-controlled deployment restriction.

### Finding Description

In `fvm/environment/contract_updater.go`, `SetContract` performs the authorization check only when the contract does not yet exist:

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
...
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: %w", ...)
}
``` [1](#0-0) 

When `exists == true`, the function falls through unconditionally and queues the update in `draftUpdates` with no authorization check whatsoever. [2](#0-1) 

The authorization list is read from the service account's on-chain storage path `authorizedAddressesToDeployContracts`: [3](#0-2) 

Governance manages this list via `SetContractDeploymentAuthorizersTransaction`. The default `ContractUpdaterParams` has `RestrictContractDeployment: true`, meaning restricted deployment is the intended production posture: [4](#0-3) 

The `isAuthorizedForDeployment` helper is only reached when `!exists`: [5](#0-4) 

By contrast, `RemoveContract` **always** checks authorization regardless of state: [6](#0-5) 

This asymmetry is the root cause. The removal path is correctly guarded; the update path is not.

### Impact Explanation

Any account that owns a contract (i.e., holds `auth(UpdateContract)` on the account where the contract lives) can replace that contract's code with arbitrary new Cadence code, even after the account has been removed from the governance-controlled authorized-deployers list. Because the Cadence runtime only requires the account owner's signature to call `signer.contracts.update(...)`, no additional privilege is needed beyond owning the account. The attacker can introduce malicious logicâ€”e.g., draining capabilities, minting tokens, or altering resource definitionsâ€”into a previously-trusted contract, bypassing the governance mechanism that was specifically designed to prevent unauthorized code changes.

### Likelihood Explanation

The Flow mainnet has been live for years; many accounts hold deployed contracts. Any such account whose owner was removed from the authorized-deployers list (or whose contract was deployed before restrictions were enabled) can immediately exploit this path. The attacker-controlled entry point is a standard user transaction signed only by the contract-owning accountâ€”no privileged keys, no staked node role, no social engineering required.

### Recommendation

Apply the same `isAuthorizedForDeployment` check to updates as to new deployments, or introduce a separate `isAuthorizedForUpdate` check backed by a distinct on-chain list. The current comment "Contract updates are always allowed" should be treated as a known governance gap and explicitly addressed. At minimum, the governance documentation should warn that removing an account from the authorized-deployers list does **not** prevent that account from updating its existing contracts.

### Proof of Concept

1. Restricted deployment is enabled on-chain (`isContractDeploymentRestricted = true`).
2. Account `A` is in `authorizedAddressesToDeployContracts`; it deploys contract `Foo`.
3. Governance removes `A` from the authorized list via `SetContractDeploymentAuthorizersTransaction`.
4. `A` submits a transaction:
   ```cadence
   transaction {
     prepare(signer: auth(UpdateContract) &Account) {
       signer.contracts.update(name: "Foo", code: "<malicious code>".decodeHex())
     }
   }
   ```
5. The FVM calls `UpdateAccountContractCode` â†’ `SetContract`. `ContractExists("Foo", A)` returns `true`, so the `!exists` branch is skipped entirely.
6. `draftUpdates` is populated with the malicious code; `Commit()` writes it to state.
7. The contract is now updated to malicious code with no authorization error, despite `A` not being in the authorized list. [7](#0-6) [8](#0-7)

func (updater *ContractUpdaterImpl) UpdateAccountContractCode(
	location common.AddressLocation,
	code []byte,
) error {
	defer updater.tracer.StartChildSpan(
		trace.FVMEnvUpdateAccountContractCode).End()

	err := updater.meter.MeterComputation(
		common.ComputationUsage{
			Kind:      ComputationKindUpdateAccountContractCode,
			Intensity: 1,
		},
	)
	if err != nil {
		return fmt.Errorf("update account contract code failed: %w", err)
	}

	err = updater.SetContract(
		location,
		code,
		updater.signingAccounts)
	if err != nil {
		return fmt.Errorf("updating account contract code failed: %w", err)
	}

	return nil
}

func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,
) error {
	// Initial contract deployments must be authorized by signing accounts.
	//
	// Contract updates are always allowed.
	exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
	if err != nil {
		return err
	}

	if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
		return fmt.Errorf(
			"deploying contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"SetContract",
				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
}
```

func (updater *ContractUpdaterImpl) RemoveContract(
	location common.AddressLocation,
	signingAccounts []flow.Address,
) (err error) {
	// check if authorized
	if !updater.isAuthorizedForRemoval(signingAccounts) {
		return fmt.Errorf("removing contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"RemoveContract",
				"removing contracts requires authorization from specific "+
					"accounts"))
	}

	u := ContractUpdate{Location: location}
	updater.draftUpdates[location] = u

	return nil
}
```

func (updater *ContractUpdaterImpl) isAuthorizedForDeployment(
	signingAccounts []flow.Address,
) bool {
	if updater.RestrictedDeploymentEnabled() {
		return updater.isAuthorized(
			signingAccounts,
			blueprints.ContractDeploymentAuthorizedAddressesPath)
	}
	return true
}
