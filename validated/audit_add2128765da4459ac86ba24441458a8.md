### Title
Unbounded `MessageBuffer` Growth via Arbitrary Waitpoint Injection - (File: `src/protocol/internal.rs`)

### Summary
The `MessageBuffer` inside every protocol instance is an unbounded `HashMap<MessageHeader, SubMessageQueue>`. Because `Protocol::message()` accepts any message with a syntactically valid header and performs no validation of the `waitpoint` field or the `from` participant, a malicious participant can inject messages keyed to arbitrary, never-consumed waitpoints, growing the buffer without limit until the process runs out of memory.

### Finding Description
Every protocol instance owns a `Comms` struct whose `incoming` field is a `MessageBuffer`:

```rust
struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}
``` [1](#0-0) 

A `MessageHeader` is a `(ChannelTag, Waitpoint)` pair where `Waitpoint = u64`, giving 2⁶⁴ distinct keys. [2](#0-1) 

The public entry point `Protocol::message(from, data)` calls `Comms::push_message`, which only checks that the byte slice is long enough and that the header parses — it performs **no validation** of whether `from` is a legitimate participant or whether the `waitpoint` is within any expected range:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);
}
``` [3](#0-2) 

`MessageBuffer::push` then unconditionally inserts a new `HashMap` entry for every unseen `(ChannelTag, Waitpoint)` pair:

```rust
fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
    let mut messages_lock = self.messages.lock().expect("lock should not fail");
    messages_lock.entry(header).or_default().send(from, message);
}
``` [4](#0-3) 

Each `SubMessageQueue` is itself backed by `futures::channel::mpsc::unbounded()`, so both the number of distinct keys in the `HashMap` and the number of messages per key are unbounded: [5](#0-4) 

The codebase's own test suite explicitly demonstrates this attack path and confirms the messages accumulate:

```rust
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;
    for i in 0..attack_count {
        let header = MessageHeader::new(ChannelTag::root_shared())
            .with_waitpoint(1_000_000 + i);
        // ...
        comms.push_message(attacker, message);
    }
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [6](#0-5) 

### Impact Explanation
A malicious participant (or any entity that can call `Protocol::message()`) can flood an honest participant's `MessageBuffer` with messages carrying arbitrary, never-polled waitpoints. Because the honest protocol logic only ever polls a small, fixed set of waitpoints per round, the injected entries are never drained. Memory grows proportionally to the number of injected messages. Sustained injection causes OOM, crashing the process and permanently denying that participant the ability to complete DKG, reshare, presign, or signing protocols.

### Likelihood Explanation
The `Protocol::message()` method is the documented, public API surface — every integrator is expected to call it to deliver network messages. A single malicious participant in any multi-party session can exploit this without any special privilege. The attacker only needs to send messages with a valid 40-byte header prefix (32-byte channel tag + 8-byte waitpoint) and an arbitrary high waitpoint value. No cryptographic material is required.

### Recommendation
1. **Cap the `HashMap` size**: Reject `push_message` calls once the number of distinct `MessageHeader` keys exceeds a protocol-specific bound (e.g., `n_participants × max_rounds × small_constant`).
2. **Cap per-key queue depth**: Replace `mpsc::unbounded` with `mpsc::channel(bound)` and drop messages that exceed the per-waitpoint capacity.
3. **Validate `from`**: In `push_message`, check that `from` belongs to the known participant list before buffering.
4. **Validate `waitpoint` range**: Reject messages whose waitpoint exceeds the maximum waitpoint the current protocol phase will ever poll.

### Proof of Concept
Construct a minimal harness that calls `Protocol::message()` in a tight loop with messages whose waitpoint field increments on each call:

```rust
let comms = Comms::new();
let attacker = Participant::from(0xdeadu32);
// Root shared channel tag (deterministic, publicly known)
let root_tag = ChannelTag::root_shared();
for i in 0u64.. {
    let header = MessageHeader::new(root_tag).with_waitpoint(1_000_000 + i);
    let mut msg = header.to_bytes().to_vec();
    msg.extend_from_slice(&[0u8; 8]); // arbitrary payload
    comms.push_message(attacker, msg);
    // HashMap grows by one entry per iteration; memory grows without bound.
}
```

Each iteration inserts one new `HashMap` entry and one `SubMessageQueue` (backed by an unbounded channel). The honest protocol never polls waitpoints above its own counter, so nothing is ever drained. The process will exhaust available memory and crash, permanently denying the honest participant the ability to complete any in-progress threshold protocol session.
