### Title
Malicious CKD Participant Can Corrupt the Coordinator's Output Without Detection — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator blindly sums participant-supplied `(big_y, big_c)` curve-point pairs with no proof of correctness. A single malicious participant can substitute arbitrary group elements, causing the coordinator to assemble a `CKDOutput` that decrypts to a wrong confidential key. No mechanism exists for the coordinator or the application to detect the substitution before accepting the result.

### Finding Description

The analog vulnerability class is **missing output validation**: just as `burn()` accepts whatever price the chain delivers with no `minOutputAmount` guard, `do_ckd_coordinator` accepts whatever curve points each participant delivers with no correctness guard.

In `compute_signature_share` every honest participant computes:

```
big_y  = y * G                          (random blinding point)
big_c  = x_i * H(pk || app_id) + y * app_pk   (masked share)
```

and then normalises both by its Lagrange coefficient before sending them to the coordinator. [1](#0-0) 

The coordinator receives these pairs and simply accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [2](#0-1) 

There is **no zero-knowledge proof**, **no Pedersen commitment**, and **no consistency check** that the received `(big_y, big_c)` was computed from the participant's actual signing share and the agreed `app_id`/`app_pk`. The `CKDOutput` struct itself carries only the two raw points: [3](#0-2) 

Contrast this with the DKG protocol, which enforces proofs of knowledge, commitment hashes, and per-share Feldman verification before accepting any contribution: [4](#0-3) 

The CKD protocol has none of these safeguards.

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

When the application calls `ckd_output.unmask(app_sk)` it computes `big_c − app_sk * big_y`. If a malicious participant injected arbitrary `(big_y', big_c')`, the result is:

```
(Σ honest big_c + big_c') − app_sk * (Σ honest big_y + big_y')
≠  msk * H(pk || app_id)
```

The coordinator returns a silently wrong confidential key. The application has no way to distinguish this from a correct derivation because the protocol provides no authenticating tag on the output. [5](#0-4) 

### Likelihood Explanation

**Medium.** Any single participant in the CKD session is a sufficient attacker — no collusion is required. The participant is already a trusted protocol member (the `ckd()` entry point only checks that the participant is in the list), so the attacker-controlled entry path is the normal `ckd()` call with a malicious implementation of the participant role. [6](#0-5) 

### Recommendation

Attach a zero-knowledge proof of correct share formation to each participant's message. Concretely, each participant should prove in zero knowledge that `big_c − y * app_pk` lies on the line `x_i * H(pk || app_id)` for the committed `x_i` (i.e., a Schnorr-style DLEQ proof relating `big_y = y*G`, `app_pk = a*G`, and `big_c = x_i*H + y*app_pk`). The coordinator must verify all proofs before summing:

```diff
  for (_, participant_output) in
      recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
  {
+     verify_ckd_share_proof(&participant_output, &key_pair.public_key, app_id, app_pk)?;
      norm_big_y += participant_output.big_y();
      norm_big_c += participant_output.big_c();
  }
```

### Proof of Concept

1. Run a 3-of-3 CKD session with participants `[A, B, C]` where `C` is malicious.
2. `C` replaces its honest `(norm_big_y_C, norm_big_c_C)` with arbitrary points `(P, Q)` before sending to the coordinator.
3. The coordinator sums `norm_big_y_A + norm_big_y_B + P` and `norm_big_c_A + norm_big_c_B + Q` with no error.
4. `ckd_output.unmask(app_sk)` returns a value that is not `msk * H(pk || app_id)`.
5. Any downstream use of the derived key (e.g., BLS signing, key agreement) silently fails or produces an attacker-chosen key. [7](#0-6)

pub struct CKDOutput {
    big_y: ElementG1,
    big_c: ElementG1,
}

impl CKDOutput {
    pub fn new(big_y: ElementG1, big_c: ElementG1) -> Self {
        Self { big_y, big_c }
    }

    /// Outputs `big_y`
    pub fn big_y(&self) -> ElementG1 {
        self.big_y
    }

    /// Outputs `big_c`
    pub fn big_c(&self) -> ElementG1 {
        self.big_c
    }

    /// Takes a secret scalar and returns
    /// s <- C − a ⋅ Y = msk ⋅ H ( `app_id` )
    pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
        self.big_c - self.big_y * secret_scalar
    }
}
