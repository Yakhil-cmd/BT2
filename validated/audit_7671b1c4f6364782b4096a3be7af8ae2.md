### Title
Unauthorized Invoke Transaction Accepted Without Signature Verification via `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point — the sole on-chain signature check — for any invoke transaction with nonce=1 when the target account has on-chain nonce=0 and **any** transaction in the mempool. An unprivileged attacker who observes a victim's pending `deploy_account` transaction can submit a forged invoke transaction with `sender_address=victim, nonce=1`, have it admitted to the mempool without signature verification, and have it executed from the victim's account once the deploy_account is sequenced.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` decides whether to skip the blockifier's `__validate__` entry point:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}

Inside the blockifier, `validate_tx` immediately returns `None` (no-op) when `validate=false`:

`__validate__` is the **only** place where the account contract verifies the ECDSA signature against its stored public key. Skipping it means the signature field is never cryptographically checked.

The guard condition relies on `account_tx_in_pool_or_recent_block`, which returns `true` if the account has **any** transaction in the pool or any committed block entry — not specifically a `deploy_account`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

The stateless validator only checks signature **length**, not cryptographic validity:

The mempool's `validate_tx` only checks nonce ordering and fee escalation — no signature check:

```rust

    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;

**Exact attack path:**

1. Victim broadcasts a `deploy_account` transaction (nonce=0). It enters the mempool. On-chain nonce for victim's address remains 0.
2. Attacker submits an `InvokeV3` transaction with `sender_address=victim_address`, `nonce=1`, arbitrary `calldata`, and a garbage/empty signature.
3. Gateway stateless validation passes (signature length ≤ max, resource bounds non-zero, etc.).
4. Gateway stateful validation:
   - `validate_nonce`: nonce=1, account_nonce=0, within `max_allowed_nonce_gap` → **passes**.
   - `validate_by_mempool`: nonce=1 ≥ account_nonce=0 → **passes**.
   - `skip_stateful_validations`: nonce==1 ✓, account_nonce==0 ✓, `account_tx_in_pool_or_recent_block(victim)` returns `true` (victim's deploy_account is in pool) → returns `true`.
   - `run_validate_entry_point` called with `validate=false` → `__validate__` **never called**.
5. Attacker's forged invoke is added to the mempool.
6. Batcher sequences: deploy_account (nonce=0) executes first, deploying victim's account. Then attacker's invoke (nonce=1) executes from victim's account with attacker-controlled calldata.

### Impact Explanation

The attacker executes arbitrary calldata from the victim's account without possessing the victim's private key. This can drain the victim's token balances, approve arbitrary spenders, or call any contract on behalf of the victim. The victim's account is compromised at the moment of its first deployment. This is a **Critical** impact: an unauthorized Starknet transaction is accepted through account validation/signature logic.

### Likelihood Explanation

- `deploy_account` transactions are publicly visible in the mempool.
- The attack requires no privileged access — any RPC caller can submit the forged invoke.
- The window of vulnerability is the time between the victim's `deploy_account` entering the mempool and being committed to a block.
- The attack is deterministic and requires no brute-force.

### Recommendation

1. **Specifically verify a `deploy_account` exists**: `account_tx_in_pool_or_recent_block` checks for any transaction from the address. The skip logic should instead query the mempool for a pending `deploy_account` transaction specifically for the target address.
2. **Verify the invoke was submitted alongside the deploy_account**: Require that both transactions arrive in the same gateway request or carry a cryptographic linkage.
3. **Alternatively, remove the skip entirely**: Require the account to be deployed before accepting invoke transactions, accepting the UX trade-off.
4. **At minimum, add a type-specific check**: Before skipping `__validate__`, confirm the mempool contains a `DeployAccount` transaction for `sender_address`, not just any transaction.

### Proof of Concept

```
1. Victim submits:
   deploy_account_tx {
     sender_address: 0xVICTIM,
     nonce: 0,
     signature: [valid_r, valid_s],
     ...
   }
   → Accepted into mempool. On-chain nonce(0xVICTIM) = 0.

2. Attacker submits:
   invoke_tx {
     sender_address: 0xVICTIM,   // attacker sets victim's address
     nonce: 1,
     calldata: [transfer_all_tokens_to_attacker],
     signature: [0x0, 0x0],      // garbage signature
     ...
   }

3. Gateway checks:
   - stateless: signature length 2 ≤ max_signature_length → OK
   - stateful nonce: 1 ≥ 0 → OK
   - validate_by_mempool: nonce 1 ≥ account_nonce 0 → OK
   - skip_stateful_validations:
       nonce==1 ✓, account_nonce==0 ✓,
       account_tx_in_pool_or_recent_block(0xVICTIM)==true ✓
       → returns true (skip __validate__)
   - run_validate_entry_point(validate=false) → __validate__ NOT called

4. Attacker's invoke added to mempool with nonce=1.

5. Batcher sequences block:
   - deploy_account executes → 0xVICTIM deployed, nonce becomes 1
   - attacker's invoke executes → calldata runs from 0xVICTIM's account
   → victim's tokens transferred to attacker
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
