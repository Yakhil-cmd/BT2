### Title
Missing Identity-Element Check on `app_pk` Allows Direct Extraction of Confidential Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` function in the Confidential Key Derivation protocol accepts a caller-supplied `app_pk` (the application's ElGamal public key) without validating that it is not the group identity element. When `app_pk = G1::identity()`, the ElGamal blinding term vanishes, causing the coordinator's aggregated output `big_c` to equal `msk · H(pk, app_id)` — the confidential derived key — in plaintext. An attacker who controls the `app_pk` input can read the BLS signature directly from the protocol output without possessing `app_sk`.

### Finding Description

The `ckd()` entry point validates participant counts, duplicates, and membership, but performs no check on `app_pk`: [1](#0-0) 

The `app_pk` value flows directly into `compute_signature_share()`: [2](#0-1) 

The critical line is:

```rust
let big_c = big_s + app_pk * y.0;   // line 174
```

When `app_pk = G1::identity()`, the term `app_pk * y.0` evaluates to the group identity, so `big_c = big_s = x_i · H(pk, app_id)`. Each participant sends their `(lambda_i · Y_i, lambda_i · big_s_i)` to the coordinator.

The coordinator aggregates without any identity check: [3](#0-2) 

The resulting `C = Σ lambda_i · x_i · H(pk, app_id) = msk · H(pk, app_id)`, which is the BLS signature — the confidential derived key itself — stored directly in `CKDOutput.big_c`.

The `CKDOutput` type exposes `big_c` via a public accessor: [4](#0-3) 

The attacker reads `ckd_output.big_c()` directly; no `app_sk` is needed to unmask anything.

The `AppId` type only enforces a maximum length, not a minimum, so an empty `app_id` is also accepted, but the primary root cause is the missing identity check on `app_pk`: [5](#0-4) 

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

The output `msk · H(pk, app_id)` is the exact secret the CKD protocol is designed to keep confidential. The protocol specification states that "the secret itself is never revealed to any entity other than the application itself, not even to individual MPC nodes." By supplying `app_pk = G1::identity()`, an attacker obtains `msk · H(pk, app_id)` for any `app_id` of their choice, without needing the corresponding `app_sk`. This breaks the core confidentiality guarantee of the CKD scheme.

### Likelihood Explanation

The `app_pk` is a caller-supplied parameter with no validation. Any entity that can invoke `ckd()` — including a malicious TEE app or a compromised coordinator — can supply the identity point. The identity element of `blstrs::G1Projective` is a well-known, trivially constructable value. No cryptographic capability is required.

### Recommendation

Add an explicit check in `ckd()` (or at the top of `compute_signature_share()`) that rejects `app_pk` equal to the group identity:

```rust
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern already used in the robust ECDSA presign and dlogeq proof code: [6](#0-5) [7](#0-6) 

### Proof of Concept

1. Attacker constructs `app_pk = blstrs::G1Projective::identity()`.
2. Attacker calls `ckd(participants, coordinator, me, key_pair, app_id, app_pk, rng)` — this succeeds because no identity check exists.
3. Inside `compute_signature_share`, each node computes `big_c = big_s + identity * y = big_s = x_i · H(pk, app_id)`.
4. Coordinator aggregates: `C = Σ lambda_i · x_i · H(pk, app_id) = msk · H(pk, app_id)`.
5. Attacker reads `ckd_output.big_c()` from the returned `CKDOutput` — this is `msk · H(pk, app_id)`, the confidential derived key, in plaintext.
6. The attacker now possesses the BLS signature for any `app_id` they chose, without ever holding `app_sk`.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
