### Title
Gateway validates transaction L2 gas price against stale `l2_gas_price` instead of `next_l2_gas_price` â€” (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `validate_resource_bounds` checks a transaction's `max_l2_gas_price` against the previous block's `l2_gas_price` (the price used in the last committed block). However, the transaction will execute in the **next** block, which uses `next_l2_gas_price` â€” the EIP-1559 adjusted price computed from the current block's gas usage. This is the direct Sequencer analog of the Ajna "wrong reference point" bug: just as the first borrower's interest accrued from the last inflator update time rather than from when debt was created, here the gateway validates against the wrong price epoch, causing it to admit transactions that will fail at execution (rising price) and reject valid transactions (falling price).

---

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds`, the code reads `strk_gas_prices.l2_gas_price` from the latest committed block and uses it as the admission threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The TODO comment is the developer's own acknowledgment of the bug. The `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs `BlockInfo` using `block_header.l2_gas_price.price_in_fri` â€” the price of the **already-committed** block â€” and never exposes `block_header.next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

The `StorageBlockHeader` stores both fields separately:

```rust
pub l2_gas_price: GasPricePerToken,      // price used in this block
pub next_l2_gas_price: GasPrice,          // EIP-1559 price for the NEXT block
``` [3](#0-2) 

The batcher's `build_proposal` sets the new block's gas price to `args.l2_gas_price`, which is `self.l2_gas_price` from the consensus context â€” the value produced by `update_l2_gas_price` â†’ `calculate_next_l2_gas_price_for_fin` â†’ `calculate_next_base_gas_price`, i.e., the EIP-1559 adjusted `next_l2_gas_price`:

```rust
let init = ProposalInit {
    ...
    l2_gas_price_fri: args.l2_gas_price,
    ...
};
``` [4](#0-3) 

The same wrong price is also used in `run_validate_entry_point` when building the `BlockContext` for blockifier validation at the gateway:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [5](#0-4) 

Both the explicit threshold check and the blockifier's `check_fee_bounds` inside the gateway use `l2_gas_price`, so they agree with each other â€” but both disagree with what the batcher will use. The blockifier's `check_fee_bounds` in the batcher will compare `max_price_per_unit` against the actual block gas price (`next_l2_gas_price`):

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [6](#0-5)

### Impact Explanation

**Case 1 â€” price rising** (`next_l2_gas_price > l2_gas_price`): A transaction with `max_l2_gas_price` in the range `[l2_gas_price, next_l2_gas_price)` passes the gateway's threshold check (validated against the lower `l2_gas_price`) and is admitted to the mempool. When the batcher executes it, `check_fee_bounds` fails because the block's actual price is `next_l2_gas_price`. The transaction is included as a reverted transaction; the user loses the fee. **Gateway admits invalid transactions.**

**Case 2 â€” price falling** (`next_l2_gas_price < l2_gas_price`): A transaction with `max_l2_gas_price` in the range `[next_l2_gas_price, min_gas_price_percentage% Ă— l2_gas_price)` is rejected by the gateway (threshold based on the higher `l2_gas_price`) even though it would succeed at execution. **Gateway rejects valid transactions.**

Both cases match the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

EIP-1559 adjusts the gas price every block. The maximum change per block is `price / gas_price_max_change_denominator`. With a full block, the price rises by up to ~12.5% (denominator = 8). The discrepancy between `l2_gas_price` and `next_l2_gas_price` is non-zero for every block that is not exactly at the gas target â€” which is the common case. Any user submitting a transaction with `max_l2_gas_price` near the current price will be affected. No special privileges are required; the trigger is ordinary transaction submission.

---

### Recommendation

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` should expose `next_l2_gas_price` from the block header and `validate_resource_bounds` should use it as the threshold. Concretely:

1. Add `next_l2_gas_price` to the `BlockInfo` returned by `get_block_info`, or expose it as a separate method on `GatewayFixedBlockStateReader`.
2. In `validate_resource_bounds`, replace `previous_block_l2_gas_price` (which reads `l2_gas_price`) with the block's `next_l2_gas_price`.
3. In `run_validate_entry_point`, replace `block_info.gas_prices.strk_gas_prices.l2_gas_price` with `next_l2_gas_price` when constructing the `BlockContext`, so the gateway's blockifier validation matches what the batcher will enforce.

---

### Proof of Concept

1. Block N is committed with `l2_gas_price = 100 fri` and `next_l2_gas_price = 112 fri` (12% EIP-1559 increase due to a full block).
2. User submits an `InvokeV3` transaction with `AllResources` bounds: `l2_gas.max_price_per_unit = 105 fri`.
3. Gateway `validate_resource_bounds`: threshold = `100% Ă— 100 = 100`; `105 â‰Ą 100` â†’ **PASS**. Transaction admitted to mempool.
4. Gateway `run_validate_entry_point` builds `BlockContext` with `l2_gas_price = 100`; blockifier `check_fee_bounds`: `105 â‰Ą 100` â†’ **PASS**. Validation succeeds.
5. Batcher builds block N+1 with `l2_gas_price_fri = 112` (from `next_l2_gas_price`).
6. Blockifier `check_fee_bounds` in batcher: `105 < 112` â†’ `MaxGasPriceTooLow` â†’ transaction reverts.
7. User's transaction is included in block N+1 as a reverted transaction; fee is charged for the failed execution.

The root cause is identical in structure to the Ajna M-4 bug: a reference value (`l2_gas_price`) is read at the start of the validation pipeline and used throughout, but the actual execution uses a different value (`next_l2_gas_price`) that was already computed and stored in the block header. The gateway never reads the correct reference point.
