### Title
Attacker-Controlled `sender_address` Bypasses Signature Validation via `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator allows an attacker to submit an invoke transaction with an arbitrary victim `sender_address` and a completely invalid (or empty) signature, and have it accepted into the mempool without any signature (`__validate__`) check. The only guard is a mempool presence check on the victim's address — which any observer can trigger by watching for a victim's `deploy_account` transaction in the mempool.

---

### Finding Description

The external bug's invariant is: **authorization must be tied to the actual caller (`msg.sender`), not to an attacker-supplied address field**. In the Sequencer, the exact analog is: **signature validation must be tied to the actual submitter, not to the attacker-supplied `sender_address` field in the transaction**.

The gateway stateful path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ← victim's address
                .await
                // ...
        }
    }
    Ok(false)
}
```

When this returns `true`, `run_validate_entry_point` sets `validate: false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

This means `StatefulValidator::perform_validations` is called with `validate = false`, so the account's `__validate__` entry point — the only mechanism that checks the transaction signature — is **never called** at the gateway level.

The `account_tx_in_pool_or_recent_block` check queries the mempool for **any** transaction from the victim's address:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This check is satisfied the moment a victim submits a `deploy_account` transaction (nonce=0) to the mempool — a publicly observable event. The check does **not** verify that the submitter of the invoke transaction is the same entity that submitted the `deploy_account` transaction.

**Attack path:**

1. Victim broadcasts a `deploy_account` transaction (nonce=0) to the gateway.
2. Attacker observes the victim's `sender_address` (the calculated contract address) in the mempool.
3. Attacker constructs an invoke transaction with:
   - `sender_address = victim_address`
   - `nonce = 1`
   - Arbitrary `calldata` (e.g., a token transfer from the victim's account)
   - An empty or invalid `signature`
4. The gateway's `skip_stateful_validations` returns `true` because:
   - The tx is Invoke with `nonce == 1`
   - `account_nonce == 0` (victim not yet deployed)
   - `account_tx_in_pool_or_recent_block(victim_address) == true`
5. `run_validate_entry_point` is called with `validate = false` — no `__validate__` call.
6. The attacker's transaction is accepted into the mempool.

---

### Impact Explanation

This matches the allowed High impact scope: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The gateway accepts an invoke transaction with an invalid signature into the mempool. Concretely:

- The attacker's transaction occupies the nonce=1 slot for the victim's address in the mempool. If the mempool enforces one transaction per (address, nonce) pair, the victim's legitimate nonce=1 invoke transaction is displaced or rejected with `DuplicateNonce`.
- The victim's `deploy_account` succeeds, but their intended first action (nonce=1 tx) is replaced by the attacker's transaction, which will fail at `__validate__` during block execution.
- For time-sensitive operations (e.g., a swap with a deadline, a liquidation), this displacement causes direct financial loss to the victim.
- The attacker pays no fees for the failing transaction (in Starknet, a transaction rejected at `__validate__` is not charged).

---

### Likelihood Explanation

- **Precondition**: The victim must have a `deploy_account` transaction in the mempool. This is a common, publicly observable event for any new account.
- **Attacker cost**: Zero — the attacker only needs to submit a transaction with a valid-looking structure (non-zero resource bounds pass stateless checks) and an arbitrary signature.
- **Detection**: The attack is silent at the gateway level; the failure only surfaces at block execution time.
- **Repeatability**: The attacker can repeat this for every new account that submits a `deploy_account` transaction.

---

### Recommendation

The `skip_stateful_validations` path must not skip signature validation entirely. Options:

1. **Remove the skip entirely**: Require the account to be deployed before accepting nonce=1 invoke transactions. Handle the UX concern at the batcher level by allowing deploy_account + invoke to be sequenced in the same block.
2. **Restrict the skip to transactions submitted in the same RPC call**: Only skip validation when the deploy_account and invoke are submitted together as a batch from the same connection, so the gateway can verify they share the same signature key.
3. **Add a lightweight signature pre-check**: Before skipping `__validate__`, verify that the transaction's signature is structurally consistent with the account's class (e.g., correct length for a standard ECDSA account), even if the full `__validate__` is deferred.

---

### Proof of Concept

```
1. Victim calls gateway: POST /gateway/add_transaction
   {
     "type": "DEPLOY_ACCOUNT",
     "class_hash": "0xABC...",
     "contract_address_salt": "0x1",
     "constructor_calldata": ["0xVICTIM_PUB_KEY"],
     "nonce": "0x0",
     "signature": ["0xVALID_SIG_R", "0xVALID_SIG_S"],
     ...
   }
   → victim_address = calculate_contract_address(salt=0x1, class=0xABC, ctor=[victim_pub_key], deployer=0)
   → mempool now contains deploy_account for victim_address

2. Attacker calls gateway: POST /gateway/add_transaction
   {
     "type": "INVOKE",
     "sender_address": "<victim_address>",
     "calldata": ["<token_contract>", "<transfer_selector>", "3", "<attacker_address>", "1000000", "0"],
     "nonce": "0x1",
     "signature": [],          ← empty / invalid signature
     "resource_bounds": { "l2_gas": { "max_amount": "0x1000", "max_price_per_unit": "0x1" } },
     ...
   }

3. Gateway stateless check: passes (non-zero resource bounds, valid address format)

4. Gateway stateful check:
   - account_nonce = get_nonce(victim_address) = 0   ✓ (not deployed yet)
   - tx.nonce = 1                                     ✓
   - account_tx_in_pool_or_recent_block(victim_address) = true  ✓ (deploy_account is in pool)
   → skip_stateful_validations returns true
   → run_validate_entry_point called with validate=false
   → __validate__ is NEVER called
   → transaction accepted into mempool

5. Victim's legitimate nonce=1 invoke tx is rejected with DuplicateNonce.

6. Block execution:
   - deploy_account executes → victim's account is deployed
   - attacker's invoke executes → __validate__ called → fails (invalid signature) → tx rejected
   - victim's intended action never executes
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
