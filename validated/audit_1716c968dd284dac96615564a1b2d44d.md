### Title
Unrestricted `ft_on_transfer` Allows Minting ERC-20 Tokens Without Actual NEP-141 Deposits - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` function in the Aurora Engine lacks any caller-origin check to verify it is being invoked as part of a legitimate NEP-141 `ft_transfer_call` flow. Any NEAR account that controls a registered NEP-141 token can call `ft_on_transfer` directly on Aurora to mint arbitrary amounts of ERC-20 tokens without depositing any actual NEP-141 tokens. Combined with the equally unprotected `deploy_erc20_token` entrypoint, an unprivileged attacker can execute the full exploit chain from scratch.

---

### Finding Description

**Root cause 1 — `deploy_erc20_token` has no access control.**

`engine/src/contract_methods/connector.rs` lines 112–158:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // only liveness check
        let bytes = io.read_input().to_vec();
        let args = DeployErc20TokenArgs::deserialize(&bytes)...;
        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;
                ...
            }
        }
    })
}
```

There is no `require_owner_only` or any other caller restriction. Any NEAR account can register any NEP-141 account ID as an ERC-20 on Aurora.

**Root cause 2 — `ft_on_transfer` has no caller-origin check.**

`engine/src/contract_methods/connector.rs` lines 62–109:

```rust
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let predecessor_account_id = env.predecessor_account_id();
        ...
        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,   // used as the NEP-141 token identity
                &args,
                &current_account_id,
                handler,
            )
        };
        ...
    })
}
```

The only guard on the ERC-20 minting path is `get_erc20_from_nep141(&self.io, token)?` inside `receive_erc20_tokens` — which succeeds whenever the predecessor's account ID is already registered. There is no check that the call arrived via a legitimate `ft_transfer_call` callback from the token contract. Any account that is registered as a NEP-141 token can call `ft_on_transfer` directly.

`engine/src/engine.rs` lines 796–843 — `receive_erc20_tokens` mints tokens unconditionally once the lookup succeeds:

---

### Impact Explanation

**Critical — Direct theft of user funds.**

An attacker who controls a registered NEP-141 account can mint an unbounded quantity of its ERC-20 mirror token on Aurora without depositing any real NEP-141 tokens. If that ERC-20 token has liquidity on any Aurora-hosted DEX (e.g., a pool the attacker seeded with real ETH to attract other LPs), the attacker can swap the counterfeit ERC-20 tokens for real bridged ETH or other tokens, draining the pool. The stolen assets are real bridged value held inside Aurora's EVM state.

---

### Likelihood Explanation

**High.** The full exploit requires no privileged access:

1. Deploy a NEAR NEP-141 token contract at any account the attacker controls (standard, permissionless NEAR operation).
2. Call `deploy_erc20_token` on Aurora — no access control, succeeds for any caller.
3. Seed a DEX pool on Aurora with real ETH to attract liquidity providers.
4. Call `ft_on_transfer` directly on Aurora **from** the attacker's NEP-141 account with `amount = <large number>` and `msg = <attacker EVM address>`.
5. Aurora mints the ERC-20 tokens with no actual deposit.
6. Swap counterfeit ERC-20 tokens for real ETH in the pool.

Every step is reachable by an unprivileged external NEAR account.

---

### Recommendation

Add a caller-origin guard inside `ft_on_transfer` that rejects direct calls. The standard NEAR pattern is to verify the call arrived as a callback (i.e., `env::promise_results_count() > 0` and `predecessor == current_account_id`), or to maintain an explicit allowlist of approved NEP-141 token contracts that may invoke `ft_on_transfer`. At minimum, `deploy_erc20_token` should be restricted to the owner or a governance-controlled allowlist so that the set of mintable ERC-20 tokens cannot be expanded by an attacker.

---

### Proof of Concept

```
1. Attacker creates NEAR account `evil-token.near` and deploys a minimal NEP-141 contract.

2. Attacker calls Aurora's `deploy_erc20_token` with args = borsh(AccountId("evil-token.near")).
   → No access control; succeeds. Aurora now has an ERC-20 mirror for evil-token.

3. Attacker seeds a Uniswap-style pool on Aurora:
   evil-token-ERC20 / WETH, depositing real WETH to attract LPs.

4. Attacker signs a NEAR transaction FROM `evil-token.near` calling Aurora's `ft_on_transfer`
   with JSON args:
     { "sender_id": "evil-token.near",
       "amount": "1000000000000000000000000",

   - predecessor_account_id = "evil-token.near"  (registered NEP-141 ✓)

   - receive_erc20_tokens mints 1e24 evil-token ERC-20 to attacker's EVM address
   - No actual NEP-141 tokens were transferred to Aurora

6. Attacker calls swap() on the DEX pool, exchanging 1e24 evil-token for all WETH in the pool.
   Real WETH (bridged user funds) is stolen.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
