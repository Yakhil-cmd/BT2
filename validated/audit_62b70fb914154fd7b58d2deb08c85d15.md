### Title
Gas Not Charged for Empty Input to BLS12-381 Mapping Precompiles — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/mappings.rs`)

---

### Summary

`Bls12381G1MappingPrecompile` (MAP_FP_TO_G1, address `0x10`) and `Bls12381G2MappingPrecompile` (MAP_FP2_TO_G2, address `0x11`) both perform an early-exit guard for zero-length input **before** charging the precompile gas cost. Every other EIP-2537 precompile in the same codebase charges gas unconditionally before any input validation. The result is an EVM semantic mismatch and resource-accounting bug: an unprivileged caller can invoke either mapping precompile with empty calldata and avoid paying the precompile-specific gas (5 500 gas for MAP_FP_TO_G1; 23 800 gas for MAP_FP2_TO_G2).

---

### Finding Description

In `mappings.rs`, both mapping precompiles share the same control-flow pattern:

```rust
// Bls12381G1MappingPrecompile::execute  (lines 20-34)
if input.len() == 0 {                          // ← early return, NO gas charged
    return Err(interface_error!(
        Bls12PrecompileInterfaceError::InvalidInputSize
    ));
}
let cost_ergs = Ergs(BLS12_381_FIELD_TO_G1_GAS * ERGS_PER_GAS);
resources.charge(&R::from_ergs_and_native(...))?;  // ← gas charged only here
if input.len() != FIELD_ELEMENT_SERIALIZATION_LEN {
    return Err(...);                           // ← gas IS charged for this path
}
``` [1](#0-0)

### Impact Explanation

EIP-2537 specifies a fixed gas cost for MAP_FP_TO_G1 (5 500) and MAP_FP2_TO_G2 (23 800) that must be consumed regardless of whether the call succeeds or fails. The standard EVM model charges precompile gas before executing the precompile body; a revert still burns the gas. By returning before the charge for the specific case of `input.len() == 0`, ZKsync OS deviates from this model:

- **Resource accounting bug**: the caller is not billed the precompile-specific gas for empty-input calls.
- **EVM semantic mismatch**: behaviour diverges from Ethereum mainnet (and from revm, the reference implementation used in the project's own differential fuzz tests).
- **Intra-codebase inconsistency**: the six other EIP-2537 precompiles all charge gas before any length check; only the two mapping precompiles do not.

---

### Likelihood Explanation

The precompile addresses `0x10` (MAP_FP_TO_G1) and `0x11` (MAP_FP2_TO_G2) are callable by any unprivileged EOA or contract. Sending a transaction with empty calldata to either address is trivial and requires no special privilege. The attacker-controlled entry path is direct: craft a `CALL` to `0x0000…0010` with zero-length input data. [7](#0-6) 

---

### Recommendation

Move the gas charge to before **all** input-length checks, mirroring the pattern used in every other EIP-2537 precompile:

```rust
// Corrected Bls12381G1MappingPrecompile::execute
let cost_ergs = Ergs(BLS12_381_FIELD_TO_G1_GAS * ERGS_PER_GAS);
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(0),
))?;
if input.len() != FIELD_ELEMENT_SERIALIZATION_LEN {   // covers both 0 and wrong length
    return Err(interface_error!(
        Bls12PrecompileInterfaceError::InvalidInputSize
    ));
}
```

Apply the same fix to `Bls12381G2MappingPrecompile`. The separate `input.len() == 0` guard can then be removed entirely, since the unified length check already handles it.

---

### Proof of Concept

1. Deploy a contract that calls `address(0x10).call{gas: 100_000}("")` (empty calldata to MAP_FP_TO_G1).
2. Observe that the call reverts (correct) but only 21 000 base gas + calldata cost is consumed — the 5 500 precompile gas is **not** deducted.
3. Repeat with `address(0x11)` (MAP_FP2_TO_G2) and observe 23 800 precompile gas is similarly not deducted.
4. Compare against a call with a non-empty but wrong-length input (e.g., 1 byte): the 5 500 / 23 800 gas **is** consumed, confirming the inconsistency is specific to the zero-length path.
