### Title
Silent Fund Loss via EvmErc20V2/ExitToNear Input Format Mismatch When `error_refund` Feature Is Absent — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

`EvmErc20V2.withdrawToNear` prepends the caller's 20-byte address to the precompile input as a refund address. The `ExitToNear` precompile only strips those 20 bytes when compiled with the `error_refund` feature. Without that feature the precompile misparses the input, returns `ERR_INVALID_AMOUNT`, and the Solidity assembly block silently ignores the failure — after the ERC-20 burn has already committed. The user's tokens are permanently destroyed with no NEAR-side transfer.

### Finding Description

**`EvmErc20V2.withdrawToNear`** constructs the precompile input as:

```
0x01 | sender(20 bytes) | amount_b(32 bytes) | recipient
``` [1](#0-0) 

The legacy **`EvmErc20.withdrawToNear`** (V1) omits the sender prefix:

```
0x01 | amount_b(32 bytes) | recipient
``` [2](#0-1) 

The `ExitToNear` precompile has two compile-time `parse_input` variants:

**With `error_refund`** — strips bytes `[1..21]` as `refund_address`, returns `&input[21..]`: [3](#0-2) 

**Without `error_refund`** — strips only the flag byte, returns `&input[1..]`: [4](#0-3) 

For flag `0x1`, the remaining slice is immediately indexed as `input[..32]` for the amount: [5](#0-4) 

Without `error_refund`, `input[..32]` is `sender[0..20] ++ amount_b[0..12]`. Interpreted as a big-endian U256, any non-zero sender address produces a value far exceeding `u128::MAX`. `parse_amount` rejects it: [6](#0-5) 

The `MIN_INPUT_SIZE` without `error_refund` is only 3 bytes, so the V2 payload (≥ 53 bytes) passes the size guard and the failure occurs at `parse_amount`, not earlier: [7](#0-6) 

The precompile returns `Err(ExitError::Other("ERR_INVALID_AMOUNT"))`, which the EVM translates to a failed inner `call` (return value 0). The Solidity assembly block captures `res` but **never checks it**: [8](#0-7) 

Because `_burn` executes unconditionally before the assembly block, the ERC-20 balance is already reduced. The outer transaction succeeds, the burn is committed, and no NEAR-side `ft_transfer` promise is ever scheduled.

### Impact Explanation

Every call to `EvmErc20V2.withdrawToNear` on an engine compiled without `error_refund` silently destroys the caller's ERC-20 tokens. The corresponding NEP-141 balance remains locked in Aurora's account with no mechanism for recovery. This is permanent, irreversible fund loss for any user who calls the function.

### Likelihood Explanation

`EvmErc20V2` was introduced specifically to carry the sender address for the `error_refund` refund path. There is no on-chain or compile-time guard preventing `EvmErc20V2` from being deployed against an engine binary that lacks the `error_refund` feature. The mismatch is invisible to users and to the contract itself. Any user holding a non-zero balance can trigger the loss by calling `withdrawToNear` with any valid recipient and any non-zero amount.

### Recommendation

1. **Enforce format compatibility at the contract level.** `EvmErc20V2.withdrawToNear` should revert if the precompile call returns 0 (i.e., add `if iszero(res) { revert(0, 0) }` after the `call`). This converts a silent loss into a visible revert, preserving the user's balance.
2. **Document the hard deployment constraint.** `EvmErc20V2` must only be deployed against an engine compiled with `error_refund`. This constraint should be enforced in deployment scripts or checked via a storage flag readable by the contract.
3. **Consider a runtime feature probe.** The engine could expose a storage key indicating which input format is expected, allowing the Solidity contract to select V1 or V2 encoding dynamically.

### Proof of Concept

Construct the V2 input manually and feed it to `ExitToNearParams::try_from` compiled without `error_refund`:

```rust
// Compiled without `error_refund`
let sender: [u8; 20] = [0xAB; 20]; // any non-zero address
let amount: u128 = 1_000;
let recipient = b"alice.near";

let mut input = vec![0x01u8];
input.extend_from_slice(&sender);                          // 20 bytes
input.extend_from_slice(&U256::from(amount).to_big_endian()); // 32 bytes
input.extend_from_slice(recipient);

// parse_input (no error_refund) returns &input[1..] = sender ++ amount_b ++ recipient
// parse_amount reads input[..32] = sender[0..20] ++ amount_b[0..12]
// U256::from_big_endian of that 32-byte slice >> u128::MAX  →  ERR_INVALID_AMOUNT
let result = ExitToNearParams::try_from(input.as_slice());
assert!(result.is_err()); // confirmed: precompile fails
```

On the Solidity side, `_burn` has already executed before the `call` opcode, so the user's balance is gone regardless of the precompile result.
