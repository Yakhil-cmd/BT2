### Title
Disabled Per-Share Verification in FROST EdDSA `aggregate()` Enables Unattributable Permanent Signing DoS — (File: src/frost/eddsa/sign.rs)

---

### Summary
In both `do_sign_coordinator_v1` and `do_sign_coordinator_v2`, the FROST EdDSA coordinator calls `aggregate()` with a deliberately empty `verifying_shares` map. This disables per-participant signature-share verification. A single malicious participant can submit an invalid signature share on every signing round, causing every aggregate to fail, while the coordinator has no mechanism to identify or exclude the culprit — permanently denying signing to all honest parties.

---

### Finding Description
In `src/frost/eddsa/sign.rs`, both coordinator paths construct the `PublicKeyPackage` with an empty `BTreeMap` for `verifying_shares`:

```rust
// do_sign_coordinator_v1, line 163
let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
```

```rust
// do_sign_coordinator_v2, line 218
let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
``` [1](#0-0) [2](#0-1) 

The `frost_ed25519::aggregate()` function, when given an empty `verifying_shares` map, skips the per-share check that would verify each participant's `round2::SignatureShare` against their individual verifying share (derived from the DKG output). It still verifies the final aggregated signature against the group public key — but only after combining all shares. If any single share is invalid, the aggregate fails and returns an error. The coordinator has no way to determine which participant submitted the bad share.

The code comment acknowledges this explicitly:

```rust
// We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
// Feature "cheater-detection" only points to a malicious participant, if there's such.
// It doesn't bring any additional guarantees.
``` [3](#0-2) 

This reasoning is incorrect from a security standpoint. Without per-share verification, the coordinator cannot distinguish a malicious participant from a network error, cannot exclude the culprit, and cannot retry with a valid subset. The malicious participant can repeat the attack on every signing attempt indefinitely.

---

### Impact Explanation
**High — Permanent denial of signing for honest parties.**

A single malicious participant in the signing group can submit a random or zeroed `round2::SignatureShare` on every signing round. Each attempt will fail at the `aggregate()` call with a `ProtocolError`. Because `verifying_shares` is empty, the coordinator cannot attribute the failure to any specific participant. The malicious participant is never identified or excluded. Honest parties are permanently unable to produce a valid threshold EdDSA signature under this participant set.

This applies to both the V1 (online nonce) and V2 (presignature-based) signing flows.

---

### Likelihood Explanation
**Medium.** The attack requires one malicious participant to be included in the signing group. In a threshold scheme with `t-of-n` signing, any participant selected for a signing round can execute this attack. No special privilege, leaked key, or cryptographic break is required — only the ability to send an arbitrary `round2::SignatureShare` message to the coordinator, which is a normal protocol message.

---

### Recommendation
Populate `verifying_shares` in the `PublicKeyPackage` with each participant's verifying share, derived from the `KeygenOutput`. The `KeygenOutput` already contains the `public_key` (group verifying key); the per-participant verifying shares can be derived from the DKG commitment or stored alongside the keygen output. With a populated map, `aggregate()` will verify each share individually before combining, allowing the coordinator to identify and exclude the malicious participant and retry with a valid subset.

---

### Proof of Concept

1. A threshold EdDSA signing session is initiated with participants `[P1, P2, P3]` and threshold `2`.
2. Malicious participant `P2` intercepts the normal signing flow and, instead of computing a valid `round2::SignatureShare`, sends a random scalar as their share to the coordinator.
3. The coordinator (`P1`) collects all shares and calls:
   ```rust
   let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
   let signature = aggregate(&signing_package, &signature_shares, &public_key_package)?;
   ```
4. `aggregate()` skips per-share verification (empty map), combines all shares, and attempts to verify the result against the group public key. Verification fails.
5. The coordinator returns `ProtocolError::AssertionFailed("signature failed to verify")`. No participant is identified as the culprit.
6. `P2` repeats this on every subsequent signing attempt. Signing is permanently denied for `P1` and `P3`.
