### Title
Greedy Regex in `parseOfferData` Always Selects Last `offer1` Match, Enabling Offer Substitution via Crafted File — (`packages/gui/src/components/offers/OfferImport.tsx`, `packages/gui/src/components/offers2/OfferBuilderImport.tsx`)

### Summary

The `parseOfferData` function in both offer import components uses a greedy regex with the `s` (dotAll) flag. When a file or pasted payload contains multiple valid `offer1` bech32 strings, the greedy `(?<leading>.*)` group always consumes everything up to the **last** `offer1` occurrence, causing the last offer — not the first — to be extracted, summarized, and presented to the user for acceptance.

### Finding Description

Both affected files define an identical `parseOfferData` function:

```ts
const matches = data.match(
  /(?<leading>.*)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
);
return [matches?.groups?.offer, ...];
``` [1](#0-0) [2](#0-1) 

The `s` flag makes `.` match newlines, and `(?<leading>.*)` is greedy. JavaScript's regex engine backtracks from the rightmost position, so when the input contains two `offer1` strings, `leading` absorbs the first one entirely and `offer` captures only the last. There is no guard, warning, or count check for multiple matches.

The extracted `offerData` is passed directly to `getOfferSummary` and then navigated to the acceptance view: [3](#0-2) 

The file size limit of 1 MB is the only constraint on input, and two bech32 offer strings fit comfortably within it: [4](#0-3) 

### Impact Explanation

An attacker who delivers a crafted `.offer` file (e.g., via a marketplace link, email, or chat) containing:

```
offer1<legitimate_bech32>
offer1<malicious_bech32>
```

causes the GUI to silently discard the first offer and present the **malicious** offer's summary for acceptance. The user, expecting to review the legitimate offer, may accept the malicious one — resulting in direct asset loss (XCH, CAT, NFT) or an unfavorable trade. This is a concrete offer-acceptance approval hijack.

### Likelihood Explanation

The attack requires only that the user open or paste a crafted offer payload — the normal, intended workflow. No special permissions, leaked keys, or host compromise are needed. The behavior is deterministic and reproducible with any two syntactically valid `offer1` strings.

### Recommendation

Replace the single greedy match with an explicit search for **all** `offer1` matches and either:
- Reject the input if more than one is found (with a clear error), or
- Always select the **first** match (use a non-greedy or anchored approach, e.g., `data.match(/offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+/)` which returns the first match by default without the `s`-flag greedy leading group).

Apply the fix identically in both:
- `packages/gui/src/components/offers/OfferImport.tsx`
- `packages/gui/src/components/offers2/OfferBuilderImport.tsx`

### Proof of Concept

```ts
// Reproducible in Node.js / browser console — no wallet needed
function parseOfferData(data) {
  const matches = data.match(
    /(?<leading>.*)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s
  );
  return matches?.groups?.offer;
}

const legitimate = 'offer1qpzry9x8gf2tvdw0s3jn54khce6mua7lAAAA';
const malicious  = 'offer1qpzry9x8gf2tvdw0s3jn54khce6mua7lBBBB';

const result = parseOfferData(`${legitimate}\n${malicious}`);
console.assert(result === malicious, 'malicious offer selected');
// Assertion passes — the malicious offer is always returned
```

Feeding a file with this content through the drag-and-drop or paste import path causes `getOfferSummary` to be called with `malicious` and the acceptance dialog to display the malicious offer's terms.
