### Title
Missing Lower-Bound Check on `threshold` in `assert_sign_inputs` Allows Invalid Threshold in EdDSA Signing - (`src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates the upper bound of `threshold` (`threshold <= participants.len()`) but omits the lower-bound check (`threshold >= 2`) that is consistently enforced in the analogous `assert_key_invariants` in `src/dkg.rs`. Because `ReconstructionLowerBound` is a plain `usize` wrapper with no enforced minimum, a caller can pass `threshold = 1` (or `0`) into the EdDSA signing path without triggering any error, causing the signing protocol to proceed with a cryptographically invalid threshold.

---

### Finding Description

`assert_sign_inputs` (`src/frost/mod.rs`, lines 120–159) performs the following threshold validation:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [1](#0-0) 

Only the upper bound is checked. The lower bound check (`threshold >= 2`) is entirely absent.

Compare this to `assert_key_invariants` in `src/dkg.rs`, which enforces **both** bounds:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [2](#0-1) 

`ReconstructionLowerBound` is a bare `usize` newtype with no enforced minimum:

```rust
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
``` [3](#0-2) 

The same partial-check pattern also appears in the `presign` function in the same file:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [4](#0-3) 

No lower-bound guard exists there either.

The structural parallel to the external report is exact:

| External Report | This Codebase |
|---|---|
| `minAmountOut_ <= maxPayout` checked | `threshold <= participants.len()` checked |
| `payout >= minAmountOut_` **not** checked | `threshold >= 2` **not** checked |
| Returns market with payout < minimum | Proceeds with threshold below cryptographic minimum |

---

### Impact Explanation

In FROST-based EdDSA signing, `threshold` governs the degree of the secret-sharing polynomial and the Lagrange interpolation used to reconstruct the aggregate signature. A key generated with `threshold = 2` (degree-1 polynomial) is incompatible with a signing session that presents `threshold = 1` (degree-0 polynomial). The Lagrange coefficients computed during aggregation will be wrong, producing a signature that is cryptographically invalid and unusable. Honest parties who receive and attempt to verify this output will find it fails verification — a corrupted, unusable signing output.

This maps to: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

`assert_sign_inputs` is a `pub` function and is the sole validation gate for the EdDSA signing path. Any library caller constructing a `ReconstructionLowerBound(1)` — whether through misconfiguration or malicious intent — passes the existing check silently. The missing guard is a single missing `if` branch, making it easy to trigger accidentally or deliberately.

---

### Recommendation

Add the lower-bound check to `assert_sign_inputs` (and to the `presign` function) to match the invariant already enforced in `assert_key_invariants`:

```rust
// In assert_sign_inputs, after the existing upper-bound check:
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
``` [5](#0-4) 

Apply the same fix to the `presign` function's threshold validation block. [4](#0-3) 

---

### Proof of Concept

1. Construct a `ReconstructionLowerBound(1)` (valid per the type — no minimum is enforced).
2. Call the EdDSA `sign` or `presign` entry point with this threshold and any two participants.
3. `assert_sign_inputs` checks `1 <= 2` (passes) and skips the missing `< 2` guard.
4. The signing protocol runs with `threshold = 1` against a key generated with `threshold = 2`.
5. Lagrange interpolation uses degree-0 coefficients on a degree-1 secret polynomial; the aggregated signature is cryptographically incorrect and fails external verification. [6](#0-5) [3](#0-2)
