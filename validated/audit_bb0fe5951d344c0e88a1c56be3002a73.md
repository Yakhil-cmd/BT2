### Title
Retroactive Silo Whitelist Activation Permanently Freezes Pre-Existing ERC-20 Token Holders' Funds - (File: `engine/src/engine.rs`)

### Summary

Aurora Engine's Silo mode includes an `Address` and `Account` whitelist that gates all EVM transaction submissions via `assert_access`. The whitelist is disabled by default and can be enabled at any time by the owner. When the whitelist is enabled after users have already bridged NEP-141 tokens into Aurora (receiving ERC-20 tokens), those users — whose addresses were never added to the whitelist — lose the ability to call `submit` and therefore cannot invoke the `ExitToNear` precompile to recover their tokens. There is no exemption for pre-existing token holders, creating a retroactive fund freeze.

### Finding Description

**Root cause — `assert_access` in `engine/src/engine.rs`:**

Every EVM transaction submitted via `submit` / `submit_with_alt_modexp` unconditionally calls `assert_access`:

```rust
// engine/src/engine.rs:1052
assert_access(&io, env, &transaction)?;
```

`assert_access` (lines 1756–1775) calls `silo::is_allow_submit`, which checks both the `Address` whitelist and the `Account` whitelist:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I, env: &E, transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
```

`is_allow_submit` (lines 136–138 of `engine/src/contract_methods/silo/mod.rs`) returns `false` if either the `Address` or `Account` whitelist is enabled and the caller is absent:

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
```

The whitelist is **disabled by default** (per `whitelist.rs` line 42: "White list is disabled by default") and can be enabled at any time by the owner via `set_whitelist_status` / `set_whitelists_statuses`.

**Exploit path:**

1. User bridges NEP-141 tokens to Aurora via `ft_on_transfer` → `receive_erc20_tokens` mints ERC-20 tokens to the user's EVM address. The user's funds are now held inside the Aurora EVM.
2. The owner later enables the `Address` whitelist (and/or `Account` whitelist) via `set_whitelist_status`.
3. The user's EVM address is not on the whitelist (they deposited before the whitelist existed).
4. The user attempts to exit their ERC-20 tokens back to NEAR by submitting an EVM transaction that calls the ERC-20 `burn` function, which triggers the `ExitToNear` precompile.
5. `submit_with_alt_modexp` calls `assert_access`, which calls `is_allow_submit`, which returns `false` because the user's address is absent from the enabled `Address` whitelist.
6. The call reverts with `EngineErrorKind::NotAllowed`. The user's ERC-20 tokens are frozen.

There is no alternative withdrawal path for ERC-20 tokens: the `withdraw` NEAR-level call only handles base ETH (it calls `engine_withdraw` on the ETH connector), not ERC-20/NEP-141 tokens. ERC-20 exit is exclusively via the `ExitToNear` precompile, which requires a successful `submit`.

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

Any user who bridged NEP-141 tokens into Aurora before the silo whitelist was activated, and whose EVM address is not subsequently added to the whitelist, cannot recover their ERC-20 tokens. The freeze persists for as long as the owner does not add the user's address or disable the whitelist. If the owner is unresponsive or adversarial, the freeze becomes permanent. The affected tokens are real user assets (bridged NEP-141 tokens), not unclaimed yield.

### Likelihood Explanation

Silo mode is an explicitly supported production feature of Aurora Engine. The whitelist is designed to be enabled post-deployment. Any Silo deployment that activates the `Address` or `Account` whitelist after users have already deposited ERC-20 tokens triggers this condition. No special attacker capability is required — the victim is any ordinary token holder whose address was not pre-populated in the whitelist.

### Recommendation

Before enabling the `Address` or `Account` whitelist, the owner must ensure all existing ERC-20 token holders are added to the whitelist. Alternatively, introduce a dedicated "exit-only" path (e.g., a separate NEAR-level `exit_erc20_to_near` method analogous to `withdraw` for base ETH) that bypasses the silo whitelist check, so that token holders can always recover their assets regardless of whitelist state. A third option is to apply the whitelist check only to transactions submitted after the whitelist was enabled (recording an activation timestamp), mirroring the recommendation in the reference report.

### Proof of Concept

1. Deploy Aurora Engine in Silo mode (no whitelist active yet).
2. User `alice.near` (EVM address `0xAlice`) calls `ft_on_transfer` on a NEP-141 token, bridging 1000 tokens into Aurora. `receive_erc20_tokens` mints 1000 ERC-20 tokens to `0xAlice`.
3. Owner calls `set_whitelist_status({ kind: Address, active: true })` — whitelist is now active with zero entries.
4. `0xAlice` submits an EVM transaction calling `erc20.burn(1000, "alice.near")` to exit tokens via `ExitToNear`.
5. `submit_with_alt_modexp` → `assert_access` → `is_allow_submit` → `is_address_allowed` checks `Whitelist::init(io, WhitelistKind::Address).is_exist(&0xAlice)` → returns `false` (address not in list).
6. Transaction reverts with `EngineErrorKind::NotAllowed`. Alice's 1000 ERC-20 tokens remain frozen in Aurora with no recovery path.
