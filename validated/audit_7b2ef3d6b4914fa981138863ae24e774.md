### Title
Gateway Admits Invoke Transactions With Unverified Signatures via `skip_stateful_validations` Deploy-Account UX Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validation path contains a deliberate UX shortcut (`skip_stateful_validations`) that completely skips the account's `__validate__` entry-point for any Invoke V3 transaction whose nonce is exactly 1 when the on-chain account nonce is 0 and the account already has any transaction in the mempool. Because the mempool's own `validate_tx` path carries no signature field and performs no cryptographic check, an unprivileged attacker can submit an Invoke transaction with an arbitrary (invalid) signature for any freshly-deploying account and have it admitted to the mempool without any signature verification.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

When all three conditions hold:

1. The incoming transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)` (hardcoded)
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`

…the function returns `true` (skip), and the caller sets `validate: !skip_validate = false`: [2](#0-1) 

With `validate: false`, `StatefulValidator::perform_validations` reaches the `ApiTransaction::Invoke` branch and immediately returns `Ok(())` without ever calling `validate_tx` (the `__validate__` entry point): [3](#0-2) 

**No signature check anywhere in the admission path**

The only other validation that runs before the skip decision is `validate_by_mempool`, which passes a `ValidationArgs` struct that contains no signature field and performs only nonce-range and fee-escalation checks: [4](#0-3) [5](#0-4) 

The stateless validator checks only that the signature array does not exceed a length limit; it never verifies the cryptographic content: [6](#0-5) 

**`account_tx_in_pool_or_recent_block` is trivially satisfiable**

The condition that gates the skip is satisfied as soon as any transaction from the target address is present in the pool or was committed in a recent block: [7](#0-6) 

A legitimate user's `deploy_account` transaction is publicly visible in the mempool the moment it is submitted, so an attacker can observe it and immediately satisfy this condition.

---

### Impact Explanation

An attacker who observes a `deploy_account` transaction for address X in the mempool can submit an Invoke transaction for X with:
- `nonce = 1`
- arbitrary `calldata` (e.g., a transfer to the attacker's address)
- a completely invalid signature (e.g., `[Felt::ZERO, Felt::ZERO]`)

The gateway will accept this transaction and forward it to the mempool. The mempool will accept it (nonce 1 > account nonce 0, no duplicate). The legitimate user's own nonce-1 transaction will then be rejected with `DuplicateNonce`.

When the batcher later executes the attacker's transaction, `__validate__` is called with the invalid signature and the transaction is rejected. However:

- The legitimate user's nonce-1 transaction has been displaced for at least one block.
- If the account contract's `__validate__` is permissive (e.g., always returns `VALIDATED`), the attacker's arbitrary calldata executes successfully — draining funds or performing unauthorized state changes.
- Even with a strict `__validate__`, the attacker can repeat the attack every time the user re-submits, creating a sustained DoS.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

- The triggering condition (deploy_account in mempool + nonce=1 invoke) is a standard, publicly documented UX flow.
- The mempool is observable by any network participant.
- No privileged access is required; any RPC client can submit the crafted transaction.
- The default `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, so the window is always open for every new account deployment. [8](#0-7) 

---

### Recommendation

1. **Verify the signature even when skipping `__validate__`**: Before setting `validate: false`, perform a lightweight off-chain ECDSA check against the transaction hash and the account's known public key (available from the deploy_account calldata). This preserves the UX benefit while closing the admission gap.

2. **Alternatively, restrict the skip to transactions whose hash matches the one pre-registered by the deploy_account**: The deploy_account transaction already carries the account's public key in its constructor calldata. The gateway can extract it and verify the invoke signature without running the full VM.

3. **At minimum, add a nonce-1 invoke to the mempool only if the corresponding deploy_account transaction in the pool was submitted by the same sender address**, preventing a third party from triggering the skip on behalf of an account they do not control.

---

### Proof of Concept

```
# Step 1 – Legitimate user submits deploy_account for address X (nonce=0).
POST /gateway/add_transaction
{ "type": "DEPLOY_ACCOUNT", "sender_address": X, "nonce": "0x0", ... valid sig ... }
# → accepted; X now appears in mempool

# Step 2 – Attacker submits invoke for X with nonce=1 and invalid signature.
POST /gateway/add_transaction
{
  "type": "INVOKE_FUNCTION",
  "sender_address": X,
  "nonce": "0x1",
  "calldata": [ ... transfer_all_to_attacker ... ],
  "signature": ["0x0", "0x0"]   # ← invalid
}
# Gateway path:
#   stateless_validator: signature length 2 ≤ 4000 → OK
#   stateful_validator:
#     account_nonce = 0 (from state)
#     tx.nonce() = 1
#     account_tx_in_pool_or_recent_block(X) = true  (deploy_account is in pool)
#     → skip_stateful_validations returns true
#     → run_validate_entry_point called with validate=false → __validate__ NOT called
#   mempool.validate_tx: nonce 1 ≥ account nonce 0, no duplicate → OK
# → ACCEPTED into mempool without any signature verification

# Step 3 – Legitimate user submits their own nonce=1 invoke.
POST /gateway/add_transaction { "type": "INVOKE_FUNCTION", "nonce": "0x1", ... valid sig ... }
# → REJECTED: MempoolError::DuplicateNonce { address: X, nonce: 1 }
```

The attacker's transaction is in the mempool with an invalid signature. If the account's `__validate__` is strict, it fails at execution time and the user can retry. If `__validate__` is permissive, the attacker's calldata executes. [1](#0-0) [2](#0-1) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
