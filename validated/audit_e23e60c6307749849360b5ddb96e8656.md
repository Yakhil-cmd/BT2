### Title
Silent Precompile Failure After `_burn` Causes Permanent Fund Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary
`EvmErc20.withdrawToNear` burns the caller's ERC-20 tokens before calling the `ExitToNear` precompile. The assembly `call()` return value is never checked and the function never reverts on precompile failure. If the precompile rejects the input (e.g., non-UTF-8 or invalid NEAR account ID in `recipient`), the burn is committed but no exit promise is scheduled, permanently locking the backing NEP-141 tokens in Aurora's account.

### Finding Description
In `EvmErc20.sol` lines 53–63, `_burn` executes unconditionally before the precompile call. The assembly block stores the call success flag in `res` but never reads it. Solidity has no visibility into the failure. The transaction succeeds from the EVM's perspective, the user's token balance is reduced to zero, but the `ExitToNear` precompile never emits the `promise_log` that schedules the NEP-141 `ft_transfer`. The NEP-141 tokens remain in Aurora's custody with no mechanism to reclaim them.

The same structural flaw exists in `EvmErc20V2.sol` lines 53–63. [9](#0-8) 

### Impact Explanation
**Critical — Permanent freezing of funds.** The EVM-side tokens are destroyed (total supply reduced). The NEP-141 tokens that backed them remain in Aurora's account. There is no admin recovery function, no re-mint path, and no way for the user to retry the exit because their EVM balance is already zero. The loss is permanent.

### Likelihood Explanation
Any user who calls `withdrawToNear` with a `recipient` that is not valid UTF-8 or not a valid NEAR account ID triggers this silently. This includes: passing a raw Ethereum address as bytes, passing an empty byte array, or passing bytes with high-bit characters. The function signature accepts `bytes memory recipient` with no on-chain validation before the burn. A user making a simple encoding mistake (e.g., passing `abi.encodePacked(someAddress)` instead of a NEAR account string) permanently loses their tokens with no error feedback.

### Recommendation
Check the precompile return value and revert on failure:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }  // <-- add this
    }
}
```

Alternatively, restructure to call the precompile first and only burn on success, though this is harder given the ERC-20 model (the precompile reads the burn amount from the call context).

Apply the same fix to `withdrawToEthereum` and both `EvmErc20.sol` and `EvmErc20V2.sol`.

### Proof of Concept
1. Deploy `EvmErc20` for a registered NEP-141 token.
2. Mint tokens to address `A`.
3. From address `A`, call `withdrawToNear(bytes("\xff\xfe"), amount)` — `\xff\xfe` is invalid UTF-8.
4. Observe: transaction succeeds, `Transfer` event emitted with `to = address(0)` (burn), balance of `A` is zero.
5. Observe: no `ExitToNear` event log, no promise log in transaction output.
6. The NEP-141 tokens remain in Aurora's account. The user has no EVM tokens to retry. Funds are permanently lost. [1](#0-0) [4](#0-3) [2](#0-1)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
