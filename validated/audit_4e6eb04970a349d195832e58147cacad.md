### Title
COA Ownership Proof Signature Replay Across Multiple COAs on the Same Cadence Account - (`File: fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` includes the target `evmAddress`. Because the `CapabilityPath` field inside the caller-supplied `COAOwnershipProof` is attacker-controlled, a signature produced to prove ownership of one COA (COA_A) can be replayed to falsely prove ownership of a different COA (COA_B) on the same Cadence account. This breaks the ERC-1271 `isValidSignature` guarantee implemented by the COA contract.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. It verifies that the supplied `signatures` are valid over `signedData` using the Cadence account's keys.
2. It verifies that the COA resource at the supplied `path` has an EVM address matching the supplied `evmAddress`.

The code itself documents the gap:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

The `COAOwnershipProof` struct (encoded in the `_sig` bytes passed to `isValidSignature`) contains a `CapabilityPath` field that is fully attacker-controlled: [2](#0-1) 

The ERC-1271 entry point in `coa.sol` passes `address(this)` as the EVM address and the caller-supplied `_sig` blob (which contains the proof including `CapabilityPath`) directly to the Cadence Arch precompile: [3](#0-2) 

The precompile routes to `validateCOAOwnershipProof` via `coaOwnershipProofValidator`, forwarding all proof fields including the attacker-chosen `CapabilityPath`: [4](#0-3) 

The signature verification step only checks that the signatures are valid over `signedData` — it never checks that `signedData` encodes `evmAddress`: [5](#0-4) 

The EVM address check only verifies that the COA at the attacker-chosen `path` matches the `evmAddress` argument — it does not verify that the signer intended to authorize that specific EVM address: [6](#0-5) 

### Impact Explanation

Any Cadence account that holds two or more COAs at different public capability paths is vulnerable. An attacker who obtains a valid `COAOwnershipProof` signature from Alice over arbitrary data `D` (e.g., from a legitimate interaction with COA_A) can construct a new proof pointing `CapabilityPath` at COA_B and call `COA_B.isValidSignature(D, forged_proof)`. The function returns `ValidERC1271Signature` (magic value `0x1626ba7e`), falsely asserting that Alice authorized `D` on behalf of COA_B. Any EVM protocol relying on ERC-1271 for authorization (token permits, NFT approvals, multisig thresholds) will accept this forged authorization, enabling unauthorized asset transfers or privileged actions on COA_B without Alice's consent.

### Likelihood Explanation

The Flow protocol places no restriction on the number of COAs a single Cadence account may create and publish at different public paths. The `EVM.createCadenceOwnedAccount` function is freely callable by any account holder. An attacker only needs: (1) a target victim with two published COA capabilities, and (2) any previously observed valid `COAOwnershipProof` signature from that victim (e.g., from a prior legitimate ERC-1271 interaction with COA_A). Both conditions are realistic as the Flow EVM ecosystem grows and users interact with multiple EVM protocols. The attack requires no privileged access and is executable from an unprivileged EVM transaction.

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. Before verifying signatures, check that the first 20 bytes of `signedData` equal `evmAddress` (or define a canonical encoding). Alternatively, prepend `evmAddress` to `signedData` before hashing inside the function itself, so the signature is always bound to the specific COA being proven.

### Proof of Concept

**Setup:** Alice's Cadence account has two COAs:
- COA_A at `/public/coaA` → EVM address `0xAAAA...`
- COA_B at `/public/coaB` → EVM address `0xBBBB...`

**Step 1 – Obtain a legitimate proof for COA_A:**
Alice signs data `D` (e.g., an EIP-712 hash from a DeFi protocol) to prove ownership of COA_A. The resulting `COAOwnershipProof` is:
```
{ Address: Alice, CapabilityPath: "coaA", KeyIndices: [0], Signatures: [sig_D] }
```

**Step 2 – Attacker forges a proof for COA_B:**
The attacker constructs a new proof by changing only `CapabilityPath`:
```
{ Address: Alice, CapabilityPath: "coaB", KeyIndices: [0], Signatures: [sig_D] }
```

**Step 3 – Attacker calls `COA_B.isValidSignature(D, forged_proof)`:**
- `address(this)` = `0xBBBB...` (COA_B's EVM address)
- The Cadence Arch precompile calls `validateCOAOwnershipProof(Alice, /public/coaB, D, [0], [sig_D], 0xBBBB...)`
- `keyList.verify(signatureSet, signedData: D, ...)` → **passes** (sig_D is a valid signature over D by Alice's key)
- `coaRef.address().bytes == evmAddress` → **passes** (COA_B's address is `0xBBBB...`)
- Returns `ValidationResult(isValid: true)`

**Result:** `isValidSignature` returns `0x1626ba7e` (valid), falsely proving Alice authorized `D` on COA_B. Any EVM contract that gates an action on this ERC-1271 check (e.g., a token permit or NFT approval) will grant the attacker unauthorized access.

        bytes memory _sig
    ) external view virtual returns (bytes4){
