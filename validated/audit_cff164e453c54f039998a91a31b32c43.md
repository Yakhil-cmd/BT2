### Title
Zero-Reward `claim_rewards` Proceeds Without Early Return, Emitting Empty Events and Wasting Gas - (File: src/pool/pool.cairo)

### Summary
`Pool::claim_rewards` in `src/pool/pool.cairo` does not guard against a zero-reward scenario. When called with no accrued rewards, it still performs a zero-value token transfer, writes updated pool member info to storage, and emits a `PoolMemberRewardClaimed` event with `amount: 0`. This is the direct analog of the `seizeInternal` zero-check omission: a state-mutating function that should be a no-op instead consumes gas, pollutes the event log, and performs unnecessary storage writes.

### Finding Description

`Pool::claim_rewards` (lines 335–377) calculates rewards, then unconditionally:
1. Writes updated `pool_member_info` to storage (line 362)
2. Calls `checked_transfer` with `rewards.into()` — which may be `0` (line 366)
3. Emits `PoolMemberRewardClaimed { pool_member, reward_address, amount: rewards }` — with `amount = 0` (lines 369–374) [1](#0-0) 

There is no guard of the form:
```cairo
if rewards.is_zero() {
    return 0;
}
```

The same pattern exists in `Staking::send_rewards_to_staker` (lines 1614–1629), which is called from both `claim_rewards` and `unstake_action` in `staking.cairo`. When `unclaimed_rewards_own == 0`, it still calls `claim_from_reward_supplier` with `amount = 0`, performs a zero transfer, and emits `StakerRewardClaimed { amount: 0 }`. [2](#0-1) 

The protocol's own test suite confirms this is reachable — `test_claim_rewards_no_rewards` explicitly tests that `claim_rewards` returns zero and the reward address balance stays zero, meaning the full function body executes with zero rewards: [3](#0-2) 

### Impact Explanation

Every zero-reward `claim_rewards` call:
- Writes to storage (pool member info checkpoint update) — unnecessary SSTORE cost
- Performs a zero-value ERC-20 `transfer` call — unnecessary cross-contract call
- Emits a `PoolMemberRewardClaimed` event with `amount: 0` — pollutes the event log used by off-chain indexers and monitoring tools

Any pool member or their reward address can call `claim_rewards` an unbounded number of times with no rate limiting. This constitutes **unbounded gas consumption** and griefing of the event log with no profit motive.

**Impact: Medium** — Griefing with no profit motive but damage to protocol (event log pollution, unnecessary storage writes); Unbounded gas consumption.

### Likelihood Explanation

The entry path requires only that the caller be a registered pool member or their reward address — no privileged role is needed. The condition (zero accrued rewards) is common: it occurs before the first attestation cycle completes, immediately after a prior `claim_rewards` call, or when the pool member's balance is below `min_for_rewards`. Any pool member can trigger this repeatedly at will. [4](#0-3) 

### Recommendation

Add an early-return guard after computing `rewards`, analogous to the fix recommended in the original report:

```cairo
fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
    // ... existing asserts ...

    let (mut rewards, updated_entry_to_claim_from) = self.calculate_rewards(...);
    rewards += pool_member_info._unclaimed_rewards_from_v0;

    // Early return if nothing to claim — avoids zero transfer, storage write, and empty event.
    if rewards.is_zero() {
        return 0;
    }

    // ... rest of function ...
}
```

Apply the same guard in `send_rewards_to_staker` in `staking.cairo`:

```cairo
fn send_rewards_to_staker(...) {
    let amount = staker_info.unclaimed_rewards_own;
    if amount.is_zero() {
        return;
    }
    // ... transfer and emit ...
}
``` [5](#0-4) 

### Proof of Concept

1. Deploy the system and stake with a pool enabled.
2. Have a delegator call `enter_delegation_pool` with a valid amount.
3. Without advancing any epochs or triggering attestation, call `pool.claim_rewards(pool_member)` — rewards are zero.
4. Observe: the call succeeds, `PoolMemberRewardClaimed { amount: 0 }` is emitted, storage is written, and a zero-value transfer is executed.
5. Repeat step 3 indefinitely — each call emits another empty event and writes storage with no state change.

This is directly confirmed by the existing test: [3](#0-2)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

fn test_claim_rewards_no_rewards() {
    let mut cfg: StakingInitConfig = Default::default();
    general_contract_system_deployment(ref :cfg);
    let token = cfg.test_info.strk_token;
    let token_address = token.contract_address();
    let token_dispatcher = IERC20Dispatcher { contract_address: token_address };

    let pool_contract = stake_with_strk_pool_enabled(:cfg);
    enter_delegation_pool_for_testing_using_dispatcher(:pool_contract, :cfg, :token);

    let pool_member = cfg.test_info.pool_member_address;
    // Balance is zero.
    let rewards = claim_rewards_for_pool_member(:pool_contract, :pool_member);
    assert!(rewards == Zero::zero());
    assert!(token_dispatcher.balance_of(cfg.pool_member_info.reward_address) == Zero::zero());

    advance_epoch_global();

    // Balance is not zero but no rewards.
    let rewards = claim_rewards_for_pool_member(:pool_contract, :pool_member);
    assert!(rewards == Zero::zero());
    assert!(token_dispatcher.balance_of(cfg.pool_member_info.reward_address) == Zero::zero());
}
