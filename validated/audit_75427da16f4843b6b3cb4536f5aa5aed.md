### Title
No Actual Slashing Enforced for HotStuff Protocol Violations — (`consensus/hotstuff/notifications/slashing_violation_consumer.go`)

### Summary
The `SlashingViolationsConsumer` is named and documented as the handler for slashable consensus offenses (double voting, equivocation, invalid proposals, double timeouts), but its entire implementation consists only of log warnings. No stake is deducted, no on-chain penalty is applied, and no challenge is raised. The `network/slashing/consumer.go` explicitly acknowledges this gap with the comment: *"This struct will be updated in the future when slashing is implemented."* A Byzantine staked consensus node can equivocate freely with zero economic consequence.

### Finding Description
`SlashingViolationsConsumer` implements three violation consumer interfaces (`ProposalViolationConsumer`, `VoteAggregationViolationConsumer`, `TimeoutAggregationViolationConsumer`) and is wired into the live consensus pipeline. Every handler — `OnDoubleVotingDetected`, `OnDoubleProposeDetected`, `OnInvalidBlockDetected`, `OnDoubleTimeoutDetected`, `OnInvalidVoteDetected`, `OnVoteForInvalidBlockDetected`, `OnInvalidTimeoutDetected` — terminates with a `c.log.Warn()` call and nothing else.

The detection path is fully functional: `VoteCollector.ensureVoteUnique()` detects equivocation via `VotesCache.AddVote()`, which returns a `DoubleVoteError`, and the notifier fires `OnDoubleVotingDetected`. The detection works; the consequence does not exist.

The same gap exists in the verification layer: `engine/verification/verifier/engine.go` handles chunk faults (`CFNonMatchingFinalState`, `CFInvalidVerifiableChunk`, `CFInvalidEventsCollection`) with `// TODO raise challenge` comments and `return nil` — no challenge is submitted against the misbehaving execution node.

### Impact Explanation
Flow's BFT security model is predicated on the assumption that Byzantine nodes lose their stake when caught misbehaving (slashing). Without slashing, a staked consensus node can:
- Double-vote across views to attempt safety violations
- Double-propose blocks (equivocation) to confuse the finalization logic
- Vote for known-invalid proposals

None of these actions incur any economic cost. The protocol's liveness and safety guarantees degrade to relying entirely on the team's own monitoring infrastructure — exactly the "protocol relies on its own bots" risk identified in the external report. If those monitoring systems fail or are unavailable, Byzantine nodes accumulate no penalty and can persist indefinitely.

### Likelihood Explanation
Any staked consensus node that turns Byzantine (key compromise, insider threat, or rational defection when slashing is absent) faces zero economic disincentive. The absence of slashing is a permanent, unconditional condition in the current codebase, not a transient configuration state.

### Recommendation
Implement actual stake-slashing logic inside `SlashingViolationsConsumer`. Each handler should submit a cryptographically provable slashing transaction (the two conflicting votes/proposals already constitute the proof) to the staking contract, deducting stake from the offending node. Until on-chain slashing is available, at minimum apply a protocol-level ejection (set `EpochParticipationStatus` to ejected) so the node loses its ability to participate further in the current epoch.

For the verification layer, implement the `// TODO raise challenge` paths in `engine/verification/verifier/engine.go` so that execution nodes submitting incorrect results face a slashing challenge.
