### Title
Anyone Can Call `update_rewards` with `disable_rewards: true` to Permanently Deny All Stakers' Consensus Block Rewards — (File: `src/staking/staking.cairo`)

---

### Summary

The `IStakingRewardsManager::update_rewards` function in the `Staking` contract is missing the access control mandated by the protocol spec ("Only starkware sequencer"). Any unprivileged caller can invoke it with `disable_rewards: true`, which marks the current block as "already updated" via the global `last_reward_block` storage slot without distributing any rewards. Because `last_reward_block` is a single global value, this permanently wastes the block's reward slot. Repeated every block, it permanently prevents every staker from earning consensus rewards.

---

### Finding Description

The spec at `docs/spec.md` lines 1644–1645 explicitly states:

> **access control**: Only starkware sequencer.

The implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1449–1507) contains **no caller validation whatsoever**:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ...
    self.last_reward_block.write(current_block_number);   // global slot consumed

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // no rewards distributed
    }
    // ... distribute rewards
}
``` [1](#0-0) 

`last_reward_block` is a single global storage value (not per-staker). Once it is written to the current block number, every subsequent call to `update_rewards` in that block fails with `REWARDS_ALREADY_UPDATED`, regardless of which staker is passed. [2](#0-1) 

The spec's access control requirement is confirmed at: [3](#0-2) 

The `IStakingRewardsManager` interface definition shows no access-control annotation in the trait itself: [4](#0-3) 

Tests confirm the function is callable by any address without any `cheat_caller_address` override: [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker calls `update_rewards(any_valid_staker, disable_rewards: true)` at the start of every block during the consensus-rewards era. Each call:

1. Passes the `REWARDS_ALREADY_UPDATED` guard (first call in the block).
2. Writes `last_reward_block = current_block`.
3. Returns immediately without distributing rewards (`disable_rewards == true`).

The sequencer's legitimate call with `disable_rewards: false` then fails with `REWARDS_ALREADY_UPDATED` for the rest of that block. No staker earns block rewards. Repeated every block, **all stakers are permanently denied consensus rewards** — their yield is frozen at zero indefinitely.

---

### Likelihood Explanation

**High.** The function is publicly callable with no privilege requirement. The attack costs only gas per block and is trivially automatable. The attacker gains nothing financially but can sustain the attack indefinitely at low cost, making it a realistic griefing vector especially against a protocol where staking rewards are the primary incentive.

---

### Recommendation

Add a sequencer-only caller check at the top of `update_rewards`, consistent with the spec:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    // ...
}
```

The check should validate `get_caller_address()` against a stored, governance-controlled sequencer address, mirroring the pattern used for `assert_caller_is_attestation_contract` in `update_rewards_from_attestation_contract`. [6](#0-5) 

---

### Proof of Concept

**Pre-conditions**: Consensus rewards are active (`is_pre_consensus()` returns `false`); at least one valid staker exists with non-zero balance.

**Attack steps**:

1. Attacker monitors the chain. At the start of block `N` (before the sequencer acts):
2. Attacker calls `update_rewards(valid_staker_address, disable_rewards: true)`.
3. The call succeeds: `last_reward_block` is set to `N`, no rewards are distributed.
4. Sequencer calls `update_rewards(designated_staker, disable_rewards: false)` — reverts with `REWARDS_ALREADY_UPDATED`.
5. No staker earns rewards for block `N`.
6. Attacker repeats at block `N+1`, `N+2`, … indefinitely.

**Result**: All stakers' consensus-era yield is permanently frozen at zero. The attack is confirmed by the assertion test: [7](#0-6) 

which shows that a single prior call to `update_rewards` (regardless of `disable_rewards`) blocks all subsequent calls in the same block with `REWARDS_ALREADY_UPDATED`.
