### Title
Gas refund for a deleted gas key incorrectly increments `FunctionCallPermission.allowance` when replaced by a `FunctionCall` key with the same public key — (`runtime/runtime/src/lib.rs`, `runtime/runtime/src/actions.rs`)

### Summary
When a gas key transaction is processed, gas is deducted from `GasKeyInfo.balance`. If the account owner subsequently deletes the gas key and adds a plain `FunctionCall` key with the same public key, the arriving gas refund receipt credits `account.amount()` correctly (fallback path) but also incorrectly increments `FunctionCallPermission.allowance` via `try_refund_allowance`, because the code cannot distinguish a gas-key-origin refund from a function-call-key-origin refund once the key type has changed.

### Finding Description
In `action_transfer_or_implicit_account_creation`:

1. `try_refund_gas_key_balance` is called; the key at `signer_public_key` is now `AccessKeyPermission::FunctionCall`, so `gas_key_info_mut()` returns `None` and the function returns `false`.
2. `action_transfer(account, deposit)` increments `account.amount()` by `deposit`.
3. Because `is_gas_refund` is still `true`, `try_refund_allowance` is called. It matches `AccessKeyPermission::FunctionCall` and, if `allowance` is `Some(...)`, increments it by `deposit` via `saturating_add`.

The allowance was never decremented for the gas key transaction (gas key transactions deduct from `GasKeyInfo.balance`, not from `FunctionCallPermission.allowance`). The net result is a free allowance increase equal to the gas refund amount.

### Impact Explanation
The corrupted value is `FunctionCallPermission.allowance`: it is incremented by the gas refund amount without a corresponding prior deduction. The account owner can exploit this to raise the spending limit of a `FunctionCall` key beyond its intended cap. Repeated cycles (gas key tx → delete gas key → re-add FunctionCall key with same pubkey → wait for refund) allow unbounded allowance inflation. The account balance is not double-spent (tokens are correctly refunded once to `account.amount()`), but the allowance limit — which is the security boundary for delegated key spending — is bypassed.

### Likelihood Explanation
Requires the account owner to deliberately delete a gas key and re-add a `FunctionCall` key with the same public key before the gas refund receipt is processed. This is a multi-step but fully unprivileged sequence of standard transactions. The window between gas key transaction execution and refund receipt processing is at least one block, giving ample time for the key swap.

### Recommendation
In `action_transfer_or_implicit_account_creation`, `try_refund_allowance` should only be called when the original transaction was signed with a `FunctionCall` key. One approach: before calling `try_refund_allowance`, check that the key at `signer_public_key` is `AccessKeyPermission::FunctionCall` **and** that no gas key with that public key existed at the time of the original transaction. A simpler conservative fix: skip `try_refund_allowance` entirely when `try_refund_gas_key_balance` returned `false` due to a key-type mismatch (key exists but is not a gas key), as opposed to a key-not-found case. This requires `try_refund_gas_key_balance` to return a tri-state (key not found / key found but wrong type / key found and credited).

### Proof of Concept
Runtime state test:
1. Set up account with gas key (GasKeyFullAccess) at `PK`, balance = 10 mNEAR.
2. Apply a gas key transaction; gas refund receipt is queued.
3. Apply a `DeleteKey(PK)` + `AddKey(PK, FunctionCall { allowance: Some(5 mNEAR), ... })` transaction.
4. Apply the gas refund receipt (e.g., 1 mNEAR).
5. Assert `account.amount()` increased by 1 mNEAR (correct).
6. Assert `FunctionCallPermission.allowance` is **still** 5 mNEAR (should be unchanged).
7. Observe that `allowance` is actually 6 mNEAR — the invariant is violated.

Note, that it's not always possible to refund the allowance, because the access key can be deleted between the moment when the transaction was
