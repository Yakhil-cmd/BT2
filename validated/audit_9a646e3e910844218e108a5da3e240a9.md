### Title
Unvalidated Received Vector Length in `do_generation_many` Allows Malicious Participant to Panic Honest Parties and Permanently Deny Triple Generation — (File: src/ecdsa/ot_based_ecdsa/triples/generation.rs)

---

### Summary

In `do_generation_many`, vectors received from other participants across three distinct protocol rounds are indexed with `[i]` for `i in 0..N` without first asserting that the received vector has length ≥ N. A single malicious participant can send a vector of length 0 (or any length < N) at any of these rounds, triggering an out-of-bounds index panic in the honest party's async task. Because Beaver triples are a hard prerequisite for OT-based ECDSA presigning and signing, this permanently denies signing for all honest parties in that session.

---

### Finding Description

`generate_triple_many<const N>` is the public entry point for batch triple generation. It calls `do_generation_many::<N>`, which internally maintains all state vectors at exactly length N. However, when deserializing and consuming messages from remote participants, no length guard is applied before indexing.

**Round 1 — commitment collection (lines 176–183):**

```rust
while all_commitments_vec.iter().any(|all_commitments| !all_commitments.full()) {
    let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
    for i in 0..N {
        all_commitments_vec[i].put(from, commitments[i]); // ← panics if commitments.len() < N
    }
}
```

A malicious participant sends a `Vec<Commitment>` of length 0. The first iteration (`i = 0`) panics.

**Round 2 — polynomial commitment reveal (lines 331–342):**

```rust
for (from, their) in
    recv_from_others::<PolynomialCommitmentsMessageMany>(&chan, wait2, &participants, me).await?
{
    for i in 0..N {
        let their_big_e = &their.big_e_v[i];   // ← panics if big_e_v.len() < N
        let their_big_f = &their.big_f_v[i];   // ← panics if big_f_v.len() < N
        let their_big_l = &their.big_l_v[i];   // ← panics if big_l_v.len() < N
        let their_randomizer = &their.randomizer_v[i]; // ← panics
        let their_phi_proof0 = &their.phi_proof0_v[i]; // ← panics
        let their_phi_proof1 = &their.phi_proof1_v[i]; // ← panics
        ...
    }
}
```

A malicious participant sends a `PolynomialCommitmentsMessageMany` whose inner `Vec` fields have length 0.

**Round 3 — private share accumulation (lines 626–632):**

```rust
for (_, c_j_i_v) in
    recv_from_others::<Vec<SerializableScalar<C>>>(&chan, wait6, &participants, me).await?
{
    for i in 0..N {
        let c_j_i = c_j_i_v[i].0; // ← panics if c_j_i_v.len() < N
        c_i_v[i] += c_j_i;
    }
}
```

The root cause is identical in all three cases: the protocol enforces the batch size N as a const-generic parameter on all internally produced data, but applies no corresponding length assertion to data received over the network before indexing into it. This is directly analogous to the reported pattern — a bound is enforced on one side of the protocol but not validated on the received side.

---

### Impact Explanation

A Rust index-out-of-bounds panic in an async future propagates as a task panic. In a tokio runtime the task terminates with `JoinError::Panicked`; in the library's own `run_protocol_common` simulator the panic unwinds the thread. Either way, the honest party's triple-generation session is irrecoverably aborted. Because OT-based ECDSA presigning requires pre-generated Beaver triples, permanent denial of triple generation equals permanent denial of ECDSA signing for honest parties — matching the **High** impact tier: *"Permanent denial of signing … for honest parties under valid protocol inputs and documented trust assumptions."*

---

### Likelihood Explanation

The attack requires only that one participant in the `generate_triple_many` session be malicious. The attacker needs to send a single well-formed but short `Vec` (e.g., length 0) during any of the three affected rounds. Serde will deserialize an empty `Vec` without error; the panic occurs only when the honest party indexes into it. No cryptographic material, no key leakage, and no external dependency is needed. The attack is trivially repeatable across sessions.

---

### Recommendation

Before the `for i in 0..N` loop in each affected receive block, assert the received vector's length:

```rust
// Round 1
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(format!(
        "commitments from {from:?}: expected {N}, got {}", commitments.len()
    )));
}

// Round 2 (check each field of PolynomialCommitmentsMessageMany)
if their.big_e_v.len() != N || their.big_f_v.len() != N || ... {
    return Err(ProtocolError::AssertionFailed(format!(
        "polynomial commitment vectors from {from:?} have wrong length"
    )));
}

// Round 3
if c_j_i_v.len() != N {
    return Err(ProtocolError::AssertionFailed(format!(
        "share vector from {_:?}: expected {N}, got {}", c_j_i_v.len()
    )));
}
```

This converts a panic into a recoverable `ProtocolError`, consistent with how all other protocol violations are handled throughout the codebase.

---

### Proof of Concept

1. Honest parties call `generate_triple_many::<4>(participants, me, threshold, rng)` with N = 4.
2. The malicious participant, instead of sending `Vec<Commitment>` of length 4 at `wait0`, sends an empty `Vec<Commitment>` (length 0).
3. The honest party deserializes the message successfully (empty `Vec` is valid).
4. The honest party enters `for i in 0..4` and evaluates `commitments[0]` — index 0 into a zero-length vector — triggering a Rust index-out-of-bounds panic.
5. The async task terminates. Triple generation is permanently aborted for that session. All subsequent ECDSA presign and sign calls that depend on these triples are denied.
