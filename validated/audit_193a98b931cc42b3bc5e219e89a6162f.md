### Title
Unbounded Loop Over User-Inflatable Balance Trace Enables Permanent Freezing of Unclaimed Yield - (`File: src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over a pool member's entire `pool_member_epoch_balance` trace in an unbounded loop. A delegator can inflate this trace by making one balance change per epoch (via `add_to_delegation_pool` or `exit_delegation_pool_intent`) without claiming rewards. After enough epochs, the loop will exceed Starknet's execution step limit, causing `claim_rewards` to permanently revert and freezing the delegator's unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over the pool member's balance trace from `entry_to_claim_from` to `pool_member_trace_length`: [1](#0-0) 

The code itself acknowledges the problem with the comment:

> `// **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.`

Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` invokes `set_member_balance` (or `increase_member_balance`), which inserts a new checkpoint into the `pool_member_epoch_balance` trace keyed at `current_epoch + K`: [2](#0-1) 

The `insert` function only deduplicates entries with the **same epoch key**. Since the key is `current_epoch + K`, each epoch in which the delegator makes a balance change produces a distinct, new entry in the trace: [3](#0-2) 

The `entry_to_claim_from` pointer advances only after a successful `claim_rewards` call: [4](#0-3) 

If the delegator never calls `claim_rewards`, the pointer stays at 0 and the loop must traverse every accumulated entry on the next claim attempt. Each loop iteration calls `find_sigma`, which performs additional storage reads against the cumulative rewards trace: [5](#0-4) 

`calculate_rewards` is also called from the read-only `pool_member_info_v1` view function, meaning any on-chain caller querying a bloated member's info also pays the cost: [6](#0-5) 

---

### Impact Explanation

After accumulating enough trace entries without claiming, the `claim_rewards` transaction will exceed Starknet's execution step limit and permanently revert. The delegator's unclaimed STRK rewards become permanently frozen and unclaimable. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

---

### Likelihood Explanation

The minimum amount for `add_to_delegation_pool` is 1 wei (`amount.is_non_zero()`): [7](#0-6) 

A delegator can therefore add 1 wei per epoch at negligible cost. Only the pool member or their reward address may call `add_to_delegation_pool`: [8](#0-7) 

This means the attacker inflates their own trace. While self-inflicted, the result is permanent loss of yield — a real economic harm. A delegator who simply forgets to claim for many epochs while continuing to top up their delegation (a normal usage pattern) can inadvertently trigger this condition without any malicious intent.

---

### Recommendation

1. **Enforce a maximum number of loop iterations per `claim_rewards` call** (e.g., cap at 100 entries per call) and allow partial claims, advancing `entry_to_claim_from` incrementally.
2. **Alternatively, enforce a minimum epoch gap between balance changes** to prevent trace inflation faster than the natural claim cadence.
3. **Remove the acknowledged "unbounded" comment** and replace it with an enforced bound.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with the minimum stake.
2. Each epoch, delegator calls `add_to_delegation_pool` with 1 wei. Each call inserts a new entry into `pool_member_epoch_balance` (keyed at `current_epoch + K`), growing the trace by 1.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs (e.g., N = 10,000), delegator calls `claim_rewards`. The `calculate_rewards` loop must iterate N times, each time calling `find_sigma` with storage reads.
5. The transaction exceeds Starknet's step limit and reverts. All accumulated unclaimed rewards are permanently frozen.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
