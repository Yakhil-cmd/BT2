### Title
WalletConnect `chia_takeOffer` Silently Forwards Attacker-Controlled `extra_conditions` to Daemon Without Displaying Them in Confirmation Dialog — (`packages/api/src/services/WalletService.ts`, `packages/gui/src/electron/commands/Commands.ts`, `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

A WalletConnect dApp that has been granted the `chia_takeOffer` permission can inject arbitrary CLVM conditions (e.g., `CREATE_COIN` to an attacker-controlled puzzle hash) via the `extra_conditions` parameter. These conditions are:

1. **Allowed through the dApp param allowlist** — `extra_conditions` is declared in the `chia_wallet.take_offer` command schema and inherited by the `chia_takeOffer` dApp command.
2. **Never shown in the confirmation dialog** — `parseCommandDisplay` builds the user-facing summary exclusively from the offer string; `extra_conditions` is completely ignored.
3. **Forwarded as-is to the daemon** — `WalletService.takeOffer` passes the full args object, including `extra_conditions`, directly to the `take_offer` RPC.

The user sees a legitimate offer summary and clicks Accept, but the signed spend bundle contains hidden conditions that transfer additional assets to the attacker.

---

### Finding Description

**Step 1 — `extra_conditions` is in the dApp param allowlist.**

`Commands.ts` defines `chia_wallet.take_offer` with `extra_conditions` as an optional `json` param: [1](#0-0) 

`DappCommands.ts` builds the `chia_takeOffer` dApp schema by inheriting the parent's `params` list when no override is provided: [2](#0-1) 

Because `chia_takeOffer`'s `dapp` entry does not override `params`, it inherits the full list including `extra_conditions`.

**Step 2 — `parseDappParams` passes `extra_conditions` through.**

The allowlist check in `parseDappParams` only verifies that each submitted key exists in `dappParamsMap`: [3](#0-2) 

Since `extra_conditions` is in the map, it passes. The `type: 'json'` branch applies no further validation — the array is accepted verbatim.

**Step 3 — The confirmation dialog never shows `extra_conditions`.**

`parseCommandDisplay` for `chia_wallet.take_offer` derives the entire user-facing summary from `params.offer` alone via `getOfferSummary`. The `extra_conditions` key is never read, decoded, or displayed: [4](#0-3) 

**Step 4 — `WalletService.takeOffer` forwards `extra_conditions` as-is to the daemon.** [5](#0-4) 

The `extraConditions?: any[]` type imposes no structural constraint. The daemon receives the raw array, appends the conditions to the spend bundle, and signs.

---

### Impact Explanation

A dApp with `chia_takeOffer` permission can cause the wallet to sign and broadcast a spend bundle that includes arbitrary CLVM conditions — such as `CREATE_COIN <attacker_puzzlehash> <large_amount>` (opcode 51) — without the user ever seeing those conditions. The user approves what appears to be a normal offer acceptance, but the resulting on-chain transaction transfers additional XCH or CAT tokens to the attacker. This is an unauthorized asset transfer with direct, irreversible financial loss.

---

### Likelihood Explanation

Any dApp that has been granted `chia_takeOffer` permission (a spending-class permission that users are explicitly warned about during pairing) can exploit this immediately. No additional privileges, leaked keys, or local compromise are required. The dApp only needs to include `extraConditions` in its `chia_takeOffer` call alongside a legitimate offer string.

---

### Recommendation

1. **Remove `extra_conditions` from the `chia_takeOffer` dApp param schema**, or add a dApp-level override that excludes it (analogous to how `chia_cancelOffer` overrides `params` to hide `secure`).
2. If `extra_conditions` must remain available to dApps, **decode and display each condition** in the confirmation dialog before the user approves, so the user can see every condition that will be included in the spend.
3. Consider adding a daemon-side guard that rejects `extra_conditions` on `take_offer` calls originating from WalletConnect sessions.

---

### Proof of Concept

```
// dApp side (has chia_takeOffer permission)
walletConnectClient.request({
  topic: session.topic,
  chainId: 'chia:mainnet',
  request: {
    method: 'chia_takeOffer',
    params: {
      offer: '<legitimate_offer_string>',   // user sees normal offer summary
      fee: '0',
      extraConditions: [
        {
          opcode: 51,                        // CREATE_COIN
          args: ['<attacker_puzzlehash>', 1_000_000_000_000]  // 1 XCH
        }
      ]
    }
  }
});
```

**Expected (broken) behavior:** User sees only the legitimate offer summary. On confirmation, the daemon appends the `CREATE_COIN` condition, signs, and broadcasts. The attacker receives 1 XCH.

**Expected (correct) behavior:** Either `extra_conditions` is rejected for dApp callers, or its contents are decoded and shown in the confirmation dialog before the user can approve.

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
