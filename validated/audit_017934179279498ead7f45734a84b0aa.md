### Title
FROST `PresignOutput` Derives `Clone`, Enabling Nonce Reuse and Private Signing Share Extraction — (File: `src/frost/mod.rs`)

---

### Summary

The `PresignOutput<C>` struct in `src/frost/mod.rs` derives `Clone`. The `sign_v2` signing entry-point correctly takes `presignature: PresignOutput` **by value** to enforce single-use at the type level, but the `Clone` derive completely undermines this protection. Any caller — including a malicious coordinator or a library consumer implementing retry logic — can clone a `PresignOutput` and submit the same FROST nonces to two distinct signing sessions. FROST nonce reuse across two messages allows algebraic recovery of the participant's private signing share.

---

### Finding Description

In `src/frost/mod.rs` lines 36–41, `PresignOutput<C>` is defined as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

The `nonces` field holds `SigningNonces<C>`, which are the per-session secret FROST nonces. These are cryptographically single-use: reusing the same nonces across two signing sessions with different messages directly reveals the nonce scalar, from which the signing share can be recovered.

The `sign_v2` function in `src/frost/eddsa/sign.rs` takes `presignature: PresignOutput` by value, which is the correct Rust idiom to enforce single-use (the value is moved and cannot be used again):

```rust
pub fn sign_v2(
    ...
    presignature: PresignOutput,
    message: Vec<u8>,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError>
``` [2](#0-1) 

However, because `PresignOutput` derives `Clone`, any caller can trivially bypass this protection:

```rust
let presig = run_presign(...);
let sig1 = sign_v2(..., presig.clone(), msg1);  // nonces used once
let sig2 = sign_v2(..., presig.clone(), msg2);  // same nonces reused
```

The library's own test code at line 595 of `src/frost/eddsa/sign.rs` explicitly uses `presign_output.clone()` before passing to `sign_v2`, demonstrating this pattern is treated as normal: [3](#0-2) 

By contrast, the OT-based ECDSA `PresignOutput` at least derives `ZeroizeOnDrop`, providing memory hygiene. The FROST `PresignOutput` has neither `ZeroizeOnDrop` nor any mechanism to mark nonces as consumed: [4](#0-3) 

The FROST presign protocol in `do_presign` generates nonces via `commit(&signing_share, &mut rng)` and returns them in `PresignOutput` without any invalidation token or use-once guard: [5](#0-4) 

---

### Impact Explanation

**Critical — Extraction of private signing shares.**

FROST nonce reuse is a textbook key-recovery attack. Given two signatures `(R, z1)` and `(R, z2)` produced with the same nonce commitment `R` but different messages `m1` and `m2`, an attacker can solve for the participant's signing share `s_i` algebraically. This is the same class of attack that broke the Sony PS3 ECDSA implementation. In a t-of-n threshold setting, recovering even one participant's signing share is a Critical disclosure of private key material, and if the attacker controls the coordinator role, they can target the threshold number of participants to reconstruct the full aggregate secret.

---

### Likelihood Explanation

**High.** The library's own test suite uses `presign_output.clone()` before calling `sign_v2`, normalizing the pattern. Any production application that:
- Implements retry logic after a failed signing round,
- Caches presign outputs for performance,
- Runs parallel signing sessions,
- Or simply stores `PresignOutput` in a `Vec` and iterates over it,

will naturally clone the struct. There is no documentation warning on `PresignOutput` itself, no `#[non_exhaustive]` guard, and no runtime check inside `sign_v2` to detect nonce reuse. The `Clone` derive is the direct, reachable root cause.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput<C>`** in `src/frost/mod.rs`. The by-value parameter in `sign_v2` already encodes the correct intent; removing `Clone` enforces it at compile time.
2. **Add `ZeroizeOnDrop`** to `PresignOutput<C>` so that nonce material is cleared from memory after the struct is dropped.
3. **Update the test** at `src/frost/eddsa/sign.rs:595` to not clone the presign output; instead, run a fresh presign for each signing session.
4. Add a doc-comment on `PresignOutput` explicitly stating that nonces are single-use and that cloning is unsafe.

---

### Proof of Concept

```rust
// Attacker-controlled coordinator or library caller:
let presig_outputs = run_presign(&keys, threshold, actual_signers, rng).unwrap();

// Clone the presign output for participant P
let presig_p = presig_outputs.iter().find(|(p, _)| p == &target).unwrap().1.clone();

// Sign message 1 with the presign output
let _sig1 = sign_v2(participants, threshold, target, coordinator,
                    keygen_output.clone(), presig_p.clone(), b"message1".to_vec());

// Sign message 2 with the SAME nonces (presig_p.nonces is identical)
let _sig2 = sign_v2(participants, threshold, target, coordinator,
                    keygen_output.clone(), presig_p.clone(), b"message2".to_vec());

// Both signing sessions use presig_p.nonces unchanged.
// The coordinator observes two signature shares (z1, z2) for the same R.
// From z1 - z2 = lambda * s_i * (c1 - c2), the signing share s_i is recovered.
```

The `Clone` derive on `PresignOutput<C>` at `src/frost/mod.rs:36` is the necessary and sufficient root cause. The `sign_v2` call at `src/frost/eddsa/sign.rs:64` accepts the cloned value without any nonce-reuse detection, completing the attack path.
