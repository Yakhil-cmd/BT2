### Title
Malicious Coordinator Can Redirect CKD Output by Substituting `app_pk`, Extracting the Confidential Derived Key — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The Confidential Key Derivation (CKD) protocol accepts `app_pk` (the application's ElGamal public key) as an unauthenticated, unverified parameter. The protocol never broadcasts `app_pk` among participants or verifies consensus on it. A malicious coordinator who initiates the protocol can silently substitute their own `app_pk'` for all participants, causing the derived secret to be encrypted under the attacker's key instead of the legitimate application's key. The coordinator then unmasks the output and extracts `msk · H(pk ‖ app_id)` — the confidential derived key.

---

### Finding Description

The `ckd()` entry point accepts `app_pk: PublicKey` as a plain parameter with no validation: [1](#0-0) 

Inside `compute_signature_share`, `app_pk` is used directly to mask the derived secret: [2](#0-1) 

The coordinator aggregates participant shares without any check that those shares were computed with the correct `app_pk`: [3](#0-2) 

Crucially, `app_pk` is never broadcast, never committed to, and never verified against any authenticated identity. The `hash_app_id_with_pk` function binds only the MPC public key and `app_id` — not `app_pk`: [4](#0-3) 

This means `app_pk` is entirely outside the protocol's integrity perimeter.

---

### Impact Explanation

**Critical — Extraction of confidential derived secrets.**

If the coordinator distributes `app_pk'` (their own ElGamal key) to all participants instead of the legitimate `app_pk`, every participant computes:

```
C_i = λ_i · (x_i · H(pk ‖ app_id)) + λ_i · y_i · app_pk'
```

The coordinator aggregates to obtain:

```
C_total = msk · H(pk ‖ app_id) + y_total · app_pk'
```

The coordinator then calls `CKDOutput::unmask(app_sk')`: [5](#0-4) 

yielding `msk · H(pk ‖ app_id)` — the full confidential derived key — which the coordinator was never supposed to learn. This directly violates the core security guarantee of CKD ("without MPC nodes learning the derived secret").

---

### Likelihood Explanation

The CKD protocol's explicit threat model includes a malicious coordinator (one of the MPC nodes). In any realistic deployment the coordinator is the party that receives the application's CKD request and then initiates the protocol with all participants, distributing `app_id` and `app_pk`. Because the protocol contains no broadcast or consensus step for `app_pk`, a malicious coordinator can substitute their own key with zero detection risk. No special capabilities beyond being the coordinator are required.

---

### Recommendation

Add a protocol-level broadcast and consensus step for `app_pk` at the start of `do_ckd_coordinator` / `do_ckd_participant`, analogous to how `session_id` is broadcast in `do_keyshare`: [6](#0-5) 

Each participant should independently broadcast the `app_pk` they were given, and every party should abort if any mismatch is detected. Alternatively, `app_pk` should be cryptographically bound to `app_id` (e.g., via a certificate or a registry entry signed by a trusted authority) so that participants can verify it independently before contributing their share.

---

### Proof of Concept

**Setup**: 3-of-3 CKD with participants P1, P2, and coordinator C (malicious).

1. Legitimate application sends `(app_id, app_pk)` to coordinator C.
2. C generates `(app_sk', app_pk') = (random scalar, app_sk' · G)`.
3. C calls `ckd(&participants, C, C, key_pair_C, app_id, app_pk', rng)` — using `app_pk'`.
4. C tells P1 and P2 to call `ckd(&participants, C, P1/P2, key_pair, app_id, app_pk', rng)` — also using `app_pk'`.
5. All participants compute shares under `app_pk'` and send them to C.
6. C aggregates: `CKDOutput { big_y: Y_total, big_c: C_total }` where `C_total = msk·H(pk‖app_id) + y_total·app_pk'`.
7. C calls `ckd_output.unmask(app_sk')` and recovers `msk · H(pk ‖ app_id)`.

The legitimate application receives either nothing or a corrupted output it cannot unmask. The coordinator has extracted the confidential derived key. [7](#0-6)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
