### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Block Rewards - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract is documented as callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any non-zero address can call it with `disable_rewards: true`, which writes the current block number into the global `last_reward_block` counter **before** the early-return guard, permanently consuming the per-block reward slot without distributing any yield. The sequencer's subsequent call for the same block reverts with `REWARDS_ALREADY_UPDATED`, and the rewards for that block are irrecoverably lost.

---

### Finding Description

The specification at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer.

The implementation, however, only calls `general_prerequisites()`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
    ...
    self.last_reward_block.write(current_block_number);   // ← written unconditionally

    if disable_rewards || self.is_pre_consensus() {
        return;                     // ← exits without distributing rewards
    }
    ...
}
``` [1](#0-0) 

`general_prerequisites` only asserts the contract is unpaused and the caller is non-zero — no sequencer role check exists: [2](#0-1) 

The global `last_reward_block` field is a single slot shared across all stakers: [3](#0-2) 

Because `last_reward_block` is written **before** the `disable_rewards` branch, calling the function with `disable_rewards: true` atomically:

1. Marks block `N` as "already rewarded" in storage.
2. Returns without crediting any staker or pool.

Any subsequent call for block `N` — including the legitimate sequencer call — hits:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [4](#0-3) 

There is no mechanism to reset `last_reward_block`, so the rewards for block `N` are permanently frozen.

The spec confirms the intended per-block reward logic: [5](#0-4) 

---

### Impact Explanation

**Permanent freezing of unclaimed yield.** For every block in which an attacker front-runs the sequencer with `update_rewards(valid_staker, disable_rewards: true)`, the STRK (and BTC) block rewards computed by `calculate_block_rewards` are never credited to any staker or pool. Because `last_reward_block` is a single global counter, one successful front-run per block is sufficient to suppress rewards for **all** stakers and all delegation pools for that block. Sustained over many blocks, this constitutes a permanent, protocol-wide loss of staker yield.

---

### Likelihood Explanation

The function is public and callable by any non-zero address. The only precondition is supplying a valid, active staker address — information that is publicly observable on-chain via `get_stakers`. Front-running on Starknet is feasible because transactions are ordered by the sequencer and an attacker can submit a transaction targeting the same block. No special privilege, leaked key, or external dependency is required.

---

### Recommendation

Add a sequencer-role check at the top of `update_rewards`, analogous to the role guards used elsewhere in the contract (e.g., `self.roles.only_security_agent()`). A dedicated `SEQUENCER` role should be introduced and verified before any state mutation occurs:

```cairo
fn update_rewards(...) {
    self.roles.only_sequencer();   // new guard
    self.general_prerequisites();
    ...
}
```

Alternatively, move the `self.last_reward_block.write(current_block_number)` call to **after** the `disable_rewards` guard so that an early return does not consume the block slot.

---

### Proof of Concept

```
Block N arrives.

1. Attacker observes any active staker address S on-chain.
2. Attacker submits:
       update_rewards(staker_address: S, disable_rewards: true)
   This transaction is included in block N.
3. Inside the call:
   - last_reward_block is written to N.
   - Function returns early; no rewards distributed.
4. Sequencer submits its own update_rewards(S, disable_rewards: false) for block N.
5. The assert `current_block_number > last_reward_block` fails → REWARDS_ALREADY_UPDATED.
6. Block N rewards are permanently lost for all stakers and pools.
7. Attacker repeats for block N+1, N+2, … to freeze all future yield.
```

The analog to the external report: just as `MembershipFactory::upgradeTier` failed to update `tiers.minted`, allowing the tier cap to be bypassed, `Staking::update_rewards` fails to guard the `last_reward_block` write behind the intended access control, allowing the per-block reward slot to be consumed without distributing yield.

            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,

            self.general_prerequisites();

            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,

            let staker_info = self.internal_staker_info(:staker_address);
