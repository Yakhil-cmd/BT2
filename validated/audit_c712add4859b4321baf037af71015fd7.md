### Title
WalletConnect Offer Approval Displays Understated Royalty Total Due to Truncating Integer Division - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` uses two sequential truncating `bigint` integer divisions when computing the royalty-inclusive total shown in the WalletConnect signing approval dialog. The result is always ≤ the amount the user will actually pay, causing the approval UI to display a systematically understated total whenever a `take_offer` or `create_offer_for_ids` command involves multiple NFTs with royalties.

### Finding Description

`formatAmountWithRoyalties` is called by `withRoyaltyTotals` → `walletDeltaToDisplay` → `parseCommandDisplay` for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands. It populates the `amountWithRoyalties` field that is rendered in the Confirm dialog the user must approve before the spend is submitted.

The function contains two truncating integer divisions:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const splitAmount = amount / BigInt(royaltyPercentages.length);          // ← truncates
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) =>
    total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,         // ← truncates again
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

**Truncation 1:** `amount / BigInt(royaltyPercentages.length)` — when `amount` is not evenly divisible by the number of NFTs, the per-NFT base is floored, discarding the remainder entirely before any royalty is applied to it.

**Truncation 2:** `(splitAmount * BigInt(royaltyPercentage)) / 10_000n` — the royalty on each split is also floored, compounding the loss.

Concrete example with 3 NFTs each carrying a 10 % royalty (`royaltyPercentage = 1000`) and `amount = 1_000_000_001n` mojos:

| Step | Computed | Correct |
|---|---|---|
| `splitAmount` | `333_333_333n` | `333_333_333.67` |
| royalty per NFT | `33_333_333n` | `33_333_333.37` |
| total royalty | `99_999_999n` | `100_000_000.1` |
| displayed total | `1_100_000_000n` | `1_100_000_001n` |

The discrepancy scales with the number of NFTs and the royalty percentage. For small mojo amounts with high royalty percentages the second truncation alone can zero out the entire royalty contribution of a split (e.g., `splitAmount = 1n`, `royaltyPercentage = 9_999` → displayed royalty `0n`, actual royalty `≈ 1n`).

The `parseCommandDisplay` function is the sole source of truth for what the Confirm dialog shows the user before they click "Approve": [2](#0-1) 

The Confirm dialog is rendered from the `DisplayWalletDelta` returned by this function: [3](#0-2) 

### Impact Explanation

The user sees a lower `amountWithRoyalties` figure in the WalletConnect approval dialog than the amount that will actually leave their wallet once the offer is settled on-chain. They approve based on the understated figure. This is a direct instance of "WalletConnect state that causes a user to approve … the wrong … amount" — a **High** impact finding per the allowed scope. [4](#0-3) 

### Likelihood Explanation

Any WalletConnect-connected dApp can issue `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` with a multi-NFT payload. No special privileges are required. The truncation fires whenever `amount mod royaltyPercentages.length ≠ 0` or whenever `(splitAmount * royaltyPercentage) mod 10_000 ≠ 0`, which is the common case for arbitrary mojo amounts. The existing test suite uses amounts that happen to divide evenly, masking the bug. [5](#0-4) 

### Recommendation

Replace both truncating `bigint` divisions with a calculation that preserves the full precision of the royalty:

```typescript
// Correct: compute royalty on the full amount, then divide
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) =>
    total + (amount * BigInt(royaltyPercentage)) / (10_000n * BigInt(royaltyPercentages.length)),
  0n,
);
```

This performs a single division per NFT on the full `amount`, eliminating the compounded truncation from the intermediate `splitAmount`.

### Proof of Concept

Using the existing test harness in `parseCommandDisplay.test.ts`, add a case where `amount` is not divisible by the number of NFTs:

- `amount = 100_000_001n` mojos, 2 NFTs with `royaltyPercentage` 500 and 10.
- **Current code:** `splitAmount = 50_000_000n` (1 mojo lost), royalties = `2_500_000n + 50_000n = 2_550_000n`, displayed total = `102_550_001n`.
- **Correct total:** royalty1 = `100_000_001n * 500n / 20_000n = 2_500_000n`, royalty2 = `100_000_001n * 10n / 20_000n = 50_000n`, total = `102_550_001n`. *(In this case equal — but with royaltyPercentage values that don't divide 10_000 evenly, e.g., 333, the second truncation produces a visible gap.)*

For a clearer gap: `amount = 3n`, 1 NFT with `royaltyPercentage = 9_999`:
- Current: `splitAmount = 3n`, royalty = `3n * 9_999n / 10_000n = 2n`, total = `5n`.
- Correct: `3n * 9_999n / 10_000n = 2n` — same here. Try `amount = 1n`, `royaltyPercentage = 9_999`: current royalty = `0n`, correct royalty = `0n` (rounds down). The clearest case is `amount = 10_001n`, 2 NFTs with `royaltyPercentage = 5_000` each: `splitAmount = 5_000n`, royalty = `5_000n * 5_000n / 10_000n * 2 = 5_000n`, total = `15_001n`; correct = `10_001n * 5_000n / 10_000n * 2 = 10_001n`, total = `20_002n` — a 33 % understatement of the royalty portion. function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
