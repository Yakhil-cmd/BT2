### Title
Permissionless `deploy_erc20_token` Combined with Unguarded `ft_on_transfer` Enables Unbacked ERC-20 Minting - (`engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` carries no caller access control beyond a liveness check. Any NEAR account can register an arbitrary NEP-141 account ID as an ERC-20 mirror on Aurora. Because `ft_on_transfer` also imposes no restriction on who may call it, an attacker who controls a NEAR account can register that account as a NEP-141 token and then directly invoke `ft_on_transfer` from it with an arbitrary `amount`, causing Aurora to mint ERC-20 tokens that are not backed by any real NEP-141 deposit. This is a bridge-accounting bug that produces unbacked ERC-20 supply and can lead to insolvency of the bridge.

---

### Finding Description

**Step 1 – Permissionless registration.**

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` checks only `require_running`; there is no `require_owner_only` or any allowlist guard:

```rust

    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // ← only guard
        ...
        DeployErc20TokenArgs::Legacy(nep141) => {
            let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;
``` [1](#0-0) 

The comment at line 148–149 ("this transaction could be executed by the owner of the contract only") is factually wrong; no such check exists in the function. [2](#0-1) 

The inner `engine::deploy_erc20_token` deploys an `EvmErc20` contract and writes the `nep141 → ERC-20 address` mapping unconditionally:

```rust
engine.register_token(address, nep141)
    .map_err(DeployErc20Error::Register)?;
``` [3](#0-2) 

`register_token` only rejects a second registration for the same NEP-141 ID; it does not verify that the NEP-141 account exists or is legitimate. [4](#0-3) 

**Step 2 – Unguarded `ft_on_transfer`.**

`ft_on_transfer` is the public NEAR entry point that mints ERC-20 tokens. It distinguishes ETH-connector calls from NEP-141 calls solely by comparing `predecessor_account_id` to the stored connector account ID. For any other predecessor it calls `receive_erc20_tokens`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id, &args, &current_account_id, handler,
    )
};
``` [5](#0-4) 

`receive_erc20_tokens` looks up the registered ERC-20 for the caller's account ID and mints `args.amount` tokens to the address encoded in `args.msg`, with no verification that any NEP-141 tokens were actually transferred:

```rust
let erc20_token = get_erc20_from_nep141(&self.io, token)?;
let erc20_admin_address = current_address(current_account_id);
let result = self.call(
    &erc20_admin_address, &erc20_token, Wei::zero(),
    setup_receive_erc20_tokens_input(&recipient, amount), ...
``` [6](#0-5) 

**Step 3 – Wrong default metadata permanently locked in.**

When the `Legacy` variant is used, `metadata` is `None`, so `setup_deploy_erc20_input` falls back to `Erc20Metadata::default()` which is `{ name: "Empty", symbol: "EMPTY", decimals: 0 }`: [7](#0-6) 

Because `register_token` rejects double-registration, a front-runner who registers a legitimate token ID (e.g., `usdc.near`) first with the `Legacy` path permanently locks in `decimals = 0` for that token's ERC-20 mirror. The owner can later call `set_erc20_metadata` to fix the on-chain Solidity metadata, but the NEP-141 → ERC-20 address mapping is immutable.

---

### Impact Explanation

Two concrete impacts exist:

1. **Bridge insolvency via unbacked minting.** An attacker who controls NEAR account `evil.near` registers it via `deploy_erc20_token` (permissionless), then calls `ft_on_transfer` directly from `evil.near` with `amount = U128::MAX` and `msg = attacker_evm_address`. Aurora mints `U128::MAX` ERC-20 tokens for `evil.near` with zero NEP-141 backing. If any on-Aurora DEX or lending protocol holds liquidity denominated in that ERC-20, the attacker can drain it, creating insolvency in the bridge's accounting.

2. **Permanent corruption of a legitimate token's ERC-20 mirror.** An attacker front-runs the registration of a high-value token (e.g., `usdc.near`) using the `Legacy` path, locking in `decimals = 0`. Every subsequent user who bridges USDC receives an ERC-20 with the wrong decimal precision. DeFi protocols that read `decimals()` from the ERC-20 contract will miscalculate amounts by up to 10^18, causing fund loss for users of those protocols.

Both impacts fall within the allowed scope: insolvency and direct theft of user funds.

---

### Likelihood Explanation

`deploy_erc20_token` is a public NEAR contract method callable by any NEAR account with no deposit or stake requirement. The attack requires only: (a) a NEAR account the attacker controls, and (b) a single transaction to register it, followed by a second transaction to call `ft_on_transfer`. No privileged access, leaked keys, or governance capture is needed. Front-running a legitimate token registration is also straightforward on NEAR because transaction ordering within a block is observable.

---

### Recommendation

1. Add an allowlist or `require_owner_only` guard to `deploy_erc20_token` so that only trusted callers (e.g., the Aurora DAO or a curated registry) can register new NEP-141 → ERC-20 mappings.
2. In `ft_on_transfer`, verify that the `predecessor_account_id` is a registered NEP-141 account before proceeding to mint. Reject calls from accounts that are not in the registry.
3. Alternatively, require that `deploy_erc20_token` be called with the `WithMetadata` variant only, which at least forces a cross-contract call to the NEP-141 contract's `ft_metadata` method, providing a weak liveness check on the target account.

# 3. Attacker calls ft_on_transfer directly from evil.near

  --accountId evil.near

#   predecessor_account_id = "evil.near" ≠ eth_connector → receive_erc20_tokens

#   calls EvmErc20(0xABCD...).mint(attacker_evm_address, 1_000_000_000_000)

# Result: attacker holds 1_000_000_000_000 unbacked ERC-20 tokens on Aurora
#         with zero NEP-141 tokens ever deposited.
