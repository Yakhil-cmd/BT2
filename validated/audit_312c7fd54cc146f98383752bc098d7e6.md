### Title
WalletConnect NFT Offer Approval Displays Understated Total Cost Due to Incorrect Multi-NFT Royalty Calculation - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount equally among all NFTs before computing each royalty, rather than applying each NFT's royalty percentage to the full amount. When a WalletConnect dApp sends a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command involving multiple NFTs with royalties, the confirmation dialog shows the user an understated `amountWithRoyalties` total. The user approves believing they will spend less than the transaction actually charges.

### Finding Description

`formatAmountWithRoyalties` computes the displayed total cost (base amount + royalties) shown in the WalletConnect `Confirm` dialog:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  ...
  const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← BUG
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
``` [1](#0-0) 

The function first divides `amount` by the number of NFTs (`splitAmount = amount / N`), then sums `splitAmount * royaltyPct / 10_000` for each NFT. This is mathematically equivalent to `amount * sum(royaltyPcts) / (N * 10_000)` — dividing the true royalty total by N.

**Correct formula:** `royaltyAmount = Σ (amount × royaltyPct_i / 10_000)`

**Actual formula used:** `royaltyAmount = Σ ((amount / N) × royaltyPct_i / 10_000)`

The result `amountWithRoyalties` is then embedded in the `DisplayWalletDelta` returned by `walletDeltaToDisplay` and rendered in the `Confirm` dialog's `WalletDeltaSection`: [2](#0-1) [3](#0-2) 

The existing test suite confirms the buggy output is the expected value:

```
// Two NFTs: royalty 500 (5%) and 10 (0.1%), amount = 100,000,000 mojos
amountWithRoyalties: '0.00010255'   // displayed (WRONG)
// Correct value: 0.0001051
``` [4](#0-3) 

### Impact Explanation

A user approving a WalletConnect `take_offer` or `create_offer_for_ids` command involving N NFTs with royalties sees a total cost understated by a factor of N in the royalty component. With N=2 and royalties of 5% + 0.1%, the displayed total is 0.00010255 XCH but the actual on-chain deduction is 0.0001051 XCH — a ~2.4% understatement that scales with N and royalty magnitudes. With 10 NFTs each carrying 10% royalty, the displayed royalty cost would be 10× lower than reality, potentially doubling the user's actual spend relative to what they approved.

This fits the High impact category: WalletConnect state causes a user to approve the wrong amount.

### Likelihood Explanation

- Requires a WalletConnect-connected dApp to issue a multi-NFT offer command (a supported, documented WalletConnect flow).
- Multi-NFT offers are a standard feature of the Chia offer protocol.
- No special privileges are required; any paired dApp can trigger this.
- The discrepancy is invisible to the user — the dialog shows a plausible-looking number with no warning.

### Recommendation

Replace the split-then-multiply formula with the correct per-NFT full-amount calculation:

```typescript
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [5](#0-4) 

### Proof of Concept

Given a `chia_wallet.take_offer` WalletConnect request where the offer summary contains:
- `requested.xch = '100000000'` (0.0001 XCH)
- Two offered NFTs with `royalty_percentage: 500` (5%) and `royalty_percentage: 10` (0.1%)

**Displayed** `amountWithRoyalties` (buggy): `0.00010255 XCH`
- `splitAmount = 100000000 / 2 = 50000000`
- royalties = `(50000000×500/10000) + (50000000×10/10000)` = `2500000 + 50000 = 2550000`
- total = `102550000` mojos = `0.00010255 XCH`

**Actual on-chain cost** (correct): `0.0001051 XCH`
- royalties = `(100000000×500/10000) + (100000000×10/10000)` = `5000000 + 100000 = 5100000`
- total = `105100000` mojos = `0.0001051 XCH`

The user approves seeing `0.00010255 XCH` but the transaction deducts `0.0001051 XCH`. The test at line 310–334 of `parseCommandDisplay.test.ts` encodes the buggy value as the expected output, confirming the defect is present and untested-against-correct-behavior. [1](#0-0) [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
