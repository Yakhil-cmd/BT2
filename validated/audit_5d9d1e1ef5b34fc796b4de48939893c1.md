### Title
Unrestricted `update_rewards` with `disable_rewards=true` Allows Any Caller to Permanently Freeze Consensus Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is publicly callable with no access control. When called with `disable_rewards=true`, it advances the global `last_reward_block` checkpoint without distributing any rewards. Because the contract enforces a strict one-call-per-block invariant on this checkpoint, any unprivileged caller can permanently prevent all stakers from receiving consensus block rewards by calling this function in every block.

### Finding Description
`IStakingRewardsManager.update_rewards` is a public, permissionless entry point:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
``` [1](#0-0) 

The function enforces a single-call-per-block invariant by checking and writing `last_reward_block`:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
...
self.last_reward_block.write(current_block_number);
``` [2](#0-1) 

After writing `last_reward_block`, if `disable_rewards` is `true`, the function returns immediately with no rewards distributed:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [3](#0-2) 

There is no role check, no `assert_caller_is_attestation_contract`, and no other access guard before the `last_reward_block` write. The only preconditions are:
1. The contract is not paused (`general_prerequisites`).
2. The supplied `staker_address` is a currently active staker with non-zero STRK balance.

Both conditions are trivially satisfiable by any observer: active staker addresses are emitted in `NewStaker` events and are permanently public.

The interface declaration confirms no restriction is documented or enforced:

```cairo
pub trait IStakingRewardsManager<TContractState> {
    fn update_rewards(
        ref self: TContractState, staker_address: ContractAddress, disable_rewards: bool,
    );
}
``` [4](#0-3) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

Consensus block rewards are distributed exactly once per block via `update_rewards`. If an attacker calls `update_rewards(any_valid_staker, disable_rewards=true)` in every block, `last_reward_block` is advanced to the current block each time. Every subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`. Because the reward for a given block is never accumulated or carried forward, those rewards are permanently lost. Sustained over time, this freezes 100% of consensus-phase yield for all stakers and delegators.

### Likelihood Explanation
**Medium-High.** The attack requires:
- Knowledge of one valid staker address (trivially obtained from `NewStaker` events).
- One transaction per block (low gas cost on Starknet).
- No special role, no stake, no privileged key.

A motivated attacker (e.g., a competing validator, a protocol saboteur) can sustain this indefinitely. Front-running the legitimate `update_rewards` call is not even necessary — calling it at any point in the block before the legitimate caller suffices.

### Recommendation
Restrict `update_rewards` to a trusted caller. The natural choice is the attestation contract, which already has a privileged path into the staking contract:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.assert_caller_is_attestation_contract(); // add this guard
    ...
}
```

Alternatively, remove the `disable_rewards` parameter entirely and handle the "no-reward" case through a separate privileged administrative function, keeping the public surface minimal.

### Proof of Concept
1. Attacker reads any `NewStaker` event to obtain `valid_staker_address`.
2. In every Starknet block, attacker submits: `staking.update_rewards(valid_staker_address, disable_rewards=true)`.
3. `last_reward_block` is set to the current block number; no rewards are distributed.
4. Any legitimate block proposer or staker calling `update_rewards(staker, disable_rewards=false)` in the same block receives `REWARDS_ALREADY_UPDATED` and reverts.
5. All consensus-phase block rewards for that block are permanently lost.
6. Repeated every block → all stakers and delegators receive zero consensus rewards indefinitely.
