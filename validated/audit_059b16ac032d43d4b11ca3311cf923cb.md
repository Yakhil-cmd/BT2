### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees via `setPoolBinAdditionalFees`, Bypassing the Protocol Fee Cap System — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` passes `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound validation. The analogous function `setPoolAdminFees` enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` caps, but no equivalent cap exists for per-bin additional fees. A pool admin can set `addFeeBuyE6 = type(uint16).max` (65535, ~6.55%) on any active bin, and that value is added unchecked to `baseFeeX64` in every swap through that bin.

---

### Finding Description

**Factory entry point — no cap check:** [1](#0-0) 

The function only enforces `onlyPoolAdmin(pool)`. There is no `if (addFeeBuyE6 > maxAdminBinFeeE6) revert` guard.

**Contrast with `setPoolAdminFees` — caps are enforced there:** [2](#0-1) 

**Pool-side `setBinAdditionalFees` — only validates bin index, not fee magnitude:** [3](#0-2) 

**Swap math consumes `addFeeBuyE6` directly, unclamped:** [4](#0-3) 

`Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)` with `addFeeBuyE6 = 65535` adds ~6.55% to `baseFeeX64` before the swap executes. The same pattern applies to `addFeeSellE6` in the sell direction. [5](#0-4) 

---

### Impact Explanation

Traders swapping through the affected bin pay `baseFee + 6.55%` in spread, on top of the notional fee. The excess fee is collected into the bin's LP balance — the pool admin does not receive it directly, but can coordinate with LP positions to extract it. More directly, traders receive materially worse execution than the protocol's cap system is designed to guarantee. This is a direct loss of trader principal above any Sherlock-relevant threshold for a single large swap.

---

### Likelihood Explanation

The pool admin is a per-pool role, explicitly described as "semi-trusted only inside caps." The attack requires only one `setPoolBinAdditionalFees` call from the pool admin — no timelock, no co-signer, no oracle manipulation. The call succeeds immediately and affects every subsequent swap through that bin until reset.

---

### Recommendation

Add a per-bin additional fee cap in `setPoolBinAdditionalFees`, mirroring the pattern used in `setPoolAdminFees`:

```solidity
uint16 public maxAdminBinAdditionalFeeE6; // e.g. 10_000 = 1%

function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

---

### Proof of Concept

1. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, type(uint16).max, 0)` — succeeds with no revert.
2. `BinState.addFeeBuyE6` for bin 0 is now `65535`.
3. A trader calls `swap` routing through bin 0 to buy token0.
4. The pool computes `currBinBuyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately `baseFee + 6.55%`.
5. The trader's effective cost is `baseFee + 6.55% + notionalFee`, far exceeding any cap enforced by `maxAdminSpreadFeeE6`.
6. Assert: effective fee > `maxAdminSpreadFeeE6 + maxProtocolSpreadFeeE6` (the documented total spread cap). [1](#0-0) [6](#0-5) [4](#0-3)
