## Full Trace

**Step 1 ŌĆö Runner forwards P2P tx to gateway**

When a transaction batch arrives from the network, the runner attaches `message_metadata: Some(broadcasted_message_metadata)` and calls `gateway_client.add_tx`. [1](#0-0) 

**Step 2 ŌĆö Gateway component handler wraps every error with `p2p_message_metadata`**

In `communication.rs`, the `handle_request` implementation extracts `p2p_message_metadata` from the input **before** calling `add_tx`, then maps **every** `StarknetError` ŌĆö regardless of its error code ŌĆö into `GatewayError::DeprecatedGatewayError { source, p2p_message_metadata }`: [2](#0-1) 

For a P2P transaction, `p2p_message_metadata` is always `Some(...)`, so the resulting `GatewayError` always carries `p2p_message_metadata: Some(...)`.

**Step 3 ŌĆö Gateway `add_tx_inner` can return `InternalError` for sequencer-side faults on a valid tx**

Multiple internal failure paths in `add_tx_inner` produce `StarknetError` with `code: UnknownErrorCode("StarknetErrorCode.InternalError")` for reasons entirely unrelated to the transaction's validity:

- `TransactionConverterError::ClassNotFound` / `ClassManagerClientError` / `ProofManagerClientError` ŌåÆ `StarknetError::internal_with_logging` [3](#0-2) 
- `StatefulTransactionValidatorFactory::instantiate_validator` failure ŌåÆ `InternalError` [4](#0-3) 
- `extract_state_nonce_and_run_validations` returning `InternalError` (e.g., state-sync unavailable) [5](#0-4) 
- `store_proof_and_spawn_archiving` / `await_proof_archiving` failures ŌåÆ `StarknetError::internal_with_logging` [6](#0-5) 
- `MempoolClientError::ClientError` ŌåÆ `StarknetError::internal_with_signature_logging` [7](#0-6) 

The `internal_error_code()` is confirmed as `UnknownErrorCode("StarknetErrorCode.InternalError")`: [8](#0-7) 

**Step 4 ŌĆö Runner's error handler has no error-code discrimination**

The runner matches on `DeprecatedGatewayError` using `..` to ignore the `source` field entirely. The only condition checked is `p2p_message_metadata: Some(...)`. If that is present, `report_peer` is called unconditionally: [9](#0-8) 

The TODO comment at lines 84ŌĆō85 is the **only** guard and explicitly acknowledges the missing discrimination.

**Step 5 ŌĆö `report_peer` leads to peer blacklisting**

`report_peer` in the network peer manager can blacklist the peer (temporarily or permanently depending on accumulated misconduct score): [10](#0-9)

## Verdict

The question's premise is **confirmed by the production code**. The `..` wildcard in the runner's match arm means that a `DeprecatedGatewayError` carrying `InternalError` (from a state-sync failure, class manager failure, proof manager failure, etc.) for a **valid** P2P transaction will trigger `report_peer` on the originating peer. The peer is then subject to blacklisting, causing its future valid transactions to be rejected from admission.

This fits the allowed impact scope: **High ŌĆö Mempool/gateway/RPC admission rejects valid transactions before sequencing**, because a blacklisted peer's subsequent valid transactions are not admitted to the mempool.

### Title
Undiscriminating `DeprecatedGatewayError` handler in `MempoolP2pRunner` blacklists peers for sequencer-internal errors, rejecting future valid P2P transactions ŌĆö (`crates/apollo_mempool_p2p/src/runner/mod.rs`)

### Summary
`MempoolP2pRunner::start` calls `report_peer` for **any** `GatewayError::DeprecatedGatewayError` that carries a `p2p_message_metadata`, without inspecting the `source` error code. Because the gateway component handler wraps every `StarknetError` ŌĆö including `InternalError` codes produced by sequencer-internal faults ŌĆö with the originating peer's metadata, a transient internal failure (state-sync unavailable, class manager error, proof manager error) while processing a valid P2P transaction causes the sending peer to be incorrectly reported and potentially blacklisted.

### Finding Description
In `crates/apollo_gateway/src/communication.rs` lines 24ŌĆō31, the `handle_request` implementation extracts `p2p_message_metadata` before calling `add_tx`, then maps every `StarknetError` to `GatewayError::DeprecatedGatewayError { source, p2p_message_metadata }`. For P2P transactions, `p2p_message_metadata` is always `Some(...)`.

In `crates/apollo_mempool_p2p/src/runner/mod.rs` lines 86ŌĆō94, the runner matches this error with `GatewayError::DeprecatedGatewayError{p2p_message_metadata: Some(p2p_message_metadata), ..}` ŌĆö the `..` wildcard ignores `source` entirely. Any `DeprecatedGatewayError` with a non-`None` metadata triggers `report_peer`, including those with `code: UnknownErrorCode("StarknetErrorCode.InternalError")`.

The TODO comment at lines 84ŌĆō85 is the only acknowledgment of this gap and provides no runtime protection.

### Impact Explanation
A legitimate peer sending a well-formed transaction during a period of sequencer-internal instability (e.g., state-sync temporarily unavailable, class manager RPC failure) will be reported to the network peer manager. Depending on accumulated misconduct score and the `ReputationModifier` used, the peer may be blacklisted for `malicious_timeout_seconds` or `unstable_timeout_millis`. During the blacklist period, all future valid transactions from that peer are silently dropped at the network layer before reaching the gateway ŌĆö a concrete admission rejection of valid transactions before sequencing.

### Likelihood Explanation
Any transient internal sequencer failure (state-sync hiccup, class manager restart, proof manager timeout) during P2P transaction processing is sufficient. No attacker action is required; the trigger is a normal operational event. The peer sending the transaction need not do anything wrong.

### Recommendation
In the runner's error handler, inspect `source.is_internal()` (which is already implemented on `StarknetError`) before calling `report_peer`. Only report the peer when the error code indicates a transaction-level fault (e.g., `DuplicatedTransaction`, `InvalidTransactionNonce`, `ValidateFailure`), not when it is an `InternalError`. The TODO comment at lines 84ŌĆō85 already identifies the correct fix direction: widen `GatewayError`'s variants to distinguish transaction faults from internal errors, or add error-code discrimination at the runner level.

async fn incoming_p2p_tx_fails_on_gateway_client() {
    let network_future = pending().boxed();
    // Create channels for sending an empty message to indicate that the tx reached the gateway
    // client.
    let (add_tx_indicator_sender, add_tx_indicator_receiver) = futures::channel::oneshot::channel();

    let message_metadata = BroadcastedMessageMetadata::get_test_instance(&mut get_rng());
    let message_metadata_clone = message_metadata.clone();
    let expected_rpc_transaction_batch =
        RpcTransactionBatch(vec![RpcTransaction::get_test_instance(&mut get_rng())]);

    let mut mock_gateway_client = MockGatewayClient::new();
    #[allow(clippy::result_large_err)]
    mock_gateway_client.expect_add_tx().return_once(move |_| {
        add_tx_indicator_sender.send(()).unwrap();
        Err(GatewayClientError::GatewayError(GatewayError::DeprecatedGatewayError {
            source: StarknetError {
                code: StarknetErrorCode::KnownErrorCode(
                    KnownStarknetErrorCode::DuplicatedTransaction,
                ),
                message: format!("Transaction with hash {} already exists.", Nonce::default()),
            },
            p2p_message_metadata: Some(message_metadata_clone),
        }))
    });

    let (mut mempool_p2p_runner, mock_network) = setup(
        network_future,
        Arc::new(mock_gateway_client),
        Arc::new(MockMempoolP2pPropagatorClient::new()),

        MAX_CONCURRENT_GATEWAY_REQUESTS,
    );

    let BroadcastNetworkMock {
        broadcasted_messages_sender: mut mock_broadcasted_messages_sender,
        reported_messages_receiver: mut mock_reported_messages_receiver,
        ..
    } = mock_network;

    let res = mock_broadcasted_messages_sender
        .send((expected_rpc_transaction_batch.clone(), message_metadata.clone()));

    res.await.expect("Failed to send message");

        // if the runner fails, there was a network issue => panic.
