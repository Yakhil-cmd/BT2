### Title
Unchecked Precompile Call Return Value After `_burn` in `withdrawToNear` Enables Permanent Fund Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

`withdrawToNear` in `EvmErc20.sol` burns the caller's ERC-20 tokens **before** invoking the `ExitToNear` precompile, and never checks the assembly `call` return value. When the precompile rejects the input (e.g., because `recipient.length` causes the total input to exceed `MAX_INPUT_SIZE = 1024`), the sub-call silently returns 0, the burn is not reverted, and no NEP-141 credit is ever scheduled. The tokens are permanently destroyed.

---

### Finding Description

**Step 1 — Burn-before-call ordering**

`withdrawToNear` unconditionally burns the caller's balance first:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← balance reduced here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

**Step 2 — Precompile enforces `MAX_INPUT_SIZE = 1024`**

`parse_input` (called unconditionally inside `ExitToNearParams::try_from`) validates the total input length:

```rust
const MAX_INPUT_SIZE: usize = 1_024;

fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
``` [2](#0-1) [3](#0-2) 

`validate_input_size` returns `Err(ExitError::Other("ERR_INVALID_INPUT"))` when `input.len() > max`: [4](#0-3) 

This error propagates via `?` through `TryFrom` at line 419 of `run`, causing the precompile to return `Err`: [5](#0-4) 

**Step 3 — EVM sub-call failure does not revert the caller**

In SputnikVM (Aurora's EVM), a precompile returning `Err` causes the `call` opcode to push `0` onto the stack. The calling contract's own state changes — including the `_burn` — are **not** rolled back. Since `res` is never read or asserted, `withdrawToNear` returns normally with the ERC-20 balance already zeroed and no NEAR promise scheduled.

---

### Impact Explanation

Any user who calls `withdrawToNear` with `recipient.length >= 992` (making `1 + 32 + 992 = 1025 > 1024`) will have their ERC-20 tokens permanently burned with zero NEP-141 credit issued. The tokens cannot be recovered. This breaks the core accounting invariant: every ERC-20 burn must produce a corresponding NEP-141 transfer.

Impact: **Critical — Permanent freezing / destruction of funds.**

---

### Likelihood Explanation

- `withdrawToNear` is a public, permissionless function callable by any ERC-20 token holder.
- The `recipient` parameter is a raw `bytes` argument with no on-chain length cap enforced by the Solidity contract.
- A user could trigger this accidentally (e.g., passing a long NEAR account ID or a malformed message) or be socially engineered into doing so.
- No admin privilege or special role is required.

---

### Recommendation

1. **Check `res` and revert on failure.** Replace the unchecked assembly block with:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                       0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Enforce the length cap in Solidity before burning**, e.g.:
   ```solidity
   require(1 + 32 + recipient.length <= 1024, "ERR_RECIPIENT_TOO_LONG");
   ```
3. **Invert the order**: validate all inputs before calling `_burn`, so any rejection reverts the entire call with no state change.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IEvmErc20 {
    function withdrawToNear(bytes memory recipient, uint256 amount) external;
    function balanceOf(address account) external view returns (uint256);
}

contract PoC {
    function exploit(address token, uint256 amount) external {
        IEvmErc20 erc20 = IEvmErc20(token);

        uint256 balanceBefore = erc20.balanceOf(address(this));

        // recipient.length = 992 → total input = 1 + 32 + 992 = 1025 > 1024
        // Precompile rejects; res = 0; never checked; burn already committed.
        bytes memory longRecipient = new bytes(992);
        erc20.withdrawToNear(longRecipient, amount);

        uint256 balanceAfter = erc20.balanceOf(address(this));

        // balanceBefore - balanceAfter == amount  (tokens burned)
        // NEP-141 balance on NEAR: unchanged       (no promise scheduled)
        // Funds permanently lost.
        assert(balanceBefore - balanceAfter == amount);
    }
}
```

The fuzz invariant to assert: for any `recipient.length` where `1 + 32 + recipient.length > 1024`, the post-call ERC-20 balance equals `pre - amount` while the NEP-141 balance on NEAR is unchanged.
