### Title
Gateway Skips Signature Verification for Invoke Transactions via `skip_stateful_validations` Bypass, Admitting Invalid Transactions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (signature verification) for any invoke transaction with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` as soon as **any** transaction from the target address is in the pool — including the deploy_account itself — an unprivileged attacker can submit an invoke with a completely invalid signature for any address that has a pending deploy_account, and the gateway will admit it without ever verifying the signature.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain `account_nonce == 0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

**Effect of `skip_validate = true`** [2](#0-1) 

When `skip_validate` is `true`, `execution_flags.validate` is set to `false`. Inside `StatefulValidator::perform_validations`, the `__validate__` call is then short-circuited: [3](#0-2) 

No signature verification occurs at all.

**The flawed guard — `account_tx_in_pool_or_recent_block`** [4](#0-3) 

This returns `true` if the address appears in the mempool state **or** the transaction pool. The deploy_account transaction itself (nonce=0) is in the transaction pool the moment it is admitted, so `account_tx_in_pool_or_recent_block` immediately returns `true` for the deploying address. The check does **not** distinguish between a deploy_account and any other transaction type.

**Attack path**

1. Alice submits a `deploy_account` for address X (nonce=0). The gateway admits it; the deploy_account is now in the tx pool.
2. `account_tx_in_pool_or_recent_block(X)` now returns `true`.
3. Attacker calls `add_transaction` with an `Invoke`: `sender_address=X`, `nonce=1`, `signature=[0xdead, 0xbeef]` (arbitrary/invalid).
4. Stateless validation passes — only signature *length* is checked, not cryptographic validity. [5](#0-4) 

5. `validate_state_preconditions` passes — nonce=1 is within `[0, max_allowed_nonce_gap]` for account_nonce=0. [6](#0-5) 

6. `validate_by_mempool` passes — no duplicate tx hash, nonce=1 is not yet in the pool. [7](#0-6) 

7. `skip_stateful_validations` returns `true` (conditions met). `run_validate_entry_point` is called with `validate=false`; `__validate__` is never invoked. [8](#0-7) 

8. The attacker's invoke with an invalid signature is admitted to the mempool.
9. Alice's legitimate invoke with nonce=1 is now rejected as `DuplicateNonce` (or requires fee escalation to displace the attacker's transaction).

**No existing guard catches this.** The mempool's `validate_tx` checks only for duplicate hashes and nonce ordering, not signature validity. [9](#0-8) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway's invariant that every admitted transaction carries a valid account signature is broken for the specific case of nonce=1 invoke transactions targeting an address with a pending deploy_account. The attacker's transaction will revert at execution time (the batcher does call `__validate__`), but it occupies the nonce=1 slot in the mempool, disrupting the deploy_account + invoke UX flow for legitimate users and forcing fee escalation.

---

### Likelihood Explanation

**Medium.** The attacker requires no special privileges — only the ability to call `add_transaction` on the public gateway API and to observe the mempool (or simply race after submitting the deploy_account themselves). The attack window is the time between the deploy_account being admitted and the account owner submitting their legitimate invoke.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists in the mempool for the target address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that inspects transaction types, rather than returning `true` for any transaction from the address.

Alternatively, require that the invoke transaction's signature is validated against the account class declared in the pending deploy_account (the class hash and constructor calldata are known at gateway time), so that even the UX-skip path performs at least a lightweight signature pre-check.

---

### Proof of Concept

```
1. Alice → gateway: add_transaction(DeployAccount { sender=X, nonce=0, class_hash=C, sig=valid })
   → admitted; deploy_account in tx_pool for address X

2. Attacker → gateway: add_transaction(Invoke { sender=X, nonce=1, calldata=[...], sig=[0xdead,0xbeef] })

   Stateless:  signature length ≤ max_signature_length  → OK
   Stateful:
     account_nonce = state.get_nonce(X) = 0
     validate_state_preconditions:
       nonce=1 in [0, max_allowed_nonce_gap]            → OK
     validate_by_mempool:
       no duplicate hash, nonce=1 not in pool           → OK
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0
       && account_tx_in_pool_or_recent_block(X)==true   → returns true (skip!)
     run_validate_entry_point:
       execution_flags.validate = false
       __validate__ NOT called                          → Ok(())

   → Attacker's invoke admitted to mempool with invalid signature.

3. Alice → gateway: add_transaction(Invoke { sender=X, nonce=1, sig=valid })
   → MempoolError::DuplicateNonce  (or requires fee escalation)

4. Batcher executes attacker's invoke → __validate__ called → fails (invalid sig) → revert.
   Alice's UX is disrupted; she must resubmit with higher fee.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
