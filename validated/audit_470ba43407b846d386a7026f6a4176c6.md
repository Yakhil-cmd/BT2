### Title
`EVM.mustRun()` Silently Passes `Status.failed` EVM Transactions Without Reverting - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` in `fvm/evm/stdlib/contract.cdc` is designed to be a "safe" wrapper around `EVM.run()`, but it explicitly allows EVM transactions with `Status.failed` (execution reverted, out-of-gas, etc.) to pass through without reverting the Cadence transaction. Any Cadence code that calls `EVM.mustRun()` for a token transfer and does not additionally assert `runResult.status == Status.successful` will silently proceed as if the EVM transfer succeeded — a direct analog to the unchecked `transferFrom` return value pattern.

---

### Finding Description

`EVM.mustRun()` is defined as:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
``` [1](#0-0) 

The assert condition `runResult.status == Status.failed || runResult.status == Status.successful` only rejects `Status.invalid` and `Status.unknown`. It explicitly passes `Status.failed` — which covers all EVM-level execution failures including reverted EVM calls (e.g., a `transferFrom` that returns `false` and reverts, an out-of-gas error, or any EVM revert). The function name `mustRun` implies a guarantee of success, but it provides no such guarantee.

The three EVM result statuses are:

```go
StatusInvalid Status = 1   // rejected before execution
StatusFailed  Status = 2   // executed but EVM returned error
StatusSuccessful Status = 3 // executed and succeeded
``` [2](#0-1) 

`EVM.run()` itself is also publicly accessible and returns a `Result` whose status is never enforced by the protocol:

```cadence
access(all)
fun run(tx: [UInt8], coinbase: EVMAddress): Result {
    pre {
        !self.isPaused(): "EVM operations are temporarily paused"
    }
    return InternalEVM.run(tx: tx, coinbase: coinbase.bytes) as! Result
}
``` [3](#0-2) 

---

### Impact Explanation

Any unprivileged Cadence transaction that:
1. Calls `EVM.mustRun()` (or `EVM.run()`) to execute an EVM token transfer (e.g., `transferFrom`, `transfer`)
2. Does not explicitly assert `runResult.status == Status.successful` after the call
3. Then performs Cadence-side state changes (e.g., releasing escrowed tokens, minting Cadence-side assets, updating accounting)

…will silently proceed even when the EVM transfer failed. This maps directly to **cross-VM asset loss** and **bridge escrow mis-accounting**: the Cadence side believes the EVM transfer succeeded and releases or mints assets, while no tokens actually moved on the EVM side.

---

### Likelihood Explanation

`EVM.mustRun()` is a public protocol function callable by any transaction sender. Its misleading name (`mustRun` implies mandatory success) creates a natural trap for Cadence developers writing cross-VM logic. The comment in the source explicitly acknowledges the behavior but does not enforce a check on callers. Any Cadence transaction or contract that uses `mustRun()` for a token transfer without a follow-up status assertion is vulnerable. The entry path requires no special privileges — only the ability to submit a Cadence transaction.

---

### Recommendation

1. **Rename or redesign `mustRun()`**: Either rename it to `runAllowingFailure()` to make the semantics explicit, or add a second variant `mustRunSuccessful()` that asserts `runResult.status == Status.successful`.
2. **Add a protocol-level guard**: Provide a helper that panics on any non-successful status:
   ```cadence
   fun mustRunSuccessful(tx: [UInt8], coinbase: EVMAddress): Result {
       let runResult = self.run(tx: tx, coinbase: coinbase)
       assert(
           runResult.status == Status.successful,
           message: "EVM.mustRunSuccessful(): EVM transaction did not succeed"
       )
       return runResult
   }
   ```
3. **Document clearly**: All callers of `EVM.run()` and `EVM.mustRun()` that perform token transfers must explicitly check `runResult.status == Status.successful` before taking any Cadence-side action.

### Proof of Concept

```cadence
import EVM from <EVMAddress>
import FungibleToken from <FTAddress>

transaction(rlpEncodedTransferFrom: [UInt8], coinbaseBytes: [UInt8; 20]) {
    prepare(account: auth(Storage) &Account) {
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)

        // mustRun does NOT revert if the EVM transferFrom fails (Status.failed)
        let result = EVM.mustRun(tx: rlpEncodedTransferFrom, coinbase: coinbase)

        // result.status may be Status.failed here — no revert occurred
        // Any Cadence-side asset release below proceeds regardless:
        // e.g., release escrowed tokens, mint Cadence-side wrapped tokens, etc.
        // → Cross-VM asset loss: EVM transfer failed, Cadence side proceeds as if it succeeded
    }
}
```

The `mustRun()` assert passes for `Status.failed`:

```cadence
assert(
    runResult.status == Status.failed || runResult.status == Status.successful,
    // ^^^ Status.failed satisfies this condition — no revert
    message: "EVM.mustRun(): The provided transaction is not valid for execution"
)
