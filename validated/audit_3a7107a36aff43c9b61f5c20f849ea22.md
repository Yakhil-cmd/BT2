### Title
Unauthorized Cross-Account Balance Burn via `DeployGlobalContractAction` — (`runtime/runtime/src/global_contracts.rs`)

---

### Summary

`action_deploy_global_contract` permanently burns `global_contract_storage_amount_per_byte * code.len()` from the **receipt receiver's** account. Because no authorization check enforces that the receipt must be a self-receipt (`predecessor_id == receiver_id`), any contract can create a cross-account promise receipt targeting an arbitrary victim account B with `DeployGlobalContractAction`, causing B's balance to be permanently drained without B's consent.

---

### Finding Description

**Step 1 — The burn happens on the receiver's account.**

In `action_deploy_global_contract`:

```rust
let storage_cost = apply_state.config.fees.storage_usage_config
    .global_contract_storage_amount_per_byte
    .saturating_mul(deploy_contract.code.len() as u128);
let Some(updated_balance) = account.amount().checked_sub(storage_cost) else { ... };
result.tokens_burnt = result.tokens_burnt.checked_add(storage_cost)...;
account.set_amount(updated_balance);
``` [1](#0-0) 

The `account` here is the **receipt receiver's** account, loaded at `apply_action_receipt` line 824 as `get_account(state_update, account_id)` where `account_id = receipt.receiver_id()`. [2](#0-1) 

**Step 2 — No authorization check exists.**

The dispatch in `apply_action` for `Action::DeployGlobalContract` simply calls `action_deploy_global_contract` with the receiver's account and no predecessor/receiver equality check: [3](#0-2) 

A search of `runtime/runtime/src/verifier.rs` for `DeployGlobalContract` returns **zero matches** — there is no validation rule restricting this action to self-receipts.

**Step 3 — A contract can target any account.**

The VM host function `promise_batch_action_deploy_global_contract_impl` only checks: view-mode prohibition, code size limit, and gas fees. It does not check that the promise's receiver equals the current account: [4](#0-3) [5](#0-4) 

A malicious contract calls `promise_batch_create("victim_B")` and appends `DeployGlobalContractAction` with a max-size code blob. The resulting receipt has `predecessor_id = attacker_contract`, `receiver_id = victim_B`. When it executes on B's shard, B's balance is permanently reduced.

**Step 4 — The burn is permanent, not a stake.**

Unlike `DeployContract` (which increases `storage_usage` and is checked against a staking requirement that can be recovered), the global contract storage cost is an explicit `account.set_amount(updated_balance)` deduction added to `result.tokens_burnt`. It is never refunded. [6](#0-5) 

---

### Impact Explanation

- `global_contract_storage_amount_per_byte = 0.0001 NEAR` and `max_contract_size = 4 MiB = 4,194,304 bytes`.
- Maximum burn per receipt: `0.0001 × 4,194,304 ≈ 419 NEAR` permanently drained from victim B.
- The attack is repeatable (each call creates a new receipt).
- In `AccountId` deploy mode, the global contract is additionally registered under B's account ID, which B cannot revoke. [7](#0-6) 

---

### Likelihood Explanation

Any unprivileged user who can deploy a contract (standard NEAR account with balance) can execute this attack. No validator, relayer, or operator privilege is required. The attacker only needs to pay gas for the cross-contract call; the storage cost is borne entirely by the victim.

---

### Recommendation

Add a check in `action_deploy_global_contract` (or in `validate_actions_with_mode` for `Action::DeployGlobalContract`) that the receipt is a self-receipt:

```rust
if receipt.predecessor_id() != receipt.receiver_id() {
    result.result = Err(ActionErrorKind::ActorNoPermission { ... }.into());
    return Ok(());
}
```

This mirrors the pattern used for other privileged actions (e.g., `Stake`, `AddKey`, `DeleteKey`) which are implicitly restricted to self-receipts by the transaction-to-receipt conversion rules.

### Proof of Concept

```rust
// Malicious contract code (pseudo-Rust NEAR contract):
#[near_bindgen]
impl Malicious {
    pub fn attack(&self, victim: AccountId) -> Promise {
        let code = vec![0u8; 4_194_304]; // max-size blob
        Promise::new(victim)
            .deploy_global_contract(code)  // DeployGlobalContractAction, CodeHash mode
    }
}
```

**Runtime state test plan:**

1. Initialize runtime with victim account B holding, e.g., 1000 NEAR.
2. Initialize attacker account A with a deployed malicious contract.
3. Submit a transaction from A calling `attack(B)`.
4. Apply the resulting action receipt on B's shard.
5. Assert:
   - `B.account.amount()` decreased by `global_contract_storage_amount_per_byte * 4_194_304`.
   - `execution_outcome.tokens_burnt` equals that storage cost (plus gas).
   - B never signed or authorized any transaction. [8](#0-7)

    account: &mut Account,

    result: &mut ActionResult,

    let storage_cost = apply_state
        .config

        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {

            account_id: account_id.clone(),

        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}
