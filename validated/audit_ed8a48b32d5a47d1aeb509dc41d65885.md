### Title
Unenforced Execution Order of Scheduled Promises Allows Any Caller to Freeze User Funds - (File: `etc/xcc-router/src/lib.rs`)

### Summary
The XCC router's `execute_scheduled` function carries no access control and accepts an arbitrary nonce from any caller. Because the promise is irrevocably removed from storage the moment `execute_scheduled` is called — regardless of whether the promise eventually succeeds or fails — any external account can execute a user's scheduled promises out of order. When a user has scheduled a sequence of logically dependent promises, out-of-order execution causes the dependent promise to fail while its storage entry is permanently deleted, leaving the funds that were supposed to flow through it stranded in the router until the user manually schedules a recovery transaction.

### Finding Description

The XCC router contract stores promises submitted by the Aurora Engine parent under monotonically increasing nonces via `schedule`: [1](#0-0) 

Only the parent (Aurora Engine) may call `schedule`, so the stored promises are legitimate user-initiated cross-contract calls. However, `execute_scheduled` imposes no caller restriction whatsoever: [2](#0-1) 

The critical detail is line 151: `self.scheduled_promises.remove(&nonce.0)` executes unconditionally before the promise is dispatched. In NEAR, a promise's eventual success or failure is asynchronous and does not roll back the storage mutation that already committed. Therefore, once `execute_scheduled(N)` returns without panicking, the entry at nonce `N` is permanently gone from `scheduled_promises`, regardless of whether the dispatched promise later succeeds or fails.

The `CrossContractCallArgs::Delayed` path in the XCC precompile is the entry point that populates these scheduled slots: [3](#0-2) 

A user who needs multiple sequential cross-contract calls (e.g., approve → swap → transfer) schedules them as separate `Delayed` promises, expecting them to be executed in nonce order. Nothing in the protocol enforces that order.

### Impact Explanation

**High — Temporary freezing of funds.**

Concrete scenario:
1. A user schedules nonce 0: `ft_transfer_call` sending 1000 wNEAR to a DEX, expecting to receive token X in the router.
2. The user schedules nonce 1: `ft_transfer` sending token X to an external address.
3. An attacker calls `execute_scheduled(1)` first. The promise is removed from storage and dispatched. It fails because the router holds no token X yet. NEAR returns any attached balance to the router, but the nonce-1 entry is gone.
4. The user calls `execute_scheduled(0)`. It succeeds; the router now holds token X.
5. Token X is stranded in the router. Nonce 1 no longer exists; the user must submit a new EVM transaction through the XCC precompile to schedule a fresh recovery promise, paying additional gas and waiting for the next block.

During the window between step 3 and the user's recovery transaction, the funds are inaccessible. If the user's EVM account lacks sufficient ETH for gas, or the router's NEAR balance is insufficient for a new promise, the freeze extends further.

### Likelihood Explanation

Medium. NEAR transactions are visible on-chain before they are finalized. An attacker monitoring router accounts for `schedule` calls can observe when a user has queued multiple promises and immediately call `execute_scheduled` with a later nonce. The attacker pays only the NEAR gas cost of the call (~5 Tgas) and gains the ability to disrupt any multi-step XCC operation. The attack is especially attractive to competitors who benefit from a user's DeFi operation failing (e.g., preventing a large swap from completing).

### Recommendation

Restrict `execute_scheduled` to the parent account (Aurora Engine), mirroring the access control already applied to `schedule` and `execute`:

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    self.assert_preconditions(); // same guard used by `schedule`
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

Alternatively, if open execution is desired for gas-efficiency reasons, enforce sequential execution by requiring `nonce == current_execution_nonce` and maintaining a separate monotonic execution counter, so promises can only be consumed in the order they were scheduled.

### Proof of Concept

1. Deploy Aurora Engine with XCC router; fund a user's router sub-account with wNEAR.
2. User submits two EVM transactions using `CrossContractCallArgs::Delayed`:
   - Tx A → router stores promise at nonce 0 (e.g., `ft_transfer_call` to a DEX).
   - Tx B → router stores promise at nonce 1 (e.g., `ft_transfer` of the received token).
3. Attacker (any NEAR account) calls:
   ```
   router_account.execute_scheduled({"nonce": "1"})
   ```
   before the user calls `execute_scheduled({"nonce": "0"})`.
4. Observe: nonce-1 promise is dispatched and fails (no token balance); nonce-1 entry is deleted from `scheduled_promises`.
5. User calls `execute_scheduled({"nonce": "0"})`; it succeeds and the router receives the token.
6. Confirm: `scheduled_promises` contains no entry for nonce 1; the received token is stranded in the router with no scheduled path to its intended destination. [2](#0-1) [1](#0-0) [3](#0-2)
