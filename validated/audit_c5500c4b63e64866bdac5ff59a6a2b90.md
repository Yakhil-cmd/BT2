### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any non-allowlisted address to bypass the deposit gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and validates only `owner`. Because `MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` with no requirement that `msg.sender == owner`, any address can pass the allowlist check by nominating an allowlisted address as `owner`.

### Finding Description
`MetricOmmPool.addLiquidity` passes both `msg.sender` and the caller-supplied `owner` to the extension: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both into the call: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` explicitly drops the first (`sender`) parameter — it is unnamed — and checks only `owner`: [3](#0-2) 

The contract's own NatSpec states it "Gates `addLiquidity` by depositor address" and the storage mapping is named `allowedDepositor`, yet the check is on the position-recipient (`owner`), not the depositor (`sender`).

Unlike `removeLiquidity`, which enforces `msg.sender == owner`, `addLiquidity` has no such guard: [4](#0-3) 

### Impact Explanation
A non-allowlisted address B calls `pool.addLiquidity(owner=A, ...)` where A is allowlisted. The extension checks `allowedDepositor[pool][A]` → passes. B pays the tokens (via the modify-liquidity callback), A receives the LP shares. The pool admin's access control is fully bypassed: B, an address the admin explicitly did not allowlist, successfully injects tokens into the pool. This breaks the core invariant of the `DepositAllowlistExtension` — that only allowlisted addresses may deposit — and constitutes a broken core pool functionality (access control bypass) under the contest's allowed impact gate.

### Likelihood Explanation
Exploitable by any EOA or contract with a direct call to `MetricOmmPool.addLiquidity`. No special privileges, flash loans, or oracle manipulation are required. The attacker only needs to know one allowlisted address (publicly readable from `allowedDepositor`).

### Recommendation
Check `sender` (the actual caller/payer) instead of `owner` in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Note: if the router (`MetricOmmPoolLiquidityAdder`) is a valid entry point, the adder itself would need to be allowlisted or the extension would need to accept a user-identity forwarded through `extensionData`. The fix must be consistent with the intended trust model for router-mediated deposits.

### Proof of Concept
```solidity
// Pool configured with DepositAllowlistExtension; only address A is allowlisted.
// address B is NOT allowlisted.

// B calls directly:
pool.addLiquidity(
    /*owner=*/ A,       // allowlisted → extension passes
    salt,
    deltas,
    callbackData,       // B's callback pays the tokens
    extensionData
);

// Result: B paid tokens, A holds LP shares, allowlist was never enforced on B.
assertGt(pool.positionBinShares(A, salt, binIdx), 0); // A has shares
// B was never blocked despite not being allowlisted
```
