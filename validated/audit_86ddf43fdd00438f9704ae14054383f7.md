### Title
Signature Validation Bypass via Fee Escalation on Nonce-1 Invoke Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature verification) for nonce-1 invoke transactions when `account_tx_in_pool_or_recent_block` returns `true`. This check verifies the presence of **any** transaction from the address in the mempool, not specifically a `deploy_account` transaction. Combined with the fee escalation mechanism, an unprivileged attacker can replace a victim's valid nonce-1 invoke transaction with an invalid-signature transaction that bypasses gateway signature validation, causing the victim's transaction to fail during execution and their fee balance to be drained.

---

### Finding Description

**The intended invariant:** Every transaction admitted to the mempool must have a valid signature, verified by the `__validate__` entry point during gateway stateful validation.

**The broken invariant:** `skip_stateful_validations` skips `__validate__` for nonce-1 invoke transactions when `account_tx_in_pool_or_recent_block` returns `true`. The comment in the code states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is circular and incorrect. The check `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the address — including a nonce-1 invoke transaction that the attacker themselves submitted in a prior step. [1](#0-0) 

**Exact attack path:**

**Step 1.** Alice submits a valid nonce-1 invoke transaction for her address (on-chain `account_nonce = 0`). At this point, `account_tx_in_pool_or_recent_block` returns `false` (no tx in pool yet), so `skip_stateful_validations` returns `false`, and the full `__validate__` entry point is executed. Alice's tx passes and enters the mempool. [2](#0-1) 

**Step 2.** Bob submits a nonce-1 invoke transaction with Alice's `sender_address`, a fee ≥10% higher than Alice's (satisfying fee escalation), and an **invalid signature**. The gateway processes it:

- `validate_state_preconditions`: passes — nonce 1 is within `[0, 0 + max_allowed_nonce_gap]`.
- `validate_by_mempool` → `mempool.validate_tx`: passes — fee escalation check only compares `tip` and `max_l2_gas_price`, not signatures.
- `skip_stateful_validations`: `account_tx_in_pool_or_recent_block(Alice)` returns **`true`** because Alice's valid tx is in the pool. The function returns `true` (skip validation).
- `run_validate_entry_point` is called with `skip_validate = true`, setting `execution_flags.validate = false`. The `__validate__` entry point is **never called**. [3](#0-2) 

**Step 3.** Bob's invalid-signature tx is admitted to the mempool and replaces Alice's valid tx via fee escalation. [4](#0-3) 

**Step 4.** The batcher picks up Bob's tx. The blockifier runs `perform_pre_validation_stage` and then calls the `__validate__` entry point. Signature verification fails. The tx is reverted, but the fee is charged from Alice's balance (the `sender_address` in the tx). [5](#0-4) 

The `account_tx_in_pool_or_recent_block` implementation confirms it checks for any transaction, not specifically a `deploy_account`: [6](#0-5) 

Fee escalation is enabled by default in production configuration: [7](#0-6) 

---

### Impact Explanation

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

- The gateway admits a transaction with an invalid signature to the mempool, violating the core admission invariant.
- Any account that has a nonce-1 invoke transaction in the mempool (the common deploy_account + invoke UX flow) is vulnerable.
- The victim's valid transaction is silently replaced by an invalid one that will fail during execution.
- The fee is charged from the victim's balance even on revert, enabling fee-balance draining.
- The attacker pays nothing: the fee is debited from the victim's `sender_address`.

---

### Likelihood Explanation

- No special privileges required; any unprivileged actor can submit transactions to the gateway.
- The attack window is the period between Alice's nonce-1 invoke tx entering the mempool and being sequenced — a common scenario in the deploy_account + invoke UX flow.
- Fee escalation is enabled by default (`enable_fee_escalation: true`, `fee_escalation_percentage: 10`).
- The attacker only needs to observe the public mempool for nonce-1 invoke transactions from accounts with `account_nonce = 0`.

---

### Recommendation

Fix `skip_stateful_validations` to specifically verify that a **`deploy_account` transaction** exists for the address in the mempool, not just any transaction. The mempool should expose a dedicated method such as `deploy_account_tx_in_pool_or_recent_block(address)` that checks only for `DeployAccount` transaction types. The current `account_tx_in_pool_or_recent_block` check is insufficient because it returns `true` for any transaction type, including the attacker's own previously submitted nonce-1 invoke transaction.

---

### Proof of Concept

```
1. Alice sends: InvokeTx { sender_address: A, nonce: 1, signature: valid_sig, tip: 100 }
   → Gateway: skip_stateful_validations returns false (no tx in pool)
   → __validate__ runs, passes. Tx enters mempool.

2. Bob sends: InvokeTx { sender_address: A, nonce: 1, signature: INVALID, tip: 111 }
   → Gateway stateless: passes (valid fields, tip > 0)
   → validate_by_mempool: passes (tip 111 > 100 * 1.10 = 110, fee escalation satisfied)
   → skip_stateful_validations:
       account_tx_in_pool_or_recent_block(A) == true  ← Alice's tx is in pool
       returns true (skip __validate__)
   → run_validate_entry_point: validate=false, __validate__ NOT called
   → Bob's invalid-sig tx admitted, replaces Alice's tx via fee escalation

3. Batcher executes Bob's tx:
   → blockifier calls __validate__ → FAILS (invalid signature)
   → Tx reverted, fee charged from Alice's balance (sender_address = A)
   → Alice's valid tx is gone; her fee balance is reduced
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }
