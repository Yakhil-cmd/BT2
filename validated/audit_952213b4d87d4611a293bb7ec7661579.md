### Title
Price-Provider Timelock Enforcement Missing: Zero-Delay Bypass Allows Atomic Oracle Swap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`createPool` accepts `params.priceProviderTimelock = 0` without any minimum-value guard. With a zero timelock, the pool admin can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in the same transaction, atomically replacing the oracle with a malicious one and immediately executing a swap at a manipulated price, draining LP principal.

---

### Finding Description

`PoolParameters.priceProviderTimelock` is the sole mechanism that forces a delay between a pool admin proposing a new price provider and activating it. The factory stores it verbatim: [1](#0-0) 

The only special-case check is whether the value equals `type(uint256).max`, which activates immutable-provider mode: [2](#0-1) 

`_validatePoolParameters` validates tokens, admin, fees, and bin geometry, but contains **no check** that `priceProviderTimelock` is above any minimum when the pool is in mutable-provider mode: [3](#0-2) 

When `priceProviderTimelock[pool] == 0`, `proposePoolPriceProvider` computes:

```solidity
uint256 executeAfter = block.timestamp + timelock;  // = block.timestamp + 0
``` [4](#0-3) 

Then `executePoolPriceProviderUpdate` enforces:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
``` [5](#0-4) 

Because `block.timestamp < block.timestamp` is always `false`, the revert never fires. Both calls succeed in the same block, making the timelock completely inoperative.

---

### Impact Explanation

The pool admin is explicitly described as "semi-trusted only inside caps and timelocks." The timelock is the only mechanism that gives LPs time to exit before a hostile oracle is activated. With `priceProviderTimelock = 0`:

- The pool admin atomically swaps the oracle to one returning an arbitrarily skewed bid/ask.
- A swap is executed at the manipulated price in the same transaction.
- LP token balances no longer cover LP claims — pool insolvency / direct loss of user principal.

This matches the allowed impact gate: **bad-price execution** and **admin-boundary break** (pool admin bypasses the timelock cap).

---

### Likelihood Explanation

Any caller of the permissionless `createPool` can set `priceProviderTimelock = 0`. A malicious pool deployer can advertise a legitimate-looking pool, attract LP deposits, then drain it in a single transaction. No privileged factory-owner action is required; the pool admin role is sufficient and is set by the pool creator at deployment.

---

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters` for mutable-provider pools:

```solidity
// In _validatePoolParameters, after the immutablePriceProvider check:
bool immutable_ = params.priceProviderTimelock == type(uint256).max;
if (!immutable_ && params.priceProviderTimelock < MINIMUM_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

`MINIMUM_PRICE_PROVIDER_TIMELOCK` should be set to a value that gives LPs meaningful time to exit (e.g., 24 hours). This mirrors the fix recommended in M-03: enforce the delay at construction/initialization rather than leaving it caller-controlled with no floor.

---

### Proof of Concept

```solidity
// Pool created with zero timelock — no factory validation rejects it
PoolParameters memory params = _defaultParams();
params.priceProviderTimelock = 0;          // <-- zero delay, passes _validatePoolParameters
address pool = factory.createPool(params);

// Attacker (pool admin) deploys a malicious oracle returning price = 0
MaliciousOracle mal = new MaliciousOracle(token0, token1);

// Step 1: propose — executeAfter = block.timestamp + 0 = block.timestamp
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(mal));

// Step 2: execute in the SAME block — block.timestamp < block.timestamp is false → no revert
vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool);   // succeeds immediately

// Step 3: swap at manipulated price, drain LP funds
// router.swap(pool, ...) — pool now reads from MaliciousOracle
``` [6](#0-5)

  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
