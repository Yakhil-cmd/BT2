### Title
Missing Lower-Bound Threshold Validation in `assert_sign_inputs` Allows Sub-Threshold Signing — (`File: src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs`, which gates all FROST EdDSA and RedJubjub signing operations, validates only the upper bound of the `threshold` parameter (`threshold > participants.len()`). It never checks the lower bound (`threshold < 2`). This is the exact check that **is** present in `assert_key_invariants` (DKG) and `validate_triple_inputs` (OT-ECDSA triple generation). A caller who passes `threshold = 1` (or `0`) bypasses the guard entirely, and the unchecked value flows directly into `construct_key_package`, which sets `min_signers = threshold` in the FROST `KeyPackage`. With `min_signers = 1`, the FROST library accepts a signing package containing only a single participant's commitment, allowing a single key-share holder to drive the signing protocol to completion and produce a cryptographic output that honest parties treat as a valid threshold signature.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds:

```rust
if threshold > participants.len() { … ThresholdTooLarge … }
if threshold < 2 { … ThresholdTooSmall … }   // ← present
``` [1](#0-0) 

`validate_triple_inputs` in the OT-ECDSA triple-generation path also enforces both bounds:

```rust
if threshold_value > participants.len() { … ThresholdTooLarge … }
if threshold_value < 2 { … ThresholdTooSmall … }   // ← present
``` [2](#0-1) 

`assert_sign_inputs`, however, only checks the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← lower-bound check is absent
``` [3](#0-2) 

`assert_sign_inputs` is the sole validation gate for both FROST EdDSA (`sign_v1`, `sign_v2`) and RedJubjub (`sign`): [4](#0-3) [5](#0-4) 

The validated (but unchecked) `threshold` value is then forwarded to `construct_key_package`, which converts it directly to `min_signers` in the FROST `KeyPackage`:

```rust
Ok(KeyPackage::new(
    identifier,
    signing_share,
    verifying_share,
    *verifying_key,
    u16::try_from(threshold.value())…,   // min_signers = threshold
))
``` [6](#0-5) 

`construct_key_package` is called for every participant role (coordinator and non-coordinator) in both `sign_v1` and `sign_v2` paths: [7](#0-6) [8](#0-7) 

The same `assert_sign_inputs` path is used by the `presign` function in `src/frost/mod.rs`, which also omits the lower-bound check: [9](#0-8) 

---

### Impact Explanation

With `threshold = 1` accepted by `assert_sign_inputs`, the FROST `KeyPackage` is constructed with `min_signers = 1`. The FROST library then permits a signing package containing only a single participant's commitment. A malicious coordinator can:

1. Collect only their own nonce commitment (ignoring others).
2. Construct a `SigningPackage` with a single entry.
3. Invoke `round2::sign` — which succeeds because `min_signers = 1`.
4. Call `aggregate` with a single signature share.

Because the Lagrange coefficient for a lone signer evaluated at 0 is 1, the resulting aggregate signature is `z = nonce + signing_share * challenge`. This verifies against the individual public key of that participant, not the group public key. Honest verifiers who check against the group public key will receive a verification failure, meaning the signing output is cryptographically corrupt and unusable. Honest parties that do not independently re-verify the group key (e.g., they trust the coordinator's output) may accept the corrupted output.

**Mapped impact:** High — Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.

---

### Likelihood Explanation

Any caller of `sign_v1`, `sign_v2` (EdDSA), or `sign` (RedJubjub) controls the `threshold` parameter directly. No special privilege is required. A misconfigured or malicious coordinator passing `threshold = 1` (e.g., by accident, analogous to the uninitialized `transferHalfLife = 0` in the reference report) will silently pass all library-level guards and corrupt the signing session for all honest participants.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs` to `assert_sign_inputs`:

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    // … existing checks …
    if threshold.value() < 2 {
        return Err(InitializationError::ThresholdTooSmall {
            threshold: threshold.value(),
            min: 2,
        });
    }
    // … rest of function …
}
```

Apply the same fix to the `presign` function in `src/frost/mod.rs` for the `PresignArguments::threshold` field. [10](#0-9)

### Proof of Concept

```rust
// Demonstrates that assert_sign_inputs accepts threshold = 1
// while assert_key_invariants correctly rejects it.

use threshold_signatures::{
    frost::assert_sign_inputs,
    dkg::assert_key_invariants,
    participants::Participant,
    ReconstructionLowerBound,
};

let p1 = Participant::from(1u32);
let p2 = Participant::from(2u32);
let participants = vec![p1, p2];
let threshold: ReconstructionLowerBound = 1usize.into();

// DKG correctly rejects threshold = 1
let dkg_result = assert_key_invariants(&participants, p1, threshold);
assert!(dkg_result.is_err()); // ThresholdTooSmall

// Signing silently accepts threshold = 1 — BUG
let sign_result = assert_sign_inputs(&participants, threshold, p1, p2);
assert!(sign_result.is_ok()); // passes, threshold = 1 flows into KeyPackage::min_signers
```

The accepted `threshold = 1` then reaches `construct_key_package` and sets `min_signers = 1` in the FROST `KeyPackage`, allowing the signing session to proceed with a single participant's share and producing a cryptographically corrupt aggregate output. [11](#0-10)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}

fn construct_key_package(
    threshold: ReconstructionLowerBound,
    me: Participant,
    signing_share: SigningShare,
    verifying_key: &VerifyingKey,
) -> Result<KeyPackage, ProtocolError> {
    let identifier = me.to_identifier()?;
    let verifying_share = signing_share.into();

    Ok(KeyPackage::new(
        identifier,
        signing_share,
        verifying_share,
        *verifying_key,
        u16::try_from(threshold.value()).map_err(|_| {
            ProtocolError::Other("threshold cannot be converted to u16".to_string())
        })?,
    ))
}
