### Title
Multi-NFT Royalty Amount Understatement in WalletConnect Offer Confirmation Dialog - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` uses two sequential BigInt floor divisions to compute the "Total Amount with Royalties" displayed in the WalletConnect confirmation dialog. When an offer involves multiple NFTs, the function first divides the fungible amount by the number of NFTs (`splitAmount = amount / BigInt(royaltyPercentages.length)`) before computing each NFT's royalty on that split amount. This systematically understates the displayed royalty by a factor of `1/N` (where N is the number of NFTs), causing users to approve WalletConnect `take_offer` or `create_offer_for_ids` transactions believing they will spend less than the actual on-chain cost.

### Finding Description

In `formatAmountWithRoyalties`:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);  // floor division #1
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,  // floor division #2
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The function computes: `sum_i( (amount / N) * royaltyPercentage_i / 10_000 )`.

The correct computation is: `sum_i( amount * royaltyPercentage_i / 10_000 )`.

These are equal only when N=1. For N≥2, the displayed royalty is `1/N` of the correct total royalty.

**Concrete example** (matching the existing test case):
- Offer: 2 NFTs, fungible amount = 100,000,000 mojos (0.0001 XCH)
- NFT1 royalty: 500 bps (5%), NFT2 royalty: 10 bps (0.1%)
- **Displayed** (buggy): `splitAmount=50_000_000`; royalty = 2,500,000 + 50,000 = 2,550,000 mojos → total 0.00010255 XCH
- **Correct**: royalty = 5,000,000 + 100,000 = 5,100,000 mojos → total 0.0001051 XCH

For a realistic 10 XCH purchase of 2 NFTs each with 5% royalty:
- Displayed total: **10.5 XCH**
- Actual on-chain total: **11 XCH**
- Discrepancy: **0.5 XCH**

The `amountWithRoyalties` string is rendered in the WalletConnect confirmation dialog (`Confirm.tsx`) as "Total Amount with Royalties" under the "You Spend" section, which is the primary financial summary the user reads before clicking Confirm. [2](#0-1) 

The actual on-chain royalty is enforced by the Chia daemon (not by this display code), so the transaction charges the correct higher amount regardless of what the dialog shows.

### Impact Explanation

A user approves a WalletConnect `take_offer` or `create_offer_for_ids` command for a multi-NFT offer believing their total cost (including royalties) is X, but the actual on-chain deduction is higher. The discrepancy is proportional to `(N-1)/N × sum_of_royalties × amount`, which is material at normal XCH values. This constitutes approval of the wrong amount via WalletConnect state corruption.

### Likelihood Explanation

Any dApp connected via WalletConnect can trigger this path by sending a `take_offer` or `create_offer_for_ids` request that includes two or more NFTs with non-zero royalties. No privileged access is required. Multi-NFT bundle offers are a supported and advertised Chia feature. The bug is deterministic and reproducible for any N≥2.

### Recommendation

Replace the `splitAmount` approach with per-NFT royalty computed on the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This mirrors the correct mathematical definition (each NFT creator receives their royalty percentage of the full fungible payment) and eliminates both the `splitAmount` floor-division loss and the conceptual error.

### Proof of Concept

The existing test at `packages/gui/src/electron/commands/parseCommandDisplay.test.ts` lines 251–335 demonstrates the bug: with `amount = 100_000_000` mojos and two NFTs (royalty_percentage 500 and 10), the test asserts `amountWithRoyalties: '0.00010255'` (102,550,000 mojos). The correct value is 105,100,000 mojos (0.0001051 XCH). The test passes because it encodes the buggy behavior as the expected output. [3](#0-2) 

A WalletConnect-connected dApp issues `take_offer` with an offer blob containing two NFTs (royalty_percentage 500 each) and a 10 XCH fungible payment. The dialog shows "Total Amount with Royalties: 10.5 XCH". The user confirms. The daemon executes the offer and deducts 11 XCH from the user's wallet.
