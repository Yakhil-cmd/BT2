### Title
Unauthorized Burn of Arbitrary Account Balances via Bank Precompile — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `burn` case in `BankContract.Run` accepts an arbitrary `address` argument and burns tokens of denom `evm/<caller_contract>` from that address with **no authorization check** that the target account consented to or initiated the burn. Any contract can drain the native-bank balance of any account that holds tokens of the contract's denom.

---

### Finding Description

In `Run`, both `mint` and `burn` share a single case block. The address to burn **from** is taken directly from `args[0]`: [1](#0-0) 

The denom is locked to the calling contract (`evm/<contract.Caller()>`), but the target account `addr` is fully attacker-controlled: [2](#0-1) 

The only guards applied before this are:

- `readonly` check (line 114)
- `amount.Sign() <= 0` (line 123)
- `checkBlockedAddr` (line 127) — only rejects module/blocked addresses from **receiving** funds; it does not verify that the caller is authorized to spend from `addr` [3](#0-2) 

There is **no check** that `addr == sdk.AccAddress(contract.Caller().Bytes())` or that `addr` has granted any allowance. The `Executor` pattern used by other precompiles (relayer, ICA) does enforce caller-signer equality, but the bank precompile does not use it: [4](#0-3) 

The `transfer` method has the identical flaw — `sender` (args[0]) is arbitrary with no authorization check: [5](#0-4) 

---

### Impact Explanation

An attacker deploys contract `C` (address `0xC…`). Tokens of denom `evm/0xC…` are the only tokens the precompile will touch when `C` calls it. The attacker can:

1. **Mint** tokens of denom `evm/0xC…` to victim `V` (mint also has no recipient restriction).
2. Later call `burn(V, amount)` from `C` — this executes `SendCoinsFromAccountToModule(ctx, V, ...)` followed by `BurnCoins`, destroying `V`'s balance without `V`'s consent.

If `evm/<C>` tokens carry real value (e.g., they back a DeFi protocol, are used as collateral, or are redeemable for IBC vouchers via a wrapper), this constitutes **unauthorized destruction of user funds** — a Critical impact under the allowed scope.

The `transfer` variant allows the same contract to silently move any holder's balance to an arbitrary recipient.

---

### Likelihood Explanation

- Requires only deploying a contract and calling the precompile — no privileged keys, no governance, no validator access.
- The `TestBank.sol` integration contract demonstrates the intended pattern is `bank.burn(msg.sender, amount)`, but nothing enforces it. [6](#0-5) 

---

### About the "Duplicate Ordering / Event Binding" Framing

The question's framing of "duplicate ordering plus event binding" does **not** describe the actual attack path. The burn path passes `nil` as the event converter to `ExecuteNativeAction`, so no Cosmos→EVM event conversion occurs and there is no log-replay surface: [7](#0-6) 

The real root cause is purely the **missing caller-authorization check** on the `addr` argument. No reordering or event manipulation is needed.

---

### Recommendation

Add an explicit check in the `burn` (and `transfer`) branch that the target address equals the calling contract's address, or implement an allowance/approval mechanism before permitting third-party burns:

```go
// burn branch — enforce caller == addr being burned from
callerAddr := sdk.AccAddress(contract.Caller().Bytes())
if !addr.Equals(callerAddr) {
    return nil, errors.New("burn: caller is not the token holder")
}
```

---

### Proof of Concept

1. Deploy `MaliciousIssuer` contract at `0xA`.
2. Call `bank.mint(victim, 1000)` from `0xA` → victim now holds 1000 `evm/0xA` tokens.
3. Call `bank.burn(victim, 1000)` from `0xA` → `SendCoinsFromAccountToModule(victim, ...)` succeeds (victim has the balance), tokens are destroyed.
4. Verify: victim's `evm/0xA` balance is 0; no signature or approval from victim was required.

Compare state root before step 2 and after step 3 — victim's balance is gone despite never signing a burn authorization. [2](#0-1)
