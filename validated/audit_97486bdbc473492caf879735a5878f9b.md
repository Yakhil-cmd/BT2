Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Breaks Conservation Invariant â€” (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows all errors from the ICP ledger `send_pb` call, returning `()` regardless of outcome. Because cycles are deposited and the notification is finalized as `Ok` before any burn confirmation is possible, a transient ledger failure permanently strands ICP in the CMC's subaccount while the corresponding cycles remain in circulation, breaking the ICP/cycles conservation invariant with no recovery path.

## Finding Description
Three code paths call `burn_and_log` after a successful cycles operation:

- `process_create_canister` at line 1945: after `do_create_canister` succeeds
- `process_mint_cycles` at line 1968: after `do_mint_cycles` succeeds  
- `process_top_up` at line 2001: after `deposit_cycles` succeeds [1](#0-0) 

Each of these functions returns `Ok(...)` regardless of whether `burn_and_log` succeeded, because `burn_and_log` always returns `()`: [2](#0-1) 

The comment at line 2015â€“2016 explicitly acknowledges the design intent: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* The `Err` arm at line 2044â€“2047 only logs; it does not propagate, schedule a retry, or record the failure in persistent state.

Back in the callers (`notify_top_up`, `notify_mint_cycles`), the result from `process_top_up`/`process_mint_cycles` is `Ok(cycles)` â€” so `NotifiedTopUp(Ok(cycles))` / `NotifiedMint(Ok(...))` is written to `blocks_notified`: [3](#0-2) [4](#0-3) 

The `is_transient_error` guard only removes the entry when the *result* is a transient error â€” but the result is `Ok`, so the entry is never removed. The notification is permanently finalized. A grep for `failed_burns`, `pending_burns`, `burn_queue`, and heartbeat/timer-based burn retry confirms no retry mechanism exists anywhere in `rs/nns/cmc/src/`.

## Impact Explanation
This constitutes **illegal minting**: cycles are created and deposited to a target canister or cycles-ledger account without the corresponding ICP being destroyed. The ICP remains in the CMC's per-principal/per-canister subaccount with no user-accessible or operator-accessible recovery path. Total ICP supply is not reduced while total cycles supply has increased. This directly matches the Critical bounty impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets."* Repeated occurrences during ledger upgrade windows accumulate silently, with no on-chain signal beyond canister logs.

## Likelihood Explanation
Any unprivileged user calling `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` during a window in which the NNS ICP ledger (`ryjl3-tyaaa-aaaaa-aaaba-cai`) is transiently unavailable â€” such as during a canister upgrade, message queue saturation, or archiving operation â€” will trigger this path. No special privileges or victim interaction are required. The race is purely environmental. Because `created_at_time: None` is set in `send_args` at line 2038, a later manual retry of the burn would succeed (no deduplication block), but no such retry is ever scheduled. Likelihood is low-to-medium per individual transaction but non-negligible across the lifetime of the protocol given periodic ledger upgrades. [5](#0-4) 

## Recommendation
1. **Burn before finalizing the notification.** Reorder the sequence so the ICP burn is attempted and confirmed before writing `NotifiedTopUp(Ok(...))` to `blocks_notified`. If the burn fails with a transient error, leave the entry as `Processing` so the user can retry.
2. **Track failed burns in persistent state.** Add a `failed_burns: Vec<(Subaccount, Tokens)>` field to `State`. On burn failure in `burn_and_log`, push the entry rather than only logging.
3. **Retry via heartbeat or timer.** Drain `failed_burns` in a `canister_heartbeat` or periodic timer, retrying each `send_pb` call until it succeeds. Since `created_at_time: None`, retries are safe and idempotent from the ledger's perspective only if the subaccount balance is still present; set `created_at_time` to enable deduplication if retries are added.
4. **At minimum**, expose a metric counter for failed burns so operators can detect and manually intervene.

## Proof of Concept
1. User sends N ICP to the CMC subaccount for `canister_id` with `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id }`.
3. CMC marks the block as `Processing`, calls `process_top_up`.
4. `deposit_cycles` succeeds â€” N cycles are deposited to `canister_id` (irreversible).
5. CMC calls `burn_and_log(sub, amount).await`; the ICP ledger is mid-upgrade; `call_protobuf` returns `Err(...)`.
6. `burn_and_log` logs the error and returns `()`.
7. `process_top_up` returns `Ok(cycles)`.
8. CMC writes `NotifiedTopUp(Ok(cycles))` to `blocks_notified` â€” permanently finalized.
9. `notify_top_up` returns `Ok(cycles)` to the user.

**Result:** N cycles exist in `canister_id`. N ICP remain in the CMC's subaccount for `canister_id`, unburned. The conservation invariant is broken. The notification cannot be retried. A deterministic integration test using PocketIC can reproduce this by intercepting the `send_pb` call to the ledger canister and returning a reject code during the `burn_and_log` window, then asserting that the CMC subaccount balance is non-zero after `notify_top_up` returns `Ok`.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
