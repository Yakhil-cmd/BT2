### Title
Inaccessible Pause Mechanism in EVM Contract — `isPaused()` Guards Present but No `pause()`/`unpause()` Functions Implemented - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

The `EVM` Cadence contract in `fvm/evm/stdlib/contract.cdc` applies `!EVM.isPaused()` pre-condition guards to every state-mutating EVM operation, but the contract never defines a `pause()` or `unpause()` function. The pause flag is stored at `/storage/evmOperationsPaused` and is only read — never written — by any code in the repository. As a result, the emergency pause mechanism is inaccessible through the contract's own interface, making all `isPaused()` guards effectively dead code.

---

### Finding Description

`EVM.isPaused()` reads a `Bool` from `/storage/evmOperationsPaused`, defaulting to `false`:

```cadence
access(all)
view fun isPaused(): Bool {
    return self.account.storage.copy<Bool>(
        from: /storage/evmOperationsPaused
    ) ?? false
}
``` [1](#0-0) 

This guard is applied to every state-mutating EVM entry point:

| `EVMAddress.deposit()` | 203–204 |
| `CadenceOwnedAccount.call()` | 643–644 |
| `CadenceOwnedAccount.callWithSigAndArgs()` | 669–670 |

| `CadenceOwnedAccount.withdrawNFT()` | 761–762 |

| `EVM.run()` | 829–830 |

A repository-wide search for `fun pause`, `fun unpause`, and `evmOperationsPaused` returns **zero matches**. The storage path `/storage/evmOperationsPaused` is never written to by any contract function or system transaction in the codebase. The contract comment acknowledges this gap:

> "Only the Governance Committee can pause the EVM transactions, with a multi-sig Cadence transaction." [5](#0-4) 

This means the only way to set the pause flag is via a raw Cadence transaction that directly writes a `Bool` to the EVM contract account's storage — a mechanism that is entirely off-chain, undocumented in the contract, and not enforced by any access control defined within the contract itself.

---

### Impact Explanation

All `!EVM.isPaused()` pre-condition guards are permanently bypassed because the flag they read (`/storage/evmOperationsPaused`) can never be set to `true` through any on-chain contract function. During an active exploit of the EVM environment — for example, a reentrancy or bridge accounting bug — the Governance Committee has no contract-level mechanism to halt `run()`, `batchRun()`, `depositTokens()`, `withdrawTokens()`, or any other state-mutating EVM operation. The entire pause subsystem is dead code. This directly parallels the reported Solidity finding: a pause guard is declared and applied, but the setter is never implemented, so the protection can never be activated.

---

### Likelihood Explanation

The EVM contract is a core system contract deployed to the Flow Service Account on every network. Any scenario requiring an emergency pause — a bridge exploit, an EVM execution bug, a critical vulnerability in `InternalEVM` — would require the Governance Committee to respond. Because no `pause()` function exists in the contract, the response path is undefined and unreliable. The likelihood that this gap matters is directly proportional to the likelihood of any EVM-layer emergency, which is non-trivial given the complexity of the EVM integration and the bridge.

---

### Recommendation

Add `pause()` and `unpause()` functions to the `EVM` contract, restricted to the contract account (or a designated governance capability):

```cadence
access(account)
fun pause() {
    self.account.storage.save(true, to: /storage/evmOperationsPaused)
}

access(account)
fun unpause() {
    self.account.storage.save(false, to: /storage/evmOperationsPaused)
}
```

This makes the pause mechanism self-contained, auditable, and callable via a standard authorized Cadence transaction without requiring raw storage manipulation.

3. Observe that `run()` at line 828, `batchRun()` at line 918, and all other guarded functions will always pass the `!EVM.isPaused()` pre-condition because the flag can never be set to `true` through any contract function.
