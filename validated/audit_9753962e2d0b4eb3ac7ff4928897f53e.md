### Title
Gateway stateful validator checks only L2 gas price, admitting transactions with zero L1/L1-data gas price that will fail blockifier pre-validation - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful `validate_resource_bounds` function fetches only the previous block's L2 gas price and validates only the transaction's `l2_gas.max_price_per_unit` against it. The L1 gas price and L1 data gas price fields in `AllResources` bounds are never checked at the gateway or mempool admission layer. A transaction with `l1_gas.max_price_per_unit = 0` and `l1_data_gas.max_price_per_unit = 0` (but a valid L2 gas price) passes every gateway check and enters the mempool, yet is guaranteed to be rejected by the blockifier's `check_fee_bounds` during `perform_pre_validation_stage` before any execution occurs and without charging the sender.

### Finding Description

### Impact Explanation

An attacker submits `InvokeV3` transactions with `AllResources` bounds where:
- `l2_gas.max_price_per_unit` ≥ current L2 threshold (passes gateway)
- `l1_gas.max_price_per_unit = 0` and `l1_data_gas.max_price_per_unit = 0`

These transactions:
1. Pass stateless validation (only L2 price checked against static floor)
2. Pass stateful gateway validation (only L2 price checked against previous block)
3. Enter the mempool
4. Are picked up by the batcher
5. Fail `check_fee_bounds` in `perform_pre_validation_stage` with `InsufficientResourceBounds`
6. Are never included in a block and the sender is never charged

This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.** The mempool is polluted with transactions that are guaranteed to fail, wasting sequencer CPU and memory without any cost to the attacker.

### Likelihood Explanation

The attack requires no special privilege. Any user can craft a V3 transaction with zero L1 gas price. The `AllResources` bound format is the standard format for V3 transactions. The gap is explicitly acknowledged in the codebase via the TODO comment. The attack is trivially repeatable at negligible cost since failed pre-validation transactions are not charged.

### Recommendation

Extend `validate_resource_bounds` to also fetch `l1_gas_price` and `l1_data_gas_price` from the previous block and validate the transaction's `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` against them, mirroring the existing L2 gas price check. The TODO comment at line 358 already identifies this as the intended fix.

### Proof of Concept

1. Construct an `InvokeV3` transaction with `AllResources` bounds:
   - `l2_gas = { max_amount: 1, max_price_per_unit: current_l2_gas_price }` (valid)
   - `l1_gas = { max_amount: 0, max_price_per_unit: 0 }` (zero — will fail blockifier)
   - `l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }` (zero — will fail blockifier)
2. Submit via the gateway `add_tx` RPC endpoint.
3. `StatelessTransactionValidator::validate_resource_bounds` passes: `max_possible_fee > 0` (from L2 gas) and `l2_gas.max_price_per_unit >= min_gas_price`.
4. `StatefulTransactionValidator::validate_resource_bounds` passes: `l2_gas.max_price_per_unit >= threshold`.
5. Transaction is admitted to the mempool.
6. Batcher picks it up; blockifier calls `perform_pre_validation_stage` → `check_fee_bounds` → `MaxGasPriceTooLow { resource: L1Gas, max_gas_price: 0, actual_gas_price: <current_l1_price> }`.
7. Transaction is dropped; sender is not charged. Repeat indefinitely.
