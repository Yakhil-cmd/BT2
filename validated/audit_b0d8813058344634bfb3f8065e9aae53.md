### Title
Pusher delegation consent signature is replayable within the deadline window, rendering `revokePusher` ineffective - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1::allowPushers` verifies a pusher's EIP-191 consent signature but never marks the signature as consumed. The only replay guard is a deadline check. Within the deadline window, the creator can replay the exact same signature any number of times, re-establishing a delegation that the pusher has explicitly revoked via `revokePusher()`. A pusher whose key is compromised cannot stop the attacker from writing into the creator's oracle namespace until the deadline expires.

---

### Finding Description

`allowPushers` signs over `(chainid, address(this), deadline, pusher, msg.sender)` and checks only that `block.timestamp <= deadline`: [1](#0-0) 

The code's own NatSpec acknowledges the replay risk but claims the deadline is sufficient protection: [2](#0-1) 

`_ensureDeadline` only checks `block.timestamp <= deadline` — it does not record that the signature was already used: [3](#0-2) 

`revokePusher` clears the mapping to `address(0)`: [4](#0-3) 

Because no nonce or "used-signature" bitmap exists, the creator can immediately call `allowPushers` again with the identical `(deadline, pusher, sig)` tuple and restore `namespaceRemapping[pusher] = creator`. The pusher's revocation is silently undone.

**Attack path:**

1. Pusher signs consent for creator with `deadline = now + 30 days`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher's key is compromised; pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator (or the attacker who extracted the original calldata from the chain) calls `allowPushers(deadline, [pusher], [sig])` with the identical arguments.
5. `namespaceRemapping[pusher] = creator` is restored. The compromised pusher key continues to write into the creator's namespace.

The `fallback` push path resolves the namespace at call time: [5](#0-4) 

So every subsequent push from the compromised key lands in the creator's namespace, overwriting legitimate price data.

---

### Impact Explanation

The creator's oracle namespace receives attacker-controlled price data. `CompressedOracleV1` is the oracle layer consumed by `PriceProvider`, which is called by `MetricOmmPool::swap` to obtain `bidPriceX64` / `askPriceX64`. Corrupted prices reach live swaps, causing bad-price execution: traders receive more output than the oracle permits or the pool receives less input than owed, directly draining pool reserves.

---

### Likelihood Explanation

The original `allowPushers` calldata is permanently visible on-chain. Any party who observed the transaction — including the creator themselves — can replay it. Deadlines in practice are set days to weeks in the future (the test suite uses `block.timestamp + 1 days`). The window between a pusher's revocation and deadline expiry is therefore substantial. The creator has a direct economic incentive to keep the delegation alive (their oracle feed depends on it), making the replay a rational action even without malicious intent.

---

### Recommendation

Track consumed signatures with a per-pusher nonce stored in contract state and include it in the signed digest:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
uint256 nonce = pusherNonce[pusher]++;
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender, nonce))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
```

Alternatively, record the `keccak256` of each accepted signature in a `mapping(bytes32 => bool) usedSignatures` and revert on reuse. Either approach ensures that `revokePusher()` permanently invalidates the prior consent.

### Proof of Concept

Add to `smart-contracts-poc/test/oracles/compressed/CompressedOracle.t.sol`:

```solidity
function testAllowPushers_replayAfterRevoke_bypassesRevocation() public {
    uint256 deadline = block.timestamp + 1 days;
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // Step 1: creator delegates pusher
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator, "delegated");

    // Step 2: pusher revokes (e.g. key compromise)
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0), "revoked");

    // Step 3: creator replays the SAME signature — deadline still valid
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);

    // Revocation was silently undone; compromised key can push again
    assertEq(oracle.namespaceRemapping(pusher), creator, "delegation restored — revoke bypassed");
}
```

The test passes against the current code, confirming that `revokePusher` provides no durable protection within the deadline window.

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
