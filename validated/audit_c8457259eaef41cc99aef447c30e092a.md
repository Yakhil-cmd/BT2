### Title
Signature Verification Bypassed for Funded Undeployed Accounts via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` unconditionally skips the blockifier `__validate__` entry point for any invoke transaction with `nonce == 1` from an account whose on-chain nonce is `0`, provided the mempool reports *any* transaction for that address. Because the only gateway check that would catch an invalid signature is the `__validate__` call, an attacker who observes a `deploy_account` transaction in the mempool can inject an invoke transaction with a completely forged signature that is accepted by the gateway and inserted into the mempool.

---

### Finding Description

The gateway stateful path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`:

```
run_pre_validation_checks
  ├─ validate_state_preconditions   (resource bounds + nonce range)
  ├─ validate_by_mempool            (duplicate hash + nonce ordering only)
  └─ skip_stateful_validations      ← returns true → __validate__ skipped
```

`skip_stateful_validations` triggers when:

1. The transaction is an `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage` — the `__validate__` entry point (which is the only place the account's signature is cryptographically verified) is never called: [3](#0-2) 

`perform_pre_validation_stage` does run, but it only checks nonce ordering (`strict_nonce_check = false`) and, if `charge_fee = true`, fee-balance sufficiency — neither of which touches the signature: [4](#0-3) 

`validate_by_mempool` (called before the skip decision) only passes `ValidationArgs` containing address, nonce, tip, and gas price — no signature field: [5](#0-4) 

The mempool's `validate_tx` checks duplicate hash and nonce ordering only: [6](#0-5) 

`account_tx_in_pool_or_recent_block` returns `true` if the account has *any* transaction in the pool or in the mempool's committed-block state — it does not verify that the existing transaction is specifically a `deploy_account`: [7](#0-6) 

---

### Impact Explanation

An attacker who observes a `deploy_account` transaction for address `A` in the mempool (or infers it from the gateway's response) can submit an `InvokeV3` transaction with `nonce = 1`, `sender_address = A`, and an **arbitrary/invalid signature**. The transaction passes every gateway check:

- Stateless: signature *length* is within `max_signature_length` (content not checked).
- Stateful nonce: `0 ≤ 1 ≤ max_allowed_nonce_gap` passes.
- Mempool `validate_tx`: nonce ordering passes, no duplicate hash.
- `skip_stateful_validations`: returns `true` → `__validate__` skipped.
- `verify_can_pay_committed_bounds`: passes if the account address has been pre-funded with STRK (required for the deploy_account to succeed anyway).

The forged transaction is then inserted into the mempool via `mempool_client.add_tx`. If the attacker sets a higher tip than the legitimate user's invoke, the mempool's fee-escalation logic will **replace** the user's valid invoke with the attacker's forged one. The forged transaction will fail at `__validate__` during batcher execution (nonce not incremented on `__validate__` failure), but the user's original transaction has already been evicted from the mempool and must be resubmitted.

Impact category: **High — Mempool/gateway admission accepts an invalid transaction (forged signature) before sequencing.**

---

### Likelihood Explanation

- The `deploy_account` transaction hash and sender address are returned by the gateway and are public.
- The attacker needs no privileged access; only a standard gateway endpoint call is required.
- The window is the time between the `deploy_account` entering the mempool and being committed — typically one block.
- The attack is cheap: one gateway call with a crafted invoke transaction.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender address is present in the mempool. Alternatively, even when `skip_validate = true`, run a lightweight off-chain ECDSA/Stark-curve signature check against the transaction hash before admitting the transaction, so that the `__validate__` skip only bypasses the on-chain VM call, not all authentication.

---

### Proof of Concept

```
1. Account A is pre-funded with STRK (required for deploy_account fees).
   On-chain nonce of A = 0.

2. Legitimate user submits deploy_account tx for A (nonce=0).
   Gateway returns tx_hash_deploy. Mempool now contains deploy_account for A.

3. Attacker submits InvokeV3 tx:
     sender_address = A
     nonce          = 1
     signature      = [0xdeadbeef]   ← completely invalid
     tip            = <higher than user's planned invoke>
     resource_bounds = valid non-zero bounds

4. Gateway stateless validation:
   - signature length 1 ≤ max_signature_length (4000) → PASS

5. Gateway stateful validation:
   - get_nonce_from_state(A) → 0
   - validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap → PASS
   - validate_by_mempool: no duplicate hash, nonce ordering OK → PASS
   - skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → check mempool
       account_tx_in_pool_or_recent_block(A) → true (deploy_account present)
       returns true
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       perform_pre_validation_stage: nonce 0≤1 OK, fee balance OK → PASS
       __validate__ NOT called → signature never verified
   → PASS

6. mempool_client.add_tx(attacker_invoke) → attacker's tx in mempool.

7. User submits legitimate invoke (nonce=1, valid signature).
   Mempool: attacker's tx already at (A, nonce=1) with higher tip.
   User's tx rejected (DuplicateNonce) or requires fee escalation above attacker's tip.

8. Batcher sequences: deploy_account (A deployed), then attacker's invoke.
   __validate__ called → invalid signature → tx rejected, nonce NOT incremented.
   User's tx is gone; user must resubmit.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,

        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

            self.check_fee_bounds(tx_context)?;

        }

        self.validate_proof_facts(&tx_context.block_context, state)?;
