### Title
Ineffective Bridge Circuit Breaker — `TurnBridge` Is a No-Op Stub That Bypasses Permission Enforcement and Permanently Disables Emergency Bridge Shutdown - (File: x/cronos/keeper/msg_server.go)

---

### Summary

The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is implemented as a stub that unconditionally returns `nil, nil` without performing any permission check or state mutation. This has two direct security consequences: (1) any unprivileged user can call `MsgTurnBridge` and receive a success response, fully bypassing the `CanTurnBridge` permission requirement; and (2) the bridge enable/disable state is never written, permanently breaking the emergency circuit breaker that is supposed to allow the admin to halt bridge operations.

---

### Finding Description

The `TurnBridge` handler at `x/cronos/keeper/msg_server.go` lines 85–87 is:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

Every other privileged handler in the same file enforces its authorization gate before acting:

- `UpdateTokenMapping` (line 73): `if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`
- `UpdatePermissions` (line 106): `if admin != msg.From`
- `StoreBlockList` (line 121): `if admin != msg.From`

`TurnBridge` performs none of these checks and writes no state. The permission system defines `CanTurnBridge = 2` in `x/cronos/keeper/permissions.go` and the `QueryPermissionsResponse` exposes `CanTurnBridge` as a distinct capability, confirming the intent that only authorized addresses should be able to toggle the bridge. The integration test `test_gravity_turn_bridge` (line 661–665) explicitly asserts that an unpermissioned caller (`community`) must be rejected and that a permissioned caller (`validator`) must succeed — behavior that is impossible with the current stub.

**Attacker path:**

1. Any unprivileged account constructs a `MsgTurnBridge` message (e.g., `enable=false`).
2. The message is broadcast and accepted by the chain.
3. `TurnBridge` returns success without checking `CanTurnBridge` — the permission gate is bypassed.
4. No bridge-state key is written; the bridge remains in its default (enabled) state regardless of the message content.
5. Repeat with `enable=true` or `enable=false` — the result is identical and the bridge state never changes.

**Secondary consequence (broken circuit breaker):**

Because the bridge state is never written, the admin also cannot disable the bridge. Any EVM hook handler that gates bridge operations on the bridge-enabled flag will always see the default "enabled" state. In an emergency (e.g., an active exploit draining funds through the bridge), the intended emergency shutdown path is permanently non-functional.

---

### Impact Explanation

**Permission bypass (High):** Any unprivileged user can call `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` authorization check that is explicitly defined and enforced everywhere else in the permission system.

**Permanent inability to disable bridge flows (High):** The bridge circuit breaker is permanently broken. The admin cannot halt bridge/conversion flows in an emergency. If a bridge exploit is discovered, the only mitigation path (turning off the bridge) is unavailable, allowing continued unauthorized asset movement until a governance upgrade is deployed — a process that takes far longer than an active exploit.

Both impacts fall within the allowed High scope: bypass of module-account authorization checks, and permanent inability to halt bridge flows under normal network assumptions.

---

### Likelihood Explanation

The entry point is a standard Cosmos SDK message (`MsgTurnBridge`) reachable by any account with gas. No privileged key, leaked secret, or special network position is required. The stub is unconditional — every call succeeds regardless of sender.

---

### Recommendation

Implement `TurnBridge` with the same pattern used by all other privileged handlers:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    // persist bridge enabled/disabled state
    params := k.GetParams(ctx)
    params.EnableGravity = msg.Enable
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &MsgTurnBridgeResponse{}, nil
}
```

The exact state field depends on the bridge parameter schema, but the critical requirements are: (a) reject unauthorized callers before any state mutation, and (b) actually persist the bridge state so that EVM hook handlers can observe it.

```
# Actual: code == 0, success — permission check bypassed, bridge state unchanged

# Returns success, but bridge state is never written; bridge remains enabled
```
