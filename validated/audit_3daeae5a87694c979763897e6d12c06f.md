### Title
Empty `AppId` Allowed in CKD Protocol Collapses Domain Separation, Corrupting Derived Key Uniqueness - (File: `src/confidential_key_derivation/app_id.rs`)

---

### Summary

`AppId::try_new` enforces only a maximum length bound and silently accepts empty byte slices. Because `AppId` is the sole domain separator in `hash_app_id_with_pk`, an empty `AppId` reduces the hash input to just the MPC network's compressed public key, making the derived BLS secret `S = msk · H(pk)` identical for every CKD request that uses an empty identifier. The `ckd()` entry point performs no minimum-length check on the supplied `app_id`, so this path is reachable by any library caller.

---

### Finding Description

`AppId::try_new` in `src/confidential_key_derivation/app_id.rs` validates only the upper bound:

```rust
// src/confidential_key_derivation/app_id.rs  L58-L68
pub fn try_new(id: impl AsRef<[u8]>) -> Result<Self, ProtocolError> {
    let id = id.as_ref();
    if id.len() > MAX_APP_ID_LEN {          // only upper bound checked
        ...
        return Err(ProtocolError::InvalidInput(err_msg));
    }
    Ok(Self(Arc::from(id)))                 // empty slice accepted silently
}
```

There is no minimum-length guard. The library's own test suite explicitly exercises `vec![]` as a valid `AppId` (line 172 of `app_id.rs`), confirming the empty case is accepted by design.

The public `ckd()` entry point in `src/confidential_key_derivation/protocol.rs` validates participant counts, duplicates, and coordinator membership, but performs **no validation on `app_id`**:

```rust
// src/confidential_key_derivation/protocol.rs  L66-L116
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,   // no emptiness check
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // ... participant checks only ...
    let fut = run_ckd_protocol(..., app_id.into(), ...);
    Ok(make_protocol(comms, fut))
}
```

Inside `compute_signature_share`, the `app_id` is passed directly to `hash_app_id_with_pk`:

```rust
// src/confidential_key_derivation/protocol.rs  L167-L171
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
let big_s = hash_point * private_share.to_scalar();
```

`hash_app_id_with_pk` concatenates the compressed network public key with `app_id` before hashing:

```rust
// src/confidential_key_derivation/mod.rs  L67-L71
pub fn hash_app_id_with_pk(pk: &VerifyingKey, app_id: &[u8]) -> ElementG1 {
    let compressed_pk = pk.to_element().to_compressed();
    let input = [compressed_pk.as_slice(), app_id].concat();
    ciphersuite::hash_to_curve(&input)
}
```

When `app_id` is empty, `input = compressed_pk` only. The hash point becomes `H(pk)`, which is **constant** across all CKD sessions sharing the same MPC network key. Consequently:

```
S  =  msk · H(pk)          (same for every empty-AppId request)
C  =  S + y · A            (varies per session only due to ephemeral y)
```

After the coordinator aggregates and the caller calls `CKDOutput::unmask(a)`:

```
unmask(a) = C − a·Y = S = msk · H(pk)
```

Every application that submits an empty `AppId` recovers the **same** BLS signature `S`. The `AppId` field, which is the only mechanism distinguishing one application's derived secret from another's, is completely neutralised.

---

### Impact Explanation

The CKD protocol's security guarantee is that each application receives a deterministic secret uniquely bound to its `app_id`. An empty `AppId` eliminates this binding: all callers using `AppId = []` share the identical underlying secret `msk · H(pk)`. This corrupts the CKD output — the derived key is no longer application-specific, violating the core isolation property the protocol is designed to enforce.

**Impact: High** — Corruption of CKD outputs so honest parties accept cryptographic outputs that are inconsistent with the documented uniqueness guarantee (different applications receiving the same derived secret).

---

### Likelihood Explanation

The empty `AppId` path requires no privilege: any caller of the public `ckd()` function can supply `AppId::try_new(vec![]).unwrap()` and the call succeeds without error. The library's own test suite treats empty `AppId` as a valid, round-trippable value, so there is no runtime signal that this is incorrect. Any integrator that does not independently enforce a minimum `app_id` length is exposed. Likelihood is **Medium** — requires a caller to pass an empty identifier, which is an easy mistake for an integrator to make given the library provides no guard.

---

### Recommendation

1. Add a minimum-length check in `AppId::try_new`:

```rust
const MIN_APP_ID_LEN: usize = 1;

pub fn try_new(id: impl AsRef<[u8]>) -> Result<Self, ProtocolError> {
    let id = id.as_ref();
    if id.is_empty() {
        return Err(ProtocolError::InvalidInput(
            "AppId must not be empty".to_string(),
        ));
    }
    if id.len() > MAX_APP_ID_LEN { ... }
    Ok(Self(Arc::from(id)))
}
```

2. Optionally add a defence-in-depth check in `ckd()` after `app_id.into()` to reject empty `AppId` at the protocol entry point.

3. Update the `test_borsh_roundtrip` test to assert that `AppId::try_new(vec![])` returns an error.
