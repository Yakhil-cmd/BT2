### Title
WalletConnect Royalty Total Understated Due to Incorrect Split-Before-Multiply Arithmetic in `formatAmountWithRoyalties` - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` computes the "Total Amount with Royalties" shown in the WalletConnect confirmation dialog by first dividing the trade amount by the number of NFTs, then multiplying each split portion by its royalty percentage. This is mathematically wrong: each NFT's royalty applies to the full trade amount, not a fractional share. The result is a displayed total that is materially lower than what the wallet will actually deduct, causing users to approve WalletConnect `take_offer` / `create_offer_for_ids` requests under a false understanding of the cost.

### Finding Description
In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` computes the royalty-inclusive total as follows:

```ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // integer division
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The `splitAmount` divides the full trade amount by the count of NFTs before computing each royalty. The correct formula is to apply each NFT's royalty percentage to the full `amount` independently and sum the results. The split-first approach produces a total that is `N` times smaller in royalty contribution than the actual on-chain cost (where `N` is the number of NFTs), plus additional truncation loss from bigint integer division.

The computed `amountWithRoyalties` string is then surfaced in the WalletConnect confirmation dialog under the "You Spend" section as **"Total Amount with Royalties"**:

```tsx
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties} XCH
  </div>
)}
``` [2](#0-1) 

This is the primary cost signal the user reads before clicking Approve.

### Impact Explanation
A malicious dapp connected via WalletConnect crafts a `take_offer` or `create_offer_for_ids` call containing multiple NFTs with high royalty percentages. The confirmation dialog displays a "Total Amount with Royalties" that is significantly lower than what the wallet will actually deduct. The user approves based on the understated figure; the actual transaction spends more XCH or CAT than the user consented to.

**Concrete example** — user pays 1 XCH for 2 NFTs each with 50% royalty (5000 basis points):

| | Displayed (split-first) | Actual (correct) |
|---|---|---|
| `splitAmount` | 500,000,000,000 mojos | — |
| royalty per NFT | 250,000,000,000 mojos | 500,000,000,000 mojos |
| total royalties | 500,000,000,000 mojos | 1,000,000,000,000 mojos |
| **Total shown** | **1.5 XCH** | **2.0 XCH** |

The user sees 1.5 XCH and approves; 2.0 XCH is deducted — a 33% overcharge relative to what was displayed.

The existing test fixture confirms the wrong value is accepted as correct:

```ts
amountWithRoyalties: '0.00010255',  // split-first result, not the true total
``` [3](#0-2) 

### Likelihood Explanation
Any dapp with an active WalletConnect session can issue `chia_wallet.take_offer` with a crafted multi-NFT offer. No privileged access is required. The user must click Approve, but the understated total is the exact information they rely on to make that decision. NFTs with royalties of 10–50% are common in the Chia ecosystem, making the discrepancy material.

### Recommendation
Replace the split-before-multiply pattern with per-NFT royalty computation on the full amount:

```ts
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [4](#0-3) 

Update the corresponding test in `parseCommandDisplay.test.ts` to assert the correct total (e.g., for the two-NFT case with royalties 500 + 10 on 100,000,000 mojos, the correct `amountWithRoyalties` is `(100000000 + 5000000 + 100000) / 1e12 = 0.0001051 XCH`, not `0.00010255`).

### Proof of Concept
1. Establish a WalletConnect session with the Chia GUI.
2. Issue `chia_wallet.take_offer` with an offer blob containing two NFTs, each with `royalty_percentage: 5000` (50%), and a requested XCH amount of `1000000000000` mojos (1 XCH).
3. Observe the WalletConnect confirmation dialog: "Total Amount with Royalties" shows **1.5 XCH**.
4. Click Approve.
5. Observe the actual wallet balance decrease by **2.0 XCH** (1 XCH base + 0.5 XCH royalty per NFT × 2 NFTs).

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
