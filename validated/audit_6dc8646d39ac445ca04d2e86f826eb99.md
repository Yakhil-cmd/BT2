### Title
Malicious CKD Participant Sends Unauthenticated Group Elements, Corrupting the Derived Confidential Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates per-participant contributions `(big_y, big_c)` by simple addition, with no zero-knowledge proof or commitment binding those values to the participant's actual key share. Any single malicious participant can substitute arbitrary group elements, causing the coordinator to output a corrupted confidential key that honest parties silently accept as valid.

---

### Finding Description

The CKD protocol is a threshold ElGamal-based scheme. Each participant `i` is supposed to compute:

```
y_i  ← random scalar
Y_i  = y_i · G
S_i  = x_i · H(pk ‖ app_id)
C_i  = S_i + y_i · app_pk
```

then Lagrange-normalize and send `(λ_i · Y_i, λ_i · C_i)` privately to the coordinator.

The coordinator aggregates all shares and outputs `(Y, C)` such that `C − app_sk · Y = msk · H(pk ‖ app_id)`.

In `do_ckd_coordinator`, the coordinator receives each participant's contribution and adds it unconditionally: [1](#0-0) 

```rust
// Receive everyone's inputs and add them together
let waitpoint = chan.next_waitpoint();

for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no proof of correct computation** attached to `(big_y, big_c)`. The `CKDOutput` struct is a plain pair of `ElementG1` values with no cryptographic binding to the sender's key share: [2](#0-1) 

The participant path (`do_ckd_participant`) simply computes the share and sends it with no proof: [3](#0-2) 

Compare this with the DKG protocol, which requires and verifies a Schnorr proof-of-knowledge for every participant's polynomial commitment before accepting it: [4](#0-3) 

No equivalent verification exists in the CKD aggregation step.

---

### Impact Explanation

A malicious participant sends `(big_y' = big_y_correct + Δ_Y, big_c' = big_c_correct + Δ_C)` for attacker-chosen group elements `Δ_Y, Δ_C`. The coordinator computes:

```
Y_final = Y_honest + Δ_Y
C_final = C_honest + Δ_C
```

The TEE unmasks with `app_sk`:

```
C_final − app_sk · Y_final
  = msk · H(pk ‖ app_id) + (Δ_C − app_sk · Δ_Y)
```

The result is no longer `msk · H(pk ‖ app_id)`. The coordinator and TEE receive and silently accept a corrupted confidential key. No error is raised because the protocol has no mechanism to detect the deviation.

This matches the **High** impact: *Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

- Any single participant out of the N−1 non-coordinator participants can trigger this unilaterally.
- No cryptographic capability beyond participation in the protocol is required.
- The attack requires only sending a crafted pair of group elements instead of the honest computation — a trivial modification for a library caller who controls their own protocol instance.
- The coordinator has no way to detect or reject the malicious contribution.

---

### Recommendation

Each participant must accompany their `(big_y, big_c)` contribution with a zero-knowledge proof of correct formation. Concretely, the participant should prove:

1. **Knowledge of `y_i`**: a Schnorr discrete-log proof that `big_y = y_i · G`.
2. **Correct ElGamal ciphertext**: a proof that `big_c = λ_i · x_i · H(pk ‖ app_id) + y_i · app_pk`, binding `y_i` to the same witness used in (1) and `x_i` to the participant's committed key share from DKG.

This is analogous to the proof-of-knowledge already required in `do_keyshare` for DKG commitments. [5](#0-4) 

---

### Proof of Concept

**Setup:** 3-party CKD with participants `P1` (coordinator), `P2`, `P3 (malicious)`.

**Honest execution** would yield `C − app_sk · Y = msk · H(pk ‖ app_id)`.

**Attack:**

1. `P3` runs `compute_signature_share` normally to obtain `(norm_big_y_3, norm_big_c_3)`.
2. Instead of sending the honest values, `P3` sends `(norm_big_y_3, norm_big_c_3 + Δ)` where `Δ` is any non-zero `ElementG1`.
3. The coordinator at line 53–54 adds the tampered `big_c` without verification: [6](#0-5) 

4. The final `CKDOutput` satisfies `C_final − app_sk · Y_final = msk · H(pk ‖ app_id) + Δ`.
5. The TEE calls `ckd_output.unmask(app_sk)` and receives a wrong key with no error. [7](#0-6) 

The deviation `Δ` is fully attacker-controlled, making the output of the CKD protocol entirely under the malicious participant's influence.

pub struct CKDOutput {
    big_y: ElementG1,
    big_c: ElementG1,
}

impl CKDOutput {
    pub fn new(big_y: ElementG1, big_c: ElementG1) -> Self {
        Self { big_y, big_c }
    }

    /// Outputs `big_y`
    pub fn big_y(&self) -> ElementG1 {
        self.big_y
    }

    /// Outputs `big_c`
    pub fn big_c(&self) -> ElementG1 {
        self.big_c
    }

    /// Takes a secret scalar and returns
    /// s <- C − a ⋅ Y = msk ⋅ H ( `app_id` )
    pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
        self.big_c - self.big_y * secret_scalar
    }
}

fn proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    me: Participant,
    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,
    rng: &mut impl CryptoRngCore,
) -> Result<Signature<C>, ProtocolError> {
    // creates an identifier for the participant
    let id = me.scalar::<C>();
    let vk_share = coefficient_commitment.eval_at_zero()?;

    // pick a random k_i and compute R_id = g^{k_id},
    // Step 2.5
    let (k, big_r) = <C>::generate_nonce(rng);

    // Step 2.6
    // compute H(domain_separator, id, me, g^{a_0}, R_id) as a scalar
    let hash = challenge::<C>(domain_separator, session_id, id, &vk_share, &big_r)?;
    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
    Ok(Signature::new(big_r, mu))
}
