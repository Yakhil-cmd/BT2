### Title
Reward Address Can Accidentally Stake Own Funds Into Pool Member's Balance - (File: src/pool/pool.cairo)

### Summary
In `Pool::add_to_delegation_pool`, when the reward address calls the function, funds are transferred from the reward address (the actual caller) rather than from the pool member. The interface documentation misleadingly states that the *pool member* needs sufficient funds and approval, but the implementation transfers from `caller_address`. This is the direct analog to the external report: a function accepts an address parameter as the intended payer, but the actual caller is not validated against that address, allowing the wrong party to accidentally pay.

### Finding Description
`add_to_delegation_pool` accepts `pool_member: ContractAddress` as a parameter and permits either the pool member or their reward address to call it. The access control check is: [1](#0-0) 

However, the fund transfer always uses `caller_address` as the source: [2](#0-1) 

While the balance is credited to the `pool_member` parameter: [3](#0-2) 

The underlying `transfer_from_delegator` utility confirms it pulls from whichever address is passed as `pool_member`: [4](#0-3) 

The interface documentation reinforces the confusion by stating the *pool member* (not the caller) must have sufficient funds and approval: [5](#0-4) 

When the reward address calls `add_to_delegation_pool(pool_member, amount)`:
- Funds are pulled **from the reward address** (the caller)
- Balance is credited **to the pool member** (the parameter)
- The reward address's tokens are now locked in the staking contract under the pool member's balance
- The reward address has no direct path to recover them; recovery requires the pool member to voluntarily exit and return the tokens

### Impact Explanation
The reward address loses their own tokens, which become permanently locked in the staking contract under the pool member's balance. This constitutes **temporary (effectively permanent without pool member cooperation) freezing of the reward address's funds** — matching the allowed High impact: *Temporary freezing of funds*.

### Likelihood Explanation
The reward address is explicitly permitted to call `add_to_delegation_pool`. If the reward address is a smart contract (e.g., a yield aggregator, auto-compounder, or multisig) that has approved the pool contract for token transfers — a natural setup for a contract that manages staking positions — any call to `add_to_delegation_pool` will silently drain the reward address's own tokens. The misleading documentation ("pool_member has sufficient funds and approval") directly increases the probability of this mistake, as an integrator reading the docs would expect the pool member's pre-approved funds to be used, not their own.

### Recommendation
The fund transfer should always pull from `pool_member`, not from `caller_address`. If the reward address is permitted to trigger a top-up, the pool member should pre-approve the pool contract, and the transfer should use `pool_member` as the sender regardless of who calls the function:

```cairo
// Always transfer from pool_member, not from caller_address
transfer_from_delegator(pool_member: pool_member, :amount, :token_dispatcher);
```

Alternatively, if the design intent is that the reward address may contribute their own funds, the interface documentation must be corrected to state that the *caller* (not the pool member) needs sufficient funds and approval, and the emitted event should reflect the actual fund source.

### Proof of Concept
1. Alice is a pool member; she sets her `reward_address` to Bob's smart contract.
2. Bob's smart contract approves the pool contract to spend its tokens (e.g., to reinvest accrued rewards).
3. Bob's smart contract calls `add_to_delegation_pool(Alice_address, 1000)`, expecting — per the documented precondition — that Alice's pre-approved funds will be used.
4. Instead, `transfer_from_delegator(pool_member: caller_address, ...)` pulls 1000 tokens **from Bob's smart contract**.
5. Alice's delegated balance increases by 1000; Bob's smart contract has lost 1000 tokens locked under Alice's pool position.
6. Bob's contract cannot recover the tokens directly; Alice must exit the pool and voluntarily return them. [6](#0-5)

        fn add_to_delegation_pool(
            ref self: ContractState, pool_member: ContractAddress, amount: Amount,
        ) -> Amount {
            // Asserts.
            self.assert_staker_is_active();
            let pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            assert!(
                caller_address == pool_member || caller_address == pool_member_info.reward_address,
                "{}",
                Error::CALLER_CANNOT_ADD_TO_POOL,
            );
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);

            // Transfer funds from the delegator to the staking contract.
            let token_dispatcher = self.token_dispatcher.read();
            let staker_address = self.staker_address.read();
            transfer_from_delegator(pool_member: caller_address, :amount, :token_dispatcher);
            self.transfer_to_staking_contract(:amount, :token_dispatcher, :staker_address);

            // Update the pool member's balance checkpoint.
            let old_delegated_stake = self.increase_member_balance(:pool_member, :amount);
            let new_delegated_stake = old_delegated_stake + amount;

            // Emit events.
            self
                .emit(
                    Events::PoolMemberBalanceChanged {
                        pool_member, old_delegated_stake, new_delegated_stake,
                    },
                );

            new_delegated_stake
        }

pub(crate) fn transfer_from_delegator(
    pool_member: ContractAddress, amount: Amount, token_dispatcher: IERC20Dispatcher,
) {
    let self_contract = get_contract_address();
    token_dispatcher
        .checked_transfer_from(
            sender: pool_member, recipient: self_contract, amount: amount.into(),
        );
}
