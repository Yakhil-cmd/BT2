### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on the pool admin's fee components, but `setPoolBinAdditionalFees` applies per-bin additional fees with **no cap validation**. Because bin additional fees are additive to the base spread fee in every swap computation, the pool admin can cause traders to pay effective fees that exceed the protocol-enforced hard cap (`HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%).

### Finding Description

`setPoolAdminFees` enforces caps before updating fees:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` passes values directly to the pool with **no cap check**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

In every swap, the bin additional fee is added directly on top of the oracle-derived base fee:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
``` [3](#0-2) 

The same pattern applies on the sell side:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [4](#0-3) 

`addFeeBuyE6` and `addFeeSellE6` are `uint16` fields, so their maximum value is 65 535, which in E6 units equals **≈ 6.55%**. [5](#0-4) 

The factory hard cap is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%). [6](#0-5) 

**Attack path:**
1. Pool admin sets base admin spread fee to `maxAdminSpreadFeeE6` (e.g. 20%) via `setPoolAdminFees` — this is within the cap.
2. Pool admin calls `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` — no cap check, succeeds.
3. For every swap touching that bin, the effective fee is `20% + 6.55% = 26.55%`, exceeding the hard cap by 6.55 percentage points.

The excess fee accrues to LPs (not the admin), but traders pay it as additional input tokens beyond what the cap system is supposed to guarantee.

### Impact Explanation
Traders executing swaps through affected bins pay effective fees up to **26.55%** when the protocol's hard cap is 20%. This is a direct loss of trader input principal — the overcharge is real token value transferred out of the trader's pocket. The pool admin is semi-trusted "only inside caps"; this path lets them exceed the cap without any on-chain guard.

### Likelihood Explanation
Any pool admin can call `setPoolBinAdditionalFees` at any time with no timelock and no cap. The call is gated only by `onlyPoolAdmin`, which is the semi-trusted role. A malicious or compromised pool admin can exploit this immediately after pool creation.

### Recommendation
Add cap validation in `setPoolBinAdditionalFees` analogous to `setPoolAdminFees`. Introduce a `maxAdminBinAdditionalFeeE6` cap (or reuse `maxAdminSpreadFeeE6`) and enforce it:

```diff
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
+  if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
+  if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce that the sum of the base spread fee and the bin additional fee does not exceed the hard cap.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20%) — the hard cap.
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
4. A trader swaps through bin 0. The effective buy fee is `baseFeeX64 + 65535/1e6 ≈ 20% + 6.55% = 26.55%`.
5. The trader pays 26.55% in fees instead of the protocol-guaranteed maximum of 20%, losing the 6.55% excess as additional input tokens. [7](#0-6) [2](#0-1) [3](#0-2)
