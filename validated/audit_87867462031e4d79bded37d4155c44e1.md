### Title
Fragile Integer-to-ASCII Conversion in `format_short_integer` Silently Drops Trailing-Zero Digits, Producing Wrong Solidity Function Selectors — (`File: supporting_crates/solidity_abi/src/impls/codable_array.rs`)

### Summary

`format_short_integer` in `codable_array.rs` is the ZKsync OS analog of the `_uintToBytes` fragility reported against UMA's `Governor`. The function converts a fixed-array size constant `N` into its ASCII decimal representation for Solidity function-selector construction. A misplaced `if integer != 0` guard silently skips any digit position whose running remainder has become zero, causing trailing zeros to be dropped entirely. The result is that array sizes whose decimal representation contains trailing zeros (10, 20, 100, 110, 200, …) produce a truncated, incorrect string. The same proc-macro (`derive_selector.rs`) that emits every `*_SOLIDITY_ABI_SELECTOR` constant used for on-chain dispatch calls this function directly, so any function whose ABI includes a fixed-size array with such a size will have a permanently wrong selector baked into the binary.

### Finding Description

```
// supporting_crates/solidity_abi/src/impls/codable_array.rs  lines 10-39
pub(crate) const fn format_short_integer(
    buffer: &mut [u8],
    offset: &mut usize,
    integer: usize,
) -> Result<(), ()> {
    assert!(integer <= u8::MAX as usize);
    let mut integer = integer;
    let mut i = 0;
    let mut divisor = 100;
    while i < 3 {
        if integer != 0 {          // ← BUG: skips the digit when the
            let q = integer / divisor; //   running remainder is 0, even
            ...write q as ASCII...     //   when a non-zero digit already
        }                              //   appeared (trailing-zero case)
        integer %= divisor;
        divisor /= 10;
        i += 1;
    }
    Ok(())
}
```

For `N = 100`: output is `"1"` (should be `"100"`).
For `N = 0`: output is `""` (empty — should be `"0"`).

The proc-macro in `derive_selector.rs` (lines 118-123) calls `format_short_integer` directly when it encounters an `Array<T, N>` parameter, then feeds the resulting string into a `keccak256` to produce the 4-byte selector constant:

The constant is then emitted as `pub const FUNC_SOLIDITY_ABI_SELECTOR: u32 = const { … };` and used at runtime to match incoming calldata.

### Impact Explanation

Any system-contract or bootloader function whose ABI signature contains a fixed-size array `T[N]` where `N` has trailing decimal zeros (10, 20, 100, 110, 200, …) will have its 4-byte selector computed from a corrupted type string. Concretely:

- A function declared as `foo(uint256[10])` will have its selector computed from the string `"foo(uint256001)"` (the `'0'` units digit is dropped), not from `"foo(uint256[10])"` or even the zero-padded `"foo(uint256010)"`.
- Any external caller (L1 bridge contract, user transaction, or system upgrade) that constructs the call using the correct Solidity ABI selector `keccak256("foo(uint256[10])")[:4]` will receive a revert or be silently misrouted.
- Two distinct array sizes can produce the same truncated string (e.g., `N=100` → `"1"` and `N=1` → `"001"` are different, but `N=10` → `"01"` and `N=1` → `"001"` are also different — however the corruption is still present and the selector is wrong for all trailing-zero sizes).

The class of impact is **EVM semantic mismatch / state-transition bug**: the bootloader's ABI dispatch layer permanently encodes wrong selectors, making affected entry points unreachable via the correct Solidity ABI, or reachable via an unintended selector that could collide with another function.

### Likelihood Explanation

The bug is latent in every compilation of the codebase. It activates the moment any production function is annotated with the `#[selector_codable]` (or equivalent) derive macro and has a fixed-size array parameter whose size has a trailing zero. The attacker does not need to do anything special: sending a correctly ABI-encoded call to such a function is sufficient to trigger a dispatch failure. The entry path is fully unprivileged (any transaction sender).

### Recommendation

Replace the digit-extraction loop with a correct implementation that does not suppress digits based on the running remainder. A simple approach:

```rust
pub(crate) const fn format_short_integer(
    buffer: &mut [u8],
    offset: &mut usize,
    integer: usize,
) -> Result<(), ()> {
    assert!(integer <= u8::MAX as usize);
    // Always emit exactly the digits needed (no leading zeros, handle 0 explicitly)
    let mut tmp = [0u8; 3];
    let mut len = 0;
    let mut n = integer;
    if n == 0 {
        tmp[0] = b'0';
        len = 1;
    } else {
        while n > 0 {
            tmp[len] = b'0' + (n % 10) as u8;
            len += 1;
            n /= 10;
        }
        // digits are in reverse order; reverse in place
        tmp[..len].reverse();
    }
    for &byte in &tmp[..len] {
        let s = unsafe { core::str::from_utf8_unchecked(core::slice::from_ref(&byte)) };
        append_ascii_str(buffer, offset, s)?;
    }
    Ok(())
}
```

Additionally, add unit tests covering `N = 0, 1, 9, 10, 11, 99, 100, 110, 200, 255` and verify the output matches the canonical Solidity ABI type-string format (no leading zeros, correct trailing zeros).
