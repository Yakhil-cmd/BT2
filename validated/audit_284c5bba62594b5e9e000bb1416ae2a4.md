### Title
Unchecked `app_pk` Parameter in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the CKD protocol, the `app_pk` (application ElGamal public key) parameter passed to `ckd()` and used inside `compute_signature_share()` is never committed to in the protocol transcript, never broadcast for cross-participant verification, and never checked against any stored value. A malicious participant can silently substitute a different `app_pk` when computing their share contribution. Because the coordinator blindly sums all participant contributions, the final `CKDOutput` is arithmetically corrupted, and the legitimate TEE application's `unmask(app_sk)` call returns a wrong value instead of `msk · H(pk ‖ app_id)`.

---

### Finding Description

`ckd()` accepts `app_pk: PublicKey` as a caller-supplied argument and passes it directly into `compute_signature_share()`:

```
src/confidential_key_derivation/protocol.rs  lines 66–116
```

Inside `compute_signature_share()` (lines 148–182), each participant computes:

```
big_s  = hash_point * private_share          // H(pk || app_id) · xᵢ
big_c  = big_s + app_pk * y                  // ElGamal encryption of big_s under app_pk
```

The protocol then normalises and sends `(λᵢ · big_y, λᵢ · big_c)` to the coordinator. The coordinator sums all contributions (lines 50–55) and returns the aggregate `CKDOutput`.

There is **no protocol step** that:
- broadcasts `app_pk` so participants can verify they all use the same value, or
- includes `app_pk` in the session transcript / commitment hash, or
- checks `app_pk` against any field in `key_pair` (the only stored trusted object).

This is structurally identical to the Tracer `claimReceipts()` bug: a caller-supplied context parameter (`market` / `app_pk`) is used directly without being verified against the trusted stored object (`receipt.tracer` / `key_pair`).

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an unusable cryptographic result.**

Let honest participants use `app_pk_legit` and one malicious participant use `app_pk_evil ≠ app_pk_legit`. After the coordinator sums all contributions:

```
sum_C = Σ_honest(big_sᵢ + app_pk_legit · yᵢ)·λᵢ  +  (big_s_evil + app_pk_evil · y_evil)·λ_evil
```

When the TEE application calls `unmask(app_sk_legit)`:

```
sum_C − app_sk_legit · sum_Y
  = msk · H(pk‖app_id)  +  (app_pk_evil − app_pk_legit) · y_evil · λ_evil
```

The residual term `(app_pk_evil − app_pk_legit) · y_evil · λ_evil` is non-zero with overwhelming probability, so the TEE receives a wrong secret. A single malicious participant is sufficient to corrupt the output for all honest parties in every CKD invocation they participate in.

---

### Likelihood Explanation

**Medium-High.** Any participant in the CKD protocol is an unprivileged library caller who supplies their own `app_pk` argument. No out-of-band enforcement prevents them from passing a different value. The attack requires no cryptographic break, no leaked key, and no trusted-host compromise — only the ability to call `ckd()` with a chosen `app_pk`.

---

### Recommendation

Include `app_pk` in the protocol transcript so all participants commit to the same value before computing their shares. Concretely:

1. Add `app_pk` to the session-ID broadcast (alongside `my_session_id` in `do_keyshare`-style round 1), or perform a dedicated broadcast of `app_pk` at the start of `run_ckd_protocol`.
2. Each participant verifies that every other participant's broadcast `app_pk` matches their own before proceeding to `compute_signature_share`.
3. Alternatively, derive `app_pk` from a value already committed to in `key_pair` (e.g., a field added during DKG), removing the free parameter entirely — analogous to the Tracer recommendation of retrieving `market` from `receipt.tracer` rather than accepting it as a parameter.

---

### Proof of Concept

**Setup:** 3-of-3 CKD session. Participants P1, P2 are honest; P3 is malicious.

1. P1 and P2 call `ckd(..., app_pk_legit, ...)` and compute correct `(norm_big_y, norm_big_c)` contributions.
2. P3 calls `ckd(..., app_pk_evil, ...)` where `app_pk_evil` is an arbitrary attacker-chosen G1 point. Inside `compute_signature_share` (line 174), P3 computes `big_c = big_s + app_pk_evil * y` instead of `big_s + app_pk_legit * y`.
3. The coordinator (any of the three) receives all three `(big_y, big_c)` pairs and sums them without any `app_pk` consistency check (lines 50–55).
4. The TEE application calls `ckd_output.unmask(app_sk_legit)` (line 54 of `mod.rs`): `big_c − app_sk_legit · big_y`. The result is `msk · H(pk‖app_id) + (app_pk_evil − app_pk_legit) · y_evil · λ_evil`, which is not the expected confidential key.
5. The existing test `test_ckd` (lines 211–283 of `protocol.rs`) would fail if P3 used a different `app_pk`, confirming the corruption — but the production protocol has no equivalent runtime check. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
