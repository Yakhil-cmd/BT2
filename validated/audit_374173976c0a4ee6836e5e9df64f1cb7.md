### Title
EVM Pause Lacks Grace Period After Unpausing, Enabling Frontrunning of Protective EVM Transactions - (File: fvm/evm/stdlib/contract.cdc)

### Summary
The Flow EVM contract's pause mechanism blocks all state-mutating EVM operations but leaves read-only query functions (`dryRun`, `dryCall`, `dryCallWithSigAndArgs`) fully available. After unpausing, there is no grace period. An unprivileged attacker can monitor EVM DeFi positions during the pause using these read-only functions, then immediately execute liquidations in the first block after unpausing — before affected users can add collateral, repay debt, or otherwise protect their positions.

### Finding Description

`EVM.isPaused()` reads a boolean from `/storage/evmOperationsPaused` and is enforced as a `pre` condition in every state-mutating EVM entry point:

- `EVMAddress.deposit` — blocks FlowToken deposits to any EVM address
- `CadenceOwnedAccount.withdraw` — blocks FlowToken withdrawals from EVM
- `CadenceOwnedAccount.call`, `EVM.run`, `EVM.batchRun` — blocks all EVM contract calls
- `CadenceOwnedAccount.deploy` — blocks EVM contract deployments
- `CadenceOwnedAccount.depositNFT/withdrawNFT`, `depositTokens/withdrawTokens` — blocks all bridge operations [1](#0-0) [2](#0-1) [3](#0-2) 

However, the contract explicitly exempts read-only functions from the pause guard:

```cadence
/// Note: dry-run functions are intentionally exempt from the isPaused() guard —
/// they perform no state mutations and remain available in read-only mode.
access(all)
fun dryRun(tx: [UInt8], from: EVMAddress): Result {
    return InternalEVM.dryRun(tx: tx, from: from.bytes) as! Result
}
``` [4](#0-3) 

`CadenceOwnedAccount.dryCall` and `dryCallWithSigAndArgs` are similarly exempt — they carry no `isPaused()` guard: [5](#0-4) 

`EVM.isPaused()` itself is a `view` function that any Cadence script or transaction can call at zero cost: [6](#0-5) 

The pause is toggled by the Governance Committee via `PauseBridgeTransaction` / a multi-sig Cadence transaction that writes `true`/`false` to `/storage/evmOperationsPaused`. There is **no grace period** mechanism anywhere in the contract after the flag is cleared. [7](#0-6) 

### Impact Explanation

EVM DeFi protocols (lending, margin trading, perpetuals) deployed on Flow EVM may accumulate positions that become liquidatable during a pause — because users cannot add collateral (`EVMAddress.deposit` blocked), repay debt (`coa.call` blocked), or close positions (`EVM.run` blocked). The moment the Governance Committee unpauses, there is no window for users to act first. An attacker who has been monitoring health factors via `dryRun`/`dryCall` throughout the pause can submit liquidation transactions in the very first block after unpausing, capturing the liquidation bonus before any victim can protect their position. This constitutes a direct cross-VM asset loss for all affected EVM DeFi users.

### Likelihood Explanation

The EVM pause is an expected operational tool ("for maintenance or any situation that requires special governance handling" per the contract comment). Any pause of non-trivial duration — during which EVM asset prices move — creates liquidatable positions. The attacker requires no special privileges: `dryRun` and `dryCall` are `access(all)` and callable from any Cadence transaction or script. The attack cost is a single Flow transaction fee. [8](#0-7) 

### Recommendation

After the `evmOperationsPaused` flag is cleared, record the unpause block height (or timestamp) and enforce a grace period during which liquidation-type EVM calls remain blocked. Concretely, `isPaused()` could return `true` for an additional N blocks after the flag is set to `false`, giving users time to restore their positions before liquidators can act.

### Proof of Concept

1. Governance Committee stores `true` at `/storage/evmOperationsPaused` — EVM enters pause mode.
2. During the pause, a user's EVM lending position accrues interest and crosses the liquidation threshold. The user cannot call `coa.call(repay...)` or `EVMAddress.deposit(collateral)` — both revert with `"EVM operations are temporarily paused"`.
3. An unprivileged attacker calls `EVM.dryRun(liquidateTx, from: attackerAddr)` repeatedly throughout the pause to confirm the position is liquidatable and to determine the exact liquidation parameters. This call succeeds because `dryRun` is explicitly exempt from the pause guard.
4. Governance Committee stores `false` at `/storage/evmOperationsPaused` — EVM unpauses.
5. In the same block (or the next), the attacker submits a real `EVM.run(liquidateTx, coinbase: attackerAddr)` transaction. Because there is no grace period, this executes before the victim's protective transaction can be included.
6. The victim's collateral is seized at a discount; the attacker captures the liquidation bonus. The victim suffers a cross-VM asset loss they could not have prevented. [9](#0-8) [10](#0-9)

func PauseBridgeTransaction(

	service flow.Address,

	txScript, _ := bridge.GetCadenceTransactionCode("cadence/transactions/bridge/admin/pause/update_bridge_pause_status.cdc", bridgeEnv, env)
	return flow.NewTransactionBodyBuilder().
		SetScript(txScript).
