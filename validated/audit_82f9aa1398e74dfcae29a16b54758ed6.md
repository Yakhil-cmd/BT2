### Title
Silent Skip of IBC Transfer for Sub-`10^10` evmDenom Amounts Causes Unmatched Bank Debit in `SendCroToIbcHandler` - (`x/cronos/keeper/evmhandlers/send_cro_to_ibc.go`, `x/cronos/keeper/ibc.go`)

---

### Summary

`SendCroToIbcHandler.Handle` unconditionally transfers `amount` evmDenom from the bridge contract to `sender` via `bankKeeper.SendCoins`, then calls `IbcTransferCoins`. When `amount < 10^10`, `IbcTransferCoins` silently `continue`s past the IBC transfer branch and returns `nil`. `Handle` interprets the nil return as success and also returns nil. The result is a committed bank debit from the contract with no corresponding IBC packet — a permanent accounting mismatch.

---

### Finding Description

**Step 1 — `Handle` debits the contract unconditionally** [1](#0-0) 

`bankKeeper.SendCoins(contractAddr, sender, coins)` runs first. If it succeeds, the evmDenom is already in `sender`'s account. Only then is `IbcTransferCoins` called.

**Step 2 — `IbcTransferCoins` silently skips for small amounts** [2](#0-1) 

For the `evmDenom` case:
```
amount8decRem = amount % 10^10
amountToBurn  = amount - amount8decRem
```
When `amount = 9_999_999_999` (< `10^10 = 10_000_000_000`):
- `amount8decRem = 9_999_999_999`
- `amountToBurn = 0`
- `amountToBurn.IsZero()` → `continue` → **no burn, no IBC send, no error**

`IbcTransferCoins` returns `nil`.

**Step 3 — `Handle` sees nil and returns nil** [3](#0-2) 

The EVM transaction commits successfully. The contract's evmDenom balance is permanently reduced by `amount`, the sender's balance is permanently increased by `amount`, and zero IBC packets are created.

**`TenPowTen` definition:** [4](#0-3) 

**Existing test confirms the silent-skip behavior is known but not guarded against in `Handle`:**

The test case `"Correct address with too small amount EVM token"` in `ibc_test.go` passes `amount=123`, expects `nil` error, and has an empty `postCheck` — confirming `IbcTransferCoins` returns nil silently. [5](#0-4) 

No analogous guard or test exists in `TestSendCroToIbcHandler` for the sub-`10^10` case. [6](#0-5) 

---

### Impact Explanation

Every evmDenom unit debited from the CroBridge contract via `Handle` must be matched by an IBC transfer or a revert. When `amount < 10^10`, neither happens: the sender gains evmDenom for free (relative to the bridge contract's reserve), and the bridge contract's bank balance is permanently drained without any cross-chain transfer. This is an unauthorized balance/accounting change for CRO — a Critical-scope impact.

---

### Likelihood Explanation

The CroBridge contract is a public-facing bridge. Any user who can call it with an amount in the range `[1, 9_999_999_999]` (a valid, non-zero uint256) triggers this path. The authorized-contract check only gates which *contract* can emit the event, not which *user* can interact with the contract. The bridge contract holds evmDenom reserves by design, making the drained asset immediately available.

---

### Recommendation

In `IbcTransferCoins`, replace the silent `continue` with an explicit error return when `amountToBurn.IsZero()`:

```go
if amountToBurn.IsZero() {
    return fmt.Errorf("amount %s is too small to bridge (minimum 10^10 %s)", c.Amount, evmParams.EvmDenom)
}
```

Alternatively, enforce the minimum-amount check at the top of `Handle` before `bankKeeper.SendCoins` is called, so the debit never occurs for sub-threshold amounts.

---
