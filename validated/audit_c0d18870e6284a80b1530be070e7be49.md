### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Single-Participant Signature Creation — (File: src/frost/mod.rs)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` accept any `ReconstructionLowerBound` value, including 1 or 0, without enforcing a minimum of 2. The DKG layer explicitly rejects `threshold < 2`, but the FROST signing layer has no equivalent guard. A malicious coordinator can invoke the signing protocol with `threshold = 1`, reducing the effective security to a single-participant scheme and producing a valid threshold signature alone.

---

### Finding Description

`ReconstructionLowerBound` is a plain newtype over `usize` with no invariant enforcement: [1](#0-0) 

`assert_sign_inputs`, the canonical pre-flight validator for FROST signing, checks only that `threshold ≤ participants.len()` and that `participants.len() ≥ 2`. It never checks `threshold ≥ 2`: [2](#0-1) 

The same gap exists in `presign`: [3](#0-2) 

By contrast, the DKG layer explicitly rejects `threshold < 2`: [4](#0-3) 

And the OT-based triple generation does the same: [5](#0-4) 

The FROST signing layer is the only entry point that omits this guard, creating an inconsistency that is silently exploitable.

---

### Impact Explanation

In FROST, the threshold controls Lagrange interpolation during share combination. With `threshold = 1`, the Lagrange coefficient for a single participant evaluates to 1 over the full secret polynomial, meaning that participant's share alone reconstructs the aggregate signing key contribution. A malicious coordinator who:

1. Calls `assert_sign_inputs` with `threshold = 1` (passes without error),
2. Includes only themselves in the participant list,
3. Runs the signing protocol,

produces a cryptographically valid threshold signature for an arbitrary message using only their own share — bypassing the t-of-n security model entirely. This matches the **Critical** allowed impact: *unauthorized creation of a valid threshold signature for attacker-chosen inputs*.

---

### Likelihood Explanation

The coordinator role is explicitly part of the threat model (the prompt lists "malicious coordinator" as a valid attacker). The coordinator controls session setup, including the `participants` slice and the `threshold` value passed to `assert_sign_inputs`. No out-of-band mechanism prevents them from supplying `threshold = 1`. The `ReconstructionLowerBound` type provides no compile-time or runtime barrier. Exploitation requires no cryptographic break, no leaked keys, and no external dependency — only a single API call with a crafted parameter.

---

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` and `validate_triple_inputs` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Optionally, enforce the invariant at the type level by giving `ReconstructionLowerBound` a checked constructor that rejects values below 2, so the gap cannot recur in future protocol additions.

---

### Proof of Concept

```rust
use threshold_signatures::frost::assert_sign_inputs;
use threshold_signatures::participants::Participant;
use threshold_signatures::ReconstructionLowerBound;

let p0 = Participant::from(0u32);
let p1 = Participant::from(1u32);
let participants = vec![p0, p1];

// threshold = 1 — passes assert_sign_inputs without error
let result = assert_sign_inputs(
    &participants,
    ReconstructionLowerBound::from(1usize), // no lower-bound check fires
    p0,   // me (also the coordinator)
    p0,   // coordinator
);
assert!(result.is_ok()); // succeeds — no ThresholdTooSmall error

// A malicious coordinator now runs the FROST signing protocol
// with threshold=1 and participants=[p0], producing a valid
// signature using only their own share.
```

The call succeeds because `assert_sign_inputs` only checks `threshold ≤ participants.len()` (1 ≤ 2 ✓) and `participants.len() ≥ 2` (2 ≥ 2 ✓), never `threshold ≥ 2`. [6](#0-5) [7](#0-6)

    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,

    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });

    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
