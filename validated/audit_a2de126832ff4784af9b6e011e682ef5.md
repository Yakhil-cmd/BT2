### Title
Permanent Loss of Bridged ERC-20 Tokens When `ft_transfer` Promise Fails in `ExitToNear` Precompile Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

### Summary
The `ExitToNear` precompile burns a user's ERC-20 tokens before scheduling an asynchronous `ft_transfer` promise to the corresponding NEP-141 contract. When the `error_refund` compile-time feature is disabled, no failure-handling callback is attached to that promise. If the `ft_transfer` fails — for example because the recipient NEAR account is not registered with the NEP-141 token, or because the NEP-141 token (e.g. bridged USDC) has blacklisted the recipient — the burned ERC-20 tokens are permanently destroyed with no recovery path.

### Finding Description
In `engine-precompiles/src/native.rs`, the `ExitToNear::run` function constructs a `callback_args` struct and decides whether to attach a callback to the outgoing promise:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

For the standard ERC-20 legacy `ft_transfer` path, `transfer_near` is `None` and — when `error_refund` is not compiled in — `refund` is also `None`. This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, triggering the branch that emits a bare `PromiseArgs::Create` with **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

The ERC-20 burn happens synchronously inside the EVM (in `EvmErc20.sol`'s `withdrawToNear`), before the promise is even dispatched. If the subsequent `ft_transfer` NEAR promise fails, there is no `exit_to_near_precompile_callback` invocation, no `refund_on_error` call, and no re-mint of the burned tokens.

The test suite explicitly acknowledges this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

### Impact Explanation
Any user who calls `withdrawToNear` on a bridged ERC-20 token (e.g. bridged USDC) and whose chosen recipient NEAR account either (a) has not registered a storage deposit with the NEP-141 contract, or (b) is blacklisted by a NEP-141 token that implements blacklist functionality, will have their ERC-20 tokens permanently burned with no refund. The funds are irrecoverable. This matches **Critical — Permanent freezing of funds**.

### Likelihood Explanation
- The entry point (`withdrawToNear`) is callable by any unprivileged EVM user with a balance of the bridged token.
- NEP-141 tokens such as bridged USDC can and do implement blacklisting; a blacklisted recipient causes `ft_transfer` to fail on the NEAR side.
- Even without blacklisting, a simple typo in the recipient account ID (pointing to an unregistered account) triggers the same failure.
- The `error_refund` feature is a non-default compile-time opt-in; without it the vulnerability is present in every build that omits the flag.
