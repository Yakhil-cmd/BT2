### Title
`setPoolAdminFeeDestination` Changes Fee Destination Without Collecting Accrued Fees, Causing Old Destination to Lose Owed Admin Fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` immediately overwrites `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. All fees that have built up under the old destination are silently redirected to the new one on the next `collectFees` call, causing a direct loss of owed funds for the old fee destination.

---

### Finding Description

Every other fee-mutating path in the factory flushes accrued fees **before** changing the fee configuration:

- `setPoolAdminFees` calls `collectFees(…, poolAdminFeeDestination[pool])` before updating `poolFeeConfig[pool]`.
- `setPoolProtocolFee` does the same. [1](#0-0) 

`setPoolAdminFeeDestination`, however, skips this flush entirely:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;          // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
``` [2](#0-1) 

At the time of the call, the pool holds two categories of uncollected admin fees:

1. **Notional fees** — tracked in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` inside the pool.
2. **Spread fees** — the surplus `balance - binTotals - notionalFee` that `collectFees` splits by the stored fee ratios. [3](#0-2) 

`collectPoolFees` (callable by anyone) passes the **current** `poolAdminFeeDestination[pool]` to the pool:

```solidity
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]          // ← now points to new destination
    );
}
``` [4](#0-3) 

After `setPoolAdminFeeDestination` executes, every subsequent `collectPoolFees` call sends **all previously accrued admin fees** to the new destination. The old destination receives nothing.

This is the direct structural analog to M-14: just as `changeHatToggle()` should call `checkHatToggle()` before switching the toggle address to sync the hat's status, `setPoolAdminFeeDestination()` must call `collectFees()` before switching the destination to sync accrued fees.

---

### Impact Explanation

The old admin fee destination loses all fees that accrued before the destination change. Because `collectPoolFees` is permissionless, a pool admin can:

1. Let fees accumulate in the pool.
2. Call `setPoolAdminFeeDestination(pool, attackerAddress)`.
3. Immediately call `collectPoolFees(pool)`.

All accrued admin fees — both notional and spread — are transferred to `attackerAddress`. The original fee destination (e.g., a DAO treasury or revenue-sharing contract) receives zero. This is a direct, on-chain loss of protocol-owed assets.
