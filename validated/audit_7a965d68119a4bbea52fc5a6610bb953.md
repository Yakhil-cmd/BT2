### Title
Pool admin bypasses global spread-fee cap via uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` to the pool with **no cap validation**, while the parallel `setPoolAdminFees` path is strictly bounded by `maxAdminSpreadFeeE6`. A pool admin can therefore impose per-bin spread fees up to `type(uint16).max` (65 535 / 1e6 ≈ 6.5 %) on every bin, regardless of what the factory owner has set as the admin spread-fee ceiling.

### Finding Description

The factory enforces a hard ceiling on admin-controlled spread fees through `maxAdminSpreadFeeE6` (itself bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20 %).

`setPoolAdminFees` correctly enforces this cap:

```solidity
// MetricOmmPoolFactory.sol  line 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, however, passes the caller-supplied values straight through with **no bounds check**:

```solidity
// MetricOmmPoolFactory.sol  line 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index:

```solidity
// MetricOmmPool.sol  line 464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

These per-bin values are **directly added to the oracle base fee** in every swap price calculation:

```solidity
// MetricOmmPool.sol  line 540-541  (getSellAndBuyPrices, mirrors live swap path)
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The global `spreadFeeE6` (the capped value) is used only for LP-fee accounting inside `SwapMath`, not for the price paid by the user. The per-bin additional fees are the sole admin-controlled lever that directly widens the bid/ask spread seen by traders.

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = addFeeSellE6 = 65535` (≈ 6.5 %) on every bin. Every subsequent swap pays a spread that is 6.5 % wider than the oracle mid-price, regardless of what `maxAdminSpreadFeeE6` the factory owner has configured. If the factory owner has set `maxAdminSpreadFeeE6 = 0` to prevent the admin from charging any spread, the admin can still extract up to 6.5 % per swap via this path. The excess fee accrues to LPs (the admin can simultaneously be an LP), constituting a direct loss of user principal on every trade.

### Likelihood Explanation

The pool admin is a semi-trusted role that is expected to operate only within factory-enforced caps. The bypass requires a single permissioned call with no timelock. Any pool whose admin is compromised or acts adversarially can immediately impose the maximum uncapped per-bin fee on all bins. The attack is silent (no revert, no event distinguishable from a legitimate fee adjustment) and reversible at will.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. The simplest fix is to bound the sum of the global admin spread fee and the per-bin additional fee against `maxAdminSpreadFeeE6`:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    uint24 adminSpread = poolFeeConfig[pool].adminSpreadFeeE6;
    if (uint256(adminSpread) + addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (uint256(adminSpread) + addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the factory owner can configure independently.

### Proof of Concept

1. Factory owner deploys a pool with `maxAdminSpreadFeeE6 = 0` (admin may charge no global spread fee).
2. Pool admin calls `factory.setPoolAdminFees(pool, 1, 0)` → reverts with `AdminFeeTooHigh` as expected.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` → **succeeds**.
4. A user swaps through bin 0. The effective buy/sell fee is `baseFeeX64 + 65535/1e6` ≈ oracle spread + 6.5 %, far above the factory owner's intended zero-admin-spread policy.
5. The excess fee is retained in the pool as LP profit; the admin (also an LP) collects it via `collectPoolFees`. [5](#0-4) [6](#0-5)
