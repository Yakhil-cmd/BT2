### Title
Unprotected `new()` Initialization Allows Any Caller to Hijack Engine Ownership Before the Deployer - (File: engine/src/contract_methods/admin.rs)

### Summary
The Aurora Engine's `new()` initialization function performs no caller authentication. Any NEAR account can race to call it between contract deployment and the deployer's own `new()` invocation, setting an attacker-controlled `owner_id` and gaining full administrative control over the engine before any user funds are deposited.

### Finding Description
`new()` in `engine/src/contract_methods/admin.rs` guards re-initialization only by checking whether state already exists:

```rust
if state::get_state(&io).is_ok() {
    return Err(b"ERR_ALREADY_INITIALIZED".into());
}
``` [1](#0-0) 

There is no check on `env.predecessor_account_id()` or `env.current_account_id()`. The public WASM export in `lib.rs` likewise adds no guard:

```rust
pub extern "C" fn new() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::admin::new(io, &env)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
``` [2](#0-1) 

The deployment process, as shown in the integration test harness, performs the WASM deployment and the `new()` call in **separate transactions**:

```rust
let res = engine_contract
    .call("new")
    .args_borsh((chain_id, engine_contract.id(), engine_contract.id(), 1_u64))
    ...
    .transact()
    .await?;
``` [3](#0-2) 

This creates an observable on-chain window — between the block that deploys the WASM and the block that calls `new()` — during which the contract is live but uninitialized. Any account that submits a `new()` call first wins, because the subsequent legitimate call will be rejected with `ERR_ALREADY_INITIALIZED`.

The `NewCallArgs` accepted by `new()` includes `owner_id`, `chain_id`, and `upgrade_delay_blocks`. An attacker who wins the race sets all of these to attacker-controlled values. [4](#0-3) 

### Impact Explanation
Once the attacker is the recorded `owner_id`, every owner-gated function becomes available to them:

- `stage_upgrade` + `deploy_upgrade` — replace the engine WASM with a malicious binary that drains all ETH and ERC-20 balances held by the contract. This is **direct theft of user funds at rest** (Critical).
- `pause_contract` — permanently freeze all user funds (Critical).
- `set_eth_connector_contract_account` — redirect the connector to a malicious contract, intercepting all future deposits and withdrawals (Critical). [5](#0-4) [6](#0-5) 

### Likelihood Explanation
NEAR's transaction pool is publicly observable. An attacker watching for a new deployment of the Aurora Engine WASM (identifiable by the well-known account ID `aurora`) can submit a `new()` call targeting the same account in the same or the immediately following block. The window exists whenever deployment and initialization are not batched into a single atomic NEAR batch transaction. The test harness confirms this is the current pattern: deploy in one transaction, call `new()` in a second. No special privilege, leaked key, or social engineering is required — only a standard NEAR account and the ability to submit a transaction.

### Recommendation
Restrict `new()` to the contract's own account by asserting `env.predecessor_account_id() == env.current_account_id()` (i.e., the call must originate from the contract itself, which is only possible inside a batch that also deploys the code). pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
