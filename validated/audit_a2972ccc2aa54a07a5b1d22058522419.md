### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required - (File: L1/starkware/solidity/stake/RewardSupplier.sol)

### Summary
The `tick()` function in `RewardSupplier.sol` is `external payable` and accepts native ETH to cover two L1→L2 message fees. When `requiredMinting()` returns zero (no pending mint requests), the entire execution body is skipped and any ETH sent by the caller is neither consumed nor refunded, permanently locking it in the contract. There is no ETH withdrawal or recovery mechanism in the contract.

### Finding Description
`RewardSupplier.tick()` is declared `external payable` and expects the caller to supply ETH to fund two downstream payable calls:

1. `bridge().depositWithMessage{value: msgFee}(...)` — bridge fee for the STRK deposit to L2
2. `messagingContract().sendMessageToL2{value: msgFee}(...)` — fee for the total-supply update message

The fee split is:
```solidity
uint256 msgFee = msg.value / 2;          // line 125
bridge().depositWithMessage{value: msgFee}(...);
msgFee = msg.value - msgFee;             // line 135
messagingContract().sendMessageToL2{value: msgFee}(...);
```

Both calls are guarded by `if (amountToMint > 0)` at line 111. When `requiredMinting()` returns `(0, 0)` — i.e., there are no pending L2→L1 mint-request messages — the function exits immediately after the check, consuming zero ETH but also issuing zero refund:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // ... all ETH-spending logic here ...
    }
    // ← silent return; msg.value is now stuck
}
```

The `RewardSupplier` contract inherits `ProxySupportImpl` and `Identity` but contains no `receive()`, `fallback()`, or ETH-rescue function. ETH that accumulates in the contract balance is irrecoverable.

### Impact Explanation
Any ETH sent to `tick()` when `amountToMint == 0` is permanently frozen in the `RewardSupplier` contract. Because there is no withdrawal path, the loss is irreversible. This matches the **High** impact category: *Permanent freezing of funds*.

### Likelihood Explanation
`tick()` is a public, permissionless function — any address can call it. A caller cannot atomically verify that `requiredMinting()` will return a non-zero value at the exact block their transaction is included, because:

- L2→L1 messages are consumed by other callers between submission and inclusion.
- The L2 `RewardSupplier` may not have emitted any new mint requests since the last `tick()`.

A caller who sends ETH to ensure the two downstream payable calls succeed (analogous to a buyer over-paying to ensure a Dutch-auction order fills) will have their ETH locked if the condition is false at inclusion time. This is a realistic, unprivileged, reachable path.

### Recommendation
Add a refund of unused ETH at the end of `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // ... existing logic ...
    }
    // Refund any unused ETH to the caller.
    if (address(this).balance > 0) {
        (bool ok, ) = msg.sender.call{value: address(this).balance}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, require `msg.value == 0` when `amountToMint == 0`, or compute the exact required fee before accepting ETH.

### Proof of Concept

1. The L2 `RewardSupplier` has no pending mint requests (all prior requests have been fulfilled or none have been issued yet).
2. Alice calls `RewardSupplier.tick{value: 1 ether}()` to trigger minting, sending ETH to cover the two message fees.
3. `requiredMinting()` returns `(0, 0)` because `messagingContract().l2ToL1Messages(msgHash) == 0`.
4. The `if (amountToMint > 0)` branch is not entered.
5. `tick()` returns. Alice's 1 ETH is now held by `RewardSupplier` with no mechanism to recover it.
6. The ETH is permanently locked. [1](#0-0) [2](#0-1) [3](#0-2)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
