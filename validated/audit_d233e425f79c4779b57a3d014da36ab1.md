I have enough information to write up the finding. Let me confirm the key details from the code.### Title
`max_l2_gas_amount` Limit Not Enforced for Declare Transactions in Gateway Stateless Validation - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidatorConfig::max_l2_gas_amount` is a configured gateway admission limit described as "Maximum allowed L2 gas amount for transactions." It is enforced for `Invoke` and `DeployAccount` transactions, but is explicitly skipped for `Declare` transactions via a hard-coded type-check branch. Any caller can submit a `Declare` transaction with an arbitrarily large `l2_gas.max_amount` (up to `u64::MAX`) and the gateway's stateless validator will accept it unconditionally, bypassing the configured cap.

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds`, the `max_l2_gas_amount` check is guarded by a type-check that silently no-ops for `Declare` transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1_210_000_000` (≈1.21 billion gas units): [2](#0-1) 

The config schema documents this field as applying to all transactions: [3](#0-2) 

The bypass is not a dead code path — it is explicitly tested and asserted to pass: [4](#0-3) 

The test `valid_l2_gas_amount_on_declare` constructs a `Declare` transaction with `l2_gas.max_amount = GasAmount(200)` against a config with `max_l2_gas_amount = 100` and asserts `Ok(())`. The corresponding negative test `test_invalid_max_l2_gas_amount` only covers `TransactionType::DeployAccount` and `TransactionType::Invoke`, confirming `Declare` is intentionally excluded from enforcement: [5](#0-4) 

### Impact Explanation

The `max_l2_gas_amount` limit exists to bound the L2 gas a single transaction may claim at the gateway admission layer, before any stateful or blockifier processing. For `Declare` transactions this bound is never applied. An unprivileged submitter can craft a `Declare` transaction with `l2_gas.max_amount = u64::MAX` (18,446,744,073,709,551,615) and the stateless validator will accept it.

Downstream, `verify_can_pay_committed_bounds` in the blockifier computes `max_amount × max_price_per_unit` to check balance sufficiency. With `max_amount = u64::MAX` and `max_price_per_unit` set to the minimum accepted price (`min_gas_price = 8_000_000_000`), the product overflows a `u64` and wraps or saturates depending on the arithmetic implementation, potentially producing a committed fee that is far smaller than the true declared obligation. This means a well-funded account is not the only prerequisite — the overflow itself may allow the balance check to pass with a normal account balance, admitting the transaction into the mempool with a declared L2 gas bound that no downstream component can safely interpret.

The impact matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The trigger is fully unprivileged: any user submitting a `Declare` transaction via the RPC/gateway endpoint can set `l2_gas.max_amount` to any value. No special role, key, or prior state is required. The stateless validator is the first and only gate that should enforce this limit; once past it, no other stateless check re-applies `max_l2_gas_amount` to `Declare` transactions.

### Recommendation

Remove the `Declare`-exemption branch and apply the `max_l2_gas_amount` check uniformly to all transaction types, as the config description states:

```rust
// Remove the type-check exemption:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher L2 gas ceiling (e.g., due to Sierra compilation overhead), introduce a separate `max_l2_gas_amount_declare` config field with an appropriate value, rather than leaving the bound entirely unenforced.
