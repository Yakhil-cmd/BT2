### Title
`skip_stateful_validations` Admits Unsigned Invoke Transactions for Undeployed Accounts, Bypassing `__validate__` Signature Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (signature verification) for any invoke transaction with nonce=1 sent to an undeployed account, provided that *any* transaction from that address is currently in the mempool. Because the mempool is public and the check does not verify that the submitter is the legitimate account owner, an unprivileged attacker can submit an invoke with a garbage signature for a victim's undeployed account, have it admitted to the mempool without signature verification, and block the victim's legitimate nonce=1 invoke.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

This means `StatefulValidator::validate` — which calls the account's `__validate__` entry point — is never invoked for this transaction at the gateway stage.

**The broken assumption**

The comment at line 441–443 states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

`account_tx_in_pool_or_recent_block` checks whether *any* transaction from the address is in the pool — it does not verify that the submitter of the current invoke is the same party who submitted the deploy_account. Because the mempool is public, any observer can see a pending deploy_account for address X and immediately submit an invoke(nonce=1, sender=X, signature=garbage).

**No other check closes the gap**

`validate_by_mempool` (called before `skip_stateful_validations`) only checks nonce ordering and fee escalation — it does not verify signatures: [3](#0-2) 

`validate_nonce` for invoke transactions allows any nonce in `[account_nonce, account_nonce + max_allowed_nonce_gap]`, so nonce=1 with account_nonce=0 passes as long as `max_allowed_nonce_gap >= 1`: [4](#0-3) 

Stateless validation only checks signature *length*, not cryptographic validity: [5](#0-4) 

**Attack path**

1. Victim broadcasts `deploy_account` for address X → enters mempool.
2. Attacker calls `add_tx` with `Invoke { nonce=1, sender=X, signature=[0x41, 0x42] }`.
3. Stateless check: signature length ≤ `max_signature_length` → passes.
4. `validate_nonce`: `0 ≤ 1 ≤ 0 + max_gap` → passes.
5. `validate_by_mempool`: nonce 1 ≥ account_nonce 0, no existing nonce-1 tx → passes.
6. `skip_stateful_validations`: nonce=1, account_nonce=0, account in mempool → returns `true`.
7. `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate = false` → `__validate__` **not called**.
8. Attacker's transaction enters the mempool occupying the nonce=1 slot for address X.
9. Victim's legitimate nonce=1 invoke is rejected (duplicate nonce) or requires fee escalation to displace the attacker's transaction.

---

### Impact Explanation

The gateway admits a transaction with an unverified (garbage) signature into the mempool, violating the invariant that only transactions passing `__validate__` are accepted. Concretely:

- The victim's legitimate nonce=1 invoke is blocked until the attacker's transaction is either evicted (TTL) or displaced via fee escalation.
- The attacker can automate this against every new `deploy_account` observed in the public mempool, systematically disrupting all new account deployments.
- This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

(The attacker's transaction will fail during block execution when `__validate__` is called with the real execution flags, so funds are not directly stolen; the harm is admission-level integrity and targeted DoS.)

---

### Likelihood Explanation

- Requires zero privileged access — any RPC caller can submit transactions.
- The mempool is public; deploy_account transactions are visible to all peers.
- The attack is trivially automatable: watch for deploy_account, immediately submit a nonce=1 invoke for the same sender address.
- No cryptographic work is required.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a **deploy_account** transaction (not just any transaction) is pending for the address. For example, expose `deploy_account_tx_in_pool(address) -> bool` from the mempool and use that as the skip condition. This preserves the UX intent (deploy + invoke in one shot) while closing the spoofing path.

---

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
