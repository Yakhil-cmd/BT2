### Title
Incorrect Mint Recipient in `mint_base_token_hook` — Tokens Always Credited to Intermediary Caller Instead of Intended User - (`system_hooks/src/call_hooks/mint_base_token.rs`)

---

### Summary

The `mint_base_token_hook` at address `0x7100` always credits minted base tokens to its immediate caller (`L2_BASE_TOKEN_ADDRESS`, `0x800a`) rather than to the intended end-user recipient. The hook's calldata interface accepts only a 32-byte amount with no recipient address field, making it structurally impossible to mint tokens to anyone other than the L2BaseToken contract itself. This is the direct analog of the external report's pattern: a privileged intermediary calls a hook/callback, and the hook uses the intermediary's identity as the beneficiary instead of the actual intended recipient.

---

### Finding Description

In `system_hooks/src/call_hooks/mint_base_token.rs`, the `mint_base_token_hook` function extracts `caller` from the `ExternalCallRequest` and passes it as the `beneficiary` all the way down to `mint_nominal_token_value`:

```
mint_base_token_hook  →  mint(caller)  →  mint_nominal_token_value(&caller, amount)
``` [1](#0-0) 

The `mint()` function receives `caller: B160` and passes it directly as the beneficiary: [2](#0-1) 

`mint_nominal_token_value` then increases the balance of `beneficiary` (which is always `L2_BASE_TOKEN_ADDRESS = 0x800a`): [3](#0-2) 

The hook's calldata interface is exactly 32 bytes — only the amount, no recipient address: [4](#0-3) 

The hook enforces that only `L2_BASE_TOKEN_ADDRESS` can call it: [5](#0-4) 

This means every successful invocation of the hook mints tokens to `0x800a` (the L2BaseToken system contract), never to the actual user who should receive them. The hook's own doc comment acknowledges this: *"Increases the **caller's** nominal token balance by the specified amount."* [6](#0-5) 

The existing test `test_mint_base_token_hook` confirms this behavior — it asserts that `L2_BASE_TOKEN_ADDRESS`'s balance increases, not any user's balance: [7](#0-6) 

By contrast, the bootloader's `transfer_from_treasury` (used for L1→L2 deposits) correctly accepts an explicit `to: &B160` parameter and credits the intended recipient directly: [8](#0-7) 

---

### Impact Explanation

Any time the L2BaseToken contract (`0x800a`) calls the mint hook to mint tokens on behalf of a user (e.g., during a protocol upgrade or bridge finalization), the minted tokens are credited to `0x800a` itself rather than to the user. The user receives nothing. The tokens accumulate at the L2BaseToken system contract address with no mechanism in the hook to redistribute them to the correct recipient. This constitutes a direct, permanent loss of base tokens for the intended recipient.

---

### Likelihood Explanation

The hook is described as being used for "system-level mints" and "during upgrades only." However, it is a live, callable system hook registered at `0x7100`. Any protocol upgrade or bridge operation that relies on the L2BaseToken contract calling this hook to credit a user will silently send those tokens to `0x800a` instead. The L2BaseToken contract is a system contract whose behavior is fixed; it cannot compensate for the hook's missing recipient parameter after the fact. The likelihood is high whenever the hook is actually invoked for user-facing minting.

---

### Recommendation

Add a recipient address to the hook's calldata interface (e.g., 64 bytes: `address recipient || uint256 amount`), and use that address as the `beneficiary` in `mint_nominal_token_value` instead of `caller`. This mirrors how `transfer_from_treasury` accepts an explicit `to` parameter:

```diff
- if calldata_len != 32 {
+ if calldata_len != 64 {
      return Ok(Err("Mint hook failure: mint called with invalid calldata"));
  }

+ let recipient = B160::try_from_be_slice(&calldata[12..32])
+     .ok_or(...)?;
  let nominal_token_value = U256::from_be_slice(&calldata[32..64]);

- mint_nominal_token_value(resources, system, &caller, &nominal_token_value)?;
+ mint_nominal_token_value(resources, system, &recipient, &nominal_token_value)?;
```

---

### Proof of Concept

The existing test already demonstrates the bug. `test_mint_base_token_hook` calls the hook from `L2_BASE_TOKEN_ADDRESS` and verifies that `L2_BASE_TOKEN_ADDRESS`'s own balance increases — confirming tokens go to the intermediary caller, not any user: [9](#0-8) 

A targeted PoC would:
1. Record `user_balance` before the call.
2. Have `L2_BASE_TOKEN_ADDRESS` call `0x7100` with a 32-byte amount.
3. Assert `user_balance` is unchanged and `L2_BASE_TOKEN_ADDRESS` balance increased by the mint amount — demonstrating the wrong recipient received the tokens.

fn test_mint_base_token_hook() {
    let mut tester = TestingFramework::new().with_minted_tokens_to_treasury();

    // L2 base token address is the only address allowed to call the mint hook
    let l2_base_token_address = address!("000000000000000000000000000000000000800a");
    // Mint hook address (0x7100)
    let mint_hook_address = address!("0000000000000000000000000000000000007100");
    let mint_amount = alloy::primitives::U256::from(3000000000000000000u64); // 3 ETH

    // Check initial balance of L2_BASE_TOKEN_ADDRESS is zero
    let initial_balance = tester
        .get_account_properties(&l2_base_token_address)
        .balance;

    // Prepare calldata: 32 bytes containing the mint amount as U256 big-endian
    let calldata = mint_amount.to_be_bytes::<32>().to_vec();

    // Create transaction from L2_BASE_TOKEN_ADDRESS to MINT_HOOK_ADDRESS
    let tx = L1TxBuilder::new()
        .from(l2_base_token_address)
        .to(mint_hook_address)
        .input(calldata)
        .value(alloy::primitives::U256::ZERO) // No ETH value needed for mint
        .gas_price(1000)
        .gas_limit(200_000)
        .build();

    let output = tester.execute_block(vec![tx]);

    // Assert transaction succeeded
    assert!(output.tx_results.iter().cloned().enumerate().all(|(i, r)| {
        let success = r.clone().is_ok_and(|o| o.is_success());
        if !success {
            println!("Transaction {} failed with: {:?}", i, r)
        }
        success
    }));

    // Check that the caller's (L2_BASE_TOKEN_ADDRESS) balance was increased by the mint amount
    let final_balance = tester
        .get_account_properties(&l2_base_token_address)
        .balance;

    let actually_minted_amount = final_balance
        .checked_sub(initial_balance)
        .expect("Some tokens should be minted");
    assert_eq!(
        actually_minted_amount, mint_amount,
        "Minted amount should match the requested mint amount"
    );
}
