### Title
WalletConnect `chia_spendClawbackCoins` Bulk Spend Under Single Ambiguous Confirmation - (`packages/gui/src/electron/commands/Commands.ts`)

### Summary

A WalletConnect dApp granted `chia_spendClawbackCoins` permission can send an array of N coin IDs in a single request. The Electron confirmation dialog renders the entire `coin_ids` array as an opaque JSON blob with no per-coin amount, direction (clawback vs. claim), or counterparty address. A user who approves the single dialog authorizes `spend_clawback_coins` for all N coins simultaneously, potentially clawing back payments they intended to let complete or claiming coins they did not intend to claim yet.

### Finding Description

**Command schema** — `chia_wallet.spend_clawback_coins` is defined with `coin_ids` typed as `'json'` and `fee` typed as `'bigint'`: [1](#0-0) 

**Humanization** — `humanizeParamValue` for `type: 'json'` calls `JSONBig.stringify(value, null, 2)`, producing a raw JSON array of opaque hex strings with no amount, direction, or counterparty enrichment: [2](#0-1) 

**Confirmation dialog** — `humanizeDappCommand` feeds these rows into the `Confirm` dialog, which renders each row as a plain label/value pair. The user sees `Coin Ids: ["0xabc...", "0xdef...", ...]` and a fee amount — nothing else: [3](#0-2) [4](#0-3) 

**Execution** — After a single user approval, `dispatchPairRequest` calls `sendCommand` with the full `parsedParams` (including all N coin IDs), which routes to `spend_clawback_coins` on the wallet daemon: [5](#0-4) 

**No per-coin limit or count validation** — `parseDappParams` accepts any JSON array for `coin_ids` without restricting its length or validating that each entry is a coin the user actually intends to spend: [6](#0-5) 

**Contrast with the normal UI** — The in-app `ClawbackClaimTransactionDialog` always operates on exactly one coin at a time, showing the specific XCH amount and counterparty address before calling `spendClawbackCoins({ coinIds: [coinId], ... })`: [7](#0-6) [8](#0-7) 

### Impact Explanation

`spend_clawback_coins` is used for two distinct operations: the **sender** clawing back a pending payment (reclaiming XCH they sent), and the **receiver** claiming a pending payment (collecting XCH they are owed). A malicious dApp can bundle all of a user's pending clawback coin IDs into one request. The user sees only an opaque hex array and a fee — no amounts, no direction, no counterparties. Approving the dialog:

- Causes the user to unintentionally claw back payments they intended to let complete (unauthorized reversal of outgoing XCH transfers).
- Or causes the user to claim all incoming clawback coins at once, potentially before they intended to.

This is a direct, on-chain asset impact: XCH is moved in ways the user did not meaningfully authorize.

### Likelihood Explanation

Any dApp that has been granted `chia_spendClawbackCoins` permission (a spending command that always requires confirmation per `isSpendCommand`) can trigger this. The user must approve the dialog, but the dialog provides no information to distinguish a 1-coin request from a 10-coin request. A user who has multiple pending clawback transactions is the target population. [9](#0-8) 

### Recommendation

1. **Enforce a maximum of one coin ID per WalletConnect `chia_spendClawbackCoins` call** in `parseDappParams`, or add a schema-level `maxItems: 1` constraint.
2. **If bulk is intentional**, enrich the confirmation dialog: for each coin ID, resolve and display the amount, direction (clawback/claim), and counterparty address before presenting the approval prompt.
3. **Add a `destructive: true` flag** to the `chia_wallet.spend_clawback_coins` command schema so the confirm button renders in the danger style.

### Proof of Concept

1. Establish a WalletConnect session with `chia_spendClawbackCoins` in the granted commands list.
2. Ensure the wallet has at least two pending clawback coins (e.g., two outgoing clawback-enabled sends still within their timelock).
3. From the dApp, send:
   ```json
   { "coin_ids": ["<coinId1>", "<coinId2>", "<coinId3>"], "fee": "0" }
   ```
4. Observe the Electron confirmation dialog: it shows `Coin Ids: ["<coinId1>", "<coinId2>", "<coinId3>"]` with no amounts or counterparties.
5. Approve the dialog.
6. Verify via the wallet RPC that all three clawback coins were spent in a single call, including any the user intended to let complete.

  // validate if all required params are provided and set the correct types
  for (const param of dappParams) {
    const { name, type, isOptional } = param;

    const value = nextParams[name];

    // verify if all isOptional !== true params are provided
    if (isOptional !== true && value === undefined) {
      throw new Error(`param is required: ${name}`);
    }

    // parse value to the correct types
    if (value !== undefined) {
      if (type === 'string') {
        nextParams[name] = String(value);
      } else if (type === 'number') {
        nextParams[name] = Number(value);
        if (Number.isNaN(nextParams[name])) {
          throw new Error(`Invalid number value for argument ${name}. Value: ${value}`);
        }
      } else if (type === 'bool') {
        if (typeof value !== 'boolean') {
          throw new Error(`Invalid boolean value for argument ${name}. Value: ${value}`);
        }
        nextParams[name] = value;
      } else if (type === 'bigint') {
        if (typeof value !== 'string' && typeof value !== 'number' && typeof value !== 'bigint') {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        if (typeof value === 'number' && !Number.isSafeInteger(value)) {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        if (typeof value === 'string') {
          const trimmed = value.trim();
          if (trimmed === '') {
            throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
          }
        }

        const bigintValue = BigInt(value);
        if (bigintValue.toString() !== value.toString()) {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        nextParams[name] = bigintValue;
      }
    }
  }
