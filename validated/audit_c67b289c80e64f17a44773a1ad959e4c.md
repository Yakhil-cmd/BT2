### Title
Gateway Admits Unsigned Invoke Transactions for Newly-Deploying Accounts via Validation Skip — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point (the account's on-chain signature check) for any invoke transaction with `nonce == 1` when the sender's on-chain nonce is `0` and the account has any transaction in the mempool or a recent block. Because the mempool is public, an unprivileged attacker can observe a victim's pending `deploy_account` transaction and immediately submit a crafted invoke with `nonce=1` and an arbitrary/invalid signature. The gateway admits this transaction without ever verifying the signature, occupying the victim's nonce-1 slot and causing the victim's legitimate invoke to be rejected as a duplicate nonce.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the nonce and resource-bound checks pass:

```
run_pre_validation_checks
  └─ validate_state_preconditions   // nonce range, resource bounds
  └─ validate_by_mempool            // duplicate-nonce check only, no sig check
  └─ skip_stateful_validations      // ← returns true → validate=false
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: !skip_validate` → `false`, so the blockifier's `StatefulValidator` returns `Ok(())` for the invoke without ever calling `__validate__`: [2](#0-1) 

The condition that triggers the skip is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
``` [3](#0-2) 

`account_tx_in_pool_or_recent_block` is a public mempool query that returns `true` for any account that has submitted a transaction (including a `deploy_account`): [4](#0-3) 

The stateless validator only checks the *length* of the signature, never its cryptographic validity: [5](#0-4) 

No other gateway-path check verifies the signature before the transaction is forwarded to the mempool.

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the public mempool can submit an invoke with `nonce=1`, any `sender_address` equal to the victim's contract address, and a syntactically valid but cryptographically invalid signature. The gateway admits this transaction to the mempool without signature verification. The victim's own legitimate invoke with `nonce=1` is subsequently rejected by the mempool as a `DuplicateNonce` error. The attacker's invalid invoke will eventually fail at `__validate__` during batcher execution (nonce not incremented), but the victim's transaction window is disrupted and must be resubmitted.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- The mempool is public; any observer can call `account_tx_in_pool_or_recent_block` or watch the P2P gossip layer to detect newly submitted `deploy_account` transactions.
- No privileged access is required; the attacker only needs to submit a standard RPC transaction.
- The race window is the time between the victim's `deploy_account` entering the mempool and being committed to a block — typically several seconds to minutes.
- The production config confirms the feature is active: `max_nonce_for_validation_skip = 0x1`. [6](#0-5) 

### Recommendation

The skip should only be granted to the same party that submitted the `deploy_account`. Two concrete options:

1. **Require a valid signature even when skipping `__validate__`**: perform a lightweight ECDSA check against the account's *expected* public key (derivable from the `deploy_account` constructor calldata) before admitting the invoke.
2. **Tie the skip to the deploy-account transaction hash**: require the invoke to carry a commitment (e.g., a hash of the pending `deploy_account` tx hash) in its calldata or a dedicated field, and verify that commitment in the gateway before granting the skip.

### Proof of Concept

1. Victim submits `deploy_account` for address `A` (nonce=0, valid signature). It enters the mempool.
2. Attacker queries `account_tx_in_pool_or_recent_block(A)` → `true`.
3. Attacker constructs an `RpcInvokeTransactionV3` with `sender_address=A`, `nonce=1`, `signature=[0x41, 0x41]` (two arbitrary felts, passes the `max_signature_length=4000` size check).
4. Attacker POSTs the transaction to the gateway's `add_transaction` endpoint.
5. Gateway path:
   - Stateless: signature length 2 ≤ 4000 → passes. [7](#0-6) 
   - `validate_state_preconditions`: nonce=1 ≤ account_nonce(0) + max_gap → passes (if gap ≥ 1). [8](#0-7) 
   - `validate_by_mempool`: no duplicate nonce=1 yet → passes.
   - `skip_stateful_validations`: nonce==1, account_nonce==0, `account_tx_in_pool_or_recent_block(A)`==true → returns `true`. [9](#0-8) 
   - `run_validate_entry_point`: `validate=false` → `__validate__` never called → `Ok(())`. [10](#0-9) 
6. Attacker's invalid invoke is added to the mempool with nonce=1 for address `A`.
7. Victim submits their own invoke (nonce=1, valid signature) → mempool returns `DuplicateNonce` → rejected.
8. Batcher executes `deploy_account` then attacker's invoke; `__validate__` fails (bad sig) → invoke rejected, nonce not incremented.
9. Victim must detect the failure and resubmit, suffering a sequencing delay and potential fee loss on retry.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
