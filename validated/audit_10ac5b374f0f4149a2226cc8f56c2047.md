### Title
Missing Access Control in `update_rewards()` Allows Any Caller to Block or Misdirect Block Reward Distribution — (File: src/staking/staking.cairo)

---

### Summary
The `update_rewards` function in `src/staking/staking.cairo` is specified in the protocol spec as callable **only by the starkware sequencer**, but the implementation enforces no such restriction. Any unprivileged address can call it, consuming the single per-block reward slot (enforced by `last_reward_block`) without distributing rewards, or misdirecting rewards to an attacker-chosen staker — permanently denying legitimate stakers their earned yield.

---

### Finding Description
The spec at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1449–1510 only calls `general_prerequisites()`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: unpaused + caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    // distributes rewards to staker + pools
```

`general_prerequisites()` at lines 1794–1797 only asserts the contract is unpaused and the caller is non-zero — **no sequencer identity check exists**:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

The global `last_reward_block` storage variable ensures only **one** call per block is accepted. An attacker exploits this by calling `update_rewards` before the sequencer each block with `disable_rewards: true`. This:
1. Writes `current_block_number` into `last_reward_block`.
2. Returns early — no rewards are calculated or transferred.
3. The sequencer's subsequent call reverts with `REWARDS_ALREADY_UPDATED`.

Alternatively, the attacker can call with `disable_rewards: false` and an attacker-controlled `staker_address`, causing rewards to flow to the wrong staker for that block.

---

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

By calling `update_rewards(staker_address: any_active_staker, disable_rewards: true)` once per block, an attacker permanently prevents any reward distribution for every block they target. Stakers and pool members accumulate zero `unclaimed_rewards_own` and pool balances receive no STRK from the reward supplier. The yield is not redirected — it is simply never credited, constituting a permanent freeze of unclaimed yield for all affected stakers and their delegators.

---

### Likelihood Explanation
**Medium.**

The attack requires the attacker's transaction to be ordered before the sequencer's `update_rewards` call within each block. On Starknet the sequencer controls ordering, but `update_rewards` is a regular external function with no protocol-level enforcement (no forced/system transaction type is used). A determined attacker who submits transactions at the start of every block, or who exploits any sequencer ordering policy, can reliably front-run the sequencer's call. The attack requires no capital, no privileged role, and no special knowledge beyond the contract's ABI.

---

### Recommendation
Add an explicit sequencer address check at the top of `update_rewards`, analogous to the pattern already used elsewhere in the codebase (e.g., `assert_caller_is_attestation_contract`, `assert_caller_is_staking_contract`):

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // store sequencer address at init; check here
    ...
}
```

Store the sequencer address in contract storage during initialization and expose a governance-gated setter for rotation.

---

### Proof of Concept

1. Deploy the system. Consensus rewards are active (`is_pre_consensus()` returns false).
2. Advance to a new block.
3. Attacker (any EOA) calls:
   ```
   IStakingRewardsManagerDispatcher { contract_address: staking }.update_rewards(
       staker_address: any_active_staker,
       disable_rewards: true,
   );
   ```
4. `last_reward_block` is updated to the current block; no rewards are distributed.
5. The sequencer's legitimate call to `update_rewards` for the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeat every block — stakers and pool members receive zero rewards indefinitely.
