### Title
Missing Caller Authorization in `BankContract::transfer` Allows Unauthorized Native Token Drain - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `transfer` method of the bank precompile (`BankContract`) at address `0x0000000000000000000000000000000000000064` accepts an arbitrary `sender` address as a parameter and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without ever verifying that `contract.Caller()` equals `sender`. Any unprivileged EVM contract can call `bank.transfer(victim, attacker, amount)` and drain the victim's native `evm/<calling_contract>` token balance without the victim's consent.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks three arguments from the call input:

```go
sender    := args[0].(common.Address)   // arbitrary, attacker-controlled
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
```

The denom is derived from the EVM caller:

```go
denom := EVMDenom(contract.Caller())   // "evm/<calling_contract>"
```

Then the transfer is executed directly:

```go
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is **no check** that `contract.Caller() == sender`. The only guard present is `checkBlockedAddr(to)`, which only prevents sending to module-blocked addresses.

Compare this with the `exec` helper in `utils.go` used by other precompiles (ICA, relayer), which explicitly enforces `caller == signer`:

```go
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
```

The bank precompile's `transfer` path has no equivalent guard.

The intended usage pattern (from `TestBank.sol`) is:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);   // ERC20 checks authorization
    return bank.transfer(msg.sender, recipient, amount);  // native transfer
}
```

The ERC20 contract is expected to pass `msg.sender` as `sender`, having already verified authorization. But the bank precompile does not enforce this — any contract can pass any address as `sender`.

The same structural flaw exists in the `BurnMethodName` case: `recipient` (the address to burn from) is taken from the input parameter, and there is no check that `contract.Caller() == recipient`, allowing any contract to burn native tokens from any address of its own denom.

### Impact Explanation

**Critical** — Unauthorized transfer of precompile-controlled assets (`evm/<contract>` native tokens).

`evm/<contract>` tokens are real, redeemable assets: a user who calls `moveToNative` on a legitimate ERC20 contract burns their ERC20 balance and receives an equivalent native `evm/<contract>` balance, which can be redeemed back via `moveFromNative`. These tokens have direct monetary value.

Attack outcome:
- A malicious contract operator can drain all users' native `evm/<malicious_contract>` balances in a single transaction per victim, with no victim interaction required after the initial deposit.
- The `burn` variant allows permanent destruction of victim balances.

### Likelihood Explanation

Any unprivileged EVM contract can call the bank precompile at `0x0000000000000000000000000000000000000064`. No special role, key, or governance access is required. The attacker only needs to have deployed a contract that users have previously interacted with (e.g., by calling `moveToNative`), which is a realistic scenario for any ERC20 token using the bank precompile pattern.

### Recommendation

Add a caller authorization check in the `TransferMethodName` case, enforcing that `contract.Caller() == sender`:

```go
case TransferMethodName:
    // ...
    sender := args[0].(common.Address)
    // ADD: verify the EVM caller is the sender
    if contract.Caller() != sender {
        return nil, errors.New("caller is not authorized to transfer from sender")
    }
    // ... rest of logic
```

Similarly, add `contract.Caller() == recipient` enforcement in the `BurnMethodName` case to prevent unauthorized burns.

This mirrors the pattern already used in `utils.go` (`exec`) for other precompiles.

### Proof of Concept

```solidity

pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract MaliciousToken {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Users call this to deposit (converts ERC20 → native evm/<this> tokens)
    function deposit(uint256 amount) external {
        // burn ERC20 from msg.sender (omitted for brevity)
        bank.mint(msg.sender, amount);  // victim now holds evm/<this> native tokens
    }

    // Step 2: Attacker calls this to drain any victim's native balance
    // No victim authorization required — sender is an arbitrary parameter
    function steal(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // Succeeds: denom = evm/<this>, from = victim, to = attacker
        // No check that msg.sender == victim
    }
}
```
