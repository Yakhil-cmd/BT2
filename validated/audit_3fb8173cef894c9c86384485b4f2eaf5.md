### Title
Signature Verification Bypass via `skip_stateful_validations` for Undeployed Accounts — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path unconditionally skips the `__validate__` entry point (which performs signature verification) for any Invoke transaction with `nonce == 1` targeting an address whose on-chain nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. Because this pool-membership check is not restricted to the submitter of the pending `deploy_account`, any unprivileged attacker can exploit a victim's pending `deploy_account` to inject an Invoke transaction with an arbitrary, invalid signature into the mempool without signature verification.

---

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have passed the account's `__validate__` entry point (signature check), or must be provably the first transaction from a freshly-deploying account submitted by the same principal.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

The three conditions are:
1. `tx.nonce() == 1` — fully attacker-controlled field in the submitted RPC transaction.
2. `account_nonce == 0` — the default value for any address that has never been deployed; satisfied for any victim address whose `deploy_account` is still pending.
3. `account_tx_in_pool_or_recent_block(sender)` — returns `true` if **any** transaction (not necessarily a `deploy_account` from the same submitter) is present in the pool or recent block for that address. [2](#0-1) 

When all three conditions hold, `skip_validate = true` is returned, which propagates to `run_validate_entry_point`: [3](#0-2) 

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate = false`, the blockifier's `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point: [4](#0-3) 

```rust
ApiTransaction::Invoke(_) => {
    let tx_context = ...;
    tx.perform_pre_validation_stage(self.state(), &tx_context)?;
    if !tx.execution_flags.validate {
        return Ok(());   // ← signature never checked
    }
```

**Attack path:**

1. Victim submits a `deploy_account` for address `X`; it enters the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. Attacker pre-funds address `X` with STRK (transfers to undeployed addresses are valid ERC-20 operations) so that `verify_can_pay_committed_bounds` inside `perform_pre_validation_stage` passes.
3. Attacker submits `Invoke { sender: X, nonce: 1, signature: <arbitrary garbage>, calldata: <anything> }`.
4. Gateway stateless validation passes (valid resource bounds, correct DA modes, etc.).
5. `validate_nonce` passes: `0 ≤ 1 ≤ max_allowed_nonce_gap`. [5](#0-4) 

6. `validate_by_mempool` passes (no duplicate hash, nonce within gap).
7. `skip_stateful_validations` returns `true` (conditions 1–3 all satisfied).
8. `run_validate_entry_point` runs with `validate = false`; `perform_pre_validation_stage` passes (nonce ≤ 1, balance covered by pre-funding); `__validate__` is **never called**.
9. The attacker's Invoke with an invalid signature is forwarded to the mempool via `mempool_client.add_tx`. [6](#0-5) 

**Parallel to the external bug:**

| External (Clearpool) | Sequencer analog |
|---|---|
| `currentAuction.tokenId == 0` (default) passes check | `account_nonce == Nonce(Felt::ZERO)` (default for undeployed address) passes check |
| `block.timestamp >= currentAuction.end` (0 ≥ 0) passes check | `tx.nonce() == Nonce(Felt::ONE)` (attacker-set) passes check |
| `currency.safeTransfer(0x0, 0)` doesn't block for some tokens | `account_tx_in_pool_or_recent_block` returns `true` for any account with any pending tx, not just the legitimate deployer |
| Anyone can call `processDebtClaim()` | Anyone can inject an unsigned Invoke into the mempool for any address with a pending `deploy_account` |

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An unprivileged attacker can inject Invoke transactions with completely invalid signatures into the mempool for any address that has a pending `deploy_account`. The admitted transaction will fail during block execution (because `AccountTransaction::new_for_sequencing` sets `validate: true` and `__validate__` is called at execution time), but the admission itself is unauthorized. Consequences include:

- **Mempool displacement:** The attacker can submit an Invoke with nonce=1 and a higher fee than the victim's legitimate Invoke, causing the victim's transaction to be evicted via fee-escalation replacement.
- **Mempool resource consumption:** Invalid transactions occupy pool capacity until execution failure.
- **Bootstrapping:** Once the attacker's nonce-1 Invoke is in the pool, `account_tx_in_pool_or_recent_block` remains `true` even after the victim's `deploy_account` is committed, extending the window for further abuse. [2](#0-1) 

---

### Likelihood Explanation

**Likelihood: Medium.**

- Requires a victim's `deploy_account` to be in the mempool (common during account onboarding).
- Requires the attacker to pre-fund the target address with a small STRK amount (trivially achievable; STRK transfers to undeployed addresses succeed).
- No privileged access, no special tooling; any user of the public gateway endpoint can trigger this.
- The attack window is bounded by the time between `deploy_account` submission and block commitment, but this window can be minutes long under normal sequencer operation.

---

### Recommendation

1. **Restrict the skip condition to deploy-account-specific pool membership.** Instead of checking `account_tx_in_pool_or_recent_block` (which returns `true` for any transaction type), add a dedicated `has_pending_deploy_account(address)` query to the mempool that only returns `true` when a `DeployAccount` transaction is present for that address.

2. **Alternatively, remove the nonce-1 bypass entirely** and require users to submit `deploy_account` first, then wait for confirmation before submitting the first Invoke. This eliminates the UX shortcut but closes the admission bypass.

3. **At minimum, document the invariant violation** so that downstream consumers (e.g., block builders) know that nonce-1 Invokes from zero-nonce accounts may have unverified signatures and must be re-validated at execution time.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
