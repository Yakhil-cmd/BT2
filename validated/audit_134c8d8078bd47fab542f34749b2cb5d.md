### Title
Silent Precompile Call Failure in `EvmErc20.sol::withdrawToNear` Causes Permanent Token Burn Without NEP-141 Transfer — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` (V1) constructs input for the `ExitToNear` precompile in a format that is structurally incompatible with the precompile's parsing logic when the `error_refund` feature is compiled in. The ERC-20 tokens are burned before the precompile is called, and the assembly `call()` return value is never checked. When the precompile fails due to the malformed input, the burn is irreversible and no NEP-141 transfer occurs, permanently destroying user funds.

---

### Finding Description

**Root cause — `EvmErc20.sol` (V1) `withdrawToNear`:**

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens burned first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    //                                    flag    32 bytes  N bytes
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

**Root cause — `EvmErc20V2.sol` (V2) `withdrawToNear` for comparison:**

```solidity
// EvmErc20V2.sol lines 53-64
bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
//                                    flag    20 bytes 32 bytes  N bytes
uint input_size = 1 + 20 + 32 + recipient.length;
``` [2](#0-1) 

V2 inserts the 20-byte `sender` address immediately after the flag byte. This matches the `error_refund` parsing path in the precompile.

**Root cause — `ExitToNear` precompile `parse_input` with `error_refund` enabled:**

```rust
// engine-precompiles/src/native.rs lines 778-785
#[cfg(feature = "error_refund")]
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);   // expects 20-byte refund address here
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))       // remaining = input after refund address
}
``` [3](#0-2) 

When V1 sends `\x01 | amount_b[32] | recipient[N]`, the precompile (with `error_refund`) reads:
- `refund_address` ← first 20 bytes of `amount_b` (wrong)
- remaining ← last 12 bytes of `amount_b` + `recipient`

Then for flag `0x01` (ERC-20 path):

```rust
// engine-precompiles/src/native.rs lines 758-771
0x1 => {
    let amount = parse_amount(&input[..32])?;   // needs 32 bytes
    let Recipient { receiver_account_id, .. } = parse_recipient(&input[32..])?;
    ...
}
``` [4](#0-3) 

The remaining slice is only `12 + len(recipient)` bytes. If `len(recipient) < 20`, the slice `&input[..32]` panics (index out of bounds in Rust), the precompile call returns failure to the EVM, and the assembly `call()` returns `0`. Since `res` is never checked and no `revert` follows, the `_burn` is already committed and permanent.

If `len(recipient) >= 20`, the precompile parses a completely wrong amount (derived from the tail of `amount_b` concatenated with the start of `recipient`) and sends it to a wrong NEAR account ID, causing partial or total fund loss.

**The `MIN_INPUT_SIZE` constants confirm the two incompatible modes:**

```rust
// engine-precompiles/src/native.rs lines 36-40
#[cfg(not(feature = "error_refund"))]
const MIN_INPUT_SIZE: usize = 3;
#[cfg(feature = "error_refund")]
const MIN_INPUT_SIZE: usize = 21;
``` [5](#0-4) 

V1 was designed for the non-`error_refund` format; V2 was designed for the `error_refund` format. Both contracts coexist in the production codebase with no enforcement that the correct version is paired with the correct engine build.

---

### Impact Explanation

When a user calls `withdrawToNear` on a V1 `EvmErc20` contract while the engine is compiled with `error_refund`:

1. `_burn` permanently destroys the user's ERC-20 tokens (Aurora-side supply decreases).
2. The precompile call fails silently (unchecked `res`).
3. No `ft_transfer` promise is created; the NEP-141 balance on NEAR is never reduced.
4. The user's tokens are gone with no recovery path.

This is a **connector/bridge accounting bug**: the ERC-20 mirror supply and the NEP-141 supply diverge permanently. The user suffers **permanent loss of funds** — matching the Critical impact tier (permanent freezing of funds).

---

### Likelihood Explanation

- Both `EvmErc20.sol` (V1) and `EvmErc20V2.sol` (V2) are present in the production contract directory. The factory mechanism (`factory_update_address_version`) allows both versions to be deployed simultaneously.
- Any V1 contract that was deployed before V2 was introduced and not subsequently upgraded remains vulnerable when the engine is compiled with `error_refund`.
- The entry path requires no privilege: any token holder can call `withdrawToNear` on a V1 ERC-20 contract.
- The `error_refund` feature is a compile-time flag; if enabled in the production WASM build, all V1 contracts are affected.

---

### Recommendation

1. **Check the return value** of the precompile `call()` in both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Revert if `res == 0` so the `_burn` is rolled back:
   ```solidity
   assembly {
       let res := call(...)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Enforce version compatibility**: the factory should prevent deploying V1 contracts when the engine is compiled with `error_refund`, or migrate all V1 contracts to V2 before enabling `error_refund
