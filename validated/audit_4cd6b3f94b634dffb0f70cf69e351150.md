### Title
Signature Bypass via `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures to Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (the account's on-chain signature check) for any invoke transaction with `nonce == 1` when the sender's on-chain nonce is `0` and the sender address already appears in the mempool or a recent block. An unprivileged attacker who observes a pending `deploy_account` transaction in the mempool can immediately submit an invoke transaction with an arbitrary/fake signature for that same account address and have it admitted to the mempool without any signature verification.

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold simultaneously:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

Condition 3 is satisfied as soon as *any* transaction from that address is in the mempool — including a `deploy_account` submitted by the legitimate owner.

**How the skip propagates**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the `validate` flag is checked and the `__validate__` call is short-circuited: [3](#0-2) 

`perform_pre_validation_stage` still runs (nonce increment on the ephemeral state, fee-bound check, balance check), but the account's `__validate__` entry point — which is the only place the cryptographic signature is verified — is never called. [4](#0-3) 

**Attack path (unprivileged, no special role required)**

1. Legitimate owner pre-funds address `A` and submits a `deploy_account` (nonce=0, valid signature). It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker submits `Invoke { sender_address: A, nonce: 1, signature: [0xDEAD, 0xBEEF] }` (completely fake signature).
3. Stateless validator passes — no signature check at this layer. [5](#0-4) 

4. Stateful validator:
   - `validate_nonce`: nonce=1 is within `[0, 0+200]` → passes. [6](#0-5) 

   - `validate_by_mempool`: no duplicate nonce=1 for A yet → passes.
   - `skip_stateful_validations`: all three conditions met → returns `true`.
   - `run_validate_entry_point`: `validate=false` → `__validate__` **never called**.

5. The attacker's invoke with a fake signature is admitted to the mempool.

**Broken invariant**

The invariant *"every transaction in the mempool has passed the account's signature verification"* is violated. The mempool rejects duplicate `(address, nonce)` pairs: [7](#0-6) 

So once the attacker's fake nonce=1 transaction occupies the slot, the legitimate owner's own nonce=1 invoke is rejected as `DuplicateNonce` until the fake transaction is executed (and fails) by the batcher.

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction.**

- The gateway admits an invoke transaction whose signature has never been verified.
- The attacker's fake transaction occupies the nonce=1 slot in the mempool, blocking the legitimate account owner from submitting their first post-deployment invoke until the fake transaction is executed and rejected by the batcher.
- The batcher wastes execution resources running `__validate__` on a transaction that will always fail.
- The attack is repeatable: every new `deploy_account` in the mempool is a target.

### Likelihood Explanation

- Fully unprivileged: requires only the ability to submit transactions to the gateway (public endpoint).
- The mempool is observable via P2P propagation; an attacker can automate detection of new `deploy_account` transactions and immediately race to submit a fake nonce=1 invoke.
- The attack is cheap (one transaction per victim account) and requires no special knowledge beyond the victim's address.

### Recommendation

1. **Do not skip signature verification entirely.** The `skip_stateful_validations` path should at minimum verify the transaction signature cryptographically against the account's declared public key (if available in state) before admitting the transaction.
2. **Scope the skip more tightly.** Instead of checking whether *any* transaction from the address is in the mempool, verify that a `deploy_account` transaction specifically (not just any transaction) is pending for that exact address, and that the incoming invoke's `tx_hash` was co-submitted in the same gateway request.
3. **Alternatively**, remove the skip entirely and require the `deploy_account` to be executed first before the invoke is admitted — accepting the UX trade-off in exchange for the security invariant.

### Proof of Concept

```
# Step 1: Pre-fund address 0xABCD with STRK.

# Step 2: Submit deploy_account (nonce=0, valid signature from owner).
POST /add_transaction
{ type: "DEPLOY_ACCOUNT", sender: 0xABCD, nonce: 0, signature: [valid_r, valid_s], ... }
# → Admitted. account_tx_in_pool_or_recent_block(0xABCD) == true.

# Step 3: Attacker submits invoke with FAKE signature (nonce=1).
POST /add_transaction
{ type: "INVOKE", sender: 0xABCD, nonce: 1, signature: [0xDEAD, 0xBEEF], calldata: [...], ... }
# → skip_stateful_validations returns true (nonce==1, account_nonce==0, address in mempool).
# → __validate__ is NOT called.
# → Transaction admitted to mempool with invalid signature.

# Step 4: Legitimate owner tries to submit their own nonce=1 invoke.
POST /add_transaction
{ type: "INVOKE", sender: 0xABCD, nonce: 1, signature: [real_r, real_s], ... }
# → Rejected: MempoolError::DuplicateNonce { address: 0xABCD, nonce: 1 }

# Step 5: Batcher eventually executes the fake invoke.
# → perform_pre_validation_stage runs (strict_nonce_check=true in batcher).
# → __validate__ runs and fails (invalid signature).
# → Transaction rejected; nonce slot freed.

# Step 6: Legitimate owner can now submit their nonce=1 invoke.
```

The exact gateway entry point is `GenericGateway::add_tx`, which calls `extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `skip_stateful_validations`: [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
