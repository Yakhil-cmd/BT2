### Title
`native_per_pubdata` Computed with Floor Division Causes Systematic Pubdata Fee Undercharge — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validation_impl.rs`, `native_per_pubdata` is computed using `wrapping_div` (floor/truncating division), while `native_per_gas` uses `div_ceil` (ceiling division). This asymmetry causes the protocol to charge less native resource for pubdata than intended whenever `pubdata_price % native_price != 0`. Through the `delta_gas` mechanism in `compute_gas_refund`, this translates directly into lower `gas_used` and thus lower fees paid to the operator — an exact structural analog to the JUSD borrow-fee-rate undercharge.

---

### Finding Description

**Root cause — `validation_impl.rs` line 142:**

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

Compare to `native_per_gas` in the same function, which correctly uses ceiling division:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

The same floor-division pattern is replicated in the API helper:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

**How the undercharge propagates:**

`native_per_pubdata` feeds into `get_resources_to_charge_for_pubdata`, which computes the native cost of pubdata:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

A lower `native_per_pubdata` means a lower native charge for pubdata. This flows into `compute_gas_refund`, where `native_used` is computed from the remaining native budget:

```rust
let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());
``` [5](#0-4) 

Then the `delta_gas` adjustment converts native consumption into gas:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
``` [6](#0-5) 

Because `native_used` is artificially low (pubdata was undercharged), `delta_gas` is smaller than it should be, so `gas_used` is smaller, and the fee paid to the operator is smaller.

The documentation confirms the intended formula:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed` [7](#0-6) 

---

### Impact Explanation

When `pubdata_price % native_price != 0`, `native_per_pubdata` is 1 less than the correct ceiling value. For a transaction generating `N` pubdata bytes:

- **Native undercharge** = `N × 1` native units
- **Gas undercharge** = `N / native_per_gas` gas units
- **Fee undercharge** = `(N / native_per_gas) × gas_price`

**Concrete example:**
- `pubdata_price = 1001`, `native_price = 1000`
- `native_per_pubdata = floor(1001/1000) = 1` (correct: `ceil = 2`)
- `native_per_gas = ceil(1000/1000) = 1`
- Transaction generates 1000 pubdata bytes, EVM gas used = 50,000
- Actual `gas_used` = 51,000; correct `gas_used` = 52,000
- At `gas_price = 1000`: user underpays by **1,000,000 wei** (~2% of total fee)

The operator systematically collects less than the intended pubdata fee on every transaction where `pubdata_price % native_price != 0`.

---

### Likelihood Explanation

The condition `pubdata_price % native_price != 0` is the normal case in production. Both `pubdata_price` and `native_price` are operator-set parameters reflecting real-world costs (L1 data availability prices and RISC-V cycle costs respectively). There is no mechanism enforcing they be exact multiples. Any unprivileged user submitting a transaction that writes to storage, emits events, or otherwise generates pubdata triggers this path. No special access is required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` in both locations:

**`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` line 142:**
```rust
// Before (floor division — undercharges pubdata):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

**`api/src/helpers.rs` line 427:**
```rust
// Before:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;

// After:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

### Proof of Concept

1. Operator sets `pubdata_price = 1001`, `native_price = 1000`, `eip1559_basefee = 1000`.
2. User submits an EIP-1559 transaction with `max_fee_per_gas = 1000`, `gas_limit = 200_000`, calling a contract that writes to 30 storage slots (generating ~960 pubdata bytes).
3. `native_per_gas = ceil(1000/1000) = 1`.
4. `native_per_pubdata = floor(1001/1000) = 1` — correct value is `ceil = 2`.
5. Native charged for pubdata = `960 × 1 = 960`; correct = `960 × 2 = 1920`.
6. `native_used` is 960 units lower than it should be.
7. `delta_gas = (native_used / 1) - gas_used` is 960 lower than it should be.
8. `gas_used` is 960 lower; at `gas_price = 1000`, user underpays by **960,000 wei**.
9. Operator receives less than the intended pubdata fee on every such transaction.
