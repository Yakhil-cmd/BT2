### Title
Unchecked Return Value of Exit Precompile `call` Causes Permanent Token Burn Without Bridge Transfer - (`File: etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first calling `_burn` on the caller's balance and then invoking the Aurora exit precompile via a low-level assembly `call`. The return value `res` of that assembly `call` is assigned but **never checked**. If the precompile call fails (returns `0`), the burn is not reverted, the function returns successfully, and no cross-chain promise is ever scheduled — permanently destroying the user's ERC-20 tokens while leaving the corresponding NEP-141 tokens locked in Aurora's account on NEAR.

---

### Finding Description

In `EvmErc20.sol`, both exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // (1) tokens destroyed — irreversible

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // (2) res is NEVER checked — silent failure
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum`: [2](#0-1) 

And is replicated verbatim in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The EVM `call` opcode returns `0` on failure and `1` on success. In Solidity assembly, a failed `call` does **not** revert the caller's execution — it simply returns `0`. Because `res` is never inspected with a `require(res == 1)` or equivalent, any failure of the exit precompile is silently swallowed. The `_burn` that already executed is a committed EVM state change and is not rolled back.

The exit precompile (`ExitToNear` at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) can fail for multiple reasons visible in the production Rust code — for example, invalid recipient encoding, gas exhaustion inside the precompile, or an `ExitError` returned from the precompile's `run` method: [5](#0-4) 

When the precompile fails, no NEAR promise is scheduled, so no `ft_transfer` is ever issued to the NEP-141 contract. The NEP-141 tokens remain permanently locked in Aurora's account on NEAR with no recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` on a deployed `EvmErc20`/`EvmErc20V2` token contract loses their ERC-20 tokens permanently if the precompile call fails silently:

- ERC-20 tokens are burned on Aurora (irreversible).
- No NEP-141 `ft_transfer` promise is scheduled on NEAR.
- The NEP-141 tokens remain locked in Aurora's NEAR account with no mechanism to reclaim them.
- The transaction appears to succeed from the caller's perspective (no revert).

This breaks the invariant that every burn of an ERC-20 mirror token must be matched by a corresponding NEP-141 transfer, causing permanent insolvency of the bridge for the affected amount.

---

### Likelihood Explanation

**Medium.** The precompile can fail under several realistic conditions:

1. A user passes a malformed or oversized `recipient` bytes argument that causes the precompile to return an error (e.g., `ERR_INVALID_RECEIVER_ACCOUNT_ID`).
2. Gas forwarded to the precompile is insufficient (`ExitError::OutOfGas`).
3. Any future precompile-level error path that returns `Err(ExitError::...)`.

In all these cases the assembly `call` returns `0`, `res` is ignored, and the burn is finalized. Any EVM user (token holder) can trigger this path without any special privilege.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is `0`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to **all four** assembly `call` sites in `EvmErc20.sol` (`withdrawToNear`, `withdrawToEthereum`) and `EvmErc20V2.sol` (`withdrawToNear`, `withdrawToEthereum`). This ensures that if the precompile call fails, the entire transaction reverts, the `_burn` is rolled back, and the user's tokens are preserved.

---

### Proof of Concept

1. Deploy `EvmErc20` for a NEP-141 token; mint `1000` tokens to `alice`.
2. `alice` calls `withdrawToNear(malformed_recipient_bytes, 1000)` where `malformed_recipient_bytes` causes the `ExitToNear` precompile to return `ERR_INVALID_RECEIVER_ACCOUNT_ID`.
3. `_burn(alice, 1000)` executes — alice's ERC-20 balance drops to `0`.
4. The assembly `call` to the precompile returns `0` (failure); `res` is ignored.
5. The function returns without reverting.
6. Observed state: alice's ERC-20 balance = `0`; NEP-141 balance of Aurora's account on NEAR = unchanged (tokens not transferred); alice's NEP-141 balance = unchanged. Funds are permanently frozen. [1](#0-0) [6](#0-5)
