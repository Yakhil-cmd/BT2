### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Suppress Block Rewards for All Stakers - (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` is documented as callable only by the Starknet sequencer, but the implementation enforces no such restriction. Any unprivileged caller can invoke it with `disable_rewards: true` against any valid staker, consuming the single global `last_reward_block` slot for that block without distributing any rewards. Because the slot is written before the `disable_rewards` guard, the legitimate sequencer is permanently locked out of distributing rewards for that block to every staker in the protocol.

---

### Finding Description

`update_rewards` in `src/staking/staking.cairo` is the consensus-era reward distribution entry point. Its spec explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation only calls `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — no sequencer identity check exists. [1](#0-0) 

The function then writes `last_reward_block` to the current block number **before** branching on `disable_rewards`: [2](#0-1) 

`last_reward_block` is a single global storage variable shared across all stakers: [3](#0-2) 

The guard that prevents double-processing is: [4](#0-3) 

Because `last_reward_block` is written unconditionally at line 1485 before the `disable_rewards` early-return at line 1487, a caller who passes `disable_rewards: true` consumes the block's reward slot while distributing nothing. Any subsequent call by the legitimate sequencer for the same block reverts with `REWARDS_ALREADY_UPDATED`.

The only prerequisite the attacker must satisfy is supplying a currently-active staker address with non-zero balance at the current epoch — information that is entirely public on-chain via `get_stakers` or event logs. [5](#0-4) 

---

### Impact Explanation

Each Starknet block can only be processed once for reward distribution. When an attacker front-runs the sequencer with `disable_rewards: true`, the rewards that would have been minted and distributed for that block are permanently lost — they are never credited to `unclaimed_rewards_own` for any staker, and the `RewardSupplier` is never notified. Because `last_reward_block` is global, a single griefing call suppresses rewards for **all** stakers and all their delegation pools for that block.

This maps directly to the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- No privileged access is required; any EOA or contract can call `update_rewards`.
- The attacker only needs to know one valid, active staker address (trivially obtained from `NewStaker` events or `get_stakers`).
- The attack is repeatable every block at the cost of gas only.
- Front-running the sequencer on Starknet is feasible because transaction ordering within a block is observable in the mempool.

---

### Recommendation

Add an access-control check matching the spec. The simplest fix is to restrict `update_rewards` to the sequencer address (or a designated rewards-manager role):

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // add this
    ...
}
```

Alternatively, remove the `disable_rewards` parameter entirely and let the sequencer simply not call the function for blocks it wishes to skip, eliminating the griefing vector while preserving the intended semantics.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker observes a pending sequencer call to `update_rewards(staker_A, false)` in the mempool.
3. Attacker submits `update_rewards(staker_A, true)` with higher priority (or simply calls it first in the same block).
4. Execution path:
   - `general_prerequisites()` passes (contract unpaused, caller non-zero). [6](#0-5) 
   - `current_block_number > last_reward_block` passes (first call this block). [4](#0-3) 
   - Staker validity checks pass (attacker used a known active staker). [5](#0-4) 
   - `last_reward_block` is written to current block. [7](#0-6) 
   - `disable_rewards == true` → early return, zero rewards distributed. [8](#0-7) 
5. Sequencer's call arrives: `current_block_number > last_reward_block` is now **false** → reverts with `REWARDS_ALREADY_UPDATED`.
6. All stakers lose their block rewards for this block permanently. The attack can be repeated every block.
