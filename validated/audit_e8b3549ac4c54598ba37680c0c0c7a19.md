### Title
Unauthenticated `ft_on_transfer` Callback Allows Arbitrary ERC-20 Token Minting Without Backing NEP-141 Transfer — (`engine/src/contract_methods/connector.rs`)

---

### Summary

`ft_on_transfer` is a public NEAR entry point on the Aurora Engine that mints ERC-20 tokens (or ETH) in response to an incoming NEP-141 token transfer. It is intended to be called only by a legitimate NEP-141 FT contract as the callback leg of `ft_transfer_call`. However, the function performs no check that it is being invoked as part of a real `ft_transfer_call` flow. Any NEAR account that is registered as a NEP-141 token in Aurora can call `ft_on_transfer` directly, supplying an arbitrary `amount` and recipient, and Aurora will mint the corresponding ERC-20 tokens with no backing transfer having occurred.

---

### Finding Description

`ft_on_transfer` is exposed as a public WASM export with no `assert_private_call()` guard and no check that `promise_results_count() > 0` (which would confirm it is executing as a callback): [1](#0-0) 

Inside the function, the only branching logic is whether `predecessor_account_id` equals the configured ETH connector account: [2](#0-1) 

- If `predecessor_account_id == connector_account_id` → `receive_base_tokens` mints ETH.
- Otherwise → `receive_erc20_tokens` mints ERC-20 tokens for the NEP-141 token whose ID equals `predecessor_account_id`.

`receive_erc20_tokens` uses the fully attacker-controlled `args.amount` and `args.msg` (recipient address) to call the ERC-20 contract as the engine admin and mint tokens: [3](#0-2) 

The `FtOnTransferArgs.sender_id` field — the analog of the "initiator" in the Astaria report — is accepted from JSON input and is never verified against `predecessor_account_id` or any other trusted source: [4](#0-3) 

There is no mechanism that forces `ft_on_transfer` to be a callback of a prior `ft_transfer_call` promise. Compare this with every other sensitive callback in the codebase, which all call `env.assert_private_call()`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Critical — Direct theft / insolvency.**

An attacker who controls a NEAR account registered as a NEP-141 token in Aurora can:

1. Register their NEAR account as a NEP-141 token via `deploy_erc20_token` (no owner restriction on the Legacy path).
2. Call `ft_on_transfer` directly from that account, supplying an arbitrary `amount` and their own EVM address as recipient.
3. Aurora mints the corresponding ERC-20 tokens to the attacker with zero NEP-141 tokens ever transferred.

The minted ERC-20 tokens are unbacked, making the bridge insolvent. The attacker can then swap or bridge these tokens for real assets.

---

### Likelihood Explanation

**High.** The entry point is a standard public NEAR function callable by any NEAR account. Registering a NEP-141 token in Aurora requires no special permission. The attack requires only a NEAR account and knowledge of the `ft_on_transfer` ABI — no flash loan, no governance capture, no privileged key.

---

### Recommendation

Add a `promise_results_count() >= 1` guard (confirming the call is a callback) and/or `env.assert_private_call()` to `ft_on_transfer`, consistent with every other callback in the codebase. Alternatively, maintain an in-flight transfer nonce or flag set by `ft_transfer_call` and cleared by `ft_on_transfer`, so that direct calls without a preceding transfer are rejected.

---

### Proof of Concept

```
1. Attacker deploys NEAR contract at `evil-token.near`.
2. Attacker calls aurora::deploy_erc20_token({nep141: "evil-token.near"})
   → Aurora creates ERC-20 mirror at address X.
3. Attacker calls aurora::ft_on_transfer from `evil-token.near`:
     { sender_id: "anyone.near", amount: "1000000000000000000000", msg: "<attacker_evm_addr>" }
   predecessor_account_id = "evil-token.near"  ≠  connector_account_id
   → receive_erc20_tokens("evil-token.near", args, ...) is called
   → get_erc20_from_nep141(io, "evil-token.near") returns address X
   → ERC-20 at X mints 1e21 tokens to attacker_evm_addr
4. No NEP-141 tokens were ever transferred. Bridge is insolvent.
``` [7](#0-6) [8](#0-7)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
