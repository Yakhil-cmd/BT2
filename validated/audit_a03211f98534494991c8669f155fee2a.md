Looking at the staking reward calculation in the consensus path of `update_rewards`, I can identify a clear accounting bug analogous to H-1.

### Title
Consensus `update_rewards` passes staker's own balance as total stake denominator, granting disproportionate block rewards - (File: src/staking/staking.cairo)

### Summary

In `StakingRewardsManagerImpl::update_rewards`, the individual staker's balance (`staker_total_strk_balance`) is passed as the `strk_total_stake` parameter to `_update_rewards`. This is the denominator used in `calculate_staker_total_staking_power` to compute the staker's proportional share of block rewards. Passing the staker's own balance as the total stake causes the staker to receive the maximum possible STRK staking power (equal to `STRK_WEIGHT_FACTOR`) regardless of their actual share of the network, resulting in a disproportionate reward claim.

### Finding Description

In the pre-consensus path (`update_rewards_from_attestation_contract`), the global total stake is correctly fetched and passed:

```cairo
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_epoch_rewards,
    :strk_total_stake,   // ← global total
    :btc_total_stake,
    ...
``` [1](#0-0) 

In the consensus path (`update_rewards`), the individual staker's balance is passed instead:

```cairo
let (staker_total_strk_balance, staker_total_btc_balance) = self
    .get_staker_total_strk_btc_balance_at_epoch(...);
...
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_block_rewards,
    strk_total_stake: staker_total_strk_balance,   // ← staker's own balance, NOT global total
    btc_total_stake: staker_total_btc_balance,
    ...
``` [2](#0-1) 

`calculate_staker_total_staking_power` uses `strk_total_stake` as the denominator:

```cairo
let strk_staking_power = mul_wide_and_div(
    lhs: staker_strk_total_amount.to_amount_18_decimals(),
    rhs: STRK_WEIGHT_FACTOR,
    div: strk_total_stake.to_amount_18_decimals(),   // ← denominator
)
``` [3](#0-2) 

When `strk_total_stake == staker_strk_total_amount`, the division yields `STRK_WEIGHT_FACTOR` (= `STAKING_POWER_BASE_VALUE × (1 − ALPHA)` = 75% of `STAKING_POWER_BASE_VALUE`), the maximum possible STRK staking power, regardless of the staker's actual network share. [4](#0-3) 

### Impact Explanation

A staker with a tiny fraction of total stake (e.g., 0.1%) calls `update_rewards(self, false)` once per block. Because `strk_total_stake` equals their own balance, their computed staking power equals `STRK_WEIGHT_FACTOR` — the full STRK allocation weight. They receive 75% of the block's total STRK rewards instead of their proportional 0.075%. All other stakers receive zero rewards for that block because `last_reward_block` is updated, blocking further calls.

This is **theft of unclaimed yield** from all other stakers — a High-severity impact under the allowed scope. [5](#0-4) 

### Likelihood Explanation

`update_rewards` is a fully public function callable by any staker with no access restriction beyond holding a non-zero stake. The `last_reward_block` guard means only the first caller per block succeeds, so a rational attacker simply calls it at the start of every block. No privileged role, leaked key, or external dependency is required. [6](#0-5) 

### Recommendation

Replace the individual staker balance with the global total staking power in the consensus path, mirroring the pre-consensus path:

```cairo
// Fetch global totals, not the individual staker's balance
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_block_rewards,
    btc_total_rewards: btc_block_rewards,
    :strk_total_stake,
    :btc_total_stake,
    ...
```

### Proof of Concept

1. Network has 1,000,000 STRK total stake. Attacker stakes the minimum (e.g., 20,000 STRK = 2% of total).
2. Each block, attacker calls `update_rewards(attacker_address, false)`.
3. Inside `_update_rewards`, `calculate_staker_total_staking_power` is called with `strk_total_stake = 20,000` (attacker's balance) and `staker_strk_total_amount = 20,000`.
4. `strk_staking_power = 20,000 × STRK_WEIGHT_FACTOR / 20,000 = STRK_WEIGHT_FACTOR` (maximum).
5. Attacker receives 75% of block rewards; correct share would be 2% × 75% = 1.5%.
6. `last_reward_block` is updated; no other staker can claim rewards for that block.
7. Over time, attacker drains the majority of block rewards that should belong to other stakers. [7](#0-6)
