### Title
Unsigned Invoke Admitted to Mempool via `skip_stateful_validations` Signature Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator bypasses the `__validate__` entry point — the only place where a transaction's cryptographic signature is verified — for invoke transactions with nonce=1 targeting an undeployed account that has a pending deploy_account in the mempool. The sole guard is an address-presence check (`account_tx_in_pool_or_recent_block`), not a signature check. An unprivileged attacker who observes any pending deploy_account can submit an invoke with an arbitrary signature for that address and have it admitted to the mempool without any signature verification.

### Finding Description

**Root cause — address-presence check substituted for signer verification:**

In `skip_stateful_validations` at lines 429–461, the gateway checks two conditions before skipping `__validate__`:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The function checks that the sender address has *any* transaction in the mempool, but never verifies that the incoming invoke's signature is valid. This is structurally identical to the external bug: the code checks that an address matches stored data (the deploy_account's contract address is in the mempool) but does not verify that the submitter is the authorized signer.

**How the skip propagates to suppress `__validate__`:**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

`StatefulValidator::perform_validations` then returns early without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

**The mempool's `validate_tx` does not check signatures:**

`ValidationArgs` carries only address, nonce, tx_hash, tip, and max_l2_gas_price — no signature field:

```rust
pub struct ValidationArgs {
    pub address: ContractAddress,
    pub account_nonce: Nonce,
    pub tx_hash: TransactionHash,
    pub tx_nonce: Nonce,
    pub tip: Tip,
    pub max_l2_gas_price: GasPrice,
}
``` [4](#0-3) 

`Mempool::validate_tx` only calls `validate_incoming_tx` (nonce ordering) and `validate_fee_escalation` (fee replacement rules):

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

No existing guard in the mempool or gateway preserves the invariant that every admitted transaction carries a valid signature.

**Fee-escalation replacement attack surface:**

`account_tx_in_pool_or_recent_block` returns `true` as soon as the legitimate user's deploy_account is in the pool:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [6](#0-5) 

When `enable_fee_escalation=true`, `validate_fee_escalation` will accept the attacker's fake invoke as a replacement for the legitimate invoke if the attacker offers a higher fee, evicting the legitimate transaction from the mempool.

### Impact Explanation

**Matching impact: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An unprivileged attacker can:
1. Submit an invoke with an arbitrary (invalid) signature for any address that has a pending deploy_account in the mempool.
2. The gateway admits the transaction without running `__validate__`, violating the invariant that all mempool transactions carry valid signatures.
3. With fee escalation enabled, the attacker can evict the legitimate user's invoke by offering a higher fee, causing a targeted DoS against users deploying new accounts.
4. The attacker's fake invoke will fail at execution time (when the batcher runs `__validate__` with `validate=true`), but the legitimate user's transaction has already been displaced.

### Likelihood Explanation

- The mempool is publicly observable; pending deploy_account transactions are visible to any network participant.
- The attack requires no privileged access — any account can submit an invoke transaction.
- The attack window is bounded by the time between deploy_account submission and its inclusion in a block, which can be minutes under normal load.
- The attack is cheap: the attacker only needs to submit one fake invoke per target address.

### Recommendation

Even when `skip_stateful_validations` returns `true`, the gateway should not completely skip signature verification. Options:

1. **Lightweight ECDSA check at the gateway:** Extract the public key from the deploy_account's `constructor_calldata` (for standard account contracts) and verify the invoke's signature against the transaction hash before admitting it to the mempool.
2. **Restrict skip to same-submitter invokes:** Track that the invoke was submitted in the same RPC call or batch as the deploy_account, rather than relying solely on mempool presence.
3. **Defer skip to batcher only:** Remove the gateway-level skip entirely and let the batcher handle the ordering; the batcher already executes deploy_account before the invoke, so `__validate__` will succeed at execution time for legitimate pairs.

### Proof of Concept

```
1. Alice submits RpcDeployAccountTransactionV3 for address X
   (constructor_calldata = [alice_public_key])
   → Gateway runs __validate_deploy__, admits to mempool.

2. Alice submits RpcInvokeTransactionV3 from X, nonce=1, valid signature.
   → skip_stateful_validations: nonce==1, account_nonce==0,
     account_tx_in_pool_or_recent_block(X) == true → skip_validate=true
   → __validate__ NOT called. Admitted to mempool.

3. Attacker observes deploy_account for X in mempool.

4. Attacker submits RpcInvokeTransactionV3 from X, nonce=1,
   fee > Alice's fee, signature = [0x1, 0x2] (arbitrary garbage).
   → skip_stateful_validations: same conditions → skip_validate=true
   → __validate__ NOT called. Mempool validate_tx: nonce OK, fee escalation
     accepted (higher fee). Alice's invoke is EVICTED.

5. Batcher picks up attacker's invoke after deploy_account executes.
   → __validate__ runs with validate=true → FAILS (invalid signature).
   → Alice's invoke is gone; attacker's invoke reverts.
   → Alice must resubmit her invoke.
```

The broken invariant is: `∀ tx ∈ mempool, tx.signature is valid under tx.sender_address`. The `skip_stateful_validations` path breaks this invariant by substituting an address-presence check for a cryptographic signer check — the direct Sequencer analog of the Phantasia `CancelNftSale` missing `is_signer` check.
