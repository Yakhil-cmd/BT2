### Title
`max_receipt_size` Limit Bypassed via Post-Validation `output_data_receivers` Mutation in `promise_return` Path — (`runtime/runtime/src/lib.rs`)

### Summary

`LimitConfig::max_receipt_size` is validated at receipt creation time, but the runtime subsequently mutates a receipt's `output_data_receivers` field **after** that validation when processing a `promise_return`. This allows an unprivileged contract to produce receipts that exceed `max_receipt_size` and enter the chain, bypassing the protocol's receipt-size invariant. The bug is acknowledged in production code and tracked as issue #12606.

### Finding Description

When a contract executes `promise_return(C)` and the parent receipt has non-empty `output_data_receivers`, the runtime appends those receivers directly to receipt C's `output_data_receivers` field: [1](#0-0) 

This mutation occurs **after** `validate_receipt` has already accepted receipt C under `ValidateReceiptMode::NewReceipt`: [2](#0-1) 

The `ValidateReceiptMode` enum itself documents the resulting invariant break: [3](#0-2) 

Because the size check is gated on `mode == ValidateReceiptMode::NewReceipt` and the post-mutation receipt is never re-validated, the final serialized receipt can exceed `max_receipt_size`. The congestion-control layer has a separate workaround that clamps oversized receipts to `max_receipt_size` to prevent them from getting stuck: [4](#0-3) 

### Impact Explanation

Any account that can deploy a contract can exploit this path. The attacker crafts a receipt C whose serialized size equals exactly `max_receipt_size` (passing validation), then calls `promise_return(C)` from a callback that itself has `output_data_receivers`. The runtime appends those receivers to C, producing a receipt whose on-chain serialized size exceeds the protocol limit. The oversized receipt is forwarded to the destination shard and stored in state, violating the `max_receipt_size` invariant. This corrupts the receipt-size accounting used by congestion control and stateless validation (state-witness size bounds), and forces every future node to handle receipts it should have rejected.

### Likelihood Explanation

Reachable by any unprivileged user who can deploy a WASM contract. The trigger requires a two-level promise DAG (`A -then-> B`) where B calls `promise_return` with a near-maximum-size receipt. No special permissions, validator role, or operator access is needed. The test `test_max_receipt_size_promise_return` in the production test suite confirms the path is reachable and that the oversized receipt does enter the chain: [5](#0-4) 

### Recommendation

After appending `output_data_receivers` to the returned receipt (lines 1028–1035 of `runtime/runtime/src/lib.rs`), re-run `validate_receipt` on the mutated receipt in `NewReceipt` mode and propagate a `NewReceiptValidationError` if the size now exceeds `max_receipt_size`. This mirrors the existing guard applied to all other newly created receipts and closes the post-validation mutation window.

### Proof of Concept

The existing integration test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete proof of concept:

1. Deploy a contract with `max_receipt_size_promise_return_method1` and `max_receipt_size_promise_return_method2`.
2. Method 1 creates DAG `[A -then-> B]`; method 2 (called as A) creates receipt C sized exactly at `max_receipt_size` and calls `promise_return(C)`.
3. The runtime appends B's `output_data_receivers` to C, making C exceed `max_receipt_size`.
4. `assert_oversized_receipt_occurred` confirms the oversized receipt is present in the chain. [6](#0-5)

#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
