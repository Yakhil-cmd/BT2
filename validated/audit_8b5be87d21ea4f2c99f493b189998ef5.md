### Title
`skip_stateful_validations` Admits Signature-Invalid Invoke Transactions for Any Account with Nonce-0 and a Pending Mempool Entry â€” (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX shortcut in the Apollo gateway's stateful validator skips the `__validate__` entry-point call (i.e., signature verification) for any Invoke transaction with `nonce = 1` whenever the sender's on-chain nonce is `0` **and** the mempool reports any transaction from that address. The proxy check (`account_tx_in_pool_or_recent_block`) is not restricted to deploy-account transactions. An unprivileged attacker who observes a victim's pending nonce-0 transaction in the mempool can immediately submit a signature-invalid Invoke with `nonce = 1` for the victim's address and have it admitted to the mempool without any signature check.

---

### Finding Description

**The UX skip path**

`skip_stateful_validations` is called inside `run_pre_validation_checks` and returns `true` when three conditions hold simultaneously:

```
tx.nonce() == Nonce(Felt::ONE)          // attacker-controlled
&& account_nonce == Nonce(Felt::ZERO)   // victim's on-chain nonce
&& account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, an Invoke with `validate = false` exits after `perform_pre_validation_stage` without ever calling `__validate__`: [3](#0-2) 

`perform_pre_validation_stage` checks nonce, fee bounds, and balance â€” but **not** the signature: [4](#0-3) 

**The overly broad proxy check**

The code comment claims the mempool presence check is sufficient because it implies either a deploy-account or a future-nonce transaction that "passed validations." This reasoning is wrong for accounts that are already deployed on-chain with nonce `0` (deployed but never transacted): [5](#0-4) 

`account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the address â€” including a legitimate nonce-0 Invoke that the victim submitted: [6](#0-5) 

**The mempool's `validate_tx` does not check signatures**

`validate_by_mempool` calls `mempool.validate_tx`, which only checks for duplicate tx hash and fee-escalation rules â€” no signature verification: [7](#0-6) 

`ValidationArgs` carries no signature field: [8](#0-7) 

---

### Impact Explanation

An attacker can inject a signature-invalid Invoke with `nonce = 1` into the mempool for any account whose on-chain nonce is `0` and that has any pending transaction visible in the mempool. The malicious transaction:

1. **Displaces the victim's legitimate nonce-1 Invoke** if the attacker offers a higher fee (fee-escalation path in the mempool replaces the existing entry).
2. **Pollutes the mempool** with an invalid transaction that the batcher will attempt to execute, fail `__validate__`, and mark as rejected â€” wasting sequencer resources.
3. **Blocks the victim's nonce-1 slot** until the rejected transaction is cleaned up by `commit_block`, forcing the victim to resubmit.

This matches the allowed impact: **High â€” Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- The mempool is observable (P2P broadcast, RPC). Any pending nonce-0 transaction from a victim is visible.
- The attacker needs only to craft an Invoke with `nonce = 1`, any arbitrary signature, and a fee above the victim's nonce-1 transaction (or submit first).
- No privileged access, no special account state, and no cryptographic capability is required.
- The production default `min_gas_price_percentage = 100` and `max_allowed_nonce_gap = 200` do not prevent this. [9](#0-8) 

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` proxy with a check that specifically confirms a **deploy-account** transaction exists for the sender (e.g., a dedicated `deploy_account_in_pool(address)` query). Alternatively, restrict the skip to accounts whose class hash is zero on-chain (i.e., truly undeployed accounts), which is the only case where `__validate__` cannot be called anyway.

---

### Proof of Concept

**Setup:** Victim has a pre-deployed account at address `X` with on-chain nonce `0`. Victim submits a legitimate Invoke with `nonce = 0`.

**Attack:**

1. Attacker observes victim's nonce-0 Invoke in the mempool â†’ `account_tx_in_pool_or_recent_block(X) == true`.
2. Attacker submits:
   ```
   RpcInvokeTransactionV3 {
       sender_address: X,
       nonce: 1,
       signature: [0xdead, 0xbeef],   // arbitrary invalid signature
       resource_bounds: { l2_gas: { max_price_per_unit: <above victim's nonce-1 tx> } },
       ...
   }
   ```
3. Gateway stateless check passes (non-zero resource bounds, valid address).
4. `validate_resource_bounds` passes (L2 gas price â‰Ą threshold).
5. `validate_nonce` passes: `0 â‰¤ 1 â‰¤ 200`.
6. `validate_by_mempool` passes (no signature check).
7. `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_in_pool==true` â†’ returns `true`.
8. `run_validate_entry_point` runs with `validate = false` â†’ `__validate__` never called.
9. Malicious Invoke is admitted to the mempool, displacing the victim's legitimate nonce-1 transaction.

**Result:** The victim's nonce-1 slot is occupied by a signature-invalid transaction. The batcher attempts execution, `__validate__` fails, the transaction is rejected, and the victim must resubmit â€” while the attacker can repeat the cycle. [10](#0-9) [11](#0-10)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
