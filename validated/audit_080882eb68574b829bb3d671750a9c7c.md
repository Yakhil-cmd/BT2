### Title
Pool Admin Can Reset Price Provider Timelock Indefinitely via Repeated `proposePoolPriceProvider` Calls — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`proposePoolPriceProvider` unconditionally overwrites `pendingPriceProviderExecuteAfter[pool]` to `block.timestamp + timelock` on every call. There is no check preventing a re-proposal while a pending provider exists. A pool admin can call this function repeatedly to reset the countdown, defeating the timelock's purpose as a user-protection boundary.

### Finding Description
In `MetricOmmPoolFactory.proposePoolPriceProvider`: [1](#0-0) 

Both storage slots are written unconditionally. There is no guard of the form:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PendingProviderAlreadyExists();
```

`priceProviderTimelock[pool]` is set once at pool creation and cannot be changed afterward: [2](#0-1) 

So the timelock duration is fixed, but the countdown start is not — the admin can reset it at will.

### Impact Explanation
The timelock exists specifically to constrain the pool admin's power over oracle changes, giving users a guaranteed window to exit before a new price provider becomes active. By resetting the countdown, the admin can:

1. Propose a malicious oracle (one returning manipulated bid/ask prices).
2. Observe users beginning to withdraw and re-propose (same or different address) to reset `pendingPriceProviderExecuteAfter[pool]` to `block.timestamp + timelock`.
3. Repeat step 2 until users stop monitoring.
4. Execute the malicious oracle at a chosen moment.
5. Swaps then execute against the manipulated oracle, causing direct loss of user principal via bad-price execution.

This is an explicit admin-boundary break: the pool admin is semi-trusted only *inside* timelocks, and this path bypasses the timelock entirely. [3](#0-2) 

### Likelihood Explanation
The pool admin role is semi-trusted. The timelock mechanism was added specifically to address audit finding M01 (admin can arbitrarily change the price provider): [4](#0-3) 

The fix introduced the timelock but did not guard against re-proposal, leaving the invariant unenforceable. Any pool admin can exploit this with a single additional transaction.

### Recommendation
Add a guard at the start of `proposePoolPriceProvider` that reverts if a pending proposal already exists:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PendingProviderAlreadyExists();
```

Alternatively, if replacing a pending proposal is intentional (e.g., to change the target), the `executeAfter` must not be reset — it should be preserved from the original proposal:

```solidity
if (pendingPriceProvider[pool] == address(0)) {
    pendingPriceProviderExecuteAfter[pool] = block.timestamp + timelock;
}
pendingPriceProvider[pool] = newPriceProvider;
```

The first option (hard revert) is simpler and fully enforces the invariant.
