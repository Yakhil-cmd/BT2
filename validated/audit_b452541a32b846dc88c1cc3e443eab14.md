### Title
Rounding in `compute_rewards_rounded_down` permanently freezes STRK dust in Pool contracts - (File: `src/pool/utils.cairo`)

### Summary
The `Pool` contract distributes rewards to delegators using integer division (`compute_rewards_rounded_down`). Because each delegator's share is truncated independently, the sum of all distributed rewards is strictly less than the total STRK transferred into the pool by the staking contract. The difference accumulates permanently in the pool contract with no recovery path.

### Finding Description
When the staking contract calls `_update_rewards`, it computes `total_pools_rewards` and transfers exactly that amount to each pool contract via `send_rewards_to_delegation_pool`. [1](#0-0) 

The pool contract then distributes these rewards to individual delegators inside `calculate_rewards` by calling `compute_rewards_rounded_down` for each balance interval: [2](#0-1) 

`compute_rewards_rounded_down` performs `amount * interest / base_value` using integer division: [3](#0-2) 

Because each delegator's reward is independently truncated, the sum of all delegators' rewards is strictly less than `total_pools_rewards`. The remainder (dust) stays in the pool contract's STRK balance. There is no function in the pool contract to sweep or redistribute this dust — `claim_rewards`, `exit_delegation_pool_action`, and `set_staker_removed` only move principal and individually-computed reward amounts, never the residual balance.

The codebase itself acknowledges this behavior in the comment at lines 144–147 of `src/pool/utils.cairo`, and the flow tests explicitly assert `pool balance < 100` after all delegators have claimed: [4](#0-3) [5](#0-4) 

### Impact Explanation
Every reward cycle, a small amount of STRK is transferred into each pool contract but can never be claimed by any delegator or recovered by any privileged role. The tokens are permanently frozen. This matches the allowed impact: **High — Permanent freezing of unclaimed yield**. The per-epoch amount is small (< 100 wei per pool per epoch), but it is cumulative across all pools and all epochs for the lifetime of the protocol.

### Likelihood Explanation
This occurs on every reward distribution cycle in every pool that has more than one delegator with non-proportional balances. It is deterministic and requires no attacker — any unprivileged delegator triggering normal `claim_rewards` calls reproduces it. Likelihood is **High**.

### Recommendation
Track the total rewards actually distributed to delegators and assign the remainder to the last claimant (or to a designated address), mirroring the Beefy fix:

```cairo
// Instead of computing each share independently and discarding the remainder:
let last_delegator_rewards = total_pool_rewards - sum_of_other_delegator_rewards;
```

Alternatively, accumulate the dust into a designated "remainder" storage slot and allow it to be swept into the next epoch's reward pool.

### Proof of Concept
1. Deploy a pool with two delegators: delegator A with balance `X`, delegator B with balance `Y`, where `X + Y` does not evenly divide `pool_rewards * base_value`.
2. Trigger a reward epoch (attest or `update_rewards`). The staking contract transfers `pool_rewards` STRK to the pool contract.
3. Both delegators call `claim_rewards`. Each receives `floor(balance * sigma / base_value)`.
4. Assert `token.balance_of(pool_contract) > 0` — the pool contract retains dust that no party can ever claim.

This is confirmed by the existing test assertion at `src/pool/tests/test.cairo:696`:
```cairo
assert!(token_dispatcher.balance_of(pool_contract) < 100);
```
which proves the pool contract retains a non-zero balance after all delegators have claimed, with no mechanism to recover it. [6](#0-5) [7](#0-6)

/// Compute the rewards from the amount and interest.
///
/// $$ rewards = amount * interest / base_value $$
/// **Note**: The Pool contract’s reward calculation logic uses integer division, discarding
/// small rounding remainders (dust) without tracking or redistributing them.
/// This results in negligible reward losses for delegators, as the total distributed rewards
/// are slightly less than the allocated amount.
pub(crate) fn compute_rewards_rounded_down(
    amount: Amount, interest: Index, base_value: Index,
) -> Amount {
    mul_wide_and_div(lhs: amount, rhs: interest, div: base_value)
        .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
}
