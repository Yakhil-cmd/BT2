### Title
Missing Minimum Threshold Lower-Bound Validation in FROST Presign/Sign Produces Corrupted Signing Outputs — (File: `src/frost/mod.rs`)

---

### Summary

Both `presign` and `assert_sign_inputs` in `src/frost/mod.rs` validate only the **upper** bound of the threshold parameter (`threshold > participants.len()`), but omit the **lower** bound check (`threshold >= 2`) that every other protocol entry-point in the library enforces. A malicious coordinator or unprivileged caller can supply `threshold = 1` (or `threshold = 0`) to the FROST EdDSA / RedJubjub signing pipeline. Because the key was generated under `threshold >= 2` (enforced by DKG), using a mismatched threshold during signing causes Lagrange interpolation with wrong coefficients, producing an invalid, unusable signature that honest parties accept as the protocol output.

---

### Finding Description

**Root cause — missing lower-bound guard in `src/frost/mod.rs`**

`assert_key_invariants` in `src/dkg.rs` and `validate_triple_inputs` in the OT-ECDSA triple module both enforce a minimum threshold of 2:

```rust
// src/dkg.rs  lines 580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

```rust
// src/ecdsa/ot_based_ecdsa/triples/generation.rs  lines 699-703
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value, min: 2,
    });
}
```

The FROST entry-points perform **no equivalent check**:

```rust
// src/frost/mod.rs  lines 71-77  (presign)
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check; threshold = 1 or 0 is silently accepted
```

```rust
// src/frost/mod.rs  lines 144-150  (assert_sign_inputs)
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← same omission
```

`ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforcement at the type level:

```rust
// src/thresholds.rs  lines 9-24
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
```

So any caller can construct `ReconstructionLowerBound(1)` or `ReconstructionLowerBound(0)` and pass it directly.

---

### Impact Explanation

FROST signing uses the threshold to select which partial signatures to collect and to compute Lagrange interpolation coefficients. The key was generated with `threshold >= 2` (DKG enforces this). If signing is invoked with `threshold = 1`:

- Only one partial signature is collected.
- The Lagrange coefficient for a single-point interpolation is 1, so the "combined" signature equals one participant's raw partial contribution.
- This is not a valid EdDSA signature over the group key; verification against the public key will fail.
- Honest participants complete the protocol and receive this corrupted, unusable output, believing the signing round succeeded.

**Mapped impact**: *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

The coordinator role is explicitly modelled as a potential adversary in this library (the coordinator is a named parameter in every signing call). A malicious coordinator constructs `PresignArguments { threshold: ReconstructionLowerBound(1), ... }` and calls `presign` / the EdDSA or RedJubjub `sign` wrappers. No special privilege beyond being a protocol participant is required. The entry-path is direct and requires zero cryptographic capability.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already apply, in both `presign` and `assert_sign_inputs` inside `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place this check immediately after converting the threshold, before any other validation, mirroring the pattern in `src/dkg.rs`.

---

### Proof of Concept

1. A key is generated via `keygen` with `threshold = 2` and `participants = [A, B, C]`. DKG enforces `threshold >= 2` — this succeeds normally.

2. A malicious coordinator calls `presign` with `PresignArguments { threshold: ReconstructionLowerBound(1), keygen_out: ... }`. The only guard is:
   ```
   1 > 3  →  false  →  no error returned
   ``` [1](#0-0) 

3. The coordinator then calls `assert_sign_inputs` with `threshold = 1`. Again, only the upper-bound is checked:
   ```
   1 > 3  →  false  →  no error returned
   ``` [2](#0-1) 

4. Signing proceeds: only participant A's partial signature is collected; Lagrange coefficient = 1; the "signature" is A's raw partial share contribution — not a valid EdDSA signature over the threshold public key.

5. Honest participants B and C complete the protocol round and receive this corrupted output, with no indication that the threshold was inconsistent with key generation.

**Contrast with DKG** (which correctly rejects `threshold = 1`): [3](#0-2) 

**Contrast with OT-ECDSA triples** (which correctly rejects `threshold < 2`): [4](#0-3) 

**`ReconstructionLowerBound` has no type-level minimum invariant**, making the missing runtime check the only line of defence: [5](#0-4)
