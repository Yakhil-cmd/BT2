### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards per-bin additional fee values directly to the pool with no validation against `maxAdminSpreadFeeE6` or `maxAdminNotionalFeeE8`. The pool admin (semi-trusted) can therefore charge effective swap fees far above the factory-owner-enforced cap on every swap that crosses a manipulated bin.

### Finding Description

The factory enforces an admin fee cap in `setPoolAdminFees`: [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

The parallel path `setPoolBinAdditionalFees` has no such guard: [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool itself also performs no cap check: [3](#0-2) 

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
```

During every swap the per-bin additional fee is added directly on top of the oracle-derived base fee: [4](#0-3) 

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so the pool admin can set them to **65 535 E6 ≈ 6.55 %** per bin, regardless of what `maxAdminSpreadFeeE6` is set to. If the factory owner has set `maxAdminSpreadFeeE6 = 0` (or any value below 65 535) to prevent admin from charging spread fees, the pool admin can still impose up to 6.55 % per-bin fees on every swap through the active bin.

### Impact Explanation

Every swap that crosses a bin with inflated `addFeeBuyE6`/`addFeeSellE6` pays the excess fee directly to the pool's spread-fee surplus, which is then collected by the admin via `collectFees`. Users lose real token value on every swap. The loss is proportional to swap size and can reach 6.55 % of notional per bin crossed — well above any `maxAdminSpreadFeeE6` the factory owner may have configured.

### Likelihood Explanation

The pool admin is a single EOA or contract address stored in `poolAdmin[pool]`. Any pool whose admin key is compromised, or any pool whose admin acts adversarially, can immediately call `setPoolBinAdditionalFees` with `addFeeBuyE6 = type(uint16).max` on the active bin. No timelock, no co-signer, no prior notice is required. The attack is silent (no revert, no unusual event beyond `BinAdditionalFeesUpdated`) and takes effect on the very next swap.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's `maxAdminSpreadFeeE6` via a factory call, or store the cap as an immutable on the pool at deploy time.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 0` (no admin spread fee permitted).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, activeBin, 65535, 65535)`.
3. No revert occurs; `_binStates[activeBin].addFeeBuyE6 = 65535` is written.
4. Next user swap through `activeBin` pays `baseFeeX64 + 65535/1e6` as the effective fee — 6.55 % above the oracle spread — instead of the capped `baseFeeX64 + 0`.
5. The excess fee accumulates as spread surplus in the pool balance and is collected by the admin via `collectPoolFees`, draining user funds beyond the factory-enforced limit.
