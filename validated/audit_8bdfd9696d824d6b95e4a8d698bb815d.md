### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Protocol's Hard Spread-Fee Limit — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap check, while every other fee-setting path in the factory is guarded by `maxAdminSpreadFeeE6` (≤ `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%). Because bin additional fees are applied *on top of* the base spread fee, a pool admin can silently push the effective per-bin fee well above the 20 % hard limit the protocol intends to enforce.

---

### Finding Description

The factory defines two hard constants and enforces them on every base-fee path:

```
uint24 internal constant HARD_MAX_SPREAD_FEE_E6  = 200_000;   // 20 %
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000; // 1 %
``` [1](#0-0) 

`setPoolAdminFees` validates against those caps before writing:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [2](#0-1) 

`setPoolBinAdditionalFees`, however, performs **no validation at all** — it forwards the caller-supplied values straight to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The same gap exists at pool-creation time: `_unpackAndValidateBinStates` unpacks `buyFee` / `sellFee` from the packed bin data and stores them in `BinState` without any range check:

```solidity
(uint16 length, uint16 buyFee, uint16 sellFee) = binData.unpack();
if (length == 0) break;
nonNegativeBinStates[k] = BinState({
    ..., addFeeBuyE6: buyFee, addFeeSellE6: sellFee
});
``` [4](#0-3) 

The interface documents these as fees applied **"on top of base spread"**: [5](#0-4) 

`uint16` allows values up to 65 535, which in E6 units equals ≈ 6.55 % additional fee per bin. Combined with a base spread fee already at the 20 % hard cap, the effective per-bin fee can reach ≈ 26.55 %, silently exceeding the protocol's own hard limit.

---

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` (or any value that pushes the total above 20 %) on any bin. Every swap that crosses that bin will be charged the inflated fee. Because the fee is taken from the trader's input before the output is computed, the trader receives fewer tokens than the oracle price and the protocol's stated fee cap would permit. Accumulated excess fees flow to the admin fee destination, constituting a direct, measurable loss of user principal on every affected swap.

---

### Likelihood Explanation

The pool admin role is semi-trusted but operates within caps. The factory's own design signals that the 20 % hard cap is a protocol-level invariant, not merely a suggestion. Any pool admin — including one who turns adversarial after deployment — can invoke `setPoolBinAdditionalFees` in a single transaction with no timelock, immediately affecting all subsequent swaps on the targeted bin. No special privilege beyond the pool admin role is required.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the guard already present in `setPoolAdminFees`:

```solidity
if (uint256(addFeeBuyE6)  + spreadFeeE6ForPool > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (uint256(addFeeSellE6) + spreadFeeE6ForPool > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Or, at minimum, cap each bin additional fee independently against `HARD_MAX_SPREAD_FEE_E6`. Apply the same check inside `_unpackAndValidateBinStates` during `createPool` so the invariant is enforced at both entry points.

---

### Proof of Concept

1. Deploy a pool with `adminSpreadFeeE6 = 0` (base spread fee = 0).
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert occurs; the pool stores `addFeeBuyE6 = 65535` on bin 0.
4. A trader swaps through bin 0; the effective spread fee applied is 65535 / 1e6 ≈ 6.55 %, far above the 0 % base fee and unchecked against `HARD_MAX_SPREAD_FEE_E6 = 200_000`.
5. Repeat with `adminSpreadFeeE6 = 200_000` (base at hard cap): total effective fee ≈ 26.55 %, exceeding the hard cap by 6.55 percentage points with no revert. [3](#0-2) [1](#0-0)
