### Title
Unprivileged Caller Can Permanently Freeze Consensus Rewards via `update_rewards` with Caller-Controlled `disable_rewards` Flag — (File: `src/staking/staking.cairo`)

---

### Summary
`update_rewards` in the Staking contract is callable by any address with no access control. It accepts a caller-controlled `disable_rewards: bool` parameter and writes to the global `last_reward_block` storage slot regardless of whether rewards are actually distributed. An attacker can call this every block with `disable_rewards: true` to consume the per-block reward slot without distributing any rewards, permanently freezing unclaimed yield for all stakers.

---

### Finding Description
`update_rewards` is exposed as part of `IStakingRewardsManager` with no caller restriction beyond `general_prerequisites()`, which only checks the contract is unpaused and the caller is non-zero. [1](#0-0) 

The function accepts a `disable_rewards: bool` parameter directly from the caller. After validating that the staker is active and has non-zero balance, it unconditionally writes `current_block_number` to the global `last_reward_block` storage slot: [2](#0-1) 

Only after this write does it branch on `disable_rewards`: [3](#0-2) 

The global `last_reward_block` field is defined as a single contract-wide value: [4](#0-3) 

Any subsequent call to `update_rewards` in the same block fails with `REWARDS_ALREADY_UPDATED`: [5](#0-4) 

---

### Impact Explanation
An attacker who calls `update_rewards(valid_staker, disable_rewards: true)` in every block:

1. Advances `last_reward_block` to the current block without distributing any rewards.
2. Causes every legitimate consensus-layer call to `update_rewards` in that block to revert with `REWARDS_ALREADY_UPDATED`.
3. Stakers' `unclaimed_rewards_own` never increases; pool `cumulative_rewards_trace` is never updated.

Sustained over time this constitutes **permanent freezing of unclaimed yield** (High impact). Even a partial attack constitutes **temporary freezing of funds** (also High impact). The attacker gains nothing financially — pure griefing — but the damage to stakers and the protocol is concrete and measurable.

---

### Likelihood Explanation
- The function is publicly callable with no role check.
- The attacker only needs to know any one active staker address (all staker addresses are emitted as `NewStaker` events and are fully public).
- The cost is one Starknet transaction per block. On Starknet, transaction fees are low, making sustained griefing economically viable.
- No special knowledge, leaked keys, or privileged access is required.

---

### Recommendation
Restrict who may supply `disable_rewards: true`. Options:

1. **Restrict the entire function** to a trusted caller (e.g., the attestation contract or a designated consensus operator role), consistent with how `update_rewards_from_attestation_contract` is already gated: [6](#0-5) 

2. **Split the function**: expose a public `update_rewards(staker_address)` that always distributes rewards, and a role-gated `update_rewards_with_disable(staker_address, disable_rewards)` callable only by the consensus layer.

3. At minimum, move the `last_reward_block.write` **after** the `disable_rewards` branch so that a call with `disable_rewards: true` does not consume the block's reward slot.

---

### Proof of Concept

```
// Attacker script — run once per block:
// 1. Pick any known active staker address (e.g., from NewStaker events).
// 2. Call:
staking_contract.update_rewards(
    staker_address: known_active_staker,
    disable_rewards: true,
);
// Result:
//   - last_reward_block is now set to the current block number.
//   - No rewards are distributed.
//   - Any legitimate consensus call to update_rewards in this block
//     reverts with REWARDS_ALREADY_UPDATED.
// Repeat every block → all stakers permanently denied consensus rewards.
```
