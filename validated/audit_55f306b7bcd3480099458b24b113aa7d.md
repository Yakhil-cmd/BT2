### Title
`MsgTurnBridge` Handler Is a No-Op Stub: Permission Check Bypassed and Bridge Circuit Breaker Permanently Broken — (`File: x/cronos/keeper/msg_server.go`)

---

### Summary

The `TurnBridge` gRPC handler in `x/cronos/keeper/msg_server.go` is implemented as a stub that immediately returns `nil, nil`. It performs no authorization check and updates no state. Any unprivileged address can call `MsgTurnBridge` and receive a success response, while the bridge emergency circuit breaker is permanently non-functional — the bridge can never be disabled regardless of who calls it.

---

### Finding Description

The `TurnBridge` message handler is defined as:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to the `UpdateTokenMapping` handler immediately above it, which correctly enforces the permission system:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

The permission constants `CanTurnBridge` and `CanChangeTokenMapping` are defined side-by-side, and the query layer correctly reports `CanTurnBridge` as a distinct permission: [3](#0-2) 

The `HasPermission` function is fully implemented and used by `UpdateTokenMapping`, but is never called by `TurnBridge`: [4](#0-3) 

The `MsgTurnBridge` proto message and its gRPC handler are fully wired into the module: [5](#0-4) 

The CLI command `CmdTurnBridge` is also fully wired, allowing any user to submit the transaction: [6](#0-5) 

The integration test `test_gravity_turn_bridge` explicitly expects that:
1. An unprivileged `community` address calling `turn_bridge("false")` should fail with a non-zero code.
2. After a permissioned `validator` calls `turn_bridge("false")`, `send_to_evm_chain` should fail. [7](#0-6) 

Both assertions are violated by the stub: the community call returns code `0` (no auth check), and the bridge remains active after the validator call (no state update).

---

### Impact Explanation

**Auth bypass (High):** Any unprivileged address can submit `MsgTurnBridge` and receive a success response (`code == 0`). The `CanTurnBridge` permission check — which is the entire purpose of the permission system for this message — is completely absent. This directly bypasses the Cronos admin/permission authorization model.

**Bridge circuit breaker permanently broken (High):** The `TurnBridge` handler never writes any bridge-enabled/disabled state. Even if the admin or a permissioned address calls it, the bridge state is unchanged. The gravity bridge emergency stop mechanism — explicitly designed to halt bridge flows in an emergency — is permanently non-functional. Bridge and conversion flows (`send_to_evm_chain`, `send_to_ibc`) cannot be halted by any on-chain action.

---

### Likelihood Explanation

The entry path is a standard, publicly-routable Cosmos SDK gRPC/REST message (`/cronos.Msg/TurnBridge`). No special privileges, leaked keys, or cryptographic breaks are required. Any address can submit the transaction. The broken state is permanent until a code fix and chain upgrade are deployed.

---

### Recommendation

Implement `TurnBridge` analogously to `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` if it fails.
2. Read the current `Params`, update the bridge-active field (e.g., `params.EnableAutoDeployment` or a dedicated `BridgeActive` param), and call `k.SetParams(ctx, params)`.
3. Emit an event recording the new bridge state and the caller.

The EVM hook handlers for `__CronosSendToEvmChain` and `__CronosSendToIbc` should then gate on the bridge-active param and return an error when the bridge is disabled.

---

### Proof of Concept

1. Any unprivileged address submits `MsgTurnBridge{Sender: attacker, Enable: false}`.
2. The handler returns `nil, nil` — the transaction succeeds with code `0`, bypassing the `CanTurnBridge` permission check.
3. No bridge state is updated. The bridge remains fully active.
4. A permissioned admin also calls `MsgTurnBridge{Enable: false}` in an emergency — same result: success response, no state change, bridge still active.
5. `send_to_evm_chain` on any CRC21/CRC20Proxy contract continues to burn tokens and emit `__CronosSendToEvmChain` events, which the EVM hook processes into outbound bridge transfers — the bridge cannot be stopped.

func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```

				return err

				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}
