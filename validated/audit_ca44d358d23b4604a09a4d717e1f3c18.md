### Title
WalletConnect `chia_spendClawbackCoins` Confirmation Dialog Omits Amount, Address, and Direction Context — (`packages/gui/src/electron/commands/Commands.ts`, `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

The WalletConnect confirmation dialog for `chia_spendClawbackCoins` displays only opaque `coin_ids` (a JSON array of hex strings) and a `fee` value. It does not show the XCH amount, the counterparty address, or the direction of the operation (claim vs. claw back). The native `ClawbackClaimTransactionDialog` always shows all three. A malicious dApp that has been granted the `chia_spendClawbackCoins` permission can trigger a spend of any pending clawback coin whose ID it knows from on-chain data, and the user will approve it without understanding what they are signing.

---

### Finding Description

**Command registration** — `chia_wallet.spend_clawback_coins` is registered in `Commands.ts` with a `dapp` entry that maps the WalletConnect method `chia_spendClawbackCoins` to it:

```
params: [
  { name: 'coin_ids', label: 'Coin Ids', type: 'json' },
  { name: 'fee',      label: 'Fee',      type: 'bigint', humanize: 'mojo-to-xch' },
],
dapp: [{ command: 'chia_spendClawbackCoins', requiresSync: true }],
``` [1](#0-0) 

The `params` array is the complete set of fields rendered in the confirmation dialog. No `amount`, `address`, or `fromOrTo` field is present.

**`parseCommandDisplay` returns `undefined` for this command** — the function that enriches the dialog with a `walletDelta` section (showing "You Spend / You Receive") only handles `take_offer` and `create_offer_for_ids`. For every other command, including `spend_clawback_coins`, it returns `undefined`, so no enriched display is produced:

```ts
export async function parseCommandDisplay(command: string, params: ...) {
  if (command === 'chia_wallet.take_offer') { ... }
  if (command === 'chia_wallet.create_offer_for_ids') { ... }
  return undefined;   // ← spend_clawback_coins falls here
}
``` [2](#0-1) 

**What the user sees in the WalletConnect dialog:**
- Title: "Confirm Clawback Spend"
- Message: "Please carefully review and confirm this clawback spend."
- Row 1 — Coin Ids: `["0xabc123..."]`
- Row 2 — Fee: `0 XCH`

**What the native `ClawbackClaimTransactionDialog` shows:**
- Title: "Claim Transaction" or "Claw Back Transaction" (direction-aware)
- Amount in XCH (human-readable)
- Address (From / To, truncated with copy-to-clipboard) [3](#0-2) 

**Execution path** — after the user clicks "Send", `main.tsx` calls `sendCommand('spend_clawback_coins', 'chia_wallet', parsedParams)` which reaches the wallet daemon and executes the spend: [4](#0-3) 

---

### Impact Explanation

A malicious dApp that holds an active WalletConnect session with `chia_spendClawbackCoins` in its approved command list can:

1. Read any pending clawback coin ID from the public blockchain.
2. Send `chia_spendClawbackCoins({ coin_ids: [knownCoinId], fee: 0 })` at any time.
3. The user sees a dialog with an opaque hex coin ID and a zero fee — no amount, no address, no indication of whether they are claiming incoming funds or reclaiming sent funds.
4. A user who approves causes an irreversible on-chain spend of the clawback coin.

This fits **High: WalletConnect state that causes a user to approve a spend of the wrong asset, amount, or status** and **High: bypass of custody/clawback restrictions with direct security impact**.

---

### Likelihood Explanation

- Coin IDs for pending clawback transactions are publicly visible on-chain; no privileged access is required.
- The only precondition is an established WalletConnect session where the user previously approved `chia_spendClawbackCoins` for the dApp — a permission users may grant to legitimate-looking dApps.
- The confirmation dialog gives no signal that distinguishes a legitimate self-initiated claim from a dApp-initiated spend.

---

### Recommendation

1. Extend `parseCommandDisplay` to handle `chia_wallet.spend_clawback_coins`: look up each coin ID via the wallet RPC, resolve the amount, counterparty address, and direction, and return a `walletDelta`-style display. [2](#0-1) 

2. Add `amount`, `address`, and `direction` (claim vs. claw back) as explicit labeled rows in the `params` schema for `chia_wallet.spend_clawback_coins`, populated by a pre-confirmation RPC lookup. [5](#0-4) 

3. Consider adding a `handler` to the dapp entry that resolves coin metadata before the dialog is shown, mirroring the pattern used for offer commands.

---

### Proof of Concept

1. Establish a WalletConnect session between the Chia GUI and a test dApp; approve `chia_spendClawbackCoins` during pairing.
2. Send a clawback transaction from the wallet; note the resulting coin ID from the blockchain explorer or wallet RPC.
3. From the test dApp, send:
   ```json
   { "method": "chia_spendClawbackCoins", "params": { "coin_ids": ["<knownCoinId>"], "fee": 0 } }
   ```
4. Observe the confirmation dialog: it shows only the hex coin ID and `0 XCH` fee — no amount, no address, no direction label.
5. Click "Send". Verify via wallet RPC that `spend_clawback_coins` was executed and the transaction appears in the wallet history.
6. Assert that the dialog never displayed the XCH amount, the counterparty address, or whether the action was a claim or a claw back — information that `ClawbackClaimTransactionDialog` always shows. [6](#0-5) [1](#0-0)

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }

  return undefined;
}
