### Title
Gateway Admits Invoke Transactions Without Signature Verification When Any Transaction for the Account Exists in the Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point (the account's signature-verification step) for any invoke transaction with `nonce == 1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because that check returns `true` for **any** transaction type in the pool — not specifically a `deploy_account` — an unprivileged attacker who observes a `deploy_account` for address A in the mempool can immediately front-run the legitimate user's first invoke by submitting a nonce-1 invoke with an arbitrary/invalid signature. The gateway admits it without running `__validate__`, and the legitimate user's nonce-1 invoke is subsequently rejected with `DuplicateNonce`.

``` [1](#0-0) 

The function returns `true` (skip validation) whenever:
1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The account's on-chain nonce is `0`
3. `account_tx_in_pool_or_recent_block` returns `true`

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for **any** transaction type in the pool — a `deploy_account`, a future-nonce invoke, or anything else. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning does not account for an attacker who submits a nonce-1 invoke *exploiting* the presence of the legitimate user's `deploy_account`.

**How the skip propagates to execution:**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` constructs the `AccountTransaction` with `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };
...
blockifier_validator.validate(account_tx)
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `validate` flag gates the `__validate__` call:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [4](#0-3) 

So the gateway's blockifier validation step runs with `__validate__` entirely disabled, and the transaction is forwarded to the mempool via `mempool_client.add_tx`.

**Pre-conditions that allow the attacker's transaction to reach this point:**

- *Stateless validation* (`StatelessTransactionValidator::validate`): no signature check; passes trivially.
- *Stateful nonce check* (`validate_nonce`): nonce 1 is within `max_allowed_nonce_gap` (default 200) of account nonce 0; passes.
- *Mempool pre-validation* (`validate_by_mempool` → `mempool.validate_tx`): checks only for duplicate tx hash and nonce-too-old; passes because no nonce-1 invoke is in the pool yet. [5](#0-4) [6](#0-5) 

### Impact Explanation

An unprivileged attacker can:

1. Observe a `deploy_account` for address A in the public mempool.
2. Immediately submit an invoke with `nonce=1` for address A carrying **arbitrary calldata and any signature**.
3. The gateway admits it without running `__validate__` (signature check bypassed).
4. The legitimate user's nonce-1 invoke is subsequently rejected with `MempoolError::DuplicateNonce`.
5. The attacker's transaction eventually fails during batcher execution (the batcher runs `__validate__` with `validate: true`), but the legitimate user's transaction has already been blocked.
6. The attacker can repeat this indefinitely, persistently preventing the legitimate user from executing their first post-deployment transaction.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."** The gateway admits an invalid transaction (invalid signature) and causes a valid transaction (the legitimate user's nonce-1 invoke) to be rejected before sequencing.

### Likelihood Explanation

- The mempool is publicly observable; no privileged access is required.
- The attacker only needs to detect a `deploy_account` and submit a competing nonce-1 invoke faster — a straightforward race condition.
- No cryptographic material or special knowledge of the target account is needed.
- The attack is repeatable at negligible cost.

### Recommendation

1. **Narrow the skip condition**: Instead of checking `account_tx_in_pool_or_recent_block` (which matches any transaction type), verify that a `deploy_account` transaction specifically exists in the pool for the sender address. This requires a new mempool query such as `deploy_account_in_pool(address)`.

2. **Use the existing but unused config field**: `StatefulTransactionValidatorConfig::max_nonce_for_validation_skip` exists but is never read by `skip_stateful_validations`. Wire it in and consider whether the skip should be limited to nonce=1 only or a configurable range.

3. **Alternatively, remove the skip entirely**: Require the user to submit the deploy_account first and wait for it to be processed before submitting the invoke. This eliminates the UX shortcut but closes the admission bypass.

### Proof of Concept

```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
