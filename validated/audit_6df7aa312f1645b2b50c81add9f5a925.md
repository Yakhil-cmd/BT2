### Title
`skip_stateful_validations` Uses `account_tx_in_pool_or_recent_block` Instead of a Deploy-Account-Specific Check, Allowing Invoke Transactions to Bypass `__validate__` Without a Corresponding `deploy_account` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is intended to skip the `__validate__` entry-point call for an invoke transaction whose nonce is 1 and whose account is not yet deployed, **only when a `deploy_account` transaction for that account is already pending in the mempool** (a UX improvement for the simultaneous deploy+invoke flow). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from that address, not specifically a `deploy_account`. An attacker can seed the mempool with a future-nonce invoke for an undeployed address, causing any subsequent invoke for that address to skip `__validate__` (signature verification) at the gateway.

---

### Finding Description

The relevant code is in `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // WRONG CHECK: returns true for ANY tx from this address
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The comment in the code states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is **incorrect**. A future-nonce invoke (nonce=1) for an undeployed address (account_nonce=0) passes `validate_nonce` because the nonce is within the allowed gap:

```rust
// validate_nonce: account_nonce=0, tx_nonce=1 passes when max_allowed_nonce_gap >= 1
let max_allowed_nonce =
    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) { ... }
``` [2](#0-1) 

And `account_tx_in_pool_or_recent_block` checks for any transaction, not a deploy_account:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

**Attack sequence:**

1. Attacker submits `invoke(nonce=1, address=X)` for an undeployed address X. This passes `validate_nonce` (nonce within gap) and is added to the mempool.
2. Attacker (or any party) submits a second `invoke(nonce=1, address=X)` with a higher fee (fee escalation replaces the first).
3. During gateway validation of the second invoke:
   - `validate_nonce`: passes (account_nonce=0, tx_nonce=1 ≤ max_allowed_nonce_gap).
   - `validate_by_mempool`: passes (fee escalation allows replacement of the first invoke).
   - `skip_stateful_validations`: `account_tx_in_pool_or_recent_block(X)` returns `true` (the first invoke is still in the pool at query time) → returns `skip_validate = true`.
4. `run_validate_entry_point` is called with `validate: false`, so `__validate__` (signature verification) is **not called**.
5. The second invoke — which may carry an invalid or attacker-crafted signature — is admitted to the mempool without any signature check.

The analog to the external bug is exact: the external bug used `isLocalToken(newLocalAddress)` (always false for a freshly generated address) instead of `isGlobalToken(globalAddress)` (the stable identity). Here, `account_tx_in_pool_or_recent_block` (true for any tx) is used instead of a deploy-account-specific check (the correct identity predicate), making the guard trivially bypassable by seeding the pool with a cheap future-nonce invoke.

The `run_validate_entry_point` path that sets `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An invoke transaction with an arbitrary or invalid signature is admitted to the mempool without `__validate__` being called at the gateway. The batcher does re-run `__validate__` with `validate: true` (via `AccountTransaction::new_for_sequencing`), so the transaction will revert on-chain if the signature is invalid. However, the gateway's admission invariant — that every accepted invoke has passed account-controlled signature verification — is broken. This enables:

- Mempool pollution with signature-unverified invokes for any undeployed address.
- Denial-of-service against the deploy+invoke UX flow: an attacker can ensure that a victim's legitimate invoke (nonce=1) skips `__validate__` at the gateway, and then replace it with a malicious invoke that also skips `__validate__`, occupying the nonce slot until the batcher reverts it.

---

### Likelihood Explanation

**Medium.** The attacker needs only to submit one cheap future-nonce invoke for the target address (which passes nonce-gap validation) before the victim's invoke arrives. No privileged access is required. The target address can be predicted (deploy_account addresses are deterministic from salt + class_hash + constructor_calldata) or observed from the mempool.

---

### Recommendation

Replace `account_tx_in_pool_or_recent_block` with a check that specifically verifies a `deploy_account` transaction exists for the address. This requires adding a new mempool API method, e.g. `deploy_account_in_pool(address) -> bool`, and using it in `skip_stateful_validations`:

```rust
// Correct check: only skip __validate__ if a deploy_account is pending
return mempool_client
    .deploy_account_in_pool(tx.sender_address())
    .await ...;
```

### Proof of Concept

```
1. Attacker calls gateway.add_tx(invoke { sender=X, nonce=1, signature=GARBAGE, fee=10 })
   - validate_nonce: account_nonce=0, tx_nonce=1 ≤ max_allowed_nonce_gap → PASS
   - validate_by_mempool: no existing tx for (X, nonce=1) → PASS
   - skip_stateful_validations: account_tx_in_pool_or_recent_block(X) = false → skip=false
   - __validate__ IS called → PASS (attacker controls X's key, signs correctly)
   → Attacker's invoke(nonce=1) is in the mempool.

2. Attacker calls gateway.add_tx(invoke { sender=X, nonce=1, signature=INVALID, fee=100 })
   - validate_nonce: account_nonce=0, tx_nonce=1 → PASS
   - validate_by_mempool: fee escalation replaces step-1 tx → PASS
   - skip_stateful_validations: account_tx_in_pool_or_recent_block(X) = true (step-1 tx still
     present at query time) → skip=true
   - __validate__ IS NOT CALLED
   → Malicious invoke(nonce=1, signature=INVALID) admitted to mempool without signature check.

3. Attacker calls gateway.add_tx(deploy_account { address=X })
   → deploy_account is in the mempool.

4. Batcher sequences the block: deploy_account(X) executes first, then malicious invoke(X,nonce=1).
   Batcher calls __validate__ with validate=true → REVERT (invalid signature).
   Block space wasted; victim's legitimate nonce=1 slot consumed.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
