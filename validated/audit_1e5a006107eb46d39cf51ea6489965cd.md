### Title
Duplicate Entries in EIP-2930/EIP-1559/EIP-4844/EIP-7702 Access List Cause Excess Native Resource Drain Without Proportional Benefit - (File: `basic_bootloader/src/bootloader/transaction/access_list.rs`)

---

### Summary

The `parse_and_warm_up_access_list` function in ZKsync OS processes every entry in a transaction's access list unconditionally, without checking for duplicate addresses or duplicate storage keys. Because each entry unconditionally charges native resources (and ergs) regardless of whether the address/slot was already warmed by a prior entry in the same list, a transaction sender can craft an access list with many duplicate entries to drain native resources from the intrinsic budget in a way that is not proportional to actual system work performed. This creates a resource-accounting bug: the intrinsic native formula pre-charges based on the raw count of entries (including duplicates), but the actual warm-up work for a duplicate is cheaper than for a fresh cold entry, causing the formula to over-charge users and — more critically — the `WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST` is charged on every call to `materialize_element` regardless of whether the entry is already warm, meaning duplicates still consume real native resources from the transaction's budget.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction/access_list.rs`, `parse_and_warm_up_access_list` iterates over every `AccessListForAddress` entry in the access list:

```rust
for AccessListForAddress { address, slots_list } in iter {
    // per-address charge
    resources.charge(&S::Resources::from_ergs_and_native(
        Ergs(evm_interpreter::gas_constants::ACCESS_LIST_ADDRESS * ERGS_PER_GAS),
        ...PER_ADDRESS_ACCESS_LIST_NATIVE_COMPUTATIONAL_OVERHEAD...
    ))?;
    resources.with_infinite_ergs(|resources| {
        system.io.touch_account(ExecutionEnvironmentType::NoEE, resources, &address)
    })?;
    for key in slots_list.iter() {
        resources.charge(&S::Resources::from_ergs_and_native(
            Ergs(evm_interpreter::gas_constants::ACCESS_LIST_STORAGE_KEY * ERGS_PER_GAS),
            ...PER_SLOT_ACCESS_LIST_NATIVE_COMPUTATIONAL_OVERHEAD...
        ))?;
        ...
        system.io.storage_touch(...)?;
    }
}
```

There is **no deduplication check** on `address` across iterations, and no deduplication check on `key` within a `slots_list` or across entries for the same address.

Inside `materialize_element` (both the Ethereum and flat storage model implementations), every call charges `WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST` unconditionally at the top:

```rust
let native = R::Native::from_computational(WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST);
resources.charge(&R::from_ergs_and_native(ergs, native))?;
```

So for a duplicate address, the native cost `WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST = 4000` is charged again even though the account is already warm and no cold-access oracle work is done.

Similarly, in `validation_impl.rs` for ZK transactions, the intrinsic native formula counts raw entries:

```rust
let mut access_list_accounts = 0;
let mut access_list_storage_keys = 0;
if let Some(iter) = transaction.access_list_iter() {
    for AccessListForAddress { address: _, slots_list } in iter {
        access_list_accounts += 1;
        access_list_storage_keys += slots_list.count as u64;
    }
}
```

This count includes duplicates, so the intrinsic native pre-charge is multiplied by the number of raw entries. The formula uses `L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_ACCESS_LIST_PER_ADDRESS` which includes `NEW_COLD_ACCOUNT_READ_COST` (worst-case cold read), but for a duplicate entry the actual cost is only `WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST`. This means:

1. **For Ethereum-type transactions**: The ergs charged per duplicate address entry is `ACCESS_LIST_ADDRESS * ERGS_PER_GAS = 2400 * 256 = 614,400 ergs`, and native `PER_ADDRESS_ACCESS_LIST_NATIVE_COMPUTATIONAL_OVERHEAD + WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST` is consumed per duplicate. A sender can pad the access list with N copies of the same address to consume `N × 614,400` ergs from their gas budget without any additional useful work.

2. **For ZK-type transactions**: The intrinsic native formula pre-charges `N × L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_ACCESS_LIST_PER_ADDRESS` for N duplicate entries, which includes `NEW_COLD_ACCOUNT_READ_COST` per entry. This over-charges the user's native budget for duplicates, potentially causing valid transactions to be rejected as `OutOfNativeResourcesDuringValidation` when they would otherwise succeed.

---

### Impact Explanation

**Resource accounting bug / valid-execution unprovability path:**

- An attacker (or any user) can submit a transaction with an access list containing many duplicate addresses or storage keys. Each duplicate causes the system to charge ergs and native resources as if it were a fresh cold entry, even though the second and subsequent touches of the same address/slot are cheap (warm-path only).
- For ZK transactions, the intrinsic native formula over-charges based on raw duplicate count, which can cause a transaction with a large duplicate-heavy access list to be rejected with `OutOfNativeResourcesDuringValidation` even if the actual native work is within budget. This is a correctness/liveness issue.
- For Ethereum-type transactions, duplicate entries drain the user's gas budget at the full `ACCESS_LIST_ADDRESS = 2400` gas rate per duplicate, which is an EVM semantic mismatch: Ethereum itself (EIP-2930) does not prohibit duplicate access list entries and charges the same gas per entry, but ZKsync OS additionally charges native resources per duplicate, creating a divergence in effective cost between Ethereum and ZKsync OS.
- The `WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST = 4000` native units are charged on every `touch_account` call regardless of warm status, so duplicates consume real native from the transaction budget.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft an EIP-2930, EIP-1559, EIP-4844, or EIP-7702 transaction with a duplicate-heavy access list. The RLP parser (`AccessListForAddress::decode_from_item`) does not reject duplicates — it only validates structure. The access list is user-controlled input signed into the transaction. This is directly reachable with no special privileges.

---

### Recommendation

In `parse_and_warm_up_access_list`, maintain a seen-set (e.g., a `BTreeSet` or `HashSet`) of addresses already processed. Before charging and touching an address, check if it has already been seen; if so, skip the charge and touch (or charge only the warm-access cost). Similarly, within each address's `slots_list`, deduplicate storage keys before charging and touching. This mirrors how Ethereum clients handle duplicate access list entries in practice (they are idempotent for warming purposes).

Alternatively, reject transactions with duplicate access list entries during validation (returning `InvalidTransaction::InvalidStructure`), which is the stricter approach.

---

### Proof of Concept

1. Craft an EIP-2930 transaction with an access list containing the same address repeated 100 times, each with the same storage key repeated 10 times.
2. Submit to ZKsync OS bootloader.
3. In `parse_and_warm_up_access_list`:
   - Iteration 1: address `A` is cold → full cold-access native cost charged, account warmed.
   - Iterations 2–100: address `A` is already warm → only `WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST = 4000` native charged per iteration (from `materialize_element` line 114), plus `PER_ADDRESS_ACCESS_LIST_NATIVE_COMPUTATIONAL_OVERHEAD` and `ACCESS_LIST_ADDRESS * ERGS_PER_GAS` ergs per iteration.
   - For ZK transactions: intrinsic formula pre-charges `100 × L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_ACCESS_LIST_PER_ADDRESS` which includes `100 × NEW_COLD_ACCOUNT_READ_COST`, vastly over-estimating actual native consumption for 99 warm hits.
4. The transaction's native budget is drained by `99 × (PER_ADDRESS_ACCESS_LIST_NATIVE_COMPUTATIONAL_OVERHEAD + WARM_ACCOUNT_CACHE_ACCESS_NATIVE_COST)` more than necessary, and the intrinsic formula rejects the transaction if the gas limit does not cover the inflated estimate.
