### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Freezing of Unclaimed Yield - (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both the staking contract and the delegation pool contract accept `address(0)` as a valid `reward_address` at every entry point. When rewards are later claimed, the protocol unconditionally transfers STRK to whatever address is stored — including the zero address — permanently burning or freezing the unclaimed yield with no recovery path.

---

### Finding Description

The external report's vulnerability class is a **token-freeze bug**: a missing guard allows a zero-valued destination address to coexist with a non-zero reward amount, causing the reward to be irrecoverably lost when the claim path executes.

The exact same structural flaw exists in Starknet Staking. Four entry points accept `reward_address` from an unprivileged caller and write it to storage without any zero-address check:

**1. `Staking::stake()`** — a new staker can supply `reward_address = 0`: [1](#0-0) 

The only guard present is `!self.does_token_exist(token_address: reward_address)`, which returns `false` for address(0) (address(0) is neither STRK nor any registered BTC token), so the check passes and zero is accepted.

**2. `Staking::change_reward_address()`** — an existing staker can update to `reward_address = 0` at any time, including after rewards have already accrued: [2](#0-1) 

**3. `Pool::enter_delegation_pool()`** — a new delegator can supply `reward_address = 0`: [3](#0-2) 

**4. `Pool::change_reward_address()`** — an existing pool member can update to `reward_address = 0`: [4](#0-3) 

Once `reward_address = 0` is stored, every downstream reward-transfer path executes unconditionally against it:

**Staking contract — `send_rewards_to_staker()`** (called by both `claim_rewards` and `unstake_action`): [5](#0-4) 

Line 1625 calls `token_dispatcher.checked_transfer(recipient: reward_address, ...)` with no zero guard.

**Pool contract — `claim_rewards()`**: [6](#0-5) 

Line 366 calls `reward_token.checked_transfer(recipient: reward_address, ...)` with no zero guard.

---

### Impact Explanation

**Scenario A — STRK ERC20 allows transfer to address(0) (burn semantics):**  
Rewards are silently transferred to address(0) and permanently destroyed. The pool member's or staker's `unclaimed_rewards` is zeroed out in storage, so there is no recovery path. This is a **permanent freezing (burning) of unclaimed yield** — High severity.

**Scenario B — STRK ERC20 reverts on transfer to address(0):**  
`claim_rewards` and `unstake_action` revert every time they are called, making it impossible for the user to retrieve their accrued rewards or complete unstaking. The user is locked out of their yield until they call `change_reward_address` with a valid address. This is a **temporary freezing of unclaimed yield / griefing** — Medium severity.

In both scenarios the impact falls within the allowed scope:
- *High: Permanent freezing of unclaimed yield* (Scenario A)
- *Medium: Griefing with damage to users* (Scenario B)

---

### Likelihood Explanation

Any unprivileged staker or delegator can trigger this by passing `reward_address = 0` to `stake()` or `enter_delegation_pool()`, or by calling `change_reward_address(0)` after joining. No privileged role, bridge compromise, or external dependency is required. The entry path is direct and permissionless. Likelihood is low-to-medium (user error or deliberate self-harm), but the missing guard is a straightforward omission that should be closed regardless.

---

### Recommendation

Add an explicit non-zero assertion on `reward_address` at every entry point that accepts or updates it:

```cairo
// In stake(), change_reward_address() (staking), enter_delegation_pool(), change_reward_address() (pool):
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the pattern already used elsewhere in the codebase (e.g., `add_token` at line 1346 of `staking.cairo` asserts `token_address.is_non_zero()`). [7](#0-6) 

---

### Proof of Concept

1. Deploy the system normally.
2. Call `Staking::stake(reward_address: 0, operational_address: <valid>, amount: <min_stake>)`. The call succeeds — no zero check exists.
3. Advance epochs and attest so that `unclaimed_rewards_own` accumulates.
4. Call `Staking::claim_rewards(staker_address)`.
5. Inside `send_rewards_to_staker`, `claim_from_reward_supplier` moves STRK from the reward supplier into the staking contract, then `checked_transfer(recipient: 0, amount: rewards)` is executed.
   - If the STRK token burns on transfer to zero: rewards are gone, `unclaimed_rewards_own` is zeroed, no recovery.
   - If the STRK token reverts: the entire transaction reverts; the staker cannot claim or unstake until they call `change_reward_address` with a valid address.

The same proof of concept applies to pool members via `Pool::enter_delegation_pool(reward_address: 0, ...)` followed by `Pool::claim_rewards(pool_member)`.

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
