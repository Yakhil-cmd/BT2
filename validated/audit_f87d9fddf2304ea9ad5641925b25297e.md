### Title
Unauthorized Contract Removal via Inconsistent On-Chain vs. Context-Flag Restriction Enforcement - (File: `fvm/environment/contract_updater.go`)

### Summary
The `ContractUpdaterImpl` in the FVM enforces deployment restriction by reading an on-chain boolean flag (`IsContractDeploymentRestrictedPath`) from the service account at runtime, but enforces removal restriction exclusively from a static context flag (`RestrictContractRemoval`), never reading an equivalent on-chain value. This asymmetry means that when the on-chain deployment restriction flag is set to `false` (unrestricted deployment), the removal restriction still falls back to the context-level flag — which in certain configurations (e.g., the emulator, or any node where `RestrictContractRemoval` defaults to `false`) allows any account holding the `RemoveContract` entitlement to remove any contract without service-account co-signing.

### Finding Description

`RestrictedDeploymentEnabled()` reads the on-chain state to determine whether deployment is restricted:

```go
func (impl *contractUpdaterStubsImpl) RestrictedDeploymentEnabled() bool {
    enabled, defined := impl.getIsContractDeploymentRestricted()
    if !defined {
        return impl.RestrictContractDeployment  // fallback
    }
    return enabled
}
``` [1](#0-0) 

`RestrictedRemovalEnabled()` never reads on-chain state — it only returns the context flag:

```go

`RemoveContract` enforces removal authorization only through `isAuthorizedForRemoval`, which delegates to `RestrictedRemovalEnabled()`:

```go
func (updater *ContractUpdaterImpl) RemoveContract(...) (err error) {
    if !updater.isAuthorizedForRemoval(signingAccounts) {
        return fmt.Errorf("removing contract failed: %w", ...)
    }
    ...
}
``` [3](#0-2) 

`isAuthorizedForRemoval` only checks the authorized-accounts list when `RestrictedRemovalEnabled()` is `true`. When it returns `false`, removal is unconditionally allowed:

```go
func (updater *ContractUpdaterImpl) isAuthorizedForRemoval(signingAccounts []flow.Address) bool {
    if updater.RestrictedRemovalEnabled() {
        return updater.isAuthorized(signingAccounts, blueprints.ContractRemovalAuthorizedAddressesPath)
    }
    return true
}
``` [4](#0-3) 

The default `ContractUpdaterParams` sets `RestrictContractRemoval: true`, but this can be overridden. The emulator explicitly sets `WithContractRemovalRestricted(!b.conf.ContractRemovalEnabled)`, and `ContractRemovalEnabled` defaults to `false`, meaning `RestrictContractRemoval = true` in the emulator. However, the `SystemChunkContext` explicitly disables removal restriction:

```go
fvm.WithContractRemovalRestricted(false),
``` [5](#0-4) 

The critical asymmetry is that for deployment, the on-chain flag can override the context flag (making restriction tighter or looser than the context default), but for removal, the on-chain flag is **never consulted**. If an operator sets `RestrictContractRemoval: false` in context (or the system chunk context does so), any account with the Cadence `RemoveContract` entitlement can remove any contract on their account without service-account co-signing, bypassing the intended governance control.

The duplicate/inconsistent pattern is the root cause: the deployment path has a two-layer check (on-chain flag → authorized-accounts list), while the removal path has only a one-layer check (context flag → authorized-accounts list), with the on-chain layer missing entirely.

### Impact Explanation

An unprivileged transaction sender who owns an account with a deployed contract can submit a transaction using `auth(RemoveContract) &Account` to call `signer.contracts.remove(name: "X")`. If `RestrictContractRemoval` is `false` in the active FVM context (e.g., system chunk context, or any deployment where the operator has not explicitly set the flag), the removal proceeds without service-account authorization. This allows unauthorized destruction of on-chain smart contracts — a permanent, irreversible state mutation. Any contracts that other accounts or dApps depend on can be destroyed, breaking dependent functionality and potentially enabling asset loss if the removed contract controlled resource custody.

### Likelihood Explanation

The `SystemChunkContext` in production execution nodes explicitly sets `WithContractRemovalRestricted(false)`, meaning the system chunk transaction runs with removal unrestricted. While the system chunk is not directly user-controlled, the inconsistency creates a fragile invariant: any future code path that creates a context with `RestrictContractRemoval: false` (or fails to set it to `true`) silently removes the removal guard. The TODO comment in the code itself acknowledges this is an incomplete implementation. The likelihood of exploitation is moderate — it requires the attacker to control an account with a deployed contract and to submit a transaction in a context where the flag is `false`.

### Recommendation

1. Implement `RestrictedRemovalEnabled()` to read the on-chain state analogously to `RestrictedDeploymentEnabled()`, using a dedicated on-chain path (e.g., `IsContractRemovalRestrictedPath`).
2. Add a corresponding on-chain storage path in `fvm/blueprints/contracts.go` for the removal restriction flag.
3. Remove the asymmetry so both deployment and removal restrictions are governed by the same two-layer (on-chain flag → authorized-accounts list) mechanism.
4. Audit all `fvm.Context` construction sites to ensure `RestrictContractRemoval` is not inadvertently set to `false` in user-reachable transaction contexts.

func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,
) error {
	// Initial contract deployments must be authorized by signing accounts.
	//
	// Contract updates are always allowed.
	exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
	if err != nil {
		return err
	}

	if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
		return fmt.Errorf(
			"deploying contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"SetContract",
				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
```
