### Title
`class_hash` Hardcoded to `ClassHash::default()` in `BroadcastedDeclareV1` Conversion Corrupts Transaction Hash and Simulation State Diff — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The `TryFrom<BroadcastedDeclareTransaction>` implementation for `ExecutableTransactionInput` unconditionally sets `class_hash: ClassHash::default()` (i.e., `Felt::ZERO`) for every V1 declare transaction. Because `class_hash` is a committed input to the Starknet V1 declare transaction hash function, this produces a deterministically wrong transaction hash for every `estimate_fee` and `simulate_transaction` call involving a V1 declare. It also causes the `simulate_transaction` state diff to report `deprecated_declared_classes: [ClassHash(0)]` instead of the actual class hash.

---

### Finding Description

At conversion time, the V1 branch of `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` explicitly writes:

```rust
class_hash: ClassHash::default(),
```

with the comment *"The blockifier doesn't need the class hash, but it uses the SN_API DeclareTransactionV0V1 which requires it."* [1](#0-0) 

The resulting `ExecutableTransactionInput::DeclareV1` carries a `DeclareTransactionV0V1` whose `class_hash` field is `Felt::ZERO` regardless of what the caller submitted.

**Hash computation path.** `calc_tx_hashes` calls `calc_tx_hash` on each input, which calls `get_transaction_hash` → `DeclareTransactionV0V1::calculate_transaction_hash` → `get_declare_transaction_v1_hash`. That function chains `transaction.class_hash.0` directly into the Pedersen hash:

```rust
.chain(&HashChain::new().chain(&transaction.class_hash.0).get_pedersen_hash())
``` [2](#0-1) 

Because `class_hash` is always zero, the computed `TransactionHash` is always wrong for any non-zero class. [3](#0-2) 

**State diff path.** `execute_transactions` extracts `deprecated_declared_class_hash` from the same zeroed field:

```rust
ExecutableTransactionInput::DeclareV1(
    DeclareTransactionV0V1 { class_hash, .. }, _, _, _,
) => Some(*class_hash),
``` [4](#0-3) 

This zero value is then passed to `induced_state_diff`, which places it verbatim into `ThinStateDiff::deprecated_declared_classes`:

```rust
deprecated_declared_classes: deprecated_declared_class_hash
    .map_or_else(Vec::new, |class_hash| vec![class_hash]),
``` [5](#0-4) 

The simulation output therefore always reports `deprecated_declared_classes: [0x0]` for any V1 declare, regardless of the actual class submitted.

---

### Impact Explanation

Two concrete wrong values are returned to callers:

1. **Wrong transaction hash** — `simulate_transaction` and `estimate_fee` compute and use a hash derived from `class_hash = 0`. Any account whose `__validate__` entry point reads `get_execution_info` (standard ECDSA accounts do) will receive this wrong hash. When validation is not skipped, the signature check will fail against the wrong hash, causing `estimate_fee` to return an execution error for a transaction that would succeed on-chain. When validation is skipped, the wrong hash is still embedded in the simulation trace returned to the caller.

2. **Wrong state diff** — `simulate_transaction` returns `deprecated_declared_classes: [0x0]` in the induced state diff instead of the actual class hash. A caller using this output to reason about what class was declared (e.g., to check whether a class is already declared before submitting) receives authoritative-looking but incorrect data.

Both impacts fall squarely within:
- *High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*
- *High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.*

---

### Likelihood Explanation

This is triggered by any unprivileged caller invoking `starknet_estimateFee` or `starknet_simulateTransactions` with a `BroadcastedDeclareTransaction::V1`. No special state, privileges, or race conditions are required. The code path is unconditional.

---

### Recommendation

Compute the actual class hash from the submitted `contract_class` before constructing the `DeclareTransactionV0V1`. The `starknet_api` crate already provides a `ClassHash` computation for deprecated contract classes. Replace the hardcoded default with:

```rust
let class_hash = sn_api_contract_class.compute_hash(); // or equivalent
```

and pass it as `class_hash` in the struct literal. This ensures the transaction hash and state diff both reflect the actual class being declared.
