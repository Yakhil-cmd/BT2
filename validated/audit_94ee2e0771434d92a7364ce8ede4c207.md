### Title
`set_eth_connector_contract_account` Strands Old Connector Funds Without Migrating Balances — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `set_eth_connector_contract_account` function updates the engine's stored ETH connector account ID without withdrawing or migrating the underlying NEAR token balances held by the old connector. Any NEAR tokens backing ETH in the Aurora EVM that remain in the old connector become inaccessible to the engine, causing all user ETH withdrawal attempts to fail against the new (empty) connector.

---

### Finding Description

`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` performs only a storage write — it replaces the stored connector account ID and serialization type, then returns: [1](#0-0) 

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        ...
        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

`set_connector_account_id` is a bare storage write with no side-effects: [2](#0-1) 

The ETH connector is an external NEAR contract that holds the NEAR tokens backing all ETH minted inside the Aurora EVM. When a user deposits ETH, the connector receives NEAR tokens and the engine mints EVM ETH. When a user withdraws, the engine burns EVM ETH and calls the connector to release the NEAR tokens. After `set_eth_connector_contract_account` is called with a new account:

1. The engine routes all future connector calls (`engine_withdraw`, `ft_on_transfer`, etc.) to the **new** connector.
2. The **old** connector still holds all previously deposited NEAR tokens.
3. The new connector has zero NEAR token balance.
4. Every user withdrawal attempt calls the new connector, which cannot release funds it does not hold — the call fails.

There is no function in the engine to pre-drain the old connector before switching, nor to post-migrate its balance. The `withdraw` path in `connector.rs` unconditionally routes to whichever account is currently stored: [3](#0-2) 

The `ft_on_transfer` path similarly reads the current connector account ID at call time: [4](#0-3) 

---

### Impact Explanation

All ETH held by users inside the Aurora EVM is backed 1:1 by NEAR tokens in the connector. After the connector account is switched without migration:

- **User withdrawals fail**: the engine calls the new connector, which holds no NEAR tokens.
- **Old connector NEAR tokens are stranded**: the engine has no path to interact with the old connector again.
- **Protocol insolvency risk**: EVM ETH balances exist with no redeemable backing in the active connector.

This constitutes **High — Temporary freezing of funds** (reversible only if the owner calls `set_eth_connector_contract_account` again to restore the old account, but the window of inaccessibility is real and user-impacting).

---

### Likelihood Explanation

`set_eth_connector_contract_account` is a documented, callable owner function exposed at the NEAR contract level: [5](#0-4) 

A legitimate upgrade scenario — deploying a new version of the external ETH connector — would naturally trigger this call. The owner acting in good faith would switch the account ID without realizing that the old connector's NEAR token reserves must first be drained and transferred. This is the same class of good-faith operational mistake as the original report.

---

### Recommendation

Before overwriting the connector account ID, the function should:

1. Read the current (old) connector account ID.
2. Query the old connector's NEAR token balance held on behalf of the engine.
3. Issue a cross-contract call to the old connector to transfer those NEAR tokens to the new connector (or to the engine itself as an intermediary).
4. Only after the transfer callback succeeds, write the new connector account ID to storage.

```rust
// Pseudocode sketch
let old_account = get_connector_account_id(&io)?;
// Schedule promise: old_account.transfer_balance_to(new_account)
// In callback: set_connector_account_id(io, &args.account);
```

---

### Proof of Concept

1. Engine is live; users have deposited ETH. Old connector (`connector-v1.near`) holds 1000 NEAR backing 1000 ETH in the EVM.
2. Owner calls `set_eth_connector_contract_account` with `connector-v2.near` (freshly deployed, zero balance).
3. Engine storage now points to `connector-v2.near`.
4. User calls `withdraw` to redeem 10 ETH → engine calls `connector-v2.near::engine_withdraw` → fails (no balance).
5. `connector-v1.near` still holds 1000 NEAR; engine has no stored reference to it and no function to call it.
6. All user ETH withdrawals are blocked until the owner manually intervenes.

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
