### Title
Malformed MessagePack Payload from Malicious Participant Causes Unhandled Deserialization Error That Permanently Halts DKG, Reshare, Refresh, and Presign Protocols - (File: src/protocol/helpers.rs)

### Summary
`recv_from_others` propagates deserialization errors from `Comms::recv` directly to the caller via the `?` operator. A malicious participant can craft a message with a syntactically valid `MessageHeader` but a malformed MessagePack payload, causing `rmp_serde::decode::from_slice` to fail. This error bubbles up through `recv_from_others` and terminates the DKG, reshare, refresh, or presign protocol for every honest participant that processes the message.

### Finding Description

**Root cause — `src/protocol/helpers.rs`, line 20:**

```rust
pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;   // ← error propagates
        if seen.put(from) {
            messages.push((from, msg));
        }
    }
    Ok(messages)
}
```

`chan.recv` ultimately calls `Comms::recv` in `src/protocol/internal.rs`:

```rust
async fn recv<T: DeserializeOwned>(
    &self,
    header: MessageHeader,
) -> Result<(Participant, T), ProtocolError> {
    let (from, data) = self.incoming.pop(header).await;
    let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| {
        ProtocolError::DeserializationError("Failed to deserialize message data".to_string())
    })?;
    let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
        rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
    Ok((from, decoded?))   // ← deserialization failure propagates
}
```

The message ingestion path (`push_message`) only validates the header, not the payload:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);   // payload never validated
}
```

So any message with a valid 40-byte header is accepted into the buffer. The payload is only decoded when `recv` is called, and a failure there propagates as a fatal `ProtocolError`.

**Contrast with the echo-broadcast path** (`src/protocol/echo_broadcast.rs`, line 179–182), which explicitly swallows deserialization errors:

```rust
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // ← errors are silently ignored
};
```

`recv_from_others` has no equivalent guard.

**Call sites where this is fatal:**

- `do_keyshare` (DKG / reshare / refresh) in `src/dkg.rs`, lines 422–426 and 514–528 — both `recv_from_others` calls propagate the error directly to the caller via `?`, aborting the entire key-generation or resharing session.
- `do_presign` in `src/ecdsa/ot_based_ecdsa/presign.rs`, lines 114 and 151 — same pattern, aborting presign.

### Impact Explanation

A single malicious participant can permanently abort any in-progress DKG, reshare, refresh, or presign session for every honest participant that processes the crafted message. The protocol has no recovery path once `recv_from_others` returns an error; the session must be restarted from scratch. This matches:

> **High: Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation

Any participant in the protocol can call `Protocol::message(from, data)` with attacker-controlled bytes. The attacker only needs to know the correct `MessageHeader` (channel tag + waitpoint), which is deterministic and derivable from the protocol structure. No cryptographic material or privileged access is required. The attack is trivially repeatable across restarts.

### Recommendation

Mirror the error-handling pattern already used in `reliable_broadcast_receive_all`. In `recv_from_others`, skip messages that fail to deserialize rather than propagating the error:

```rust
while !seen.full() {
    let Ok((from, msg)) = chan.recv(waitpoint).await else {
        continue;   // discard malformed messages, keep waiting
    };
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

This is safe because the protocol already validates message content (proof-of-knowledge, commitment hash, share validity) after collection; a participant that sends only garbage will be detected as absent or malicious at those later checks.

### Proof of Concept

1. Honest participants `A`, `B`, `C` start DKG. Each calls `chan.send_many(wait_round_1, &commitment_hash)`.
2. Malicious participant `M` (also in the participant list) constructs a message:
   - First 40 bytes: valid `MessageHeader` with the correct shared-channel tag and `waitpoint = wait_round_1`.
   - Remaining bytes: `0xFF 0xFF 0xFF …` (invalid MessagePack).
3. `M` delivers this message to participant `A` via `protocol_A.message(M, crafted_bytes)`.
4. `A`'s `recv_from_others` loop calls `chan.recv(wait_round_1)`, which pops `M`'s message, calls `rmp_serde::decode::from_slice` on the garbage payload, gets `Err(...)`, and returns `Err(ProtocolError::Other(...))`.
5. `do_keyshare` propagates this error; `A`'s DKG session terminates with an error.
6. `M` repeats for `B` and `C`. The entire DKG round is aborted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
