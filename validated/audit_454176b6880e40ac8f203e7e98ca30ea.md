### Title
Gas Refund Minted to Zero Address Instead of Transaction Sender When `refund_recipient` Is Unset — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In `process_l1_transaction`, when an L1→L2 transaction has `reserved[1]` (the `refund_recipient` field) set to the zero address, the unused-gas refund is minted directly to `address(0)` and permanently burned. ZKsync OS lacks the zero-address fallback to `transaction.from` that the ZKsync Era bootloader implements, causing the transaction sender to lose their entire gas refund.

---

### Finding Description

After execution, `process_l1_transaction` computes `to_refund_recipient` (the unused-gas portion of the deposit) and mints it to the address stored in `transaction.reserved[1]`:

```rust
// process_l1_transaction.rs line 336-359
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
    mint_base_token::<S, Config>(
        system,
        system_functions,
        memories.reborrow(),
        &to_refund_recipient,
        &refund_recipient,   // ← zero address if reserved[1] == 0
        l1_chain_id,
        &mut inf_resources,
        tracer,
        validator,
    )?;
}
``` [1](#0-0) 

There is no check for `refund_recipient == address(0)` and no fallback to `transaction.from`. The ZKsync Era bootloader (the predecessor) explicitly handles this case:

```solidity
address refundRecipient = _transaction.reserved[1] != 0
    ? address(uint160(_transaction.reserved[1]))
    : _transaction.from;
```

The `L1TxBuilder` test utility confirms that zero is the default value for `refund_recipient`:

```rust
refund_recipient: self.refund_recipient.unwrap_or_default(),
// unwrap_or_default() → Address::ZERO
``` [2](#0-1) 

The `ZKsyncL1Tx` struct encodes `refund_recipient` into `reserved[1]`:

```rust
let reserved = [
    self.to_mint,
    U256::from(refund_recipient),  // zero if not set
    U256::ZERO,
    U256::ZERO,
];
``` [3](#0-2) 

The `to_refund_recipient` amount is non-zero whenever `gas_limit > gas_used` (success path) or `total_deposited > pay_to_operator` (failure path): [4](#0-3) 

`mint_base_token` calls `transfer_from_treasury`, which subtracts from the treasury and adds to the recipient — in this case `address(0)`: [5](#0-4) 

---

### Impact Explanation

When `reserved[1] == 0` and `to_refund_recipient > 0`, the refund amount `gas_price × (gas_limit − gas_used)` is transferred from the treasury to `address(0)` and permanently lost. On a failed transaction the loss is `total_deposited − pay_to_operator`, which can be the entire deposit minus the operator fee. The treasury balance is correctly debited, but the funds are irrecoverably burned rather than returned to the sender. This is a direct loss of user funds with no recovery path.

---

### Likelihood Explanation

The default `refund_recipient` for any L1→L2 transaction that does not explicitly set `reserved[1]` is `address(0)`. Because virtually every L1→L2 transaction consumes less gas than its `gas_limit`, `to_refund_recipient` will be positive in the common case. Any user or bridge contract that submits an L1→L2 transaction without specifying a non-zero `refund_recipient` will silently lose their gas refund on every transaction.

---

### Recommendation

Add a zero-address check before using `reserved[1]` as the refund recipient, falling back to `transaction.from` exactly as ZKsync Era does:

```rust
let refund_recipient_raw = transaction.reserved[1].read();
let refund_recipient = if refund_recipient_raw.is_zero() {
    transaction.from.read()
} else {
    u256_to_b160_checked(refund_recipient_raw)
};
``` [6](#0-5) 

---

### Proof of Concept

1. Submit an L1→L2 priority transaction with `reserved[1] = 0` (the default), `gas_price = 1000`, `gas_limit = 100_000`, and any calldata that consumes fewer than 100,000 gas units.
2. After execution, `to_refund_recipient = 1000 × (100_000 − gas_used) > 0`.
3. `mint_base_token` is called with `to = address(0)`.
4. The treasury balance decreases by `to_refund_recipient`.
5. `address(0)` balance increases by `to_refund_recipient`.
6. The transaction sender's balance is unchanged — the refund is permanently burned.

This is directly observable using the existing `test_treasury_based_token_distribution_regression` test framework, which already uses `refund_recipient = address(0)` as the default and verifies that the refund goes to the zero address rather than the sender. [7](#0-6)
