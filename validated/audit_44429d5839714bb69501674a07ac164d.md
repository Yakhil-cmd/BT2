### Title
`validateCOAOwnershipProof` Does Not Bind `signedData` to `evmAddress`, Enabling Cross-COA Signature Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's keys signed `signedData`, then checks that the COA at the given capability path matches `evmAddress`. However, it never enforces that `signedData` encodes `evmAddress`. A signature produced over any 32-byte value for one COA can therefore be replayed to pass `isValidSignature` (ERC-1271) on a *different* COA owned by the same Cadence account. The code itself acknowledges this gap but dismisses it as low-risk based on a historical assumption that is not a protocol invariant.

---

### Finding Description

`EVM.validateCOAOwnershipProof` accepts caller-supplied `signedData`, `keyIndices`, `signatures`, and `evmAddress`. It performs two checks:

1. The signatures are cryptographically valid over `signedData`.
2. The COA resource at the supplied `path` has an EVM address equal to `evmAddress`.

What it does **not** check is that `signedData` commits to `evmAddress`. The binding between the signed payload and the specific COA being authenticated is entirely absent.

The COA Solidity contract exposes this as ERC-1271:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig) external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig)
    );
    ...
}
```

`address(this)` is passed as `evmAddress`, and `_hash` is passed as `signedData`. The `_sig` blob is a caller-supplied `COAOwnershipProof` that contains the Cadence account address, capability path, key indices, and raw signatures. Because `signedData` is not required to include `evmAddress`, any signature Alice produced over `_hash` for COA_A is equally valid when presented against COA_B — provided the proof blob is constructed to point at COA_B's capability path.

The code comment at line 1003–1009 explicitly acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The mitigation offered — "Flow wallets historically create at most one COA per account" — is a social/historical assumption, not a protocol invariant. Nothing in the Cadence type system or the EVM contract prevents a single Cadence account from owning multiple COAs.

---

### Impact Explanation

An attacker who observes Alice calling `COA_A.isValidSignature(D, proof_A)` on-chain (all EVM transactions are public) can extract Alice's signature over `D`. The attacker then constructs `proof_B` — identical signatures, but with COA_B's capability path — and calls `COA_B.isValidSignature(D, proof_B)`. The call returns `ValidERC1271Signature` (0x1626ba7e) without Alice's consent.

Any EVM protocol that uses ERC-1271 to authorize asset transfers, NFT approvals, or governance votes on COA_B is now bypassable. This is a cross-VM asset loss vector: EVM-side assets held by or delegatable through COA_B can be moved without the Cadence account holder's authorization for that specific COA.

---

### Likelihood Explanation

The attack requires:
1. Alice owns two or more COAs (currently uncommon but not prevented).
2. Alice has previously signed some 32-byte value `D` in a context where the signature is observable on-chain (e.g., a prior `isValidSignature` call on COA_A, or any other EVM transaction that embeds a COA ownership proof).
3. A DApp uses ERC-1271 on COA_B with the same `D`.

All three conditions are realistic as the Flow EVM ecosystem grows and multi-COA patterns emerge. The attacker needs no privileged access, no private keys, and no staked node role — only the ability to submit an EVM transaction.

---

### Recommendation

`validateCOAOwnershipProof` should enforce that `signedData` encodes `evmAddress` as part of the signed payload. Concretely, the function should require that `signedData` equals `keccak256(abi.encode(evmAddress, applicationData))` or a similar domain-separated commitment that binds the signature to the specific COA being authenticated. Alternatively, the function should internally reconstruct the expected signed payload from `evmAddress` and reject proofs where the signature does not cover it.

---

### Proof of Concept

**Setup:**
- Alice's Cadence account: `0xAlice`
- COA_A at `/public/coaA`, EVM address `0xCOA_A`
- COA_B at `/public/coaB`, EVM address `0xCOA_B`
- Alice signs `D = keccak256("some application data")` for COA_A

**Step 1 — Alice legitimately uses COA_A:**
```
COA_A.isValidSignature(D, proof_A)
// proof_A = { Address: 0xAlice, CapabilityPath: "coaA", KeyIndices: [0], Signatures: [sig_alice_over_D] }
// → ValidERC1271Signature ✓
```
This transaction is public on-chain; `sig_alice_over_D` is extractable.

**Step 2 — Attacker replays against COA_B:**
```
COA_B.isValidSignature(D, proof_B)
// proof_B = { Address: 0xAlice, CapabilityPath: "coaB", KeyIndices: [0], Signatures: [sig_alice_over_D] }
```

Inside `validateCOAOwnershipProof`:
- `keyList.verify(signatureSet, signedData: D, ...)` → **true** (Alice's key did sign D)
- `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coaB)?.address()` → `0xCOA_B`
- `evmAddress == 0xCOA_B` → **true** (precompile passed `address(this)`)
- Returns `ValidationResult(isValid: true)` → `ValidERC1271Signature`

Alice never authorized this action on COA_B. Any DApp that trusted this ERC-1271 result to release COA_B's assets has been deceived. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
