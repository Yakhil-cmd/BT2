### Title
Hardcoded 50/50 ETH Fee Split in `tick()` Causes Permanent Inability to Mint Rewards When Message Fees Diverge - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` sends two distinct L1→L2 messages and splits `msg.value` exactly 50/50 between them. Because the two messages target different L2 handlers with different computational costs, their required fees can diverge. When they do, no caller can supply a `msg.value` that satisfies both messages simultaneously, permanently blocking the reward minting pipeline and freezing unclaimed yield.

---

### Finding Description

In `L1/starkware/solidity/stake/RewardSupplier.sol`, the `tick()` function performs two payable cross-chain operations:

1. `bridge().depositWithMessage{value: msgFee}(...)` — deposits minted STRK tokens via StarkGate to the L2 `RewardSupplier`.
2. `messagingContract().sendMessageToL2{value: msgFee}(...)` — sends a `totalSupply` update to the L2 `MintingCurve` contract.

The fee allocation is:

```solidity
uint256 msgFee = msg.value / 2;          // line 125 — half to bridge deposit
bridge().depositWithMessage{value: msgFee}(...);

msgFee = msg.value - msgFee;             // line 135 — remaining half to messaging
messagingContract().sendMessageToL2{value: msgFee}(...);
``` [1](#0-0) 

The split is always exactly 50/50 regardless of the actual fee requirements of each message. The two messages differ in nature:

- The bridge deposit (`depositWithMessage`) triggers an L2 `on_receive` handler in the `RewardSupplier` contract, which performs storage writes and token accounting.
- The `sendMessageToL2` to `mintingCurve()` triggers `update_total_supply`, which also performs storage writes.

Starknet L1→L2 message fees are proportional to the L2 gas consumed by the handler. If these two handlers have different L2 gas costs — which is expected since `on_receive` and `update_total_supply` are distinct functions — their required fees will differ. Because the split is hardcoded at 50/50, there is no `msg.value` a caller can provide that correctly allocates the right fee to each message when the required fees are unequal. The underfunded message will revert at the Starknet messaging contract level, causing the entire `tick()` call to revert. [2](#0-1) 

---

### Impact Explanation

`tick()` is the sole mechanism by which:
- Minted STRK tokens are bridged from L1 to the L2 `RewardSupplier`.
- The L2 `MintingCurve` receives updated total supply data for accurate inflation calculations.

If `tick()` cannot execute, `l1_pending_requested_amount` on L2 never decreases (tokens never arrive), `unclaimed_rewards` cannot be satisfied, and `claim_rewards` calls from the staking contract will fail once the L2 `RewardSupplier`'s token balance is exhausted. [3](#0-2) 

This constitutes **temporary (potentially permanent) freezing of unclaimed yield**: stakers and delegators accumulate rewards on L2 that cannot be paid out because the L1 minting pipeline is broken.

**Impact: High** — Freezing of unclaimed yield for all stakers and delegators.

---

### Likelihood Explanation

`tick()` is a public, permissionless function callable by any address. The Starknet L1→L2 fee market is dynamic; fees are set by the sequencer and can change with network upgrades or congestion. The two target handlers (`on_receive` in `RewardSupplier` and `update_total_supply` in `MintingCurve`) have different code paths and storage operations, making equal fee requirements coincidental rather than guaranteed. Any protocol upgrade that changes either handler's L2 gas cost will break the 50/50 assumption. Additionally, the `getMaxL1MsgFee()` cap enforced by the messaging contract means an overpayment to one message does not compensate for underpayment to the other. [4](#0-3) 

**Likelihood: Medium** — Requires fee divergence between the two message types, which is plausible at any protocol upgrade or fee schedule change.

---

### Recommendation

Allow the caller of `tick()` to specify the fee for each message independently, rather than hardcoding a 50/50 split. For example:

```solidity
function tick(uint256 bridgeMsgFee, uint256 mintingCurveMsgFee) external payable {
    require(msg.value >= bridgeMsgFee + mintingCurveMsgFee, "INSUFFICIENT_FEE");
    // ...
    bridge().depositWithMessage{value: bridgeMsgFee}(...);
    messagingContract().sendMessageToL2{value: mintingCurveMsgFee}(...);
}
```

This mirrors the recommendation in the reference report: allow callers to specify per-message bridge fees rather than forcing an equal split.

---

### Proof of Concept

1. Assume the Starknet sequencer sets the minimum L1→L2 fee for `on_receive` (bridge deposit handler) at `X` wei and for `update_total_supply` (MintingCurve handler) at `Y` wei, where `X ≠ Y` (e.g., `X = 3e14`, `Y = 1e14`).

2. A caller invokes `tick()` with `msg.value = X + Y = 4e14` wei (the exact total needed).

3. Inside `tick()`:
   - `msgFee = msg.value / 2 = 2e14` — allocated to `depositWithMessage`.
   - The bridge call requires `X = 3e14` but receives only `2e14` → **reverts**.

4. The caller tries `msg.value = 2 * X = 6e14` to ensure the bridge gets enough:
   - `msgFee = 3e14` → bridge succeeds.
   - Remaining `msgFee = 3e14` → `sendMessageToL2` receives `3e14` but only needs `1e14` — this succeeds but wastes `2e14` wei.
   - However, if `Y > X` instead, the second message reverts and no `msg.value` can fix it without the split being adjustable.

5. With a fixed 50/50 split, whenever `X ≠ Y`, either: (a) the caller must overpay one message to cover the other, or (b) if the cheaper message's fee is less than half the total but the expensive message needs more than half, no valid `msg.value` exists that passes both calls — `tick()` is permanently broken.

6. Result: L2 `RewardSupplier` never receives minted tokens; `claim_rewards` exhausts the existing balance; staker and delegator reward withdrawals revert. [5](#0-4) [6](#0-5)
