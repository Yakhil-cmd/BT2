### Title
COA Ownership Proof Signature Replay Across Multiple COAs Owned by the Same Cadence Account - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced to prove ownership of one COA (Cadence-Owned Account) can be replayed by any caller to pass ownership validation for a *different* COA belonging to the same Cadence account. The vulnerability is self-documented in the source code and is reachable by any unprivileged Cadence script or transaction author.

### Finding Description

`EVM.validateCOAOwnershipProof` is an `access(all)` Cadence function in the deployed EVM system contract. It accepts a Cadence `address`, a `path` to a COA capability, arbitrary `signedData`, `keyIndices`, `signatures`, and an `evmAddress` to verify against. [1](#0-0) 

The function verifies that the keys of the Cadence account at `address` produced valid signatures over `signedData`, then checks that the COA resource stored at `path` has an EVM address matching `evmAddress`. [2](#0-1) 

The critical flaw: **`signedData` is never checked to contain `evmAddress`**. The signature is verified only against the Cadence account's keys and the opaque `signedData` blob. The `evmAddress` and `path` parameters are supplied by the *caller*, not the signer. The code itself acknowledges this: [3](#0-2) 

If a Cadence account owns two COAs — stored at `/public/coaA` (EVM address `0xAAAA`) and `/public/coaB` (EVM address `0xBBBB`) — a signature produced over some `signedData` to prove ownership of `0xAAAA` is cryptographically identical to a valid signature for `0xBBBB`, because the signed bytes contain no COA address binding. Any caller can invoke `validateCOAOwnershipProof` with the intercepted `(signedData, signatures, keyIndices)` tuple but substitute `evmAddress = 0xBBBB` and `path = /public/coaB`. The function will return `isValid: true`.

The EVM precompile path (`verifyCOAOwnershipProof` in `fvm/evm/precompiles/arch.go`) is not directly vulnerable because it always passes `address(this)` — the calling COA — as the EVM address: [4](#0-3) 

However, the Cadence-level `EVM.validateCOAOwnershipProof` is `access(all)` and callable by any script or transaction with attacker-controlled arguments. [5](#0-4) 

### Impact Explanation

Any off-chain authentication system or on-chain Cadence contract that calls `EVM.validateCOAOwnershipProof` directly — rather than through the EVM precompile — and relies on the returned `isValid: true` to grant access tied to a specific COA EVM address, can be deceived. An attacker who intercepts or observes a valid proof for COA_A can replay it to impersonate ownership of COA_B (same Cadence account owner). This enables unauthorized COA ownership attestation, which can be exploited in ERC-1271 off-chain authentication flows, cross-VM bridge authorization checks, or any Cadence contract that gates actions on a specific EVM address using this proof primitive.

### Likelihood Explanation

The precondition — a single Cadence account owning more than one COA — is explicitly called out as the risk scenario in the code comment. While Flow wallets historically create at most one COA per account, the protocol imposes no such restriction. Any user or contract can create multiple COAs on a single Cadence account via `EVM.createCadenceOwnedAccount()`. The function is `access(all)`, so any unprivileged transaction sender or script author can call it with attacker-chosen `evmAddress` and `path` arguments while reusing an observed signature.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` before accepting the proof as valid. Concretely, the function should verify that the first 20 bytes (or a well-defined prefix) of `signedData` equal `evmAddress`, or define a canonical signed message format that commits to the COA EVM address. This mirrors the recommendation in the original Gateway report: include the target account address in the signed message.

### Proof of Concept

```cadence
// Attacker script — no special privileges required
import EVM from <EVM_CONTRACT_ADDRESS>

// Assume:
//   - flowAccount owns two COAs:
//       coaA at /public/coaA  (EVM address 0xAAAA...AAAA)
//       coaB at /public/coaB  (EVM address 0xBBBB...BBBB)
//   - The attacker observed a valid proof for coaA:
//       signedData, keyIndices, signatures  (signed by flowAccount's key)

access(all)
fun main(
    flowAccount: Address,
    signedData: [UInt8],          // captured from a legitimate coaA proof
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    coaBAddress: [UInt8; 20]      // 0xBBBB...BBBB — the target COA
): EVM.ValidationResult {

    // Replay the coaA signature against coaB by substituting path and evmAddress.
    // validateCOAOwnershipProof never checks that signedData encodes coaBAddress.
    return EVM.validateCOAOwnershipProof(
        address: flowAccount,
        path: /public/coaB,       // attacker-chosen path
        signedData: signedData,   // original bytes — no coaB address inside
        keyIndices: keyIndices,
        signatures: signatures,
        evmAddress: coaBAddress   // attacker-chosen target
    )
    // Returns ValidationResult(isValid: true, problem: nil)
    // even though the signer never authorized coaB.
}
