### Title
Missing Share Verification in CKD Coordinator Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` accepts participant shares without any cryptographic proof of correctness. A single malicious participant can send arbitrary group elements, causing the coordinator to compute and output a corrupted `CKDOutput`, resulting in an incorrect confidential derived key accepted by all honest parties.

### Finding Description
In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55), the coordinator receives `(norm_big_y, norm_big_c)` from each participant via `recv_from_others` and unconditionally accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no verification that:
1. `norm_big_y` is correctly formed as `λ_i · y_i · G` for some valid ephemeral scalar `y_i`
2. `norm_big_c` is correctly formed as `λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` using the participant's actual private share `x_i` (verifiable against their public key share from DKG)

The correct per-participant computation is defined in `compute_signature_share` (lines 148–181), but participants are never held to it. The coordinator blindly incorporates whatever group elements it receives into the final `CKDOutput::new(norm_big_y, norm_big_c)` at line 56.

This is structurally analogous to the PriceFeed Case 4 bug: just as the PriceFeed transitions from the Tellor-fallback state back to Chainlink and accepts its price without cross-validating it against Tellor, the CKD coordinator transitions from "collecting shares" to "finalizing output" and accepts each participant's contribution without validating it against any expected structure or the participant's known public key share.

All other multi-round protocols in this library (DKG, reshare, presign) include explicit cryptographic checks — proof-of-knowledge verification (`verify_proof_of_knowledge`), commitment hash binding (`verify_commitment_hash`), share consistency checks (`validate_received_share`), and exponent interpolation checks — before accepting participant contributions. The CKD protocol has none of these.

### Impact Explanation
A single malicious participant can corrupt the aggregate `CKDOutput`. The coordinator outputs `(Y_corrupted, C_corrupted)` and honest parties have no mechanism to detect the corruption. When the application unmasks using `app_sk`:

```
C_corrupted - app_sk · Y_corrupted
= (C_honest + δ_C) - app_sk · (Y_honest + δ_Y)
= msk · H(pk, app_id) + δ_C - app_sk · δ_Y
```

Unless `δ_C = app_sk · δ_Y` (which the attacker cannot enforce without knowing `app_sk`), the derived confidential key is wrong. TEE applications relying on this key will silently derive an incorrect secret — data encrypted under the correct key becomes unrecoverable, or TEE computations produce incorrect results. This matches: **High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation
Any single participant in the CKD protocol can trigger this. No special privilege is required beyond being a listed participant. The attacker sends their protocol message in the single communication round and the coordinator has no recourse. With `n` participants, any one of the `n-1` non-coordinator participants is a viable attacker.

### Recommendation
Add a zero-knowledge proof of correct share formation alongside each participant's `(norm_big_y, norm_big_c)` message. Concretely, the participant should prove in zero knowledge that:
- `norm_big_y = λ_i · y_i · G` for a known `y_i` (a Schnorr PoK on `norm_big_y`)
- `norm_big_c` is consistent with the participant's public key share from DKG (a sigma protocol relating `norm_big_c`, `norm_big_y`, `app_pk`, and the participant's public key share)

The coordinator must verify these proofs before incorporating any share into the aggregate, mirroring the `_bothOraclesLiveAndUnbrokenAndSimilarPrice` cross-check recommended for PriceFeed Case 4.

### Proof of Concept
1. Malicious participant `P_m` is one of the `n` participants in the CKD protocol.
2. Instead of calling `compute_signature_share`, `P_m` sends `(G, G)` — the BLS12-381 G1 generator — as their `(norm_big_y, norm_big_c)`.
3. The coordinator at line 53–54 adds these unconditionally: `norm_big_y += G`, `norm_big_c += G`.
4. The final output is `CKDOutput::new(Y_honest + G, C_honest + G)`.
5. The application unmasks: `(C_honest + G) - app_sk · (Y_honest + G) = msk · H(pk, app_id) + (1 - app_sk) · G`, which is not `msk · H(pk, app_id)`.
6. All honest parties accept this corrupted output as the legitimate CKD result; there is no round in which they can detect the deviation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?

        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
