### Title
Malicious Participant Can Corrupt CKD Output via Unverified Share Contributions — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator aggregates participant share contributions `(norm_big_y, norm_big_c)` with no cryptographic verification that each contribution was honestly computed. A single malicious participant can inject arbitrary group elements, causing the coordinator to produce a structurally valid but cryptographically incorrect `CKDOutput`. The requester then unmasks a wrong derived key, silently corrupting the CKD result.

### Finding Description
The external report's vulnerability class is **missing enforcement on a specific code path**: in Webpack/LavaMoat, ContextModules (dynamically-loaded modules) were not wrapped by LavaMoat's runtime enforcement, allowing a compromised module to bypass policy restrictions. The analog here is that the CKD coordinator path applies no enforcement on the shares it receives from participants — the enforcement that shares are correctly formed exists only implicitly in the honest participant's own computation, not in any coordinator-side check.

In `do_ckd_coordinator` the coordinator collects one `CKDOutput` from every other participant and unconditionally accumulates it:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation that each participant is supposed to perform is:

```
norm_big_y = λ_i · y_i · G
norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
``` [2](#0-1) 

There is no zero-knowledge proof, commitment, or consistency check that binds the received `(big_y, big_c)` pair to the participant's actual private share `x_i` or to the correct `app_id`/`app_pk`. The coordinator has no way to distinguish a correctly-formed share from an arbitrary pair of group elements.

The `recv_from_others` helper only enforces that exactly one message is received per participant; it performs no content validation. [3](#0-2) 

### Impact Explanation
The final `CKDOutput` is `(Y, C)` where `Y = Σ norm_big_y_i` and `C = Σ norm_big_c_i`. The requester recovers the derived key as `C − app_sk · Y`. If any participant substitutes arbitrary `(big_y', big_c')`, the recovered value deviates from `msk · H(pk ‖ app_id)` by an attacker-controlled offset. The requester silently accepts a wrong derived key with no indication of failure. This maps to: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs** — High impact.

### Likelihood Explanation
Any single participant in the CKD session can trigger this. Unlike the threshold ECDSA schemes (which tolerate up to `t` malicious parties), the CKD protocol requires all `n` participants to contribute correctly; there is no threshold redundancy. The attacker needs only to be a legitimate participant and deviate from the protocol in the share-sending step — no special capability is required.

### Recommendation
Add a non-interactive proof of correct share formation. Each participant should accompany `(norm_big_y, norm_big_c)` with a Chaum-Pedersen-style DLEQ proof demonstrating that the same scalar `λ_i · y_i` was used in both `norm_big_y = λ_i · y_i · G` and the `y`-component of `norm_big_c`, and that `norm_big_c` encodes the correct private share `x_i` relative to the public key. The coordinator must verify all proofs before accumulating contributions, rejecting and identifying any malicious participant.

### Proof of Concept
```
// Malicious participant deviates in do_ckd_participant:
// Instead of computing norm_big_y and norm_big_c honestly,
// send arbitrary group elements to the coordinator.

let fake_big_y = ElementG1::generator() * Scalar::from(1234u64); // arbitrary
let fake_big_c = ElementG1::generator() * Scalar::from(5678u64); // arbitrary
chan.send_private(waitpoint, coordinator, &(fake_big_y, fake_big_c))?;

// The coordinator at do_ckd_coordinator blindly adds these:
//   norm_big_y += fake_big_y   ← no verification
//   norm_big_c += fake_big_c   ← no verification
//
// Result: coordinator returns CKDOutput with corrupted (Y, C).
// Requester computes: C - app_sk * Y  ≠  msk * H(pk || app_id)
// The derived key is silently wrong.

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
