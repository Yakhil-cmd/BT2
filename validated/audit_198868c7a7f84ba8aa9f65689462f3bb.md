### Title
Unauthenticated `e_j` Share in Round 1 Allows Any Participant to Abort Presign for All Honest Parties — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

In `do_presign`, each participant broadcasts their Lagrange-weighted share `e_i` of the Beaver triple's `c` component. The only validation applied to received shares is a zero-check. Because no per-share commitment exists, a single malicious participant can send any non-zero scalar, causing the aggregate consistency check to fail for every honest party with no mechanism to identify the culprit.

---

### Finding Description

In Round 1 of `do_presign`, each participant computes `e_i = lambda_me * triple0.0.c` and broadcasts it. [1](#0-0) 

Received shares are accumulated with only a zero-check guard: [2](#0-1) 

The aggregate is then verified against the public commitment `big_e = args.triple0.1.big_c`: [3](#0-2) 

`big_e` is the commitment to the **aggregate** `e = sum(e_j)`, not to individual shares. [4](#0-3) 

There is no per-share commitment (no `e_j * G` published and verified), and no ZKP binding `e_j` to the triple. A malicious participant can send `e_j = Scalar::ONE` — which passes the `is_zero()` check — causing the aggregate `e` to be wrong. The check `big_e != (ProjectivePoint::GENERATOR * e).to_affine()` then fails for **every** honest party.

Although `recv_from_others` returns `(Participant, value)` pairs so the sender identity is known, honest parties cannot determine **which** received `e_j` is incorrect without per-share commitments. [5](#0-4) 

The same structural gap exists for `alpha_j`/`beta_j` in Round 2 (lines 151–168), but the `e_j` path is the earliest abort point.

---

### Impact Explanation

**High — Permanent denial of presign for honest parties.**

A single malicious participant can unconditionally abort every presign session by sending any non-zero `e_j` that differs from their real share. Honest parties cannot identify the malicious party from the error, so they cannot exclude them and retry successfully. The presign protocol is permanently blocked as long as the malicious party participates.

---

### Likelihood Explanation

Any participant in the presign protocol can trivially execute this attack — no special knowledge or cryptographic capability is required beyond participation. The attack requires only sending a single wrong scalar value. It is repeatable across every presign invocation.

---

### Recommendation

Introduce per-share commitments for `e_j` during triple generation (e.g., publish `e_j * G` for each participant as part of `TriplePub` or as a separate broadcast in Round 1 of presign). Each honest party can then verify `e_j * G` against the received scalar before accumulating. Alternatively, use a verifiable secret sharing scheme for the `c` component of triple0 so that individual shares can be checked against the public commitment polynomial.

---

### Proof of Concept

```
1. Set up presign with N honest parties + 1 malicious party.
2. Malicious party intercepts its own send at wait0 and substitutes
   e_j = Scalar::ONE instead of lambda_me * triple0.0.c.
3. All honest parties receive the tampered e_j, accumulate a wrong e,
   and hit the check at line 127:
       big_e != (ProjectivePoint::GENERATOR * e).to_affine()
4. All honest parties return:
       ProtocolError::AssertionFailed("received incorrect shares of kd")
5. The error message contains no participant identifier.
   Honest parties cannot determine who sent the bad share.
``` [6](#0-5)

    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }
