### Title
Signature Bypass via `skip_stateful_validations` Admits Unsigned Invoke Transactions to Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX shortcut for the deploy-account + invoke flow unconditionally skips the `__validate__` entry-point for any invoke transaction with `nonce == 1` whenever the sender address already appears in the mempool or a recent block. Because the only check is a presence test (`account_tx_in_pool_or_recent_block`), an unprivileged attacker who observes a victim's pending `deploy_account` transaction can immediately inject a forged invoke transaction with an arbitrary signature into the mempool without any cryptographic verification.

However, the check does **not** verify that the existing mempool entry is specifically a `deploy_account` transaction. Any transaction type for that address satisfies the condition.

### Impact Explanation

An attacker can inject an invoke transaction with a fabricated signature for any account whose `deploy_account` transaction is currently pending in the mempool. The transaction is admitted to the mempool without any cryptographic verification of the sender's identity. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The injected transaction will be rejected by the batcher during execution (because `__validate__` is enforced there with `validate: true`), but the mempool admission invariant — that every accepted transaction has been cryptographically verified by the account's `__validate__` entry point — is broken at the gateway boundary.

### Likelihood Explanation

The trigger is fully unprivileged and requires only:
1. Observing a pending `deploy_account` transaction in the public mempool (trivially observable).
2. Submitting an invoke transaction with `nonce = 1`, a valid-length but arbitrary signature, and resource bounds above the minimum threshold.

No private key material, privileged access, or special network position is required.

### Recommendation

Replace the presence-only check with a type-specific check: only skip `__validate__` when the mempool contains a **`deploy_account`** transaction for the sender address, not any arbitrary transaction. Alternatively, expose a dedicated `has_pending_deploy_account(address)` query on the mempool that checks the transaction type of the pending entry.

### Proof of Concept

1. Alice submits `deploy_account` for address `A` (class hash `C`, salt `S`). On-chain nonce for `A` is `0`. The transaction enters the mempool.

2. Attacker constructs an `RpcInvokeTransactionV3` with:
   - `sender_address = A`
   - `nonce = 1`
   - `signature = [Felt::ONE, Felt::TWO]` (two felts — passes the length check since `max_signature_length` is typically ≥ 2)
   - `resource_bounds.l2_gas.max_price_per_unit` ≥ `min_gas_price`

3. Gateway stateless validation passes (signature length ≤ limit, nonce DA mode = L1, no proof data).

4. `convert_rpc_tx_to_internal` computes the tx hash and produces an `ExecutableTransaction::Invoke` with `nonce = 1`.

5. `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(A)` → `Nonce(0)` (account not yet deployed).
   - `validate_nonce`: `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: nonce/fee checks pass; no signature check.
   - `skip_stateful_validations`: `nonce == 1 && account_nonce == 0` → calls `account_tx_in_pool_or_recent_block(A)` → `true` (Alice's deploy_account is in the pool) → returns `true`.

6. `run_validate_entry_point` is called with `skip_validate = true` → `execution_flags.validate = false` → `perform_validations` returns `Ok(())` without calling `__validate__`.

7. The attacker's invoke transaction with a forged signature is added to the mempool via `mempool_client.add_tx(...)`. [7](#0-6) [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
