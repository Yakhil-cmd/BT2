### Title
Unvalidated Sender Lookup in Echo Broadcast Terminates DKG/Reshare/Refresh — (File: src/protocol/echo_broadcast.rs)

---

### Summary

In `reliable_broadcast_receive_all`, the `from` participant received from the network is used to look up its index via `participants.index(from)?`. If `from` is not in the participants list, the `?` operator propagates `ProtocolError::InvalidIndex`, immediately terminating the entire broadcast session. Because `do_broadcast` is called three times inside `do_keyshare` (DKG, reshare, refresh), any party outside the participants list can permanently deny key generation by injecting a single message with an unrecognized sender ID.

---

### Finding Description

In `reliable_broadcast_receive_all`, incoming messages are processed in a loop. At line 199, the code validates that the session ID (`sid`) embedded in the message matches the sender's actual index in the participants list:

```rust
// src/protocol/echo_broadcast.rs:199
if state_sid.finish_send || sid != participants.index(from)? {
    continue;
}
```

`participants.index(from)` returns `Err(ProtocolError::InvalidIndex)` when `from` is absent from the participants list. The `?` operator propagates this error, causing `reliable_broadcast_receive_all` to return immediately with an error and abort the entire broadcast session. [1](#0-0) 

By contrast, the `recv_from_others` helper in `src/protocol/helpers.rs` correctly handles unknown senders: it calls `seen.put(from)`, which returns `false` for non-participants and simply continues the loop — no error is propagated. [2](#0-1) 

The `ParticipantCounter::put` method explicitly documents and implements this safe-ignore behavior: [3](#0-2) 

The `Protocol::message` trait accepts any `from: Participant` without validating membership in the participants list: [4](#0-3) 

The internal `push_message` function similarly performs no membership check before buffering the message: [5](#0-4) 

`do_broadcast` is called three times inside `do_keyshare` in `src/dkg.rs`:

- Line 362: session ID broadcast
- Line 435: commitment and proof-of-knowledge broadcast
- Line 531: success broadcast

A single injected message with an unknown `from` at any of these points causes `do_keyshare` to return an error, aborting DKG, reshare, or refresh for the affected participant. [6](#0-5) [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A party outside the participants list — or any network-level adversary able to inject a message — can abort any in-progress DKG, reshare, or refresh session by sending one message whose `from` participant ID is not present in the participants list. Because the attacker can repeat this on every retry, honest parties are permanently denied key generation, reshare, and refresh under any network model that does not authenticate and filter senders at the transport layer. The library does not document or enforce such a transport-layer guarantee, making this a library-level defect.

**Impact: High — Permanent denial of key generation, reshare, and refresh for honest parties.**

---

### Likelihood Explanation

The `Protocol::message` API is public and accepts arbitrary `from` values with no membership check. Any integrator whose network layer does not enforce that `from` is a known participant is vulnerable — a common omission since the library does not require or document it. A single malicious or misconfigured node outside the participant set is sufficient to trigger the abort on every protocol run.

---

### Recommendation

Replace the error-propagating lookup with a graceful skip, consistent with how `recv_from_others` handles unknown senders:

```rust
// Before (line 199 of echo_broadcast.rs):
if state_sid.finish_send || sid != participants.index(from)? {
    continue;
}

// After:
let Ok(from_index) = participants.index(from) else {
    continue; // silently discard messages from non-participants
};
if state_sid.finish_send || sid != from_index {
    continue;
}
```

This makes `reliable_broadcast_receive_all` consistent with `recv_from_others` and ensures that messages from non-participants are silently discarded rather than causing a protocol abort.

---

### Proof of Concept

1. Start a DKG session with participants `[A, B, C]` and threshold 2.
2. Before participant A completes the first `do_broadcast` call (session ID exchange at `dkg.rs:362`), inject a message into A's channel with `from = D` (where `D ∉ {A, B, C}`) and any syntactically valid `(sid, vote)` payload.
3. Participant A's `reliable_broadcast_receive_all` receives the message, calls `participants.index(D)`, receives `Err(ProtocolError::InvalidIndex)`, and the `?` at line 199 propagates the error.
4. `do_broadcast` returns an error; `do_keyshare` returns an error; the DKG session for participant A is aborted.
5. Repeating this injection on every retry permanently prevents DKG completion for honest parties.
