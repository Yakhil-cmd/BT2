### Title
WalletConnect `chia_sendTransaction` Hides `wallet_id` from Confirmation Dialog, Enabling Covert CAT Token Transfers - (File: `packages/gui/src/electron/commands/Commands.ts`)

### Summary
The `wallet_id` parameter in the `chia_sendTransaction` WalletConnect command is marked `hide: true` in the confirmation dialog schema, so it is excluded from the user-visible confirmation rows. A malicious dApp granted `chia_sendTransaction` can supply any `wallet_id` — including a CAT wallet — while the user sees the amount rendered as XCH via the unconditional `mojo-to-xch` humanizer. The user approves what appears to be an XCH send but the daemon executes a CAT token transfer to the attacker's address.

### Finding Description

In `Commands.ts`, the `chia_wallet.send_transaction` schema marks `wallet_id` with `hide: true`: [1](#0-0) 

The `hide: true` flag causes `wallet_id` to be omitted from the main confirmation rows (it only appears in the collapsed "Raw data" section). The `amount` field is always humanized with `mojo-to-xch` regardless of the actual wallet type: [2](#0-1) 

`parseDappParams` accepts a dApp-supplied `wallet_id` — the test suite explicitly confirms that `walletId: 2` overrides the default: [3](#0-2) 

The default is `wallet_id: 1` (standard XCH wallet), but the dApp can override it: [4](#0-3) 

When the dApp supplies `wallet_id: 2` (a CAT wallet), the confirmation dialog shown to the user via `openReactDialog` displays:

- **Amount**: X XCH (actually X CAT mojos divided by 10¹², a misleading figure)
- **Fee**: Y XCH
- **Address**: attacker-controlled address
- **wallet_id**: *not shown* [5](#0-4) 

The same `parsedParams` object — including the hidden `wallet_id` — is forwarded unchanged to the daemon after the user clicks "Send": [6](#0-5) 

### Impact Explanation

A malicious dApp granted `chia_sendTransaction` can silently redirect the spend to any wallet the user holds. By targeting a CAT wallet with a large balance and supplying an amount whose `mojo-to-xch` rendering looks innocuous (e.g., `1 000 000 000 000` CAT mojos displays as `1 XCH`), the attacker causes the user to approve a transfer of a large number of CAT tokens to an attacker-controlled address. The user suffers direct, irreversible CAT token loss while believing they approved a routine 1 XCH send.

This matches the **High** impact criterion: *WalletConnect state causes a user to approve the wrong asset, identity, amount, or destination.*

### Likelihood Explanation

The attacker must control a dApp that the victim has already paired and granted `chia_sendTransaction`. Once paired, the dApp can call `chia_getWallets` (if also granted, which is common) to enumerate wallet IDs, or simply iterate integers starting at 2. No cryptographic material, leaked keys, or host compromise is required. The attack is a single crafted `chia_sendTransaction` call.

### Recommendation

1. Remove `hide: true` from `wallet_id` in the `chia_wallet.send_transaction` schema, or replace the raw integer with a resolved wallet name/type in the confirmation rows.
2. Derive the humanization unit (`mojo-to-xch` vs. `mojo-to-cat`) from the actual wallet type of the resolved `wallet_id`, not from a static schema field.
3. Display the wallet name and asset type prominently in the confirmation dialog so the user can verify which asset is being sent.

### Proof of Concept

1. Attacker deploys a dApp and convinces the user to pair it, granting `chia_sendTransaction` and `chia_getWallets`.
2. dApp calls `chia_getWallets`; response includes `{ wallet_id: 2, type: "CAT", name: "SomeToken" }` with a balance of 1 000 000 CAT tokens (= 1 000 000 000 000 CAT mojos if 1 CAT = 1 000 mojos).
3. dApp sends:
   ```json
   { "walletId": 2, "amount": "1000000000000", "fee": "0", "address": "<attacker>" }
   ```
4. Confirmation dialog shows: **"Send 1 XCH to \<attacker\>"** — `wallet_id` is absent from the visible rows.
5. User clicks **Send**.
6. Daemon executes `send_transaction` on wallet 2, transferring 1 000 000 CAT tokens to the attacker. [7](#0-6) [8](#0-7) [9](#0-8)
