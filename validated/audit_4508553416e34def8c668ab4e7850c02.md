### Title
Missing `threshold >= 2` Lower-Bound Validation in FROST Signing Allows Signing with Threshold = 1 — (File: `src/frost/mod.rs`)

---

### Summary

`assert_key_invariants` enforces that `threshold >= 2` during DKG/keygen, but the analogous signing-path validators — `assert_sign_inputs` and `presign` in `src/frost/mod.rs` — omit this lower-bound check entirely. A malicious coordinator or unprivileged library caller can supply `threshold = 1` to the FROST signing flow, bypassing the minimum-threshold invariant that the protocol specification mandates. This causes Lagrange interpolation during signing to use incorrect coefficients, producing an invalid, unusable signature that honest participants accept as a completed protocol output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces two threshold bounds:

```
// Step 1.1
if threshold > participants.len() { ... ThresholdTooLarge ... }
// Step 1.1
if threshold < 2 { ... ThresholdTooSmall { threshold, min: 2 } ... }
``` [1](#0-0) 

The protocol specification in `docs/dkg.md` explicitly states the invariant:

> 1.1 Each $P_i$ asserts that $1 < \mathsf{threshold} < N$. [2](#0-1) 

However, `assert_sign_inputs` — the public validation function for FROST signing — only checks the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

The lower-bound check (`threshold < 2 → ThresholdTooSmall`) is entirely absent. The same omission exists in the FROST `presign` entry-point:

```rust
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [4](#0-3) 

Neither function rejects `threshold = 1`. A caller supplying `threshold = 1` with `participants.len() >= 2` passes all validation gates and enters the signing protocol.

By contrast, the OT-based ECDSA presign path is indirectly protected: it requires the presign threshold to match the triple threshold (`args.threshold != args.triple0.1.threshold`), and triples are generated through `validate_triple_inputs`, which does enforce `threshold_value < 2 → ThresholdTooSmall`. No equivalent indirect protection exists for the FROST path. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

When `threshold = 1` is accepted by `assert_sign_inputs` or FROST `presign`, the signing protocol proceeds with Lagrange coefficients computed for a single-participant interpolation set. Because the key was generated with a polynomial of degree `threshold_keygen − 1 ≥ 1`, a single share does not correctly reconstruct the secret. The aggregated FROST signature `(R, z)` will fail the standard verification equation `z·G = R + challenge·X`, producing a cryptographically invalid, unusable output. Honest participants complete their portion of the protocol and receive what appears to be a finished signing transcript, but the output is silently broken.

This matches the allowed impact: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The entry point is any caller of the public API functions `assert_sign_inputs` or `presign` in `src/frost/mod.rs`. No special privilege is required beyond being a participant or coordinator who initiates a signing session. A malicious coordinator can trivially supply `threshold = 1` when constructing `PresignArguments` or calling `assert_sign_inputs`, causing all honest participants to waste their nonce material and produce an invalid signature without any protocol-level error being raised.

---

### Recommendation

Add the lower-bound check to both `assert_sign_inputs` and the FROST `presign` function, mirroring the check already present in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the upper-bound check in both locations:
- `src/frost/mod.rs` — `assert_sign_inputs` (after line 150)
- `src/frost/mod.rs` — `presign` (after line 77)

---

### Proof of Concept

1. A key is generated with `threshold = 2` and `participants = [P1, P2, P3]` via `assert_key_invariants` (which correctly enforces `threshold >= 2`).
2. A malicious coordinator constructs `PresignArguments { threshold: 1.into(), keygen_out, ... }` and calls `presign(participants, me, &args, rng)`.
3. The check `args.threshold.value() > participants.len()` evaluates `1 > 3 = false` — no error.
4. The check `threshold < 2` is absent — no error.
5. The presign protocol completes normally, producing `PresignOutput` with nonces and commitments.
6. The coordinator calls `assert_sign_inputs(&participants, 1usize, me, coordinator)`.
7. The check `threshold.value() > participants.len()` evaluates `1 > 3 = false` — no error.
8. The check `threshold < 2` is absent — no error. `assert_sign_inputs` returns `Ok(participants)`.
9. Signing proceeds with `threshold = 1`; Lagrange interpolation uses a single-participant coefficient of 1, producing `z = nonce_i + share_i · challenge`.
10. The aggregated signature fails verification: `z·G ≠ R + challenge·X` because `share_i·G ≠ X` (the public key encodes a degree-1 polynomial, not a single share).
11. Honest participants have consumed their nonces and accepted a completed-but-invalid signing output with no error indication from the library.
