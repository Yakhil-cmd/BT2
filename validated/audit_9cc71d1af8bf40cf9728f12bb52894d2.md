### Title
Unbounded `stakers` Vec Growth Enables Griefing via Unbounded Gas Consumption in `get_stakers()` - (File: src/staking/staking.cairo)

### Summary
The `stakers: Vec<ContractAddress>` storage variable in the Staking contract grows monotonically as new stakers join, but staker addresses are **never removed** from it even after they fully unstake. Any unprivileged user can repeatedly stake and unstake (recovering their tokens after the exit window) to inflate this Vec at the cost of only gas fees. The `get_stakers()` function iterates over the **entire** Vec on every call, making its gas cost proportional to the total number of ever-registered stakers, enabling a griefing attacker to make this function prohibitively expensive or non-functional.

### Finding Description
In `src/staking/staking.cairo`, the `stake()` function appends every new staker address to the `stakers` Vec:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
```

The storage declaration explicitly acknowledges this is permanent:

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) [2](#0-1) 

The `get_stakers()` function (part of `IStakingConsensus`) iterates over the **full range** of this Vec on every invocation:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    // ... compute staking power, public key, peer id
}
``` [3](#0-2) 

Each iteration performs a storage read (`staker_address_ptr.read()`) regardless of whether the staker is active. Inactive stakers are skipped via `continue`, but the storage read still occurs for every entry. An attacker who creates N staker accounts, stakes the minimum amount, waits for the exit window, and unstakes all of them permanently inflates the Vec by N entries. After `unstake_action()`, the attacker recovers all staked tokens; the only permanent cost is gas fees.

The `remove_staker()` internal function called from `unstake_action()` clears staker info and pool data but does **not** remove the address from the `stakers` Vec. [4](#0-3) 

### Impact Explanation
`get_stakers()` is the consensus mechanism's entry point for enumerating active stakers for a given epoch. As the `stakers` Vec grows unboundedly, the gas cost of `get_stakers()` grows linearly with the total number of ever-registered stakers. A sufficiently motivated attacker can inflate this Vec to the point where `get_stakers()` exceeds gas limits or becomes prohibitively expensive, degrading or blocking the consensus mechanism's ability to enumerate stakers. This constitutes **unbounded gas consumption / griefing with damage to the protocol** — matching the allowed Medium impact. [5](#0-4) 

### Likelihood Explanation
The attack requires only gas fees — staked tokens are fully recovered after the exit wait window (default 1 week, max 12 weeks). No privileged access is needed. Any unprivileged address can call `stake()` with the minimum stake amount. The minimum stake raises the capital cost temporarily (locked for the exit window), but the attacker recovers all tokens. The inflation is **permanent**: once the Vec is inflated, it cannot be deflated, and the gas cost of `get_stakers()` is permanently elevated. [6](#0-5) 

### Recommendation
- **Short term**: Remove staker addresses from the `stakers` Vec when they fully unstake (i.e., in `unstake_action()` / `remove_staker()`), or maintain a separate compact active-staker set that supports deletion.
- **Long term**: Cap the maximum number of registered stakers, or implement a lazy-deletion scheme (e.g., a bitmap or a separate active-index structure) so that `get_stakers()` only iterates over currently active stakers.

### Proof of Concept
1. Attacker creates N accounts (e.g., N = 100,000).
2. Each account calls `stake()` with the minimum stake amount, appending their address to `stakers`.
3. After the exit wait window, each account calls `unstake_intent()` then `unstake_action()`, recovering all staked tokens.
4. The `stakers` Vec now has N additional permanently-inactive entries.
5. Any subsequent call to `get_stakers()` iterates over N extra entries, each requiring a storage read.
6. With sufficient N, `get_stakers()` exceeds gas limits or becomes prohibitively expensive, preventing the consensus mechanism from enumerating active stakers.

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
