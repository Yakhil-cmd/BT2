### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Overly Broad Mempool Presence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is intended to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 **only when a corresponding `deploy_account` transaction is pending in the mempool**. However, the actual check — `account_tx_in_pool_or_recent_block(sender_address)` — returns `true` for **any** transaction associated with that address, not exclusively a `deploy_account`. An attacker who observes a `deploy_account` (or any nonce=0 invoke for a deployed account) in the mempool can submit a nonce=1 invoke with an **invalid signature** for the same address, bypass `__validate__` at the gateway, and have the forged transaction admitted to the mempool.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements the deploy-account + invoke UX shortcut: [1](#0-0) 

The skip condition fires when:
- The incoming invoke has `tx.nonce() == Nonce(Felt::ONE)`, and
- The on-chain account nonce is `Nonce(Felt::ZERO)`, and
- `account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`.

The third predicate is implemented as: [2](#0-1) 

This returns `true` if the address appears in **either** the mempool pool (`tx_pool.contains_account`) **or** the mempool's committed-block state (`state.contains_account`). The code comment acknowledges the ambiguity:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch ("future nonces that passed validations") is the loophole. For any account that is already deployed on-chain with nonce=0 (e.g., deployed via the `deploy` syscall rather than `deploy_account`), an attacker can:

1. Submit a valid nonce=0 invoke for that address — this passes `__validate__` because the contract exists, and is admitted to the pool.
2. Submit a nonce=1 invoke with an **invalid signature** for the same address.
3. The gateway evaluates: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true` → `skip_validate = true`.
4. `run_validate_entry_point` is called with `execution_flags.validate = false`, so `__validate__` is **never called** at the gateway. [3](#0-2) 

The forged nonce=1 invoke is then forwarded to the mempool via `validate_by_mempool` / `add_tx`.

A second, more impactful variant exploits the **intended** deploy_account + invoke UX flow:

1. A legitimate user submits `deploy_account` (nonce=0) + `invoke` (nonce=1) simultaneously.
2. The `deploy_account` enters the mempool pool, making `account_tx_in_pool_or_recent_block` return `true`.
3. An attacker submits a nonce=1 invoke for the **same address** with a higher tip and an invalid signature.
4. The skip condition fires (deploy_account is in pool → `account_tx_in_pool_or_recent_block=true`).
5. `__validate__` is skipped; the attacker's transaction is admitted.
6. The mempool's fee-escalation logic may **replace** the legitimate user's nonce=1 invoke with the attacker's forged one. [4](#0-3) 

### Impact Explanation

**Gateway/mempool admission accepts an invalid transaction** (invalid signature bypasses `__validate__`). The forged transaction is included in a block; during block execution the blockifier calls `__validate__` with `validate=true` (default flags), the signature check fails, and the transaction reverts. Consequences:

- The legitimate user's nonce=1 invoke can be displaced from the mempool by the attacker's higher-fee forged transaction, permanently losing the user's transaction.
- The forged transaction occupies a block slot and charges the account fees for the revert.
- Any account with nonce=0 and a pending transaction in the mempool is a target.

This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The deploy_account + invoke UX flow is an explicitly supported and tested production feature: [5](#0-4) 

Any user exercising this flow creates the precondition (deploy_account in pool, account_nonce=0) that an attacker can exploit. The attack requires only observing the mempool and submitting a crafted transaction — no privileged access is needed.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the address. The mempool should expose a dedicated `deploy_account_in_pool(address)` query, or the mempool's `account_tx_in_pool_or_recent_block` should be split into separate predicates so the gateway can distinguish between a pending deploy_account and any other pending transaction.

Alternatively, restrict the skip to only fire when the mempool confirms a `deploy_account` transaction (not just any transaction) is present for the sender address, eliminating the "future nonces that passed validations" loophole.

### Proof of Concept

**Variant A — deployed-via-syscall account (nonce=0 on-chain):**

1. Account `X` is deployed via `deploy` syscall; on-chain nonce = 0, class hash set.
2. Attacker submits `invoke(sender=X, nonce=0, sig=valid)` → passes `__validate__` → admitted to pool.
3. Attacker submits `invoke(sender=X, nonce=1, sig=INVALID)`.
4. Gateway: `skip_stateful_validations` → `nonce==1 && account_nonce==0 && pool_contains(X)==true` → `skip=true`.
5. `__validate__` not called; transaction admitted to mempool.
6. Block execution: `__validate__` called with `validate=true` → invalid signature → revert.

**Variant B — racing the deploy_account + invoke UX flow:**

1. User submits `deploy_account(X, nonce=0)` + `invoke(X, nonce=1, sig=valid, tip=T)`.
2. `deploy_account` enters pool; `account_tx_in_pool_or_recent_block(X) = true`.
3. Attacker submits `invoke(X, nonce=1, sig=INVALID, tip=T+1)`.
4. Gateway skips `__validate__`; attacker's transaction admitted.
5. Mempool fee-escalation replaces user's nonce=1 invoke with attacker's forged one. [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

/// Generates a deploy account transaction followed by an invoke transaction from the same account.
/// The first invoke_tx can be inserted to the first block right after the deploy_tx due to
/// the skip_validate feature. This feature allows the gateway to accept this transaction although
/// the account does not exist yet.
pub fn create_deploy_account_tx_and_invoke_tx(
    tx_generator: &mut MultiAccountTransactionGenerator,
    account_id: AccountId,
) -> Vec<RpcTransaction> {
    let undeployed_account_tx_generator = tx_generator.account_with_id_mut(account_id);
    assert!(!undeployed_account_tx_generator.is_deployed());
    let deploy_tx = undeployed_account_tx_generator.generate_deploy_account();

    vec![deploy_tx, invoke_tx]
}
