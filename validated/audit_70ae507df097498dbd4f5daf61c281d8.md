### Title
`MsgTurnBridge` Handler Is a No-Op — Bridge Circuit Breaker Permanently Broken - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` message handler in the Cronos `x/cronos` module is a stub that unconditionally returns `nil, nil` without performing any state change. The proto-defined `MsgTurnBridge` message, the `CanTurnBridge` permission bit, and the CLI command all exist and are reachable by any authorized sender, but calling the message has zero effect. The bridge can never be disabled, making the circuit-breaker mechanism permanently inoperative.

### Finding Description

`MsgTurnBridge` is a fully wired Cosmos SDK message: it is declared in `proto/cronos/tx.proto`, has a generated gRPC handler, a CLI command (`CmdTurnBridge`), a dedicated permission constant (`CanTurnBridge`), and is exercised in the integration test suite. The entire permission infrastructure for turning the bridge on/off is present.

However, the actual server-side implementation in `x/cronos/keeper/msg_server.go` is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

No permission check, no state write, no bridge-enabled flag is ever set. The handler silently succeeds (`code == 0`) while doing nothing.

Compare this with `UpdateTokenMapping`, which correctly checks `CanChangeTokenMapping` and calls `RegisterOrUpdateTokenMapping`: [2](#0-1) 

The `CanTurnBridge` permission constant is defined alongside `CanChangeTokenMapping` but is never consumed by any live code path: [3](#0-2) 

The proto service declares `TurnBridge` as the mechanism to disable or enable the gravity bridge: [4](#0-3) 

The integration test confirms the intended behavior: after `turn_bridge("false")` succeeds with `code == 0`, subsequent `send_to_evm_chain` calls are expected to fail: [5](#0-4) 

Because the handler is a no-op, the bridge-enabled state is never written, and `send_to_evm_chain` continues to succeed regardless of what the admin sends.

### Impact Explanation

**High — Bypass of Cronos bridge authorization/control checks.**

The bridge circuit breaker is the primary emergency mechanism for halting gravity bridge operations. With `TurnBridge` permanently broken:

- Any unprivileged user can continue to call `send_to_evm_chain` / `send_to_ethereum` on CRC20/CRC21 contracts and emit bridge events that the EVM hooks process, even after the admin has "disabled" the bridge.
- The admin's `CanTurnBridge` permission and any delegated address granted that permission via `UpdatePermissions` are rendered meaningless.
- In an active exploit scenario (e.g., a gravity bridge vulnerability), the admin has no on-chain mechanism to stop token outflows, leading to potential unauthorized bridge transfers of CRC20/CRC21/gravity-wrapped assets.

This matches the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation

The entry path is fully reachable by any unprivileged user: they simply submit bridge transactions. The admin's attempted mitigation (calling `MsgTurnBridge`) silently succeeds but has no effect. No special privileges, leaked keys, or cryptographic breaks are required. The vulnerability is permanent until a chain upgrade patches the handler.

### Recommendation

Implement `TurnBridge` to:
1. Verify the sender has `CanTurnBridge` permission (mirroring the `UpdateTokenMapping` pattern).
2. Write a bridge-enabled boolean flag to the KV store.
3. Have the EVM hook handlers (in `x/cronos/keeper/evmhandlers/`) read that flag and reject bridge events when the bridge is disabled. [1](#0-0) 

### Proof of Concept

1. Deploy a CRC21 token and bridge some amount to Cronos (as in `test_gravity_turn_bridge`).
2. As the admin, send `MsgTurnBridge{Sender: admin, Enable: false}`. The transaction returns `code == 0`.
3. Immediately call `send_to_evm_chain` on the CRC21 contract from an unprivileged address.
4. Observe the transaction succeeds (`status == 1`) — the bridge was never disabled.
5. Confirm by searching the entire codebase: `TurnBridge` in `msg_server.go` contains only `return nil, nil` with no state mutation. [1](#0-0) [6](#0-5)
