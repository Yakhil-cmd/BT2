### Title
Gateway Signature Validation Bypass via `skip_stateful_validations` for Existing Accounts with Nonce Zero — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for any invoke transaction with `nonce == 1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The pool membership check is not restricted to `DeployAccount` transactions — it returns `true` for any account that has *any* pending transaction in the pool. An attacker who observes a valid `nonce=0` invoke from an already-deployed account can immediately submit a second invoke with `nonce=1` and an **invalid signature**; the gateway admits it without calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` is designed to improve UX for the `deploy_account + invoke` flow: when a user sends both transactions simultaneously, the account does not yet exist on-chain, so `__validate__` would fail. The function therefore skips signature verification when:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (from on-chain state).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

Condition 4 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool, not only `DeployAccount`. `state.contains_account` returns `true` for any account in the `staged` or `committed` maps. [3](#0-2) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags = ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

`StatefulValidator::perform_validations` then returns `Ok(())` before calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [5](#0-4) 

**Attack path:**

1. Account `A` is already deployed on-chain with `nonce = 0` (it has never sent a transaction).
2. Account `A` submits a valid `Invoke` with `nonce = 0`. The gateway calls `__validate__`, verifies the signature, and admits the transaction to the mempool. Now `tx_pool.contains_account(A) == true`.
3. An attacker submits an `Invoke` with `nonce = 1` from account `A` with an **arbitrary/invalid signature**.
4. Gateway evaluation:
   - `account_nonce` from state = `0` ✓
   - `tx.nonce() == 1` ✓
   - `account_tx_in_pool_or_recent_block(A) == true` ✓ (due to the nonce-0 invoke, not a deploy_account)
   - `skip_validate = true` → `__validate__` is **not called**
5. The invalid transaction is admitted to the mempool without signature verification.

The `validate_by_mempool` call (`mempool.validate_tx`) only checks for duplicate hashes and stale nonces — it does not verify signatures. [6](#0-5) 

---

### Impact Explanation

The gateway accepts an `Invoke` transaction carrying an invalid (or entirely absent) signature. This violates the invariant that every admitted transaction has passed account-level authorization. The transaction will revert during blockifier execution (since `__validate__` is called with `validate=true` during block building), but the sequencer has already:

- Admitted an unauthorized transaction to the mempool.
- Committed resources to propagate, store, and attempt to execute it.

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

---

### Likelihood Explanation

The precondition — an already-deployed account with on-chain `nonce = 0` that has a pending `nonce = 0` invoke in the mempool — is common. Any account that has just been deployed and is sending its first transaction satisfies it. The attacker only needs to observe the mempool (public) and submit a crafted transaction before the `nonce = 0` tx is committed. No privileged access is required.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` heuristic with a check that is specific to `DeployAccount` transactions. The mempool should expose a method such as `has_deploy_account_in_pool(address)` that only returns `true` when a `DeployAccount` transaction for that address is pending. Alternatively, restrict the skip to cases where `tx_pool.contains_deploy_account(sender)` is true, so that existing accounts with `nonce = 0` are not incorrectly treated as undeployed accounts.

---

### Proof of Concept

```
1. Deploy account A on-chain (nonce becomes 0 in state).
2. Submit valid Invoke(sender=A, nonce=0, sig=VALID) → gateway calls __validate__, admits to mempool.
3. Submit Invoke(sender=A, nonce=1, sig=0xdeadbeef_INVALID) to gateway.
4. Gateway: account_nonce=0, tx.nonce=1, account_tx_in_pool_or_recent_block=true → skip_validate=true.
5. __validate__ is NOT called. Transaction is admitted to mempool.
6. Batcher executes nonce=0 tx (succeeds), then nonce=1 tx → __validate__ called → REVERT.
   Gateway has accepted an unauthorized transaction.
```

The root cause is in `skip_stateful_validations` at: [7](#0-6) 

which uses `account_tx_in_pool_or_recent_block` — a check that is true for *any* account with a pending transaction, not exclusively for accounts awaiting deployment.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
