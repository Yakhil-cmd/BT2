Audit Report

## Title
SNS Voting Rewards Permanently Lost When Proposals Settle With Zero Voter Participation - (File: rs/sns/governance/src/governance.rs, rs/sns/governance/src/types.rs)

## Summary
The SNS `distribute_rewards` function permanently discards the accumulated `rewards_purse_e8s` when proposals settle in a round where no neuron cast an eligible vote. The rollover mechanism in `rewards_rolled_over()` only triggers when `settled_proposals` is empty, but the function unconditionally populates `settled_proposals` even when `distributed_e8s_equivalent == 0`, causing the entire purse to be silently dropped with no recovery path.

## Finding Description
In `rs/sns/governance/src/governance.rs`, `distribute_rewards` computes `rewards_purse_e8s` starting from `e8s_equivalent_to_be_rolled_over()` plus new round rewards: [1](#0-0) 

When `total_reward_shares == dec!(0)`, the function logs a warning and skips maturity distribution, leaving `distributed_e8s_equivalent = 0`: [2](#0-1) 

Critically, the function does **not** return early. It continues to settle all `considered_proposals` and writes a new `RewardEvent` with `settled_proposals` populated: [3](#0-2) 

The rollover logic in `rs/sns/governance/src/types.rs` defines `rewards_rolled_over()` as: [4](#0-3) 

Because `settled_proposals` is non-empty, `rewards_rolled_over()` returns `false`, so `e8s_equivalent_to_be_rolled_over()` returns `0`: [5](#0-4) 

On the next call to `distribute_rewards`, the purse starts from zero — the entire prior accumulated balance is permanently lost. No minting occurs, no neuron receives maturity, and no recovery path exists.

## Impact Explanation
This is a **permanent, unrecoverable loss of SNS voting rewards** — a governance asset. The loss scales with `supply × reward_rate × round_duration`. For a newly launched SNS with a high initial reward rate (e.g., 10% annualized), a single missed round destroys a material fraction of the expected annual inflation budget. This matches the allowed High impact: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The condition is reachable by any unprivileged user:

1. Any account holding enough SNS tokens to pay the proposal rejection fee can submit a proposal.
2. If the SNS community fails to vote before the voting period expires, `total_reward_shares == 0` and the purse is destroyed.
3. This is especially likely for newly launched SNS instances where neurons have not yet configured following relationships, or during low-activity periods.

No privileged access is required. The attacker only needs to submit a proposal and wait for the voting period to expire with zero participation.

## Recommendation
Change `rewards_rolled_over()` in `rs/sns/governance/src/types.rs` to also return `true` when proposals settled but no rewards were distributed:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.settled_proposals.is_empty()
        || (self.distributed_e8s_equivalent == 0
            && self.total_available_e8s_equivalent.unwrap_or(0) > 0)
}
```

Alternatively, `e8s_equivalent_to_be_rolled_over` could unconditionally return `total_available_e8s_equivalent - distributed_e8s_equivalent`, which is the correct undistributed balance regardless of whether proposals were settled.

## Proof of Concept
1. Launch SNS with `VotingRewardsParameters`: `round_duration = 7 days`, `initial_rate = 10%`.
2. Submit a proposal; let it expire with zero votes cast (`ReadyToSettle`).
3. `run_periodic_tasks()` calls `distribute_rewards(supply)`:
   - `rewards_purse_e8s = supply * 0.10 / 52` (substantial amount)
   - `considered_proposals = [proposal_1]` (non-empty)
   - `total_reward_shares = 0` → skip maturity distribution
   - `distributed_e8s_equivalent = 0`
   - New `RewardEvent`: `settled_proposals=[proposal_1]`, `distributed=0`, `total_available=purse`
4. Next round: `distribute_rewards()` called again:
   - `rewards_rolled_over()` → `settled_proposals.is_empty()` = `false` → returns `false`
   - `e8s_equivalent_to_be_rolled_over()` → returns `0`
   - `rewards_purse_e8s` starts from `0` — entire prior purse permanently lost.

A deterministic integration test can confirm this by asserting that `total_available_e8s_equivalent` from round N appears as the rollover base in round N+1 when `distributed_e8s_equivalent == 0`.
