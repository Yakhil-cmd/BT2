Audit Report

## Title
Scheduler Finalization Overhead Scales with Total Canister Count, Enabling Round-Throttling DoS - (File: `rs/execution_environment/src/scheduler.rs`)

## Summary
At the end of every inner-loop iteration, the IC scheduler deducts `instruction_overhead_per_canister_for_finalization * state.num_canisters()` from the shared round instruction budget. This overhead is charged against the round-level budget (not any canister's cycles), and `state.num_canisters()` counts all canisters including idle ones. An unprivileged attacker who deploys a large number of idle canisters can inflate this per-iteration deduction to exhaust the round budget after one or two iterations, preventing multi-hop inter-canister calls from completing within a single round and reducing subnet throughput for all users.

## Finding Description
In `rs/execution_environment/src/scheduler.rs` at lines 559–563, after each inner-loop iteration:

```rust
round_limits.instructions -= as_round_instructions(
    self.config
        .instruction_overhead_per_canister_for_finalization
        * state.num_canisters() as u64,
);
```

`RoundInstructions` is a signed `i64` type (`AmountOf<RoundInstructionsTag, i64>`), so this subtraction can drive the budget negative. At the top of the next iteration, `instructions_reached()` checks `self.instructions <= RoundInstructions::from(0)` and breaks the loop if true.

The initial round budget is computed as:
```
round_instructions = max_instructions_per_round - max(max_instructions_per_slice, max_instructions_per_install_code_slice) + 1
                   = 4B - 2B + 1 ≈ 2,000,000,001
```

With `INSTRUCTION_OVERHEAD_PER_CANISTER_FOR_FINALIZATION = 12_000` and `DEFAULT_MAX_NUMBER_OF_CANISTERS = 120_000`, the per-iteration overhead at maximum canister count is `12_000 * 120_000 = 1,440,000,000` instructions — 72% of the effective round budget. After two inner-loop iterations, the cumulative overhead alone (`2 * 1.44B = 2.88B`) exceeds the ~2B budget, causing the loop to terminate before a third iteration can run. Any cross-canister call pattern requiring three iterations (call → response → callback) is starved.

There is no cap on `state.num_canisters()` in this calculation, no mechanism to charge idle canisters for the finalization work they impose, and no guard preventing the overhead from consuming the entire round budget. The constant was calibrated at 5,000 canisters; at 120,000 canisters it is 24× larger than the calibration point.

The existing test `finalization_overhead_per_canister_reduces_instruction_budget` in `rs/execution_environment/src/scheduler/tests/limits.rs` (lines 572–645) explicitly demonstrates and asserts this behavior: with sufficient per-canister overhead, the callback iteration does not run.

## Impact Explanation
This is a platform-level DoS matching the allowed impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** At near-maximum canister counts, every round's inner loop terminates after at most two iterations, preventing any three-hop inter-canister call from completing in a single round. This increases latency and reduces throughput for all users on the affected subnet indefinitely, for as long as the attacker's idle canisters remain deployed. The effect is persistent and requires no ongoing attacker action after initial deployment.

## Likelihood Explanation
Any unprivileged user can call `create_canister` on an application subnet. The only gate is `DEFAULT_MAX_NUMBER_OF_CANISTERS = 120_000` and the canister creation fee of 5,000,000 cycles per canister. At 120,000 canisters the one-time cost is 600 billion cycles (~0.6 XDR). After deployment, idle canisters pay only negligible storage fees. No privileged access, governance majority, or node compromise is required. The attack is repeatable across any application subnet and is persistent once deployed.

## Recommendation
1. **Cap the canister count used in the overhead calculation** to a safe maximum (e.g., the calibration point of 5,000) or use the count of canisters with pending messages rather than the total.
2. **Re-calibrate `INSTRUCTION_OVERHEAD_PER_CANISTER_FOR_FINALIZATION`** against the actual maximum canister count, or reduce `DEFAULT_MAX_NUMBER_OF_CANISTERS` so that `12_000 * N * max_iterations` remains well below the effective round budget.
3. **Charge idle canisters for finalization overhead** by deducting proportionally from each canister's cycles balance rather than from the shared round budget.

## Proof of Concept
The existing unit test at `rs/execution_environment/src/scheduler/tests/limits.rs:572–645` is a minimal reproducible proof. It constructs a scheduler with 3 canisters and a cross-canister call pattern that normally requires 3 inner-loop iterations. With `instruction_overhead_per_canister_for_finalization = SLICE_INSTRUCTIONS` (10) and 3 canisters, the overhead per iteration is 30, which exhausts the budget after 2 iterations and prevents the callback from running. The test asserts `expected_iterations == 2` and `inner_round_loop_consumed_max_instructions == 1`.

To reproduce at production scale: deploy 120,000 idle canisters on an application subnet. The per-iteration overhead becomes `12_000 * 120_000 = 1.44B` instructions against an effective round budget of ~2B, limiting the inner loop to at most 2 iterations per round and starving all 3-hop inter-canister calls. #[test]
fn finalization_overhead_per_canister_reduces_instruction_budget() {
    const SLICE: u64 = 10;
    const SLICE_INSTRUCTIONS: NumInstructions = NumInstructions::new(SLICE);

    /// Tests the scheduling and execution of two active canisters (canister0 and
    /// canister1) and one inactive canister. Ignoring overhead it should execute 3
    /// iterations:
    ///
    ///  * Iteration 1: canister0 ingress  (10 instructions)
    ///  * Iteration 2: canister1 call     (10 instructions, inducted from iter 1)
    ///  * Iteration 3: canister0 callback (10 instructions, inducted from iter 2)
    ///
    /// ```text
    /// budget = max_instructions_per_round - max_instructions_per_slice + 1
    ///        = 80 - 10 + 1 = 71
    /// ```
    fn test_sequence(finalization_overhead: NumInstructions, expected_iterations: u64) {
        let config = SchedulerConfig {
            scheduler_cores: 2,
            max_instructions_per_round: SLICE_INSTRUCTIONS * 8,
            max_instructions_per_message: SLICE_INSTRUCTIONS,
            max_instructions_per_slice: SLICE_INSTRUCTIONS,
            max_instructions_per_install_code_slice: SLICE_INSTRUCTIONS,
            instruction_overhead_per_execution: NumInstructions::from(0),
            instruction_overhead_per_canister: NumInstructions::from(0),
            instruction_overhead_per_canister_for_finalization: finalization_overhead,
            ..SchedulerConfig::application_subnet()
        };

        let mut test = SchedulerTestBuilder::new()
            .with_scheduler_config(config)
            .build();
        let canister0 = test.create_canister();
        let canister1 = test.create_canister();
        test.send_ingress(
            canister0,
            ingress(SLICE).call(other_side(canister1, SLICE), on_response(SLICE)),
        );
        let _canister2 = test.create_canister();

        test.execute_round(ExecutionRoundType::OrdinaryRound);

        let metrics = &test.scheduler().metrics;
        assert_eq!(
            metrics.inner_loop_processed_non_zero_inputs_count.get(),
            expected_iterations,
        );
        assert_eq!(
            metrics.inner_round_loop_consumed_max_instructions.get(),
            if expected_iterations == 3 { 0 } else { 1 }
        );
        assert_eq!(
            test.state()
                .metadata
                .subnet_metrics
                .update_transactions_total,
            expected_iterations
        );
    }

    // Baseline: without overhead, the inner loop runs 3 iterations:
    //  * Iteration 1: 71 → 61 (exec) → 61 (overhead)
    //  * Iteration 2: 61 → 51 (exec) → 51 (overhead)
    //  * Iteration 3: 51 → 41 (exec) → 41 (overhead)
    test_sequence(NumInstructions::from(0), 3);

    // With overhead = 10 and 3 canisters, the overhead per iteration is 30:
    //  * Iteration 1: 71 → 61 (exec) → 31 (overhead)
    //  * Iteration 2: 31 → 21 (exec) → -9 (overhead)
    //  * Iteration 3: budget ≤ 0 → break
    //
    // Only 2 iterations execute.
    test_sequence(SLICE_INSTRUCTIONS, 2);
```
