### Title
`addLiquidityWeighted` DoS via per-bin ceiling rounding accumulation exceeding `maxAmountToken0`/`maxAmountToken1` — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`addLiquidityWeighted` uses a probe-then-pay pattern. The probe returns `need0`/`need1` (computed with per-bin ceiling rounding), the adder scales weight shares down to fit within `max0`/`max1`, then executes the paying add. Because the pay step also applies per-bin ceiling rounding independently for each bin, the sum of per-bin ceiling amounts for the scaled shares can exceed `max0`/`max1`, causing the callback to revert with `MaxAmountExceeded` and permanently DoS-ing the weighted add for any user who sets caps at the probe-returned values.

---

### Finding Description

**Probe step** (`addLiquidityWeighted`, lines 106–112):

`LiquidityLib.addLiquidity` is called with `w.shares`. For each bin it computes:
```
amount0Scaled[i] = ceil(binBalance[i] * w.shares[i] / binTotalShares[i])   // line 109
```
then converts the sum to external units:
```
need0 = ceil(sum(amount0Scaled[i]) / token0ScaleMultiplier)                 // line 142
```
The callback reverts with `LiquidityProbe(need0, need1)`.

**Scale step** (`_scaleWeightsToShares`, lines 231–240):

```solidity
scaleWad0 = Math.mulDiv(max0, WAD, need0);          // rounds DOWN
scaleWad  = min(scaleWad0, scaleWad1);
scaledShares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);  // rounds DOWN
```

Both `mulDiv` calls use the default floor rounding, so `scaleWad ≤ max0·WAD/need0`.

**Pay step** (`_addLiquidity`, lines 194–206):

`LiquidityLib.addLiquidity` is called again with `scaledShares`. It again applies per-bin ceiling rounding:
```
amount0Scaled_pay[i] = ceil(binBalance[i] * scaledShares[i] / binTotalShares[i])
amount0Added = ceil(sum(amount0Scaled_pay[i]) / token0ScaleMultiplier)
```

The callback then checks:
```solidity
if (amount0Delta > max0 || amount1Delta > max1) revert MaxAmountExceeded(...);  // line 165
```

**The mismatch:** The probe computed `need0` as the ceiling of the *total* scaled sum for `w.shares`. The scale factor is derived from this total. But the pay step applies ceiling rounding *per bin* for `scaledShares`. With N bins, each bin can independently round up by 1 scaled unit, so the total can exceed `need0 * (scaledShares/w.shares)` by up to N units, pushing `amount0Added` above `max0`.

**Concrete minimal example (2 bins, `token0ScaleMultiplier = 1`):**

| Parameter | Value |
|---|---|
| `binBalance[0] = binBalance[1]` | `1` |
| `binTotalShares[0] = binTotalShares[1]` | `2` |
| `w.shares[0] = w.shares[1]` | `2` |
| `max0` | `1` |

- Probe: `need0_scaled = ceil(1·2/2) + ceil(1·2/2) = 2`, `need0 = 2`
- Scale: `scaleWad = floor(1·1e18/2) = 5e17`, `scaledShares = [1, 1]`
- Pay: `totalScaled = ceil(1·1/2) + ceil(1·1/2) = 1 + 1 = 2`, `amount0Added = 2`
- Callback: `2 > max0=1` → **`MaxAmountExceeded` revert**

The transaction reverts. Pool state is rolled back (the `try/catch` in `_addLiquidity` re-reverts). No liquidity is added.

---

### Impact Explanation

`addLiquidityWeighted` is the primary user-facing function for proportional multi-bin liquidity addition. Any user who sets `maxAmountToken0 = need0` (the value the probe itself returned as the required amount) will have their transaction revert with `MaxAmountExceeded`. The function is broken for all multi-bin deposits where per-bin fractional amounts exist — a common condition in any pool with non-trivial bin balances. This is a DoS of a core liquidity flow.

---

### Likelihood Explanation

The condition triggers whenever:
1. Two or more bins are included in the weight delta (standard usage)
2. At least one bin has `binBalance[i] * scaledShares[i]` not divisible by `binTotalShares[i]` (extremely common)
3. The user sets `maxAmountToken0` to the probe-returned `need0` (the natural, documented usage)

All three conditions hold in ordinary pool operation. The bug is deterministic and reproducible without any privileged access or special token behavior.

---

### Recommendation

In `_scaleWeightsToShares`, compute `scaleWad` using ceiling division so the scaled shares are conservative enough that per-bin ceiling rounding in the pay step cannot push the total above `max0`:

```solidity
// Use ceiling division so the scale factor is a lower bound
uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0, Math.Rounding.Floor);
```

Alternatively, after computing `scaledShares`, perform a second probe with `scaledShares` to get the exact `amount0Added_pay`, and reduce `scaledShares` by one unit per bin until `amount0Added_pay ≤ max0`. The simplest fix is to subtract 1 from `scaleWad` before applying it, ensuring the scaled shares always produce a pay amount strictly below `max0`.

  function _scaleWeightsToShares(LiquidityDelta calldata w, uint256 max0, uint256 max1, uint256 need0, uint256 need1)
    internal
    pure
    returns (LiquidityDelta memory out)
  {
    uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
    uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
    uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;

    uint256 n = w.binIdxs.length;
    out.binIdxs = new int256[](n);
    out.shares = new uint256[](n);
    for (uint256 i; i < n; i++) {
      out.binIdxs[i] = w.binIdxs[i];
      out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
      if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
    }
  }
