### Title
Missing `verify_not_in_validate_mode` Guard in `DeprecatedSyscallHintProcessor::replace_class` Allows State Mutation During `__validate__` — (`crates/blockifier/src/execution/deprecated_syscalls/hint_processor.rs`)

---

### Summary

`DeprecatedSyscallHintProcessor::replace_class` (lines 795–807) unconditionally calls `state.set_class_hash_at(...)` without first calling `verify_not_in_validate_mode`. Every other state-mutating deprecated syscall that is supposed to be forbidden during `__validate__` (e.g., `get_sequencer_address`) calls this guard. The omission means a Cairo0 account whose `__validate__` entry point invokes `replace_class` will permanently overwrite its own class hash in the blockifier's cached state, and that mutation is never rolled back because the revert machinery only fires on execution failure, not on successful validation.

---

### Finding Description

`verify_not_in_validate_mode` is defined at lines 286–295 and is called by `get_sequencer_address` at line 730. [1](#0-0) 

`replace_class` at lines 795–807 performs no such check: [2](#0-1) 

The function directly calls `state.get_compiled_class` (to verify the class is declared) and then `state.set_class_hash_at(syscall_handler.storage_address, request.class_hash)`, mutating the cached state unconditionally.

`validate_tx` in `AccountTransaction` creates a `new_validate` context and calls `validate_call.execute(state, &mut context, ...)` on the same mutable `state` object that will be used for `__execute__`. After `validate_tx` returns `Ok(Some(validate_call_info))` there is no rollback of state changes made during validation. [3](#0-2) 

The `EntryPointRevertInfo` struct tracks `original_class_hash` and `original_values` for revert support, but this mechanism is only triggered when an execution *fails* (panics/reverts). A successful `__validate__` that calls `replace_class` leaves the mutation in place. [4](#0-3) 

The OS-level `update_class_hash_in_execution_context` re-reads the class hash from state *after* validation completes, meaning `__execute__` will run under the attacker-chosen replacement class. [5](#0-4) 

---

### Impact Explanation

An unprivileged user deploys a Cairo0 account contract whose `__validate__` calls `replace_class` with a pre-declared class hash. If validation succeeds:

1. The account's class hash in the blockifier's cached state is permanently overwritten with the attacker-chosen value.
2. `__execute__` runs under the replacement class, not the originally deployed class.
3. The final committed state diff records the wrong class hash for the account address.

This is a **Critical** wrong-state impact: the class hash stored for the account address after the transaction is incorrect, and the `__execute__` phase dispatches to the wrong code.

---

### Likelihood Explanation

Any user who can deploy a Cairo0 account contract and declare a second class can trigger this. No operator or privileged role is required. The only prerequisite is that the replacement class is already declared on-chain (or declared in the same block before the invoke). Cairo0 account contracts are still supported by the sequencer.

---

### Recommendation

Add `syscall_handler.verify_not_in_validate_mode("replace_class")?;` as the first statement in `DeprecatedSyscallHintProcessor::replace_class`, mirroring the pattern used by `get_sequencer_address`:

```rust
fn replace_class(
    request: ReplaceClassRequest,
    _vm: &mut VirtualMachine,
    syscall_handler: &mut Self,
) -> DeprecatedSyscallResult<ReplaceClassResponse> {
    syscall_handler.verify_not_in_validate_mode("replace_class")?;  // ADD THIS
    syscall_handler.state.get_compiled_class(request.class_hash)?;
    syscall_handler
        .state
        .set_class_hash_at(syscall_handler.storage_address, request.class_hash)?;
    Ok(ReplaceClassResponse {})
}
```

Also audit `SyscallHandlerBase::replace_class` in `syscall_base.rs` (lines 369–377) for the same omission in the Cairo1 path. [6](#0-5) 

---

### Proof of Concept

```rust
// Pseudocode for a Rust integration test
// 1. Declare a replacement Cairo0 class R.
// 2. Deploy a Cairo0 account A whose __validate__ does:
//      replace_class(R)   // no validate-mode guard blocks this
//      return ()          // validation succeeds
// 3. Execute an invoke transaction from A.
// 4. After execute(), read state.get_class_hash_at(A.address).
// 5. Assert it equals the ORIGINAL class hash of A.
//    → The assertion FAILS: the class hash is R, not the original.
```

The test would confirm that `state.get_class_hash_at(account_address)` returns the replacement class hash `R` rather than the original class hash, demonstrating that `replace_class` mutated state during `__validate__` without being blocked.
