### Title
Gateway `validate_resource_bounds` Checks Against Stale `l2_gas_price` Instead of `next_l2_gas_price` from Block Header — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful resource-bounds check compares a transaction's `max_l2_gas_price` against `previous_block.gas_prices.strk_gas_prices.l2_gas_price` — the price that was *used inside* the previous block — rather than `next_l2_gas_price`, the EIP-1559-derived price that the batcher will actually apply to the next block. The code itself contains a `TODO` acknowledging this: `// TODO(Arni): getnext_l2_gas_price from the block header.` Because the two values diverge whenever usage is non-flat, the gateway admits transactions that the batcher will reject, and rejects transactions the batcher would accept.

---

### Finding Description

`validate_resource_bounds` in `StatefulTransactionValidator` reads the L2 gas price as:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;          // ← price used *inside* block N
``` [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs `BlockInfo` from `block_header.l2_gas_price.price_in_fri`, which is the price field of the block's own gas-price record — not `block_header.next_l2_gas_price`. [2](#0-1) 

The batcher, however, uses `next_l2_gas_price` (stored in the block header by the consensus orchestrator) as the L2 gas price for the block it is building. `next_l2_gas_price` is computed by the EIP-1559-style `calculate_next_l2_gas_price_for_fin` and written into the proposal init; it can differ substantially from the current block's `l2_gas_price` whenever utilisation is high or low. [3](#0-2) 

The gateway threshold check is:

```
threshold = (min_gas_price_percentage / 100) × previous_block.l2_gas_price
admit if tx.max_l2_gas_price ≥ threshold
```

The blockifier's `perform_pre_validation_stage` check (applied during actual execution) is:

```
reject if tx.max_l2_gas_price < block_context.gas_prices.l2_gas_price   (= next_l2_gas_price)
``` [4](#0-3) 

These two checks use different reference prices. The gateway check is a fractional floor on the *old* price; the blockifier check is an exact floor on the *new* price.

`run_validate_entry_point` also builds its `BlockContext` from the same stale `get_block_info()` result (only incrementing the block number, not updating gas prices), so the blockifier validation that runs inside the gateway also uses the wrong price and does not catch the discrepancy. [5](#0-4) 

---

### Impact Explanation

**Admission of invalid transactions (price rising):** During high-utilisation periods `next_l2_gas_price > previous_block.l2_gas_price`. A transaction with `max_l2_gas_price` satisfying the gateway threshold (≥ `min_gas_price_percentage%` of the old price) but below `next_l2_gas_price` passes gateway and enters the mempool. When the batcher attempts to execute it, `perform_pre_validation_stage` raises `InsufficientResourceBounds / MaxGasPriceTooLow` and the transaction is dropped — wasting sequencer resources and degrading user experience with a misleading "accepted" response.

**Rejection of valid transactions (price falling):** During low-utilisation periods `next_l2_gas_price < previous_block.l2_gas_price`. A transaction with `max_l2_gas_price` above `next_l2_gas_price` but below the gateway threshold (a fraction of the now-higher old price) is incorrectly rejected at the gateway even though the batcher would accept and execute it successfully.

Both directions match the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The EIP-1559 mechanism adjusts `next_l2_gas_price` every block based on gas used vs. target. Any sustained deviation from the target utilisation level (common during traffic spikes or lulls) causes `next_l2_gas_price` to diverge from `previous_block.l2_gas_price`. No special attacker capability is required; ordinary users submitting transactions during normal network activity trigger both failure modes. The `TODO` comment in the source confirms the developers are aware the wrong field is being read.

---

### Recommendation

Replace the stale `l2_gas_price` read with `next_l2_gas_price` from the block header. Concretely:

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a dedicated method) to also return `block_header.next_l2_gas_price`.
2. In `validate_resource_bounds`, use `next_l2_gas_price` as the reference price instead of `previous_block.gas_prices.strk_gas_prices.l2_gas_price`.
3. In `run_validate_entry_point`, populate `block_info.gas_prices` with the prices that will actually apply to the next block (i.e., derived from `next_l2_gas_price`) so the blockifier pre-validation inside the gateway is consistent with what the batcher will enforce.

---

### Proof of Concept

**Setup:** A network where blocks are consistently full, causing `next_l2_gas_price` to be 12.5 % higher than `previous_block.l2_gas_price` each block (EIP-1559 max increase). After a few blocks, `next_l2_gas_price = 1.125 × P_prev`.

**Config:** `min_gas_price_percentage = 90` (threshold = 0.9 × P_prev).

**Transaction:** `max_l2_gas_price = 0.95 × P_prev`.

**Gateway check:**
```
threshold = 0.90 × P_prev
tx.max_l2_gas_price (0.95 × P_prev) ≥ threshold → ADMITTED
```

**Batcher execution check:**
```
block.l2_gas_price = next_l2_gas_price = 1.125 × P_prev
tx.max_l2_gas_price (0.95 × P_prev) < 1.125 × P_prev → REJECTED with MaxGasPriceTooLow
```

The transaction is accepted by the gateway, enters the mempool, is selected by the batcher, and then fails pre-validation — the exact admission/execution mismatch described in the impact scope. [6](#0-5) [7](#0-6)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
