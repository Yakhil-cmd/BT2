### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Unclaimed Yield via Trace Inflation - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the delegation pool contract contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire `pool_member_epoch_balance` trace. A pool member (or their reward address) can inflate this trace by repeatedly calling `add_to_delegation_pool` or `exit_delegation_pool_intent` across many epochs without ever claiming rewards. Once the trace is large enough, every subsequent `claim_rewards` call will exceed Starknet's gas budget, permanently freezing the pool member's accumulated unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` implementation appends a **new** checkpoint whenever the key (epoch) differs from the last entry's key — i.e., one new entry per distinct `current_epoch + K` value: [2](#0-1) [3](#0-2) 

`set_member_balance` is called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`: [4](#0-3) [5](#0-4) 

The `entry_to_claim_from` cursor is only advanced when `claim_rewards` **successfully completes**: [6](#0-5) 

If a pool member never calls `claim_rewards`, `entry_to_claim_from` stays at `0` and the loop must traverse the entire accumulated trace on the next call.

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

Once the trace is large enough that a single `claim_rewards` transaction exceeds Starknet's `max_computation_budget`, the call reverts every time. Because `entry_to_claim_from` is only updated on success, there is no way to partially drain the trace. The pool member's accumulated rewards become permanently unclaimable.

The same unbounded loop is also exercised by the public view `pool_member_info_v1`, which calls `calculate_rewards` without updating state: [7](#0-6) 

This means even querying the member's reward balance will fail once the trace is large enough.

---

### Likelihood Explanation

**Medium.** The attack requires one trace entry per epoch. Starknet epochs are short (roughly one week in the current design), so a pool member who adds to their delegation every epoch and defers claiming for an extended period (e.g., one to two years) can accumulate hundreds of entries. The developers themselves flag this risk inline: *"This loop is unbounded but unlikely to exceed gas limits."* The qualifier "unlikely" is not a guarantee, and the cost per iteration (storage read + arithmetic) is non-trivial. A motivated attacker who controls the reward address can also call `add_to_delegation_pool` on behalf of the victim each epoch to inflate the trace without the victim's cooperation: [8](#0-7) 

---

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` and `exit_delegation_pool_intent` if the trace would exceed a safe bound (e.g., 200–500 entries).
2. **Alternatively, implement partial reward claiming.** Allow `claim_rewards` to accept a `max_entries` parameter and advance `entry_to_claim_from` incrementally across multiple transactions, so a large trace does not permanently lock rewards.
3. **Remove the reward-address permission to call `add_to_delegation_pool`.** This eliminates the third-party inflation vector.

---

### Proof of Concept

```
1. Pool member Alice enters the delegation pool with the minimum amount.
2. Each epoch, Alice (or her reward address) calls add_to_delegation_pool
   with 1 unit. Each call in a new epoch appends one entry to
   pool_member_epoch_balance (because get_epoch_plus_k() returns a new key).
3. Alice never calls claim_rewards, so entry_to_claim_from stays at 0.
4. After N epochs, pool_member_epoch_balance has N entries.
5. Alice calls claim_rewards. calculate_rewards loops over all N entries,
   each requiring a storage read (pool_member_trace.at(i)) plus arithmetic.
6. For sufficiently large N, the transaction exceeds max_computation_budget
   and reverts.
7. Because entry_to_claim_from was never updated, every future claim_rewards
   call also reverts. Alice's accumulated yield is permanently frozen.
``` [9](#0-8) [10](#0-9)

            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.

            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
