### Title
Missing Access Control on `update_rewards` Allows Any Caller to Consume Block Reward Slots Without Distributing Rewards - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in the staking contract is intended to be callable only by the Starkware sequencer (per the protocol spec), but the implementation contains no access control check. Any unprivileged caller can invoke it with `disable_rewards: true`, which updates the global `last_reward_block` storage variable without distributing any rewards. Because `last_reward_block` is a global one-per-block gate, this permanently burns the reward slot for that block, preventing the legitimate sequencer from ever distributing rewards for it.

### Finding Description
The spec at `docs/spec.md` lines 1644–1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `StakingRewardsManagerImpl::update_rewards` in `src/staking/staking.cairo` lines 1449–1507 contains no role check — only `self.general_prerequisites()` (a pause check):

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validation ...
    self.last_reward_block.write(current_block_number);   // ← slot consumed here
    if disable_rewards || self.is_pre_consensus() {
        return;                                            // ← no rewards distributed
    }
    // ... actual reward distribution ...
}
```

The `disable_rewards` parameter is fully caller-controlled. When set to `true`, the function:
1. Writes `current_block_number` into `last_reward_block` (line 1485).
2. Returns immediately without distributing any rewards (line 1487–1489).

Because `last_reward_block` is a **global** (not per-staker) variable, once it is written for a given block number, the `REWARDS_ALREADY_UPDATED` assertion at line 1454–1458 prevents any subsequent call to `update_rewards` for that same block — including the legitimate sequencer call.

### Impact Explanation
An attacker who calls `update_rewards(any_valid_active_staker, disable_rewards: true)` on every block permanently prevents all stakers from receiving consensus-era block rewards. The `last_reward_block` slot is consumed with zero reward output, and the sequencer's legitimate call for that block will always revert with `REWARDS_ALREADY_UPDATED`. This constitutes **permanent freezing of unclaimed yield** for all stakers and their delegators — matching the **High** impact tier.

### Likelihood Explanation
The entry point is fully public (no role, no fee, no special state required). The attacker only needs to know any valid, active staker address (trivially obtainable from on-chain events). The cost is one transaction per block. The attack is sustainable indefinitely and requires no privileged access, leaked keys, or external dependencies.

### Recommendation
Add the missing sequencer-only access control to `update_rewards`, consistent with the spec. For example, add a role check analogous to `self.roles.only_sequencer()` (or the equivalent role used by the protocol) as the first statement in the function body, before any state is read or written.

### Proof of Concept

1. Consensus rewards are activated (`consensus_rewards_first_epoch` is set and reached).
2. A valid staker `S` exists with non-zero balance.
3. Attacker (any EOA) calls `update_rewards(S, disable_rewards: true)` at block `B`.
   - `last_reward_block` is written to `B` (line 1485).
   - Function returns early; no rewards distributed (line 1487–1489).
4. The legitimate sequencer attempts `update_rewards(S, disable_rewards: false)` at block `B`.
   - Assertion `current_block_number > self.last_reward_block.read()` fails → `REWARDS_ALREADY_UPDATED` panic.
5. Staker `S` (and all delegators) receive zero rewards for block `B`.
6. Repeating steps 3–5 every block results in permanent, total suppression of all consensus rewards.
