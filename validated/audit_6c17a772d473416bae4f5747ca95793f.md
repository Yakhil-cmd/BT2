### Title
Insufficient L1→L2 Message Fee in `tick()` Causes Minted Reward Tokens to Be Permanently Frozen in StarkGate Bridge - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.sol`'s `tick()` function is callable by any unprivileged account with `msg.value = 0`. When called with zero ETH, the function irreversibly consumes L2→L1 mint-request messages and mints STRK tokens, then deposits them to the StarkGate bridge with a zero L1→L2 message fee. Because Starknet's sequencer requires a non-zero fee to process L1→L2 messages, the L2 `on_receive` callback on the reward supplier is never triggered, the minted tokens are permanently locked in the bridge, and stakers can never claim the corresponding rewards.

---

### Finding Description

`tick()` in `RewardSupplier.sol` performs a two-phase operation:

**Phase 1 – irreversible on-chain state changes (L1):**
1. Consumes up to `MAX_MESSAGES_TO_PROCESS_PER_TICK` (5) L2→L1 mint-request messages via `messagingContract().consumeMessageFromL2(...)`.
2. Mints `amountToMint` STRK tokens via `mintManager().mintRequest(token(), amountToMint)`.

**Phase 2 – L1→L2 message delivery (requires ETH fee):**
3. Deposits the minted tokens to StarkGate with fee `msg.value / 2`.
4. Sends a total-supply update to the L2 MintingCurve with fee `msg.value - msg.value / 2`. [1](#0-0) 

There is **no minimum fee validation** anywhere in `tick()`. If `msg.value = 0`, both fees are 0. The `IStarknetMessaging` interface documents that `sendMessageToL2` is payable and "the paid amount is the message fee." [2](#0-1) 

Starknet's sequencer requires a non-zero fee to process L1→L2 messages. A zero-fee message is queued in the messaging contract (stored as `msg_fee + 1 = 1` in `l1ToL2Messages`) but is never automatically executed by the sequencer. As a result:

- The L2 `on_receive` callback in `reward_supplier.cairo` is never invoked.
- `l1_pending_requested_amount` is never decremented.
- The reward supplier's token balance remains zero.
- `claim_rewards` reverts with `AMOUNT_TOO_HIGH` when the staking contract tries to distribute rewards. [3](#0-2) [4](#0-3) 

**No recovery path exists.** `cancelL1ToL2Message` in `IStarknetMessaging` can only be called by the original sender of the message. The bridge contract sent the deposit message, not `RewardSupplier.sol`, so the reward supplier cannot cancel or retry the deposit. [5](#0-4) 

---

### Impact Explanation

Each call to `tick()` with `msg.value = 0` consumes up to 5 L2→L1 mint-request messages and mints up to `5 × 1,300,000 STRK = 6,500,000 STRK` per invocation. [6](#0-5) 

Those tokens are minted and locked in the StarkGate bridge forever (no recovery mechanism). The L2 reward supplier never receives them, so `claim_rewards` will revert for all stakers and delegators whose rewards depend on that batch. This constitutes **permanent freezing of unclaimed yield** for all protocol participants.

---

### Likelihood Explanation

`tick()` is a public, permissionless, payable function — any EOA or contract can call it with `msg.value = 0` at any time. No special role, key, or privilege is required. The only precondition is that pending L2→L1 mint-request messages exist (a normal operating condition whenever rewards accrue). A griefing attacker pays only Ethereum gas (a few dollars) to freeze millions of STRK in rewards. [7](#0-6) 

---

### Recommendation

Add a minimum fee guard at the top of `tick()` before any state-changing operations:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        uint256 minFee = messagingContract().getMaxL1MsgFee(); // or a configurable minimum
        require(msg.value >= 2 * minFee, "INSUFFICIENT_MSG_FEE");
        // ... rest of logic
    }
}
```

Alternatively, enforce that `msg.value > 0` and that each sub-fee is non-zero before proceeding with the irreversible mint and consume steps.

---

### Proof of Concept

1. The L2 reward supplier calls `send_mint_request_to_l1_reward_supplier()`, queuing N L2→L1 messages. [8](#0-7) 

2. An unprivileged attacker calls `RewardSupplier.tick{value: 0}()` on L1.
3. `tick()` consumes the L2→L1 messages (irreversible), mints `N × 1,300,000 STRK`, and calls `bridge().depositWithMessage{value: 0}(...)`.
4. The StarkGate bridge queues an L1→L2 message with fee = 0. The Starknet sequencer never processes it.
5. The L2 `on_receive` is never called; `l1_pending_requested_amount` remains elevated. [9](#0-8) 

6. When any staker or delegator calls `claim_rewards`, the staking contract calls `reward_supplier.claim_rewards(amount)`, which attempts `checked_transfer` of tokens the reward supplier does not hold — the call reverts with `AMOUNT_TOO_HIGH`. [10](#0-9) 

7. The minted STRK tokens are permanently locked in the StarkGate bridge with no recovery mechanism available to the reward supplier.

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
