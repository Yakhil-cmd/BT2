### Title
Reverted DeclareV0/DeclareV1 Unconditionally Populates `deprecated_declared_classes` in Simulation State Diff — (`crates/apollo_rpc_execution/src/lib.rs`, `execution_utils.rs`)

### Summary

In `execute_transactions`, the `deprecated_declared_class_hash` is extracted from the transaction input **before** execution and passed unconditionally to `induced_state_diff` regardless of whether the blockifier execution reverted. `induced_state_diff` then unconditionally inserts it into `ThinStateDiff.deprecated_declared_classes`. The `simulate_transactions` path returns this corrupted diff to callers without checking for revert.

### Finding Description

In `execute_transactions` (`lib.rs:733-753`):

```rust
let deprecated_declared_class_hash = match &tx {
    ExecutableTransactionInput::DeclareV0(
        DeclareTransactionV0V1 { class_hash, .. }, _, _, _,
    ) => Some(*class_hash),
    ExecutableTransactionInput::DeclareV1(
        DeclareTransactionV0V1 { class_hash, .. }, _, _, _,
    ) => Some(*class_hash),
    _ => None,
};
let blockifier_tx = to_blockifier_tx(...);
let tx_execution_info_result =
    blockifier_tx.execute(&mut transactional_state, &block_context);  // may revert
let state_diff =
    induced_state_diff(&mut transactional_state, deprecated_declared_class_hash)?; // no revert check
transactional_state.commit();
let execution_info = tx_execution_info_result.map_err(...)?;  // revert checked AFTER
``` [1](#0-0) 

`induced_state_diff` then does:

```rust
deprecated_declared_classes: deprecated_declared_class_hash
    .map_or_else(Vec::new, |class_hash| vec![class_hash]),
``` [2](#0-1) 

The `deprecated_declared_class_hash` is sourced from the attacker-controlled `tx.class_hash` field, not from the actual blockifier state diff. There is no check on `tx_execution_info_result` before calling `induced_state_diff`. When the transaction reverts (e.g., `execution_info.revert_error` is `Some(...)`), the class hash is still injected into `deprecated_declared_classes`.

The `simulate_transactions` function passes `induced_state_diff` directly into `TransactionSimulationOutput` without inspecting `revert_error`: [3](#0-2) 

By contrast, `estimate_fee` correctly detects the revert and returns early before the corrupted diff is ever exposed: [4](#0-3) 

The existing test even has a `TODO` acknowledging the missing reverted-transaction coverage: [5](#0-4) 

### Impact Explanation

Any caller of `starknet_simulateTransactions` (or `starknet_traceTransaction` / `starknet_traceBlockTransactions`) who submits a DeclareV0 or DeclareV1 with insufficient account balance (or any other condition that causes a blockifier revert) will receive a `SimulatedTransaction` whose `state_diff.deprecated_declared_classes` contains the attacker-supplied class hash, even though the class was never actually declared. This is an authoritative-looking wrong value returned by the RPC simulation endpoint.

Impact category: **High — RPC simulation returns an authoritative-looking wrong value.**

### Likelihood Explanation

Triggering a revert on a DeclareV0/V1 requires no privileges. Any user can submit a `starknet_simulateTransactions` request with a DeclareV1 whose `max_fee` is below the actual fee or whose account has zero balance. This is a fully unprivileged, deterministic trigger.

### Recommendation

Before calling `induced_state_diff`, inspect `tx_execution_info_result` (or the resulting `execution_info.revert_error`). If the transaction reverted, pass `None` as `deprecated_declared_class_hash` so that `deprecated_declared_classes` remains empty:

```rust
let tx_execution_info_result =
    blockifier_tx.execute(&mut transactional_state, &block_context);
let is_reverted = tx_execution_info_result
    .as_ref()
    .ok()
    .and_then(|info| info.revert_error.as_ref())
    .is_some();
let effective_deprecated_hash = if is_reverted {
    None
} else {
    deprecated_declared_class_hash
};
let state_diff =
    induced_state_diff(&mut transactional_state, effective_deprecated_hash)?;
```

### Proof of Concept

1. Call `starknet_simulateTransactions` with a DeclareV1 transaction where the sender account has zero ETH balance and `charge_fee` is not skipped.
2. The blockifier executes the transaction, charges fee, finds insufficient balance, and sets `execution_info.revert_error = Some(...)`.
3. `execute_transactions` calls `induced_state_diff(..., Some(class_hash))` before checking the revert.
4. The returned `SimulatedTransaction.transaction_trace.state_diff.deprecated_declared_classes` contains the class hash.
5. Assert that `deprecated_declared_classes` is non-empty despite the revert — this assertion passes, confirming the bug.
