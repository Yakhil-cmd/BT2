Audit Report

## Title
Non-Disclosed Rule and Incident UUIDs Exposed to Anonymous Callers via Unguarded Query Methods - (`rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`, `rs/boundary_node/rate_limits/canister/getter.rs`)

## Summary
Both `ConfigConfidentialityFormatter` and `RuleConfidentialityFormatter` redact only `rule_raw` and `description` for non-disclosed rules, leaving `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` intact in responses to `RestrictedRead` callers. Because `inspect_message` is only invoked by the IC runtime for ingress update messages and never for non-replicated query calls, any anonymous principal can invoke `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` as non-replicated queries, bypassing the `inspect_message` gate and harvesting all rule/incident UUIDs and version metadata across all historical config versions.

## Finding Description

**Root cause — formatter does not redact identifiers.**

`ConfigConfidentialityFormatter::format` iterates over rules and, for non-disclosed ones (`disclosed_at.is_none()`), sets only `rule.description = None` and `rule.rule_raw = None`. The fields `id` (mapped to `rule_id` in the API type) and `incident_id` are never cleared: [1](#0-0) 

`RuleConfidentialityFormatter::format` has the identical gap: [2](#0-1) 

**`inspect_message` does not protect query calls.**

The hook gates `get_config` (as a replicated query / ingress update) for authorized callers and rejects all other ingress methods: [3](#0-2) 

However, all three read methods are annotated `#[query]`: [4](#0-3) 

The IC runtime never invokes `inspect_message` for non-replicated query calls. Any anonymous principal can therefore reach the getter logic directly.

**`ConfigGetter::get` always populates `id` and `incident_id` before formatting.**

`OutputRule` is constructed with both identifiers unconditionally set: [5](#0-4) 

After `self.formatter.format(config)` is applied, `rule_id` and `incident_id` survive in the response: [6](#0-5) 

**`IncidentGetter::get` similarly exposes `added_in_version` and `removed_in_version`.**

`OutputRuleMetadata` is built with all identifier and version fields set before the formatter is applied: [7](#0-6) 

**Existing test confirms the behavior.**

The unit test for `RestrictedRead` callers explicitly asserts that `rule_id` and `incident_id` are non-empty in the redacted response: [8](#0-7) 

## Impact Explanation
An anonymous caller can enumerate every `rule_id` and `incident_id` UUID across all historical config versions by issuing non-replicated query calls to `get_config` for versions 1 through `current_version`. With harvested `incident_id` values, the caller can further call `get_rules_by_incident_id` to obtain the complete set of rule UUIDs together with `added_in_version` and `removed_in_version` for every historical and active non-disclosed rule. This fully exposes the structural topology of the rate-limit rule set — the very information the disclosure mechanism is designed to protect — without ever being authorized. This constitutes a meaningful confidentiality breach of a boundary node infrastructure component, matching the **Medium** impact tier: a meaningful security impact against boundary/API infrastructure, though constrained to identifier and version metadata rather than full rule content.

## Likelihood Explanation
The attack requires no privileges, no keys, no social engineering, and no special tooling — only the ability to issue a standard IC query call. The `inspect_message` bypass is a structural property of the IC runtime, not a configuration issue. The formatter gap is confirmed by the existing unit test. Any anonymous principal on the IC network can reproduce this immediately and repeatedly.

## Recommendation
1. **Redact `rule_id` and `incident_id` for non-disclosed rules** in both `ConfigConfidentialityFormatter` and `RuleConfidentialityFormatter`, replacing them with opaque placeholders (e.g., empty string or a fixed sentinel UUID) so that identifiers of non-disclosed rules are never returned to `RestrictedRead` callers.
2. **Add an explicit in-method access-level check** at the top of `get_rule_by_id` and `get_rules_by_incident_id`, returning an authorization error for `RestrictedRead` callers, consistent with the intent expressed in the `inspect_message` "else" branch that rejects these methods for all ingress callers.
3. Consider adding an in-method guard to `get_config` that enforces the same `FullAccess`/`FullRead` restriction that `inspect_message` intends for the replicated-query path, so the restriction applies to non-replicated query calls as well.

## Proof of Concept
The existing unit test in `getter.rs` already constitutes a proof of concept. The following assertions pass against the current code, confirming the leak:

```rust
// RestrictedRead caller receives non-null rule_id and incident_id for a non-disclosed rule
let response = getter_unauthorized.get(&Some(1)).unwrap();
let rule = &response.config.rules[0]; // rule_id_1, disclosed_at: None
assert!(!rule.rule_id.is_empty());      // PASSES — UUID is exposed
assert!(!rule.incident_id.is_empty());  // PASSES — UUID is exposed
assert!(rule.rule_raw.is_none());       // PASSES — content is redacted
assert!(rule.description.is_none());   // PASSES — description is redacted
```

To exercise the non-replicated query path end-to-end, deploy the canister to a local replica or PocketIC instance and issue `dfx canister call <id> get_config '(null)'` as the anonymous principal. The response will contain `rule_id` and `incident_id` fields populated for non-disclosed rules.
