### Title
Unprivileged Caller Can Permanently Suppress Per-Block Consensus Rewards for All Stakers via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in `Staking.cairo` has no access control beyond a zero-address check. Any unprivileged caller can invoke it with `disable_rewards: true` for any valid active staker. This atomically marks the current block as "rewarded" by writing to the global `last_reward_block` storage slot — without distributing any rewards. Because `last_reward_block` is a single global per-block lock, this permanently prevents every staker from receiving rewards for that block.

---

### Finding Description

`update_rewards` is part of `IStakingRewardsManager` and is the sole mechanism for distributing per-block consensus rewards in V3. Its access control is:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
``` [1](#0-0) 

`general_prerequisites()` only asserts the contract is unpaused and the caller is non-zero:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [2](#0-1) 

There is no role check, no whitelist, and no restriction on who may call this function or what value they pass for `disable_rewards`.

Inside the function, the global `last_reward_block` is written **before** the early-return guard on `disable_rewards`:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [3](#0-2) 

`last_reward_block` is declared as a single global field:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [4](#0-3) 

Any subsequent call to `update_rewards` in the same block — including the legitimate one from the consensus mechanism — will fail:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [5](#0-4) 

The attacker only needs to supply any valid, active staker address (all are public on-chain). The staker existence and activity checks are:

```cairo
let staker_info = self.internal_staker_info(:staker_address);
let curr_epoch = self.get_current_epoch();
assert!(
    self.is_staker_active(:staker_address, epoch_id: curr_epoch),
    "{}",
    Error::INVALID_STAKER,
);
...
assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);
``` [6](#0-5) 

When `disable_rewards: true` is passed, the function returns after writing `last_reward_block` without ever calling `_update_rewards`, so `reward_supplier_dispatcher.update_unclaimed_rewards_from_staking_contract` is never called and no rewards are minted or credited to any staker or pool for that block.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Each block in consensus mode is entitled to a share of the yearly mint. When `update_rewards` is called with `disable_rewards: true` by an attacker, the block's reward slot is consumed and the rewards are permanently unrecoverable — the `unclaimed_rewards` counter in `RewardSupplier` is never incremented, no L1 mint request is triggered for that block's share, and no staker or pool receives yield. Repeated across many blocks, this causes unbounded, permanent loss of staker and delegator yield with no recovery path.

---

### Likelihood Explanation

**High.** The attack requires:
1. Any non-zero Starknet address (no tokens, no stake, no role needed).
2. Knowledge of one active staker address — trivially obtained from on-chain events (`NewStaker`).
3. A single transaction per block to front-run the legitimate `update_rewards` call.

There is no economic cost to the attacker beyond gas. The attack is repeatable every block indefinitely.

---

### Recommendation

Restrict `update_rewards` to a trusted caller. The most appropriate fix is to add a role check (e.g., `only_app_governor` or a dedicated `REWARDS_UPDATER` role), or to assert that the caller is the attestation contract or a whitelisted consensus component. The `disable_rewards` parameter should also be removed from the public interface or restricted to privileged callers only.

---

### Proof of Concept

1. Staker `S` is active with non-zero STRK balance. Their address is visible from the `NewStaker` event.
2. Attacker `A` (any EOA) monitors the chain. At the start of block `B`:
   - `A` calls `Staking.update_rewards(staker_address: S, disable_rewards: true)`.
   - `general_prerequisites()` passes (contract unpaused, `A` is non-zero).
   - Staker `S` is active and has non-zero balance — checks pass.
   - `last_reward_block` is written to `B`.
   - Function returns early — no rewards distributed.
3. The legitimate consensus call `update_rewards(staker_address: S, disable_rewards: false)` arrives in the same block `B`:
   - `assert!(current_block_number > self.last_reward_block.read())` → `B > B` → **FAILS** with `REWARDS_ALREADY_UPDATED`.
4. Block `B`'s rewards are permanently lost. No staker or pool receives yield for block `B`.
5. Repeat for every block.
