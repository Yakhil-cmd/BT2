### Title
WalletConnect Offer Approval Dialog Underestimates Royalty Cost for Multi-NFT Offers - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in the WalletConnect/dapp approval pipeline incorrectly divides the fungible asset amount by the number of NFTs before applying each NFT's royalty percentage. For an offer containing N NFTs, the displayed "Total Amount with Royalties" is underestimated by a factor of N, causing a user to approve a `take_offer` or `create_offer_for_ids` WalletConnect request believing they will spend significantly less than they actually will.

### Finding Description
In `parseCommandDisplay.ts`, the `formatAmountWithRoyalties` function computes the royalty-inclusive total shown in the WalletConnect confirmation dialog:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The `splitAmount` divides the fungible amount by the number of NFTs before applying each royalty. For N NFTs each with royalty percentage P on amount A, the code computes:

```
royaltyAmount = N × ((A / N) × P / 10_000) = A × P / 10_000
```

The correct calculation is:

```
royaltyAmount = N × (A × P / 10_000)
```

The code produces the same royalty total regardless of how many NFTs are in the offer. The `royaltyPercentages` array is collected from all NFTs on the opposite side of the trade via `royaltyPercentagesForSide`, and then the same (wrong) split is applied to every fungible asset line: [2](#0-1) 

This display path is used exclusively in the WalletConnect/dapp approval flow (`parseCommandDisplay` → `walletDeltaToDisplay` → `withRoyaltyTotals`), not in the interactive Offer Builder, which correctly delegates royalty calculation to the wallet backend via `useCalculateRoyaltiesForNFTsQuery`. [3](#0-2) 

The `Confirm.tsx` dialog renders the `amountWithRoyalties` field prominently as "Total Amount with Royalties" for both XCH and CAT lines: [4](#0-3) 

### Impact Explanation
A user approving a WalletConnect `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` request that involves multiple NFTs with royalties will see a displayed total cost that is N times lower than the actual royalty burden (where N is the number of NFTs). For example, an offer with 2 NFTs each carrying a 5% royalty on 1 XCH would display "Total Amount with Royalties: 1.05 XCH" but the actual on-chain cost is 1.10 XCH. The user approves based on the wrong amount, resulting in more XCH/CAT spent than they consented to.

This matches the allowed High impact: *"WalletConnect state that causes a user to approve the wrong amount."*

### Likelihood Explanation
Any dapp with an active WalletConnect session can send a `chia_wallet.take_offer` command. The dapp constructs the offer blob; the GUI calls `getOfferSummary` on it and then runs the local (broken) royalty estimate. No special privilege is required beyond having a WalletConnect pairing. The discrepancy grows linearly with the number of NFTs and their royalty percentages, making it straightforward to craft an offer where the displayed amount is materially lower than the actual cost.

### Recommendation
Replace the `splitAmount` division with the correct per-NFT royalty accumulation:

```typescript
// Correct: apply each NFT's royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Alternatively, delegate royalty calculation to the wallet backend (as the interactive Offer Builder already does) rather than computing it locally in the Electron main process. [5](#0-4) 

### Proof of Concept
The existing test suite already demonstrates the incorrect behavior. The test "shows the take-offer fungible total with multiple NFT creator royalties" uses two NFTs with royalties of 500 bp and 10 bp on 100,000,000 mojos (0.0001 XCH):

- **Displayed** (code output): `0.00010255 XCH` — royalty = 2,550,000 mojos
- **Correct** (full royalty per NFT): royalty = (100,000,000 × 500/10,000) + (100,000,000 × 10/10,000) = 5,000,000 + 100,000 = 5,100,000 mojos → total `0.0001051 XCH`

The test asserts the wrong value as expected, confirming the bug is baked in: [6](#0-5) 

A malicious dapp can exploit this by constructing a `take_offer` payload with many high-royalty NFTs. The approval dialog will show a fraction of the true cost, and the user will approve a transaction that drains significantly more XCH or CAT than displayed.

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
