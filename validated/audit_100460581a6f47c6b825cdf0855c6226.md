### Title
Unauthenticated Peer Can Inject Forged `FinalExecutionOutcomeView` into RPC Transaction Status Cache - (`chain/client/src/view_client_actor.rs`)

### Summary

Any connected peer can send a crafted `T2MessageBody::TxStatusResponse` for a transaction hash `H` that has a pending `tx_status_requests` entry. The handler stores the forged `FinalExecutionOutcomeView` into the LRU response cache with no source authentication or content validation. The next RPC call for `H` returns the forged outcome directly to the caller.

### Finding Description

**Step 1 — Network ingress, no authentication.**

At `network_state/mod.rs:968-970`, any connected peer's `T2MessageBody::TxStatusResponse` is forwarded unconditionally to `ViewClientActor`:

```rust
T2MessageBody::TxStatusResponse(tx_result) => {
    self.client.send_async(TxStatusResponse(tx_result.into())).await.ok();
    None
}
```

No peer identity check, no validator-origin check, no signature on the payload. [1](#0-0) 

**Step 2 — Handler extracts `tx_hash` from attacker-controlled data.**

`Handler<TxStatusResponse>::handle` at `view_client_actor.rs:1726` derives the cache key from the incoming payload itself:

```rust
let tx_hash = tx_result.transaction_outcome.id;  // attacker-controlled
``` [2](#0-1) 

**Step 3 — The only guard is a pending-request existence check.**

```rust
if request_manager.tx_status_requests.pop(&tx_hash).is_some() {
    request_manager.tx_status_response.put(tx_hash, *tx_result);
}
```

This checks only whether `tx_hash` has a pending entry in `tx_status_requests`. It does **not** verify that the response originated from the expected validator, nor does it validate any field of `FinalExecutionOutcomeView`. [3](#0-2) 

**Step 4 — `tx_status_requests` is populated with publicly known hashes.**

When a node does not track the shard for a transaction, `get_tx_status` inserts the tx hash into `tx_status_requests` and broadcasts a `TxStatusRequest` to the validator. Transaction hashes are derived from transaction content and are publicly observable on the network. [4](#0-3) 

**Step 5 — Forged outcome is returned to RPC caller without validation.**

`get_tx_status` checks `tx_status_response` first and returns its contents directly:

```rust
if let Some(res) = request_manager.tx_status_response.pop(&tx_hash) {
    request_manager.tx_status_requests.pop(&tx_hash);
    let status = self.get_tx_execution_status(&res)?;
    return Ok(TxStatusView {
        execution_outcome: Some(FinalExecutionOutcomeViewEnum::FinalExecutionOutcome(res)),
        status,
    });
}
```

No cross-check against chain state, no Merkle proof, no signature verification. [5](#0-4) 

This result propagates through `tx_status_fetch` → `tx_status_common` → the JSON-RPC `tx` / `EXPERIMENTAL_tx_status` endpoints. [6](#0-5) 

### Impact Explanation

The exact corrupted value is `FinalExecutionOutcomeView::status` (and the full `receipts_outcome` array) returned by the RPC `tx` / `EXPERIMENTAL_tx_status` endpoints. An attacker can forge:

- `FinalExecutionStatus::SuccessValue(...)` for a transaction that failed or has not yet executed.
- `FinalExecutionStatus::Failure(...)` for a transaction that succeeded.

Applications (exchanges, bridges, DeFi contracts) that rely on this RPC response to confirm settlement are directly affected. The corrupted value is a client-proof invariant violation: the node asserts a finalized outcome that is not anchored to any block or Merkle proof.

### Likelihood Explanation

- Connecting as a P2P peer is permissionless on mainnet.
- Transaction hashes are publicly observable (broadcast on the network when submitted).
- The race window between the node inserting into `tx_status_requests` and the legitimate validator response arriving is non-trivial (network round-trip to the validator).
- The attacker only needs to win the race once; the LRU cache entry is consumed on the first `get_tx_status` call, so the forged value is delivered to exactly the polling RPC client.

### Recommendation

Validate the source of `TxStatusResponse` before caching. Options:

1. Record the expected validator `AccountId` alongside the pending entry in `tx_status_requests`, and require the responding peer's announced account to match before accepting the response.
2. Require the `FinalExecutionOutcomeView` to be accompanied by a Merkle proof anchored to a known finalized block header, and verify it before caching.
3. As a minimal mitigation, ignore `TxStatusResponse` messages from peers whose announced account does not match the validator the request was sent to.

### Proof of Concept

```
1. Connect to the target node as a peer (permissionless).
2. Submit (or observe) a transaction T with hash H to the node,
   where the node does not track T's shard.
   → node inserts H into tx_status_requests and sends TxStatusRequest to validator.
3. Before the validator responds, send a T2MessageBody::TxStatusResponse
   with transaction_outcome.id = H and
   status = FinalExecutionStatus::SuccessValue(b"forged".to_vec()).
4. The handler pops H from tx_status_requests (guard passes) and stores
   the forged FinalExecutionOutcomeView in tx_status_response.
5. The next RPC tx_status_fetch poll for H returns the forged outcome
   with status = SuccessValue, even if T actually failed or is pending.
```
