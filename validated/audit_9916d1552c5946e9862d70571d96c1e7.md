Audit Report

## Title
`incident_id` of Undisclosed Security Incidents Exposed to Anonymous Callers via `inspect_message` Bypass and Missing Redaction — (`rs/boundary_node/rate_limits/canister/canister.rs`, `confidentiality_formatting.rs`, `types.rs`)

## Summary

The `inspect_message` hook in the rate-limit canister is annotated to gate `get_config`, but per the IC protocol specification it is only invoked for ingress messages; the `get_config` handler is declared `#[query]`, so any non-replicated query call reaches it unconditionally. Once inside the handler, `ConfigConfidentialityFormatter::format` redacts `rule_raw` and `description` for non-disclosed rules but leaves `incident_id` untouched, serialising the UUID of every undisclosed security incident into the wire response for any anonymous caller.

## Finding Description

**Root cause 1 — `inspect_message` does not fire for `#[query]` calls.**

`canister.rs` L31 names the constant `REPLICATED_QUERY_METHOD`, and L48–55 traps any non-privileged caller who sends `get_config` as an ingress message. However, `get_config` is declared `#[query]` at L110, making it a non-replicated query. The IC specification guarantees that `inspect_message` is never invoked for non-replicated queries; the guard is therefore structurally inert for the only call type an anonymous user would use.

**Root cause 2 — `incident_id` is not redacted by `ConfigConfidentialityFormatter`.**

`getter.rs` L130–146 checks the access level and, for non-`FullAccess`/`FullRead` callers, delegates to `self.formatter.format(config)`. `confidentiality_formatting.rs` L17–28 sets `is_redacted = true` and nulls `rule_raw` and `description` for rules where `disclosed_at.is_none()`, but never touches `incident_id`. The `OutputRule → api::OutputRule` conversion in `types.rs` L78–86 then unconditionally serialises `incident_id: value.incident_id.to_string()` into the response.

**Exploit flow:**

1. Anonymous caller sends `dfx canister call <id> get_config '(null)' --query`.
2. `inspect_message` is never invoked; `get_config` executes with `AccessLevel::RestrictedRead`.
3. `ConfigGetter::get` builds `OutputRule` structs with `incident_id` populated from storage.
4. `ConfigConfidentialityFormatter::format` nulls `rule_raw`/`description` but leaves `incident_id` intact.
5. The response contains the UUID of every undisclosed security incident alongside `active_since` and `version`.

## Impact Explanation

The rate-limit canister is an in-scope boundary-node infrastructure component whose explicit security model is that `incident_id` for non-disclosed rules must remain confidential until `disclose_rules` is called. The bypass exposes the UUID of every active, undisclosed security incident to any anonymous caller at zero cost, undermining the confidentiality guarantee of the boundary-node rate-limit system. This constitutes a concrete boundary/API infrastructure security impact: the exact moment a new incident-linked config is committed is observable (timing oracle), and the incident UUID is linkable to public incident trackers or on-chain activity to confirm that a security response is in progress. This maps to the High impact class: *Significant boundary/API infrastructure security impact with concrete user or protocol harm*.

## Likelihood Explanation

The attack requires no privileges, no cycles, no authentication, and no special tooling. A single unauthenticated HTTP query to any replica serving the canister suffices. The `inspect_message` bypass is a structural property of the IC protocol, not a misconfiguration, so it is reliably and repeatably exploitable on mainnet by any party.

## Recommendation

1. **Remove the `inspect_message` guard for `get_config`** — it provides no protection for query calls. Enforce access control exclusively inside the handler: if the resolved access level is `RestrictedRead`, return an explicit authorization error rather than a redacted response, or accept that the endpoint is public and ensure all confidential fields are fully redacted.
2. **Redact `incident_id` in `ConfigConfidentialityFormatter::format`** for rules where `disclosed_at.is_none()`, replacing it with an empty string or a stable opaque placeholder so the UUID is not observable before disclosure.
3. Apply the same fix to `RuleConfidentialityFormatter` and the `IncidentGetter` path, which have the same `incident_id` exposure for non-disclosed rules.

## Proof of Concept

```bash
# Zero privileges, zero cycles, any anonymous caller:
dfx canister call <rate_limit_canister_id> get_config '(null)' --query
# Returns ConfigResponse { version: N, active_since: T, config: { is_redacted: true,
#   rules: [{ rule_id: "...", incident_id: "<UUID of undisclosed incident>",
#             rule_raw: null, description: null }, ...] } }

# After version increments, retrieve the new config:
dfx canister call <rate_limit_canister_id> get_config '(opt N)' --query
# incident_id of newly added, non-disclosed rules is visible in the response.
```

The `--query` flag sends a non-replicated query call. `inspect_message` is never invoked. The response includes `incident_id` for every rule regardless of `disclosed_at` status, confirmed by the unit test at `getter.rs` L358–384 which asserts that `incident_id` is present in the redacted response for `AccessLevel::RestrictedRead`.
