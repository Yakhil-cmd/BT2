Audit Report

## Title
Unbounded `ingress_expiries` Loop in `construction_payloads` Enables Single-Request OOM DoS — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC-1 Rosetta node accepts attacker-controlled `ingress_start` and `ingress_end` values from request metadata and uses them to drive an unbounded `while` loop that accumulates `EnvelopeContent::Call` structs and signing payloads into heap-allocated `Vec`s. No upper bound is placed on `ingress_end` or on the number of loop iterations. A single unauthenticated HTTP request with `ingress_end = u64::MAX` causes the Rosetta node process to exhaust heap memory and crash.

## Finding Description
In `construction_payloads` (`rs/rosetta-api/icrc1/src/construction_api/services.rs`, lines 162–167), the loop is:

```rust
let mut ingress_expiries = vec![];
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
``` [1](#0-0) 

The only two guards before this loop are:

1. `ingress_start >= ingress_end` → error (line 148)
2. `ingress_end < now + ingress_interval` → error (line 154) [2](#0-1) 

Neither guard bounds `ingress_end` from above. The constants are:
- `MAX_INGRESS_TTL = 300s`, `PERMITTED_DRIFT = 60s` → `ingress_interval = 240s = 240,000,000,000 ns`
- `INGRESS_INTERVAL_OVERLAP = 120s = 120,000,000,000 ns` [3](#0-2) [4](#0-3) 

The actual step per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP = 120s = 120,000,000,000 ns`. With `ingress_end = u64::MAX ≈ 1.84×10¹⁹ ns` and `ingress_start ≈ now ≈ 1.7×10¹⁸ ns`, the loop runs approximately `(1.84e19 - 1.7e18) / 1.2e11 ≈ 139,000,000 iterations` — more than the claim's estimate because the claim used `ingress_interval` as the step rather than the actual smaller step.

Each iteration allocates an `EnvelopeContent::Call` struct (containing `canister_id`, `method_name`, `arg` bytes, `nonce`, `sender`, `ingress_expiry`) and a `SigningPayload` into growing `Vec`s: [5](#0-4) 

The accumulated `envelope_contents` is then CBOR+hex-serialized into the `unsigned_transaction` string: [6](#0-5) 

## Impact Explanation
The Rosetta node process exhausts heap memory and is OOM-killed or panics, taking down the entire ICRC-1 Rosetta API. This prevents transaction construction, signing, and submission for all users. This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"** — a single ~100-byte HTTP request suffices, with no volumetric component. The ICRC-1 Rosetta API is explicitly in scope as a financial integration component.

## Likelihood Explanation
The `/construction/payloads` endpoint is a public HTTP endpoint requiring no authentication. The exploit requires no privileges, no keys, no prior state, and no volumetric traffic. The crafted `ingress_end = u64::MAX` trivially passes both existing guards. The attack is repeatable: after the node restarts, a single request crashes it again.

## Recommendation
Add a hard cap on the number of ingress expiries before entering the loop, or validate the window size before the loop:

```rust
const MAX_INGRESS_WINDOW_NS: u64 = 24 * 3600 * 1_000_000_000; // 24 hours
if ingress_end.saturating_sub(ingress_start) > MAX_INGRESS_WINDOW_NS {
    return Err(Error::processing_construction_failed(
        &"ingress_end exceeds maximum allowed window of 24 hours"
    ));
}
```

Alternatively, break out of the loop once `ingress_expiries.len()` exceeds a reasonable maximum (e.g., 360 for a 24-hour window at 4-minute intervals).

## Proof of Concept
```bash
curl -X POST http://<rosetta-node>:8081/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [<valid_transfer_operation>],
    "public_keys": [<valid_public_key>],
    "metadata": {
      "ingress_start": 1700000000000000000,
      "ingress_end": 18446744073709551615
    }
  }'
```

A unit test can also reproduce this deterministically by calling `construction_payloads(ops, Some(metadata_with_max_ingress_end), ...)` with `ingress_end = u64::MAX` and asserting that it returns an error rather than running indefinitely.
