### Title
Unprivileged Caller Can Permanently Freeze Consensus Block Rewards via `update_rewards` with `disable_rewards: true` — (File: src/staking/staking.cairo)

---

### Summary

The `IStakingRewardsManager.update_rewards` function is publicly callable with no access control. It accepts a `disable_rewards` boolean that, when `true`, causes the function to consume the global `last_reward_block` slot for the current block without distributing any rewards. Because `last_reward_block` is a single global variable, only one `update_rewards` call can succeed per block. An attacker can front-run every legitimate call with `disable_rewards: true`, permanently destroying that block's consensus rewards for all stakers.

---

### Finding Description

`update_rewards` is defined in `IStakingRewardsManager` and implemented in `StakingRewardsManagerImpl`: [1](#0-0) 

The implementation has no caller restriction — `general_prerequisites()` only checks that the contract is unpaused and the caller is non-zero: [2](#0-1) 

Regardless of the `disable_rewards` value, the function always writes the current block number to the **global** `last_reward_block`: [3](#0-2) 

When `disable_rewards` is `true`, the function returns immediately after that write, distributing nothing: [4](#0-3) 

The guard at the top of the function enforces that only one call per block can succeed: [5](#0-4) 

Because `last_reward_block` is global (not per-staker), a single attacker call with `disable_rewards: true` for any valid staker address consumes the entire block's reward slot. Any subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`. The rewards for that block are permanently lost — they are never added to `unclaimed_rewards` in the `RewardSupplier`.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.** An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block prevents all stakers from ever accumulating consensus block rewards. The rewards are not deferred — they are silently discarded because `_update_rewards` is never reached and `update_unclaimed_rewards_from_staking_contract` is never called on the `RewardSupplier`. [6](#0-5) 

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

**High.** The attack requires no special role, no funds at risk, and no complex setup. Any address can call `update_rewards` with an arbitrary `staker_address` (any currently active staker) and `disable_rewards: true`. The only cost is gas per block. On Starknet, transaction ordering within a block is deterministic and observable, making front-running straightforward. The attacker has no profit motive but can cause continuous, irreversible damage to the protocol's reward distribution.

---

### Recommendation

1. **Restrict `disable_rewards: true` to a privileged caller** (e.g., a governance role or the staking contract itself). If `disable_rewards` is only needed during migration, gate it behind `only_app_governor()` or equivalent.
2. **Alternatively**, remove the `disable_rewards` parameter from the public interface entirely and handle the migration case through a separate privileged function.
3. **Consider making `last_reward_block` per-staker** if the intent is for each staker to independently claim per-block rewards, which would also eliminate the single-slot-per-block bottleneck.

---

### Proof of Concept

1. Consensus rewards are active (`current_epoch >= consensus_rewards_first_epoch`).
2. Staker Alice has an active stake and her agent submits `update_rewards(alice, disable_rewards: false)` to earn her block reward.
3. Attacker observes the pending transaction and front-runs it with `update_rewards(alice, disable_rewards: true)` (or any other valid staker address).
4. Attacker's transaction executes first: `last_reward_block` is set to the current block number; the early-return path is taken; no rewards are distributed.
5. Alice's transaction executes next and reverts: `current_block_number > last_reward_block` is now `false`.
6. The block's consensus rewards are permanently lost — they are never credited to `unclaimed_rewards` in `RewardSupplier`.
7. The attacker repeats this every block, permanently freezing all consensus block rewards across the entire protocol.
