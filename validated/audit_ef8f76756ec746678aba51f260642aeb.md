### Title
Recreated Account Permanently Deleted After CREATE2 Collision ŌĆö (`fvm/evm/emulator/state/stateDB.go`)

### Summary

`StateDB.Commit` contains a logic error: when an account is both self-destructed **and** recreated within the same transaction (the standard CREATE2-to-existing-address path), the `deleted = true` + `continue` guard unconditionally skips the `IsCreated` branch, causing the account to be permanently deleted from persistent state instead of being recreated with its carried-over balance.

### Finding Description

**Step 1 ŌĆö `DeltaView.CreateAccount` on an existing address**

When CREATE2 targets an address that already exists (has balance and/or code), the EVM calls `CreateAccount`. The function:

1. Detects the address exists and is not yet destructed
2. Calls `SelfDestruct(addr)` internally ŌåÆ sets `toBeDestructed[addr] = balance` and zeroes `balances[addr]`
3. Immediately overwrites `balances[addr] = balance` (carry-over)
4. Sets `recreated[addr] = {}` and `created[addr] = {}` [1](#0-0) 

After this call, the delta view holds **both** `toBeDestructed[addr]` (set) **and** `created[addr]` (set) simultaneously.

**Step 2 ŌĆö `StateDB.Commit` processes the address**

The commit loop iterates over dirty addresses and checks `HasSelfDestructed` first: [2](#0-1) 

Because `toBeDestructed[addr]` was set in Step 1, `HasSelfDestructed` returns `true`. The account is deleted from `baseView`, `deleted = true` is set, and `continue` is executed ŌĆö **unconditionally skipping** the `IsCreated` branch: [3](#0-2) 

The `recreated` map, which the code's own comment describes as "a flag used to track accounts that has been flagged for destruction but recreated later", is **never consulted** in `Commit`. It is only used in `DeltaView.GetState` to isolate old storage slots: [4](#0-3) 

**Result:** The account is deleted from persistent state and never recreated. The carried-over balance is lost. The new contract code is never stored.

### Impact Explanation

Any EVM transaction that uses CREATE2 to deploy to an address that already holds a non-zero balance (a common pattern: pre-fund a deterministic address, then deploy) will:

- Permanently delete the target account from state
- Lose the carried-over balance (funds destroyed, not transferred)
- Fail to persist the new contract deployment

This violates the EVM invariant that a recreated account must survive commit with its carried-over balance and new code intact.

### Likelihood Explanation

CREATE2 collisions with pre-funded addresses are a well-known and intentional EVM pattern (e.g., counterfactual instantiation in state channels, factory patterns). Any user following this pattern on Flow EVM will trigger the bug. An attacker can also deliberately pre-fund a CREATE2 address and then deploy to it to destroy the balance.

### Recommendation

In `StateDB.Commit`, after deleting a self-destructed account, check whether the address is also marked as recreated (`IsCreated`). If so, do not `continue` ŌĆö instead fall through to the `IsCreated` branch to recreate the account with its carried-over balance and new code:

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    ...
    deleted = true
}
// If the account was recreated in the same tx, do NOT skip it
if deleted && !db.IsCreated(addr) {
    continue
}
```

### Proof of Concept

1. Deploy a contract `A` that uses CREATE2 with a known salt and init code, targeting address `T`
2. Send ETH to `T` before deployment (making it exist with balance)
3. Call `A.deploy(salt, initCode)` ŌĆö this triggers `CreateAccount(T)` on an existing address
4. After the transaction commits, assert `stateDB.Exist(T)` ŌĆö it returns false (account deleted)
5. Assert `stateDB.GetBalance(T)` ŌĆö returns 0 (balance lost)

The state-test can be run on the Flow EVM emulator with no modifications to the node software. [5](#0-4) [6](#0-5)

func (db *StateDB) Commit(finalize bool) (hash.Hash, error) {
	// return error if any has been accumulated
	if db.cachedError != nil {
		return nil, wrapError(db.cachedError)
	}

	var err error

	// iterate views and collect dirty addresses and slots
	addresses := make(map[gethCommon.Address]struct{})
	slots := make(map[types.SlotAddress]struct{})
	for _, view := range db.views {
		for key := range view.DirtyAddresses() {
			addresses[key] = struct{}{}
		}
		for key := range view.DirtySlots() {
			slots[key] = struct{}{}
		}
	}

	// sort addresses
	sortedAddresses := make([]gethCommon.Address, 0, len(addresses))
	for addr := range addresses {
		sortedAddresses = append(sortedAddresses, addr)
	}

	sort.Slice(sortedAddresses,
		func(i, j int) bool {
			return bytes.Compare(sortedAddresses[i][:], sortedAddresses[j][:]) < 0
		})

	updateCommitter := NewUpdateCommitter()
	// update accounts
	for _, addr := range sortedAddresses {
		deleted := false
		// first we need to delete accounts
		if db.HasSelfDestructed(addr) {
			err = db.baseView.DeleteAccount(addr)
			if err != nil {
				return nil, wrapError(err)
			}
			err = updateCommitter.DeleteAccount(addr)
			if err != nil {
				return nil, wrapError(err)
			}
			deleted = true
		}
		if deleted {
			continue
		}

		bal := db.GetBalance(addr)
		nonce := db.GetNonce(addr)
		code := db.GetCode(addr)
		codeHash := db.GetCodeHash(addr)
		// create new accounts
		if db.IsCreated(addr) {
			err = db.baseView.CreateAccount(
				addr,
				bal,
				nonce,
				code,
				codeHash,
			)
			if err != nil {
				return nil, wrapError(err)
			}
			err = updateCommitter.CreateAccount(addr, bal, nonce, codeHash)
			if err != nil {
				return nil, wrapError(err)
			}
			continue
		}
