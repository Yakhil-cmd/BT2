### Title
`Debug` Derive on Secret-Bearing Types Enables Private Key Share and Presign Secret Disclosure — (`File: src/lib.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/triples/mod.rs`)

---

### Summary

Multiple production types that hold secret cryptographic material — private signing shares, presign nonce scalars, and Beaver triple scalars — unconditionally derive `Debug`. Any caller that formats these types (via logging, error handling, panic output, or tracing frameworks) will emit the raw secret scalar bytes into whatever output sink is configured. This is a direct analog to the symbol-cli file-permissions bug: in both cases, sensitive material is made readable to parties that should never see it, through a default/automatic mechanism the developer did not explicitly restrict.

---

### Finding Description

The following production types derive `Debug` while containing secret scalar fields:

**1. `KeygenOutput<C>` — long-term private key share** [1](#0-0) 

`private_share: SigningShare<C>` is the participant's long-term secret key share. `#[derive(Debug)]` causes `{:?}` formatting to recurse into `SigningShare`, which in `frost_core` exposes the underlying scalar.

**2. `PresignOutput` (OT-based ECDSA) — per-signature nonce secrets** [2](#0-1) 

Fields `k` and `sigma` are secret nonce shares. Exposure of `k` alone is sufficient to recover the private key from a completed signature.

**3. `RerandomizedPresignOutput` (OT-based ECDSA)** [3](#0-2) 

Same fields, rerandomized but still secret.

**4. `PresignOutput` (Robust ECDSA) — four secret nonce scalars** [4](#0-3) 

Fields `c`, `e`, `alpha`, `beta` are all secret shares of the presign nonces.

**5. `RerandomizedPresignOutput` (Robust ECDSA)** [5](#0-4) 

**6. `TripleShare` — Beaver triple secret scalars** [6](#0-5) 

The module's own documentation states explicitly:

> "It's important that the value of the underlying scalars in the triple is kept secret, otherwise the private key used to create a signature with that triple could be recovered." [7](#0-6) 

Yet `TripleShare` derives `Debug` with all three secret scalars `a`, `b`, `c` fully exposed.

---

### Impact Explanation

**Critical — Disclosure of private signing shares and presign secrets.**

- If `KeygenOutput` is debug-formatted (e.g., `tracing::debug!("{:?}", keygen_output)`, `unwrap_or_else(|e| panic!("{:?}", keygen_output))`), the `private_share` scalar is emitted verbatim. An attacker who can read application logs, crash dumps, or error telemetry recovers the full private key share and can reconstruct the group private key with `t` such shares.
- If `PresignOutput` or `TripleShare` is debug-formatted, the nonce/triple scalars are exposed. Per the module documentation, recovering a triple's scalars allows recovering the private key from any signature produced with that triple.

Both impacts fall squarely within: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

**High.** These are public library types returned directly to callers:

- `KeygenOutput` is the return type of `keygen()`, `reshare()`, and `refresh()` — the primary entry points of the library.
- `PresignOutput` and `TripleShare` are stored and passed between protocol phases by callers.
- Rust logging and tracing frameworks (`log`, `tracing`, `env_logger`) routinely format values with `{:?}`. Error-handling patterns such as `.expect("keygen failed: {:?}")` or `anyhow`/`thiserror` context chains that capture `Debug` representations are ubiquitous in Rust applications.
- The `Debug` derive is unconditional and requires no special configuration to trigger — any caller that follows standard Rust idioms for logging or error reporting will inadvertently leak secrets.

---

### Recommendation

**Short term:** Remove `#[derive(Debug)]` from all types containing secret scalar fields (`KeygenOutput`, `PresignOutput`, `RerandomizedPresignOutput`, `TripleShare`). Implement `Debug` manually for each, emitting only non-secret fields and a redacted placeholder for secret fields, e.g.:

```rust
impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        f.debug_struct("KeygenOutput")
            .field("private_share", &"[REDACTED]")
            .field("public_key", &self.public_key)
            .finish()
    }
}
```

**Long term:** Adopt a `secrecy`-crate pattern or a newtype wrapper (e.g., `Secret<T>`) that provides a `Debug` implementation that always redacts, and use it for all scalar fields in secret-bearing structs. Add a CI lint (e.g., a custom Clippy rule or `cargo-geiger`-style audit) that flags any `#[derive(Debug)]` on types whose fields include `Scalar`, `SigningShare`, or similar secret types.

---

### Proof of Concept

```rust
use threshold_signatures::{keygen, KeygenOutput};
use threshold_signatures::participants::Participant;

fn main() {
    let participants = vec![Participant::from(1u32), Participant::from(2u32)];
    let me = participants[0];
    let rng = rand::rngs::OsRng;

    // keygen returns a Protocol; run it to completion (simplified)
    let protocol = keygen::<frost_secp256k1::Secp256K1Sha256>(
        &participants, me, 2usize, rng
    ).unwrap();
    // ... run protocol to get keygen_output: KeygenOutput<_>

    // The following line emits the private_share scalar to stdout/logs:
    println!("{:?}", keygen_output);
    // Output includes: KeygenOutput { private_share: SigningShare(<raw scalar bytes>), ... }

    // Same for TripleShare:
    println!("{:?}", triple_share);
    // Output includes: TripleShare { a: <scalar>, b: <scalar>, c: <scalar> }
    // Per module docs, this allows recovering the private key from any signature.
}
```

Any logging call, `unwrap()` with a debug message, or panic handler that captures `KeygenOutput`, `PresignOutput`, or `TripleShare` via `{:?}` will expose the secret scalars to whoever can read that output.
