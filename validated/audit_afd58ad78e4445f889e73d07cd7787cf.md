### Title
`addLiquidity` Executes on Paused Pools, Allowing Users to Deposit Principal Into a Compromised Pool - (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.addLiquidity` lacks the `whenNotPaused` modifier that guards `swap`. When a pool is paused at any level (admin-pause = 1, protocol-pause = 2), users can still deposit token principal into the pool. If the pool was paused precisely because of a live vulnerability (e.g., a compromised price provider), those newly deposited funds are exposed to that vulnerability the moment the pool is unpaused.

---

### Finding Description

`MetricOmmPool` defines a two-level pause system via `pauseLevel` (0 = active, 1 = admin-paused, 2 = protocol-paused) and enforces it through the `whenNotPaused` modifier:

```solidity
// MetricOmmPool.sol line 174-177
modifier whenNotPaused() {
    _checkNotPaused();
    _;
}
// line 643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

`swap` correctly applies this guard:

```solidity
// MetricOmmPool.sol line 224
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
```

But `addLiquidity` does not:

```solidity
// MetricOmmPool.sol line 182-196
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    ...
    _beforeAddLiquidity(...);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(...);
    _afterAddLiquidity(...);
}
```

`LiquidityLib.addLiquidity` pulls tokens from the caller via a callback and credits them to the pool's internal bin accounting. There is no check that `pauseLevel == 0` before this transfer occurs.

`removeLiquidity` is also unguarded, which is arguably intentional (LPs must always be able to exit). The asymmetry — exit allowed, entry also allowed — is the defect. A paused pool should not accept new principal.

---

### Impact Explanation

The realistic loss path:

1. A pool is paused (level 1 or 2) because a security incident is detected — e.g., the mutable `priceProvider` is compromised or a bin-accounting bug is found.
2. Users who are unaware of the pause (e.g., routers, aggregators, or direct callers that do not read `pauseLevel` before depositing) call `addLiquidity`. The call succeeds and their tokens are transferred into the pool.
3. The pool admin or protocol owner unpauses the pool after believing the issue is resolved (or before it is fully resolved).
4. The attacker exploits the original vulnerability — e.g., using the compromised price provider to drain bins — and the newly deposited principal is among the drained funds.

The deposited tokens are real user principal. The loss is direct and bounded only by how much liquidity is added during the pause window.

---

### Likelihood Explanation

- Pause events are rare but are the exact scenario where the guard matters most.
- Routers and aggregators (e.g., `MetricOmmSimpleRouter`) call `addLiquidity` on behalf of users without checking `pauseLevel` first; they would succeed silently.
- The factory's `pausePool` / `protocolPausePool` paths are callable by semi-trusted roles (pool admin, protocol owner), so a pause can be triggered by a valid actor responding to an incident — not a malicious setup assumption.
- The trigger for the deposit is fully unprivileged: any address can call `addLiquidity`.

---

### Recommendation

Apply `whenNotPaused` to `addLiquidity`:

```solidity
function addLiquidity(...)
    external
    whenNotPaused                          // add this
    nonReentrant(PoolActions.ADD_LIQUIDITY)
    returns (uint256 amount0Added, uint256 amount1Added)
{
    ...
}
```

`removeLiquidity` should remain unguarded (or guarded only at level 2 / protocol-pause) so that LPs can always exit. The asymmetry should be intentional and documented: deposits blocked, withdrawals open.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Assume pool, factory, admin, token0, token1, and a valid LiquidityDelta are set up.

// Step 1: Admin pauses the pool (e.g., responding to a price provider incident)
factory.pausePool(pool);  // sets pauseLevel = 1

// Step 2: Verify swap is blocked
pool.swap(...);  // reverts with PoolPaused ✓

// Step 3: addLiquidity is NOT blocked — user deposits principal
uint256 amount0 = 1_000e18;
uint256 amount1 = 1_000e6;
// Caller approves tokens and calls:
(uint256 a0, uint256 a1) = pool.addLiquidity(
    user, salt, deltas, callbackData, extensionData
);
// Succeeds. Tokens are now inside the paused (potentially compromised) pool.

// Step 4: Admin unpauses
factory.unpausePool(pool);  // sets pauseLevel = 0

// Step 5: Attacker exploits the original vulnerability and drains the pool,
//         including the principal deposited in Step 3.
```

The call in Step 3 succeeds because `addLiquidity` carries only `nonReentrant(PoolActions.ADD_LIQUIDITY)` and no `whenNotPaused` check. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
