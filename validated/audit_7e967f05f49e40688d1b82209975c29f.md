### Title
Missing `me`-in-participants Check in `generate_triple` / `generate_triple_many` — (`File: src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

`validate_triple_inputs`, the shared validation helper called by both `generate_triple` and `generate_triple_many`, checks participant count, threshold bounds, and duplicate participants, but **omits the check that `me` is a member of the participant list**. Every other protocol entry-point in the library performs this check. Proceeding without it causes `do_generation_many` to execute with an invalid local identity, corrupting the Beaver-triple output that is a required input to OT-based ECDSA presigning.

---

### Finding Description

`validate_triple_inputs` (lines 681–708) performs four checks before returning a validated `ParticipantList`: [1](#0-0) 

1. `participants.len() < 2`
2. `threshold_value > participants.len()`
3. `threshold_value < 2`
4. Duplicate participants via `ParticipantList::new`

It does **not** check `!participants.contains(me)`.

Both public entry-points delegate to this helper without adding the missing check: [2](#0-1) [3](#0-2) 

Every other protocol constructor in the library performs the `me`-in-participants check. For example, `presign` (OT-based): [4](#0-3) 

`sign` (OT-based): [5](#0-4) 

`ckd`: [6](#0-5) 

`assert_key_invariants` (keygen/reshare): [7](#0-6) 

When `me ∉ participants`, `do_generation_many` proceeds into the protocol body. Two concrete consequences arise:

**a) `participants.others(me)` returns the full participant set.**
At line 284, the party iterates over `participants.others(me)` to send private polynomial evaluations. Because `me` is not in the list, `others(me)` returns *all* participants. The party therefore sends private shares to every participant, including parties that do not expect a message from this identity, and it never sends to itself (which it should, since it is supposed to be a real participant). [8](#0-7) 

**b) `ParticipantMap::put(me, …)` inserts an out-of-range identity.**
At line 172, the party inserts its own commitment into a `ParticipantMap` keyed on `participants`. If `me` is not a valid index in that map, the insertion is either silently dropped or panics, leaving the commitment map in an inconsistent state. [9](#0-8) 

Both effects corrupt the triple `(a, b, c)` such that `a·b ≠ c` or the public polynomial commitments are inconsistent across parties, producing an unusable or cryptographically invalid `TripleGenerationOutput`.

---

### Impact Explanation

Beaver triples are the sole randomness source for OT-based ECDSA presigning: [10](#0-9) 

A corrupted triple fed into `presign` produces a `PresignOutput` whose internal consistency invariants (`a·b = c mod q`) do not hold. When that presignature is consumed by `sign`, the resulting signature share reconstruction fails or produces an invalid ECDSA signature, permanently denying signing for all honest parties who relied on that triple. Because triples are generated in batches (`generate_triple_many`) and cached for future signing sessions, a single corrupted batch can block an entire signing pipeline.

**Mapped impact:** High — Corruption of presign outputs so honest parties accept inconsistent cryptographic outputs / permanent denial of signing.

---

### Likelihood Explanation

The entry-point is a public library function. Any caller that misconfigures `me` (e.g., passes the wrong local identity, or passes a participant list that does not include the local node) will silently bypass the missing guard and enter the corrupted execution path. No adversarial intent is required; a misconfigured honest participant is sufficient. The absence of the check is especially surprising because every sibling function in the same codebase performs it, making the omission easy to overlook during code review.

---

### Recommendation

Add the `me`-in-participants check inside `validate_triple_inputs`, immediately after `ParticipantList::new` succeeds, mirroring the pattern used in every other protocol constructor:

```rust
fn validate_triple_inputs(
    participants: &[Participant],
    me: Participant,          // <-- add me as a parameter
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    // ... existing checks ...
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ADD: ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    Ok((participants, threshold))
}
```

Propagate `me` through the call-sites in `generate_triple` and `generate_triple_many`.

### Proof of Concept

```
Caller invokes:
  generate_triple(
      participants = [P1, P2, P3],   // me (P4) is absent
      me = P4,
      threshold = 2,
      rng,
  )

validate_triple_inputs passes (len=3 ≥ 2, threshold=2 ≤ 3, no duplicates, no me-check).

do_generation_many executes:
  - participants.others(P4) == [P1, P2, P3]  (all, since P4 ∉ list)
  - Party sends private shares e(P1), e(P2), e(P3) to P1, P2, P3
    but never sends e(P4) to itself (which P4 should receive as a participant)
  - ParticipantMap::put(P4, commitment) inserts an out-of-range key

Result: TripleGenerationOutput is corrupted (a·b ≠ c).
Downstream presign using this triple produces an invalid PresignOutput.
Signing with that presignature fails for all honest parties → permanent DoS of signing.
```

fn validate_triple_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    let threshold = threshold.into();
    let threshold_value = threshold.value();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if threshold_value > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold_value,
            max: participants.len(),
        });
    }
    if threshold_value < 2 {
        return Err(InitializationError::ThresholdTooSmall {
            threshold: threshold_value,
            min: 2,
        });
    }
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;
    Ok((participants, threshold))
