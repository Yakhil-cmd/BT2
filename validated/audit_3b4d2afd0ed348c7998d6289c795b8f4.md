### Title
Missing Access Control on `update_rewards` Allows Anyone to Permanently Block Per-Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
`IStakingRewardsManager::update_rewards` is publicly callable by any address. The protocol specification explicitly restricts this function to "Only starkware sequencer," but the implementation contains no caller check. An unprivileged attacker can call `update_rewards` with `disable_rewards: true` for any valid active staker, consuming the single per-block reward slot and permanently preventing the sequencer from distributing rewards for that block.

### Finding Description
The `StakingRewardsManagerImpl::update_rewards` function is exposed as a public ABI entry point via `#[abi(embed_v0)]`:

```cairo
// src/staking/staking.cairo:1447-1507
#[abi(embed_v0)]
impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
    fn update_rewards(
        ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
    ) {
        self.general_prerequisites();                          // only checks pause
        let current_block_number = starknet::get_block_number();
        assert!(
            current_block_number > self.last_reward_block.read(),
            "{}",
            Error::REWARDS_ALREADY_UPDATED,
        );
        // ... staker validity checks ...

        // Update last block rewards.
        self.last_reward_block.write(current_block_number);   // ← slot consumed here

        if disable_rewards || self.is_pre_consensus() {
            return;                                           // ← exits without distributing
        }
        // ... reward distribution ...
    }
}
```

There is no `get_caller_address()` check anywhere in this function. The protocol specification at `docs/spec.md` line 1645 states the access control requirement is **"Only starkware sequencer"**, but this is never enforced in code.

The critical ordering is:
1. `last_reward_block` is written to `current_block_number` unconditionally (line 1485) before the `disable_rewards` branch.
2. The `REWARDS_ALREADY_UPDATED` guard (`current_block_number > self.last_reward_block.read()`) then prevents any subsequent call in the same block.

An attacker who calls `update_rewards(staker_address, disable_rewards: true)` before the sequencer does will:
- Pass all validity checks (staker exists, is active, has non-zero balance)
- Consume the block's reward slot by writing `last_reward_block = current_block_number`
- Return early without distributing any rewards
- Leave the sequencer unable to call `update_rewards` for that block (reverts with `REWARDS_ALREADY_UPDATED`)

### Impact Explanation
The rewards for the targeted block are permanently lost. `last_reward_block` is a monotonically-advancing pointer; once a block number is written, no function resets it backwards. The staker's `unclaimed_rewards_own` and the pool's balance are never incremented for that block. This constitutes **permanent freezing of unclaimed yield** for every staker and pool member whose rewards would have accrued in the attacked block.

The attacker can repeat this every block, continuously suppressing all reward accrual across the entire protocol.

### Likelihood Explanation
The entry point is fully public — no token approval, no privileged role, no special state is required. The attacker only needs to know any valid active staker address, which is observable on-chain from `stake` events. The attack is cheap (a single transaction per block) and can be automated. There is no economic cost to the attacker and no recovery path for the protocol.

### Recommendation
Add a caller check at the top of `update_rewards` to restrict it to the authorized sequencer address, consistent with the specification:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ...
}
```

Alternatively, if the sequencer address is not stored on-chain, restrict the caller to the attestation contract (which is already trusted for `update_rewards_from_attestation_contract`) or introduce a dedicated `REWARDS_MANAGER` role.

### Proof of Concept
1. Staker `S` is active and has non-zero balance.
2. At block `B`, before the sequencer submits its transaction, attacker calls:
   ```
   IStakingRewardsManager(staking_contract).update_rewards(S, disable_rewards=true)
   ```
3. The call passes all checks, writes `last_reward_block = B`, and returns without distributing rewards.
4. The sequencer's subsequent call for block `B` reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker `S` and all pool members receive zero rewards for block `B`.
6. Attacker repeats every block, permanently suppressing all reward accrual.
