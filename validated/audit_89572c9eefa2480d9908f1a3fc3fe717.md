### Title
Presign Protocol Hangs Indefinitely When Any Participant Withholds Round 2 Contribution — (File: `src/ecdsa/robust_ecdsa/presign.rs`)

---

### Summary

The Robust ECDSA presign protocol unconditionally waits for **all** N = 2·`max_malicious`+1 participants to deliver their Round 2 broadcast before it can proceed. A single malicious participant within the declared tolerance can permanently block presigning—and therefore all subsequent signing—for every honest party by simply withholding its `(big_r_me, w_me)` message. No capital, no cryptographic capability, and no external resource is required.

---

### Finding Description

After each participant computes and broadcasts its share in Round 2, `do_presign` collects responses with an unbounded loop:

```rust
// src/ecdsa/robust_ecdsa/presign.rs  lines 162–168
while !signingshares_map.full() {
    let (from, (big_r_p, w_p)): (_, (_, SigningShare<C>)) = chan.recv(wait_round_2).await?;
    signingshares_map.put(from, SerializableScalar(w_p.to_scalar()));
    verifyingshares_map.put(from, big_r_p);
}
```

`signingshares_map.full()` returns `true` only when **every** participant has contributed an entry. There is no timeout, no partial-set fallback, and no mechanism to proceed with a strict subset. The same unconditional-wait pattern is present in `recv_from_others` (helpers.rs lines 19–24), which is reused in DKG (`do_keyshare`) and signing (`do_sign_coordinator`).

The initialization guard enforces `participants.len() == 2*max_malicious+1` exactly:

```rust
// src/ecdsa/robust_ecdsa/presign.rs  lines 75–79
if participants.len() != robust_ecdsa_threshold {
    return Err(InitializationError::BadParameters(
        "the number of participants during presigning must be exactly 2*max_malicious+1 ...".to_string(),
    ));
}
```

This leaves zero slack: every single participant is required. A participant who sends Round 1 polynomial evaluations (to appear honest) but then silently drops its Round 2 broadcast causes the `while` loop to block forever on every honest node.

The polynomial interpolation of `w` at line 242 requires exactly `2*threshold+1` points, which equals N, so the dependency on all participants is also algorithmic, not merely an implementation oversight:

```rust
// src/ecdsa/robust_ecdsa/presign.rs  lines 236–242
let (w_2tp1_identifiers, _) = identifiers
    .split_at_checked(2 * threshold + 1) ...;
let (w_2tp1_verifying_shares, _) = signingshares
    .split_at_checked(2 * threshold + 1) ...;
let w = Polynomial::eval_interpolation(w_2tp1_identifiers, w_2tp1_verifying_shares, None)?;
```

The same liveness gap exists in `do_sign_coordinator` (sign.rs lines 138–143), where the coordinator waits for all N−1 signature shares via `recv_from_others` before it can assemble the final signature.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

Once presigning is blocked, no valid `PresignOutput` is produced. Because presignatures are single-use and consumed per signature, honest parties are permanently unable to produce signatures for the blocked session. Restarting requires out-of-band coordination to select a new participant set, which the same attacker can block again at zero additional cost, making the denial effectively permanent under adversarial conditions.

---

### Likelihood Explanation

Any participant within the declared `max_malicious` tolerance can execute this attack at zero cost: they simply do not transmit their Round 2 message. The attacker can selectively target specific signing sessions (e.g., high-value or time-sensitive transactions) while participating honestly in others, making detection difficult. Unlike the Morpho analog—where the attacker must lock up significant capital to borrow all pool liquidity—here the only requirement is membership in the participant set.

---

### Recommendation

1. Introduce a per-round deadline after which the protocol proceeds with the shares received so far, provided at least `2*threshold+1` shares are available for `w` interpolation and `threshold+1` for exponent interpolation.
2. Run presign with `N > 2*max_malicious+1` participants so that up to `max_malicious` dropouts can be tolerated by selecting any `2*threshold+1`-subset of respondents.
3. Record which participant failed to contribute so the caller can exclude them from future sessions and attribute the abort.

---

### Proof of Concept

1. Instantiate a presign session with `N = 5` participants and `max_malicious = 2`.
2. Participant P1 honestly sends its Round 1 polynomial evaluations to all others.
3. P1 then silently drops its Round 2 `(big_r_me, w_me)` broadcast—no message is sent.
4. All four honest participants reach `while !signingshares_map.full()` (presign.rs line 162) and block indefinitely; `signingshares_map` never reaches `full()` because P1's slot remains `None`.
5. No `PresignOutput` is produced; signing is permanently blocked for this session with no error surfaced to honest parties.
