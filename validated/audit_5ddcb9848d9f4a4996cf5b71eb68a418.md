### Title
Staker-Controlled `stake` Input in Target Block Hash Enables Attestation Window Manipulation — (File: `src/attestation/attestation.cairo`)

---

### Summary

The `_calculate_target_attestation_block` function uses the staker's **current, mutable `stake` amount** as an input to the Poseidon hash that determines the target attestation block. Because `stake` is directly controllable by the staker via `increase_stake`, a staker can manipulate their stake to shift their assigned target block to any convenient position within the epoch, bypassing the pseudo-random window assignment and claiming rewards even after missing their assigned window.

---

### Finding Description

In `_calculate_target_attestation_block`, the target block is computed as:

```cairo
let hash = PoseidonTrait::new()
    .update(staking_attestation_info.stake().into())   // ← mutable, staker-controlled
    .update(staking_attestation_info.epoch_id().into())
    .update(staking_attestation_info.staker_address().into())
    .finalize();
let block_offset: u256 = hash
    .into() % (staking_attestation_info.epoch_len() - attestation_window.into())
    .into();
let target_attestation_block = staking_attestation_info.current_epoch_starting_block()
    + block_offset.try_into().unwrap();
``` [1](#0-0) 

The `stake` value is fetched live inside `get_attestation_info_by_operational_address` using `get_staker_total_strk_balance_at_epoch` at the **current epoch**:

```cairo
let stake = self
    .get_staker_total_strk_balance_at_epoch(
        :staker_address, :staker_pool_info, :epoch_id,
    )
    .to_strk_native_amount();
``` [2](#0-1) 

Because `increase_stake` updates the staker's own balance trace for the current epoch, calling it changes the `stake` value that feeds into the hash, which in turn changes the computed `target_attestation_block`. The `AttestationInfo` struct confirms `stake` is the live current-epoch balance, not a snapshot fixed at epoch start: [3](#0-2) 

**Attack path:**

1. Staker S is assigned target block T for epoch E. Their attestation window `[T + 11, T + attestation_window]` has already passed (i.e., `current_block > T + attestation_window`).
2. S computes off-chain: for each candidate `delta`, compute `Poseidon(stake + delta, epoch_id, staker_address)` and derive the resulting `target_block'`.
3. S finds a `delta` such that `target_block' + MIN_ATTESTATION_WINDOW ≤ current_block ≤ target_block' + attestation_window`.
4. S calls `increase_stake(delta)` — a single transaction transferring `delta` STRK.
5. S calls `attest(block_hash_of_target_block')` and successfully triggers `update_rewards_from_attestation_contract`. [4](#0-3) 

The off-chain search is computationally trivial (Poseidon hash evaluation). The expected number of candidates to test before finding a valid window is approximately `epoch_len / (attestation_window - MIN_ATTESTATION_WINDOW)`, which is on the order of tens to hundreds of iterations for typical parameters.

---

### Impact Explanation

The protocol explicitly intends that rewards are only distributed to stakers who attest within their **pseudo-randomly assigned window**, verifying liveness during that specific period. A staker who was offline during their assigned window should not receive epoch rewards. By manipulating `stake`, a staker can retroactively obtain a valid window at any point during the epoch, claiming rewards they are not entitled to. This is **theft of unclaimed yield** — the staker extracts rewards the protocol intended to withhold.

---

### Likelihood Explanation

**Medium.** The attack requires the staker to hold extra STRK tokens for the `increase_stake` call. However: the off-chain computation is free and instant; only one on-chain transaction is needed; and the extra stake is fully recoverable via `unstake_intent` / `unstake_action` after the epoch. Any staker with a modest token reserve can execute this every epoch with negligible overhead.

---

### Recommendation

Remove `stake` from the hash inputs. The target block should be determined solely by values that are **immutable within an epoch** and not staker-controlled:

```cairo
let hash = PoseidonTrait::new()
    .update(staking_attestation_info.epoch_id().into())
    .update(staking_attestation_info.staker_address().into())
    .finalize();
```

`staker_address` and `epoch_id` are both fixed for the duration of an epoch and cannot be manipulated by the staker after registration, preserving the pseudo-random assignment guarantee.

---

### Proof of Concept

```
Epoch E, epoch_len = 1000, attestation_window = 20, MIN_ATTESTATION_WINDOW = 11
Staker S: address = 0xABC, current stake = 1_000_000 STRK
Current block = epoch_start + 900  (late in epoch)

Step 1: Compute target block for current stake:
  hash = Poseidon(1_000_000e18, E, 0xABC)
  block_offset = hash % (1000 - 20) = 50
  target_block = epoch_start + 50
  Window = [epoch_start+61, epoch_start+70]  ← already passed at block 900

Step 2: Off-chain search for delta:
  Try delta = 1 wei: block_offset' = 889 → window [900, 909] ← VALID at block 900

Step 3: S calls increase_stake(1 wei)
Step 4: S calls attest(block_hash_of(epoch_start + 889))
  → _validate_attestation passes
  → update_rewards_from_attestation_contract called
  → S receives full epoch rewards despite being offline during assigned window [61,70]
``` [5](#0-4)
