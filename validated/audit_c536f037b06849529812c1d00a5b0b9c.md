### Title
`current_epoch()` Panics with Arithmetic Underflow Before the First Epoch Starts — (`File: src/staking/objects.cairo`)

---

### Summary

`EpochInfo::current_epoch()` contains an arithmetic underflow that causes a panic whenever it is called before the first epoch begins (i.e., when `get_block_number() < starting_block`). Because `EpochInfo::new()` explicitly permits a future `starting_block`, and because `current_epoch()` is called by virtually every public staking function, the entire staking contract becomes non-functional from deployment until the first epoch block is reached. If any user manages to deposit funds before that block (via a path that does not call `current_epoch()`), those funds are temporarily frozen.

---

### Finding Description

`EpochInfo::new()` enforces only `starting_block >= get_block_number()`, meaning the first epoch is allowed to start at a future block number. [1](#0-0) 

The helper `update_done_in_this_epoch()` returns `true` whenever the current block is still before `starting_block`: [2](#0-1) 

`current_epoch()` uses this guard to return `*self.starting_epoch - 1` in that branch: [3](#0-2) 

`update_done_in_this_epoch()` was designed for the post-`set_epoch_info()` case, where `starting_epoch` is already a non-zero next-epoch ID, making `starting_epoch - 1` a valid current epoch. However, in the **initial** state, `starting_epoch` is set to `STARTING_EPOCH = 0`: [4](#0-3) 

When `starting_block` is in the future and `starting_epoch == 0`, the expression `*self.starting_epoch - 1` computes `0 - 1` on an unsigned integer, which **panics** in Cairo. The code comment acknowledges this: *"This function fails before the first epoch."* [5](#0-4) 

Because the staking contract has ~96 call sites for `current_epoch()` (confirmed by grep), every public entry point that touches epoch logic — `stake()`, `unstake_intent()`, `unstake_action()`, `claim_rewards()`, `attest()`, etc. — will revert with a panic before the first epoch block is reached. [6](#0-5) 

---

### Impact Explanation

**Temporary freezing of funds / griefing (High/Medium).**

- If any deposit path does not internally call `current_epoch()`, a user could lock tokens in the contract before the first epoch, then be unable to call `unstake_intent()` (which does call `current_epoch()`) until the starting block is reached.
- Even if all deposit paths also call `current_epoch()` (and therefore also revert), the entire protocol is non-functional for the window between deployment and `starting_block`, which constitutes griefing and damage to the protocol.
- The window can be arbitrarily long because `EpochInfo::new()` places no upper bound on how far in the future `starting_block` may be.

---

### Likelihood Explanation

The `EpochInfo::new()` constructor explicitly allows `starting_block > get_block_number()`. A deployment or governance action that sets a future starting block — a reasonable operational choice to synchronize a protocol launch — triggers this path deterministically. No attacker action is required; the condition is met by normal, intended usage. [7](#0-6) 

---

### Recommendation

Guard `current_epoch()` against the initial pre-epoch state. Before computing `starting_epoch - 1`, verify that `starting_epoch > 0` (i.e., at least one `set_epoch_info()` update has occurred). If `starting_epoch == 0` and `get_block_number() < starting_block`, the function should either return `STARTING_EPOCH` directly or revert with a descriptive error such as `"Epoch has not started yet"` rather than underflowing.

```cairo
fn current_epoch(self: @EpochInfo) -> Epoch {
    if self.update_done_in_this_epoch() {
        // Guard: only safe after at least one update (starting_epoch > 0)
        assert!(*self.starting_epoch > 0, "Epoch has not started yet");
        return *self.starting_epoch - 1;
    }
    ((get_block_number() - *self.starting_block) / self.epoch_len_in_blocks().into())
        + *self.starting_epoch
}
```

Callers that must operate before the first epoch (e.g., a hypothetical `cancel` path) should check `get_block_number() < epoch_info.starting_block` and handle that case explicitly, mirroring the recommendation from the original TwabRewards report.

---

### Proof of Concept

1. Deploy the staking contract with `starting_block = get_block_number() + 1000`.
2. `EpochInfo::new()` accepts this because `starting_block >= get_block_number()`. `starting_epoch` is stored as `0`.
3. Before block `starting_block` is reached, any caller invokes `stake()` (or any other public function that internally calls `self.epoch_info.read().current_epoch()`).
4. Inside `current_epoch()`: `update_done_in_this_epoch()` returns `true` (since `get_block_number() < starting_block`).
5. The function attempts to return `*self.starting_epoch - 1 = 0 - 1`.
6. Cairo's checked arithmetic panics with an underflow, reverting the transaction.
7. All public staking operations are blocked until block `starting_block` is reached. [3](#0-2) [2](#0-1)
