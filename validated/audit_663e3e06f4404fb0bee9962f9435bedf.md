### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is `external payable` and accepts ETH as fees for two L1→L2 messages. When there are no pending L2 mint requests (`amountToMint == 0`), the entire `msg.value` is silently accepted but never used and never refunded. Because the contract has no ETH-withdrawal or recovery mechanism, any ETH sent in this path is permanently locked.

---

### Finding Description

`tick()` splits `msg.value` between two downstream calls only inside the `if (amountToMint > 0)` branch:

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol  lines 107-143
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // half to bridge

        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // half to messaging
    }
    // ← no else-branch, no refund, no revert
}
``` [1](#0-0) 

When `amountToMint == 0` the function returns immediately after the `requiredMinting()` call. Any ETH attached to the call accumulates in the contract's balance with no path to recover it: there is no `receive()` / `fallback()`, no admin `withdraw()`, and no `payable(msg.sender).transfer(msg.value)` guard. [2](#0-1) 

The contract inherits `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl` — none of which introduce an ETH-recovery path. [3](#0-2) 

Additionally, even when `amountToMint > 0`, the fee split uses integer division (`msg.value / 2`), so an odd `msg.value` causes one wei to be permanently stranded in the contract on every such call. [4](#0-3) 

---

### Impact Explanation

Any ETH sent to `tick()` when `requiredMinting()` returns zero is permanently locked in the `RewardSupplier` contract. There is no recovery path. This constitutes a **permanent freezing of the caller's funds** and ongoing griefing of any keeper/bot infrastructure that calls `tick()` speculatively or in a polling loop. The protocol's operational continuity depends on keepers being willing to fund `tick()` calls; repeated silent ETH loss degrades that incentive.

Matched allowed impact: **Medium — Griefing with no profit motive but damage to users or protocol** (and potentially **High — Permanent freezing of funds** depending on the ETH amounts involved).

---

### Likelihood Explanation

`tick()` is an unrestricted public function — any address can call it. Keeper bots, automation scripts, or integrators that poll `tick()` without first calling `requiredMinting()` will silently lose ETH on every no-op invocation. The `requiredMinting()` view function exists but is not enforced inside `tick()`, so callers who do not read the source carefully will not know they must pre-check it. This is a realistic and reachable scenario. [5](#0-4) 

---

### Recommendation

1. **Revert when no work is needed and ETH is attached**: add `require(amountToMint > 0 || msg.value == 0, "NO_MINT_REQUIRED")` at the top of `tick()`, or simply revert the whole call when `amountToMint == 0` and `msg.value > 0`.
2. **Refund unused ETH**: if the function is intended to be callable with ETH even when no minting is needed, refund `msg.value` to `msg.sender` in the no-op branch.
3. **Fix the odd-wei leak**: use `msg.value - msgFee` (already done for the second fee) but ensure the first fee is computed as `msg.value / 2` and the remainder is explicitly the second fee — which the code already does — but document that any rounding remainder stays in the contract, or add a final `if (address(this).balance > 0) payable(msg.sender).transfer(address(this).balance)` sweep.

---

### Proof of Concept

1. L2 reward supplier has no pending mint requests → `requiredMinting()` returns `(0, 0)`.
2. A keeper calls `tick{value: 1 ether}()`.
3. `amountToMint == 0` → the `if` branch is skipped entirely.
4. The function returns. The `RewardSupplier` contract now holds 1 ETH with no mechanism to recover it.
5. Repeat on every polling interval; ETH accumulates permanently in the contract. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
