### Title
Tautological Safety Check in `compute_gas_refund` Allows Unguarded Unsigned Subtraction Underflow - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, a `u64` subtraction `total_gas_refund = gas_limit - gas_used` is performed at line 83, and then the result is checked with `require_internal!(total_gas_refund <= gas_limit, ...)` at lines 85–89. Because `total_gas_refund` is defined as `gas_limit - gas_used` (both `u64`), the check `total_gas_refund <= gas_limit` is a mathematical tautology for any non-wrapping subtraction — it can never be false. The guard is therefore dead code that provides no protection against the condition it was intended to catch (`gas_used > gas_limit`).

### Finding Description

In `compute_gas_refund`:

```rust
let total_gas_refund = gas_limit - gas_used;          // line 83 — plain u64 subtraction
// ...
require_internal!(
    total_gas_refund <= gas_limit,                     // line 86 — always true
    "Gas refund greater than gas limit",
    system
)?;
```

`gas_limit` and `gas_used` are both `u64`. For any valid (non-wrapping) subtraction, `gas_limit - gas_used` is always `<= gas_limit` by arithmetic identity. The check can never return an error in the normal case.

The only scenario where the check could fire is when `gas_used > gas_limit`, which would cause the subtraction at line 83 to **underflow first** — panicking in debug builds or wrapping to a huge `u64` in release builds — before the `require_internal!` guard is ever evaluated. The guard is therefore structurally unable to protect against the condition it names.

The `gas_used` value at line 83 is the result of several adjustments:

1. Initial: `gas_used = gas_limit - (remaining_ergs / ERGS_PER_GAS)` — bounded by `gas_limit` [1](#0-0) 
2. EVM refund subtracted: `gas_used -= evm_refund` where `evm_refund <= gas_used / 5` — still bounded [2](#0-1) 
3. Floored to `minimal_gas_used`: `gas_used = max(gas_used, minimal_gas_used)` — **can exceed `gas_limit` if `minimal_gas_used > gas_limit`** [3](#0-2) 
4. Native delta added: `gas_used += delta_gas as u64` — can further increase `gas_used` [4](#0-3) 

The intended safety check is placed **after** the subtraction that it is supposed to guard, making it a tautology for the normal path and unreachable for the abnormal path. [5](#0-4) 

### Impact Explanation

If `minimal_gas_used` (passed in from callers such as `context.minimal_ergs_to_charge.0 / ERGS_PER_GAS` in the ZK flow or `context.minimal_gas_to_charge` in the Ethereum flow) exceeds `gas_limit`, the subtraction at line 83 underflows. In a debug/proving build this is a panic that halts the RISC-V prover. In a release/sequencer build the value wraps to a near-`u64::MAX` number, which is then stored in `total_gas_refund` and returned inside `RefundInfo`. Callers use `gas_used` from `RefundInfo` to compute operator payment and user refund amounts; a corrupted `gas_used` would cause incorrect token transfers — either over-refunding the sender or under-paying the operator — constituting a resource-accounting bug and potential fund-loss path. [6](#0-5) 

### Likelihood Explanation

`compute_gas_refund` is called on every transaction through both the Ethereum and ZK transaction flows. [7](#0-6) [8](#0-7)  The `minimal_gas_used` parameter is derived from intrinsic gas charges computed during validation; a transaction sender can influence `gas_limit` directly. If an attacker submits a transaction with a `gas_limit` below the intrinsic minimum (e.g., just above the validation threshold but below the execution minimum), `minimal_gas_used` can exceed `gas_used` after the EVM refund step, and the `max` at line 56 can push `gas_used` above `gas_limit`. The tautological check at lines 85–89 will not catch this.

### Recommendation

Replace the post-subtraction tautological check with a pre-subtraction guard:

```rust
// Before line 83:
require_internal!(
    gas_used <= gas_limit,
    "Gas used exceeds gas limit",
    system
)?;
let total_gas_refund = gas_limit - gas_used;
// Remove the now-redundant check at lines 85-89
```

Alternatively, use `checked_sub` and propagate the error:

```rust
let total_gas_refund = gas_limit
    .checked_sub(gas_used)
    .ok_or(internal_error!("gas used exceeds gas limit"))?;
```

### Proof of Concept

1. Submit a transaction with `gas_limit = G` where `G` is just above the validation intrinsic gas threshold.
2. The execution path sets `minimal_gas_used = M` where `M > G` (e.g., due to a minimum charge floor exceeding the supplied gas limit).
3. At line 56, `gas_used = max(gas_used_before, M) = M > G = gas_limit`.
4. At line 83, `gas_limit - gas_used` underflows: panic in debug, wraps in release.
5. The `require_internal!` check at lines 85–89 evaluates `total_gas_refund <= gas_limit` — this is `true` for the non-wrapping case (tautology, never fires) and only accidentally catches the wrapping case in release mode after the damage is done.
6. In release mode, the wrapped `total_gas_refund` is never stored, but `gas_used = M > gas_limit` is returned in `RefundInfo` and used to compute operator payment and user refund, corrupting both. [9](#0-8)
