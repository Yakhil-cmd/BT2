### Title
Missing `chain_id` Validation in `interop_root_reporter_event_hook` Allows Injection of Interop Roots with Arbitrary Source Chain ID — (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` in ZKsync OS processes `InteropRootAdded` events emitted by `L2_INTEROP_ROOT_STORAGE_ADDRESS` and records them into the system's interop root storage. The hook verifies the event emitter (via hook registration keyed on `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW`) but performs **no validation on the `chain_id` field** extracted from `topics[1]`. This is the direct structural analog to the Gnosis bug: the emitter address is checked, but the source chain identity is not.

---

### Finding Description

In `interop_root_reporter_event_hook`:

```rust
let root = Bytes32::from_array(data[64..96].try_into().unwrap());
let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
system.io.add_interop_root(
    ExecutionEnvironmentType::NoEE,
    resources,
    InteropRoot {
        root,
        block_or_batch_number,
        chain_id,   // ← accepted without any validation
    },
)?;
``` [1](#0-0) 

The `chain_id` is read directly from `topics[1]` and forwarded to `add_interop_root` with zero validation. The `InteropRoot` struct documents the invariant "must be non-zero" in a comment, but neither the hook nor `push_root` enforces it: [2](#0-1) [3](#0-2) 

The hook is registered as an event hook keyed on `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW` (0x10008): [4](#0-3) 

This means it fires for **any** `InteropRootAdded` event emitted from that address, regardless of who called the contract. The hook checks the emitter identity but not the source chain identity — the exact pattern from the Gnosis report.

The missing checks are:
1. `chain_id != 0` (documented invariant, unenforced)
2. `chain_id` is not the current chain's own ID (self-referential root injection)
3. `chain_id` is not validated against any whitelist of expected source chains

---

### Impact Explanation

Every accepted `InteropRoot` is folded into the `interop_roots_rolling_hash` committed in the batch output:

```rust
// For each root: rolling_hash = keccak256(old_rolling_hash || chain_id || block_number || root_hash)
data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
``` [5](#0-4) 

This hash is part of the `BatchOutput` committed to the settlement layer: [6](#0-5) 

An injected root with a spoofed `chain_id` (e.g., `chain_id = 0`, or the ID of a chain the attacker controls) would:
- Corrupt the `interop_roots_rolling_hash` in the proven batch output
- Allow the attacker to forge a valid-looking interop root for an arbitrary source chain
- Enable downstream forgery of cross-chain asset transfers or message proofs that rely on this root

---

### Likelihood Explanation

The hook fires whenever `L2_INTEROP_ROOT_STORAGE_ADDRESS` emits `InteropRootAdded`. The normal path is via a service transaction (`ZKsyncServiceTx`) restricted to the `SERVICE_DESTINATION_WHITELIST`: [7](#0-6) 

However, `L2_INTEROP_ROOT_STORAGE_ADDRESS` is an EVM contract. Regular L2 transactions can call it with arbitrary calldata and selectors not in the whitelist. If the deployed contract at `0x10008` exposes any function that emits `InteropRootAdded` without an `onlyOperator`-equivalent guard, an unprivileged caller can trigger the hook with attacker-controlled `chain_id`. The hook itself provides no backstop — it performs no caller check and no `chain_id` validation.

---

### Recommendation

Add explicit validation in `interop_root_reporter_event_hook` before calling `add_interop_root`:

```rust
if chain_id == U256::ZERO {
    return Err(internal_error!("Interop root reporter: chain_id must be non-zero").into());
}
// Optionally: verify chain_id != current chain's own ID
// Optionally: verify chain_id is in a known-source-chain registry
``` [1](#0-0) 

Additionally, `InteropRootStorage::push_root` should enforce the documented invariant at the storage layer: [3](#0-2) 

---

### Proof of Concept

1. Deploy or interact with `L2_INTEROP_ROOT_STORAGE_ADDRESS` (0x10008) via a regular L2 transaction calling any function that emits `InteropRootAdded(chain_id=0, block_or_batch_number=N, roots=[fake_root])`.
2. The `interop_root_reporter_event_hook` fires, reads `chain_id = 0` from `topics[1]`, and calls `system.io.add_interop_root(...)` with no rejection.
3. `push_root` stores the `InteropRoot { root: fake_root, block_or_batch_number: N, chain_id: 0 }` unconditionally.
4. `calculate_interop_roots_rolling_hash` folds this root into the batch's `interop_roots_rolling_hash`.
5. The corrupted hash is committed in `BatchOutput::interop_roots_rolling_hash` and proven on the settlement layer, anchoring a forged interop root. [8](#0-7) [9](#0-8) [6](#0-5)

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())

/// Calculates a rolling hash over a sequence of interop roots.
/// This creates a cumulative digest that can be verified on settlement layers.
///
/// For each root: rolling_hash = keccak256(old_rolling_hash || chain_id || block_number || root_hash)
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
