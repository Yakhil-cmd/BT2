### Title
Unbounded loop over pool member balance checkpoints in `calculate_rewards` can permanently freeze unclaimed yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change checkpoint trace since their last claim. A pool member who repeatedly changes their delegation balance across many epochs without claiming rewards accumulates an ever-growing trace. Once the trace is large enough, the `claim_rewards` transaction will exceed Starknet's transaction gas limit, permanently freezing the pool member's unclaimed yield with no recovery path.

---

### Finding Description

`calculate_rewards` is called from `claim_rewards` and iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` up to the current epoch:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
```

The developers themselves flag this as unbounded. [1](#0-0) 

Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends exactly one new `PoolMemberBalanceCheckpoint` to the trace via `PoolMemberBalanceTrace::insert`. [2](#0-1) 

The `entry_to_claim_from` cursor is only advanced when `claim_rewards` succeeds. [3](#0-2) 

Each loop iteration performs:
1. A storage read of the checkpoint (`pool_member_trace.at(...)`)
2. A `find_sigma` call that reads up to three entries from `cumulative_rewards_trace`
3. Arithmetic

On Starknet, storage reads are among the most expensive operations. If a pool member makes one balance change per epoch for an extended period without claiming, the trace grows linearly. Once the trace is large enough that a single `claim_rewards` call cannot complete within the transaction gas limit, the pool member's accumulated rewards are permanently inaccessible — there is no pagination mechanism and no alternative entry point to partially drain the trace.

The `add_to_delegation_pool` function allows the pool member **or their registered `reward_address`** to add stake. [4](#0-3) 

This means a third party who controls or impersonates the `reward_address` can spam small additions each epoch to inflate the victim's trace, accelerating the freeze.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.** Once the pool member's balance trace grows beyond the gas-limit threshold, every call to `claim_rewards` reverts. The pool member's accumulated STRK rewards remain locked in the pool contract forever. There is no administrative escape hatch, no partial-claim function, and no way to compact or reset the trace.

This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

- A pool member who changes their balance once per epoch and never claims will accumulate one checkpoint per epoch. With Starknet epochs on the order of hours, a year of activity yields thousands of checkpoints.
- Each loop iteration costs multiple storage reads. Starknet's per-transaction resource limits are finite and well-documented.
- The developers explicitly acknowledge the risk in the source comment ("This loop is unbounded but unlikely to exceed gas limits"), confirming awareness of the failure mode.
- The `reward_address` griefing path (a third party calling `add_to_delegation_pool` with dust amounts each epoch) requires only a small token budget and is permissionless.
- Long-term passive delegators who set-and-forget are the most natural victims.

---

### Recommendation

1. **Paginate `claim_rewards`**: Accept an optional `max_checkpoints` parameter so a pool member can drain their trace incrementally across multiple transactions.
2. **Alternatively**, cap the number of balance-change checkpoints that can accumulate between claims (e.g., enforce a maximum trace length and revert `add_to_delegation_pool` / `exit_delegation_pool_intent` if the cap is reached without an intervening claim).
3. Remove or replace the comment "unlikely to exceed gas limits" with a concrete bound or a pagination guarantee.

---

### Proof of Concept

1. Deploy the staking system and create a pool member.
2. In a loop over N epochs: call `add_to_delegation_pool` with a minimal amount (or `exit_delegation_pool_intent` followed by re-entry) to append one checkpoint per epoch. Do **not** call `claim_rewards`.
3. After N epochs, call `claim_rewards`. Measure gas consumed. Observe that gas grows linearly with N.
4. At sufficiently large N (determined by Starknet's transaction gas ceiling), `claim_rewards` reverts, and the pool member's accumulated rewards are permanently frozen.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
