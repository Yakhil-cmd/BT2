### Title
Unprotected Legacy Transactions Accepted Without Chain-ID Binding, Enabling Cross-Chain Replay — (`basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS's bootloader explicitly accepts pre-EIP-155 "unprotected" legacy transactions (those with `v = 27` or `v = 28`). Because the signing hash for such transactions covers only `[nonce, gasPrice, gasLimit, to, value, data]` with no chain identifier, a transaction signed by a user on any other EVM chain (Ethereum mainnet, another L2, etc.) is cryptographically valid on ZKsync OS and can be replayed by an unprivileged attacker to drain the victim's funds.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155-protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
``` [1](#0-0) 

`is_eip155()` returns `false` precisely when `v == 27 || v == 28`:

```rust
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
``` [2](#0-1) 

When `is_eip155()` is false, the signing hash is computed over only the six payload fields — **no chain ID is included**. The transaction is then stored as the `Legacy` variant (not `LegacyWithEIP155`) and proceeds through full validation:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    Self::Legacy(tx, sig_data)
};
Ok((tx, sig_hash))
``` [3](#0-2) 

There is no rejection of the `Legacy` (unprotected) variant anywhere in the validation pipeline. The `sig_parity_r_s` method also handles it without error:

```rust
RlpEncodedTxInner::Legacy(_, sig) => {
    let parity = sig.v - U256::from(27) == U256::ONE;
    Some((parity, sig.r, sig.s)) // prechecked
}
``` [4](#0-3) 

The ecrecover-based signature check in the Ethereum validation path then verifies this chain-ID-free hash against the recovered signer, and if it matches the `from` address, the transaction is accepted and executed: [5](#0-4) 

---

### Impact Explanation

Any pre-EIP-155 legacy transaction that a user signed on Ethereum mainnet (or any other EVM chain) — with `v = 27` or `v = 28` — carries a signature over a chain-agnostic hash. An attacker who observes such a transaction on another chain can submit the identical RLP bytes to ZKsync OS. If the victim has funds at the same address on ZKsync OS, the bootloader will:

1. Parse the transaction as `RlpEncodedTxInner::Legacy`.
2. Compute the same chain-ID-free signing hash.
3. Recover the same signer address via ecrecover.
4. Accept and execute the transaction — transferring value or executing arbitrary calldata — without the victim's knowledge or consent on ZKsync OS.

This results in direct, irreversible loss of user funds.

---

### Likelihood Explanation

Pre-EIP-155 transactions are still produced today by certain hardware wallets, old tooling, and some dApps that do not enforce EIP-155. Any such transaction broadcast on Ethereum mainnet is publicly visible in the mempool and on-chain. An attacker monitoring the Ethereum mempool or historical blocks for unprotected legacy transactions targeting addresses that also hold funds on ZKsync OS can replay them with zero additional cost. The attacker needs no privileged access — only the ability to submit a transaction to ZKsync OS.

---

### Recommendation

**Short term:** Reject unprotected legacy transactions (those with `v = 27` or `v = 28`) at the parsing stage. In `parse_and_compute_signed_hash`, after calling `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, check `sig_data.is_eip155()` and return `Err(InvalidTransaction::InvalidChainId)` if it is false. This mirrors the behavior already enforced for all typed transaction formats (EIP-2930, EIP-1559, EIP-4844, EIP-7702).

**Long term:** Audit all signature verification paths in the bootloader to confirm that every accepted signature scheme binds the signed data to the ZKsync OS chain ID, preventing cross-chain and cross-network replay.

---

### Proof of Concept

1. On Ethereum mainnet, Alice signs a legacy ETH transfer to Bob with `v = 27` (no EIP-155). The raw RLP bytes are publicly visible.
2. Alice also holds ETH at the same address on ZKsync OS.
3. Attacker Eve takes the identical raw RLP bytes and submits them to ZKsync OS.
4. The bootloader parses the transaction as `RlpEncodedTxInner::Legacy`, computes `keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the Ethereum signing hash — and recovers Alice's address via ecrecover.
5. The transaction passes all validation checks (nonce, balance, signature) and executes, transferring Alice's ZKsync OS funds to Bob without Alice's authorization on ZKsync OS.
