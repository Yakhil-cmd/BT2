### Title
Malicious Coordinator Can Present Inconsistent Rerandomized Presignatures to Different Signers, Enabling Secret Key Extraction — (File: `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The Robust ECDSA `sign()` function accepts an externally-supplied `RerandomizedPresignOutput` per participant without any cross-participant consistency check on the `(msg_hash, tweak, participants)` inputs used during rerandomization. A malicious coordinator controls what rerandomized presignature each signer receives and can silently present different `(h, ε)` pairs to different participants within the same signing session. Because all participants send their shares only to the coordinator, the coordinator can collect shares from two inconsistent views of the same presignature and recover the aggregate secret key via a standard ECDSA nonce-reuse attack.

---

### Finding Description

**Root cause — missing cross-participant presignature consistency enforcement in `sign()`**

The `sign()` entry point at `src/ecdsa/robust_ecdsa/sign.rs` (lines 33–108) validates:

- `participants.len() == 2 * max_malicious + 1` (lines 86–90)
- `msg_hash != 0` (lines 91–95)
- Coordinator and self are in the participant list (lines 52–65)

It does **not** validate that the caller-supplied `presignature: RerandomizedPresignOutput` was derived from the same `(msg_hash, tweak, participants)` that the `sign()` call itself is using, nor does it verify that every other participant received an identical rerandomized presignature.

`RerandomizedPresignOutput` is produced externally by a `rerandomize()` call that takes `(participants, msg_hash, tweak, presign_output, public_key, entropy)` as free inputs. The coordinator is the entity that distributes these rerandomized presignatures to each signer before the signing round begins. Nothing in the protocol or the library code forces the coordinator to use the same `(h, ε)` for every participant.

**Exploit flow**

1. Presigning completes normally with `N1 = 2t+1` participants, producing a shared `PresignOutput` (nonce `k`, point `R`).
2. Malicious coordinator calls `rerandomize(participants, h₁, ε₁, presign, pk, ρ)` → `rps_A`.
3. Malicious coordinator calls `rerandomize(participants, h₂, ε₂, presign, pk, ρ)` → `rps_B` (same underlying `k`, different `δ`).
4. Coordinator distributes `rps_A` to a subset of signers and `rps_B` to the remaining signers (or runs two parallel sessions with the same presignature).
5. Each participant independently calls `sign(…, presignature=rps_X, msg_hash=hₓ)`. All local checks pass because `participants.len() == 2t+1` and `msg_hash != 0`.
6. Participants send their signature shares **only to the coordinator** (line 121 of `sign.rs`). No participant can observe what the others received.
7. The coordinator now holds shares from two inconsistent views of the same nonce. Because the underlying nonce `k` is shared and the rerandomization scalars `δ₁`, `δ₂` are known to the coordinator (it chose `ρ`, `ε₁`, `ε₂`), the coordinator can apply standard ECDSA nonce-reuse algebra to recover the aggregate secret key `x`.

The security documentation at `docs/ecdsa/robust_ecdsa/signing.md` (lines 147–181) explicitly acknowledges this risk and states: *"ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers."* However, the library code provides **no enforcement mechanism** for this requirement. The `sign()` function trusts the caller-supplied `presignature` entirely.

The structural parallel to the external report is exact: just as `enableTradingWithWeights()` consumed weights set by `disableTrading()` without re-checking the `allowance()` invariant, `sign()` consumes a `RerandomizedPresignOutput` set externally by the coordinator without re-checking that the presignature is consistent with the `msg_hash` and participant set the function itself is operating on.

---

### Impact Explanation

**Critical — extraction/reconstruction of the aggregate private signing key.**

A malicious coordinator can recover the full secret key `x` from two signing sessions that reuse the same presignature nonce with different `(h, ε)` values. This satisfies the allowed impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

Once `x` is recovered, the coordinator can forge arbitrary signatures for any message without participation from any honest party.

---

### Likelihood Explanation

**High.** The coordinator is a required, explicitly trusted role in the signing protocol. The attack requires no external assumptions, no cryptographic breaks, and no leaked keys. The coordinator already controls the distribution of rerandomized presignatures to each participant. The only precondition is that the coordinator is malicious — which is exactly the threat model the security documentation warns about but the code does not defend against.

---

### Recommendation

1. **Bind `(msg_hash, tweak, participants)` inside `RerandomizedPresignOutput`** and verify at the start of `sign()` that the embedded values match the arguments passed to `sign()`. This prevents a coordinator from handing a participant a presignature rerandomized under a different `(h, ε)` than the one the participant believes it is signing.

2. **Add a participant-to-participant consistency round**: before computing signature shares, have each participant broadcast a commitment to their received `(msg_hash, tweak, presignature.big_r)` and abort if any mismatch is detected. This removes the coordinator's ability to present split views.

3. **Enforce `N1 = N2 = 2t+1` at the presign level as well**, so that the presigning participant set is recorded inside `PresignOutput` and verified against the signing participant set during rerandomization.

/// To reduce risk in this implementation, require `N1 = N2 = 2 * max_malicious + 1`,
/// ensure all participants agree on `(msg_hash, tweak, participants)` when creating
/// `RerandomizedPresignOutput`, never reuse a presignature, and do not sign with
/// `msg_hash == 0`.
pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

4. **Do not sign with $h = 0$** (the zero message hash).
   This input enables a related algebraic split-view attack in the modified scheme when
   $N_1 > 2t + 1$.
