### Title
`skip_stateful_validations` Bypasses `__validate__` for Nonce-1 Invoke When Any Mempool Transaction Exists, Not Only `deploy_account` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call for any invoke transaction with `nonce == 1` whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. That helper returns `true` for **any** transaction type in the pool, not exclusively a `deploy_account`. An attacker can therefore admit an invoke with an invalid signature (or one that would otherwise fail `__validate__`) into the mempool without the gateway ever calling the account's validation entry point.

---

### Finding Description

`skip_stateful_validations` is the gateway-side UX shortcut that lets a user broadcast `deploy_account` (nonce 0) and `invoke` (nonce 1) simultaneously before the account exists on-chain. [1](#0-0) 

The skip condition fires when:

1. The transaction is `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in state).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The code comment reads:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of that disjunction is the flaw. `account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

It returns `true` for **any** transaction type — invoke, declare, or deploy-account — that is currently in the pool or was seen in a recent committed block. It does not distinguish a `deploy_account` from a regular `invoke`.

**Concrete attack path (deployed account, nonce 0 in state):**

1. Account `X` is deployed on-chain with nonce `0` (has never executed a transaction).
2. Attacker submits a valid `invoke` with `nonce = 0` for `X`. This passes `__validate__` normally and is added to the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.
3. Attacker submits a second `invoke` with `nonce = 1` for `X` carrying an **invalid signature** (or calldata that would fail the account's custom `__validate__` logic).
4. Gateway flow:
   - `validate_state_preconditions`: passes — nonce 1 is within the allowed gap from account nonce 0.
   - `validate_by_mempool`: passes — no duplicate hash, nonce within range; the mempool does not verify signatures.
   - `skip_stateful_validations`: condition `nonce == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block == true` is satisfied → returns `true`.
   - `run_validate_entry_point` is called with `skip_validate = true`, so `execution_flags.validate = false`. [3](#0-2) 

5. The invalid nonce-1 invoke is added to the mempool **without any signature or `__validate__` check**.

**Concrete attack path (undeployed account, deploy_account in mempool):**

1. Legitimate user broadcasts `deploy_account` (nonce 0) for address `X`. It enters the mempool; `account_tx_in_pool_or_recent_block(X)` becomes `true`.
2. Attacker — who does not know `X`'s private key — submits an `invoke` with `nonce = 1` for `X` with an arbitrary/invalid signature.
3. Same gateway flow as above: `skip_stateful_validations` returns `true` because the deploy_account is in the pool.
4. The attacker's invalid nonce-1 invoke enters the mempool. If the attacker sets a higher fee than the legitimate owner's nonce-1 invoke, fee-escalation logic replaces the legitimate transaction with the attacker's invalid one. [4](#0-3) 

The batcher always constructs transactions with `validate: true` via `AccountTransaction::new_for_sequencing`: [5](#0-4) 

So the invalid transaction is ultimately **rejected** during block execution. However, the gateway invariant — that every admitted transaction has passed its account's `__validate__` entry point or is provably safe to skip — is broken.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

- Any invoke with `nonce = 1` targeting an address that has *any* pending transaction in the mempool bypasses the gateway's `__validate__` call.
- An attacker can inject transactions with invalid signatures, wrong calldata, or any payload that would normally be rejected by the account contract's validation logic.
- In the fee-escalation scenario the attacker can displace a legitimate user's nonce-1 invoke with an invalid one, forcing the user to resubmit and delaying their transaction.
- Mempool resources are consumed by transactions that will always be rejected at execution time.

---

### Likelihood Explanation

The precondition — an address with `account_nonce == 0` in state and any transaction in the mempool — is routinely satisfied during normal operation: every new account deployment creates exactly this window. The attack requires no privileged access and no special on-chain state beyond observing the public mempool.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the sender address. Expose a dedicated mempool query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, and use that in `skip_stateful_validations` instead of the current broad presence check.

```rust
// Proposed fix sketch
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .deploy_account_in_pool(tx.sender_address())  // type-specific check
                .await
                ...
        }
    }
    Ok(false)
}
```

### Proof of Concept

```
State:  account X deployed, nonce = 0 in chain state.

Step 1: Submit invoke_A  { sender: X, nonce: 0, signature: VALID }
        → gateway calls __validate__ → passes → added to mempool
        → account_tx_in_pool_or_recent_block(X) == true

Step 2: Submit invoke_B  { sender: X, nonce: 1, signature: INVALID }
        → validate_state_preconditions: nonce 1 ∈ [0, 0+gap] → OK
        → validate_by_mempool: no duplicate, nonce in range → OK
        → skip_stateful_validations:
              nonce == 1 ✓
              account_nonce == 0 ✓
              account_tx_in_pool_or_recent_block(X) == true ✓
              → returns true (skip __validate__)
        → run_validate_entry_point called with skip_validate=true
              → execution_flags.validate = false
              → __validate__ NOT called
        → invoke_B admitted to mempool with invalid signature

Step 3: Batcher fetches invoke_B, executes with validate=true
        → __validate__ called → fails (invalid signature)
        → invoke_B rejected from block

Broken invariant: the gateway admitted a transaction that fails
account validation, violating the admission guarantee.
``` [1](#0-0) [2](#0-1) [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
