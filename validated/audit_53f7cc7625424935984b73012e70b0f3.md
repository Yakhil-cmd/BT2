### Title
Unvalidated `refund_recipient` in L1→L2 Transactions Allows Attacker to Redirect Treasury Refunds to Arbitrary Addresses, Including the Treasury Itself, Causing L2AssetTracker Accounting Inflation - (File: `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The `process_l1_transaction` function in the ZKsync OS bootloader does not validate the `refund_recipient` field (`reserved[1]`) of L1→L2 priority transactions. An attacker can set this field to any arbitrary address — including the `BASE_TOKEN_HOLDER_ADDRESS` (the treasury itself) — causing the L2AssetTracker to be notified of a token bridging event that produces no net balance change, inflating the tracker's accounting and potentially halting block processing.

---

### Finding Description

In `validate_structure` for ABI-encoded L1/upgrade transactions, the `reserved[1]` field (the `refund_recipient`) is explicitly left unvalidated with a developer `// TODO: validate address?` comment: [1](#0-0) 

At execution time in `process_l1_transaction`, the refund recipient is read directly from `transaction.reserved[1].read()` and passed to `mint_base_token` without any address constraint: [2](#0-1) 

`mint_base_token` first calls `notify_l2_asset_tracker` (which notifies the L2AssetTracker of a bridging event for the full refund amount), then calls `transfer_from_treasury` which subtracts from the treasury and adds to the `to` address: [3](#0-2) 

`transfer_from_treasury` performs two sequential balance updates — subtract from treasury, add to `to`: [4](#0-3) 

If the attacker sets `refund_recipient = BASE_TOKEN_HOLDER_ADDRESS`, the treasury subtracts X then immediately adds X back (net zero), but `notify_l2_asset_tracker` has already been called with amount X, falsely reporting a bridging event to the L2AssetTracker.

---

### Impact Explanation

1. **L2AssetTracker accounting inflation**: The L2AssetTracker's `handleFinalizeBaseTokenBridgingOnL2` is called with the refund amount even though no net token movement occurred. This inflates the tracker's view of total bridged supply. The comment in the code notes that if the L2AssetTracker reverts, it is treated as a **fatal block-processing error**: [5](#0-4) 

2. **Potential chain halt**: If repeated false notifications cause the L2AssetTracker to enter an inconsistent state and revert on a subsequent legitimate transaction, block processing halts entirely.

3. **Redirect of refund funds**: An attacker can also redirect their gas refund to any arbitrary address (e.g., a contract they control), bypassing the intended behavior where the refund goes to the transaction sender or a designated recipient validated on L1.

---

### Likelihood Explanation

Any user who can submit an L1→L2 priority transaction can set `refund_recipient` to an arbitrary value. The ZKsync OS bootloader explicitly skips validation of this field (confirmed by the `// TODO: validate address?` comment). The L1 contracts are expected to enforce this, but the bootloader provides no defense-in-depth check. This is directly reachable by an unprivileged transaction sender with no special access required.

---

### Recommendation

1. In `validate_structure` for `L1_L2_TX_TYPE` and `UPGRADE_TX_TYPE`, validate that `reserved[1]` encodes a valid 20-byte Ethereum address (i.e., the upper 12 bytes are zero) and that it is not a reserved system address such as `BASE_TOKEN_HOLDER_ADDRESS`.

2. Resolve the `// TODO: validate address?` at: [1](#0-0) 

3. In `process_l1_transaction`, add an explicit guard before calling `mint_base_token` for the refund that rejects `refund_recipient == BASE_TOKEN_HOLDER_ADDRESS` (and other system-reserved addresses).

---

### Proof of Concept

1. Craft an L1→L2 priority transaction with:
   - `gas_limit` = 100,000
   - `gas_price` = 1,000
   - `to_mint` (`reserved[0]`) = `gas_limit * gas_price` (minimum deposit)
   - `refund_recipient` (`reserved[1]`) = `BASE_TOKEN_HOLDER_ADDRESS` (0x0000...010001 or the configured treasury address)
2. Submit the transaction. The transaction body executes (or reverts), consuming some gas.
3. The bootloader computes `to_refund_recipient = prepaid_fee - pay_to_operator > 0`.
4. `mint_base_token` is called with `to = BASE_TOKEN_HOLDER_ADDRESS`:
   - `notify_l2_asset_tracker` fires with the refund amount → L2AssetTracker records a bridging event.
   - `transfer_from_treasury` subtracts X from treasury, then adds X back to treasury → net zero.
5. The L2AssetTracker's internal supply accounting is now inflated by X tokens that were never actually moved.
6. Repeat to accumulate accounting drift until the L2AssetTracker reverts on a legitimate transaction, halting block processing.
