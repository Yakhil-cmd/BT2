### Title
Missing Upper-Bound Validation on `threshold` vs `participants.len()` in `assert_key_invariants` Allows Permanently Unusable DKG Output - (File: src/dkg.rs)

### Summary
`assert_key_invariants` enforces a lower bound (`threshold >= 2`) but omits the upper-bound check (`threshold <= participants.len()`). A caller can invoke `keygen` or `reshare` with a threshold exceeding the number of participants. The DKG protocol completes without error, distributing shares that can never reconstruct the secret key, permanently denying signing to all honest parties.

### Finding Description
`assert_key_invariants` is the shared validation gate for both `keygen` and `reshare` (via `assert_reshare_keys_invariants`). Its body is:

```rust
// src/dkg.rs ~line 580
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
let participants =
    ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;
if !participants.contains(me) {
    return Err(InitializationError::MissingParticipant { ... });
}
Ok(participants)
``` [1](#0-0) 

There is no guard of the form `threshold <= participants.len()`. Contrast this with two other validation helpers in the same codebase that do enforce the upper bound:

- `validate_triple_inputs` (OT-based triple generation): [2](#0-1) 

- `assert_sign_inputs` (FROST signing): [3](#0-2) 

Both of these correctly reject `threshold > participants.len()` with `InitializationError::ThresholdTooLarge`. The DKG path does not.

When `threshold > participants.len()`, each participant generates a polynomial of degree `threshold - 1`, evaluates it at the (fewer) participant identifiers, and distributes shares. The commitment length checks inside `verify_proof_of_knowledge` only verify that the commitment has exactly `threshold` coefficients — they do not compare `threshold` against the participant count: [4](#0-3) 

The protocol therefore runs to completion and returns `KeygenOutput` structs to all participants. The shares are mathematically valid evaluations of a degree-`(threshold-1)` polynomial, but since there are fewer than `threshold` participants, Lagrange interpolation can never reconstruct the secret. Every subsequent signing attempt will fail because no quorum of size `threshold` can ever be assembled.

The same gap is inherited by `assert_reshare_keys_invariants`, which calls `assert_key_invariants` for the new threshold/participants without adding the missing upper-bound check: [5](#0-4) 

### Impact Explanation
**High — Permanent denial of signing for honest parties.**

Once a DKG or reshare completes with `threshold > participants.len()`, the resulting key material is permanently unusable. No signing session can ever succeed because the required quorum size exceeds the total number of key-share holders. The key cannot be recovered or repaired without running a fresh DKG, which requires out-of-band coordination and discards the existing public key. All on-chain or off-chain assets bound to that public key become inaccessible.

### Likelihood Explanation
The library is consumed as a Rust crate by application-layer coordinators. A misconfigured or malicious coordinator can trivially pass `threshold = N+1` with `N` participants. The existing test suite only exercises the lower-bound path (`ThresholdTooSmall`): [6](#0-5) 

There is no test asserting that `threshold > participants.len()` is rejected at the keygen/reshare entry point, confirming the gap is untested and reachable.

### Recommendation
Add the following guard inside `assert_key_invariants`, immediately after the lower-bound check:

```rust
if threshold > participants_slice.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold,
        max: participants_slice.len(),
    });
}
```

This mirrors the pattern already used in `validate_triple_inputs` and `assert_sign_inputs`. Add a corresponding test analogous to `keygen__should_fail_if_threshold_is_below_limit` that verifies `keygen` and `reshare` both return `ThresholdTooLarge` when `threshold > participants.len()`.

### Proof of Concept

```rust
// Pseudocode demonstrating the missing check
let participants = generate_participants(3); // 3 participants
let threshold = 5;                           // threshold > participants.len()

// assert_key_invariants passes: threshold >= 2 ✓, no duplicates ✓, me present ✓
// No check: threshold (5) <= participants.len() (3) — MISSING

let result = keygen::<Secp256K1Sha256>(
    &participants,
    participants[0],
    threshold,   // 5 > 3, but no error is returned
    rng,
);

// result is Ok(KeygenOutput { ... }) — DKG completed "successfully"
// But signing will always fail: you need 5 shares, only 3 exist.
// The key is permanently unusable.
```

The `ThresholdTooLarge` error variant already exists in `InitializationError`: [7](#0-6) 

It is simply never emitted from the DKG initialization path.

    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_threshold: impl Into<ReconstructionLowerBound>,
    old_participants: &[Participant],
) -> Result<(ParticipantList, ParticipantList), InitializationError> {
    let threshold = usize::from(threshold.into());
    let old_threshold = usize::from(old_threshold.into());

    let participants = assert_key_invariants(participants, me, threshold)?;

    let old_participants =
        ParticipantList::new(old_participants).ok_or(InitializationError::DuplicateParticipants)?;

    // Step 1.1
    if old_participants.intersection(&participants).len() < old_threshold {
        return Err(InitializationError::NotEnoughParticipantsForNewThreshold {
            threshold: old_threshold,
            participants: old_participants.intersection(&participants).len(),
        });

    // Step 1.1

            "party {me:?} is present in the old participant list but provided no share"
        )));
    }
    Ok((participants, old_participants))
}
