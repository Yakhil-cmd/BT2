### Title
Gateway Skips `__validate__` for Invoke Transactions Whose Sender Has Any Mempool Entry, Allowing Signature-Free Admission Under a Victim's Address - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator returns `true` (skip `__validate__`) for any invoke transaction with `nonce == 1` whose `sender_address` has **any** transaction present in the mempool or a recent block. Because the check is keyed only on the sender address and not on the submitter's identity, an attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a second invoke transaction that claims the victim's address as `sender_address`, carries an invalid signature, and is admitted to the mempool without any signature verification.

### Finding Description

**Invariant broken**: Every transaction admitted to the mempool must carry a valid signature from its declared `sender_address`.

**Root cause** — `skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ← keyed on attacker-supplied address
                .await
                ...
        }
    }
    Ok(false)
}

The earlier `validate_by_mempool` call does **not** close this gap: `ValidationArgs` carries only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — no signature field — and `Mempool::validate_tx` checks only nonce ordering and fee escalation. [4](#0-3) [5](#0-4) 

`account_tx_in_pool_or_recent_block` returns `true` whenever the address appears in the pool or the committed-block state, regardless of who submitted the matching transaction:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [6](#0-5) 

**Attack path**:

1. Victim submits `deploy_account` (nonce 0) and a legitimate `invoke` (nonce 1) to the gateway.
2. The `deploy_account` is admitted first; the victim's address now appears in the mempool.
3. Before the victim's `invoke` reaches the mempool, the attacker submits a crafted `invoke` with `sender_address = victim`, `nonce = 1`, and an arbitrary/empty signature.
4. Gateway stateful validation:
   - `validate_nonce`: passes (`0 ≤ 1 ≤ max_allowed_nonce_gap`).
   - `validate_by_mempool`: passes (no duplicate nonce yet, no fee escalation issue).
   - `skip_stateful_validations`: returns `true` because `victim_address` is in the mempool.
   - `run_validate_entry_point`: `validate = false`; `__validate__` is **never called**.
5. The attacker's invalid transaction is inserted into the mempool under the victim's address at nonce 1.
6. The victim's legitimate `invoke` (nonce 1) is now rejected with `MempoolError::DuplicateNonce`. [7](#0-6) 

### Impact Explanation

The gateway admits an invoke transaction that carries no valid signature from its declared `sender_address`. This directly violates the admission invariant. Concretely:

- The victim's legitimate nonce-1 invoke transaction is evicted from the admission path with `DuplicateNonce`.
- The attacker's invalid transaction occupies the victim's nonce-1 slot in the mempool.
- At batcher execution time `__validate__` is called with `validate = true`; it fails, and the batcher rejects the transaction — but the victim's slot has already been squatted.
- The attacker can repeat this indefinitely, permanently blocking the victim's account from progressing through the `deploy_account + invoke` UX flow.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The preconditions are observable on-chain: any account submitting a `deploy_account` transaction is visible in the public mempool. The attack requires no privileged access, no special tokens, and no prior relationship with the victim. The race window is the time between the `deploy_account` being admitted and the victim's `invoke` being admitted — a window that is reliably exploitable by any mempool-watching bot.

### Recommendation

The `skip_stateful_validations` check must verify that the transaction in the mempool for the sender's address is specifically a `deploy_account` transaction, not just any transaction. Alternatively, the gateway should still run a lightweight signature check (e.g., verify the ECDSA/Stark signature against the transaction hash and the sender's public key stored in the class) even when skipping the full `__validate__` entry-point execution. A minimal fix is to change `account_tx_in_pool_or_recent_block` to `deploy_account_tx_in_pool_or_recent_block`, ensuring the skip only applies when the account's own deploy-account transaction is pending. [8](#0-7)

```rust
/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
