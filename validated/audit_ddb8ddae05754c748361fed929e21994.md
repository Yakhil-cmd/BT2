### Title
`FeeEstimation.overall_fee` Includes L2 Gas Cost Not Reflected in Any Returned Field, Violating the Published Formula - (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`tx_execution_output_to_fee_estimation` populates `overall_fee` from `receipt.fee`, which is computed by `GasVector::cost()` and includes L2 gas cost (`l2_gas * l2_gas_price`). However, the returned `FeeEstimation` struct has no `l2_gas_consumed` field, and the struct doc-comment, the OpenRPC spec, and the RPC API description all state that `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`. When a v3 transaction with `AllResources` bounds consumes nonzero L2 gas under a block context with nonzero `l2_gas_price`, `overall_fee` is strictly greater than that formula by `l2_gas * l2_gas_price`, and clients have no way to reconstruct the correct value from the returned fields.

### Finding Description

**Fee computation path:**

`GasVector::cost()` in `crates/starknet_api/src/execution_resources.rs` computes:

```
fee = l1_gas * l1_gas_price
    + l1_data_gas * l1_data_gas_price
    + l2_gas * (l2_gas_price + tip)
``` [1](#0-0) 

`TransactionReceipt::from_params()` calls `get_fee_by_gas_vector(gas, ...)` where `gas` is the full `GasVector` including `l2_gas`, so `receipt.fee` includes the L2 gas cost: [2](#0-1) 

**Fee estimation output path:**

`tx_execution_output_to_fee_estimation` then builds `FeeEstimation` as:
- `gas_consumed` ← `gas_vector.l1_gas` (L1 gas only)
- `data_gas_consumed` ← `gas_vector.l1_data_gas` (L1 data gas only)
- `overall_fee` ← `receipt.fee` (includes L2 gas cost)
- No `l2_gas_consumed` field exists [3](#0-2) 

The struct doc-comment explicitly states the invariant that is violated:

> The total amount of fee. This is equal to: `gas_consumed * gas_price + data_gas_consumed * data_gas_price`. [4](#0-3) 

The OpenRPC spec repeats the same broken invariant: [5](#0-4) 

**The TODO in the code acknowledges the gap:**

> TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price. [6](#0-5) 

**When is L2 gas tracked separately?**

For v3 transactions with `AllResources` bounds, `get_gas_vector_computation_mode()` returns `GasVectorComputationMode::All`, meaning L2 gas is tracked as a distinct component in `GasVector.l2_gas` and priced at `l2_gas_price`. For v3 transactions with only `L1Gas` bounds (`NoL2Gas` mode), L2 gas is converted to L1 gas equivalent and folded into `l1_gas`, so the inconsistency does not arise in that mode.

The blockifier fee test confirms the L2 gas contribution is real and non-trivial: [7](#0-6) 

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` for a v3 transaction with `AllResources` bounds that consumes L2 gas receives a `FeeEstimation` where:

```
overall_fee  ≠  gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price
```

The discrepancy is exactly `l2_gas * l2_gas_price`. Since no `l2_gas_consumed` field is returned, clients cannot detect or correct this. Wallets and dApps that verify the fee estimate using the published formula will compute a lower value than `overall_fee`, causing them to either reject valid fee estimates or silently accept an inflated fee without understanding why.

### Likelihood Explanation

This triggers for every v3 invoke/declare/deploy-account transaction that uses `AllResources` bounds (the standard v3 format) and executes any Cairo 1 code, since Cairo 1 execution always consumes Sierra gas (L2 gas). The block context always has a nonzero `l2_gas_price` in production. This is therefore the common case for all modern v3 transactions.

### Recommendation

Add an `l2_gas_consumed: Felt` field to `FeeEstimation` populated from `gas_vector.l2_gas.0`, and update the doc-comment and OpenRPC spec description to include the L2 gas term:

```
overall_fee = gas_consumed * l1_gas_price
            + data_gas_consumed * l1_data_gas_price
            + l2_gas_consumed * l2_gas_price
```

### Proof of Concept

```rust
// In tx_execution_output_to_fee_estimation, after building FeeEstimation:
let gas_vector = tx_execution_output.execution_info.receipt.gas;
let reconstructed = gas_vector.l1_gas.0 as u128 * l1_gas_price.0
    + gas_vector.l1_data_gas.0 as u128 * l1_data_gas_price.0;
// When l2_gas > 0 and l2_gas_price > 0:
assert_ne!(
    fee_estimation.overall_fee.0,
    reconstructed,
    "overall_fee includes l2_gas cost ({} * {}) not reflected in returned fields",
    gas_vector.l2_gas.0,
    l2_gas_price.0
);
```

A concrete unit test would: construct a `BlockContext` with nonzero `l2_gas_price` (e.g., `l2_gas_price = 5`), execute a Cairo 1 invoke transaction with `AllResources` bounds (which always produces nonzero `l2_gas`), call `tx_execution_output_to_fee_estimation`, and assert `overall_fee.0 > gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`. The difference equals `receipt.gas.l2_gas.0 * l2_gas_price.0`.

    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
