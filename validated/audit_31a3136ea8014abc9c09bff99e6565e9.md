### Title
State Desync via `pause_contract` Racing `factory_update_address_version` Callback — (`engine/src/contract_methods/xcc.rs`)

### Summary

`factory_update_address_version` gates its entire body behind `require_running`, including the `set_code_version_of_address` write. Because NEAR promise callbacks execute in a separate receipt from the originating call, the owner can call `pause_contract` in the window between the router deploy receipt and the callback receipt. The callback then aborts with `ERR_PAUSED` before updating the engine's `code_version_of_address` record, leaving the on-chain router at the new version while the engine's storage still reflects the old version (or `None` for a first-time deploy). The desync is not self-healing: `factory_update_address_version` is gated by `assert_private_call`, so no external actor can invoke it directly to repair the record.

### Finding Description

**Step 1 — `fund_xcc_sub_account` schedules deploy + callback**

`xcc::fund_xcc_sub_account` detects `AddressVersionStatus::DeployNeeded`, builds a batch promise (create/deploy/initialize), and attaches `factory_update_address_version` as a callback carrying `AddressVersionUpdateArgs { address, version: latest_code_version }`. [1](#0-0) 

**Step 2 — Router is deployed on-chain (separate NEAR receipt)**

The batch executes in its own receipt. The router contract now exists on-chain at `latest_code_version`.

**Step 3 — Owner calls `pause_contract` (separate NEAR transaction)**

`pause_contract` requires only owner authorization and that the engine is currently running. It sets `state.is_paused = true` and commits. [2](#0-1) 

**Step 4 — `factory_update_address_version` callback fires and aborts**

The callback receipt executes. The very first thing it does is `require_running(&state::get_state(&io)?)`. Because the engine is now paused, this returns `ERR_PAUSED` and the function returns early — `set_code_version_of_address` is never reached. [3](#0-2) 

**Step 5 — Permanent desync for new-account case**

For a first-time deploy, `get_code_version_of_address` returns `None`. `AddressVersionStatus::new` therefore produces `DeployNeeded { create_needed: true }`. Every subsequent `fund_xcc_sub_account` call for that address will schedule a `CreateAccount` action against an account that already exists on NEAR. NEAR rejects the batch; the callback receives `Some(false)` and returns `ERR_ROUTER_DEPLOY_FAILED`; the version is still never written. The loop is unbreakable without a contract upgrade because `factory_update_address_version` enforces `assert_private_call` — no external caller can invoke it to repair the record. [4](#0-3) [5](#0-4) 

### Impact Explanation

For any EVM address whose XCC router was being deployed for the first time when the race occurs:

- The router sub-account holds NEAR (transferred during the deploy batch). Those funds are inaccessible via the XCC precompile path because the engine perpetually believes the account needs to be created.
- The user cannot use XCC from that EVM address until the engine is upgraded to add a manual version-repair path.
- This constitutes **temporary (potentially permanent without upgrade) freezing of funds** in the XCC router sub-account.

For the upgrade case (`create_needed: false`), the impact is limited to one extra re-deploy after resume, which succeeds — no freeze.

### Likelihood Explanation

- The owner must call `pause_contract` in the narrow window between the deploy receipt and the callback receipt. On NEAR, these are separate blocks, so the window is at least one block wide.
- Pausing for maintenance, emergency response, or upgrades is a routine operation. The owner has no visibility into pending callbacks.
- No malicious intent is required; an accidental pause during routine maintenance is sufficient.
- Likelihood is **low-to-medium** given the timing requirement, but the impact for affected users is severe and unrecoverable without a contract upgrade.

### Recommendation

Remove the `require_running` guard from `factory_update_address_version`. This callback is already protected by `assert_private_call` (only the engine itself can invoke it) and by `promise_result_check` (it only proceeds if the deploy succeeded). The pause flag is irrelevant to the correctness of recording a version that was already successfully deployed. Alternatively, if a pause check is desired, the callback should silently succeed (no-op) rather than abort, so the version is still written. [3](#0-2) 

### Proof of Concept

```
1. Deploy engine; call factory_update to set router bytecode (version 1).
2. Call fund_xcc_sub_account for address A with sufficient NEAR.
   → Batch receipt: CreateAccount + Transfer + DeployContract + initialize
   → Callback receipt scheduled: factory_update_address_version(A, 1)
3. Before the callback receipt executes, call pause_contract as owner.
4. Callback receipt executes:
   - require_running → ERR_PAUSED → abort
   - set_code_version_of_address never called
5. Assert: get_code_version_of_address(A) == None  (desync confirmed)
6. Resume contract. Call fund_xcc_sub_account for address A again.
   → AddressVersionStatus::DeployNeeded { create_needed: true }
   → Batch: CreateAccount on already-existing account → NEAR rejects
   → Callback: Some(false) → ERR_ROUTER_DEPLOY_FAILED
   → Version still None
7. Repeat step 6 indefinitely — XCC for address A is permanently broken.
```

pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
