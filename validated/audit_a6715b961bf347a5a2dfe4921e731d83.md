### Title
Unconditional Zero-Amount Token Transfer in `transfer_to_pools_when_unstake` Can Permanently Freeze Staker STRK Funds — (`src/staking/staking.cairo`)

### Summary
`transfer_to_pools_when_unstake` unconditionally calls `checked_transfer` for every pool a staker has opened, including when the pool's delegated balance is zero. If the BTC token (or any future token added via `add_token`) reverts on zero-value transfers, `unstake_action` will always revert, permanently freezing the staker's own STRK stake inside the contract.

### Finding Description
In `src/staking/staking.cairo`, the internal function `transfer_to_pools_when_unstake` iterates over all pools associated with a staker and transfers each pool's delegated balance back to the pool contract:

```cairo
for (pool_contract, token_address) in staker_pool_info.pools {
    let pool_balance = self.get_delegated_balance(:staker_address, :pool_contract);
    let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
    ...
    token_dispatcher
        .checked_transfer(
            recipient: pool_contract,
            amount: pool_balance.to_native_amount(:decimals).into(),
        );
}
``` [1](#0-0) 

There is no guard of the form `if pool_balance.is_non_zero()` before the transfer. If `pool_balance` is zero (e.g., a BTC pool that was opened but never received any delegations, or whose delegators all exited before the staker's `unstake_action`), the call becomes `checked_transfer(pool_contract, 0)`.

This function is called unconditionally from `unstake_action`:

```cairo
self.transfer_to_pools_when_unstake(
    :staker_address, staker_pool_info: staker_pool_info.as_non_mut(),
);
``` [2](#0-1) 

The protocol explicitly supports adding non-STRK tokens (e.g., BTC) via `add_token` / `set_open_for_delegation`: [3](#0-2) 

For these external tokens, zero-transfer behavior is not guaranteed. Some ERC20 implementations (including certain wrapped BTC variants) revert on zero-value transfers.

A parallel issue exists in `send_rewards_to_staker`, which calls `claim_from_reward_supplier` and then `checked_transfer` with `amount = staker_info.unclaimed_rewards_own` — which can be zero — but this path only uses STRK, which does not revert on zero transfers: [4](#0-3) 

Similarly, `Pool::claim_rewards` calls `checked_transfer` with a potentially-zero `rewards` value, but again only for STRK: [5](#0-4) 

The critical path is the BTC token in `transfer_to_pools_when_unstake`.

### Impact Explanation
If the BTC token reverts on zero-value transfers, any staker who has opened a BTC pool with zero delegated balance at the time of `unstake_action` will find that call permanently reverting. Their STRK own-stake is locked inside the staking contract with no recovery path, since `unstake_action` is the only mechanism to retrieve it after `unstake_intent` has been called.

This matches the **Temporary/Permanent freezing of funds** impact category (High).

### Likelihood Explanation
The scenario is reachable by any staker who:
1. Calls `set_open_for_delegation` for the BTC token (a normal, permissionless staker action).
2. Has zero BTC delegated balance at the time of `unstake_action` (e.g., opened the pool but no one delegated, or all delegators exited first).
3. The BTC token contract reverts on zero-value transfers.

Steps 1 and 2 are normal protocol usage. Step 3 depends on the specific BTC token deployed. Given that the protocol is designed to support arbitrary ERC20 tokens via governance, and some ERC20 implementations do revert on zero transfers, this is a realistic combination.

### Recommendation
Add a zero-amount guard before each transfer in `transfer_to_pools_when_unstake`:

```cairo
if pool_balance.is_non_zero() {
    token_dispatcher
        .checked_transfer(
            recipient: pool_contract,
            amount: pool_balance.to_native_amount(:decimals).into(),
        );
}
```

Apply the same pattern defensively to `send_rewards_to_staker` and `Pool::claim_rewards` for future-proofing against token changes.

### Proof of Concept
1. Governance adds BTC token (a zero-transfer-reverting ERC20) via `add_token`.
2. Staker calls `stake(...)` and then `set_open_for_delegation(btc_token_address)` — BTC pool is created with zero delegated balance.
3. No delegators join the BTC pool.
4. Staker calls `unstake_intent()` — succeeds, exit window starts.
5. After the exit window, staker calls `unstake_action(staker_address)`.
6. Internally, `transfer_to_pools_when_unstake` iterates pools, finds BTC pool with `pool_balance = 0`, and calls `btc_token.checked_transfer(btc_pool_contract, 0)`.
7. BTC token reverts on zero transfer → `unstake_action` reverts.
8. Staker's STRK stake is permanently locked; `unstake_action` will always revert for this staker.
