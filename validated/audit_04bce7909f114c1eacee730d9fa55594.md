### Title
WalletConnect Offer Approval Displays Underestimated Royalty Total When Multiple NFTs Are Involved - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in the WalletConnect command display pipeline incorrectly divides the spending amount by the number of NFTs before computing each NFT's royalty contribution. This causes the `amountWithRoyalties` shown in the WalletConnect approval dialog to be systematically lower than the amount that will actually be deducted from the user's wallet, leading users to approve transactions under false pretenses.

### Finding Description

In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` is called for every fungible asset line (XCH or CAT) when building the WalletConnect confirmation display for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`: [1](#0-0) 

The critical error is on line 363:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);  // BUG
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The code divides the full spending `amount` by the count of NFTs (`royaltyPercentages.length`) to get `splitAmount`, then applies each NFT's royalty percentage to that reduced `splitAmount`. The correct approach is to apply each NFT's royalty to the **full** `amount`.

**Buggy formula** (what the code does):
```
royaltyTotal = (amount / N) * sum(royaltyPct_i) / 10_000
```

**Correct formula**:
```
royaltyTotal = amount * sum(royaltyPct_i) / 10_000
```

**Concrete proof using the existing test case** (`parseCommandDisplay.test.ts` lines 251–335): [2](#0-1) 

- Spending: 100,000,000 mojos (0.0001 XCH)
- NFT 1 royalty: 500 (5%), NFT 2 royalty: 10 (0.1%)

| | Buggy (current) | Correct |
|---|---|---|
| splitAmount | 50,000,000 | N/A |
| royaltyAmount | 2,550,000 | 5,100,000 |
| `amountWithRoyalties` shown | **0.00010255 XCH** | **0.0001051 XCH** |

The test itself encodes the wrong expected value (`0.00010255`), confirming the bug is present and untested-against-correct-behavior. With 3 NFTs the royalty is divided by 3; with 10 NFTs by 10 — the underestimation grows linearly with NFT count.

The `withRoyaltyTotals` function passes the collected `royaltyPercentages` from all opposite-side NFTs into `formatAmountWithRoyalties` for each fungible line: [3](#0-2) 

This display result is then surfaced in the WalletConnect confirmation dialog via `walletDeltaToDisplay`: [4](#0-3) 

### Impact Explanation

The `amountWithRoyalties` field is the primary figure shown to the user in the WalletConnect approval dialog when they are about to sign a `take_offer` or `create_offer_for_ids` command. The actual on-chain transaction is constructed by the Chia wallet backend and correctly applies each NFT's full royalty to the full amount. The display, however, shows a lower number. A user who sees "you will spend 0.00010255 XCH" and approves will actually have more deducted. The discrepancy scales with both the number of NFTs and their royalty percentages — a malicious dApp offering 10 NFTs each with a 10% royalty would show the user only 1/10th of the true royalty cost.

This matches the **High** impact criterion: WalletConnect state causes a user to approve the wrong amount.

### Likelihood Explanation

Any unprivileged dApp connected via WalletConnect can craft a `take_offer` payload containing multiple NFTs with royalties. No special permissions beyond a standard WalletConnect pairing are required. The user must only have WalletConnect enabled and accept a multi-NFT offer — a normal user action.

### Recommendation

Remove the `splitAmount` division. Apply each NFT's royalty to the full `amount`:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Apply each NFT's royalty to the full amount, not a split amount
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

Update the test expectation for the multi-NFT case from `'0.00010255'` to `'0.0001051'`.

### Proof of Concept

1. A dApp connects via WalletConnect and sends a `chia_wallet.take_offer` request containing an offer where the user must spend 1 XCH (1,000,000,000,000 mojos) to receive 2 NFTs, each with a 10% royalty (royalty_percentage: 1000).
2. The GUI calls `parseCommandDisplay`, which calls `formatAmountWithRoyalties` with `amount = 1_000_000_000_000n` and `royaltyPercentages = [1000, 1000]`.
3. **Buggy display**: `splitAmount = 500_000_000_000`, `royaltyAmount = 100_000_000_000`, shown as **1.1 XCH**.
4. **Actual deduction**: `royaltyAmount = 200_000_000_000`, actual cost is **1.2 XCH**.
5. User approves thinking they spend 1.1 XCH; wallet deducts 1.2 XCH — a 0.1 XCH loss per such approval.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
