### Title
`ReplaceabilityComponent` Initializer Not Called in `MintingCurve` and `RewardSupplier` Constructors - (File: `src/minting_curve/minting_curve.cairo`, `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

`MintingCurve` and `RewardSupplier` both include `ReplaceabilityComponent` and publicly expose its upgrade interface via `ReplaceabilityImpl`, but their constructors never call `self.replaceability.initialize()`. Every other contract in the protocol (`Staking`, `Pool`, `Attestation`) correctly calls this initializer. The omission leaves the replaceability subsystem in an uninitialized state, permanently disabling the upgrade mechanism for these two critical contracts.

---

### Finding Description

Both `MintingCurve` and `RewardSupplier` declare `ReplaceabilityComponent` and embed `ReplaceabilityImpl`:

`MintingCurve` component declaration and ABI embedding: [1](#0-0) 

`MintingCurve` constructor — `replaceability.initialize()` is absent: [2](#0-1) 

`RewardSupplier` component declaration and ABI embedding: [3](#0-2) 

`RewardSupplier` constructor — `replaceability.initialize()` is absent: [4](#0-3) 

Contrast with every other contract in the protocol, which correctly calls the initializer:

`Staking` constructor calls `self.replaceability.initialize(upgrade_delay: Zero::zero())`: [5](#0-4) 

`Pool` constructor calls `self.replaceability.initialize(upgrade_delay: Zero::zero())`: [6](#0-5) 

`Attestation` constructor calls `self.replaceability.initialize(upgrade_delay: Zero::zero())`: [7](#0-6) 

A secondary confirming signal: `Staking`, `Pool`, and `Attestation` all import `InternalReplaceabilityTrait` (the trait that exposes `initialize`): [8](#0-7) [9](#0-8) [10](#0-9) 

`MintingCurve` and `RewardSupplier` do **not** import `InternalReplaceabilityTrait`, confirming the `initialize` call was never added to either contract.

`ReplaceabilityComponent::initialize` (part of `InternalReplaceabilityTrait`) sets up the component's internal state — including `upgrade_delay` and, consistent with the Starkware replaceability pattern, an `initialized` flag. Without this call, `add_new_implementation` and `replace_to` will either revert (if they assert the `initialized` flag) or operate with an unset internal state, permanently breaking the upgrade path for both contracts.

---

### Impact Explanation

`RewardSupplier` is the sole contract responsible for holding and distributing all STRK staking rewards. It manages `unclaimed_rewards`, handles `claim_rewards` calls from the staking contract, and sends L1 mint requests via `send_mint_request_to_l1_reward_supplier`. If a bug is discovered in any of these flows and the upgrade mechanism is broken, the protocol cannot deploy a fix. The result is **permanent freezing of unclaimed yield** for all stakers and delegators.

`MintingCurve` controls the `yearly_mint` inflation calculation that determines total reward issuance. A broken upgrade path means any miscalculation in `compute_yearly_mint` or a corrupted `c_num` cannot be corrected, risking **protocol insolvency**.

**Impact: High — Permanent freezing of unclaimed yield / Protocol insolvency.**

---

### Likelihood Explanation

The contracts are already deployed with this omission baked into their constructors. The protocol has a demonstrated history of requiring upgrades (V1→V2→V3 for both `RewardSupplier` and `MintingCurve`), making future upgrade needs highly probable. The missing `initialize` call cannot be corrected post-deployment without a full redeployment and migration of state, which is itself blocked by the broken replaceability mechanism.

---

### Recommendation

1. Import `InternalReplaceabilityTrait` in both `MintingCurve` and `RewardSupplier`.
2. Add `self.replaceability.initialize(upgrade_delay: Zero::zero());` to both constructors, following the pattern used by `Staking`, `Pool`, and `Attestation`.

---

### Proof of Concept

1. Deploy `MintingCurve` or `RewardSupplier` with the current constructor (no `replaceability.initialize()` call).
2. As the upgrade governor, call `add_new_implementation` with a valid `ImplementationData` struct.
3. The call reverts because `ReplaceabilityComponent` was never initialized — its internal state (including any `initialized` flag) remains at the Cairo default of zero/false.
4. `replace_to` is equally unreachable.
5. The upgrade governor is permanently unable to upgrade either contract.

For comparison, the same sequence on `Staking` (which correctly calls `replaceability.initialize()`) succeeds, as exercised throughout the protocol's upgrade test suite. [2](#0-1) [4](#0-3)
