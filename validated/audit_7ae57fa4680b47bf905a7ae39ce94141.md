### Title
Global `last_reward_block` Allows Any Staker to Permanently Block Consensus Reward Distribution for All Other Stakers — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function uses a single **global** `last_reward_block` variable to gate reward distribution. Because this variable is shared across all stakers and the function has no access control, any staker can call it at every block, setting `last_reward_block` to the current block and making it impossible for any other staker to receive consensus rewards in that block. Repeated across every block, this permanently freezes unclaimed yield for all other stakers.

---

### Finding Description

`update_rewards` is a public function in `IStakingRewardsManager` with no caller restriction beyond `general_prerequisites()` (which only checks for non-zero caller and unpaused state): [1](#0-0) 

The gate that prevents double-rewarding is a **global** storage variable: [2](#0-1) 

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

Immediately after the check, the global variable is updated: [3](#0-2) 

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);
```

The storage declaration confirms this is a single global slot, not a per-staker mapping: [4](#0-3) 

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
```

Because `staker_address` is a caller-supplied parameter (not derived from `get_caller_address()`), any account can invoke `update_rewards(attacker_staker_address, false)`. Once called at block N, `last_reward_block == N`, and every other staker's call at block N reverts with `REWARDS_ALREADY_UPDATED`. The attacker repeats this at block N+1, N+2, … indefinitely.

The rewards actually distributed per call are computed as: [5](#0-4) 

So the attacker receives their own consensus block rewards on every block while all other stakers receive nothing.

The `disable_rewards` flag makes the attack even cheaper: calling with `disable_rewards = true` still writes `last_reward_block` but skips the reward transfer entirely, enabling pure griefing at minimal cost: [6](#0-5) 

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
```

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield / Theft of unclaimed yield.**

An attacker who calls `update_rewards(attacker_staker, false)` at every block:
- Receives their own proportional consensus rewards for every block.
- Prevents every other staker from ever successfully calling `update_rewards`, permanently freezing their unclaimed consensus-phase yield.

The attacker need not hold a large stake; even a minimal staker can execute this attack. The only cost is the gas for one transaction per block.

---

### Likelihood Explanation

**High.** The entry point is fully public, requires no special role, no minimum stake beyond the protocol minimum, and no external conditions. The attack is deterministic and can be automated with a simple bot. There is no on-chain mechanism to prevent or penalise it.

---

### Recommendation

Replace the global `last_reward_block` with a **per-staker** mapping:

```cairo
// Storage
last_reward_block: Map<ContractAddress, BlockNumber>,
```

Update the check and write in `update_rewards`:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(staker_address),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
// ...
self.last_reward_block.write(staker_address, current_block_number);
```

This ensures each staker can be rewarded at most once per block independently of all other stakers.

---

### Proof of Concept

1. Attacker stakes the minimum amount, obtaining `attacker_staker` address.
2. At every block N (consensus rewards phase), attacker submits `update_rewards(attacker_staker, false)`.
3. `last_reward_block` is set to N; attacker receives their block rewards.
4. Legitimate staker calls `update_rewards(legitimate_staker, false)` in the same block N → reverts: `REWARDS_ALREADY_UPDATED`.
5. Attacker repeats at N+1, N+2, … — legitimate stakers accumulate zero consensus rewards indefinitely.

**Analogy to the reference report:** Just as the Nouns DAO attacker front-ran legitimate proposal creation with a bogus proposal to set the `lastProposal` anchor and capture a larger auction-revenue window, the Starknet Staking attacker front-runs every legitimate `update_rewards` call by writing the global `last_reward_block` anchor first — redirecting the block's reward to themselves and locking out all other stakers.
