### Title
Off-by-One in `SliceVec::resize` Causes Valid EVM Heap Expansions to Fail as OutOfGas - (`File: zk_ee/src/memory/slice_vec.rs`)

### Summary

`SliceVec::resize` uses a strict `>=` guard against the backing memory length, which incorrectly rejects a resize to exactly the backing capacity. In the EVM interpreter, this error is silently mapped to `OutOfGas`, causing a valid transaction to be rejected with a wrong error code — a state-transition / valid-execution unprovability bug.

### Finding Description

`SliceVec<T>` is a fixed-capacity vector backed by a pre-allocated `&mut [MaybeUninit<T>]` slice. Its `resize` method contains the following guard:

```rust
pub fn resize(&mut self, new_length: usize, padding: T) -> Result<(), ()> {
    if new_length >= self.memory.len() {   // ← off-by-one: should be `>`
        return Err(());
    }
    ...
    self.length = new_length;
    Ok(())
}
``` [1](#0-0) 

When `new_length == self.memory.len()`, the condition is `true` and the function returns `Err(())`. However, `new_length == self.memory.len()` is a perfectly valid request: it means "use every slot in the backing allocation." The correct guard is `new_length > self.memory.len()`.

The EVM heap is a `SliceVec<'a, u8>`. Every memory-touching opcode (`MLOAD`, `MSTORE`, `CALLDATACOPY`, `CODECOPY`, `RETURNDATACOPY`, etc.) calls `resize_heap_implementation`, which calls `heap.resize(new_heap_size, 0)` and maps any `Err` to `EvmError::OutOfGas`:

```rust
heap.resize(new_heap_size, 0)
    .map_err(|_| ExitCode::EvmError(EvmError::OutOfGas))?;
``` [2](#0-1) 

`new_heap_size` is computed as `max_offset.next_multiple_of(32)`: [3](#0-2) 

Whenever `new_heap_size` equals exactly the backing memory capacity of the heap `SliceVec`, the resize is rejected and the transaction aborts with `OutOfGas` — even though the transaction had sufficient gas and the access was within the EVM memory limit (`u32::MAX - 31`).

### Impact Explanation

A transaction whose EVM heap expansion lands exactly on the backing-memory boundary is incorrectly terminated with `OutOfGas`. Because the error is indistinguishable from a genuine gas exhaustion, the transaction is reverted. Any ETH or token transfer, contract deployment, or state change inside that transaction is lost. The user pays the full gas fee for a transaction that the protocol should have executed successfully. This is a **valid-execution unprovability / state-transition bug**: the forward execution produces a revert that the prover cannot justify from the EVM specification.

### Likelihood Explanation

The EVM heap backing memory is pre-allocated with a fixed capacity. The `new_heap_size` is always a multiple of 32. Whenever the system allocates the heap backing buffer to a size that is itself a multiple of 32 (the common case), any transaction that drives the heap to exactly that size triggers the bug. An attacker who knows the backing capacity can craft a contract that performs a single memory access at `offset = capacity - 32`, forcing `new_heap_size = capacity` and reliably triggering the incorrect `OutOfGas`.

### Recommendation

Change the guard in `SliceVec::resize` from `>=` to `>`:

```rust
// Before (off-by-one):
if new_length >= self.memory.len() {
    return Err(());
}

// After (correct):
if new_length > self.memory.len() {
    return Err(());
}
``` [1](#0-0) 

This allows the `SliceVec` to be resized to its full backing capacity, which is the semantically correct behaviour.

### Proof of Concept

1. Deploy a contract whose constructor or fallback executes a single `MSTORE` at offset `N - 32`, where `N` is the EVM heap backing-memory capacity (a multiple of 32).
2. `resize_heap_implementation` computes `new_heap_size = (N - 32 + 32).next_multiple_of(32) = N`.
3. `heap.resize(N, 0)` is called; the guard `N >= N` is `true`; `Err(())` is returned.
4. The error is mapped to `EvmError::OutOfGas`; the transaction reverts.
5. The same transaction on a reference EVM (Geth/REVM) succeeds, demonstrating the divergence.
