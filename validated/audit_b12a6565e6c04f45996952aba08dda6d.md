Audit Report

## Title
Missing `GlobalConcurrencyLimitLayer` on Synchronous Call Endpoints Allows Unbounded Shared Resource Contention - (File: `rs/http_endpoints/public/src/lib.rs`)

## Summary
The IC replica's HTTP server applies `GlobalConcurrencyLimitLayer` to `/api/v2/canister/.../call` but explicitly omits it for the synchronous call endpoints `/api/v3/canister/.../call`, `/api/v4/canister/.../call`, and `/api/v4/subnet/.../call`. A developer TODO comment at line 606 acknowledges this gap. Without a concurrency cap, an attacker can hold arbitrarily many concurrent requests in-flight on these endpoints, exhausting the Tokio blocking thread pool via unbounded `spawn_blocking` calls for cryptographic signature verification and contending on the shared `ingress_filter` mutex, degrading the node's ability to process legitimate ingress messages.

## Finding Description
In `make_router()` in `rs/http_endpoints/public/src/lib.rs`, `call_v2_router` is wrapped with `service_builder(GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests))` (default: 50), which includes `HandleErrorLayer`, `load_shed()`, and the concurrency cap. The three synchronous routers are merged with no such layer:

```rust
// L603-609
.merge(http_handler.call_v2_router.layer(service_builder(
    GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests),
)))
// TODO(CON-1574): see if there is any reasonable explicit concurrency limit we could use here.
.merge(http_handler.call_v3_router)
.merge(http_handler.call_v4_router)
.merge(http_handler.subnet_call_v4_router)
```

Every request reaching these routers calls `IngressValidator::validate_ingress_message()`, which:

1. Acquires a read-lock on the shared `ingress_throttler: Arc<RwLock<dyn IngressPoolThrottler>>` (line 230) — this check only rejects when the ingress pool is already full, not when concurrent HTTP requests are excessive.
2. Calls `tokio::task::spawn_blocking` for cryptographic signature verification (line 327) — each concurrent request consumes one thread from Tokio's blocking thread pool (default cap: 512).
3. Acquires and clones the shared `ingress_filter: Arc<Mutex<IngressFilterService>>` (line 343) — the mutex is held briefly for a clone, but under high concurrency this serializes all callers.

The `ingress_filter` is the same `Arc<Mutex<...>>` instance shared across all call variants, constructed once at line 290. The existing guards (`ingress_throttler` threshold check, `ingress_tx.capacity()` check, global `load_shed()`, and `request_timeout_seconds`) do not bound the number of *concurrently in-flight* requests: the throttler check fires only after the pool fills from successfully submitted messages, the channel capacity check fires only after the downstream channel fills, and the global load-shedder has no per-endpoint concurrency counter.

An attacker sends N > `max_call_concurrent_requests` (e.g., N = 200) concurrent well-formed POST requests to `/api/v3/canister/{id}/call`. Each request passes body-size and content-type checks, then enters `validate_ingress_message`. Each spawns a `spawn_blocking` task for crypto verification. With N = 512+, the Tokio blocking thread pool saturates, stalling all `spawn_blocking` work across the process (including other replica subsystems). Even below that threshold, the serialized `ingress_filter` mutex clone and the registry lookups under high concurrency degrade throughput for legitimate callers on all call variants, since the `ingress_filter` is shared.

## Impact Explanation
This is a node-level availability impact: an unprivileged attacker can degrade or stall a replica's ingress processing by exploiting the absent concurrency bound on the primary modern call endpoints. This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** The attack is not raw volumetric — it exploits a specific, developer-acknowledged code gap (missing concurrency limit) to exhaust a bounded shared resource (Tokio blocking thread pool, `ingress_filter` mutex) with a moderate, targeted number of concurrent requests that would be rejected on the v2 endpoint.

## Likelihood Explanation
The `/api/v3` and `/api/v4` call endpoints are the primary endpoints used by `ic-agent` and all current IC tooling. They are publicly reachable through boundary nodes. The attack requires only the ability to send concurrent HTTP POST requests with a valid CBOR content-type header and any CBOR-encoded body — no special privileges, no valid identity, no subnet knowledge beyond the target canister ID. The developer TODO at line 606 confirms this is a known, unresolved gap rather than an intentional design choice. The default `max_call_concurrent_requests` is 50; sending 51+ concurrent requests to v3/v4 immediately exceeds the protection level that v2 enforces.

## Recommendation
Apply `GlobalConcurrencyLimitLayer::new(config.max_call_concurrent_requests)` (with `HandleErrorLayer` and `load_shed()` via the existing `service_builder` closure) to `call_v3_router`, `call_v4_router`, and `subnet_call_v4_router` in `make_router()`, consistent with how `call_v2_router` is protected. This directly resolves the acknowledged TODO at line 606. If a different limit is appropriate for synchronous endpoints (due to their blocking-wait semantics), a separate config field (e.g., `max_sync_call_concurrent_requests`) should be introduced and applied to all three routers.

## Proof of Concept
1. Start a local replica or use a test node with `max_call_concurrent_requests: 1` in the HTTP handler config.
2. Send 2 concurrent POST requests to `/api/v2/canister/aaaaa-aa/call` with a valid CBOR body and `Content-Type: application/cbor`. Observe the second returns `429 Too Many Requests`.
3. Send 2 concurrent POST requests to `/api/v3/canister/aaaaa-aa/call` with the same body. Observe both are accepted and processed concurrently — no 429 is returned.
4. Scale to N = 512 concurrent requests to `/api/v3/.../call`. Observe Tokio blocking thread pool saturation: `spawn_blocking` calls from other subsystems begin queuing, and node metrics show blocking thread exhaustion.
5. Alternatively, write an integration test mirroring `test_load_shedding_update_call` in `rs/http_endpoints/public/tests/load_shed_test.rs` (which tests v2 load shedding) but targeting `Call::V3` — assert that the second concurrent request returns `429`. This test will fail against the current code, confirming the vulnerability.
