### Title
Missing L2 Sequencer Uptime Check in `PriceProviderL2` and `ProtectedPriceProviderL2` Allows Stale-Price Swaps After Sequencer Restart — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are the L2-specific price providers used by Metric OMM pools deployed on Base and other L2s. Both contracts rely exclusively on a `_isStale` timestamp comparison against `MAX_TIME_DELTA` for oracle freshness. Neither contract checks a Chainlink sequencer uptime feed before serving prices. After an L2 sequencer outage and restart, a grace-period window exists where the last pre-downtime oracle report is still within `MAX_TIME_DELTA` but reflects a price that is materially stale relative to actual market conditions. Swaps execute against this stale price, allowing traders to drain LP funds.

---

### Finding Description

Both L2 price providers call `IPricedOracle.price(feedId, pool)` to retrieve oracle data and then apply a single staleness gate:

```solidity
// PriceProviderL2.sol lines 214-217
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
```

`_isStale` only checks whether `block.timestamp - refTime > MAX_TIME_DELTA`. It does not consult a sequencer uptime feed. [1](#0-0) 

When the L2 sequencer goes offline, Chainlink Data Streams keepers cannot submit new reports (they are L2 transactions). The stored `oracleData[feedId]` freezes at the last pre-downtime value. Once the sequencer restarts, keepers begin pushing fresh reports, but there is a window — bounded by `MAX_TIME_DELTA` — during which the stale pre-downtime report is still accepted as fresh. During this window, `getBidAndAskPrice()` returns bid/ask prices derived from the frozen, pre-downtime oracle value. [2](#0-1) 

The same gap exists identically in `ProtectedPriceProviderL2._computeBidAsk()`: [3](#0-2) 

Notably, the codebase already contains a `ChainlinkVerifierL2` artifact in the contract registry that exposes `sequencerUptimeFeed()` and `GRACE_PERIOD` — the exact Chainlink-recommended pattern — but this contract is not wired into either `PriceProviderL2` or `ProtectedPriceProviderL2`. [4](#0-3) 

`MAX_TIME_DELTA` is set at construction and can be up to 7 days, but even a short value (e.g., 60 seconds, as suggested by deployment scripts) creates a meaningful exploitation window immediately after sequencer restart. [5](#0-4) 

---

### Impact Explanation

If the price of a pool's base token moves significantly during sequencer downtime (e.g., ETH drops 5%), the stale pre-downtime price is still served as valid. A trader can immediately swap against the pool at the inflated stale ask price, receiving more tokens than the current market value warrants. The LP bears the loss: their reserves are depleted at an off-market rate. This is a direct loss of LP principal, satisfying the "Critical/High/Medium direct loss of user principal" impact gate.

---

### Likelihood Explanation

Base (a primary deployment target) has experienced sequencer outages. The attack requires no special privilege — any EOA can call `swap()` on a pool immediately after sequencer restart. The attacker only needs to monitor the sequencer status feed and have a transaction ready. The window is bounded by `MAX_TIME_DELTA`, but even a 10-second window is exploitable by a bot. [6](#0-5) 

---

### Recommendation

Add a sequencer uptime check to `_getBidAndAskPrice()` in both `PriceProviderL2` and `ProtectedPriceProviderL2`. Wire in the `sequencerUptimeFeed` (already modeled in `ChainlinkVerifierL2`) and enforce a `GRACE_PERIOD` (e.g., 3600 seconds) after sequencer restart before allowing price reads:

```solidity
(, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
// answer == 0 means sequencer is up
if (answer != 0) revert SequencerDown();
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

This check should execute before the staleness check in `_getBidAndAskPrice()` / `_computeBidAsk()`.

---

### Proof of Concept

1. ETH/USDC pool on Base uses `PriceProviderL2` with `MAX_TIME_DELTA = 60`.
2. ETH price is $3,000. Sequencer goes offline. ETH price drops to $2,700 on CEX.
3. Sequencer restarts. The last stored oracle report (price = $3,000, `refTime = T`) is 30 seconds old — within `MAX_TIME_DELTA`.
4. `_isStale(T, block.timestamp, 60, ...)` returns `false`. `getBidAndAskPrice()` returns bid/ask derived from $3,000.
5. Attacker calls `swap()` on the pool, buying ETH at the stale $3,000 ask price.
6. Attacker immediately sells ETH on CEX at $2,700 — net loss to LPs is $300 per ETH swapped.
7. No sequencer uptime check exists to block this; the pool executes the swap normally. [7](#0-6) [8](#0-7)

```text
    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }
```
