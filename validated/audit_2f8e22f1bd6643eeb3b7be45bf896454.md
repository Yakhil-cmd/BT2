### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield — (`src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in `src/pool/pool.cairo` contains an unbounded loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace since their last claim. The developers themselves acknowledge this in a code comment. A pool member who makes many balance changes across different epochs without claiming rewards will grow this trace without bound. When they eventually call `claim_rewards`, the transaction can exhaust the Starknet gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) is called by the public `claim_rewards` function. It loops over every entry in the pool member's `pool_member_epoch_balance` trace that has not yet been processed:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
```

The trace grows by one entry per epoch in which the pool member modifies their balance. Every call to `enter_delegation_pool`, `add_to_delegation_pool`, `exit_delegation_pool_intent`, or `enter_delegation_pool_from_staking_contract` invokes `set_member_balance` or `increase_member_balance`, which appends a new checkpoint at `current_epoch + K`:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(
        balance: amount,
        cumulative_rewards_trace_idx: self.cumulative_rewards_trace_length() + 1,
    );
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
```

The `entry_to_claim_from` cursor is stored in `pool_member_info` and advanced after each successful `claim_rewards` call. If a pool member never claims, the cursor stays at zero and the loop must traverse the entire trace on the next claim. If the trace has grown to N entries (one per epoch of balance change), the loop executes N iterations, each performing a `find_sigma` lookup into the `cumulative_rewards_trace` storage vector.

---

### Impact Explanation

If the loop exceeds the Starknet transaction gas limit, `claim_rewards` reverts. Because the cursor is only advanced on a successful claim, every subsequent attempt also reverts. The pool member's accumulated STRK rewards are permanently locked in the pool contract with no recovery path. This matches the allowed impact: **permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

Any pool member who actively adjusts their delegation (e.g., calling `add_to_delegation_pool` or `exit_delegation_pool_intent` in many different epochs) without periodically calling `claim_rewards` will grow their trace. This is a realistic usage pattern for delegators who rebalance their stake across epochs but defer reward collection. Additionally, the pool member's `reward_address` is a separate account that is also authorized to call `add_to_delegation_pool` (line 229), meaning a distinct party can grow the trace on behalf of the pool member. The developers themselves flagged this risk in the comment at line 858.

---

### Recommendation

1. **Paginate claims**: Accept an optional `max_entries` parameter in `claim_rewards` so the caller can process the trace in bounded chunks, advancing `entry_to_claim_from` incrementally across multiple transactions.
2. **Enforce a claim cadence**: Require or incentivize pool members to claim before making additional balance changes, preventing unbounded trace growth.
3. **Cap trace growth**: Merge consecutive balance changes within the same epoch into a single entry (the `trace.insert` at `get_epoch_plus_k()` already does this for the same target epoch, but multiple changes across different epochs still accumulate).

---

### Proof of Concept

1. Alice enters a delegation pool and never calls `claim_rewards`.
2. Alice calls `add_to_delegation_pool` once per epoch for N epochs (each call appends one entry to `pool_member_epoch_balance`).
3. After N epochs, Alice calls `claim_rewards`. The `calculate_rewards` loop must iterate N times, each iteration reading from `pool_member_epoch_balance` and calling `find_sigma` into `cumulative_rewards_trace`.
4. For sufficiently large N, the transaction exhausts the Starknet gas limit and reverts.
5. All subsequent `claim_rewards` calls also revert because `entry_to_claim_from` was never advanced. Alice's rewards are permanently frozen.

The exact N required depends on Starknet's per-transaction gas ceiling and the cost of each `find_sigma` storage read, but the code comment at line 858 explicitly acknowledges the loop is unbounded. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.

            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
