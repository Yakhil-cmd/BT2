### Title
Invoke Signature Validation Bypassed via Overly Broad `account_tx_in_pool_or_recent_block` Check in `skip_stateful_validations` - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call for an Invoke transaction with `nonce=1` when the on-chain account nonce is `0`, to support the UX pattern of submitting `deploy_account + invoke` simultaneously. However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the sender address in the mempool, not exclusively a `deploy_account` transaction. An attacker who observes any pending transaction from a target account (e.g., the `deploy_account` itself, or a nonce-0 invoke from a freshly deployed account) can submit a forged `invoke` with `nonce=1` and an invalid signature. The gateway admits this transaction to the mempool without ever calling `__validate__`, violating the invariant that every admitted invoke transaction must pass account-controlled signature verification.

### Finding Description

**Step 1 — Nonce and resource-bounds checks pass.**

`validate_state_preconditions` calls `validate_nonce`. For an Invoke with `tx_nonce=1` and `account_nonce=0`, the general branch applies:

```
account_nonce(0) <= tx_nonce(1) <= max_allowed_nonce
```

This passes. [1](#0-0) 

**Step 2 — Mempool nonce check passes.**

`validate_by_mempool` calls `mempool.validate_tx`. The mempool's `validate_incoming_tx` resolves the account nonce via `resolve_nonce`. If the target account has a nonce-0 transaction in the pool (not yet staged), the resolved nonce is `0` (falling back to `incoming_account_nonce=0`). Since `tx_nonce(1) >= resolved_nonce(0)`, no `NonceTooOld` error is raised. [2](#0-1) 

**Step 3 — `skip_stateful_validations` returns `true` (skip) based on an overly broad check.**

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
```

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in `state.staged`, `state.committed`, or `tx_pool`. It does **not** verify that the pooled transaction is a `deploy_account`. [3](#0-2) 

`account_tx_in_pool_or_recent_block` implementation: [4](#0-3) 

**Step 4 — `__validate__` is skipped entirely.**

`run_validate_entry_point` sets `execution_flags.validate = !skip_validate = false` and calls `blockifier_validator.perform_validations`. Inside `perform_validations`, for an Invoke transaction:

```rust
if !tx.execution_flags.validate {
    return Ok(());   // returns before __validate__ is called
}
```

The account's `__validate__` entry point — which performs signature verification — is never executed. [5](#0-4) [6](#0-5) 

### Impact Explanation

The gateway admits an Invoke transaction with an invalid or forged signature to the mempool without any account-level authorization check. This directly satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

Concrete consequences:
- A forged nonce-1 invoke occupies the victim's nonce-1 slot in the mempool.
- If fee escalation is enabled, the attacker can replace the victim's legitimate nonce-1 invoke with the forged one by paying a higher fee.
- The forged transaction will be included in a block proposal, fail at execution time (when `__validate__` runs in the blockifier), and waste block space.
- The victim's nonce is incremented by the failed transaction, forcing them to resubmit with nonce-2 and losing the original nonce-1 transaction.

### Likelihood Explanation

The preconditions are reachable by any unprivileged user:
1. The target account must have `nonce=0` in on-chain state — this is the normal state for any account that has just been deployed or is in the process of being deployed.
2. Any transaction from the target address must be in the mempool — the attacker can observe the mempool (e.g., the victim's own `deploy_account` transaction) to satisfy this condition.

The `deploy_account + invoke` UX pattern is explicitly documented and encouraged, making condition 2 routinely satisfied during normal user flows.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address)` that only returns `true` when a `DeployAccount` transaction (not any transaction) is present for the given address.

Alternatively, the skip condition should be tightened to require that the account contract does not yet exist on-chain (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`), which is the true invariant that makes skipping `__validate__` safe.

### Proof of Concept

**State before attack:**
- Account `A` is deployed on-chain; `nonce_A = 0` in state (never sent a transaction).
- Alice submits `invoke(nonce=0, calldata=X, sig=valid_sig)` — this passes all checks including `__validate__` and enters the mempool.

**Attack:**
1. Attacker observes Alice's nonce-0 invoke in the mempool.
2. Attacker submits `invoke(nonce=1, calldata=DRAIN_FUNDS, sig=0xdeadbeef)` from address `A`.

**Gateway processing of attacker's transaction:**
- `validate_nonce`: `0 <= 1 <= max_allowed_nonce` → passes. [1](#0-0) 
- `validate_by_mempool`: `tx_nonce(1) >= resolved_nonce(0)` → passes. [2](#0-1) 
- `skip_stateful_validations`: `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` (Alice's nonce-0 invoke is in pool) → returns `true`. [7](#0-6) 
- `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate=false`, `__validate__` is never called. [8](#0-7) 

**Result:** The forged `invoke(nonce=1, sig=0xdeadbeef)` is admitted to the mempool. If fee escalation is enabled and the attacker pays a higher fee than Alice's legitimate nonce-1 transaction, Alice's transaction is replaced. The forged transaction is included in a block, fails at execution (when `__validate__` runs in the blockifier), and Alice's nonce-1 slot is permanently consumed.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
