### Title
Commission Change Without Reward Checkpoint Allows Retroactive Epoch Reward Theft From Delegators - (File: src/staking/staking.cairo)

### Summary

In the V2 (attestation-based) reward mode, a staker holding an active `CommissionCommitment` can increase their commission to `max_commission` immediately before submitting their epoch attestation. Because `calculate_staker_pools_rewards` reads the **current** commission from storage at attestation time and applies it to the **entire epoch's** pool rewards, delegators who participated throughout the epoch under a lower commission rate receive zero (or reduced) rewards for that epoch. The staker pockets the difference as commission.

### Finding Description

Reward calculation for pool members is triggered once per epoch when the staker calls `attest()` on the Attestation contract, which in turn calls `update_rewards_from_attestation_contract` on the Staking contract. Inside `_update_rewards`, the commission split is performed by `calculate_staker_pools_rewards`:

```cairo
// src/staking/staking.cairo ~line 1964
let commission = staker_pool_info.commission();
``` [1](#0-0) 

This reads the **live** commission value from storage at the moment of attestation and applies it to the entire epoch's pool rewards. There is no per-epoch commission snapshot, no checkpoint written when commission changes, and no delay between a commission change and its effect on reward splits.

`set_commission` / `update_commission` writes the new value directly to storage with no epoch delay:

```cairo
// src/staking/staking.cairo ~line 1599-1600
staker_pool_info.commission.write(Option::Some(commission));
``` [2](#0-1) 

Normally, commission can only decrease. However, `set_commission_commitment` allows a staker to set a `max_commission` ceiling, after which `update_commission` permits raising commission to any value ≤ `max_commission`:

```cairo
// src/staking/staking.cairo ~line 1583-1589
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
}
``` [3](#0-2) 

**Attack sequence (V2 attestation mode):**

1. Staker sets commission to `0%` and calls `set_commission_commitment` with `max_commission = 10000` (100%).
2. Delegators join the pool, attracted by the 0% commission rate.
3. The epoch progresses; delegators accumulate entitlement to pool rewards.
4. Right before the attestation window closes, the staker calls `set_commission(10000)` — raising commission to 100%.
5. Staker attests; `update_rewards_from_attestation_contract` fires, reads commission = 100%, and routes **all** pool rewards to the staker as commission.
6. Delegators receive zero rewards for the entire epoch.
7. Staker calls `set_commission(0)` again (allowed because the commitment is still active and `0 != 10000`), restoring the attractive rate for the next epoch.
8. Repeat indefinitely.

The pool contract's `update_rewards_from_staking_contract` only receives the already-split `pool_rewards` amount (post-commission), so the pool's `cumulative_rewards_trace` records zero growth for that epoch: [4](#0-3) 

### Impact Explanation

Delegators who staked for an entire epoch under an advertised 0% commission receive **zero yield** for that epoch. The staker captures 100% of the pool's share of epoch rewards as commission. This is a direct, repeatable theft of delegators' unclaimed yield. Impact: **High — Theft of unclaimed yield**.

### Likelihood Explanation

Any staker who (a) has a pool and (b) holds an active `CommissionCommitment` with `max_commission > current_commission` can execute this attack. Setting up a commitment is a permissionless staker action. The attack requires no privileged access, no external dependencies, and can be repeated every epoch. Likelihood: **High**.

### Recommendation

Record a per-epoch commission snapshot at the time rewards are calculated, or introduce an epoch delay before a commission increase takes effect (analogous to the `K`-epoch delay already used for balance changes). Concretely, when `set_commission` increases the commission (within a commitment), the new value should only become effective starting from epoch `current_epoch + K`, mirroring the balance-trace pattern used for stake changes:

```cairo
// Analogous to insert_staker_own_balance pattern
staker_pool_info.commission_trace.insert(
    key: self.get_epoch_plus_k(), value: Option::Some(commission)
);
```

`calculate_staker_pools_rewards` would then query `commission_at_epoch(curr_epoch)` instead of the live value.

### Proof of Concept

```
Epoch N begins.
  Staker commission = 0%.  CommissionCommitment: max=100%, expires epoch N+10.
  Delegator A delegates 1000 STRK.  Pool balance = 1000 STRK.
  Staker own balance = 1000 STRK.  Total = 2000 STRK.

Block B (last block of attestation window, epoch N):
  Staker calls set_commission(10000).   // commission → 100%, immediate storage write
  Staker calls attest(block_hash).
    → attestation contract calls update_rewards_from_attestation_contract(staker)
    → calculate_staker_pools_rewards reads commission = 10000
    → pool_rewards_including_commission = epoch_rewards * 1000/2000
    → commission_rewards = pool_rewards_including_commission * 10000/10000 = 100%
    → pool_rewards forwarded to pool contract = 0
  Delegator A's cumulative_rewards_trace gains 0 for epoch N.

Staker calls set_commission(0).   // back to 0% for next epoch

Epoch N+1 begins.  Delegator B joins, attracted by 0% commission.  Repeat.
```

Delegator A calls `claim_rewards` after epoch N and receives **0 STRK** despite having been the sole pool member for the entire epoch. [5](#0-4) [6](#0-5) [7](#0-6)

        fn update_commission(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
            old_commission: Commission,
            commission: Commission,
        ) {
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                if self.is_commission_commitment_active(:commission_commitment) {
                    assert!(
                        commission <= commission_commitment.max_commission,
                        "{}",
                        Error::INVALID_COMMISSION_WITH_COMMITMENT,
                    );
                    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
                } else {
                    assert!(
                        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
                    );
                }
            } else {
                assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
            }

            // Update commission in storage.
            staker_pool_info.commission.write(Option::Some(commission));

            // Emit event.
            self
                .emit(
                    Events::CommissionChanged {
                        staker_address, old_commission, new_commission: commission,
                    },
                );
