### Title
Unbounded `pool_member_epoch_balance` Trace Iteration in `calculate_rewards` Permanently Freezes Delegator Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract iterates over every entry in a delegator's `pool_member_epoch_balance` trace in a single, unbounded loop. Because there is no cap on how many trace entries a delegator can accumulate (one new entry is appended per epoch in which a balance change occurs), a delegator who makes frequent balance changes across many epochs without claiming will eventually find that `claim_rewards` permanently reverts out-of-gas, freezing all accrued yield with no recovery path.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function is called by both `claim_rewards` and `pool_member_info_v1`. It iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` up to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` implementation only deduplicates when the incoming key equals the last stored key; otherwise it appends a new checkpoint:

```cairo
} else {
    // Checkpoint keys must be non-decreasing.
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a **different epoch** appends one new entry. Both functions are callable by any delegator with no rate limit: [3](#0-2) [4](#0-3) 

The `entry_to_claim_from` pointer is stored in `pool_member_info` and is only advanced on a **successful** `claim_rewards` execution:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
...
self.write_pool_member_info(:pool_member, :pool_member_info);
``` [5](#0-4) 

If the loop OOGs, the entire transaction reverts and the pointer is never advanced. There is no partial-claim or paginated-claim mechanism anywhere in the interface. [6](#0-5) 

---

### Impact Explanation

Once the trace is large enough that a single `claim_rewards` call cannot complete within the Starknet gas limit, the delegator's accumulated yield is **permanently inaccessible**. The rewards remain recorded in the contract's accounting but can never be transferred out. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

A delegator who calls `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch across hundreds of epochs without ever claiming will silently accumulate trace entries. This is a realistic pattern for active delegators who top up or partially exit regularly. The protocol provides no warning, no maximum trace depth, and no batched-claim escape hatch. The codebase itself acknowledges the loop is unbounded with the comment "unlikely to exceed gas limits," which is an assumption, not a guarantee. [7](#0-6) 

---

### Recommendation

1. **Cap trace depth**: Enforce a `MAX_BALANCE_TRACE_ENTRIES` constant (analogous to the unused `L2_TO_L1_LOGS_COMMITMENT_BYTES` in the ZkSync report) and assert it is not exceeded in `set_member_balance`.
2. **Paginated claiming**: Expose a `claim_rewards_up_to(entry_limit)` variant that processes at most `entry_limit` trace entries per call, advancing `entry_to_claim_from` incrementally so a delegator with a large trace can drain it over multiple transactions.
3. **Claim-on-balance-change**: Force a reward settlement (advancing `entry_to_claim_from`) every time `set_member_balance` is called, so the trace never grows beyond a small constant number of unprocessed entries.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` in epoch 0.
2. For epochs 1 … N, delegator calls `add_to_delegation_pool` with a minimal amount each epoch, never calling `claim_rewards`. Each call appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: epoch + K, ...)`. [8](#0-7) 
3. After N epochs the trace has N entries. `entry_to_claim_from` is still 0.
4. Delegator calls `claim_rewards`. `calculate_rewards` enters the `while entry_to_claim_from < pool_member_trace_length` loop and must iterate all N entries in one transaction. [9](#0-8) 
5. For sufficiently large N the transaction runs out of gas and reverts. `entry_to_claim_from` remains 0.
6. All subsequent `claim_rewards` calls also revert. The delegator's yield is permanently frozen.
