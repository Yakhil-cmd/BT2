### Title
Malicious Participant Sends Malformed Message to Permanently Abort DKG/Sign/Reshare via Unhandled Deserialization Error in `recv_from_others` — (`src/protocol/helpers.rs`)

---

### Summary

`recv_from_others` propagates deserialization errors from `chan.recv()` directly to the caller via `?`. A malicious participant can inject a message with a syntactically valid routing header but an invalid MessagePack payload, causing the deserialization to fail and the entire protocol future to terminate with a `ProtocolError::DeserializationError`. Because the protocol is driven by a single async future, this terminates DKG, signing, reshare, or refresh for every honest party in the session. The malicious participant can repeat this on every restart, permanently denying honest parties the ability to complete any of these operations.

---

### Finding Description

**Root cause — `src/protocol/helpers.rs` lines 19–23:**

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // ← propagates DeserializationError
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

`chan.recv` (in `src/protocol/internal.rs` lines 330–341) pops the next buffered message for the given `MessageHeader` and calls `rmp_serde::decode::from_slice` on the payload. If decoding fails, it returns `Err(ProtocolError::DeserializationError(...))`. The `?` in `recv_from_others` propagates this error immediately, unwinding the entire protocol future.

**Message acceptance — `src/protocol/internal.rs` lines 286–296:**

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);
}
```

A message is accepted into the buffer as long as it is ≥ 40 bytes and the first 40 bytes form a parseable `MessageHeader` (channel tag + waitpoint). The payload after byte 40 is never validated at ingestion time. Any participant can therefore craft a message with a legitimate channel tag and waitpoint but with garbage bytes as the payload.

**Contrast with `reliable_broadcast_receive_all` (`src/protocol/echo_broadcast.rs` lines 179–182):**

```rust
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,          // ← deserialization errors are silently skipped
};
```

The echo-broadcast path already handles this correctly by ignoring bad messages and looping. `recv_from_others` does not apply the same defensive pattern.

**Attack path:**

1. Attacker is a legitimate participant in a DKG, reshare, refresh, or FROST signing session.
2. At any round that calls `recv_from_others`, the attacker sends a message whose first 40 bytes encode the correct channel tag and waitpoint for that round, followed by arbitrary non-MessagePack bytes.
3. `push_message` accepts the message into the buffer.
4. When `recv_from_others` pops the message, `rmp_serde::decode::from_slice` fails.
5. The `?` propagates `ProtocolError::DeserializationError` out of `recv_from_others`, out of `do_keyshare` / `do_sign_coordinator_v1` / etc., and into `ProtocolExecutor::poke`, which stores it as the final result.
6. Every subsequent `poke()` call returns `Action::Return(Err(...))`, permanently terminating the session for all honest parties.
7. On restart the attacker repeats step 2, indefinitely blocking completion.

---

### Impact Explanation

**High — Permanent denial of DKG, reshare, refresh, and signing for honest parties.**

- DKG and reshare require all listed participants; excluding the attacker is not possible without changing the participant set and restarting from scratch.
- FROST signing requires at least `threshold` participants; if the attacker is one of the threshold signers chosen for a session, they can abort every signing attempt for that session.
- Because the attacker can repeat the injection on every restart with zero cost (a single crafted UDP/TCP packet), the denial is effectively permanent under the documented assumption that participants may be malicious.

---

### Likelihood Explanation

Any participant in the protocol — including a newly joined participant with no prior trust — can execute this attack. No cryptographic material, no privileged access, and no coordination with other parties is required. The attacker only needs to know the channel tag and waitpoint for the current round, both of which are deterministic and publicly derivable from the protocol structure. Likelihood is **high**.

---

### Recommendation

Mirror the defensive pattern already used in `reliable_broadcast_receive_all`: skip messages that fail deserialization instead of propagating the error.

```rust
pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        // Ignore messages that fail deserialization; wait for the next one.
        let Ok((from, msg)) = chan.recv(waitpoint).await else {
            continue;
        };
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
}
```

To prevent a malicious participant from flooding the channel with garbage and stalling the loop indefinitely, the caller should also enforce a per-participant message budget or track which participants have already been `seen` before attempting deserialization.
