### Title
Unbounded Iteration in `recursive_promise_create` over `NearPromise::And` Exhausts NEAR Gas via XCC Precompile — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `recursive_promise_create` function iterates over an arbitrarily large `promises` vector in the `NearPromise::And` variant with no size cap. An unprivileged EVM user can craft a `CrossContractCallArgs` payload encoding a `NearPromise::And([...N promises...])` with an arbitrarily large N. The EVM gas charged by the XCC precompile is proportional only to the **byte length** of the input, not to the number of promises. Because each promise in the `And` variant can be encoded compactly, a small byte-length input can encode a very large N, causing the router's `execute` call to consume far more NEAR gas than was accounted for, exhausting the NEAR gas budget and causing the XCC call to fail mid-execution.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, the `recursive_promise_create` function handles the `NearPromise::And` variant by iterating over every element of the `promises` vector with no upper bound check:

```rust
NearPromise::And(promises) => {
    let indices: Vec<PromiseIndex> = promises
        .iter()
        .map(Self::recursive_promise_create)
        .collect();
    env::promise_and(&indices)
}
``` [1](#0-0) 

Each call to `Self::recursive_promise_create` is itself recursive and invokes NEAR host functions (`env::promise_create`, `env::promise_then`, `env::promise_and`, etc.), each of which carries a non-trivial NEAR gas cost. With N promises in the `And` variant, the NEAR gas consumed scales linearly with N.

Similarly, `add_batch_actions` (enabled under the `all-promise-actions` feature) iterates over an unbounded `actions` slice:

```rust
fn add_batch_actions(id: PromiseIndex, actions: &[PromiseAction]) {
    for action in actions.iter() {
        match action { ... }
    }
}
``` [2](#0-1) 

The EVM gas charged by the XCC precompile is computed as:

```
EVM_gas = CROSS_CONTRACT_CALL_BASE + input_length * CROSS_CONTRACT_CALL_BYTE
``` [3](#0-2) 

This is purely a function of the **serialized byte length** of the `CrossContractCallArgs` input. A `NearPromise::And` with N minimal `Create` promises can be encoded in O(N) bytes, but the NEAR gas consumed by `recursive_promise_create` is also O(N) in NEAR host function calls — and the per-call NEAR gas cost of each `env::promise_create` / `env::promise_and` invocation is orders of magnitude higher than the `4 EVM gas/byte` rate implies. There is no hard limit on N enforced anywhere in the router or the precompile.

---

### Impact Explanation

**High — Temporary freezing of funds.**

When the router's `execute` call is dispatched by the engine as a NEAR promise, any wNEAR tokens that the EVM user transferred to the router (as part of the XCC flow) prior to the `execute` call are already in transit. If `execute` exhausts its NEAR gas budget during the unbounded iteration in `recursive_promise_create`, the NEAR promise fails. The `send_refund` path is never reached, and the wNEAR tokens attached to the sub-promises are not automatically returned to the user. The funds remain locked in the router sub-account until an explicit recovery action is taken, constituting a temporary freeze of user funds. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The XCC precompile is a production-facing interface reachable by any EVM contract or EOA on Aurora. Constructing a `CrossContractCallArgs` with a large `NearPromise::And` requires only knowledge of the Borsh encoding format, which is publicly documented. No privileged access, leaked keys, or governance capture is required. The attacker pays EVM gas proportional to byte length, which is cheap relative to the NEAR gas exhaustion they can induce.

---

### Recommendation

Introduce a hard limit on the number of promises in a `NearPromise::And` variant and on the number of actions in a batch, enforced in `recursive_promise_create` and `add_batch_actions` respectively. Additionally, the EVM gas cost model in the XCC precompile should account for the number of promises/actions in the input, not just the raw byte length, to ensure the charged EVM gas accurately reflects the NEAR gas that will be consumed.

---

### Proof of Concept

1. Attacker deploys an EVM contract on Aurora that calls the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372`. [5](#0-4) 
2. The contract encodes a `CrossContractCallArgs` containing `PromiseArgs::Recursive(NearPromise::And([p1, p2, ..., pN]))` where each `pi` is a minimal `NearPromise::Simple(SimpleNearPromise::Create(...))`. With N = 10,000, the serialized input is on the order of tens of kilobytes (cheap in EVM gas), but the router must call `env::promise_create` N times plus `env::promise_and` once.
3. The engine dispatches a NEAR promise to `router.execute(args)` with the NEAR gas budget derived from the EVM gas limit.
4. Inside `execute`, `recursive_promise_create` iterates over all N promises, exhausting the NEAR gas budget before completing. [6](#0-5) 
5. The NEAR promise fails. Any wNEAR transferred to the router as part of the XCC setup is not refunded via `send_refund`, leaving user funds temporarily frozen in the router sub-account.
