### Title
Pool Admin Bypasses Fee Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` allows the pool admin to write any `uint16` value (0–65535, i.e., 0–6.5535% in E6 units) into a bin's `addFeeBuyE6`/`addFeeSellE6` fields with no cap check, while the parallel global-fee setter `setPoolAdminFees()` enforces `maxAdminSpreadFeeE6`. The per-bin additional fee is added directly to the oracle-derived base fee in every swap path, so an uncapped value causes traders to pay more than the factory-owner-intended ceiling on every swap through that bin.

### Finding Description

`setPoolAdminFees` enforces the factory-owner-set cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no equivalent check — it forwards the raw caller-supplied values straight to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` only validates the bin index, not the fee magnitude:

```solidity
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

The stored per-bin fee is then added to the oracle-derived base fee in every swap iteration across all four swap paths:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) [5](#0-4) [6](#0-5) 

`BinState.addFeeBuyE6` is `uint16`, so the maximum settable value is 65535 (6.5535% in E6 units). The factory hard cap `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) is never consulted for this path. [7](#0-6) [8](#0-7) 

### Impact Explanation

Every swap through the affected bin pays `baseFeeX64 + 65535/1e6` instead of `baseFeeX64 + (capped value)/1e6`. The difference is extracted from the trader's output and retained in the pool as LP fee (of which the admin's share flows to `poolAdminFeeDestination`). This is a direct, per-swap loss of trader principal bounded by the `uint16` ceiling (≤ 6.5535% additional fee per bin). The factory owner's intent to limit admin fee extraction via `maxAdminSpreadFeeE6` is fully bypassed for the per-bin channel.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly expected to be constrained by caps. The call path is a single transaction with no timelock. Any pool admin can execute this immediately after pool creation. The scope rules explicitly identify `setBinAdditionalFees` as an admin-boundary bypass surface.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, analogous to `setPoolAdminFees`:

```solidity
if (uint256(addFeeBuyE6) + uint256(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
// or enforce each independently against a dedicated per-bin cap
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` factory state variable and enforce it in both `setPoolBinAdditionalFees` and at pool creation when initial bin fees are packed into `nonNegativeBinDataArray`/`negativeBinDataArray`.

### Proof of Concept

```solidity
// Pool admin calls with max uint16 values
factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);

// Swap in bin 0: effective buy fee = baseFeeX64 + 65535/1e6
// baseFeeX64 is oracle-spread-derived; 65535/1e6 ≈ 6.5535% additional
// If maxAdminSpreadFeeE6 = 1000 (0.1%), the admin has extracted 65x the intended cap
// Trader receives ~6.5% less token0 than oracle price implies
```

The `test_setPoolBinAdditionalFees_updatesStorage_emitsEvent` test in `MetricOmmPoolFactory.t.sol` already confirms the write succeeds with arbitrary values and no revert. [9](#0-8)
