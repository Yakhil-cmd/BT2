### Title
`require_running` Guard on `exit_to_near_precompile_callback` Causes Permanent Fund Loss When Contract Is Paused Mid-Flight - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `exit_to_near_precompile_callback` function enforces `require_running` at the start of its execution. This callback is the second leg of a two-step NEAR promise chain initiated by the `ExitToNear` precompile. The first leg (the `ft_transfer` / `near_withdraw` call) burns the user's ERC-20 tokens or deducts ETH from their Aurora balance **before** the callback fires. If the Aurora Engine contract is paused between the first promise executing and the callback receipt being processed, the callback reverts with `ERR_PAUSED`, the refund path inside the callback is never reached, and the user's tokens are permanently lost.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) constructs a two-promise chain:

1. **Base promise**: calls `ft_transfer` (or `near_withdraw` for wNEAR unwrap) on the NEP-141 contract. At this point the user's ERC-20 tokens have already been burned on the Aurora side.
2. **Callback**: calls `exit_to_near_precompile_callback` on the Aurora Engine itself.

The callback is responsible for two critical outcomes:
- On success of the base promise: forward the unwrapped NEAR to the recipient (`transfer_near` path).
- On failure of the base promise: **refund the user's tokens** via `engine::refund_on_error` (`refund` path).

The callback implementation begins with:

```rust
// engine/src/contract_methods/connector.rs, line 202-203
let state = state::get_state(&io)?;
require_running(&state)?;
```

`require_running` returns `Err(ERR_PAUSED)` if `state.is_paused == true`. This causes the entire callback to abort before it can reach either the `transfer_near` branch or the `refund` branch.

The attack/failure scenario:
1. User calls an ERC-20 `withdrawToNear` (or ETH exit). The ERC-20 tokens are burned; the base promise is scheduled.
2. The owner calls `pause_contract` (or the contract is paused for any operational reason) before the callback receipt is processed.
3. The callback fires, hits `require_running`, and panics with `ERR_PAUSED`.
4. The refund path is never executed. The user's tokens are gone — burned on Aurora, never transferred on NEAR.

There is no automatic recovery mechanism. The `refund_on_error` path inside the callback is the **only** on-chain refund path for this flow.

---

### Impact Explanation

**Critical — Permanent freezing / direct loss of user funds.**

- The user's ERC-20 tokens (or ETH) are burned from their Aurora balance at EVM execution time, before any NEAR promise fires.
- If the callback cannot execute, those tokens are unrecoverable on-chain. There is no admin `refundGame`-equivalent function in Aurora Engine that can restore burned ERC-20 balances.
- The `refund_on_error` path inside the callback is the sole recovery mechanism, and it is blocked by `require_running`.

---

### Likelihood Explanation

**Medium.** The Aurora Engine owner can pause the contract at any time via `pause_contract`. Pauses are used during upgrades, security incidents, or operational maintenance. NEAR processes receipts asynchronously across blocks; the window between the base promise executing and the callback receipt being processed can span multiple blocks, making a race condition with a pause realistic. Any user who initiates an `ExitToNear` during a period that overlaps with a pause is at risk.

---

### Recommendation

Remove `require_running` from `exit_to_near_precompile_callback`. This callback is a private self-call (`env.assert_private_call()` is already enforced) and is only ever scheduled by the engine itself as the second leg of an exit flow. It is not a user-initiated entry point and should not be subject to the contract-wide pause guard. The refund path in particular **must** be able to execute regardless of pause state to protect user funds.

Similarly, audit `withdraw_wnear_to_router` and `mirror_erc20_token_callback` for the same pattern, as both also call `require_running` and are callbacks in multi-step promise chains.

---
