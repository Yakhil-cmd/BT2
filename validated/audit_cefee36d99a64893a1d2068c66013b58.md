### Title
Skewed Reward Denominator in Consensus `update_rewards` Path — (`File: src/staking/staking.cairo`)

### Summary
In the consensus-based reward path, `update_rewards` passes the **individual staker's own balance** as the `strk_total_stake` denominator to `_update_rewards`, instead of the **global total staking power**. The attestation-based path correctly uses `get_current_total_staking_power()`. This mirrors M-03 exactly: the denominator used for proportional reward calculation is skewed, causing the calling staker to receive a disproportionate share of block rewards at the expense of all other stakers.

---

### Finding Description

The `Staking` contract exposes two reward-update paths:

**Attestation path** (`update_rewards_from_attestation_contract`, lines 1394–1423):
```cairo
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_epoch_rewards,
    btc_total_rewards: btc_epoch_rewards,
    :strk_total_stake,   // ← global total (correct denominator)
    :btc_total_stake,
    ...
);
```

**Consensus path** (`update_rewards`, lines 1474–1500):
```cairo
let (staker_total_strk_balance, staker_total_btc_balance) = self
    .get_staker_total_strk_btc_balance_at_epoch(
        :staker_address, :staker_pool_info, epoch_id: curr_epoch,
    );
...
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_block_rewards,
    btc_total_rewards: btc_block_rewards,
    strk_total_stake: staker_total_strk_balance,  // ← individual balance (wrong denominator)
    ...
);
```

The parameter `strk_total_stake` in `_update_rewards` is the denominator for proportional reward calculation (as confirmed by the attestation path, which passes the global total). In the consensus path, it is instead set to `staker_total_strk_balance` — the individual staker's own balance.

When `_update_rewards` computes the staker's proportional share:

```
staker_reward = total_rewards × staker_balance / strk_total_stake
```

With the wrong denominator:

```
staker_reward = block_rewards × staker_balance / staker_balance = block_rewards
```

The calling staker receives **100% of block rewards** instead of their proportional share.

The global total staking power is correctly tracked in `tokens_total_stake_trace` and is available via `get_current_total_staking_power()` / `get_total_staking_power_at_epoch()`, but the consensus path never queries it.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Any staker can call `update_rewards` (no access control) once per block. By doing so, they receive the entire block reward allocation rather than their proportional share. All other stakers are permanently deprived of their yield for that block. A staker with even a small stake can repeatedly call `update_rewards` every block to drain the entire reward stream.

---

### Likelihood Explanation

**High.** `update_rewards` is a public, permissionless function callable by any staker once per block. The only precondition is that the staker is active and has a non-zero balance at the current epoch. A rational staker has a direct financial incentive to call this function every block to capture 100% of block rewards.

---

### Recommendation

In `update_rewards`, replace the individual staker balance with the global total staking power as the denominator, consistent with the attestation path:

```cairo
// Replace:
let (staker_total_strk_balance, staker_total_btc_balance) = self
    .get_staker_total_strk_btc_balance_at_epoch(...);
...
strk_total_stake: staker_total_strk_balance,

// With:
let (strk_total_stake, btc_total_stake) = self
    .get_total_staking_power_at_epoch(epoch_id: curr_epoch);
...
strk_total_stake: strk_total_stake,
```

---

### Proof of Concept

1. Staker A holds 1% of total staked STRK. Staker B holds 99%.
2. Each block, Staker A calls `update_rewards(staker_address: A, disable_rewards: false)`.
3. `staker_total_strk_balance` = Staker A's balance (1% of total).
4. `_update_rewards` computes: `block_rewards × A_balance / A_balance = block_rewards`.
5. Staker A receives 100% of block rewards every block.
6. Staker B receives 0 rewards despite holding 99% of stake.
