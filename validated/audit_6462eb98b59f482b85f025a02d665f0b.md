Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Breaks ICP/Cycles Conservation Invariant - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The Cycles Minting Canister's `burn_and_log` function is intentionally designed to swallow all errors from the ICP ledger burn operation. After cycles are irreversibly deposited to a target canister, if the subsequent ICP burn fails (e.g., due to a transient inter-canister call rejection during a ledger upgrade), the ICP remains unburned in the CMC's subaccount while the notification is permanently recorded as successfully processed. This breaks the ICP/cycles conservation invariant: cycles are created without the corresponding ICP being destroyed.

## Finding Description

In `process_top_up` (L1985–2012), after `deposit_cycles` succeeds, `burn_and_log` is called and its result is unconditionally discarded:

```rust
match deposit_cycles(canister_id, cycles, true, limiter_to_use).await {
    Ok(()) => {
        burn_and_log(sub, amount).await;  // result discarded
        Ok(cycles)
    }
``` [1](#0-0) 

The same pattern appears in `process_mint_cycles` (L1966–1968):

```rust
match do_mint_cycles(to_account, cycles, deposit_memo).await {
    Ok(deposit_result) => {
        burn_and_log(sub, amount).await;  // result discarded
``` [2](#0-1) 

`burn_and_log` (L2014–2049) is explicitly typed to return `()` and only logs errors:

```rust
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    ...
    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
``` [3](#0-2) 

After `process_top_up` or `process_mint_cycles` returns `Ok(...)`, the caller immediately records the notification as permanently processed:

```rust
state.blocks_notified.insert(
    block_index,
    NotificationStatus::NotifiedTopUp(result.clone()),
);
if is_transient_error(&result) {
    state.blocks_notified.remove(&block_index);
}
``` [4](#0-3) 

Because `process_top_up` returns `Ok(cycles)` regardless of whether the burn succeeded, `is_transient_error` evaluates to `false` and the notification is never removed. Any subsequent call with the same `block_index` hits the `NotifiedTopUp` occupied-entry branch and returns an `InvalidTransaction` error, permanently preventing retry. [5](#0-4) 

The `burn_and_log` call uses `call_protobuf` to invoke `send_pb` on the ICP ledger. This is a standard inter-canister call that can fail with `RejectionCode::SysTransient` (code 4) when the ledger canister is being upgraded or is temporarily unavailable — a routine operational event on the IC. [6](#0-5) 

## Impact Explanation

When `burn_and_log` fails silently:

1. Cycles have been irreversibly deposited to the target canister.
2. The ICP remains in the CMC's subaccount (keyed by `canister_id` or `caller`), unburned.
3. The notification block is permanently marked as successfully processed — no retry is possible.
4. The ICP is locked in the CMC's subaccount with no automated recovery path.
5. The ICP total supply is not reduced while the cycles supply has increased — the conservation invariant is broken.

This constitutes **illegal minting**: cycles are created without destroying the corresponding ICP. This is a significant ledger/CMC security impact with concrete protocol harm, matching the High impact category: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation

The ICP ledger is upgraded periodically via NNS governance proposals. During the upgrade window, inter-canister calls to the ledger fail with `SysTransient`. Any `notify_top_up` or `notify_mint_cycles` call that completes its `deposit_cycles`/`do_mint_cycles` step just before or during a ledger upgrade will hit this path. The failure is not directly attacker-controlled, but it is a real, recurring operational scenario. Given the CMC processes continuous high-value transactions, the probability of at least one occurrence per ledger upgrade cycle is non-negligible. The vulnerable window is narrow but real.

## Recommendation

`burn_and_log` should be refactored to return a `Result`. On burn failure after a successful cycles deposit, the CMC should either:

1. **Retry queue**: Record the `(subaccount, amount)` pair in persistent state and have a background heartbeat task retry pending burns against the ledger.
2. **Deferred success**: Do not mark the notification as `NotifiedTopUp(Ok(...))` until the burn is confirmed. Instead, use a new `NotificationStatus::PendingBurn(cycles)` state that allows the notification to be retried for the burn step only (not the cycles deposit step, which is already complete).

The design comment "we don't want to reject the transaction notification because then it could be retried" is valid for the cycles-deposit step, but the burn is a separate, independent operation. Its failure should be tracked and retried independently without re-depositing cycles.

## Proof of Concept

1. User sends `X` ICP to the CMC subaccount for canister `C` and calls `notify_top_up(block_index)`.
2. CMC calls `deposit_cycles` — canister `C` receives cycles. This succeeds.
3. CMC calls `burn_and_log`; the ICP ledger is mid-upgrade and returns `(SysTransient, "canister is upgrading")`.
4. `burn_and_log` logs the error and returns `()`.
5. `process_top_up` returns `Ok(cycles)`.
6. CMC records `NotifiedTopUp(Ok(cycles))` in `blocks_notified`; `is_transient_error` is `false` so the entry is kept.
7. Any subsequent call to `notify_top_up` with the same `block_index` returns `InvalidTransaction`.
8. `X` ICP remains in the CMC's subaccount for canister `C`. It is never burned. Cycles exist without corresponding ICP destruction.

A deterministic integration test can reproduce this by mocking the ICP ledger's `send_pb` endpoint to return a `SysTransient` rejection on the first call after `deposit_cycles` succeeds, then asserting that the CMC subaccount balance is non-zero and the notification is permanently recorded as `Ok`.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
