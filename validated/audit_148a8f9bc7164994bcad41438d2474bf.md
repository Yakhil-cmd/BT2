### Title
Unverified Participant Contributions in CKD Protocol Allow Any Malicious Participant to Corrupt the CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly sums `(norm_big_y, norm_big_c)` values received from every participant with no proof of correctness. A single malicious participant can substitute arbitrary group elements, causing the coordinator to assemble a corrupted `CKDOutput` that the TEE application will silently accept and unmask into a wrong confidential key.

---

### Finding Description

In `do_ckd_coordinator` the coordinator collects each participant's contribution and adds it directly to the running totals:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The correct contribution from participant `i` is:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

computed inside `compute_signature_share` (lines 148–181). Nothing in the coordinator path checks that the received pair satisfies this relation. There is no zero-knowledge proof, no commitment-then-reveal, and no consistency check against the participant's public key share. Any participant can replace their honest contribution with arbitrary `(P, Q) ∈ G₁²` and the coordinator has no means of detection.

Compare this with the DKG protocol (`src/dkg.rs`), which uses a commit-then-reveal scheme (`verify_commitment_hash`, lines 222–236) and a Schnorr proof of knowledge (`internal_verify_proof_of_knowledge`, lines 145–166) to authenticate every participant's polynomial commitment before it is used. The CKD protocol has no equivalent safeguard.

---

### Impact Explanation

The final CKD output is:

```
Y_total = Σ norm_big_y_i
C_total = Σ norm_big_c_i
```

The TEE application unmasks with its secret `app_sk`:

```
confidential_key = C_total − app_sk · Y_total
```

If participant `j` sends `(P_j, Q_j)` instead of the correct pair, the coordinator computes:

```
Y_total = Y_honest + P_j   (wrong)
C_total = C_honest + Q_j   (wrong)
```

The TEE application then derives:

```
C_honest + Q_j − app_sk · (Y_honest + P_j)
```

which is not the intended `msk · H(pk ‖ app_id)`. The output is an unusable, attacker-perturbed group element. Because the coordinator returns this value as a valid `CKDOutput` with no error, honest parties (coordinator and TEE) accept it without suspicion.

**Matched impact**: *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

The attack requires only that one participant in the CKD session be malicious. No threshold of colluders is needed; no cryptographic primitive needs to be broken; no external assumption is required. The attacker simply deviates from the protocol by sending a chosen pair of group elements instead of the honest computation. The entry path is the standard `ckd()` public API (`src/confidential_key_derivation/protocol.rs`, lines 66–116), reachable by any library caller who is listed as a participant.

---

### Recommendation

Add a zero-knowledge proof that each participant's `(norm_big_y, norm_big_c)` is correctly formed with respect to their public key share. Concretely, participant `i` should prove in zero knowledge:

- Knowledge of `y_i` such that `norm_big_y_i = λ_i · y_i · G`
- Knowledge of `y_i` and `x_i` such that `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`, where `x_i · G₂` is the publicly known verification share from the DKG output.

This is a standard linear-relation proof (a Chaum–Pedersen / dlog-equality proof) and is consistent with the ZKP infrastructure already present in `src/crypto/proofs/dlogeq.rs`. The coordinator should verify each proof before adding the contribution to the running totals, mirroring the pattern used in `do_keyshare` (`src/dkg.rs`, lines 452–476).

---

### Proof of Concept

1. Run a CKD session with participants `{P1, P2, P3}` and coordinator `P1`.
2. Participant `P2` is malicious. Instead of calling `compute_signature_share`, it sends `(G₁, G₁)` (the generator point for both fields) to the coordinator.
3. The coordinator executes lines 50–55 of `protocol.rs`, adding `G₁` to both `norm_big_y` and `norm_big_c` without any check.
4. The coordinator returns a `CKDOutput` containing the corrupted totals.
5. The TEE application calls `ckd_output.unmask(app_sk)` and obtains a group element that is not `msk · H(pk ‖ app_id)`.
6. Every subsequent operation that depends on the confidential key (e.g., deriving a TEE secret) silently uses the wrong value. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```

```rust
fn verify_commitment_hash<C: Ciphersuite>(
    session_id: &HashOutput,
    participant: Participant,
    domain_separator: &mut DomainSeparator,
    commitment: &VerifiableSecretSharingCommitment<C>,
    all_hash_commitments: &ParticipantMap<'_, HashOutput>,
) -> Result<(), ProtocolError> {
    let actual_commitment_hash = all_hash_commitments.index(participant)?;
    let commitment_hash =
        domain_separate_hash(domain_separator, &(&participant, &commitment, &session_id))?;
    if *actual_commitment_hash != commitment_hash {
        return Err(ProtocolError::InvalidCommitmentHash);
    }
    Ok(())
}
```
