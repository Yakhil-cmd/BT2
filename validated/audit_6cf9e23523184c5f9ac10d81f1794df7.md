### Title
`vm_resources_to_execution_resources` Returns `UnknownBuiltin` Error for Valid `range_check96`/`add_mod`/`mul_mod` Builtins, Breaking RPC Simulation and Tracing — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`vm_resources_to_execution_resources` has an explicit TODO acknowledging that `range_check96`, `add_mod`, and `mul_mod` builtins are unhandled. Any transaction whose execution produces these builtins — which blockifier fully supports since versioned constants 0.13.2 — causes the RPC trace/simulation conversion to return `Err(ExecutionError::UnknownBuiltin)` instead of a valid trace, even though blockifier executed the transaction successfully.

---

### Finding Description

The match in `vm_resources_to_execution_resources` covers only 8 builtins: [1](#0-0) 

Lines 385–388 contain a developer TODO explicitly noting the three missing variants, and the `_` wildcard at line 389–391 returns `Err(ExecutionError::UnknownBuiltin { builtin_name })` for all of them.

This function is called unconditionally from `FunctionInvocation::try_from` during trace/simulation conversion: [2](#0-1) 

Blockifier's versioned constants have tracked `range_check96`, `add_mod`, and `mul_mod` since at least version 0.13.2, and `entry_point_execution.rs` and `bouncer.rs` both reference these builtins as first-class resources that can appear in real execution output. A Sierra contract that exercises modular arithmetic (e.g., using `u96`-range operations or the `mod_builtin`) will produce non-zero counts for these builtins in `VmExecutionResources::builtin_instance_counter`.

---

### Impact Explanation

When a user calls `starknet_simulateTransactions` or `starknet_traceTransaction` for a transaction whose contract uses any of these three builtins, the RPC handler receives `Err(ExecutionError::UnknownBuiltin)` from the conversion layer and returns an error response — even though blockifier executed the transaction successfully and produced a valid `CallInfo`. The simulation/trace result is corrupted: the caller receives an error instead of a valid trace, violating the invariant that any successfully-executed transaction is traceable.

Impact: **High** — RPC simulation and tracing return an error for valid, successfully-executed transactions.

---

### Likelihood Explanation

`range_check96` is used by Sierra's 96-bit range-check operations, which are emitted by standard arithmetic on `u96`/`u128` types in Cairo contracts. Any deployed contract using these types will trigger the bug. No special privileges are required — any unprivileged user can deploy such a contract and submit a transaction against it.

---

### Recommendation

Add the three missing arms to the match in `vm_resources_to_execution_resources`:

```rust
BuiltinName::range_check96 => {
    builtin_instance_counter.insert(Builtin::RangeCheck96, count)
}
BuiltinName::add_mod => {
    builtin_instance_counter.insert(Builtin::AddMod, count)
}
BuiltinName::mul_mod => {
    builtin_instance_counter.insert(Builtin::MulMod, count)
}
```

Verify that the corresponding `Builtin` enum variants exist in the RPC types crate; if not, add them and ensure downstream serialization matches the Starknet spec.

---

### Proof of Concept

```rust
use cairo_vm::types::builtin_name::BuiltinName;
use cairo_vm::vm::runners::cairo_runner::ExecutionResources as VmExecutionResources;
use std::collections::HashMap;

#[test]
fn range_check96_causes_unknown_builtin_error() {
    let mut counters = HashMap::new();
    counters.insert(BuiltinName::range_check96, 1usize);
    let vm_resources = VmExecutionResources {
        n_steps: 100,
        n_memory_holes: 0,
        builtin_instance_counter: counters,
    };
    let gas = GasVector::default();
    let result = vm_resources_to_execution_resources(vm_resources, gas);
    assert!(matches!(
        result,
        Err(ExecutionError::UnknownBuiltin { builtin_name })
            if builtin_name == BuiltinName::range_check96
    ));
}
```

This test confirms the function returns `Err(UnknownBuiltin)` for a `VmExecutionResources` that blockifier can produce for any contract using 96-bit range checks — a common Sierra pattern.
