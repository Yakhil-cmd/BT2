### Title
Cross-Chain Replay of Pre-EIP-155 Legacy Transactions Bypasses Chain ID Validation, Enabling Fund Theft Across Aurora Deployments - (File: `engine/src/engine.rs`)

### Summary

The `submit_with_alt_modexp` function in `engine/src/engine.rs` validates the chain ID only when `transaction.chain_id` is `Some`. Pre-EIP-155 legacy transactions (signed with `v=27` or `v=28`) produce `chain_id: None` after normalization, causing the chain ID guard to be silently skipped. A valid signed transaction intended for one Aurora deployment (e.g., a silo or testnet) can be replayed verbatim on any other Aurora deployment where the sender has the same nonce, directly stealing the sender's funds on the replayed chain.

### Finding Description

**Root cause — missing domain separation for pre-EIP-155 legacy transactions:**

`LegacyEthSignedTransaction::sender()` in `engine-transactions/src/legacy.rs` accepts `v=27` or `v=28` and returns `chain_id: None`:

```rust
27..=28 => (
    None,
    u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
),
``` [1](#0-0) 

`NormalizedEthTransaction::try_from` for the `Legacy` variant propagates this as `chain_id: tx.chain_id()`, which is `None`: [2](#0-1) 

In `submit_with_alt_modexp`, the chain ID guard uses an `if let Some(...)` pattern:

```rust
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [3](#0-2) 

When `chain_id` is `None`, the entire guard short-circuits and the transaction proceeds to nonce checking and execution with **no chain binding**. The nonce check that follows only prevents replay on the *same* chain — it does not prevent replay on a *different* Aurora deployment where the sender happens to share the same nonce. [4](#0-3) 

The public `submit` and `submit_with_args` NEAR entry points both funnel through this same `submit_with_alt_modexp` path: [5](#0-4) 

**Exploit path (end-to-end):**

1. Victim signs a pre-EIP-155 legacy transaction (`v=27` or `v=28`, no chain ID) on Aurora Mainnet (chain_id `1313161554`) — e.g., a transfer of 10 ETH to address `B`, nonce=0.
2. Attacker observes the raw RLP bytes of this transaction on-chain or in the mempool.
3. Victim also has 10 ETH on Aurora Testnet (chain_id `1313161555`) with nonce=0 (common for users who have used both networks from a fresh address).
4. Attacker submits the identical RLP bytes to Aurora Testnet's `submit` entry point.
5. The engine parses the transaction, recovers the same sender address (ECDSA recovery is chain-agnostic for pre-EIP-155), finds `chain_id=None`, skips the chain ID guard, passes the nonce check (nonce=0 matches), and executes the transfer — draining the victim's 10 ETH on Testnet.

The same attack applies across any pair of Aurora silo deployments that share a user with the same nonce.

**Contradiction with stated security posture:**

`CHANGES.md` at v2.4.0 explicitly states:
> "Security improvement: Engine will no longer accept EVM transactions without a chain ID as part of their signature." [6](#0-5) 

The current production code contradicts this: `v=27`/`v=28` transactions are parsed successfully and the chain ID guard is bypassed for `None`. The unit test `test_eth_signed_no_chain_sender` confirms the parser accepts such transactions today. [7](#0-6) 

### Impact Explanation

**Critical — Direct theft of user funds at-rest.**

Any user who:
- signs a pre-EIP-155 transaction (e.g., via an older wallet, a custom script, or any tooling that omits chain_id), **and**
- holds funds on a second Aurora deployment (mainnet, testnet, or any silo) at the same nonce

can have that transaction replayed by an unprivileged attacker, transferring the user's ETH or triggering arbitrary EVM calls (including ERC-20 transfers) on the second chain without the user's consent. The attacker needs no privileged access — only the ability to call the public `submit` NEAR method with the observed transaction bytes.

### Likelihood Explanation

**Medium.** Modern wallets (MetaMask, hardware wallets) default to EIP-155 signing, but:
- Developers using raw `eth_sign` or custom scripts frequently omit chain_id.
- Aurora's multi-deployment model (mainnet + testnet + many silo chains) means the same EOA address and nonce=0 is common across deployments for new users.
- The attack requires no special capability beyond submitting a NEAR transaction to the `submit` method, which is publicly callable.

### Recommendation

In `submit_with_alt_modexp`, reject transactions where `chain_id` is `None` (pre-EIP-155 format) rather than silently skipping the guard:

```rust
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This aligns the code with the stated security improvement from v2.4.0 and eliminates the cross-chain replay surface entirely.

### Proof of Concept

```
# 1. Victim signs a pre-EIP-155 transfer on Aurora Mainnet (chain_id omitted, v=27 or v=28)
raw_tx = rlp_encode(nonce=0, gas_price=..., gas_limit=..., to=B, value=10_ETH, data=b"", v=27, r=..., s=...)

# 2. Attacker submits the IDENTICAL bytes to Aurora Testnet's `submit` entry point
near call aurora submit --args raw_tx --accountId attacker.near  # on testnet RPC

# 3. Engine flow on Testnet:
#    - EthTransactionKind::try_from(raw_tx) -> Legacy(tx) with v=27
#    - LegacyEthSignedTransaction::chain_id() -> None
#    - NormalizedEthTransaction { chain_id: None, ... }
#    - submit_with_alt_modexp: `if let Some(chain_id) = None` -> guard skipped
#    - check_nonce(testnet_state, sender, nonce=0) -> passes (victim's testnet nonce is 0)
#    - EVM executes: 10 ETH transferred from victim to B on Testnet
```

The victim loses 10 ETH on Testnet without having signed anything for Testnet. The attacker gains nothing directly but can direct the funds to any address `B` they control by first obtaining a pre-EIP-155 signature from the victim for a transfer to `B`.
