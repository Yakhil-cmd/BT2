### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator blindly aggregates `(norm_big_y, norm_big_c)` shares from all participants with no proof of correctness. A single malicious participant can send arbitrary group elements, silently corrupting the derived confidential key that the coordinator returns. There is no final consistency check that would catch the manipulation.

### Finding Description
In `do_ckd_coordinator` (lines 44–57), the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and accumulates them with simple field addition:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each participant is supposed to compute, in `compute_signature_share`:

- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · A)` [2](#0-1) 

and send these to the coordinator via a private channel:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

No zero-knowledge proof, commitment, or any other binding mechanism accompanies these values. The coordinator has no way to distinguish a correctly computed share from an arbitrary group element pair. This is structurally identical to the oracle-dependency flaw in the external report: the protocol relies on a single, unverified source of truth per participant, and any one of them can silently poison the aggregate.

Contrast this with the robust ECDSA signing path, where the coordinator performs a final `sig.verify(&public_key, &msg_hash)` check that catches corrupted signature shares: [4](#0-3) 

No equivalent correctness check exists in the CKD coordinator path.

### Impact Explanation
**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The correct aggregate satisfies `C − a·Y = x · H(pk, app_id)`, where `a` is the application's secret key and `x` is the threshold master secret. If a malicious participant substitutes arbitrary `(big_y', big_c')`, the coordinator produces:

```
Y_total  = Y_honest + Δy
C_total  = C_honest + Δc
```

When the application unmasks: `C_total − a·Y_total = x·H(pk,app_id) + Δc − a·Δy`, which is a wrong, unusable key. The malicious participant cannot recover `x` from this (they do not know `a`), but they permanently corrupt the CKD output for every honest party that relies on it.

### Likelihood Explanation
**High.** Any single participant in the CKD session can execute this attack. It requires no special capability beyond being a protocol participant, no cryptographic break, and no external oracle. The attack is a single-round substitution of two group elements with no detection path.

### Recommendation
Require each participant to accompany their `(norm_big_y, norm_big_c)` with a Sigma-protocol (e.g., a Chaum–Pedersen proof) demonstrating that:

1. `norm_big_y` is a scalar multiple of `G` consistent with their committed randomness `y_i`.
2. `norm_big_c` is correctly formed as `λ_i · (x_i · H(pk, app_id) + y_i · A)`, binding to their known public share `λ_i · x_i · G` (derivable from the DKG verifying key).

Alternatively, add a post-aggregation consistency check analogous to the ECDSA coordinator's `sig.verify` call, if such a check is feasible without revealing the secret.

### Proof of Concept
1. Honest participants `P_1, …, P_{n-1}` compute correct shares and send them to the coordinator `P_c`.
2. Malicious participant `P_m` sends `(big_y' = G, big_c' = G)` (arbitrary non-zero points) instead of their correct share.
3. The coordinator accumulates without complaint:
   ```
   norm_big_y += G   // wrong
   norm_big_c += G   // wrong
   ```
4. `CKDOutput::new(norm_big_y, norm_big_c)` is returned as `Some(ckd_output)`.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `x·H(pk,app_id) + G − app_sk·G`, which is not the intended confidential key and is indistinguishable from a random group element to the application.
6. All honest parties have accepted a silently corrupted CKD output with no error or warning.
