### Title
Gateway Signature Verification Bypass via `skip_stateful_validations` UX Path Admits Invalid Invoke Transactions into Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path unconditionally skips the blockifier `__validate__` entry point — which is the only place account signature verification occurs — for any invoke transaction with `nonce=1` targeting an account whose on-chain nonce is `0` and that has any transaction present in the mempool pool or recent-block state. An unprivileged attacker who observes a pending `deploy_account` transaction for address A can immediately submit an invoke transaction for A with `nonce=1` and a completely invalid signature; the gateway will admit it to the mempool without ever verifying the signature.

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have passed account-level signature verification (the `__validate__` entry point).

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

When `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)`, the function queries `account_tx_in_pool_or_recent_block` and, if it returns `true`, returns `true` (skip validation). The mempool check is: [2](#0-1) 

This returns `true` for **any** transaction in the pool or recent-block state for that address — not specifically a `deploy_account`. Any attacker who can observe a pending `deploy_account` for address A (the mempool is public) satisfies this condition.

**How the skip propagates to `run_validate_entry_point`:** [3](#0-2) 

`validate: !skip_validate` is set to `false`. Inside `validate_tx`: [4](#0-3) 

The function returns `Ok(None)` immediately — the `__validate__` entry point, which calls the account's ECDSA signature check, is never executed.

**Full attacker-controlled path:**

1. Stateless validation (`StatelessTransactionValidator::validate`) checks calldata/signature *length* and resource bounds, but not cryptographic validity. [5](#0-4) 

2. `convert_rpc_tx_to_internal` computes the tx hash and converts fields; no signature verification. [6](#0-5) 

3. `extract_state_nonce_and_run_validations` reads `account_nonce=0`, calls `run_pre_validation_checks` (nonce `0 ≤ 1 ≤ 200` passes, `validate_by_mempool` checks only for duplicates/nonce conflicts), then `skip_stateful_validations` returns `true`, and `run_validate_entry_point` is called with `skip_validate=true`. [7](#0-6) 

4. The transaction is forwarded to the mempool with an invalid signature. [8](#0-7) 

### Impact Explanation

The gateway/mempool admission invariant is broken: an invoke transaction with an arbitrary (invalid) signature is accepted and stored in the mempool. The gateway returns a valid-looking `tx_hash` to the caller. The batcher will later attempt to execute the transaction, run `__validate__`, fail, and reject it — but by then the invalid transaction has occupied mempool capacity and consumed batcher resources. An attacker monitoring the public mempool for `deploy_account` transactions can continuously inject invalid invoke transactions for every such account, degrading mempool quality and wasting batcher execution cycles.

This matches the stated impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The `deploy_account` transactions in the mempool are observable (the mempool is public). The condition to trigger the bypass (`nonce=1`, `account_nonce=0`, any pool entry for the address) is trivially satisfiable for any account with a pending deployment. No privileged access, special key material, or network position is required. The attacker only needs to submit a well-formed RPC invoke transaction with an arbitrary signature field.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction (not just any transaction) is pending for the address. Alternatively, restrict the skip to cases where the mempool can confirm the pending transaction is a `deploy_account` type:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use a dedicated check:
mempool_client.has_pending_deploy_account(tx.sender_address())
```

If a type-specific check is not feasible, the skip should be removed entirely and the UX concern addressed by having the batcher tolerate a short window where the deploy_account has not yet been committed.

### Proof of Concept

1. Submit a `deploy_account` transaction for a fresh address A to the gateway. The gateway admits it; the mempool now has A in its pool.

2. Immediately submit an `invoke` transaction for address A with `nonce=1` and a signature of `[Felt::ONE, Felt::ONE]` (invalid ECDSA values):

```
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "version": "0x3",
  "sender_address": "<address A>",
  "nonce": "0x1",
  "signature": ["0x1", "0x1"],   // invalid signature
  "calldata": [...],
  "resource_bounds": { ... valid bounds ... },
  ...
}
```

3. The gateway returns a `tx_hash` (admission success). The invalid invoke is now in the mempool. Confirmed by querying `starknet_getTransactionStatus` — it will show `RECEIVED` until the batcher rejects it.

The bypass is triggered at: [9](#0-8) 

because `account_tx_in_pool_or_recent_block` returns `true` (the deploy_account is in the pool), causing `skip_validate=true` and the `__validate__` entry point to be skipped entirely.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
