### Title
Unvalidated `app_pk` Parameter in `ckd()` Allows Coordinator to Recover Confidential Derived Key in Plaintext — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry-point in `src/confidential_key_derivation/protocol.rs` accepts an `app_pk: PublicKey` (`blstrs::G1Projective`) parameter without any validity check. If a caller supplies the group identity element (`G1Projective::identity()`) as `app_pk`, the ElGamal masking term `app_pk * y` collapses to the identity, and the coordinator's aggregated `big_c` output equals the confidential derived key `msk · H(pk ‖ app_id)` in plaintext — a value the coordinator is explicitly not supposed to learn.

---

### Finding Description

The CKD protocol is an ElGamal-style masked computation. Each participant computes:

```
big_c_i = x_i · H(pk ‖ app_id)   +   app_pk · y_i
          ^^^^^^^^^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^^
          secret share contribution     randomness mask
```

The coordinator aggregates all Lagrange-weighted shares:

```
big_C = Σ λ_i · big_c_i = msk · H(pk ‖ app_id)  +  app_pk · Y
```

The client unmasks with `app_sk`: `big_C − app_sk · big_Y = msk · H(pk ‖ app_id)`.

The masking term `app_pk · Y` is the only thing preventing the coordinator from reading the derived key directly. The `ckd()` function validates participants, coordinator membership, and duplicate detection, but **never checks that `app_pk` is a non-identity point**: [1](#0-0) 

The `app_pk` is forwarded without inspection all the way into `compute_signature_share()`: [2](#0-1) 

Specifically, line 174 is the vulnerable computation:

```rust
let big_c = big_s + app_pk * y.0;   // if app_pk == identity → big_c == big_s
``` [3](#0-2) 

When `app_pk = G1Projective::identity()`:
- `app_pk * y.0 = identity` for every participant
- `big_c_i = x_i · H(pk ‖ app_id)` (unmasked secret share)
- Coordinator aggregates: `big_C = msk · H(pk ‖ app_id)` — the confidential derived key in plaintext

The coordinator's `CKDOutput.big_c` field directly holds the secret: [4](#0-3) 

Note that `BLS12381G1Group::deserialize()` does reject the identity element, but that guard only applies to wire-format deserialization — it is never invoked when `app_pk` is passed as a native Rust value through the `ckd()` API: [5](#0-4) 

---

### Impact Explanation

**Critical — Disclosure of a confidential derived secret.**

The entire security property of the CKD protocol is that the coordinator (an MPC node holding a share of `msk`) must never learn `msk · H(pk ‖ app_id)`. With `app_pk = identity`, the coordinator's `CKDOutput.big_c` equals that value directly. Any party who can invoke `ckd()` with a crafted `app_pk` — including a malicious client or a compromised node that substitutes the identity for the real `app_pk` — causes the coordinator to hold the confidential derived key in plaintext, defeating the protocol's confidentiality guarantee entirely.

---

### Likelihood Explanation

The attack requires only passing `blstrs::G1Projective::identity()` (a trivially constructable Rust value) as `app_pk`. No cryptographic knowledge, key material, or privileged access is needed. Any unprivileged library caller can trigger this path.

---

### Recommendation

Add an explicit identity-point check on `app_pk` at the top of `ckd()`, mirroring the check already present in `BLS12381G1Group::deserialize()`:

```rust
if app_pk.is_identity().into() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
``` [6](#0-5) 

---

### Proof of Concept

```rust
// Attacker supplies the identity element as app_pk
let malicious_app_pk = blstrs::G1Projective::identity();

// All MPC nodes are called with app_pk = identity
let protocol = ckd(
    &participants,
    coordinator,
    me,
    key_pair,
    app_id,
    malicious_app_pk,   // <-- identity element, no validation error
    rng,
).unwrap();

// After protocol execution, coordinator's CKDOutput.big_c
// equals msk · H(pk ‖ app_id) in plaintext.
// The coordinator can read it directly without knowing app_sk.
let ckd_output = /* coordinator result */;
let confidential_derived_key = ckd_output.big_c(); // == msk · H(pk ‖ app_id)
```

The `unmask` function confirms the equivalence: with `app_pk = identity`, `app_sk = 0`, so `big_c − 0 · big_y = big_c = msk · H(pk ‖ app_id)`.
