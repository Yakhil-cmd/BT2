### Title
Unprivileged Caller Can Permanently Freeze All Staker Block Rewards via `update_rewards` with `disable_rewards=true` - (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is callable by **any address** with no access control, despite the protocol specification explicitly requiring "Only starkware sequencer." An attacker can call it with `disable_rewards = true` on every block, consuming the global `last_reward_block` slot and permanently preventing the legitimate sequencer from distributing consensus rewards to any staker for that block.

---

### Finding Description

The `update_rewards` function in `StakingRewardsManagerImpl` enforces only a single guard: the current block number must be strictly greater than `last_reward_block`. There is no caller identity check. [1](#0-0) 

The function signature and its spec: [2](#0-1) 

The spec explicitly states access control is "Only starkware sequencer": [3](#0-2) 

The critical logic path is:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);  // ← written BEFORE the guard

if disable_rewards || self.is_pre_consensus() {
    return;  // ← exits without distributing rewards
}
``` [4](#0-3) 

`last_reward_block` is a **single global storage slot** shared across all stakers. Once it is written to block N, no further call to `update_rewards` can succeed in block N — for any staker — because the guard `current_block_number > self.last_reward_block.read()` will revert with `REWARDS_ALREADY_UPDATED`. [5](#0-4) 

---

### Impact Explanation

An attacker calls `update_rewards(any_valid_staker_address, disable_rewards: true)` as the first transaction in every block. This:

1. Passes all validation (staker exists, has non-zero balance, block is new).
2. Writes `last_reward_block = current_block`.
3. Returns immediately without distributing any rewards.

The legitimate sequencer's subsequent call to `update_rewards(..., disable_rewards: false)` in the same block reverts with `REWARDS_ALREADY_UPDATED`. The block's rewards are **permanently lost** — there is no mechanism to retroactively credit missed blocks.

This matches the allowed impact: **Permanent freezing of unclaimed yield** (High).

---

### Likelihood Explanation

- The function is publicly callable with no role check.
- The attacker needs only a valid staker address (publicly readable from events/storage) and gas to front-run the sequencer each block.
- The attack is cheap: one transaction per block, no capital required.
- It is sustainable indefinitely, denying all consensus-phase rewards to all stakers.

---

### Recommendation

Add an access control check at the top of `update_rewards` to restrict callers to the authorized sequencer address, consistent with the spec:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.roles.only_sequencer(); // ← enforce "Only starkware sequencer"
    ...
}
```

Alternatively, store `last_reward_block` per-staker rather than globally, so a griefing call on one staker cannot block rewards for all others.

---

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. Attacker monitors the mempool and submits `update_rewards(victim_staker, disable_rewards: true)` at the start of block N, before the sequencer's call.
3. `last_reward_block` is set to N; attacker's tx succeeds with no reward distribution.
4. Sequencer submits `update_rewards(victim_staker, disable_rewards: false)` in block N → reverts `REWARDS_ALREADY_UPDATED`.
5. Victim staker's `unclaimed_rewards_own` is unchanged; block N rewards are permanently gone.
6. Repeat every block.

The test suite already demonstrates that a second call in the same block reverts regardless of the `disable_rewards` flag: [6](#0-5)
