### Title
Gateway Skips `__validate__` Signature Check for Nonce-1 Invoke When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call — the only place where the account's cryptographic signature is verified — for any Invoke V3 transaction with `nonce == 1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because that mempool query returns `true` for **any** transaction from the sender (not only a `deploy_account`), an unprivileged attacker can submit an Invoke with an arbitrary/invalid signature for a victim's undeployed account, have it accepted by the gateway, and — if the attacker outbids the victim on fees — have it replace the victim's legitimate first Invoke in the mempool.

---

### Finding Description

The gateway stateful validation path is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range + resource bounds)
       ├─ validate_by_mempool            (mempool nonce + fee-escalation check)
       └─ skip_stateful_validations      ← decides whether __validate__ runs
  └─ run_validate_entry_point           (calls __validate__ unless skip=true)
```

`skip_stateful_validations` is:

```rust

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by that flag:

The design comment reads: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* [4](#0-3) 

This reasoning is broken. `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the sender address — including a `deploy_account` submitted by the victim. An attacker who observes the victim's `deploy_account` (nonce 0) in the mempool can immediately submit their own Invoke with `nonce = 1` and a completely invalid signature. The three conditions for skipping are all satisfied:

| Condition | Value |
|---|---|
| `tx.nonce() == Nonce(Felt::ONE)` | ✓ attacker sets nonce=1 |
| `account_nonce == Nonce(Felt::ZERO)` | ✓ account not yet deployed |
| `account_tx_in_pool_or_recent_block` | ✓ victim's deploy_account is in mempool |

The stateless validator only checks signature **length**, not validity. [5](#0-4)  The mempool's `validate_tx` checks nonce range and fee escalation, not the signature. [6](#0-5)  No other guard verifies the signature before the transaction is forwarded to the mempool.

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

Concrete attack:

1. Victim broadcasts `deploy_account` (nonce 0) + `invoke` (nonce 1).
2. Attacker observes the `deploy_account` in the mempool.
3. Attacker submits `invoke` (nonce 1, victim's address, arbitrary/invalid signature, higher tip/gas price than victim's invoke).
4. Gateway: stateless check passes (signature length ≤ limit), conversion succeeds, nonce-range check passes (0 ≤ 1 ≤ max_gap), `validate_by_mempool` passes fee-escalation (attacker outbids), `skip_stateful_validations` returns `true`, `__validate__` is never called.
5. `mempool.add_tx` replaces the victim's nonce-1 invoke with the attacker's.
6. Block is built: `deploy_account` succeeds; attacker's invoke runs `__validate__` on the now-deployed account → signature fails → transaction reverts.
7. Victim's legitimate first invoke is permanently displaced; the account is deployed but the intended first action never executes.

The gateway has accepted a transaction whose signature was never verified — a direct violation of the admission invariant.

---

### Likelihood Explanation

The attack requires only:
- Observing the mempool for `deploy_account` transactions (public information).
- Submitting a crafted Invoke with nonce 1 and a higher fee than the victim's.

No privileged access, no special keys, no dependency on external services. The window is the time between the victim's `deploy_account` entering the mempool and being included in a block — typically multiple seconds to minutes.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **`deploy_account` transaction** for the sender is pending. The skip should only be granted when the sequencer can confirm that the account's deployment is queued, not merely that any transaction from that address exists in the mempool. Alternatively, require that the nonce-1 invoke carry a valid signature even in the skip-validate path by running `__validate__` unconditionally and only skipping the fee/balance pre-checks.
