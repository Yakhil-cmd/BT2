Audit Report

## Title
`approve_genesis_kyc` Hotkey Inflation Causes Permanent KYC Proposal Execution Failure - (File: `rs/nns/governance/src/neuron_store.rs`)

## Summary
`approve_genesis_kyc` builds its neuron map using `get_neuron_ids_readable_by_caller`, which returns neurons where the target principal is either the controller **or a hotkey**. The 1000-neuron cap is enforced against this inflated count, but the actual KYC mutation only fires for neurons where the principal is the controller. An attacker holding 1001+ neurons can add any target principal as a hotkey to those neurons, causing the cap to be exceeded and the entire proposal execution to fail with `PreconditionFailed` — leaving all principals in the batch un-KYC'd.

## Finding Description
In `rs/nns/governance/src/neuron_store.rs`, `approve_genesis_kyc` collects neuron IDs via `get_neuron_ids_readable_by_caller`: [1](#0-0) 

The test at `neuron_store_tests.rs` confirms this function returns neurons for both the controller and any hotkey principal: [2](#0-1) 

The cap check then rejects the entire operation if the combined count exceeds 1000: [3](#0-2) 

However, the mutation loop only KYC-verifies neurons where the principal is the **controller**, not a hotkey: [4](#0-3) 

The mismatch is: hotkey-linked neurons are counted toward the cap but never mutated. An attacker with 1001 neurons calls `AddHotKey { new_hot_key: P }` on all of them (no consent from `P` required). When the proposal executes via: [5](#0-4) 

`get_neuron_ids_readable_by_caller(P)` returns 1001+ entries, the cap is exceeded, and `Err(PreconditionFailed)` is returned — no neurons are KYC'd.

## Impact Explanation
Genesis neurons with `kyc_verified = false` cannot spawn child neurons and cannot disburse their stake after dissolving. An attacker can permanently block KYC approval for any targeted principal by maintaining the hotkey relationship across all retry proposals. This constitutes a significant NNS governance impact with concrete, lasting harm to affected genesis neuron holders — matching the **High ($2,000–$10,000)** impact class for NNS/governance security impact with concrete user harm.

## Likelihood Explanation
The attacker must hold 1001+ neurons (minimum ~1001 ICP at stake). Large ICP holders exist on mainnet. Adding a hotkey is a cheap, permissionless `manage_neuron` call requiring no consent from the hotkey principal. The NNS voting period is 4 days, giving ample time to add hotkeys after a proposal is submitted. The attack is repeatable indefinitely at negligible ongoing cost (hotkeys can be re-added after any retry proposal is submitted). `ApproveGenesisKyc` is rare but remains a live, executable governance action type.

## Recommendation
Replace `get_neuron_ids_readable_by_caller` in `approve_genesis_kyc` with a lookup that returns only neurons **controlled** by the principal (i.e., `neuron.controller() == principal`). The set used to enforce the 1000-neuron cap must match the set that will actually be mutated. Alternatively, filter `neuron_id_to_principal` after collection to retain only entries where the principal is the controller before applying the cap check. [1](#0-0) 

## Proof of Concept
1. Attacker holds 1001 neurons, each controlled by their own principal `A`.
2. NNS submits and passes `ApproveGenesisKyc { principals: [P] }` for genesis principal `P`.
3. Before execution, attacker calls `manage_neuron` → `AddHotKey { new_hot_key: P }` on all 1001 neurons (no consent from `P` needed).
4. Proposal executes: `get_neuron_ids_readable_by_caller(P)` returns 1001 neuron IDs (attacker's neurons via hotkey + `P`'s own neurons).
5. `neuron_id_to_principal.len() > 1000` → `Err(PreconditionFailed)` → proposal marked failed, `P`'s neurons remain `kyc_verified = false`.
6. Any retry proposal listing `P` is blocked the same way. Attacker re-adds hotkeys after each new proposal is submitted.

A deterministic unit test can reproduce this by constructing a `NeuronStore` with 1001 neurons all having `P` as a hotkey, calling `approve_genesis_kyc` with `principals: [P]`, and asserting the returned error is `PreconditionFailed`.
