### Title
Pool admin can set `drawdownE6 = 1e6` to silently disable stop-loss protection for all bins — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
An off-by-one in `_validateDrawdown` permits `drawdownE6 = E6 = 1e6`. When this value is stored and later read in `_afterSwapOracleStopLoss`, `floorMultiplier` becomes `0`, making the breach condition in `_applyWatermark` a `uint256 < 0` comparison that is permanently `false`. The stop-loss ceases to block any swap.

### Finding Description
`_validateDrawdown` rejects only values strictly greater than `E6`:

```solidity
if (drawdownE6 > E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
```

A pool admin calls `proposeOracleStopLossDrawdown(pool, 1e6)`, waits for the timelock, then calls `executeOracleStopLossDrawdown`. The stored `drawdownE6` becomes `1e6`. On the next swap, `_afterSwapOracleStopLoss` computes:

```solidity
uint256 floorMultiplier = E6 - drawdown;  // = 0
```

`_applyWatermark` then evaluates:

```solidity
breached = metric < (hwm * 0) / E6;  // always false
```

No revert is ever issued. The extension continues to update watermarks (ratcheting them up on new highs) but never triggers a stop-loss, regardless of how much bin value has been extracted.

### Impact Explanation
The stop-loss extension is the primary mechanism protecting LP principal from value extraction through swaps. With it disabled, a pool admin (or any actor who can influence the oracle mid-price) can drain bin balances to near-zero through repeated swaps without the extension halting execution. LPs suffer direct loss of principal with no on-chain recourse once the drawdown is executed.

### Likelihood Explanation
The pool admin role is semi-trusted and the timelock is the only delay. Any pool admin who proposes `drawdown = 1e6` and waits for the timelock can execute this. LPs must monitor pending proposals and exit before execution — a realistic failure mode, especially for passive LPs.

### Recommendation
Change the validation to reject `drawdownE6 == E6` as well:

```solidity
// Before
if (drawdownE6 > E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);

// After
if (drawdownE6 >= E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
```

Alternatively, add a symmetric early-exit alongside the `drawdown == 0` guard:

```solidity
if (drawdown == 0 || drawdown >= E6) return;
```

### Proof of Concept
1. Deploy pool with `OracleValueStopLossExtension`, `drawdownE6 = 500_000` (50%), `timelock = 0` for test simplicity.
2. Seed a bin with token balances; trigger a swap to record a high watermark.
3. Pool admin calls `proposeOracleStopLossDrawdown(pool, 1_000_000)` → `executeOracleStopLossDrawdown(pool)`.
4. Reduce bin balances to 1 wei (simulate near-total extraction).
5. Trigger `afterSwap`. Assert: no `OracleStopLossTriggered` revert fires.
6. Restore `drawdownE6 = 500_000` and repeat step 5. Assert: `OracleStopLossTriggered` reverts as expected.
