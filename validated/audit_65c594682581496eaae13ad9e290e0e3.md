### Title
Any contract can overwrite another account's `AccountId`-mode global contract via unauthenticated cross-contract `DeployGlobalContractByAccountId` action — (File: runtime/runtime/src/global_contracts.rs)

### Summary

`action_deploy_global_contract` in `runtime/runtime/src/global_contracts.rs` stores the global contract under `account_id` (the receipt's `receiver_id`) when `deploy_mode == AccountId`. There is no check that `predecessor_id == receiver_id`. The host function `promise_batch_action_deploy_global_contract_by_account_id` in `runtime/near-vm-runner/src/logic/logic.rs` also imposes no such restriction. A malicious contract can therefore create a cross-contract promise targeting any victim account and append a `DeployGlobalContractByAccountId` action carrying arbitrary WASM, overwriting the global contract stored under the victim's account ID without the victim's consent.

### Finding Description

`GlobalContractDeployMode::AccountId` is designed so that the deploying account "owns" the global contract slot keyed by its account ID, and can update it for all downstream users. The protocol already enforces `predecessor_id == receiver_id` for `DeployContract`, `Stake`, `AddKey`, `DeleteKey`, and `DeleteAccount` (documented in `docs/RuntimeSpec/Actions.md` lines 26–32). `DeployGlobalContract` is absent from that list.

**Call path:**

1. Attacker deploys a malicious contract.
2. Malicious contract calls host function `promise_batch_action_deploy_global_contract_by_account_id(promise_idx_to_victim, ...)`.
3. In `promise_batch_action_deploy_global_contract_impl` (`runtime/near-vm-runner/src/logic/logic.rs`), `promise_idx_to_receipt_idx_with_sir` returns `sir = false` (cross-contract), but **no guard rejects the call**.
4. The action is appended to a receipt with `predecessor_id = attacker_contract`, `receiver_id = victim_account`.
5. At execution, `apply_action` dispatches to `action_deploy_global_contract` (`runtime/runtime/src/global_contracts.rs` line 23). No `predecessor_id == receiver_id` check exists.
6. `initiate_distribution` is called with `account_id = victim_account`. For `AccountId` mode, `id = GlobalContractIdentifier::AccountId(victim_account)` (line 154). The global contract slot keyed by the victim's account ID is overwritten with the attacker's WASM.
7. Storage cost is deducted from the **victim's** balance (`account.set_amount(updated_balance)`, line 49).

The nonce mechanism (`check_and_update_nonce`) only prevents stale distribution receipts from overwriting newer ones; it does not prevent an attacker from issuing a fresh deployment with a higher nonce.

### Impact Explanation

- **Code identity corruption**: Every account that previously called `UseGlobalContract(GlobalContractIdentifier::AccountId(victim_account_id))` now executes the attacker's WASM when invoked. The attacker can insert arbitrary logic (drain attached deposits, corrupt storage, emit false logs).
- **Balance drain**: Each overwrite deducts `global_contract_storage_amount_per_byte × code_len` from the victim's balance. Repeated attacks drain the victim's NEAR.
- **Irreversibility**: The victim cannot prevent further overwrites; the attacker can keep issuing new deployments with monotonically increasing nonces.

### Likelihood Explanation

Any unprivileged user can deploy a contract and invoke the host function. No special key, validator role, or governance access is required. The only prerequisite is that the victim account holds enough NEAR to cover one storage payment; any funded account qualifies. The attack is cheap (gas + one storage payment per overwrite) and fully scriptable.

### Recommendation

Add a `predecessor_id == receiver_id` guard inside `action_deploy_global_contract` for `GlobalContractDeployMode::AccountId`, mirroring the existing restriction on `DeployContract`. Alternatively, enforce the restriction at the host-function level by checking `sir == true` in `promise_batch_action_deploy_global_contract_impl` when `mode == AccountId` and returning `HostError::ProhibitedInView` or a new `HostError::RequiresSelfPromise`.

### Proof of Concept

```rust
// Attacker's contract (pseudo-code using NEAR SDK host functions)
#[no_mangle]
pub fn attack() {
    // Create a promise targeting the victim account
    let promise = env::promise_batch_create("victim.near");
    // Append DeployGlobalContractByAccountId with malicious WASM
    env::promise_batch_action_deploy_global_contract_by_account_id(
        promise,
        MALICIOUS_WASM,
    );
    env::promise_return(promise);
}
```

After execution:
- `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId("victim.near") }` contains the attacker's WASM.
- All accounts whose `account.contract()` is `AccountContract::GlobalByAccount("victim.near")` now execute attacker code on every function call.
- `victim.near`'s balance is reduced by `global_contract_storage_amount_per_byte × malicious_wasm_len`.

pub(crate) fn action_deploy_global_contract(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    account_id: &AccountId,
    apply_state: &ApplyState,
    deploy_contract: &DeployGlobalContractAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let _span = tracing::debug_span!(target: "runtime", "action_deploy_global_contract").entered();

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {
        result.result = Err(ActionErrorKind::LackBalanceForState {
            account_id: account_id.clone(),
            amount: storage_cost,
        }
        .into());
        return Ok(());
    };
    result.tokens_burnt =
        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;
    account.set_amount(updated_balance);

    initiate_distribution(
        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}

fn initiate_distribution(
    state_update: &mut TrieUpdate,
    account_id: AccountId,
    contract_code: Arc<[u8]>,
    deploy_mode: &GlobalContractDeployMode,
    current_shard_id: ShardId,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let id = match deploy_mode {
        GlobalContractDeployMode::CodeHash => {
            GlobalContractIdentifier::CodeHash(hash(&contract_code))
        }
        GlobalContractDeployMode::AccountId => {
            GlobalContractIdentifier::AccountId(account_id.clone())
        }
    };
    // Increment the nonce and write it to state immediately to prevent multiple
    // distributions with the same nonce from being initiated. This requires
    // allowing the same nonce in the freshness check when applying the
    // distribution receipt.
    let nonce = increment_nonce(state_update, &id)?;
    let distribution_receipt =
        GlobalContractDistributionReceipt::new(id, current_shard_id, vec![], contract_code, nonce);
    let distribution_receipts =
        Receipt::new_global_contract_distribution(account_id, distribution_receipt);
    // No need to set receipt_id here, it will be generated as part of apply_action_receipt
    result.new_receipts.push(distribution_receipts);
    Ok(())
