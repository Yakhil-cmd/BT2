Looking at the codebase for an analog to the "parallel arrays without length validation" vulnerability class, I need to find places where multiple parallel `Vec` fields received from network participants are indexed without length checks.

The most concrete analog is in `do_generation_many` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs`. The function receives `PolynomialCommitmentsMessageMany` (and other multi-Vec messages) from each participant and immediately indexes into their `Vec` fields using `[i]` for `i in 0..N` ã with no prior check that each Vec actually contains N elements.

---

### Title
Missing Vector Length Validation on Received Multi-Vec Messages in `do_generation_many` Allows Malicious Participant to Panic-Abort Triple Generation ã (File: `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
`do_generation_many` receives several messages from protocol participants that each contain multiple parallel `Vec` fields, all expected to have exactly `N` elements. None of these received Vecs are length-validated before being indexed with `[i]` for `i in 0..N`. A malicious participant can send any of these messages with fewer than `N` elements, triggering an index-out-of-bounds panic in the honest party's async task, aborting triple generation and permanently denying OT-based ECDSA signing for honest parties.

### Finding Description

`do_generation_many<const N: usize>` processes five distinct received message types, each containing `Vec` fields that must have exactly `N` elements. In every case the code directly indexes with `[i]` inside a `for i in 0..N` loop, with no prior length check:

**1. `Vec<Commitment>` received at `wait0`:** [1](#0-0) 
```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]); // panics if commitments.len() < N
}
```

**2. `PolynomialCommitmentsMessageMany` received at `wait2`** ã six parallel Vecs, all indexed without length checks: [2](#0-1) 
```rust

    let their_big_e    = &their.big_e_v[i];      // panics if len < N

    let their_big_l    = &their.big_l_v[i];      // panics if len < N
```

The message structs are `#[derive(Serialize, Deserialize)]` with plain `Vec` fields: [7](#0-6) 

Serde imposes no constraint that all Vec fields share the same length, and no constraint that any Vec has exactly `N` elements. A malicious participant can craft a valid serialization of any of these message types with fewer than `N` elements in any field. Deserialization succeeds; the subsequent `[i]` index panics.

### Impact Explanation

Triple generation via `generate_triple` / `generate_triple_many` is the mandatory prerequisite for OT-based ECDSA presigning and signing: [8](#0-7) 

A panic inside the async future aborts the honest party's protocol task. Because the malicious participant can repeat this attack in every session they are part of, any participant who is a required member of the signing group can permanently deny triple generation ã and therefore signing ã for all honest parties in that group. This matches **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions**.

### Likelihood Explanation

The attack requires only that the adversary be a legitimate participant in a `generate_triple_many` session (which is the normal precondition for signing). No cryptographic capability is needed ã only the ability to send a well-formed but length-truncated serialized message. The attack is deterministic and repeatable every session.

### Recommendation

Before the `for i in 0..N` loop over any received Vec, validate that every parallel Vec field has exactly `N` elements and return a `ProtocolError` on mismatch. For example, at the `wait2` receive site:

```rust
if their.big_e_v.len() != N
    || their.big_f_v.len() != N
    || their.big_l_v.len() != N
    || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N
    || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(format!(
        "message from {from:?} has wrong number of elements (expected {N})"
    )));
}
```

Apply the same pattern to all five other receive sites (`wait0`, `wait3`, `wait4`, `wait5`, `wait6`).

### Proof of Concept

1. Honest parties call `generate_triple_many::<4>(...)` (N = 4).
2. Malicious participant serializes a `PolynomialCommitmentsMessageMany` where `big_f_v` is an empty `Vec` (0 elements) while all other fields have 4 elements. This is a valid msgpack encoding.
3. Malicious participant sends this message at `wait2`.
4. Honest party deserializes successfully (serde does not enforce length equality).
5. Honest party enters `for i in 0..4` and evaluates `their.big_f_v[0]` ã index out of bounds panic.
6. The async task aborts; `generate_triple_many` returns an error/panic to the caller.
7. No triples are produced; OT-based ECDSA signing is denied for this session.
8. Malicious participant repeats in every subsequent session, permanently blocking signing.
