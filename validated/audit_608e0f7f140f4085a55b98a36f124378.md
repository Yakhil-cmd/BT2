Audit Report

## Title
Unprivileged Cycle-Funded DoS via `OUTCALL_CAPACITY` Exhaustion Causes Minter to Return `TemporarilyUnavailable` — (`rs/bitcoin/checker/src/state.rs`, `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
`check_transaction` is a public update method with no caller restriction beyond a cycles payment. An unprivileged attacker can exhaust all `OUTCALL_CAPACITY` slots by sending concurrent calls with distinct uncached txids. While slots are held, the minter's `check_utxo` receives `CheckTransactionRetriable::HighLoad` and immediately returns `UpdateBalanceError::TemporarilyUnavailable` without retrying, blocking ckBTC minting for all users for the duration of the attack.

## Finding Description
`check_transaction` at [1](#0-0)  accepts calls from any principal; the only gate is a cycles acceptance check at [2](#0-1) .

`FetchGuard::new` decrements a shared `OUTCALL_CAPACITY` counter and returns `FetchGuardError::NoCapacity` when it reaches zero: [3](#0-2) 

The guard is held for the entire duration of the HTTP outcall; `Drop` restores the slot only after the outcall completes: [4](#0-3) 

When `new_fetch_guard` fails, `try_fetch_tx` returns `TryFetchResult::HighLoad`: [5](#0-4) 

This propagates to `CheckTransactionRetriable::HighLoad` in `check_transaction_inputs`: [6](#0-5) 

In `check_utxo`, the retry loop (`MAX_CHECK_TRANSACTION_RETRY = 10`) only continues on `NotEnoughCycles`. On `Retriable(HighLoad)` it immediately returns an error without retrying: [7](#0-6) 

## Impact Explanation
All users calling `update_balance` during the attack window receive `UpdateBalanceError::TemporarilyUnavailable` and must retry manually. This is a concrete application-level DoS against ckBTC minting — a financial integration component — matching the allowed impact: **High ($2,000–$10,000): Application/platform-level DoS with concrete user and protocol harm, not based on raw volumetric DDoS.** The attack exploits a specific finite-capacity design flaw, not raw traffic volume.

## Likelihood Explanation
The attack requires sustained cycle expenditure (capacity × `CHECK_TRANSACTION_CYCLES_REQUIRED` per round-trip, repeated continuously). This is an economic deterrent but not a hard barrier. The checker canister is publicly accessible on mainnet. The DoS is temporary and self-healing: once the attacker's HTTP outcalls complete, `FetchGuard::drop` restores capacity. The attacker must continuously send new calls to sustain the effect.

## Recommendation
1. Add retry logic for `Retriable(HighLoad)` in `check_utxo` (with bounded retries and backoff), mirroring the existing `NotEnoughCycles` `continue` path at [8](#0-7) 
2. Consider per-caller rate limiting or a higher minimum cycle cost to make sustained slot exhaustion economically prohibitive.
3. Increase `OUTCALL_CAPACITY` or implement per-caller slot quotas to reduce blast radius.

## Proof of Concept
A state-machine test can:
1. Acquire all `OUTCALL_CAPACITY` `FetchGuard`s simultaneously using distinct uncached txids.
2. Assert that `FetchGuard::new` returns `Err(FetchGuardError::NoCapacity)` on the next attempt, as defined at [9](#0-8) 
3. Simulate a minter `check_utxo` call receiving `CheckTransactionRetriable::HighLoad` and assert it returns `UpdateBalanceError::TemporarilyUnavailable` without iterating further, as shown at [7](#0-6)

impl FetchGuard {
    pub fn new(txid: Txid) -> Result<Self, FetchGuardError> {
        let guard = OUTCALL_CAPACITY.with(|capacity| {
            let mut capacity = capacity.borrow_mut();
            if *capacity > 0 {
                *capacity -= 1;
                Ok(FetchGuard(txid))
            } else {
                Err(FetchGuardError::NoCapacity)
            }
        })?;
        set_fetch_status(txid, FetchTxStatus::PendingOutcall);
        Ok(guard)
    }

        OUTCALL_CAPACITY.with(|capacity| {

            *capacity += 1;

            // Only clear the status when it is still `PendingOutcall`

    }
