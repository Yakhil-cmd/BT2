### Title
Attacker Can Bypass Signature Validation for Invoke Transactions Targeting Undeployed Accounts via `skip_stateful_validations` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call for any invoke transaction whose `nonce == 1` and whose `sender_address` has any transaction in the mempool or a recent block. An attacker who observes a victim's pending `deploy_account` transaction in the mempool can submit an invoke transaction with the victim's `sender_address`, `nonce = 1`, arbitrary calldata, and a garbage signature. The gateway admits this transaction without ever calling `__validate__`, violating the invariant that every admitted transaction must carry a valid account signature.

### Finding Description

`skip_stateful_validations` is the Sequencer analog of the VaderPoolV2 `from`-argument bug: both allow an arbitrary caller to substitute a victim's identity into a privileged operation without the victim's authorization.

The exact code path is:

1. **Stateless validation** (`StatelessTransactionValidator::validate`) performs no signature check; it only validates field formats and resource bounds.
2. **Conversion** (`TransactionConverter::convert_rpc_tx_to_internal`) computes the transaction hash but does not verify the signature.
3. **Stateful validation** (`extract_state_nonce_and_run_validations`) calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations`, the invoke branch immediately returns `Ok(())` without calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:79-81
if !tx.execution_flags.validate {
    return Ok(());
}
```

The three conditions that trigger the skip are all attacker-controllable or observable:
- `tx.nonce() == Nonce(Felt::ONE)` ÔÇö attacker sets this field
- `account_nonce == Nonce(Felt::ZERO)` ÔÇö observable: victim's account is not yet deployed
- `account_tx_in_pool_or_recent_block(sender_address)` ÔÇö observable: victim's `deploy_account` is in the mempool

### Impact Explanation

The gateway admits an invoke transaction with a garbage signature for a victim's undeployed account. This breaks the admission invariant: **every transaction admitted to the mempool must carry a signature that would pass the account's `__validate__` entry point**.

Concrete consequences:
1. **Invalid transaction admission**: The gateway accepts a transaction it should reject, matching the High impact category "Mempool/gateway/RPC admission accepts invalid transactions."
2. **Nonce-slot griefing**: The attacker's transaction occupies the victim's nonce-1 slot in the mempool. If the attacker pays a higher fee, `validate_fee_escalation` will replace the victim's legitimate nonce-1 transaction with the attacker's invalid one. The victim's legitimate transaction is evicted and must be resubmitted.
3. **Block-space waste**: The invalid transaction will be included in a block proposal, fail during `__validate__` at execution time (since `AccountTransaction::new_for_sequencing` sets `validate: true`), and be dropped ÔÇö wasting block space and sequencer resources.

### Likelihood Explanation

The attack requires only:
1. Observing the public mempool for a pending `deploy_account` transaction (trivially observable).
2. Submitting a well-formed invoke transaction with the victim's address, `nonce = 1`, any calldata, and any signature bytes.

No privileged access, no special contract deployment, and no cryptographic capability is required. Any unprivileged network participant can execute this attack against any account that is in the process of being deployed.

### Recommendation

Remove the `account_tx_in_pool_or_recent_block` shortcut from the gateway's stateful validation path. The UX goal (allowing a nonce-1 invoke to be submitted alongside a `deploy_account`) can be preserved by deferring the signature check to block-building time rather than skipping it entirely at the gateway. Alternatively, if the skip must remain, the gateway should verify that the transaction's signature is structurally valid (e.g., correct length and format) even when `__validate__` execution is deferred, so that obviously forged signatures are rejected before mempool admission.

### Proof of Concept

**Setup**: Victim V submits a `deploy_account` transaction (nonce 0) to the gateway. It is accepted and sits in the mempool. `account_tx_in_pool_or_recent_block(V)` now returns `true`.

**Attack**:
```
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "version": "0x3",
  "sender_address": <V>,          // victim's address
  "nonce": "0x1",                 // exactly 1
  "calldata": [...drain funds...],
  "signature": ["0xdeadbeef", "0xdeadbeef"],  // garbage
  "resource_bounds": { ... valid ... }
}
```

**Trace through gateway**:
1. `StatelessTransactionValidator::validate` ÔÇö passes (no signature check).
2. `convert_rpc_tx_to_internal` ÔÇö passes (hash computed, no sig check).
3. `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(V)` Ôćĺ `0` (not deployed).
   - `validate_nonce`: nonce 1 is within `[0, max_gap]` Ôćĺ passes.
   - `validate_by_mempool` Ôćĺ passes (no duplicate hash).
   - `skip_stateful_validations`: nonce==1 Ôťô, account_nonce==0 Ôťô, `account_tx_in_pool_or_recent_block(V)` Ôćĺ `true` Ôťô Ôćĺ returns `true`.
   - `run_validate_entry_point(skip_validate=true)` Ôćĺ `validate=false` Ôćĺ `StatefulValidator` returns `Ok(())` without calling `__validate__`.
4. **Transaction admitted to mempool with garbage signature.**

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
