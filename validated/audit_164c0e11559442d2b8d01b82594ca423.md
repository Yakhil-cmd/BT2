### Title
`MsgTurnBridge` Is a No-Op — Bridge Circuit Breaker Has No Effect - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` message handler, which is intended to allow a permissioned address to disable the Gravity/IBC bridge in an emergency, is implemented as a complete no-op. It returns `nil, nil` without writing any state. As a result, the bridge can never be disabled regardless of admin or permissioned-address action.

### Finding Description
`MsgTurnBridge` is a defined Cosmos SDK message type with a dedicated permission bit (`CanTurnBridge`). The `UpdatePermissions` flow correctly grants and stores this permission. However, the handler that is supposed to act on it does nothing:

```go
// x/cronos/keeper/msg_server.go:85-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

No bridge-enabled/disabled flag is ever written to the KV store. Because no state is ever set, any downstream check of that flag (in IBC handlers, EVM hooks, or gravity bridge handlers) would always see the default "enabled" value. The circuit breaker is permanently non-functional.

This is the direct Cronos analog of the external report: in the external bug, `cancelled[orderHash] = true` is written but never read; here, the bridge-disable state is never written at all, producing the same outcome — the intended control action has no effect.

### Impact Explanation
The Gravity Bridge circuit breaker is the primary emergency response mechanism for bridge exploits. With `TurnBridge` being a no-op, a permissioned address or admin invoking `MsgTurnBridge` to halt a live bridge exploit will succeed at the transaction level (no error returned) but will not stop any bridge or conversion flows. This constitutes a **bypass of Cronos admin/bridge authorization controls** — the admin's authority over the bridge is silently nullified.

Impact classification: **High** — Bypass of Cronos admin bridge authorization check.

### Likelihood Explanation
The vulnerability is always present and requires no special conditions to trigger. Any time an admin or permissioned address attempts to use the bridge circuit breaker (e.g., during a live exploit), the call silently succeeds but has zero effect. The `CanTurnBridge` permission is already defined and grantable, making this a reachable, unprivileged-triggerable path (the attacker simply continues exploiting the bridge knowing the circuit breaker cannot stop them).

### Recommendation
Implement `TurnBridge` to write a bridge-enabled boolean to the KV store (analogous to how `StoreBlockList` writes to `types.KeyPrefixBlockList`), and add a guard in all bridge-initiating paths (IBC transfer handlers, EVM log handlers for `__CronosSendToEvmChain`, and gravity bridge hooks) that checks this flag and aborts if the bridge is disabled.

### Proof of Concept
1. Admin grants `CanTurnBridge` permission to address `A` via `MsgUpdatePermissions`.
2. Address `A` submits `MsgTurnBridge{Enable: false}`.
3. Transaction succeeds with code 0 and no error.
4. Bridge state in KV store is unchanged (no key written).
5. A user calls `send_to_evm_chain` on a CRC21 contract — the EVM hook fires, tokens are burned, and the Gravity bridge outbound transfer is queued as normal.
6. The bridge was never disabled; the circuit breaker had no effect. [1](#0-0) [5](#0-4) [4](#0-3)
