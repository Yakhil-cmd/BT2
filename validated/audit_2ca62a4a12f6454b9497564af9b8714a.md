### Title
L2AssetTracker Revert After Protocol Upgrade Permanently Halts Chain for Pending L1 Priority Deposits — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function unconditionally treats any revert of `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` as a fatal, block-halting error. Because L1 priority transactions cannot be skipped or invalidated (doing so would itself halt the chain via the priority queue), any pending L1→L2 priority transaction carrying a deposit (`to_mint > 0`) becomes an irremovable chain-stopper if the `L2AssetTracker` contract is subsequently upgraded to a version that reverts that function. This is the direct ZKsync OS analog of the USDC bridge bug: a legitimate protocol state change (migration/upgrade) invalidates already-queued messages, causing permanent fund loss — here escalated to a full chain halt.

---

### Finding Description

Every L1→L2 priority transaction with `total_deposited > 0` triggers `mint_base_token`, which first calls `notify_l2_asset_tracker` and then `transfer_from_treasury`. [1](#0-0) 

Inside `notify_l2_asset_tracker`, the bootloader calls `handleFinalizeBaseTokenBridgingOnL2(fromChainId, amount)` on the `L2AssetTracker` contract (address `0x800b`) impersonating `L2_BASE_TOKEN_ADDRESS` (0x800a): [2](#0-1) 

If the call reverts for **any** reason, the code explicitly treats it as a fatal system error that stops block processing entirely: [3](#0-2) 

The comment in the code acknowledges the assumption:

> "If no contract is deployed at L2AssetTracker, the call succeeds silently… However, we are certain that L2AssetTracker is available after the upgrade." [4](#0-3) 

This assumption is fragile. The `L2AssetTracker` is an upgradeable EVM contract. Its `handleFinalizeBaseTokenBridgingOnL2` function has internal guards (asset registration check via `isAssetRegistered`, `assetMigrationNumber` matching, pausability) that can legitimately revert after a protocol upgrade changes the contract's state. The predeployed contract setup confirms these state variables exist and are initialized: [5](#0-4) 

The `notify_l2_asset_tracker` call is made **three times** per L1 deposit transaction — for the value mint, operator fee, and refund — all on `FORMAL_INFINITE` resources for the post-execution calls: [6](#0-5) [7](#0-6) 

The fatal error propagates up through `process_l1_transaction` and halts the entire block. Since L1 transactions cannot be invalidated (the priority queue design enforces this), the chain is permanently stuck: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

**Impact: Critical — permanent chain halt.**

If the `L2AssetTracker` is upgraded to a version that reverts `handleFinalizeBaseTokenBridgingOnL2` (e.g., due to changed asset registration state, a new access control modifier, a paused state, or a bug in the new implementation), any pending L1 priority transaction with `to_mint > 0` will cause block processing to halt with a fatal internal error. Because L1 transactions cannot be skipped or invalidated, the chain cannot advance past that transaction. All user funds on L2 become inaccessible. This is strictly worse than the USDC bridge bug (individual fund lock) — it locks the entire chain.

---

### Likelihood Explanation

**Likelihood: Low.**

The scenario requires two conditions to coincide:
1. A user submits an L1→L2 priority transaction with a deposit (`to_mint > 0`) that is queued but not yet processed.
2. The `L2AssetTracker` is subsequently upgraded (a legitimate protocol operation) to a version whose `handleFinalizeBaseTokenBridgingOnL2` reverts for the given parameters.

This mirrors the USDC bug's likelihood profile: unlikely in normal operation, but the consequence of the race condition is catastrophic and irreversible. The risk is elevated during any protocol upgrade that touches `L2AssetTracker`, which is a planned and recurring event.

---

### Recommendation

1. **Degrade gracefully instead of halting**: Change `notify_l2_asset_tracker` so that a revert of `handleFinalizeBaseTokenBridgingOnL2` does not halt block processing. Instead, log the failure and continue — analogous to how the USDC fix wraps the mint in a try/catch. The accounting inconsistency concern cited in the comment can be addressed by emitting an L2→L1 log so the discrepancy is observable and recoverable on L1.

2. **Decouple the fatal path from the L2AssetTracker state**: The bootloader should not treat an upgradeable application contract's revert as a chain-fatal condition. Consider making the `notify_l2_asset_tracker` call best-effort, with a fallback that records the missed notification for later reconciliation.

3. **Add a pre-upgrade invariant check**: Before any `L2AssetTracker` upgrade, verify that `handleFinalizeBaseTokenBridgingOnL2` will still succeed for the current asset configuration, and that no L1 priority transactions with deposits are pending in the queue.

---

### Proof of Concept

**Attack sequence:**

1. User Alice submits an L1→L2 priority transaction with `to_mint = 1 ETH` on L1. The transaction enters the priority queue and is not yet processed on L2.

2. The protocol governance executes an upgrade transaction that deploys a new `L2AssetTracker` implementation. The new implementation adds a `whenNotPaused` modifier to `handleFinalizeBaseTokenBridgingOnL2`, or changes the `assetMigrationNumber` check, or deregisters the base token asset — any change that causes the function to revert.

3. The operator attempts to include Alice's L1 priority transaction in the next block. The bootloader calls `notify_l2_asset_tracker` with `amount = 1 ETH`.

4. The upgraded `L2AssetTracker` reverts `handleFinalizeBaseTokenBridgingOnL2`.

5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.

6. `process_l1_transaction` propagates the fatal error. Block processing halts.

7. The operator cannot skip Alice's transaction (it is in the priority queue). The chain is permanently stuck. All L2 user funds are inaccessible.

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
}
