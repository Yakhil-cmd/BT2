### Title
CKD Coordinator Accepts Participant Contributions Without Proof of Correctness - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator in `do_ckd_coordinator` blindly accumulates participant-supplied `(big_y, big_c)` group elements with no proof of correctness or consistency check. A single malicious participant can send arbitrary curve points, silently corrupting the final `CKDOutput` for all honest parties and causing the derived confidential key to be wrong.

### Finding Description

In `do_ckd_coordinator` the coordinator collects each participant's share and sums them unconditionally:

```rust
// src/confidential_key_derivation/protocol.rs lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant computes its share via `compute_signature_share` (lines 148–182), which produces:

- `big_y = y_i · G` (a blinding commitment)
- `big_c = x_i · H(pk ‖ app_id) + y_i · A` (the masked key share)

These are then Lagrange-weighted and sent privately to the coordinator. The coordinator has no mechanism to verify that the received `(big_y, big_c)` pair was actually derived from the sender's legitimate key share `x_i` and a consistent blinding scalar `y_i`. There is no ZK proof, no consistency check against the participant's public key share, and no identity-element guard on the received points.

The root cause is structurally identical to the external report: a function that aggregates contributions is called with zero validation on each input, analogous to calling `depositUnderlying` with `minOut = 0`.

**Attacker-controlled entry path:**
1. A malicious participant `P_m` participates in the CKD protocol.
2. Instead of calling `compute_signature_share` honestly, `P_m` constructs an arbitrary `CKDOutput` with attacker-chosen `(big_y', big_c')`.
3. `P_m` sends this forged output privately to the coordinator via `chan.send_private`.
4. The coordinator's `recv_from_others` delivers the forged values; the loop at lines 50–55 adds them without any check.
5. The final `CKDOutput::new(norm_big_y, norm_big_c)` is now poisoned.

### Impact Explanation

The corrupted `CKDOutput` causes `ckd_output.unmask(app_sk)` to return a wrong group element instead of `msk · H(pk ‖ app_id)`. Every downstream consumer of the derived confidential key receives an incorrect secret. Because the CKD output carries no authentication tag, honest parties cannot distinguish a corrupted output from a legitimate one. This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

Any single participant in the CKD protocol can mount this attack with no special privilege. The participant role is the lowest-trust role in the protocol. The attack requires only the ability to send a message to the coordinator, which is the normal protocol flow. No cryptographic break, no key leakage, and no coordinator compromise is needed.

### Recommendation

Each participant should accompany its `(big_y, big_c)` with a non-interactive ZK proof demonstrating:
1. Knowledge of `y_i` such that `big_y = y_i · G`.
2. That `big_c - y_i · A` equals `x_i · H(pk ‖ app_id)`, where `x_i · G` is the participant's publicly committed key share from DKG.

This is a discrete-log-equality proof (the same `dlogeq` primitive already used in OT-based ECDSA triple generation at `src/ecdsa/ot_based_ecdsa/triples/generation.rs`). The coordinator must verify each proof before accumulating the contribution, and abort if any proof fails.

### Proof of Concept

A malicious participant replaces its honest `compute_signature_share` output with an arbitrary pair, e.g., `(big_y' = r · G, big_c' = s · G)` for random scalars `r, s`. It sends this pair to the coordinator. The coordinator's loop adds `(r · G, s · G)` to the running sum. The resulting `CKDOutput` satisfies:

```
norm_big_y = Σ_{i≠m} λ_i · y_i · G  +  r · G
norm_big_c = Σ_{i≠m} λ_i · C_i      +  s · G
```

Unmasking gives `norm_big_c − app_sk · norm_big_y = msk · H(pk ‖ app_id) + (s − app_sk · r) · G`, which is not the correct confidential key for any honest `app_sk` unless `s = app_sk · r` (negligible probability for random `r, s`). The honest coordinator and all downstream consumers silently accept this wrong value. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
