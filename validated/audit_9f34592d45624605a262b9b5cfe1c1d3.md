### Title
Missing Caller Authorization in `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` is documented as restricted to "Only starkware sequencer" but the implementation enforces no such check. Any unprivileged address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for that block and causing all stakers to permanently lose that block's consensus rewards.

---

### Finding Description

The spec explicitly states the access control for `update_rewards` is **"Only starkware sequencer"**: [1](#0-0) 

The actual implementation's only gate is `general_prerequisites()`, which checks only for pause state and a non-zero caller: [2](#0-1) 

There is no role check, no sequencer check, and no caller identity check anywhere in `update_rewards`: [3](#0-2) 

The `last_reward_block` is a **single global storage slot** shared across all stakers: [4](#0-3) 

The function writes to it unconditionally before checking `disable_rewards`, then returns early without distributing any rewards: [5](#0-4) 

Because `last_reward_block` is global, once it is written for block N, every subsequent call in block N reverts with `REWARDS_ALREADY_UPDATED`. The block's rewards are permanently unrecoverable — there is no catch-up mechanism.

---

### Impact Explanation

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` at the start of every block:

1. Consumes the one-call-per-block slot (sets `last_reward_block` to current block).
2. Exits before any rewards are calculated or credited.
3. Blocks the legitimate sequencer call for that block.
4. Causes **permanent, irrecoverable loss** of that block's consensus rewards for every staker and every delegation pool.

Sustained over many blocks, this zeroes out all consensus-era yield. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

---

### Likelihood Explanation

The function is public, requires no stake, no token approval, and no privileged role. The only requirement is that a valid active staker address exists (trivially satisfied on mainnet). The attack costs only gas per block and can be automated. Likelihood is **High**.

---

### Recommendation

Add a sequencer/operator role check at the top of `update_rewards`, consistent with the spec. For example, assert that `get_caller_address()` equals the registered sequencer address or holds the appropriate role, before writing to `last_reward_block`.

---

### Proof of Concept

1. Deploy the staking contract with two active stakers, A and B, in the consensus-rewards epoch.
2. From an unprivileged EOA, call `update_rewards(staker_address: A, disable_rewards: true)` at the first transaction of block N.
3. Observe: `last_reward_block` is now N; staker A's `unclaimed_rewards_own` is unchanged.
4. Have the legitimate sequencer call `update_rewards(staker_address: B, disable_rewards: false)` — it reverts with `REWARDS_ALREADY_UPDATED`.
5. Advance to block N+1 and repeat. After K blocks, both stakers have zero accumulated consensus rewards despite being eligible every block.

The test `update_rewards_disable_rewards_consensus_rewards_flow_test` already demonstrates that `disable_rewards: true` with consensus active produces zero rewards and that a second call in the same block reverts — confirming the mechanism is exactly as described: [6](#0-5)
