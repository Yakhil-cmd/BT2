### Title
Unbounded `self.stakers` Vec growth permanently degrades `get_stakers` gas cost — (`src/staking/staking.cairo`)

### Summary
The `self.stakers` Vec is append-only: every `stake()` call pushes a new address, and `unstake_action` never removes it. `get_stakers` iterates the **full** Vec with one storage read per entry (`is_staker_active`), making its gas cost O(all-time staker count) regardless of how many are currently active.

### Finding Description

**Root cause 1 — Vec is append-only:**

`stake()` unconditionally pushes every new staker: [1](#0-0) 

The storage comment is explicit: [2](#0-1) 

**Root cause 2 — `remove_staker` never touches the Vec:**

`unstake_action` calls `remove_staker`, which only zeroes `staker_info` and the operational-address mapping. The `stakers` Vec entry is left intact: [3](#0-2) 

**Root cause 3 — `get_stakers` iterates the full Vec:**

Every call to `get_stakers` walks `self.stakers.into_iter_full_range()` and performs a storage read (`staker_info.read`) per entry via `is_staker_active`: [4](#0-3) 

`is_staker_active` reads `self.staker_info` from storage for every entry — one storage read per ghost staker: [5](#0-4) 

**Root cause 4 — address reuse is blocked, so each attacker account inflates the Vec permanently:**

`stake()` calls `assert_staker_address_not_reused` (line 303), preventing a previously-unstaked address from re-staking. This means each address occupies exactly one permanent slot in the Vec. [6](#0-5) 

### Impact Explanation
`get_stakers` is the consensus query endpoint (`IStakingConsensus`). After N accounts stake-then-fully-exit, every future call to `get_stakers` costs O(N) storage reads even when zero stakers are active. Because the Vec is permanent and unbounded, the function can be made arbitrarily expensive, eventually exceeding block gas limits and making consensus queries permanently unusable. [7](#0-6) 

### Likelihood Explanation
The attack requires only unprivileged accounts and the minimum stake amount. The staked funds are **fully returned** after `unstake_action`, so the net cost to the attacker is only transaction gas. The attack is permanent and irreversible — there is no mechanism to prune the Vec.

### Recommendation
Options:
1. **Swap-and-pop on removal**: When a staker is removed, swap its Vec slot with the last entry and pop the tail.
2. **Separate active-staker set**: Maintain a separate, prunable data structure for active stakers and iterate only that in `get_stakers`.
3. **Minimum stake economic deterrent**: Raise the minimum stake high enough that the cost of inflating the Vec is prohibitive — but this is not a code fix.

### Proof of Concept
```
for i in 0..N:
    accounts[i].stake(min_amount)          // stakers Vec grows by 1 each time
    accounts[i].unstake_intent()
    advance_time(exit_wait_window)
    accounts[i].unstake_action()           // staker_info = None, Vec entry stays

// Now stakers Vec has N entries, all inactive.
// Every get_stakers() call does N storage reads.
get_stakers(current_epoch)                 // O(N) reads, gas grows linearly with N
```
