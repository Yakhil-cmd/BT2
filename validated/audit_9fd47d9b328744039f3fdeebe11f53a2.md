### Title
Permanent NEAR Token Loss on Global Contract Re-deployment in AccountId Mode — (`runtime/runtime/src/global_contracts.rs`)

### Summary

`action_deploy_global_contract` permanently burns the full storage cost for every call, including updates to an already-deployed `AccountId`-mode global contract, without refunding the storage cost of the contract being replaced. Because the old contract's trie slot is silently overwritten, the deployer loses NEAR proportional to the old contract's size on every update — a permanent, unrecoverable accounting loss.

### Finding Description

`action_deploy_global_contract` computes and burns a storage cost equal to `global_contract_storage_amount_per_byte × new_code.len()`: [1](#0-0) 

The function then calls `initiate_distribution`, which eventually writes the new code to `TrieKey::GlobalContractCode { identifier: AccountId(account_id) }`, silently overwriting whatever was there before: [2](#0-1) 

There is no guard that checks whether a contract already exists under the same `AccountId` identifier, and no mechanism to refund the storage cost of the displaced contract. The deployer's `account.amount()` is reduced by `rate × new_len` while the old contract's `rate × old_len` worth of tokens — already burnt on the first deployment — is never credited back.

Contrast this with `action_deploy_contract` (regular per-account contracts), which explicitly calls `clear_account_contract_storage_usage` before charging for the new code, so the deployer only pays the net difference: [3](#0-2) 

The `AccountId` deploy mode is explicitly designed for mutable, upgradeable global contracts: [4](#0-3) 

Updating such a contract is therefore a normal, expected operation — not an edge case.

### Impact Explanation

Every update to an `AccountId`-mode global contract permanently burns `global_contract_storage_amount_per_byte × old_code.len()` NEAR with no corresponding storage benefit. At the current rate of `0.0001 N/byte`, a 4 MiB contract costs ≈ 419 NEAR per update. The loss is:

- **Permanent** — tokens are added to `result.tokens_burnt`, not locked as a stake.
- **Cumulative** — each subsequent update incurs the full cost again.
- **Unrecoverable** — no refund path exists in the protocol.

The exact corrupted field is `account.amount()`: it is reduced by `storage_cost_new` on every call without being credited `storage_cost_old` for the freed trie slot.

### Likelihood Explanation

Any unprivileged account holder can deploy a global contract in `AccountId` mode and later update it. Updating is the primary advertised use-case of `AccountId` mode. The existing test `test_global_contract_update` exercises exactly this path but never asserts the deployer's balance after the second deployment, so the loss goes undetected: [5](#0-4) 

### Recommendation

Before charging the storage cost in `action_deploy_global_contract`, check whether a contract already exists under the same `AccountId` identifier. If it does, read the existing code length from the trie (analogous to `get_code_len_or_default` used for regular contracts), compute the net cost `rate × (new_len − old_len)`, and charge only that delta (or issue a refund if the new contract is smaller). This mirrors the accounting already applied in `action_deploy_contract` via `clear_account_contract_storage_usage`. [6](#0-5) 

### Proof of Concept

1. Account `alice.near` deploys a 4 MiB global contract in `AccountId` mode. Her balance decreases by ≈ 419 NEAR (burnt).
2. Alice later deploys a 3 MiB updated version of the same contract. Her balance decreases by another ≈ 314 NEAR (burnt). The old 4 MiB trie slot is overwritten and freed.
3. Net accounting: Alice has paid ≈ 733 NEAR in storage fees but currently occupies only 3 MiB of global contract storage (worth ≈ 314 NEAR). The ≈ 419 NEAR paid for the first deployment is permanently lost with no corresponding storage benefit.
4. Repeating step 2 N times multiplies the loss by N, with no upper bound other than Alice's balance.
