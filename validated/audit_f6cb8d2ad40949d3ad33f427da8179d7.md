### Title
XCC Router `add_batch_actions` Is `unimplemented!()` Without `all-promise-actions` Feature, Causing Permanent Loss of User wNEAR — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router contract's `add_batch_actions` function panics with `unimplemented!()` when compiled without the `all-promise-actions` feature flag. Because that feature is **not** in the default feature set, the production build of the router will panic at runtime whenever a user submits a cross-contract call whose `PromiseArgs` resolves to a `SimpleNearPromise::Batch`. If the user has no pre-existing router account, their wNEAR (≥ 2 NEAR) is irrevocably transferred to the engine's implicit EVM address before the panic occurs, with no refund path.

---

### Finding Description

In `etc/xcc-router/Cargo.toml`, the `all-promise-actions` feature is declared but **not** included in `default`:

```toml
[features]
default = []
all-promise-actions = []
```

The router's `add_batch_actions` has two conditional implementations:

```rust
#[cfg(not(feature = "all-promise-actions"))]
fn add_batch_actions(_id: PromiseIndex, _actions: &[PromiseAction]) {
    unimplemented!("NEAR batch transactions are not supported. ...")
}

#[cfg(feature = "all-promise-actions")]
fn add_batch_actions(id: PromiseIndex, actions: &[PromiseAction]) { ... }
```

`add_batch_actions` is called from `recursive_promise_create` when the promise contains a `SimpleNearPromise::Batch` variant:

```rust
SimpleNearPromise::Batch(batch) => {
    let target = batch.target_account_id.as_ref().parse().unwrap();
    let id = env::promise_batch_create(&target);
    Self::add_batch_actions(id, &batch.actions);  // panics without feature
    id
}
```

`recursive_promise_create` is called from `promise_create`, which is called from both `execute` and `execute_scheduled`:

```rust
pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The XCC precompile (`engine-precompiles/src/xcc.rs`) accepts user-controlled borsh-encoded `CrossContractCallArgs` from EVM calldata, deserializes the inner `PromiseArgs`, and forwards it verbatim to the router's `execute` or `schedule` method:

```rust
let args = CrossContractCallArgs::try_from_slice(input)
    .map_err(|_| ExitError::Other(Cow::from(consts::ERR_INVALID_INPUT)))?;
let (promise, attached_near) = match args {
    CrossContractCallArgs::Eager(call) => {
        ...
        args: borsh::to_vec(&call)...,
```

A user can craft `call` as `PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch(...)))`. The precompile performs no structural validation that would reject this variant.

Critically, **before** the router is called, the precompile transfers the user's wNEAR to the engine's implicit EVM address if `required_near > 0`:

```rust
if required_near != ZERO_YOCTO {
    let engine_implicit_address = aurora_engine_sdk::types::near_account_to_evm_address(
        self.engine_account_id.as_bytes(),
    );
    let tx_data = transfer_from_args(...);
    let (exit_reason, return_value) =
        handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
```

This EVM-level wNEAR transfer is committed to EVM state as part of the `submit` transaction. The subsequent NEAR promise to the router is a separate async call. When the router panics, the NEAR promise fails, but the wNEAR transfer is already finalized. The engine's implicit address is not user-controlled, so the funds are unrecoverable by the user.

---

### Impact Explanation

**High — Permanent freezing of user funds.**

For a user with no existing router account, `required_near = attached_near + STORAGE_AMOUNT` where `STORAGE_AMOUNT = 2_000_000_000_000_000_000_000_000` yoctoNEAR (2 NEAR). The user's wNEAR is transferred to the engine's implicit EVM address in the EVM execution phase. When the router's `execute` call panics, the NEAR promise fails. There is no callback or refund mechanism that returns the wNEAR to the user. The funds are permanently inaccessible to the user.

---

### Likelihood Explanation

**Medium.** The `PromiseArgs::Recursive` variant with a `SimpleNearPromise::Batch` is a documented, borsh-serializable type that any EVM user can encode and submit as calldata to the XCC precompile. No special privilege is required. The user only needs to construct the correct borsh encoding. Any user who attempts to use NEAR batch actions via XCC (a feature the type system explicitly supports) will trigger this path.

---

### Recommendation

Either:
1. Enable `all-promise-actions` in the default feature set of the xcc-router crate, or
2. Replace the `unimplemented!()` stub with a graceful error return (e.g., `env::panic_str("ERR_BATCH_NOT_SUPPORTED")`) **before** any wNEAR transfer occurs — ideally by validating the `PromiseArgs` variant in the XCC precompile before committing the wNEAR transfer, so the EVM transaction reverts cleanly and the user's funds are not moved.

---

### Proof of Concept

1. Deploy an EVM contract on Aurora that calls the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372`.
2. Craft calldata as borsh-encoded `CrossContractCallArgs::Eager(PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch({ target_account_id: "some.near", actions: [PromiseAction::CreateAccount] }))))`.
3. Ensure the calling EVM address has ≥ 2 NEAR worth of wNEAR approved to the XCC precompile.
4. Submit the EVM transaction via Aurora's `submit` endpoint.
5. Observe: the EVM transaction succeeds, 2 NEAR of wNEAR is transferred from the user's EVM address to the engine's implicit address.
6. Observe: the subsequent NEAR call to the router's `execute` method panics with `unimplemented!("NEAR batch transactions are not supported...")`.
7. The user's 2 NEAR of wNEAR is permanently inaccessible.
