### Title
Unbounded Replay of COA Ownership Proof Signatures — No Nonce, No Expiry, No Used-Proof Tracking - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies Flow account signatures over arbitrary `signedData` to prove Cadence-Owned Account (COA) ownership, but the proof contains no nonce, no expiry timestamp, and no on-chain mechanism to mark a proof as consumed. Any valid proof observed from on-chain calldata can be replayed indefinitely by an unprivileged attacker against any protocol that uses this function for one-time authorization. The code itself acknowledges a related cross-address replay gap in a comment but dismisses it as low-risk.

### Finding Description

`EVM.validateCOAOwnershipProof` is a public (`access(all)`) function in the EVM system contract. It accepts a Cadence account address, a capability path, arbitrary `signedData`, key indices, signatures, and an EVM address, then verifies that the account's keys signed `signedData` and that the account holds a COA at the given EVM address. [1](#0-0) 

The function performs three checks:
1. Signature weight threshold over `signedData` using `Crypto.KeyList.verify`.
2. COA resource exists at `path`.
3. COA's EVM address matches the supplied `evmAddress`. [2](#0-1) 

**What is missing:** There is no nonce field in `COAOwnershipProof`, no expiry timestamp, and no on-chain set of consumed proofs. Once a valid `(address, path, signedData, keyIndices, signatures, evmAddress)` tuple is produced and observed on-chain, it passes `validateCOAOwnershipProof` forever. [3](#0-2) 

The code itself acknowledges a related gap — that `signedData` does not need to encode `evmAddress`, enabling cross-COA replay for accounts that own multiple COAs: [4](#0-3) 

The EVM-side entry point is the Cadence Arch precompile `verifyCOAOwnershipProof(address,bytes32,bytes)`, which decodes the ABI-encoded proof and delegates to `coaOwnershipProofValidator`, which calls `validateCOAOwnershipProof`: [5](#0-4) 

The COA contract's ERC-1271 implementation (`isValidSignature`) calls this precompile directly, passing `address(this)` as the COA address and the caller-supplied `_hash` as `signedData`: [6](#0-5) 

**Exploit flow:**

1. Victim COA owner signs `signedData = H` (e.g., a permit hash, a DAO vote hash, a one-time withdrawal authorization) and submits a transaction that calls an EVM contract's `isValidSignature(H, encodedProof)`.
2. The `encodedProof` (containing the Flow account address, capability path, key indices, and raw signatures) is visible in EVM transaction calldata — no privileged access needed to extract it.
3. An attacker extracts the proof and replays it in any future call to `verifyCOAOwnershipProof` or `EVM.validateCOAOwnershipProof` with the same `H`. The call returns `true` indefinitely.
4. Any EVM or Cadence protocol that uses this result for one-time authorization (permit, vote, withdrawal gate) is bypassed.

The secondary cross-address replay: if the same Cadence account owns COA-A (at `/public/coaA`) and COA-B (at `/public/coaB`), a proof produced for COA-A can be replayed against COA-B by substituting the `path` and `evmAddress` arguments, because `signedData` is not required to commit to the EVM address. [7](#0-6) 

### Impact Explanation

Any EVM or Cadence smart contract that uses `validateCOAOwnershipProof` / `verifyCOAOwnershipProof` / `COA.isValidSignature` for one-time authorization is vulnerable to replay. An attacker who observes a valid proof on-chain can impersonate the COA owner for any future call using the same `signedData`. Concrete impacts include:

- **Unauthorized asset movement**: A protocol that gates a withdrawal behind a COA ownership proof can be drained by replaying a previously observed proof.
- **Privilege escalation**: A Cadence protocol that grants elevated access upon proof verification can be bypassed.
- **Cross-COA impersonation**: For accounts with multiple COAs, a proof for one COA authenticates the attacker as the owner of any other COA owned by the same account.

### Likelihood Explanation

- The proof is fully visible in EVM transaction calldata; no privileged access is required to extract it.
- `EVM.validateCOAOwnershipProof` is `access(all)` — callable from any Cadence script or transaction.
- The EVM precompile is callable from any EVM contract or EOA.
- The COA ERC-1271 `isValidSignature` is a standard interface actively used by DeFi protocols for permit-style authorization.
- The only mitigation is application-level nonce tracking, which is not enforced or documented as a requirement at the protocol level (the comment only mentions it as a suggestion for "off-chain authentication flows").

### Recommendation

1. Add a `nonce` field to `COAOwnershipProof` and maintain an on-chain `usedNonces` mapping inside `validateCOAOwnershipProof` (or require callers to supply and consume nonces).
2. Add an `expiry` (block height or timestamp) field to `COAOwnershipProof` and reject proofs past their expiry.
3. Enforce that `signedData` commits to `evmAddress` (remove the acknowledged cross-address replay gap) by hashing `evmAddress` into the signed payload inside the protocol, rather than relying on callers.
4. Document clearly that any caller of `validateCOAOwnershipProof` must implement its own nonce/expiry tracking if used for one-time authorization.

### Proof of Concept

```cadence
// Step 1: Victim submits a legitimate EVM tx calling isValidSignature(H, proof)
// The proof is visible in EVM calldata.

// Step 2: Attacker extracts proof from calldata and calls validateCOAOwnershipProof
// directly from a Cadence script (no signing required):

import EVM from <EVMContractAddress>

access(all)
fun main(
    address: Address,          // victim's Flow address (from calldata)
    path: PublicPath,          // /public/coa (from calldata)
    signedData: [UInt8],       // H (from calldata)
    keyIndices: [UInt64],      // from calldata
    signatures: [[UInt8]],     // from calldata — the victim's actual signatures
    evmAddress: [UInt8; 20]    // victim's COA address (from calldata)
): EVM.ValidationResult {
    // Returns isValid: true — indefinitely, with no victim consent
    return EVM.validateCOAOwnershipProof(
        address: address,
        path: path,
        signedData: signedData,
        keyIndices: keyIndices,
        signatures: signatures,
        evmAddress: evmAddress
    )
}
```

The result is `isValid: true` on every replay. Any EVM contract calling `COA.isValidSignature(H, proof)` with the replayed proof also returns `ValidERC1271Signature (0x1626ba7e)`. [8](#0-7) [9](#0-8)

		return nil, err

		hash,
