### Title
Gateway Skips `__validate__` for Invoke Transactions with Nonce=1 Based on Insufficient Mempool Presence Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the blockifier `__validate__` entry-point call for an Invoke transaction whose nonce is exactly `1` and whose sender's on-chain nonce is `0`, whenever `account_tx_in_pool_or_recent_block` returns `true`. The check is intended to confirm that a `deploy_account` transaction is pending, but the underlying mempool query returns `true` for **any** transaction type in the pool for that address. An attacker who first submits a valid `deploy_account` (which must pass full validation) can immediately follow with an Invoke carrying an **invalid signature** at nonce=1; the gateway accepts it without ever calling `__validate__`, and the transaction enters the mempool.

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when:
- the transaction is an Invoke,
- `tx.nonce() == Nonce(Felt::ONE)`,
- `account_nonce == Nonce(Felt::ZERO)`, and
- `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

**What `account_tx_in_pool_or_recent_block` actually checks:** [2](#0-1) 

It returns `true` if the address has **any** transaction in the pool or appears in the mempool's staged/committed state maps — it does **not** verify that the pending transaction is a `deploy_account`. [3](#0-2) 

**What is skipped when `skip_validate = true`:** [4](#0-3) 

`execution_flags.validate` is set to `false`, so the blockifier's `StatefulValidator::perform_validations` returns immediately without calling `__validate__`: [5](#0-4) 

**The full gateway admission path:** [6](#0-5) 

`validate_state_preconditions` and `validate_by_mempool` only check nonce ordering and resource bounds — neither verifies the transaction signature. The only signature check is the `__validate__` entry-point call, which is the step being skipped.

**The comment's flawed assumption:** [7](#0-6) 

The comment claims it is "sufficient to check if the account exists in the mempool." In practice, for a non-deployed account (on-chain nonce = 0), the only transaction that can legitimately reach the mempool is a valid `deploy_account`. However, the code does not enforce this — it accepts the presence of **any** transaction type as proof that a `deploy_account` is pending.

### Impact Explanation

An attacker can:

1. Submit a valid `deploy_account` for a fresh address `A` (must pass full gateway validation including `__validate_deploy__`). This places `A` in the mempool's `tx_pool`.
2. Immediately submit an Invoke with nonce=1 for address `A` carrying an **invalid or forged signature**.
3. `skip_stateful_validations` returns `true` (nonce=1, account_nonce=0, `A` is in pool).
4. `run_validate_entry_point` is called with `execution_flags.validate = false`.
5. The gateway accepts the invalid Invoke and forwards it to the mempool.

The invalid Invoke is admitted to the mempool without any signature verification at the gateway layer. This violates the gateway admission invariant: **every accepted transaction must have a valid `__validate__` result**.

At block execution time the blockifier will run `__validate__` and reject the transaction, so no state corruption occurs. The concrete impact is:

- The gateway accepts an invalid (unsigned/wrongly-signed) Starknet transaction — matching the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions before sequencing."*
- Mempool resources are consumed by transactions that will always be rejected at execution.
- The attacker can repeat this for every `deploy_account` they submit, scaling the attack proportionally.

### Likelihood Explanation

The trigger is fully unprivileged and requires only two sequential RPC calls. The first (`deploy_account`) must be economically valid (requires funds), but the second (invalid Invoke) is free to forge. The condition `nonce == 1 && account_nonce == 0` is the normal state for any new account during the deploy+invoke UX flow, making this a commonly reachable code path.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a query that specifically confirms a **`deploy_account` transaction** is pending for the sender address. Alternatively, add a dedicated mempool API `deploy_account_in_pool(address) -> bool` that inspects transaction types, and use that in `skip_stateful_validations`. The current check conflates "account has any pending transaction" with "account has a pending deploy_account," which is the broken invariant.

### Proof of Concept

```
1. Generate a fresh keypair → address A (not yet deployed on-chain).

2. POST /add_transaction  →  deploy_account for A
   - Valid class_hash, valid __validate_deploy__ signature, sufficient balance.
   - Gateway runs full blockifier validation; deploy_account enters mempool.
   - account_tx_in_pool_or_recent_block(A) now returns true.

3. POST /add_transaction  →  invoke for A, nonce=1, INVALID signature (e.g., all-zero sig)
   - Stateless validation passes (no signature check there).
   - validate_state_preconditions: nonce 1 >= account_nonce 0, within gap → passes.
   - validate_by_mempool: nonce 1 >= 0 → passes.
   - skip_stateful_validations: nonce==1, account_nonce==0,
       account_tx_in_pool_or_recent_block(A)==true → returns true.
   - run_validate_entry_point called with validate=false → __validate__ NOT called.
   - Gateway returns tx_hash; invalid invoke is now in the mempool.

4. Observe: the mempool contains an invoke with an invalid signature for address A.
   At execution time the blockifier rejects it, but the gateway admission invariant
   was violated — an invalid transaction was accepted.
```

The exact corrupted value is `execution_flags.validate = false` set at [8](#0-7) 

for a transaction whose `__validate__` entry point would return an error if actually called.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
