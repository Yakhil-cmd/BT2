### Title
Anyone Can Force `exit_delegation_pool_action` on Behalf of Any Pool Member - (File: src/pool/pool.cairo)

### Summary
`exit_delegation_pool_action(pool_member)` in `pool.cairo` accepts an arbitrary `pool_member` address with no caller validation. Any unprivileged caller can complete another user's pending exit, preventing the pool member from canceling their intent or switching to a different delegation pool.

### Finding Description
`exit_delegation_pool_action` accepts a `pool_member` address parameter and performs the full exit sequence — calling `remove_from_delegation_pool_action` on the staking contract and transferring `unpool_amount` tokens to `pool_member` — without ever checking that `get_caller_address() == pool_member`. [1](#0-0) 

The function's only guards are that the pool member has a pending intent and that the exit window has elapsed: [2](#0-1) 

There is no `get_caller_address()` call anywhere in the function body. [3](#0-2) 

By contrast, every other sensitive pool function that accepts an address parameter does validate the caller. For example, `add_to_delegation_pool` asserts `caller_address == pool_member || caller_address == pool_member_info.reward_address`: [4](#0-3) 

And `claim_rewards` performs the same check: [5](#0-4) 

The same missing-caller-check pattern exists in `unstake_action` in `staking.cairo`, which also accepts `staker_address` without validating the caller: [6](#0-5) 

### Impact Explanation
A pool member who has set an exit intent retains two legitimate options after the exit window elapses:

1. **Cancel the intent** by calling `exit_delegation_pool_intent(amount: 0)`, which sets `unpool_time = None` and keeps the member in the pool.
2. **Switch pools** by calling `switch_delegation_pool`, which requires `unpool_time.is_some()` and `unpool_amount > 0`.

An attacker who calls `exit_delegation_pool_action(victim)` first zeroes out `unpool_amount` and clears `unpool_time`, making both options impossible. The victim's tokens are returned to their address (no direct theft), but they are forcibly ejected from the pool and must re-enter and wait a full exit window again before they can switch pools. This constitutes **griefing with damage to the user** — the victim loses accrued delegation rewards for the forced-out period and is denied the ability to execute a pool switch they had prepared.

This maps to the **Medium** allowed impact: *Griefing with no profit motive but damage to users or protocol*.

### Likelihood Explanation
The attack requires only that:
- A pool member has a pending exit intent with a non-zero `unpool_amount` and the exit window has elapsed.
- The attacker observes the mempool for a `switch_delegation_pool` or `exit_delegation_pool_intent(0)` transaction and frontruns it.

Both conditions are easily observable on-chain. No privileged access, leaked key, or external dependency is required. Any unprivileged address can execute the attack.

### Recommendation
Add a caller check at the top of `exit_delegation_pool_action`, mirroring the pattern used in `add_to_delegation_pool` and `claim_rewards`:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let caller = get_caller_address();
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    assert!(
        caller == pool_member || caller == pool_member_info.reward_address,
        "{}",
        Error::CALLER_CANNOT_EXIT_POOL,
    );
    // ... rest of function
```

Apply the same fix to `unstake_action` in `staking.cairo`:

```cairo
fn unstake_action(ref self: ContractState, staker_address: ContractAddress) -> Amount {
    self.general_prerequisites();
    let caller = get_caller_address();
    let staker_info = self.internal_staker_info(:staker_address);
    assert!(
        caller == staker_address || caller == staker_info.reward_address,
        "{}",
        Error::CALLER_CANNOT_UNSTAKE,
    );
    // ... rest of function
```

### Proof of Concept

1. Alice (pool member) calls `exit_delegation_pool_intent(amount: 100)`. Exit window starts.
2. Exit window elapses. Alice decides she wants to switch to a better-performing pool and submits `switch_delegation_pool(to_staker, to_pool, amount: 100)`.
3. Attacker observes Alice's transaction in the mempool and submits `exit_delegation_pool_action(alice_address)` with higher gas.
4. Attacker's transaction executes first: `unpool_amount` is zeroed, `unpool_time` is cleared, and 100 tokens are transferred to Alice's address.
5. Alice's `switch_delegation_pool` transaction reverts because `pool_member_info.unpool_amount` is now 0 (`AMOUNT_TOO_HIGH`).
6. Alice must re-enter the pool, wait a full exit window, and retry — losing delegation rewards for the entire forced-out period. [7](#0-6)
