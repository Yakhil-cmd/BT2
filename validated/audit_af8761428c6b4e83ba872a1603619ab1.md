### Title
ERC-20 Mirror Deployed with Zero Decimals via Unrestricted Legacy Path — (File: `engine/src/engine.rs`, `engine-types/src/parameters/connector.rs`)

### Summary

The `Erc20Metadata::default()` struct hard-codes `decimals: 0`. The `deploy_erc20_token` contract method has no access-control guard and accepts a raw Borsh-encoded `AccountId` (the "Legacy" format), which causes the ERC-20 mirror to be deployed with `decimals = 0` instead of the correct NEP-141 value. Any unprivileged NEAR account can trigger this path to front-run a legitimate token deployment and permanently register the mirror with zero decimals.

### Finding Description

**Root cause 1 — Wrong default decimal value.**

`Erc20Metadata::default()` sets `decimals` to `0`: [1](#0-0) 

**Root cause 2 — Legacy path passes `None` metadata, falling back to the zero-decimal default.**

In `deploy_erc20_token` (connector method), the `Legacy` branch calls `engine::deploy_erc20_token` with `metadata = None`: [2](#0-1) 

`setup_deploy_erc20_input` then calls `unwrap_or_default()`, materialising `decimals = 0` into the EVM constructor calldata: [3](#0-2) 

**Root cause 3 — No access control on `deploy_erc20_token`.**

The public entry point only checks `require_running`; there is no `require_owner_only` guard: [4](#0-3) 

**Root cause 4 — The Legacy deserialization path is always reachable.**

`DeployErc20TokenArgs::deserialize` falls back to `Legacy` whenever the input is a plain Borsh-encoded `AccountId`: [5](#0-4) 

**Consequence in the EVM contract.**

`EvmErc20` / `EvmErc20V2` store `_decimals` exactly as supplied in the constructor and return it from `decimals()`: [6](#0-5) 

Once the token is registered via `register_token`, a second deployment for the same NEP-141 is rejected, so the zero-decimal value is permanent unless the admin manually calls `set_erc20_metadata`.

### Impact Explanation

Any DeFi protocol on Aurora (AMM, lending market, aggregator) that reads `decimals()` to scale amounts will treat every raw token unit as one whole token. For a token with true decimals = 6 (e.g., USDC), a deposit of 1 000 000 units (= 1 USDC) appears to the protocol as 1 000 000 whole USDC. An attacker can exploit this price discrepancy to drain liquidity pools or borrow against inflated collateral — constituting direct theft of user funds held in those protocols on Aurora.

**Impact: Critical — direct theft of user funds.**

### Likelihood Explanation

- `deploy_erc20_token` is callable by any NEAR account with no privilege requirement.
- The Legacy input format (a plain Borsh `AccountId`) is trivially constructable.
- An attacker only needs to submit the call before the legitimate operator does for a not-yet-bridged NEP-141 token.
- NEAR transaction ordering is deterministic and observable in the mempool, making front-running straightforward.

**Likelihood: Medium** (requires timing a front-run before the first legitimate deployment of a target token).

### Recommendation

1. **Add access control** to `deploy_erc20_token` (e.g., `require_owner_only`) so only the contract owner can register new ERC-20 mirrors.
2. **Remove or harden the Legacy path**: either reject plain `AccountId` inputs entirely and require the `WithMetadata` variant, or fetch `ft_metadata` even in the legacy branch before deploying.
3. **Change `Erc20Metadata::default()`** to use a sentinel that causes an explicit error rather than silently deploying with `decimals = 0`.

### Proof of Concept

1. Identify a NEP-141 token `target.near` not yet bridged to Aurora.
2. Construct `borsh::to_vec(&AccountId::from("target.near"))` — a valid Legacy input.
3. Call `deploy_erc20_token` on the Aurora Engine contract from any NEAR account with this payload.
4. The engine executes `engine::deploy_erc20_token(nep141, None, ...)`, which calls `setup_deploy_erc20_input` with `erc20_metadata.unwrap_or_default()` → `decimals = 0`.
5. The ERC-20 mirror is deployed and registered. All subsequent calls to `deploy_erc20_token` for `target.near` fail with "already registered."
6. Any AMM on Aurora that prices `target.near`'s ERC-20 using `decimals()` now reads `0`, inflating apparent token value by `10^correct_decimals` and enabling the attacker to extract funds from the pool.
