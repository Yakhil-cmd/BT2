### Title
Off-by-One in DKG Threshold Upper-Bound Check Allows `threshold == N`, Enabling Permanent Denial of Signing - (File: src/dkg.rs)

### Summary
`assert_key_invariants` uses a strict `>` comparison to enforce the upper bound on the DKG threshold, but the protocol specification requires a strict `<` relationship (`threshold < N`). The boundary case `threshold == N` is silently accepted, allowing a DKG to complete with a configuration that requires all N participants for every future signing operation.

### Finding Description
The PedPop+ protocol specification in `docs/dkg.md` line 50 states:

> Each P_i asserts that `1 < threshold < N`.

Both inequalities are strict. The lower bound is correctly enforced: `threshold < 2` is rejected, enforcing `threshold > 1`. However, the upper bound check in `assert_key_invariants` (`src/dkg.rs` line 573) uses `>` instead of `>=`:

```rust
// Step 1.1
// validate threshold
if threshold > participants.len() {          // ← should be >=
    return Err(InitializationError::ThresholdTooLarge {
        threshold,
        max: participants.len(),
    });
}
```

When `threshold == participants.len()` (i.e., `threshold == N`), the condition `threshold > N` is `false`, so no error is returned. The DKG proceeds and produces valid-looking key shares backed by a degree-`N-1` polynomial. Reconstruction requires all N shares. Every subsequent call to `sign`, `presign`, or `reshare` will silently require all N participants to be simultaneously available.

`assert_key_invariants` is the single validation gate called by both `keygen` (via `src/lib.rs` line 99) and `reshare`/`refresh` (via `assert_reshare_keys_invariants`, `src/dkg.rs` line 649), so the off-by-one propagates to all three DKG entry points.

### Impact Explanation
With `threshold == N` accepted, the resulting key material has zero fault tolerance: the absence of any single participant permanently prevents signing. A malicious participant who participates in DKG and then goes permanently offline causes irreversible loss of signing capability for all remaining honest parties. There is no recovery path short of a full reshare — which itself requires all N participants (including the offline one) to be present, since the old threshold is N.

This maps to: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation
The library's API accepts `threshold == N` without error, making it indistinguishable from a valid configuration to callers. A malicious coordinator or participant who controls the threshold parameter (e.g., in a deployment where threshold is negotiated or suggested by one party) can propose `threshold == N`. Because the library does not reject it, honest participants proceed through DKG. Once the malicious party goes offline after DKG completes, signing is permanently denied. The attack requires no cryptographic capability — only the ability to influence the threshold value passed to `keygen`.

### Recommendation
Change the comparison from `>` to `>=` in `assert_key_invariants` to enforce the strict upper bound `threshold < N` as specified:

```rust
// Step 1.1
// validate threshold: spec requires 1 < threshold < N (strict on both sides)
if threshold >= participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold,
        max: participants.len(),
    });
}
```

**Spec reference** — `docs/dkg.md` line 50 mandates `threshold < N` (strict): [1](#0-0) 

**Vulnerable check** — `assert_key_invariants` uses `>` instead of `>=`, silently accepting `threshold == N`: [2](#0-1) 

**Propagation to `keygen`** — `assert_key_invariants` is the sole validation gate for `keygen`: [3](#0-2) 

**Propagation to `reshare`/`refresh`** — `assert_reshare_keys_invariants` delegates to `assert_key_invariants` for the new threshold, so both reshare and refresh inherit the same off-by-one: [4](#0-3) 

**Concrete scenario:**
1. Coordinator proposes `threshold = N` (e.g., `threshold = 3`, `participants = [P1, P2, P3]`).
2. Each participant calls `keygen(..., threshold=3)`. The check `3 > 3` is `false`; no error is raised.
3. DKG completes; each party holds a share of a degree-2 polynomial. All 3 shares are required to reconstruct.
4. Malicious P3 goes permanently offline.
5. P1 and P2 attempt to sign. Signing requires 3 shares; only 2 are available. Signing is permanently denied.
6. Reshare to recover also requires the old threshold (3) participants to be present — P3 is gone, so reshare is also permanently blocked.
