Now I have enough context to write the finding. The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` uses a split-amount model with two layers of integer truncation to compute the "Total Amount with Royalties" shown in the WalletConnect confirmation dialog. This is structurally analogous to the external bug: a rounding/truncation error in one calculation produces a displayed total that is lower than the actual amount that will be deducted, causing a user to approve a transaction spending more than shown.

### Title
WalletConnect Confirmation Dialog Displays Understated "Total Amount with Royalties" Due to Integer Truncation in Multi-NFT Offers - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` uses two layers of `BigInt` integer division to estimate royalties for multi-NFT WalletConnect offers. Both divisions truncate, producing a "Total Amount with Royalties" that is systematically lower than the actual amount the wallet will deduct. A user viewing the WalletConnect confirmation dialog sees a lower spend figure and approves; the on-chain transaction deducts more than displayed.

### Finding Description
When a WalletConnect dApp sends `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` for an offer that includes multiple NFTs with royalties, `parseCommandDisplay` calls `formatAmountWithRoyalties` to compute the `amountWithRoyalties` string shown in the "You Spend" section of the confirmation dialog. [1](#0-0) 

The function contains two truncating integer divisions:

```typescript
// Line 363: splits the total amount equally among NFTs — truncates remainder
const splitAmount = amount / BigInt(royaltyPercentages.length);

// Line 365: computes royalty per NFT from the already-truncated splitAmount — truncates again
(total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
```

The resulting `totalAmount = amount + royaltyAmount` is then rendered in `Confirm.tsx` as the authoritative "Total Amount with Royalties" label the user reads before clicking Confirm. [2](#0-1) 

The regular offer builder (`OfferBuilderXCHSection`, `OfferBuilderTokensSection`) avoids this by calling the `calculateRoyaltiesForNFTs` RPC and summing the exact per-NFT royalty amounts returned by the daemon. `parseCommandDisplay` does not call that RPC; it uses the split-amount approximation exclusively. [3](#0-2) 

### Impact Explanation
The user's only signal of total cost in the WalletConnect approval flow is the `amountWithRoyalties` field. When it is understated, the user approves a transaction believing they are spending less XCH or CAT than the blockchain will actually deduct. This constitutes displaying the wrong amount in a signing-approval context, matching the **High** impact criterion: *"WalletConnect state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation
Any WalletConnect-connected dApp can trigger this by submitting a `take_offer` or `create_offer_for_ids` command for an offer that bundles two or more NFTs with non-zero royalties. No privileged access, leaked keys, or social engineering is required. The discrepancy grows with the number of NFTs and the magnitude of royalty percentages, and is present in every such offer regardless of dApp intent.

### Recommendation
Replace the split-amount approximation in `formatAmountWithRoyalties` with the same RPC-backed approach used by the offer builder: call `calculateRoyaltiesForNFTs` (or read the royalty amounts already resolved from the offer summary) and sum the exact per-NFT royalty amounts. If an async call is not feasible at that point in the pipeline, sum `(amount * BigInt(royaltyPercentage)) / 10_000n` per NFT without splitting — this at least removes the split-induced truncation and matches the on-chain calculation model.

### Proof of Concept
**Setup:** WalletConnect dApp sends `chia_wallet.take_offer` for an offer where the user spends **1 XCH** (1,000,000,000,000 mojos) to receive **2 NFTs**, each with `royalty_percentage = 500` (5%).

**`formatAmountWithRoyalties` calculation (displayed):**
```
splitAmount = 1_000_000_000_000n / 2n = 500_000_000_000n
royaltyAmount = (500_000_000_000n * 500n) / 10_000n   // NFT1 = 25_000_000_000n
              + (500_000_000_000n * 500n) / 10_000n   // NFT2 = 25_000_000_000n
              = 50_000_000_000n
totalAmount   = 1_050_000_000_000n  →  displayed: "1.05 XCH"
```

**Actual on-chain royalty (per NFT on full amount):**
```
NFT1 royalty = 1_000_000_000_000n * 500n / 10_000n = 50_000_000_000n
NFT2 royalty = 1_000_000_000_000n * 500n / 10_000n = 50_000_000_000n
actual total = 1_100_000_000_000n  →  actual spend: "1.1 XCH"
```

The confirmation dialog shows **1.05 XCH**; the wallet deducts **1.1 XCH** — a 50,000,000,000 mojo (0.05 XCH) understatement. The user approves based on the lower figure. [4](#0-3) [5](#0-4)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
