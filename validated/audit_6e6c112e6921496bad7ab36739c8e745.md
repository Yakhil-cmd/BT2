### Title
Unbounded `stakers` Vec Growth Causes `get_stakers` Unbounded Gas Consumption - (File: `src/staking/staking.cairo`)

---

### Summary

The `stakers` storage `Vec<ContractAddress>` in the Staking contract grows monotonically with every new staker and is **never pruned** — even after a staker fully unstakes. The `get_stakers` consensus function iterates the entire Vec on every call. Because any unprivileged address can stake (paying only the minimum stake), an attacker can bloat this Vec without bound, eventually making `get_stakers` uncallable due to gas exhaustion.

---

### Finding Description

**Root cause — unbounded Vec, never pruned:**

The storage declaration explicitly documents the design flaw:

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

Every successful call to `stake()` unconditionally appends the caller's address:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
``` [2](#0-1) 

Neither `unstake_intent` nor `unstake_action` removes the address from the Vec. `unstake_action` only clears the staker's pool list and deletes their `staker_info` record — the Vec entry remains forever. [3](#0-2) 

**Affected consumer — full-range iteration in `get_stakers`:**

`get_stakers`, the consensus-layer function that returns the active staker set for a given epoch, iterates the **entire** Vec on every call:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
``` [4](#0-3) 

Inactive/exited stakers are skipped via `is_staker_active`, but the loop still reads every entry from storage. Each storage read costs gas, so a Vec of N entries costs O(N) gas regardless of how many are active.

**No cap exists:** There is no maximum staker count enforced anywhere in `stake()`. [5](#0-4) 

---

### Impact Explanation

`get_stakers` is the function the consensus layer calls to determine which validators are active for a given epoch. Once the `stakers` Vec grows large enough to exceed Starknet's per-transaction resource limits, every call to `get_stakers` will revert with an out-of-gas / resource-exhaustion error. The consensus layer loses the ability to read the active staker set, permanently breaking the consensus reward and validator-selection mechanism.

This matches the allowed impact: **Medium — Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption.**

---

### Likelihood Explanation

- Any address can call `stake()` by providing the minimum stake amount and the required token approval.
- After unstaking, the address is blocked from re-staking (`assert_staker_address_not_reused`), but a fresh address can always be used.
- An attacker willing to cycle through many funded addresses (each staking the minimum, then unstaking to recover funds after the exit window) can grow the Vec at the cost of transaction fees and temporary capital lock-up.
- The attack is permissionless and requires no privileged access.

---

### Recommendation

1. **Remove stakers from the Vec on `unstake_action`**: Replace the Vec with a data structure that supports O(1) deletion (e.g., swap-and-pop pattern), or maintain a separate active-staker count and skip-list.
2. **Alternatively, enforce a maximum staker count** at `stake()` time to bound the Vec size.
3. **Paginate `get_stakers`**: Accept an offset/limit parameter so callers can retrieve the staker set in chunks, preventing a single call from iterating the entire Vec.

---

### Proof of Concept

1. Deploy the Staking contract with `min_stake = M`.
2. Create N unique addresses, each funded with `M` STRK tokens.
3. For each address, call `stake(...)` — each call appends one entry to `stakers`.
4. Optionally call `unstake_intent()` + `unstake_action()` on each to recover funds; the Vec entries remain.
5. Call `get_stakers(epoch_id)`. Observe that the function must iterate all N entries. As N grows toward Starknet's per-call step limit, the call reverts with resource exhaustion.
6. The consensus layer is now unable to retrieve the active staker set. [6](#0-5)

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
