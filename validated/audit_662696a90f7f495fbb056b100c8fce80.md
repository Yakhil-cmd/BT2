### Title
Unprivileged Caller Can Permanently Suppress Per-Block Reward Distribution via `disable_rewards` Toggle - (File: `src/staking/staking.cairo`)

### Summary
`update_rewards` in the `Staking` contract accepts a caller-controlled `disable_rewards: bool` parameter. Because there is no access control on this function, any unprivileged address can call it with `disable_rewards = true` once per block. Doing so advances the global `last_reward_block` sentinel **before** the early return, permanently consuming the reward slot for that block without distributing any rewards to stakers. This is a direct analog to the `displayMessage` toggle: an untrusted caller controls a boolean that silently bypasses the security-critical operation (reward distribution instead of message display).

### Finding Description
`IStakingRewardsManager::update_rewards` is a public, permissionless function:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);   // ← sentinel advanced HERE

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // ← rewards silently skipped
    }
    // actual reward calculation and distribution follows
    ...
    self._update_rewards(...);
}
``` [1](#0-0) 

The critical ordering is:
1. `last_reward_block` is written to `current_block_number` unconditionally (line 1485).
2. Only **after** that write does the `disable_rewards` branch return early (line 1487–1489).

Because `last_reward_block` is a single global value shared across all stakers, any subsequent legitimate call to `update_rewards` in the same block will hit the `REWARDS_ALREADY_UPDATED` assertion and revert. The block's reward slot is permanently consumed with zero rewards distributed. [2](#0-1) 

There is no role check — `general_prerequisites()` only enforces the pause flag: [3](#0-2) 

### Impact Explanation
**High — Theft of unclaimed yield.**

Block rewards that should have been credited to stakers via `_update_rewards` are never calculated or distributed. They are not deferred or frozen; they are simply never created. Every block for which an attacker fires this call results in a permanent, irrecoverable loss of yield for all active stakers and their delegators. Sustained over many blocks, this drains the entire consensus-reward stream.

### Likelihood Explanation
**High.** The function is public and permissionless. The only cost to the attacker is gas per block. The attack can be fully automated (e.g., a bot that front-runs the legitimate consensus call every block). No privileged key, bridge compromise, or external dependency is required — only the ability to submit a transaction.

### Recommendation
Remove the `disable_rewards` parameter from the public interface entirely, mirroring the fix applied in the referenced report (removing `displayMessage`). If a "no-op" path is genuinely needed for the transition period, gate it behind a privileged role (e.g., `only_app_governor`) so that an unprivileged caller cannot suppress reward distribution. Additionally, move the `last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard so that a suppressed call does not consume the block's reward slot.

### Proof of Concept
1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. A valid staker `S` exists with non-zero balance.
3. Attacker `A` (any EOA) calls `update_rewards(S, disable_rewards: true)` at block `N`.
4. `last_reward_block` is set to `N`; the function returns before `_update_rewards`.
5. The legitimate consensus caller attempts `update_rewards(S, disable_rewards: false)` at block `N` — it reverts with `REWARDS_ALREADY_UPDATED`.
6. No rewards are distributed for block `N`. Repeating steps 3–5 every block permanently zeroes all staker yield. [4](#0-3)

                    btc_total_stake: staker_total_btc_balance,
