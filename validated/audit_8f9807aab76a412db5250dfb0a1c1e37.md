### Title
Unbounded Loop Over Pool Member Balance Trace Enables Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire `pool_member_epoch_balance` trace. Because the trace grows by one checkpoint per epoch whenever a pool member changes their balance, and because there is no mechanism to claim rewards in partial batches or to prune the trace, a pool member who accumulates enough checkpoints without claiming will eventually find their `claim_rewards` transaction exceeding the Starknet transaction gas/step limit, permanently freezing their unclaimed yield with no recovery path.

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:**

In `src/pool/pool.cairo` at line 857–877, the `calculate_rewards` internal function iterates from `entry_to_claim_from` to `pool_member_trace_length`, reading one storage slot per iteration and calling `find_sigma` (which itself performs a binary search over `cumulative_rewards_trace`) on every step:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

The developers themselves flag this as unbounded.

**How the trace grows:**

`set_member_balance` (called by `add_to_delegation_pool`, `exit_delegation_pool_intent`, and `enter_delegation_pool`) inserts a new checkpoint into `pool_member_epoch_balance` whenever the target epoch (`current_epoch + K`) differs from the last stored key. Because `K = 2` is a constant, each balance change in a new epoch appends one new checkpoint. The `insert` logic in `src/pool/pool_member_balance_trace/trace.cairo` lines 166–172 only deduplicates within the same epoch; changes across different epochs always append.

**No partial-claim mechanism:**

`claim_rewards` in `src/pool/pool.cairo` lines 335–377 always processes from the stored `entry_to_claim_from` to the current epoch in a single transaction. There is no parameter to limit the range, and `entry_to_claim_from` is only advanced after a successful full execution. If the transaction runs out of gas mid-loop, the state is not updated and the next attempt starts from the same position — making the freeze permanent.

**Attacker-controlled entry path:**

1. A pool member (or their reward address, which can call `add_to_delegation_pool` spending its own tokens) makes one balance change per epoch — either via `add_to_delegation_pool` (minimum 1 wei) or `exit_delegation_pool_intent` — without ever calling `claim_rewards`.
2. After N epochs of balance changes, the trace has N checkpoints.
3. When `claim_rewards` is eventually called, the loop must iterate all N checkpoints, each requiring multiple storage reads plus a binary search over `cumulative_rewards_trace`.
4. Once N is large enough to exceed the Starknet step limit, the transaction always reverts and the yield is permanently frozen.

A malicious reward address (set by the pool member, or a contract the pool member interacts with) can accelerate this by calling `add_to_delegation_pool` with 1 wei each epoch, spending negligible tokens to inflate the victim pool member's trace.

### Impact Explanation

Once the trace is large enough that `calculate_rewards` exceeds the transaction step limit, the pool member's accumulated yield becomes permanently unclaimable. There is no admin function, no pruning mechanism, and no partial-claim path to recover from this state. This matches the **High** impact category: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

The attack requires accumulating one checkpoint per epoch without claiming. With `K = 2` and realistic epoch durations, a pool member who makes frequent balance changes and defers claiming for an extended period (or a malicious reward address spending ~1 wei/epoch) can reach the threshold. The code comment at line 858 explicitly acknowledges the loop is unbounded, confirming the developers are aware of the risk but have not mitigated it. Likelihood is **Medium**: it requires sustained activity over many epochs, but there is no on-chain enforcement preventing it.

### Recommendation

1. **Add a `max_iterations` parameter to `claim_rewards`** so callers can process rewards in bounded batches, advancing `entry_to_claim_from` incrementally across multiple transactions.
2. **Cap the number of balance-change checkpoints** per pool member (e.g., enforce a maximum trace length and require claiming before further balance changes are allowed).
3. **Alternatively**, restructure reward accounting to avoid per-epoch iteration entirely (e.g., store a running cumulative index at each balance change so rewards can be computed in O(1) per checkpoint pair).

### Proof of Concept

```
Epoch 0:  pool_member calls enter_delegation_pool(amount=MIN_STAKE)
           → trace length = 1

Epoch 1:  pool_member calls add_to_delegation_pool(amount=1)
           → trace length = 2

Epoch 2:  pool_member calls add_to_delegation_pool(amount=1)
           → trace length = 3

... (repeat for N epochs without ever calling claim_rewards) ...

Epoch N:  pool_member calls claim_rewards
           → calculate_rewards loops N times
           → each iteration: storage read (pool_member_trace.at) + find_sigma binary search
           → if N is large enough, transaction exceeds Starknet step limit → REVERT
           → entry_to_claim_from unchanged → next call also reverts → yield permanently frozen
```

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
