### Title
Unvalidated Participant Shares in CKD Coordinator Allow Any Malicious Participant to Corrupt the Derived Confidential Key Output — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates `(norm_big_y, norm_big_c)` group elements received from each participant with no proof of correctness. A single malicious participant can send arbitrary group elements, causing the final `CKDOutput` to be cryptographically invalid and the derived confidential key to be wrong. This is a direct analog of the spot-price manipulation in the external report: just as an attacker injects a manipulated price into an aggregation function that trusts it blindly, a malicious CKD participant injects manipulated group elements into an aggregation function that trusts them blindly.

---

### Finding Description

`do_ckd_coordinator` in `src/confidential_key_derivation/protocol.rs` receives one `CKDOutput` per non-coordinator participant and unconditionally adds each element into the running totals:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

No proof of knowledge, commitment, or consistency check is performed on the received `(norm_big_y, norm_big_c)` pair before it is added. Compare this with the honest computation in `compute_signature_share`, which derives these values from the participant's private share `x_i` and a fresh random scalar `y`:

```rust
let big_s = hash_point * private_share.to_scalar();   // x_i · H(pk ‖ app_id)
let big_c = big_s + app_pk * y.0;                     // x_i·H + y·A
let norm_big_y = big_y * lambda_i;                    // λ_i · y·G
let norm_big_c = big_c * lambda_i;                    // λ_i · (x_i·H + y·A)
``` [2](#0-1) 

Because `do_ckd_participant` sends these values with no accompanying NIZK proof, the coordinator has no way to distinguish a correctly-formed share from an arbitrary group element:

```rust
fn do_ckd_participant(...) -> Result<CKDOutputOption, ProtocolError> {
    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
    Ok(None)
}
``` [3](#0-2) 

Contrast this with the OT-based ECDSA presign, which validates received scalar shares against public commitments before using them (`big_e == e * G`, `alpha * G == K + A`, `beta * G == X + B`): [4](#0-3) 

And with DKG, which validates every received secret share against the sender's public polynomial commitment via `validate_received_share`: [5](#0-4) 

The CKD protocol has no equivalent guard.

---

### Impact Explanation

The final `CKDOutput` is `(Y, C)` where the confidential key is recovered as `C − sk_app · Y = msk · H(pk ‖ app_id)`. If any participant substitutes arbitrary group elements for their share, the sum `(Y, C)` no longer encodes the correct master secret key contribution. The coordinator and any downstream consumer will accept this corrupted output as legitimate (there is no post-aggregation verification), and the derived confidential key will be wrong — rendering it unusable for any TEE application that depends on it.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

- Only one malicious participant is required.
- The attack requires no special capability: the participant simply sends two arbitrary `ElementG1` values instead of the correctly computed ones.
- There is no throttle, rate limit, or post-hoc audit that would detect or prevent this.
- The coordinator has no way to distinguish a valid share from a forged one.

---

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` with a NIZK proof of correct formation. Concretely, a DLOGEQ proof can demonstrate that `norm_big_y = λ_i · y · G` and `norm_big_c = λ_i · (x_i · H + y · A)` share the same `y`, and a separate proof (or use of the public verification key share) can bind `x_i · H` to the participant's known public key share. The coordinator must verify all proofs before adding any share to the running total. The existing `dlog.rs` and `dlogeq.rs` primitives in `src/crypto/proofs/` provide the building blocks for this.


---

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` and malicious participant `P_m` all call `ckd(...)`.
2. `P_m` intercepts the `do_ckd_participant` path and, instead of calling `compute_signature_share`, sends `(ElementG1::identity(), ElementG1::identity())` to the coordinator.
3. The coordinator executes:
   ```
   norm_big_y += identity  →  no change to Y
   norm_big_c += identity  →  no change to C
   ```
   This silently removes `P_m`'s legitimate contribution `λ_m · (x_m · H + y_m · A)` from `C`, so the final `C` encodes only a partial sum of the master secret key. The coordinator returns this as a valid `CKDOutput`.
4. Any consumer that calls `ckd_output.unmask(app_sk)` obtains `C − sk_app · Y ≠ msk · H(pk ‖ app_id)`, i.e., the wrong confidential key, with no error or indication of failure.

Alternatively, `P_m` can send `(G, G)` (the generator) to shift the output by a known offset, potentially allowing `P_m` to predict or control the derived key if `app_sk` is known to them.

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}
