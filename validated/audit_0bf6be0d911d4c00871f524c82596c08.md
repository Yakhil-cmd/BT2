### Title
`PresignOutput` Derives `Clone`, Enabling One-Time-Use Nonce/Presignature Reuse and Private Key Extraction — (File: `src/frost/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

All three `PresignOutput` types in the library — for FROST EdDSA, OT-based ECDSA, and Robust ECDSA — derive `Clone`. These types contain one-time-use cryptographic nonce material (FROST `SigningNonces`, ECDSA nonce shares `k`/`sigma`/`c`/`e`/`alpha`/`beta`). Because `Clone` is derived, any caller can trivially duplicate a presign output and pass it to the signing function multiple times. The library's own documentation explicitly forbids presignature reuse, but the type system provides no enforcement. This is the direct analog to the SoftClayContract bug: a one-time-use resource is not invalidated after use, allowing it to be consumed repeatedly.

---

### Finding Description

**FROST (`src/frost/mod.rs`, line 36):**

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
```

`SigningNonces<C>` are FROST round-1 nonces. The FROST specification mandates they are used exactly once. Deriving `Clone` allows a caller to clone the `PresignOutput` and call `sign_v2` multiple times with the same nonces.

The test code in `src/frost/eddsa/sign.rs` (line 595) already demonstrates this pattern in practice:

```rust
sign_v2(..., presign_output.clone(), msg.clone())
```

**OT-based ECDSA (`src/ecdsa/ot_based_ecdsa/mod.rs`, line 40):**

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,      // nonce share
    pub sigma: Scalar,  // nonce-key product share
}
```

The presign comment at `src/ecdsa/ot_based_ecdsa/presign.rs` line 18–19 states: *"it's crucial that a presignature is never reused."* Yet `Clone` is derived.

**Robust ECDSA (`src/ecdsa/robust_ecdsa/mod.rs`, line 26):**

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub c: Scalar,
    pub e: Scalar,
    pub alpha: Scalar,
    pub beta: Scalar,
}
```

`docs/ecdsa/robust_ecdsa/signing.md` line 176–177 states: *"Never reuse a presignature, even across failed, aborted, or partially completed signing sessions."* The type still derives `Clone`.

The `RerandomizedPresignOutput` types for both ECDSA variants also derive `Clone` (lines 54 and 42 respectively), compounding the issue at the rerandomization layer.

---

### Impact Explanation

**FROST nonce reuse:** If a participant clones their `PresignOutput` and calls `sign_v2` twice (for the same or different messages), the same `SigningNonces` are used in two FROST round-2 signing operations. In Schnorr-based schemes, reusing a nonce across two signatures with different challenges directly exposes the participant's signing share via linear algebra: given `s1 = r + c1*x` and `s2 = r + c2*x` (same `r`), `x = (s1 - s2) / (c1 - c2)`. This constitutes **Critical: extraction of private signing shares**.

**ECDSA nonce reuse:** For OT-based and Robust ECDSA, reusing the nonce share `k` across two signatures with different message hashes `h1`, `h2` allows standard ECDSA nonce-reuse key recovery. The Robust ECDSA documentation explicitly names this attack vector at `docs/ecdsa/robust_ecdsa/signing.md` line 152–154. This constitutes **Critical: extraction of private signing shares / aggregate secret material**.

---

### Likelihood Explanation

The `Clone` derive is part of the public API surface. Any integrator who:
- Stores presign outputs in a database or cache for later use,
- Retries a failed signing session using the same presign output,
- Runs parallel signing sessions from the same presign output,

will trigger nonce reuse. The test suite itself calls `.clone()` on `PresignOutput` (line 595 of `src/frost/eddsa/sign.rs`), normalizing this pattern for library consumers. The likelihood is **High** because the API actively facilitates the unsafe operation.

---

### Recommendation

Remove `Clone` from all `PresignOutput` types. One-time-use cryptographic material should be consumed by value and not be cloneable at the type level. Rust's ownership system can enforce this invariant for free:

- `src/frost/mod.rs`: Remove `Clone` from `PresignOutput<C>`.
- `src/ecdsa/ot_based_ecdsa/mod.rs`: Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`.
- `src/ecdsa/robust_ecdsa/mod.rs`: Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`.

All signing functions already take `PresignOutput` by value (move semantics), so removing `Clone` will cause a compile error at any call site that attempts to reuse a presign output, making the invariant machine-checked rather than documentation-only.

### Proof of Concept

**FROST nonce reuse (two signatures, same nonces):**

```rust
// Attacker/misconfigured coordinator clones the presign output
let presign_output: PresignOutput<Ed25519Sha512> = run_presign(...);
let presign_clone = presign_output.clone(); // <-- Clone is derived, compiles fine

// First signing session
let sig1 = sign_v2(participants, threshold, me, coordinator,
                   keygen_output.clone(), presign_output, msg1.clone());

// Second signing session — same nonces reused
let sig2 = sign_v2(participants, threshold, me, coordinator,
                   keygen_output.clone(), presign_clone, msg2.clone());

// Both succeed. An observer with sig1 and sig2 recovers the signing share:
// s1 = nonce + c1 * x_i,  s2 = nonce + c2 * x_i
// x_i = (s1 - s2) / (c1 - c2)
```

The `Clone` derive on `PresignOutput<C>` at `src/frost/mod.rs` line 36 makes this compile without warning. The same pattern applies identically to OT-based ECDSA (`src/ecdsa/ot_based_ecdsa/mod.rs` line 40) and Robust ECDSA (`src/ecdsa/robust_ecdsa/mod.rs` line 26).
