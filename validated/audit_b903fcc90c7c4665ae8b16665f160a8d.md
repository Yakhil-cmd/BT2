### Title
`protocolUnpausePool` Sets Wrong Pause Level, Leaving Pool Swap-Blocked After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`protocolUnpausePool` is supposed to undo a protocol-level pause and restore the pool to active. Instead it transitions the pool from level 2 (protocol-paused) to level 1 (admin-paused), leaving swaps permanently blocked until the pool admin takes a separate action. If the pool was active (level 0) before the protocol pause, the protocol unpause silently leaves it in a state the admin never set, with no mechanism for the protocol owner to complete the restoration.

### Finding Description

The pool pause system uses three levels:

```
0 = active
1 = paused by admin
2 = paused by protocol
``` [1](#0-0) 

The factory enforces the following transitions:

| Function | Caller | Requires | Sets to |
|---|---|---|---|
| `pausePool` | pool admin | cur == 0 | 1 |
| `unpausePool` | pool admin | cur == 1 | 0 |
| `protocolPausePool` | owner | cur == 0 **or** 1 | 2 |
| `protocolUnpausePool` | owner | cur == 2 | **1** ← bug | [2](#0-1) 

`protocolPausePool` accepts pools that are already at level 0 (active): [3](#0-2) 

So the common path is:

1. Pool is **active** (level 0).
2. Protocol owner calls `protocolPausePool` → pool moves to level 2.
3. Protocol owner calls `protocolUnpausePool` → pool moves to level **1** (admin-paused), **not 0**.
4. Swaps are still blocked. The pool admin must now call `unpausePool` to reach level 0.

The factory stores no record of the pre-protocol-pause level, so the protocol owner has no way to complete the restoration unilaterally.

This is the direct analog of the VoteEscrowDelegation bug: a function intended to remove a restriction (`removeDelegation` / `protocolUnpausePool`) silently operates on the wrong target (wrong checkpoint key / wrong pause level), leaving the restriction fully in place.

### Impact Explanation

`swap` is the only pool action gated by `whenNotPaused`: [4](#0-3) 

After `protocolUnpausePool`, the pool sits at level 1, so `_checkNotPaused` still reverts: [5](#0-4) 

All swap traffic is blocked. LPs can still withdraw (no `whenNotPaused` on `removeLiquidity`), but the pool is commercially dead until the pool admin acts. If the pool admin is unavailable, compromised, or deliberately uncooperative, the pool remains swap-blocked indefinitely despite the protocol owner having already "unpaused" it.

### Likelihood Explanation

The trigger is the normal operational path: protocol pauses an active pool (level 0 → 2) and then unpauses it. This is the expected use of the protocol-pause mechanism (e.g., emergency response followed by recovery). Every such cycle leaves the pool at level 1 instead of 0. No special attacker is required; the bug fires on every legitimate protocol-unpause of a previously-active pool.

### Recommendation

`protocolUnpausePool` should restore the pool to fully active (level 0), not to admin-paused (level 1):

```diff
 function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
     (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-    if (cur != 2) revert InvalidPauseTransition(cur, 1);
-    IMetricOmmPoolFactoryActions(pool).setPause(1);
+    if (cur != 2) revert InvalidPauseTransition(cur, 0);
+    IMetricOmmPoolFactoryActions(pool).setPause(0);
 }
```

If the intent is to allow the admin to keep the pool paused after a protocol unpause, the pre-protocol-pause level should be stored in factory state and restored on unpause. The current code provides no such storage, so the only coherent fix is to restore to level 0.

### Proof of Concept

1. Pool is deployed and active (pauseLevel == 0).
2. Protocol owner calls `protocolPausePool(pool)`.
   - Requires `cur == 0 || cur == 1` ✓ → `setPause(2)` → pool is at level 2.
3. Emergency resolved; protocol owner calls `protocolUnpausePool(pool)`.
   - Requires `cur == 2` ✓ → `setPause(1)` → pool is at level **1**.
4. Any user calls `swap(...)`.
   - `_checkNotPaused()` reads `pauseLevel == 1 != 0` → reverts with `PoolPaused`.
5. Protocol owner has no further lever; only the pool admin can call `unpausePool` (requires `cur == 1`) to reach level 0.
6. If the pool admin is absent or hostile, swaps are permanently blocked. [2](#0-1) [6](#0-5)
