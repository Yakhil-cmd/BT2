Looking at the Cronos codebase for an analog to the "pause/unpause functions are inaccessible" vulnerability class — where a critical control mechanism is defined but never properly implemented or exposed.

The most direct analog is in `x/cronos/keeper/msg_server.go`:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

This is a complete no-op. Meanwhile, `x/cronos/keeper/permissions.go` defines a dedicated permission constant for it:

```go
const (
    CanChangeTokenMapping uint64 = 1 << iota // 1
    CanTurnBridge                             // 2
    All = CanChangeTokenMapping | CanTurnBridge // 3
)
```

The permission exists, the proto message exists, but the handler does nothing and checks nothing.

### Title
TurnBridge Message Handler is a No-Op, Permanently Disabling Bridge Emergency Controls — (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` gRPC message handler is fully defined in the proto interface and has a dedicated `CanTurnBridge` permission constant, but its implementation unconditionally returns `nil, nil` without performing any state change or permission check. This makes it permanently impossible for authorized parties to disable the bridge in an emergency, and simultaneously allows any unprivileged caller to invoke the handler without authorization.

### Finding Description
In `x/cronos/keeper/msg_server.go`, the `TurnBridge` handler is:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly gates on `CanChangeTokenMapping`:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

The `CanTurnBridge` permission constant is defined alongside `CanChangeTokenMapping` but is never referenced anywhere in the handler: [3](#0-2) 

Two distinct defects are present simultaneously:
1. **No permission check**: Any unprivileged caller can submit `MsgTurnBridge` and receive a success response (`nil` error), bypassing the `CanTurnBridge` authorization gate entirely.
2. **No state mutation**: Even a legitimately authorized caller (admin or address with `CanTurnBridge`) cannot actually toggle the bridge state, because the function body is empty.

### Impact Explanation
This is a **High** impact finding under two categories:

- **Bypass of bridge authorization checks**: An unprivileged user can call `MsgTurnBridge` and receive a success response without holding `CanTurnBridge` permission. The permission system is silently bypassed.
- **Permanent inability to control bridge/conversion flows**: If a security incident (e.g., a bridge exploit draining CRC20/CRC21 tokens) requires the bridge to be halted, the intended emergency control mechanism is permanently non-functional. Authorized parties cannot stop ongoing bridge operations through the designed path.

### Likelihood Explanation
The defect is unconditional — every invocation of `TurnBridge` is a no-op regardless of caller, parameters, or chain state. There is no code path that makes it functional. Any attacker exploiting a bridge vulnerability benefits from the permanent inability of admins to shut it down.

### Recommendation
Implement `TurnBridge` to:
1. Check that the caller holds `CanTurnBridge` permission (mirroring the pattern in `UpdateTokenMapping`).
2. Write a `bridge_enabled` boolean flag to the KV store.
3. Have bridge-sensitive paths (e.g., `ConvertVouchersToEvmCoins`, `IbcTransferCoins`) read and enforce that flag before proceeding.

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    k.SetBridgeEnabled(ctx, msg.Enable)
    return &types.MsgTurnBridgeResponse{}, nil
}
```

### Proof of Concept
1. Submit a `MsgTurnBridge{Enable: false}` from any unprivileged address.
2. Observe: the transaction succeeds (no error returned).
3. Observe: the bridge state is unchanged — `ConvertVouchersToEvmCoins` and `IbcTransferCoins` continue to process normally.
4. An admin attempting to halt the bridge during an active exploit receives a success response but achieves nothing; the exploit continues unimpeded. [1](#0-0) [4](#0-3) [5](#0-4)

		return err
	}

	params := k.GetParams(ctx)
	evmParams := k.GetEvmParams(ctx)
	for _, c := range coins {
		switch c.Denom {
		case params.IbcCroDenom:
			if params.IbcCroDenom == "" {
				return errorsmod.Wrap(types.ErrIbcCroDenomEmpty, "ibc is disabled")
			}

			// Send ibc tokens to escrow address
			err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, acc, types.ModuleName, sdk.NewCoins(c))
			if err != nil {
				return err
			}
			// Compute new amount, because basecro is a 8 decimals token, we need to multiply by 10^10 to make it
			// a 18 decimals token
			amount18dec := sdk.NewCoin(evmParams.EvmDenom, c.Amount.Mul(sdkmath.NewIntFromBigInt(types.TenPowTen)))

			// Mint new evm tokens
			if err := k.bankKeeper.MintCoins(

				return err
			}

			// Send evm tokens to receiver

			); err != nil {
				return err
			}

		default:
			err := k.ConvertCoinFromNativeToCRC21(ctx, common.BytesToAddress(acc.Bytes()), c, params.EnableAutoDeployment)
			if err != nil {
				return err
			}
		}
	}
	defer func() {
