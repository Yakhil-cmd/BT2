### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees via `setPoolBinAdditionalFees`, Bypassing `maxAdminSpreadFeeE6` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to `pool.setBinAdditionalFees` with **no cap validation**, while the parallel admin fee setter `setPoolAdminFees` explicitly enforces `maxAdminSpreadFeeE6`. A pool admin can set `addFeeBuyE6 = type(uint16).max = 65535` (6.5535 % in E6 units) on any bin, and the swap engine adds this value on top of the oracle spread fee, causing traders to pay materially more than the configured cap implies.

### Finding Description

`setPoolAdminFees` enforces the cap:

```solidity
// MetricOmmPoolFactory.sol:414
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has **no equivalent check**:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` also has no cap — it only validates the bin index:

```solidity
// MetricOmmPool.sol:469-473
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

In every swap direction the stored `addFeeBuyE6` / `addFeeSellE6` is added directly to `baseFeeX64` before computing the fee charged to the trader:

```solidity
// MetricOmmPool.sol:910 (and 999, 1177)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The resulting `currBinBuyFeeX64` drives `feeAmountScaled`, which is added to `amountInScaled` — the actual token amount the trader must supply:

```solidity
uint256 feeAmountScaled = Math.ceilDiv(amountInScaled * currBinBuyFeeX64, ONE_X64);
amountInScaled += feeAmountScaled;
``` [5](#0-4) 

`maxAdminSpreadFeeE6` defaults to `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) and can be lowered by the factory owner. `type(uint16).max = 65535` represents 6.5535 % — an additional fee the pool admin can impose on any single bin with no factory-level guard. [6](#0-5) 

### Impact Explanation

Traders swapping through an affected bin pay up to 6.5535 % more than the oracle spread alone, on top of whatever `spreadFeeE6` is already configured. The excess fee accrues to LPs (not to the admin directly), but the trader suffers a direct, quantifiable loss of swap output. For any non-trivial swap size this exceeds Sherlock Medium thresholds. The `getSellAndBuyPrices` view also incorporates `addFeeBuyE6` into the quoted ask price, so the quoted price is consistent with the inflated fee — traders have no on-chain signal that the bin fee exceeds the advertised cap. [7](#0-6) 

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly expected to operate only within caps and timelocks. The factory enforces this for `setPoolAdminFees` but omits it for `setPoolBinAdditionalFees`. Any pool admin — including one who turns adversarial after deployment — can exploit this immediately with a single transaction, with no timelock.

### Recommendation

Add a cap check in `MetricOmmPoolFactory.setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (uint256(addFeeBuyE6)  + uint256(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
// or enforce each side independently:
if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, add the same guard inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's `maxAdminSpreadFeeE6` via a callback, or store the cap as an immutable on the pool at deployment time.

### Proof of Concept

1. Deploy factory with default `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Create a pool with `adminSpreadFeeE6 = 0` (admin spread fee at zero, well within cap).
3. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
4. Observe `_binStates[0].addFeeBuyE6 == 65535` — no revert.
5. Perform a swap through bin 0; the effective fee applied is `baseFeeX64 + 65535/1e6` in X64 units.
6. Assert the trader's input cost exceeds what `maxAdminSpreadFeeE6 = 200_000` alone would imply, confirming the cap is bypassed. [2](#0-1) [8](#0-7)
