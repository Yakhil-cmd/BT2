### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

The `setPoolBinAdditionalFees` path in `MetricOmmPoolFactory` applies no cap check on `addFeeBuyE6` / `addFeeSellE6`, allowing a semi-trusted pool admin to impose per-bin fees that exceed the factory-enforced `maxAdminSpreadFeeE6` ceiling, causing traders to pay more than the protocol's hard limit permits.

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` correctly enforces the cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` passes the values straight through to the pool with no validation:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` likewise performs no cap check beyond the `int8 bin` range: [3](#0-2) 

At swap time the per-bin fee is added directly on top of the base spread fee:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
``` [4](#0-3) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so their maximum value is **65 535** (≈ 6.55 % in E6 units). Because no cap is applied, the pool admin can set any bin's additional fee to this maximum, adding 6.55 % on top of whatever base spread fee is already in effect — a value that can exceed `maxAdminSpreadFeeE6` (`HARD_MAX_SPREAD_FEE_E6 = 200 000`, i.e. 20 %) by itself, and that the factory owner cannot override by lowering `maxAdminSpreadFeeE6` (which only affects the base component). [5](#0-4) 

There is also no timelock on this call, unlike the price-provider rotation path.

### Impact Explanation

Every trader whose swap routes through the targeted bin pays `addFeeBuyE6` or `addFeeSellE6` extra on top of the capped base fee. At the uint16 maximum this is an additional 6.55 % extracted from the trader's output (or added to their input), constituting a direct loss of user principal that the factory's cap system is designed to prevent. The factory owner cannot remedy this by reducing `maxAdminSpreadFeeE6` because that cap does not govern bin additional fees.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly expected to operate only within factory-enforced caps. The call requires no timelock, no co-signer, and no special precondition beyond holding the `poolAdmin` role for the target pool. Any pool admin who wishes to extract excess fees can do so immediately and silently.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. For example, enforce that `addFeeBuyE6` and `addFeeSellE6` each do not exceed `maxAdminSpreadFeeE6` (or a dedicated `maxAdminBinAdditionalFeeE6` constant), and revert with `AdminFeeTooHigh()` if the limit is breached.

### Proof of Concept

1. Factory is deployed with `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — accepted, base admin spread fee is at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**, bin 0 now carries an additional 6.55 % buy/sell fee.
4. A trader swaps through bin 0 and pays ≈ 26.55 % total spread fee (20 % base + 6.55 % bin additional), far above the 20 % hard cap the factory is supposed to enforce.
5. The factory owner lowers `maxAdminSpreadFeeE6` to 0 — the base fee drops, but the bin additional fee of 65 535 remains unchanged and continues to drain traders. [2](#0-1) [3](#0-2)
