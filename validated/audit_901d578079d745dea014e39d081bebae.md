### Title
Missing Lower-Bound Validation for `threshold` in FROST `assert_sign_inputs` — (File: src/frost/mod.rs)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that the threshold does not exceed the participant count, but never checks that the threshold is at least 2. A caller supplying `threshold = 1` (or `threshold = 0`) bypasses the guard entirely. The same gap exists in the FROST `presign` function. This is directly analogous to the TokenBridge finding: a critical numeric parameter is accepted without a non-trivial lower-bound check, allowing the protocol to proceed with a fundamentally invalid configuration.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
// upper bound
if threshold > participants.len() { … ThresholdTooLarge … }
// lower bound
if threshold < 2 { … ThresholdTooSmall … }
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` only enforces the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no lower-bound check; threshold = 1 or 0 is silently accepted
``` [2](#0-1) 

The same omission appears in the FROST `presign` entry point:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no lower-bound check
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforcement of its own, so any value including 0 or 1 can be constructed and passed in: [4](#0-3) 

---

### Impact Explanation

FROST signing uses the threshold to compute Lagrange interpolation coefficients over the participants' signing shares. The shares were generated during DKG with a polynomial of degree `t − 1` (where `t ≥ 2`). If a caller passes `threshold = 1` at signing time, the interpolation treats the polynomial as degree-0 (a constant), computing coefficients that are inconsistent with the actual share structure. The resulting aggregated signature share will not verify against the public key. Honest parties complete the protocol and accept the output as a valid signing transcript, but the produced signature is cryptographically unusable — a corruption of the signing output.

**Matched impact**: *High — Corruption of sign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs.*

---

### Likelihood Explanation

`assert_sign_inputs` is a public library function. Any library caller — including a malicious coordinator who controls the threshold argument passed to the FROST signing entry points — can supply `threshold = 1`. No special privilege, leaked key, or cryptographic break is required. The call path is direct: caller → `assert_sign_inputs` → signing protocol proceeds with invalid threshold. [5](#0-4) 

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces, in both `assert_sign_inputs` and the FROST `presign` function:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing pattern in `assert_key_invariants`: [6](#0-5) 

---

### Proof of Concept

1. Call `assert_sign_inputs` with a valid participant list of size 3, `me` and `coordinator` both present, but `threshold = 1`.
2. Observe that the function returns `Ok(participants)` — no error is raised.
3. Proceed to the FROST signing protocol with this threshold.
4. The Lagrange coefficients are computed for a degree-0 polynomial; the aggregated signature share does not satisfy the verification equation for the public key produced by DKG (which used `threshold = 2`).
5. All honest parties complete the protocol and accept the corrupted output.

The inconsistency is confirmed by comparing the guard in `assert_key_invariants` (which rejects `threshold < 2`) with the missing guard in `assert_sign_inputs` (which accepts any value ≤ `participants.len()`). [7](#0-6) [8](#0-7)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
