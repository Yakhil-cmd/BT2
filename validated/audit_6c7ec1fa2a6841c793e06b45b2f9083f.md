### Title
`FeeEstimation` omits `l2_gas_consumed`, making `overall_fee` description wrong and leaving clients unable to derive correct `l2_gas.max_amount` resource bounds — (File: `crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. The struct's own doc comment states `overall_fee` equals `gas_consumed * gas_price + data_gas_consumed * data_gas_price`, but the actual value stored in `overall_fee` is the full receipt fee, which includes the L2 gas cost. The two quantities diverge for every V3 `AllResources` transaction that consumes non-zero L2 gas — which is the common case for Cairo execution. Clients that trust the documented formula receive an authoritative-looking undercount of the fee, and have no way to derive the `l2_gas.max_amount` they must supply in their resource bounds.

---

### Finding Description

In `tx_execution_output_to_fee_estimation`:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),      // l1_gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // l1_data_gas only
    l1_data_gas_price,
    l2_gas_price,                                  // price present …
    // l2_gas_consumed is ABSENT                   // … but consumed is not
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes l2_gas cost
    unit: tx_execution_output.price_unit,
})
```

The `overall_fee` is taken from the blockifier receipt, which is computed by `GasVector::cost` as:

```
fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)
```

The struct's doc comment, however, says:

```
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
```

This description omits the L2 gas term entirely. The internal TODO acknowledges the gap:

```
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

The OpenAPI schema (`starknet_api_openrpc.json`) repeats the same wrong formula in its `FEE_ESTIMATE` description.

A client following the documented formula computes:

```
client_fee = gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price
```

For any transaction with non-zero L2 gas (all Cairo-VM / Sierra execution), `client_fee < overall_fee`. The gap equals `l2_gas_consumed × l2_gas_price`, which can be the dominant term for compute-heavy transactions (the sample RPC record shows `l2_gas_consumed = 0xb56b6` at `l2_gas_price = 0x1dcd65000`, contributing ~`0x151eb86f3ed400` FRI — the entire `overall_fee`).

Because `l2_gas_consumed` is absent, a client building a V3 `AllResources` transaction has no API-provided value to place in `l2_gas.max_amount`. Setting it to zero causes the blockifier to reject the transaction with `MaxGasAmountExceeded { resource: L2Gas }` at post-execution check. Setting it to an arbitrary large value causes over-payment. Neither outcome is derivable from the documented interface.

---

### Impact Explanation

**Matches:** "High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."

The `overall_fee` field carries a documented invariant (`= gas_consumed × P1 + data_gas_consumed × P2`) that is false for every V3 transaction with non-zero L2 gas. Any client or tooling that reconstructs the fee from the breakdown — to verify the estimate, to set resource bounds, or to display a cost breakdown — receives a systematically wrong (lower) number. The missing `l2_gas_consumed` also means the resource-bounds field that gates execution (`l2_gas.max_amount`) cannot be set correctly from the fee estimation response alone, causing valid transactions to be rejected or users to over-commit funds.

---

### Likelihood Explanation

All V3 `AllResources` transactions that execute Cairo code consume L2 gas. This is the standard transaction type for Starknet v0.13.3+. The discrepancy is present in every such fee estimation call. Any wallet, SDK, or dApp that calls `starknet_estimateFee` and uses the returned breakdown to populate resource bounds is affected.

---

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`.
2. Correct the doc comment on `overall_fee` to include the L2 gas term.
3. Update the OpenAPI `FEE_ESTIMATE` schema description to match.
4. Add a test asserting `gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price + l2_gas_consumed × l2_gas_price == overall_fee` for a transaction with non-zero L2 gas.

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}

    /// Computes the cost (in fee token units) of the gas vector (panicking on overflow).
    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }

    pub fn check_all_gas_amounts_within_bounds(

        gas_vector: &GasVector,
    ) -> FeeCheckResult<()> {
        // TODO(Arni): Consider refactoring the returned error. The first failed check will hide
        // future checks.
        for (resource, max_amount, actual_amount) in [
            (L1Gas, max_amount_bounds.l1_gas, gas_vector.l1_gas),
            (L2Gas, max_amount_bounds.l2_gas, gas_vector.l2_gas),
            (L1DataGas, max_amount_bounds.l1_data_gas, gas_vector.l1_data_gas),
        ] {
            if max_amount < actual_amount {
                return Err(FeeCheckError::MaxGasAmountExceeded {
                    resource,
                    max_amount,
                    actual_amount,
                });
            }
        }

        Ok(())
    }
