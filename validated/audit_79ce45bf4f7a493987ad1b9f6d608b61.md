### Title
Silent `unwrap_or(NonzeroGasPrice::MIN)` Substitution in RPC Fee Estimation Returns Authoritative-Looking Wrong Values for Zero-Price Blocks - (`File: crates/apollo_rpc_execution/src/lib.rs`)

### Summary

`prepare_block_execution` in `crates/apollo_rpc_execution/src/lib.rs` silently replaces every zero gas price with `NonzeroGasPrice::MIN` (= 1 wei/fri) instead of returning an error. This is the direct Sequencer-native analog of the deprecated Chainlink `latestAnswer` returning 0 silently: a missing/invalid oracle value is swallowed and a wrong sentinel is fed downstream, producing authoritative-looking but incorrect fee-estimation and simulation results.

### Finding Description

`prepare_block_execution` builds the `BlockContext` used by every RPC simulation, fee-estimation, and tracing call. When the stored block header contains a zero gas price — which is the documented case for all pre-0.13.1 Starknet blocks — the code does not return an error. Instead it silently substitutes `NonzeroGasPrice::MIN` (the constant `1`) for every affected price field:

```rust
// TODO(yair): What to do about blocks pre 0.13.1 where the data gas price were 0?
gas_prices: GasPrices {
    eth_gas_prices: GasPriceVector {
        l1_gas_price: NonzeroGasPrice::new(l1_gas_price.price_in_wei)
            .unwrap_or(NonzeroGasPrice::MIN),          // ← silent 0 → 1
        l1_data_gas_price: NonzeroGasPrice::new(l1_data_gas_price.price_in_wei)
            .unwrap_or(NonzeroGasPrice::MIN),
        l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_wei)
            .unwrap_or(NonzeroGasPrice::MIN),
    },
    strk_gas_prices: GasPriceVector {
        l1_gas_price: NonzeroGasPrice::new(l1_gas_price.price_in_fri)
            .unwrap_or(NonzeroGasPrice::MIN),
        l1_data_gas_price: NonzeroGasPrice::new(l1_data_gas_price.price_in_fri)
            .unwrap_or(NonzeroGasPrice::MIN),
        l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_fri)
            .unwrap_or(NonzeroGasPrice::MIN),
    },
},
``` [1](#0-0) 

The resulting `BlockContext` is then passed directly to `tx_execution_output_to_fee_estimation`, which reads the gas prices back out and returns them as the `l1_gas_price`, `l1_data_gas_price`, and `l2_gas_price` fields of the `FeeEstimation` response:

```rust
let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
    gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
    ...
);
``` [2](#0-1) 

The `NonzeroGasPrice::MIN` constant is `GasPrice(1)`: [3](#0-2) 

So for any block whose stored header has a zero price in any of the six price fields, the RPC layer reports a gas price of 1 wei/fri — a value that is off by many orders of magnitude from the real price — without any error, warning to the caller, or indication that the value is synthetic.

The TODO comment confirms this is an unresolved design gap, not a deliberate safe default: [4](#0-3) 

### Impact Explanation

Every call to `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_traceTransaction` that targets a block with any zero-valued gas price field will receive a `FeeEstimation` whose `l1_gas_price`, `l1_data_gas_price`, or `l2_gas_price` is `1` instead of the correct value. The `overall_fee` computed from `gas_vector × gas_price` is therefore also wrong by the same factor. The response carries no flag indicating the price is synthetic; callers treat it as authoritative. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

### Likelihood Explanation

Pre-0.13.1 Starknet mainnet blocks have zero `l1_data_gas_price` and zero `l2_gas_price` fields in their stored headers. Any unprivileged caller can trigger the path by passing a historical `BlockId` to `starknet_estimateFee`. No special account, key, or privileged role is required.

### Recommendation

Replace the silent `.unwrap_or(NonzeroGasPrice::MIN)` calls with an explicit error path. When a stored gas price is zero, `prepare_block_execution` should return an `ExecutionError` (e.g., `GasPriceParsingFailure`) so the RPC layer can surface a clear error to the caller rather than a plausible-looking but wrong fee estimate. If backward compatibility with pre-0.13.1 blocks is required, document the substitution explicitly and propagate a warning field in the response.

```rust
// Recommended: fail loudly instead of silently substituting 1
l1_gas_price: NonzeroGasPrice::new(l1_gas_price.price_in_wei)
    .map_err(|_| ExecutionError::GasPriceParsingFailure(l1_gas_price.price_in_wei))?,
```

### Proof of Concept

1. Identify any Starknet mainnet block with number < the 0.13.1 upgrade height (where `l1_data_gas_price` and `l2_gas_price` were not yet tracked and are stored as zero).
2. Call `starknet_estimateFee` with any valid invoke transaction and `block_id = {"block_number": <pre-0.13.1 block>}`.
3. Observe that the returned `FeeEstimation.l1_data_gas_price` and `FeeEstimation.l2_gas_price` are `0x1` (1 wei/fri) rather than the correct historical values, and that `overall_fee` is computed from those wrong prices.
4. No error is returned; the response is structurally identical to a correct fee estimate. [5](#0-4) [6](#0-5)

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
