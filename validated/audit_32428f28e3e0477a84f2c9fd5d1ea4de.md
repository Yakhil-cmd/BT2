### Title
Stale `lastDecayTs` When `totalShares == 0` Allows Stop-Loss Watermark to Decay to Zero, Bypassing LP Drawdown Protection — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` skips `_checkAndUpdateWatermarks` for any bin whose `totalShares == 0`. This means `hwmS.lastDecayTs` is never advanced while a bin is empty. When liquidity is later re-added and a swap occurs, the elapsed time `dt = block.timestamp - hwmS.lastDecayTs` is computed from the stale timestamp, causing the watermark to decay by the full accumulated idle period in a single step. If `decayPerSecondE8 * dt >= 1e8`, the watermark collapses to zero, and the stop-loss never fires — even if the current per-share value is far below the configured drawdown floor.

`_decayed` collapses the watermark to zero when `ratePerSecondE8 * dt >= 1e8`:

```solidity
// lines 319-324
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;              // ← watermark wiped
    return hwm - (hwm * factor) / E8;
}

### Impact Explanation

An LP who deposits into a bin that previously had all liquidity removed (and whose `lastDecayTs` is therefore stale) receives no stop-loss protection on the first swap after re-entry. The watermark silently resets to whatever the current per-share value is — even if that value is already below the drawdown floor relative to the historical high. Subsequent swaps that drain the bin further are also unprotected until the watermark ratchets up again. This is a direct loss of LP principal beyond the drawdown threshold the extension was configured to enforce.

### Likelihood Explanation

The condition is reachable through normal, unprivileged operation:

1. LP1 adds liquidity; watermark `W` is set, `lastDecayTs = T1`.
2. LP1 removes all liquidity; `totalShares = 0`; `lastDecayTs` stays at `T1`.
3. Time passes. At the default illustrative rate of `decayPerSecondE8 = 58` (~5 %/day), the watermark fully decays after `1e8 / 58 ≈ 20 days` of idle time.
4. LP2 adds liquidity.
5. Any swap touching that bin computes `dt = now − T1 ≥ 20 days`, decays the watermark to 0, resets it to the current (possibly already-impaired) metric, and does not revert.

No privileged role, malicious token, or exotic setup is required.

### Recommendation

Advance `lastDecayTs` for empty bins without computing or checking watermarks, so the clock is never stale when liquidity returns:

```solidity
for (uint256 i = 0; i < count; i++) {
    uint256 totalShares = PoolStateLibrary._decodeBinTotalShares(shares[i]);
    if (totalShares == 0) {
        // Advance the decay clock even when the bin is empty,
        // so the watermark is not over-decayed on re-entry.
        BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdxs[i]];
        if (hwmS.lastDecayTs != 0) {
            hwmS.lastDecayTs = uint32(block.timestamp);
        }
        continue;
    }
    ...
    _checkAndUpdateWatermarks(...);
}
```

Alternatively, apply the decay lazily inside `_checkAndUpdateWatermarks` but cap `dt` at the time the bin last had non-zero shares (requires tracking that timestamp separately).

  function _checkAndUpdateWatermarks(
    address pool_,
    int8 binIdx,
    uint256 metricT0,
    uint256 metricT1,
    uint256 floorMultiplier,
    uint256 decayRate,
    bool zeroForOne
  ) private {
    BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdx];
    uint256 dt = block.timestamp - hwmS.lastDecayTs;

    (uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
