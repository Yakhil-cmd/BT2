### Title
Missing Validation of Participant Shares in CKD Coordinator Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function in the Confidential Key Derivation protocol blindly sums participant-supplied `(big_y, big_c)` ElGamal share pairs without any proof-of-correctness check. A single malicious participant can inject arbitrary group elements, causing the coordinator to accept and return a `CKDOutput` that decrypts to an attacker-shifted value instead of the legitimate `msk · H(pk, app_id)`.

### Finding Description
In `do_ckd_coordinator`, the coordinator collects each participant's `CKDOutput` and accumulates it directly:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each honest participant is supposed to compute:

- `big_y_i = y_i · G`
- `big_c_i = λ_i · x_i · H(pk, app_id) + y_i · app_pk`

and send `(λ_i · big_y_i, λ_i · big_c_i)` to the coordinator. The coordinator then sums these to obtain an ElGamal encryption `(Y, C)` of `msk · H(pk, app_id)` under `app_pk`. [2](#0-1) 

There is **no NIZK proof** (or any other check) that the received `(big_y, big_c)` pair was computed using the participant's actual key share `x_i` and a consistent randomness `y_i`. No identity-element check is performed either. The coordinator accepts whatever values arrive over the channel.

Contrast this with the DKG protocol, which requires and verifies a Schnorr proof-of-knowledge for every participant's polynomial commitment before accepting it: [3](#0-2) 

And with both ECDSA signing variants, which verify the assembled signature against the public key before returning it: [4](#0-3) [5](#0-4) 

The CKD protocol has no equivalent final-output or per-share validation.

### Impact Explanation
A malicious participant `j` sends:

- `big_y_j' = λ_j · y_j · G + Δ_Y`
- `big_c_j' = λ_j · (x_j · H(pk, app_id) + y_j · app_pk) + Δ_C`

for arbitrary group elements `Δ_Y`, `Δ_C`. After summation the coordinator holds:

```
Y' = Y + Δ_Y
C' = msk · H(pk, app_id) + y · app_pk + Δ_C
```

Decryption by the application yields `C' − app_sk · Y' = msk · H(pk, app_id) + Δ_C − app_sk · Δ_Y`, which is an attacker-controlled shift of the legitimate derived key. The coordinator, an honest party, accepts and returns this corrupted `CKDOutput` with no error.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
The `ckd` entry point accepts no threshold parameter; the protocol requires every listed participant to contribute. [6](#0-5) 

Therefore a **single** malicious participant is sufficient to corrupt the output. There is no redundancy or majority vote to absorb one bad actor. Any participant that has been compromised (or is acting adversarially) can silently shift the derived key to an arbitrary value, with no detection mechanism available to the coordinator.

### Recommendation
Add a Chaum–Pedersen (or equivalent) NIZK proof of discrete-log equality to each participant's share, proving that `big_y_i = y_i · G` and `big_c_i = λ_i · x_i · H(pk, app_id) + y_i · app_pk` for the **same** `y_i`. The coordinator must verify all proofs before accumulating shares. Additionally, after accumulation, verify that neither `big_Y` nor `big_C` is the group identity. This mirrors the proof-of-knowledge pattern already used in `do_keyshare` for DKG commitments.

### Proof of Concept
1. Deploy the CKD protocol with participants `[P1, P2, P3]`; `P1` is the coordinator.
2. `P2` (malicious) computes its legitimate share but replaces `big_c` with `big_c + Δ_C` for an arbitrary non-zero `Δ_C` before sending to `P1`.
3. `P1` runs `do_ckd_coordinator`, sums all shares without validation, and returns `CKDOutput::new(norm_big_y, norm_big_c + Δ_C)`.
4. The application calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk, app_id) + Δ_C − app_sk · 0 = msk · H(pk, app_id) + Δ_C`, a value that differs from the expected derived key by `Δ_C`.
5. All honest parties accept this output; no error is raised anywhere in `src/confidential_key_derivation/protocol.rs`. [7](#0-6)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
