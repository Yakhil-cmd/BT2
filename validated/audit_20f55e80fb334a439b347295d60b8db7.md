Audit Report

## Title
Unprivileged Caller Can Frontrun Victim's `update_balance` to Cause Persistent `AlreadyProcessing` DoS - (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
The `update_balance` endpoint accepts an optional `owner` principal without verifying that the caller is authorized to act on behalf of that principal. The `balance_update_guard` is keyed on the resolved account (not the caller), and is held across multiple async inter-canister calls. Any unprivileged attacker can continuously call `update_balance(owner = victim_principal)` to keep the guard occupied for the victim's account, causing every legitimate call from the victim to return `AlreadyProcessing` and permanently blocking ckBTC minting for confirmed on-chain deposits.

## Finding Description
In `update_balance` (`rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, lines 164–168), the target account is resolved from the caller-supplied `args.owner` with no authorization check:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),  // attacker sets this to victim's principal
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

The only existing check (line 149) only prevents setting `owner` to the minter's own principal — it does not require `args.owner == Some(caller)`.

In `guard.rs` (lines 45–60), `Guard::new` inserts the account into `state.update_balance_accounts` and returns `GuardError::AlreadyProcessing` if already present. The guard is held for the entire async function lifetime, spanning at minimum two sequential `get_utxos` calls to the Bitcoin canister (lines 175–183 and 225–236). The guard is only released on `Drop` at function exit (guard.rs lines 63–66).

The `AlreadyProcessing` guard error maps directly to `UpdateBalanceError::AlreadyProcessing` (update_balance.rs lines 108–116) and is returned to the caller.

**Exploit flow:**
1. Attacker (any non-anonymous principal) calls `update_balance({ owner = opt principal "VICTIM"; subaccount = null })`.
2. Attacker's call acquires `balance_update_guard` for the victim's account and suspends at the first `get_utxos` await.
3. Victim calls `update_balance({ owner = null; subaccount = null })` — resolves to the same account — and immediately receives `AlreadyProcessing`.
4. Attacker pre-queues additional calls so the guard is re-acquired as soon as it is released, maintaining continuous coverage.

The `MAX_CONCURRENT = 100` limit (guard.rs line 6) does not protect against this: a targeted single-account attack requires only one slot.

## Impact Explanation
A victim with confirmed on-chain BTC deposits is permanently denied the ability to mint ckBTC for as long as the attacker continues. The victim's funds are not stolen but are inaccessible in ckBTC form indefinitely. Additionally, each attacker call forces the minter canister to perform Bitcoin canister round-trips, draining the minter's cycles at the attacker's direction. This constitutes a significant ck-token / Chain Fusion security impact with concrete, sustained user harm, matching the **High** bounty impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation
The attack requires only a valid non-anonymous IC principal and knowledge of the victim's principal ID (which is public on-chain). No BTC, ckBTC, or special permissions are needed. The attacker pays only standard IC ingress fees (a few cycles per call), while the minter bears the cost of Bitcoin canister calls. The attack is trivially scriptable, can be sustained indefinitely, and is targeted — it does not require exhausting the global `MAX_CONCURRENT` limit.

## Recommendation
Restrict `update_balance` so that an unprivileged ingress caller cannot specify an `owner` other than themselves. The minimal fix is to add an authorization check immediately after resolving `caller_account`:

```rust
if let Some(owner) = args.owner {
    if owner != caller {
        ic_cdk::trap("unauthorized: cannot update balance on behalf of another principal");
    }
}
```

This mirrors the approach in `retrieve_btc_with_approval` (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, lines 248–263), which always derives `caller_account` directly from `ic_cdk::api::msg_caller()` without an overridable owner field.

If third-party notification by a helper canister is a desired feature, the guard should be keyed on the **caller** rather than the resolved owner, so an attacker's guard cannot block the victim's own call.

## Proof of Concept
```
// Attacker (any non-anonymous principal) submits in a tight loop:
update_balance({ owner = opt principal "VICTIM_PRINCIPAL_ID"; subaccount = null })

// Victim submits their legitimate call:
update_balance({ owner = null; subaccount = null })
// → Returns: Err(AlreadyProcessing)
// because the attacker's in-flight call holds balance_update_guard for victim's account
// across the async get_utxos() Bitcoin canister call.
```

A deterministic integration test using PocketIC can confirm this by: (1) initializing the ckBTC minter, (2) submitting an attacker call with `owner = victim_principal` and pausing execution at the first `get_utxos` await point, (3) submitting the victim's call and asserting it returns `AlreadyProcessing`.
