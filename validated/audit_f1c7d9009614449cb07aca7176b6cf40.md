### Title
Missing Validation of Participant Contributions in CKD Coordinator Allows Output Corruption - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` blindly aggregates group elements received from all participants without any proof of correctness or commitment verification. A single malicious participant can send arbitrary `(big_y, big_c)` values, causing the coordinator to produce and return a corrupted `CKDOutput` that honest parties will accept as valid.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives a `CKDOutput` from every other participant and unconditionally adds the contained group elements into the running totals: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

There is no zero-knowledge proof, commitment binding, or any other cryptographic check on the received `(big_y, big_c)` pairs. The protocol simply trusts whatever each participant sends.

Contrast this with the DKG protocol, which enforces correctness of every participant's contribution through a proof-of-knowledge check and a commitment-hash binding before any share is accepted: [2](#0-1) 

Similarly, the Robust ECDSA presign protocol validates received shares via exponent-interpolation consistency checks before using them: [3](#0-2) 

And the Robust ECDSA signing coordinator verifies the final assembled signature against the public key before returning it: [4](#0-3) 

No equivalent safeguard exists in the CKD coordinator path.

---

### Impact Explanation

The `CKDOutput` encodes `(Y, C)` where the honest protocol computes:

- `Y = Σ λᵢ · yᵢ · G` (blinding factor)
- `C = Σ λᵢ · (xᵢ · H(pk, app_id) + yᵢ · app_pk)` (masked derived key)

The caller unmasks the output as `C − app_sk · Y = msk · H(pk, app_id)`.

A malicious participant who sends `(big_y', big_c')` instead of their honest contribution shifts the final result to:

```
C' − app_sk · Y' = msk · H(pk, app_id) + (big_c' − app_sk · big_y')
```

The additive error term `(big_c' − app_sk · big_y')` is non-zero for any arbitrary choice of `(big_y', big_c')` that the attacker makes without knowing `app_sk`. The coordinator returns this corrupted output with no error, and the honest caller has no way to detect the corruption.

This maps directly to the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.** [5](#0-4) 

---

### Likelihood Explanation

The CKD protocol is an N-of-N aggregation: the coordinator waits for a contribution from **every** other participant before producing the output. [6](#0-5) 

Therefore a **single** malicious participant out of N is sufficient to corrupt the output. The attacker-controlled entry path is straightforward: any participant that has been legitimately enrolled in the protocol can send a malformed `CKDOutput` message to the coordinator during the single communication round. No special privilege, leaked key, or external assumption is required.

---

### Recommendation

Each participant should attach a non-interactive zero-knowledge proof (e.g., a Schnorr proof of discrete-log equality) demonstrating that `big_y` and `big_c` were computed correctly from their committed key share and the agreed `app_id`. The coordinator must verify all proofs before aggregating. This is the same pattern already applied in `do_keyshare` via `verify_proof_of_knowledge`. [7](#0-6) 

Alternatively, a commitment-then-reveal round (analogous to the `commitment_hash` round in DKG) can bind each participant to their contribution before the coordinator aggregates.

---

### Proof of Concept

1. Honest participants `P1 … P_{N-1}` and malicious participant `Pm` all call `ckd(...)` with the same `(participants, coordinator, app_id, app_pk)`.
2. `Pm` intercepts the outgoing message to the coordinator and replaces `(norm_big_y, norm_big_c)` with arbitrary non-identity group elements `(G, G)`.
3. The coordinator receives all contributions, adds them without verification, and returns `CKDOutput::new(norm_big_y + G, norm_big_c + G)`.
4. The caller invokes `ckd_output.unmask(app_sk)` and obtains `msk · H(pk, app_id) + (G − app_sk · G)`, which is not the intended derived key.
5. No error is raised; the corrupted output is silently accepted. [1](#0-0)

fn verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    threshold: ReconstructionLowerBound,
    participant: Participant,
    old_participants: Option<ParticipantList>,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: Option<&Signature<C>>,
) -> Result<(), ProtocolError> {
    let threshold = threshold.value();
    match proof_of_knowledge {
        // if participant did not send anything but he is actually an old participant
        None => {
            // if basic dkg or participant is old
            if old_participants.is_none_or(|p| p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since previous line did not abort, then we know participant is new indeed
            // check the commitment length is threshold - 1
            if commitment.coefficients().len() != threshold - 1 {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }
            // nothing to verify
            Ok(())
        }
        // now we know the proof is not none
        Some(proof_of_knowledge) => {
            // if participant sent something but he is actually a new participant
            if old_participants.is_some_and(|p| !p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since the previous did not abort, we know the participant is old or we are dealing with a dkg
            if commitment.coefficients().len() != threshold {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }

            // creating an identifier as required by the syntax of verify_proof_of_knowledge of frost_core
            internal_verify_proof_of_knowledge(
                session_id,
                domain_separator,
                participant,
                commitment,
                proof_of_knowledge,
            )
        }
    }
}
