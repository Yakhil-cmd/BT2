### Title
Accrued Admin Fees Redirected to New Destination When `setPoolAdminFeeDestination` Is Called Without Prior Collection — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. All fees that accumulated while the old destination was active are subsequently paid to the new destination on the next `collectPoolFees` call, permanently denying the old destination its earned share.

---

### Finding Description

Every other fee-mutating function in `MetricOmmPoolFactory` — `setPoolAdminFees` and `setPoolProtocolFee` — calls `collectFees` on the pool **before** writing any new configuration, so that accrued balances are settled at the rates and destination that were in effect when those fees were earned.

`setPoolAdminFeeDestination` does not follow this pattern:

```solidity
// MetricOmmPoolFactory.sol L438-447
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;          // ← destination overwritten
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
    // ← no collectFees call; all accrued fees now belong to the new address
}
``` [1](#0-0) 

By contrast, `setPoolAdminFees` explicitly collects at the **old** destination before any state change:

```solidity
// MetricOmmPoolFactory.sol L417-425
PoolFeeConfig memory c = poolFeeConfig[pool];
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6,
    c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8,
    c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination, before update
);
``` [2](#0-1) 

The pool's `collectFees` distributes both the **spread surplus** (implicit in the token balance above LP bins) and the **notional fee accumulators** (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) to whichever `adminFeeDestination_` is passed at call time: [3](#0-2) 

Because `collectPoolFees` (permissionless) and all subsequent fee-change calls will now pass the **new** destination, every token unit that accrued under the old destination is irrecoverably redirected.

---

### Impact Explanation

The old admin fee destination permanently loses all fees that accrued before the destination change. Both spread-surplus tokens and notional-fee accumulator tokens are affected. The loss is proportional to the admin fee share and the volume traded since the last collection — potentially large for high-volume pools with infrequent collection. The new destination receives a windfall it did not earn.

This is a direct loss of protocol/admin fee principal with no recovery path once `collectPoolFees` is called after the destination update.

---

### Likelihood Explanation

`setPoolAdminFeeDestination` is a routine operational action (treasury rotation, multisig upgrade, key compromise response). Any pool admin performing a standard treasury handover triggers the loss. No special market conditions or attacker cooperation is required — the pool admin's own legitimate call is sufficient.

---

### Recommendation

Call `collectFees` on the pool using the **current** (old) destination before overwriting `poolAdminFeeDestination[pool]`, mirroring the pattern already used in `setPoolAdminFees`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the old destination first
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed with `adminFeeDestination = A`. Admin spread fee is non-zero.
2. Swaps occur; spread surplus and notional fees accumulate in the pool.
3. Pool admin calls `setPoolAdminFeeDestination(pool, B)`. No collection occurs; `poolAdminFeeDestination[pool]` is now `B`.
4. Anyone calls `collectPoolFees(pool)`. `collectFees` is invoked with `adminFeeDestination_ = B`.
5. All accrued admin fees — including those earned while `A` was the destination — are transferred to `B`.
6. `A` receives nothing despite being the entitled destination during the accrual period.

The `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` accumulators are zeroed after this call, making the loss permanent. [4](#0-3) [5](#0-4)
