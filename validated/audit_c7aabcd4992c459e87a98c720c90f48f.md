### Title
Stale Pending Block Fallback Forces `l1_da_mode=Calldata`, Corrupting `use_kzg_da` in `BlockContext` and Producing Wrong Fee Estimates for Blob-DA Networks — (`crates/apollo_rpc/src/pending.rs`, `crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When the in-memory pending data is stale (its `parent_block_hash` does not match the latest committed block), `read_pending_data` constructs a `DeprecatedPendingBlock` fallback. The `l1_da_mode()` accessor for the `Deprecated` variant unconditionally returns `L1DataAvailabilityMode::Calldata`. This value flows through `client_pending_data_to_execution_pending_data` into `ExecutionPendingData.l1_da_mode`, then into `create_block_context` where it sets `use_kzg_da = false` in the `BlockContext`. On a Blob-DA network, this causes `simulateTransactions` and `estimateFee` (with `block_id=Pending`) to compute DA gas as L1 calldata gas instead of L1 data gas, returning a materially wrong fee estimate.

---

### Finding Description

**Step 1 — Stale detection and fallback construction.**

`read_pending_data` compares `pending_data.block.parent_block_hash()` against the latest committed block hash. On mismatch it builds:

```rust
PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock {
    parent_block_hash: latest_header.block_hash,
    eth_l1_gas_price: latest_header…l1_gas_price.price_in_wei,
    strk_l1_gas_price: latest_header…l1_gas_price.price_in_fri,
    timestamp: …,
    sequencer_address: …,
    starknet_version: …,
    ..Default::default()
})
``` [1](#0-0) 

`DeprecatedPendingBlock` has no `l1_da_mode` field; the struct does not carry it. [2](#0-1) 

**Step 2 — `l1_da_mode()` always returns `Calldata` for the `Deprecated` variant.**

```rust
pub fn l1_da_mode(&self) -> L1DataAvailabilityMode {
    match self {
        // In older versions, all blocks were using calldata.
        PendingBlockOrDeprecated::Deprecated(_) => L1DataAvailabilityMode::Calldata,
        PendingBlockOrDeprecated::Current(block) => block.l1_da_mode,
    }
}
``` [3](#0-2) 

**Step 3 — Conversion propagates the wrong mode.**

`client_pending_data_to_execution_pending_data` reads `l1_da_mode` directly from the block accessor:

```rust
l1_da_mode: client_pending_data.block.l1_da_mode(),
``` [4](#0-3) 

**Step 4 — `create_block_context` converts `l1_da_mode` to `use_kzg_da`.**

```rust
pending_data.l1_da_mode,   // ← always Calldata in stale path
…
let use_kzg_da = if override_kzg_da_to_false { false } else { l1_da_mode.is_use_kzg_da() };
```

`Calldata.is_use_kzg_da()` returns `false`, so `use_kzg_da = false` regardless of the actual network DA mode. [5](#0-4) 

`use_kzg_da` is then embedded in `BlockInfo` and used by blockifier to route DA gas: `true` → L1 data gas (blob pricing); `false` → L1 gas (calldata pricing). [6](#0-5) 

**Step 5 — All Pending-block RPC execution paths are affected.**

`simulate_transactions`, `estimate_fee`, and `estimate_message_fee` all call `read_pending_data` → `client_pending_data_to_execution_pending_data` → `exec_simulate_transactions` / `exec_estimate_fee` with the corrupted `maybe_pending_data`. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

On a Blob-DA network (mainnet today), blob gas (`l1_data_gas`) is priced differently from calldata gas (`l1_gas`). When `use_kzg_da` is forced to `false`, the blockifier computes DA gas as L1 calldata gas. For a V3 invoke transaction with non-trivial state changes, the DA component of the fee is computed against the wrong price vector, producing a fee estimate that diverges from the correct Blob-DA estimate. The returned `fee_estimation` is authoritative-looking and will be used by wallets and dApps to set resource bounds, causing either over-payment or under-funded transactions.

---

### Likelihood Explanation

The stale pending path is a normal operational condition. It is entered in the window between any new block being committed and the pending data being refreshed. On a live network this window is non-zero and recurs every block. Any unprivileged caller invoking `starknet_estimateFee` or `starknet_simulateTransactions` with `block_id=Pending` during that window receives the corrupted estimate. No special privileges, keys, or timing attacks are required.

---

### Recommendation

In the stale fallback branch of `read_pending_data`, copy `l1_da_mode` (and `l1_data_gas_price`) from the latest committed block header rather than relying on the `DeprecatedPendingBlock` defaults. The simplest fix is to use a `PendingBlock` (the `Current` variant) for the fallback, populated from the committed header fields, so that `l1_da_mode` is explicitly set to `header.l1_da_mode`. Alternatively, add an `l1_da_mode` field to `DeprecatedPendingBlock` and populate it from the header in the fallback path.

---

### Proof of Concept

1. Commit a block with `l1_da_mode = Blob` and non-zero `l1_data_gas_price`.
2. Do **not** update the in-memory `pending_data` (leave its `parent_block_hash` pointing to the previous block).
3. Call `starknet_simulateTransactions` with `block_id = Pending` and a V3 invoke transaction that writes storage (generating DA gas).
4. Observe that `read_pending_data` enters the stale branch and constructs a `DeprecatedPendingBlock`.
5. Assert that `l1_da_mode()` on the returned block is `Calldata` and `use_kzg_da = false` in the resulting `BlockContext`.
6. Compare the returned `fee_estimation` against a simulation using `block_id = BlockNumber(latest)` (which reads `l1_da_mode = Blob` from storage): the DA gas component will differ, confirming the wrong fee is returned in the stale pending path.
