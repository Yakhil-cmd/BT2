### Title
Missing Proof-of-Correctness Validation for CKD Participant Shares Allows Malicious Participant to Corrupt Derived Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` receives `(norm_big_y, norm_big_c)` share contributions from all participants and directly accumulates them without any cryptographic proof of correctness. Analogous to the Chainlink oracle trusting `price` without checking `timeStamp != 0` or `answeredInRound >= roundId`, the coordinator here trusts received elliptic-curve elements without verifying they were honestly computed. A single malicious participant can inject arbitrary group elements to corrupt the final CKD output.

### Finding Description
`compute_signature_share` (lines 148–182) defines the honest computation each participant must perform:

```
big_y  = y * G                          // randomness commitment
big_s  = H(pk || app_id) * private_share // secret contribution
big_c  = big_s + app_pk * y             // ElGamal-style masking
```

The coordinator's `do_ckd_coordinator` (lines 35–58) then collects every participant's `(norm_big_y, norm_big_c)` and blindly sums them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

There is no zero-knowledge proof, consistency check, or any other validation that the received `(big_y, big_c)` pair satisfies the required relationship `big_c = H(pk||app_id)*x_i + app_pk * y_i` for the participant's actual key share `x_i`. The coordinator has no mechanism to distinguish a correctly-formed share from an arbitrary pair of group elements.

Because the protocol requires **all** participants to contribute (not merely a threshold subset), a single dishonest participant can deviate freely. The `recv_from_others` call waits for every participant in `participants` before the coordinator proceeds, so the malicious share is always incorporated. [1](#0-0) [2](#0-1) 

### Impact Explanation
The final `CKDOutput { big_y, big_c }` is consumed by the caller via `unmask(app_sk)`, which computes `big_c − app_sk * big_y` to recover the confidential derived key. If any participant injects an arbitrary `(Δ_Y, Δ_C)` addend, the recovered key becomes `correct_key + (Δ_C − app_sk * Δ_Y)`, which is an attacker-controlled offset from the intended value. Honest parties will accept and use this corrupted output without any indication of tampering, satisfying the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable or attacker-biased cryptographic outputs**. [3](#0-2) 

### Likelihood Explanation
The attack requires only that one participant in the CKD session behaves maliciously — a realistic assumption in any multi-party deployment. No special cryptographic capability is needed; the attacker simply sends two arbitrary `G1` points instead of the honestly computed share. The protocol provides no detection mechanism, so the attack succeeds silently every time it is attempted.

### Recommendation
Each participant should accompany their `(norm_big_y, norm_big_c)` with a non-interactive zero-knowledge proof of correct formation — specifically a proof of knowledge of `(y_i, x_i)` such that:
- `big_y = y_i * G`
- `big_c = H(pk||app_id) * x_i + app_pk * y_i`
- `x_i` is consistent with the participant's public key share (verifiable from the DKG output)

The coordinator must verify all proofs before accumulating the shares, mirroring the Chainlink fix of adding `require(timeStamp != 0)` and `require(answeredInRound >= roundId)` before trusting the returned price.

### Proof of Concept
1. Honest participants run `compute_signature_share` and send correct `(norm_big_y, norm_big_c)` to the coordinator.
2. Malicious participant `P_m` instead sends `(Δ_Y, Δ_C)` — two arbitrary `G1` points chosen to bias the output.
3. `do_ckd_coordinator` calls `recv_from_others`, receives `P_m`'s contribution, and adds it unconditionally:
   ```rust
   norm_big_y += Δ_Y;   // no check
   norm_big_c += Δ_C;   // no check
   ```
4. The coordinator returns `CKDOutput::new(norm_big_y, norm_big_c)` containing the corrupted sum.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `correct_key + (Δ_C − app_sk * Δ_Y)` — an attacker-controlled value — believing it to be the legitimate confidential derived key. [4](#0-3)

    mut chan: SharedChannel,

        norm_big_c += participant_output.big_c();

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
