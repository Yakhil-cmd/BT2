### Title
Staker Can Manipulate Target Attestation Block by Changing Stake Mid-Epoch to Bypass Missed Window - (`src/attestation/attestation.cairo`)

---

### Summary

The target attestation block is computed via a Poseidon hash of `(stake, epoch_id, staker_address)`. The `stake` input is read live from the staker's balance at the **current epoch** at the time `attest` is called. Because `increase_stake` updates the balance trace at the current epoch immediately, a staker can call `increase_stake` with a chosen amount to change the hash output and thus their target attestation block. This allows a staker who has already missed their attestation window to find a new stake amount that places a new target block within the still-open window, and attest successfully to claim epoch rewards they should have forfeited.

---

### Finding Description

`_calculate_target_attestation_block` in `src/attestation/attestation.cairo` computes:

```
hash = Poseidon(stake, epoch_id, staker_address)
block_offset = hash % (epoch_len - attestation_window)
target_block = epoch_start + block_offset
``` [1](#0-0) 

The `stake` value is fetched in `get_attestation_info_by_operational_address` as the staker's total STRK balance **at the current epoch**: [2](#0-1) 

`increase_stake` writes a new checkpoint into the staker's own balance trace at the current epoch, so any subsequent read of `balance_at_epoch(current_epoch)` returns the updated value: [3](#0-2) 

The window validity check only tests the **current** target block (derived from the current stake), not the original one assigned at epoch start: [4](#0-3) 

**Attack path:**

1. Staker's original target block window closes (`target_block + attestation_window < current_block`).
2. Staker computes off-chain: for which `new_stake` does `Poseidon(new_stake, epoch_id, staker_address) % (epoch_len - attestation_window)` produce a `block_offset` satisfying `current_block - attestation_window ≤ epoch_start + block_offset ≤ current_block - MIN_ATTESTATION_WINDOW`?
3. Staker calls `increase_stake(staker_address, delta)` where `delta = new_stake - current_stake`.
4. Staker calls `attest(block_hash)` — the new target block is within the window, the check passes, and `update_rewards_from_attestation_contract` is triggered.

The search space is all `new_stake > current_stake`. Since Poseidon is deterministic and public, the staker iterates off-chain until a valid amount is found. The valid target-block range has width `attestation_window - MIN_ATTESTATION_WINDOW`; for any `attestation_window > 11` (the minimum), this range is non-empty and the attack succeeds with probability `(attestation_window - 11) / (epoch_len - attestation_window)` per attempt. [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker who missed their randomised attestation window should receive zero rewards for that epoch. By manipulating their stake amount, they can retroactively satisfy the window check and call `update_rewards_from_attestation_contract`, receiving a full epoch's worth of STRK rewards they are not entitled to. The rewards are minted from protocol inflation and credited directly to the staker's `unclaimed_rewards_own`. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The staker must hold additional STRK to call `increase_stake`, but the staked tokens are not lost — they remain staked and earn future rewards. The off-chain Poseidon search is trivial (public inputs, deterministic function). The attack is profitable whenever `attestation_window > MIN_ATTESTATION_WINDOW`, which is a valid and expected governance configuration. Any staker who misses their window has a direct financial incentive to attempt this.

---

### Recommendation

Freeze the `stake` value used in the attestation hash at the **start** of the epoch rather than reading it live. One approach: record the staker's balance at `epoch_start` in `get_attestation_info_by_operational_address` by reading `balance_at_epoch(epoch_id - 1)` (the balance confirmed at the previous epoch boundary), consistent with how consensus rewards already use a K-epoch-delayed balance to prevent manipulation. [7](#0-6) 

Alternatively, store the target attestation block in contract storage at the first block of each epoch (or on first `attest` call) so it cannot be recomputed after the fact.

---

### Proof of Concept

```
Epoch length = 100 blocks, attestation_window = 20, MIN_ATTESTATION_WINDOW = 11
Epoch starts at block 1000.

Staker's original stake S₀ gives:
  block_offset = Poseidon(S₀, epoch_id, addr) % 80 = 5
  target_block = 1005
  valid window = [1016, 1025]

Current block = 1030 → window already closed, staker missed it.

Staker searches off-chain for S₁ > S₀ such that:
  Poseidon(S₁, epoch_id, addr) % 80 ∈ [10, 19]   (i.e., target_block ∈ [1010, 1019])
  → window for that target_block includes block 1030.

Staker calls increase_stake(delta = S₁ - S₀).
Staker calls attest(block_hash_of_target_block).
→ _assert_attest_in_window passes, rewards are credited.
``` [8](#0-7)

        fn _calculate_target_attestation_block(
            self: @ContractState, staking_attestation_info: StakingAttestationInfo,
        ) -> BlockNumber {
            // Compute staker hash for the attestation.
            let hash = PoseidonTrait::new()
                .update(staking_attestation_info.stake().into())
                .update(staking_attestation_info.epoch_id().into())
                .update(staking_attestation_info.staker_address().into())
                .finalize();
            // Calculate staker's block number in this epoch.
            let attestation_window = self.attestation_window.read();
            let block_offset: u256 = hash
                .into() % (staking_attestation_info.epoch_len() - attestation_window.into())
                .into();
            // Calculate actual block number for attestation.
            let target_attestation_block = staking_attestation_info.current_epoch_starting_block()
                + block_offset.try_into().unwrap();
            target_attestation_block
        }

        fn _assert_attest_in_window(self: @ContractState, target_attestation_block: BlockNumber) {
            let attestation_window = self.attestation_window.read();
            let current_block_number = get_block_number();
            let min_block = target_attestation_block + MIN_ATTESTATION_WINDOW.into();
            let max_block = target_attestation_block + attestation_window.into();
            assert!(
                min_block <= current_block_number && current_block_number <= max_block,
                "{}",
                Error::ATTEST_OUT_OF_WINDOW,
            );
        }

        fn update_rewards_from_attestation_contract(
            ref self: ContractState, staker_address: ContractAddress,
        ) {
            // Prerequisites and asserts.
            self.general_prerequisites();
            assert!(self.is_pre_consensus(), "{}", Error::CONSENSUS_REWARDS_IS_ACTIVE);
            self.assert_caller_is_attestation_contract();
            let mut staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);

            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            // Get current epoch data.
            let (strk_epoch_rewards, btc_epoch_rewards) = reward_supplier_dispatcher
                .calculate_current_epoch_rewards();
            let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let curr_epoch = self.get_current_epoch();
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_epoch_rewards,
                    btc_total_rewards: btc_epoch_rewards,
                    :strk_total_stake,
                    :btc_total_stake,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
