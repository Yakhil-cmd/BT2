### Title
Unvalidated `app_pk` Identity Element Bypasses ElGamal Encryption in CKD, Exposing Confidential Derived Key to Coordinator — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary
The `ckd()` function accepts a caller-supplied `app_pk: PublicKey` (a BLS12-381 G1 element) with no check that it is a non-identity point. If a caller supplies the identity element as `app_pk`, the ElGamal encryption in `compute_signature_share` degenerates: the ciphertext collapses to the raw secret share, and the coordinator reconstructs the plaintext confidential derived key `msk · H(pk, app_id)` — material that must never be visible to any MPC node.

---

### Finding Description

The `ckd()` entry point performs four guard checks before launching the protocol: [1](#0-0) 

None of those checks touch `app_pk`. The value flows unchanged into `compute_signature_share`: [2](#0-1) 

Inside that function the per-participant ciphertext component is:

```
big_c = big_s + app_pk * y   // line 174
```

where `big_s = hash_point * private_share` is the participant's secret contribution.

When `app_pk = G1::identity()`:

```
app_pk * y  =  identity * y  =  identity
big_c       =  big_s + identity  =  big_s
```

The coordinator then aggregates:

```
sum(lambda_i * big_c_i)  =  sum(lambda_i * big_s_i)
                          =  msk · H(pk, app_id)
``` [3](#0-2) 

This is the raw confidential derived key — the exact value the ElGamal layer is designed to keep hidden from every MPC node, including the coordinator.

---

### Impact Explanation

The CKD design guarantee, stated in the documentation, is that the secret is never revealed to any entity other than the application itself, not even to individual MPC nodes. The coordinator is an MPC node. When `app_pk` is the identity, the coordinator's `CKDOutput.big_c` field holds `msk · H(pk, app_id)` in plaintext, directly violating that guarantee. This constitutes **disclosure of a confidential derived secret** — a Critical-tier impact under the allowed scope.

---

### Likelihood Explanation

The `app_pk` parameter is supplied by the calling application (the TEE app or the developer contract). A malicious or compromised application can trivially construct the identity element of G1 and pass it. No cryptographic capability, no key material, and no privileged access is required. The identity element is a valid, serializable BLS12-381 G1 point accepted by `blstrs` without error.

---

### Recommendation

Add an explicit non-identity check for `app_pk` inside `ckd()` before the protocol is launched, analogous to the existing participant-set guards:

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern already used for `msg_hash` in the robust ECDSA `sign()` function: [4](#0-3) 

---

### Proof of Concept

```rust
use threshold_signatures::confidential_key_derivation::{ckd, AppId};
use blstrs::G1Projective; // identity() is the additive identity

let app_pk_identity = G1Projective::identity(); // zero point

// ckd() accepts this without error
let protocol = ckd(
    &participants,
    coordinator,
    me,
    key_pair,
    AppId::try_from(b"victim-app").unwrap(),
    app_pk_identity,   // <-- identity element, no validation
    rng,
).unwrap();

// After running the protocol, the coordinator's CKDOutput.big_c
// equals  msk · H(pk, app_id)  in plaintext.
// The coordinator can read this value directly — no app_sk needed.
let ckd_output = run_and_collect_coordinator_output(protocol);
let raw_confidential_key = ckd_output.big_c(); // == msk · H(pk, app_id)
```

The coordinator, which is an MPC node that must never learn the confidential key, now holds it in plaintext because the ElGamal term `y · app_pk` vanished when `app_pk` was the identity.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
