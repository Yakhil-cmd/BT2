### Title
Dust Pool Balance Causes Permanent Freezing of Delegation Rewards - (`src/pool/pool.cairo`)

### Summary

When a pool's total delegated balance falls below `min_delegation_for_rewards` (1 STRK = `10^18` units), `compute_rewards_per_unit` silently returns zero, causing the epoch's pool rewards to be untracked. However, the staking contract still physically transfers the STRK reward tokens into the pool contract. Because no delegator's cumulative sigma increases, those tokens can never be claimed and are permanently frozen in the pool contract.

### Finding Description

`exit_delegation_pool_intent` in `pool.cairo` enforces only an upper-bound check (`amount <= total_amount`) with no minimum remaining pool balance guard: [1](#0-0) 

This allows any delegator to reduce the pool's active balance to an arbitrarily small value. When the staking contract later calls `update_rewards_from_staking_contract`, the pool contract's `compute_rewards_per_unit` checks: [2](#0-1) 

If `total_stake < min_delegation_for_rewards` (i.e., `< 10^18` for STRK), it returns `Zero::zero()`, so the cumulative sigma is not advanced. The code comment explicitly acknowledges this: [3](#0-2) 

Meanwhile, in `update_pool_rewards` in `staking.cairo`, the STRK tokens are **unconditionally** transferred to the pool contract before `update_rewards_from_staking_contract` is called: [4](#0-3) 

Because the cumulative sigma was not advanced, no delegator can ever claim those tokens via `claim_rewards`. There is no sweep or recovery function in the pool contract, so the tokens are permanently frozen.

The `STRK_CONFIG.min_for_rewards` constant is `10^18`: [5](#0-4) 

### Impact Explanation

Every epoch in which the pool balance is below `10^18` STRK units, the full pool reward allocation for that epoch is transferred into the pool contract but is irrecoverable. This constitutes **permanent freezing of unclaimed yield**. The frozen amount scales with the staker's share of total stake and the epoch reward rate; for a staker with meaningful self-stake but a sub-threshold pool balance, this can represent a significant and growing quantity of STRK.

### Likelihood Explanation

The exit wait window is measured in weeks. A delegator who calls `exit_delegation_pool_intent` with a partial amount that drops the pool balance below `10^18` units will cause every reward update during the entire wait window to freeze rewards. This can occur:

- **Naturally**: a pool that starts near the threshold and has one delegator partially exit.
- **Maliciously**: a delegator intentionally reduces the pool balance to just below `10^18` to grief other delegators or the staker, at negligible personal cost (they are exiting anyway).

`enter_delegation_pool` only requires `amount.is_non_zero()`: [6](#0-5) 

So an attacker can enter with 1 wei, then immediately exit, leaving the pool at a dust balance.

### Recommendation

1. Add a minimum remaining pool balance check in `exit_delegation_pool_intent`: after computing `new_delegated_stake`, assert that either `new_delegated_stake == 0` (full exit) or `new_delegated_stake >= min_delegation_for_rewards`.
2. Alternatively, in `update_rewards_from_staking_contract`, when `compute_rewards_per_unit` returns zero, do **not** transfer the reward tokens to the pool contract (or hold them in escrow to be credited when the balance recovers).
3. Add a minimum amount check in `enter_delegation_pool` analogous to the staking contract's `min_stake` check.

### Proof of Concept

1. Staker S opens a pool. Delegator D enters with `10^18 + 1` STRK (just above threshold).
2. D calls `exit_delegation_pool_intent(amount: 2)`, leaving pool balance at `10^18 - 1`.
3. The exit wait window begins (weeks).
4. Each epoch, the staking contract computes non-zero `pool_rewards` for S's pool (proportional to S's share of total stake), transfers those STRK to the pool contract via `send_rewards_to_delegation_pool`, then calls `update_rewards_from_staking_contract(rewards: pool_rewards, pool_balance: 10^18 - 1)`.
5. `compute_rewards_per_unit` returns 0 because `10^18 - 1 < 10^18`. The cumulative sigma is unchanged.
6. After the exit wait window, D exits. The pool balance is now 0. All STRK transferred during the wait window is permanently stuck in the pool contract with no mechanism to recover it.

        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
