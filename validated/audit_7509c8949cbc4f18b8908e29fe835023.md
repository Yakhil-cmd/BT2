Audit Report

## Title
Duplicate `SignedIngressId` Entries in `StrippedBlockProposal` Enable K-Fold Concurrent RPC Amplification — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary

A Byzantine subnet peer can craft a `StrippedBlockProposal` containing K duplicate `SignedIngressId` entries in the `ingress_messages` field. Because no deduplication occurs at deserialization, in `BlockProposalAssembler::new`, or in `missing_stripped_messages`, the `assemble_message` loop spawns K concurrent `get_or_fetch` tokio tasks for the same ingress message ID. If the message is absent from the local ingress pool, all K tasks issue simultaneous outbound RPCs to peers before the assembly is aborted. The attack is repeatable for every block proposal advertisement, `HASHES_IN_BLOCKS_ENABLED` is `true`, and no consensus integrity is compromised — the damage is purely resource exhaustion on the victim replica.

## Finding Description

**Step 1 — No deduplication at deserialization.**

`TryFrom<pb::StrippedBlockProposal>` in `stripped.rs` converts the protobuf `repeated StrippedIngressMessage ingress_messages` field directly into a `Vec<SignedIngressId>` with no uniqueness check: [1](#0-0) 

**Step 2 — No deduplication in `BlockProposalAssembler::new`.**

The constructor maps every element of `stripped_ingress_payload.ingress_messages` into `self.ingress_messages: Vec<(SignedIngressId, Option<SignedIngress>)>` without deduplication: [2](#0-1) 

**Step 3 — `missing_stripped_messages` returns K duplicate IDs.**

`missing_artifacts()` iterates over `self.ingress_messages` and returns every entry whose `Option` is `None`. With K duplicate entries, all K are `None` initially, so K duplicate `StrippedMessageId::Ingress` values are returned: [3](#0-2) 

**Step 4 — K concurrent tasks spawned.**

`assemble_message` iterates over the returned IDs and calls `join_set.spawn(get_or_fetch(...))` for each one, spawning K concurrent async tasks before any can complete: [4](#0-3) 

**Step 5 — K RPCs issued before abort.**

Each `get_or_fetch` task checks the local ingress pool first; if the message is absent, it enters `download_stripped_message` and issues an RPC to a random peer. All K tasks run concurrently, so up to K RPCs are in-flight simultaneously. Only after the *second* task completes does `try_insert` return `AlreadyInserted` (because `find` returns the first matching entry, which is already `Some` after the first task fills it), causing `assemble_message` to return `Unwanted` and drop `join_set`, aborting the remaining K-2 tasks: [5](#0-4) [6](#0-5) 

**Step 6 — The `ingress_messages` list is unauthenticated.**

The `StrippedBlockProposal` wire format contains the signed `pruned_block_proposal_proto` (authenticated by the block maker's signature) but the `ingress_messages` list is unsigned metadata added by the forwarding peer. Any Byzantine peer can take a legitimate stripped block and replace the `ingress_messages` list with K duplicates before forwarding it. The proto definition confirms `ingress_messages` is a plain `repeated` field with no authentication binding it to the signed block: [7](#0-6) 

`HASHES_IN_BLOCKS_ENABLED = true` confirms this code path is active on mainnet: [8](#0-7) 

**Amplification bound.**

`MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` applies to the *consensus-validated* ingress payload, not to the unauthenticated `ingress_messages` list in the stripped proposal. There is no count or size validation on this list at the P2P layer. Each `StrippedIngressMessage` proto entry is ~72 bytes (IngressMessageId + 32-byte hash). The QUIC transport enforces `MAX_MESSAGE_SIZE_BYTES` on incoming messages, but this is in the MB range, allowing tens of thousands of duplicate entries per crafted message: [9](#0-8) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact class: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

A single crafted `StrippedBlockProposal` causes K simultaneous outbound RPCs and K tokio tasks on the victim replica. The attack is repeatable for every block proposal advertisement (one per consensus round, ~1 second). Sustained amplification degrades the victim replica's CPU, memory, and network resources, potentially causing it to fall behind in consensus processing and reducing subnet availability. No invalid block is ever accepted; the impact is purely resource exhaustion.

## Likelihood Explanation

- `HASHES_IN_BLOCKS_ENABLED = true` means the code path is live on mainnet.
- The attacker only needs to be a Byzantine subnet peer below the fault threshold (f < n/3). A single Byzantine node suffices.
- No cryptographic material needs to be forged; the attacker reuses a legitimately signed `pruned_block_proposal_proto` and only modifies the unsigned `ingress_messages` list.
- The attack is repeatable for every block proposal advertisement with no cooldown.
- No victim mistakes or external dependencies are required.

## Recommendation

Deduplicate or bound `ingress_messages` at the earliest possible point:

1. **Preferred — In `TryFrom<pb::StrippedBlockProposal>`**: Reject the message if `ingress_messages.len()` exceeds `MAX_INGRESS_MESSAGES_PER_BLOCK`, and/or deduplicate by collecting into a `BTreeSet`/`HashSet` and rejecting on duplicate detection.
2. **Alternative — In `BlockProposalAssembler::new`**: Use a `BTreeSet` or `HashSet` instead of a `Vec` for `ingress_messages`, which naturally deduplicates.
3. **Fallback — In `missing_stripped_messages`**: Deduplicate the returned iterator before it is consumed by `assemble_message`.

Option 1 is preferred as it rejects malformed input at the trust boundary and prevents the malformed data from propagating further.

## Proof of Concept

```rust
// Construct a StrippedBlockProposal with K=1000 duplicate ingress entries
let ingress_id = fake_ingress_message("fake_1").id();
let duplicate_ids = vec![ingress_id.clone(); 1000];
let stripped = fake_stripped_block_proposal_with_messages(duplicate_ids);
let assembler = BlockProposalAssembler::new(stripped);

// missing_stripped_messages returns 1000 duplicate IDs — no deduplication
let missing = assembler.missing_stripped_messages();
assert_eq!(missing.len(), 1000); // all 1000 are returned

// assemble_message would spawn 1000 concurrent get_or_fetch tasks,
// issuing up to 1000 simultaneous RPCs for the same ingress message
// if it is absent from the local ingress pool.
```

The existing test infrastructure in `rs/p2p/artifact_downloader/src/fetch_stripped_artifact/test_utils.rs` (`fake_stripped_block_proposal_with_messages`) directly supports constructing this scenario as a unit test. A deterministic integration test can verify that `missing_stripped_messages()` returns K entries for K duplicates and that `assemble_message` spawns K tasks.

    fn try_insert(
        &mut self,
        signed_ingress_id: SignedIngressId,
        ingress_message: SignedIngress,
    ) -> Result<(), InsertionError> {
        // We can have at most 1000 elements in the vector, so it should be reasonably fast to do a
        // linear scan here.
        let (_, ingress) = self
            .ingress_messages
            .iter_mut()
            .find(|(id, _maybe_ingress)| *id == signed_ingress_id)
            .ok_or(InsertionError::NotNeeded)?;

        if ingress.is_some() {
            Err(InsertionError::AlreadyInserted)
        } else {
            *ingress = Some(ingress_message);
            Ok(())
