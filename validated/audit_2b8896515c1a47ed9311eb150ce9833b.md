Audit Report

## Title
Unbounded Synchronous Neuron Iteration in SNS Governance `compute_ballots_for_new_proposal` Enables Governance DoS — (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `compute_ballots_for_new_proposal` function iterates over every neuron in `self.proto.neurons` without any instruction-limit guard, and is called synchronously inside the `make_proposal` update call. As neuron count grows, the per-call instruction cost grows linearly. Beyond the IC's per-message instruction limit (~5 billion instructions on application subnets), every `make_proposal` call traps, permanently freezing proposal submission for the affected SNS. NNS governance has already addressed the identical pattern by moving ballot computation to a timer-based snapshot; SNS governance has not received this mitigation.

## Finding Description
`compute_ballots_for_new_proposal` at `rs/sns/governance/src/governance.rs:5255–5280` iterates the full `self.proto.neurons` `BTreeMap`, calling `v.dissolve_delay_seconds(now_seconds)` and `v.voting_power(...)` per neuron — both involving multiple arithmetic operations — with no early exit, no batching, and no instruction-limit check. This function is called synchronously from `make_proposal` at line 3557–3559, which is an update call reachable by any principal holding SNS tokens with `SubmitProposal` permission. A search for `is_message_over_threshold` and `noop_self_call_if_over_instructions` in `rs/sns/governance/src/governance.rs` returns zero matches, confirming no guard exists. The same function is also called from `compute_cached_metrics` (the timer path) at line 5388, which would also trap under the same conditions.

By contrast, NNS governance's `make_proposal` delegates to `compute_ballots_for_standard_proposal` (`rs/nns/governance/src/governance.rs:5486`), which reads from a pre-computed `VOTING_POWER_SNAPSHOTS` populated by a dedicated timer task (`SnapshotVotingPowerTask`). The NNS voting cascade separately uses `is_message_over_threshold` with a `HARD_VOTING_INSTRUCTIONS_LIMIT` of 750 billion instructions. Neither mechanism exists in SNS governance.

The claim incorrectly states SNS has no `MAX_NUMBER_OF_NEURONS` constant — the constant does exist in `rs/sns/governance/src/types.rs`. However, the structural vulnerability is independent of the cap value: there is no instruction-limit guard inside the loop, and the computation runs synchronously in the update call path. If the enforced cap is high enough (or if it is not enforced at neuron creation time for all code paths), the instruction limit can be reached.

## Impact Explanation
When the number of SNS neurons is large enough that a single pass through `self.proto.neurons` exceeds the IC's per-message instruction limit, every call to `make_proposal` traps. Because the loop runs before any state mutation, no proposal can ever be submitted again. The SNS governance canister becomes permanently unable to accept new proposals — a complete governance freeze for the affected SNS. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS with concrete user and protocol harm** to an in-scope SNS governance canister.

## Likelihood Explanation
Any principal who holds SNS tokens can stake a neuron and call `make_proposal`. The attacker does not need to cause neuron growth themselves; organic growth through token swaps, staking, and airdrops is sufficient. The NNS governance team already identified this exact pattern as a problem and fixed it; the SNS governance has not received the same fix. The attack is repeatable and permanent once the threshold is crossed — no recovery path exists without an upgrade.

## Recommendation
1. **Move ballot computation off the update call path**: Adopt the same snapshot mechanism used by NNS governance — compute the electoral roll in a periodic timer task and read from the snapshot inside `make_proposal`.
2. **Add an instruction-limit guard** inside the loop (analogous to `is_message_over_threshold` / `HARD_VOTING_INSTRUCTIONS_LIMIT` in `rs/nns/governance/src/voting.rs:24`) so that if the loop cannot complete in one message, it fails gracefully rather than trapping.
3. **Verify and enforce `MAX_NUMBER_OF_NEURONS`** at all neuron-creation code paths to ensure the cap is low enough that a full iteration cannot exceed the instruction limit.

## Proof of Concept
1. Deploy an SNS with a low minimum stake so neurons can be created cheaply.
2. Stake enough neurons (organically or via many small stakes) to push `self.proto.neurons.len()` past the instruction-limit threshold for the loop at `rs/sns/governance/src/governance.rs:5255–5280`.
3. Call `make_proposal` from any neuron with `SubmitProposal` permission.
4. Observe the update call trap due to instruction limit exhaustion.
5. Confirm every subsequent `make_proposal` call also traps — the SNS is governance-frozen.

A deterministic integration test can be written using PocketIC: populate the SNS neuron map with a large number of stub neurons (bypassing the cap if it is not enforced at the storage level), then call `make_proposal` and assert the canister traps rather than returning an error.
