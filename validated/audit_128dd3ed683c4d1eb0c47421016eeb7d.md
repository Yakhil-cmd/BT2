### Title
Malicious Participant Can Send Divergent `SigningCommitments` via `send_many` in `do_presign`, Permanently Breaking `sign_v2` for Honest Parties ŌĆö (`src/frost/mod.rs`, `src/frost/eddsa/sign.rs`)

---

### Summary

`do_presign` in `src/frost/mod.rs` uses `chan.send_many` (explicitly documented as having **no reliable-broadcast guarantee**) to distribute `SigningCommitments`. A malicious participant can deliver different commitments to different honest parties. Because `sign_v2` has each party independently construct a `SigningPackage` from its own local `commitments_map` ŌĆö with no reconciliation step ŌĆö the coordinator's `aggregate()` call receives signature shares computed against incompatible `SigningPackage`s and fails, permanently breaking the signing round.

---

### Finding Description

**Root cause ŌĆö `do_presign` (`src/frost/mod.rs` lines 104ŌĆō111):**

Each participant generates `(nonces, commitments)` and broadcasts via `send_many`:

```rust
chan.send_many(commit_waitpoint, &commitments)?;

for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
    commitments_map.insert(from.to_identifier()?, commitment);
}
```

`send_many` is explicitly documented in `docs/network-layer.md` as:

> "peer-to-peer sending with no security guarantees used by one sender in destination to multiple receiver."

There is no echo-broadcast, no cross-validation, and no consistency check. A malicious participant P_m can use `send_private` to deliver commitment `C1` to the coordinator and `C2` to other participants. Each honest party silently stores whatever it received.

**Divergence propagates into `sign_v2`:**

`do_sign_coordinator_v2` (`src/frost/eddsa/sign.rs` lines 189ŌĆō190) constructs its `SigningPackage` directly from its own `presignature.commitments_map`:

```rust
let signing_package =
    frost_ed25519::SigningPackage::new(presignature.commitments_map, message.as_slice());
```

`do_sign_participant_v2` (`src/frost/eddsa/sign.rs` line 339) does the same independently:

```rust
let signing_package = SigningPackage::new(presignature.commitments_map, message);
let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)?;
```

**Critically, unlike `sign_v1`**, the coordinator does NOT send the `SigningPackage` to participants. In `sign_v1` (line 137), the coordinator broadcasts the `SigningPackage` so all parties use the same one. In `sign_v2`, each party uses its own local view ŌĆö there is no reconciliation.

**Consequence at `aggregate()`:**

In FROST, `round2::sign` computes a binding factor `Žü_i` that depends on the full `commitments_map` in the `SigningPackage`. If the coordinator has `commitments_map[P_m] = C1` and participants have `commitments_map[P_m] = C2`, they compute different binding factors and produce incompatible signature shares. The coordinator's `aggregate()` call (line 219):

```rust
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
```

fails because the shares were computed against a different `SigningPackage` than the one passed to `aggregate()`.

---

### Impact Explanation

Honest parties complete `presign` and accept a `PresignOutput` that appears valid but contains divergent `commitments_map` entries. The subsequent `sign_v2` call permanently fails at `aggregate()`. The presign output is unusable and cannot be recovered ŌĆö a new presign round must be initiated. This matches the **High** impact category: *corruption of presign outputs so honest parties accept unusable cryptographic outputs*.

---

### Likelihood Explanation

The attack requires only a single malicious participant who controls their own protocol executor. The `Protocol` trait's `poke()` returns `Action::SendMany(data)`, and the external executor is responsible for delivery. A malicious executor can trivially deliver different payloads to different recipients instead of the same one. No cryptographic assumption is broken; no privileged access is required. The attack is fully within the documented threat model (malicious participants).

---

### Recommendation

Replace `send_many` with `echo_broadcast` (`do_broadcast`) for the commitment exchange round in `do_presign`. The `echo_broadcast` primitive already exists in `src/protocol/echo_broadcast.rs` and is used in the DKG protocol for exactly this purpose ŌĆö ensuring all honest parties agree on the same value from each sender even under Byzantine faults (up to `ŌīŖ(N-1)/3Ōīŗ` malicious parties).

### Proof of Concept

**Call sequence:**

1. P_m runs `presign()` but its executor delivers `C1` to the coordinator and `C2` to all other participants at `commit_waitpoint`.
2. All parties complete `do_presign` successfully ŌĆö no error is raised, since there is no consistency check.
3. Coordinator holds `commitments_map[P_m] = C1`; other participants hold `commitments_map[P_m] = C2`.
4. `sign_v2` is called. Coordinator constructs `SigningPackage_coord` with `C1`; participants construct `SigningPackage_i` with `C2`.
5. Participants compute `signature_share_i = round2::sign(SigningPackage_i, ...)` ŌĆö binding factor uses `C2`.
6. Coordinator calls `aggregate(SigningPackage_coord, {signature_share_i}, ...)` ŌĆö binding factor uses `C1`.
7. FROST's internal share verification inside `aggregate()` fails ŌåÆ `ProtocolError::AssertionFailed`.

**Minimal unit test sketch:**

```rust
// In presign mock executor: when P_m's SendMany fires,
// deliver commitments_C1 to coordinator, commitments_C2 to others.
// Assert that sign_v2 returns Err(ProtocolError::AssertionFailed).
```
