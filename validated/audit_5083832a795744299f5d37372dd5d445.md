### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo)

### Summary
The `change_reward_address()` function in `src/staking/staking.cairo` and `src/pool/pool.cairo` does not validate that the supplied `reward_address` is non-zero. A staker or pool member can set their reward address to `ContractAddress::zero()`, causing all subsequent reward claims to transfer tokens to the zero address, permanently freezing their unclaimed yield.

### Finding Description
`change_reward_address()` in `src/staking/staking.cairo` (line 517) only asserts that the new address is not a registered token address, but performs no zero-address check:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;   // written without zero check
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

The same omission exists in `pool.cairo`'s `change_reward_address()`, which has **no validation at all** beyond confirming the caller is a pool member:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    let pool_member = get_caller_address();
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    pool_member_info.reward_address = reward_address;   // no zero check
    self.write_pool_member_info(:pool_member, :pool_member_info);
    ...
}
``` [2](#0-1) 

The same gap is present at entry points: `stake()` in `staking.cairo` checks `reward_address` only against the token registry, not for zero: [3](#0-2) 

And `enter_delegation_pool()` in `pool.cairo` checks `reward_address` only against the token address, not for zero: [4](#0-3) 

The protocol's own specification lists `REWARD_ADDRESS_IS_TOKEN` as an error for `change_reward_address` in both contracts, implying the intent to guard against invalid reward addresses, yet the zero-address case is silently omitted. [5](#0-4) [6](#0-5) 

### Impact Explanation
When a staker's `reward_address` is `ContractAddress::zero()`, every call to `claim_rewards` transfers the accrued STRK to address(0). Depending on the ERC-20 implementation, this either:
- **Reverts** — permanently freezing the staker's unclaimed yield in the contract with no recovery path, or
- **Succeeds** — permanently burning the tokens.

Both outcomes constitute **permanent freezing of unclaimed yield**, which is a High-severity impact under the allowed scope.

### Likelihood Explanation
Any staker or pool member can trigger this by calling `change_reward_address(0)` (or by supplying zero at `stake()` / `enter_delegation_pool()` time). While deliberate self-harm is unlikely, the scenario is realistic for:
- Buggy front-ends or integration scripts that pass an uninitialized address variable.
- Automated staking bots that construct calldata incorrectly.

No privileged access is required; the call path is fully reachable by an unprivileged staker or delegator.

### Recommendation
Add an explicit non-zero assertion before writing the reward address in all four affected entry points:

```diff
// staking.cairo – change_reward_address
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
+   assert!(reward_address.is_non_zero(), "Invalid reward address");
    assert!(!self.does_token_exist(token_address: reward_address), ...);
    ...
}

// pool.cairo – change_reward_address
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    let pool_member = get_caller_address();
+   assert!(reward_address.is_non_zero(), "Invalid reward address");
    ...
}

// pool.cairo – enter_delegation_pool
fn enter_delegation_pool(ref self: ContractState, reward_address: ContractAddress, amount: Amount) {
    ...
+   assert!(reward_address.is_non_zero(), "Invalid reward address");
    assert!(token_address != reward_address, ...);
    ...
}

// staking.cairo – stake
fn stake(ref self: ContractState, reward_address: ContractAddress, ...) {
    ...
+   assert!(reward_address.is_non_zero(), "Invalid reward address");
    assert!(!self.does_token_exist(token_address: reward_address), ...);
    ...
}
```

### Proof of Concept
1. Staker calls `staking.change_reward_address(reward_address: ContractAddress::zero())`.
2. The function passes all existing checks (zero is not a registered token address) and writes `reward_address = 0` to storage.
3. Staker accumulates rewards over subsequent epochs.
4. Staker (or anyone) calls `staking.claim_rewards(staker_address)`.
5. The contract transfers the accrued STRK to `ContractAddress::zero()` — tokens are permanently lost or the call reverts, freezing the yield with no recovery mechanism.
