Audit Report

## Title
Case-Insensitive Bech32 Bypass of Minter-Address Guard in ckBTC Peg-Out — (`File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary
The `retrieve_btc` and `retrieve_btc_with_approval` endpoints guard against withdrawals to the minter's own Bitcoin address using a plain case-sensitive string comparison. Because the minter's canonical address string is always lowercase, but the bech32 parser accepts any case variant and decodes it to the same address bytes, a caller supplying the minter's address in uppercase bypasses the guard, has their ckBTC irreversibly burned, and causes the minter to send BTC to its own UTXO pool with no reimbursement path.

## Finding Description
**Root cause — case-sensitive guard vs. case-insensitive parser:**

In `retrieve_btc` (line 156–160) and `retrieve_btc_with_approval` (line 254–258), the guard is:

```rust
let main_address_str = state::read_state(|s| runtime.derive_minter_address_str(s));
if args.address == main_address_str {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

`derive_minter_address_str` calls `display()` → `encode_bech32()`, which always produces a lowercase bech32 string (e.g. `bc1q…`). [1](#0-0) 

The guard comparison is a plain `==` on `&str`, so it is case-sensitive. [2](#0-1) 

The bech32 parser (`parse_bip173_address`) normalises the HRP to lowercase before comparing:

```rust
if found_hrp.to_lowercase() != expected_hrp { … }
``` [3](#0-2) 

The bech32 data characters are also case-insensitive per BIP-173, so the `bech32` crate decodes `BC1Q…` and `bc1q…` to identical bytes. The existing unit test in `address.rs` proves this explicitly — both the lowercase and uppercase forms parse to the same `P2wpkhV0` byte array: [4](#0-3) 

**Exploit flow:**

1. Attacker obtains the minter's canonical address string (publicly available via `get_minter_info`).
2. Attacker converts it to uppercase: `BC1Q0JRXZ4JH59T5QSU7L0Y59KPFDMGJCQ60WLEE3H`.
3. Attacker calls `retrieve_btc` with the uppercase address.
4. Guard at line 158 does **not** trap — the strings differ in case.
5. `BitcoinAddress::parse` at line 173 succeeds and returns the same `P2wpkhV0` bytes as the minter's address. [5](#0-4) 
6. `burn_ckbtcs` is called — ckBTC is irreversibly burned on the IC ledger.
7. The `RetrieveBtcRequest` is queued with `address = parsed_address`, which is the minter's own address. [6](#0-5) 
8. The minter broadcasts a Bitcoin transaction to itself; the user receives no BTC.
9. `reimbursement_account` is only consulted on Bitcoin transaction failure — a successful send to the minter's own address is not a failure, so no reimbursement is triggered.

The identical guard and parser ordering exist in `retrieve_btc_with_approval`: [7](#0-6) 

## Impact Explanation
An unprivileged ckBTC holder permanently loses their entire withdrawal amount: ckBTC is irreversibly burned on the IC ledger and the corresponding BTC is deposited into the minter's own UTXO pool. There is no on-chain recovery path once the Bitcoin transaction is confirmed. This constitutes a concrete, permanent loss of in-scope chain-key/ck-token assets, matching the **High** impact category: *"Significant Chain Fusion, ck-token, ledger… security impact with concrete user or protocol harm."*

## Likelihood Explanation
The attack requires no special privileges — any ckBTC holder can trigger it. The minter's mainnet address is publicly known. Realistic triggering scenarios include: a QR code scanner or hardware wallet that uppercases bech32 output, or deliberate self-harm. The attack is trivially repeatable and requires no on-chain setup beyond holding ckBTC.

## Recommendation
Replace the string comparison with a comparison of the **parsed** `BitcoinAddress` structs, which are case- and format-agnostic. After parsing `args.address`, compare the resulting `BitcoinAddress` against the minter's derived address:

```rust
let parsed_address = BitcoinAddress::parse(&args.address, btc_network)?;
let minter_address = state::read_state(|s| runtime.derive_minter_address(s));
if parsed_address == minter_address {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

Apply the same fix to `retrieve_btc_with_approval`. This closes the bypass for all bech32 case variants and any future address encoding differences.

## Proof of Concept
1. Call `get_minter_info` to obtain the minter's canonical address, e.g. `bc1q0jrxz4jh59t5qsu7l0y59kpfdmgjcq60wlee3h`.
2. Convert to uppercase: `BC1Q0JRXZ4JH59T5QSU7L0Y59KPFDMGJCQ60WLEE3H`.
3. Call `retrieve_btc({ address = "BC1Q0JRXZ4JH59T5QSU7L0Y59KPFDMGJCQ60WLEE3H", amount = <balance> })`.
4. Observe: guard at line 158 does not trap; `BitcoinAddress::parse` returns the same `P2wpkhV0` bytes; ckBTC is burned; withdrawal request is accepted targeting the minter's own address.
5. Confirm via a unit test: assert that `BitcoinAddress::parse("BC1Q…", Network::Mainnet) == BitcoinAddress::parse("bc1q…", Network::Mainnet)` (already proven by the existing test at `address.rs` lines 471–480), and that the guard string comparison `"BC1Q…" == "bc1q…"` returns `false`. [8](#0-7)
