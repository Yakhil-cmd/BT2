### Title
Unanchored `native_price` / `pubdata_price` Oracle Values Enable Prover-Side Fee-Parameter Substitution — (`docs/bootloader/bootloader.md`, `zk_ee/src/system/metadata/zk_metadata.rs`, `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_price` and `pubdata_price` are consumed from the oracle as part of `BlockMetadataFromOracle` and drive every native-resource and pubdata-fee calculation in the bootloader. Neither value is committed to in the block header (public inputs) or in `ProofData`. A party controlling the proving-mode oracle input can therefore substitute arbitrary values for these two fields, causing the proven state transition to diverge from the one the sequencer executed — the direct ZKsync OS analog of the NFTPairWithOracle oracle-swap attack.

---

### Finding Description

The bootloader documentation explicitly acknowledges the gap:

> "The block header should determine the block fully, i.e. include all the inputs needed to execute the block. Currently it misses `gas_per_pubdata` and `native_price`, but we already working on design and implementation to solve this issue." [1](#0-0) 

`BlockMetadataFromOracle` carries both fields and serialises them for oracle consumption: [2](#0-1) [3](#0-2) 

`ProofData`, the only structure that is committed to as a public input, contains only `state_root_view` and `last_block_timestamp` — not `native_price` or `pubdata_price`: [4](#0-3) 

During ZK-transaction validation, `native_per_gas` is computed directly from the oracle-supplied `native_price`:

```
native_per_gas = ceil(gas_price / native_price)
native_limit   = gas_limit * native_per_gas
``` [5](#0-4) 

Because `native_price` is not pinned in the public inputs, the prover can feed a value that differs from the one the sequencer used. The same applies to `pubdata_price`, which governs the native cost of every byte of pubdata written: [6](#0-5) 

---

### Impact Explanation

**Forward / proving divergence leading to invalid state commitment.**

| Manipulation | Effect |
|---|---|
| Prover sets `native_price` ≫ sequencer value | `native_per_gas` shrinks → transactions that exhausted native resources in forward execution now succeed in proving → a different (larger) set of state writes is proven |
| Prover sets `native_price` ≪ sequencer value | `native_per_gas` grows → transactions that succeeded in forward execution now run out of native resources in proving → valid state writes are omitted from the proven state |
| Prover sets `pubdata_price` = 0 | Pubdata cost is zero → transactions rejected by the sequencer for exceeding pubdata limits pass in proving |

In all cases the L1 verifier accepts the proof (because `native_price` is absent from the public inputs), but the proven state root corresponds to a block the sequencer never executed. This is a critical state-transition integrity violation.

---

### Likelihood Explanation

The entry path is the proving-mode oracle (CSR-based `CsrBasedIOOracle`), which is controlled by whoever runs the RISC-V prover. The Immunefi scope explicitly lists "prover/forward execution input" as an attacker-controlled entry path. The manipulation requires no special key or governance action beyond controlling the oracle data fed to the RISC-V machine — the same level of access needed to run the prover at all. The gap is self-documented in the codebase, confirming it is a real, reachable condition rather than a theoretical one. [7](#0-6) 

---

### Recommendation

Include `native_price` and `pubdata_price` in the public inputs that are hashed into the batch commitment (alongside `eip1559_basefee`, `block_number`, `timestamp`, etc.). The `BlockMetadataFromOracle` serialisation already covers these fields; they simply need to be part of the committed public-input hash verified on L1. Until that is done, the prover can silently substitute any pricing values it chooses. [8](#0-7) 

---

### Proof of Concept

1. Sequencer runs forward execution with `native_price = 10`, `pubdata_price = 100`. Transaction T succeeds and writes storage slot S → state root R₁.
2. Malicious prover runs proving execution with `native_price = 1_000_000` (oracle-injected). `native_per_gas` collapses to near zero; T now exhausts native resources and is reverted. State root R₂ ≠ R₁.
3. The proof is generated for R₂. Because neither `native_price` nor `pubdata_price` appears in the public inputs committed on L1, the verifier accepts the proof.
4. The L2 state diverges from what users observed during sequencer execution — storage slot S is not written, and any funds or contract state depending on T are lost or corrupted. [9](#0-8) [10](#0-9)

impl<NDS: NonDeterminismCSRSourceImplementation> IOOracle for CsrBasedIOOracle<NDS> {
    type RawIterator<'a> = CsrBasedIOOracleIterator<NDS>;

    fn raw_query<'a, I: UsizeSerializable + UsizeDeserializable>(
        &'a mut self,
        query_type: u32,
        input: &I,
    ) -> Result<Self::RawIterator<'a>, InternalError> {
        const {
            assert!(core::mem::size_of::<usize>() == core::mem::size_of::<u32>());
        }
        NDS::csr_write_impl(query_type as usize);
        let iter_to_write = UsizeSerializable::iter(input);
        // write length
        let iterator_len = iter_to_write.len();
        assert!(iterator_len == <I as UsizeSerializable>::USIZE_LEN);
        NDS::csr_write_impl(iterator_len);
        // write content
        let mut remaining_len = iterator_len;
        for value in iter_to_write {
            assert!(remaining_len != 0);
            NDS::csr_write_impl(value);
            remaining_len -= 1;
        }
        assert!(remaining_len == 0);
        // we can expect that length of the result is returned via read
        let remaining_len = NDS::csr_read_impl();
        let it = CsrBasedIOOracleIterator::<NDS> {
            remaining: remaining_len,
            _marker: core::marker::PhantomData,
        };

        Ok(it)
    }
