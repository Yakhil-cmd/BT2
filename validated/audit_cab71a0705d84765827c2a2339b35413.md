### Title
Signature Verification Bypassed for Invoke Transactions via `skip_stateful_validations` — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary
The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point call (the only cryptographic signature check) for any invoke transaction with `tx_nonce == 1` targeting an account whose on-chain nonce is `0` and that has any entry in the mempool. An attacker who observes a victim's `deploy_account` transaction in the mempool can submit a malicious invoke transaction with nonce=1 for the victim's address, which is admitted to the mempool without signature verification.

### Finding Description

In `skip_stateful_validations` the gateway checks three conditions before returning `true` (skip validation):

```
tx_nonce == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
account_tx_in_pool_or_recent_block(sender_address) == true
``` [1](#0-0) 

When the function returns `true`, `run_validate_entry_point` sets `validate: false` in the execution flags: [2](#0-1) 

With `validate: false`, the blockifier's `StatefulValidator::perform_validations` skips the `__validate__` entry-point call entirely: [3](#0-2) 

`__validate__` is the **only** place where the account's signature is cryptographically verified during gateway admission. The stateless validator only checks signature *length*, not validity: [4](#0-3) 

The code comment asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This assumption is incorrect. The presence of a `deploy_account` transaction for address A in the mempool says nothing about whether a *different* invoke transaction for address A carries a valid signature.

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in the pool or in the committed-nonce state: [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce gaps and duplicates — it carries no signature field: [6](#0-5) 

### Impact Explanation

**Impact: High** — Mempool/gateway admission accepts an invalid (unauthorized) transaction before sequencing.

When the malicious invoke transaction is later executed by the batcher, `__validate__` *is* called and the transaction reverts. However:
- The victim's nonce is incremented (Starknet increments nonces even for reverted transactions), potentially invalidating any pending nonce=1 transaction the victim intended to submit.
- The victim's account is charged fees for the reverted transaction (fees are collected even on revert).
- The mempool can be polluted with invalid transactions for any undeployed account that has a `deploy_account` tx in flight.

### Likelihood Explanation

**Likelihood: High** — The attack requires no special privileges. The mempool is observable (P2P propagation), account addresses are deterministic, and any account being deployed must be pre-funded (satisfying the gateway's balance check). The attacker simply needs to race a nonce=1 invoke submission against the victim's `deploy_account` transaction.

### Recommendation

Do not skip signature verification entirely for the deploy-account + invoke UX flow. Instead, retrieve the account's class hash from the pending `deploy_account` transaction and use it to verify the invoke transaction's signature before admission, even though the account is not yet on-chain. Alternatively, restrict the skip to cases where the *same sender* submitted both the `deploy_account` and the invoke (verified by matching the transaction hash stored in the mempool), or require the invoke to carry a proof-of-knowledge of the account's private key.

### Proof of Concept

1. Alice submits `deploy_account` (nonce=0) for address A, pre-funded with STRK.
2. Bob observes Alice's `deploy_account` tx in the mempool.
3. Bob submits `invoke` (nonce=1, sender=A, calldata=drain_funds, signature=garbage).
4. Gateway: `validate_nonce` passes (1 ≥ 0, within `max_allowed_nonce_gap=200`). [7](#0-6) 
5. Gateway: `validate_by_mempool` passes (no duplicate nonce=1 for A yet).
6. Gateway: `skip_stateful_validations` → `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` → returns `true`. [8](#0-7) 
7. Gateway: `run_validate_entry_point` called with `validate=false`; `__validate__` is **not** called; Bob's tx is admitted to the mempool. [9](#0-8) 
8. Batcher includes both transactions in a block. Alice's `deploy_account` executes (nonce 0→1). Bob's invoke executes: `__validate__` is called, rejects the invalid signature → transaction reverts.
9. Alice's account nonce is now 2; she paid fees for Bob's unauthorized transaction; her own intended nonce=1 invoke (if any) is now stale.
