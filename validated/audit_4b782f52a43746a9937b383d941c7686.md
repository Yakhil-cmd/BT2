### Title
Missing `evmAddress` Binding in `validateCOAOwnershipProof` Enables Cross-COA Ownership Proof Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

The `access(all)` function `validateCOAOwnershipProof` in the EVM system contract verifies that a set of signatures is valid over `signedData` and that the COA at the given `path` matches `evmAddress`, but it does **not** enforce that `signedData` encodes `evmAddress`. This is the direct Flow analog of the keychain weak-access-control pattern: a default of "no domain binding" is used where a secure binding is required. Any unprivileged Cadence script or transaction can call this `access(all)` function with attacker-chosen arguments, making the entry path fully externally reachable. A signature produced for one COA can be replayed to prove ownership of a different COA owned by the same Cadence account, bypassing the per-COA authentication boundary.

### Finding Description

`validateCOAOwnershipProof` is declared `access(all)` and callable by any Cadence script or transaction without restriction. [1](#0-0) 

The function performs two checks:

1. **Signature validity** (lines 1082–1086): verifies the supplied signatures over `signedData` using the account's keys with domain tag `"FLOW-V0.0-user"`.
2. **COA address match** (lines 1095–1109): borrows the `CadenceOwnedAccount` capability from the account at the caller-supplied `path` and compares its EVM address byte-by-byte against the caller-supplied `evmAddress`. [2](#0-1) 

What is **absent** is any check that `signedData` encodes `evmAddress`. The code itself documents this gap: [3](#0-2) 

Because `signedData` is never bound to a specific EVM address, the two checks are **independent**: the signature check passes for any COA owned by the same Cadence account (all share the same account keys), and the address check passes for whichever COA the attacker supplies in `path`/`evmAddress`. An attacker who observes a valid proof for COA_A can replay it against COA_B by substituting COA_B's `path` and `evmAddress` while keeping `signedData`, `keyIndices`, and `signatures` unchanged.

The `COAOwnershipProof` struct (encoded in the `bytes` argument) carries `CapabilityPath` inside it, so the attacker must also supply a proof whose `CapabilityPath` matches the target COA's published path. Because COA capabilities are published at public paths (e.g. `/public/coa`, `/public/coa2`), these paths are discoverable on-chain by any observer. [4](#0-3) 

The EVM-side precompile path (`isValidSignature` in the COA Solidity contract) is **not** vulnerable because it always passes `address(this)` as the EVM address argument, so the address is fixed to the calling contract: [5](#0-4) 

However, `validateCOAOwnershipProof` is also callable directly from Cadence scripts (as shown in the test harness), and any off-chain or on-chain authentication flow that calls it directly is exposed. [6](#0-5) 

### Impact Explanation

An attacker who obtains a legitimately produced COA ownership proof for COA_A (e.g., by observing it in a public transaction, an on-chain event, or an API response) can replay it to obtain a `ValidationResult { isValid: true }` for COA_B, a different COA owned by the same Cadence account. Any Cadence contract or off-chain system that uses `validateCOAOwnershipProof` as an authentication gate for COA_B — for example, to authorize a bridge withdrawal, a privileged contract call, or an ERC-1271-based DeFi action — would incorrectly grant access. The impact class is **entitlement/capability bypass**: the attacker bypasses the per-COA authentication boundary without holding the COA resource or the account's private key.

### Likelihood Explanation

Likelihood is **low-to-medium**. The precondition is that the victim holds two or more COAs under the same Cadence account and that a valid proof for one of them is observable (e.g., emitted in a transaction or returned by an API). While Flow wallets historically create at most one COA per account, the protocol imposes no such limit, and multi-COA patterns are expected to grow as EVM-on-Flow adoption increases. No privileged access, leaked keys, or social engineering is required; the attacker only needs to observe a proof that the victim has already published.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` before accepting the proof as valid. Concretely, add a pre-condition or an explicit check that the 20-byte EVM address appears at a well-known offset within `signedData`, or require callers to pass a structured payload (e.g., `hash(evmAddress || challenge)`) so that the signature is cryptographically bound to the specific COA being authenticated. The EVM precompile path already passes the calling COA's address as the `evmAddress` argument; the same binding must be enforced at the Cadence layer for direct callers.

### Proof of Concept

Assume a Cadence account `0xALICE` owns two COAs:
- COA_A stored at `/storage/coaA`, published at `/public/coaA` with EVM address `0xAAAA…`
- COA_B stored at `/storage/coaB`, published at `/public/coaB` with EVM address `0xBBBB…`

**Step 1 – Victim produces a proof for COA_A** (e.g., for a login challenge `D`):
```
proof_A = COAOwnershipProof {
    Address:        0xALICE,
    CapabilityPath: "coaA",
    KeyIndices:     [0],
    Signatures:     [sign(D, aliceKey0)],
}
signedData = D
evmAddress = 0xAAAA…
```

**Step 2 – Attacker replays the proof against COA_B** by calling `validateCOAOwnershipProof` directly from a Cadence script:
```cadence
import EVM from 0x…

access(all)
fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address:    0xALICE,
        path:       /public/coaB,          // COA_B's path
        signedData: D,                     // same signedData
        keyIndices: [0],                   // same key index
        signatures: [sign(D, aliceKey0)],  // same signature
        evmAddress: 0xBBBB…               // COA_B's EVM address
    )
}
```

**Result**: The function returns `ValidationResult { isValid: true, problem: nil }` because:
- `sign(D, aliceKey0)` is a valid signature over `D` using Alice's key (check at line 1082 passes).
- The COA at `/public/coaB` has EVM address `0xBBBB…` (check at lines 1095–1109 passes).
- No check verifies that `D` encodes `0xBBBB…`.

The attacker has successfully proven ownership of COA_B using a signature that was produced for COA_A. [7](#0-6) [8](#0-7)
