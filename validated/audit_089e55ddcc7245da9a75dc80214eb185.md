### Title
WalletConnect `take_offer` Confirmation Displays Understated Royalty Total for Multi-NFT Offers - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly splits the fungible payment amount equally across all NFTs before applying each NFT's royalty percentage. The correct calculation applies each royalty percentage to the **full** payment amount. When a dApp sends a `chia_wallet.take_offer` WalletConnect command for an offer containing multiple NFTs, the `amountWithRoyalties` shown in the approval dialog is understated, causing the user to approve a transaction whose true cost is higher than displayed.

### Finding Description

In `formatAmountWithRoyalties`:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← wrong
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The function divides `amount` by the number of NFTs (`royaltyPercentages.length`) and applies each NFT's royalty to that fraction. The correct formula is to apply each royalty percentage to the full `amount`:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

**Concrete example** (confirmed by the existing test at line 251–335):

- Payment: 100,000,000 mojos (0.0001 XCH)
- NFT1 royalty: 500 bps (5%), NFT2 royalty: 10 bps (0.1%)

| | Buggy code | Correct |
|---|---|---|
| splitAmount | 50,000,000 | — |
| royalty1 | 2,500,000 | 5,000,000 |
| royalty2 | 50,000 | 100,000 |
| **Total shown** | **102,550,000 (0.00010255 XCH)** | **105,100,000 (0.0001051 XCH)** |

The test at line 320 asserts `amountWithRoyalties: '0.00010255'`, which is the buggy understated value. [2](#0-1) 

This `amountWithRoyalties` field is the value displayed in the WalletConnect confirmation dialog to the user before they approve the offer acceptance. [3](#0-2) 

### Impact Explanation

A dApp connected via WalletConnect sends `chia_wallet.take_offer` with an offer containing multiple NFTs that each carry royalties. The GUI's approval dialog shows the user a total cost that is lower than the actual on-chain cost. The user approves believing they will spend X XCH; the actual spend is X + (understated royalty delta). The discrepancy grows with the number of NFTs and the magnitude of their royalty percentages. This constitutes spoofing of WalletConnect state that causes a user to approve the wrong amount.

### Likelihood Explanation

Any dApp paired via WalletConnect can trigger this path by constructing a valid multi-NFT offer. No special privileges, leaked keys, or social engineering are required beyond the standard WalletConnect pairing that the user has already approved. Multi-NFT offers are a supported and documented feature of the Chia offer protocol.

### Recommendation

Replace the split-amount logic with the correct per-NFT full-amount royalty sum:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [4](#0-3) 

### Proof of Concept

The existing test at lines 251–335 of `parseCommandDisplay.test.ts` already encodes the buggy expected value (`0.00010255`). Changing the formula to apply each royalty to the full amount would produce `0.0001051`, demonstrating the understatement. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
