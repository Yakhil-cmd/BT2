### Title
`COAOwnershipProof` Structure Lacks Deadline, Enabling Indefinite Signature Replay - (File: `fvm/evm/types/proof.go`)

---

### Summary

The `COAOwnershipProof` structure used by the `verifyCOAOwnershipProof` Cadence Arch precompile contains no deadline, expiration block, or nonce field. Any signature a COA owner produces over arbitrary `signedData` is valid indefinitely. The only revocation path is revoking the underlying Cadence account key — a drastic, account-wide action. This is a direct structural analog to the ParaSpace missing-deadline bug: a user who signs a COA ownership proof for one purpose (e.g., an EVM marketplace order) is permanently committed to that proof, and an adversary can replay it at any future time.

---

### Finding Description

`COAOwnershipProof` is defined in `fvm/evm/types/proof.go`:

```go
type COAOwnershipProof struct {
    KeyIndices     KeyIndices
    Address        FlowAddress
    CapabilityPath PublicPath
    Signatures     Signatures
}
``` [1](#0-0) 

There is no `Deadline`, `ExpiryBlock`, or `Nonce` field. The extended context type `COAOwnershipProofInContext` adds only `SignedData` and `EVMAddress` — still no time-bound:

```go
type COAOwnershipProofInContext struct {
    COAOwnershipProof
    SignedData SignedData
    EVMAddress Address
}
``` [2](#0-1) 

The Cadence-side verifier `validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` checks only: (1) signature cryptographic validity, (2) key revocation status, and (3) COA address match. It performs **no time-bound or nonce check**: [3](#0-2) 

The contract's own NatSpec comment acknowledges a related replay concern but explicitly dismisses it as "low-risk" and does not address the missing deadline: [4](#0-3) 

The precompile entry point `proofVerifier.Run` in `fvm/evm/precompiles/arch.go` decodes the proof and calls the validator with no temporal validation: [5](#0-4) 

The `coaOwnershipProofValidator` in `fvm/evm/handler/precompiles.go` forwards the proof directly to `validateCOAOwnershipProof` with no deadline enforcement: [6](#0-5) 

The COA contract's ERC-1271 `isValidSignature` implementation in `fvm/evm/handler/coa/coa.sol` calls `verifyCOAOwnershipProof` with the caller's address and the supplied `_hash`, meaning any EVM contract can trigger proof verification against a stored proof: [7](#0-6) 

---

### Impact Explanation

A COA owner who signs a `COAOwnershipProof` over a specific `signedData` hash (e.g., an EVM marketplace order hash, a loan authorization hash, or any off-chain commitment) is permanently bound to that proof. Because the proof carries no expiry and the only revocation path is revoking the Cadence account key (which invalidates all account operations, not just this proof), the user has no targeted way to cancel a previously issued proof.

An adversary holding a captured proof can:
1. Wait for market conditions to become favorable (e.g., NFT price drops, collateral value falls).
2. Submit the old proof to any EVM contract that calls `verifyCOAOwnershipProof` or `isValidSignature` with the matching hash.
3. Force the COA owner into the previously authorized action (sale, loan, transfer) at the original terms, against their current intent.

This matches the ParaSpace impact class: **unauthorized asset movement / permanent commitment to a signed obligation with no revocation mechanism**.

---

### Likelihood Explanation

The `verifyCOAOwnershipProof` precompile is a publicly callable EVM precompile reachable by any EVM contract or EVM transaction sender without any privilege. The ERC-1271 `isValidSignature` path in `coa.sol` is the standard integration point for EVM marketplaces and DeFi protocols. Any user who has ever signed a COA ownership proof for an EVM application (marketplace listing, loan authorization, DAO vote) and whose proof was captured by a counterparty is exposed. The attacker needs only the previously broadcast proof bytes and the matching hash — both are observable on-chain or off-chain from the original interaction.

---

### Recommendation

1. Add a `Deadline` (block height or timestamp) field to `COAOwnershipProof` in `fvm/evm/types/proof.go`.
2. Enforce the deadline inside `validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` by comparing against the current Flow block height or timestamp before accepting the proof.
3. Optionally add a per-address nonce to prevent replay of proofs with the same deadline across different contexts.

---

### Proof of Concept

**Setup:**
- Alice owns a COA and lists an NFT for sale on an EVM marketplace at price P. She signs a `COAOwnershipProof` over the order hash H and submits it to the marketplace.

**Attack:**
1. The NFT price rises. Alice wants to cancel the listing.
2. Alice has no way to invalidate the proof — `validateCOAOwnershipProof` has no deadline check and no nonce. Revoking her Cadence account key would break all her other transactions.
3. Bob (attacker) holds the original encoded proof bytes (observable from the original marketplace transaction).
4. Bob calls the marketplace contract's fill-order function, supplying the old proof. The marketplace calls `isValidSignature(_hash=H, _sig=encodedProof)` on Alice's COA.
5. `isValidSignature` calls the `verifyCOAOwnershipProof` precompile → `validateCOAOwnershipProof` → verifies Alice's key is still active and the signature over H is valid → returns `true`.
6. The marketplace executes the sale at price P, transferring Alice's NFT to Bob at the stale price, against Alice's current intent.
