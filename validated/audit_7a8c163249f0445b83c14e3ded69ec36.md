### Title
Pool admin bypasses `maxAdminSpreadFeeE6` cap via uncapped `setPoolBinAdditionalFees`, enabling instant front-run fee extraction from traders — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` applies **no cap validation** on `addFeeBuyE6`/`addFeeSellE6`, allowing the pool admin to set per-bin additional fees up to `type(uint16).max` = 65 535 (≈ 6.55 % in E6 terms) with **no timelock**. This directly bypasses the `maxAdminSpreadFeeE6` cap that `setPoolAdminFees` enforces, and lets the pool admin front-run any pending swap by atomically spiking the bin fee in the same block.

---

### Finding Description

**Cap enforcement is asymmetric.** `setPoolAdminFees` guards the global admin spread component:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` passes the caller-supplied values straight through with **zero cap check**:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — only a bin-index range check:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

**These per-bin fees are applied directly inside every swap iteration** on top of the oracle-derived base fee:

```solidity
// MetricOmmPool.sol L999 (buy-token0 exact-in path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears in all four swap directions (exact-in/out × token0/token1): [5](#0-4) [6](#0-5) [7](#0-6) 

**`BinState.addFeeBuyE6` / `addFeeSellE6` are `uint16`**, so the unchecked ceiling is 65 535 ≈ 6.55 % per bin, per direction: [8](#0-7) 

The factory hard-caps `maxAdminSpreadFeeE6` at 200 000 (20 %), but the factory owner can lower it to any value — including below 65 535. Once lowered, `setPoolAdminFees` is capped at the new lower value, yet `setPoolBinAdditionalFees` remains uncapped at `type(uint16).max`. The pool admin therefore retains the ability to impose a 6.55 % per-bin fee regardless of what `maxAdminSpreadFeeE6` is set to. [9](#0-8) 

There is also **no timelock** on `setPoolBinAdditionalFees`, unlike the price-provider rotation path which enforces `priceProviderTimelock`. The fee change takes effect in the same transaction, enabling a mempool front-run. [10](#0-9) 

---

### Impact Explanation

A malicious pool admin can:

1. Observe a pending swap in the mempool.
2. Front-run it with `setPoolBinAdditionalFees(pool, curBin, 65535, 65535)` — no timelock, no cap.
3. The victim's swap executes against the spiked fee: the effective buy/sell fee for that bin becomes `baseFeeX64 + 65535/1e6` instead of the expected `baseFeeX64`.
4. For an exact-input swap the trader receives fewer output tokens; for an exact-output swap the trader pays more input tokens.
5. The excess fee accrues as `binLpFeeAmount` (stays in the bin, benefiting LPs) and a proportional slice goes to protocol/admin via `protocolFeeAmountScaled`.

The router's `amountOutMinimum` / `amountInMaximum` slippage guards can catch this only if the user set a tight bound. Many integrations pass `amountOutMinimum = 0` or a loose maximum, leaving the trader fully exposed. [11](#0-10) [12](#0-11) 

---

### Likelihood Explanation

The pool admin is explicitly classified as semi-trusted "only inside caps and timelocks." The absence of any cap on `setPoolBinAdditionalFees` means the admin can exceed the intended boundary without any privileged escalation. The attack requires only the pool admin key and knowledge of a pending swap — both realistic conditions for a malicious or compromised admin.

---

### Recommendation

1. **Add a cap check** in `setPoolBinAdditionalFees` (factory side) against `maxAdminSpreadFeeE6` (or introduce a dedicated `maxBinAdditionalFeeE6` storage variable):

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

2. **Add a timelock** (or at minimum a minimum-notice event) for bin-fee changes, consistent with the price-provider rotation pattern, so traders can observe and react before the change takes effect.

---

### Proof of Concept

```
Setup:
  factory.setFeeCaps(maxProtocolSpread=5_000, maxAdminSpread=1_000, ...);
  // maxAdminSpreadFeeE6 = 1_000 (0.1%)

  pool admin calls setPoolAdminFees(pool, 1_000, 0);
  // global admin spread = 0.1% — within cap

Attack:
  1. Trader broadcasts exactInputSingle(amountIn=1_000_000, amountOutMinimum=0, ...)
  2. Pool admin sees the pending tx, front-runs with:
       setPoolBinAdditionalFees(pool, curBin, 65_535, 65_535)
     // addFeeBuyE6 = 65_535 ≈ 6.55% — NO revert, no cap check
  3. Trader's swap executes:
       effectiveBuyFeeX64 = baseFeeX64 + mulDiv(65_535, ONE_X64, 1e6)
     // trader pays ~6.55% extra on top of oracle spread
  4. Trader receives ~6.55% fewer output tokens than quoted.
  5. Pool admin resets: setPoolBinAdditionalFees(pool, curBin, 0, 0)
```

The `maxAdminSpreadFeeE6 = 1_000` cap is completely bypassed; `setPoolBinAdditionalFees` accepts `65_535` without reverting.
