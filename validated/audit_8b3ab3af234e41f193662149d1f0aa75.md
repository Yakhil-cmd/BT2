### Title
Malformed Message Causes Permanent Protocol Termination via Unhandled Deserialization Error in `recv_from_others` — (File: `src/protocol/helpers.rs`)

---

### Summary

The `recv_from_others` helper function propagates deserialization errors directly to the caller with no retry or skip logic. A single malicious participant can inject a message with a syntactically valid routing header but garbage payload bytes, causing every honest party that calls `recv_from_others` to receive a fatal `ProtocolError::DeserializationError`, permanently aborting DKG, reshare, refresh, or signing.

---

### Finding Description

**Root cause — `recv_from_others` propagates deserialization errors:** [1](#0-0) 

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // ← ? propagates error
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

The `?` operator on `chan.recv(...)` means that any `Err` returned by the inner deserialization path immediately unwinds the entire protocol future. There is no `continue`, no skip, and no way to resume.

**Why deserialization can fail — `Comms::recv` pops-then-decodes:** [2](#0-1) 

```rust
async fn recv<T: DeserializeOwned>(...) -> Result<(Participant, T), ProtocolError> {
    let (from, data) = self.incoming.pop(header).await;   // message consumed
    let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| {
        ProtocolError::DeserializationError(...)
    })?;
    let decoded = rmp_serde::decode::from_slice(message_data)...;
    Ok((from, decoded?))   // ← error returned, `from` identity is lost
}
```

The message is **consumed** from the queue before deserialization is attempted. If the payload is garbage, the error is returned without the sender's identity, so the honest party cannot even identify the culprit.

**Why any participant can craft a valid routing header:** [3](#0-2) 

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);
}
```

The only validation is that the first 40 bytes parse as a `MessageHeader` (32-byte deterministic channel tag + 8-byte little-endian waitpoint). The channel tag for the shared channel is a fixed, publicly computable SHA-256 hash: [4](#0-3) 

The waitpoints are sequential counters starting at 0, incremented by each `next_waitpoint()` call. Because the protocol structure is public, an attacker can predict the exact waitpoint for any round.

**Contrast with the echo-broadcast path, which correctly ignores errors:** [5](#0-4) 

```rust
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // ← bad message skipped, protocol continues
};
```

`echo_broadcast` uses `continue` on error; `recv_from_others` uses `?`. This inconsistency means the DKG rounds that rely on `recv_from_others` are unprotected.

**Affected call sites in DKG:** [6](#0-5) 

```rust
for (from, their_commitment_hash) in
    recv_from_others(&chan, wait_round_1, &participants, me).await?
``` [7](#0-6) 

```rust
for (from, signing_share_from) in
    recv_from_others(&chan, wait_round_3, &participants, me).await?
```

Both the commitment-hash round and the secret-share round are vulnerable.

---

### Impact Explanation

A single malicious participant (one who is legitimately part of the participant set) can abort DKG, reshare, refresh, or any other protocol that calls `recv_from_others`. Because the error does not carry the sender's identity, honest parties cannot identify or exclude the attacker. The attacker can repeat the injection on every retry, making the denial permanent.

This matches: **High — Permanent denial of key generation, reshare, refresh, or signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

- The attacker is any participant already enrolled in the protocol (no external capability required).
- The shared channel tag is a deterministic, publicly derivable constant.
- Waitpoints are sequential and predictable from the public protocol specification.
- Crafting the attack message requires only: prepend the 40-byte header, append arbitrary garbage bytes.
- No cryptographic material, no side channel, no privileged access needed.

---

### Recommendation

Mirror the error-handling pattern already used in `echo_broadcast`. In `recv_from_others`, replace the propagating `?` with a `match` that logs and skips messages that fail deserialization:

```rust
while !seen.full() {
    let result: Result<(Participant, T), _> = chan.recv(waitpoint).await;
    let (from, msg) = match result {
        Ok(v) => v,
        Err(_) => continue,   // skip malformed message, keep waiting
    };
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

Additionally, `Comms::recv` should return the sender's identity alongside any deserialization error so that callers can attribute and report misbehaving participants.

---

### Proof of Concept

```
Participants: [H1, H2, M]  (M is malicious)
Protocol: DKG, round 1 (commitment-hash exchange)
Shared channel tag: ChannelTag::root_shared()  (fixed SHA-256 constant)
Target waitpoint: 0  (first next_waitpoint() call in do_keyshare)

Attack:
  header = root_shared_tag_bytes || 0u64_le   // 40 bytes
  payload = [0xDE, 0xAD, 0xBE, 0xEF]          // invalid msgpack

  M sends (header || payload) to H1 and H2.

Result:
  H1 calls recv_from_others(wait=0).
  Comms::recv pops M's message, rmp_serde::decode fails.
  ProtocolError::DeserializationError propagates via `?`.
  do_keyshare returns Err, DKG aborted for H1.
  Same for H2.
  M repeats on every retry → permanent denial.
```
