### Title
Adversary Resets Watermark Decay Clock via Non-Blocked Swaps, Extending Stop-Loss Duration by 2–4× - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

---

### Summary

`_checkAndUpdateWatermarks` unconditionally resets `lastDecayTs = block.timestamp` after every swap, even when the watermark is actively in breach. An adversary can make tiny swaps in the non-blocked direction to repeatedly reset the decay clock, converting the intended linear decay into a much slower exponential decay. This extends the breach duration by 2–4× for realistic parameters, keeping one direction of swaps blocked far longer than the protocol intends.

---

### Finding Description

The `OracleValueStopLossExtension` uses a lazy linear decay model for per-bin high watermarks. The `_decayed` function computes:

```
decayed = hwm * (1 - rate * dt / E8)   if rate * dt < E8
decayed = 0                              if rate * dt >= E8
``` [1](#0-0) 

The intended design is that if no swap touches a bin for `E8 / rate` seconds, the watermark decays to zero and the ratchet re-arms from the current (lower) metric, resolving the breach. For `rate = 58` (~5%/day), this threshold is `1e8 / 58 ≈ 19.95 days`.

The flaw is in `_checkAndUpdateWatermarks`: after checking and potentially reverting on a breach, it **always** writes `hwmS.lastDecayTs = uint32(block.timestamp)` regardless of whether the watermark is in breach: [2](#0-1) 

The breach check is direction-aware:
- `breach0 && zeroForOne` → revert (blocks token0-in swaps)
- `breach1 && !zeroForOne` → revert (blocks token1-in swaps)

When only one direction is blocked (e.g., `breach0` is true but `breach1` is false), swaps in the non-blocked direction (`zeroForOne = false`) succeed and still execute line 284, resetting `lastDecayTs` for **both** the token0 and token1 watermarks of every touched bin. [3](#0-2) 

An adversary can make minimal-size `zeroForOne = false` swaps at intervals shorter than `E8 / rate` seconds. Each swap:
1. Applies partial linear decay to the stored watermark: `hwm_new = hwm_old * (1 - rate * dt / E8)`
2. Resets `lastDecayTs` to now

This converts the intended single-interval linear decay into a compounding exponential decay. The breach resolves when `hwm_decayed * floorMultiplier / E6 ≤ metric`, i.e., when `hwm_decayed ≤ metric * E6 / floorMultiplier = x * hwm_initial`.

- **Without adversary (linear):** `T_linear = (1 - x) * E8 / rate`
- **With adversary (exponential, daily swaps):** `T_exp ≈ -ln(x) * E8 / rate`

For `rate = 58`, `x = 0.1` (metric is 10% of breach threshold):
- `T_linear ≈ 17.96 days`
- `T_exp ≈ 45.95 days` — **2.56× longer**

For `x = 0.02`:
- `T_linear ≈ 19.56 days`
- `T_exp ≈ 67.1 days` — **3.43× longer**

The adversary's cost is gas plus any pool trading fees per swap. On low-fee pools the cost is negligible.

---

### Impact Explanation

One direction of swaps is blocked for 2–4× longer than the protocol's decay model intends. During this extended period, traders cannot execute swaps in the blocked direction and the pool's core swap functionality is degraded. The stop-loss extension is specifically designed to protect LPs by allowing the watermark to decay and re-arm; the adversary defeats this recovery mechanism, locking the pool in a one-sided halt for a materially extended window.

---

### Likelihood Explanation

Any unprivileged user can trigger this by making tiny swaps in the non-blocked direction. The attacker needs only to submit one swap per interval shorter than `E8 / rate` seconds. For `rate = 58` this is once every ~20 days; for lower rates the window is even wider. Motivation exists for any party holding a short position in the blocked token or wishing to suppress liquidity in one direction.

---

### Recommendation

Do not reset `lastDecayTs` when the watermark remains in breach after the swap. Only advance the decay clock when the bin's watermark is not in breach (i.e., when the metric is at or above the floor). A minimal fix:

```solidity
bool anyBreach = (breach0 && zeroForOne) || (breach1 && !zeroForOne);
// Only advance the clock when no breach is active for this bin
if (!anyBreach) {
    hwmS.lastDecayTs = uint32(block.timestamp);
}
hwmS.token0 = uint104(hwm0);
hwmS.token1 = uint104(hwm1);
```

Alternatively, track a separate `cumulativeDecayedSeconds` that accumulates elapsed time regardless of swap activity, so the watermark always decays at the configured linear rate independent of swap frequency.

### Proof of Concept

Setup: pool with `OracleValueStopLossExtension`, `drawdownE6 = 500_000` (50%), `decayPerSecondE8 = 58` (~5%/day), bin 0 with `hwm0 = 1000`, current `metricT0 = 100` (breach: `100 < 1000 * 0.5 = 500`). `zeroForOne = true` swaps are blocked; `zeroForOne = false` swaps succeed.

**Without adversary:**
- Breach resolves at `T = (1 - 100/500) * 1e8 / 58 ≈ 13.8 days` (linear decay from 1000 to 200).

**With adversary (daily `zeroForOne = false` swaps):**
- After each swap, `hwm` is multiplied by `(1 - 58 * 86400 / 1e8) ≈ 0.9499`.
- Breach resolves when `hwm_n * 0.5 ≤ 100`, i.e., `0.9499^N ≤ 0.2`.
- `N = ceil(log(0.2) / log(0.9499)) ≈ 32 swaps → 32 days`.

The adversary extends the one-sided swap halt from **~14 days to ~32 days** by making one cheap swap per day, with no privileged access required. [4](#0-3) [1](#0-0) [5](#0-4)

  function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
  }
