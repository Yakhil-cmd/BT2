### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

The `stake`, `change_reward_address` (staking contract), `enter_delegation_pool`, and `change_reward_address` (pool contract) functions accept a user-controlled `reward_address` parameter without validating it is non-zero. A staker or pool member can set their reward address to the zero address, causing all future reward transfers to be permanently sent to address `0` and irretrievably lost.

---

### Finding Description

**In `src/staking/staking.cairo`:**

The `stake` function validates that `reward_address` is not a token address, but performs no zero-address check: [1](#0-0) 

The `change_reward_address` function applies the same incomplete validation: [2](#0-1) 

When rewards are distributed, `send_rewards_to_staker` unconditionally transfers to whatever address is stored as `reward_address`: [3](#0-2) 

**In `src/pool/pool.cairo`:**

`enter_delegation_pool` only checks that `reward_address` is not the token address: [4](#0-3) 

`change_reward_address` in the pool contract applies the same incomplete check: [5](#0-4) 

When pool rewards are claimed, they are transferred directly to the stored `reward_address`: [6](#0-5) 

**The inconsistency is notable**: the protocol already defines a `ZERO_ADDRESS` error in `GenericError` and enforces a zero-address check on the *caller* via `assert_caller_is_not_zero()`, but omits this check for the `reward_address` parameter: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

If a staker or pool member sets `reward_address` to `0` (either at entry via `stake`/`enter_delegation_pool`, or later via `change_reward_address`), all subsequent reward transfers — including those triggered by `claim_rewards` and `unstake_action` — will be sent to the zero address. On Starknet, ERC20 transfers to address `0` succeed and the tokens are permanently unrecoverable. This constitutes **permanent freezing of unclaimed yield**, matching the allowed High impact category.

---

### Likelihood Explanation

Any staker or pool member can trigger this with a single transaction. No privileged role is required. The zero address is a valid `ContractAddress` value in Cairo and passes all existing checks (`!does_token_exist(Zero::zero())` returns `true` since zero is not a registered token). A user could set this accidentally (e.g., passing a default/uninitialized value) or be socially engineered into doing so. The protocol's own `ZERO_ADDRESS` error and caller-zero check demonstrate awareness of this class of issue, making the omission for `reward_address` a clear oversight.

---

### Recommendation

Add a non-zero assertion for `reward_address` in all four entry points:

- `stake` in `src/staking/staking.cairo`
- `change_reward_address` in `src/staking/staking.cairo`
- `enter_delegation_pool` in `src/pool/pool.cairo`
- `change_reward_address` in `src/pool/pool.cairo`

For example, before the existing token-address check:
```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

---

### Proof of Concept

**Staker path:**
1. Staker calls `stake(reward_address: 0, operational_address: <valid>, amount: min_stake)`.
2. The call succeeds — `!does_token_exist(0)` is `true`, and `0` is not a registered token.
3. Staker accumulates `unclaimed_rewards_own` over epochs via attestation/consensus rewards.
4. Anyone calls `claim_rewards(staker_address)` or `unstake_action(staker_address)`.
5. `send_rewards_to_staker` executes `token_dispatcher.checked_transfer(recipient: 0, amount: rewards)`.
6. Rewards are transferred to address `0` and permanently lost.

**Pool member path:**
1. Pool member calls `enter_delegation_pool(reward_address: 0, amount: <valid>)`.
2. The call succeeds — `token_address != 0` is `true` for any real token.
3. Pool member accumulates rewards over epochs.
4. Anyone calls `pool.claim_rewards(pool_member)`.
5. `reward_token.checked_transfer(recipient: 0, amount: rewards)` executes.
6. Rewards are permanently lost.
