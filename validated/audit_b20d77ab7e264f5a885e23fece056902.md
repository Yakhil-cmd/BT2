### Title
NFT Seller Net Proceeds Always Equals Full Asking Price Due to Commented-Out Royalty Deduction — (`File: packages/gui/src/components/offers/utils.ts`)

### Summary

`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` always sets `nftSellerNetAmount` equal to the full offer `amount`, because the correct deduction formula (`amount - royaltyAmount - makerFee`) is permanently commented out. As a result, the "You will receive" label shown to an NFT seller during offer creation, and the "Net Proceeds" label shown to a buyer reviewing a `TokenForNFT` offer, always display the full asking price rather than the actual net amount after royalties are deducted.

### Finding Description

In `calculateNFTRoyalties` (`packages/gui/src/components/offers/utils.ts`, lines 306–329):

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The correct formula — `amount - royaltyAmount - makerFee` — is commented out. `nftSellerNetAmount` is therefore always equal to `amount` (the full asking price), regardless of the royalty percentage or maker fee.

This value propagates to two user-facing surfaces:

1. **NFT Offer Editor** (`packages/gui/src/components/offers/NFTOfferEditor.tsx`, line 365): When a user selects "Sell an NFT" (`NFTForToken` tab), the label "You will receive" renders `nftSellerNetAmount`. Because `nftSellerNetAmount == amount`, the seller always sees the full asking price as what they will receive, never the net after royalties.

2. **NFT Offer Viewer** (`packages/gui/src/components/offers/NFTOfferViewer.tsx`, lines 595–597): When a buyer views a `TokenForNFT` offer, the "Net Proceeds" section renders `nftSaleInfo?.nftSellerNetAmount`. The tooltip explicitly states "The net proceeds include the asking price, minus any associated creator fees," but the displayed value never subtracts those fees.

The `royaltyAmount` is correctly computed on line 312 and displayed separately as "Creator Fee," so the total amount requested is correct. Only the `nftSellerNetAmount` field — the net the seller actually pockets — is wrong.

### Impact Explanation

An NFT seller creating an offer sees "You will receive: 10 XCH" when the actual net (after, say, a 10% royalty) is 9 XCH. The seller may set their asking price based on this incorrect "net proceeds" figure, effectively underpricing their NFT. The royalty is enforced at the blockchain/protocol level, so the seller receives less than the UI indicated at offer-creation time. This constitutes display of the wrong amount in the offer confirmation flow, which can cause a user to sign and broadcast an offer they would not have created had the correct net amount been shown.

### Likelihood Explanation

Any NFT with a non-zero `royaltyPercentage` triggers this path. The NFT offer editor and viewer are standard, reachable UI flows requiring no special privileges. Any user who creates or reviews an NFT-for-token offer is affected.

### Recommendation

Uncomment and restore the correct `nftSellerNetAmount` calculation:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

Ensure the result is clamped to zero or surfaced as an error when negative (the existing `showNegativeAmountWarning` guard in `NFTOfferEditor.tsx` already handles the display side of this).

### Proof of Concept

1. Open the Chia GUI and navigate to **Offers → Create an Offer**.
2. Select "Sell an NFT" tab. Choose an NFT whose `royaltyPercentage > 0` (e.g., 10%).
3. Enter an asking price of 10 XCH.
4. Observe: "Creator Fee: 1 XCH" is shown correctly, but "You will receive: **10 XCH**" is shown instead of the correct **9 XCH**.
5. The seller proceeds to create and sign the offer believing they will net 10 XCH; on acceptance they receive 9 XCH.
