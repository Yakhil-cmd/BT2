### Title
Hardcoded `L1_TX_NATIVE_PRICE` Bypasses Operator-Controlled Dynamic Native Price for L1→L2 Transactions, Enabling Underpriced Proving Resource Consumption - (File: `basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded to `10` and used as the native price for all L1→L2 transactions, bypassing the operator's dynamically-set `native_price` from block metadata. This creates a fixed exchange rate between gas price and native (proving) resources that cannot adapt to actual proving costs — a direct analog to hardcoding a stablecoin peg.

---

### Finding Description

For L2 transactions, the native price is dynamically set by the operator via block metadata (`BlockMetadataFromOracle::native_price`). This allows the operator to adjust the price to reflect actual proving costs at any time. However, for L1→L2 transactions, `L1_TX_NATIVE_PRICE` is used as a hardcoded constant: [1](#0-0) 

The TODO comment explicitly acknowledges the value is a placeholder with no principled derivation. The constant is imported and applied in the L1 transaction processing path: [2](#0-1) 

For L2 transactions, the native-per-gas ratio is computed dynamically using the operator-supplied `native_price` from block metadata: [3](#0-2) 

The `native_price` field is operator-controlled and lives in `BlockMetadataFromOracle`: [4](#0-3) [5](#0-4) 

The resource limit derivation (from docs) is:
```
native_per_gas := gas_price / native_price
native_limit   := gas_limit * native_per_gas
```

For L2 transactions, `native_price` is the operator-set value (e.g., `1000`). For L1→L2 transactions, `native_price` is always `L1_TX_NATIVE_PRICE = 10`, regardless of what the operator has set. The `create_resources_for_tx` function then uses `native_per_gas == 0` as the `free_native` flag: [6](#0-5) 

---

### Impact Explanation

When proving costs increase and the operator raises `native_price` for L2 transactions, L1→L2 transactions remain priced at the hardcoded rate of `10`. An attacker can exploit this divergence:

- **L2 transaction** with `gas_price = 100`, operator `native_price = 1000`:
  `native_per_gas = ceil(100/1000) = 1`, `native_limit = gas_limit * 1`

- **L1→L2 transaction** with same `gas_price = 100`, hardcoded `L1_TX_NATIVE_PRICE = 10`:
  `native_per_gas = ceil(100/10) = 10`, `native_limit = gas_limit * 10`

The attacker receives **10× more native (proving) resources** per unit of gas price than an equivalent L2 transaction. The sequencer/prover must absorb the extra proving cost. At scale, an attacker flooding the system with computationally expensive L1→L2 transactions forces the protocol to prove far more work than it was compensated for, leading to direct financial losses for the operator.

The `MAX_NATIVE_COMPUTATIONAL` cap limits per-transaction abuse but does not prevent sustained batched exploitation across many transactions. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The TODO comment (`// TODO (EVM-1157): find a reasonable value for it.`) explicitly acknowledges the value is not yet properly calibrated. As the protocol matures, proving costs will fluctuate with ETH price, hardware costs, and circuit complexity changes. The operator already has the mechanism to adjust `native_price` for L2 transactions — the gap between that dynamic value and the hardcoded L1 value is the exploitable surface. Any unprivileged user can submit L1→L2 transactions; no special access is required.

---

### Recommendation

Replace the hardcoded `L1_TX_NATIVE_PRICE` with the operator-supplied `native_price` from block metadata, the same source used for L2 transactions. If a floor or separate L1-specific price is needed for DDoS protection, it should be a governance-updatable parameter rather than a compile-time constant. At minimum, add a runtime check that `L1_TX_NATIVE_PRICE` does not diverge from the current block `native_price` by more than a configurable factor.

---

### Proof of Concept

1. Operator observes proving costs have increased 100×; sets `native_price = 1000` in block metadata for L2 transactions.
2. Attacker submits an L1→L2 transaction with `gas_price = 50`, `gas_limit = 10_000_000`, containing a tight loop of expensive EVM opcodes (e.g., `KECCAK256`, `SSTORE`).
3. L2 equivalent would yield `native_per_gas = ceil(50/1000) = 1`, `native_limit = 10_000_000`.
4. L1→L2 transaction yields `native_per_gas = ceil(50/10) = 5`, `native_limit = 50_000_000` — 5× more proving budget for the same payment.
5. Attacker repeats across many transactions; the prover must prove 5× the work it was paid for.
6. At `MAX_NATIVE_COMPUTATIONAL` cap, each transaction consumes the maximum proving budget at a fraction of the intended cost. [8](#0-7)
