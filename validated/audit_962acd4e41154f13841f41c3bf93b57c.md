### Title
Cross-COA Signature Replay via Missing `evmAddress` Binding in `validateCOAOwnershipProof` - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` verifies that the supplied signatures are cryptographically valid over `signedData` but does **not** enforce that `signedData` encodes the target `evmAddress`. Because the signed payload is not bound to a specific COA address, a valid proof produced for COA_A can be replayed verbatim against COA_B when both COAs are owned by the same Cadence account. The vulnerability is explicitly acknowledged in the source code comment but is not mitigated at the protocol level.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies that the supplied `signatures` over `signedData` are valid for the Cadence account's keys.
2. Verifies that the COA resource at the supplied `path` has the given `evmAddress`.

The code itself documents the gap:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

Because `signedData` is never required to commit to `evmAddress`, the two checks are decoupled. An attacker who observes a valid proof `{Cadence address A, /public/coaA, keyIndices, signatures over D}` can construct a new proof `{Cadence address A, /public/coaB, keyIndices, same signatures over D}` and submit it against COA_B.

The EVM-side entry point is the COA's ERC-1271 `isValidSignature` implementation:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig) external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig)
    );
    ...
}
``` [2](#0-1) 

Here `address(this)` is always the calling COA's own EVM address, and `_hash` is the attacker-supplied message hash. The `_sig` (the encoded `COAOwnershipProof`) is also fully attacker-controlled. The precompile handler decodes these and calls `validateCOAOwnershipProof`: [3](#0-2) 

The Cadence function then verifies signatures over `signedData` (step 1) and checks that the COA at the given `path` matches `evmAddress` (step 2), but never checks that `signedData` encodes `evmAddress`: [4](#0-3) 

**Concrete replay path:**

1. Victim owns Cadence account `A` with two COAs: `COA_A` at `/public/coaA` and `COA_B` at `/public/coaB`.
2. Victim produces a valid proof for `COA_A` by signing arbitrary data `D` with their Cadence key (domain tag `"FLOW-V0.0-user"`).
3. Attacker observes the proof on-chain (it is submitted as EVM calldata, which is public).
4. Attacker constructs a new proof: same `{Cadence address A, keyIndices, signatures over D}` but with `path = /public/coaB`.
5. Attacker calls `COA_B.isValidSignature(D, new_proof)`.
6. The precompile calls `validateCOAOwnershipProof(A, /public/coaB, D, keyIndices, signatures, COA_B.address)`.
7. Signatures over `D` are valid (same Cadence account keys) ✓
8. COA at `/public/coaB` has address `COA_B.address` ✓
9. Returns `isValid: true` — COA_B appears to have authorized `D`.

### Impact Explanation

A successful replay causes `isValidSignature` on COA_B to return `ValidERC1271Signature (0x1626ba7e)` for a message the COA_B owner never intended to authorize. Any ERC-1271-aware EVM protocol (token permit systems, NFT marketplace approvals, ERC-4337 account abstraction, DeFi vault authorizations) that trusts this return value will treat the forged authorization as legitimate. This can lead to unauthorized asset transfers from COA_B — a direct cross-VM asset loss.

### Likelihood Explanation

The precondition is that the victim holds **two or more COAs under the same Cadence account**. The comment in the source code dismisses this as low-risk because "Flow wallets historically create at most one COA per account," but:

- The protocol imposes no limit on the number of COAs per account.
- As the Flow EVM ecosystem matures, multi-COA patterns (e.g., one COA per application, one COA per vault) are a natural usage pattern.
- The attacker needs only to observe a single valid user-tagged signature from the victim's Cadence account — no privileged access is required.

Likelihood is **medium** given current ecosystem state and **high** as multi-COA usage grows.

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. The simplest fix is to require that `signedData` is the hash of `evmAddress || application_data`, or to hash `evmAddress` into the signed payload before verification:

```cadence
// Require signedData to be keccak256(evmAddress || userPayload)
// OR verify that the first 20 bytes of signedData equal evmAddress
```

Alternatively, the function can construct the expected signed message internally by prepending `evmAddress` to whatever payload the caller provides, ensuring the signature is always bound to the specific COA being validated. This mirrors the nonce-mapping recommendation in the original report: bind each authorization to its unique target so it cannot be replayed elsewhere.

### Proof of Concept

```
Attacker-controlled inputs to COA_B.isValidSignature(_hash=D, _sig=proof):

proof = RLP-encode({
    Address:        victim_cadence_account,   // same as original
    CapabilityPath: "coaB",                   // CHANGED from "coaA"
    KeyIndices:     [0],                      // same as original
    Signatures:     [sig_over_D],             // REPLAYED from COA_A proof
})

Call: COA_B.isValidSignature(D, proof)
  → verifyCOAOwnershipProof(address(COA_B), D, proof)
  → validateCOAOwnershipProof(victim, /public/coaB, D, [0], [sig_over_D], COA_B.bytes)
      check 1: keyList.verify(sig_over_D, signedData=D) → true  (same Cadence keys)
      check 2: coaRef.address().bytes == COA_B.bytes    → true  (path points to COA_B)
  → returns ValidationResult(isValid: true)
  → isValidSignature returns 0x1626ba7e  ← forged ERC-1271 approval for COA_B
```
