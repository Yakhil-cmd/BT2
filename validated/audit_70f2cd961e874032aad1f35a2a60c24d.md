### Title
Attacker Bypasses Signature Verification via `skip_stateful_validations` to Inject Unauthorized Invoke Transaction — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature, designed to allow a `deploy_account + invoke` pair to be submitted simultaneously, uses an overly broad membership check (`account_tx_in_pool_or_recent_block`) to decide whether to skip the `__validate__` entry point. Because that check returns `true` for **any** transaction in the pool for the target address — not specifically a `deploy_account` — an attacker can exploit a victim's pending `deploy_account` to inject an invoke transaction with an **invalid signature** that bypasses all gateway signature verification and is admitted to the mempool.

---

### Finding Description

**Invariant broken:** Every invoke transaction admitted to the mempool must have its signature verified by the account's `__validate__` entry point before admission.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function skips `__validate__` when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

The inline comment claims this is safe because the presence of any transaction in the pool implies a `deploy_account` was submitted. That reasoning is wrong: the check is performed against the **victim's** address, not the attacker's. The attacker's invoke tx is the first submission from the attacker for that address; it has never passed any signature check.

**`account_tx_in_pool_or_recent_block` is too broad:** [3](#0-2) 

It returns `true` if the address has **any** transaction in `tx_pool` or appears in `state` (staged/committed). It does not distinguish `deploy_account` from `invoke` or `declare`.

**No other guard catches an invalid signature:**

The stateless validator only checks signature **length**, not cryptographic validity: [4](#0-3) 

`validate_by_mempool` only checks nonce ordering: [5](#0-4) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false`, so `__validate__` is never called at the gateway: [6](#0-5) 

**`run_pre_validation_checks` call order confirms no signature check occurs before the skip decision:** [7](#0-6) 

---

### Impact Explanation

**Matching impact:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

An attacker submits an invoke tx with an arbitrary/invalid signature for a victim's undeployed account address. The gateway admits it to the mempool without any signature check. When the batcher later executes the block:

1. The victim's `deploy_account` (nonce 0) executes and deploys the account.
2. The attacker's invoke (nonce 1) executes; `__validate__` is now called by the blockifier (default `ExecutionFlags` has `validate: true`); the invalid signature causes the tx to **revert**.
3. The victim's newly deployed account is charged fees for a transaction they never authorized.

The attacker can target any account whose `deploy_account` is visible in the mempool — a publicly observable event. The attack requires no privileged access and no on-chain funds from the attacker.

---

### Likelihood Explanation

- The victim's `deploy_account` transaction is visible in the mempool (public information).
- The attacker only needs to submit a well-formed invoke tx (valid resource bounds, valid nonce=1, any signature of valid length) targeting the victim's address.
- The window is open from the moment the victim's `deploy_account` enters the mempool until it is committed.
- No special capability, key material, or privileged role is required.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account`** transaction exists for the address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type stored in `tx_pool` for the address at nonce 0. Only if a `deploy_account` is confirmed pending should `__validate__` be skipped for the subsequent invoke.

Alternatively, the gateway can inspect the transaction type of the nonce-0 entry for the address directly before granting the skip, without requiring a new mempool API.

---

### Proof of Concept

**Setup:**
- Victim generates a new keypair; computes account address `A` deterministically.
- Victim submits `deploy_account` tx for address `A` (nonce=0, valid signature). It enters the mempool.

**Attack:**
1. Attacker observes `deploy_account` for address `A` in the mempool → `account_tx_in_pool_or_recent_block(A)` is now `true`.
2. Attacker constructs an `Invoke V3` transaction:
   - `sender_address = A`
   - `nonce = 1`
   - `signature = [0x1, 0x2]` (two arbitrary felts, satisfying the length check of `max_signature_length`)
   - `resource_bounds` set to valid non-zero values
3. Attacker submits the tx to the gateway.

**Gateway processing:**
- Stateless validator: signature length 2 ≤ `max_signature_length` → **passes**.
- `validate_nonce`: `0 ≤ 1 ≤ 200` → **passes**.
- `validate_by_mempool`: nonce not too old → **passes**.
- `skip_stateful_validations`: nonce==1 ✓, account_nonce==0 ✓, `account_tx_in_pool_or_recent_block(A)` == true ✓ → returns `true`.
- `run_validate_entry_point` called with `validate: false` → `__validate__` **skipped**.
- Tx admitted to mempool. ✓

**Execution:**
- Batcher sequences: `deploy_account` (nonce 0) executes → account deployed.
- Batcher sequences: attacker's invoke (nonce 1) executes → `__validate__` called → invalid ECDSA signature → tx **reverts**.
- Victim's account is charged fees for the reverted unauthorized transaction.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
