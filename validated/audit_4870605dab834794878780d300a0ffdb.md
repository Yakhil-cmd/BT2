### Title
Invoke Transaction Signature Validation Unconditionally Bypassed via `skip_stateful_validations` Nonce Condition - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway stateful validator skips the `__validate__` entry-point call for any Invoke transaction whose nonce is exactly `1` and whose sender's on-chain nonce is `0`, provided the mempool reports any transaction for that address. The check used — `account_tx_in_pool_or_recent_block` — does not verify that the pooled transaction is a `DeployAccount`; it returns `true` for any transaction type. An unprivileged attacker can therefore submit a `DeployAccount` (valid) followed by an `Invoke(nonce=1, arbitrary_signature)` and have the invoke admitted to the mempool without any signature or `__validate__` verification.

### Finding Description

`skip_stateful_validations` (lines 429–461) returns `true` when three conditions hold simultaneously:

1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain state)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }` and passes it to `StatefulValidator::perform_validations`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by `tx.execution_flags.validate`: [3](#0-2) 

The mempool check that gates the skip is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

`MempoolState::contains_account` returns `true` if the address appears in either `staged` or `committed`: [5](#0-4) 

`TransactionPool::contains_account` returns `true` for **any** transaction type for that address: [6](#0-5) 

The code comment claims this is sufficient to prove a `DeployAccount` exists: [7](#0-6) 

This claim is structurally correct for the normal case (an undeployed account can only have a `DeployAccount` with nonce 0 pass gateway validation), but the check does not enforce it. Specifically:

- `state.contains_account` also returns `true` when the address is in `staged` — i.e., a `DeployAccount` was already handed to the batcher but the block has not yet been committed. During this window, an attacker can submit `Invoke(nonce=1, bad_sig)` and have it admitted without `__validate__`.
- `tx_pool.contains_account` does not distinguish `DeployAccount` from any other transaction type. If a future code path allows a non-`DeployAccount` transaction with nonce 0 to enter the pool for an undeployed account, the skip would fire incorrectly.

**Attack path (unprivileged):**

1. Attacker generates fresh address `A` (controls private key).
2. Submits valid `DeployAccount(nonce=0, valid_sig)` for `A` → passes full gateway validation, enters `tx_pool`.
3. Immediately submits `Invoke(nonce=1, arbitrary_sig)` for `A`.
4. Gateway evaluates: `account_nonce=0`, `tx_nonce=1`, `account_tx_in_pool_or_recent_block(A)=true` → `skip_validate=true` → `__validate__` is **not called**.
5. The invoke is admitted to the mempool with an unverified, potentially invalid signature.
6. During block production, `DeployAccount` executes first; then `Invoke(nonce=1)` is executed by the blockifier with `validate=true` — `__validate__` is called and the transaction **reverts** if the signature is invalid.

The broken invariant: **every transaction admitted to the mempool must have passed `__validate__` (signature verification) at the gateway level**. The `skip_stateful_validations` path breaks this invariant unconditionally for `(nonce=1, account_nonce=0, any_tx_in_pool)`.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An invoke transaction bearing an arbitrary or malformed signature is admitted to the mempool without any cryptographic verification. The transaction will revert during execution (blockifier calls `__validate__` with `validate=true` during sequencing), so no persistent state is corrupted. However, the admission invariant is violated: the mempool contains a transaction that would fail its own account's validation logic. This can be used to:

- Pollute the mempool with transactions that will always revert, consuming sequencer resources.
- Bypass gateway-level rate-limiting or fee checks that depend on `__validate__` succeeding.
- In accounts with non-standard `__validate__` logic (multisig, session keys, paymasters), submit transactions that would never pass the account's own policy — they are admitted, consume a nonce slot, and revert.

### Likelihood Explanation

**Medium.** The attacker must control a fresh address (trivial — generate a keypair) and submit two transactions in sequence. No privileged access is required. The window is always open as long as the `DeployAccount` remains in the pool (until the block is committed). The `committed_nonce_retention_block_count` default of 100 blocks means the window can persist across many blocks. [8](#0-7) 

### Recommendation

1. **Narrow the pool check**: In `skip_stateful_validations`, instead of calling `account_tx_in_pool_or_recent_block` (which matches any transaction type), query the pool specifically for a `DeployAccount` transaction

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
