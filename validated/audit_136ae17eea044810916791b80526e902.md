### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Deployed Accounts with Any Pending Transaction — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry point only when a `deploy_account` transaction is pending in the mempool for an undeployed account (UX improvement for the deploy+invoke pattern). However, it uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from the account — not just `deploy_account` transactions. An already-deployed account that has any pending nonce-0 invoke in the mempool can submit a nonce-1 invoke with an **invalid signature** and have it admitted to the mempool without `__validate__` ever running at the gateway.

### Finding Description

`run_pre_validation_checks` calls `skip_stateful_validations` to decide whether to run the blockifier's `__validate__` entry point: [1](#0-0) 

`skip_stateful_validations` fires when `tx.nonce() == 1 && account_nonce == 0`, then delegates the "is a deploy_account pending?" question to `account_tx_in_pool_or_recent_block`: [2](#0-1) 

The mempool implementation of `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the address — it checks `state.contains_account` (any committed account) OR `tx_pool.contains_account` (any pooled transaction): [3](#0-2) 

The code comment in `skip_stateful_validations` acknowledges this but incorrectly reasons: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The second branch of that reasoning is the flaw: a nonce-0 **invoke** (not a deploy_account) from an already-deployed account also satisfies the check, yet the account is fully deployed and `__validate__` must run.

When `skip_validate = true`, `run_validate_entry_point` sets `validate: !skip_validate = false` and the blockifier's `StatefulValidator` returns early without calling `__validate__`: [4](#0-3) [5](#0-4) 

No other step in the gateway pipeline verifies the transaction signature. `validate_by_mempool` only checks for duplicate hashes and fee-escalation rules: [6](#0-5) 

`convert_rpc_tx_to_internal_and_executable_txs` performs compiled-class-hash validation and proof verification but not signature verification: [7](#0-6) 

### Impact Explanation

A transaction with an **invalid signature** (or any payload that would fail `__validate__`) is admitted to the mempool without the gateway ever running the account's validation entry point. This breaks the gateway's admission invariant: *"only transactions whose `__validate__` passes are forwarded to the mempool."*

Matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The precondition is common in normal operation: any deployed account (on-chain nonce = 0) that has submitted a single nonce-0 invoke which is still pending in the mempool satisfies the trigger. An attacker who controls such an account (or can observe one) can immediately exploit the bypass. No privileged access is required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool`, or the gateway should inspect the transaction type of the pooled transaction before deciding to skip `__validate__`.

### Proof of Concept

1. Deploy account `A` on-chain; its nonce in state is `0`.
2. Submit a valid `Invoke V3` from `A` with `nonce = 0` and a correct signature. It passes all gateway checks (including `__validate__`) and enters the mempool. Now `tx_pool.contains_account(A) == true`.
3. Craft a second `Invoke V3` from `A` with `nonce = 1` and a **deliberately invalid signature** (e.g., all-zero signature bytes).
4. Submit it to the gateway. The gateway calls `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(A)` → `Nonce(0)`.
   - `validate_state_preconditions`: nonce `1` is within `[0, max_allowed_nonce_gap]` → passes.
   - `validate_by_mempool`: no duplicate hash, no fee-escalation violation → passes.
   - `skip_stateful_validations`: `tx.nonce() == 1 && account_nonce == 0` → calls `account_tx_in_pool_or_recent_block(A)` → `true` (the nonce-0 invoke is still pooled) → returns `true`.
5. `run_validate_entry_point` is called with `skip_validate = true` → `execution_flags.validate = false` → `StatefulValidator::perform_validations` returns `Ok(())` without executing `__validate__`.
6. The invalid-signature transaction is forwarded to the mempool and accepted.

The gateway has admitted a transaction whose signature was never verified, violating the admission invariant. [2](#0-1) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
