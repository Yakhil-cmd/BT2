### Title
Zero Echo-Broadcast Thresholds for n ≤ 3 Allow Malicious Participant to Equivocate and Corrupt DKG Public Key Output — (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

When the participant count `n` is 2 or 3, `echo_ready_thresholds` returns `(0, 0)` for both the echo and ready thresholds. Because every phase-advancement check in `reliable_broadcast_receive_all` uses strict greater-than (`> echo_t`, `> ready_t`, `> 2 * ready_t`), a single self-simulated vote immediately satisfies every threshold. A malicious participant can therefore send a different `Send` value to each honest party, causing them to deliver inconsistent data from the same broadcast instance. When this equivocation occurs during the DKG commitment broadcast (Round 3 of `do_keyshare`), honest parties compute and accept different public keys, permanently corrupting the DKG output without any detection.

---

### Finding Description

**Root cause — `echo_ready_thresholds` returns zero for n ≤ 3:** [1](#0-0) 

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // echo_t = 0, ready_t = 0
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

With `echo_t = 0` and `ready_t = 0`, the three phase-advancement guards in `reliable_broadcast_receive_all` become:

| Guard | Condition | Effective requirement |
|---|---|---|
| Echo → Ready | `count > echo_t` | `count > 0` (1 vote) |
| Ready → Amplify | `count > ready_t` | `count > 0` (1 vote) |
| Amplify → Deliver | `count > 2 * ready_t` | `count > 0` (1 vote) | [2](#0-1) [3](#0-2) 

Each honest party's own self-simulated echo immediately satisfies all three guards, so the party advances through all phases before receiving any echo from other honest parties. The `finish_echo = true` flag is set immediately, causing all subsequently received echoes from other parties to be silently dropped.

**Equivocation attack with n = 3 (2 honest + 1 malicious):**

Participants: A (honest), B (honest), C (malicious sender).

1. C sends `Send(value_A)` to A and `Send(value_B)` to B (where `value_A ≠ value_B`).
2. A echoes `Echo(value_A)` to all; A's own simulated echo immediately satisfies `count > 0` → A sends `Ready(value_A)`, sets `finish_echo = true`.
3. B echoes `Echo(value_B)` to all; B's own simulated echo immediately satisfies `count > 0` → B sends `Ready(value_B)`, sets `finish_echo = true`.
4. A's simulated `Ready(value_A)` satisfies `count > 0` twice → A delivers `value_A`.
5. B's simulated `Ready(value_B)` satisfies `count > 0` twice → B delivers `value_B`.

A and B have irreconcilably different views of C's broadcast.

**How this corrupts DKG in `do_keyshare`:**

`do_keyshare` calls `do_broadcast` twice:

- **Round 1** (session ID exchange): [4](#0-3) 
- **Round 3** (commitment + proof-of-knowledge broadcast): [5](#0-4) 

For the Round 3 attack, C can be honest in Round 1 (so A and B share the same `session_id`), then equivocate in Round 3:

- C sends `(commitment_C_A, proof_C_A)` to A and `(commitment_C_B, proof_C_B)` to B, where `commitment_C_A(0) ≠ commitment_C_B(0)`.
- In Round 2, C sends `H(C, commitment_C_A, session_id)` to A and `H(C, commitment_C_B, session_id)` to B (Round 2 is a direct `send_many`, not a broadcast): [6](#0-5) 
- A verifies: `H(C, commitment_C_A, session_id) == received_hash` ✓ [7](#0-6) 
- B verifies: `H(C, commitment_C_B, session_id) == received_hash` ✓
- Both proofs of knowledge are valid because C chose both polynomials.
- A computes `public_key_A = commitment_A(0) + commitment_B(0) + commitment_C_A(0)`.
- B computes `public_key_B = commitment_A(0) + commitment_B(0) + commitment_C_B(0)`.
- `public_key_A ≠ public_key_B`.

The final `broadcast_success` check broadcasts `(true, session_id)`. Since A and B share the same `session_id` (C was honest in Round 1), this check passes and the inconsistency is **never detected**: [8](#0-7) 

---

### Impact Explanation

**Impact: High** — Corruption of DKG outputs so honest parties accept inconsistent public keys.

Honest parties A and B complete the DKG protocol without error but hold shares of two different secrets corresponding to two different public keys. Any threshold signature produced by a quorum including A will not verify against B's public key, and vice versa. The DKG output is permanently unusable for its intended purpose. The same attack applies to reshare and refresh, since both call `do_keyshare` internally. [9](#0-8) 

---

### Likelihood Explanation

**Likelihood: High.**

- The minimum allowed participant count is 2 (`participants.len() < 2` check in `assert_key_invariants`), so n = 3 is a minimal, realistic deployment. [10](#0-9) 
- The attack requires only one malicious participant who controls their own message delivery — no external assumptions, no key leakage, no cryptographic breaks.
- The malicious participant simply sends two different byte strings to two different honest parties during the `do_broadcast` call. This is a standard equivocation attack that the echo broadcast is specifically designed to prevent, but fails to prevent when n ≤ 3.
- The attack is silent: no error is returned to honest parties, and the corrupted public keys are returned as valid `KeygenOutput`.

---

### Recommendation

Replace the special-case zero-threshold branch with a formula that is safe for all n ≥ 2. The standard echo broadcast from Cachin–Guerraoui–Rodrigues requires n ≥ 3f + 1 to tolerate f Byzantine faults. For n = 3, f = 0 is the only safe assumption, meaning the echo and ready thresholds should be set to values that require all n parties to agree (i.e., `echo_t = n - 1`, `ready_t = 0` for f = 0). Alternatively, enforce a minimum participant count of 4 at the API level to guarantee at least one Byzantine fault can be tolerated by the echo broadcast layer.

At minimum, document clearly that for n ≤ 3 the echo broadcast provides **no equivocation protection**, and that callers must not include any potentially malicious participant when n ≤ 3.

### Proof of Concept

**Setup**: n = 3, participants A (index 0), B (index 1), C (index 2, malicious). Threshold = 2.

**Step 1**: C participates honestly in Round 1 (session ID broadcast). A and B obtain the same `session_id`.

**Step 2**: In Round 2, C sends:
- `commitment_hash_A = H(C, commitment_C_A, session_id)` to A
- `commitment_hash_B = H(C, commitment_C_B, session_id)` to B

where `commitment_C_A` and `commitment_C_B` are commitments to two different polynomials with different constant terms.

**Step 3**: In Round 3 (`do_broadcast` for commitments), C sends:
- `Send((commitment_C_A, proof_C_A))` to A
- `Send((commitment_C_B, proof_C_B))` to B

**Step 4**: Because `echo_ready_thresholds(3) = (0, 0)`, A's own simulated echo of `commitment_C_A` immediately satisfies `count > 0`, advancing A to deliver `commitment_C_A` before receiving B's echo of `commitment_C_B`. B symmetrically delivers `commitment_C_B`.

**Step 5**: Both A and B pass all verification checks (commitment hash, proof of knowledge) because C crafted both consistently with `session_id`.

**Step 6**: A returns `KeygenOutput { public_key: pk_A }` and B returns `KeygenOutput { public_key: pk_B }` where `pk_A ≠ pk_B`. The DKG is silently corrupted. [1](#0-0) [11](#0-10) [12](#0-11) [4](#0-3) [5](#0-4)
