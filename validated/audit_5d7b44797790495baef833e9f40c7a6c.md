### Title
Valid Proof-Bearing Invoke Transaction Rejected on GCS Archive Timeout — (`crates/apollo_gateway/src/gateway.rs`)

---

### Summary

`GenericGateway::await_proof_archiving` enforces a hard 5-second timeout on the GCS proof archive write. When GCS is slow or degraded, this timeout fires and the function returns a fatal `StarknetError::internal`, which is propagated with `?` in `add_tx_inner` before `mempool_client.add_tx` is ever called. A fully valid proof-bearing invoke transaction — one that has passed every stateless and stateful canonical check and whose proof has already been committed to the proof manager — is therefore rejected at the gateway admission boundary solely because of an archival side-effect.

---

### Finding Description

The execution order in `add_tx_inner` is:

1. Stateless validation — passes.
2. Transaction conversion + proof verification — passes.
3. Stateful validation — passes.
4. `store_proof_and_spawn_archiving` — spawns the GCS write task **and** synchronously commits the proof to the proof manager. Both succeed.
5. `await_proof_archiving` — waits up to `PROOF_ARCHIVE_WRITE_TIMEOUT` (5 s) for the GCS task.
6. `mempool_client.add_tx` — **never reached** if step 5 times out. [1](#0-0)

### Impact Explanation

The admission decision — accept or reject — is coupled to a non-canonical archival side-effect (GCS write). Any GCS latency spike above 5 seconds, or any transient GCS error, causes the gateway to return an internal error to the submitter for a transaction that is canonically valid and whose proof is already stored. The transaction is never forwarded to the mempool. The submitter must retry, but the proof is already in the proof manager, creating an inconsistent intermediate state. This directly matches the "High" impact category: **valid transactions rejected before sequencing**.

---

### Likelihood Explanation

GCS write latency exceeding 5 seconds is a realistic operational event (cold-start, regional degradation, large proof payload, network jitter). The `GcsProofArchiveWriter` has no retry mechanism (noted in a `TODO` comment at line 66 of `proof_archive_writer.rs`): [7](#0-6) 

Any proof-bearing invoke transaction submitted during such a window is affected. The submitter is an unprivileged user; no operator or admin access is required.

---

### Recommendation

Decouple the archival write from the admission decision. Options in increasing robustness:

- **Log-and-continue**: on GCS timeout or error, log and increment the metric but return `Ok(())` so the transaction proceeds to the mempool. The GCS write is archival, not canonical.
- **Background retry queue**: enqueue failed GCS writes for async retry rather than blocking or failing the admission path.
- **Separate the concerns entirely**: move GCS archiving to a post-mempool hook so that admission is never gated on external storage availability.

The proof manager store (the canonical store) already succeeds before the GCS timeout fires, so the transaction's proof is not lost; only the archival copy is missing.
