### Title
`EVM.validateCOAOwnershipProof` Does Not Bind `signedData` to `evmAddress`, Enabling Cross-COA Signature Replay - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's key signatures are valid over an arbitrary `signedData` blob, then checks that the COA at the given `path` matches the provided `evmAddress`. However, the function does not enforce that `signedData` encodes `evmAddress`. This is directly analogous to the ERC865 bug: the signed payload is missing a critical binding field (the target address), so a valid signature produced for one COA can be replayed to prove ownership of a different COA owned by the same Cadence account.

### Finding Description

`EVM.validateCOAOwnershipProof` is an `access(all)` Cadence function callable by any unprivileged Cadence script or transaction. It accepts:

- `address`: the Cadence account
- `path`: the `PublicPath` where the COA capability is published
- `signedData`: arbitrary bytes the caller signed
- `keyIndices` / `signatures`: the signing keys and their signatures
- `evmAddress`: the expected EVM address of the COA

The function verifies the signatures over `signedData` and then checks that the COA at `path` has address `evmAddress`. Crucially, `signedData` is never required to contain `evmAddress`. [1](#0-0) 

The code's own comment acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The signature verification step: [2](#0-1) 

...verifies only that the signatures are valid over `signedData` with the `"FLOW-V0.0-user"` domain tag. There is no check that `signedData` encodes the target `evmAddress`.

The COA address check: [3](#0-2) 

...only verifies that the COA at `path` matches `evmAddress`, but since `path` is also attacker-controlled, an attacker can supply a different path pointing to a different COA.

### Impact Explanation

If a Cadence account holds two COAs at different public paths (e.g., `/public/coaA` and `/public/coaB`), a signature produced by the account owner to prove ownership of COA_A (EVM address A) can be replayed by an attacker to forge a proof of ownership of COA_B (EVM address B):

1. Victim signs `signedData` to prove ownership of COA_A at `/public/coaA`.
2. Attacker calls `EVM.validateCOAOwnershipProof` with the same `signedData` and signatures, but substitutes `path=/public/coaB` and `evmAddress=B`.
3. Signature verification passes (same data, same keys).
4. COA address check passes (COA at `/public/coaB` has address B).
5. Function returns `isValid: true` — a forged proof of ownership of COA_B.

Any dApp or Cadence contract that calls `EVM.validateCOAOwnershipProof` directly to gate access to on-chain assets is vulnerable to this cross-COA replay. The function is `access(all)` and reachable by any unprivileged transaction sender. [4](#0-3) 

### Likelihood Explanation

The EVM precompile path (`verifyCOAOwnershipProof` in `arch.go`) is safe because it always passes `address(this)` as the COA address: [5](#0-4) [6](#0-5) 

However, the Cadence-level `EVM.validateCOAOwnershipProof` is `access(all)` and is directly callable by any script or transaction. The likelihood is **medium-low**: it requires a Cadence account to hold multiple COAs at different paths (not enforced by the protocol), and a dApp that uses the Cadence function directly rather than the EVM precompile. Both conditions are realistic as the ecosystem matures and multi-COA patterns emerge.

### Recommendation

Enforce that `signedData` encodes `evmAddress` inside `validateCOAOwnershipProof`, or derive the message to verify as `hash(signedData || evmAddress)` rather than verifying over bare `signedData`. This binds each signature to a specific COA address and prevents cross-COA replay, matching the security property the ERC865 report recommends for its analogous case.

### Proof of Concept

```cadence
// Attacker transaction demonstrating cross-COA replay
import EVM from 0xEVMAddress

transaction(
    victimAddress: Address,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    coaBPath: PublicPath,
    coaBEvmAddress: [UInt8; 20]
) {
    execute {
        // Victim signed `signedData` to prove ownership of COA_A.
        // Attacker replays the same signature against COA_B at a different path.
        let result = EVM.validateCOAOwnershipProof(
            address: victimAddress,
            path: coaBPath,           // different path → different COA
            signedData: signedData,   // same signed data, no evmAddress binding
            keyIndices: keyIndices,
            signatures: signatures,
            evmAddress: coaBEvmAddress
        )
        // result.isValid == true — forged proof of COA_B ownership
        assert(result.isValid, message: "replay succeeded")
    }
}
