### Title
`uint32` Overflow in `_afterTimelock` Allows Pool Admin to Instantly Bypass Stop-Loss Timelock ŌĆö (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` casts `block.timestamp + timelock` (a `uint256` sum) down to `uint32`. When `timelock` is set to any value exceeding `type(uint32).max ŌłÆ block.timestamp` (Ōēł 80 years from July 2026, well within the `uint32` domain of 136 years), the truncation wraps the result to a **past** timestamp. Every subsequent `_requireElapsed` check then passes immediately, letting the pool admin propose and execute drawdown/decay/watermark changes in a single block ŌĆö collapsing the LP-protection guarantee the timelock is meant to enforce.

---

### Finding Description

`_afterTimelock` computes the unlock timestamp for all pending stop-loss parameter changes:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space (no overflow there), but the result is then **truncated to `uint32`**. If `block.timestamp + timelock > 2^32 ŌłÆ 1 = 4 294 967 295`, the high bits are silently discarded.

At the current epoch (~1 753 000 000, July 2026), overflow occurs when `timelock > 2 541 967 295` seconds (Ōēł 80.6 years). `type(uint32).max = 4 294 967 295` seconds (Ōēł 136 years) is a valid `uint32` value and is accepted without any bounds check in both `initialize` and `proposeOracleStopLossTimelock`.

Concrete arithmetic with `timelock = type(uint32).max`:

```
block.timestamp + type(uint32).max
= 1_753_000_000 + 4_294_967_295
= 6_047_967_295

uint32(6_047_967_295) = 6_047_967_295 mod 2^32
= 6_047_967_295 ŌłÆ 4_294_967_296
= 1_752_999_999          ŌåÉ one second in the PAST
```

`_requireElapsed` then checks:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

`executeAfter` is zero-extended to `uint256` for the comparison. `1_753_000_000 < 1_752_999_999` is **false**, so the revert is never triggered. Every proposal becomes immediately executable.

Neither `initialize` nor `proposeOracleStopLossTimelock` validates the timelock value:

```solidity

(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock accepted unconditionally
```

```solidity
// proposeOracleStopLossTimelock L78-84 ŌĆö no bounds check on newTimelock
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
}
```

### Impact Explanation

The stop-loss extension is the sole on-chain mechanism protecting LP shares from value erosion: it blocks swaps when per-share value at the oracle mid falls below the high-watermark floor. The timelock is the only window LPs have to exit before the admin can tighten or disable that floor.

Once the overflow is triggered the admin can, in a single block:

1. Call `proposeOracleStopLossDrawdown(pool, E6)` ŌĆö sets drawdown to 100 %, making `floorMultiplier = 0`, so `metric < 0` is never true and the stop-loss never fires.
2. Immediately call `executeOracleStopLossDrawdown(pool)` ŌĆö `_requireElapsed` passes because `executeAfter` is a past timestamp.
3. Repeat for decay (`proposeOracleStopLossDecay` / `executeOracleStopLossDecay`) and watermarks.

With the stop-loss silently disabled, swaps that drain LP value below the previously advertised floor proceed unchecked. LPs who entered the pool relying on the stop-loss guarantee suffer direct principal loss with no on-chain recourse.

This is an **admin-boundary break**: the pool admin is semi-trusted only within the timelock envelope; the overflow lets the admin collapse that envelope to zero without LPs having any reaction window.

---

### Likelihood Explanation

The attack requires the pool admin to:

1. Wait for the current timelock to elapse (one-time cost, bounded by the initial timelock).
2. Propose `newTimelock = type(uint32).max` ŌĆö a value that *appears* to be a 136-year lock, making it look like a conservative tightening to any LP monitoring the event log.
3. Execute the change after the current timelock.
4. Immediately collapse all stop-loss parameters.

Steps 2ŌĆō4 are atomic from the LP's perspective (no warning window after step 3). The deceptive appearance of `type(uint32).max` as an extremely long timelock makes the attack hard to detect from event data alone. Any pool admin who is willing to act maliciously after an initial trust period can execute this.

---

### Recommendation

1. **Bound the timelock at write time** in both `initialize` and `proposeOracleStopLossTimelock`. A safe upper bound is `365 days` (31 536 000 seconds), which is far below the overflow threshold and covers any realistic LP-protection window:

```solidity
uint256 internal constant MAX_TIMELOCK = 365 days;

function _validateTimelock(uint256 t) private pure {
    if (t > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(t);
}
```

2. **Perform the timestamp arithmetic in `uint256`** and only cast to `uint32` after the bound check, or store `executeAfter` as `uint64`/`uint256` to eliminate the truncation entirely.

3. **Add the same guard to `initialize`** alongside the existing `_validateDrawdown` and `_validateDecay` calls.
