### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract contains an unbounded loop that iterates over a pool member's entire `pool_member_epoch_balance` trace since their last claim checkpoint. A pool member who makes balance changes across many epochs without claiming rewards will accumulate an arbitrarily large trace. Eventually, `claim_rewards` will exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield with no recovery path.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    // ... compute rewards per balance segment ...
    entry_to_claim_from += 1;
}
```

The code itself acknowledges the loop is unbounded. [1](#0-0) 

The trace grows via `set_member_balance`, which is called by `enter_delegation_pool`, `add_to_delegation_pool`, and `exit_delegation_pool_intent`. The underlying `insert` logic appends a new checkpoint entry whenever the epoch key differs from the last entry — meaning each epoch in which a pool member changes their balance adds exactly one new entry to the trace. [2](#0-1) 

The `entry_to_claim_from` cursor is only advanced and persisted on a **successful** `claim_rewards` call: [3](#0-2) 

There is no partial-claim mechanism. If the loop runs out of gas, the transaction reverts, `entry_to_claim_from` is not updated, and the pool member is permanently unable to claim their rewards.

`calculate_rewards` is also called from the view function `pool_member_info_v1`, meaning even reading the member's state will eventually fail. [4](#0-3) 

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

A pool member who has accumulated N epochs of balance changes without claiming rewards will have N entries in their `pool_member_epoch_balance` trace that must all be processed in a single `claim_rewards` call. Once N is large enough to exceed the Starknet transaction gas limit, the member's accumulated STRK rewards are permanently locked — there is no subset-processing function and no administrative recovery path.

---

### Likelihood Explanation

**Medium.** The trace grows at most once per epoch per pool member (the `insert` deduplicates same-epoch writes). A pool member who calls `add_to_delegation_pool` or `exit_delegation_pool_intent` in each epoch and never calls `claim_rewards` will accumulate one entry per epoch. Starknet's per-transaction resource limits are finite; the code comment itself concedes the loop is unbounded. Over a sufficiently long protocol lifetime (months to years of active delegation with frequent balance changes and infrequent claiming), this threshold becomes reachable without any adversarial intent — it is a natural usage pattern.

---

### Recommendation

1. **Add a partial-claim function** analogous to the BarnBridge PR#37 fix: expose a public function that processes only a bounded number of `pool_member_epoch_balance` entries per call, advancing `entry_to_claim_from` incrementally and persisting the intermediate state.
2. **Enforce a maximum trace depth** by requiring pool members to claim rewards before their unclaimed trace exceeds a safe bound, or by automatically triggering a partial claim on each balance-change operation.
3. **Document a concrete safe upper bound** for the number of unclaimed epochs, and add a guard that reverts balance-change operations if the pool member's unclaimed trace would exceed that bound.

---

### Proof of Concept

1. Deploy the staking system and create a staker with a pool.
2. Create a pool member and call `enter_delegation_pool`.
3. For each of N epochs (e.g., N = 500):
   - Advance one epoch.
   - Call `add_to_delegation_pool` with a minimal amount (1 unit) — this appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`.
   - Do **not** call `claim_rewards`.
4. After N epochs, call `claim_rewards`. The `calculate_rewards` loop must iterate over all N entries. At a sufficient N, the call exceeds the Starknet transaction gas limit and reverts.
5. Observe that `entry_to_claim_from` remains at 0 (state not updated), and every subsequent `claim_rewards` attempt also reverts — the pool member's accumulated yield is permanently frozen.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
