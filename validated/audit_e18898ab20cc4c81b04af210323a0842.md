### Title
Mandatory `collectFees` pre-step in `setPoolAdminFees` and `setPoolProtocolFee` blocks fee governance and protocol-fee extraction when `adminFeeDestination` is USDC/USDT-blacklisted — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

Both `setPoolAdminFees` and `setPoolProtocolFee` unconditionally call `pool.collectFees(…, adminFeeDestination)` as a mandatory first step before updating fee configuration. If the `adminFeeDestination` is blacklisted by USDC or USDT (both in scope), the `safeTransfer` inside `collectFees` reverts, causing the entire fee-update call to fail. The same revert also blocks the permissionless `collectPoolFees` path, trapping all accrued protocol and admin fees inside the pool.

---

``` [2](#0-1) 

**Step 3 — `collectPoolFees` (permissionless) also routes through the same path:** [3](#0-2) 

**Step 4 — Inside `MetricOmmPool.collectFees`, tokens are transferred to `adminFeeDestination_` via `safeTransfer`:**

```solidity
// MetricOmmPool.sol lines 416-421
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);
}
``` [4](#0-3) 

**Step 5 — `transferToken0/1` uses `SafeERC20.safeTransfer`, which reverts on USDC/USDT blacklist:** [5](#0-4) 

If `adminFeeDestination` is blacklisted by USDC or USDT after pool creation, every call to `collectFees` reverts. Because `setPoolAdminFees`, `setPoolProtocolFee`, and `collectPoolFees` all invoke `collectFees` unconditionally before doing anything else, all three entry points are simultaneously bricked. The fee-update logic (the actual goal) never executes.

The pool admin can recover by calling `setPoolAdminFeeDestination`, which does **not** call `collectFees`: [6](#0-5) 

However, until the admin acts:
- The protocol owner (`onlyOwner`) cannot update protocol fees for the pool via `setPoolProtocolFee`.
- The pool admin cannot update admin fees via `setPoolAdminFees`.
- No one (permissionless path) can extract accrued fees via `collectPoolFees`.

---

### Impact Explanation

All accrued spread and notional fees remain locked inside the pool contract. The protocol owner is blocked from a core governance action (`setPoolProtocolFee`) by a condition entirely outside their control — the pool admin's fee destination being blacklisted. Fee rates cannot be adjusted for the affected pool until the pool admin changes the destination. If the pool admin is unresponsive or the admin key is lost, the block is permanent.

---

### Likelihood Explanation

USDC and USDT both maintain on-chain blacklists and are explicitly in scope per the contest rules. Any pool whose token0 or token1 is USDC/USDT and whose `adminFeeDestination` is later blacklisted (e.g., a sanctioned multisig or exchange address) triggers this condition. The blacklisting event is external and can occur post-deployment without any action by the protocol.

---

### Recommendation

Decouple fee collection from fee configuration updates. Two options:

1. **Separate the steps**: Remove the mandatory `collectFees` call from `setPoolAdminFees` and `setPoolProtocolFee`. Let callers invoke `collectPoolFees` independently before changing rates.

2. **Try/catch around collection**: Wrap the `collectFees` call in a try/catch so that a transfer failure is logged but does not block the fee-rate update:

```solidity
try IMetricOmmPoolCollectFees(pool).collectFees(...) {} catch {}
// proceed with fee config update regardless
```

---

### Proof of Concept

1. Deploy a pool with USDC as `token0` and any token as `token1`.
2. Set `adminFeeDestination` to address `A`.
3. Execute several swaps so spread and notional fees accumulate.
4. USDC blacklists address `A` (e.g., via regulatory action).
5. Call `collectPoolFees(pool)` → reverts inside `transferToken0(A, amount)`.
6. Call `factory.setPoolAdminFees(pool, newFee, 0)` as pool admin → same revert, fee update never applies.
7. Call `factory.setPoolProtocolFee(pool, newFee, 0)` as factory owner → same revert, protocol cannot adjust its own fee rate.
8. All accrued fees remain in the pool; fee governance for this pool is frozen until the pool admin calls `setPoolAdminFeeDestination`.
