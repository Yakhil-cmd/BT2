### Title
Unrestricted `update_rewards` with `disable_rewards: true` Allows Any Caller to Permanently Consume Per-Block Reward Slots Without Distributing Rewards - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in `staking.cairo` has no access control and accepts a caller-controlled `disable_rewards` boolean. Any unprivileged address can call `update_rewards(staker_address: any_valid_staker, disable_rewards: true)` to advance the global `last_reward_block` counter without distributing any rewards, permanently destroying that block's reward allocation and blocking all legitimate stakers from claiming rewards in the same block.

---

### Finding Description

`update_rewards` is the V3 consensus-rewards entry point. Its design enforces a strict one-call-per-block invariant via `last_reward_block`:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

Critically, `last_reward_block` is written **before** the `disable_rewards` guard is evaluated:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);   // ← slot consumed here

if disable_rewards || self.is_pre_consensus() {
    return;                                            // ← rewards silently skipped
}
``` [1](#0-0) 

The only gate on the function is `general_prerequisites()`, which only checks that the contract is not paused and the caller is not the zero address: [2](#0-1) 

There is no check that the caller is the block proposer, the staker themselves, or any privileged role. `disable_rewards` is a raw caller-supplied boolean: [3](#0-2) 

The result: any address can call `update_rewards(staker_address: <any active staker>, disable_rewards: true)` in any block, advance `last_reward_block` to that block, and cause the function to return without distributing rewards. Every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`. The block's reward allocation is permanently destroyed — it is not accumulated or deferred.

---

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users or protocol.**

An attacker with no financial gain can:

1. Call `update_rewards(staker_address: <any valid staker>, disable_rewards: true)` once per block.
2. Consume the per-block reward slot without distributing rewards.
3. Prevent every legitimate staker from claiming block rewards for that block.
4. Repeat indefinitely at the cost of one transaction per block.

Block rewards that are not distributed are not accumulated; they are simply lost. Sustained execution permanently freezes unclaimed yield across all stakers, escalating the impact toward High.

---

### Likelihood Explanation

**High.** The function is fully public, requires no tokens, no privileged role, and no special knowledge. The only cost is gas for one transaction per block. On Starknet, transaction costs are low enough to make this economically viable as a sustained griefing campaign. No front-running is required — the attacker simply calls the function at the start of each block before any legitimate staker does.

---

### Recommendation

1. **Restrict the caller**: Add an access control check so that only the block proposer (or a designated rewards manager role) can call `update_rewards`.
2. **Alternatively, reorder the write**: Move `self.last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard, so that a no-op call does not consume the per-block slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number); // moved here
// ... distribute rewards
```

---

### Proof of Concept

1. Block `N` begins. `last_reward_block` = `N-1`.
2. Attacker calls `update_rewards(staker_address: alice, disable_rewards: true)`.
   - `current_block_number (N) > last_reward_block (N-1)` → assertion passes.
   - `last_reward_block` is written to `N`. [4](#0-3) 
   - `disable_rewards == true` → early return, no rewards distributed. [5](#0-4) 
3. Alice (the legitimate block proposer) calls `update_rewards(staker_address: alice, disable_rewards: false)`.
   - `current_block_number (N) > last_reward_block (N)` → **false** → reverts with `REWARDS_ALREADY_UPDATED`. [6](#0-5) 
4. Block `N`'s rewards are permanently lost.
5. Attacker repeats in block `N+1`, `N+2`, … indefinitely.
