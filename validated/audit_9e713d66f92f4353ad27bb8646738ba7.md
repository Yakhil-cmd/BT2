### Title
Unprotected Legacy Transactions (v=27/28) Accepted Without Chain ID — Cross-Chain Replay Attack Possible - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS's bootloader explicitly accepts pre-EIP-155 "unprotected" legacy transactions (those with `v = 27` or `v = 28`). Because the signing hash for these transactions is computed without including the chain ID, a valid transaction on ZKsync OS is simultaneously valid on Ethereum mainnet and any other EVM-compatible chain — and vice versa. An attacker who observes an unprotected legacy transaction on one chain can replay it on ZKsync OS (or replay a ZKsync OS transaction elsewhere), draining the sender's funds.

---

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, the code branches on whether the signature is EIP-155-protected:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy — NO chain ID in hash
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
} else {
    // EIP-155 protected: chain_id is mixed into the hash
    ...
}
``` [1](#0-0) 

`is_eip155()` returns `false` exactly when `v == 27 || v == 28`: [2](#0-1) 

The caller in `RlpEncodedTxInner::parse_and_compute_signed_hash` stores such transactions as the `Self::Legacy` variant — there is no rejection path: [3](#0-2) 

The `Self::Legacy` variant is then processed through the full validation and execution pipeline. The `InvalidTransaction::InvalidChainId` error is only raised for typed transactions (EIP-2930, EIP-1559, EIP-4844, EIP-7702); it is never raised for unprotected legacy transactions. [4](#0-3) 

---

### Impact Explanation

A user who signs a legacy transaction with `v = 27` or `v = 28` on ZKsync OS produces a signature that is equally valid on Ethereum mainnet, any other EVM chain, or any future ZKsync chain with a different chain ID. An attacker who observes such a transaction on one chain can submit the identical raw bytes to ZKsync OS (or vice versa). If the sender's nonce on the target chain matches the nonce in the transaction — which is trivially true for fresh accounts (nonce = 0) and possible for accounts with matching nonce histories — the transaction executes, transferring funds or deploying contracts without the user's intent on that chain.

The impact is direct loss of user funds (ETH/token transfers) or unauthorized contract deployments from the victim's account.

---

### Likelihood Explanation

- Many legacy wallets, hardware wallets in compatibility mode, and pre-EIP-155 tooling still produce `v = 27/28` signatures.
- Users who share a private key across Ethereum mainnet and ZKsync OS (the common case) are exposed.
- For new accounts (nonce = 0), no nonce alignment is required — the replay succeeds immediately.
- The attacker's only cost is the gas fee on the target chain; no privileged access is needed.

---

### Recommendation

Reject unprotected legacy transactions at the parsing stage. Add an explicit check after the `is_eip155()` branch:

```rust
if legacy_signature.is_eip155() == false {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

Alternatively, add a dedicated `InvalidTransaction::UnprotectedLegacyTransaction` variant and reject it in `parse_and_compute_signed_hash` before the `Self::Legacy` arm is ever reached. This mirrors the approach already taken for all typed transaction formats, which unconditionally require a matching chain ID. [5](#0-4) 

---

### Proof of Concept

1. On Ethereum mainnet, user Alice sends a legacy ETH transfer with `v = 27` (unprotected), nonce = 0, to address Bob, value = 1 ETH.
2. Attacker Eve observes the raw RLP-encoded transaction bytes on Ethereum.
3. Alice also holds ETH on ZKsync OS at the same address (same private key), with nonce = 0.
4. Eve submits the identical raw bytes to ZKsync OS.
5. `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` computes the signing hash without chain ID (the `is_eip155() == false` branch), recovers Alice's address correctly, and the transaction passes signature validation.
6. Nonce check passes (both nonces are 0).
7. ZKsync OS executes the transfer: Alice loses 1 ETH on ZKsync OS without having authorized it there.

The reverse direction (ZKsync OS → Ethereum mainnet) is equally valid.
