### Title
ETH Sent to `tick()` Is Permanently Frozen When No Mint Requests Are Pending - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is `payable` and accepts ETH to cover L1→L2 messaging fees. When there are no pending L2→L1 mint request messages (`amountToMint == 0`), the entire fee-spending block is skipped and `msg.value` is neither used nor returned. There is no ETH withdrawal or recovery mechanism anywhere in the contract or its parent contracts, so the funds are permanently frozen.

---

### Finding Description

`tick()` is declared `external payable` with no access control: [1](#0-0) 

The ETH is only consumed inside the `if (amountToMint > 0)` branch: [2](#0-1) 

When `requiredMinting()` returns `amountToMint == 0` — which happens whenever there are no pending L2→L1 messages from `mintRequestSource` — the function returns silently after the check, leaving `msg.value` untouched in the contract: [3](#0-2) 

A `grep` across all L1 Solidity files confirms there is no `receive()`, `fallback()`, `withdraw`, or any ETH-recovery function anywhere in the contract hierarchy. The ETH has no exit path.

---

### Impact Explanation

Any caller who sends ETH with `tick()` when `amountToMint == 0` permanently loses that ETH to the contract balance. Because there is no recovery mechanism, this constitutes **permanent freezing of the caller's funds**.

This maps to the allowed impact: **High — Temporary/Permanent freezing of funds**.

---

### Likelihood Explanation

- `tick()` is a public, permissionless function — any EOA or contract can call it.
- The natural operational pattern is to call `tick()` with ETH to pay for the two L1→L2 messages. A caller has no on-chain guarantee that messages are still pending by the time their transaction lands (another `tick()` call in the same block could consume them first).
- A race condition between two concurrent `tick()` callers means the second caller's ETH is frozen with certainty.
- Even a single caller who calls `tick()` speculatively (without first reading `requiredMinting()` off-chain) loses their ETH.

---

### Recommendation

Return `msg.value` to the caller when the minting branch is not entered:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Return unused ETH to the caller
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, remove `payable` from `tick()` and require callers to send ETH only through a separate, guarded entry point, or have the contract hold a pre-funded ETH balance for fees.

---

### Proof of Concept

1. Alice calls `tick{value: 0.01 ether}()`.
2. Concurrently, Bob's `tick()` (sent earlier, no ETH) is mined first and consumes all pending L2→L1 mint messages.
3. When Alice's transaction executes, `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. Alice's `0.01 ETH` sits in `RewardSupplier`'s balance with no withdrawal path.
6. Repeat for every subsequent caller who sends ETH to `tick()` when no messages are pending. [3](#0-2)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
