### Title
Presignature Replay via Non-Consuming `rerandomize_presign` Enables ECDSA Nonce Reuse and Secret Key Recovery - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both OT-based and Robust ECDSA `PresignOutput` structs derive `Clone` and `Serialize`/`Deserialize`, and `rerandomize_presign()` accepts `PresignOutput` by shared reference (`&PresignOutput`), leaving the presignature unconsumed. A malicious coordinator can therefore induce honest participants to rerandomize and sign with the same presignature across two distinct signing sessions (different `msg_hash` values), producing two sets of signature shares whose nonces are multiplicatively related. This is sufficient for standard ECDSA nonce-reuse secret key recovery.

---

### Finding Description

**Root cause — `PresignOutput` is not consumed after rerandomization.**

In `src/ecdsa/ot_based_ecdsa/mod.rs`, `PresignOutput` is declared as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
```

`rerandomize_presign` takes it by shared reference:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { … }
```

The same pattern exists in `src/ecdsa/robust_ecdsa/mod.rs` for the Robust ECDSA `PresignOutput` (`big_r`, `c`, `e`, `alpha`, `beta`).

The `sign()` function in both schemes takes `RerandomizedPresignOutput` by value (consuming it), which appears to enforce single use. However, because `rerandomize_presign` does not consume the underlying `PresignOutput`, a participant can call `rerandomize_presign` again on the same `PresignOutput` with a different `RerandomizationArguments` (different `msg_hash`) and produce a fresh `RerandomizedPresignOutput` for a second signing session. The library provides no flag, counter, or type-level mechanism to detect or prevent this.

The security documentation explicitly mandates: *"Never reuse a presignature, even across failed, aborted, or partially completed signing sessions"* and warns that reuse enables secret key recovery via multiplicatively related nonces. No enforcement exists in the production code.

**Exploit path — malicious coordinator:**

1. All `N = 2t+1` participants (including the coordinator) complete presigning. Each holds their own `PresignOutput`.
2. Coordinator initiates signing session 1 with `msg_hash_1`. Each participant calls `rerandomize_presign(&presign_output, args_1)` → `rerandomized_1`, then `sign(…, rerandomized_1, msg_hash_1)`. The coordinator collects all `s_i` shares but deliberately aborts (never returns the final signature).
3. Coordinator initiates signing session 2 with `msg_hash_2 ≠ msg_hash_1`. Each honest participant, having no record that their `PresignOutput` was already used, calls `rerandomize_presign(&presign_output, args_2)` → `rerandomized_2`, then `sign(…, rerandomized_2, msg_hash_2)`. The coordinator collects all `s_i` shares.
4. The coordinator now holds two complete sets of signature shares. The underlying nonce `k` (or `alpha`/`beta`/`c`/`e` in Robust ECDSA) is the same in both sessions, scaled by `delta_1^{-1}` and `delta_2^{-1}` respectively. Since `delta_i = HKDF(…, msg_hash_i, …)` is publicly computable, the ratio `delta_1 / delta_2` is known. Standard ECDSA nonce-reuse algebra recovers the secret key.

---

### Impact Explanation

The attack yields full recovery of the aggregate ECDSA secret key from two signing sessions that reuse the same presignature nonce material. This falls squarely under:

> **Critical: Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets.**

The documentation itself confirms: *"the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks"* and *"a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."*

---

### Likelihood Explanation

The coordinator is one of the `2t+1` signing participants. The security model permits up to `t` malicious parties; a malicious coordinator is therefore an in-scope threat. The attack requires only two sequential signing sessions (both initiated by the coordinator), no external resources, no cryptographic breaks, and no privileged access beyond the coordinator role. Honest participants have no observable signal that their presignature was already consumed in a prior session.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput`** in both `src/ecdsa/ot_based_ecdsa/mod.rs` and `src/ecdsa/robust_ecdsa/mod.rs`. This prevents callers from copying the presignature.
2. **Change `rerandomize_presign` to consume `PresignOutput` by value** (`presignature: PresignOutput` instead of `presignature: &PresignOutput`). Rust's ownership system then statically enforces single use: once rerandomized, the `PresignOutput` is gone and cannot be rerandomized again.
3. Apply the same treatment to the FROST `PresignOutput` in `src/frost/mod.rs`, which also derives `Clone` and contains secret `SigningNonces`.

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }

signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the

few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
