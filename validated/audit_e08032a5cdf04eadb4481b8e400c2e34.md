### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Accounts with Pending Deploy - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally bypasses the `__validate__` entry point (signature verification) for any invoke transaction with `nonce=1` targeting an account whose address appears in the mempool or a recent block. Because the check does not verify that the caller is the legitimate account owner, an unprivileged attacker can inject an unsigned invoke for any victim account that has a pending `deploy_account` transaction, causing the gateway to admit an invalid transaction and reject the victim's legitimate invoke.

### Finding Description

`extract_state_nonce_and_run_validations` is the single entry point for stateful gateway validation:

```
get_nonce_from_state → run_pre_validation_checks → run_validate_entry_point
```

Inside `run_pre_validation_checks`, the call sequence is:

1. `validate_state_preconditions` — checks resource bounds and nonce range
2. `validate_by_mempool` — checks duplicate hash / nonce via `mempool.validate_tx`
3. `skip_stateful_validations` — decides whether to skip `__validate__`

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
```

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or any committed block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, so `StatefulValidator::perform_validations` returns immediately after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point:

```rust
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());   // ← signature never checked
}
```

The invariant the code relies on is stated in the comment: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This invariant is broken. The check does not verify **who** submitted the transaction that caused the account to appear in the mempool. A victim's `deploy_account` causes `account_tx_in_pool_or_recent_block` to return `true` for the victim's address. Any third party can then submit an invoke with `nonce=1` for that address, and the gateway will skip signature verification entirely.

The only remaining gate — `verify_can_pay_committed_bounds` inside `perform_pre_validation_stage` — checks the fee-token balance at the account address. Because Starknet users must fund their account address **before** deploying it (the fee token balance is stored in the ERC-20 contract keyed by address, independent of deployment), this check passes for any normally-funded account.
