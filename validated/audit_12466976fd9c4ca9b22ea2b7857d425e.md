### Title
Missing Access Control on `update_rewards` Allows Anyone to Permanently Freeze Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `StakingRewardsManagerImpl` is documented to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards=true` to consume the per-block reward slot without distributing rewards, permanently destroying that block's yield for all stakers and delegators.

### Finding Description
The spec for `update_rewards` explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation in `src/staking/staking.cairo` performs no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker existence checks ...
    self.last_reward_block.write(current_block_number);   // consumes the slot
    if disable_rewards || self.is_pre_consensus() {
        return;                                            // exits without distributing
    }
    // ... reward distribution ...
}
```

The only guard is `REWARDS_ALREADY_UPDATED`, which checks that `current_block_number > last_reward_block`. Once any caller writes `last_reward_block = current_block_number`, the sequencer's legitimate call for that block will revert with `REWARDS_ALREADY_UPDATED`.

An attacker calls `update_rewards(any_active_staker, disable_rewards=true)` at the start of every block. The function:
1. Passes the block-number guard.
2. Writes `last_reward_block = current_block_number`.
3. Returns early — zero rewards distributed.

The sequencer's subsequent call for the same block fails. The block's rewards are permanently unclaimable.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

Every block's STRK (and BTC) rewards that should flow to stakers and delegators are permanently destroyed. No staker or delegator can ever claim those rewards; they are not deferred, they are simply never credited. Sustained griefing eliminates all protocol yield.

### Likelihood Explanation
**High.** The entry point is fully public, requires no tokens, no privileged role, and no setup beyond knowing any active staker address (which is public on-chain). The attack is repeatable every block at negligible cost (one transaction per block). There is no economic barrier.

### Recommendation
Add an explicit sequencer-only guard at the top of `update_rewards`, consistent with the spec. For example, store the sequencer address at construction time and assert:

```cairo
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

Alternatively, if the sequencer address is not known at deploy time, use a role-based check (e.g., `SEQUENCER_ROLE`) via the existing `RolesComponent`.

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker observes any active staker address `S`.
3. At block `N`, attacker calls:
   ```
   staking.update_rewards(staker_address=S, disable_rewards=true)
   ```
4. `last_reward_block` is written to `N`; no rewards are distributed.
5. Sequencer attempts `update_rewards(S, false)` in the same block → reverts with `REWARDS_ALREADY_UPDATED`.
6. Block `N`'s rewards are permanently lost.
7. Repeat for every block.
