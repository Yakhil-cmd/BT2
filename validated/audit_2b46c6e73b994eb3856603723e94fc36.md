### Title
Signing Participant Set Not Verified Against Rerandomization Binding, Enabling Split-View Key Extraction — (`src/ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The `RerandomizationArguments` struct binds a presignature to a specific signing context including the `participants` list via HKDF. However, `RerandomizedPresignOutput` does not store the `participants` it was bound to, and `sign()` accepts a `participants` argument that is never verified to match the one used during rerandomization. A malicious coordinator can present different participant lists to different signers, causing them to compute different `delta` values from the same presignature. This creates multiplicatively related nonces across two signing sessions, enabling full secret key extraction via a standard ECDSA nonce-reuse attack.

---

### Finding Description

**Phase 1 — Rerandomization binding includes `participants`:**

In `src/ecdsa/mod.rs`, `RerandomizationArguments::derive_randomness()` computes `delta` via HKDF over `(pk, tweak, msg_hash, big_r, participants, entropy)`:

```rust
for participant in self.participants.participants() {
    concatenation.extend_from_slice(&participant.bytes());
}
let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);
hk.expand(&concatenation, &mut okm)...
``` [1](#0-0) 

The `participants` field is therefore a cryptographic input to `delta`. Two calls with different `participants` produce different `delta` values, yielding different rerandomized nonces `R' = delta * R`.

**Phase 2 — `RerandomizedPresignOutput` does not store the bound `participants`:**

The output struct for both OT-based and Robust ECDSA contains only `(big_r, alpha/k, beta/sigma, e)` — no record of which `participants` were used to derive `delta`:

```rust
pub struct RerandomizedPresignOutput {
    big_r: AffinePoint,
    e: Scalar,
    alpha: Scalar,
    beta: Scalar,
}
``` [2](#0-1) 

**Phase 3 — `sign()` accepts an independent `participants` argument with no consistency check:**

`robust_ecdsa::sign::sign()` takes `participants: &[Participant]` as a free parameter. It enforces `len == 2t+1` and `msg_hash != 0`, but never checks that `participants` matches the list embedded in the rerandomization that produced the `RerandomizedPresignOutput`:

```rust
pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<...>
``` [3](#0-2) 

The same gap exists in `ot_based_ecdsa::sign::sign()`. [4](#0-3) 

**The security documentation explicitly identifies this as a required invariant that the code does not enforce:**

> "Ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers." [5](#0-4) 

---

### Impact Explanation

A malicious coordinator runs two signing sessions against the same presignature:

- **Session A**: presents `participants = [P1, P2, P3]` to all signers → each signer computes `delta_A = HKDF(..., [P1,P2,P3], ...)` → nonce `R_A = delta_A * R` → valid signature `(R_A, s_A)`
- **Session B**: presents `participants = [P1, P2, P4]` to all signers → each signer computes `delta_B = HKDF(..., [P1,P2,P4], ...)` → nonce `R_B = delta_B * R` → valid signature `(R_B, s_B)`

Since `R_A = delta_A * R` and `R_B = delta_B * R`, the relationship `R_A = (delta_A / delta_B) * R_B` is known to the coordinator (both deltas are derived from public inputs). This is ECDSA nonce reuse with a known scalar relationship. Standard algebraic recovery then yields the secret key `x`.

The security documentation confirms this exact attack class:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [6](#0-5) 

**Impact: Critical** — full extraction of the aggregate ECDSA secret key.

---

### Likelihood Explanation

The attack requires a malicious coordinator, which is an explicitly modeled adversary in this library (the coordinator role is a designated participant in every signing call). The coordinator controls what `participants` list is presented to each signer during rerandomization setup. No out-of-band consensus mechanism is enforced by the library. The attack requires two signing sessions using the same presignature, which the library also does not prevent at the code level (presignature non-reuse is documented as a caller responsibility).

**Likelihood: Medium** — requires a malicious coordinator and presignature reuse, both of which are realistic in deployments where the coordinator is a semi-trusted orchestration node.

---

### Recommendation

1. **Store the bound `participants` inside `RerandomizedPresignOutput`** so that `sign()` can verify consistency:

```rust
pub struct RerandomizedPresignOutput {
    big_r: AffinePoint,
    e: Scalar,
    alpha: Scalar,
    beta: Scalar,
    bound_participants: ParticipantList, // add this
}
```

2. **In `sign()`, assert that `participants == presignature.bound_participants`** before proceeding. This mirrors the existing `big_r` consistency check in `rerandomize_presign()`:

```rust
if presignature.big_r != args.big_r {
    return Err(ProtocolError::IncompatibleRerandomizationInputs);
}
``` [7](#0-6) 

Apply the same pattern for `participants` in `sign()`.

3. Apply the same fix to `ot_based_ecdsa::RerandomizedPresignOutput` and its `sign()` function.

### Proof of Concept

```
Setup: N=5, t=2, so 2t+1=5 participants [P1..P5], presignature (R, shares).

Step 1: Coordinator rerandomizes for Session A:
  args_A = RerandomizationArguments { participants: [P1,P2,P3,P4,P5], msg_hash: h, ... }
  delta_A = HKDF(entropy; pk, tweak, h, R, [P1,P2,P3,P4,P5])
  Each signer computes RerandomizedPresignOutput with delta_A → nonce R_A = delta_A * R

Step 2: Coordinator rerandomizes for Session B (same presignature, different participants):
  args_B = RerandomizationArguments { participants: [P1,P2,P3,P4,P6], msg_hash: h, ... }
  delta_B = HKDF(entropy; pk, tweak, h, R, [P1,P2,P3,P4,P6])  ← different!
  Each signer computes RerandomizedPresignOutput with delta_B → nonce R_B = delta_B * R

Step 3: sign() in both sessions accepts the coordinator-supplied participants list
  without checking it matches the list used in rerandomize_presign().
  Both sessions complete, yielding (R_A, s_A) and (R_B, s_B).

Step 4: Coordinator knows delta_A, delta_B (both derived from public inputs).
  R_A = delta_A * R, R_B = delta_B * R → R_A = (delta_A/delta_B) * R_B
  Standard ECDSA related-nonce key recovery:
    s_A = (h + r_A * x) / (delta_A * k)
    s_B = (h + r_B * x) / (delta_B * k)
  Solving for x yields the master secret key.
```
