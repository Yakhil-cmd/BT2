### Title
Unauthorized Encryption Key Overwrite via Missing Signer Check in `RegisterEncryptionKey` - (`x/e2ee/keeper/keeper.go`)

### Summary

`Keeper.RegisterEncryptionKey` writes `req.Address → req.Key` to the KV store with no check that `req.Address` equals the transaction signer. Any unprivileged account can overwrite any other account's registered age public key.

### Finding Description

`Keeper.RegisterEncryptionKey` unconditionally stores the key for whatever address is in the message: [1](#0-0) 

The inner `registerEncryptionKey` helper only validates that `address` is a valid bech32 string before writing to the store: [2](#0-1) 

`ValidateBasic` on the message validates address format and key format, but never compares `Address` to the signer: [3](#0-2) 

There is no `GetSigners()` implementation that enforces `req.Address == signer`, and no signer check anywhere in the keeper. The CLI happens to populate `Address` from the from-address, but this is a client-side convention with no on-chain enforcement: [4](#0-3) 

### Impact Explanation

An attacker submits a valid, fee-paying transaction `MsgRegisterEncryptionKey{Address: victim_bech32, Key: attacker_age_pubkey}`. The keeper writes `attacker_age_pubkey` under `victim_bech32` in the KV store. All subsequent callers of the `Key` or `Keys` query for the victim's address receive the attacker's public key. Any party that then encrypts a message to the victim using the registered key will produce ciphertext decryptable only by the attacker. This is a permanent, on-chain corruption of e2ee key state with direct security impact, matching the allowed High impact category: *"Corruption of … e2ee key/message state with direct security impact."*

### Likelihood Explanation

The attack requires only a funded account and a single transaction. No special privileges, leaked keys, or external assumptions are needed. The message is accepted by `ValidateBasic` as long as `victim_bech32` is a valid address and `attacker_age_pubkey` is a valid age X25519 recipient string.

### Recommendation

In `Keeper.RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    signers := sdkCtx.TxSigner() // or retrieve via msg server signer extraction
    // compare req.Address bytes to the signer bytes; return error if mismatch
    ...
}
```

Alternatively, implement `GetSigners() []sdk.AccAddress` on `MsgRegisterEncryptionKey` returning the decoded `Address` bytes, which the Cosmos SDK ante-handler will then enforce must be present in the transaction's signer set — and add a keeper-level check for defense in depth.

### Proof of Concept

```go
// unit test sketch
func TestRegisterEncryptionKey_SignerMismatch(t *testing.T) {
    k, ctx := setupKeeper(t)

    victim := sdk.AccAddress([]byte("victim______________"))
    attacker := sdk.AccAddress([]byte("attacker____________"))

    // victim registers their own key first
    _, err := k.RegisterEncryptionKey(ctx, &types.MsgRegisterEncryptionKey{
        Address: victim.String(),
        Key:     validAgeKey1,
    })
    require.NoError(t, err)

    // attacker (ctx signer = attacker) overwrites victim's key
    // ctx does NOT contain victim as signer — no check exists
    _, err = k.RegisterEncryptionKey(ctx, &types.MsgRegisterEncryptionKey{
        Address: victim.String(),
        Key:     attackerAgeKey,
    })
    require.NoError(t, err) // succeeds — this is the bug

    resp, _ := k.Key(ctx, &types.KeyRequest{Address: victim.String()})
    require.Equal(t, attackerAgeKey, resp.Key) // victim's key is now attacker's
}
```

}
