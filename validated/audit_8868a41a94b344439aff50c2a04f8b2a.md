### Title
Missing `l2_gas_consumed` Field Causes `overall_fee` to Be Irreconcilable with Documented Formula in `FeeEstimation` — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`tx_execution_output_to_fee_estimation` constructs a `FeeEstimation` whose `overall_fee` is taken from `receipt.fee`, which is computed over the full `GasVector` (L1 gas + L1 data gas + **L2 gas**). However, the struct exposes only `gas_consumed` (L1 gas) and `data_gas_consumed` (L1 data gas), with no `l2_gas_consumed` field. The struct's own doc comment asserts the invariant `overall_fee = gas_consumed * gas_price + data_gas_consumed * data_gas_price`, which is provably false for any v3 invoke transaction that consumes nonzero L2 gas under a nonzero `l2_gas_price`.

---

### Finding Description

`FeeEstimation` is defined as:

```rust
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
pub overall_fee: Fee,
``` [1](#0-0) 

`tx_execution_output_to_fee_estimation` populates it as:

```rust
let gas_vector = tx_execution_output.execution_info.receipt.gas;
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    ...
})
``` [2](#0-1) 

`receipt.fee` is computed in `TransactionReceipt::from_params` by calling `get_fee_by_gas_vector` with the **full** `GasVector`:

```rust
let gas = tx_resources.to_gas_vector(..., &gas_mode);
let fee = tx_context.tx_info.get_fee_by_gas_vector(&block_info, gas, effective_tip);
``` [3](#0-2) 

`GasVector::cost` sums all three components:

```rust
for (gas, price, resource) in [
    (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
    (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
] { ... }
``` [4](#0-3) 

For v3 transactions using `AllResourceBounds`, the gas mode is `GasVectorComputationMode::All`, which produces a nonzero `l2_gas` component. Regression tests confirm this concretely — a v3 invoke with KZG DA produces `l2_gas: 653175` alongside `l1_gas: 16023`: [5](#0-4) 

The developers acknowledge this gap with a TODO:

> `TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.` [6](#0-5) 

---

### Impact Explanation

Any client (wallet, dApp, block explorer) that calls `starknet_estimateFee` or `starknet_simulateTransactions` for a v3 invoke transaction receives a `FeeEstimation` where:

- `overall_fee` = `l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * l2_gas_price`
- But the documented formula yields: `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`

These two values differ by exactly `l2_gas * l2_gas_price`. The RPC response is authoritative-looking but internally inconsistent: the returned fields cannot reconstruct `overall_fee`. Clients that verify or display the fee breakdown will compute a wrong value. This falls under **High — RPC fee estimation returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

This is triggered by any v3 invoke transaction (the standard transaction type for STRK-fee-paying accounts) submitted to `starknet_estimateFee` or `starknet_simulateTransactions`. No special attacker capability is required — any unprivileged user submitting a normal v3 transaction triggers the inconsistency. The L2 gas component is nonzero for all such transactions in `GasVectorComputationMode::All`.

---

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`. Update the doc comment to reflect the correct three-term formula: `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price + l2_gas_consumed * l2_gas_price`.

---

### Proof of Concept

```rust
// In tx_execution_output_to_fee_estimation, with a v3 tx that consumed L2 gas:
let est = tx_execution_output_to_fee_estimation(&output, &block_context).unwrap();

let reconstructed = est.gas_consumed.to_u128().unwrap() * est.l1_gas_price.get()
    + est.data_gas_consumed.to_u128().unwrap() * est.l1_data_gas_price.get();

// overall_fee includes l2_gas * l2_gas_price; reconstructed does not.
assert_ne!(est.overall_fee.0, reconstructed); // PASSES — invariant is violated
```

The regression test at `crates/blockifier/src/transaction/transactions_test.rs` already shows `l2_gas: 653175` for a v3 KZG-DA transaction. With any nonzero `l2_gas_price` (standard for STRK-fee blocks), the discrepancy is `653175 * l2_gas_price` Fri.
