### Title
Unauthenticated Invoke Transaction Bypasses Signature Validation via `skip_stateful_validations` for Any Account with a Pending Deploy-Account - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature check) for any invoke transaction with nonce=1 targeting an account whose address appears in the mempool or a recent block. Because the check does not verify that the submitter of the invoke is the same party who submitted the `deploy_account`, any external observer can inject an invoke with a garbage signature for a victim's freshly-deploying account, and the gateway will admit it without signature verification.

### Finding Description
`skip_stateful_validations` (lines 429–461 of `stateful_transaction_validator.rs`) returns `true` — meaning "skip `__validate__`" — when all three conditions hold:

1. The transaction is `ExecutableTransaction::Invoke`.
2. The transaction nonce equals `Nonce(Felt::ONE)`.
3. The account's on-chain nonce is `Nonce(Felt::ZERO)` (not yet deployed).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

The fourth condition is satisfied by **any** address that has **any** transaction in the mempool or any committed block — it is not restricted to the party who submitted the `deploy_account`. The comment in the code acknowledges this looseness: "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

This causes `StatefulValidator::perform_validations` to skip the `__validate__` call entirely: [4](#0-3) 

The full gateway admission flow (`add_tx_inner`) runs stateless validation, converts the transaction, then calls `extract_state_nonce_and_run_validations` — which internally calls `run_pre_validation_checks` → `skip_stateful_validations` → `run_validate_entry_point`. No other step verifies the signature when `skip_validate=true`. [5](#0-4) 

**Attack scenario:**

1. Victim submits a `deploy_account` for address `A`. It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker observes the pending `deploy_account` and immediately submits `Invoke(sender=A, nonce=1, signature=<garbage>)`.
3. The gateway checks: tx is Invoke, nonce=1, on-chain nonce of `A` is 0, `A` is in the mempool → `skip_stateful_validations` returns `true`.
4. `run_validate_entry_point` is called with `skip_validate=true`, so `__validate__` (ECDSA check) is never executed.
5. `validate_by_mempool` runs before the skip check and only verifies nonce range / duplicate hash — it does not verify the signature. [6](#0-5) 

6. The attacker's unsigned invoke is admitted to the mempool at nonce=1 for address `A`.
7. When the victim subsequently submits their legitimate invoke with nonce=1, the mempool rejects it as `DuplicateNonce` (or forces the victim to pay a higher fee for fee escalation). [7](#0-6) 

### Impact Explanation
This is a **High** impact finding matching the criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The gateway admits a transaction with an invalid (attacker-controlled) signature into the mempool. The attacker can:
- Block the victim's first post-deployment invoke by occupying nonce=1 with an unsigned transaction.
- Force the victim to pay a higher fee (fee escalation) to displace the attacker's transaction.
- Cause the victim's account to execute a reverted transaction as its first action, wasting gas.

The attacker's invalid invoke will eventually revert during batcher execution (since the batcher runs `__validate__` with full execution flags), but the admission-level invariant — *only transactions with valid signatures are accepted* — is broken at the gateway.

### Likelihood Explanation
**High.** The attack requires only:
- Monitoring the public mempool for `deploy_account` transactions (no privileged access needed).
- Submitting an invoke with nonce=1 for the target address with any signature before the victim does.

The default `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, so the window is exactly nonce=1 — the victim's first invoke after deployment, which is the most critical transaction. [8](#0-7) 

### Recommendation
The `skip_stateful_validations` check must be restricted to the party who submitted the `deploy_account`. Two options:

1. **Require the deploy_account transaction hash**: The gateway should only skip validation if the caller provides the hash of the pending `deploy_account` transaction (as `PyValidator::should_run_stateful_validations` does via `deploy_account_tx_hash`), and verify that hash belongs to the same sender address. This prevents third parties from exploiting the skip. [9](#0-8) 

2. **Verify the deploy_account is from the same sender**: Query the mempool to confirm that the pending transaction for the address is specifically a `deploy_account` (not just any transaction), and that the invoke's signature is consistent with the class hash declared in that `deploy_account`.

gateway.add_tx(attacker_invoke, None).await.unwrap(); // succeeds

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
