Audit Report

## Title
Rate-Limit Charged Before Cycle Delivery Confirmed, Enabling Hourly Global Minting DoS — (`rs/nns/cmc/src/main.rs`, `rs/nns/cmc/src/limiter.rs`)

## Summary
`ensure_balance` permanently increments the global per-hour minting rate limiter and calls `ic0_mint_cycles128` before the management canister `deposit_cycles` call is made. If that inter-canister call fails, the ICP is refunded to the caller but the limiter increment is never rolled back. An unprivileged attacker can exhaust the 150P-cycles/hour global limit in a single transaction, recover their ICP (minus a small fee), and block all legitimate cycle minting for up to one hour.

## Finding Description
In `deposit_cycles` (`rs/nns/cmc/src/main.rs`, lines 2110–2138), `ensure_balance` is called unconditionally before the `await` point:

```rust
if mint_cycles {
    ensure_balance(cycles, limiter_to_use)?;   // (A) limiter charged, cycles minted into CMC
}
let res: CallResult<()> = ic_cdk::api::call::call_with_payment128(
    candid::Principal::management_canister(),
    METHOD_DEPOSIT_CYCLES, ..., u128::from(cycles),
).await;                                        // (B) yield point — may fail
res.map_err(...)?;                              // (C) propagates error upward
```

Inside `ensure_balance` (lines 2306–2325), the limiter is charged and cycles are minted into the CMC's own balance before returning:

```rust
with_state_mut(|state| {
    limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;  // limiter charged
    state.total_cycles_minted += cycles_to_mint;
    Ok::<_, String>(())
})?;
let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);               // cycles minted
```

In `limiter.rs` (lines 54, 77–78), `check_and_add_cycles` calls `self.add(now, cycles_to_mint)` which increments `total_count` with no compensating decrement path anywhere in the codebase.

On the IC, `await` is a commit point: state mutations before it are durable. If step (B) returns an error (e.g., target canister does not exist, is stopped, or is frozen), step (C) propagates the error to `process_top_up` (lines 1999–2011), which calls `refund_icp` and returns the ICP to the caller. The limiter's `total_count` remains permanently incremented.

The same pattern exists in `do_mint_cycles` (line 2151), meaning `notify_mint_cycles` is also affected.

## Impact Explanation
This is an application/platform-level DoS matching the High impact class. The `base_limiter` is shared across all callers. Once exhausted (150 × 10¹⁵ cycles), every subsequent call to `notify_top_up`, `notify_mint_cycles`, and `notify_create_canister` that requires new minting will fail with "More than N cycles have been minted in the last 3600 seconds." The CMC is a core NNS canister; blocking cycle minting for up to one hour disrupts all users attempting to top up canisters or create new canisters via ICP→cycles conversion across the entire Internet Computer.

## Likelihood Explanation
The attacker entry point is the public `#[update]` `notify_top_up` endpoint — no privilege is required. The ICP transfer to a non-existent canister's subaccount is valid on the ledger. The management canister's `deposit_cycles` reliably fails for non-existent canister IDs. The net cost per attack cycle is only `TOP_UP_CANISTER_REFUND_FEE` (a small ICP fee), as the principal is otherwise refunded. The attack is repeatable every hour once the CMC's accumulated balance is depleted by legitimate operations.

## Recommendation
Move `ensure_balance` (and therefore `check_and_add_cycles` / `ic0_mint_cycles128`) to after the management canister call succeeds. Alternatively, implement a compensating rollback: if `call_with_payment128` returns an error, subtract the previously added cycles from the limiter and decrement `state.total_cycles_minted` before returning. The simplest structural fix is to restructure `deposit_cycles` so that `ensure_balance` is only called on the success path, or to separate the minting step from the delivery step and only commit the limiter charge once delivery is confirmed.

## Proof of Concept
State-machine test sketch:
1. Configure CMC with a mock management canister that always rejects `deposit_cycles`.
2. Fund attacker with enough ICP to convert to 150P cycles; attacker sends ICP to CMC subaccount of a non-existent canister.
3. Attacker calls `notify_top_up { block_index, canister_id: non_existent }`.
4. Assert: `notify_top_up` returns `Err(NotifyError::Refunded { ... })` and attacker ICP balance is restored (minus fee).
5. Assert: `state.base_limiter.get_count()` equals 150P cycles — rate limit exhausted.
6. Assert: a subsequent legitimate `notify_top_up` from a different user returns `Err` containing "More than 150000000000000000 cycles have been minted in the last 3600 seconds."
