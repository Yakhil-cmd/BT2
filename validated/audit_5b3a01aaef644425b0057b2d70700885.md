### Title
NFT Offer Viewer Displays Incorrect "Net Proceeds" Due to Commented-Out Royalty Deduction in `calculateNFTRoyalties` - (File: packages/gui/src/components/offers/utils.ts)

### Summary
The `calculateNFTRoyalties` function hardcodes `nftSellerNetAmount = amount` (the full offered price) with the correct royalty-deducting formula commented out. This causes the NFT offer viewer to display the full offered price as "Net Proceeds" to an NFT seller reviewing an imported offer, instead of the actual net amount after creator royalties are deducted. A seller who accepts based on this display receives less XCH/CAT than the UI claimed.

### Finding Description
In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as simply `amount`, with the correct calculation commented out:

```typescript
// packages/gui/src/components/offers/utils.ts, lines 314-317
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

`nftSellerNetAmount` is consumed in two places in `NFTOfferViewer.tsx`:

1. **`overrideNFTSellerAmount`** (lines 397–402): computed as `chiaToMojo(nftSaleInfo?.nftSellerNetAmount)` and passed to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow` to override the "You will receive" row. Because `nftSellerNetAmount = amount`, this resolves to the same mojo value already in the offer summary, so the override is a no-op — but it was intended to show the post-royalty net.

2. **"Net Proceeds" label** (line 596): directly renders `nftSaleInfo?.nftSellerNetAmount` as the prominent "Net Proceeds" figure shown to the NFT seller. Because `nftSellerNetAmount = amount`, this shows the full offered price, not the actual proceeds after royalties.

The tooltip at lines 562–568 explicitly states that for the `TokenForNFT` case, "those creator fees will be paid from the offered purchase price," confirming the seller receives `amount − royaltyAmount`, not `amount`. The "Net Proceeds" label contradicts this.

### Impact Explanation
An NFT seller viewing an imported `TokenForNFT` offer sees a prominently labeled "Net Proceeds: X XCH" equal to the full offered price. On-chain, the seller actually receives `X − (royaltyPercentage/100 × X)` XCH, with the remainder going to the NFT creator. For a 10% royalty on a 1 XCH offer, the seller sees "Net Proceeds: 1 XCH" but receives 0.9 XCH. The seller approves the offer based on a materially incorrect amount display — a direct accounting spoofing impact on offer acceptance.

This satisfies the **High** impact class: "Corruption, spoofing, or unsafe trust of … offer … state that causes a user to approve … the wrong … amount."

### Likelihood Explanation
Any unprivileged actor can create a `TokenForNFT` offer for any NFT that has royalties enabled (a common, standard NFT property). The victim (NFT owner) who imports and views the offer is shown the wrong "Net Proceeds" figure. No special privileges, leaked keys, or social engineering beyond sharing a valid offer file are required. The royalty amount is shown separately in smaller text, but the prominent "Net Proceeds" label explicitly claims to represent the post-royalty net, creating a misleading authoritative display.

### Recommendation
Restore the correct `nftSellerNetAmount` calculation in `calculateNFTRoyalties`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

This ensures the "Net Proceeds" label and the `overrideNFTSellerAmount` passed to the summary row both reflect the actual post-royalty amount the seller will receive.

### Proof of Concept
1. Mint an NFT with a 10% royalty (`royaltyPercentage = 1000`).
2. As an attacker/buyer, create a `TokenForNFT` offer: offer 1 XCH for the victim's NFT.
3. Share the offer file with the NFT owner (victim).
4. Victim imports the offer into the GUI and views `NFTOfferDetails`.
5. GUI renders:
   - "NFT Purchase Price: 1 XCH"
   - "Creator Fee (10%): 0.1 XCH"
   - **"Net Proceeds: 1 XCH"** ← incorrect; should be 0.9 XCH
6. Victim accepts the offer believing they will receive 1 XCH.
7. On-chain settlement: victim receives 0.9 XCH; 0.1 XCH goes to the NFT creator's royalty address.

The discrepancy is rooted in `nftSellerNetAmount = amount` at line 314 of `utils.ts`, propagated through `nftSaleInfo` in `NFTOfferViewer.tsx` lines 377–394, and rendered at line 596. [1](#0-0) [2](#0-1) [3](#0-2)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
