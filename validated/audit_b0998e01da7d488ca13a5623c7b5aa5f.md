### Title
Unprivileged caller can invoke `update_rewards` with `disable_rewards: true` to permanently deny block rewards to all stakers — (`File: src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` carries no access-control guard. Any unprivileged caller can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` cursor to the current block **without distributing any rewards**. Every subsequent legitimate call for that same block is rejected with `REWARDS_ALREADY_UPDATED`, permanently destroying the block's reward allocation for all stakers.

---

### Finding Description

`update_rewards` is a public, permissionless entry point in `StakingRewardsManagerImpl`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();          // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.          ← written BEFORE the disable_rewards branch
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;                            ← exits with no reward distribution
    }
    ...
``` [1](#0-0) 

The critical ordering is:

1. `last_reward_block` is **written** at line 1485 before the `disable_rewards` guard at line 1487.
2. There is **no caller check** — `general_prerequisites` only asserts the contract is unpaused and the caller is non-zero.
3. The `disable_rewards` parameter is accepted from any caller. [2](#0-1) 

Because `last_reward_block` is a **single global** (not per-staker), one attacker transaction per block is sufficient to poison the cursor for every staker simultaneously. [3](#0-2) 

The sibling path `update_rewards_from_attestation_contract` — which **does** distribute rewards — is correctly gated:

```cairo
fn update_rewards_from_attestation_contract(...) {
    ...
    self.assert_caller_is_attestation_contract();
    ...
``` [4](#0-3) 

`update_rewards` lacks the analogous guard, mirroring the external report's pattern exactly: a sibling function has the restriction; the vulnerable function does not.

---

### Impact Explanation

**High — Permanent freezing / theft of unclaimed yield.**

When an attacker calls `update_rewards(any_active_staker, disable_rewards: true)` at block N:

- `last_reward_block` is set to N.
- No STRK block rewards are computed or credited to any staker.
- All subsequent calls for block N revert with `REWARDS_ALREADY_UPDATED`.
- The block-N reward allocation is **permanently unrecoverable** — there is no replay or catch-up mechanism.

Repeated once per block, the attacker can suppress the entire consensus-rewards stream indefinitely, constituting permanent freezing of unclaimed yield for all stakers and their delegators.

---

### Likelihood Explanation

**High.** The attack requires:

- A single L2 transaction per block (gas cost only).
- No privileged role, no leaked key, no external dependency.
- Any non-zero address qualifies as the caller.

The only precondition is that at least one active staker with non-zero balance exists, which is always true in a live deployment.

---

### Recommendation

Restrict `update_rewards` to a trusted caller (e.g., the attestation contract, a designated consensus contract, or the staker/operational address themselves). The simplest fix is to add an access-control assertion analogous to `assert_caller_is_attestation_contract` used in `update_rewards_from_attestation_contract`, or to remove the `disable_rewards` parameter from the public interface and expose it only through a privileged internal path.

---

### Proof of Concept

1. Consensus rewards are active (`consensus_rewards_first_epoch` has been set and the current epoch has passed it).
2. Attacker (any non-zero address) calls:
   ```
   staking.update_rewards(staker_address=<any_active_staker>, disable_rewards=true)
   ```
   at block N.
3. `last_reward_block` is written to N; no rewards are distributed.
4. The legitimate consensus mechanism (or any other caller) attempts:
   ```
   staking.update_rewards(staker_address=<staker>, disable_rewards=false)
   ```
   at the same block N → reverts with `REWARDS_ALREADY_UPDATED`.
5. Block-N rewards are permanently lost for all stakers.
6. Attacker repeats step 2 every block to suppress all future consensus rewards.
