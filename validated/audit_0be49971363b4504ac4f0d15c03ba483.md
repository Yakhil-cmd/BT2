### Title
Gateway Admits Unsigned Invoke Transactions via `skip_stateful_validations` Bypass — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

---

### Summary

The `skip_stateful_validations` function in the gateway skips the `__validate__` entry point (signature verification) for any invoke transaction with `nonce == 1` sent from an account whose address appears in `account_tx_in_pool_or_recent_block`. An unprivileged attacker who observes a victim's pending `deploy_account` transaction in the mempool can submit a fake invoke with `nonce=1` from the victim's address, carrying an arbitrary or empty signature. The gateway admits this transaction to the mempool without ever running `__validate__`, occupying the victim's nonce-1 slot and forcing fee escalation or blocking the legitimate invoke.

---

### Finding Description

`skip_stateful_validations` is a UX feature that allows a user to submit `deploy_account` + `invoke(nonce=1)` simultaneously, before the account is deployed on-chain. The gateway skips `__validate__` for the invoke when two conditions hold:

1. The invoke's `nonce == 1` and the on-chain account nonce is `0` (account not yet deployed).
2. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

The second condition is satisfied by **any** transaction from that address in the mempool — not specifically the account's own `deploy_account`. The mempool's `account_tx_in_pool_or_recent_block` checks `state.contains_account || tx_pool.contains_account`: [2](#0-1) 

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate == false` for an invoke, execution returns immediately without calling `__validate__`: [4](#0-3) 

**Attack scenario:**

1. Victim submits `deploy_account` for address `X` → admitted to mempool.
2. Attacker submits `invoke(nonce=1, sender=X, signature=<garbage>)`.
3. Gateway calls `skip_stateful_validations`: `tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(X) == true` (victim's `deploy_account` is in pool).
4. `skip_validate = true` → `__validate__` is never called → fake invoke admitted to mempool.
5. Victim's legitimate `invoke(nonce=1)` must now compete via fee escalation or is blocked.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate `tx_hash` and nonce ordering — it performs no signature verification: [5](#0-4) 

The batcher later re-creates the transaction with `AccountTransaction::new_for_sequencing` (which always sets `validate: true`): [6](#0-5) 

So the batcher will reject the fake invoke when `__validate__` fails during execution — but by then the mempool slot is occupied and the victim's legitimate transaction is displaced.

---

### Impact Explanation

The gateway's admission invariant is broken: an invoke transaction with an invalid or absent signature is accepted into the mempool without running `__validate__`. The corrupted value is the **admission decision** — the mempool accepts a transaction that should have been rejected at the gateway. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The attacker cannot execute arbitrary calls from the victim's account (the batcher's `__validate__` will reject the fake invoke), but the attack:
- Occupies the victim's nonce-1 slot in the mempool.
- Forces the victim to pay a higher tip for fee escalation to displace the fake transaction.
- Can be repeated for every new `deploy_account` observed in the mempool, creating a systematic griefing vector against all new account deployments.

---

### Likelihood Explanation

The attack requires only:
1. Mempool visibility (public RPC endpoint).
2. The ability to submit a transaction (no funds required — the fake invoke is rejected by the batcher before fee collection).

The attack window is the interval between the victim's `deploy_account` being admitted and the victim submitting their `invoke(nonce=1)`. This window is predictable and can be exploited programmatically by monitoring the mempool.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender's address is pending. Alternatively, add a mempool-level check that the `deploy_account` transaction hash matches the one expected for the sender's address (which is deterministic from `class_hash`, `salt`, and `constructor_calldata`). A simpler mitigation is to require that the `deploy_account` transaction hash be provided by the submitter and verified against the mempool before skipping `__validate__`, analogous to the `deploy_account_tx_hash` parameter already present in the `native_blockifier` path: [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
