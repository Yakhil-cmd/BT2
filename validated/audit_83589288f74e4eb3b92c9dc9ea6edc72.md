### Title
Unchecked `approve()` Return Value in Delegation Pool Breaks Staking for Non-Standard ERC20 Tokens - (File: src/pool/pool.cairo)

---

### Summary

The `transfer_to_staking_contract` internal function in `Pool` calls `token_dispatcher.approve(...)` using the raw `IERC20DispatcherTrait` without verifying the returned `bool`. Every other ERC20 operation in the codebase uses the `CheckedIERC20DispatcherTrait` wrappers (`checked_transfer`, `checked_transfer_from`), but this single `approve` call is inconsistently left unchecked. For any ERC20 token whose `approve()` can silently return `false` instead of reverting, the delegation entry-points `enter_delegation_pool` and `add_to_delegation_pool` become permanently non-functional.

---

### Finding Description

`transfer_to_staking_contract` is the internal helper called by both public delegation entry-points:

```
// src/pool/pool.cairo  lines 668-682
fn transfer_to_staking_contract(
    self: @ContractState,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,   // raw dispatcher
    staker_address: ContractAddress,
) {
    let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
    token_dispatcher
        .approve(spender: staking_pool_dispatcher.contract_address, amount: amount.into());
    //  ^^^^^^ return value (bool) is silently discarded

    staking_pool_dispatcher.add_stake_from_pool(:staker_address, :amount);
}
``` [1](#0-0) 

The callers are:

- `enter_delegation_pool` (line 199): `transfer_from_delegator` → `transfer_to_staking_contract`
- `add_to_delegation_pool` (line 239): `transfer_from_delegator` → `transfer_to_staking_contract` [2](#0-1) [3](#0-2) 

In contrast, every other ERC20 call in the codebase imports and uses `CheckedIERC20DispatcherTrait`:

```
// pool.cairo line 47
use starkware_utils::erc20::erc20_utils::CheckedIERC20DispatcherTrait;
``` [4](#0-3) 

Examples of consistent checked usage elsewhere:

- `transfer_from_delegator` → `checked_transfer_from` (pool/utils.cairo line 25)
- `add_stake_from_pool` → `checked_transfer_from` (staking.cairo line 1032)
- `remove_from_delegation_pool_action` → `checked_transfer` (staking.cairo line 1137) [5](#0-4) [6](#0-5) 

There is no `checked_approve` anywhere in the codebase (`grep` returns zero matches), confirming the gap is unintentional.

---

### Impact Explanation

When `approve()` silently returns `false` (non-reverting failure, as seen in certain non-standard ERC20 tokens):

1. The allowance on the staking contract is **not set**.
2. The immediately following `add_stake_from_pool` call invokes `checked_transfer_from(pool → staking, amount)`, which reverts with `INSUFFICIENT_ALLOWANCE` because the allowance was never granted.
3. The entire transaction reverts atomically.

The result is that **`enter_delegation_pool` and `add_to_delegation_pool` are permanently non-functional** for any pool whose underlying token has non-standard `approve()` semantics. Delegators cannot stake, and therefore cannot earn any yield from that pool. This constitutes permanent freezing of unclaimed yield / griefing of users with no attacker profit motive.

**Allowed impact matched**: *Griefing with no profit motive but damage to users or protocol* (Medium); *Permanent freezing of unclaimed yield* (High) if delegators are locked out of earning rewards entirely.

---

### Likelihood Explanation

The protocol explicitly supports multi-token staking beyond STRK (e.g., BTC) via `IStakingTokenManager`. [7](#0-6)  Any future token added whose `approve()` deviates from the standard (returns `false` instead of reverting) will trigger this bug for every delegator in that pool. The STRK token itself uses OpenZeppelin's standard implementation and is unaffected today, but the code path is live and reachable by any unprivileged delegator calling `enter_delegation_pool` or `add_to_delegation_pool`.

---

### Recommendation

Replace the raw `approve` call with a checked variant that asserts the return value is `true`, consistent with every other ERC20 call in the codebase:

```cairo
// Before (unsafe):
token_dispatcher
    .approve(spender: staking_pool_dispatcher.contract_address, amount: amount.into());

// After (safe):
token_dispatcher
    .checked_approve(spender: staking_pool_dispatcher.contract_address, amount: amount.into());
```

If `CheckedIERC20DispatcherTrait` does not yet expose `checked_approve`, add it following the same pattern as `checked_transfer` / `checked_transfer_from`.

---

### Proof of Concept

1. Deploy a non-standard ERC20 token whose `approve()` always returns `false` without reverting.
2. Add this token to the staking protocol via `add_token` (governance action).
3. A staker opens a delegation pool for this token via `set_open_for_delegation`.
4. Any delegator calls `enter_delegation_pool(reward_address, amount)`:
   - `transfer_from_delegator` succeeds (moves tokens from delegator to pool).
   - `transfer_to_staking_contract` calls `approve(staking, amount)` → returns `false`, discarded.
   - `add_stake_from_pool` → `checked_transfer_from(pool, staking, amount)` → reverts: `INSUFFICIENT_ALLOWANCE`.
   - Entire transaction reverts; delegator cannot enter the pool.
5. All subsequent calls to `enter_delegation_pool` and `add_to_delegation_pool` for this token revert identically. The pool is permanently non-functional; no delegator can earn yield from it. [8](#0-7)
