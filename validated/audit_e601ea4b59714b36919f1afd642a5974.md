### Title
Signature Validation Bypass for Invoke Transactions via Front-Running `deploy_account` in Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call for any invoke transaction with `nonce == 1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because that check returns `true` for **any** transaction from the address in the mempool — not specifically a `deploy_account` — an attacker who observes a legitimate `deploy_account` in the mempool can immediately submit an invoke with an invalid signature for the same address, have it admitted without signature verification, and thereby block the legitimate user's first invoke from entering the mempool.

### Finding Description

In `skip_stateful_validations` the gateway implements a UX shortcut: when a user sends `deploy_account` + `invoke(nonce=1)` simultaneously, the invoke's `__validate__` is skipped at the gateway because the account does not exist on-chain yet. [1](#0-0) 

The skip condition is:

```
tx.nonce() == 1  &&  account_nonce == 0  &&  account_tx_in_pool_or_recent_block(sender) == true
```

When all three hold, `skip_validate = true` is returned, and `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, so the blockifier never calls `__validate__`. [2](#0-1) 

The mempool-side check that feeds the third condition is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

This returns `true` for **any** transaction type from that address — it does not distinguish `deploy_account` from `invoke`. The code comment acknowledges this looseness: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* [4](#0-3) 

**Attack path:**

1. Alice submits `deploy_account` for address `X` (nonce 0). It is admitted to the mempool; `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. Bob observes Alice's `deploy_account` in the mempool (public mempool gossip or RPC).
3. Bob submits `invoke(sender=X, nonce=1, signature=[0,0])` — an invalid signature.
4. Gateway stateful validation: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → `skip_validate=true`. The `__validate__` entry point is **never called**. Bob's invalid invoke is admitted.
5. Alice submits her legitimate `invoke(sender=X, nonce=1, valid_signature)`. The mempool rejects it with `DuplicateNonce` because Bob's transaction already occupies `(X, nonce=1)`. [5](#0-4) 

The nonce-gap check in `validate_nonce` does not block Bob's invoke because `nonce=1` is within `[account_nonce=0, account_nonce + max_allowed_nonce_gap]`. [6](#0-5) 

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction and rejects a valid one before sequencing.**

An invoke transaction carrying an invalid (or zero) signature is admitted to the mempool without any signature check. Simultaneously, the legitimate user's first invoke after account deployment is rejected with `DuplicateNonce`. The user must either wait for Bob's transaction to expire (TTL) or pay a higher fee via fee escalation to displace it. Bob's invalid invoke will revert at execution time, but the admission decision is already wrong and the legitimate transaction is blocked.

### Likelihood Explanation

**Medium.** The attacker needs only to monitor the public mempool (or the gateway's RPC endpoint) for `deploy_account` transactions, then race to submit an invoke at `nonce=1` for the same address. No privileged access is required. The window is the time between Alice's `deploy_account` being admitted and Alice's `invoke` being submitted — a window that is intentionally wide because the UX feature is designed for users who submit both transactions together.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions. For example, maintain a separate set of addresses that have a pending `deploy_account` in the mempool, and only skip `__validate__` when the address is in that set. Alternatively, perform a lightweight off-chain signature check (e.g., ECDSA verification against the public key embedded in the constructor calldata) even when skipping the full blockifier validation.

### Proof of Concept

```

   → Gateway admits it; mempool now has X in tx_pool.

2. Submit RpcInvokeTransactionV3 {

       nonce: 1,

         validate_by_mempool: no tx at (X,1) yet → OK

           → returns true (skip)

         ExecutionFlags { validate: false, ... }
         __validate__ is never called
   → Invalid invoke admitted to mempool.

3. Submit legitimate RpcInvokeTransactionV3 {
       sender_address: X, nonce: 1, signature: <valid>
   }
   → Mempool rejects: DuplicateNonce { address: X, nonce: 1 }

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
