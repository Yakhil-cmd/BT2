Looking at the factory, admin, and fee paths in Metric OMM for a validation function that claims to enforce bounds but doesn't.

**Key observation:** `setPoolAdminFees` in the factory validates `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` before writing, but `setPoolBinAdditionalFees` — the other pool-admin fee-setting path — passes `addFeeBuyE6` / `addFeeSellE6` straight through to the pool with zero cap check.

---

### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Unchecked `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` to the pool without validating them against any factory cap, while every other admin fee-setting path in the same contract enforces `maxAdminSpreadFeeE6`. A pool admin can therefore charge per-bin fees that exceed the factory owner's intended ceiling, causing traders to lose more funds than the protocol's fee-cap system is designed to allow.

### Finding Description
The factory maintains `maxAdminSpreadFeeE6` (bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000`) and enforces it in `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol line 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

The parallel admin path `setPoolBinAdditionalFees` performs no such check:

```solidity
// MetricOmmPoolFactory.sol lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes:

```solidity
// MetricOmmPool.sol lines 469-473
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

In the swap path these bin-level fees are **added on top of** the base spread fee:

```solidity
// MetricOmmPool.sol lines 540-541
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so the pool admin can set them to **65 535** (≈ 6.55 % in E6 units) regardless of what `maxAdminSpreadFeeE6` is set to. If the factory owner has tightened the cap to, say, 1 000 (0.1 %), the pool admin can still impose a 6.55 % per-bin surcharge, making the effective total fee far exceed the owner-intended ceiling.

### Impact Explanation
Traders swapping through a targeted bin pay a fee that can be up to `uint16.max` (65 535 E6 ≈ 6.55 %) above the base spread fee, bypassing the factory owner's `maxAdminSpreadFeeE6` cap entirely. This is a direct, quantifiable loss of trader principal routed to the pool admin, constituting an admin-boundary break where a semi-trusted actor exceeds the caps the protocol is designed to enforce.

### Likelihood Explanation
Any pool admin — a role that can be transferred to an adversarial address — can call `setPoolBinAdditionalFees` at any time with no timelock. The call requires no special conditions beyond holding the `poolAdmin` role. The factory owner has no mechanism to prevent or reverse the fee setting after the fact without pausing the pool.

### Recommendation
Mirror the cap enforcement already present in `setPoolAdminFees`. Add a check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the factory owner can configure independently.

### Proof of Concept
1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 1_000` (0.1 %) via `setFeeCaps`.
2. Pool is created; pool admin is set to `attacker`.
3. `attacker` calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — no revert, no cap check.
4. Any trader swapping through bin 0 now pays `spreadFeeE6 + 65_535` in total spread cost, far exceeding the 0.1 % cap the factory owner intended.
5. The excess fee accrues as surplus in the pool and is collected by the protocol/admin via `collectFees`, with the admin portion flowing to `poolAdminFeeDestination` — a direct transfer of trader funds beyond the sanctioned limit.
