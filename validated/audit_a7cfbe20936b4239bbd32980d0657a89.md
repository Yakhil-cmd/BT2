### Title
Unvalidated `old_participants` List in Reshare Enables Malicious Participant to Corrupt Lagrange Coefficients and Permanently Deny Reshare — (File: src/dkg.rs)

---

### Summary
In `do_reshare`, each participant locally computes their secret contribution using a Lagrange coefficient derived from a caller-supplied `old_participants` list. This list is never broadcast or cross-validated across participants. A malicious old participant can supply a different (but locally invariant-passing) `old_participants` list, causing them to compute a wrong Lagrange coefficient and contribute an inconsistent polynomial to the new DKG. The resulting new public key will not match the old one, causing the protocol to abort and permanently denying reshare for honest parties.

---

### Finding Description

In `do_reshare` (`src/dkg.rs`, lines 600–635), each participant computes their secret contribution as:

```rust
let intersection = old_participants.intersection(&participants);
let secret = old_signing_key
    .map(|x_i| {
        intersection
            .lagrange::<C>(me)
            .map(|lambda| lambda * x_i.to_scalar())
    })
    .transpose()?
    .unwrap_or_else(<C::Group as Group>::Field::zero);
``` [1](#0-0) 

The `old_participants` list is passed by the caller and used to compute the intersection with new participants. The Lagrange coefficient is computed over this intersection. However, the `old_participants` list is **never broadcast or verified against other participants' views** — it is purely a local, caller-controlled input.

The pre-flight check `assert_reshare_keys_invariants` only validates that the intersection is at least `old_threshold` in size:

```rust
if old_participants.intersection(&participants).len() < old_threshold {
    return Err(InitializationError::NotEnoughParticipantsForNewThreshold { ... });
}
``` [2](#0-1) 

It does **not** verify that all participants agree on the same `old_participants` list. The session ID is computed only from broadcast random nonces — the `old_participants` list is never included in any cross-participant transcript:

```rust
let session_ids = do_broadcast(&mut chan, &participants, me, my_session_id).await?;
let session_id = domain_separate_hash(&mut domain_separator, &session_ids)?;
``` [3](#0-2) 

A malicious old participant can therefore pass a proper subset of the true `old_participants` list (e.g., `{A, B}` instead of `{A, B, C}`), provided the subset still satisfies the invariant check. This changes their Lagrange coefficient, making their polynomial contribution inconsistent with the honest participants' contributions. The resulting aggregate public key will not match the old one, triggering the abort check:

```rust
if old_vk != verifying_key {
    return Err(ProtocolError::AssertionFailed(
        "new public key does not match old public key".to_string(),
    ));
}
``` [4](#0-3) 

This is structurally analogous to the external report: just as the attacker there chose a buy-price `X` (a caller-controlled subset parameter) to manipulate which contributions were counted, here the attacker chooses an `old_participants` list (a caller-controlled subset parameter) to manipulate which Lagrange coefficients are computed — in both cases the subset passes local validation but is inconsistent with the honest parties' view.

---

### Impact Explanation

**High: Permanent denial of reshare for honest parties under valid protocol inputs and documented trust assumptions.**

A single malicious old participant can always cause the reshare protocol to abort by supplying a locally-valid but incorrect `old_participants` list. Because the `old_participants` list is never included in any broadcast or transcript, honest parties cannot identify the culprit or distinguish this from a legitimate protocol failure. If the malicious participant is among the minimum required `old_threshold` contributors, the reshare is permanently blocked — honest parties cannot complete key migration, threshold change, or participant rotation.

---

### Likelihood Explanation

Any malicious old participant can trigger this with zero cryptographic effort. The attack requires only knowledge of the true `old_participants` list (public information) and the ability to call `reshare` with a different list that still satisfies the local invariant check (`intersection.len() >= old_threshold`). No special privileges, leaked keys, or external assumptions are needed. The attack is deterministic and repeatable.

---

### Recommendation

Include the `old_participants` list in the session ID computation so that all participants commit to the same list before computing Lagrange coefficients:

```rust
// In do_keyshare / do_reshare, broadcast old_participants as part of the session transcript
let session_id = domain_separate_hash(
    &mut domain_separator,
    &(&session_ids, &old_participants),  // bind old_participants into session ID
)?;
```

Alternatively, add an explicit broadcast-and-verify round at the start of `do_reshare` where each participant broadcasts their `old_participants` list and aborts if any participant's list differs from the majority. This ensures all participants compute Lagrange coefficients over the same intersection, eliminating the inconsistency.

---

### Proof of Concept

1. Old participants: `{A, B, C}` with `old_threshold = 2`. New participants: `{A, B, C, D}` with `new_threshold = 2`.
2. Malicious participant A calls `reshare` with `old_participants = {A, B}` (omitting C).
3. Local invariant check: `intersection({A, B}, {A, B, C, D}) = {A, B}`, size `2 >= old_threshold 2` — **passes**.
4. A computes Lagrange coefficient `λ_A` over `{A, B}` instead of the correct `{A, B, C}`, yielding a different scalar.
5. A contributes polynomial `f_A(x)` with constant term `λ_A * x_A` (wrong) to the new DKG.
6. Honest participants B and C compute their Lagrange coefficients over `{A, B, C}` (correct).
7. The aggregate new public key `Σ f_i(0) * G` does not equal the old public key.
8. All honest participants hit the check at `src/dkg.rs:491` and abort.
9. Reshare is permanently denied for honest parties B, C, D — key migration is blocked indefinitely.
