### Title
`rerandomize_presign` Does Not Consume `PresignOutput`, Enabling Nonce Reuse and Secret Key Extraction — (File: `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

`RerandomizedPresignOutput::rerandomize_presign` accepts `presignature: &PresignOutput` — a shared borrow — instead of consuming the value. The library's own documentation mandates that each `PresignOutput` is one-time use and must be destroyed after consumption. Because the API does not enforce this at the type level, the same `PresignOutput` can be rerandomized multiple times with different `RerandomizationArguments`, producing multiple valid `RerandomizedPresignOutput` values from the same underlying nonce material. Two signing sessions that share the same nonce expose the secret key via standard ECDSA nonce-reuse algebra.

---

### Finding Description

The library explicitly documents the one-time-use requirement in multiple places:

> "Each output is consumed **exactly once** (one-time use)." [1](#0-0) 

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [2](#0-1) 

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [3](#0-2) 

Despite this, the production function signature is:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,          // shared borrow — NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
``` [4](#0-3) 

Because `presignature` is a shared reference, the caller retains full ownership of the `PresignOutput` after the call returns. The Rust type system places no obstacle on calling `rerandomize_presign` a second time with the same value and a different `RerandomizationArguments` (different `msg_hash`, `tweak`, `entropy`, or `participants`).

Compounding this, `PresignOutput` derives `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
``` [5](#0-4) 

Even if the call site were changed to pass by value, a caller could still clone the struct before passing it. Together, the borrow-based API and the `Clone` impl mean the library provides no mechanism — compile-time or runtime — to enforce the one-time-use invariant.

The rerandomization formula scales the nonce by `delta`:

```rust
let rerandomized_big_r = presignature.big_r * delta;
let rerandomized_k     = presignature.k * inv_delta;
let rerandomized_sigma = (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;
``` [6](#0-5) 

Two calls with `delta1` and `delta2` produce nonces `k/delta1` and `k/delta2`. These are multiplicatively related: `(k/delta1) / (k/delta2) = delta2/delta1`, a known ratio. Two ECDSA signatures sharing multiplicatively related nonces allow full secret-key recovery with simple linear algebra — no brute force required.

The security considerations section of the robust ECDSA documentation confirms the exact attack:

> "if different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [7](#0-6) 

The library documents the danger but does not close it in code.

---

### Impact Explanation

Two signatures produced from the same `PresignOutput` with different `RerandomizationArguments` satisfy:

```
s1 = (h1 + r1 * x) / (k / delta1)
s2 = (h2 + r2 * x) / (k / delta2)
```

where `r_i = x_coord(R_i)` and `R_i = presignature.big_r * delta_i`. Since `delta1`, `delta2`, `h1`, `h2`, `r1`, `r2`, `s1`, `s2` are all known to the coordinator, the system has one unknown (`x`, the secret key) and is trivially solvable. This matches the **Critical** impact: extraction of private signing shares / aggregate secret material.

---

### Likelihood Explanation

A malicious coordinator controls which `RerandomizationArguments` are presented to participants for each signing session. Nothing in the protocol prevents the coordinator from presenting two different argument sets that both reference the same presignature. Participants have no in-library mechanism to detect or refuse a second use of a presignature they have already consumed. The attack requires only two completed signing sessions and standard arithmetic — no cryptographic break, no side channel, no external assumption.

---

### Recommendation

1. **Change `rerandomize_presign` to consume `PresignOutput` by value** so the Rust move semantics prevent a second call without an explicit clone:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // consumed, not borrowed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

2. **Remove `Clone` from `PresignOutput`** (and the analogous robust-ECDSA type) so that callers cannot trivially circumvent the move-based protection.

3. **Add a runtime guard** (e.g., a `used: AtomicBool` inside `PresignOutput`) as a defence-in-depth measure for serialization/deserialization paths where move semantics cannot be relied upon.

---

### Proof of Concept

```rust
// After a presigning session each participant holds their PresignOutput.
let presign: PresignOutput = /* result of presign protocol */;

// Session 1 — legitimate signing request
let args1 = RerandomizationArguments::new(pk, tweak, msg_hash_1, presign.big_r, participants.clone(), entropy_1);
let rerandomized_1 = RerandomizedPresignOutput::rerandomize_presign(&presign, &args1).unwrap();
// ... run sign protocol, coordinator collects (R1, s1)

// Session 2 — malicious coordinator reuses the same presignature with a different message
let args2 = RerandomizationArguments::new(pk, tweak, msg_hash_2, presign.big_r, participants.clone(), entropy_2);
let rerandomized_2 = RerandomizedPresignOutput::rerandomize_presign(&presign, &args2).unwrap();
// ... run sign protocol, coordinator collects (R2, s2)

// Coordinator now holds (R1,s1,h1) and (R2,s2,h2) with multiplicatively related nonces.
// Solve: x = (s1*h2 - s2*h1) / (s2*r1*delta_ratio - s1*r2) — secret key recovered.
```

The second call to `rerandomize_presign` compiles and runs without error because `presign` is still in scope and the function accepts a shared reference. [8](#0-7)

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
```
