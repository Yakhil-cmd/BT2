### Title
Missing `app_pk` identity-point check allows extraction of confidential derived secret — (`src/confidential_key_derivation/protocol.rs`)

### Summary
`ckd()` and `compute_signature_share()` accept `app_pk = G1Projective::identity()` without error. When the identity point is used as the ElGamal encryption key, the blinding term `app_pk * y` collapses to the group identity, causing `CKDOutput.big_c` to equal `msk * H(pk||app_id)` — the confidential derived secret — in plaintext.

### Finding Description
The ElGamal blinding in `compute_signature_share` is:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;   // line 174
``` [1](#0-0) 

If `app_pk` is the group identity `O`, then `O * y = O` for any scalar `y`, so `big_c = big_s = x_i * H(pk||app_id)`. After Lagrange-weighted aggregation by the coordinator:

```rust
norm_big_y += participant_output.big_y();
norm_big_c += participant_output.big_c();
``` [2](#0-1) 

`big_c` becomes `msk * H(pk||app_id)` — the exact value `unmask(a)` is supposed to produce only when the legitimate holder of `a` calls it. The `ckd()` entry point performs no validation of `app_pk`: [3](#0-2) 

### Impact Explanation
`CKDOutput.big_c` is the confidential derived secret `s = msk * H(pk||app_id)` in plaintext. It is returned to the coordinator and, per the protocol design, transmitted on-chain. Any observer of the on-chain output reads `s` directly without needing `app_sk`. This is a full disclosure of the confidential derived secret, matching the Critical impact tier: *Extraction, reconstruction, or disclosure of confidential derived secrets*.

### Likelihood Explanation
`app_pk` is a caller-supplied `PublicKey` (`blstrs::G1Projective`) with no type-level restriction preventing the identity point. Any entity that can invoke `ckd()` — including the TEE app submitting a request on-chain — can supply `app_pk = G1Projective::identity()`. No cryptographic assumption needs to be broken; it is a pure input-validation gap.

### Recommendation
Add an explicit identity-point check in `ckd()` before the protocol runs:

```rust
if app_pk == PublicKey::identity() {
    return Err(InitializationError::InvalidInput("app_pk must not be the identity point".into()));
}
```

This check belongs in `ckd()` at `src/confidential_key_derivation/protocol.rs` immediately after the existing participant-list validations (after line 101).

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
