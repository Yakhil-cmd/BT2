### Title
Malicious CKD Participant Can Silently Corrupt Derived Key Output via Unverified Contribution Aggregation — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol aggregates each participant's `(norm_big_y, norm_big_c)` contribution in a stepwise, unverified manner. Because no zero-knowledge proof is required to demonstrate that a contribution was computed from the participant's actual key share and consistent randomness, any malicious participant can inject arbitrary curve points. The coordinator blindly sums all inputs and returns a silently corrupted `CKDOutput` that honest parties accept as valid.

---

### Finding Description

In `compute_signature_share`, each participant computes:

```
norm_big_y = λ_i · y_i · G
norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

and sends the pair privately to the coordinator. [1](#0-0) 

The coordinator then aggregates every received pair with no verification whatsoever:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [2](#0-1) 

There is no check that:
1. The `norm_big_c` component was formed using the participant's actual key share `x_i` (consistent with the public polynomial from DKG).
2. The same ephemeral scalar `y_i` was used in both `norm_big_y` and `norm_big_c`.

A malicious participant can send any two arbitrary `G1` points. The coordinator sums them into the final `CKDOutput` and returns it as `Some(ckd_output)`. [3](#0-2) 

The `do_ckd_participant` path has no proof generation either — it simply calls `compute_signature_share` and forwards the result: [4](#0-3) 

The `CKDOutput::unmask` function then computes `C − a · Y` and returns the result as the confidential derived key with no integrity check: [5](#0-4) 

---

### Impact Explanation

**HIGH — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A single malicious participant can cause every honest party (including the coordinator) to silently accept a `CKDOutput` whose `big_y` and `big_c` fields are attacker-chosen. The application calling `unmask(app_sk)` will receive a wrong curve point instead of `msk · H(pk ‖ app_id)`. Because the protocol produces no transcript or commitment that would allow post-hoc detection, the corruption is invisible to all honest parties. The derived key is permanently wrong for that invocation, and no retry mechanism is specified.

---

### Likelihood Explanation

Any participant in the CKD participant list — an unprivileged library caller who has been admitted to the protocol — can trigger this. No special privilege beyond being listed in `participants` is required. The attacker does not need to compromise keys, break cryptographic primitives, or control the coordinator. The attack is deterministic and requires sending two arbitrary `G1` points in place of a legitimate contribution. [6](#0-5) 

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation. Concretely, the participant should prove in zero knowledge:

1. **Consistency of randomness:** `norm_big_y = λ_i · y_i · G` and the `y_i · app_pk` term in `norm_big_c` use the same scalar `y_i` (a Chaum–Pedersen / DLEQ proof over the pair `(G, app_pk)`).
2. **Correct key share:** `norm_big_c − norm_big_y · (app_pk / G)` equals `λ_i · x_i · H(pk ‖ app_id)`, consistent with the public polynomial committed during DKG (a proof of discrete-log equality against the participant's public key share).

The coordinator must verify both proofs before adding a contribution to the running sum.

---

### Proof of Concept

**Setup:** 3-of-3 CKD session. Participants are `P1` (honest), `P2` (honest), `P3` (malicious).

1. `P1` and `P2` compute correct `(norm_big_y_i, norm_big_c_i)` and send them to the coordinator.
2. `P3` sends `(G, G)` — the generator point for both fields — instead of its real contribution.
3. The coordinator (say `P1`) executes:
   ```rust
   norm_big_y += G;   // P3's fake big_y
   norm_big_c += G;   // P3's fake big_c
   ```
4. The final `CKDOutput` is `(Y_honest + G, C_honest + G)`.
5. The application calls `unmask(app_sk)`:
   ```
   (C_honest + G) − app_sk · (Y_honest + G)
   = (msk · H(pk‖app_id) + G − app_sk · G)
   ≠ msk · H(pk‖app_id)
   ```
6. The derived key is wrong. No error is raised. All honest parties accept the output. [2](#0-1) [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
