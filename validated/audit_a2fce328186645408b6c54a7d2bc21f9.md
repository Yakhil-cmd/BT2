### Title
Malicious Participant Can Force `w = 0` in Robust ECDSA Presign via Rushing Adversary, Causing Permanent Denial of Signing — (File: src/ecdsa/robust_ecdsa/presign.rs)

---

### Summary

In the robust ECDSA presign protocol, round-2 `w_me` values are broadcast without a prior commitment. A malicious participant acting as a rushing adversary can delay their broadcast until all honest participants' `w_me` values are received, then craft a `w_me` that forces the interpolated `w = 0`. This triggers `ProtocolError::ZeroScalar` for every honest participant, permanently denying the presign operation.

---

### Finding Description

In `do_presign()`, each participant computes:

```
w_me = a_me * k_me + b_me
```

and broadcasts it in round 2 via `send_many` at line 152:

```rust
chan.send_many(wait_round_2, &(&big_r_me, &SigningShare::<C>::new(w_me)))?;
``` [1](#0-0) 

After all `w_me` values are collected, every participant interpolates `w` using all `2t+1` shares and checks for zero at step 3.6:

```rust
let w = Polynomial::eval_interpolation(w_2tp1_identifiers, w_2tp1_verifying_shares, None)?;

// Step 3.6
// check w is non-zero
if w.0.is_zero().into() {
    return Err(ProtocolError::ZeroScalar);
}
``` [2](#0-1) 

This check exists because line 315 calls `w.0.invert().unwrap()`, which would panic on zero:

```rust
let c_me = w.0.invert().unwrap() * shares.a();
``` [3](#0-2) 

**The attack:** A malicious participant `j` acts as a rushing adversary — they delay their `w_me` broadcast until they have received all honest participants' `w_i` values. They then compute:

```
w_j = -( sum(lambda_i * w_i for i ≠ j) ) / lambda_j
```

Since `lambda_j ≠ 0` for any valid participant identifier, this is always solvable. The malicious participant sends this crafted `w_j`, causing the interpolated `w = 0` for all participants.

There is **no commitment scheme** for `w_me` values in round 2. The `send_many` broadcast is not preceded by a hash commitment, so a rushing adversary can adaptively choose their value after observing all honest participants' values. Furthermore, individual `w_me` values are not verifiable (there are no public commitments to `a_me` or `b_me`), so the malicious participant cannot be identified from the protocol transcript.

The presign validation enforces `participants.len() == 2*max_malicious+1`, meaning all participants are used in the interpolation:

```rust
if participants.len() != robust_ecdsa_threshold {
    return Err(InitializationError::BadParameters(
        "the number of participants during presigning must be exactly 2*max_malicious+1 ...".to_string(),
    ));
}
``` [4](#0-3) 

With `t` malicious participants out of `2t+1` total, the malicious coalition controls `t` of the `w_i` values and needs to satisfy only one linear equation (`w = 0`), which is always achievable.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.** The malicious participant can repeat this attack on every presign attempt. Because no participant can identify which `w_me` was crafted (there is no per-share verification), the malicious participant cannot be excluded from future rounds based on protocol evidence alone. Every presign invocation involving the malicious participant will fail with `ProtocolError::ZeroScalar`. [5](#0-4) 

---

### Likelihood Explanation

**High.** The rushing adversary model is a standard and well-documented adversary capability in MPC protocols. Any malicious participant who controls their network stack (i.e., can delay outgoing messages) can execute this attack. No special cryptographic capability is required — only the ability to receive before sending in round 2, which is trivially achievable outside the library's own async state machine.

---

### Recommendation

Add a **commit-then-reveal** scheme for `w_me` (and `big_r_me`) in round 2:

1. **Commit sub-round:** Each participant broadcasts `H(w_me || big_r_me || nonce)`.
2. **Reveal sub-round:** After all commitments are received, each participant broadcasts the actual `(w_me, big_r_me, nonce)`.

This prevents the rushing adversary from adaptively choosing `w_me` based on honest participants' values, because the commitment is binding before any values are revealed.

Alternatively, add a zero-knowledge proof that `w_me` is correctly formed with respect to the polynomial commitments already exchanged in round 1, allowing honest participants to identify and exclude the malicious party.

---

### Proof of Concept

Setup: 3 participants `{P1, P2, P3}`, `max_malicious = 1`, `P3` is malicious.

1. Round 1 completes normally — all participants exchange polynomial evaluations privately.
2. Round 2 begins. `P3` delays their broadcast.
3. `P3` receives `w_1` from `P1` and `w_2` from `P2`.
4. `P3` computes Lagrange coefficients `lambda_1, lambda_2, lambda_3` at 0 for the three participant identifiers.
5. `P3` computes `w_3 = -(lambda_1 * w_1 + lambda_2 * w_2) / lambda_3`.
6. `P3` broadcasts `(big_r_3, w_3)` (with any `big_r_3`).
7. All participants compute `w = lambda_1*w_1 + lambda_2*w_2 + lambda_3*w_3 = 0`.
8. All participants hit the check at line 246 and return `ProtocolError::ZeroScalar`.
9. No participant can determine that `P3` caused the failure.
10. `P3` repeats on every retry. [6](#0-5)
