### Title
`new_settlement_layer_chain_id_storage` Is Never Rolled Back on Frame Revert - (`File: basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIOStateSnapshot` omits a snapshot handle for `new_settlement_layer_chain_id_storage`. Consequently, `finish_io_frame` never rolls back that sub-storage when a frame is reverted. If a transaction triggers `update_settlement_layer_chain_id` (via the `system_context_event_hook`) and then reverts, the settlement-layer chain-ID update persists in `new_settlement_layer_chain_id_storage` even though every other side-effect of that transaction is correctly undone. This produces an irrecoverable state inconsistency that panics at block finalization.

---

### Finding Description

`FullIO` owns six rollbackable sub-storages: [1](#0-0) 

The snapshot struct that is used to roll them back is: [2](#0-1) 

`FullIOStateSnapshot` captures handles for `storage`, `transient_storage`, `logs_storage`, `events_storage`, and `interop_root_storage` — but **not** for `new_settlement_layer_chain_id_storage`.

`start_io_frame` snapshots five of the six sub-storages and returns the snapshot: [3](#0-2) 

`finish_io_frame` rolls back those same five sub-storages when a rollback handle is supplied, but `new_settlement_layer_chain_id_storage.finish_frame(...)` is never called: [4](#0-3) 

`NewSettlementLayerChainIdStorage` does expose a `start_frame` / `finish_frame` API that is fully capable of rolling back an update: [5](#0-4) 

The update is written during execution via `update_settlement_layer_chain_id`, which is called by `system_context_event_hook` whenever the `SettlementLayerChainIdUpdated` event is emitted: [6](#0-5) [7](#0-6) 

At block finalization, `read_batch_context_inputs` asserts that if `new_settlement_layer_chain_id_storage` holds a value it must match the value actually stored in the SystemContext slot: [8](#0-7) 

If the transaction that emitted the event reverted, the SystemContext storage slot was rolled back (via `storage.finish_frame`) but `new_settlement_layer_chain_id_storage` was not, so the two values diverge and the assertion panics.

Additionally, `NewSettlementLayerChainIdStorage::update` guards against a second update within the same block: [9](#0-8) 

Because the stale update is never cleared, any subsequent legitimate service transaction that tries to set the SL chain ID in the same block will hit this guard and fail with an internal error.

---

### Impact Explanation

A transaction that emits `SettlementLayerChainIdUpdated` and then reverts (e.g., OOG after the event hook fires, or an explicit revert in the outer call) leaves `new_settlement_layer_chain_id_storage` in a permanently dirty state for the rest of the block. Two concrete consequences:

1. **Block finalization panic** — the `assert_eq!` in `read_batch_context_inputs` fires because the storage slot was rolled back but the in-memory counter was not. The block cannot be sealed.
2. **Denial of legitimate SL chain-ID updates** — the "already updated" guard permanently blocks any further update in the same block, even though the first update was reverted.

Both outcomes are irreversible within the block; the sequencer must discard the block entirely.

---

### Likelihood Explanation

The trigger requires a transaction that (a) causes the SystemContext contract to emit `SettlementLayerChainIdUpdated` and (b) subsequently reverts. Concretely:

- A service transaction with a gas limit that is exhausted after the event is emitted but before the call returns successfully.
- A service transaction where the SystemContext call succeeds (emitting the event) but an outer wrapper reverts.

Service transactions are operator-controlled, so the direct attacker surface is the operator or anyone who can influence the gas limit or calldata of a service transaction. Given that the block-finalization panic is a hard crash with no recovery path within the block, even a single such occurrence is a critical availability failure.

---

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and include it in both `start_io_frame` and `finish_io_frame`, mirroring the pattern used for `interop_root_storage`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    new_sl_chain_id: NewSettlementLayerChainIdSnapshotId, // add this
}
```

In `start_io_frame`:
```rust
let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();
// include in returned snapshot
```

In `finish_io_frame`:
```rust
self.new_settlement_layer_chain_id_storage
    .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id));
```

### Proof of Concept

1. Block contains a service transaction that calls `SystemContext.setSettlementLayerChainId(42)`.
2. The SystemContext contract emits `SettlementLayerChainIdUpdated(42)`.
3. The `system_context_event_hook` fires → `new_settlement_layer_chain_id_storage` is set to `Some(42)`.
4. The transaction runs out of gas (or the outer call reverts for any reason).
5. `finish_io_frame(Some(rollback_handle))` is called:
   - `storage.finish_frame(...)` → SystemContext slot `0` is rolled back to the old chain ID.
   - `events_storage.finish_frame(...)` → the event is removed.
   - `new_settlement_layer_chain_id_storage` is **not** rolled back → still holds `Some(42)`.
6. Block finalization calls `read_batch_context_inputs`:
   - `settlement_layer_chain_id` is read from storage → old value (e.g., `1`).
   - `new_settlement_layer_chain_id_storage.value()` → `Some(42)`.
   - `assert_eq!(42, 1)` → **panic**, block cannot be finalized.
