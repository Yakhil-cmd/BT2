### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Transaction Senders to Lose Gas Refunds - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the `delta_gas` reconciliation between native resource consumption and EVM gas consumption is applied only in one direction. When `delta_gas > 0` (native cost exceeds gas cost), `gas_used` is correctly increased. However, when `delta_gas < 0` (native cost is less than gas cost), `gas_used` is never reduced. The result is that users are systematically overcharged: the excess gas is silently consumed rather than refunded. A developer-acknowledged `TODO` comment at the exact site confirms this is a known open issue.

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements the double-resource reconciliation step described in `docs/double_resource_accounting.md`. After execution, it computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
```

The intent, per the documentation, is to align the EVM-gas charge with the actual native (proving) cost. The docs state:

> "If `deltaGas > 0`, we add it to `gasUsed`… Finally, any remaining gas left is refunded as usual."

The code at lines 75–80 implements only the positive branch:

```rust
if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` — meaning the native resource consumption is *less* than the EVM gas consumption — `gas_used` is left unchanged. The user is charged for the full EVM gas consumed, even though the actual proving cost was lower. The `|delta_gas|` units of gas are effectively confiscated: they are neither refunded to the sender nor paid to the operator as a legitimate fee.

The downstream effect is direct: `refund_and_commit_fee` (both in `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs` lines 452–458 and `ethereum/mod.rs` lines 508–518) computes the token refund as `gas_price * (gas_limit - gas_used)`. Because `gas_used` is inflated by `|delta_gas|`, the refund is reduced by exactly `gas_price * |delta_gas|`.

### Impact Explanation

Every transaction where `native_used / native_per_gas < gas_used` results in the sender receiving a smaller refund than they are entitled to. The shortfall per transaction is:

```
loss = gas_price * |delta_gas|
     = gas_price * (gas_used - native_used / native_per_gas)
```

Since `native_per_gas = gas_price / native_price`, this simplifies to:

```
loss = gas_used * gas_price - native_used * native_price
```

This is the difference between what the user paid (in token terms) and what the actual proving cost was. The tokens are neither burned nor credited anywhere — they are simply not returned. This constitutes a direct, permanent loss of user funds with no compensating benefit to any protocol participant.

### Likelihood Explanation

The condition `delta_gas < 0` is reachable through ordinary, unprivileged L2 transactions. It is triggered whenever EVM gas consumption exceeds the native-resource-equivalent gas. This happens in two common scenarios:

1. **High gas price / high priority fee**: `native_per_gas = gas_price / native_price` grows with `gas_price`, so `native_used / native_per_gas` shrinks. A user who sets a high `max_priority_fee_per_gas` to get fast inclusion will have a large `native_per_gas`, making `delta_gas` more negative.

2. **Compute-heavy, low-pubdata transactions**: Transactions that consume many EVM opcodes but write little to storage produce high `gas_used` and low `native_used` (pubdata dominates native cost). The ratio `native_used / native_per_gas` can easily fall below `gas_used`.

Both scenarios are routine in production. No special access, governance role, or oracle manipulation is required.

### Recommendation

Apply the `delta_gas` adjustment symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, subject to the `minimal_gas_used` floor already established at line 56:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // delta_gas < 0: native cost was lower than gas cost; refund the difference
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This matches the documented intent ("any remaining gas left is refunded as usual") and resolves the open `TODO` at line 80.

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `gas_price = 10_000` (user sets a high priority fee)
- `native_price = 100`
- `native_per_gas = gas_price / native_price = 100`
- After execution: `gas_used (from ergs) = 60_000`, `native_used = 3_000_000`

**Calculation:**
```
delta_gas = (3_000_000 / 100) - 60_000
          = 30_000 - 60_000
          = -30_000
```

**Current behavior (buggy):**
- `delta_gas < 0` → branch not taken → `gas_used` stays at `60_000`
- Refund = `gas_price * (gas_limit - gas_used)` = `10_000 * 40_000` = `400_000_000`

**Correct behavior:**
- `gas_used` should be reduced to `max(60_000 - 30_000, minimal_gas_used)` = `30_000`
- Refund = `10_000 * 70_000` = `700_000_000`

**User loss per transaction:** `300_000_000` base tokens (the `30_000 * 10_000` gap). Any unprivileged sender submitting a compute-heavy transaction with a non-trivial gas price will silently lose this amount on every execution.
