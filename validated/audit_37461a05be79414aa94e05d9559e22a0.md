### Title
Caller's attached NEAR deposit is permanently locked in `WalletContract` when intermediate cross-contract calls fail in `address_check_callback` or `nep_141_storage_balance_callback` — (`runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

---

### Summary

The `WalletContract` tracks an external caller's attached deposit via `CallerDeposit` specifically to refund it if the eventual action fails. `rlp_execute_callback` correctly issues that refund. However, two intermediate callbacks — `address_check_callback` and `nep_141_storage_balance_callback` — have multiple early-return failure paths that drop `caller_deposit` without issuing any refund, permanently locking the caller's NEAR inside the wallet contract.

---

### Finding Description

`inner_rlp_execute` captures the caller's attached deposit into a `CallerDeposit` struct and passes it through the promise chain so that `rlp_execute_callback` can refund it on failure: [1](#0-0) 

For two `TransactionKind` variants, the promise chain goes through an intermediate callback before reaching `rlp_execute_callback`:

- **`EOABaseTokenTransfer` with address check**: `inner_rlp_execute` calls the address registrar, then `address_check_callback(target, action, caller_deposit)`.
- **`ERC20Transfer`**: `inner_rlp_execute` calls `storage_balance_of`, then `nep_141_storage_balance_callback(token_id, receiver_id, action, caller_deposit)`. [2](#0-1) 

In `address_check_callback`, four early-return paths exist that never touch `caller_deposit`:

1. `PromiseResult::Failed` (registrar call panicked or ran out of gas): [3](#0-2) 

2. Unexpected registrar response (JSON parse failure): [4](#0-3) 

3. Address maps to an existing named account, non-self signer: [5](#0-4) 

4. `action_to_promise` error: [6](#0-5) 

In `nep_141_storage_balance_callback`, the same pattern repeats:

1. `PromiseResult::Failed` (`storage_balance_of` panicked or ran out of gas): [7](#0-6) 

2. Unexpected NEP-141 response (JSON parse failure): [8](#0-7) 

3. Non-`FunctionCall` action when receiver is unregistered: [9](#0-8) 

4. `action_to_promise` error in the `Some(_)` branch: [10](#0-9) 

In every one of these paths, `caller_deposit` is received as a parameter, then silently dropped. The NEAR it represents was already transferred into the wallet contract's balance when `rlp_execute` was called, and no subsequent receipt returns it.

By contrast, `rlp_execute_callback` — which is only reached when the intermediate call succeeds — correctly issues the refund: [11](#0-10) 

---

### Impact Explanation

Any NEAR attached by an external caller to `rlp_execute` is permanently locked inside the wallet contract whenever the intermediate registrar or NEP-141 `storage_balance_of` call fails. The wallet contract is deployed as a global contract for all ETH-implicit accounts on NEAR; the locked balance accumulates in the wallet contract's account and is irrecoverable by the caller. This is a direct, permanent financial loss for any relayer or user who attaches a deposit and whose transaction routes through either of these two intermediate callbacks.

---

### Likelihood Explanation

Both intermediate calls can fail without any attacker involvement:
- The address registrar contract can panic, run out of gas, or be temporarily unavailable.
- Any NEP-141 token contract's `storage_balance_of` can panic or run out of gas.

A malicious token contract deployed by an attacker could deliberately panic on `storage_balance_of` to guarantee the deposit is locked. Since `rlp_execute` is `#[payable]` and external callers routinely attach deposits for base-token or ERC-20 transfers, the conditions for triggering this loss are reachable in normal operation.

---

### Recommendation

In every early-return failure path inside `address_check_callback` and `nep_141_storage_balance_callback`, issue a refund transfer before returning, mirroring the logic already present in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This should be applied to all four early-return sites in `address_check_callback` and all four in `nep_141_storage_balance_callback`.

---

### Proof of Concept

**Trigger path for `address_check_callback`:**

1. An external relayer calls `rlp_execute` on an ETH-implicit wallet contract, attaching e.g. 1 NEAR as deposit. The Ethereum transaction encodes an `EOABaseTokenTransfer` to another ETH-implicit address that requires an address-registrar check.
2. `inner_rlp_execute` creates `caller_deposit = Some(CallerDeposit { account_id: relayer, yocto_near: 1_000_000_000_000_000_000_000_000 })` and schedules `registrar.lookup(address).then(address_check_callback(..., caller_deposit))`.
3. The registrar call panics (out of gas, contract bug, etc.), so `env::promise_result(0) == PromiseResult::Failed`.
4. `address_check_callback` hits the `PromiseResult::Failed` arm at line 142 and returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` — the receipt succeeds from the runtime's perspective, so no automatic deposit refund is issued.
5. The relayer's 1 NEAR is now permanently held in the wallet contract's balance with no mechanism to recover it.

**Trigger path for `nep_141_storage_balance_callback`:**

1. An external relayer calls `rlp_execute` with an attached deposit. The Ethereum transaction encodes an ERC-20 transfer (emulated `ft_transfer_call`).
2. `inner_rlp_execute` schedules `token_contract.storage_balance_of(receiver).then(nep_141_storage_balance_callback(..., caller_deposit))`.
3. The token contract's `storage_balance_of` panics (malicious or buggy contract), so `PromiseResult::Failed`.
4. `nep_141_storage_balance_callback` hits line 204 and returns an error response without refunding `caller_deposit`.
5. The attached NEAR is locked in the wallet contract.

/// A data type to keep track of the deposit given by an external caller.
/// This allows us to refund the caller's deposit if the cross-contract call fails.
#[derive(Debug, PartialEq, Eq, Clone, serde::Serialize, serde::Deserialize)]
pub struct CallerDeposit {
    pub account_id: AccountId,
    pub yocto_near: NonZeroU128,

impl CallerDeposit {
    pub fn new(context: &ExecutionContext) -> Option<Self> {
        // Only track for external (non-self) callers

            return None;
        }

        NonZeroU128::new(context.attached_deposit.as_yoctonear()).map(|yocto_near| Self {
            account_id: context.predecessor_account_id.clone(),
            yocto_near,
        })
    }
