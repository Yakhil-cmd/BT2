### Title
`MsgTurnBridge` Lacks Permission Check and Has No Implementation, Permanently Disabling the Bridge Circuit Breaker - (File: x/cronos/keeper/msg_server.go)

### Summary
`MsgTurnBridge` is explicitly designated as a permission-restricted message in ADR-009 (requiring `CanTurnBridge`), but its keeper implementation performs zero authorization checks and returns `nil, nil` with no state change. Any unprivileged user can call it and receive a success response, while no authorized user can actually disable the gravity bridge in an emergency.

### Finding Description
In `x/cronos/keeper/msg_server.go`, the `TurnBridge` handler is a stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

ADR-009 explicitly lists `MsgTurnBridge` as one of exactly two messages that must require permission:

> "For now, there are only two messages that require permission: `MsgUpdateTokenMapping` and `MsgTurnBridge`." [2](#0-1) 

The permission system (`HasPermission`, `CanTurnBridge`) is fully implemented and working. `UpdateTokenMapping` correctly enforces it:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [3](#0-2) 

The `CanTurnBridge` constant is defined alongside `CanChangeTokenMapping`: [4](#0-3) 

`TurnBridge` calls neither `HasPermission` nor any bridge state mutation. The gRPC handler routes all `MsgTurnBridge` calls directly to this stub: [5](#0-4) 

The integration test `test_gravity_turn_bridge` confirms the intended behavior — a non-admin call should fail with a permission error, and an admin call should actually disable the bridge: [6](#0-5) 

Neither outcome is achievable with the current stub.

### Impact Explanation
Two distinct impacts compound each other:

1. **Auth bypass (High):** Any unprivileged address can submit `MsgTurnBridge` and receive a success response (`code == 0`), completely bypassing the `CanTurnBridge` permission check that ADR-009 mandates. This is a direct bypass of Cronos bridge authorization.

2. **Permanent inability to disable the bridge (High):** Because the function performs no state change, the gravity bridge circuit breaker is permanently non-functional. In an emergency (e.g., an active bridge exploit draining funds), the admin or any permissioned address cannot halt bridge operations. This is a permanent inability to protect bridge/conversion flows under normal network assumptions.

### Likelihood Explanation
The `MsgTurnBridge` endpoint is publicly reachable via gRPC by any address that can submit a Cosmos SDK transaction. No special preconditions are required. The attacker path is a single transaction submission. The broken circuit breaker is always present regardless of attacker action.

### Recommendation
Implement `TurnBridge` with:
1. A permission check mirroring `UpdateTokenMapping`: `if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`
2. An actual bridge state mutation (e.g., setting a `BridgeEnabled` parameter in module params or a dedicated KV store key) that downstream bridge hooks read before processing outgoing transfers.

### Proof of Concept
```
1. Attacker submits: MsgTurnBridge{Sender: "cosmos1attacker...", Enable: false}
   → Transaction succeeds (code == 0), no permission error.
   → Bridge state: unchanged (still enabled).

2. Admin submits: MsgTurnBridge{Sender: "cosmos1admin...", Enable: false}
   → Transaction succeeds (code == 0).
   → Bridge state: still unchanged (still enabled).

3. Active bridge exploit continues draining funds; circuit breaker is inoperable.
```

The root cause is identical in class to the external report: `FCFSMint()` should only be callable by whitelisted users but has no whitelist check; `MsgTurnBridge` should only be callable by addresses with `CanTurnBridge` permission but has no permission check. Both are auth-bypass stubs that accept any caller.
