### Title
Fixed 2,300-Gas Stipend in COA `Deposit` and `Transfer` Direct Calls Permanently Blocks FLOW Delivery to EVM Smart Contracts with Non-Trivial Receive Logic - (`File: fvm/evm/types/call.go`)

---

### Summary

Flow EVM's COA (Cadence-Owned Account) `Deposit` and `Transfer` operations construct `DirectCall` messages with a hardcoded gas limit of `21,000 + 2,300 = 23,300` gas. Only 2,300 gas is available for the recipient's EVM `receive()` or `fallback()` function — the same stipend that Solidity's deprecated `transfer()` imposes. Any EVM smart contract whose `receive()` or `fallback()` consumes more than 2,300 gas will permanently be unable to receive FLOW tokens via these COA primitives, and the calling Cadence transaction will revert every time it tries.

---

### Finding Description

In `fvm/evm/types/call.go`, three constants are defined:

```go
IntrinsicFeeForTokenTransfer = gethParams.TxGas  // 21_000

// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300

DepositCallGasLimit  = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

These constants are consumed by all three token-movement constructors:

- `NewDepositCall` sets `GasLimit: DepositCallGasLimit` (23,300)
- `NewWithdrawCall` sets `GasLimit: WithdrawCallGasLimit` (23,300)
- `NewTransferCall` sets `GasLimit: DefaultGasLimitForTokenTransfer` (23,300) [2](#0-1) 

In `fvm/evm/handler/handler.go`, `Account.Deposit` and `Account.Transfer` invoke these constructors unconditionally with no way for the caller to supply a higher gas limit:

```go
func (a *Account) Deposit(v *types.FLOWTokenVault) {
    res, err := a.fch.executeAndHandleCall(
        types.NewDepositCall(bridge, a.address, v.Balance(), bridgeAccount.Nonce()),
        ...
    )
    panicOnErrorOrInvalidOrFailedState(res, err)
}

func (a *Account) Transfer(to types.Address, balance types.Balance) {
    res, err := a.executeAndHandleAuthorizedCall(
        types.NewTransferCall(a.address, to, balance, a.Nonce()),
        ...
    )
    panicOnErrorOrInvalidOrFailedState(res, err)
}
``` [3](#0-2) 

When the EVM execution runs out of gas (because the recipient's `receive()` needs more than 2,300 gas), `panicOnErrorOrInvalidOrFailedState` panics with an EVM error, reverting the entire Cadence transaction:

```go
if res != nil && res.Failed() {
    panic(fvmErrors.NewEVMError(res.VMError))
}
``` [4](#0-3) 

The Cadence-level `CadenceOwnedAccount.deposit` and `CadenceOwnedAccount.withdraw` in `contract.cdc` both ultimately route through these same Go-layer calls, so the restriction is protocol-wide. [5](#0-4) 

---

### Impact Explanation

Any EVM smart contract deployed on Flow EVM whose `receive()` or `fallback()` function consumes more than 2,300 gas — for example, a vault that emits an event on receipt, a multisig that updates accounting state, or any proxy contract that adds call overhead — **can never receive FLOW tokens** via the COA `Deposit` or `Transfer` primitives. Every attempt reverts the Cadence transaction. Because the gas cap is a compile-time constant with no override path, this is a permanent, protocol-enforced exclusion. FLOW tokens intended for such contracts are not lost (the sender's transaction reverts), but the recipient contract is structurally unreachable via these primitives, making it impossible to fund or interact with it using FLOW through the COA bridge layer.

---

### Likelihood Explanation

The condition is easily triggered by any unprivileged user who deploys an EVM contract with a non-trivial `receive()` function. Common patterns — emitting a `Received` event, updating a balance mapping, calling into another contract — all exceed 2,300 gas. No special privileges, leaked keys, or node compromise are required. The attacker-controlled entry path is: deploy such a contract, then observe that any COA `Transfer` or `Deposit` targeting it always reverts.

---

### Recommendation

Replace the hardcoded `DefaultGasLimitForTokenTransfer` stipend with a caller-supplied gas limit parameter in `NewDepositCall` and `NewTransferCall`, similar to how `NewDeployCall` and `NewContractCall` already accept a `gasLimit uint64` argument. Expose this parameter through the Cadence `CadenceOwnedAccount.deposit` and `transfer` functions so callers can specify sufficient gas for the recipient's logic. As a minimum safe default, use a value substantially higher than 2,300 (e.g., the block gas limit cap), and document that the 2,300-gas assumption is only valid for EOA recipients.

---

### Proof of Concept

1. Deploy an EVM contract on Flow EVM with a `receive()` function that emits one event (costs ~1,500 gas for the LOG opcode plus topic hashing — already close to the 2,300 limit; adding a single SSTORE pushes it well over).
2. From a Cadence transaction, create a COA and call `cadenceOwnedAccount.deposit(from: <-vault)` targeting that contract's EVM address, or call `Account.Transfer(contractAddress, balance)` from the handler layer.
3. Observe that `panicOnErrorOrInvalidOrFailedState` panics with an out-of-gas EVM error and the Cadence transaction reverts every time, regardless of how much FLOW the sender holds.

// Withdraw deducts the balance from the account and

func (a *Account) Withdraw(b types.Balance) *types.FLOWTokenVault {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMWithdraw).End()

	res, err := a.executeAndHandleAuthorizedCall(
		types.NewWithdrawCall(
			a.fch.addressAllocator.NativeTokenBridgeAddress(),
			a.address,
			b,
			a.Nonce(),
		),
		b,
		true,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)

	return types.NewFlowTokenVault(b)
}
