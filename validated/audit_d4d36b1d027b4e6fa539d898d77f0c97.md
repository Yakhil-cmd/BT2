### Title
Missing Access Control on `update_rewards` Allows Any Caller to Block Per-Block Reward Distribution for All Stakers - (File: src/staking/staking.cairo)

### Summary
`update_rewards` in the `Staking` contract is exposed as a public external function with no caller restriction, despite the protocol specification requiring it to be callable only by the Starkware sequencer. Because the function updates a single global `last_reward_block` storage variable, any unprivileged caller can invoke it once per block for any staker, permanently preventing the sequencer from distributing rewards to all other stakers in that block.

### Finding Description
`update_rewards` is implemented under `#[abi(embed_v0)]` in `StakingRewardsManagerImpl`, making it a fully public external entrypoint. [1](#0-0) 

The function's only guard against repeated calls is a global `last_reward_block` check: [2](#0-1) 

`last_reward_block` is a single contract-wide storage variable (not per-staker). Once any call to `update_rewards` succeeds in a given block — regardless of which staker address is passed — `last_reward_block` is set to that block number, and every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

The protocol specification explicitly restricts this function: [3](#0-2) 

No `assert_caller_is_sequencer` or equivalent check exists in the implementation. The function accepts any `staker_address` that is currently active, which is public on-chain information.

### Impact Explanation
An attacker who is (or knows of) any active staker can call `update_rewards(staker_address: <any_active_staker>, disable_rewards: false)` once per block. This:

1. Sets `last_reward_block` to the current block.
2. Causes the sequencer's own call for every other staker in that block to revert with `REWARDS_ALREADY_UPDATED`.
3. Permanently forfeits that block's rewards for all stakers the sequencer would have served — the missed block can never be retroactively processed.

Repeated every block, this permanently freezes all unclaimed yield accrual for the entire staker set. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

The attacker gains nothing financially; this is pure griefing. The `disable_rewards: true` path still updates `last_reward_block`, so even a zero-reward call blocks the sequencer.

### Likelihood Explanation
The entry path requires no privilege: any EOA can call `update_rewards` on the deployed `Staking` contract with any known active staker address. Active staker addresses are discoverable from on-chain events (`NewStaker`). The cost is a single transaction per block. The attack is trivially repeatable and requires no special setup.

### Recommendation
Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the configured sequencer address (analogous to the existing `CALLER_IS_NOT_ATTESTATION_CONTRACT` guard used in `update_rewards_from_attestation_contract`): [4](#0-3) 

Alternatively, restrict the function to an `onlySequencer` role stored in contract configuration, consistent with how other privileged callers are enforced throughout the codebase.

### Proof of Concept
1. Deploy the staking system; let staker `S` stake normally.
2. Advance epochs so `S` is eligible for consensus rewards.
3. From any unprivileged address `A` (not the sequencer), call:
   ```
   IStakingRewardsManager(staking).update_rewards(staker_address: S, disable_rewards: false)
   ```
4. Observe: call succeeds, `last_reward_block` is now set to the current block, `StakerRewardsUpdated` event is emitted.
5. The sequencer now attempts `update_rewards` for any other staker `S2` in the same block — it reverts with `REWARDS_ALREADY_UPDATED`.
6. `S2`'s block rewards are permanently lost. Repeat step 3 every block to freeze all reward accrual indefinitely. [5](#0-4) [6](#0-5)
