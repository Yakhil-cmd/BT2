### Title
Pausing Engine Between `add_relayer_key` Batch and `store_relayer_key_callback` Locks NEAR Allowance — (`engine/src/contract_methods/admin.rs`)

### Summary

`add_relayer_key` creates a single `PromiseBatchAction` containing two actions: `AddFunctionCallKey` (which deducts the NEAR allowance from the engine account balance) and a `FunctionCall` to `store_relayer_key_callback`. In NEAR's execution model these are **two separate receipts**: the batch receipt (which adds the key and schedules the callback) is atomic, but the callback receipt executes independently in a later block. If `pause_contract` is called in that window, `store_relayer_key_callback` hits `require_running` and returns `ERR_PAUSED`, leaving the NEAR function-call key on the engine account with its allowance locked but the key absent from the internal registry. The only removal path, `remove_relayer_key`, also requires `require_running` and checks the internal registry, so it cannot reclaim the allowance even after the engine is resumed.

### Finding Description

`add_relayer_key` builds a batch promise: [1](#0-0) 

The first action (`AddFunctionCallKey`) deducts `allowance` NEAR from the engine account balance and adds the key. The second action schedules `store_relayer_key_callback` as a new receipt. These two receipts are **not** atomic with each other.

`store_relayer_key_callback` unconditionally checks `require_running` before writing to the internal registry: [2](#0-1) 

`require_running` returns `ERR_PAUSED` if `state.is_paused` is true: [3](#0-2) 

`pause_contract` can be called by the owner at any time while the engine is running: [4](#0-3) 

`remove_relayer_key` also requires `require_running` and calls `engine::remove_function_call_key`, which checks the internal registry: [5](#0-4) 

Because the key was never written to the internal registry (callback failed), `remove_relayer_key` cannot find it and cannot issue the `DeleteKey` promise that would return the allowance to the account.

### Impact Explanation

The NEAR allowance attached to `AddFunctionCallKey` is deducted from the engine account balance at the time the batch receipt executes. If the callback fails, the key exists on the engine account but is invisible to the engine's own key-management logic. The allowance remains locked until an out-of-band mechanism (e.g., the owner adding a full-access key via `attach_full_access_key` and then manually deleting the orphaned function-call key) is used. This constitutes **temporary freezing of funds** (the NEAR allowance is inaccessible through any normal engine interface until manual admin intervention).

### Likelihood Explanation

The window between the batch receipt and the callback receipt spans at least one NEAR block. An owner performing routine maintenance (pausing for an upgrade, migration, or emergency) during that window — without knowing a `store_relayer_key_callback` is pending — is a realistic operational scenario. No attacker compromise is required; the owner's own legitimate `pause_contract` call is sufficient.

### Recommendation

Remove the `require_running` guard from `store_relayer_key_callback`. The callback is a private, self-scheduled receipt (`env.assert_private_call()` is already enforced) and its only job is to record a key that was already added by the preceding batch action. Blocking it on pause state creates the invariant violation. Alternatively, if the callback fails for any reason, emit a `DeleteKey` promise to undo the `AddFunctionCallKey` and refund the allowance.

### Proof of Concept

1. Deploy the engine with a configured `key_manager`.
2. `key_manager` calls `add_relayer_key` with `attached_deposit >= 100` yoctoNEAR.
3. The batch receipt executes: `AddFunctionCallKey` adds the key and deducts the allowance from the engine account balance.
4. Before the `store_relayer_key_callback` receipt executes, the owner calls `pause_contract`.
5. `store_relayer_key_callback` executes: `require_running` returns `ERR_PAUSED`; `engine::add_function_call_key` is never called.
6. Assert: the NEAR function-call key exists on the engine account (verifiable via `view_access_key`), but the internal registry has no entry for it.
7. Assert: calling `remove_relayer_key` (even after `resume_contract`) fails because the key is absent from the internal registry.
8. Assert: the allowance NEAR is locked in the engine account with no standard reclaim path.

pub fn remove_relayer_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;

        require_running(&state)?;
        require_key_manager_only(&state, &env.predecessor_account_id())?;

        let args: RelayerKeyArgs = serde_json::from_slice(&io.read_input().to_vec())
            .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;

        engine::remove_function_call_key(&mut io, &args.public_key)?;

        let action = PromiseAction::DeleteKey {
            public_key: args.public_key,
        };
        let promise = PromiseBatchAction {
            target_account_id: env.current_account_id(),
            actions: vec![action],
        };

        let promise_id = handler.promise_create_batch(&promise);
        handler.promise_return(promise_id);

        Ok(())
    })
}
