### Title
`receive_base_tokens` Bypasses Silo Address Whitelist, Freezing Deposited ETH in Non-Whitelisted Addresses — (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode with the `Address` whitelist enabled, `receive_erc20_tokens` correctly redirects incoming ERC-20 tokens to a configured fallback address when the intended recipient is not whitelisted. However, the parallel function `receive_base_tokens` performs no such check: it credits base ETH tokens directly to any caller-supplied recipient address without consulting the Silo whitelist. Because non-whitelisted addresses are blocked from submitting transactions by `assert_access`, any base ETH deposited to a non-whitelisted address is frozen until an owner manually intervenes.

---

### Finding Description

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to one of two paths depending on whether the predecessor is the ETH connector account:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)          // base ETH path
} else {
    engine.receive_erc20_tokens(...)           // ERC-20 path
};
```

`receive_erc20_tokens` (lines 818–822 of `engine/src/engine.rs`) enforces the Silo whitelist:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

`receive_base_tokens` (lines 773–790 of `engine/src/engine.rs`) has no equivalent guard:

```rust
pub fn receive_base_tokens(&mut self, args: &FtOnTransferArgs) -> ... {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;          // no whitelist check
    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance.checked_add(amount)...;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

The Silo whitelist check that gates transaction submission (`assert_access` in `engine/src/engine.rs` lines 1756–1775) is entirely separate from the deposit path and is never consulted during `ft_on_transfer`. The result is that base ETH can be minted to any EVM address, but only whitelisted addresses can subsequently move those funds.

---

### Impact Explanation

**High — Temporary freezing of funds.**

When the Silo `Address` whitelist is active, base ETH deposited to a non-whitelisted EVM address is irrecoverable by the recipient: `assert_access` blocks every `submit` call from that address. The tokens remain frozen until the Silo owner either adds the address to the whitelist or uses a privileged `call` to move the balance. This is a direct, concrete fund-freeze reachable through the normal bridge deposit flow, not a theoretical edge case.

---

### Likelihood Explanation

Silo mode with an active `Address` whitelist is an explicitly supported production configuration (documented in `WhitelistKind::Address`). The deposit path (`ft_transfer_call` on the ETH connector → `ft_on_transfer` → `receive_base_tokens`) is the standard mechanism for bridging ETH into Aurora. Any NEAR account can trigger it by specifying an arbitrary EVM recipient in the message field. The freeze can occur accidentally (user sends to a not-yet-whitelisted address) or deliberately (griefing: attacker sends dust to a victim's non-whitelisted address, locking it out of the fallback mechanism). No special privileges are required.

---

### Recommendation

Apply the same fallback-address guard to `receive_base_tokens` that already exists in `receive_erc20_tokens`:

```rust
pub fn receive_base_tokens(&mut self, args: &FtOnTransferArgs) -> ... {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut receipient = message_data.recipient;

    // Mirror the Silo whitelist guard from receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
    {
        receipient = fallback_address;
    }

    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance.checked_add(amount)...;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

---

### Proof of Concept

**Setup**: Deploy Aurora in Silo mode. Enable the `Address` whitelist. Configure a fallback address. Do **not** whitelist Alice's EVM address.

**Steps**:
1. Alice (or any NEAR account) calls `ft_transfer_call` on the ETH connector contract, targeting the Aurora engine, with `msg` set to Alice's EVM address hex and `amount = 1_000_000`.
2. The ETH connector calls `ft_on_transfer` on Aurora.
3. `ft_on_transfer` detects the predecessor is the connector account and calls `receive_base_tokens`.
4. `receive_base_tokens` parses Alice's address from `msg` and calls `set_balance`, crediting 1,000,000 wei to Alice's non-whitelisted EVM address — **no whitelist check occurs**.
5. Alice attempts to submit any EVM transaction. `assert_access` → `silo::is_allow_submit` → `is_address_allowed` finds Alice's address absent from the `Address` whitelist → returns `EngineErrorKind::NotAllowed`.
6. Alice's 1,000,000 wei is frozen.

**Contrast**: Repeat with an ERC-20 token. `receive_erc20_tokens` detects Alice is not whitelisted and mints to the fallback address instead — funds are recoverable. The base ETH path has no such protection. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
