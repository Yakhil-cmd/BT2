### Title
Unchecked NEAR Promise Result in `ExitToNear` Precompile Causes Permanent Loss of Bridged Funds - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile burns EVM-side ETH or ERC-20 tokens and schedules a NEAR `ft_transfer` promise **without any failure callback**. If the NEAR-side promise fails, the burned tokens are permanently lost with no refund path — a direct analog to the GroupBuy unchecked `call` return value bug.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) is the bridge exit path: it burns ETH or ERC-20 tokens on the EVM side and schedules a NEAR `ft_transfer` (or `ft_transfer_call`) promise to deliver the equivalent NEP-141 tokens to the recipient.

The refund callback is gated behind the `error_refund` feature flag: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

When `error_refund` is absent and `transfer_near_args` is `None` (the common case for plain ETH and ERC-20 exits), `callback_args` equals its default value, so the promise is created **without any callback**: [2](#0-1) 

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

On the callback side, the refund branch is only reachable when `args.refund` is `Some(...)`, which requires `error_refund` to have been compiled in: [3](#0-2) 

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    ...
    None
} else if let Some(args) = args.refund {   // ← None when error_refund is off
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op on failure
};
```

The engine's own integration test explicitly documents this loss: [4](#0-3) 

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

When the NEAR-side `ft_transfer` promise fails (e.g., Aurora's NEP-141 balance is insufficient, the recipient account does not exist, or any transient NEAR-side condition causes a revert), the EVM-side tokens have already been burned. Because there is no callback to re-mint or refund them, the funds are permanently destroyed. This is not a theoretical edge case: the test suite explicitly models and confirms the loss scenario.

---

### Likelihood Explanation

**High.** The `ft_transfer` promise can fail for multiple realistic reasons that are outside the user's control:

- The Aurora contract's NEP-141 balance is temporarily drained (as demonstrated in the test by calling `ft_transfer` to drain the account before the exit).
- The recipient NEAR account does not exist or is not registered with the NEP-141 contract.
- Transient NEAR runtime conditions (gas exhaustion on the NEAR side, storage deposit missing).

Any unprivileged EVM user who calls the `ExitToNear` precompile (directly or via an ERC-20 `withdrawToNear`) is exposed to this loss if the feature is absent from the deployed binary.

---

### Recommendation

1. Make `error_refund` a **default feature** in `engine-precompiles/Cargo.toml` and `engine/Cargo.toml` so it is always compiled into production builds.
2. Alternatively, remove the feature flag entirely and unconditionally attach the refund callback to every `ExitToNear` promise, mirroring the fix applied to GroupBuy (`require` that the call succeeded).
3. Add a deployment-time assertion or CI gate that rejects any WASM build produced without `error_refund`.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Alice holds 1 ETH in the EVM and calls the `ExitToNear` precompile to bridge it to `alice.near`.
3. Before the NEAR `ft_transfer` promise executes, the Aurora contract's NEP-141 ETH balance is drained to zero (e.g., by a concurrent exit or an admin `ft_transfer`).
4. The `ft_transfer` promise fails on the NEAR side.
5. Because no callback exists, `exit_to_near_precompile_callback` is never invoked with refund args.
6. Alice's 1 ETH is permanently gone: burned in the EVM, never delivered on NEAR.

The test `test_exit_to_near_eth_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 717–780) reproduces exactly this scenario and confirms the balance loss when `error_refund` is absent. [5](#0-4)
