### Title
COA Ownership Proof Signature Replayable Across Chains via Missing Domain Binding in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over an arbitrary `signedData` blob using the fixed domain tag `"FLOW-V0.0-user"`, but never asserts that `signedData` encodes the target COA's EVM address, the chain ID, or any other deployment-specific context. A valid COA ownership proof produced on one Flow network (e.g., testnet) can be replayed verbatim on another (e.g., mainnet) to forge ERC-1271 authentication for the same Cadence account's COA on the target chain. The code itself acknowledges the absence of this binding in a comment but dismisses it as low-risk; the ERC-1271 integration in `coa.sol` makes the risk concrete and externally reachable.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. It verifies that the provided `signatures` are valid over `signedData` under the fixed domain tag `"FLOW-V0.0-user"` using the Cadence account's keys.
2. It checks that the COA resource at the given `path` has an EVM address matching the supplied `evmAddress` argument. [1](#0-0) 

Critically, neither check binds `signedData` to `evmAddress`, the chain ID, or any other deployment-specific value. The code explicitly acknowledges this gap: [2](#0-1) 

The COA Solidity contract (`fvm/evm/handler/coa/coa.sol`) implements ERC-1271 by forwarding `isValidSignature(_hash, _sig)` directly to the `verifyCOAOwnershipProof` precompile, passing `address(this)` as the COA address and the caller-supplied `_hash` as `signedData`: [3](#0-2) 

The precompile decodes `_sig` as an encoded `COAOwnershipProof` and delegates to `validateCOAOwnershipProof`: [4](#0-3) 

Because `signedData` (`_hash`) is never required to commit to the chain ID or the specific COA address, a proof signed on one chain is cryptographically valid on any other chain where the same Cadence account (same keys) exists.

**Attack path:**

1. Alice holds a Cadence account with the same key pair on both Flow testnet and mainnet (common for users who test before deploying).
2. Alice signs `FLOW-V0.0-user || hash_X` on testnet — for example, to authorize a gasless ERC-1271 permit on a testnet DeFi protocol. The encoded `COAOwnershipProof` (containing her Cadence address, capability path, key indices, and signatures) is observable on-chain in the transaction calldata.
3. An attacker extracts this proof from testnet.
4. On mainnet, the attacker calls `isValidSignature(hash_X, encoded_proof)` on Alice's mainnet COA (same EVM address, same Cadence account structure).
5. `validateCOAOwnershipProof` runs on mainnet: the signatures verify (same keys), and the COA at the given path has the correct EVM address. The function returns `isValid: true`.
6. The attacker has forged a valid ERC-1271 signature for Alice's mainnet COA over `hash_X` — without Alice ever signing anything on mainnet.

The `COAOwnershipProofInContext` struct confirms that `SignedData` is passed through without any chain-binding enforcement: [5](#0-4) 

### Impact Explanation

An unprivileged attacker who can observe a COA ownership proof on one Flow network can replay it on another network where the same Cadence account exists. Through the ERC-1271 `isValidSignature` interface, this allows the attacker to impersonate the COA owner for any EVM-side operation that relies on ERC-1271 authorization (token permit approvals, gasless transaction authorization, NFT marketplace order signing, multisig wallet authentication, etc.) — without the account owner's consent on the target chain. This constitutes unauthorized authorization of on-chain asset operations.

### Likelihood Explanation

The preconditions are realistic: many Flow users create accounts on testnet before mainnet using the same key material, and COA ownership proofs are emitted in transaction calldata where they are publicly observable. The `validateCOAOwnershipProof` function is `access(all)` and callable from any Cadence script or EVM transaction, requiring no privileged access. The ERC-1271 surface is actively used by DeFi protocols integrating with COA wallets.

### Recommendation

Enforce that `signedData` commits to the deployment context. Specifically, `validateCOAOwnershipProof` should assert that `signedData` encodes at least the target `evmAddress` (20 bytes) and the chain ID. The simplest approach is to require callers to sign a structured message such as `keccak256(abi.encode(chainId, evmAddress, applicationData))` and verify this structure inside `validateCOAOwnershipProof` before accepting the proof. Alternatively, incorporate the chain ID into the domain separation tag (analogous to EIP-712's `domainSeparator`) so that signatures are chain-scoped at the cryptographic level.

### Proof of Concept

```
// Testnet: Alice signs hash_X for her COA at testnet path "coa"
// Attacker observes the encoded COAOwnershipProof on-chain:
//   { Address: alice_cadence_addr, CapabilityPath: "coa",
//     KeyIndices: [0], Signatures: [sig_over_hash_X] }

// Mainnet: Attacker calls isValidSignature on Alice's mainnet COA
// coa.sol::isValidSignature(hash_X, encoded_proof)
//   -> verifyCOAOwnershipProof(address(this)=alice_mainnet_coa, hash_X, encoded_proof)
//   -> validateCOAOwnershipProof(
//        address: alice_cadence_addr,   // same on both chains
//        path: "coa",                   // same on both chains
//        signedData: hash_X,            // same bytes, no chain binding
//        keyIndices: [0],
//        signatures: [sig_over_hash_X], // valid: same keys on both chains
//        evmAddress: alice_mainnet_coa  // passes: COA at "coa" == alice_mainnet_coa
//      )
// Result: isValid = true  ← forged ERC-1271 authorization on mainnet
```

The root cause is at `fvm/evm/stdlib/contract.cdc` lines 1082–1086: `signedData` is verified cryptographically but never checked to contain `evmAddress` or chain context, making the proof portable across any deployment of the same Cadence account. [6](#0-5)
