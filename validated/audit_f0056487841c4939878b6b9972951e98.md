### Title
Unbounded Loop in `calculate_rewards` Over Pool Member Balance Trace Can Permanently Freeze Delegator Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` internal function in `src/pool/pool.cairo` iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because there is no cap on how many entries can accumulate in that trace, a delegator who makes balance changes across many epochs without claiming rewards can grow the trace until the `claim_rewards` transaction exceeds the Starknet gas limit, permanently freezing their unclaimed yield.

The code itself acknowledges the risk with the comment at lines 857–858:

> `// **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.`

### Finding Description

**How the trace grows**

Every call to `set_member_balance` (and its wrapper `increase_member_balance`) appends a new checkpoint to the `pool_member_epoch_balance` `Vec` for the calling pool member, keyed at `current_epoch + K`. [1](#0-0) 

The `insert` logic in the trace only *updates* the last entry when the key is identical; otherwise it *pushes* a new entry. [2](#0-1) 

Because the key is `current_epoch + K`, any call made in a different epoch than the previous call produces a distinct key and therefore a new entry. The following public entry-points all call `set_member_balance` / `increase_member_balance`:

- `enter_delegation_pool` → `set_member_balance`
- `add_to_delegation_pool` → `increase_member_balance` → `set_member_balance`
- `exit_delegation_pool_intent` → `set_member_balance`
- `enter_delegation_pool_from_staking_contract` (pool switch) → `set_member_balance` / `increase_member_balance` [3](#0-2) [4](#0-3) 

**The unbounded loop**

`claim_rewards` calls `calculate_rewards`, which iterates from `entry_to_claim_from` (the index saved after the last successful claim) up to the first trace entry whose epoch is ≥ the current epoch: [5](#0-4) 

Each iteration performs at least one `find_sigma` call, which itself reads from the `cumulative_rewards_trace` storage vector: [6](#0-5) 

Storage reads are among the most expensive operations in the Cairo VM. If a delegator accumulates N balance-change entries between two consecutive `claim_rewards` calls, the next claim iterates N times, each with multiple storage reads.

**Attack path**

1. Delegator enters the pool and calls `add_to_delegation_pool` (minimum non-zero amount) once per epoch, never calling `claim_rewards`.
2. Each call in a new epoch appends one entry to the trace; `entry_to_claim_from` stays at 0.
3. After enough epochs the delegator's trace contains hundreds or thousands of entries.
4. Any subsequent call to `claim_rewards` (by the delegator or their reward address) iterates over all accumulated entries and runs out of gas, reverting.
5. Because `entry_to_claim_from` is only advanced inside a successful `claim_rewards`, the state is never updated and every future attempt also reverts — permanently freezing the yield.

There is no on-chain cap on the number of entries in `pool_member_epoch_balance`. [7](#0-6) 

### Impact Explanation

A delegator's unclaimed STRK (or BTC) pool rewards become permanently unclaimable once the trace is large enough to cause an out-of-gas revert. The funds are locked in the pool contract with no recovery path, because every code path that would transfer them (`claim_rewards`, `exit_delegation_pool_action` which also reads rewards, `switch_delegation_pool`) goes through `calculate_rewards`.

This matches the allowed impact: **Permanent freezing of unclaimed yield (High)**.

### Likelihood Explanation

- Any unprivileged delegator can trigger this against themselves (self-griefing) or, more relevantly, it can happen organically: a long-term delegator who adjusts their stake frequently and claims rewards infrequently will naturally accumulate a large trace.
- The only cost to the attacker is gas for repeated `add_to_delegation_pool` calls (minimum amount each time) across many epochs.
- Starknet epochs are short; over months of normal protocol operation a diligent delegator could easily accumulate hundreds of entries.
- The code comment at line 857 explicitly acknowledges the loop is unbounded, confirming the developers are aware of the risk but have not bounded it. [8](#0-7) 

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` calls that would push the trace beyond a safe bound (e.g., 500 entries).
2. **Alternatively, require `claim_rewards` before each balance change** so that `entry_to_claim_from` is always advanced and the loop never iterates over more than a small constant number of entries.
3. **Batch-claim support:** Allow `claim_rewards` to accept a `max_entries` parameter so a delegator with a large trace can drain it incrementally across multiple transactions.

### Proof of Concept

```
1. Deploy the staking system and create a staker with a STRK pool.
2. Create a delegator with sufficient STRK.
3. Call `enter_delegation_pool(reward_address, min_amount)`.
4. In a loop, for i in 0..N_EPOCHS:
     a. advance_epoch()          // moves to a new epoch
     b. call add_to_delegation_pool(pool_member, 1)  // appends one new trace entry
   // Do NOT call claim_rewards at any point.
5. After N_EPOCHS iterations, pool_member_epoch_balance trace has N_EPOCHS entries.
6. Call claim_rewards(pool_member).
7. Observe: the transaction reverts with out-of-gas once N_EPOCHS is large enough
   (empirically determine the threshold; the loop body performs multiple storage reads
   per iteration via find_sigma).
8. All subsequent claim_rewards calls also revert — yield is permanently frozen.
```

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
