### Title
Incorrect Royalty Amount Calculation in WalletConnect `take_offer` Confirmation Dialog Causes User to Approve Wrong Spend Amount - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` uses a mathematically incorrect formula to compute the total royalty obligation when a `chia_wallet.take_offer` WalletConnect command involves multiple NFTs with royalties. It divides the payment amount by the number of NFTs before computing each royalty, producing a displayed total that is systematically understated. A user shown this confirmation dialog approves a transaction believing they are spending less than the wallet will actually deduct.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect `take_offer` command parsing to compute the `amountWithRoyalties` field shown in the confirmation dialog before the user approves the spend. [1](#0-0) 

The function contains this logic:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The actual royalty each NFT creator receives on-chain is computed independently on the **full** payment amount. The correct formula is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The implemented formula computes instead:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
               = (amount / N) × Σ (royaltyPercentage_i / 10_000)
```

This is smaller than the correct value by a factor of N (the number of NFTs). The on-chain settlement pays the correct royalties; only the pre-approval display is wrong.

The existing test encodes and validates the buggy output: [2](#0-1) 

The test asserts `amountWithRoyalties: '0.00010255'` for a 0.0001 XCH payment with two NFTs carrying 5% and 0.1% royalties. The correct value is 0.0001051 XCH (royalties of 5,000,000 + 100,000 mojos on the full amount, not on the halved amount).

The `withRoyaltyTotals` function passes the collected `royaltyPercentages` array directly to `formatAmountWithRoyalties` for every fungible line on the spending side: [3](#0-2) 

This result is surfaced through `walletDeltaToDisplay` and returned as the `walletDelta` field of `parseCommandDisplay`, which drives the WalletConnect confirmation UI: [4](#0-3) 

### Impact Explanation

The WalletConnect confirmation dialog shows the user an `amountWithRoyalties` value that is smaller than what the wallet will actually spend. The understatement grows with the number of NFTs: with N NFTs the displayed royalty total is 1/N of the correct value. A user who relies on this figure to decide whether to approve the transaction is deceived into authorizing a spend of a larger amount than they consented to. This maps directly to the allowed High impact: **WalletConnect state that causes a user to approve the wrong amount**.

### Likelihood Explanation

Any DApp that sends a `chia_wallet.take_offer` WalletConnect request for an offer containing two or more NFTs with non-zero royalties will trigger this path. No special attacker capability is required beyond constructing a valid multi-NFT offer, which is a standard Chia offer type. The user has no way to detect the discrepancy from the confirmation dialog alone.

### Recommendation

Remove the `splitAmount` division. Each royalty should be computed on the full `amount`:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

The test expectation for the two-NFT case should be updated to `'0.0001051'`.

### Proof of Concept

Given a `chia_wallet.take_offer` WalletConnect request for an offer where:
- The taker pays **0.0001 XCH** (100,000,000 mojos)
- Two NFTs are received: NFT-A with 5% royalty (`royalty_percentage: 500`) and NFT-B with 0.1% royalty (`royalty_percentage: 10`)

**Buggy display (N=2):**
- `splitAmount` = 100,000,000 / 2 = 50,000,000
- royalty from NFT-A = 50,000,000 × 500 / 10,000 = 2,500,000 mojos
- royalty from NFT-B = 50,000,000 × 10 / 10,000 = 50,000 mojos
- displayed total = 102,550,000 mojos = **0.00010255 XCH**

**Correct on-chain settlement:**
- royalty from NFT-A = 100,000,000 × 500 / 10,000 = 5,000,000 mojos
- royalty from NFT-B = 100,000,000 × 10 / 10,000 = 100,000 mojos
- actual total = 105,100,000 mojos = **0.0001051 XCH**

The user sees 0.00010255 XCH in the approval dialog but the wallet deducts 0.0001051 XCH — a ~2.5% discrepancy in this example, scaling worse with more NFTs or higher royalty percentages.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
