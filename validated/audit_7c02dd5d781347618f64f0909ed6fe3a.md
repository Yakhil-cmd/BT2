### Title
Invoke Transaction with Nonce=1 Bypasses Signature Verification via Overly Broad `account_tx_in_pool_or_recent_block` Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for any invoke transaction with `nonce=1` when the account's on-chain nonce is `0` and **any** transaction from that sender address exists in the mempool. The guard `account_tx_in_pool_or_recent_block` is too broad: it returns `true` for any pending transaction from the address, not exclusively for a `deploy_account`. An unprivileged attacker who observes a pending `deploy_account` (or any nonce-0 invoke) for a target address can submit a forged invoke with nonce=1 and an arbitrary signature, and the gateway will admit it to the mempool without ever calling `__validate__`.

### Finding Description

In `skip_stateful_validations` the skip condition is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` returns `true` whenever the address has **any** transaction in the pool or appears in a recently committed block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };
// ...
blockifier_validator.validate(account_tx)
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false` for an invoke transaction, the function returns `Ok(())` immediately after `perform_pre_validation_stage`, never reaching the `__validate__` call:

```rust
ApiTransaction::Invoke(_) => {
    tx.perform_pre_validation_stage(self.state(), &tx_context)?;
    if !tx.execution_flags.validate {
        return Ok(());
    }
    // `__validate__` call.
    let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [4](#0-3) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate hashes and nonce ordering — it does not verify signatures: [5](#0-4) 

The code comment acknowledges the intent but overstates the guarantee: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* A nonce-0 invoke (not a deploy_account) also satisfies the check, yet it does not establish that the account is deployed or that the nonce-1 invoke carries a valid signature. [6](#0-5) 

### Impact Explanation

An attacker can submit a forged invoke transaction (arbitrary signature, arbitrary calldata) for any address that has a pending transaction in the mempool, and the gateway will admit it without signature verification. This violates the invariant that every transaction admitted to the mempool has passed account-level signature validation. The matching impact category is:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The forged transaction will fail at batcher execution time (the batcher sets `validate: true` independently), so no wrong state is committed. However, the mempool is polluted with cryptographically invalid transactions, enabling targeted DoS against any address with a pending deploy_account.

### Likelihood Explanation

Mempool contents are observable (P2P gossip, RPC). Any observer who sees a `deploy_account` for address X can immediately submit a forged nonce-1 invoke for X. No privileged access, no special tooling, and no knowledge of the account's private key is required. The only prerequisite is that the target address has a pending transaction in the mempool with on-chain nonce still at 0.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account`** transaction is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the existing pool should be filtered by transaction type before the skip decision is made. Alternatively, remove the skip entirely and instead allow `__validate__` to be called against a simulated post-deploy state.

### Proof of Concept

1. Victim broadcasts `deploy_account` for address `X` (nonce=0, valid signature). It enters the mempool; `tx_pool.contains_account(X)` becomes `true`.
2. Attacker submits `invoke` for address `X` with `nonce=1` and a completely arbitrary signature (e.g., all zeros).
3. Gateway `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(X)` → `Nonce(0)` (account not yet deployed on-chain).
   - `validate_nonce`: nonce=1 is within `max_allowed_nonce_gap` of 0 → passes.
   - `validate_by_mempool`: checks duplicate hash and nonce ordering → passes.
   - `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
4. `run_validate_entry_point` is called with `skip_validate=true` → `execution_flags.validate=false`.
5. `blockifier_validator.validate(account_tx)` calls `perform_validations`, hits `if !tx.execution_flags.validate { return Ok(()); }` and returns without ever calling `__validate__`.
6. The forged invoke is forwarded to the mempool via `mempool_client.add_tx(...)` and accepted.

The forged transaction now occupies a mempool slot for address `X` at nonce=1. Repeating step 2 with different tx hashes (e.g., varying calldata) allows the attacker to flood the mempool with invalid transactions for any address that has a pending `deploy_account`.
