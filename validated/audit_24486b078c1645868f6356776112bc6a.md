### Title
Presignature Reuse via Clone-able `PresignOutput` Enables Private Key Extraction — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/presign.rs`)

### Summary

The `PresignOutput` struct in both the OT-based and robust ECDSA implementations derives `Clone` and is accepted by `rerandomize_presign()` via shared reference (`&PresignOutput`), meaning the same presignature can be rerandomized multiple times with different `(msg_hash, tweak)` inputs. A malicious coordinator who participates in presigning can silently reuse the same presignature across two or more signing sessions with different messages, producing signatures that share a multiplicatively related nonce. This is a classical ECDSA nonce-reuse scenario that allows full private key recovery.

### Finding Description

**Root cause — `PresignOutput` is `Clone` and `rerandomize_presign` borrows it:**

In `src/ecdsa/ot_based_ecdsa/mod.rs` the presignature output type is declared as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
``` [1](#0-0) 

The rerandomization entry point takes the presignature by shared reference:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
``` [2](#0-1) 

Because `rerandomize_presign` borrows `&PresignOutput` rather than consuming it, the same `PresignOutput` value can be passed to `rerandomize_presign` an unlimited number of times with different `RerandomizationArguments` (different `msg_hash`, `tweak`, or `big_r`-derived randomness). The library provides no use-tracking, no invalidation flag, and no session binding on the presignature object.

**Exploit path (malicious coordinator):**

1. The coordinator participates in the presigning protocol and obtains a `PresignOutput` containing `(big_r, k_i, sigma_i)`.
2. The coordinator calls `rerandomize_presign(&presign_output, &args_A)` to produce `RerandomizedPresignOutput` for message hash `h_A`.
3. The coordinator calls `rerandomize_presign(&presign_output, &args_B)` a second time with a different `h_B` (or different tweak `ε_B`), producing a second `RerandomizedPresignOutput`.
4. The coordinator runs two concurrent signing sessions — one with `h_A` and one with `h_B` — each using a different rerandomization of the same underlying nonce `k`.
5. Both resulting ECDSA signatures `(R_A, s_A)` and `(R_B, s_B)` are derived from the same secret nonce `k` (only scaled by different `delta` values). The relationship between the two nonces is known to the coordinator (it chose both `delta_A` and `delta_B`). Standard ECDSA nonce-reuse algebra then recovers the private key `x`.

The security documentation explicitly acknowledges this attack class:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [3](#0-2) 

The documentation mandates "Never reuse a presignature" but the library enforces nothing at the API level. [4](#0-3) 

The same structural issue exists in the robust ECDSA `PresignOutput`: [5](#0-4) 

The `RerandomizedPresignOutput` also derives `Clone`, so even after rerandomization the value can be duplicated and fed to `sign()` a second time: [6](#0-5) 

The `sign()` function accepts `presignature: RerandomizedPresignOutput` by value (a move), which would normally prevent reuse in safe Rust — but because `Clone` is derived, a caller can trivially call `.clone()` before passing it, bypassing the move-based protection entirely. [7](#0-6) 

### Impact Explanation

**Critical — private key extraction.**

ECDSA nonce reuse with a known nonce relationship is a textbook key-recovery attack. Given two signatures `(R_A, s_A)` and `(R_B, s_B)` produced from the same presignature with known rerandomization scalars `delta_A` and `delta_B`, the coordinator can solve for the master secret key `x` algebraically. This constitutes full extraction of the aggregate secret material, matching the Critical impact tier: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

### Likelihood Explanation

The coordinator role is a normal, non-privileged participant in the signing protocol. Any participant who is assigned the coordinator role — or who can influence which participant acts as coordinator — can execute this attack. No external assumptions, leaked keys, or cryptographic breaks are required. The attack requires only that the coordinator retain the `PresignOutput` value (trivially possible since it is `Clone` and `Serialize`) and initiate two signing sessions before the presignature is discarded.

### Recommendation

1. **Consume `PresignOutput` on rerandomization.** Change `rerandomize_presign` to take `presignature: PresignOutput` (by value, moving it) instead of `presignature: &PresignOutput`. This makes the Rust type system enforce single-use at the call site without any runtime overhead.

2. **Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`.** Deriving `Clone` on secret nonce material is dangerous because it silently bypasses move-based single-use enforcement. If serialization is needed for persistence, implement it explicitly with a documented warning.

3. **Add a session-binding check.** Bind the presignature to a unique session identifier (e.g., a random nonce committed to during presigning) and verify in `sign()` that the session ID matches, preventing cross-session reuse even if the type-system protection is somehow circumvented.

### Proof of Concept

```rust
// Coordinator holds presign_output: PresignOutput after presigning completes.

// Rerandomize the SAME presignature for two different messages.
let rerandomized_a = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,   // <-- shared reference, not consumed
    &args_for_msg_a,
).unwrap();

let rerandomized_b = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,   // <-- same presign_output reused
    &args_for_msg_b,   // different msg_hash / tweak
).unwrap();

// Run two signing sessions concurrently.
// sig_a = (R_a, s_a) and sig_b = (R_b, s_b) share nonce k (up to known scalar).
// Given delta_a, delta_b, h_a, h_b, R_a, R_b, s_a, s_b:
//   s_a = (h_a * k/delta_a + Rx_a * sigma/delta_a)
//   s_b = (h_b * k/delta_b + Rx_b * sigma/delta_b)
// Two equations, two unknowns (k, sigma) → solve for x = sigma/k = private key.
```

The `PresignOutput` struct's `Clone` derive and `rerandomize_presign`'s `&PresignOutput` signature are the necessary and sufficient conditions for this attack to be reachable by an unprivileged coordinator participant.
