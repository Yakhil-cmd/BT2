### Title
Gateway Admits Invoke Transactions with Unverified Signatures via `skip_stateful_validations` Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (which performs on-chain signature verification) for any Invoke transaction whose nonce equals 1 and whose sender address already has any transaction in the mempool or a recent block. An unprivileged attacker who observes a `deploy_account` transaction in the mempool can submit a crafted Invoke with nonce=1 and an arbitrary fake signature for that same address. The gateway admits the transaction without ever verifying the signature, inserting it into the mempool and occupying the nonce=1 slot. The legitimate user's first post-deployment Invoke is then rejected with `DuplicateNonce`.

---

### Finding Description

**Broken invariant:** Every transaction admitted to the mempool must have passed signature verification (the `__validate__` entry point).

**Root cause — `skip_stateful_validations`:**

```
crates/apollo_gateway/src/stateful_transaction_validator.rs, lines 429–460
```

The function returns `true` (skip) when all three conditions hold:

1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)` (nonce = 1)
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

**Effect on `run_validate_entry_point`:**

When `skip_validate = true`, the execution flags are set to `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

`StatefulValidator::perform_validations` then short-circuits before calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

**Mempool nonce-uniqueness enforcement:**

The mempool rejects a second transaction with the same `(address, nonce)` pair as `DuplicateNonce`:

```rust
if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
    return Err(MempoolError::DuplicateNonce { address, nonce });
}
``` [4](#0-3) 

Once the attacker's fake transaction occupies nonce=1, the legitimate user's Invoke is rejected.

**`account_tx_in_pool_or_recent_block` trigger condition:**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

A pending `deploy_account` in the pool satisfies `tx_pool.contains_account`, so the condition is met as soon as the victim submits their deployment.

**Stateless validation does not catch this:**

`StatelessTransactionValidator::validate` checks only the *length* of the signature array, not its cryptographic validity: [6](#0-5) 

An attacker can supply a single-element signature `[Felt::ONE]` (or any value within the length limit) and pass all stateless checks.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions.**

An attacker can insert an Invoke transaction with a completely fake signature into the mempool for any address that has a pending `deploy_account`. The nonce=1 slot for that address is permanently occupied until the fake transaction is executed (and reverted) by the batcher. During that window the legitimate user's first post-deployment Invoke is rejected with `DuplicateNonce`, effectively blocking the account from operating immediately after deployment. With fee escalation disabled (the default), the legitimate user cannot replace the fake transaction at all.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Pending `deploy_account` transactions are publicly observable via the gateway RPC.
- The attacker needs no privileged access, no private key, and no special knowledge beyond the target address.
- The race window is the time between the victim's `deploy_account` submission and their first Invoke submission — a window that is deliberately widened by the UX feature itself (users are encouraged to submit both together).
- The only friction is that the attacker must submit before the victim's Invoke arrives, which is trivially achievable by monitoring the mempool.

---

### Recommendation

**Short term:** Before returning `skip_validate = true`, perform a lightweight off-chain ECDSA check against the account's expected public key (retrievable from state for already-known account classes such as OpenZeppelin). For accounts whose class is not yet known (truly undeployed), require the `deploy_account` and the Invoke to arrive in the same atomic batch submission so the gateway can verify both together.

**Long term:** Decouple the UX skip from the signature-verification skip. The `__validate__` entry point can still be called even when `strict_nonce_check = false`; the nonce leniency and the signature skip are orthogonal. Setting `validate: true` while keeping `strict_nonce_check: false` would preserve the UX benefit while restoring the admission invariant.

---

### Proof of Concept

1. Victim submits `deploy_account` for address `0xVICTIM`. The gateway admits it; the mempool now satisfies `account_tx_in_pool_or_recent_block(0xVICTIM) == true`.

2. Attacker constructs an `RpcInvokeTransactionV3` with:
   - `sender_address = 0xVICTIM`
   - `nonce = 1`
   - `signature = [Felt::ONE]` (fake, one element — within `max_signature_length`)
   - valid `resource_bounds` above `min_gas_price`

3. Attacker submits to the gateway. The gateway executes `add_tx_inner`:
   - `stateless_tx_validator.validate` passes (signature length ≤ limit). [7](#0-6) 
   - `convert_rpc_tx_to_internal_and_executable_txs` succeeds (Invoke, no class compilation).
   - `extract_state_nonce_and_run_validations` is called:
     - `get_nonce_from_state` returns `Nonce(0)` (account not on-chain). [8](#0-7) 
     - `run_pre_validation_checks`:
       - `validate_state_preconditions`: nonce=1 is within `[0, max_allowed_nonce_gap]`. ✓
       - `validate_by_mempool`: no existing nonce=1 for `0xVICTIM`. ✓
       - `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block` = true → returns `true`. [9](#0-8) 
     - `run_validate_entry_point(skip_validate=true)`: `validate: false` → `__validate__` is **not called**. [2](#0-1) 
   - Transaction is forwarded to the mempool and accepted.

4. Victim submits their legitimate Invoke with nonce=1 and a valid signature. The mempool rejects it: `DuplicateNonce { address: 0xVICTIM, nonce: 1 }`. [4](#0-3) 

5. The attacker's fake transaction is eventually pulled by the batcher, the `__validate__` entry point is called during execution, it fails (invalid signature), and the transaction is reverted — but the victim's nonce=1 slot has been consumed and the legitimate transaction was never executed.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
