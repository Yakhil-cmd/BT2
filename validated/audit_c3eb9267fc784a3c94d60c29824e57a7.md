### Title
`__validate__` balance drain breaks fee-transfer invariant in `run_revertible`, enabling fee-free DoS — (File: `crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

In `run_revertible`, when a post-execution balance check fails with `InsufficientFeeTokenBalance`, the recommended fee is set to `actual_fee` under the explicit assumption that reverting execution state changes will restore sufficient balance. However, `__validate__` runs directly on the outer `state` (not a transactional sub-state), so its balance changes are **not** reverted when execution is aborted. A malicious account can drain its fee-token balance inside `__validate__`, causing the subsequent fee transfer to fail, which causes `execute_raw` to return an error, which causes the batcher to abort the entire transactional state — reverting even the nonce increment — so the attacker pays nothing and can repeat indefinitely.

---

### Finding Description

**Step 1 — Pre-validation passes.**
`perform_pre_validation_stage` calls `verify_can_pay_committed_bounds`, which reads the balance and confirms `balance_before ≥ max_possible_fee`. [1](#0-0) 

**Step 2 — `__validate__` runs on the outer state.**
Inside `run_revertible`, `validate_tx` is called directly on `state` (the outer `TransactionalState`). Its writes are committed into `state` immediately. [2](#0-1) 

**Step 3 — Execution runs in a transactional sub-state.**
`execution_state = TransactionalState::create_transactional(state)` layers on top of `state`. When `execution_state.abort()` is called, only the execute-phase writes are rolled back; the validate-phase writes in `state` survive. [3](#0-2) 

**Step 4 — Post-execution check fires `InsufficientFeeTokenBalance`.**
`PostExecutionReport::new` reads the balance from `execution_state` (which already reflects the validate drain). It detects `balance_after_execute < actual_fee` and returns `InsufficientFeeTokenBalance`. [4](#0-3) 

**Step 5 — Broken invariant: `recommended_fee = actual_fee`.**
`FeeCheckReport::from_fee_check_error` sets `recommended_fee = actual_fee` for the `InsufficientFeeTokenBalance` variant. The comment explicitly states the assumption: *"the sender initially could cover the resource bounds … after reverting execution state changes the user **can** cover the fee."* This assumption is false when `__validate__` has already drained the balance. [5](#0-4) 

**Step 6 — Fee transfer fails.**
Back in `execute_raw`, `handle_fee` is called with `state` (which still carries the validate-phase drain) and `final_fee = actual_fee`. The ERC-20 `transfer` call reverts because `balance_after_validate < actual_fee`, and `execute_fee_transfer` returns `ExecuteFeeTransferError`. [6](#0-5) [7](#0-6) 

**Step 7 — `execute_raw` returns an error; batcher aborts everything.**
The `?` on `handle_fee` propagates the error out of `execute_raw`. The batcher's outer transactional state is aborted, reverting **all** changes — including the nonce increment from `handle_nonce` and the validate-phase balance drain. The attacker's balance is fully restored. [8](#0-7) 

---

### Impact Explanation

**Incorrect fee accounting with economic impact (Critical).**

- The sequencer executes the full validate + execute pipeline but receives zero fee.
- The nonce is not incremented, so the attacker can resubmit the same transaction indefinitely.
- The attacker's balance is always restored (transactional abort), making the attack cost-free.
- The sequencer wastes L2 gas, CPU, and state-access resources on every iteration with no compensation.

---

### Likelihood Explanation

Any user who deploys a custom Cairo account contract (which is a standard Starknet operation) can implement a malicious `__validate__` that calls `transfer` on the fee token. The only precondition is holding `max_possible_fee + ε` tokens at the time of gateway admission — a trivially satisfiable condition. No privileged access, no special network position, and no dependency on external contracts is required.

---

### Recommendation

Run `__validate__` inside its own `TransactionalState` sub-layer (analogous to how `run_execute` is wrapped), and commit that sub-layer only after a successful fee transfer. Alternatively, after `execution_state.abort()`, re-read the balance from `state` and cap `recommended_fee` to `min(actual_fee, balance_after_validate)` before attempting the fee transfer. The existing comment in `from_fee_check_error` already documents the broken assumption; the fix is to make the assumption hold structurally rather than relying on caller discipline.

---

### Proof of Concept

1. Deploy an account contract whose `__validate__` calls `fee_token.transfer(attacker_alt_address, entire_balance)`.
2. Fund the account with `balance = max_possible_fee + 1`.
3. Submit an invoke V3 transaction. Gateway stateful check passes (`balance ≥ max_possible_fee`).
4. Blockifier executes:
   - `perform_pre_validation_stage`: nonce incremented, balance check passes.
   - `validate_tx` on `state`: entire balance transferred out → `balance_in_state = 0`.
   - `execution_state` created on top of `state`.
   - `run_execute` in `execution_state`: no-op.
   - `PostExecutionReport::new(&mut execution_state, …)`: `balance = 0 < actual_fee` → `InsufficientFeeTokenBalance`.
   - `execution_state.abort()`: execute writes rolled back; `state` still has `balance = 0`.
   - `recommended_fee = actual_fee`.
5. `handle_fee(state, …, actual_fee)`: ERC-20 transfer fails (balance = 0) → `ExecuteFeeTransferError`.
6. `execute_raw` returns `Err(…)`. Batcher aborts outer transactional state.
7. All changes reverted: nonce back to original, balance back to `max_possible_fee + 1`.
8. Repeat from step 3 — zero cost to attacker, full execution cost to sequencer each iteration.

    fn execute_fee_transfer(
        state: &mut dyn State,
        tx_context: Arc<TransactionContext>,
        actual_fee: Fee,
    ) -> TransactionExecutionResult<CallInfo> {
        // The least significant 128 bits of the amount transferred.
        let lsb_amount = Felt::from(actual_fee.0);
        // The most significant 128 bits of the amount transferred.
        let msb_amount = Felt::ZERO;

        let TransactionContext { block_context, tx_info } = tx_context.as_ref();
        let storage_address = tx_context.fee_token_address();
        // The fee contains the cost of running this transfer, and the token contract is
        // well known to the sequencer, so there is no need to limit its run.
        let mut remaining_gas_for_fee_transfer =
            block_context.versioned_constants.os_constants.gas_costs.base.default_initial_gas_cost;
        let fee_transfer_call = CallEntryPoint {
            class_hash: None,
            code_address: None,
            entry_point_type: EntryPointType::External,
            entry_point_selector: selector_from_name(constants::TRANSFER_ENTRY_POINT_NAME),
            calldata: calldata![
                *block_context.block_info.sequencer_address.0.key(), // Recipient.
                lsb_amount,
                msb_amount
            ],
            storage_address,
            caller_address: tx_info.sender_address(),
            call_type: CallType::Call,

            initial_gas: remaining_gas_for_fee_transfer,
        };
        let mut context = EntryPointExecutionContext::new_invoke(
            tx_context,
            true,
            SierraGasRevertTracker::new(GasAmount(remaining_gas_for_fee_transfer)),
        );

        Ok(fee_transfer_call
            .execute(state, &mut context, &mut remaining_gas_for_fee_transfer)
            .map_err(|error| Box::new(TransactionFeeError::ExecuteFeeTransferError(error)))?)
    }

impl PostExecutionReport {
    /// Verifies the actual cost can be paid by the account. If not, reports an error and the fee
    /// that should be charged in revert flow.
    pub fn new<S: StateReader>(
        state: &mut S,
        tx_context: &TransactionContext,
        tx_receipt: &TransactionReceipt,
        charge_fee: bool,
    ) -> TransactionExecutionResult<Self> {
        let TransactionReceipt { fee, gas, .. } = tx_receipt;

        // If fee is not enforced, no need to check post-execution.
        if !charge_fee {
            return Ok(Self(FeeCheckReport::success_report(*fee)));
        }

        // First, compare the actual resources used against the upper bound(s) defined by the
        // sender.
        let cost_within_bounds_result =
            FeeCheckReport::check_actual_cost_within_bounds(tx_context, tx_receipt);

        // Next, verify the actual cost is covered by the account balance, which may have changed
        // after execution. If the above check passes, the pre-execution balance covers the actual
        // cost for sure.
        let can_pay_fee_result = FeeCheckReport::check_can_pay_fee(state, tx_context, tx_receipt);

        for fee_check_result in [cost_within_bounds_result, can_pay_fee_result] {
            match fee_check_result {
                Ok(_) => continue,
                Err(TransactionExecutionError::FeeCheckError(fee_check_error)) => {
                    // Found an error; set the recommended fee based on the error variant and
                    // current context, and return the report.
                    return Ok(Self(FeeCheckReport::from_fee_check_error(
                        *fee,
                        *gas,
                        fee_check_error,
                        tx_context,
                    )));
                }
                Err(other_error) => return Err(other_error),
            }
        }

        Ok(Self(FeeCheckReport::success_report(*fee)))
    }
