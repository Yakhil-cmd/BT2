Audit Report

## Title
ckERC20 Deposit Helper Contracts Emit Caller-Supplied Amount Instead of Actual Received Amount, Enabling ckERC20 Undercollateralization for Fee-on-Transfer Tokens - (File: rs/ethereum/cketh/minter/ERC20DepositHelper.sol, rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol)

## Summary

Both ckERC20 deposit helper contracts emit deposit events using the caller-supplied `amount` parameter rather than the actual token balance change at the minter address. The IC ckETH minter canister parses this event-logged value and mints exactly that many ckERC20 tokens with no cross-check against actual ERC20 received. For any fee-on-transfer ERC20 token, the minter would receive `amount - fee` but mint `amount` ckERC20 tokens, permanently undercollateralizing the ckERC20 supply. USDT — a live, explicitly supported ckERC20 token — contains a built-in fee mechanism (`basisPointsRate`, `maximumFee`) that is currently set to zero but can be enabled at any time by the Tether contract owner.

## Finding Description

**Root cause — Solidity layer:**

In `ERC20DepositHelper.sol`, `deposit()` calls `safeTransferFrom` and then emits `ReceivedErc20` with the caller-supplied `amount`:

```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
``` [1](#0-0) 

In `DepositHelperWithSubaccount.sol`, `depositErc20()` follows the same pattern:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
``` [2](#0-1) 

Neither contract measures the actual balance change at `minterAddress` before and after the transfer. The emitted `amount` is always the caller-supplied value.

**Root cause — Rust minter layer:**

`ReceivedErc20LogParser` and `ReceivedEthOrErc20LogParser` parse the `value` field directly from the event data bytes and store it in `ReceivedErc20Event.value`: [3](#0-2) [4](#0-3) 

The `mint()` function in `deposit.rs` then calls the ICRC-1 ledger with `amount: event.value()`, minting exactly the event-logged value: [5](#0-4) 

`event.value()` returns the value parsed from the log, which is the caller-supplied `amount`: [6](#0-5) 

There is no balance-before/after check at any layer. The minter has no mechanism to detect that the actual ERC20 received differs from the event-logged amount.

**Exploit flow:**

1. USDT contract owner enables a non-zero `basisPointsRate` (e.g., 10 = 0.1%).
2. User calls `depositErc20(USDT, 1_000_000, principal, subaccount)` on the helper contract.
3. `safeTransferFrom` executes: minter receives `999_000` USDT (fee of `1_000` deducted by USDT contract).
4. Helper emits `ReceivedEthOrErc20(USDT, user, 1_000_000, ...)` — full `1_000_000`.
5. Minter scrapes the log, parses `value = 1_000_000`, mints `1_000_000` ckUSDT to the user.
6. After N deposits: minter holds `N × 999_000` USDT but has minted `N × 1_000_000` ckUSDT. The `1_000 × N` shortfall means the last depositors cannot redeem their full ckUSDT balance.

**Existing checks are insufficient:** The minter checks that the ERC20 contract address is supported and that the sender is not blocklisted, but performs no accounting check against the actual ERC20 balance held. The ICRC-1 ledger mint call simply accepts the amount provided by the minter canister.

## Impact Explanation

**Impact class: High — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.**

If USDT's fee mechanism is enabled, every ckUSDT deposit permanently mints more ckUSDT than the USDT backing held by the minter. The discrepancy accumulates monotonically with each deposit. Eventually, the minter holds insufficient USDT to honor all outstanding ckUSDT redemptions, breaking the 1:1 backing guarantee and causing direct financial loss to the last users to withdraw. USDT is a live, explicitly supported ckERC20 token: [7](#0-6) [8](#0-7) 

The same vulnerability applies to any other fee-on-transfer ERC20 token that is or becomes supported. The code defect is in production IC Chain Fusion infrastructure.

## Likelihood Explanation

**Likelihood: Medium.** The vulnerability is latent and requires a third-party action — the Tether contract owner enabling the USDT fee mechanism — to become exploitable. Currently `basisPointsRate = 0` on Ethereum mainnet, so no fee is charged and no undercollateralization occurs. However: (1) the fee capability is a documented, built-in feature of the USDT contract; (2) Tether has historically modified contract parameters; (3) once fees are active, no attacker capability is required — any ordinary user depositing USDT triggers the undercollateralization; (4) the code defect also affects any future fee-on-transfer token added to the supported list. The SECURITY.md exclusion for "depegging of an external stablecoin" does not apply here — the issue is ckUSDT becoming undercollateralized relative to USDT, not USDT depegging from USD.

## Recommendation

**Short-term (Solidity):** In both helper contracts, measure the actual balance change at `minterAddress` before and after `safeTransferFrom`, and emit the event with the actual received amount:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

**Long-term (Rust minter):** Add a policy to reject fee-on-transfer tokens from the supported token list, or add an on-chain ERC20 balance verification step before minting ckERC20 tokens.

## Proof of Concept

**Deterministic integration test plan (PocketIC / state-machine test):**

1. Deploy a mock ERC20 token contract that deducts a 0.1% fee on every `transferFrom` (i.e., recipient receives `amount * 999 / 1000`).
2. Register this token as a supported ckERC20 token in the minter.
3. Call `depositErc20(mockToken, 1_000_000, principal, subaccount)` on the helper contract.
4. Assert: minter's ERC20 balance increased by `999_000`.
5. Assert: ckERC20 ledger minted `1_000_000` to the user (demonstrating the discrepancy).
6. Repeat N times and assert: `ckERC20 total_supply = N × 1_000_000` while `minter ERC20 balance = N × 999_000`, confirming accumulating undercollateralization.

The existing integration test `test_erc20_deposit` in `rs/tests/cross_chain/ic_xc_cketh_test.rs` already asserts `minter_balance_after - minter_balance_before == deposit_amount` [9](#0-8) 
— this assertion would fail with a fee-on-transfer token, confirming the bug is not currently tested for.
