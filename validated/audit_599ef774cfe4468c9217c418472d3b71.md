### Title
Balance Inflation via Direct ERC20 Transfer Suppresses L1 Mint Requests, Causing Temporary Freeze of Reward Claims — (`src/reward_supplier/reward_supplier.cairo`)

---

### Summary

An unprivileged attacker can directly transfer STRK tokens to `RewardSupplier` (bypassing StarkGate). Because `request_funds` reads the raw ERC20 balance to compute `credit`, the inflated balance suppresses L1 mint requests for multiple epochs. Once the donated tokens are consumed by legitimate pool-reward claims, the balance drops to near zero. The next `_update_rewards` call in the staking contract then atomically calls `update_unclaimed_rewards_from_staking_contract` (which sends a mint request but does not yet receive funds) followed immediately by `claim_from_reward_supplier` — which reverts because the actual ERC20 balance is insufficient. This temporarily freezes reward distribution for all stakers until L1 funds arrive.

---

### Finding Description

**Root cause — `request_funds` trusts the raw ERC20 balance:** [1](#0-0) 

`balance` is read directly from the ERC20 contract. Any tokens sent to `RewardSupplier` via a plain `transfer()` call (not through StarkGate's `on_receive`) inflate this value without incrementing `l1_pending_requested_amount`. The threshold guard: [2](#0-1) 

is therefore bypassed for as long as `balance + l1_pending >= unclaimed + threshold`. The threshold is only `base_mint_amount / 2`: [3](#0-2) 

**The atomic failure window — `_update_rewards` in staking.cairo:**

The staking contract's internal `_update_rewards` does two things in the same transaction:

1. Calls `update_unclaimed_rewards_from_staking_contract` (which calls `request_funds` and may send an L1 mint message, but the funds have not arrived yet).
2. Immediately calls `claim_from_reward_supplier` for pool rewards. [4](#0-3) 

`claim_rewards` on the reward supplier only checks the accounting invariant (`amount <= unclaimed_rewards`) but then calls `checked_transfer`, which reverts if the actual ERC20 balance is insufficient: [5](#0-4) 

When the donated tokens have been consumed by prior claims and the balance is near zero, step 2 reverts even though step 1 correctly sent the mint request — because the L1→L2 bridge message has not yet been processed.

---

### Impact Explanation

The revert in `claim_from_reward_supplier` propagates up through `_update_rewards` → `update_rewards`. This means no staker can have their rewards updated (and therefore no staker or pool member can claim rewards) until the L1 bridge message is processed and `on_receive` is called. On Starknet mainnet, L1→L2 message finality typically takes hours to over a day. During this window, all reward claims are frozen.

---

### Likelihood Explanation

The attacker must sacrifice `X ≥ N × (per-epoch reward rate)` STRK tokens to suppress mint requests for `N` epochs, where `N` must be large enough that the balance drops below a single epoch's pool rewards. With `BASE_MINT_AMOUNT = 1,300,000 STRK` and `threshold = 650,000 STRK`, a donation of ~10× `base_mint_amount` suppresses requests for roughly 10+ epochs. The attacker has no profit motive; this is a griefing attack. Likelihood is low but the path is fully permissionless and concrete.

---

### Recommendation

Do not use the raw ERC20 balance as a measure of protocol-controlled liquidity. Instead, track an internal `deposited_balance` storage variable that is incremented only inside `on_receive` (the StarkGate callback) and decremented inside `claim_rewards`. Replace the `balance_of` call in `request_funds` with this internal counter:

```cairo
// Instead of:
let balance = token_dispatcher.balance_of(account: get_contract_address());

// Use:
let balance = self.deposited_balance.read();
```

This makes `credit` immune to unsolicited ERC20 transfers and correctly reflects only protocol-controlled liquidity.

---

### Proof of Concept

```
// Concrete multi-epoch scenario (BASE_MINT_AMOUNT = 1_300_000 STRK, threshold = 650_000 STRK)
// Per-epoch pool rewards ≈ R STRK

1. attacker.transfer(reward_supplier, 10 * BASE_MINT_AMOUNT)
   // balance = 13_000_000 STRK; l1_pending = 0

2. For epochs 1..12:
   staking._update_rewards(staker)
     -> update_unclaimed_rewards_from_staking_contract(R)
        -> request_funds: credit = (13_000_000 - k*R) >= unclaimed + threshold  // no request
     -> claim_from_reward_supplier(pool_R)  // succeeds, balance decreases

3. Epoch 13: balance ≈ 0
   staking._update_rewards(staker)
     -> update_unclaimed_rewards_from_staking_contract(R)
        -> request_funds: credit < unclaimed + threshold  // sends L1 mint message
           l1_pending += BASE_MINT_AMOUNT  (in-transit, not yet received)
     -> claim_from_reward_supplier(pool_R)
        -> claim_rewards(pool_R)
           -> checked_transfer(staking_contract, pool_R)
              // balance ≈ 0 < pool_R  →  REVERT

// All reward updates are frozen until on_receive() is called by StarkGate.
```
