### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `IStakingRewardsManager` has no caller restriction despite the specification requiring "Only starkware sequencer." Any unprivileged address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for the current block without distributing any rewards. Because `last_reward_block` is written **before** the `disable_rewards` early-return, the sequencer is permanently blocked from distributing rewards for that block.

### Finding Description
The two-step flow in `update_rewards` is:

1. Write `last_reward_block = current_block_number` (unconditionally).
2. If `disable_rewards || self.is_pre_consensus()`, return early — no rewards distributed. [1](#0-0) 

Because step 1 always executes before step 2, any caller who passes `disable_rewards: true` consumes the global slot without distributing rewards. The subsequent `REWARDS_ALREADY_UPDATED` guard then blocks every other call in the same block: [2](#0-1) 

The spec explicitly states access should be restricted: [3](#0-2) 

Yet the implementation contains no such check — the tests confirm this by calling `update_rewards` directly without any caller-address cheating: [4](#0-3) 

The `last_reward_block` storage variable is global (not keyed per staker), so a single poisoned call blocks reward distribution for **all** stakers in that block: [5](#0-4) 

The analog to the original report is direct: just as a shared nonce per account can be consumed by any permissioned user to block a legitimate user's transaction, the shared `last_reward_block` can be consumed by any unprivileged caller to block the sequencer's reward distribution for an entire block.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

When an attacker calls `update_rewards(valid_staker, disable_rewards: true)` in block N:
- `last_reward_block` is set to N.
- No rewards are distributed.
- The sequencer's subsequent call for block N reverts with `REWARDS_ALREADY_UPDATED`.
- The block-N rewards are permanently unrecoverable for every staker.

This satisfies the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

### Likelihood Explanation
**Low.** On Starknet the sequencer controls transaction ordering and would normally place its own `update_rewards` call first in each block. The attack succeeds only when the sequencer omits the call for a given block (e.g., due to a bug, downtime, or a block where no staker is selected). Because staker addresses and their active status are fully public, the attacker has all the information needed to craft a valid call the moment such a gap appears.

### Recommendation
Add an explicit sequencer-only guard at the top of `update_rewards`, analogous to the attestation-contract guard used in `update_rewards_from_attestation_contract`: [6](#0-5) 

Store the authorized sequencer address in contract storage and assert `get_caller_address() == sequencer_address` before any state mutation in `update_rewards`.

Alternatively, move the `self.last_reward_block.write(current_block_number)` call to **after** the `disable_rewards` check so that a no-op call does not consume the slot.

### Proof of Concept
1. Consensus rewards are active (post `consensus_rewards_first_epoch`).
2. Attacker observes that the sequencer has not yet called `update_rewards` in block N.
3. Attacker calls `update_rewards(active_staker_address, disable_rewards: true)`.
   - Passes `REWARDS_ALREADY_UPDATED` check (new block).
   - Passes staker-existence and balance checks (public staker chosen).
   - Writes `last_reward_block = N`.
   - Returns early — zero rewards distributed.
4. Sequencer calls `update_rewards(staker_address, disable_rewards: false)` in block N.
   - Reverts: `current_block_number > self.last_reward_block.read()` is false.
   - Error: `REWARDS_ALREADY_UPDATED`.
5. Block-N rewards are permanently lost for all stakers.
