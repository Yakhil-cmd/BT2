Looking at the CKD protocol in `src/confidential_key_derivation/protocol.rs`, I can identify a direct analog to the oracle price manipulation vulnerability: the coordinator blindly aggregates participant-provided cryptographic contributions without any proof of correctness, just as `MagicLpAggregator` blindly uses pool reserves without cross-checking against Chainlink.

---

### Title
Unverified Participant Contributions in CKD Protocol Allow Malicious Participant to Corrupt Confidential Key Derivation Output — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator sums `(norm_big_y, norm_big_c)` values received from each participant with no cryptographic proof of correctness. A single malicious participant can substitute arbitrary group elements, causing the coordinator to derive a permanently incorrect confidential key.

### Finding Description

In `do_ckd_coordinator`, the coordinator collects every participant's share and adds them together unconditionally: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The protocol specification requires each participant `i` to compute: [2](#0-1) 

```
norm_big_y = λᵢ · yᵢ · G
norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)
```

where `xᵢ` is the participant's private share and `yᵢ` is a fresh random scalar. The coordinator has no way to verify that the received `(norm_big_y, norm_big_c)` pair was actually computed this way, because:

1. No zero-knowledge proof accompanies the contribution.
2. No commitment to `xᵢ · H(pk ‖ app_id)` is checked.
3. `recv_from_others` performs only message-routing, not content validation. [3](#0-2) 

The final unmasked confidential key is:

```
unmask(app_sk) = norm_big_c_total − app_sk · norm_big_y_total
               = msk · H(pk ‖ app_id)   [when all parties are honest]
```

If one participant substitutes `(Δ_Y, Δ_C)` for their legitimate share, the coordinator computes:

```
norm_big_c_total = honest_sum_C + Δ_C
norm_big_y_total = honest_sum_Y + Δ_Y
unmask(app_sk)   = msk · H(pk ‖ app_id) + (Δ_C − app_sk · Δ_Y)
```

By choosing `Δ_C ≠ app_sk · Δ_Y`, the attacker shifts the derived confidential key by an arbitrary, attacker-controlled offset. The coordinator has no mechanism to detect this deviation.

This is structurally identical to the oracle report: pool reserves (participant contributions) are consumed directly without cross-checking against a reliable reference (a ZK proof or public-key commitment).

### Impact Explanation

A single malicious participant can permanently corrupt the CKD output accepted by the coordinator. The coordinator will believe it holds a valid `CKDOutput`, but `unmask` will return a key that does not equal `msk · H(pk ‖ app_id)`. Any downstream system relying on this key (e.g., a TEE that uses it to decrypt secrets) will silently operate on a wrong key. This matches:

> **High: Corruption of CKD outputs so honest parties accept inconsistent… unusable cryptographic outputs.**

### Likelihood Explanation

Every participant in the CKD session is a potential attacker. No special privilege is required beyond being a listed participant. The attack requires only sending two arbitrary group elements instead of the protocol-specified values — a trivial deviation for any party that controls its own network stack or runs a modified client.

### Recommendation

Attach a non-interactive zero-knowledge proof (e.g., a Sigma protocol / Chaum-Pedersen proof) to each participant's `(norm_big_y, norm_big_c)` contribution, proving that:

```
norm_big_c − norm_big_y · app_pk / λᵢ = xᵢ · H(pk ‖ app_id)
```

without revealing `xᵢ` or `yᵢ`. The coordinator must verify all proofs before summing contributions, analogous to how the DKG protocol verifies proofs of knowledge before accepting polynomial commitments. [4](#0-3) 

### Proof of Concept

A malicious participant `P_m` sends:
- `norm_big_y = G` (generator, instead of `λ_m · y_m · G`)
- `norm_big_c = app_pk` (instead of the correct value)

The coordinator computes:
```
final_norm_big_y = Σ_{i≠m} λᵢ·yᵢ·G  +  G
final_norm_big_c = Σ_{i≠m} λᵢ·(xᵢ·H + yᵢ·A)  +  A
```

Unmasking:
```
final_norm_big_c − app_sk · final_norm_big_y
= msk·H(pk‖app_id) + A − app_sk·G
= msk·H(pk‖app_id) + app_sk·G − app_sk·G   [only if A = app_sk·G, i.e. app_pk]
```

In the general case where `Δ_C ≠ app_sk · Δ_Y`, the result is `msk·H + noise`, which is an incorrect and unpredictable confidential key. The coordinator accepts it silently because no verification step exists. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn internal_verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    participant: Participant,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: &Signature<C>,
) -> Result<(), ProtocolError> {
    // creates an identifier for the participant
    let id = participant.scalar::<C>();
    let vk_share = commitment
        .coefficients()
        .first()
        .ok_or_else(|| ProtocolError::AssertionFailed("Empty coefficient list".to_string()))?;

    let big_r = proof_of_knowledge.R();
    let z = proof_of_knowledge.z();
    let c = challenge::<C>(domain_separator, session_id, id, vk_share, big_r)?;
    if *big_r != <C::Group>::generator() * *z - vk_share.value() * c.to_scalar() {
        return Err(ProtocolError::InvalidProofOfKnowledge(participant));
    }
    Ok(())
}
