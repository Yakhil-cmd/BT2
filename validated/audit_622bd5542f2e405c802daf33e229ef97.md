### Title
Missing `max_l2_gas_amount` Upper Bound for `Declare` Transactions in `StatelessTransactionValidator::validate_resource_bounds` - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary
The stateless gateway validator enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for every transaction type **except** `Declare`. An explicit branch silently skips the check for Declare transactions, allowing a Declare with `l2_gas.max_amount = u64::MAX` to pass all gateway admission checks. The gap is acknowledged by a TODO comment in the source. This is the direct Sequencer analog of the external report's "missing upper bound" pattern: a setter-level guard is absent for one variant, and downstream logic proceeds with an unconstrained value.

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds` the upper-bound guard reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
```

The `if let RpcTransaction::Declare(_) = tx { }` arm is an intentional no-op: it matches Declare and does nothing, so the `else if` branch is never reached for Declare transactions. Every other transaction type (Invoke, DeployAccount) is correctly rejected when `l2_gas.max_amount > max_l2_gas_amount` (default 1,210,000,000). Declare transactions are exempt with no compensating check anywhere in the stateful validator or blockifier pre-validation path.

The stateful validator's `validate_resource_bounds` only checks the L2 gas **price** against the previous block's price; it never checks the gas **amount**. The blockifier's `perform_pre_validation_stage` → `check_fee_bounds` only enforces a **lower** bound (`minimal_gas_amount ≤ resource_bounds.max_amount`). No layer between the RPC endpoint and execution enforces an upper bound on `l2_gas.max_amount` for Declare transactions.

The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts this behaviour: a Declare with `l2_gas.max_amount = 200` passes even when the configured limit is `100`.

### Impact Explanation

**Impact: High — Mempool/gateway/RPC admission accepts transactions that should be rejected before sequencing.**

Any unprivileged user can submit a `RpcDeclareTransaction::V3` with `l2_gas.max_amount = u64::MAX` (18,446,744,073,709,551,615) and a valid Sierra class. The transaction passes:
- `StatelessTransactionValidator::validate` (the Declare branch is a no-op for the amount check)
- `StatefulTransactionValidator::validate_resource_bounds` (only checks price)
- Blockifier pre-validation (only checks lower bound)

The gateway's admission invariant — "no transaction may claim more L2 gas than `max_l2_gas_amount`" — is broken for the entire Declare transaction type. The `max_l2_gas_amount` parameter exists precisely to bound the resource claim a single transaction can make at admission time; exempting Declare transactions removes that bound entirely for that type.

Downstream execution is bounded by the block's global `validate_max_sierra_gas` / `execute_max_sierra_gas` constants, so no incorrect state or fee results from a single transaction. However, the gateway's first line of defence against oversized resource claims is bypassed, and the admitted value propagates unchecked through the mempool and into the blockifier's `max_steps` computation, which derives the per-transaction step ceiling from `max_amount`.

### Likelihood Explanation

Trivially reachable by any external caller. No special account state, privileged role, or prior on-chain action is required. The attacker only needs to construct a syntactically valid `RpcDeclareTransaction::V3` with an arbitrary `l2_gas.max_amount` value and a valid Sierra class body (which must pass the separate `validate_declare_tx` checks on Sierra version and class size). The TODO comment and the dedicated positive test `valid_l2_gas_amount_on_declare` confirm the path is reachable and intentionally left open.

### Recommendation

Remove the Declare exemption and apply the same `max_l2_gas_amount` ceiling to all transaction types, or introduce a separate, explicitly documented `max_l2_gas_amount_declare` limit. The TODO comment at line 78 already flags this as pending work:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // Apply the same or a separate upper-bound check here.
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
    }
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    …
}
```

### Proof of Concept

1. Construct a `RpcDeclareTransaction::V3` with a valid Sierra class (passes `validate_sierra_version`, `validate_class_length`, `validate_entry_points_sorted_and_unique`) and set:
   - `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`
   - `resource_bounds.l2_gas.max_price_per_unit = GasPrice(min_gas_price)` (≥ 8,000,000,000)
   - Any non-zero L1 or L1-data bound to satisfy `ZeroResourceBounds`
2. Submit via `POST /gateway/add_transaction`.
3. `StatelessTransactionValidator::validate` reaches the `validate_resource_bounds` call; the `if let RpcTransaction::Declare(_) = tx { }` arm matches and returns without error; the `else if` branch is never evaluated.
4. The transaction proceeds through stateful validation (price check passes), mempool admission, and blockifier pre-validation (lower-bound check passes with `u64::MAX ≥ minimal_gas_amount`).
5. The gateway has admitted a Declare transaction whose `l2_gas.max_amount` exceeds `max_l2_gas_amount` by a factor of ~15, violating the admission invariant enforced for every other transaction type.
