### Title
Final DKG Integrity Broadcast Hardcodes Success, Making Failure-Detection Dead Code — (File: `src/dkg.rs`)

---

### Summary

`broadcast_success` in `src/dkg.rs` is documented to accept an error parameter and broadcast `false` when a participant detects a local failure during DKG. The implementation omits that parameter entirely and hardcodes the broadcast payload to `(true, session_id)`. The receiving-side check that is supposed to abort the protocol when any participant signals failure is therefore permanently dead code and can never trigger.

---

### Finding Description

The function comment explicitly describes the intended design:

```
/// This function takes err as input.
/// If err is None then broadcast success
/// otherwise, broadcast failure
``` [1](#0-0) 

However, the actual function signature carries no `err` parameter, and the broadcast payload is hardcoded to `(true, session_id)`:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,          // ← no `err` parameter
) -> Result<(), ProtocolError> {
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    //                                                   ^^^^ always true
``` [2](#0-1) 

The receiving side then checks whether any participant broadcast `false`:

```rust
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant seems to have failed its checks. Aborting Protocol!"
    ));
}
``` [3](#0-2) 

Because every participant unconditionally broadcasts `true`, this guard is permanently unreachable — dead code. The `false` path described in the comment has no implementation path to reach it.

`broadcast_success` is invoked as the final step of `do_keyshare`, which underlies DKG, reshare, and refresh: [4](#0-3) 

---

### Impact Explanation

The final-round integrity check — the only mechanism by which a participant can signal to all peers that their local DKG checks failed — is non-functional. Two concrete consequences follow:

1. **Dead integrity gate.** If a participant reaches `broadcast_success` after a local inconsistency that was not caught by an earlier early-return (e.g., a future code path, or a subtle edge case in share accumulation), they will still broadcast `true`. All other participants will accept the DKG output as valid, potentially accepting inconsistent or unusable key material.

2. **Permanent denial of key generation.** Because the failure-signaling path (`false`) is missing, a participant that *does* abort early (before calling `broadcast_success`) simply drops out of the reliable broadcast without sending anything. The `do_broadcast` call on all remaining honest participants blocks indefinitely waiting for that participant's message, permanently stalling DKG/reshare/refresh for all honest parties.

This matches the allowed impact: **High — Corruption of DKG outputs so honest parties accept inconsistent outputs, or Permanent denial of key generation for honest parties.**

---

### Likelihood Explanation

The missing `err` parameter is reachable by any participant in any DKG, reshare, or refresh session. A single malicious participant who participates honestly through all validation rounds and then aborts just before the final broadcast (or who triggers any local error after passing share validation) will cause all honest participants to hang indefinitely. No privileged access is required; any participant in the protocol can trigger this.

---

### Recommendation

Restore the intended design described in the comment. Add an `err: Option<ProtocolError>` parameter to `broadcast_success` and broadcast `(err.is_none(), session_id)`. Call the function unconditionally at the end of `do_keyshare`, passing any accumulated error, so that participants can signal failure gracefully and the receiving-side check becomes reachable:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
    err: Option<&ProtocolError>,   // ← restore intended parameter
) -> Result<(), ProtocolError> {
    let success = err.is_none();
    let vote_list = do_broadcast(chan, participants, me, (success, session_id)).await?;
    // ... existing checks now reachable
}
```

---

### Proof of Concept

1. Initiate a DKG session with `n` honest participants and one participant `M` that is configured to abort after completing share validation (i.e., after `validate_received_share` succeeds but before `broadcast_success` is called).
2. `M` returns a `ProtocolError` from `do_keyshare` and never calls `broadcast_success`, so it never sends a message to `do_broadcast`.
3. All `n-1` honest participants reach `broadcast_success`, call `do_broadcast`, and block waiting for `M`'s message.
4. Because `do_broadcast` (echo-broadcast) requires all `n` participants to deliver a message and there is no timeout, all honest participants hang indefinitely.
5. The check `if !vote_list.iter().all(|&(boolean, _)| boolean)` — which was supposed to let `M` signal failure by broadcasting `false` — is never reached because `false` cannot be broadcast. [5](#0-4)
