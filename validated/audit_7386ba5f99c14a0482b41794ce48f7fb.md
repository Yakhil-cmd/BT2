### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Global Spread Fee Cap System — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` applies no cap on `addFeeBuyE6` / `addFeeSellE6`, while every other admin fee path enforces `maxAdminSpreadFeeE6`. A pool admin can write `uint16` max (65 535 = 6.5535 % in E6 units) onto any bin — including currently inactive ones — with no timelock, and the value takes effect the moment the pool's cursor enters that bin.

---

### Finding Description

**Global fee path — capped:**

`setPoolAdminFees` enforces:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

The hard ceiling is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %). [2](#0-1) 

**Per-bin fee path — uncapped:**

`setPoolBinAdditionalFees` passes the caller-supplied values straight through to the pool with no validation beyond the bin-index range check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool-level function only validates the bin index:

```solidity
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [4](#0-3) 

**Fee application during swap:**

The per-bin fee is added directly on top of the oracle-derived base spread fee:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [5](#0-4) 

`uint16` max = 65 535 → 6.5535 % additive on top of whatever the base spread already is. Combined with a base spread at the 20 % hard cap, the effective per-bin fee reaches 26.5 %, far above the cap the system is supposed to enforce.

---

### Impact Explanation

Traders who swap into a bin that has a pre-set `addFeeBuyE6 = 65535` pay 6.5535 % more than the global spread cap permits, with no on-chain warning or timelock. The excess fee is extracted from the trader's input and accrues to the pool (LP benefit / admin extraction depending on fee-split config). This is a direct loss of trader principal above Sherlock thresholds, caused by an admin-boundary break: the pool admin exceeds the cap that the fee system is designed to enforce.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly expected to operate within caps. The call path is two transactions (set fee on inactive bin, wait for price to move) and requires no special conditions beyond holding the `poolAdmin` role. The `BinAdditionalFeesUpdated` event is emitted but there is no timelock, so the fee can be set and take effect in the same block as a price move.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool) analogous to the one in `setPoolAdminFees`:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Optionally, add a timelock consistent with the price-provider rotation timelock so that traders have advance notice before a per-bin fee change takes effect.

---

### Proof of Concept

1. Pool is deployed with bins `[-2, -1, 0, 1, 2]`; current active bin is `0`.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 2, 65535, 65535)` — no revert, no timelock.
3. Oracle price moves upward; the swap cursor advances to bin `2`.
4. The next swap into bin `2` computes `buyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, charging the trader 6.5535 % extra on top of the base spread.
5. Assert: effective fee on bin `2` > `maxAdminSpreadFeeE6` — the cap is bypassed. [6](#0-5) [3](#0-2)
