### Title
EIP-7623 Minimum Gas Floor Uses `intrinsic_gas` Instead of `floor_tokens_gas_cost`, Causing Overpayment for Data-Heavy Transactions - (File: `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`, `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the `eip_7623` feature is enabled, the `minimal_gas_used` floor passed to `compute_gas_refund` is set to `TX_INTRINSIC_GAS + floor_tokens_gas_cost` (i.e., `21000 + tokens × 10`). Per EIP-7623, the minimum gas that must be *charged* is only `floor_tokens_gas_cost = tokens × 10`. Because the intrinsic gas (21,000) is already embedded in the `gas_used` value derived from remaining resources, adding it again to the floor causes every data-heavy transaction to be overcharged by up to 21,000 gas.

---

### Finding Description

In both the Ethereum STF path and the ZK path, the EIP-7623 block computes a single `intrinsic_gas` value that serves two purposes:

1. **Validity gate** — `require!(intrinsic_gas <= tx_gas_limit, ...)` — correctly checks that the gas limit is at least `21000 + tokens × 10`.
2. **Minimum gas floor** — the same `intrinsic_gas` value is returned as `minimal_gas_used` and later passed to `compute_gas_refund`.

**Ethereum STF path** (`ethereum/validation_impl.rs`, lines 145–162):

```rust
#[cfg(feature = "eip_7623")]
{
    let floor_tokens_gas_cost = num_tokens.saturating_mul(TOTAL_COST_FLOOR_PER_TOKEN);
    let intrinsic_gas = TX_INTRINSIC_GAS.saturating_add(floor_tokens_gas_cost);
    require!(intrinsic_gas <= tx_gas_limit, ...)?;
    (num_tokens, intrinsic_gas)   // ← minimal_gas_used = 21000 + tokens×10  (WRONG)
}
``` [1](#0-0) 

**ZK path** (`zk/validation_impl.rs`, via `compute_calldata_tokens`, lines 530–535):

```rust
#[cfg(feature = "eip_7623")]
{
    let floor_tokens_gas_cost = num_tokens.saturating_mul(TOTAL_COST_FLOOR_PER_TOKEN);
    let intrinsic_gas = TX_INTRINSIC_GAS.saturating_add(floor_tokens_gas_cost);
    (num_tokens, intrinsic_gas)   // ← minimal_gas_used = 21000 + tokens×10  (WRONG)
}
``` [2](#0-1) 

This `minimal_gas_used` is then used as the floor in `compute_gas_refund`:

```rust
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);
``` [3](#0-2) 

The `gas_used` value at that point is already `gas_limit − remaining_ergs/ERGS_PER_GAS`, which already includes the 21,000 intrinsic gas deducted upfront in `create_resources_for_tx`. [4](#0-3) 

Per EIP-7623, the minimum gas that must be *charged* is `floor_gas = tokens × TOTAL_COST_FLOOR_PER_TOKEN`, not `21000 + floor_gas`. The correct return value should be `(num_tokens, floor_tokens_gas_cost)`.

---

### Impact Explanation

For a data-heavy transaction (e.g., 1,000 non-zero calldata bytes):

| Quantity | Value |
|---|---|
| `tokens` | 4,000 |
| `floor_gas` (correct EIP-7623 floor) | 40,000 |
| `gas_used` from execution (intrinsic only) | 21,000 |
| **Correct minimum charged** | `max(21000, 40000)` = **40,000** |
| **Actual minimum charged (buggy)** | `max(21000, 61000)` = **61,000** |
| **Overcharge** | **21,000 gas** |

The sender is charged 21,000 extra gas units on every data-heavy transaction where the EIP-7623 floor exceeds actual execution gas. The operator receives this excess. This is a direct, quantifiable financial loss for users.

---

### Likelihood Explanation

Any unprivileged transaction sender can trigger this by submitting a transaction with sufficient calldata to make `tokens × 10 > gas_used_from_execution`. This is a common pattern (e.g., large calldata to a contract). The only prerequisite is that the `eip_7623` compile-time feature is enabled; the EVM tester index confirms EIP-7623 tests are active in this repository. [5](#0-4) 

---

### Recommendation

Return only `floor_tokens_gas_cost` (not `intrinsic_gas`) as the second tuple element in both EIP-7623 blocks:

```rust
// Ethereum path
(num_tokens, floor_tokens_gas_cost)   // was: (num_tokens, intrinsic_gas)

// ZK path (compute_calldata_tokens)
(num_tokens, floor_tokens_gas_cost)   // was: (num_tokens, intrinsic_gas)
```

The validity gate (`require!(intrinsic_gas <= tx_gas_limit, ...)`) is correct and should remain unchanged. Only the returned `minimal_gas_used` floor needs to be `floor_tokens_gas_cost`.

---

### Proof of Concept

1. Enable the `eip_7623` feature.
2. Construct a transaction with 1,000 non-zero calldata bytes, `gas_limit = 62,000`, `gas_price = 1`.
3. `tokens = 4,000`, `floor_gas = 40,000`, `intrinsic_gas = 61,000`.
4. Transaction passes validity check (`61,000 ≤ 62,000`).
5. Execution uses only intrinsic gas → `gas_used_from_resources = 21,000`.
6. `compute_gas_refund` applies floor: `max(21000, 61000) = 61,000` gas charged.
7. User receives refund of only `(62,000 − 61,000) × gas_price = 1,000` gas worth of tokens.
8. **Expected** per EIP-7623: `max(21000, 40000) = 40,000` gas charged; refund = `22,000` gas worth of tokens.
9. User is overcharged by 21,000 gas units. [6](#0-5) [7](#0-6)
