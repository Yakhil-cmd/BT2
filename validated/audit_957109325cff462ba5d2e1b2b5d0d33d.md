### Title
User-Controlled `executionEffort = 0` in Scheduled Transaction Callbacks Bypasses Computation Limit - (`fvm/blueprints/scheduled_callback.go`)

### Summary

The `callbackArgsFromEvent` function in `fvm/blueprints/scheduled_callback.go` enforces only an **upper bound** on the user-supplied `executionEffort` field from the `PendingExecution` event, but no **lower bound**. When a user schedules a callback with `executionEffort = 0`, the resulting callback transaction is built with `SetComputeLimit(0)`. The FVM's `TransactionProcedure.ComputationLimit` treats `GasLimit = 0` as "unset" and silently falls back to `DefaultComputationLimit` (100,000 computation units). The user's callback therefore executes with up to 100K computation units regardless of what they paid for, consuming block computation budget at the protocol's expense.

### Finding Description

**Step 1 – User-controlled effort flows from event into transaction:**

In `callbackArgsFromEvent` (`fvm/blueprints/scheduled_callback.go`), the `executionEffort` field is decoded from the `PendingExecution` event emitted by the `FlowTransactionScheduler` Cadence contract. The function caps effort at `flow.DefaultMaxTransactionGasLimit` (upper bound) but performs **no minimum check**: [1](#0-0) 

If `effort = 0`, the function returns `0` without warning or clamping.

**Step 2 – Zero effort is passed directly to `SetComputeLimit`:**

`generateExecuteCallbacksTransaction` uses the returned `effort` value directly as the transaction's gas limit: [2](#0-1) 

**Step 3 – `GasLimit = 0` triggers silent fallback to `DefaultComputationLimit`:**

`TransactionProcedure.ComputationLimit` explicitly documents that `GasLimit = 0` is treated as "unset by user" and falls back first to `ctx.ComputationLimit`, then to `DefaultComputationLimit` (100,000): [3](#0-2) 

This behavior is confirmed by the existing test comment: *"gas limit of zero is ignored by runtime"*. [4](#0-3) 

**Step 4 – The scheduled transaction context does not set a `ComputationLimit`:**

`ScheduledTransactionContext` inherits `ComputationLimit` from the parent `vmCtx` without overriding it. Since the parent context's `ComputationLimit` is typically `0` (not explicitly set), the fallback to `DefaultComputationLimit` (100K) is guaranteed: [5](#0-4) 

**Step 5 – The same flaw exists in the v0 system collection builder:**

`builderV0.callbackArgsFromEvent` in `model/access/systemcollection/system_collection_v0.go` has the identical missing lower-bound check: [6](#0-5) 

**Step 6 – No test covers `effort = 0`:**

The test suite covers the upper-bound case (`effort > DefaultMaxTransactionGasLimit`) but has no test for `effort = 0`, leaving the lower-bound bypass undetected: [7](#0-6) 

### Impact Explanation

A user who schedules a callback with `executionEffort = 0` (paying minimal or zero fees for computation) has their callback executed with up to `DefaultComputationLimit` (100,000) computation units. This computation is consumed from the block's budget at the protocol's expense, not the user's. An attacker can schedule many such callbacks to consume block computation budget without paying for it — directly analogous to the EVM hook vulnerability where the hook's gas is paid by the claimer rather than the hook author.

### Likelihood Explanation

The attacker entry path is: submit a transaction to the `FlowTransactionScheduler` Cadence contract scheduling a callback with `executionEffort = 0`. If the Cadence contract permits this value (no minimum enforcement on-chain), the Go execution layer has no protection. The Go code path is fully reachable by any unprivileged user who can submit a transaction to the scheduler contract.

### Recommendation

Add a minimum lower-bound check in `callbackArgsFromEvent` (and its v0 counterpart). If `effort == 0`, either reject the callback or clamp it to a protocol-defined minimum (e.g., `1`):

```go
const minCallbackEffort = uint64(1)

if effort == 0 {
    log.Warn().Msg("effort is zero, setting to minimum")
    effort = minCallbackEffort
}
if effort > flow.DefaultMaxTransactionGasLimit {
    log.Warn().Uint64("effort", effort).Msg("effort exceeds max, capping")
    effort = flow.DefaultMaxTransactionGasLimit
}
```

Apply the same fix to `builderV0.callbackArgsFromEvent` in `model/access/systemcollection/system_collection_v0.go`.

### Proof of Concept

1. A user submits a transaction to `FlowTransactionScheduler` scheduling a callback with `executionEffort = 0`.
2. The scheduler emits `PendingExecution(id: X, executionEffort: 0, ...)`.
3. `callbackArgsFromEvent` decodes `effort = 0`, passes the upper-bound check (`0 <= DefaultMaxTransactionGasLimit`), and returns `effort = 0`.
4. `generateExecuteCallbacksTransaction` builds a transaction with `GasLimit = 0`.
5. At execution time, `TransactionProcedure.ComputationLimit` sees `GasLimit = 0`, falls back to `ctx.ComputationLimit = 0`, then to `DefaultComputationLimit = 100_000`.
6. The callback executes with 100K computation units — 100,000× more than the user paid for — consuming block budget at the protocol's expense. [1](#0-0) [3](#0-2) [8](#0-7)

func callbackArgsFromEvent(event flow.Event) (uint64, uint64, error) {
	cadenceId, cadenceEffort, err := ParsePendingExecutionEvent(event)
	if err != nil {
		return 0, 0, err
	}

	effort := uint64(cadenceEffort)

	if effort > flow.DefaultMaxTransactionGasLimit {
		log.Warn().Uint64("effort", effort).Msg("effort is greater than max transaction gas limit, setting to max")
		effort = flow.DefaultMaxTransactionGasLimit
	}

	return uint64(cadenceId), uint64(effort), nil
}

	// TODO replace tx.Gas with individual limits for computation and memory

	// decide computation limit
	computationLimit := proc.Transaction.GasLimit
	// if the computation limit is set to zero by user, fallback to the gas limit set by the context
	if computationLimit == 0 {
		computationLimit = ctx.ComputationLimit
		// if the context computation limit is also zero, fallback to the default computation limit
		if computationLimit == 0 {
			computationLimit = DefaultComputationLimit
		}
	}
	return computationLimit
}
