### Title
Unauthorized Transfer and Burn of Native `evm/` Tokens via Unauthenticated Bank Precompile Caller Check - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract` precompile at address `0x0000000000000000000000000000000000000064` exposes `transfer` and `burn` methods that accept the **source address as a calldata argument** but perform no check that the calling contract is authorized to act on behalf of that address. Any unprivileged smart contract can transfer or burn `evm/<its_own_address>` denomination native tokens from any holder without the holder's consent.

### Finding Description
In `BankContract.Run()`, the `transfer` case reads the `sender` address directly from calldata:

```go
sender := args[0].(common.Address)   // attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // "evm/<calling_contract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is no check that `contract.Caller() == sender`. The denom is scoped to the calling contract (`evm/<caller>`), but the **source of funds** is an arbitrary address supplied by the caller. Any contract can therefore drain `evm/<its_address>` tokens from any holder.

The same flaw exists in the `burn` case: the address to burn from (`recipient` in the code, line 121) is taken from calldata with no authorization check that the caller is permitted to burn on behalf of that address. [1](#0-0) [2](#0-1) 

### Impact Explanation
**Critical.** Any unprivileged contract can:
1. Call `bank.transfer(victimAddress, attackerAddress, amount)` — this executes `bankKeeper.SendCoins(ctx, victim, attacker, evm/<callerContract> coins)` with no consent from `victim`.
2. Call `bank.burn(victimAddress, amount)` — this destroys `evm/<callerContract>` tokens held by `victim` without their consent.

This constitutes an **unauthorized transfer and burn of precompile-controlled assets** (`evm/` denomination native tokens), which is a Critical impact under the allowed scope. [3](#0-2) 

### Likelihood Explanation
The `evm/` denomination tokens are a live feature. `TestBank.sol` demonstrates the exact pattern where users convert ERC20 tokens into native `evm/<contract>` bank tokens via `moveToNative()`. Any user who holds `evm/<contractAddress>` tokens is vulnerable to the contract at that address draining them. The attacker only needs to deploy a contract and call the bank precompile — no special privileges required. [4](#0-3) 

### Recommendation
For `transfer`, verify that the calling contract is the authorized sender:
```go
// Enforce: only the calling contract may initiate a transfer on behalf of itself
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer on behalf of sender")
}
```
For `burn`, similarly verify that `contract.Caller() == recipient` (the address being burned from), or require an explicit approval mechanism before allowing a contract to burn another address's tokens.

### Proof of Concept
1. Attacker deploys `MaliciousContract` at address `0xATK`. This contract exposes:
   ```solidity
   function steal(address victim, address attacker, uint256 amount) external {
       IBankModule(0x0000000000000000000000000000000000000064)
           .transfer(victim, attacker, amount);
   }
   ```
2. Victim previously called `MaliciousContract.moveToNative(100)`, converting 100 ERC20 tokens into 100 `evm/0xATK` native bank tokens.
3. Attacker calls `MaliciousContract.steal(victim, attacker, 100)`.
4. The bank precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, 100 evm/0xATK)` — no approval from victim was required.
5. Victim's 100 `evm/0xATK` tokens are transferred to the attacker. [5](#0-4)
