### Title
Unrestricted `update_rewards` with `disable_rewards: true` Allows Permanent Griefing of Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in the Staking contract has no access control and accepts a caller-controlled `disable_rewards` flag. Because `last_reward_block` is written **before** the `disable_rewards` early-return check, any unprivileged caller can consume each block's reward slot without distributing rewards, permanently preventing all stakers from accruing consensus-era yield.

### Finding Description

`StakingRewardsManagerImpl::update_rewards` is a public function gated only by `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero. There is no role check. [1](#0-0) 

The function writes `last_reward_block` to the current block number unconditionally, **before** branching on `disable_rewards`: [2](#0-1) 

If `disable_rewards` is `true`, the function returns immediately without calling `_update_rewards`, so no rewards are distributed to any staker: [3](#0-2) 

Because `last_reward_block` is now equal to the current block, every subsequent call to `update_rewards` in the same block fails: [4](#0-3) 

`last_reward_block` is a single global storage slot shared across all stakers: [5](#0-4) 

The `general_prerequisites` guard provides no role protection: [6](#0-5) 

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` in every block permanently prevents all stakers from receiving consensus-era block rewards. The `last_reward_block` slot is consumed without distributing yield, and no legitimate staker can reclaim that block's rewards. This constitutes **permanent freezing of unclaimed yield** for the entire staker set as long as the attacker continues the griefing.

This maps to the allowed impact: **Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation

- The entry point is fully public; no token balance, staking position, or privileged role is required.
- The attacker only needs to know any one active staker address (all are publicly enumerable via the `stakers` vector).
- The cost per block is a single cheap Starknet transaction.
- The attack is sustainable indefinitely.

### Recommendation

Add an access-control check so that only the staker themselves (or their operational/reward address) can call `update_rewards`. Alternatively, move the `self.last_reward_block.write(current_block_number)` write to **after** the `disable_rewards` guard so that a disabled call does not consume the block slot.

### Proof of Concept

1. Staker Alice stakes and becomes active with a non-zero STRK balance.
2. Attacker (any address) calls `staking.update_rewards(alice_address, disable_rewards: true)` in block N.
3. `last_reward_block` is set to N; no rewards are distributed.
4. Alice calls `staking.update_rewards(alice_address, disable_rewards: false)` in the same block N → reverts with `REWARDS_ALREADY_UPDATED`.
5. Attacker repeats step 2 in every subsequent block.
6. Alice (and every other staker) accrues zero consensus rewards indefinitely.

The root cause is the unconditional write at: [7](#0-6) 

followed by the unchecked early return at: [8](#0-7)
