### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees (up to 6.55%), Bypassing Protocol Fee Cap System — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` (both `uint16`, max 65535 ≈ 6.55% in E6 units) to `MetricOmmPool.setBinAdditionalFees` with no cap check. The global admin spread fee path (`setPoolAdminFees`) is bounded by `maxAdminSpreadFeeE6` (itself bounded by `HARD_MAX_SPREAD_FEE_E6`), but the per-bin additional fee path has no analogous guard. A pool admin can therefore set per-bin fees to `uint16.max` and extract up to 6.55% additional fee from every swap through that bin, bypassing the protocol's fee cap system entirely.

### Finding Description
The factory's `setPoolAdminFees` enforces:
```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```
But `setPoolBinAdditionalFees` has no equivalent check:
```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```
The pool's `setBinAdditionalFees` also has no cap check — it stores the raw `uint16` values. During every swap through the affected bin, the swap math adds `addFeeBuyE6` or `addFeeSellE6` directly to the base fee, increasing the amount the trader must pay. The excess fee accrues to LPs (and a protocol share), draining trader principal.

### Impact Explanation
Traders swapping through a bin with `addFeeBuyE6 = 65535` or `addFeeSellE6 = 65535` pay up to 6.55% additional fee on top of the base spread fee. This is a direct loss of trader principal into LP surplus, with no protocol-enforced ceiling. The pool admin can apply this to any or all bins, making the pool effectively a fee-extraction trap for traders who cannot observe or anticipate the per-bin fee state.

### Likelihood Explanation
The pool admin is explicitly modeled as semi-trusted (not fully trusted) — the entire cap system exists to constrain admin actions. Any pool admin (which could be a compromised key, a malicious deployer, or a DAO vote) can call this function at any time with no timelock and no cap. The absence of a cap is a structural gap, not a configuration choice.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Either reuse `maxAdminSpreadFeeE6` or introduce a dedicated `maxBinAdditionalFeeE6` storage variable bounded by a hard ceiling:
```solidity
if (addFeeBuyE6 > maxBinAdditionalFeeE6 || addFeeSellE6 > maxBinAdditionalFeeE6)
    revert BinAdditionalFeeTooHigh();
```
Apply the same check inside `MetricOmmPool.setBinAdditionalFees` as a defense-in-depth guard.

### Proof of Concept
1. Deploy a pool with one bin (bin index 0), any valid oracle and token pair.
2. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Execute a swap through bin 0.
4. Observe that the trader's effective fee is `baseFee + 65535/1e6` ≈ `baseFee + 6.55%`, far exceeding `maxAdminSpreadFeeE6`.
5. Assert that the trader received materially less than the oracle mid-price permits, with the difference accruing to LP balances — confirming direct principal loss with no protocol-enforced bound.
