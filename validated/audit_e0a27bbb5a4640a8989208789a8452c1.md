### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via `setPoolBinAdditionalFees`, Charging Traders Uncapped Per-Bin Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the global admin spread fee, but `setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no cap check. A pool admin can set per-bin additional fees to `uint16.max` (65535 = 6.5535%) regardless of the factory's configured `maxAdminSpreadFeeE6`, bypassing the protocol's admin fee ceiling entirely.

---

### Finding Description

`setPoolAdminFees` explicitly guards against exceeding `maxAdminSpreadFeeE6`: [1](#0-0) 

`setPoolBinAdditionalFees` has no such guard — it forwards the caller-supplied values directly: [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index, not the fee magnitude: [3](#0-2) 

The uncapped value is then stored directly into `BinState.addFeeBuyE6` / `addFeeSellE6`: [4](#0-3) 

And applied verbatim to every swap through that bin: [5](#0-4) 

---

### Impact Explanation

A pool admin can set `addFeeBuyE6 = 65535` (6.5535%) on any bin, far exceeding a `maxAdminSpreadFeeE6` of, say, 1000 (0.1%). Every swap routed through that bin pays the inflated fee. The excess fee accrues to the pool's LP fee pool (not directly to the admin), but traders suffer direct principal loss above the protocol-guaranteed cap on every affected swap. This is a direct, per-swap loss of user funds caused by an admin-boundary break that the `maxAdminSpreadFeeE6` mechanism was explicitly designed to prevent.

---

### Likelihood Explanation

The pool admin is a semi-trusted role bounded by caps and timelocks. The bypass requires only a single call to `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535`. No special conditions, no timelock, no co-conspirator. Any pool admin can execute this immediately after pool creation.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

---

### Proof of Concept

1. Deploy factory with `maxAdminSpreadFeeE6 = 1000` (0.1%).
2. Create a pool; pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert occurs — `setBinAdditionalFees` only checks bin index bounds.
4. Read `BinState` for bin 0: `addFeeBuyE6 == 65535`, `addFeeSellE6 == 65535`.
5. Execute a swap through bin 0: effective fee = `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, which is 6.5535% above the oracle mid-price, far exceeding the 0.1% cap.
6. Confirm `setPoolAdminFees(pool, 1001, 0)` reverts with `AdminFeeTooHigh` — proving the bin path is the only uncapped route.
