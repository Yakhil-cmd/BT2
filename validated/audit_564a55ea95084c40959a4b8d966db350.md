### Title
Unbounded Growth of `stakers` Vector Causes Unbounded Gas Consumption in `get_stakers` - (File: src/staking/staking.cairo)

### Summary
Every call to `stake()` appends the staker's address to the `stakers` storage vector, but `remove_staker()` never removes entries from it. The consensus-critical `get_stakers()` function iterates over the **entire** vector on every call. An unprivileged attacker can inflate this vector without bound by repeatedly staking with fresh addresses and then unstaking, causing `get_stakers()` to consume ever-increasing gas until it becomes uncallable.

### Finding Description

When a new staker registers, their address is permanently appended to the `stakers` vector: [1](#0-0) 

When a staker exits via `unstake_action()` → `remove_staker()`, the function clears `staker_info`, zeroes the own balance, and clears the operational address mapping, but **never removes the address from the `stakers` vector**: [2](#0-1) 

The `get_stakers()` function — called by the consensus layer to determine the active validator set — iterates over `self.stakers.into_iter_full_range()`, which covers every address ever pushed, including all exited stakers: [3](#0-2) 

The `assert_staker_address_not_reused` guard prevents re-use of the same address, so each attack cycle requires a fresh address: [4](#0-3) 

This means the vector is strictly monotonically growing. Every staker who has ever existed — active or not — permanently occupies a slot that `get_stakers()` must visit.

### Impact Explanation

`get_stakers()` is a consensus-critical read function. As the `stakers` vector grows, each call to `get_stakers()` consumes proportionally more gas. Once the vector is large enough, the function will exceed the Starknet block gas limit and revert on every call, permanently preventing the consensus layer from reading the validator set. This constitutes **unbounded gas consumption** leading to a denial-of-service on a protocol-critical function.

This matches the allowed impact: **Medium — Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption**.

### Likelihood Explanation

The attack is cheap and permissionless. The attacker only needs to hold `min_stake` tokens per cycle; after the `exit_wait_window` elapses they recover their principal. Running many addresses in parallel amortises the wait. No privileged role is required. The only cost is gas per cycle, making sustained inflation economically feasible.

### Recommendation

Remove the staker's address from the `stakers` vector inside `remove_staker()`. Because Cairo's `Vec` does not support O(1) removal, the standard approach is a swap-and-pop pattern: swap the exiting staker's slot with the last element, then pop the last element. Alternatively, maintain a separate `active_stakers_count` and use a doubly-linked list or a tombstone bitmap so that `get_stakers()` can skip dead entries without iterating them.

### Proof of Concept

1. Deploy the staking system with `min_stake = S`.
2. For `i = 1 … N`, using a fresh address `addr_i` each time:
   a. Fund `addr_i` with `S` STRK and approve the staking contract.
   b. Call `stake(addr_i, S, …)` → `stakers` vector length becomes `i`.
   c. Call `unstake_intent()` from `addr_i`.
   d. Advance time past `exit_wait_window`.
   e. Call `unstake_action(addr_i)` → principal returned, but `stakers[i-1]` is never cleared.
3. After `N` cycles, call `get_stakers(epoch_id)`.
4. Observe that the function iterates all `N` slots (most of which are dead), consuming `O(N)` gas.
5. For sufficiently large `N`, the call exceeds the block gas limit and reverts, making the validator set permanently unreadable.

        fn remove_staker(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_info: InternalStakerInfoLatest,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
        ) {
            self.insert_staker_own_balance(:staker_address, own_balance: Zero::zero());

            self
                .emit(
                    Events::DeleteStaker {
                        staker_address,
                        reward_address: staker_info.reward_address,
                        operational_address,
                        pool_contracts,
                    },
                );
        }
