### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire `pool_member_epoch_balance` trace. Because a pool member can grow this trace without bound by repeatedly making balance changes across epochs, a sufficiently large trace will cause any `claim_rewards` call to exceed Starknet's step/gas limit, permanently freezing the pool member's unclaimed rewards.

### Finding Description

`Pool::calculate_rewards` iterates over every entry in the `pool_member_epoch_balance` trace from `entry_to_claim_from` up to `pool_member_trace_length`: [1](#0-0) 

The code itself acknowledges the risk with the comment:

> **Note**: The loop iterates over the balance changes in the pool member's balance trace. **This loop is unbounded but unlikely to exceed gas limits.**

The trace grows by one checkpoint per epoch whenever a pool member calls:
- `enter_delegation_pool` → `set_member_balance` → `trace.insert(key: epoch+K, ...)`
- `add_to_delegation_pool` → `increase_member_balance` → `trace.insert(key: epoch+K, ...)`
- `exit_delegation_pool_intent` → `set_member_balance` → `trace.insert(key: epoch+K, ...)` [2](#0-1) [3](#0-2) 

The `insert` function only deduplicates within the same epoch key; balance changes across different epochs each produce a distinct checkpoint: [4](#0-3) 

`calculate_rewards` is called unconditionally inside `claim_rewards` before any state is written: [5](#0-4) 

Because the state update (`entry_to_claim_from` advancement) only persists if the transaction succeeds, a trace that is too large to process in one call will cause every future `claim_rewards` attempt to revert, with no partial-progress escape hatch.

Additionally, `find_sigma` is invoked once per loop iteration inside `calculate_rewards`: [6](#0-5) 

If `find_sigma` itself performs a storage scan, the effective complexity is super-linear, worsening the gas impact.

### Impact Explanation

A pool member who accumulates a sufficiently large balance trace — by making balance changes across many epochs without claiming rewards — will find that `claim_rewards` permanently reverts due to exceeding Starknet's step limit. Their accrued STRK rewards become permanently frozen in the pool contract with no recovery path, because:

1. There is no partial-claim mechanism (the loop must complete in one transaction).
2. `entry_to_claim_from` is only written on success, so every retry starts the same failing loop.

This matches the allowed impact: **Permanent freezing of unclaimed yield (High)** and **Unbounded gas consumption (Medium)**.

### Likelihood Explanation

**Medium.** The entry path requires only unprivileged pool member actions (`add_to_delegation_pool`, `exit_delegation_pool_intent`) repeated across epochs. Starknet epochs are short (on the order of hours to days), so a long-lived pool member who actively manages their position — or one who deliberately alternates deposits and partial-exit intents — can accumulate hundreds of trace entries over months. The protocol has no cap on trace length and no mechanism to prune old entries.

### Recommendation

1. **Bound the loop**: Introduce a maximum number of iterations per `claim_rewards` call and persist `entry_to_claim_from` after each partial run, allowing rewards to be claimed in multiple transactions.
2. **Limit trace growth**: Enforce a maximum number of balance-change checkpoints per pool member, or require a minimum epoch gap between successive balance changes.
3. **Prune processed entries**: After `entry_to_claim_from` advances past old checkpoints, allow those entries to be deleted from storage to keep the active trace length bounded.

### Proof of Concept

1. Pool member calls `enter_delegation_pool` in epoch 1 → trace length = 1.
2. In each subsequent epoch N, pool member calls `add_to_delegation_pool(1)` → trace length grows by 1 per epoch.
3. Pool member never calls `claim_rewards`.
4. After ~10,000 epochs, trace length ≈ 10,000.
5. Pool member (or their reward address) calls `claim_rewards`.
6. `calculate_rewards` enters the `while` loop and iterates 10,000 times, each iteration calling `find_sigma` (additional storage reads).
7. Transaction exceeds Starknet's step limit and reverts.
8. `entry_to_claim_from` is not updated; every subsequent `claim_rewards` call also reverts.
9. Pool member's entire accrued STRK reward balance is permanently frozen. [7](#0-6)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
