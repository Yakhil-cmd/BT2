### Title
Presigning Participant Set Not Bound to Signing Participant Set Enables Split-View Secret Key Extraction — (`src/ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/`)

---

### Summary

The robust ECDSA scheme's `RerandomizationArguments` binds a presignature to the **signing** participant set (P2) via HKDF, but does not include the **presigning** participant set (P1). Because no code-level constraint enforces P1 = P2 or |P1| = 2t+1, a malicious coordinator can run presigning with N1 > 2t+1 participants and then drive two parallel signing sessions with different overlapping subsets of size 2t+1. Both sessions consume the same presignature nonce R with different Lagrange linearization coefficients, producing two signatures whose nonces are multiplicatively related — enabling full secret key extraction via standard ECDSA nonce-reuse algebra.

---

### Finding Description

`RerandomizationArguments` is the structure that is supposed to bind a presignature to a unique signing context before it is consumed:

```rust
// src/ecdsa/mod.rs ~line 94
pub struct RerandomizationArguments {
    pub pk: AffinePoint,
    pub tweak: Tweak,
    pub msg_hash: [u8; 32],
    pub big_r: AffinePoint,
    pub participants: ParticipantList,   // ← signing set P2 only
    pub entropy: [u8; 32],
}
```

`derive_randomness()` feeds `pk`, `tweak`, `msg_hash`, `big_r`, and the **signing** participants into HKDF-SHA3-256 to produce the rerandomization scalar δ. The **presigning** participant set P1 is never included. [1](#0-0) [2](#0-1) 

Because P1 is absent from the binding, the same `PresignOutput` (same `big_r`, same secret shares k_i, σ_i) can be rerandomized independently for two different signing sets P2 and P2′ that are distinct subsets of P1. Each rerandomization produces a different δ, so the two signing sessions see different R′ = δ·R and different s-share formulas — but both are algebraically derived from the **same** underlying nonce k. This is precisely the split-view attack described in the library's own security-considerations section: [3](#0-2) 

The security-considerations document lists four mitigations (enforce N1 = N2 = 2t+1, agree on (h,ε,participants), never reuse a presignature, reject h=0), but all four are stated as **recommendations to the caller** — none are enforced in the production signing or presigning code paths. [4](#0-3) 

The analog to the external report is direct:

| External Report | This Codebase |
|---|---|
| Staking tokens deposited in both `StakingRewards` and `ConvexStakingWrapper` simultaneously | Presignature consumed in two signing sessions (P2, P2′) simultaneously |
| Shelter activates for `ConvexStakingWrapper` but does not wipe balance from `MasterChef` | Signing session 1 "uses" the presignature but does not invalidate it for session 2 |
| User claims double Concur rewards | Coordinator obtains two signatures with related nonces → full key extraction |

### Impact Explanation

**Critical — Extraction of the aggregate secret signing key.**

Given two signatures (R1, s1) and (R2, s2) produced from the same presignature nonce k under different linearization coefficients λ(P2) and λ(P2′), the relationship between s1 and s2 is linear in the secret x. Standard ECDSA nonce-reuse algebra recovers x directly. The library's own documentation confirms: *"the secret key can be recovered using standard ECDSA nonce-reuse attacks"* and *"a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."* [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The attacker must be the coordinator (a designated role in the protocol). A malicious coordinator controls which participant subsets are assembled for each signing session and can present different (h, ε, participants) to different signers without any of the honest signers being able to detect the inconsistency, because each signer only communicates with the coordinator in the signing round. [6](#0-5) 

---

### Recommendation

1. **Include the presigning participant set P1 in `RerandomizationArguments`** and add it to the HKDF input in `derive_randomness()`. This cryptographically binds each rerandomized presignature to a unique (P1, P2) pair, making it impossible to rerandomize the same presignature for two different signing sets.

2. **Enforce N1 = N2 = 2t+1 in code**, not just in documentation. At presign initialization, record the presigning participant set. At sign initialization, assert that the signing participant set equals the presigning participant set exactly.

3. **Enforce single-use of presignatures at the library level** by consuming (zeroing) the `PresignOutput` after the first rerandomization call, preventing any second use even across aborted sessions.

---

### Proof of Concept

1. Coordinator runs `presign` with P1 = {A, B, C, D, E} (N1 = 5, t = 2, so 2t+1 = 5 — but the code does not prevent N1 = 6 or more).
2. Coordinator constructs `RerandomizationArguments` with P2 = {A, B, C}, msg_hash = h, tweak = ε, entropy = ρ → derives δ1, produces rerandomized presignature RP1.
3. Coordinator constructs a second `RerandomizationArguments` with P2′ = {C, D, E}, same h, same ε, same ρ → derives δ2 ≠ δ1, produces rerandomized presignature RP2.
4. Coordinator runs two signing sessions in parallel: session 1 with {A,B,C} using RP1, session 2 with {C,D,E} using RP2.
5. Both sessions produce valid-looking signatures (R1, s1) and (R2, s2). Because both R1 and R2 are scalar multiples of the same base nonce point, the secret key x satisfies a linear system solvable from (R1, s1, R2, s2, h, ε). [7](#0-6) [8](#0-7)

```rust
    /// Derives a random string from the public key, tweak, message hash, presignature R,
    /// set of participants and the entropy.
    ///
    /// Outputs a random string computed as HKDF(entropy, pk, hash, R, participants)
    pub fn derive_randomness(&self) -> Result<Scalar, ProtocolError> {
        // create a string containing (pk, msg_hash, big_r, sorted(participants))
        let pk_encoded_point = self.pk.to_encoded_point(true);
        let encoded_pk: &[u8] = pk_encoded_point.as_bytes();
        let encoded_tweak: &[u8] = &<Secp256K1ScalarField as Field>::serialize(&self.tweak.value());
        let encoded_msg_hash: &[u8] = &self.msg_hash;
        let big_r_encoded_point = self.big_r.to_encoded_point(true);
        let encoded_big_r: &[u8] = big_r_encoded_point.as_bytes();

        // concatenate all the bytes
        let mut concatenation = Vec::new();
        // 1 byte counter, used in the unlikely case that the hash result is 0
        concatenation.extend_from_slice(&[0u8, 1]);
        concatenation.extend_from_slice(encoded_pk);
        concatenation.extend_from_slice(encoded_tweak);
        concatenation.extend_from_slice(encoded_msg_hash);
        concatenation.extend_from_slice(encoded_big_r);
        // Append each ParticipantId's
        for participant in self.participants.participants() {
            concatenation.extend_from_slice(&participant.bytes());
        }

        // initiate hkdf with the salt and with some `good' entropy
        let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);

        let mut delta = Scalar::ZERO;
        // If the randomness created is 0 then we want to generate a new randomness
        while bool::from(delta.is_zero()) {
            // Generate randomization out of HKDF(counter, entropy, pk, msg_hash, big_r, participants, )
            // where entropy is a public but unpredictable random string.
            // The counter depends on the number of times we enter into this loop
            let mut okm = [0u8; 32];

            hk.expand(&concatenation, &mut okm)
                .map_err(|_| ProtocolError::HashingError)?;

            // derive the randomness delta
            delta = Scalar::from_repr(okm.into()).unwrap_or(
                // if delta falls outside the field
                // probability is negligible: in the order of 1/2^224
                Scalar::ZERO,
            );
            // Increment the counter, the probability that this overflows is astronomically low
            let concatenation_0 = concatenation
                .first_mut()
                .ok_or(ProtocolError::InvalidIndex)?;
            *concatenation_0 += 1;
        }
        Ok(delta)
    }
```

```markdown
# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

4. **Do not sign with $h = 0$** (the zero message hash).
   This input enables a related algebraic split-view attack in the modified scheme when
   $N_1 > 2t + 1$.
```

```markdown
## Threshold

The threshold parameter is `MaxMalicious`, denoted `t`. Both presigning and signing require **exactly** `N = 2t + 1` participants. This constraint is enforced at initialization and prevents split-view attacks where different subsets sign different messages using shares from the same presignature.

Additionally, `msg_hash == 0` is rejected to prevent a related-key split-view attack.
```
