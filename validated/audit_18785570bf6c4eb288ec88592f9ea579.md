### Title
Unresponsive Coordinator Permanently Blocks Honest Participants in FROST EdDSA `sign_v1` and RedJubjub `sign` Protocols — (`src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`)

---

### Summary

In the FROST EdDSA `sign_v1` and RedJubjub `sign` protocols, non-coordinator participants enter an unbounded blocking loop waiting for a message that **only the coordinator can send**. If the coordinator becomes unresponsive (crash, network partition, or deliberate refusal) after collecting participant commitments, all honest participants are permanently stuck. There is no timeout, no fallback, and no mechanism for any other party to advance the protocol. This is a direct structural analog to the Audius H12 bug: a single privileged party is the only one who can advance a shared protocol state, and their absence permanently locks all other honest parties.

---

### Finding Description

**FROST EdDSA `sign_v1` — `do_sign_participant_v1`** (`src/frost/eddsa/sign.rs`, lines 234–301):

1. Each non-coordinator participant generates fresh nonces and sends their commitment to the coordinator (step 1.2, line 265).
2. The participant then enters an infinite `loop` at line 273, blocking on `chan.recv(r2_wait_point).await?`, waiting exclusively for the `SigningPackage` from the coordinator:

```rust
let signing_package = loop {
    let (from, signing_package): (_, frost_ed25519::SigningPackage) =
        chan.recv(r2_wait_point).await?;
    if from != coordinator {
        continue;
    }
    break signing_package;
};
```

Only the coordinator constructs and sends the `SigningPackage` (line 137 in `do_sign_coordinator_v1`). No other participant can send it. If the coordinator is unresponsive after receiving commitments, this loop never terminates.

**RedJubjub `sign` — `do_sign_participant`** (`src/frost/redjubjub/sign.rs`, lines 197–235):

The same pattern occurs for the `Randomizer`. Participants wait in an infinite loop (lines 217–223) for the coordinator to send the `Randomizer`:

```rust
let randomizer = loop {
    let (from, randomizer): (_, Randomizer) = chan.recv(wait_round_1).await?;
    if from != coordinator {
        continue;
    }
    break randomizer;
};
```

Only the coordinator computes and sends the `Randomizer` (line 150 in `do_sign_coordinator`). If the coordinator is unresponsive, participants are permanently blocked, and their presignature nonces (already committed to in `presignature.commitments_map`) are permanently consumed — they cannot be safely reused.

**Contrast with other protocols**: In OT-based ECDSA (`do_sign_participant`, `src/ecdsa/ot_based_ecdsa/sign.rs`) and Robust ECDSA (`do_sign_participant`, `src/ecdsa/robust_ecdsa/sign.rs`), participants send their share to the coordinator and immediately return `Ok(None)` — they do not block waiting for coordinator output. The blocking pattern is unique to the FROST-based protocols.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

A malicious or unresponsive coordinator can halt the signing session for all honest participants indefinitely. In `sign_v1`, participants have already committed their nonces; in RedJubjub `sign`, participants have consumed their presignature material. Neither can safely restart the same session. The library exposes no abort or coordinator-rotation mechanism at the protocol level. The only recovery path is for the application layer to drop the entire protocol instance and restart from scratch — but the library provides no documented guidance for this, and the consumed presignature nonces in RedJubjub cannot be recovered.

---

### Likelihood Explanation

**Medium.** The coordinator is a single designated participant passed in at call time. Any of the following realistic scenarios triggers the denial:
- The coordinator crashes or loses connectivity after receiving commitments but before sending the signing package / randomizer.
- A malicious participant is assigned the coordinator role and deliberately withholds the signing package to block a specific signing session.
- Network partitioning isolates the coordinator from the rest of the group.

No special cryptographic capability is required — the attacker only needs to be assigned the coordinator role and then go silent.

---

### Recommendation

1. **Add a timeout to the coordinator-wait loops** in `do_sign_participant_v1` and `do_sign_participant` (RedJubjub). If no message is received within a configurable deadline, return a `ProtocolError` so the caller can abort and retry with a different coordinator.
2. **Document the coordinator liveness assumption** explicitly in the public API (`sign_v1`, `sign_v2`, `sign` for RedJubjub) so callers understand that coordinator unavailability permanently blocks the session.
3. **Consider allowing any threshold-many participants to reconstruct and broadcast the signing package**, removing the coordinator as a single point of liveness failure — analogous to the Audius fix of allowing anyone to call `claimRewards`.
