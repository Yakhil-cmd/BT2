### Title
Per-bin additional fees are uncapped, allowing pool admin to bypass the protocol fee cap system and overcharge traders - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no upper-bound check. Every other admin fee setter enforces a factory-level cap, but the per-bin path is entirely outside that cap system. A pool admin can set bin additional fees up to `uint16` max (65 535 E6 ≈ 6.55%) on any active bin, charging traders more than the protocol's intended hard ceiling and causing direct loss of swap input tokens.

---

### Finding Description

The factory defines hard fee ceilings and enforces them on every admin fee path — except one.

`setPoolAdminFees` checks both `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` before writing: [1](#0-0) 

`setFeeCaps` itself is bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) and `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` (1%): [2](#0-1) 

But `setPoolBinAdditionalFees` forwards the caller-supplied values straight to the pool with no validation beyond the bin-index range check: [3](#0-2) 

The pool's `setBinAdditionalFees` likewise only validates the bin index: [4](#0-3) 

During every swap the per-bin additional fee is added directly to `baseFeeX64` before the SwapMath call: [5](#0-4) 

The same pattern applies to all four swap directions (`addFeeBuyE6` on lines 999, `addFeeSellE6` on lines 1088 and 1177). Because `addFeeBuyE6` / `addFeeSellE6` are `uint16`, the maximum settable value is 65 535, which in E6 units is ≈ 6.55% — entirely outside the cap system and not tracked in `poolFeeConfig`.

---

### Impact Explanation

A pool admin can set `addFeeBuyE6 = 65535` and/or `addFeeSellE6 = 65535` on any bin. Every swap routed through that bin pays an additional ≈ 6.55% fee on top of the already-capped spread fee, directly reducing the output (or increasing the input) the trader receives. This is a direct, quantifiable loss of user swap principal that bypasses the protocol's hard fee ceiling. If set on the active bin, every swap in the pool is affected until the admin resets the value.

---

### Likelihood Explanation

The trigger is a pool admin calling `setPoolBinAdditionalFees` with `addFeeBuyE6` or `addFeeSellE6` at or near `type(uint16).max`. Pool admins are semi-trusted and are explicitly expected to be constrained by caps; this path has no such constraint. A malicious or compromised admin can exploit it immediately, with no timelock and no protocol override short of a protocol-level pause.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and mirror it in `setBinAdditionalFees`) analogous to the checks already present in `setPoolAdminFees`:

```solidity
// In MetricOmmPoolFactory.setPoolBinAdditionalFees:
if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` cap that the factory owner can configure, keeping the bin-fee ceiling independent of but bounded by the same hard limit.

---

### Proof of Concept

1. Factory is deployed; `maxAdminSpreadFeeE6 = 200_000` (20% hard cap).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. A trader calls `swap` routing through bin 0. The effective fee applied is:
   ```
   baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)   // ≈ baseFee + 6.55%
   ``` [6](#0-5) 
4. The trader pays ≈ 6.55% more than the oracle spread alone, exceeding the protocol's intended 20% hard ceiling when combined with the base spread fee — with no on-chain guard having fired.
