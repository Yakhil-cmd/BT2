### Title
Missing Sender Address Blocklist in Gateway Stateless Validation Allows Any Address to Submit Transactions - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary
`StatelessTransactionValidator::validate()` contains an explicit TODO acknowledging that sender address blocking was planned but never implemented. No data structure, configuration field, or enforcement logic exists to block specific sender addresses from submitting transactions through the gateway. Any address — including known malicious or sanctioned ones — can submit invoke and deploy-account transactions unconditionally.

### Finding Description
In `StatelessTransactionValidator::validate()`, line 34 reads:

```rust
// TODO(Arni, 1/5/2024): Add a mechanism that validate the sender address is not blocked.
``` [1](#0-0) 

The `StatelessTransactionValidatorConfig` struct has no field for a blocked-addresses list, and the `add_tx_inner()` flow in `gateway.rs` calls `self.stateless_tx_validator.validate(&tx)` without any sender-address blocking check ever executing. [2](#0-1) [3](#0-2) 

By contrast, the gateway **does** implement address-based admission control for declare transactions: `GatewayStaticConfig` carries `authorized_declarer_accounts` and `check_declare_permissions()` enforces it before stateless validation runs. [4](#0-3) [5](#0-4) 

This demonstrates the codebase already has the pattern for address-level admission control, but the analogous blocklist for invoke and deploy-account transactions was never built.

### Impact Explanation
The gateway's stateless admission path — the first and cheapest enforcement point — cannot reject transactions from any sender address. An operator who needs to block a specific address (e.g., a sanctioned account, a known attacker flooding the mempool) has no mechanism to do so at the gateway layer. Every such transaction passes stateless validation, proceeds through conversion and stateful validation, and enters the mempool. This matches the **High** impact category: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation
Triggering the gap requires no privilege: any user submitting an invoke or deploy-account transaction from any address exercises the missing check. The TODO comment dated 1 May 2024 confirms the gap has been known and unaddressed for over a year.

### Recommendation
1. Add a `blocked_sender_addresses: Vec<ContractAddress>` (or `HashSet`) field to `StatelessTransactionValidatorConfig`.
2. In `StatelessTransactionValidator::validate()`, immediately after `validate_contract_address`, check whether `tx.sender_address()` is in the blocked set and return a `StatelessTransactionValidatorError::BlockedSenderAddress` if so.
3. Expose the field in `SerializeConfig::dump()` so it can be set via the node's configuration file.

### Proof of Concept
1. Deploy any account on Starknet.
2. Submit an invoke transaction from that account to the gateway endpoint.
3. Observe that `StatelessTransactionValidator::validate()` runs all checks **except** the missing sender-address block check (line 34 is a comment, not code).
4. The transaction passes stateless validation regardless of the sender address, proceeds to stateful validation, and is admitted to the mempool — even if the operator intended to block that address. [6](#0-5)
