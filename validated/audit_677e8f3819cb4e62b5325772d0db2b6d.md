Audit Report

## Title
Caller Identity Not Verified in `refresh_buyer_tokens` Bypasses Confirmation Text Consent Mechanism - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal with no check that it matches the actual caller. Because the SNS confirmation text is stored publicly on-chain in the swap's `Init` parameters, any unprivileged caller can supply both a victim's principal and the correct confirmation text, registering the victim's ICP deposit as a confirmed participation without the victim's knowledge or consent. If the swap commits, the victim's ICP is swept to the SNS treasury and they receive SNS tokens in exchange — an outcome they never explicitly agreed to.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 128–143, the `refresh_buyer_tokens` endpoint parses `arg.buyer` directly into a `PrincipalId` with no `buyer == caller()` guard:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no authorization check
};
```

The inner `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` lines 1149–1150 then validates the confirmation text against the SNS-configured value:

```rust
self.validate_confirmation_text(confirmation_text)?;
```

`validate_confirmation_text` (lines 363–384 of `swap.rs`) performs only a string equality check against `self.init_or_panic().confirmation_text` — it does not verify that the caller is the principal whose participation is being registered. The confirmation text is stored in the public `Init` struct (proto field `confirmation_text`, `swap.proto` line 325), readable by anyone via `get_state`.

After `refresh_buyer_token_e8s` succeeds, the victim is inserted into `self.buyers` (lines 1285–1288 of `swap.rs`). The `error_refund_icp` path (lines 1950–1959 of `swap.rs`) then blocks any refund while the buyer has ICP in escrow:

```rust
if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
    if let Some(transfer) = &buyer_state.icp
        && transfer.transfer_success_timestamp_seconds == 0
    {
        return ErrorRefundIcpResponse::new_precondition_error(...)
    }
}
```

This behavior is confirmed by the integration test in `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–944: after the wealthy user transfers ICP, the **default identity** (a different principal) successfully calls `refresh_buyer_tokens` with `wealthy_user_identity.principal_id` as the buyer and the call succeeds (`unwrap()` with no error). The test explicitly validates this cross-caller registration as working behavior.

## Impact Explanation
This matches the allowed High impact: **Significant SNS security impact with concrete user or protocol harm**. An SNS that configures a `confirmation_text` does so to require explicit per-participant consent before ICP is locked in. An attacker who reads the public confirmation text can register any victim who has ICP sitting in the swap subaccount, locking their funds in escrow and, if the swap commits, causing their ICP to be swept to the SNS treasury and replaced with SNS tokens — without the victim ever calling `refresh_buyer_tokens` themselves. The victim cannot recover their ICP via `error_refund_icp` while the swap is OPEN and their participation is registered.

## Likelihood Explanation
The preconditions are low-cost and observable on-chain: (1) the swap must be OPEN, (2) the victim must have transferred ICP to the swap subaccount (a normal first step, visible on the ICP ledger), and (3) the attacker must know the confirmation text (stored in the public `Init` struct, readable via `get_state`). No special privilege, key material, or majority is required. Any unprivileged ingress sender can execute the attack. It is repeatable for every victim who has deposited ICP but not yet called `refresh_buyer_tokens`.

## Recommendation
Add a caller authorization check in the canister endpoint before using the `buyer` field:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Caller {} is not authorized to refresh tokens for buyer {}", caller, buyer);
        }
        buyer
    };
    ...
}
```

Alternatively, remove the `buyer` override entirely and always derive the principal from `caller()`. The `buyer` override field has no legitimate use case that requires a third party to register participation on behalf of another principal.

## Proof of Concept
The existing integration test in `rs/tests/nns/sns/lib/src/sns_deployment.rs` lines 919–944 already demonstrates the cross-caller registration path succeeding. A targeted PoC:

1. Deploy an SNS swap with `confirmation_text = "I agree"`.
2. Alice transfers ICP to the swap subaccount derived from her principal (observable on the ICP ledger).
3. Eve (any principal) calls:
   ```
   refresh_buyer_tokens({ buyer: "<Alice's principal>", confirmation_text: Some("I agree") })
   ```
4. The canister reads Alice's subaccount balance, validates the confirmation text (supplied by Eve), and inserts Alice into `self.buyers`.
5. Alice's `error_refund_icp` call is now blocked with "ICP cannot be refunded as principal Alice has N ICP (e8s) in escrow".
6. When the swap commits, `sweep_icp` transfers Alice's ICP to the SNS treasury and she receives SNS tokens — without ever having consented via `refresh_buyer_tokens`.

A deterministic unit test can be written against `refresh_buyer_token_e8s` directly, passing a `buyer` principal different from any simulated "caller" context, with the correct `confirmation_text`, and asserting that the buyer is inserted into `self.buyers`.
