### Title
Unprivileged EVM Caller Can Permanently Close Any IBC Channel via Relayer Precompile — (`x/cronos/keeper/precompiles/relayer.go`, `utils.go`)

### Summary
The relayer precompile's `ChannelCloseInit` handler performs no authorization check beyond verifying that the protobuf `signer` field matches the EVM `msg.sender`. Because an attacker fully controls both values, any unprivileged EVM address can permanently close any open IBC channel.

### Finding Description
`RelayerContract.Run` dispatches `ChannelCloseInit` to the generic `exec()` helper: [1](#0-0) 

Inside `exec()`, the only guard is: [2](#0-1) 

`signers[0]` is taken directly from the attacker-controlled `MsgChannelCloseInit.signer` field (a plain string in the protobuf). The attacker sets `signer` to their own EVM address bytes, calls the precompile from that same address, and `caller == e.caller` is satisfied. There is no allowlist, governance gate, or capability check on who may invoke channel-close operations.

`MsgChannelCloseInit` in ibc-go carries no proof requirement and imposes no signer authorization — the signer is only used for ICS-29 fee accounting. The IBC keeper will execute the close for any valid channel/port pair regardless of who the signer is. [3](#0-2) 

### Impact Explanation
A successful call transitions the channel state to `CLOSED`. Once closed, the channel cannot be reopened; a new channel must be negotiated. All in-flight IBC transfers on the closed channel are permanently disrupted. This satisfies the **High** impact criterion: *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows, precompile calls, IBC transfers."*

### Likelihood Explanation
The attack requires only a funded EVM account and knowledge of the target channel/port IDs (both publicly observable on-chain). No privileged access, leaked keys, or external assumptions are needed. The call is a single EVM transaction.

### Recommendation
Add an authorized-relayer allowlist or governance-controlled permission check before dispatching any channel lifecycle operation (open/close) through the precompile. At minimum, `ChannelCloseInit` and `ChannelCloseConfirm` should be gated behind an on-chain allowlist of approved relayer addresses, consistent with how production IBC relayers are permissioned in other Cosmos chains.
