### Title
Missing Caller Authorization in `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function is documented as restricted to "Only starkware sequencer" but the implementation enforces no such check. Any non-zero address can call it, consume the global `last_reward_block` slot for the current block with `disable_rewards: true`, and permanently discard that block's rewards before the legitimate sequencer call arrives.

---

### Finding Description

The spec explicitly states the access control for `update_rewards`:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation, however, only calls `general_prerequisites()`: [2](#0-1) 

`general_prerequisites()` performs exactly two checks — contract-not-paused and caller-not-zero — with no role or identity gate: [3](#0-2) 

After passing those checks, the function unconditionally writes the current block number into the **global** `last_reward_block` storage variable: [4](#0-3) 

`last_reward_block` is a single global slot, not per-staker. The guard at the top of `update_rewards` is:

```
current_block_number > self.last_reward_block.read()
```

Once written, every subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`. [5](#0-4) 

---

### Impact Explanation

An attacker who front-runs the sequencer's `update_rewards` call with `disable_rewards: true` causes:

1. `last_reward_block` is set to the current block.
2. The function returns early — no rewards are distributed to any staker or pool.
3. The sequencer's legitimate call for that block reverts with `REWARDS_ALREADY_UPDATED`.
4. There is no retroactive mechanism to recover the skipped block's rewards.

Because the design is one `update_rewards` call per block (enforced by the global `last_reward_block` gate), every block the attacker front-runs results in **permanently lost yield** for all active stakers and their delegators. The attacker can repeat this every block at negligible cost (only gas).

**Impact**: High — Permanent freezing of unclaimed yield (continuous, protocol-wide).

---

### Likelihood Explanation

- Requires no privileged role, no leaked key, no external dependency.
- The target function is public on-chain.
- The attacker only needs to know any valid active staker address (readable from public events or `get_stakers`).
- Front-running on Starknet is feasible via mempool observation or by submitting in the same block before the sequencer's transaction.
- The attack is cheap and repeatable every block.

---

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the authorized sequencer address (stored in contract state), consistent with the spec's stated access control. For example:

```cairo
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

This mirrors the pattern already used in `update_rewards_from_attestation_contract`, which correctly enforces `CALLER_IS_NOT_ATTESTATION_CONTRACT`. [6](#0-5) 

---

### Proof of Concept

1. Deploy the staking contract with two active validators (past the K-epoch activation window).
2. At the start of any block, before the sequencer submits its `update_rewards` call, submit:
   ```
   update_rewards(staker_address=<any_valid_staker>, disable_rewards=true)
   ```
   from any non-zero EOA.
3. Observe: `last_reward_block` is now set to the current block; the call returns without distributing rewards.
4. The sequencer's `update_rewards` call for the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Claim rewards for both stakers — both show zero increment for that block.
6. Repeat every block. Cumulative rewards for all stakers remain permanently at zero regardless of elapsed time.
