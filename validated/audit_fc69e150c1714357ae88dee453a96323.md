### Title
TOCTOU Race in `skip_stateful_validations` Allows Invoke Transactions with Invalid Signatures to Bypass Gateway Signature Verification and Enter the Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function queries the mempool to decide whether to skip the `__validate__` entry-point call (signature verification) for an invoke transaction with nonce=1 submitted alongside a deploy_account. The check (`account_tx_in_pool_or_recent_block`) and the subsequent `mempool_client.add_tx` call are not atomic. Between these two operations the deploy_account can be evicted from the mempool (TTL expiry or capacity-overflow eviction), causing the invoke to be admitted without any signature check even though the precondition that justified skipping validation no longer holds.

---

### Finding Description

**Root cause — non-atomic check-then-act in `extract_state_nonce_and_run_validations`:**

```
extract_state_nonce_and_run_validations          (stateful_transaction_validator.rs:158-179)
  └─ run_pre_validation_checks                   (line 175-176)
       └─ skip_stateful_validations              (line 429-461)
            └─ account_tx_in_pool_or_recent_block  ← CHECK (T1)
  └─ run_validate_entry_point(skip_validate=true)  ← sets validate=false, no __validate__ call
```

Then, back in `gateway.rs`:

```
store_proof_and_spawn_archiving  (line 268-271)   ← async gap
await_proof_archiving            (line 282-284)   ← async gap
mempool_client.add_tx            (line 286)       ← ACTION (T2)
```

`skip_stateful_validations` (lines 429–461) returns `true` (skip) when:
- the transaction is `Invoke` with `tx.nonce() == Nonce(ONE)`, and
- `account_nonce == Nonce(ZERO)`, and
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = !skip_validate = false` (line 312), so `validate_tx` returns `Ok(None)` immediately (line 999–1001 of `account_transaction.rs`) — the `__validate__` entry point is never called.

The mempool's `account_tx_in_pool_or_recent_block` (line 697–700 of `mempool.rs`) checks `self.state.contains_account` (staged/committed maps) **or** `self.tx_pool.contains_account`. If the deploy_account is only in the pool (not yet committed), and it is evicted between T1 and T2, the account disappears from both maps. The invoke is then added to the mempool with `skip_validate=true` having been decided on stale information.

**Eviction vectors between T1 and T2:**

1. **TTL expiry** — `remove_expired_txs` is called inside `add_tx` (line 481 of `mempool.rs`). A concurrent `add_tx` from another gateway request can trigger TTL cleanup that removes the deploy_account.
2. **Capacity-overflow eviction** — `handle_capacity_overflow` (line 434) can evict the deploy_account when the mempool is near capacity and a new transaction arrives.
3. **Proof-archiving latency** — for invoke transactions carrying proof facts, `await_proof_archiving` (line 282–284 of `gateway.rs`) introduces a non-trivial async gap, widening the race window. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

---

### Impact Explanation

When the race fires, an invoke transaction carrying an **invalid or attacker-controlled signature** is admitted to the mempool without the `__validate__` entry point ever being called at the gateway layer. The broken invariant is:

> *An invoke with nonce=1 may skip gateway signature verification only if a deploy_account for the same address is present in the mempool at the moment the invoke is added.*

The batcher uses `AccountTransaction::new_for_sequencing` (line 147–155 of `account_transaction.rs`) which sets `validate: true`, so the `__validate__` entry point **is** re-run at execution time and the invalid transaction will revert. However, the gateway has already admitted an invalid transaction — a transaction that would have been rejected had the check been atomic.

**Impact category:** High — Mempool/gateway admission accepts invalid transactions before sequencing. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

**Low–Medium.** The attacker must:

1. Submit a valid deploy_account for address A so it enters the mempool.
2. Submit an invoke for address A (nonce=1) with an invalid signature.
3. Cause the deploy_account to be evicted between the `account_tx_in_pool_or_recent_block` check and the `add_tx` call.

Step 3 is the hard part for TTL-based eviction (default TTL is 60 s, race window is milliseconds). However, capacity-overflow eviction is attacker-controllable: by flooding the mempool with high-priority transactions the attacker can force the deploy_account out during the async gap introduced by `await_proof_archiving` for proof-bearing invoke transactions. The gateway processes concurrent `add_tx` requests, so multiple racing requests can be submitted simultaneously. [8](#0-7) 

---

### Recommendation

1. **Re-verify after add:** After `mempool_client.add_tx` succeeds, check whether the deploy_account is still present. If it has been evicted, reject the invoke with an appropriate error.
2. **Atomic check-and-add:** Extend the mempool's `add_tx` API to accept a precondition (e.g., "account must be present") that is evaluated atomically inside the mempool actor, eliminating the TOCTOU window.
3. **Tighten the skip condition:** Only skip validation when the deploy_account transaction hash is explicitly tracked (not just "any account transaction"), so the skip cannot be triggered by unrelated future-nonce transactions that happen to be in the mempool for the same address.

---

### Proof of Concept

```
1. Fill the mempool to near-capacity with low-priority transactions from many addresses.

2. Send a valid deploy_account for address A (nonce=0, valid signature).
   → deploy_account enters the mempool pool.

3. Concurrently send an invoke for address A (nonce=1, INVALID signature).
   → Gateway enters add_tx_inner:
      a. Stateless validation passes (signature length check only, not cryptographic).
      b. convert_rpc_tx_to_internal_and_executable_txs runs (async, includes proof archiving gap).
      c. extract_state_nonce_and_run_validations:
           account_nonce = 0  (A not yet deployed)
           skip_stateful_validations:
             tx.nonce() == 1 && account_nonce == 0  → true
             account_tx_in_pool_or_recent_block(A)  → true  (deploy_account still in pool)
             skip_validate = true
           run_validate_entry_point(skip_validate=true):
             execution_flags.validate = false
             __validate__ NOT called  ← signature never checked

4. During await_proof_archiving (or via a concurrent add_tx that triggers
   handle_capacity_overflow), the deploy_account is evicted from the pool.

5. mempool_client.add_tx(invoke_with_invalid_sig) succeeds.
   → Invalid invoke is now in the mempool.

6. Batcher calls get_txs, receives the invoke.
   convert_to_executable_blockifier_tx wraps it with validate=true.
   execute_raw → perform_pre_validation_stage → validate_tx → __validate__ called.
   __validate__ fails (invalid signature) → transaction reverts / is rejected.
```

The gateway has admitted an invalid transaction. The batcher catches it, but the admission invariant is broken. [1](#0-0) [9](#0-8) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn add_tx_validations(
        &mut self,
        tx_reference: TransactionReference,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
    ) -> MempoolResult<()> {
        self.validate_incoming_tx(tx_reference, account_nonce)?;
        let replaced_tx_reference = self.validate_fee_escalation(tx_reference)?;

        // The replaced transaction is still pooled, so its bytes still count toward
        // `size_in_bytes()`. Credit what its removal will free: a same-size bump nets to zero (no
        // overflow handling), and a larger replacement only needs room for the delta, consistent
        // with how a fresh next-nonce transaction is treated. The removal happens only after
        // capacity is confirmed below, so a rejected incoming transaction never strands the
        // account.
        let freed_bytes = replaced_tx_reference.map_or(0, |reference| {
            self.tx_pool
                .get_by_tx_hash(reference.tx_hash)
                .expect("Replacement target from pool must exist.")
                .total_bytes()
        });

        if self.exceeds_capacity(tx, freed_bytes) {
            self.handle_capacity_overflow(tx, account_nonce, freed_bytes)?;
        }

        // Capacity is confirmed: this is the final, infallible mutation before the incoming
        // transaction is inserted by the caller.
        if let Some(existing_tx_reference) = replaced_tx_reference {
            self.remove_replaced_tx(existing_tx_reference);
        }

        Ok(())
    }
