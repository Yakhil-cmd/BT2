### Title
Malicious Participant Can Corrupt CKD Output by Supplying Fake `public_key` in `KeygenOutput` — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

A valid participant can pass a `KeygenOutput` with an arbitrary `public_key` to `ckd()`. Because `compute_signature_share` hashes `key_pair.public_key` without any validation against the agreed-upon master verifying key, the malicious participant's contribution is computed over a different hash point. The coordinator aggregates all contributions blindly, producing a `CKDOutput` that unmasks to a value that is not `msk · H(pk_real ∥ app_id)`.

---

### Finding Description

`KeygenOutput.public_key` is a fully public, mutable field: [1](#0-0) 

`compute_signature_share` uses it directly to compute the hash point, with no check that it matches any externally agreed-upon master verifying key: [2](#0-1) 

The coordinator aggregates every participant's `(norm_big_y, norm_big_c)` pair with no consistency check on which `pk` each participant used: [3](#0-2) 

**Attack path:**

1. Attacker is a valid member of the participant list.
2. Attacker calls `ckd()` with `KeygenOutput { public_key: pk_fake, private_share: x_j }` where `pk_fake` is any G2 point ≠ `pk_real`.
3. Their contribution becomes `λ_j · x_j · H(pk_fake ∥ app_id) + λ_j · y_j · A`.
4. Honest participants contribute `λ_i · x_i · H(pk_real ∥ app_id) + λ_i · y_i · A`.
5. The coordinator sums all `C` values. After unmasking (`C − a·Y`), the result is:

```
Σ_{honest i} λ_i · x_i · H(pk_real ∥ app_id)
  + λ_j · x_j · H(pk_fake ∥ app_id)
```

This is not equal to `msk · H(pk_real ∥ app_id)`.

---

### Impact Explanation

**High — Corruption of CKD output.** Honest parties receive and accept a `CKDOutput` that unmasks to a cryptographically wrong value. Any application relying on the derived key (e.g., a TEE application expecting `msk · H(pk_real ∥ app_id)`) will silently receive a corrupted secret. The attacker cannot predict the output (they do not know `Σ_{honest} λ_i · x_i`), so this does not rise to Critical, but it permanently corrupts the CKD result for that session.

---

### Likelihood Explanation

Any participant who has completed DKG can trivially mount this attack by constructing a `KeygenOutput` with a modified `public_key` before calling `ckd()`. The field is `pub`, no serialization or deserialization barrier prevents it, and the `ckd()` entry point performs no validation on it. [4](#0-3) 

---

### Recommendation

Before computing the hash point, validate that `key_pair.public_key` matches the expected master verifying key. One approach: require callers to pass the expected `VerifyingKey` separately (or derive it from the participant list's commitments), and assert equality at the start of `compute_signature_share`. Alternatively, have all participants broadcast their `public_key` and have the coordinator reject any contribution whose embedded `pk` differs from the agreed value.

---

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
```

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
```
