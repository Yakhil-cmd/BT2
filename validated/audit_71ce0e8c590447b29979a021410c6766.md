### Title
`tick()` Does Not Refund ETH When No Minting Is Required — (`File: L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in the L1 `RewardSupplier` contract is `payable` and accepts ETH to cover L1→L2 messaging fees. However, when `requiredMinting()` returns zero (i.e., there are no pending mint requests to consume), the function silently exits without using or refunding `msg.value`. Any ETH sent in this case is permanently locked in the contract with no recovery path.

---

### Finding Description

`tick()` is declared `external payable` and splits `msg.value` between two cross-layer calls only inside the `if (amountToMint > 0)` branch:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // consumes half

        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // consumes rest
    }
    // ← if amountToMint == 0, msg.value is never used and never returned
}
``` [1](#0-0) 

When `amountToMint == 0`, the function body is a no-op. There is no `require(amountToMint > 0)` guard to revert and return the ETH, no explicit refund, and no `receive()` / withdrawal function visible in the contract to recover stranded ETH.

The race condition that triggers this:

1. Keeper A calls `requiredMinting()` off-chain and observes `amountToMint > 0`.
2. Keeper A submits `tick{value: X}()`.
3. Keeper B's `tick()` transaction is mined first, consuming all pending L2→L1 mint-request messages.
4. Keeper A's transaction executes: `requiredMinting()` now returns `(0, 0)`, the `if` branch is skipped, and `X` ETH is permanently locked in the contract.

This is structurally identical to the SpeedBumpPriceGate bug: a caller sends ETH to guarantee a state-dependent operation succeeds, the state changes before execution, the operation is skipped, and the ETH is not returned.

---

### Impact Explanation

ETH sent by the caller is permanently locked in the `RewardSupplier` L1 contract. There is no withdrawal function or `receive()` fallback that would allow recovery. The loss is permanent and proportional to the fee the keeper sent to cover messaging costs. Under competitive keeper conditions (multiple bots racing to call `tick()`), this can happen repeatedly.

**Impact category:** High — Permanent freezing of funds (caller's ETH is irrecoverably locked).

---

### Likelihood Explanation

The `tick()` function has no access control; any address can call it. In a live system, multiple keeper bots are expected to monitor and call `tick()` when mint requests accumulate. A race between two keepers is a realistic, foreseeable event. The probability increases as the system matures and more independent operators run keeper infrastructure.

---

### Recommendation

Add one of the following mitigations:

**Option A — Revert when there is nothing to do:**
```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    require(amountToMint > 0, "NOTHING_TO_MINT");
    // ... rest of logic
}
```
This causes the losing keeper's transaction to revert, returning their ETH automatically.

**Option B — Refund unused ETH:**
```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        if (msg.value > 0) {
            (bool ok,) = msg.sender.call{value: msg.value}("");
            require(ok, "ETH_REFUND_FAILED");
        }
        return;
    }
    // ... rest of logic
}
```

### Proof of Concept

1. Observe `requiredMinting()` returns `(N, M)` with `N > 0`.
2. Submit `tick{value: 1 ether}()` from address `A`.
3. Before `A`'s tx is mined, address `B` calls `tick{value: 0.001 ether}()` and it is mined first, consuming all messages.
4. `A`'s tx executes: `requiredMinting()` returns `(0, 0)`, the `if` block is skipped, `1 ether` remains in the contract.
5. Verify: `address(RewardSupplier).balance` increased by `1 ether` with no corresponding outflow. [1](#0-0)

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)

            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
