Audit Report

## Title
Unbounded `ingress_expiries` Vec allocation via uncapped `ingress_end` causes OOM DoS in `construction_payloads` — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function accepts attacker-controlled `ingress_start` and `ingress_end` values from the HTTP request metadata with no upper-bound validation on `ingress_end`. Submitting `ingress_start=0, ingress_end=u64::MAX` bypasses both existing guards and causes the while loop at line 163 to run for an astronomically large number of iterations, pushing one `u64` per iteration into an unbounded `Vec`, exhausting process memory and crashing the ICRC1 Rosetta node. In release mode (wrapping arithmetic), the loop is effectively infinite due to `ingress_start` wrapping past `u64::MAX`.

## Finding Description

**Constants confirmed from source:**
- `MAX_INGRESS_TTL = 300s`, `PERMITTED_DRIFT = 60s` → `ingress_interval = 240s = 240_000_000_000 ns`
- `INGRESS_INTERVAL_OVERLAP = 120s = 120_000_000_000 ns`
- Loop step: `ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64) = 120_000_000_000 ns`

**Guard 1 (line 148):** `if ingress_start >= ingress_end` → `0 >= u64::MAX` → `FALSE` → no error.

**Guard 2 (line 154):** `if ingress_end < now + ingress_interval`. With `now ≈ 1.75×10¹⁸ ns` (year 2026) and `ingress_interval ≈ 2.4×10¹¹ ns`, the sum is `≈1.75×10¹⁸`, far below `u64::MAX ≈ 1.84×10¹⁹`. Condition is `FALSE` → no error.

**The loop (lines 162–167):**
```rust
let mut ingress_expiries = vec![];
while ingress_start < ingress_end {
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start +=
        ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
```
With `ingress_start=0` and `ingress_end=u64::MAX`, each iteration increments `ingress_start` by `120_000_000_000`. After `≈153,722,867` iterations, `ingress_start` overflows `u64::MAX`. In **release mode** (Rust wrapping arithmetic), `ingress_start` wraps to a small value and the loop condition `ingress_start < u64::MAX` becomes `TRUE` again, creating an **infinite loop** with continuous heap allocation. In debug mode, the plain addition `ingress_start += ...` panics on overflow after ~153M iterations. Either way, the process is killed by OOM or panic before returning.

The `ingress_end` field is a plain `Option<u64>` with no range constraint at the type level. The HTTP endpoint at `/construction/payloads` (registered in `main.rs` line 380) passes the deserialized metadata directly to `services::construction_payloads` with no additional validation layer.

## Impact Explanation

A single unauthenticated HTTP POST to `/construction/payloads` with `ingress_end=u64::MAX` causes the Rosetta process to either loop infinitely with continuous heap allocation (release mode) or panic after ~153M iterations (debug mode). In both cases the process crashes and does not recover without a restart. This is a complete denial-of-service against the ICRC1 Rosetta node, matching the **High** impact category: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation

The `/construction/payloads` endpoint is a standard public Rosetta API endpoint with no authentication or rate limiting visible in the router configuration. The attack requires a single JSON POST with two integer fields set to extreme values. Any network-reachable client can trigger it. The attack is trivially repeatable after each restart.

## Recommendation

Add an explicit upper-bound check on `ingress_end` before the while loop, capping the allowed window to a reasonable maximum (e.g., 24 hours):

```rust
const MAX_INGRESS_WINDOW_NS: u64 = 24 * 60 * 60 * 1_000_000_000u64; // 24h in ns
if ingress_end > now + MAX_INGRESS_WINDOW_NS {
    return Err(Error::processing_construction_failed(&format!(
        "ingress_end {ingress_end} exceeds maximum allowed window"
    )));
}
```

Alternatively, cap the number of loop iterations directly:

```rust
const MAX_INGRESS_EXPIRIES: usize = 1000;
while ingress_start < ingress_end {
    if ingress_expiries.len() >= MAX_INGRESS_EXPIRIES {
        return Err(Error::processing_construction_failed("Too many ingress intervals"));
    }
    ingress_expiries.push(ingress_start + ingress_interval);
    ingress_start += ingress_interval.saturating_sub(INGRESS_INTERVAL_OVERLAP.as_nanos() as u64);
}
```

Also replace the plain addition `ingress_start + ingress_interval` with `ingress_start.saturating_add(ingress_interval)` to prevent overflow-induced infinite loops in release mode.

## Proof of Concept

```rust
#[test]
fn test_construction_payloads_oom_dos() {
    use std::time::{Duration, SystemTime, UNIX_EPOCH};
    use candid::Principal;
    use rosetta_core::objects::{PublicKey, CurveType};

    let now = UNIX_EPOCH + Duration::from_secs(1_750_000_000);
    // Provide a minimal valid public key (Ed25519, 32 zero bytes)
    let pk = PublicKey {
        hex_bytes: "0".repeat(64),
        curve_type: CurveType::Edwards25519,
    };
    // This call will either OOM-crash the process (release) or panic on overflow (debug)
    // before returning Err — demonstrating the missing upper-bound guard.
    let result = construction_payloads(
        vec![/* valid transfer operations */],
        Some(ConstructionPayloadsRequestMetadata {
            ingress_start: Some(0),
            ingress_end: Some(u64::MAX),
            created_at_time: None,
            memo: None,
        }),
        &Principal::anonymous(),
        vec![pk],
        now,
    );
    assert!(result.is_err()); // never reached
}
```
