### Title
Missing Service Event Types in Cadence-to-Go Conversion Dispatch Table Causes Silent Discard of `EjectNode` and `SetEpochExtensionViewCount` Events - (File: `model/convert/service_event.go`)

### Summary
The Flow protocol defines seven service event types, but the Cadence-to-Go conversion dispatch table (`convert.ServiceEvent`) and the system-contracts registry (`ServiceEvents` struct) only cover five of them. The two newest types — `ServiceEventEjectNode` and `ServiceEventSetEpochExtensionViewCount` — are absent from both the registry and the conversion switch. As a result, when the on-chain Cadence contracts emit either of these events, the execution layer silently discards them: they are never converted, never included in execution receipts, and never reach the protocol state machine. The state-machine handlers that were written to process them are therefore unreachable dead code.

### Finding Description

**Seven types are defined; only five are registered.**

`model/flow/service_event.go` declares all seven canonical service-event type constants:

```
ServiceEventSetup
ServiceEventCommit
ServiceEventRecover
ServiceEventVersionBeacon
ServiceEventProtocolStateVersionUpgrade
ServiceEventSetEpochExtensionViewCount   ← missing from registry
ServiceEventEjectNode                    ← missing from registry
``` [1](#0-0) 

**The system-contracts registry omits the two newest types.**

`fvm/systemcontracts/system_contracts.go` builds the `ServiceEvents` struct that the execution layer uses to recognise which Cadence events are service events. It populates only five fields:

```go
events := &ServiceEvents{
    EpochSetup:                  event(ContractNameEpoch, EventNameEpochSetup),
    EpochCommit:                 event(ContractNameEpoch, EventNameEpochCommit),
    EpochRecover:                event(ContractNameEpoch, EventNameEpochRecover),
    VersionBeacon:               event(ContractNameNodeVersionBeacon, EventNameVersionBeacon),
    ProtocolStateVersionUpgrade: event(ContractNameNodeVersionBeacon, EventNameProtocolStateVersionUpgrade),
}
``` [2](#0-1) 

There are no `EventNameEjectNode` or `EventNameSetEpochExtensionViewCount` constants defined, and no corresponding fields in the struct. [3](#0-2) 

**The conversion function falls through to an error for both missing types.**

`convert.ServiceEvent` is the Cadence-to-Go dispatch table. It switches on the five registered event-type strings; any other type hits the `default` branch and returns an error, so the event is dropped:

```go
switch event.Type {
case events.EpochSetup.EventType():          return convertServiceEventEpochSetup(event)
case events.EpochCommit.EventType():         return convertServiceEventEpochCommit(event)
case events.EpochRecover.EventType():        return convertServiceEventEpochRecover(event)
case events.VersionBeacon.EventType():       return convertServiceEventVersionBeacon(event)
case events.ProtocolStateVersionUpgrade.EventType(): return convertServiceEventProtocolStateVersionUpgrade(event)
default:
    return nil, fmt.Errorf("invalid event type: %s", event.Type)
}
``` [4](#0-3) 

**The protocol state machine handlers for the missing types are dead code.**

`SetValueStateMachine.EvolveState` was written to consume `ServiceEventSetEpochExtensionViewCount`: [5](#0-4) 

`EpochStateMachine.evolveActiveStateMachine` was written to consume `ServiceEventEjectNode`: [6](#0-5) 

Neither handler is ever reached because the events are discarded before they can be included in an execution receipt.

**The Go types themselves document the intended on-chain origin.**

`EjectNode` carries the comment *"service event emitted when a node has to be ejected from the network"*: [7](#0-6) 

`SetEpochExtensionViewCount` carries the comment *"service event emitted by the FlowServiceAccount for updating the `EpochExtensionViewCount` parameter"*: [8](#0-7) 

Both are clearly intended to be live, on-chain service events, not future placeholders.

### Impact Explanation

**`EjectNode` silently dropped:** Any on-chain governance transaction that emits an `EjectNode` event to remove a misbehaving node from the identity table has no effect. The ejected node's `EpochParticipationStatus` is never updated; it continues to be treated as an active, valid participant in consensus, collection, execution, and verification. A node that has been formally ejected by governance retains full protocol participation rights indefinitely.

**`SetEpochExtensionViewCount` silently dropped:** The `EpochExtensionViewCount` KV-store parameter, which governs how long an epoch extension lasts during fallback mode, can never be updated at runtime. The governance mechanism for adjusting this safety-critical parameter is permanently broken.

Both impacts are analogous to the reported Solidity bug: a lookup table that is too short for the full enumeration silently returns a zero/default result for the highest-tier entries, making the corresponding feature inoperative.

### Likelihood Explanation

The Cadence contracts (`FlowEpoch`, `FlowServiceAccount`) are the authoritative source of these events. The Go-side state machines were written specifically to consume them, confirming that the contracts are expected to emit them. Any governance action that triggers either event type will silently fail with no on-chain or off-chain indication of the failure, making the bug hard to detect in production.
