### Title
Unchecked Exit Precompile Call Return Value Causes Silent Token Burn Without Bridge-Out - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear()` and `withdrawToEthereum()` by first burning the caller's ERC-20 tokens via `_burn()`, then invoking the Aurora exit precompile via a raw assembly `call()`. The return value of that `call()` — which signals success or failure — is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, the transaction does not revert: the tokens are permanently destroyed with no corresponding bridge-out, resulting in irreversible loss of user funds.

### Finding Description

In `EvmErc20.sol`, both exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here — irreversible
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no require(res != 0)
    }
}
``` [1](#0-0) 

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {

The identical pattern is present in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The EVM `call()` opcode returns `0` on failure and `1` on success. In Solidity assembly, a failed sub-call does **not** automatically revert the caller — the caller must explicitly check `res` and revert if it is `0`. Neither contract does this.

The exit precompile (`ExitToNear` at `0xe9217bc7...`, `ExitToEthereum` at `0xb0bd02f6...`) can fail for several reasons visible in the Rust precompile implementation:

- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — malformed NEAR account ID in `recipient`
- `ERR_INVALID_AMOUNT` — amount exceeds `u128::MAX`
- `ERR_TARGET_TOKEN_NOT_FOUND` — ERC-20 not registered in the NEP-141 map
- `OutOfGas` — insufficient gas forwarded to the precompile [5](#0-4) 

When any of these conditions occur, the precompile returns an error exit reason, the assembly `call()` returns `0`, but because `res` is never checked, the outer transaction **succeeds**. The `_burn()` that already executed is not rolled back.

### Impact Explanation

**Critical — Permanent freezing of funds.**

The burn is irreversible. Once `_burn()` executes and the precompile call silently fails, the user's ERC-20 tokens are gone from Aurora's EVM state and no corresponding NEP-141 or ETH transfer is initiated on the NEAR/Ethereum side. The funds are permanently destroyed with no recovery path.

### Likelihood Explanation

**Medium.** Every user of a bridged ERC-20 token who calls `withdrawToNear()` or `withdrawToEthereum()` is exposed. Failure conditions are reachable without any privileged access:

- Passing a syntactically invalid NEAR account ID as `recipient` (e.g., an empty byte string, or a string with invalid characters) triggers `ERR_INVALID_RECEIVER_ACCOUNT_ID` in the precompile.
- Passing an amount exceeding `u128::MAX` triggers `ERR_INVALID_AMOUNT`.
- Calling with insufficient gas causes `OutOfGas` inside the precompile.

Any of these causes a silent burn. A user making a mistake with the recipient encoding loses their tokens permanently.

### Recommendation

After the assembly `call()`, check `res` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to all four assembly blocks across both `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures that if the exit precompile fails for any reason, the entire transaction reverts, the `_burn()` is rolled back, and the user's tokens are preserved.

### Proof of Concept

1. Deploy a bridged ERC-20 (`EvmErc20`) on Aurora for some NEP-141 token.
2. Bridge tokens to Aurora so a user holds a non-zero ERC-20 balance.
3. Call `withdrawToNear(bytes(""), amount)` — passing an empty byte string as `recipient`.
4. Inside the precompile, `parse_recipient` fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID` because an empty string is not a valid NEAR account ID.
5. The precompile returns an error; the assembly `call()` returns `0`.
6. Because `res` is never checked, the transaction succeeds.
7. The user's ERC-20 balance is now zero (burned), no NEP-141 transfer was initiated, and the tokens are permanently lost. [6](#0-5) [1](#0-0)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
