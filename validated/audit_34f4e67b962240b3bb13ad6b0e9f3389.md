### Title
First Pool Created by Factory Is Not Recognized as a Valid Pool by `isPool` — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.isPool` uses `poolToIdx[pool] != 0` as its membership test. Because Solidity initializes `nextPoolIdx` to `0` and the first `createPool` call assigns `poolToIdx[firstPool] = 0`, `isPool(firstPool)` returns `false` for the very first pool ever deployed by the factory. Any downstream consumer that gates on `isPool` — most critically the oracle's `register` function — will reject the first pool, permanently preventing it from registering with the protected oracle and making all swaps revert if the pool was wired to a `ProtectedPriceProvider`.

### Finding Description
In `MetricOmmPoolFactory.createPool`, pool indices are assigned from `nextPoolIdx`, which is a `uint256` state variable that Solidity initialises to `0`:

```solidity
uint256 poolIdx = nextPoolIdx;   // == 0 for the first pool
nextPoolIdx++;
idxToPool[poolIdx] = pool;
poolToIdx[pool] = poolIdx;       // poolToIdx[firstPool] = 0
```

The membership predicate is:

```solidity
function isPool(address pool) external view override returns (bool) {
    return poolToIdx[pool] != 0;   // 0 != 0 → false for the first pool
}
```

`poolToIdx` maps every address that was never registered to `0` (the default). The first legitimate pool is also stored at `0`, so it is indistinguishable from an unregistered address. Every subsequent pool (index ≥ 1) is correctly identified.

The oracle's `register` path calls `IPoolFactory(factory).isPool(pool)` before whitelisting a pool for a feed. Because `isPool` returns `false` for the first pool, `register` reverts, and the pool can never be added to `registeredPool[feedId][pool]`. When the pool later calls `oracle.price(feedId, pool)` through its `ProtectedPriceProvider`, the oracle requires `registeredPool[feedId][pool]` and reverts, causing every swap on that pool to revert.

### Impact Explanation
The first pool deployed by the factory is permanently unable to register with the protected oracle. If it was created with a `ProtectedPriceProvider` or `ProtectedPriceProviderL2`, every call to `swap` will revert at the oracle read, making the pool completely unusable for trading. LP funds deposited into the pool are locked (no swaps, no arbitrage to rebalance), and the pool cannot be fixed post-deployment because the registry entry is immutable.

### Likelihood Explanation
The bug is deterministic and affects exactly the first pool created by any fresh factory deployment. `createPool` is permissionless, so any caller can trigger it. The pool admin has no remediation path: the factory registry entry cannot be corrected, and the oracle registration cannot be forced through because `isPool` will always return `false` for that pool.

### Recommendation
Initialise `nextPoolIdx` to `1` in the factory constructor so that the first assigned index is `1` and `poolToIdx[pool] != 0` correctly identifies all factory-created pools:

```solidity
constructor(...) {
    nextPoolIdx = 1;   // reserve 0 as the "not registered" sentinel
    ...
}
```

Alternatively, change the `isPool` predicate to use an explicit boolean mapping (`isRegisteredPool[pool]`) that is set to `true` in `createPool`, removing the dependency on index value.

### Proof of Concept
```solidity
// Foundry test
function test_isPool_firstPoolReturnsFalse() public {
    address pool = factory.createPool(_defaultPoolParams());
    // First pool gets poolToIdx == 0, so isPool returns false
    assertFalse(factory.isPool(pool), "first pool incorrectly not recognized");
    // All subsequent pools are fine
    address pool2 = factory.createPool(_defaultPoolParams2());
    assertTrue(factory.isPool(pool2));
}
```
