### Title
Free Share Minting in Completely-Drained Bins Dilutes Existing LP Positions — (`metric-core/contracts/libraries/LiquidityLib.sol`)

### Summary
When a pool bin holds existing shares but both token balances have been drained to zero by swaps, `LiquidityLib.addLiquidity` allows any caller to mint an arbitrary number of shares for zero token cost. This is the direct Metric OMM analog of the wxETH inflation attack: instead of a first depositor inflating the exchange rate to steal from subsequent depositors, an attacker exploits a drained bin to dilute existing LPs' claims on future tokens.

### Finding Description

In `LiquidityLib.addLiquidity`, when `binTotalSharesVal > 0` (existing shares are present), the required token contribution is computed as:

```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
``` [1](#0-0) 

If `binState.token0BalanceScaled == 0` and `binState.token1BalanceScaled == 0` (bin completely drained by swaps), both `amount0Scaled` and `amount1Scaled` evaluate to `Math.ceilDiv(0, binTotalSharesVal) = 0`. The shares are unconditionally minted into the position and the bin total:

```solidity
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
``` [2](#0-1) 

After the loop, `totalToken0ToAddScaled = 0` and `totalToken1ToAddScaled = 0`, so `_deltasScaledToExternal` returns `(0, 0)`. The callback guard:

```solidity
if (amount0Added > 0 || amount1Added > 0) {
    IMetricOmmModifyLiquidityCallback(msg.sender)
        .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
``` [3](#0-2) 

…is never entered. No tokens are transferred, yet the attacker's shares are permanently recorded in pool storage.

A bin reaches this state naturally: bins above the current price hold only token0; a large swap that buys all token0 leaves `token0BalanceScaled = 0` while the bin's share accounting remains intact. The same applies symmetrically to token1-only bins below the current price.

### Impact Explanation

An attacker who observes a bin being completely drained can immediately call `addLiquidity` with an arbitrarily large `sharesToAdd` (subject only to the 1,000-share minimum enforced by `minimalMintableLiquidity`) and pay zero tokens. When the price later moves back through the bin and refills it with tokens via subsequent swaps, the attacker captures:

```
attackerShares / (existingShares + attackerShares)
```

of those tokens. Existing LPs — who paid real tokens to acquire their shares — lose a proportional fraction of the tokens they would otherwise have received. The attacker's cost is gas only. [4](#0-3) 

### Likelihood Explanation

Bins being completely drained is a realistic scenario in any active pool, particularly for bins adjacent to the current price during directional price moves. The attack requires no special role — `addLiquidity` is permissionless and callable by any address directly on the pool. The attacker only needs to monitor on-chain state for a drained bin and submit a transaction before the price reverses. [5](#0-4) 

### Recommendation

In `LiquidityLib.addLiquidity`, after computing `amount0Scaled` and `amount1Scaled` for a non-empty bin (`binTotalSharesVal > 0`), require that at least one is non-zero before minting shares:

```diff
} else {
    amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
    amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
+   if (amount0Scaled == 0 && amount1Scaled == 0) {
+       revert IMetricOmmPoolActions.ZeroTokenContribution();
+   }
}
```

This mirrors the fix applied to `wxETH::stake` (checking `mintAmount > 0` before accepting the deposit). It ensures that shares can only be minted when the caller contributes at least one scaled unit of token value to the bin.

### Proof of Concept

1. Pool is deployed with bins `[-5, 4]`. Alice calls `addLiquidity` for bin 4 (above current price) with `sharesToAdd = 10,000`, paying token0. State: `token0BalanceScaled = X`, `binTotalShares[4] = 10,000`.

2. A large swap buys all token0 from bin 4. State: `token0BalanceScaled = 0`, `binTotalShares[4] = 10,000` (shares unchanged).

3. Bob calls `addLiquidity` for bin 4 with `sharesToAdd = 1,000,000`:
   - `amount0Scaled = Math.ceilDiv(0 * 1,000,000, 10,000) = 0`
   - `amount1Scaled = Math.ceilDiv(0 * 1,000,000, 10,000) = 0`
   - Callback not triggered; Bob pays 0 tokens.
   - State: `binTotalShares[4] = 1,010,000`, Bob owns 1,000,000 shares.

4. Price reverses; a swap refills bin 4 with token1. Bob now owns `1,000,000 / 1,010,000 ≈ 99%` of the bin's tokens. Alice, who paid real token0, owns only `10,000 / 1,010,000 ≈ 1%`. [6](#0-5)
