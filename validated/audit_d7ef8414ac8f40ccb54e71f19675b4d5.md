### Title
Lack of Consensus on `app_pk` in CKD Protocol Allows Malicious Participant to Corrupt Derived Secret Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function accepts `app_pk` (the application's ElGamal public key) as a caller-supplied parameter with no cross-participant consensus or binding. Each MPC node independently provides its own `app_pk` value. A malicious participant can substitute a different `app_pk`, causing the coordinator to aggregate shares encrypted to inconsistent keys. The resulting `CKDOutput` is silently accepted as valid but cannot be correctly unmasked by the application, permanently destroying the derived secret.

---

### Finding Description

The CKD protocol derives a confidential secret for a TEE application by having each participant compute an ElGamal-encrypted share and send it to the coordinator for aggregation.

In `compute_signature_share`, each participant computes:

```
C_i = S_i + y_i * app_pk
```

where `app_pk` is the application's ephemeral public key `A = a * G₁`. [1](#0-0) 

The coordinator then aggregates all received shares by simple summation: [2](#0-1) 

**The critical flaw**: `app_pk` is passed as an independent parameter to each participant's `ckd()` call and is never broadcast, echo-broadcast, or otherwise consensus-verified across participants. [3](#0-2) 

There is no step in `do_ckd_coordinator` or `do_ckd_participant` that checks whether all participants used the same `app_pk`. The coordinator receives `(norm_big_y, norm_big_c)` tuples and sums them blindly. [4](#0-3) 

If a malicious participant `m` uses `app_pk_m ≠ app_pk`, the aggregated ciphertext becomes:

```
C_total = Σ λ_i * (S_i + y_i * app_pk)   [honest]
        + λ_m * (S_m + y_m * app_pk_m)    [malicious]
```

When the application unmasks with its private key `a` (where `app_pk = a * G₁`):

```
C_total - a * Y_total = MSK * H(pk, app_id) + λ_m * y_m * (app_pk_m - app_pk)
```

The residual term `λ_m * y_m * (app_pk_m - app_pk)` is a non-zero random G₁ point, making the derived secret permanently unrecoverable. No error is returned; the coordinator outputs a structurally valid but cryptographically corrupted `CKDOutput`.

This is directly analogous to the reported bridge vulnerability: just as `msg.sender` is assumed to be the correct destination address on the target chain without explicit user specification, `app_pk` is assumed to be consistent across all participants without any protocol-level enforcement.

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an unusable cryptographic output.**

The coordinator and all honest participants complete the protocol without error. The `CKDOutput` is structurally well-formed and passes all existing checks. However, the TEE application receives a derived secret that is permanently inaccessible — the confidential key is irreversibly corrupted. The application's secret derivation fails silently.

---

### Likelihood Explanation

Any single malicious MPC participant can trigger this. The attacker only needs to call `ckd()` with a self-chosen `app_pk` (e.g., a key they control or a random point). No special privileges, leaked keys, or external assumptions are required. The attack is deterministic and requires no cryptographic breaks. In a deployment where even one of N nodes is adversarial, this attack is trivially executable on any CKD request.

---

### Recommendation

Add `app_pk` to the echo-broadcast round so all participants reach consensus on a single value before computing their shares. Concretely:

1. Include `app_pk` in the existing broadcast phase (alongside `commitment` and `proof_of_knowledge`), or add a dedicated broadcast round at the start of `run_ckd_protocol`.
2. Each participant verifies that the `app_pk` they received via broadcast matches the one they were given locally; abort if they differ.
3. Alternatively, bind `app_pk` into the `AppId` hash so that a mismatched `app_pk` produces a different hash point, making the shares cryptographically incompatible and detectable.

---

### Proof of Concept

1. Honest participants P₁, P₂ call `ckd(participants, coordinator, me, key_pair, app_id, app_pk_real, rng)` with the correct application public key `app_pk_real = a * G₁`.
2. Malicious participant P₃ calls `ckd(participants, coordinator, me, key_pair, app_id, app_pk_evil, rng)` where `app_pk_evil` is an arbitrary G₁ point.
3. P₁ and P₂ send `(λ₁·Y₁, λ₁·C₁)` and `(λ₂·Y₂, λ₂·C₂)` to the coordinator, with `C_i = S_i + y_i * app_pk_real`.
4. P₃ sends `(λ₃·Y₃, λ₃·C₃)` with `C₃ = S₃ + y₃ * app_pk_evil`.
5. The coordinator sums all shares and returns `CKDOutput { big_y, big_c }` — no error.
6. The application calls `ckd_output.unmask(a)` and obtains `MSK * H(pk, app_id) + λ₃ * y₃ * (app_pk_evil - app_pk_real)`, which is a random, unusable G₁ point rather than the intended derived secret. [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
