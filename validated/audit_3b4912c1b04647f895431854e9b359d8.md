### Title
Permanent Fund Loss on Failed `ExitToNear` Promise Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns a user's ERC-20 tokens (or deducts ETH) from their EVM balance before scheduling an asynchronous `ft_transfer` promise on the NEP-141 contract. When the `error_refund` Cargo feature is not compiled in — which is the default — no refund callback is registered. If the `ft_transfer` promise fails (e.g., the recipient is not registered with the NEP-141 contract), the burned tokens are permanently unrecoverable. There is no mechanism to recover them.

---

### Finding Description

**Root cause — `engine-precompiles/src/native.rs`:**

When the `ExitToNear` precompile runs, it constructs `ExitToNearPrecompileCallbackArgs` with the `refund` field conditionally compiled: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

Without `error_refund`, `refund` is always `None`. The promise dispatch logic then checks whether a callback is even needed: [2](#0-1) 

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback at all
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

For a standard ERC-20 exit (non-wNEAR), both `refund` and `transfer_near` are `None`, so `callback_args == default()` and **no callback is scheduled**. If `ft_transfer` fails, there is no handler to execute.

**Callback handler — `engine/src/contract_methods/connector.rs`:**

Even when a callback is scheduled (e.g., wNEAR unwrap path where `transfer_near` is `Some`), the failure branch does nothing without `error_refund`: [3](#0-2) 

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op; tokens are gone
};
```

**Feature flag is not a default — `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`:** [4](#0-3) 

```toml
[features]
default = ["std"]
...
error_refund = []
``` [5](#0-4) 

```toml
[features]
default = ["std"]
contract = ["log", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
```

`error_refund` is absent from both `default` and `contract` (the production WASM build feature). The production binary is therefore compiled without it.

**The fund-loss path is explicitly acknowledged in the test suite:** [6](#0-5) 

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

```rust
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

The codebase itself documents that without `error_refund`, tokens are destroyed with no recovery path.

---

### Impact Explanation

Any EVM user who calls the `ExitToNear` precompile (directly or via an ERC-20 `withdraw`/`burn` function) with a NEAR recipient account that is not registered with the target NEP-141 contract will have their ERC-20 tokens (or ETH) permanently destroyed. The EVM-side burn is committed atomically within the `submit` call; the NEAR-side `ft_transfer` is asynchronous and can fail independently. With no refund callback in the production build, the discrepancy is never corrected. This constitutes **permanent freezing/destruction of user funds**.

---

### Likelihood Explanation

Medium-to-high. The failure condition — calling `ft_transfer` on a NEP-141 contract for an account that has not called `storage_deposit` — is a routine user mistake on NEAR. Any Aurora EVM user bridging tokens to NEAR for the first time, or specifying a fresh NEAR account as the recipient, can trigger this. No special privileges or adversarial setup are required; the user only needs to submit a normal EVM transaction.

---

### Recommendation

Enable `error_refund` as part of the `contract` (production) feature set, or make it a default feature. Alternatively, unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` and always schedule the callback, so that a failed `ft_transfer` always triggers a re-mint of the burned tokens to the original sender's EVM address.

---

### Proof of Concept

1. Alice holds 100 units of `erc20_token` on Aurora (backed by `nep141.near`).
2. Alice calls `erc20_token.withdraw(100, "alice_new.near")`, which internally calls the `ExitToNear` precompile.
3. The precompile burns Alice's 100 ERC-20 tokens atomically within the EVM execution.
4. A `ft_transfer` promise is scheduled on `nep141.near` targeting `alice_new.near`.
5. `alice_new.near` has never called `storage_deposit` on `nep141.near`; the `ft_transfer` panics and fails.
6. Because `error_refund` is not compiled in, `callback_args == default()`, so **no callback was scheduled**.
7. Alice's 100 ERC-20 tokens are gone from the EVM. `nep141.near` still holds the backing NEP-141 tokens on Aurora's behalf. Alice has no recourse — the funds are permanently frozen in the connector with no withdrawal path for her.
