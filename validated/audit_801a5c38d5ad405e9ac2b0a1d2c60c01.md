### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` — the production ERC-20 contracts deployed for every bridged NEP-141 token on Aurora — burn the caller's tokens **before** calling the `ExitToNear` / `ExitToEthereum` precompile via inline assembly. The return value `res` of the `call()` opcode is captured but **never checked**. If the precompile call fails (returns `0`), the function does not revert, the tokens are permanently destroyed on the EVM side, and no corresponding NEP-141 transfer is ever scheduled on NEAR.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
// EvmErc20.sol lines 53–62
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}
```

The identical pattern exists in `withdrawToEthereum` (lines 65–76) and in both functions of `EvmErc20V2.sol` (lines 53–77). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` (`ExitToNear`) can return failure (`res = 0`) under several conditions visible in the Rust precompile implementation:

- `get_nep141_from_erc20` fails (ERC-20 address not registered in engine storage)
- `exit_erc20_token_to_near` returns an `ExitError` (e.g., `ERR_ETH_ATTACHED_FOR_ERC20_EXIT`, invalid recipient)
- The precompile's required gas exceeds the remaining gas budget (`ExitError::OutOfGas`) [5](#0-4) [6](#0-5) 

When any of these conditions occur, the EVM `call()` opcode returns `0`. Because `res` is never tested and no `revert` is issued, the outer transaction succeeds. The `_burn` that already executed is not rolled back. The promise log that would schedule the NEAR-side `ft_transfer` is never emitted. The user's tokens are gone with no recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `withdrawToNear` or `withdrawToEthereum` on a bridged ERC-20 token while the precompile call fails will have their tokens permanently destroyed. There is no refund mechanism at the EVM level for this failure mode (the `error_refund` feature only handles NEAR-promise-level failures, not EVM-level precompile call failures). The burned tokens cannot be re-minted because the mint function is `onlyAdmin`. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The precompile succeeds under normal operation. However, the failure path is reachable by any unprivileged token holder in the following realistic scenarios:

1. **Gas exhaustion**: A transaction that calls `withdrawToNear` with a gas limit just above the intrinsic cost but below `EXIT_TO_NEAR_GAS` will cause the precompile to return `ExitError::OutOfGas`. The `call(gas(), ...)` forwards remaining gas; if it is insufficient, `res = 0` and tokens are silently burned.
2. **Unregistered token**: If an ERC-20 contract is deployed but its NEP-141 mapping is not yet stored in engine state, `get_nep141_from_erc20` returns an error, the precompile fails, and `res = 0`.
3. **Engine paused / state corruption**: Any condition that causes the precompile to return an error propagates silently. [8](#0-7) 

---

### Recommendation

Check the return value of the assembly `call()` and revert if it is zero. Apply this fix to all four affected functions in both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile fails, the entire transaction (including the `_burn`) is reverted, preserving the user's token balance.

---

### Proof of Concept

1. Deploy a bridged ERC-20 token (`EvmErc20`) on Aurora with a NEP-141 mapping.
2. Mint tokens to address `A`.
3. Construct a transaction from `A` calling `withdrawToNear(recipient, amount)` with a gas limit set just below `EXIT_TO_NEAR_GAS` (the precompile's required gas), so the precompile call runs out of gas and returns `0`.
4. Observe: the transaction succeeds (EVM status = success), `A`'s ERC-20 balance is zero, but no `ft_transfer` promise was scheduled on NEAR — the tokens are permanently lost. [1](#0-0) [9](#0-8)
