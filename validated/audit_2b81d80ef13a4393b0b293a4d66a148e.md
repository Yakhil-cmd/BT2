### Title
Missing Access Control on `update_rewards()` Allows Any Caller to Force Reward Distribution — (File: src/staking/staking.cairo)

---

### Summary
The `update_rewards` function in the Staking contract has no caller validation. The specification explicitly restricts this function to "Only starkware sequencer," but the implementation performs no such check. Any unprivileged address can call it, forcing reward distribution for a given block and permanently blocking the legitimate sequencer from calling it for that same block.

---

### Finding Description
`IStakingRewardsManager::update_rewards` is the consensus-era reward distribution entry point. It accepts a `staker_address` and a `disable_rewards: bool` flag. When `disable_rewards` is `false`, it calculates and distributes block rewards to the staker and their pools. When `true`, it only advances `last_reward_block` without distributing anything.

The specification at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1449–1500 contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ← no get_caller_address() check here
``` [1](#0-0) 

The spec entry confirming the intended restriction: [2](#0-1) 

`last_reward_block` is a single contract-wide value, not per-staker: [3](#0-2) 

Once any caller sets it to the current block, the sequencer's call for that block reverts with `REWARDS_ALREADY_UPDATED`.

---

### Impact Explanation
**High — Theft of unclaimed yield.**

The sequencer uses `disable_rewards: true` to mark a block as processed without distributing rewards (e.g., for blocks where a staker should not earn, during transition periods, or when the sequencer deliberately withholds distribution). An attacker who front-runs with `disable_rewards: false` forces reward distribution for that block. Because `last_reward_block` is now equal to the current block, the sequencer's subsequent call fails with `REWARDS_ALREADY_UPDATED`, and the block cannot be re-processed. Rewards that should not have been distributed are now credited to the staker's `unclaimed_rewards_own` and transferred to pools — constituting theft of unclaimed yield from the protocol's reward budget.

Additionally, the attacker can choose *which* staker receives rewards for a given block, bypassing the sequencer's intended ordering and staker selection.

---

### Likelihood Explanation
**High.** The function is publicly callable with no role check, no stored sequencer address, and no protocol-level restriction. Any EOA or contract can call it. The only constraint is the one-call-per-block limit (`REWARDS_ALREADY_UPDATED`), which the attacker satisfies by simply calling first.

---

### Recommendation
Store the authorized sequencer address in contract storage and assert it at the top of `update_rewards`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
```

Alternatively, use the existing roles framework (e.g., an `OPERATOR` or dedicated `SEQUENCER` role) consistent with how `update_rewards_from_attestation_contract` restricts its caller: [4](#0-3) 

---

### Proof of Concept

1. Sequencer intends to call `update_rewards(staker_A, disable_rewards: true)` at block N to mark the block without distributing rewards (e.g., staker_A missed attestation).
2. Attacker observes the pending transaction in the mempool and submits `update_rewards(staker_A, disable_rewards: false)` with higher priority.
3. Attacker's call executes first:
   - `last_reward_block` is written to block N.
   - Block rewards are calculated and credited to `staker_A.unclaimed_rewards_own`.
   - Pool rewards are transferred to staker_A's pools via `update_pool_rewards`.
4. Sequencer's call executes next and reverts: `assert!(current_block_number > self.last_reward_block.read())` fails with `REWARDS_ALREADY_UPDATED`.
5. staker_A has received rewards for a block where the sequencer intended to withhold them. The block cannot be re-processed. [5](#0-4)
