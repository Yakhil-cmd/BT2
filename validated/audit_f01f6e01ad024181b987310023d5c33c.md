### Title
Zero `priceProviderTimelock` Bypasses Oracle-Change Delay, Enabling Atomic Price-Provider Swap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`createPool` accepts `priceProviderTimelock = 0` without any minimum-value guard. When the stored timelock is zero, `proposePoolPriceProvider` sets `executeAfter = block.timestamp + 0 = block.timestamp`, and `executePoolPriceProviderUpdate` immediately passes its `block.timestamp < execAfter` check. The pool admin can therefore propose and execute an oracle replacement atomically in the same block — or even the same transaction — with no delay for LPs to react.

---

### Finding Description

`_validatePoolParameters` validates tokens, admin address, fee caps, and initial amounts, but contains **no check on `params.priceProviderTimelock`**: [1](#0-0) 

The only sentinel the factory recognises is `type(uint256).max` (immutable mode). Every other value — including `0` — is silently accepted as "mutable mode with delayed execution": [2](#0-1) 

The value is stored verbatim: [3](#0-2) 

When `priceProviderTimelock[pool] == 0`, `proposePoolPriceProvider` computes:

```
executeAfter = block.timestamp + 0  →  executeAfter == block.timestamp
``` [4](#0-3) 

`executePoolPriceProviderUpdate` then evaluates:

```
if (block.timestamp < block.timestamp)  →  false  →  no revert
``` [5](#0-4) 

Both calls succeed in the same block. The `PoolParameters` struct documents `priceProviderTimelock` only as "delay for mutable provider rotation; `type(uint256).max` means immutable provider" — no minimum is stated or enforced: [6](#0-5) 

---

### Impact Explanation

The timelock is the **only** constraint that keeps the pool admin "semi-trusted inside caps and timelocks." With a zero timelock the admin can:

1. Deploy a malicious `IPriceProvider` that returns an arbitrarily manipulated bid/ask.
2. In the same block: call `proposePoolPriceProvider` → `executePoolPriceProviderUpdate` → `swap` (or `removeLiquidity`) against the pool at the manipulated price.
3. LPs have zero blocks to observe the oracle change and exit.

This satisfies **bad-price execution** (stale/manipulated bid/ask reaches a pool swap) and **admin-boundary break** (pool admin bypasses the timelock cap the system relies on to bound admin power).

---

### Likelihood Explanation

`createPool` is **permissionless** — any address can deploy a pool with `priceProviderTimelock = 0`. The pool admin (often the same as the creator) immediately holds the ability to rotate the oracle with no delay. The mistake can also occur accidentally: a creator who intends "no timelock needed yet" or "use the default" may supply `0` without realising it eliminates the protection entirely — the exact accidental-misconfiguration scenario the external report describes.

---

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters`. For example:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 hours;

// inside _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) {
    revert PriceProviderTimelockTooShort();
}
```

This preserves the immutable-mode sentinel (`type(uint256).max`) while ensuring every mutable pool has a meaningful delay before an oracle change can be finalised.

---

### Proof of Concept

```solidity
// 1. Deploy pool with zero timelock (permissionless)
PoolParameters memory params = _defaultParams();
params.priceProviderTimelock = 0;          // NOT type(uint256).max → mutable mode
address pool = factory.createPool(params); // succeeds, no revert

// 2. Admin deploys a manipulated oracle
MockOracle badOracle = new MockOracle();
badOracle.setTokens(token0, token1);
badOracle.setBidAndAskPrice(MANIPULATED_BID, MANIPULATED_ASK);

// 3. In the same block: propose + execute atomically
vm.startPrank(admin);
factory.proposePoolPriceProvider(pool, address(badOracle));
// executeAfter == block.timestamp → check passes immediately
factory.executePoolPriceProviderUpdate(pool);  // succeeds, no warp needed
vm.stopPrank();

// 4. Pool now prices swaps from the manipulated oracle
// Admin (or a colluding swapper) executes a swap at the bad price
// LPs had zero blocks to react
```
