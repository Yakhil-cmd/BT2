### Title
Permanent ETH/ERC-20 Fund Freeze When `ExitToNear` Promise Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary
When the `error_refund` compile-time feature is not enabled (the default build), the `ExitToNear` precompile sets `refund: None` unconditionally and attaches no failure-handling callback to the outgoing NEAR promise. If the `ft_transfer` or `ft_transfer_call` promise fails at runtime (e.g., recipient account not registered with the NEP-141 contract), the ETH or ERC-20 tokens that were already deducted from the user's EVM state are permanently frozen with no recovery path.

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles two exit paths:

- **ETH (base token)**: ETH is transferred from the caller's EVM address to the precompile address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` as part of the EVM call's `apparent_value`. A NEAR `ft_transfer` promise is then scheduled on the eth-connector.
- **ERC-20**: The ERC-20 contract burns the tokens (calling the precompile), and a NEAR `ft_transfer` promise is scheduled on the NEP-141 contract.

In both cases, the EVM-side deduction is **final and irreversible** before the NEAR promise executes. The refund logic is gated behind a compile-time feature flag:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent, `callback_args` equals the default (both fields `None`), so the promise is created as a bare `Create` with no callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

If the NEAR-side `ft_transfer` fails, `exit_to_near_precompile_callback` is never invoked, so `refund_on_error` is never called, and the EVM state is never corrected. [3](#0-2) 

The refund path in `refund_on_error` â€” which would re-mint burned ERC-20 tokens or transfer ETH back from the precompile address â€” is only reachable when a callback with a non-`None` `refund` field is present: [4](#0-3) 

The precompile address `exit_to_near::ADDRESS` is a deterministic hash-derived address with no controlling private key, so any ETH stranded there is unrecoverable. [5](#0-4) 

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

- **ETH path**: ETH is moved to the precompile address inside the EVM before the promise fires. If the promise fails and `error_refund` is off, the ETH is permanently stranded at an uncontrolled address.
- **ERC-20 path**: ERC-20 tokens are burned (supply reduced) before the promise fires. If the promise fails, the corresponding NEP-141 tokens remain locked in Aurora's account and the user's ERC-20 balance is gone.

The integration tests explicitly document this loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [7](#0-6) 

### Likelihood Explanation

**Medium.** Any unprivileged EVM user can trigger this by calling `exitToNear` with a recipient NEAR account that is not registered with the eth-connector or the target NEP-141 contract. The `ft_transfer` call will fail on the NEAR side, the EVM-side deduction is already committed, and without `error_refund` there is no callback to restore the user's balance. The attacker-controlled input is simply the recipient account ID in the precompile calldata.

### Recommendation

1. Remove the `error_refund` feature gate and make the refund callback unconditional. The `refund_call_args` function and `ExitToNearPrecompileCallbackArgs::refund` field should always be populated for both ETH and ERC-20 exits.
2. Alternatively, validate that the recipient NEAR account exists and is registered before deducting any EVM-side balance, analogous to the fix recommended for VaultBooster (`deposit` should revert on unsupported tokens).

### Proof of Concept

1. User holds ETH on Aurora (EVM balance > 0).
2. User submits an EVM transaction calling the `exitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x00`, attaching ETH as `msg.value`, and specifying a recipient NEAR account that is not registered with the eth-connector (e.g., `"nonexistent.near"`).
3. The EVM deducts the ETH from the user's balance and credits it to the precompile address â€” this is final.
4. Aurora schedules a `ft_transfer` promise on the eth-connector for `"nonexistent.near"`.
5. The eth-connector rejects the transfer (account not registered); the promise fails.
6. Because `error_refund` is not enabled, `callback_args.refund == None`, no callback fires, and `refund_on_error` is never called.
7. The ETH remains at `exit_to_near::ADDRESS` permanently. The user's funds are irrecoverably frozen. [8](#0-7) [2](#0-1)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
