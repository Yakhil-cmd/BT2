Audit Report

## Title
Unprivileged Caller Can DoS Victim's ckBTC Deposit Minting via `update_balance` Owner Spoofing - (File: rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs)

## Summary
The `update_balance` endpoint accepts an `owner: Option<Principal>` argument and constructs the per-account concurrency guard from the caller-supplied value without any authorization check. Any unprivileged caller can continuously invoke `update_balance` with a victim's principal as `owner`, holding the victim's account lock for the duration of each async Bitcoin canister round-trip and returning `AlreadyProcessing` to the victim's own calls. This allows sustained denial of ckBTC minting for targeted accounts.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`, the effective account is derived from the caller-supplied argument:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),  // attacker-controlled
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

The only pre-guard check is:

```rust
if args.owner.unwrap_or(caller) == runtime.id() {
    ic_cdk::trap("cannot update minter's balance");
}
```

This rejects only the minter's own canister ID; any other principal is accepted without restriction.

`balance_update_guard` calls `Guard::new`, which inserts the account into the global `update_balance_accounts: BTreeSet<Account>`. While the guard is held, any concurrent call for the same account hits:

```rust
if accounts.contains(&account) {
    return Err(GuardError::AlreadyProcessing);
}
```

The guard is held across the async `get_utxos` call to the Bitcoin canister (at minimum one consensus round-trip, potentially several). The guard is only released on `Drop` after the full async execution completes.

**Exploit flow:**
1. Attacker calls `update_balance({ owner: Some(victim), subaccount: None })`.
2. Minter inserts `{ owner: victim, subaccount: None }` into `update_balance_accounts` and issues `get_utxos` for victim's derived BTC address.
3. Victim calls `update_balance({ owner: None, subaccount: None })`; minter constructs the same account, finds it locked, returns `UpdateBalanceError::AlreadyProcessing`.
4. Attacker's call completes with `NoNewUtxos` (attacker has no UTXOs at victim's address); guard is released.
5. Attacker immediately re-submits step 1. Steps 2–5 repeat.

The victim's calls consistently lose the race as long as the attacker re-submits before the victim's next ingress message is processed. The `MAX_CONCURRENT` limit of 100 also allows simultaneous targeting of up to 100 distinct accounts.

## Impact Explanation
**High.** A victim who has sent BTC to their ckBTC deposit address cannot call `update_balance` to mint ckBTC. Their Bitcoin funds are locked on-chain with no path to convert them to ckBTC tokens for the duration of the attack. This constitutes a concrete, sustained denial of a core ck-token financial operation (ckBTC minting) caused by an unprivileged attacker with no minimum stake or deposit. This matches the allowed High impact: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation
**Medium.** The attack requires no special privileges — any ingress sender or canister can call `update_balance` with an arbitrary `owner`. The victim's principal is publicly observable from their ckBTC ledger account or prior on-chain activity. The cost is limited to cycles for repeated inter-canister calls to the Bitcoin canister. The attacker cannot guarantee a mathematically permanent lock (the victim may win the race in a given round), but can sustain the DoS indefinitely at low cost, making it practically severe for high-value targets.

## Recommendation
Restrict `update_balance` so that the effective `owner` must equal the caller:

```rust
let caller_account = Account {
    owner: caller,  // always use caller; ignore args.owner
    subaccount: args.subaccount,
};
```

If third-party deposit notification is a required feature (e.g., a smart contract depositing on behalf of a user), introduce a separate non-locking notification path or require the owner to pre-authorize the caller via an on-chain allowlist, so the guard is never acquirable by an unauthorized third party.

## Proof of Concept
A deterministic PocketIC integration test can prove this:

1. Initialize a local ckBTC minter with a mock Bitcoin canister that introduces a configurable delay on `get_utxos` responses.
2. Submit `update_balance({ owner: Some(victim_principal), subaccount: None })` from attacker identity; do not await completion.
3. While the attacker's call is in-flight (guard held), submit `update_balance({ owner: None, subaccount: None })` from victim identity.
4. Assert victim's call returns `Err(UpdateBalanceError::AlreadyProcessing)`.
5. Assert attacker's call eventually returns `Err(UpdateBalanceError::NoNewUtxos)`.
6. Repeat steps 2–5 in a loop; assert victim never receives `Ok(...)` while attacker continuously re-submits.
