### Title
Cross-Context Signature Replay in `validateCOAOwnershipProof` Allows Forged COA Ownership Proof - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` is an `access(all)` Cadence function that verifies signatures over caller-supplied `signedData` to prove that a Cadence account controls a specific COA (Cadence-Owned Account). The function does not enforce that `signedData` commits to the `evmAddress` being proven. This is the direct Flow analog of the external report's signature replay vulnerability: a valid signature produced for one context (one COA or one dApp purpose) can be replayed by an unprivileged attacker to forge a passing ownership proof for a different COA owned by the same Cadence account.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` (lines 1001–1116) accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. It verifies that the provided signatures over `signedData` meet the account's key-weight threshold, then checks that the COA resource at `path` has an EVM address matching `evmAddress`. The critical gap is that **the function never checks whether `signedData` encodes `evmAddress`**. The code itself documents this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The function is declared `access(all)`, making it callable by any Cadence script, transaction, or EVM contract author without restriction: [2](#0-1) 

The signature verification step at lines 1082–1086 verifies signatures over `signedData` with domain tag `"FLOW-V0.0-user"`, but `evmAddress` is never part of the signed message: [3](#0-2) 

The `evmAddress` check at lines 1095–1109 only verifies that the COA resource at `path` matches the caller-supplied `evmAddress` — it does not verify that the signer intended to authorize that specific `evmAddress`: [4](#0-3) 

**Attack path:**

1. Alice owns a Cadence account with two COAs: `COA_A` (high-value, at `/public/coa_a`) and `COA_B` (low-value, at `/public/coa_b`).
2. A dApp asks Alice to prove ownership of `COA_B` by signing arbitrary `signedData` (e.g., a challenge nonce) that does not encode `COA_B`'s EVM address.
3. An attacker observes Alice's on-chain proof (the `signedData` + `signatures` are public inputs to any transaction or script).
4. The attacker calls `EVM.validateCOAOwnershipProof` with Alice's original `signedData` and `signatures`, but substitutes `address = Alice's Cadence address`, `path = /public/coa_a`, and `evmAddress = COA_A.address`.
5. The function returns `isValid: true` for `COA_A` — a COA Alice never authorized.
6. Any dApp or EVM contract that gates privileged actions (asset transfers, governance votes, bridge withdrawals) on the result of `validateCOAOwnershipProof` will incorrectly grant the attacker authorization over `COA_A`.

The `coaOwnershipProofValidator` in `fvm/evm/handler/precompiles.go` (lines 115–153) delegates directly to `validateCOAOwnershipProof` without adding any binding between `signedData` and `evmAddress`: [5](#0-4) 

The EVM-side `isValidSignature` in `coa.sol` (line 118) always passes `address(this)` as the COA address, which prevents replay via that specific path. However, the Cadence-callable `EVM.validateCOAOwnershipProof` is directly reachable by any unprivileged Cadence script or transaction author, and the `evmAddress` parameter is fully attacker-controlled: [6](#0-5) 

### Impact Explanation

An attacker who observes a valid COA ownership proof (signature + signedData) for any COA owned by a target Cadence account can replay it to forge a passing `validateCOAOwnershipProof` result for any other COA owned by the same account. Any Cadence contract, EVM contract, or off-chain authentication flow that uses `EVM.validateCOAOwnershipProof` to gate asset transfers, bridge operations, or privileged state changes — without independently verifying that `signedData` commits to the specific `evmAddress` — is vulnerable to unauthorized authorization. This can lead to direct theft or unauthorized movement of assets controlled by the victim's COA.

### Likelihood Explanation

The likelihood is medium. The preconditions are:
- A Cadence account must own more than one COA (not enforced against by the protocol).
- A dApp or contract must use `EVM.validateCOAOwnershipProof` directly (not exclusively through the EVM precompile's `address(this)` path) and must not independently bind `signedData` to `evmAddress`.
- The attacker must observe a prior valid proof for the victim account.

The function is `access(all)` and is part of the public EVM system contract API. As the Flow EVM ecosystem grows and more dApps build off-chain authentication flows on top of `validateCOAOwnershipProof`, the attack surface expands. The code's own comment acknowledges this risk for "off-chain authentication flows."

### Recommendation

Enforce at the protocol level that `signedData` must commit to `evmAddress`. The simplest fix is to have `validateCOAOwnershipProof` internally prepend or hash `evmAddress` into the message before verification, rather than relying on callers to do so:

```cadence
// Inside validateCOAOwnershipProof, before calling keyList.verify:
let boundData = evmAddress.toConstantSized<[UInt8; 20]>()!.toVariableSized().concat(signedData)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: boundData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

This ensures that a signature produced for one COA's EVM address cannot be replayed for a different COA, regardless of how callers construct `signedData`. The EVM precompile path and `coa.sol`'s `isValidSignature` would need to be updated to produce signatures over the same bound message format.

### Proof of Concept

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>

// Attacker script: replay Alice's proof for COA_B against COA_A
access(all)
fun main(
    aliceCadenceAddress: Address,
    coaAPath: PublicPath,          // /public/coa_a (high-value COA)
    coaAEvmAddress: [UInt8; 20],   // EVM address of COA_A
    // Observed from Alice's prior proof for COA_B:
    observedSignedData: [UInt8],
    observedKeyIndices: [UInt64],
    observedSignatures: [[UInt8]]
): EVM.ValidationResult {
    // Replay Alice's COA_B signature against COA_A
    return EVM.validateCOAOwnershipProof(
        address: aliceCadenceAddress,
        path: coaAPath,
        signedData: observedSignedData,   // same signedData, no evmAddress binding
        keyIndices: observedKeyIndices,
        signatures: observedSignatures,
        evmAddress: coaAEvmAddress        // substituted: COA_A instead of COA_B
    )
    // Returns isValid: true if Alice's key weight >= 1000
    // and COA_A exists at coaAPath — Alice never authorized this
}
