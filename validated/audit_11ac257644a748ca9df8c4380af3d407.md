### Title
Missing Zero-Address Validation for L1→L2 Refund Recipient Permanently Burns Base Tokens - (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The L1→L2 transaction processing pipeline in ZKsync OS does not validate that the `refund_recipient` field is non-zero before minting the gas refund. An L1 sender who sets `refund_recipient = address(0)` causes the bootloader to permanently transfer base tokens from the treasury to `address(0)`, making them irretrievable. The developers themselves flagged this gap with a `// TODO: validate address?` comment in the structure-validation function.

---

### Finding Description

L1→L2 transactions encode a `refund_recipient` address in `reserved[1]`. After execution, the bootloader computes the unused-gas refund and mints it from the treasury to that address. The validation function `validate_structure()` explicitly skips any check on this field:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
``` [1](#0-0) 

The refund recipient is then extracted with `u256_to_b160_checked`, which only asserts the value fits in 160 bits — it performs no zero-address check:

```rust
pub fn u256_to_b160_checked(src: U256) -> B160 {
    assert!(src.as_limbs()[3] == 0 && src.as_limbs()[2] < (1u64 << 32));
    ...
}
``` [2](#0-1) 

The extracted address is passed directly to `mint_base_token`, which calls `transfer_from_treasury`:

```rust
let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
mint_base_token::<S, Config>(
    system, system_functions, memories.reborrow(),
    &to_refund_recipient,
    &refund_recipient,   // ← can be address(0)
    ...
)
``` [3](#0-2) 

`transfer_from_treasury` subtracts from the treasury and adds to `to` with no zero-address guard:

```rust
system.io.update_account_nominal_token_balance(
    ..., treasury_address, nominal_token_value, true, ...)?;   // debit treasury
system.io.update_account_nominal_token_balance(
    ..., to, nominal_token_value, false, ...)?;                // credit `to`
``` [4](#0-3) 

The existing regression test explicitly uses `refund_recipient = address(0)` and asserts the transaction succeeds, confirming the path is live:

```rust
let refund_recipient = address!("0000000000000000000000000000000000000000"); // refund recipient (zero address)
``` [5](#0-4) 

---

### Impact Explanation

When `refund_recipient = address(0)` and `to_refund_recipient > 0` (which is true whenever `gas_limit > gas_used`, i.e., virtually every transaction), the bootloader:

1. Debits `to_refund_recipient` tokens from the treasury (`BASE_TOKEN_HOLDER_ADDRESS`).
2. Credits those tokens to `address(0)`.

The tokens credited to `address(0)` are permanently inaccessible — they cannot be recovered by any user or protocol mechanism. This is a direct, irreversible loss of base tokens from the protocol treasury. The treasury is the canonical reserve for all L1→L2 deposits; draining it to `address(0)` reduces the total supply of accessible base tokens on L2.

---

### Likelihood Explanation

The attack requires only that an L1 sender set `refund_recipient = address(0)` in their L1→L2 transaction. This is an unprivileged, externally-submitted operation — no special role, key, or governance approval is needed. The `ZKsyncL1Tx` struct exposes `refund_recipient` as a plain `Address` field with no constraints at the encoding layer. [6](#0-5) 

The `L1TxBuilder` defaults `refund_recipient` to `Address::default()` (i.e., `address(0)`) when the caller omits it:

```rust
refund_recipient: self.refund_recipient.unwrap_or_default(),
``` [7](#0-6) 

This means any L1 sender who forgets to set the refund recipient — or any contract that omits it — will silently burn their gas refund on every L1→L2 transaction.

---

### Recommendation

In `validate_structure()` inside `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`, replace the `// TODO: validate address?` stub with an explicit non-zero check for `reserved[1]` on `L1_L2_TX_TYPE` and `UPGRADE_TX_TYPE` transactions:

```rust
Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
    // Refund recipient must not be the zero address.
    let refund_recipient = self.reserved[1].read();
    if refund_recipient.is_zero() {
        return Err(());
    }
}
```

Alternatively, if a zero refund recipient is intended to mean "refund to sender", redirect the refund to `from` when `reserved[1]` is zero inside `process_l1_transaction`.

---

### Proof of Concept

**Entry path (no special privileges required):**

1. Construct an L1→L2 transaction with `refund_recipient = address(0)`, a non-zero `gas_limit`, and any `gas_price > 0`.
2. Submit it through the L1 bridge (or directly in a test via `L1TxBuilder::new().from(...).to(...).gas_limit(...).gas_price(...).build()`).
3. The transaction executes. `gas_used < gas_limit` (normal case).
4. `to_refund_recipient = (gas_limit - gas_used) * gas_price > 0`.
5. `process_l1_transaction` calls `mint_base_token(..., &to_refund_recipient, &B160::ZERO, ...)`.
6. `transfer_from_treasury` debits the treasury and credits `address(0)`.
7. `tester.get_balance(&address(0))` increases by `to_refund_recipient`; those tokens are permanently inaccessible.

The existing test at `tests/instances/transactions/src/lib.rs:1843` already demonstrates this exact scenario and asserts it succeeds, serving as a ready-made proof of concept. [8](#0-7)

pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
{
    system_log!(
        system,
        "Transferring {nominal_token_value:?} tokens from treasury to {to:?}\n"
    );

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
            true, // true = subtract from balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Treasury transfer failed: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::TreasuryTransferFailed)
                }
                _ => wrap_error!(e),
            }
        })?;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            to,
            nominal_token_value,
            false, // false = add to balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Error while minting: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::MintingBalanceOverflow)
                }
                _ => wrap_error!(e),
            }
        })?;

    Ok(())
}
