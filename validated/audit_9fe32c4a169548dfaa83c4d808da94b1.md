### Title
Off-by-One in DKG Threshold Upper-Bound Check Allows N-of-N Key Generation - (File: src/dkg.rs)

### Summary
`assert_key_invariants` in `src/dkg.rs` uses a strict `>` comparison to reject an oversized threshold, but the protocol specification requires a strict `<` relationship between threshold and participant count. This allows a caller to pass `threshold == N` (all participants), which the library silently accepts and runs DKG with, producing a key that can never be signed without every single participant — a permanent denial-of-signing condition.

### Finding Description

The protocol specification in `docs/dkg.md` (Step 1.1) states:

> Each $P_i$ asserts that $1 < \mathsf{threshold} < N$.

The upper bound is **strict**: threshold must be strictly less than the number of participants.

The implementation in `assert_key_invariants` enforces:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [1](#0-0) 

This check only rejects `threshold > N`, but **allows `threshold == N`**. The spec requires `threshold < N` (strict), so the correct guard is `threshold >= participants.len()`.

The spec document explicitly states the strict inequality: [2](#0-1) 

`assert_key_invariants` is the shared validation entry point called by all three public DKG operations — `keygen`, `reshare`, and `refresh` — via `assert_reshare_keys_invariants`: [3](#0-2) [4](#0-3) 

### Impact Explanation

When `threshold == N` is accepted:

1. DKG completes successfully, producing shares for all N participants.
2. The underlying polynomial has degree `N-1`, requiring all N shares to reconstruct.
3. Every subsequent signing, reshare, or refresh operation requires **all N participants** simultaneously.
4. If any single participant is absent, crashes, or refuses to cooperate, signing is **permanently impossible** — the key is effectively locked forever.

This maps directly to the allowed High impact: **Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions**.

### Likelihood Explanation

The `keygen`, `reshare`, and `refresh` public APIs accept `threshold` as a caller-supplied parameter with no type-level restriction preventing `threshold == N`. A library integrator who wants "require all participants" semantics, or who makes an off-by-one error (e.g., passing `participants.len()` instead of `participants.len() - 1`), will silently produce an unusable key. The library's own error type `ThresholdTooLarge` implies it intends to reject this case, making the missing `==` branch a plausible oversight. The call path is fully reachable by any unprivileged library caller.

### Recommendation

Change the comparison in `assert_key_invariants` from strict greater-than to greater-than-or-equal, to enforce the spec's strict upper bound `threshold < N`:

```rust
// Before (incorrect — allows threshold == N):
if threshold > participants.len() {

// After (correct — enforces threshold < N per spec):
if threshold >= participants.len() {
``` [5](#0-4) 

### Proof of Concept

1. Call `keygen` with `participants = [P1, P2, P3]` (N=3) and `threshold = 3`.
2. `assert_key_invariants` checks `3 > 3` → false → no error returned; DKG proceeds.
3. DKG produces degree-2 polynomial shares for all three participants.
4. Attempt to sign with any two participants (threshold=3 requires all 3).
5. Signing fails because the reconstruction requires all 3 shares.
6. If P3 is permanently offline, the key can never be used — permanent DoS on signing for P1 and P2.
