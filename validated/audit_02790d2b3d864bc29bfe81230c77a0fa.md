### Title
Users Can Delete All Access Keys From a NEAR Account, Permanently Locking It Out — (`File: runtime/runtime/src/access_keys.rs`)

### Summary
`action_delete_key` contains no guard preventing removal of the last access key from an account. A user who submits a `DeleteKeyAction` for their sole remaining key will succeed, leaving the account with an empty key set. Every subsequent transaction will fail with `AccessKeyNotFound`, and the account's funds become permanently inaccessible with no recovery path.

### Finding Description
`action_delete_key` is the production execution handler for `DeleteKeyAction`. Its full logic is:

```rust
pub(crate) fn action_delete_key(
    config: &RuntimeConfig,
    state_update: &mut TrieUpdate,
    account: &mut Account,
    result: &mut ActionResult,
    account_id: &AccountId,
    delete_key: &DeleteKeyAction,
) -> Result<(), RuntimeError> {
    let access_key = get_access_key(state_update, account_id, &delete_key.public_key)?;
    if let Some(access_key) = access_key {
        if let Some(gas_key_info) = access_key.gas_key_info() {
            delete_gas_key(...)?;
        } else {
            delete_regular_key(...);
        }
    } else {
        result.result = Err(ActionErrorKind::DeleteKeyDoesNotExist { ... }.into());
    }
    Ok(())
}
```

The only guard is `DeleteKeyDoesNotExist` — the key being deleted must exist. There is no check on how many keys remain after deletion. When the deleted key is the last one, `remove_access_key` is called unconditionally, writing the empty state to the trie. [1](#0-0) 

The downstream `delete_regular_key` helper simply calls `remove_access_key` and adjusts `storage_usage` with no count check: [2](#0-1) 

The integration test `test_delete_key_last` explicitly confirms this succeeds and leaves the account with zero keys: [3](#0-2) 

The TODO comment inside that test even acknowledges a secondary consequence: because the signing key is gone, nonce-based retry logic breaks, but the root issue — that the deletion itself succeeds — is not flagged as a bug. [4](#0-3) 

### Impact Explanation
After the last key is removed, every future transaction signed by any key will be rejected at the pre-inclusion validation stage with `InvalidAccessKeyError::AccessKeyNotFound`. The account's NEAR balance, any deployed contract, and all stored state become permanently inaccessible. There is no protocol-level recovery mechanism (no "default key", no guardian, no admin override). The corrupted value is the account's access-key trie namespace: it transitions to the empty set, making `get_access_key` return `None` for every public key forever.

**Impact: High** — permanent, irreversible loss of account control and all associated funds.

### Likelihood Explanation
**Likelihood: Low** — a user must deliberately (or accidentally) submit a `DeleteKeyAction` targeting their only remaining key. Accidental deletion is plausible during key-rotation workflows where the user deletes the old key before confirming the new one was added. A malicious contract could also trigger this via `promise_batch_action_delete_key` if the account grants it a full-access key. [5](#0-4) 

### Recommendation
Add a guard inside `action_delete_key` (or inside `delete_regular_key` / `delete_gas_key`) that counts remaining access keys before performing the removal. If the key to be deleted is the last one, set `result.result` to a new `ActionErrorKind::DeleteLastKey` error and return without modifying state. Alternatively, enforce the invariant at the action-list validation layer in `action_validation.rs` by rejecting any receipt whose action list would leave the account with zero keys.

### Proof of Concept
1. Account `alice.near` is created with a single full-access key `K`.
2. Alice submits a transaction signed with `K` containing `Action::DeleteKey { public_key: K }`.
3. `action_delete_key` is called; `get_access_key` returns `Some(K)`, the `else` branch is not taken, `delete_regular_key` calls `remove_access_key`, and `result.result` remains `Ok(())`.
4. The transaction succeeds on-chain (`FinalExecutionStatus::SuccessValue([])`).
5. Alice's account now has zero access keys in the trie.
6. Any subsequent transaction from any key is rejected: `InvalidTxError::InvalidAccessKeyError(AccessKeyNotFound { account_id: "alice.near", public_key: ... })`.
7. Alice's balance is permanently locked with no recovery path. [6](#0-5)

pub fn test_delete_key_last(node: impl Node) {
    let account_id = &node.account_id().unwrap();
    let node_user = node.user();
    let root = node_user.get_state_root();

    assert!(node_user.get_access_key(account_id, &node.signer().public_key()).is_ok());
    let transaction_result = node_user.delete_key(account_id.clone(), node.signer().public_key());

    match transaction_result {
        Ok(transaction_result) => {
            assert_eq!(transaction_result.status, FinalExecutionStatus::SuccessValue(Vec::new()));
            assert_eq!(transaction_result.receipts_outcome.len(), 1 + extra_refund_outcomes());
        }
        Err(err) => {
            // TODO(#6724): This is a wrong error, the transaction actually
            // succeeds. We get an error here when we retry the tx and the second
            // time around it fails. Normally, retries are handled by nonces, but we
            // forget the nonce when we delete a key!
            assert_eq!(
                err,
                CommitError::Server(ServerError::TxExecutionError(
                    TxExecutionError::InvalidTxError(InvalidTxError::InvalidAccessKeyError(
                        InvalidAccessKeyError::AccessKeyNotFound {
                            account_id: account_id.clone(),
                            public_key: node.signer().public_key().into(),
                        },
                    ))
                ))
            )
        }
    }

    let new_root = node_user.get_state_root();
    assert_ne!(new_root, root);

    assert!(node_user.get_access_key(account_id, &node.signer().public_key()).is_err());
