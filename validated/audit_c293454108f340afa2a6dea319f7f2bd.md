### Title
Unauthorized Invoke Transaction Admitted Without Signature Verification via `skip_stateful_validations` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is designed to improve UX for the deploy_account + invoke bundle pattern. However, it checks only that **any** transaction from the sender's address exists in the mempool — not that a deploy_account specifically exists. An attacker who observes a victim's deploy_account in the mempool can submit a forged Invoke with `sender_address=victim`, `nonce=1`, and an invalid signature. The gateway admits it without ever calling `__validate__`, because `execution_flags.validate` is set to `false`. During sequencing, `__validate__` fails and the victim is charged the validation fee.

### Finding Description

**Step 1 — `skip_stateful_validations` condition is attacker-satisfiable.** [1](#0-0) 

The function returns `true` when all three hold:
- The tx is `Invoke`
- `tx.nonce() == Nonce(Felt::ONE)` — attacker sets this
- `account_nonce == Nonce(Felt::ZERO)` — victim not yet deployed
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

**Step 2 — `account_tx_in_pool_or_recent_block` is satisfied by the victim's own deploy_account.** [2](#0-1) 

This returns `true` if the address has **any** tx in the pool or recent block — not specifically a deploy_account. The victim's deploy_account (nonce=0) already in the mempool satisfies this. The code comment acknowledges this: "either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

**Step 3 — `run_validate_entry_point` sets `validate=false` and skips `__validate__`.** [3](#0-2) 

When `skip_validate=true`, `execution_flags.validate = !skip_validate = false`.

**Step 4 — `perform_validations` returns `Ok(())` after pre-validation without calling `__validate__`.** [4](#0-3) 

For Invoke transactions, after `perform_pre_validation_stage`, the code checks `if !tx.execution_flags.validate { return Ok(()); }` — the `__validate__` call is never reached.

**Step 5 — `perform_pre_validation_stage` passes for the attacker's tx.** [5](#0-4) 

- `handle_nonce` with `strict_nonce_check=false`: accepts `account_nonce(0) <= tx_nonce(1)` ✓
- `verify_can_pay_committed_bounds`: reads the victim's fee token balance. Passes if the victim pre-funded their address (the standard Starknet onboarding pattern) ✓
- `validate_proof_facts`: passes for a standard Invoke ✓

**Step 6 — Gateway's `validate_nonce` also passes.** [6](#0-5) 

For Invoke with `account_nonce=0`, `tx_nonce=1`: `0 <= 1 <= 0 + max_allowed_nonce_gap`. The default gap is ≥ 1 (confirmed by the `should_skip_validation` test case which uses `Default::default()` config and expects the nonce check to pass).

**Step 7 — Economic impact during sequencing.**

When the sequencer later executes the block:
1. Victim's deploy_account (nonce=0) executes → account deployed, nonce becomes 1
2. Attacker's Invoke (nonce=1) executes → `__validate__` is called with the invalid signature → fails → transaction reverts → victim is charged the validation fee

### Impact Explanation

- **Unauthorized transaction admitted**: An Invoke with an invalid (attacker-crafted) signature is accepted by the gateway without signature verification, bypassing the core account abstraction security guarantee.
- **Economic harm**: The victim's pre-funded balance is charged the validation fee for a transaction they never authorized.
- **Nonce slot griefing**: The attacker's tx occupies the victim's nonce=1 slot. If the victim also submits a legitimate Invoke with nonce=1, the mempool's duplicate/replacement policy determines which survives.

### Likelihood Explanation

The attack requires:
1. Observing a victim's deploy_account tx in the mempool (public information)
2. Knowing the victim's address (deterministic from deploy_account parameters, also public)
3. Submitting an Invoke with `sender_address=victim`, `nonce=1`, any signature, before the deploy_account is sequenced

All three conditions are trivially satisfiable by any unprivileged network participant. The victim must have pre-funded their address (standard practice), which is the only prerequisite on the victim's side.

### Recommendation

`skip_stateful_validations` must verify that the account's existing mempool entry is specifically a **deploy_account** transaction, not just any transaction. The current check (`account_tx_in_pool_or_recent_block`) is too broad — it returns `true` for any account that has ever had a transaction in the mempool, including accounts that have already been deployed and have future-nonce Invoke transactions queued.

A concrete fix: query the mempool for the specific deploy_account tx hash for the sender address (e.g., a dedicated `get_deploy_account_tx_hash` RPC), or store and check the tx type of the nonce=0 entry for the address.

### Proof of Concept

```
1. Victim calls gateway with DeployAccount(sender=V, nonce=0, sig=valid_sig)
   → Mempool accepts it. account_tx_in_pool_or_recent_block(V) == true.

2. Attacker calls gateway with Invoke(sender=V, nonce=1, sig=GARBAGE)
   → validate_nonce: 0 <= 1 <= gap → OK
   → validate_by_mempool: no duplicate → OK
   → skip_stateful_validations:
       tx is Invoke ✓, nonce==1 ✓, account_nonce==0 ✓,
       account_tx_in_pool_or_recent_block(V)==true ✓
       → returns true
   → run_validate_entry_point: execution_flags.validate=false
   → perform_pre_validation_stage: nonce 0<=1 OK, balance OK
   → returns Ok(()) — __validate__ never called
   → Gateway returns 200 OK, tx admitted to mempool

3. Sequencer block:
   - Executes DeployAccount(V, nonce=0) → V deployed, nonce=1
   - Executes Invoke(V, nonce=1, sig=GARBAGE):
       __validate__ called → signature check fails → revert
       Victim V charged validation fee
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
