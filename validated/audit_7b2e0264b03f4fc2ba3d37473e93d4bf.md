### Title
Gateway Admits Invoke Transactions with Invalid Signatures by Exploiting `skip_stateful_validations` via Any Pending Pool Transaction - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the `__validate__` entry point (the account's signature-verification function) for an invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender. That helper returns `true` if **any** transaction from the address is in the pool — including a victim's pending `deploy_account` (nonce=0). An attacker who observes a victim's `deploy_account` in the mempool can immediately submit an invoke transaction with nonce=1 from the victim's address carrying an arbitrary/invalid signature. The gateway's pre-validation chain (`validate_by_mempool` → `skip_stateful_validations`) passes without ever calling `__validate__`, and the invalid transaction is admitted to the mempool.

### Finding Description

**Exact code path:**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which executes three steps in order:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
``` [1](#0-0) 

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, and `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`: [3](#0-2) [4](#0-3) 

**The broken invariant:**

The code comment claims:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This is wrong. `account_tx_in_pool_or_recent_block` returns `true` if **any** transaction from the address is in `tx_pool` or in the committed state — it does not distinguish a `deploy_account` (nonce=0) from any other transaction type: [5](#0-4) [6](#0-5) 

A victim's `deploy_account` (nonce=0) in the pool satisfies `tx_pool.contains_account(victim_address) == true`. This is a different nonce slot (0) from the attacker's invoke (nonce=1), so `validate_by_mempool` does not detect a conflict: [7](#0-6) 

`validate_incoming_tx` only rejects if `tx_nonce < resolved_account_nonce`. With `tx_nonce=1` and `account_nonce=0`, this passes. `validate_fee_escalation` only rejects if a tx with the **same nonce** already exists in the pool; since only nonce=0 exists, it also passes. [8](#0-7) 

**Step-by-step exploit:**

1. Victim broadcasts `deploy_account` for address `A` (nonce=0). It enters the mempool pool; `tx_pool.contains_account(A)` becomes `true`.
2. Attacker reads `A` from the public mempool and crafts an invoke tx: `sender=A`, `nonce=1`, `signature=[0xff, 0xff, ...]` (garbage).
3. Gateway stateless checks pass (no signature check there).
4. `validate_state_preconditions`: `account_nonce=0`, `tx_nonce=1`, within `max_allowed_nonce_gap` → passes.
5. `validate_by_mempool`: no existing nonce=1 tx for `A`, nonce not too old → passes.
6. `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)=true` → returns `true`.
7. `run_validate_entry_point` with `validate=false` → `__validate__` never called → `Ok(())`.
8. Invalid invoke tx is forwarded to the mempool and stored.

### Impact Explanation

- **Invalid transaction admitted**: An invoke transaction with an arbitrary/invalid signature bypasses the gateway's signature-verification step and enters the mempool. This directly violates the "High. Mempool/gateway/RPC admission accepts invalid transactions" impact category.
- **Nonce-slot griefing**: The attacker's invalid nonce=1 tx occupies the victim's first post-deploy nonce slot. The victim's own valid invoke tx (nonce=1) is subsequently rejected with `DuplicateNonce` by `validate_by_mempool`, blocking the victim from using their newly-deployed account until the invalid tx is evicted or the victim pays a fee-escalation premium.
- **Block-space waste**: When the batcher executes the block, the invalid tx runs `__validate__` for the first time, fails, and reverts — consuming block resources without economic justification.
- **Scalable attack**: Any `deploy_account` visible in the mempool is a target. An attacker can submit one invalid invoke per pending `deploy_account` at negligible cost (only stateless checks are paid).

### Likelihood Explanation

`deploy_account` transactions are publicly visible in the mempool. The attacker needs only to read the sender address from any pending `deploy_account` and submit a crafted invoke with nonce=1. No privileged access, no cryptographic secret, and no race condition beyond submitting before the victim's own nonce=1 invoke is required. The skip-validate UX feature is always active (no configuration flag disables it).

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. Add a dedicated mempool query such as `deploy_account_in_pool(sender_address)` that inspects the transaction type of the pooled entry at nonce=0, and use that as the guard in `skip_stateful_validations`:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use:
mempool_client.deploy_account_in_pool(tx.sender_address())
```

This ensures the skip path is only triggered when the account's own `deploy_account` is genuinely pending, not merely because any transaction from that address happens to be in the pool.

### Proof of Concept

```
State:
  - Account A: on-chain nonce = 0 (not deployed)
  - Mempool pool: { deploy_account(sender=A, nonce=0, valid_sig) }

Attacker submits:
  InvokeV3 {
    sender_address: A,
    nonce: 1,
    signature: [0xdead, 0xbeef],   // invalid
    calldata: [...],
    resource_bounds: { l2_gas: { max_amount: X, max_price_per_unit: Y } },
    ...
  }

Gateway trace:
  1. StatelessTransactionValidator::validate  → Ok  (no sig check)
  2. get_nonce_from_state(A)                  → Nonce(0)
  3. validate_state_preconditions             → Ok  (0 ≤ 1 ≤ max_gap)
  4. validate_by_mempool                      → Ok  (no nonce=1 conflict)
  5. skip_stateful_validations:
       nonce==1 ✓, account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(A)  → true  (deploy_account is in pool)
       returns true
  6. run_validate_entry_point(skip_validate=true)
       execution_flags.validate = false
       StatefulValidator::perform_validations → returns Ok() immediately
  7. Tx forwarded to mempool → ADMITTED

Result:
  Mempool now contains:
    deploy_account(A, nonce=0, valid)
    invoke(A, nonce=1, INVALID signature)   ← admitted without __validate__

  Victim's subsequent invoke(A, nonce=1, valid_sig) → rejected: DuplicateNonce
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
