### Title
Missing Slippage Protection on `removeLiquidity` Exposes LPs to Sandwich Attacks - (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.removeLiquidity` accepts no minimum-output parameters, and no periphery wrapper exists that adds such a guard. An LP removing liquidity is fully exposed to a sandwich attack: an adversary can front-run with a swap that shifts the bin cursor to an unfavorable position, causing the LP to receive a worse token ratio, then back-run to capture the spread.

### Finding Description

`MetricOmmPool.removeLiquidity` signature is:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
``` [1](#0-0) 

There are no `minAmount0Out` / `minAmount1Out` parameters, and the returned values are never compared against any caller-supplied threshold inside the function.

The amount each LP receives is determined by the current `curBinIdx` / `curPosInBin` cursor, which is an internal state variable that any swap can move. [2](#0-1) 

The periphery layer provides slippage protection only for swaps (`amountOutMinimum` / `amountInMaximum` in `MetricOmmSimpleRouter`) and for *adding* liquidity (`maxAmountToken0` / `maxAmountToken1` checked in `metricOmmModifyLiquidityCallback`). [3](#0-2) 

No `removeLiquidity` wrapper exists anywhere in `metric-periphery/contracts/`. 

Unlike the original AlgebraPool finding (where the judge downgraded severity because only a *contract* caller could be affected), `removeLiquidity` in Metric OMM enforces `msg.sender == owner`, meaning **EOAs call it directly** and are the primary victims. [4](#0-3) 

### Impact Explanation

Within a bin, the cursor position determines the marginal price and therefore the token0/token1 split an LP receives on withdrawal. A swap that moves the cursor from position A to position B changes that split. An attacker who:

1. Observes a pending `removeLiquidity` in the mempool,
2. Front-runs with a swap that pushes the cursor toward the end of the bin (draining the more-valuable token from the bin),
3. Lets the LP's `removeLiquidity` execute — the LP receives less of the valuable token and more of the cheap token,
4. Back-runs by swapping back,

captures the spread at the LP's expense. The LP has no on-chain mechanism to revert if the received amounts fall below acceptable thresholds.

### Likelihood Explanation

- `removeLiquidity` is callable by any EOA that owns a position — the most common user path.
- No periphery guard exists; users have no alternative protected entry point.
- The attack requires only a standard sandwich bot with mempool visibility, which is routine on all EVM chains where Metric OMM is deployed.
- The pool is oracle-priced, but the cursor is internal state and is freely movable by swaps, so oracle integrity does not prevent this attack.

### Recommendation

Add `minAmount0Out` and `minAmount1Out` parameters to `removeLiquidity` (or to a periphery wrapper) and revert if the returned amounts fall below those thresholds:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,   // <-- add
    uint256 minAmount1Out,   // <-- add
    bytes calldata extensionData
) external ... returns (uint256 amount0Removed, uint256 amount1Removed) {
    ...
    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out)
        revert InsufficientOutput(amount0Removed, amount1Removed, minAmount0Out, minAmount1Out);
}
```

Alternatively, add a `MetricOmmPoolLiquidityAdder.removeLiquidityWithMinOut` wrapper in the periphery that reads the expected amounts via a probe (mirroring the existing `addLiquidityWeighted` probe pattern) and reverts if the actual output is below the caller's floor.

### Proof of Concept

1. Alice holds shares in bin 0 and submits `removeLiquidity(alice, salt, deltas, "")`.
2. Attacker sees the transaction in the mempool.
3. Attacker front-runs: calls `swap(zeroForOne=false, ...)` to buy token0, moving the cursor toward the high end of bin 0. Bin 0 now holds mostly token1 and little token0.
4. Alice's `removeLiquidity` executes. `LiquidityLib.removeLiquidity` computes her share of the bin at the new cursor position — she receives far less token0 (the more valuable asset) and more token1 than she expected.
5. Attacker back-runs: sells token0 back, restoring the cursor and pocketing the spread.
6. Alice has no revert path because `removeLiquidity` imposes no output floor. [1](#0-0) [5](#0-4)
