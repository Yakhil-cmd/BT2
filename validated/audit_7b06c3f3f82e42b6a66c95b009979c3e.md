### Title
Permanent Fund Freeze When `ExitToNear` Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

When a user exits ETH or ERC-20 tokens from Aurora to NEAR via the `ExitToNear` precompile, the EVM balance is burned first and a NEAR cross-contract `ft_transfer` promise is scheduled asynchronously. If that promise fails and the `error_refund` compile-time feature is not enabled, no refund callback is attached, and the burned tokens are permanently lost. This violates the bridge accounting invariant `total_evm_eth_supply ≤ total_nep141_held_by_aurora_contract` and results in a permanent freeze of user funds.

### Finding Description

The `ExitToNear` precompile's `run` function constructs a `callback_args` struct whose `refund` field is conditionally compiled: [1](#0-0) 

When `error_refund` is **not** enabled, `refund` is `None`. If `transfer_near_args` is also `None` (the common case for regular ETH and ERC-20 exits), `callback_args` equals its `Default` value, and the code takes the no-callback branch: [2](#0-1) 

A bare `PromiseArgs::Create` is emitted with no error-handling callback. If the `ft_transfer` or `ft_transfer_call` promise fails on the NEP-141 side, execution returns to Aurora with no mechanism to restore the user's burned EVM balance.

The callback that *would* perform the refund is `exit_to_near_precompile_callback`: [3](#0-2) 

This branch is only reachable when a callback was attached in the first place. Without `error_refund`, it is never scheduled.

The `refund_on_error` function itself is correct when called: [4](#0-3) 

But it is never invoked for the failing promise when the feature is absent.

The test suite explicitly documents this behavior as a known gap: [5](#0-4) [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract, connector paused, or any revert in the NEP-141 contract), the user's EVM ETH or ERC-20 tokens are irreversibly burned. The NEP-141 tokens remain held by the Aurora contract, but the corresponding EVM balance is gone. The bridge accounting invariant `total_evm_eth = total_nep141_transferred_out` is violated: Aurora holds more NEP-141 than it should, and the user holds nothing. There is no recovery path.

### Likelihood Explanation

**Medium.** The `ft_transfer` promise can fail for several realistic reasons reachable by an ordinary EVM user:

1. Sending to a NEAR account not registered with the NEP-141 contract (storage deposit not paid).
2. The NEP-141 contract being paused.
3. Any revert in the NEP-141 `ft_transfer` logic.

The entry path is fully unprivileged: any EVM account can call `withdrawToNear` on an `EvmErc20` contract or call the `ExitToNear` precompile directly with ETH value. [7](#0-6) 

### Recommendation

1. **Enable `error_refund` unconditionally in production builds**, or promote it from a feature flag to always-on behavior. The refund callback path already exists and is correct; it simply must always be attached.
2. If keeping the feature flag, gate the `ExitToNear` precompile's `run` function so it returns an error at the EVM level when `error_refund` is disabled and the promise could fail, preventing the burn from occurring without a safety net.
3. Analogous to the original report's recommendation of `min(balance, slashAmount)`: if a refund cannot be fully executed (e.g., the precompile address balance is insufficient), the engine should take whatever is available rather than returning `ERR_REFUND_FAILURE` and leaving the user with nothing.

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora as an ERC-20 (user holds ERC-20 balance X).
3. Call `withdrawToNear("unregistered.near", X)` on the ERC-20 contract.
4. The ERC-20 tokens are burned (`_burn` in `EvmErc20.withdrawToNear`), the `ExitToNear` precompile fires, and a bare `ft_transfer` promise is scheduled with no callback.
5. The `ft_transfer` fails because `unregistered.near` has no storage deposit with the NEP-141 contract.
6. No callback fires. The user's ERC-20 balance is 0. The NEP-141 tokens remain with Aurora. Funds are permanently frozen.

This is confirmed by the existing test `test_exit_to_near_refund`: [8](#0-7)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
