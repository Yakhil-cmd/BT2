### Title
Gateway Skips Signature Verification for Invoke Transactions with Nonce=1 Based on Overly Broad Mempool Presence Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway stateful validator bypasses the `__validate__` entry-point call (i.e., account signature verification) for any invoke transaction with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that mempool query returns `true` for **any** transaction from the address — not specifically a `deploy_account` — an unprivileged attacker can front-run a legitimate user's first post-deploy invoke by submitting an invoke with a forged/wrong signature, causing the gateway to admit it without signature verification.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the nonce and resource-bound checks pass:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
``` [1](#0-0) 

The skip condition fires when:
1. The transaction is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account does not yet exist on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [2](#0-1) 

When the skip fires, `run_validate_entry_point` is called with `skip_validate = true`, which sets `execution_flags.validate = false` and passes the transaction to `StatefulValidator::validate` **without** calling `__validate__`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the `__validate__` call is entirely skipped and `Ok(())` is returned immediately: [4](#0-3) 

The mempool's `account_tx_in_pool_or_recent_block` checks `MempoolState::contains_account` (staged/committed nonces) **or** `TransactionPool::contains_account` (any transaction in the pool): [5](#0-4) [6](#0-5) 

Neither check distinguishes a `deploy_account` transaction from any other transaction type. The code comment acknowledges this is intentional ("it means that either it has a deploy_account transaction **or** transactions with future nonces that passed validations"), but this reasoning is circular: a prior invoke with `nonce=1` could itself have been admitted via this same skip, anchoring the chain to a deploy_account that may have since been evicted.

**Attack path:**

1. Victim submits `deploy_account` for address `A` → admitted to mempool.
2. Attacker observes the pending `deploy_account` in the mempool.
3. Attacker submits `Invoke(sender=A, nonce=1, signature=<forged>)`.
4. Gateway evaluates `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓ (A does not exist on-chain)
   - `account_tx_in_pool_or_recent_block(A) == true` ✓ (deploy_account is in pool)
5. `__validate__` is **not called**; the forged-signature invoke is admitted to the mempool.
6. Victim's legitimate `Invoke(sender=A, nonce=1)` is now rejected by the mempool as `DuplicateNonce`.
7. When the batcher executes the attacker's invoke, `__validate__` is called (execution flags are set fresh), the forged signature fails, the transaction reverts, and the nonce is incremented — permanently blocking the victim's nonce-1 invoke.

---

### Impact Explanation

An invalid transaction carrying a forged signature is admitted to the gateway and mempool without signature verification. This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:
- The victim's first post-deploy invoke (nonce=1) is permanently displaced by the attacker's forged transaction.
- The victim must resubmit with nonce=2 (or higher), disrupting the expected deploy+invoke UX flow.
- The reverted attacker transaction charges a small fee from the victim's newly-deployed account balance.

---

### Likelihood Explanation

Medium. The attacker must monitor the public mempool for pending `deploy_account` transactions and front-run the victim's invoke submission. No privileged access is required. The mempool is observable, and the race window is the time between the `deploy_account` being admitted and the victim submitting their invoke.

---

### Recommendation

`skip_stateful_validations` should verify that the transaction present in the mempool for the sender address is specifically a `deploy_account` transaction (i.e., `tx_type == DeployAccount`), not merely any transaction. The mempool API should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type, rather than relying on the generic `account_tx_in_pool_or_recent_block` check. Alternatively, the skip should only fire when the mempool explicitly confirms a `deploy_account` with `nonce=0` is present for the sender.

---

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
