### Title
Missing Zero-Address Validation on `reward_address` Enables Permanent Freezing of Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

`change_reward_address` in `src/pool/pool.cairo` and `enter_delegation_pool` in the same file accept a `reward_address` parameter without validating it is non-zero. A pool member can set their `reward_address` to `0x0`, which passes all existing guards. When `claim_rewards` is subsequently called, accumulated STRK rewards are transferred to address zero and are permanently lost.

The same root cause exists in `src/staking/staking.cairo` for both `stake` and `change_reward_address`.

---

### Finding Description

**Vulnerability class**: Missing zero-address validation on a critical address parameter.

In `pool/pool.cairo`, `change_reward_address` performs only one guard:

```
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [1](#0-0) 

There is no `assert!(reward_address.is_non_zero(), ...)` check. Because `token_address` is a real deployed contract address, `token_address != 0x0` is always true, so passing `reward_address = 0x0` clears the only guard and the zero address is written to storage.

`enter_delegation_pool` has the identical gap — it only checks `token_address != reward_address`:

```
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [2](#0-1) 

In `staking/staking.cairo`, `change_reward_address` only checks `!does_token_exist(reward_address)`, which returns `false` for address zero (zero is not a registered token), so zero passes: [3](#0-2) 

The same gap exists in `stake`: [4](#0-3) 

Once `reward_address` is zero, `claim_rewards` in `pool/pool.cairo` unconditionally transfers accumulated STRK to it:

```
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [5](#0-4) 

The STRK ERC-20 `checked_transfer` to address zero does not revert on Starknet — it succeeds and the tokens are burned/lost.

The errors module defines `ZERO_ADDRESS` but it is never applied to `reward_address` in these entry points: [6](#0-5) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Any accumulated STRK rewards that are claimed while `reward_address` is zero are transferred to address zero and are irrecoverable. The pool member's entire unclaimed reward balance (including `_unclaimed_rewards_from_v0` carried over from V0) is destroyed in a single `claim_rewards` call. [7](#0-6) 

The same applies to a staker's `unclaimed_rewards_own` in `staking.cairo` via `send_rewards_to_staker`. [8](#0-7) 

---

### Likelihood Explanation

**Medium.**

The entry path is fully unprivileged — any pool member or staker can call `change_reward_address` or `enter_delegation_pool` / `stake` with `reward_address = 0x0`. No special role or key is required. The scenario is reachable through:

1. User error (e.g., passing an uninitialized address variable).
2. A front-end bug that submits a zero address.
3. An automated keeper or bot that calls `claim_rewards` on behalf of a pool member whose reward address was accidentally zeroed.

The pool member can in principle fix the address before claiming, but the window between the bad `change_reward_address` call and a subsequent `claim_rewards` (which any caller matching `pool_member || reward_address` can trigger) is a realistic loss scenario. [9](#0-8) 

---

### Recommendation

Add an explicit non-zero assertion for `reward_address` in all four affected entry points:

- `pool/pool.cairo` → `change_reward_address` (line 505)
- `pool/pool.cairo` → `enter_delegation_pool` (line 182)
- `staking/staking.cairo` → `change_reward_address` (line 517)
- `staking/staking.cairo` → `stake` (line 288)

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

The `ZERO_ADDRESS` error variant already exists in `GenericError` and is used elsewhere in the codebase (e.g., `add_token`), so this is a one-line fix per site. [6](#0-5) 

---

### Proof of Concept

1. Pool member calls `enter_delegation_pool(reward_address: 0x0, amount: X)` — passes all checks because `token_address != 0x0`.
2. Rewards accumulate over several epochs via `update_rewards_from_staking_contract`.
3. Pool member (or any caller equal to `pool_member`) calls `claim_rewards(pool_member)`.
4. `claim_rewards` reads `reward_address = 0x0` from storage and executes `checked_transfer(recipient: 0x0, amount: rewards)`.
5. STRK tokens are transferred to address zero and permanently lost.

Alternatively, an existing pool member with a valid reward address calls `change_reward_address(0x0)` — the single guard `token_address != 0x0` passes — and the same outcome follows on the next `claim_rewards` call. [10](#0-9) [11](#0-10) [12](#0-11)
