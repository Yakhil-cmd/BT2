### Title
Staker Can Instantly Raise Commission to Maximum via Same-Block Commitment + `set_commission`, Stealing Delegator Yield - (File: src/staking/staking.cairo)

### Summary
A staker can atomically set a `commission_commitment` with `max_commission = 10000` (100%) and immediately call `set_commission(10000)` in the same block. Because the commitment becomes active the moment it is written and `set_commission` only checks `commission <= max_commission` while the commitment is active, the commission jumps from any low value to 100% with no delay. Delegators cannot exit before the change takes effect; they must wait through the full `exit_wait_window` (up to 12 weeks) while all pool rewards flow to the staker.

### Finding Description
The `set_commission_commitment` function writes the commitment to storage and returns immediately. The commitment is active as soon as it is stored. In the same block (or the very next transaction), the staker calls `set_commission` with any value ≤ `max_commission`. The `update_commission` internal function checks only `commission <= commission_commitment.max_commission` and writes the new commission to storage with no further delay.

The code itself acknowledges this gap at line 745–746:

> **Note**: Current commission increase safeguards still allow for sudden commission changes.

The relevant code path:

1. `set_commission_commitment` stores the commitment immediately with no activation delay.
2. `set_commission` → `update_commission` writes the new commission to storage in the same call.
3. The next `update_rewards_from_attestation_contract` call uses the new commission, routing all pool rewards to the staker.

Delegators who joined under a low-commission pool have no on-chain mechanism to exit before the change takes effect; `exit_delegation_pool_intent` starts a timer equal to `exit_wait_window` (default 1 week, max 12 weeks), during which all rewards accrue at the new 100% commission rate.

### Impact Explanation
All pool rewards earned by delegators from the moment of the commission change until they complete the exit window are redirected to the staker. For a pool with significant delegated stake and a multi-week exit window, this constitutes a material theft of unclaimed yield from every delegator in the pool. The delegators' principal is recoverable, but the yield accrued during the exit window is permanently lost to them.

**Impact class**: High — Theft of unclaimed yield.

### Likelihood Explanation
Any staker who has attracted delegators can execute this in two sequential transactions in the same block. No external conditions, oracle manipulation, or third-party compromise are required. The only prerequisite is that the staker does not already have an active commitment, which is trivially satisfied for any staker who has never set one. The attack is economically rational whenever the value of captured yield exceeds the reputational cost of losing delegators.

### Recommendation
Introduce a mandatory activation delay between setting a `commission_commitment` and being permitted to raise commission under it. Specifically:

- Require that `set_commission` with an upward change may only be called at least `K` epochs (or one `exit_wait_window`) after the commitment was set, giving delegators time to observe the commitment on-chain and exit before the higher commission takes effect.
- Alternatively, make commission increases take effect only at `current_epoch + K`, not immediately, mirroring the epoch-delayed pattern already used for staking-power changes.

### Proof of Concept

```
// Epoch N, block B:
// Tx 1 – staker sets commitment allowing 100% commission
staking.set_commission_commitment(max_commission: 10000, expiration_epoch: N + 1);

// Tx 2 – same block, staker immediately raises commission to 100%
staking.set_commission(commission: 10000);

// Commission is now 10000/10000 = 100%, effective immediately.
// Next attestation in epoch N routes ALL pool rewards to the staker.
// Delegators must call exit_delegation_pool_intent and wait exit_wait_window
// (≥ 1 week) before recovering their principal; all yield during that window
// is captured by the staker.
```
