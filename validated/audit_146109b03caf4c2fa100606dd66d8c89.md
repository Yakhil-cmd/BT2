### Title
Gateway Stateful Validator Uses Stale Previous-Block L2 Gas Price for Resource Bounds Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the L2 gas price from the **previous block's** `BlockInfo` (the price that was in effect for the already-committed block), while the block being built will execute at the **next** L2 gas price (the EIP-1559-adjusted value stored as `next_l2_gas_price` in the previous block's header). A transaction whose `l2_gas.max_price_per_unit` falls in the gap between `threshold × previous_block_price` and `next_block_price` passes gateway admission and enters the mempool, but is rejected by the blockifier during actual batcher execution with `MaxGasPriceTooLow`.

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
```

The TODO comment explicitly acknowledges that the code should read `next_l2_gas_price` from the block header instead of `l2_gas_price`. The field `next_l2_gas_price` is already present in `BlockHeaderWithoutHash` (confirmed by its use in sync tests) and represents the EIP-1559-adjusted price for the block currently being built. [1](#0-0) 

The same stale price propagates into `run_validate_entry_point`, which constructs the `BlockContext` for blockifier validation by taking the previous block's `BlockInfo` and only incrementing `block_number`, leaving all gas prices at their previous-block values:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [2](#0-1) 

The blockifier's `check_fee_bounds` (called inside `perform_pre_validation_stage`) then checks `resource_bounds.max_price_per_unit >= block_info.gas_prices.gas_price_vector(fee_type).l2_gas_price`, using the same stale previous-block price: [3](#0-2) 

Both gateway checks therefore pass for a transaction whose `l2_gas.max_price_per_unit` is above `previous_block_l2_gas_price` but below `next_block_l2_gas_price`. When the batcher executes the transaction, it uses the actual next-block gas prices set by the consensus orchestrator, and `check_fee_bounds` fails with `MaxGasPriceTooLow`, causing the transaction to revert.

The `next_l2_gas_price` is already computed and stored in the block header (used by the sync path): [4](#0-3) 

### Impact Explanation

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**. The gateway admits transactions that will deterministically fail during batcher execution due to an insufficient gas price. An attacker who observes that the current block is above the EIP-1559 gas target (causing `next_l2_gas_price > l2_gas_price`) can craft transactions priced in the gap, flood the mempool with entries that consume mempool capacity and batcher execution slots, and cause nonce increments without fee payment (since `handle_nonce` runs before `check_fee_bounds` in `perform_pre_validation_stage`, incrementing the nonce even when the fee check subsequently fails).
