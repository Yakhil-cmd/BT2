Looking at the code carefully across the full call chain.

**Step 1: Identify the target and caller**

`set_state_header` is called from `StateHeaderValidationRequest` handler in `client_actor.rs`, which is triggered by the state sync downloader after downloading a header from a network peer. [1](#0-0) [2](#0-1) 

**Step 2: Trace the Merkle proof check — the critical gap**

In `set_state_header`, the chunk inclusion check is:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { ... }
``` [3](#0-2) 

`verify_path` only checks that the item hashes to the root via the provided path — it does **not** check which position/index in the tree the item occupies: [4](#0-3) 

The `chunk_headers_root` is a Merkle root over all shards' chunks ordered by shard index. Shard X's chunk at position X has a valid Merkle proof that reconstructs the root. That proof passes `verify_path` even when the caller claims `shard_id=Y`, because the function only checks path→root consistency, not position.

**Step 3: No shard_id cross-check exists**

`validate_chunk_proofs` checks internal chunk consistency (hash, tx root, receipts root) but never checks `chunk.shard_id()`: [5](#0-4) 

There is no line anywhere in `set_state_header` that asserts `chunk.shard_id() == shard_id`. The header is stored keyed by the parameter `shard_id`, not by the chunk's embedded shard_id: [6](#0-5) 

**Step 4: Downstream corruption in `apply_state_part`**

`set_state_part` reads the stored header and extracts `state_root` from the chunk's inner header — which is shard X's `prev_state_root`: [7](#0-6) 

`apply_state_part` then derives `shard_uid` from the parameter `shard_id=Y` and writes trie nodes under shard Y's `ShardUId`-prefixed namespace: [8](#0-7) 

Shard X's trie nodes (account balances, contract code, access keys, delayed receipts) are written into shard Y's trie and flat storage namespace.

**Step 5: Receipt proof check is not a blocker**

The receipt proof check at step 4e uses the parameter `shard_id` (Y) to hash receipts, and the attacker can supply valid on-chain receipt proofs for shard Y (publicly available). The `state_root_node` validation (step 5) checks against `chunk_inner.prev_state_root()` — shard X's root — which the attacker can satisfy with a valid node for shard X's trie. [9](#0-8)

### Title
Missing shard_id-vs-chunk.shard_id() cross-check in `set_state_header` allows cross-shard trie namespace corruption — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to shard X while the `shard_id` parameter is Y. The Merkle inclusion proof is verified only for path→root consistency, not for the chunk's position in the tree. No code checks `chunk.shard_id() == shard_id`. The header is stored under `StateHeaderKey(Y, sync_hash)` with shard X's `prev_state_root`, causing subsequent `set_state_part` and `apply_state_part` calls to write shard X's trie nodes into shard Y's `ShardUId`-prefixed trie and flat storage namespace.

### Finding Description
In `set_state_header` (`chain/chain/src/state_sync/adapter.rs`, lines 368–532):

1. `validate_chunk_proofs` verifies internal chunk consistency but never checks `chunk.shard_id()` against the `shard_id` parameter.
2. The `verify_path` call at line 394 verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is included in `sync_prev_block_header.chunk_headers_root()`. Because `verify_path` only checks path→root consistency (not position), shard X's chunk with its own valid Merkle proof at position X passes this check even when `shard_id=Y`.
3. No subsequent guard checks `chunk.shard_id() == shard_id`.
4. The header is stored as `StateHeaderKey(shard_id=Y, sync_hash)` with the embedded chunk from shard X.
5. `set_state_part` extracts `state_root` from the stored chunk (shard X's root) and validates parts against it.
6. `apply_state_part` derives `shard_uid` from `shard_id=Y` and writes all trie changes under shard Y's namespace.

### Impact Explanation
A syncing node that accepts a crafted state header has shard Y's trie and flat storage populated with shard X's account subtree, contract code, access keys, delayed receipt queue, and buffered receipt data. The node's post-sync state root for shard Y equals shard X's `prev_state_root` rather than the correct value. Any subsequent block application, view query, or proof generation for shard Y operates on the wrong state, violating the shard-namespace isolation invariant.

### Likelihood Explanation
Any node can register as a snapshot host and serve state headers. A syncing node selects snapshot hosts from the network and sends `StateRequestHeader`; a malicious peer responds with a crafted `VersionedStateResponse` containing shard X's chunk under `shard_id=Y`. No validator or operator privilege is required. The attack requires running a NEAR peer node, which is permissionless.

### Recommendation
Add an explicit check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, consider using `verify_path_with_index` (which exists in the codebase at `core/primitives/src/merkle.rs` line 121) to verify that the chunk's Merkle proof corresponds to the correct shard index position in the block's chunk list, providing defense in depth. [10](#0-9) 

### Proof of Concept
1. Build a two-shard `TestEnv`.
2. Produce several blocks so both shards have non-empty state.
3. On client[0], call `get_state_response_header(shard_id=0, sync_hash)` to obtain a valid header for shard 0.
4. On client[1] (the syncing node), call `set_state_header(shard_id=1, sync_hash, header_for_shard_0)`.
5. Assert the call returns `Ok(())` (it will, because shard 0's chunk IS in the block's Merkle tree).
6. Read back the stored header via `get_state_header(shard_id=1, sync_hash)` and assert `header.chunk_prev_state_root() == shard_0_state_root`.
7. Apply state parts for shard 0 under `shard_id=1` and call `set_state_finalize(shard_id=1, sync_hash)`.
8. Assert that the resulting trie state root for shard 1 equals shard 0's state root rather than shard 1's.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
