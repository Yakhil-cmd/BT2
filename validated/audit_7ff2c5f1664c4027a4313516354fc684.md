### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions with Nonce=1 from Undeployed Accounts - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point (which performs account signature verification) for any invoke transaction with `nonce=1` from an undeployed account that has any transaction in the mempool. An attacker who submits a valid `deploy_account` transaction can immediately follow it with an invoke carrying an invalid or forged signature, and the gateway will admit that invoke to the mempool without ever calling `__validate__`.

### Finding Description

The call chain in `extract_state_nonce_and_run_validations` is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range + resource bounds)
       ├─ validate_by_mempool            (nonce-too-old check only)
       └─ skip_stateful_validations      ← returns true when conditions met
  └─ run_validate_entry_point(skip_validate=true)
       └─ ExecutionFlags { validate: false }  ← __validate__ is NOT called
```

`skip_stateful_validations` returns `true` when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == 1`.
3. `account_nonce == 0` (account not yet deployed on-chain).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false }`, which causes `StatefulValidator::perform_validations` to return immediately after `perform_pre_validation_stage` without ever calling `tx.validate_tx(...)` (the `__validate__` entry point): [2](#0-1) [3](#0-2) 

Neither `validate_state_preconditions` nor `validate_by_mempool` checks the cryptographic signature: [4](#0-3) [5](#0-4) 

The stateless validator only checks signature *length*, not validity: [6](#0-5) 

The `account_tx_in_pool_or_recent_block` check only verifies that the account address appears in the mempool pool or committed state — it does not verify that the pooled transaction is specifically a `deploy_account`, nor does it verify the nonce=1 invoke's signature: [7](#0-6) 

### Impact Explanation

The gateway admits an invoke transaction with an invalid (e.g., all-zero) signature to the mempool. This is an unauthorized transaction: the account's `__validate__` entry point would reject it, but it is never called at admission time. The transaction will eventually fail/revert when the batcher executes it, but it has already been accepted as a valid pending transaction. This directly satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The attacker must:
1. Compute a fresh account address (deterministic from `class_hash`, `salt`, `constructor_calldata`).
2. Pre-fund that address with enough STRK to cover the `deploy_account` fee.
3. Submit a valid `deploy_account` (valid signature required — the attacker controls the key).
4. Immediately submit an invoke with `nonce=1` and an arbitrary/invalid signature.

Step 4 costs nothing beyond the `deploy_account` fee already paid. The attacker can repeat this across many fresh addresses to flood the mempool with signature-invalid invokes. The design comment acknowledges the skip is intentional for UX but does not account for adversarial use: [8](#0-7) 

### Recommendation

Move the `__validate__` call outside the `skip_validate` guard, or replace the skip with a weaker form that still verifies the signature. Concretely, `run_validate_entry_point` should always set `validate: true`; the UX goal (accepting the first invoke before the deploy_account is mined) can be preserved by relaxing only the *nonce strictness* (`strict_nonce_check = false`) while keeping signature validation active. Alternatively, mirror the approach in `PyValidator::should_run_stateful_validations`, which gates the skip on a caller-supplied `deploy_account_tx_hash` that is verified out-of-band, rather than a mempool presence check that any attacker can satisfy: [9](#0-8) 

### Proof of Concept

```
1. Attacker picks class_hash C, salt S, calldata D → derives address A.
2. Attacker funds address A with enough STRK for deploy_account fees.
3. Attacker submits deploy_account(class_hash=C, salt=S, calldata=D, nonce=0)
   with a VALID signature → passes full __validate_deploy__ check → enters mempool.
4. Attacker submits invoke(sender=A, nonce=1, calldata=..., signature=[0x0, 0x0])
   with an INVALID signature.
5. Gateway stateful validator:
     account_nonce = get_nonce(A) = 0          ← account not yet on-chain
     validate_state_preconditions: nonce 1 ∈ [0, max_gap] → OK
     validate_by_mempool: nonce 1 ≥ 0 → OK
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → true
       account_tx_in_pool_or_recent_block(A) → true (deploy_account is there)
       → returns true (skip)
     run_validate_entry_point(skip_validate=true):
       ExecutionFlags { validate: false }
       StatefulValidator::perform_validations → returns after perform_pre_validation_stage
       __validate__ is NEVER called
6. Invoke with invalid signature is added to the mempool.
7. Batcher executes deploy_account (succeeds), then invoke → __validate__ fails → revert.
   Invalid transaction was admitted and consumed sequencer execution resources.
``` [10](#0-9)

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
