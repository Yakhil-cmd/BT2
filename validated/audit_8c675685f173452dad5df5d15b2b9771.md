### Title
Silent Failure in `exit_to_near_precompile_callback` Permanently Burns User wNEAR ERC-20 Tokens on Failed `near_withdraw` Promise ŌĆö (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile's wNEAR-unwrap path (`near_withdraw` + callback) attaches a `exit_to_near_precompile_callback` because `transfer_near` is `Some`, but if the `near_withdraw` promise fails, the callback silently falls into an `else { None }` branch and performs no refund. The user's EVM-side wNEAR ERC-20 tokens have already been burned before the promise was created, so they are permanently lost. The underlying NEAR-side wNEAR remains locked in the Aurora contract with no recovery path.

---

### Finding Description

**Step 1 ŌĆö ERC-20 tokens are burned before the promise is dispatched.**

In `exit_erc20_token_to_near` (`engine-precompiles/src/native.rs`), when the caller uses the `:unwrap` suffix on the wNEAR ERC-20 address, the function returns `method = "near_withdraw"` and `transfer_near_args = Some(TransferNearArgs { ... })`. [1](#0-0) 

The ERC-20 burn happens inside the EVM execution that calls the precompile. By the time the precompile returns its `PrecompileOutput`, the EVM state has already committed the burn.

**Step 2 ŌĆö `callback_args` is non-default only because `transfer_near` is `Some`; `refund` is always `None` without the feature.**

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,                          // ŌåÉ always None without the feature
    transfer_near: transfer_near_args,     // ŌåÉ Some(...) for the wNEAR path
};
``` [2](#0-1) 

Because `transfer_near` is `Some`, `callback_args != ExitToNearPrecompileCallbackArgs::default()`, so a callback **is** attached: [3](#0-2) 

**Step 3 ŌĆö The callback silently ignores promise failure when `refund` is `None`.**

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success: transfer NEAR
    ...
    None
} else if let Some(args) = args.refund {
    // failure with refund args: re-mint ERC-20
    ...
} else {
    None   // ŌåÉ reached when near_withdraw fails AND refund is None
};
``` [4](#0-3) 

When `near_withdraw` fails (e.g., gas exhaustion with the 10 TGas budget, or contract rejection) and `error_refund` is not compiled in, execution reaches the final `else { None }` branch. No re-mint occurs, no error is raised, and `Ok(None)` is returned. The burned EVM-side wNEAR tokens are gone permanently. [5](#0-4) 

**Step 4 ŌĆö The `error_refund` feature is not unconditionally enabled.**

The feature changes the wire format of the precompile input (adds 20 bytes for `refund_address`), making it an opt-in compile-time choice: [6](#0-5) 

The integration tests explicitly acknowledge that without the feature, tokens are not refunded: [7](#0-6) 

---

### Impact Explanation

**Critical ŌĆö Permanent freezing of funds.**

A user who calls the `ExitToNear` precompile with wNEAR ERC-20 and the `:unwrap` suffix loses their entire bridged wNEAR balance if the `near_withdraw` NEAR promise fails. The EVM-side ERC-20 tokens are burned before the promise is dispatched; the NEAR-side wNEAR remains locked in the Aurora contract with no mechanism to recover it. There is no admin escape hatch for individual user balances.

---

### Likelihood Explanation

**Medium.** The `near_withdraw` promise is dispatched with only `FT_TRANSFER_GAS = 10_000_000_000_000` (10 TGas). [8](#0-7) 

If the wNEAR contract's `near_withdraw` method consumes more than 10 TGas (e.g., under congestion or after a contract upgrade), the promise fails and the silent-failure path is triggered. Any EVM user who calls the wNEAR exit precompile is exposed. The `error_refund` feature being absent from the build is the enabling condition.

---

### Recommendation

1. **Unconditionally attach a refund callback for the wNEAR unwrap path.** The `transfer_near` path already attaches a callback; extend it to also carry `refund` args regardless of the `error_refund` feature flag, so that `near_withdraw` failures re-mint the burned ERC-20 tokens.

2. **In `exit_to_near_precompile_callback`, treat the `else { None }` branch as an error** when `transfer_near` was set (i.e., the wNEAR path), rather than silently returning `Ok(None)`. A failed `near_withdraw` with no refund path should panic or return an explicit error so the NEAR runtime can surface the failure.

3. **Audit the 10 TGas budget** for `near_withdraw` against real-world wNEAR contract gas consumption to ensure it is sufficient.

---

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature.
2. Bridge wNEAR into Aurora as an ERC-20.
3. Call the `ExitToNear` precompile with flag `0x1`, amount, and recipient `"some.near:unwrap"`.
4. The EVM burns the wNEAR ERC-20 tokens and schedules `near_withdraw` with 10 TGas.
5. Arrange for `near_withdraw` to fail (e.g., by providing insufficient total gas to the outer `submit` call so the inner promise runs out of gas).
6. `exit_to_near_precompile_callback` executes: `promise_result(0)` is `Failed`, `args.refund` is `None` ŌåÆ falls into `else { None }` ŌåÆ returns `Ok(None)`.
7. Observe: user's EVM wNEAR balance is zero; NEAR-side wNEAR is still held by the Aurora contract; no re-mint occurred.

The burned tokens are unrecoverable.
