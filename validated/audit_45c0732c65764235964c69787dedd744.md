Looking at the actual code, I need to trace the full call chain to assess whether this is a real, exploitable vulnerability.

**Step 1: `handleCounterOffer` in `OfferIncomingTable.tsx`** [1](#0-0) 

The `puzzleHash` is read directly from `notification.puzzleHash` with no validation against the actual offer maker's on-chain identity, then immediately converted to a bech32m address and passed into router state.

**Step 2: Type definition — who controls `puzzleHash`?** [2](#0-1) 

`NotificationCounterOffer.puzzleHash` is a plain `string` with no schema enforcement. Notifications can originate from `'BLOCKCHAIN'` (anyone can broadcast a Chia notification coin to any puzzle hash), so an attacker can craft a `COUNTER_OFFER` notification with an arbitrary `puzzleHash`.

**Step 3: Validation in `useValidNotifications`** [3](#0-2) 

The only filter is `offerState.offer?.valid` — it checks that the embedded offer file is a valid Chia offer, **not** that `puzzleHash` matches the offer maker's actual puzzle hash. A crafted notification with a valid offer file but a malicious `puzzleHash` passes this filter.

**Step 4: Counter button enablement** [4](#0-3) 

The Counter button is enabled whenever `puzzleHash` is truthy and the offer is not a DataLayer offer — no further checks.

**Step 5: Address flows into `CreateOfferBuilder` and `onOfferCreated`** [5](#0-4) 

The attacker-derived `address` is forwarded to `onOfferCreated`, which is `handleOfferCreated` in `OfferManager.tsx`. [6](#0-5) 

`handleOfferCreated` converts `ph` (already a bech32m string from `handleCounterOffer`) through `toBech32m` again and passes it to `OfferShareDialog` as `address`. This `address` is the destination to which the counter-offer notification is sent — i.e., who gets notified and can accept the counter-offer.

**Assessment**

The described path is **concrete and code-confirmed**:

- Attacker broadcasts a `COUNTER_OFFER` blockchain notification to the victim's wallet containing a valid offer file but with `puzzleHash` set to the attacker's own puzzle hash.
- `useValidNotifications` passes it through (offer is valid).
- Victim clicks Counter; `handleCounterOffer` converts the attacker's `puzzleHash` to a bech32m address with no cross-check against the offer's actual maker.
- The counter-offer notification is sent to the attacker's address instead of the legitimate offer maker's address.
- The attacker receives the counter-offer, can accept it (if they hold the requested assets), and the legitimate offer maker is never notified.

The claim that assets are "sent to attacker's address" is slightly imprecise — in Chia's offer model, assets are locked in the offer file, not transferred at creation time. The actual impact is that the attacker receives the counter-offer notification and can accept the counter-offer, while the legitimate maker is silently excluded. This fits the High scope rule: *unsafe trust of notification state that causes a user to send the wrong destination*.

### Title
Attacker-Controlled `puzzleHash` in Incoming Offer Notification Redirects Counter-Offer to Attacker — (`packages/gui/src/components/offers2/OfferIncomingTable.tsx`)

### Summary
`handleCounterOffer` derives the counter-offer destination address directly from `notification.puzzleHash` without validating it against the actual offer maker's identity. An attacker who can broadcast a blockchain notification (a standard Chia capability) can supply an arbitrary puzzle hash, causing the victim's counter-offer notification to be delivered to the attacker instead of the legitimate offer maker.

### Finding Description
In `OfferIncomingTable.tsx` `handleCounterOffer` (lines 177–206), `notification.puzzleHash` is read and converted to a bech32m address via `toBech32m`, then stored in router state as `address`. This value flows through `CreateOfferBuilder` → `onOfferCreated` → `handleOfferCreated` → `OfferShareDialog`, where it determines the recipient of the counter-offer notification. No step validates that `puzzleHash` matches the cryptographic identity of the offer maker embedded in the offer file. `useValidNotifications` only checks `offer.valid`, not the `puzzleHash` field.

### Impact Explanation
The attacker receives the counter-offer notification and can accept the counter-offer. The legitimate offer maker is never notified. If the attacker holds the assets the victim requested, they can complete the trade on their own terms. At minimum, the victim's counter-offer is silently misdirected, preventing the intended trade.

### Likelihood Explanation
Chia's blockchain notification system allows anyone to send a notification coin to any puzzle hash. Crafting a `COUNTER_OFFER` notification with a valid offer file (copied from a real offer) and a malicious `puzzleHash` requires no special privileges. The victim only needs to click the Counter button, which is a normal user action.

### Recommendation
In `handleCounterOffer`, derive the counter-offer destination address from the offer file's cryptographic maker identity (e.g., from `offerSummary` or the offer's coin spend conditions), not from the notification's `puzzleHash` field. If `puzzleHash` must be used, validate it against the maker's puzzle hash extracted from the offer file before using it.

### Proof of Concept
1. Obtain any valid Chia offer file (or URL).
2. Broadcast a blockchain notification to the victim's wallet of type `COUNTER_OFFER`, embedding the valid offer file and setting `puzzleHash` to the attacker's own puzzle hash.
3. Victim opens the GUI; the notification appears in the Incoming Offers table with the Counter button enabled (offer is valid, `puzzleHash` is truthy).
4. Victim clicks Counter; `handleCounterOffer` converts the attacker's `puzzleHash` to a bech32m address and navigates to the offer builder with `address = attacker_bech32m_address`.
5. Victim completes and submits the counter-offer; `OfferShareDialog` sends the counter-offer notification to the attacker's address.
6. Assert: the attacker's wallet receives the counter-offer notification; the legitimate offer maker receives nothing.
