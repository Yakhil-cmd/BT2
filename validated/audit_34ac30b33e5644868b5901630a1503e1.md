### Title
Malicious Coordinator Can Present Different `(msg_hash, tweak)` to Different Participants in Robust ECDSA Signing, Enabling Secret Key Extraction — (`src/ecdsa/robust_ecdsa/sign.rs`)

### Summary

The Robust ECDSA signing protocol (`src/ecdsa/robust_ecdsa/sign.rs`) contains no in-protocol step where participants broadcast and reach consensus on the `(msg_hash, tweak, participants)` used for presignature rerandomization before submitting their signature shares. A malicious coordinator can present different rerandomization contexts to different honest participants, causing them to produce signature shares over different effective nonces derived from the same raw presignature. This is a textbook ECDSA nonce-reuse scenario that allows full recovery of the aggregate private signing key.

### Finding Description

**Root cause and code path:**

The Robust ECDSA signing flow is split into two caller-side steps before the protocol runs:

1. Each participant independently calls `RerandomizedPresignOutput::rerandomize_presign(presignature, &args)` in `src/ecdsa/robust_ecdsa/mod.rs` (lines 55–86), where `args` is a `RerandomizationArguments` struct containing `msg_hash`, `tweak`, `participants`, and `entropy`. This produces a participant-local `RerandomizedPresignOutput` whose `alpha`, `beta`, and `big_r` fields are deterministically derived from those inputs.

2. Each participant then calls `sign(participants, coordinator, max_malicious, me, public_key, presignature, msg_hash)` in `src/ecdsa/robust_ecdsa/sign.rs` (lines 33–108), passing in their locally-computed `RerandomizedPresignOutput` and their local `msg_hash`.

Inside the protocol, `do_sign_participant` (lines 111–124) immediately computes its signature share and sends it to the coordinator:

```rust
fn do_sign_participant(...) -> Result<SignatureOption, ProtocolError> {
    let s_me = compute_signature_share(presignature, msg_hash, participants, me)?;
    let wait_round = chan.next_waitpoint();
    chan.send_private(wait_round, coordinator, &s_me)?;
    Ok(None)
}
```

There is **no round** in which participants broadcast their `(msg_hash, tweak, participants)` or their rerandomized `big_r` to each other and verify consistency before submitting shares. The protocol is a single-round: compute share, send to coordinator, done.

The security documentation at `docs/ecdsa/robust_ecdsa/signing.md` (lines 172–174) explicitly requires:

> "Ensure all participants agree on $(h, \epsilon)$ and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers."

But the library provides **zero protocol-level enforcement** of this requirement. The `sign()` function accepts any `RerandomizedPresignOutput` without verifying it is consistent with what other participants received.

**Exploit flow:**

A malicious coordinator orchestrates two signing sessions using the same raw `PresignOutput`:

- **Session A**: Coordinator tells participants $P_1, \ldots, P_{2t+1}$ to rerandomize with context $(h_A, \epsilon_A, \rho_A)$, producing rerandomized nonce $R_A = \delta_A \cdot R$.
- **Session B**: Coordinator tells the same participants to rerandomize with context $(h_B, \epsilon_B, \rho_B)$ where $(h_B, \epsilon_B) \neq (h_A, \epsilon_A)$, producing rerandomized nonce $R_B = \delta_B \cdot R$.

Since both $\delta_A$ and $\delta_B$ are derived from the same raw $R$ and the same secret polynomial shares, the resulting nonces are multiplicatively related: $R_B = (\delta_B / \delta_A) \cdot R_A$. The two resulting signatures $(R_A, s_A)$ and $(R_B, s_B)$ satisfy the standard ECDSA nonce-reuse relation, from which the private key $x$ can be recovered algebraically.

The documentation at `docs/ecdsa/robust_ecdsa/signing.md` (lines 156–158) confirms this:

> "a novel split-view attack exists that can extract the secret key using as few as $2t + 2$ presigning participants, with as few as two signing sessions."

**Contrast with FROST/EdDSA:** The EdDSA signing protocol at `docs/eddsa/signing.md` (line 64) includes an explicit step: "Each $P_i$ verifies that $m = m^*$" — participants receive the message from the coordinator and verify it matches their expectation. The Robust ECDSA signing protocol has no equivalent step.

### Impact Explanation

**Critical — Extraction, reconstruction, or disclosure of the aggregate private signing key.**

A malicious coordinator with no special privileges beyond being a designated participant can, across two signing sessions using the same presignature, recover the full aggregate private key $x$ shared among all honest participants. This completely breaks the threshold security guarantee: the attacker gains the ability to forge arbitrary signatures for any message without further participation from honest parties.

### Likelihood Explanation

The coordinator role is a normal participant role assigned per signing session. Any participant who is selected as coordinator (or who can influence coordinator selection) can execute this attack. The attack requires only two signing sessions with the same presignature, which is a realistic scenario given that presignatures are generated in advance and may be reused accidentally or under coordinator manipulation. The attack is algebraically straightforward once two inconsistent signature pairs are obtained.

### Recommendation

Add an explicit consensus round at the start of the signing protocol where each participant broadcasts a commitment to their `(msg_hash, tweak, participants, big_r_rerandomized)` tuple, and all participants verify that every other participant committed to the same values before computing and sending their signature share. This mirrors the message-verification step already present in the FROST/EdDSA protocol (`docs/eddsa/signing.md` line 64). Alternatively, require participants to broadcast their rerandomized `big_r` and verify it matches the expected value before proceeding.

### Proof of Concept

Setup: 5 participants, `max_malicious = 2`, so `N = 2t+1 = 5`. One presignature `P` with raw nonce `R` is generated.

**Session A:** Coordinator tells all 5 participants to call:
```rust
RerandomizedPresignOutput::rerandomize_presign(&P, &args_A)
// args_A: msg_hash = h_A, tweak = ε_A, entropy = ρ_A
```
Each participant computes `s_i^A` using `msg_hash = h_A` and sends to coordinator. Coordinator obtains signature `(R_A, s_A)`.

**Session B:** Coordinator tells all 5 participants to call:
```rust
RerandomizedPresignOutput::rerandomize_presign(&P, &args_B)
// args_B: msg_hash = h_B ≠ h_A, tweak = ε_B, entropy = ρ_B
```
Each participant computes `s_i^B` using `msg_hash = h_B` and sends to coordinator. Coordinator obtains signature `(R_B, s_B)`.

At no point does `do_sign_participant` in `src/ecdsa/robust_ecdsa/sign.rs` (lines 111–124) verify that `presignature.big_r` or `msg_hash` is consistent with what other participants received. The coordinator collects both `(R_A, s_A)` and `(R_B, s_B)`, which are ECDSA signatures with multiplicatively related nonces over the same key, and applies standard nonce-reuse key recovery to extract the private key $x$. pub fn sign(
```rust
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
```

```rust
impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
```

```markdown
# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.
```
