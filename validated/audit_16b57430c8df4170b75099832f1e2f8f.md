### Title
Unchecked `u32`-to-`u16` Truncation of `tx_number` in L2→L1 Log Serialization Corrupts Merkle Tree Root - (`File: zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

In `zk_ee/src/common_structs/logs_storage.rs`, when converting an internal `LogContent` record to the wire-format `L2ToL1Log`, the transaction number is silently truncated from `u32` to `u16` with no bounds check. If a block contains more than 65,535 transactions, every subsequent log will carry a wrong `tx_number_in_block`, corrupting the L2→L1 Merkle tree root that is committed to L1.

---

### Finding Description

`GenericLogContent` stores the transaction number as a `u32`:

```rust
pub struct GenericLogContent<IOTypes: SystemIOTypesConfig, A: Allocator = Global> {
    pub tx_number: u32,
    ...
}
``` [1](#0-0) 

The wire-format `L2ToL1Log` stores the same field as a `u16`:

```rust
pub struct L2ToL1Log {
    ...
    pub tx_number_in_block: u16,
    ...
}
``` [2](#0-1) 

The `From` conversion between the two types performs an unchecked narrowing cast:

```rust
tx_number_in_block: m.tx_number as u16,
``` [3](#0-2) 

Rust's `as` cast on integers silently truncates the high-order bits when the source value exceeds the target type's range. For any transaction with index ≥ 65,536 (`u16::MAX + 1`), the resulting `tx_number_in_block` wraps around to an incorrect value (e.g., transaction 65,536 → `tx_number_in_block = 0`, colliding with the first transaction in the block).

The corrupted field is then serialised directly into the 88-byte log encoding that feeds the Keccak-based Merkle tree:

```rust
buffer[2..4].copy_from_slice(&self.tx_number_in_block.to_be_bytes());
``` [4](#0-3) 

and is also written to pubdata:

```rust
result_keeper.pubdata(&self.tx_number_in_block.to_be_bytes());
``` [5](#0-4) 

The Merkle root computed over these hashes is what gets committed to L1. A wrong `tx_number_in_block` produces a wrong leaf hash, hence a wrong root.

Logs are pushed with the current block transaction number via `push_message` and `push_l1_l2_tx_log`, both of which accept `tx_number: u32`: [6](#0-5) 

The block-level counter `current_transaction_number` is incremented unconditionally for every successfully processed transaction:

```rust
block_data.current_transaction_number += 1;
``` [7](#0-6) 

There is no guard that prevents `current_transaction_number` from exceeding 65,535 before the truncation occurs.

---

### Impact Explanation

- **Incorrect L2→L1 Merkle root committed to L1.** Every log emitted by a transaction with index ≥ 65,536 carries a silently wrong `tx_number_in_block`. The Merkle tree root derived from these logs is therefore incorrect.
- **Log attribution collision.** Transactions 65,536 and 0 would produce logs with the same `tx_number_in_block = 0`, making it impossible to correctly attribute logs to their originating transactions on L1.
- **Pubdata integrity violation.** The pubdata written to L1 contains the wrong two-byte field for every affected log, breaking any L1-side proof or verification that relies on the log encoding.
- **State-transition / forward-proving divergence.** The forward (sequencer) execution and the proving execution both perform the same truncation, so they agree with each other but both commit an incorrect root to L1. L1 contracts that verify the Merkle root against expected transaction attribution will accept an invalid state.

---

### Likelihood Explanation

ZKsync OS explicitly targets 10,000 TPS. A block covering even a few seconds at that throughput can contain tens of thousands of transactions. The minimum gas cost per transaction (21,000 gas) combined with a sufficiently large block gas limit makes it feasible to include more than 65,535 transactions in a single block. An attacker can deliberately spam minimal-cost transactions (e.g., simple ETH transfers) to push the transaction counter past the `u16` boundary within a single block, triggering the truncation for any log-emitting transaction that follows.

---

### Recommendation

Replace the unchecked cast with a checked conversion and return an error (or panic in debug mode) if the transaction number exceeds `u16::MAX`:

```rust
tx_number_in_block: u16::try_from(m.tx_number)
    .expect("tx_number exceeds u16::MAX; block transaction limit must be enforced"),
```

Additionally, enforce a hard block-level cap on the number of transactions that is strictly less than 65,536, consistent with the `u16` field width used in the L2→L1 log format. Document this constraint explicitly alongside `MAX_NUMBER_OF_LOGS`. [8](#0-7) 

---

### Proof of Concept

1. Construct a block containing 65,537 transactions. The first 65,535 are simple ETH transfers (no L2→L1 logs). Transaction 65,536 is an L1→L2 priority transaction that emits an `L1TxLog`.
2. The bootloader increments `current_transaction_number` to 65,536 and calls `push_l1_l2_tx_log(65536, tx_hash, true)`.
3. In `From<&LogContent> for L2ToL1Log`, `m.tx_number as u16` evaluates to `0` (65,536 mod 65,536 = 0).
4. The log is serialised with `tx_number_in_block = 0` instead of `65536`.
5. The Merkle tree root is computed over this corrupted leaf and committed to L1.
6. On L1, any contract that checks which transaction a log belongs to will incorrectly attribute the log to transaction 0, not transaction 65,536. [9](#0-8) [10](#0-9)
