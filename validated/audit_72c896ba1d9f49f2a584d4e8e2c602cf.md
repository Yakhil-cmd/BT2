### Title
Unauthorized Token Transfer and Burn via Missing Caller Authorization in Bank Precompile — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `BankContract` precompile at address `0x0000...0064` exposes `transfer(address sender, address recipient, uint256 amount)` and `burn(address addr, uint256 amount)` methods that accept an arbitrary source/target address as a caller-supplied argument. Neither method verifies that the supplied address equals `contract.Caller()` or has otherwise authorized the operation. Any EVM contract can therefore transfer or burn `evm/<calling_contract>` Cosmos-native tokens from any victim address that holds that denom, without the victim's consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles two state-mutating paths that lack caller authorization:

**`transfer` path (lines 167–200):**

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`from` is derived entirely from the first ABI argument. There is no check that `sender == contract.Caller()`. The denom is `evm/<contract.Caller()>`, so the calling contract controls both the denom namespace and the source address of the transfer.

**`burn` path (lines 113–156):**

```go
recipient := args[0].(common.Address)  // for burn, this is the address burned FROM
...
addr := sdk.AccAddress(recipient.Bytes())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
```

`addr` is again fully caller-supplied. There is no check that `addr == contract.Caller()`. Any contract can burn `evm/<itself>` tokens from any address.

The only guard present is `checkBlockedAddr`, which only prevents sending to module-blocked addresses — it does not protect the *source* address. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

**Critical — Unauthorized transfer and burn of CRC20/EVM-native Cosmos tokens.**

The Cosmos SDK's `bankKeeper.SendCoins` and `SendCoinsFromAccountToModule` are privileged module-level calls that bypass EVM-side `approve`/`allowance` semantics. Once a user converts EVM-side tokens to Cosmos-native `evm/<contract>` tokens (e.g., via `moveToNative` as shown in `TestBank.sol`), those Cosmos-side balances are fully exposed to the calling contract with no consent mechanism. [3](#0-2) 

A malicious token contract can:
1. Drain all `evm/<attacker_contract>` Cosmos-native balances from any holder via `transfer(victim, attacker, balance)`.
2. Destroy all `evm/<attacker_contract>` Cosmos-native balances from any holder via `burn(victim, balance)`.

This directly maps to the allowed Critical impact: *unauthorized transfer/burn of CRC20/EVM-native assets*.

---

### Likelihood Explanation

**Medium-High.** The bank precompile is a documented, publicly accessible precompile at a fixed address (`0x0000...0064`). Any deployed EVM contract can call it. The attack requires victims to hold Cosmos-native `evm/<attacker_contract>` tokens, which occurs whenever users call `moveToNative`-style functions on a token contract. A malicious token issuer can exploit this immediately after users convert their balances. The integration test `test_bank_precompiles.py` confirms the `transfer` and `burn` flows are live and functional. [4](#0-3) 

---

### Recommendation

In both the `transfer` and `burn` cases, enforce that the source address equals `contract.Caller()`:

**`transfer`:** Add before `SendCoins`:
```go
if sender != contract.Caller() {
    return nil, errors.New("transfer sender must be the calling contract")
}
```

**`burn`:** Add before `SendCoinsFromAccountToModule`:
```go
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the calling contract")
}
```

This mirrors the fix applied to SPTV2's `mintAdvanced`: when acting on behalf of another address, require that the address is the caller itself (or provide an explicit authorization mechanism such as an EVM-side `approve` check before the Cosmos-side operation). [5](#0-4) 

---

### Proof of Concept

1. Deploy `MaliciousToken` at address `0xATK`. It exposes:
   ```solidity
   IBankModule bank = IBankModule(0x0000000000000000000000000000000000000064);

   function stealNative(address victim, uint256 amount) external {
       bank.transfer(victim, msg.sender, amount);
   }

   function destroyNative(address victim, uint256 amount) external {
       bank.burn(victim, amount);
   }
   ```

2. Victim calls `bank.mint(victim, 1000)` from `MaliciousToken` (or acquires `evm/0xATK` tokens via any conversion path), giving them 1000 `evm/0xATK` Cosmos-native tokens.

3. Attacker calls `MaliciousToken.stealNative(victim, 1000)`:
   - `bank.transfer(victim, attacker, 1000)` is dispatched to the precompile.
   - `contract.Caller()` = `0xATK`, so denom = `evm/0xATK`.
   - `from` = victim (no authorization check).
   - `bankKeeper.SendCoins(ctx, victim, attacker, 1000 evm/0xATK)` executes.
   - Victim's Cosmos-native balance is drained to zero; attacker receives 1000 `evm/0xATK`.

4. Alternatively, attacker calls `MaliciousToken.destroyNative(victim, 1000)`:
   - `bank.burn(victim, 1000)` executes `SendCoinsFromAccountToModule(victim, ...)` then `BurnCoins`.
   - Victim's 1000 `evm/0xATK` tokens are permanently destroyed. [6](#0-5) [5](#0-4)
