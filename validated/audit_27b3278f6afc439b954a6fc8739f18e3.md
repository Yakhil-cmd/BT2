### Title
Unsolicited StarkGate Deposits Inflate Balance and Suppress L1 Mint Requests — (`src/reward_supplier/reward_supplier.cairo`)

### Summary
`RewardSupplier.on_receive` accepts STRK deposits from any depositor via StarkGate and decrements `l1_pending_requested_amount` without crediting `unclaimed_rewards`. Because `request_funds` uses `credit = balance + l1_pending_requested_amount` to decide whether to send new L1 mint messages, an attacker who deposits STRK through StarkGate inflates `balance` while zeroing `l1_pending_requested_amount`, suppressing future mint requests. The deposited tokens can never be explicitly recovered (no sweep function exists), and the accounting divergence between `balance` and `unclaimed_rewards` persists until the inflated balance is gradually consumed by legitimate reward claims.

### Finding Description

`on_receive` is the StarkGate callback invoked whenever STRK is bridged to the `RewardSupplier`. The code explicitly notes that the depositor is not checked: [1](#0-0) 

The function only decrements `l1_pending_requested_amount`:

```
if amount_u128 > l1_pending_requested_amount {
    self.l1_pending_requested_amount.write(Zero::zero());
} else {
    l1_pending_requested_amount -= amount_u128;
    self.l1_pending_requested_amount.write(l1_pending_requested_amount);
}
```

It never touches `unclaimed_rewards`. The `request_funds` internal function computes: [2](#0-1) 

```
let credit = balance + l1_pending_requested_amount;
let debit  = unclaimed_rewards;
if credit < debit + threshold { /* send L1 mint messages */ }
```

An attacker who sends X STRK via StarkGate causes:
- `balance` to increase by X
- `l1_pending_requested_amount` to decrease by `min(X, l1_pending_requested_amount)`

Net effect: `credit` is unchanged or increases, while `debit` (`unclaimed_rewards`) is unchanged. `request_funds` therefore concludes that no new L1 mint messages are needed, even though the inflated balance consists of tokens that are not tracked by `unclaimed_rewards` and cannot be claimed through `claim_rewards`: [3](#0-2) 

`claim_rewards` enforces `amount <= unclaimed_rewards`, so the attacker's tokens are permanently inaccessible to any explicit claim path. They are consumed only indirectly as legitimate reward transfers draw down the shared balance.

The `NormalizedAmount` system and the `to_native_amount` conversion used elsewhere in the staking contract are not the root cause here; the issue is purely the missing `unclaimed_rewards` credit in `on_receive`. [4](#0-3) 

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users or protocol.**

1. **Suppressed L1 mint requests.** After a large unsolicited deposit, `credit ≥ debit + threshold` holds for many epochs. `request_funds` sends no new L1 messages during this window. The protocol relies on L1 minting for long-term reward supply; suppressing requests delays the replenishment pipeline.

2. **Permanent accounting divergence.** The deposited tokens are never credited to `unclaimed_rewards`. They are silently consumed by legitimate reward claims, but there is no mechanism to recover them or to reconcile the divergence. If the contract is paused before the balance is fully consumed, the excess tokens are frozen with no recovery path.

3. **`l1_pending_requested_amount` corruption.** Legitimate pending L1 requests are prematurely zeroed, breaking the invariant that `l1_pending_requested_amount` tracks only protocol-initiated mints.

### Likelihood Explanation

**Medium.** StarkGate is a public bridge; any address can initiate a deposit to any L2 contract address. No privileged role, leaked key, or external dependency compromise is required. The attacker only needs enough STRK to make `credit ≥ debit + threshold` hold for a meaningful number of epochs. The comment in the code ("the deposit can be done by anyone") confirms the developers are aware of this path but did not add a guard.

### Recommendation

In `on_receive`, add the received amount to `unclaimed_rewards` (or to a separate "excess balance" counter) so that `request_funds` always operates on accurate credit/debit figures. Alternatively, reject deposits whose `depositor` does not match the stored `l1_reward_supplier` address, which would close the unsolicited-deposit path entirely.

### Proof of Concept

1. Protocol state: `unclaimed_rewards = R`, `balance = 0`, `l1_pending_requested_amount = R` (a legitimate mint request is in flight).
2. Attacker calls StarkGate `deposit(recipient=reward_supplier, amount=2R)`.
3. StarkGate calls `reward_supplier.on_receive(l2_token=STRK, amount=2R, ...)`.
4. Inside `on_receive`: `2R > R` → `l1_pending_requested_amount` written to `0`; `balance` is now `2R`.
5. Next epoch: staking contract calls `update_unclaimed_rewards_from_staking_contract(rewards)` → `unclaimed_rewards = R + rewards` → `request_funds` is called.
6. `credit = 2R + 0 = 2R`; `debit = R + rewards ≈ R`; `credit ≥ debit + threshold` → **no L1 mint message sent**.
7. The legitimate L1 mint (already in flight) arrives: `on_receive` called again, `l1_pending_requested_amount` stays `0`, `balance = 3R`.
8. `claim_rewards(R)` succeeds (balance sufficient), `unclaimed_rewards = 0`, `balance = 2R`.
9. The attacker's `2R` tokens remain in the balance, inaccessible via any claim path, and continue to suppress mint requests for subsequent epochs until the balance is depleted. [5](#0-4) [2](#0-1) [6](#0-5)

            ref self: ContractState,
            l2_token: ContractAddress,
            amount: u256,
            depositor: EthAddress,
            message: Span<felt252>,
        ) -> bool {
            // Note that the deposit can be done by anyone (not just the L1 reward supplier), so

            // These messages accepted only from the token bridge.
            assert!(
                get_caller_address() == self.starkgate_address.read(),

            );
            // The bridge may serve multiple tokens, only the correct token may be received.

                "{}",
                Error::UNEXPECTED_TOKEN,
            );
            let amount_u128: Amount = amount
                .try_into()
                .expect_with_err(GenericError::AMOUNT_TOO_HIGH);
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            if amount_u128 > l1_pending_requested_amount {
                self.l1_pending_requested_amount.write(Zero::zero());
            } else {
                l1_pending_requested_amount -= amount_u128;
                self.l1_pending_requested_amount.write(l1_pending_requested_amount);
            }
            true
        }

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
