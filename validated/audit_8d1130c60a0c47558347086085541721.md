### Title
`ArrayVec` Overflow Panic in Multiblock Batch Log Accumulation Halts the Prover - (`basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/post_tx_op_proving_multiblock_batch.rs`)

---

### Summary

In the multiblock batch proving path, each block's L2→L1 logs are appended into a shared `ArrayVec<Bytes32, 16384>` via `LogsStorage::apply_to_array_vec`. The per-block log limit (`MAX_NUMBER_OF_LOGS = 16_384`) exactly equals the `ArrayVec` capacity. A user who fills one block with 16,384 L2→L1 messages saturates the batch-level array; any subsequent block in the same batch that emits even one log causes an unconditional `ArrayVec::push` panic, making the entire batch unprovable and halting the chain's ability to finalize state on L1.

---

### Finding Description

**Root cause — `apply_to_array_vec` uses the panicking `push` with no bounds guard:**

```rust
// zk_ee/src/common_structs/logs_storage.rs  lines 311-316
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.push(log.hash())   // panics when full
    });
}
``` [1](#0-0) 

This function is called once per block during multiblock-batch finalization:

```rust
// post_tx_op_proving_multiblock_batch.rs  lines 109-110
io.logs_storage
    .apply_to_array_vec(&mut batch_data.logs_storage);
``` [2](#0-1) 

The batch-level accumulator is declared with a fixed capacity of 16,384:

```rust
// batch_data.rs  line 27
pub logs_storage: ArrayVec<Bytes32, 16384>,
``` [3](#0-2) 

**The per-block limit exactly matches the array capacity.** The block-level guard is:

```rust
// zk/mod.rs  lines 84-90
} else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
    Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
``` [4](#0-3) 

`MAX_NUMBER_OF_LOGS = 16_384` (strictly greater-than check), so a block may legally contain exactly 16,384 logs. [5](#0-4) 

After one such block is appended to the batch accumulator, the `ArrayVec` is completely full. When the prover processes the next block and calls `apply_to_array_vec` again, `ArrayVec::push` panics unconditionally (this is the documented behavior of `arrayvec::ArrayVec::push` when at capacity). The panic propagates out of the RISC-V proving binary, making the batch unprovable.

**Attacker-controlled entry path:**

1. Attacker sends many transactions that each call the L1 Messenger system contract (`0x8008`), which calls the L1 messenger hook (`0x7001`) via `send_to_l1_inner`, which calls `system.io.emit_l1_message`, which calls `logs_storage.push_message`. [6](#0-5) [7](#0-6) 

2. The attacker fills one block with exactly 16,384 such messages, saturating the per-block limit.
3. The sequencer includes this block as the first block of a multiblock batch.
4. Any subsequent block in the same batch that emits even one log (e.g., any L1 priority transaction, which always appends a result log via `push_l1_l2_tx_log`) triggers the panic. [8](#0-7) 

---

### Impact Explanation

The panic occurs inside the RISC-V proving binary during `post_op` for a multiblock batch. The prover cannot recover: the batch is permanently unprovable. In a ZK rollup, unproven batches cannot be finalized on L1, so the chain's state commitment on the settlement layer stalls. This is a **complete DoS on batch finalization** for any multiblock batch that contains a saturated block followed by any block with logs.

---

### Likelihood Explanation

- The L1 Messenger hook is publicly callable by any user who can interact with the L1 Messenger system contract (`0x8008`), which is a standard user-facing contract.
- Filling a block with 16,384 L2→L1 messages requires paying gas for each message, but is economically feasible for a motivated attacker.
- The condition is deterministic: once a block with 16,384 logs is the first block of a multiblock batch, the next block with any log triggers the panic with certainty.
- L1 priority transactions always emit a result log, so the second block condition is trivially satisfied by any L1 transaction.

---

### Recommendation

1. **Immediate fix:** Replace `array_vec.push(log.hash())` with `array_vec.try_push(log.hash()).expect(...)` and add a pre-check, or use a dynamically sized structure for the batch-level log accumulator.
2. **Correct the capacity invariant:** The batch-level `ArrayVec` capacity must be `MAX_NUMBER_OF_LOGS * max_blocks_per_batch`, not `MAX_NUMBER_OF_LOGS`. Alternatively, enforce that the sum of logs across all blocks in a batch cannot exceed 16,384 at the sequencer level before batch construction.
3. **Use `try_push` and propagate an error** rather than panicking, so the prover can return a structured error instead of aborting.

---

### Proof of Concept

```
Block 1: attacker sends 16,384 transactions each calling L1Messenger.sendToL1(...)
  → logs_storage for block 1 has 16,384 entries
  → check_for_block_limits: logs_used == 16384, NOT > 16384, so block is accepted

Multiblock batch finalization (prover, RISC-V):
  Block 1 post_op:
    io.logs_storage.apply_to_array_vec(&mut batch_data.logs_storage)
    → pushes 16,384 hashes → ArrayVec is now FULL (len == capacity == 16384)

Block 2: any L1 priority transaction
  → push_l1_l2_tx_log adds 1 log to block 2's logs_storage

  Block 2 post_op:
    io.logs_storage.apply_to_array_vec(&mut batch_data.logs_storage)
    → first push: ArrayVec::push panics ("ArrayVec is full")
    → prover binary aborts
    → batch is permanently unprovable
    → chain finalization on L1 stalls
```

        //   previous step).

            keccak256_native_cost::<Self::Resources>(L2_TO_L1_LOG_SERIALIZE_SIZE).as_u64()
                + 3 * keccak256_native_cost::<Self::Resources>(64).as_u64()
                + keccak256_native_cost::<Self::Resources>(data.len()).as_u64();

            + EVENT_STORAGE_BASE_NATIVE_COST
            + EVENT_DATA_PER_BYTE_COST * (data.len() as u64);
