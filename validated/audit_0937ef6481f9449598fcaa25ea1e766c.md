### Title
`skip_stateful_validations` ignores `max_nonce_for_validation_skip` config, enabling signature-bypass admission when the skip feature is configured to be disabled â€” (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `StatefulTransactionValidatorConfig::max_nonce_for_validation_skip` field is defined, serialized in `config_schema.json`, documented, and used correctly in the Python-path `PyValidator`. However, the Rust gateway's `skip_stateful_validations` free function hardcodes `Nonce(Felt::ONE)` and never reads the config field. When an operator sets `max_nonce_for_validation_skip` to `0` to disable the UX skip feature, the gateway still bypasses `__validate__` for nonce-1 invoke transactions from undeployed accounts, admitting them without signature verification.

### Finding Description

`StatefulTransactionValidatorConfig` declares:

```rust
pub max_nonce_for_validation_skip: Nonce,  // default: Nonce(Felt::ONE)
``` [1](#0-0) 

This field is serialized in the production config schema: [2](#0-1) 

The Python-path `PyValidator::should_run_stateful_validations` correctly reads and applies this bound:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [3](#0-2) 

The Rust gateway's `skip_stateful_validations` free function, however, hardcodes the nonce check to exactly `Nonce(Felt::ONE)` and never receives or reads the config:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [4](#0-3) 

The caller `run_pre_validation_checks` has full access to `self.config` but does not pass `self.config.max_nonce_for_validation_skip` to the free function: [5](#0-4) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, meaning the account's `__validate__` entry point (which checks the signature) is not called during gateway admission: [6](#0-5) 

### Impact Explanation

**Impact: High â€” Mempool/gateway admission accepts invalid transactions before sequencing.**

When an operator sets `max_nonce_for_validation_skip` to `Nonce(Felt::ZERO)` to disable the UX skip feature entirely, the Rust gateway ignores this setting. An attacker can:

1. Submit a `DeployAccount` transaction for an address they control to the mempool.
2. Submit an `Invoke` transaction with `nonce = 1` from that same address, carrying an **invalid or forged signature**.
3. The gateway evaluates `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)` â€” both true â€” and calls `account_tx_in_pool_or_recent_block`, which returns `true` because the deploy_account is in the pool.
4. `skip_validate = true` is returned; the `__validate__` entry point is never called at the gateway.
5. The invalid transaction is admitted to the mempool and forwarded to the batcher without signature verification.

The batcher will eventually revert the transaction during execution, but the gateway's admission invariant â€” that every accepted transaction has passed account validation â€” is broken. This wastes mempool slots, batcher execution resources, and block space on reverted transactions.

Symmetrically, if `max_nonce_for_validation_skip` is set to `Nonce(Felt::TWO)` to extend the skip window, the gateway still only skips for nonce 1, causing valid nonce-2 invoke transactions from undeployed accounts to be **rejected** even though the operator intended them to be admitted.

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which matches the hardcoded constant, so the default deployment is unaffected. The issue is triggered when an operator changes this config value â€” a documented, serialized, production config field â€” and expects the change to take effect. Because the field is dead code in the Rust gateway path, the change silently has no effect, creating a gap between operator intent and actual behavior.

### Recommendation

Convert `skip_stateful_validations` from a free function to a method on `StatefulTransactionValidator`, or pass `self.config.max_nonce_for_validation_skip` as an explicit parameter, and replace the hardcoded `Nonce(Felt::ONE)` with the config value:

```rust
// In run_pre_validation_checks:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,  // pass config
).await?;

// In skip_stateful_validations:
if tx.nonce() == Nonce(Felt::ONE)
    && tx.nonce() <= max_nonce_for_validation_skip  // use config
    && account_nonce == Nonce(Felt::ZERO)
{
```

### Proof of Concept

1. Configure the gateway with `max_nonce_for_validation_skip = 0x0` (operator intends to disable the skip feature).
2. Submit `DeployAccount` for address `A` (attacker-controlled) to the gateway â€” it passes all validations and enters the mempool.
3. Submit `Invoke` with `sender_address = A`, `nonce = 1`, and a **garbage signature** (e.g., all zeros).
4. `skip_stateful_validations` evaluates: `nonce == 1` âś“, `account_nonce == 0` âś“, `account_tx_in_pool_or_recent_block(A)` â†’ `true` âś“.
5. `skip_validate = true`; `run_validate_entry_point` is called with `validate = false`; `__validate__` is never invoked.
6. The transaction with the invalid signature is accepted by the gateway and inserted into the mempool â€” contradicting the operator's explicit configuration to disable this bypass. [7](#0-6) [8](#0-7)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
