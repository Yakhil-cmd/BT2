### Title
Missing Access Control on `update_rewards` Allows Anyone to Permanently Skip Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is specified to be callable only by the Starkware sequencer, but the implementation contains no such access control check. Any unprivileged caller can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` pointer without distributing rewards, permanently burning that block's yield for all stakers and delegators.

### Finding Description
The protocol specification at `docs/spec.md` line 1644–1645 explicitly states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1447–1507 does not enforce this. The function is exposed via `#[abi(embed_v0)]` with no caller check:

```cairo
#[abi(embed_v0)]
impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
    fn update_rewards(
        ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
    ) {
        self.general_prerequisites();   // only checks pause state
        let current_block_number = starknet::get_block_number();
        assert!(
            current_block_number > self.last_reward_block.read(),
            "{}",
            Error::REWARDS_ALREADY_UPDATED,
        );
        // ... staker validation ...
        self.last_reward_block.write(current_block_number);  // global write

        if disable_rewards || self.is_pre_consensus() {
            return;   // exits without distributing rewards
        }
        // ... reward distribution ...
    }
}
```

`last_reward_block` is a **single global storage variable** (not per-staker), declared at `src/staking/staking.cairo` line 187:

```cairo
last_reward_block: BlockNumber,
```

When any caller invokes `update_rewards(valid_staker, disable_rewards: true)`:
1. `last_reward_block` is written to the current block number.
2. The function returns early, distributing zero rewards.
3. Any subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`.

Because `last_reward_block` is global, a single call with any valid `staker_address` and `disable_rewards: true` poisons the entire block for every staker and every delegation pool.

### Impact Explanation
Each block that is "consumed" by the attacker with `disable_rewards: true` is permanently lost: the block number can never be revisited, so the rewards that would have been distributed are never minted or credited. Stakers' `unclaimed_rewards_own` and pool balances are never incremented for those blocks. This constitutes **permanent freezing of unclaimed yield** for all stakers and delegators across the protocol.

Allowed impact matched: **High — Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation
The Starknet sequencer is centralized and controls intra-block transaction ordering, so it can in principle always place its own `update_rewards` call first. However:
- The sequencer is not obligated to call `update_rewards` in every block; blocks where it does not call it are fully open to the attacker.
- The attacker only needs to submit a transaction to the public mempool; no special privilege, stake, or capital is required.
- The attack is cheap (one transaction per block) and can be sustained indefinitely.
- The missing check is a straightforward spec-vs-code divergence with no compensating control in the code.

### Recommendation
Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the authorized sequencer address (stored in contract storage or a role), consistent with the spec's stated access control of "Only starkware sequencer."

### Proof of Concept
1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker observes a new block `N` where `N > last_reward_block`.
3. Attacker calls `update_rewards(any_valid_staker, disable_rewards: true)`.
4. `last_reward_block` is set to `N`; no rewards are distributed.
5. The legitimate sequencer attempts `update_rewards(staker, disable_rewards: false)` in the same block → reverts with `REWARDS_ALREADY_UPDATED`.
6. Block `N`'s rewards are permanently lost for all stakers and delegators.
7. Repeat for every subsequent block.
