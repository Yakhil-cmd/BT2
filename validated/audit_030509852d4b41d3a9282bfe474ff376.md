### Title
Stale `wnear_address` Read in `withdraw_wnear_to_router` Callback Permanently Freezes User wNEAR When `factory_set_wnear_address` Is Updated Mid-Flight - (File: `engine/src/contract_methods/xcc.rs`)

---

### Summary

The XCC flow is a two-phase operation split across NEAR receipts. In phase 1 (EVM execution), the XCC precompile transfers the user's wNEAR ERC-20 tokens to the engine's implicit address using the `wnear_address` stored at that moment. In phase 2 (a subsequent NEAR receipt), the `withdraw_wnear_to_router` callback re-reads `wnear_address` from storage to call `withdrawToNear`. If `factory_set_wnear_address` is called between these two phases, the callback uses a different address than the one used in phase 1, causing the callback to fail while the user's wNEAR tokens remain permanently locked in the engine's implicit EVM address.

---

### Finding Description

**Phase 1 — EVM execution (XCC precompile):**

In `engine-precompiles/src/xcc.rs`, when `required_near != ZERO_YOCTO`, the precompile reads `wnear_address` from storage and calls `transferFrom` on the wNEAR ERC-20 contract to move the user's tokens to the engine's implicit address:

```rust
let wnear_address = state::get_wnear_address(&self.io);   // reads wnear_address NOW
// ...
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
```

This EVM state change is committed atomically with the EVM execution. The user's wNEAR tokens are now in the engine's implicit EVM address.

The engine then schedules a `withdraw_wnear_to_router` self-call with args `WithdrawWnearToRouterArgs { target, amount }` — **the `wnear_address` is not included in these args**.

**Phase 2 — `withdraw_wnear_to_router` callback (subsequent NEAR receipt):**

In `engine/src/contract_methods/xcc.rs` line 43, the callback re-reads `wnear_address` from storage independently:

```rust
let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
```

It then calls `withdrawToNear` on this address to burn the wNEAR and send NEAR to the router.

**The gap:**

`factory_set_wnear_address` in `engine/src/contract_methods/xcc.rs` lines 102–115 can be called by the owner at any time with no guard for in-flight XCC calls:

```rust
pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        // ...
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));  // immediate update
        Ok(())
    })
}
```

If the owner calls `factory_set_wnear_address` between phase 1 and phase 2, the callback reads the new address and attempts `withdrawToNear` on a contract where the engine's implicit address holds no tokens. The call fails. The chained router `execute` also fails. The user's wNEAR tokens transferred in phase 1 remain permanently locked in the engine's implicit EVM address with no recovery path.

---

### Impact Explanation

The user's wNEAR ERC-20 tokens are transferred from the user to the engine's implicit EVM address during phase 1 (committed EVM state). When phase 2 fails due to the stale address, there is no automatic refund or rollback of the EVM-level `transferFrom`. The engine's implicit address is not directly controllable by users, and no standard recovery function exists to return the stuck tokens. This constitutes **permanent freezing of user funds**.

---

### Likelihood Explanation

The owner legitimately calls `factory_set_wnear_address` when migrating to a new wNEAR contract (e.g., a contract upgrade). NEAR processes receipts asynchronously; multiple XCC calls can be in-flight simultaneously. The window between phase 1 and phase 2 spans at least one NEAR block. On a busy engine instance, many users could have in-flight XCC calls at the time of a wNEAR migration, making this a realistic scenario during any wNEAR contract upgrade.

---

### Recommendation

The `wnear_address` used in phase 1 must be the same address used in phase 2. Two approaches:

1. **Include `wnear_address` in `WithdrawWnearToRouterArgs`**: Serialize the address at promise-creation time so the callback uses the original value regardless of subsequent storage changes.

2. **Guard `factory_set_wnear_address`**: Reject the update if any XCC calls are in-flight (analogous to the original report's recommendation of using a pending-value pattern that only takes effect at the end of the relevant operation).

Option 1 is simpler and directly mirrors the fix recommended in the original report.

---

### Proof of Concept

1. User calls the XCC precompile with `required_near > 0`. Phase 1 executes: `transferFrom` moves 2 NEAR worth of wNEAR (old address `A`) from the user to the engine's implicit EVM address. `withdraw_wnear_to_router` is scheduled with `{target: user_router, amount: 2_NEAR}`.

2. Before the `withdraw_wnear_to_router` receipt executes, the owner calls `factory_set_wnear_address(B)` where `B` is a new wNEAR contract address.

3. The `withdraw_wnear_to_router` receipt executes. Line 43 of `engine/src/contract_methods/xcc.rs` reads `wnear_address = B`. It calls `withdrawToNear` on contract `B` for 2 NEAR. The engine's implicit address has no balance on contract `B`, so the EVM call reverts. `withdraw_wnear_to_router` returns `ERR_WITHDRAW_FAILED`.

4. The chained router `execute` receipt is skipped (callback of failed promise).
