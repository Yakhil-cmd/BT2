### Title
`validate_tx` Omits `add_ready_declares()`, Causing Gateway to Reject Valid Fee-Escalation Replacements After Declare Delay Expires - (`crates/apollo_mempool/src/mempool.rs`)

### Summary

The mempool's `validate_tx` (called by the gateway as a pre-admission check) does not call `add_ready_declares()` before evaluating delayed-declare conflicts, while `add_tx` does. Once a declare transaction's `declare_delay` has elapsed, `validate_tx` still finds it in `delayed_declares` and returns `DuplicateNonce` for any replacement at the same `(address, nonce)`. Because the gateway calls `validate_tx` first and aborts on rejection, valid fee-escalation replacements are permanently blocked at the gateway until an unrelated transaction triggers `add_ready_declares()` inside `add_tx`.

### Finding Description

**Two divergent code paths in the mempool:**

`add_tx` calls `add_ready_declares()` at the top of its body, which drains all expired entries from `delayed_declares` into `tx_pool`: [1](#0-0) 

```rust
pub fn add_tx(&mut self, args: AddTransactionArgs) -> MempoolResult<()> {
    let mut account_nonce_updates = self.remove_expired_txs();
    if !self.is_fifo() {
        self.add_ready_declares();   // ← moves expired declares to tx_pool
    }
    ...
```

`validate_tx` does **not** call `add_ready_declares()`: [2](#0-1) 

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;   // ← no add_ready_declares first
    Ok(())
}
```

`validate_fee_escalation` calls `validate_no_delayed_declare_front_run`, which rejects **any** transaction at the same `(address, nonce)` as a delayed declare, regardless of whether the delay has already expired: [3](#0-2) 

```rust
fn validate_no_delayed_declare_front_run(&self, tx_reference: TransactionReference) -> MempoolResult<()> {
    if self.delayed_declares.contains(tx_reference.address, tx_reference.nonce) {
        return Err(MempoolError::DuplicateNonce { address: tx_reference.address, nonce: tx_reference.nonce });
    }
    Ok(())
}
```

**Gateway flow that exposes the divergence:**

The gateway always calls `validate_tx` (via `validate_by_mempool`) before calling `add_tx`. If `validate_tx` returns an error, `add_tx` is never reached: [4](#0-3) 

`validate_by_mempool` is called inside `run_pre_validation_checks`: [5](#0-4) 

**The broken invariant:** `validate_tx` is supposed to predict whether `add_tx` will accept the transaction. After the `declare_delay` expires, `validate_tx` returns `DuplicateNonce` for a valid fee-escalation replacement, but `add_tx` would accept it (because `add_ready_declares()` would first drain the expired declare into `tx_pool`, after which normal fee-escalation logic applies).

**Production `declare_delay` values:** [6](#0-5) 

The deployment config sets `declare_delay` to 20 seconds; the default is 1 second. [7](#0-6) 

### Impact Explanation

After a declare transaction's delay expires, the gateway permanently rejects any fee-escalation replacement submitted by the user with a spurious `DuplicateNonce` error. The user is stuck with a low-fee declare that cannot be replaced until an unrelated transaction from any sender happens to trigger `add_ready_declares()` inside `add_tx`. In a low-traffic environment the user may be unable to replace their declare at all, causing it to sit in the delayed queue until TTL expiry. This matches: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

### Likelihood Explanation

Any unprivileged user can trigger this deterministically:
1. Submit a declare transaction → it enters `delayed_declares`.
2. Wait for `declare_delay` to elapse (20 s in production).
3. Submit a fee-escalation replacement declare at the same `(address, nonce)`.
4. The gateway calls `validate_tx`, which finds the original declare still in `delayed_declares` and returns `DuplicateNonce`, aborting before `add_tx` is ever called.

No special privileges, no malicious peer, no race condition required.

### Recommendation

`validate_tx` should mirror the expiry logic of `add_tx`. The minimal fix is to check whether the delayed declare at `(address, nonce)` has already expired before returning `DuplicateNonce` in `validate_no_delayed_declare_front_run`. Alternatively, `validate_tx` could skip the `validate_no_delayed_declare_front_run` check entirely for entries whose `submission_time + declare_delay ≤ now`, since those entries would have been drained by `add_ready_declares()` before `add_tx` runs its own validation.
