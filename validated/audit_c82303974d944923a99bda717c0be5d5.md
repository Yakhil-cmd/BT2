### Title
Missing Participant-Set Validation Between Triple Generation and Presigning — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign` function in the OT-based ECDSA implementation explicitly omits verifying that the triples supplied via `PresignArguments` were generated by the same participant set that is running the presigning protocol. Only the threshold value is cross-checked. A malicious coordinator can supply triples generated under a different participant set, causing Lagrange interpolation during presigning to use incorrect coefficients and producing a cryptographically corrupted presignature that honest parties will accept as valid output.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the public entry point `presign()` validates several initialization conditions but contains an explicit, documented omission:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [1](#0-0) 

The code checks that `args.threshold` matches the threshold embedded in both triples, but it does **not** check that the participant set embedded in `triple0` and `triple1` matches the `participants` slice passed to `presign()`. The comment acknowledges this omission and justifies it with an assumption ("presumably they need to have been present in order to have shares") that is not enforced by any runtime check.

The analog to the external report is direct: just as the Solana program used `liquidity_pool` in a reward calculation without verifying it matched `loan.liquidity_pool`, this code uses triple shares in Lagrange-interpolation-based presigning without verifying the triples' participant set matches the presigning participant set.

During `do_presign`, each participant's triple share is evaluated at their own index within the **presigning** participant list to compute Lagrange coefficients. If the triple was generated under participant set `{A, B, C}` but presigning runs under `{A, B, D}`, participant `A`'s Lagrange coefficient is computed using `D`'s index rather than `C`'s index. The resulting linear combination is arithmetically wrong, producing a presignature whose internal consistency checks (if any) may still pass locally but whose cryptographic relationship to the shared secret key is broken. [2](#0-1) 

---

### Impact Explanation

**Impact: High — Corruption of presign outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

When a presignature is produced from mismatched triples, the resulting `PresignOutput` is internally consistent from each honest party's local perspective (no local check fails) but is cryptographically invalid relative to the shared secret key. When this corrupted presignature is subsequently used in `sign()`, the resulting signature will be invalid and unverifiable under the group public key. Honest parties have no way to distinguish this from a legitimate presignature at the presign output stage, so they accept and store it. Every signing attempt using this presignature will fail, permanently denying signing capability for that presignature slot.

Additionally, if the attacker (as coordinator) controls which triples are paired with which presigning sessions, they can selectively corrupt presignatures for chosen messages or participants, causing targeted denial of signing.

---

### Likelihood Explanation

**Likelihood: Medium-High.**

The coordinator role is explicitly modeled as a potentially malicious participant throughout this codebase. The coordinator controls which `PresignArguments` (including which triple shares) are distributed to participants for a given presigning session. A malicious coordinator can trivially supply triple shares from a session with a different participant set — the only runtime check is on the threshold value, not the participant identity set. No out-of-band mechanism enforces the pairing. The attack requires no cryptographic capability beyond participation as coordinator. [1](#0-0) 

---

### Recommendation

Add an explicit check that the participant set recorded in each triple matches the participant set passed to `presign()`. The `TripleGenerationOutput` (or its associated metadata) should carry the participant list used during generation. Before proceeding, verify:

```rust
if args.triple0.1.participants != participants || args.triple1.1.participants != participants {
    return Err(InitializationError::BadParameters(
        "Triple participant set must match presigning participant set".to_string(),
    ));
}
```

This is the direct analog of the patch in the external report (`has_one = liquidity_pool`, `has_one = collection_info`): enforce that the state object used in the computation was produced for the same context as the current operation. [1](#0-0) 

---

### Proof of Concept

1. Honest parties `{A, B, C}` complete triple generation, producing triple shares for participant set `S1 = {A, B, C}` with threshold `t`.
2. A malicious coordinator initiates a presigning session with participant set `S2 = {A, B, D}` (same threshold `t`, different member `D` replacing `C`).
3. The coordinator distributes to `A` and `B` their triple shares from the `S1` session (which pass the only runtime check: `threshold == t`).
4. `presign()` proceeds without error. Each party computes Lagrange coefficients using `S2`'s participant indices, but their triple shares encode evaluations at `S1`'s participant indices. The resulting linear combination is arithmetically incorrect.
5. `do_presign` completes and returns a `PresignOutput` to all parties. No local error is raised.
6. When `sign()` is called using this `PresignOutput`, the produced signature fails verification under the group public key. The signing capability for this presignature is permanently lost.
7. The coordinator can repeat this for every presignature, causing permanent denial of signing for honest parties. [3](#0-2)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
