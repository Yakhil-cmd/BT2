### Title
Signature Bypass via `skip_stateful_validations` Allows Unauthenticated Invoke Transactions into Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX feature for the `deploy_account + invoke` flow unconditionally skips the `__validate__` entry-point call (i.e., signature verification) for any Invoke transaction with `nonce == 1` submitted against an address whose on-chain nonce is `0` and that already has a transaction in the mempool. Because the mempool's own `validate_tx` path performs no signature check either, an attacker can inject an Invoke transaction carrying an **arbitrary/invalid signature** for any victim address that has a pending `deploy_account` in the mempool. The invalid transaction is admitted and occupies the victim's `nonce=1` slot, causing the victim's legitimate first post-deploy Invoke to be rejected as a duplicate nonce (or requiring a fee-escalation race to displace the attacker's entry).

---

### Finding Description

**Gateway stateful path — `skip_stateful_validations`**

`extract_state_nonce_and_run_validations` is the top-level stateful validation entry point:

```
account_nonce  ← get_nonce_from_state(sender)
skip_validate  ← run_pre_validation_checks(tx, account_nonce, mempool)
                   └─ validate_state_preconditions   (nonce range, resource bounds)
                   └─ validate_by_mempool            (duplicate hash / nonce check only)
                   └─ skip_stateful_validations      ← returns true when conditions met
run_validate_entry_point(tx, skip_validate)
```

`skip_stateful_validations` returns `true` when all three conditions hold:

```rust

        return mempool_client
            .account_tx_in_pool_or_recent_block(tx.sender_address())
            .await ...

}
```

When `skip_validate == true`, `run_validate_entry_point` builds the `AccountTransaction` with `validate: false`:

```rust
// lines 310-312

let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations` (blockifier), the branch for Invoke transactions is:

```rust

`validate_tx` in `AccountTransaction` also short-circuits immediately when `validate == false`:

```rust

    return Ok(None);
}
```

**Mempool `validate_tx` does not check signatures**

`validate_by_mempool` calls `mempool_client.validate_tx(ValidationArgs)`. `ValidationArgs` carries only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — no signature field. The mempool's `validate_tx` only checks for duplicate hashes and nonce validity:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
```

No path in the gateway or mempool verifies the cryptographic signature of the Invoke transaction when `skip_validate == true`.

---

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool can:

1. Craft an Invoke transaction for the victim's address with `nonce = 1` and an **arbitrary signature** (e.g., all zeros).
2. Submit it to the gateway. The three conditions for `skip_stateful_validations` are satisfied: tx is Invoke, tx nonce = 1, on-chain account nonce = 0, and `account_tx_in_pool_or_recent_block` returns `true` for the victim's address.
3. The gateway skips `__validate__`, the mempool performs no signature check, and the attacker's transaction is admitted.
4. The attacker's invalid-signature transaction now occupies `(victim_address, nonce=1)` in the mempool.
5. When the victim submits their legitimate Invoke (nonce=1):
   - With fee escalation **disabled**: rejected as `MempoolError::DuplicateNonce`.
   - With fee escalation **enabled**: the victim must outbid the attacker's tip/gas price to displace the entry; if the attacker sets a high tip, this becomes a griefing/extortion vector.
6. The attacker's transaction will fail at execution time (when `__validate__` is called with `validate: true` by the batcher), but the victim's legitimate transaction has already been displaced from the mempool.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

- **No privilege required.** The attacker only needs to watch the public mempool for `deploy_account` transactions.
- **Trivially constructable.** Any Invoke transaction with `nonce=1` and a zero/random signature passes all gateway checks under the skip condition.
- **Deterministic trigger.** The three conditions (`Invoke`, `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true`) are reliably satisfied whenever a user submits the standard `deploy_account + invoke` UX flow.
- **Race window is wide.** The attacker has the entire time between the victim's `deploy_account` being admitted and the victim's Invoke being submitted — typically seconds to minutes.

---

### Recommendation

The root cause is that the gateway cannot call `__validate__` for an account that does not yet exist on-chain, so it skips signature verification entirely. Mitigations:

1. **Bind the skip to the same submission batch.** Only skip `__validate__` when the Invoke transaction was submitted in the same HTTP request as the `deploy_account` (e.g., via a multi-transaction endpoint), so the gateway can verify they share the same sender without needing the account to exist.
2. **Require a pre-image commitment.** Require the Invoke transaction to include a hash commitment to the `deploy_account` transaction hash, verifiable without executing the account contract.
3. **Defer admission.** Do not admit the Invoke to the mempool until the `deploy_account` has been executed and the account exists on-chain. This sacrifices the UX benefit but eliminates the attack surface.
4. **At minimum**, add a note in the mempool that the admitted Invoke was signature-unverified, and re-verify it before handing it to the batcher.

### Proof of Concept

```

Step 1 — Victim submits deploy_account:
  POST /gateway/add_transaction
  { type: "DEPLOY_ACCOUNT", sender_address: V, nonce: 0, signature: <valid>, ... }
  → Admitted to mempool. account_tx_in_pool_or_recent_block(V) == true.

Step 2 — Attacker submits forged invoke:
  POST /gateway/add_transaction
  { type: "INVOKE", sender_address: V, nonce: 1, signature: [0x0, 0x0], ... }

  Gateway evaluation:
    account_nonce = get_nonce_from_state(V) = 0
    skip_stateful_validations:
      tx.nonce() == 1  ✓
      account_nonce == 0  ✓
      account_tx_in_pool_or_recent_block(V) == true  ✓
      → returns true
    run_validate_entry_point(tx, skip_validate=true):
      ExecutionFlags { validate: false, ... }
      StatefulValidator::perform_validations → returns Ok(()) without calling __validate__
  → Attacker's tx admitted to mempool at (V, nonce=1).

Step 3 — Victim submits legitimate invoke:
  POST /gateway/add_transaction
  { type: "INVOKE", sender_address: V, nonce: 1, signature: <valid ECDSA>, ... }

    validate_fee_escalation → DuplicateNonce (fee escalation disabled)

  execute_raw → perform_pre_validation_stage → handle_nonce (increments nonce)
             → run_or_revert → validate_tx (validate=true) → __validate__ called
             → signature [0x0, 0x0] fails → tx reverted/rejected
  But victim's nonce=1 tx is already gone from the mempool.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
