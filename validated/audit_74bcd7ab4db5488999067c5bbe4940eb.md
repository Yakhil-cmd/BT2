## Code Path Trace

**Step 1 — Attacker-controlled entrypoint: blockchain notification parsing**

In `useBlockchainNotifications.tsx`, the `ph` field from the raw on-chain notification message is extracted and stored directly as `puzzleHash` with no validation that it matches the actual sender's address: [1](#0-0) 

Anyone can send a notification on-chain to any wallet address. The `ph` field in the JSON payload is fully attacker-controlled.

**Step 2 — Validity filter does NOT validate puzzleHash**

`useValidNotifications.ts` filters notifications by checking only `offerState.offer?.valid` — it validates the offer itself, not the `puzzleHash`: [2](#0-1) 

A notification with a crafted `puzzleHash` passes this filter as long as the embedded offer URL resolves to a valid offer.

**Step 3 — handleCounterOffer trusts the notification's puzzleHash unconditionally** [3](#0-2) 

`puzzleHash` is taken directly from the notification, converted to a bech32m address, and passed as `address` in the navigation state to the offer builder. There is no check that this puzzle hash belongs to the actual on-chain sender of the notification transaction.

**Step 4 — address flows into CreateOfferBuilder and onOfferCreated** [4](#0-3) 

The `address` prop is passed through to `onOfferCreated`, which in `OfferManager.tsx` converts it and passes it to `OfferShareDialog` as the destination for the counter-offer notification: [5](#0-4)

## Impact Correction

The claim states "funds from the counter-offer go to the attacker." This is **incorrect as stated**. The `address` field is used for **notification routing** (where the counter-offer notification is sent), not as a fund destination. Fund destinations in Chia offers are determined by the cryptographic spend bundle, not by this UI field.

The **actual impact** is:
- The counter-offer notification is misdirected to the attacker's address instead of the legitimate offer creator's address.
- The legitimate offer creator never receives the counter-offer notification.
- The attacker receives the counter-offer notification and can choose to **accept** it on the victim's terms.

For the attacker to profit, they must take an additional action (accepting the counter-offer). The victim still receives whatever they specified in the counter-offer terms. This is a notification misdirection attack, not direct fund theft.

---

## Verdict

The vulnerability is **real and the attack path is concrete**: an unprivileged attacker can send an on-chain notification with an arbitrary `puzzleHash`, which passes the offer validity filter and is unconditionally trusted by `handleCounterOffer` to pre-populate the counter-offer return address. There is no guard verifying the `puzzleHash` against the actual transaction sender.

However, the claimed impact — "funds go to the attacker without the victim noticing" — is **overstated**. The `address` controls notification routing, not fund destination. The attacker must still accept the counter-offer to profit, and the victim receives what they requested.

Under the scope rules, this is "unsafe trust of notification state that causes a user to send [a counter-offer notification] to the wrong destination." This meets the **High** threshold for notification spoofing/misdirection, but the specific PoC claim (automatic fund loss) does not hold as described.

### Title
Unverified `puzzleHash` in blockchain notification pre-populates counter-offer return address with attacker-controlled value — (`packages/gui/src/components/offers2/OfferIncomingTable.tsx`)

### Summary
An attacker who can send an on-chain notification (a standard Chia wallet feature) can embed an arbitrary `puzzleHash` in the notification payload. The GUI trusts this value unconditionally and uses it as the counter-offer notification return address, misdirecting the victim's counter-offer notification to the attacker.

### Finding Description
`useBlockchainNotifications.tsx` parses `data.ph` from the raw message and stores it as `puzzleHash` with no verification against the actual sender. `useValidNotifications` only checks offer validity, not the puzzle hash. `handleCounterOffer` converts this unverified hash to a bech32m address and passes it to the offer builder. The offer builder passes it to `onOfferCreated`, which uses it as the destination for the counter-offer notification.

### Impact Explanation
The counter-offer notification is sent to the attacker's address instead of the legitimate offer creator's. The attacker can accept the counter-offer. The legitimate offer creator is silently excluded from the counter-offer flow.

### Likelihood Explanation
Sending an on-chain notification requires only a small XCH amount and is a standard, documented feature. The attacker needs no special access.

### Recommendation
When parsing a blockchain notification, verify that the `puzzleHash` (`ph` field) matches the puzzle hash of the coin that sent the notification (i.e., the actual on-chain sender), not just accept it as a self-reported field in the message payload.

### Proof of Concept
1. Attacker crafts a notification payload: `{"v":1,"t":1,"d":{"u":"<valid_offer_url>","ph":"<attacker_puzzle_hash>"}}`
2. Attacker sends this notification on-chain to the victim's wallet address.
3. Victim sees the notification; the offer is valid, so it passes the filter.
4. Victim clicks "Counter" — the offer builder opens with `toBech32m(attacker_puzzle_hash, ...)` pre-populated as the return address.
5. Victim creates a counter-offer; the counter-offer notification is sent to the attacker's address.
6. The legitimate offer creator receives nothing; the attacker receives the counter-offer and can accept it.
