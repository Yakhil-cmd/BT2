### Title
EVM `dryRun` Simulation Uses Stale Block Context (Timestamp/Height), Causing Simulation-to-Execution Divergence and Relayer Financial Loss - (File: `fvm/evm/handler/handler.go`, `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.dryRun` simulates an EVM transaction against the block proposal's timestamp and block number at the time the script executes. When the relayer subsequently submits the real transaction via `EVM.run` in a later Cadence transaction, the block proposal has advanced to a new height and timestamp. EVM contracts that gate execution on `block.timestamp` or `block.number` (e.g., `require(block.timestamp < deadline)`) will return `StatusSuccessful` during dry-run but revert during actual execution. The relayer pays the Flow transaction fee regardless of EVM execution outcome, resulting in a direct financial loss.

---

### Finding Description

`EVM.dryRun` is exposed as a public, unpermissioned Cadence function callable from any script or transaction:

```cadence
// fvm/evm/stdlib/contract.cdc lines 853-865
/// Simulates running unsigned RLP-encoded transaction using
/// the from address as the signer.
/// The transaction state changes are not persisted.
/// This is useful for gas estimation or calling view contract functions.
/// Note: dry-run functions are intentionally exempt from the isPaused() guard —
/// they perform no state mutations and remain available in read-only mode.
access(all)
fun dryRun(tx: [UInt8], from: EVMAddress): Result {
    return InternalEVM.dryRun(
        tx: tx,
        from: from.bytes,
    ) as! Result
}
``` [1](#0-0) 

Internally, `dryRunTx` in the Go handler fetches the **current** block proposal and derives the EVM block context from it:

```go
// fvm/evm/handler/handler.go lines 608-615
bp, err := h.getBlockProposal()
if err != nil {
    return nil, err
}
ctx, err := h.getBlockContext(bp)
if err != nil {
    return nil, err
}
``` [2](#0-1) 

`getBlockContext` directly injects `bp.Timestamp` and `bp.Height` as the EVM block timestamp and block number:

```go
// fvm/evm/handler/handler.go lines 730-750
return types.BlockContext{
    ChainID:        types.EVMChainIDFromFlowChainID(h.flowChainID),
    BlockNumber:    bp.Height,
    BlockTimestamp: bp.Timestamp,
    ...
}, nil
``` [3](#0-2) 

`DryRunTransaction` then executes the EVM transaction against this context without persisting state:

```go
// fvm/evm/emulator/emulator.go lines 299-330
func (bl *BlockView) DryRunTransaction(
    tx *gethTypes.Transaction,
    from gethCommon.Address,
) (*types.Result, error) {
    proc, err := bl.newProcedure()
    ...
    msg.SkipNonceChecks = true
    msg.SkipTransactionChecks = true
    // run and return without committing the state changes
    return proc.run(msg, tx.Hash(), tx.Type())
}
``` [4](#0-3) 

When the relayer calls `EVM.run` in a **subsequent Cadence transaction**, `getBlockProposal()` returns a new proposal at a later height with a later timestamp. Any EVM contract logic conditioned on `block.timestamp` or `block.number` (e.g., a Uniswap-style deadline check `require(block.timestamp <= deadline, "expired")`) will evaluate against a different value than during the dry run. The simulation returns `StatusSuccessful`; the actual execution reverts.

The Flow transaction fee is charged to the payer (the relayer) regardless of EVM execution outcome, because the Cadence transaction itself succeeds — only the inner EVM call reverts. The relayer has no on-chain mechanism to detect this divergence before submission.

---

### Impact Explanation

**Impact: Medium**

Any relayer or bundler that uses `EVM.dryRun` to pre-validate EVM transactions before submitting them via `EVM.run` is exposed to financial loss. The Flow transaction fee is deducted from the relayer's account even when the EVM transaction reverts due to a time-based condition that was valid at dry-run time but expired by execution time. This is a direct, measurable loss of FLOW tokens for the relayer, proportional to the gas limit of the Cadence transaction.

---

### Likelihood Explanation

**Likelihood: Medium**

Time-based guards are ubiquitous in EVM DeFi contracts (swap deadlines, auction windows, vesting cliffs, time-locked operations). Any relayer infrastructure that uses `EVM.dryRun` for pre-flight checks — a natural and documented use case per the contract's own NatSpec — is susceptible whenever the block timestamp advances past a contract-enforced deadline between the script call and the transaction submission. No special privileges or attacker-controlled state are required; the block timestamp advancing is a normal protocol property.

---

### Recommendation

1. **Expose block context in dry-run results**: Return the `BlockTimestamp` and `BlockNumber` used during the dry run in the `Result` struct so callers can detect staleness before submitting.
2. **Add a deadline parameter to `EVM.dryRun`**: Allow callers to specify a maximum acceptable block timestamp; if the current proposal's timestamp exceeds it, return a distinct error code rather than a success result.
3. **Document the divergence risk**: The NatSpec for `EVM.dryRun` should explicitly warn that results are only valid for the block at which the script executes and may not reflect execution conditions in a subsequent transaction.

---

### Proof of Concept

1. Deploy an EVM contract on Flow EVM with a time-gated function:
   ```solidity
   function swap(uint deadline) external {
       require(block.timestamp <= deadline, "expired");
       // ... swap logic
   }
   ```

2. Call `EVM.dryRun` in a Cadence **script** at block height H (timestamp T), passing `deadline = T + 5 seconds`. The dry run executes against block proposal at height H, timestamp T. Result: `StatusSuccessful`.

3. Submit a Cadence **transaction** calling `EVM.run` with the same EVM transaction. By the time the transaction is included and executed, the block proposal is at height H+N, timestamp T+6 seconds. The EVM contract reverts: `block.timestamp (T+6) > deadline (T+5)`.

4. The Cadence transaction succeeds (no Cadence-level error), but the EVM result is `StatusFailed`. The relayer's Flow account is debited the full transaction fee.

The root cause is that `dryRunTx` at [2](#0-1)  and `getBlockContext` at [3](#0-2)  bind the simulation to the block proposal at script-execution time, while `EVM.run` in a subsequent transaction binds to a later proposal with no cross-check or staleness guard.

	return types.BlockContext{
		ChainID:                types.EVMChainIDFromFlowChainID(h.flowChainID),

// DryRunTransaction runs an unsigned transaction without persisting the state
func (bl *BlockView) DryRunTransaction(
	tx *gethTypes.Transaction,
	from gethCommon.Address,
) (*types.Result, error) {
	// create a new procedure
	proc, err := bl.newProcedure()
	if err != nil {
		return nil, err
	}

	// convert tx into message
	msg, err := gethCore.TransactionToMessage(
		tx,
		GetSigner(bl.config),
		proc.config.BlockContext.BaseFee,
	)

	// we can ignore invalid signature errors since we don't expect signed transactions
	if !errors.Is(err, gethTypes.ErrInvalidSig) {
		return nil, err
	}

	// use the from as the signer
	msg.From = from
	// we need to skip nonce/transaction checks for dry run
	msg.SkipNonceChecks = true
	msg.SkipTransactionChecks = true

	// run and return without committing the state changes
	return proc.run(msg, tx.Hash(), tx.Type())
}
