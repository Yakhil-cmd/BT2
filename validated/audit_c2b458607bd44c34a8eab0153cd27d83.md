### Title
Malicious CKD Participant Can Silently Corrupt Confidential Key Derivation Output Without Verification - (File: src/confidential_key_derivation/protocol.rs)

### Summary
In `do_ckd_coordinator`, the coordinator accumulates `(big_y, big_c)` contributions from all participants and sums them directly into the final `CKDOutput` without any cryptographic proof that each contribution is correctly formed. A single malicious participant can send arbitrary group elements, silently corrupting the derived confidential key delivered to the TEE application.

### Finding Description
The CKD protocol computes a masked secret `C = Σ λᵢ·(xᵢ·H(pk,app_id) + yᵢ·app_pk)` and a blinding term `Y = Σ λᵢ·yᵢ·G`. The TEE application then unmasks via `C − a·Y = msk·H(pk,app_id)`.

In `do_ckd_coordinator`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` and adds them unconditionally: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no proof-of-correct-formation attached to each participant's message. Specifically, no participant is required to prove:
1. That `big_y = λᵢ·yᵢ·G` for some scalar `yᵢ` they know.
2. That `big_c = λᵢ·(xᵢ·H(pk,app_id) + yᵢ·app_pk)` using their actual secret share `xᵢ` and the same `yᵢ`.

The coordinator simply returns the accumulated `CKDOutput` with no post-verification: [2](#0-1) 

Compare this to the ECDSA signing coordinator, which at least verifies the final assembled signature against the public key before returning it: [3](#0-2) 

No equivalent correctness check exists in the CKD coordinator path.

The `compute_signature_share` function that each participant is *supposed* to call is: [4](#0-3) 

A malicious participant is free to skip this computation entirely and send any `(ElementG1, ElementG1)` pair instead.

### Impact Explanation
**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

When a malicious participant sends arbitrary `(big_y', big_c')`, the coordinator produces:
- `Y_final = Y_honest + big_y'`
- `C_final = C_honest + big_c'`

The TEE application calls `unmask(a)`: [5](#0-4) 

This yields `C_final − a·Y_final = msk·H(pk,app_id) + big_c' − a·big_y'`, which is not the correct confidential key. The corruption is completely silent: neither the coordinator nor any honest participant can detect it, because no verification step exists after aggregation.

### Likelihood Explanation
Any single participant below the threshold can trigger this. The attack requires no special knowledge — the attacker simply sends two arbitrary `G1` points instead of their honest contribution. The protocol has a single communication round for CKD participants, so the attack is trivially executable on every CKD invocation the malicious participant is part of.

### Recommendation
Each participant should attach a non-interactive zero-knowledge proof of correct formation alongside their `(big_y, big_c)` contribution. Concretely, a Chaum-Pedersen proof can demonstrate that `big_c − big_y·(app_pk/G)` lies on the correct line through `λᵢ·xᵢ·H(pk,app_id)`, binding the contribution to the participant's committed public share. The coordinator must verify all such proofs before aggregating. Alternatively, a post-aggregation pairing check (analogous to the ECDSA coordinator's `sig.verify` call) could detect a corrupted output if a reference value is available.

### Proof of Concept
1. Honest participants P1, P2 and malicious participant P3 run `ckd(...)`.
2. P1 and P2 call `compute_signature_share` and send correct `(norm_big_y, norm_big_c)` to the coordinator.
3. P3 instead sends `(G1::generator(), G1::generator())` — two arbitrary non-zero points.
4. The coordinator in `do_ckd_coordinator` adds all three contributions without any check.
5. The returned `CKDOutput` contains `Y_final = Y_honest + G` and `C_final = C_honest + G`.
6. The TEE application calls `ckd_output.unmask(app_sk)` and receives `msk·H(pk,app_id) + G − app_sk·G`, which is a random-looking wrong key.
7. No error is raised anywhere in the library; the corruption is invisible to all honest parties.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
