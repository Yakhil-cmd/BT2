### Title
Double Floor Division in Spread Fee Split Permanently Locks Protocol/Admin Spread Fee Dust in Pool — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

In `collectFees`, the spread fee surplus is split between admin and protocol using **two independent floor divisions**. Because both quotients round down, their sum can be 1 wei (scaled) less than `surplus*Scaled` per token per call. That residue is never distributed and is permanently locked in the pool. The notional fee split in the same function already uses the correct subtraction pattern; the spread fee split does not.

---

### Finding Description

Inside `collectFees` the spread surplus is partitioned as:

```solidity
uint256 spreadFee0ToAdminScaled =
    spreadSumE6 == 0 ? 0 : (surplus0Scaled * adminSpreadFeeE6_) / spreadSumE6;

uint256 spreadFee0ToProtocolScaled =
    spreadSumE6 == 0 ? 0 : (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
``` [1](#0-0) 

Because `adminSpreadFeeE6_ + protocolSpreadFeeE6_ = spreadSumE6`, integer floor division gives:

```
floor(S * A / T) + floor(S * P / T)  ≤  S   (can be S − 1)
```

whenever `S * A mod T ≠ 0`. The 1-wei shortfall is never recovered; it stays in the pool's balance as uncollectable spread-fee dust.

The **notional** fee split in the same function already applies the correct pattern — compute one leg, derive the other by subtraction — so no dust is lost there:

```solidity
uint256 notionalFee0ToProtocolScaled =
    notionalFee0AmountScaled - notionalFee0ToAdminScaled;
``` [2](#0-1) 

The asymmetry between the two splits is the root cause. The external report's analog is exact: `(amount * 1e18 − amount * X) / 1e18` vs. `amount − (amount * X) / 1e18`.

`collectPoolFees` on the factory is **permissionless** — any address may call it at any time:

```solidity
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(...);
}
``` [3](#0-2) 

The existing test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive` mirrors the split math but uses a surplus that does **not** subtract `notionalFeeToken*Scaled`, so it documents a different scenario (over-allocation when the wrong surplus is used). It does not cover or guard against the under-allocation caused by double floor division on the correct surplus. [4](#0-3) 

---

### Impact Explanation

Up to **1 wei (scaled) per token per `collectFees` call** is permanently locked in the pool and never reaches either the admin fee destination or the protocol (factory). Because `collectPoolFees` is permissionless and is expected to be called by keepers on an operational schedule, the locked amount accumulates monotonically. For low-decimal tokens (e.g. USDC with `TOKEN_0_SCALE_MULTIPLIER = 1e12`) the locked scaled wei corresponds to a fractional native unit, but for 18-decimal pairs the locked amount is 1 native wei per call. Neither admin nor protocol can recover it; the pool's `surplus*Scaled` accounting permanently absorbs it.

---

### Likelihood Explanation

The loss occurs whenever `surplus0Scaled * adminSpreadFeeE6_ mod spreadSumE6 ≠ 0`, which is the generic case for any non-trivial surplus and non-round fee ratio. Both `setPoolAdminFees` and `setPoolProtocolFee` trigger `collectFees` internally before updating rates, and `collectPoolFees` is callable by anyone, so the condition is reached on every normal operational cycle.

---

### Recommendation

Compute the protocol spread leg by subtraction, mirroring the notional fee pattern already present in the same function:

```solidity
// current (lossy)
uint256 spreadFee0ToProtocolScaled =
    spreadSumE6 == 0 ? 0 : (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;

// fix (lossless)
uint256 spreadFee0ToProtocolScaled =
    spreadSumE6 == 0 ? 0 : surplus0Scaled - spreadFee0ToAdminScaled;
```

Apply the same change for `spreadFee1ToProtocolScaled`.

---

### Proof of Concept

**Concrete numeric example** (token0, scaled units):

| Variable | Value |
|---|---|
| `surplus0Scaled` | 5 |
| `adminSpreadFeeE6_` | 1 |
| `protocolSpreadFeeE6_` | 2 |
| `spreadSumE6` | 3 |

Current code:
- `spreadFee0ToAdminScaled = (5 × 1) / 3 = 1` (floor of 1.667)
- `spreadFee0ToProtocolScaled = (5 × 2) / 3 = 3` (floor of 3.333)
- **Sum = 4 < 5 → 1 wei locked**

With fix:
- `spreadFee0ToAdminScaled = 1`
- `spreadFee0ToProtocolScaled = 5 − 1 = 4`
- **Sum = 5 = surplus → no loss**

Calling `collectPoolFees` N times with this surplus and fee ratio locks N wei permanently. Since `collectPoolFees` is permissionless and is called routinely by keepers, the locked amount grows without bound and is irrecoverable.
