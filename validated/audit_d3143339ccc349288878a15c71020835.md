### Title
Gateway L2 Gas Price Admission Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` — (`crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs`)

---

### Summary

The gateway's stateful transaction validator checks a submitted transaction's `max_price_per_unit` against the **current block's** `l2_gas_price`, but the transaction will actually be executed in the **next block**, which uses `next_l2_gas_price`. These two values diverge whenever the EIP-1559-style fee market adjusts the price between blocks. The result is that the gateway admits transactions whose gas price is too low for the next block (false admission) and rejects transactions whose gas price is sufficient for the next block (false rejection). A developer TODO in the source code explicitly acknowledges this is wrong.

---

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` from the latest committed block header. It populates `gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price` — the price that was used **inside** the already-committed block: [1](#0-0) 

The block header also carries `next_l2_gas_price` — the price computed by the EIP-1559 mechanism that **will** be used in the next block — but this field is silently dropped and never placed into `BlockInfo`.

`validate_resource_bounds` then reads `l2_gas_price` from this `BlockInfo` and labels it `previous_block_l2_gas_price`, using it as the reference for the minimum-price threshold check. A developer TODO at line 229 explicitly flags that `next_l2_gas_price` should be used instead: [2](#0-1) 

`run_validate_entry_point` also builds its `BlockContext` from the same stale `BlockInfo` (only bumping `block_number`, not the gas price), so the blockifier's own `check_fee_bounds` inside gateway validation also runs against the wrong price: [3](#0-2) 

The block header struct confirms both fields exist and are distinct: [4](#0-3) 

The `SyncStateReaderFactory` confirms the gateway always snapshots the latest committed block, so `next_l2_gas_price` is always available but never forwarded: [5](#0-4) 

---

### Impact Explanation

**When gas price is rising** (`next_l2_gas_price > l2_gas_price`):

- The gateway threshold is `min_gas_price_percentage% × l2_gas_price`.
- A transaction with `max_price_per_unit` in the range `[threshold, next_l2_gas_price)` passes both `validate_resource_bounds` and the gateway's blockifier validation (which also uses the stale price).
- The batcher then rejects the same transaction during block building because `max_price_per_unit < next_l2_gas_price` fails `check_fee_bounds`.
- **Result**: invalid transactions are admitted to the mempool and wasted block-building cycles occur.

**When gas price is falling** (`next_l2_gas_price < l2_gas_price`):

- The gateway threshold is higher than it should be.
- Transactions with `max_price_per_unit` in `[threshold_correct, threshold_stale)` are rejected by the gateway even though they would succeed in the next block.
- **Result**: valid transactions are rejected before sequencing.

Both directions match the "High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing" impact category.

---

### Likelihood Explanation

The EIP-1559 mechanism adjusts `next_l2_gas_price` every block based on gas consumption relative to the target. During periods of sustained high or low load the price drifts monotonically, making the stale-price window persistent rather than transient. The discrepancy is present on every transaction submission whenever `next_l2_gas_price ≠ l2_gas_price`, which is the normal operating condition of a live fee market.

---

### Recommendation

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` should populate the L2 gas price slot from `block_header.next_l2_gas_price` rather than `block_header.l2_gas_price`:

```rust
// current (wrong)
l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,

// correct
l2_gas_price: block_header.next_l2_gas_price.try_into()?,
```

The same correction is needed for the wei denomination. This directly resolves the TODO at line 229 of `stateful_transaction_validator.rs` and ensures both the explicit threshold check and the blockifier's `check_fee_bounds` inside gateway validation use the price that will actually govern the next block.

---

### Proof of Concept

1. Observe that `StorageBlockHeader` stores both `l2_gas_price` (price used in the committed block) and `next_l2_gas_price` (price for the next block).
2. Observe that `get_block_info_from_sync_client` reads only `l2_gas_price` and discards `next_l2_gas_price`.
3. Submit a transaction with `max_price_per_unit = l2_gas_price` (passes gateway threshold when `min_gas_price_percentage = 100`).
4. If the fee market has increased the price (`next_l2_gas_price > l2_gas_price`), the batcher's `check_fee_bounds` will reject the transaction during block building because `max_price_per_unit < next_l2_gas_price`.
5. The transaction was admitted by the gateway but is unsequenceable — matching the "admission accepts invalid transactions" impact.
