### Title
Unprivileged Caller Can Permanently Freeze Consensus Rewards by Consuming `last_reward_block` Slot via `disable_rewards=true` - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function has no access control and writes the global `last_reward_block` **before** checking the `disable_rewards` flag. Any unprivileged caller can invoke `update_rewards(any_active_staker, disable_rewards=true)` every block to consume the block's reward slot without distributing rewards, permanently preventing all stakers from receiving consensus rewards.

---

### Finding Description

The vulnerability class from the external report is a **one-directional state-transition bug**: a periodic update function can move state in only one direction (deactivate but not activate). The analog here is that `update_rewards` can consume the block's reward slot (preventing rewards) but can never release it.

`update_rewards` is part of `IStakingRewardsManager` and enforces a single reward distribution per block via the global `last_reward_block` storage variable. Critically, this variable is written **before** the `disable_rewards` guard: [1](#0-0) 

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
```

The function's only entry guards are `general_prerequisites()` (unpaused + caller non-zero) — there is no role check: [2](#0-1) 

The global `last_reward_block` is a single slot shared across all stakers: [3](#0-2) 

Once written in block N, every subsequent call in block N fails with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The attacker only needs a valid active staker with non-zero balance, which is trivially discoverable from the public `stakers` vector: [5](#0-4) 

The staker's balance check that must pass before the write: [6](#0-5) 

---

### Impact Explanation

An attacker calling `update_rewards(any_active_staker, disable_rewards=true)` in every block sets `last_reward_block` to the current block without distributing any rewards. All legitimate calls in the same block revert. Repeated indefinitely, this **permanently freezes unclaimed consensus yield** for every staker in the protocol.

This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

High. The attack requires:
- No privileged role or capital
- One transaction per block (cheap on Starknet)
- Any active staker address (publicly readable from `stakers` vector)

The attacker has no profit motive but causes severe, sustained damage to all stakers.

---

### Recommendation

1. **Add access control**: Restrict `update_rewards` to a trusted caller (e.g., the consensus mechanism or a dedicated role), mirroring how `update_rewards_from_attestation_contract` is restricted via `assert_caller_is_attestation_contract`.
2. **Alternatively, move the write**: Place `last_reward_block.write(current_block_number)` after the `disable_rewards` check so that a `disable_rewards=true` call does not consume the block slot.

---

### Proof of Concept

1. Attacker reads any active staker address from the public `stakers` vector.
2. Each block, attacker calls `update_rewards(staker_address, disable_rewards=true)`.
3. `last_reward_block` is set to the current block number; function returns early — no rewards distributed.
4. Any legitimate `update_rewards` call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeated every block → all stakers permanently receive zero consensus rewards.
