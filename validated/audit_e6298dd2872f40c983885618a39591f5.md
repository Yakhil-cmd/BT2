### Title
Bridged Cadence Assets Permanently Locked in Escrow When EVM-Side Tokens Are Transferred to Non-COA Addresses - (`File: fvm/evm/stdlib/contract.cdc`)

### Summary
The Flow EVM bridge has a structural asymmetry: Cadence assets can be bridged to any EVM address (including non-COA EOAs), but the only path to bridge assets back from EVM to Cadence requires a `CadenceOwnedAccount` (COA) resource. When bridged ERC-20 or ERC-721 tokens are transferred within EVM to a non-COA address (via `EVM.run()`), the corresponding Cadence-side escrowed assets become permanently locked with no redemption path.

### Finding Description
The `BridgeAccessor` interface defines two asymmetric sets of operations:

**Deposit (unrestricted):** `depositNFT` and `depositTokens` accept any `EVMAddress` as destination with no restriction on whether it is a COA. [1](#0-0) 

**Withdrawal (COA-restricted):** `withdrawNFT` and `withdrawTokens` require `caller: auth(Call) &CadenceOwnedAccount` — only a COA resource holder can invoke the bridge back to Cadence. [2](#0-1) 

The `CadenceOwnedAccount.depositTokens` and `depositNFT` methods always deposit to the COA's own EVM address (`self.address()`), so the initial bridge deposit lands at a COA address. However, `EVM.run()` is `access(all)` and allows any unprivileged caller to submit a signed EVM transaction: [3](#0-2) 

A standard ERC-20 `transfer()` or ERC-721 `transferFrom()` call submitted via `EVM.run()` moves the bridged tokens from the COA's EVM address to any arbitrary EOA. Once at a non-COA address, the holder has no `CadenceOwnedAccount` resource and therefore cannot call `withdrawTokens` or `withdrawNFT`. The Cadence-side escrow (`FlowEVMBridgeTokenEscrow`, `FlowEVMBridgeNFTEscrow`) continues to hold the original Cadence assets indefinitely.

The `Account` interface itself documents this restriction: [4](#0-3) 

### Impact Explanation
**Cross-VM asset loss / bridge escrow mis-accounting.** When bridged Cadence fungible tokens or NFTs are transferred within EVM to a non-COA address:

1. The Cadence-side escrow permanently holds the original assets — they cannot be released because the only release trigger is a successful `withdrawTokens`/`withdrawNFT` call from a COA.
2. The EVM-side token holder at the non-COA address has no protocol-level path to redeem the underlying Cadence asset.
3. This creates a two-tier EVM token market: COA-held tokens are redeemable for their Cadence backing; non-COA-held tokens are not, despite being identical ERC-20/ERC-721 tokens on-chain.
4. The Cadence escrow balance diverges from the circulating EVM token supply that is actually redeemable, undermining the bridge's accounting invariant.

### Likelihood Explanation
**High.** `EVM.run()` is callable by any unprivileged transaction sender with no special role or key. Standard ERC-20/ERC-721 transfers are the most common EVM operation. Any user who receives bridged tokens at a COA address and then transfers them to a regular EOA (e.g., to a centralized exchange deposit address, a hardware wallet EOA, or any non-Flow EVM wallet) triggers this condition. This is a routine, expected user action in any EVM ecosystem and requires no special knowledge or attack setup.

### Recommendation
1. **Enforce COA-only destinations at bridge deposit time:** In `BridgeAccessor.depositNFT` and `depositTokens`, verify that the `to` EVM address corresponds to a registered COA before completing the bridge operation.
2. **Alternatively, implement a permissionless bridge-back path:** Allow any EVM address that holds the bridged ERC-20/ERC-721 tokens to initiate a bridge-back by submitting a signed EVM transaction (via `EVM.run()`) that calls a bridge precompile or contract, specifying a Cadence recipient address. This removes the COA requirement for the return path.
3. **At minimum, emit a clear warning** in `depositNFT`/`depositTokens` documentation and in the bridge UI that EVM-side transfers to non-COA addresses permanently forfeit the Cadence redemption path.

### Proof of Concept

**Step 1 — Bridge Cadence FT to EVM (legitimate operation):**
```cadence
// User calls on their COA:
myCOA.depositTokens(vault: <-myFungibleTokenVault, feeProvider: feeRef)
// Result: Cadence FT escrowed, ERC-20 minted to myCOA.address() in EVM
```

**Step 2 — Transfer EVM tokens to a non-COA address via `EVM.run()`:**
```cadence
// Any unprivileged sender submits a signed ERC-20 transfer tx:
let result = EVM.run(tx: signedERC20TransferToEOA, coinbase: someAddress)
// Result: ERC-20 tokens now at eoaAddress (non-COA) in EVM
```

**Step 3 — Attempt bridge-back from non-COA address (fails — no path exists):**
```cadence
// eoaAddress holder has no CadenceOwnedAccount resource.
// withdrawTokens() requires: caller: auth(Call) &CadenceOwnedAccount
// There is no alternative bridge-back entry point.
// The Cadence-side escrow is permanently locked.
```

Root cause — `withdrawTokens` requires a COA caller: [5](#0-4) 

Root cause — `BridgeAccessor.withdrawTokens` interface enforces COA caller: [6](#0-5) 

Root cause — `EVM.run()` is unrestricted, enabling EVM-native transfers to any address: [3](#0-2)
