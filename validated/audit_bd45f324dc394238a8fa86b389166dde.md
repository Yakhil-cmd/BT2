### Title
`rerandomize_presign` Does Not Consume `PresignOutput`, Enabling Nonce Reuse and Private Key Extraction — (File: `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

`RerandomizedPresignOutput::rerandomize_presign` accepts `presignature: &PresignOutput` (a shared reference) rather than consuming the `PresignOutput` by value. This means the presignature is never invalidated after rerandomization. A caller can invoke `rerandomize_presign` on the same `PresignOutput` for two different messages, reusing the underlying nonce material `(k, sigma)`. Two signatures produced from the same presignature expose a linear system that allows full private key extraction.

---

### Finding Description

The library's own documentation is explicit:

> "Each output is consumed **exactly once** (one-time use)."
> "It's **critical** that the output is then destroyed." [1](#0-0) [2](#0-1) 

Despite this, the public API for consuming a presignature is:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← shared reference, NOT moved/consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
``` [3](#0-2) 

Because `presignature` is taken by reference, Rust's ownership system does **not** drop or zeroize the `PresignOutput` after the call. The `ZeroizeOnDrop` derive on `PresignOutput` only fires when the value is actually dropped: [4](#0-3) 

The caller retains full ownership of the original `PresignOutput` and can call `rerandomize_presign` again with a different `RerandomizationArguments` (i.e., a different message hash `h`), producing a second `RerandomizedPresignOutput` backed by the same underlying nonce `k`.

The same structural issue exists in the robust ECDSA path, where `PresignOutput` similarly carries secret nonce shares `(c, e, alpha, beta)`: [5](#0-4) 

---

### Impact Explanation

**Critical — Extraction of private signing shares.**

When the same `PresignOutput` is rerandomized for two messages `h1` and `h2` with rerandomization scalars `delta1` and `delta2`, the two resulting signatures satisfy:

```
s1 * delta1 = (h1 + R1x * tweak) * k  +  R1x * sigma
s2 * delta2 = (h2 + R2x * tweak) * k  +  R2x * sigma
```

All quantities on the left and the coefficients `h1, h2, R1x, R2x, delta1, delta2, tweak` are public. This is a 2×2 linear system in the secret unknowns `k` and `sigma`. Solving:

```
k = (R2x * s1 * delta1 - R1x * s2 * delta2) / (R2x * h1 - R1x * h2)
```

With `k` recovered, `sigma` follows immediately, and since `sigma` encodes `x * k` (the product of the private key share and nonce), the private key `x` is directly exposed. This matches the allowed impact: **extraction/reconstruction of private signing shares or nonce material**.

---

### Likelihood Explanation

Any integrator who stores a `PresignOutput` in a pool and calls `rerandomize_presign` for each signing request — a natural and documented usage pattern — will silently reuse presignatures. The library provides no runtime guard, no consumption enforcement, and no warning at the call site. The `ZeroizeOnDrop` gives a false sense of security because it only fires on drop, not on borrow. The mistake is easy to make and hard to detect without auditing caller code. [6](#0-5) 

---

### Recommendation

Change `rerandomize_presign` to consume the `PresignOutput` by value, so Rust's ownership system enforces one-time use at compile time:

```rust
// Before (unsafe — does not consume):
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// After (safe — consumed and zeroized on drop):
pub fn rerandomize_presign(
    presignature: PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Apply the same fix to the robust ECDSA `rerandomize_presign` counterpart. This mirrors the pattern described in the orchestration docs: the output must be destroyed after use, and the API should enforce this structurally. [7](#0-6) 

---

### Proof of Concept

1. Party `P_i` runs presigning and obtains `presign_out: PresignOutput` containing secret `(k_i, sigma_i)`.
2. Coordinator calls `rerandomize_presign(&presign_out, &args_msg1)` → `rp1` for message `h1`. **`presign_out` is still alive.**
3. Coordinator calls `rerandomize_presign(&presign_out, &args_msg2)` → `rp2` for message `h2`. **Same `k_i, sigma_i` used again.**
4. Both signing rounds complete, producing `(R1, s1)` and `(R2, s2)`.
5. Attacker observes both signatures. Since `delta1, delta2, R1, R2, h1, h2, tweak` are all public, the attacker solves the linear system above to recover `k` and `sigma`, then extracts the private key `x`. [8](#0-7) [4](#0-3)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
