### Title
`protocolUnpausePool` Always Transitions to Admin-Paused (Level 1) Instead of Active (Level 0), Leaving Pool Permanently Swap-Blocked After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`protocolUnpausePool()` is designed to reverse a protocol-level emergency pause, but it unconditionally sets `pauseLevel` to `1` (admin-paused) rather than `0` (active). When the protocol pauses a pool that was previously active (level 0 → 2), calling `protocolUnpausePool()` moves it to level 1 — still paused — instead of restoring it to active. Swaps remain blocked and the pool admin must separately call `unpausePool()` to fully restore operation, a step the protocol owner has no way to force.

### Finding Description

The pause-level state machine is:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolPausePool()` accepts pools at level 0 **or** 1 and moves them to level 2: [1](#0-0) 

`protocolUnpausePool()` requires level 2 and **always** sets level 1, never 0: [2](#0-1) 

When the pool was at level 0 before the protocol pause, the round-trip is:

```
0 (active) → protocolPausePool → 2 → protocolUnpausePool → 1 (admin-paused)
```

The pool is still paused. `swap()` carries `whenNotPaused`, which reverts on any non-zero `pauseLevel`: [3](#0-2) [4](#0-3) 

The protocol owner has no path to set level 0 directly — `unpausePool()` is `onlyPoolAdmin`, not `onlyOwner`: [5](#0-4) 

So after `protocolUnpausePool()`, the protocol owner believes the pool is restored, but swaps remain blocked until the pool admin independently calls `unpausePool()`.

### Impact Explanation

Swaps are the core revenue-generating operation of every pool. After `protocolUnpausePool()` the pool is still at level 1, so every `swap()` call reverts with `PoolPaused`. LPs cannot trade, arbitrageurs cannot rebalance, and the pool accumulates price divergence from the oracle. This is an unusable swap flow — a broken core pool functionality impact within the allowed gate.

### Likelihood Explanation

The most common emergency scenario is pausing an **active** pool (level 0 → 2). Every such pause followed by `protocolUnpausePool()` triggers the bug. The protocol owner has no indication from the transaction that the pool is still paused; the call succeeds and emits `PauseLevelUpdated(2, 1)`. The pool admin may be a multisig with its own latency, extending the outage indefinitely.

### Recommendation

`protocolUnpausePool()` should restore the pool to level 0 (active), not level 1. If the intent is to hand control back to the admin, the function should be renamed and documented accordingly, and a separate `protocolFullUnpausePool()` that sets level 0 should be provided. The simplest correct fix:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0); // restore to active
}
```

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` — pool moves to level 2.
3. Protocol owner calls `protocolUnpausePool(pool)` — pool moves to level **1**, not 0.
4. Any user calls `pool.swap(...)` — reverts with `PoolPaused`.
5. Protocol owner has no function to set level 0; only the pool admin's `unpausePool()` can do so.
6. If the pool admin is unavailable (multisig, key rotation, etc.), swaps are permanently blocked despite the protocol owner having "unpaused" the pool. [6](#0-5)
