### Title
Gateway L2 Gas Price Threshold Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Mempool Admission/Rejection Divergence from Blockifier - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`validate_resource_bounds` in the gateway stateful validator reads the **current block's** `l2_gas_price` from the block header to threshold-check incoming transactions. However, the batcher executes those same transactions against the **next block's** L2 gas price (`next_l2_gas_price`), which is an EIP-1559-adjusted value stored separately in the block header. The two values diverge whenever blocks are over- or under-utilized. A TODO comment in the code explicitly acknowledges this is wrong. The result is that the gateway admits transactions that the blockifier will reject during pre-validation (or rejects transactions the blockifier would accept), breaking the admission invariant.

---

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` by reading `block_header.l2_gas_price.price_in_fri` — the price that was used **inside** the latest finalized block: [1](#0-0) 

This `BlockInfo` is then consumed by `validate_resource_bounds`, which extracts `strk_gas_prices.l2_gas_price` and calls it `previous_block_l2_gas_price`: [2](#0-1) 

The TODO comment at line 229 explicitly flags the bug: [3](#0-2) 

The threshold check then admits any transaction whose `max_price_per_unit >= (min_gas_price_percentage / 100) * block_N.l2_gas_price`: [4](#0-3) 

Meanwhile, the batcher builds the next block using `next_l2_gas_price` — the EIP-1559-adjusted price stored in the block header — as the block's actual L2 gas price: [5](#0-4) 

The blockifier's `check_fee_bounds` then enforces `tx.max_price_per_unit >= block_context.block_info.gas_prices.strk_gas_prices.l2_gas_price`, which equals `block_N.next_l2_gas_price`: [6](#0-5) 

`next_l2_gas_price` is computed by `calculate_next_base_gas_price` (EIP-1559): when blocks are full it rises; when under-utilized it falls: [7](#0-6) 

The two prices are structurally distinct fields in the block header: [8](#0-7) 

---

### Impact Explanation

**Scenario A — Rising price (blocks over-utilized):**
`next_l2_gas_price > l2_gas_price`. The gateway admits transactions with `max_price_per_unit` in the range `[threshold * l2_gas_price, next_l2_gas_price)`. These transactions enter the mempool but fail `check_fee_bounds` during blockifier pre-validation in the batcher, wasting mempool capacity and causing user-visible failures for transactions the gateway explicitly accepted.

**Scenario B — Falling price (blocks under-utilized) with `min_gas_price_percentage = 100`:**
`next_l2_gas_price < l2_gas_price`. The gateway rejects transactions with `max_price_per_unit` in `[next_l2_gas_price, l2_gas_price)`. These transactions are valid for the next block but are incorrectly turned away at the gateway.

Both scenarios match: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The EIP-1559 mechanism adjusts the L2 gas price every block. Any sustained period of high or low block utilization causes `next_l2_gas_price` to diverge from `l2_gas_price`. This is the normal operating condition of the network, not an edge case. The divergence is bounded per block by `gas_price_max_change_denominator` but accumulates across consecutive full or empty blocks. No special attacker capability is required — any user submitting a transaction during a price-change period can trigger the mismatch.

---

### Recommendation

In `validate_resource_bounds`, read `next_l2_gas_price` from the block header instead of `l2_gas_price`. The `GatewayFixedBlockStateReader` trait and its `GatewayFixedBlockSyncStateClient` implementation should expose `next_l2_gas_price` (already present in `BlockHeaderWithoutHash`) so the validator can compare the transaction's `max_price_per_unit` against the price that will actually govern the next block's execution. This directly resolves the acknowledged TODO at line 229.

---

### Proof of Concept

1. Observe the latest finalized block has `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (blocks have been full for several consecutive blocks).

2. Submit an `InvokeV3` transaction with `AllResourceBounds` where `l2_gas.max_price_per_unit = X` such that:
   - `X >= (min_gas_price_percentage / 100) * P` → passes `validate_tx_l2_gas_price_within_threshold`
   - `X < P'` → fails `check_fee_bounds` in the blockifier

3. The gateway accepts the transaction and returns a transaction hash to the user.

4. The batcher builds the next block using `P'` as the L2 gas price. When it pulls the transaction from the mempool and calls `perform_pre_validation_stage`, `check_fee_bounds` fires `ResourceBoundsError::MaxGasPriceTooLow` for L2Gas and the transaction is dropped.

5. The user's transaction was accepted by the gateway but never sequenced — the admission decision was wrong.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
```
