### Title
`FeeEstimation` RPC Response Omits `l2_gas_consumed`, Making `overall_fee` Irreconcilable and Preventing Correct Resource-Bound Setting - (File: crates/apollo_rpc_execution/src/objects.rs)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` includes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is computed from the actual receipt fee (which correctly includes L2 gas costs), yet the struct's own documentation states it "equals to `gas_consumed * gas_price + data_gas_consumed * data_gas_price`" — an equation that is false whenever L2 gas is non-zero. Clients cannot reconstruct `overall_fee` from the returned fields, cannot determine the correct `l2_gas.max_amount` to set in their resource bounds, and will submit transactions that are rejected at execution time.

### Finding Description

`tx_execution_output_to_fee_estimation` in `crates/apollo_rpc_execution/src/objects.rs` builds the `FeeEstimation` response:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_price,                                          // price present …
    overall_fee: tx_execution_output.execution_info.receipt.fee,  // … but l2_gas_consumed absent
    unit: tx_execution_output.price_unit,
})
```

`receipt.fee` is the true fee: `l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * l2_gas_price`. The struct exposes `l2_gas_price` but never `l2_gas_consumed`, so the L2 component is silently folded into `overall_fee` with no way for the caller to extract it. The in-code TODO acknowledges this explicitly:

```
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

The OpenRPC schema at `crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json` likewise describes `overall_fee` as "equals to `gas_consumed*gas_price + data_gas_consumed*data_gas_price`" — omitting the L2 term entirely. The transaction-prover service's own fixture files show the correct decomposition (`l2_gas_consumed: "0xb56b6"`, `l2_gas_price: "0x1dcd65000"`, `overall_fee: "0x151eb86f3ed400"`), confirming the RPC path is the outlier.

The broken invariant is:

> `overall_fee == gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`

This invariant is stated as true in both the struct comment and the OpenRPC spec, but is false for any V3 transaction that consumes L2 gas.

### Impact Explanation

**High — RPC fee estimation returns an authoritative-looking wrong value.**

A wallet or SDK calls `starknet_estimateFee`, receives `overall_fee = X`, and attempts to verify or derive `l2_gas.max_amount` from the returned fields. Because `l2_gas_consumed` is absent, the client computes:

```
apparent_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price
             < overall_fee   (by the hidden l2_gas * l2_gas_price term)
```

The client either:
1. Sets `l2_gas.max_amount = 0` (or any value derived from the incomplete breakdown), causing the transaction to revert at `check_fee_bounds` / `PostExecutionReport` with `MaxGasAmountExceeded` for L2Gas.
2. Trusts `overall_fee` but cannot set a tight `l2_gas.max_amount`, so it either over-provisions (economic waste) or under-provisions (execution failure).

This is the direct Starknet analog of the external report's hardcoded gas limit: the estimation API does not return the information needed to correctly size the gas limit, causing transactions to fail.

### Likelihood Explanation

Every V3 (`AllResources`) transaction on a network running Starknet ≥ 0.14 consumes non-zero L2 gas. The `starknet_estimateFee` and `starknet_simulateTransactions` endpoints are the canonical pre-submission tools for all wallets and SDKs. Any client that follows the documented formula to reconstruct or verify `overall_fee` will observe a discrepancy, and any client that derives `l2_gas.max_amount` from the returned fields alone will produce an under-specified transaction.

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it in `tx_execution_output_to_fee_estimation`:

```rust
l2_gas_consumed: gas_vector.l2_gas.0.into(),
```

Update the struct comment and the OpenRPC `FEE_ESTIMATE` schema to reflect the correct formula:

```
overall_fee = gas_consumed * l1_gas_price
            + data_gas_consumed * l1_data_gas_price
            + l2_gas_consumed * l2_gas_price
```

### Proof of Concept

1. Submit any V3 invoke transaction to `starknet_estimateFee`.
2. Observe the response: `overall_fee` is non-zero, `gas_consumed` and `data_gas_consumed` may be zero or small, `l2_gas_consumed` is absent.
3. Compute `apparent_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`.
4. Observe `apparent_fee < overall_fee`; the difference is `l2_gas * l2_gas_price`.
5. Construct a V3 transaction with `l2_gas.max_amount` derived from `apparent_fee / l2_gas_price` (the only derivable value); submit it.
6. The transaction is rejected by `check_fee_bounds` with `MaxGasAmountTooLow { resource: L2Gas }` because the actual L2 gas required exceeds the bound set from the incomplete estimation.

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
