### Title
Gateway Admits Signature-Invalid Invoke Transactions via `skip_stateful_validations` Race Window — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point check for an invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that check returns `true` as soon as **any** transaction for the address is in the pool — not specifically a `deploy_account` — an unprivileged attacker who observes a victim's `deploy_account(nonce=0)` entering the mempool can immediately submit a signature-invalid `invoke(nonce=1)` for the same address. The gateway admits it without calling `__validate__`, it occupies the (address, nonce=1) slot in the mempool, and the victim's legitimate `invoke(nonce=1)` is subsequently rejected as `DuplicateNonce`. The attack is free (rejected transactions pay no fees and produce no nonce increment) and can be repeated indefinitely.

---

### Finding Description

**Relevant code — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip `__validate__`) when:
1. The transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)` (nonce=1)
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`

The comment claims condition 4 is sufficient because it implies "either it has a deploy_account transaction or transactions with future nonces that passed validations." This is the broken invariant: the check returns `true` for **any** transaction in the pool for that address, not specifically a `deploy_account`.

**`account_tx_in_pool_or_recent_block` implementation:** [2](#0-1) 

It returns `true` if the address appears in either the pool or the committed-state map — with no type filter.

**Gateway validation flow:** [3](#0-2) 

`run_pre_validation_checks` calls `validate_by_mempool` (nonce-gap check only) and then `skip_stateful_validations`. When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

**Batcher re-executes with `validate: true`:**

When the batcher converts a transaction for sequencing it calls `new_for_sequencing`, which always sets `validate: true`: [5](#0-4) 

So the attacker's invalid invoke reaches the blockifier with `validate: true`, `__validate__` is called, it fails (invalid signature), and the transaction is **rejected** — no nonce increment, no fee payment.

**Attack steps:**

1. Victim submits `deploy_account(nonce=0)` for address A → admitted to mempool.
2. Attacker observes the mempool, submits `invoke(nonce=1, sender=A, signature=GARBAGE)`.
3. Gateway: `account_nonce=0`, `tx_nonce=1`, `account_tx_in_pool_or_recent_block(A)=true` (victim's `deploy_account` is in pool) → `skip_validate=true` → `__validate__` skipped → attacker's invoke admitted.
4. Victim submits legitimate `invoke(nonce=1, sender=A)` → mempool rejects with `DuplicateNonce`.
5. Batcher executes `deploy_account(nonce=0)` → account A deployed.
6. Batcher executes attacker's `invoke(nonce=1)` → `__validate__` fails → **rejected** (no state change, no fee).
7. Victim resubmits `invoke(nonce=1)` → admitted.
8. Attacker repeats from step 2 — indefinitely, at zero cost.

---

### Impact Explanation

**Matching impact:** *High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

The gateway admits a signature-invalid invoke transaction into the mempool. This occupies the `(address, nonce=1)` slot, causing the victim's legitimate first post-deploy invoke to be rejected with `DuplicateNonce`. The attacker pays no fees (rejected transactions are rolled back entirely). The attack can be sustained indefinitely, permanently preventing the victim's first invoke from being sequenced until the attacker stops.

---

### Likelihood Explanation

Any unprivileged network participant who can observe the mempool (e.g., via the gateway's public RPC) can execute this attack. The only requirement is reacting to a `deploy_account` submission before the victim's `invoke(nonce=1)` arrives. Because the gateway processes transactions sequentially and the UX pattern explicitly sends `deploy_account` first, there is a reliable race window. No special privileges, keys, or resources are required.

---

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction (not just any transaction) is present in the mempool for the sender address. Expose a dedicated `has_deploy_account_in_pool(address)` method on the mempool that filters by transaction type. This closes the race window: an attacker cannot submit a `deploy_account` for the victim's address (it would fail `__validate_deploy__` at the gateway), so the type-specific check cannot be spoofed.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
