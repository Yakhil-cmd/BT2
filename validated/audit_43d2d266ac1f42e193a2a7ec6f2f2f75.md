### Title
Presignature Reuse Not Enforced at API Level Enables Private Key Extraction — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both ECDSA schemes expose `rerandomize_presign` as a function that accepts a **shared reference** (`&PresignOutput`) rather than consuming ownership (`PresignOutput`). Combined with `PresignOutput` deriving `Clone`, the library provides no type-system or runtime enforcement of the documented "one-time use" invariant for presignatures. A malicious coordinator can rerandomize the same presignature with different `(msg_hash, tweak)` pairs, run two signing sessions, and recover the aggregate private key via standard ECDSA nonce-reuse algebra.

---

### Finding Description

The documentation is explicit that presignatures are one-time-use material:

> *"It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase."*
> — `docs/ecdsa/ot_based_ecdsa/orchestration.md`, lines 70–71

> *"**Never reuse a presignature**, even across failed, aborted, or partially completed signing sessions."*
> — `docs/ecdsa/robust_ecdsa/signing.md`, line 176

Despite this, the API design makes reuse trivial:

**OT-based ECDSA** — `src/ecdsa/ot_based_ecdsa/mod.rs`:

```rust
// PresignOutput is Clone — can be duplicated freely
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }

// rerandomize_presign takes a *reference* — original is never consumed
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // <-- shared reference, not ownership
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
``` [1](#0-0) [2](#0-1) 

**Robust ECDSA** — `src/ecdsa/robust_ecdsa/mod.rs`:

```rust

pub struct PresignOutput { ... }

pub fn rerandomize_presign(
    presignature: &PresignOutput,   // <-- shared reference, not ownership
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
``` [3](#0-2) [4](#0-3) 

Because `rerandomize_presign` takes `&PresignOutput`, the Rust borrow checker never moves or drops the presignature. A caller can invoke `rerandomize_presign` on the same `PresignOutput` an arbitrary number of times with different `RerandomizationArguments`, producing multiple distinct `RerandomizedPresignOutput` values that all share the same underlying nonce `R`. Each can then be passed to a separate `sign()` call.

The robust ECDSA documentation explicitly names the consequence:

> *"If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."*
> — `docs/ecdsa/robust_ecdsa/signing.md`, lines 152–154 [5](#0-4) 

The library enforces `N1 = N2 = 2t+1` to prevent split-view attacks that use *different participant subsets*, but it does **not** prevent the same presignature from being rerandomized twice with different `(msg_hash, tweak)` values and used in two signing sessions with the *same* participant set. The `N1 = N2` guard is therefore insufficient to close the attack surface. [6](#0-5) 

---

### Impact Explanation

Two signatures `(R', s₁)` and `(R', s₂)` produced from the same presignature nonce (even after rerandomization with the same `delta`) satisfy a linear relation in the secret key. Standard ECDSA nonce-reuse recovery yields the aggregate private key `x` directly. This constitutes **Critical** impact: extraction of the aggregate secret signing material.

---

### Likelihood Explanation

The malicious coordinator is an explicitly modeled adversary in the robust ECDSA threat model (the documentation dedicates a full security-considerations section to coordinator-driven split-view attacks). The coordinator controls `RerandomizationArguments` construction and can call `rerandomize_presign` twice on the same `PresignOutput` before initiating two signing sessions. No out-of-band coordination or cryptographic break is required — the attack is a straightforward API misuse that the type system permits. [7](#0-6) 

---

### Recommendation

1. **Change `rerandomize_presign` to consume ownership** — accept `PresignOutput` (by value) instead of `&PresignOutput`. Rust's move semantics will then statically prevent a second rerandomization of the same presignature value.

2. **Remove `Clone` from `PresignOutput`** — deriving `Clone` on both `ot_based_ecdsa::PresignOutput` and `robust_ecdsa::PresignOutput` allows callers to duplicate the value before passing it, circumventing the ownership-consumption fix above. Removing `Clone` closes this escape hatch. [1](#0-0) [3](#0-2) 

---

### Proof of Concept

```rust
// Attacker is the coordinator; presig was produced by an honest presigning round.
let presig: PresignOutput = run_presigning(...);

// Rerandomize the SAME presignature twice with different (msg_hash, tweak).
// This compiles and runs without error because rerandomize_presign takes &PresignOutput.
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presig, &args_for_msg1)?;
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presig, &args_for_msg2)?;

// Run two signing sessions — both use the same underlying nonce R.
let sig1 = run_signing(participants, rerand1, msg_hash1);
let sig2 = run_signing(participants, rerand2, msg_hash2);

// sig1 and sig2 share a multiplicatively related nonce.
// Standard ECDSA nonce-reuse algebra recovers the private key x.
```

The `rerandomize_presign` call on line 66 of `src/ecdsa/ot_based_ecdsa/mod.rs` and line 55 of `src/ecdsa/robust_ecdsa/mod.rs` are the necessary vulnerable steps; no external assumption or cryptographic break is required. [8](#0-7) [9](#0-8)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }

```markdown
# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
