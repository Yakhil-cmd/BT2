### Title
No Minimum `priceProviderTimelock` Allows Pool Admin to Atomically Bypass Oracle Timelock and Execute Bad-Price Swaps — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool` accepts `priceProviderTimelock = 0` without any minimum-value guard. When the timelock is zero, `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` can be called in the same transaction (or same block), making the timelock protection completely ineffective. A pool admin can atomically rotate the pool's oracle to a malicious price provider and immediately execute swaps at manipulated prices, draining LP principal.

---

### Finding Description

`_validatePoolParameters` enforces fee caps, non-zero tokens, non-zero admin, and non-zero initial amounts, but applies **no lower-bound check** on `priceProviderTimelock`:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol
function _validatePoolParameters(PoolParameters calldata params) internal view {
    // ... token, admin, fee checks ...
    // NO check: priceProviderTimelock >= MIN_TIMELOCK
}
```

The only special-case for `priceProviderTimelock` is the immutable-mode sentinel:

```solidity
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
```

Any value below `type(uint256).max` — including **zero** — creates a mutable-provider pool and is stored verbatim in `priceProviderTimelock[pool]`.

In `proposePoolPriceProvider`, the deadline is computed as:

```solidity
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp when timelock == 0
pendingPriceProviderExecuteAfter[pool] = executeAfter;
```

In `executePoolPriceProviderUpdate`, the guard is:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
```

When `execAfter == block.timestamp`, the condition `block.timestamp < block.timestamp` is **false**, so the revert is never triggered. Both calls succeed in the same transaction.

---

### Impact Explanation

The pool admin is described as "semi-trusted only inside caps and timelocks." The timelock is the sole constraint on oracle rotation. With `priceProviderTimelock = 0`, the admin can:

1. Deploy a malicious `IPriceProvider` that returns an arbitrarily skewed bid/ask (e.g., bid ≈ 0, ask ≈ ∞ for one direction).
2. In a single transaction: call `proposePoolPriceProvider(pool, maliciousOracle)` → call `executePoolPriceProviderUpdate(pool)` → call `swap(...)`.
3. The swap executes against the malicious oracle's prices, allowing the admin (or a colluding party) to extract all LP token balances at near-zero cost.

This is a direct loss of LP principal — the pool's token balances fail to cover LP claims after the manipulated swap settles.

---

### Likelihood Explanation

- `createPool` is **permissionless** once `poolDeployer` is set; any caller can create a pool with `priceProviderTimelock = 0`.
- The pool admin role is assigned at creation and can be any address, including the pool creator themselves.
- No off-chain monitoring can prevent the atomic propose+execute+swap sequence from completing within a single block.
- The only prerequisite is that LPs have deposited liquidity into the pool — a normal operational state.

---

### Recommendation

Enforce a minimum timelock in `_validatePoolParameters`:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 2 days; // example

function _validatePoolParameters(PoolParameters calldata params) internal view {
    // existing checks ...
    if (
        params.priceProviderTimelock != type(uint256).max &&
        params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
    ) revert PriceProviderTimelockTooShort();
}
```

This mirrors the pattern already used for fee caps (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) — the factory enforces upper bounds on fees and should enforce a lower bound on the timelock that protects LPs from instant oracle rotation.

---

### Proof of Concept

```
1. Factory owner deploys MetricOmmPoolFactory + MetricOmmPoolDeployer.
2. Attacker calls createPool({
       priceProvider: legitimateOracle,
       priceProviderTimelock: 0,   // <-- zero, not type(uint256).max
       admin: attacker,
       ...
   })
3. LPs call addLiquidity → pool accumulates token0/token1 balances.
4. Attacker deploys MaliciousOracle implementing IPriceProvider:
       token0() → pool.token0()
       token1() → pool.token1()
       getBidAndAskPrice() → (1, type(uint128).max - 1)  // near-zero bid
5. In one transaction, attacker calls:
       a. factory.proposePoolPriceProvider(pool, address(maliciousOracle))
          → pendingPriceProviderExecuteAfter[pool] = block.timestamp + 0 = block.timestamp
       b. factory.executePoolPriceProviderUpdate(pool)
          → block.timestamp < block.timestamp == false → no revert
          → pool.priceProvider = maliciousOracle
       c. pool.swap(...)  // executes at malicious bid/ask, drains LP balances
6. LP funds are extracted; pool balances no longer cover LP claims.
```
