### Title
Missing `msg_hash` Non-Zero Validation in OT-Based ECDSA Signing Enables Private Key Extraction via Split-View Attack - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` accepts a zero `msg_hash` without any validation. The robust ECDSA counterpart explicitly rejects `msg_hash == 0` to prevent a split-view key-extraction attack. Because the OT-based variant lacks this guard, a malicious coordinator can present `h = 0` to one threshold subset of participants and `h' ≠ 0` to another subset within the same signing session, reconstruct both `r·σ` and `h'·k + r·σ`, and algebraically recover the aggregate private key `x`.

---

### Finding Description

**Root cause — missing validation:**

`src/ecdsa/ot_based_ecdsa/sign.rs` `sign()` (lines 22–76) performs several input checks (participant count, coordinator membership, threshold size) but never checks whether `msg_hash` is zero: [1](#0-0) 

By contrast, `src/ecdsa/robust_ecdsa/sign.rs` explicitly rejects a zero hash: [2](#0-1) 

The robust ECDSA README documents the reason: [3](#0-2) 

**Signing formula and the algebraic attack:**

Each participant computes their signature share as:

```
s_i = h · k_i + r · σ_i          (Spec 1.3)
``` [4](#0-3) 

The coordinator sums Lagrange-weighted shares to obtain `s = h·k + r·σ`, where `σ = k·x` (nonce times private key).

**Split-view attack with `h = 0`:**

A malicious coordinator controlling a signing session with `2t` participants (where `t` is the threshold) proceeds as follows:

1. Present `h = 0` to participants `{P_1 … P_t}` (group A).
2. Present `h' ≠ 0` to participants `{P_{t+1} … P_{2t}}` (group B).
3. Collect all `s_i` shares. Using Lagrange interpolation on group A's `t` shares, reconstruct:
   - `A = r · σ = r · k · x`
4. Using group B's `t` shares, reconstruct:
   - `B = h' · k + r · σ`
5. Compute `B − A = h' · k`, then `k = (B − A) / h'`.
6. Recover the private key: `x = A / (r · k)`.

The OT-based ECDSA signing spec confirms the formula `s = k·(H(m) + r·x)`: [5](#0-4) 

The security documentation for the robust variant explicitly names this class of attack: [6](#0-5) 

---

### Impact Explanation

**Impact: Critical — Extraction of the aggregate private signing key.**

A malicious coordinator (a role reachable without any privileged key material — any participant can act as coordinator) can recover the full private key `x` in a single signing session by exploiting the missing `h = 0` guard. Once `x` is known, the attacker can forge arbitrary signatures for any message under the threshold group's public key, completely breaking the scheme's security guarantee.

---

### Likelihood Explanation

**Likelihood: High.**

- The coordinator role is explicitly part of the protocol and is reachable by any participant.
- The `sign` function's public API accepts `msg_hash: Scalar` with no restriction; passing `Scalar::ZERO` is trivially possible.
- The attack requires only `2t` cooperating participants (two threshold-sized subsets), which is a realistic participant count.
- The robust ECDSA variant already identified and patched this exact class of attack, confirming the threat model is understood and the OT-based variant's omission is an oversight. [7](#0-6) 

---

### Recommendation

Add a non-zero check for `msg_hash` in `src/ecdsa/ot_based_ecdsa/sign.rs`, mirroring the guard already present in the robust ECDSA variant:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid split-view key-extraction attacks".to_string(),
    ));
}
```

This should be inserted in `sign()` before the protocol future is spawned, consistent with the pattern at: [2](#0-1)

### Proof of Concept

```
Participants: P1, P2, P3, P4  (threshold t = 2, so 2t = 4 participants)
Presignature: (R, k_1, σ_1, k_2, σ_2, k_3, σ_3, k_4, σ_4)

Malicious coordinator C:
  → sends msg_hash = 0   to P1, P2  (group A)
  → sends msg_hash = h'  to P3, P4  (group B)

Group A shares (h=0):
  s_1 = 0·k_1 + r·σ_1 = r·σ_1
  s_2 = 0·k_2 + r·σ_2 = r·σ_2
  Lagrange-reconstruct: A = r·σ = r·k·x

Group B shares (h=h'):
  s_3 = h'·k_3 + r·σ_3
  s_4 = h'·k_4 + r·σ_4
  Lagrange-reconstruct: B = h'·k + r·σ

Key recovery:
  k = (B − A) / h'
  x = A / (r · k)
```

The `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` will accept `msg_hash = Scalar::ZERO` without error and proceed to execute the protocol, giving the coordinator all the shares needed to complete the attack. [8](#0-7)

    participants: &[Participant],

            participants: participants.len(),

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });

        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });

    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

4. **Do not sign with $h = 0$** (the zero message hash).
   This input enables a related algebraic split-view attack in the modified scheme when
   $N_1 > 2t + 1$.
