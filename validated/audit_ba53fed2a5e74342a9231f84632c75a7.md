### Title
Missing Validation of Participant Share Binding to Protocol `app_pk` in CKD Coordinator — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator aggregates `CKDOutput` shares received from participants without any verification that each share was computed using the agreed-upon `app_pk`. A malicious participant can substitute an arbitrary `app_pk'` in their computation and send a crafted share to the coordinator. The coordinator blindly sums all shares, producing a `CKDOutput` that cannot be correctly unmasked with the legitimate `app_sk`, permanently corrupting the derived confidential key for all honest parties.

---

### Finding Description

The CKD protocol is designed so that each participant computes a share `(norm_big_y, norm_big_c)` where:

```
big_y  = y * G
big_s  = private_share * H(pk || app_id)
big_c  = big_s + app_pk * y          ← ElGamal encryption under app_pk
```

The coordinator collects these shares and sums them to produce the final `CKDOutput`. The unmask operation is `sum_big_c - app_sk * sum_big_y = sum_big_s`, which recovers the aggregate secret only if every participant used the same `app_pk = app_sk * G`.

In `do_ckd_coordinator` at `src/confidential_key_derivation/protocol.rs` lines 50–55:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no proof of correct computation** and **no binding check** that the received `(big_y, big_c)` was formed using the protocol-agreed `app_pk`. The coordinator accepts and aggregates any `CKDOutput` values unconditionally.

This is the direct analog of the external report's `issuance.parent_bond == *bond_account_info.key` missing check: just as that function failed to verify the issuance was associated with the correct bond before performing a critical state transition, `do_ckd_coordinator` fails to verify that each participant's share is associated with the correct `app_pk` before aggregating into the final output. [1](#0-0) 

The `compute_signature_share` function shows the intended computation, where `app_pk` is used to form the ElGamal ciphertext component `big_c = big_s + app_pk * y`: [2](#0-1) 

No ZK proof or consistency check is generated or verified to bind `big_c` to the specific `app_pk` agreed upon by the protocol session. [3](#0-2) 

---

### Impact Explanation

A malicious participant sends `(big_y_m, big_c_m)` where `big_c_m = big_s_m + app_pk_attacker * y_m` (using an attacker-controlled `app_pk_attacker ≠ app_pk`). The coordinator computes:

```
sum_big_c - app_sk * sum_big_y
  = (honest_sum_big_s + big_s_m + app_pk_attacker * y_m) - app_sk * (honest_sum_big_y + y_m * G)
  = honest_sum_big_s + big_s_m + (app_pk_attacker - app_sk * G) * y_m
```

Since `app_pk_attacker ≠ app_sk * G`, the residual term `(app_pk_attacker - app_sk * G) * y_m` is non-zero and unknown to the coordinator. The resulting `CKDOutput` is permanently corrupted and cannot be correctly unmasked by any holder of `app_sk`. The derived confidential key is irrecoverably wrong.

**Impact class**: High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker needs only to be a legitimate protocol participant (no key leakage, no external compromise required). The attack requires only that the malicious participant deviate from the protocol by substituting `app_pk_attacker` in their local `compute_signature_share` call before sending to the coordinator. There is no detection mechanism in the current code.

---

### Recommendation

Add a non-interactive zero-knowledge proof of correct ElGamal encryption that binds each participant's `(big_y, big_c)` to the session-agreed `app_pk`. Specifically, each participant should prove knowledge of `y` such that `big_y = y * G` and `big_c - big_s = app_pk * y` (a standard Chaum-Pedersen proof of discrete log equality). The coordinator must verify this proof before accepting and aggregating any participant's share.

Alternatively, at minimum, include a consistency check: each participant should commit to `app_pk` as part of their message, and the coordinator should reject any share whose embedded `app_pk` does not match the session parameter.

---

### Proof of Concept

1. Honest participants `P_1, ..., P_{n-1}` run `compute_signature_share` correctly with the agreed `app_pk`.
2. Malicious participant `P_n` runs a modified `compute_signature_share` substituting `app_pk_attacker = attacker_sk * G` for `app_pk`, producing `big_c_n = big_s_n + app_pk_attacker * y_n`.
3. `P_n` sends `(norm_big_y_n, norm_big_c_n)` to the coordinator via `chan.send_private`.
4. The coordinator in `do_ckd_coordinator` receives all shares and sums them unconditionally (lines 50–55).
5. The resulting `CKDOutput` satisfies `sum_big_c - app_sk * sum_big_y ≠ msk * H(pk || app_id)`.
6. Any consumer calling `ckd_output.unmask(app_sk)` receives a garbage value, permanently denying the correct confidential derived key to all honest parties. [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
