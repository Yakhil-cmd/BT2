Audit Report

## Title
Unbounded Async Task Accumulation via `MaybeWantsLater` Stash with `SLOT_TABLE_NO_LIMIT` for HTTPS Outcalls — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

## Summary
The https-outcalls artifact type is configured with `SLOT_TABLE_NO_LIMIT` (`usize::MAX`), meaning a single Byzantine subnet peer can advertise an unbounded number of distinct slot updates, each with a `CanisterHttpResponseId` whose callback ID exceeds `highest_accepted_request_id`. Each unique artifact ID spawns a new `process_slot_update` async task that blocks indefinitely in the `MaybeWantsLater` stash path, causing unbounded memory growth on the victim replica.

## Finding Description

**Root cause — no slot limit for https-outcalls:**
`SLOT_TABLE_NO_LIMIT` is defined as `usize::MAX` and is passed as the `slot_limit` for the https-outcalls consensus manager channel. [1](#0-0) [2](#0-1) 

The slot table guard at receiver.rs line 373 is `peer_slot_table_len < self.slot_limit`. With `slot_limit = usize::MAX`, this guard is never triggered, so every new `(peer_id, slot_number)` pair is unconditionally inserted. [3](#0-2) 

**Task spawning per unique artifact ID:**
For each new artifact ID not already in `active_assembles`, a new `process_slot_update` task is spawned. The `active_assembles` map deduplicates by artifact ID, but an attacker using a distinct `CanisterHttpResponseId` per slot bypasses this entirely. [4](#0-3) 

**Bouncer returns `MaybeWantsLater` for high callback IDs:**
Any `CanisterHttpResponseId` with a callback ID above `next_callback_id + MAX_NUMBER_OF_REQUESTS_AHEAD` (345 slots) returns `MaybeWantsLater`. [5](#0-4) 

**Tasks block indefinitely in the stash:**
`should_download` drops the artifact payload but keeps the task alive, blocking on `bouncer_watcher.changed()`. The bouncer refreshes every 3 seconds; as long as the advertised IDs remain above `highest_accepted_request_id`, every refresh still returns `MaybeWantsLater`. [6](#0-5) 

**`all_peers_deleted_artifact` never fires:**
The `process_slot_update` select races `assemble_artifact` against `all_peers_deleted_artifact`. The latter only fires when the peer counter becomes empty (peer retracts) or the sender is dropped (task completes). A Byzantine peer that never retracts keeps the peer counter non-empty, and the sender in `active_assembles` is only dropped after the task completes — a circular dependency that keeps all N tasks alive indefinitely. [7](#0-6) 

**Attack flow:**
1. Byzantine peer sends N slot updates, each with a distinct `SlotNumber` (0…N−1) and a distinct `CanisterHttpResponseId` with callback ID > `next_callback_id + 345`.
2. All N entries are inserted into the slot table (no limit).
3. N `process_slot_update` tasks are spawned, each blocking in `should_download` on `bouncer_watcher.changed()`.
4. The Byzantine peer never retracts; `all_peers_deleted_artifact` never fires.
5. Memory grows O(N) with no upper bound.

## Impact Explanation
Memory exhaustion on the targeted replica node. As tasks accumulate, heap usage grows until the OS OOM-kills the process or the replica becomes unresponsive, terminating its consensus participation. This maps to **High** severity: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS." If multiple nodes are targeted simultaneously, subnet availability is directly threatened.

## Likelihood Explanation
The attacker must be a valid protocol peer — a single Byzantine subnet node suffices. No cryptographic material beyond normal peer authentication is required. The attack requires only sending crafted slot update messages over the existing QUIC transport. The `SLOT_TABLE_NO_LIMIT` configuration for https-outcalls is the enabling condition; ingress (which uses `SLOT_TABLE_LIMIT_INGRESS = 50_000`) is not affected by the same unbounded path. [8](#0-7) 

## Recommendation
1. **Apply a finite slot limit for https-outcalls.** Replace `SLOT_TABLE_NO_LIMIT` with a bound derived from the maximum number of concurrent HTTPS outcall requests, e.g., `MAX_NUMBER_OF_REQUESTS_AHEAD` (345) or a small multiple thereof, analogous to `SLOT_TABLE_LIMIT_INGRESS`.
2. **Bound the `MaybeWantsLater` stash size.** Track the number of stashed tasks globally or per-peer in `should_download` and return `false` (treat as `Unwanted`) when the limit is exceeded.
3. **Add a timeout to stashed tasks.** If a task remains in `MaybeWantsLater` for longer than a configurable number of bouncer refresh cycles without transitioning to `Wants`, treat it as `Unwanted` and terminate the task.

## Proof of Concept
A deterministic unit test using the existing `ReceiverManagerBuilder` harness:

```rust
#[tokio::test]
async fn unbounded_task_accumulation_https_outcalls() {
    // Simulate SLOT_TABLE_NO_LIMIT (usize::MAX)
    let (mut mgr, _channels) = ReceiverManagerBuilder::new()
        .with_slot_limit(usize::MAX)
        .build();
    let cancellation = CancellationToken::new();
    let n = 10_000usize;
    for slot in 0..n {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(slot as u64),
                commit_id: CommitId::from(1),
                // Each slot uses a distinct artifact ID (simulating distinct high callback IDs)
                update: Update::Id(slot as u64 + 1_000_000),
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    // All N tasks are alive, blocked in MaybeWantsLater stash
    assert_eq!(mgr.artifact_processor_tasks.len(), n);
    // Memory grows O(N) with no upper bound enforced
}
```

A fuzz test injecting artifacts with IDs in `(highest_accepted_request_id, u64::MAX]` and asserting `artifact_processor_tasks.len()` is bounded by a constant would also reproduce this.
