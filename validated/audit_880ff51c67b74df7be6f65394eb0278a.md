### Title
Missing External Setter for `avg_block_duration` Prevents Admin Correction of Block Reward Calculation - (`File: src/reward_supplier/reward_supplier.cairo`)

### Summary

The `RewardSupplier` contract stores `avg_block_duration`, a critical parameter that directly controls per-block reward amounts. The developers themselves flagged it with `// TODO: Setter.` and `// TODO: View?` comments. No public or privileged external function exists to set this value; it can only be updated via the internal `set_avg_block_duration()` function, which is called automatically once per epoch. If the automatic calculation produces an incorrect value (e.g., due to edge-case block timing), the protocol has no mechanism to correct it without a full contract upgrade.

### Finding Description

In `src/reward_supplier/reward_supplier.cairo`, the storage field `avg_block_duration` is declared with explicit TODO notes acknowledging the missing setter:

```cairo
/// Average block duration in units of 1 / BLOCK_DURATION_SCALE seconds.
// TODO: Setter.
// TODO: View?
avg_block_duration: u64,
``` [1](#0-0) 

The value is initialized to `DEFAULT_AVG_BLOCK_DURATION` (= `3 * BLOCK_DURATION_SCALE` = 300, representing 3.00 seconds) in the constructor:

```cairo
self.avg_block_duration.write(DEFAULT_AVG_BLOCK_DURATION);
``` [2](#0-1) 

It is then updated automatically once per epoch by the internal `set_avg_block_duration()` function, which is called from `update_current_epoch_block_rewards()`: [3](#0-2) 

The internal updater calculates the average block duration from the difference between the current and previous epoch's block snapshot, then **clamps** the result to `[min_block_duration, max_block_duration]` from `block_duration_config`: [4](#0-3) 

Crucially, `block_duration_config` **does** have an external privileged setter (`set_block_duration_config` in `IRewardSupplierConfig`), but `avg_block_duration` itself does not: [5](#0-4) 

The `avg_block_duration` value feeds directly into the per-block reward formula:

```cairo
let total_rewards = mul_wide_and_div(
    lhs: yearly_mint,
    rhs: avg_block_duration.into(),
    div: BLOCK_DURATION_SCALE.into() * SECONDS_IN_YEAR.into(),
)
``` [6](#0-5) 

### Impact Explanation

`avg_block_duration` is a linear multiplier in the block reward formula. If it is stuck at an incorrect value — whether the default (300) during the first two epochs, or a clamped/miscalculated value thereafter — every staker and delegator receiving block-based consensus rewards will receive a proportionally wrong amount for the entire duration until a contract upgrade is deployed. An `avg_block_duration` that is too low permanently under-pays stakers (theft of unclaimed yield); one that is too high over-mints and drains the reward supplier (protocol insolvency). The absence of a setter means neither the app governor nor any other role can correct the value in-flight.

**Allowed impact matched**: High — Theft of unclaimed yield / permanent freezing of unclaimed yield.

### Likelihood Explanation

The automatic `set_avg_block_duration()` calculation is correct under normal conditions, so the issue is not constantly triggered. However, the first epoch after deployment (or after an upgrade via the EIC) always uses the hardcoded default because the snapshot starts at zero and the early-return path is taken:

```cairo
if snapshot_block_number.is_zero() || snapshot_timestamp.is_zero() {
    return;
}
``` [7](#0-6) 

Additionally, if Starknet's block production rate deviates significantly from the default (e.g., during congestion or protocol changes), the clamping mechanism will pin `avg_block_duration` to the configured min or max, and without a setter there is no way to adjust the actual stored value to reflect reality. The EIC for the reward supplier sets `avg_block_duration` only at upgrade time and cannot be re-run: [8](#0-7) 

### Recommendation

Add a privileged external setter for `avg_block_duration` to `IRewardSupplierConfig`, gated behind `only_app_governor`, with the same range validation already applied in `set_avg_block_duration()`:

```cairo
fn set_avg_block_duration(ref self: ContractState, avg_block_duration: u64) {
    self.roles.only_app_governor();
    let block_duration_config = self.block_duration_config.read();
    assert!(
        avg_block_duration >= block_duration_config.min_block_duration
            && avg_block_duration <= block_duration_config.max_block_duration,
        "{}",
        Error::INVALID_AVG_BLOCK_DURATION,
    );
    self.avg_block_duration.write(avg_block_duration);
    // Emit event.
}
```

Also add a corresponding view function so off-chain tooling can read the current value without relying on storage inspection.

### Proof of Concept

1. The `RewardSupplier` contract is deployed. `avg_block_duration` is written to `DEFAULT_AVG_BLOCK_DURATION` (300 = 3.00 s) in the constructor. [9](#0-8) 
2. Epoch 1 begins. The staking contract calls `update_current_epoch_block_rewards()`. Inside, `set_avg_block_duration()` is called; because `snapshot_block_number == 0`, it returns early without updating `avg_block_duration`. The value remains 300. [7](#0-6) 
3. Block rewards for epoch 1 are computed using `avg_block_duration = 300`. If the actual average block time is, say, 6 seconds (600 in scaled units), rewards are **halved** for all stakers and delegators for the entire epoch. [10](#0-9) 
4. No external caller — including the app governor — can invoke any function to correct `avg_block_duration`. The `IRewardSupplierConfig` interface exposes only `set_block_duration_config`, not a setter for `avg_block_duration`. [11](#0-10) 
5. The under-payment persists until a contract upgrade is deployed and a new EIC is executed, which may span multiple epochs. [8](#0-7)

        fn set_avg_block_duration(ref self: ContractState) {
            let current_block_number = starknet::get_block_number();
            let current_timestamp = starknet::get_block_timestamp();
            let (snapshot_block_number, snapshot_timestamp) = self.block_snapshot.read();
            // Sanity asserts.
            assert!(
                current_block_number > snapshot_block_number,
                "{}",
                InternalError::INVALID_BLOCK_NUMBER,
            );
            assert!(
                current_timestamp > snapshot_timestamp.into(),
                "{}",
                InternalError::INVALID_BLOCK_TIMESTAMP,
            );
            self
                .block_snapshot
                .write((current_block_number, Timestamp { seconds: current_timestamp }));
            // If this is the first time we're setting the block snapshot, can't calculate avg block
            // time yet.
            if snapshot_block_number.is_zero() || snapshot_timestamp.is_zero() {
                return;
            }
            let time_delta = current_timestamp - snapshot_timestamp.into();
            // *Note*: `num_blocks` should match the epoch length in blocks. This calculation is
            // expected to run on the first block of each epoch, assuming `update_rewards` is called
            // every block.
            // We calculate `num_blocks` instead of using the configured value to keep the average
            // accurate even if some calls are missed.
            let num_blocks = current_block_number - snapshot_block_number;
            let mut calculated_block_duration = mul_wide_and_div(
                lhs: time_delta, rhs: BLOCK_DURATION_SCALE, div: num_blocks,
            )
                .expect_with_err(err: Error::BLOCK_DURATION_OVERFLOW);
            // Adjust calculated_block_duration with min and max block duration.
            let block_duration_config = self.block_duration_config.read();
            calculated_block_duration = calculated_block_duration
                .clamp(
                    block_duration_config.min_block_duration,
                    block_duration_config.max_block_duration,
                );
            self.avg_block_duration.write(calculated_block_duration);
        }
```
