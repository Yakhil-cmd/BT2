Audit Report

## Title
Missing Anonymous Principal Validation in Neuron Controller Assignment via `claim_or_refresh_neuron_by_memo_and_controller` - (File: rs/nns/governance/src/governance.rs)

## Summary
The NNS Governance canister's `claim_or_refresh_neuron_by_memo_and_controller` function accepts a caller-supplied `controller` principal without validating that it is not the anonymous principal. Because the IC protocol permits anonymous ingress messages and the governance canister contains no rejection of anonymous callers, an attacker can create a neuron whose controller is the anonymous principal, allowing any network participant calling as anonymous to exercise full neuron control including casting NNS governance votes.

## Finding Description
At `rs/nns/governance/src/governance.rs:5858`, the controller is resolved as:

```rust
let controller = memo_and_controller.controller.unwrap_or(*caller);
```

When `memo_and_controller.controller` is `Some(PrincipalId::new_anonymous())`, the anonymous principal is passed directly to `claim_neuron(subaccount, controller, ...)` at line 5867 and persisted as the neuron's authoritative controller. No validation of the controller value occurs at this site, at the `By::MemoAndController` dispatch point (lines 6123–6130), or anywhere else in `rs/nns/governance/src/governance.rs` — a grep for `anonymous` or `new_anonymous` in that file returns zero matches. The `MemoAndController.controller` field is a fully caller-controlled `Option<PrincipalId>`. The `By::MemoAndController` path is reachable by any ingress sender without any pre-existing neuron ownership. The IC protocol allows the anonymous principal to send ingress messages; no boundary-node or canister-level guard in the governance canister blocks it. Once the neuron exists with the anonymous principal as controller, the governance canister's own neuron authorization logic (which checks `caller == neuron.controller` or hotkey membership) accepts any anonymous caller as authorized for all neuron operations on that neuron.

Note: the PoC's citation of `validate_controller` from `rs/execution_environment/src/execution/common.rs` is incorrect — that function governs canister management operations, not neuron authorization. The actual authorization path is the governance canister's internal controller/hotkey check, which is equally permissive once the anonymous principal is stored as controller.

The identical missing check exists in `rs/sns/governance/src/governance.rs` at the SNS equivalent of this function.

## Impact Explanation
This matches the **High** bounty impact: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." An attacker who funds a neuron with sufficient ICP and a long dissolve delay can acquire meaningful NNS voting power exercisable by any anonymous caller, directly undermining NNS governance integrity. The constraint is the ICP cost of staking, which bounds severity below Critical but does not eliminate the concrete governance harm.

## Likelihood Explanation
The attack requires no privileged access, no key compromise, and no social engineering. The only prerequisite is possessing enough ICP to meet `neuron_minimum_stake_e8s`. The steps are deterministic and repeatable. Any motivated actor wishing to acquire pseudonymous or shared NNS voting power can execute this at will.

## Recommendation
Add an explicit rejection of the anonymous principal before assigning the controller in `claim_or_refresh_neuron_by_memo_and_controller` in both `rs/nns/governance/src/governance.rs` and `rs/sns/governance/src/governance.rs`:

```rust
let controller = memo_and_controller.controller.unwrap_or(*caller);
if controller == PrincipalId::new_anonymous() {
    return Err(GovernanceError::new_with_message(
        ErrorType::InvalidPrincipal,
        "The anonymous principal cannot be set as a neuron controller.",
    ));
}
```

As defense-in-depth, add the same guard inside `Neuron::set_controller` in `rs/nns/governance/src/neuron/types.rs`.

## Proof of Concept
1. Compute `subaccount = compute_neuron_staking_subaccount(PrincipalId::new_anonymous(), memo=42)`.
2. Transfer `neuron_minimum_stake_e8s` ICP to the NNS governance canister's ledger address for that subaccount (callable from any authenticated identity).
3. Call `manage_neuron` (from any identity, including authenticated) with:
   ```
   ManageNeuron {
     command: ClaimOrRefresh {
       by: MemoAndController { memo: 42, controller: Some(<anonymous_principal>) }
     }
   }
   ```
4. The call succeeds; a neuron is created with `controller = anonymous_principal`.
5. Subsequently, any caller using the anonymous principal sends `manage_neuron` with `RegisterVote`, `Follow`, or `Disburse` targeting this neuron's ID. The governance canister's internal controller check passes because `anonymous_principal == neuron.controller`, and the operation is executed.
