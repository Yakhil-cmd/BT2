### Title
Unrestricted `exit_delegation_pool_action` Allows Any Caller to Execute Pool Member Exit, Blocking Pool Switches - (File: src/pool/pool.cairo)

---

### Summary
`exit_delegation_pool_action` in the Pool contract accepts an arbitrary `pool_member` address and performs no caller authorization check. Any unprivileged address can invoke it for any pool member whose exit window has elapsed, forcibly clearing that member's undelegate intent and transferring their funds. This permanently blocks the pool member from calling `switch_delegation_pool`, which requires a live `unpool_time`, forcing them to restart the full exit/re-entry cycle.

---

### Finding Description
`exit_delegation_pool_action` (pool.cairo lines 295–333) takes `pool_member: ContractAddress` as a parameter and checks only that an intent exists and the time window has passed. There is no assertion that `get_caller_address() == pool_member` or any equivalent authorization guard. [1](#0-0) 

The function then unconditionally zeroes `unpool_amount` and `unpool_time` in the pool member's stored info and transfers the funds to `pool_member`: [2](#0-1) 

`switch_delegation_pool`, the only path for a pool member to move stake to a different staker without exiting, requires `pool_member_info.unpool_time.is_some()`: [3](#0-2) 

Once `exit_delegation_pool_action` is executed by a third party, `unpool_time` is `None`, and any subsequent `switch_delegation_pool` call by the pool member reverts with `MISSING_UNDELEGATE_INTENT`. The pool member must re-enter the pool and wait through the full exit window again before they can switch.

---

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users.**

The attacker gains nothing financially; funds are transferred to the correct `pool_member` address. However, the pool member suffers:
- Loss of the ability to switch delegation pools atomically (the primary use-case for having a live intent).
- Forced re-entry into the pool and a full additional exit-wait-window delay before they can switch.
- Potential loss of rewards during the forced re-entry period, since delegation stake only counts after the `K`-epoch activation delay.

For pool members that are smart contracts (e.g., a DAO treasury or a yield aggregator), the forced receipt of tokens at an unexpected time can also break internal accounting or trigger unintended state transitions.

---

### Likelihood Explanation
**Medium.** The attack requires no special privilege — any EOA or contract can call `exit_delegation_pool_action` with an arbitrary `pool_member` address. The only precondition is that the target's exit window has elapsed, which is publicly observable on-chain. A griefing actor monitoring mempool or epoch boundaries can reliably front-run a pool member's `switch_delegation_pool` transaction.

---

### Recommendation
Add a caller authorization check at the top of `exit_delegation_pool_action`, mirroring the pattern used in `add_to_delegation_pool` and `claim_rewards`:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let caller = get_caller_address();
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    assert!(
        caller == pool_member || caller == pool_member_info.reward_address,
        "{}",
        Error::CALLER_CANNOT_EXIT_POOL,
    );
    // ... rest of function
}
```

This is consistent with how `add_to_delegation_pool` (pool.cairo lines 227–232) and `claim_rewards` (pool.cairo lines 338–344) restrict callers to the pool member or their reward address. [4](#0-3) 

---

### Proof of Concept

1. **Pool member** (victim, address `V`) calls `exit_delegation_pool_intent(amount)` — sets `unpool_time = T` and `unpool_amount = X`.
2. Time advances past `T` (exit window elapses).
3. **Victim** prepares a `switch_delegation_pool(to_staker, to_pool, X)` transaction to move stake to a preferred staker.
4. **Attacker** (address `A`) observes the pending transaction and calls `exit_delegation_pool_action(V)` first (front-run or same block).
5. `exit_delegation_pool_action` executes: `unpool_time = None`, `unpool_amount = 0`, tokens transferred to `V`.
6. Victim's `switch_delegation_pool` call now reverts: `pool_member_info.unpool_time.is_some()` is `false` → `MISSING_UNDELEGATE_INTENT`.
7. Victim must call `enter_delegation_pool` again, wait `K` epochs for stake activation, then call `exit_delegation_pool_intent` again and wait another full exit window — suffering both reward loss and operational disruption. [5](#0-4) [6](#0-5)

        fn exit_delegation_pool_action(
            ref self: ContractState, pool_member: ContractAddress,
        ) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let unpool_time = pool_member_info
                .unpool_time
                .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
            assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberExitAction {
                        pool_member, unpool_amount: pool_member_info.unpool_amount,
                    },
                );

            // Perform removal action in the staking contract, receiving funds if needed.
            // Note that if the intent was done after the staker was removed (unstake_action),
            // the funds will already be in the pool contract, and the following call will do
            // nothing.
            let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
            staking_pool_dispatcher
                .remove_from_delegation_pool_action(identifier: pool_member.into());

            let unpool_amount = pool_member_info.unpool_amount;
            pool_member_info.unpool_amount = Zero::zero();
            pool_member_info.unpool_time = Option::None;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer delegated amount to the pool member.
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());

            unpool_amount
        }
