### Title
Missing Lower Bound on `threshold` in FROST Signing Validation Allows Malicious Coordinator to Corrupt Signing Output - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that `threshold` does not exceed the participant count, but omit the lower-bound check (`threshold >= 2`) that every other analogous validation function in the codebase enforces. A malicious coordinator can supply `threshold = 1` to FROST EdDSA or RedJubjub signing, bypassing the guard entirely. The signing protocol then proceeds with a structurally incorrect threshold, producing an invalid/unusable signature while consuming honest participants' one-time presignature nonces.

---

### Finding Description

Every other threshold-validation helper in the codebase enforces **both** an upper bound and a lower bound of 2:

| Function | File | Upper bound | Lower bound ≥ 2 |
|---|---|---|---|
| `assert_key_invariants` | `src/dkg.rs` | ✓ | ✓ |
| `validate_triple_inputs` | `src/ecdsa/ot_based_ecdsa/triples/generation.rs` | ✓ | ✓ |
| `assert_sign_inputs` | `src/frost/mod.rs` | ✓ | **✗ missing** |
| `presign` | `src/frost/mod.rs` | ✓ | **✗ missing** |

In `assert_key_invariants`, both checks are present: [1](#0-0) 

In `validate_triple_inputs`, both checks are present: [2](#0-1) 

In `assert_sign_inputs`, only the upper bound is checked; the lower bound is absent: [3](#0-2) 

In `presign`, the same pattern: only the upper bound is checked: [4](#0-3) 

`ReconstructionLowerBound` is a plain `usize` newtype with no internal invariant enforcing a minimum of 2: [5](#0-4) 

So `threshold = 1` (or `threshold = 0`) is accepted by both `assert_sign_inputs` and `presign` without error.

---

### Impact Explanation

**High: Corruption of signing outputs so honest parties accept unusable cryptographic outputs.**

In FROST, the signing threshold determines how many signature shares are Lagrange-interpolated to reconstruct the final signature. The key shares were generated with a threshold ≥ 2 during DKG (which enforces the lower bound). If `threshold = 1` is passed at signing time:

- Only one signature share is collected.
- The Lagrange coefficient for a single-point interpolation is 1, so the aggregated value is simply that one participant's share — not the correct reconstruction of the degree-(t−1) polynomial used during keygen.
- The resulting aggregated signature is cryptographically invalid and will fail verification.
- Honest participants have irrevocably consumed their one-time presignature nonces (FROST nonces must never be reused), so those presignatures are permanently invalidated.

This maps directly to the allowed High impact: *"Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs."*

---

### Likelihood Explanation

The coordinator role is an explicit, documented trust boundary in this library. The `sign` functions in `src/frost/eddsa/sign.rs` and `src/frost/redjubjub/sign.rs` call `assert_sign_inputs` with a caller-supplied `threshold`. A malicious or compromised coordinator can trivially pass `threshold = 1` as a `usize` (or via `ReconstructionLowerBound::from(1usize)`). No cryptographic capability is required — it is a pure parameter-manipulation attack reachable through the public API.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already enforce, in both `assert_sign_inputs` and `presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place this check immediately after the upper-bound check in both functions, mirroring the pattern in `src/dkg.rs` lines 580–582 and `src/ecdsa/ot_based_ecdsa/triples/generation.rs` lines 699–703.

---

### Proof of Concept

1. Run DKG for 3 participants with `threshold = 2` (accepted, lower bound enforced).
2. Run FROST presigning for those participants.
3. Call `assert_sign_inputs` (or the public `sign` entry point) with `threshold = 1` and the same 3 participants.
4. Observe: no `InitializationError` is returned — the call succeeds.
5. The signing protocol proceeds, collects only 1 signature share, performs Lagrange interpolation with a single point (coefficient = 1), and emits an aggregated signature that is not a valid EdDSA/RedJubjub signature over the intended message.
6. The presignature nonces of all honest participants are consumed and cannot be reused.

Contrast: calling `keygen` with `threshold = 1` returns `InitializationError::ThresholdTooSmall { threshold: 1, min: 2 }` immediately, demonstrating the inconsistency. [6](#0-5) [7](#0-6)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
