### Title
Missing `app_pk` Identity-Element Validation in `ckd()` Bypasses ElGamal Masking, Directly Exposing the Confidential Derived Key — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry point and its inner `compute_signature_share()` helper accept the caller-supplied `app_pk` (`ElementG1 = blstrs::G1Projective`) without checking whether it is the group identity element. When `app_pk = G1Projective::identity()`, the ElGamal blinding term `app_pk * y` collapses to the identity, so every participant's ciphertext share `C_i` equals the unblinded BLS share `x_i · H(pk ‖ app_id)`. The coordinator then aggregates these into `C = msk · H(pk ‖ app_id)` — the confidential key itself — and publishes it on-chain in the clear. No private scalar `a` is needed to recover the secret; it is directly readable from the protocol output.

---

### Finding Description

**Root cause — `compute_signature_share` (lines 148–182):**

```rust
// src/confidential_key_derivation/protocol.rs
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,          // ← no identity check
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    let y = Scalar::random(rng);
    let big_y = ElementG1::generator() * y;

    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
    let big_s = hash_point * private_share.to_scalar();   // x_i · H(pk‖app_id)

    let big_c = big_s + app_pk * y;   // ← if app_pk == identity → big_c = big_s
    ...
}
```

When `app_pk` is the identity element:

- `app_pk * y = G1Projective::identity()` (scalar multiplication of identity is identity)
- `big_c = big_s = x_i · H(pk ‖ app_id)` — the raw BLS share, unblinded
- After Lagrange-weighted aggregation by the coordinator: `C = Σ λ_i · big_c_i = msk · H(pk ‖ app_id)`

The CKD output `(Y, C)` is published on-chain. With `app_pk = 0`, `C` **is** the confidential key `s = msk · H(pk ‖ app_id)`. Any observer reads it directly; no knowledge of `a` is required.

**Entry path:**

The `ckd()` initializer performs four checks (participant count, duplicates, self-membership, coordinator-membership) but **no check on `app_pk`**:

```rust
// src/confidential_key_derivation/protocol.rs  lines 66–117

    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,

    // checks: len >= 2, no duplicates, me ∈ participants, coordinator ∈ participants
    // NO check: app_pk != G1Projective::identity()
    ...
}
```

`app_pk` flows unchanged into `run_ckd_protocol` → `do_ckd_participant` / `do_ckd_coordinator` → `compute_signature_share`.

---

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

The entire security guarantee of CKD is that `msk · H(pk ‖ app_id)` is hidden inside the ElGamal ciphertext `C = msk · H(pk ‖ app_id) + a · Y`. When `app_pk = identity`, the blinding term vanishes and `C` equals the secret directly. The on-chain CKD response then leaks the confidential key to every observer, permanently breaking the confidentiality property for that `(pk, app_id)` pair.

---

### Likelihood Explanation

The `app_pk` value originates from the requesting application and is relayed through the MPC contract to each node, which calls `ckd()`. The library imposes no guard; the MPC contract layer is explicitly described as out-of-scope for `app_pk` validation (the docs only discuss `app_id` authentication). A buggy application that zero-initialises its key pair, or a malicious actor who submits a crafted CKD request with `app_pk = G1Projective::identity()`, triggers the exposure with a single on-chain transaction. No cryptographic break, no privileged key, and no internal access are required.

---

### Recommendation

Add an explicit identity-element check in `ckd()` before the protocol starts, mirroring the pattern already used for other invalid inputs:

```rust
// After the coordinator-membership check, before make_protocol:
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Optionally, repeat the check inside `compute_signature_share` as a defence-in-depth assertion.

---

### Proof of Concept

1. Run `ckd()` on all `N` participants with `app_pk = blstrs::G1Projective::identity()` and any valid `app_id`.
2. Observe the coordinator output `CKDOutput { big_y, big_c }`.
3. Compute `expected = hash_app_id_with_pk(&public_key, &app_id) * msk`.
4. Assert `big_c == expected` — the confidential key is directly in `big_c`, with no need for the private scalar `a`.

This is reproducible by adapting the existing `test_ckd` unit test in `src/confidential_key_derivation/protocol.rs` (lines 210–283): replace `app_pk = ElementG1::generator() * app_sk` with `app_pk = ElementG1::identity()` and verify that `ckd_output.big_c == hash_app_id_with_pk(&pk, &app_id) * msk`. pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
