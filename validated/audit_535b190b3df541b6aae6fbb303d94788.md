### Title
Updating `set_eth_connector_contract_account` Redirects All Bridge Withdrawals Without Migrating Existing Bridged Funds - (File: engine/src/contract_methods/connector.rs)

### Summary

The Aurora Engine exposes `set_eth_connector_contract_account`, a post-deployment setter that allows the owner to replace the ETH connector contract account at any time. Because all `withdraw` and `ft_transfer` operations are forwarded to whatever account is currently stored in this slot, replacing it without migrating the existing bridged ETH balances held by the old connector causes those funds to become permanently inaccessible to users.

### Finding Description

The function `set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` unconditionally overwrites the stored connector account ID and serialization type with no migration step, no balance check, and no freeze of in-flight operations:

```rust
// engine/src/contract_methods/connector.rs, lines 417-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

The stored connector account ID is the sole routing target for every subsequent `withdraw` call. The `withdraw` function in the same file reads the connector account from storage and forwards the call there:

```rust
// engine/src/contract_methods/connector.rs, lines 43-59
pub fn withdraw<I: IO + Copy + PromiseHandler, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    require_running(&state::get_state(&io)?)?;
    env.assert_one_yocto()?;
    let args: WithdrawCallArgs = io.read_input_borsh()?;
    let args = borsh::to_vec(&EngineWithdrawCallArgs { ... }).unwrap();
    return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)
}
```

The `return_promise` call targets the connector account read from storage at call time. Similarly, `ft_on_transfer` checks `predecessor_account_id == get_connector_account_id(&io)?` to decide whether incoming tokens represent bridged ETH:

```rust
// engine/src/contract_methods/connector.rs, lines 81-90
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
};
```

After the connector account is replaced:
1. All existing bridged ETH balances tracked by the **old** connector contract are orphaned — the engine no longer routes `withdraw` calls to the old connector, so users cannot redeem their ETH.
2. Deposits from the old connector via `ft_on_transfer` are no longer recognized as base-token deposits, breaking the bridge in both directions.
3. There is no on-chain mechanism to drain or migrate the old connector's NEP-141 balances to the new one.

The NEAR-side entrypoint is exposed as a public `extern "C"` function:

```rust
// engine/src/lib.rs, lines 700-707
pub extern "C" fn set_eth_connector_contract_account() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::connector::set_eth_connector_contract_account(io, &env)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
```

### Impact Explanation

**Critical — Permanent freezing of funds.**

All ETH that has been bridged into Aurora is custodied by the ETH connector contract. The engine's internal EVM balances are backed 1:1 by NEP-141 tokens held by that connector. If the connector account is replaced, the engine's `withdraw` path routes to the new (empty) connector, while the old connector still holds the real backing tokens. Users who attempt to exit Aurora via `withdraw` will call `engine_withdraw` on the new connector, which has no balance, causing the withdrawal to fail or produce no real token transfer. The old connector's tokens are permanently stranded with no on-chain path to recover them.

### Likelihood Explanation

The setter is owner-only, so exploitation requires the owner to call it. However, the vulnerability class is identical to the reported StWSX issue: a legitimate administrative action (migrating to a new connector) causes permanent fund loss for all users who have not already exited. Given that Aurora is a live bridge with significant TVL, any connector migration performed without a prior full-drain of the old connector would trigger this loss. The likelihood is **medium** — it is a realistic operational scenario (connector upgrade/migration) that the protocol explicitly supports via this setter, but the setter provides no safety net.

### Recommendation

1. Remove `set_eth_connector_contract_account` as a live setter, or gate it behind a time-lock with a mandatory zero-balance check on the old connector before the switch takes effect.
2. If migration is required, implement a two-phase process: (a) pause deposits/withdrawals, (b) verify the old connector's total supply is zero (all users have exited), then (c) switch the account.
3. Alternatively, follow the same resolution as the StWSX audit: remove the setter entirely and perform migration via a new contract deployment with a redemption-and-redeposit flow.

### Proof of Concept

1. Users bridge ETH into Aurora. The ETH connector contract (`old_connector`) holds N ETH-equivalent NEP-141 tokens. Users have N ETH balance in the Aurora EVM.
2. Owner calls `set_eth_connector_contract_account` with `account = new_connector` (a freshly deployed, empty connector).
3. `set_connector_account_id` overwrites the stored account ID to `new_connector`.
4. A user calls `withdraw` on the Aurora engine to exit their ETH. The engine calls `engine_withdraw` on `new_connector`. `new_connector` has zero balance and cannot honor the withdrawal. The call fails or burns nothing from `old_connector`.
5. The user's EVM balance is debited (or the call reverts), but no real ETH is released. The N tokens in `old_connector` are permanently inaccessible because no engine method routes to `old_connector` anymore.
