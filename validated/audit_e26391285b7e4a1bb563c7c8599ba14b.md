### Title
Malicious Old Participant Can Corrupt Reshare/Refresh by Supplying Inconsistent `old_participants` List, Causing Permanent Denial of Key Rotation — (`src/dkg.rs`)

### Summary

The `do_reshare` function in `src/dkg.rs` computes each participant's Lagrange coefficient over the **locally-computed** intersection of `old_participants` and the new `participants` list. There is no protocol-level broadcast or agreement mechanism to ensure all participants use the same `old_participants` list. A malicious participant who holds a valid old signing share can silently supply a different `old_participants` list (e.g., one that omits some old participants), causing them to compute a wrong Lagrange coefficient and contribute an incorrect secret polynomial. The resulting aggregate public key will not match the old public key, causing the protocol to abort. Honest parties receive only a generic "new public key does not match old public key" error and cannot identify the culprit, making the denial of reshare/refresh effectively permanent.

### Finding Description

In `src/dkg.rs`, `do_reshare` computes the Lagrange-linearized secret contribution as:

```rust
let intersection = old_participants.intersection(&participants);
let secret = old_signing_key
    .map(|x_i| {
        intersection
            .lagrange::<C>(me)
            .map(|lambda| lambda * x_i.to_scalar())
    })
    .transpose()?
    .unwrap_or_else(<C::Group as Group>::Field::zero);
``` [1](#0-0) 

The `intersection` is derived entirely from the caller-supplied `old_participants` parameter. No round of the protocol broadcasts or cross-validates this list. Each participant independently calls `reshare()` with their own view of `old_participants`:

```rust
pub fn reshare<C: Ciphersuite>(
    old_participants: &[Participant],
    ...
    new_participants: &[Participant],
    ...
``` [2](#0-1) 

The only pre-flight check in `assert_reshare_keys_invariants` is:

```rust
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(...));
}
``` [3](#0-2) 

There is **no check** that prevents a participant who is genuinely in the old set from providing a *different* `old_participants` list (e.g., one that omits peer `C`). The `assert_keyshare_inputs` guard only checks whether the caller's own `old_participants` contains `me`: [4](#0-3) 

Because the `old_participants` list is never broadcast, honest participants B and C verify A's proof-of-knowledge using their own (correct) `old_participants = {A, B, C}`. A sends a valid `Some(proof)` (A is old in their view), so the PoK check passes: [5](#0-4) 

The inconsistency is only detected after all shares have been distributed, when the aggregate public key is compared to the old one:

```rust
if old_vk != verifying_key {
    return Err(ProtocolError::AssertionFailed(
        "new public key does not match old public key".to_string(),
    ));
}
``` [6](#0-5) 

At this point the protocol aborts with a generic error. No participant identity is attached to the failure.

**Concrete attack scenario** (threshold-2-of-3 refresh, participants {A, B, C}):

- Correct intersection for Lagrange: `{A, B, C}` → coefficients `λ_A`, `λ_B`, `λ_C` satisfying `x_A·λ_A + x_B·λ_B + x_C·λ_C = x`.
- Malicious A uses `old_participants = {A, B}` → intersection `{A, B}` → A contributes `x_A · λ_A({A,B})` instead of `x_A · λ_A({A,B,C})`.
- Sum of contributions: `x_A·λ_A({A,B}) + x_B·λ_B({A,B,C}) + x_C·λ_C({A,B,C}) ≠ x`.
- Resulting public key ≠ old public key → abort.
- B and C see only the generic assertion error; A is not identified.

### Impact Explanation

This maps to **High: Permanent denial of reshare/refresh for honest parties under valid protocol inputs**. A single malicious participant holding a legitimate old share can abort every reshare/refresh attempt without being identified by the protocol. Because the library provides no culprit-identification mechanism on this error path, honest parties cannot exclude the attacker and retry successfully without out-of-band forensics. Key rotation — a critical proactive-security operation — is permanently blocked.

### Likelihood Explanation

Any participant who holds a valid old signing share and wishes to block key rotation can trivially execute this attack by passing a truncated `old_participants` slice to `reshare()`. No cryptographic capability beyond holding a legitimate share is required. The attack is silent and unattributable from the protocol transcript alone.

### Recommendation

1. **Bind `old_participants` into the session transcript.** Include the sorted `old_participants` list in the session-ID hash (Round 1 broadcast) so that any participant using a different list will produce a mismatched session ID, causing an early, attributable abort.

2. **Broadcast and cross-validate `old_participants`.** Add a dedicated broadcast round where each participant sends their `old_participants` list; abort with `MaliciousParticipant` if any two lists differ.

3. **Add the missing asymmetric guard** in `assert_reshare_keys_invariants`: if `!old_participants.contains(me) && old_signing_key.is_some()`, return an error immediately, closing the complementary misconfiguration path.

### Proof of Concept

```
Setup: participants {A=1, B=2, C=3}, threshold=2, secret x.
Correct Lagrange over {A,B,C}: λ_A = 3/2, λ_B = -3, λ_C = 3/2  (example values)
Correct contributions: x_A·(3/2) + x_B·(-3) + x_C·(3/2) = x  ✓

Attack: A calls reshare() with old_participants=[A,B] instead of [A,B,C].
Malicious Lagrange over {A,B}: λ_A({A,B}) = 2, λ_B({A,B}) = -1
A contributes: x_A·2  (instead of x_A·(3/2))

Aggregate secret in new shares:
  x_A·2 + x_B·(-3) + x_C·(3/2)  ≠  x

New public key = (x_A·2 + x_B·(-3) + x_C·(3/2))·G  ≠  x·G = old public key

Protocol aborts at src/dkg.rs:491 with generic error.
B and C cannot determine A was the cause.
```
