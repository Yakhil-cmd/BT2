### Title
`set_eth_connector_contract_account()` Updates Connector Reference Without Migrating Supply State, Permanently Freezing User ETH Withdrawals - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` updates only the stored connector account ID pointer in engine storage. It does not migrate the existing ETH fungible-token supply from the old connector contract to the new one. After the call, every withdrawal, `ft_transfer`, and `ft_transfer_call` operation is routed to the new connector, which holds zero supply, causing all such operations to fail and permanently freezing the ETH of every existing Aurora user.

---

### Finding Description

Aurora's ETH connector is an external NEAR fungible-token contract. The engine stores a single pointer to it:

```
EthConnectorStorageId::EthConnectorAccount  →  <connector account id>
```

All outbound connector calls are dispatched through `return_promise`, which reads this pointer at call time: [1](#0-0) 

This single pointer is used for `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, and `storage_withdraw`.

The owner (or a private caller) can update the pointer via `set_eth_connector_contract_account`: [2](#0-1) 

The function writes only two values — the new account ID and the serialization type: [3](#0-2) 

It performs **no migration** of the old connector's fungible-token supply, user storage registrations, or balance records. The old connector contract retains the entire ETH supply; the new connector starts with zero supply.

Inbound minting is equally broken. `ft_on_transfer` gates base-token minting on the predecessor matching the stored connector account ID: [4](#0-3) 

After the pointer is changed, the old connector can no longer trigger `receive_base_tokens`, so new deposits through the old connector are also rejected.

The structural parallel to the reported `setSalesAddress()` bug is exact:

| Original (`UnikuraPhygitalCollection`) | Aurora Engine |
|---|---|
| `_salesAddress` updated in `UnikuraPhygitalCollectionStorage` | Connector account ID updated in `EthConnectorStorageId::EthConnectorAccount` |
| `ERC721AStorage` still records old address as token owner | Old connector contract still holds the entire ETH fungible-token supply |
| `_mintTransferFrom` compares against new `_salesAddress` → fails | `return_promise` routes withdrawals to new connector with zero supply → fails |

---

### Impact Explanation

After `set_eth_connector_contract_account` is called with a new account:

1. **Withdrawals fail**: `withdraw` is routed to the new connector, which has no supply to burn. Every user's attempt to bridge ETH back to Ethereum is rejected.
2. **FT transfers fail**: `ft_transfer` and `ft_transfer_call` are routed to the new connector, which has no registered storage or balances.
3. **New deposits fail**: `ft_on_transfer` rejects calls from the old connector, so in-flight deposits are also broken.

All EVM-side ETH balances remain in engine storage but become unwithdrawable. This constitutes **permanent freezing of funds** (Critical) for every ETH holder on Aurora at the time of the call.

---

### Likelihood Explanation

The function is callable by the engine owner or via a private call. It is a legitimate administrative operation — the expected use case is upgrading or replacing the connector contract. There is no on-chain guard, migration step, or supply-transfer logic enforced before or after the pointer swap. Any owner-initiated connector upgrade that does not manually coordinate a full supply migration on the old connector before calling this function will trigger the freeze. Given that the function exists precisely to be called during connector upgrades, the likelihood of it being invoked without a correct migration procedure is moderate.

---

### Recommendation

`set_eth_connector_contract_account` should either:

1. **Enforce a zero-supply precondition**: Require that the old connector's total supply is zero (i.e., all ETH has been withdrawn) before allowing the pointer to be changed, or
2. **Atomically migrate supply**: Coordinate a cross-contract call sequence that transfers the full fungible-token supply and all user storage registrations from the old connector to the new one before updating the pointer, or
3. **Remove the function**: Keep the connector account as an immutable constant set at initialization, analogous to the fix applied in the referenced `UnikuraPhygitalCollection` audit (`b898120427d9748162dccbb37a79524d40f20f40`).

---

### Proof of Concept

1. Alice deposits 10 ETH into Aurora. The old connector (`old.connector.near`) mints 10 ETH-FT and calls `ft_on_transfer` on the engine. The engine credits Alice's EVM address with 10 ETH. `EthConnectorStorageId::EthConnectorAccount = "old.connector.near"`.

2. The engine owner deploys a new connector contract (`new.connector.near`) with zero supply and calls:
   ```
   set_eth_connector_contract_account({ account: "new.connector.near", withdraw_serialize_type: Borsh })
   ```
   `EthConnectorStorageId::EthConnectorAccount` is now `"new.connector.near"`. No supply is transferred.

3. Alice calls `withdraw` on the engine to bridge her 10 ETH back to Ethereum. `return_promise` dispatches the call to `new.connector.near`. The new connector has zero supply and cannot burn 10 ETH-FT → the call fails.

4. Alice's 10 EVM ETH is permanently locked. She cannot withdraw, transfer via `ft_transfer`, or recover her funds through any engine interface. [2](#0-1) [1](#0-0) [4](#0-3)

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
