### Title
Unbounded loop in `calculate_rewards` over `pool_member_epoch_balance` trace can permanently freeze pool member's unclaimed yield - (File: `src/pool/pool.cairo`)

---

### Summary
The `calculate_rewards` function in the delegation pool contract iterates over an unbounded `pool_member_epoch_balance` trace. A pool member who makes balance changes across many epochs without claiming rewards accumulates one new trace entry per epoch. If the trace grows large enough, the `claim_rewards` call will exceed Starknet's gas limit, permanently freezing the pool member's unclaimed yield. The developers themselves acknowledged this risk with an inline comment.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function contains an explicit unbounded loop:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` (the index saved from the last claim) up to the current trace length. Each iteration performs a storage read (`pool_member_trace.at(entry_to_claim_from)`) and calls `find_sigma`, which itself reads from `cumulative_rewards_trace_vec`.

**How the trace grows:**

The trace is backed by a `Vec<PoolMemberBalanceCheckpoint>`. The `insert` function appends a new checkpoint whenever the incoming key (epoch) differs from the last stored key; otherwise it updates in place. [2](#0-1) 

`set_member_balance` always inserts with key `current_epoch + K`: [3](#0-2) 

This means **one new trace entry is appended per epoch** in which the pool member changes their balance. The following public entry points all call `set_member_balance` (directly or via `increase_member_balance`):

- `enter_delegation_pool` — line 201 [4](#0-3) 
- `add_to_delegation_pool` — line 242 [5](#0-4) 
- `exit_delegation_pool_intent` — line 278 [6](#0-5) 

**Mitigation already present (partial):**

`entry_to_claim_from` is stored per pool member and advanced after each successful `claim_rewards` call, so the loop only iterates over entries accumulated *since the last claim*. [7](#0-6) 

However, if a pool member makes balance changes in many consecutive epochs without ever claiming, the unclaimed window grows without bound.

---

### Impact Explanation

If the `pool_member_epoch_balance` trace grows large enough that the loop in `calculate_rewards` exceeds Starknet's per-transaction gas limit, `claim_rewards` will revert on every call. Because `entry_to_claim_from` is only advanced inside a successful `claim_rewards` execution, there is no way to partially drain the trace. The pool member's entire accumulated unclaimed yield is permanently frozen in the contract with no recovery path.

This matches the allowed impact: **Permanent freezing of unclaimed yield (High)** / **Unbounded gas consumption (Medium)**.

---

### Likelihood Explanation

- Epoch duration in the deployed configuration is ~9 000 seconds (~2.5 hours); `K = 2`. [8](#0-7) 
- A pool member calling `add_to_delegation_pool` once per epoch accumulates ~1 460 entries per year, ~14 600 entries per decade.
- Each loop iteration involves multiple storage reads and arithmetic; at sufficient scale this will exceed Starknet's gas ceiling.
- Only the pool member themselves can grow their own trace (access control on `add_to_delegation_pool` restricts callers to `pool_member` or `reward_address`), so this is self-inflicted rather than a third-party attack. [9](#0-8) 
- The developers explicitly acknowledged the risk: *"This loop is unbounded but unlikely to exceed gas limits."* [10](#0-9) 

**Likelihood: Low** — requires years of continuous activity with no reward claims. The scenario is realistic for long-lived, active delegators who never claim.

---

### Recommendation

1. **Checkpoint-based pagination**: Allow `claim_rewards` to accept an optional `max_entries` parameter so a pool member can drain the trace in multiple transactions, advancing `entry_to_claim_from` incrementally.
2. **Enforce a maximum trace depth**: Cap the number of unclaimed balance-change entries (e.g., require a claim before a new balance change is accepted once the unclaimed window exceeds a threshold).
3. **Periodic forced claims**: Require or incentivize pool members to claim rewards at least once every N epochs to keep the unclaimed window bounded.

---

### Proof of Concept

1. Pool member calls `enter_delegation_pool` in epoch `E`. Trace length = 1.
2. Pool member calls `add_to_delegation_pool` with a minimal amount once per epoch for `N` consecutive epochs without ever calling `claim_rewards`. Each call in a new epoch appends one entry; trace length grows to `N + 1`. [11](#0-10) 
3. Pool member calls `claim_rewards`. `calculate_rewards` is invoked with `entry_to_claim_from = 0` and `pool_member_trace_length = N + 1`. [12](#0-11) 
4. The loop executes `N + 1` iterations, each reading from storage and calling `find_sigma`. [13](#0-12) 
5. For sufficiently large `N`, the transaction exceeds Starknet's gas limit and reverts. Because `entry_to_claim_from` was never written back, every subsequent `claim_rewards` call also reverts. The pool member's unclaimed yield is permanently frozen.
