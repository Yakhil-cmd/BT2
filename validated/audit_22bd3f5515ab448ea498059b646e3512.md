### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions When Any Prior Account Transaction Exists in Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the blockifier `__validate__` entry-point call for any invoke transaction with `nonce=1` whenever `account_tx_in_pool_or_recent_block` returns `true`. The check is intended to cover only the "deploy_account + invoke" UX scenario, but the mempool query returns `true` for **any** prior transaction from that account — including a valid invoke at `nonce=0` from an already-deployed account. An attacker can therefore admit an invoke transaction carrying an invalid signature into the mempool by first submitting one valid invoke (nonce=0) and then a second invoke (nonce=1) with an arbitrary or forged signature.

### Finding Description

In `skip_stateful_validations`:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, which causes `StatefulValidator::perform_validations` to skip the `__validate__` entry-point call entirely: [2](#0-1) [3](#0-2) 

The mempool's `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or any committed nonce in the recent-block history — not specifically a `DeployAccount` transaction: [4](#0-3) [5](#0-4) 

The code comment acknowledges the ambiguity but incorrectly concludes it is safe:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of that disjunction — a prior invoke at nonce=0 from an already-deployed account — does not justify skipping `__validate__` for the nonce=1 invoke.

### Impact Explanation

An attacker controlling a deployed account with on-chain nonce=0 (e.g., a genesis/pre-deployed account, or any account whose deploy_account was committed but whose nonce was reset to 0 in a reorg scenario) can:

1. Submit `invoke_A` (nonce=0, valid signature) → passes all checks including `__validate__`, enters mempool.
2. Submit `invoke_B` (nonce=1, **invalid or forged signature**) → `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block` is `true` due to `invoke_A`; `__validate__` is never called; `invoke_B` is admitted to the mempool.

The gateway's only cryptographic admission gate (`__validate__`) is bypassed. `invoke_B` will fail at execution time in the batcher, but it has already been admitted, consuming a mempool slot and potentially enabling targeted DoS of the mempool with signature-invalid transactions at the cost of one valid transaction per batch.

This matches the allowed impact: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The condition `account_nonce == 0 && tx_nonce == 1` is reachable for any pre-deployed account (genesis accounts, accounts deployed via direct state writes in test/staging environments, or any account whose deploy_account was committed and nonce is still 0 due to a reorg). The attacker needs no special privilege beyond the ability to submit two transactions.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction exists for the sender address in the mempool. Alternatively, add a transaction-type filter so that `skip_stateful_validations` only returns `true` when the mempool contains a `DeployAccount` (not an `Invoke`) for the account:

```rust
// Pseudocode
let has_deploy_account = mempool_client
    .deploy_account_tx_in_pool(tx.sender_address())
    .await?;
return Ok(has_deploy_account);
```

If a dedicated `deploy_account_tx_in_pool` query is not feasible, the function should at minimum document and enforce that the only valid reason to skip `__validate__` is the presence of an unprocessed `DeployAccount` transaction, and the mempool API should expose that distinction.

### Proof of Concept

1. Deploy an account contract at address `A` via genesis or direct state write so that `state.get_nonce(A) == 0` but the contract class exists.
2. Submit `invoke_A`: sender=`A`, nonce=0, valid ECDSA signature over the correct tx hash. This passes `StatelessTransactionValidator::validate`, `validate_nonce` (0 ∈ [0, max_gap]), `validate_resource_bounds`, `validate_by_mempool`, and `run_validate_entry_point` (with `skip_validate=false`, `__validate__` runs and succeeds). `invoke_A` enters the mempool.
3. Submit `invoke_B`: sender=`A`, nonce=1, **signature = [0x0, 0x0]** (invalid). `validate_nonce` passes (1 ∈ [0, max_gap]). `validate_by_mempool` passes (nonce not too old). `skip_stateful_validations` calls `account_tx_in_pool_or_recent_block(A)` → `true` (due to `invoke_A`) → returns `true`. `run_validate_entry_point` sets `validate=false`; `__validate__` is never called. `invoke_B` is admitted to the mempool with an invalid signature. ```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
