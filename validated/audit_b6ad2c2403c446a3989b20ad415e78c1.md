### Title
Unchecked Precompile Call Return Value in `withdrawToNear` Causes Permanent ERC-20/NEP-141 Accounting Discrepancy — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

In `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear` function burns the caller's ERC-20 tokens **before** calling the `ExitToNear` precompile via a low-level assembly `call`. The return value of that call is captured in `res` but **never checked**. If the precompile call fails for any reason (e.g., invalid NEAR recipient account ID), the ERC-20 burn is not reverted, the corresponding NEP-141 tokens remain permanently locked inside Aurora, and no refund path exists. This creates a permanent discrepancy between the ERC-20 mirror `totalSupply` and the actual NEP-141 tokens held by the engine — an exact structural analog to H-12.

---

### Finding Description

`EvmErc20.sol` `withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // (1) burn is final

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure
    }
}
``` [1](#0-0) 

`EvmErc20V2.sol` has the identical pattern: [2](#0-1) 

**Step-by-step failure path:**

1. A user calls `withdrawToNear(recipient, amount)` where `recipient` is any byte sequence that does not parse as a valid NEAR `AccountId` (e.g., empty bytes, bytes containing spaces or uppercase letters, or a string exceeding the 64-character NEAR account ID limit).
2. `_burn(_msgSender(), amount)` executes and succeeds — ERC-20 `totalSupply` is permanently reduced.
3. The assembly `call` invokes the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`.
4. Inside the precompile, `ExitToNearParams::try_from(input)` or `get_nep141_from_erc20` fails, returning `ExitError`. The EVM `call` opcode returns `0`.
5. Because `res` is never checked and no `require(res != 0)` guard exists, the Solidity function returns normally — **no revert**.
6. No NEAR promise is scheduled; the NEP-141 `ft_transfer` never happens.
7. The `error_refund` callback mechanism (`exit_to_near_precompile_callback`) is **never triggered** because it only fires when a promise was actually scheduled and then failed — not when the precompile itself returned an error before scheduling anything. [3](#0-2) [4](#0-3) 

The precompile's own validation that can cause a silent-failure-from-the-caller's-perspective: [5](#0-4) 

The `error_refund` feature only covers the case where the NEP-141 `ft_transfer` promise is scheduled and then fails asynchronously: [6](#0-5) 

It provides zero protection when the precompile call itself returns an error before any promise is created.

---

### Impact Explanation

**Permanent freezing of funds (High).**

After a failed `withdrawToNear`:
- The ERC-20 `totalSupply` is permanently reduced by `amount`.
- Aurora's NEP-141 balance for that token is **not** reduced — the tokens remain locked inside the engine contract with no recovery path.
- The ERC-20 mirror `totalSupply` no longer reflects the actual NEP-141 tokens held by Aurora, breaking the 1:1 invariant that the bridge depends on.
- The affected user's NEP-141 tokens are permanently inaccessible: they cannot re-mint ERC-20 tokens (no admin-controlled re-mint for users), and the NEP-141 tokens cannot be retrieved because no `ft_transfer` was ever issued.

---

### Likelihood Explanation

**Medium.** The trigger is reachable by any unprivileged EVM user:

- A user who accidentally passes an empty `recipient` (`bytes("")`) or a recipient with invalid NEAR account ID characters (spaces, uppercase, leading/trailing dots, etc.) will silently lose their tokens.
- A malicious actor can deliberately call `withdrawToNear` with an invalid recipient to permanently destroy their own ERC-20 tokens while keeping the NEP-141 tokens locked — breaking the bridge's accounting invariant for that token.
- No special privilege, admin key, or governance action is required.

---

### Recommendation

Add an explicit revert if the precompile call fails:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ERR_EXIT_TO_NEAR_FAILED");
}
```

Because `_burn` precedes the precompile call, the `require` will cause the EVM to revert the entire transaction (including the burn) if the precompile returns failure, preserving the 1:1 accounting invariant. Apply the same fix to `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora via `deploy_erc20_token` → `ft_on_transfer`. User receives `N` ERC-20 tokens; Aurora holds `N` NEP-141 tokens.
2. User calls `erc20.withdrawToNear(bytes(""), N)` (empty recipient).
3. `_burn` succeeds: ERC-20 `totalSupply` drops from `N` to `0`.
4. Precompile call fails silently (empty bytes cannot be parsed as a valid NEAR `AccountId`).
5. No NEAR promise is scheduled; no `exit_to_near_precompile_callback` fires.
6. Query `erc20.totalSupply()` → `0`. Query NEP-141 `ft_balance_of(aurora)` → `N`. The discrepancy is permanent. The user's `N` NEP-141 tokens are frozen inside Aurora forever.

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
