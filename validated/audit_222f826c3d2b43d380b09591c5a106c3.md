### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check Using Overly Broad Mempool Condition — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway stateful validator skips the `__validate__` entry-point call (which is the account's signature-verification step) for any invoke transaction with `nonce=1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because that helper returns `true` for **any** transaction type in the pool — not specifically a `deploy_account` — an unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can submit a second invoke transaction with an arbitrary/invalid signature for the same account, have it admitted by the gateway without signature verification, and permanently occupy the victim's `nonce=1` slot.

---

### Finding Description

`skip_stateful_validations` is the sequencer's analog of the external report's "fallback authorization path that uses the wrong identity." The external bug used `msg.sender` (the intermediary contract) instead of the actual initiating user; here the gateway uses the presence of *any* account transaction in the mempool as a proxy for "this account authorized the incoming transaction," which is equally wrong.

**Relevant code path:**

`extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `skip_stateful_validations` [1](#0-0) 

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ← checks ANY tx in pool, not specifically deploy_account
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
```

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate=false`, `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without ever calling `__validate__`: [3](#0-2) 

```rust
ApiTransaction::Invoke(_) => {
    tx.perform_pre_validation_stage(self.state(), &tx_context)?;
    if !tx.execution_flags.validate {
        return Ok(());   // ← __validate__ never called
    }
```

`account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type: [4](#0-3) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

The code comment in `skip_stateful_validations` claims this is safe because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The reasoning is flawed: future-nonce transactions that passed `__validate__` proved *those* transactions' signatures are valid, not the signature of the *current* nonce=1 transaction being submitted.

---

### Impact Explanation

**High — Gateway/mempool admission accepts an invalid (unauthorized) transaction before sequencing.**

An attacker can submit an invoke transaction with an arbitrary signature for any account whose `deploy_account` is pending in the mempool. The gateway accepts it without calling `__validate__`. The transaction is forwarded to the mempool and occupies the victim's `nonce=1` slot. When the batcher later executes it, `__validate__` is called (the batcher always validates), the invalid signature causes a revert, and the nonce is permanently incremented to `2`. The victim's legitimate `nonce=1` transaction is rejected by the mempool as a duplicate nonce and can never be submitted again at that nonce.

Concrete consequences:
- **Unauthorized transaction admitted** by the gateway (broken admission invariant).
- **Permanent nonce disruption**: victim's first post-deploy transaction is blocked; all subsequent transactions must use `nonce ≥ 2`.
- **Fee drain**: if the account was pre-funded, the reverted transaction still charges a fee from the victim's balance.

---

### Likelihood Explanation

- The attack window is the period between a `deploy_account` entering the mempool and being committed to a block — a window that is observable by any mempool participant.
- The attack requires no privilege: any party can submit a transaction to the gateway.
- The attacker needs only to know the victim's contract address (derivable from the `deploy_account` transaction fields).
- The attack is cheap: one gateway call with a crafted invoke transaction.
- Accounts that use the deploy-account-plus-invoke UX flow (the intended use case for this feature) are the exact population at risk.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the account in the mempool. Add a new mempool API such as `deploy_account_in_pool(address) -> bool` that inspects transaction types, and use that in `skip_stateful_validations`. Alternatively, remove the skip entirely and require users to wait for the `deploy_account` to be committed before submitting the first invoke (accepting the UX tradeoff). [5](#0-4) 

---

### Proof of Concept

```
1. Victim X generates a key pair; computes their contract address A.
2. X submits deploy_account { sender: A, nonce: 0, signature: <valid> } to the gateway.
   → Gateway accepts; deploy_account enters the mempool.

3. Attacker observes A in the mempool via account_tx_in_pool_or_recent_block.
4. Attacker submits invoke { sender: A, nonce: 1, signature: [0xdead, 0xbeef], calldata: [...] }.

5. Gateway stateful validation:
   a. get_nonce_from_state(A) → 0  (account not deployed yet)
   b. validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap → passes
   c. validate_by_mempool: nonce/fee check → passes
   d. skip_stateful_validations:
        tx.nonce() == 1 && account_nonce == 0 → true
        account_tx_in_pool_or_recent_block(A) → true (deploy_account is there)
        → returns skip_validate = true
   e. run_validate_entry_point: validate=false → __validate__ NOT called
   → Gateway accepts the attacker's transaction.

6. Attacker's invoke enters the mempool at (A, nonce=1).

7. X tries to submit their legitimate invoke { sender: A, nonce: 1, signature: <valid> }.
   → Mempool rejects: DuplicateNonce at (A, 1).

8. Batcher executes deploy_account → A deployed, on-chain nonce = 1.
9. Batcher executes attacker's invoke → __validate__ called → invalid signature → REVERT.
   → On-chain nonce incremented to 2.

10. X's legitimate nonce=1 transaction is permanently invalid.
    X must resubmit all future transactions starting at nonce=2.
```

```rust
/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
