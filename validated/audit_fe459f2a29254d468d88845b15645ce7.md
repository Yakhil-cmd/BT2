### Title
Unprivileged Caller Can Permanently Freeze All Stakers' Consensus Rewards via `update_rewards(disable_rewards=true)` — (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is callable by any unprivileged address. An attacker can invoke it with `disable_rewards=true` to mark the current block as processed without distributing any rewards. Because `last_reward_block` is a single global storage slot written unconditionally before the `disable_rewards` guard, one such call per block is sufficient to block reward distribution for every staker in the system. Repeating this every block permanently freezes all unclaimed consensus yield.

### Finding Description
`update_rewards` is exposed through `StakingRewardsManagerImpl` with `#[abi(embed_v0)]`. Its only gate is `general_prerequisites`, which checks that the contract is not paused and the caller is not the zero address — no role or identity check is performed.

The critical ordering inside the function is:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);   // ← written unconditionally

if disable_rewards || self.is_pre_consensus() {
    return;                                            // ← exits without distributing
}
```

`last_reward_block` is a single global value (not per-staker):

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
```

Because the write happens before the guard, any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

An attacker therefore needs only to call `update_rewards(any_active_staker, disable_rewards=true)` once per block to consume the block's reward slot for the entire protocol.

The contrast with the pre-consensus path is instructive: `update_rewards_from_attestation_contract` is gated by `assert_caller_is_attestation_contract`, making it callable only by the trusted attestation contract. `update_rewards` has no equivalent restriction.

### Impact Explanation
By calling `update_rewards(active_staker, true)` at the first transaction of every block, an attacker permanently prevents all stakers from accumulating consensus rewards. This constitutes **permanent freezing of unclaimed yield** for all participants in the protocol — a High-severity impact under the allowed scope.

### Likelihood Explanation
The function is publicly callable with no access control beyond a non-zero-address check. Active staker addresses are fully discoverable on-chain via emitted `NewStaker` events. The attacker gains nothing financially but can sustain the attack indefinitely at the cost of gas alone, making this a realistic griefing scenario.

### Recommendation
Two complementary fixes should be applied:

1. **Move `last_reward_block.write` after the `disable_rewards` guard**, so that a call with `disable_rewards=true` does not consume the block's reward slot:
   ```cairo
   if disable_rewards || self.is_pre_consensus() {
       return;
   }
   self.last_reward_block.write(current_block_number); // moved here
   ```

2. **Add access control** so that only the staker themselves, a designated relayer, or the attestation contract can call `update_rewards`. This mirrors the existing restriction on `update_rewards_from_attestation_contract`.

### Proof of Concept
1. Consensus rewards are active (`is_pre_consensus()` returns `false`; `consensus_rewards_first_epoch` has been set and passed).
2. Staker A is registered with a non-zero STRK balance that has been effective for at least K epochs.
3. At block N, the attacker calls `update_rewards(staker_A_address, disable_rewards=true)`.
4. `last_reward_block` is written to N; the function returns early — no rewards are distributed.
5. Any legitimate call to `update_rewards` at block N reverts with `REWARDS_ALREADY_UPDATED`.
6. All stakers lose rewards for block N.
7. The attacker repeats at blocks N+1, N+2, … permanently freezing all consensus yield across the protocol.
