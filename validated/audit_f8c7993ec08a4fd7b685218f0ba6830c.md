### Title
Unauthorized Token Transfer via Missing Caller-Sender Authorization Check in Bank Precompile `transfer` Method - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract.Run()` handler for the `transfer` method in the bank precompile accepts an arbitrary `sender` address from ABI-decoded input but never verifies that `contract.Caller()` equals that `sender`. Any unprivileged caller can therefore drain `evm/<caller_address>`-denominated native bank tokens from any victim who holds them, without the victim's consent.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks three arguments from the ABI input: `sender`, `recipient`, and `amount`. The `from` address used in `bankKeeper.SendCoins` is set directly to the ABI-supplied `sender`, while the token denomination is derived from `contract.Caller()` (i.e., `evm/<caller_address>`). There is no check that `contract.Caller() == sender`.

```go
// bank.go lines 175-192 (simplified)
sender := args[0].(common.Address)   // attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
from := sdk.AccAddress(sender.Bytes())  // arbitrary victim address
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())    // evm/<attacker_address>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))  // no auth check
```

Compare this with the `mint`/`burn` cases in the same file, which correctly use `contract.Caller()` to derive the denom but operate on a caller-supplied `recipient` — those cases do not need a caller-vs-sender check because they only affect the recipient. The `transfer` case is different: it moves tokens *out of* an arbitrary `from` address.

The `exec()` helper used by the relayer precompile does enforce `caller == signer` (see `utils.go` lines 46-48), but the bank precompile does not use that helper and has no equivalent guard.

### Impact Explanation
Any address that has previously minted `evm/<attacker_address>` tokens to users (via the `mint` method of the same precompile) can later call `transfer(victim, attacker, amount)` to reclaim or redirect those tokens from any holder without the holder's authorization. This is an unauthorized transfer of precompile-controlled native bank assets — a Critical impact under the allowed scope.

### Likelihood Explanation
The bank precompile is a live, permissionlessly callable precompile at address `0x0000...0064`. Any EOA or contract can call it. The only precondition is that the victim holds tokens of denom `evm/<caller_address>`, which is satisfied whenever the caller has previously minted such tokens to users. This is the normal intended use of the `mint`/`transfer` pair, making the precondition realistic.

### Recommendation
Add a check in the `TransferMethodName` case that the ABI-supplied `sender` matches `contract.Caller()`:

```go
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer from sender")
}
```

This mirrors the pattern already used in `utils.go` (`exec` function) where `caller != signer` is rejected.

### Proof of Concept

1. Attacker EOA `0xATK` calls bank precompile `mint(0xVICTIM, 1000)` — this mints 1000 `evm/0xATK` tokens to `0xVICTIM`.
2. Attacker calls bank precompile `transfer(0xVICTIM, 0xATK, 1000)`:
   - `sender = 0xVICTIM` (ABI argument, attacker-controlled)
   - `recipient = 0xATK`
   - `denom = evm/0xATK` (derived from `contract.Caller() = 0xATK`)
   - `bankKeeper.SendCoins(ctx, 0xVICTIM, 0xATK, 1000 evm/0xATK)` executes with no authorization check.
3. 1000 `evm/0xATK` tokens are transferred from `0xVICTIM` to `0xATK` without `0xVICTIM`'s consent.
