### Title
Unprivileged Caller Can Permanently Freeze Staker Block Rewards via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

### Summary
Any unprivileged caller can invoke `update_rewards` with `disable_rewards: true` on behalf of any staker. This marks the current block as "rewards already updated" (`last_reward_block`) without distributing any rewards. Because the contract enforces one reward update per block, the legitimate call with `disable_rewards: false` is permanently blocked for that block. Repeating this every block permanently freezes a staker's unclaimed yield.

### Finding Description
`update_rewards` in `StakingRewardsManagerImpl` is a public ABI function with no caller access control. Its only gate is `general_prerequisites()` (pause check) and a per-block uniqueness guard:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // block consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // rewards skipped
    }
    ...
``` [1](#0-0) 

The attacker's flow:
1. At the start of every block, call `update_rewards(victim_staker, disable_rewards: true)`.
2. `last_reward_block` is written to the current block number.
3. The function returns early — no STRK or BTC block rewards are computed or forwarded to the staker or their pool.
4. Any subsequent legitimate call to `update_rewards(victim_staker, false)` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. That block's rewards are permanently unrecoverable; the epoch accumulates zero rewards for the targeted staker.

The `disable_rewards` flag is a caller-supplied boolean with no privilege check, making this fully attacker-controlled.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

A staker who is continuously targeted never accumulates block rewards. Because `last_reward_block` is a single global slot per staker call, and the block window is gone once the block is finalized, the lost rewards cannot be retroactively recovered. Over time this is equivalent to a complete denial of yield for the targeted staker(s).

### Likelihood Explanation
**High.** The attack requires only a standard Starknet transaction per block. Gas costs on Starknet are low. The attacker needs no special role, no stake, and no prior relationship with the victim. The function is publicly callable with no rate-limiting beyond the one-per-block guard, which the attacker exploits rather than circumvents.

### Recommendation
Restrict `update_rewards` to a trusted caller (e.g., the sequencer operator address, the attestation contract, or a dedicated rewards-manager role). Alternatively, derive `disable_rewards` from on-chain state rather than accepting it as a caller-supplied parameter, so no external party can suppress reward distribution.

### Proof of Concept
1. Staker `S` is registered and active in the consensus rewards phase.
2. Attacker `A` (any EOA) monitors the mempool/block production.
3. At each new block `B`, `A` submits: `staking.update_rewards(S, disable_rewards: true)`.
4. `last_reward_block` is set to `B`; no rewards are distributed.
5. Any honest call `staking.update_rewards(S, false)` in block `B` reverts with `REWARDS_ALREADY_UPDATED`.
6. After `N` blocks, staker `S` has accumulated zero block rewards despite being fully active and attesting correctly. [2](#0-1)
