### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Unverified Share Values — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` blindly aggregates `CKDOutput` values from all participants without any cryptographic proof of correct computation. A malicious participant can send arbitrary `(big_y, big_c)` values, causing the coordinator to output a corrupted CKD result that does not correspond to the correct derived key, directly matching the "blind acceptance without validation" vulnerability class of the external report.

---

### Finding Description

In `do_ckd_coordinator` at lines 50–55, the coordinator receives `CKDOutput` from every participant and unconditionally accumulates the values:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

`recv_from_others` confirms the sender is a known participant, but it performs no cryptographic check on the *content* of the message. The protocol specification requires each participant to compute:

- `big_y = lambda_i · y_i · G` (blinding term)
- `big_c = lambda_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` (masked share)

as implemented in `compute_signature_share`: [2](#0-1) 

There is no zero-knowledge proof, commitment-then-reveal, or any other binding mechanism that forces a participant to send correctly computed values. The coordinator discards the sender identity (`_`) and blindly adds whatever `(big_y, big_c)` pair arrives.

This is the direct analog of the external report: just as the withdrawal precompile blindly accepted a `Transfer` without checking its source and target (allowing a CALLCODE with source = target to trigger a withdrawal without moving funds), the CKD coordinator blindly accepts participant share outputs without checking whether they were honestly computed, allowing a malicious participant to inject arbitrary group elements into the aggregation.

---

### Impact Explanation

The coordinator outputs `CKDOutput { big_y, big_c }` where the application later recovers the confidential key as:

```
confidential_key = big_c − app_sk · big_y
```

Expected result: `msk · H(pk ‖ app_id)`.

If participant `P_m` sends `(0, 0)` instead of their honest share, the coordinator's sum omits `P_m`'s Lagrange-weighted contribution. The application then computes:

```
confidential_key = Σ_{i≠m} λ_i · x_i · H(pk ‖ app_id)  ≠  msk · H(pk ‖ app_id)
```

More generally, `P_m` can send any `(Δ_y, Δ_c)`, shifting the output by an attacker-controlled delta. The coordinator and all honest parties accept this corrupted output as the protocol result with no error signal.

**Impact class (allowed scope):** *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.* [3](#0-2) 

---

### Likelihood Explanation

Any legitimate participant in the CKD session is the attacker. No leaked keys, side channels, or cryptographic breaks are required. The attacker simply deviates from the protocol by sending `(0, 0)` or any other group elements in place of their honest `(norm_big_y, norm_big_c)`. The coordinator has no mechanism to detect or reject this deviation. The attack is single-round and requires no coordination with other parties.

---

### Recommendation

1. **Proof of correct computation:** Require each participant to attach a Schnorr-style zero-knowledge proof that `big_c` and `big_y` were computed from their committed signing share `x_i` and a fresh random `y_i`. The coordinator verifies all proofs before aggregating.
2. **Commitment-then-reveal:** Have participants commit to `(big_y, big_c)` in a first round (hash commitment), then open in a second round. This prevents adaptive manipulation but does not prevent sending incorrect values; a ZKP is still needed for full protection.
3. **Verifiable secret sharing integration:** Bind the CKD share computation to the public polynomial commitments already established during DKG, so the coordinator can verify `big_c` is consistent with the participant's public key share.

---

### Proof of Concept

1. Honest participants `P_1 … P_{n-1}` and malicious `P_m` all join a CKD session with coordinator `C`.
2. `P_m` computes the honest `(norm_big_y_m, norm_big_c_m)` but instead sends `(0, 0)` to `C`.
3. `recv_from_others` accepts the message (sender is a valid participant, message deserializes correctly).
4. The coordinator accumulates `norm_big_y += 0` and `norm_big_c += 0`, silently dropping `P_m`'s share.
5. `C` outputs `CKDOutput { big_y: Σ_{i≠m} …, big_c: Σ_{i≠m} … }`.
6. The application calls `ckd_output.unmask(app_sk)` and obtains `Σ_{i≠m} λ_i · x_i · H(pk ‖ app_id)` instead of `msk · H(pk ‖ app_id)`.
7. All honest parties accept this incorrect derived key as the protocol output, with no error or abort. [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
