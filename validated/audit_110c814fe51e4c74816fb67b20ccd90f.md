### Title
Missing Sign Validation on WalletConnect Bigint Amount/Fee Parameters Allows Negative Value Spoofing in Confirmation Dialog - (File: packages/gui/src/electron/commands/parseDappParams.ts)

### Summary
`parseDappParams` validates `bigint`-typed WalletConnect parameters for type, safe-integer range, and canonical format, but performs **no sign check**. A malicious WalletConnect dapp can supply a negative `amount` or `fee` (e.g., `"-1000000000000"`) to `chia_sendTransaction`, `chia_spendCAT`, `chia_cancelOffer`, and similar spend commands. The value passes the gate, is forwarded to the confirmation dialog, and is rendered as a negative XCH/CAT amount — causing the user to approve a transaction whose displayed amount does not match any legitimate spend.

### Finding Description
In `parseDappParams.ts`, the `bigint` branch (lines 75–96) checks:
- that the value is a `string`, `number`, or `bigint`
- that numeric values are safe integers
- that string values are non-empty and canonical

It does **not** check `bigintValue < 0n`. [1](#0-0) 

By contrast, the dedicated `parseMojos` utility — used elsewhere in the codebase — explicitly rejects negative values by default: [2](#0-1) 

The `humanizeParamValue` display path for `mojo-to-xch` calls `formatMojoXch`, which casts the value directly to `string | number` and passes it to `mojoToChiaLocaleString` **without** calling `parseMojos`: [3](#0-2) 

So a negative bigint flows through `parseDappParams` → confirmation dialog display → daemon call without any rejection.

The `amount` and `fee` fields for `chia_sendTransaction` and `chia_spendCAT` are both declared `type: 'bigint'` with `humanize: 'mojo-to-xch'` / `humanize: 'mojo-to-cat'`: [4](#0-3) 

Note: `formatMojoCat` does call `parseMojos` and would reject a negative CAT amount at display time, but `formatMojoXch` does not, so XCH `amount` and all `fee` fields are unprotected at the display layer as well.

### Impact Explanation
A malicious WalletConnect dapp sends `chia_sendTransaction` with `amount: "-1000000000000000"`. The GUI:
1. Accepts it as `-1000000000000000n` through `parseDappParams`
2. Renders it as a negative XCH figure in the confirmation dialog
3. Forwards the negative value to the wallet daemon on user approval

The user sees a nonsensical or misleading amount (e.g., "-1,000,000 XCH") and may approve without understanding what they are signing. This is a **High** impact: spoofing of the displayed amount in a WalletConnect spend approval, causing the user to approve the wrong amount/status.

### Likelihood Explanation
Any paired WalletConnect dapp — an unprivileged actor — can craft this payload. No leaked keys, host compromise, or social engineering beyond the normal WalletConnect pairing flow is required. The attacker only needs to be a connected dapp.

### Recommendation
Add a negativity check in the `bigint` branch of `parseDappParams` for parameters that represent mojos (amounts and fees). The simplest fix is to call `parseMojos(bigintValue)` (which already rejects negatives by default) instead of raw `BigInt(value)`, or add an explicit `if (bigintValue < 0n) throw new Error(...)` guard after line 91. Alternatively, introduce a `nonNegativeBigint` param type distinct from `bigint` and use it for all amount/fee fields. [5](#0-4) 

### Proof of Concept
A WalletConnect dapp sends the following JSON payload for `chia_sendTransaction`:

```json
{"amount":"-1000000000000000","fee":"0","address":"txch1validaddress"}
```

`parseDappParams('chia_sendTransaction', payload)` returns `{ amount: -1000000000000000n, fee: 0n, address: "txch1validaddress", wallet_id: 1 }` without error. The confirmation dialog renders the amount via `mojoToChiaLocaleString(-1000000000000000n)`, displaying a negative XCH value. If the user confirms, the daemon receives `amount: -1000000000000000`.
