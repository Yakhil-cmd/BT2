### Title
Malicious Participant Can Silently Corrupt CKD Output via Unverified Contribution Aggregation — (`src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator blindly aggregates `(norm_big_y, norm_big_c)` values received from participants with no cryptographic verification that each contribution is correctly formed. A single malicious participant can inject arbitrary group elements, silently corrupting the final ElGamal ciphertext and causing all honest parties to accept an incorrect confidential key derivation output.

### Finding Description

In `do_ckd_coordinator` the coordinator collects each participant's share and sums them unconditionally:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The correct contribution from participant `i` is:

- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

computed inside `compute_signature_share`: [2](#0-1) 

No proof-of-correct-formation accompanies the message. The coordinator performs no consistency check — not against the sender's public key share, not against the Lagrange coefficient, and not against any commitment. The protocol has no final verification step analogous to the signature-verify call present in the OT-based ECDSA signing coordinator: [3](#0-2) 

or the `aggregate()` call in FROST that rejects bad shares: [4](#0-3) 

The CKD path has neither guard. Compare this with the DKG protocol, which verifies every participant's polynomial commitment and proof-of-knowledge before accepting any contribution: [5](#0-4) 

The structural parallel to the external report is exact: just as the Paraspace oracle consumed a spot price from an attacker-controlled low-TVL pool without cross-validation, the CKD coordinator consumes attacker-controlled group elements without cross-validation.

### Impact Explanation

The aggregated output `(big_y, big_c)` is an ElGamal ciphertext intended to encrypt `msk · H(pk ‖ app_id)` under `app_pk`. A malicious participant who substitutes arbitrary `(big_y', big_c')` shifts the ciphertext by an attacker-chosen offset. The TEE that calls `ckd_output.unmask(app_sk)` will recover a wrong scalar; the derived confidential key is silently wrong. All honest parties — including the coordinator — accept and store this corrupted output with no error signal.

This matches: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

- Any single participant in the CKD session can mount the attack.
- The attack requires only sending two arbitrary `G1` points instead of the correct ones — no cryptographic capability beyond participation is needed.
- The attack is undetectable at runtime because there is no post-aggregation verification.
- The attacker does not need to be the coordinator; a non-coordinator participant suffices.

### Recommendation

Add a zero-knowledge proof of correct formation alongside each `(norm_big_y, norm_big_c)` message. Concretely, each participant should prove in zero-knowledge that:

1. `norm_big_y = λ_i · y_i · G` for a known `y_i` (a Schnorr PoK of `y_i`).
2. `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` is consistent with the participant's public key share `x_i · G2` (a Chaum-Pedersen proof linking the discrete log of `norm_big_c - λ_i · x_i · H(...)` in base `app_pk` to the same `y_i`).

The coordinator must verify all proofs before aggregating. Alternatively, use a verifiable secret sharing scheme for the `y_i` randomness so that each contribution can be checked against a public commitment.

### Proof of Concept

```
Setup: N=3 participants P1 (honest), P2 (honest), P3 (malicious), coordinator = P1.

1. P1 and P2 compute correct (norm_big_y_1, norm_big_c_1) and (norm_big_y_2, norm_big_c_2)
   and send them to P1 (coordinator).

2. P3 sends (R, S) where R, S are arbitrary random G1 points unrelated to its
   private share or the Lagrange coefficient.

3. Coordinator executes lines 50-55:
       big_y = norm_big_y_1 + norm_big_y_2 + R
       big_c = norm_big_c_1 + norm_big_c_2 + S

4. The resulting CKDOutput is returned as Ok(Some(ckd_output)).

5. TEE calls ckd_output.unmask(app_sk):
       result = big_c - app_sk * big_y
              = (correct_c + S) - app_sk * (correct_y + R)
              = msk·H(pk,app_id) + S - app_sk·R   ← wrong key, no error raised.

6. All honest parties store and use this corrupted key with no indication of failure.
``` [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
