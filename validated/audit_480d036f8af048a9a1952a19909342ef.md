### Title
Pool Admin Can Instantly Raise Swap Fees to Maximum Cap Without Timelock, Enabling Front-Run Fee Extraction — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` and `setPoolBinAdditionalFees` allow the pool admin to change live swap-fee parameters in a single transaction with no delay. The protocol already enforces a configurable `priceProviderTimelock` for oracle rotations, but applies no equivalent delay to fee mutations. A pool admin can front-run any pending swap, raise fees to the hard cap in the same block, and collect the inflated fee revenue immediately.

---

### Finding Description

`MetricOmmPoolFactory` exposes two pool-admin fee setters:

**`setPoolAdminFees`** — no timelock, only a cap check:

```solidity
function setPoolAdminFees(address pool, uint24 newAdminSpreadFeeE6, uint24 newAdminNotionalFeeE8)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
    ...
    IMetricOmmPoolFactoryActions(pool)
      .setPoolFees(c.protocolSpreadFeeE6 + c.adminSpreadFeeE6, ...);
}
``` [1](#0-0) 

The caps default to `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) and `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` (1 %), and the factory owner can set them anywhere up to those hard limits via `setFeeCaps`. [2](#0-1) 

**`setPoolBinAdditionalFees`** — no timelock, no cap at all:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool-level `setBinAdditionalFees` writes the raw `uint16` values directly into `BinState` with no upper-bound check, so the admin can set per-bin additional fees up to 65 535 (≈ 6.55 % in E6 units) on any bin. [4](#0-3) 

Both fees are applied directly during `_executeSwap` and `getSellAndBuyPrices`: [5](#0-4) 

By contrast, the oracle-rotation path enforces a mandatory delay stored in `priceProviderTimelock[pool]` and checked in `executePoolPriceProviderUpdate`: [6](#0-5) 

The asymmetry is the root cause: oracle changes are timelocked; fee changes are not.

---

### Impact Explanation

A pool admin who observes a large pending swap in the mempool can, in the same block and with higher gas priority:

1. Call `setPoolAdminFees` to raise `adminSpreadFeeE6` from 0 to `maxAdminSpreadFeeE6` (up to 20 %) and `adminNotionalFeeE8` to `maxAdminNotionalFeeE8` (up to 1 %).
2. Optionally call `setPoolBinAdditionalFees` on the active bin to add up to 6.55 % more.
3. Let the victim's swap execute at the inflated rate.
4. Call `collectPoolFees` to harvest the excess.
5. Restore fees to their original values.

The victim trader receives far less output than the quoted price implied, constituting a direct loss of principal. For a $1 M swap at a 20 % spread fee the loss is up to $200 K in a single transaction. The attack requires no special privilege beyond the pool-admin role, which is semi-trusted per the contest scope.

---

### Likelihood Explanation

The pool admin is explicitly classified as semi-trusted ("pool admin is semi-trusted only inside caps and timelocks"). The fee-change path has no timelock, so the semi-trust assumption is violated. The attack is a single atomic transaction requiring no external dependencies, no flash loan, and no oracle manipulation. Any pool admin — whether malicious from inception or compromised after deployment — can execute it.

---

### Recommendation

Introduce a fee-change timelock analogous to `priceProviderTimelock`. Concretely:

- Add a `feeChangeTimelock[pool]` mapping (set at `createPool`, similar to `priceProviderTimelock`).
- Replace the immediate `setPoolAdminFees` with a two-step `proposePoolAdminFees` / `executePoolAdminFees` pattern that enforces `block.timestamp >= proposedAt + feeChangeTimelock[pool]`.
- Apply the same pattern to `setPoolBinAdditionalFees`.
- Additionally, add an explicit upper-bound cap check inside `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool) so that per-bin fees cannot silently reach the `uint16` maximum.

---

### Proof of Concept

```
Setup:
  pool deployed with adminSpreadFeeE6 = 0, maxAdminSpreadFeeE6 = 200_000

Attack (single block, admin txs ordered before victim):

  Tx 1 (admin): factory.setPoolAdminFees(pool, 200_000, 1_000_000)
    → pool.setPoolFees(protocolSpread + 200_000, protocolNotional + 1_000_000)
    → spreadFeeE6 and notionalFeeE8 updated immediately in pool slot 0

  Tx 2 (victim): pool.swap(recipient, zeroForOne=true, amountSpecified=1_000_000e6, ...)
    → _getBidAndAskPriceX64() returns oracle bid/ask
    → SwapMath applies spreadFeeE6 = 200_000 (20%) + notionalFeeE8 = 1_000_000 (1%)
    → victim receives ~21% less token1 than the pre-attack quote implied

  Tx 3 (admin): factory.collectPoolFees(pool)
    → inflated spread surplus and notional fee balance transferred to adminFeeDestination

  Tx 4 (admin): factory.setPoolAdminFees(pool, 0, 0)   // restore, avoid detection
```

The same sequence works with `setPoolBinAdditionalFees` targeting the active bin, stacking an additional ≤ 6.55 % on top.
