### Title
No-Op `TurnBridge` Implementation Permanently Disables Bridge Circuit-Breaker — (`x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` msg server handler is an unimplemented stub that returns `nil, nil` unconditionally. It performs no permission check and modifies no state. The bridge emergency disable mechanism is permanently non-functional.

### Finding Description

`msgServer.TurnBridge` at line 85–87 of `x/cronos/keeper/msg_server.go` is:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly gates on `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` before acting: [2](#0-1) 

The `CanTurnBridge` permission constant is defined and intended for exactly this guard: [3](#0-2) 

The proto schema confirms `MsgTurnBridge` carries an `enable bool` field that is supposed to toggle bridge state: [4](#0-3) 

A codebase-wide search for `BridgeEnabled`, `bridge_enabled`, `IsBridgeEnabled`, and `enable.*bridge` returns zero results — confirming no bridge-enabled state is ever written or read anywhere. The toggle was never implemented.

### Impact Explanation

Any address holding `CanTurnBridge` permission (or the Cronos admin) that submits `MsgTurnBridge{Enable: false}` to halt the bridge in an emergency receives a success response but achieves nothing. Bridge operations (IBC transfers, voucher conversions, gravity bridge flows) continue uninterrupted regardless of the message. The circuit-breaker that operators rely on to contain a live exploit or bridge incident is silently inoperative.

This maps to **High: Bypass of bridge authorization checks** — the permission-gated bridge control mechanism is entirely bypassed (no check, no effect), and **High: Permanent inability for authorized parties to halt bridge/conversion flows**.

### Likelihood Explanation

The function is reachable by any transaction sender (no permission check blocks submission). The broken behavior is permanent until a governance upgrade patches the implementation. Any operator attempting to use the emergency disable during an incident will be silently misled by the success response.

### Recommendation

Implement `TurnBridge` to:
1. Check `HasPermission(ctx, signers, CanTurnBridge)` and return `sdkerrors.ErrUnauthorized` on failure.
2. Persist the `Enable` flag into module params (add a `BridgeEnabled bool` field to `Params`) via `k.SetParams`.
3. Gate all bridge entry-points (IBC hooks, voucher conversion, gravity handlers) on this flag.

### Proof of Concept

```go
// Submit MsgTurnBridge{Enable: false} from an authorized sender
msg := types.NewMsgTurnBridge(authorizedSender, false)
resp, err := msgServer.TurnBridge(ctx, msg)
// err == nil, resp == nil  ← success response
// Bridge state: unchanged — no BridgeEnabled key exists in the KV store
// Bridge operations: still accepted
```

A unit test asserting that the KV store contains a `BridgeEnabled=false` entry after the call will fail, confirming the no-op.
