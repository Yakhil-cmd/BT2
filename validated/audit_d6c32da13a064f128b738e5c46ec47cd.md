### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a distinct epoch appends a new entry to this trace, a delegator who makes balance changes across many epochs without claiming rewards will accumulate a trace large enough to cause `claim_rewards` to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over the pool member's balance trace with no upper bound: [1](#0-0) 

The comment on line 858 explicitly acknowledges the risk: *"This loop is unbounded but unlikely to exceed gas limits."* This is the same class of assumption that was invalidated in the tBTC SPV proof issue — a gas-bounded operation is dismissed as "unlikely" to fail without a hard cap.

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`: [2](#0-1) 

The `insert` implementation in `PoolMemberBalanceTrace` only updates the last entry if the key matches; otherwise it **appends** a new checkpoint: [3](#0-2) 

This means every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a **different epoch** appends one new entry. The `entry_to_claim_from` cursor stored in `pool_member_info` advances only upon a successful `claim_rewards`: [4](#0-3) 

Therefore, a pool member who makes balance changes across N distinct epochs without claiming will force the loop to iterate N times on the next `claim_rewards` call. Each iteration performs at least one storage read (`pool_member_trace.at(entry_to_claim_from)`) and a `find_sigma` call that reads from `cumulative_rewards_trace_vec`: [5](#0-4) 

The same unbounded `calculate_rewards` call is also triggered by the view function `pool_member_info_v1`: [6](#0-5) 

Once the trace is large enough to exceed the gas limit, both `claim_rewards` and `pool_member_info_v1` revert permanently. There is no pagination or partial-claim mechanism.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A pool member whose `pool_member_epoch_balance` trace grows beyond the gas-limit threshold can never successfully call `claim_rewards`. All accumulated STRK rewards are permanently inaccessible. The principal (delegated tokens) is recoverable via `exit_delegation_pool_intent` + `exit_delegation_pool_action`, which do not invoke `calculate_rewards`, but the yield is irrecoverably frozen. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** A delegator who dollar-cost-averages into a pool — calling `add_to_delegation_pool` once per epoch — grows the trace by one entry per epoch. Starknet's per-transaction resource limits are significantly lower than Ethereum's 30 M gas, and each loop iteration involves multiple storage reads plus arithmetic. Empirically, a few hundred iterations of storage-heavy loops can exhaust Starknet's step/resource budget. A delegator active for several hundred epochs without claiming (e.g., a long-term holder who delegates and forgets) reaches this threshold naturally. No adversarial action by a third party is required; the delegator's own normal usage triggers the freeze.

---

### Recommendation

1. **Introduce a `max_iterations` parameter** to `claim_rewards` (or an internal cap) so the loop processes at most `N` trace entries per call, advancing `entry_to_claim_from` and returning partial rewards. Subsequent calls continue from where the previous left off.
2. **Alternatively**, enforce a maximum trace length per pool member by requiring that `claim_rewards` is called before a configurable number of unclaimed balance-change epochs accumulates (e.g., reject `add_to_delegation_pool` if `pool_member_trace_length - entry_to_claim_from > MAX_UNCLAIMED_ENTRIES`).
3. **Benchmark** the gas cost of `calculate_rewards` as a function of loop iterations on Starknet mainnet to establish a concrete safe upper bound, analogous to the tBTC recommendation to benchmark SPV proof verification costs.

---

```text
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
```
