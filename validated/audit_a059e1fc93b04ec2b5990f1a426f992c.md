### Title
Owner Can Permanently Freeze All User Funds by Calling `pause_contract` - (File: engine/src/contract_methods/admin.rs)

### Summary
The Aurora Engine owner can call `pause_contract` to set `state.is_paused = true`, which causes every user-facing state-changing function — including `withdraw`, `submit`, `ft_on_transfer`, and all EVM execution paths — to revert via `require_running`. If the owner never calls `resume_contract`, all user ETH and ERC-20 token balances held inside Aurora are permanently frozen with no user-accessible escape path.

### Finding Description
`pause_contract` in `engine/src/contract_methods/admin.rs` sets `state.is_paused = true` and persists it to storage. [1](#0-0) 

The guard `require_running` returns `Err(ERR_PAUSED)` whenever `state.is_paused` is `true`. [2](#0-1) 

Every user-reachable fund-movement entry point calls this guard unconditionally before doing any work:

- `withdraw` (ETH out of Aurora → NEAR): [3](#0-2) 

- `submit` / `submit_with_args` (all EVM transactions, including ERC-20 transfers and DEX interactions): [4](#0-3) 

- `ft_on_transfer` (NEP-141 → ERC-20 bridging): [5](#0-4) 

- `exit_to_near_precompile_callback` (callback that finalises ExitToNear bridge-out): [6](#0-5) 

- `ft_transfer`, `ft_transfer_call`, `storage_withdraw`, `storage_unregister`: [7](#0-6) 

- XCC functions (`withdraw_wnear_to_router`, `fund_xcc_sub_account`, `factory_update`): [8](#0-7) 

`resume_contract` is the only way to unset the flag, and it is also gated behind `require_owner_only`. [9](#0-8) 

There is no time-lock, multi-sig requirement, or DAO veto on `pause_contract`. A single owner key is sufficient to freeze the entire protocol indefinitely.

### Impact Explanation
When the contract is paused:
- Users cannot call `withdraw` to move their ETH from Aurora back to NEAR.
- Users cannot call `submit` to execute any EVM transaction, including ERC-20 `transfer` or DEX swaps.
- The `ExitToNear` and `ExitToEthereum` precompiles are unreachable because they are invoked from within EVM execution, which itself requires `require_running` to pass.
- No alternative withdrawal path exists in the contract.

All ETH and ERC-20 token balances held inside Aurora are effectively frozen. This constitutes **temporary (or permanent) freezing of funds**, matching the High impact tier in the allowed scope.

### Likelihood Explanation
Likelihood is **Low**. Exploiting this requires the owner account to act maliciously or be compromised. However, the owner is a single NEAR account with no on-chain time-lock or governance check on `pause_contract`, making the attack trivially executable once the key is in adversarial hands — identical to the centralization risk described in the reference report.

### Recommendation
Remove `require_running` from `withdraw` (and analogously from `exit_to_near_precompile_callback`) so that users can always retrieve their already-deposited funds even when the contract is paused. Administrative pausing should only block new deposits and new EVM execution, not existing fund withdrawals. Alternatively, introduce a time-locked or multi-sig-controlled pause mechanism so that no single key can freeze user funds indefinitely.

### Proof of Concept
1. Owner calls `pause_contract` on the Aurora Engine NEAR contract.
   - `state.is_paused` is set to `true` in storage.
2. Any user calls `withdraw` with their ETH recipient address and amount.
   - `require_running` reads `state.is_paused == true` and returns `Err(ERR_PAUSED)`.
   - The NEAR transaction fails; no ETH is moved.
3. Any user submits an EVM transaction via `submit` (e.g., an ERC-20 `transfer`).
   - Same `require_running` guard fires; the transaction is rejected.
4. Owner never calls `resume_contract`.
   - All user balances remain locked in Aurora with no on-chain recourse. [1](#0-0) [10](#0-9) [11](#0-10)

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}

pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
