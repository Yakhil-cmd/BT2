### Title
Duplicate entries in `tx.Authorizers` bypass multi-authorizer signature requirements — (File: `fvm/environment/transaction_info.go`)

---

### Summary

The `Authorizers` field of a `flow.TransactionBody` is a user-controlled slice. No stage of the transaction validation pipeline rejects or deduplicates repeated addresses in that slice. Because every signature-presence and key-weight check uses a map keyed by address, a single account's one valid signature satisfies the check for every occurrence of that address. The FVM then forwards the raw, un-deduplicated slice verbatim to the Cadence runtime as the signing-accounts list. A transaction with `Authorizers = [addr, addr]` causes the runtime to receive two `&Account` references to the same account while only one valid signature was provided, effectively bypassing any multi-authorizer gate enforced by a Cadence contract.

---

### Finding Description

**Stage 1 — Access/Collection Node validation (`access/validator/validator.go`, `checkAccounts`)**

The authorizer-signature loop (lines 452–460) iterates over `tx.Authorizers` and, for each entry, performs a map lookup into `observedEnvelopeSig` / `observedPayloadSig` (both `map[flow.Address]bool`). Because the key is the address alone, a second occurrence of the same address passes the check identically to the first. There is no set-membership guard that would detect a repeated address. [1](#0-0) 

The comment block above `checkAccounts` explicitly lists the invariants it enforces (no duplicate *key* signing, at least one signature per authorizer, etc.) but says nothing about duplicate *authorizer addresses*, confirming the omission is not intentional. [2](#0-1) 

**Stage 2 — Execution Node FVM verification (`fvm/transactionVerifier.go`, `verifyTransaction`)**

The weight-check loop (lines 249–264) iterates over `tx.Authorizers` and calls `hasSufficientKeyWeight(payloadWeights, addr, ...)`. `payloadWeights` is a `map[flow.Address]int` built from the already-deduplicated signature entries. A duplicate authorizer address passes the weight check on every iteration because the map value is unchanged between iterations. [3](#0-2) 

**Stage 3 — Cadence runtime authorizer construction (`fvm/environment/transaction_info.go`, `NewTransactionInfo`)**

Lines 128–135 append every element of `params.TxBody.Authorizers` to `runtimeAddresses` with no deduplication. `GetSigningAccounts()` (lines 165–170) returns this slice verbatim. The Cadence runtime uses this list to bind `&Account` values to the `prepare` block parameters. [4](#0-3) [5](#0-4) 

The three stages together mean: an attacker who submits `Authorizers = [addr, addr]` with one valid signature for `addr` clears all validation gates and causes the Cadence runtime to receive `[addr, addr]` as the signing-accounts list.

---

### Impact Explanation

Any Cadence transaction whose `prepare` block accepts N `&Account` parameters implicitly requires N distinct authorizing accounts. By supplying `Authorizers = [addr, …, addr]` (N times), an attacker satisfies the authorizer-count check with a single account and a single signature. The Cadence runtime then provides N `&Account` references to the same account.

Contracts that rely on the presence of N distinct signing accounts to gate privileged operations — multi-party asset transfers, multi-sig vaults, two-party escrow agreements, or any pattern of the form "account A and account B must both consent" — can be bypassed: the attacker's single account appears to satisfy all N authorization slots. This is an **entitlement/capability bypass** leading to **unauthorized account mutation**: the attacker can trigger state changes that the contract author intended to require consent from multiple independent parties.

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction to the Access API (gRPC `SendTransaction` or REST `POST /transactions`), which is open to any unprivileged user. No staked node, no leaked key, and no admin privilege is needed. The attacker controls the `Authorizers` slice directly as part of the transaction body they construct and sign. The only prerequisite is knowledge of a deployed contract that gates a valuable operation behind a multi-authorizer `prepare` block without independently verifying that the two `&Account` references belong to distinct addresses.

---

### Recommendation

**Short term:** Add a duplicate-address guard in `checkAccounts` (`access/validator/validator.go`) that rejects any transaction whose `Authorizers` slice contains repeated addresses, analogous to the existing duplicate-key-signature check. The same guard should be added in `verifyTransaction` (`fvm/transactionVerifier.go`) as a defense-in-depth measure at the execution layer.

**Long term:** Deduplicate `tx.Authorizers` in `NewTransactionInfo` (`fvm/environment/transaction_info.go`) before constructing `runtimeAddresses`, or add an assertion that the slice is already duplicate-free, so that even if a malformed transaction somehow reaches execution, the Cadence runtime never receives repeated signing accounts.

---

### Proof of Concept

```
// Attacker controls `addr` and its private key.
// Target contract requires two independent accounts to authorize:
//
//   transaction {
//     prepare(partyA: &Account, partyB: &Account) {
//       // sensitive two-party operation (e.g., release escrow)
//     }
//   }

tx := flow.TransactionBody{
    Script:      <above script>,
    Authorizers: []flow.Address{addr, addr},  // same address twice
    Payer:       addr,
    ProposalKey: flow.ProposalKey{Address: addr, KeyIndex: 0, SequenceNumber: n},
}

// One payload signature for addr satisfies both authorizer slots:
tx.PayloadSignatures  = []flow.TransactionSignature{sigForAddr}
tx.EnvelopeSignatures = []flow.TransactionSignature{envelopeSigForAddr}

// checkAccounts (validator.go:452-460):
//   iteration 1: observedPayloadSig[addr] == true  → pass
//   iteration 2: observedPayloadSig[addr] == true  → pass  (no duplicate-address guard)
//
// verifyTransaction (transactionVerifier.go:249-264):
//   iteration 1: payloadWeights[addr] >= threshold → pass
//   iteration 2: payloadWeights[addr] >= threshold → pass  (same map value)
//
// NewTransactionInfo (transaction_info.go:128-135):
//   runtimeAddresses = [addr, addr]
//
// Cadence runtime: prepare(partyA=&addr, partyB=&addr)
//   → two-party gate bypassed with one account's signature
```
