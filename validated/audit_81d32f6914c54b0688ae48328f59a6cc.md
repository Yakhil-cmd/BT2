### Title
`setPoolAdminFeeDestination` updates fee recipient without collecting accrued spread fees, causing prior destination to lose owed fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` changes the admin fee recipient address without first flushing accrued spread fees to the old destination. Every other fee-config mutator (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before writing new state. The missing flush means all spread-fee surplus earned under the old destination is silently redirected to the new one.

---

### Finding Description

The spread fee is not stored in a dedicated accumulator. It is computed on demand as the pool's token surplus:

```
surplus = balance() * SCALE - binTotals.scaled - notionalFeeScaled
``` [1](#0-0) 

When `collectFees` is called, that surplus is split between protocol and admin using `adminFeeDestination_` — a value read from `poolAdminFeeDestination[pool]` at call time: [2](#0-1) 

Both `setPoolAdminFees` and `setPoolProtocolFee` flush accrued fees **before** writing new config, so the old destination receives what it earned: [3](#0-2) 

`setPoolAdminFeeDestination` performs **no such flush**: [4](#0-3) 

After the destination is overwritten, the next `collectFees` call (permissionless, callable by anyone) sends the entire accumulated surplus — including the portion earned before the change — to the new address. The old destination receives nothing.

---

### Impact Explanation

Any spread-fee surplus that accrued while the old `adminFeeDestination` was set is permanently redirected to the new address. Because `collectFees` is permissionless, an adversarial pool admin can:

1. Let surplus accumulate over many swaps.
2. Call `setPoolAdminFeeDestination` to point to a self-controlled address.
3. Immediately call `collectPoolFees` to drain the full surplus.

The old destination (e.g., a separate treasury multisig) loses all owed admin spread fees with no recourse. Notional fees are unaffected (they are tracked in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` and cleared on collect), but spread fees — which can be the dominant fee type — are fully at risk. [5](#0-4) 

---

### Likelihood Explanation

The pool admin is semi-trusted and can call `setPoolAdminFeeDestination` at any time with no timelock and no cap. The trigger requires only a single transaction from the pool admin. In any deployment where the admin fee destination is a separate entity from the pool admin (e.g., a DAO treasury vs. an operator multisig), this path is reachable without any external precondition.

---

### Recommendation

Mirror the pattern used in `setPoolAdminFees`: call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before overwriting the mapping.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before switching.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed; `adminFeeDestination = Treasury`.
2. 1 000 swaps execute, accumulating a spread-fee surplus of S tokens in the pool balance.
3. Pool admin calls `setPoolAdminFeeDestination(pool, AttackerWallet)` — no fees collected.
4. Anyone calls `collectPoolFees(pool)` — the factory reads `poolAdminFeeDestination[pool] == AttackerWallet` and sends the admin share of S to `AttackerWallet`.
5. `Treasury` receives 0 despite having earned the admin share of S.

The inconsistency is directly visible by comparing `setPoolAdminFees` (lines 417–425, which calls `collectFees` first) with `setPoolAdminFeeDestination` (lines 438–447, which does not). [6](#0-5)
