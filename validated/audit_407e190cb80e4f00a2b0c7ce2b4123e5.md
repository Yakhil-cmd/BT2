### Title
Rolling-Window Digest Eviction Allows Same Physical Key to Accumulate Multiple Weight Slots, Bypassing Multi-Sig Threshold — (`fvm/environment/account-key-metadata/metadata.go`, `fvm/transactionVerifier.go`)

---

### Summary

The account-key deduplication system uses a rolling window of only `MaxStoredDigests=2` fast-hash digests to detect duplicate keys at add-time. Once a key's digest is evicted from that window, the same physical public key can be re-added as a brand-new "unique" key with its own independent weight entry. Because the transaction verifier aggregates weight by `(address, keyIndex)` pair — never by physical public-key bytes — an attacker can accumulate weight from multiple key-index slots that all correspond to a single private key, crossing the `AccountKeyWeightThreshold` with one signing key.

---

### Finding Description

**Digest eviction — `appendDigest`**

`AppendUniqueKeyMetadata` calls `appendDigest`, which enforces a hard cap of `MaxStoredDigests=2`:

```
digestCount = 1 + len(m.digestBytes)/digestSize
if digestCount > int(m.maxStoredDigests) {
    // Remove digest from front
    m.digestBytes = slices.Delete(m.digestBytes, 0, removeCount*digestSize)
    m.startIndexForDigests += uint32(removeCount)
}
``` [1](#0-0) 

`AppendDuplicateKeyMetadata` — the path taken when a duplicate is detected — **never calls `appendDigest`**, so the window only advances on unique-key additions. [2](#0-1) 

**Duplicate detection — `FindDuplicateKey`**

Detection is limited to the current window contents. If a key's digest has been evicted, `findDuplicateDigest` returns `false` and the key is treated as unique: [3](#0-2) 

The code explicitly acknowledges this is intentional and not 100%: [4](#0-3) 

**Weight aggregation — `getAccountKeysAndAggregateWeights`**

For every signature in the transaction, the verifier looks up the key by `(address, keyIndex)` and adds its stored weight. There is no check that two different key indices share the same physical public key:

```go
signature.aggregateWeights[signature.Address] += accountKey.Weight
``` [5](#0-4) 

**Signature deduplication — `newSignatureEntries`**

The only deduplication guard is on `(address, keyIndex)` pairs. Two signatures from different key indices that carry the same physical public key are both accepted: [6](#0-5) 

**Signature verification — `verify`**

Each signature is verified against `entry.accountKey.PublicKey`. Because all re-added copies of the same physical key resolve to the same public key bytes (via the stored-key lookup), the same cryptographic signature validates for every copy: [7](#0-6) 

---

### Impact Explanation

An attacker who controls a multi-sig account (e.g., 2-of-2, each key weight 500) can, while they still hold all keys, execute the setup sequence below and thereafter authorize transactions unilaterally with a single private key — permanently bypassing the multi-sig requirement. This constitutes unauthorized account manipulation: the account's intended authorization policy is violated without the cooperation of the other key holder(s).

---

### Likelihood Explanation

The exploit requires no privileged access, no leaked keys, no staked-node control, and no brute force. It is fully executable by any account owner through normal Cadence `addKey` transactions. The only prerequisite is that the attacker must have had sufficient combined weight to authorize the setup transactions at some point (e.g., during initial account setup or while temporarily holding all keys). The steps are deterministic and reproducible on the emulator.

---

### Recommendation

1. **Verifier-level dedup by public key bytes**: In `getAccountKeysAndAggregateWeights`, track a set of already-seen `(address, encodedPublicKey)` pairs and skip weight accumulation for any key index whose physical public key has already been counted.
2. **Persistent full-key dedup on add**: Replace or supplement the rolling-window digest with a persistent bloom filter or a full-key set so that eviction cannot create a bypass window.
3. **Invariant test**: Assert that for any account, the sum of weights across all non-revoked key indices sharing the same physical public key bytes never exceeds `AccountKeyWeightThreshold` when only one private key is held.

### Proof of Concept

**Setup** — account with key_A (weight 500) and key_B (weight 500), both keys held by attacker:

| Step | Action | Digest window after | Notes |
|------|--------|-------------------|-------|
| 0 | Create account with key_A (idx 0, w=500) | — | key metadata not yet created |
| 1 | Add unique_key_1 (idx 1, w=1) | `[digest(key_A), digest(unique_key_1)]` | key_A digest enters window |
| 2 | Add unique_key_2 (idx 2, w=1) | `[digest(unique_key_1), digest(unique_key_2)]` | **key_A digest evicted** |
| 3 | Add key_A again (idx 3, w=500) | `[digest(unique_key_2), digest(key_A)]` | `FindDuplicateKey` returns false; stored as new unique key at storedKeyIndex=3 |

**Exploit transaction** — signed with key_A's private key only, for key indices 0 and 3:

- `newSignatureEntries`: both `(addr, 0)` and `(addr, 3)` are distinct → both accepted
- `getAccountKeysAndAggregateWeights`: weight[addr] += 500 (idx 0) + 500 (idx 3) = **1000 ≥ threshold**
- `verifySignatures`: both signatures verify against key_A's public key ✓

The transaction is authorized using a single private key on an account that was intended to require two independent key holders.

func (m *KeyMetadataAppender) appendDigest(digest uint64) {
	digestCount := 1 + len(m.digestBytes)/digestSize

	if digestCount > int(m.maxStoredDigests) {
		// Remove digest from front
		removeCount := digestCount - int(m.maxStoredDigests)
		m.digestBytes = slices.Delete(m.digestBytes, 0, removeCount*digestSize)

		// Adjust digest start index
		m.startIndexForDigests += uint32(removeCount)
	}

	var digestBytes [digestSize]byte
	binary.BigEndian.PutUint64(digestBytes[:], digest)

	m.digestBytes = append(m.digestBytes, digestBytes[:]...)
}
```

	}

	return nil
}
