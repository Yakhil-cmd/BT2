### Title
`DeterministicStateInit` inside `DelegateAction` validated against outer transaction receiver instead of delegate action receiver â€” (File: `runtime/runtime/src/action_validation.rs`)

### Summary
In `validate_delegate_action`, when a `DelegateAction` wraps a `DeterministicStateInit` inner action, the receiver identity used for the cryptographic derivation check is the **outer transaction's receiver** (`receiver` parameter) rather than the **delegate action's own receiver** (`delegate_action.receiver_id()`). This is the exact same wrong-identity-for-authorization pattern as the external report: the check is performed against the wrong account, causing legitimate meta-transactions to be unconditionally rejected and allowing a crafted transaction to pass pre-inclusion validation while targeting the wrong deterministic account.

### Finding Description

In `validate_delegate_action` (`runtime/runtime/src/action_validation.rs`, lines 180â€“208), the function receives two distinct account identities:

- `receiver` â€” the outer transaction's `receiver_id` (the account that will unpack the `DelegateAction`)
- `delegate_action.receiver_id()` â€” the account the inner actions will actually execute on [1](#0-0) 

Before `ProtocolFeature::FixDelegatedDeterministicStateInit` is enabled, the code selects `receiver` (the outer tx receiver) as `inner_receiver` and passes it to `validate_actions_with_mode`, which in turn calls `validate_deterministic_state_init`: [2](#0-1) 

`validate_deterministic_state_init` enforces the core invariant: `derive_near_deterministic_account_id(state_init) == receiver_id`. With the wrong `receiver_id` supplied, this check is performed against the outer tx receiver instead of the delegate action's actual target: [3](#0-2) 

**Attack path 1 â€” Legitimate use unconditionally blocked:**

A user submits a meta-transaction where:
- outer tx `receiver = user_account` (the sender's own account, which unpacks the delegate)
- `delegate_action.receiver_id = det_account_a = derive(state_init_a)`
- inner action: `DeterministicStateInit { state_init: state_init_a }`

The buggy check evaluates `derive(state_init_a) == user_account`, which is always false. Every legitimate `DeterministicStateInit` inside a meta-transaction is rejected at pre-inclusion validation.

**Attack path 2 â€” Crafted tx passes pre-inclusion, fails at receipt:**

An attacker who controls `det_account_b = derive(state_init_b)` submits:
- outer tx `receiver = det_account_b`
- `delegate_action.receiver_id = det_account_a` (a different deterministic account)
- inner action: `DeterministicStateInit { state_init: state_init_b }`

The buggy check evaluates `derive(state_init_b) == det_account_b` â†’ passes. The transaction is admitted. The resulting receipt targets `det_account_a` with `state_init_b`. Receipt-level validation (`validate_receipt` â†’ `validate_action_receipt` â†’ `validate_deterministic_state_init`) then checks `derive(state_init_b) == det_account_a` â†’ fails, so no state is corrupted. However, the relayer's gas is consumed and the pre-inclusion invariant is broken.

The code comment in the production file explicitly acknowledges both effects: [4](#0-3) 

The exploit scenario is confirmed by the integration test `try_meta_tx_deterministic_receiver_exploit`: [5](#0-4) 

### Impact Explanation

**Scope: pre-inclusion transaction validation.**

The broken invariant is: *the `DeterministicStateInit` receiver-derivation check must be performed against the account that will actually execute the action* (`delegate_action.receiver_id()`), not the account that unpacks the delegate (`outer_tx.receiver`). Using the wrong identity causes:

1. Every legitimate `DeterministicStateInit` inside a `DelegateAction` (meta-transaction) to be rejected â€” the feature is completely unusable via meta-tx.
2. A crafted transaction to pass pre-inclusion validation while targeting the wrong deterministic account; receipt-level validation prevents state corruption, but the pre-inclusion gate is bypassed.

### Likelihood Explanation

Any unprivileged user submitting a `DelegateAction` containing `DeterministicStateInit` triggers the wrong-identity check. No special privilege is required. The feature is designed for use with meta-transactions (NEP-616), so every user of the feature is affected.

### Recommendation

Use `delegate_action.receiver_id()` as the receiver for inner-action validation, not the outer transaction's receiver. The fix is already present behind `ProtocolFeature::FixDelegatedDeterministicStateInit`:

```rust
// Correct:
let inner_receiver = delegate_action.receiver_id();
// Buggy (pre-fix):
// let inner_receiver = receiver;  // outer tx receiver â€” wrong identity
``` [6](#0-5) 

### Proof of Concept

1. Deploy `det_account_b = derive(state_init_b)` as a deterministic account and add a full-access key to it.
2. Craft a `DelegateAction` with `sender_id = det_account_b`, `receiver_id = det_account_a` (a different deterministic account), and inner action `DeterministicStateInit { state_init: state_init_b }`.
3. Wrap it in an outer transaction with `receiver = det_account_b`.
4. Submit. Pre-fix: tx passes pre-inclusion validation (check: `derive(state_init_b) == det_account_b` âś“), but the resulting receipt fails execution (check: `derive(state_init_b) == det_account_a` âś—). Post-fix (`FixDelegatedDeterministicStateInit`): tx is rejected at pre-inclusion with `InvalidDeterministicStateInitReceiver`.

The exact scenario is exercised in `try_meta_tx_deterministic_receiver_exploit` and `test_deterministic_state_init_meta_tx_receiver_check`: [7](#0-6)

fn validate_delegate_action(
    limit_config: &LimitConfig,
    delegate_action: VersionedDelegateActionRef<'_>,
    receiver: &AccountId,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    let actions = delegate_action.get_actions();
    let inner_receiver =
        if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
            // This is the correct receiver id to use for the check.
            delegate_action.receiver_id()
        } else {
            // This is a bug fixed with `FixDelegatedDeterministicStateInit` that
            // validated against the wrong id. This makes it impossible to
            // initialize deterministic accounts from meta transactions.
            // The bug cannot be abused, if someone crafts a state init that passes
            // validation here, it will fail when it is checked as incoming receipt.
            receiver
        };
    validate_actions_with_mode(
        limit_config,
        &actions,
        inner_receiver,
        current_protocol_version,
        mode,
    )?;
    Ok(())
}

fn validate_deterministic_state_init(
    limit_config: &LimitConfig,
    action: &DeterministicStateInitAction,
    receiver_id: &AccountId,
) -> Result<(), ActionsValidationError> {
    validate_global_contract_identifier(action.state_init.code())?;

    let derived_id = derive_near_deterministic_account_id(&action.state_init);

    if derived_id != *receiver_id {
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver {
            derived_id,
            receiver_id: receiver_id.clone(),
        });
    }
