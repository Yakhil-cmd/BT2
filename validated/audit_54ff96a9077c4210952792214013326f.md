### Title
WalletConnect `take_offer` Confirmation Dialog Understates Total Royalty Cost for Multi-NFT Offers - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in the WalletConnect confirmation pipeline incorrectly computes the "Total Amount with Royalties" for offers containing multiple NFTs. It divides the fungible payment amount equally among all NFTs before applying each NFT's royalty percentage, producing a displayed total that is `1/N` of the correct royalty burden (where N is the number of NFTs). A malicious dApp can exploit this to present an artificially low spending total in the approval dialog, causing the user to authorize an offer that costs significantly more than shown.

### Finding Description

When a WalletConnect-connected dApp sends `chia_wallet.take_offer`, the main process calls `parseCommandDisplay` to build the confirmation dialog. For the spending side, `withRoyaltyTotals` calls `formatAmountWithRoyalties` to compute `amountWithRoyalties`: [1](#0-0) 

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

This computes: `royaltyAmount = (amount / N) × Σ(royaltyPercentage_i) / 10000`

The correct formula — matching how Chia's royalty puzzles actually charge — is: `royaltyAmount = amount × Σ(royaltyPercentage_i) / 10000`

The code is off by a factor of N. For two NFTs the displayed royalty is half the actual royalty; for three NFTs it is one-third, and so on.

The test suite itself documents this discrepancy: with two NFTs carrying royalties of 500 bp (5 %) and 10 bp (0.1 %), and a payment of 100 000 000 mojos (0.0001 XCH), the test asserts `amountWithRoyalties: '0.00010255'`: [2](#0-1) 

The correct total is 0.0001051 XCH (royalty = 5 100 000 mojos), but the dialog shows 0.00010255 XCH (royalty = 2 550 000 mojos) — exactly half.

The understated value is rendered in the WalletConnect approval dialog under "Total Amount with Royalties": [3](#0-2) 

The dialog is opened after `parseCommandDisplay` runs, before the command is executed: [4](#0-3) 

### Impact Explanation

A user reviewing the WalletConnect approval dialog sees "Total Amount with Royalties: X XCH" and uses that figure to decide whether to accept. Because the figure is understated by a factor of N, the user may:

- Approve an offer they would have rejected had they seen the true cost, resulting in an unexpected XCH/CAT balance reduction.
- Approve when their spendable balance is between the displayed and actual totals, causing the transaction to fail after approval (or succeed and drain more funds than consented to).

This matches the **High** impact class: "WalletConnect state that causes a user to approve the wrong amount."

### Likelihood Explanation

Any dApp paired via WalletConnect can issue `chia_wallet.take_offer` with an offer blob containing multiple NFTs with non-trivial royalty percentages. No special privilege is required beyond an established WalletConnect session. The discrepancy grows linearly with the number of NFTs, making it straightforward to engineer a large gap between displayed and actual cost.

### Recommendation

Replace the split-then-sum calculation with a sum-then-apply calculation:

```typescript
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test assertions to reflect the corrected values.

### Proof of Concept

1. Attacker operates a dApp paired to the victim's wallet via WalletConnect.
2. Attacker crafts an offer blob containing two NFTs: NFT-A (royalty 500 bp / 5 %) and NFT-B (royalty 10 bp / 0.1 %), requesting 0.0001 XCH from the taker.
3. Attacker sends `chia_wallet.take_offer` via WalletConnect.
4. The confirmation dialog displays:
   - **You Spend:** 0.0001 XCH
   - **Total Amount with Royalties:** 0.00010255 XCH ← understated
5. Victim approves, expecting to spend 0.00010255 XCH.
6. The blockchain charges the correct royalties: 5 000 000 + 100 000 = 5 100 000 mojos, for a true total of **0.0001051 XCH** — ~2.4 % more than shown.

The root cause is in `formatAmountWithRoyalties`: [5](#0-4)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
