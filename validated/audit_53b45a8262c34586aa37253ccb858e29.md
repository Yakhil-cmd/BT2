### Title
`uint32` Overflow in `_afterTimelock` Allows Pool Admin to Bypass Stop-Loss Timelock — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

The `_afterTimelock` helper in `OracleValueStopLossExtension` truncates `block.timestamp + timelock` to `uint32`. When the pool admin sets a `timelock` value large enough to overflow `uint32`, the resulting `executeAfter` wraps to a **past** timestamp, making every `_requireElapsed` check pass immediately. The pool admin can therefore execute any pending stop-loss parameter change (drawdown, decay, watermarks) with zero delay, defeating the LP-protection guarantee the timelock is designed to enforce.

---

### Finding Description

`_afterTimelock` is:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space and then **silently truncated** to `uint32`. `uint32` overflows at 4,294,967,296 (≈ year 2106). With `block.timestamp ≈ 1,750,000,000` (July 2025), any `timelock > 2,544,967,295` (~80.7 years) causes the sum to exceed `uint32.max`, wrapping to a value **smaller than the current timestamp**.

The `_requireElapsed` guard is:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

Because `executeAfter` (a `uint32`) is implicitly widened to `uint256` for the comparison, a wrapped value like `1,749,999,999` is always less than `block.timestamp ≈ 1,750,000,000`, so the check passes immediately.

The `timelock` field is set at initialization with **no upper-bound validation**:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock is stored without any cap check
oracleStopLossConfig[pool] = PoolStopLossConfig({...timelock: timelock...});
``` [3](#0-2) 

The same absence of validation applies to `proposeOracleStopLossTimelock`, which accepts any `uint32 newTimelock`. [4](#0-3) 

---

### Impact Explanation

The stop-loss extension is the primary mechanism protecting LPs from sudden pool-admin parameter changes. Its comment explicitly states: *"Drawdown and decay changes are timelocked so LPs can react."* [5](#0-4) 

With the timelock bypassed, the pool admin can:

1. **Instantly set `drawdownE6 = E6` (100%)** → `floorMultiplier = E6 − E6 = 0` → the breach condition `metric < (hwm × 0) / E6 = 0` is never true → stop-loss is fully disabled. [6](#0-5) 

2. **Instantly zero out watermarks** → `_applyWatermark` with `hwm = 0` always returns `(metric, false)` → no swap is ever blocked. [7](#0-6) 

With the stop-loss disabled, swaps that would previously have been reverted (e.g., during oracle price stress or manipulation) proceed unchecked, exposing LP principal to direct loss. This is a **broken core pool functionality** impact: the extension guard that is supposed to be the last line of defense against value drain is silently removed.

---

### Likelihood Explanation

A malicious pool admin sets `timelock = type(uint32).max` (or any value > ~2.54 billion) at pool creation. LPs observe a displayed timelock of ~136 years and consider the pool safe. The admin immediately bypasses it. The attack requires only a single malicious actor with pool-admin rights — a role that is explicitly semi-trusted and whose boundary the timelock is designed to enforce.

---

### Recommendation

Add a maximum timelock cap in `initialize` and `proposeOracleStopLossTimelock`, and use `uint256` for the `executeAfter` computation and storage to eliminate the truncation entirely:

```solidity
uint256 internal constant MAX_TIMELOCK = 365 days; // or another safe bound

function _afterTimelock(address pool_) private view returns (uint256) {
    uint256 tl = oracleStopLossConfig[pool_].timelock;
    return block.timestamp + tl;
}

// In initialize:
require(timelock <= MAX_TIMELOCK, "timelock too large");

// In proposeOracleStopLossTimelock:
require(newTimelock <= MAX_TIMELOCK, "timelock too large");
```

Store `pendingTimelockExecuteAfter`, `pendingDrawdownExecuteAfter`, `pendingDecayExecuteAfter`, and `executeAfter` in `PendingHighWatermarks` as `uint256` (or `uint64`) rather than `uint32`.

### Proof of Concept

```
block.timestamp  = 1_750_000_000   (July 2025)
timelock         = 4_294_967_295   (type(uint32).max, ~136 years)

_afterTimelock:
  sum            = 1_750_000_000 + 4_294_967_295 = 6_044_967_295
  uint32(sum)    = 6_044_967_295 % 4_294_967_296 = 1_749_999_999   ← past timestamp

_requireElapsed(1_749_999_999):
  block.timestamp (1_750_000_000) < 1_749_999_999  → false → no revert

Attack sequence:
1. Pool admin deploys pool with timelock = type(uint32).max
2. LPs join, seeing a "136-year" timelock
3. Admin calls proposeOracleStopLossDrawdown(pool, 1_000_000 /*E6*/)
   → executeAfter = 1_749_999_999 (past)
4. Admin immediately calls executeOracleStopLossDrawdown(pool)
   → _requireElapsed passes → drawdownE6 set to E6
5. floorMultiplier = 0 → stop-loss disabled
6. Swaps that drain LP value now proceed without revert
```
