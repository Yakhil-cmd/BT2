### Title
Missing Sequencer-Only Access Control on `update_rewards` Allows Any Caller to Consume Per-Block Reward Slot and Permanently Freeze Consensus Rewards - (File: `src/staking/staking.cairo`)

### Summary

The `update_rewards` function in the `Staking` contract is documented as callable "Only starkware sequencer" but the implementation enforces no such restriction. Any unprivileged address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for the current block without distributing any rewards. If repeated every block, this permanently freezes all consensus-era staker and delegator yield.

### Finding Description

The protocol spec explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation of `update_rewards` calls only `general_prerequisites()`: [2](#0-1) 

`general_prerequisites` enforces only two things — the contract is not paused and the caller is not the zero address: [3](#0-2) 

There is no check that `get_caller_address()` equals the Starkware sequencer address. After passing those two trivial checks, the function unconditionally writes `last_reward_block = current_block_number` **before** the `disable_rewards` branch: [4](#0-3) 

Because `last_reward_block` is a single global counter (not per-staker), once it is set to the current block, any subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`: [5](#0-4) 

The analog to the original report is direct: just as an external minter pre-empting a sequential token ID permanently bricks the sale, an external caller pre-empting `last_reward_block` with `disable_rewards: true` permanently bricks per-block reward distribution for that block. Repeated every block, it freezes all consensus rewards indefinitely.

### Impact Explanation

- **Permanent freezing of unclaimed yield (High)**: An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block prevents the sequencer from ever distributing consensus-era block rewards. Stakers and delegators accumulate zero `unclaimed_rewards_own` / pool rewards for the entire duration of the attack. Because `last_reward_block` is global, a single attacker transaction per block is sufficient to suppress rewards for all stakers simultaneously.

### Likelihood Explanation

- Any non-zero address can call `update_rewards`; no stake, role, or special permission is required.
- The attacker only needs to know one valid, active staker address, which is trivially obtained from on-chain `NewStaker` events.
- The cost is one cheap Starknet transaction per block.
- The attack is self-sustaining: the attacker loses nothing and can run it indefinitely.

### Recommendation

Add an explicit sequencer-only guard at the top of `update_rewards`, consistent with the spec. For example, store the sequencer address in contract storage during construction and assert it at the start of the function:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
}
```

Alternatively, if the sequencer address is not known at deploy time, use a role-based check (e.g., `self.roles.only_sequencer()`).

### Proof of Concept

```cairo
// Attacker (any address) calls this every block before the sequencer does:
staking_rewards_dispatcher.update_rewards(
    staker_address: any_valid_staker,
    disable_rewards: true,   // consumes the slot, distributes nothing
);

// Sequencer's legitimate call in the same block now reverts:
// "REWARDS_ALREADY_UPDATED" because current_block_number == last_reward_block
staking_rewards_dispatcher.update_rewards(
    staker_address: any_valid_staker,
    disable_rewards: false,  // <-- REVERTS
);
```
