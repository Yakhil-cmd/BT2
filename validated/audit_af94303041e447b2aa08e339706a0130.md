### Title
`PresignOutput` Not Consumed on Rerandomization Enables Presignature Reuse and Secret Key Extraction — (`File: src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both the robust ECDSA and OT-based ECDSA schemes expose `rerandomize_presign()` as a function that takes `&PresignOutput` (a shared reference), leaving the original presignature live and reusable. Because `PresignOutput` also derives `Clone` and `Serialize/Deserialize`, a malicious coordinator can trivially initiate two or more signing sessions against the same presignature with different `(h, ε)` pairs. The resulting signatures carry multiplicatively related nonces, enabling full secret-key extraction via a standard ECDSA nonce-reuse attack — a split-view attack explicitly described in the library's own security documentation.

---

### Finding Description

`RerandomizedPresignOutput::rerandomize_presign()` in both ECDSA variants accepts the presignature by shared reference:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,          // ← reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) [2](#0-1) 

The `PresignOutput` struct derives `Clone`, `Serialize`, and `Deserialize`, so it can be stored, copied, and passed to `rerandomize_presign()` an arbitrary number of times:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub c: Scalar, pub e: Scalar, ... }
``` [3](#0-2) [4](#0-3) 

The downstream `sign()` function does consume `RerandomizedPresignOutput` by value, but this is too late: the original `PresignOutput` is already still alive and can be rerandomized again for a second session. [5](#0-4) 

The library's own security documentation explicitly identifies the consequence:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [6](#0-5) 

The library enforces `N = 2t+1` to prevent *different subsets* from signing with the same presignature, and rejects `msg_hash == 0`: [7](#0-6) [8](#0-7) 

However, there is **no enforcement** preventing the *same* subset from signing twice with the same presignature and different `(h, ε)` values. The documented constraint "Never reuse a presignature" is advisory only; the type system does not enforce it.

---

### Impact Explanation

A malicious coordinator initiates two signing sessions against the same `PresignOutput` with distinct `(h₁, ε₁)` and `(h₂, ε₂)`. Each honest participant independently calls `rerandomize_presign(&presign_output, args_i)` — the reference signature means the presignature is not consumed. The coordinator collects both sets of linearized signature shares `{s_i^(1)}` and `{s_i^(2)}`. Because the underlying nonce `R` is the same in both sessions (only scaled by different `delta` values), the nonces are multiplicatively related (`R₂ = R₁ · (delta₂/delta₁)`), satisfying the precondition for the split-view key-extraction attack described in the docs. The aggregate secret key is recoverable.

This matches the allowed critical impact: **Extraction of aggregate secret material / private signing shares**.

---

### Likelihood Explanation

The coordinator is a designated participant in every signing session and is explicitly named in the security documentation as an entity that "must not be able to present different message hashes, tweaks, or participant lists to different signers." [9](#0-8) 

A malicious coordinator needs only to:
1. Retain the `PresignOutput` values from a completed presigning round (they participated and hold their own share; they can also store the serialized outputs of others if the application persists them).
2. Initiate a first signing session with `(h₁, ε₁)` and collect all `s_i^(1)`.
3. Initiate a second signing session with the same presignature and `(h₂, ε₂)` — the library imposes no barrier.
4. Apply the nonce-reuse attack offline.

No external compromise, leaked keys, or cryptographic breaks are required. The attack is entirely within the library's documented threat model.

---

### Recommendation

Change `rerandomize_presign()` to consume `PresignOutput` by value in both ECDSA variants:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // ← consume, not borrow
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

This enforces one-time use at the type level: once a presignature has been rerandomized, the original value is moved and cannot be rerandomized again. Callers who need to store presignatures before use should serialize them; the `Serialize/Deserialize` derives remain appropriate for that purpose. The `Clone` derive on `PresignOutput` should also be removed or gated behind a clearly-named `#[cfg(test)]` or `dangerous_clone()` method to prevent accidental duplication in production paths.

The same fix applies to `src/ecdsa/ot_based_ecdsa/mod.rs`.

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
