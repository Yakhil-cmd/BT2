Audit Report

## Title
Silent Integer Truncation of `length` Parameter in ICRC-1 Ledger `get_blocks`/`get_transactions` on wasm32 - (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary
Both `get_transactions` and `get_blocks` query endpoints in the ICRC-1 ledger cast the caller-supplied `length` parameter from `u64` to `usize` using an unchecked `as` cast. On the wasm32 target used by all IC canisters, `usize` is 32 bits, so any `length` value that is a multiple of `2^32` (most critically `(u32::MAX as u64) + 1 = 4_294_967_296`) silently truncates to `0`. The downstream `block_locations` call then returns an empty range and empty archive pointers, causing the endpoint to return zero blocks and zero archive callbacks with no error — silently hiding ledger history from the caller.

## Finding Description
In `rs/ledger_suite/icrc1/ledger/src/main.rs` lines 793–797 and 802–806, both handlers call `req.as_start_and_length()` and immediately cast the returned `u64` length to `usize` with an unchecked `as`:

```rust
// line 797
Access::with_ledger(|ledger| ledger.get_transactions(start, length as usize))
// line 806
Access::with_ledger(|ledger| ledger.get_blocks(start, length as usize))
```

`as_start_and_length()` in `packages/icrc-ledger-types/src/icrc3/blocks.rs` (lines 64–83) only caps `length` at `u64::MAX` via `to_u64()` — it performs no wasm32-`usize` guard. On wasm32, `u64 as usize` is equivalent to `u64 as u32`; the upper 32 bits are discarded. For `length = 4_294_967_296`, the result is `0`. The call chain then reaches `block_locations(self, start, 0)` in `rs/ledger_suite/common/ledger_canister_core/src/ledger.rs` (line 509), which constructs an empty requested range, yielding empty `local_blocks` and empty `archived_blocks`. The response is structurally valid but contains zero blocks and zero archive pointers.

By contrast, the ICP ledger explicitly guards this in `rs/ledger_suite/icp/ledger/src/main.rs` line 1065:
```rust
let locations = block_locations(&*ledger, start, length.min(usize::MAX as u64) as usize);
```
The ICP ledger also has a dedicated regression test at `rs/ledger_suite/icp/ledger/tests/tests.rs` lines 1884–1886 with the comment: *"If this is cast (in a wasm32 ledger) using `as usize`, it will overflow to 0u32."*

The `icrc3_get_blocks` path in the same ICRC-1 ledger library is correctly guarded at `rs/ledger_suite/icrc1/ledger/src/lib.rs` line 1120:
```rust
let length = max_length.min(length).min(usize::MAX as u64) as usize;
```
The `get_blocks` and `get_transactions` paths were not updated when this fix was applied to `icrc3_get_blocks`.

## Impact Explanation
The ICRC-1 ledger is deployed on mainnet for ckBTC, ckETH, ckERC20, and SNS tokens — all in-scope financial infrastructure. Any caller can cause these endpoints to return a structurally valid but semantically empty response (zero blocks, zero archive pointers). Downstream indexers, wallets, and chain-fusion bridges relying on these endpoints to reconstruct ledger history will observe a complete absence of blocks, potentially causing missed token transfers in off-chain accounting, incorrect balance reconstruction, and denial of historical data availability. This constitutes a significant ck-token and ledger security impact with concrete protocol harm, matching the High ($2,000–$10,000) impact tier.

## Likelihood Explanation
The `length` field is a `Nat` (arbitrary-precision integer) in the Candid interface, so any anonymous query caller can supply a value exceeding `u32::MAX` with no special privilege. The trigger value `(u32::MAX as u64) + 1` is not a value normal clients would use, but the DFINITY platform team's own regression test for the ICP ledger demonstrates the exact truncation scenario, confirming awareness of the wasm32 hazard. The ICRC-1 ledger simply lacks the equivalent fix. Likelihood is low-to-medium: not triggered by typical clients, but trivially triggerable by any caller who crafts the specific value, and the ICP team's own test confirms the platform considers this a real risk.

## Recommendation
Apply the same guard used in the ICP ledger and in `icrc3_get_blocks` to both affected functions in `rs/ledger_suite/icrc1/ledger/src/main.rs`:

```rust
fn get_transactions(req: GetTransactionsRequest) -> GetTransactionsResponse {
    let (start, length) = req
        .as_start_and_length()
        .unwrap_or_else(|msg| ic_cdk::api::trap(&msg));
    Access::with_ledger(|ledger| {
        ledger.get_transactions(start, length.min(usize::MAX as u64) as usize)
    })
}

fn get_blocks(req: GetBlocksRequest) -> GetBlocksResponse {
    let (start, length) = req
        .as_start_and_length()
        .unwrap_or_else(|msg| ic_cdk::api::trap(&msg));
    Access::with_ledger(|ledger| {
        ledger.get_blocks(start, length.min(usize::MAX as u64) as usize)
    })
}
```

Alternatively, move the `usize::MAX` cap into `as_start_and_length()` itself so all callers benefit automatically.

## Proof of Concept
Mirror the ICP ledger's existing regression test structure (`rs/ledger_suite/icp/ledger/tests/tests.rs` lines 1878–1900) for the ICRC-1 ledger:

1. Install the ICRC-1 ledger canister in a `StateMachine` or PocketIC environment.
2. Seed several transactions so the ledger contains blocks.
3. Call `get_blocks` with `start = Nat::from(0u64)` and `length = Nat::from((u32::MAX as u64) + 1)`.
4. On the unfixed wasm32 canister, `length as usize` evaluates to `0`, `block_locations` returns an empty range, and `res.blocks` is empty despite the ledger containing blocks.
5. Assert `!res.blocks.is_empty()` — this assertion fails on the unfixed canister, confirming the truncation.
