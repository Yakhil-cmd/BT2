### Title
Signature Verification Skipped for Invoke Transactions When Account Has Pending Deploy-Account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator returns `true` whenever an incoming invoke transaction has `nonce == 1` and the sender's on-chain nonce is `0` and the mempool reports any transaction for that address. When this flag is `true`, `run_validate_entry_point` sets `validate: false`, completely bypassing the `__validate__` entry-point call that performs cryptographic signature verification. An unprivileged attacker can exploit this to inject an invoke transaction bearing an arbitrary (or empty) signature for any victim account that has a pending `deploy_account` transaction in the mempool.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

Condition 3 is satisfied as soon as the victim submits their own `deploy_account` transaction, because `account_tx_in_pool_or_recent_block` checks only whether any transaction for that address exists in the pool or recent commit history: [2](#0-1) 

**Signature verification is then unconditionally skipped**

When `skip_validate == true`, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `validate` flag gates the entire `__validate__` call: [4](#0-3) 

The stateless validator only checks signature *length*, not cryptographic validity: [5](#0-4) 

Therefore, an invoke transaction with a completely fabricated signature passes all gateway checks and is admitted to the mempool.

---

### Impact Explanation

**Mempool admission of unauthorized transactions (High)**

Any attacker can submit an invoke transaction for a victim's address (with arbitrary calldata and an invalid/empty signature) the moment the victim's `deploy_account` appears in the mempool. The transaction passes all gateway checks and is stored in the mempool.

**Fee drain from victim's account (Critical)**

During block execution, `new_for_sequencing` always sets `validate: true`: [6](#0-5) 

The `__validate__` entry point is therefore called at sequencing time. With an invalid signature it fails, the transaction reverts, but Starknet's fee-charging semantics still deduct fees from the victim's account (the `sender_address`). An attacker can submit many such transactions, draining the victim's balance without ever holding the victim's private key.

**Nonce slot occupation**

The attacker's invoke occupies nonce slot 1 in the mempool. If the victim also submits a legitimate invoke with nonce 1, the mempool may accept both (no duplicate-nonce rejection in `validate_incoming_tx`), leading to ordering ambiguity or displacement of the legitimate transaction. [7](#0-6) 

---

### Likelihood Explanation

The precondition is simply that the victim has submitted a `deploy_account` transaction that is visible in the mempool. This is a normal, public, observable event. No privileged access, leaked keys, or special network position is required. The attacker only needs to know the victim's contract address (deterministic and publicly derivable) and submit a crafted invoke before the deploy_account is committed to a block.

---

### Recommendation

1. **Verify the signature even when skipping the `__validate__` entry point.** The UX skip is intended to avoid a failed `__validate__` call because the account contract does not exist yet — not to skip cryptographic signature verification entirely. A lightweight off-chain ECDSA check against the transaction hash should be performed before admission regardless of the `skip_validate` flag.

2. **Restrict `account_tx_in_pool_or_recent_block` to deploy-account transactions only.** The current check returns `true` for any transaction type, which is broader than the stated intent ("a deploy_account transaction exists for this account").

3. **Alternatively, reject invoke transactions for undeployed accounts at the gateway level** and require the user to submit the deploy_account and invoke in a single atomic bundle, verifying the bundle's signature before admission.

---

### Proof of Concept

```
Precondition:
  victim_address = deterministic_address(class_hash, salt, constructor_calldata)

Step 1 — Victim submits deploy_account (nonce=0):
  POST /add_transaction
  { type: DEPLOY_ACCOUNT, sender: victim_address, nonce: 0, signature: <valid> }
  → Admitted to mempool.
  → mempool.account_tx_in_pool_or_recent_block(victim_address) == true

Step 2 — Attacker submits invoke (nonce=1, invalid signature):
  POST /add_transaction
  { type: INVOKE, sender: victim_address, nonce: 1,
    calldata: [drain_all_funds_selector, ...],
    signature: [0x0, 0x0]   ← completely invalid }

Gateway path:
  stateless_validator.validate()  → passes (signature length ≤ max_signature_length)
  stateful_validator.extract_state_nonce_and_run_validations():
    account_nonce = get_nonce_from_state(victim_address) = 0
    run_pre_validation_checks():
      validate_nonce()  → 0 ≤ 1 ≤ 200, passes
      validate_by_mempool() → passes
      skip_stateful_validations():
        nonce==1 ✓, account_nonce==0 ✓,
        account_tx_in_pool_or_recent_block(victim_address)==true ✓
        → returns true
    run_validate_entry_point(skip_validate=true):
      ExecutionFlags { validate: false }
      __validate__ is NOT called
      → returns Ok(())
  → Transaction admitted to mempool.

Step 3 — Block is produced:
  deploy_account (nonce=0) executed → victim's account deployed
  attacker's invoke (nonce=1) executed with validate=true:
    __validate__ called → signature [0x0, 0x0] is invalid → REVERT
    Fee charged to victim_address  ← victim loses funds
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
