### Title
Truncating Integer Division in `native_per_pubdata` Computation Silently Zeroes Pubdata Native Cost — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validation_impl.rs`, the per-pubdata-byte native resource rate is computed with a plain floor (`wrapping_div`) division. When `pubdata_price < native_price`, the result truncates to **0**. Every downstream pubdata-cost check then charges **zero** native resources for pubdata, allowing any L2 transaction to generate unlimited pubdata without paying the native resource cost that the protocol intends to collect.

---

### Finding Description

At line 142 of `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 999`, `native_price = 1000`), the result is **0**. The `ok_or` guard only rejects values that overflow a `u64`; it does not reject zero. [1](#0-0) 

By contrast, `native_per_gas` is computed with `div_ceil`, which guarantees a non-zero result whenever the numerator is non-zero:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The asymmetry is the root cause: `native_per_gas` rounds up; `native_per_pubdata` rounds down and can reach zero.

`native_per_pubdata` flows into three critical places:

**1. Intrinsic pubdata overhead in `create_resources_for_tx`:**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
```
When `native_per_pubdata = 0`, the native limit is not reduced for intrinsic pubdata at all. [3](#0-2) 

**2. Execution pubdata charging in `get_resources_to_charge_for_pubdata`:**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
```
When `native_per_pubdata = 0`, `native = 0` regardless of how many pubdata bytes were written. [4](#0-3) 

**3. Post-execution pubdata sufficiency check in `check_enough_resources_for_pubdata`:**
```rust
let enough = resources.has_enough(&resources_for_pubdata);
```
When `resources_for_pubdata` is zero native, `enough` is always `true`. [5](#0-4) 

---

### Impact Explanation

When `pubdata_price < native_price`, `native_per_pubdata` silently becomes 0. The entire native-resource accounting for pubdata is bypassed:

- The transaction's native limit is not reduced for intrinsic pubdata.
- No native resources are charged for any pubdata written during execution.
- The post-execution pubdata check always passes.

A transaction that writes many storage slots (each generating 32+ bytes of pubdata) pays only EVM gas for `SSTORE` operations, but pays **zero** native resources for the resulting pubdata. The protocol's double-resource accounting model — which is specifically designed to ensure pubdata costs are covered — is completely defeated for the pubdata dimension.

This is a **resource accounting bug** with a direct funds-loss path: the operator/sequencer bears the L1 data-availability cost of publishing the pubdata, but the transaction sender was never charged for it in native resources.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic. The documentation describes `native_price` as "the cost of processing a single RISC-V cycle" and `pubdata_price` as the cost of publishing one byte of data. These are independent operator-set parameters. Any block context where `pubdata_price` is set below `native_price` (e.g., `native_price = 1000`, `pubdata_price = 999`) triggers the bug. An unprivileged transaction sender only needs to observe the public block context and submit a storage-heavy transaction. [6](#0-5) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` (ceiling division) for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (floor division — can produce 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This ensures that even when `pubdata_price < native_price`, at least 1 native unit is charged per pubdata byte, preserving the invariant that pubdata is never free.

---

### Proof of Concept

1. Operator sets block context: `native_price = 1000`, `pubdata_price = 999`.
2. Attacker submits an L2 EIP-1559 transaction with `max_fee_per_gas = 1000`, `gas_limit = 10_000_000`, calling a contract that executes 100 `SSTORE` operations to fresh slots.
3. During validation, `native_per_pubdata = 999 / 1000 = 0` (integer floor division).
4. `intrinsic_pubdata_overhead = 0 * L2_TX_INTRINSIC_PUBDATA = 0` — native limit not reduced.
5. After execution, `get_resources_to_charge_for_pubdata` returns `(pubdata_bytes, 0_native)`.
6. `check_enough_resources_for_pubdata` returns `true` (0 native needed ≤ any remaining native).
7. `compute_gas_refund` charges 0 native for pubdata; `delta_gas` from pubdata is 0.
8. Transaction succeeds. ~3200 bytes of pubdata are published to L1 at operator cost; the sender paid only EVM gas for `SSTORE`. [1](#0-0) [7](#0-6) [8](#0-7)

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
```
