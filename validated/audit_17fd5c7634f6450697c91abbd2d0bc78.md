### Title
Bank Precompile `transfer` Method Allows Any Contract to Move `evm/<denom>` Tokens From Arbitrary Holders Without Authorization - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The bank precompile's `transfer` (and `burn`) methods accept an arbitrary `sender` address from call arguments and execute `bankKeeper.SendCoins` (or `SendCoinsFromAccountToModule`) on behalf of that address without verifying the caller is authorized to act for it. Any deployed EVM contract can transfer or burn `evm/<its_own_address>`-denominated native bank tokens from any holder without the holder's consent.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` directly from ABI-encoded call data and uses it as the `from` address in a native bank transfer:

```go
// bank.go L175-L192
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())          // denom = "evm/<caller_contract>"
amt := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is no check that `sender == contract.Caller()`. The denom is scoped to the calling contract (`evm/<caller_address>`), but the `from` address is fully attacker-controlled.

The same pattern exists in the shared `MintMethodName, BurnMethodName` case: for `burn`, `args[0]` (named `recipient` in the code) is the address whose tokens are burned, again with no check that it equals `contract.Caller()`:

```go
// bank.go L121-L148
recipient := args[0].(common.Address)   // address to burn FROM
addr := sdk.AccAddress(recipient.Bytes())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
``` [1](#0-0) [2](#0-1) 

The bank precompile is registered as a stateful precompile reachable by any EVM contract. [3](#0-2) 

### Impact Explanation
**Critical — Unauthorized transfer and burn of `evm/<contract_address>`-denominated native bank assets.**

A malicious contract deployed by an unprivileged attacker can:

1. **Theft via `transfer`**: Call `bank.transfer(victim, attacker, amount)` where `victim` holds `evm/<malicious_contract_address>` tokens. The precompile executes `bankKeeper.SendCoins(victim, attacker, ...)` with no authorization check, moving the victim's tokens to the attacker.

2. **Destruction via `burn`**: Call `bank.burn(victim, amount)` to destroy `evm/<malicious_contract_address>` tokens held by any address.

The denom is scoped to the calling contract, so the attacker must first cause victims to hold tokens of that denom (e.g., by minting them via `bank.mint`). Once victims hold those tokens, the attacker can drain or destroy them at will.

This directly matches the Critical impact category: *unauthorized transfer/burn of EVM-denominated assets*.

### Likelihood Explanation
An unprivileged attacker can deploy a contract, call `bank.mint(victim, amount)` to issue `evm/<attacker_contract>` tokens to victims (e.g., as part of a fake airdrop or DeFi protocol), and then call `bank.transfer(victim, attacker, amount)` to reclaim them. No privileged keys, governance access, or leaked secrets are required. The entire attack path is reachable through standard `MsgEthereumTx` transactions.

### Recommendation
Add a caller-authorization check in both the `transfer` and `burn` cases of the bank precompile's `Run` method:

- In `TransferMethodName`: require `sender == contract.Caller()` before executing `SendCoins`.
- In `BurnMethodName`: require `recipient == contract.Caller()` before executing `SendCoinsFromAccountToModule`.

Alternatively, implement an ERC20-style allowance mechanism so token holders can explicitly authorize a contract to act on their behalf. [4](#0-3) 

### Proof of Concept

1. Attacker deploys `MaliciousToken` contract at address `0xDEAD`.
2. `MaliciousToken` calls `bankPrecompile.mint(victim, 1000)` → victim now holds `1000 evm/0xDEAD` native bank tokens.
3. Attacker calls `MaliciousToken.steal(victim)` which internally calls `bankPrecompile.transfer(victim, attacker, 1000)`.
4. The precompile executes `bankKeeper.SendCoins(victim_addr, attacker_addr, [{denom:"evm/0xDEAD", amount:1000}])` — no authorization check fires.
5. Victim's `evm/0xDEAD` balance is now 0; attacker holds 1000.

The same flow with `bankPrecompile.burn(victim, 1000)` destroys the victim's balance instead.
