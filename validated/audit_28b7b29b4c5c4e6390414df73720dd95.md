### Title
ETH (`msg.value`) Permanently Lost in `tick()` When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary
The `tick()` function in the L1 `RewardSupplier` contract is `payable` and accepts ETH to cover L1→L2 messaging fees. However, the ETH is only consumed inside the `if (amountToMint > 0)` branch. If `tick()` is called when there are no pending mint requests (`amountToMint == 0`), the entire `msg.value` is silently accepted and permanently locked in the contract with no refund and no recovery path.

---

### Finding Description

`RewardSupplier.tick()` is declared `external payable`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... uses msg.value here ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // If amountToMint == 0: function returns, msg.value is never used or refunded.
}
``` [1](#0-0) 

When `amountToMint == 0`, the function exits immediately after the `if` block. There is no `else` branch, no `require(msg.value == 0)` guard, and no refund of `msg.value`. The ETH is accepted by the contract and becomes permanently inaccessible — the contract has no ETH withdrawal function.

The `requiredMinting()` view function can be called off-chain to check whether minting is needed, but the result can change between the off-chain check and the on-chain execution (e.g., another caller's `tick()` transaction processes the pending messages first). A caller who sends ETH based on a stale off-chain read will lose all of it. [2](#0-1) 

---

### Impact Explanation

Any ETH sent with a `tick()` call that lands when `amountToMint == 0` is permanently locked in the `RewardSupplier` contract. The contract inherits `ProxySupportImpl` and `RewardSupplierStorage` but exposes no ETH sweep or rescue function. The funds are irrecoverable by the sender. This constitutes **direct, permanent loss of user funds in motion** — matching the Critical impact tier: *"Direct theft of user funds, whether at-rest or in-motion."* [3](#0-2) 

---

### Likelihood Explanation

Likelihood is **Low**. The scenario requires a caller to send non-zero ETH to `tick()` in a transaction that executes when no mint messages are pending. This can happen realistically via a race condition: a caller reads `requiredMinting()` off-chain, sees pending messages, prepares a transaction with ETH, but a competing `tick()` call is mined first and drains the pending queue. The original caller's transaction then executes with `amountToMint == 0` and loses all ETH. No privileged access is required — `tick()` is open to any address.

---

### Recommendation

Add a guard at the top of `tick()` that reverts if `msg.value > 0` and no minting will occur, or unconditionally refund any unused ETH after the `if` block:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else {
        require(msg.value == 0, "NO_MINT_REQUIRED_NO_FEE_NEEDED");
    }
}
```

Alternatively (stricter): require `msg.value == 0` when `amountToMint == 0` before entering the function body, mirroring the recommendation from the original report to enforce exact payment.

---

### Proof of Concept

1. L2 `RewardSupplier` sends one mint-request message to L1 (pending in the messaging contract).
2. Caller A calls `requiredMinting()` off-chain → sees `amountToMint = TOKENS_PER_MINT_REQUEST`, `numMsgsToConsume = 1`.
3. Caller A submits `tick{value: 0.01 ether}()`.
4. Before Caller A's tx is mined, Caller B submits `tick{value: 0.01 ether}()` with higher gas and gets mined first, consuming the pending message.
5. Caller A's tx is now mined: `requiredMinting()` returns `(0, 0)`, the `if` branch is skipped, and Caller A's `0.01 ether` is permanently locked in the contract. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
