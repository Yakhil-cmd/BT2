### Title
Missing Error-Callback in `ExitToEthereum` Precompile Causes Permanent Fund Freeze When eth-connector `withdraw` Is Paused/Caged - (File: engine-precompiles/src/native.rs)

---

### Summary

The `ExitToEthereum` precompile schedules a `withdraw` promise on the eth-connector as a fire-and-forget `PromiseArgs::Create` with **no error callback**. If the eth-connector's `withdraw` is paused (analogous to DSS `cage`), the user's ETH or burned ERC-20 tokens are permanently locked with no recovery path. The `ExitToNear` precompile has an explicit error-refund callback (`exit_to_near_precompile_callback`); `ExitToEthereum` does not.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run` function constructs a `withdraw` promise targeting the eth-connector and wraps it as `PromiseArgs::Create`:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

`PromiseArgs::Create` means there is **no callback** registered for the failure case. Compare this to `ExitToNear`, which wraps its promise in `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`:

```rust
PromiseArgs::Callback(PromiseWithCallbackArgs {
    base: transfer_promise,
    callback: PromiseCreateArgs {
        target_account_id: self.current_account_id.clone(),
        method: "exit_to_near_precompile_callback".to_string(),
        ...
    },
})
``` [2](#0-1) 

That callback explicitly handles the failure branch by calling `engine::refund_on_error`, which either re-mints burned ERC-20 tokens or transfers ETH back from the precompile address to the user: [3](#0-2) [4](#0-3) 

`ExitToEthereum` has no equivalent. When the `withdraw` promise fails:

- **ETH exit (flag `0x00`)**: The user's ETH was already deducted from their EVM balance and transferred to `exit_to_ethereum::ADDRESS` (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`). It is permanently stranded there.
- **ERC-20 exit (flag `0x01`)**: The ERC-20 tokens were already burned by the calling contract before the precompile ran. There is no re-mint path. [5](#0-4) 

The eth-connector exposes a `PAUSE_WITHDRAW` admin flag (bit 1 of `PausedMask`) that can pause its `withdraw` method: [6](#0-5) 

When `withdraw` is paused on the eth-connector, every `ExitToEthereum` call silently fails its promise, and all funds sent through that path are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ETH or ERC-20 tokens routed through `ExitToEthereum` while the eth-connector's `withdraw` is paused are irrecoverably lost. ETH accumulates in the `exit_to_ethereum::ADDRESS` precompile address (which has no withdrawal interface), and ERC-20 tokens are burned with no remint. Unlike the DSS cage scenario (which merely freezes funds), this results in **permanent destruction of user assets**.

---

### Likelihood Explanation

The eth-connector owner can legitimately pause `withdraw` as an emergency measure (the `PAUSE_WITHDRAW` flag exists precisely for this purpose, as shown in `test_withdraw_from_near_pausability`). This is a realistic operational event — not a key compromise — analogous to Maker admins calling `cage`. Any user who calls `ExitToEthereum` during the pause window permanently loses their funds. The window need not be long; a single block is sufficient. [7](#0-6) 

---

### Recommendation

Add an error-refund callback to `ExitToEthereum` mirroring the `ExitToNear` pattern:

1. Encode a `PromiseArgs::Callback` instead of `PromiseArgs::Create`, with a new `exit_to_ethereum_precompile_callback` method as the error handler.
2. In that callback, if the promise result is not `Successful`, call `engine::refund_on_error` with the appropriate `RefundCallArgs` (ETH transfer back from `exit_to_ethereum::ADDRESS`, or ERC-20 re-mint).
3. Ensure the `error_refund` feature gate (already used in `ExitToNear`) is consistently applied or removed so both exit paths have equivalent safety guarantees. [8](#0-7) 

---

### Proof of Concept

1. Owner calls `pa_pause_feature` on the eth-connector with key `"withdraw"` (sets `PAUSE_WITHDRAW` bit).
2. User submits an EVM transaction calling the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with `value = X ETH`.
3. EVM execution succeeds: X ETH is deducted from the user's balance and credited to `exit_to_ethereum::ADDRESS`.
4. The engine's `filter_promises_from_logs` schedules the `PromiseArgs::Create(withdraw_promise)` — no callback registered.
5. NEAR runtime executes the `withdraw` call on the eth-connector; it panics/reverts because `PAUSE_WITHDRAW` is set.
6. No callback fires. X ETH remains in `exit_to_ethereum::ADDRESS` with no extraction interface.
7. Even after the owner unpauses `withdraw`, the stranded ETH cannot be recovered — there is no retry or sweep mechanism. [9](#0-8) [10](#0-9)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
