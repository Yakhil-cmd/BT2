### Title
Participant Set in `RerandomizationArguments` Not Enforced to Match Participant Set in `sign()` — (`File: src/ecdsa/ot_based_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The signing participant set is specified in two independent locations — inside `RerandomizationArguments` (used to derive the rerandomization scalar `delta`) and as a separate argument to `sign()` (used for Lagrange interpolation). There is no protocol-level check that these two participant sets are identical. A malicious coordinator can present different participant sets to different honest signers, causing each signer to compute Lagrange coefficients over a different set, producing inconsistent signature shares. This is the split-view attack explicitly documented as catastrophic in `docs/ecdsa/robust_ecdsa/signing.md`, and it is not prevented by any in-library enforcement.

---

### Finding Description

`RerandomizationArguments` contains a `participants: ParticipantList` field that is hashed into the HKDF derivation of the rerandomization scalar `delta`:

```rust
// src/ecdsa/mod.rs:157-159
for participant in self.participants.participants() {
    concatenation.extend_from_slice(&participant.bytes());
}
``` [1](#0-0) 

This `delta` is then used in `rerandomize_presign()` to transform each participant's presignature shares. The resulting `RerandomizedPresignOutput` does **not** store which participant set was used. [2](#0-1) 

When `sign()` is called, a fresh `participants` argument is accepted and used directly for Lagrange interpolation in `compute_signature_share()`:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs:148-153
let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
let k_i = lambda * presignature.k;
let sigma_i = lambda * presignature.sigma;
``` [3](#0-2) 

The `sign()` initialization only checks that `participants.len() >= threshold` — it never verifies that the participant set matches the one embedded in `RerandomizationArguments`: [4](#0-3) 

The same structural gap exists in the robust ECDSA variant. The `sign()` function there enforces `participants.len() == 2*max_malicious+1` but still does not cross-check against `RerandomizationArguments.participants`: [5](#0-4) 

The library's own documentation explicitly identifies this as a required invariant that the library does not enforce:

> "Ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers." [6](#0-5) 

---

### Impact Explanation

A malicious coordinator orchestrates the signing session. It can:

1. Construct `RerandomizationArguments` with participant set **P₁** and distribute rerandomized presignatures to all signers.
2. Call `sign()` on signer A with participant set **P₁** and on signer B with participant set **P₂** (differing by one member).
3. Each signer computes Lagrange coefficients over a different set, producing signature shares `s_A` and `s_B` that are algebraically inconsistent — they are shares of different linear combinations of the secret nonce.
4. The coordinator collects both shares. As documented in `docs/ecdsa/robust_ecdsa/signing.md` lines 151–158, combining signature shares from overlapping but distinct signing sets over the same presignature nonce enables standard ECDSA nonce-reuse key extraction.

This maps to **Critical** impact: extraction of the aggregate secret signing key material via a split-view attack enabled by the missing participant-set consistency check. [7](#0-6) 

---

### Likelihood Explanation

The coordinator role is a documented participant in the protocol. A malicious coordinator is an explicit threat model entry in the library's security scope. No external compromise is required — the coordinator simply passes different `participants` slices to different signers' `sign()` calls. The library provides no mechanism to detect or prevent this. The attack requires only two signing sessions over the same presignature nonce, which is a realistic scenario.

---

### Recommendation

The `RerandomizedPresignOutput` should store the participant set (or a binding commitment to it) that was used during rerandomization. The `sign()` function should verify that the `participants` argument matches the stored set before proceeding. Concretely:

- Add a `participants_hash: [u8; 32]` field to `RerandomizedPresignOutput` (both OT-based and robust variants), populated during `rerandomize_presign()` by hashing the `RerandomizationArguments.participants`.
- At the start of `sign()`, recompute the hash of the provided `participants` and assert it equals `presignature.participants_hash`, returning `InitializationError::BadParameters` on mismatch.

This mirrors the fix recommended in the original Atlas report: add an on-protocol check at the entry point of the signing function to ensure the two independently-specified values are consistent.

---

### Proof of Concept

**Step 1**: Coordinator constructs `RerandomizationArguments` with participants `[A, B, C]` and rerandomizes presignatures for all three. [8](#0-7) 

**Step 2**: Coordinator calls `sign()` for participant A with `participants = [A, B, C]` and for participant B with `participants = [A, B, D]`. Both calls pass the size check (`participants.len() >= threshold`). [9](#0-8) 

**Step 3**: Participant A computes `lambda_A` over `{A,B,C}`; participant B computes `lambda_B` over `{A,B,D}`. These are different scalars applied to the same presignature nonce share, producing shares of different linear combinations of the secret nonce `k`. [10](#0-9) 

**Step 4**: The coordinator collects both shares. Because both shares are derived from the same underlying nonce `k` but with different Lagrange weights, the coordinator now holds two equations in the secret key `x` and nonce `k`, enabling key extraction via standard ECDSA nonce-reuse algebra — exactly the attack described in the library's own documentation. [7](#0-6)

    coordinator: Participant,

    msg_hash: Scalar,

    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });

    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,

    // Round 1
    // Linearize ki
    // Spec 1.1

    // Compute si = h * ki + Rx * sigmai
