### Title
Unchecked Subtraction Underflow in Gas Refund Calculation — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` performs an unchecked subtraction `gas_limit - gas_used` at line 83. The post-hoc `require_internal!` guard at lines 85–89 checks the *result* of the already-executed subtraction rather than validating the operands beforehand. When `gas_used` exceeds `gas_limit` — which is reachable for L2 transactions when the EIP-7623 `minimal_gas_used` floor exceeds the transaction's gas limit — the subtraction underflows in release mode (wrapping to a huge value), and the guard returns an `InternalError` instead of a proper validation error.

---

### Finding Description

In `refund_calculation.rs`, `gas_used` is built up through several steps:

1. **Initial assignment** (line 31–33): `gas_used = gas_limit - remaining_ergs/ERGS_PER_GAS` — safe, `gas_used ≤ gas_limit`.
2. **EVM refund subtraction** (line 48): `gas_used -= evm_refund` where `evm_refund ≤ gas_used/5` — safe.
3. **EIP-7623 floor enforcement** (line 56): `gas_used = max(gas_used, minimal_gas_used)` — **unsafe if `minimal_gas_used > gas_limit`**.
4. **Native adjustment** (lines 69–79): can only raise `gas_used` up to `gas_limit` (since `native_used ≤ gas_limit * native_per_gas`), so this step alone cannot push `gas_used` past `gas_limit`.

After step 3, if `minimal_gas_used > gas_limit`, then `gas_used > gas_limit`. The unchecked subtraction at line 83 then underflows:

```rust
// refund_calculation.rs line 83
let total_gas_refund = gas_limit - gas_used;   // ← unchecked; wraps in release mode
```

The guard that follows:

```rust
// lines 85–89
require_internal!(
    total_gas_refund <= gas_limit,
    "Gas refund greater than gas limit",
    system
)?;
```

checks the *wrapped result*, not the original `gas_used ≤ gas_limit` invariant. In release mode the wrapped value is `u64::MAX - (gas_used - gas_limit) + 1`, which is far larger than `gas_limit`, so the guard fires and returns an `InternalError`. In debug mode the subtraction panics before the guard is reached.

**Root cause — missing cap on `minimal_gas_used` for L2 transactions:**

In `zk/validation_impl.rs` line 98, `minimal_gas_used` is computed from `compute_calldata_tokens`:

```rust
let (calldata_tokens, minimal_gas_used) = compute_calldata_tokens(system, calldata);
```

The only guard against `minimal_gas_used > tx_gas_limit` is:

```rust
// lines 99–104
#[cfg(feature = "eip_7623")]
require!(
    minimal_gas_used <= tx_gas_limit,
    InvalidTransaction::EIP7623IntrinsicGasIsTooLow,
    system
)?;
```

This check is **feature-gated** and absent in production builds without `eip_7623`. `minimal_gas_used` is then stored in the transaction context and forwarded to `compute_gas_refund` uncapped.

By contrast, L1 transactions are explicitly protected at `process_l1_transaction.rs` line 537:

```rust
let minimal_gas_used = minimal_gas_used.min(gas_limit);
```

No equivalent cap exists for L2 transactions.

**Concrete reachability condition:**

`minimal_gas_used = calldata_tokens × TOTAL_COST_FLOOR_PER_TOKEN` (= `calldata_tokens × 10`).  
`intrinsic_gas = TX_INTRINSIC_GAS + calldata_tokens × CALLDATA_TOKEN_GAS_COST` (= `21_000 + calldata_tokens × 4`).

For `minimal_gas_used > gas_limit` while `intrinsic_gas ≤ gas_limit` (so the transaction passes `create_resources_for_tx`):

```
calldata_tokens × 10 > gas_limit
21_000 + calldata_tokens × 4 ≤ gas_limit
```

Solving: `gas_limit > 35_000`. Example: `gas_limit = 50_000`, `calldata_tokens = 5_001` (≈1250 non-zero bytes):

- `minimal_gas_used = 50_010 > 50_000` ✓  
- `intrinsic_gas = 41_004 ≤ 50_000` ✓ (passes validation)

---

### Impact Explanation

An attacker submits an L2 transaction with a `gas_limit` slightly above 35 000 and enough non-zero calldata bytes to satisfy the condition above. The transaction passes all validation checks in `create_resources_for_tx` (because `intrinsic_gas ≤ gas_limit`), executes its body, and then fails with an `InternalError` during refund calculation. This `InternalError` propagates as a `BootloaderSubsystemError`, which is a fundamentally different error class from a validation error. Depending on bootloader error-handling policy, this can:

- Cause the block to be treated as invalid (halting the chain in the worst case).
- Produce a divergence between forward execution and proving, since the prover may handle `InternalError` differently from a normal transaction revert.
- In debug/test builds, cause an outright panic, crashing the node.

---

### Likelihood Explanation

The condition is trivially constructable by any unprivileged L2 transaction sender: choose any `gas_limit` between ~36 000 and the block gas limit, and supply enough non-zero calldata bytes. No special privileges, leaked keys, or oracle manipulation are required. The `eip_7623` feature flag is absent in production, leaving the path permanently open.

---

### Recommendation

1. **Replace the unchecked subtraction** with a checked variant and validate *before* subtracting:

```rust
// refund_calculation.rs
if gas_used > gas_limit {
    return Err(internal_error!("gas_used exceeds gas_limit"));
}
let total_gas_refund = gas_limit - gas_used; // now safe
```

2. **Cap `minimal_gas_used` unconditionally** for L2 transactions in `zk/validation_impl.rs`, mirroring the L1 treatment in `process_l1_transaction.rs` line 537:

```rust
let minimal_gas_used = minimal_gas_used.min(tx_gas_limit);
```

   or, if EIP-7623 semantics require rejection, move the `require!` check outside the feature gate.

3. **Remove the misleading post-hoc guard** (`require_internal!(total_gas_refund <= gas_limit, ...)`) or replace it with the pre-subtraction check above.

---

### Proof of Concept

**Entry path:**

1. Attacker submits an RLP-encoded EIP-1559 L2 transaction with `gas_limit = 50_000` and 1 251 non-zero calldata bytes (≈ 5 004 tokens).
2. `validate_and_compute_fee_for_transaction` → `compute_calldata_tokens` returns `minimal_gas_used = 50_040`.
3. `create_resources_for_tx` computes `intrinsic_gas = 21_000 + 5_004 × 4 = 41_016 ≤ 50_000` → **passes**.
4. Transaction body executes normally.
5. `compute_gas_refund` is called with `minimal_gas_used = 50_040`, `gas_limit = 50_000`.
6. After `max(gas_used, 50_040)`, `gas_used = 50_040 > 50_000 = gas_limit`.
7. Line 83: `let total_gas_refund = 50_000u64 - 50_040u64` → **wraps to `u64::MAX - 39`** in release mode / **panics** in debug mode.
8. `require_internal!` fires → `InternalError` propagates as `BootloaderSubsystemError`.
