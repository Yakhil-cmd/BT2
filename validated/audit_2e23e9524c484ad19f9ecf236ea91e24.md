### Title
Gateway L2 Gas Price Threshold Validated Against Current Block Price Instead of Next Block Price, Causing Incorrect Admission Decisions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful `validate_resource_bounds` check compares a transaction's `max_price_per_unit` against a threshold derived from the **current (latest committed) block's** L2 gas price. However, the block the transaction will actually be included in uses the **next block's** L2 gas price, which is computed by the EIP-1559-style fee market and stored as `next_l2_gas_price` in the block header. A developer TODO comment in the code explicitly acknowledges this discrepancy. The mismatch causes the gateway to admit transactions that will fail `check_fee_bounds` in the batcher, and to reject transactions that would succeed in the batcher — both matching the "High: admission accepts invalid / rejects valid transactions" impact scope.

---

In `validate_resource_bounds`, the gateway reads the L2 gas price from the latest committed block:

    executable_tx.resource_bounds(),

``` [1](#0-0) 

This price is then used to compute an admission threshold:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...);
}
``` [2](#0-1) 

The **correct** reference price is `next_l2_gas_price`, which is computed by the fee market algorithm after each block and stored in the block header:

```rust
next_l2_gas_price: self.l2_gas_price,  // set in update_state_sync_with_new_block
``` [3](#0-2) 

The fee market computes this via `calculate_next_l2_gas_price_for_fin`, which applies EIP-1559 logic:

```rust
pub fn calculate_next_base_gas_price(price, gas_used, gas_target, min_gas_price) -> GasPrice {
    let price_change = (price_u256 * gas_delta) / denominator;
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change }
}
``` [4](#0-3) 

The `GatewayFixedBlockSyncStateClient` constructs a `BlockInfo` from the block header but only copies `l2_gas_price` (the current block's price), not `next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
    ...
},
``` [5](#0-4) 

So `next_l2_gas_price` is never surfaced to the gateway, and the threshold is always computed from the stale current-block price.

Meanwhile, the batcher builds the block context from `ProposalInit.l2_gas_price_fri`, which is set to `self.l2_gas_price` — the **already-updated** next-block price computed by `update_l2_gas_price`:

```rust
fn update_l2_gas_price(&mut self, height: BlockNumber, l2_gas_used: GasAmount) {
    self.l2_gas_price = self.calculate_next_l2_gas_price(height, l2_gas_used);
}
``` [6](#0-5) 

The batcher then enforces this price in `check_fee_bounds`:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [7](#0-6)

### Impact Explanation

Two concrete failure modes arise from the price mismatch:

**Mode A — Invalid transactions admitted (gas price rising):**
When the network is congested, `next_l2_gas_price > current_l2_gas_price`. A transaction with `max_price_per_unit` above the gateway threshold (derived from the lower current price) but below `next_l2_gas_price` passes gateway admission and enters the mempool. When the batcher executes it, `check_fee_bounds` rejects it because the transaction's offered price is below the block's actual price. The transaction occupies a mempool slot and wastes batcher execution cycles.

**Mode B — Valid transactions rejected (gas price falling, high `min_gas_price_percentage`):**
When load drops, `next_l2_gas_price < current_l2_gas_price`. The threshold is computed from the higher current price. A transaction with `max_price_per_unit` above `next_l2_gas_price` but below the threshold is rejected by the gateway even though it would succeed in the batcher. Users are denied service for economically valid transactions.

Both modes match: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The EIP-1559 fee market adjusts the L2 gas price every block based on gas usage relative to the target. Under sustained high load (full blocks), the price increases by approximately `1/gas_price_max_change_denominator` per block. Under low load, it decreases symmetrically. The `min_gas_price_percentage` config provides a buffer, but:

- The buffer is applied to the **wrong** (current) price, not the next price.
- During rapid price transitions (e.g., a sudden traffic spike), the divergence between current and next price can exceed the buffer within a few blocks.
- The condition is unprivileged: any user submitting a transaction with a price in the gap between the two thresholds triggers it.

The TODO comment `// TODO(Arni): getnext_l2_gas_price from the block header.` is a first-party acknowledgment that the current implementation is incorrect.

---

### Recommendation

1. Extend `GatewayFixedBlockStateReader` to expose `next_l2_gas_price` from the block header (it is already stored in `BlockHeaderWithoutHash.next_l2_gas_price`).
2. In `validate_resource_bounds`, replace `previous_block_l2_gas_price` with the `next_l2_gas_price` read from the latest committed block header.
3. Update `GatewayFixedBlockSyncStateClient.get_block_info_from_sync_client` to include `next_l2_gas_price` in the returned data, or add a dedicated `get_next_l2_gas_price` method to the trait.

### Proof of Concept

**Setup:**
- Latest committed block: `block_N`, with `l2_gas_price = 100 fri`, `next_l2_gas_price = 112 fri` (network at 112% of gas target for several blocks).
- `min_gas_price_percentage = 50` (threshold = 50 fri).
- Batcher is building `block_N+1` with `l2_gas_price_fri = 112 fri`.

**Attack (Mode A — invalid transaction admitted):**
1. Attacker submits an `InvokeV3` transaction with `l2_gas.max_price_per_unit = 60 fri`.
2. Gateway calls `validate_resource_bounds`: threshold = 50% × 100 = 50 fri; 60 ≥ 50 → **admitted**.
3. Transaction enters the mempool.
4. Batcher pulls the transaction and calls `perform_pre_validation_stage` → `check_fee_bounds` with block price = 112 fri: 60 < 112 → **`MaxGasPriceTooLow` error**.
5. Transaction is dropped by the batcher; mempool slot was wasted.

**Attack (Mode B — valid transaction rejected, high percentage config):**
1. `min_gas_price_percentage = 95`. Latest block: `l2_gas_price = 100 fri`, `next_l2_gas_price = 88 fri` (low load).
2. User submits a transaction with `l2_gas.max_price_per_unit = 90 fri`.
3. Gateway threshold = 95% × 100 = 95 fri; 90 < 95 → **rejected with `GAS_PRICE_TOO_LOW`**.
4. But the batcher would have accepted it: 90 ≥ 88 → `check_fee_bounds` passes.
5. User's valid transaction is incorrectly denied.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
