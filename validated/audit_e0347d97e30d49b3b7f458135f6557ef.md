### Title
Contract Update Bypasses Deployment Authorization Restriction — (`fvm/environment/contract_updater.go`)

---

### Summary

`ContractUpdaterImpl.SetContract` enforces the `isAuthorizedForDeployment` check only when a contract does not yet exist (`!exists`). When a contract already exists, the update path skips the authorization check entirely, with the comment "Contract updates are always allowed." This directly contradicts the documented intent of `SetContractDeploymentAuthorizersTransaction`, which explicitly covers "deploy/**update** contracts," and the `GetAuthorizedAccounts` docstring, which lists "deploy, **update**, or remove." Any account that already has a contract deployed can replace its bytecode with arbitrary code without being in the authorized list, even when restricted deployment mode is active.

---

### Finding Description

**Root cause — `fvm/environment/contract_updater.go`, lines 379–395:**

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
...
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: ...")
}
```

The guard condition is `!exists && !isAuthorizedForDeployment(...)`. When `exists == true` (an update), the entire authorization check is short-circuited. The update is unconditionally queued into `draftUpdates` and committed by `Commit()`.

**Documented intent contradicts the implementation:**

- `fvm/blueprints/contracts.go` line 35: `SetContractDeploymentAuthorizersTransaction` is described as "updating list of authorized accounts allowed to **deploy/update** contracts."
- `fvm/environment/contract_updater.go` lines 223–225: `GetAuthorizedAccounts` is described as "Used to determine which accounts are permitted to **deploy, update, or remove** contracts."

There is no `ContractUpdateAuthorizedAddressesPath` and no `isAuthorizedForUpdate` function. The deployment authorized-address list (`ContractDeploymentAuthorizedAddressesPath`) is the intended gate for updates, but `SetContract` never consults it when `exists == true`.

**Attacker-controlled entry path:**

1. Restricted deployment mode is active (`RestrictedDeploymentEnabled() == true`).
2. Attacker controls an account that already has a contract deployed (e.g., deployed before the restriction was enabled, or deployed when the account was previously authorized).
3. Attacker submits a transaction calling `account.contracts.update(name: "X", code: <malicious bytecode>)`.
4. The Cadence runtime calls `UpdateAccountContractCode` → `SetContract`.
5. `ContractExists` returns `true` → the `isAuthorizedForDeployment` check is skipped.
6. The malicious bytecode is queued and committed to chain state.

No privileged keys are required. The attacker only needs to own an account with an existing contract.

---

### Impact Explanation

When the protocol enables restricted deployment mode to lock down who may modify on-chain contracts (e.g., during a security incident or controlled upgrade window), any account that already has a contract deployed can silently replace that contract's bytecode with arbitrary code. This is an unauthorized state change: the attacker modifies on-chain contract logic without being in the authorized list, bypassing the governance-equivalent restriction. Downstream callers of the updated contract (other contracts, dApps, protocol infrastructure) will execute the attacker-controlled code. The impact is unauthorized mutation of on-chain contract state, equivalent to the Audius `upgradeToAndCall` bypass.

---

### Likelihood Explanation

The precondition — owning an account with an existing contract — is trivially satisfied by any account that deployed a contract before the restriction was enabled, or any account that was previously authorized and later removed from the list. On a live network, many such accounts exist. The attacker entry point is a standard user transaction with no special privileges. Likelihood is high whenever restricted deployment mode is active.

---

### Recommendation

Apply `isAuthorizedForDeployment` to both the new-deployment and the update branch in `SetContract`:

```go
if !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("setting contract failed: %w",
        errors.NewOperationAuthorizationErrorf(
            "SetContract",
            "deploying or updating contracts requires authorization from specific accounts"))
}
```

Remove the `!exists &&` guard so the check applies unconditionally when restricted deployment is enabled. Update the inline comment to reflect that updates are also gated. Align `SetContractDeploymentAuthorizersTransaction`'s behavior with the code.

---

### Proof of Concept

**Setup:** Restricted deployment mode enabled; authorized list = `[serviceAccount]`; attacker account `0xABCD` has contract `Foo` already deployed (from before restriction was enabled).

**Attacker transaction:**
```cadence
transaction {
  prepare(acct: auth(UpdateContract) &Account) {
    let maliciousCode = "access(all) contract Foo { access(all) fun steal() { /* drain funds */ } }"
    acct.contracts.update(name: "Foo", code: maliciousCode.utf8)
  }
}
// Signed by: 0xABCD only (not in authorized list)
```

**FVM execution trace:**
1. `UpdateAccountContractCode(location={0xABCD, "Foo"}, code=maliciousCode)` called.
2. `SetContract` called; `ContractExists("Foo", 0xABCD)` → `true`.
3. Condition `!exists && !isAuthorizedForDeployment(...)` → `false && ...` → **check skipped**.
4. `draftUpdates[{0xABCD,"Foo"}] = ContractUpdate{code: maliciousCode}` queued.
5. `Commit()` calls `accounts.SetContract("Foo", 0xABCD, maliciousCode)` → committed to ledger.

**Result:** `0xABCD`'s contract `Foo` now contains attacker-controlled bytecode, despite `0xABCD` not being in the authorized list. The authorization restriction is fully bypassed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
