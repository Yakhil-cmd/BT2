### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions with Nonce=1, Admitting Invalid-Signature Transactions to the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point execution for any Invoke transaction with `nonce=1` when the sender account has on-chain `nonce=0` and **any** transaction for that account exists in the mempool. Because `__validate__` is the sole mechanism that verifies the transaction's cryptographic signature at the gateway level, an attacker who observes a legitimate `deploy_account` transaction in the mempool can submit an Invoke transaction with an invalid/forged signature for the same account and have it admitted to the mempool without any signature check.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

Condition 4 is satisfied by the presence of **any** transaction for the account in the mempool — not specifically a `deploy_account` transaction. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incomplete: an attacker can exploit the window after a legitimate `deploy_account` is admitted to inject their own Invoke with nonce=1.

**How the skip propagates to disable `__validate__`:** [2](#0-1) 

When `skip_validate=true`, `execution_flags.validate` is set to `false`. [3](#0-2) 

`StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`, so no signature verification occurs. [4](#0-3) 

`validate_tx` also short-circuits on `execution_flags.validate == false`, confirming no entry-point call is made.

**Attack scenario:**

1. Alice submits a `deploy_account` transaction for her account (on-chain nonce=0). It passes full gateway execution validation and enters the mempool.
2. Attacker observes Alice's `deploy_account` in the public mempool.
3. Attacker submits an Invoke transaction with `nonce=1` for Alice's account address, carrying an **invalid/forged signature**.
4. Gateway calls `skip_stateful_validations`: `tx.nonce()=1`, `account_nonce=0`, `account_tx_in_pool_or_recent_block=true` → returns `true`.
5. `run_validate_entry_point` sets `validate=false`; `__validate__` is never called.
6. The invalid-signature Invoke transaction is admitted to the mempool. [5](#0-4) 

The `extract_state_nonce_and_run_validations` method is the single gateway entry point that orchestrates nonce fetch → pre-validation → skip decision → entry-point execution. The skip decision at line 175–177 directly controls whether `__validate__` runs.

**`validate_by_mempool` does not compensate:** [6](#0-5) 

The mempool's `validate_tx` checks nonce ordering and duplicates but performs no cryptographic signature verification, so it does not catch the invalid signature.

---

### Impact Explanation

The gateway mempool admission path accepts an Invoke transaction whose cryptographic signature has never been verified. This directly matches the allowed High impact:

> *High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

Concrete consequences:
- **Mempool pollution**: The attacker can inject arbitrarily many invalid-signature Invoke transactions for any account that has a pending `deploy_account` in the mempool.
- **Transaction displacement**: If the mempool supports fee escalation for the same `(address, nonce)` slot, the attacker's higher-fee invalid transaction can displace Alice's legitimate Invoke transaction, preventing it from being sequenced.
- **Block-space waste**: The batcher will pull the invalid transaction, attempt execution (where `__validate__` runs with `validate=true` and fails), and waste block capacity.

---

### Likelihood Explanation

- The mempool is publicly observable; any pending `deploy_account` transaction is visible to all network participants.
- No privileged access is required. Any external actor can craft an Invoke transaction with `nonce=1` for any address with a pending `deploy_account`.
- The attack window is the entire time the `deploy_account` transaction remains unconfirmed in the mempool, which can be minutes to hours.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** exists in the mempool for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that only returns `true` when the pending transaction is of type `DeployAccount`. This preserves the UX feature (deploy + invoke in one batch) while closing the bypass for arbitrary invalid-signature Invoke transactions.

Alternatively, require the caller to supply the `deploy_account` transaction hash (as the `native_blockifier` `PyValidator` already does via `deploy_account_tx_hash: Option<TransactionHash>`), and verify that hash is present in the mempool as a `DeployAccount` transaction before skipping validation. [7](#0-6) 

The `PyValidator::should_run_stateful_validations` in `native_blockifier` already takes an explicit `deploy_account_tx_hash` parameter and checks `deploy_account_tx_hash.is_some()` — a stricter guard than the gateway's `account_tx_in_pool_or_recent_block`. The gateway path should adopt the same pattern.

---

### Proof of Concept

```
1. Alice broadcasts: deploy_account_tx { sender: 0xALICE, nonce: 0, sig: valid_sig }
   → Gateway: full execution validation passes → mempool admits it.

2. Attacker queries mempool, sees 0xALICE has a pending deploy_account.

3. Attacker broadcasts: invoke_tx { sender: 0xALICE, nonce: 1, sig: 0xDEADBEEF (invalid) }

4. Gateway stateful validator:
   - get_nonce_from_state(0xALICE) → Nonce(0)          [account not deployed]
   - validate_nonce: nonce=1 ≤ 0 + max_gap → passes
   - validate_by_mempool: nonce ordering OK → passes
   - skip_stateful_validations:
       tx.nonce() == 1 ✓
       account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(0xALICE) == true ✓  (Alice's deploy_account is there)
       → returns true (SKIP)
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       StatefulValidator::perform_validations → returns Ok(()) immediately
       __validate__ is NEVER called
   → Invalid-signature invoke_tx admitted to mempool ✓

5. Batcher pulls attacker's invoke_tx, executes with validate=true:
   __validate__ runs → signature check fails → transaction reverts.
   Block space wasted; if fee escalation displaced Alice's valid invoke, Alice's tx is lost.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
