### Title
Unbounded EIP-6110 Deposit Event Iteration in Block Finalization Without Per-Request Cap — (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

The `eip6110_events_parser` function iterates over **all** events accumulated during a block with no upper bound on the number of deposit events processed. Unlike the EIP-7002 and EIP-7251 system-part handlers — which enforce `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16` and `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2` respectively — the EIP-6110 path has no analogous cap. An attacker who fills a block with transactions that emit deposit events from the deposit contract address forces the block-finalization routine to perform unbounded SHA-256 hashing work that is entirely outside user-gas accounting, constituting a resource-accounting bug and a valid-execution unprovability path in the proving context.

---

### Finding Description

`eip6110_events_parser` is called unconditionally during `post_op` (block finalization) in both the sequencing path (`post_tx_op_sequencing.rs`) and the proving path (`post_tx_op_proving.rs`). Its implementation is:

```rust
for event in system.io.events_iterator() {          // iterates ALL block events
    if event.address != &DEPOSIT_CONTRACT_ADDRESS {
        continue;
    }
    if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
        // SHA-256 hashes 192 bytes of deposit data per matching event
        let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
            panic!("invalid deposit event structure");
        };
    }
}

There is no constant analogous to `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK` or `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK` for deposits: [2](#0-1) [3](#0-2) 

The EIP-7002 and EIP-7251 handlers both apply `core::cmp::min(num_in_queue, MAX_*_PER_BLOCK)` before entering their loops: [4](#0-3) [5](#0-4) 

No equivalent guard exists for EIP-6110. The `validate_and_write_event_data` function performs SHA-256 hashing of 192 bytes of deposit data per matching event: [6](#0-5) 

All of this work executes during `post_op`, which runs with `FORMAL_INFINITE` resources — entirely outside the user-gas budget charged during transaction execution. [7](#0-6) [8](#0-7)

### Impact Explanation

**Resource accounting bug / valid-execution unprovability.**

1. **Proving context (RISC-V):** Block finalization is part of the state-transition function compiled to RISC-V for ZK proof generation. Unbounded SHA-256 hashing work in `eip6110_events_parser` directly inflates the RISC-V cycle count for the block. A sufficiently large number of deposit events can push the cycle count beyond the prover's capacity, making a validly-executed block **unprovable**. This is a liveness break: the sequencer produces a valid block that the prover cannot prove, stalling the chain.

2. **Sequencer context:** The same loop runs in `post_tx_op_sequencing.rs` during block sealing. Excessive deposit events cause unbounded CPU work at block finalization time, degrading sequencer throughput and potentially causing a denial-of-service on block production.

3. **Divergence between sequencer and prover:** Because the sequencer runs the loop to completion while the prover may time out or run out of cycles, the two can diverge on whether a block is valid, breaking the forward/proving equivalence invariant.

---

### Likelihood Explanation

The deposit contract (`0x00000000219ab540356cBB839Cbe05303d7705Fa`) is a standard Ethereum predeploy. Any user with sufficient ETH can call it and emit a `DepositEvent`. The cost per deposit event is bounded by the LOG4 opcode cost plus 576 bytes of calldata/data cost — roughly 6,000–7,000 gas per event. With a 10–15M gas block limit, an attacker can pack approximately 1,400–2,500 deposit events into a single block. Each event triggers 192 bytes of SHA-256 hashing work in `validate_and_write_event_data` that is uncharged. This is directly attacker-controlled via normal, unprivileged transactions.

---

### Recommendation

Apply a `MAX_DEPOSIT_REQUESTS_PER_BLOCK` cap (the Ethereum EIP-6110 spec defines this as `8192`, but a tighter value matching the block gas limit is appropriate) before entering the deposit-event loop, consistent with the pattern already used for EIP-7002 and EIP-7251. Additionally, consider counting matching deposit events and breaking early once the cap is reached, rather than iterating the entire event list.

---

### Proof of Concept

1. Deploy or interact with the EIP-6110 deposit contract at `0x00000000219ab540356cBB839Cbe05303d7705Fa` on the ZKsync OS Ethereum-mode chain.
2. Craft N transactions (where N × ~6,500 gas ≤ block gas limit) each calling the deposit contract with valid 576-byte deposit data, causing N `DepositEvent` log emissions.
3. Submit all N transactions in a single block.
4. Observe that `eip6110_events_parser` iterates all N events during `post_op`, performing `N × 192` bytes of SHA-256 hashing with no gas charged to the submitter.
5. In the proving path (`post_tx_op_proving.rs` → `eip6110_events_parser`), the RISC-V cycle count for block finalization grows linearly with N, unbounded by any per-block cap, while EIP-7002 and EIP-7251 are capped at 16 and 2 respectively. [9](#0-8) [10](#0-9)

                panic!("invalid deposit event structure");

fn validate_and_write_event_data(
    event: GenericEventContentWithTxRef<'_, MAX_EVENT_TOPICS, EthereumIOTypesConfig>,
    requests_hasher: &mut impl crypto::sha256::Digest,
    logger: &mut impl Logger,
) -> Result<(), ()> {
    let data = event.data;
    if data.len() != 576 {
        return Err(());
    }
    let mut chunks = data.as_chunks::<32>().0.iter();
    validate_u16_at_most(chunks.next().unwrap(), 160)?;
    validate_u16_at_most(chunks.next().unwrap(), 256)?;
    validate_u16_at_most(chunks.next().unwrap(), 320)?;
    validate_u16_at_most(chunks.next().unwrap(), 384)?;
    validate_u16_at_most(chunks.next().unwrap(), 512)?;
    drop(chunks);

    validate_u16_at_most(data[160..192].try_into().unwrap(), 48)?;
    validate_u16_at_most(data[256..288].try_into().unwrap(), 32)?;
    validate_u16_at_most(data[320..352].try_into().unwrap(), 8)?;
    validate_u16_at_most(data[384..416].try_into().unwrap(), 96)?;
    validate_u16_at_most(data[512..544].try_into().unwrap(), 8)?;

    logger_log!(logger, "Processing EIP-6110 deposit event with:");

    logger_log!(logger, "\nPubkey = ");
    let pubkey = &data[192..][..48];
    let _ = logger.log_data(pubkey.iter().copied());
    requests_hasher.update(pubkey);

    logger_log!(logger, "\nWithdrawal credentials = ");
    let withdrawal_credentials = &data[288..][..32];
    let _ = logger.log_data(withdrawal_credentials.iter().copied());
    requests_hasher.update(withdrawal_credentials);

    let amount = &data[352..][..8];
    logger_log!(
        logger,
        "\nAmount = {}",
        u64::from_le_bytes(amount.try_into().unwrap())
    );
    requests_hasher.update(amount);

    logger_log!(logger, "\nSignature = ");
    let signature = &data[416..][..96];
    let _ = logger.log_data(signature.iter().copied());
    requests_hasher.update(signature);

    let index = &data[544..][..8];
    logger_log!(
        logger,
        "\nIndex = {}\n",
        u64::from_le_bytes(index.try_into().unwrap())
    );
    requests_hasher.update(index);

    Ok(())
}
