Audit Report

## Title
Unsanitized Memo Interpolation Enables Markdown Injection into ICRC-21 GenericDisplayMessage — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

`ConsentMessage::add_memo` decodes the ICRC-1 memo bytes as UTF-8 and splices the result verbatim into a Markdown-formatted `GenericDisplayMessage` string without escaping backticks or newlines. A backtick in the memo closes the surrounding code-span fence early, allowing arbitrary Markdown — including fake `**To:**` fields — to be injected into the consent message returned by `icrc21_canister_call_consent_message`. Hardware wallets that render this string as Markdown (the intended ICRC-21 behavior) will display the injected content as legitimate transaction fields, undermining the core security guarantee of the ICRC-21 consent mechanism.

## Finding Description

In `add_memo` (`responses.rs` lines 284–295), the memo bytes are decoded as UTF-8 (or hex-encoded if invalid), then interpolated directly into the Markdown string:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

No escaping of backticks, newlines, or any other Markdown control characters is applied to `memo_str`. [1](#0-0) 

If `memo_str` contains a backtick followed by a newline and Markdown markup, the code-span opened by the leading `` ` `` is closed early and the remainder is rendered as raw Markdown. For example, a memo of `` `\n\n**To:**\n`victim-address `` produces:

```
\n\n**Memo:**\n``\n\n**To:**\n`victim-address`
```

which renders as a legitimate-looking `**To:**` field containing `victim-address`.

The `ConsentMessageBuilder::build` method adds the memo **after** all real transaction fields (including the real `**To:**` field), so the injected field appears at the end of the message: [2](#0-1) 

The real `**To:**` (attacker's address) is added at line 206, before the memo: [3](#0-2) 

On a hardware wallet with a small scrolling display, the injected field appearing last can be mistaken for the authoritative destination address. The `FieldsDisplayMessage` variant is not affected because it stores fields as structured `Vec<(String, Value)>` and never concatenates user content into a Markdown string. [4](#0-3) 

## Impact Explanation

The ICRC-21 consent mechanism exists specifically to protect users from malicious dapps: even if a dapp is compromised or adversarial, the user's hardware wallet calls `icrc21_canister_call_consent_message` on the trusted ledger canister to obtain an authoritative description of what the transaction does. This vulnerability breaks that guarantee. A malicious dapp can craft a `TransferArg` with `to = attacker_address` and a memo containing Markdown injection, causing the trusted ledger to return a consent message that displays a fake `**To:**` field pointing to a victim-controlled or decoy address. A user who approves based on the injected display authorizes a transfer to the attacker's actual address. This constitutes **unauthorized access to user funds / ledger assets** — a High-severity impact under the allowed ICP bounty scope.

## Likelihood Explanation

The `icrc21_canister_call_consent_message` endpoint is fully public; any caller can invoke it with a crafted `TransferArg`. The ICRC-1 memo field accepts up to 32 bytes, which is more than sufficient for the injection payload. The 500-byte arg size limit is easily satisfied. The attack requires a malicious dapp (the standard ICRC-21 threat model) and a hardware wallet that renders `GenericDisplayMessage` as Markdown, which is the intended behavior per the ICRC-21 specification. No privilege, key, or governance majority is required.

## Recommendation

Escape all Markdown control characters in `memo_str` before interpolation. At minimum, replace backticks and strip newlines:

```rust
let safe_memo = memo_str
    .replace('`', "\\`")
    .replace('\n', " ")
    .replace('\r', "");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, use a Markdown-escaping library, or switch the memo display to a `FieldsDisplayMessage` field (which is already injection-safe by design).

## Proof of Concept

```rust
#[test]
fn memo_injection_creates_fake_to_field() {
    use icrc_ledger_types::icrc21::lib::{GenericMemo, Icrc21Function};
    use icrc_ledger_types::icrc21::responses::ConsentMessage;

    let mut msg = ConsentMessage::GenericDisplayMessage(String::new());
    msg.add_intent(Icrc21Function::Transfer, Some("ICP".to_string()));
    msg.add_account("To", "attacker-principal".to_string());
    // Memo closes the code span and injects a fake To: field
    let evil_memo = b"`\n\n**To:**\n`victim-principal".to_vec();
    msg.add_memo(GenericMemo::Icrc1Memo(evil_memo.into()));

    let rendered = match msg {
        ConsentMessage::GenericDisplayMessage(s) => s,
        _ => panic!(),
    };
    // With the current code this assertion FAILS, proving injection:
    // the rendered string contains two **To:** fields
    assert_eq!(rendered.matches("**To:**").count(), 1);
}
```

The test fails on the current code because `rendered.matches("**To:**").count()` returns 2 — one real field and one injected field — demonstrating the injection.
