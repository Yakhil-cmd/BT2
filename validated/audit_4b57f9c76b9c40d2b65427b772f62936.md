### Title
Unchecked `cancelOffer` Return Value Causes Silent Failure and False Cancellation State in Offer Conflict Resolution - (File: packages/gui/src/components/offers2/CancelOfferList.tsx)

---

### Summary
In `CancelOfferList.tsx`, the `cancelOffer` RTK Query mutation is called without `.unwrap()`, meaning RPC failures are silently swallowed. The `onOfferCanceled` state-update callback is then called unconditionally regardless of whether the cancellation actually succeeded. In the offer-conflict-resolution flow (`OfferEditorCancelConflictingOffersDialog`), this causes the UI to falsely report that a blocking offer has been cancelled and recalculate spendable balances upward, allowing the user to proceed to create a new offer while the old conflicting offer remains live on-chain.

---

### Finding Description

In `handleCancelOffer` inside `CancelOfferList.tsx`:

```ts
await cancelOffer({ tradeId, secure, fee });   // line 61 — no .unwrap()
onOfferCanceled(tradeId, secure, fee);          // line 62 — called unconditionally
```

RTK Query mutations, when awaited **without** `.unwrap()`, always resolve to `{ data }` or `{ error }` — they never throw. Any RPC-level failure (network drop, daemon error, wallet locked, etc.) is silently discarded. The `onOfferCanceled` callback fires regardless.

`CancelOfferList` is consumed by `OfferEditorCancelConflictingOffersDialog` via two callbacks:

- `onCancelOffer1` — called when an offer **must** be cancelled to free locked assets before a new offer can be created
- `onCancelOffer2` — called for offers that share the same locked assets as the new offer

Both callbacks update local React state (`setAssetsToUnlock`, `setAssetsBetterUnlocked`) to remove the "cancelled" offer and recalculate spendable amounts upward. The dialog's "Proceed" gate (`offersRequiredToBeCanceled.length === 0`) then evaluates to true, and the user is allowed to confirm and submit the new offer — all while the old conflicting offer is still active on-chain.

A second instance of the same pattern exists in `OfferManager.tsx` `relistOffer`:

```ts
await cancelOffer({ tradeId, secure: false, fee: 0 });  // line 106 — no .unwrap()
// ... immediately navigates to offer builder
navigate('/dashboard/offers/builder', { ... });
```

If the cancellation fails, the user is sent to the offer builder to create a replacement offer while the original offer remains live.

---

### Impact Explanation

**High.** When `cancelOffer` fails silently:

1. The conflict-resolution dialog removes the offer from its required-cancellation list and adds its locked assets back to the computed spendable balance.
2. The user sees a "sufficient balance" state and is permitted to proceed.
3. The new offer is created and potentially shared.
4. The old conflicting offer is still active and can be accepted by any counterparty who holds the offer file.
5. If the old offer is accepted, the user's XCH/CAT/NFT assets are transferred without their knowledge or intent.

This matches the allowed High impact: *"Corruption or unsafe trust of offer state that causes a user to sign/send the wrong asset or display the wrong status."*

---

### Likelihood Explanation

The failure path is reachable by any transient condition: a momentary RPC timeout, a wallet daemon restart, a network interruption, or a wallet that is briefly locked. No attacker privilege is required — the attacker only needs to hold a copy of the victim's previously shared offer file and wait for the victim to attempt (and silently fail) to cancel it.

---

### Recommendation

Replace the fire-and-forget call with `.unwrap()` so that failures propagate as thrown exceptions, and wrap in try/catch to surface an error to the user:

```ts
// CancelOfferList.tsx — handleCancelOffer
try {
  await cancelOffer({ tradeId, secure, fee }).unwrap();
  onOfferCanceled(tradeId, secure, fee);
} catch (e) {
  showError(e); // surface the failure; do NOT call onOfferCanceled
}
```

Apply the same fix to `relistOffer` in `OfferManager.tsx`:

```ts
try {
  await cancelOffer({ tradeId, secure: false, fee: 0 }).unwrap();
  navigate('/dashboard/offers/builder', { ... });
} catch (e) {
  showError(e);
}
```

### Proof of Concept

1. User has an open offer locking 1 XCH and shares the offer file with a counterparty.
2. User opens the offer builder to create a new offer using the same 1 XCH.
3. `OfferEditorCancelConflictingOffersDialog` appears, listing the old offer as a required cancellation.
4. User clicks **Cancel** on the conflicting offer.
5. At the moment `cancelOffer` RPC fires, the wallet daemon is briefly unavailable (e.g., restarting). The RPC returns `{ error: ... }`.
6. Because `.unwrap()` is absent, no exception is thrown. `onCancelOffer1` fires, removes the offer from `assetsToUnlock`, and recalculates spendable balance to show 1 XCH available.
7. `offersRequiredToBeCanceled.length` is now 0; the dialog's confirm button is enabled.
8. User confirms and creates the new offer.
9. The counterparty, still holding the original offer file, calls `take_offer` and the old offer is accepted on-chain.
