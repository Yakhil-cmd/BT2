### Title
Unsigned Invoke Transaction Admitted to Mempool via `skip_stateful_validations` Overly-Broad Presence Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is a UX feature that allows an invoke with nonce=1 to bypass the `__validate__` entry-point call when the account's `deploy_account` (nonce=0) is still pending in the mempool. The guard that enables this bypass — `account_tx_in_pool_or_recent_block(sender_address)` — returns `true` for **any** transaction type in the pool, not only `deploy_account`. An attacker who observes a victim's `deploy_account` in the mempool can immediately submit an invoke with `sender_address = victim`, `nonce = 1`, and an **arbitrary/invalid signature**. Because the presence check passes, `__validate__` is skipped, and the gateway admits the unsigned transaction into the mempool.

---

### Finding Description

**Step 1 — UX skip path in `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip `__validate__`) when:
- the incoming tx is an `Invoke`,
- `tx.nonce() == 1`,
- `account_nonce == 0` (account not yet deployed), and
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

**Step 2 — The presence check is not type-specific** [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or in a recently committed block. It does not verify that the pooled transaction is a `deploy_account`.

The code comment acknowledges this: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The second clause is the flaw: an attacker's own previously-injected transaction (or the victim's legitimate `deploy_account`) satisfies the check.

**Step 3 — `__validate__` is suppressed when skip=true** [3](#0-2) 

When `skip_validate = true`, `execution_flags.validate` is set to `false`. The blockifier's `validate_tx` then returns `Ok(None)` immediately without executing the account's `__validate__` entry point: [4](#0-3) 

**Step 4 — Mempool-level checks do not verify signatures**

`validate_by_mempool` (called before `skip_stateful_validations`) only checks nonce ordering and fee-escalation rules: [5](#0-4) 

No cryptographic signature verification occurs at the mempool level.

**Step 5 — The full gateway admission path** [6](#0-5) 

After `extract_state_nonce_and_run_validations` returns without error, the transaction is forwarded to the mempool via `add_tx`. The unsigned invoke is now a live mempool entry for the victim's address.

---

### Impact Explanation

An attacker can inject an invoke transaction with an **invalid signature** for any victim whose `deploy_account` is pending in the mempool. The gateway's admission invariant — every accepted transaction must have its signature verified by the account's `__validate__` entry point — is broken. The admitted transaction occupies the victim's nonce=1 slot. The victim's legitimate invoke with nonce=1 is then rejected as `DuplicateNonce` (fee escalation disabled) or forced to pay a higher fee to replace it (fee escalation enabled). The attacker can repeat this continuously at negligible cost, permanently blocking the victim's first post-deploy invoke.

This matches the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."**

---

### Likelihood Explanation

- The victim's `deploy_account` is publicly observable in the mempool.
- The attacker only needs to know the victim's address (public) and submit a single invoke with nonce=1 and any bytes as the signature.
- No privileged access, no special contract, no funds required.
- The attack window is the entire time the victim's `deploy_account` is pending (potentially many blocks).

---

### Recommendation

Replace the overly-broad `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction exists for the sender's address in the mempool. Concretely, add a new mempool query such as `deploy_account_tx_in_pool(sender_address)` that inspects the transaction type, and use that in `skip_stateful_validations` instead of the generic presence check.

Alternatively, restrict the skip path so that it only applies when the **caller** is the same entity that submitted the `deploy_account` (e.g., by requiring the invoke to arrive in the same gateway request batch as the `deploy_account`), or require a minimal signature check even in the skip path.

---

### Proof of Concept

1. **Victim** submits a `deploy_account` transaction for address `V` (nonce=0). It enters the mempool. `account_tx_in_pool_or_recent_block(V)` now returns `true`.

2. **Attacker** constructs an `RpcInvokeTransactionV3` with:
   - `sender_address = V`
   - `nonce = 1`
   - `calldata` = arbitrary (e.g., drain victim's funds via a call)
   - `signature` = `[]` (empty or random bytes — passes the stateless size check)

3. Attacker POSTs this to `POST /gateway/add_transaction`.

4. Gateway flow:
   - `stateless_tx_validator.validate`: passes (empty signature is within `max_signature_length`). [7](#0-6) 
   - `extract_state_nonce_and_run_validations`: fetches `account_nonce = 0` for address `V`.
   - `validate_nonce`: `0 <= 1 <= 0 + max_gap` → passes. [8](#0-7) 
   - `validate_by_mempool`: no existing nonce=1 tx for `V` → passes.
   - `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(V)==true` → returns `true`. [9](#0-8) 
   - `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate=false` → `__validate__` not called → returns `Ok(())`.

5. Transaction is forwarded to the mempool and accepted. The attacker's unsigned invoke now occupies `(V, nonce=1)`.

6. Victim submits their legitimate invoke with `nonce=1` and a valid signature → rejected with `DuplicateNonce`.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
