### Title
Gateway fee validation uses previous block's L2 gas price instead of next block's, causing admission/rejection mismatch - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` validates a transaction's `max_price_per_unit` against the **previous block's** L2 gas price. The same previous-block gas prices are forwarded into the blockifier's `check_fee_bounds` inside `run_validate_entry_point`. However, the batcher executes transactions against the **next block's** L2 gas price (computed by the EIP-1559 fee market). When the next block's price is higher than the previous block's, the gateway admits transactions that the batcher will reject. When it is lower, the gateway rejects transactions that the batcher would have accepted.

### Finding Description

In `validate_resource_bounds`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The developer's own TODO comment acknowledges the wrong price is being used. The threshold check rejects any transaction whose `l2_gas.max_price_per_unit < min_gas_price_percentage% × previous_block_price`. [2](#0-1) 

In `run_validate_entry_point`, the blockifier validation block context is built from the same previous-block info, with only `block_number` incremented — gas prices are **not** updated to the next block's values:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [3](#0-2) 

So `check_fee_bounds` inside `perform_pre_validation_stage` also runs against the previous block's gas prices during gateway admission. [4](#0-3) 

The batcher, by contrast, receives the **next block's** gas prices from the consensus orchestrator (via `propose_block_input.block_info`) and builds a fresh `BlockContext` with those prices: [5](#0-4) 

The batcher also pushes the next block's L2 gas price to the mempool: [6](#0-5) 

The next block's L2 gas price is computed by the EIP-1559 fee market (`calculate_next_base_gas_price`), which can increase the price by up to `price / gas_price_max_change_denominator` per block when blocks are full: [7](#0-6) 

When the batcher executes a transaction, `perform_pre_validation_stage` calls `check_fee_bounds` with the **next block's** gas prices. If `tx.l2_gas.max_price_per_unit < next_block_l2_gas_price`, the transaction fails with `InsufficientResourceBounds` / `MaxGasPriceTooLow` — a hard pre-validation rejection, not a revert. [8](#0-7) 

### Impact Explanation

**Incorrect admission (primary impact):** A transaction submitted with `max_price_per_unit = previous_block_l2_gas_price` satisfies the gateway's threshold (with default `min_gas_price_percentage = 100`). If the next block's price is higher (EIP-1559 increase due to full blocks), the batcher's `check_fee_bounds` rejects the transaction with `MaxGasPriceTooLow`. The gateway has admitted a transaction that can never be sequenced.

**Incorrect rejection (secondary impact):** If the next block's price is lower than the previous block's (EIP-1559 decrease due to under-utilized blocks), a transaction with `max_price_per_unit` between the two prices is rejected by the gateway even though the batcher would have accepted it.

Both outcomes match the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The EIP-1559 fee market is always active. Whenever consecutive blocks are above the gas target, the next block's L2 gas price rises above the previous block's. With `gas_price_max_change_denominator` controlling the rate, even a single full block produces a measurable price increase. Any user submitting a transaction with `max_price_per_unit` exactly at the previous block's price (the minimum the gateway accepts) will have their transaction admitted but then silently dropped by the batcher. This is a normal operating condition, not an edge case.

### Recommendation

Replace the previous-block L2 gas price lookup with the **next block's** L2 gas price. The consensus orchestrator already computes this value (`calculate_next_l2_gas_price_for_fin`). It should be made available to the gateway — for example, by exposing it in the block header or via a dedicated RPC call — so that `validate_resource_bounds` and the blockifier block context in `run_validate_entry_point` both use the price the batcher will actually enforce.

The TODO comment at line 229 already identifies the fix: `// TODO(Arni): getnext_l2_gas_price from the block header.` [9](#0-8) 

### Proof of Concept

1. Observe the current L2 gas price from the latest committed block: `P_prev`.
2. Submit a V3 invoke transaction with `l2_gas.max_price_per_unit = P_prev`. The gateway's `validate_tx_l2_gas_price_within_threshold` passes because `P_prev >= 100% × P_prev`.
3. The transaction enters the mempool.
4. The batcher starts proposing the next block. The consensus orchestrator computes `P_next = calculate_next_base_gas_price(P_prev, gas_used, gas_target, min_price)`. If the previous block was full (`gas_used > gas_target`), then `P_next > P_prev`.
5. The batcher builds a `BlockContext` with `l2_gas_price = P_next` and calls `perform_pre_validation_stage` on the transaction.
6. `check_fee_bounds` compares `max_price_per_unit = P_prev` against `actual_gas_price = P_next`. Since `P_prev < P_next`, it returns `MaxGasPriceTooLow`.
7. The transaction is rejected by the batcher despite having been admitted by the gateway.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
