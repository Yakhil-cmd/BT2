### Title
Secret Signing Shares and Presign Nonce Material Exposed via `Debug` Trait Derivation - (File: `src/lib.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/triples/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Multiple production types holding private signing shares, Beaver triple secret scalars, and presign nonce shares unconditionally derive `Debug`, causing the raw scalar values to be emitted in plaintext whenever these types are formatted with `{:?}`. Any library caller that logs, panics on, or error-formats these types will inadvertently disclose the secret material. The codebase's own documentation explicitly states that exposure of `TripleShare` scalars allows recovery of the private signing key.

---

### Finding Description

The following production types derive `Debug` without a custom redacting implementation:

**1. `KeygenOutput<C>` — `src/lib.rs` line 48**
```rust
#[derive(Debug, Clone, Deserialize, Serialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,   // ← raw scalar exposed via {:?}

```
`SigningShare<C>` from `frost_core` implements `Debug` by printing the inner scalar. Any `{:?}` formatting of `KeygenOutput` emits the private signing share in hex.

**2. `PresignArguments` — `src/ecdsa/ot_based_ecdsa/mod.rs` line 23**

---

### Impact Explanation

**Critical — Extraction, reconstruction, or disclosure of private signing shares, presign secrets, and nonce material.**

- Logging `KeygenOutput` via `{:?}` directly discloses the participant's private signing share `x_i`. With `t` or more such shares an attacker reconstructs the master secret key.
- Logging `TripleShare` via `{:?}` discloses the Beaver triple scalars `(a, b, c)`. Per the codebase's own documentation, this allows recovery of the private signing key from any signature produced with that triple.
- Logging `PresignOutput` via `{:?}` discloses the per-signature nonce shares `k`/`sigma` (OT-based) or `c`/`e`/`alpha`/`beta` (Robust). Nonce reuse or disclosure is a classical ECDSA private-key recovery vector.
- `PresignArguments` aggregates all of the above into a single type that is the direct public API input to `presign()`, maximizing the blast radius of a single accidental log line.

---

### Likelihood Explanation

**High.** `Debug` is the default formatting trait used by Rust's `println!("{:?}", ...)`, `dbg!(...)`, `tracing`/`log` macros, `unwrap()`/`expect()` panic messages, and test assertion failures. Library integrators routinely log protocol inputs and outputs for observability. Because `PresignArguments` is a public API struct passed by value into `presign()`, any integrator that logs it (e.g., `tracing::debug!(?args, "starting presign")`) immediately leaks both the private share and the triple scalars. No attacker capability beyond reading application logs is required.

---

### Recommendation

1. **Remove `Debug` from all types holding secret scalar material.** Replace with a custom `Debug` implementation that redacts secret fields:
   ```rust
   impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
       fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
           f.debug_struct("KeygenOutput")
            .field("private_share", &"<redacted>")
            .field("public_key", &self.public_key)
            .finish()
       }
   }
   ```
2. Apply the same pattern to `TripleShare`, `PresignOutput` (both variants), `RerandomizedPresignOutput` (both variants), and `PresignArguments`.
3. Consider using the [`secrecy`](https://crates.io/crates/secrecy) crate to wrap scalar fields so that `Debug` redaction is enforced at the type level rather than per-struct.
4. Add a `clippy` lint or CI check that forbids `#[derive(Debug)]` on types containing `Scalar` or `SigningShare` fields.

### Proof of Concept

```rust
use threshold_signatures::{keygen, KeygenOutput, participants::Participant, ReconstructionLowerBound};
use rand_core::OsRng;
use frost_secp256k1::Secp256K1Sha256;

let participants = vec![Participant::from(0u32), Participant::from(1u32)];
let me = Participant::from(0u32);
// Run DKG and obtain KeygenOutput
let protocol = keygen::<Secp256K1Sha256>(&participants, me, 2usize, OsRng).unwrap();
// ... drive protocol to completion, obtaining `output: KeygenOutput<Secp256K1Sha256>`

// Any of the following common patterns leaks the raw private_share scalar:
println!("{:?}", output);                          // direct debug print
tracing::debug!(?output, "keygen complete");       // tracing instrumentation
let _ = format!("{output:?}");                     // string formatting
assert_eq!(output, expected);                      // test assertion failure message
```

For `PresignArguments`, the same applies at the `presign()` call site, where both `TripleShare` scalars and `private_share` are emitted together. The snapshot files already demonstrate that serialization of these types produces raw hex scalars (e.g., `"private_share": "aa4d3f7ff43da60bce9803cf12619a231f96fbf6e7e4b688bdd69aa8a35dd82e"`), confirming the `Debug` output is equally revealing. //! This module contains the types and protocols related to triple generation.
```rust
//!
//! The cait-sith signing protocol makes use of *committed* Beaver Triples.
//! A triple is a value of the form `(a, b, c), (A, B, C)`, such that
//! `c = a * b`, and `A = a * G`, `B = b * G`, `C = c * G`. This is a beaver
//! triple along with commitments to its values in the form of group elements.
//!
//! The signing protocols make use of a triple where the scalar values `(a, b, c)`
//! are secret-shared, and the commitments are public. Each signature requires
//! two triples. These triples can be generated in advance without knowledge
//! of the secret key used to sign. It's important that the value of the underlying
//! scalars in the triple is kept secret, otherwise the private key used to create
//! a signature with that triple could be recovered.
```
