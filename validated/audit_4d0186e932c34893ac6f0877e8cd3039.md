### Title
Missing Lower-Bound Validation on `threshold` in FROST `assert_sign_inputs` Allows Corrupted Signing Output — (File: `src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` function in `src/frost/mod.rs` validates that the threshold does not exceed the participant count but omits the minimum-threshold check (`threshold >= 2`) that is explicitly enforced during key generation. A caller can pass `threshold = 1` (or `0`), bypassing the security invariant, causing FROST signing to proceed with incorrect Lagrange coefficients and produce a cryptographically invalid, unusable signature.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` correctly enforces a lower bound:

```rust
// Step 1.1
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` only validates the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

There is no corresponding lower-bound check. `ReconstructionLowerBound` is a plain `usize` wrapper with no minimum enforced at the type level:

```rust
pub struct ReconstructionLowerBound(usize);
``` [3](#0-2) 

A caller can therefore pass `threshold = 1` (or `0`) to `assert_sign_inputs`. The function will accept it and return a valid `ParticipantList`, allowing the FROST signing protocol to proceed. The same gap exists in the FROST `presign` function, which also only checks the upper bound:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [4](#0-3) 

In contrast, the OT-based triple generation (`validate_triple_inputs`) and DKG both enforce `threshold >= 2`, making the omission in the FROST signing path an inconsistency with the rest of the library. [5](#0-4) 

---

### Impact Explanation

In FROST, each participant computes a signing share:

```
z_i = nonce_i + λ_i · s_i · challenge
```

where `λ_i` is the Lagrange coefficient evaluated at 0 using the set of `threshold` participants. When `threshold = 1`, the Lagrange coefficient for the single-point interpolation collapses to `λ_i = 1` for every participant, regardless of their actual identifier. The signing shares are then summed by the coordinator using these wrong coefficients, producing a scalar `s` that does not correspond to the correct reconstruction of the secret key. The resulting `(R, s)` signature fails verification against the public key.

This maps directly to the allowed High impact: **Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

`assert_sign_inputs` is a `pub` function and is the sole validation gate for FROST EdDSA and RedJubjub signing. A malicious coordinator who controls the signing session parameters can supply `threshold = 1` to honest participants' signing calls. Because the library performs no cross-check between the threshold used at key generation and the threshold supplied at signing time, honest participants have no way to detect the mismatch before the protocol runs. The attack requires no cryptographic capability — only the ability to invoke the public API with an unchecked integer argument.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already applies:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This check should be inserted in `assert_sign_inputs` (after the participant-count check) and in the FROST `presign` function, mirroring the pattern already used in `assert_key_invariants` and `validate_triple_inputs`. [6](#0-5) 

---

### Proof of Concept

1. Generate a key with `threshold = 2` and `N = 3` participants via the DKG (this succeeds because `assert_key_invariants` enforces `threshold >= 2`).
2. Call the FROST signing function with the same participants but pass `threshold = 1`.
3. `assert_sign_inputs` accepts the call: `1 > 3` is false (upper-bound check passes), and there is no lower-bound check.
4. Each participant computes `z_i = nonce_i + 1 · s_i · challenge` (Lagrange coefficient = 1).
5. The coordinator sums the shares and assembles a signature `(R, s)`.
6. Verification of `(R, s)` against the public key fails because `s` does not reconstruct the correct secret.
7. All honest parties receive an unusable, invalid signature — the signing output is permanently corrupted for this session.

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
