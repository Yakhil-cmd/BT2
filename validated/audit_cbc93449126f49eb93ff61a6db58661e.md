### Title
Missing Validation of Participant Contributions in CKD Coordinator Allows Output Corruption — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function aggregates `(norm_big_y, norm_big_c)` values received from participants without any proof of correct computation. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a corrupted CKD output that honest parties will accept as valid.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's contribution and blindly sums them: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is supposed to compute their contribution in `compute_signature_share` as: [2](#0-1) 

- `norm_big_y = lambda_i * y_i * G`
- `norm_big_c = lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)`

where `x_i` is their secret signing share. However, there is **no zero-knowledge proof** (e.g., a DLEQ proof) attached to the message sent to the coordinator verifying that `norm_big_c` was formed using the participant's actual secret share `x_i` and the same `y_i` used for `norm_big_y`. The coordinator performs no such check before aggregating: [3](#0-2) 

The `recv_from_others` helper simply delivers raw deserialized values with no cryptographic binding to the sender's public key share.

This is directly analogous to the reference vulnerability: just as the staking withdraw instruction accepted any `amount` without checking the staker's authorized balance, the CKD coordinator accepts any `(norm_big_y, norm_big_c)` pair without checking it is consistent with the sender's public key share.

---

### Impact Explanation

The final CKD output is:

```
C - app_sk * Y  =  msk * H(pk, app_id)  +  Δc - app_sk * Δy
```

where `(Δy, Δc)` is the malicious participant's deviation from their correct contribution. Because the attacker does not know `app_sk`, they cannot cancel the deviation, so the unmasked confidential key is permanently wrong. The coordinator and all honest parties accept this corrupted output as the protocol result with no indication of failure.

**Impact class**: High — Corruption of CKD outputs so honest parties accept an unusable cryptographic output.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attack requires only that the participant send arbitrary `ElementG1` values instead of their correct contribution. No special knowledge (private keys of others, `app_sk`, etc.) is needed. The protocol has no round that would expose the deviation to other participants before the coordinator finalizes the output.

---

### Recommendation

Each participant should attach a **DLEQ proof** (or equivalent Schnorr proof) to their message demonstrating that `norm_big_c` was formed using the same discrete-log relationship as `norm_big_y` and their committed public key share. Concretely, the coordinator should verify, for each participant `i` with public key share `X_i = x_i * G2`:

- `norm_big_y = lambda_i * y_i * G1` (binding `y_i` via the proof)
- `norm_big_c = lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)` (binding `x_i` to `X_i`)

Only after all proofs pass should the coordinator aggregate the contributions.

---

### Proof of Concept

1. Honest participants `{1, 2, 3}` run the CKD protocol with coordinator = participant 1.
2. Participant 3 (malicious) intercepts the `do_ckd_participant` call and instead of computing `compute_signature_share`, sends `(ElementG1::identity(), ElementG1::identity())` (or any arbitrary points) to the coordinator via the private channel.
3. The coordinator's loop at line 50–55 adds these values without rejection.
4. The final `CKDOutput` returned by `do_ckd_coordinator` is `(Y + Δy, C + Δc)` where `(Δy, Δc)` are the malicious deviations.
5. When the application calls `ckd_output.unmask(app_sk)`, the result is `msk * H(pk, app_id) + Δc - app_sk * Δy ≠ msk * H(pk, app_id)`.
6. The derived confidential key is permanently wrong; no error is raised by the protocol. [4](#0-3)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
