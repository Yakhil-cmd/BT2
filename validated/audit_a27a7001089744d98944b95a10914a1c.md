### Title
Unauthenticated `sender` Argument in Bank Precompile `transfer` and `burn` Allows Unauthorized Native Token Theft - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The bank precompile (`0x0000000000000000000000000000000000000064`) exposes `transfer(address sender, address recipient, uint256 amount)` and `burn(address addr, uint256 amount)` methods. Neither method verifies that the `sender`/`addr` argument matches the actual EVM caller (`contract.Caller()`). Any contract can therefore transfer or burn `evm/<callerContract>` native tokens from any victim address without authorization.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks a caller-supplied `sender` address and passes it directly to `bankKeeper.SendCoins` as the source of funds:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // denom = "evm/<callerContract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, ...)   // from = attacker-controlled
``` [1](#0-0) 

There is no guard of the form `if sender != contract.Caller() { return error }`. The denom is scoped to the calling contract (`evm/<callerContract>`), but the source address is fully attacker-controlled.

The same flaw exists in the `BurnMethodName` case: the `addr` argument (the address whose tokens are burned) is caller-supplied and never validated against `contract.Caller()`:

```go
recipient := args[0].(common.Address)   // address to burn FROM
...
addr := sdk.AccAddress(recipient.Bytes())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
``` [2](#0-1) 

### Impact Explanation

Any unprivileged EVM contract can:

1. Call `bank.transfer(victimAddress, attackerAddress, amount)` — transfers `evm/<maliciousContract>` native tokens from `victimAddress` to `attackerAddress` with no consent from the victim.
2. Call `bank.burn(victimAddress, amount)` — destroys `evm/<maliciousContract>` native tokens held by `victimAddress` with no consent.

Users who hold `evm/<contractAddress>` native tokens (obtained, for example, by calling `moveToNative` on a contract like `TestBank`) can have those tokens stolen or destroyed by the contract itself or by any contract that can invoke the bank precompile with the victim's address as the `sender` argument.

This is a **Critical** impact: unauthorized transfer and burn of precompile-controlled assets (`evm/` denom native tokens). [3](#0-2) 

### Likelihood Explanation

The bank precompile is callable by any EVM contract. No special privilege, key, or governance action is required. An attacker deploys a contract, mints tokens to victims (or waits for victims to acquire them), then calls `transfer` or `burn` with the victim's address as the `sender`/`addr` argument. The entry path is fully unprivileged and reachable on-chain. [4](#0-3) 

### Recommendation

Add a caller-authentication check in both the `TransferMethodName` and `BurnMethodName` cases. For `transfer`, enforce that `sender == contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender is not authenticated: transfer can only be initiated by the token holder")
}
```

For `burn`, enforce that `recipient == contract.Caller()` (i.e., a contract can only burn its own tokens, not another user's):

```go
if recipient != contract.Caller() {
    return nil, errors.New("burn target is not authenticated: can only burn caller's own tokens")
}
``` [5](#0-4) 

### Proof of Concept

1. Deploy `MaliciousContract` on Cronos.
2. Alice calls `MaliciousContract.moveToNative(100)` which internally calls `bank.mint(alice, 100)`, crediting Alice with 100 `evm/<MaliciousContract>` native tokens.
3. Attacker calls `MaliciousContract.steal(alice, attacker, 100)` which internally calls `bank.transfer(alice, attacker, 100)`.
4. The bank precompile executes `bankKeeper.SendCoins(alice, attacker, 100 evm/<MaliciousContract>)` — no authorization check is performed.
5. Alice's 100 native tokens are transferred to the attacker.

The same attack applies to `burn`: replace step 3–4 with `bank.burn(alice, 100)` to destroy Alice's tokens. [1](#0-0) [6](#0-5)
