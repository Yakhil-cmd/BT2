### Title
Gateway L2 Gas Price Admission Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's stateful admission check for L2 gas price compares a transaction's `max_price_per_unit` against the **previous block's execution price** (`block_header.l2_gas_price`), not the **EIP-1559-adjusted price for the next block** (`block_header.next_l2_gas_price`). This is the direct sequencer analog of the external report's stale-variable bug: a rate check is performed against an outdated value, causing the gateway to admit transactions that will fail at blockifier execution (when gas usage is above target and price is rising) and to reject transactions that are actually valid (when gas usage is below target and price is falling).

The code even contains a self-documenting TODO acknowledging the wrong field is being used.

### Finding Description

In `validate_resource_bounds`, the gateway reads the L2 gas price from the previous block's `BlockInfo`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;   // ← l2_gas_price of the *last committed* block
``` [1](#0-0) 

This value is `block_header.l2_gas_price.price_in_fri` — the price that was used to execute transactions **in** the previous block. The `GatewayFixedBlockSyncStateClient` builds `BlockInfo` from the block header and maps `block_header.l2_gas_price.price_in_fri` into `strk_gas_prices.l2_gas_price`, but never reads `block_header.next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
    // next_l2_gas_price is never exposed
},
``` [2](#0-1) 

The block header carries a distinct field `next_l2_gas_price` — the EIP-1559-adjusted price computed at block finalization that **will** be used for the next block:

```rust
pub struct BlockHeaderWithoutHash {
    pub l2_gas_price: GasPricePerToken,      // price used IN this block
    pub l2_gas_consumed: GasAmount,
    pub next_l2_gas_price: GasPrice,          // price for the NEXT block
    ...
}
``` [3](#0-2) 

`next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` using the EIP-1559 formula and stored in the block header at finalization time: [4](#0-3) 

It is also written into the `SyncBlock` that the gateway reads: [5](#0-4) 

The admission threshold is computed as:

```
threshold = (min_gas_price_percentage / 100) * previous_block_l2_gas_price
``` [6](#0-5) 

Because `previous_block_l2_gas_price` ≠ `next_l2_gas_price`, the threshold is wrong.

### Impact Explanation

**Case 1 — False admission (gas usage above target, price rising):**  
When the previous block consumed more gas than the EIP-1559 target, `next_l2_gas_price > l2_gas_price`. A transaction with `max_price_per_unit` in the range `[l2_gas_price, next_l2_gas_price)` passes the gateway check but will be rejected by the blockifier's `check_fee_bounds` with `MaxGasPriceTooLow`. The gateway admits a transaction that cannot be sequenced.

**Case 2 — False rejection (gas usage below target, price falling):**  
When the previous block consumed less gas than the target, `next_l2_gas_price < l2_gas_price`. A transaction with `max_price_per_unit` in the range `[next_l2_gas_price, l2_gas_price)` is valid for the next block but is rejected by the gateway with `GAS_PRICE_TOO_LOW`. A valid transaction is denied admission.

Both cases match the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The EIP-1559 price adjustment is applied every block. Any block whose gas consumption deviates from the target (the common case) produces a `next_l2_gas_price` that differs from `l2_gas_price`. No special attacker capability is required — any user submitting a transaction during a period of non-target gas usage will encounter the wrong admission decision. The `min_gas_price_percentage` config defaults to 100, making the threshold equal to the full stale price, maximizing the window of incorrect decisions. [7](#0-6) 

### Recommendation

In `validate_resource_bounds`, read `next_l2_gas_price` from the block header instead of `l2_gas_price`. This requires:

1. Extending `GatewayFixedBlockStateReader::get_block_info` (or adding a new method) to expose `block_header.next_l2_gas_price`.
2. In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, populate the returned value from `block_header.next_l2_gas_price` rather than `block_header.l2_gas_price.price_in_fri`.
3. Rename the local variable from `previous_block_l2_gas_price` to `next_block_l2_gas_price` and remove the TODO comment.

### Proof of Concept

1. Observe the last committed block has `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (high-usage block).
2. Submit a V3 transaction with `l2_gas.max_price_per_unit = P` (equal to the stale price, above the threshold `min_gas_price_percentage * P / 100 = P` when percentage = 100).
3. Gateway `validate_resource_bounds` computes `threshold = P`, sees `P >= P`, and admits the transaction.
4. The blockifier's `check_fee_bounds` computes `actual_gas_price = P'` (the next block's price from `BlockContext`), sees `P < P'`, and returns `ResourceBoundsError::MaxGasPriceTooLow`.
5. The transaction was admitted through the gateway but cannot be executed — the gateway's admission decision was wrong. [8](#0-7)
