### Title
Changing ETH Connector Contract Account Does Not Migrate Existing Balances, Permanently Freezing User Funds - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `set_eth_connector_contract_account` function updates the stored connector account ID but does not migrate existing NEP-141 token balances from the old connector to the new one. After the update, all engine connector operations are silently rerouted to the new (empty) connector, making every existing user balance permanently inaccessible through the engine interface.

---

### Finding Description

`set_eth_connector_contract_account` (connector.rs:418–438) writes a new connector `AccountId` to engine storage and returns immediately:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I, env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        ...
        set_connector_account_id(io, &args.account);          // ← only updates the pointer
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
``` [1](#0-0) 

Every user-facing connector operation — `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw` — resolves the target account at call time via `get_connector_account_id`, then forwards the call through `return_promise`:

```rust
fn return_promise<I: IO + PromiseHandler, E: Env>(
    mut io: I, env: &E, method: &str, args: Vec<u8>, deposit: Yocto,
) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
        target_account_id: get_connector_account_id(&io)?,   // ← live lookup
        ...
    };
    ...
}
``` [2](#0-1) 

For example, `withdraw` and `ft_transfer` both delegate entirely through this helper: [3](#0-2) [4](#0-3) 

Because `get_connector_account_id` reads the value that was just overwritten, **all subsequent calls are forwarded to the new connector account**. The old connector account — which holds every user's NEP-141 ETH balance — is no longer reachable through any engine entrypoint. No balance migration, no drain-first requirement, and no guard against a non-zero balance on the old account exist anywhere in `set_eth_connector_contract_account`. [5](#0-4) 

---

### Impact Explanation

All bridged ETH balances held by users in the old connector account become permanently inaccessible through the Aurora Engine. Users cannot `withdraw`, `ft_transfer`, or perform any NEP-141 operation on their funds because every such call is now dispatched to the new connector, which starts with zero balances. This constitutes a **permanent freezing of funds** (Critical).

---

### Likelihood Explanation

The function is callable by the engine owner or via a private (self) call. A legitimate upgrade or migration of the ETH connector contract — a plausible operational action — is sufficient to trigger the freeze. No malicious intent is required; the design flaw activates on any invocation that switches to a non-empty-predecessor connector account.

---

### Recommendation

Before accepting a new connector account, `set_eth_connector_contract_account` should either:

1. **Require the old connector to have a zero total supply** (i.e., no outstanding user balances) before the switch is permitted, or
2. **Initiate a balance migration** — transferring all NEP-141 balances from the old connector to the new one atomically as part of the setter — before updating the stored account ID.

---

### Proof of Concept

1. Users deposit ETH via the bridge; the old connector account accumulates NEP-141 balances for each user.
2. The engine owner calls `set_eth_connector_contract_account` with a freshly deployed connector account (e.g., during an upgrade).
3. `set_connector_account_id` overwrites the stored account ID; no balance migration occurs.
4. A user calls `withdraw` or `ft_transfer` on the engine. `return_promise` resolves `get_connector_account_id()` to the **new** connector, which has no record of the user's balance.
5. The call fails or returns zero; the user's funds remain locked in the old connector account, which is no longer reachable through any engine entrypoint.
6. All user funds are permanently frozen.

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(

        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_connector_account_id<I: IO>(io: &I) -> Result<AccountId, ContractError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(errors::ERR_CONNECTOR_STORAGE_KEY_NOT_FOUND)
    .and_then(|x| {
        x.to_value()
            .map_err(|_| errors::ERR_BORSH_DESERIALIZE.as_bytes())
    })
    .map_err(Into::into)
}

pub fn set_connector_account_id<I: IO + Copy>(mut io: I, account_id: &AccountId) {
    io.write_borsh(
        &construct_contract_key(EthConnectorStorageId::EthConnectorAccount),
        account_id,
    );
}
