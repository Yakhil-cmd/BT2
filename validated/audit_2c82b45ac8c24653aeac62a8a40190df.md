### Title
`tx_execution_output_to_fee_estimation` Returns `overall_fee` That Includes L2 Gas Cost But Omits `l2_gas_consumed`, Making the Fee Irreconcilable with the Documented Formula — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`tx_execution_output_to_fee_estimation` populates `FeeEstimation::overall_fee` from `receipt.fee`, which is computed by `GasVector::cost` and includes the L2 gas component (`l2_gas * l2_gas_price`). However, the returned struct exposes only `gas_consumed` (L1 gas) and `data_gas_consumed` (L1 data gas) — there is no `l2_gas_consumed` field. The struct's own docstring and the RPC OpenAPI spec both declare `overall_fee = gas_consumed * gas_price + data_gas_consumed * data_gas_price`. Any v3 invoke transaction that consumes nonzero L2 gas under a block context with nonzero `l2_gas_price` produces a response where `overall_fee` is strictly greater than that formula, and clients have no way to reconstruct or verify the correct value from the returned fields.

---

### Finding Description

**`tx_execution_output_to_fee_estimation`** (objects.rs lines 161–183) builds the `FeeEstimation` struct as follows: [1](#0-0) 

```
gas_consumed       = gas_vector.l1_gas
data_gas_consumed  = gas_vector.l1_data_gas
overall_fee        = receipt.fee
```

`receipt.fee` is computed in `TransactionReceipt::from_params` (receipt.rs lines 114–118) by calling `get_fee_by_gas_vector`, which delegates to `GasVector::cost`: [2](#0-1) 

`GasVector::cost` (execution_resources.rs lines 156–186) sums all three gas components: [3](#0-2) 

```
fee = l1_gas * l1_gas_price
    + l1_data_gas * l1_data_gas_price
    + l2_gas * (l2_gas_price + tip)
```

So `overall_fee` includes the L2 gas cost. But the `FeeEstimation` struct's docstring says: [4](#0-3) 

> "The total amount of fee. This is equal to: gas_consumed * gas_price + data_gas_consumed * data_gas_price."

And the RPC OpenAPI spec repeats this invariant: [5](#0-4) 

> "equals to gas_consumed\*gas_price + data_gas_consumed\*data_gas_price"

The `l2_gas_price` field is present in the struct but `l2_gas_consumed` is absent. The developer TODO at lines 104–105 explicitly acknowledges this gap: [6](#0-5) 

> `// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.`

For any v3 transaction using `AllResourceBounds` (the standard modern format) that consumes nonzero L2 gas, the discrepancy is:

```
overall_fee - (gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price)
= l2_gas_consumed * l2_gas_price   (> 0)
```

---

### Impact Explanation

Clients (wallets, dApps, block explorers) that call `starknet_estimateFee` or `starknet_simulateTransactions` receive a `FeeEstimation` where `overall_fee` cannot be reconstructed from the other returned fields using the documented formula. The `l2_gas_price` is returned but `l2_gas_consumed` is not, so the client cannot even compute the discrepancy. This is an authoritative-looking wrong decomposition: the total is correct but the breakdown is incomplete, violating the spec invariant. Any client that validates `overall_fee` against the formula (e.g., for fee transparency or anti-manipulation checks) will incorrectly flag valid estimates as wrong, or silently accept an unverifiable fee.

---

### Likelihood Explanation

This triggers for every v3 transaction with `AllResourceBounds` that consumes L2 gas — which is the standard path for all modern Starknet transactions. The `GasVectorComputationMode::All` path (used for v3 transactions) routes computation costs into `l2_gas` rather than `l1_gas`. The condition is met by any normal user submitting a v3 invoke transaction; no special privileges or crafted inputs are required.

---

### Recommendation

Add `l2_gas_consumed: gas_vector.l2_gas.0.into()` to the `FeeEstimation` struct and populate it in `tx_execution_output_to_fee_estimation`. Update the struct docstring and the RPC OpenAPI spec description to reflect the correct three-component formula:

```
overall_fee = gas_consumed * l1_gas_price
            + data_gas_consumed * l1_data_gas_price
            + l2_gas_consumed * l2_gas_price
```

---

### Proof of Concept

Construct a `FeeEstimation` from a v3 invoke execution with nonzero L2 gas and assert the invariant violation:

```rust
// block context: l2_gas_price = 1000 (nonzero)
// transaction: consumes l2_gas = 500, l1_gas = 10, l1_data_gas = 0
// receipt.fee = 10 * l1_gas_price + 500 * 1000 = 10 * l1_gas_price + 500_000

let estimation = tx_execution_output_to_fee_estimation(&output, &block_context).unwrap();

let reconstructed = estimation.gas_consumed.to_u128().unwrap()
    * estimation.l1_gas_price.get()
    + estimation.data_gas_consumed.to_u128().unwrap()
    * estimation.l1_data_gas_price.get();

// overall_fee includes l2 gas cost; reconstructed does not
assert_ne!(estimation.overall_fee.0, reconstructed);
// overall_fee > reconstructed by exactly l2_gas_consumed * l2_gas_price
```

The assertion passes (they are unequal), confirming the invariant stated in the docstring and RPC spec is violated for any v3 transaction consuming L2 gas.

        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
