**How ETH gets stranded:** A user calls `addLiquidityExactShares{value: X}` where `X > amount_needed`. The `pay` function consumes only `amount_needed` ETH; the remainder stays in the adder. **The attack:**
1. Alice calls `addLiquidityExactShares{value: 2 ETH}` for a pool needing 1 ETH. 1 ETH remains in the adder.
2. Alice does not call `refundETH` (separate tx, or simply forgot).
3. Bob calls `addLiquidityExactShares` (no ETH sent) for a WETH pool needing 2 WETH.
4. In the callback, `pay(WETH, bob, pool, 2e18)` fires. `nativeBalance = 1 ETH > 0`, so the adder wraps 1 ETH → WETH → pool, then pulls only 1 WETH from Bob via `safeTransferFrom`.
5. Bob receives full liquidity (2 WETH worth). Alice loses 1 ETH.

**Why the `receive()` guard does not help:** `receive()` blocks plain ETH transfers from non-WETH addresses, but ETH sent as `msg.value` in a function call bypasses `receive()` entirely. So ETH from `addLiquidityExactShares{value: X}` accumulates in the contract balance without restriction. [4](#0-3) 

**Does the callback guard prevent this?** No. The callback validates the caller is the expected pool and that amounts are within caps, but it does not check whether the ETH used in `pay` originated from the current transaction. [5](#0-4)

### Title
Stranded ETH in `MetricOmmPoolLiquidityAdder` is silently consumed by any subsequent WETH liquidity add, causing direct loss of funds for the original depositor — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay` uses `address(this).balance` globally, not scoped to the current transaction's `msg.value`. ETH left in the adder from a prior transaction (by any user who did not call `refundETH`) is automatically consumed to fund any subsequent WETH payment, transferring that ETH to a different user's liquidity position.

### Finding Description
When `pay(token=WETH, payer, pool, value)` is called during `metricOmmModifyLiquidityCallback`, it reads the adder's full native balance:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
```

There is no mechanism to distinguish ETH from the current `msg.value` versus ETH stranded from a previous call. Any ETH remaining in the contract (because a prior user sent excess ETH and did not call `refundETH`) is silently consumed for the current payer's benefit.

### Impact Explanation
Direct loss of user principal. The user who left ETH in the adder loses it permanently; it is wrapped and transferred to a pool as part of a different user's liquidity deposit. The attacker (subsequent caller) receives full liquidity while paying less WETH than owed.

### Likelihood Explanation
The `payable` entry points (`addLiquidityExactShares`, `addLiquidityWeighted`) accept ETH directly. Users who overshoot `msg.value` and omit `refundETH` — a common pattern when not using multicall — will strand ETH. Any other user can then exploit it by calling a WETH-pool liquidity add with no ETH sent.

### Recommendation
Track the ETH balance at the start of each top-level call (e.g., store `msg.value` in transient storage alongside the pay context) and limit `pay` to consuming only that amount. Alternatively, enforce that `pay` only uses ETH equal to `msg.value` of the current call, reverting if the contract holds residual ETH from prior transactions.

### Proof of Concept
```solidity
// 1. Alice sends excess ETH, forgets refundETH
adder.addLiquidityExactShares{value: 2 ether}(pool, alice, 0, delta, 2 ether, 0, "");
// Pool only needed 1 ETH; 1 ETH remains in adder.

// 2. Bob exploits stranded ETH
uint256 bobWethBefore = weth.balanceOf(bob);
vm.prank(bob);
adder.addLiquidityExactShares(pool, bob, 1, delta2, 2 ether, 0, ""); // no ETH sent

// Assert: adder used Alice's 1 ETH + pulled 1 WETH from Bob
assertEq(address(adder).balance, 0);                          // Alice's ETH consumed
assertEq(bobWethBefore - weth.balanceOf(bob), 1 ether);       // Bob paid only 1 WETH
// Bob received 2 WETH worth of liquidity
```
