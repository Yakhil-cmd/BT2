### Title
Division-Before-Multiplication in `native_per_pubdata` Computation Causes Systematic Pubdata Native-Resource Undercharging — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The bootloader computes `native_per_pubdata` by performing integer division of `pubdata_price / native_price` first, then multiplying the truncated quotient by the actual pubdata bytes consumed. This division-before-multiplication ordering causes a systematic floor-rounding loss: every transaction that produces pubdata pays fewer native resources than the protocol intends, allowing a sender to consume pubdata at a discount relative to the configured price ratio.

---

### Finding Description

In the ZK-transaction validation path, `native_per_pubdata` is derived as:

```rust
// validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This truncated-integer quotient is then used in two downstream multiplications:

**1. Intrinsic pubdata overhead deduction** (inside `create_resources_for_tx`):

```rust
// gas_helpers.rs line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
``` [2](#0-1) 

**2. Runtime pubdata charging** (inside `get_resources_to_charge_for_pubdata`):

```rust
// gas_helpers.rs lines 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

The same pattern is replicated in the public API helper:

```rust
// api/src/helpers.rs line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
// line 438
let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
``` [4](#0-3) 

The mathematically correct order is:

```
native_cost = (pubdata_price × pubdata_bytes) / native_price
```

The current order is:

```
native_cost = floor(pubdata_price / native_price) × pubdata_bytes
```

The difference is `(pubdata_price mod native_price) / native_price` native units **per pubdata byte**, which is always in `[0, 1)`. Over `N` pubdata bytes the total underpayment is bounded by `N` native units. Critically, when `pubdata_price < native_price` the quotient floors to **zero**, meaning the sender pays **no** native resources for pubdata at all regardless of how many bytes are written.

Contrast with `native_per_gas`, which correctly uses `div_ceil` (rounds up, conservative):

```rust
// validation_impl.rs line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [5](#0-4) 

The asymmetry — ceiling for gas, floor for pubdata — is inconsistent and systematically favours the sender on the pubdata dimension.

---

### Impact Explanation

**Resource accounting bug / valid-execution unprovability boundary.**

- Every L2 transaction that writes pubdata pays up to `pubdata_bytes` fewer native units than the protocol price ratio demands.
- When `pubdata_price < native_price` (a ratio the operator can legitimately set during low-cost periods), `native_per_pubdata` collapses to `0` and the sender's native budget is never reduced by pubdata consumption at all. A sender can then write the maximum allowed pubdata per transaction at zero native cost, exhausting the block's pubdata capacity without proportional resource expenditure.
- The `withheld` resource pool (excess native above `MAX_NATIVE_COMPUTATIONAL`) that is specifically reserved for pubdata payment is computed from the same underestimated `native_per_pubdata`, so the safety valve is also undersized. [6](#0-5) 

---

### Likelihood Explanation

The entry path is fully unprivileged: any L2 transaction sender can submit a transaction that writes storage (pubdata). The rounding loss occurs on every such transaction unconditionally. The `pubdata_price < native_price` degenerate case (zero charge) is reachable whenever the operator configures those block-context values in that ratio, which is a normal operational scenario (e.g., cheap L1 data availability periods).

---

### Recommendation

Replace the division-first pattern with multiplication-first at the point of actual charging. Pass `pubdata_price` and `native_price` separately into `get_resources_to_charge_for_pubdata` and compute:

```rust
let native = current_pubdata_spent
    .checked_mul(pubdata_price)
    .and_then(|v| Some(v / native_price))
    ...
```

Or, if the intermediate `native_per_pubdata` scalar must be retained for interface compatibility, use `div_ceil` (matching `native_per_gas`) so the rounding is at least conservative:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427. [7](#0-6) 

---

### Proof of Concept

**Setup:** `pubdata_price = 7`, `native_price = 10`, transaction writes `1000` pubdata bytes.

**Current behaviour:**
```
native_per_pubdata = floor(7 / 10) = 0
native_charged     = 0 × 1000     = 0   ← sender pays nothing for pubdata
```

**Correct behaviour:**
```
native_charged = (7 × 1000) / 10 = 700
```

The sender avoids 700 native units of pubdata cost entirely. Scaled to the maximum pubdata per block, an attacker can fill the block's pubdata quota while paying zero native resources for it, degrading the economic accounting that the native-resource system is designed to enforce.
