Audit Report

## Title
Unbounded Loop Over All Proposal Ballots and Neurons in `distribute_rewards` Causes Instruction-Limit DoS of SNS Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary

The SNS Governance canister's `distribute_rewards` function performs two unbounded nested loops — one over all `ReadyToSettle` proposals × their full ballot maps, and one over all voting neurons — in a single synchronous heartbeat execution with no instruction-limit guard. If the combined work exceeds the IC's 40B-instruction per-message limit, the heartbeat traps, `latest_reward_event` is never updated, and the same proposals remain `ReadyToSettle` on the next heartbeat, creating a permanent DoS loop that permanently breaks voting reward distribution for the SNS.

## Finding Description

`distribute_rewards` at [1](#0-0)  is called synchronously from `run_periodic_tasks` (the canister heartbeat) at [2](#0-1) .

The function first collects all `ReadyToSettle` proposals with no cap: [3](#0-2) 

**Loop 1** iterates over every proposal × every ballot entry in that proposal's `HashMap<String, Ballot>`, performing string parsing and Decimal arithmetic per entry: [4](#0-3) 

**Loop 2** iterates over every neuron that voted, calling `get_neuron_result_mut` and mutating maturity: [5](#0-4) 

A grep for `instruction_counter`, `is_over_instructions_limit`, or any instruction-limit guard in `rs/sns/governance/src/governance.rs` returns zero matches. There is no batching, no early-break, and no cursor to resume from.

By contrast, NNS governance already has a `RewardsDistributionInProgress` state machine with an explicit `is_over_instructions_limit` guard that breaks and resumes across ticks: [6](#0-5) 

The IC's per-message instruction limit is confirmed at: [7](#0-6) 

If `distribute_rewards` traps mid-execution, all state mutations are rolled back. The `latest_reward_event` is written only at the very end of the function (after both loops), so it is never updated. On the next heartbeat, `should_distribute_rewards` returns `true` again, `distribute_rewards` is called again with the same `ReadyToSettle` proposals, and it traps again — indefinitely.

## Impact Explanation

Permanent DoS of SNS voting reward distribution: no neuron maturity is ever increased, SNS token holders permanently lose voting rewards, and the SNS governance heartbeat is permanently degraded for this task. This matches the allowed High impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Application/platform-level DoS … not based on raw volumetric DDoS."* The SNS cannot self-recover without a canister upgrade.

## Likelihood Explanation

Medium. An unprivileged SNS neuron holder (no admin key or governance majority required) can submit proposals by paying the proposal fee. Proposals enter `ReadyToSettle` automatically after their voting period expires regardless of outcome. With thousands of neurons from swap participation and tens of settled proposals accumulating in a single reward round, the O(proposals × neurons) work can exceed 40B instructions. The attack is repeatable and self-sustaining once triggered.

## Recommendation

Apply the same batched-distribution pattern already used in NNS governance (`rs/nns/governance/src/reward/distribution.rs`):

1. After computing `neuron_id_to_reward_shares`, persist it to stable storage as a `RewardsDistributionInProgress`-equivalent structure.
2. In each heartbeat/timer tick, process a bounded chunk of neurons, checking `ic_cdk::api::instruction_counter()` after each mutation and breaking early if the limit is approached.
3. Resume from the saved cursor on the next tick until distribution is complete, then write the `latest_reward_event`.

Additionally, consider capping the number of proposals that can simultaneously be in `ReadyToSettle` state to bound worst-case work per round.

## Proof of Concept

1. Deploy an SNS with N neurons (e.g., N = 10,000 via swap participation).
2. Submit M proposals in rapid succession (e.g., M = 50), each paying the proposal fee. Neurons auto-vote via following, so each proposal accumulates N ballots.
3. Wait for all proposals' voting periods to expire, placing them all in `ReadyToSettle` state.
4. On the next reward round, `distribute_rewards` is invoked. It must process M × N = 500,000 ballot entries (string parsing + Decimal arithmetic each) plus N neuron maturity mutations. Total instruction cost exceeds 40B.
5. The heartbeat traps; `latest_reward_event` is not updated; all M proposals remain `ReadyToSettle`.
6. Every subsequent heartbeat repeats step 4–5. Reward distribution is permanently broken.

A deterministic integration test can reproduce this by instantiating an SNS with a large neuron count via PocketIC, submitting many proposals, advancing time past their voting periods, and asserting that `distribute_rewards` traps (instruction limit exceeded) and that `latest_reward_event` is unchanged across multiple heartbeat invocations.
