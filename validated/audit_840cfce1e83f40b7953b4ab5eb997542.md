The code is clear. Let me confirm the exact mismatch between `target_shard_index` (computed from E1's layout) and `outcome_roots` (from E2's block after resharding).

### Title
Stale `target_shard_index` from pre-split epoch used to index post-split `outcome_roots`, producing a wrong `outcome_root_proof` — (`chain/client/src/view_client_actor.rs`)

---

### Summary

`ViewClientActor::handle(GetExecutionOutcome)` computes `target_shard_index` from the **old epoch's** shard layout, then calls `get_next_block_hash_with_new_chunk`, which can cross a resharding boundary and return a child shard ID belonging to the **new epoch's** layout. The stale index is then used without recomputation to select an entry from `outcome_roots` built from the new-epoch block, silently producing a Merkle proof that commits to the **wrong shard's** outcome root.

---

### Finding Description

The handler in `view_client_actor.rs` proceeds as follows:

**Step 1 — index computed from old epoch (E1):** [1](#0-0) 

`epoch_id` and `shard_layout` are taken from the block that contains the outcome (epoch E1). `target_shard_index` is the position of `target_shard_id` inside E1's shard layout and is never updated again.

**Step 2 — `get_next_block_hash_with_new_chunk` may cross a resharding boundary:** [2](#0-1) 

When the walk crosses into a new epoch with a different shard layout, `shard_ids` is replaced with the children of the original shard in the new layout, and the function returns `(block_hash, child_shard_id)` where `child_shard_id` belongs to E2.

**Step 3 — stale index used against new-epoch `outcome_roots`:** [3](#0-2) 

The destructured `target_shard_id` is now a child shard in E2, but `target_shard_index` still holds the E1 value. `outcome_roots` is collected from block `h` (in E2, which has a different number of shards and a different `shard_id → shard_index` mapping). The bounds check at line 1174 only catches the case where the stale index is ≥ the new length; it does **not** detect the case where the index is in-bounds but points to the wrong shard. Line 1182 then emits a Merkle proof rooted at the wrong shard's outcome root.

**Why `shard_id ≠ shard_index` makes this concrete:**

In V2/V3 layouts, shard IDs and shard indices are explicitly decoupled: [4](#0-3) 

A concrete example using the V2 test layout (shard_ids = `[3, 8, 4, 7]`, indices `0, 1, 2, 3`): parent shard 1 (V1, index 1) splits into children `[7, 8]` (V2 indices 3 and 1 respectively). If `get_next_block_hash_with_new_chunk` returns child shard 7, the correct index is 3, but the stale `target_shard_index` is 1 — so the proof is built from `outcome_roots[1]` (shard 8's root) instead of `outcome_roots[3]` (shard 7's root). [5](#0-4) 

---

### Impact Explanation

The `outcome_root_proof` field of `GetExecutionOutcomeResponse` is a `MerklePath` that is supposed to prove that the outcome root for the correct shard is committed to in the block header. When `target_shard_index` is stale, the path proves membership of a **different shard's** outcome root. Any light client, bridge, or relayer that verifies this proof against the block's `outcome_root` will find it invalid (or, worse, will accept a proof that commits to the wrong shard's state). This breaks the core proof-of-execution invariant that nearcore's RPC is expected to uphold.

---

### Likelihood Explanation

The bug is triggered by any unprivileged user who:
1. Submits a transaction (or receipt) that executes in epoch E1, and
2. Queries `GetExecutionOutcome` after a shard split has occurred (epoch E2).

Both actions require only standard RPC access. The resharding boundary is a scheduled, protocol-level event, so the window is predictable and repeatable.

---

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` using the new block's epoch shard layout before indexing into `outcome_roots`:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout =
        self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... build outcome_roots and proof as before
}
```

The stale `target_shard_index` computed at line 1155 should not be used inside the `if let Some(...)` branch at all.

---

### Proof of Concept

A test-loop test spanning a V1→V2 (or V2→V3) shard-split epoch boundary:

1. Configure a chain with a scheduled resharding (e.g., shard 1 splits into children 7 and 8 in V2).
2. Submit a transaction from an account that maps to shard 1 in E1; wait for it to be included and finalized.
3. Advance the chain past the epoch boundary so E2 is active.
4. Call `GetExecutionOutcome` with the transaction hash.
5. Assert that `outcome_root_proof` verifies correctly: reconstruct `outcome_roots` from the returned `block_hash`, compute `merklize(&outcome_roots)`, and check that the proof path for the **child shard** (7 or 8) validates against the block's outcome root.
6. Without the fix, the proof will be built from the wrong index and verification will fail (or succeed against the wrong root).
