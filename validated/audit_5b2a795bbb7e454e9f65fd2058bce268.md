### Title
Unrestricted `disable_rewards` Parameter Allows Any Caller to Permanently Freeze Any Staker's Consensus Rewards — (File: src/staking/staking.cairo)

---

### Summary
The public `update_rewards` function accepts a caller-controlled `disable_rewards` flag with no access control. Because the global `last_reward_block` is written unconditionally before the `disable_rewards` early-return, any unprivileged address can call `update_rewards(victim, disable_rewards: true)` once per block to consume the per-block reward slot without distributing rewards, permanently starving a target staker of consensus yield.

---

### Finding Description
`update_rewards` is exposed on the `IStakingRewardsManager` ABI with two caller-supplied parameters: `staker_address` and `disable_rewards`. The only gate is `general_prerequisites()`, which checks the contract is not paused and the caller is non-zero — no check that the caller is the staker, the staker's reward address, or any privileged role.

The critical ordering inside the function is:

```
self.last_reward_block.write(current_block_number);   // ← always written

if disable_rewards || self.is_pre_consensus() {
    return;                                            // ← exits before _update_rewards
}
```

`last_reward_block` is a **single global slot** (not per-staker). The re-entry guard at the top of the function is:

```
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

Once an attacker writes `last_reward_block` for block N with `disable_rewards: true`, no further call to `update_rewards` can succeed in block N — for any staker. The victim's legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

The attacker repeats this at the first transaction of every block. Because the victim has no way to front-run the attacker in every block, the victim's consensus rewards are permanently frozen for as long as the attack is sustained.

---

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

The victim staker accrues zero consensus-phase block rewards for every block in which the attacker fires first. There is no recovery path: the victim cannot call `update_rewards` again in the same block, and the missed block reward is never retroactively credited. Sustained over many blocks, this constitutes permanent loss of yield.

---

### Likelihood Explanation
**High.** The function is permissionlessly callable by any non-zero address. The attacker needs only to submit one transaction per block (a low, predictable gas cost). A competing staker has a direct economic incentive: by suppressing the victim's rewards, the attacker's own proportional share of the total reward pool grows. No privileged access, leaked key, or external dependency is required.

---

### Recommendation
Restrict who may supply `disable_rewards: true`. Options include:
1. Require `get_caller_address() == staker_address || get_caller_address() == staker_info.reward_address` before honouring `disable_rewards: true`.
2. Move the `disable_rewards` logic into a separate, role-gated function and make the public `update_rewards` always distribute rewards.
3. Write `last_reward_block` only after the `disable_rewards` guard, so a disabled call does not consume the block slot.

---

### Proof of Concept

```
// Attacker (any address) executes once per block:
staking_contract.update_rewards(
    staker_address: victim_staker,
    disable_rewards: true,
);
// Effect:
//   last_reward_block ← current_block_number   (line 1485)
//   returns early, no rewards distributed       (line 1487)
//
// Victim's subsequent call in the same block:
staking_contract.update_rewards(
    staker_address: victim_staker,
    disable_rewards: false,
);
// Reverts: current_block_number == last_reward_block  (lines 1453-1458)
//          → REWARDS_ALREADY_UPDATED
```
