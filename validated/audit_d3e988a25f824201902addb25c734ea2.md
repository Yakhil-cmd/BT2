### Title
`StatelessTransactionValidator::validate_resource_bounds` unconditionally applies `min_gas_price` to `l2_gas.max_price_per_unit` for all transaction types, incorrectly rejecting valid L1-gas-only transactions at gateway admission — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces a `min_gas_price` floor on `l2_gas.max_price_per_unit` for every transaction type (Declare, DeployAccount, Invoke) without first checking whether the transaction actually uses L2 gas. A transaction that legitimately specifies only L1 gas bounds — with `l2_gas.max_price_per_unit = 0` and `l2_gas.max_amount = 0` — passes the `ZeroResourceBounds` guard (because L1 fee is non-zero) but is then unconditionally rejected with `MaxGasPriceTooLow`. Real on-chain transactions (e.g., `deploy_account_v3` with `l2_gas.max_price_per_unit = 0x0`) demonstrate this is a live pattern.

### Finding Description

In `validate_resource_bounds`:

```rust
// line 71-76
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

This check fires for **every** transaction regardless of whether `l2_gas.max_amount` is zero. The production default is `min_gas_price = 8_000_000_000`. [1](#0-0) 

The `ZeroResourceBounds` guard immediately above it only requires that *some* resource bound is non-zero, so an L1-only transaction (l1_gas non-zero, l2_gas all-zero) passes that guard and then hits the unconditional L2 price floor. [2](#0-1) 

The test suite exposes the contradiction: `test_positive_flow` includes a `valid_l1_gas` case that sets `l2_gas = Default::default()` (all zeros) and expects `Ok(())`, but it uses `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` which sets `min_gas_price = 0` to avoid the failure. Under the production config (`min_gas_price = 8_000_000_000`) the same transaction is rejected. [3](#0-2)

A real archived transaction (`deploy_account_v3`) from the replay corpus has `l2_gas.max_price_per_unit = 0x0`, confirming this is not a theoretical edge case: [5](#0-4) 

The analogous `max_l2_gas_amount` check correctly exempts Declare transactions with a type guard, but no such guard exists for the `min_gas_price` check: [6](#0-5) 

### Impact Explanation

**High — Mempool/gateway admission rejects valid transactions before sequencing.**

Any transaction that specifies only L1 (or L1-data) gas bounds with `l2_gas.max_price_per_unit = 0` is unconditionally rejected at the stateless validation stage. The transaction never reaches the mempool or blockifier. Because the check is stateless and deterministic, every such submission fails permanently with no recourse for the sender.

### Likelihood Explanation

**High.** The `AllResourceBounds` structure explicitly supports independent L1/L2/L1-data gas fields. The existing test corpus and the archived `deploy_account_v3` transaction both demonstrate that `l2_gas.max_price_per_unit = 0` is a real, valid submission pattern. Any wallet or SDK that constructs a V3 transaction using only L1 gas bounds will trigger this rejection on the production gateway.

### Recommendation

Gate the `min_gas_price` check on whether the transaction actually allocates L2 gas:

```rust
if resource_bounds.l2_gas.max_amount.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

This mirrors the existing pattern used for `max_l2_gas_amount` (which already skips Declare) and aligns with the intent expressed in the `valid_l1_gas` positive test case.

### Proof of Concept

1. Configure the gateway with `validate_resource_bounds = true` and `min_gas_price = 8_000_000_000` (production defaults).
2. Submit any V3 transaction (Invoke, DeployAccount, or Declare) with:
   ```
   l1_gas.max_amount        = 0x18          (non-zero)
   l1_gas.max_price_per_unit = 0x5e95b9f35336 (non-zero)
   l2_gas.max_amount        = 0x0
   l2_gas.max_price_per_unit = 0x0
   ```
3. `ZeroResourceBounds` passes because `l1_gas` fee > 0.
4. `min_gas_price` check fires: `0 < 8_000_000_000` → `Err(MaxGasPriceTooLow)`.
5. The transaction is rejected at the stateless gateway before reaching the mempool, despite being a structurally valid transaction with sufficient L1 gas to cover execution.
