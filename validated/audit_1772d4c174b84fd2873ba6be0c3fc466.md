### Title
Missing Session ID in Triple Generation Transcript Enables Cross-Session Proof Replay — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

The `create_transcript` function in the OT-based ECDSA Beaver triple generation protocol constructs a Fiat-Shamir transcript that binds dlog proofs only to the static public parameters (participant list and threshold), with no session-unique identifier. A malicious participant can replay their polynomial commitment and dlog proof verbatim from a prior triple generation session into any future session sharing the same participant set and threshold, silently contributing the same secret polynomial across sessions and producing correlated Beaver triples.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/triples/generation.rs`, `create_transcript` builds the Fiat-Shamir context used for the dlog proofs (`phi_proof0`, `phi_proof1`) that bind each participant to their polynomial commitments:

```rust
fn create_transcript(
    participants: &ParticipantList,
    threshold: ReconstructionLowerBound,
) -> Result<Transcript, ProtocolError> {
    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);
    transcript.message(b"group", NAME);
    let enc = rmp_serde::encode::to_vec(participants)...;
    transcript.message(b"participants", &enc);
    transcript.message(
        b"threshold",
        &u64::try_from(threshold.value())...to_be_bytes(),
    );
    Ok(transcript)
}
``` [1](#0-0) 

No session-unique value is included. The transcript is fully determined by the static public parameters. Contrast this with the DKG protocol, which explicitly generates per-session randomness: each participant generates a random 32-byte `sid_i`, broadcasts it, and the combined `sid = H(sid_1, …, sid_N)` is incorporated into every challenge computation. [2](#0-1) 

The dlog proof verifier in `src/crypto/proofs/dlog.rs` derives its Fiat-Shamir challenge from the transcript plus the statement (commitment) plus the nonce commitment `R`:

```rust
pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError> {
    transcript.message(NEAR_DLOG_STATEMENT_LABEL, &statement.encode()?);
    let big_k = C::Group::generator() * proof.s.0 - *statement.public * proof.e.0;
    let ser = C::Group::serialize(&big_k)...;
    transcript.message(NEAR_DLOG_COMMITMENT_LABEL, ser.as_ref());
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOG_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, TranscriptRng>(&mut rng);
    Ok(e == proof.e.0)
}
``` [3](#0-2) 

Because the transcript context is identical across all sessions with the same participants and threshold, a malicious participant P_m can execute the following replay:

1. **Session 1**: P_m generates polynomial commitment `X_m = g^{x_m}` and proof `(R_m, e_m, s_m)`. The challenge `e_m = H(transcript, X_m, R_m)` is computed and the proof is broadcast in the `PolynomialCommitmentsMessage`. [4](#0-3) 

2. **Session 2** (same participants, same threshold): P_m submits the identical `PolynomialCommitmentsMessage` — same `big_e`, `big_f`, `big_l`, `randomizer`, `phi_proof0`, `phi_proof1`.

3. The challenge recomputed during verification is `H(transcript, X_m, R_m)` — identical to session 1 — so `e == proof.e` holds and the proof passes.

P_m has now contributed the same secret polynomial `f_{k_m}` to both sessions without detection. Honest participants have no way to distinguish a replayed message from a freshly generated one.

---

### Impact Explanation

Correlated Beaver triples violate the independence assumption required for the security of the OT-based ECDSA presigning protocol. The library's own security documentation explicitly warns about the consequence of nonce-share correlation:

> If different subsets of size at least 2t + 1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks. [5](#0-4) 

When P_m reuses polynomial `f_{k_m}`, their nonce share `k_m` is identical across sessions. The aggregate nonces `k^{(1)}` and `k^{(2)}` differ only by the contributions of honest participants, but P_m knows the fixed offset `k_m` in both. Combined with the public signature outputs and nonce points `R^{(1)}`, `R^{(2)}` from two signing sessions, P_m can exploit the algebraic relationship to recover the aggregate secret key `x`.

**Impact**: **High** — Corruption of presign outputs: honest parties accept presignatures derived from correlated triples, breaking the security proof of the protocol. **Potentially Critical** — Nonce-share correlation enabling secret key extraction via standard ECDSA nonce-reuse algebra across signing sessions.

---

### Likelihood Explanation

The attack requires only that a single malicious participant takes part in two or more triple generation sessions with the same participant set and threshold — a routine occurrence in any long-running MPC deployment where the same nodes repeatedly generate fresh triples for signing. The attack is entirely passive: the replayed `PolynomialCommitmentsMessage` is indistinguishable from a legitimately generated one, so honest participants cannot detect it. No special capabilities beyond normal protocol participation are required.

---

### Recommendation

Include a session-unique identifier in the transcript, following the same pattern already used in the DKG protocol. A random session ID should be agreed upon by all participants at the start of each triple generation session (e.g., via a commit-then-reveal round) and absorbed into the transcript before any proofs are generated or verified:

```rust
fn create_transcript(
    session_id: &[u8],          // <-- add per-session random value
    participants: &ParticipantList,
    threshold: ReconstructionLowerBound,
) -> Result<Transcript, ProtocolError> {
    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);
    transcript.message(b"group", NAME);
    transcript.message(b"session_id", session_id);  // <-- bind to session
    transcript.message(b"participants", &enc);
    transcript.message(b"threshold", &...);
    Ok(transcript)
}
```

This ensures that a proof generated in session 1 produces a different challenge in session 2, making cross-session replay cryptographically infeasible.

---

### Proof of Concept

1. Configure a 3-participant MPC network with threshold `t = 1`.
2. Run triple generation **session 1**. Malicious participant P_m records the full `PolynomialCommitmentsMessage` they broadcast (fields: `big_e`, `big_f`, `big_l`, `randomizer`, `phi_proof0`, `phi_proof1`).
3. Run triple generation **session 2** with the same 3 participants and threshold `t = 1`.
4. P_m submits the identical `PolynomialCommitmentsMessage` from step 2 instead of generating fresh values.
5. Observe that `phi_proof0` and `phi_proof1` pass verification — the `create_transcript` output is byte-for-byte identical, so the Fiat-Shamir challenge is unchanged.
6. Inspect the resulting Beaver triple shares from both sessions: P_m's polynomial contribution `f_{k_m}` is identical, meaning `k_m^{(1)} = k_m^{(2)}`.
7. Use the two correlated presignatures in signing sessions on distinct messages and demonstrate that the algebraic relationship between the nonce shares allows recovery of the secret key `x`.

fn challenge<C: Ciphersuite>(
    domain_separator: &mut DomainSeparator,
    session_id: &HashOutput,
    id: Scalar<C>,
    vk_share: &CoefficientCommitment<C>,
    big_r: &Element<C>,
) -> Result<Challenge<C>, ProtocolError> {
    let mut preimage = vec![];
    let serialized_id = <C::Group as Group>::Field::serialize(&id);

    // Should not return Error
    // The function should not be called when the first coefficient is zero
    let serialized_vk_share = vk_share.serialize().map_err(|_| {
        ProtocolError::AssertionFailed(
            "The verification share
        could not be serialized as it is null"
                .to_string(),
        )
    })?;

    let serialized_big_r = <C::Group>::serialize(big_r).map_err(|_| {
        ProtocolError::AssertionFailed(
            "The group element R
        could not be serialized as it is the identity"
                .to_string(),
        )
    })?;

    preimage.extend_from_slice(&domain_separator.to_le_bytes());
    preimage.extend_from_slice(session_id.as_ref());
    preimage.extend_from_slice(serialized_id.as_ref());
    preimage.extend_from_slice(serialized_vk_share.as_ref());
    preimage.extend_from_slice(serialized_big_r.as_ref());

    domain_separator.increment();

    let hash = C::HDKG(&preimage[..]).ok_or(ProtocolError::DKGNotSupported)?;
    Ok(Challenge::from_scalar(hash))
}

/// Verify that a proof attesting to the validity of some statement.
/// We use a transcript in order to verify the Fiat-Shamir transformation.
pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError> {
    transcript.message(NEAR_DLOG_STATEMENT_LABEL, &statement.encode()?);

    let big_k = C::Group::generator() * proof.s.0 - *statement.public * proof.e.0;

    // Create a serialization of big_k
    // Raises error if the big_k turned out to be the identity element
    let ser = C::Group::serialize(&big_k).map_err(|_| ProtocolError::IdentityElement)?;

    transcript.message(NEAR_DLOG_COMMITMENT_LABEL, ser.as_ref());
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOG_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, TranscriptRng>(&mut rng);

    Ok(e == proof.e.0)
}
