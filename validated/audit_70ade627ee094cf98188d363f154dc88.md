### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `pool.cairo` contains an unbounded loop that iterates over every entry in a delegator's `pool_member_epoch_balance` trace. Because each balance change in a new epoch appends a new entry to this trace with no cap, a delegator who accumulates enough balance-change entries without claiming rewards will eventually find that their `claim_rewards` transaction exceeds the Starknet gas limit and permanently reverts, freezing their unclaimed yield with no recovery path.

---

### Finding Description

`calculate_rewards` is called by both `claim_rewards` and `pool_member_info_v1`. It iterates over the entire `pool_member_epoch_balance` trace starting from `entry_to_claim_from`: [1](#0-0) 

The developers themselves flag this:

> `// **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.`

The trace grows via `set_member_balance`, which is called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`: [2](#0-1) 

The underlying `insert` implementation appends a **new** checkpoint whenever the epoch key differs from the last entry's key: [3](#0-2) 

So every balance change made in a distinct epoch permanently adds one entry to the trace. The only pruning mechanism is `entry_to_claim_from`, which advances after a **successful** `claim_rewards` call: [4](#0-3) 

If `claim_rewards` reverts (out-of-gas), `entry_to_claim_from` is never updated, so every subsequent attempt starts the same oversized loop and also reverts — a permanent freeze.

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

Once the trace is large enough that `claim_rewards` exceeds the Starknet transaction gas limit, the delegator's accumulated STRK rewards become permanently inaccessible. There is no administrative function, no compaction mechanism, and no alternative claim path that bypasses `calculate_rewards`. The funds remain locked in the pool contract forever.

---

### Likelihood Explanation

**Medium.**

- A delegator who calls `add_to_delegation_pool` (minimum amount: 1 wei) once per epoch, without claiming rewards in between, adds one trace entry per epoch.
- Starknet epochs are short (on the order of hours to days). Over months of active participation, a delegator can accumulate hundreds to thousands of entries.
- Each loop iteration performs multiple storage reads (`pool_member_trace.at(entry_to_claim_from)` and `find_sigma`), making it expensive in terms of gas steps.
- The code comment itself acknowledges the loop is unbounded and relies on the assumption that gas limits will not be hit — an assumption that weakens as the protocol ages and delegators accumulate history.
- No minimum-amount floor prevents micro-deposits from being used to inflate the trace cheaply.

---

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject balance changes that would push the trace beyond a safe bound (e.g., 500 entries).
2. **Alternatively, compact the trace on each successful `claim_rewards` call** by truncating all entries before the new `entry_to_claim_from`.
3. **Remove the "unlikely to exceed gas limits" assumption** from the comment and replace it with an enforced invariant.

---

### Proof of Concept

```
// Attacker is a delegator in an active pool.
// They call add_to_delegation_pool(1) once per epoch, never claiming rewards.
// After N epochs, pool_member_epoch_balance trace has N entries.
// When claim_rewards is finally called:
//   calculate_rewards loops from entry_to_claim_from=0 to N
//   Each iteration: storage read (pool_member_trace.at) + find_sigma (another storage read)
//   At sufficiently large N, transaction exceeds gas limit and reverts.
//   entry_to_claim_from remains 0.
//   All subsequent claim_rewards calls also revert.
//   Delegator's accumulated STRK yield is permanently frozen in the pool contract.

// Minimal reproduction sequence (pseudocode):
for epoch in 0..N:
    advance_epoch()
    add_to_delegation_pool(pool_member=attacker, amount=1)
    // do NOT call claim_rewards

// Now attempt to claim:
claim_rewards(pool_member=attacker)
// → OOG revert if N is large enough
// → entry_to_claim_from unchanged
// → all future claim_rewards calls also revert
```

The entry point is `add_to_delegation_pool` in `pool.cairo` (line 221), callable by any pool member with no privileged role required. [5](#0-4)
