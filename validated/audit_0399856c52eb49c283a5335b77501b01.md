### Title
NEAR Tokens Permanently Stuck in XCC Router Due to Unnecessary `#[payable]` on `execute_scheduled` With No Withdrawal Mechanism — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC Router contract is marked `#[payable]` and is intentionally callable by any unprivileged account. However, the router contract has no general withdrawal function. Any NEAR attached to a call of `execute_scheduled` is silently accepted and permanently frozen in the router sub-account, because the attached deposit is never forwarded to the scheduled promise and no recovery path exists.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, the `execute_scheduled` function is declared `#[payable]`: [1](#0-0) 

The inline comment on lines 146–148 explicitly states this function is intentionally open to any caller: [2](#0-1) 

When `execute_scheduled` runs, it reconstructs the stored `PromiseArgs` and calls `Self::promise_create(promise)`. The `attached_balance` forwarded to the cross-contract call is whatever was encoded inside the `PromiseArgs` struct at scheduling time — **not** the NEAR deposited by the caller of `execute_scheduled`. The caller's deposit is absorbed by the router account and never forwarded: [3](#0-2) 

The only NEAR-sending function in the router is `send_refund`, which unconditionally transfers a hardcoded constant `REFUND_AMOUNT` (2 NEAR) back to the parent: [4](#0-3) [5](#0-4) 

There is no function in the router that allows the parent or any other entity to withdraw an arbitrary amount of NEAR. The `deploy_upgrade` function is also `#[payable]`: [6](#0-5) 

Any NEAR deposited via either of these payable functions accumulates in the router sub-account with no recovery path.

---

### Impact Explanation

**Permanent freezing of funds.** Any NEAR attached to a call of `execute_scheduled` by any external account is irrecoverably locked in the router sub-account. Because each EVM address that uses XCC gets its own router sub-account (e.g., `<address>.aurora`), and because `execute_scheduled` is callable by anyone, an attacker or a mistaken user can cause NEAR to be permanently frozen in any router sub-account. The parent (Aurora engine) has no admin withdrawal path — `send_refund` only sends the fixed 2 NEAR constant, and there is no other egress.

---

### Likelihood Explanation

**Medium.** The `execute_scheduled` function is publicly callable with no access control. A user who wants to "fund" the execution of a scheduled promise may reasonably attach NEAR, expecting it to be forwarded. Additionally, a griefing attacker can deliberately send dust NEAR to any router sub-account to permanently lock it. The router sub-accounts are created automatically by the engine for every EVM address that uses XCC, so the attack surface is broad.

---

### Recommendation

Remove the `#[payable]` attribute from `execute_scheduled`. The function does not need to receive NEAR — the `attached_balance` for the underlying cross-contract call is encoded inside the stored `PromiseArgs` and is sourced from the router's own balance (funded via `fund_xcc_sub_account`). Removing `#[payable]` causes the NEAR SDK to automatically panic if any deposit is attached, preventing accumulation entirely — the same fix applied in the referenced PR 37 for `StaticAttester`.

Alternatively, add a `withdraw` function restricted to the parent account that transfers an arbitrary amount of NEAR back to the parent.

---

### Proof of Concept

1. Alice's EVM address `0xAAAA...` has a router sub-account `aaaa....aurora` with a scheduled promise at nonce `0`.
2. Bob (any unprivileged account) calls `aaaa....aurora::execute_scheduled({"nonce": "0"})` with `1 NEAR` attached.
3. The function removes the scheduled promise, calls `Self::promise_create(promise)` — which uses the `attached_balance` from the stored `PromiseArgs`, not Bob's deposit — and returns the promise.
4. Bob's `1 NEAR` is now held by `aaaa....aurora`.
5. The only egress is `send_refund`, which sends exactly `REFUND_AMOUNT` (2 NEAR) to the parent. If the router balance is less than 2 NEAR, `send_refund` will fail. If it is more, the excess is permanently stuck.
6. The parent (Aurora engine) has no `withdraw` or `drain` method targeting router sub-accounts, so the NEAR cannot be recovered without a contract upgrade. [1](#0-0) [4](#0-3)
