### Title
`protocolUnpausePool` Sets Wrong Pause Level (1 Instead of 0), Leaving Pool Permanently Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool()` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). After the protocol owner "unpauses" a pool, it remains paused at admin level. Every swap continues to revert with `PoolPaused()` until the pool admin separately calls `unpausePool()`.

---

### Finding Description

The pool defines three pause levels:

> `0 = active, 1 = paused by admin, 2 = paused by protocol` [1](#0-0) 

`_checkNotPaused()` reverts for **any** non-zero level: [2](#0-1) 

`protocolUnpausePool` is the protocol owner's function to lift a protocol-level pause. It reads the current level, asserts it is 2, then calls `setPause(1)` — not `setPause(0)`: [3](#0-2) 

Compare with the admin's `unpausePool`, which correctly targets level 0: [4](#0-3) 

And `protocolPausePool`, which can pause from level 0 **or** level 1: [5](#0-4) 

The resulting state machine has a broken path:

```
Active (0) --[protocolPausePool]--> Protocol-Paused (2)
                                         |
                              [protocolUnpausePool]
                                         |
                                         v
                              Admin-Paused (1)  ← still paused!
                                         |
                              [unpausePool by admin]
                                         |
                                         v
                                    Active (0)
```

When a pool that was **active (0)** is protocol-paused (→ 2) and then protocol-unpaused (→ 1), it lands in admin-paused state even though the pool admin never issued a pause. The pool admin must now separately call `unpausePool` to restore activity. If the admin is unresponsive or unaware, the pool is permanently stuck at level 1.

This is the direct analog of the Union Finance bug: the wrong target value is passed to the state-update call (`setPause(1)` instead of `setPause(0)`), leaving the system in a broken state that the intended caller cannot self-correct.

---

### Impact Explanation

Every call to `protocolUnpausePool` leaves the pool paused. `swap`, `addLiquidity`, and `removeLiquidity` all gate on `whenNotPaused` / `_checkNotPaused()`, so all user-facing pool operations remain disabled. LPs cannot exit, traders cannot swap, and the pool is functionally insolvent from a usability standpoint until the admin acts. [6](#0-5) 

---

### Likelihood Explanation

This triggers on **every** invocation of `protocolUnpausePool` — there are no low-probability prerequisites. Any emergency pause/unpause cycle by the protocol owner reproduces the broken state unconditionally.

---

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
// Before
IMetricOmmPoolFactoryActions(pool).setPause(1);

// After
IMetricOmmPoolFactoryActions(pool).setPause(0);
```

If the intent is to hand control back to the admin after a protocol unpause, that should be documented explicitly and the function renamed; as written, the name `protocolUnpausePool` implies full restoration to active state.

---

### Proof of Concept

1. Pool is active: `pauseLevel == 0`.
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel = 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` → `pauseLevel = 1` (still paused).
4. Any user calls `swap(...)` → reverts `PoolPaused()`.
5. Pool admin must call `unpausePool(pool)` → `pauseLevel = 0` (finally active).

Without step 5, the pool is permanently paused despite the protocol owner having "unpaused" it.
