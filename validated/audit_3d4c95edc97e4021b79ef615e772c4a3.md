Audit Report

## Title
GTC `donate_account` Permanently Locks User Out of Neuron Recovery After Partial Transfer Failure - (File: rs/nns/gtc/src/lib.rs)

## Summary
`AccountState::transfer` unconditionally removes each neuron ID from `self.neuron_ids` and always returns `Ok(())` regardless of whether individual `transfer_gtc_neuron` cross-canister calls succeed or fail. After `transfer` returns, `donate_account` unconditionally sets `has_donated = true`. Any subsequent call to `claim_neurons` or `donate_account` is permanently blocked, leaving neurons that failed to transfer stranded under GTC control with no recovery path.

## Finding Description
In `AccountState::transfer` (`rs/nns/gtc/src/lib.rs`, lines 188–209), the loop over `neuron_ids` calls `GovernanceCanister::transfer_gtc_neuron` and then, at line 192, unconditionally executes `self.neuron_ids.retain(|id| id != &neuron_id)` before inspecting the result. Whether the call returns `Ok` or `Err`, the neuron ID is removed from `neuron_ids`. Failed neurons are pushed to `failed_transferred_neurons` (line 204) but there is no mechanism to retry them. The function always returns `Ok(())` at line 209.

Back in `donate_account` (lines 89–90), `account.transfer(...).await?` propagates only a hard error from `transfer` itself (which never occurs for individual neuron failures), then `account.has_donated = true` is set unconditionally.

From this point:
- `claim_neurons` (lines 54–56) returns `Err("Account has previously donated its funds")` whenever `has_donated` is true.
- `transfer` (lines 177–178) returns `Err("Account has already donated its funds")` whenever `has_donated` is true, blocking any retry via `donate_account`.

By contrast, `forward_whitelisted_unclaimed_accounts` (lines 118–126) correctly sets `has_forwarded = true` only inside the `Ok(_)` arm, demonstrating the developers knew the correct pattern but did not apply it to `donate_account`.

The `has_donated` field is a permanent boolean in the proto with no partial-state tracking (`gtc.proto`, lines 56–57), so there is no on-chain representation of "partially donated" that could enable recovery.

## Impact Explanation
A GTC account holder who calls `donate_account` during any transient failure of the Governance canister (or the ledger call inside `transfer_gtc_neuron`) will permanently lose control of all neurons that failed to transfer. Those neurons remain staked in Governance under GTC canister control, are no longer in `neuron_ids`, and cannot be claimed, donated, or recovered through any GTC interface. This constitutes permanent, irreversible loss of genesis allocation neurons — real ICP-staked governance assets — matching the **High** impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds."

## Likelihood Explanation
The `donate_account` flow issues one cross-canister call per neuron to Governance, which itself calls the ICP ledger. SR accounts hold up to 48 neurons, meaning up to 48 sequential async await points. Any transient canister error (cycles exhausted, canister stopped/upgrading, ledger temporarily unavailable) on any one of those calls triggers the partial-failure scenario. The IC canister messaging system can return errors for many reasons. The probability per individual call is low in normal operation, but across 48 calls and across all remaining GTC accounts it is non-negligible, and the consequence is permanent and irreversible for the affected user.

## Recommendation
1. **Only remove a neuron ID from `self.neuron_ids` after a confirmed successful transfer.** Move `self.neuron_ids.retain(...)` inside the `Ok(_)` arm of the match.
2. **Do not set `has_donated = true` if any neuron transfer failed.** Check `self.failed_transferred_neurons.is_empty()` before returning `Ok(())` from `transfer`, or propagate a partial-failure error to `donate_account` so it does not set `has_donated`.
3. **Allow retrying `donate_account`** when `failed_transferred_neurons` is non-empty and `has_donated` is false, consistent with the pattern already used in `forward_whitelisted_unclaimed_accounts`.

## Proof of Concept
1. A GTC account holder with multiple SR neurons calls `donate_account`.
2. The first N neurons transfer successfully; on neuron N+1, the Governance canister returns a transient error.
3. In `AccountState::transfer` (line 192), neuron N+1's ID is removed from `self.neuron_ids` and placed in `failed_transferred_neurons` (line 204). The loop continues; remaining neurons also fail.
4. `transfer` returns `Ok(())` (line 209).
5. `donate_account` sets `account.has_donated = true` (line 90).
6. User calls `claim_neurons` to recover failed neurons → `Err("Account has previously donated its funds")` (lines 54–56).
7. User calls `donate_account` again to retry → `Err("Account has already donated its funds")` from `transfer` (lines 177–178).
8. Failed neurons remain in Governance under GTC control, permanently inaccessible.
