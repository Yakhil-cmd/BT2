### Title
Direct DSToken `burn` on ModuleCRC21/ModuleCRC20 Permanently Locks IBC Vouchers in Contract Escrow — (File: `contracts/src/ModuleCRC21.sol`, `contracts/src/ModuleCRC20.sol`)

### Summary

`ModuleCRC21` and `ModuleCRC20` both inherit from `DSToken`, which exposes a public `burn(uint wad)` function that allows any token holder to burn their own CRC21 tokens directly. For IBC-voucher-backed tokens (`ibc/...` denom), this permanently locks the corresponding native IBC vouchers in the contract's Cosmos bank account, creating an irrecoverable accounting discrepancy: the CRC21 `totalSupply` decreases while the escrowed native tokens remain stuck forever.

### Finding Description

**Token accounting invariant for IBC vouchers:**

When `ConvertCoinFromNativeToCRC21` is called for a non-source (`ibc/...`) denom, the keeper:
1. Sends native IBC tokens to the contract's Cosmos bank address (escrow)
2. Calls `mint_by_cronos_module` to mint CRC21 tokens to the user [1](#0-0) 

The invariant is: `CRC21.totalSupply == native_IBC_tokens_held_at_contract_address`.

**The break:**

`ModuleCRC21` inherits from `DSToken` and does **not** override the public `burn(uint wad)` function. DSToken's implementation allows any holder to burn their own tokens without any authorization check:

```solidity
// DSToken (inherited, not overridden in ModuleCRC21)
function burn(uint wad) public {
    burn(msg.sender, wad);
}
function burn(address guy, uint wad) public {
    // no auth check when guy == msg.sender
    balanceOf[guy] = sub(balanceOf[guy], wad);
    totalSupply    = sub(totalSupply, wad);
    emit Burn(guy, wad);
}
```

`ModuleCRC21` only defines `unsafe_burn` as `internal` and `burn_by_cronos_module` as module-gated, but neither overrides the inherited public `burn`: [2](#0-1) [3](#0-2) 

When a user calls `burn(amount)` directly:
- CRC21 `totalSupply` decreases by `amount`
- No `__CronosSendToIbc` or `__CronosSendToEvmChain` event is emitted
- No EVM hook fires
- The native IBC tokens remain escrowed in the contract's bank account permanently

The only path to release the escrowed native tokens is `ConvertCoinFromCRC21ToNative`, which requires the caller to hold CRC21 tokens — but those have been burned. There is no recovery path.

The same issue exists in `ModuleCRC20`, which also inherits DSToken's public `burn` without overriding it: [4](#0-3) 

### Impact Explanation

**Critical — Unauthorized burn / permanent accounting change for IBC vouchers.**

For every unit burned via the direct `burn` path:
- The native IBC vouchers are permanently locked in the contract's Cosmos bank address with no mechanism to release them.
- The IBC escrow on the source chain retains the corresponding tokens in escrow indefinitely, since no IBC transfer packet is ever sent to release them.
- `CRC21.totalSupply` permanently underrepresents the native tokens held at the contract address, breaking the supply invariant that the entire bridge accounting relies on.

This is the direct Cronos analog of the Linea report: burning on the "wrapped" side without updating the "source" side escrow, causing permanent token loss and supply accounting corruption for IBC vouchers.

### Likelihood Explanation

Any unprivileged user holding CRC21 tokens for an IBC-voucher-backed contract can trigger this with a single direct call to `burn(amount)`. No special role, key, or governance action is required. The entry path is fully reachable from the EVM layer.

### Recommendation

Override the inherited `burn(uint wad)` and `burn(address, uint)` functions in both `ModuleCRC21` and `ModuleCRC20` to unconditionally revert:

```solidity
function burn(uint) public override {
    revert("direct burn not allowed; bridge back to source chain");
}
function burn(address, uint) public override {
    revert("direct burn not allowed; bridge back to source chain");
}
```

Users who wish to destroy their tokens should bridge them back to the source chain via `send_to_ibc`, which correctly releases the escrowed native tokens and initiates the IBC transfer.

### Proof of Concept

1. IBC voucher `ibc/AAAA...` is mapped to a deployed `ModuleCRC21` contract at address `C`.
2. User calls `ConvertCoinFromNativeToCRC21` with 100 `ibc/AAAA...` tokens.
   - 100 native tokens are sent to `C`'s Cosmos bank account. [5](#0-4) 
   - `C.totalSupply` becomes 100.
3. User calls `C.burn(100)` directly (DSToken public function, not overridden).
   - `C.totalSupply` becomes 0.
   - No EVM hook fires. No IBC transfer is initiated.
4. The 100 native `ibc/AAAA...` tokens remain in `C`'s Cosmos bank account forever.
5. `ConvertCoinFromCRC21ToNative` cannot be called (user has 0 CRC21 tokens).
6. The source chain's IBC escrow retains the tokens permanently locked. [6](#0-5) [7](#0-6)
