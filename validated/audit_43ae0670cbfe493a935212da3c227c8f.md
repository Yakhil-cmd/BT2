Looking at the production code directly:

### Title
`vm_resources_to_execution_resources` hardcodes `gas_consumed` to zero in all RPC execution traces — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`vm_resources_to_execution_resources` unconditionally sets `ExecutionResources.gas_consumed` to `StarknetApiGasVector::default()` (all-zero) regardless of the actual gas consumed. Every `FunctionInvocation` returned by `starknet_traceTransaction`, `starknet_simulateTransactions`, and related RPC methods therefore reports zero `gas_consumed`, even for V3 transactions with significant L2/Sierra gas consumption.

---

### Finding Description

The function signature accepts a `GasVector` argument (destructured as `{ l1_gas, l1_data_gas, l2_gas }`), correctly assigns it to `da_gas_consumed`, but then hardcodes `gas_consumed` to the default (zero):

```rust
// crates/apollo_rpc_execution/src/objects.rs, lines 363–401
fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    GasVector { l1_gas, l1_data_gas, l2_gas }: GasVector,
) -> ExecutionResult<ExecutionResources> {
    // ...
    Ok(ExecutionResources {
        steps: u64_from_usize(vm_resources.n_steps),
        builtin_instance_counter,
        memory_holes: u64_from_usize(vm_resources.n_memory_holes),
        da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
        gas_consumed: StarknetApiGasVector::default(),   // ← always zero
    })
}
``` [1](#0-0) 

The call site passes `receipt.da_gas` as the gas vector for every invocation type (validate, execute, fee-transfer): [2](#0-1) 

So `da_gas_consumed` is populated from `receipt.da_gas`, but `gas_consumed` — which per the Starknet spec should reflect the L2/Sierra gas consumed by the individual call — is never populated from any source. The `ExecutionResources` struct defines both fields as distinct: [3](#0-2) 

---

### Impact Explanation

Every `FunctionInvocation.execution_resources.gas_consumed` value returned by `starknet_traceTransaction` and `starknet_simulateTransactions` is `{l1_gas: 0, l1_data_gas: 0, l2_gas: 0}` for all transactions. For V3 transactions that consume nonzero L2 (Sierra) gas, the per-call trace is factually wrong. Tooling, block explorers, and developers relying on this field to understand per-call gas breakdown receive systematically incorrect data from an authoritative-looking RPC response.

This matches the allowed High impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

This is a hardcoded constant, not a conditional path. It triggers for **every** traced or simulated transaction — no special attacker input is required. Any V3 invoke transaction with nonzero L2 gas consumption demonstrates it immediately.

---

### Recommendation

Pass the actual per-call gas consumed (from `call_info.resources` or the receipt's `gas` field) into `vm_resources_to_execution_resources` and assign it to `gas_consumed` instead of `StarknetApiGasVector::default()`. The `da_gas_consumed` field should continue to receive `receipt.da_gas`. The two fields represent distinct quantities and must be sourced independently.

---

### Proof of Concept

```rust
// Demonstrates the hardcoded zero — no special setup needed.
let result = vm_resources_to_execution_resources(
    VmExecutionResources { n_steps: 10, ..Default::default() },
    GasVector { l1_gas: GasAmount(100), l1_data_gas: GasAmount(50), l2_gas: GasAmount(200) },
).unwrap();

// da_gas_consumed is correctly populated:
assert_eq!(result.da_gas_consumed.l1_gas, GasAmount(100));
assert_eq!(result.da_gas_consumed.l2_gas, GasAmount(200));

// gas_consumed is always zero regardless of input:
assert_eq!(result.gas_consumed, GasVector::default()); // passes — this is the bug

```rust
fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    GasVector { l1_gas, l1_data_gas, l2_gas }: GasVector,
) -> ExecutionResult<ExecutionResources> {
    let mut builtin_instance_counter = HashMap::new();
    for (builtin_name, count) in vm_resources.builtin_instance_counter {
        if count == 0 {
            continue;
        }
        let count = u64_from_usize(count);
        match builtin_name {
            BuiltinName::output => continue,
            BuiltinName::pedersen => builtin_instance_counter.insert(Builtin::Pedersen, count),
            BuiltinName::range_check => builtin_instance_counter.insert(Builtin::RangeCheck, count),
            BuiltinName::ecdsa => builtin_instance_counter.insert(Builtin::Ecdsa, count),
            BuiltinName::bitwise => builtin_instance_counter.insert(Builtin::Bitwise, count),
            BuiltinName::ec_op => builtin_instance_counter.insert(Builtin::EcOp, count),
            BuiltinName::keccak => builtin_instance_counter.insert(Builtin::Keccak, count),
            BuiltinName::poseidon => builtin_instance_counter.insert(Builtin::Poseidon, count),
            BuiltinName::segment_arena => {
                builtin_instance_counter.insert(Builtin::SegmentArena, count)
            }
            // TODO(DanB): what about the following?
            // BuiltinName::range_check96 => todo!(),
            // BuiltinName::add_mod => todo!(),
            // BuiltinName::mul_mod => todo!(),
            _ => {
                return Err(ExecutionError::UnknownBuiltin { builtin_name });
            }
        };
    }
    Ok(ExecutionResources {
        steps: u64_from_usize(vm_resources.n_steps),
        builtin_instance_counter,
        memory_holes: u64_from_usize(vm_resources.n_memory_holes),
        da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
        gas_consumed: StarknetApiGasVector::default(),
    })
}
