The vulnerability is real. Here is the full analysis:

---

### Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Non-Allowlisted Addresses to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`, i.e., the economic actor paying tokens) and instead checks `allowedDepositor[pool][owner]`. Because `MetricOmmPool.addLiquidity` explicitly supports the operator pattern (`msg.sender != owner`), any non-allowlisted address can call `addLiquidity(owner=allowlistedAddr)` and pass the allowlist check, depositing tokens into a restricted pool.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives both `sender` (the actual caller of `addLiquidity`) and `owner` (the position owner), but the `sender` parameter is unnamed and completely ignored:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [1](#0-0) 

The pool passes `msg.sender` as `sender` to the extension:

```solidity
// metric-core/contracts/MetricOmmPool.sol
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

The pool's own NatSpec explicitly documents the operator pattern — `msg.sender` pays but need not equal `owner`: [3](#0-2) 

So the check `allowedDepositor[pool][owner]` passes whenever `owner` is allowlisted, regardless of who `sender` is.

---

### Impact Explanation

The deposit allowlist is rendered ineffective. A non-allowlisted address can:

1. Call `pool.addLiquidity(owner=allowlistedAddr, salt, deltas, callbackData, extensionData)` directly.
2. The extension checks `allowedDepositor[pool][allowlistedAddr]` → passes.
3. The attacker's tokens are pulled via the modify-liquidity callback (to `msg.sender` = attacker).
4. LP shares are minted to `allowlistedAddr`.

The pool admin's intended access restriction is completely circumvented. Tokens from a non-allowlisted source enter the pool. The allowlisted address receives LP shares they did not initiate. This is a direct admin-boundary break: an unprivileged address bypasses a pool-admin-configured access control.

---

### Likelihood Explanation

- Requires only a pool with `DepositAllowlistExtension` in `beforeAddLiquidity` order and at least one allowlisted address (both normal production conditions).
- The attacker needs no special role, no privileged access, and no malicious pool setup.
- The attack is a single direct call to `pool.addLiquidity` with `owner` set to any known allowlisted address.
- The operator pattern (`msg.sender != owner`) is explicitly supported and documented, making this a straightforward exploit path.

---

### Recommendation

Check `sender` (the actual depositor/payer), not `owner`, in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The `allowedDepositor` mapping and `setAllowedToDeposit` admin function should also be reviewed to ensure they are keyed on the depositing address (sender), not the position owner. [4](#0-3)

### Proof of Concept

```solidity
// Setup: pool with DepositAllowlistExtension in beforeAddLiquidity order
// allowedDepositor[pool][allowlistedAddr] = true
// attacker is NOT allowlisted

// Attacker calls directly (no periphery needed):
pool.addLiquidity(
    allowlistedAddr,   // owner = allowlisted address
    salt,
    deltas,
    callbackData,      // attacker implements IMetricOmmModifyLiquidityCallback and pays tokens
    extensionData
);

// Extension checks allowedDepositor[pool][allowlistedAddr] → true → passes
// Attacker's tokens are pulled, LP shares minted to allowlistedAddr
// Allowlist bypass confirmed
```

The `MetricOmmPoolLiquidityAdder` periphery test `test_exactShares_canAddOnBehalfOfAnotherOwner` already demonstrates that `alice` can add liquidity with `owner=bob` — the same operator pattern the attacker exploits. [5](#0-4)
