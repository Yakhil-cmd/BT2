### Title
`addLiquidityWeighted` Reverts Due to Cascaded Rounding in Probe-Scale-Pay Flow - (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`addLiquidityWeighted` performs a two-step probe-then-pay pattern. The scale factor and per-bin shares are both computed with floor division, while the pool's `addLiquidity` computes per-bin token amounts with `ceilDiv`. The sum of per-bin `ceilDiv` results for the scaled shares can exceed `maxAmountToken0` / `maxAmountToken1`, causing the callback to revert with `MaxAmountExceeded` even when the user's caps are set exactly to the probe-derived token requirement scaled by the computed ratio.

---

### Finding Description

`addLiquidityWeighted` executes three steps:

**Step 1 – Probe**: calls `addLiquidity` with the raw weight shares; the callback reverts with `LiquidityProbe(need0, need1)` carrying the total token amounts required.

**Step 2 – Scale** (`_scaleWeightsToShares`):

```solidity
uint256 scaleWad0 = Math.mulDiv(max0, WAD, need0);   // floor
uint256 scaleWad  = min(scaleWad0, scaleWad1);
out.shares[i]     = Math.mulDiv(w.shares[i], scaleWad, WAD); // floor
``` [1](#0-0) 

Both `scaleWad` and `scaled_shares[i]` are floor-divided.

**Step 3 – Pay**: calls `addLiquidity` with the scaled shares. Inside `LiquidityLib.addLiquidity`, each bin's token amount is computed with `ceilDiv`:

```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
``` [2](#0-1) 

The total is then converted to external units with another `ceilDiv`:

```solidity
(amount0Added, amount1Added) = _deltasScaledToExternal(totalToken0ToAddScaled, totalToken1ToAddScaled, ctx, Math.Rounding.Ceil);
``` [3](#0-2) 

The callback then enforces:

```solidity
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
``` [4](#0-3) 

The invariant that must hold — `sum(ceilDiv(balance_i * scaled_shares_i, totalShares_i)) ≤ max0` — is not guaranteed. The probe's `need0` is the sum of per-bin `ceilDiv` results for the weight shares. The scale factor is derived from that sum. But when `ceilDiv` is applied independently to each bin for the scaled shares, the per-bin ceiling increments accumulate and the total can exceed `max0`.

---

### Impact Explanation

`addLiquidityWeighted` is a core periphery liquidity-provision path. When the rounding condition is triggered, the paying `addLiquidity` call reverts inside the callback, the pool state is rolled back, and the user's transaction fails with `MaxAmountExceeded`. The user cannot add liquidity via this function for the affected parameter combination, breaking the weighted deposit flow. No funds are lost, but the function is rendered unusable for valid inputs.

---

### Likelihood Explanation

The condition requires multiple bins with non-trivial existing liquidity such that the per-bin `ceilDiv` increments sum to more than `max0`. This is not an edge case: any pool with ≥ 2 active bins and a non-integer scale ratio can trigger it. The probability grows with the number of bins in the weight vector.

---

### Recommendation

In `_scaleWeightsToShares`, subtract 1 from `scaleWad` before applying it to shares (i.e., use `scaleWad - 1` when `scaleWad > 0`). This ensures the scaled shares are strictly below the proportional threshold, so the per-bin `ceilDiv` accumulation cannot push the total above `max0`. Alternatively, pass `max0 + N` (where `N` is the number of bins) as the effective cap inside the callback for the weighted pay call, reserving headroom for per-bin ceiling increments.

---

### Proof of Concept

Concrete arithmetic (all values in internal scaled units, `scale_multiplier = 1`):

| Parameter | Value |
|---|---|
| Bins | A and B, each with `balance0 = 3`, `totalShares = 2` |
| Weight shares per bin | `w_shares = 4` |
| `max0` | `9` |

**Probe:**
- `need0_A = ceilDiv(3 × 4, 2) = 6`
- `need0_B = ceilDiv(3 × 4, 2) = 6`
- `need0 = 12`

**Scale:**
- `scaleWad = floor(9 × 1e18 / 12) = 750000000000000000`
- `scaled_shares = floor(4 × 750000000000000000 / 1e18) = 3`

**Pay:**
- `amount0_A = ceilDiv(3 × 3, 2) = 5`
- `amount0_B = ceilDiv(3 × 3, 2) = 5`
- `total amount0 = 10`

**Callback check:** `10 > max0 = 9` → `revert MaxAmountExceeded(10, …, 9, …)`

The user set `max0 = 9` expecting to pay exactly `need0 × scaleWad/WAD = 12 × 0.75 = 9`, but the paying call requires 10 due to cascaded rounding. The `addLiquidityWeighted` call reverts.
