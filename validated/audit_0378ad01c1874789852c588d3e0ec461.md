### Title
Unbounded loop in `calculate_rewards` allows permanent freezing of pool member's unclaimed yield via trace inflation — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the `Pool` contract iterates over a pool member's entire balance-change trace without any upper bound. The developers explicitly acknowledge this: *"This loop is unbounded but unlikely to exceed gas limits."* A pool member can inflate this trace by calling `add_to_delegation_pool` once per epoch while never claiming rewards. After enough epochs, the `claim_rewards` transaction will exceed the Starknet gas limit, permanently freezing the pool member's accumulated unclaimed yield with no recovery path.

---

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:** [1](#0-0) 

The loop iterates from `entry_to_claim_from` to `pool_member_trace_length`, processing every balance-change checkpoint that has not yet been accounted for. There is no cap on the number of iterations.

**How the trace grows:**

`set_member_balance` inserts a new checkpoint into `pool_member_epoch_balance` at key `current_epoch + K`: [2](#0-1) 

The underlying `insert` function in the trace appends a **new** checkpoint only when the epoch key differs from the last stored key: [3](#0-2) 

Because the key is always `current_epoch + K`, each distinct epoch produces a distinct key, so every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends one new entry.

**Attacker-controlled entry path:**

`add_to_delegation_pool` is callable by the pool member (or their reward address) with no privileged access: [4](#0-3) 

**Why `entry_to_claim_from` does not protect against this:**

`entry_to_claim_from` advances only when `claim_rewards` succeeds. If the pool member never calls `claim_rewards`, the field stays at its initial value while the trace grows unboundedly: [5](#0-4) 

**Exploit path:**

1. Pool member joins via `enter_delegation_pool`.
2. Each epoch, pool member calls `add_to_delegation_pool` with the minimum amount (1 STRK = `10^18` FRI, per `STRK_CONFIG.min_for_rewards`).
3. Pool member never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs, `pool_member_epoch_balance` has N checkpoints.
5. The next `claim_rewards` call must iterate all N entries; once N is large enough to exhaust the Starknet per-transaction gas budget, the call reverts permanently.
6. The pool member's accumulated STRK rewards are frozen with no recovery mechanism.

---

### Impact Explanation

**Impact category:** High — Permanent freezing of unclaimed yield.

The pool member's STRK rewards become permanently inaccessible. There is no partial-claim mechanism, no admin escape hatch, and no way to reset `entry_to_claim_from` without a successful `claim_rewards` call. The funds are not stolen but are irrecoverably locked in the contract.

---

### Likelihood Explanation

**Medium.** Two realistic paths exist:

1. **Deliberate self-griefing / protocol griefing:** A pool member intentionally inflates their trace (1 STRK per epoch) while withholding reward claims. The cost is proportional to the number of epochs needed to hit the gas ceiling.
2. **Organic accumulation:** A long-term delegator who rarely claims rewards will naturally accumulate one trace entry per epoch. Depending on Starknet's gas limit and the per-iteration cost (storage reads + `find_sigma` call), a delegator who has been active for hundreds of epochs without claiming could hit the limit without any malicious intent.

The code comment *"unlikely to exceed gas limits"* is an assertion without supporting analysis of the actual per-iteration gas cost or the Starknet block gas ceiling.

---

### Recommendation

1. **Cap the loop:** Introduce a `max_iterations` parameter or a hard constant, and allow `claim_rewards` to process rewards in batches, storing intermediate progress.
2. **Enforce periodic claims:** Require pool members to claim rewards at least once every N epochs, preventing unbounded trace accumulation.
3. **Compact the trace:** After a successful `claim_rewards`, truncate all processed entries from the trace (entries before `entry_to_claim_from`) to keep the trace size bounded.

---

### Proof of Concept

```
Epoch 0:  pool_member calls enter_delegation_pool(amount=1e18)
          → trace: [(epoch=K, balance=1e18)]  length=1

Epoch 1:  pool_member calls add_to_delegation_pool(amount=1e18)
          → trace: [(K,1e18),(K+1,2e18)]  length=2

Epoch 2:  pool_member calls add_to_delegation_pool(amount=1e18)
          → trace: [(K,1e18),(K+1,2e18),(K+2,3e18)]  length=3

...

Epoch N:  trace length = N+1, entry_to_claim_from = 0

pool_member calls claim_rewards()
  → calculate_rewards iterates N+1 times
  → if N+1 > gas_limit / per_iteration_cost → OUT_OF_GAS, reverts
  → pool member's accumulated rewards are permanently frozen
```

The `pool_member_info_v1` view function also calls `calculate_rewards` with the same parameters: [6](#0-5) 

meaning even read queries for the pool member's state will fail once the trace is large enough, making the frozen state completely unobservable from outside the contract.
