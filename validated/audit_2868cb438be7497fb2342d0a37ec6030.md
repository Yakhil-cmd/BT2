### Title
Remove-then-redeploy authorization bypass allows unauthorized contract code replacement — (`fvm/environment/contract_updater.go`)

### Summary
`SetContract` determines whether a deployment authorization check is needed by querying committed on-chain storage via `accounts.ContractExists`. It does not inspect `draftUpdates`. An attacker authorized only for contract removal can stage a removal for a location, then immediately call `SetContract` for the same location within the same transaction. Because the removal is not yet committed, `ContractExists` returns `true`, the deployment authorization guard is skipped, and the new code overwrites the staged removal in `draftUpdates`. At `Commit()` time the contract is still present in storage, so the operation is classified as an update and the malicious code is written unconditionally.

### Finding Description
`ContractUpdaterImpl.SetContract` checks `accounts.ContractExists` (committed state) to decide whether to enforce deployment authorization:

```go
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return ... // authorization error
}
updater.draftUpdates[location] = ContractUpdate{Location: location, Code: code}
``` [4](#0-3) 

`RemoveContract` stages a removal by writing `ContractUpdate{Code: nil}` to `draftUpdates` but does not touch committed storage:

```go
u := ContractUpdate{Location: location}
updater.draftUpdates[location] = u
``` [5](#0-4) 

Because `draftUpdates` is a plain `map[common.AddressLocation]ContractUpdate` and both operations use the same key, a subsequent `SetContract` call for the same location silently overwrites the staged removal. The `SetContract` function has no awareness that a removal was pending for that key. [6](#0-5) 

At `Commit()`, `accounts.ContractExists` is queried again against committed storage (still `true`), so the operation is classified as an update and `accounts.SetContract` is called with the attacker-supplied code, with no authorization check ever having been performed for deployment. [7](#0-6) 

### Impact Explanation
An account authorized only for contract removal can replace any contract it can remove with arbitrary new Cadence code. Because Cadence resource types are identified by their defining contract, replacing a contract redefines all resource types declared in it. This enables:
- Unauthorized redefinition of resource types (e.g., fungible tokens, NFTs).
- Construction or duplication of resources under the new type definition.
- Destruction of existing resources whose type invariants are violated by the new definition.

This directly violates the invariant that replacing contract code requires deployment authorization, and constitutes an execution-layer bypass of resource semantics.

### Likelihood Explanation
The preconditions are:
1. `RestrictedDeploymentEnabled()` returns `true` (the default on mainnet).
2. The attacker controls an account that is on the removal-authorized list but not the deployment-authorized list.
3. The target contract already exists in storage.

Condition 2 is realistic: the removal and deployment authorized-address lists are stored separately (`ContractRemovalAuthorizedAddressesPath` vs `ContractDeploymentAuthorizedAddressesPath`) and can have different members. [8](#0-7) 

The attack is a single transaction with two Cadence stdlib calls (`contracts.remove` then `contracts.add`), executable on any standard Flow node without any privileged node access.

### Recommendation
In `SetContract`, before querying `accounts.ContractExists`, also check whether a pending removal exists in `draftUpdates` for the same location. If `draftUpdates[location].Code` is nil/empty (i.e., a removal is staged), treat the operation as a new deployment and enforce deployment authorization regardless of what committed storage reports. Alternatively, disallow calling `SetContract` for a location that already has a pending removal in the same transaction.

// Setup: contract "C" already deployed at attacker's address.

func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,

	// Initial contract deployments must be authorized by signing accounts.
	//

				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
}
