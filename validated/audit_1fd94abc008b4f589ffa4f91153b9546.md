### Title
Unrestricted `disable_rewards` Flag in `update_rewards` Allows Any Caller to Permanently Block Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The public `update_rewards` function in `StakingRewardsManagerImpl` accepts a caller-controlled `disable_rewards: bool` parameter. When set to `true`, the function still writes the current block number to the global `last_reward_block` storage slot but returns before distributing any rewards. Because `last_reward_block` is a single global gate that prevents any staker from receiving rewards in the same block, an unprivileged caller can call `update_rewards(any_valid_staker, disable_rewards: true)` every block to permanently starve all stakers of consensus rewards.

### Finding Description

`update_rewards` is part of `IStakingRewardsManager` and carries no role guard beyond `general_prerequisites()` (unpaused + non-zero caller). [1](#0-0) 

The function first writes `last_reward_block` unconditionally, then branches on `disable_rewards`: [2](#0-1) 

Because `last_reward_block` is a **global** (not per-staker) storage variable, once it is written for block N, the assertion on line 1455 will revert for every subsequent call in the same block, regardless of which staker is targeted: [3](#0-2) 

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` in every block therefore consumes the sole per-block reward slot for the entire protocol without distributing a single token.

### Impact Explanation

Every staker and delegator in the consensus-rewards phase loses all block-level rewards for as long as the attack continues. The `unclaimed_rewards_own` field of every staker remains at zero; pool members' `cumulative_rewards_trace` is never advanced. This constitutes **permanent freezing of unclaimed yield** for all participants, matching the allowed High impact, or at minimum **griefing with no profit motive but damage to users or protocol** (Medium). [4](#0-3) 

### Likelihood Explanation

- No privileged role is required; any EOA or contract can call `update_rewards`.
- The only precondition is supplying a valid, active staker address with non-zero balance at the current epoch — trivially satisfied by reading any existing staker from the public `stakers` vector.
- On Starknet, transaction fees are low enough that calling once per block (~3 s) is economically feasible for a motivated attacker.
- The attack is fully atomic per block and requires no coordination.

### Recommendation

1. **Restrict who may set `disable_rewards: true`**: add a check that the caller is the staker, their reward address, or a trusted protocol role (e.g., `security_agent`).
2. **Alternatively**, separate the "mark block consumed without rewarding" path into a privileged internal function and keep the public `update_rewards` always distributing rewards when called.
3. **At minimum**, move the `last_reward_block.write` to after the `disable_rewards` guard so that a call with `disable_rewards: true` does not consume the block slot.
