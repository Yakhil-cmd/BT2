### Title
Rushing Adversary Can Force `w(0) = 0` via Crafted Round-2 `w_me`, Permanently Aborting Presign for All Honest Parties — (`src/ecdsa/robust_ecdsa/presign.rs`)

---

### Summary

In `do_presign`, the round-2 `w_me` values are broadcast without any prior commitment. A rushing adversary controlling even one of the `t` malicious participants can observe all honest parties' `w_me` values before sending their own, then craft a `w_me` that drives the Lagrange interpolation `w(0)` to zero. The `ZeroScalar` guard at step 3.6 fires before the `W == g^w` consistency check at step 3.12, so the forgery is never detected and every honest party aborts with `ProtocolError::ZeroScalar`.

---

### Finding Description

**Round-2 broadcast has no prior commitment to `w_me`.**

At step 2.5, each participant broadcasts `(big_r_me, w_me)` in a single `send_many` call: [1](#0-0) 

The only commitment sent alongside `w_me` is `big_r_me = g^{k_me}`, which binds `k_me` but says nothing about `w_me = a_me * k_me + b_me`. [2](#0-1) 

**A rushing adversary can observe honest `w_me` values before sending their own.**

Honest parties call `chan.send_many` and then `chan.recv` in a loop. A malicious participant's implementation can simply delay its own `send_many` until it has received all honest parties' messages — the standard rushing-adversary capability. No commit-then-reveal mechanism prevents this.

**Step 3.5 interpolates `w(0)` from all 2t+1 shares.** [3](#0-2) 

The interpolated value is:

```
w(0) = Σ_{i=1}^{2t+1}  λ_i · w_i
```

With `t` malicious participants the attacker controls `t` free variables and needs to satisfy one scalar equation `w(0) = 0`. This is always solvable: for the simplest case `t = 1`, the attacker sets

```
w_j = −(Σ_{i ∈ honest} λ_i · w_i) / λ_j
```

**Step 3.6 aborts before the consistency check at step 3.12.** [4](#0-3) 

The `W == g^w` check that would expose the inconsistency between the crafted `w_me` and the attacker's actual `a_me` share is at step 3.12: [5](#0-4) 

Because `ZeroScalar` is returned at line 247 before execution ever reaches line 303, the forgery is never caught. Every honest party receives `Err(ProtocolError::ZeroScalar)` with no indication of which participant sent the bad value.

---

### Impact Explanation

Every honest party aborts presign. Because the error carries no attribution, the honest parties cannot identify and exclude the malicious participant. The attacker can repeat the attack on every subsequent presign attempt, permanently denying the ability to produce presignatures (and therefore signatures) for the key. This matches the **High** impact category: *Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions*.

---

### Likelihood Explanation

- Requires controlling exactly one participant out of `2t+1` — the minimum adversary budget the protocol is supposed to tolerate.
- Requires only network-level rushing (delay one's own send until others' messages arrive), which is a standard and realistic capability for any participant who controls their own network stack.
- No cryptographic assumption needs to be broken; the attack is pure protocol manipulation.

---

### Recommendation

Bind `w_me` to a commitment before the broadcast round, then open it. Concretely:

1. **Commit phase**: each party broadcasts `H(w_me || nonce)` before sending `w_me`.
2. **Reveal phase**: each party sends `(w_me, nonce)`; others verify the commitment.

This prevents a rushing adversary from choosing `w_me` after seeing honest values. Alternatively, restructure the check order so that the `W == g^w` exponent-consistency check (step 3.12) is performed *before* the `ZeroScalar` abort (step 3.6), so that a party who sends a `w_me` inconsistent with their committed `W_me` is identified and excluded rather than silently aborting the whole session.

---

### Proof of Concept

Simulation (N=3, t=1):

1. Start presign with participants `{P1, P2, P3}`, `max_malicious = 1`. `P3` is the attacker.
2. `P1` and `P2` send their honest `(big_r_i, w_i)` in round 2.
3. `P3` (rushing) receives `w_1` and `w_2`, computes Lagrange coefficients `λ_1, λ_2, λ_3` for interpolation at 0, then sets:
   ```
   w_3 = −(λ_1·w_1 + λ_2·w_2) / λ_3
   ```
4. `P3` sends `(big_r_3, w_3)` (with any `big_r_3`).
5. All three parties compute `w(0) = λ_1·w_1 + λ_2·w_2 + λ_3·w_3 = 0`.
6. All three parties hit the guard at line 246–247 and return `Err(ProtocolError::ZeroScalar)`. [6](#0-5) 

Assert: all parties return `Err(ZeroScalar)`; no presignature is produced; no party can identify `P3` as the culprit.
