### Title
Unchecked Precompile Call Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then invoking the Aurora exit precompile via inline assembly. The assembly `call` opcode stores its success flag in `res`, but `res` is never inspected. When the precompile returns an error (e.g., invalid NEAR recipient, amount exceeding `u128::MAX`, or unregistered ERC-20), the EVM inner call silently returns 0, the outer transaction does not revert, and the already-burned tokens are permanently destroyed with no corresponding release on the NEAR side.

---

### Finding Description

In `EvmErc20.sol` and `EvmErc20V2.sol`, the withdrawal functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum` and in both versions of the contract: [2](#0-1) [3](#0-2) 

The `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` and `ExitToEthereum` at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` both return `ExitError::Other(...)` (which maps to `PrecompileFailure::Error`) for a range of user-triggerable conditions: [4](#0-3) [5](#0-4) 

When a precompile returns `PrecompileFailure::Error`, the EVM `call` opcode returns 0 (failure) but execution of the calling contract continues normally — the outer transaction is **not** reverted. Because `_burn` executes unconditionally before the assembly block, the tokens are already destroyed at the point the precompile fails.

Concrete user-triggerable failure paths inside the precompile:

| Condition | Error returned |
|---|---|
| `recipient` bytes not a valid NEAR account ID | `ERR_INVALID_RECEIVER_ACCOUNT_ID` |
| `recipient` bytes exceed `MAX_INPUT_SIZE` | `ERR_INVALID_INPUT` |
| `amount > u128::MAX` (valid `uint256`, rejected by `parse_amount`) | `ERR_INVALID_AMOUNT` |
| ERC-20 address not registered in the NEP-141 map | `ERR_TARGET_TOKEN_NOT_FOUND` | [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently, the ERC-20 tokens are burned on the Aurora side (reducing the user's balance to zero for the withdrawn amount) while the corresponding NEP-141 tokens on NEAR remain locked in the Aurora engine account forever. There is no recovery path: the ERC-20 supply is reduced, the NEAR-side tokens are not released, and no refund mechanism is triggered because the EVM transaction itself succeeds.

---

### Likelihood Explanation

**Medium.** Any unprivileged token holder can trigger this by:

1. Calling `withdrawToNear` with a `recipient` that is not a valid NEAR account ID (e.g., contains uppercase letters, starts with `-`, or exceeds the 64-character limit). NEAR account ID validation is enforced inside the precompile via `parse_recipient` → `AccountId::parse`, but the Solidity function accepts arbitrary `bytes` with no pre-validation.
2. Calling `withdrawToNear` or `withdrawToEthereum` with `amount > u128::MAX`. The Solidity parameter type is `uint256`; the precompile rejects values above `u128::MAX` via `parse_amount`, but the ERC-20 contract imposes no such cap before burning.

Both paths require only a standard EVM call from any address that holds tokens.

---

### Recommendation

Add a revert on precompile call failure in both functions of both contracts:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, validate `recipient` length and `amount <= type(uint128).max` in Solidity before burning, so the burn is never reached when the precompile would reject the input.

---

### Proof of Concept

1. Deploy `EvmErc20` (or use an existing bridged token).
2. Mint tokens to address `A`.
3. From address `A`, call `withdrawToNear(bytes("INVALID ACCOUNT!"), amount)` — the recipient contains spaces and uppercase letters, which are rejected by NEAR's account ID rules.
4. Inside the precompile, `parse_recipient` → `AccountId::parse` fails → `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")` → inner `call` returns 0.
5. `res` is not checked; the outer transaction succeeds.
6. Observe: address `A`'s ERC-20 balance is reduced by `amount`; the NEP-141 balance of the Aurora engine account on NEAR is unchanged; the tokens are permanently lost. [8](#0-7) [1](#0-0)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
