## Title
Rate-Limit Consumed Without Cycle Delivery Enables Global Minting DoS â€” (`rs/nns/cmc/src/main.rs`, `deposit_cycles` / `ensure_balance`)

## Summary
In `deposit_cycles`, `ensure_balance` synchronously charges the rate limiter and calls `ic0_mint_cycles128` before the async management canister `deposit_cycles` call. If that call fails (e.g., targeting a non-existent canister), the limiter is permanently charged for the current window while the ICP is refunded to the caller. A single ~1500 ICP transaction (refunded minus fees) can exhaust the 150P/hour global limit, blocking all legitimate minting for up to one hour.

## Finding Description
In `deposit_cycles` (`rs/nns/cmc/src/main.rs`, lines 2116â€“2128), `ensure_balance` is called synchronously at line 2117 before the `call_with_payment128` await at line 2120. Inside `ensure_balance` (lines 2306â€“2325), `limiter_to_use.check_and_add_cycles` permanently records the minting event in the limiter's `time_windows` deque and increments `total_count` (line 2316), then `total_cycles_minted` is incremented (line 2317), and `ic0_mint_cycles128` mints cycles into the CMC's own balance (line 2322) â€” all before any await point. When the management canister rejects `deposit_cycles` (e.g., because the target canister does not exist), the IC returns the attached cycles to the CMC, but `base_limiter.total_count` retains the charged amount with no compensating decrement. `process_top_up` (lines 1999â€“2010) then calls `refund_icp` on failure (line 2005), returning the ICP to the attacker minus only transfer fees. There is no rollback path for the limiter in either `deposit_cycles` or `process_top_up`.

## Impact Explanation
The `DEFAULT_CYCLES_LIMIT` is 150P/hour (line 83). A single `notify_top_up` call for ~1500 ICP targeting a non-existent canister exhausts this limit. All subsequent `notify_top_up`, `notify_create_canister`, and `notify_mint_cycles` calls routed through `ensure_balance` with `BaseLimit` will be rejected with "try again later" for up to one hour. This is a concrete application/platform-level DoS on the NNS CMC â€” a core financial infrastructure canister â€” matching the High ($2,000â€“$10,000) impact class. A partial self-limiting factor exists: after the first attack, the CMC retains the 150P minted cycles in its own balance, so `cycles_to_mint = cycles - current_balance` (line 2313) would be near zero on a second attack until legitimate operations drain the CMC's balance. The first-hour DoS is fully achievable regardless.

## Likelihood Explanation
No privileged access is required â€” any principal can call `notify_top_up`. The attacker needs ~1500 ICP upfront (refunded minus ~0.0002 ICP in fees). A non-existent canister ID is trivially constructable. The `blocks_notified` deduplication (lines 1181â€“1206) prevents reuse of the same block index, but each new ICP transfer creates a new block index, making the attack repeatable each hour once the CMC's retained balance is drained by legitimate operations.

## Recommendation
Move `ensure_balance` (limiter charge + `ic0_mint_cycles128`) to after the management canister call succeeds, or implement a rollback: if `call_with_payment128` returns an error, decrement `base_limiter` by the previously charged amount and subtract from `total_cycles_minted`. The invariant must be: the limiter is only charged when cycles are verifiably delivered to the target canister.

## Proof of Concept
State-machine test:
1. Configure CMC with a mock management canister that always rejects `deposit_cycles`.
2. Send 1500 ICP to the CMC subaccount for a non-existent target canister ID.
3. Call `notify_top_up` with that block index.
4. Assert: `base_limiter.get_count()` equals 150P (limit exhausted).
5. Assert: attacker's ICP balance is restored minus ~0.0002 ICP in fees.
6. Assert: a legitimate `notify_top_up` from a different user now returns `NotifyError::Refunded` with "try again later".
