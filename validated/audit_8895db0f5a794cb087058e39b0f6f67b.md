### Title
Missing G2 Subgroup Check in BN254 ecpairing Precompile Allows False-Positive Pairing Results — (`basic_system/src/system_functions/bn254_pairing_check.rs`)

---

### Summary

The BN254 `ecpairing` precompile implementation in `bn254_pairing_check_inner` validates G2 points only for curve membership (via `check()`), but never calls `is_in_correct_subgroup_assuming_on_curve()`. This violates EIP-197, which mandates rejection of inputs not in the prime-order subgroup. An unprivileged caller can supply a G2 point that lies on the BN254 twist curve but belongs to the cofactor subgroup (order dividing h₂, not r). The Ate pairing maps any such cofactor point to 1 in GT for all G1 inputs, causing the precompile to return `true` for semantically invalid pairing equations.

---

### Finding Description

In `bn254_pairing_check_inner`, G2 point validation is:

```rust
let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
g2_point.check().map_err(|_| ())?;   // ← only curve equation check
``` [1](#0-0) 

The `Valid::check()` method from arkworks for `Affine<SWCurveConfig>` only verifies that the point satisfies the curve equation y² = x³ + b over Fq2. It does **not** verify prime-order subgroup membership.

The subgroup check IS implemented in the codebase:

```rust
fn is_in_correct_subgroup_assuming_on_curve(point: &G2Affine) -> bool {
    // Checks that [p]P = [6X^2]P
    let x_times_point = point.mul_bigint(SIX_X_SQUARED);
    let p_times_point = p_power_endomorphism(point);
    x_times_point.eq(&p_times_point)
}
``` [2](#0-1) 

But it is never called in the pairing precompile path. By contrast, the BLS12-381 pairing precompile correctly calls `parse_g2_with_subgroup_check` which explicitly invokes `is_in_correct_subgroup_assuming_on_curve()`: [3](#0-2) 

The BN254 G2 cofactor is:

```
h₂ = 21888242871839275222246405745257275088844257914179612981679871602714643921549
``` [4](#0-3) 

This is a large non-trivial value, so the cofactor subgroup contains many non-identity points. For any cofactor point Q_cofactor = [r]Q′ (where r is the prime order), the Ate pairing satisfies:

```
e(P, Q_cofactor) = e(P, [r]Q′) = e(P, Q′)^r = 1  (since GT has order r)
```

Therefore, the precompile computes `result.0.is_one()` as `true` for any G1 input paired with a cofactor G2 point: [5](#0-4) 

---

### Impact Explanation

Any smart contract using `ecpairing` (precompile `0x08`) for signature verification or ZK proof verification can be bypassed. An attacker constructs a cofactor G2 point Q_cofactor (on the curve, passes `check()`, but not in G2), submits it as a pairing input, and the precompile returns `1` (true) regardless of the G1 input. This allows forging pairing equations — for example, making `e(A, Q_cofactor) * e(B, Q_cofactor) = 1` hold for arbitrary A, B — which breaks any protocol relying on ecpairing for cryptographic soundness.

---

### Likelihood Explanation

The attack is fully unprivileged: it requires only crafting a valid transaction calling a contract that invokes ecpairing. Computing a cofactor G2 point is straightforward: take any point Q on E′(Fq2) not in G2 and compute [r]Q. The coordinates are public knowledge. No special access, leaked keys, or governance control is needed.

---

### Recommendation

After constructing the G2 point and calling `check()`, add an explicit subgroup check:

```rust
let g2_point = G2Affine::new_unchecked(g2_x, g2_y);
g2_point.check().map_err(|_| ())?;
// Add this:
if !<crypto::bn254::curves::g2::Config as SWCurveConfig>
    ::is_in_correct_subgroup_assuming_on_curve(&g2_point) {
    return Err(());
}
```

This mirrors the pattern already used correctly in the BLS12-381 pairing precompile. [6](#0-5) 

---

### Proof of Concept

1. Pick any generator Q of E′(Fq2) that is not in G2 (e.g., a random point on the twist curve).
2. Compute Q_cofactor = [r]Q. This point is on the curve (passes `check()`), has order dividing h₂, and is NOT in the prime-order subgroup G2.
3. Encode the pair (G1_gen, Q_cofactor) as a 192-byte input per EIP-197 encoding.
4. Call the ecpairing precompile with this input.
5. **Expected per EIP-197**: revert (invalid input — point not in subgroup).
6. **Actual**: returns `0x0000...0001` (true), because e(G1_gen, Q_cofactor) = 1 in GT.

The question's proof idea incorrectly predicts the result should be `false`; the actual result is `true` (1), which is the false positive that enables the exploit. Any contract checking `ecpairing(...) == 1` as a validity condition can be trivially satisfied with cofactor points.
