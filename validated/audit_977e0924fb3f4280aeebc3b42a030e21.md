### Title
Single-Path L1 Deposit Processing with No Fallback Bricks Block Execution When `L2AssetTracker` Reverts — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

Every L1→L2 deposit transaction unconditionally calls `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` through `notify_l2_asset_tracker`. If that call reverts for any reason, the bootloader treats it as a fatal, unrecoverable error and halts block processing entirely. There is no fallback path. This is structurally identical to the Reth.sol issue: a single external dependency with no alternative, whose failure bricks the protocol.

---

### Finding Description

In `process_l1_transaction.rs`, the function `notify_l2_asset_tracker` is invoked for every L1 deposit with `amount > 0`. It encodes a call to `handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)` on the `L2AssetTracker` contract and executes it via `run_single_interaction`. If the call returns a failure result, the code explicitly propagates a fatal `BootloaderSubsystemError` with no recovery path:

```rust
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    )
    .into());
}
``` [1](#0-0) 

The `l1_chain_id` passed to `handleFinalizeBaseTokenBridgingOnL2` is read from `L2AssetTracker` storage slot 154 via `read_l1_chain_id`. If that slot is zero (uninitialized or reset), the call is made with `_fromChainId = 0`. Any guard in the `L2AssetTracker` Solidity contract that rejects a zero chain ID (e.g., `require(_fromChainId != 0)`) would cause an immediate revert, triggering the fatal path. [2](#0-1) 

`notify_l2_asset_tracker` is called up to three times per L1 transaction — for the value mint, the operator fee, and the refund — meaning any of these three calls can trigger the fatal halt. [3](#0-2) 

The code comment acknowledges the single-path design explicitly:

> *"Failure halts block processing — if the asset tracker reverts, the chain's token accounting would be inconsistent, so we treat it as fatal rather than silently continuing with incorrect bookkeeping."* [4](#0-3) 

There is no alternative mechanism, no graceful degradation, and no retry path.

---

### Impact Explanation

If `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` reverts for any reason — uninitialized storage (`l1_chain_id == 0`), a specific deposit amount triggering an internal guard, a contract upgrade introducing a regression, or any other revert condition — the bootloader returns a `BootloaderSubsystemError` that propagates up and prevents the block from being finalized. All subsequent blocks are also blocked. The entire L1→L2 deposit flow is permanently bricked until the `L2AssetTracker` contract is fixed and redeployed, which itself requires a governance upgrade.

The existing test `test_treasury_insufficient_balance_failure` confirms that a `BootloaderSubsystemError` from within L1 transaction processing causes `execute_block_no_panic` to return `Err`, i.e., a block-level failure, not a transaction-level revert. [5](#0-4) 

---

### Likelihood Explanation

The attacker-controlled entry path is straightforward: submit any L1 deposit transaction with `amount > 0`. This unconditionally invokes `notify_l2_asset_tracker`. The trigger condition depends on `L2AssetTracker` internals, but two concrete scenarios exist without requiring privileged access:

1. **Uninitialized `L1_CHAIN_ID`**: If `L2AssetTracker` storage slot 154 is zero (e.g., before initialization or after a storage reset), `read_l1_chain_id` returns 0, and `handleFinalizeBaseTokenBridgingOnL2(0, amount)` is called. Any `require(_fromChainId != 0)` guard in the Solidity contract causes a revert.

2. **Amount-dependent revert**: The `amount` parameter is fully user-controlled (it is the deposit value). If `L2AssetTracker` has any overflow check, maximum deposit cap, or arithmetic that reverts for specific amounts, an attacker can craft a deposit to trigger it.

The code comment itself acknowledges the fragility: *"we are certain that L2AssetTracker is available after the upgrade"* — the same reasoning dismissed in the Reth.sol report ("the deposit pool is mostly always full"). [6](#0-5) 

---

### Recommendation

Mirror the fix recommended in the external report: add a fallback path. Specifically:

1. **Do not treat `L2AssetTracker` revert as a fatal block-level error.** Instead, log the failure and continue block processing. The asset tracker is an accounting helper, not a gating mechanism for deposit validity.
2. **Alternatively**, if the asset tracker revert must be treated as fatal for accounting correctness, add a pre-call check (analogous to `RocketDepositPool.getExcessBalance`) to verify the `L2AssetTracker` is in a callable state before invoking it, and skip the call if it is not.
3. **At minimum**, wrap the `notify_l2_asset_tracker` call in a recoverable error handler that allows the block to continue with a logged warning rather than a hard halt.

---

### Proof of Concept

1. Deploy ZKsync OS with `L2AssetTracker` at `L2_ASSET_TRACKER_ADDRESS` but with storage slot 154 (`L1_CHAIN_ID`) unset (zero).
2. Submit an L1 deposit transaction with `to_mint > 0` and `amount > 0`.
3. The bootloader calls `read_l1_chain_id`, which reads slot 154 and returns `U256::ZERO`.
4. `notify_l2_asset_tracker` encodes `handleFinalizeBaseTokenBridgingOnL2(0, amount)` and calls it.
5. If `L2AssetTracker` has `require(_fromChainId != 0)`, the call reverts.
6. `asset_tracker_result.failed()` returns `true`.
7. The bootloader returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
8. Block execution fails entirely — confirmed by the pattern in `test_treasury_insufficient_balance_failure` where a `BootloaderSubsystemError` from L1 tx processing causes `execute_block_no_panic` to return `Err`. [7](#0-6)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
