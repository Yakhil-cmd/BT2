### Title
Inaccurate `balance()` View Function Does Not Account for Minimum Withdrawal Constraint - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
The `EVM.CadenceOwnedAccount.balance()` and `EVM.EVMAddress.balance()` view functions return the raw attoflow balance without accounting for the minimum withdrawal constraint of 1e10 attoflow enforced by `withdraw()`. This creates a discrepancy between what the view functions report and what can actually be withdrawn, misleading users and causing failed transactions — a direct analog to the sUSX `maxWithdraw`/`maxRedeem` not accounting for `mintCap`.

### Finding Description
In `fvm/evm/stdlib/contract.cdc`, both `EVMAddress.balance()` and `CadenceOwnedAccount.balance()` are `view` functions that return the raw attoflow balance with no constraint checking:

```cadence
access(all)
view fun balance(): Balance {
    return Balance(attoflow: InternalEVM.balance(address: self.addressBytes))
}
``` [1](#0-0) 

The underlying `InternalEVM.balance` host function in `fvm/evm/impl/impl.go` simply reads the raw EVM state balance with no floor check:

```go
return interpreter.UIntValue{BigInt: account.Balance()}
``` [2](#0-1) 

However, the `withdraw()` function enforces a hard minimum of 1e10 attoflow (the smallest UFix64-representable FLOW unit, 1e-8 FLOW) via `types.AttoFlowBalanceIsValidForFlowVault`:

```go
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
``` [3](#0-2) 

Where the constraint is defined as:

```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
``` [4](#0-3) 

`UFixToAttoConversionMultiplier` is 1e10 (= 10^(18-8)). [5](#0-4) 

The same rounding check is also enforced at the emulator level in `fvm/evm/emulator/emulator.go`:

```go
if !types.AttoFlowBalanceIsValidForFlowVault(call.Value) {
    return types.NewInvalidResult(..., types.ErrWithdrawBalanceRounding), nil
}
``` [6](#0-5) 

If an EVM account holds any balance between 1 and 9,999,999,999 attoflow, `balance()` reports a non-zero value, but every withdrawal attempt will revert. The view function provides no indication that the reported balance is below the minimum withdrawable threshold.

The `withdraw()` Cadence function itself documents the constraint only in a comment, not in any queryable way:

```cadence
/// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
/// given that Flow Token Vaults use UFix64 to store balances.
``` [7](#0-6) 

### Impact Explanation
Any user or contract that queries `balance()` to determine how much they can withdraw receives an inaccurate result when their balance is below 1e10 attoflow. Transactions attempting to withdraw the reported balance will revert with `ErrWithdrawBalanceRounding`. This is a bridge escrow mis-accounting analog: the view layer reports assets as accessible when the execution layer will reject the withdrawal. No unauthorized asset access occurs, but users are misled about their withdrawable balance, causing failed transactions and potential confusion about fund availability.

### Likelihood Explanation
This is reachable by any unprivileged user who:
1. Deposits a small amount of FLOW into an EVM account (COA or direct `EVMAddress.deposit()`). The deposit path has no minimum floor — any UFix64 amount is accepted.
2. Queries `balance()` to check their balance before withdrawing.
3. Attempts to withdraw the reported balance.

A user can also end up with a sub-1e10 attoflow residual balance after a partial withdrawal (e.g., withdrawing a truncated UFix64 amount leaves a remainder). The `balance()` view function will report this residual as available, but it can never be withdrawn.

### Recommendation
Add a `maxWithdrawable()` view function that rounds the raw balance down to the nearest 1e10 attoflow multiple, accurately reflecting the maximum amount that can be withdrawn:

```cadence
access(all)
view fun maxWithdrawable(): Balance {
    let raw = InternalEVM.balance(address: self.addressBytes)
    // Round down to nearest 1e10 attoflow (UFix64 precision floor)
    let aligned = raw - (raw % 10000000000)
    return Balance(attoflow: aligned)
}
```

Alternatively, document the constraint directly in the `balance()` return value or emit a warning when the balance is below the minimum threshold.

### Proof of Concept
1. A user deposits 5,000,000,000 attoflow (5e9 < 1e10) into a COA.
2. `cadenceOwnedAccount.balance()` returns `Balance(attoflow: 5000000000)` — a non-zero value suggesting funds are available.
3. The user attempts: `cadenceOwnedAccount.withdraw(balance: EVM.Balance(attoflow: 5000000000))`
4. The transaction panics: `"withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"` (`ErrWithdrawBalanceRounding`).
5. The `balance()` view function gave no indication this withdrawal would fail.

This is confirmed by the existing test at `fvm/evm/evm_test.go` lines 2274–2338, which demonstrates that withdrawing 9,999,999,999 attoflow fails with `ErrWithdrawBalanceRounding` while `balance()` would have reported that amount as the account's balance: [8](#0-7)
