### Title
Missing Emitter-Address Verification in Event Hooks Allows Unauthorized System State Updates - (`system_hooks/src/event_hooks/system_context.rs`, `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `system_context_event_hook` and `interop_root_reporter_event_hook` event hooks update critical system state (settlement layer chain ID and interop roots) when a matching LOG event is emitted from a registered system contract address. Unlike every call hook in the codebase — which explicitly checks `caller != EXPECTED_ADDRESS` before acting — the event hooks perform no emitter-identity verification. Furthermore, the `SystemEventHook` function signature does not even receive the emitter address, making it structurally impossible to add such a check. Any contract at the registered address that emits the matching event signature will unconditionally trigger the hook and mutate system state.

---

### Finding Description

**Root cause — missing caller check in event hooks:**

All four call hooks explicitly guard against unauthorized callers:

- `l1_messenger_hook`: `if caller != L1_MESSENGER_ADDRESS { return empty; }` [1](#0-0) 
- `set_bytecode_on_address_hook`: `if caller != CONTRACT_DEPLOYER_ADDRESS && caller != COMPLEX_UPGRADER_ADDRESS { return empty; }` [2](#0-1) 
- `mint_base_token_hook`: `if caller != L2_BASE_TOKEN_ADDRESS { return empty; }` [3](#0-2) 
- `contract_deployer_temp_hook`: `if caller != COMPLEX_UPGRADER_ADDRESS { return empty; }` [4](#0-3) 

The event hooks have no equivalent guard. `system_context_event_hook` calls `system.io.update_settlement_layer_chain_id(...)` with the chain ID taken directly from `topics[1]`, with no check on who emitted the event: [5](#0-4) 

`interop_root_reporter_event_hook` calls `system.io.add_interop_root(...)` with root, chain_id, and block_or_batch_number taken directly from topics/data, with no check on who emitted the event: [6](#0-5) 

**Structural impossibility of adding the check:**

The `SystemEventHook` function signature does not include the emitter address as a parameter — only `topics`, `data`, `caller_ee` (execution environment type byte), `system`, and `resources` are passed: [7](#0-6) 

The dispatch mechanism in `try_intercept_event` uses the emitter address only as a lookup key to find the hook, but does not forward it into the hook function: [8](#0-7) 

The hook is invoked from `System::emit_event`, which passes `ee_type as u8` (not the emitter address) as `caller_ee`: [9](#0-8) 

**Exploit path:**

The `system_context_event_hook` is registered for `SYSTEM_CONTEXT_ADDRESS_LOW`: [10](#0-9) 

The `interop_root_reporter_event_hook` is registered for `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW`: [11](#0-10) 

The `setSettlementLayerChainId` function is intended to be called only via service transactions (type `0x7d`), as reflected in the `SERVICE_DESTINATION_WHITELIST`: [12](#0-11) 

However, this whitelist only restricts service transaction routing. It does not prevent a regular EVM transaction from calling the SystemContext contract at `SYSTEM_CONTEXT_ADDRESS` directly. If the deployed SystemContext Solidity contract exposes `setSettlementLayerChainId` (or any other function that emits `SettlementLayerChainIdUpdated`) without an access control guard, any unprivileged user can:

1. Send a regular EVM transaction calling `setSettlementLayerChainId(arbitrary_chain_id)` on `SYSTEM_CONTEXT_ADDRESS`
2. The contract emits `SettlementLayerChainIdUpdated(arbitrary_chain_id)` as a LOG
3. `System::emit_event` dispatches to `system_context_event_hook` because the emitter address matches
4. The hook calls `update_settlement_layer_chain_id(arbitrary_chain_id)` with no further validation
5. The settlement layer chain ID is overwritten with an attacker-controlled value

The same path applies to `interop_root_reporter_event_hook` for injecting arbitrary interop roots.

---

### Impact Explanation

**Settlement layer chain ID manipulation:** The settlement layer chain ID is included in the batch public input and governs cross-chain messaging and L1 integration. An attacker setting it to an arbitrary value corrupts the batch commitment, breaks L1 verification, and can permanently disrupt the rollup's ability to settle on L1. This is a critical state-transition integrity failure.

**Interop root injection:** Injecting arbitrary interop roots with attacker-controlled `chain_id`, `block_or_batch_number`, and `root` values corrupts the interop root storage used for cross-chain interoperability. This can enable fraudulent cross-chain message acceptance or denial of legitimate messages.

---

### Likelihood Explanation

The likelihood is conditional on the SystemContext and L2InteropRootStorage Solidity contracts lacking access control on the event-emitting functions. The Rust-level hook provides zero defense-in-depth — unlike every call hook — so the entire security boundary rests on the Solidity contracts. The `SERVICE_DESTINATION_WHITELIST` in the Rust bootloader does not protect against direct EVM calls to these contracts. Any user with gas can submit a regular EVM transaction to `SYSTEM_CONTEXT_ADDRESS`.

---

### Recommendation

1. **Add the emitter address to the `SystemEventHook` function signature**, mirroring the `caller` parameter available in all call hooks. Pass the emitter address from `System::emit_event` through `try_intercept_event` into the hook function.

2. **Add explicit emitter-address checks inside each event hook**, analogous to the pattern used in all call hooks:
   - `system_context_event_hook`: verify emitter is `SYSTEM_CONTEXT_ADDRESS`
   - `interop_root_reporter_event_hook`: verify emitter is `L2_INTEROP_ROOT_STORAGE_ADDRESS`

3. **Ensure the Solidity contracts** at `SYSTEM_CONTEXT_ADDRESS` and `L2_INTEROP_ROOT_STORAGE_ADDRESS` enforce that only the bootloader or a service transaction can invoke the event-emitting functions.

---

### Proof of Concept

```
1. Deploy or use the existing SystemContext contract at SYSTEM_CONTEXT_ADDRESS.
2. Confirm setSettlementLayerChainId(uint256) is callable without access control.
3. Submit a regular EVM transaction:
     from: attacker_eoa
     to:   SYSTEM_CONTEXT_ADDRESS (0x000000000000000000000000000000000000800b)
     data: abi.encodeCall(setSettlementLayerChainId, (999999))
4. The contract emits SettlementLayerChainIdUpdated(999999).
5. system_context_event_hook fires (no caller check), calls
   update_settlement_layer_chain_id(999999).
6. The settlement layer chain ID is now 999999 — attacker-controlled.
7. The next batch public input includes the corrupted chain ID,
   breaking L1 verification.
``` [13](#0-12) [14](#0-13)
