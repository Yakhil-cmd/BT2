### Title
Blob Gas Fee Collected from Sender but Never Distributed to Operator — (`basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`)

### Summary

In `ZkTransactionFlow::refund_and_commit_fee`, the blob gas fee (`fee_for_blob_gas`) is charged from the sender upfront as part of `fee_to_prepay`, but is never distributed to the coinbase/operator during the refund phase. This is directly analogous to the Caviar flashloan bug: a fee is collected from the user but the intended recipient (the operator) receives nothing from it. The inconsistency is sharpest when the `burn_base_fee` feature is disabled — in that configuration, all execution gas fees go to the operator, but blob gas fees are silently absorbed into the system without being credited to anyone.

---

### Finding Description

**Step 1 — Blob gas fee is charged upfront during validation.**

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, for EIP-4844 transactions, `fee_for_blob_gas` is computed and added to `fee_to_prepay`:

```
fee_to_prepay = gas_fee_amount + fee_for_blob_gas
``` [1](#0-0) 

**Step 2 — The full `fee_to_prepay` (including blob gas) is deducted from the sender.**

`precharge_fee` in `zk/mod.rs` deducts the entire `fee_to_prepay` from the sender's balance, which includes the blob gas component. [2](#0-1) 

**Step 3 — `refund_and_commit_fee` never distributes the blob gas fee.**

In `refund_and_commit_fee`, the function:
- Refunds unused execution gas to the sender: `gas_price * (tx_gas_limit - gas_used)`
- Pays the operator: `gas_price_for_operator * gas_used`

The blob gas fee is never refunded to the sender (correct per EIP-4844) and is **never paid to the coinbase/operator**. It simply disappears from the accounting. [3](#0-2) 

**Step 4 — The `burn_base_fee` flag creates a direct inconsistency.**

When `burn_base_fee` is disabled, `gas_price_for_operator = gas_price` — meaning the operator is supposed to receive the full gas price for execution. Yet blob gas fees are always burned regardless of this flag:

```rust
let gas_price_for_operator = if cfg!(feature = "burn_base_fee") {
    context.gas_price.saturating_sub(base_fee)
} else {
    context.gas_price  // operator gets everything
};
// blob gas fee: never handled here
``` [4](#0-3) 

The same pattern exists in the Ethereum flow, where `blob_gas_used` is tracked in `EthereumTxContext` but never used in fee distribution: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

For every EIP-4844 transaction with blobs processed by `ZkTransactionFlow`:

- The sender pays `blob_base_fee * blob_gas_used` as part of `fee_to_prepay`.
- This amount is deducted from the sender's balance and never credited to the operator/coinbase.
- When `burn_base_fee` is disabled (ZK rollup mode where the operator is supposed to receive all fees), the operator loses the entire blob gas fee revenue.
- The tokens are effectively destroyed without any explicit burn mechanism, causing a permanent reduction in circulating supply.

In a ZK rollup context, the operator posts blobs to L1 and bears the cost. The blob gas fee is the mechanism to compensate the operator for this cost. Silently burning it means the operator is under-compensated for every blob-carrying transaction.

---

### Likelihood Explanation

Any unprivileged user can submit an EIP-4844 (type 3) transaction with blob versioned hashes. The ZK validation path explicitly parses blobs and computes `fee_for_blob_gas`. The bug is triggered unconditionally for every such transaction — no special conditions are required. The attacker-controlled entry path is simply submitting a valid EIP-4844 transaction to the ZKsync OS sequencer. [7](#0-6) 

---

### Recommendation

In `ZkTransactionFlow::refund_and_commit_fee`, after paying the operator for execution gas, also distribute the blob gas fee to the coinbase when `burn_base_fee` is disabled. When `burn_base_fee` is enabled, the blob gas fee can remain burned (consistent with EIP-4844 semantics). Concretely:

```rust
// After paying execution gas to operator...
if !cfg!(feature = "burn_base_fee") && context.blob_gas_used > 0 {
    let blob_fee = system.get_blob_base_fee_per_gas()
        .checked_mul(U256::from(context.blob_gas_used))
        .ok_or(internal_error!("blob fee overflow"))?;
    // pay blob_fee to coinbase
}
```

The same fix should be evaluated for `EthereumTransactionFlow::refund_and_commit_fee` depending on whether EVM-equivalent blob fee burning is intended there.

---

### Proof of Concept

1. Submit an EIP-4844 transaction with 1 blob (`blob_gas_used = GAS_PER_BLOB = 131072`) and `blob_base_fee = 1000`.
2. `fee_to_prepay` includes `131072 * 1000 = 131,072,000` extra tokens for blob gas.
3. After execution, `refund_and_commit_fee` refunds unused execution gas and pays `priority_fee * gas_used` to coinbase.
4. The `131,072,000` blob gas tokens are deducted from the sender but never credited to the coinbase.
5. With `burn_base_fee` disabled, the operator receives `gas_price * gas_used` for execution but receives `0` for blob gas — the blob fee is silently lost. [8](#0-7) [9](#0-8)

    fn refund_and_commit_fee<Config: BasicBootloaderExecutionConfig>(
        system: &mut System<S>,
        transaction: &Transaction<<S as SystemTypes>::Allocator>,
        context: &mut Self::TransactionContext,
        _tracer: &mut impl Tracer<S>,
    ) -> Result<(), BootloaderSubsystemError> {
        // here we refund the user, then we will transfer fee to the operator

        if context.tx_gas_limit > context.gas_used {
            system_log!(system, "Gas price for refund is {:?}\n", &context.gas_price);

            // refund
            let refund_recipient = transaction.from();
            let token_to_refund =
                context.gas_price * U256::from(context.tx_gas_limit - context.gas_used); // can not overflow

            // First refund the sender. Routed through `intrinsic_resources` so
            // the native charge (precharged by the intrinsic formula) can be
            // verified under `verify_intrinsic_native`.
            context
                .intrinsic_resources
                .with_infinite_ergs(|resources| {
                    system.io.update_account_nominal_token_balance(
                        ExecutionEnvironmentType::NoEE,
                        resources,
                        &refund_recipient,
                        &token_to_refund,
                        false,
                        Config::SIMULATION,
                    )
                })
                .map_err(|e| match e {
                    // Balance errors can not be cascaded
                    SubsystemError::Cascaded(CascadedError(inner, _)) => match inner {},
                    SubsystemError::LeafUsage(InterfaceError(ie, _)) => match ie {
                        BalanceError::InsufficientBalance => {
                            unreachable!("Cannot be insufficient when incrementing balance")
                        }
                        BalanceError::Overflow => {
                            interface_error!(BootloaderInterfaceError::CantPayRefundOverflow)
                        }
                    },
                    other => wrap_error!(other),
                })?;
        }

        // Next we pay the operator
        // ARCHITECTURE NOTE: Fee payment is split into two phases:
        // 1. Deduct full fee from sender at transaction start (in pay_for_transaction)
        // 2. Transfer actual payment to operator after execution (here)
        // This ensures sender has sufficient funds before execution begins

        // EIP-1559 compatibility: When burn_base_fee is enabled, only priority fees
        // go to the operator. Base fees are effectively "burned" (not transferred anywhere).
        let gas_price_for_operator = if cfg!(feature = "burn_base_fee") {
            let base_fee = system.get_eip1559_basefee();
            // We use saturating arithmetic to allow the caller of this method to
            // allow gas_price < base_fee. This can be used, for example, for
            // transaction simulation
            context.gas_price.saturating_sub(base_fee)
        } else {
            context.gas_price
        };

        system_log!(
            system,
            "Gas price for coinbase fee is {:?}\n",
            &gas_price_for_operator
        );

        let token_to_pay_operator = U256::from(context.gas_used)
            .checked_mul(gas_price_for_operator)
            .ok_or(internal_error!("gu*gpfo"))?;

        let coinbase = system.get_coinbase();
        // Operator payment native is precharged by the intrinsic formula too.
        context
            .intrinsic_resources
            .with_infinite_ergs(|resources| {
                system.io.update_account_nominal_token_balance(
                    ExecutionEnvironmentType::NoEE,
                    resources,
                    &coinbase,
                    &token_to_pay_operator,
                    false,
                    Config::SIMULATION,
                )
            })
            .map_err(|e| match e {
                // Balance errors can not be cascaded
                SubsystemError::Cascaded(CascadedError(inner, _)) => match inner {},
                SubsystemError::LeafUsage(InterfaceError(ie, _)) => match ie {
                    BalanceError::InsufficientBalance => {
                        unreachable!("Cannot be insufficient when incrementing balance")
                    }
                    BalanceError::Overflow => {
                        interface_error!(BootloaderInterfaceError::CantPayOperatorOverflow)
                    }
                },
                other => wrap_error!(other),
            })?;

        Ok(())
    }
