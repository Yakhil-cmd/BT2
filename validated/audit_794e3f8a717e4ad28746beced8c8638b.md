### Title
WalletConnect `create_offer_for_ids` Royalty Amount Spoofing via Dapp-Controlled `driver_dict` — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
A paired WalletConnect dapp can supply a manipulated `driver_dict` with a falsified `royalty_percentage` for `create_offer_for_ids`. The GUI's confirmation dialog computes and displays `amountWithRoyalties` entirely from the dapp-controlled parameter, with no cross-check against the NFT's authoritative on-chain royalty data. The user approves the offer creation seeing an incorrect total cost; the actual royalty enforced on-chain when the offer is accepted differs from what was shown.

### Finding Description

`parseCommandDisplay` handles `chia_wallet.create_offer_for_ids` by calling `createOfferRoyaltyPercentages`, which reads royalty percentages exclusively from the dapp-supplied `params.driver_dict`: [1](#0-0) 

```
const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);
``` [2](#0-1) 

`createOfferRoyaltyPercentages` iterates over `walletDelta` keys and calls `royaltyPercentageForDriverInfo(driverDict[assetId])`, where `driverDict` is `params.driver_dict` verbatim from the dapp: [3](#0-2) 

The resulting royalty percentages feed `formatAmountWithRoyalties`, which computes the `amountWithRoyalties` string shown in the WalletConnect confirmation dialog: [4](#0-3) 

The confirmation dialog renders this value prominently under "Total Amount with Royalties": [5](#0-4) 

Contrast this with the `take_offer` path, where royalty percentages are fetched from `nftGetInfo` (a daemon call against on-chain data), overriding any value from the offer summary's `infos`: [6](#0-5) 

For `create_offer_for_ids` there is no equivalent authoritative lookup — the dapp's `driver_dict` value is used as-is.

### Impact Explanation

A malicious paired dapp sends `create_offer_for_ids` with `driver_dict` omitting or zeroing `royalty_percentage`. The user's confirmation dialog shows `amountWithRoyalties` equal to the base amount (e.g., `1 XCH`). The actual NFT royalty enforced on-chain when the offer is accepted is higher (e.g., `1.025 XCH`). The user approves the offer creation believing the total cost is `1 XCH`; when the offer is accepted their wallet is debited `1.025 XCH`. Conversely, a dapp could inflate the displayed royalty to deter the user from accepting a legitimate offer.

This is WalletConnect state spoofing that causes the user to approve an offer displaying the wrong amount — matching the High impact criterion.

### Likelihood Explanation

Requires a paired WalletConnect dapp (user has already approved the pairing). Any dapp with `chia_wallet.create_offer_for_ids` permission can trigger this. The `driver_dict` parameter is fully dapp-controlled and passes through `parseDappParams` without royalty-percentage bounds or cross-validation. [7](#0-6) 

### Recommendation

For `create_offer_for_ids`, resolve the royalty percentage for each NFT asset ID via `nftGetInfo` (the same authoritative path used by `take_offer`) and use that value to compute `amountWithRoyalties`, ignoring or only using the `driver_dict` value as a fallback when `nftGetInfo` fails. Add a cap check (`royalty_percentage <= 10000`) consistent with `parseRoyaltyPercentage`. [8](#0-7) 

### Proof of Concept

1. Pair a WalletConnect dapp with `chia_wallet.create_offer_for_ids` permission.
2. Send the following session request:
```json
{
  "method": "chia_wallet.create_offer_for_ids",
  "params": {
    "offer": { "1": "-1000000000000", "<nft_launcher_id>": "1" },
    "driver_dict": {
      "<nft_launcher_id>": {
        "type": "singleton",
        "launcher_id": "0x<nft_launcher_id>",
        "also": {
          "type": "metadata",
          "also": {
            "type": "ownership",
            "transfer_program": {
              "type": "royalty transfer program",
              "royalty_percentage": "0"
            }
          }
        }
      }
    }
  }
}
```
3. The confirmation dialog shows `Total Amount with Royalties: 1 XCH`.
4. The NFT's actual on-chain royalty (e.g., 2.5%) is enforced at offer acceptance, deducting `1.025 XCH` from the user's wallet — `0.025 XCH` more than displayed. [7](#0-6) [3](#0-2)

function createOfferRoyaltyPercentages(
  walletDelta: WalletDelta,
  driverDict: Record<string, unknown>,
): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  for (const assetId of Object.keys(walletDelta.spending)) {
    royaltyPercentages.spending[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  for (const assetId of Object.keys(walletDelta.receiving)) {
    royaltyPercentages.receiving[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  return royaltyPercentages;
}

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
