### Title
`setPoolAdminFeeDestination` redirects already-accrued admin fees to the new destination without flushing, causing loss of owed fees to the old fee destination — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` immediately without first collecting accrued fees. Every other fee-configuration mutator in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) flushes accrued fees to the current destination before updating configuration. `setPoolAdminFeeDestination` does not, so all fees that accrued while the old destination was active are silently redirected to the new destination when `collectPoolFees` is next called — which is permissionless and can be triggered by anyone.

### Finding Description

The factory stores the admin fee recipient in the mutable mapping `poolAdminFeeDestination[pool]`. [1](#0-0) 

When fees are collected, the pool sends the admin share to whatever address is currently in that slot at collection time: [2](#0-1) 

`setPoolAdminFees` and `setPoolProtocolFee` both call `collectFees` **before** updating configuration, ensuring fees accrued under the old config go to the correct destination: [3](#0-2) [4](#0-3) 

`setPoolAdminFeeDestination` does **not** follow this pattern — it directly overwrites the slot with no flush: [5](#0-4) 

`collectPoolFees` is permissionless, so any keeper or third party can trigger collection immediately after the destination change: [6](#0-5) 

The two independently-mutable surfaces — `poolAdminFeeDestination[pool]` (changed by `setPoolAdminFeeDestination`) and the accrued fee balance (accumulated by swaps) — drift out of sync under a normal admin rotation. The contract is the only place the invariant "fees accrued under destination D_A are paid to D_A" can be enforced, and it is not.

### Impact Explanation

`adminFeeDestination` is a distinct parameter from `admin` in `PoolParameters`, designed to be a separate treasury contract. [7](#0-6) 

When the admin rotates the treasury address (a routine operational action), all previously accrued but uncollected admin fees are redirected to the new treasury. The old treasury loses its entire uncollected balance — a direct, permanent loss of owed protocol fees proportional to the pool's trading volume since the last collection.

### Likelihood Explanation

Treasury rotation is a normal operational event (key rotation, multisig upgrade, entity restructuring). The documentation explicitly notes "Coordinate with treasury operations" for this function, confirming it is expected to be called in production. Because `setPoolAdminFees` auto-flushes, a reasonable admin would not expect `setPoolAdminFeeDestination` to require a manual pre-flush. The permissionless `collectPoolFees` means a keeper or MEV bot can cement the loss in the same block as the destination change.

### Recommendation

Add a `collectFees` flush at the start of `setPoolAdminFeeDestination`, mirroring the pattern in `setPoolAdminFees`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+       c.protocolSpreadFeeE6,
+       c.adminSpreadFeeE6,
+       c.protocolNotionalFeeE8,
+       c.adminNotionalFeeE8,
+       poolAdminFeeDestination[pool]   // flush to OLD destination first
+   );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

```
State:
  poolAdminFeeDestination[pool] = D_A  (old treasury)
  adminSpreadFeeE6 = 50_000            (5%)

Step 1 – Swaps occur, surplus accumulates.
  surplus0Scaled = 1_000_000 (scaled units)
  Admin share owed to D_A = 1_000_000 * 50_000 / (protocolFee + 50_000)
  ≈ 500_000 scaled → ~500 token0 at 1e18 scale

Step 2 – Admin calls setPoolAdminFeeDestination(pool, D_B).
  poolAdminFeeDestination[pool] = D_B
  No collectFees call. Accrued surplus unchanged.

Step 3 – Anyone calls collectPoolFees(pool).
  collectFees(..., adminFeeDestination_ = D_B)
  → 500 token0 transferred to D_B
  → D_A receives 0

Result: D_A loses 500 token0 it was owed. D_B receives fees it did not earn.
The vetoing-SC analog: D_A is SC_A (took the action / earned the fees);
D_B is SC_B (holds the live slot at resolution time / receives the consequence).
```
