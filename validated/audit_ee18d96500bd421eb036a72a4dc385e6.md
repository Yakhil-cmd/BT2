### Title
Unprivileged Caller Can Grief Staker by Calling `update_rewards` with `disable_rewards: true`, Permanently Preventing Block Reward Distribution - (`src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable with no access control. It accepts a `disable_rewards: bool` parameter. The global `last_reward_block` guard is written **before** rewards are distributed. An attacker can call `update_rewards(victim_staker, disable_rewards: true)` to consume the reward slot for the current block without distributing any rewards, permanently preventing the legitimate staker from receiving block rewards for that block.

---

### Finding Description

In `update_rewards` (`staking.cairo` lines 1448–1507), the function:

1. Checks `current_block_number > last_reward_block` (the guard).
2. Writes `last_reward_block = current_block_number` **before** distributing rewards.
3. If `disable_rewards == true`, returns immediately without distributing rewards. [1](#0-0) 

The function is exposed through the public `IStakingRewardsManager` interface with no role-based access control — `general_prerequisites()` only checks that the contract is unpaused and the caller is non-zero. [2](#0-1) [3](#0-2) 

The `last_reward_block` is a **single global** storage variable — once consumed for block N, no other call to `update_rewards` can succeed in block N. [4](#0-3) 

The ordering bug is the direct analog to the Maverick finding: in Maverick, the bin-movement state update is deferred to the **end** of the swap, allowing a zero-value swap to trigger it early. Here, the `last_reward_block` guard is written at the **beginning** of `update_rewards`, before the actual reward work is done. The `disable_rewards: true` flag acts as the "zero-value swap" — it triggers the state update (consuming the slot) while skipping the actual work (reward distribution).

---

### Impact Explanation

An attacker calls `update_rewards(staker_address, disable_rewards: true)` in block N. This:

- Sets `last_reward_block = N` (consuming the slot).
- Distributes zero rewards.
- Causes any subsequent call to `update_rewards` in block N to revert with `REWARDS_ALREADY_UPDATED`. [5](#0-4) 

The staker permanently loses their block rewards for block N. The rewards are never allocated via `update_unclaimed_rewards_from_staking_contract`, so they are irretrievably lost to the staker. [6](#0-5) 

**Impact**: Medium — Griefing with no profit motive but permanent damage to staker yield.

---

### Likelihood Explanation

Any unprivileged address can call `update_rewards` with `disable_rewards: true` targeting any valid, active staker. The only cost to the attacker is gas. The attack is repeatable every block. In the consensus rewards phase (V3), this directly and permanently denies block rewards to the targeted staker. [7](#0-6) 

---

### Recommendation

Add access control to `update_rewards` so that only an authorized caller (e.g., the staker themselves, their operational address, or a trusted consensus contract) can invoke it. Alternatively, restrict the `disable_rewards: true` path to privileged callers only, preventing unprivileged actors from consuming the reward slot without distributing rewards.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Block N is produced. Staker S is eligible for block rewards.
3. Attacker (any address) calls `update_rewards(S, disable_rewards: true)` in block N.
   - `last_reward_block` is set to N.
   - No rewards are distributed to S.
4. Staker S (or anyone on their behalf) calls `update_rewards(S, disable_rewards: false)` in block N.
   - Reverts: `current_block_number > last_reward_block` is false → `REWARDS_ALREADY_UPDATED`.
5. Staker S permanently loses block rewards for block N. [8](#0-7)
