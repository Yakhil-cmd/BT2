### Title
ETH Sent to System Hook Addresses by Unauthorized Callers is Permanently Locked — (`basic_bootloader/src/bootloader/runner.rs`, `system_hooks/src/call_hooks/`)

---

### Summary

When an EVM contract or EOA sends a non-zero `nominal_token_value` (ETH) to a system hook address (e.g., `0x7100`, `0x7001`, `0x7002`) while not being the authorized caller, the ETH is irrecoverably transferred to the hook address. The hook's "pretend to be an empty account" path returns a **success** result, which causes the bootloader to commit the transfer frame rather than rolling it back. The ETH is permanently locked at the system hook address with no withdrawal mechanism.

---

### Finding Description

The execution path in `execute_call` in `runner.rs` is:

1. A global snapshot frame is opened.
2. `perform_transfer_if_required` executes the ETH transfer from caller to callee (the hook address) **before** the hook is invoked.
3. `call_to_special_address_execute_callee_frame` is called, which invokes the hook.
4. If the hook returns a **success** result, `finish_global_frame(None)` is called — **committing** the transfer.
5. If the hook returns a **failure** result, `finish_global_frame(Some(&rollback_handle))` is called — **reverting** the transfer. [1](#0-0) 

The system hooks for unauthorized callers return **success** (pretend to be an empty account):

```rust
// mint_base_token_hook
if caller != L2_BASE_TOKEN_ADDRESS {
    // Pretend to be an empty account
    return Ok((
        make_return_state_from_returndata_region(available_resources, &[]),
        return_memory,
    ));
}
``` [2](#0-1) 

The same pattern exists in `l1_messenger_hook`, `set_bytecode_on_address_hook`, and `contract_deployer_temp_hook`: [3](#0-2) [4](#0-3) [5](#0-4) 

Because the hook returns `Ok(success_result)`, the frame commit path is taken:

```rust
self.system
    .finish_global_frame(if reverted {
        Some(&rollback_handle)
    } else {
        None   // <-- ETH transfer committed here
    })
``` [6](#0-5) 

The hook addresses (`0x7001`, `0x7002`, `0x7100`, `0x8006`) are pure Rust system hooks with no EVM bytecode. They cannot call SELFDESTRUCT or any other mechanism to recover ETH. Any ETH transferred to them is permanently locked. [7](#0-6) 

---

### Impact Explanation

A user or contract that accidentally sends ETH alongside a call to a system hook address (e.g., `CALL 0x7100 value=1 ether`) while not being the authorized caller will have their ETH permanently transferred to the hook address. There is no recovery path. This is a direct, irreversible loss of user funds.

The affected hook addresses are:
- `0x7001` — L1 Messenger hook
- `0x7002` — Set Bytecode on Address hook
- `0x7100` — Mint Base Token hook
- `0x8006` — Contract Deployer Temp hook

---

### Likelihood Explanation

Likelihood is **low-to-medium**. System hook addresses are not standard user-facing contract addresses, but they are in the low address range that EVM contracts can call with arbitrary `value`. A contract that iterates over addresses or a user who mistakenly sends ETH to a system address (e.g., confusing `0x800a` L2BaseToken with `0x7100` the mint hook) would permanently lose funds. The `0x7100` mint hook is particularly relevant because the L2BaseToken contract at `0x800a` is a user-facing address, and `0x7100` is its internal hook — a developer could confuse the two.

---

### Recommendation

In each hook's unauthorized-caller early-return path, check whether `nominal_token_value` is non-zero and return a **failure** result instead of a success result if ETH was attached. This causes `finish_global_frame(Some(&rollback_handle))` to be called, reverting the ETH transfer:

```rust
if caller != L2_BASE_TOKEN_ADDRESS {
    if nominal_token_value != U256::ZERO {
        // Revert to return ETH to caller
        return Ok((make_error_return_state(available_resources), return_memory));
    }
    // Pretend to be an empty account
    return Ok((
        make_return_state_from_returndata_region(available_resources, &[]),
        return_memory,
    ));
}
```

Apply the same fix to `l1_messenger_hook`, `set_bytecode_on_address_hook`, and `contract_deployer_temp_hook`.

---

### Proof of Concept

1. Deploy a contract with sufficient ETH balance.
2. From the contract, execute `CALL` to `0x7100` (MINT_HOOK_ADDRESS) with `value = 1 ether` and any calldata, from any address other than `0x800a`.
3. The call returns success (empty returndata).
4. Verify that `0x7100` now holds 1 ETH and the caller's balance decreased by 1 ETH.
5. Verify there is no mechanism to recover the ETH from `0x7100`.

The existing test `test_mint_base_token_hook_rejects_non_zero_value` only tests the **authorized** caller (`0x800a`) with non-zero value — it does not test the **unauthorized** caller with non-zero value, which is the vulnerable path. [8](#0-7)

#[test]
fn test_mint_base_token_hook_rejects_non_zero_value() {
    let mut tester = TestingFramework::new().with_minted_tokens_to_treasury();

    let l2_base_token_address = address!("000000000000000000000000000000000000800a");
    let mint_hook_address = address!("0000000000000000000000000000000000007100");
    let mint_amount = alloy::primitives::U256::from(3000000000000000000u64);
    let call_value = alloy::primitives::U256::from(1u64);

    tester.set_balance(l2_base_token_address, call_value);

    let initial_balance = tester
        .get_account_properties(&l2_base_token_address)
        .balance;

    let calldata = mint_amount.to_be_bytes::<32>().to_vec();

    let tx = L1TxBuilder::new()
        .from(l2_base_token_address)
        .to(mint_hook_address)
        .input(calldata)
        .value(call_value)
        .gas_price(0)
        .gas_limit(200_000)
        .build();

    let output = tester.execute_block(vec![tx]);
    let tx_result = output.tx_results[0]
        .as_ref()
        .expect("Mint hook call should be processed");
    assert!(
        !tx_result.is_success(),
        "Mint hook should fail when called with non-zero value"
    );

    let final_balance = tester
        .get_account_properties(&l2_base_token_address)
        .balance;

    let balance_delta = final_balance
        .checked_sub(initial_balance)
        .expect("Final balance should not be below initial balance");
    assert!(
        balance_delta <= call_value,
        "Mint amount should not be credited when value is non-zero"
    );
}
