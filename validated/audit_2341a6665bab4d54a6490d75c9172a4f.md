### Title
Gateway Stateful Validator Skips `__validate__` for Invoke Transactions with Nonce=1 When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful path skips the `__validate__` entry point (account signature verification) for any invoke transaction with `tx.nonce() == 1` and `account_nonce == 0` whenever `account_tx_in_pool_or_recent_block` returns `true`. The guard is intended to support the UX pattern of submitting `deploy_account + invoke` atomically, but the mempool check is too broad: it returns `true` for **any** pending transaction from that account, not only a `deploy_account`. An already-deployed account (class hash set, on-chain nonce = 0) that has a legitimate nonce-0 invoke in the mempool satisfies the condition, allowing an attacker to submit a nonce-1 invoke with a garbage signature that bypasses `__validate__` and is admitted to the mempool.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` contains the following logic:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
``` [1](#0-0) 

When this returns `true`, `skip_validate = true` is propagated to `run_validate_entry_point`, which sets `execution_flags.validate = !skip_validate = false`, causing the blockifier's `StatefulValidator` to skip the `__validate__` call entirely for that transaction. [2](#0-1) 

The mempool's `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or any recently committed block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is incorrect. An already-deployed account (class hash set, on-chain nonce = 0) can have a legitimate nonce-0 invoke in the mempool. That invoke is not a `deploy_account`, and its presence does not imply the new nonce-1 transaction has a valid signature.

The nonce validation in `validate_nonce` permits `account_nonce (0) <= incoming_tx_nonce (1) <= max_allowed_nonce`, so the nonce-1 invoke passes the pre-check: [4](#0-3) 

After `skip_validate = true` is returned, `run_validate_entry_point` builds an `AccountTransaction` with `validate: false` and calls `blockifier_validator.validate(account_tx)`, which in `StatefulValidator::perform_validations` short-circuits at `if !tx.execution_flags.validate { return Ok(()); }` without ever calling `__validate__`: [5](#0-4) 

The transaction with the invalid signature is then forwarded to the mempool via `mempool_client.add_tx(add_tx_args)`. [6](#0-5) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An unprivileged attacker can inject invoke transactions with arbitrary (invalid) signatures into the mempool. The corrupted value is `skip_validate = true` when it should be `false`, causing `execution_flags.validate = false` for a transaction whose signer has not been verified. The transaction is admitted with a valid-looking `tx_hash` (computed from the chain_id and transaction fields) but an unverified signature. When the batcher later executes the transaction, `__validate__` will run and the transaction will revert, but the admission invariant — every accepted invoke must have passed signature verification — is broken at the gateway boundary.

### Likelihood Explanation

The attack requires:
1. An account that is already deployed (has a class hash) with on-chain nonce = 0 (i.e., it has never sent a committed transaction).
2. Any legitimate nonce-0 invoke from that account already in the mempool.
3. Submission of a nonce-1 invoke with a garbage signature.

All three conditions are achievable by any unprivileged user. The attacker can be the same party who submitted the nonce-0 invoke, or a third party who observes the mempool. The `max_allowed_nonce_gap` default of 200 means nonce-1 is always within the accepted range when account_nonce = 0. [7](#0-6) 

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** for the sender address is pending in the mempool. Alternatively, add a dedicated `deploy_account_in_pool(address)` mempool query that only returns `true` when the pending transaction is of type `DeployAccount`. This preserves the intended UX shortcut while closing the bypass for already-deployed accounts.

### Proof of Concept

1. Deploy account `A` (class hash set, on-chain nonce = 0).
2. Submit a legitimate invoke with nonce = 0 from `A` → gateway accepts it, mempool now contains `A`'s nonce-0 invoke.
3. Craft an invoke transaction for `A` with nonce = 1 and a garbage signature (e.g., `[Felt::ONE, Felt::TWO]`).
4. Submit to gateway. Gateway path:
   - `validate_nonce`: `0 <= 1 <= 200` → passes.
   - `validate_by_mempool`: mempool accepts (no duplicate nonce).
   - `skip_stateful_validations`: `tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` → returns `true`.
   - `run_validate_entry_point`: `execution_flags.validate = false` → `__validate__` is never called.
5. Gateway calls `mempool_client.add_tx(...)` → nonce-1 invoke with garbage signature is now in the mempool.
6. Batcher retrieves and attempts to execute the transaction; `__validate__` runs and the transaction reverts, but the invalid transaction was admitted. [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
