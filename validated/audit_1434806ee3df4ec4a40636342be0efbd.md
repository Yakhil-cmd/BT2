### Title
COA Ownership Proof Signature Reuse Across Different EVM Addresses - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` does not enforce that the caller-supplied `signedData` encodes the target `evmAddress`. A valid signature produced for one COA-ownership context (e.g., for COA address `0xAAAA`) can be replayed verbatim against a different COA address (`0xBBBB`) owned by the same Cadence account, because the cryptographic check is performed only over `signedData` — which is fully attacker-controlled and need not contain `evmAddress`.

### Finding Description

`EVM.validateCOAOwnershipProof` accepts six parameters: `address` (Cadence account), `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The function:

1. Verifies that the provided `signatures` are valid over `signedData` using the account's keys.
2. Checks that the COA resource at `path` has an EVM address matching `evmAddress`.

The critical gap is that **`signedData` is never checked to contain `evmAddress`**. The code itself acknowledges this in a comment:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The function is callable by any unprivileged Cadence script or EVM transaction via the `verifyCOAOwnershipProof` Cadence Arch precompile. The precompile at `fvm/evm/precompiles/arch.go` calls `coaOwnershipProofValidator` which invokes `EVM.validateCOAOwnershipProof` with caller-supplied `signedData` and `evmAddress`.

### Impact Explanation

An attacker who obtains a valid `(signedData, signatures)` pair that was originally produced by a victim to prove ownership of COA address `0xAAAA` can replay that exact pair in a call to `verifyCOAOwnershipProof(0xBBBB, hash, encodedProof)` where `encodedProof` references the same Cadence account but a different `CapabilityPath` pointing to a different COA (`0xBBBB`). The signature check passes (same `signedData`, same keys), and the EVM-address check passes (the COA at the new path has address `0xBBBB`). The result is a forged proof of ownership of `0xBBBB`.

Any EVM contract relying on `ERC-1271` via the COA's `isValidSignature` (which calls `verifyCOAOwnershipProof`) or any off-chain authentication flow using `EVM.validateCOAOwnershipProof` directly is vulnerable to accepting a forged ownership proof for a COA the signer never intended to authorize.

### Likelihood Explanation

The preconditions are:
- The victim Cadence account must own **more than one COA** (or the attacker must be able to publish a capability at a different path pointing to a COA they control but that is linked to the victim's account).
- The attacker must have observed a prior valid `(signedData, signatures)` pair from the victim.

The code comment explicitly acknowledges this risk and notes it is "low-risk" only because "Flow wallets historically create at most one COA per account." However, this is a social/operational assumption, not a protocol-level enforcement. Any application that creates multiple COAs per account, or any future wallet that does so, is immediately exposed. The entry path is fully unprivileged: any EVM transaction can call the `verifyCOAOwnershipProof` precompile.

### Recommendation

Enforce inside `EVM.validateCOAOwnershipProof` that `signedData` encodes `evmAddress` — for example, by requiring `signedData` to be the ABI or Cadence encoding of `evmAddress` (or a hash thereof), or by prepending `evmAddress` to `signedData` before the cryptographic verification step. This mirrors the fix recommended in the original report: bind the signed payload to the specific target context (here, the EVM address) so that a signature produced for one address cannot be replayed for another.

### Proof of Concept

1. Victim account `0xAlice` owns two COAs: `COA_A` at `/public/coaA` (EVM address `0xAAAA`) and `COA_B` at `/public/coaB` (EVM address `0xBBBB`).
2. A legitimate protocol asks Alice to prove ownership of `COA_A`. Alice signs arbitrary `signedData = [0x01, 0x02, ...]` with her key. The resulting `(signedData, signatures)` pair is published or observed by an attacker.
3. The attacker constructs a `COAOwnershipProof` with `Address = 0xAlice`, `CapabilityPath = /public/coaB`, `KeyIndices = [0]`, `Signatures = [aliceSig]` (the same signature from step 2).
4. The attacker calls `EVM.validateCOAOwnershipProof(address: 0xAlice, path: /public/coaB, signedData: [0x01, 0x02, ...], keyIndices: [0], signatures: [aliceSig], evmAddress: 0xBBBB)`.
5. The `keyList.verify(signatureSet: ..., signedData: [0x01, 0x02, ...], domainSeparationTag: "FLOW-V0.0-user")` check passes — the signature is valid over `signedData`.
6. The COA at `/public/coaB` has address `0xBBBB`, so the address-match check passes.
7. The function returns `ValidationResult(isValid: true, ...)` — a forged proof of ownership of `0xBBBB`.
