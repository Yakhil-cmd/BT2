### Title
Missing Zero-Address Check for `refund_recipient` in L1→L2 Transaction Processing Causes Permanent Token Loss - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary

The `validate_structure()` function for ABI-encoded L1→L2 and upgrade transactions explicitly skips zero-address validation for `reserved[1]` (the refund recipient field), as acknowledged by a `// TODO: validate address?` comment. Downstream, `process_l1_transaction` unconditionally mints the gas refund to whatever address is in `reserved[1]`, including `address(0)`. Any user who submits an L1→L2 transaction with `refund_recipient = address(0)` permanently burns their refund tokens with no recovery path.

### Finding Description

**Root cause 1 — validation gap in `validate_structure`:** [1](#0-0) 

The `reserved[1]` field (refund recipient) is the only address-typed field in the ABI-encoded transaction that receives no validation. Every other structural constraint is enforced here; this one is explicitly deferred with `// TODO: validate address?`.

**Root cause 2 — unconditional mint to the unvalidated address:** [2](#0-1) 

`u256_to_b160_checked` only asserts that the value fits in 160 bits (upper limbs are zero); it does **not** check for `B160::ZERO`. [3](#0-2) 

When `to_refund_recipient > U256::ZERO` and `reserved[1] == 0`, `mint_base_token` is called with `refund_recipient = B160::ZERO`, permanently crediting address(0).

**Confirmed by an existing test that treats this as expected behavior:** [4](#0-3) [5](#0-4) 

The test explicitly sets `refund_recipient = address(0)` and asserts that the refund amount lands at address(0), confirming the protocol accepts and executes this path without error.

### Impact Explanation

Any unused-gas refund tokens minted to `address(0)` are permanently irrecoverable. The treasury balance is debited, the tokens are created at address(0), and no mechanism exists to retrieve them. This constitutes a permanent, unrecoverable loss of user funds for every L1→L2 transaction where:
- `reserved[1]` is zero (either accidentally or due to a bridge/UI bug), and
- there is any unused gas (`to_refund_recipient > 0`).

Because L1→L2 transactions always over-provision gas (the L1 bridge charges for `gas_limit`, not `gas_used`), a non-zero refund is the common case, making accidental burns likely.

### Likelihood Explanation

The `refund_recipient` field defaults to `address(0)` in the `L1TxBuilder` helper: [6](#0-5) 

Any bridge integration, SDK, or tooling that omits the refund recipient will silently burn the refund. The `// TODO: validate address?` comment confirms the developers are aware of the gap but have not closed it. Likelihood is **medium**: accidental omission by integrators is realistic, and the protocol provides no guard.

### Recommendation

**Short term:** Add a zero-address check inside `validate_structure` for `reserved[1]` on `L1_L2_TX_TYPE` and `UPGRADE_TX_TYPE`:

```rust
Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
    // Reject zero refund recipient to prevent permanent token loss
    if self.reserved[1].read().is_zero() {
        return Err(());
    }
}
```

**Long term:** Replace the `// TODO: validate address?` comment with enforced validation. Audit all other address-typed fields in ABI-encoded transactions for similar gaps. Consider adding a property-based test that asserts no tokens are ever minted to `address(0)` as a protocol invariant.

### Proof of Concept

1. Construct an L1→L2 transaction (`tx_type = 0x7F`) with `reserved[1] = 0` (refund recipient = `address(0)`), any non-zero `gas_limit`, and `gas_price > 0`.
2. Submit it to the ZKsync OS bootloader.
3. `validate_structure` accepts the transaction (the `// TODO` branch is a no-op).
4. The transaction executes; `gas_used < gas_limit` (typical for any non-trivial gas limit).
5. `to_refund_recipient = gas_price * (gas_limit - gas_used) > 0`.
6. `refund_recipient = u256_to_b160_checked(0) = B160::ZERO`.
7. `mint_base_token(..., &to_refund_recipient, &B160::ZERO, ...)` is called — tokens are permanently minted to `address(0)`.
8. The treasury balance decreases; the refund is unrecoverable.

This is directly confirmed by the existing regression test `test_treasury_based_token_distribution_regression` which uses `refund_recipient = address(0)` and verifies the refund lands there. [1](#0-0) [2](#0-1)

    assert!(src.as_limbs()[3] == 0 && src.as_limbs()[2] < (1u64 << 32));

        result.as_limbs_mut()[0] = src.as_limbs()[0];
