### Title
Missing Access Control on `update_rewards` Allows Any Caller to Freeze Consensus Reward Distribution — (File: `src/staking/staking.cairo`)

---

### Summary

The `IStakingRewardsManager::update_rewards` function is documented as callable "Only starkware sequencer" but the implementation enforces no such check. Any unprivileged public caller can invoke it with `disable_rewards: true` to advance the global `last_reward_block` state without distributing rewards, permanently blocking the sequencer from distributing rewards in that block for all stakers.

---

### Finding Description

The protocol specification explicitly restricts `update_rewards` to the Starkware sequencer:

> **Access control**: Only starkware sequencer. [1](#0-0) 

However, the implementation only calls `general_prerequisites()`: [2](#0-1) 

And `general_prerequisites()` only checks for pause state and non-zero caller — no sequencer identity check: [3](#0-2) 

The function unconditionally writes to the global `last_reward_block` storage variable before the `disable_rewards` branch: [4](#0-3) 

Because `last_reward_block` is a single global (not per-staker), the guard at the top of the function prevents **any** second call in the same block for **any** staker: [5](#0-4) 

An attacker therefore needs only one transaction per block — calling `update_rewards(any_valid_active_staker, disable_rewards: true)` — to consume the block's single reward-distribution slot without distributing anything. The sequencer's subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

The `disable_rewards: true` path is the critical enabler: it still writes `last_reward_block` but skips all reward accounting: [6](#0-5) 

The attacker must supply a valid, active staker address with non-zero balance (all public information from on-chain events), but no stake of their own is required.
