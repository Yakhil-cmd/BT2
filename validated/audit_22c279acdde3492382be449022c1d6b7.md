### Title
Gateway Skips Signature Validation for Invoke Transactions with Nonce=1 When Any Pending Transaction Exists for the Sender — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point for any invoke transaction with `nonce=1` when the sender's address appears in the mempool or a recent block. Because the check is not restricted to deploy-account transactions, an attacker who observes a victim's pending `deploy_account` can submit an invoke with a garbage signature for the victim's address, bypass all cryptographic signature verification at the gateway, and have the invalid transaction admitted to the mempool. Via the mempool's fee-escalation mechanism the attacker can then displace the victim's legitimate nonce-1 invoke at zero cost (the invalid transaction reverts during batcher execution, so no fee is charged).

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks`, which is itself called from `extract_state_nonce_and_run_validations`:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range, resource bounds)
       ├─ validate_by_mempool            (duplicate hash / fee-escalation check — no sig)
       └─ skip_stateful_validations      ← decides whether __validate__ runs
```

The skip condition is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

With `validate = false`, `validate_tx` in the blockifier returns `Ok(None)` immediately without calling the account's `__validate__` entry point:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [3](#0-2) 

Neither the stateless validator nor `validate_by_mempool` performs cryptographic signature verification. The stateless validator only checks signature *length*:

```rust
fn validate_tx_signature_size(...) {
    let signature_length = signature.0.len();
    if signature_length > self.config.max_signature_length { ... }
``` [4](#0-3) 

`ValidationArgs` (used by `validate_by_mempool`) carries no signature field at all:

```rust
pub struct ValidationArgs {
    pub address: ContractAddress,
    pub account_nonce: Nonce,
    pub tx_hash: TransactionHash,
    pub tx_nonce: Nonce,
    pub tip: Tip,
    pub max_l2_gas_price: GasPrice,
}
``` [5](#0-4) 

The code comment justifying the broad check reads:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [6](#0-5) 

This reasoning is flawed: `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type from that address, not specifically a `deploy_account`. An attacker can exploit this by placing any transaction from the victim's address into the mempool first (or simply observing the victim's own `deploy_account`), then submitting an invoke with a forged signature.

The mempool supports fee escalation: a new transaction at the same `(address, nonce)` replaces an existing one if it offers a sufficiently higher tip and gas price:

```rust
fn validate_fee_escalation(...) -> MempoolResult<Option<TransactionReference>> {
    ...
    if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
        return Err(MempoolError::DuplicateNonce { address, nonce });
    }
    Ok(Some(existing_tx_reference))
}
``` [7](#0-6) 

Because the attacker's transaction will revert during batcher execution (the account's `__validate__` will reject the garbage signature), **no fee is charged**, making the attack free to repeat.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions and rejects valid transactions before sequencing.**

An attacker can:
1. Admit an unsigned (garbage-signature) invoke transaction into the mempool for any account that has a pending `deploy_account`.
2. Use fee escalation to replace the victim's legitimate nonce-1 invoke with the attacker's invalid one.
3. The victim's first post-deploy call is silently dropped; the attacker's invalid transaction reverts during execution at zero cost.

This breaks the admission invariant: the gateway is supposed to guarantee that every transaction in the mempool has passed signature verification before sequencing.

---

### Likelihood Explanation

**Medium.** The attacker only needs to:
- Monitor the public mempool (or RPC) for `deploy_account` transactions — trivially observable.
- Submit an invoke with the victim's address, `nonce=1`, any calldata, and a garbage signature within the configured `max_signature_length`.
- Offer a tip/gas-price above the fee-escalation threshold to displace the victim's legitimate invoke.

No privileged access, no special knowledge, and no on-chain funds are required (the attacker's transaction reverts before fee transfer).

---

### Recommendation

1. **Restrict the skip condition to deploy-account transactions specifically.** Instead of `account_tx_in_pool_or_recent_block`, add a dedicated mempool query such as `deploy_account_in_pool(sender_address)` that returns `true` only when a `DeployAccount` transaction is present for that address.

2. **Alternatively**, require the invoke transaction to carry the hash of the associated `deploy_account` transaction (as the legacy `py_validator.rs` path does via `deploy_account_tx_hash`), and verify that hash against the mempool before skipping validation. [8](#0-7) 

---

### Proof of Concept

```
1. Alice submits deploy_account (nonce=0) to the gateway.
   → Admitted to mempool; account_tx_in_pool_or_recent_block(Alice) = true.

2. Bob crafts an invoke:
     sender_address = Alice's address
     nonce          = 1
     calldata       = [0x0]          (arbitrary)
     signature      = [0x1, 0x2]    (garbage, within max_signature_length)
     resource_bounds = valid non-zero values
     tip            = Alice's tip * 2   (triggers fee escalation)

3. Bob submits the invoke to the gateway.

4. Gateway stateless validation:
   - signature length 2 ≤ max_signature_length → PASS
   - resource bounds non-zero → PASS

5. Gateway stateful validation:
   - validate_nonce: nonce=1, account_nonce=0, within gap → PASS
   - validate_by_mempool: no duplicate hash → PASS
   - skip_stateful_validations:
       nonce == 1 ✓, account_nonce == 0 ✓,
       account_tx_in_pool_or_recent_block(Alice) == true ✓
       → returns true (skip __validate__)
   - run_validate_entry_point called with validate=false → __validate__ NOT called

6. Bob's invoke is admitted to the mempool.
   Fee escalation replaces Alice's legitimate invoke (if already present).

7. Batcher executes:
   - Alice's deploy_account executes → account deployed.
   - Bob's invoke executes → __validate__ called → garbage signature rejected → REVERT.
   - No fee charged to Bob.
   - Alice's legitimate invoke was displaced and is never executed.
```

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }
