### Title
`TurnBridge` Is a No-Op Stub — Bridge Circuit-Breaker Permanently Non-Functional - ([File: `x/cronos/keeper/msg_server.go`])

### Summary

The `TurnBridge` message handler is an unimplemented stub that unconditionally returns `(nil, nil)`. Any sender — permissioned or not — receives a success response, but no permission check is performed and no state is ever written. The bridge enabled/disabled flag is never modified, making the emergency circuit-breaker permanently non-functional.

### Finding Description

The entire body of `TurnBridge` is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly gates on `HasPermission` and then calls `RegisterOrUpdateTokenMapping` to write state: [2](#0-1) 

`TurnBridge` does neither. The ADR and integration tests confirm the intended design: `MsgTurnBridge` is a *restricted* message requiring `CanTurnBridge` permission, and its purpose is to write a bridge-enabled flag that the gravity bridge hooks read to halt operations. [3](#0-2) 

The integration test `test_gravity_turn_bridge` asserts `rsp["code"] == 0` after calling `turn_bridge("false", from_="validator")` and then expects bridge sends to fail — but because the handler is a stub, the bridge state is never written and bridge operations continue unimpeded. [4](#0-3) 

### Impact Explanation

The bridge circuit-breaker is completely non-functional. When an operator with `CanTurnBridge` permission submits `MsgTurnBridge{enable: false}` in an emergency (e.g., a bridge exploit in progress), they receive a success response but the bridge remains open. Unauthorized bridge operations (minting of gravity-bridged tokens, `send_to_ethereum` transfers) continue without interruption. This is a **High** severity bypass of a critical admin/bridge authorization control.

### Likelihood Explanation

The vulnerability is trivially reachable: `MsgTurnBridge` is a registered message type, the CLI command `turn-bridge` exists, and the handler is wired up in the gRPC router. Any transaction broadcast triggers it. The only precondition is that the message type is registered, which it is. [5](#0-4) 

### Recommendation

Implement `TurnBridge` with the same pattern as `UpdateTokenMapping`:
1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` if it fails.
2. Write the `enable` flag into the module's KV store (or params).
3. Ensure the gravity bridge hooks read this flag before processing packets/batches.

### Proof of Concept

1. Grant `CanTurnBridge` permission to address `A` via `MsgUpdatePermissions`.
2. Submit `MsgTurnBridge{sender: A, enable: false}`.
3. Observe `code == 0` in the response (success).
4. Query the bridge-enabled state — it is unchanged (still enabled).
5. Submit a `send_to_ethereum` or `send_to_cosmos` bridge transaction — it succeeds, proving the circuit-breaker had no effect.
