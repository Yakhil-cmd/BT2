### Title
Overly Broad `skip_stateful_validations` Check Allows Admission of Invoke Transactions with Invalid Signatures — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry point (which performs account-contract signature verification) for the deploy-account + invoke UX pattern. However, the guard condition is too broad: it skips validation whenever *any* transaction for the account is in the mempool, not only when a `deploy_account` transaction is pending. An unprivileged attacker who observes a legitimate user's nonce-0 invoke in the mempool can immediately submit a nonce-1 invoke with an **invalid signature** for the same account, bypassing gateway signature verification and getting the invalid transaction admitted to the mempool.

---

### Finding Description

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three conditions hold:

1. The incoming transaction is an `Invoke` with `tx_nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

Condition 3 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` whenever **any** transaction for the account is in the pool — including a regular nonce-0 invoke submitted by a legitimate user for an already-deployed account. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incorrect for the case of an already-deployed account (nonce=0 on-chain) with a pending nonce-0 invoke.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

The blockifier's `perform_validations` then returns early without calling `__validate__`: [4](#0-3) 

`__validate__` is the account contract's signature-verification entry point. Skipping it means the gateway never verifies the transaction signature. The pre-validation stage (`handle_nonce`, `check_fee_bounds`, `verify_can_pay_committed_bounds`) still runs, but none of those checks verify the signature. [5](#0-4) 

The mempool's `validate_tx` path also does not check signatures — it only checks for duplicate hashes and nonce gaps: [6](#0-5) 

---

### Impact Explanation

An attacker who observes a legitimate user's nonce-0 invoke in the mempool (visible via P2P or RPC) can submit a nonce-1 invoke with an **arbitrary invalid signature** for the same account. The gateway admits it to the mempool without signature verification. The invalid transaction will fail during block execution (the batcher calls `__validate__` unconditionally), but while it sits in the mempool it:

- Occupies the nonce-1 slot for the victim account, blocking the legitimate user's nonce-1 transaction from being admitted (rejected as a duplicate nonce)
- Forces the legitimate user to pay a higher fee to replace it via fee escalation
- Pollutes the mempool with transactions that are guaranteed to fail on execution

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- No privileged access required; any network participant can submit transactions to the gateway
- The trigger condition (a pending nonce-0 invoke for a target account) is observable via the public P2P network or RPC mempool queries
- The attack is cheap: the attacker only needs to craft a syntactically valid invoke transaction with a wrong signature and submit it

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction is pending for the account. One approach: add a dedicated mempool query `has_pending_deploy_account(address) -> bool` and use that instead. Alternatively, store the transaction type alongside the account state in the mempool so the gateway can distinguish deploy-account entries from regular invoke entries.

The corrected guard should be:

```rust
// Only skip validation if a deploy_account tx is pending for this account.
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .has_pending_deploy_account(tx.sender_address())  // type-specific check
        .await
        ...
}
```

### Proof of Concept

```
1. Account A is deployed on-chain; on-chain nonce = 0.
   (No committed mempool history for A, so state.contains_account(A) = false.)

2. Legitimate user submits:
     Invoke { sender: A, nonce: 0, signature: valid_sig, calldata: [...] }
   → Gateway: account_nonce=0, tx_nonce=0 → skip_stateful_validations returns false
     → __validate__ is called → valid signature passes → tx admitted to mempool
   → tx_pool.contains_account(A) = true

3. Attacker submits:
     Invoke { sender: A, nonce: 1, signature: GARBAGE, calldata: [malicious] }
   → Gateway stateless checks: pass (valid structure, resource bounds, etc.)
   → validate_nonce: account_nonce=0, tx_nonce=1, 0 ≤ 1 ≤ max_gap → pass
   → validate_by_mempool: no duplicate hash/nonce → pass
   → skip_stateful_validations:
       tx_nonce == 1 ✓, account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(A) = true (from step 2) ✓
       → returns true (skip __validate__)
   → __validate__ NOT called → invalid signature never checked
   → Attacker's tx admitted to mempool

4. Legitimate user tries to submit their own nonce-1 invoke:
   → Mempool rejects it as DuplicateNonce (attacker's tx already holds nonce-1 slot)
   → Legitimate user must pay higher fee to replace it via fee escalation

5. Batcher picks up attacker's nonce-1 tx, calls __validate__ → fails → tx rejected.
   Legitimate user's transaction is delayed or must be resubmitted.
``` [7](#0-6) [2](#0-1)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

        &self,
        state: &mut S,
