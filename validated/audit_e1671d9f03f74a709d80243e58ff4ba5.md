### Title
Gateway Admits Invoke Transactions with Arbitrary Signatures by Skipping `__validate__` for Accounts with Pending `deploy_account` - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the gateway's stateful validator unconditionally skips the `__validate__` entry point (the account's signature-verification function) for any invoke transaction with `nonce=1` submitted against an account whose on-chain nonce is `0` and that has any transaction in the mempool. An unprivileged attacker can exploit this to inject an invoke transaction carrying an arbitrary or empty signature for any victim account that has a pending `deploy_account` in the mempool, and the gateway will admit it without ever calling the account's `__validate__` function.

---

### Finding Description

`skip_stateful_validations` implements a UX shortcut: when a user submits a `deploy_account` + `invoke` pair simultaneously, the invoke (nonce=1) cannot be validated against the account contract because the contract does not yet exist on-chain. The function therefore skips the `__validate__` call when three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. The transaction's nonce equals `Felt::ONE`.
3. The account's committed on-chain nonce equals `Felt::ZERO`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When all four conditions are satisfied the function returns `true`, which propagates back to `run_validate_entry_point`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

`validate: false` is then passed into `StatefulValidator::perform_validations`, which explicitly returns `Ok(())` before calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

The `__validate__` entry point is the only place where the account contract verifies the transaction signature. Skipping it means the signature field is never inspected.

`account_tx_in_pool_or_recent_block` returns `true` whenever the account address appears in the mempool pool or in the mempool's committed-block state:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

A victim account satisfies this condition the moment its legitimate `deploy_account` transaction enters the mempool. The attacker does not need to submit the `deploy_account`; they only need to observe that one exists (e.g., via the public RPC).

After validation passes, `add_tx_inner` in the gateway calls `mempool_client.add_tx(add_tx_args)` unconditionally: [5](#0-4) 

Neither `validate_by_mempool` nor `mempool.add_tx` inspect the signature; they only check nonce ordering and fee escalation rules: [6](#0-5) 

The attacker's invoke transaction is therefore stored in the mempool alongside the victim's legitimate `deploy_account`.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid transaction.**

The gateway's invariant is that every admitted transaction has passed account-level signature verification. This invariant is broken for any invoke with `nonce=1` targeting an account whose `deploy_account` is pending. The attacker's transaction carries an arbitrary signature and arbitrary calldata, yet receives a success response from the gateway and is stored in the mempool. When the batcher later executes the block, the `__validate__` entry point will be called with `validate=true` (the batcher uses default `ExecutionFlags`), the fake signature will fail, and the transaction will be marked rejected — but the admission invariant was already violated at the gateway layer.

---

### Likelihood Explanation

Any account that submits a `deploy_account` transaction is immediately vulnerable for the window between mempool admission and block commitment. This is a routine, publicly observable event. The attacker needs only to:

1. Monitor the mempool or RPC for a `deploy_account` transaction.
2. Craft an invoke with `sender_address = victim`, `nonce = 1`, any calldata, any (or empty) signature.
3. Submit it to the gateway.

No privileged access, no special knowledge of the victim's private key, and no on-chain funds are required.

---

### Recommendation

The skip-validation shortcut must be tied to the **transaction hash** of the corresponding `deploy_account`, not merely to the existence of any transaction for that address in the mempool. Specifically:

- Require the caller to supply the `deploy_account` transaction hash alongside the invoke (as the `native_blockifier` Python path already does via `deploy_account_tx_hash`). [7](#0-6) 

- In `skip_stateful_validations`, verify that the mempool contains a `deploy_account` transaction (not just any transaction) for the sender address, and that its hash matches the supplied value.
- Alternatively, require the invoke's signature to be verifiable against the **expected** account class (derived from the pending `deploy_account`'s `class_hash` and constructor arguments) before skipping the on-chain `__validate__` call.

---

### Proof of Concept

```
1. Alice submits DeployAccount(class_hash=C, salt=S, constructor=[K_pub])
   → gateway admits it; mempool.account_tx_in_pool_or_recent_block(alice) == true
   → alice's on-chain nonce remains 0

2. Attacker observes alice's address via starknet_getTransactionByHash or mempool gossip.

3. Attacker crafts:
   InvokeV3 {
       sender_address: alice,
       nonce: 1,
       calldata: [<arbitrary>],
       signature: [],          // empty — never verified
       resource_bounds: <valid>,
       ...
   }

4. Attacker calls POST /add_transaction with the above.

5. Gateway flow:
   a. stateless_tx_validator.validate() — passes (empty signature is within length limit)
   b. convert_rpc_tx_to_internal_and_executable_txs() — succeeds
   c. extract_state_nonce_and_run_validations():
      - get_nonce_from_state(alice) → Nonce(0)          ✓ condition 3
      - validate_nonce: 0 ≤ 1 ≤ 0+max_gap              ✓
      - validate_by_mempool: no duplicate, nonce ok      ✓
      - skip_stateful_validations:
          tx.nonce()==1 && account_nonce==0              ✓ conditions 1,2,3
          account_tx_in_pool_or_recent_block(alice)==true ✓ condition 4
          → returns true (skip __validate__)
      - run_validate_entry_point(skip_validate=true):
          ExecutionFlags { validate: false, ... }
          StatefulValidator::perform_validations → Ok(()) without calling __validate__
   d. mempool_client.add_tx(attacker_invoke) → Ok(())

6. Gateway returns 200 OK with attacker's tx_hash.
   Attacker's invoke (with fake signature) is now in the mempool.
``` [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
