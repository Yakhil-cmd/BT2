### Title
COA Contract Missing `fallback() payable` Causes Value+Calldata Transfers to Revert - (`File: fvm/evm/handler/coa/coa.sol`)

### Summary
The `COA` contract deployed for every Cadence-Owned Account defines only a `receive() external payable` function but no `fallback() external payable` function. Any EVM call that sends value together with non-empty calldata that does not match a known function selector will revert, silently failing the transfer. This is the direct analog of M-03: the `receive` path handles plain ETH transfers, but the missing `fallback` path means value+data calls fail where a true EOA would succeed.

### Finding Description
The `COA` contract in `fvm/evm/handler/coa/coa.sol` is the on-chain smart-contract wallet deployed at every COA address. It implements several ERC token-receiver hooks and a `receive()` function:

```solidity
receive() external payable  {
}
``` [1](#0-0) 

There is no `fallback()` function anywhere in the contract. [2](#0-1) 

In Solidity, the EVM dispatches to `receive()` only when `msg.data` is empty. When `msg.data` is non-empty and no function selector matches, the EVM dispatches to `fallback()`. Because `fallback()` is absent, any call of the form `address(coa).call{value: v}(data)` where `data` does not match a known selector will revert with an out-of-gas / no-fallback error, and the value will not be transferred.

The `EVM.cdc` contract documentation explicitly states that COAs "accept native token transfers": [3](#0-2) 

The `coa_abi.json` confirms only `receive` (payable) is present and no `fallback` entry exists: [4](#0-3) 

### Impact Explanation
Any EVM protocol that sends value together with calldata to a COA address — a common DeFi pattern such as `call{value: amount}(abi.encodeWithSelector(...))` — will receive a failed result instead of a successful transfer. The value is not moved. This breaks cross-VM asset flows where an EVM contract treats a COA address as an EOA-equivalent recipient. The `Account.Transfer` path in the handler uses `NewTransferCall` with `Data: nil`, so plain transfers work; but the general `Call` path with non-empty data and non-zero value targeting a COA will fail at the EVM level. [5](#0-4) [6](#0-5) 

### Likelihood Explanation
Any unprivileged EVM transaction sender or COA owner can trigger this by issuing a `ContractCallSubType` direct call (or a standard EVM transaction) to a COA address with non-zero value and non-empty calldata. No special privileges are required. The pattern of `call{value: x}(data)` is ubiquitous in DeFi (ERC-777 hooks, proxy forwarding, multi-call patterns), making accidental triggering realistic.

### Recommendation
Add a `fallback() external payable {}` function to the `COA` contract, mirroring the fix recommended in M-03:

```solidity
receive() external payable {
}

fallback() external payable {
}
```

This ensures that calls with non-matching calldata and non-zero value succeed (value is accepted) rather than reverting, making COA addresses behave like EOAs for value-transfer purposes, consistent with the documented intent.

### Proof of Concept
1. Deploy a COA via `EVM.createCadenceOwnedAccount()` and fund it.
2. From another EVM account, execute:
   ```solidity
   (bool ok,) = address(coa).call{value: 1 ether}(abi.encode(uint256(42)));
   // ok == false  ← revert, value not transferred
   ```
3. Contrast with calling a plain EOA address (no code) with the same call: the EVM transfers the value and returns `ok == true`.
4. The discrepancy arises because the COA has code but no `fallback`, so the EVM reverts instead of accepting the value. [7](#0-6)
