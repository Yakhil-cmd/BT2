### Title
Gateway `skip_stateful_validations` bypasses `__validate__` signature check when any mempool transaction exists for account, not only a deploy_account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for an Invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender. The function's own comment acknowledges the proxy is imprecise: it returns `true` for *any* account transaction in the mempool, including future-nonce Invoke transactions (nonce ≥ 2) that were admitted under the 200-nonce gap allowance — not only for a pending `deploy_account`. An attacker who observes that a target account has a future-nonce Invoke in the mempool can submit a malicious Invoke with nonce=1 and an arbitrary/invalid signature; the gateway skips `__validate__` and admits the transaction.

### Finding Description

`skip_stateful_validations` is the sequencer-native analog of the stale-accounting bug in the external report. In the external report, `strategyUnderlying` is a cached balance variable that becomes stale after `updateStrategy` is called without resetting it; downstream code (`_transferBaseToken`) trusts the stale value and delivers less collateral than owed. Here, `account_tx_in_pool_or_recent_block` is a cached proxy for "a `deploy_account` is pending", but it becomes stale/incorrect the moment any other transaction type for that account enters the mempool — and downstream code (`run_validate_entry_point`) trusts the proxy and skips signature verification.

The gateway stateful-validation path is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range, resource bounds)
       ├─ validate_by_mempool            (duplicate hash / nonce-too-old)
       └─ skip_stateful_validations      ← broken invariant here
  └─ run_validate_entry_point(skip_validate=true)  ← __validate__ skipped
```

`skip_stateful_validations` fires when all three conditions hold:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())   // ← proxy
        .await ...
}
```

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

It returns `true` for **any** transaction type. Meanwhile, `validate_nonce` in the gateway admits Invoke transactions with nonces 0 through `account_nonce + max_allowed_nonce_gap` (default 200):

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 287-296
_ => {
    let max_allowed_nonce =
        Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
    if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
        return Err(...);
    }
}
```

So an account with `account_nonce=0` can have an Invoke with nonce=2 in the mempool (no `deploy_account` required). Once that future-nonce Invoke is present, `account_tx_in_pool_or_recent_block` returns `true`, and the skip condition fires for any subsequent Invoke with nonce=1 — regardless of its signature.

### Impact Explanation

The gateway admits an Invoke transaction with an invalid/attacker-controlled signature to the mempool without running `__validate__`. This matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:
- The nonce=1 slot for the victim account is occupied by the malicious transaction, blocking the legitimate user's nonce=1 transaction (requires fee escalation to displace).
- The batcher wastes execution resources on a transaction that will revert when `__validate__` is finally called during block execution (execution flags default to `validate: true`).
- An attacker can repeat this for any account that has a future-nonce Invoke in the mempool, enabling sustained mempool pollution and targeted DoS.

### Likelihood Explanation

The nonce gap of 200 is the production default. Users routinely submit batches of transactions with sequential nonces; a gap at nonce=1 (e.g., submitting nonce=0 deploy_account + nonce=2 Invoke while nonce=1 is absent) is a realistic occurrence. No privileged access is required; the attacker only needs to observe the public mempool.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` proxy with a check that specifically confirms a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool`, or the existing function should be narrowed to inspect transaction type. Alternatively, mirror the `PyValidator::should_run_stateful_validations` approach and require the caller to pass the concrete `deploy_account_tx_hash` so the gateway can verify the specific transaction exists, rather than relying on the presence of any account transaction.

### Proof of Concept

```
State:  account A, account_nonce = 0 (not deployed), max_allowed_nonce_gap = 200

Step 1 – Legitimate user submits Invoke(nonce=2) for account A.
         validate_nonce:  0 ≤ 2 ≤ 200  → OK
         __validate__:    runs, signature valid  → admitted to mempool
         account_tx_in_pool_or_recent_block(A) now returns true

Step 2 – Attacker submits Invoke(nonce=1) for account A with INVALID signature.
         validate_nonce:  0 ≤ 1 ≤ 200  → OK
         validate_by_mempool: nonce=1 not in pool, no duplicate hash  → OK
         skip_stateful_validations:
             tx.nonce() == 1  ✓
             account_nonce == 0  ✓
             account_tx_in_pool_or_recent_block(A) == true  ✓  (due to nonce=2 Invoke)
             → returns true (skip __validate__)
         run_validate_entry_point(skip_validate=true): __validate__ NOT called
         → malicious Invoke(nonce=1) admitted to mempool without signature check

Step 3 – Batcher picks up Invoke(nonce=1), calls __validate__ with validate=true.
         Signature invalid → transaction reverts.
         Nonce=1 slot was occupied; legitimate user's nonce=1 tx was blocked.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
