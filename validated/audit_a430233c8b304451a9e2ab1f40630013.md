### Title
Post-SELFDESTRUCT Value Transfer Burns ETH Permanently in Same-Transaction Deconstruction — (`basic_system/src/system_implementation/flat_storage_model/account_cache.rs`)

### Summary

`finish_tx` unconditionally overwrites the entire `AccountProperties` of any account flagged `is_marked_for_deconstruction` with `AccountProperties::TRIVIAL_VALUE` (balance = 0). Because `mark_for_deconstruction` only transfers the balance that existed *at the moment SELFDESTRUCT was called*, any ETH credited to the same address *after* that call but *before* `finish_tx` is silently zeroed with no corresponding credit anywhere — permanently removing it from the total supply.

### Finding Description

**Step 1 — Flag is set only for same-tx-deployed contracts.**

`mark_for_deconstruction` sets `is_marked_for_deconstruction = true` only when `should_be_deconstructed` is true:

```
let in_constructor = account_data.current().value().observable_bytecode_len == 0;
let should_be_deconstructed = account_data.current().metadata().basic.deployed_in_tx
    == Some(cur_tx)
    || in_constructor;
``` [1](#0-0) 

This is the EIP-6780 restriction: only contracts deployed in the current transaction are actually deconstructed.

**Step 2 — Balance is transferred at SELFDESTRUCT time, not at end-of-tx.**

When `should_be_deconstructed` is true and beneficiary ≠ self, the balance is transferred immediately via `transfer_nominal_token_value_inner`, leaving C's balance at 0: [2](#0-1) 

**Step 3 — SELFDESTRUCT exits with a success code, parent frame continues.**

`selfdestruct` returns `Err(ExitCode::SelfDestruct)` which is a *success* exit code (value `0x03`), not a revert. The parent frame resumes execution normally. [3](#0-2) [4](#0-3) 

**Step 4 — No guard prevents crediting ETH to a marked-for-deconstruction account.**

`transfer_nominal_token_value_inner` and `update_nominal_token_value_inner` contain no check for `is_marked_for_deconstruction`. A CALL with value to address C after SELFDESTRUCT succeeds and credits C's balance: [5](#0-4) 

**Step 5 — `finish_tx` unconditionally zeros the entire account.**

At end of transaction, `finish_tx` iterates pending changes and, for any account with `is_marked_for_deconstruction = true`, sets `*x = AccountProperties::TRIVIAL_VALUE` — which includes `balance: U256::ZERO` — with no transfer of the current balance to anyone:

```rust
current.value.update(|x, metadata| {
    metadata.basic.is_marked_for_deconstruction = false;
    *x = AccountProperties::TRIVIAL_VALUE;   // balance zeroed, no transfer
    Ok(())
})?;
``` [6](#0-5) 

`AccountProperties::TRIVIAL_VALUE` has `balance: U256::ZERO` by definition: [7](#0-6) 

### Impact Explanation

Any ETH sent to a same-tx-deployed contract C after C has called SELFDESTRUCT is permanently destroyed. The sender's balance is debited (via `transfer_nominal_token_value_inner`), C's balance is credited, and then `finish_tx` zeros C's balance without crediting the beneficiary or returning the ETH. The total ETH supply decreases by the transferred amount. This is a direct, irreversible loss of funds.

### Likelihood Explanation

The scenario requires a contract that:
1. Deploys a sub-contract C in the same transaction (or C self-destructs in its constructor)
2. C calls SELFDESTRUCT
3. The parent frame subsequently sends ETH to C's address

This pattern is reachable by any unprivileged user deploying a crafted contract. A malicious factory contract can trigger this to burn ETH belonging to any caller that sends value through it. The attacker does not need any privileged role.

### Recommendation

In `finish_tx`, before zeroing the account, check whether the current balance exceeds zero and, if so, transfer it to the recorded beneficiary (which should be stored at `mark_for_deconstruction` time), or at minimum assert that the balance is zero to catch the invariant violation early. The simplest fix is to record the beneficiary address in the account metadata at SELFDESTRUCT time and re-transfer any residual balance in `finish_tx` before applying `TRIVIAL_VALUE`.

### Proof of Concept

```
// Attacker deploys this contract with value V1
contract Factory {
    address public child;
    
    constructor() payable {
        // Step 1: deploy C with 0 ETH
        child = address(new Child());
        // At this point: child.is_marked_for_deconstruction = true,
        //                child.balance = 0 (transferred to beneficiary in constructor)
        
        // Step 2: send V2 ETH to child AFTER selfdestruct
        (bool ok,) = child.call{value: V2}("");
        require(ok); // succeeds: no code at child, plain ETH transfer
        // child.balance = V2 in cache
        
        // finish_tx runs: child.balance -> 0, V2 burned
    }
}

contract Child {
    constructor() {
        // in_constructor = true (observable_bytecode_len == 0)
        // should_be_deconstructed = true
        selfdestruct(payable(msg.sender)); // balance=0 transferred, flag set
    }
}
```

Invariant violated: `sum(balances_before) + V2 == sum(balances_after) + gas_fees` — the left side exceeds the right side by `V2`. [8](#0-7)

    pub fn finish_tx(
        &mut self,
        storage: &mut NewStorageWithAccountPropertiesUnderHash<A, SF, M, R, P>,
    ) -> Result<(), InternalError> {
        self.current_tx_id += 1;

        // Actually deconstructing accounts
        self.cache.apply_to_last_record_of_pending_changes(
            |key, (_initial, current), cache_appearance| {
                if current.value.metadata().basic.is_marked_for_deconstruction {
                    // NOTE: it can only happen if the account is initially empty,
                    // so we need to make sure that it was observed earlier - when bytecode was deployed
                    assert!(cache_appearance.is_value_observed());
                    current.value.update(|x, metadata| {
                        metadata.basic.is_marked_for_deconstruction = false;
                        *x = AccountProperties::TRIVIAL_VALUE;
                        Ok(())
                    })?;
                    storage
                        .0
                        .clear_state_impl(key)
                        .expect("must clear state for code deconstruction in same TX");
                }
                Ok(())
            },
        )?;

        Ok(())
    }
