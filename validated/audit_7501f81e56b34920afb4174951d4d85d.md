### Title
Non-Whitelisted EVM Addresses Can Receive ERC-20 Tokens When No Fallback Address Is Configured, Causing Permanent Fund Freeze — (`engine/src/engine.rs`)

---

### Summary

In Aurora Engine's Silo mode, the `Address` whitelist check for ERC-20 token receipt is gated behind a `let Some(fallback_address)` guard. When no ERC-20 fallback address is configured, the whitelist check is never evaluated, allowing any non-whitelisted EVM address to receive bridged ERC-20 tokens. Because the same `Address` whitelist blocks those addresses from submitting EVM transactions, the received tokens become permanently frozen with no recovery path.

---

### Finding Description

In `engine/src/engine.rs`, the `receive_erc20_tokens` function determines the final recipient of bridged NEP-141 tokens:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

This is a combined `if let … && …` guard. The whitelist check `is_allow_receive_erc20_tokens` is only reached when `get_erc20_fallback_address` returns `Some`. When no fallback address is stored (i.e., `None`), the entire block is skipped unconditionally, and the original `recipient` — regardless of whitelist membership — receives the minted ERC-20 tokens.

`is_allow_receive_erc20_tokens` itself checks only the `Address` whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [2](#0-1) 

The same `Address` whitelist is enforced for transaction submission via `assert_access` → `is_allow_submit`:

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [3](#0-2) 

So a non-whitelisted address that receives ERC-20 tokens via the bridge cannot submit any EVM transaction to move them, including calls to the `ExitToNear` precompile.

The fallback address is stored and retrieved independently of the whitelist enable/disable state: [4](#0-3) 

A silo operator can enable the `Address` whitelist without ever setting a fallback address, or can later clear silo params (`set_silo_params(None)`) while leaving the whitelist active — both configurations produce the vulnerable state.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the `Address` whitelist is enabled but no ERC-20 fallback address is configured:

1. ERC-20 tokens are minted to a non-whitelisted EVM address via `receive_erc20_tokens`.
2. The non-whitelisted address is blocked from submitting any EVM transaction by `assert_access`.
3. There is no admin recovery function to forcibly transfer ERC-20 tokens out of an arbitrary EVM address.
4. The tokens are permanently frozen inside the Aurora EVM state with no on-chain recovery path. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The vulnerable state arises in two realistic configurations:

- A silo operator enables the `Address` whitelist for access control but does not configure silo params (no fallback address). The code comment explicitly notes the fallback logic "works only if the fallback address is set by `set_silo_params`", implying operators may treat these as independent features.
- A silo operator later calls `set_silo_params(None)` to exit silo mode but forgets to disable the `Address` whitelist, leaving the whitelist active with no fallback.

Once the vulnerable state exists, any NEAR account (no special privilege required) can trigger the freeze by calling `ft_transfer_call` on any registered NEP-141 token with a non-whitelisted EVM address as the message recipient. [6](#0-5) 

---

### Recommendation

Decouple the whitelist check from the fallback address guard. The whitelist check should be evaluated independently:

```rust
// Enforce whitelist regardless of fallback address presence
if !silo::is_allow_receive_erc20_tokens(&self.io, &recipient) {
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io) {
        recipient = fallback_address;
    } else {
        return Err(/* ERR_ADDRESS_NOT_WHITELISTED */);
    }
}
```

This ensures that when the `Address` whitelist is enabled, a non-whitelisted recipient is always either redirected to the fallback address or the transfer is rejected — never silently accepted.

---

### Proof of Concept

**Setup**: Silo mode with `Address` whitelist enabled, no fallback address configured.

1. Owner enables the `Address` whitelist:
   ```
   set_whitelist_status({ kind: Address, active: true })
   ```
2. Owner does **not** call `set_silo_params` (or calls `set_silo_params(None)`), so `get_erc20_fallback_address` returns `None`.
3. Attacker (any NEAR account) calls `ft_transfer_call` on a registered NEP-141 token:
   ```json
   { "receiver_id": "aurora", "amount": "1000000", "msg": "<non_whitelisted_evm_address_hex>" }
   ```
4. Aurora's `ft_on_transfer` → `receive_erc20_tokens` is invoked.
5. At line 818, `get_erc20_fallback_address` returns `None` → the `if let Some(…) && …` block is skipped entirely.
6. `recipient` remains the non-whitelisted address; ERC-20 tokens are minted to it via `self.call(…, setup_receive_erc20_tokens_input(&recipient, amount), …)`.
7. The non-whitelisted address attempts to call `withdrawToNear` on the ERC-20 contract → `assert_access` rejects it with `NotAllowed`.
8. Tokens are permanently frozen. [7](#0-6) [2](#0-1) [5](#0-4)

pub fn get_erc20_fallback_address<I: IO>(io: &I) -> Option<Address> {
    let key = erc20_fallback_address_key();
    io.read_storage(&key)?.to_value().ok()
}

/// Set ERC-20 fallback address.
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();

    if let Some(address) = address {
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);
    }

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,

    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
