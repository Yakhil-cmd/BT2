### Title
Missing Lower-Bound Check on `threshold` in FROST Signing Validation Allows Sub-Threshold Protocol Execution - (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `frost::presign` in `src/frost/mod.rs` validate that `threshold` does not exceed the participant count, but never enforce the minimum value of `2`. The DKG entry points (`assert_key_invariants` in `src/dkg.rs`) correctly enforce both bounds. A caller supplying `threshold = 1` (or `0`) to FROST signing bypasses the minimum-threshold guard, allowing the signing protocol to proceed with a threshold that is cryptographically inconsistent with the key material, corrupting the signing output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both sides of the threshold range:

```rust
// src/dkg.rs  lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The FROST signing entry points perform only the upper-bound check and omit the lower-bound check entirely.

`assert_sign_inputs` (used by every FROST sign call):

```rust
// src/frost/mod.rs  lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no check: threshold.value() < 2
``` [2](#0-1) 

`frost::presign` has the same gap:

```rust
// src/frost/mod.rs  lines 71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no check: args.threshold.value() < 2
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no constructor-level validation, so any value — including `0` or `1` — is accepted without error:

```rust
// src/thresholds.rs  lines 9-24
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
``` [4](#0-3) 

The same gap exists in the OT-based ECDSA `sign` entry point, which checks `participants.len() < threshold` but never `threshold < 2`: [5](#0-4) 

---

### Impact Explanation

FROST signing uses Lagrange interpolation over the contributing participants' shares. The key was generated with `threshold ≥ 2`, meaning the secret polynomial has degree `≥ 1`. If `threshold = 1` is accepted at the signing layer, the protocol may proceed collecting only one share and computing a Lagrange coefficient of `1` for that single participant. The reconstructed nonce/signature scalar will equal the raw share value rather than the interpolated secret, producing a signature that either fails on-chain verification or, in degenerate cases, leaks the participant's raw signing share through the output. Either outcome constitutes corruption of the signing output that honest parties may accept as a completed protocol run.

**Matched impact**: High — Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.

---

### Likelihood Explanation

`ReconstructionLowerBound` is a public type accepted directly from the caller at every FROST signing entry point. Any library consumer — including a malicious coordinator constructing `PresignArguments` — can supply `threshold = 1` without triggering any error. The missing check is a single missing `if` branch, making accidental or adversarial triggering straightforward.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces to both `assert_sign_inputs` and `frost::presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same fix to `ot_based_ecdsa::sign` (after converting `threshold` to `usize`). Alternatively, enforce the invariant inside `ReconstructionLowerBound::new` so it is impossible to construct an invalid value.

---

### Proof of Concept

```rust
use threshold_signatures::frost;
use threshold_signatures::participants::Participant;
use threshold_signatures::ReconstructionLowerBound;

let participants = vec![Participant::from(1u32), Participant::from(2u32)];
let me = Participant::from(1u32);
let coordinator = Participant::from(2u32);

// threshold = 1 — below the cryptographic minimum of 2
// assert_sign_inputs returns Ok(…) instead of ThresholdTooSmall
let result = frost::assert_sign_inputs(
    &participants,
    ReconstructionLowerBound::from(1usize),
    me,
    coordinator,
);
assert!(result.is_ok()); // passes — missing lower-bound check confirmed

// Similarly for presign:
// frost::presign(&participants, me, &PresignArguments { threshold: 1.into(), … }, rng)
// returns Ok(protocol) and the protocol executes with threshold=1
```

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
