### Title
Missing Zero Address Validation in `change_reward_address` Permanently Freezes Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
Neither `Staking::change_reward_address` nor `Pool::change_reward_address` validates that the supplied `reward_address` is non-zero. A staker or pool member can overwrite their reward address with `0x0`, after which every subsequent reward transfer targets the zero address, permanently destroying all unclaimed yield. Because `unstake_action` calls `send_rewards_to_staker` before returning principal, a zero reward address can also block the full unstake flow.

### Finding Description
`Staking::change_reward_address` (line 517) performs exactly one guard: it rejects a token contract address via `does_token_exist`. There is no `assert!(reward_address.is_non_zero(), …)` guard. [1](#0-0) 

`Pool::change_reward_address` (line 505) mirrors the same pattern: it only rejects the pool's own token address and writes the caller-supplied value directly to storage. [2](#0-1) 

The same gap exists at entry time: `Staking::stake` and `Pool::enter_delegation_pool` both check that `reward_address` is not a token address but never assert it is non-zero. [3](#0-2) [4](#0-3) 

### Impact Explanation
Once `reward_address` is set to `0x0`:

1. **Permanent loss of unclaimed yield** — every `claim_rewards` call transfers accrued STRK to address zero; those tokens are unrecoverable.
2. **Potential permanent freeze of principal** — `unstake_action` calls `send_rewards_to_staker` *before* returning the staked principal. If the STRK ERC-20 implementation reverts on a zero-address recipient (standard OpenZeppelin behaviour), `unstake_action` reverts unconditionally, trapping the staker's entire principal with no recovery path. [5](#0-4) 

This matches **High: Permanent freezing of unclaimed yield / unclaimed royalties** and potentially **High: Temporary freezing of funds**.

### Likelihood Explanation
The entry path is fully unprivileged: any active staker or pool member can call `change_reward_address` at any time without restriction. A user error (e.g., passing an uninitialised variable, a scripting bug, or a front-end that serialises `None` as `0`) is a realistic trigger. The absence of any on-chain guard means the contract provides no safety net.

### Recommendation
Add a non-zero assertion at the top of every function that accepts a `reward_address` parameter:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

Apply this to:
- `Staking::stake` — `reward_address` parameter
- `Staking::change_reward_address` — `reward_address` parameter
- `Pool::enter_delegation_pool` — `reward_address` parameter
- `Pool::change_reward_address` — `reward_address` parameter

### Proof of Concept

1. Deploy the system normally and stake with a valid reward address.
2. Call `staking.change_reward_address(reward_address: 0x0)` as the staker — the call succeeds; no assertion fires.
3. Advance epochs so rewards accrue.
4. Call `staking.claim_rewards(staker_address)` — rewards are transferred to `0x0` and are permanently lost.
5. Call `staking.unstake_intent()`, wait for the exit window, then call `staking.unstake_action(staker_address)` — if the STRK token reverts on zero-address transfers, this call panics inside `send_rewards_to_staker`, and the staker's principal is permanently frozen with no recovery path. [6](#0-5) [7](#0-6)

        fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
            // Prerequisites and asserts.
            self.general_prerequisites();
            assert!(
                !self.does_token_exist(token_address: reward_address),
                "{}",
                GenericError::REWARD_ADDRESS_IS_TOKEN,
            );
            let staker_address = get_caller_address();
            let mut staker_info = self.internal_staker_info(:staker_address);
            let old_address = staker_info.reward_address;

            // Update reward_address and commit to storage.
            staker_info.reward_address = reward_address;
            self.write_staker_info(:staker_address, :staker_info);

            // Emit event.
            self
                .emit(
                    Events::StakerRewardAddressChanged {
                        staker_address, new_address: reward_address, old_address,
                    },
                );
        }
