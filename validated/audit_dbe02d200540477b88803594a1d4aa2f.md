### Title
Malicious Participant Can Permanently Abort Triple Generation via Crafted DLOG Proof Triggering `IdentityElement` Error — (`src/crypto/proofs/dlog.rs`)

### Summary

`dlog::verify` returns `Err(ProtocolError::IdentityElement)` instead of `Ok(false)` when the reconstructed commitment `big_k = s·G − e·X` is the group identity. A malicious participant who knows their own secret `x` can trivially craft `(e, s=e·x)` to force this condition, causing every honest verifier to abort triple generation with an anonymous error — permanently blocking ECDSA signing without identifying the cheater.

### Finding Description

In `dlog::verify`, the reconstructed commitment is computed and then serialized: [1](#0-0) 

`C::Group::serialize` rejects the identity element (as required by the group abstraction), so the `.map_err` converts that into `Err(ProtocolError::IdentityElement)`. This is a protocol-level error, not a proof-rejection signal.

At every call site in `do_generation_many`, the result is consumed with `?`: [2](#0-1) [3](#0-2) [4](#0-3) 

When `verify` returns `Err`, the `?` short-circuits before reaching the `AssertionFailed("dlog proof from {from:?} failed to verify")` branch that carries the cheater's identity. The error that propagates is the anonymous `ProtocolError::IdentityElement`.

**Exploit construction** (requires only knowledge of own witness `x`):
1. Attacker picks any scalar `e` and sets `s = e · x`.
2. Verifier computes `big_k = s·G − e·X = e·x·G − e·x·G = identity`.
3. `C::Group::serialize(&identity)` fails → `Err(ProtocolError::IdentityElement)` propagates via `?`.
4. Protocol aborts; no participant is identified as malicious.

### Impact Explanation

Triple generation (`do_generation_many`) is a prerequisite for every ECDSA signing operation in the OT-based path. A single malicious participant can abort every attempt indefinitely. Because the error is anonymous (`IdentityElement`, not `AssertionFailed` with `from`), honest parties have no basis to exclude the attacker. This constitutes **permanent denial of signing** for all honest parties.

### Likelihood Explanation

Any participant in the protocol knows their own secret `x` by definition. The crafted proof requires only one scalar multiplication (`s = e·x`). No cryptographic assumption needs to be broken. The attack is repeatable with zero cost per attempt.

### Recommendation

`verify` must treat an identity `big_k` as a proof-verification failure, not a protocol error. Replace the hard error with a soft rejection:

```rust
// Instead of:
let ser = C::Group::serialize(&big_k).map_err(|_| ProtocolError::IdentityElement)?;

// Use:
let ser = match C::Group::serialize(&big_k) {
    Ok(s) => s,
    Err(_) => return Ok(false), // identity commitment → invalid proof
};
```

This ensures `verify` returns `Ok(false)` for any malformed proof (including the identity-commitment case), allowing callers to attribute the failure to the specific sender and return `AssertionFailed` with the cheater's identity.

### Proof of Concept

```rust
// In src/crypto/proofs/dlog.rs tests:
#[test]
fn test_identity_commitment_returns_false_not_err() {
    use k256::{ProjectivePoint, Scalar};
    use elliptic_curve::Field;
    use frost_secp256k1::Secp256K1Sha256;
    use frost_core::serialization::SerializableScalar;

    let mut rng = rand::thread_rng();
    let x = Scalar::random(&mut rng);
    let big_x = ProjectivePoint::GENERATOR * x;
    let statement = Statement::<Secp256K1Sha256> { public: &big_x };

    // Pick arbitrary e; set s = e*x so that s*G - e*X = identity
    let e = Scalar::random(&mut rng);
    let s = e * x;
    let proof = Proof {
        e: SerializableScalar(e),
        s: SerializableScalar(s),
    };

    let transcript = Transcript::new(b"protocol");
    let result = verify(&mut transcript.fork(b"party", &[1]), statement, &proof);

    // Currently returns Err(IdentityElement); should return Ok(false)
    assert_eq!(result, Ok(false), "got {:?}", result);
}
```

Running this test against the current code will produce `Err(ProtocolError::IdentityElement)`, confirming the bug. The fix makes it return `Ok(false)`, allowing the caller in `generation.rs` to emit `AssertionFailed("dlog proof from {from:?} failed to verify")` and identify the cheater.
