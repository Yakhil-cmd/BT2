### Title
Pool Admin Can Bypass Oracle Rotation Timelock by Overwriting Pending Proposal — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`proposePoolPriceProvider` contains no guard against overwriting an in-flight pending proposal. A pool admin can call it again at any time to silently replace the pending oracle and reset `pendingPriceProviderExecuteAfter` to `now + timelock`, giving the replacement oracle only the remaining seconds of the original window as effective notice — or zero seconds if the overwrite happens at `T-1`.

### Finding Description
`proposePoolPriceProvider` unconditionally overwrites `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]` without checking whether a pending proposal already exists:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol:487-489
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
```

There is no `if (pendingPriceProvider[pool] != address(0)) revert(...)` guard. The attack sequence:

1. Admin proposes oracle A at `T0` → `executeAfter = T0 + L` (users observe proposal for A).
2. At `T0 + L - 1` (one second before expiry), admin proposes oracle B → `executeAfter = T0 + L - 1 + L`.
3. Oracle B is now pending with a fresh full timelock, but users had only **1 second** of on-chain visibility for B.
4. After `L` more seconds, admin executes oracle B.

The admin can also repeat step 2 indefinitely to prevent any oracle from ever being executed (griefing the rotation), or to redirect to a different oracle with arbitrarily short notice.

### Impact Explanation
The timelock was introduced specifically to give LPs and traders advance notice of oracle changes so they can exit before a potentially adverse oracle takes effect (see `AuditFindings.md` M01 resolution). Bypassing it allows the admin to:

- Switch to a different (potentially manipulated) oracle for the same token pair with near-zero effective notice.
- A malicious oracle that returns bad bid/ask prices causes swaps to execute at incorrect prices, resulting in direct loss of user principal or LP assets — qualifying as bad-price execution above Sherlock thresholds.

### Likelihood Explanation
The pool admin is a semi-trusted role explicitly constrained by timelocks. The bypass requires only repeated calls to a function the admin already controls, with no external dependencies. Any pool with `priceProviderTimelock < type(uint256).max` is affected.

### Recommendation
Add a check at the top of `proposePoolPriceProvider` that reverts if a pending proposal already exists:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();
```

Alternatively, require the admin to explicitly cancel the pending proposal (via a new `cancelPoolPriceProviderProposal`) before submitting a new one, preserving the full timelock window for every proposal.

### Proof of Concept
```solidity
// 1. Admin proposes oracle A; timelock = 1 day
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(oracleA));
uint256 firstExpiry = factory.pendingPriceProviderExecuteAfter(pool);

// 2. Warp to T-1 (one second before expiry)
vm.warp(firstExpiry - 1);

// 3. Admin overwrites with oracle B — clock resets
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(oracleB));

// 4. Assert: pending is now B, and executeAfter is in the future (full timelock from now)
assertEq(factory.pendingPriceProvider(pool), address(oracleB));
assertGt(factory.pendingPriceProviderExecuteAfter(pool), block.timestamp);
// Oracle B had only 1 second of visibility before the clock reset
// Original timelock for A is fully bypassed
``` [3](#0-2)
