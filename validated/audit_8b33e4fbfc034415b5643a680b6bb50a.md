### Title
`protocolUnpausePool` Hardcodes Wrong Target Pause Level, Leaving Pool Permanently Admin-Paused After Protocol Pause/Unpause Cycle — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` always calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). When the protocol pauses an active pool (level 0 → 2) and then unpauses it (level 2 → 1), the pool remains paused at admin level. All swaps are blocked until the pool admin separately calls `unpausePool`, even though the admin never paused the pool and may be unaware action is required.

---

### Finding Description

The pause level semantics are: `0 = active`, `1 = paused by admin`, `2 = paused by protocol`.

`protocolPausePool` can pause from **either** level 0 or level 1 to level 2: [1](#0-0) 

`protocolUnpausePool` always transitions to level 1, regardless of what the pre-pause state was: [2](#0-1) 

The hardcoded target `1` is the wrong value when the pool was originally at level 0. The correct value for that case is `0`. This is the direct analog of the Lido bug: just as `requestsToClaim.length + 1` is passed instead of `getLastCheckpointIndex()`, here `1` is passed instead of the correct restored state (`0`), producing a wrong result (pool still paused) rather than the expected outcome (pool active).

The admin-unpause path confirms level 1 is a paused state, not active: [3](#0-2) 

The `swap` function is gated by `whenNotPaused`, which reverts on any non-zero pause level: [4](#0-3) 

---

### Impact Explanation

After a protocol pause/unpause cycle on an active pool, the pool is stuck at pause level 1. Every `swap` call reverts with `PoolPaused`. LP assets are effectively trapped — LPs cannot remove liquidity through normal swap-dependent flows, and no new swaps settle. The pool admin must call `unpausePool` to restore level 0, but the admin never paused the pool and receives no on-chain signal that action is required. If the admin is unresponsive, the pool remains permanently paused.

---

### Likelihood Explanation

This triggers every time the protocol pauses an active pool (level 0) and then unpauses it — the primary emergency-pause scenario. The protocol owner would naturally expect `protocolUnpausePool` to be the exact inverse of `protocolPausePool` when the pool was originally active. The two-step recovery (protocol unpause → admin unpause) is not documented and is not enforced by any on-chain notification to the admin.

---

### Recommendation

Store the pre-protocol-pause level and restore it on unpause, or — simpler — change `protocolUnpausePool` to target level 0 directly:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0); // restore to active
}
```

If preserving any pre-existing admin pause is required, the factory must record the level before `protocolPausePool` and restore it in `protocolUnpausePool`.

---

### Proof of Concept

1. Pool is at pause level 0 (active). `swap()` succeeds.
2. Protocol owner calls `protocolPausePool(pool)`. Pool transitions 0 → 2. `swap()` reverts with `PoolPaused`.
3. Protocol owner calls `protocolUnpausePool(pool)`. Pool transitions 2 → **1** (admin-paused). `swap()` still reverts with `PoolPaused`.
4. Admin must call `unpausePool(pool)` (1 → 0) before swaps resume — but the admin never paused the pool and receives no signal.
5. If the admin is unresponsive, the pool is permanently paused at level 1, blocking all swaps and trapping LP assets. [2](#0-1) [5](#0-4) [6](#0-5)
