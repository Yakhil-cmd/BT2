Audit Report

## Title
Silent Burn Failure in Cycles Minting Canister Breaks ICP/Cycles Economic Invariant - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the CMC deliberately discards ICP ledger burn errors after successfully minting cycles or creating canisters. If the ICP ledger is temporarily unavailable (e.g., during a routine NNS upgrade), cycles are minted or canisters are created without the corresponding ICP being burned. The block index is then permanently marked as processed, preventing any retry. The unburned ICP is stranded in CMC's subaccount with no protocol-level recovery mechanism.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` is defined to explicitly swallow errors:

```rust
/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
``` [1](#0-0) 

On ledger call failure, the error is only logged; no error is propagated and no state is recorded for retry: [2](#0-1) 

`burn_and_log` is called after every successful operation in all three notify paths:

- `process_create_canister`: [3](#0-2) 
- `process_top_up`: [4](#0-3) 
- `process_mint_cycles`: [5](#0-4) 

After `burn_and_log` returns, the caller immediately inserts a final `NotifiedTopUp`/`NotifiedCreateCanister`/`NotifiedMint` status into `blocks_notified`, permanently closing the notification: [6](#0-5) 

Any subsequent call with the same block index hits the `Entry::Occupied` branch and returns the cached result, making retry impossible: [7](#0-6) 

The `NotificationStatus` enum has no `PendingBurn` or equivalent variant, and the `StateV2` struct has no field to track failed burns for deferred retry: [8](#0-7) [9](#0-8) 

## Impact Explanation
Cycles are minted (or canisters created) without the corresponding ICP being burned. This constitutes illegal minting — the ICP/cycles economic invariant is violated. The unburned ICP accumulates permanently in CMC's subaccount: it is not refunded to the user, not re-burned by any timer or heartbeat, and not accessible for any other protocol purpose. This is a concrete, persistent protocol accounting violation in a core NNS financial canister, matching the High impact category: "Significant NNS/ledger security impact with concrete user or protocol harm."

## Likelihood Explanation
The ICP ledger undergoes routine upgrades on the NNS subnet. During the upgrade window (typically seconds to a few minutes), the ledger is unavailable and inter-canister calls to it will be rejected. Any unprivileged user who sends ICP to CMC's subaccount and calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` while the ledger is mid-upgrade will trigger this path. No special privileges, keys, or coordination are required. The user does not need to predict the upgrade window precisely — the CMC processes the burn asynchronously after the main operation succeeds, so even a notification submitted just before an upgrade can hit the failure window. The condition is repeatable across every NNS ledger upgrade cycle.

## Recommendation
Track failed burns in persistent CMC state. Add a `PendingBurn { subaccount: Subaccount, amount: Tokens }` variant to `NotificationStatus` (or a separate `pending_burns: Vec<(Subaccount, Tokens)>` field in `StateV2`). When `burn_and_log` fails, record the pending burn instead of silently discarding it. Process pending burns on a heartbeat or timer tick, retrying the ledger `send_pb` call until it succeeds. This is analogous to how ckBTC/ckETH minters handle failed mint retries and preserves both retry-safety (the original notification is still finalized) and supply conservation correctness.

## Proof of Concept
1. User sends N ICP to `CMC_CANISTER_ID` subaccount with `MEMO_TOP_UP_CANISTER` for target canister C.
2. User calls `notify_top_up(block_index, C)` on CMC.
3. CMC sets `blocks_notified[block_index] = Processing`.
4. CMC calls `deposit_cycles(C, cycles, ...)` → succeeds; cycles are deposited to C.
5. CMC calls `burn_and_log(sub, amount)` → ICP ledger is mid-upgrade; `call_protobuf` returns `Err((RejectionCode::CanisterError, "..."))`.
6. `burn_and_log` logs the error and returns `()`.
7. CMC sets `blocks_notified[block_index] = NotifiedTopUp(Ok(cycles))`.
8. Any subsequent `notify_top_up` call with the same `block_index` returns the cached `Ok(cycles)` — no retry of the burn is possible.
9. **Result**: N ICP worth of cycles minted; N ICP permanently stranded in CMC's subaccount; ICP supply not reduced as the protocol requires.

A deterministic integration test can reproduce this by: (a) deploying CMC and a mock ICP ledger in a PocketIC environment, (b) configuring the mock ledger to reject `send_pb` calls, (c) calling `notify_top_up`, and (d) asserting that cycles were deposited while the CMC subaccount balance on the mock ledger remains unchanged and no burn block was recorded.
