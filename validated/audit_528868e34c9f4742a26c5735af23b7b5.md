### Title
Staker Can Instantly Increase Commission via Commitment to Steal Pool Member Yield - (File: src/staking/staking.cairo)

### Summary
A staker holding an active `CommissionCommitment` can instantly raise their commission to `max_commission` in the same epoch that rewards are distributed. Because pool members are locked behind a multi-week exit wait window, they cannot exit before the inflated commission is applied, resulting in theft of their unclaimed yield.

### Finding Description
Without a commission commitment, `update_commission` enforces a strict monotone-decrease invariant:

```
} else {
    assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
}
``` [1](#0-0) 

However, once a `CommissionCommitment` is active, the invariant is replaced by a ceiling check only:

```
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission, ...
    );
    assert!(commission != old_commission, ...);
}
``` [2](#0-1) 

`max_commission` is bounded only by `COMMISSION_DENOMINATOR = 10000` (100 %):

```
assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
``` [3](#0-2) 

The commission written to storage is read **at reward-distribution time** with no epoch delay:

```
let commission = staker_pool_info.commission();
for (pool_contract, token_address) in staker_pool_info.pools {
    ...
    let (commission_rewards, pool_rewards) = split_rewards_with_commission(
        rewards_including_commission: pool_rewards_including_commission, :commission,
    );
``` [4](#0-3) 

The developers themselves flag this gap:

```
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
``` [5](#0-4) 

Pool members who wish to exit must first call `exit_delegation_pool_intent` and then wait the full `exit_wait_window` (default 1 week, max 12 weeks) before funds are returned:

```
assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);
``` [6](#0-5) 

Because epochs are shorter than the exit wait window, a pool member who observes a commitment with a high `max_commission` cannot exit before the next reward distribution.

### Impact Explanation
The staker captures pool rewards that rightfully belong to delegators. For a pool with `pool_balance` delegated stake and a commission jump from 1 % to 99.99 %, delegators receive 0.01 % of their proportional epoch rewards instead of 99 %. This constitutes **theft of unclaimed yield** (High severity per the allowed impact scope).

### Likelihood Explanation
The attack requires only two permissionless staker-controlled transactions (`set_commission_commitment` then `set_commission`) and no external dependencies. The staker has a direct financial incentive. The exit wait window guarantees that delegators cannot react in time. The attack is repeatable every epoch for the duration of the commitment (up to one year).

### Recommendation
Apply the same epoch-delay mechanism used for balance changes (the K-parameter) to commission increases. A commission increase should be recorded with an effective epoch of `current_epoch + K`, so that pool members have at least K epochs to observe the pending change and submit an exit intent before it takes effect. Alternatively, prohibit commission increases entirely (removing the commitment-based exception) and require stakers to unstake and re-stake with a new commission if they wish to raise it.

### Proof of Concept

```
// Step 1 – Staker opens pool with 1 % commission.
staking.stake(reward_address, operational_address, amount);
staking.set_commission(commission: 100);          // 1 %
staking.set_open_for_delegation(STRK_TOKEN_ADDRESS);

// Step 2 – Staker sets commitment allowing up to 99.99 % commission.
staking.set_commission_commitment(
    max_commission: 9999,
    expiration_epoch: current_epoch + 10,
);

// Step 3 – Delegators join the pool, seeing 1 % commission.
pool.enter_delegation_pool(reward_address, large_amount);

// Step 4 – Delegators cannot exit: exit_wait_window > epoch length.
// pool.exit_delegation_pool_intent(large_amount) starts the clock,
// but unpool_time = now + exit_wait_window > next epoch boundary.

// Step 5 – Just before update_rewards is called for the epoch,
//           staker raises commission to 99.99 %.
staking.set_commission(commission: 9999);

// Step 6 – update_rewards executes with commission = 9999.
//   pool_rewards_including_commission = X
//   commission_rewards = X * 9999 / 10000  ← staker keeps this
//   pool_rewards       = X * 1    / 10000  ← delegators receive this
//
// Delegators lose ~99 % of their expected epoch yield.
```

The root cause is in `update_commission` at `src/staking/staking.cairo` lines 1573–1609 (no epoch delay on commission increases) and the live commission read in `calculate_staker_pools_rewards` at lines 1964–1991.
