### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` Enables Unauthorized ERC-1271 Identity Assertion - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target EVM address or capability path. A signature produced to prove ownership of one Cadence-Owned Account (COA) can be replayed by any unprivileged caller to forge a valid ERC-1271 proof for a different COA owned by the same Cadence account. The function's own inline comment acknowledges this gap but treats it as low-risk based on a behavioral assumption that is not enforced by the protocol.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies that the provided `signatures` are cryptographically valid over `signedData` using the Cadence account's keys.
2. Verifies that the COA resource stored at the caller-supplied `path` has an EVM address matching the caller-supplied `evmAddress`.

The function's own comment at lines 1003–1009 states:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

Because `signedData` is not required to commit to `evmAddress` or `path`, the two checks are entirely decoupled. An attacker who observes a valid `COAOwnershipProof` for COA_A can construct a structurally identical proof with `CapabilityPath` changed to point to COA_B, reuse the same `signatures`, and pass it to `validateCOAOwnershipProof` with `evmAddress = COA_B.address`. Both checks pass independently, and the function returns `isValid: true`.

The concrete attack path runs through the ERC-1271 `isValidSignature` function in the deployed COA contract (`fvm/evm/handler/coa/coa.sol`, line 114–125):

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig) external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig)
    );
```

Here `address(this)` is the calling COA's EVM address (COA_B), `_hash` is `signedData`, and `_sig` is the attacker-supplied encoded `COAOwnershipProof`. The attacker controls the `CapabilityPath` field inside `_sig`. The precompile handler in `fvm/evm/precompiles/arch.go` (`DecodeABIEncodedProof`, lines 219–243) decodes these fields and passes them directly to `validateCOAOwnershipProof` via `coaOwnershipProofValidator` in `fvm/evm/handler/precompiles.go` (lines 115–153).

Step-by-step exploit:

1. Alice owns two COAs: COA_A at `/public/coa1` (EVM address `0xAAAA`) and COA_B at `/public/coa2` (EVM address `0xBBBB`).
2. Alice signs `data` to prove ownership of COA_A and publishes proof_A: `{Address: alice, CapabilityPath: "coa1", KeyIndices: [0], Signatures: [sig]}`.
3. Attacker observes proof_A (e.g., from a public transaction or off-chain authentication flow).
4. Attacker constructs proof_B: `{Address: alice, CapabilityPath: "coa2", KeyIndices: [0], Signatures: [sig]}` — identical except for `CapabilityPath`.
5. Attacker calls `COA_B.isValidSignature(data, encode(proof_B))`.
6. The precompile calls `validateCOAOwnershipProof(alice, /public/coa2, data, [0], [sig], 0xBBBB)`.
7. Signature check: `sig` is valid over `data` with alice's key → **passes**.
8. COA address check: COA at `/public/coa2` has address `0xBBBB` == `evmAddress` → **passes**.
9. Returns `0x1626ba7e` (`ValidERC1271Signature`).

Alice never authorized COA_B for `data`. The attacker has forged a valid ERC-1271 proof for COA_B.

### Impact Explanation

Any EVM contract that uses `isValidSignature` (ERC-1271) to gate privileged actions — NFT marketplace approvals, DeFi protocol authorizations, multisig wallet confirmations, permit-style flows — will accept the attacker's forged proof as Alice's authorization for COA_B. This constitutes unauthorized account mutation and cross-VM asset loss: the attacker can impersonate Alice's COA_B in any EVM protocol that trusts ERC-1271 without additional context binding.

### Likelihood Explanation

The attack requires Alice to own multiple COAs at different public paths. The inline comment acknowledges this is currently uncommon ("Flow wallets historically create at most one COA per account"), but the protocol imposes no limit. As the Flow EVM ecosystem matures and protocols or power users create multiple COAs per account (e.g., for role separation, multi-tenant dApps, or bridge integrations), the precondition becomes increasingly realistic. The attacker needs only to observe one valid proof — which may be emitted in a public transaction — and can replay it immediately with no privileged access.

### Recommendation

Enforce within `validateCOAOwnershipProof` that `signedData` commits to `evmAddress` (and optionally `path`) before accepting the proof. Concretely, the function should verify that `signedData` encodes `evmAddress` as a prefix or structured field, rejecting proofs where `signedData` is opaque. Alternatively, the function should hash `evmAddress || path || signedData` and require signatures over that composite, eliminating cross-COA portability of signatures.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}

func coaOwnershipProofValidator(contractAddress flow.Address, backend backends.Backend) func(proof *types.COAOwnershipProofInContext) (bool, error) {
	return func(proof *types.COAOwnershipProofInContext) (bool, error) {
		value, err := backend.Invoke(
			environment.ContractFunctionSpec{
				AddressFromChain: func(_ flow.Chain) flow.Address {
					return contractAddress
				},
				LocationName: "EVM",
				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}
		data, ok := value.(cadence.Struct)
		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		isValidValue := cadence.SearchFieldByName(data, ValidationResultTypeIsValidFieldName)
		if isValidValue == nil {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
}
