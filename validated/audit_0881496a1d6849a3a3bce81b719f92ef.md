Audit Report

## Title
OFAC Blocklist Bypass via Proxy Contract — `msg.sender` Reflects Proxy, Not Sanctioned Originator - (File: `rs/ethereum/cketh/minter/src/deposit.rs`, `rs/ethereum/cketh/minter/src/blocklist.rs`)

## Summary

The ckETH/ckERC20 minter enforces OFAC sanctions screening by checking only the `from_address` field extracted from deposit event logs, which corresponds to `msg.sender` inside `DepositHelperWithSubaccount.sol`. When a sanctioned address routes a deposit through an intermediary (proxy) smart contract, `msg.sender` inside the helper is the proxy's address — not the sanctioned originator. Since the proxy is not on the blocklist, `is_blocked` returns `false` and minting proceeds to the sanctioned entity's IC principal, violating the stated invariant: *"ETH is not accepted from nor sent to addresses on this list."*

## Finding Description

**Step 1 — Helper contract emits `msg.sender` as the depositor identity:**

`DepositHelperWithSubaccount.sol` unconditionally emits `msg.sender` as the `owner` field in both deposit functions: [1](#0-0) [2](#0-1) 

When a proxy contract calls `depositEth` or `depositErc20`, `msg.sender` = proxy address, not the sanctioned EOA that initiated the chain.

**Step 2 — All three log parsers extract `from_address` from the `msg.sender` topic:**

`ReceivedEthLogParser` reads `from_address` from `topics[1]`: [3](#0-2) 

`ReceivedErc20LogParser` and `ReceivedEthOrErc20LogParser` both read `from_address` from `topics[2]`: [4](#0-3) [5](#0-4) 

**Step 3 — Blocklist check operates solely on this `from_address`:**

`register_deposit_events` calls `is_blocked(&event.from_address())` with no check against `tx.origin` or any other field identifying the ultimate transaction signer: [6](#0-5) 

**Step 4 — `is_blocked` is a static OFAC SDN list lookup:**

The blocklist is generated from the OFAC SDN XML list and performs a binary search. A proxy address not on the list returns `false`, and `event.into_deposit()` is called, triggering minting: [7](#0-6) 

The blocklist is explicitly sourced from OFAC SDN data: [8](#0-7) 

**Existing guards are insufficient:** The existing tests (`should_block_deposit_from_blocked_address`) only verify that a direct deposit from a listed address is blocked. They do not test the proxy-forwarding scenario where `from_address` is an unlisted proxy contract. [9](#0-8) 

## Impact Explanation

A sanctioned entity can receive ckETH or ckERC20 tokens by deploying any smart contract not on the blocklist as a proxy and having it call the official helper contract with the sanctioned entity's IC principal as the recipient. The minter observes `owner = proxy_address`, passes `is_blocked(proxy_address) == false`, and mints to the sanctioned entity's IC account. This constitutes a **significant ck-token security impact with concrete protocol harm** — the OFAC sanctions screening mechanism, which is the protocol's primary compliance control for ckETH/ckERC20, is rendered entirely ineffective against proxy-routed deposits. This matches the High ($2,000–$10,000) impact tier: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."*

## Likelihood Explanation

The attack requires no privileged access, no key compromise, and no protocol-level attack. Any Ethereum user can deploy a forwarding contract in a single transaction at the cost of Ethereum gas. The technique is well-known and trivially executable. The sanctioned entity does not need to interact with the IC directly — the IC principal encoded in the deposit can belong to any account they control. The attack is repeatable indefinitely.

## Recommendation

The minter cannot reliably enforce sanctions purely at the `msg.sender` layer because Ethereum does not prevent contract-to-contract calls. Recommended mitigations:

1. **Emit and check `tx.origin` in addition to `msg.sender`** — add a `tx.origin` field to the `ReceivedEthOrErc20` event in the helper contract, parse it in the minter's log parsers, and block the deposit if either `msg.sender` or `tx.origin` is on the blocklist. `tx.origin` is the EOA that signed the Ethereum transaction and cannot be spoofed by a proxy.
2. **Deploy a new helper contract** emitting both fields and upgrade the minter to parse the additional topic.
3. If the design intentionally only screens the immediate caller, document that proxy-based bypasses are accepted risk and out of scope.

Options 1/2 require a new helper contract deployment and a minter upgrade to parse the additional field.

## Proof of Concept

```solidity
// Proxy contract (not on blocklist)
contract Proxy {
    function forwardDeposit(
        address helper,
        bytes32 principal,
        bytes32 subaccount
    ) external payable {
        // msg.sender inside depositEth will be address(this), not tx.origin
        CkDeposit(helper).depositEth{value: msg.value}(principal, subaccount);
    }
}
```

Reproduction steps:
1. Deploy `Proxy` from `SAMPLE_BLOCKED_ADDRESS` (= `0x0330070FD38Ec3bB94F58FA55D40368271E9e54A`).
2. Call `proxy.forwardDeposit{value: 1 ether}(helperAddress, encodedPrincipal, 0x00...)` from `SAMPLE_BLOCKED_ADDRESS`.
3. The helper emits `ReceivedEthOrErc20(address(0), proxy_address, 1 ether, encodedPrincipal, 0x00...)`.
4. The minter scrapes the log, calls `is_blocked(proxy_address)` → `false`.
5. `register_deposit_events` calls `event.into_deposit()` → minting proceeds.
6. Assert: `is_blocked(SAMPLE_BLOCKED_ADDRESS) == true`, `is_blocked(proxy_address) == false`, and ckETH is minted to the sanctioned entity's IC principal.

A deterministic integration test can be written using the existing `CkEthSetup`/`CkErc20Setup` test harness by constructing a log entry where `from_address` is an unlisted proxy address while `tx.origin` (if added) would be `SAMPLE_BLOCKED_ADDRESS`, confirming the current code mints without blocking.
