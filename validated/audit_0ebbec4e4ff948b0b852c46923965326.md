### Title
Invoke Transactions with Nonce=1 Bypass Signature Validation When Sender Has Any Pending Transaction in Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry-point call (the on-chain signature check) for any Invoke transaction with `nonce=1` whose sender address appears in the mempool or recent-block state. Because the presence check (`account_tx_in_pool_or_recent_block`) is not type-restricted to `DeployAccount` transactions, an unprivileged attacker who observes a pending `DeployAccount` for address A can immediately submit an Invoke with `nonce=1` and an **arbitrary/wrong signature** for A, and the gateway will admit it to the mempool without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the mempool's nonce/duplicate check but **before** the blockifier's `__validate__` entry-point execution:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range, resource bounds)
       ├─ validate_by_mempool            (duplicate hash, nonce-too-old only)
       └─ skip_stateful_validations      ← returns true → __validate__ skipped
  └─ run_validate_entry_point(skip_validate=true)
       └─ ExecutionFlags { validate: false }  → __validate__ NOT called
```

The skip condition is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
```

`account_tx_in_pool_or_recent_block` is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)   // staged or committed nonce map
        || self.tx_pool.contains_account(account_address)  // ANY tx type in pool
}
```

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool for that address — not only `DeployAccount`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is circular for the very first nonce-1 Invoke being evaluated: it skips validation because a `DeployAccount` is present, yet the `DeployAccount` itself provides no proof that the Invoke's signature is valid.

**Attack path:**

1. Alice broadcasts a `DeployAccount` for address A (nonce=0). It enters the mempool's `tx_pool`.
2. Attacker observes the pending `DeployAccount` (mempool is public).
3. Attacker crafts an Invoke for address A with `nonce=1` and a **completely wrong signature** (e.g., all-zero bytes).
4. Gateway stateless validator passes (signature size is within limits).
5. `validate_by_mempool` passes (not a duplicate, nonce 1 ≥ account nonce 0).
6. `skip_stateful_validations`: `nonce==1 && account_nonce==0` → `account_tx_in_pool_or_recent_block(A)` → `true` (Alice's `DeployAccount` is in the pool) → returns `true`.
7. `run_validate_entry_point(skip_validate=true)` → `ExecutionFlags { validate: false }` → `__validate__` is **never called**.
8. The invalid Invoke is forwarded to the mempool and admitted.

The attacker pays no fee (the transaction will fail at `__validate__` during batcher execution and be rejected without fee charge), so this attack is **free to repeat** for every observed `DeployAccount`.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

- Invalid Invoke transactions (arbitrary signatures) are admitted to the mempool without signature verification.
- Each such transaction consumes mempool capacity (`capacity_in_bytes`). An attacker can flood the mempool with invalid Invokes for every observed pending `DeployAccount`, potentially exhausting capacity and causing legitimate transactions to be evicted or rejected.
- The batcher will eventually reject these transactions when `__validate__` fails during execution, but by then mempool slots have been consumed and sequencer CPU has been wasted.
- No fees are charged for rejected-at-validate transactions, making the attack economically free.

### Likelihood Explanation

**Medium.** The mempool is publicly observable. Any `DeployAccount` transaction broadcast to the gateway is visible to any network participant. The attacker only needs to submit a single malformed Invoke per observed `DeployAccount`. No privileged access, special keys, or on-chain state is required.

### Recommendation

1. **Type-restrict the skip condition.** Instead of checking whether *any* transaction for the address is in the mempool, verify that a `DeployAccount` transaction specifically is present. The mempool's `tx_pool` should expose a `has_deploy_account_for_address` query, or the `AddTransactionArgs` stored in the pool should be inspected for transaction type.

2. **Alternatively**, remove the UX shortcut entirely and require users to wait for the `DeployAccount` to be confirmed before submitting the first Invoke, or handle the skip only at the batcher level (where the `DeployAccount` is already in the same proposal batch and the Invoke can be validated in sequence).

3. **At minimum**, add a guard that the transaction in the pool for the address is of type `DeployAccount` (nonce=0), not an arbitrary future-nonce Invoke that itself benefited from the same skip.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
