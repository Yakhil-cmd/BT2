### Title
Missing Participant-Set Binding in OT-Based ECDSA Presign Allows Triple Reuse Across Mismatched Participant Sets — (`File: src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign` function in the OT-based ECDSA implementation validates that the **threshold** of the supplied Beaver triples matches the presign threshold, but **explicitly omits** the analogous check that the **participant set** used during triple generation matches the participant set used during presigning. This is a partial invariant enforcement bug: the threshold half of the binding constraint is checked, but the participant-set half is not, mirroring the external report's pattern where `f(tm)` is checked but `f(0)` is not.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the initialization guard reads:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [1](#0-0) 

The comment is a self-documented admission that the participant-set invariant is unenforced. The triple output stores a `threshold` field (accessed as `args.triple0.1.threshold`) but does **not** store the participant set used during generation. Consequently, there is no structural mechanism to enforce that the participant set at presign time matches the participant set at triple-generation time.

The Beaver triple protocol produces shares `(a_i, b_i, c_i)` such that `Σ λ_i · a_i = A`, `Σ λ_i · b_i = B`, `A·B = C`, where `λ_i` are Lagrange coefficients computed over the **triple-generation participant set**. When presign runs with a **different participant set**, it recomputes Lagrange coefficients over the new set. The algebraic relationship `A·B = C` no longer holds under the new coefficients, so the presignature nonce `k` and the blinded key share `χ` are computed incorrectly.

This is structurally identical to the external report's finding: the codebase enforces one half of a two-part invariant (threshold equality) while leaving the other half (participant-set equality) entirely unenforced, with the gap explicitly noted in a comment rather than fixed.

---

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

A malicious coordinator who controls triple assignment can supply triples generated by participant set `{P1, P2, P3}` to a presign session running with participant set `{P1, P2, P4}` (same threshold, different members). Because the Lagrange coefficients differ between the two sets, the reconstructed nonce `k` and the masked key share `χ` are arithmetically wrong. The presign protocol completes without error (no check catches the mismatch), and honest parties store a corrupted `PresignOutput`. Any subsequent signing attempt using that output produces an invalid ECDSA signature. Depending on the exact arithmetic divergence, partial information about individual secret key shares may also be exposed through the incorrect linear combination.

---

### Likelihood Explanation

**Medium.** The coordinator role is required to assign triples to presign sessions. A malicious or compromised coordinator can trivially exploit this by reusing triples across sessions with different participant sets. No cryptographic capability is needed — only the ability to supply `PresignArguments` with mismatched triples, which is the normal API surface. The explicit comment in the code confirms the gap is known and accepted as a trust assumption, making it a documented but unenforced invariant.

---

### Recommendation

1. **Store the participant set in the triple output.** The triple generation protocol should record the sorted participant list (or a collision-resistant hash of it) alongside the threshold in its output type.
2. **Enforce participant-set equality at presign initialization.** Add a check analogous to the existing threshold check:
   ```rust
   if args.participants_hash != args.triple0.1.participants_hash
       || args.participants_hash != args.triple1.1.participants_hash {
       return Err(InitializationError::BadParameters(
           "Participant set must match the participant set of both triples".to_string(),
       ));
   }
   ```
3. **Remove the trust-assumption comment** once the structural fix is in place, so the invariant is enforced rather than documented.

---

### Proof of Concept

1. Coordinator runs triple generation with participants `{P1, P2, P3}`, threshold `t = 2`. Triple output records only `threshold = 2`; participant set is discarded.
2. Coordinator initiates a presign session with participants `{P1, P2, P4}`, threshold `t = 2`, supplying the triples from step 1 as `args.triple0` and `args.triple1`.
3. The threshold check at line 43 passes: `2 == 2 && 2 == 2`.
4. The participant-set check is explicitly omitted (line 38–40 comment).
5. `do_presign` proceeds; Lagrange coefficients are computed over `{P1, P2, P4}` instead of `{P1, P2, P3}`.
6. The reconstructed nonce `k` and masked key share `χ` are arithmetically incorrect because the triple relationship `A·B = C` was established under the Lagrange basis of `{P1, P2, P3}`.
7. All honest parties store a corrupted `PresignOutput` with no error. Signing with this output produces an invalid signature, permanently denying signing capability for this presign round. [2](#0-1)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
