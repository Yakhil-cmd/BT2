### Title
Malicious CKD Participant Can Corrupt Derived Key Output Without Proof of Correct Computation - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The CKD coordinator (`do_ckd_coordinator`) aggregates participant contributions `(norm_big_y, norm_big_c)` without any cryptographic proof that each contribution was correctly computed from the participant's actual secret share. A malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that honest parties accept as valid, permanently destroying the correctness of the derived confidential key.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `CKDOutput` and unconditionally adds it to the running aggregate: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant is expected to send `(λ_i · y_i · G,  λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk))` where `x_i` is their DKG secret share and `y_i` is a fresh random scalar, as computed in `compute_signature_share`: [2](#0-1) 

However, the coordinator performs **no verification** that:
1. `norm_big_y` is of the form `λ_i · y_i · G` for any `y_i`, and
2. `norm_big_c` encodes the participant's actual DKG share `x_i` and the **same** `y_i`.

A malicious participant can bypass `do_ckd_participant`: [3](#0-2) 

and instead send arbitrary group elements directly over the `SharedChannel`. The coordinator has no Sigma proof, commitment binding, or consistency check to detect or reject such malformed contributions.

The analog to the AutoTrader report is exact: just as the Solo contract accepted any arbitrary contract as an `AutoTrader` and called `getTradeCost()` on it without on-chain validation of the returned value, the CKD coordinator accepts any arbitrary `(big_y, big_c)` pair from any registered participant without cryptographic validation of the returned contribution.

---

### Impact Explanation

A malicious participant corrupts the aggregate:

- `norm_big_y_corrupted = big_y_attacker + Σ_{honest i} λ_i · y_i · G`
- `norm_big_c_corrupted = big_c_attacker + Σ_{honest i} λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

When the TEE application unmasks with `app_sk`:

```
norm_big_c_corrupted − app_sk · norm_big_y_corrupted
  = (big_c_attacker − app_sk · big_y_attacker)
    + Σ_{honest i} λ_i · x_i · H(pk ‖ app_id)
  ≠ msk · H(pk ‖ app_id)
```

The coordinator outputs this corrupted `CKDOutput` as valid with no error. Honest parties have no mechanism to detect the corruption. The TEE application receives an incorrect confidential key, permanently breaking the CKD output for that `app_id`.

**Matched impact:** *High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

Any registered participant in the CKD protocol can trigger this. The attacker needs only to be a valid member of the `participants` list and to send arbitrary bytes over the communication channel instead of the library's legitimate computation. No leaked keys, trusted-host compromise, or cryptographic break is required. The attack is single-round and requires no coordination.

---

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation. Concretely, a Chaum-Pedersen proof can demonstrate that:

```
norm_big_c − norm_big_y · (app_pk / G) = λ_i · x_i · H(pk ‖ app_id)
```

where `x_i · G` is the participant's public verification share from DKG (already committed to and publicly known). This proves the contribution is consistent with the participant's committed secret share without revealing `x_i` or `y_i`. The coordinator should verify this proof before adding each contribution to the aggregate.

---

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` and malicious participant `P_m` are all registered in `participants`.
2. `P_m` does not call `compute_signature_share`. Instead it sends `(big_y' = identity, big_c' = identity)` to the coordinator over the channel.
3. The coordinator in `do_ckd_coordinator` (line 50–55) adds these zero contributions without error:
   - `norm_big_y = Σ_{i≠m} λ_i · y_i · G`
   - `norm_big_c = Σ_{i≠m} λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`
4. Unmasked result: `Σ_{i≠m} λ_i · x_i · H(pk ‖ app_id) ≠ msk · H(pk ‖ app_id)`.
5. The coordinator returns `Some(ckd_output)` with the corrupted value. The TEE application receives and uses an incorrect confidential key with no indication of failure. [4](#0-3)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
