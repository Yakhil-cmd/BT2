### Title
Missing Association Check Between `to_staker` and `to_pool` in `switch_delegation_pool` Enables Cross-Staker Accounting Corruption - (File: `src/pool/pool.cairo`)

### Summary

The `switch_delegation_pool` function in the Pool contract accepts caller-controlled `to_staker` and `to_pool` parameters without any on-chain validation that `to_pool` is the registered delegation pool for `to_staker`. An unprivileged pool member with an active unpool intent can supply a mismatched pair, causing the staking contract to credit the switched amount to the wrong staker's delegated balance while simultaneously registering the pool member in an unrelated pool. This corrupts the staking contract's accounting and can permanently freeze the affected pool members' funds.

### Finding Description

`switch_delegation_pool` in `src/pool/pool.cairo` (lines 379–429) is callable by any pool member who has an active unpool intent. It accepts two fully caller-controlled parameters: `to_staker` (the destination staker) and `to_pool` (the destination pool contract). It then forwards both directly to the staking contract via `switch_staking_delegation_pool`: [1](#0-0) 

The staking contract's `IStakingPool::switch_staking_delegation_pool` interface (lines 221–228) lists only two conditions:

1. An intent entry exists for `(caller_pool, identifier)`.
2. `to_staker` is not in exit window. [2](#0-1) 

There is **no condition** requiring that `to_pool` is the registered delegation pool for `to_staker`. The interface's documented flow (lines 209–228) shows the staking contract will:

- Increase `to_staker`'s delegated balance by `switched_amount` (step 2).
- Increase `total_stake` by `switched_amount` (step 3).
- Call `to_pool.enter_delegation_pool_from_staking_contract` (step 5). [3](#0-2) 

`enter_delegation_pool_from_staking_contract` in the pool contract only asserts the caller is the staking contract, so it will accept the call regardless of whether `to_pool` belongs to `to_staker`: [4](#0-3) 

The pool's own interface documentation lists `to_pool` being the pool for `to_staker` only as a **precondition** (not enforced on-chain): [5](#0-4) 

### Impact Explanation

When an attacker calls `switch_delegation_pool(to_staker=StakerA, to_pool=PoolB)` where PoolB belongs to StakerB:

1. The staking contract credits `switched_amount` to `staker_delegated_balance_trace[StakerA][PoolB]` and increases `total_stake`.
2. PoolB registers the attacker as a pool member with `switched_amount`.
3. PoolB's `staker_address` is StakerB (set at construction). All subsequent pool operations (exit intent, exit action) call the staking contract with `staker_address = StakerB`.
4. When PoolB members later call `exit_delegation_pool_intent`, the staking contract attempts to reduce StakerB's delegated balance — but the funds were credited to StakerA. This mismatch causes either a revert or underflow, **permanently freezing** PoolB members' funds.
5. `total_stake` is permanently inflated, diluting all other stakers' relative staking power and rewards.

Impact: **Temporary/permanent freezing of delegator funds** (High) and **griefing of the protocol's accounting** (Medium).

### Likelihood Explanation

Any pool member who has called `exit_delegation_pool_intent` (a normal, permissionless user action) can immediately exploit this. No privileged role, leaked key, or external dependency is required. The attacker only needs a non-zero `unpool_amount`. [6](#0-5) 

### Recommendation

In the staking contract's `switch_staking_delegation_pool` implementation, add an explicit check that `to_pool` is the registered delegation pool for `to_staker` before crediting the balance. Concretely, verify:

```
assert!(
    self.staker_pool_info.entry(to_staker).pools.contains(to_pool),
    "to_pool is not the registered pool for to_staker"
);
```

This mirrors the fix described in the external report: enforce the association at the account-validation level, not merely as a documented precondition.

### Proof of Concept

1. Pool member Alice is in PoolA (associated with StakerA). She calls `exit_delegation_pool_intent(amount=X)`, setting `unpool_amount=X`.
2. Alice calls `PoolA.switch_delegation_pool(to_staker=StakerA_legit, to_pool=PoolB, amount=X)` where PoolB is StakerB's pool.
3. PoolA calls `staking.switch_staking_delegation_pool(to_staker=StakerA_legit, to_pool=PoolB, switched_amount=X, data=..., identifier=Alice)`.
4. The staking contract (no association check): increases `staker_delegated_balance_trace[StakerA_legit][PoolB]` by X, increases `total_stake` by X, calls `PoolB.enter_delegation_pool_from_staking_contract(amount=X, data=...)`.
5. PoolB registers Alice as a member with balance X, using StakerB as its `staker_address`.
6. Alice later calls `PoolB.exit_delegation_pool_intent(amount=X)`. PoolB calls `staking.remove_from_delegation_pool_intent(staker_address=StakerB, identifier=Alice, amount=X)`. The staking contract tries to reduce StakerB's delegated balance — but X was never added to StakerB's balance. The call reverts or corrupts StakerB's accounting, freezing Alice's funds permanently.         fn switch_delegation_pool(
            ref self: ContractState,
            to_staker: ContractAddress,
            to_pool: ContractAddress,
            amount: Amount,
        ) -> Amount {
            // Asserts.
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            assert!(
                pool_member_info.unpool_time.is_some(),
                "{}",
                GenericError::MISSING_UNDELEGATE_INTENT,
            );
            assert!(amount <= pool_member_info.unpool_amount, "{}", GenericError::AMOUNT_TOO_HIGH);
            let reward_address = pool_member_info.reward_address;

            // Update pool_member_info and write to storage.
            pool_member_info.unpool_amount -= amount;
            if pool_member_info.unpool_amount.is_zero() {
                // unpool_amount is zero, clear unpool_time.
                pool_member_info.unpool_time = Option::None;
            }
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Serialize the switch pool data and invoke the staking contract to switch pool.
            let switch_pool_data = SwitchPoolData { pool_member, reward_address };
            let mut serialized_data = array![];
            switch_pool_data.serialize(ref output: serialized_data);
            self
                .staking_pool_dispatcher
                .read()
                .switch_staking_delegation_pool(
                    :to_staker,
                    :to_pool,
                    switched_amount: amount,
                    data: serialized_data.span(),
                    identifier: pool_member.into(),
                );

                    :staker_address, identifier: pool_member.into(), :amount,

    /// Registers an intention to remove `amount` FRI of pooled stake from the staking contract.
    /// Returns the timestmap when the pool is allowed to remove the `amount` for `identifier`.
    ///
    /// The flow, if making a brand new intent:
    /// 1. Update rewards for `staker_address`.
    /// 2. Decrease the staker's pooled amount by `amount`.
    /// 3. Decrease total_stake by `amount`.
    /// 4. Calculate the timestamp when the pool may perform remove_from_delegation_pool_action for
    ///    this `amount` and `identifier` (notate it as unpool_time for following use).
    /// 5. Create an entry in pool_exit_intents map for this `identifier` and pool contract address
    ///    with the value being `UndelegateIntentValue { amount, unpool_time, token_address }`.
    /// 6. Return unpool_time.
    ///
    /// The function supports overriding intentions, upwards and downwards, *which recalculates the
    /// unpool_time and restarts the timer*. This slightly changes the flow, meaning that if the
    /// pool already has an intent for this `identifier`, the flow remains the same except for
    /// points 2 and 3:
    /// * If the amount to be removed is greater in the previous intent, the staker's pooled amount
    ///   and total_stake will be *decreased* by the difference between the new and the old amount.
    /// * If the amount to be removed is smaller in the previous intent, the staker's pooled amount
    ///   and total_stake will be *increased* by the difference between the old and the new amount.
    ///
    /// If the amount to be removed is zero, any existing intent associated with this `identifier`
    /// will be removed, and the returned unpool time will be zero.
    fn remove_from_delegation_pool_intent(
        ref self: TContractState,
        staker_address: ContractAddress,
        identifier: felt252,
        amount: Amount,
    ) -> Timestamp;
