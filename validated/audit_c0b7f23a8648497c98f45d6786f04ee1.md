### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Skip Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is documented to be callable "Only starkware sequencer" but has no on-chain caller restriction. Any unprivileged address can call it with `disable_rewards: true`, which permanently marks the current block as processed (`last_reward_block` is updated) without distributing any rewards. The sequencer's subsequent call for the same block will revert with `REWARDS_ALREADY_UPDATED`, permanently freezing that block's unclaimed yield for all stakers and delegators.

### Finding Description

The spec for `update_rewards` explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation in `src/staking/staking.cairo` contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks CONTRACT_IS_PAUSED
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validation ...

    // Permanently marks this block as processed:
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;   // <-- exits with NO rewards distributed
    }
    // ... reward distribution ...
}
``` [1](#0-0) 

`general_prerequisites()` only enforces the pause check via `assert_is_unpaused`: [2](#0-1) 

The spec's access control requirement is documented but never enforced in code: [3](#0-2) 

The `last_reward_block` write at line 1485 is unconditional — it happens before the `disable_rewards` branch. Once written, no further call for the same block number can succeed: [4](#0-3) 

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

When an attacker calls `update_rewards(any_active_staker, disable_rewards: true)`:
1. `last_reward_block` is set to the current block number.
2. The function returns immediately with zero rewards distributed.
3. The sequencer's own `update_rewards` call for the same block reverts with `REWARDS_ALREADY_UPDATED`.
4. That block's rewards are permanently lost — there is no recovery path, no retry mechanism, and no way to retroactively credit the skipped block.

This affects all stakers and their delegators, since `update_rewards` is the sole mechanism for distributing consensus-era block rewards. [5](#0-4) 

### Likelihood Explanation

**Likelihood: Medium.**

- The function is publicly callable with no role check.
- An attacker only needs to submit a transaction with `disable_rewards: true` targeting any active staker before the sequencer's system call lands.
- On Starknet, the sequencer controls ordering but must include user transactions. A determined attacker can repeatedly attempt this across many blocks, causing cumulative yield loss.
- The attack requires no capital, no special role, and no prior setup beyond knowing an active staker address (which is public on-chain).

### Recommendation

Add an explicit sequencer-only caller check at the top of `update_rewards`, consistent with the spec's stated access control:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // enforce spec requirement
    // ...
}
```

Alternatively, if the intent is to allow any caller, the `disable_rewards` parameter should be removed and the function should always distribute rewards when consensus is active, preventing griefing via the `disable_rewards: true` path.

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker identifies any active staker address `S`.
3. Attacker submits: `staking.update_rewards(S, disable_rewards: true)`.
4. Transaction is included in block `B`. `last_reward_block` is set to `B`. No rewards are distributed.
5. Sequencer submits its own `update_rewards(S, disable_rewards: false)` for block `B`.
6. Transaction reverts: `REWARDS_ALREADY_UPDATED` (current block `B` is not `> last_reward_block` which is also `B`).
7. Block `B`'s rewards for all stakers and delegators are permanently frozen/lost. [6](#0-5)

            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,

            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,

            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self

                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake

            self.last_reward_block.write(current_block_number);

                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
