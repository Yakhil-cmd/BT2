### Title
Missing `max_l2_gas_amount` Upper-Bound Check for Declare Transactions Allows Admission-Control Bypass - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions via an empty `if let` branch. An unprivileged user can submit a Declare transaction with `l2_gas.max_amount` set to any value up to `u64::MAX`, bypassing the gateway's admission-control limit that exists specifically to prevent resource exhaustion.

### Finding Description

In `validate_resource_bounds`, the check reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The empty `if let RpcTransaction::Declare(_) = tx {}` arm is a deliberate no-op that causes the `max_l2_gas_amount` guard to be skipped entirely for Declare transactions. The production default for `max_l2_gas_amount` is `1,210,000,000`. [2](#0-1) 

This is confirmed by the test `valid_l2_gas_amount_on_declare`, which explicitly asserts that a Declare transaction with `l2_gas.max_amount = 200` (above a configured limit of `100`) passes validation: [3](#0-2) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* threshold against the previous block price — it does not re-enforce the `max_l2_gas_amount` ceiling: [4](#0-3) 

The only downstream guard is `verify_can_pay_committed_bounds`, which checks `max_amount × max_price_per_unit ≤ balance`. This is not equivalent to the missing check: an attacker with sufficient STRK balance can set `l2_gas.max_amount` to any value above `1,210,000,000` and still pass the balance check, provided `max_price_per_unit` is set to the minimum (`min_gas_price = 8,000,000,000`). [5](#0-4) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` set above `max_l2_gas_amount` (e.g., `u64::MAX`) is admitted through the stateless gateway check and proceeds to stateful validation and the mempool. The `l2_gas.max_amount` field sets the initial Sierra gas budget for execution (`initial_user_gas_bound` in the OS). If the bouncer uses declared bounds for pessimistic block-space reservation, a single Declare with an enormous `l2_gas.max_amount` can monopolize the block's L2 gas budget, preventing other transactions from being included. Even without bouncer impact, the gateway's admission-control invariant — that no transaction with `l2_gas.max_amount > max_l2_gas_amount` enters the mempool — is broken for Declare transactions.

**Impact category:** High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.

### Likelihood Explanation

Any unprivileged user can craft a valid Declare transaction (valid Sierra class, valid signature, valid nonce) and set `l2_gas.max_amount` to an arbitrarily large value. The only prerequisite is sufficient STRK balance to satisfy `verify_can_pay_committed_bounds`. No special privileges or insider access are required.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx {}` arm and apply the same `max_l2_gas_amount` upper-bound check to Declare transactions as is applied to Invoke and DeployAccount transactions. The TODO comment at line 78 already acknowledges this gap; it should be resolved rather than deferred.

```rust
// Apply to all transaction types uniformly:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas ceiling (e.g., for large Sierra classes), a separate `max_l2_gas_amount_declare` config field should be introduced rather than removing the check entirely.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with:
   - A valid Sierra class (passes `validate_class_length`, `validate_sierra_version`, `validate_entry_points_sorted_and_unique`)
   - `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` (or any value > `1,210,000,000`)
   - `resource_bounds.l2_gas.max_price_per_unit = GasPrice(8_000_000_000)` (minimum allowed)
   - A sender account with STRK balance ≥ `8_000_000_000 × 1_210_000_001` (just above the Invoke limit)
2. Submit via `POST /add_transaction`.
3. Observe: `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never triggered for Declare.
4. The transaction proceeds to stateful validation and enters the mempool with `l2_gas.max_amount = u64::MAX`, bypassing the admission-control limit that would have rejected an equivalent Invoke or DeployAccount transaction. [6](#0-5)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}
