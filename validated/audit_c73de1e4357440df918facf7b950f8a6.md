### Title
Cross-COA Proof Replay via Unsigned `CapabilityPath` — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`validateCOAOwnershipProof` verifies account key signatures over `signedData` only. The `CapabilityPath` field inside the RLP-encoded `COAOwnershipProof` is **never included in what is signed**. An attacker who has observed a valid proof for COA_A can swap `CapabilityPath` to point at COA_B (on the same Flow account), keep the original signatures, and pass the modified proof to `verifyCOAOwnershipProof` with `evmAddress = coa_B_address`. All checks pass and the function returns `isValid: true`, falsely asserting ownership of COA_B.

---

### Finding Description

**Signed surface vs. proof surface mismatch**

`COAOwnershipProof` (RLP-encoded) carries four fields:

```
KeyIndices | Address | CapabilityPath | Signatures
``` [1](#0-0) 

The signatures are verified exclusively over `signedData`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // ← only this is signed
    domainSeparationTag: "FLOW-V0.0-user"
)
``` [2](#0-1) 

`CapabilityPath` is decoded from the proof struct and used to borrow the COA resource, but it is **not** part of the signed message. The code itself acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [3](#0-2) 

**`evmAddress` is caller-supplied, not enforced**

In the arch precompile, `evmAddress` (called `caller`) is read from ABI-encoded calldata at offset 0 — it is **not** derived from `msg.sender` or any on-chain identity:

```go
caller, err := ReadAddress(input, index)
...
return types.NewCOAOwnershipProofInContext(hash, types.Address(caller), encodedProof)
``` [4](#0-3) 

The comment's claim that "the EVM-side precompile always passes the calling COA's address as the evmAddress argument" is a convention, not an enforcement. Any EVM contract can call the precompile with an arbitrary `evmAddress`.

**Validation logic after signature check**

After signatures pass, the function borrows the COA at the attacker-supplied `path` and checks `coaRef.address().bytes == evmAddress`:

```cadence
if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
    let coaAddressBytes = coaRef.address().bytes
    for index, item in coaAddressBytes {
        if item != evmAddress[index] { ... }
    }
    return ValidationResult(isValid: true, problem: nil)
}
``` [5](#0-4) 

If the attacker supplies `path = /public/coa2` and `evmAddress = coa2Address`, both checks pass trivially — the COA at that path does have that address. The only thing that was supposed to bind the proof to COA_A is the `CapabilityPath`, which is unsigned.

---

### Impact Explanation

An attacker who has observed a valid on-chain proof for COA_A (same Flow account, same `signedData`) can:

1. Copy `{KeyIndices, Address, Signatures}` verbatim.
2. Replace `CapabilityPath = 'coa1'` with `CapabilityPath = 'coa2'`.
3. Call `verifyCOAOwnershipProof(coa2EvmAddr, signedData, modifiedEncodedProof)` via the arch precompile.
4. Receive `isValid: true`.

Any EVM contract relying on this result for ERC-1271 signature validation or access control will grant the attacker the same permissions as if the victim had explicitly signed for COA_B. Bridged assets held in COA_B (ERC-20/ERC-721 tokens, FLOW) can be transferred without the victim's consent for that specific action.

---

### Likelihood Explanation

Preconditions required:
- Victim Flow account has **two COAs** published at different public paths (unusual today, but nothing in the protocol prevents it, and multi-COA setups are a natural evolution for power users and protocols).
- Attacker has observed a valid proof for COA_A with the target `signedData` (proofs are submitted in EVM transactions and are fully on-chain/observable).

The code comment explicitly acknowledges the attack class exists and defers mitigation to callers, meaning the protocol itself provides no guard.

---

### Recommendation

Bind the proof to a specific COA by including the `evmAddress` (or the `CapabilityPath`) inside `signedData` before signing. The simplest fix is to require callers to hash `evmAddress || applicationData` and sign that composite value, then verify the binding inside `validateCOAOwnershipProof`. Alternatively, include `CapabilityPath` in the signed payload at the protocol level so that swapping the path invalidates the signatures.

---

### Proof of Concept

```
1. Deploy emulator. Create Flow account V with two COAs:
   - COA1 stored at /storage/coa1, published at /public/coa1 → evmAddr1
   - COA2 stored at /storage/coa2, published at /public/coa2 → evmAddr2

2. V signs signedData = sha3("hello") with key[0].
   Build proof1 = COAOwnershipProof{KeyIndices:[0], Address:V, CapabilityPath:"coa1", Signatures:[sig]}
   Encode proof1 → encodedProof1.

3. Verify proof1 legitimately:
   verifyCOAOwnershipProof(evmAddr1, signedData, encodedProof1) → true  ✓

4. Attacker builds proof2 = COAOwnershipProof{KeyIndices:[0], Address:V, CapabilityPath:"coa2", Signatures:[sig]}
   (identical except CapabilityPath; sig is unchanged and still valid over signedData)
   Encode proof2 → encodedProof2.

5. Call verifyCOAOwnershipProof(evmAddr2, signedData, encodedProof2).
   - keyList.verify(signedData) → true  (sig unchanged)
   - borrow COA at /public/coa2 → succeeds
   - coaRef.address() == evmAddr2 → true
   → returns isValid: true  ✗ (should be false)

6. Assert that step 5 returns false → assertion fails, confirming the vulnerability.
```

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
```
