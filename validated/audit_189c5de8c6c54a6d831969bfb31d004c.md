### Title
Silent Truncation of Gas Refund When Native Resource Consumption Is Less Than EVM Gas Consumption — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary
In `compute_gas_refund`, the double-resource accounting adjustment (`delta_gas`) is only applied when it is positive (native cost exceeds EVM gas cost). When `delta_gas < 0` — meaning the user consumed more EVM gas than native resources warrant — the negative delta is silently discarded. The user's `gas_used` is not reduced, so the user is charged for more gas than their actual native resource consumption requires. The unrefunded amount is permanently lost to the user with no mechanism for recovery. A developer-acknowledged `// TODO: return delta_gas to gas_used?` comment marks the exact unresolved location.

### Finding Description
ZKsync OS implements a dual-resource model: EVM gas (ergs) and a native resource that models proving/pubdata cost. After execution, `compute_gas_refund` reconciles the two by computing:

```
delta_gas = (native_used / native_per_gas) - gas_used
```

Per `docs/double_resource_accounting.md`:

> If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. … Finally, any remaining gas left is refunded as usual.

The documentation describes only the positive case. The code handles only the positive case:

```rust
if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` — i.e., the user's EVM gas consumption exceeds what native resource consumption would imply — the negative delta is silently dropped. `gas_used` is left at the higher EVM-derived value. The user's refund is computed as:

```
refund = (gas_limit - gas_used) * gas_price
```

instead of the correct:

```
refund = (gas_limit - (gas_used + delta_gas)) * gas_price
       = (gas_limit - gas_used + |delta_gas|) * gas_price
```

The difference `|delta_gas| * gas_price` is silently transferred to the operator as an overpayment. There is no tracking of this shortfall and no way for the user to recover it. This is structurally identical to the Ajna pattern: an entitlement is silently capped, the remainder is not tracked, and the user permanently loses the difference.

### Impact Explanation
Every user whose transaction has high EVM gas consumption but low native resource consumption (computation-heavy, pubdata-light transactions) overpays fees by `|delta_gas| * gas_price` tokens. For a transaction with `gas_limit = 1,000,000`, `gas_price = 1,000`, and a `delta_gas` of `-50,000`, the user loses `50,000,000` base-token units per transaction. The loss is permanent and unrecoverable — the bootloader does not track the shortfall, and there is no claim mechanism.

### Likelihood Explanation
The condition `delta_gas < 0` is triggered whenever a transaction's EVM gas consumption exceeds `native_used / native_per_gas`. This is a normal operating condition for any computation-heavy, pubdata-light transaction (e.g., complex arithmetic, cryptographic operations, or loops that do not write storage). It requires no special privileges — any unprivileged L2 transaction sender can trigger it simply by submitting such a transaction. The `// TODO` comment confirms the developers are aware the case is unhandled.

### Recommendation
Apply the negative delta symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, subject to the `minimal_gas_used` floor:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native resource consumption is less than EVM gas consumption.
    // Reduce gas_used so the user receives the correct refund.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This ensures the user's refund accurately reflects their actual native resource consumption, consistent with the stated design intent in `docs/double_resource_accounting.md`.

### Proof of Concept
1. Submit an L2 transaction with high EVM gas consumption and low pubdata (e.g., a tight computation loop with no storage writes), with a non-zero `gas_price` and a `native_per_gas > 0`.
2. After execution, `native_used` is small (low pubdata), so `native_used / native_per_gas < gas_used_evm`, giving `delta_gas < 0`.
3. In `compute_gas_refund` at line 75, the `if delta_gas > 0` branch is not taken; `gas_used` is not reduced.
4. The user's refund is `(gas_limit - gas_used_evm) * gas_price` instead of `(gas_limit - gas_used_evm + |delta_gas|) * gas_price`.
5. The user permanently loses `|delta_gas| * gas_price` tokens, which are silently paid to the operator as an overpayment.
