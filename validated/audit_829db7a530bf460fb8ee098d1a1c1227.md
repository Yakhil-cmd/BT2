### Title
Missing `gas_per_pubdata_limit` Enforcement for L2 ZK Transactions Allows Unexpected Pubdata-Driven Reverts with Full Gas Loss - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

The ZK (ABI-encoded) L2 transaction format includes a `gas_per_pubdata_limit` field that users set to cap the pubdata price they are willing to accept. However, the L2 transaction validation path in `validate_and_compute_fee_for_transaction` never reads or enforces this field. The block's `pubdata_price` is consumed unconditionally. If the operator raises `pubdata_price` between a user's transaction submission and its inclusion, the transaction executes at the higher pubdata rate, exhausts native resources, is silently reverted, and the user forfeits the full gas fee — with no ability to have the transaction rejected at validation time.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the function `validate_and_compute_fee_for_transaction` computes the pubdata cost rate exclusively from the block-level oracle value:

```rust
// line 106-107
let pubdata_price = system.get_pubdata_price();
let native_price  = system.get_native_price();
// ...
// line 142-143
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

The `gas_per_pubdata_limit` field that the user signed into the transaction is accessible via `Transaction::gas_per_pubdata_limit()`:

```rust
// basic_bootloader/src/bootloader/transaction/mod.rs line 150-155
pub fn gas_per_pubdata_limit(&self) -> U256 {
    match self {
        Self::Rlp(_) => U256::ZERO,
        Self::Abi(tx) => U256::from(tx.gas_per_pubdata_limit.read()),
    }
}
```

This field is **only consumed** in the L1 transaction path (`process_l1_transaction.rs` line 80):

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
```

It is **never read** inside `validate_and_compute_fee_for_transaction` for L2 ZK transactions. No comparison of `pubdata_price` against `gas_per_pubdata_limit` is performed, and no `PubdataPriceTooHigh`-style rejection is issued when the block's pubdata price exceeds the user's stated limit.

When the block's `pubdata_price` is higher than the user's `gas_per_pubdata_limit`, the transaction proceeds with the elevated `native_per_pubdata`. If the resulting pubdata charge exhausts the user's native budget, the post-execution check in `mod.rs` forces a revert and burns all remaining ergs:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs line 886-890
if !has_enough {
    execution_result = execution_result.to_reverted();
    // Burn all remaining ergs.
    context.resources.main_resources.exhaust_ergs();
```

The user loses the full gas fee with no recourse.

---

### Impact Explanation

**Impact: Medium**

A user who submits a ZK L2 transaction with `gas_per_pubdata_limit = X` expecting it to act as a slippage guard receives no protection. If the operator-controlled `pubdata_price` rises above `X` before the transaction is included:

1. The transaction is validated and pre-charged at the higher pubdata rate.
2. Execution proceeds, generates pubdata, and the post-execution native check fails.
3. The transaction is reverted and `gas_used` is set to the full `gas_limit`.
4. The user pays the maximum possible fee for a transaction that produced no state change.

This is a direct financial loss for the user, analogous to a DeFi user receiving far fewer assets than previewed due to a fee change between submission and execution.

---

### Likelihood Explanation

**Likelihood: Medium**

`pubdata_price` and `native_price` are operator-set block-level parameters read from the oracle (`BlockMetadataFromOracle`). They can change every block. A user who estimates gas at one pubdata price and whose transaction is delayed by even one block faces this risk. The `gas_per_pubdata_limit` field exists precisely to guard against this, but its non-enforcement makes the guard illusory for L2 transactions.

---

### Recommendation

In `validate_and_compute_fee_for_transaction`, after computing `native_per_pubdata`, derive the effective gas-per-pubdata from the block parameters and compare it against the user's `gas_per_pubdata_limit`. If the block's effective pubdata price exceeds the user's limit, reject the transaction with `InvalidTransaction::PubdataPriceTooHigh` (the error variant already exists). This mirrors the enforcement already present for L1 transactions and the `BaseFeeGreaterThanMaxFee` check for EIP-1559 gas.

---

### Proof of Concept

1. User signs a ZK L2 transaction with `gas_per_pubdata_limit = 100` (gas per pubdata byte) and `gas_limit = 200_000`.
2. At submission time, `pubdata_price / native_price = 80` (within the user's limit).
3. Before inclusion, the operator raises `pubdata_price` so that `pubdata_price / native_price = 500`.
4. `validate_and_compute_fee_for_transaction` reads `pubdata_price` from the block oracle (line 106), computes `native_per_pubdata = 500`, and never reads `gas_per_pubdata_limit`.
5. The transaction passes validation and is pre-charged.
6. Execution generates 300 bytes of pubdata → native cost = `300 × 500 = 150_000` native units, exceeding the user's native budget.
7. `check_enough_resources_for_pubdata` returns `false`; the transaction is reverted and all ergs are burned.
8. The user pays `200_000 × gas_price` in fees for a no-op transaction, despite having set `gas_per_pubdata_limit = 100` to prevent exactly this outcome. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
