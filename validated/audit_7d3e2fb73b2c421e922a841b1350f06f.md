Audit Report

## Title
Unbounded `ingress_messages` deserialization in `TryFrom<pb::StrippedBlockProposal>` enables Byzantine peer to spawn unbounded Tokio tasks and exhaust replica resources — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

## Summary
`TryFrom<pb::StrippedBlockProposal>` performs no upper-bound check on the `ingress_messages` repeated field before collecting it into a `Vec`. A Byzantine peer below the consensus fault threshold can serve a crafted `StrippedBlockProposal` with an arbitrarily large `ingress_messages` list under a valid, currently-wanted block proposal hash. This causes `BlockProposalAssembler::new` to allocate a proportionally large `Vec`, and `assemble_message` to spawn one indefinitely-looping Tokio task per entry, exhausting replica memory and saturating the task scheduler until the bouncer fires `Unwanted`.

## Finding Description

**Root cause — no count bound in deserialization.**
`TryFrom<pb::StrippedBlockProposal>` validates that `pruned_block_proposal` is present, that the pruned block's ingress payload is empty, and that `unstripped_consensus_message_id` is a `BlockProposal` hash. It does not check `ingress_messages.len()`: [1](#0-0) 

**Proportional allocation in `BlockProposalAssembler::new`.**
Every `SignedIngressId` in the stripped payload is mapped into a `Vec<(SignedIngressId, Option<SignedIngress>)>` with no cap. The developer comment at line 440 ("We can have at most 1000 elements") is an unenforced assumption, not a guard: [2](#0-1) 

**One Tokio task spawned per entry.**
`assemble_message` calls `assembler.missing_stripped_messages()` (which returns all N entries since none are in the local pool) and spawns a `get_or_fetch` task for each: [3](#0-2) 

**Each task loops indefinitely.**
`download_stripped_message` uses `with_max_elapsed_time(None)`, so each of the N tasks retries with exponential backoff forever until the `JoinSet` is dropped: [4](#0-3) 

**ID check does not constrain content.**
The `id()` implementation for `MaybeStrippedConsensusMessage::StrippedBlockProposal` is derived solely from `unstripped_consensus_message_id`. A Byzantine peer can set this field to a valid, currently-wanted block proposal hash while inflating `ingress_messages` to an arbitrary count — the ID check in `FetchArtifact::download_artifact` passes: [5](#0-4) 

**Bouncer checks only the artifact ID.**
`BouncerFactoryWrapper` delegates to the consensus pool bouncer which evaluates only the `ConsensusMessageId` (height + hash). It has no visibility into the number of ingress message IDs embedded in the stripped payload: [6](#0-5) 

## Impact Explanation
A single Byzantine peer can trigger allocation of a `Vec` with tens of thousands of entries (bounded only by the QUIC message size limit, not by any protocol-level ingress count limit) and spawn an equal number of Tokio tasks per targeted block proposal. Each task holds a stack frame and loops indefinitely. With multiple concurrent block proposals in flight (the consensus bouncer allows up to `LOOK_AHEAD` heights), the total task count multiplies. This causes application/platform-level DoS — consensus blocking and subnet availability impact — matching the **High ($2,000–$10,000)** bounty tier: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation
The attacker needs only to be a subnet peer (below the Byzantine fault threshold) and to observe a block proposal ID that the victim is currently downloading — both are trivially achievable in normal subnet operation. No privileged access, key material, or majority corruption is required. The crafted message passes all existing validation gates (ID check, bouncer check) before the resource allocation occurs. The attack is repeatable for every new block proposal the victim attempts to download.

## Recommendation
Add an upper-bound check on `ingress_messages.len()` (and `stripped_idkg_dealings.len()`) inside `TryFrom<pb::StrippedBlockProposal>`, before the `collect()` call, returning `ProxyDecodeError` if either count exceeds the protocol-enforced maximum (`max_ingress_messages_per_block` from the registry, default 1000, for ingress; `dkg_dealings_per_block` for IDKG dealings). This enforces the invariant already assumed by the comment at assembler.rs line 440.

## Proof of Concept
```rust
// Craft a StrippedBlockProposal proto with N fake ingress_messages entries.
let mut proto = pb::StrippedBlockProposal {
    pruned_block_proposal: Some(pb::BlockProposal::default()),
    unstripped_consensus_message_id: Some(/* valid wanted block proposal ID */),
    ingress_messages: (0..100_000)
        .map(|i| pb::StrippedIngressMessage {
            stripped: Some(pb::IngressMessageId {
                expiry: 0,
                message_id: vec![i as u8; 32],
            }),
            ingress_bytes_hash: vec![0u8; 32],
        })
        .collect(),
    stripped_idkg_dealings: vec![],
};
// Serve this proto in response to the victim's RPC for the wanted block proposal ID.
// TryFrom succeeds (no count check), BlockProposalAssembler::new allocates 100,000-entry Vec,
// assemble_message spawns 100,000 get_or_fetch tasks each looping with with_max_elapsed_time(None).
// Assert: memory usage and spawned task count are unbounded before the bouncer fires Unwanted.
```
A deterministic integration test can verify this by constructing the proto directly, calling `StrippedBlockProposal::try_from`, then `BlockProposalAssembler::new`, and asserting that `missing_stripped_messages()` returns 100,000 entries — confirming the unbounded allocation path without requiring a live subnet.
