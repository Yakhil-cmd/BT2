### Title
Missing Zero-Address Validation in `change_reward_address` Enables Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
Both `change_reward_address` functions in `staking.cairo` and `pool.cairo` accept any `ContractAddress` as the new reward destination but only guard against the token contract address. Neither function checks that the supplied `reward_address` is not the zero address (`0x0`). A staker or pool member who sets their reward address to zero will have all future reward transfers directed to the zero address, permanently freezing their unclaimed yield.

### Finding Description
In `staking.cairo`, `change_reward_address` validates only that the new address is not a registered token:

```cairo
// src/staking/staking.cairo  lines 517-530
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No zero-address check
    staker_info.reward_address = reward_address;
    ...
}
```

In `pool.cairo`, `change_reward_address` performs an equivalent single-token guard:

```cairo
// src/pool/pool.cairo  lines 505-516
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No zero-address check
    pool_member_info.reward_address = reward_address;
    ...
}
```

Once the zero address is stored, every subsequent reward disbursement targets it:

- **Staker path** — `send_rewards_to_staker` (staking.cairo line 1625): `token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into())` — transfers to `0x0`.
- **Pool member path** — `claim_rewards` (pool.cairo line 366): `reward_token.checked_transfer(recipient: reward_address, amount: rewards.into())` — transfers to `0x0`.

Depending on the ERC-20 implementation, the transfer either silently burns the tokens (permanent loss) or reverts on every call (permanent DoS on `claim_rewards`). In either case the yield is permanently frozen.

### Impact Explanation
Any staker or pool member who sets their reward address to zero loses the ability to ever receive their accrued rewards. All STRK rewards that accumulate from that point forward are either burned or cause every `claim_rewards` call to revert, permanently freezing unclaimed yield. This maps directly to the allowed High impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The entry point is fully unprivileged — any registered staker calls `staking.cairo::change_reward_address` and any pool member calls `pool.cairo::change_reward_address`. No special role or key is required. The scenario is realistic via scripting error, front-end bug, or a malicious actor who controls the staker/pool-member key and wishes to grief themselves or sabotage a shared-custody arrangement. Because the call is irreversible (there is no recovery path once zero is stored and rewards begin accumulating), even a single accidental invocation causes permanent harm.

### Recommendation
Add an explicit zero-address guard in both functions, mirroring the pattern used in the external report:

```cairo
// staking.cairo  change_reward_address
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");

// pool.cairo  change_reward_address
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

This is consistent with the existing `REWARD_ADDRESS_IS_TOKEN` guard and closes the analogous gap identified in the external report.

### Proof of Concept

1. A staker is registered and has accumulated unclaimed rewards.
2. The staker calls `staking.cairo::change_reward_address(reward_address: 0x0)`.
   - The only check (`does_token_exist`) passes because `0x0` is not a registered token.
   - `staker_info.reward_address` is written as `0x0`.
3. The staker (or anyone) calls `claim_rewards(staker_address)`.
   - `send_rewards_to_staker` executes `checked_transfer(recipient: 0x0, amount: ...)`.
   - Tokens are either burned or the call reverts — the staker can never recover their yield.
4. The identical sequence applies to a pool member via `pool.cairo::change_reward_address` followed by `pool.cairo::claim_rewards`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
