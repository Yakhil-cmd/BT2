### Title
COA Ownership Proof Does Not Bind `signedData` to `evmAddress`, Enabling Cross-COA Replay - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's keys signed `signedData`, and separately checks that the COA at the given capability path matches `evmAddress`. However, it never enforces that `signedData` encodes `evmAddress`. A signature produced to prove ownership of COA_A is therefore cryptographically valid when replayed against COA_B, as long as both COAs are owned by the same Cadence account. This allows an unprivileged attacker to forge ERC-1271 (`isValidSignature`) approvals for any COA whose Cadence owner has ever published a valid proof for a different COA.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verify that the supplied signatures are valid over `signedData` using the Cadence account's keys.
2. Verify that the COA resource stored at `CapabilityPath` has address `evmAddress`. [1](#0-0) 

These two checks are never joined: `signedData` is never required to commit to `evmAddress`. The code itself documents this gap: [2](#0-1) 

The `COAOwnershipProof` struct that is RLP-encoded and submitted by the caller contains `CapabilityPath` as a mutable, caller-controlled field: [3](#0-2) 

`CapabilityPath` is **not** covered by the signatures (signatures are over `signedData` only). An attacker who observes a valid proof `{Address=A, CapabilityPath=/public/coa1, Signatures=sig(H)}` for COA_A can construct `{Address=A, CapabilityPath=/public/coa2, Signatures=sig(H)}` and submit it to prove ownership of COA_B, because:

- `sig(H)` is still valid (same Cadence account keys, same hash)
- The COA at `/public/coa2` is COA_B, so the address check passes

The EVM entry point is the COA contract's `isValidSignature`, which passes `address(this)` as `evmAddress` and the caller-supplied `_sig` bytes directly to the precompile: [4](#0-3) 

`_sig` is the RLP-encoded `COAOwnershipProof`, which the attacker controls entirely (they only need to reuse the signatures from a previously observed proof and change `CapabilityPath`).

The precompile decodes the proof and passes it to `validateCOAOwnershipProof`: [5](#0-4) [6](#0-5) 

### Impact Explanation

An attacker can forge a valid ERC-1271 `isValidSignature` response for any COA whose Cadence owner has ever published a COA ownership proof for any other COA under the same account. Any on-chain protocol that relies on `isValidSignature` for authorization (token approvals, governance votes, permit-style flows, multisig membership) will accept the forged proof, granting the attacker the ability to act as the COA owner without possessing the Cadence account's private keys.

### Likelihood Explanation

The precondition is that the victim Cadence account holds at least two COAs at different public capability paths, and that a valid proof for one of them has been submitted on-chain (making the signatures observable). Both conditions are achievable without any privileged access: any Cadence account can create multiple COAs, and proofs are submitted in ordinary EVM transactions. The attack requires no admin access, no key compromise, and no staked-node control.

### Recommendation

Enforce that `signedData` encodes `evmAddress` inside `validateCOAOwnershipProof`, or add a protocol-level check that rejects any proof where `signedData` does not include the bytes of `evmAddress`. The EVM-side precompile already has `evmAddress` available as the first ABI argument; it should hash `evmAddress` into the data that is verified against the signatures before delegating to the Cadence function.

### Proof of Concept

**Setup:** Alice's Cadence account `0xALICE` has two COAs:
- COA_A at EVM address `0xCOA_A`, capability published at `/public/coa1`
- COA_B at EVM address `0xCOA_B`, capability published at `/public/coa2`

**Step 1 – Observe a valid proof for COA_A on-chain:**
```
proof_A = RLP{
  KeyIndices:     [0],
  Address:        0xALICE,
  CapabilityPath: "coa1",
  Signatures:     [sig_over_H],   // Alice signed hash H
}
// verifyCOAOwnershipProof(0xCOA_A, H, proof_A) → true
```

**Step 2 – Attacker constructs a replay proof for COA_B:**
```
proof_B = RLP{
  KeyIndices:     [0],
  Address:        0xALICE,
  CapabilityPath: "coa2",         // changed; not covered by signatures
  Signatures:     [sig_over_H],   // reused unchanged
}
```

**Step 3 – Attacker calls `isValidSignature` on COA_B:**
```solidity
COA_B.isValidSignature(H, proof_B)
// internally calls: verifyCOAOwnershipProof(address(COA_B), H, proof_B)
```

**Step 4 – `validateCOAOwnershipProof` evaluation:**
- `keyList.verify(signatureSet, signedData=H, ...)` → **true** (Alice's key signed H)
- `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coa2).address()` → `0xCOA_B` == `evmAddress` → **true**
- Returns `ValidationResult(isValid: true)`

**Result:** `isValidSignature` returns `0x1626ba7e` (valid), granting the attacker ERC-1271 authorization as the owner of COA_B without Alice's consent. [7](#0-6)

		return nil, err
