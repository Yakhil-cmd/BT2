### Title
Period-Boundary Double-Mint via `block.timestamp`-Epoch Reset in `PeriodMintLimit` — (`L1/starkware/solidity/stake/PeriodMintLimit.sol`)

---

### Summary

`PeriodMintLimit.checkAndUpdatePeriodicalQuota` uses `block.timestamp / MINTING_PERIOD_DURATION` as a discrete period index. Because the index resets at exact multiples of 604800 seconds, two calls to the permissionless `RewardSupplier.tick()` that straddle a period boundary each see a fresh, zero-initialized accounting slot and each pass the `PERIOD_MINT_CAP` check independently. With ≥10 pending L2→L1 mint-request messages, an unprivileged caller can mint 2 × 6,500,000e18 STRK within seconds, doubling the intended weekly inflation cap.

---

### Finding Description

**Root cause — `PeriodMintLimit.sol` lines 33–36:**

```solidity
function periodAccountingSlot(address token) internal view returns (bytes32) {
    uint256 period_index = block.timestamp / MINTING_PERIOD_DURATION;   // integer division
    return keccak256(abi.encode(MINTING_PERIOD_DURATION, token, period_index));
}
``` [1](#0-0) 

The slot is keyed on `period_index = block.timestamp / 604800`. At timestamp `T = k * 604800 - 1` the index is `k-1`; at `T = k * 604800` it is `k`. These are two entirely independent storage slots, each starting at zero.

**`checkAndUpdatePeriodicalQuota` only checks the current slot:**

```solidity
function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
    bytes32 periodSlot = periodAccountingSlot(token);
    uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
    uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
    require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
    periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;
}
``` [2](#0-1) 

There is no look-back into the previous slot, no sliding-window accumulator, and no cross-slot carry.

**`tick()` is fully permissionless:**

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    ...
    mintManager().mintRequest(token(), amountToMint);
    ...
}
``` [3](#0-2) 

No `onlyOwner`, no role check, no caller restriction.

**Per-tick ceiling equals exactly `PERIOD_MINT_CAP`:**

```solidity
uint256 constant TOKENS_PER_MINT_REQUEST = 1_300_000 * 10**18;
uint256 constant MAX_MESSAGES_TO_PROCESS_PER_TICK = 5;
// 5 × 1_300_000e18 = 6_500_000e18 == PERIOD_MINT_CAP
``` [4](#0-3) [5](#0-4) 

A single `tick()` call can consume exactly the full weekly cap in one shot.

---

### Impact Explanation

Two `tick()` calls separated by a single block timestamp crossing a period boundary each pass `checkAndUpdatePeriodicalQuota` independently, minting 13,000,000e18 STRK in seconds instead of the intended 6,500,000e18 per week. The extra 6,500,000e18 tokens are bridged to the L2 reward supplier and distributed as staking rewards, inflating supply 2× beyond the protocol's safety cap. This directly reduces the real value of all outstanding unclaimed yield held by stakers and token holders — a High-severity impact under "Theft of unclaimed yield."

---

### Likelihood Explanation

- `tick()` requires no privilege — any EOA can call it.
- Period boundaries occur every 7 days; no validator collusion is needed. The attacker simply monitors the chain and submits two transactions around the natural boundary.
- The only precondition is ≥10 unconsumed L2→L1 mint-request messages. These accumulate naturally whenever the L2 reward supplier has requested funds that have not yet been processed on L1 (e.g., during any multi-day gap in `tick()` calls, which is expected in normal operation).
- The exploit is deterministic and local-testable with a fork test.

---

### Recommendation

Replace the fixed-epoch slot with a **sliding-window accumulator** that tracks the total minted in the last `MINTING_PERIOD_DURATION` seconds, or carry forward the remainder from the previous slot into the new one:

```solidity
// Option A: carry-forward
uint256 prev_index = period_index - 1;
bytes32 prevSlot = keccak256(abi.encode(MINTING_PERIOD_DURATION, token, prev_index));
uint256 prevMinted = periodMintAccounting()[prevSlot];
// weight by how far into the new period we are
uint256 elapsed = block.timestamp % MINTING_PERIOD_DURATION;
uint256 carry = prevMinted * (MINTING_PERIOD_DURATION - elapsed) / MINTING_PERIOD_DURATION;
require(carry + mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
```

Alternatively, enforce a **global monotonic mint counter** with a per-period high-water mark that cannot be reset by a boundary crossing.

---

### Proof of Concept

```solidity
// Precondition: ≥10 L2→L1 mint-request messages pending in messagingContract.
// Step 1 — call tick() in a block where block.timestamp = k * 604800 - 1
//   period_index = k-1, slot[k-1] goes from 0 → 6_500_000e18 ✓ (passes cap check)
rewardSupplier.tick{value: fee}();

// Step 2 — call tick() in the very next block where block.timestamp = k * 604800
//   period_index = k,   slot[k]   goes from 0 → 6_500_000e18 ✓ (passes cap check)
rewardSupplier.tick{value: fee}();

// Result: 13_000_000e18 STRK minted in ~12 seconds.
// Invariant violated: no more than PERIOD_MINT_CAP should be mintable
// in any 604800-second window.
```

An invariant fuzz test asserting `sum(minted in any 604800-second sliding window) ≤ PERIOD_MINT_CAP` would catch this immediately.
