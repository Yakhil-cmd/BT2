After thorough analysis of the `hirayap/threshold-signatures--018` codebase, I mapped the external report's vulnerability class — **missing initialization/validation of a required prerequisite state causing protocol failure** — to the CKD subsystem.

The external report's pattern: a required state (cardinality) is not initialized, so every call to the oracle reverts. The analog here: a required constraint (threshold) is never validated in `ckd()`, so the protocol silently produces a cryptographically wrong output when called with a sub-threshold participant set.

---

### Title
Missing Threshold Validation in `ckd()` Allows Corrupted CKD Output with Sub-Threshold Participant Sets — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` function does not accept or validate a threshold parameter. Unlike every other protocol entry point in the library (`sign`, `presign`, `keygen`, `reshare`), which all enforce `participants.len() >= threshold`, `ckd()` accepts any participant set with ≥ 2 members. When called with fewer participants than the DKG threshold, the Lagrange interpolation inside `compute_signature_share` silently produces an incorrect scalar, and the coordinator aggregates a cryptographically wrong derived key with no error returned.

### Finding Description

Every other protocol entry point in the library validates the threshold against the participant count. For example, OT-based ECDSA `sign()`: [1](#0-0) 

And `keygen()` via `assert_key_invariants`: [2](#0-1) 

The `ckd()` entry point performs only four checks — minimum 2 participants, no duplicates, `me` present, `coordinator` present — and has **no threshold parameter at all**: [3](#0-2) 

Inside `compute_signature_share`, the Lagrange coefficient is computed directly from the supplied `participants` list: [4](#0-3) 

The Lagrange coefficient `λ_i` is defined over the supplied participant set. For the CKD output to equal `msk · H(pk ‖ app_id)`, the participant set must be the same set (or a valid superset) used during DKG with at least `threshold` members, so that `Σ λ_i · x_i = msk`. If the participant set is smaller than the threshold, `Σ λ_i · x_i ≠ msk`, and the coordinator aggregates a wrong value with no error: [5](#0-4) 

### Impact Explanation

The coordinator receives a `CKDOutput` whose `big_c` field encodes `msk' · H(pk ‖ app_id) + blinding` where `msk' ≠ msk`. The derived confidential key is cryptographically incorrect and unusable. Any downstream consumer that calls `ckd_output.unmask(app_sk)` obtains a wrong key. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

A malicious coordinator controls which `participants` slice is advertised to honest parties for a given CKD session. By advertising a participant list smaller than the DKG threshold (e.g., omitting one or more old participants), the coordinator causes all honest parties to call `ckd()` with the truncated list. The protocol completes without error on every node, but the coordinator's aggregated output is wrong. No cryptographic break or leaked key is required — only the ability to specify the participant list, which is the coordinator's normal role.

### Recommendation

Add a `threshold: impl Into<ReconstructionLowerBound>` parameter to `ckd()` and enforce:

```rust
if participants.len() < threshold.value() {
    return Err(InitializationError::NotEnoughParticipantsForThreshold {
        threshold: threshold.value(),
        participants: participants.len(),
    });
}
```

This mirrors the pattern already used in `sign()`, `presign()`, `keygen()`, and `reshare()`. [6](#0-5) 

### Proof of Concept

1. Run DKG with participants `[P1, P2, P3]` and `threshold = 3`. Each party obtains a `KeygenOutput` with `public_key = pk`.
2. Call `ckd(&[P1, P2], coordinator, me, key_pair, app_id, app_pk, rng)` — only 2 of the 3 required participants.
3. `ckd()` returns `Ok(protocol)` — no initialization error.
4. Run the protocol to completion. The coordinator aggregates `λ_1·x_1 + λ_2·x_2` (Lagrange over `{P1,P2}`) instead of `λ_1·x_1 + λ_2·x_2 + λ_3·x_3` (Lagrange over `{P1,P2,P3}`). Since `threshold = 3`, the two-party Lagrange sum does not equal `msk`.
5. `ckd_output.unmask(app_sk)` returns a key that is not `msk · H(pk ‖ app_id)` — the derived key is wrong and unusable.

    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
