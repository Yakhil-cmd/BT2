### Title
Unserialized `session_request` Processing Allows Concurrent Confirmation Dialogs and Double Transaction Execution — (`packages/gui/src/components/walletConnect/WalletConnectProvider.tsx`)

### Summary
`processSessionRequest` is registered as a plain async event listener with no mutex, queue, or in-flight guard. A malicious dApp can send two identical `chia_sendTransaction` `session_request` events in rapid succession, causing two independent async invocations to race through the full pipeline concurrently, each opening its own confirmation dialog. If the user approves both (e.g., dialogs appear stacked or the user clicks through quickly), two transactions are dispatched, resulting in double asset loss.

### Finding Description

The `session_request` listener is registered in `WalletConnectProvider` with no concurrency control:

```
client.on('session_request', handleSessionRequest);   // line 418
```

`handleSessionRequest` immediately `await`s `processSessionRequest(event)`, which is itself a plain `async` function with no in-flight guard. [1](#0-0) 

When two `session_request` events arrive within the same JS turn, the first `await` inside `processSessionRequest` (the `window.permissionsAPI.findPair` IPC call at line 335) yields the event loop, allowing the second event to enter `processSessionRequest` concurrently. Both invocations then independently traverse the full pipeline: [2](#0-1) 

Both reach `currentProcess(pairTopic, method, commandParams, ...)` (line 366), which calls `handleProcess` in `useWalletConnectCommand`, which calls `window.permissionsAPI.dispatchAsPair(...)` (IPC to main). [3](#0-2) 

In the main process, `ipcMain.handle` does **not** serialize concurrent invocations of the same channel — each IPC call gets its own independent handler execution: [4](#0-3) 

Both invocations reach `dispatchPairRequest`, which has no mutex or in-flight guard: [5](#0-4) 

Both then reach the `confirm` callback in the `DISPATCH_AS_PAIR` handler, which calls `openReactDialog(mainWindow, Confirm, ...)` — creating a **new BrowserWindow dialog for each invocation**: [6](#0-5) 

There is no deduplication, no pending-request set, and no concurrency limiter anywhere in the chain. The `limit.ts` and `throttleAsync.ts` utilities exist in the codebase but are not applied to the `session_request` pipeline.

### Impact Explanation

If the user approves both dialogs (which can happen when dialogs appear stacked, or when the user clicks through quickly), two `chia_sendTransaction` calls are dispatched to the daemon. The user intended to authorize one transaction but two are executed — direct double asset loss in XCH or CAT tokens.

### Likelihood Explanation

Any connected dApp (already past the pairing/approval gate) can trigger this by sending two identical `session_request` messages within ~100 ms. No additional privileges are required. The WalletConnect SDK does not deduplicate or serialize `session_request` events on the client side. The attack is repeatable and locally testable.

### Recommendation

Serialize `session_request` processing with a per-topic (or global) async queue/mutex. The simplest fix is to wrap `handleSessionRequest` with the existing `limit(1)` utility applied per pairing topic, so a second request for the same topic cannot enter the confirmation flow until the first resolves. Alternatively, maintain a `Set` of in-flight request IDs and reject duplicates before they reach `dispatchAsPair`.

### Proof of Concept

1. Pair a dApp with `chia_sendTransaction` granted.
2. From the dApp, send two identical `chia_sendTransaction` `session_request` messages to the same session topic within 100 ms.
3. Observe: two separate confirmation dialogs appear in the Chia GUI.
4. Approve both dialogs.
5. Assert: two `send_transaction` RPC calls are dispatched to the wallet daemon, resulting in two on-chain transactions.

Expected (correct) behavior: only one confirmation dialog appears; the second request is queued or rejected until the first is resolved.

export async function dispatchPairRequest<T>(
  topic: string,
  command: string,
  params: Record<string, unknown>,
  process: (context: DispatchPairRequestContext) => Promise<T>,
  confirm: () => Promise<boolean>,
): Promise<T> {
  const [loggedInFingerprint, isMainnetValue] = await Promise.all([getLoggedInFingerprint(), isMainnet()]);

  const pair = findPair(topic);
  if (!pair) {
    throw new WcError(`Pair not found`, WcErrorCode.USER_REJECTED);
  }

  // verify if the command is allowed for this pair
  if (!pair.commands.includes(command)) {
    throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const { fingerprint } = params;

  // verify if the network is the same as the pair's network
  if (isMainnetValue !== pair.mainnet) {
    throw new WcError(`Network mismatch`, WcErrorCode.UNSUPPORTED_CHAINS);
  }

  // verify if the requested fingerprint is allowed for this pair
  const requestedFingerprint = fingerprint ?? loggedInFingerprint;
  if (typeof requestedFingerprint !== 'number' || !requestedFingerprint || requestedFingerprint !== pair.fingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const context = {
    pair,
    fingerprint: requestedFingerprint,
  };

  // Dapps may not switch the active key for an existing pair.
  if (fingerprint !== undefined && fingerprint !== loggedInFingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  // if command is bypassed return true
  if (pair.bypass.includes(command)) {
    return process(context);
  }

  const isAllowed = await confirm();
  if (isAllowed === true) {
    return process(context);
  }

  throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
