### Title
Missing Participant-Set Validation Against Triple's Recorded Participants Allows Malicious Coordinator to Permanently Deny Presigning - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in the OT-based ECDSA module accepts Beaver triples without verifying that the triple's recorded participant set (`TriplePub.participants`) is consistent with the presigning participant set. The `TriplePub` struct explicitly stores the participants who generated the triple, but this field is never read or validated during presigning. A malicious coordinator can supply triples generated by a mismatched participant set, causing the Lagrange linearization to be computed with wrong coefficients, which deterministically fails the protocol's own consistency checks and permanently denies signing for all honest parties.

---

### Finding Description

**Root cause — explicit omission of participant-set validation:**

`src/ecdsa/ot_based_ecdsa/presign.rs`, lines 38–40, contains a deliberate comment acknowledging the missing check:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [1](#0-0) 

The `TriplePub` struct records the exact participant set that generated the triple:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    pub threshold: ReconstructionLowerBound,
}
``` [2](#0-1) 

The `presign()` initialization function only validates that the threshold values match, but never compares `args.triple0.1.participants` or `args.triple1.1.participants` against the presigning `participants` argument:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [3](#0-2) 

**How the mismatch corrupts the protocol:**

Inside `do_presign`, the Lagrange coefficient `lambda_me` is computed from the **presigning** participant set:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
``` [4](#0-3) 

But `k_i` and `e_i` are shares of the triple polynomial evaluated at `me`'s identifier, which was generated under the **triple generation** participant set. If the two sets differ, the Lagrange coefficients are wrong for the shares in hand. The protocol then immediately hits a hard consistency check:

```rust
if big_e != (ProjectivePoint::GENERATOR * e).to_affine() {
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of kd".to_string(),
    ));
}
``` [5](#0-4) 

Because the sum of wrongly-linearized `e_i` shares no longer equals the public commitment `big_e` from the triple, this check fails deterministically, aborting the presign protocol for every honest participant.

**Attacker-controlled entry path:**

The coordinator is the party that distributes `TriplePub` values to presigning participants. Since `TriplePub` is a plain public struct with no authentication binding it to a specific presigning session, a malicious coordinator can trivially supply a `TriplePub` from a different triple generation session (different `participants` field) to any or all presigning participants. The `presign()` API accepts this without complaint.

The orchestration documentation explicitly requires `P_triples ⊇ P_presign` as a protocol invariant:

```
P ⊇ P0 ⊇ P1 ⊇ P2
``` [6](#0-5) 

This invariant is documented but not enforced in code.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

A malicious coordinator can supply triples whose `TriplePub.participants` field does not match the presigning participant set. Because `do_presign` computes Lagrange coefficients from the presigning set but applies them to shares generated under a different set, the `E =?= e*G` consistency check (and the `alpha*G =?= K+A` check at line 162) will fail deterministically. Every presign attempt using coordinator-supplied mismatched triples will abort with `ProtocolError::AssertionFailed`. Since the coordinator controls which `TriplePub` is handed to each participant, this denial is permanent and repeatable across all signing sessions. [7](#0-6) 

---

### Likelihood Explanation

The coordinator role requires no privileged key material — it is simply a designated participant. The attack requires only that the coordinator pass a `TriplePub` from a different generation session, which is trivially achievable since `TriplePub` is a serializable public struct. No cryptographic break is needed. Any deployment where the coordinator is not fully trusted (e.g., a rotating or semi-trusted MPC node) is directly exposed.

---

### Recommendation

In `presign()`, after constructing the `ParticipantList`, add an explicit check that the presigning participant set is a subset of each triple's recorded participant set:

```rust
// Validate that the presigning participants are a subset of the triple's generation participants
let triple0_participants = ParticipantList::new(&args.triple0.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants = ParticipantList::new(&args.triple1.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;

for p in participants.participants() {
    if !triple0_participants.contains(*p) || !triple1_participants.contains(*p) {
        return Err(InitializationError::BadParameters(
            "presigning participants must be a subset of the triple generation participants".to_string()
        ));
    }
}
```

This enforces the documented invariant `P_triples ⊇ P_presign` at the API boundary, preventing a malicious coordinator from supplying triples generated by an incompatible participant set.

### Proof of Concept

**Setup:** Three participants `{P1, P2, P3}` generate triple `T_A` with threshold 2. A separate set `{P4, P5, P6}` generates triple `T_B` with threshold 2.

**Attack:**
1. Malicious coordinator initiates presigning with participants `{P1, P2}` and threshold 2.
2. Coordinator supplies `T_B`'s `TriplePub` (from `{P4, P5, P6}`) to P1 and P2, paired with P1/P2's own `TripleShare` values from `T_A`.
3. `presign()` passes initialization: participant count ≥ 2, threshold matches `T_B.threshold` (both 2), no duplicate participants, `me` is present.
4. Inside `do_presign`, P1 computes `lambda_P1` using Lagrange interpolation over `{P1, P2}`. P1's share `e_i` was generated for `{P1, P2, P3}` — the Lagrange basis is wrong.
5. The sum `e = Σ lambda_j * e_j` over `{P1, P2}` does not equal the scalar whose commitment is `big_e` from `T_B`.
6. Check `big_e != (G * e).to_affine()` triggers `ProtocolError::AssertionFailed("received incorrect shares of kd")`.
7. Presigning aborts. The coordinator repeats this for every signing attempt, permanently denying signing. [8](#0-7) [9](#0-8)
