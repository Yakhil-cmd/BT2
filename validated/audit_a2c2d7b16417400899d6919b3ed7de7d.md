### Title
Missing Sequencer-Only Guard on `update_rewards` Allows Any Caller to Permanently Freeze Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `staking.cairo` is specified to be callable only by the Starkware sequencer, but no access-control check enforces this. Any unprivileged address can call it with `disable_rewards: true`, which writes `last_reward_block` to the current block **before** the `disable_rewards` guard is evaluated. The sequencer is then permanently blocked from distributing rewards for that block, causing permanent freezing of unclaimed yield.

### Finding Description
`IStakingRewardsManager::update_rewards` is the sole mechanism for distributing per-block consensus rewards to stakers and their delegation pools. The spec explicitly states its access control is "Only starkware sequencer" (docs/spec.md line 1645). The implementation, however, only calls `general_prerequisites()` (which checks the pause flag) and a block-deduplication guard:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks paused
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.  <-- written unconditionally
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;                    // returns WITHOUT distributing rewards
    }
    ...
```

The critical ordering flaw: `last_reward_block` is committed to storage **before** the `disable_rewards` branch. Once any caller writes `last_reward_block = N`, the assertion `current_block_number > self.last_reward_block.read()` will revert for every subsequent call at block N, including the legitimate sequencer call.

`general_prerequisites()` is shared with user-callable functions (`stake`, `unstake_intent`, `set_commission`, etc.) and therefore cannot contain a sequencer check. Unit tests confirm this: `update_rewards` is invoked in tests without any `cheat_caller_address_once` setup, and the first expected error is `STAKER_NOT_EXISTS`, not an access-control error.

### Impact Explanation
An attacker who calls `update_rewards(staker_address, disable_rewards: true)` at block N:
1. Passes all checks (contract unpaused, block N not yet recorded, staker exists and active).
2. Causes `last_reward_block = N` to be written.
3. The function returns without distributing any rewards.
4. The sequencer's subsequent call at block N reverts with `REWARDS_ALREADY_UPDATED`.
5. Rewards for block N are permanently lost — they are never credited to `unclaimed_rewards_own` or sent to pools.

This constitutes **permanent freezing of unclaimed yield**, a High-severity impact under the allowed scope.

### Likelihood Explanation
The function is publicly callable on-chain with no role restriction. An attacker needs only to monitor the mempool for the sequencer's `update_rewards` transaction and front-run it with `disable_rewards: true`. Alternatively, the attacker can call it first in any block before the sequencer does. The cost is a single transaction per block. The attack can be sustained indefinitely, draining all future block rewards for any targeted staker.

### Recommendation
Add a sequencer-only access control check at the top of `update_rewards`, analogous to how `update_rewards_from_attestation_contract` restricts callers to the attestation contract. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.roles.only_sequencer();   // <-- add this guard
    ...
```

Alternatively, move the `self.last_reward_block.write(current_block_number)` call to **after** the `disable_rewards` branch so that a call with `disable_rewards: true` does not consume the block slot.

### Proof of Concept
1. Consensus rewards are active (`is_pre_consensus()` returns false).
2. Staker `S` is registered and has non-zero balance at the current epoch.
3. At block `N`, attacker calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
4. Execution path: `general_prerequisites()` passes → block dedup passes → `last_reward_block.write(N)` → `disable_rewards == true` → early return.
5. Sequencer calls `staking.update_rewards(staker_address: S, disable_rewards: false)` at block `N` → reverts with `REWARDS_ALREADY_UPDATED`.
6. Staker `S` receives zero rewards for block `N`. The reward tokens that would have been minted and credited are permanently unclaimable.
