### Title
Stranded ETH on the router is silently consumed by any subsequent WETH-input swap via the partial-native branch in `pay()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay()` uses the router's entire native ETH balance as a first-priority funding source for any WETH-input swap. ETH left on the router by a prior user (via a `refundETH`-less multicall or direct `payable` call with excess `msg.value`) is wrapped and forwarded to the pool on behalf of the next WETH-input swapper, reducing that swapper's WETH pull by the stranded amount and permanently destroying the prior user's ETH.

### Finding Description
The `pay()` function in `PeripheryPayments` implements a "native-first" WETH settlement strategy: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` and, if positive but less than `value`, wraps and forwards all of it before pulling the remainder from `payer` via `transferFrom`. There is no check that the native balance was contributed by the current `payer`. Any ETH previously stranded on the router — from any prior user — is silently consumed.

ETH is stranded when a user sends `msg.value` to a `payable` entry point (`multicall`, `exactInputSingle`, `exactOutputSingle`) with more ETH than the swap requires and omits a `refundETH()` step: [2](#0-1) [3](#0-2) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) does not prevent stranding because `msg.value` in a `payable` function call is credited to the contract's balance without invoking `receive()`. [4](#0-3) 

### Impact Explanation
Direct loss of user funds. The prior user's stranded ETH is irreversibly transferred to the pool as WETH on behalf of the next swapper. The prior user cannot recover it via `refundETH()` because the balance is gone. The beneficiary swapper pays less WETH than the pool is owed from them, but the pool receives the correct total — the loss falls entirely on the prior user.

### Likelihood Explanation
Medium. The multicall + `refundETH()` pattern is the documented native ETH flow: [5](#0-4) 

Users who send excess ETH without `refundETH()` — a common mistake — are immediately vulnerable. An attacker can monitor the mempool for such transactions and back-run with a WETH-input swap to steal the stranded ETH. The attack requires no special role, no malicious pool, and no non-standard token behavior.

### Recommendation
Track the ETH contributed by the current call context (e.g., via a transient storage slot set at the `payable` entry point to `msg.value`) and cap the native-balance consumption in `pay()` to that tracked amount. Alternatively, require that the native balance used in `pay()` equals exactly `msg.value` of the current top-level call, reverting if the router holds any pre-existing ETH balance at swap entry.
