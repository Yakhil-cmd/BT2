### Title
Missing Lower-Bound Threshold Validation in FROST Signing Enables Single-Participant Signature Forgery - (File: src/frost/mod.rs)

---

### Summary

The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` validate the threshold parameter only against an upper bound (must not exceed participant count) but impose no lower bound. A malicious coordinator can supply `threshold = 1` to the FROST signing entry points, reducing the required quorum to a single participant and allowing that participant to produce a cryptographically valid threshold signature without the cooperation of any other party.

---

### Finding Description

**Root cause — missing lower-bound guard in `assert_sign_inputs`:**

`src/frost/mod.rs` lines 120–159 define the shared validation helper used by all FROST signing paths:

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants { ... });
    }
    // ...
    // validate threshold
    if threshold.value() > participants.len() {          // ← only upper bound
        return Err(InitializationError::ThresholdTooLarge { ... });
    }
    // ← NO lower-bound check; threshold = 1 (or 0) is silently accepted
    Ok(participants)
}
```

The same omission exists in the FROST `presign` entry point (lines 71–77):

```rust
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← threshold = 1 passes through unchecked
```

**Contrast with the DKG and triple-generation paths, which do enforce a minimum:**

`src/ecdsa/ot_based_ecdsa/triples/generation.rs` lines 699–703:
```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
```

The DKG test `keygen__should_fail_if_threshold_is_below_limit` (`src/dkg.rs` lines 738–759) explicitly confirms that `threshold = 1` must be rejected with `ThresholdTooSmall { threshold, min: 2 }` for key generation. No equivalent guard exists for FROST signing.

**Attack path:**

1. A legitimate keygen is run with `threshold = 2` (or higher), producing valid key shares for all participants.
2. A malicious coordinator initiates a FROST signing session, supplying `threshold = 1` to `assert_sign_inputs` / `presign`.
3. `assert_sign_inputs` accepts `threshold = 1` because `1 ≤ participants.len()`.
4. The FROST signing protocol now requires only **one** signature share to aggregate a valid final signature.
5. The coordinator, holding its own share, completes the signing round unilaterally and produces a cryptographically valid EdDSA or RedJubjub threshold signature over an attacker-chosen message — without the cooperation of any other key-share holder.

---

### Impact Explanation

**Severity: Critical**

The threshold security property of the scheme is entirely defeated. A single malicious coordinator can produce a valid threshold signature for any message of their choice using only their own key share, bypassing the quorum requirement that is the fundamental security guarantee of the protocol. This maps directly to the allowed critical impact: *Unauthorized creation of a valid threshold signature for attacker-chosen inputs.*

---

### Likelihood Explanation

The coordinator role is explicitly listed as a valid attacker profile in `RESEARCHER.md`. The coordinator controls the `threshold` argument passed to the FROST signing entry points. No out-of-band mechanism prevents a coordinator from supplying `threshold = 1`; the library itself is the only enforcement point, and it fails to enforce the minimum. Exploitation requires no cryptographic capability — only the ability to call the public API with a crafted parameter.

---

### Recommendation

Add a lower-bound check in `assert_sign_inputs` and in the FROST `presign` entry point, consistent with the guard already present in `validate_triple_inputs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be placed immediately after the `threshold.into()` conversion in both `assert_sign_inputs` (`src/frost/mod.rs` line 126) and `presign` (`src/frost/mod.rs` line 72), mirroring the pattern already enforced in `validate_triple_inputs` (`src/ecdsa/ot_based_ecdsa/triples/generation.rs` lines 699–703) and validated by the DKG test suite (`src/dkg.rs` lines 738–759).

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
