### Title
Missing Lower-Bound Validation on `threshold` in FROST Presign/Sign Allows Coordinator to Corrupt Signing Outputs — (File: `src/frost/mod.rs`)

---

### Summary

The `presign` and `assert_sign_inputs` functions in `src/frost/mod.rs` validate that `threshold` does not exceed `participants.len()`, but omit the symmetric lower-bound check (`threshold >= 2`) that is enforced in `assert_key_invariants` (`src/dkg.rs`). A malicious coordinator can supply `threshold = 0` or `threshold = 1` to honest participants, who will accept it without error and proceed into the signing round, producing a cryptographically invalid (unusable) signature.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on `threshold`: [1](#0-0) 

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

The FROST `presign` function performs only the upper-bound check and silently accepts `threshold = 0` or `threshold = 1`: [2](#0-1) 

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no lower-bound check; threshold = 0 or 1 passes here
```

`assert_sign_inputs`, called by every participant before the online signing round, has the identical gap: [3](#0-2) 

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no lower-bound check
```

Because each honest participant independently calls `assert_sign_inputs` with the coordinator-supplied `threshold`, a coordinator who broadcasts `threshold = 1` (or `0`) to all participants will pass every participant's local validation. The participants then enter `do_presign` / the signing round with an invalid threshold. [4](#0-3) 

With `threshold = 1`, FROST Lagrange interpolation reconstructs only a single share `x_i` rather than the actual secret, producing a signature scalar that does not satisfy the verification equation. With `threshold = 0`, the interpolation sums over an empty set, yielding `s = 0`, which is unconditionally rejected by any verifier. In both cases the signing round completes without a protocol-level error, but the output signature is cryptographically unusable.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

Every honest participant completes the signing protocol without detecting the invalid threshold. The coordinator-aggregated signature is structurally well-formed but cryptographically invalid. Honest parties have no in-protocol signal that the output is wrong; they must discover the failure only at external verification time. A malicious coordinator can repeat this on every signing request, permanently denying the ability to produce a valid threshold signature.

---

### Likelihood Explanation

The coordinator role is filled by one of the signing participants (validated by `assert_sign_inputs` line 153). Any participant who is selected as coordinator can supply an out-of-range threshold. No key material or external compromise is required — only the ability to initiate a signing session with a crafted `threshold` value. [5](#0-4) 

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces, in both `presign` and `assert_sign_inputs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing check in `src/dkg.rs` lines 580–582 and closes the asymmetry between key-generation and signing validation. [6](#0-5) 

---

    me: Participant,

    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {

            role: "coordinator",
            participant: coordinator,
