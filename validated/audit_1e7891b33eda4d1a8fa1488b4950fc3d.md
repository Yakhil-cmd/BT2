### Title
Unprotected `update_rewards` Allows Any Caller to Permanently Freeze Staker and Delegator Yield - (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` every block, consuming the global `last_reward_block` slot and preventing the legitimate sequencer from ever distributing rewards — permanently freezing unclaimed yield for all stakers and delegators.

---

### Finding Description

The protocol specification explicitly restricts `update_rewards` to the Starkware sequencer:

> **access control:** Only starkware sequencer.
> (`docs/spec.md`, lines 1644–1645)

The implementation in `src/staking/staking.cairo` (`StakingRewardsManagerImpl`, lines 1447–1507) enforces only three conditions:

1. Contract is not paused (`general_prerequisites()`).
2. `current_block_number > self.last_reward_block.read()` — not already updated this block.
3. The supplied `staker_address` exists and is active with non-zero balance.

There is **no check on `get_caller_address()`**. The function is callable by any address.

The critical ordering flaw is that `last_reward_block` is written **before** the `disable_rewards` branch:

```cairo
// line 1485
self.last_reward_block.write(current_block_number);

// line 1487
if disable_rewards || self.is_pre_consensus() {
    return;   // ← exits without distributing rewards, but slot is consumed
}
```

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` in a given block:
- Passes all three checks (no caller restriction, block is new, staker is valid).
- Writes `last_reward_block = current_block_number`.
- Returns immediately with zero rewards distributed.
- The sequencer's subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

By repeating this every block (a simple bot), the attacker permanently prevents any block rewards from being distributed to stakers or their delegation pool members.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

All stakers and all delegation pool members accumulate zero block rewards for as long as the attack is sustained. The `unclaimed_rewards_own` field of every staker and the `unclaimed_rewards` of every pool member remain at zero. No funds are stolen, but all yield is frozen indefinitely. The attack is protocol-wide: a single call per block affects every staker simultaneously because `last_reward_block` is a single global storage slot.

---

### Likelihood Explanation

**High.** The function is publicly callable with no economic barrier. The attacker only needs to submit one transaction per block (a trivially cheap bot). There is no profit motive required — a competitor, a griever, or a short-seller of STRK has clear incentive. The attack is fully permissionless and requires no privileged access, leaked keys, or external dependencies.

---

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the configured sequencer address (or a dedicated role), consistent with the specification. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    self.general_prerequisites();
    // ...
}
```

Alternatively, store the sequencer address in contract storage and expose a governance function to update it, mirroring the pattern used for `attestation_contract` in `update_rewards_from_attestation_contract`.

**Attack steps:**

1. Wait for consensus rewards to be active (`is_pre_consensus()` returns `false`).
2. Each block, call `update_rewards(<any_active_staker>, disable_rewards: true)` before the sequencer does.
3. The call succeeds: `last_reward_block` is set to the current block, zero rewards are distributed.
4. The sequencer's call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers and delegators receive zero block rewards indefinitely.
