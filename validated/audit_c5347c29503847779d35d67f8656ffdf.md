### Title
Missing Zero-Address Validation for `reward_address` Enables Permanent Freezing of Unclaimed Yield — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `change_reward_address()` in `staking.cairo` and `pool.cairo` validate that the new `reward_address` is not a registered token contract address, but neither validates that it is non-zero. A staker or pool member can set their `reward_address` to `0`, causing all subsequent reward transfers to either permanently burn accrued yield (if the STRK ERC20 silently accepts zero-address transfers) or revert on every `claim_rewards` / `unstake_action` call (if the ERC20 panics on zero-address transfers), temporarily freezing the staker's principal as well.

---

### Finding Description

**Vulnerability class**: Token-freeze bug / incomplete validation (direct analog of the BeraBitcoin pattern).

In `staking.cairo`, `change_reward_address` checks that the new address is not a token contract but omits a zero-address guard:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();          // only checks caller != 0
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // ← no assert!(reward_address.is_non_zero(), ...)
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

The same omission exists in `pool.cairo`'s `change_reward_address`:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // ← no assert!(reward_address.is_non_zero(), ...)
    let pool_member = get_caller_address();
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    pool_member_info.reward_address = reward_address;
    self.write_pool_member_info(:pool_member, :pool_member_info);
    ...
}
``` [2](#0-1) 

The same gap exists at entry time in `stake()` and `enter_delegation_pool()`: [3](#0-2) [4](#0-3) 

The asymmetry mirrors the BeraBitcoin pattern exactly: the **caller** address is validated (non-zero, via `assert_caller_is_not_zero()` inside `general_prerequisites()`), but the **reward destination** address is not. [5](#0-4) 

When rewards are later disbursed, `send_rewards_to_staker` unconditionally transfers to whatever `reward_address` is stored:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [6](#0-5) 

And `claim_rewards` in the pool does the same:

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [7](#0-6) 

`unstake_action` calls `send_rewards_to_staker` **before** returning the principal, so a zero `reward_address` with non-zero accrued rewards causes the entire `unstake_action` to revert, blocking principal withdrawal: [8](#0-7) 

---

### Impact Explanation

**Scenario A — ERC20 reverts on zero-address transfer (standard OpenZeppelin Cairo behaviour):**

1. Staker calls `change_reward_address(0)` (or stakes with `reward_address = 0`).
2. Rewards accumulate in `unclaimed_rewards_own`.
3. Staker calls `unstake_action` → `send_rewards_to_staker` → `checked_transfer(recipient: 0, amount: rewards)` → ERC20 panics → entire `unstake_action` reverts.
4. The staker's principal is **temporarily frozen**: they cannot exit until they call `change_reward_address` again with a valid address.

**Scenario B — ERC20 silently accepts zero-address transfer (burn):**

1. Same setup.
2. `claim_rewards` or `unstake_action` succeeds, but all accrued STRK yield is sent to address `0` and permanently destroyed.
3. **Permanent freezing/burning of unclaimed yield.**

Both scenarios apply equally to pool members via `pool.cairo`'s `change_reward_address` and `claim_rewards`.

Matched allowed impacts:
- **High: Permanent freezing of unclaimed yield** (Scenario B).
- **High: Temporary freezing of funds** (Scenario A — principal locked until recovery action).

---

### Likelihood Explanation

Any unprivileged staker or pool member can trigger this by calling `change_reward_address(0)` directly, or by supplying `reward_address = 0` at entry (`stake` / `enter_delegation_pool`). No privileged role, bridge compromise, or external dependency is required. The call is permissionless and costs only gas. Accidental triggering (e.g., a front-end bug passing an uninitialized address) is also realistic.

---

### Recommendation

Add a non-zero guard for `reward_address` in every function that accepts or updates it:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply this in:
- `staking.cairo`: `stake()`, `change_reward_address()`
- `pool.cairo`: `enter_delegation_pool()`, `change_reward_address()`

The `GenericError::ZERO_ADDRESS` variant already exists in the shared error enum. [9](#0-8) 

---

### Proof of Concept

```
1. Deploy the system (staking + pool contracts).
2. Staker calls stake(reward_address: 0, operational_address: op, amount: min_stake).
   → Succeeds; reward_address stored as 0.
3. Advance K epochs; staker attests → rewards accumulate in unclaimed_rewards_own.
4. Staker calls unstake_intent().
5. Advance past exit_wait_window.
6. Staker calls unstake_action(staker_address).
   → send_rewards_to_staker fires checked_transfer(recipient: 0, amount: rewards).
   → STRK ERC20 panics: "ERC20: transfer to the zero address".
   → unstake_action reverts; principal remains locked in the staking contract.
7. (Recovery) Staker calls change_reward_address(valid_address), then retries unstake_action.
   → Succeeds; but if Scenario B applies, all accrued yield was already burned.
```

        fn unstake_action(ref self: ContractState, staker_address: ContractAddress) -> Amount {
            // Prerequisites and asserts.
            self.general_prerequisites();
            let mut staker_info = self.internal_staker_info(:staker_address);
            let unstake_time = staker_info
                .unstake_time
                .expect_with_err(Error::MISSING_UNSTAKE_INTENT);
            assert!(Time::now() >= unstake_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Send rewards to staker's reward address.
            // It must be part of this function's flow because staker_info is about to be erased.
            let token_dispatcher = strk_token_dispatcher();
            self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
            // Update staker info to storage (it will be erased later).
            // This is done here to avoid re-entrancy.
            self.write_staker_info(:staker_address, :staker_info);

            let staker_amount = self.get_own_balance(:staker_address).to_strk_native_amount();
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            self.remove_staker(:staker_address, :staker_info, :staker_pool_info);

            // Return stake to staker.
            token_dispatcher
                .checked_transfer(recipient: staker_address, amount: staker_amount.into());
            // Return delegated stake to pools and zero their balances.
            self
                .transfer_to_pools_when_unstake(
                    :staker_address, staker_pool_info: staker_pool_info.as_non_mut(),
                );
            // Clear staker pools.
            staker_pool_info.pools.clear();
            staker_amount
        }
