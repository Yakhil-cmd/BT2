### Title
`MsgTurnBridge` Handler Missing Authorization Check and State Mutation — (`File: x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` gRPC message handler in `x/cronos/keeper/msg_server.go` is a stub that returns `nil, nil` unconditionally. It performs no `CanTurnBridge` permission check and writes no state. Any unprivileged address can submit `MsgTurnBridge` and receive a success response, fully bypassing the bridge authorization gate. Simultaneously, no authorized party — including the Cronos admin — can actually disable the gravity bridge, permanently breaking the emergency circuit-breaker.

### Finding Description
`UpdateTokenMapping` correctly gates execution behind `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`:

```go
// x/cronos/keeper/msg_server.go:68-82
func (k msgServer) UpdateTokenMapping(...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, ...)
    }
    ...
}
```

The sibling handler `TurnBridge`, which is supposed to enable/disable the gravity bridge and requires the `CanTurnBridge` permission, is instead:

```go
// x/cronos/keeper/msg_server.go:84-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

No permission check. No state write. The handler is a no-op stub that always succeeds.

The `CanTurnBridge` permission bit (`2`) is defined alongside `CanChangeTokenMapping` (`1`) in the permission system, and `HasPermission` is the correct guard to call:

```go
// x/cronos/keeper/permissions.go:13-16
const (
    CanChangeTokenMapping uint64 = 1 << iota // 1
    CanTurnBridge                             // 2
    All = CanChangeTokenMapping | CanTurnBridge // 3
)
```

The proto definition and CLI both expose `MsgTurnBridge` as a live, routable transaction type with a `sender` field and an `enable` bool. The gRPC handler is registered and reachable on-chain.

### Impact Explanation
Two distinct impacts:

1. **Authorization bypass (High):** Any unprivileged address can broadcast `MsgTurnBridge` and receive a `code: 0` success response. The `CanTurnBridge` permission check — the only guard protecting bridge-control authority — is entirely absent. This directly satisfies: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

2. **Permanent inability to disable the bridge (High):** Because the handler writes no state, no party — including the Cronos admin or a permissioned address — can ever disable the gravity bridge. The circuit-breaker designed to halt bridge operations in an emergency is permanently non-functional. This satisfies: *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows … under normal network assumptions."*

### Likelihood Explanation
The entry path is a standard signed Cosmos SDK transaction (`MsgTurnBridge`) with no preconditions beyond a valid bech32 sender address. Any funded account can reach it. The integration test `test_gravity_turn_bridge` explicitly expects `from_="community"` to fail with a non-zero code, but the stub returns success for every caller, confirming the bypass is live.

### Recommendation
Restore the full implementation of `TurnBridge`:

1. Add the `CanTurnBridge` permission check mirroring `UpdateTokenMapping`:
   ```go
   if !k.HasPermission(ctx, []sdk.AccAddress{senderAddr}, CanTurnBridge) {
       return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
   }
   ```
2. Write the bridge-enabled flag to the module's param store (e.g., update `Params.EnableAutoDeployment` or a dedicated `BridgeActive` param via `SetParams`), so the state change is actually persisted and enforced by downstream bridge logic.

### Proof of Concept
```bash
# Any unprivileged address succeeds — no CanTurnBridge permission required
cronosd tx cronos turn-bridge false \
  --from <any_funded_address> \
  --chain-id cronos_777-1 -y

# Expected (correct): code != 0, ErrUnauthorized
# Actual (buggy):     code == 0, empty MsgTurnBridgeResponse
```

The stub at [1](#0-0)  returns `nil, nil` with no guard, while the correct permission constant [2](#0-1)  and the working pattern in `UpdateTokenMapping` [3](#0-2)  both exist in the same file. The proto-registered handler [4](#0-3)  routes live transactions to this stub. The integration test at [5](#0-4)  confirms the expected behavior (community should fail, validator should succeed) that the stub violates.
