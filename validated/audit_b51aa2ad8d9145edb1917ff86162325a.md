### Title
Missing Access Control on `update_rewards` Allows Anyone to Permanently Deny Block Rewards — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is documented as callable "Only starkware sequencer" but contains no caller validation. Any unprivileged address can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` to the current block without distributing any rewards. Because the function enforces a strict "one call per block" invariant, the legitimate sequencer's subsequent call for that block is permanently rejected with `REWARDS_ALREADY_UPDATED`, causing all stakers to lose their consensus block rewards for that block.

---

### Finding Description

`IStakingRewardsManager::update_rewards` is the V3 consensus-rewards entry point. The spec explicitly restricts it:

> **access control**: Only starkware sequencer.
> (`docs/spec.md`, line 1645)

The implementation, however, performs no such check:

```cairo
// src/staking/staking.cairo  lines 1449-1510
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validity checks ...
    self.last_reward_block.write(current_block_number);   // ← global write, no auth
    if disable_rewards || self.is_pre_consensus() {
        return;                                            // ← exits without rewards
    }
    // ... reward distribution ...
}
``` [1](#0-0) 

The critical sequence is:

1. `last_reward_block` is written to `current_block_number` **before** the `disable_rewards` branch.
2. If `disable_rewards == true`, the function returns immediately — no rewards are minted or credited.
3. Any subsequent call for the same block (including the legitimate sequencer's) hits `REWARDS_ALREADY_UPDATED` and reverts.

`last_reward_block` is a single global storage slot (not per-staker), so one attacker call poisons the entire block for every staker. [2](#0-1) 

The spec confirms the global, single-call-per-block semantics:

> **pre-condition**: Rewards did not distributed for the current block yet.
> (`docs/spec.md`, line 1643) [3](#0-2) 

The interface exposes this function publicly with no role guard:

```cairo
// src/staking/interface.cairo  lines 303-311
#[starknet::interface]
pub trait IStakingRewardsManager<TContractState> {
    fn update_rewards(
        ref self: TContractState, staker_address: ContractAddress, disable_rewards: bool,
    );
}
``` [4](#0-3) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who front-runs the sequencer's `update_rewards` call for block `N` with `disable_rewards: true`:

- Advances `last_reward_block` to `N` with zero rewards distributed.
- Permanently blocks the sequencer from crediting block-`N` rewards to any staker.
- Stakers' `unclaimed_rewards_own` is never incremented for that block; the yield is lost forever (it is not deferred to the next block).

Repeated across every block, this completely halts consensus reward accrual for the entire protocol.

---

### Likelihood Explanation

**High.** The attack requires:

- No special role, no funds, no prior state.
- Only knowledge of a single valid `staker_address` (publicly readable from events or `staker_info_v1`).
- A single transaction per block, callable by any EOA or contract.

On Starknet, where transaction ordering is sequencer-controlled, a determined attacker can submit the griefing call at the start of every block. There is no economic barrier.

---

### Recommendation

Add a sequencer-only guard at the top of `update_rewards`, consistent with the spec's stated access control. The existing `RolesComponent` already provides the infrastructure:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.roles.only_operator();   // or a dedicated SEQUENCER role
    self.general_prerequisites();
    // ... rest unchanged ...
}
```

Alternatively, restrict via the `app_governor` / `operator` role already used elsewhere in the contract, or introduce a dedicated `SEQUENCER` role assigned to the Starknet sequencer address.

### Proof of Concept

```
Block N arrives.

1. Attacker (any address) calls:
       staking.update_rewards(staker_address=<any_valid_staker>, disable_rewards=true)

2. Inside update_rewards:
   - general_prerequisites() passes (contract not paused).
   - current_block_number (N) > last_reward_block → assertion passes.
   - last_reward_block.write(N)   ← global slot updated.
   - disable_rewards == true → early return, NO rewards distributed.

3. Sequencer calls:
       staking.update_rewards(staker_address=<staker>, disable_rewards=false)

4. Inside update_rewards:
   - current_block_number (N) > last_reward_block (N) → FALSE → PANIC: REWARDS_ALREADY_UPDATED.

5. All stakers receive zero block rewards for block N.
   Repeating this every block permanently freezes all consensus yield.
``` [5](#0-4) [6](#0-5)
