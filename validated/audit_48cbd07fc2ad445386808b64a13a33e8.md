### Title
Unvalidated `sender` Argument in Bank Precompile `transfer` Allows Unauthorized Token Theft - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer` method accepts an arbitrary `sender` address from ABI-encoded call input without verifying it matches `contract.Caller()`. Any EVM contract can therefore transfer `evm/<its_own_address>` native Cosmos tokens from any holder to any recipient without the holder's authorization.

### Finding Description
The `BankContract.Run` function in `x/cronos/keeper/precompiles/bank.go` handles three state-mutating methods: `mint`, `burn`, and `transfer`. For `mint` and `burn`, the denom is derived from `contract.Caller()` and the recipient/burner is taken from ABI input — this is safe because the contract controls its own denom. However, for `transfer`, both the `sender` and `recipient` are taken from ABI input, while the denom is still derived from `contract.Caller()`:

```go
// bank.go lines 175-192
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())   // evm/0x<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no check that `sender == contract.Caller()` or that the holder has approved the transfer. The precompile unconditionally executes `bankKeeper.SendCoins(from, to, ...)` where `from` is fully attacker-controlled.

This is the direct analog to the external report's root cause: just as `blockHead.hash` from an untrusted remote node was used without length/content validation to construct signed operations, here the `sender` field from an untrusted EVM call input is used without identity validation to execute a Cosmos bank transfer.

### Impact Explanation
Any EVM contract at address `0xC` can call the bank precompile's `transfer(victim, attacker, amount)` and move `evm/0xC` tokens from `victim` to `attacker` without `victim`'s consent. `evm/` denom tokens are native Cosmos bank module tokens managed by the precompile; they are real transferable assets. This constitutes an **unauthorized transfer of precompile-controlled assets** — Critical impact per the allowed scope.

### Likelihood Explanation
The attacker only needs to deploy a contract and call the precompile at address `0x64` (decimal 100). No privileged keys, governance, or validator compromise is required. Any user who holds `evm/<contract>` tokens (e.g., received via a `mint` call from that contract) is immediately at risk. The precompile is reachable from any EVM transaction. [2](#0-1) 

### Recommendation
In the `transfer` case, replace the ABI-supplied `sender` with `contract.Caller()` as the authoritative source of the sending address, mirroring how `mint`/`burn` derive the denom from `contract.Caller()`. If third-party-initiated transfers are intentional, add an explicit on-chain approval/allowance mechanism before executing `SendCoins`.

```go
// Replace:
sender := args[0].(common.Address)
from := sdk.AccAddress(sender.Bytes())

// With:
from := sdk.AccAddress(contract.Caller().Bytes())
```

### Proof of Concept

1. Attacker deploys `MaliciousContract` at address `0xC`.
2. `MaliciousContract` calls bank precompile (`0x64`) method `mint(victim_addr, 1000)` → victim receives 1000 `evm/0xC` tokens in the Cosmos bank module.
3. `MaliciousContract` calls bank precompile method `transfer(victim_addr, attacker_addr, 1000)`.
4. The precompile executes `bankKeeper.SendCoins(victim_addr, attacker_addr, [{1000, "evm/0xC"}])` with no authorization check.
5. Victim loses 1000 `evm/0xC` tokens; attacker gains them. Victim never signed or approved the transfer.
