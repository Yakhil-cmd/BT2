### Title
`protocolUnpausePool` Sets Pause Level to 1 (Admin-Paused) Instead of 0 (Active), Leaving Pool Permanently Paused After Protocol Unpause - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`. After the protocol owner invokes this function, the pool transitions from protocol-paused (2) to admin-paused (1) ŌĆö not to active (0). Swaps remain blocked, and the pool admin must issue a separate `unpausePool` call to restore trading. If the pool was active before the protocol pause, the admin is placed in a paused state they never chose.

### Finding Description
The pause-level semantics are documented inline:

> `0 = active, 1 = paused by admin, 2 = paused by protocol`

The four factory transitions are:

| Function | Caller | Allowed `cur` | Target |
|---|---|---|---|
| `pausePool` | pool admin | 0 | 1 |
| `unpausePool` | pool admin | 1 | 0 |
| `protocolPausePool` | owner | 0 or 1 | 2 |
| `protocolUnpausePool` | owner | 2 | **1** ŌåÉ bug |

`protocolUnpausePool` (line 402) calls `setPause(1)`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ŌåÉ should be setPause(0)
}
```

`_checkNotPaused` blocks swaps for any `pauseLevel != 0`:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

So after `protocolUnpausePool` completes, `pauseLevel == 1` and every `swap` call still reverts with `PoolPaused`. The pool admin must separately call `unpausePool` (which requires `cur == 1`, so the precondition is met) to reach `pauseLevel == 0`.

The wrong-value analog to the external report is exact: just as `redeemOzUSD` passed `msg.sender` where `_from` was required, `protocolUnpausePool` passes `1` where `0` is required ŌĆö a single wrong literal that corrupts the intended state transition.

### Impact Explanation
Every call to `protocolUnpausePool` on a pool that was active (state 0) before the protocol pause leaves the pool in admin-paused state (1). Swaps ŌĆö the core revenue-generating operation ŌĆö remain blocked. If the pool admin is unresponsive, the pool is permanently frozen for traders. LPs retain the ability to remove liquidity (no `whenNotPaused` guard on `removeLiquidity`), so principal is not directly lost, but the pool's primary function is broken until a second privileged action is taken.

### Likelihood Explanation
The trigger is a routine, expected protocol operation: the owner pauses a pool for safety and later unpauses it. No adversarial setup is required. The bug fires on every such round-trip where the pool was active before the protocol pause, which is the common case.

### Recommendation
Change the target pause level in `protocolUnpausePool` from `1` to `0`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // restore to active
}
```

If the intended design is to preserve the pre-protocol-pause admin state, the factory must store that state before calling `protocolPausePool` and restore it here.

### Proof of Concept

1. Pool is deployed; `pauseLevel == 0` (active).
2. Owner calls `protocolPausePool(pool)` ŌåÆ `pauseLevel == 2`.
3. Owner calls `protocolUnpausePool(pool)` ŌåÆ `setPause(1)` ŌåÆ `pauseLevel == 1`.
4. Any trader calls `swap(...)` ŌåÆ `_checkNotPaused()` ŌåÆ `revert PoolPaused()`.
5. Pool admin must call `unpausePool(pool)` to reach `pauseLevel == 0`.
6. If admin is absent, swaps are permanently blocked despite the owner having "unpaused" the pool. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
