### Title
`MsgTurnBridge` Handler Is a No-Op Stub with No Authorization Check, Permanently Disabling the Bridge Circuit Breaker — (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is an unimplemented stub that unconditionally returns `nil, nil` — success — without performing any authorization check and without mutating any state. The `CanTurnBridge` permission constant is defined and the permission system is wired, but the handler never consults it. Any unprivileged user can submit `MsgTurnBridge` and receive a success response, while the bridge enable/disable state is never actually changed. The Gravity bridge emergency circuit breaker is therefore permanently non-functional.

### Finding Description

`TurnBridge` is the designated emergency circuit breaker for the Gravity bridge. ADR-009 explicitly lists `MsgTurnBridge` as one of the two messages that must require the `CanTurnBridge` permission before execution. The permission constant is declared in `x/cronos/keeper/permissions.go` and `HasPermission` is correctly called in `UpdateTokenMapping`. However, the `TurnBridge` handler itself was never implemented:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare with `UpdateTokenMapping`, which correctly gates on `HasPermission`: [2](#0-1) 

The `CanTurnBridge` bit is defined but never enforced in the handler: [3](#0-2) 

The proto definition confirms `MsgTurnBridge` carries a `sender` and an `enable` bool that are supposed to control bridge state: [4](#0-3) 

**Attacker path:**

1. Any unprivileged address constructs a `MsgTurnBridge{Sender: <attacker>, Enable: false}`.
2. The transaction passes `ValidateBasic` (only checks bech32 validity of `Sender`).
3. The `BlockAddressesDecorator` ante handler does not block it (no rule against `MsgTurnBridge` from unprivileged senders).
4. The handler executes, returns `nil, nil` — the transaction is committed as successful.
5. No bridge state is changed. The bridge remains in its current state regardless of the `Enable` field.

The analog to the external report's "event-based auth" class: the external report describes authentication that is a simple boolean check decoupled from the protected resource. Here, the authorization check is entirely absent — the handler never reads `msg.Sender`, never calls `HasPermission`, and never writes any state. The `CanTurnBridge` permission exists in the system but is never bound to the action it is supposed to protect.

### Impact Explanation

**High — Bypass of bridge authorization check + permanent inability to exercise the bridge circuit breaker.**

- The `CanTurnBridge` permission guard is completely absent from the handler, meaning the authorization boundary for bridge control is bypassed by every caller, privileged or not.
- Because the handler is a no-op, the Gravity bridge enable/disable state can never be changed. If a bridge exploit is discovered, the admin and governance have no on-chain mechanism to halt bridge flows. Funds bridging through Gravity during an active exploit cannot be stopped.
- Any user can submit `MsgTurnBridge` and receive a success receipt, creating false assurance that the bridge state was changed.

This satisfies: *"High: Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks"* and *"High: Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows … under normal network assumptions."*

### Likelihood Explanation

The entry path requires no privileges — any funded address can submit `MsgTurnBridge`. The stub is reachable on mainnet today. The only precondition is a valid bech32 sender address and enough gas.

### Recommendation

Implement `TurnBridge` with the same pattern as `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` if it fails.
2. Write the `enable` flag to the Gravity bridge module's params or a dedicated KV key so that the bridge end-blocker and send-to-Ethereum handler respect it.
3. Add a unit test asserting that an unprivileged sender receives `ErrUnauthorized` and that a permissioned sender actually mutates the bridge-enabled state.

### Proof of Concept

```
# Any address can call TurnBridge and receive code=0 (success)
cronosd tx cronos turn-bridge false \
  --from <any_unprivileged_key> \
  --chain-id cronosmainnet_25-1 \
  --gas auto -y

# Result: tx committed, code=0, but bridge state is unchanged.
# Admin calling turn-bridge also has no effect — bridge cannot be halted.
```
