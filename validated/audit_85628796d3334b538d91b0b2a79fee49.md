### Title
Unrestricted `update_rewards` with `disable_rewards: true` Permanently Freezes All Consensus Block Rewards - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function has no access control and accepts a caller-supplied `disable_rewards` boolean. Any unprivileged address can call `update_rewards(any_active_staker, disable_rewards: true)` in every block, consuming the global `last_reward_block` slot without distributing any rewards. This permanently freezes consensus block rewards for all stakers in the protocol.

### Finding Description
`update_rewards` in `src/staking/staking.cairo` is a public function with no caller restriction. It accepts an arbitrary `staker_address` (any active staker) and a `disable_rewards` boolean. The critical ordering flaw is that the function unconditionally writes `last_reward_block` to the current block number **before** checking `disable_rewards`:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);   // line 1485 — always executes

if disable_rewards || self.is_pre_consensus() {
    return;                                            // line 1487-1489 — exits with no rewards
}
```

The global `last_reward_block` acts as a one-slot-per-block lock:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

Because the slot is consumed before the `disable_rewards` guard, any caller who invokes `update_rewards(valid_staker, true)` in a block burns that block's reward opportunity for the entire protocol. There is no nonce, sequence, or caller identity check that would prevent an attacker from doing this in every block.

The only preconditions the attacker must satisfy are:
- Provide a `staker_address` that is active and has non-zero balance at the current epoch — trivially met using any publicly observable staker address from on-chain events.
- The contract must not be paused (`general_prerequisites()`).

### Impact Explanation
**Permanent freezing of unclaimed yield** for all stakers. As long as the attacker submits one transaction per block calling `update_rewards(any_active_staker, disable_rewards: true)`, no staker in the protocol can ever accumulate consensus block rewards. The `last_reward_block` slot is consumed without distributing anything, and the assertion on line 1454–1458 causes every legitimate call in that block to revert with `REWARDS_ALREADY_UPDATED`. This matches the allowed High impact: *Permanent freezing of unclaimed yield or unclaimed royalties*.

### Likelihood Explanation
High. The attack requires:
- No privileged role, no stake, no leaked key.
- One transaction per block (cheap on Starknet).
- Any active staker address, which is public from `NewStaker` events.

The attacker has a clear economic motive if they hold a competing position or wish to damage the protocol.

### Recommendation
1. Add an access-control check to `update_rewards` restricting callers to an authorized address (e.g., the attestation contract or a designated sequencer/operator role).
2. Move `self.last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard, so that a no-op call does not consume the block's reward slot.
3. Consider removing `disable_rewards` from the public ABI entirely and handling the pre-consensus / migration path through an internal or governance-gated function.

### Proof of Concept
1. Attacker observes any active staker address `S` from on-chain `NewStaker` events.
2. In block `N`, attacker calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
3. Execution path:
   - `general_prerequisites()` passes (contract not paused).
   - `current_block_number (N) > last_reward_block` passes (first call this block).
   - Staker `S` is active with non-zero balance — passes.
   - **`last_reward_block` is written to `N`** (line 1485).
   - `disable_rewards == true` → function returns early, zero rewards distributed (line 1487–1489).
4. Any legitimate call to `update_rewards` in block `N` now reverts: `current_block_number (N) > last_reward_block (N)` is **false** → `REWARDS_ALREADY_UPDATED`.
5. Attacker repeats step 2 in every subsequent block.
6. All stakers are permanently frozen from receiving consensus block rewards. [1](#0-0) [2](#0-1)
