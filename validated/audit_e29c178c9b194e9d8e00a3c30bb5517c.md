### Title
`setPoolAdminFeeDestination` Overwrites Fee Recipient Without Collecting Accrued Fees, Causing Loss of Owed Admin Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory::setPoolAdminFeeDestination()` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every subsequent `collectPoolFees` call then routes all previously-earned admin fees — both spread-surplus and notional-accumulator balances — to the new destination, permanently depriving the old recipient of funds it was owed.

---

### Finding Description

The protocol has two fee types that accumulate silently on the pool between explicit collection calls:

1. **Spread fees** — held as implicit surplus: `balance * scaleMultiplier − binTotals.scaledToken − notionalFeeScaled`. They grow with every swap and are only realized when `collectFees` is called.
2. **Notional fees** — tracked explicitly in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`, reset to zero only inside `collectFees`.

`collectPoolFees` reads `poolAdminFeeDestination[pool]` at call time and passes it directly to `collectFees`, which transfers the admin share of **all** currently-accrued fees to that address: [1](#0-0) 

Inside `collectFees`, the admin share of both surplus and notional accumulators is sent to `adminFeeDestination_`: [2](#0-1) 

The sibling mutators `setPoolAdminFees` and `setPoolProtocolFee` both flush fees **before** changing any fee-related state, establishing the correct pattern: [3](#0-2) 

`setPoolAdminFeeDestination` breaks this pattern entirely — it overwrites the destination with no prior flush: [4](#0-3) 

After the call, the old destination has no on-chain claim to the fees it earned. The next `collectPoolFees` (permissionless — any caller) sends those fees to the new address.

---

### Impact Explanation

The old `adminFeeDestination` — which may be a DAO treasury, revenue-sharing contract, or any address distinct from the pool admin — permanently loses all admin fees that accrued since the last collection. Both the spread surplus (proportional to total swap volume × admin spread rate) and the notional accumulator balance are redirected. This is a direct, irreversible loss of owed protocol-fee assets with no recovery path.

---

### Likelihood Explanation

The pool admin is semi-trusted and has a legitimate reason to update the fee destination (e.g., rotating a treasury address). The trigger requires no special conditions: the pool admin calls `setPoolAdminFeeDestination` at any time while fees have accrued. Because `collectPoolFees` is permissionless, an adversarial watcher can front-run the destination change with a `collectPoolFees` call to drain fees to the old address before the update, but the reverse is also true — the pool admin can call `setPoolAdminFeeDestination` and immediately trigger `collectPoolFees` to redirect all accrued fees to the new address, stealing from the old recipient.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: collect accrued fees at the **old** destination before updating the mapping.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

+   // Flush fees owed to the current destination before rotating.
+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+       c.protocolSpreadFeeE6,
+       c.adminSpreadFeeE6,
+       c.protocolNotionalFeeE8,
+       c.adminNotionalFeeE8,
+       poolAdminFeeDestination[pool]
+   );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is created with `adminFeeDestination = treasury`.
2. Swaps occur; spread surplus and notional accumulators grow.
3. Pool admin calls `setPoolAdminFeeDestination(pool, attacker)` — no fees are flushed.
4. Anyone calls `collectPoolFees(pool)` — all accrued admin fees (spread + notional) are sent to `attacker`.
5. `treasury` receives nothing despite having earned those fees.

The corrupted value is `poolAdminFeeDestination[pool]`: it is overwritten before the accrued fee balance it governs is settled, causing the entire admin-fee surplus and notional accumulator to be misrouted to the new address.
