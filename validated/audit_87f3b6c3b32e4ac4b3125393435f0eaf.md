### Title
Incorrect Multi-NFT Royalty Amount Displayed in WalletConnect Signing Confirmation Causes Users to Approve Offers With Understated Spend - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each royalty, causing the "Total Amount with Royalties" shown in the WalletConnect signing confirmation dialog to be systematically understated whenever an offer involves more than one NFT with royalties. A user approves a `take_offer` or `create_offer_for_ids` WalletConnect request believing they will spend less than the blockchain will actually deduct.

### Finding Description

`formatAmountWithRoyalties` is called by `withRoyaltyTotals`, which is called by `walletDeltaToDisplay`, which is called by `parseCommandDisplay` for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands. The result populates the `amountWithRoyalties` field on each spending line, which is rendered as "Total Amount with Royalties" in the `Confirm` dialog's "You Spend" section.

The faulty logic:

```js
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

For N NFTs with royalty percentages `r₁, r₂, …, rₙ`, the code computes:

```
royaltyAmount = (amount / N) * (r₁ + r₂ + … + rₙ) / 10000
```

The correct formula is:

```
royaltyAmount = amount * (r₁ + r₂ + … + rₙ) / 10000
```

The two are equal only when N = 1. For N ≥ 2 the displayed royalty is underestimated by a factor of N, and BigInt integer division introduces additional truncation loss.

The `Confirm` dialog renders this field directly as the authoritative spend figure the user sees before clicking "Approve": [2](#0-1) 

`parseCommandDisplay` is wired into the WalletConnect approval path in `main.tsx`: [3](#0-2) 

### Impact Explanation

When a WalletConnect-connected dApp sends a `take_offer` request containing an offer with two or more NFTs that carry royalties, the confirmation dialog shows a "Total Amount with Royalties" that is N times smaller than the actual royalty component. The user approves believing they will spend X, but the blockchain deducts X + (full royalties). This is a direct, concrete asset loss caused by incorrect state displayed in the WalletConnect approval flow.

### Likelihood Explanation

Any WalletConnect-paired dApp can craft or relay a multi-NFT offer. No special privileges are required. Multi-NFT offers are a supported and documented feature of the Chia offer protocol. The bug is triggered automatically whenever the offer contains ≥ 2 NFTs with non-zero royalty percentages on the side opposite the fungible asset.

### Recommendation

Replace the `splitAmount` division with the full `amount` for each royalty term:

```js
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This matches the per-NFT royalty semantics used by the Chia wallet backend and eliminates the N-fold underestimation.

### Proof of Concept

1. Pair a WalletConnect dApp with the Chia GUI.
2. The dApp sends `chia_wallet.take_offer` with an offer where the user spends 10 XCH (10,000,000,000,000 mojos) to receive two NFTs, each with a 5 % royalty (`royaltyPercentage = 500`).
3. Correct "Total Amount with Royalties": `10 XCH + (10 * 5%) + (10 * 5%) = 11 XCH`.
4. Displayed "Total Amount with Royalties" (current code):
   - `splitAmount = 10,000,000,000,000 / 2 = 5,000,000,000,000`
   - `royaltyAmount = (5,000,000,000,000 * 500 / 10000) * 2 = 500,000,000,000`
   - `totalAmount = 10,500,000,000,000` → **10.5 XCH**
5. The user sees **10.5 XCH** and approves; the blockchain deducts **11 XCH** — a 0.5 XCH discrepancy per such offer. [4](#0-3) [5](#0-4) [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
