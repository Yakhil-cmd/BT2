Audit Report

## Title
Unconditional `pop_front()` Before Duplicate Check in `enqueue_transaction` Allows Silent Eviction of Legitimate Pending Bitcoin Transactions — (File: `rs/bitcoin/adapter/src/transaction_store.rs`)

## Summary
`enqueue_transaction` unconditionally evicts the oldest cache entry whenever `len() == TX_CACHE_SIZE` (250), even when the incoming transaction is a duplicate already present in the store. Because the duplicate guard (`or_insert_with`) executes *after* the eviction, a duplicate submission at full capacity causes a net loss of one distinct cached transaction with no error, log warning, or retry. Any canister can reach this path via the public `bitcoin_send_transaction` management API, which is unrestricted per `ic00_permissions.rs`.

## Finding Description
In `rs/bitcoin/adapter/src/transaction_store.rs`, `enqueue_transaction` (lines 83–103):

```rust
if self.transactions.len() == TX_CACHE_SIZE {   // line 92
    self.transactions.pop_front();               // line 97 — unconditional eviction
}
self.transactions
    .entry(txid)
    .or_insert_with(|| TransactionInfo::new(&transaction)); // line 99–101 — no-op for duplicates
``` [1](#0-0) 

The capacity check and `pop_front()` at lines 92–97 execute before the duplicate guard. When the store is at 250 entries and a duplicate `txid` is submitted: `pop_front()` removes the oldest entry, then `or_insert_with` is a no-op because the key already exists. Net result: the store shrinks from 250 to 249, permanently losing the evicted transaction from the adapter's broadcast queue.

The comment at line 82 states the invariant: *"If the transaction is not known, the transaction is added"* — eviction should only occur for genuinely new transactions. The implementation violates this invariant. [2](#0-1) 

**Exploit flow:**
1. Victim submits a Bitcoin transaction V (it enters the cache as the oldest entry).
2. Attacker calls the public `bitcoin_send_transaction` management API 249 times with distinct valid Bitcoin transactions A1…A249. Per `ic00_permissions.rs`, `BitcoinSendTransaction` has `allow_remote_subnet_sender: true` and `allow_only_nns_subnet_sender: false` — no privilege required. [3](#0-2) 
3. These flow through the Bitcoin consensus payload builder → `bitcoin_send_transaction_internal` → adapter gRPC server → `TransactionManagerRequest::SendTransaction` → `enqueue_transaction`. Cache is now full at 250: [V, A1…A249]. [4](#0-3) 
4. Attacker submits a duplicate of A1. `pop_front()` evicts V (the oldest). `or_insert_with` is a no-op for A1. Cache: [A1…A249], size 249. V is gone.
5. `advertise_txids` will never broadcast V again. [5](#0-4) 

The `send_transaction_internal` privileged check at `rs/execution_environment/src/bitcoin.rs:86` gates only the *internal* variant called directly by canisters; the system-level call from the consensus layer is what ultimately reaches the adapter, and the public API is the unrestricted entry point for any canister. [6](#0-5) 

## Impact Explanation
The evicted transaction is silently removed from the `LinkedHashMap` in the adapter's in-memory cache. Because all subnet nodes receive the same ordered sequence of `SendTransaction` requests via IC consensus, the attack simultaneously corrupts the cache on every node running the adapter. The victim's Bitcoin transaction may never reach the Bitcoin P2P network within the 10-minute TTL window. For ckBTC withdrawals, this means the ckBTC is burned but the corresponding BTC transaction is never broadcast — a concrete, permanent loss of user funds. This matches: **High ($2,000–$10,000) — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.**

## Likelihood Explanation
Any canister can call the public `bitcoin_send_transaction` API without special privileges. The attacker needs to submit 249 syntactically valid Bitcoin transactions (non-trivial cycles cost but well within reach of a motivated attacker) and then re-submit one as a duplicate. No admin role, key material, consensus corruption, or social engineering is required. The attack is repeatable: after each duplicate submission, the cache shrinks by one, allowing the attacker to drain additional victim transactions.

## Recommendation
Check for the duplicate *before* deciding to evict:

```rust
pub fn enqueue_transaction(&mut self, raw_tx: &[u8]) {
    if let Ok(transaction) = deserialize::<Transaction>(raw_tx) {
        let txid = transaction.compute_txid();
        if self.transactions.contains_key(&txid) {
            return; // duplicate — no eviction, no insertion
        }
        self.metrics.txn_ops.with_label_values(&["insert", "enqueued"]).inc();
        if self.transactions.len() == TX_CACHE_SIZE {
            self.metrics.txn_ops.with_label_values(&["remove", "pushed_out"]).inc();
            self.transactions.pop_front();
        }
        self.transactions.insert(txid, TransactionInfo::new(&transaction));
    }
}
```

This preserves the FIFO eviction policy while ensuring eviction only occurs for genuinely new transactions.

## Proof of Concept

```rust
#[test]
fn test_duplicate_evicts_distinct_transaction() {
    let mut manager = make_transaction_manager();

    // Fill cache with TX_CACHE_SIZE distinct transactions
    let mut txids = vec![];
    for i in 0..TX_CACHE_SIZE {
        let mut tx = get_transaction();
        tx.lock_time = LockTime::from_height(i.try_into().unwrap()).unwrap();
        let raw = serialize(&tx);
        manager.enqueue_transaction(&raw);
        txids.push(tx.compute_txid());
    }
    assert_eq!(manager.transactions.len(), TX_CACHE_SIZE);

    // Re-submit the second transaction (duplicate) — cache is still at TX_CACHE_SIZE
    let mut dup_tx = get_transaction();
    dup_tx.lock_time = LockTime::from_height(1u32).unwrap();
    let raw_dup = serialize(&dup_tx);
    manager.enqueue_transaction(&raw_dup);

    // BUG: oldest entry (index 0) was evicted; duplicate was not inserted
    assert_eq!(manager.transactions.len(), TX_CACHE_SIZE,
        "FAIL: oldest tx was silently evicted by a duplicate submission");
    assert!(manager.transactions.get(&txids[0]).is_some(),
        "FAIL: oldest tx is gone from the cache");
}
```

This test can be added directly to the existing `#[cfg(test)]` block in `rs/bitcoin/adapter/src/transaction_store.rs` alongside the existing `test_adapter_transaction_cache_full` test. [7](#0-6)
