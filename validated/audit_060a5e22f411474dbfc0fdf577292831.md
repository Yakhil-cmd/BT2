### Title
`exit_to_near_precompile_callback` Fails Under Contract Pause, Permanently Freezing In-Flight wNEAR Unwrap Funds - (`engine/src/contract_methods/connector.rs`)

---

### Summary

When a user initiates a wNEAR-to-NEAR unwrap via the `ExitToNear` precompile, the wNEAR ERC-20 tokens are burned on Aurora and a `near_withdraw` NEAR promise is scheduled asynchronously. If the contract is paused between the initial EVM transaction and the firing of the `exit_to_near_precompile_callback`, the callback fails at the `require_running` guard. The NEAR that was successfully unwrapped from wNEAR accumulates in Aurora's account, the user's wNEAR is permanently burned, and there is no retry or recovery path.

---

### Finding Description

The `ExitToNear` precompile handles wNEAR unwrapping by scheduling a two-step NEAR promise chain:

1. **Step 1**: Call `near_withdraw` on the wNEAR NEP-141 contract (burns wNEAR, sends NEAR to Aurora's account).
2. **Step 2**: Callback `exit_to_near_precompile_callback` on Aurora, which transfers the unwrapped NEAR to the user's target account.

The callback is always scheduled for the wNEAR unwrap path because `transfer_near_args` is `Some(...)`, making `callback_args != default()`: [1](#0-0) 

Inside `exit_to_near_precompile_callback`, the very first substantive check is `require_running(&state)?`: [2](#0-1) 

If the contract is paused (via `pause_contract`) between Step 1 and Step 2, `require_running` returns an error, the callback panics/fails, and the NEAR transfer to the user at lines 215–227 never executes: [3](#0-2) 

The `pause_contract` function sets `state.is_paused = true`: [4](#0-3) 

NEAR callbacks in the NEAR runtime are not automatically retried on failure. Once the callback fails, the NEAR that was unwrapped from wNEAR sits in Aurora's account permanently, and the user's wNEAR is already burned with no re-mint path.

The same issue applies to the `error_refund`-feature ERC-20 exit path: if `ft_transfer` fails and the contract is paused, the refund branch at line 231–239 is never reached: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

For the wNEAR unwrap path (no feature flag required):
- The user's wNEAR ERC-20 tokens are burned atomically in the initial EVM transaction.
- The NEAR unwrapped from wNEAR accumulates in Aurora's NEAR account.
- The user receives nothing and has no mechanism to recover their funds.
- There is no sweep function, no retry mechanism, and no admin recovery path for the stuck NEAR.

The `ExitToNear` precompile address is `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`: [6](#0-5) 

---

### Likelihood Explanation

The condition requires the contract owner to call `pause_contract` while a wNEAR unwrap is in-flight (i.e., after the initial EVM transaction commits but before the NEAR callback fires). This is the same admin-triggered emergency condition as the original report's shelter mechanism. wNEAR unwraps are a normal user operation on Aurora, and any pause during high-activity periods (precisely when emergency pauses are most likely) will affect multiple in-flight unwraps simultaneously.

---

### Recommendation

1. **Remove `require_running` from `exit_to_near_precompile_callback`**, or replace it with a weaker guard that allows already-scheduled callbacks to complete. The callback is `assert_private_call`-protected and cannot be triggered by an external attacker; it only fires as a NEAR callback from a promise the engine itself scheduled.

2. **Alternatively**, before pausing the contract, ensure all in-flight `ExitToNear` callbacks have resolved (analogous to the original report's recommendation to call `massUpdatePools` before making structural changes).

3. **Add a recovery function** callable by the owner that can re-execute a failed `exit_to_near_precompile_callback` given its original arguments, so that funds are not permanently frozen if a pause races with an in-flight callback.

---

### Proof of Concept

**Attack path:**

1. User calls wNEAR ERC-20's `withdrawToNear` on Aurora, which burns wNEAR and triggers the `ExitToNear` precompile.
2. The precompile schedules: `near_withdraw` on wNEAR NEP-141 → callback `exit_to_near_precompile_callback` on Aurora. [7](#0-6) 
3. Admin calls `pause_contract` (emergency scenario). `state.is_paused` is set to `true`. [8](#0-7) 
4. `near_withdraw` executes successfully on the wNEAR contract; NEAR lands in Aurora's account.
5. `exit_to_near_precompile_callback` fires. `require_running(&state)?` returns `Err` because `is_paused == true`. The callback fails. [9](#0-8) 
6. The NEAR transfer to the user's target account (lines 215–227) never executes. The user's wNEAR is permanently burned. The NEAR is permanently stranded in Aurora's account with no recovery path.

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
