### Title
Missing Validation of `app_pk` Identity Element Breaks ElGamal Blinding in CKD Protocol — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` public entry point accepts the application's ElGamal public key `app_pk` (`PublicKey = blstrs::G1Projective`) without checking whether it is the group identity element. When `app_pk` is the identity, the blinding term `y_i · A` in every participant's contribution collapses to zero, causing the coordinator to receive the raw BLS signature `msk · H(pk, app_id)` — the confidential derived key — in the clear. This violates the protocol's core security requirement that no single MPC node can compute the derived secret `s`.

### Finding Description

The CKD protocol is an ElGamal-based scheme. Each participant computes:

```
C_i = S_i + y_i · A     where S_i = x_i · H(pk, app_id)
```

and sends `(λ_i · Y_i, λ_i · C_i)` to the coordinator. The coordinator aggregates:

```
C = msk · H(pk, app_id) + a · Y
```

The TEE app then recovers `sig = C − a · Y = msk · H(pk, app_id)`.

The blinding relies entirely on `A` being a valid non-identity point. The `ckd()` function performs no such check:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,          // ← accepted without any validation
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // participant/coordinator checks only — no app_pk check
``` [1](#0-0) 

Inside `compute_signature_share`, the blinding step is:

```rust
let big_c = big_s + app_pk * y.0;   // if app_pk == identity → big_c == big_s
``` [2](#0-1) 

When `app_pk` is the identity element, `app_pk * y.0 = identity` for every participant, so every `C_i = S_i = x_i · H(pk, app_id)`. The coordinator aggregates these into:

```
C = Σ λ_i · C_i = Σ λ_i · x_i · H(pk, app_id) = msk · H(pk, app_id)
```

The coordinator's output `CKDOutput { big_y, big_c }` now contains `big_c = msk · H(pk, app_id)` — the confidential key — directly readable by the coordinator. [3](#0-2) 

The `PublicKey` type alias is `blstrs::G1Projective`, which freely represents the identity: [4](#0-3) 

No identity check exists anywhere in the `ckd()` initialization path, unlike the group-element checks present in `verify_signature`: [5](#0-4) 

### Impact Explanation

The protocol's stated security requirement is: *"No single node in the MPC network should be capable of computing `s`. This avoids key leakage in the case a single TEE is compromised."* [6](#0-5) 

When `app_pk = identity`, the coordinator receives `C = msk · H(pk, app_id)` in the clear and can directly compute the confidential derived key `s = HKDF(msk · H(pk, app_id))`. This is a **Critical** impact: **disclosure of a confidential derived secret** to a single MPC node, directly violating the no-single-node-leakage guarantee.

### Likelihood Explanation

The `app_pk` value originates from the TEE app's on-chain submission. A malicious or buggy TEE app (or any entity that can submit a CKD request on-chain) can supply `app_pk = G1::identity()`. The MPC library accepts this value without rejection and proceeds to execute the protocol. No privilege beyond the ability to submit a CKD request is required. The `ckd()` function is a public API with no guard on this input.

### Recommendation

Add an explicit identity-element check on `app_pk` at the start of `ckd()`, mirroring the pattern already used in `verify_signature`:

```rust
pub fn ckd(
    ...
    app_pk: PublicKey,
    ...
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // Reject the identity element — it breaks ElGamal blinding
    if bool::from(app_pk.is_identity()) {
        return Err(InitializationError::BadParameters(
            "app_pk must not be the identity element".to_string(),
        ));
    }
    // existing checks follow ...
```

    me: Participant,
    key_pair: KeygenOutput,

    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,

    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
