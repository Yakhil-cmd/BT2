### Title
Unauthorized Burn and Transfer of Any User's Native Tokens via Missing Caller Authorization in Bank Precompile — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `BankContract` precompile at address `0x0000000000000000000000000000000000000064` exposes `burn(address, uint256)` and `transfer(address, address, uint256)` methods that operate on an arbitrary victim address supplied as a calldata argument. Neither method verifies that `contract.Caller()` is authorized to spend from that address. Any unprivileged EVM contract can therefore burn or steal native `evm/<callerAddress>`-denom tokens from any holder of those tokens without their consent.

---

### Finding Description

In `BankContract.Run`, the shared `MintMethodName`/`BurnMethodName` branch unpacks the first argument as `recipient` (the address whose tokens are burned) and derives the denom from `contract.Caller()`:

```go
recipient := args[0].(common.Address)          // victim address — caller-supplied
...
denom := EVMDenom(contract.Caller())            // "evm/<callerContract>"
...
// BURN path — no check that contract.Caller() == recipient
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
``` [1](#0-0) 

The `TransferMethodName` branch similarly accepts a caller-supplied `sender` address and moves funds from it without any authorization check:

```go
sender := args[0].(common.Address)             // victim address — caller-supplied
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
...
// TRANSFER path — no check that contract.Caller() == sender
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [2](#0-1) 

The only guard present is `checkBlockedAddr`, which only prevents sending to module-blocked addresses and does nothing to verify the caller's authority over the `sender`/`recipient` being debited. [3](#0-2) 

The interface definition confirms these are open, unpermissioned entry points callable by any contract: [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized burn and transfer of native assets.**

The denom affected is `evm/<callerContractAddress>`. A user who holds tokens of that denom (e.g., received via `mint` from the same contract, or via a `transfer`) can have those tokens burned or stolen by the contract at any time without their consent. Because the bank precompile is the canonical mechanism for CRC20-style contracts to manage their native Cosmos SDK token representation, this allows any deployed contract to:

1. **Burn** any holder's native `evm/<contract>` tokens — permanent, irreversible loss.
2. **Transfer** any holder's native `evm/<contract>` tokens to an attacker-controlled address — theft.

This directly matches the Critical impact category: *unauthorized burn, transfer, or balance/accounting change for precompile-controlled assets*.

---

### Likelihood Explanation

**High.** No preconditions beyond deploying a contract and having a victim hold tokens of the attacker's denom. The bank precompile is publicly accessible at a fixed address (`0x0000000000000000000000000000000000000064`) to any EVM contract. No admin key, governance vote, or special permission is required.

---

### Recommendation

In both the `burn` and `transfer` branches, assert that the EVM caller is authorized to debit the source address:

- **`burn`**: require `contract.Caller() == recipient` (the caller can only burn their own tokens), or implement an allowance/approval mechanism.
- **`transfer`**: require `contract.Caller() == sender` (the caller can only transfer from themselves), or check an on-chain allowance granted by `sender` to `contract.Caller()`.

```go
// burn — add before the bankKeeper calls:
if contract.Caller() != recipient {
    return nil, errors.New("caller is not authorized to burn tokens from this address")
}

// transfer — add before the bankKeeper calls:
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer tokens from this address")
}
```

---

### Proof of Concept

1. Attacker deploys `MaliciousContract` at address `0xATTACKER`.
2. `MaliciousContract` calls `bank.mint(victimAddress, 1000)` → victim now holds 1000 `evm/0xATTACKER` native tokens.
3. `MaliciousContract` calls `bank.burn(victimAddress, 1000)` → the precompile executes `SendCoinsFromAccountToModule` + `BurnCoins` against `victimAddress` with no authorization check. Victim's 1000 tokens are permanently destroyed.
4. Alternatively at step 3: `MaliciousContract` calls `bank.transfer(victimAddress, attackerEOA, 1000)` → victim's tokens are moved to the attacker.

No signature, approval, or governance action from the victim is required at any step.

	to, err := sdk.AccAddressFromBech32(addr.String())

		return err
	}
	if bc.bankKeeper.BlockedAddr(to) {
		return errorsmod.Wrapf(errortypes.ErrUnauthorized, "%s is not allowed to receive funds", to.String())

	return nil
