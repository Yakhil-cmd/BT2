Audit Report

## Title
ckETH Minter Indefinitely Queues Withdrawal Requests Without Reimbursement on Persistent InsufficientTransactionFee - (`File: rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary
When `create_transactions_batch` encounters `CreateTransactionError::InsufficientTransactionFee` for a `WithdrawalRequest::CkEth` request, it calls `reschedule_withdrawal_request` and moves the request to the back of the queue with no reimbursement scheduled and no retry limit. The user's ckETH has already been irreversibly burned on the IC ledger. If Ethereum gas fees remain elevated above the withdrawal amount for an extended or indefinite period, the user's funds are locked with no recovery path — no cancellation, no timeout, and no reimbursement event type exists for this case.

## Finding Description
**Burn path** (`rs/ethereum/cketh/minter/src/main.rs`, lines 291–336): `withdraw_eth` checks `amount >= cketh_minimum_withdrawal_amount`, then calls `client.burn_from(...)`. On success, it emits `EventType::AcceptedEthWithdrawalRequest` and enqueues the request. The burn is irreversible at this point.

**Transaction creation path** (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 1122–1134): `create_transaction` for `WithdrawalRequest::CkEth` computes `max_transaction_fee = gas_fee_estimate.to_price(gas_limit).max_transaction_fee()` and returns `Err(CreateTransactionError::InsufficientTransactionFee)` if `withdrawal_amount < max_transaction_fee`.

**Error handler** (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 281–291): The match arm for `CreateTransactionError::InsufficientTransactionFee` calls only `reschedule_withdrawal_request(request)`. No `ReimbursementRequest` is constructed, no `EventType::FailedErc20WithdrawalRequest` (or any equivalent ckETH event) is emitted, and `process_reimbursement` is never invoked for this request.

**Missing event type** (`rs/ethereum/cketh/minter/src/state/event.rs`, lines 17–174): The `EventType` enum contains `FailedErc20WithdrawalRequest` (variant 20) which triggers `record_reimbursement_request` in `apply_state_transition` (audit.rs lines 146–153), but there is no `FailedEthWithdrawalRequest` variant. The reimbursement infrastructure for ckETH exists (`ReimbursementIndex::CkEth`, `process_reimbursement` at withdraw.rs lines 46–130) but is never triggered from the `InsufficientTransactionFee` path.

**Contrast with ckERC20** (`rs/ethereum/cketh/minter/src/main.rs`, lines 506–531): When the ckERC20 burn fails, the minter explicitly constructs a `ReimbursementRequest` and emits `EventType::FailedErc20WithdrawalRequest`, which feeds into `process_reimbursement`. No equivalent path exists for the ckETH `InsufficientTransactionFee` case.

**Gas limit constant** (`rs/ethereum/cketh/minter/src/withdraw.rs`, line 43): `CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 21_000`. For a minimum withdrawal amount of 0.03 ETH (30,000,000,000,000,000 wei), the fee threshold is ~1,428 Gwei. For lower configured minimums the threshold is proportionally lower and more reachable during congestion events.

## Impact Explanation
This is a concrete user-funds loss scenario within the ckETH Chain Fusion integration. A user's ckETH is burned on the IC ledger (permanently reducing their balance), no ETH is ever sent on Ethereum, and no ckETH is ever minted back. The request loops in the pending queue indefinitely with no timeout, no cancellation endpoint, and no reimbursement event. This matches the allowed impact: **Significant Chain Fusion / ck-token security impact with concrete user or protocol harm** (High, $2,000–$10,000). The loss per affected user equals their full withdrawal amount.

## Likelihood Explanation
Any unprivileged IC principal can call `withdraw_eth`. The trigger requires gas fees to exceed the user's withdrawal amount at the time `create_transactions_batch` runs. With `CKETH_WITHDRAWAL_TRANSACTION_GAS_LIMIT = 21_000`, the required gas price scales inversely with the configured minimum withdrawal amount. The `cketh_minimum_withdrawal_amount` is a static deployment-time parameter that does not track real-time gas. The gas fee estimate is fetched live via `lazy_refresh_gas_fee_estimate` from Ethereum JSON-RPC providers. During extreme Ethereum congestion (historical peaks: 500–1,000 Gwei), a sufficiently low minimum withdrawal amount makes this condition reachable. No privileged access is required. The condition is self-reinforcing: once triggered, every timer tick re-queues the request without relief.

## Recommendation
1. **Immediate fix**: In the `CreateTransactionError::InsufficientTransactionFee` match arm in `create_transactions_batch` (`withdraw.rs` lines 281–291), add a retry counter or maximum queue age to the `EthWithdrawalRequest`. Once the threshold is exceeded, construct a `ReimbursementRequest` and emit a new `EventType::FailedEthWithdrawalRequest` event (mirroring `FailedErc20WithdrawalRequest`) so `process_reimbursement` mints back the burned ckETH.
2. **Structural**: Add a `FailedEthWithdrawalRequest(ReimbursementRequest)` variant to `EventType` and a corresponding `apply_state_transition` arm that calls `record_reimbursement_request` with `ReimbursementIndex::CkEth`.
3. **Preventive**: Dynamically gate `withdraw_eth` against the current gas fee estimate so the burn is rejected before it occurs if the amount cannot cover fees at acceptance time.

## Proof of Concept
1. Deploy the minter with `cketh_minimum_withdrawal_amount = X`.
2. Alice calls `withdraw_eth(amount = X, recipient = addr)`. The minter burns `X` ckETH from Alice's ledger account (`main.rs` lines 301–312). `AcceptedEthWithdrawalRequest` is emitted and the request is enqueued.
3. Simulate a gas spike such that `gas_fee_estimate.to_price(21_000).max_transaction_fee() > X` (achievable in a PocketIC or integration test by mocking the JSON-RPC gas fee response).
4. Trigger `process_retrieve_eth_requests` → `create_transactions_batch`. Observe that `create_transaction` returns `Err(CreateTransactionError::InsufficientTransactionFee)` (`transactions/mod.rs` lines 1127–1133).
5. Observe that `reschedule_withdrawal_request` is called (`withdraw.rs` line 290) and no `ReimbursementRequest` is recorded in state.
6. Verify via `reimbursement_requests_iter()` that the iterator is empty — `process_reimbursement` has nothing to process.
7. Repeat step 4 indefinitely; Alice's ckETH balance remains reduced and no ETH is sent. Confirm no `ReimbursedEthWithdrawal` event is ever emitted.
8. Contrast: repeat with `withdraw_erc20` and a failing ckERC20 burn; observe `FailedErc20WithdrawalRequest` is emitted and `process_reimbursement` mints back the ckETH fee (`main.rs` lines 506–531).
