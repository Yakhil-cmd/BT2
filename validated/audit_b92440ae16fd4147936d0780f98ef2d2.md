### Title
Pre-EIP-155 Legacy Transactions Accepted Without Chain ID Validation Enables Cross-Chain Replay - (File: engine/src/engine.rs, engine-transactions/src/legacy.rs)

### Summary
Aurora Engine's `submit` entry point accepts legacy (pre-EIP-155) Ethereum transactions signed with `v = 27` or `v = 28`. For these transactions, no chain ID is embedded in the signature, and the engine's chain ID validation is gated on `transaction.chain_id` being `Some`. Because pre-EIP-155 transactions yield `chain_id = None`, the guard is silently skipped, allowing any such transaction to be replayed verbatim on Aurora from Ethereum mainnet (or any other EVM chain), and vice versa.

### Finding Description
In `engine-transactions/src/legacy.rs`, `LegacyEthSignedTransaction::sender()` decodes the `v` field:

```rust
let (chain_id, rec_id) = match self.v {
    0..=26 | 29..=34 => return Err(Error::InvalidV),
    27..=28 => (None, ...),   // pre-EIP-155: no chain ID
    _ => (Some((self.v - 35) / 2), ...),
};
``` [1](#0-0) 

`chain_id()` correspondingly returns `None` for `v ∈ {27, 28}`. [2](#0-1) 

In `engine/src/engine.rs`, `submit_with_alt_modexp` validates the chain ID only when it is present:

```rust
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [3](#0-2) 

When `transaction.chain_id` is `None` (pre-EIP-155), the entire guard is bypassed and the transaction is executed unconditionally, regardless of which chain it was originally signed for.

### Impact Explanation
**Critical — Direct theft of user funds.**

A pre-EIP-155 transaction signed by Alice on Ethereum mainnet (transferring ETH to Bob) can be submitted byte-for-byte to Aurora's `submit` method. If Alice holds the same address on Aurora with a matching nonce and sufficient balance, the transfer executes on Aurora without Alice's knowledge or consent, draining her Aurora funds. The reverse direction (Aurora → Ethereum) is equally valid for any EVM chain that still accepts pre-EIP-155 transactions.

### Likelihood Explanation
**Medium.** Pre-EIP-155 transactions (v=27/28) are still produced by certain wallets, hardware signers, and tooling that predates EIP-155 or that explicitly omit chain ID for chain-agnostic signing. Aurora Engine explicitly accepts them (the `27..=28` branch is not an error path). An attacker only needs to observe a broadcast pre-EIP-155 transaction on any public mempool and replay it on Aurora before the victim's nonce advances.

### Recommendation
**Short term:** Reject pre-EIP-155 legacy transactions (v=27/28) at the `submit` entry point by treating a missing chain ID as an error rather than a pass-through. Add an explicit check:

```rust
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

**Long term:** Audit all transaction-kind parsers (`EthTransactionKind`) to confirm that every accepted variant carries a chain ID that is validated against the engine state before execution.

### Proof of Concept
1. Alice signs a pre-EIP-155 ETH transfer on Ethereum mainnet (v=27): `nonce=N, to=Bob, value=1 ETH`.
2. Alice has the same address on Aurora with nonce=N and ≥1 ETH balance.
3. Attacker copies the raw RLP-encoded transaction bytes from the Ethereum mempool.
4. Attacker calls Aurora's `submit` NEAR method with those bytes as input.
5. `LegacyEthSignedTransaction::chain_id()` returns `None`; the chain ID guard in `submit_with_alt_modexp` is skipped; `check_nonce` passes (nonce matches); the transfer executes on Aurora, crediting Bob and debiting Alice without her consent. [3](#0-2) [4](#0-3)

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);
        let message_hash = sdk::keccak(rlp_stream.as_raw());
        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
            .map_err(|_| Error::EcRecover)
    }
