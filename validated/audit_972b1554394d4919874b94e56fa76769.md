### Title
Staker Can Immediately Increase Commission Before Reward Distribution, Stealing Delegators' Unclaimed Yield — (`File: src/staking/staking.cairo`)

---

### Summary

The `set_commission` function allows a staker to immediately increase their commission rate to any value up to `max_commission` when an active `CommissionCommitment` exists. Because the new commission is written to storage instantly and read at the next reward distribution, a staker can front-run reward distributions to steal all pool rewards from delegators.

---

### Finding Description

The internal `update_commission` function writes the new commission directly to storage with no delay or epoch-offset: [1](#0-0) 

At reward distribution time, `calculate_staker_pools_rewards` reads the commission live from storage: [2](#0-1) 

The `split_rewards_with_commission` utility then applies the current commission to split rewards between the staker and delegators: [3](#0-2) 

When a `CommissionCommitment` is active, `update_commission` only enforces `commission <= max_commission` and `commission != old_commission` — it does **not** require the commission to decrease: [4](#0-3) 

`set_commission_commitment` allows `max_commission` to be set as high as `COMMISSION_DENOMINATOR` (10000 = 100%): [5](#0-4) 

The codebase itself acknowledges this gap: [6](#0-5) 

**Attack sequence:**

1. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
2. Staker immediately calls `set_commission(10000)` — commission jumps to 100% in the same block.
3. The next call to `update_rewards_from_attestation_contract` (attestation path) or `update_rewards` (consensus path) distributes rewards using the new 100% commission.
4. `split_rewards_with_commission` returns `(pool_rewards, 0)` — all pool rewards go to the staker as commission; delegators receive zero.
5. Delegators cannot exit immediately; the `exit_wait_window` (default 1 week) traps their stake. [7](#0-6) 

---

### Impact Explanation

Delegators lose all unclaimed yield for the affected distribution period. The staker captures 100% of pool rewards as commission. Because the exit wait window prevents immediate withdrawal, delegators remain exposed for at least one week after the commission increase. This constitutes **theft of unclaimed yield** (High severity).

---

### Likelihood Explanation

Any staker who operates a delegation pool can execute this attack in two transactions. The only prerequisite is that no active commitment already exists. The commitment is public on-chain, but the window between commitment creation and commission increase can be a single block, giving delegators no practical time to react before the next reward distribution. The attack is repeatable after each commitment expires.

---

### Recommendation

Commission **increases** should not take effect immediately. Apply the same epoch-offset pattern used for balance changes elsewhere in the protocol: write the new (higher) commission with a key of `current_epoch + K`, so it only becomes effective K epochs in the future. This gives delegators time to observe the pending increase and exit before it applies. Commission **decreases** can remain immediate, as they benefit delegators.

---

### Proof of Concept

```
// Staker has commission = 500 (5%), active delegation pool, delegators have accrued rewards.

// Step 1 – set commitment allowing 100% commission
staking.set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1);

// Step 2 – immediately raise commission to 100%
staking.set_commission(commission: 10000);
// update_commission writes 10000 to storage instantly (line 1600)

// Step 3 – next reward distribution (attestation or update_rewards)
// calculate_staker_pools_rewards reads commission = 10000
// split_rewards_with_commission(pool_rewards_including_commission, 10000)
//   → commission_rewards = pool_rewards_including_commission * 10000 / 10000 = all rewards
//   → pool_rewards = 0
// Delegators receive 0 STRK; staker receives all pool rewards as commission.

// Step 4 – delegators attempt to exit but must wait exit_wait_window (≥ 1 week)
```

pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}
