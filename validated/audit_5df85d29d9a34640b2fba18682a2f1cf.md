### Title
Missing Zero `msg_hash` Validation in OT-Based ECDSA `sign` Allows Signing of Zero-Hash Messages and Presign Secret Leakage - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` does not validate that `msg_hash` is non-zero before initiating the signing protocol. The sibling robust ECDSA `sign` function in `src/ecdsa/robust_ecdsa/sign.rs` explicitly rejects a zero `msg_hash` with the comment "to avoid potential split view attacks." The missing check in the OT-based path allows a malicious coordinator or caller to drive the protocol with `msg_hash = 0`, producing a valid threshold ECDSA signature for the zero message and enabling the coordinator to isolate each participant's `sigma_i` presign secret share from the received signature shares.

---

### Finding Description

The robust ECDSA `sign` function explicitly guards against a zero message hash: [1](#0-0) 

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
```

The OT-based ECDSA `sign` function performs several input validations (participant count, threshold bounds, coordinator presence) but contains **no equivalent check**: [2](#0-1) 

The protocol proceeds directly to `fut_wrapper` with an unchecked `msg_hash: Scalar`. The inner `compute_signature_share` function computes each participant's share as: [3](#0-2) 

```rust
let k_i = lambda * presignature.k;
let sigma_i = lambda * presignature.sigma;
let r = x_coordinate(&presignature.big_r);
Ok(msg_hash * k_i + r * sigma_i)
```

When `msg_hash = 0`, this collapses to:

```
s_i = r * sigma_i
```

The `k_i` term vanishes entirely. The coordinator receives `s_j` from every other participant and, since `r` is public, can trivially compute:

```
sigma_j = s_j / r
```

for every participant `j`. These are the Lagrange-weighted shares of the presign secret `sigma = k * x` (nonce × private key). The coordinator already holds their own `sigma_me` share, so they now possess **all** Lagrange-weighted shares of `sigma` and can reconstruct `sigma = k * x` in full.

The final verification in `do_sign_coordinator` does not catch this: [4](#0-3) 

```rust
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed("signature failed to verify".to_string()));
}
```

The signature `(r, s)` with `s = r * sigma` is mathematically valid for `msg_hash = 0`, so verification passes and the protocol returns a valid signature.

---

### Impact Explanation

**Two distinct impacts arise:**

1. **Unauthorized valid threshold signature for attacker-chosen input (CRITICAL):** A malicious coordinator passes `msg_hash = 0` to `sign()`. All honest participants execute the protocol and produce shares. The coordinator assembles a valid ECDSA signature for the zero message. Honest participants believe they participated in a legitimate signing session, but the resulting signature is for an attacker-controlled (zero) message hash — a message they never intended to authorize.

2. **Disclosure of presign secrets (CRITICAL):** With `msg_hash = 0`, the coordinator receives `s_j = r * sigma_j` and can compute `sigma_j = s_j / r` for every participant. The `sigma_j` values are Lagrange-weighted shares of `sigma = k * x`, a presign secret. The coordinator reconstructs `sigma = k * x` in full. While `sigma` alone does not directly yield `x` (since `k` is not known in scalar form), this constitutes disclosure of presign secret material that the protocol is designed to keep hidden from any single party, and it enables further attacks if the coordinator can correlate multiple sessions or obtain `k` through other means.

---

### Likelihood Explanation

The entry path is direct and requires no cryptographic break. Any party that calls `sign()` — including the coordinator role — controls the `msg_hash` argument. The coordinator is a participant in the OT-based signing flow and is the natural entity to initiate signing. Passing `msg_hash = 0` is a single-parameter substitution. The robust ECDSA path explicitly documents and blocks this exact scenario, confirming the developers are aware of the risk class; the OT-based path simply lacks the guard.

---

### Recommendation

Add the same zero-hash guard present in the robust ECDSA `sign` function to the OT-based ECDSA `sign` function, immediately after the existing participant-count and threshold checks:

```rust
// In src/ecdsa/ot_based_ecdsa/sign.rs, inside `sign()`, before creating the protocol:
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks and presign secret leakage".to_string(),
    ));
}
```

This mirrors the guard already present in `src/ecdsa/robust_ecdsa/sign.rs` at lines 91–95 and closes the inconsistency between the two signing paths.

---

### Proof of Concept

**Setup:** Three participants run DKG to obtain key shares. They run presign to obtain `PresignOutput { big_r, k, sigma }` for each participant. The coordinator then calls:

```rust
// Attacker-controlled call with msg_hash = Scalar::ZERO
sign(
    &participants,
    coordinator,
    threshold,
    me,
    public_key,
    rerandomized_presign,
    Scalar::ZERO,   // <-- zero hash, no check rejects this
)
```

**During protocol execution** (`compute_signature_share`):
```
s_i = 0 * (lambda_i * k_i) + r * (lambda_i * sigma_i)
    = r * lambda_i * sigma_i
```

**Coordinator receives** `s_j` from each participant `j ≠ coordinator`:
```
sigma_j_recovered = s_j / r   // r is public; division is field inversion + multiply
```

**Coordinator reconstructs** the full presign secret:
```
sigma = sum_j(sigma_j_recovered) + sigma_coordinator = k * x
```

**Protocol completes** successfully: `sig.verify(&public_key, &Scalar::ZERO)` returns `true`, and `Ok(Some(sig))` is returned — a valid threshold ECDSA signature for the zero message, with presign secrets disclosed to the coordinator. [2](#0-1) [1](#0-0) [5](#0-4) [4](#0-3)

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<Scalar, ProtocolError> {
    // Round 1
    // Linearize ki
    // Spec 1.1
    let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
    let k_i = lambda * presignature.k;

    // Linearize sigmai
    // Spec 1.2
    let sigma_i = lambda * presignature.sigma;

    // Compute si = h * ki + Rx * sigmai
    // Spec 1.3
    let r = x_coordinate(&presignature.big_r);
    Ok(msg_hash * k_i + r * sigma_i)
