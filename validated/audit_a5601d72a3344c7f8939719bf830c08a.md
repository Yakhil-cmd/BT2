### Title
Attacker-Controlled `sender_address` in `skip_stateful_validations` Bypasses Signature Verification for Any Address with a Pending Deploy-Account Transaction - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function uses the attacker-controlled `sender_address` field of an incoming invoke transaction to query `account_tx_in_pool_or_recent_block`. When that query returns `true`, the function returns `true` (skip), which propagates to `run_validate_entry_point` setting `validate: false` in `ExecutionFlags`. The blockifier's `StatefulValidator::perform_validations` then returns `Ok(())` immediately after `perform_pre_validation_stage` — without ever calling the `__validate__` entry point — so the account signature is never verified. Any attacker can trigger this path for any victim address that has a deploy-account transaction in the mempool, submitting an invoke with an arbitrary/invalid signature that is admitted without cryptographic verification.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // tx.sender_address() is fully attacker-controlled
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}
```

The three conditions that must hold for the skip to fire are:

| Condition | Who controls it |
|---|---|
| `tx` is `Invoke` | Attacker |
| `tx.nonce() == 1` | Attacker |
| `account_nonce == 0` | State of `tx.sender_address()` — also attacker-chosen |
| `account_tx_in_pool_or_recent_block(tx.sender_address())` | Attacker chooses the address |

All four conditions are satisfiable by the attacker by pointing `sender_address` at any victim address B that (a) has not yet been deployed (nonce = 0 in state) and (b) has a deploy-account transaction already in the mempool.

`perform_pre_validation_stage` checks nonce ordering and fee bounds — neither of which verifies the cryptographic signature. The `__validate__` entry point, which is the only place the account contract verifies the signature, is skipped entirely.

**`account_tx_in_pool_or_recent_block` checks only address presence, not ownership:**

    self.state.contains_account(account_address)

There is no check that the entity querying this function is the legitimate owner of `account_address`. Any caller can pass any address.

### Impact Explanation

An attacker can submit an invoke transaction for any victim address B that has a pending deploy-account in the mempool, using an arbitrary/invalid signature. The gateway admits the transaction without signature verification. Concrete consequences:

1. **Invalid transactions admitted to the mempool** — the gateway's stateful validation invariant (every admitted invoke has passed `__validate__`) is broken.
2. **Front-running / displacement** — the mempool enforces one transaction per `(address, nonce)` slot. The attacker's invalid invoke with `nonce=1` for address B can displace the victim's legitimate invoke with `nonce=1`, preventing the victim from using the deploy-account + invoke UX flow.
3. **Block space waste** — the invalid invoke will be included in a block and revert during execution (when `__validate__` IS called by the batcher), consuming block resources.

This matches the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The attack requires no special privilege. The attacker only needs to:
1. Observe that some address B has a deploy-account transaction in the mempool (observable from the public mempool API or network).
2. Submit a single invoke transaction with `sender_address = B`, `nonce = 1`, and any signature.

The deploy-account + invoke UX flow is an explicitly supported and documented feature, so victim addresses in this state will exist regularly in production.

### Recommendation

The `skip_stateful_validations` function must verify that the caller is the legitimate owner of `sender_address`, not merely that some transaction for that address exists in the pool. Since the account contract does not yet exist on-chain, a direct `__validate__` call is impossible, but ownership can be established by one of the following approaches:

1. **Require the deploy-account and invoke to arrive in the same gateway request** (atomic submission), so the gateway can cryptographically link them without relying on a mempool state query.
2. **Verify the invoke signature against the expected public key derived from the deploy-account's constructor calldata** (applicable when the account class uses a known key-extraction convention).
3. **Restrict the skip to transactions submitted by the same source as the deploy-account** (e.g., track a session token or require the invoke to commit to the deploy-account's transaction hash in its calldata).

At minimum, add a check that the deploy-account transaction already in the pool was submitted for the exact same `sender_address` as the incoming invoke, and that no other invoke with `nonce=1` for that address is already present, to prevent displacement attacks.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
impl<S: StateReader> StatefulValidatorTrait for StatefulValidator<S> {
    fn validate(&mut self, account_tx: AccountTransaction) -> StatefulValidatorResult<()> {
        self.perform_validations(account_tx)
    }
}

/// Manages state related transaction validations for pre-execution flows.
pub struct StatefulValidator<S: StateReader> {
    tx_executor: TransactionExecutor<S>,
}

impl<S: StateReader> StatefulValidator<S> {
    pub fn create(state: CachedState<S>, block_context: BlockContext) -> Self {
        let tx_executor =
            TransactionExecutor::new(state, block_context, TransactionExecutorConfig::default());
        Self { tx_executor }
    }

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
```
