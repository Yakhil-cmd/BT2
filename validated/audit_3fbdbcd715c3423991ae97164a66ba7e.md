### Title
ETH Permanently Locked in `exit_to_near::ADDRESS` When NEAR-Side `ft_transfer` Fails Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary

When a user calls the `ExitToNear` precompile to bridge ETH (base token) to NEAR, the ETH is transferred from the user's EVM balance into the `exit_to_near::ADDRESS` precompile address. A NEAR-side `ft_transfer` promise is then scheduled. If that promise fails and the `error_refund` compile-time feature is **not** enabled, no callback is registered to return the ETH, leaving it permanently locked in `exit_to_near::ADDRESS` with no recovery path.

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` function constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is disabled
    transfer_near: transfer_near_args,
};
```

For a base ETH exit (`ExitToNearParams::BaseToken`), `transfer_near_args` is also `None` (see `exit_base_token_to_near` returning `None` for the fifth tuple element). This means `callback_args == ExitToNearPrecompileCallbackArgs::default()`, so the promise is constructed as a bare `PromiseArgs::Create` with **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

If the NEAR-side `ft_transfer` fails (e.g., the recipient account is not registered with the eth-connector), there is no callback to invoke `refund_on_error`. The ETH that was credited to `exit_to_near::ADDRESS` during the EVM execution remains there permanently. The only code path that can move ETH out of `exit_to_near::ADDRESS` is `engine::refund_on_error`, which is exclusively called from `exit_to_near_precompile_callback` when `args.refund` is `Some(...)` — a branch that is unreachable when `error_refund` is disabled.

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH deducted from the user's EVM balance and credited to `exit_to_near::ADDRESS` is irrecoverable when `error_refund` is disabled and the NEAR-side `ft_transfer` fails. There is no admin function, no governance path, and no user-callable method that can drain `exit_to_near::ADDRESS` back to the original sender. The `refund_on_error` function exists for this purpose but is gated behind the `error_refund` feature and is only reachable via the callback that is never registered in this code path.

### Likelihood Explanation

**Medium.** The `ft_transfer` to the eth-connector will fail whenever the NEAR recipient account is not registered as a storage depositor with the eth-connector contract. This is a realistic condition: any user who specifies a NEAR account that has never interacted with the eth-connector will trigger the failure. The `error_refund` feature is a compile-time opt-in; if the production WASM is built without it, every such failed exit permanently destroys the user's ETH.

### Recommendation

Ensure the `error_refund` feature is unconditionally enabled in all production builds, or restructure the base-ETH exit path so that a refund callback is always registered regardless of the feature flag. The callback should invoke `refund_on_error` to transfer ETH from `exit_to_near::ADDRESS` back to the original sender whenever the NEAR-side promise fails.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Fund an EVM address `A` with ETH.
3. From address `A`, call the `ExitToNear` precompile (flag `0x00`) with `value = X ETH` and a NEAR recipient account that is **not** registered with the eth-connector.
4. Observe: `X ETH` is deducted from `A`'s EVM balance and credited to `exit_to_near::ADDRESS`. The NEAR-side `ft_transfer` fails. No callback fires. `exit_to_near::ADDRESS` now holds `X ETH` with no mechanism to recover it.
5. Confirm: `eth_balance_of(A) == initial - X` and `eth_balance_of(exit_to_near::ADDRESS) == X`, permanently.
