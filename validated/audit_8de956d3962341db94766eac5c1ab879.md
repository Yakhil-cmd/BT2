### Title
`StateDB.Commit()` Permanently Deletes Recreated Accounts — (`fvm/evm/emulator/state/stateDB.go`)

---

### Summary

`StateDB.Commit()` contains a logic flaw: when `CreateAccount` is called on an address that already exists in the base view, it internally calls `SelfDestruct` (populating `toBeDestructed`) **and** sets `created[addr]`. At commit time, `Commit()` sees `HasSelfDestructed == true`, calls `DeleteAccount`, sets `deleted = true`, and then unconditionally `continue`s — permanently skipping the `IsCreated` branch that would have recreated the account. The account is erased from the base view with no recreation and no balance carry-over.

---

### Finding Description

**Step 1 — `DeltaView.CreateAccount` on an existing address**

When `CreateAccount(addr)` is called on an address that already exists (e.g., an EOA with balance): [1](#0-0) 

- `HasSelfDestructed(addr)` returns false → `SelfDestruct(addr)` is called, writing `toBeDestructed[addr] = originalBalance` and zeroing `balances[addr]`.
- `d.balances[addr] = balance` then restores the carried-over balance.
- `d.recreated[addr]` and `d.created[addr]` are both set.

After this call, the delta view simultaneously has:
- `toBeDestructed[addr]` = original balance (non-zero)
- `created[addr]` = set
- `recreated[addr]` = set

**Step 2 — `StateDB.Commit()` processes the address** [2](#0-1) 

The commit loop for `addr`:
1. `db.HasSelfDestructed(addr)` → **true** (because `toBeDestructed[addr]` is set) → `baseView.DeleteAccount(addr)` is called, `deleted = true`.
2. `if deleted { continue }` — **unconditionally skips** the rest of the loop body.
3. `db.IsCreated(addr)` (line 495) and `baseView.CreateAccount(...)` (line 496) are **never reached**.

The account is deleted from the base view and never recreated. The `recreated` map set in `DeltaView` is never consulted by `Commit()`.

---

### Impact Explanation

- **Account erasure**: An existing EOA or contract at address X is permanently deleted from the base view after a contract deployment to X in the same transaction.
- **Balance destruction**: The balance that `CreateAccount` correctly carried over into `balances[addr]` is never written to the base view — it is silently lost.
- **Contract non-deployment**: The new contract code and nonce are never persisted. The deployment appears to succeed at the EVM level (no revert), but the resulting state is corrupted.

This violates the EVM invariant (and the explicit comment in `CreateAccount`: *"carry over the balance and reset the data from the original account"*) that a recreated account must survive `Commit()` with its new code and carried-over balance.

---

### Likelihood Explanation

The path is directly reachable via a standard EVM transaction:

1. Fund address X (making it an EOA in the base view).
2. Submit a `CREATE2` deployment transaction targeting address X (the salt and deployer address are fully attacker-controlled).
3. The EVM calls `StateDB.CreateAccount(X)` during contract creation.
4. `Commit()` deletes X and never recreates it.

No privileged access, no staked node control, and no social engineering is required. Any user who can submit EVM transactions on Flow can trigger this.

---

### Recommendation

In `StateDB.Commit()`, change the guard at line 486 from an unconditional skip to one that also checks whether the account was recreated:

```go
// stateDB.go, inside the Commit() loop
if deleted && !db.IsCreated(addr) {
    continue
}
```

This ensures that when an account is both self-destructed and recreated in the same transaction (the `recreated` case), the `IsCreated` branch is still reached and `baseView.CreateAccount(...)` is called with the correct carried-over balance, nonce, and new code.

---

### Proof of Concept

```
State before tx:
  baseView[X] = EOA { balance: 100 ETH, nonce: 3, code: nil }

Transaction:
  CREATE2 targeting address X, deployer D, salt S
  → EVM calls StateDB.CreateAccount(X)
    → DeltaView: toBeDestructed[X]=100, balances[X]=100, created[X]=set, recreated[X]=set
  → EVM sets code, nonce=1 via SetCode/SetNonce
  → StateDB.Commit() called:
      HasSelfDestructed(X) == true → baseView.DeleteAccount(X), deleted=true
      if deleted { continue }   ← IsCreated branch never reached
      baseView.CreateAccount never called

State after tx:
  baseView[X] = <does not exist>   ← WRONG: should exist with new code + 100 ETH
```

Assert: `db.Exist(X)` returns `false` after commit. Balance 100 ETH is gone. Contract code is absent. [3](#0-2) [4](#0-3)
