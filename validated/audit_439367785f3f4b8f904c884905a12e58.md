### Title
Missing Non-Zero Validation for `gas_per_pubdata_limit` in L1→L2 Transactions Enables Zero-Cost Pubdata - (File: `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

`AbiEncodedTransaction::validate_structure()` enforces that `gas_per_pubdata_limit` is **zero** for non-L1 transaction types, but never enforces that it is **non-zero** for L1→L2 (`0x7F`) transactions. An L1→L2 transaction submitted with `gas_per_pubdata_limit = 0` passes all ZKsync OS validation and is executed with `native_per_pubdata = 0`, meaning the transaction pays nothing for any pubdata it generates.

---

### Finding Description

**Root cause — `validate_structure()` omits the required non-zero check:**

```rust
// gas_per_pubdata_limit should be zero for non L1 transactions
match tx_type {
    Self::UPGRADE_TX_TYPE | Self::L1_L2_TX_TYPE => {}   // ← no check at all for L1 types
    _ => {
        if self.gas_per_pubdata_limit.read() != 0 {
            return Err(());
        }
    }
}
``` [1](#0-0) 

The `L1_L2_TX_TYPE` arm is a no-op: any value of `gas_per_pubdata_limit`, including `0`, is accepted without error.

**Propagation — the zero value flows directly into resource accounting:**

In `process_l1_transaction`, the field is read and passed to `prepare_and_check_resources` without any guard:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();   // may be 0
``` [2](#0-1) 

Inside `prepare_and_check_resources`, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = (gas_per_pubdata as u64)
    .checked_mul(native_per_gas)
    .unwrap_or_else(|| { ... u64::MAX });
``` [3](#0-2) 

When `gas_per_pubdata == 0`, `native_per_pubdata = 0`. This value is then used throughout the entire transaction lifecycle — intrinsic pubdata overhead, post-execution pubdata charging, and refund computation — all of which multiply by `native_per_pubdata`. Every pubdata-cost term collapses to zero.

**Contrast with L2 path:** The ZK L2 validation path (`validation_impl.rs`) derives `native_per_pubdata` from the block-level `pubdata_price / native_price`, which is always validated non-zero. L1 transactions bypass this path entirely and rely solely on the transaction-supplied `gas_per_pubdata_limit`. [4](#0-3) 

---

### Impact Explanation

An L1→L2 transaction with `gas_per_pubdata_limit = 0` executes with `native_per_pubdata = 0`. All pubdata generated by that transaction — storage writes, L2→L1 logs, bytecode publications — is accounted at zero cost. The transaction is still bounded by the block-level pubdata limit, but within that limit the attacker pays nothing for pubdata while the rollup must still publish it to L1. This is a **resource accounting bug**: pubdata consumption is decoupled from payment, undermining the economic model that protects the rollup from pubdata-based DoS.

---

### Likelihood Explanation

The code comment in `prepare_and_check_resources` explicitly acknowledges the missing OS-level guard:

> *"Note that the 'validation errors' are practically unreachable, as gas_limit, gas_price and gas_per_pubdata are either checked or set by the L1 contracts."* [5](#0-4) 

This means the ZKsync OS deliberately relies on L1 contracts as the sole enforcement layer. If the L1 bridge contracts are upgraded, misconfigured, or contain a bug that permits `gas_per_pubdata_limit = 0`, the OS has no independent defense. Any unprivileged user who can submit an L1→L2 priority transaction (the normal public interface) with that field set to zero would trigger the vulnerability.

---

### Recommendation

Add an explicit non-zero check inside `validate_structure()` for the `L1_L2_TX_TYPE` case:

```rust
match tx_type {
    Self::L1_L2_TX_TYPE => {
        if self.gas_per_pubdata_limit.read() == 0 {
            return Err(());
        }
    }
    Self::UPGRADE_TX_TYPE => {}
    _ => { ... }
}
```

This mirrors the existing pattern used for other type-specific required fields (e.g., the signature length check at lines 280–291) and provides defense-in-depth independent of L1 contract behavior. [6](#0-5) 

---

### Proof of Concept

1. Craft an ABI-encoded L1→L2 transaction (`tx_type = 0x7F`) with `gas_per_pubdata_limit = 0` and a non-trivial calldata that writes to multiple storage slots.
2. Submit it through the L1 priority queue (or directly inject it in a test via `L1TxBuilder` with `.gas_per_pubdata_byte_limit(0)`).
3. Observe that `prepare_and_check_resources` computes `native_per_pubdata = 0`.
4. The transaction executes, generates pubdata (storage diffs), and the post-execution pubdata charge is `0 * pubdata_used = 0` — the sender is never charged for pubdata.
5. Repeat up to the block pubdata limit to exhaust rollup pubdata capacity at zero marginal cost. [7](#0-6) [8](#0-7)
