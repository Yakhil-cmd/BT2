### Title
Unredacted `Debug` Derive on Secret-Bearing Structs Discloses Private Signing Shares and Presign Secrets - (File: `src/lib.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/triples/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary
Multiple production structs containing secret cryptographic material derive `Debug` without redacting secret scalar values. This allows private signing shares, presign nonce secrets, and Beaver triple scalars to be disclosed through debug formatting (e.g., logging with `{:?}`). This is the direct Rust analog to the iOS Keychain "accessible under weaker conditions than intended" vulnerability class: sensitive material is reachable through an interface — the `Debug` trait — that provides no access restriction, mirroring the use of `AlwaysThisDeviceOnly` or `AfterFirstUnlockThisDeviceOnly` instead of a passcode-gated attribute.

---

### Finding Description

The following production structs derive `Debug` and contain secret scalar material:

**1. `KeygenOutput<C>`** — `src/lib.rs:48`
```rust
#[derive(Debug, Clone, Deserialize, Serialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,   // SECRET signing share
    #[zeroize[skip]]
    pub public_key: VerifyingKey<C>,
}
```
The `private_share` field is the participant's secret signing share. The `Debug` derive causes it to be printed as a hex scalar whenever `{:?}` is used on this struct. The test snapshot at `src/frost/eddsa/snapshots/threshold_signatures__frost__eddsa__test__reshare_determinism.snap` confirms the scalar is rendered verbatim (e.g., `"b216a69fce7975ee44b615f71c58de1b2df1e85a4205ac7678e147078b68d909"`).

### Impact Explanation

Any caller that formats these structs with `{:?}` — in log statements, error messages, or assertion output — will emit the raw secret scalars. For `TripleShare`, the library's own documentation confirms that exposure of `a`, `b`, `c` enables full private key recovery. For `KeygenOutput`, exposure of `private_share` directly discloses the participant's secret signing share. For `PresignOutput`, exposure of `k`/`sigma` or `c`/`e`/`alpha`/`beta` discloses presign nonce material sufficient to reconstruct the signing nonce and break signature unforgeability.

This maps to the **Critical** allowed impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

**Medium.** The `Debug` derive is explicitly present on all public-facing secret-bearing structs, signaling to callers that it is safe to use. It is a standard Rust idiom to log structs with `{:?}` in error paths, tracing spans, and integration tests. Any node operator who instruments their signing pipeline with structured logging (a common production practice) and includes these structs in log output will expose secret material. The `PresignArguments` struct is particularly dangerous because it aggregates all secret inputs into a single `Debug`-printable value.

---

### Recommendation

Remove the `Debug` derive from all structs containing secret scalar fields, or implement `Debug` manually to redact secret fields:

```rust
impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        f.debug_struct("KeygenOutput")
            .field("private_share", &"[REDACTED]")
            .field("public_key", &self.public_key)
            .finish()
    }
}
```

Apply the same pattern to `PresignOutput`, `RerandomizedPresignOutput`, `TripleShare`, and `PresignArguments` across both ECDSA sub-modules. This is the direct analog to the iOS remediation: replacing a permissive accessibility attribute with a restrictive one that limits exposure to only the intended access path.

```rust
// Attacker-observable disclosure path:
let keygen_output: KeygenOutput<Secp256K1Sha256> = run_dkg(...);
// Any of the following exposes private_share verbatim:
