### Title
Missing Minimum Threshold Validation in FROST Signing Enables Single-Participant Signature Forgery - (File: src/frost/mod.rs)

### Summary
The `assert_sign_inputs` function and the `presign` function in `src/frost/mod.rs` validate that the signing threshold does not exceed the participant count, but fail to enforce a minimum threshold of 2. A malicious coordinator can initiate a FROST signing session with `threshold=1` (or even `threshold=0`), which passes all input checks. With `threshold=1`, the coordinator needs only their own signature share to produce a valid aggregate FROST signature, bypassing the multi-party consent requirement entirely.

### Finding Description

In `src/frost/mod.rs`, `assert_sign_inputs` (the shared validation gate for EdDSA and RedJubjub signing) performs the following threshold check:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [1](#0-0) 

This is only an upper-bound check. There is **no lower-bound check** (`threshold >= 2`). A `threshold` of `1` or `0` passes silently.

The same pattern appears in the `presign` function:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

This is in direct contrast to `assert_key_invariants` in `src/dkg.rs`, which correctly enforces a minimum threshold of 2:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [3](#0-2) 

The `ReconstructionLowerBound` type itself imposes no minimum:

```rust
pub struct ReconstructionLowerBound(usize);
``` [4](#0-3) 

So `ReconstructionLowerBound(1)` or `ReconstructionLowerBound(0)` are both constructible and accepted by `assert_sign_inputs` and `presign`.

### Impact Explanation

In the FROST protocol, the signing threshold determines how many participants' signature shares must be combined to produce a valid aggregate signature. If a malicious coordinator calls the signing function with `threshold=1`:

1. All honest participants pass `assert_sign_inputs` without error (threshold=1 satisfies `1 <= participants.len()`).
2. Each honest participant computes and sends their signature share to the coordinator.
3. With `threshold=1`, the coordinator needs only **one** signature share — their own — to reconstruct a valid aggregate FROST signature over any message hash they choose.
4. The coordinator can sign arbitrary attacker-chosen messages without the consent of the other participants, producing a cryptographically valid threshold signature.

This directly matches the Critical impact: **Unauthorized creation of a valid threshold signature for attacker-chosen inputs**.

### Likelihood Explanation

The coordinator role is a documented trust assumption in the FROST protocol as implemented here. A malicious coordinator controls the signing session parameters passed to `assert_sign_inputs`. Since the library exposes `assert_sign_inputs` as a public function and `ReconstructionLowerBound` accepts any `usize`, a malicious coordinator can trivially supply `threshold=1` without any library-level rejection. No cryptographic break or external compromise is required — only control of the coordinator role, which is an explicitly modeled adversary in this library.

### Recommendation

Add a minimum threshold check to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`, mirroring the guard already present in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the upper-bound check in both functions.

### Proof of Concept

```rust
// Malicious coordinator sets threshold=1 for a 3-participant signing session.
// assert_sign_inputs accepts this because 1 <= 3 (upper-bound check passes,
// lower-bound check is absent).
let participants = vec![coordinator, participant_b, participant_c];
let threshold = ReconstructionLowerBound::from(1usize); // passes all checks

let signing_participants = assert_sign_inputs(
    &participants,
    threshold,
    coordinator,
    coordinator,
).unwrap(); // succeeds — no ThresholdTooSmall error

// Coordinator now runs the FROST signing protocol.
// With threshold=1, the coordinator aggregates only their own share
// and produces a valid signature over an arbitrary msg_hash,
// without participant_b or participant_c contributing or consenting.

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
