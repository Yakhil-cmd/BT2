Audit Report

## Title
Permanent ckBTC Loss Without Reimbursement When `BuildTxError::AmountTooLow` Occurs in `submit_pending_requests` — (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary
The ckBTC minter validates withdrawal requests against a stale cached fee-based minimum (`fee_based_retrieve_btc_min_amount`), burns the user's ckBTC, then re-fetches a fresh fee estimate in `submit_pending_requests`. If Bitcoin network fees spike between these two points, `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, and the minter finalizes the request as `FinalizedStatus::AmountTooLow` without issuing any reimbursement mint. The burned ckBTC is permanently lost.

## Finding Description
**Root cause:** Two separate fee estimates are used — one stale cached value for admission gating in `retrieve_btc` and one fresh live value for transaction construction in `submit_pending_requests` — with no reconciliation between them.

**Exploit flow:**

1. `retrieve_btc` reads `s.fee_based_retrieve_btc_min_amount` (set during the last periodic `estimate_fee_per_vbyte` call at low fees, e.g. 1.5 sat/vbyte → minimum = 100,000 sats). The request passes the check.
2. `burn_ckbtcs` is called at `retrieve_btc.rs:210`, permanently burning the user's ckBTC and enqueuing the request.
3. `submit_pending_requests` calls `estimate_fee_per_vbyte(runtime).await` at `lib.rs:358` for a fresh estimate. If fees have spiked (e.g. to 710+ sat/vbyte), `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`.
4. The `AmountTooLow` branch at `lib.rs:412–434` calls `state::audit::remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` for each request in the batch. This function only records an event and pushes to finalized requests — it does not schedule any reimbursement.
5. The `InvalidTransaction` branch at `lib.rs:400–410` calls `reimburse_canceled_requests`, proving the omission in the `AmountTooLow` branch is a defect, not intentional design.
6. `WithdrawalReimbursementReason` contains only an `InvalidTransaction` variant; there is no `AmountTooLow` variant, making reimbursement structurally impossible on this path without a code change.
7. The `reimbursement_account` field is populated on the `RetrieveBtcRequest` at `retrieve_btc.rs:218–221` but is never accessed in the `AmountTooLow` finalization path.

**Why existing checks fail:** The admission check in `retrieve_btc` is intentionally based on a cached value to avoid an async call on every withdrawal. There is no guard that re-validates the amount against the fresh fee estimate obtained in `submit_pending_requests` before finalizing without reimbursement.

## Impact Explanation
This is a concrete, permanent loss of user funds in ckBTC, an in-scope chain-key ledger asset. Any unprivileged user who submits `retrieve_btc` at or near the advertised minimum withdrawal amount during a period of low fees will have their ckBTC burned with no recovery path if fees spike before the batch is processed. The `DustOutput` path at `lib.rs:436–467` has the same defect for the matched request. This matches the allowed impact: **High — Significant ck-token security impact with concrete user fund harm**.

## Likelihood Explanation
No special privileges are required. The attack is passively triggered by normal Bitcoin network congestion. Bitcoin fees of 500–1,000+ sat/vbyte have occurred during Ordinals/BRC-20 congestion events in 2023. At 710 sat/vbyte, a single-input P2WPKH transaction (~141 vbytes) costs ~100,110 sats in fees, exceeding the 100,000 sat minimum withdrawal. The window between `retrieve_btc` acceptance and `submit_pending_requests` execution spans multiple IC rounds. Any user withdrawing at the advertised minimum during a low-fee period is at risk if fees spike before their batch is processed. The condition is repeatable and requires no victim mistake beyond using the protocol as documented.

## Recommendation
In `submit_pending_requests`, when `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput` is encountered for a request, call `reimburse_canceled_requests` (mirroring the `InvalidTransaction` path) instead of calling only `remove_retrieve_btc_request`. Add a `WithdrawalReimbursementReason::AmountTooLow` variant to `reimbursement/mod.rs`. The reimbursement amount should be `request.amount` minus a bounded fee (consistent with the existing assertion in `reimburse_canceled_requests` that the fee does not exceed `retrieve_btc_min_amount`).

## Proof of Concept
State-machine test outline:

1. Initialize minter with `retrieve_btc_min_amount = 100_000` on Mainnet.
2. Set fee percentiles to `[FeeRate::from_millis_per_byte(1_500); 100]` and call `refresh_fee_percentiles` → `fee_based_retrieve_btc_min_amount` = 100,000 sats.
3. Call `retrieve_btc(amount = 100_000, address = <valid P2WPKH>)` → succeeds, ckBTC burned, request enqueued.
4. Set fee percentiles to `[FeeRate::from_millis_per_byte(1_000_000); 100]` (1,000 sat/vbyte) before advancing the timer.
5. Advance time past `max_time_in_queue_nanos` to trigger `submit_pending_requests`.
6. Assert: `retrieve_btc_status(block_index) == FinalizedStatus::AmountTooLow`.
7. Assert: user's ckBTC balance is not restored (no reimbursement mint occurred).
8. Assert: `pending_withdrawal_reimbursements` does not contain an entry for `block_index`.

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
