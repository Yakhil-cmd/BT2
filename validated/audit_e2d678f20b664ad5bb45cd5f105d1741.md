Audit Report

## Title
ckERC20 Minter Mints Unbacked Tokens for Fee-on-Transfer ERC20 Deposits - (`rs/ethereum/cketh/minter/ERC20DepositHelper.sol` / `DepositHelperWithSubaccount.sol`)

## Summary

Both ckERC20 deposit helper contracts emit deposit events using the caller-supplied `amount` argument rather than the actual tokens received by the minter address. For fee-on-transfer ERC20 tokens (e.g., XAUt, which is already a supported ckERC20 token and charges a 0.02% transfer fee), the minter receives `amount - fee` on-chain but the IC minter canister reads `amount` from the log and mints that full amount of ckERC20. This permanently inflates the ckERC20 supply beyond the ERC20 collateral held, breaking the 1:1 backing invariant and eventually causing withdrawal failures for the last redeemers.

## Finding Description

**Root cause — Solidity contracts emit pre-fee `amount`:**

`ERC20DepositHelper.sol` L498–503:
```solidity
function deposit(address erc20_address, uint256 amount, bytes32 principal) public {
    IERC20 erc20Token = IERC20(erc20_address);
    erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
    emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
}
```

`DepositHelperWithSubaccount.sol` L511–532:
```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```

For a fee-on-transfer token, `safeTransferFrom` delivers `amount - fee` to `minterAddress`, but the event is emitted with the original `amount`.

**IC minter reads event value verbatim:**

`parser.rs` L86–97 parses the event data word directly into `value: Erc20Value::from_be_bytes(value_bytes)` with no balance-delta verification.

**Minter mints exactly `event.value()`:**

`deposit.rs` L73–81 calls `client.transfer(TransferArg { amount: event.value(), ... })`, minting the full pre-fee amount to the user.

**No existing guard:** There is no check anywhere in the minter that validates the actual ERC20 balance change against the logged value. The minter trusts the event data unconditionally.

**Currently affected token:** XAUt (Tether Gold, `0x68749665FF8D2d112Fa859AA293F07A622782F38`) is already in the supported ckERC20 token list (`rs/ethereum/cketh/docs/ckerc20.adoc` L46–47) and charges a 0.02% fee on every transfer. Every XAUt deposit today triggers this discrepancy.

## Impact Explanation

Every deposit of a fee-on-transfer token mints `fee` more ckERC20 than the ERC20 collateral actually held. The ckERC20 supply grows unbacked by the cumulative transfer fees. When users later withdraw ckERC20 back to ERC20, the minter's Ethereum address will eventually lack sufficient ERC20 balance to fulfill all withdrawals. The last withdrawers permanently lose funds proportional to the total accumulated fee inflation. This is **illegal minting** and **protocol insolvency** for the affected ckERC20 token — a direct match for the Critical/High bounty impact: *"Theft, permanent loss, illegal minting, or protocol insolvency involving in-scope chain-key/ledger assets."*

## Likelihood Explanation

XAUt is already a supported ckERC20 token and charges a 0.02% fee on every transfer. No special privilege, governance majority, or key compromise is required — any unprivileged user calling `depositErc20` with XAUt triggers the discrepancy on every deposit. The vulnerability is automatic and cumulative: each deposit widens the insolvency gap by exactly the transfer fee amount. The attack requires only a normal deposit call.

## Recommendation

In both helper contracts, measure the minter's actual balance change around the `transferFrom` call and emit the event with the measured delta:

```solidity
function depositErc20(address erc20Address, uint256 amount, bytes32 principal, bytes32 subaccount) public {
    require(erc20Address != ZERO_ADDRESS, "ERC20: depositErc20 from the zero address");
    IERC20 erc20Token = IERC20(erc20Address);
    uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
    erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
    uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
    emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
}
```

Apply the same pattern to `ERC20DepositHelper.sol`'s `deposit` function.

## Proof of Concept

1. XAUt (`0x68749665FF8D2d112Fa859AA293F07A622782F38`) is already a supported ckERC20 token, confirmed in `rs/ethereum/cketh/docs/ckerc20.adoc` L46–47. XAUt charges a 0.02% fee on every `transferFrom`.
2. User calls `depositErc20(xautAddress, 1_000_000, principal, subaccount)` on `DepositHelperWithSubaccount`.
3. `safeTransferFrom` executes: minter receives `999_800` XAUt (after 200-unit fee).
4. Event emitted: `ReceivedEthOrErc20(xautAddress, user, 1_000_000, principal, subaccount)` — using the original `1_000_000`.
5. IC minter scrapes the log, `parser.rs` L86–97 reads `value = 1_000_000` from event data, `deposit.rs` L80 mints `1_000_000` ckXAUt to the user.
6. Minter holds only `999_800` XAUt on Ethereum but has issued `1_000_000` ckXAUt.
7. After N such deposits, the ckXAUt supply exceeds the XAUt collateral by `N × 200`. The last `N × 200` ckXAUt units are unbacked; corresponding withdrawal requests will fail, causing permanent loss of funds for those users.

A deterministic integration test can be written using the existing `CkErc20Setup` test harness in `rs/ethereum/cketh/test_utils/src/ckerc20.rs`, deploying a mock fee-on-transfer ERC20 contract, performing a deposit, and asserting that the ckERC20 ledger total supply exceeds the mock ERC20 balance held by the minter address.
