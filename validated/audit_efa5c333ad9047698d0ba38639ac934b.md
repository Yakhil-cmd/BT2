### Title
`protocolUnpausePool` transitions pool to admin-paused (level 1) instead of active (level 0), leaving swaps permanently blocked without admin cooperation — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). After a protocol pause/unpause cycle on an originally-active pool, the pool's `swap` function remains blocked because `_checkNotPaused` reverts on any non-zero pause level. The pool admin must separately call `unpausePool` to restore swap functionality — an action the protocol owner cannot force.

---

### Finding Description

The pool tracks three pause levels:

```
0 = active, 1 = paused by admin, 2 = paused by protocol
```

The `_checkNotPaused` guard used by `swap` reverts on **any** non-zero level:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [1](#0-0) 

`protocolPausePool` correctly accepts pools at level 0 **or** 1 (both non-protocol-paused states):

```solidity
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
``` [2](#0-1) 

But `protocolUnpausePool` always targets level **1**, not level 0:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1); // ← sets admin-paused, not active
}
``` [3](#0-2) 

The resulting state machine has an asymmetry: `protocolPausePool` can consume a level-0 (active) pool, but `protocolUnpausePool` never restores it to level 0. The pool is left at level 1 (admin-paused), where `swap` still reverts.

The admin-side `unpausePool` enforces `cur == 1` before setting level 0:

```solidity
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
``` [4](#0-3) 

The protocol owner has no path to call `unpausePool` — it is `onlyPoolAdmin`. If the pool admin is unresponsive, the pool's swap functionality is permanently blocked even after the protocol owner has "unpaused" it.

---

### Impact Explanation

After a `protocolPausePool` → `protocolUnpausePool` cycle on a pool that was originally at level 0 (active), the pool is left at level 1 (admin-paused). The `swap` function remains blocked for all users. `addLiquidity` and `removeLiquidity` are not gated by `whenNotPaused`, so LPs can still withdraw, but no trading is possible. The protocol owner cannot restore swap functionality unilaterally — they must rely on the pool admin to call `unpausePool`. This breaks the core invariant that the protocol owner can fully reverse a protocol-imposed pause.

---

### Likelihood Explanation

This is a deterministic bug triggered every time `protocolUnpausePool` is called on a pool that was active (level 0) before the protocol pause. No special conditions, price values, or timing are required. The only prerequisite is the normal protocol pause/unpause operational flow.

---

### Recommendation

Change `protocolUnpausePool` to set the pool to level 0 (active) instead of level 1 (admin-paused):

```diff
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-   if (cur != 2) revert InvalidPauseTransition(cur, 1);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   if (cur != 2) revert InvalidPauseTransition(cur, 0);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the design intent is to require admin sign-off before swaps resume after a protocol pause, this should be explicitly documented and the function renamed (e.g., `protocolDowngradeToAdminPause`). As written, the function name implies full restoration of pool activity.

---

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → pool transitions to level 2.
3. Emergency is resolved; protocol owner calls `protocolUnpausePool(pool)` → pool transitions to level **1** (admin-paused), not 0.
4. Any user calls `pool.swap(...)` → reverts with `PoolPaused()` because `pauseLevel == 1 != 0`.
5. Protocol owner has no function to call `setPause(0)` directly — only `protocolUnpausePool` (which goes to 1) or `protocolPausePool` (which goes to 2).
6. Pool admin must call `unpausePool(pool)` to restore level 0. If admin is absent or uncooperative, swaps remain blocked indefinitely.
