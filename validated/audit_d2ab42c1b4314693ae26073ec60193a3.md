### Title
WalletConnect `chia_sendTransaction` `puzzle_decorator` Rendered as Opaque JSON — `clawbackTimelock` Unit Not Humanized, Enabling Attacker-Controlled Timelock Injection — (`packages/gui/src/electron/commands/Commands.ts`)

### Summary

A WalletConnect dApp that has been granted `chia_sendTransaction` permission can inject an arbitrary `puzzle_decorator` value (e.g., `[{decorator:'CLAWBACK',clawbackTimelock:1}]`) into the command. The approval dialog renders the entire `puzzle_decorator` field as raw pretty-printed JSON via `JSONBig.stringify`, displaying `"clawbackTimelock": 1` with no unit label. The user cannot determine whether `1` means 1 second, 1 minute, 1 day, or any other duration, and therefore cannot make an informed decision before approving.

### Finding Description

**Entrypoint:** A paired WalletConnect dApp sends `chia_sendTransaction` with `puzzle_decorator=[{decorator:'CLAWBACK',clawbackTimelock:1}]`.

**Parsing path:**

`parseDappParams` in `packages/gui/src/electron/commands/parseDappParams.ts` validates that `puzzle_decorator` is a declared param (it is, as `type: 'json'`, `isOptional: true`) and passes the value through without any content validation. [1](#0-0) 

**Schema definition:** In `Commands.ts`, `puzzle_decorator` is declared as `type: 'json'` with no `humanize` property — unlike `amount` and `fee` which have `humanize: 'mojo-to-xch'`. [2](#0-1) 

Compare with the humanized fields: [3](#0-2) 

**Humanization path:** `humanizeParamValue` in `humanizeParamValue.ts` handles `type: 'json'` by calling `JSONBig.stringify(value, null, 2)` — producing a raw JSON string with no unit annotation. [4](#0-3) 

**Dialog rendering:** `Confirm.tsx` renders each row's `value` as plain text. The user sees:

```
Puzzle Decorator
[
  {
    "decorator": "CLAWBACK",
    "clawbackTimelock": 1
  }
]
``` [5](#0-4) 

The integer `1` has no unit label. The user cannot distinguish `clawbackTimelock: 1` (1 second — effectively no protection) from `clawbackTimelock: 86400` (1 day) or `clawbackTimelock: 31536000` (1 year).

**Execution path to RPC:** After user approval, `parsedParams` (including the injected `puzzle_decorator`) is forwarded directly to the `chia_wallet` RPC `send_transaction` command. [6](#0-5) 

### Impact Explanation

Two concrete attack scenarios:

1. **Clawback stripping (1-second timelock):** The dApp injects `clawbackTimelock: 1`. The user approves, believing the value is reasonable. The 1-second window expires before the user can act, eliminating clawback protection entirely. The user sent funds believing they had a meaningful reclaim window.

2. **Recipient lockout (attacker-chosen long timelock):** The dApp injects `clawbackTimelock: 31536000` (1 year). The recipient cannot claim the funds for a year. The sender approved a transaction with a year-long clawback they did not intend, which may be used to harass or lock a recipient's expected funds.

In both cases the user approved a transaction carrying an attacker-controlled security parameter without being shown its meaning in human-readable form. This fits: *"WalletConnect state that causes a user to approve a send with the wrong status."*

### Likelihood Explanation

- Requires only that the user has previously approved the dApp's `chia_sendTransaction` permission — a normal, expected pairing step.
- No additional privileges, key leakage, or host compromise required.
- The dApp controls the full `puzzle_decorator` JSON payload; there is no server-side or GUI-side content validation on the nested `clawbackTimelock` value.
- Locally testable: craft a WalletConnect session, send `chia_sendTransaction` with `puzzleDecorator:[{decorator:'CLAWBACK',clawbackTimelock:1}]`, observe the approval dialog.

### Recommendation

In the `chia_wallet.send_transaction` command schema, replace the opaque `type: 'json'` rendering for `puzzle_decorator` with a dedicated humanizer that:

1. Detects `decorator === 'CLAWBACK'` entries.
2. Converts `clawbackTimelock` (seconds) to a human-readable string (e.g., `"1 second"`, `"2 days 3 hours"`).
3. Displays the result as `"Clawback timelock: 2 days 3 hours"` rather than raw JSON.

The existing `getClawbackTimeInSeconds` / `getOfferExpirationTimeAsTuple` utilities in `packages/wallets/src/hooks/useClawbackDefaultTime.tsx` already implement the inverse conversion and can be reused. [7](#0-6) 

### Proof of Concept

```
1. Pair a WalletConnect dApp that requests chia_sendTransaction permission.
2. User approves the pairing.
3. dApp sends:
   {
     "method": "chia_sendTransaction",
     "params": {
       "fingerprint": <paired_fingerprint>,
       "amount": "1000000000",
       "fee": "0",
       "address": "<valid_xch_address>",
       "puzzleDecorator": [{"decorator": "CLAWBACK", "clawbackTimelock": 1}]
     }
   }
4. Approval dialog appears showing:
     Puzzle Decorator
     [
       {
         "decorator": "CLAWBACK",
         "clawbackTimelock": 1
       }
     ]
5. User sees "1" with no unit — cannot determine this is 1 second.
6. User clicks "Send".
7. Transaction is broadcast with a 1-second clawback timelock.
   Clawback protection is effectively nullified.
```

export function getClawbackTimeInSeconds(clawbackTimeObject: ClawbackDefaultTimeInput) {
  const { days, hours, minutes } = stringPropertiesToNumbers(clawbackTimeObject);
  const daysInSeconds = days * 24 * 60 * 60;
  const hoursInSeconds = hours * 60 * 60;
  const minutesInSeconds = minutes * 60;
  return daysInSeconds + hoursInSeconds + minutesInSeconds;
}
