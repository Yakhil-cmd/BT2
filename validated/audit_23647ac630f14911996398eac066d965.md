### Title
Missing Zero-Address Validation for `reward_address` in `change_reward_address` and `enter_delegation_pool` - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

Both the Staking contract's `change_reward_address` and the Pool contract's `enter_delegation_pool` accept a `reward_address` parameter without validating it against the zero address. If a staker or pool member accidentally supplies `ContractAddress::zero()`, all subsequently claimed rewards are transferred to address(0) and permanently burned, with no protocol-level guard to prevent it.

---

### Finding Description

**`change_reward_address` — `src/staking/staking.cairo` lines 517–540**

The function validates only that `reward_address` is not a registered token address, but performs no zero-address check:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;
    staker_info.reward_address = reward_address;   // written unconditionally
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

**`enter_delegation_pool` — `src/pool/pool.cairo` lines 182–219**

Similarly, only a token-address check is present; zero is not rejected:

```cairo

    ref self: ContractState, reward_address: ContractAddress, amount: Amount,

    ...
    assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);

``` [2](#0-1) 

The `GenericError::ZERO_ADDRESS` variant exists in the shared error module and is already used elsewhere (e.g., `add_token`), confirming the pattern is known but was not applied here: [3](#0-2) 

---

### Impact Explanation

When `claim_rewards` executes, it transfers the accrued reward amount to the stored `reward_address`. If that address is zero, the ERC-20 transfer succeeds on Starknet (zero address is a valid recipient) and the tokens are permanently burned. The staker or pool member loses all unclaimed yield accumulated up to the point they notice and call `change_reward_address` again. This constitutes **permanent freezing / destruction of unclaimed yield** for the affected account.

---

### Likelihood Explanation

Any staker or pool member can trigger this by passing `0` as `reward_address` — either by mistake (copy-paste error, uninitialized variable in a script) or via a front-end bug. The call is permissionless and requires no privileged role. Because the protocol provides no guard, a single erroneous transaction is sufficient to route all future reward claims to the burn address until the user manually corrects it.

---

### Recommendation

Add an explicit zero-address assertion in both functions, mirroring the pattern already used for `add_token`:

```cairo
// In change_reward_address and enter_delegation_pool:
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply the same guard to the `reward_address` parameter of `stake()` in `src/staking/staking.cairo` (line 290) for consistency. [4](#0-3) 

---

### Proof of Concept

1. A staker calls `change_reward_address(reward_address: Zero::zero())` on the Staking contract.
2. No assertion fires; `staker_info.reward_address` is written as `0x0`.
3. The staker (or anyone triggering reward distribution) calls `claim_rewards`.
4. The contract transfers the accrued STRK reward to `0x0`; the tokens are burned and irrecoverable.
5. The staker must call `change_reward_address` again with a valid address to stop further losses, but rewards already sent to `0x0` are gone.

The same sequence applies to a pool member calling `enter_delegation_pool(reward_address: Zero::zero(), amount: ...)`.
