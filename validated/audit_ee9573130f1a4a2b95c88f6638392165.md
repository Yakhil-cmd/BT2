### Title
Gateway Stateful Validator Checks L2 Gas Price Against Current Block Instead of Next Block — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful resource-bounds check and blockifier pre-validation both use the **current committed block's** `l2_gas_price` as the reference price, but the transaction will actually execute in the **next** block, which uses `next_l2_gas_price` from the current block's header. This is the direct Sequencer analog of the external report's "L2 upgrade timing precedes L1 upgrade" class: a timing/ordering mismatch between the price used at admission and the price enforced at execution.

### Finding Description

**Step 1 – `validate_resource_bounds` reads the wrong price.**

In `run_pre_validation_checks` → `validate_state_preconditions` → `validate_resource_bounds`, the code explicitly reads the **current** block's L2 gas price:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

The TODO comment is the developers' own acknowledgement that `next_l2_gas_price` should be used instead.

**Step 2 – `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` never reads `next_l2_gas_price`.**

The `BlockInfo` struct returned by the fixed-block reader is built from `block_header.l2_gas_price.price_in_fri` (the current block's price). The `block_header.next_l2_gas_price` field — which is what the sequencer will actually use for the next block — is silently discarded. [2](#0-1) 

**Step 3 – `run_validate_entry_point` increments the block number but keeps the stale gas prices.**

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
// gas_prices are NOT updated to next_l2_gas_price
let block_context = BlockContext::new(block_info, ...);
``` [3](#0-2) 

The blockifier's `check_fee_bounds` inside `perform_pre_validation_stage` then compares `resource_bounds.max_price_per_unit` against `block_info.gas_prices.strk_gas_prices.l2_gas_price` — which is still block N's price, not block N+1's price. [4](#0-3) 

**Step 4 – The actual next block uses `next_l2_gas_price`.**

The consensus orchestrator stores `next_l2_gas_price` in the block header and uses it as the L2 gas price for the following block: [5](#0-4) 

The EIP-1559 fee market can move `next_l2_gas_price` significantly above or below the current block's `l2_gas_price` depending on block utilization. [6](#0-5) 

### Impact Explanation

**Case A – Price increase (`next_l2_gas_price > l2_gas_price`, block above gas target):**
A transaction with `max_price_per_unit = l2_gas_price` passes both the gateway's `validate_resource_bounds` check and the blockifier's `check_fee_bounds` at admission time (both use block N's price). However, when the batcher builds block N+1 with the actual `next_l2_gas_price`, `check_fee_bounds` fails. The transaction is admitted to the mempool but rejected during block building — wasted sequencer resources, potential DoS vector.

**Case B – Price decrease (`next_l2_gas_price < l2_gas_price`, block below gas target):**
A transaction with `max_price_per_unit = next_l2_gas_price` (perfectly valid for block N+1) is rejected at the gateway because the check compares against the higher block N price. Valid transactions are incorrectly rejected.

Both cases match the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The EIP-1559 mechanism continuously adjusts `next_l2_gas_price` based on block utilization. Any block that is above or below the gas target produces a `next_l2_gas_price` that differs from the current `l2_gas_price`. This is the normal operating condition of the fee market, not an edge case. The divergence grows with sustained high or low utilization. No special privileges are required — any user submitting a transaction triggers this path.

### Recommendation

In `validate_resource_bounds`, replace the read of `l2_gas_price` with `next_l2_gas_price` from the block header (as the TODO comment already notes). Similarly, in `run_validate_entry_point`, when constructing the `BlockContext` for block N+1, replace `block_info.gas_prices.strk_gas_prices.l2_gas_price` with the `next_l2_gas_price` value from the block header. The `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` should expose `next_l2_gas_price` so both call sites can use it.

### Proof of Concept

1. Observe current block N has `l2_gas_price = 100` and `next_l2_gas_price = 110` (block was above gas target).
2. Submit an invoke V3 transaction with `l2_gas.max_price_per_unit = 100`.
3. Gateway `validate_resource_bounds` computes threshold as `100 * min_gas_price_percentage / 100`. With `min_gas_price_percentage = 100`, threshold = 100. Check passes (`100 >= 100`).
4. Gateway `run_validate_entry_point` builds `BlockContext` with `block_number = N+1` but `l2_gas_price = 100`. Blockifier `check_fee_bounds` passes (`100 >= 100`).
5. Transaction is admitted to the mempool.
6. Batcher builds block N+1 with actual `l2_gas_price = 110` (from `next_l2_gas_price`).
7. Blockifier `check_fee_bounds` in block N+1 fails: `100 < 110` → `MaxGasPriceTooLow` for L2Gas.
8. Transaction is dropped from the block; the mempool accepted an invalid transaction.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
