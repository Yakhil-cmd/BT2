### Title
Unregistered Contract Emitting `__CronosSendToAccount` / `__CronosSendToIbc` Events Causes Entire EVM Transaction Revert via `PostTxProcessing` ŌĆö (File: x/cronos/keeper/evmhandlers/send_to_account.go)

---

### Summary

`LogProcessEvmHook.PostTxProcessing` iterates over every log in the EVM receipt and, for any log whose topic matches a known Cronos bridge event, invokes the corresponding handler. `SendToAccountHandler.Handle` (and `SendToIbcHandler.handle`) return a hard error when the emitting contract is not registered in the token mapping. Because `PostTxProcessing` propagates that error upward, the entire EVM transaction is reverted at the Cosmos layer ŌĆö even if the EVM execution itself succeeded and the call to the malicious contract was wrapped in a Solidity `try/catch`. Any unprivileged contract that receives control flow can exploit this to permanently block the outer transaction.

---

### Finding Description

`LogProcessEvmHook.PostTxProcessing` in `x/cronos/keeper/evm_hooks.go` processes every log in the receipt:

```go
handler, ok := h.handlers[log.Topics[0]]
if !ok {
    continue
}
err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
if err != nil {
    return err   // ŌåÉ propagates to Cosmos, reverts entire tx
}
``` [1](#0-0) 

`SendToAccountHandler.Handle` returns a hard error for any contract whose address is not registered in the denomŌåÆcontract mapping:

```go
denom, found := h.cronosKeeper.GetDenomByContract(ctx, contract)
if !found {
    return fmt.Errorf("contract %s is not connected to native token", contract)
}
``` [2](#0-1) 

`SendToIbcHandler.handle` has the identical guard: [3](#0-2) 

The event signature is a fixed ABI hash of `__CronosSendToAccount(address,uint256)`. Any contract can emit a log with that topic. The handler is selected purely by `log.Topics[0]`; there is no prior check that `log.Address` is a registered contract before the handler is invoked. [4](#0-3) 

The existing test suite confirms this path: the "not enough balance, expect fail" case uses an unregistered contract and asserts `suite.Require().Error(err)` from `PostTxProcessing`. [5](#0-4) 

---

### Impact Explanation

When `PostTxProcessing` returns an error, Ethermint reverts all state changes from the EVM execution. This is a **Cosmos-layer revert**, not an EVM-layer revert. Solidity `try/catch` and low-level calls with gas limits only isolate EVM-layer reverts; they cannot prevent a Cosmos-layer revert triggered by `PostTxProcessing`. Therefore, any transaction that calls a malicious contract ŌĆö even through a fully isolated `try/catch` callback ŌĆö will be silently reverted if the malicious contract emits `__CronosSendToAccount` or `__CronosSendToIbc`.

This maps to the **High** allowed impact: *Permanent or long-lived inability for honest users or validators to process valid transactions under normal network assumptions.* A concrete example is a lending protocol that notifies a borrower's contract via `try/catch` before executing a liquidation. The borrower deploys a contract that emits `__CronosSendToAccount`; every liquidation attempt against them is permanently blocked. This also breaks EVM equivalence: the same contract is safe on any other EVM chain but lethal on Cronos.

---

### Likelihood Explanation

- No special privilege is required. Any EOA can deploy a contract that emits `__CronosSendToAccount(recipient, amount)`.
- The attack is triggered passively: the malicious contract only needs to be called once during the victim's transaction.
- DeFi protocols using callback patterns (liquidation hooks, flash-loan receivers, ERC-777 hooks, ERC-1363 receivers) are directly exposed.
- The attacker does not need to know the internals of Cronos; emitting an event with the right 4-byte topic is trivial.

---

### Recommendation

Change `SendToAccountHandler.Handle` and `SendToIbcHandler.handle` to **silently ignore** (log and return `nil`) when the emitting contract is not registered in the token mapping, consistent with how both handlers already treat ABI decode failures:

```go
denom, found := h.cronosKeeper.GetDenomByContract(ctx, contract)
if !found {
    // log and ignore ŌĆö unregistered contracts must not abort the outer tx
    h.cronosKeeper.Logger(ctx).Error("SendToAccount from unregistered contract", "contract", contract)
    return nil
}
```

Only errors that arise from a **registered** contract's bridge operation (e.g., insufficient balance, IBC transfer failure) should propagate as hard errors, because those represent genuine invariant violations for that contract's own funds.

---

### Proof of Concept

1. Attacker deploys `MaliciousCallback`:
   ```solidity
   contract MaliciousCallback {
       event __CronosSendToAccount(address recipient, uint256 amount);
       function onLiquidationNotify() external {
           emit __CronosSendToAccount(address(this), 1);
           // does NOT revert ŌĆö logs are kept in receipt
       }
   }
   ```

2. Attacker registers `MaliciousCallback` as their liquidation callback address in a DeFi lending protocol that uses `try/catch` to notify borrowers.

3. Liquidator submits a liquidation transaction. The lending contract executes:
   ```solidity
   try IBorrower(borrower).onLiquidationNotify() {} catch {}
   // EVM execution succeeds; log is in receipt
   ```

4. EVM execution completes successfully. `PostTxProcessing` iterates the receipt logs, finds `__CronosSendToAccount` emitted by `MaliciousCallback` (unregistered), calls `SendToAccountHandler.Handle`, which returns `"contract 0xŌĆ” is not connected to native token"`. [2](#0-1) 

5. `LogProcessEvmHook.PostTxProcessing` returns that error; Ethermint reverts the entire transaction. [6](#0-5) 

6. The liquidation is permanently blocked. The attacker's loan position is never liquidated.

func (h LogProcessEvmHook) PostTxProcessing(ctx sdk.Context, _ *core.Message, receipt *ethtypes.Receipt) error {
	addLogToReceiptFunc := newFuncAddLogToReceipt(receipt)
	for _, log := range receipt.Logs {
		if len(log.Topics) == 0 {
			continue
		}
		handler, ok := h.handlers[log.Topics[0]]
		if !ok {
			continue
		}
		err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
		if err != nil {
			return err
		}
	}
	return nil
}
