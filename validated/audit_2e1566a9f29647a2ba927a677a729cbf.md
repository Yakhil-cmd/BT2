### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Delegator's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract contains an unbounded loop over a per-delegator balance checkpoint trace (`pool_member_epoch_balance`). Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a distinct epoch appends a new checkpoint to this trace. Because `claim_rewards` processes all pending checkpoints atomically with no partial-claim mechanism, a delegator who accumulates enough checkpoints without claiming will find their `claim_rewards` transaction permanently exceeds the block gas limit, irreversibly freezing their unclaimed yield.

---

### Finding Description

`claim_rewards` in `src/pool/pool.cairo` delegates reward computation to `calculate_rewards`:

```cairo
fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
    ...
    let (mut rewards, updated_entry_to_claim_from) = self
        .calculate_rewards(
            :pool_member,
            from_checkpoint: pool_member_info.reward_checkpoint,
            :until_checkpoint,
            entry_to_claim_from: pool_member_info.entry_to_claim_from,
        );
    ...
    pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
    pool_member_info.reward_checkpoint = until_checkpoint;
    ...
}
```

Inside `calculate_rewards`, the developers themselves flag the risk:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
```

Each iteration performs:
1. One storage read from `pool_member_epoch_balance` via `pool_member_trace.at(entry_to_claim_from)`.
2. Up to three storage reads from `cumulative_rewards_trace_vec` inside `find_sigma` / `find_sigma_standard_case`.

The trace grows because `set_member_balance` (called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`) calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` function only deduplicates when the incoming key equals the last stored key; otherwise it appends a new checkpoint:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
```

Because `key = current_epoch + K`, every balance change in a different epoch appends a distinct checkpoint. A delegator who adjusts their delegation once per epoch accumulates one checkpoint per epoch indefinitely.

`entry_to_claim_from` is only advanced inside a successful `claim_rewards` execution. If the transaction reverts due to gas exhaustion, the pointer is never updated, and the delegator has no way to process a subset of checkpoints — there is no partial-claim path.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace for a delegator grows large enough that iterating over all unprocessed entries in a single transaction exceeds the Starknet block gas limit, every subsequent `claim_rewards` call for that delegator will revert. Because `entry_to_claim_from` is only committed on success, the delegator is permanently unable to recover their accrued rewards. The funds remain locked in the Pool contract with no recovery path.

---

### Likelihood Explanation

Any delegator who:
- Calls `add_to_delegation_pool` or `exit_delegation_pool_intent` in many distinct epochs (one call per epoch is sufficient to grow the trace by one entry per epoch), **and**
- Does not call `claim_rewards` frequently enough to keep `entry_to_claim_from` advancing,

will eventually reach the gas-limit threshold. This is a realistic usage pattern for active delegators who compound their stake regularly. The protocol imposes no upper bound on the trace length and no minimum claim frequency. The developers' own inline comment — *"This loop is unbounded but unlikely to exceed gas limits"* — confirms awareness of the risk without any mitigation.

Additionally, `add_to_delegation_pool` is callable by the pool member **or their `reward_address`**, meaning a malicious or compromised reward address can accelerate trace growth on behalf of the pool member without the pool member's per-call consent.

---

### Recommendation

1. **Add a partial-claim mechanism**: Allow `claim_rewards` to accept an optional `max_checkpoints` parameter and advance `entry_to_claim_from` incrementally across multiple transactions.
2. **Cap trace growth**: Enforce a maximum number of unprocessed checkpoints (e.g., require a claim before a new balance change is accepted if the pending count exceeds a threshold).
3. **Compact the trace on claim**: After processing checkpoints up to `entry_to_claim_from`, truncate or compact the already-processed prefix so the effective loop length stays bounded.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` at epoch `E`.
2. For each subsequent epoch `E+1, E+2, ..., E+N`, the delegator calls `add_to_delegation_pool` with a small amount. Each call in a new epoch appends one entry to `pool_member_epoch_balance` (since `get_epoch_plus_k()` returns a strictly larger key each epoch).
3. The delegator never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After `N` epochs, `pool_member_trace_length == N`.
5. The delegator calls `claim_rewards`. `calculate_rewards` enters the loop and must execute `N` iterations, each performing ~4 storage reads.
6. For sufficiently large `N`, the transaction exceeds the block gas limit and reverts.
7. `entry_to_claim_from` remains 0. Every future `claim_rewards` attempt also reverts. The delegator's accumulated yield is permanently frozen.

```text
pub(crate) fn find_sigma_standard_case(
    cumulative_rewards_trace_vec: StorageBase<Trace>,
    cumulative_rewards_trace_idx: VecIndex,
    target_epoch: Epoch,
) -> Amount {
    // Three entries in the cumulative rewards trace are relevant (idx, idx - 1, idx - 2).
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx);
    if epoch < target_epoch {
        return sigma;
    }
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx - 1);
    if epoch < target_epoch {
        return sigma;
    }
    // Note: When handling a checkpoint from the old version, it never reaches here.
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx - 2);
    assert!(epoch < target_epoch, "{}", InternalError::INVALID_EPOCH_IN_TRACE);
    sigma
}
```
