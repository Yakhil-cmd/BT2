### Title
Cross-COA Signature Replay in `EVM.validateCOAOwnershipProof` Enables Unauthorized ERC-1271 Authentication - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced to prove ownership of one Cadence-Owned Account (COA) can be replayed by an attacker to pass ERC-1271 authentication for a different COA belonging to the same Cadence account, because the signed payload is decoupled from the COA identity being proven.

### Finding Description

`EVM.validateCOAOwnershipProof` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The function verifies that the provided signatures are valid over `signedData` using the account's keys, then separately checks that the COA resource at `path` has an EVM address matching `evmAddress`.

The critical gap is that `signedData` is never required to contain `evmAddress`. The two checks — signature validity and COA address matching — are independent. The code itself acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The signature verification at line 1082–1086 checks only that `signedData` was signed by the account's keys: [2](#0-1) 

The COA address check at lines 1095–1105 is a separate, independent step that verifies the COA resource at `path` matches `evmAddress`, but this check is not cryptographically bound to the signature: [3](#0-2) 

The EVM-side ERC-1271 entry point in `coa.sol` calls the precompile with `address(this)` as the COA address: [4](#0-3) 

In `DecodeABIEncodedProof`, the `caller` (= `address(this)`, the calling COA's EVM address) is used as `EVMAddress`, and the attacker-controlled `encodedProof` bytes supply the `CapabilityPath`: [5](#0-4) 

The `COAOwnershipProof` struct that is RLP-decoded from the attacker-supplied `_sig` bytes contains the `CapabilityPath` field, which the attacker can freely set: [6](#0-5) 

### Impact Explanation

**Unauthorized ERC-1271 authentication for a COA the signer never authorized.**

Concrete scenario:
1. Alice holds a Cadence account with two COAs: COA-A published at `/public/coaA` and COA-B published at `/public/coaB`.
2. Alice signs hash `H` (not including any COA address) to prove ownership of COA-A for some legitimate ERC-1271 check. The resulting `COAOwnershipProof` encodes `CapabilityPath = "coaA"`.
3. An attacker observes this proof and constructs a modified proof with `CapabilityPath = "coaB"` but the same `Signatures` and `KeyIndices`.
4. The attacker calls `COA-B.isValidSignature(H, modified_proof)`. The precompile passes `address(COA-B)` as `evmAddress` and the modified proof as `encodedProof`.
5. Inside `validateCOAOwnershipProof`: the signature check passes (Alice's key signed `H`); the COA address check passes (COA-B is at `/public/coaB` and its address matches `address(COA-B)`).
6. The function returns `isValid: true` — COA-B is authenticated for hash `H` without Alice ever signing for COA-B.

Any EVM contract relying on ERC-1271 (`isValidSignature`) for access control — such as ERC-4337 account abstraction entry points, multisig wallets, or token-gated contracts — will grant the attacker the same rights as COA-B's legitimate owner.

### Likelihood Explanation

**Medium.** The precondition of a Cadence account holding multiple COAs is not enforced against by the protocol; any transaction can create additional COAs. The attacker only needs to observe a single valid COA ownership proof (e.g., from a public on-chain transaction or an off-chain authentication flow) and modify the `CapabilityPath` field in the RLP-encoded proof bytes before replaying it. No privileged access, staked node control, or key compromise is required — the attack is executable by any unprivileged EVM transaction sender calling `isValidSignature` on a COA contract.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` cryptographically commits to `evmAddress`. Concretely, before the `keyList.verify` call, assert that `signedData` contains the canonical encoding of `evmAddress` (e.g., require `signedData == hash(evmAddress || application_data)`). Alternatively, adopt a structured signing scheme (analogous to EIP-712) that binds the signature to the specific COA EVM address and the Cadence account address, preventing cross-COA and cross-account replay. [2](#0-1) 

### Proof of Concept

**Setup:**
- Alice's Cadence account `0xALICE` has two COAs:
  - COA-A at `/storage/coaA` (published at `/public/coaA`), EVM address `0xCOA_A`
  - COA-B at `/storage/coaB` (published at `/public/coaB`), EVM address `0xCOA_B`

**Step 1 — Obtain a legitimate proof for COA-A:**
Alice signs hash `H` (arbitrary bytes, no COA address included) with her account key index 0. The resulting `COAOwnershipProof`:
```
{KeyIndices: [0], Address: 0xALICE, CapabilityPath: "coaA", Signatures: [sig_H]}
```
RLP-encode this as `proof_A`.

**Step 2 — Attacker constructs a replayed proof for COA-B:**
The attacker RLP-decodes `proof_A`, changes `CapabilityPath` from `"coaA"` to `"coaB"`, and re-encodes as `proof_B`. The `Signatures` field is unchanged.

**Step 3 — Attacker calls ERC-1271 on COA-B:**
```solidity
// called on COA-B's deployed contract at 0xCOA_B
bytes4 result = IERC1271(0xCOA_B).isValidSignature(H, proof_B);
// result == 0x1626ba7e  (ValidERC1271Signature)
```

**What happens inside the precompile:**
- `coa.sol:isValidSignature` calls `verifyCOAOwnershipProof(address(this)=0xCOA_B, H, proof_B)`
- `DecodeABIEncodedProof` sets `EVMAddress = 0xCOA_B`, `SignedData = H`, decodes `proof_B` → `CapabilityPath = "coaB"`
- `validateCOAOwnershipProof(0xALICE, /public/coaB, H, [0], [sig_H], 0xCOA_B)` is invoked
- Signature check: `sig_H` is valid over `H` under Alice's key 0 → **passes**
- COA address check: COA at `/public/coaB` has address `0xCOA_B` == `evmAddress` → **passes**
- Returns `ValidationResult(isValid: true)`

COA-B is now authenticated for hash `H` that Alice only ever signed in the context of COA-A. [7](#0-6) [8](#0-7) [9](#0-8)

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
