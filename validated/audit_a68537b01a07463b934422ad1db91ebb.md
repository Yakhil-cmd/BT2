### Title
Gateway Skips `__validate__` Signature Check for Invoke Transactions with Nonce=1 When Deploy-Account Is in Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway unconditionally skips the `__validate__` entry-point execution for any Invoke V3 transaction whose nonce equals `1` and whose sender account has on-chain nonce `0`, provided the mempool reports any transaction for that address. Because no signature verification is performed at the gateway when this path is taken, an attacker can submit an Invoke transaction carrying an arbitrary or invalid signature and have it admitted to the mempool without the account contract ever checking it.

### Finding Description

`extract_state_nonce_and_run_validations` in `StatefulTransactionValidator` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

The skip condition is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
```

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`: [2](#0-1) 

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate = false`, `ValidatableTransaction::validate_tx` returns `Ok(None)` immediately: [3](#0-2) 

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
```

The `account_tx_in_pool_or_recent_block` predicate is satisfied by any of:
- a transaction for the address currently in the pool (`tx_pool.contains_account`)
- the address appearing in the mempool's staged or committed state [4](#0-3) 

The `validate_by_mempool` call that precedes the skip check only verifies nonce ordering and duplicate-hash rejection; it performs no signature check: [5](#0-4) 

The stateless validator checks only signature *length*, not validity: [6](#0-5) 

The `StatefulTransactionValidatorConfig` defines `max_nonce_for_validation_skip` but this field is **never read** inside `skip_stateful_validations`; the nonce-1 threshold is hardcoded: [7](#0-6) 

### Impact Explanation

The gateway admits an Invoke transaction with an invalid (or entirely fabricated) signature into the mempool without running the account's `__validate__` entry point. This directly satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concretely:
- Invalid-signature Invoke transactions occupy mempool capacity, displacing legitimate transactions.
- The batcher will eventually attempt execution, call `__validate__` (with `validate: true` in default `ExecutionFlags`), observe the signature failure, and drop the transaction — but only after wasting batcher CPU and mempool slots.
- An attacker who controls one valid `DeployAccount` transaction (which itself passes `__validate_deploy__`) can submit an unbounded stream of nonce-1 Invoke transactions with arbitrary signatures, each admitted without signature verification.

### Likelihood Explanation

The precondition is low-cost: the attacker needs exactly one valid `DeployAccount` transaction in the mempool (or a recently committed block entry for the address). Once that is satisfied, the skip path is triggered for every subsequent nonce-1 Invoke from that address. The attacker pays fees only for the `DeployAccount`; the invalid Invoke transactions are admitted for free and dropped only at batcher execution time.

### Recommendation

1. **Remove or gate the skip path on a verified precondition.** The UX goal (deploy-account + invoke in one shot) can be preserved by verifying the signature offline (e.g., using the class hash from the `DeployAccount` calldata) rather than skipping verification entirely.
2. **Use the `max_nonce_for_validation_skip` config field** that is already defined in `StatefulTransactionValidatorConfig` but never read in `skip_stateful_validations`. Wire it into the condition so operators can disable the skip entirely by setting it to `Nonce(Felt::ZERO)`.
3. **Add a test** that submits a nonce-1 Invoke with an invalid signature after a `DeployAccount` is in the mempool and asserts that the gateway rejects it.

### Proof of Concept

```
1. Attacker derives a fresh contract address A and constructs a valid
   DeployAccount tx (nonce=0) for A — passes __validate_deploy__ normally.

2. Attacker submits DeployAccount(A, nonce=0) to the gateway.
   → Accepted; A is now in tx_pool.

3. Attacker constructs Invoke(sender=A, nonce=1, signature=[0xDEAD, 0xBEEF]).
   (Signature is arbitrary garbage.)

4. Gateway stateless check: signature length ≤ max_signature_length → OK.
   convert_rpc_tx_to_internal: computes tx_hash, no signature check.
   extract_state_nonce_and_run_validations:
     account_nonce = get_nonce_from_state(A) = 0   (not yet on-chain)
     validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap → OK
     validate_by_mempool: nonce 1 ≥ resolved nonce 0 → OK
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → true
       account_tx_in_pool_or_recent_block(A) → true (DeployAccount is in pool)
       returns skip_validate = true
     run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       blockifier_validator.validate(account_tx) → validate_tx returns Ok(None)
       __validate__ is NEVER called.

5. Invoke(A, nonce=1, sig=garbage) is inserted into the mempool.
   Repeat step 3 with different garbage signatures (different tx_hashes)
   to fill mempool capacity.
``` [8](#0-7) [1](#0-0)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
