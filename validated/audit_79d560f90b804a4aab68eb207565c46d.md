### Title
Floored Division in `native_per_pubdata` Causes Systematic Undercharging of Pubdata Native Resource Costs — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's L2 transaction validation, `native_per_pubdata` — the native resource units charged per byte of pubdata — is computed using floor (truncating) integer division. This mirrors the Eggs Finance rounding bug: when the division is not exact, the protocol always rounds in the user's favor, systematically undercharging for pubdata. Any unprivileged user who generates pubdata benefits from this discount, and the operator is underpaid for the actual proving cost of that pubdata.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` the conversion from pubdata price to native units per pubdata byte is:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs  line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer (floor) division. The true ratio is `pubdata_price / native_price` (a rational number), but the protocol stores only `⌊pubdata_price / native_price⌋`. Every pubdata byte is therefore charged `(pubdata_price % native_price) / native_price` fewer native units than the true cost.

The same floored value is replicated verbatim in the public API helper:

```rust
// api/src/helpers.rs  line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

A second, independent rounding-down occurs in `compute_gas_refund` when converting native consumption back to gas:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs  line 72
(native_used / native_per_gas) as i64 - (gas_used as i64)
```

`native_used / native_per_gas` is again floor division. The documentation explicitly defines `deltaGas := (nativeUsed / nativePerGas) - gasUsed` and states this adjustment ensures gas estimation covers native costs — but the floor means the user is charged one fewer gas unit whenever `native_used % native_per_gas != 0`.

Both divisions round in the user's favor (protocol receives less), exactly the wrong direction for protocol solvency.

---

### Impact Explanation

**`native_per_pubdata` undercharge (primary):**

Let `pubdata_price = k · native_price − 1` for integer `k ≥ 2`. Then:

```
native_per_pubdata = ⌊(k·native_price − 1) / native_price⌋ = k − 1
true ratio                                                   ≈ k
```

For `k = 2` (e.g., `pubdata_price = 19`, `native_price = 10`):
- `native_per_pubdata = 1` instead of `≈ 1.9`
- A transaction writing 1 000 pubdata bytes is charged **1 000 native units** instead of **1 900 native units** — a ~47 % discount on pubdata cost.
- The operator is underpaid by 900 native units × `gas_price / native_per_gas` ≈ 900 × `native_price` base tokens per such transaction.

The operator cannot recover this loss after the fact; the fee is fixed at validation time and the user's balance is only debited once.

**`delta_gas` undercharge (secondary):**

At most 1 gas unit per transaction is lost. Individually negligible, but it compounds with the pubdata undercharge and accumulates across the block.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are operator-set block-level parameters. In practice they are independent quantities (pubdata cost in base tokens vs. proving cycle cost in base tokens) and there is no enforcement that one is a multiple of the other. Any deployment where `pubdata_price % native_price ≠ 0` — the common case — triggers the undercharge on every transaction that writes pubdata. An unprivileged user needs only submit a normal transaction that touches storage (which generates pubdata) to benefit.

---

### Recommendation

Round `native_per_pubdata` **up** (ceiling division) so the protocol never charges less than the true cost:

```rust
// validation_impl.rs line 142 — replace wrapping_div with div_ceil
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427.

For `delta_gas`, round up so the protocol always recovers the full native cost in gas terms:

```rust
// refund_calculation.rs line 72 — replace / with div_ceil
(native_used.div_ceil(native_per_gas)) as i64 - (gas_used as i64)
```

Note that `native_per_gas` itself is already computed with `div_ceil` (see `validation_impl.rs` line 135), which is the correct direction for the limit; the refund path must mirror that ceiling to stay consistent.

### Proof of Concept

**Setup:**
- `native_price = 10`, `pubdata_price = 19`
- `native_per_pubdata = ⌊19/10⌋ = 1` (current code)
- `native_per_pubdata_correct = ⌈19/10⌉ = 2`

**Transaction:** writes 500 storage slots → ~500 × 64 = 32 000 pubdata bytes (rough estimate).

| | Current (floor) | Correct (ceil) |
|---|---|---|
| Native charged for pubdata | 32 000 | 64 000 |
| Operator shortfall | 32 000 native units | 0 |

At `gas_price = 100`, `native_price = 10`, `native_per_gas = ⌈100/10⌉ = 10`:
- Shortfall in base tokens ≈ 32 000 / 10 × 100 = **320 000 tokens per transaction**.
