### Title
`max_receipt_size` Limit Not Enforced After `promise_return` Mutates `output_data_receivers` - (File: `runtime/runtime/src/lib.rs`)

### Summary

The `max_receipt_size` limit is enforced when a new receipt is first created and validated, but the runtime subsequently mutates that already-validated receipt by appending `output_data_receivers` from the parent receipt (the `promise_return` path). No re-validation of the receipt size occurs after this mutation. Any unprivileged user who deploys a contract can craft a receipt that is exactly at the size limit, trigger `promise_return`, and cause an oversized receipt to be forwarded across shards. The nearcore codebase explicitly acknowledges this as a known bug (issue #12606).

### Finding Description

**Invariant:** Every receipt forwarded across shards must satisfy `borsh_size(receipt) <= max_receipt_size` (currently 4,194,304 bytes).

**Root cause — two-phase mutation without re-check:**

Phase 1 — validation: In `apply_action_receipt`, after each action executes, newly created receipts are validated:

```rust
validate_receipt(
    &apply_state.config.wasm_config.limit_config,
    receipt,
    apply_state.current_protocol_version,
    ValidateReceiptMode::NewReceipt,   // ← enforces max_receipt_size
)
``` [1](#0-0) 

Phase 2 — post-validation mutation: After all actions in the receipt have been applied, the runtime checks whether the result is `ReturnData::ReceiptIndex` (i.e., `promise_return`). If so, it appends the parent receipt's `output_data_receivers` directly into the already-validated child receipt:

```rust
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        // Modifying a new receipt instead of sending data
        match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
            ReceiptEnum::Action(new_action_receipt) | ... =>
                new_action_receipt.output_data_receivers
                    .extend_from_slice(&action_receipt.output_data_receivers()),
            ...
        }
    }
``` [2](#0-1) 

There is **no size re-check** after this mutation. The receipt that was exactly at `max_receipt_size` before the mutation now exceeds it.

**Two concrete bypass paths (both confirmed by tests):**

1. **`promise_return` path** (`test_max_receipt_size_promise_return`): Contract creates receipt C at exactly `max_receipt_size`, calls `promise_return(C)` inside a `then`-chained DAG. The runtime appends the parent's `output_data_receivers` to C, pushing it over the limit.

2. **Value-return path** (`test_max_receipt_size_value_return`): Contract returns a value of exactly `max_receipt_size` bytes. The runtime wraps it in a `DataReceipt`, which then exceeds the limit. [3](#0-2) 

**Existing guards and why they fail:**

`validate_receipt` with `ValidateReceiptMode::NewReceipt` correctly checks the size limit:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [4](#0-3) 

But this check runs **before** the `output_data_receivers` mutation. When the oversized receipt is later processed as an incoming or delayed receipt, it is validated with `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check:

```rust

The congestion control layer also has a special-case workaround that clamps the reported size of oversized receipts to `max_receipt_size` to prevent them from getting stuck:

```rust

// ...See https://github.com/near/nearcore/issues/12606

    size = max_receipt_size;

``` [6](#0-5) 

### Impact Explanation

Any account that can deploy a contract (any unprivileged user on mainnet) can produce receipts that exceed `max_receipt_size`. The corrupted value is the receipt's borsh-serialized byte length, which exceeds the 4,194,304-byte protocol limit. Consequences include:

- The `max_receipt_size` invariant is broken for the entire network; oversized receipts are forwarded across shards and stored in state.
- State witness sizes can exceed the `main_storage_proof_size_soft_limit` and `per_receipt_storage_proof_size_limit` bounds, degrading stateless validation.
- The bandwidth scheduler receives incorrect size signals (mitigated by the clamping workaround, but the workaround itself is a protocol deviation).
- The `ExistingReceipt` validation mode must permanently remain more permissive than `NewReceipt`, creating a persistent two-tier validation system.

### Likelihood Explanation

Likelihood is **Medium**. The trigger requires deploying a contract and crafting a receipt of exactly `max_receipt_size` bytes, which requires knowledge of the receipt encoding format and the limit value. However, both are public protocol constants, and the test suite demonstrates the exact construction. No special privileges are required beyond a funded account.

### Recommendation

After the `output_data_receivers` mutation at `runtime/runtime/src/lib.rs` lines 1028–1035, re-validate the size of the mutated receipt before adding it to `result.new_receipts`. If the post-mutation size exceeds `max_receipt_size`, set `result.result` to `Err(ActionErrorKind::NewReceiptValidationError(ReceiptValidationError::ReceiptSizeExceeded { ... }))` and discard the oversized receipt. This mirrors the existing guard pattern used for all other new receipts.
