### Title
Unprivileged Caller Can Permanently Freeze All Staker Rewards by Invoking `update_rewards` with `disable_rewards: true` ŌĆö (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the `Staking` contract is publicly callable with no access control. When called with `disable_rewards: true`, it unconditionally writes the current block number into the global `last_reward_block` storage slot **before** checking the flag, then returns early without distributing any rewards. Because `last_reward_block` is a single global slot that gates every reward distribution, any unprivileged caller can consume the "reward slot" for a block, causing all subsequent legitimate `update_rewards` calls in that block to revert with `REWARDS_ALREADY_UPDATED`. Repeated every block, this permanently freezes unclaimed yield for every staker and pool member in the protocol.

---

### Finding Description

`update_rewards` is the sole entry point through which per-block consensus rewards are distributed to stakers and their delegation pools. Its control flow is:

```
update_rewards(staker_address, disable_rewards):
  1. general_prerequisites()          // only checks pause flag
  2. assert current_block > last_reward_block
  3. validate staker is active & has non-zero balance
  4. last_reward_block ŌåÉ current_block   ŌåÉ STATE WRITTEN HERE
  5. if disable_rewards || is_pre_consensus: return  ŌåÉ EARLY EXIT
  6. calculate & distribute block rewards
``` [1](#0-0) 

The critical ordering flaw: `last_reward_block` is committed to storage at step 4, **before** the `disable_rewards` guard at step 5. Any caller who passes `disable_rewards: true` with a valid, active staker address will:

- Advance `last_reward_block` to the current block number, and
- Return without distributing any rewards.

Every subsequent call to `update_rewards` in the same block then hits the assertion at step 2 and reverts. [2](#0-1) 

There is no access control on `update_rewards` beyond `general_prerequisites()`, which only checks the pause flag. [3](#0-2) 

The `staker_address` argument must belong to an active staker with non-zero balance, but every staker address is publicly readable from the `stakers` storage vector. [4](#0-3) 

**Analog to the external report:** In `DynamicAfterFee`, the `targetDelta` is reset after every swap and can be front-run to make the entire output amount consumed as a fee. Here, `last_reward_block` is the analogous "reset-after-each-operation" state: an attacker front-runs the legitimate consensus call with `disable_rewards: true`, consuming the slot and leaving all stakers with zero rewards for that block.

---

### Impact Explanation

`last_reward_block` is a **single global slot** shared across all stakers. One successful griefing call per block is sufficient to deny rewards to every staker and every delegation pool member in the protocol for that block. Performed continuously, this constitutes **permanent freezing of unclaimed yield** ŌĆö a High-severity impact under the allowed scope.

The attacker gains nothing financially; the sole effect is that stakers and pool members never accumulate rewards, which also undermines the economic security model of the network (validators have no incentive to attest).

---

### Likelihood Explanation

- The function is unconditionally public; no role, key, or privileged access is required.
- Valid staker addresses are enumerable on-chain from the `stakers` Vec.
- The attacker's transaction only needs to be sequenced before the legitimate consensus `update_rewards` call in the same block ŌĆö achievable by submitting with a competitive fee on Starknet.
- The per-call cost to the attacker is a single cheap transaction; the per-block damage to the protocol is the entire epoch's worth of block rewards.

---

### Recommendation

1. **Move `last_reward_block` write after the early-exit guard.** If `disable_rewards` is true, do not advance `last_reward_block`; let the slot remain available for a legitimate call later in the same block.
2. **Add caller access control.** Restrict `update_rewards` to a trusted caller (e.g., the attestation contract, a consensus operator role, or the staker themselves) so that the `disable_rewards` path cannot be triggered by an arbitrary address.

---

### Proof of Concept

```
// Attacker script (pseudocode, one call per block)
loop every block:
    valid_staker = staking.stakers[0]          // any active staker, public data
    staking.update_rewards(valid_staker, disable_rewards=true)
    // last_reward_block is now == current_block
    // no rewards distributed
    // all subsequent update_rewards calls in this block revert with REWARDS_ALREADY_UPDATED
```

Step-by-step:

1. Attacker reads any active staker address from the public `stakers` vector.
2. At the start of each block, attacker submits `update_rewards(staker, true)`.
3. The contract validates the staker (passes), writes `last_reward_block = current_block`, then returns early ŌĆö zero rewards distributed.
4. The legitimate consensus call to `update_rewards(staker, false)` arrives later in the same block and reverts: `current_block_number > last_reward_block` is false.
5. No staker or pool member receives rewards for that block.
6. Repeated indefinitely, all unclaimed yield is permanently frozen. [5](#0-4) [2](#0-1)
