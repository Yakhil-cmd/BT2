### Title
Account Over Storage Capacity Cannot Incrementally Reduce Storage Usage - (File: `fvm/transactionStorageLimiter.go`)

### Summary
An account that is over storage capacity cannot perform incremental storage-reducing transactions because `CheckStorageLimits` requires the account's storage usage to be below capacity after **every** transaction that touches storage. This is the direct Flow analog of the Size M-09 bug: a post-operation health check that prevents incremental recovery, locking the account in a degraded state.

### Finding Description
In Flow's FVM, storage capacity is proportional to an account's FLOW token balance. After every transaction body executes, `normalExecution()` in `transactionInvoker.go` calls `CheckStorageLimits`:

```go
// transactionInvoker.go lines 350–391
err = executor.txnBodyExecutor.Execute()          // body runs, may reduce storage
...
bodySnapshot, err = executor.txnState.CommitNestedTransaction(bodyTxnId)
...
err = executor.CheckStorageLimits(               // checked AFTER body
    executor.ctx, executor.env,
    bodySnapshot,
    executor.proc.Transaction.Payer,
    maxTxFees)
```

`checkStorageLimits` in `transactionStorageLimiter.go` then evaluates the **post-transaction** storage usage against capacity for every address whose registers were written:

```go
// transactionStorageLimiter.go lines 154–162
for i, address := range addresses {
    capacity := environment.StorageMBUFixToBytesUInt(resultArray.Values[i])
    if usages[i] > capacity {
        return errors.NewStorageCapacityExceededError(address, usages[i], capacity)
    }
}
```

The check is unconditional: it does not distinguish between a transaction that **increased** storage (bad) and one that **decreased** storage but left the account still over capacity (a legitimate recovery step). If the account is still over capacity after the transaction, the entire transaction is reverted, including the storage reduction.

`getStorageCheckAddresses` includes every address whose registers appear in the `WriteSet` of the committed body snapshot. Any storage-removal operation writes to the account's registers (to update the storage index), so it is always included in the check.

### Impact Explanation
An account that is over storage capacity is locked out of performing incremental storage-reducing transactions. The only escape paths are:

1. **Receive FLOW tokens** from an external party in a single transaction to raise capacity above current usage.
2. **Remove all excess data in one transaction** — impossible if the data is spread across many independently-managed resources (e.g., a contract that stores N NFTs, each requiring a separate transaction to destroy).

If neither escape path is available, the account's stored assets are permanently inaccessible: the owner cannot destroy resources, remove keys, or perform any storage-modifying operation to recover. This is a user-asset-locking impact, not merely a performance degradation.

### Likelihood Explanation
Any account that:
- Stores a significant amount of data (NFTs, DeFi positions, contract state), **and**
- Has a FLOW token balance that can decrease (withdrawals, fee deductions, token price drops affecting collateral)

is at risk. The scenario is realistic for any production DeFi or NFT application on Flow. An account that legitimately stored data when FLOW prices were high may find itself over capacity after a price drop, with no way to incrementally free storage.

### Recommendation
The check should permit transactions that **improve** the storage situation even if the account remains over capacity. Concretely, before the transaction body executes, record the pre-transaction storage usage for each affected address. After the body executes, allow the transaction if:
- `post_usage <= capacity` (fully resolved — current passing condition), **or**
- `post_usage < pre_usage` (storage is decreasing — incremental recovery).

Only revert if `post_usage > pre_usage` while already over capacity (the account is making things worse). This mirrors the fix recommended for Size M-09: measure the ratio before and after, and revert only if the state worsens.

### Proof of Concept
1. Account A has `1000` bytes used, `800` bytes capacity (over capacity by `200` bytes).
2. Account A submits a transaction to destroy one resource, reducing storage by `50` bytes.
3. After the body executes: `950` bytes used, `800` bytes capacity — still over by `150` bytes.
4. `checkStorageLimits` evaluates `950 > 800` → returns `NewStorageCapacityExceededError`.
5. `normalExecution` returns the error; `errorExecution` reverts all body changes.
6. Account A is back to `1000` bytes used — the recovery step was entirely undone.
7. Account A must remove `≥ 200` bytes in a **single** transaction to escape, which is impossible if each resource requires its own transaction.

func (limiter TransactionStorageLimiter) getStorageCheckAddresses(
	ctx Context,
	snapshot *snapshot.ExecutionSnapshot,
	payer flow.Address,
	maxTxFees uint64,
) []flow.Address {
	// Multiple updated registers might be from the same address.  We want to
	// duplicated the addresses to reduce check overhead.
	dedup := make(map[flow.Address]struct{}, len(snapshot.WriteSet)+1)
	addresses := make([]flow.Address, 0, len(snapshot.WriteSet)+1)

	// In case the payer is not updated, include it here.  If the maxTxFees is
	// zero, it doesn't matter if the payer is included or not.
	if maxTxFees > 0 {
		dedup[payer] = struct{}{}
		addresses = append(addresses, payer)
	}

	sc := systemcontracts.SystemContractsForChain(ctx.Chain.ChainID())
	for id := range snapshot.WriteSet {
		address, ok := addressFromRegisterId(id)
		if !ok {
			continue
		}

		if limiter.shouldSkipSpecialAddress(ctx, address, sc) {
			continue
		}

		_, ok = dedup[address]
		if ok {
			continue
		}

		dedup[address] = struct{}{}
		addresses = append(addresses, address)
	}

	slices.SortFunc(
		addresses,
		func(a flow.Address, b flow.Address) int {
			// reverse order to maintain compatibility with previous
			// implementation.
			return bytes.Compare(b[:], a[:])
		})
	return addresses
}

// NewStorageCapacityExceededError constructs a new CodedError which indicates
// that an account used more storage than it has storage capacity.
func NewStorageCapacityExceededError(
	address flow.Address,
	storageUsed uint64,
	storageCapacity uint64,
) CodedError {
	return NewCodedError(
		ErrCodeStorageCapacityExceeded,
		"The account with address (%s) uses %d bytes of storage which is "+
			"over its capacity (%d bytes). Capacity can be increased by "+
			"adding FLOW tokens to the account.",
		address,
		storageUsed,
		storageCapacity)
}
