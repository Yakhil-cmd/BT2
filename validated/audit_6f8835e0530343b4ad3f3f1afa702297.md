### Title
Weight Bit-Overflow Corrupts `revoked` Flag in Account Key Metadata Encoding ŌĆö (`fvm/environment/account-key-metadata/weight_and_revoked_group.go`)

---

### Summary

`encodeWeightAndRevokedStatusGroup` packs a `uint16` weight and a boolean `revoked` flag into a single 2-byte field. The high bit (`0x8000`) is reserved for `revoked`, leaving only 15 bits for weight. The function accepts a full `uint16` weight without masking or validating it to 15 bits. Any weight value Ōēź 32768 (`0x8000`) sets the high bit, which is decoded as `revoked = true` with a silently truncated weight. No caller in the production path validates that weight fits in 15 bits before encoding.

---

### Finding Description

The encoding layout is documented as:

> value (2 bytes): revoked status is the high bit and weight is the remaining 15 bits.

The encoder:

```go
// weight_and_revoked_group.go L287-L300
func encodeWeightAndRevokedStatusGroup(runLength uint16, revoked bool, weight uint16) []byte {
    var b [weightAndRevokedStatusGroupSize]byte
    binary.BigEndian.PutUint16(b[:], runLength)
    value := weight          // ŌåÉ full uint16, no masking to 15 bits
    if revoked {
        value |= revokedMask // 0x8000
    }
    binary.BigEndian.PutUint16(b[weightAndRevokedStatusIndex:], value)
    return b[:]
}
```

The decoder:

```go
// weight_and_revoked_group.go L315-L321
func parseWeightAndRevokedStatus(b []byte, off int) (revoked bool, weight uint16) {
    weightAndRevoked := binary.BigEndian.Uint16(b[off:])
    weight  = weightAndRevoked & weightMask   // 0x7fff ŌĆö drops bit 15
    revoked = (weightAndRevoked & revokedMask) > 0  // 0x8000
    return revoked, weight
}
```

If `weight = 0x8000` (32768) is encoded:
- `value = 0x8000` (bit 15 set by the weight itself)
- Decoded: `weight = 0`, `revoked = true` ŌĆö **key appears revoked with zero weight**

If `weight = 0x83E8` (33768 = 32768 + 1000):
- Decoded: `weight = 1000`, `revoked = true` ŌĆö **key appears revoked with full weight**

The call chain that reaches this encoder without any 15-bit bound check:

1. Cadence runtime ŌåÆ `NewAccountPublicKey` (weight is `int`, no upper bound check)
2. `appendPublicKey` in `accounts.go` performs `uint16(publicKey.Weight)` ŌĆö truncates to 16 bits but does **not** check `< 0x8000`
3. `appendKeyMetadataToAccountStatusRegister` ŌåÆ `AppendAccountPublicKeyMetadata` ŌåÆ `AppendUniqueKeyMetadata` / `AppendDuplicateKeyMetadata` ŌåÆ `appendWeightAndRevokedStatus` ŌåÆ `encodeWeightAndRevokedStatusGroup`

`AccountPublicKey.Validate()` explicitly notes the missing weight check with a TODO comment:

```go
// model/flow/account.go L80
// - (TODO) It specifies a negative key weight
```

No check for weight > 32767 exists anywhere in the chain.

---

### Impact Explanation

An unprivileged transaction sender who adds a key with weight Ōēź 32768 to any account they control will have that key's metadata silently corrupted on-chain:

- The key is stored as `revoked = true` even though it was never revoked through the normal revocation path.
- The stored weight is `weight & 0x7fff`, not the intended value.

Concrete consequences:
1. **Phantom revocation**: A freshly added key immediately appears revoked to all on-chain and off-chain consumers (e.g., `validateCOAOwnershipProof` in `contract.cdc` rejects revoked keys; off-chain indexers tracking key state are misled).
2. **Weight corruption**: A key encoded with weight = 33768 (0x83E8) is decoded as weight = 1000, revoked = true. The attacker can craft a key that appears to have any weight Ōēż 32767 while also appearing revoked ŌĆö or weight = 0 while appearing revoked.
3. **Authorization bypass potential**: If a multi-sig account relies on key weight thresholds and an attacker can add keys to that account (e.g., via a shared contract account), corrupted weight values could shift the effective signing weight in unexpected ways.
4. **Irreversibility**: Flow has no "unrevoke" operation. Once the metadata is written with `revoked = true`, the key is permanently unusable for signing, even though the account owner never issued a revocation.

---

### Likelihood Explanation

- Any Flow transaction can call `AuthAccount.keys.add(publicKey:hashAlgorithm:weight:)` with an arbitrary integer weight.
- The Cadence `weight` parameter is typed as `UFix64` in the stdlib contract but is converted to `int` in the Go runtime layer with no upper-bound enforcement before reaching the encoder.
- No existing guard in `NewAccountPublicKey`, `appendPublicKey`, or `encodeWeightAndRevokedStatusGroup` rejects weight Ōēź 32768.
- The bug is triggered by a single ordinary transaction ŌĆö no privileged access, no staked node, no quorum required.

---

### Recommendation

Mask or validate the weight to 15 bits before encoding, making the library self-contained:

```go
func encodeWeightAndRevokedStatusGroup(runLength uint16, revoked bool, weight uint16) []byte {
    var b [weightAndRevokedStatusGroupSize]byte
    binary.BigEndian.PutUint16(b[:], runLength)
    // Enforce 15-bit limit; weight must not set the revoked flag bit
    value := weight & weightMask  // 0x7fff
    if revoked {
        value |= revokedMask
    }
    binary.BigEndian.PutUint16(b[weightAndRevokedStatusIndex:], value)
    return b[:]
}
```

Additionally, add an explicit validation in `appendWeightAndRevokedStatus` and in `AccountPublicKey.Validate()` / `NewAccountPublicKey` to reject weight > `weightMask` (32767) with a clear error, rather than silently truncating.

### Proof of Concept

A transaction adds a key with weight = 32768 to an account:

