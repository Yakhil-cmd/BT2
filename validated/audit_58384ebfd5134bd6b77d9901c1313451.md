### Title
Hardcoded `L1_TX_NATIVE_PRICE` Creates Stale Native Resource Budget for L1→L2 Transactions — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is a compile-time constant (`= 10`) used exclusively to compute the native resource budget for every L1→L2 priority transaction. Unlike L2 transactions — where `native_price` is a per-block operator-supplied value — L1→L2 transactions bypass the operator's live `native_price` and always divide `gas_price` by this hardcoded 10. The constant carries an explicit `TODO (EVM-1157): find a reasonable value for it.` comment, acknowledging it has never been properly calibrated. When the operator's live `native_price` diverges from 10, the native budget allocated to L1→L2 transactions becomes either severely over-provisioned (attacker subsidy / block-native DoS) or under-provisioned (L1→L2 execution failures for legitimate users).

---

### Finding Description

The native resource limit for any transaction is:

```
nativePerGas  = gasPrice / nativePrice
nativeLimit   = gasLimit * nativePerGas
```

For L2 transactions, `nativePrice` is read from `BlockMetadataFromOracle.native_price`, which the operator updates every block to reflect actual proving costs.

For L1→L2 transactions, `prepare_and_check_resources` ignores the operator's value and substitutes the hardcoded constant:

```rust
// basic_bootloader/src/bootloader/constants.rs:64-66
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
```

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs:453-454
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
```

The documentation explicitly states this design choice:

> `nativePrice` be a constant set by the operator … **Note: for L1→L2 transactions we use a code constant instead of one provided by operator.**

`L1_TX_NATIVE_PRICE` is a `pub const` in Rust; it cannot be changed at runtime. Changing it requires a full protocol upgrade and redeployment.

**Scenario A — Over-provisioning (operator raises `native_price` above 10):**

If proving costs increase and the operator sets `native_price = 1000` for L2 txs, L1→L2 txs still compute `nativePerGas = gasPrice / 10`, giving them 100× more native resources per gas unit than L2 txs at the same `gasPrice`. An unprivileged attacker can submit L1→L2 transactions containing computationally expensive calldata/operations, consuming the block's native resource budget at 1/100th the market rate. Once the block's native limit is exhausted, no further transactions (L1 or L2) can be included, halting block progress.

**Scenario B — Under-provisioning (operator lowers `native_price` below 10):**

If proving becomes cheaper and the operator sets `native_price = 1`, L1→L2 txs compute `nativePerGas = gasPrice / 10`, giving them 10× fewer native resources than L2 txs. Any L1→L2 transaction whose execution consumes more native resources than this reduced budget will revert. Since L1→L2 transactions cannot be invalidated or cancelled once in the priority queue, the user's intended operation fails and the deposited funds are returned minus gas fees — the transaction is permanently stuck in a failed state with no recourse.

---

### Impact Explanation

**Scenario A (over-provisioning):** An unprivileged attacker can submit L1→L2 transactions that consume the block's native resource budget at a fraction of the market rate. This can prevent other transactions from being included in the block, constituting a resource-exhaustion DoS against the sequencer. The attacker's cost is bounded only by the L1 gas price, not by the actual proving cost.

**Scenario B (under-provisioning):** Legitimate users' L1→L2 transactions revert during execution due to native resource exhaustion. The user's deposited value is returned, but the intended operation (e.g., token bridge, contract call) is not executed. Because L1→L2 transactions are irrevocable once submitted to the priority queue, there is no mechanism to retry with a higher native budget without submitting a new transaction on L1.

---

### Likelihood Explanation

The operator adjusts `native_price` per block in production (confirmed by the block re-executor reading `block_metadata.result.native_price` and the `BlockMetadataFromOracle.native_price` field). Any divergence between the live operator `native_price` and the hardcoded `10` — which the TODO comment acknowledges has not been calibrated — triggers one of the two scenarios above. The likelihood is **medium-high**: the constant is explicitly flagged as unresolved, and the operator's `native_price` is expected to change over time as proving hardware and workloads evolve.

---

### Recommendation

Replace the compile-time constant with a runtime-configurable value that the operator can update without a protocol upgrade:

1. Add `l1_tx_native_price: U256` to `BlockMetadataFromOracle` (alongside the existing `native_price`).
2. In `prepare_and_check_resources`, read `system.metadata.l1_tx_native_price()` instead of the hardcoded `L1_TX_NATIVE_PRICE`.
3. Enforce on L1 that the submitted `gasPrice` satisfies `gasPrice / l1_tx_native_price >= minimum_native_per_gas` before accepting the transaction into the priority queue, so that the L2-side budget is always sufficient.

This mirrors the fix recommended for `WormholeTransceiver`: make the resource-pricing parameter mutable and keep it synchronized with actual costs.
