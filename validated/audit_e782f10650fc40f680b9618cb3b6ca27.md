### Title
Unprivileged Caller Can Permanently Suppress Block Rewards via `disable_rewards=true` in `update_rewards` — (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable and accepts an attacker-controlled `disable_rewards: bool` parameter. Any non-zero address can invoke `update_rewards(valid_staker, true)` to advance the global `last_reward_block` checkpoint without distributing block rewards, permanently preventing reward distribution for that block for all stakers.

---

### Finding Description

`StakingRewardsManagerImpl::update_rewards` is exposed as a public ABI function gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero. [1](#0-0) 

The function unconditionally writes `current_block_number` to the global `last_reward_block` storage slot **before** inspecting the `disable_rewards` flag: [2](#0-1) 

When `disable_rewards=true`, execution returns immediately after that write, skipping the entire reward calculation and distribution path (`_update_rewards`, `update_unclaimed_rewards_from_staking_contract`, pool reward transfers). [3](#0-2) 

Because `last_reward_block` is a single global value, any subsequent legitimate call to `update_rewards` for the same block reverts with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The block rewards that were never credited are permanently unrecoverable — the reward supplier's `unclaimed_rewards` is never incremented, and no staker or pool receives the yield for that block.

---

### Impact Explanation

In the consensus rewards phase (post `consensus_rewards_first_epoch`), every block carries a STRK and BTC block reward computed from the minting curve. An attacker who calls `update_rewards(any_valid_staker, true)` once per block suppresses 100 % of block rewards for all stakers and all delegation pools for that block. Repeated across many blocks this constitutes **permanent freezing of unclaimed yield** for the entire protocol — matching the High impact tier.

---

### Likelihood Explanation

- The attacker needs only a valid, active staker address, which is publicly observable from `NewStaker` events or the `stakers` vector.
- No privileged role, leaked key, or external dependency is required.
- The call costs only gas; the attacker has no profit motive but can cause continuous protocol damage.
- The attack is executable on every block once consensus rewards are active.

---

### Recommendation

Restrict the `disable_rewards` parameter to a privileged caller (e.g., only the staker themselves, or a governance/operator role), or remove it from the public interface entirely and handle the pre-consensus early-return solely via `is_pre_consensus()`. The simplest fix is to assert that `get_caller_address() == staker_address` when `disable_rewards=true`, or to make `disable_rewards` an internal-only concern.

---

### Proof of Concept

1. Consensus rewards are active (`current_epoch >= consensus_rewards_first_epoch`).
2. Attacker observes any valid, active staker address `S` from on-chain events.
3. At the first transaction of block `B`, attacker calls:
   ```
   staking.update_rewards(S, disable_rewards=true)
   ```
4. `last_reward_block` is written to `B`; the function returns without distributing rewards.
5. The legitimate sequencer/staker call `update_rewards(S, false)` for block `B` reverts with `REWARDS_ALREADY_UPDATED`.
6. Block `B`'s STRK and BTC rewards are permanently lost — `unclaimed_rewards` in `RewardSupplier` is never incremented for this block, and no staker or pool receives yield.
7. Repeating step 3 every block suppresses all consensus block rewards indefinitely.
