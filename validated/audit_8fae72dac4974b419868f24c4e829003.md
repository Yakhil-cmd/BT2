### Title
Missing Refund Callback on Failed `ft_transfer` When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is absent (it is **not** a default feature), the `ExitToNear` precompile schedules a bare `PromiseArgs::Create` for the NEP-141 `ft_transfer` with no callback. If that promise fails — e.g., because the recipient is not registered with the NEP-141 contract — the ERC-20 tokens that were already burned are never re-minted, and the corresponding NEP-141 balance remains locked in Aurora's account with no user-accessible recovery path.

### Finding Description

**Step 1 — Feature flag is not a default.**

`engine/Cargo.toml` and `engine-precompiles/Cargo.toml` both list `default = ["std"]`; `error_refund` is an opt-in feature. [1](#0-0) [2](#0-1) 

**Step 2 — `refund` is hardcoded to `None` when the feature is off.**

Inside `ExitToNear::run()`, the `callback_args` struct is built with `refund: None` unconditionally when `error_refund` is disabled, and `transfer_near` is also `None` for the standard ERC-20 `ft_transfer` path (the `_` arm of `exit_erc20_token_to_near` returns `None` for `transfer_near_args`). [3](#0-2) [4](#0-3) 

**Step 3 — `callback_args == default()` suppresses the callback entirely.**

The gate at line 470 compares `callback_args` to `ExitToNearPrecompileCallbackArgs::default()` (both `Option` fields are `None`). When the comparison is true, only `PromiseArgs::Create(transfer_promise)` is emitted — no `PromiseArgs::Callback` wrapper, so `exit_to_near_precompile_callback` is never scheduled. [5](#0-4) [6](#0-5) 

**Step 4 — Callback logic that would re-mint is unreachable.**

`exit_to_near_precompile_callback` contains the only re-mint path (`engine::refund_on_error`), reached only when `args.refund` is `Some(...)`. Since no callback is ever scheduled, this code is dead for the affected flow. [7](#0-6) [8](#0-7) 

**Step 5 — The test suite explicitly confirms the fund loss.**

`test_exit_to_near_refund` calls `exit_to_near` targeting `"unregistered.near"` (which causes `ft_transfer` to fail) and then asserts that, without `error_refund`, the ERC-20 balance is permanently reduced by `FT_EXIT_AMOUNT` with no re-mint: [9](#0-8) 

The same pattern is confirmed for ETH exits in `test_exit_to_near_eth_refund`. [10](#0-9) 

### Impact Explanation

An unprivileged EVM user calls `withdrawToNear` (or any ERC-20 exit) targeting a NEAR account that is not registered with the NEP-141 contract. The ERC-20 contract burns the tokens and calls the precompile. The precompile emits only a bare `ft_transfer` promise. That promise fails on-chain. Because no callback was attached, no re-mint is triggered. The user's ERC-20 balance is permanently decremented, and the NEP-141 tokens remain locked in Aurora's account. **Impact: High — temporary (potentially permanent) freezing of user funds.**

### Likelihood Explanation

- `error_refund` is not a default feature; any build that omits it (including the default `cargo build`) is affected.
- The triggering condition (recipient not registered with the NEP-141 contract) is a normal, user-reachable scenario requiring no special privilege.
- The attacker only needs to call the standard ERC-20 `withdrawToNear` interface with a valid but unregistered NEAR account ID.

### Recommendation

Enable `error_refund` as a **default** feature in both `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`, or restructure the callback-scheduling logic so that a callback is always attached for ERC-20 exits regardless of the feature flag, ensuring a re-mint is triggered on any `ft_transfer` failure.

### Proof of Concept

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` already proves the scenario when compiled **without** `--features error_refund`:

1. Bridge NEP-141 tokens to Aurora ERC-20.
2. Call `exit_to_near` targeting `"unregistered.near"`.
3. Assert `erc20_balance == FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (tokens burned, not refunded).
4. Assert `nep_141_balance_of(aurora) == FT_TRANSFER_AMOUNT` (NEP-141 tokens stuck in Aurora's account).

This matches the `#[cfg(not(feature = "error_refund"))]` branch already present in the test, confirming the fund freeze without any code modification. [11](#0-10)
