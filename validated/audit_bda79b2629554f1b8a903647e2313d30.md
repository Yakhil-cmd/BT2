Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Breaks ICP/Cycles Conservation Invariant - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the CMC is intentionally designed to return `()` and swallow all errors, as documented by its own comment. When called after a successful cycles deposit in `process_top_up`, `process_mint_cycles`, and `process_create_canister`, any failure of the ICP ledger burn call results in cycles being permanently minted without a corresponding ICP burn. The ICP remains locked in the CMC's per-user subaccount with no recovery path, and the block is marked processed so the notification cannot be retried.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` has the signature `async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens)` — it returns `()` by design. The comment at line 2015–2016 explicitly states: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."*

The three call sites follow the same pattern:

- `process_create_canister` (L1945): after `do_create_canister` succeeds
- `process_mint_cycles` (L1968): after `do_mint_cycles` succeeds  
- `process_top_up` (L2001): after `deposit_cycles` succeeds

In all three, cycles are irrevocably deposited to the recipient first. Then `burn_and_log` calls `call_protobuf(ledger_canister_id, "send_pb", send_args).await` (L2040). If this inter-canister call fails (e.g., ledger mid-upgrade, CMC cycles exhaustion, transient rejection), the error is only logged (L2044–2047) and the function returns `()`. The caller proceeds to return `Ok(...)`. Because the notification is now considered processed, the same block index cannot be re-notified. The ICP sits permanently in the CMC subaccount with no automated recovery.

The design rationale — preventing retry-based double minting — is sound in intent, but the chosen implementation creates a permanent conservation break on any burn failure rather than a recoverable inconsistency.

## Impact Explanation
This matches the High impact class: *"Significant... ledger... security impact with concrete user or protocol harm."* Cycles are minted without a corresponding ICP burn, breaking the ICP/cycles conservation invariant that underpins the exchange rate. The unburned ICP is permanently locked in the CMC subaccount (inaccessible to anyone), while the equivalent cycles circulate freely. Repeated exploitation across multiple transactions during a ledger upgrade window could accumulate a meaningful unburned ICP balance, inflating the effective circulating cycles supply relative to the burned ICP backing it. The impact is permanent and unrecoverable per affected transaction.

## Likelihood Explanation
An unprivileged user cannot directly force the ICP ledger to be unavailable, but ledger upgrades are routine governance events that temporarily stop the canister. A user who monitors NNS upgrade proposals can time a `notify_top_up` or `notify_mint_cycles` call to the upgrade window with no special privileges beyond holding ICP. The cost is only the ICP sent (which the attacker retains in the CMC subaccount, unburned). The attack is repeatable across upgrade windows and scales linearly with the amount sent.

## Recommendation
The existing design tradeoff (swallow burn errors to prevent retry-based double minting) is understandable but the chosen resolution is too aggressive. Preferred fixes:

1. **Persist and retry**: Record a "pending burn" entry in CMC state on burn failure and retry via a timer task (consistent with how the CMC handles other deferred operations). This preserves the no-retry guarantee while eventually completing the burn.
2. **Propagate and halt**: Return an error from `burn_and_log`, but only after verifying the cycles deposit is idempotent/reversible — if the cycles ledger deduplicates by block index, a retry would safely no-op the deposit and successfully complete the burn.
3. **At minimum, emit a certified metric or alert**: If neither retry nor revert is feasible, the CMC should expose the stuck subaccount balance as a certified state variable so operators can detect and manually remediate conservation breaks.

## Proof of Concept
1. User sends N ICP to the CMC subaccount for canister `C` (`MEMO_TOP_UP_CANISTER`).
2. An NNS upgrade proposal for the ICP ledger is submitted and executed (public, observable on-chain).
3. During the ledger's stopped/upgrading window, user calls `notify_top_up` for canister `C`.
4. CMC executes `deposit_cycles(C, cycles, ...)` — succeeds; `C` receives cycles.
5. CMC calls `burn_and_log`; the `send_pb` call to the ledger returns a rejection (canister stopped).
6. `burn_and_log` logs the error and returns `()`.
7. `process_top_up` returns `Ok(cycles)` to the user; block index is marked processed.
8. Result: `C` holds new cycles; N ICP remains in the CMC subaccount unburned; the block cannot be re-notified. Conservation invariant is permanently broken for this transaction.

A deterministic integration test can reproduce this by deploying a mock ICP ledger that rejects `send_pb` calls, running `notify_top_up`, and asserting that (a) the cycles were deposited, (b) the CMC subaccount balance is non-zero, and (c) re-notification returns `AlreadyProcessed`.
