### Title
Unprivileged Caller Can Permanently Suppress Block Rewards via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

### Summary
`IStakingRewardsManager::update_rewards` is a public function with no caller restriction. It accepts a `disable_rewards: bool` parameter. Any unprivileged address can call it with `disable_rewards: true` for any staker, consuming the one-call-per-block slot while skipping reward distribution, permanently destroying that block's rewards for the targeted staker.

### Finding Description
`update_rewards` is exposed as a public ABI function under `StakingRewardsManagerImpl`: [1](#0-0) 

The only gate is `general_prerequisites`, which exclusively checks that the contract is not paused and the caller is not the zero address: [2](#0-1) 

There is no check that the caller is the staker, the staker's operational address, or any privileged role.

Inside the function, the very first state-mutating action is writing the current block number to `last_reward_block`: [3](#0-2) 

Immediately after, if `disable_rewards` is `true`, the function returns without distributing any rewards: [4](#0-3) 

Because `last_reward_block` is already updated, any subsequent legitimate call in the same block fails with `REWARDS_ALREADY_UPDATED`: [5](#0-4) 

The block rewards that were skipped are never re-queued or recoverable; they are permanently lost.

### Impact Explanation
An attacker calling `update_rewards(victim_staker, disable_rewards: true)` every block for a target staker permanently prevents that staker (and their delegators) from accumulating consensus block rewards. The rewards are not deferred — the per-block reward calculation window is consumed and discarded. This constitutes **permanent freezing of unclaimed yield** (High impact).

### Likelihood Explanation
The call is cheap (no token transfer, no approval needed) and requires no special role or stake. A griever can automate this for every block against any staker. Front-running is not even required — calling it first in any block is sufficient. Likelihood is **High**.

### Recommendation
Add a caller restriction to `update_rewards`. Only the staker's registered operational address (or the staker address itself) should be permitted to call this function with `disable_rewards: true`. Alternatively, remove the `disable_rewards` parameter from the public interface entirely and handle the disable-rewards logic internally based on on-chain state.

### Proof of Concept
1. Staker `S` is registered with operational address `O`.
2. Attacker `A` (any address) calls `update_rewards(S, disable_rewards: true)` at block `N`.
3. `last_reward_block` is set to `N`; no rewards are distributed.
4. Operational address `O` attempts to call `update_rewards(S, disable_rewards: false)` in the same block `N`.
5. The call reverts with `REWARDS_ALREADY_UPDATED` because `current_block_number == last_reward_block`.
6. Staker `S` and all its delegators receive zero rewards for block `N`.
7. Attacker repeats every block — staker earns no consensus rewards indefinitely.
