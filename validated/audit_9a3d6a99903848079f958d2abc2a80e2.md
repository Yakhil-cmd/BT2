### Title
Gateway Stateless Validator Unconditionally Applies L2 Gas Price Minimum to All Transactions, Incorrectly Rejecting Valid L1-Gas-Only and Client-Side Proving Transactions — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` applies a `min_gas_price` floor check against `l2_gas.max_price_per_unit` for **every** transaction type without exception. This check was designed for L2-gas-bearing transactions but is incorrectly applied to all transactions, including those that legitimately carry a zero L2 gas price (L1-gas-only transactions and client-side proving transactions). In production, where `min_gas_price` defaults to `8_000_000_000`, any transaction with `l2_gas.max_price_per_unit = 0` is rejected at the gateway even when it is otherwise valid.

---

### Finding Description

In `validate_resource_bounds`, after the zero-bounds guard, the following unconditional check fires for every transaction type (Invoke, Declare, DeployAccount):

```rust
// line 71
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

The production default is `min_gas_price: 8_000_000_000`.

A transaction that sets only `l1_gas` (or only `l1_data_gas`) and leaves `l2_gas` at its zero default passes the first guard (because `max_possible_fee(Tip::ZERO)` is non-zero from the L1 gas component) but immediately fails the second guard because `l2_gas.max_price_per_unit = 0 < 8_000_000_000`.

The same applies to client-side proving transactions: `validate_zero_fee_resource_bounds` in `crates/starknet_transaction_prover/src/proving/virtual_snos_prover.rs` explicitly **requires** all `max_price_per_unit` fields to be zero. Such transactions are accepted by the OS and appear in real blocks (e.g., `block_post_0_14_2.json` contains invoke transactions with `L2_GAS.max_price_per_unit = 0x0` and non-empty `proof_facts`), yet the stateless validator would reject them before they ever reach proof verification.

The unit test `valid_l1_gas` only passes because `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` sets `min_gas_price: 0`, masking the production failure entirely.

By contrast, the stateful validator's analogous check (`validate_tx_l2_gas_price_within_threshold`) correctly handles this case by matching on `ValidResourceBounds::L1Gas(_)` and explicitly skipping the check with the comment `// No validation required for legacy transactions.` — but the stateless validator has no such guard.

---

### Impact Explanation

Any transaction with `l2_gas.max_price_per_unit = 0` is permanently rejected at the gateway stateless path with `MaxGasPriceTooLow`. Affected classes:

1. **L1-gas-only transactions** — valid V3 transactions that allocate only L1 gas, leaving L2 gas price at zero.
2. **Client-side proving (SNOS) transactions** — explicitly require all `max_price_per_unit` fields to be zero; these are real on-chain transactions that the sequencer's own prover produces.

Impact tier: **High — Mempool/gateway admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The production `StatelessTransactionValidatorConfig::default()` sets `validate_resource_bounds: true` and `min_gas_price: 8_000_000_000`. Any user or protocol component submitting a transaction with zero L2 gas price hits this rejection unconditionally. Client-side proving is an explicitly supported feature (`allow_client_side_proving: true` by default), making this a systematic rejection of an entire transaction class.

---

### Recommendation

Guard the `min_gas_price` check so it only fires when the transaction actually intends to use L2 gas:

```rust
if resource_bounds.l2_gas.max_price_per_unit.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
```

Alternatively, mirror the stateful validator's pattern and skip the check entirely when `l2_gas.max_amount == 0 && l2_gas.max_price_per_unit == 0`.

---

### Proof of Concept

Using the existing test harness with the **production** config:

```rust
let config = StatelessTransactionValidatorConfig::default();
// min_gas_price = 8_000_000_000, validate_resource_bounds = true

let tx = rpc_tx_for_testing(
    TransactionType::Invoke,
    RpcTransactionArgs {
        resource_bounds: AllResourceBounds {
            l1_gas: NON_EMPTY_RESOURCE_BOUNDS, // non-zero L1 gas
            // l2_gas defaults to { max_amount: 0, max_price_per_unit: 0 }
            ..Default::default()
        },
        ..Default::default()
    },
);

let validator = StatelessTransactionValidator { config };
// Returns Err(MaxGasPriceTooLow { gas_price: 0, min_gas_price: 8_000_000_000 })
// even though the transaction is valid and would be accepted by the OS.
assert!(validator.validate(&tx).is_err());
```

The same failure occurs for any invoke transaction carrying `proof_facts` with `l2_gas.max_price_per_unit = 0`, which is the exact format produced by the virtual SNOS prover and observed in `block_post_0_14_2.json`.

    let bounds = &tx.resource_bounds;
    let mut violations = Vec::new();

    if bounds.l1_gas.max_price_per_unit != GasPrice(0) {
        violations
            .push(format!("l1_gas.max_price_per_unit = {}", bounds.l1_gas.max_price_per_unit.0));
    }
    if bounds.l2_gas.max_price_per_unit != GasPrice(0) {
        violations
            .push(format!("l2_gas.max_price_per_unit = {}", bounds.l2_gas.max_price_per_unit.0));
    }
    if bounds.l1_data_gas.max_price_per_unit != GasPrice(0) {
        violations.push(format!(
            "l1_data_gas.max_price_per_unit = {}",
            bounds.l1_data_gas.max_price_per_unit.0
        ));
    }
    if tx.tip != Tip(0) {
        violations.push(format!("tip = {}", tx.tip.0));
    }

    if !violations.is_empty() {
        return Err(VirtualSnosProverError::InvalidTransactionInput(format!(
            "Proving is client-side — no fees are charged. The following fields must be zero but \
             were not: [{}]. Set all max_price_per_unit fields and tip to 0x0. Note: max_amount \
             fields are fine to set — l2_gas.max_amount controls the gas limit enforced by the OS \
             (use the value from starknet_estimateFee, or 100000000 as a safe upper bound). \
             l1_gas.max_amount and l1_data_gas.max_amount do not affect OS execution.",
            violations.join(", ")
        )));
    }
