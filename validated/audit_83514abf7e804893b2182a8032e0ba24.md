### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Is Called With No Pending Mint Requests — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary
`RewardSupplier.tick()` is `payable` and accepts ETH to cover L1→L2 messaging fees. However, when there are no pending L2 mint requests (`amountToMint == 0`), the entire fee-spending block is skipped and any ETH sent with the call is permanently locked in the contract. No `receive()`, `fallback()`, or withdrawal function exists anywhere in the L1 contract suite to recover it.

---

### Finding Description
`tick()` is declared `external payable` and splits `msg.value` between two fee-bearing calls inside an `if (amountToMint > 0)` guard:

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        ...
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // ← if amountToMint == 0, msg.value is silently accepted and never spent
}
``` [1](#0-0) 

When `amountToMint == 0`, execution falls through the function with no use of `msg.value`. The ETH is credited to the contract's balance with no mechanism to recover it — a grep across all L1 Solidity files confirms zero `receive()`, `fallback()`, or `withdraw` definitions exist.

---

### Impact Explanation
Any ETH sent to `tick()` during a no-op call is permanently frozen in the `RewardSupplier` contract. This constitutes **griefing with damage to the caller / protocol operator** (Medium impact per the allowed scope). The ETH is not user staking principal (STRK), but it is real ETH paid as L1→L2 message fees, and it is irrecoverable.

---

### Likelihood Explanation
`tick()` has no access control — any address may call it. A realistic race condition exists: a keeper observes pending L2 mint requests via `requiredMinting()`, constructs a `tick()` call with ETH, but a competing transaction processes those messages first. By the time the keeper's transaction executes, `amountToMint == 0` and the ETH is locked. This is a standard mempool race condition requiring no special privileges. [2](#0-1) 

---

### Recommendation
Add a guard that reverts when ETH is sent but there is nothing to process:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        require(msg.value == 0, "NO_MINT_REQUIRED_ETH_REJECTED");
        return;
    }
    ...
}
```

Alternatively, refund unused ETH at the end of the function, or add a permissioned ETH-recovery function.

---

### Proof of Concept
1. L2 `RewardSupplier` has one pending `mintRequest` message on L1.
2. Keeper A calls `tick{value: 0.01 ether}()` — transaction is in the mempool.
3. Keeper B's `tick{value: 0}()` is mined first, consuming the message.
4. Keeper A's transaction executes: `requiredMinting()` returns `(0, 0)`, the `if` block is skipped, `0.01 ether` is credited to `RewardSupplier`'s balance.
5. No function exists to recover the ETH. It is permanently locked. [3](#0-2)
