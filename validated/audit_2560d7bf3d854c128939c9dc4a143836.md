### Title
`DeleteAccountAction` with `beneficiary_id == account_id` silently burns the entire account balance — (`File: runtime/runtime/src/actions.rs`)

---

### Summary

`action_delete_account` in nearcore's runtime creates a balance-refund receipt addressed to `delete_account.beneficiary_id`, then immediately removes the account from state. No guard checks whether `beneficiary_id` equals `account_id`. When they are equal the refund receipt is addressed to a now-deleted account; the system-refund transfer fails, and the protocol's own rule ("if a refund fails, the amount is burnt") destroys the entire balance.

---

### Finding Description

`action_delete_account` executes in two sequential steps with no self-referential guard:

**Step 1 – enqueue balance refund to beneficiary** (before account removal): [1](#0-0) 

**Step 2 – remove the account from state**: [2](#0-1) 

`Receipt::new_balance_refund` constructs a system-origin Transfer receipt (`predecessor_id = "system"`) addressed to `beneficiary_id`: [3](#0-2) 

When `beneficiary_id == account_id`, that receipt is routed to the account that was just deleted. On execution, `check_account_existence` reaches `check_transfer_to_nonexisting_account`. Because the predecessor is `"system"`, `implicit_account_creation_eligible` is false (the code comment at line 843 explicitly states *"Refunds don't automatically create accounts"* and *"Account deletion with beneficiary creates a refund, so it'll not create a new account"*), so the transfer returns `AccountDoesNotExist`: [4](#0-3) 

The protocol specification then burns the failed refund amount:

> "If the execution of a refund fails, the refund amount is burnt." [5](#0-4) 

No validation in `action_validation.rs` or `action_delete_account` rejects or redirects the case `beneficiary_id == account_id`: [6](#0-5) 

---

### Impact Explanation

**Accounting scope — permanent, irreversible balance destruction.**

The entire `account.amount()` of the deleted account is burned. The total NEAR supply is reduced by that amount without any corresponding fee or intentional burn. The invariant stated in the protocol documentation — *"the tokens are transferred to `beneficiary_id`"* — is violated: [7](#0-6) 

---

### Likelihood Explanation

The trigger is a single, valid, user-submitted `DeleteAccountAction` with `beneficiary_id` set to the same value as the account being deleted. This is reachable via:

1. **Direct transaction** — any account owner signs `DeleteAccount { beneficiary_id: self }`.
2. **Contract-initiated deletion** — a contract calls `promise_batch_action_delete_account` with `current_account_id()` as the beneficiary: [8](#0-7) 

The default `delete_account` helper in the integration-test user API already passes `signer_id` as both `receiver_id` and `beneficiary_id`, meaning any caller who passes the same account for both fields hits this path: [9](#0-8) 

No privileged role is required. The input is fully unprivileged-user-controlled.

---

### Recommendation

Add a guard at the start of `action_delete_account` (or in `validate_action` for `DeleteAccount`) that rejects or redirects the self-referential case:

```rust
if &delete_account.beneficiary_id == account_id {
    // Either return an error, or redirect the balance to the predecessor.
    result.result = Err(ActionErrorKind::DeleteAccountWithSelfBeneficiary {
        account_id: account_id.clone(),
    }.into());
    return Ok(());
}
```

Alternatively, redirect the balance to `receipt.predecessor_id()` when `beneficiary_id == account_id`, matching the intent that the balance should not be destroyed. [10](#0-9) 

---

### Proof of Concept

```
1. Account "alice.near" holds 100 NEAR.
2. Alice signs and submits:
     Transaction {
       signer_id:   "alice.near",
       receiver_id: "alice.near",
       actions: [DeleteAccount { beneficiary_id: "alice.near" }]
     }
3. action_delete_account runs:
   a. account_balance = 100 NEAR
   b. new_receipts.push(Receipt::new_balance_refund("alice.near", 100 NEAR))
   c. remove_account(state_update, "alice.near")   // alice deleted
4. The refund receipt is processed:
   a. receiver_id = "alice.near" — account does not exist
   b. check_transfer_to_nonexisting_account → AccountDoesNotExist (system refund, no implicit creation)
   c. Receipt fails → 100 NEAR is burnt per protocol rule
5. Result: alice's 100 NEAR is permanently destroyed; no account receives it.
```

pub(crate) fn action_delete_account(
    state_update: &mut TrieUpdate,
    account: &mut Option<Account>,
    actor_id: &mut AccountId,
    receipt: &Receipt,
    result: &mut ActionResult,
    account_id: &AccountId,
    delete_account: &DeleteAccountAction,
    config: &RuntimeConfig,
    current_protocol_version: ProtocolVersion,
) -> Result<(), StorageError> {
    let account_ref = account.as_ref().unwrap();
    let account_storage_usage = if ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage
        .enabled(current_protocol_version)
    {
        let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
        account_ref.storage_usage().saturating_sub(contract_storage)
    } else {
        // Legacy behavior: only subtracts local contract code, misses the
        // global contract identifier overhead.
        let account_storage_usage = account_ref.storage_usage();
        let code_len = get_code_len_or_default(
            state_update,
            account_id.clone(),
            account_ref.local_contract_hash().unwrap_or_default(),
        )?;
        debug_assert!(
            code_len == 0 || account_storage_usage > code_len,
            "account storage usage should be larger than code size. storage usage: {}, code size: {}",
            account_storage_usage,
            code_len
        );
        account_storage_usage.saturating_sub(code_len)
    };
    if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
        result.result =
            Err(ActionErrorKind::DeleteAccountWithLargeState { account_id: account_id.clone() }
                .into());
        return Ok(());
    }
    let gas_key_balance_to_burn = compute_gas_key_balance_sum(state_update, account_id)?;
    if gas_key_balance_to_burn > GasKeyInfo::MAX_BALANCE_TO_BURN {
        result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh {
            account_id: account_id.clone(),
            public_key: None,
            balance: gas_key_balance_to_burn,
        }
        .into());
        return Ok(());
    }
    // We use current amount as a pay out to beneficiary.
    let account_balance = account_ref.amount();
    if account_balance > Balance::ZERO {
        result
            .new_receipts
            .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
    }
    let remove_result = remove_account(state_update, account_id)?;
    result.tokens_burnt =
        result.tokens_burnt.checked_add(gas_key_balance_to_burn).ok_or_else(|| {
            StorageError::StorageInconsistentState("tokens_burnt overflow".to_string())
        })?;
    if remove_result.gas_key_nonce_count > 0 {
        let compute = storage_removes_compute(
            &config.wasm_config.ext_costs,
            remove_result.gas_key_nonce_count,
            remove_result.gas_key_nonce_total_key_bytes,
            AccessKey::NONCE_VALUE_LEN * remove_result.gas_key_nonce_count,
        );
        result.compute_usage = safe_add_compute(result.compute_usage, compute).map_err(|_| {
            StorageError::StorageInconsistentState("compute_usage overflow".to_string())
        })?;
    }
    *actor_id = receipt.predecessor_id().clone();
    *account = None;
    Ok(())
}

fn check_transfer_to_nonexisting_account(
    config: &RuntimeConfig,
    account_id: &AccountId,
    implicit_account_creation_eligible: bool,
) -> Result<(), ActionError> {
    if implicit_account_creation_eligible
        && account_is_implicit(account_id, config.wasm_config.eth_implicit_accounts)
    {
        // OK. It's implicit account creation.
        // Notes:
        // - Transfer action has to be the only action in the transaction to avoid
        // abuse by hijacking this account with other public keys or contracts.
        // - Refunds don't automatically create accounts, because refunds are free and
        // we don't want some type of abuse.
        // - Account deletion with beneficiary creates a refund, so it'll not create a
        // new account.
        Ok(())
    } else {
        Err(ActionErrorKind::AccountDoesNotExist { account_id: account_id.clone() }.into())
    }
}
