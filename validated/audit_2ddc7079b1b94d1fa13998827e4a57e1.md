### Title
Gateway L2 Gas Price Admission Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful resource-bounds validation reads the **current block's** `gas_prices.strk_gas_prices.l2_gas_price` to check whether an incoming transaction's `max_price_per_unit` is acceptable. The correct value is `next_l2_gas_price` — the EIP-1559-derived price that will govern the **next** block the transaction will actually be included in. The codebase itself acknowledges this with an explicit TODO comment at the call site. Because the two values diverge whenever the network is not at the gas target, the gateway either admits transactions whose max price is too low for the real execution block (rising price) or rejects transactions whose max price is sufficient (falling price).

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds`, the gateway fetches the latest committed block's `BlockInfo` and extracts `gas_prices.strk_gas_prices.l2_gas_price`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The `BlockHeaderWithoutHash` struct carries two distinct L2 gas price fields:

- `gas_prices.strk_gas_prices.l2_gas_price` — the price that was **used** for the block that was just committed.
- `next_l2_gas_price` — the EIP-1559-adjusted price that **will be used** for the next block. [2](#0-1) 

`next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` using the actual gas consumed in the just-committed block and the EIP-1559 formula:

```rust
let next_l2_gas_price = calculate_next_l2_gas_price_for_fin(
    args.l2_gas_price,
    args.build_param.height,
    info.l2_gas_used,
    ...
);
``` [3](#0-2) 

The formula adjusts the price proportionally to the deviation of `gas_used` from `gas_target`:

```rust
let price_change = (price_u256 * gas_delta) / denominator;
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [4](#0-3) 

Snapshot tests confirm the two values diverge by ~1% per block under normal congestion conditions:

```
high_congestion: 30_312_500_000  (from base 30_000_000_000)
low_congestion:  29_687_500_000
``` [5](#0-4) 

The gateway's `run_validate_entry_point` also builds its `BlockContext` from the same stale `block_info`, so the blockifier's `check_fee_bounds` inside gateway validation is equally misaligned with the price the batcher will actually apply. [6](#0-5) 

### Impact Explanation

A transaction submitted to the gateway will be executed in the **next** block at `next_l2_gas_price`. The gateway validates the transaction's `max_price_per_unit` against the **previous** block's `l2_gas_price`. Two failure modes follow:

1. **Rising price (high congestion):** `next_l2_gas_price > l2_gas_price`. A transaction with `max_price_per_unit` between the two values passes gateway admission but will be rejected by the batcher's fee check at execution time, wasting mempool resources and causing user-visible failures for transactions that appeared valid at submission.

2. **Falling price (low congestion):** `next_l2_gas_price < l2_gas_price`. A transaction with `max_price_per_unit` above `next_l2_gas_price` but below the stale threshold is incorrectly rejected at the gateway, denying admission to a transaction that would have been perfectly valid in the next block.

Both modes constitute the gateway making an **authoritative-looking wrong admission decision** on every block where gas usage deviates from the target — which is the normal operating condition.

### Likelihood Explanation

The EIP-1559 price adjustment fires on every block where `gas_used ≠ gas_target`. In practice this is nearly every block. The divergence between `l2_gas_price` and `next_l2_gas_price` is bounded per block (~1% in the test snapshots) but is cumulative across sustained congestion or low-usage periods. The bug is triggered by any transaction whose `max_price_per_unit` falls in the gap between the two values, which is a predictable and exploitable range.

### Recommendation

Replace the stale `gas_prices.strk_gas_prices.l2_gas_price` read with `next_l2_gas_price` from the same block header, as the TODO comment already prescribes:

```rust
async fn validate_resource_bounds(
    &self,
    executable_tx: &ExecutableTransaction,
) -> StatefulTransactionValidatorResult<()> {
    if self.config.validate_resource_bounds {
        let block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        // Use next_l2_gas_price: the price that will govern the block this tx enters.
        let next_l2_gas_price = block_info.next_l2_gas_price;
        self.validate_tx_l2_gas_price_within_threshold(
            executable_tx.resource_bounds(),
            next_l2_gas_price,
        )?;
    }
    Ok(())
}
```

The `BlockInfo` / `BlockHeaderWithoutHash` already carries `next_l2_gas_price` as a first-class field; no additional RPC calls or state reads are required. [7](#0-6) 

### Proof of Concept

1. Observe the latest committed block has `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (high-congestion block).
2. Craft an `InvokeV3` transaction with `resource_bounds.l2_gas.max_price_per_unit = P + ε` (just above `P`, below `P'`).
3. Submit via the gateway. `validate_resource_bounds` compares against `P` → passes.
4. The batcher builds the next block at price `P'`. The blockifier's `check_fee_bounds` compares `max_price_per_unit = P + ε` against `P'` → fails with `MaxGasPriceTooLow`.
5. The transaction was admitted to the mempool and consumed sequencer resources despite being unexecutable at the actual block price.
