### Title
Unbounded L2→L1 Log Accumulation Across Multiblock Batch Causes Prover Panic - (`basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`, `zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

In the multiblock batch proving mode, L2→L1 log hashes from each block are accumulated into a fixed-capacity `ArrayVec<Bytes32, 16384>` inside `ZKBatchDataKeeper`. The per-block log limit is enforced at 16,384 entries, but no cross-block limit is enforced before appending into the batch-level accumulator. When the total log count across all blocks in a batch exceeds 16,384, `ArrayVec::push` panics unconditionally, aborting the prover.

---

### Finding Description

`ZKBatchDataKeeper` holds a `logs_storage: ArrayVec<Bytes32, 16384>` field that accumulates L2→L1 log hashes across every block in a multiblock batch. [1](#0-0) 

At the end of each block's post-tx operation, `apply_to_array_vec` is called unconditionally: [2](#0-1) 

`apply_to_array_vec` iterates over all logs in the block and calls `array_vec.push(log.hash())` for each one: [3](#0-2) 

`ArrayVec::push` in Rust **panics** when the array is at capacity. There is no bounds check before pushing.

The per-block log limit is enforced in `check_for_block_limits` using `MAX_NUMBER_OF_LOGS = 16_384`: [4](#0-3) 

This limit is per-block and resets between blocks. The `logs_storage` accumulator in `ZKBatchDataKeeper` is never reset between blocks and has the same capacity (16,384) as the per-block maximum. Therefore, a batch containing two blocks each with 8,193 logs (both individually valid) would attempt to push 16,386 entries into a 16,384-capacity `ArrayVec`, causing a panic. [5](#0-4) 

---

### Impact Explanation

The panic aborts the proving process for the entire multiblock batch. This is a **denial of service against the ZKsync OS prover** in multiblock batch mode. The prover cannot complete, the batch cannot be finalized, and the chain halts until the issue is resolved. This matches the Immunefi scope for ZKsync OS proving infrastructure.

---

### Likelihood Explanation

L2→L1 logs are emitted by any contract calling the L1 messenger system hook, which is accessible to any unprivileged transaction sender. An attacker can craft transactions that emit the maximum number of logs per block. If the sequencer groups two or more such blocks into a single multiblock batch, the accumulated log count exceeds the `ArrayVec` capacity. The attacker does not need any privileged access — only the ability to submit transactions that emit L2→L1 logs.

---

### Recommendation

Before calling `apply_to_array_vec`, check that the remaining capacity in `batch_data.logs_storage` is sufficient to hold the current block's logs. If not, either return an error or enforce a stricter cross-block log limit at the batch level. Alternatively, replace `ArrayVec<Bytes32, 16384>` with a dynamically-allocated `Vec` that can grow beyond 16,384 entries, and enforce the Merkle tree capacity separately at finalization time.

---

### Proof of Concept

1. Attacker deploys a contract that calls the L1 messenger system hook in a loop, emitting L2→L1 logs.
2. Attacker submits transactions filling Block 1 with 8,193 L2→L1 logs (within the per-block limit of 16,384).
3. Attacker submits transactions filling Block 2 with 8,193 L2→L1 logs (also within the per-block limit).
4. The sequencer groups Block 1 and Block 2 into a single multiblock batch.
5. During proving, `post_op` for Block 1 calls `io.logs_storage.apply_to_array_vec(&mut batch_data.logs_storage)`, pushing 8,193 hashes into the `ArrayVec` (capacity 16,384, now at 8,193).
6. `post_op` for Block 2 calls `apply_to_array_vec` again, attempting to push 8,193 more hashes. At entry 16,385, `ArrayVec::push` panics.
7. The prover aborts. The multiblock batch cannot be finalized. [6](#0-5) [3](#0-2) [2](#0-1)
