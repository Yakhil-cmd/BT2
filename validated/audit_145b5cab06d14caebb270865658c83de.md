### Title
Signature-Bypass via `skip_stateful_validations` Allows Unauthenticated Invoke Admission for Undeployed Accounts — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX shortcut for the `deploy_account + invoke` flow unconditionally skips the `__validate__` entry-point call for any Invoke V3 transaction with `nonce == 1` sent to an address that has *any* transaction in the mempool. An attacker who observes a victim's pending `deploy_account` can front-run the victim's first invoke by submitting a malicious invoke with an arbitrary signature. The gateway admits it without signature verification; the victim's legitimate invoke is then rejected by the mempool as a duplicate nonce.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip `__validate__`) when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the function returns immediately after `perform_pre_validation_stage`, never calling `validate_tx` (the `__validate__` entry point): [3](#0-2) 

`perform_pre_validation_stage` only checks nonce ordering, fee bounds, and proof facts — it does **not** verify the transaction signature: [4](#0-3) 

The mempool check `account_tx_in_pool_or_recent_block` returns `true` if the address has *any* transaction in the pool or a recently committed block — it does not distinguish a `deploy_account` from any other transaction type: [5](#0-4) 

**Attack path:**

1. Victim broadcasts `deploy_account` (nonce 0) → admitted to mempool; `account_tx_in_pool_or_recent_block(victim_addr)` now returns `true`.
2. Attacker pre-funds `victim_addr` with the minimum STRK required to pass `verify_can_pay_committed_bounds` (the only remaining gateway-side check).
3. Attacker submits `Invoke(sender=victim_addr, nonce=1, signature=<garbage>)`.
4. Gateway: `skip_stateful_validations` returns `true` → `__validate__` is never called → transaction admitted to mempool.
5. Victim submits their legitimate `Invoke(nonce=1)` → mempool rejects it with `DuplicateNonce` (fee escalation disabled by default).
6. Attacker's malicious invoke is eventually executed in a block: `__validate__` runs, signature fails, transaction reverts, fee is deducted from `victim_addr`.
7. Account nonce advances to 2; victim's original nonce-1 invoke is permanently invalidated.

The `max_nonce_for_validation_skip` config field exists in `StatefulTransactionValidatorConfig` but is **not used** by `skip_stateful_validations` (which hardcodes `nonce == 1`), so there is no operator-tunable guard: [6](#0-5) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions and rejects valid transactions before sequencing.**

- An invoke transaction carrying an arbitrary/invalid signature is admitted to the mempool, violating the invariant that every admitted transaction must be authorised by the account's `__validate__` logic.
- The victim's correctly-signed invoke is rejected as a duplicate nonce, permanently blocking their first post-deploy action at that nonce.
- The victim's pre-funded balance is drained by the fee charged on the reverted malicious invoke.

---

### Likelihood Explanation

The attack is fully unprivileged and requires only:
- Monitoring the public mempool for `deploy_account` transactions (trivial).
- Sending a small STRK transfer to the deterministic account address to satisfy the balance check (low cost).
- Submitting a single malicious invoke before the victim does (race condition, but the attacker controls timing after observing the deploy_account).

The default production config (`max_allowed_nonce_gap = 200`, fee escalation disabled) makes the duplicate-nonce rejection reliable. [7](#0-6) 

---

### Recommendation

1. **Restrict the skip to the exact sender**: before returning `true`, verify that the transaction in the pool for `sender_address` is specifically a `deploy_account` transaction, not just any transaction. The current check is too broad.
2. **Alternatively, use `max_nonce_for_validation_skip` from config** (the field already exists but is unused in `skip_stateful_validations`) and add a type check that the pooled transaction is a `deploy_account`.
3. Consider whether the UX shortcut is worth the admission risk; a stricter approach would be to require the `deploy_account` and the first invoke to arrive in the same RPC batch and be validated atomically.

---

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
