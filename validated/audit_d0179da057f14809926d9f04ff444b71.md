### Title
Missing Fee Estimation Function for `tick()` Cross-Chain Calls - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary
`RewardSupplier.sol`'s `tick()` function is `payable` and internally forwards ETH as fees to two separate L1→L2 cross-chain calls, but no helper function exists to let callers determine the correct `msg.value` to supply. This is the direct analog of the missing `quoteLayerZeroFee()` pattern: callers who send too little ETH cause `tick()` to revert, blocking the reward minting and bridging pipeline.

---

### Finding Description
`tick()` is declared `external payable` and makes two payable cross-chain calls:

```solidity
// RewardSupplier.sol line 125-141
uint256 msgFee = msg.value / 2;
bridge().depositWithMessage{value: msgFee}(
    token(), amountToMint, mintDestination(), new uint256[](0)
);

msgFee = msg.value - msgFee;
messagingContract().sendMessageToL2{value: msgFee}(
    mintingCurve(), UPDATE_TOTAL_SUPPLY_SELECTOR, messagePayload
);
``` [1](#0-0) 

The fee is split 50/50 (`msg.value / 2` each) between `IBridge.depositWithMessage` and `IStarknetMessaging.sendMessageToL2`. Neither the `RewardSupplier` contract nor any related contract exposes a fee-quoting helper. The `IStarknetMessaging` interface does define `getMaxL1MsgFee()` for this purpose: [2](#0-1) 

but it is never called or surfaced in `RewardSupplier.sol`. A grep across all L1 Solidity files for `quoteFee`, `quoteL1`, `estimateFee`, and `getMaxL1MsgFee` returns zero matches, confirming no fee estimation path exists. [3](#0-2) 

---

### Impact Explanation
`tick()` is the sole entry point for the entire reward minting and bridging pipeline (L2 mint-request consumption → `MintManager.mintRequest` → `depositWithMessage` → `sendMessageToL2`). [4](#0-3) 

If `msg.value` is insufficient, `sendMessageToL2` reverts (Starknet messaging enforces a minimum fee), rolling back the entire transaction including the already-consumed L2→L1 messages. Rewards are not minted or bridged to L2, causing **temporary freezing of unclaimed yield** until a caller supplies the correct fee. There is also no excess-ETH refund path in `tick()`, so over-paying permanently loses ETH.

**Impact: Medium** — Griefing with no profit motive but damage to protocol operations (reward distribution stalls).

---

### Likelihood Explanation
`tick()` is `external` with no access control, callable by any unprivileged address. [5](#0-4) 

Because no fee-quoting function exists, every caller must guess the correct `msg.value`. The 50/50 hardcoded split compounds the problem: even if the total ETH sent is sufficient in aggregate, the wrong split can cause one of the two calls to fail. Likelihood is **High**.

---

### Recommendation
Add a public `quoteTickFee()` view function that queries both the bridge and the messaging contract for their required fees and returns the total `msg.value` the caller must supply:

```solidity
function quoteTickFee() external view returns (uint256 bridgeFee, uint256 messagingFee) {
    bridgeFee  = messagingContract().getMaxL1MsgFee();
    messagingFee = messagingContract().getMaxL1MsgFee();
    // Or query the bridge's own fee estimation if available.
}
```

Replace the hardcoded `msg.value / 2` split with the actual quoted fees, and revert with a descriptive error if `msg.value` is insufficient.

---

### Proof of Concept
1. An L2 `RewardSupplier` sends a mint-request message to L1 (normal protocol operation).
2. A caller invokes `tick()` with `msg.value = 1 wei` (or any amount below the Starknet messaging minimum fee).
3. `consumeMessageFromL2` succeeds (no ETH required).
4. `bridge().depositWithMessage{value: 0}(...)` may succeed or revert depending on bridge implementation.
5. `messagingContract().sendMessageToL2{value: 0}(...)` reverts because Starknet messaging requires `msg.value >= 1` (enforced by the core contract).
6. The entire transaction reverts; no rewards are minted or bridged.
7. Because there is no `quoteTickFee()`, the caller has no on-chain way to determine the correct amount, and the failure repeats until someone empirically discovers the right value off-chain. [6](#0-5) [7](#0-6)
