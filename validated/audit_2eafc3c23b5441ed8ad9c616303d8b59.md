### Title
Sub-Threshold EVM Balances Permanently Locked with No Recovery Mechanism - (File: fvm/evm/stdlib/contract.cdc)

### Summary
The `CadenceOwnedAccount.withdraw()` function enforces a minimum withdrawal threshold of `1e10 attoflow` (0.00000001 FLOW) due to the precision gap between EVM's 18-decimal attoflow and Cadence's 8-decimal `UFix64`. Any FLOW balance below this threshold in an EVM account is permanently locked with no recovery path — no emergency sweep, no protocol-level reclaim, and no alternative withdrawal route exists.

### Finding Description
EVM on Flow stores balances in attoflow (1e-18 FLOW), while Cadence's `FlowToken.Vault` uses `UFix64` (1e-8 FLOW precision). The conversion multiplier is `1e10` attoflow per UFix64 unit.

The `withdraw` function in `CadenceOwnedAccount` enforces this constraint in two places:

1. **Direct rejection**: If the requested withdrawal amount is below `1e10 attoflow`, the call panics with `ErrWithdrawBalanceRounding`. [1](#0-0) 

2. **Silent truncation**: If the requested amount is above `1e10 attoflow` but not a multiple of it, the amount is truncated to the nearest multiple, and the remainder stays in the EVM account. [2](#0-1) 

The truncation remainder is always `< 1e10 attoflow` and therefore can never be withdrawn. The `AttoFlowBalanceIsValidForFlowVault` guard confirms this hard floor: [3](#0-2) 

The `withdraw` function in the Cadence contract itself also documents this limitation but provides no recovery path: [4](#0-3) 

Sub-threshold balances accumulate through:
- **Truncation remainders on every non-aligned withdrawal**: A user withdrawing `1_230_000_789_912_345_678 attoflow` receives `1.23000078 FLOW` back; the remaining `9_912_345_678 attoflow` stays locked in EVM forever.
- **EVM gas fee remainders**: After paying gas (denominated in attoflow), the leftover may fall below `1e10 attoflow`.
- **Dust deposits from EVM contracts**: Any EVM contract can send sub-threshold amounts to a COA address.

There is no protocol-level function, admin escape hatch, or owner-callable sweep to recover these balances. The only operations available on a COA are `withdraw` (blocked by the threshold), `call`/`deploy` (EVM-side only), and bridge functions (which also require minimum amounts). The `ErrWithdrawBalanceRounding` error is defined as a permanent hard stop: [5](#0-4) 

### Impact Explanation
FLOW tokens below `1e10 attoflow` in any EVM account are permanently irrecoverable to Cadence. Every withdrawal of a non-UFix64-aligned balance silently leaves a locked remainder. Over time, across all COA accounts on the network, this constitutes a growing pool of permanently locked FLOW with no protocol mechanism to reclaim it. This is a cross-VM asset loss: tokens exist in the EVM state but can never re-enter the Cadence token supply.

### Likelihood Explanation
This is triggered by any unprivileged user performing a normal COA withdrawal with a non-UFix64-aligned balance — a routine operation. The test suite explicitly demonstrates the truncation behavior: [6](#0-5) 

Any EVM transaction that leaves a sub-threshold remainder (gas fees, dust transfers, partial withdrawals) creates permanently locked funds. No special attacker capability is required.

### Recommendation
Add a protocol-level or owner-callable mechanism to recover sub-threshold EVM balances. Options include:
- A `sweepDust()` function on `CadenceOwnedAccount` that aggregates sub-threshold remainders across multiple accounts into a single withdrawable amount.
- A protocol-level burn/reclaim path that credits sub-threshold balances back to a designated recovery vault.
- Alternatively, modify the withdrawal logic to round up to the nearest `1e10 attoflow` boundary (minting the difference from the protocol reserve) so no remainder is ever stranded.

### Proof of Concept
1. User deposits `2.34 FLOW` into a COA.
2. User calls `cadenceOwnedAccount.withdraw(balance: EVM.Balance(attoflow: 1_230_000_789_912_345_678))`.
3. The `newInternalEVMTypeWithdrawFunction` truncates: `1_230_000_789_912_345_678 / 1e10 = 123000078` UFix64 units → `1_230_000_780_000_000_000 attoflow` is returned as `1.23000078 FLOW`.
4. Remainder in EVM: `1_230_000_789_912_345_678 - 1_230_000_780_000_000_000 = 9_912_345_678 attoflow`.
5. User attempts `cadenceOwnedAccount.withdraw(balance: EVM.Balance(attoflow: 9_912_345_678))` → panics with `ErrWithdrawBalanceRounding` (`9_912_345_678 < 1e10`).
6. The `9_912_345_678 attoflow` is permanently locked in the EVM account with no recovery path. [7](#0-6) [8](#0-7)

// AttoFlowBalanceIsValidForFlowVault returns true if the given balance,
// represented as atto-flow, can be stored in a Flow Vault, without loss
// in precision.
//
// Warning! The smallest unit of Flow token that a Flow Vault (Cadence)
// can store, is 1e-8 .
// This means the minimum balance, in atto-flow, that can be stored in a
// Flow Vault, is 1e10 .
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
	return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}

        /// Withdraws the balance from the cadence owned account's balance.
        /// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
        /// given that Flow Token Vaults use UFix64 to store balances.
        /// In other words, the smallest withdrawable amount is 1e10 attoFlow.
        /// Amounts smaller than 1e10 attoFlow, will cause the function to panic
        /// with: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".
        /// If the given balance conversion to UFix64 results in rounding loss,
        /// the withdrawal amount will be truncated to the maximum precision for UFix64.
        ///
        /// @param balance: The EVM balance to withdraw
        ///
        /// @return A FlowToken Vault with the requested balance
        access(Owner | Withdraw)
        fun withdraw(balance: Balance): @FlowToken.Vault {
            pre {
                !EVM.isPaused(): "EVM operations are temporarily paused"
            }

            if balance.isZero() {
                return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
            }
            let vault <- InternalEVM.withdraw(
                from: self.addressBytes,
                amount: balance.attoflow
            ) as! @FlowToken.Vault
            emit FLOWTokensWithdrawn(
                address: self.address().toString(),
                amount: balance.inFLOW(),
                withdrawnUUID: vault.uuid,
                balanceAfterInAttoFlow: self.balance().attoflow
            )
            return <-vault
        }
