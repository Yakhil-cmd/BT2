### Title
Global Contract Namespace Hijack via Cross-Contract `DeployGlobalContract(AccountId)` — (`runtime/runtime/src/global_contracts.rs`)

### Summary
Any contract can register attacker-supplied code under an arbitrary victim account's `TrieKey::GlobalContractCode { identifier: AccountId(victim) }` by creating a cross-contract promise targeting the victim with `DeployGlobalContract { deploy_mode: AccountId }`. No authorization check verifies that the receipt's `predecessor_id` equals its `receiver_id` before writing the global contract trie key.

### Finding Description

The host function `promise_batch_action_deploy_global_contract_by_account_id` in `runtime/near-vm-runner/src/logic/logic.rs` accepts any `promise_idx`, including promises targeting accounts other than the current contract. [1](#0-0) 

The action is appended to the receipt without any ownership check: [2](#0-1) 

At execution time, `action_deploy_global_contract` is called with `account_id` = the receipt's `receiver_id`: [3](#0-2) 

Inside `initiate_distribution`, when `deploy_mode == AccountId`, the global contract identifier is derived directly from `account_id` (the receiver), with no check that the predecessor owns that account: [4](#0-3) 

The action validator `validate_deploy_global_contract_action` only checks code size — no ownership or predecessor/receiver equality check: [5](#0-4) 

The resulting trie write is `TrieKey::GlobalContractCode { identifier: AccountId(victim) }` with attacker-supplied bytes, propagated to all shards via the distribution receipt mechanism. [6](#0-5) 

### Impact Explanation

The exact corrupted value is `TrieKey::GlobalContractCode { identifier: AccountId(victim_account_id) }` written with attacker-controlled `code: Vec<u8>`. Any account that subsequently calls `UseGlobalContract(AccountId(victim_account_id))` will have its `AccountContract` set to `GlobalByAccount(victim_account_id)` and will execute the attacker's code on every function call. The victim account's own ability to manage its global contract entry is also overwritten (or preempted), since the attacker can write to the key before the victim does, or overwrite it after.

Storage cost is deducted from the **victim's** balance (not the attacker's), so the attacker also causes an unauthorized balance reduction on the victim account.

### Likelihood Explanation

The attack requires only a deployed contract and a cross-contract call. The victim account must have sufficient balance to cover storage cost (`global_contract_storage_amount_per_byte * code.len()`), which is true for any normally-funded account. No validator or privileged access is needed.

### Recommendation

In `action_deploy_global_contract` (or in `validate_deploy_global_contract_action` for the `AccountId` mode), add a check that the receipt's `predecessor_id` equals `receiver_id` (i.e., the action is self-directed). Cross-contract `DeployGlobalContract(AccountId)` receipts should be rejected with an authorization error. Alternatively, enforce this at the host-function level by requiring that the promise target for `deploy_global_contract_by_account_id` must be `current_account_id`.

### Proof of Concept

**Test-loop test outline:**

1. Create accounts `attacker`, `victim`, `user`.
2. Deploy a contract to `attacker` that, when called, does:
   ```rust
   let p = env::promise_batch_create("victim");
   env::promise_batch_action_deploy_global_contract_by_account_id(p, ATTACKER_WASM);
   ```
3. Call the attacker contract from `attacker`.
4. Wait for the cross-contract receipt to execute at `victim`.
5. From `user`, submit `UseGlobalContract(AccountId("victim"))`.
6. Call a function on `user`'s account and assert it executes `ATTACKER_WASM` (e.g., logs a sentinel string only present in the attacker's code).
7. Assert `victim`'s balance was reduced by the storage cost without `victim` authorizing it.
