### Title
Malicious Participant Can Corrupt CKD Output Without Proof of Correct Computation — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator()` function in the Confidential Key Derivation (CKD) protocol aggregates `(norm_big_y, norm_big_c)` contributions from all participants without any cryptographic proof that each contribution was computed correctly from the participant's secret share. A single malicious participant can send arbitrary group elements in place of their honest contribution. The coordinator has no mechanism to detect this deviation, producing a corrupted `(Y, C)` output. The TEE app will then compute an invalid BLS signature and reject the result, permanently denying the CKD service for that request.

---

### Finding Description

In `do_ckd_coordinator()` (lines 35–57 of `src/confidential_key_derivation/protocol.rs`), the coordinator computes its own share and then collects contributions from every other participant via `recv_from_others`, adding each received `(big_y, big_c)` pair directly into the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The protocol specification requires each participant `i` to send:

```
norm_big_y = λ_i · (y_i · G₁)
norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · A)
```

where `x_i` is the participant's secret share and `y_i` is a fresh random scalar. There is **no zero-knowledge proof** or any other verification that the received `(norm_big_y, norm_big_c)` satisfies this relation. The coordinator blindly sums whatever values it receives.

**Attacker-controlled entry path:**

1. A malicious participant is a legitimate member of the `participants` list (no privilege required beyond being a protocol participant).
2. When `do_ckd_participant()` is called on the malicious node, instead of computing the correct share, it sends arbitrary group elements `(big_y', big_c')` to the coordinator via `chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))`.
3. `recv_from_others` accepts the message because the sender is a known participant — it enforces *who* can send but not *what* they send.
4. The coordinator adds the malicious values into the aggregate, producing a wrong `(Y, C)`.
5. The TEE app receives the corrupted output, computes `sig = C + (−a)·Y`, which is not a valid BLS signature over the expected public key `pk`, and rejects it.

The root cause is the absence of a proof-of-correct-computation check — the direct analog of the missing `onlyEntityAdmin` modifier in the reference report. Just as any child of an object ID could trigger an unauthorized token transfer, any participant in the CKD participant list can inject arbitrary values into the aggregation without authorization of correctness.

---

### Impact Explanation

**High — Corruption of CKD outputs / Permanent denial of CKD for honest parties.**

Because the protocol requires contributions from **all** `n` participants (no threshold subset is used; `recv_from_others` collects from every participant except `me`), a single malicious participant is sufficient to corrupt every CKD execution. The TEE app will always receive an invalid `(Y, C)` pair and will always reject it. There is no fallback or retry mechanism that can exclude the malicious participant without a full protocol reconfiguration. This constitutes a permanent, targeted denial of the CKD service under valid protocol inputs.

---

### Likelihood Explanation

**High.** Any participant in the authorized participant list — a role that requires no privileged access beyond being enrolled in the MPC network — can trigger this by deviating from the protocol in a single message. The deviation requires no cryptographic capability: sending random group elements is trivial. The malicious participant cannot be identified or excluded by the coordinator because no proof of correctness is required or checked.

---

### Recommendation

Add a zero-knowledge proof of correct computation to each participant's contribution. Specifically, each participant should attach a Chaum-Pedersen-style DLEQ proof demonstrating that `norm_big_c` and `norm_big_y` share the same discrete-log relationship with respect to the participant's committed secret share. The coordinator must verify all proofs before aggregating. Participants who fail to provide a valid proof should be identified as malicious and excluded, consistent with the approach already used in the DKG protocol (`verify_proof_of_knowledge` in `src/dkg.rs`).

---

### Proof of Concept

1. Participant `P_malicious` is enrolled in a CKD session with `n = 3` participants.
2. When `do_ckd_participant()` executes on `P_malicious`, it replaces the honest computation with:
   ```rust
   // Instead of compute_signature_share(...)
   let fake_big_y = ElementG1::generator(); // arbitrary non-zero point
   let fake_big_c = ElementG1::generator(); // arbitrary non-zero point
   chan.send_private(waitpoint, coordinator, &(fake_big_y, fake_big_c))?;
   ```
3. The coordinator in `do_ckd_coordinator()` receives this via `recv_from_others` (line 50–55) and adds it to the aggregate without any check.
4. The resulting `(Y, C)` is incorrect.
5. The TEE app computes `sig = C + (−a)·Y ≠ msk · H(pk, app_id)` and the BLS verification fails.
6. The CKD request is denied. Repeating this for every session permanently blocks the service. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
