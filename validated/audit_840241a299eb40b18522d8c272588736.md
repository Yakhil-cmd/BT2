### Title
Cross-Address COA Ownership Proof Signature Replay — (`fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of signatures over caller-supplied `signedData` proves a Cadence account controls a specific COA (Cadence-Owned Account). Because the function does not enforce that `signedData` encodes the target `evmAddress`, a signature produced to prove ownership of one COA can be replayed verbatim to prove ownership of any other COA held by the same Cadence account. The code itself acknowledges this gap in a comment but leaves it unmitigated at the protocol level.

### Finding Description

`validateCOAOwnershipProof` is an `access(all)` Cadence function in the EVM system contract. Its signature is:

```cadence
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
```

The function:
1. Verifies `signatures` over `signedData` using keys from the Cadence account at `address`.
2. Borrows the COA capability at `path` and checks its EVM address equals `evmAddress`.

The critical gap: `signedData` is **not required to contain `evmAddress`**. The function verifies the signature is valid for the account, then separately checks that the COA at `path` matches `evmAddress`. These two checks are independent — the signature does not commit to which COA is being proven.

The code explicitly acknowledges this in a comment:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The function is `access(all)` — any unprivileged Cadence script or transaction can call it with arbitrary `path` and `evmAddress` values. The EVM-side precompile (`verifyCOAOwnershipProof`) is protected because it always passes `address(this)` as `evmAddress`, but direct Cadence-level callers receive no such protection.

**Exploit path:**

1. Cadence account `A` holds two COAs: `COA_1` at `/public/coa1` (EVM address `0xAAAA…`) and `COA_2` at `/public/coa2` (EVM address `0xBBBB…`). Nothing in the protocol prevents multiple COAs per account.
2. Account `A` signs `data` to prove ownership of `COA_1` for some protocol. The proof `(signedData=data, keyIndices=[0], signatures=[sig])` is submitted on-chain and becomes observable.
3. An attacker calls `validateCOAOwnershipProof(address: A, path: /public/coa2, signedData: data, keyIndices: [0], signatures: [sig], evmAddress: 0xBBBB…)`.
4. Step 1 of the function: signatures verify correctly — `sig` is a valid signature from account `A`'s key over `data`.
5. Step 2 of the function: the COA at `/public/coa2` has address `0xBBBB…` — matches.
6. The function returns `isValid: true`, falsely certifying that account `A` proved ownership of `COA_2`, even though `A` only ever signed for `COA_1`.

### Impact Explanation

Any Cadence-level protocol or off-chain authentication flow that calls `validateCOAOwnershipProof` directly (not through the EVM precompile) and uses the result to authorize actions on a specific COA is vulnerable. An attacker who observes a valid proof for one COA can replay it to impersonate the account owner for any other COA they hold. Concrete consequences include:

- Unauthorized authorization of EVM asset transfers or contract calls gated by COA ownership proof.
- Bypass of access controls in Cadence contracts that use `validateCOAOwnershipProof` for authentication.
- Cross-COA identity confusion in bridge or DeFi protocols that rely on this proof.

The ERC-1271 `isValidSignature` path in `coa.sol` (which calls the precompile) is **not** affected because the precompile always binds `evmAddress` to `address(this)`. Only direct Cadence callers are exposed.

### Likelihood Explanation

- **Precondition**: A Cadence account must hold more than one COA. The protocol does not prevent this; `EVM.createCadenceOwnedAccount()` can be called multiple times and stored at different paths.
- **Attacker capability**: No privileged access required. Any unprivileged Cadence script can call `validateCOAOwnershipProof` with arbitrary arguments.
- **Proof availability**: A valid proof for COA-1 becomes observable the moment it is submitted on-chain (e.g., as a transaction argument or event). The attacker only needs to observe it once.
- **Mitigation gap**: The protocol provides no nonce, no binding of `signedData` to `evmAddress`, and no on-chain invalidation mechanism.

### Recommendation

Enforce at the protocol level that `signedData` encodes `evmAddress`. The simplest fix is to hash `evmAddress` into the data that is verified:

```cadence
// Inside validateCOAOwnershipProof, before calling keyList.verify:
let expectedSignedData = HashAlgorithm.SHA3_256.hash(
    signedData.concat(evmAddress.toConstantSized()!)
)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: expectedSignedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Alternatively, document a mandatory encoding convention and enforce it with a pre-condition check, or add a nonce to the signed payload to prevent temporal replay as well.

### Proof of Concept

```cadence
// Setup: account A creates two COAs
transaction {
    prepare(account: auth(Capabilities, SaveValue) &Account) {
        let coa1 <- EVM.createCadenceOwnedAccount()
        account.storage.save(<-coa1, to: /storage/coa1)
        let cap1 = account.capabilities.storage.issue<&EVM.CadenceOwnedAccount>(/storage/coa1)
        account.capabilities.publish(cap1, at: /public/coa1)

        let coa2 <- EVM.createCadenceOwnedAccount()
        account.storage.save(<-coa2, to: /storage/coa2)
        let cap2 = account.capabilities.storage.issue<&EVM.CadenceOwnedAccount>(/storage/coa2)
        account.capabilities.publish(cap2, at: /public/coa2)
    }
}

// Account A signs `data` to prove ownership of COA_1 (evmAddress1).
// Attacker observes (signedData, keyIndices, signatures) from on-chain.

// Attacker replays the proof for COA_2 (evmAddress2):
access(all) fun main(
    accountA: Address,
    signedData: [UInt8],       // same bytes A signed for COA_1
    keyIndices: [UInt64],      // same key indices
    signatures: [[UInt8]],     // same signatures
    evmAddress2: [UInt8; 20]   // COA_2's EVM address
): EVM.ValidationResult {
    // path changed to /public/coa2, evmAddress changed to COA_2's address
    return EVM.validateCOAOwnershipProof(
        address: accountA,
        path: /public/coa2,
        signedData: signedData,
        keyIndices: keyIndices,
        signatures: signatures,
        evmAddress: evmAddress2
    )
    // Returns isValid: true — replay succeeds
}
```
