### Title
WalletConnect Multi-NFT Offer Royalty Amount Understated Due to Incorrect Equal-Split Accounting - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible payment amount equally among all NFTs before applying each NFT's royalty percentage. When a WalletConnect-connected dapp sends a `chia_wallet.take_offer` containing multiple NFTs with royalties, the confirmation dialog displays a materially understated total cost, causing the user to approve a transaction believing they will spend less than the wallet will actually deduct.

### Finding Description

`formatAmountWithRoyalties` computes the royalty-inclusive total shown in the WalletConnect approval dialog:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← wrong
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

The `splitAmount` division distributes the fungible payment equally across all NFTs before multiplying by each royalty rate. The correct formula applies each NFT's royalty percentage to the **full** fungible amount:

```
correct royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
buggy  royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
```

These are equal only when N = 1. For N > 1 the buggy formula underestimates royalties by a factor of N.

**Concrete example** (confirmed by the existing test at line 251 of `parseCommandDisplay.test.ts`):

- Offer: pay 100,000,000 mojos (0.0001 XCH) for 2 NFTs with royalties 5 % and 0.1 %
- Displayed `amountWithRoyalties`: **0.00010255 XCH** (buggy)
- Correct total: 0.0001 XCH + (0.0001 × 5 %) + (0.0001 × 0.1 %) = **0.0001051 XCH**
- Understatement: ~2.4 % in this case; grows linearly with N

The test at line 320 asserts the buggy value `'0.00010255'` as the expected output, confirming the incorrect behavior is baked in. [1](#0-0) [2](#0-1) 

### Impact Explanation

`parseCommandDisplay` is invoked inside `processSessionRequest` in `WalletConnectProvider.tsx` to build the human-readable confirmation dialog shown to the user before any WalletConnect command is executed. [3](#0-2) 

When a dapp sends `chia_wallet.take_offer` with a multi-NFT offer, `walletDeltaToDisplay` calls `withRoyaltyTotals`, which calls `formatAmountWithRoyalties` to populate `amountWithRoyalties` on each fungible spending line. [4](#0-3) 

The user reads the understated figure and approves. The Chia wallet backend then executes the offer and deducts the correct (higher) royalty-inclusive amount from the user's spendable balance. The user has approved a transaction displaying the wrong spending amount — a direct fit for the **High** impact category: *"WalletConnect state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation

Any dapp that has an active WalletConnect session can trigger this path by sending a `chia_wallet.take_offer` request containing two or more NFTs with non-zero royalties. No special privilege, leaked key, or social engineering beyond the already-established WalletConnect pairing is required. The understatement scales with the number of NFTs and their royalty rates, making it straightforward to craft an offer where the displayed cost is meaningfully lower than the actual cost.

### Recommendation

Replace the equal-split logic with a sum of per-NFT royalties applied to the full amount:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Apply each NFT's royalty to the full amount, not a split portion
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
```

Update the corresponding test assertions in `parseCommandDisplay.test.ts` to reflect the corrected values. [5](#0-4) 

### Proof of Concept

1. Establish a WalletConnect session with the Chia GUI wallet.
2. Send a `chia_wallet.take_offer` session request whose offer blob encodes:
   - Two NFTs offered by the maker: NFT-A (5 % royalty) and NFT-B (1 % royalty)
   - 1 XCH requested from the taker
3. Observe the WalletConnect confirmation dialog: `amountWithRoyalties` displays **1.03 XCH** (buggy: `(0.5 × 5 %) + (0.5 × 1 %) = 0.03`).
4. Approve the offer.
5. The wallet backend deducts **1.06 XCH** (correct: `(1 × 5 %) + (1 × 1 %) = 0.06`).
6. The user has approved a transaction that costs 2× the displayed royalty overhead.

The gap grows proportionally with the number of NFTs and their royalty rates, allowing a dapp to craft offers where the displayed cost is arbitrarily lower than the actual cost within the constraints of valid royalty percentages. [6](#0-5) [4](#0-3)

function royaltyPercentagesForSide(lines: DisplayWalletDeltaItem[]): number[] {
  return lines
    .filter((line): line is Extract<DisplayWalletDeltaItem, { kind: 'nft' }> => line.kind === 'nft')
    .map((line) => line.royaltyPercentage)
    .filter(
      (royaltyPercentage): royaltyPercentage is number => royaltyPercentage !== undefined && royaltyPercentage > 0,
    );
}

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

async function walletDeltaToDisplay(
  walletDelta: WalletDelta,
  walletInfos: Record<string, WalletInfo>,
  assetKinds: AssetDisplayKinds,
  royaltyPercentages: AssetRoyaltyPercentages,
  fee?: bigint,
): Promise<DisplayWalletDelta> {
  const { spending, receiving } = walletDelta;
  const spendingItems = await Promise.all(
    Object.entries(spending).map(async ([key, value]) => ({
      key,
      line: await parseWalletDeltaItem(
        key,
        value,
        walletInfos,
        assetKinds.spending[key],
        royaltyPercentages.spending[key],
      ),
    })),
  );
  const receivingItems = await Promise.all(
    Object.entries(receiving).map(async ([key, value]) => ({
      key,
      line: await parseWalletDeltaItem(
        key,
        value,
        walletInfos,
        assetKinds.receiving[key],
        royaltyPercentages.receiving[key],
      ),
    })),
  );
  const spendingLines = spendingItems.map(({ line }) => line);
  const receivingLines = receivingItems.map(({ line }) => line);

  return {
    spending: withRoyaltyTotals(spendingItems, spending, receivingLines),
    receiving: withRoyaltyTotals(receivingItems, receiving, spendingLines),
    fee: fee !== undefined ? mojoToChiaLocaleString(fee) : undefined,
  };
}
