### Title
Missing Zero-Address Check in `change_reward_address` Permanently Freezes Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Both `Staking.change_reward_address` and `Pool.change_reward_address` accept a `reward_address` argument without validating it is non-zero. Setting the reward address to the zero address causes all subsequent `claim_rewards` calls to permanently transfer accrued yield to the zero address, freezing it irreversibly.

### Finding Description
`Staking.change_reward_address` (staking.cairo:517–540) and `Pool.change_reward_address` (pool.cairo:505–526) each accept a `reward_address: ContractAddress` parameter and update the stored reward destination for the caller. The only validation performed is that the address is not a registered token address (`REWARD_ADDRESS_IS_TOKEN`). There is no check that `reward_address.is_non_zero()`. [1](#0-0) [2](#0-1) 

Once the reward address is set to zero, every subsequent call to `claim_rewards` in the staking contract invokes `send_rewards_to_staker`, which reads `staker_info.reward_address` and calls `checked_transfer(recipient: reward_address, ...)` — transferring tokens to address zero. [3](#0-2) 

The same applies in the pool contract, where `claim_rewards` reads `pool_member_info.reward_address` and calls `checked_transfer(recipient: reward_address, ...)`. [4](#0-3) 

The same zero-address gap exists at initial registration: `Staking.stake` accepts `reward_address` and only checks it is not a token address, with no zero check. [5](#0-4) 

### Impact Explanation
Any staker or pool member who sets their reward address to zero (accidentally, via a buggy script, or due to a UI/SDK default) will have all future reward claims permanently transferred to the zero address. The tokens are unrecoverable. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

### Likelihood Explanation
The call path is fully reachable by any unprivileged staker or pool member. The scenario mirrors the external report exactly: an automation script or integration that leaves `reward_address` uninitialized (defaulting to `0`) and calls `change_reward_address` would silently redirect all future yield to the zero address. No privileged access is required.

### Recommendation
Add a non-zero assertion at the top of both `change_reward_address` implementations, and also in `Staking.stake` and `Pool.enter_delegation_pool`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

The error variant `ZERO_ADDRESS` ("Zero address") already exists in the error catalogue. [6](#0-5) 

### Proof of Concept

1. A staker is registered and has accrued rewards.
2. The staker (or an automation script with an uninitialized variable) calls:
   ```cairo
   staking.change_reward_address(reward_address: 0.try_into().unwrap());
   ```
   This succeeds — the only check (`REWARD_ADDRESS_IS_TOKEN`) passes because zero is not a registered token.
3. Any subsequent call to `staking.claim_rewards(staker_address)` executes `send_rewards_to_staker`, which reads `reward_address = 0` from storage and calls `checked_transfer(recipient: 0, amount: rewards)`.
4. The STRK tokens are transferred to address zero and are permanently lost.
5. The identical path exists for pool members via `Pool.change_reward_address` followed by `Pool.claim_rewards`.
