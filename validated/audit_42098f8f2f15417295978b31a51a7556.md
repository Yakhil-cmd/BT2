### Title
Bridge Circuit Breaker Permanently Inoperative: `TurnBridge` Is a No-Op — (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is implemented as a complete no-op, returning `nil, nil` without performing any state mutation, permission check, or event emission. The bridge can never be disabled by any caller — including the admin or a permissioned address — permanently breaking the emergency circuit breaker that is the only mechanism to halt bridge/conversion flows.

### Finding Description
`TurnBridge` at lines 84–87 of `x/cronos/keeper/msg_server.go`:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

The function:
1. Performs **zero permission checks** — unlike `UpdateTokenMapping`, which calls `k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` before acting. [2](#0-1) 
2. Writes **no state** — no `BridgeEnabled` parameter is toggled, no KV store entry is set.
3. Returns a **nil error** — the caller receives a success response and has no way to detect that nothing happened.

Any address (privileged or not) can submit `MsgTurnBridge` and receive a success response. The bridge state is never altered. The `CanTurnBridge` permission bit defined in the permissions system exists solely to gate this message, but the gate leads nowhere. [3](#0-2) 

The integration test in `test_gravity.py` confirms the intended contract: after `turn_bridge("false", from_="validator")` returns code 0, subsequent bridge operations are expected to fail. Because `TurnBridge` is a no-op, those operations continue to succeed, silently violating the invariant. [4](#0-3) 

ADR-009 explicitly designates `MsgTurnBridge` as the emergency circuit breaker for the bridge module, intended to halt bridge activity in security incidents. [5](#0-4) 

### Impact Explanation
**High — Bypass of Cronos admin bridge authorization control.**

The admin's sole on-chain mechanism to halt bridge/conversion flows is permanently broken. In the event of a bridge exploit, a compromised CRC20/CRC21 contract, or any other emergency requiring the bridge to be stopped, the admin calls `TurnBridge(false)`, receives a success response, and believes the bridge is disabled — while all bridge and conversion operations continue unimpeded. This satisfies: *"High: Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation
Certainty, not likelihood: the code path is statically verifiable. Every invocation of `MsgTurnBridge` returns success and changes nothing. The only question is whether an emergency arises that requires the circuit breaker — at which point the failure is guaranteed.

### Recommendation
Implement `TurnBridge` to:
1. Verify the caller holds `CanTurnBridge` permission via `k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`.
2. Persist the enabled/disabled state (e.g., update a `BridgeEnabled` field in `Params` via `k.SetParams`).
3. Ensure all bridge entry points (`SendToIbc`, `SendToEVMChain`, gravity hooks) gate on the stored `BridgeEnabled` value.
4. Emit an event recording the state change and the caller.

### Proof of Concept
1. Attacker begins exploiting a bridge vulnerability (e.g., draining CRC20 tokens via a malicious `SendToIbc` log).
2. Admin submits `MsgTurnBridge{Enable: false}` from the permissioned validator address.
3. Chain returns `code: 0` — apparent success.
4. No state is written; `BridgeEnabled` is never set to `false`.
5. Attacker's bridge transactions continue to be processed by EVM hooks and IBC handlers.
6. Admin has no remaining on-chain mechanism to halt the bridge without a governance upgrade. [1](#0-0)
