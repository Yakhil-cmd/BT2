### Title
`max_l2_gas_amount` Upper-Bound Invariant Not Enforced for Declare Transactions in Gateway Stateless Validation — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The `StatelessTransactionValidator` enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. Any user can submit a Declare transaction with `l2_gas.max_amount` set to an arbitrarily large value (up to `u64::MAX`), bypassing the admission invariant the sequencer is designed to enforce. No downstream guard restores the invariant before the transaction enters the mempool.

---

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds`, the check for `max_l2_gas_amount` is gated behind an explicit type-dispatch that silently skips Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1,210,000,000`. [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts this gap: a Declare with `l2_gas.max_amount = 200` passes validation even when `max_l2_gas_amount = 100`. [3](#0-2) 

The full `validate` call-chain shows that `validate_resource_bounds` is the only place this ceiling is checked; `validate_tx_size` (always called) does not check gas amounts, and the stateful validator only checks the L2 gas *price* against the previous block, not the gas *amount*. [4](#0-3) [5](#0-4) 

After stateless validation, the gateway converts the transaction and calls `extract_state_nonce_and_run_validations`, which runs nonce/fee-price checks but has no `max_l2_gas_amount` guard. [6](#0-5) 

The mempool's `validate_tx` checks nonce validity and fee-escalation rules only; it does not inspect `l2_gas.max_amount`. [7](#0-6) 

The blockifier's `perform_pre_validation_stage` checks that resource bounds are *sufficient* (not too low) and that the account can pay the maximum possible fee, but it does not enforce an upper ceiling on `l2_gas.max_amount`. [8](#0-7) 

**Attack path:**

1. Craft a Declare V3 transaction with `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit = 0` (L1 gas bounds set non-zero to satisfy the zero-bounds check).
2. Submit via `POST /add_transaction`.
3. Stateless validation passes — the Declare branch is empty.
4. Stateful validation passes — the fee-price check only looks at `max_price_per_unit`, not `max_amount`; `verify_can_pay_committed_bounds` computes `max_possible_fee = actual_l1_gas_cost + 0` (L2 price is zero), which the account can cover.
5. The transaction is admitted to the mempool and sequenced with `l2_gas.max_amount = u64::MAX`, violating the invariant the system is designed to enforce.

---

### Impact Explanation

The gateway's explicit admission invariant — "no transaction may have `l2_gas.max_amount > max_l2_gas_amount`" — is broken for every Declare transaction. This matches the **High** impact tier: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

Concretely:
- A Declare with `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit = 0` is admitted and sequenced with zero L2 gas fee, bypassing the economic bound the operator configured.
- The oversized gas budget is passed to the OS as the initial gas limit for `__validate_declare__`, potentially allowing the validation entry point to consume more gas than the operator intended to permit per transaction.
- The invariant violation is permanent once the transaction is in the mempool; no later stage rejects it.

---

### Likelihood Explanation

The trigger requires only a standard Declare V3 transaction with a crafted `l2_gas` field. No privileged access, special account state, or race condition is needed. Any user with a funded account can submit such a transaction through the public HTTP endpoint or via P2P propagation (which also routes through the gateway). The TODO comment in the source confirms the developers are aware the check is absent.

---

### Recommendation

Move the `max_l2_gas_amount` check outside the `Declare`-skipping branch so it applies to all transaction types:

```rust
// Apply to all transaction types, including Declare.
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Remove the `// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.` comment and the empty `if let RpcTransaction::Declare(_) = tx {}` branch. [1](#0-0)

fn declare_bypasses_max_l2_gas_amount() {

                l1_gas: NON_EMPTY_RESOURCE_BOUNDS,   // satisfies zero-bounds check

                    max_amount: GasAmount(u64::MAX),  // violates max_l2_gas_amount

                },
                ..Default::default()
            },
            ..Default::default()
        },
    );

    // Passes — the Declare branch skips the max_l2_gas_amount check entirely.
