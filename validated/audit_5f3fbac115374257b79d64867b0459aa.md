### Title
ERC-20 Burn Without Guaranteed NEP-141 Transfer in `ExitToNear` Precompile Creates Permanent Fund Loss — (File: `engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is absent, the `ExitToNear` precompile burns a user's ERC-20 tokens atomically inside the EVM but schedules the corresponding NEP-141 `ft_transfer` as an asynchronous NEAR promise with **no failure callback**. If that promise fails, the ERC-20 tokens are permanently destroyed while the NEP-141 tokens remain locked in Aurora's account — an exact accounting split analogous to the reported `disburseJob` inconsistency.

### Finding Description

`ExitToNear::run()` constructs a `transfer_promise` targeting the NEP-141 contract's `ft_transfer` (or `ft_transfer_call`) method. Whether a callback is attached depends on a compile-time branch: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← no refund info compiled in
    transfer_near: transfer_near_args,
};
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← bare promise, no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { base: transfer_promise, callback: … })
};
```

For a standard ERC-20 exit (not wNEAR unwrap), `transfer_near_args` is `None`. Without `error_refund`, `refund` is also `None`, so `callback_args` equals the default, and the promise is emitted **without any callback**. The ERC-20 burn that preceded this call is already committed to EVM state; there is no mechanism to reverse it if `ft_transfer` fails.

The `exit_erc20_token_to_near` helper confirms the ERC-20 address and amount are taken directly from the EVM execution context (the caller is the ERC-20 contract that already burned the tokens): [2](#0-1) 

The same pattern applies to the ETH base-token path in `exit_base_token_to_near`, where the ETH value is already deducted from the sender's EVM balance before the promise is scheduled. [3](#0-2) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails (e.g., recipient account not registered with the NEP-141 contract, connector paused, insufficient gas in the promise), the outcome is:

- ERC-20 tokens: **permanently burned** (EVM state already committed).
- NEP-141 tokens: **remain in Aurora's account** (transfer never executed).

The user's funds are irrecoverably lost. The NEP-141 balance held by Aurora exceeds what the remaining ERC-20 supply can redeem, creating a persistent accounting split between the EVM mirror and the underlying NEP-141 asset — the direct analog of the `jobBalances`/token-balance inconsistency in the reference report.

The integration test explicitly documents this outcome: [4](#0-3) 

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

### Likelihood Explanation

**Medium.** Any unprivileged EVM user who calls `ExitToNear` with a NEAR recipient that is not storage-registered with the target NEP-141 contract will trigger this path. NEP-141 storage registration is a prerequisite that many users are unaware of. The failure is silent from the EVM perspective (the EVM transaction succeeds), so users have no on-chain indication that their tokens were lost until they check balances manually.

### Recommendation

1. **Enable `error_refund` unconditionally** in all production builds, or promote it from a feature flag to always-on behavior. The callback in `exit_to_near_precompile_callback` already handles the refund correctly when the feature is present: [5](#0-4) 

2. For `ExitToEthereum`, which has **no** refund callback at all regardless of feature flags, apply the same callback pattern: [6](#0-5) 

3. Consider adding an invariant check (e.g., in a periodic reconciliation or via the `Accounting` struct) that asserts Aurora's NEP-141 balance equals the total ERC-20 supply for each bridged token: [7](#0-6) 

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to an ERC-20 on Aurora.
2. As `ft_owner`, call the ERC-20's `withdrawToNear` targeting `"unregistered.near"` (an account with no NEP-141 storage deposit).
3. The EVM transaction succeeds; ERC-20 tokens are burned.
4. The `ft_transfer` NEAR promise fails (unregistered recipient).
5. Without `error_refund`, no callback fires; ERC-20 balance is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` while Aurora's NEP-141 balance remains `FT_TRANSFER_AMOUNT`.
6. The user has permanently lost `FT_EXIT_AMOUNT` tokens with no recourse.

This is confirmed by the existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` lines 623–665, which explicitly asserts the token loss when `error_refund` is absent. [8](#0-7)

use aurora_engine_types::U256;
use core::cmp::Ordering;

/// This struct tracks changes to the supply of a U256 quantity.
/// It is used in our code to keep track of the total supply of ETH on Aurora.
/// This struct is intentionally designed to avoid doing subtraction as much as possible
/// to avoid complexities of signed values and over/underflow.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Default)]
pub struct Accounting {
    gained: U256,
    lost: U256,
}

impl Accounting {
    pub fn change(&mut self, amount: Change) {
        match amount.new_value.cmp(&amount.old_value) {
            Ordering::Greater => {
                let net_gained = amount.new_value - amount.old_value;
                self.gained = self.gained.saturating_add(net_gained);
            }
            Ordering::Less => {
                let net_lost = amount.old_value - amount.new_value;
                self.lost = self.lost.saturating_add(net_lost);
            }
            Ordering::Equal => (),
        }
    }

    pub fn remove(&mut self, amount: U256) {
        self.lost = self.lost.saturating_add(amount);
    }

    #[must_use]
    pub fn net(&self) -> Net {
        match self.gained.cmp(&self.lost) {
            Ordering::Equal => Net::Zero,
            Ordering::Greater => Net::Gained(self.gained - self.lost),
            Ordering::Less => Net::Lost(self.lost - self.gained),
        }
    }
}
