### Title
EVM Storage Refund Cancelled by Native-Resource Delta Adjustment — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

`compute_gas_refund` applies an EVM SSTORE refund to `gas_used` (reducing it), but then computes `delta_gas` against `native_used`, which was never reduced by any corresponding amount. Because the two resource metrics diverge at exactly the refund amount, `delta_gas` is positive and adds the refund back, fully cancelling it. Users who clear storage slots are overcharged by up to 20 % of their gas fees.

### Finding Description

In `compute_gas_refund` (`refund_calculation.rs`):

```
// Step 1 – gas used from ergs
gas_used = gas_limit - remaining_ergs / ERGS_PER_GAS;   // e.g. 1000

// Step 2 – EVM SSTORE refund (EIP-3529, capped at 1/5)
evm_refund = min(refund_counter / ERGS_PER_GAS, gas_used / 5);  // e.g. 200
gas_used -= evm_refund;                                          // → 800

// Step 3 – native used (NOT reduced by evm_refund)
native_used = full_native_limit - remaining_native;  // e.g. 1000 * native_per_gas

// Step 4 – delta_gas bridges the two metrics
delta_gas = (native_used / native_per_gas) - gas_used;  // = 1000 - 800 = 200
if delta_gas > 0 { gas_used += delta_gas; }             // → 1000
```

`gas_used` is reduced by `evm_refund` (an EVM-layer accounting adjustment), but `native_used` reflects the actual native cycles consumed for those same SSTORE operations — it is never reduced. When `delta_gas` is computed, the two metrics diverge by exactly `evm_refund`, so `delta_gas ≈ evm_refund` and the refund is silently added back. The final `gas_used` equals the pre-refund value, and the user pays `gas_used * gas_price` as if no refund existed.

The EVM refund counter is populated by `refund_for_storage_write` in `basic_system/src/system_implementation/system/mod.rs` (lines 132–148) whenever a non-zero storage slot is cleared. The `compute_gas_refund` function is called for every L2 transaction (via `before_refund` in `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`, line 427) and every L1 transaction (via `process_l1_transaction`, line 266).

### Impact Explanation

Any user who clears storage slots (SSTORE from non-zero to zero) receives zero net benefit from the EVM refund. The operator is overpaid by up to `gas_used / 5 * gas_price`. For a 1 M gas transaction at 1 gwei, the maximum overcharge is 200 000 gas × 1 gwei = 0.0002 ETH per transaction. This is a direct, repeatable financial loss for users and a corresponding gain for the operator, breaking EVM-equivalent fee semantics.

### Likelihood Explanation

Clearing storage is a routine operation in DeFi (resetting allowances, deleting mappings, zeroing balances). Any unprivileged user submitting a standard EVM transaction that clears at least one storage slot will trigger the bug. No special privileges, oracle manipulation, or governance access are required.

### Recommendation

Compute `delta_gas` against `gas_used` **before** the EVM refund is subtracted, so the two metrics remain on the same basis:

```rust
// Compute delta_gas against pre-refund gas_used
let gas_used_before_refund = gas_limit - resources.ergs().0.div_floor(ERGS_PER_GAS);
resources.exhaust_ergs();

let evm_refund = { /* ... as today ... */ };

let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());

let delta_gas = (native_used / native_per_gas) as i64 - (gas_used_before_refund as i64);
let gas_used = gas_used_before_refund + delta_gas.max(0) as u64;
let gas_used = gas_used.saturating_sub(evm_refund);  // apply refund after delta
let gas_used = gas_used.max(minimal_gas_used);
```

This ensures the native-resource adjustment and the EVM refund are applied independently rather than one cancelling the other.

### Proof of Concept

**Setup:** `gas_limit = 1 000 000`, `native_per_gas = 2`, transaction clears 5 storage slots.

- EVM gas consumed: 900 000 (900 000 ergs / ERGS_PER_GAS remaining = 100 000 left)
- `evm_refund = min(5 × 4800, 900 000 / 5) = 24 000`
- `gas_used` after refund: `900 000 − 24 000 = 876 000`
- `native_used = 900 000 × 2 = 1 800 000`
- `delta_gas = (1 800 000 / 2) − 876 000 = 900 000 − 876 000 = 24 000`
- `gas_used += 24 000` → **900 000** (refund fully cancelled)

**Expected (EVM-compatible):** user pays for 876 000 gas.
**Actual:** user pays for 900 000 gas — overcharged by 24 000 gas × `gas_price`. [1](#0-0) [2](#0-1) [3](#0-2)
