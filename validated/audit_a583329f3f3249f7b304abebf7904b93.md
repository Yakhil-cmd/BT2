Audit Report

## Title
Unbounded `ingress_end` in `construction_payloads` Enables OOM DoS via Unbounded Envelope-Generation Loop — (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC1 Rosetta API accepts an attacker-controlled `ingress_end` value with no upper-bound validation. The two existing guards only enforce ordering and a minimum future distance, leaving the while-loop at lines 163–167 free to execute ~139 million iterations when `ingress_end = u64::MAX`, allocating gigabytes of heap memory and crashing the Rosetta node. The endpoint requires no authentication.

## Finding Description
`construction_payloads` in [1](#0-0)  computes `ingress_interval = (MAX_INGRESS_TTL − PERMITTED_DRIFT).as_nanos() = 240 × 10⁹ ns`, and `INGRESS_INTERVAL_OVERLAP` is 120 s [2](#0-1) , giving a loop step of `240e9 − 120e9 = 120 × 10⁹ ns` per iteration.

The two guards before the loop are:

1. `ingress_start >= ingress_end` → error [3](#0-2) 
2. `ingress_end < now + ingress_interval` → error [4](#0-3) 

Guard 2 only enforces a **lower** bound (ingress_end must be at least one interval in the future). It imposes no **upper** bound. With `ingress_end = u64::MAX ≈ 1.844 × 10¹⁹ ns` and `ingress_start = now ≈ 1.75 × 10¹⁸ ns`, both guards pass, and the loop [5](#0-4)  executes approximately `(u64::MAX − now) / 120e9 ≈ 139,000,000` iterations, pushing a `u64` into `ingress_expiries` each time (~1.1 GB for the vec alone).

The resulting `ingress_expiries` vec is then passed to `handle_construction_payloads`, which allocates a full `EnvelopeContent::Call` struct (containing `Vec<u8>` fields for `arg`, `nonce`, method name, etc.) for every entry [6](#0-5) , multiplying the memory pressure by several times.

The endpoint handler applies no authentication — it accepts any `Json<ConstructionPayloadsRequest>` directly [7](#0-6) .

The identical unbounded loop pattern is also present in the ICP Rosetta implementation [8](#0-7) .

## Impact Explanation
A single unauthenticated HTTP POST causes the Rosetta process to exhaust available memory (OOM kill) or spin for an extended period, making the node unavailable. This matches the allowed High impact: **"Application/platform-level DoS, crash … or subnet availability impact not based on raw volumetric DDoS"** and **"Significant … Rosetta … security impact with concrete user or protocol harm"** — all legitimate ledger transfers routed through the affected Rosetta node are blocked for the duration of the crash/restart cycle.

## Likelihood Explanation
The endpoint requires no credentials. The malicious payload is a standard JSON body with one field (`metadata.ingress_end`) set to `u64::MAX`. Any attacker who can reach the Rosetta HTTP port can trigger this with a single `curl` command. The attack is repeatable: the node can be crashed again immediately after restart.

## Recommendation
Add an explicit upper-bound check immediately after the existing guards, before the loop:

```rust
const MAX_INGRESS_WINDOW: u64 =
    (24 * 60 * 60) * 1_000_000_000u64; // 24 h in nanoseconds
if ingress_end > now + MAX_INGRESS_WINDOW {
    return Err(Error::processing_construction_failed(
        "ingress_end exceeds maximum allowed window",
    ));
}
```

This caps the loop to at most `24h / 120s = 720` iterations. Apply the same fix to the ICP Rosetta implementation at [8](#0-7) .

## Proof of Concept

```rust
#[test]
fn test_construction_payloads_oom_dos() {
    use std::time::{Duration, SystemTime};
    let now_sys = SystemTime::now();
    let now = now_sys
        .duration_since(SystemTime::UNIX_EPOCH)
        .unwrap()
        .as_nanos() as u64;

    // Guard 1 passes: ingress_start (now) < ingress_end (u64::MAX)
    // Guard 2 passes: u64::MAX >= now + ingress_interval
    // Loop runs ~139,000,000 iterations → OOM
    let result = construction_payloads(
        vec![/* valid transfer operation */],
        Some(ConstructionPayloadsRequestMetadata {
            ingress_start: Some(now),
            ingress_end: Some(u64::MAX),
            created_at_time: None,
            memo: None,
        }),
        &Principal::anonymous(),
        vec![/* valid public key */],
        now_sys,
    );
    // Without the fix this OOMs; with the fix it must return Err.
    assert!(result.is_err(), "must reject astronomically large ingress_end");
}
```
