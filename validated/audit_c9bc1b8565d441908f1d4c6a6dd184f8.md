### Title
Unchecked Exit Precompile Call After `_burn` in `withdrawToNear`/`withdrawToEthereum` Leads to Permanent Token Loss — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` irreversibly burn a user's ERC-20 tokens **before** calling the Aurora exit precompile via inline assembly. The assembly `call` return value is never checked. If the precompile call fails for any reason — including the NEP-141 connector contract having transfer restrictions analogous to rETH's deposit delay — the tokens are permanently destroyed with no corresponding NEAR-side transfer and no refund path.

---

### Finding Description

In both bridge token contracts, `withdrawToNear` and `withdrawToEthereum` follow the same unsafe pattern:

**Step 1 — Irreversible burn:** [1](#0-0) [2](#0-1) 

**Step 2 — Unchecked precompile call:** [3](#0-2) [4](#0-3) 

The `res` variable from the assembly `call` is assigned but **never inspected**. In EVM semantics, a low-level `call()` that fails returns `0` without reverting the caller. Since `_burn` has already executed, the user's ERC-20 balance is gone. If the precompile returns an `ExitError`, no promise log is emitted, no NEAR-side `ft_transfer` or `withdraw` is ever scheduled, and the tokens are unrecoverable.

The same pattern is identically reproduced in `withdrawToEthereum` in both contracts: [5](#0-4) [6](#0-5) 

**Conditions under which the precompile fails (ExitError):**

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `ExitError` in several reachable cases:
- `ERR_TARGET_TOKEN_NOT_FOUND` — the ERC-20→NEP-141 mapping is absent for the calling contract address. [7](#0-6) 
- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — the recipient bytes are not a valid NEAR account ID. [8](#0-7) 
- `ERR_INVALID_INPUT` — input size outside `[MIN_INPUT_SIZE, MAX_INPUT_SIZE]`. [9](#0-8) 

The `ExitToEthereum` precompile has the same failure modes: [10](#0-9) 

**Compounding issue — no refund callback for `ExitToEthereum`:**

Even when the precompile succeeds at the EVM level and a NEAR promise is scheduled, `ExitToEthereum` creates only a bare `PromiseArgs::Create` with **no callback**: [11](#0-10) 

If the NEAR-side `withdraw` call on the connector contract subsequently fails (e.g., due to the connector being paused — the direct analog of rETH's deposit delay), there is no refund path. By contrast, `ExitToNear` conditionally attaches a `exit_to_near_precompile_callback` when the `error_refund` feature is compiled in: [12](#0-11) 

`ExitToEthereum` has no equivalent protection at all.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` on any `EvmErc20`/`EvmErc20V2` mirror token and whose precompile call fails silently loses their entire `amount` of ERC-20 tokens with no recourse. The tokens are burned in the EVM state; no NEAR-side transfer is ever initiated; and no refund mechanism exists. The loss is permanent and irreversible.

---

### Likelihood Explanation

**Medium.**

The failure conditions are reachable by an ordinary EVM user:

1. A user supplies a `recipient` bytes value that is syntactically too long (> `MAX_INPUT_SIZE - 33` bytes) or not a valid NEAR account ID — the precompile returns `ERR_INVALID_INPUT` or `ERR_INVALID_RECEIVER_ACCOUNT_ID`, the assembly `call` returns 0, and the burn is not rolled back.
2. If the NEP-141↔ERC-20 mapping is ever removed or corrupted (e.g., during a migration), `ERR_TARGET_TOKEN_NOT_FOUND` is returned for every user attempting to exit, silently burning all their tokens.
3. Analogous to the rETH deposit delay: if the Aurora eth-connector contract is paused (a supported administrative action), the NEAR-side `withdraw` call fails. For `ExitToEthereum` exits, there is no refund callback, so all in-flight exits result in permanent token loss.

---

### Recommendation

1. **Check the assembly `call` return value and revert on failure** in both `EvmErc20.sol` and `EvmErc20V2.sol`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures `_burn` is atomically rolled back if the precompile fails, mirroring the protection that rETH's `_beforeTokenTransfer` hook provides (it reverts the entire transfer on delay violation).

2. **Add an error-refund callback to `ExitToEthereum`** analogous to the one already present in `ExitToNear`, so that NEAR-side `withdraw` failures trigger a re-mint of the burned ERC-20 tokens. [12](#0-11) 

---

### Proof of Concept

1. Deploy or use an existing `EvmErc20` mirror token on Aurora.
2. Call `withdrawToNear(recipient, amount)` where `recipient` is a bytes value that is a valid length but contains characters that make it an invalid NEAR account ID (e.g., `bytes("INVALID ACCOUNT!!!")`).
3. The `_burn` executes, reducing the caller's balance by `amount`.
4. The precompile call returns `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`, so the assembly `call` returns `0`.
5. The Solidity function returns normally (no revert).
6. Observe: the caller's ERC-20 balance is reduced by `amount`; no NEAR-side `ft_transfer` was ever scheduled; the tokens are permanently lost.

The same outcome occurs for `withdrawToEthereum` and for `EvmErc20V2`. [13](#0-12) [14](#0-13)
