### Title
Unvalidated `sender` Argument in Bank Precompile `transfer` Allows Unauthorized Native Token Theft - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract` precompile at address `0x0000000000000000000000000000000000000064` exposes a `transfer(address sender, address recipient, uint256 amount)` method. The `sender` argument is taken directly from user-supplied calldata and used as the `from` address in `bankKeeper.SendCoins` with no check that `sender == contract.Caller()` or that the calling contract has any authorization to move tokens on behalf of `sender`. Any unprivileged contract can therefore drain `evm/<callerContract>` native tokens from any victim account that holds them.

### Finding Description
In `BankContract.Run`, the `TransferMethodName` case unpacks `sender` from the ABI-encoded input and passes it directly to `bankKeeper.SendCoins`:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(to); err != nil { ... }
denom := EVMDenom(contract.Caller())          // evm/<callingContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The only guard applied to `sender` is `checkBlockedAddr` on the **recipient**, not the sender. There is no assertion that `from == contract.Caller()` (i.e., that the calling contract is the token issuer moving its own escrow balance) nor any allowance/approval check. The denom is scoped to `evm/<contract.Caller()>`, so the calling contract controls which token is moved, but it can move that token **from any address** to any address.

The `mint` method (same precompile, same caller-scoped denom) allows a contract to mint `evm/<self>` tokens to arbitrary recipients. Once users hold such balances, the same contract — or any other contract that can be made to call the precompile — can invoke `transfer(victimAddr, attackerAddr, amount)` to steal them without the victim's consent.

The `burn` case has the same structural flaw: `addr` (the address whose tokens are burned) is the first user-supplied argument with no authorization check, allowing any contract to destroy `evm/<callerContract>` tokens held by any account.

### Impact Explanation
This is a **Critical** unauthorized transfer/burn of CRC20/native assets. An attacker who deploys a contract that:
1. Mints `evm/<attackerContract>` tokens to users (e.g., as part of a DeFi protocol), and then
2. Calls `bank.transfer(victimAddr, attackerAddr, amount)` via the precompile

can steal the full native-bank balance of every user who holds `evm/<attackerContract>` tokens. Because the denom is caller-scoped, the attacker controls both the issuance and the theft vector within the same contract. The Cosmos bank module executes the transfer with no further authorization check.

### Likelihood Explanation
Any unprivileged EVM contract can call the bank precompile at `0x0000000000000000000000000000000000000064`. No admin key, governance vote, or special permission is required. The attack is reachable in a single EVM transaction.

### Recommendation
In the `TransferMethodName` case, enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}
```

This mirrors the authorization model used by `mint`/`burn` (where the denom is already scoped to `contract.Caller()`) and prevents any contract from moving tokens it does not own. Apply the same guard to the `burn` case: verify that `addr == contract.Caller()` before burning.

### Proof of Concept
1. Attacker deploys `MaliciousToken` contract. It calls `bank.mint(userAddr, 1000)` to distribute `evm/<MaliciousToken>` tokens to users.
2. Users accumulate `evm/<MaliciousToken>` native balances (visible via `bank.balanceOf`).
3. Attacker calls `MaliciousToken.steal(victimAddr, attackerAddr, 1000)`, which internally calls `bank.transfer(victimAddr, attackerAddr, 1000)` on the precompile.
4. `BankContract.Run` executes `bankKeeper.SendCoins(ctx, victimAddr, attackerAddr, {evm/<MaliciousToken>: 1000})` with no authorization check.
5. Victim's entire `evm/<MaliciousToken>` balance is transferred to the attacker.

The root cause is identical to the GroupBuy analog: a user-supplied address parameter (`_market` there, `sender` here) is passed to a privileged operation without any whitelist or ownership validation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
