### Title
Missing Proof-of-Correctness Validation for Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` aggregates `CKDOutput` contributions from all participants by simply adding received group elements together, with no proof-of-correctness or any other validation that each contribution was computed using the participant's actual private share and the correct `app_id`/`app_pk`. This is the direct analog of the Chainlink staleness bug: just as `getPrice` consumed oracle data without checking `answeredInRound >= roundId` or `updatedAt`, the CKD coordinator consumes participant data without checking that it was produced from the correct inputs. A single malicious participant can inject arbitrary group elements, causing every honest party to accept a corrupted, unusable CKD output.

---

### Finding Description

**Root cause — `do_ckd_coordinator` (lines 35–58):**

```rust
async fn do_ckd_coordinator(...) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    let waitpoint = chan.next_waitpoint();
    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();   // ← no validation
        norm_big_c += participant_output.big_c();   // ← no validation
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
``` [1](#0-0) 

Each honest participant computes their contribution in `compute_signature_share` as:

```
big_s  = H(pk ‖ app_id) · private_share_i
big_c  = big_s + app_pk · y_i
norm_big_y = (y_i · G) · λ_i
norm_big_c = big_c · λ_i
``` [2](#0-1) 

The coordinator receives `(norm_big_y, norm_big_c)` over the channel and adds them directly into the running aggregate. There is **no zero-knowledge proof** (or any other check) that the received pair was computed from the participant's actual `private_share`, the correct `hash_point = H(pk ‖ app_id)`, or the correct `app_pk`. The only structural check is that the deserialized type is `CKDOutput`; the content is never validated.

**Contrast with DKG:** The DKG protocol (`src/dkg.rs`) requires every participant to supply a Schnorr proof-of-knowledge over their committed polynomial, and every commitment is bound to the session ID via `verify_proof_of_knowledge` and `verify_commitment_hash`. [3](#0-2)  The CKD protocol has no equivalent safeguard.

**Exploit path:**

1. A malicious participant `P_m` joins a legitimate CKD session with a valid key share.
2. Instead of calling `compute_signature_share`, `P_m` sends an arbitrary pair `(A, B)` — e.g., both set to the group identity, or to values that steer the aggregate toward an attacker-chosen point.
3. The coordinator receives `(A, B)` via `recv_from_others`, adds them to `norm_big_y` and `norm_big_c` without any check, and returns the corrupted `CKDOutput` as the protocol result.
4. The TEE application calls `ckd_output.unmask(app_sk)` on the corrupted output and derives an incorrect confidential key — one that does not equal `H(pk ‖ app_id) · msk`. [4](#0-3) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The coordinator is the sole party that sees the aggregated result and returns it as the protocol output. All honest parties (including the TEE application) accept this output as authoritative. Because the confidential key derivation is deterministic given correct inputs, a corrupted output cannot be distinguished from a correct one without an independent reference. The TEE application will silently use an incorrect derived key, breaking any downstream operation that depends on it (e.g., decryption, authentication, secret provisioning).

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. No privileged access, leaked key material, or cryptographic break is required — only the ability to send a crafted message over the authenticated channel, which every participant already possesses. The attack is a single-message deviation from the honest protocol and requires no coordination.

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct computation — specifically, a proof that `norm_big_c - norm_big_y_scaled` lies on the correct coset defined by `H(pk ‖ app_id)` and the participant's committed public share. A discrete-log equality proof (analogous to the `dlogeq` proofs already used in triple generation) would suffice: prove that the discrete log of `norm_big_c - app_pk · (norm_big_y / G)` with respect to `H(pk ‖ app_id)` equals the discrete log of the participant's public share `C_i(0)` with respect to the generator `G`. The coordinator must verify this proof before adding the contribution to the aggregate.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
