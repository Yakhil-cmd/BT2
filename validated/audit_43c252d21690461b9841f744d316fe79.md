### Title
Unchecked Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Freeze - (File: etc/eth-contracts/contracts/EvmErc20.sol)

---

### Summary

The `withdrawToNear` and `withdrawToEthereum` functions in `EvmErc20.sol` and `EvmErc20V2.sol` irreversibly burn a user's ERC-20 tokens **before** calling the `ExitToNear`/`ExitToEthereum` precompile via inline assembly. The return value `res` of the `call` opcode is captured but **never checked**. If the precompile returns an error (EVM-level failure), the `call` opcode returns `0`, the burn is not reverted, and the NEAR-side transfer is never initiated — permanently destroying the user's funds.

---

### Finding Description

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear` and `withdrawToEthereum` follow the same pattern:

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // <-- tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum` and in both functions of `EvmErc20V2.sol`: [2](#0-1) [3](#0-2) [4](#0-3) 

The `ExitToNear` precompile's `run` function returns `Err(ExitError)` in multiple reachable paths — for example, when `ExitToNearParams::try_from(input)` fails due to an invalid NEAR recipient account ID, when `get_nep141_from_erc20` cannot find a registered NEP-141 mapping, or when `exit_erc20_token_to_near` returns an error: [5](#0-4) [6](#0-5) 

When the precompile returns `Err(ExitError)`, the EVM `call` opcode sets `res = 0`. Because `res` is never tested, the Solidity function returns normally. The `_burn` executed before the assembly block is **not** rolled back — it is a permanent state change in the EVM sub-call context.

**The `error_refund` mechanism in `EvmErc20V2.sol` does not mitigate this.** The refund callback (`exit_to_near_precompile_callback`) is only scheduled when the precompile itself **succeeds** and the downstream NEAR promise fails. If the precompile fails at the EVM level, the callback is never registered: [7](#0-6) 

The engine deploys `EvmErc20V2.bin` when the `error_refund` feature is enabled, and `EvmErc20.bin` otherwise — both binaries carry the same unchecked-`res` defect: [8](#0-7) 

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

Any ERC-20 tokens burned by `_burn` before a failing precompile call are irrecoverably destroyed. The NEP-141 balance on the NEAR side is never decremented (the `ft_transfer` promise is never created), so the accounting diverges: the EVM supply shrinks while the NEAR supply stays the same, but the user receives nothing. The lost tokens cannot be recovered by any on-chain mechanism.

---

### Likelihood Explanation

**Medium.** The precompile fails at the EVM level whenever the caller-supplied `recipient` bytes do not decode to a valid NEAR account ID (e.g., contains uppercase letters, spaces, or exceeds the 64-character limit). NEAR account ID validation is non-trivial and not enforced by the Solidity function signature (`bytes memory recipient`). A user or integrating contract that passes a malformed recipient — including a hex-encoded Ethereum address mistakenly passed to `withdrawToNear` — will silently lose their tokens. No privileged access is required; any token holder can trigger this path.

---

### Recommendation

Add an explicit revert inside the assembly block if the precompile call fails:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` and to both functions in `EvmErc20V2.sol`. Additionally, consider validating the `recipient` bytes against NEAR account ID rules in Solidity before burning, so users receive a descriptive error rather than a silent revert.

---

### Proof of Concept

1. Alice holds 1000 units of an ERC-20 mirror token on Aurora (backed by a NEP-141 on NEAR).
2. Alice calls `withdrawToNear(bytes("INVALID ACCOUNT ID WITH SPACES"), 1000)`.
3. `_burn(Alice, 1000)` executes — Alice's EVM balance drops to 0.
4. The assembly `call` invokes the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`.
5. Inside `ExitToNear::run`, `ExitToNearParams::try_from(input)` fails because `"INVALID ACCOUNT ID WITH SPACES"` is not a valid NEAR account ID — the precompile returns `Err(ExitError)`.
6. The EVM `call` opcode sets `res = 0`. The assembly block exits without reverting.
7. `withdrawToNear` returns successfully. No NEAR promise is created. No `ft_transfer` is issued.
8. Alice's 1000 ERC-20 tokens are permanently destroyed; she receives nothing on NEAR. [1](#0-0) [9](#0-8)
