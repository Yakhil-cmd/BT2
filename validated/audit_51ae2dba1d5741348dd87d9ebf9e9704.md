### Title
Gateway Stateful Validator Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` for Admission, Causing Incorrect Admission Decisions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `StatefulTransactionValidator` validates a transaction's L2 gas price against the **current committed block's `l2_gas_price`** field, but the block that will actually execute the transaction uses **`next_l2_gas_price`** (the EIP-1559-computed price stored in the same block header). This off-by-one reference causes the gateway to accept transactions the batcher will reject (when gas price is rising) and to reject transactions the batcher would accept (when gas price is falling). The code itself acknowledges the bug with a TODO comment.

---

### Finding Description

The `StorageBlockHeader` stores two distinct gas price fields:

- `l2_gas_price`: the gas price **used for the current block**
- `next_l2_gas_price`: the gas price **pre-computed for the next block** via the EIP-1559 mechanism [1](#0-0) 

The gateway's `validate_resource_bounds` reads the previous block's info and extracts `gas_prices.strk_gas_prices.l2_gas_price`, which maps to `header.l2_gas_price.price_in_fri` — the price for the **already-committed** block, not the price for the **next** block being built. The TODO comment in the code explicitly acknowledges this is wrong: [2](#0-1) 

The same stale price is also used in `run_validate_entry_point`, which constructs the block context for blockifier validation. It increments `block_number` to simulate the next block but leaves all gas prices unchanged from the previous block: [3](#0-2) 

The batcher, however, receives the gas price from the consensus orchestrator's EIP-1559 state (`self.l2_gas_price`), which equals `committed_block.next_l2_gas_price`. The batcher's `get_block_info` helper also reads only `header.l2_gas_price.price_in_fri` (the current block's price), confirming that the next block's actual execution price is `next_l2_gas_price`, not `l2_gas_price`: [4](#0-3) 

The EIP-1559 update formula changes the gas price by approximately `price / 333` per block (~0.3%): [5](#0-4) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

Two concrete failure modes:

1. **Gas price rising** (`next_l2_gas_price > l2_gas_price`, i.e., high block utilization): The gateway accepts a transaction with `tx.max_price = l2_gas_price`, but the batcher rejects it because `tx.max_price < next_l2_gas_price`. The transaction occupies a mempool slot and is never sequenced.

2. **Gas price falling** (`next_l2_gas_price < l2_gas_price`, i.e., low block utilization): The gateway rejects a transaction with `tx.max_price = next_l2_gas_price` because `tx.max_price < l2_gas_price * threshold`. The batcher would have accepted it. Legitimate users are denied service.

The `validate_tx_l2_gas_price_within_threshold` check uses `min_gas_price_percentage = 100` by default, meaning the threshold is exactly `l2_gas_price`: [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

This divergence is present in **every block** where gas usage is non-zero, because the EIP-1559 mechanism always produces `next_l2_gas_price ≠ l2_gas_price` under any non-target utilization. The gap grows with sustained high or low utilization. Any user who sets `max_price_per_unit` to exactly the current block's gas price (a natural choice) will be affected.

---

### Recommendation

Replace the `l2_gas_price` read in `validate_resource_bounds` with `next_l2_gas_price` from the block header, as the TODO comment already prescribes:

```rust
// In validate_resource_bounds:
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info_with_next_price()  // or extend get_block_info to return next_l2_gas_price
    .await?
    .next_l2_gas_price;
```

Similarly, in `run_validate_entry_point`, after incrementing `block_number`, also update `block_info.gas_prices.strk_gas_prices.l2_gas_price` (and the ETH equivalent) to `next_l2_gas_price` from the header, so the blockifier's `check_fee_bounds` uses the same price the batcher will use.

### Proof of Concept

```
Block N committed with:
  l2_gas_price     = 10_000_000_000 fri  (used for block N)
  next_l2_gas_price = 10_030_030_030 fri  (= 10B + 10B/333, for block N+1)

User submits invoke tx with:
  resource_bounds.l2_gas.max_price_per_unit = 10_000_000_000 fri

Gateway stateful validation (validate_resource_bounds):
  threshold = 10_000_000_000 * 100% = 10_000_000_000
  10_000_000_000 >= 10_000_000_000  → PASS ✓

Gateway blockifier validation (check_fee_bounds, run_validate_entry_point):
  block_context.l2_gas_price = 10_000_000_000  (stale, from block N)
  10_000_000_000 >= 10_000_000_000  → PASS ✓

Transaction admitted to mempool.

Batcher builds block N+1 with:
  block_context.l2_gas_price = 10_030_030_030  (= block N's next_l2_gas_price)
  check_fee_bounds: 10_000_000_000 < 10_030_030_030  → FAIL ✗

Transaction is rejected at execution time; mempool slot wasted.
```
