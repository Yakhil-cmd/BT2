### Title
Reverted Invoke Transaction Omits Execute-Phase Resources from `InvokeTransactionTrace::execution_resources` in `starknet_simulateTransactions` — (`crates/apollo_rpc/src/v0_8/execution.rs`)

### Summary

When a reverted Invoke transaction is converted to an RPC `InvokeTransactionTrace`, the `execution_resources` aggregation unconditionally substitutes `ExecutionResources::default()` (zero) for the execute phase. The blockifier receipt **does** track the steps consumed before revert as `n_reverted_steps`, but the RPC conversion layer never reads the receipt — it only sums per-invocation resources. The result is that `starknet_simulateTransactions` returns an authoritative-looking `execution_resources` value that silently omits all execute-phase VM work for any reverted Invoke.

---

### Finding Description

**Step 1 — Blockifier tracks reverted steps in the receipt, not in `execute_call_info`.**

`ValidateExecuteCallInfo::new_reverted` always sets `execute_call_info: None`: [1](#0-0) 

The actual steps consumed before the revert are captured as `execution_steps_consumed` and passed to `TransactionReceipt::from_account_tx` as `n_reverted_steps`: [2](#0-1) 

A blockifier test explicitly asserts this: `assert!(tx_execution_info.receipt.resources.computation.n_reverted_steps > 0)`: [3](#0-2) 

**Step 2 — The execution-layer trace conversion discards the receipt.**

`TryFrom<TransactionExecutionInfo> for InvokeTransactionTrace` (in `apollo_rpc_execution`) converts a reverted transaction to `FunctionInvocationResult::Err(revert_reason_string)` — a bare string with no resource data. The receipt is never consulted for the execute phase: [4](#0-3) 

**Step 3 — The RPC conversion layer substitutes zero for the missing execute resources.**

In `From<(ExecutionTransactionTrace, ThinStateDiff)> for TransactionTrace`, the `Err` arm hard-codes `ExecutionResources::default()`: [5](#0-4) 

The final `execution_resources` field is then:

```
validate_execution_resources.unwrap_or_default()
  + ExecutionResources::default()   // ← always zero for reverted Invoke
  + fee_transfer_execution_resources.unwrap_or_default()
``` [6](#0-5) 

The execute-phase VM steps, builtins, and gas consumed before the revert are entirely absent from the returned value.

---

### Impact Explanation

`starknet_simulateTransactions` returns an `InvokeTransactionTrace` whose top-level `execution_resources` field is structurally wrong for any reverted Invoke transaction. The field is described in the OpenRPC spec as "the resources consumed by the transaction, includes both computation and data": [7](#0-6) 

Clients that use this field to audit resource consumption, build fee models, or verify that a simulation matches an on-chain receipt will receive a silently underreported value. The fee itself is charged correctly (it is derived from the receipt, not from the trace), but the simulation trace misrepresents the actual work performed.

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a `BroadcastedTransaction::Invoke` whose `__execute__` reverts (e.g., calling a contract that panics, or exceeding a resource bound). No special privileges are required. The condition is deterministic and reproducible.

---

### Recommendation

In `From<(ExecutionTransactionTrace, ThinStateDiff)> for TransactionTrace`, the `Err` arm should not substitute `ExecutionResources::default()`. Instead, the execute-phase resources for reverted transactions should be sourced from `TransactionExecutionInfo::receipt.resources.computation.n_reverted_steps` (and the corresponding sierra gas / builtins). This requires either:

1. Propagating the receipt alongside the `ExecutionTransactionTrace` so the RPC conversion layer can read `n_reverted_steps`, or
2. Extending `FunctionInvocationResult::Err` to carry an `ExecutionResources` field populated from the receipt at the `objects.rs` conversion layer.

---

### Proof of Concept

The following logic (expressible as a Rust unit test against `exec_simulate_transactions`) confirms the issue:

1. Deploy an account and a contract whose `__execute__` always reverts after consuming a non-trivial number of steps.
2. Call `exec_simulate_transactions` with `validate=true`, `charge_fee=true`.
3. Assert `simulated_tx.transaction_trace` is `TransactionTrace::Invoke(trace)` where `trace.execute_invocation` is `FunctionInvocationResult::Err(_)`.
4. Assert `trace.execution_resources` equals `validate_resources + fee_transfer_resources` (i.e., the execute-phase contribution is zero).
5. Assert `trace.execution_resources` is **strictly less than** the resources recorded in `TransactionExecutionInfo::receipt` (which includes `n_reverted_steps > 0`).

Step 4 will pass, confirming the underreported value. Step 5 confirms the discrepancy against the authoritative receipt.
