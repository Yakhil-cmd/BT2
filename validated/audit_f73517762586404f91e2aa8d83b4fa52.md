### Title
Gateway `verify_can_pay_committed_bounds` balance check does not account for total committed fees of other pending transactions from the same account - (File: crates/blockifier/src/fee/fee_utils.rs)

### Summary
`verify_can_pay_committed_bounds` checks whether an account's current on-chain balance covers the `max_possible_fee` of the **incoming** transaction only. It does not subtract the total `max_possible_fee` already committed by other pending transactions from the same account sitting in the mempool. An unprivileged sender can therefore submit N sequential transactions each declaring `max_possible_fee ≈ balance`, all of which pass gateway admission, while the account can only pay for the first one.

### Finding Description
During gateway stateful validation, `extract_state_nonce_and_run_validations` calls `run_validate_entry_point`, which internally calls `AccountTransaction::perform_pre_validation_stage`. That function calls `verify_can_pay_committed_bounds`:

```rust
// crates/blockifier/src/fee/fee_utils.rs:173-203
pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let committed_fee = tx_context.max_possible_fee();          // only THIS tx
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    ...
}
```

`get_balance_and_if_covers_fee` reads the **current committed-block balance** of the sender and checks `balance >= committed_fee_of_this_tx`. It has no knowledge of other pending transactions from the same sender already in the mempool.

The mempool's own `validate_tx` path also does not fill this gap — it only validates nonce ordering and fee escalation:

```rust
// crates/apollo_mempool/src/mempool.rs:402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
```

Neither `validate_incoming_tx` nor `validate_fee_escalation` sums the `max_possible_fee` of all already-pooled transactions for the same address.

### Impact Explanation
An account with balance `B` can submit `N` transactions (nonces 0…N-1), each with `max_possible_fee = B`. Every submission passes the gateway balance check independently (`B ≤ B`). All `N` transactions are admitted to the mempool. At execution time, transaction 0 pays its fee and reduces the balance; transactions 1…N-1 will each fail with `InsufficientFeeTokenBalance` and be reverted. The gateway has accepted `N-1` transactions that it should have rejected, matching the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

### Likelihood Explanation
Any unprivileged account can trigger this with a single STRK balance and sequential nonce submissions. No special permissions, contract deployment, or privileged access is required. The gateway's stateful validation path is the standard ingestion path for all transactions.

### Recommendation
When admitting a new transaction, the gateway (or mempool's `validate_tx`) should compute the **total** `max_possible_fee` across all pending transactions already pooled for the same sender address and verify:

```
balance >= sum(max_possible_fee(pending_txs_for_sender)) + max_possible_fee(new_tx)
```

This mirrors the fix recommended in the external report: check the total locked liquidity across all consumers of the shared resource, not just the current one.

### Proof of Concept
1. Account `A` has on-chain balance = 100 STRK.
2. Submit `Tx1` (nonce=0, `max_possible_fee`=100 STRK) → `verify_can_pay_committed_bounds` passes (100 ≤ 100). Admitted.
3. Submit `Tx2` (nonce=1, `max_possible_fee`=100 STRK) → `verify_can_pay_committed_bounds` reads the same committed-block balance (still 100 STRK, `Tx1` not yet executed) and passes (100 ≤ 100). Admitted.
4. Submit `Tx3` … `TxN` similarly — all admitted.
5. Batcher executes `Tx1`: fee paid, balance drops to ~90 STRK.
6. Batcher executes `Tx2`: `check_can_pay_fee` fails with `InsufficientFeeTokenBalance`; transaction reverts. Revert fee charged.
7. `Tx3`…`TxN` all revert similarly.

The gateway admitted `N-1` transactions that were guaranteed to fail at execution time because the total committed fee (`N × 100 STRK`) far exceeded the account's balance (`100 STRK`), directly mirroring the external bug's pattern of over-committing a shared resource. /// Returns the current fee balance and a boolean indicating whether the balance covers the fee.
pub fn get_balance_and_if_covers_fee(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
    fee: Fee,
) -> TransactionFeeResult<(Felt, Felt, bool)> {
    let tx_info = &tx_context.tx_info;
    let (balance_low, balance_high) =
        state.get_fee_token_balance(tx_info.sender_address(), tx_context.fee_token_address())?;
    // TODO(Dori,1/10/2023): If/when fees can be more than 128 bit integers, this should be updated.
    let has_sufficient_balance = balance_high > Felt::ZERO || balance_low >= Felt::from(fee.0);
    if !has_sufficient_balance {
        debug!(
            "Fee token balance check: sender_address={}, tx_hash={}, balance={}, expected fee={}",
            tx_info.sender_address(),
            tx_info.transaction_hash(),
            balance_low,
            fee.0
        );
    }
    Ok((balance_low, balance_high, has_sufficient_balance))
}

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}

    // Performs static checks before executing validation entry point.
    // Note that nonce is incremented during these checks.
    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
