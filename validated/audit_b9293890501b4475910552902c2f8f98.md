Audit Report

## Title
Missing ICP/XDR Rate Staleness Check Allows Treasury Valuation Cap Bypass — (File: `rs/sns/governance/token_valuation/src/lib.rs`)

## Summary
`CmcBased30DayMovingAverageXdrsPerIcpClient::get()` fetches the ICP/XDR rate from the CMC and uses `response.data.xdr_permyriad_per_icp` directly without inspecting `response.data.timestamp_seconds`. This rate feeds the `ProposalsAmountTotalUpperBound` computation that gates `TransferSnsTreasuryFunds` and `MintSnsTokens` proposals. When the CMC's stored average rate is stale (due to sustained XRC HTTP-outcall failures), the treasury's XDR valuation is underestimated, allowing proposals that exceed the intended 7-day transfer cap to be submitted, voted on, and executed.

## Finding Description
In `rs/sns/governance/token_valuation/src/lib.rs` at lines 435–459, `CmcBased30DayMovingAverageXdrsPerIcpClient::get()` calls `get_average_icp_xdr_conversion_rate` on the CMC and immediately extracts the rate value:

```rust
let xdr_per_icp =
    Decimal::from(response.data.xdr_permyriad_per_icp) * *UNITS_PER_PERMYRIAD;
Ok(xdr_per_icp)
```

`response.data.timestamp_seconds` is never read. The `IcpXdrConversionRate` struct in `rs/nns/cmc/src/lib.rs` (lines 487–497) carries this field explicitly to record when the market data was queried.

The CMC refreshes its `average_icp_xdr_conversion_rate` only when the XRC successfully returns a new rate (every 5 minutes via `REFRESH_RATE_INTERVAL_SECONDS` in `rs/nns/cmc/src/exchange_rate_canister.rs` line 16). If the XRC's HTTP outcalls fail continuously, the CMC's stored average is not updated; its `timestamp_seconds` remains at the last successful computation.

The fetched rate flows into `try_get_balance_valuation_factors` (`rs/sns/governance/token_valuation/src/lib.rs` lines 141–191), which assembles `ValuationFactors`. This is passed to `ProposalsAmountTotalUpperBound::in_tokens` (`rs/sns/governance/proposals_amount_total_limit/src/lib.rs` lines 66–114), which classifies the treasury as small (≤ 100,000 XDR → `NoLimit`), medium (≤ 1,200,000 XDR → 25% fraction), or large (> 1,200,000 XDR → 300,000 XDR absolute cap).

The existing `MIN_XDRS_PER_ICP = dec!(1)` floor (`rs/sns/governance/proposals_amount_total_limit/src/lib.rs` line 64) only clamps rates below 1 XDR/ICP. It does not protect against a stale rate that is above 1 XDR/ICP but materially below the true current rate (e.g., a stale 5 XDR/ICP when the true rate is 10 XDR/ICP).

Critically, the valuation captured at proposal submission time is stored in `ActionAuxiliary::TransferSnsTreasuryFunds` and reused verbatim at execution time by `transfer_sns_treasury_funds_amount_is_small_enough_at_execution_time_or_err` (`rs/sns/governance/src/proposal.rs` lines 2600–2658). A stale valuation at submission time therefore persists through execution even if the rate recovers.

NNS governance, by contrast, maintains its own cached rate and checks staleness via `should_refresh_xdr_rate` (`rs/nns/governance/src/governance.rs` lines 6336–6347), which enforces a 1-day freshness window. No equivalent check exists in the SNS path.

## Impact Explanation
This matches the **High** impact class: "Unauthorized access to … canister-controlled funds where exploitation requires meaningful per-target work or other constraints."

Concrete scenario: a treasury holding 200,000 ICP at a true rate of 10 XDR/ICP is worth 2,000,000 XDR (large regime; 300,000 XDR cap ≈ 30,000 ICP per 7-day window). With a stale rate of 4 XDR/ICP, the same treasury appears as 800,000 XDR (medium regime; 25% cap = 50,000 ICP). A proposal for 50,000 ICP passes the submission check and is stored with the stale valuation; at execution time the same stale valuation is reused, so the transfer proceeds — 67% more than the intended limit.

In the boundary case where a stale rate pushes a medium treasury below 100,000 XDR, the `NoLimit` branch applies and the entire treasury balance can be proposed and executed in a single proposal.

## Likelihood Explanation
The XRC canister fetches prices via HTTP outcalls to external exchanges. Transient failures (exchange downtime, network partitions, rate-limiting) cause the XRC to return errors, which cause the CMC's heartbeat to skip the rate update. A sustained outage of hours to days is realistic without requiring any subnet-majority corruption or privileged access. No attacker action is required to create the stale-rate condition; the attacker only needs to submit a `TransferSnsTreasuryFunds` proposal during the outage window. The SNS community voting on the proposal has no on-chain signal that the rate is stale, since the valuation displayed in the proposal rendering is derived from the same stale rate.

## Recommendation
After receiving the CMC response in `CmcBased30DayMovingAverageXdrsPerIcpClient::get()`, check that `response.data.timestamp_seconds` is within an acceptable freshness window before using the rate:

```rust
let now_seconds = /* ic_cdk::api::time() / 1_000_000_000 */;
let rate_age_seconds = now_seconds.saturating_sub(response.data.timestamp_seconds);
const MAX_RATE_AGE_SECONDS: u64 = 2 * 24 * 60 * 60; // 48 hours
if rate_age_seconds > MAX_RATE_AGE_SECONDS {
    return Err(ValuationError::new_external(format!(
        "ICP/XDR rate is stale: age {} s exceeds limit {} s",
        rate_age_seconds, MAX_RATE_AGE_SECONDS,
    )));
}
```

A 24–48 hour threshold is consistent with NNS governance's own 1-day staleness check (`should_refresh_xdr_rate`). Returning an error causes `try_get_balance_valuation_factors` to propagate the error, which causes `treasury_valuation_if_proposal_amount_is_small_enough_or_err` to reject the proposal at submission time — the correct behavior when the rate cannot be trusted.

## Proof of Concept
1. Deploy a local replica with the SNS governance canister and a mock CMC that returns a fixed `average_icp_xdr_conversion_rate` with a `timestamp_seconds` set 72 hours in the past and `xdr_permyriad_per_icp` set to 40,000 (4 XDR/ICP).
2. Fund the SNS ICP treasury with 200,000 ICP (true value at 10 XDR/ICP = 2,000,000 XDR; large regime cap = 30,000 ICP).
3. Submit a `TransferSnsTreasuryFunds` proposal for 50,000 ICP. With the stale rate (4 XDR/ICP), the treasury appears as 800,000 XDR (medium regime; 25% cap = 50,000 ICP). The proposal passes the submission valuation check.
4. Observe that the proposal is accepted for voting (not rejected at submission). Pass the proposal via governance vote.
5. Observe that execution succeeds, transferring 50,000 ICP — 67% more than the 30,000 ICP that the true rate would have permitted.
6. To test the `NoLimit` boundary: set the mock rate to 8,000 permyriad (0.8 XDR/ICP, below `MIN_XDRS_PER_ICP = 1`); the floor clamps it to 1 XDR/ICP. Then set it to 12,000 permyriad (1.2 XDR/ICP, above the floor) with a treasury of 80,000 ICP: apparent value = 96,000 XDR < 100,000 XDR → `NoLimit` → entire treasury can be proposed in one action.
