### Title
Stateless Gateway Unconditionally Applies L2 Gas Price Floor to L1-Gas-Only Transactions, Causing Incorrect Admission Rejection — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` always checks `l2_gas.max_price_per_unit` against the configured `min_gas_price` floor, regardless of whether the transaction actually allocates any L2 gas. In production the floor is `8_000_000_000` FRI. Any transaction that sets `l2_gas.max_amount = 0` (an L1-gas-only transaction) also leaves `l2_gas.max_price_per_unit = 0`, so the comparison `0 < 8_000_000_000` is always true and the gateway returns `MaxGasPriceTooLow`, permanently blocking a class of otherwise-valid transactions at the stateless admission gate.

### Finding Description

`StatelessTransactionValidator::validate_resource_bounds` (lines 56–88) performs two checks after confirming that at least one resource bound is non-zero:

```rust
// line 71-76
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
``` [1](#0-0) 

The check is unconditional: it fires even when `l2_gas.max_amount == 0`, i.e., when the transaction carries no L2 gas budget at all and the price field is semantically irrelevant.

The production default for `min_gas_price` is `8_000_000_000`: [2](#0-1) [3](#0-2) 

The test suite that exercises the "valid L1-gas-only" path silently avoids the issue by overriding `min_gas_price` to `0`:

```rust
static DEFAULT_VALIDATOR_CONFIG_FOR_TESTING: LazyLock<...> =
    LazyLock::new(|| StatelessTransactionValidatorConfig {
        validate_resource_bounds: false,
        min_gas_price: 0,   // ← masks the production defect
        ...
    });
``` [4](#0-3) 

The positive test case `valid_l1_gas` therefore passes in CI but would fail under the production configuration: [5](#0-4) 

The analog in the external report is exact: `getZetaFromToken` always routes through WETH as an intermediate; when the input token *is* WETH the path degenerates to WETH→WETH→ZETA and reverts. Here, `validate_resource_bounds` always routes through the L2 gas price check as an intermediate; when the transaction *is* L1-gas-only the check degenerates to `0 < min_gas_price` and rejects.

### Impact Explanation

Every transaction submitted to the gateway with `AllResourceBounds { l1_gas: <non-zero>, l2_gas: ResourceBounds::default(), ... }` is rejected at the stateless validation stage with `MaxGasPriceTooLow`. The rejection happens before any stateful check, before mempool admission, and before blockifier execution. The transaction is never sequenced. This matches the allowed impact: **"Mempool/gateway/RPC admission rejects valid transactions before sequencing."**

### Likelihood Explanation

L1-gas-only `AllResources` transactions are a documented, supported transaction shape (the test `valid_l1_gas` exists precisely to cover this case). Any wallet or SDK that constructs a transaction with only `l1_gas` set and leaves `l2_gas` at its zero default will be silently rejected in production. The trigger requires no privilege and no special state — a single crafted (or legitimately constructed) RPC transaction suffices.

### Recommendation

Guard the L2 gas price check with a non-zero amount condition:

```rust
if resource_bounds.l2_gas.max_amount.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
```

Equivalently, update the test `valid_l1_gas` to use the production `min_gas_price` value so the regression is caught in CI.

### Proof of Concept

1. Configure the gateway with the production defaults (`validate_resource_bounds: true`, `min_gas_price: 8_000_000_000`).
2. Submit any `RpcTransaction` (Invoke, Declare, or DeployAccount) with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      ResourceBounds { max_amount: 1, max_price_per_unit: 1 },
       l2_gas:      ResourceBounds::default(),   // max_price_per_unit = 0
       l1_data_gas: ResourceBounds::default(),
   }
   ```
3. `StatelessTransactionValidator::validate` calls `validate_resource_bounds`.
4. The zero-fee guard passes (l1_gas contributes a non-zero fee).
5. The price check fires: `0 < 8_000_000_000` → `MaxGasPriceTooLow` is returned.
6. The transaction is rejected at the gateway; it never reaches the mempool or blockifier. [6](#0-5)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
