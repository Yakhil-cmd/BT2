### Title
Missing Whitelist Check in `receive_base_tokens` Allows Non-Whitelisted Addresses to Receive and Permanently Freeze Base Tokens in Silo Mode - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, Aurora Engine enforces an `Address` whitelist to control which EVM addresses may receive tokens. The `receive_erc20_tokens` function correctly checks this whitelist and redirects tokens to the configured `erc20_fallback_address` when the recipient is not whitelisted. However, the sibling function `receive_base_tokens` — called when the ETH connector bridges base tokens (ETH) into Aurora — performs **no whitelist check at all**. A non-whitelisted EVM address can therefore receive base tokens directly, bypassing the silo restriction. Because that address is also blocked from submitting transactions by `is_allow_submit`, the minted tokens become permanently frozen.

---

### Finding Description

`receive_erc20_tokens` in `engine/src/engine.rs` contains the following guard:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

This redirects ERC-20 tokens to the `erc20_fallback_address` when the intended recipient is not in the `Address` whitelist.

`receive_base_tokens`, called for the same bridging flow when the predecessor is the ETH connector, contains no equivalent check:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    ...
}
``` [2](#0-1) 

The dispatch between the two paths occurs in `ft_on_transfer`:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
};
``` [3](#0-2) 

`is_allow_receive_erc20_tokens` checks only the `Address` whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [4](#0-3) 

The same `Address` whitelist also gates transaction submission via `is_allow_submit`:

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [5](#0-4) 

The `erc20_fallback_address` is documented as the address that receives tokens when the intended recipient is not whitelisted: [6](#0-5)

### Impact Explanation

When Silo mode is active with the `Address` whitelist enabled:

1. A user bridges base tokens (ETH) to Aurora by calling `ft_transfer_call` on the ETH connector, specifying a non-whitelisted EVM address as the recipient in the message.
2. The ETH connector calls `ft_on_transfer` on Aurora. Because `predecessor_account_id == connector_account_id`, `receive_base_tokens` is invoked.
3. `receive_base_tokens` mints the full token amount directly to the non-whitelisted address — no whitelist check, no fallback redirect.
4. The non-whitelisted address now holds base tokens but cannot submit any transaction to spend them (blocked by `is_allow_submit`).
5. The tokens are permanently frozen in the non-whitelisted address. The `erc20_fallback_address` — which the silo operator configured to receive such tokens — receives nothing.

**Impact**: Permanent freezing of user funds. The silo operator's fallback mechanism is silently bypassed for base token deposits.

---

### Likelihood Explanation

- Silo mode with an active `Address` whitelist is a supported, documented production configuration.
- Any user holding ETH on NEAR can trigger this by calling `ft_transfer_call` on the ETH connector with a non-whitelisted EVM address as the recipient — no special privileges required.
- The asymmetry between `receive_base_tokens` and `receive_erc20_tokens` is a straightforward omission that is easy to trigger accidentally (e.g., a user who was whitelisted, then removed, bridges tokens to their old address).

---

### Recommendation

Apply the same fallback-redirect guard in `receive_base_tokens` that exists in `receive_erc20_tokens`:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut receipient = message_data.recipient;

    // Mirror the ERC-20 whitelist guard for base tokens in silo mode.
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
    {
        receipient = fallback_address;
    }

    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    ...
}
```

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a non-zero `erc20_fallback_address` and `fixed_gas`.
2. Enable the `Address` whitelist via `set_whitelist_status(WhitelistKind::Address, active: true)`.
3. Do **not** add `victim_address` to the `Address` whitelist.
4. From any NEAR account holding ETH, call `ft_transfer_call` on the ETH connector contract, with `receiver_id = aurora`, `amount = X`, and `msg` encoding `victim_address` as the EVM recipient.
5. Observe: `receive_base_tokens` mints `X` ETH to `victim_address` with no whitelist check.
6. Attempt any EVM transaction from `victim_address` — it is rejected by `is_allow_submit` because `victim_address` is not in the `Address` whitelist.
7. The `X` ETH is permanently frozen. The `erc20_fallback_address` received nothing, contrary to the silo operator's intent.
