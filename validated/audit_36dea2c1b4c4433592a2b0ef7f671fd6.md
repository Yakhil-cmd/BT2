### Title
Owner Can Bypass Upgrade Time-Lock by Setting `upgrade_delay_blocks` to Zero - (File: `engine/src/contract_methods/admin.rs`)

### Summary

The Aurora Engine implements an upgrade time-lock via `upgrade_delay_blocks` to give users time to react before a new contract version is deployed. However, `set_upgrade_delay_blocks` imposes no minimum value constraint, allowing the owner to reduce the delay to zero at any time — including immediately before staging a malicious upgrade — rendering the time-lock protection illusory.

### Finding Description

The upgrade flow works as follows:

1. `stage_upgrade` computes and stores `delay_block_height = env.block_height() + state.upgrade_delay_blocks`.
2. `deploy_upgrade` enforces `if io.block_height() <= index { panic ERR_NOT_ALLOWED_TOO_EARLY }`. [1](#0-0) [2](#0-1) 

The intent is that users have `upgrade_delay_blocks` NEAR blocks to observe a staged upgrade and exit before it takes effect.

However, `set_upgrade_delay_blocks` unconditionally writes the caller-supplied value with no lower-bound check: [3](#0-2) 

The `EngineState` field `upgrade_delay_blocks` is a plain `u64` with no enforced minimum: [4](#0-3) 

The bypass sequence:

1. Owner calls `set_upgrade_delay_blocks(0)` → `state.upgrade_delay_blocks = 0`.
2. Owner calls `stage_upgrade(malicious_code)` → `delay_block_height = current_block + 0 = current_block`.
3. In the very next NEAR block, `deploy_upgrade` is called (callable by anyone per the public entrypoint) → `block_height > delay_block_height` passes, malicious code is deployed. [5](#0-4) 

This is structurally identical to the Holdefi H01 pattern: the admin bypasses a time-restriction not by waiting, but by manipulating the parameter that controls the restriction immediately before the protected action.

### Impact Explanation

If the owner deploys a malicious upgrade (e.g., one that redirects ETH connector withdrawals, mints unbacked ETH, or drains bridged ERC-20 balances), all user funds held in the Aurora EVM are at risk. The upgrade time-lock is the only on-chain mechanism giving users advance notice. Setting it to zero eliminates that window entirely.

**Impact: Critical — direct theft of user funds at rest.**

### Likelihood Explanation

Requires a malicious or compromised owner account. The time-lock exists precisely because this threat model is acknowledged by the protocol. The bypass is trivially executable in two sequential NEAR transactions with no additional preconditions.

### Recommendation

1. Enforce a non-zero minimum on `set_upgrade_delay_blocks` (e.g., reject values below a protocol-defined constant such as 1000 blocks).
2. Apply a time-lock to changes of `upgrade_delay_blocks` itself, so the owner cannot reduce the delay and immediately exploit the reduced window.

### Proof of Concept

```
// Block N
owner.call("set_upgrade_delay_blocks", { upgrade_delay_blocks: 0 })
owner.call("stage_upgrade", malicious_wasm_bytes)
// delay_block_height stored = N + 0 = N

// Block N+1 (anyone can call deploy_upgrade)
anyone.call("deploy_upgrade")
// block_height (N+1) > index (N) → check passes → malicious code deployed

pub fn set_upgrade_delay_blocks<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let args: SetUpgradeDelayBlocksArgs = io.read_input_borsh()?;
        state.upgrade_delay_blocks = args.upgrade_delay_blocks;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })

    /// Deploy staged upgrade.
    #[unsafe(no_mangle)]
    pub extern "C" fn deploy_upgrade() {
        // This function is intentionally not implemented in `contract_methods`
        // because it only makes sense in the context of the NEAR runtime.
        let mut io = Runtime;
        let state = state::get_state(&io).sdk_unwrap();
        require_running(&state)
            .map_err(ContractError::msg)
            .sdk_unwrap();
        let index = internal_get_upgrade_index();
        if io.block_height() <= index {
            sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
        }
        Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
    }
