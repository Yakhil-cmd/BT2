### Title
`skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures by Trusting Any Mempool Entry for the Sender Address - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (i.e., signature verification) for an invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. That helper returns `true` if **any** transaction exists in the mempool for that address — not specifically a deploy-account transaction owned by the same sender. An unprivileged attacker who observes a victim's pending deploy-account in the mempool can immediately submit an invoke transaction with nonce=1 and an arbitrary/invalid signature from the victim's address. The gateway will skip `__validate__`, admit the transaction, and the batcher will later execute it: the nonce is incremented and a fee is charged against the victim's pre-funded balance before the invalid `__validate__` reverts the call.

### Finding Description

**Broken invariant.** The gateway is supposed to skip `__validate__` only when the account's *own* deploy-account is pending and the account contract does not yet exist on-chain. The actual check is weaker: it only verifies that *some* transaction for the address exists in the mempool or a recent block.

**Root cause — `skip_stateful_validations`:**

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}

**Attack path:**

1. Victim Alice submits `deploy_account` for address `0xABC` (nonce=0). It enters the mempool; `tx_pool.contains_account(0xABC)` is now `true`.
2. Attacker Bob submits `invoke(sender=0xABC, nonce=1, signature=<garbage>)`.
3. Gateway stateful path: `account_nonce = 0`, `tx.nonce() == 1`, `account_tx_in_pool_or_recent_block(0xABC) == true` → `skip_validate = true` → `__validate__` is never called → transaction admitted.
4. Batcher executes Alice's `deploy_account` (success, nonce → 1), then Bob's invoke.
5. Blockifier calls `handle_nonce` (nonce 1 == state nonce 1 → increments to 2), then calls `__validate__` → invalid signature → reverts.
6. Alice's nonce is now 2; a fee is charged against her pre-funded balance; her own nonce=1 invoke is now stale. [6](#0-5) 

### Impact Explanation

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**. An unprivileged attacker can inject an invoke transaction with an arbitrary signature into the mempool for any address that has a pending deploy-account. Beyond admission, the batcher will execute the invalid transaction: the victim's nonce is permanently incremented and a fee is deducted from their pre-funded balance before the revert, constituting a direct economic loss and a denial-of-service on the victim's own nonce=1 transaction.

### Likelihood Explanation

Any pending deploy-account transaction is observable in the public mempool. The attacker needs only to submit a single invoke with nonce=1 and any signature before the victim's own nonce=1 invoke is submitted. No privileged access, no special knowledge, and no on-chain state is required beyond observing the mempool.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction for the exact sender address is pending in the mempool. Alternatively, require that the pending transaction for the address is of type `DeployAccount` before skipping `__validate__`. A targeted fix:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await

// Use a new mempool query:
mempool_client.has_pending_deploy_account(tx.sender_address()).await
```

This ensures the skip is only granted when the account's own deploy-account is genuinely pending, not when any attacker-observable transaction exists for the address.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
