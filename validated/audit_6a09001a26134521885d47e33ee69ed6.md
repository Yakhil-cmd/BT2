### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap check, while the parallel `setPoolAdminFees` path enforces `maxAdminSpreadFeeE6`. A pool admin can therefore set per-bin additional fees up to the full `uint16` range (65 535 E6 ≈ 6.55 %) on every bin, bypassing the factory-owner-controlled spread-fee cap entirely and charging traders more than the protocol's documented maximum.

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` correctly enforces the factory-owner-set cap before updating the base admin spread fee:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, `setPoolBinAdditionalFees` performs no analogous check and passes the caller-supplied values straight through to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` likewise applies no cap — it only validates the bin index: [3](#0-2) 

During every swap the per-bin additional fee is added directly on top of the oracle-derived base fee before the quote is presented to the trader:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern applies to the sell direction: [5](#0-4) 

`uint16` allows values up to 65 535, which in E6 units equals ≈ 6.55 %. The factory hard-max for spread fees is 20 % (`HARD_MAX_SPREAD_FEE_E6 = 200_000`), but the factory owner may configure `maxAdminSpreadFeeE6` to any value below that — including 0. Regardless of what `maxAdminSpreadFeeE6` is set to, the pool admin can always layer up to 6.55 % additional fee per bin through this unchecked path. [6](#0-5) 

### Impact Explanation

Traders executing swaps through any bin whose `addFeeBuyE6` or `addFeeSellE6` has been set to the maximum pay a fee that exceeds the factory-owner-enforced cap. The excess fee accrues inside the pool as LP surplus and is later distributed to LPs and the admin fee destination — meaning the pool admin effectively extracts value from traders beyond the protocol's stated limit. This is a direct loss of user funds (overpayment on every swap touching the affected bin) and constitutes an admin-boundary break: the pool admin exceeds the `maxAdminSpreadFeeE6` cap the factory owner intended to enforce.

### Likelihood Explanation

The pool admin role is semi-trusted and is expected to operate only within factory-owner-set caps. Any pool admin — whether malicious or compromised — can call `setPoolBinAdditionalFees` at any time with no timelock, no cap guard, and no protocol-owner approval. The call is a single transaction with no preconditions beyond holding the pool admin role.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the one in `setPoolAdminFees`. Because bin additional fees are additive on top of the base admin spread fee, the natural bound is `maxAdminSpreadFeeE6`:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap so the factory owner can tune the two limits independently.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 5_000` (0.5 %) via `setFeeCaps`.
2. Pool admin calls `setPoolAdminFees(pool, 5_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; bin 0 now carries ≈ 6.55 % additional fee on both sides.
4. A trader swaps through bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65_535, ONE_X64, 1e6)` — approximately 6.55 % above the oracle mid-price spread, far exceeding the 0.5 % cap the factory owner intended.
5. The excess is retained in the pool as LP surplus and flows to the admin fee destination on the next `collectPoolFees` call, constituting a direct extraction from traders beyond the protocol's stated maximum. [7](#0-6) [2](#0-1)
