### Title
Bridge Circuit Breaker (`MsgTurnBridge`) Is a No-Op — Bridge Cannot Be Disabled After Admin Invocation - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is a stub that unconditionally returns `nil, nil` without performing any state change. As a result, calling `MsgTurnBridge{Enable: false}` succeeds (returns no error) but leaves the bridge fully operational. Any unprivileged user can continue to invoke bridge operations — `MsgTransferTokens`, `MsgConvertVouchers`, and EVM-hook-triggered `__CronosSendToIbc` / `__CronosSendToEvmChain` flows — after an admin has "disabled" the bridge, bypassing the intended circuit-breaker authorization boundary.

### Finding Description

**Root cause — empty handler:**

```go
// x/cronos/keeper/msg_server.go  lines 84-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil   // ← no state written, no permission checked
}
```

The handler neither checks the `CanTurnBridge` permission (defined in `permissions.go`) nor writes any bridge-enabled flag to the KV store. The module `Params` proto (`proto/cronos/cronos.proto`) contains no `bridge_enabled` field, so there is nowhere for the flag to be stored even if the handler tried.

**Missing guard in bridge operations:**

Because no flag is ever written, the downstream bridge paths have nothing to check:

- `IbcTransferCoins` (`x/cronos/keeper/ibc.go`) — no bridge-status guard.
- `ConvertVouchersToEvmCoins` (`x/cronos/keeper/ibc.go`) — no bridge-status guard.
- `SendToIbcHandler.Handle` / `SendToIbcV2Handler.Handle` (`x/cronos/keeper/evmhandlers/send_to_ibc.go`) — no bridge-status guard.
- `SendCroToIbcHandler.Handle` (`x/cronos/keeper/evmhandlers/send_cro_to_ibc.go`) — checks only the `CroBridgeContractAddresses` allowlist, not a bridge-enabled flag.

**Exploit path:**

1. Admin (or any address, since the permission check is also absent) submits `MsgTurnBridge{Enable: false}`. The tx is accepted with code 0.
2. `TurnBridge` returns `nil, nil`; no state is mutated.
3. Unprivileged user calls `MsgTransferTokens` or deploys a contract that emits `__CronosSendToIbc` / `__CronosSendToEvmChain`.
4. `IbcTransferCoins` / EVM log handlers execute normally — the bridge is still live.
5. The integration test in `integration_tests/test_gravity.py` (lines 660–680) explicitly asserts that `send_to_evm_chain` **must fail** after `turn_bridge("false")`; with the empty handler this assertion would be violated.

### Impact Explanation

**Severity: High** — matches "Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."

The `TurnBridge` circuit breaker is the sole mechanism for an authorized operator to halt bridge flows in an emergency (e.g., a discovered vulnerability in the IBC or gravity-bridge path). Because the handler is a no-op, this authorization boundary is permanently bypassed: no matter what the admin submits, unprivileged users retain full, uninterrupted access to all bridge operations. In a live incident this directly enables continued unauthorized asset movement across chains.

### Likelihood Explanation

Exploitation requires no special privilege. Any user who holds tokens and wishes to bridge them can do so after the admin has "disabled" the bridge. The only precondition is that the admin has attempted to invoke the circuit breaker — a scenario that arises precisely when the bridge is under active threat.

### Recommendation

1. Add a `bridge_enabled bool` field to `Params` in `proto/cronos/cronos.proto`.
2. Implement `TurnBridge` to (a) verify `CanTurnBridge` permission via `HasPermission`, and (b) write the flag via `SetParams`.
3. Add a guard at the top of `IbcTransferCoins`, `ConvertVouchersToEvmCoins`, and each EVM log handler's `Handle` method that returns an error when `params.BridgeEnabled == false`.

### Proof of Concept

```
// 1. Admin submits MsgTurnBridge{Enable: false}
//    → handler returns nil, nil; no state written

// 2. Attacker submits MsgTransferTokens{From: attacker, To: "cosmos1...", Coins: [...]}
//    → IbcTransferCoins is called
//    → no bridge-enabled check exists
//    → ibcSendTransfer executes successfully
//    → tokens leave Cronos despite the "disabled" bridge
```

The integration test at `integration_tests/test_gravity.py:670-674` independently confirms the expected (but unimplemented) behaviour: `send_to_evm_chain` is asserted to return `status == 0` (failure) after `turn_bridge("false")`, which the current no-op handler cannot enforce.
