### Title
Token Donation Suppresses L1 Mint Requests, Causing Temporary Reward Freeze - (File: src/reward_supplier/reward_supplier.cairo)

### Summary

The `request_funds` internal function in `RewardSupplier` uses `token_dispatcher.balance_of(account: get_contract_address())` to determine whether to send an L1 mint request. An unprivileged attacker can donate STRK tokens directly to the reward supplier contract address, inflating the observed balance and suppressing L1 mint requests. Once the inflated balance is consumed by legitimate reward claims, the contract has no tokens left and `claim_rewards` calls revert, temporarily freezing unclaimed yield for all stakers and delegators.

### Finding Description

In `src/reward_supplier/reward_supplier.cairo`, the internal `request_funds` function reads the contract's live token balance via `balance_of`:

```cairo
fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())   // <-- manipulable
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    let credit = balance + l1_pending_requested_amount;
    let debit = unclaimed_rewards;

    let base_mint_amount = self.base_mint_amount.read();
    let threshold = compute_threshold(base_mint_amount);
    if credit < debit + threshold {          // suppressed when balance is inflated
        // ... send L1 mint request messages
    }
}

`request_funds` is called from `update_unclaimed_rewards_from_staking_contract`, which is itself called by the staking contract each time rewards are distributed: [2](#0-1) 

The `unclaimed_rewards` state variable correctly tracks what is owed to stakers, but the **credit side** of the solvency check is derived from the live `balance_of` call rather than a tracked state variable. Because ERC-20 tokens can be transferred to any address without calling a contract function, the balance is externally manipulable.

### Impact Explanation

**High — Temporary freezing of unclaimed yield.**

1. An attacker donates a large amount of STRK to the reward supplier address.
2. The inflated `balance` causes `credit >= debit + threshold`, so no L1 mint request is emitted.
3. Legitimate reward claims drain the balance over time (the donated tokens are consumed paying out rewards).
4. When the balance reaches zero, `claim_rewards` attempts a token transfer that reverts because the contract holds no tokens, even though `unclaimed_rewards` is non-zero.
5. All stakers and delegators are unable to claim their accrued rewards until the next `update_unclaimed_rewards_from_staking_contract` call detects the deficit and a new L1 mint request is processed — an L1→L2 round-trip that can take hours to days. [3](#0-2) 

### Likelihood Explanation

**Medium.** The attack requires only a direct ERC-20 transfer to the reward supplier address — no privileged role, no bridge compromise, no governance access. The cost to the attacker is the donated tokens (which are consumed paying legitimate rewards, so the attacker loses them with no recovery). The attack is therefore a pure griefing vector. Given that the reward supplier address is public and the STRK token is freely transferable, any motivated adversary can execute this at any time.

### Recommendation

Replace the live `balance_of` call in `request_funds` with a tracked internal state variable (e.g., `available_balance`) that is incremented only when tokens arrive via the controlled `on_receive` path and decremented when `claim_rewards` transfers tokens out. This mirrors the fix recommended in the original GVault report: use a state variable instead of `balanceOf` to track the contract's asset balance. [4](#0-3) 

### Proof of Concept

1. Deploy the protocol normally. Let the reward supplier accumulate `unclaimed_rewards = U` and `balance = B` where `B < U + threshold` (i.e., an L1 mint request is normally due).
2. Before `update_unclaimed_rewards_from_staking_contract` is called, attacker calls `STRK.transfer(reward_supplier_address, D)` where `D` is large enough that `B + D >= U + threshold`.
3. `update_unclaimed_rewards_from_staking_contract` is called by the staking contract. Inside `request_funds`, `balance = B + D`, `credit = B + D + l1_pending_requested_amount >= debit + threshold`. The `if` branch is skipped — **no L1 mint request is sent**.
4. Stakers and delegators call `claim_rewards` on the staking contract, which calls `reward_supplier.claim_rewards(amount)`. Each call drains the balance. After `B + D` tokens are consumed, the balance is zero.
5. The next `claim_rewards` call attempts `token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into())` with a zero balance — **the transfer reverts**, freezing all unclaimed yield until the next epoch's `update_unclaimed_rewards_from_staking_contract` detects the deficit and a new L1 mint request completes.         fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
