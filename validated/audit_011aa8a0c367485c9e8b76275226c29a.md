### Title
Missing Minimum Threshold Validation in `assert_sign_inputs` Helper Allows Signing with Threshold < 2 — (`File: src/frost/mod.rs`)

### Summary

The `assert_sign_inputs` helper function in `src/frost/mod.rs` is the shared validation gate for all FROST signing entry points (`sign_v1`, `sign_v2` in EdDSA, and `sign` in RedJubjub). It validates participant count, uniqueness, coordinator membership, and an upper-bound on threshold — but it **never checks that `threshold >= 2`**. The analogous DKG helper `assert_key_invariants` in `src/dkg.rs` does enforce this lower bound. The validation logic is split across two separate helpers, and the signing helper is missing the critical minimum-threshold guard. This is the direct analog of the Atlendis pattern: a required invariant is enforced in one factory-style helper but absent from the helper that governs a different protocol phase.

---

### Finding Description

`assert_key_invariants` (`src/dkg.rs`) enforces `threshold >= 2`:

```rust
// Step 1.1
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

`assert_sign_inputs` (`src/frost/mod.rs`) only enforces an upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

There is **no lower-bound check**. A caller may pass `threshold = 0` or `threshold = 1` and `assert_sign_inputs` will return `Ok(participants)`.

All three public FROST signing entry points delegate exclusively to `assert_sign_inputs`:

- `frost::eddsa::sign::sign_v1` — calls `assert_sign_inputs` at line 47
- `frost::eddsa::sign::sign_v2` — calls `assert_sign_inputs` at line 73
- `frost::redjubjub::sign::sign` — calls `assert_sign_inputs` at line 50

After validation passes, the threshold is forwarded to `construct_key_package`, which converts it to `u16` and passes it directly to `frost_ed25519::KeyPackage::new` (or the RedJubjub equivalent) as `min_signers`. With `min_signers = 1`, the FROST library's internal signing-package verification accepts a package containing only a single commitment, and `aggregate` can produce a valid signature from a single signature share.

---

### Impact Explanation

With `threshold = 1` accepted by `assert_sign_inputs`, the `KeyPackage` is constructed with `min_signers = 1`. FROST's `aggregate` function verifies that the signing package contains at least `min_signers` commitments. With `min_signers = 1`, a signing package containing only the coordinator's own commitment satisfies this check. In the `sign_v2` flow, the `PresignOutput` (including `commitments_map`) is supplied as a caller-controlled argument. A malicious coordinator can craft a `PresignOutput` whose `commitments_map` contains only their own commitment, call `sign_v2` with `threshold = 1`, and drive `do_sign_coordinator_v2` to produce a valid FROST signature from a single share — bypassing the threshold requirement entirely.

**Impact class:** Critical — Unauthorized creation of a valid threshold signature for attacker-chosen inputs.

---

### Likelihood Explanation

The entry point is the public library API. Any caller of `frost::eddsa::sign_v1`, `sign_v2`, or `frost::redjubjub::sign` can supply an arbitrary `threshold` value. The `assert_sign_inputs` function is also `pub`, so a developer writing a new signing wrapper who calls it directly will assume it enforces all threshold invariants. The missing check is invisible at the call site. Likelihood is **Medium-High**: the API is directly reachable by any library consumer, and the missing guard is not documented.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` to `assert_sign_inputs`:

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    // ... existing checks ...
    if threshold.value() < 2 {
        return Err(InitializationError::ThresholdTooSmall {
            threshold: threshold.value(),
            min: 2,
        });
    }
    // ... rest of checks ...
}
```

Long-term, consolidate threshold invariants into a single canonical validator (or into the `ReconstructionLowerBound` type constructor itself) so that no protocol phase can be entered with an invalid threshold regardless of which helper is called.
