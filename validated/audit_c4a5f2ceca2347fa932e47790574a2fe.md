### Title
Gateway Admits Unsigned Invoke Transactions for Undeployed Accounts via Overly Broad `skip_stateful_validations` Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (which performs account signature verification) for invoke transactions with `nonce == 1` when `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction type from the account — not exclusively a `deploy_account` transaction. An attacker who observes a victim's pending `deploy_account` transaction in the mempool can submit an invoke transaction for the victim's account with `nonce = 1` and an arbitrary (invalid) signature. The gateway will skip signature verification and admit the transaction to the mempool.

---

### Finding Description

`skip_stateful_validations` is the UX feature that allows a user to submit a `deploy_account` + `invoke` pair atomically. The gateway skips the blockifier `__validate__` call for the invoke when the account is not yet deployed but has a pending deploy:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs L429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                // ...
        }
    }
    Ok(false)
}
```

The comment reads: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

The mempool implementation of `account_tx_in_pool_or_recent_block` is:

```rust
// crates/apollo_mempool/src/mempool.rs L697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` for **any** transaction type from the account — it does not verify that the matching transaction is specifically a `deploy_account`. The assumption in the comment is therefore incorrect: an attacker can cause `account_tx_in_pool_or_recent_block` to return `true` for a victim's address by observing the victim's own `deploy_account` transaction already in the mempool, then submitting a forged invoke.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate = true`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs L311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

This sets `validate = false`. Inside `StatefulValidator::perform_validations`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs L79-81
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
```

The `__validate__` entry point — which is the account contract's signature verification — is **never called**. The transaction is admitted to the mempool with an arbitrary signature.

The nonce pre-check in `validate_nonce` allows `nonce = 1` when `account_nonce = 0` because the default `max_allowed_nonce_gap >= 1`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs L287-295
_ => {
    let max_allowed_nonce =
        Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
    if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
        return Err(create_error(...));
    }
}
```

---

### Impact Explanation

An attacker can submit an invoke transaction for a victim's undeployed account with an invalid/forged signature, and the gateway will accept it into the mempool without any signature verification. This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

Concrete consequences:
1. The attacker's invalid invoke (nonce=1) occupies the nonce=1 slot in the mempool for the victim's account.
2. The victim's own legitimate invoke (nonce=1) may be blocked or require fee escalation to displace the attacker's transaction.
3. When the batcher eventually executes the block, `new_for_sequencing` sets `validate: true` and `strict_nonce_check: true`, so `__validate__` IS called during execution and the attacker's tx is rejected — but only after it has already been sequenced into a block attempt, consuming batcher resources and delaying the victim's transaction.

---

### Likelihood Explanation

The precondition — a victim's `deploy_account` transaction visible in the mempool — is observable by any network participant. The `deploy_account` + `invoke` UX flow is explicitly documented and tested as a supported pattern, making the window predictable and repeatable. No privileged access is required; any unprivileged actor can submit an RPC transaction to the gateway.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the account, not just any transaction. For example, expose a `deploy_account_tx_in_pool` query from the mempool that only returns `true` when the pending transaction for the account at nonce=0 is of type `DeployAccount`. Alternatively, require the caller to supply the expected `deploy_account` transaction hash and verify it is present in the mempool as a `DeployAccount` type before skipping validation.

---

### Proof of Concept

**Setup:**
- Victim generates a new account keypair. Account address = `V`, private key unknown to attacker.
- Victim submits `deploy_account` tx (nonce=0) to the gateway. It passes all checks and enters the mempool.

**Attack:**
1. Attacker observes the mempool and sees a `deploy_account` tx for address `V` with `account_nonce = 0`.
2. Attacker crafts an `InvokeV3` transaction: `sender_address = V`, `nonce = 1`, `calldata = [transfer all ETH to attacker]`, `signature = [0x0, 0x0]` (arbitrary invalid bytes).
3. Attacker submits this tx to the gateway.
4. Gateway calls `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(V)` → `Nonce(0)` ✓
   - `validate_nonce`: `0 <= 1 <= 0 + max_gap` → passes ✓
   - `validate_by_mempool`: nonce gap check passes ✓
   - `skip_stateful_validations`: `nonce == 1 && account_nonce == 0` → calls `account_tx_in_pool_or_recent_block(V)` → `true` (victim's deploy_account is in pool) → returns `true`
   - `run_validate_entry_point(skip_validate=true)`: `validate = false` → `__validate__` is **never called**
5. Gateway returns success. Attacker's invalid invoke tx is now in the mempool at nonce=1 for address `V`.
6. Victim tries to submit their own legitimate invoke tx (nonce=1) — it is rejected as a duplicate nonce or requires fee escalation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
