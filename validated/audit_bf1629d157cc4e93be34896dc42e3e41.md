### Title
Incorrect `amountWithRoyalties` Calculation in WalletConnect Offer Confirmation Underreports Total XCH/CAT Cost When Multiple NFTs Are Present - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before applying each NFT's royalty percentage. When a WalletConnect dApp sends a `take_offer` or `create_offer_for_ids` command containing multiple NFTs, the `amountWithRoyalties` field shown in the confirmation dialog is understated by a factor of N (the number of NFTs). The user approves believing they will spend less than the blockchain will actually deduct in royalties.

### Finding Description

`formatAmountWithRoyalties` is the function responsible for computing the total fungible cost (base amount + royalties) shown in the WalletConnect signing confirmation dialog. [1](#0-0) 

The flawed logic:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Each NFT's royalty is computed on `amount / N` instead of on the full `amount`. The correct formula is:

```
royaltyAmount = Σ (amount × Rᵢ / 10_000)
```

The buggy formula produces:

```
royaltyAmount = Σ ((amount / N) × Rᵢ / 10_000)  =  (1/N) × Σ (amount × Rᵢ / 10_000)
```

So the displayed royalty total is exactly `1/N` of the correct value.

The `royaltyPercentages` array is populated from the NFT driver info embedded in the offer summary: [2](#0-1) 

And `withRoyaltyTotals` feeds this into the confirmation display: [3](#0-2) 

The existing test suite encodes the wrong expected value, confirming the bug is baked in: [4](#0-3) 

With two NFTs (royalty_percentage 500 = 5% and 10 = 0.1%) and 100,000,000 mojos (0.0001 XCH):
- **Displayed**: `0.00010255` XCH (test-asserted value)
- **Correct**: `0.0001051` XCH (`0.0001 + 0.0001×0.05 + 0.0001×0.001`)

The discrepancy grows linearly with N. With 10 NFTs each at 10% royalty, the displayed royalty is 10× too small.

### Impact Explanation

The `amountWithRoyalties` field is the only place in the WalletConnect confirmation dialog where the user sees the true cost of accepting an NFT offer. When it is understated, the user approves a transaction believing they will spend less XCH or CAT than the blockchain will actually deduct. This is a direct case of displaying the wrong amount in a WalletConnect approval context, matching the High-severity criterion: *"WalletConnect state that causes a user to approve… the wrong… amount."*

The actual on-chain royalty deduction is computed by the wallet backend and is always correct; the bug is confined to the pre-approval display. However, the confirmation dialog is the user's only opportunity to review and reject the transaction before signing.

### Likelihood Explanation

Any WalletConnect-connected dApp can construct a `take_offer` payload containing two or more NFTs with royalties. No special privileges are required. The trigger condition (≥2 NFTs with royalties in one offer) is a normal, supported use case in the Chia ecosystem. The bug is deterministic and reproducible on every such offer.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be applied to the full fungible amount:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Apply each NFT's royalty to the full amount, not a split fraction
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

Update the corresponding test assertions in `parseCommandDisplay.test.ts` to reflect the corrected values.

### Proof of Concept

**Scenario**: WalletConnect dApp sends `chia_wallet.take_offer` with an offer containing two NFTs — NFT-A (royalty_percentage = 500, i.e. 5%) and NFT-B (royalty_percentage = 10, i.e. 0.1%) — requesting 100,000,000 mojos (0.0001 XCH) from the user.

**Buggy path** (`formatAmountWithRoyalties`):
```
splitAmount = 100_000_000 / 2 = 50_000_000
royaltyAmount = (50_000_000 × 500) / 10_000 + (50_000_000 × 10) / 10_000
             = 2_500_000 + 50_000 = 2_550_000
totalAmount  = 100_000_000 + 2_550_000 = 102_550_000  →  "0.00010255 XCH"
```

**Correct calculation**:
```
royaltyAmount = (100_000_000 × 500) / 10_000 + (100_000_000 × 10) / 10_000
             = 5_000_000 + 100_000 = 5_100_000
totalAmount  = 100_000_000 + 5_100_000 = 105_100_000  →  "0.0001051 XCH"
```

The confirmation dialog shows **0.00010255 XCH** but the blockchain deducts **0.0001051 XCH** — a 2.4× understatement of the royalty cost. The existing test at line 310–335 of `parseCommandDisplay.test.ts` asserts the wrong value, confirming the bug is present and untested-against. [5](#0-4)

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
