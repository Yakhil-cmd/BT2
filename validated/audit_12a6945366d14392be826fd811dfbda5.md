### Title
Minimum Threshold Check Bypassed in FROST Signing — (`src/frost/mod.rs`)

---

### Summary

The minimum threshold invariant (`threshold >= 2`) is enforced during key generation via `assert_key_invariants` in `src/dkg.rs`, but this same lower-bound check is **absent** from `assert_sign_inputs` in `src/frost/mod.rs`. A malicious coordinator or any unprivileged library caller can invoke FROST signing with `threshold = 1`, causing all honest participants to compute signature shares using Lagrange coefficients for a threshold-1 scheme. Because the key was generated under threshold ≥ 2, the resulting combined signature is cryptographically invalid and unusable — corrupting the signing output for all honest parties.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces three threshold constraints before DKG proceeds:

```rust
// src/dkg.rs lines 572-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The minimum-threshold guard (`threshold < 2`) is the critical invariant: it prevents a key from being generated under a threshold that would allow a single party to reconstruct the secret.

`assert_sign_inputs` in `src/frost/mod.rs`, which validates inputs for every FROST signing invocation, enforces only the upper-bound check and omits the lower-bound entirely:

```rust
// src/frost/mod.rs lines 127-150
if participants.len() < 2 {
    return Err(InitializationError::NotEnoughParticipants { ... });
}
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← NO check: threshold < 2
``` [2](#0-1) 

Because the signing threshold is supplied independently by the caller at signing time — it is not cryptographically bound to the threshold used during key generation — a caller can freely pass `threshold = 1` to `assert_sign_inputs` and have it accepted.

For contrast, the OT-based triple generation (`validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs`) **does** enforce `threshold < 2`, and the OT-based presign further requires `args.threshold == args.triple0.1.threshold`, so the OT path is indirectly protected. The FROST path has no equivalent binding. [3](#0-2) 

---

### Impact Explanation

In FROST, the threshold parameter passed to signing directly controls the Lagrange interpolation coefficients each participant uses when computing their partial signature share. If `threshold = 1` is supplied:

- Each participant computes a Lagrange coefficient as if only one share is needed to reconstruct the secret (coefficient = 1 for the sole participant, 0 for all others in a 1-of-N scheme).
- The key was generated under threshold ≥ 2, so the actual secret is encoded across at least two shares with degree-1 polynomial coefficients.
- The combined signature produced by summing these incorrectly-weighted partial shares does not correspond to the actual signing key and will fail standard ECDSA/EdDSA verification.

All honest participants complete the protocol without error (no abort), yet the output signature is cryptographically unusable. This matches the allowed impact: **"Corruption of sign outputs so honest parties accept unusable cryptographic outputs."**

Severity: **High**

---

### Likelihood Explanation

The `assert_sign_inputs` function is a public API entry point. Any library caller — including a malicious coordinator who orchestrates which participants sign and with what parameters — can supply `threshold = 1` directly. No special privilege, leaked key material, or cryptographic break is required. The coordinator role is the natural, documented entry point for initiating signing sessions, making this trivially reachable.

---

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` to `assert_sign_inputs`:

```rust
// src/frost/mod.rs — inside assert_sign_inputs
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
``` [4](#0-3) 

Additionally, consider binding the signing threshold to the threshold recorded in the `KeygenOutput` so that callers cannot supply an arbitrary value that diverges from the key-generation parameters.

---

### Proof of Concept

1. Run DKG with `participants = [A, B, C]`, `threshold = 2`. This succeeds because `assert_key_invariants` enforces `threshold >= 2`. All three parties receive valid key shares for a 2-of-3 scheme.

2. Invoke FROST signing with the same participants but pass `threshold = 1`. `assert_sign_inputs` accepts this because the `threshold < 2` guard is absent.

3. Each participant computes their partial signature share using Lagrange coefficients for a 1-of-3 scheme (each coefficient = 1 for themselves, 0 for others).

4. The coordinator aggregates the partial shares. The resulting signature does not verify against the public key produced in step 1, because the Lagrange interpolation was performed at the wrong degree.

5. All honest parties observe a successful protocol run with no errors, but the output signature is invalid — a silent corruption of the signing output. [5](#0-4) [2](#0-1)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
