### Title
`_afterTimelock` uint32 Truncation Silently Wraps `executeAfter` to a Past Timestamp, Bypassing All OracleValueStopLossExtension Timelocks — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
`_afterTimelock` casts `block.timestamp + timelock` to `uint32`. When `timelock` is large enough that the sum exceeds `type(uint32).max`, the result wraps to a value smaller than `block.timestamp`. Every subsequent `_requireElapsed` check passes immediately, so drawdown, decay, watermark, and timelock-change proposals all execute with zero delay regardless of the configured timelock.

### Finding Description
`_afterTimelock` computes the unlock timestamp as:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space and then truncated to `uint32`. If `block.timestamp + timelock > type(uint32).max` (≈ 4,294,967,295), the truncation wraps the result to a value that is less than `block.timestamp`.

At the current epoch (~1,753,000,000 s), any `timelock > 2,541,967,295 s` (~80.6 years) triggers the wrap. `type(uint32).max` itself (≈ 136 years) is a valid `uint32` value and produces `executeAfter ≈ block.timestamp − 1`.

The guard that enforces the delay is:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

When `executeAfter` has wrapped to a past value, `block.timestamp < executeAfter` is `false`, so the revert never fires. Every proposal — drawdown, decay, watermarks, and the timelock itself — can be executed in the same block it was proposed.

`initialize` decodes and stores the timelock without any bounds check:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// no _validateTimelock
``` [3](#0-2) 

`proposeOracleStopLossTimelock` similarly stores a new timelock without validation, so the overflow can be introduced after deployment as well. [4](#0-3) 

### Impact Explanation
The `OracleValueStopLossExtension` timelock is the sole mechanism that gives LPs advance notice before the pool admin can change drawdown thresholds, decay rates, or high-watermark baselines. With the timelock bypassed:

- The admin can set `drawdownE6 = 0` in one transaction, which causes `if (drawdown == 0) return;` to skip all stop-loss checks, silently disabling LP protection. [5](#0-4) 
- The admin can immediately raise `decayPerSecondE8` to `E8`, collapsing all watermarks to zero on the next swap, making the stop-loss permanently inert.
- LPs who entered the pool trusting a displayed large timelock (e.g., `type(uint32).max`) have no actual protection window.

This is a broken core-functionality / admin-boundary-break impact: the timelock invariant that bounds semi-trusted admin power is silently voided.

### Likelihood Explanation
The overflow requires `timelock > ~80.6 years` at current block times. A pool creator who intends to make the timelock "effectively permanent" by setting `type(uint32).max` (a natural sentinel value) triggers the bug exactly. No special privilege beyond pool-admin role at creation is needed. The factory passes `extensionInitData` through without inspecting the timelock field, so the factory provides no backstop. [6](#0-5) 

### Recommendation
1. **Validate the timelock** in `initialize` and `proposeOracleStopLossTimelock`. Enforce an upper bound (e.g., `≤ 365 days`) that keeps `block.timestamp + timelock` safely within `uint32` range for the foreseeable future.
2. **Use safe arithmetic** in `_afterTimelock`: perform the addition in `uint256`, check that the result fits in `uint32` before casting, and revert otherwise.
3. Mirror the pattern used in the factory's mutable price-provider timelock, which stores `executeAfter` as `uint256` and never truncates: [7](#0-6)

```solidity

pragma solidity ^0.8.35;

// Scenario: pool admin sets timelock = type(uint32).max at initialization.
// _afterTimelock wraps executeAfter to block.timestamp - 1.
// All proposals execute immediately — timelock is fully bypassed.

uint32 timelock = type(uint32).max; // 4_294_967_295 s ≈ 136 years

// _afterTimelock:
uint256 sum = block.timestamp + uint256(timelock);
// sum ≈ 1_753_000_000 + 4_294_967_295 = 6_047_967_295
uint32 executeAfter = uint32(sum);
// uint32(6_047_967_295) = 6_047_967_295 - 4_294_967_296 = 1_752_999_999
// ≈ block.timestamp - 1  ← in the past

// _requireElapsed:
// block.timestamp < executeAfter  →  1_753_000_000 < 1_752_999_999  →  false
// → no revert → proposal executes immediately

// Admin then calls executeOracleStopLossDrawdown in the same block:
// drawdownE6 is set to 0 → stop-loss disabled for all LPs.
```
