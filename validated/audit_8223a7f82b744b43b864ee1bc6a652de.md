### Title
Cross-Chain Signature Replay via Accepted Unprotected Legacy Transactions (Pre-EIP-155) - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

---

### Summary

ZKsync OS's bootloader explicitly accepts pre-EIP-155 "unprotected" legacy transactions (those with `v = 27` or `v = 28`). For these transactions, the signing hash is computed over only the six transaction fields (`nonce, gasPrice, gasLimit, to, value, data`) with no chain ID included. This means any pre-EIP-155 transaction signed on Ethereum mainnet or any other EVM chain is cryptographically valid on ZKsync OS as well, enabling a direct cross-chain signature replay attack. An unprivileged attacker can take a publicly visible pre-EIP-155 transaction from another chain and submit it to ZKsync OS to drain the victim's funds if the nonce matches.

---

### Finding Description

**Root Cause**

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, when the parsed `v` value equals `27` or `28` (i.e., `is_eip155()` returns `false`), the code takes the "Unprotected legacy" branch and computes the signing hash over only the 6-field RLP payload — with no chain ID:

    hasher.update(inner_slice);

**Exploit Flow**

1. Attacker observes a pre-EIP-155 transaction on Ethereum mainnet (or any other EVM chain) with `v = 27` or `v = 28`. Such transactions are publicly visible on-chain.
2. The victim's address on ZKsync OS is the same as on Ethereum (deterministic address derivation from the same private key).
3. The attacker submits the identical raw transaction bytes to ZKsync OS.
4. The bootloader computes the signing hash over the same 6 fields (no chain ID), recovers the same signer address, and the signature check passes.
5. If the victim's nonce on ZKsync OS matches the nonce in the replayed transaction (e.g., nonce = 0 for a fresh account, or the attacker waits/manipulates to align nonces), the transaction executes — transferring funds or calling contracts as the victim.

---

### Impact Explanation

An attacker can replay any pre-EIP-155 transaction from Ethereum mainnet (or any other chain) on ZKsync OS without possessing the victim's private key. The concrete impact is unauthorized ETH/token transfers and arbitrary contract calls executed as the victim. For fresh accounts (nonce = 0), exploitation is immediate with zero preconditions beyond observing a historical pre-EIP-155 transaction from the victim's address on any other chain. This constitutes direct loss of user funds.

---

### Likelihood Explanation

Pre-EIP-155 transactions (v = 27/28) were common before EIP-155 was widely adopted (pre-2016 Ethereum) and are still occasionally produced by older wallets, hardware signers, and certain DeFi protocols. Any account that ever sent a pre-EIP-155 transaction on any EVM chain and later bridges assets to ZKsync OS is at risk if their nonce on ZKsync OS matches the replayed transaction's nonce. The attacker requires no privileged access — only the ability to submit a transaction to ZKsync OS, which is fully unprivileged.

---

### Recommendation

Reject unprotected legacy transactions (v = 27 or v = 28) entirely at the bootloader level. ZKsync OS is a new chain with no backward-compatibility obligation to pre-EIP-155 transactions. All accepted legacy transactions should be required to use EIP-155 chain-ID binding (the `LegacyWithEIP155` path). The `RlpEncodedTxInner::Legacy` variant and its associated unprotected hash path should be removed or gated behind an explicit opt-in that is disabled by default.

---

### Proof of Concept

1. On Ethereum mainnet, find any historical pre-EIP-155 transaction from address `A` with nonce `N` (e.g., from Etherscan, filter by `v = 0x1b` or `v = 0x1c`).
2. Ensure address `A` has funds on ZKsync OS and its current nonce is `N`.
3. Submit the raw transaction bytes unchanged to ZKsync OS.
4. The bootloader parses it as `RlpEncodedTxInner::Legacy`, computes `sig_hash = keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the Ethereum mainnet signing hash — and `ecrecover` returns address `A`.
5. Nonce check passes (`old_nonce == N`), and the transaction executes on ZKsync OS as if signed by `A`, transferring `value` ETH to `to`.

The critical code path is:
- `LegacyPayloadParser::try_parse_and_hash_for_signature_verification` (lines 89–94) — chain-ID-free hash
- `RlpEncodedTxInner::parse_and_compute_signed_hash` (lines 127–131) — accepts `Legacy` variant
- `validation_impl.rs` (lines 195–245) — ecrecover against the chain-ID-free hash, no additional chain-ID check for `Legacy` variant [1](#0-0) [5](#0-4) [6](#0-5)
