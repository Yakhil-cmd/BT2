### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Any Account with a Pending Deploy-Account - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path skips the `__validate__` entry-point call (the only place where the account's signature is verified) for any invoke transaction whose nonce is `1` and whose sender address has *any* transaction present in the mempool. Because the mempool presence check does not verify that the caller of the current invoke is the account owner, an attacker can inject an invoke with an arbitrary or empty signature for any account that has a pending deploy-account transaction, and the gateway will accept it without ever verifying the signature.

### Finding Description

**Invariant broken:** Every transaction admitted by the gateway must either pass the account's `__validate__` entry point or be provably from the account owner. The `skip_stateful_validations` path breaks this invariant.

### Impact Explanation

**Attack scenario:**

1. Victim pre-funds address `A` and submits a valid `deploy_account` transaction (nonce 0). The gateway accepts it; it enters the mempool.
2. Attacker observes the mempool and submits an `invoke` transaction for address `A` with nonce=1 and an **empty or forged signature**.
3. Gateway stateful validation: `account_nonce == 0`, `tx.nonce() == 1`, `account_tx_in_pool_or_recent_block(A) == true` → `skip_validate = true` → `__validate__` is **not called** → the attacker's unsigned invoke is **admitted to the mempool**.
4. During block execution: the deploy-account executes first (nonce 0→1), then the attacker's invoke executes (nonce 1). The blockifier calls `__validate__` with the invalid signature → the transaction fails validation. The fee is charged from the victim's account balance.

**Concrete corrupted values / broken invariants:**

- The gateway emits an `InternalRpcTransaction` for an invoke whose signature was never verified. This is an invalid admission decision.
- The victim's account balance is debited for a transaction they never signed.
- The attacker occupies the victim's nonce-1 slot in the mempool, potentially delaying or displacing the victim's own legitimate nonce-1 invoke.

**Impact category:** High — Mempool/gateway admission accepts invalid (unsigned) transactions before sequencing.

### Likelihood Explanation

- Triggering the condition requires only that the target account has a pending deploy-account in the mempool, which is a publicly observable state.
- No privileged access is required; any unprivileged actor can submit an RPC transaction.
- The attack is cheap: the attacker pays nothing (the fee is charged to the victim's account on execution failure).
- The window of vulnerability is the entire time the deploy-account transaction is pending in the mempool (potentially many blocks).

### Recommendation

Replace the bare existence check with a check that specifically confirms a **deploy-account** transaction is pending for the sender address, and additionally verify that the current invoke transaction carries a valid signature before skipping `__validate__`. The minimal fix is:

```diff
-   return mempool_client
-       .account_tx_in_pool_or_recent_block(tx.sender_address())
-       .await
-       ...;
+   let has_deploy_account = mempool_client
+       .deploy_account_in_pool_or_recent_block(tx.sender_address())
+       .await
+       ...?;
+   if has_deploy_account {
+       // Still run __validate__ to verify the signature; only skip the
+       // "account does not exist yet" pre-condition failure.
+       return Ok(false);  // do NOT skip __validate__
+   }
```

Alternatively, keep the UX skip but enforce that the signature is verified through a lightweight off-chain ECDSA check before admitting the transaction, so that only the account owner can benefit from the skip.

### Proof of Concept

**Step-by-step (no code execution required; all paths are statically traceable):**

1. Submit a valid `RpcDeployAccountTransactionV3` for address `A` to the gateway. It passes all checks and enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

2. Submit an `RpcInvokeTransactionV3` for address `A` with:
   - `nonce = 1`
   - `signature = []` (empty)
   - valid resource bounds and tip (so fee checks pass; the account has balance from step 1 pre-funding)

3. Gateway stateless validation (`StatelessTransactionValidator::validate`): passes — empty signature is within `max_signature_length`, nonce DA mode is L1, resource bounds are non-zero.

4. Gateway stateful validation (`extract_state_nonce_and_run_validations`):
   - `get_nonce_from_state(A)` → `Nonce(0)` (account not yet deployed)
   - `validate_state_preconditions`: nonce check passes (`0 ≤ 1 ≤ max_allowed_nonce_gap`), resource bounds check passes
   - `validate_by_mempool`: passes (no duplicate tx_hash, nonce ordering OK)
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`
   - `run_validate_entry_point` called with `validate: false` → `__validate__` **never called**

5. The unsigned invoke is stored as an `InternalRpcTransaction` and forwarded to the mempool. The gateway returns success to the attacker.

6. During block execution: deploy-account runs (nonce 0→1), then the attacker's invoke runs. Blockifier calls `__validate__` with the empty signature → fails. Fee is charged from `A`'s balance. Victim's nonce-1 slot was consumed by the attacker's transaction.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
