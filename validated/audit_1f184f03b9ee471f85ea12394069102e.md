## Full Call Chain Trace

**Step 1 — Attacker-controlled `ph` enters `prepareNotifications`**

In `useBlockchainNotifications`, the on-chain notification message is decoded and the `ph` field is extracted with zero validation: [1](#0-0) 

Any hex string the attacker places in `ph` is stored verbatim as `puzzleHash` in the notification object.

**Step 2 — `puzzleHash` flows into `OfferIncomingTable.handleCounterOffer`** [2](#0-1) 

`toBech32m` accepts any valid hex string — it only throws on non-hex characters: [3](#0-2) 

So `toBech32m(attacker_puzzlehash, 'xch')` succeeds and produces a valid `xch1…` address. This address is then passed as `address` in the navigation state to `/dashboard/offers/builder`.

**Step 3 — `CreateOfferBuilder` receives and propagates the address** [4](#0-3) 

After the counter-offer file is created, `onOfferCreated({ offerRecord, offerData, address })` is called, passing the attacker's address forward.

**Step 4 — `handleOfferCreated` opens `OfferShareDialog` with attacker's address** [5](#0-4) 

The attacker's address is passed to `OfferShareDialog`, which pre-populates `NotificationSendDialog` with it.

**Step 5 — `NotificationSendDialog` sends 0.0001 XCH to attacker's address**

The address field in `NotificationSendDialog` is rendered `disabled` — the user cannot change it: [6](#0-5) 

If the user clicks "Send Message," `sendNotification` is called with `target = fromBech32m(attacker_address)`, sending 0.0001 XCH to the attacker.

## Impact Correction — The Question's Framing Is Partially Wrong

The question claims "the counter-offer **payment** goes to the attacker." This is **not accurate**.

The `address` field does **not** affect the Chia offer file itself. The offer file is created by `createOfferForIds` with no reference to `address`: [7](#0-6) 

The actual XCH/CAT/NFT trade in the counter-offer is unaffected. The `address` is only used as the **notification destination** after offer creation.

The real impact is:
- The victim's counter-offer notification (0.0001 XCH) is sent to the attacker's address instead of the legitimate offer sender's address.
- The attacker receives the counter-offer URL embedded in the notification, allowing them to potentially accept the counter-offer before the intended recipient.
- The `NotificationSendDialog` address field is `disabled` — the user cannot correct it.

---

## Verdict

The attack path is **real and concrete**:

1. Attacker broadcasts a notification with `ph=<attacker_puzzlehash>` on-chain.
2. No validation occurs at any point in `prepareNotifications` → `handleCounterOffer` → `toBech32m` → navigation state → `handleOfferCreated` → `NotificationSendDialog`.
3. The victim's notification send (0.0001 XCH) is misdirected to the attacker, and the attacker receives the counter-offer URL.

However, the specific claim that "the counter-offer **payment** goes to the attacker" is **false** — only the notification fee and URL are misdirected. The offer file itself is unaffected.

Under the scope rules, the impact qualifies as **High** ("unsafe trust of notification state that causes a user to send to the wrong destination"), but the question's stated root impact (counter-offer payment theft) is overstated. The actual financial loss is bounded by the notification fee (0.0001 XCH), and the user must explicitly click "Send Message" — they can close the dialog without sending.

An on-chain attacker can craft a notification with an arbitrary `ph` (puzzleHash) field. The GUI parses this field without validation and uses it verbatim as the destination address for the victim's counter-offer notification send dialog. The address field is rendered disabled, so the victim cannot correct it. export default function toBech32m(value: string, prefix: string): string {
```typescript
  if (value.startsWith(prefix)) {
    return value;
  }

  const pureHash = removePrefix(value, '0x');
  const hexBytes = Buffer.from(pureHash, 'hex');

  if (pureHash.length / 2 !== hexBytes.length) {
    throw new Error('Invalid hex string');
  }

  const words = bech32m.toWords(hexBytes);

  return bech32m.encode(prefix, words);
}
```
