### Title
`skip_stateful_validations` Bypasses Signature Validation for Deployed Accounts via Overly Broad Mempool Presence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip the blockifier `__validate__` entry-point call for an invoke transaction with nonce=1 when the account's deploy_account has not yet been processed. However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type from that address, not only a `deploy_account`. An attacker who already has a deployed account (nonce=0 in state) can first submit a valid invoke with nonce=0 to seed the mempool, then submit a second invoke with nonce=1 carrying a forged signature. The gateway skips `__validate__` for the second transaction and admits it to the mempool without signature verification.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. The transaction nonce equals `Nonce(Felt::ONE)`.
3. The on-chain account nonce equals `Nonce(Felt::ZERO)`.

When all three hold, it calls `account_tx_in_pool_or_recent_block(sender_address)` and, if that returns `true`, returns `true` (skip validation): [1](#0-0) 

The inline comment claims this is safe because an account with nonce=0 in state can only have a `deploy_account` in the mempool. That claim is false. Any deployed account whose nonce is still 0 in the committed state can have a regular invoke with nonce=0 in the mempool (it passed blockifier `__validate__` with a valid signature). Once that invoke is present, `account_tx_in_pool_or_recent_block` returns `true`: [2](#0-1) 

The result is that `run_validate_entry_point` is called with `validate: !skip_validate` = `false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the `__validate__` call is skipped entirely and the function returns `Ok(())`: [4](#0-3) 

The production `max_allowed_nonce_gap` is 200, so nonce=1 passes the gateway nonce check for an account at nonce=0: [5](#0-4) [6](#0-5) 

The analog to the external report is exact:

| External report | Sequencer |
|---|---|
| `request_self_service_share_transfer` does not call `only_when_self_service_enabled` | `skip_stateful_validations` does not verify the mempool entry is specifically a `deploy_account` |
| Passes `false` for `self_service` when it should be `true` | Sets `validate: false` (skips `__validate__`) when the precondition for doing so is not actually met |

---

### Impact Explanation

An invoke transaction carrying an arbitrary/forged signature is admitted to the mempool without the account's `__validate__` entry point being called at the gateway. The transaction hash is computed from the transaction fields (not the signature), so the hash is valid and the transaction is forwarded to the batcher. When the batcher executes it with `validate=true` (the default `ExecutionFlags`), `__validate__` runs and fails, causing a revert. The fee is still charged (or the transaction is dropped if the account cannot pay), but the sequencer has accepted and processed an unauthorized transaction. This matches the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The precondition is a deployed account whose on-chain nonce is still 0. This is the normal state of any freshly deployed account before its first transaction is committed. The attacker needs only two sequential RPC calls: one valid invoke (nonce=0) to seed the mempool, then one forged invoke (nonce=1). No privileged access, no special network position, and no race condition is required.

---

### Recommendation

**Short term:** In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions. The mempool should expose a method such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type stored in the pool. Alternatively, add a guard that the account has no deployed class hash in state (i.e., `get_class_hash_at(address) == ClassHash::default()`), which is the true invariant the comment intended to rely on.

**Long term:** Add a negative unit test that seeds the mempool with a nonce=0 invoke for a deployed account and then asserts that a nonce=1 invoke with an invalid signature is **rejected** by `run_pre_validation_checks`, not admitted.

  "type": "INVOKE",
  "sender_address": "0xX",
  "nonce": "0x1",
  "signature": ["0xdeadbeef", "0xdeadbeef"],   // forged
  ...
}
// Gateway path:
//   validate_nonce: 0 <= 1 <= 200  → OK
//   validate_by_mempool             → OK (no duplicate, nonce ordering OK)

//     → returns true (skip __validate__)
//   run_validate_entry_point(skip_validate=true):
//     execution_flags.validate = false
//     StatefulValidator::perform_validations → returns Ok(()) without calling __validate__
// → forged invoke admitted to mempool.

// Step 3 — batcher executes the forged invoke with validate=true.
//   __validate__ runs, fails (bad signature) → transaction reverts.
//   Fee is charged (or tx dropped if balance insufficient).

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
