### Title
Malicious Participant Can Silently Corrupt CKD Output Due to Missing Contribution Verification — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator aggregates participant contributions without any cryptographic verification of their correctness. A single malicious participant can send arbitrary `(norm_big_y, norm_big_c)` group elements, silently corrupting the derived key output returned to the requester. Unlike every other signing protocol in this codebase (OT-based ECDSA, Robust ECDSA, EdDSA), the CKD protocol has no final integrity check, making the corruption undetectable by the coordinator.

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 44–57), the coordinator receives a `CKDOutput` from every other participant and blindly accumulates it:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each honest participant is supposed to send:
- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

computed inside `compute_signature_share`: [2](#0-1) 

There is **no zero-knowledge proof** attached to these values, and the coordinator performs **no consistency check** — not against the participant's public key share, not against the group public key, and not against the final aggregated output. Any participant can substitute arbitrary curve points.

**Contrast with every other protocol in the codebase.** The OT-based ECDSA coordinator verifies the assembled signature before returning it: [3](#0-2) 

The Robust ECDSA coordinator does the same: [4](#0-3) 

The EdDSA specification (step 2.7) likewise mandates that the coordinator asserts `(R, s)` is a valid EdDSA signature before outputting it. [5](#0-4) 

The CKD protocol has no equivalent gate. The coordinator simply wraps whatever it accumulated and returns `Some(ckd_output)`.

**Attack path:**
1. Attacker is a legitimate participant holding a valid key share (no privilege required beyond protocol membership).
2. During the CKD round, instead of computing the correct `(norm_big_y, norm_big_c)`, the attacker sends an arbitrary pair — e.g., `(G, G)`, `(0, 0)`, or a carefully chosen offset `(0, Δ·G)`.
3. The coordinator sums all contributions including the attacker's malformed one and returns the corrupted `CKDOutput`.
4. The requester unmasks with `app_sk` and obtains `msk · H(pk ‖ app_id) + Δ` — a deterministically wrong derived key — with no indication that anything went wrong.

### Impact Explanation

The output of the CKD protocol is a confidential derived key `msk · H(pk ‖ app_id)`. A malicious participant can shift this value by any group element of their choice. The requester receives a silently wrong key. Because the coordinator does not know `app_sk`, it cannot verify the output; because the requester does not know `msk`, it cannot independently verify the result either. The corruption is end-to-end undetectable within the protocol.

This matches the **High** allowed impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation

Any single participant in a CKD session — with no special privilege beyond holding a key share — can trigger this. The attack requires only sending two arbitrary group elements instead of the correct ones. It is trivially reachable by a malicious participant or a compromised node.

### Recommendation

Add a verifiable contribution mechanism. Each participant should accompany `(norm_big_y, norm_big_c)` with a Maurer/Schnorr proof demonstrating that:
- `norm_big_y` is a scalar multiple of `G` for a known exponent, and
- `norm_big_c` is correctly formed relative to the participant's public key share and `app_pk`.

The coordinator should verify all proofs before aggregating. This is consistent with how the DKG protocol verifies proofs of knowledge and commitment hashes before accepting shares: [6](#0-5) 

### Proof of Concept

```
Setup: 3 participants P1 (honest), P2 (honest), P3 (malicious), coordinator = P1.
       Threshold = 2. app_id fixed. app_pk = app_sk · G known to requester only.

Round 1 (honest):
  P1 sends (λ1·y1·G,  λ1·(x1·H + y1·app_pk)) to coordinator.
  P2 sends (λ2·y2·G,  λ2·(x2·H + y2·app_pk)) to coordinator.

Round 1 (malicious):
  P3 sends (0, Δ·G) for any attacker-chosen scalar Δ ≠ 0.

Coordinator aggregates:
  Y_agg = λ1·y1·G + λ2·y2·G + 0
  C_agg = λ1·(x1·H + y1·app_pk) + λ2·(x2·H + y2·app_pk) + Δ·G

Requester unmasks:
  C_agg - app_sk·Y_agg = msk·H(pk‖app_id) + Δ·G   ← wrong key, no error raised.
```

The coordinator returns `Some(CKDOutput { big_y: Y_agg, big_c: C_agg })` with no error, and the requester silently derives the wrong key.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
