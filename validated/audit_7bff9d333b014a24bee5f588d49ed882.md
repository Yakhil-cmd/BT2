### Title
Missing `max_commission` parameter in `switch_delegation_pool` allows staker to front-run and impose higher commission on delegators - (File: src/pool/pool.cairo)

---

### Summary
`switch_delegation_pool` and `enter_delegation_pool` in `pool.cairo` accept no `max_commission` guard. A staker who has set an active `commission_commitment` can front-run a delegator's switch/entry transaction by calling `set_commission` to raise their commission up to `max_commission`, causing the delegator to be locked into a pool with a higher commission rate than they expected and permanently receiving less yield.

---

### Finding Description

`switch_delegation_pool` lets a pool member move their stake from one staker's pool to another:

```cairo
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
) -> Amount {
``` [1](#0-0) 

There is no `max_commission` parameter. The delegator specifies the destination pool but has no on-chain protection against the commission changing between when they observe it off-chain and when their transaction is included.

`set_commission` writes the new commission directly to storage with no time-lock:

```cairo
staker_pool_info.commission.write(Option::Some(commission));
``` [2](#0-1) 

The `update_commission` logic allows a staker who holds an active `commission_commitment` to **increase** commission to any value up to `max_commission`:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
``` [3](#0-2) 

Without a commitment, commission can only decrease. But with an active commitment, the staker can freely raise commission up to `max_commission` in a single transaction, with immediate effect.

The same missing-parameter issue exists in `enter_delegation_pool`:

```cairo
fn enter_delegation_pool(
    ref self: ContractState, reward_address: ContractAddress, amount: Amount,
) {
``` [4](#0-3) 

---

### Impact Explanation

A delegator who switches to (or enters) a pool expecting 5% commission can end up locked into a pool charging 50% commission if the staker front-runs with a `set_commission(50%)` call (assuming `max_commission = 50%` in the commitment). All future rewards earned by that delegator are permanently reduced by the commission delta. This constitutes **theft of unclaimed yield** (High impact).

---

### Likelihood Explanation

The precondition is that the target staker has an active `commission_commitment` with a `max_commission` meaningfully above their current commission. This is a legitimate protocol feature (`set_commission_commitment` is a public staker function), so adversarial stakers can set it up deliberately. On Starknet, a staker can monitor the mempool and submit a `set_commission` transaction before the delegator's `switch_delegation_pool` is sequenced. No privileged access, leaked key, or external dependency is required — only a staker acting in their own financial interest.

---

### Recommendation

Add a `max_commission: Commission` parameter to `switch_delegation_pool` (and `enter_delegation_pool` / `add_to_delegation_pool`) and assert inside the function that the destination pool's current commission does not exceed it:

```cairo
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
    max_commission: Commission,   // <-- add this
) -> Amount {
    // ... existing checks ...
    let current_commission = self.get_commission_from_staking_contract();
    assert!(current_commission <= max_commission, "commission exceeds max");
    // ...
}
```

---

### Proof of Concept

1. Staker calls `set_commission_commitment(max_commission: 5000, expiration_epoch: E+100)` and currently has `commission = 100` (1%).
2. Delegator observes 1% commission and submits `switch_delegation_pool(to_staker, to_pool, amount)`.
3. Staker observes the pending transaction and immediately submits `set_commission(5000)` (50%), which is accepted because `5000 <= max_commission` and `5000 != 100`.
4. Staker's `set_commission` is sequenced first; commission is now 50%.
5. Delegator's `switch_delegation_pool` is sequenced next. `enter_delegation_pool_from_staking_contract` is called on the destination pool with no commission check. [5](#0-4) 
6. Delegator is now a member of the pool at 50% commission. Every future `claim_rewards` call transfers 50% of their earned rewards to the staker instead of the expected 1%, constituting ongoing theft of unclaimed yield. [6](#0-5)
