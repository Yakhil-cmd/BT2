### Title
Gateway Resource-Bounds Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Wrong Admission Decisions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` checks a transaction's `max_price_per_unit` against the **current block's** L2 gas price (`block_header.l2_gas_price.price_in_fri`). The correct reference is `block_header.next_l2_gas_price`, the EIP-1559-computed price that will govern the **next** block in which the transaction will actually execute. The mismatch causes the gateway to reject valid transactions and admit invalid ones whenever the EIP-1559 mechanism moves the price between blocks.

---

### Finding Description

`validate_resource_bounds` reads the reference price from `get_block_info()`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

`get_block_info()` is implemented in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, which converts `BlockHeaderWithoutHash` to `BlockInfo`. It populates `strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price.price_in_fri` — the **current block's** price:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

`BlockHeaderWithoutHash` carries a separate field `next_l2_gas_price` — the EIP-1559-adjusted price for the upcoming block:

```rust
pub struct BlockHeaderWithoutHash {
    pub l2_gas_price: GasPricePerToken,
    pub next_l2_gas_price: GasPrice,
    ...
}
``` [3](#0-2) 

`next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` using the EIP-1559 formula and stored in the block header:

```rust
next_l2_gas_price: self.l2_gas_price,
``` [4](#0-3) 

`get_block_info_from_sync_client` never reads `block_header.next_l2_gas_price` and `BlockInfo` has no field for it, so the value is silently dropped. The gateway therefore validates every incoming transaction against the **previous block's** price, not the price that will be enforced when the transaction executes.

The developer-acknowledged TODO comment confirms this is a known incorrect value:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
``` [5](#0-4) 

---

### Impact Explanation

The threshold check is:

```rust
let threshold = (gas_price_threshold_multiplier * previous_block_l2_gas_price.get().0).to_integer();
if tx_l2_gas_price.0 < threshold { return Err(...) }
``` [6](#0-5) 

Because `previous_block_l2_gas_price` is the wrong value, two classes of error arise:

1. **False rejections (valid transactions denied):** When the next block's price is lower than the current block's price (under-utilized block → EIP-1559 decreases price), a transaction whose `max_price_per_unit` falls in the range `[next_l2_gas_price × threshold%, current_l2_gas_price × threshold%)` is rejected at the gateway even though it satisfies the actual next-block price. Legitimate users are denied service.

2. **False admissions (invalid transactions accepted):** When the next block's price is higher than the current block's price (over-utilized block → EIP-1559 increases price), a transaction whose `max_price_per_unit` falls in the range `[current_l2_gas_price × threshold%, next_l2_gas_price × threshold%)` passes gateway validation but will fail `check_fee_bounds` inside the blockifier during block building, polluting the mempool with transactions that cannot be included.

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The EIP-1559 mechanism adjusts `next_l2_gas_price` every block based on gas usage relative to `gas_target`. Any block that is not exactly at the target causes a price divergence between `l2_gas_price` and `next_l2_gas_price`. This is the normal operating condition of the fee market, so the wrong reference price is used on every block where gas usage deviates from the target — which is essentially always. No special attacker capability is required; any user submitting a transaction is affected.

---

### Recommendation

1. Expose `next_l2_gas_price` through `GatewayFixedBlockStateReader` (either add it to `BlockInfo` or return it as a separate field).
2. In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, read `block_header.next_l2_gas_price` and surface it.
3. In `validate_resource_bounds`, replace `gas_prices.strk_gas_prices.l2_gas_price` with the `next_l2_gas_price` value, resolving the TODO comment.

---

### Proof of Concept

**Setup:** Deploy a sequencer with `validate_resource_bounds = true` and `min_gas_price_percentage = 90`. Produce a block that is 50% full (gas used < gas target), causing EIP-1559 to decrease `next_l2_gas_price` relative to `l2_gas_price`.

**Concrete numbers:**
- Current block: `l2_gas_price = 100 FRI`, `next_l2_gas_price = 95 FRI` (block under-utilized)
- Threshold using current price: `100 × 90% = 90 FRI`
- Threshold using correct next price: `95 × 90% = 85 FRI`

**Attack / user impact:**
- Submit a transaction with `max_price_per_unit = 87 FRI`.
- Gateway computes threshold as `90 FRI` (using wrong `l2_gas_price = 100`), sees `87 < 90`, and **rejects** the transaction with `GAS_PRICE_TOO_LOW`.
- The transaction is valid for the next block (87 ≥ 85), so a legitimate user is incorrectly denied.

Conversely, with an over-utilized block (`next_l2_gas_price = 105 FRI`):
- Submit a transaction with `max_price_per_unit = 92 FRI`.
- Gateway computes threshold as `90 FRI` (using wrong `l2_gas_price = 100`), sees `92 ≥ 90`, and **admits** the transaction.
- Blockifier enforces `next_l2_gas_price = 105 FRI`, threshold = `94 FRI`; `92 < 94` → transaction fails fee bounds during block building, wasting mempool capacity.
