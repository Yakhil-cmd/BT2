### Title
Presignature One-Time-Use Not Enforced at Type or Runtime Level, Enabling Secret Key Extraction by Malicious Coordinator — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based and Robust ECDSA `PresignOutput` types are freely cloneable and serializable. The `rerandomize_presign()` function accepts a shared reference (`&PresignOutput`), leaving the raw presignature intact after use. The library provides no mechanism — no consumed flag, no type-level linear ownership, no runtime guard — to prevent a participant from rerandomizing and signing with the same presignature twice. A malicious coordinator can exploit this by orchestrating two signing sessions against the same presignature with different `(h, ε)` inputs, producing two signatures with multiplicatively related nonces from which the aggregate secret key is directly recoverable.

---

### Finding Description

The OT-based ECDSA `PresignOutput` is declared with `#[derive(Clone, Serialize, Deserialize)]`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
``` [1](#0-0) 

The rerandomization entry point takes the presignature by shared reference, not by value:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← shared reference; presignature is NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
``` [2](#0-1) 

The same pattern holds for the Robust ECDSA variant:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← shared reference
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
``` [3](#0-2) 

Because `PresignOutput` is `Clone` and `Serialize/Deserialize`, any caller can retain a copy of the raw presignature after the first signing session completes. Nothing in the library prevents calling `rerandomize_presign` a second time with different `(msg_hash, tweak, entropy)` arguments, producing a second `RerandomizedPresignOutput` from the same underlying nonce material.

The security documentation explicitly acknowledges the danger but relies entirely on caller discipline:

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [4](#0-3) 

No code-level guard enforces this invariant.

---

### Impact Explanation

**Critical — Extraction of aggregate secret key material.**

For the Robust ECDSA scheme, the security note states directly:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [5](#0-4) 

Two signatures `(R, s₁)` and `(R', s₂)` produced from the same presignature satisfy `R' = δ₂/δ₁ · R` and expose a linear relation between the nonce shares. Standard ECDSA lattice/nonce-reuse techniques recover the full secret key `x` from as few as two such signatures. The same nonce-reuse consequence applies to the OT-based scheme.

---

### Likelihood Explanation

A malicious coordinator controls which presignature each participant is asked to use and which `(msg_hash, tweak, entropy)` arguments are supplied for rerandomization. Because `PresignOutput` is `Clone` and `Serialize/Deserialize`, the coordinator (or any party that stores presignatures) can trivially present the same presignature to participants in two sequential or parallel signing sessions. Participants have no in-library mechanism to detect that a presignature has already been consumed; the library exposes no unique presignature identifier, no consumed marker, and no session registry. The attack requires only a malicious coordinator — an explicitly in-scope threat actor — and normal use of the public API.

---

### Recommendation

1. **Consume `PresignOutput` by value in `rerandomize_presign`**: Change the signature to `presignature: PresignOutput` (owned). This makes Rust's ownership system enforce single use at the type level — once rerandomized, the raw presignature is moved and cannot be reused.

2. **Remove `Clone` from `PresignOutput`** (both OT-based and Robust variants). Serialization can be retained for storage/transport, but deserialization should produce a value that is immediately consumed on first use.

3. **Alternatively**, wrap `PresignOutput` in a newtype that implements `Drop` with a consumed flag and panics (or returns an error) on second use.

---

### Proof of Concept

```rust
// Attacker-controlled coordinator scenario (Robust ECDSA)
let presign_out: PresignOutput = /* obtained from presigning phase */;

// Session 1: sign message h1 with tweak ε1
let args1 = RerandomizationArguments::new(pk, tweak1, h1_bytes, presign_out.big_r, participants.clone(), entropy1);
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args1).unwrap();
// participant sends s_i^1 to coordinator

// Session 2: sign message h2 with tweak ε2 — SAME presign_out, still alive
let args2 = RerandomizationArguments::new(pk, tweak2, h2_bytes, presign_out.big_r, participants.clone(), entropy2);
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args2).unwrap();
// participant sends s_i^2 to coordinator

// Coordinator now holds two signatures (R1,s1) and (R2,s2) with related nonces.
// Standard ECDSA nonce-reuse recovery extracts the secret key x.
```

`rerandomize_presign` accepts `&PresignOutput` and `PresignOutput` derives `Clone`, so both calls succeed without any library-level objection. [6](#0-5)

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
