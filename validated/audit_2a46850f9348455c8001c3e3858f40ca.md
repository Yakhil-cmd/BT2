### Title
No-In-Flight Guard in `dispatchPairRequest` Allows Concurrent WalletConnect Spending Commands to Execute Without Re-Confirmation - (File: packages/gui/src/electron/utils/dispatchPairRequest.ts)

### Summary
`dispatchPairRequest` performs authorization checks (pair validity, fingerprint, bypass list) and then calls `process(context)` to execute a wallet command, but never sets any "in-use" or "in-flight" flag before the external call. A malicious dapp can send multiple concurrent WalletConnect `session_request` events for the same spending command; all concurrent invocations pass the authorization checks simultaneously and all reach `process(context)`, causing N wallet transactions to be submitted instead of one.

### Finding Description

`dispatchPairRequest` in `packages/gui/src/electron/utils/dispatchPairRequest.ts` is the main authorization gate for every WalletConnect command dispatched from the Electron main process:

```
// verify all checks …
if (pair.bypass.includes(command)) {
    return process(context);          // ← no in-flight guard
}
const isAllowed = await confirm();
if (isAllowed === true) {
    return process(context);          // ← no in-flight guard
}
``` [1](#0-0) 

The function is invoked from the `DISPATCH_AS_PAIR` IPC handler in `main.tsx`, which is itself triggered by `window.permissionsAPI.dispatchAsPair(...)` in the renderer: [2](#0-1) 

The renderer calls `dispatchAsPair` inside `handleProcess` in `useWalletConnectCommand.tsx`: [3](#0-2) 

`handleProcess` is called from `processSessionRequest` in `WalletConnectProvider.tsx`, which is registered as the `session_request` event handler with no serialization or deduplication: [4](#0-3) 

`processSessionRequest` itself has no in-flight guard either: [5](#0-4) 

The entire call chain — from `session_request` event to `process(context)` — contains no mutex, no pending-request set, and no "mark as in-use before executing" step. This is the direct CEI analog: the check (`pair.bypass.includes(command)`) is performed, but the pair's state is never updated to "in-use" before the external call (`process(context)`), so N concurrent invocations all pass the check and all execute.

**Bypass path (most severe):** When a command is in `pair.bypass`, `process(context)` is called immediately with no `await` before it and no confirmation dialog. N concurrent requests all reach this branch simultaneously and all execute N wallet commands. [6](#0-5) 

The UI explicitly supports bypassable spending commands: [7](#0-6) 

**Confirmation path:** Even for non-bypassed commands, `await confirm()` opens an async dialog window. While that dialog is pending, a second concurrent request also reaches `await confirm()` and opens a second dialog. If the user approves both (or is confused into doing so), both `process(context)` calls execute. [8](#0-7) 

### Impact Explanation

For commands in `pair.bypass` (spending commands the user opted to allow without per-request confirmation), a malicious dapp can submit N concurrent `session_request` events and cause N `chia_sendTransaction`, `chia_takeOffer`, or equivalent spending commands to execute against the user's wallet — all without any additional user approval. The user intended to grant "execute this command without asking me each time," not "execute this command an unlimited number of times simultaneously." The result is unauthorized spend/transfer of XCH, CAT, or NFT assets.

### Likelihood Explanation

Requires the user to have paired with a malicious dapp and to have granted bypass for at least one spending command. Once those conditions are met, the exploit is fully autonomous: the dapp sends concurrent `session_request` frames over the WalletConnect relay and the GUI executes all of them. No further user interaction is needed.

### Recommendation

Introduce a per-`(topic, command)` in-flight set in `dispatchPairRequest` (or at the `DISPATCH_AS_PAIR` IPC handler level). Before any check or external call, insert the request key; remove it in a `finally` block. Reject any concurrent request whose key is already present. This mirrors the CEI fix recommended in the original report: update state (mark in-use) before making the external call, so re-entrant/concurrent invocations are blocked.

```typescript
const inFlight = new Set<string>();

export async function dispatchPairRequest(...) {
  const key = `${topic}:${command}`;
  if (inFlight.has(key)) {
    throw new WcError('Request already in progress', WcErrorCode.INTERNAL_ERROR);
  }
  inFlight.add(key);
  try {
    // existing checks and process(context) call
  } finally {
    inFlight.delete(key);
  }
}
```

### Proof of Concept

1. User pairs with a malicious dapp and grants `chia_sendTransaction` with bypass confirmation enabled.
2. Malicious dapp simultaneously sends 10 `session_request` events over WalletConnect, each requesting `chia_sendTransaction` with `amount=<wallet_balance/10>`.
3. All 10 events fire `handleSessionRequest` concurrently in `WalletConnectProvider.tsx`.
4. All 10 reach `dispatchPairRequest`; all pass `pair.bypass.includes('chia_sendTransaction')` simultaneously (no in-flight guard).
5. All 10 call `process(context)` → `sendCommand('send_transaction', ...)` → 10 transactions submitted to the daemon.
6. User's wallet is drained without any confirmation dialog appearing.
