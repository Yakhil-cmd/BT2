Looking at the exact code path in `runtime/runtime/src/ext.rs`:

**`submit_promise_resume_data_with_yield_id`** (lines 421–433) calls `get_data_id_for_yield_id` to resolve the `user_yield_id` → `data_id`, then delegates to `submit_promise_resume_data`.


The critical flaw: `has_promise_yield_status` checks only for key **existence**, not the **value**. After the first resume, the status is overwritten to `ResumeInitiated` — but the key is still present. The `YieldIdToDataId` mapping is also not cleared on resume (only on timeout via `remove_yield_id_mappings`). So a second call with the same `user_yield_id` within the same function call will:
1. Resolve the same `data_id` via the still-present `YieldIdToDataId` entry
2. Find `has_promise_yield_status` = `true` (key exists, value = `ResumeInitiated`)
3. Create a **second** `PromiseResumeDataReceipt` for the same `data_id`
4. Return `Ok(true)` again

This is a real protocol invariant violation.

### Title
Double-Resume of a Yield via `submit_promise_resume_data_with_yield_id` When Status Is Already `ResumeInitiated` — (`runtime/runtime/src/ext.rs`)

### Summary
`submit_promise_resume_data` does not distinguish between `PromiseYieldStatus::Yielded` and `PromiseYieldStatus::ResumeInitiated` when deciding whether to create a resume receipt. Because `has_promise_yield_status` is a pure existence check and the `YieldIdToDataId` mapping is not cleared on resume, a contract can call `promise_yield_resume_with_yield_id` twice with the same `user_yield_id` in a single execution, producing two `PromiseResumeDataReceipt`s for the same `data_id`.

### Finding Description

In `submit_promise_resume_data`: [1](#0-0) 

`has_promise_yield_status` returns `true` for **any** stored `PromiseYieldStatus` value, including `ResumeInitiated`. After the first resume call sets the status to `ResumeInitiated`, the key remains in the trie. The `YieldIdToDataId` mapping is also not removed at resume time: [2](#0-1) 

The `YieldIdToDataId` entry is only cleaned up via `remove_yield_id_mappings`, which is called from `lib.rs` on timeout processing, not on resume: [3](#0-2) 

So within a single contract function call, calling `promise_yield_resume_with_yield_id` twice with the same `user_yield_id` succeeds both times, each time calling `create_promise_resume_receipt` and returning `Ok(true)`.

The `PromiseYieldStatus` variants are: [4](#0-3) 

The guard that prevents duplicate yields at creation time (`has_yield_id_mapping` check in `create_promise_yield_receipt_with_id`) has no equivalent guard at resume time: [5](#0-4) 

### Impact Explanation

Two `PromiseResumeDataReceipt`s are emitted for the same `data_id`. The exact corrupted value is the `PromiseYieldReceipt` callback's input data: whichever of the two data receipts is processed second will overwrite `ReceivedData` for that `data_id` before the callback executes (if both arrive in the same block), or the second receipt will arrive after the callback has already consumed and cleaned up the first, leaving an orphaned `ReceivedData` entry. In the first case, the callback receives attacker-chosen data regardless of what a legitimate first resume submitted. In the second case, a stale `ReceivedData` entry persists in state, leaking storage.

### Likelihood Explanation

Any contract that uses `promise_yield_create_with_id` / `promise_yield_resume_with_yield_id` (the named-yield API) can trigger this by calling the resume host function twice in one execution. No special privileges are required — this is reachable from ordinary contract Wasm via the `promise_yield_resume_with_yield_id` host function exposed through `VMLogic`.

### Recommendation

In `submit_promise_resume_data`, replace the existence check with a value check. Only proceed if the stored status is exactly `PromiseYieldStatus::Yielded`:

```rust
let yield_status =
    get_promise_yield_status(self.trie_update, &self.account_id, data_id)
        .map_err(wrap_storage_error)?;

let is_resumable = match yield_status {
    Some(PromiseYieldStatus::Yielded) => true,
    _ => has_promise_yield_receipt(...),  // legacy path without status
};
```

Additionally, clear the `YieldIdToDataId` / `DataIdToYieldId` mappings at resume time (not only at timeout), so that a second call to `submit_promise_resume_data_with_yield_id` with the same `user_yield_id` returns `Ok(false)` immediately.

### Proof of Concept

1. Deploy a contract with a method that:
   - Calls `promise_yield_create_with_id(receiver=self, user_yield_id=[0u8;32], ...)` → gets `data_id`
   - Calls `promise_yield_resume_with_yield_id(user_yield_id=[0u8;32], data=b"first")` → asserts returns `true`
   - Calls `promise_yield_resume_with_yield_id(user_yield_id=[0u8;32], data=b"second")` → **currently returns `true`**, should return `false`
2. Observe that two `PromiseResumeDataReceipt`s are present in the outgoing receipts for the same `data_id`.
3. Confirm the callback receives `b"second"` (the second resume payload) instead of `b"first"`, or that a stale `ReceivedData` entry persists after callback execution.
