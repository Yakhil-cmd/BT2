### Title
Stale Program Cache on Access/Observer Nodes After Contract Update — (`module/state_synchronization/indexer/util.go`)

### Summary

`programInvalidator.ShouldInvalidateEntries()` only checks `inv.invalidateAll`, but `ShouldInvalidateEntry()` also checks the `inv.invalidated` map. Because `TableTransaction.AddInvalidator` gates entirely on `ShouldInvalidateEntries()`, any block that contains contract update events but no service-account-authorized transaction causes the invalidator to be silently dropped. The program cache on Access/Observer nodes is never invalidated for those contracts, and all subsequent scripts executed on those nodes use stale contract code.

### Finding Description

**Root cause — inconsistent `ShouldInvalidateEntries` vs `ShouldInvalidateEntry`:**

In `module/state_synchronization/indexer/util.go`, `programInvalidator` has two methods:

```go
func (inv *programInvalidator) ShouldInvalidateEntries() bool {
    return inv.invalidateAll          // ← only checks invalidateAll
}

func (inv *programInvalidator) ShouldInvalidateEntry(...) bool {
    _, ok := inv.invalidated[location]
    return inv.invalidateAll || ok    // ← also checks the invalidated map
}
``` [1](#0-0) 

**Gate in `TableTransaction.AddInvalidator`:**

```go
func (txn *TableTransaction[TKey, TVal]) AddInvalidator(
    invalidator TableInvalidator[TKey, TVal],
) {
    if invalidator == nil || !invalidator.ShouldInvalidateEntries() {
        return   // ← silently dropped when invalidateAll==false
    }
    txn.invalidators = append(...)
}
``` [2](#0-1) 

**Caller in `updateProgramCache`:**

```go
tx.AddInvalidator(&accessInvalidator{
    programs: &programInvalidator{
        invalidated:   updatedContracts,
        invalidateAll: hasAuthorizedTransaction(collections, c.serviceAddress),
    },
    ...
})
``` [3](#0-2) 

When `hasAuthorizedTransaction` returns `false` (no service-account tx in the block) but `findContractUpdates` returns a non-empty map (one or more contracts were updated), `invalidateAll=false` and `invalidated={loc:{}}`. `ShouldInvalidateEntries()` returns `false`, so `AddInvalidator` returns immediately without appending anything to `txn.invalidators`.

**Consequence at commit time:**

```go
if txn.invalidators.ShouldInvalidateEntries() {   // false → entire block skipped
    ...
    table.invalidators = append(table.invalidators, txn.invalidators...)
}
``` [4](#0-3) 

The invalidator is never appended to `table.invalidators`. Future transactions whose `snapshotTime` is past the update will call `ApplicableInvalidators` and find nothing, so cached programs for the updated contracts are never marked invalid and never evicted.

### Impact Explanation

Access and Observer nodes use `derivedChainData` to cache parsed/compiled Cadence programs. Scripts executed on these nodes call `GetOrComputeProgram`, which returns the cached (now stale) program without recomputing it. Any script or read-only query that depends on the updated contract executes against the old code until the cache is eventually cleared by a future service-account-authorized transaction. This produces silently incorrect script results returned to users and dApps.

### Likelihood Explanation

This is triggered by any ordinary (non-service-account) contract update — a common operation on Flow. The attacker does not need special privileges; any account that deploys or updates a contract triggers the path. The bug is deterministic and reproducible on a local network.

### Recommendation

Fix `ShouldInvalidateEntries()` in `programInvalidator` to also check whether the `invalidated` map is non-empty:

```go
func (inv *programInvalidator) ShouldInvalidateEntries() bool {
    return inv.invalidateAll || len(inv.invalidated) > 0
}
``` [5](#0-4) 

### Proof of Concept

1. Create a `programInvalidator` with `invalidateAll=false` and `invalidated=map[loc:{}]`.
2. Call `tx.AddInvalidator(&accessInvalidator{programs: inv, ...})`.
3. Call `tx.Commit()`.
4. Assert `block.programs.InvalidatorsForTestingOnly()` is empty — it will be.
5. Create a new `TableTransaction` with `snapshotTime > 0`, call `GetOrCompute` for the updated contract's location — it returns the stale cached entry without recomputing. [6](#0-5)
