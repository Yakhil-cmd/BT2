### Title
Missing Sender Authorization in Bank Precompile `transfer` Allows Unauthorized Native Token Theft - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The `BankContract.Run` method's `TransferMethodName` case accepts an arbitrary `sender` address from the caller's input and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `contract.Caller() == sender`. Any EVM contract can therefore drain native tokens (denom `evm/<contract_address>`) from any holder's account without the holder's consent.

### Finding Description

The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes four methods: `mint`, `burn`, `balanceOf`, and `transfer`. For `mint` and `burn`, the denom is derived from `contract.Caller()`, so only the calling contract can issue or destroy its own tokens — no additional authorization is needed. For `transfer`, the same denom scoping applies, but the `sender` (the account whose balance is debited) is taken directly from caller-supplied input with no check that it equals `contract.Caller()`:

```go
// lines 175-192
sender := args[0].(common.Address)          // attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())       // arbitrary victim
...
denom := EVMDenom(contract.Caller())         // evm/<calling_contract>
amt := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, ...)  // no auth on `from`
``` [1](#0-0) 

The intended usage shown in the integration test contract passes `msg.sender` as the first argument:

```solidity
// TestBank.sol line 37
return bank.transfer(msg.sender, recipient, amount);
``` [2](#0-1) 

But the precompile enforces nothing — any address may be supplied. By contrast, the other precompiles (`relayer`, `ica`) use the `exec` helper in `utils.go` which explicitly verifies `caller == signer`:

```go
// utils.go lines 46-49
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
``` [3](#0-2) 

The bank precompile's `transfer` path has no equivalent guard. [4](#0-3) 

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

Any EVM contract holding the `evm/<contract_address>` denom can call `transfer(victim, attacker, balance)` and move the victim's entire native-token balance to an attacker-controlled address in a single transaction. The victim's Cosmos-side balance is debited via `bankKeeper.SendCoins` with no signature or approval from the victim. This satisfies the Critical criterion: *unauthorized transfer of precompile-controlled assets*.

### Likelihood Explanation

The bank precompile is a live, registered precompile at address `0x0000000000000000000000000000000000000064`. [5](#0-4) 

Any unprivileged user can deploy an EVM contract and call the precompile directly. No special keys, governance access, or leaked secrets are required. The only precondition is that the victim holds a non-zero balance of the denom `evm/<attacker_contract_address>`, which the attacker can arrange by first calling `mint` to distribute tokens and then calling `transfer` to reclaim them — or by targeting any existing holder of the denom.

### Recommendation

Add a check that the `sender` argument equals `contract.Caller()` before executing the transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
    ...
```

Alternatively, remove the `sender` argument entirely and always use `contract.Caller()` as the debit address, consistent with how `mint` and `burn` are scoped.

### Proof of Concept

1. Attacker deploys `MaliciousContract` at address `0xABC...`.
2. `MaliciousContract` calls `bank.mint(victim, 1000)` → victim now holds 1000 `evm/0xABC...` native tokens on the Cosmos side.
3. `MaliciousContract` calls `bank.transfer(victim, attacker_eoa, 1000)`.
4. The bank precompile executes `bankKeeper.SendCoins(ctx, victim_cosmos_addr, attacker_cosmos_addr, [{denom:"evm/0xABC...", amount:1000}])` with no authorization check on `victim_cosmos_addr`.
5. Victim's balance is zero; attacker holds 1000 `evm/0xABC...` tokens. No victim signature was required at any step.
