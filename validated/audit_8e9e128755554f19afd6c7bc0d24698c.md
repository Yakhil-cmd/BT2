### Title
Overly Broad `skip_stateful_validations` Check Allows Signature-Bypass Admission of Invoke Transactions — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip `__validate__` (signature verification) for an invoke transaction with nonce=1 when the account's on-chain nonce is 0, as a UX convenience for the deploy_account + invoke flow. However, the guard condition — `account_tx_in_pool_or_recent_block` — is too broad: it returns `true` if **any** transaction from the account is in the mempool, not specifically a `deploy_account` transaction. An attacker can exploit this to inject an invoke transaction with an arbitrary/invalid signature into the mempool, bypassing the `__validate__` entry point entirely.

---

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls three sub-checks in sequence:

1. `validate_state_preconditions` — checks nonce range and resource bounds
2. `validate_by_mempool` — checks for duplicate nonces/hashes
3. `skip_stateful_validations` — decides whether to skip `__validate__` [1](#0-0) 

The `skip_stateful_validations` function skips `__validate__` when all three conditions hold:

```
tx.nonce() == Nonce(Felt::ONE)
&& account_nonce == Nonce(Felt::ZERO)
&& account_tx_in_pool_or_recent_block(sender_address) == true
``` [2](#0-1) 

The comment in the code states the rationale:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The `account_tx_in_pool_or_recent_block` implementation checks `MempoolState::contains_account` (staged or committed) OR `TransactionPool::contains_account`: [3](#0-2) [4](#0-3) 

This returns `true` for **any** pending transaction from the address — including a regular nonce=0 invoke — not exclusively a `deploy_account` transaction.

**Attack scenario:**

1. Account A is deployed via a `deploy` syscall from another contract (not via `deploy_account`). Its on-chain nonce is 0 and it has valid contract code.
2. The legitimate owner of account A submits a nonce=0 invoke transaction. It passes `__validate__` (signature is valid) and enters the mempool.
3. An attacker submits a nonce=1 invoke from account A with an **arbitrary/invalid signature**.
4. Gateway stateful validation:
   - `validate_nonce`: `account_nonce=0 ≤ tx_nonce=1 ≤ max_allowed_nonce` → passes
   - `validate_by_mempool`: no duplicate → passes
   - `skip_stateful_validations`: `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true` (the nonce=0 invoke is in the pool) → returns `true`
5. `run_validate_entry_point` is called with `skip_validate=true`, setting `execution_flags.validate = false`: [5](#0-4) 

6. Inside `StatefulValidator::perform_validations`, when `validate=false`, the `__validate__` call is skipped entirely and `Ok(())` is returned: [6](#0-5) 

7. The attacker's nonce=1 invoke with an invalid signature is **admitted to the mempool** without any signature verification.

A second variant exists even for the intended deploy_account + invoke UX: if the legitimate user submitted a deploy_account (nonce=0) and a nonce=1 invoke, an attacker can submit a **higher-fee** nonce=1 invoke with an invalid signature. The mempool's fee-escalation logic will accept it (displacing the legitimate nonce=1 invoke), since `__validate__` is skipped: [7](#0-6) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway admits invoke transactions with invalid/arbitrary signatures into the mempool. These transactions will fail during blockifier execution (when `__validate__` is run during block building with `strict_nonce_check=false`), but they have already been admitted. Consequences:

- **Mempool pollution / DoS**: An attacker can flood the mempool with signature-invalid transactions for any account that has nonce=0 in state and a pending transaction in the pool.
- **Displacement of legitimate transactions**: Via fee escalation, an attacker can replace a legitimate user's nonce=1 invoke with an invalid-signature transaction, causing the legitimate transaction to be evicted. The attacker's transaction then fails during execution, leaving the user's nonce=1 slot permanently occupied until the invalid transaction is pruned.

---

### Likelihood Explanation

**Medium.** The preconditions are:
1. The target account must have nonce=0 in on-chain state (deployed via `deploy` syscall, or a fresh account with a pending deploy_account).
2. The target account must have at least one pending transaction in the mempool.

Condition 2 is observable by any network participant (mempool is public). Condition 1 is common for contracts deployed via factory patterns. The attacker needs no privileged access — only the ability to submit RPC transactions.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should track deploy_account transactions separately. The current check conflates "account has any pending transaction" with "account has a pending deploy_account," which is the root of the bypass.

Alternatively, restrict the skip condition to only apply when the account has **no code** at the sender address (i.e., the account truly does not exist yet), which is the only case where skipping `__validate__` is semantically safe.

---

### Proof of Concept

```
1. Deploy contract C via a factory (deploy syscall). C has nonce=0 in state, has code.
2. Owner of C submits: Invoke(sender=C, nonce=0, sig=valid_sig, calldata=...) → admitted to mempool (passes __validate__).
3. Attacker submits: Invoke(sender=C, nonce=1, sig=0xdeadbeef_invalid, calldata=attacker_calldata)
   - validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce → OK
   - validate_by_mempool: no duplicate nonce=1 → OK
   - skip_stateful_validations:
       tx.nonce() == 1 ✓
       account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(C) == true ✓ (nonce=0 invoke is in pool)
     → returns true (skip __validate__)
   - run_validate_entry_point(skip_validate=true) → __validate__ NOT called → Ok(())
4. Attacker's nonce=1 invoke with invalid signature is in the mempool.
5. If attacker sets resource_bounds higher than owner's nonce=1 invoke (if any),
   fee escalation evicts the legitimate transaction.
``` [2](#0-1) [8](#0-7) [3](#0-2) [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    /// Validates an incoming transaction and handles fee escalation.
    fn add_tx_validations(
        &mut self,
        tx_reference: TransactionReference,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
    ) -> MempoolResult<()> {
        self.validate_incoming_tx(tx_reference, account_nonce)?;
        let replaced_tx_reference = self.validate_fee_escalation(tx_reference)?;

        // The replaced transaction is still pooled, so its bytes still count toward
        // `size_in_bytes()`. Credit what its removal will free: a same-size bump nets to zero (no
        // overflow handling), and a larger replacement only needs room for the delta, consistent
        // with how a fresh next-nonce transaction is treated. The removal happens only after
        // capacity is confirmed below, so a rejected incoming transaction never strands the
        // account.
        let freed_bytes = replaced_tx_reference.map_or(0, |reference| {
            self.tx_pool
                .get_by_tx_hash(reference.tx_hash)
                .expect("Replacement target from pool must exist.")
                .total_bytes()
        });

        if self.exceeds_capacity(tx, freed_bytes) {
            self.handle_capacity_overflow(tx, account_nonce, freed_bytes)?;
        }

        // Capacity is confirmed: this is the final, infallible mutation before the incoming
        // transaction is inserted by the caller.
        if let Some(existing_tx_reference) = replaced_tx_reference {
            self.remove_replaced_tx(existing_tx_reference);
        }

        Ok(())
    }
