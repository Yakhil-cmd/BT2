### Title
Unprivileged Caller Can Permanently Suppress Per-Block Consensus Rewards via `disable_rewards` Flag in `update_rewards` - (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is callable by any non-zero address with no role restriction. It unconditionally writes `last_reward_block` to the current block number **before** checking the `disable_rewards` flag. An attacker can call `update_rewards(valid_staker, disable_rewards: true)` to consume the block's reward slot without distributing any rewards, permanently destroying that block's consensus rewards for all stakers.

---

### Finding Description

In `src/staking/staking.cairo`, the `update_rewards` function (part of `IStakingRewardsManager`) has the following structure:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();          // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);   // ← written unconditionally

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // ← exits WITHOUT distributing rewards
    }
    // ... reward distribution logic follows
``` [1](#0-0) 

The `general_prerequisites()` helper only enforces that the contract is not paused and the caller is non-zero: [2](#0-1) 

There is **no role check** on `update_rewards`. Any externally-owned address can call it. The critical flaw is that `last_reward_block` is written at line 1485 **before** the `disable_rewards` branch at line 1487. Once `last_reward_block` equals the current block number, every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`. The rewards that would have been distributed for that block are permanently lost — they are never added to `unclaimed_rewards` in the `RewardSupplier`, and the block number can never be revisited. [3](#0-2) 

The analog to the external report is direct: just as `verify_signature()` accepted scheme byte `3` unconditionally without verifying any proof, `update_rewards` accepts the caller-supplied `disable_rewards: true` unconditionally without verifying that the caller is authorized to suppress rewards, and in both cases the critical operation (proof verification / reward distribution) is silently skipped while the side-effect (marking the action as done) is permanently committed.

---

### Impact Explanation

This is in the **consensus rewards phase** (after `consensus_rewards_first_epoch` is set). In that phase, `update_rewards` is the sole mechanism for distributing per-block STRK and BTC rewards to stakers and their delegation pools. [4](#0-3) 

If an attacker calls `update_rewards(valid_staker, disable_rewards: true)` once per block, **zero consensus rewards are ever distributed** to any staker or delegator. The `unclaimed_rewards` balance in `RewardSupplier` is never incremented, so no funds flow to stakers or pools. This constitutes **permanent freezing of unclaimed yield** for all protocol participants — a High-severity impact under the allowed scope.

Even a partial attack (suppressing rewards for a subset of blocks) causes proportional permanent yield loss, since each block's reward window is non-recoverable.

---

### Likelihood Explanation

- **No privilege required**: The attacker only needs a non-zero address and a valid `staker_address` (all staker addresses are publicly emitted via `NewStaker` events).
- **No capital at risk**: The attacker spends only gas.
- **Repeatable every block**: The `last_reward_block` guard resets each block, so the attacker must call once per block to sustain the attack — feasible with an automated script.
- **No front-running dependency on Starknet**: The attacker does not need to race against a specific transaction; they simply need to be the first caller of `update_rewards` in any given block, which is trivially achievable by submitting the transaction at the start of each block.

Likelihood: **High** — zero-cost, no privilege, fully automatable.

---

### Recommendation

1. **Restrict the caller**: Add an access-control check so that only a designated role (e.g., `only_operator` or `only_block_proposer`) or a specific trusted contract address can call `update_rewards`. This mirrors how `update_rewards_from_attestation_contract` is gated: [5](#0-4) 

2. **Write `last_reward_block` only after rewards are confirmed to be distributed**: Move the `self.last_reward_block.write(current_block_number)` call to after the `disable_rewards` / `is_pre_consensus` guard, so that a suppressed call does not consume the block's reward slot.

3. **Remove or restrict the `disable_rewards` parameter**: If this flag is only needed for internal migration/testing purposes, it should be removed from the public interface or gated behind a privileged role.

---

### Proof of Concept

1. Governance sets `consensus_rewards_first_epoch` to epoch `E`. The protocol enters consensus rewards mode.
2. At the start of block `B` in epoch `E`, attacker (any EOA) calls:
   ```
   staking.update_rewards(staker_address=<any_valid_staker>, disable_rewards=true)
   ```
3. Inside `update_rewards`:
   - `general_prerequisites()` passes (contract not paused, caller non-zero). [6](#0-5) 
   - `last_reward_block` is written to block `B`. [7](#0-6) 
   - `disable_rewards == true` → function returns early, no rewards distributed. [8](#0-7) 
4. The legitimate block-proposer calls `update_rewards(staker_address, disable_rewards=false)` in the same block `B`.
5. The call reverts: `current_block_number (B) > last_reward_block (B)` is false → `REWARDS_ALREADY_UPDATED`. [9](#0-8) 
6. Block `B`'s rewards are permanently lost. Repeating this every block eliminates all consensus rewards indefinitely.
