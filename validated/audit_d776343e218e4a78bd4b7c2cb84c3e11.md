## Analysis

I need to trace the exact code path for `skip_stateful_validations` and `account_tx_in_pool_or_recent_block` to determine whether the described bypass is real.

### Code Path Trace

**Step 1: `skip_stateful_validations` trigger conditions** [1](#0-0) 

The function fires when: `tx` is `ExecutableTransaction::Invoke`, `tx.nonce() == Nonce(Felt::ONE)`, AND `account_nonce == Nonce(Felt::ZERO)`. It then calls `account_tx_in_pool_or_recent_block` and returns its result directly — no type-check on what kind of transaction is in the pool.

**Step 2: `account_tx_in_pool_or_recent_block` implementation** [2](#0-1) 

Returns `true` if the address appears in `self.state` (staged/committed) OR `self.tx_pool`. There is **no filter on transaction type** — any invoke, declare, or deploy_account for that address returns `true`.

**Step 3: `MempoolState::contains_account`** [3](#0-2) 

Checks `staged` or `committed` maps — again, no transaction-type discrimination.

**Step 4: What happens when `skip_validate = true`** [4](#0-3) 

`execution_flags.validate = !skip_validate = false`. Then `StatefulValidator::perform_validations` checks this flag: [5](#0-4) 

If `validate = false`, it returns `Ok(())` immediately — `__validate__` (signature verification) is **never called**.

**Step 5: The comment's flawed assumption** [6](#0-5) 

The comment claims presence in the mempool implies "either a deploy_account or transactions with future nonces that passed validations." This is incorrect: a nonce=0 **invoke** (not deploy_account) for an already-deployed account with `account_nonce=0` can be in the mempool, and it satisfies `account_tx_in_pool_or_recent_block` just as well as a deploy_account would.

**Step 6: Nonce validation allows nonce=1 with account_nonce=0** [7](#0-6) 

For invoke transactions, nonce=1 with account_nonce=0 is accepted as long as `max_allowed_nonce_gap >= 1` (the default).

### Concrete Attack Scenario

**Precondition**: Account A is deployed on-chain, `account_nonce = 0` (no transactions committed yet).

1. Attacker submits **invoke(nonce=0, valid_signature)** for address A → passes `validate_nonce` (nonce=0 == account_nonce=0), passes `validate_by_mempool`, `skip_stateful_validations` returns `false` (pool empty for A), `__validate__` runs and passes → **tx admitted to mempool, `tx_pool.contains_account(A) = true`**.

2. Attacker submits **invoke(nonce=1, arbitrary_invalid_signature)** for address A:
   - `validate_nonce`: nonce=1 ∈ [0, 0+gap], passes.
   - `validate_by_mempool` → `validate_incoming_tx`: nonce=1 ≥ account_nonce=0, no existing nonce=1 tx, passes.
   - `skip_stateful_validations`: nonce=1, account_nonce=0 → calls `account_tx_in_pool_or_recent_block(A)` → **returns `true`** (nonce=0 invoke is in pool) → returns `true`.
   - `run_validate_entry_point` with `validate=false` → `__validate__` **skipped**.
   - **Invalid-signature invoke admitted to mempool.**

### Title
**`skip_stateful_validations` bypasses `__validate__` for nonce=1 invoke when any (non-deploy_account) transaction from the same sender is in the mempool** — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
`skip_stateful_validations` is intended to skip `__validate__` only when a `deploy_account` is pending in the mempool (account not yet deployed). However, `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type in the pool for the sender address. An attacker who first places a valid nonce=0 invoke for an already-deployed account (account_nonce=0) can then submit a nonce=1 invoke with an arbitrary invalid signature, which bypasses `__validate__` entirely and is admitted to the mempool.

### Finding Description
In `skip_stateful_validations`: [8](#0-7) 

The function checks `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)`, then delegates the "is a deploy_account pending?" question to `account_tx_in_pool_or_recent_block`. But that function: [2](#0-1) 

…returns `true` for **any** transaction type in the pool or recent block for that address. The code comment acknowledges this but incorrectly reasons that any pooled transaction implies a deploy_account was submitted. In reality, a nonce=0 invoke for an already-deployed account (account_nonce=0) satisfies the check equally.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [9](#0-8) 

And `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`: [5](#0-4) 

### Impact Explanation
An invoke transaction with an **arbitrary invalid signature** is admitted to the mempool without signature verification. This is a concrete corrupted admission value: the mempool contains a transaction that would fail `__validate__` at execution time. Impact: **High — mempool/gateway admission accepts invalid transactions**.

### Likelihood Explanation
Requires an already-deployed account with `account_nonce=0` (no committed transactions) and a valid nonce=0 invoke already in the mempool for that address. The attacker can be a third party who observes the mempool and submits the nonce=1 invoke with an invalid signature. The `max_allowed_nonce_gap` default allows nonce=1 with account_nonce=0.

### Recommendation
In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a **deploy_account** transaction (not just any transaction) is pending for the sender address. Alternatively, add a dedicated mempool API `deploy_account_tx_in_pool_or_recent_block` that filters by transaction type.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
