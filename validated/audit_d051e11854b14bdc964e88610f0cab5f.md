### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Protocol Hard Fee Cap - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolBinAdditionalFees` applies no upper-bound validation on `addFeeBuyE6` / `addFeeSellE6`, allowing the pool admin to set per-bin spread additions up to `type(uint16).max` = 65 535 (≈ 6.55 % in E6 units) on any bin. Because these values are added directly on top of the global spread fee inside every swap path, the effective per-bin fee can exceed the protocol's hard cap of 20 % (`HARD_MAX_SPREAD_FEE_E6 = 200_000`), breaking the admin-boundary invariant and directly extracting excess value from traders.

### Finding Description

The factory enforces hard caps on every other admin-controlled fee parameter:

- `setPoolAdminFees` checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` and reverts. [1](#0-0) 

- `setFeeCaps` itself is bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %). [2](#0-1) 

`setPoolBinAdditionalFees`, however, passes the caller-supplied values straight through to the pool with no validation whatsoever:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool's `setBinAdditionalFees` likewise performs no cap check — it only validates the bin index range:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [4](#0-3) 

During every swap step the per-bin additional fee is added directly on top of `baseFeeX64` (which already encodes the global spread fee):

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The same uncapped addition appears in `getSellAndBuyPrices`:

```solidity
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [9](#0-8) 

`uint16` allows values up to 65 535, which in E6 units equals ≈ 6.55 %. Combined with a global spread fee already at the 20 % hard cap, the effective per-bin fee reaches ≈ 26.55 % — 33 % above the protocol ceiling.

### Impact Explanation

Every trader whose swap routes through a bin with a maximised `addFeeBuyE6` / `addFeeSellE6` pays a fee that exceeds the protocol's hard cap. The excess fee accrues inside the pool's LP balances (it is not separately tracked), so it is effectively extracted from traders and redistributed to LPs controlled by the pool admin. This is a direct, quantifiable loss of trader principal on every affected swap, satisfying the Sherlock Medium/High threshold.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly bounded by caps. The call path is a single permissioned transaction with no timelock. Any pool admin who wishes to extract excess fees can do so immediately and silently — the only on-chain signal is a `BinAdditionalFeesUpdated` event, which most traders do not monitor. Because pool creation is permissionless, a malicious actor can deploy a pool, attract liquidity, and then raise per-bin fees to the maximum.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and mirror it in `setBinAdditionalFees` as a defence-in-depth guard):

```solidity
// In MetricOmmPoolFactory.setPoolBinAdditionalFees
uint24 maxBinFee = maxAdminSpreadFeeE6; // or a dedicated per-bin cap
if (uint256(addFeeBuyE6) + uint256(addFeeSellE6) > maxBinFee) revert BinFeeTooHigh();
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` constant (e.g. equal to `HARD_MAX_SPREAD_FEE_E6`) and enforce it on both the buy and sell legs independently.

### Proof of Concept

1. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
2. No revert — `addFeeBuyE6 = addFeeSellE6 = 65535` (≈ 6.55 % each) is stored on bin 0.
3. A trader calls `pool.swap(...)` routing through bin 0.
4. The swap math computes effective fee = `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`.
5. If `spreadFeeE6 = 200_000` (20 %), the effective bin fee is ≈ 26.55 %, exceeding the 20 % hard cap.
6. The trader receives fewer output tokens than the hard cap permits; the excess remains in the pool, benefiting the admin-controlled LP position. [3](#0-2) [4](#0-3) [10](#0-9)
