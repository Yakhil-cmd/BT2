### Title
Unvalidated Participant-Supplied CKD Shares Accepted Without Proof, Enabling Any Single Malicious Participant to Corrupt the Derived Confidential Key - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

In `do_ckd_coordinator`, the coordinator aggregates `CKDOutput` values (`big_y`, `big_c`) received from each participant by blindly summing them. There is no zero-knowledge proof or consistency check that each participant's submitted group elements were honestly derived from their actual key share. A single malicious participant can submit arbitrary elliptic curve points, permanently corrupting the final CKD output and causing all honest parties to accept an incorrect derived confidential key.

---

### Finding Description

The CKD protocol computes a threshold ElGamal encryption of `msk · H(pk ∥ app_id)`. Each participant `i` is supposed to compute:

```
big_y_i  = y_i · G                          (random blinding)
big_c_i  = x_i · H(pk ∥ app_id) + y_i · app_pk   (ElGamal share)
```

scaled by their Lagrange coefficient `λ_i`, and send `(λ_i · big_y_i, λ_i · big_c_i)` to the coordinator.

The coordinator in `do_ckd_coordinator` receives these values and sums them: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no verification** that the received `big_y` and `big_c` are correctly formed. Specifically, there is no proof that:
- `big_y = y · G` for some scalar `y` the participant committed to
- `big_c = x_i · H(pk ∥ app_id) + y · app_pk` using the participant's actual key share `x_i`

This is in stark contrast to the DKG protocol, which enforces proofs of knowledge and commitment hashes before accepting any participant contribution: [2](#0-1) 

The `compute_signature_share` function that honest participants call is entirely local and unverifiable by the coordinator: [3](#0-2) 

A malicious participant simply never calls `compute_signature_share` and instead sends arbitrary `(big_y', big_c')` group elements directly.

---

### Impact Explanation

The client recovers the confidential key as:

```
S = big_c - app_sk · big_y  =  msk · H(pk ∥ app_id)
```

If a malicious participant `m` sends `(big_y', big_c')` instead of their correct share, the coordinator outputs:

```
Y_corrupt = Y_correct + (big_y' - λ_m · y_m · G)
C_corrupt = C_correct + (big_c' - λ_m · (x_m · H(pk ∥ app_id) + y_m · app_pk))
```

The client then computes:

```
C_corrupt - app_sk · Y_corrupt
  = msk · H(pk ∥ app_id) + (big_c' - app_sk · big_y')
```

Since `app_sk` is unknown to the malicious participant, the additive error `big_c' - app_sk · big_y'` is an unpredictable non-zero group element. The derived key is permanently wrong and unusable. The honest coordinator and all honest participants have no way to detect this corruption because no output verification step exists in the CKD protocol.

**Matched allowed impact:** High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker only needs to be a registered participant (a valid `Participant` ID in the list). No threshold of colluders is required — one malicious party suffices. The attack requires no special knowledge: the attacker simply sends two arbitrary `G1` points instead of their honest contribution. The coordinator has no mechanism to reject or detect the malformed input. [4](#0-3) 

---

### Recommendation

Each participant should accompany their `(big_y, big_c)` submission with a non-interactive zero-knowledge proof (e.g., a Chaum-Pedersen proof or a Sigma protocol) demonstrating that:

1. `big_y = y · G` for a known discrete log `y`
2. `big_c = x_i · H(pk ∥ app_id) + y · app_pk` using the same `y` and the participant's committed key share `x_i`

The coordinator must verify all proofs before aggregating. This is the standard approach used in verifiable threshold ElGamal decryption schemes (e.g., Pedersen's DKG, PVSS). Alternatively, the coordinator can perform a consistency check by verifying the final output against the known public key `msk · G2` using a pairing check, though this only detects corruption after the fact and does not identify the culprit.

### Proof of Concept

**Setup:** 3 participants, threshold 2, coordinator = participant 0. Participant 2 is malicious.

**Honest flow (expected):**
- Each participant calls `compute_signature_share`, producing correct `(norm_big_y_i, norm_big_c_i)`.
- Coordinator sums all three; client recovers `msk · H(pk ∥ app_id)`.

**Attack:**
- Participant 2 skips `compute_signature_share` entirely.
- Instead, participant 2 sends `CKDOutput { big_y: G1::generator(), big_c: G1::identity() }` (arbitrary points).
- The coordinator at line 53–55 adds these without any check: [5](#0-4) 

- The resulting `(Y_corrupt, C_corrupt)` is returned as `Some(ckd_output)`.
- The client calls `ckd_output.unmask(app_sk)`: [6](#0-5) 

- The result is `msk · H(pk ∥ app_id) + (G1::identity() - app_sk · G1::generator())` — a wrong key with no error raised.
- All honest parties accept this corrupted output as valid, permanently breaking the CKD for this `app_id`.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
