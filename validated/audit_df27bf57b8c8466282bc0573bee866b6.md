### Title
Gateway Signature Validation Bypass via `skip_stateful_validations` for Invoke Transactions with Nonce=1 - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX feature that skips `__validate__` (signature verification) for invoke transactions submitted alongside a `deploy_account` transaction contains a broken invariant: the skip condition checks only that *any* transaction from the sender's address exists in the mempool, not that a `deploy_account` specifically exists. An unprivileged attacker can exploit this to inject an invoke transaction with a completely invalid signature into the mempool, bypassing the account's `__validate__` entry point entirely.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip signature check) when three conditions hold simultaneously:

1. The incoming transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet committed on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate=false` the function returns `Ok(())` immediately, before the `__validate__` call: [3](#0-2) 

The broken invariant is in condition 4. The code comment claims:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [4](#0-3) 

But `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from that address in the pool — including a `deploy_account` transaction that has not yet been executed (and whose `__validate_deploy__` has therefore not yet run). An attacker who observes a victim's `deploy_account` tx in the public mempool can immediately submit an `Invoke(nonce=1)` with an arbitrary garbage signature for the same address. The gateway will:

1. Pass stateless validation (no signature check there)
2. Pass `validate_state_preconditions` (nonce 1 ≥ account_nonce 0)
3. Pass `validate_by_mempool` (future nonce accepted; `ValidationArgs` carries no signature)
4. Evaluate `skip_stateful_validations` → `true` (deploy_account is in pool)
5. Call `run_validate_entry_point(skip_validate=true)` → `__validate__` is **never called**
6. Admit the transaction to the mempool [5](#0-4) 

### Impact Explanation

**Mempool admission of an invalid (unsigned) transaction.** The attacker's `Invoke(nonce=1)` with a garbage signature is accepted by the gateway and enters the mempool. If the attacker sets a higher tip than the victim's legitimate `Invoke(nonce=1)`, the mempool's fee-escalation logic will evict the victim's valid transaction and replace it with the attacker's invalid one. The attacker's transaction will later fail `__validate__` during block execution and revert, but the victim's transaction has already been permanently removed from the mempool.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The attack window is the period between a user submitting `deploy_account` and that transaction being committed on-chain. This window is publicly observable (mempool is public). The attack requires no privileged access, no special knowledge, and no cryptographic work — only knowledge of the victim's address (derivable from the `deploy_account` transaction itself). The attacker needs to submit a single transaction.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the pool for the sender address. The mempool should expose a `deploy_account_in_pool(address)` query, or the existing check should be narrowed to only match `DeployAccount` transaction types. Alternatively, require that the `deploy_account` transaction hash be provided by the caller (as the `native_blockifier` path does via `deploy_account_tx_hash`) and verify it matches a pending `DeployAccount` in the pool. [6](#0-5) 

### Proof of Concept

```
1. Victim submits RpcDeployAccountTransaction for address A (nonce=0).
   → deploy_account tx enters mempool; account_tx_in_pool_or_recent_block(A) = true.

2. Attacker crafts RpcInvokeTransaction:
     sender_address = A
     nonce          = 1
     signature      = [0xdead, 0xbeef]   // arbitrary garbage
     tip            = victim_tip + 1     // outbid victim

3. Attacker submits to gateway.

4. StatelessTransactionValidator::validate passes (no signature check).

5. StatefulTransactionValidator::extract_state_nonce_and_run_validations:
   - get_nonce_from_state(A) → Nonce(0)   (A not yet on-chain)
   - run_pre_validation_checks:
       validate_state_preconditions: nonce 1 in [0, max_gap] ✓
       validate_by_mempool: ValidationArgs has no signature field ✓
       skip_stateful_validations:
           tx.nonce() == 1 ✓
           account_nonce == 0 ✓
           account_tx_in_pool_or_recent_block(A) == true ✓
           → returns true
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       StatefulValidator::perform_validations → returns Ok(()) immediately
       __validate__ is NEVER called

6. Attacker's tx (invalid signature) is admitted to mempool.
   Victim's Invoke(nonce=1) is evicted by fee escalation.

7. Block execution: attacker's tx fails __validate__, reverts.
   Victim's tx is permanently lost from mempool.
``` [7](#0-6) [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
