### Title
Signature-Skipped Invoke Accepted into Mempool via `skip_stateful_validations` UX Feature — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX feature, designed to let users submit a `deploy_account` + `invoke` pair simultaneously, allows an attacker to inject an invoke transaction with nonce=1 from a victim's not-yet-deployed address **without a valid signature**. The gateway skips the `__validate__` entry point entirely, the mempool accepts the unsigned invoke, and the victim's legitimate invoke is rejected as a duplicate nonce — a mempool admission invariant violation.

---

### Finding Description

The function `skip_stateful_validations` (lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`) implements a UX shortcut: when an invoke transaction has `nonce == 1`, the on-chain account nonce is `0` (account not yet deployed), and `account_tx_in_pool_or_recent_block(sender_address)` returns `true`, the function returns `true` (skip validation). [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs an `AccountTransaction` with `ExecutionFlags { validate: false, … }`: [2](#0-1) 

Inside the blockifier, `validate_tx` short-circuits immediately when `execution_flags.validate == false`, returning `Ok(None)` without ever calling the account's `__validate__` entry point: [3](#0-2) 

The `account_tx_in_pool_or_recent_block` check only verifies that the account has **any** transaction in the mempool pool or state — it does not verify that the transaction is specifically a `deploy_account`, nor does it verify the signature of the incoming invoke: [4](#0-3) 

The mempool's `validate_tx` (called via `validate_by_mempool` before `skip_stateful_validations`) checks for duplicate `tx_hash` and nonce conflicts, but only against already-queued transactions — it does not cryptographically verify the signature: [5](#0-4) 

The stateless validator (`StatelessTransactionValidator::validate`) only checks signature **length**, not cryptographic validity: [6](#0-5) 

When the batcher later executes the transaction, it uses `AccountTransaction::new_for_sequencing` which sets `validate: true`, so `__validate__` **will** run at execution time and the attacker's invoke will be rejected — but by then the victim's legitimate invoke has already been displaced from the mempool. [7](#0-6) 

---

### Impact Explanation

**Broken invariant:** The gateway must never admit an invoke transaction into the mempool without verifying its signature (via `__validate__`).

**Exact corrupted value:** The mempool contains an invoke transaction with an invalid/attacker-controlled signature at nonce=1 for the victim's address. The victim's valid invoke at nonce=1 is rejected with `MempoolError::DuplicateNonce`. [8](#0-7) 

The attacker's invoke fails at block execution time (invalid signature → `__validate__` panics/reverts), but the victim's invoke was already evicted from the mempool. The victim must resubmit after the block is committed. This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The attack requires only:
1. Observing a `deploy_account` transaction in the public mempool (trivially visible).
2. Submitting an invoke with `nonce=1` from the same address with any signature before the victim submits their invoke.
3. No privileged access, no special knowledge of the victim's private key.

The attack window is the interval between the victim's `deploy_account` entering the mempool and the block being committed. During this window, the attacker can occupy the nonce=1 slot, delaying the victim's invoke by at least one block.

---

### Recommendation

The `skip_stateful_validations` path should not bypass signature verification entirely. Options:

1. **Perform a lightweight ECDSA/Stark signature check** on the invoke transaction before skipping `__validate__`, using the public key embedded in the `constructor_calldata` of the pending `deploy_account` (for standard account contracts).
2. **Require the invoke to be submitted in the same RPC call as the `deploy_account`** (atomic pair submission), so the gateway can bind them together and reject third-party invokes for the same slot.
3. **Do not skip `__validate__` at the gateway level.** Instead, accept the nonce gap in the mempool and let the batcher handle ordering — the invoke will naturally execute after the `deploy_account` in the same block.

---

### Proof of Concept

```
1. Victim calls add_tx(deploy_account, address=A, nonce=0, valid_sig).
   → Mempool accepts. account_tx_in_pool_or_recent_block(A) = true.

2. Attacker calls add_tx(invoke, sender=A, nonce=1, sig=[0xdead, 0xbeef]).
   Gateway stateless check: signature length ≤ max → OK.
   Gateway stateful:
     account_nonce = get_nonce(A) = 0  (A not deployed on-chain)
     validate_nonce: 0 ≤ 1 ≤ 0+gap → OK
     validate_by_mempool: no duplicate hash/nonce yet → OK
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → check mempool
       account_tx_in_pool_or_recent_block(A) = true → return true (SKIP)
     run_validate_entry_point(skip_validate=true):
       ExecutionFlags { validate: false }
       validate_tx() → Ok(None)  // __validate__ never called
   → Gateway pushes attacker's invoke to mempool. ACCEPTED.

3. Victim calls add_tx(invoke, sender=A, nonce=1, valid_sig).
   validate_by_mempool → MempoolError::DuplicateNonce { address: A, nonce: 1 }
   → REJECTED.

4. Block built:
   deploy_account(A) executes → A deployed, nonce bumped to 1.
   attacker's invoke(A, nonce=1) executes:
     new_for_sequencing → ExecutionFlags { validate: true }
     __validate__ runs → invalid signature → PanicInValidate / ValidateCairo0Error
     → transaction rejected, no state change, nonce NOT bumped.

5. Victim must resubmit their invoke (delayed ≥1 block).
   Attacker can repeat for every new deploy_account+invoke UX flow.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
