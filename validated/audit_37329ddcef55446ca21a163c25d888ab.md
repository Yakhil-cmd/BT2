### Title
Unauthenticated Fee-Escalation Invoke Bypasses `__validate__` via Overly Broad `skip_stateful_validations` Check - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry-point call for any invoke transaction whose nonce is 1 and whose sender's on-chain nonce is 0, provided `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction in the pool for that address — not exclusively a `deploy_account`. Because the mempool supports fee escalation (replacing an existing same-nonce transaction with a higher-fee one), an attacker can submit a fee-escalating invoke with arbitrary calldata and an invalid signature for any account that is in the `(account_nonce=0, pool_nonce=1)` state, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

**Root cause — `skip_stateful_validations` (lines 429–461)**

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The comment claims the pool-presence check is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second branch is the problem: a future-nonce invoke that was itself admitted via this same skip path is treated as proof of legitimacy for the next caller.

**`account_tx_in_pool_or_recent_block` is type-agnostic**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for any transaction type in the pool — invoke, declare, or deploy_account.

**Fee escalation allows replacement without re-validation**

`validate_by_mempool` (called before `skip_stateful_validations`) only checks nonce ordering and fee escalation thresholds; it does not validate the signature.

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [3](#0-2) 

If the attacker's tip and `max_l2_gas_price` exceed the escalation threshold, `validate_fee_escalation` returns `Ok(Some(existing_tx))` and the existing tx is removed. [4](#0-3) 

**Gateway validation is then skipped**

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
self.run_validate_entry_point(executable_tx, skip_validate).await?;
``` [5](#0-4) 

With `skip_validate = true`, `run_validate_entry_point` sets `validate: false` and returns immediately without calling `__validate__`. [6](#0-5) 

**Batcher executes with `validate: true` — but the legitimate tx is already gone**

`AccountTransaction::new_for_sequencing` always sets `validate: true`:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        only_query: false,
        charge_fee: enforce_fee(&tx, false),
        validate: true,
        strict_nonce_check: true,
    };
    ...
}
``` [7](#0-6) 

So the batcher will call `__validate__` on the attacker's tx. For a standard account the wrong signature causes a revert, but the fee is still charged from the victim's balance and the victim's original invoke is permanently gone from the mempool.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid (unauthorized, wrong-signature) transactions before sequencing.**

Concrete effects:
1. The victim's legitimate nonce-1 invoke is evicted from the mempool and never executed.
2. The attacker's unsigned/mis-signed invoke is admitted and sequenced in its place.
3. When the batcher executes it, `__validate__` reverts the tx, but the account still pays the revert fee — draining the newly-deployed account's balance.
4. For accounts whose `__validate__` is permissive (e.g., multisig with threshold=0, or any account that accepts arbitrary signatures), the attacker's arbitrary calldata executes successfully on behalf of the victim.

### Likelihood Explanation

**Medium.** The precondition — an account with on-chain nonce 0 that has a pending nonce-1 invoke in the mempool — is the exact, common "deploy_account + invoke" UX flow that the feature was designed to support. Any attacker monitoring the public mempool can observe these accounts and submit a fee-escalating replacement before the block is sealed. No privileged access is required; only a higher tip and `max_l2_gas_price` are needed.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender address. For example, expose a `has_pending_deploy_account(address)` query on the mempool that inspects transaction type, and use that in `skip_stateful_validations`. This preserves the UX intent while closing the fee-escalation bypass.

### Proof of Concept

```
1. Victim submits deploy_account(nonce=0) for address A.
   → Mempool: pool = {deploy_account(A, nonce=0)}
   → account_tx_in_pool_or_recent_block(A) = true

2. Victim submits invoke(nonce=1, calldata=legitimate, sig=valid) for address A.
   → Gateway: account_nonce=0, tx.nonce=1
   → validate_by_mempool: passes (no existing nonce-1 tx)
   → skip_stateful_validations: account_tx_in_pool_or_recent_block(A)=true → skip=true
   → __validate__ NOT called
   → Mempool: pool = {deploy_account(A,0), invoke(A,1,legitimate)}

3. Attacker submits invoke(nonce=1, calldata=malicious, sig=garbage,
                           tip > victim_tip * (1 + fee_escalation_pct/100),
                           max_l2_gas_price > victim_price * (1 + fee_escalation_pct/100))
   → Gateway: account_nonce=0, tx.nonce=1
   → validate_by_mempool: validate_fee_escalation passes → victim's tx marked for removal
   → skip_stateful_validations: account_tx_in_pool_or_recent_block(A)=true → skip=true
   → __validate__ NOT called
   → Victim's invoke(A,1,legitimate) removed; attacker's invoke(A,1,malicious) inserted
   → Mempool: pool = {deploy_account(A,0), invoke(A,1,malicious)}

4. Batcher seals block:
   - Executes deploy_account(A,0) → account A deployed
   - Executes invoke(A,1,malicious) with validate=true (new_for_sequencing)
     → __validate__ called → signature invalid → REVERT
     → fee charged from A's balance
     → victim's intended action never runs
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }
