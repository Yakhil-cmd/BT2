### Title
Invoke Transaction with Invalid Signature Bypasses `__validate__` via Overly Broad `skip_stateful_validations` Pool Check - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry-point check for any Invoke transaction with nonce 1 when the account's on-chain nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction type in the pool, not only a `deploy_account`. An attacker can therefore first submit a valid future-nonce invoke (nonce ≥ 2) for a pre-deployed account with nonce 0 to seed the pool, then submit a second invoke with nonce 1 carrying an arbitrary/invalid signature. The gateway skips `__validate__` for the nonce-1 invoke and admits it to the mempool without signature verification.

### Finding Description

`skip_stateful_validations` (lines 429–461) returns `true` (skip `__validate__`) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` **and** `account_tx_in_pool_or_recent_block(sender)` is `true`. [1](#0-0) 

The inline comment justifies condition 3 as: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

The second branch of that reasoning is incorrect. A nonce-2 invoke that passed `__validate__` proves only that the account's `__validate__` accepted **that specific transaction** (its calldata, signature, and nonce). It says nothing about the validity of a distinct nonce-1 invoke with different calldata and a different (possibly forged) signature.

`account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

It returns `true` for **any** transaction type in the pool or in a recently committed block, not exclusively for a `deploy_account`. The `MempoolState::contains_account` helper confirms this: [3](#0-2) 

The stateless validator checks only signature **size**, not cryptographic validity: [4](#0-3) 

`validate_by_mempool` checks only for duplicate hashes and stale nonces, not signatures: [5](#0-4) 

When `skip_validate = true`, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }` and the blockifier's `StatefulValidator::perform_validations` returns early without calling `__validate__`: [6](#0-5) [7](#0-6) 

The existing unit test `test_skip_validate` confirms the flaw: the `should_skip_validation` case passes `contains_tx = true` (any transaction in pool) and asserts that validation is skipped, without distinguishing between a `deploy_account` and a future-nonce invoke: [8](#0-7) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an Invoke transaction with an arbitrary/invalid signature for any pre-deployed account (on-chain nonce = 0) that already has a future-nonce transaction in the pool. The invalid transaction occupies nonce slot 1, blocking the victim's higher-nonce transactions from being sequenced until the invalid transaction is processed and reverted by the blockifier. The attacker can repeat this after each revert to sustain a targeted nonce-slot DoS against specific accounts.

### Likelihood Explanation

**Medium.** The precondition is an account with on-chain nonce 0 that has a future-nonce invoke in the pool. Pre-deployed accounts (system contracts, genesis-deployed wallets, multisigs) satisfy this. The nonce gap allowed by `max_allowed_nonce_gap` (shown as ≥ 10 in tests) makes it straightforward to seed the pool with a valid nonce-2 invoke before submitting the invalid nonce-1 invoke. [9](#0-8) 

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a targeted check that verifies a `deploy_account` transaction specifically exists in the pool for the sender address. Only when a `deploy_account` is pending is it safe to skip `__validate__` for the immediately following nonce-1 invoke, because the account contract does not yet exist on-chain and `__validate__` cannot be called anyway.

Alternatively, restrict the skip to the case where `tx_pool.contains_deploy_account(sender_address)` returns `true`, ensuring the skip is never triggered by a future-nonce invoke that happens to be in the pool.

### Proof of Concept

```
1. Pre-deployed account A exists on-chain with nonce = 0 (e.g., a genesis-deployed multisig).

2. Attacker submits InvokeV3 from A, nonce = 2, valid signature.
   - Passes stateless validation.
   - Passes validate_nonce (0 ≤ 2 ≤ max_allowed_nonce_gap).
   - Passes validate_by_mempool.
   - skip_stateful_validations: nonce ≠ 1 → returns false (do NOT skip).
   - run_validate_entry_point: __validate__ is called → passes (valid signature).
   - Tx admitted to mempool. A is now in tx_pool.

3. Attacker submits InvokeV3 from A, nonce = 1, signature = [Felt::ZERO] (invalid).
   - Passes stateless validation (signature length = 1 ≤ max_signature_length).
   - Passes validate_nonce (0 ≤ 1 ≤ max_allowed_nonce_gap).
   - Passes validate_by_mempool (no duplicate hash, nonce not stale).
   - skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(A) == true  ✓  (nonce-2 tx is in pool)
       → returns true (SKIP __validate__).
   - run_validate_entry_point: skipped entirely.
   - Invalid nonce-1 invoke admitted to mempool without signature verification.

4. A's valid nonce-2 invoke is now stuck behind the invalid nonce-1 invoke.
   When the batcher executes nonce-1, __validate__ fails → tx reverts.
   Attacker repeats step 3 to sustain the DoS.
```

    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())

async fn validate_by_mempool(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,

    let validation_args = ValidationArgs::new(tx, account_nonce);

        .await
        .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
