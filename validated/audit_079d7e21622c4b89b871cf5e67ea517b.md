### Title
Unbounded Loop in `calculate_rewards` Allows Delegator to Permanently Freeze Own Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a delegator's `pool_member_epoch_balance` trace. An unprivileged delegator can grow this trace by one entry per epoch (e.g., by calling `add_to_delegation_pool` with a minimal amount each epoch). After enough epochs without claiming, the loop iteration count grows proportionally, eventually causing `claim_rewards` to exceed Starknet's gas limits and permanently bricking the delegator's unclaimed yield.

### Finding Description

The `calculate_rewards` internal function iterates over every entry in the `pool_member_epoch_balance` trace between the delegator's last reward checkpoint and the current epoch: [1](#0-0) 

The developers themselves acknowledge the risk in a comment at line 858: *"This loop is unbounded but unlikely to exceed gas limits."*

The trace is backed by a `Vec<PoolMemberBalanceCheckpoint>` that grows by appending a new entry whenever `set_member_balance` is called with a key (epoch) different from the last stored key: [2](#0-1) 

`set_member_balance` always uses `current_epoch + K` as the key: [3](#0-2) 

This means every call to a balance-modifying function in a **new epoch** appends a fresh checkpoint. The functions that trigger this are:

- `enter_delegation_pool` → `set_member_balance` [4](#0-3) 
- `add_to_delegation_pool` → `increase_member_balance` → `set_member_balance` [5](#0-4) 
- `exit_delegation_pool_intent` → `set_member_balance` [6](#0-5) 

An attacker calling `add_to_delegation_pool` with `amount = 1` (the minimum non-zero value) once per epoch will append one new trace entry per epoch. The `claim_rewards` function passes `pool_member_info.entry_to_claim_from` as the loop start index, which is only advanced when `claim_rewards` is successfully called: [7](#0-6) 

If the attacker never calls `claim_rewards`, `entry_to_claim_from` stays at 0 and the loop must traverse the entire trace on the next call.

### Impact Explanation

After accumulating enough trace entries (one per epoch), a call to `claim_rewards` will iterate over all of them in a single transaction. Once the trace is large enough that the loop's gas cost exceeds Starknet's per-transaction gas limit, `claim_rewards` will always revert. The delegator's accrued rewards become permanently inaccessible — a **permanent freezing of unclaimed yield**.

This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

### Likelihood Explanation

The attack requires only that the delegator call `add_to_delegation_pool(pool_member, 1)` once per epoch. The minimum non-zero amount is 1 token unit. The cost is one transaction per epoch. There are no privileged roles, no external dependencies, and no special preconditions beyond holding a minimal delegation. The attacker controls the timing entirely. The protocol's own comment acknowledges the loop is unbounded.

### Recommendation

1. **Cap the trace size**: Enforce a maximum number of entries in `pool_member_epoch_balance` per pool member (e.g., `K + 2` entries, since only the last few epochs are relevant for balance lookups).
2. **Checkpoint on claim**: Require that `claim_rewards` is called at least once every N epochs, or automatically advance `entry_to_claim_from` during balance-modifying operations so the loop never needs to traverse more than a bounded window.
3. **Batch-claim with pagination**: Allow `claim_rewards` to accept a `max_iterations` parameter so rewards can be claimed in bounded chunks.

### Proof of Concept

1. Delegator calls `enter_delegation_pool(reward_address, min_amount)` in epoch `E`. Trace: 1 entry at epoch `E + K`.
2. Each subsequent epoch `E+i`, delegator calls `add_to_delegation_pool(self, 1)`. Trace grows: 1 new entry per epoch.
3. After `N` epochs without calling `claim_rewards`, the trace has `N + 1` entries and `entry_to_claim_from = 0`.
4. Delegator calls `claim_rewards(pool_member)`. The loop in `calculate_rewards` iterates over all `N + 1` entries.
5. For sufficiently large `N`, the transaction exceeds the gas limit and reverts.
6. Every subsequent `claim_rewards` call also reverts. The delegator's accumulated rewards are permanently frozen. [8](#0-7)

            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(

                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
