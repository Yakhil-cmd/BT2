### Title
Truncating Division in `native_per_pubdata` Computation Allows Zero-Cost Pubdata Generation — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, the `native_per_pubdata` rate is computed using truncating integer division (`wrapping_div`), while the analogous `native_per_gas` rate uses ceiling division (`div_ceil`). When `pubdata_price < native_price`, the truncation produces `native_per_pubdata = 0`, causing every pubdata-charging call downstream to charge zero native resources — regardless of how much pubdata the transaction actually generates.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two per-unit native rates are computed as follows:

**`native_per_gas`** (line 135) — uses `div_ceil`, rounding in the protocol's favor:
```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

**`native_per_pubdata`** (line 142) — uses `wrapping_div`, truncating toward zero:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the expression `pubdata_price.wrapping_div(native_price)` evaluates to `0`. The resulting `native_per_pubdata = 0` is then threaded through the entire fee pipeline:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx` (line 352):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // = 0 * intrinsic_pubdata = 0
   ```
   No native is deducted upfront for intrinsic pubdata. [2](#0-1) 

2. **Post-execution pubdata check** via `get_resources_to_charge_for_pubdata` (line 430–432):
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata);
   // = pubdata_bytes * 0 = 0
   ```
   The check `check_enough_resources_for_pubdata` always passes, and zero native is charged regardless of actual pubdata consumed. [3](#0-2) 

3. **Refund calculation** in `compute_gas_refund` (line 69–72):
   ```rust
   let delta_gas = if native_per_gas == 0 { 0 } else {
       (native_used / native_per_gas) as i64 - (gas_used as i64)
   };
   ```
   There is no analogous guard for `native_per_pubdata == 0`; the native cost of pubdata is simply invisible to the gas accounting. [4](#0-3) 

---

### Impact Explanation

`native` resources in ZKsync OS represent proving/computational cost. Pubdata (state diffs written to L1) is a primary driver of proving cost and L1 data cost. When `native_per_pubdata = 0`, a transaction can perform an unbounded number of storage writes — each generating pubdata — without consuming any native resource budget. This:

- Allows any L2 transaction sender to generate disproportionate pubdata relative to the gas/native fees paid.
- Undermines the native resource accounting model that is supposed to bound proving cost per block.
- Can be used to bloat state diffs, increasing L1 data costs and prover load without commensurate fee payment — a resource accounting bypass with direct protocol-level financial impact.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are independent block-level parameters sourced from `system.get_pubdata_price()` and `system.get_native_price()`. [5](#0-4)  There is no enforced invariant in the bootloader that `pubdata_price >= native_price`. Any block configuration where `pubdata_price < native_price` — which is a realistic market condition when pubdata is cheap relative to native computation — silently sets `native_per_pubdata = 0` for every transaction in that block. All unprivileged L2 transaction senders in such a block can exploit this without any special access.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` computation, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, protocol-favorable, matches native_per_gas pattern):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This ensures that when `pubdata_price > 0`, at least 1 native unit is charged per pubdata byte, preventing the silent zero-cost bypass. [6](#0-5) 

---

### Proof of Concept

**Setup:** Block configured with `native_price = 10`, `pubdata_price = 5` (a realistic scenario where pubdata is cheaper than native computation).

**Step 1:** `native_per_pubdata` computation:
```
pubdata_price.wrapping_div(native_price) = 5 / 10 = 0  (integer truncation)
native_per_pubdata = 0
```

**Step 2:** Attacker submits an L2 transaction that performs many `SSTORE` operations (e.g., writing to 1000 distinct storage slots), generating ~33,000 bytes of pubdata.

**Step 3:** In `get_resources_to_charge_for_pubdata`:
```
native = 33_000 * native_per_pubdata = 33_000 * 0 = 0
```
`check_enough_resources_for_pubdata` returns `true` (enough resources), charging 0 native.

**Step 4:** In `create_resources_for_tx`, the intrinsic pubdata overhead is also 0:
```
intrinsic_pubdata_overhead = 0 * L2_TX_INTRINSIC_PUBDATA = 0
```

**Result:** The attacker pays only the EVM gas cost for the `SSTORE` opcodes, but zero native resource cost for the pubdata generated — bypassing the pubdata fee mechanism entirely. The contrast with `native_per_gas` (which uses `div_ceil` and would yield `ceil(gas_price / native_price) >= 1`) makes the inconsistency and its exploitability clear. [7](#0-6) [8](#0-7)
