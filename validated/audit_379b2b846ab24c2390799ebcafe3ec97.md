### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` — (`fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over caller-supplied `signedData` but does not enforce that `signedData` encodes the target `evmAddress`. Because the signature is bound only to the raw data bytes and not to the specific COA being proved, a signature produced for one COA's ownership proof is cryptographically valid for any other COA owned by the same Cadence account. An unprivileged attacker who has observed a prior ownership-proof signature from a victim can replay it against a different COA of that same account, causing `isValidSignature` (ERC-1271) to return `ValidERC1271Signature` for a COA the attacker does not control.

### Finding Description

**Root cause — `fvm/evm/stdlib/contract.cdc` lines 1082–1086:**

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // ← caller-supplied; evmAddress NOT required here
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The function signature accepts `evmAddress` as a separate parameter and checks it only *after* the cryptographic verification succeeds (lines 1095–1109). The signed payload is never required to contain `evmAddress`. The code comment at lines 1003–1009 explicitly acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

**Precompile entry path — `fvm/evm/precompiles/arch.go` lines 219–243 (`DecodeABIEncodedProof`):**

```go
return types.NewCOAOwnershipProofInContext(
    hash,
    types.Address(caller),   // EVMAddress = address(this) from the EVM call
    encodedProof,
)
```

The `EVMAddress` used in the Cadence call is taken from the EVM `address` argument (i.e., `address(this)` in `coa.sol`), not from the proof bytes. The proof bytes (`encodedProof`) carry only the Cadence `Address`, `CapabilityPath`, `KeyIndices`, and `Signatures`.

**ERC-1271 entry point — `fvm/evm/handler/coa/coa.sol` line 118:**

```solidity
cadenceArch.staticcall(abi.encodeWithSignature(
    "verifyCOAOwnershipProof(address,bytes32,bytes)",
    address(this), _hash, _sig));
```

**Attack path:**

1. Victim holds a Cadence account with two COAs: `COA_A` published at `/public/coaA` and `COA_B` published at `/public/coaB`.
2. Victim legitimately signs data `D` to prove ownership of `COA_A`. The resulting proof `(cadenceAddr, /public/coaA, keyIndices, sigs_over_D)` is observed on-chain or off-chain.
3. Attacker constructs a new proof `(cadenceAddr, /public/coaB, keyIndices, sigs_over_D)` — identical signatures, different path.
4. Attacker submits an EVM transaction calling `COA_B.isValidSignature(D, crafted_proof)`.
5. The precompile calls `validateCOAOwnershipProof(cadenceAddr, /public/coaB, D, keyIndices, sigs_over_D, COA_B_address)`.
6. `keyList.verify(signedData: D, ...)` succeeds — the signatures are valid over `D` regardless of which COA is targeted.
7. The COA capability at `/public/coaB` is borrowed and its address matches `COA_B_address` — check passes.
8. `isValidSignature` returns `0x1626ba7e` (`ValidERC1271Signature`) for `COA_B`, which the victim never authorized.

### Impact Explanation

An attacker can make any ERC-1271-aware EVM contract accept a forged signature on behalf of a victim's `COA_B`. Concrete consequences include:

- **Unauthorized token approvals / permit signatures**: ERC-20 `permit`, ERC-721 `permit`, and similar flows that rely on ERC-1271 for smart-contract wallets will accept the forged proof, allowing the attacker to drain tokens from `COA_B` or authorize arbitrary EVM operations.
- **Unauthorized DAO / multisig votes**: Any on-chain governance or multisig that uses ERC-1271 to count COA votes will count the attacker's replayed proof as a valid vote from the victim.
- **Cross-protocol authorization bypass**: Any protocol that calls `isValidSignature` to gate privileged actions (e.g., bridge withdrawals, lending collateral releases) will grant those actions to the attacker.

The precondition — a victim with two COAs under one Cadence account — is not enforced against by the protocol. The `EVM.createCadenceOwnedAccount()` function can be called multiple times, and the `validateCOAOwnershipProof` function is `access(all)`, callable by any Cadence script or transaction.

### Likelihood Explanation

- The protocol does not prevent a Cadence account from holding multiple COAs.
- `validateCOAOwnershipProof` is `access(all)` and callable from any Cadence script or EVM transaction.
- The attacker only needs to observe one prior ownership-proof signature from the victim (e.g., from a previous ERC-1271 call, a bridge interaction, or an off-chain authentication flow).
- No privileged access, leaked keys, or staked node control is required.
- The code comment itself acknowledges the replay risk, confirming the design gap is known.

### Recommendation

Enforce that `signedData` encodes `evmAddress` inside `validateCOAOwnershipProof`, analogous to the PayrollManager fix of including `safeAddress` in the signed digest:

```cadence
// Before verifying, assert that signedData encodes evmAddress
// (e.g., the last 20 bytes of signedData must equal evmAddress)
// OR enforce this at the precompile layer by hashing (evmAddress ++ callerSuppliedData)
// before passing it as signedData.
```

Alternatively, at the Go precompile layer (`DecodeABIEncodedProof`), construct `signedData` as `keccak256(evmAddress ++ hash)` so the signed payload is always bound to the specific COA address, making cross-COA replay cryptographically impossible.

### Proof of Concept

```
Precondition:
  - Cadence account 0xALICE has COA_A at /public/coaA (EVM addr 0xEEE1)
                              and COA_B at /public/coaB (EVM addr 0xEEE2)

Step 1 – Alice legitimately proves ownership of COA_A for data D:
  proof_A = {
    address:        0xALICE,
    capabilityPath: /public/coaA,
    keyIndices:     [0],
    signatures:     [sig_alice_over_D],   // sig over D with "FLOW-V0.0-user" tag
  }
  COA_A.isValidSignature(D, encode(proof_A))  → ValidERC1271Signature  ✓ (legitimate)

Step 2 – Attacker replays with COA_B path, same signatures:
  proof_B = {
    address:        0xALICE,
    capabilityPath: /public/coaB,         // ← changed path
    keyIndices:     [0],
    signatures:     [sig_alice_over_D],   // ← same signatures, never signed for COA_B
  }
  COA_B.isValidSignature(D, encode(proof_B))

Step 3 – Precompile decodes:
  EVMAddress = address(COA_B) = 0xEEE2   (from address(this))
  signedData = D
  Calls validateCOAOwnershipProof(0xALICE, /public/coaB, D, [0], [sig_alice_over_D], 0xEEE2)

Step 4 – Cadence verification:
  keyList.verify(signedData: D, ...)      → true  (sig_alice_over_D is valid over D)
  borrow COA at /public/coaB → address = 0xEEE2 == evmAddress  → true

Step 5 – Result:
  isValidSignature returns 0x1626ba7e (ValidERC1271Signature) for COA_B
  Alice never authorized this.
```

```go
func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
```
