### Title
Direct Token Transfer to `RewardSupplier` Suppresses L1 Mint Requests, Causing Temporary Freeze of Reward Claims - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `request_funds` internal function in `RewardSupplier` reads the contract's live ERC20 token balance via `balance_of(get_contract_address())` to decide whether to send L1 mint requests. Because any address can directly transfer STRK tokens to the `RewardSupplier` contract, an attacker can inflate this balance reading, suppressing legitimate L1 mint requests. As rewards are subsequently distributed and the donated tokens are consumed, the contract's actual balance eventually falls below `unclaimed_rewards`, causing `claim_rewards` to revert and temporarily freezing reward distribution for all stakers and delegators.

---

### Finding Description

In `request_funds` (called from `update_unclaimed_rewards_from_staking_contract`), the contract computes a `credit` value as:

```
credit = balance_of(reward_supplier_address) + l1_pending_requested_amount
``` [1](#0-0) 

The `balance` term is read directly from the ERC20 token contract at call time. This is the exact same pattern as the referenced vulnerability: a live `balance_of` read that can be manipulated by a direct token transfer.

If `credit >= debit + threshold`, no L1 mint request is sent and `l1_pending_requested_amount` is not updated: [2](#0-1) 

The `claim_rewards` function later performs a `checked_transfer` to the staking contract. If the actual token balance is insufficient (because L1 minting was suppressed), this transfer reverts: [3](#0-2) 

The `unclaimed_rewards` storage variable is a separate accounting counter that grows independently of the actual balance. The `assert!(amount <= unclaimed_rewards)` check passes, but the subsequent `checked_transfer` fails when the real balance is depleted.

---

### Impact Explanation

An attacker who donates a sufficiently large amount of STRK directly to the `RewardSupplier` contract inflates the `balance` reading in `request_funds`. This suppresses L1 mint requests for a period proportional to the donated amount divided by the reward rate. Once the donated tokens are consumed by normal reward claims, the contract's balance falls below `unclaimed_rewards`. All subsequent calls to `claim_rewards` (invoked by the staking contract during reward distribution) revert, temporarily freezing reward claims for every staker and delegator in the system.

The freeze persists until a new `update_unclaimed_rewards_from_staking_contract` call detects the deficit and sends a new L1 mint request, and then until the L1→L2 bridge message is processed — a delay that can span hours to days.

**Impact category**: Temporary freezing of unclaimed yield (High).

---

### Likelihood Explanation

The attack requires only a direct ERC20 transfer to a known contract address — no privileged role, no special access, and no protocol interaction beyond holding STRK tokens. The attacker loses the donated tokens (they are consumed as rewards), making this a griefing attack. The cost scales with the reward rate and `base_mint_amount`, but for a protocol with significant TVL the cost may be acceptable to a motivated attacker. Likelihood: **Medium**.

---

### Recommendation

Replace the live `balance_of` read in `request_funds` with a tracked internal balance variable that is updated only through controlled entry points (`on_receive`, `claim_rewards`). This eliminates the ability to manipulate the credit calculation via direct transfers:

```cairo
// Instead of:
let balance = token_dispatcher.balance_of(account: get_contract_address());

// Use an internal tracked balance:
let balance = self.tracked_balance.read();
```

Alternatively, compute `credit` purely from `l1_pending_requested_amount` without reading the live token balance, since the live balance is already implicitly accounted for by the `unclaimed_rewards` counter and the pending L1 amount.

---

### Proof of Concept

```
// Setup: RewardSupplier has balance=0, unclaimed_rewards=R, l1_pending=0
// Normally: credit=0 < R+threshold → L1 mint request sent

// Attack step 1: Attacker directly transfers D STRK to RewardSupplier
STRK.transfer(reward_supplier_address, D)

// Attack step 2: Staking contract calls update_unclaimed_rewards_from_staking_contract(rewards)
// Inside request_funds:
//   balance = D  (inflated by donation)
//   credit  = D + 0 = D
//   debit   = R + rewards
// If D >= debit + threshold → no L1 mint request sent

// Attack step 3: Normal reward claims consume the donated D tokens over time
// balance approaches 0, unclaimed_rewards keeps growing

// Attack step 4: claim_rewards(amount) is called by staking contract
//   assert!(amount <= unclaimed_rewards)  ← passes (accounting is fine)
//   checked_transfer(staking_contract, amount)  ← REVERTS (balance < amount)

// Result: All staker/delegator reward claims are frozen until L1 bridge delivers new tokens
```

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
