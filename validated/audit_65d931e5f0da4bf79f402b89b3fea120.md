### Title
`native_per_pubdata` Truncates to Zero via Floor Division When `pubdata_price < native_price`, Eliminating All Native Resource Charges for Pubdata - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

In `validate_and_compute_fee_for_transaction`, `native_per_pubdata` is computed using `wrapping_div` (floor/truncating division). When the operator-set `pubdata_price` is less than `native_price` (e.g., `pubdata_price = 2`, `native_price = 100`), the result truncates to zero. This is directly analogous to the AMM bug where `_a = (x * y) / 1e18 = 0` when `x*y < 1e18` collapses the invariant `k` to zero. With `native_per_pubdata = 0`, every downstream pubdata cost calculation returns zero native resources, making pubdata completely free regardless of how much state a transaction writes. Any unprivileged transaction sender can exploit this to write unbounded pubdata at zero native cost, forcing the operator to bear L1 data publication costs without compensation.

### Finding Description

**Root cause — floor division truncates to zero:**

In `validation_impl.rs` line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `0`. This is inconsistent with how `native_per_gas` is computed on line 135, which correctly uses `div_ceil`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

The same floor-division bug appears in `api/src/helpers.rs` line 427:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

**Propagation — zero `native_per_pubdata` nullifies all pubdata cost checks:**

`native_per_pubdata = 0` flows into three critical paths:

1. **`create_resources_for_tx`** (gas_helpers.rs line 352): `intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata) = 0`. No native is deducted for intrinsic pubdata at transaction setup.

2. **`get_resources_to_charge_for_pubdata`** (gas_helpers.rs line 430–432): `native = current_pubdata_spent.checked_mul(0) = 0`. Returns zero native resources regardless of actual pubdata written.

3. **`check_enough_resources_for_pubdata`** (gas_helpers.rs line 454): `resources.has_enough(&zero_resources)` always returns `true`. The post-execution pubdata sufficiency check always passes.

**Concrete trigger condition:**

The condition `pubdata_price < native_price` is realistic and observable in production configurations. The test at `tests/instances/transactions/src/lib.rs:1236` uses `native_price = 100, pubdata_price = 2`, which silently produces `native_per_pubdata = 0`. The operator intends to charge for pubdata (price is non-zero) but the rounding error eliminates the charge entirely.

### Impact Explanation

When `native_per_pubdata = 0`:

- Any transaction sender can write the maximum number of storage slots permitted by their EVM gas limit, generating hundreds to thousands of pubdata bytes, at zero native resource cost.
- The operator publishes this data to L1 and pays the associated DA costs without receiving any native resource compensation from the transaction.
- Repeated across many transactions in a block, this constitutes a sustained financial drain on the operator: the attacker pays only EVM gas (which is bounded by the block gas limit) while the operator absorbs unbounded L1 data costs.
- There is no minimum `native_per_pubdata` guard analogous to the AMM's `MINIMUM_K` check.

**Severity: High** — direct financial loss to the operator/protocol, reachable by any unprivileged transaction sender whenever `pubdata_price < native_price`.

### Likelihood Explanation

The condition `pubdata_price < native_price` is not an edge case. The default `BlockContext` in the test rig sets `pubdata_price = U256::ZERO` and `native_price = U256::from(10)`, meaning the default configuration already produces `native_per_pubdata = 0`. The Ethereum block runner sets `pubdata_price: U256::ZERO` unconditionally. Any production deployment where the operator sets a non-zero but small `pubdata_price` relative to `native_price` (a plausible pricing configuration) silently enables this exploit for all transactions in every affected block.

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to zero when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, never zero unless pubdata_price is zero):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, consider adding a minimum `native_per_pubdata` guard (analogous to `MINIMUM_K` in the AMM fix) when `pubdata_price > 0` to ensure the operator always receives at least 1 native unit per pubdata byte when pubdata pricing is enabled.

### Proof of Concept

**Trigger condition:** Set `native_price = 100`, `pubdata_price = 2` (operator intends to charge for pubdata, but `2 / 100 = 0` via floor division).

**Attacker transaction:** Submit an EIP-1559 transaction calling a contract that writes 20 storage slots (each slot write generates ~64 bytes of pubdata = ~1280 bytes total):

```solidity
// Contract that writes 20 storage slots
mapping(uint256 => uint256) s;
fallback() external {
    for (uint256 i = 0; i < 20; i++) {
        s[i] = 0xdeadbeef;
    }
}
```

**Expected behavior:** With `pubdata_price = 2` and `native_price = 100`, the operator intends `native_per_pubdata = ceil(2/100) = 1`. For 1280 bytes of pubdata, the transaction should consume 1280 native units from the native resource budget.

**Actual behavior:** `native_per_pubdata = 2 / 100 = 0` (floor division). `get_resources_to_charge_for_pubdata` returns 0 native. `check_enough_resources_for_pubdata` returns `true`. The transaction succeeds consuming zero native resources for 1280 bytes of pubdata. The operator publishes 1280 bytes to L1 at their own cost.

This is confirmed by the existing test at `tests/instances/transactions/src/lib.rs:1236` which uses exactly `native_price = 100, pubdata_price = 2` and expects the transaction to succeed — it does, but silently with `native_per_pubdata = 0`.
