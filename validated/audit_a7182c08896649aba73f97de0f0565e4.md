### Title
`update_rewards()` Liveness DoS: Unprivileged Caller Consumes Per-Block Reward Slot, Permanently Freezing Unclaimed Yield — (File: `src/staking/staking.cairo`)

---

### Summary

`StakingRewardsManagerImpl::update_rewards` in `src/staking/staking.cairo` uses a single global per-block gate (`last_reward_block`) to prevent double-distribution of rewards. The function has **no caller access control**, allowing any unprivileged address to call it with `disable_rewards: true`, consuming the block's reward slot without distributing any rewards. The legitimate sequencer's subsequent call for the same block then unconditionally reverts with `REWARDS_ALREADY_UPDATED`, permanently losing that block's rewards for all stakers.

---

### Finding Description

`update_rewards` at `src/staking/staking.cairo` lines 1449–1500 enforces a per-block gate using a single global storage variable `last_reward_block`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause state
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validity checks ...
    self.last_reward_block.write(current_block_number);   // ← gate consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // ← no rewards distributed
    }
    // ... reward distribution ...
```

**Root cause**: `last_reward_block` is written unconditionally before the `disable_rewards` branch. Any caller who passes the staker-validity checks can write `last_reward_block = current_block_number` and return early, leaving the slot consumed but rewards unissued.

**No access control**: The spec states "Only starkware sequencer" for this function, but the code enforces no such restriction. `general_prerequisites()` only checks the pause flag — the spec's own error list for `update_rewards` contains `CONTRACT_IS_PAUSED`, `REWARDS_ALREADY_UPDATED`, `STAKER_NOT_EXISTS`, and `INVALID_STAKER`, with no sequencer-only error. Flow tests call `update_rewards` directly without any caller spoofing, confirming the function is open to any address.

**Global gate, not per-staker**: `last_reward_block` is a single contract-wide `BlockNumber` field. One successful call per block — regardless of which staker is named — blocks all further calls for that block.

**Attack path**:
1. Attacker observes any active staker address (public on-chain).
2. In every block, attacker front-runs the sequencer by calling `update_rewards(valid_staker, disable_rewards: true)`.
3. The call passes all checks, writes `last_reward_block = current_block_number`, and returns without distributing rewards.
4. The sequencer's call for the intended staker reverts with `REWARDS_ALREADY_UPDATED`.
5. That block's rewards are permanently lost for all stakers.
6. Repeated every block → all stakers earn zero rewards indefinitely.

---

### Impact Explanation

Each missed block permanently loses the block reward for all stakers and their delegators. The `last_reward_block` gate is strict (`>`), so there is no catch-up mechanism: a consumed slot cannot be reclaimed in a later block. An attacker who front-runs the sequencer every block causes **permanent freezing of all unclaimed yield** across the entire protocol.

This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- The function is callable by any address with no on-chain access control.
- The attacker only needs one valid staker address, which is trivially obtained from on-chain events (`NewStaker`).
- The cost is one transaction per block; on Starknet, transaction fees are low.
- The attacker can automate front-running with a simple bot watching the mempool or block production.
- No special privileges, leaked keys, or external dependencies are required.

---

### Recommendation

Add an explicit sequencer-only (or operator-only) access check at the top of `update_rewards`, consistent with the spec's stated intent:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.roles.only_sequencer();   // enforce the spec's "Only starkware sequencer"
    self.general_prerequisites();
    // ...
```

Alternatively, if a dedicated sequencer role is not available, consider separating the `last_reward_block` write from the early-return path so that a `disable_rewards: true` call does **not** consume the block slot unless rewards were actually due.

---

### Proof of Concept

```
1. Deploy the system with one staker (staker_address = S).
2. Advance K epochs so the staker has non-zero balance.
3. Enable consensus rewards.
4. In block B, attacker calls:
       update_rewards(staker_address: S, disable_rewards: true)
   → last_reward_block is written to B; function returns early; no rewards issued.
5. Sequencer calls:
       update_rewards(staker_address: S, disable_rewards: false)
   → assert!(B > B) fails → REWARDS_ALREADY_UPDATED panic.
6. staker_claim_rewards(S) returns 0.
7. Repeat steps 4–6 for every block → staker accumulates zero rewards forever.
```
