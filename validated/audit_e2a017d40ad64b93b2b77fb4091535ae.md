### Title
COA Ownership Proof Signature Replay Across Different COA Addresses - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced by a Flow account key for one COA can be replayed against a different COA owned by the same Cadence account, causing the proof to validate as legitimate for the wrong EVM address. The code itself acknowledges this gap in a comment but treats it as low-risk. The ERC-1271 surface (`COA.isValidSignature`) makes this directly reachable by any unprivileged EVM caller.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies that the provided `signatures` are valid over `signedData` using the Cadence account keys at `address`, then checks that the COA resource at `path` has an EVM address matching `evmAddress`. However, the function **never checks that `signedData` encodes `evmAddress`**:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // ← no enforcement that evmAddress is in here
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The code comment at line 1003–1009 explicitly acknowledges this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The EVM-side entry point is `COA.isValidSignature` in `fvm/evm/handler/coa/coa.sol`:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig
        )
    );
```

Here `_hash` is the EVM-side message hash (e.g., a Permit2 hash, an EIP-712 struct hash, etc.) and `_sig` is the RLP-encoded `COAOwnershipProof`. The `_hash` is passed as `signedData` to `validateCOAOwnershipProof`. Because `signedData` is not required to contain the COA's EVM address, a signature produced for COA-A (owned by Flow account F) is equally valid when submitted to COA-B (also owned by Flow account F), as long as the same Flow key signed the same `signedData`.

The `verifyCOAOwnershipProof` precompile in `fvm/evm/precompiles/arch.go` passes `address(this)` (the calling COA) as the `evmAddress` argument, so the EVM-side path is safe in isolation. But `EVM.validateCOAOwnershipProof` is also a public `access(all)` Cadence function callable directly from any Cadence script or transaction, where the caller fully controls all arguments including `evmAddress`. An attacker can supply a signature obtained for COA-A and pass COA-B's address as `evmAddress`, and the function will return `isValid: true` if the same Flow account key signed the same `signedData` and COA-B's capability is at the supplied path.

---

### Impact Explanation

Any Cadence script or EVM contract that uses `EVM.validateCOAOwnershipProof` or `COA.isValidSignature` as an authentication gate can be bypassed via cross-COA replay. If a Flow account controls multiple COAs (e.g., one per application), a signature obtained legitimately for one COA can be replayed to impersonate ownership of another COA. EVM protocols that rely on ERC-1271 (`isValidSignature`) for authorization — such as Permit2, EIP-712 typed-data approvals, or multisig schemes — can be tricked into accepting a signature that was never intended for the target COA address.

---

### Likelihood Explanation

The precondition is that a single Flow Cadence account controls more than one COA, which is explicitly supported by the protocol (there is no one-COA-per-account limit). The comment in the code itself acknowledges the replay scenario. Any unprivileged EVM transaction sender or Cadence script author can trigger this by calling `COA.isValidSignature` or `EVM.validateCOAOwnershipProof` directly with a recycled proof. No privileged access is required.

---

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` — for example, by hashing `evmAddress || signedData` before passing to `keyList.verify`, or by requiring callers to include the EVM address in the signed payload and verifying it inside the function. This mirrors the ERC-7739 defensive rehashing recommendation from the original report: bind the signature to the specific target address so it cannot be replayed against a different one.

---

### Proof of Concept

1. Flow account `F` creates two COAs: `COA-A` at `/public/coaA` and `COA-B` at `/public/coaB`.
2. A legitimate verifier asks `F` to prove ownership of `COA-A` by signing `signedData = hash("some_message")` with key index 0.
3. `F` produces `proofA = COAOwnershipProof{KeyIndices:[0], Address:F, CapabilityPath:"coaA", Signatures:[sig]}`.
4. An attacker constructs `proofB = COAOwnershipProof{KeyIndices:[0], Address:F, CapabilityPath:"coaB", Signatures:[sig]}` — identical signature, different path.
5. The attacker calls `EVM.validateCOAOwnershipProof(address: F, path: /public/coaB, signedData: hash("some_message"), keyIndices: [0], signatures: [sig], evmAddress: COA-B_address)`.
6. `keyList.verify` succeeds because `sig` is valid over `signedData` with key 0 of account `F`.
7. The COA address check at line 1095–1105 passes because `COA-B_address` matches the COA stored at `/public/coaB`.
