### Title
`skip_stateful_validations` Bypasses Signature Verification for Deployed Accounts with Nonce 0 — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the `__validate__` entry-point call for the deploy-account + invoke UX pattern. However, its guard condition is too broad: it fires for **any** deployed account whose on-chain nonce is still 0 and that already has any transaction in the mempool, not only for accounts whose deploy-account is pending. An attacker who controls such an account can submit an invoke transaction with a completely invalid signature, have it accepted into the mempool without signature verification, and repeat this indefinitely at near-zero cost.

### Finding Description

`skip_stateful_validations` (lines 429–461) returns `true` — meaning "skip `__validate__`" — when all three conditions hold:

1. The transaction is an `Invoke`.
2. The transaction nonce equals `Nonce(Felt::ONE)`.
3. The on-chain account nonce equals `Nonce(Felt::ZERO)`.
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

The code comment claims this is safe because the mempool containing a transaction for the account implies "either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is the problem: a **deployed** account with on-chain nonce 0 that has a valid nonce-0 invoke already in the mempool satisfies all four conditions. `account_tx_in_pool_or_recent_block` checks only whether the address appears in the pool or the committed-block state — it does not distinguish deploy-account transactions from ordinary invokes. [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false` and calls the blockifier validator without ever invoking `__validate__`: [3](#0-2) 

The full call chain is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce + resource bounds)
       ├─ validate_by_mempool            (duplicate / gap check)
       └─ skip_stateful_validations      ← returns true (BUG)
  └─ run_validate_entry_point(skip_validate=true)  ← __validate__ never called
``` [4](#0-3) 

### Impact Explanation

The gateway accepts an invoke transaction with an arbitrary (e.g., all-zero) signature into the mempool without any cryptographic verification. The transaction will later be rejected by the batcher when `__validate__` is called during execution, so no state is corrupted. However:

- The mempool is polluted with unverifiable transactions.
- The batcher wastes execution resources on transactions that will always fail `__validate__`.
- The attacker pays no fees for the rejected transactions (Starknet does not charge fees when `__validate__` fails during execution).
- The attack can be repeated continuously as long as the attacker keeps a valid nonce-0 invoke in the mempool.

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

### Likelihood Explanation

The preconditions are easy to satisfy: any deployed account whose first transaction (nonce 0) is still pending in the mempool is vulnerable. A single attacker-controlled account is sufficient. No privileged access is required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **deploy-account** transaction is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pending transaction for the address. Alternatively, restrict the skip condition to addresses that have no deployed class hash in state (i.e., the account truly does not exist yet), which is the only case where skipping `__validate__` is semantically correct.

### Proof of Concept

1. Deploy account `A` (on-chain nonce becomes 0, class hash set).
2. Submit a valid invoke from `A` with nonce 0 and a correct signature → accepted into mempool. Now `account_tx_in_pool_or_recent_block(A)` returns `true`.
3. Submit an invoke from `A` with nonce 1 and a completely invalid signature (e.g., `[0x0, 0x0]`).
4. Gateway evaluation of `skip_stateful_validations`:
   - `tx.nonce() == Nonce(1)` ✓
   - `account_nonce == Nonce(0)` ✓ (on-chain state not yet updated)
   - `account_tx_in_pool_or_recent_block(A)` → `true` ✓
   - Returns `true` → `__validate__` is skipped.
5. The invalid-signature invoke passes all gateway checks and is inserted into the mempool.
6. Batcher picks up both transactions; nonce-0 executes successfully; nonce-1 fails `__validate__` and is rejected — no fees charged to the attacker.
7. Repeat step 3 indefinitely to flood the mempool. [5](#0-4) [2](#0-1) [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
