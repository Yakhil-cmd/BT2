### Title
Invoke transactions with invalid signatures bypass gateway signature validation via overly broad `skip_stateful_validations` for accounts with nonce=0 in state — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function skips the validate entry point (which performs signature verification) for invoke transactions with nonce=1 when the account's on-chain nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. The check is designed for the UX case where a user sends a `deploy_account` and an `invoke` simultaneously before the account exists on-chain. However, `account_tx_in_pool_or_recent_block` returns `true` not only when a `deploy_account` is pending, but also when the account **already exists in state** with nonce=0 and has any transaction in the mempool or a recent block. An attacker can exploit this to submit invoke transactions with invalid signatures for any recently-deployed account, bypassing gateway signature validation entirely and admitting invalid transactions into the mempool.

---

### Finding Description

In `skip_stateful_validations` (lines 429–461 of `stateful_transaction_validator.rs`), the gateway skips the validate entry point when all three conditions hold:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The account's current on-chain nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs `ExecutionFlags { validate: false, … }`, meaning the account's `__validate__` entry point — which verifies the transaction signature — is **never called** at the gateway level. [2](#0-1) 

The code comment at line 436 states: *"the account nonce is zero, meaning the account was not deployed yet."* This assumption is **incorrect**. An account nonce of 0 in state can mean either:

- **Case A (intended):** The account does not yet exist on-chain; a `deploy_account` is pending in the mempool. Skipping validation is correct here because there is no deployed contract to call `__validate__` on.
- **Case B (unintended):** The account **already exists** on-chain with nonce=0 (e.g., just deployed in a recent block) and has any transaction in the mempool or a recent block. The account **can** validate transactions, but the gateway skips validation anyway.

The `account_tx_in_pool_or_recent_block` check does not distinguish between these two cases. Its own comment acknowledges it returns `true` for *"either a deploy_account transaction OR transactions with future nonces that passed validations"* — the second branch explicitly covers already-deployed accounts. [3](#0-2) 

**Concrete attack path:**

1. Account X is deployed via `deploy_account` (included in a recent block). Its on-chain nonce is now 0.
2. `account_tx_in_pool_or_recent_block(X)` returns `true` because the `deploy_account` appears in a recent block.
3. Attacker submits an `Invoke` for account X with `nonce=1` and a **completely invalid signature** (e.g., all zeros).
4. Gateway nonce check passes: `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap`.
5. `skip_stateful_validations` returns `true` (conditions 1–3 all satisfied).
6. `run_validate_entry_point` is called with `validate: false` — the `__validate__` entry point is **not invoked**.
7. The invalid transaction is forwarded to the mempool and admitted.

During block execution, `new_for_sequencing` always sets `validate: true`, so the `__validate__` entry point is called and the transaction fails. However, the transaction has already been **admitted to the mempool** with an invalid signature, violating the gateway's admission invariant. [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
