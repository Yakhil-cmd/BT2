### Title
Gateway Admission Accepts Invoke Transactions with Invalid Signatures via `skip_stateful_validations` Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator uses `account_tx_in_pool_or_recent_block` as a mutable, non-cryptographic condition to skip the `__validate__` entry point (signature verification) for invoke transactions with nonce=1. An unprivileged attacker who observes a pending `deploy_account` transaction in the mempool for account X can submit an invoke transaction with nonce=1 and an **invalid signature** for account X. The gateway accepts it without running `__validate__`, admitting an unauthorized transaction into the mempool.

---

### Finding Description

**The skip condition** in `skip_stateful_validations`:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
```

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `validate: !skip_validate = false`:

```rust
let strict_nonce_check = false;
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations` for an Invoke transaction:

```rust
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());   // ← returns here; __validate__ is never called
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
```

`perform_pre_validation_stage` runs the nonce check (with `strict_nonce_check = false`, so `account_nonce(0) <= tx_nonce(1)` passes) and the fee/balance check, but **the account's `__validate__` entry point — which verifies the ECDSA signature — is never executed**.

**The security gate is mutable and non-cryptographic.** `account_tx_in_pool_or_recent_block` returns `true` if:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`contains_account` checks the `staged` or `committed` maps, and `tx_pool.contains_account` checks whether any transaction for the address is in the pool. This is analogous to the external report's balance-based nonce: it is a mutable, non-monotonic condition used as a security gate, not a cryptographic proof of authorization.

**Attack path:**

1. Legitimate user submits `deploy_account` for address X (valid signature, nonce=0). The transaction enters the mempool. `tx_pool.contains_account(X)` is now `true`.
2. Attacker observes the pending `deploy_account` via P2P propagation.
3. Attacker crafts `invoke(sender=X, nonce=1, calldata=<arbitrary>, signature=<invalid>)` with a fee high enough to pass `verify_can_pay_committed_bounds` (account X is typically pre-funded before deployment).
4. Gateway evaluates: `tx.nonce()==1` ✓, `account_nonce==0` ✓, `account_tx_in_pool_or_recent_block(X)==true` ✓ → `skip_validate=true`.
5. `run_validate_entry_point` is called; `__validate__` is **not** called; the transaction is accepted.
6. The invalid invoke enters the mempool alongside the legitimate `deploy_account`.
7. When the batcher executes the block, `deploy_account` runs first (nonce=0), then the invalid invoke is attempted. The blockifier calls `__validate__` with `strict_nonce_check=true`; the signature fails; the transaction is rejected at execution time — but it has already consumed mempool capacity and batcher resources.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering (`tx_nonce >= account_nonce`), not signature validity. A different-hash transaction (different signature bytes) passes the duplicate check.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid (unauthorized) transactions before sequencing.**

The gateway's invariant — "every admitted transaction must carry a valid account signature" — is broken for the `deploy_account + invoke` UX path. An attacker can:

- Flood the mempool with signature-invalid invoke transactions for any account that has a pending `deploy_account` and a non-zero pre-funded balance.
- Force the batcher to speculatively execute and then discard these transactions, wasting block-building resources.
- Displace legitimate transactions via fee-escalation replacement (the attacker submits a higher-fee invalid transaction, evicting the legitimate nonce=1 invoke).

---

### Likelihood Explanation

**Medium.** The attacker requires:
1. Visibility into pending `deploy_account` transactions (available via P2P propagation in a public sequencer).
2. The target account to be pre-funded (standard practice; accounts must hold STRK before deployment to pay fees).
3. Submission of the invalid invoke before the `deploy_account` is committed (a window of one or more blocks).

No privileged access, no special keys, and no on-chain state manipulation are required.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, not merely the address presence. This closes the gap between the intended invariant ("skip validation only when a deploy_account is pending") and the implemented check ("skip validation when any transaction is pending").

---

### Proof of Concept

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn handle_nonce(
        state: &mut dyn State,
        tx_info: &TransactionInfo,
        strict: bool,
    ) -> TransactionPreValidationResult<()> {
        if tx_info.is_v0() {
            return Ok(());
        }

        let address = tx_info.sender_address();
        let account_nonce = state.get_nonce_at(address)?;
        let incoming_tx_nonce = tx_info.nonce();
        let valid_nonce = if strict {
            account_nonce == incoming_tx_nonce
        } else {
            account_nonce <= incoming_tx_nonce
        };
        if valid_nonce {
            return Ok(state.increment_nonce(address)?);
        }
        Err(TransactionPreValidationError::InvalidNonce {
            address,
            account_nonce,
            incoming_tx_nonce,
        })
    }
