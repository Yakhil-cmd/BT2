### Title
Staker Can Front-Run Reward Distribution by Instantly Increasing Commission via `set_commission_commitment` - (File: `src/staking/staking.cairo`)

---

### Summary

A staker who has set a `commission_commitment` can instantly increase their commission rate to any value up to `max_commission` (including 100%) with no delay or timelock. By front-running a reward distribution call (`update_rewards_from_attestation_contract` or `update_rewards`), the staker can steal the delegators' share of rewards, then immediately lower the commission back to attract more delegators.

---

### Finding Description

The `set_commission_commitment` function allows a staker to set a `CommissionCommitment` with a `max_commission` up to 10000 (100%) and an `expiration_epoch` as close as `current_epoch + 1`. [1](#0-0) 

While an active commitment exists, `update_commission` permits the staker to set commission to **any value** up to `max_commission` — including values **higher** than the current commission — with no delay: [2](#0-1) 

The protocol itself acknowledges this gap in a developer note: [3](#0-2) 

At reward distribution time, `calculate_staker_pools_rewards` reads the commission from storage at that exact moment: [4](#0-3) 

The `split_rewards_with_commission` utility then applies the current commission to split rewards between the staker and the pool: [5](#0-4) 

---

### Impact Explanation

A staker can steal 100% of delegators' unclaimed yield for any reward distribution event. With `commission = 10000` (100%), `commission_rewards = pool_rewards_including_commission` and `pool_rewards = 0`, meaning delegators receive nothing for that epoch. This is **theft of unclaimed yield** — a High-severity impact.

---

### Likelihood Explanation

Any staker with a pool can execute this attack. The steps are:
1. Call `set_commission_commitment(max_commission=10000, expiration_epoch=current_epoch+1)` — no special privilege required.
2. Monitor the mempool for `update_rewards_from_attestation_contract` or `update_rewards` calls.
3. Front-run with `set_commission(10000)`.
4. After reward distribution, call `set_commission(low_value)` to restore the attractive rate.

On Starknet, sequencer ordering is controlled, but the staker controls their own operational address and can time transactions within the same block. The attack is repeatable every epoch.

---

### Recommendation

Introduce a minimum delay (e.g., one full epoch) between when a commission increase via `set_commission` takes effect and when it applies to reward calculations. Alternatively, apply commission changes only starting from `current_epoch + K` (matching the balance checkpoint delay), so delegators have time to exit before a higher commission takes effect.

---

### Proof of Concept

```
// Staker sets low commission to attract delegators
staking.set_commission(commission: 100); // 1%

// Delegators enter pool...

// Staker sets commitment allowing up to 100% commission
staking.set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1);

// Staker sees update_rewards_from_attestation_contract about to be called
// Front-runs it:
staking.set_commission(commission: 10000); // 100%

// Attestation contract calls update_rewards_from_attestation_contract
// -> calculate_staker_pools_rewards reads commission = 10000
// -> split_rewards_with_commission: commission_rewards = all pool rewards, pool_rewards = 0
// -> Delegators receive 0 rewards for this epoch

// Staker resets commission to attract more delegators
staking.set_commission(commission: 100); // 1% again
```

The `calculate_staker_pools_rewards` function reads `commission` from live storage at line 1964, so the front-run commission value is used directly. [6](#0-5)

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));

pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}
