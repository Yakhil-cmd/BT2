### Title
Malicious Participant Can Permanently Abort EdDSA Signing Sessions Due to Disabled Cheater Detection - (File: `src/frost/eddsa/sign.rs`)

### Summary
In `do_sign_coordinator_v1` and `do_sign_coordinator_v2`, the coordinator collects signature shares from every participant in the session and calls `aggregate()` with an intentionally empty `verifying_shares` map, disabling per-share cheater detection. A single malicious participant can send a crafted invalid signature share, causing `aggregate()` to fail and permanently aborting the signing session. Because individual shares are never verified, the coordinator cannot identify the culprit, and the attacker can repeat the abort in every subsequent session.

### Finding Description
Both coordinator paths in `src/frost/eddsa/sign.rs` construct a `PublicKeyPackage` with an empty `BTreeMap` for `verifying_shares` and pass it directly to `aggregate()`:

```rust
// do_sign_coordinator_v1, lines 163-165
let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
```

The same pattern appears in `do_sign_coordinator_v2` at lines 218–220. The inline comment acknowledges the trade-off:

```
// We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
```

Prior to this call, `recv_from_others` collects one share from **every** participant in the session (not merely a threshold subset):

```rust
// lines 150-153
for (from, signature_share) in recv_from_others(&chan, r2_wait_point, &participants, me).await?
{
    signature_shares.insert(from.to_identifier()?, signature_share);
}
```
