### Title
Missing Caller Validation in `interop_root_reporter_event_hook` Allows Unauthorized Interop Root Injection - (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

### Summary

The `interop_root_reporter_event_hook` event hook, which is the OS-level mechanism that commits interop roots into the system's `InteropRootStorage`, performs no check on who triggered the event. The service-transaction whitelist restricts what *service transactions* may call, but it does not prevent regular EVM transactions from calling `L2_INTEROP_ROOT_STORAGE_ADDRESS` directly. Because the hook fires on the event signature alone, any transaction that causes `L2_INTEROP_ROOT_STORAGE_ADDRESS` to emit `InteropRootAdded` will have its root unconditionally committed to the system state.

### Finding Description

**Intended restriction path:**
Service transactions are parsed in `ServiceTx::decode_list_body`, which enforces `SERVICE_DESTINATION_WHITELIST`. Only `(L2_INTEROP_ROOT_STORAGE_ADDRESS, ADD_INTEROP_ROOTS_IN_BATCH_SELECTOR)` is whitelisted for interop-root import. This whitelist is a restriction *on service transactions* — it says service transactions may only call those destinations. [1](#0-0) 

**The bypass path:**
Regular EVM transactions are not subject to `SERVICE_DESTINATION_WHITELIST` at all. A regular transaction can call `L2_INTEROP_ROOT_STORAGE_ADDRESS` with `addInteropRootsInBatch` calldata. If the deployed contract at that address does not enforce its own access control, the call succeeds and `InteropRootAdded` is emitted.

**The missing OS-level check:**
`interop_root_reporter_event_hook` only validates the event signature and ABI layout. It does not check the transaction type, the caller of the contract, or any authorization flag. The `_caller_ee` parameter is explicitly unused (prefixed `_`): [2](#0-1) 

Upon matching the signature, the hook unconditionally calls `system.io.add_interop_root(...)`: [3](#0-2) 

**Contrast with call hooks that do check callers:**
`l1_messenger_hook` explicitly rejects any caller that is not `L1_MESSENGER_ADDRESS`: [4](#0-3) 

`set_bytecode_on_address_hook` rejects callers that are neither `CONTRACT_DEPLOYER_ADDRESS` nor `COMPLEX_UPGRADER_ADDRESS`: [5](#0-4) 

The event hook for interop roots has no equivalent guard.

**`InteropRootStorage.push_root` also performs no validation:** [6](#0-5) 

Once the hook fires, the root is committed with no further checks.

### Impact Explanation

An attacker who can cause `L2_INTEROP_ROOT_STORAGE_ADDRESS` to emit `InteropRootAdded` via a regular transaction (i.e., if the contract lacks access control or has a callable path that does not enforce it) can inject an arbitrary `(chain_id, block_or_batch_number, root)` tuple into the system's interop root storage. This root is then used by the interop system to validate cross-chain messages. A fabricated root could allow the attacker to claim cross-chain messages that never occurred on the source chain, enabling theft of bridged assets or unauthorized state transitions. The impact is equivalent to the Sentiment scenario: the OS-level accounting (interop root storage) includes roots that bypassed the intended restriction path.

### Likelihood Explanation

The service transaction whitelist is enforced only for service-typed transactions. Any regular L2 transaction can target `L2_INTEROP_ROOT_STORAGE_ADDRESS`. The exploitability depends on whether the deployed contract enforces its own access control. System contracts in ZKsync are sometimes designed with the assumption that OS-level restrictions are sufficient; if `addInteropRootsInBatch` in the contract does not check `msg.sender` (or checks it against a role that can be obtained), the full attack path is open to any unprivileged sender. Even if the current contract is safe, the OS hook provides zero defense-in-depth: a future upgrade or a contract bug immediately becomes exploitable.

### Recommendation

Add an explicit transaction-type or caller check inside `interop_root_reporter_event_hook`, mirroring the pattern used by call hooks:

1. Pass the transaction type (service vs. regular) into the event hook context, or expose a flag on the system indicating the current transaction is a service transaction.
2. In `interop_root_reporter_event_hook`, return an `internal_error` (or silently ignore) if the current transaction is not a service transaction.
3. Alternatively, register the event hook only for the specific service-transaction execution context, so it cannot fire during regular transaction processing.

This ensures that even if the contract's own access control is bypassed, the OS-level hook will not commit the root.

### Proof of Concept

1. Deploy (or use the existing) `L2_INTEROP_ROOT_STORAGE_ADDRESS` contract in a block where `addInteropRootsInBatch` has no caller restriction.
2. Submit a regular EIP-1559 transaction (not a service transaction) calling `L2_INTEROP_ROOT_STORAGE_ADDRESS` with `addInteropRootsInBatch` calldata containing an attacker-chosen `(chain_id, block_or_batch_number, root)`.
3. The contract emits `InteropRootAdded(chain_id, block_or_batch_number, [root])`.
4. `interop_root_reporter_event_hook` fires (registered at `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW`), passes all format checks, and calls `system.io.add_interop_root(...)`. [7](#0-6) 

5. The fabricated root is now in `InteropRootStorage` and will be included in the block's rolling hash, indistinguishable from a legitimately imported root.

        L2_INTEROP_ROOT_STORAGE_ADDRESS,

    ),

fn whitelisted(to: B160, data: &[u8]) -> bool {
    let selector: [u8; 4] = match data.get(..4).and_then(|bytes| bytes.try_into().ok()) {
        Some(selector) => selector,
        None => return false,
    };
    SERVICE_DESTINATION_WHITELIST.contains(&(to, selector))
}
