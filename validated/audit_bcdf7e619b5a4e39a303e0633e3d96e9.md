### Title
Gateway Signature Bypass via `skip_stateful_validations` Admits Unsigned Invoke Transactions Through Fee Escalation - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point for any Invoke transaction with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true`. Because that predicate returns `true` for **any** transaction from the account in the pool — not specifically a `deploy_account` — an attacker can submit an Invoke with an invalid signature and a higher fee, have it admitted to the mempool via fee escalation, and displace the legitimate Invoke, all without the gateway ever calling `__validate__`.

---

### Finding Description

**Invariant from the external report translated to the sequencer:**
The external bug's invariant is: *state must be cleared before an external call that can re-enter*. The sequencer analog is: *the `__validate__` entry point must be called before a transaction is admitted to the mempool, unless the skip condition is provably safe*. The skip condition here is not provably safe.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function skips `__validate__` when three conditions hold simultaneously:
1. The transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The code comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

That assumption is **circular and incorrect**. The first Invoke with `nonce=1` submitted alongside a `deploy_account` **also** skips `__validate__` (because the `deploy_account` is already in the pool when it arrives). So the pool can contain an Invoke with `nonce=1` that was never validated. Any subsequent Invoke with `nonce=1` and a higher fee will also skip `__validate__`, because `account_tx_in_pool_or_recent_block` returns `true` for the unvalidated Invoke already in the pool.

**`account_tx_in_pool_or_recent_block` is not deploy-account-specific:** [2](#0-1) 

It returns `true` if the account has **any** transaction in `tx_pool` or in the mempool's committed/staged state — not specifically a `deploy_account`.

**`validate_by_mempool` allows fee-escalation replacement before `skip_stateful_validations` is checked:** [3](#0-2) 

The call order is:
1. `validate_state_preconditions` — nonce range check only.
2. `validate_by_mempool` — calls `mempool.validate_tx`, which includes `validate_fee_escalation`. A higher-fee Invoke with the same nonce passes here.
3. `skip_stateful_validations` — returns `true` (skip `__validate__`) because the account is in the pool. [4](#0-3) 

**`run_validate_entry_point` propagates the skip flag directly:** [5](#0-4) 

When `skip_validate=true`, `execution_flags.validate=false`, so `validate_tx` returns `Ok(None)` immediately without executing the account's `__validate__` entry point. [6](#0-5) 

**Attack flow:**

1. User submits `deploy_account` for address A (passes full validation, enters mempool).
2. User submits `Invoke(nonce=1)` for A with a valid signature. Gateway: `account_tx_in_pool_or_recent_block=true` → `skip_stateful_validations=true` → `__validate__` skipped → admitted.
3. Attacker observes both transactions in the public mempool.
4. Attacker submits `Invoke(nonce=1)` for A with an **invalid signature** but a **higher fee** than step 2.
5. Gateway: `validate_by_mempool` passes (fee escalation allows replacement). `skip_stateful_validations` returns `true` (account in pool). `__validate__` is **not called**. Invalid Invoke is admitted, replacing the legitimate one.
6. Batcher executes: `deploy_account` succeeds; invalid Invoke is executed, `__validate__` fails at the blockifier level, Invoke is reverted.
7. The user's legitimate Invoke is permanently displaced from the mempool.

---

### Impact Explanation

This is a **High** impact finding matching the allowed scope: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

The gateway admits an Invoke transaction whose signature would fail `__validate__` at execution time. The concrete consequence is that an attacker can censor any user's first post-deploy Invoke by replacing it with an invalid transaction via fee escalation. The user's transaction is permanently removed from the mempool; the attacker's transaction is reverted at execution. The attacker pays only the fee for a reverted transaction to achieve censorship.

---

### Likelihood Explanation

- The `deploy_account` and its paired Invoke are publicly visible in the mempool.
- The attacker only needs to submit a higher-fee Invoke with an arbitrary (invalid) signature.
- No privileged access, no special contract, no ERC-777-style callback is required.
- The attack is repeatable: every time the user resubmits the Invoke, the attacker can displace it again.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is present in the pool for the sender address. The mempool should expose a dedicated predicate such as `deploy_account_in_pool(address) -> bool` that inspects transaction types, rather than returning `true` for any transaction type.

Alternatively, instead of skipping `__validate__` entirely, run it against a speculative state that includes the pending `deploy_account` (i.e., simulate the post-deploy state), so the signature is still verified.

---

### Proof of Concept

```
// Precondition: account A not yet deployed (on-chain nonce = 0).

// Step 1: User submits deploy_account for A (valid, passes full validation).
POST /add_transaction { deploy_account, sender: A, nonce: 0, valid_sig }
// → admitted, account A now in mempool pool

// Step 2: User submits legitimate invoke (nonce=1, valid sig).

// Gateway: account_tx_in_pool_or_recent_block(A) = true
//          skip_stateful_validations = true → __validate__ skipped → admitted

POST /add_transaction { invoke, sender: A, nonce: 1, fee: 200, INVALID_sig }
// Gateway: validate_by_mempool passes (fee escalation: 200 > 100)
//          skip_stateful_validations = true (A still in pool) → __validate__ skipped
//          → admitted, replaces user's legitimate invoke

// Result: user's invoke is gone; attacker's invalid invoke is reverted at execution.
// User must resubmit; attacker can repeat indefinitely.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
