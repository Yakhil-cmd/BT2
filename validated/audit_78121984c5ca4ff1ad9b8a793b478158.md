Audit Report

## Title
Unbounded Synchronous Scan in `create_sns_neuron_recipes` Can Permanently Stick SNS Swap Finalization Lock - (`File: rs/sns/swap/src/swap.rs`)

## Summary

`finalize_swap` acquires a lock (`finalize_swap_in_progress = true`) before crossing multiple `await` boundaries, committing the lock to stable state. After those boundaries, `create_sns_neuron_recipes` is called synchronously with two unbounded loops over all buyers and all Neurons' Fund participants, with no instruction-limit guard. If the IC instruction budget is exhausted during this synchronous call, the current message traps and rolls back only its own state changes — the lock, committed in a prior message, remains set. All subsequent `finalize_swap` calls return immediately with an error, leaving the swap permanently stuck in `Lifecycle::Committed` until a canister upgrade clears the lock.

## Finding Description

**Lock acquisition and commitment:**

`finalize` acquires the lock synchronously at line 1506, before any `await`: [1](#0-0) 

`lock_finalize_swap` sets `finalize_swap_in_progress = Some(true)` in canister heap state: [2](#0-1) 

`finalize_inner` then crosses two `await` boundaries before calling `create_sns_neuron_recipes`: [3](#0-2) 

On the IC, each `await` is a message boundary. After the first `await` at line 1558 (`sweep_icp`), the lock is committed to stable state and cannot be rolled back by any trap in a subsequent message.

**Unbounded synchronous scan:**

`create_sns_neuron_recipes` (called synchronously at line 1588) contains two nested loops with no instruction-limit guard:

- Loop 1: iterates over all entries in `self.buyers` (up to `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` total neuron recipes): [4](#0-3) 

- Loop 2: iterates over all `cf_participants` × their `cf_neurons` (up to `MAX_NEURONS_FUND_PARTICIPANTS = 5,000` NF neurons): [5](#0-4) 

A grep over the entire swap source confirms zero `is_over_instructions_limit` calls anywhere in `swap.rs`.

**Participant scale bounds:**

`MAX_NEURONS_FUND_PARTICIPANTS` is 5,000: [6](#0-5) 

The integration test confirms the combined ceiling: [7](#0-6) 

This ceiling is a **memory bound**, not an instruction-count bound. At maximum scale, `create_sns_neuron_recipes` must process `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS + MAX_SNS_NEURONS_PER_BASKET × MAX_NEURONS_FUND_PARTICIPANTS` recipe-creation operations (hashing, struct allocation, `Vec::extend`) in a single synchronous message execution.

**Lock-stuck acknowledgment in code:**

The code itself acknowledges the lock-stuck risk but only for panics, not instruction-limit traps: [8](#0-7) 

A second unbounded synchronous scan exists in `claim_swap_neurons` (called after three prior `await`s), which builds a full `claimable_neurons_index` over all `neuron_recipes` before its first `await`: [9](#0-8) 

**Why existing checks are insufficient:**

The `has_created_neuron_recipes` idempotency flag (lines 843, 915) only skips already-processed entries on retry — it does not prevent instruction exhaustion on the first call. The participant cap at line 1181 bounds memory, not instruction cost. There is no chunked-processing or instruction-limit check anywhere in the synchronous path. [10](#0-9) 

## Impact Explanation

If `create_sns_neuron_recipes` traps due to instruction exhaustion:

1. The IC rolls back only the current message's state changes.
2. The `finalize_swap_in_progress = true` lock, committed across a prior `await`, is not rolled back.
3. Every subsequent call to `finalize_swap` hits the lock check at line 1506 and returns immediately with `"The Swap canister has finalize_swap call already in progress"`.
4. The swap is permanently stuck in `Lifecycle::Committed` with the lock held.
5. Direct participants cannot receive their SNS neurons; ICP already swept to SNS governance cannot be reclaimed without a canister upgrade.

This matches the allowed impact: **High — Application/platform-level DoS with concrete user and protocol harm** (SNS swap funds and neuron distribution permanently blocked, recovery requiring privileged canister upgrade intervention).

## Likelihood Explanation

- `finalize_swap` is a public update call with no access control; any principal can invoke it.
- The trigger condition is a swap near `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` direct participants with significant NF participation — a realistic scenario for high-profile SNS launches.
- No attacker action is required beyond calling `finalize_swap` after the swap reaches `Lifecycle::Committed`; the trap is caused by the scale of legitimate participant data, not by any injected input.
- The IC's per-message instruction limit (40 billion instructions for update calls) is a hard ceiling; at maximum participant counts, the combined cost of hashing, struct allocation, and `Vec::extend` across 100,000+ recipe-creation operations is plausibly sufficient to exhaust it, and no benchmarks exist to prove otherwise.

## Recommendation

1. Add an `is_over_instructions_limit()` check inside both loops in `create_sns_neuron_recipes`, persisting partial progress via the existing `has_created_neuron_recipes` idempotency flag, and return a partial `SweepResult` so the caller can retry across multiple messages.
2. Apply the same chunked-processing pattern to the `claim_swap_neurons` pre-loop (lines 1675–1697), splitting the index-building step across messages if needed.
3. Implement a post-upgrade hook that unconditionally clears `finalize_swap_in_progress` — the comment at line 1528 acknowledges this need but it is not implemented.
4. Add an instruction-count benchmark for `create_sns_neuron_recipes` at maximum participant counts, analogous to the existing memory-bound test in `constraints_dependencies.rs`.

## Proof of Concept

1. Launch an SNS swap with `min_participant_icp_e8s` at minimum and `neuron_basket_construction_parameters.count` set to a large value (e.g., 10).
2. Fill the swap to near `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` direct participants and ensure maximum NF participation (`MAX_NEURONS_FUND_PARTICIPANTS = 5,000`).
3. Allow the swap to reach `Lifecycle::Committed`.
4. Call `finalize_swap` from any principal.
5. `sweep_icp` completes (first `await`) → lock is committed to stable state.
6. `create_sns_neuron_recipes` is called synchronously over ~100,000+ recipe-creation operations → IC instruction limit trap.
7. Observe: `finalize_swap_in_progress = true` persists in canister state; all subsequent `finalize_swap` calls return `"The Swap canister has finalize_swap call already in progress"`.
8. Swap is permanently stuck; participant funds cannot be distributed without a canister upgrade to clear the lock.

A deterministic integration test can reproduce this using PocketIC with a mocked environment that simulates instruction exhaustion mid-loop, verifying that `is_finalize_swap_locked()` returns `true` after the trap while `Lifecycle` remains `Committed`.

    pub fn lock_finalize_swap(&mut self) -> Result<(), String> {
        match self.is_finalize_swap_locked() {
            true => Err("The Swap canister has finalize_swap call already in progress".to_string()),
            false => {
                self.finalize_swap_in_progress = Some(true);
                Ok(())
            }
        }
