### Title
`recreated` Account Balance Permanently Destroyed on Commit Due to Missing Recreated-Account Path — (`fvm/evm/emulator/state/stateDB.go`, `fvm/evm/emulator/state/delta.go`)

---

### Summary

`DeltaView.CreateAccount` correctly sets both `toBeDestructed[A]` (via an internal `SelfDestruct` call) and `created[A]` when deploying to a pre-existing address, and records the pre-destruction balance in `balances[A]`. However, `StateDB.Commit` has no code path for the "destructed AND recreated" case: it checks `HasSelfDestructed`, calls `DeleteAccount`, sets `deleted = true`, and then unconditionally `continue`s — permanently skipping the `IsCreated` branch. The recreated account is never written to the base view, and the carried-over balance is silently discarded.

---

### Finding Description

**Step 1 — `DeltaView.CreateAccount` on a pre-existing account**

When the EVM executes a CREATE2 targeting address A that already exists in the base ledger with balance X: [1](#0-0) 

- `HasSelfDestructed(A)` → false, so the `!destructed` branch runs.
- `d.SelfDestruct(A)` is called → `toBeDestructed[A] = X`, `balances[A] = 0`.
- Immediately after, `balances[A] = X` (balance carried over — the intended legacy behaviour).
- `recreated[A]` and `created[A]` are both set.

After this call the delta view simultaneously has `toBeDestructed[A]` set **and** `created[A]` set.

**Step 2 — `StateDB.Commit` ignores the recreated flag** [2](#0-1) 

The commit loop:
1. Calls `db.HasSelfDestructed(A)` → **true** (because `toBeDestructed[A]` is set).
2. Calls `db.baseView.DeleteAccount(A)` — removes A from persistent storage.
3. Sets `deleted = true`.
4. `if deleted { continue }` — **unconditionally skips** the `IsCreated` branch.

`db.IsCreated(A)` would return true (line 495), but it is never reached. `db.GetBalance(A)` would return X (line 490), but it is never reached either. The recreated account is never written to the base view. Balance X is permanently lost.

The `recreated` map is populated in `DeltaView` but is **never consulted anywhere in `StateDB.Commit`**.

---

### Impact Explanation

Any EVM balance held at address A before a CREATE2 deployment to that address is silently and permanently destroyed at commit time. The execution layer reports success (no error is returned), but the base view ends up with A deleted and zero balance — violating the EVM invariant that deploying to a pre-funded address must carry the balance into the new contract. This constitutes an irreversible, attacker-triggerable loss of on-chain EVM funds.

---

### Likelihood Explanation

The trigger requires only a standard EVM transaction: pre-fund a deterministic CREATE2 address (by sending ETH to it), then deploy a contract to that address in a subsequent transaction. Both steps are fully attacker-controlled through the normal Flow Access API transaction-submission path. No privileged node role, leaked key, or social engineering is required.

---

### Recommendation

In `StateDB.Commit`, after calling `DeleteAccount`, check whether the address was also recreated before issuing the unconditional `continue`:

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    if err != nil { return nil, wrapError(err) }
    err = updateCommitter.DeleteAccount(addr)
    if err != nil { return nil, wrapError(err) }
    deleted = true
}
// Allow the IsCreated path to run for recreated accounts
if deleted && !db.IsCreated(addr) {
    continue
}
```

This mirrors how Geth's `statedb.go` handles the same case: it finalises the deletion of old storage but still writes the new account state when the address was recreated in the same transaction.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract Deployer {
    // 1. Pre-fund the CREATE2 target address off-chain (send 1 ETH to it).
    // 2. Call deploy(); the contract is created at the pre-funded address.
    // 3. After the tx, query the balance of the deployed contract — it is 0.
    //    The 1 ETH has been permanently destroyed.
    function deploy(bytes32 salt, bytes memory initCode) external {
        address target;
        assembly {
            target := create2(0, add(initCode, 0x20), mload(initCode), salt)
        }
        require(target != address(0), "deploy failed");
        // target.balance == 0 despite 1 ETH being pre-funded
    }
}
```

Run against the Flow EVM emulator / localnet:
1. Compute `target = keccak256(0xff ++ deployer ++ salt ++ keccak256(initCode))[12:]`.
2. Send 1 ETH to `target` (it becomes a plain balance-holding address in the base view).
3. Call `Deployer.deploy(salt, initCode)`.
4. Assert `target.balance == 0` post-commit — the 1 ETH is gone.
