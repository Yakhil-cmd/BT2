### Title
`setPoolAdminFeeDestination` Redirects Accumulated Fees Without Prior Collection, Causing Loss of Owed Admin Fees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accumulated fees to the old destination. Every other fee-parameter mutator in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before writing new state. The missing flush means all fees that accrued under the old destination — both notional fees tracked in `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` and spread-fee surplus — are silently redirected to the new destination on the next `collectPoolFees` call.

### Finding Description

Two fee types accumulate inside `MetricOmmPool` between collection events:

1. **Notional fees** — explicitly tracked in `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled`, incremented on every swap.
2. **Spread fees** — held as surplus: `balance * SCALE_MULTIPLIER − binTotals.scaledToken − notionalFeeScaled`.

Both are distributed to `adminFeeDestination_` (the address passed into `collectFees`) proportionally to the admin fee share.

`setPoolAdminFees` and `setPoolProtocolFee` both call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before writing any new state:

```solidity
// setPoolAdminFees — lines 418-425
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6,
    c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8,
    c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination receives accrued fees
);
```

`setPoolAdminFeeDestination` does **not**:

```solidity
// lines 438-447
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

After this call, `collectPoolFees` (callable by anyone) reads the updated mapping and passes the **new** address to `collectFees`, sending all previously accrued fees to the new destination.

The frontrunning vector mirrors the external report: the admin can observe a pending `collectPoolFees` transaction in the mempool and front-run it with `setPoolAdminFeeDestination`, atomically redirecting the entire accumulated balance to an address they control before the collection settles.

### Impact Explanation

All admin-share fees that accrued under the old destination are permanently redirected to the new destination. The old destination — which may be a DAO treasury, a multisig, or a third-party beneficiary — receives nothing for the period it was the legitimate recipient. The loss is bounded by the total admin-share of fees accumulated since the last collection, which can be substantial on a high-volume pool.

### Likelihood Explanation

The trigger is a single `onlyPoolAdmin` call with no timelock. The admin is semi-trusted; the design pattern established by `setPoolAdminFees` and `setPoolProtocolFee` (collect-before-update) shows the intent is that accumulated fees should always be settled to the current destination before any state change. The missing flush is a deviation from that invariant, reachable on every destination change.

### Recommendation

Apply the same collect-before-update pattern used by `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the current destination before switching.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool accumulates notional fees (`notionalFeeToken0Scaled > 0`) and spread surplus over N swaps.
2. Old admin fee destination is address A (e.g., a DAO treasury).
3. Pool admin calls `setPoolAdminFeeDestination(pool, addressB)`.
   - No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is now address B.
4. Anyone (or the admin) calls `collectPoolFees(pool)`.
   - Factory reads `poolAdminFeeDestination[pool]` → address B.
   - All accumulated admin-share fees are transferred to address B.
5. Address A receives zero, despite being the legitimate recipient for the entire accrual period.

The admin can also front-run a pending `collectPoolFees` mempool transaction with step 3, atomically stealing the pending payout.
