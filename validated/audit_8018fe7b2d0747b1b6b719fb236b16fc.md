### Title
Malicious Participant Can Corrupt CKD Output by Using Mismatched `app_id` or `app_pk` - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
In the Confidential Key Derivation (CKD) protocol, the coordinator accumulates cryptographic contributions from all participants without verifying that each participant used the agreed-upon `app_id` and `app_pk`. A malicious participant can compute their share using a different `app_id` or `app_pk`, corrupting the final CKD output so that honest parties accept an unusable or incorrect derived key.

### Finding Description
The CKD protocol is a one-round protocol where each participant independently computes a contribution `(norm_big_y, norm_big_c)` using their private share, the application identifier `app_id`, and the application public key `app_pk`. The coordinator collects these contributions and sums them.

In `compute_signature_share` (lines 148–182 of `src/confidential_key_derivation/protocol.rs`), each participant computes:

```
hash_point = H(public_key || app_id)
big_s      = hash_point * private_share_i
big_c      = big_s + app_pk * y_i
```

The coordinator then accumulates in `do_ckd_coordinator` (lines 50–55):

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no protocol-level mechanism** to verify that each participant's contribution was computed using the same `app_id` and `app_pk` as the coordinator and honest participants. Each participant supplies these values locally as function arguments to `ckd(...)`. A malicious participant simply passes a different `app_id'` or `app_pk'` to their local invocation, causing their `hash_point'` and `big_c'` to be computed in a different context. The coordinator blindly adds this mismatched contribution to the running sum.

This is structurally identical to the external report's root cause: a shared accumulated value (`endRoundId` / the CKD sum) is not scoped to the correct identifier (`symbol` / `app_id`), and a caller can inject a contribution computed under a different identifier to corrupt the result.

### Impact Explanation
The final CKD output `(Y, C)` is supposed to satisfy:

```
C = msk · H(pk, app_id) + Y · app_sk   (ElGamal encryption of the derived key)
```

If a malicious participant uses `app_id'` instead of `app_id`, their contribution to `C` includes `lambda_i · x_i · H(pk, app_id')` instead of `lambda_i · x_i · H(pk, app_id)`. The accumulated `C` no longer decrypts to `msk · H(pk, app_id)` under `app_sk`. The coordinator outputs a `CKDOutput` that is cryptographically invalid for the intended application. Honest parties accept this corrupted output, permanently denying the application a correct derived key for that `app_id`.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
Any single malicious participant in the CKD protocol can trigger this. The attacker only needs to call the public `ckd(...)` API with a different `app_id` or `app_pk` argument than what was agreed out-of-band. No cryptographic capability, leaked key, or external compromise is required. The attack is trivially reachable by any participant who deviates from the protocol.

### Recommendation
The coordinator must verify that each received contribution was computed under the correct `app_id` and `app_pk`. One approach is to include a zero-knowledge proof of correct computation alongside each `(norm_big_y, norm_big_c)` contribution — specifically, a proof that `big_s = H(pk, app_id) * x_i` for the agreed `app_id` and the participant's committed public share. Alternatively, participants can commit to the `app_id` and `app_pk` they used (e.g., by hashing them into a broadcast message) before sending their contribution, allowing the coordinator to reject mismatched inputs.

### Proof of Concept
1. Honest participants agree out-of-band to derive a key for `app_id = b"target_app"` and a known `app_pk`.
2. Malicious participant `P_m` calls `ckd(..., app_id = b"wrong_app", app_pk, ...)` locally.
3. `P_m`'s `compute_signature_share` computes `hash_point = H(pk, b"wrong_app")` and produces `(norm_big_y_m, norm_big_c_m)` under the wrong context.
4. `P_m` sends this to the coordinator via the normal protocol channel.
5. The coordinator at lines 50–55 of `src/confidential_key_derivation/protocol.rs` adds `norm_big_c_m` to the running sum without any check.
6. The final `CKDOutput` does not equal `msk · H(pk, b"target_app")` encrypted under `app_pk`. The application's call to `ckd_output.unmask(app_sk)` returns an incorrect group element, permanently corrupting the derived key for `b"target_app"`. [1](#0-0) [2](#0-1) [3](#0-2)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
