### Title
Unbounded `stakers` Vec Growth Enables Griefing of `get_stakers()` via Repeated Stake-Unstake Cycles - (File: src/staking/staking.cairo)

### Summary
The `stakers` storage Vec in the `Staking` contract grows permanently with every `stake()` call and is never pruned when stakers exit. The `get_stakers()` function iterates over the entire Vec on every invocation. An unprivileged actor can inflate the Vec to an arbitrary size at low cost (staked funds are fully recovered after unstaking), causing `get_stakers()` to consume unbounded gas and eventually become uncallable.

### Finding Description
Every call to `stake()` unconditionally appends the caller's address to the `stakers` Vec: [1](#0-0) 

The storage declaration explicitly documents that stakers are never removed: [2](#0-1) 

`unstake_action()` erases the staker's `staker_info` record but does **not** remove the address from the `stakers` Vec: [3](#0-2) 

`get_stakers()` iterates over the **full range** of the Vec on every call, reading each entry from storage to check whether it is still active: [4](#0-3) 

Because the Vec is append-only and never pruned, an attacker can grow it to an arbitrary size by repeatedly registering new staker addresses (each at `min_stake`) and then unstaking them. The staked tokens are fully returned after the exit-wait window, so the only recurring cost is gas.

### Impact Explanation
`get_stakers()` is the critical entry point used by the consensus layer to enumerate the active validator set for a given epoch: [5](#0-4) 

If the Vec grows large enough, every call to `get_stakers()` will exceed the step/gas limit and revert. The consensus layer can no longer retrieve the validator set, which damages protocol operation. This matches the **Medium** impact category: *Griefing with no profit motive but damage to users or protocol* and *Unbounded gas consumption*.

### Likelihood Explanation
- Any unprivileged address can call `stake()` followed by `unstake_intent()` + `unstake_action()`.
- The staked `min_stake` tokens are fully recovered after the exit-wait window; the only cost is gas fees.
- The same STRK balance can be recycled across different addresses (transfer → stake → unstake → transfer → repeat), so the capital requirement is just `min_stake` held at any one time.
- Starknet transaction fees are low, making a large number of registrations economically feasible.
- The attack is cumulative and irreversible: each cycle permanently adds one dead entry to the Vec.

### Recommendation
Remove the staker's address from the `stakers` Vec (or mark it as deleted with a tombstone flag) inside `unstake_action()` / `remove_staker()`. Alternatively, replace the append-only Vec with an iterable map that supports deletion, or add a separate active-staker index that is maintained alongside the Vec.

### Proof of Concept
1. Attacker holds `min_stake` STRK.
2. Attacker calls `stake(reward, op_addr_1, min_stake)` from address `A1` → `stakers` Vec length = L+1.
3. Attacker calls `unstake_intent()` from `A1`; waits for exit-wait window; calls `unstake_action(A1)` → `min_stake` returned, but `A1` remains in `stakers` Vec.
4. Attacker transfers `min_stake` to address `A2`, repeats steps 2–3 → Vec length = L+2.
5. After N cycles, `stakers` Vec has L+N entries, all inactive.
6. Any call to `get_stakers(epoch_id)` now iterates over L+N entries, performing storage reads for each.
7. For sufficiently large N, the step count exceeds the Starknet execution limit, causing `get_stakers()` to revert on every invocation.
8. The consensus layer can no longer enumerate the validator set.
