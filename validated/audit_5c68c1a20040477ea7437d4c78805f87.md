### Title
Incorrect `nftSellerNetAmount` Calculation Displays Wrong Net Proceeds to NFT Sellers in Offer Viewer — (File: `packages/gui/src/components/offers/utils.ts`)

---

### Summary

In `calculateNFTRoyalties`, the variable `nftSellerNetAmount` is unconditionally assigned the raw asking price (`amount`) instead of the intended net-after-royalties value (`amount - royaltyAmount`). This incorrect value is surfaced to NFT sellers in the offer viewer as "Net Proceeds," causing them to see a higher amount than they will actually receive, and potentially inducing acceptance of offers they would otherwise reject.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, the function `calculateNFTRoyalties` computes display values for NFT offer summaries:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The commented-out block reveals the intended formula: `amount - royaltyAmountString - makerFee`. Instead, `nftSellerNetAmount` is hardcoded to `amount` — the full offered price — regardless of royalties. This is the direct analog to the external report's bug: a wrong variable (the gross amount) is used where the correct variable (the net amount) was intended.

This incorrect value propagates into `NFTOfferViewer.tsx` via `overrideNFTSellerAmount`:

```typescript
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

And is rendered as "Net Proceeds" in the offer viewer for the `TokenForNFT` exchange type:

```typescript
{exchangeType === NFTOfferExchangeType.TokenForNFT && (
  ...
  <Trans>Net Proceeds</Trans>
  ...
  <FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} />
)}
``` [3](#0-2) 

The same incorrect value is also shown in `NFTOfferEditor.tsx` under "They will receive" for the `TokenForNFT` tab: [4](#0-3) 

---

### Impact Explanation

When a buyer creates a `TokenForNFT` offer (offering XCH/CAT to purchase an NFT that has royalties enabled), the NFT seller who views the offer in the GUI sees "Net Proceeds: X" where X equals the full offered amount. In reality, the seller receives `X - royaltyAmount` because the royalty is deducted from the offered amount and paid to the NFT creator. For an NFT with a 10% royalty and a 10 XCH offer, the seller sees "Net Proceeds: 10 XCH" but actually receives 9 XCH. This display spoofing of the offer amount causes a user to approve a transaction based on a materially incorrect amount, satisfying the High impact criterion: *"offer state that causes a user to approve the wrong amount."*

---

### Likelihood Explanation

Any NFT with a non-zero `royaltyPercentage` triggers this path. The `TokenForNFT` offer flow is a standard user-facing feature. No special attacker capability is required — a buyer simply creates a standard offer for an NFT with royalties. The NFT seller, viewing the offer through the GUI's offer viewer, will always see the inflated "Net Proceeds" figure. The discrepancy scales with royalty percentage and offer size.

---

### Recommendation

In `calculateNFTRoyalties`, restore the intended net calculation for `nftSellerNetAmount`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [1](#0-0) 

Apply this only for the `TokenForNFT` exchange type (where royalties are deducted from the offered amount), and keep `nftSellerNetAmount = amount` for `NFTForToken` (where the seller receives the full asking price and the buyer pays royalties on top).

---

### Proof of Concept

1. Mint or obtain an NFT with `royaltyPercentage = 1000` (10%).
2. As a buyer, create a `TokenForNFT` offer: offer 10 XCH for the NFT.
3. As the NFT seller, open the offer in the GUI's offer viewer (`NFTOfferViewer`).
4. Observe "Net Proceeds: 10 XCH" displayed.
5. Accept the offer.
6. Observe that the seller's wallet receives 9 XCH (10 XCH − 10% royalty = 9 XCH), not the 10 XCH shown in the GUI.

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
