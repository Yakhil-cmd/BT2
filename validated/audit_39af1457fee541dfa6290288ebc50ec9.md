### Title
Pusher Consent Signature Replay Reinstates Revoked Delegation After `revokePusher()` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` validates a pusher's EIP-191 consent signature but never marks that signature as consumed. A creator who saved the original calldata can replay the identical `(deadline, pusher, sig)` tuple after the pusher has called `revokePusher()`, silently re-establishing the delegation within the same deadline window. The pusher's unilateral exit right is therefore illusory for the entire lifetime of the original deadline.

---

### Finding Description

`allowPushers` computes the signed hash as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The five-field tuple contains no nonce, no per-pusher sequence counter, and no "used-signature" bitmap. The only replay guard is the deadline check:

```solidity
function _ensureDeadline(uint256 deadline) internal view virtual {
    require(block.timestamp <= deadline, DeadlineExceeded());
}
``` [2](#0-1) 

`revokePusher()` only zeroes `namespaceRemapping[msg.sender]`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [3](#0-2) 

It does **not** invalidate the original signature. Because the signed message is identical on every replay (same `chainid`, `oracle`, `deadline`, `pusher`, `creator`), the creator can call `allowPushers` again with the exact same arguments and the contract accepts it, writing `namespaceRemapping[pusher] = creator` a second time.

The code comment on `allowPushers` itself acknowledges the risk but treats the deadline as a sufficient guard:

> "The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it." [4](#0-3) 

The deadline only prevents replay *after* it expires. Within the deadline window — which can be hours or days — the revocation is completely ineffective.

---

### Impact Explanation

After the creator replays the signature:

- `namespaceRemapping[pusher] = creator` is restored.
- Every subsequent `fallback()` push from the pusher's key lands in the **creator's namespace**, not the pusher's own namespace. [5](#0-4) 

If the pusher is an active price-data bot that continues pushing after revoking (expecting pushes to go to its own namespace), those pushes instead overwrite the creator's oracle slots. A pool whose `IPriceProvider` reads from the creator's `CompressedOracleV1` namespace will consume this data for bid/ask pricing. The pusher's data — which may be intended for a different feed or may be stale relative to the creator's intended source — drives live swap math, constituting bad-price execution.

Additionally, the pusher's own namespace receives no updates (pushes are redirected), so any pool or provider reading the pusher's own feeds sees a stale or zero price, which every consumer rejects as stale — effectively a denial of service on the pusher's own feeds.

---

### Likelihood Explanation

- The creator observes the original `allowPushers` transaction on-chain and trivially saves the calldata.
- The pusher's `revokePusher()` transaction is also public; the creator can front-run or immediately follow with the replay.
- No privileged access is required — `allowPushers` is fully permissionless.
- The window is as wide as the original deadline (commonly hours to days in oracle infrastructure).
- The pusher's only mitigation is to stop pushing entirely, which is an operational workaround, not a protocol guarantee.

---

### Recommendation

Track consumed signatures per pusher. The simplest approach is a per-pusher nonce that is incremented on every successful `allowPushers` call and included in the signed hash:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusherNonce[pusher],   // <-- add nonce
        pusher, msg.sender
    ))
);
pusherNonce[pusher]++;
```

Alternatively, increment `pusherNonce[pusher]` inside `revokePusher()` so that any previously issued signature is immediately invalidated on revocation, without changing the delegation flow.

### Proof of Concept

```solidity
// 1. Pusher signs consent with deadline = now + 1 day
uint256 deadline = block.timestamp + 1 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation
address[] memory pushers = new address[](1);
pushers[0] = pusher;
bytes[] memory sigs = new bytes[](1);
sigs[0] = sig;
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs);
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation active

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // pusher believes they exited

// 4. Creator replays the SAME signature — no new consent from pusher
vm.prank(creator);
oracle.allowPushers(deadline, pushers, sigs); // succeeds: deadline still valid
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation silently restored

// 5. Pusher's next push lands in creator's namespace, not pusher's own
uint56 tsMs = uint56(block.timestamp * 1000);
uint48 raw = _packRaw(1_500_000, 4, 2);
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, raw, tsMs));
assertTrue(ok);
// Creator's namespace is updated with pusher's data (unintended)
assertGt(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, 0);
// Pusher's own namespace remains empty (their feeds are stale)
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher, 0, 0)).price, 0);
```
