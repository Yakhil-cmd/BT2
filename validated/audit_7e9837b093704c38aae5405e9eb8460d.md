### Title
Single Malicious Participant Can Permanently Abort DKG/Reshare/Refresh via Invalid Signing Share - (File: src/dkg.rs)

### Summary
In `do_keyshare`, the DKG/reshare/refresh protocol unconditionally requires every participant to supply a valid signing share. A single malicious participant can send a cryptographically invalid share to any one honest participant, causing that participant to abort with `ProtocolError::InvalidSecretShare`. Because the aborting participant never reaches `broadcast_success`, all remaining honest participants block indefinitely in that final broadcast round, permanently failing the session. A single participant — far below the documented N/3 BFT tolerance — can repeat this attack to deny key generation, reshare, or refresh indefinitely.

### Finding Description

`do_keyshare` in `src/dkg.rs` is the shared implementation for `keygen`, `reshare`, and `refresh`. In Round 5 it collects a signing-share from **every** other participant and immediately verifies each one:

```rust
// src/dkg.rs lines 514–528
for (from, signing_share_from) in
    recv_from_others(&chan, wait_round_3, &participants, me).await?
{
    let full_commitment_from = all_full_commitments.index(from)?;
    validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
    my_signing_share = my_signing_share + signing_share_from.to_scalar();
}
```

`validate_received_share` performs a VSS consistency check:

```rust
// src/dkg.rs lines 259–286
secret_share.verify().map_err(|e| {
    if let Error::InvalidSecretShare { .. } = e {
        ProtocolError::InvalidSecretShare(from)
    } else { ... }
})?;
```

If the check fails for **any** participant, the `?` operator propagates the error immediately, aborting the entire `do_keyshare` future for the receiving participant. That participant never reaches:

```rust
// src/dkg.rs line 531
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

`broadcast_success` calls `do_broadcast`, which is the echo-broadcast protocol that requires **all N** participants to deliver a READY message before any participant can return:

```rust
// src/dkg.rs lines 307–338
async fn broadcast_success(...) {
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    ...
    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(...);
    }
}
```

Because the aborted participant never sends its READY message, every other honest participant blocks indefinitely inside `broadcast_success`. The session is unrecoverable without a full restart.

The attack path is:

1. Malicious participant P_m participates honestly through Rounds 1–4 (so all honest parties accept its commitments and proofs).
2. In Round 4 (Step 4.6), P_m sends a **valid** share to every participant except target P_h, to whom it sends a random scalar that fails the VSS check.
3. P_h's `validate_received_share` returns `ProtocolError::InvalidSecretShare(P_m)`.
4. P_h's `do_keyshare` future terminates with an error; P_h never calls `broadcast_success`.
5. All other honest participants are now permanently blocked waiting for P_h's READY message in `broadcast_success`.
6. The DKG session fails for all N participants.
7. P_m can repeat this on every subsequent attempt.

This is the direct analog of the "push vs pull" vulnerability class: the protocol **pushes** a mandatory verification over every participant's contribution in a single sequential loop, with no mechanism to skip or quarantine a bad contributor and proceed with the remaining honest majority.

### Impact Explanation

A single malicious participant — one out of N, far below the documented N/3 BFT threshold — can abort every DKG, reshare, or refresh session. Because no signing can occur until a DKG session completes, this constitutes permanent denial of key generation, reshare, and refresh for all honest parties. The protocol provides no complaint mechanism, no way to identify and exclude the culprit, and no way to resume the session. The README documents that the library "can only tolerate N/3 malicious parties," but the share-distribution step has zero tolerance for a participant sending an invalid share.

**Allowed impact matched**: *High — Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs and documented trust assumptions.*

### Likelihood Explanation

High. The attack requires only that a participant deviate from the protocol in Round 4 by sending one random scalar instead of the correct polynomial evaluation. No cryptographic capability, no key material, and no special network position is required. The attacker participates honestly through all earlier rounds (so their commitments pass verification) and only deviates in the final private-share delivery step, making detection difficult without a complaint mechanism.

### Recommendation

Implement a **complaint/dispute round** after share collection: if a participant receives an invalid share, it broadcasts a complaint (including the offending share and the sender's commitment) so all honest parties can verify the complaint and exclude the malicious participant. The DKG can then proceed with the remaining N−1 participants, provided the threshold is still met. This is the standard fix used in protocols such as GJKR99 and is the direct analog of the "pull" pattern recommended in the external report.

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}

    participants: &ParticipantList,
    me: Participant,

    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;

        return Err(ProtocolError::AssertionFailed(
            "A participant

        ));

        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
}

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
}
