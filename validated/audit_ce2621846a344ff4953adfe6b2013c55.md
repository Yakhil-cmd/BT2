### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Silently Discards Negative Delta, Permanently Overcharging Users — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` adjusts `gas_used` upward when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`), but performs **no adjustment** when EVM gas consumption exceeds native resource consumption (`delta_gas < 0`). The excess gas the user already paid is never returned. A developer `TODO` comment at the exact branch explicitly acknowledges the missing path. This is a direct resource-accounting analog to the reported pattern: a value is accumulated (the user's overpayment) but never consumed or refunded.

---

### Finding Description

In `compute_gas_refund` the double-resource reconciliation block reads:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs  lines 66-81
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // native cost > gas cost → charge extra gas
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?   ← acknowledged missing branch
}
```

`delta_gas` is defined as `(native_used / native_per_gas) − gas_used`.

| Sign of `delta_gas` | Meaning | Code action |
|---|---|---|
| `> 0` | Native proving cost exceeds EVM gas cost | `gas_used` increased → user pays more |
| `< 0` | EVM gas cost exceeds native proving cost | **Nothing** → user overpays silently |
| `= 0` | Costs balanced | No change |

When `delta_gas < 0` the user has already been charged for `gas_used` units of EVM gas (deducted from their balance in `precharge_fee`), but the actual proving cost only warrants `native_used / native_per_gas` gas units. The difference — `abs(delta_gas)` gas units — is never subtracted from `gas_used`, so the final refund `gas_limit − gas_used` is smaller than it should be. The operator (coinbase) receives the excess.

The `TODO: return delta_gas to gas_used?` comment at line 80 is the developers' own acknowledgement that this branch is unimplemented.

The double-resource-accounting documentation (`docs/double_resource_accounting.md`) only describes the `deltaGas > 0` path and is silent on the negative case, confirming the missing branch is not a deliberate design choice but an incomplete implementation.

---

### Impact Explanation

Every transaction whose EVM gas consumption exceeds its native resource consumption (in gas-equivalent units) results in the user paying more than the actual proving cost. The excess is silently transferred to the operator via the fee payment in `refund_and_commit_fee`. Over many transactions this constitutes a systematic, protocol-level drain of user funds with no mechanism to recover the overpayment. The magnitude per transaction is `abs(delta_gas) × gas_price` native tokens.

---

### Likelihood Explanation

The `delta_gas < 0` case is reachable by any ordinary unprivileged transaction sender. It occurs whenever a transaction performs significant EVM computation (high `gas_used`) but produces little pubdata and has low proving complexity (low `native_used`). Examples include pure arithmetic-heavy contracts, large in-memory loops, or any call that does not write storage. No special privileges, oracle manipulation, or governance access are required. The condition is triggered by the normal transaction submission path.

---

### Recommendation

Apply the symmetric adjustment: when `delta_gas < 0`, reduce `gas_used` by `abs(delta_gas)` (clamped so `gas_used` does not fall below `minimal_gas_used`):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // TODO resolved: return excess gas to user
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This mirrors the existing `max(gas_used, minimal_gas_used)` guard already applied at line 56 and ensures the user is charged only for the greater of EVM gas and native-equivalent gas, not always the EVM gas figure.

---

### Proof of Concept

1. Deploy a contract that performs a large number of pure arithmetic operations (e.g., a tight loop of `ADD`/`MUL` opcodes) with no storage writes and no pubdata emission.
2. Submit a ZKsync L2 transaction calling that contract with a gas price such that `native_per_gas > 0`.
3. After execution, observe that `native_used / native_per_gas < gas_used` (the loop is cheap to prove but expensive in EVM gas).
4. `delta_gas` is negative; the `if delta_gas > 0` branch is skipped; `gas_used` is not reduced.
5. The user's refund is `gas_limit − gas_used` instead of the correct `gas_limit − (native_used / native_per_gas)`.
6. The operator's balance increases by the extra `abs(delta_gas) × gas_price` tokens that should have been refunded to the sender.
