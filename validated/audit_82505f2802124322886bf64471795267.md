### Title
Unprivileged Caller Can Permanently Suppress All Block Rewards via Global `last_reward_block` — (`File: src/staking/staking.cairo`)

### Summary

`update_rewards` is documented as "Only starkware sequencer" but the implementation enforces no such restriction. Any address can call it with `disable_rewards=true`, which writes the global `last_reward_block` and returns early without distributing rewards. Because `last_reward_block` is a single global slot, every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`. The block's rewards are permanently lost.

### Finding Description

The spec states access control for `update_rewards` is "Only starkware sequencer." [1](#0-0) 

The actual `general_prerequisites()` enforcer only checks pause state and non-zero caller — no role or sequencer check: [2](#0-1) 

`update_rewards` calls only `general_prerequisites()` before proceeding: [3](#0-2) 

`last_reward_block` is a **single global** `BlockNumber` in storage (not per-staker): [4](#0-3) 

The function unconditionally writes `last_reward_block` to the current block **before** checking `disable_rewards`, then returns early if `disable_rewards=true` or `is_pre_consensus()`: [5](#0-4) 

### Impact Explanation

An attacker calls `update_rewards(any_active_staker, disable_rewards=true)` once per block. This:
1. Passes all guards (non-zero caller, staker exists, staker active, non-zero balance).
2. Writes `last_reward_block = current_block`.
3. Returns without distributing any rewards.

Any legitimate sequencer call in the same block hits `REWARDS_ALREADY_UPDATED` and reverts. The block's rewards are **permanently unrecoverable** — there is no mechanism to retroactively credit a missed block. Repeating this every block freezes all staker and delegator yield indefinitely.

**Impact: High — Permanent freezing of unclaimed yield.**

### Likelihood Explanation

The function is public, requires no tokens, no stake, and no privileged role. The only cost is gas per block. Any address can execute this attack continuously.

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, analogous to how other privileged functions use role checks. The check should verify `get_caller_address()` matches the authorized sequencer address (or a dedicated role), consistent with the spec's stated access control.

### Proof of Concept

1. Deploy the staking system with two active validators (past K epochs, consensus rewards enabled).
2. From an arbitrary EOA (no stake, no role), call `update_rewards(staker_A, disable_rewards=true)` at block N.
3. Observe: `last_reward_block` is now N; staker_A receives zero rewards.
4. Have the legitimate sequencer attempt `update_rewards(staker_B, disable_rewards=false)` in the same block N.
5. Observe: reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeat step 2 every block. After many blocks, compare cumulative `unclaimed_rewards_own` for all stakers against the expected model — it remains zero.

The existing test `test_update_rewards_assertions_before_consensus` already demonstrates that `REWARDS_ALREADY_UPDATED` fires on the second same-block call regardless of caller identity, confirming the global gate is exploitable: [6](#0-5)
