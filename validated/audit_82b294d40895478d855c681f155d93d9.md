### Title
Permanent ERC-20 Token Loss Due to Unchecked Precompile Call Return Value in `withdrawToNear` - (File: `etc/eth-contracts/contracts/EvmErc20.sol` and `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` by first burning the caller's tokens unconditionally, then calling the `ExitToNear` precompile via low-level assembly. The return value of that `call()` is captured in `res` but **never checked**. If the precompile call fails for any reason, the ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer to the NEAR recipient — a silent, irrecoverable fund loss.

---

### Finding Description

In `EvmErc20.sol`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

The identical pattern exists in `EvmErc20V2.sol`: [2](#0-1) 

The `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` can return failure (causing `res = 0`) in several reachable conditions:

1. **Invalid NEAR account ID as recipient** — `parse_recipient` inside the precompile validates the bytes as a NEAR `AccountId`. If the bytes do not satisfy NEAR's account ID format (2–64 chars, lowercase alphanumeric/dots/underscores/hyphens), the precompile returns `ExitError`, the `call()` returns 0, but the outer transaction continues normally.
2. **ETH connector account not configured in storage** — `get_eth_connector_contract_account` reads from storage and returns an error if the key is absent.
3. **NEP-141 ↔ ERC-20 mapping absent** — `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND` if the mapping is missing. [3](#0-2) [4](#0-3) 

In all these cases, the EVM `call()` returns 0 into `res`, the Solidity function returns normally (no revert), and the burned tokens are gone forever. The `error_refund` callback mechanism in `EvmErc20V2.sol` only handles NEAR-side `ft_transfer` failures **after** the precompile call succeeds — it does not protect against the precompile call itself failing. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

When `withdrawToNear` is called:
- The user's ERC-20 tokens are burned from the EVM state.
- If the precompile call fails silently, no NEAR promise is created.
- No NEP-141 tokens are transferred to the recipient.
- There is no refund path; the tokens are permanently destroyed.

This matches the allowed impact scope: *permanent freezing of funds* (the tokens cease to exist on either chain).

---

### Likelihood Explanation

**Medium.** The most user-reachable trigger is providing a `recipient` byte string that is not a valid NEAR account ID (e.g., uppercase characters, too short, too long, or containing invalid characters). Because `recipient` is typed as `bytes` (not `string` or a validated type), no on-chain validation occurs before the burn. A user making a typo or using a programmatic interface that passes raw bytes will silently lose their entire `amount`. This requires no special privileges — any token holder can trigger it.

---

### Recommendation

Reverse the order of operations: validate and execute the precompile call **before** burning tokens, or check `res` and revert on failure:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    uint res;
    assembly {
        res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(res == 1, "ExitToNear precompile call failed");

    _burn(sender, amount);  // burn only after confirmed success
}
```

This ensures tokens are only destroyed when the cross-chain exit is successfully initiated.

---

### Proof of Concept

1. Deploy an `EvmErc20` token via `deploy_erc20_token` on Aurora.
2. Bridge NEP-141 tokens to mint ERC-20 balance for address `A`.
3. Address `A` calls `withdrawToNear(bytes("INVALID ACCOUNT!!"), amount)` — the recipient contains uppercase letters and spaces, which are not valid NEAR account ID characters.
4. `_burn(A, amount)` executes — ERC-20 balance of `A` drops to zero.
5. The precompile's `parse_recipient` fails on the invalid account ID bytes; the `call()` returns `res = 0`.
6. No revert occurs; the transaction succeeds.
7. `A`'s ERC-20 balance is 0. No NEP-141 tokens arrive at any NEAR account. Funds are permanently lost. [1](#0-0) [2](#0-1) [6](#0-5)
