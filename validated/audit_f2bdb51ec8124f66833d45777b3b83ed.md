### Title
Malicious Participant Can Corrupt CKD Output Without Detection by Sending Arbitrary Contribution Shares - (File: src/confidential_key_derivation/protocol.rs)

### Summary

The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` aggregates `(norm_big_y, norm_big_c)` contributions from all participants by simple addition, with no cryptographic verification that each participant's share was computed correctly from their actual private key share. Any participant in the protocol can send arbitrary group elements, causing the coordinator to output a silently corrupted `CKDOutput`. This is the direct analog of the missing slippage check: an operation that consumes participant-supplied values and produces a protocol output without validating those values against any expected reference.

### Finding Description

In `do_ckd_coordinator` (lines 35–57 of `src/confidential_key_derivation/protocol.rs`), the coordinator computes its own share via `compute_signature_share`, then collects every other participant's `(norm_big_y, norm_big_c)` pair and accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

The correct contribution from participant `i` is:

- `norm_big_y_i = λ_i · y_i · G₁`
- `norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)`

computed in `compute_signature_share`: [2](#0-1) 

The coordinator has no mechanism to verify that the received `norm_big_c_i` was formed using the participant's actual private share `x_i`. There are no zero-knowledge proofs, no commitments, and no consistency check against the participant's public key share from the DKG output. The only check that occurs is implicit deserialization of valid curve points.

A malicious participant `P_m` can deviate by sending `(0, 0)` or any arbitrary `(δ_Y, δ_C)` pair. The coordinator then computes:

- `Y' = Σ_{i≠m} norm_big_y_i + δ_Y`
- `C' = Σ_{i≠m} norm_big_c_i + δ_C`

When the application calls `unmask(app_sk)` on this output, it computes `C' − app_sk · Y'`. With `(δ_Y, δ_C) = (0, 0)`:

```
C' − app_sk · Y' = Σ_{i≠m}(λ_i · x_i) · H(pk, app_id)
```

which equals `(msk − λ_m · x_m) · H(pk, app_id)` — a silently wrong confidential key, not equal to the correct `msk · H(pk, app_id)`. With arbitrary `(δ_Y, δ_C)`, the malicious participant can shift the derived key by any chosen offset `δ_C − app_sk · δ_Y`.

The non-coordinator participant path in `do_ckd_participant` simply sends the share with no broadcast or echo mechanism, so the coordinator is the sole aggregator and the only party that could detect the deviation — but it performs no such check: [3](#0-2) 

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept an unusable or attacker-shifted confidential derived key.**

The `CKDOutput` produced by the coordinator is silently wrong. Every honest party that calls `unmask` on it receives a derived key that does not correspond to `msk · H(pk, app_id)`. The application-level secret derived from the threshold key material is therefore either garbage or a value controlled by the malicious participant's chosen offset. This falls squarely within the allowed impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

### Likelihood Explanation

Any participant enrolled in the CKD session — a role that requires no privilege beyond being in the `participants` list — can execute this attack. The attack requires no cryptographic knowledge: sending `(0, 0)` or any syntactically valid pair of curve points is sufficient. There is no round of blame attribution, no broadcast consistency check, and no final verification of the aggregate output against the known public key. The attack is silent and undetectable by the coordinator or any other honest participant.

### Recommendation

Add a zero-knowledge proof of correct share formation to each participant's contribution. Concretely, each participant should prove in zero knowledge that `norm_big_c_i` was formed as `λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` using the same `x_i` committed to in the DKG public key share and the same `y_i` used to form `norm_big_y_i`. A standard sigma protocol (e.g., a Chaum–Pedersen proof of discrete-log equality) can prove that `norm_big_c_i − norm_big_y_i · app_sk = λ_i · x_i · H(pk, app_id)` without revealing `x_i` or `y_i`, using the participant's public key share from the DKG output as the reference. The coordinator must verify all proofs before aggregating.

Alternatively, adopt an echo-broadcast or commit-then-reveal pattern so that all participants can cross-check each other's contributions before the coordinator finalises the output.

### Proof of Concept

**Setup:** 3 participants `[P1, P2, P3]`, coordinator = `P1`, threshold = 2. All hold valid DKG key shares summing to `msk`.

**Attack:**
1. `P3` intercepts the protocol at the point where it would call `compute_signature_share` and instead fabricates `(δ_Y, δ_C) = (G1::identity(), G1::identity())` (the group identity / zero element).
2. `P3` sends this fabricated pair to `P1` via `chan.send_private`.
3. `P1`'s `do_ckd_coordinator` loop adds `(0, 0)` to its running totals.
4. The resulting `CKDOutput` satisfies `C' − app_sk · Y' = (λ_1·x_1 + λ_2·x_2) · H(pk, app_id)` instead of `msk · H(pk, app_id)`.
5. Any consumer calling `ckd_output.unmask(app_sk)` receives the wrong confidential key with no error or warning.

**Expected correct output:** `msk · H(pk, app_id)`
**Actual output:** `(msk − λ_3·x_3) · H(pk, app_id)` — silently wrong, accepted by all honest parties. [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
