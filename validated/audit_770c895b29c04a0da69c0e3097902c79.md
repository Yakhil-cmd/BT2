### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: etc/eth-contracts/contracts/EvmErc20.sol)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens before calling the Aurora exit precompile via a low-level assembly `call`. The return value of that call (`res`) is captured but never checked. If the precompile call fails for any reason, the tokens are permanently destroyed with no corresponding NEP-141 tokens released on NEAR — an exact structural analog to the unsafe `.transfer()` pattern in the reference report.

---

### Finding Description

In `EvmErc20.sol`, the `withdrawToNear` and `withdrawToEthereum` functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) 

The same unchecked pattern appears in `withdrawToEthereum`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) 

The `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) validates its input and returns `0` (failure) for conditions such as an invalid NEAR account ID, a recipient that is too long, or any internal precompile error. Because `res` is never inspected and no `revert` is issued on failure, the Solidity function returns `success` to the caller even when the precompile rejected the exit request. The tokens have already been burned at that point and cannot be recovered.

The reference report's root cause — a critical transfer operation whose failure is silently swallowed — maps directly onto this pattern: `.transfer()` returning `false` without revert ↔ `call()` returning `0` without revert.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

When the precompile call fails silently:
- The user's ERC-20 balance is reduced to zero (`_burn` succeeded).
- No NEP-141 tokens are released on NEAR (the precompile did not execute the exit).
- There is no on-chain record of failure; the transaction receipt shows success.
- The tokens are unrecoverable: the ERC-20 supply is deflated, and the backing NEP-141 tokens remain locked in the Aurora contract on NEAR with no mechanism to reclaim them.

---

### Likelihood Explanation

**Medium.**

The precompile rejects input for several user-reachable reasons:
- A NEAR account ID that is syntactically invalid or exceeds the maximum length.
- A recipient byte string that is empty or malformed.
- Any future precompile-side validation added without a corresponding Solidity-side guard.

A user who mistypes a recipient, or a third-party contract that constructs the recipient programmatically with a bug, will silently burn tokens. No privileged access or attacker compromise is required — any token holder can trigger this by calling `withdrawToNear` or `withdrawToEthereum` directly.

---

### Recommendation

Check `res` inside the assembly block and revert on failure, mirroring the safe-transfer pattern:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile rejects the exit, the entire transaction reverts, the `_burn` is rolled back, and the user's tokens are preserved. Apply the same fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` (or use an existing bridged token).
2. Mint tokens to `address(this)` via the admin.
3. Call `withdrawToNear(invalidRecipient, amount)` where `invalidRecipient` is a byte string that the `ExitToNear` precompile will reject (e.g., an account ID exceeding 64 bytes, or containing invalid characters).
4. Observe: the transaction succeeds (no revert), the caller's ERC-20 balance is zero, and no NEP-141 tokens are released on NEAR.
5. The burned tokens are permanently lost. [1](#0-0) [4](#0-3)
