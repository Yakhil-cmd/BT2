### Title
Malicious Participant Can Abort DKG/Signing by Sending a Malformed Message to `recv_from_others` - (File: src/protocol/helpers.rs)

### Summary

`recv_from_others` propagates deserialization errors via `?` instead of skipping malformed messages. A single malicious participant in the protocol set can send an undeserializable message at any waitpoint consumed by `recv_from_others`, causing the entire DKG, reshare, refresh, or signing protocol to abort for all honest parties. This is the direct analog of M-15: just as a malicious GMX keeper could supply invalid block numbers to trigger uncaught errors that cancel orders instead of retrying, a malicious threshold-signatures participant can supply malformed bytes to trigger an uncaught `DeserializationError` that aborts the protocol instead of skipping the bad message.

---

### Finding Description

`recv_from_others` is the central helper used to collect one message from every other participant in a round:

```rust
// src/protocol/helpers.rs lines 19-23
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // <-- ? propagates error
    if seen.put(from) {
        messages.push((from, msg));
    }
}
``` [1](#0-0) 

`chan.recv` ultimately calls `Comms::recv`, which attempts to deserialize the raw bytes with `rmp_serde`:

```rust
// src/protocol/internal.rs lines 338-340
let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
    rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
Ok((from, decoded?))   // <-- ? propagates DeserializationError / Other
``` [2](#0-1) 

If a malicious participant sends bytes that fail `rmp_serde` deserialization, `decoded?` returns `ProtocolError::Other(...)`. The `?` in `recv_from_others` immediately propagates this error to the caller, which in turn propagates it up through the entire protocol stack via `?`.

This affects every protocol round that uses `recv_from_others`:

- **DKG Round 1** (commitment hash collection): [3](#0-2) 

- **DKG Round 5** (secret share collection): [4](#0-3) 

- **FROST signing** (signature share collection): [5](#0-4) 

The contrast with the echo-broadcast protocol is instructive. `reliable_broadcast_receive_all` — used in DKG rounds 2 and 3 — explicitly ignores receive errors and continues the loop:

```rust
// src/protocol/echo_broadcast.rs lines 179-182
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // <-- errors are silently skipped
};
``` [6](#0-5) 

`recv_from_others` makes no such allowance. The inconsistency means that the rounds protected by echo-broadcast are resilient to malformed messages, while the rounds using `recv_from_others` are not.

---

### Impact Explanation

A single malicious participant (who is legitimately part of the participant set) can permanently deny DKG, reshare, refresh, or signing for all honest parties by repeatedly sending a malformed message at any `recv_from_others` waitpoint. Every retry attempt can be aborted the same way. This matches the allowed High impact:

> **High: Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

The attack requires only that the adversary be a legitimate participant in the protocol — no key material, no external compromise, and no cryptographic break is needed. The attacker simply sends garbage bytes with a valid `MessageHeader` prefix (so `push_message` routes it correctly) but an invalid `rmp_serde` payload. This is trivially constructable by any participant who controls their own network stack. The attack is repeatable on every protocol invocation, making denial effectively permanent.

---

### Recommendation

Mirror the error-handling pattern already used in `reliable_broadcast_receive_all`. In `recv_from_others`, catch deserialization errors and continue the loop rather than propagating them:

```rust
// src/protocol/helpers.rs
while !seen.full() {
    let Ok((from, msg)) = chan.recv(waitpoint).await else {
        continue;   // skip malformed messages, do not abort
    };
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

This ensures that a malicious participant who sends an undeserializable message is simply ignored, and the protocol waits for a valid message from that participant (or times out under the caller's timeout policy, as documented).

### Proof of Concept

1. Honest participants `[A, B, C]` initiate DKG with threshold 2.
2. Malicious participant `M` (also in the participant set) constructs a message with a valid `MessageHeader` for `wait_round_1` but with a body that is not valid `rmp_serde`-encoded `HashOutput` (e.g., all-zero bytes after the header).
3. `M` sends this message to participant `A`.
4. `A`'s `recv_from_others` call at `wait_round_1` receives `M`'s message, `Comms::recv` fails deserialization, returns `ProtocolError::Other(...)`, and `recv_from_others` propagates it via `?`.
5. `do_keyshare` returns `Err(...)`, and `A`'s DKG protocol terminates with an error.
6. `M` repeats this on every retry, permanently preventing DKG completion.
