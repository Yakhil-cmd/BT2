### Title
Cross-COA Signature Replay via Unsigned `CapabilityPath` in COA Ownership Proof — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies cryptographic signatures over caller-supplied `signedData` but does not require `signedData` to commit to the target EVM address or the `CapabilityPath` field of the proof. Because the `COAOwnershipProof` is an RLP-encoded blob whose `CapabilityPath` field is fully attacker-controlled and is never covered by the signatures, an attacker who observes a valid proof for COA-A can surgically replace `CapabilityPath` with the path of COA-B (owned by the same Cadence account) and replay the unmodified signatures to obtain a passing ERC-1271 `isValidSignature` result for COA-B — a COA the signer never authorized.

---

### Finding Description

**Root cause — unsigned proof fields**

`COAOwnershipProof` is defined in `fvm/evm/types/proof.go`:

```
COAOwnershipProof {
    KeyIndices     []uint64
    Address        FlowAddress
    CapabilityPath PublicPath   // ← NOT signed
    Signatures     []Signature  // signs only signedData (_hash)
}
``` [1](#0-0) 

The signatures inside the proof are produced over `signedData` (the `bytes32 _hash` passed to `isValidSignature`), not over the proof struct itself. `CapabilityPath` is therefore a free, unsigned field that any caller can set to an arbitrary value.

**Verification logic**

`validateCOAOwnershipProof` performs two independent checks:

1. Verify the signatures over `signedData` using the account's keys.
2. Borrow the COA resource at the caller-supplied `path` and assert its EVM address equals the caller-supplied `evmAddress`. [2](#0-1) 

Neither check binds `signedData` to `evmAddress` or to `CapabilityPath`. The code itself acknowledges this gap:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [3](#0-2) 

**EVM entry path — `coa.sol` ERC-1271**

The COA smart-contract wallet implements ERC-1271 `isValidSignature`:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4)
{
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig   // address(this) = this COA's EVM addr
        )
    );
    ...
}
``` [4](#0-3) 

`_sig` is the RLP-encoded `COAOwnershipProof` and is entirely attacker-supplied. `address(this)` is the calling COA's EVM address and is fixed by the EVM runtime — but `CapabilityPath` inside `_sig` is not.

**Exploit flow**

Assume Alice owns a Cadence account with two COAs:
- COA-A stored at `/public/coaA`, EVM address `0xAAA`
- COA-B stored at `/public/coaB`, EVM address `0xBBB`

1. Alice legitimately signs `_hash` for COA-A, producing proof `P_A`:
   ```
   P_A = { KeyIndices:[0], Address:Alice, CapabilityPath:"/public/coaA", Signatures:[sig] }
   ```
2. Attacker observes `P_A` (it is passed as a public argument to `isValidSignature`).
3. Attacker RLP-decodes `P_A`, replaces `CapabilityPath` with `"/public/coaB"`, re-encodes → `P_B`. Signatures are unchanged and remain valid because they cover only `_hash`.
4. Attacker calls `COA-B.isValidSignature(_hash, RLP(P_B))`.
5. The Cadence Arch precompile invokes `validateCOAOwnershipProof(Alice, /public/coaB, _hash, [0], [sig], 0xBBB)`.
6. **Check 1** — signatures over `_hash` with Alice's key 0: **PASS** (same sig, same data).
7. **Check 2** — COA at `/public/coaB` has address `0xBBB`: **PASS**.
8. Returns `isValid: true` → `isValidSignature` returns `0x1626ba7e` (valid).

COA-B's ERC-1271 identity is now forged using a signature Alice only produced for COA-A. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

ERC-1271 is the standard mechanism by which smart-contract wallets (including COAs) authorize off-chain-signed messages on-chain. DeFi protocols use it for permit-style token approvals, order-book signatures, and cross-protocol authentication. A forged `isValidSignature` response for COA-B allows an attacker to:

- Authorize token transfers or approvals on behalf of COA-B without the owner's consent.
- Pass any ERC-1271-gated action (e.g., EIP-2612 `permit`, Gnosis Safe message hashing, NFT marketplace listings) as if COA-B's owner signed it.

The impact is **unauthorized movement of assets** held by or approved to COA-B.

---

### Likelihood Explanation

**Preconditions:**
- The victim Cadence account must own at least two COAs at distinct capability paths. The comment notes this is currently uncommon ("Flow wallets historically create at most one COA per account"), but the protocol imposes no such limit, and multi-COA accounts are a natural pattern for applications managing multiple EVM identities.
- The attacker must observe one valid proof for COA-A. Because proofs are passed as calldata to public EVM functions, they are visible on-chain.

**No privileged access required.** Any unprivileged EVM transaction sender can call `isValidSignature` on COA-B with the modified proof. The modification itself (RLP re-encoding with a different `CapabilityPath`) requires only standard tooling.

Likelihood is **medium** today (multi-COA accounts are rare) and rises as the ecosystem matures.

---

### Recommendation

Enforce that `signedData` commits to the target EVM address. The simplest fix is to require, inside `validateCOAOwnershipProof`, that `signedData` is a structured message that includes `evmAddress` as a mandatory field, and reject proofs where it does not. Alternatively, include `evmAddress` (and optionally `CapabilityPath`) in the data that is cryptographically signed, so that a proof is bound to exactly one COA and cannot be transplanted to another.

### Proof of Concept

```python
import rlp

# Step 1: observe Alice's valid proof for COA-A (from on-chain calldata)
P_A = rlp.decode(observed_proof_bytes)
# P_A = [key_indices, alice_address, b"/public/coaA", [sig_bytes]]

# Step 2: replace CapabilityPath only
P_B = [P_A[0], P_A[1], b"/public/coaB", P_A[3]]
modified_proof = rlp.encode(P_B)

# Step 3: call COA-B's isValidSignature with the same _hash and modified proof
# isValidSignature(_hash, modified_proof) → 0x1626ba7e  (VALID)
```

The signatures in `P_A[3]` are over `_hash` alone and remain cryptographically valid after the `CapabilityPath` substitution, because `CapabilityPath` is never included in the signed content. [7](#0-6) [8](#0-7)
