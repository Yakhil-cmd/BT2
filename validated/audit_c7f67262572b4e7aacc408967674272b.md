### Title
Hardcoded `L1_TX_NATIVE_PRICE = 10` Causes Resource Accounting Mismatch for L1→L2 Transactions - (File: `basic_bootloader/src/bootloader/constants.rs`)

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded at `10` in the bootloader constants. This value is used to compute the native resource limit for all L1→L2 priority transactions, instead of the operator-provided `native_price` used for L2 transactions. If the actual `native_price` diverges significantly from `10`, L1→L2 transactions receive a disproportionate native resource budget relative to what they pay, enabling resource accounting abuse.

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant `L1_TX_NATIVE_PRICE` is hardcoded at `10` with an explicit TODO acknowledging it is not properly calibrated: [1](#0-0) 

This constant is consumed in `prepare_and_check_resources` inside `process_l1_transaction.rs`, where it replaces the operator-provided `native_price` for all L1→L2 priority transactions: [2](#0-1) 

The resulting `native_per_gas` is computed as `gas_price / L1_TX_NATIVE_PRICE` (= `gas_price / 10`).

For L2 transactions, the same ratio is computed using the operator-provided `native_price` from block metadata: [3](#0-2) 

The `native_price` for L2 transactions comes from `BlockMetadataFromOracle.native_price`, which is set by the operator to reflect the actual cost of a RISC-V proving cycle: [4](#0-3) 

The documentation explicitly acknowledges this divergence: [5](#0-4) 

The native resource limit per transaction is `nativeLimit = gasLimit * (gasPrice / nativePrice)`. If the operator sets `native_price = 1000` for L2 transactions (a realistic value), but L1 transactions use `L1_TX_NATIVE_PRICE = 10`, then:

- L1 tx with `gas_price = 1000`: `native_per_gas = 1000 / 10 = 100`
- L2 tx with `gas_price = 1000`: `native_per_gas = 1000 / 1000 = 1`

L1→L2 transactions receive **100× more native resources** per gas unit than L2 transactions paying the same price.

### Impact Explanation

**Impact: Medium**

An attacker submitting L1→L2 priority transactions with high `gas_price` and `gas_limit` receives a native resource budget far exceeding what they effectively pay for in proving costs. This allows:

1. **Subsidized computation**: Computationally expensive operations (e.g., heavy EVM execution, precompile calls) that would be rejected for L2 transactions due to insufficient native resources can be executed via L1→L2 transactions at a fraction of the cost.
2. **Block native resource exhaustion**: A single L1→L2 transaction with a large `gas_limit` and high `gas_price` can consume the block's native resource budget disproportionately, crowding out legitimate L2 transactions.
3. **Inverse scenario**: If `native_price` is set below `10`, L1→L2 transactions receive fewer native resources than they should, causing legitimate priority transactions to fail with out-of-native errors even when the sender paid sufficient fees on L1.

### Likelihood Explanation

**Likelihood: Medium**

The operator-set `native_price` for L2 transactions reflects real-time proving costs and is expected to vary. The TODO comment `// TODO (EVM-1157): find a reasonable value for it.` directly confirms the value `10` is a placeholder. Any deployment where the actual `native_price` differs from `10` by more than a small factor triggers this mismatch. The entry path requires only submitting a standard L1→L2 priority transaction — no privileged access is needed.

### Recommendation

Replace the hardcoded `L1_TX_NATIVE_PRICE` with a value that is either:
1. Derived from the operator-provided `native_price` in `BlockMetadataFromOracle` (with a floor to preserve anti-censorship guarantees), or
2. Set to a carefully calibrated constant that reflects the actual proving cost per RISC-V cycle, and updated as part of protocol upgrades when proving costs change.

The anti-censorship rationale (preventing the operator from setting `native_price` arbitrarily high to block L1 txs) can be preserved by capping the effective `native_price` used for L1 txs at a protocol-defined maximum rather than using a fixed low value.

### Proof of Concept

1. Operator sets `native_price = 1000` in block metadata (realistic for a production deployment).
2. Attacker submits an L1→L2 priority transaction with `gas_price = 10000`, `gas_limit = 1_000_000`.
3. L1 tx: `native_per_gas = 10000 / 10 = 1000`, `native_limit = 1_000_000 * 1000 = 1_000_000_000`.
4. Equivalent L2 tx: `native_per_gas = 10000 / 1000 = 10`, `native_limit = 1_000_000 * 10 = 10_000_000`.
5. The L1 tx receives **100× the native resource budget** of an equivalent L2 tx, allowing it to execute computationally intensive operations that would exhaust native resources for any L2 transaction at the same gas price. [1](#0-0) [6](#0-5)
