### Title
WalletConnect `chia_pushTransactions` / `chia_pushTx` Confirmation Dialog Renders Opaque Spend Bundle JSON, Enabling Malicious Transaction Approval — (File: `packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

The WalletConnect confirmation dialogs for `chia_pushTransactions` (`chia_wallet.push_transactions`) and `chia_pushTx` (`chia_full_node.push_tx`) render the entire transaction bundle / spend bundle as a raw JSON blob. No human-readable decoding of coin IDs, amounts, or destination addresses is performed. A malicious dapp can craft a spend bundle that transfers the user's XCH, CAT, or NFT to an attacker-controlled address; the user has no means to detect this from the confirmation dialog before clicking "Push."

---

### Finding Description

The `chia_wallet.push_transactions` command schema declares the `transactions` parameter with `type: 'json'`: [1](#0-0) 

The `chia_full_node.push_tx` command similarly declares `spend_bundle` with `type: 'json'`: [2](#0-1) 

The `humanizeDappCommand` function resolves all parameter rows by calling `humanizeParams` against the schema: [3](#0-2) 

For `type: 'json'` parameters, `humanizeParams` serialises the value as a raw JSON string — there is no special-case handler that decodes a Chia spend bundle or transaction array into human-readable coin IDs, amounts, or puzzle hashes. The only field that receives humanisation is the optional `fee` (declared `humanize: 'mojo-to-xch'`). [4](#0-3) 

The `Confirm` dialog renders these rows verbatim: [5](#0-4) 

The `ConfirmDisplay` type supports a `walletDelta` section that could show a human-readable balance change, but neither `chia_pushTransactions` nor `chia_pushTx` has a custom `handler` that would compute or populate it: [6](#0-5) 

The full dispatch path in `main.tsx` confirms that, absent a `handler`, the command is forwarded directly to the wallet/full-node daemon with `parsedParams` unchanged, after the user clicks "Push": [7](#0-6) 

The `parseDappParams` test confirms that a dapp can supply arbitrary nested spend-bundle data (e.g. `aggregatedSignature`, coin spends) and it passes through the allowlist unchanged: [8](#0-7) 

---

### Impact Explanation

A malicious dapp that has obtained `chia_pushTransactions` or `chia_pushTx` permission can submit a spend bundle that:

- Spends the user's XCH/CAT/NFT coins to an attacker-controlled puzzle hash.
- Includes a small, legitimate-looking fee (the only humanised field) to avoid suspicion.

The user sees "Transactions: [opaque JSON blob]" and "Fee: 0.001 XCH" in the confirmation dialog. Because the spend bundle is not decoded into coin IDs, amounts, or destination addresses, the user cannot distinguish a benign transaction from one that drains their wallet. Clicking "Push" executes the malicious spend bundle on-chain.

This maps to the allowed High impact: *"unsafe trust of WalletConnect state that causes a user to approve… the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation

The attacker must first obtain a WalletConnect pairing and have `chia_pushTransactions` or `chia_pushTx` included in the approved command list. Both are standard WalletConnect flows; the Chia GUI explicitly exposes these commands to dapps. Once permission is granted, every subsequent call to `chia_pushTransactions` triggers only the opaque JSON confirmation — no additional privilege is required.

---

### Recommendation

1. **Decode spend bundles in the confirmation dialog.** For `transactions` and `spend_bundle` parameters, parse each `CoinSpend` entry and display the coin ID, puzzle hash, and amount in human-readable form (mojos → XCH/CAT).
2. **Populate `walletDelta` via a daemon call.** Before showing the confirmation dialog, call the wallet daemon to simulate the spend bundle and compute the net balance change per asset; render it in the existing `WalletDeltaSection`.
3. **Alternatively, restrict `chia_pushTransactions` to trusted/developer contexts** and require an explicit, separate high-privilege permission tier that warns users this command bypasses all amount/address validation.

---

import { getDappCommandSchema } from './getDappCommandSchema';
import { humanizeParams } from './humanizeParams';

export async function humanizeDappCommand(dappCommand: string, data: Record<string, unknown>, networkPrefix?: string) {
  const dappCommandSchema = getDappCommandSchema(dappCommand);

  const rows = await humanizeParams(dappCommandSchema.params, data, networkPrefix);

  return {
    destructive: dappCommandSchema.destructive === true,
    title: dappCommandSchema.title(),
    message: dappCommandSchema.message(),
    confirmLabel: dappCommandSchema.confirmLabel(),
    rows,
  };
}
