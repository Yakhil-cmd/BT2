### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Fee Cap via Unchecked `setPoolBinAdditionalFees` — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` applies no cap check on `addFeeBuyE6` / `addFeeSellE6`, while the structurally identical `setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8`. A pool admin can therefore set per-bin additional fees up to the `uint16` maximum (65 535 E6 ≈ 6.55%) on top of the already-capped base spread, making the effective swap fee exceed the protocol-enforced ceiling and causing traders to pay more than the cap permits.

---

### Finding Description

The factory exposes two admin-gated fee setters. `setPoolAdminFees` explicitly guards both components against the factory-wide caps:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, by contrast, passes the caller-supplied values straight through to the pool with no bounds check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

During every swap the pool adds the per-bin additional fee directly on top of the oracle-derived base fee before computing the trade:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [3](#0-2) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [4](#0-3) 

The `BinState` struct stores `addFeeBuyE6` and `addFeeSellE6` as `uint16`, giving a maximum value of 65 535 E6 ≈ 6.55%: [5](#0-4) 

The hard cap `maxOwnerSpreadCapE6` is 20% (200 000 E6): [6](#0-5) 

With the base spread already at the 20% cap, a pool admin can push the effective per-bin fee to ≈ 26.55%, a 32.7% overshoot of the protocol ceiling, with no revert.

---

### Impact Explanation

Traders swapping through a bin with `addFeeBuyE6 = 65535` or `addFeeSellE6 = 65535` pay a total spread fee that exceeds `maxAdminSpreadFeeE6`. The excess fee is retained in the bin as LP revenue, so LPs benefit at the direct expense of traders. This is a broken admin-boundary invariant: the pool admin exceeds the fee cap the factory owner set, causing direct loss of trader principal on every affected swap.

---

### Likelihood Explanation

The trigger requires only the pool admin (a semi-trusted role) to call `setPoolBinAdditionalFees` with a `uint16` value above the effective cap remainder. No special conditions, timelocks, or external state are needed. The call is permissioned only by `onlyPoolAdmin`, which is a normal operational role. Any pool admin who wishes to extract additional value from traders — or who is compromised — can execute this immediately.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the guard in `setPoolAdminFees`. The simplest fix is to enforce that the per-bin additional fee does not exceed `maxAdminSpreadFeeE6`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce that `baseFee + addFee ≤ maxAdminSpreadFeeE6` at the pool level in `setBinAdditionalFees`.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20%) and `maxAdminNotionalFeeE8 = 0`.
2. Pool is created with `adminSpreadFeeE6 = 200_000` (at the cap).
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`. No revert occurs.
4. A trader calls `swap` through bin 0. The effective buy fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately 26.55% total — exceeding the 20% cap.
5. The trader receives fewer tokens than the fee cap permits; the excess is retained in the bin as LP profit. [2](#0-1) [7](#0-6)
