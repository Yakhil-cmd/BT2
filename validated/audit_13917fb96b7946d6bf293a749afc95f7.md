Audit Report

## Title
Unbounded Synchronous BFS in SNS Governance Vote Cascade Exceeds Instruction Limit - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS Governance canister's `cast_vote_and_cascade_follow` is a plain synchronous `fn` that performs an unbounded BFS over the entire follower graph with no instruction-limit guard. Any neuron holder with the `Vote` permission can trigger this via the public `manage_neuron` ingress endpoint. With a sufficiently large follower graph, the message exceeds the IC's 40 billion instruction hard limit, traps, rolls back the vote, and permanently prevents cascade voting on that proposal.

## Finding Description

**Root cause:** `cast_vote_and_cascade_follow` at [1](#0-0)  is declared as a plain `fn` (not `async fn`). Its BFS loop at [2](#0-1)  contains no call to any instruction-counter check. A grep across the entire SNS governance codebase for `is_message_over_threshold`, `noop_self_call_if_over_instructions`, `HARD_VOTING`, or `SOFT_VOTING` returns zero matches.

**Call path:** The public `#[update] manage_neuron` endpoint at [3](#0-2)  calls `governance.manage_neuron`, which calls `register_vote`, which calls `Governance::cast_vote_and_cascade_follow` synchronously at [4](#0-3) . There is no `await` point between ingress receipt and the BFS completing.

**Contrast with NNS:** The NNS equivalent is `async fn cast_vote_and_cascade_follow` at [5](#0-4) , which uses a `ProposalVotingStateMachine`, checks `over_soft_message_limit()` inside the BFS at [6](#0-5) , and defines explicit soft/hard limits at [7](#0-6) . The SNS received no analogous refactor.

**Hard limit:** The IC enforces `MAX_INSTRUCTIONS_PER_MESSAGE = 40 * B` (40 billion instructions) at [8](#0-7) . Exceeding it traps the message and rolls back all state changes.

## Impact Explanation

When the BFS traversal over a large follower graph exceeds 40 billion instructions, the message traps. The vote is rolled back — the voting neuron's ballot remains `Unspecified`. Because the follower graph is unchanged, every subsequent `RegisterVote` call by that neuron on that proposal will also trap. This permanently prevents cascade voting on the proposal, blocking quorum from being reached and freezing SNS governance decisions. This matches the allowed High impact: **"Significant SNS security impact with concrete user or protocol harm"** and **"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS."**

## Likelihood Explanation

The attacker only needs the standard, unprivileged `Vote` permission on any neuron in the SNS. The `manage_neuron` endpoint is publicly reachable via ingress. No governance majority, special keys, or node control is required. The trigger condition — a hub neuron followed by a large number of other neurons — arises organically as SNS DAOs grow, and can also be deliberately constructed by an attacker who creates many neurons and sets them to follow a target neuron. The NNS governance team explicitly recognized this class of bug as production-critical (evidenced by the NNS refactor), confirming the realistic likelihood at scale.

## Recommendation

Refactor SNS `cast_vote_and_cascade_follow` to mirror the NNS implementation:
1. Convert it to an `async fn` and introduce a `VotingStateMachine` that checkpoints BFS progress in stable storage.
2. Add soft/hard instruction-limit checks inside the BFS loop, analogous to `over_soft_message_limit()` / `noop_self_call_if_over_instructions` used in `rs/nns/governance/src/voting.rs`.
3. When the soft limit is reached, persist remaining BFS state and offload continuation to a timer task, as done in NNS governance.

## Proof of Concept

1. Deploy an SNS with N neurons (e.g., N = 50,000) where neurons 2..N all follow neuron 1 on a given proposal action type.
2. Create a proposal of that action type; all N neurons receive ballots.
3. As neuron 1's controller, submit `manage_neuron { RegisterVote { proposal_id, vote: Yes } }`.
4. The SNS governance canister enters `cast_vote_and_cascade_follow` and iterates over all N−1 followers in a single synchronous BFS with no instruction guard.
5. The message traps with `CanisterInstructionLimitExceeded`; the vote is rolled back; neuron 1's ballot remains `Unspecified`.
6. All subsequent `RegisterVote` calls by neuron 1 on this proposal also trap (follower graph unchanged), permanently blocking cascade voting.

A deterministic integration test using PocketIC can reproduce this by creating 50,000 neurons with the described following topology and asserting that `manage_neuron { RegisterVote }` returns a trap error rather than `Ok`.
