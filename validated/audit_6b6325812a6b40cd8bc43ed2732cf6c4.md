### Title
Signature Validation Bypass via `skip_stateful_validations` for Undeployed Accounts — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (which performs signature verification) for invoke transactions with nonce=1 targeting undeployed accounts, based solely on whether **any** transaction for that account exists in the mempool. An unprivileged attacker who observes a pending `deploy_account` for address X can submit an invoke from X with nonce=1 and an entirely invalid signature; the gateway admits it to the mempool without ever calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` (lines 429–461) returns `true` — causing `run_validate_entry_point` to set `execution_flags.validate = false` and skip `__validate__` — when all three conditions hold:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The code comment claims this is safe because the mempool entry "means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, `account_tx_in_pool_or_recent_block` checks only whether **any** transaction for the address exists in the pool or recent block state — it does not verify the type is `DeployAccount`: [2](#0-1) 

The gateway's `run_pre_validation_checks` calls `validate_by_mempool` (which checks nonce gap and fee escalation only, not signatures) and then `skip_stateful_validations`. When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

The `StatefulValidator::validate` path in blockifier then short-circuits before calling `__validate__`: [4](#0-3) 

The transaction is then forwarded to the mempool via `mempool_client.add_tx`: [5](#0-4) 

---

### Impact Explanation

An attacker can inject an invoke transaction carrying an **arbitrary/invalid signature** into the mempool for any undeployed account that has a pending `deploy_account` transaction. This matches the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Concrete consequences:
- **Mempool pollution**: The attacker floods the mempool with signature-invalid invokes for accounts about to be deployed.
- **Transaction censorship via fee escalation**: If the attacker's invalid invoke has the same `(address, nonce=1)` as the legitimate user's invoke and offers a higher fee, the mempool's fee-escalation logic replaces the legitimate invoke with the attacker's. The legitimate user's transaction is evicted. The attacker's transaction is later rejected during block execution (because `__validate__` runs with `validate: true` in `new_for_sequencing`), but the legitimate transaction is already gone. [6](#0-5) 

---

### Likelihood Explanation

- Requires observing a pending `deploy_account` transaction for a target address (visible in the public mempool).
- No privileged access needed; any unprivileged caller of the gateway's `add_tx` endpoint can trigger this.
- The deploy_account + invoke UX pattern is explicitly documented and tested as a supported flow, making target accounts predictable. [7](#0-6) 

---

### Recommendation

1. **Verify transaction type in the mempool check**: Instead of `account_tx_in_pool_or_recent_block`, query specifically whether a `DeployAccount` transaction exists for the address. This closes the gap between the comment's stated invariant and the actual check.

2. **Alternatively, verify the deploy_account hash**: Pass the expected `deploy_account` transaction hash (as `native_blockifier`'s `PyValidator::should_run_stateful_validations` does via `deploy_account_tx_hash`) and confirm it is present in the mempool before skipping validation. [8](#0-7) 

3. **Add a test**: Cover the case where `account_tx_in_pool_or_recent_block` returns `true` due to a non-deploy-account transaction, and assert that `skip_validate` is `false`.

---

### Proof of Concept

```
1. Alice submits RpcDeployAccountTransaction for address X (nonce=0).
   → Accepted into mempool; account_tx_in_pool_or_recent_block(X) = true.

2. Attacker submits RpcInvokeTransaction from X with:
     nonce = 1
     signature = [0x0, 0x0]   // invalid

3. Gateway stateful validation:
     account_nonce = 0  (X not deployed)
     tx.nonce()    = 1
     account_tx_in_pool_or_recent_block(X) = true  ← Alice's deploy_account
   → skip_stateful_validations returns true
   → run_validate_entry_point sets execution_flags.validate = false
   → __validate__ is NOT called; invalid signature is never checked.

4. Attacker's invoke is forwarded to mempool.add_tx.
   If attacker's tip > Alice's invoke tip, fee escalation replaces Alice's invoke.

5. Block execution:
     Alice's deploy_account executes (account deployed).
     Attacker's invoke executes with validate=true → __validate__ fails → rejected.
     Alice's invoke is gone from the mempool; she must resubmit.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
