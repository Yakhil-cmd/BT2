### Title
Pool Contract `claim_rewards` Executes During Paused Staking State - (File: src/pool/pool.cairo)

### Summary
The `Pool` contract has no pause mechanism and does not check the `Staking` contract's pause state. The `Pool.claim_rewards()` function can be called by any pool member or reward address while the `Staking` contract is paused, because it operates entirely on the Pool's own internal state without making any call into the Staking contract.

### Finding Description
The `Staking` contract implements a pause mechanism via `IStakingPause` and enforces it across all state-changing functions. Every function in `Staking` that moves funds or changes state checks the pause flag and reverts with `"Contract is paused"`. [1](#0-0) 

The `Pool` contract, however, has **no pause mechanism whatsoever** — no `is_paused` storage variable, no pause check, and no call to the Staking contract's `is_paused()` before executing state changes. [2](#0-1) 

The `Pool.claim_rewards()` function calculates rewards entirely from the Pool's own `cumulative_rewards_trace` and transfers STRK directly to the `reward_address` without ever calling into the Staking contract: [3](#0-2) 

The reward token transfer at line 365–366 moves STRK out of the Pool contract with no pause guard: [4](#0-3) 

By contrast, every Pool function that *does* call into Staking (`enter_delegation_pool`, `add_to_delegation_pool`, `exit_delegation_pool_intent`, `exit_delegation_pool_action`, `switch_delegation_pool`) will fail during a pause because the Staking-side entry points are guarded. Only `claim_rewards` — which is self-contained — succeeds. [5](#0-4) 

### Impact Explanation
**High — Theft of unclaimed yield.**

The pause is the protocol's emergency stop. If a reward inflation bug causes the `cumulative_rewards_trace` inside Pool contracts to be inflated before the security agent pauses Staking, pool members can immediately call `Pool.claim_rewards()` to drain those inflated rewards. The pause on the Staking contract does not prevent this because `claim_rewards` never touches the Staking contract. STRK is transferred out of the Pool to reward addresses, and the protocol cannot recover those funds.

### Likelihood Explanation
**Medium.** Any pool member or reward address (unprivileged callers) can call `Pool.claim_rewards()` at any time, including during a pause. The only prerequisite is that rewards have already been credited to the Pool's `cumulative_rewards_trace` before the pause was triggered — a realistic condition since rewards accumulate continuously and a pause is typically reactive (triggered after a bug is noticed).

### Recommendation
Add a pause-awareness check to `Pool.claim_rewards()` (and any other Pool function that moves funds). The Pool should either:
1. **Short-term**: Query `IStakingDispatcher { contract_address: staking_contract }.is_paused()` at the top of `claim_rewards` and revert if `true`.
2. **Long-term**: Give the Pool contract its own pause storage variable that mirrors the Staking contract's pause state, updated via a dedicated callback or by the same security agent role.

### Proof of Concept
1. Pool member Alice has accrued rewards stored in the Pool's `cumulative_rewards_trace`.
2. A reward inflation bug is discovered; the security agent calls `Staking.pause()`.
3. Alice calls `Pool.claim_rewards(pool_member: alice)`.
4. The function reads `cumulative_rewards_trace` (Pool-local storage), calculates inflated rewards, and calls `reward_token.checked_transfer(recipient: alice_reward_address, amount: inflated_rewards)`.
5. The call succeeds — no pause check exists in `Pool.claim_rewards()`.
6. Alice receives inflated STRK rewards despite the Staking contract being paused. [3](#0-2) [6](#0-5)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
