### Title
`skip_stateful_validations` admits unsigned invoke transactions for any account with a pending deploy-account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the blockifier `__validate__` entry-point call (the account's signature check) for any invoke transaction whose nonce is 1, whose sender's on-chain nonce is 0, and whose sender address appears in the mempool or a recent block. Because the only prerequisite is the presence of *any* transaction from that address in the mempool, an unprivileged attacker who observes a victim's deploy-account transaction can immediately submit an invoke transaction with an arbitrary/invalid signature on behalf of the victim's not-yet-deployed account, and the gateway will accept it into the mempool without ever verifying the signature.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip blockifier validation) when all three conditions hold:

1. The transaction is an `ExecutableTransaction::Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)`.
3. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, so the blockifier's `StatefulValidator::perform_validations` returns immediately without calling `__validate__`: [3](#0-2) [4](#0-3) 

**Why the prerequisite check is insufficient**

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in the mempool pool *or* in the mempool's committed-nonce state: [5](#0-4) [6](#0-5) 

The comment in the gateway code says this is "sufficient" because it implies a deploy-account tx exists. However, the check is on the *sender address of the attacker's invoke tx*, not on the type of transaction already in the pool. Any transaction from that address — including the victim's own deploy-account tx — satisfies the condition.

**No existing guard catches the invalid signature**

- The stateless validator only checks signature *size*, not cryptographic validity.
- The mempool's `validate_tx` checks only nonce ordering, duplicate tx hash, and fee escalation — no signature field is present in `ValidationArgs`: [7](#0-6) [8](#0-7) 

**End-to-end exploit path**

1. Victim submits a deploy-account transaction for a new address `A`. The mempool now contains a tx from `A`, so `account_tx_in_pool_or_recent_block(A)` returns `true`.
2. Attacker submits an invoke transaction: `sender_address = A`, `nonce = 1`, `signature = [0x41, 0x41]` (arbitrary, valid-sized bytes).
3. Gateway stateless check passes (signature size is within limit).
4. `extract_state_nonce_and_run_validations` reads on-chain nonce for `A` = 0.
5. `run_pre_validation_checks` → `validate_state_preconditions` passes (nonce 1 ≥ 0, within gap), `validate_by_mempool` passes (nonce not duplicate yet, fee ok).
6. `skip_stateful_validations` returns `true` because nonce=1, account_nonce=0, and `A` is in the mempool.
7. `run_validate_entry_point` is called with `skip_validate = true` → `validate = false` → `__validate__` is never executed.
8. The invalid invoke tx is forwarded to the mempool via `add_tx`. [9](#0-8) 

---

### Impact Explanation

**Mempool/gateway admission accepts invalid transactions.** The gateway accepts an invoke transaction whose signature has never been verified against the account's public key. This maps directly to:

> *High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

Secondary consequence: the attacker's invalid nonce-1 tx occupies the nonce-1 slot for the victim's account. With fee escalation disabled (`enable_fee_escalation = false`, the default), the victim's legitimate nonce-1 invoke tx is rejected with `MempoolError::DuplicateNonce`: [10](#0-9) 

This allows an attacker to continuously grief every new account deployment by front-running the first post-deploy invoke transaction with an unsigned decoy, forcing the victim to wait for the decoy to fail at execution time before retrying.

---

### Likelihood Explanation

High. The attack requires no privileged access. The attacker only needs to:
- Watch the public mempool for deploy-account transactions (observable via the gateway's P2P propagation or RPC).
- Submit a crafted invoke tx with nonce=1 and any valid-sized signature.

The window is open from the moment the deploy-account tx enters the mempool until the block containing it is committed.

---

### Recommendation

Add an ownership/authorization check inside `skip_stateful_validations` analogous to the `_isApprovedOrOwner` fix in the external report. The simplest correct fix is to require that the transaction carries a valid signature before skipping blockifier validation, or to narrow the skip condition so it only applies when the gateway itself received the deploy-account tx in the same request batch (i.e., tie the skip to a session-scoped token rather than a mempool presence check). A minimal patch:

```rust
// In skip_stateful_validations, before returning true,
// run a lightweight ECDSA pre-check or require the caller
// to supply the deploy_account_tx_hash that was co-submitted.
// Do NOT skip __validate__ based solely on mempool presence.
```

Alternatively, mirror the `PyValidator::should_run_stateful_validations` approach, which requires the caller to explicitly supply the `deploy_account_tx_hash` as proof of co-submission: [11](#0-10) 

The gateway's `skip_stateful_validations` should adopt the same pattern — only skip when the deploy-account tx hash is explicitly provided by the submitter and matches a known pending tx, rather than relying on a mempool presence check that any observer can satisfy.

```
  "resource_bounds": { ... },          # meets min_gas_price

}

```

The gateway returns a success response. The invalid tx now occupies nonce-1 for address `A`. The victim's legitimate nonce-1 invoke tx is subsequently rejected with `DuplicateNonce` until the attacker's tx is executed and fails in the next block.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
```
