### Title
Cross-Chain Replay of `DelegateAction` Signatures Due to Missing Chain-ID Binding — (`File: core/primitives/src/action/delegate.rs`)

### Summary

`DelegateAction` (meta-transaction) signatures are computed over a payload that contains no chain identifier. The `SignableMessage` discriminant is a fixed NEP number, identical across every NEAR network (mainnet, testnet, betanet, localnet). A malicious relayer who receives a `SignedDelegateAction` intended for one network can submit it on any other network where the sender's account and key exist and the nonce is valid, executing the inner actions without the sender's consent on the wrong chain.

### Finding Description

`DelegateAction` is the meta-transaction primitive introduced by NEP-366. A user signs a `DelegateAction` off-chain and hands it to a relayer, who wraps it in an outer `SignedTransaction` and submits it on-chain.

The signed payload is produced by `get_nep461_hash()`:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
``` [1](#0-0) 

`SignableMessage` prepends a `MessageDiscriminant` — a fixed `u32` derived from the NEP number (366 or 611):

```rust
const NEP_366_META_TRANSACTIONS: u32 = 366;
// discriminant = MIN_ON_CHAIN_DISCRIMINANT + 366  (same on every NEAR network)
``` [2](#0-1) 

The `DelegateAction` struct itself contains no chain ID, genesis hash, or any network-scoping field:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
``` [3](#0-2) 

`apply_delegate_action` verifies the signature with `signed_delegate_action.verify()`, which only checks the NEP discriminant + content hash — no chain ID:

```rust
if !signed_delegate_action.verify() {
    result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
    return Ok(());
}
``` [4](#0-3) 

The outer `SignedTransaction` does bind to a specific chain via `block_hash` (checked by `check_transaction_validity_period`), but the inner `DelegateAction` signature is independent of the outer transaction. A relayer constructs the outer transaction; the user only signs the inner `DelegateAction`. The relayer can therefore wrap the same `SignedDelegateAction` in a fresh outer transaction pointing to any chain. [5](#0-4) 

### Impact Explanation

A malicious relayer who receives a `SignedDelegateAction` signed for testnet can submit it on mainnet (or any other NEAR network) by wrapping it in a new outer transaction with a valid mainnet `block_hash`. If the sender's account exists on the target chain with the same public key and a nonce lower than the one in the `DelegateAction`, the inner actions execute without the sender's authorization for that chain. Possible consequences include unauthorized token transfers, access key additions or deletions, contract deployments, and arbitrary function calls — all attributed to the sender's account on the wrong chain.

**Exact corrupted value:** The sender's account state on the unintended chain — balance, access keys, contract code, or contract storage — is modified by actions the sender never authorized for that chain.

### Likelihood Explanation

- Many users share the same key pair across testnet and mainnet (common practice for developers and power users).
- Relayers are third-party services; the protocol explicitly requires users to trust them to submit to the correct chain.
- The nonce condition (`DelegateAction.nonce > current_ak_nonce` on the target chain) is frequently satisfiable, especially for accounts that are less active on the target chain.
- `max_block_height` limits the time window but does not prevent cross-chain replay within that window.

### Recommendation

Include the chain's genesis hash (or a dedicated chain-ID field) in the `DelegateAction` signed payload, analogous to how `SignedTransaction` binds to a specific chain via `block_hash`. Concretely, add a `chain_id: CryptoHash` field to `DelegateAction` (and `DelegateActionV2`) and incorporate it into `get_nep461_hash()`. Alternatively, include the genesis block hash in the `SignableMessage` wrapper so that the discriminant is chain-scoped. This is the NEAR analog of the Ethereum recommendation to include `chainID` in all signature schemes.

### Proof of Concept

1. Alice holds the same ED25519 key on both testnet and mainnet, with mainnet access-key nonce = 5.
2. Alice signs a `DelegateAction` for testnet: `nonce = 6`, `max_block_height = 9_999_999`, `actions = [Transfer { deposit: 100 NEAR }]`, `receiver_id = "eve.near"`.
3. Alice hands the `SignedDelegateAction` to a relayer she believes is testnet-only.
4. The malicious relayer constructs a new outer `SignedTransaction` on mainnet: `block_hash` = current mainnet head hash, `receiver_id = alice.near`, wrapping Alice's `SignedDelegateAction`.
5. `apply_delegate_action` on mainnet calls `signed_delegate_action.verify()` → passes (same NEP discriminant, same key, same payload hash).
6. Nonce check: `6 > 5` → passes.
7. `max_block_height` check: mainnet block height < 9_999_999 → passes.
8. 100 NEAR is transferred from Alice's mainnet account to `eve.near` without Alice's consent. [6](#0-5) [7](#0-6)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }

    // Generate a new receipt from DelegateAction.
    let new_receipt = Receipt::V0(ReceiptV0 {
        predecessor_id: sender_id.clone(),
        receiver_id: delegate_action.receiver_id().clone(),
        receipt_id: CryptoHash::default(),

        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: action_receipt.signer_id().clone(),
            signer_public_key: action_receipt.signer_public_key().clone(),
            gas_price: action_receipt.gas_price(),
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: delegate_action.get_actions(),
        }),
    });

    // Note, Relayer prepaid all fees and all things required by actions: attached deposits and attached gas.
    // If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction.
    // Gas is refunded to the signer, this is Relayer.
    // Some contracts refund the deposit. Usually they refund the deposit to the predecessor and this is sender_id/Sender from DelegateAction.
    // Therefore Relayer should verify DelegateAction before submitting it because it spends the attached deposit.

    let prepaid_send_fees = total_prepaid_send_fees(&apply_state.config, action_receipt.actions())?;
    let required_cost = receipt_required_cost(apply_state, &new_receipt)?;
    // This gas will be burnt by the receiver of the created receipt.
    // Compute costs of that are not relevant at this point, the "used" gas is
    // only reserved for execution later, potentially on a different shard.
    result.gas_used = result.gas_used.checked_add_result(required_cost.gas)?;
    // This gas was prepaid on Relayer shard. Need to burn it because the receipt is going to be sent.
    // gas_used is incremented because otherwise the gas will be refunded. Refund function checks only gas_used.
    result.gas_used = result.gas_used.checked_add_result(prepaid_send_fees.gas)?;
    result.gas_burnt = result.gas_burnt.checked_add_result(prepaid_send_fees.gas)?;
    result.compute_usage = safe_add_compute(result.compute_usage, prepaid_send_fees.compute)?;
    result.new_receipts.push(new_receipt);

    Ok(())
}
