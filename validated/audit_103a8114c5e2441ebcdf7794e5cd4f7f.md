### Title
Single Malicious Participant at Index > Threshold Can Abort All Honest Parties via Inconsistent `big_r_p` — (`src/ecdsa/robust_ecdsa/presign.rs`)

### Summary

In `do_presign`, the exponent interpolation check unconditionally aborts the entire protocol when any participant at sorted index > `threshold` sends a `big_r_p` that does not lie on the polynomial interpolated from the first `threshold+1` values. There is no mechanism to identify and exclude the offending participant and continue. A single malicious participant at index > `threshold` can therefore permanently deny presigning to all honest parties.

### Finding Description

The `presign` entry point enforces that the participant count equals exactly `2*max_malicious + 1`: [1](#0-0) 

Inside `do_presign`, after collecting all `big_r_p` values in Round 2, the code splits participants into two groups:

- **Basis group** (indices `0..=threshold`): their `big_r_p` values are accepted unconditionally as the interpolation basis.
- **Check group** (indices `threshold+1..2*threshold`): each value is verified against the polynomial defined by the basis group. [2](#0-1) 

If any check-group participant's `big_r_p` does not match the interpolated value, the protocol immediately returns an error: [3](#0-2) 

There is no logic to:
1. Record *which* participant failed the check.
2. Exclude that participant and re-run interpolation with the remaining honest parties.
3. Produce a partial result or signal a retry without the offender.

### Impact Explanation

A malicious participant whose sorted ID places them at index > `threshold` sends a random group element as `big_r_p`. Every honest party independently runs the same check, all see the mismatch, and all abort with `AssertionFailed("Exponent interpolation check failed.")`. Because the protocol requires exactly `2t+1` participants and provides no exclusion/retry path, the attacker can repeat this on every presign attempt, permanently denying presigning.

This matches the allowed High impact: **Permanent denial of presigning for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation

- The attacker only needs to be one of the `2t+1` participants with a sorted ID placing them at index > `threshold`. With `t ≥ 1`, there are `t` such slots.
- No cryptographic assumption needs to be broken; the attacker simply sends an arbitrary group element.
- The attack is repeatable on every presign invocation.
- The participant's identity is not revealed by the abort error, so attribution is difficult.

### Recommendation

The check group loop should record the identity of each failing participant rather than aborting immediately. After the loop, if any participants failed, they should be excluded and the protocol should either:
- Re-run interpolation using only the remaining honest participants (if enough remain), or
- Return a structured error that names the offending participants so the caller can retry without them.

This is the standard "identify-and-exclude" pattern used in robust MPC protocols.

### Proof of Concept

With `threshold = 2` (so `n = 5` participants), participant at sorted index 3 (index > 2) sends a random point as `big_r_p`:

```
participants sorted: [P0, P1, P2, P3, P4]
basis group (indices 0..=2): P0, P1, P2  -- accepted unconditionally
check group (indices 3..4):  P3, P4      -- verified against interpolation

Malicious P3 sends big_r_p = random_group_element (not on the polynomial)

do_presign line 208: big_r_i != verifying_share  --> true
do_presign line 209: return Err(AssertionFailed("Exponent interpolation check failed."))

All 5 honest parties abort. Protocol cannot complete.
```

The check at line 208 correctly detects the deviation but incorrectly treats detection as a reason to abort the entire protocol rather than to exclude the offender. [4](#0-3)
