### Title
Pre-EIP-155 Legacy Transaction Accepted Without Chain-ID Binding Enables Cross-Chain Replay — (`engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

---

### Summary

Aurora Engine accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`) that carry no chain-ID in their signed payload. The chain-ID validation in `submit_with_alt_modexp` is guarded by an `if let Some(chain_id)` branch, so when `chain_id` is `None` the check is silently skipped. Any such transaction is therefore valid on **every** EVM chain that accepts pre-EIP-155 signatures, including Ethereum mainnet, making it trivially replayable by an unprivileged attacker.

---

### Finding Description

**Root cause — missing chain-ID binding in the signed structure**

`LegacyEthSignedTransaction::sender()` in `engine-transactions/src/legacy.rs` extracts the chain ID from `v`:

```rust
27..=28 => (
    None,
    u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
),
```

When `v ∈ {27, 28}` the function returns `chain_id = None` and calls `rlp_append_unsigned` with `chain_id = None`, so the chain ID is **not included in the signed message hash**. [1](#0-0) 

`NormalizedEthTransaction` propagates this as `chain_id: Option<u64>`: [2](#0-1) 

And the conversion from a legacy transaction preserves `None`: [3](#0-2) 

**Root cause — chain-ID validation is conditional**

In `submit_with_alt_modexp`, the only chain-ID check is:

```rust
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

When `chain_id` is `None` the entire guard is skipped; the transaction is accepted regardless of which chain it was originally intended for. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Aurora users routinely reuse the same private key / address on Ethereum mainnet and Aurora (both are EVM-compatible). If a user submits a pre-EIP-155 transaction on Aurora (e.g., an ETH transfer or an ERC-20 approval), an attacker who observes it can broadcast the identical raw bytes to Ethereum mainnet. Because the signed payload contains no chain-ID, the signature is equally valid there. If the sender's nonce on Ethereum mainnet matches, the transaction executes and the user's mainnet funds are drained. The reverse direction (Ethereum → Aurora replay) is equally possible.

---

### Likelihood Explanation

- Pre-EIP-155 transactions are still produced by some wallets, hardware signers, and scripts that omit the chain-ID field.
- Aurora explicitly accepts them (the `27..=28` branch is not rejected anywhere in the pipeline).
- Ethereum mainnet also accepts pre-EIP-155 transactions, so the replay target is always available.
- Nonce alignment is the only practical barrier; for a fresh account or a first transaction it is trivially satisfied.

---

### Recommendation

Reject pre-EIP-155 legacy transactions outright by treating `v ∈ {27, 28}` as an error in `submit_with_alt_modexp`, or equivalently change the chain-ID guard to:

```rust
match transaction.chain_id {
    None => return Err(EngineErrorKind::InvalidChainId.into()),
    Some(chain_id) if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) => {
        return Err(EngineErrorKind::InvalidChainId.into());
    }
    _ => {}
}
```

This mirrors the approach taken by EIP-2930 and EIP-1559 transactions, which always embed a mandatory `chain_id` field. [5](#0-4) 

---

### Proof of Concept

1. Alice holds 1 ETH on both Ethereum mainnet (nonce = 5) and Aurora mainnet (nonce = 5).
2. Alice signs a pre-EIP-155 legacy transaction (`v = 27`) on Aurora: `{nonce:5, to: Bob, value: 1 ETH}`.
3. Aurora Engine accepts it (chain-ID check skipped because `chain_id = None`); Alice's Aurora ETH is sent to Bob.
4. Attacker takes the identical RLP-encoded bytes and broadcasts them to Ethereum mainnet.
5. Ethereum mainnet also accepts the pre-EIP-155 signature; Alice's mainnet ETH is sent to Bob.
6. Alice has lost 1 ETH on Ethereum mainnet without ever intending to sign a mainnet transaction.

The attacker-controlled entry path is the public `submit` NEAR contract method, which is callable by any unprivileged account. [6](#0-5)
