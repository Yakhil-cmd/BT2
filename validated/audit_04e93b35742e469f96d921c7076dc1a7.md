### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures, Enabling Unauthorized Fee Drain on Victim Accounts - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call for any Invoke transaction with `nonce == 1` when the account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because the check is keyed only on the *sender address* (not on who submitted the deploy-account), an attacker can submit an Invoke transaction with a forged/invalid signature for any victim address that has a pending deploy-account transaction in the mempool. The gateway accepts the transaction without ever running the account's `__validate__` entry point. When the batcher later executes the block, `__validate__` is called with `validate: true` (the default for sequencing), the signature check fails, the transaction reverts, and the victim's account is charged fees for a transaction they never authorized.

### Finding Description

**Invariant broken:** Every Invoke transaction admitted to the mempool must have its signature verified by the account's `__validate__` entry point before admission.

**Root cause — `skip_stateful_validations`:**

```
crates/apollo_gateway/src/stateful_transaction_validator.rs
``` [1](#0-0) 

The function returns `true` (skip validation) when:
1. The transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

Condition 4 is satisfied by *any* transaction from that address in the pool — including a deploy-account transaction submitted by the legitimate owner. The check does not verify that the *attacker's* invoke transaction carries a valid signature.

**How the skip propagates to the blockifier gateway call:** [2](#0-1) 

When `skip_validate = true`, `ExecutionFlags { validate: false, … }` is set, so `StatefulValidator::perform_validations` returns immediately after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point. [3](#0-2) 

**Why the batcher still calls `__validate__`:**

The batcher converts internal consensus transactions via `convert_to_executable_blockifier_tx`, which calls `BlockifierTransaction::new_for_sequencing`: [4](#0-3) 

`new_for_sequencing` always sets `validate: true` and `strict_nonce_check: true`: [5](#0-4) 

So the gateway skips `__validate__` but the batcher runs it. The transaction with the forged signature reverts during block execution, and the fee for the gas consumed up to the revert point is charged from the victim's account.

**Full attack path:**

1. Victim submits a valid `deploy_account` transaction for address `A` → accepted by gateway and mempool.
2. Attacker submits an `Invoke` transaction for address `A`, `nonce = 1`, with an **invalid/forged signature**.
3. Gateway stateless validation: passes (no signature check at this layer).
4. Gateway stateful validation:
   - `validate_state_preconditions`: passes (`nonce = 1` is within the allowed nonce gap for `account_nonce = 0`).
   - `validate_by_mempool`: passes (mempool `validate_tx` checks nonce range, not signatures).
   - `skip_stateful_validations`: returns `true` (nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)` = true because victim's deploy-account is in pool).
   - `run_validate_entry_point` called with `skip_validate = true` → `__validate__` **not called**.
5. Attacker's invalid-signature invoke tx is admitted to the mempool.
6. Batcher executes the block: deploy-account runs first (account deployed), then the attacker's invoke tx runs with `validate: true` → `__validate__` called → signature invalid → **tx reverts**.
7. If the victim's account has sufficient balance, **fees are charged from the victim's account** for a transaction they never authorized.

### Impact Explanation

This matches **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

An unprivileged attacker can cause the gateway to admit an Invoke transaction with an invalid signature into the mempool. The concrete harm is unauthorized fee drain from the victim's account: the batcher executes the forged transaction, `__validate__` fails, the transaction reverts, and the fee for gas consumed during validation is deducted from the victim's balance. The nonce is also incremented and then rolled back on revert, but the fee is not.

### Likelihood Explanation

- The attacker only needs to observe the mempool for pending `deploy_account` transactions (publicly visible).
- The attack window is the time between the victim's `deploy_account` entering the mempool and being committed to a block — a window of at least one block time.
- No privileged access is required; any user can submit an `Invoke` transaction to the gateway.
- The attacker does not need to know the victim's private key; any invalid signature suffices.

### Recommendation

Replace the address-only `account_tx_in_pool_or_recent_block` check with a check that also verifies the signature of the incoming invoke transaction before skipping `__validate__`. Concretely, `skip_stateful_validations` should only return `true` when the gateway can confirm that the *same sender* submitted the deploy-account transaction (e.g., by checking that a deploy-account transaction with the matching sender address and nonce 0 is present in the mempool), **and** should still run a lightweight signature pre-check. Alternatively, remove the `skip_stateful_validations` shortcut entirely and require wallets to submit the deploy-account and invoke transactions in separate RPC calls, or enforce that the gateway always runs `__validate__` regardless of the deploy-account UX feature.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
