### Title
Missing `msg_hash = 0` Validation in OT-Based ECDSA Signing Allows Dangerous Signature Production - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

### Summary

The OT-based ECDSA `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` accepts a zero `msg_hash` without rejection. The sibling Robust ECDSA `sign` function in `src/ecdsa/robust_ecdsa/sign.rs` explicitly rejects `msg_hash = 0` to prevent a documented algebraic attack. The missing guard in the OT-based path allows a malicious or buggy caller to produce a valid threshold ECDSA signature for the zero message hash, which reveals the product `k·x` (nonce times secret key) in the signature scalar `s = R_x · k · x`, enabling secret key extraction if the presignature is ever reused across sessions.

### Finding Description

`src/ecdsa/robust_ecdsa/sign.rs` lines 91–95 explicitly guard against `msg_hash = 0`:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
``` [1](#0-0) 

The OT-based ECDSA `sign` function at `src/ecdsa/ot_based_ecdsa/sign.rs` lines 22–76 performs no equivalent check. It validates participant counts, coordinator membership, and threshold bounds, but passes `msg_hash` directly into the protocol without any zero-value guard: [2](#0-1) 

The signing formula (from `compute_signature_share`) is `s_i = h · k_i + R_x · σ_i`. When `h = 0`, this collapses to `s_i = R_x · σ_i`, and the aggregated `s = R_x · σ` where `σ = k · x` (nonce times secret key). The resulting signature `(R, s)` is a valid ECDSA signature for the zero message hash, and the scalar `s / R_x = k · x` is now public. [3](#0-2) 

The code carries only a documentation warning, not an enforced check:

> **WARNING** You must absolutely hash an actual message before passing it to this function. Allowing the signing of arbitrary scalars *is* a security risk, and this function only tolerates this risk to allow for genericity. [4](#0-3) 

### Impact Explanation

**Critical — Unauthorized creation of a valid threshold signature for attacker-chosen inputs; potential extraction of aggregate secret material.**

When `msg_hash = 0` is passed:

1. All honest participants compute `s_i = R_x · λ_i · σ_i` and send it to the coordinator.
2. The coordinator aggregates `s = R_x · σ` and the final signature `(R, s)` passes the `sig.verify` check at line 129 (ECDSA verification with `h = 0` is algebraically valid).
3. The public signature scalar `s` reveals `k · x = s / R_x`. If the same presignature is ever reused — even in a failed or aborted session — an attacker obtains two equations in `k` and `x` and can solve for the secret key `x`. [5](#0-4) 

### Likelihood Explanation

The OT-based ECDSA `sign` function is a public library API. Any caller — including a malicious application, a compromised orchestration layer, or a buggy integration — can supply `msg_hash = 0`. The Robust ECDSA path enforces the guard at the API boundary; the OT-based path does not. The `RerandomizationArguments::new` constructor also accepts an all-zero `msg_hash: [u8; 32]` without validation, making it straightforward to construct a zero-hash signing context. [6](#0-5) 

### Recommendation

Add the same zero-hash guard to `src/ecdsa/ot_based_ecdsa/sign.rs` immediately after the threshold check, mirroring the Robust ECDSA implementation:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0: signing with a zero hash reveals nonce-key product material".to_string(),
    ));
}
```

Additionally, consider adding a zero-hash check inside `RerandomizationArguments::new` or `derive_randomness` so that the guard is enforced before the presignature is rerandomized, preventing the dangerous context from being constructed at all.

### Proof of Concept

1. Run a standard OT-based ECDSA keygen and presign to obtain `(R, k_i, σ_i)` shares.
2. Call `ot_based_ecdsa::sign::sign(participants, coordinator, threshold, me, pk, presignature, Scalar::ZERO)` — this returns `Ok(protocol)` without error.
3. Execute the protocol. The coordinator receives `s_i = R_x · λ_i · σ_i` from each participant, aggregates `s = R_x · σ`, and the final `sig.verify` passes because ECDSA verification with `h = 0` reduces to `(r/s) · X = R`, which holds when `s = r · x / k`.
4. The public signature scalar `s` satisfies `s / R_x = k · x`. If the same presignature is reused in a second session with any nonzero `h'`, the two equations `s = R_x · k · x` and `s' = k^{-1} · (h' + R_x · x)` yield a quadratic in `k` that recovers the secret key `x`.

The Robust ECDSA path rejects this at step 2 with `"msg_hash cannot be 0"`. The OT-based path does not. [7](#0-6) [8](#0-7)

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<Scalar, ProtocolError> {
    // Round 1
    // Linearize ki
    // Spec 1.1
    let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
    let k_i = lambda * presignature.k;

    // Linearize sigmai
    // Spec 1.2
    let sigma_i = lambda * presignature.sigma;

    // Compute si = h * ki + Rx * sigmai
    // Spec 1.3
    let r = x_coordinate(&presignature.big_r);
    Ok(msg_hash * k_i + r * sigma_i)
