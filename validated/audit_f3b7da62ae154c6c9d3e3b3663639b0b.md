### Title
Unbounded `while` Loop in `Hashchain::move_to_block` Causes NEAR Gas Exhaustion, Temporarily Freezing All Aurora Engine Operations - (File: engine-hashchain/src/hashchain.rs)

### Summary

Every Aurora Engine transaction-processing method (`submit`, `call`, `deploy_code`, `ft_on_transfer`, etc.) calls `load_hashchain`, which invokes `Hashchain::move_to_block` to catch up skipped NEAR blocks. `move_to_block` contains an unbounded `while` loop that executes one `keccak` hash per skipped block. After any period of Aurora inactivity, the accumulated block gap causes this loop to exhaust the NEAR 300 Tgas per-transaction gas limit, making every subsequent transaction fail and temporarily freezing all user funds in the Aurora EVM.

### Finding Description

**Root cause — the unbounded loop:**

In `engine-hashchain/src/hashchain.rs`, `move_to_block` iterates from the stored `current_block_height` up to the current NEAR block height, calling `compute_block_hashchain` (which calls `keccak`) on every iteration:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}

`load_hashchain` is called unconditionally at the start of `with_hashchain` and `with_logs_hashchain`:

**Gas arithmetic:**

The NEAR `keccak256` host function costs approximately 1.5 Tgas per call. With a 300 Tgas per-transaction limit, only ~200 loop iterations are affordable before the transaction is killed with out-of-gas. NEAR produces roughly one block per second, so a gap of ~200 NEAR blocks (~3–4 minutes of Aurora inactivity) is sufficient to make every subsequent transaction fail.

**Exploit path:**

1. Aurora Engine is idle for ≥ ~200 NEAR blocks (e.g., during a maintenance window, upgrade pause, or organic low-activity period).
2. Any unprivileged user submits any transaction: `submit`, `call`, `ft_on_transfer`, etc.
3. `with_logs_hashchain` / `with_hashchain` calls `load_hashchain`, which calls `move_to_block(current_near_block_height)`.
4. The loop runs for the full gap, exhausting NEAR gas before the actual EVM logic executes.
5. The NEAR transaction panics with out-of-gas. The hashchain state is not updated (the write only happens after the loop).
6. Every subsequent transaction repeats step 3–5 because the stored `current_block_height` is never advanced.
7. The engine is permanently stuck until an admin calls `start_hashchain` to reset the hashchain.

### Impact Explanation

All Aurora Engine entry points that process user funds go through `with_hashchain` or `with_logs_hashchain`. Once the block gap exceeds ~200 blocks, **every** call fails at the NEAR gas level before any EVM logic runs:

- `submit` / `submit_with_args` — EVM transactions cannot execute; ETH and ERC-20 transfers are blocked.
- `ft_on_transfer` — NEP-141 bridge deposits into Aurora fail; tokens are returned to sender, but in-flight bridge state may be inconsistent.
- `call` / `deploy_code` — contract interactions and deployments fail.

Users holding ETH or ERC-20 tokens inside the Aurora EVM cannot move or withdraw them. This constitutes **temporary freezing of funds** (High severity). The freeze persists until an admin resets the hashchain via `start_hashchain`, which requires the engine to be paused first.

### Likelihood Explanation

NEAR produces ~1 block/second. A gap of ~200 blocks is only ~3–4 minutes of complete Aurora inactivity. This threshold is easily crossed during:

- Planned maintenance or upgrade windows (the engine is paused, no transactions land).
- Periods of organic low activity on a testnet or newly deployed instance.
- Any network congestion that prevents relayer transactions from landing for a few minutes.

No privileged access is required to trigger the failure — any unprivileged user submitting the first transaction after the gap is sufficient to observe the freeze. The attacker does not need to do anything active; they simply wait.

### Recommendation

Cap the number of iterations in `move_to_block`. If the gap exceeds a safe maximum (e.g., 100 blocks), skip intermediate hashes or store only the final block height without iterating, similar to how the hashchain is reset in `start_hashchain`. Alternatively, limit the loop to a fixed maximum per call and allow catch-up across multiple transactions.

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled.
2. Record the current NEAR block height `H0` at which the hashchain was last written.
3. Wait (or simulate) until the NEAR block height reaches `H0 + 300`.
4. Submit any EVM transaction via `submit(rlp_bytes)`.
5. Observe the NEAR transaction receipt: status is `Failure` with reason `Exceeded the prepaid gas`.
6. Repeat step 4 — every attempt fails identically because `load_hashchain` re-enters the same loop from `H0` each time (the hashchain state is never updated on failure).
7. Confirm that `ft_on_transfer` (bridge deposit) also fails, preventing users from moving funds.

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
