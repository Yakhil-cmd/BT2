### Title
Off-by-One in Upper Threshold Bound Allows `threshold == N`, Contradicting Spec's Strict `threshold < N` Requirement - (File: src/dkg.rs)

---

### Summary

The protocol specification in `docs/dkg.md` mandates the strict inequality `1 < threshold < N` for DKG Round 1 validation. The implementation in `assert_key_invariants` enforces the upper bound with `threshold > participants.len()` (strict greater-than), which permits the boundary value `threshold == N`. This is the same class of off-by-one inconsistency as the NounsDAO M-02 report: the spec uses a strict bound (`<`) but the code uses a non-strict bound (`>`), leaving the boundary value `threshold == N` in an accepted-but-should-be-rejected state.

---

### Finding Description

`docs/dkg.md` line 50 specifies:

> **1.1 Each P_i asserts that 1 < threshold < N.**

The implementation of this check in `assert_key_invariants` (`src/dkg.rs`) is:

```rust
// Step 1.1
// validate threshold
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold,
        max: participants.len(),
    });
}
// Step 1.1
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

The lower bound check (`threshold < 2`) correctly enforces `threshold >= 2`, matching `1 < threshold`. However, the upper bound check (`threshold > participants.len()`) only rejects when `threshold > N`, **not** when `threshold == N`. The spec requires `threshold < N` (strict), so `threshold == N` must also be rejected. The code comment even says `// Step 1.1`, acknowledging it is implementing the spec constraint, yet the operator is wrong.

This function is the sole gatekeeper for both `keygen` and `reshare`/`refresh`:

```rust
// src/lib.rs line 99
let participants = assert_key_invariants(participants, me, threshold)?;
```

```rust
// src/dkg.rs line 649
let participants = assert_key_invariants(participants, me, threshold)?;
```

The same off-by-one exists in `validate_triple_inputs` (`src/ecdsa/ot_based_ecdsa/triples/generation.rs` line 693) and `assert_sign_inputs` (`src/frost/mod.rs` line 145), both of which use `threshold_value > participants.len()` / `threshold.value() > participants.len()`.

---

### Impact Explanation

When `threshold == N` is accepted:

- All N participants must cooperate to reconstruct the secret and produce a signature.
- A single participant who drops out, crashes, or acts maliciously by refusing to participate permanently prevents any signing, key reshare, or refresh from completing.
- Honest parties who completed DKG successfully are left with shares that can never be used.

This maps to **High: Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

Any unprivileged library caller can pass `threshold == N` directly to the public `keygen`, `reshare`, or `refresh` API. The library silently accepts it. A malicious coordinator who sets `threshold == N` during DKG can later refuse to participate in signing, permanently locking out all honest parties. An honest caller who passes `threshold == N` expecting the library to enforce the spec's constraint will receive no error and will unknowingly create an unusable key.

---

### Recommendation

Change the upper-bound check in `assert_key_invariants` from strict greater-than to greater-than-or-equal, to match the spec's strict `threshold < N`:

```rust
// Before (allows threshold == N):
if threshold > participants.len() {

// After (correctly rejects threshold == N per spec):
if threshold >= participants.len() {
```

Apply the same fix to `validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` and `assert_sign_inputs` in `src/frost/mod.rs`.
