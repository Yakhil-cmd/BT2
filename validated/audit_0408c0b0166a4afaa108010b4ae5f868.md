### Title
Invoke Transaction with Invalid Signature Admitted to Mempool via `skip_stateful_validations` UX Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway stateful path unconditionally skips the `__validate__` entry point for any Invoke transaction with `nonce == 1` when the account has **any** transaction present in the mempool pool or committed state. An unprivileged attacker can exploit this by first submitting a valid `deploy_account` transaction (which seeds the mempool with the account address), then submitting an Invoke transaction with an **invalid signature** at `nonce == 1`. The gateway admits the second transaction to the mempool without ever calling the account's `__validate__` entry point, violating the invariant that every admitted transaction must pass account-level signature validation.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

Condition 4 is satisfied by **any** transaction for that address in the pool or committed state — not specifically a `deploy_account`: [2](#0-1) [3](#0-2) 

**Effect on the validate entry point:**

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, the Invoke branch returns early without calling `__validate__`: [5](#0-4) 

`perform_pre_validation_stage` (which is still called) only checks nonce, fee bounds, balance, and proof facts — it never inspects the transaction signature: [6](#0-5) 

**Attack path:**

1. Attacker derives a fresh account address `A` (deterministic from `class_hash`, `salt`, `constructor_calldata`).
2. Attacker submits a valid `deploy_account` for `A`. This passes all gateway validations and is inserted into the mempool pool, making `tx_pool.contains_account(A) == true`.
3. Attacker submits an Invoke for `A` with `nonce = 1` and a **completely invalid signature** (e.g., all-zero felts).
4. Gateway stateful path:
   - `get_nonce_from_state` → `0` (account not yet on-chain).
   - `validate_nonce`: `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap` — passes when `max_allowed_nonce_gap ≥ 1` (required for the UX feature to function at all).
   - `validate_by_mempool` → passes (only checks for duplicate hash and `NonceTooOld`).
   - `skip_stateful_validations` → `true` (deploy_account is in pool).
   - `run_validate_entry_point` → `execution_flags.validate = false` → returns `Ok(())` without calling `__validate__`.
5. The Invoke with the invalid signature is forwarded to the mempool and admitted.

The `validate_by_mempool` call does not close this gap — it only checks nonce ordering: [7](#0-6) 

---

### Impact Explanation

**Broken invariant:** Every transaction admitted to the mempool must have passed account-level signature validation (`__validate__`). The skip path violates this for Invoke transactions at `nonce == 1` from undeployed accounts.

**Concrete wrong admission decision:** An Invoke transaction carrying an arbitrary, attacker-controlled, cryptographically invalid signature is accepted by the gateway and inserted into the mempool. This matches the **High** impact tier: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

The invalid Invoke will fail at execution time (the account's `__validate__` will reject it when the block is built), but the sequencer has already consumed gateway validation resources, mempool slots, and batcher bandwidth for a transaction that was never legitimate.

---

### Likelihood Explanation

- Requires no privileged access; any external user can submit RPC transactions.
- The only prerequisite is submitting one valid `deploy_account` (which itself requires a valid signature for the deployer key, but the deployer key is under the attacker's control for a fresh address).
- `max_allowed_nonce_gap ≥ 1` is a necessary precondition, but it is also a necessary precondition for the UX feature itself to function, so it must be enabled in any deployment that supports the deploy+invoke UX flow.
- The attack is repeatable: the attacker can generate unlimited fresh addresses and flood the mempool with signature-less Invokes, each backed by one valid `deploy_account`.

---

### Recommendation

1. **Narrow the skip condition**: instead of checking `account_tx_in_pool_or_recent_block` (which matches any transaction type), verify that the mempool specifically contains a `deploy_account` transaction for the sender address. This requires the mempool to expose a `has_deploy_account_in_pool(address)` query.
2. **Alternatively**, perform a lightweight off-chain signature check (e.g., ECDSA/Stark curve verification against the public key embedded in the constructor calldata) before granting the skip, so that at least the cryptographic binding is verified even when the account contract does not yet exist on-chain.
3. **At minimum**, document the invariant break explicitly and gate the skip on a stricter condition so that the UX improvement does not become a free admission bypass.

---

### Proof of Concept

```
# Step 1 – derive a fresh account address A from known class_hash + salt
A = compute_contract_address(class_hash, salt, constructor_calldata, deployer=0)

# Step 2 – submit a valid deploy_account for A (attacker controls the key)
POST /gateway/add_transaction
  { type: DEPLOY_ACCOUNT, class_hash, salt, constructor_calldata,
    nonce: 0, signature: <valid_sig>, resource_bounds: ... }
# → accepted; A now appears in tx_pool

# Step 3 – submit an Invoke for A with nonce=1 and a garbage signature
POST /gateway/add_transaction
  { type: INVOKE, sender_address: A,
    nonce: 1, signature: [0x0, 0x0],   # ← invalid
    calldata: [...], resource_bounds: ... }
# → gateway: account_nonce=0, tx_nonce=1, skip_stateful_validations=true
#            __validate__ is NOT called
# → accepted into mempool without signature check  ← broken invariant
``` [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
