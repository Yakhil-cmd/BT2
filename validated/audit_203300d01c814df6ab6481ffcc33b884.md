### Title
Unauthorized Invoke Transaction Admitted Without Signature Verification via `skip_stateful_validations` Overly-Broad Mempool Presence Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (account signature verification) for any invoke transaction with `nonce == 1` when `account_tx_in_pool_or_recent_block` returns `true`. The check is too broad: it returns `true` for **any** transaction in the pool for that address, not only a deploy-account transaction submitted by the legitimate owner. An unprivileged attacker who observes a pending `DeployAccount` for address X can submit an invoke with `nonce = 1` for X carrying an invalid or absent signature, and the gateway will admit it to the mempool without ever calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the nonce and resource-bounds checks pass: [1](#0-0) 

The function body: [2](#0-1) 

The condition that triggers the skip is:

```
tx.nonce() == Nonce(Felt::ONE)  &&  account_nonce == Nonce(Felt::ZERO)
    &&  mempool_client.account_tx_in_pool_or_recent_block(sender) == true
```

`account_tx_in_pool_or_recent_block` is implemented as: [3](#0-2) [4](#0-3) 

`tx_pool.contains_account(address)` returns `true` whenever **any** transaction for that address is in the pool — including a `DeployAccount` submitted by the legitimate owner. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but it does not distinguish *who* submitted those transactions.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`: [5](#0-4) 

The blockifier's `perform_validations` then short-circuits before calling `__validate__`: [6](#0-5) 

**Attack path:**

1. Legitimate user submits a valid `DeployAccount` for address X (nonce=0, valid `__validate_deploy__`). It is admitted to the mempool.
2. Attacker observes the pending `DeployAccount` for X in the mempool (public information).
3. Attacker constructs an `InvokeV3` transaction for X with `nonce = 1`, arbitrary `calldata`, and a **fabricated or empty signature**.
4. Gateway stateless validation passes (resource bounds, calldata size, DA modes, contract address — all fine).
5. `validate_state_preconditions` passes: `validate_resource_bounds` passes (gas price above threshold), `validate_nonce` passes (`0 ≤ 1 ≤ 200`). [7](#0-6) [8](#0-7) 

6. `validate_by_mempool` passes (no duplicate hash, no duplicate nonce).
7. `skip_stateful_validations` returns `true` — X is in the pool via the legitimate `DeployAccount`.
8. `run_validate_entry_point` is called with `validate: false`; `__validate__` is **never invoked**.
9. The attacker's invoke is forwarded to the mempool and admitted. [9](#0-8) 

---

### Impact Explanation

The gateway/mempool admission layer accepts an invoke transaction that carries no valid account signature. This directly satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The admitted transaction will be returned by `get_txs` and handed to the batcher. During block execution the batcher will call `__validate__` (with `validate: true`), causing the transaction to revert. However:

- The invalid transaction occupies a mempool slot, consuming capacity and potentially displacing legitimate transactions.
- If the account's `__execute__` function does not independently re-verify the signature (e.g., a multisig that relies solely on `__validate__`), the execution phase could succeed, constituting an unauthorized state change — escalating to **Critical**.
- An attacker can repeat this for every account that has a pending `DeployAccount`, creating a sustained admission-layer DOS.

---

### Likelihood Explanation

`DeployAccount` transactions are publicly visible in the mempool. Any observer can immediately submit a crafted invoke with `nonce = 1` for the target address. No privileged access, no private key material, and no special tooling are required. The only prerequisite is a pending `DeployAccount` for the target, which is a normal, common operation.

---

### Recommendation

Replace the overly-broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`DeployAccount` transaction** for the sender address is present in the pool. The mempool should expose a dedicated `deploy_account_in_pool(address)` query, or the pool contents should be filtered by transaction type before the skip decision is made.

Additionally, the `max_nonce_for_validation_skip` field present in `StatefulTransactionValidatorConfig` is never consulted by `skip_stateful_validations` in the Rust gateway (it is only used in `PyValidator`): [10](#0-9) 

This field should be wired into `skip_stateful_validations` to bound the nonce range for which the skip is permitted.

---

### Proof of Concept

```
# Prerequisites
# - Sequencer running with default config (validate_resource_bounds = true)
# - Account X does not exist on-chain (nonce = 0)

# Step 1: Legitimate owner submits DeployAccount for X
POST /gateway/add_transaction
{
  "type": "DEPLOY_ACCOUNT",
  "sender_address": "<X>",
  "nonce": "0x0",
  "signature": ["<valid_sig_r>", "<valid_sig_s>"],
  ...
}
# → Admitted to mempool. tx_pool.contains_account(X) == true.

# Step 2: Attacker submits Invoke for X with nonce=1, INVALID signature
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "sender_address": "<X>",
  "nonce": "0x1",
  "signature": ["0xdeadbeef", "0xdeadbeef"],   # fabricated
  "calldata": ["<arbitrary_calldata>"],
  ...
}

# Gateway flow:
# validate_contract_address(X)          → Ok (X is non-zero)
# validate_resource_bounds(tx)          → Ok (gas price above threshold)
# validate_nonce(tx, account_nonce=0)   → Ok (0 ≤ 1 ≤ 200)
# validate_by_mempool(tx)               → Ok (no duplicate)
# skip_stateful_validations(tx, 0)      → true  ← X is in pool via DeployAccount
# run_validate_entry_point(skip=true)   → __validate__ NOT called
# → Transaction admitted to mempool with fabricated signature
```

The attacker's invoke is now in the mempool. It will be returned by `get_txs` and executed by the batcher. If the account's `__execute__` does not re-verify the signature, the execution succeeds without authorization.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

pub struct StatefulTransactionValidatorConfig {
    // If true, ensures the max L2 gas price exceeds (a configurable percentage of) the base gas

    // Minimum gas price as percentage of threshold to accept transactions.
    pub min_gas_price_percentage: u8, // E.g., 80 to require 80% of threshold.
