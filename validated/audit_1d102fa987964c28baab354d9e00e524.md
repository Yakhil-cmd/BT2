### Title
Users Cannot Withdraw Funds from Aurora When Contract Is Paused - (`engine/src/contract_methods/connector.rs`)

### Summary
When the Aurora Engine contract is paused via `pause_contract`, the `withdraw` function enforces a `require_running` guard that reverts all user withdrawal attempts. Simultaneously, `submit` and `submit_with_args` are also blocked, cutting off the EVM-level exit paths (`ExitToNear`, `ExitToEthereum`). A separate `pause_precompiles` mechanism can independently block only the exit precompiles while the rest of the contract remains live. In both cases, users holding ETH or ERC-20 tokens on Aurora are unable to exit their funds for the duration of the pause.

### Finding Description

**Root cause 1 — `pause_contract` blocks the NEAR-level `withdraw` path**

`pause_contract` sets `state.is_paused = true`: [1](#0-0) 

`require_running` returns `Err(ERR_PAUSED)` whenever that flag is set: [2](#0-1) 

The public `withdraw` function (NEAR-level ETH → Ethereum bridge exit) calls `require_running` as its very first guard, before any user-controlled logic: [3](#0-2) 

`submit` and `submit_with_args` carry the same guard: [4](#0-3) [5](#0-4) 

Because `submit` is blocked, any EVM transaction that would call the `ExitToNear` or `ExitToEthereum` precompiles is also blocked. There is no alternative exit path available to an unprivileged user while `is_paused == true`.

**Root cause 2 — `pause_precompiles` independently blocks EVM-level exits**

Even when the contract is *running*, an authorized account can call `pause_precompiles` to set `EXIT_TO_NEAR` and/or `EXIT_TO_ETHEREUM` flags: [6](#0-5) 

The precompile dispatcher checks this flag before executing any precompile and returns a fatal `ERR_PAUSED` error: [7](#0-6) 

This means `submit` succeeds at the NEAR level but every EVM transaction attempting to call `ExitToNear` (`0xe9217bc7…`) or `ExitToEthereum` fails with `ExitFatal::Other("ERR_PAUSED")`, confirmed by the existing test: [8](#0-7) 

### Impact Explanation

**High — Temporary freezing of funds.**

Users who have deposited ETH or ERC-20 tokens into Aurora cannot exit their assets during any pause period. This mirrors the M-05 scenario exactly: a user may need to withdraw to repay a borrow on another protocol or meet another financial commitment, but every exit path is gated behind the pause flag. The NEAR-level `withdraw` is blocked by `require_running`; the EVM-level exits (`ExitToNear`, `ExitToEthereum`) are blocked either by the same `require_running` on `submit`, or independently by `pause_precompiles`. No unprivileged bypass exists.

### Likelihood Explanation

**Medium.**

`pause_contract` is owner-only and is expected to be used transiently (e.g., during an upgrade or incident response). `pause_precompiles` is callable by any account in the authorized ACL, which is a slightly wider set. Neither scenario requires a malicious actor — a legitimate, well-intentioned pause for security reasons is sufficient to trigger the fund freeze. The pause duration is unbounded in the protocol.

### Recommendation

1. **For `pause_contract`:** Remove the `require_running` guard from `withdraw` (and consider `ft_on_transfer` for the refund path) so that users can always exit their funds even when the contract is paused for other operations.
2. **For `pause_precompiles`:** Consider whether `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` should ever be pausable, or provide an alternative NEAR-level withdrawal path that remains available regardless of precompile pause state.

### Proof of Concept

**Scenario A — `pause_contract` blocks all exits:**

1. Owner calls `pause_contract` → `state.is_paused = true`.
2. User calls `withdraw` (NEAR-level) → `require_running` returns `Err(ERR_PAUSED)` → reverts.
3. User calls `submit` with an EVM tx calling `ExitToNear` → `require_running` returns `Err(ERR_PAUSED)` → reverts.
4. User has no remaining exit path.

**Scenario B — `pause_precompiles` blocks EVM exits while contract is live:**

1. Authorized account calls `pause_precompiles` with mask `0b11` (both `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM`).
2. User calls `submit` with an EVM tx calling `ExitToNear` at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` → precompile dispatcher returns `PrecompileFailure::Fatal { exit_status: ExitFatal::Other("ERR_PAUSED") }` → EVM transaction fails.
3. User calls `submit` with an EVM tx calling `ExitToEthereum` → same fatal failure.
4. NEAR-level `withdraw` still works in this scenario, but EVM-held ERC-20 balances have no exit path until precompiles are resumed.

pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
