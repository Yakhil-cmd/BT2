### Title
Live Token Balance Read in `request_funds` Can Be Manipulated to Suppress L1 Mint Requests, Causing Temporary Reward Distribution Failure - (File: src/reward_supplier/reward_supplier.cairo)

---

### Summary

The `request_funds` internal function in `RewardSupplier` reads the contract's live ERC20 token balance to decide whether to send L1 mint requests. Because any unprivileged actor can inflate this balance via a direct ERC20 `transfer` to the reward supplier address, the L1 mint request logic can be suppressed. When the donated tokens are subsequently consumed by reward payouts, the reward supplier's actual balance may fall below what is needed to fulfill `claim_rewards`, causing `update_rewards` to revert and temporarily halting reward distribution for stakers.

---

### Finding Description

In `src/reward_supplier/reward_supplier.cairo`, the internal `request_funds` function (lines 301–331) reads the live ERC20 balance of the reward supplier contract itself:

```cairo
let balance: Amount = token_dispatcher
    .balance_of(account: get_contract_address())
    .try_into()
    .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

let credit = balance + l1_pending_requested_amount;
let debit = unclaimed_rewards;

if credit < debit + threshold {
    // send L1 mint request
}
``` [1](#0-0) 

The `balance` value is a live read from the STRK ERC20 contract. ERC20 `transfer` is permissionless — any address can call `IERC20(STRK_TOKEN_ADDRESS).transfer(reward_supplier_address, X)` to inflate `balance` without going through `on_receive` (which is the only controlled entry point that updates `l1_pending_requested_amount`). [2](#0-1) 

The donated tokens are **not** tracked in `unclaimed_rewards` (which only grows via `update_unclaimed_rewards_from_staking_contract`). As rewards are distributed each block, `claim_rewards` consumes the reward supplier's token balance:

```cairo
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [3](#0-2) 

If the donated tokens are consumed by these payouts before the next L1 request is fulfilled (L1→L2 message delay), the reward supplier's actual token balance can fall below the amount needed for `claim_rewards`, causing the `checked_transfer` to revert.

`claim_rewards` is called from `_update_rewards` in the staking contract, which is invoked by `update_rewards` (the per-block consensus reward function): [4](#0-3) 

If `claim_rewards` reverts, the entire `update_rewards` transaction reverts, meaning `last_reward_block` is not updated and stakers receive no rewards for that block. [5](#0-4) 

---

### Impact Explanation

If the reward supplier's balance is drained below the per-block pool reward amount before L1 replenishment arrives, `update_rewards` reverts every block until the L1 message is processed. During this window:

- Stakers receive no new rewards for affected blocks (unclaimed yield is not accrued).
- Pool rewards are not transferred to delegation pool contracts.
- The sequencer must retry `update_rewards` each block, causing operational disruption.

This matches the **Medium** allowed impact: *Griefing with no profit motive but damage to users or protocol*.

---

### Likelihood Explanation

**Low–Medium.** The attacker must donate STRK tokens proportional to `pool_rewards_per_block × L1_message_delay_in_blocks` to bridge the gap between suppressed L1 requests and actual balance depletion. The attacker permanently loses the donated tokens. The attack is permissionless (no privileged role required) and requires only a direct ERC20 transfer to the reward supplier address. The effect is temporary — once the donated tokens are consumed, `request_funds` will trigger a new L1 request on the next block.

---

### Recommendation

Replace the live `balance_of` read in `request_funds` with an internal accounting variable that is updated only through controlled entry points (`on_receive` and `claim_rewards`). For example, maintain a `tracked_balance` storage variable that is incremented in `on_receive` and decremented in `claim_rewards`. Use `tracked_balance` instead of `balance_of(get_contract_address())` in the credit calculation. This eliminates the dependency on externally manipulable state.

---

### Proof of Concept

1. Attacker calls `IERC20(STRK_TOKEN_ADDRESS).transfer(reward_supplier_address, D)` where `D` is large enough to satisfy `balance + D + l1_pending_requested_amount >= unclaimed_rewards + threshold`.
2. On the next block, `update_rewards` → `update_unclaimed_rewards_from_staking_contract` → `request_funds` is called.
3. `request_funds` reads `balance = original_balance + D`, computes `credit >= debit + threshold`, and **does not** send an L1 mint request.
4. `claim_rewards(pool_rewards)` is called in the same transaction, transferring pool rewards to the staking contract. `balance` decreases by `pool_rewards`.
5. This repeats each block. After approximately `D / pool_rewards_per_block` blocks, the donated tokens are consumed and `balance` returns to `original_balance`.
6. Meanwhile, `unclaimed_rewards` has grown by `staker_rewards_per_block × N` blocks (staker rewards are not immediately claimed). If `original_balance < unclaimed_rewards` at this point and no L1 tokens have arrived yet (L1 message delay), `claim_rewards` attempts `checked_transfer(amount)` where `balance < amount`.
7. The ERC20 transfer reverts → `_update_rewards` reverts → `update_rewards` reverts → stakers receive no rewards for that block.
8. The disruption persists until the L1 mint message is processed and tokens arrive via `on_receive`.

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
