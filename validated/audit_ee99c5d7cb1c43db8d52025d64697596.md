### Title
`skip_stateful_validations` bypasses `__validate__` for invoke nonce=1 when any pooled tx exists for the account, not only a pending deploy_account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip gateway `__validate__` only when a `deploy_account` is pending in the mempool (UX improvement for simultaneous deploy+invoke). However, `account_tx_in_pool_or_recent_block` returns `true` for **any** pooled transaction type. An attacker can exploit this to admit an invoke with an invalid signature into the mempool by ensuring any tx from the target account is already pooled.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` checks three conditions before skipping `__validate__`: [1](#0-0) 

The third condition delegates to: [2](#0-1) 

This returns `true` for **any** transaction type in the pool — invoke, declare, or deploy_account. The comment at line 441 claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is wrong: an invoke nonce=0 for a deployed account with state nonce=0 passes `__validate__` and is admitted to the mempool, yet it is not a deploy_account.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

And `StatefulValidator::perform_validations` returns early before calling `__validate__`: [4](#0-3) 

### Impact Explanation

An invalid invoke transaction (with an arbitrary/wrong signature) for nonce=1 targeting an account with state nonce=0 is admitted to the mempool without signature verification. This matches the **High** impact scope: mempool/gateway admission accepts invalid transactions before sequencing.

The invalid transaction will revert during execution (the batcher sets `validate=true` independently), so on-chain state is not corrupted. However:
- The mempool slot for nonce=1 of the victim account is occupied by an invalid transaction.
- The legitimate user's nonce=1 transaction may be rejected as a duplicate nonce or require fee escalation to replace it.
- The attacker can repeat this for any account that has nonce=0 in state and any pooled tx.

### Likelihood Explanation

The preconditions are realistic and common:
1. Account is deployed with state nonce=0 (any freshly deployed account).
2. The legitimate user submits invoke nonce=0 (normal first use of the account).
3. Attacker observes the mempool and submits invoke nonce=1 with an invalid signature.

No privileged access is required. The attacker only needs to know the target account address and submit a well-formed invoke with nonce=1.

### Recommendation

In `skip_stateful_validations`, replace the `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a **deploy_account** transaction is pending for the sender address. Alternatively, add a dedicated mempool API `deploy_account_in_pool(address) -> bool` that only returns `true` when a deploy_account tx is pooled for that address.

### Proof of Concept

```
1. Deploy account A (deploy_account committed, state nonce = 0, contract code exists).
2. User submits invoke(sender=A, nonce=0, valid_signature) → admitted to mempool.
   - tx_pool.contains_account(A) = true
3. Attacker submits invoke(sender=A, nonce=1, signature=0xdeadbeef):
   - validate_nonce: 0 ≤ 1 ≤ max_gap → OK
   - validate_by_mempool: no sig check → OK
   - skip_stateful_validations:
       invoke ✓, nonce==1 ✓, account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(A) = true (user's nonce=0 tx)
       → returns true
   - run_validate_entry_point: execution_flags.validate = false
   - StatefulValidator::perform_validations: returns Ok(()) before __validate__
   → Invalid invoke nonce=1 admitted to mempool without signature check.
```
