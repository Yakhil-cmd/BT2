Audit Report

## Title
CMC Minting Rate Limiter Bypass via Inflated Canister Balance in `ensure_balance` - (File: rs/nns/cmc/src/main.rs)

## Summary
The `ensure_balance` function computes `cycles_to_mint` as the saturating difference between the requested cycles and the CMC's live balance, then checks the rate limiter against that delta rather than the total requested amount. Because any canister can deposit cycles into the CMC via the management canister's `deposit_cycles` call (no consent required from the CMC), an attacker can inflate the CMC's balance before a conversion, causing `cycles_to_mint` to saturate to zero and the limiter to record nothing, while the full conversion still completes. This bypasses the 150 × 10¹⁵ cycles/hour cap and permanently undercounts `total_cycles_minted`.

## Finding Description

**Root cause — saturating subtraction used as the limiter input:**

`ensure_balance` at line 2313 computes:
```rust
let cycles_to_mint = cycles - current_balance;  // Cycles::sub is saturating_sub
``` [1](#0-0) 

The `Cycles` subtraction operator is confirmed to be `saturating_sub`: [2](#0-1) 

The limiter then receives `cycles_to_mint` (the delta), not `cycles` (the total): [3](#0-2) 

**Attack vector — inflating the CMC's balance:**

The IC management canister's `deposit_cycles` method allows any canister to deposit cycles into any target canister without the target's consent. An attacker calls `deposit_cycles` targeting the CMC's principal with amount X. The CMC's balance rises by X. On the next `ensure_balance(Y, ...)` call, if `X + B >= Y`, then `cycles_to_mint = 0`, the limiter records 0, and the full Y cycles are delivered from the pre-loaded balance. The ICP is still burned; the limiter is untouched.

**All three public conversion paths funnel through `ensure_balance`:**
- `notify_top_up` → `process_top_up` → `deposit_cycles` → `ensure_balance`
- `notify_create_canister` → `do_create_canister` → `ensure_balance`
- `notify_mint_cycles` → `process_mint_cycles` → `do_mint_cycles` → `ensure_balance` [4](#0-3) [5](#0-4) 

**`total_cycles_minted` is also undercounted** by the same delta, corrupting the on-chain metric: [6](#0-5) 

## Impact Explanation

An unprivileged attacker can repeatedly bypass the 150 × 10¹⁵ cycles/hour NNS minting rate cap, allowing arbitrarily large ICP burns per hour. The rate cap exists to bound the rate at which ICP supply is reduced and new cycles enter the economy; bypassing it enables economic manipulation of ICP supply at scale. The `total_cycles_minted` counter is permanently undercounted, breaking protocol-level auditability. This constitutes a significant NNS security impact with concrete protocol harm, fitting the **High** ($2,000–$10,000) bounty tier.

## Likelihood Explanation

No special privileges are required. Any canister on any subnet can call the management canister's `deposit_cycles` targeting the CMC. The attacker's deposited cycles are not destroyed — they are consumed by the CMC to fulfill the conversion and effectively returned to whoever receives the converted cycles. If the attacker is also the conversion recipient, the net cost is near zero (cycles recycled, only ICP burned at the normal rate). The attack is repeatable within a single hour window and requires no governance majority, subnet compromise, or leaked keys.

## Recommendation

Check the limiter against `cycles` (the total amount to deliver), not `cycles_to_mint` (the delta to newly mint). The CMC's pre-existing balance — regardless of how it was funded — is irrelevant to the rate-limiting policy:

```rust
fn ensure_balance(cycles: Cycles, limiter_to_use: CyclesMintingLimiterSelector) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance;

    with_state_mut(|state| {
        // Rate-limit against `cycles` (total delivered), not `cycles_to_mint` (delta minted)
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

As a defense-in-depth measure, the CMC should also reject or burn any cycles received outside its own minting flow (e.g., via a `canister_inspect_message` or by not accepting cycles in non-minting handlers).

## Proof of Concept

**Minimal integration test plan (PocketIC or local replica):**

1. Deploy a test canister with a large cycle balance on a local replica.
2. Record the CMC's `base_limiter` count (should be near the 150e15 cap after priming).
3. From the test canister, call the management canister's `deposit_cycles` with the CMC as target and amount = `cap - current_limiter_count + 1` (enough to cover the next conversion).
4. Trigger `notify_mint_cycles` or `notify_top_up` for a conversion that would normally exceed the remaining cap.
5. Observe: the call succeeds (no limiter rejection), `total_cycles_minted` is not incremented by the full conversion amount, and the limiter's internal count remains below the cap.
6. Repeat step 3–5 to demonstrate the cap can be exceeded arbitrarily within a single hour window.

    pub fn check_and_add_cycles(
        &mut self,
        now: SystemTime,
        cycles_to_mint: Cycles,
        limit: Cycles,
    ) -> Result<(), String> {
        self.purge_old(now);
        let count = self.get_count();

        if count + cycles_to_mint > limit {
            LIMITER_REJECT_COUNT.with(|count| {
                count.set(count.get().saturating_add(1));
            });

            return Err(format!(
                "More than {} cycles have been minted in the last {} seconds, please try again later.",
                limit,
                self.get_max_age().as_secs(),
            ));
        }
        self.add(now, cycles_to_mint);
        Ok(())
