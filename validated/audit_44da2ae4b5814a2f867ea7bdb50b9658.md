### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required or When Excess Is Sent — (`File: L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
`RewardSupplier.tick()` is `payable` and accepts ETH to cover L1→L2 bridge and messaging fees. However, when `amountToMint == 0` (no pending mint requests), the entire `msg.value` is silently accepted and permanently locked in the contract. Even when minting does occur, any ETH beyond what the bridge and messaging contracts actually consume is permanently locked. Unlike the analogous Uniswap/Numoen pattern where a `refundETH()` escape hatch exists, `RewardSupplier` has no ETH recovery mechanism whatsoever.

### Finding Description
`tick()` is declared `external payable` and splits `msg.value` between two downstream calls:

```solidity
uint256 msgFee = msg.value / 2;
bridge().depositWithMessage{value: msgFee}(...);   // line 126

msgFee = msg.value - msgFee;
messagingContract().sendMessageToL2{value: msgFee}(...);  // line 137
```

The entire body is guarded by `if (amountToMint > 0)`. When `requiredMinting()` returns zero (no L2 mint-request messages are pending), the function exits immediately after line 109 without touching `msg.value`. Any ETH sent in that call is permanently trapped.

Even when minting does occur, the caller must guess the exact fees required by `depositWithMessage` and `sendMessageToL2`. These fees are set by the StarkGate bridge and the Starknet messaging contract respectively and are not knowable on-chain before the call. Callers routinely over-provision ETH to avoid reverts. The surplus is never returned.

`RewardSupplier` inherits from `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl`. None of these parent contracts define a `receive()`, `fallback()`, or any ETH-withdrawal function. There is no `refundETH()` analog. The contract has no mechanism to recover trapped ETH.

### Impact Explanation
Any ETH sent to `tick()` that is not consumed by the bridge or messaging contract is permanently frozen in `RewardSupplier`. Because `tick()` is callable by any account (the spec diagram labels the caller `anyAccount`), this affects every operator or keeper who calls `tick()`. The loss is permanent and irrecoverable — matching the **permanent freezing of funds** impact category.

### Likelihood Explanation
`tick()` is a public keeper function intended to be called regularly by external parties. Callers must supply ETH to cover two separate L1→L2 fees whose exact values are not statically known. Over-provisioning is the standard safe practice. Additionally, any call made when the L2 mint-request queue is empty (a normal steady-state condition between reward epochs) locks 100% of the supplied ETH. Both scenarios are routine operational conditions, making this a high-frequency loss path.

### Recommendation
1. Refund any unused ETH to `msg.sender` at the end of `tick()`:
   ```solidity
   uint256 remaining = address(this).balance;
   if (remaining > 0) {
       (bool ok, ) = msg.sender.call{value: remaining}("");
       require(ok, "ETH_REFUND_FAILED");
   }
   ```
2. Alternatively, revert when `amountToMint == 0` and `msg.value > 0`, making the no-op case explicit.
3. Add an admin ETH-rescue function as a backstop for any residual balance.

### Proof of Concept

**Scenario A — No pending mint requests:**
1. The L2 reward supplier has not yet emitted any `MintRequest` messages (e.g., between epochs).
2. A keeper calls `tick{value: 0.1 ether}()`.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` branch is skipped entirely.
5. `0.1 ether` is now held by `RewardSupplier` with no way to recover it.

**Scenario B — Minting occurs but caller over-provisions:**
1. One mint request is pending; keeper calls `tick{value: 0.1 ether}()`.
2. `depositWithMessage` consumes `0.03 ether`; `sendMessageToL2` consumes `0.02 ether`.
3. The remaining `0.05 ether` stays in the contract permanently.

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
