### Title
Gateway Signature Validation Bypass via `skip_stateful_validations` Allows Invalid Invoke Transactions into Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway's stateful transaction validator unconditionally skips the `__validate__` entry point (signature verification) for any invoke transaction with `nonce == 1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because that check returns `true` for **any** transaction in the mempool for the address — not specifically a `deploy_account` — an attacker can submit an invoke with an invalid signature, bypass gateway-level signature verification, and have the transaction admitted to the mempool. Via fee escalation, the attacker can then displace the legitimate user's invoke, causing it to be lost while the attacker's invalid transaction consumes the nonce during execution.

### Finding Description

`skip_stateful_validations` is an intentional UX feature: when a user sends `deploy_account` (nonce 0) + `invoke` (nonce 1) simultaneously, the invoke should not fail at the gateway solely because the account does not yet exist on-chain. The function checks three conditions before returning `true` (skip `__validate__`):

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is then skipped entirely: [3](#0-2) 

The flaw is in condition 4. `account_tx_in_pool_or_recent_block` returns `true` if **any** transaction for the address exists in the pool or was committed in a recent block — it does not verify that the pooled transaction is specifically a `deploy_account`: [4](#0-3) 

The code comment acknowledges the reasoning: "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is incorrect: an attacker can make `account_tx_in_pool_or_recent_block` return `true` for an address they do not control simply by observing that the legitimate owner's `deploy_account` is in the mempool.

**Attack path:**

1. Legitimate user submits `deploy_account` (nonce 0) + `invoke` (nonce 1, valid signature) simultaneously. Both are admitted to the mempool.
2. Attacker observes the `deploy_account` for address `A` in the mempool (publicly visible).
3. Attacker submits a fake `invoke` for `A` with `nonce = 1`, **invalid signature**, and a **higher fee** than the legitimate invoke.
4. Gateway stateful validation:
   - `validate_state_preconditions`: nonce check passes (`0 ≤ 1 ≤ 0 + max_gap`). [5](#0-4) 
   - `validate_by_mempool`: mempool's `validate_tx` passes (no duplicate hash; fee escalation check passes because attacker's fee is higher). [6](#0-5) 
   - `skip_stateful_validations`: returns `true` (nonce=1, account_nonce=0, deploy_account is in pool → `account_tx_in_pool_or_recent_block` = `true`).
   - `run_validate_entry_point(skip_validate=true)`: `__validate__` is **not called**. Transaction passes gateway validation.
5. Transaction is added to the mempool via `add_tx`. Fee escalation removes the legitimate invoke and inserts the attacker's fake invoke.
6. Block execution: `deploy_account` runs (account deployed, nonce → 1). Attacker's fake invoke runs: `__validate__` is called, fails (invalid signature), transaction reverts, nonce → 2.
7. The legitimate invoke (nonce 1) is gone from the mempool. The user must resubmit with nonce 2.

### Impact Explanation

An invalid transaction (carrying a wrong/arbitrary signature) bypasses gateway-level signature verification and is admitted to the mempool. Via fee escalation, it can displace the legitimate user's invoke. During execution the fake invoke fails `__validate__`, consumes the nonce, and may charge a fee if the account has balance. The user's `deploy_account + invoke` UX flow is broken; the user must detect the failure and resubmit with the incremented nonce. This matches the scope criterion: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The attack is straightforward to execute: the mempool is publicly observable, the attacker only needs to monitor for `deploy_account` transactions, and the only cost is paying a marginally higher fee than the victim. The attack window is the time between the `deploy_account` entering the mempool and the block being committed — typically several seconds to minutes.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the address in the mempool. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that only returns `true` when the pooled transaction is of type `DeployAccount`. This mirrors the ENS fix: enforce that the entity being relied upon (the deploy_account) is actually present, not just that *some* transaction for the address exists.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
