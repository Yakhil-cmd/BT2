### Title
Malicious Participant Can Corrupt CKD Output via Unvalidated Contributions — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary
The CKD coordinator blindly aggregates participant-supplied `(norm_big_y, norm_big_c)` pairs without any cryptographic validation. A single malicious participant can send arbitrary values, causing the coordinator to produce a corrupted `CKDOutput` that decrypts to the wrong confidential key, permanently denying honest parties a valid CKD result.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's contribution and sums them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

There is no ZK proof, commitment binding, or any other mechanism to verify that a participant's `(norm_big_y, norm_big_c)` was honestly computed from their actual key share and the agreed `app_id`. The correct values are:

- `norm_big_y = lambda_i * y_i * G`
- `norm_big_c = lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)`

computed inside `compute_signature_share`: [2](#0-1) 

A malicious participant can substitute any group elements for these values. The coordinator has no way to detect the substitution.

This contrasts sharply with both ECDSA signing variants, which validate the final signature before returning it:

- OT-based: `if !sig.verify(&public_key, &msg_hash)` [3](#0-2) 
- Robust: `if !sig.verify(&public_key, &msg_hash)` [4](#0-3) 

The CKD protocol has no equivalent end-to-end check.

---

### Impact Explanation

The TEE unmasks the output as `confidential_key = C - app_sk * Y`. The correct result is `msk * H(pk, app_id)`. If a malicious participant sends `(G_arbitrary, C_arbitrary)` instead of their honest share, the aggregated `(Y_final, C_final)` will be wrong, and the TEE will derive an incorrect confidential key — one that is neither the intended derived secret nor a value the attacker controls (since `app_sk` is the TEE's secret). The CKD output is permanently corrupted for that invocation.

**Impact: High** — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs, matching the allowed impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

Any participant in the CKD protocol can execute this attack by sending `(ElementG1::identity(), ElementG1::identity())` or any other arbitrary pair. No special cryptographic knowledge is required — only the ability to participate in the protocol. In a realistic MPC deployment, a single compromised or malicious node is sufficient.

**Likelihood: Medium** — Requires one malicious participant; no external assumptions needed.

---

### Recommendation

Add cryptographic binding of each participant's contribution to their committed key share. Concretely:

1. **ZK proof of correct formation**: Each participant proves in zero-knowledge that `norm_big_c` is correctly formed relative to `norm_big_y` and their public key share commitment (e.g., a Chaum-Pedersen proof that `norm_big_c - norm_big_y * app_pk` lies on the correct subspace).
2. **Commitment-then-reveal**: Participants commit to `(norm_big_y, norm_big_c)` before revealing, and the coordinator verifies the hash matches before accepting.
3. **Final output check**: After aggregation, verify the output satisfies the expected algebraic relation using the known public key `msk_pub = G2 * msk` (analogous to how the ECDSA coordinator verifies the final signature).

---

### Proof of Concept

```
Setup: 3 participants, one malicious (participant M).

1. Honest participants P1, P2 compute correct (norm_big_y_i, norm_big_c_i).
2. Malicious participant M sends (ElementG1::identity(), ElementG1::identity()) 
   instead of their correct contribution.
3. Coordinator sums:
     Y_final = norm_big_y_P1 + norm_big_y_P2 + 0   (missing M's lambda_M * y_M * G)
     C_final = norm_big_c_P1 + norm_big_c_P2 + 0   (missing M's lambda_M * (x_M * H + y_M * app_pk))
4. TEE decrypts: C_final - app_sk * Y_final
     = (lambda_P1*x_P1 + lambda_P2*x_P2) * H(pk, app_id)   ≠   msk * H(pk, app_id)
5. TEE derives wrong confidential key; CKD is permanently corrupted for this call.
```

The attack requires zero cryptographic capability — only the ability to send a message in the protocol round handled by `recv_from_others` at the coordinator's waitpoint. [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
