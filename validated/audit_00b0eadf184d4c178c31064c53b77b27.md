### Title
Reward Address Zero Validation Missing Allows Permanent Loss of Staker/Pool Member Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `stake()` and `change_reward_address()` in the staking contract, and `enter_delegation_pool()` and `change_reward_address()` in the pool contract, accept `ContractAddress::zero()` as a valid `reward_address`. When rewards are later claimed, they are unconditionally transferred to that zero address and permanently lost.

---

### Finding Description

The staking contract's `stake()` function validates that `reward_address` is not a token contract address, but performs no check against the zero address: [1](#0-0) 

Similarly, `change_reward_address()` in the staking contract only rejects token addresses: [2](#0-1) 

The pool contract's `claim_rewards()` reads the stored `reward_address` and transfers directly to it with no zero-address guard: [3](#0-2) 

The staking contract's internal `send_rewards_to_staker()` does the same — it reads `reward_address` from storage and calls `checked_transfer` unconditionally: [4](#0-3) 

This function is invoked both from `claim_rewards()` and from `unstake_action()`, meaning the zero-address transfer can occur at any reward-claiming point in the staker lifecycle. [5](#0-4) 

---

### Impact Explanation

Any STRK rewards accumulated by a staker or pool member whose `reward_address` is zero are transferred to `ContractAddress::zero()` on the next `claim_rewards` or `unstake_action` call. Starknet's ERC20 does not revert on transfers to address 0; the tokens are sent and permanently inaccessible. This constitutes **permanent freezing of unclaimed yield**, which is in the allowed High-severity impact scope.

---

### Likelihood Explanation

A staker or pool member can set `reward_address` to zero either:
- At registration time via `stake(reward_address: 0, ...)` or `enter_delegation_pool(reward_address: 0, ...)`
- At any later time via `change_reward_address(reward_address: 0)`

No privileged role is required. The call is reachable by any unprivileged staker or delegator. Accidental misconfiguration (e.g., passing an uninitialized address) is a realistic scenario, and the protocol provides no safety net.

---

### Recommendation

Add a zero-address guard in all four entry points:

- `stake()` in `src/staking/staking.cairo`
- `change_reward_address()` in `src/staking/staking.cairo`
- `enter_delegation_pool()` in `src/pool/pool.cairo`
- `change_reward_address()` in `src/pool/pool.cairo`

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::REWARD_ADDRESS_IS_ZERO);
```

This mirrors the existing `REWARD_ADDRESS_IS_TOKEN` guard pattern already present in the codebase.

---

### Proof of Concept

**Staker path:**

1. Staker calls `staking.stake(reward_address: ContractAddress::zero(), operational_address: op, amount: min_stake)`.
   - Passes all existing checks (not a token address, not already staked, amount sufficient).
   - `reward_address = 0` is written to `staker_info`.
2. Several epochs pass; attestations are submitted; `staker_info.unclaimed_rewards_own` accumulates a non-zero value.
3. Anyone calls `staking.claim_rewards(staker_address)`.
4. `send_rewards_to_staker` executes:
   - `reward_address = staker_info.reward_address` → `0`
   - `claim_from_reward_supplier(...)` pulls the reward tokens into the staking contract.
   - `token_dispatcher.checked_transfer(recipient: 0, amount: rewards)` — succeeds, tokens sent to address 0.
5. Rewards are permanently lost. `staker_info.unclaimed_rewards_own` is zeroed out; no retry is possible.

**Pool member path (identical logic):**

1. Pool member calls `pool.enter_delegation_pool(reward_address: 0, amount: ...)`.
2. Rewards accumulate.
3. `pool.claim_rewards(pool_member)` transfers to `reward_address = 0` at line 366 of `pool.cairo`. [6](#0-5) [7](#0-6)
