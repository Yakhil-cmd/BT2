### Title
Unauthenticated Invoke Transaction Bypasses `__validate__` Signature Check for Undeployed Accounts via `skip_stateful_validations` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway's stateful validator skips the blockifier `__validate__` entry-point call for any invoke transaction with `nonce == 1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The check is supposed to confirm that the account owner submitted a `deploy_account` transaction, but it only verifies that *any* transaction for that address exists in the mempool — it does not verify that the submitter of the nonce-1 invoke is the same party who submitted the `deploy_account`. An unprivileged attacker who observes a `deploy_account` transaction in the mempool can immediately submit a nonce-1 invoke transaction for the victim's address with an arbitrary (invalid) signature and arbitrary calldata, and the gateway will accept it without running `__validate__`.

### Finding Description

The broken invariant is in `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs L429-L461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                // ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is flawed. `account_tx_in_pool_or_recent_block` returns `true` for any account that has *any* transaction in the pool, regardless of who submitted it:

```rust
// crates/apollo_mempool/src/mempool.rs L697-L700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, causing the blockifier's `StatefulValidator` to skip the `__validate__` call entirely:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs L311-L312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The `StatefulValidator::perform_validations` in the blockifier confirms: if `execution_flags.validate == false`, the `__validate__` call is skipped and the function returns `Ok(())`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs L79-L81
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

**Attack sequence:**

1. Victim submits `deploy_account` (nonce=0) for address `A` → accepted into mempool.
2. Attacker observes the mempool and submits `Invoke(sender=A, nonce=1, calldata=<arbitrary>, signature=<garbage>)`.
3. Gateway calls `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` (victim's deploy_account is there) → returns `true`.
4. `run_validate_entry_point` is called with `skip_validate=true` → `__validate__` is never called → attacker's tx is accepted into the mempool.
5. If the attacker's nonce-1 tx is admitted first, the victim's own nonce-1 invoke is subsequently rejected by the mempool as a duplicate nonce.
6. At execution time, the batcher executes the attacker's tx with `validate=true` (fresh `ExecutionFlags`); `__validate__` runs and fails → tx reverts → fees are charged to the victim's newly-deployed account.

The analog to the external bug is exact: `_verifySender` checked only `added` (a boolean that was never cleared on removal); `skip_stateful_validations` checks only `account_tx_in_pool_or_recent_block` (a boolean that does not verify the submitter identity), allowing a revoked/unauthorized party to pass the authorization gate.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An unprivileged attacker can inject a cryptographically invalid (wrong-signature) invoke transaction for any not-yet-deployed account that has a pending `deploy_account` in the mempool. Consequences:

- The attacker's invalid tx occupies the nonce-1 slot, blocking the victim's legitimate nonce-1 invoke from entering the mempool (duplicate-nonce rejection).
- At execution time the attacker's tx reverts, but fees are charged to the victim's account, draining balance.
- The victim must resubmit their nonce-1 invoke in a later block, disrupting the intended atomic deploy+invoke UX.

### Likelihood Explanation

The `deploy_account` + `invoke(nonce=1)` pattern is the documented and tested UX flow (see `create_deploy_account_tx_and_invoke_tx` in integration tests). Any attacker monitoring the public mempool can trivially trigger this for every new account deployment. No special privilege or prior relationship with the victim is required. [5](#0-4) 

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the exact sender address is pending in the mempool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query that only returns `true` when a `DeployAccount` transaction (not any transaction) is present for that address. This mirrors the fix recommended in the external report: the authorization check must verify the *complete* state (deploy_account present AND submitted by the same sender), not a stale/incomplete boolean.

### Proof of Concept

```
1. Start a local sequencer node.
2. Generate a fresh account keypair (address A, private key K).
3. Fund address A with enough STRK to cover fees.
4. Submit deploy_account(address=A, nonce=0, signature=valid(K)) → tx_hash_deploy.
5. Immediately submit invoke(sender=A, nonce=1, calldata=[0xdead], signature=[0xbad]) 
   from a DIFFERENT keypair (attacker).
6. Observe: gateway returns 200 OK and a tx_hash for the attacker's invoke.
7. Submit the victim's own invoke(sender=A, nonce=1, calldata=[0xcafe], signature=valid(K)).
8. Observe: gateway returns InvalidTransactionNonce (duplicate nonce rejected).
9. Wait for block inclusion: deploy_account succeeds; attacker's invoke reverts; 
   victim's account balance is reduced by the fee for the failed tx.
``` [6](#0-5) [7](#0-6)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

/// The first invoke_tx can be inserted to the first block right after the deploy_tx due to
