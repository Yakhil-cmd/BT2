### Title
Unbounded `stakers` Vec Growth Enables Griefing of `get_stakers()` with Unbounded Gas Consumption - (File: src/staking/staking.cairo)

---

### Summary

The `stakers` storage Vec in the Staking contract permanently accumulates every address that has ever called `stake()`, but entries are **never removed** when a staker unstakes. The `get_stakers()` function iterates over the **entire** Vec on every call. An unprivileged attacker can cheaply inflate this Vec by cycling through stake/unstake with many addresses, causing `get_stakers()` to consume unbounded gas and disrupting the consensus layer's ability to retrieve the active validator set.

---

### Finding Description

The `stakers` Vec is declared in storage with an explicit developer note acknowledging the design: [1](#0-0) 

Every call to `stake()` appends to this Vec unconditionally: [2](#0-1) 

The internal `remove_staker()` function — called during `unstake_action()` — clears `staker_info`, zeroes the own balance, and removes the operational address mapping, but **never touches the `stakers` Vec**: [3](#0-2) 

`get_stakers()`, which is part of `IStakingConsensus` and is the entry point used by the consensus layer to enumerate active validators, iterates over the **full range** of the Vec on every invocation: [4](#0-3) 

Inactive (unstaked) entries are skipped with a `continue`, but every slot still costs a storage read (`staker_address_ptr.read()`) plus the `is_staker_active` check. With N dead entries, the function performs at minimum N storage reads before reaching any live staker.

---

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption.**

`get_stakers()` is the consensus-layer query for the active validator set. If the Vec is bloated with thousands of dead entries, the function's gas cost grows linearly with the total number of historical stakers, not the number of currently active ones. At sufficient scale this causes:

- The function to exceed block gas limits when called on-chain (e.g., from the Attestation contract or any on-chain consumer).
- Off-chain callers (sequencer, consensus clients) to time out or receive stale/incomplete data, degrading protocol liveness.

---

### Likelihood Explanation

**High.** The attack requires no special privilege. Any address can call `stake()` with `min_stake` STRK, then call `unstake_intent()` and `unstake_action()` after the exit window to recover the full principal. The net cost to the attacker is only gas fees per cycle. With enough addresses and cycles the Vec grows arbitrarily large. The exit wait window (default one week) is the only rate-limiting factor, but it does not prevent the attack — it only paces it.

---

### Recommendation

1. **Remove from Vec on unstake**: Replace `Vec<ContractAddress>` with a data structure that supports O(1) removal (e.g., a swap-and-pop pattern using a `Map<u64, ContractAddress>` index alongside a length counter), and remove the entry in `remove_staker()`.
2. **Alternatively, cap iteration**: Introduce a separate active-staker count and reject `get_stakers()` calls that would iterate beyond a safe bound, or maintain a separate active-staker set that is updated on stake/unstake.
3. **Short-term mitigation**: Add a `max_stakers` cap enforced in `stake()` to bound the Vec size, analogous to `maxNumberOfKeys` in the referenced report.

---

### Proof of Concept

1. Attacker generates N addresses (e.g., N = 10,000).
2. Each address approves `min_stake` STRK to the Staking contract and calls `stake(reward_address, operational_address, min_stake)`.
   - Each call executes `self.stakers.push(staker_address)` at line 348.
3. Each address calls `unstake_intent()`, waits one week, then calls `unstake_action()` to recover STRK.
   - `remove_staker()` is called but does **not** remove the address from `stakers`.
4. `stakers.len()` is now N (all dead entries).
5. Any caller of `get_stakers(epoch_id)` now triggers a loop of N iterations, each performing at least one storage read and one `is_staker_active` check, all returning `false` and `continue`-ing.
6. At N ≈ tens of thousands, the gas cost exceeds practical on-chain limits, breaking the consensus layer's ability to enumerate validators.
