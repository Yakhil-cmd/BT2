### Title
Missing Minimum Threshold Lower-Bound Check in FROST Signing/Presigning Allows Protocol Corruption - (File: src/frost/mod.rs)

### Summary
`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate the `threshold` parameter with only an upper-bound check (`threshold > participants.len()`), but omit the lower-bound check (`threshold < 2`) that is consistently enforced in every other protocol entry point in the codebase. A caller can pass `threshold = 1` (or `threshold = 0`) to the FROST signing or presigning protocol, bypassing the minimum threshold requirement and causing the protocol to produce a cryptographically unusable signature while consuming honest participants' presignature nonces.

### Finding Description

`assert_sign_inputs` in `src/frost/mod.rs` performs the following threshold validation:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [1](#0-0) 

The lower-bound check is entirely absent. Compare this with `assert_key_invariants` in `src/dkg.rs`, which enforces **both** bounds:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [2](#0-1) 

The same dual-bound pattern is also enforced in `validate_triple_inputs` for OT-based ECDSA triple generation: [3](#0-2) 

The `presign` function in `src/frost/mod.rs` has the identical omission — it only checks the upper bound: [4](#0-3) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no minimum enforced at the type level, so any value including 0 or 1 is accepted: [5](#0-4) 

### Impact Explanation

When `threshold = 1` is passed to `assert_sign_inputs`, the FROST signing protocol proceeds with only one participant's share. Because the key was generated with `threshold ≥ 2` (enforced by `assert_key_invariants`), the Lagrange interpolation over a single share produces the wrong secret scalar. The resulting signature fails cryptographic verification, yielding an unusable output. Honest participants have irreversibly consumed their presignature nonces (FROST nonces must never be reused), permanently denying them the ability to sign with those nonces.

When `threshold = 0` is passed, the behavior is undefined — Lagrange interpolation over zero points may panic or silently produce a zero scalar, again corrupting the signing output.

**Matched impact**: High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs, and permanent denial of signing for honest parties whose nonces are consumed.

### Likelihood Explanation

The entry point is the public `assert_sign_inputs` function, callable by any library user or coordinator constructing a signing session. The `ReconstructionLowerBound` type accepts any `usize`, so passing `1` or `0` requires no special privilege. A malicious coordinator orchestrating a signing session can trivially supply a sub-threshold value, causing all participants to waste their presignature material.

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the upper-bound check in both functions, mirroring the pattern at `src/dkg.rs:580-582`.

### Proof of Concept

1. Generate a FROST key with `threshold = 2` and `n = 3` participants using `keygen` (succeeds, lower-bound enforced by `assert_key_invariants`).
2. Call `assert_sign_inputs(participants, /*threshold=*/1u8, me, coordinator)` — the call **succeeds** because only the upper-bound check fires (`1 > 3` is false) and the missing lower-bound check is never reached.
3. Proceed to the FROST signing round with `threshold = 1`. Only one participant's share enters Lagrange interpolation; the reconstructed nonce scalar is wrong.
4. The coordinator assembles the final signature; `sig.verify(public_key, msg_hash)` returns `false`.
5. All participants have consumed their `SigningNonces`, which cannot be reused per FROST security requirements — the signing session is permanently lost.
