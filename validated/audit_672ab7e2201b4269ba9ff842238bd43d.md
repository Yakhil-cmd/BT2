### Title
Gateway L2 Gas Price Threshold Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`validate_resource_bounds` in the gateway's stateful validator reads the **current block's** `l2_gas_price` from the block header to compute the admission threshold, but transactions submitted now will be executed in the **next block** at `next_l2_gas_price`. Because `BlockHeaderWithoutHash` carries both fields and the EIP-1559 adjustment can move them apart by up to `1/gas_price_max_change_denominator` per block, the gateway systematically admits transactions that will fail execution (when the price rises) and rejects valid transactions (when the price falls).

---

### Finding Description

`BlockHeaderWithoutHash` stores two distinct L2 gas price fields:

- `l2_gas_price: GasPricePerToken` — the price **used in the block that was just committed**
- `next_l2_gas_price: GasPrice` — the price **that will be applied to the next block**, computed via EIP-1559 from `l2_gas_consumed` vs `gas_target` [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` by reading only `block_header.l2_gas_price`, silently dropping `block_header.next_l2_gas_price`: [2](#0-1) 

`validate_resource_bounds` then calls `get_block_info()` and uses the returned `strk_gas_prices.l2_gas_price` as the reference price. The TODO comment in the code itself acknowledges the wrong field is being read: [3](#0-2) 

`validate_tx_l2_gas_price_within_threshold` computes `threshold = min_gas_price_percentage * previous_block_l2_gas_price` and rejects any transaction whose `max_price_per_unit` is below it: [4](#0-3) 

The correct reference price is `next_l2_gas_price`, which is what the batcher will actually use when building the next block. The two values diverge every block via `calculate_next_base_gas_price`: [5](#0-4) 

---

### Impact Explanation

**Case 1 — price rising (next > current):** After a high-usage block, `next_l2_gas_price > l2_gas_price`. The gateway threshold is computed from the lower stale price, so transactions with `max_price_per_unit` in the range `[threshold_stale, next_l2_gas_price)` pass gateway admission. These transactions enter the mempool but will fail during batcher execution because their declared max price is below the actual block price. The gateway **admits invalid transactions**.

**Case 2 — price falling (next < current):** After a low-usage block, `next_l2_gas_price < l2_gas_price`. The gateway threshold is computed from the higher stale price, so transactions with `max_price_per_unit` in the range `[threshold_correct, threshold_stale)` are rejected even though they are perfectly executable in the next block. The gateway **rejects valid transactions**.

Both cases match the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The EIP-1559 adjustment fires every block. Any block whose `l2_gas_consumed` differs from `gas_target` produces a `next_l2_gas_price ≠ l2_gas_price`. This is the normal operating condition, not an edge case. Any user submitting a transaction immediately after a high- or low-usage block will be affected. No special privileges or coordination are required.

---

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, expose `block_header.next_l2_gas_price` alongside the existing fields — either by adding it to the returned `BlockInfo` or by returning it separately. Then in `validate_resource_bounds`, replace the call that reads `strk_gas_prices.l2_gas_price` with the `next_l2_gas_price` value from the same block header. The TODO comment at line 229 already identifies this exact fix.

---

### Proof of Concept

1. Block N is committed with `l2_gas_price = 1000 fri` and `l2_gas_consumed >> gas_target`, so `next_l2_gas_price = 1003 fri` (EIP-1559 increase).
2. `min_gas_price_percentage = 100`. Gateway threshold = `1 * 1000 = 1000 fri`.
3. User submits an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = 1001 fri`.
4. `validate_resource_bounds` reads `previous_block_l2_gas_price = 1000 fri` (stale), computes `threshold = 1000`, sees `1001 >= 1000` → **admits the transaction**.
5. Batcher builds block N+1 at `next_l2_gas_price = 1003 fri`. The transaction's max price `1001 < 1003` → the transaction **fails execution** (fee check error or revert).

The gateway has admitted a transaction that cannot be executed, polluting the mempool and wasting sequencer resources. The inverse scenario (price falling) causes valid transactions to be incorrectly rejected at the gateway.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
