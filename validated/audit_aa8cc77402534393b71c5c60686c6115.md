### Title
FROST `PresignOutput` Missing `ZeroizeOnDrop` on Secret Nonce Material Unlike All ECDSA Presign Outputs - (File: `src/frost/mod.rs`)

---

### Summary

The `PresignOutput<C>` struct in `src/frost/mod.rs` holds secret per-signature nonce material (`SigningNonces<C>`) but does not derive `ZeroizeOnDrop`. Every ECDSA presign output in the same codebase explicitly derives `ZeroizeOnDrop` for its secret fields. This inconsistency means FROST nonce material is not guaranteed to be cleared from memory when the struct is dropped, leaving it potentially recoverable from process memory, swap, or core dumps.

---

### Finding Description

`src/frost/mod.rs` defines the FROST presign output as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,   // ← secret per-signature nonces
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

No `ZeroizeOnDrop` (or `Zeroize`) derivation is present on this struct.

By contrast, both ECDSA presign output types explicitly derive `ZeroizeOnDrop`:

```rust
// OT-based ECDSA
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub k: Scalar, pub sigma: Scalar, }
``` [2](#0-1) 

```rust
// Robust ECDSA
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub c: Scalar, pub e: Scalar, ... }
``` [3](#0-2) 

`TripleShare` (also secret) likewise derives `ZeroizeOnDrop`: [4](#0-3) 

The inconsistency is compounded inside the signing functions. In `do_sign_participant_v1` and `do_sign_coordinator_v1` the nonces are wrapped in `Zeroizing::new(nonces)` before use, but in the v2 paths (`do_sign_coordinator_v2`, `do_sign_participant_v2`) the nonces are consumed directly from `presignature.nonces` without any `Zeroizing` wrapper:

```rust
// v2 coordinator – no Zeroizing wrapper
let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)
``` [5](#0-4) 

```rust
// v1 coordinator – correctly wrapped
let nonces = Zeroizing::new(nonces);
``` [6](#0-5) 

The same asymmetry exists in `src/frost/redjubjub/sign.rs`: the participant path wraps nonces in `Zeroizing`, but the coordinator path does not. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

`SigningNonces<C>` contains the hiding and binding nonces used in FROST round-2 signing. These are secret scalars. In FROST, a participant's signature share is:

```
z_i = nonce_i + lambda_i * x_i * c
```

If `nonce_i` is recovered from memory, an attacker can solve for the private key share:

```
x_i = (z_i - nonce_i) / (lambda_i * c)
```

Because `PresignOutput<C>` does not implement `ZeroizeOnDrop`, the nonce scalars are not guaranteed to be overwritten when the struct is dropped. They may persist in heap memory, process swap, or a core dump. This matches the allowed Critical impact: **disclosure of nonce material enabling reconstruction of private signing shares**.

---

### Likelihood Explanation

The `PresignOutput<C>` struct is a public, serializable, cloneable type. It is passed by value into signing functions and dropped at the end of those call frames. Any memory-disclosure primitive reachable after that point (heap inspection, swap-file read, crash dump, or a co-located process with shared memory) can expose the nonce scalars. The attacker does not need to break any cryptographic primitive; they only need to read memory that was not cleared. The inconsistency with the ECDSA paths shows the codebase's own security model requires zeroization here.

---

### Recommendation

1. Derive `ZeroizeOnDrop` on `PresignOutput<C>` in `src/frost/mod.rs`, mirroring the ECDSA presign outputs:

```rust
use zeroize::ZeroizeOnDrop;

#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    #[zeroize(skip)]
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
```

2. Consistently wrap nonces in `Zeroizing::new(...)` in all v2 signing paths (`do_sign_coordinator_v2`, `do_sign_participant_v2` in both `eddsa` and `redjubjub`) to match the v1 paths.

---

### Proof of Concept

1. Run the FROST EdDSA `sign_v2` protocol to completion; `PresignOutput` is dropped at the end of `fut_wrapper_v2`.
2. Because `PresignOutput<C>` has no `ZeroizeOnDrop`, the heap allocation holding `SigningNonces` is returned to the allocator without being overwritten.
3. A subsequent heap allocation of the same size will contain the raw nonce bytes.
4. Combined with the observed signature share `z_i` and the public challenge `c`, the private key share `x_i` is directly computable. [1](#0-0) [9](#0-8)
