### Title
`total_stake` is incorrectly increased during `switch_staking_delegation_pool` when the original staker was unstaking — (`src/staking/staking.cairo`)

---

### Summary

`switch_staking_delegation_pool` unconditionally calls `add_to_total_stake` for the switched amount. However, when the original staker was already in the process of unstaking at the time the exit intent was created, the amount was **not** removed from `total_stake` during `remove_from_delegation_pool_intent` (it had already been removed during `unstake_intent`). The unconditional `add_to_total_stake` in the switch path therefore inflates `total_stake` without any corresponding new token deposit — a direct analog to the ZeroLocker MERGE_TYPE supply inflation bug.

---

### Finding Description

**Root cause — `remove_from_delegation_pool_intent` skips total_stake update when staker is unstaking:** [1](#0-0) 

```cairo
// Do not update the total stake when the staker is in the process of unstaking,
// since its delegated stake is already excluded from the total stake.
if staker_info.unstake_time.is_none() {
    self
        .update_total_stake_according_to_delegated_stake_changes(
            :token_address, :old_delegated_stake, :new_delegated_stake,
        )
}
```

When `staker_info.unstake_time.is_some()`, the intent creation does **not** touch `total_stake`. This is correct because `unstake_intent` already removed the full delegated amount: [2](#0-1) 

```cairo
// Write off the delegated stake from the total stake.
for (pool_contract, token_address) in self.staker_pool_info.entry(staker_address).pools {
    let amount = self.get_delegated_balance(:staker_address, :pool_contract);
    self.remove_from_total_stake(:token_address, :amount);
    ...
}
```

**Root cause — `switch_staking_delegation_pool` adds to total_stake unconditionally:** [3](#0-2) 

```cairo
// Update `to_staker`'s delegated stake amount, and add to total stake.
...
self.add_to_total_stake(:token_address, amount: normalized_switched_amount);
```

There is no check on whether the `from_pool`'s intent was created while the original staker was unstaking. The function always adds `switched_amount` to `total_stake`, regardless of whether the amount was already excluded from `total_stake` at intent time.

**Accounting trace for the vulnerable scenario:**

| Step | Action | `total_stake` delta |
|---|---|---|
| 1 | `unstake_intent` (staker A, delegated = X) | `−X` |
| 2 | `exit_delegation_pool_intent(X)` (staker A unstaking) | `0` (skipped) |
| 3 | `unstake_action` (staker A exits, tokens leave staking contract) | `0` |
| 4 | `switch_delegation_pool(amount=X)` → `switch_staking_delegation_pool` | `+X` ← **incorrect** |

Net: `total_stake` is inflated by `X` with no new tokens deposited.

The spec for `switch_staking_delegation_pool` states unconditionally "Add requested amount to total stake": [4](#0-3) 

but does not account for the special case where the amount was never subtracted from `total_stake` during the intent phase.

---

### Impact Explanation

`total_stake` feeds directly into `yearly_mint` via `get_current_total_staking_power`: [5](#0-4) 

```cairo
let (total_stake, _) = staking_dispatcher.get_current_total_staking_power();
let yearly_mint = self.compute_yearly_mint(
    total_stake: total_stake.to_strk_native_amount(), :total_supply,
);
```

The formula is `C * sqrt(total_stake * total_supply)`: [6](#0-5) 

An inflated `total_stake` causes `yearly_mint` to return a higher value than warranted, causing the protocol to mint and distribute more STRK rewards than it should. This constitutes **theft of unclaimed yield** (High impact).

Additionally, `enter_delegation_pool_from_staking_contract` is called on `to_pool` crediting the pool member with `switched_amount`, but the actual tokens were already transferred out of the staking contract to `from_pool` during `unstake_action`. When the pool member later attempts to withdraw via `remove_from_delegation_pool_action`, the staking contract will attempt to transfer tokens it no longer holds, causing a revert — **temporary freezing of funds** (High impact).

---

### Likelihood Explanation

High. The scenario is a standard user flow: a delegator's staker enters the exit window, the delegator creates an exit intent, the staker completes the exit, and the delegator switches to a new staker. This exact sequence is exercised in the existing flow tests: [7](#0-6) 

No privileged access is required. Any unprivileged delegator can trigger this by switching after their staker exits.

---

### Recommendation

In `switch_staking_delegation_pool`, check whether the original staker was unstaking when the intent was created. One approach is to store a flag in `UndelegateIntentValue` indicating whether the amount was already excluded from `total_stake`. Only call `add_to_total_stake` if the amount was previously subtracted (i.e., the original staker was **not** unstaking at intent time):

```cairo
// Only add to total stake if the amount was removed during intent creation.
// If the original staker was unstaking, the amount was already excluded
// from total_stake during unstake_intent and was NOT re-subtracted during
// remove_from_delegation_pool_intent.
if !undelegate_intent_value.created_while_staker_unstaking {
    self.add_to_total_stake(:token_address, amount: normalized_switched_amount);
}
```

### Proof of Concept

1. Staker A stakes `S` STRK and opens a pool.
2. Delegator D delegates `X` tokens to staker A's pool.
3. Staker A calls `unstake_intent` → `total_stake` decreases by `S + X`.
4. Delegator D calls `exit_delegation_pool_intent(X)` → `total_stake` unchanged (staker A is unstaking).
5. Staker A calls `unstake_action` → staker A removed, `X` tokens transferred to pool A.
6. Staker B stakes `S` STRK and opens a pool. `total_stake = S`.
7. Delegator D calls `switch_delegation_pool(to_staker=B, to_pool=B_pool, amount=X)`.
8. `switch_staking_delegation_pool` executes → `add_to_total_stake(X)` → `total_stake = S + X`.

**Expected**: `total_stake = S` (only staker B's own stake; the `X` tokens are in pool A, not the staking contract).
**Actual**: `total_stake = S + X` — inflated by `X`, causing `yearly_mint` to return a higher value and minting excess rewards.
