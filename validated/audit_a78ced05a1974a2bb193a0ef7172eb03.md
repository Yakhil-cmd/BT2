### Title
Unverified `key_pair.public_key` in CKD Protocol Allows Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, each participant's `key_pair.public_key` (the master public key from DKG) is used to compute the hash point `H(pk ‖ app_id)` inside `compute_signature_share` without being verified against any established or shared value. A malicious participant can supply a `KeygenOutput` with a substituted `public_key`, causing their contribution to be computed over a different hash point. The coordinator blindly aggregates all shares, producing a corrupted `CKDOutput` that honest parties accept as valid.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, `compute_signature_share` (line 148) computes:

```rust
// line 168
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
// line 171
let big_s = hash_point * private_share.to_scalar();
// line 174
let big_c = big_s + app_pk * y.0;
```

The `key_pair.public_key` is the master public key established during DKG. All participants must use the **same** master public key for the Lagrange-interpolated sum to equal `msk · H(pk ‖ app_id)`. However, the CKD protocol never broadcasts or cross-checks `key_pair.public_key` among participants. Each participant independently supplies their own `KeygenOutput`, and the protocol trusts the caller to provide the correct value.

The public entry point `ckd()` (line 66) validates:
- Participant list uniqueness
- That `me` is in the participant list
- That `coordinator` is in the participant list

But it performs **no verification** that `key_pair.public_key` matches any agreed-upon master public key. There is no broadcast round, no transcript binding, and no consistency check across participants.

The coordinator in `do_ckd_coordinator` (line 35) simply receives each participant's `(norm_big_y, norm_big_c)` and sums them:

```rust
// lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

No validation of the received shares is performed before aggregation.

---

### Impact Explanation

The correct CKD output satisfies:

```
big_c = msk · H(pk ‖ app_id) + app_pk · big_y
```

If malicious participant `j` substitutes `pk_malicious` for `pk` in their `key_pair.public_key`, their contribution becomes:

```
norm_big_c_j = λ_j · (x_j · H(pk_malicious ‖ app_id) + app_pk · y_j)
```

The aggregated output becomes:

```
big_c = Σ_{i≠j} λ_i · x_i · H(pk ‖ app_id)
      + λ_j · x_j · H(pk_malicious ‖ app_id)
      + app_pk · big_y
```

This is not equal to `msk · H(pk ‖ app_id) + app_pk · big_y`. When the application unmasks with `app_sk`:

```
confidential_key = big_c − app_sk · big_y  ≠  msk · H(pk ‖ app_id)
```

Honest parties (the application) accept this corrupted output as the CKD result, receiving an unusable or attacker-influenced confidential derived key. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

---

### Likelihood Explanation

Any participant in the CKD protocol is an unprivileged library caller who supplies their own `KeygenOutput`. There is no external enforcement that `key_pair.public_key` matches the DKG-established master public key. A malicious participant needs only to construct a `KeygenOutput` with an arbitrary `public_key` field — a trivial struct substitution requiring no cryptographic capability. The attack is reachable on every CKD invocation where at least one participant is adversarial.

---

### Recommendation

Add a consistency-verification step for `key_pair.public_key` across all participants before computing shares. Concretely:

1. **Broadcast and verify**: In an initial round, each participant broadcasts `key_pair.public_key`. Every participant verifies that all received values are identical before proceeding to `compute_signature_share`.
2. **Bind to transcript**: Include `key_pair.public_key` in the protocol transcript/session hash so that any divergence is detectable.
3. **Validate at entry**: At minimum, the `ckd()` entry point should accept the expected master public key as a separate, authoritative parameter and verify that `key_pair.public_key` matches it before starting the protocol.

---

### Proof of Concept

```
Setup: 3 participants A (honest), B (honest), C (malicious).
       Master public key from DKG: pk.
       C constructs KeygenOutput { private_share: x_C, public_key: pk_malicious }.

Step 1: C calls compute_signature_share with pk_malicious.
        hash_point_C = H(pk_malicious ‖ app_id)   ← wrong hash
        big_s_C      = x_C · hash_point_C          ← wrong contribution
        big_c_C      = big_s_C + app_pk · y_C

Step 2: Coordinator aggregates:
        big_c = λ_A·x_A·H(pk‖app_id) + λ_B·x_B·H(pk‖app_id)
              + λ_C·x_C·H(pk_malicious‖app_id)
              + app_pk · big_y
              ≠ msk · H(pk ‖ app_id) + app_pk · big_y

Step 3: Application unmasks:
        confidential_key = big_c − app_sk · big_y
                         ≠ msk · H(pk ‖ app_id)

Result: Application accepts a corrupted confidential derived key.
        No error is raised anywhere in the protocol.
```

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
