### Title
Missing Cross-Phase Participant-Set Validation in OT-Based ECDSA Presigning - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary
The `presign` function in the OT-based ECDSA module explicitly omits the check that the presigning participant set is a subset of the triple generation participant set. The `TriplePub` struct records which participants generated each triple, but this metadata is never validated against the presigning participants. A malicious coordinator can include a participant who was not present during triple generation, causing honest parties to produce an unusable presignature or triggering a protocol abort.

### Finding Description
The orchestration model requires a strict subset relationship across phases:

```
P (keygen) ⊇ P0 (triples) ⊇ P1 (presign) ⊇ P2 (sign)
```

`TriplePub` stores the participants who generated the triple:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    pub threshold: ReconstructionLowerBound,
}
``` [1](#0-0) 

The `presign` function receives both the presigning `participants` and `args.triple0.1` / `args.triple1.1` (the `TriplePub` values containing the triple generation participants). However, it explicitly skips the subset check:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [2](#0-1) 

The only cross-phase check that is performed is a threshold equality check:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [3](#0-2) 

This verifies the threshold value matches but never verifies that `participants ⊆ triple0.1.participants` and `participants ⊆ triple1.1.participants`. The `TriplePub.participants` field is stored but never consumed by `do_presign`.

Inside `do_presign`, Lagrange coefficients are computed relative to the presigning participant set:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
``` [4](#0-3) 

If the presigning set differs from the triple generation set, these Lagrange coefficients are computed over the wrong domain, making the linearized shares cryptographically inconsistent with the triple's public commitments (`big_a`, `big_b`, `big_c`, `big_e`).

### Impact Explanation
When a participant included in the presigning set was not present during triple generation, they hold no valid triple share. Their contribution causes the global consistency checks inside `do_presign` to fail:

```rust
if big_e != (ProjectivePoint::GENERATOR * e).to_affine() {
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of kd".to_string(),
    ));
}
``` [5](#0-4) 

```rust
if (ProjectivePoint::GENERATOR * alpha != big_k + big_a)
``` [6](#0-5) 

All honest parties abort with a `ProtocolError`, producing no usable presignature. This maps to **High: Corruption of presign outputs so honest parties accept unusable cryptographic outputs**, since the presigning round completes with no valid output for any honest participant.

Even in the subtler case where the presigning set is a valid subset of the triple generation set but uses a different Lagrange basis (e.g., presigning over `{A, B}` using triples generated over `{A, B, C}` with threshold 3), the linearization step applies coefficients from the wrong interpolation domain, silently corrupting the presignature values `(k_i, sigma_i)` before any check can catch it.

### Likelihood Explanation
The attacker is a malicious coordinator or any library caller who constructs `PresignArguments`. The `PresignArguments` struct is a plain public struct with no encapsulation:

```rust
pub struct PresignArguments {
    pub triple0: (TripleShare, TriplePub),
    pub triple1: (TripleShare, TriplePub),
    pub keygen_out: KeygenOutput,
    pub threshold: ReconstructionLowerBound,
}
``` [7](#0-6) 

Any caller can freely mix a `TriplePub` from one participant set with a `participants` slice from a different set. The coordinator role is explicitly listed as a potential adversary in the library's threat model (the robust ECDSA module documents split-view attacks by malicious coordinators). A malicious coordinator who controls which participants are invited to a presigning session can trivially trigger this by including a participant who was absent from the triple generation round.

### Recommendation
Add an explicit subset check in `presign` before constructing the protocol:

```rust
let triple0_participants: std::collections::HashSet<_> =
    args.triple0.1.participants.iter().copied().collect();
let triple1_participants: std::collections::HashSet<_> =
    args.triple1.1.participants.iter().copied().collect();

for p in participants.participants() {
    if !triple0_participants.contains(p) || !triple1_participants.contains(p) {
        return Err(InitializationError::BadParameters(
            format!("Participant {p:?} was not present during triple generation"),
        ));
    }
}
```

This mirrors the existing cross-phase threshold check and enforces the `P1 ⊆ P0` invariant documented in `docs/ecdsa/ot_based_ecdsa/orchestration.md`. [8](#0-7) 

### Proof of Concept
1. Run triple generation with participants `{A, B, C}`, threshold 2. Each party receives a `(TripleShare, TriplePub)` where `TriplePub.participants = [A, B, C]`.
2. Construct a presigning session with participants `{A, B, D}` where `D` was absent from step 1. Pass `D` a random `TripleShare`.
3. Call `presign(&[A, B, D], D, PresignArguments { triple0: (random_share, triple_pub_from_step1), ... })`. The function accepts the call because the only check performed is `args.threshold == triple0.1.threshold`.
4. Inside `do_presign`, `D` sends a random `e_i`. The global check `big_e != (G * e).to_affine()` fails, and all honest parties (`A`, `B`) receive `ProtocolError::AssertionFailed`, producing no usable presignature for this round.
