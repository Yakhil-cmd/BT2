### Title
Unprivileged Caller Can Permanently Deny Block Rewards to All Stakers via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

### Summary
The public `update_rewards` function in the Staking contract accepts a caller-controlled `disable_rewards: bool` parameter. It unconditionally writes the current block number to the global `last_reward_block` storage variable **before** checking `disable_rewards`. Because `last_reward_block` is a single global gate, any unprivileged caller can invoke `update_rewards(any_valid_staker, disable_rewards: true)` once per block to permanently forfeit that block's rewards for every staker and delegator in the protocol.

### Finding Description
`update_rewards` is gated only by `general_prerequisites()`, which checks the pause flag and a non-zero caller. No role or identity check restricts who may supply the `disable_rewards` argument. [1](#0-0) 

The function immediately writes `current_block_number` to `last_reward_block`: [2](#0-1) 

Only **after** that write does it branch on `disable_rewards`: [3](#0-2) 

Because `last_reward_block` is a single global variable (not per-staker), once it is advanced to block `N` with `disable_rewards: true`, every subsequent call to `update_rewards` for any staker in block `N` reverts with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The rewards that would have been calculated and forwarded to `reward_supplier_dispatcher.update_unclaimed_rewards_from_staking_contract` are never requested; they are permanently lost. [5](#0-4) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

When consensus rewards are active (`is_pre_consensus()` returns `false`), each block produces STRK and BTC block rewards that are split among all stakers and their delegators. By calling `update_rewards(any_active_staker, disable_rewards: true)` once per block, an attacker permanently forfeits the entire protocol's block rewards for that block. Repeating this every block eliminates all yield indefinitely. Stakers and delegators never receive the rewards they are owed; the yield is not deferred — it is gone.

### Likelihood Explanation
**High.** The function is public with no access control beyond a non-zero caller check. The attacker needs only a valid staker address (publicly readable from the `stakers` vector) and enough gas for one transaction per block. There is no economic barrier and no way for stakers to prevent it.

### Recommendation
1. **Remove `disable_rewards` from the public interface entirely**, or restrict it to a privileged role (e.g., `only_app_governor`).
2. Alternatively, move the `last_reward_block` write to **after** the `disable_rewards` guard so that a skipped-reward call does not consume the block's reward slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
// Only advance last_reward_block when rewards are actually processed.
self.last_reward_block.write(current_block_number);
// ... reward calculation ...
```

### Proof of Concept
1. Consensus rewards are active (`consensus_rewards_first_epoch` has been set and the current epoch has passed it).
2. Attacker reads any active staker address from the public `stakers` storage vector.
3. At the start of block `N`, attacker calls:
   ```
   staking.update_rewards(staker_address, disable_rewards: true)
   ```
4. `last_reward_block` is written to `N`; no rewards are distributed.
5. The legitimate staker (or keeper) attempts:
   ```
   staking.update_rewards(staker_address, disable_rewards: false)
   ```
   This reverts with `REWARDS_ALREADY_UPDATED` because `current_block_number (N) > last_reward_block (N)` is false.
6. All stakers and their delegators permanently lose block `N`'s rewards.
7. Repeating step 3 every block eliminates all protocol yield indefinitely at the cost of one transaction per block.
