### Title
`FeeEstimation` omits `l2_gas_consumed`, causing `starknet_estimateFee` / `starknet_simulateTransactions` to return an internally inconsistent fee breakdown unusable for constructing valid V3 resource bounds - (File: `crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is computed from the full `GasVector` (L1 + L1-data + L2 gas), while `gas_consumed` and `data_gas_consumed` only reflect L1 and L1-data gas respectively. This makes the response internally inconsistent: the documented formula `overall_fee = gas_consumed Ă— l1_gas_price + data_gas_consumed Ă— l1_data_gas_price` silently under-counts the fee whenever L2 gas is consumed, and callers have no way to derive the L2 gas amount needed to set valid `l2_gas.max_amount` resource bounds for V3 transactions.

---

### Finding Description

In `tx_execution_output_to_fee_estimation`, the `GasVector` field `l2_gas` is silently dropped:

```rust
// crates/apollo_rpc_execution/src/objects.rs  lines 161-183
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed:      gas_vector.l1_gas.0.into(),       // L1 gas only
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),  // L1-data gas only
        l1_data_gas_price,
        l2_gas_price,
        // gas_vector.l2_gas is never used here
        overall_fee: tx_execution_output.execution_info.receipt.fee, // includes L2 gas cost
        unit: tx_execution_output.price_unit,
    })
}

The OpenAPI spec for `FEE_ESTIMATE` documents the formula as:

> `overall_fee` â€¦ equals `gas_consumed*gas_price + data_gas_consumed*data_gas_price` [3](#0-2) 

This formula is wrong for any V3 transaction that consumes L2 gas: `overall_fee` will exceed `gas_consumed Ă— l1_gas_price + data_gas_consumed Ă— l1_data_gas_price` by exactly `l2_gas_consumed Ă— l2_gas_price`.

The downstream consequence mirrors the ERC-4626 `maxWithdraw` / `withdrawalPenalty` pattern: `check_fee_bounds` in `perform_pre_validation_stage` enforces that `resource_bounds.l2_gas.max_amount â‰Ą minimal_l2_gas`:

```rust

A caller who faithfully uses the `estimate_fee` response to populate V3 resource bounds has no field from which to read `l2_gas_consumed`. Setting `l2_gas.max_amount = 0` (the only value derivable from the response) causes `InsufficientResourceBounds { MaxGasAmountTooLow { resource: L2Gas } }` at pre-validation, rejecting the transaction.

---

### Impact Explanation

**High â€” RPC fee estimation returns an authoritative-looking wrong value.**

1. **Broken formula**: Any caller computing `gas_consumed Ă— l1_gas_price + data_gas_consumed Ă— l1_data_gas_price` obtains a fee lower than `overall_fee` for every Cairo 1 / V3 transaction that consumes L2 gas. The discrepancy is not a rounding error; it is the entire L2 gas cost.

2. **Unusable for resource-bound construction**: A wallet or SDK that calls `estimate_fee` to determine the `l2_gas.max_amount` field for a subsequent V3 transaction cannot derive that value from the response. The only safe workaround is to set an arbitrary large L2 gas bound, which defeats the purpose of fee estimation and may cause the transaction to be rejected by the gateway's `max_l2_gas_amount` cap.

3. **Scope**: `starknet_estimateFee`, `starknet_simulateTransactions`, and `starknet_estimateMessageFee` all call `tx_execution_output_to_fee_estimation` and are all affected. [5](#0-4) 

---

### Likelihood Explanation

Every V3 (`AllResources`) transaction that invokes a Cairo 1 contract consumes L2 gas. This is the standard transaction type on the current Starknet protocol. The affected RPC endpoints are publicly reachable. Any wallet, SDK, or dApp that calls `estimate_fee` and then uses the result to construct resource bounds will encounter this inconsistency.

---

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`.
2. Update the OpenAPI description of `overall_fee` to include the L2 gas term: `gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price + l2_gas_consumed*l2_gas_price`.
3. Remove the TODO comment once the field is added.

---

### Proof of Concept

1. Submit a V3 invoke transaction targeting any Cairo 1 contract to `starknet_estimateFee`.
2. Receive `FeeEstimation { gas_consumed: G1, data_gas_consumed: G2, l2_gas_price: P2, overall_fee: F, â€¦ }`.
3. Compute `G1 Ă— l1_gas_price + G2 Ă— l1_data_gas_price`. Observe the result is strictly less than `F` by `l2_gas_consumed Ă— P2`.
4. Construct a V3 transaction with `l2_gas.max_amount = 0` (the only value derivable from the response).
5.     #[instrument(skip(self, transactions), level = "debug", err, ret)]
    async fn estimate_fee(
        &self,
        transactions: Vec<BroadcastedTransaction>,
        simulation_flags: Vec<SimulationFlag>,
        block_id: BlockId,
    ) -> RpcResult<Vec<FeeEstimation>> {
        trace!("Estimating fee of transactions: {:#?}", transactions);
        let validate = !simulation_flags.contains(&SimulationFlag::SkipValidate);

        let storage_txn = self.storage_reader.begin_ro_txn().map_err(internal_server_error)?;

        let maybe_pending_data = if let BlockId::Tag(Tag::Pending) = block_id {
            Some(client_pending_data_to_execution_pending_data(
                read_pending_data(&self.pending_data, &storage_txn).await?,
                self.pending_classes.read().await.clone(),
            ))
        } else {
            None
        };

        let executable_txns =
            transactions.into_iter().map(|tx| tx.try_into()).collect::<Result<_, _>>()?;

        let block_number = get_accepted_block_number(&storage_txn, block_id)?;
        let block_not_reverted_validator =
            BlockNotRevertedValidator::new(block_number, &storage_txn)?;
        drop(storage_txn);
        let state_number = StateNumber::unchecked_right_after_block(block_number);
        let execution_config = self.execution_config;

        let chain_id = self.chain_id.clone();
        let reader = self.storage_reader.clone();
        let class_manager_client =
            create_class_manager_client(self.class_manager_client.clone()).await;

        let estimate_fee_result = tokio::task::spawn_blocking(move || {
            exec_estimate_fee(
                executable_txns,
                &chain_id,
                reader,
                maybe_pending_data,
                state_number,
                block_number,
                &execution_config,
                validate,
                DONT_IGNORE_L1_DA_MODE,
                class_manager_client,
            )
        })
        .await
        .map_err(internal_server_error)?;

        block_not_reverted_validator.validate(&self.storage_reader)?;

        match estimate_fee_result {
            Ok(Ok(fees)) => Ok(fees),
            Ok(Err(reverted_tx)) => {
                Err(ErrorObjectOwned::from(JsonRpcError::<TransactionExecutionError>::from(
                    TransactionExecutionError {
                        transaction_index: reverted_tx.index,
                        execution_error: reverted_tx.revert_reason,
                    },
                )))
            }
            Err(err) => Err(internal_server_error(err)),
        }
    }
