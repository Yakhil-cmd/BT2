Let me look at how the class manager is populated for deprecated classes at gateway submission time, and how `ExecutionStateReader` is constructed.

Based on my thorough analysis of the code, here is my assessment:

### Title
Missing `is_declared` Check for Cairo 0 (V0) Classes in `get_compiled_class` When Class Manager Is Active — (`crates/apollo_rpc_execution/src/state_reader.rs`)

### Summary

`ExecutionStateReader::get_compiled_class` applies a state-number declaration check for `ContractClass::V1` (Sierra) classes but unconditionally returns `RunnableCompiledClass::V0` for `ContractClass::V0` (deprecated) classes when `class_manager_handle` is `Some`. A deprecated class that exists in the class manager's persistent storage but is **not declared at the queried `state_number`** (e.g., because the declaring block was reverted) will be returned without error, allowing its bytecode to be executed.

### Finding Description

In `get_compiled_class` (lines 115–141), the two branches diverge:

**V1 branch — declaration check enforced:** [1](#0-0) 

**V0 branch — declaration check absent:** [2](#0-1) 

The TODO comment at line 136 explicitly acknowledges the missing check: `"Verify cairo0 as well after get_class_definition_block_number is fixed."` The reason the check was deferred is that `is_contract_class_declared` (used for V1) calls `get_class_definition_block_number`, which queries only the **Sierra** (`declared_classes_block_table`) table: [3](#0-2) 

For deprecated classes the correct table is `deprecated_declared_classes_block_table`, accessed via `get_deprecated_class_definition_block_number`: [4](#0-3) 

The class manager (`FsClassStorage` / `CachedClassStorage`) stores deprecated classes persistently and has **no revert mechanism**. When a block is reverted, the storage-layer declaration is removed: [5](#0-4) 

But the class manager retains the class: [6](#0-5) 

The `ExecutionStateReader` is constructed with `class_manager_handle: Some(...)` in the RPC execution path used for simulation, fee estimation, and tracing: [7](#0-6) 

### Impact Explanation

If a deprecated class is in the class manager but not declared at the queried `state_number`, `get_compiled_class` returns `Ok(RunnableCompiledClass::V0(...))` instead of `Err(StateError::UndeclaredClassHash(...))`. Any transaction that makes a `library_call` syscall with that class hash will execute the reverted/undeclared deprecated bytecode. This is a **wrong compiled class selected for execution** — the exact corrupted value is the `RunnableCompiledClass::V0` artifact returned for a class hash that has no valid declaration at the current state number.

This falls under: **Critical. Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution.**

### Likelihood Explanation

**Low.** The precondition is that a deprecated class must be present in the class manager but absent from the declared-classes state at the queried `state_number`. In practice this requires a block revert (e.g., during a sequencer reorg), which is not directly attacker-controlled. An attacker who previously submitted a `Declare V0` transaction (knowing the class hash) could opportunistically exploit the window after a revert by submitting an invoke transaction that `library_call`s the reverted hash. The window exists as long as the class manager retains the class (permanently, since there is no revert path) and the block state does not re-declare it.

### Recommendation

Replace the unconditional V0 return with a declaration check using `get_deprecated_class_definition_block_number` (the correct table for deprecated classes), mirroring the V1 branch:

```rust
ContractClass::V0(deprecated_contract_class) => {
    let is_declared = txn
        .get_state_reader()?
        .get_deprecated_class_definition_block_number(&class_hash)?
        .is_some_and(|block_number| self.state_number.is_after(block_number));
    if is_declared {
        Ok(RunnableCompiledClass::V0(deprecated_contract_class.try_into()?))
    } else {
        Err(StateError::UndeclaredClassHash(class_hash))
    }
}
```

This resolves the TODO and closes the asymmetry between V0 and V1 handling.

### Proof of Concept

1. Construct an `ExecutionStateReader` with `class_manager_handle: Some(...)` pointing to a mock class manager that returns `ContractClass::V0(DeprecatedContractClass::default())` for a chosen `class_hash`.
2. Leave the storage writer with **no state diff** declaring that `class_hash` (simulating a never-declared or reverted class).
3. Call `exec_state_reader.get_compiled_class(class_hash)`.
4. **Observed:** `Ok(RunnableCompiledClass::V0(...))` is returned.
5. **Expected:** `Err(StateError::UndeclaredClassHash(class_hash))`.

The existing test infrastructure in `crates/apollo_rpc_execution/src/state_reader_test.rs` (function `set_execution_state_reader`) already provides the scaffolding to write this test — the V1 analogue (`class_not_declared_but_in_class_manager`) is already tested and correctly returns `UndeclaredClassHash`; the V0 analogue is absent.
