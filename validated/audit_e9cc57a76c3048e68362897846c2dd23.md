Audit Report

## Title
Silent ICP Burn Failure Allows Cycles Minting Without Burning ICP - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally returns `()` regardless of whether the ICP ledger burn call succeeds or fails. After cycles or a canister have been delivered and the notification permanently consumed, a failed burn leaves ICP stranded in the CMC's subaccount with no retry path, creating cycles in circulation that are not backed by burned ICP and violating the ICP/cycles conservation invariant.

## Finding Description
`burn_and_log` (lines 2014–2049) is explicitly designed to swallow errors:

```rust
// Line 2015-2016
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    ...
    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
```

Both branches return `()`. Its three callers — `process_create_canister` (line 1945), `process_mint_cycles` (line 1968), and `process_top_up` (line 2001) — all pattern-match on the success of the cycles/canister delivery operation and then call `burn_and_log(...).await` before returning `Ok(...)`. The `Ok` result propagates back to the `notify_*` handler, which then permanently records the block as `NotifiedTopUp` / `NotifiedCreateCanister` / `NotifiedMint` (lines 1214–1222). Once that status is written, any subsequent call with the same block index returns the cached result without re-attempting the burn. There is no heartbeat, timer, or durable failure record to retry a failed burn.

The exploit path:
1. User sends ICP to the CMC subaccount and calls `notify_top_up` (or `notify_create_canister` / `notify_mint_cycles`).
2. CMC marks the block `Processing`, successfully deposits cycles into the target canister.
3. CMC calls `burn_and_log`; the ICP ledger rejects the inter-canister call (e.g., `CanisterStopping` during a ledger upgrade).
4. `burn_and_log` logs the error and returns `()`.
5. `process_top_up` returns `Ok(cycles)`; the handler writes `NotifiedTopUp(Ok(cycles))`.
6. The ICP remains in the CMC's subaccount. A second `notify_top_up` call returns the cached `Ok(cycles)` — no second burn is attempted.

Existing guards are insufficient: `is_transient_error` only removes the `Processing` entry on transient errors from the cycles/canister delivery step, not from `burn_and_log`. The deduplication cache (`blocks_notified`) prevents double-delivery but also permanently prevents any retry of the burn.

## Impact Explanation
This constitutes illegal minting of cycles: cycles are created and delivered to users without the corresponding ICP being destroyed. The ICP/cycles conservation invariant — that every cycle in existence corresponds to burned ICP at the prevailing XDR rate — is violated. The ICP is not stolen but is permanently stranded in the CMC's subaccount with no automated recovery. Over repeated ledger upgrade windows, the discrepancy accumulates silently, inflating the effective ICP supply relative to cycles in circulation. This matches the allowed impact: **illegal minting of ICP/Cycles assets with concrete protocol harm** (High, $2,000–$10,000), as it requires a specific timing condition (ledger upgrade) and the per-occurrence amounts are bounded by individual notify calls, but the violation is permanent and cumulative.

## Likelihood Explanation
The ICP ledger is upgraded periodically via NNS governance proposals. During the upgrade window, the ledger canister is briefly unavailable and will reject inter-canister calls with a `CanisterStopping` or `CanisterNotFound` error. Any `notify_*` call that completes its cycles delivery step during this window will trigger the silent burn failure. No attacker capability beyond submitting a normal unprivileged ingress message is required. The entry path is fully unprivileged and the timing window, while narrow, recurs with every ledger upgrade.

## Recommendation
`burn_and_log` should be refactored to return a `Result`. If the design intent is to never fail the notification (to prevent double-delivery of cycles), the CMC should record failed burns in durable state — for example, a `pending_burns: Vec<(Subaccount, Tokens)>` field — and retry them in a `heartbeat` or timer callback, analogous to how `blocks_notified` tracks notification status. At minimum, a failed burn should be recorded in stable state so that operators and NNS monitoring can detect and remediate the conservation discrepancy. The current approach of logging only to canister logs is insufficient because logs are not durable across upgrades.

## Proof of Concept
1. Deploy a local replica with the CMC and ICP ledger canisters.
2. Send ICP to the CMC's top-up subaccount for a target canister.
3. Stop the ICP ledger canister (simulating an upgrade window): `dfx canister stop <ledger_id>`.
4. Call `notify_top_up` on the CMC with the corresponding block index.
5. Observe: `notify_top_up` returns `Ok(cycles)`; the target canister receives cycles; the CMC logs a burn failure.
6. Restart the ledger canister.
7. Call `notify_top_up` again with the same block index: returns cached `NotifiedTopUp(Ok(cycles))` — no burn is attempted.
8. Query the CMC's subaccount on the ledger: the original ICP amount is still present, unburned.
9. Confirm cycles were minted and delivered without ICP being destroyed.
