### Title
COA Ownership Proof Weight Threshold Bypass via Duplicate Key Index in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` in `contract.cdc` allows an attacker to bypass the key-weight threshold (1000) by supplying the same key index multiple times in the caller-controlled `keyIndices` array. The `seenAccountKeyIndices` map prevents the key from being added to `keyList` more than once, but it does **not** prevent the same `keyListIndex` from appearing multiple times in `signatureSet`. Because `Crypto.KeyList.verify()` accumulates weight for every valid signature entry without deduplicating by key index, a single low-weight key can be made to count N times, bypassing the threshold.

---

### Finding Description

In `validateCOAOwnershipProof` the loop at lines 1035–1080 of `contract.cdc` iterates over the caller-supplied `signatures` array:

```cadence
for signatureIndex, signature in signatures {
    let accountKeyIndex = Int(keyIndices[signatureIndex]!)
    var keyListIndex = 0

    if !seenAccountKeyIndices.containsKey(accountKeyIndex) {
        // key added to keyList once; keyListIndex assigned
        keyList.add(key.publicKey, hashAlgorithm: key.hashAlgorithm, weight: key.weight / 1000.0)
        keyListIndex = keyListLength
        keyListLength = keyListLength + 1
        seenAccountKeyIndices[accountKeyIndex] = keyListIndex
    } else {
        // duplicate accountKeyIndex: reuse existing keyListIndex
        keyListIndex = seenAccountKeyIndices[accountKeyIndex]!
    }

    // ← appended in BOTH branches, no guard on keyListIndex uniqueness
    signatureSet.append(Crypto.KeyListSignature(keyIndex: keyListIndex, signature: signature))
}

let isValid = keyList.verify(signatureSet: signatureSet, ...)
``` [1](#0-0) 

The `seenAccountKeyIndices` map deduplicates entries in `keyList` (so the key's weight is registered only once). However, `signatureSet` receives a new `Crypto.KeyListSignature` entry on **every** loop iteration, including repeated ones. The comment at line 1072 — *"`Crypto.KeyList.verify()` knows how to handle duplicate keys"* — refers to duplicate keys in the key list, not to duplicate key indices in the signature set. [2](#0-1) 

`Crypto.KeyList.verify()` processes each entry in `signatureSet` independently and accumulates weight for every valid signature. There is no deduplication by `keyIndex` inside `verify()`. Consequently, if `keyIndices = [k, k, …, k]` (N repetitions) and each corresponding signature is valid, the weight of key `k` is added N times, yielding a total weight of `N × (key.weight / 1000.0)`.

---

### Impact Explanation

An attacker who controls a Cadence account with a key of weight `W` where `W < 1000` but `W × N ≥ 1000` for some integer `N` can forge a passing COA ownership proof by repeating the same key index `N` times. This bypasses the weight threshold that enforces multi-key authorization for accounts with partial-weight keys (e.g., a 2-of-2 multisig where each key has weight 500).

A successful bypass allows the attacker to:
1. Pass `ERC-1271 isValidSignature` checks in EVM contracts via the COA contract's `isValidSignature` function.
2. Satisfy any EVM-side authorization gate that calls `verifyCOAOwnershipProof` through the Cadence Arch precompile.
3. Authorize EVM operations (token transfers, contract calls, etc.) that are gated on COA ownership proof, potentially draining assets or executing unauthorized state changes. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

The attack is fully attacker-controlled and requires no privileged access:

- The `COAOwnershipProof` (including `keyIndices` and `signatures`) is passed as raw bytes in the EVM call to the Cadence Arch precompile at the well-known address. Any EVM contract or EOA can craft this payload.
- The attacker only needs to own a Cadence account with a key of weight < 1000 — a common configuration for multi-sig accounts (e.g., two keys of weight 500 each).
- The attacker signs `signedData` with their own key (which they legitimately possess) and simply repeats the key index N times.
- No special node access, leaked keys, or social engineering is required. [5](#0-4) [6](#0-5) 

---

### Recommendation

Reject the proof if any `accountKeyIndex` appears more than once in `keyIndices`. Add the following check inside the loop, before appending to `signatureSet`:

```cadence
// After resolving keyListIndex in the else branch:
} else {
    // Duplicate key index in the proof — reject to prevent weight multiplication
    return ValidationResult(
        isValid: false,
        problem: "EVM.validateCOAOwnershipProof(): Duplicate key index \(accountKeyIndex) in proof."
    )
}
```

Alternatively, track which `keyListIndex` values have already been added to `signatureSet` and skip or reject duplicates before calling `keyList.verify()`. [7](#0-6) 

---

### Proof of Concept

1. Create a Cadence account `A` with a single key at index 0 with weight **500** (below the 1000 threshold).
2. Create and store a COA for account `A` at `/storage/coa`, publish capability at `/public/coa`.
3. Sign `signedData` with key 0 to produce `validSig`.
4. Craft a `COAOwnershipProof`:
   - `address` = A
   - `path` = `/public/coa`
   - `keyIndices` = `[0, 0]`
   - `signatures` = `[validSig, validSig]`
5. ABI-encode the proof and call `verifyCOAOwnershipProof(address(COA), hash, encodedProof)` on the Cadence Arch precompile from any EVM contract.
6. **Expected (secure) result**: validation fails because key 0 has weight 500 < 1000.
7. **Actual result**: `keyList` contains key 0 with weight `0.5`; `signatureSet` contains two entries both pointing to `keyListIndex = 0`; `keyList.verify()` accumulates `0.5 + 0.5 = 1.0 ≥ 1.0` → **returns `true`**, bypassing the threshold. [8](#0-7) [9](#0-8)
