### Title
`TurnBridge` Message Handler Is a No-Op, Permanently Defeating the Bridge Emergency-Disable Control — (File: `x/cronos/keeper/msg_server.go`)

---

### Summary

The `TurnBridge` gRPC message handler in `x/cronos/keeper/msg_server.go` is implemented as a complete no-op: it performs no permission check, writes no state, and always returns `(nil, nil)`. Any caller—privileged or not—can submit a `MsgTurnBridge` transaction that is accepted on-chain, yet the bridge enable/disable state is never altered. The admin's emergency circuit-breaker is permanently inoperative.

---

### Finding Description

`MsgTurnBridge` is the designated mechanism for the Cronos admin (or a permissioned address holding `CanTurnBridge`) to halt bridge activity in an emergency. ADR-009 explicitly lists it as a restricted, permission-gated message. The permission constant `CanTurnBridge` is defined and the `HasPermission` helper exists to enforce it. Yet the server-side handler is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Contrast this with every other restricted handler in the same file, which all perform explicit authorization checks before touching state:

- `UpdateTokenMapping` checks `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` [2](#0-1) 
- `UpdatePermissions` checks `admin != msg.From` [3](#0-2) 
- `StoreBlockList` checks `admin != msg.From` [4](#0-3) 
- `UpdateParams` checks `msg.Authority != k.authority` [5](#0-4) 

`TurnBridge` does none of these things and writes nothing to the KV store.

The permission bit `CanTurnBridge` is defined and wired into the permission system:

```go
CanTurnBridge = 2
All           = CanChangeTokenMapping | CanTurnBridge
``` [6](#0-5) 

ADR-009 confirms the intent:

> "For now, there are only two messages that require permission: `MsgUpdateTokenMapping` and `MsgTurnBridge`." [7](#0-6) 

The integration test for the gravity bridge explicitly verifies that after `turn_bridge false` succeeds, subsequent `send_to_evm_chain` calls revert — confirming that downstream bridge logic is expected to consult the bridge-enabled state that `TurnBridge` is supposed to set. Because the handler is a no-op, that state is never written, so the bridge remains permanently enabled regardless of what the admin submits.

---

### Impact Explanation

**High — Bypass of Cronos admin bridge authorization checks.**

The admin's only on-chain mechanism to halt bridge flows in an emergency is `MsgTurnBridge`. Because the handler is a no-op:

1. The bridge-enabled flag is never set to `false` in state.
2. Any downstream check that gates bridge operations on that flag will always see the default (enabled) value.
3. An unprivileged user can continue to submit bridge transactions even after the admin has broadcast a `MsgTurnBridge(enable=false)` that was accepted on-chain.
4. Simultaneously, an unprivileged user can broadcast their own `MsgTurnBridge(enable=true)` — it will also succeed (no permission check) and also do nothing, but it demonstrates the complete absence of access control.

The net effect is that the bridge circuit-breaker — a critical safety control — is permanently bypassed.

---

### Likelihood Explanation

The condition is always present in the current codebase; no special attacker action is required. Any on-chain emergency that would normally prompt the admin to disable the bridge (e.g., a CRC20 exploit, an IBC accounting bug, a gravity-bridge drain) will leave the bridge running because the disable path is silently inoperative.

---

### Recommendation

Implement `TurnBridge` to:

1. Verify the caller holds `CanTurnBridge` permission (mirror the pattern in `UpdateTokenMapping`).
2. Persist the `enable` flag to the KV store (e.g., under a dedicated `KeyBridgeEnabled` key in `x/cronos/types`).
3. Have all bridge-entry points (gravity `SendToEthereum`, IBC send hooks, CRC21 `send_to_evm_chain` precompile path) read and enforce that flag before processing.

---

### Proof of Concept

1. Deploy Cronos locally with the current codebase.
2. As an **unprivileged** account (no `CanTurnBridge` permission), broadcast:
   ```
   cronosd tx cronos turn-bridge false --from attacker
   ```
3. Observe: transaction is accepted on-chain with code `0` and no error.
4. As the **admin**, broadcast the same message with `from_="validator"` (who holds `CanTurnBridge`).
5. Observe: transaction is also accepted with code `0`.
6. Attempt a bridge operation (e.g., `send_to_evm_chain` on a CRC21 contract).
7. Observe: the bridge operation **succeeds** — the bridge was never disabled, confirming the no-op.

The integration test `test_gravity_turn_bridge` in `integration_tests/test_gravity.py` codifies exactly this expected behavior (bridge should reject after `turn_bridge false`), and it would fail against the current implementation because the handler never writes the disabled state. [8](#0-7)

func (k msgServer) UpdatePermissions(goCtx context.Context, msg *types.MsgUpdatePermissions) (*types.MsgUpdatePermissionsResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)
	admin := k.Keeper.GetParams(ctx).CronosAdmin
	// if admin is empty, no sender could be equal to it
	if admin != msg.From {
		return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
	}
	acc, err := sdk.AccAddressFromBech32(msg.Address)
	if err != nil {
		return nil, err
	}
	k.SetPermissions(ctx, acc, msg.Permissions)

	return &types.MsgUpdatePermissionsResponse{}, nil
}
