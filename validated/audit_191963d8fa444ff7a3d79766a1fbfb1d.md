### Title
Unbounded `MessageBuffer` HashMap Allows Malicious Participant to Exhaust Honest Node Memory — (File: `src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` inside `Comms` uses an unbounded `HashMap<MessageHeader, SubMessageQueue>` to store incoming messages keyed by `(channel_tag, waitpoint)`. Because `push_message` performs no validation that the `waitpoint` field corresponds to any step the honest protocol will ever poll, a malicious participant can inject an unlimited number of messages with arbitrary waitpoint values. Each unique `(channel, waitpoint)` pair creates a new `HashMap` entry backed by an `UnboundedSender/UnboundedReceiver` pair, and these entries are never evicted. The codebase contains a developer-written unit test that explicitly confirms this attack path.

---

### Finding Description

`MessageBuffer` is the central incoming-message store for every protocol instance: [1](#0-0) 

Every call to `push_message` parses the first 40 bytes of the raw wire message as a `MessageHeader` (32-byte `ChannelTag` + 8-byte `waitpoint`) and inserts it into the map: [2](#0-1) 

The `or_default()` call at line 238 silently creates a new `SubMessageQueue` — backed by `futures::channel::mpsc::unbounded()` — for every previously-unseen `(channel, waitpoint)` pair: [3](#0-2) 

The `ChannelTag` for the shared channel is deterministic and publicly computable from the domain separator constant alone: [4](#0-3) 

Because `waitpoint` is a raw `u64` parsed directly from the wire with no allowlist check, an attacker can craft messages with the correct 32-byte channel tag and any of 2⁶⁴ distinct waitpoint values. Each such message creates a new `HashMap` entry that is never removed. The developers themselves wrote a unit test that proves this: [5](#0-4) 

The test is named `attacker_can_fill_message_buffer_with_unused_waitpoints` and asserts that 512 injected messages with high-numbered waitpoints produce exactly 512 distinct `HashMap` entries — confirming the unbounded growth.

The public entry point is `Protocol::message()`, which every honest executor calls for every received network message: [6](#0-5) 

---

### Impact Explanation

A malicious participant sends a continuous stream of messages whose first 40 bytes encode the correct shared-channel tag and a unique, never-polled `waitpoint` value. Each message allocates:
- One `HashMap` entry (key + `SubMessageQueue` struct)
- One `UnboundedSender` + `UnboundedReceiver` pair (heap-allocated MPSC channel)
- The message payload itself buffered in the channel

There is no cap, no eviction, and no back-pressure. The honest node's heap grows without bound until the OS kills the process (OOM) or the node operator intervenes. This permanently denies the honest party the ability to complete any in-progress DKG, reshare, refresh, presign, or signing session, matching the allowed impact:

> **Medium: Griefing or resource-exhaustion by a malicious caller or participant causing unbounded CPU, memory, bandwidth, or non-terminating work beyond documented behavior.**

---

### Likelihood Explanation

- The `ChannelTag` for the shared channel is derived from a public constant (`NEAR_CHANNEL_TAGS_DOMAIN` + `"root shared"`), so any participant can compute it without any secret material.
- The `waitpoint` field is a plain `u64` with no protocol-level constraint; the attacker has 2⁶⁴ distinct values to use.
- The attack requires only that the attacker be a registered participant (or be able to deliver raw bytes to the victim's `Protocol::message()` call), which is the normal operating assumption for any threshold-signature session.
- No cryptographic break, no key leakage, and no trusted-party compromise is required.

---

### Recommendation

1. **Allowlist waitpoints**: Before inserting into the `HashMap`, check that the `(channel, waitpoint)` pair corresponds to a waitpoint that has been allocated by the local protocol instance (e.g., track the maximum allocated waitpoint per channel and reject anything beyond it).
2. **Bound the HashMap**: Enforce a maximum number of entries (e.g., `n_participants × max_rounds`) and drop or error on overflow.
3. **Evict completed entries**: Once a waitpoint's `SubMessageQueue` has been fully consumed, remove it from the `HashMap`.
4. **Per-sender rate limiting**: Reject more than one message per `(sender, waitpoint)` pair, consistent with the `ParticipantCounter` pattern already used in `recv_from_others`.

### Proof of Concept

The developers' own test in `src/protocol/internal.rs` (lines 530–554) is a direct proof of concept. To reproduce:

```rust
// Attacker knows the shared channel tag (public constant).
let comms = Comms::new();
let attacker = Participant::from(99_u32);

for i in 0..u64::MAX {   // unbounded in practice
    let header = MessageHeader::new(ChannelTag::root_shared())
        .with_waitpoint(1_000_000 + i);   // waitpoints the honest code never polls
    let mut message = header.to_bytes().to_vec();
    message.extend_from_slice(&[0u8; 8]);  // minimal valid payload
    comms.push_message(attacker, message); // one new HashMap entry per iteration
}
// HashMap grows without bound; process OOMs.
```

Each iteration inserts one new `HashMap<MessageHeader, SubMessageQueue>` entry. The honest protocol's `recv()` calls never poll these waitpoints, so the entries are never consumed or freed. The existing test asserts `messages.len() == 512` after 512 injections, confirming the linear growth. [7](#0-6) [5](#0-4)

struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}

impl MessageBuffer {
    fn new() -> Self {
        Self {
            messages: Arc::new(std::sync::Mutex::new(HashMap::new())),
        }
    }

    /// Push a message into this buffer.
    ///
    /// We also need the header for the message, and the participant who sent it.
    fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
        let mut messages_lock = self.messages.lock().expect("lock should not fail");
        messages_lock.entry(header).or_default().send(from, message);
    }
