### Title
Third-party invoke with nonce=1 bypasses `__validate__` via `skip_stateful_validations`, admitted to mempool without signature verification - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature, designed to allow a user to submit `deploy_account + invoke` in one batch, uses `account_tx_in_pool_or_recent_block` as its guard. That guard returns `true` for **any** transaction in the pool for the target address — not specifically a `deploy_account`. A third party who observes a pending `deploy_account` in the mempool can submit an invoke with nonce=1 for that address with an arbitrary (invalid) signature, bypass `__validate__` entirely at the gateway, and have the transaction admitted to the mempool. This blocks the legitimate account owner's first invoke via `DuplicateNonce` and constitutes free, repeatable DoS.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when an incoming invoke has `tx.nonce() == 1` and `account_nonce == 0`, and `account_tx_in_pool_or_recent_block` returns `true`: [1](#0-0) 

When it fires, `run_validate_entry_point` is called with `skip_validate = true`, which sets `ExecutionFlags { validate: false }`, so `validate_tx` returns `Ok(None)` immediately — the account's `__validate__` entry point is never executed and the signature is never checked: [2](#0-1) [3](#0-2) 

The guard that is supposed to justify skipping validation is: [4](#0-3) 

This checks `self.state.contains_account` (committed/staged) **or** `self.tx_pool.contains_account` — any transaction type for that address, not specifically a `deploy_account`. The inline comment claims this is sufficient: [5](#0-4) 

The claim is wrong. A `deploy_account` for address X is public and its deterministic address is observable. A third party can submit an invoke with nonce=1 for X immediately after the `deploy_account` lands in the pool. The mempool's `validate_tx` (called by `validate_by_mempool` **before** `skip_stateful_validations`) only checks for duplicate tx hash and nonce ordering — it does not verify the signature or that the account exists on-chain: [6](#0-5) 

So the attacker's invoke clears all pre-checks and is admitted without `__validate__`.

The `MempoolStaticConfig` default and production deployment config both have `enable_fee_escalation: true` with `fee_escalation_percentage: 10`: [7](#0-6) [8](#0-7) 

With fee escalation enabled, the legitimate owner can replace the attacker's invoke by paying ≥10% more. But the attacker can immediately re-submit with a higher fee, forcing the owner into an escalating fee war. With fee escalation disabled the owner is fully blocked until the attacker's invoke is rejected by the batcher and removed via `commit_block`'s `rejected_tx_hashes`: [9](#0-8) [10](#0-9) 

During block execution the batcher uses default `ExecutionFlags { validate: true }`, so `__validate__` is called and the attacker's invoke is rejected — but only after it has already occupied the nonce=1 slot in the mempool and blocked the owner's transaction.

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction and rejects a valid one before sequencing.**

- The gateway admits an invoke with an unverified (invalid) signature to the mempool.
- The legitimate account owner's first post-deployment invoke is rejected with `DuplicateNonce` or forced into a fee-escalation war.
- The attacker pays zero fees (the invalid invoke is rejected by the batcher, never included in a block, so no fee is charged).
- The attack is repeatable: after each rejection cycle the attacker can re-submit, creating persistent DoS against any newly deploying account.

### Likelihood Explanation

- Requires no privilege: any observer of the public mempool can execute this.
- The target address is deterministic from the `deploy_account` fields (class hash, salt, constructor calldata) and is visible in the mempool before the block is built.
- Cost to attacker: only the gas to submit the invoke transaction to the gateway (which is rejected before execution, so no L2 fee is charged).

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` guard with a check that specifically verifies a `deploy_account` transaction exists for the address. Options:

1. Add a `deploy_account_in_pool(address)` query to the mempool that inspects transaction type, and use it in `skip_stateful_validations`.
2. Tag `deploy_account` transactions in the mempool state so `account_tx_in_pool_or_recent_block` can distinguish them from invoke transactions.
3. Alternatively, require the caller to supply the `deploy_account` tx hash (as the Python-based `PyValidator::should_run_stateful_validations` does via `deploy_account_tx_hash: Option<TransactionHash>`) and verify it exists in the mempool before skipping validation. [11](#0-10) 

### Proof of Concept

```
1. Owner submits deploy_account for address X (nonce=0).
   → Mempool admits it; tx_pool.contains_account(X) = true.

2. Attacker submits Invoke { sender=X, nonce=1, signature=[0xdead...] }.
   Gateway path:
     a. validate_state_preconditions: account_nonce=0, tx_nonce=1 → passes nonce range check.
     b. validate_by_mempool: no duplicate hash, nonce 1 ≥ 0 → Ok(()).
     c. skip_stateful_validations:
          tx.nonce()==1 && account_nonce==0 → check account_tx_in_pool_or_recent_block(X)
          → true (deploy_account is in pool)
          → returns true (skip).
     d. run_validate_entry_point(skip_validate=true):
          ExecutionFlags { validate: false } → __validate__ never called.
   → Attacker's invoke admitted to mempool.

3. Owner submits Invoke { sender=X, nonce=1, signature=[valid] }.
   Gateway path:
     b. validate_by_mempool → validate_fee_escalation:
          attacker's tx already at (X, nonce=1) → DuplicateNonce (or forced 10%+ fee bump).
   → Owner's invoke rejected or forced to pay more.

4. Batcher executes block:
   deploy_account(X) succeeds → account deployed.
   attacker's invoke(X, nonce=1) → __validate__ called → fails (invalid sig) → rejected.
   commit_block sends rejected_tx_hashes → attacker's invoke removed from mempool.

5. Attacker immediately re-submits step 2 → cycle repeats indefinitely at zero cost.
```

    fn remove_rejected_txs(
        &mut self,
        rejected_tx_hashes: IndexSet<TransactionHash>,
        rewound_tx_hashes: &IndexSet<TransactionHash>,
    ) -> AddressToNonce {
        if !rejected_tx_hashes.is_empty() {
            debug!("Removed rejected transactions from mempool: {:?}", rejected_tx_hashes);
        }
        let mut rejected_txs_counter = 0;
        let mut account_nonce_updates = AddressToNonce::new();

        for tx_hash in rejected_tx_hashes {
            // In FIFO mode, if a rejected transaction was rewound, skip removal (keep in pool and
            // queue). Otherwise, remove it from both pool and queue.
            if rewound_tx_hashes.contains(&tx_hash) {
                continue;
            }

            if let Ok(tx) = self.tx_pool.remove(tx_hash) {
                self.tx_queue.remove_by_address(tx.contract_address());
                rejected_txs_counter += 1;
                self.decrement_stuck_txs_if_gap_account(tx.contract_address(), 1);
                account_nonce_updates
                    .entry(tx.contract_address())
                    .and_modify(|nonce| *nonce = (*nonce).min(tx.nonce()))
                    .or_insert(tx.nonce());
            } else {
                continue; // Transaction hash unknown to mempool, from a different node.
            }

            // TODO(clean_accounts): remove address with no transactions left after a block cycle /
            // TTL.
        }
        metric_count_rejected_txs(rejected_txs_counter);
        account_nonce_updates
    }

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
