### Title
Offer Cancel Button Hidden for `PENDING_CANCEL` State, Preventing Re-cancellation if Cancellation Transaction Fails - (File: packages/gui/src/components/offers/OfferManager.tsx, packages/gui/src/components/offers2/CancelOfferList.tsx)

### Summary
Both `OfferManager.tsx` and `CancelOfferList.tsx` compute `canCancel` by checking only for `PENDING_ACCEPT` or `PENDING_CONFIRM` states, excluding `PENDING_CANCEL`. The backend `cancel_offer` RPC has no such restriction. If a cancellation transaction is submitted (moving the offer to `PENDING_CANCEL`) but then fails or is dropped from the mempool, the offer remains live on-chain but the GUI permanently hides the Cancel button. The user cannot re-cancel through the GUI, leaving the offer open to acceptance by a taker.

### Finding Description
The `OfferState` enum defines six states including `PENDING_CANCEL`, which represents an offer for which a cancellation transaction has been submitted but not yet confirmed on-chain.

In `OfferManager.tsx`:
```typescript
const canCancel = status === OfferState.PENDING_ACCEPT || status === OfferState.PENDING_CONFIRM;
``` [1](#0-0) 

In `CancelOfferList.tsx` (used in the conflict-resolution dialog when creating new offers):
```typescript
const canCancel = status === OfferState.PENDING_ACCEPT || status === OfferState.PENDING_CONFIRM;
``` [2](#0-1) 

Both checks exclude `PENDING_CANCEL`. The `OfferState` enum explicitly includes this as a distinct, valid state: [3](#0-2) 

The backend `cancel_offer` RPC (exposed via `useCancelOfferMutation`) does not enforce this GUI-side restriction and will accept a cancellation request for an offer regardless of its current pending state. The GUI wrapper is more restrictive than the underlying RPC layer, mirroring the exact pattern in the external report.

### Impact Explanation
When a user cancels an offer, the offer transitions to `PENDING_CANCEL`. If the cancellation transaction is dropped from the mempool (e.g., due to low fee, network congestion, or a full-node restart), the offer remains live and spendable on-chain, but the GUI continues to show `PENDING_CANCEL` and hides the Cancel button. The user, believing the offer is being cancelled, stops monitoring it. A taker can still accept the offer on-chain, causing an unwanted transfer of XCH, CAT, or NFT assets without the user's current consent. The user has no GUI path to re-cancel.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of RPC, event, offer, NFT metadata, DataLayer, notification, or WalletConnect state that causes a user to … send … the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation
Cancellation transactions can be dropped from the mempool under normal network conditions (fee too low, mempool pressure, node restart). The `PENDING_CANCEL` state is a standard, reachable state for any user who has initiated a cancellation. Once in this state, the user has no GUI recourse. The attacker path requires only that a taker monitors the mempool for offers in `PENDING_CANCEL` and accepts them before the cancellation confirms — a known MEV-style pattern.

### Recommendation
Include `PENDING_CANCEL` in the `canCancel` condition in both files:

```typescript
// OfferManager.tsx and CancelOfferList.tsx
const canCancel =
  status === OfferState.PENDING_ACCEPT ||
  status === OfferState.PENDING_CONFIRM ||
  status === OfferState.PENDING_CANCEL;
```

For `canCancelWithTransaction` in `OfferManager.tsx`, the existing logic (`canCancel && status === OfferState.PENDING_ACCEPT`) already correctly restricts the "cancel on blockchain" option to `PENDING_ACCEPT` only, so no change is needed there. [1](#0-0) 

### Proof of Concept
1. User creates an offer (state: `PENDING_ACCEPT`).
2. User clicks Cancel → `cancelOffer` RPC is called → offer transitions to `PENDING_CANCEL`.
3. The cancellation transaction is dropped from the mempool (low fee / congestion).
4. Offer remains live on-chain; GUI shows `PENDING_CANCEL` with no Cancel button (`canCancel` is `false`).
5. Taker, monitoring the chain, calls `take_offer` on the still-valid offer file.
6. Offer is accepted on-chain; user's XCH/CAT/NFT is transferred without their current consent.
7. User has no GUI path to re-cancel; the Cancel button is permanently hidden for `PENDING_CANCEL` state. [4](#0-3) [3](#0-2)
