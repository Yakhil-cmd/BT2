### Title
Missing Verification of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates `(big_y, big_c)` shares from all participants without any proof-of-correctness verification. A malicious participant can send values computed under a different `app_id`, a different `app_pk`, or entirely arbitrary group elements. The coordinator blindly sums all contributions, producing a `CKDOutput` that does not correspond to the correct confidential key for the intended application. Honest parties accept this corrupted output with no indication of failure.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects one `CKDOutput` per participant and accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is expected to compute their share as:

```
big_y  = y_i · G₁
big_c  = private_share_i · H(pk ‖ app_id) + app_pk · y_i
``` [2](#0-1) 

The coordinator has no mechanism to verify that a received `(big_y, big_c)` pair was actually computed using the agreed-upon `app_id`, the correct `app_pk`, or the participant's genuine private share. There is no zero-knowledge proof, commitment, or consistency check attached to the contribution. The `app_id` and `app_pk` are purely local inputs to each party; nothing in the protocol enforces that all parties used the same values.

This is structurally identical to the reported smart-contract bug: just as `Inbox._fulfill()` accepted a fill on any chain because `route.destination == block.chainid` was never checked, `do_ckd_coordinator` accepts any contribution because no check of the form "was this share computed for the correct `app_id` and `app_pk`?" is ever performed.

A malicious participant `P_m` can:

1. Compute `(big_y_m, big_c_m)` using a different `app_id'` (or arbitrary scalars).
2. Send the result to the coordinator via the single waitpoint.
3. The coordinator sums it into the aggregate without rejection.

The resulting `CKDOutput` satisfies:

```
unmask(app_sk) = (msk − λ_m · sk_m) · H(pk ‖ app_id)
               + λ_m · sk_m · H(pk ‖ app_id')
```

which is not the correct confidential key `msk · H(pk ‖ app_id)`.

---

### Impact Explanation

**High.** The coordinator and all honest parties accept a `CKDOutput` that does not encode the correct confidential key for the intended `app_id`. Any TEE application that subsequently calls `unmask` will obtain a wrong secret, making data encrypted or authenticated under the expected key permanently inaccessible or invalid. This matches the allowed impact: *Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

**Medium.** Any single participant in the CKD session can trigger this unilaterally. No special privilege, leaked key, or external assumption is required — only the ability to deviate from the protocol when computing and sending `(big_y, big_c)`. The attack is deterministic and silent (no error is returned to honest parties).

---

### Recommendation

Each participant should attach a zero-knowledge proof of correct computation alongside their `(big_y, big_c)` contribution, proving knowledge of `y_i` and that `big_c` was formed as `private_share_i · H(pk ‖ app_id) + app_pk · y_i` with respect to the public verification share. The coordinator must verify all proofs before aggregating. Alternatively, a commit-then-reveal round can bind each participant's contribution to the agreed `app_id` and `app_pk` before the coordinator sums them.

---

### Proof of Concept

1. Honest participants `P_1, P_2` and malicious participant `P_3` run `ckd(participants, coordinator, …, app_id_A, app_pk)`.
2. `P_3` locally calls `compute_signature_share` with `app_id_B ≠ app_id_A` (or sends `(G₁, G₁)` directly).
3. `P_3` sends the resulting `(norm_big_y_3, norm_big_c_3)` to the coordinator via `chan.send_private`.
4. The coordinator executes the loop at lines 50–55 of `protocol.rs`, adding `P_3`'s malformed values to `norm_big_y` and `norm_big_c` without any check.
5. `CKDOutput::new(norm_big_y, norm_big_c)` is returned as `Some(ckd_output)`.
6. The caller invokes `ckd_output.unmask(app_sk)` and obtains a value that differs from `msk · H(pk ‖ app_id_A)`, silently producing the wrong confidential key.
