### Title
`receive_erc20_tokens()` Mints ERC-20 Mirror Tokens Without Verifying Actual NEP-141 Transfer Amount, Combined With Permissionless `deploy_erc20_token` - (File: `engine/src/engine.rs`, `engine/src/contract_methods/connector.rs`)

### Summary

The `ft_on_transfer` entry point in Aurora Engine routes to `receive_erc20_tokens()` when the caller is any registered NEP-141 account. This function mints ERC-20 mirror tokens based entirely on the caller-supplied `args.amount` with no verification that the caller actually transferred that amount of NEP-141 tokens. Because `deploy_erc20_token` (Legacy variant) has no access control, any unprivileged NEAR account can register a malicious NEP-141 contract and then call `ft_on_transfer` directly with an arbitrary amount, minting unlimited ERC-20 tokens on Aurora.

### Finding Description

**Root cause 1 — Permissionless NEP-141 registration:**

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` has no `require_owner_only` guard for the `Legacy` variant. The only check is `require_running`: [1](#0-0) 

Any NEAR account can call this to register an arbitrary NEP-141 account ID and get a corresponding ERC-20 mirror deployed on Aurora.

**Root cause 2 — No supply/transfer verification in `receive_erc20_tokens`:**

When `ft_on_transfer` is called and `predecessor_account_id != connector_account_id`, it unconditionally calls `receive_erc20_tokens` with the caller-supplied `args.amount`: [2](#0-1) 

Inside `receive_erc20_tokens`, the amount is taken directly from `args.amount` and passed to `setup_receive_erc20_tokens_input`, which encodes a call to `mint()` on the `EvmErc20` contract. There is no check that the caller actually transferred that many NEP-141 tokens, no cap against the NEP-141 total supply, and no cross-contract verification: [3](#0-2) 

The `EvmErc20.mint()` function itself also has no supply cap — it is `onlyAdmin` (the Aurora engine contract) and calls `_mint` unconditionally: [4](#0-3) 

The same applies to `EvmErc20V2`: [5](#0-4) 

The `setup_receive_erc20_tokens_input` function confirms the mint call is constructed with the raw caller-supplied amount: [6](#0-5) 

### Impact Explanation

**Critical — Direct theft of user funds.**

An attacker who mints unbacked ERC-20 mirror tokens can immediately trade them on any DEX deployed within Aurora's EVM (e.g., Uniswap forks) in exchange for real ETH or other legitimate ERC-20 tokens held by liquidity providers. The liquidity providers lose their real assets in exchange for worthless, unbacked mirror tokens. The attacker exits with real value.

### Likelihood Explanation

**Medium.** The attack requires two permissionless on-chain steps: (1) deploying a NEAR contract that calls `ft_on_transfer` with an inflated amount, and (2) registering it via `deploy_erc20_token`. Both steps require no special privileges. The attacker needs only NEAR gas fees. The attack is repeatable and fully automated.

### Recommendation

1. **Add access control to `deploy_erc20_token`**: Restrict the Legacy variant to the contract owner or a whitelist, preventing arbitrary NEP-141 registrations.
2. **Verify transfer amount in `receive_erc20_tokens`**: Cross-check `args.amount` against the actual NEP-141 balance change (e.g., via a callback that reads the NEP-141 balance before and after), or enforce that `ft_on_transfer` can only be triggered through the standard `ft_transfer_call` flow where the NEP-141 contract has already deducted the tokens.
3. **Add a supply cap to `EvmErc20.mint()`**: Track and enforce a maximum mintable supply equal to the total NEP-141 tokens locked in the Aurora engine account.

### Proof of Concept

1. Attacker deploys `evil.near` — a NEAR contract whose only function is to call `ft_on_transfer` on the Aurora engine with `args.amount = u128::MAX` and `args.msg = <attacker_evm_address>`.
2. Attacker calls `deploy_erc20_token` on Aurora with `DeployErc20TokenArgs::Legacy("evil.near")`. No access control blocks this. Aurora deploys an `EvmErc20` mirror and registers the mapping.
3. Attacker triggers `evil.near` to call `ft_on_transfer` on Aurora. `predecessor_account_id = "evil.near"` ≠ `connector_account_id`, so `receive_erc20_tokens` is invoked.
4. `get_erc20_from_nep141` succeeds (mapping exists from step 2). `setup_receive_erc20_tokens_input` encodes `mint(<attacker_evm_address>, u128::MAX)`. The EVM executes `EvmErc20.mint()` with no supply cap check.
5. Attacker now holds `u128::MAX` ERC-20 mirror tokens. They swap them on a DEX within Aurora for real ETH held by liquidity providers, draining the pool.

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
```

```rust
    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
```

```rust
#[must_use]
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);

    [selector, tail.as_slice()].concat()
}
```
