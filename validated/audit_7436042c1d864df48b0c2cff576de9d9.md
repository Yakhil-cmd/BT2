### Title
Incorrect `nftSellerNetAmount` in `calculateNFTRoyalties` Causes Inflated "Net Proceeds" Display to NFT Seller on Offer Acceptance - (File: packages/gui/src/components/offers/utils.ts)

### Summary

`calculateNFTRoyalties` in `utils.ts` sets `nftSellerNetAmount` to the raw offer `amount` instead of `amount - royaltyAmount`. For `TokenForNFT` offers (a buyer offering XCH/CAT for an NFT), the royalty is paid *from* the offered purchase price, so the seller's actual net proceeds are `amount - royaltyAmount`. The GUI displays the inflated full `amount` as "Net Proceeds" to the NFT seller on the offer acceptance screen, causing them to approve an offer under the false belief they will receive more than they actually will.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The correct formula is commented out. The live code unconditionally assigns `nftSellerNetAmount = amount` regardless of `exchangeType`. [1](#0-0) 

For `NFTOfferExchangeType.NFTForToken` (seller sets the price, buyer pays price + royalty on top), `nftSellerNetAmount = amount` is correct — the seller receives the full amount they requested.

For `NFTOfferExchangeType.TokenForNFT` (buyer offers a fixed token amount, royalty is deducted *from* that amount), the seller actually receives `amount - royaltyAmount`. The tooltip in the viewer explicitly states this:

> "those creator fees will be paid from the offered purchase price" [2](#0-1) 

But `nftSellerNetAmount` still equals `amount`, so the "Net Proceeds" figure shown to the seller is inflated by the full royalty amount.

`nftSellerNetAmount` flows into two user-facing surfaces in `NFTOfferViewer.tsx`:

1. **"Net Proceeds" label** — displayed prominently to the NFT seller (taker) when they view an imported `TokenForNFT` offer and decide whether to accept it: [3](#0-2) 

2. **`overrideNFTSellerAmount`** — computed from `nftSellerNetAmount` and passed into `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow` to override the displayed token amount in the "In exchange for" row: [4](#0-3) [5](#0-4) 

Additionally, in `NFTOfferEditor.tsx`, the negative-amount guard that is supposed to block offer creation when royalties exceed the sale price never fires because `nftSellerNetAmount` is always positive: [6](#0-5) 

### Impact Explanation

An NFT seller imports a `TokenForNFT` offer (e.g., a buyer offers 1 XCH for their NFT). The NFT has a 10% creator royalty. The GUI displays:

- **Net Proceeds: 1 XCH** ← shown to seller (incorrect)
- Actual proceeds: **0.9 XCH** (royalty of 0.1 XCH is deducted from the offered amount)

The seller approves the offer believing they will receive 1 XCH but actually receives 0.9 XCH. The discrepancy equals the full royalty amount and scales with both the offer size and royalty percentage. This is a direct, concrete financial loss caused by the GUI displaying the wrong amount at the offer-acceptance decision point.

This matches the **High** impact criterion: *"Corruption, spoofing, or unsafe trust of … offer … state that causes a user to … display the wrong … amount … or status"* and *"causes a user to approve … the wrong … amount."*

### Likelihood Explanation

Any `TokenForNFT` offer involving an NFT with a non-zero royalty percentage triggers this bug. NFT royalties are common (many NFTs carry 3–10% royalties). The seller's only signal of their net proceeds is the "Net Proceeds" field, which is always wrong in this scenario. No special attacker action is required — the offer maker simply creates a standard offer.

### Recommendation

In `calculateNFTRoyalties`, restore the correct branch for `TokenForNFT`:

```typescript
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? parseFloat((amount - parseFloat(royaltyAmountString)).toFixed(12))
    : amount;
``` [7](#0-6) 

### Proof of Concept

1. Create an NFT with a 10% royalty.
2. A buyer creates a `TokenForNFT` offer: 1 XCH for the NFT.
3. The NFT seller imports the offer file into the GUI.
4. On the offer acceptance screen, observe **"Net Proceeds: 1 XCH"**.
5. Accept the offer; the seller's wallet receives **0.9 XCH** (0.1 XCH goes to the royalty address).
6. The displayed "Net Proceeds" was 11% higher than the actual proceeds.

The root cause is `nftSellerNetAmount: number = amount` at [8](#0-7) 
with the correct formula commented out directly below it. [9](#0-8)
