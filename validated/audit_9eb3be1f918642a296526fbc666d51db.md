### Title
Unchecked Zero Address for `refund_recipient` in L1 Transaction Processing Causes Permanent Base Token Loss - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary
In `process_l1_transaction`, the `refund_recipient` address decoded from `transaction.reserved[1]` is never validated against the zero address before `mint_base_token` is called. Any L1→L2 priority transaction that leaves `refund_recipient` unset (defaulting to `address(0)`) or explicitly sets it to zero will have its gas refund permanently transferred to the zero address, resulting in an irrecoverable loss of base tokens from the treasury.

### Finding Description
In `process_l1_transaction`, after computing the refund amount, the bootloader reads the refund recipient directly from the transaction's reserved field and immediately calls `mint_base_token` with it:

```rust
// Line 337

mint_base_token::<S, Config>(
    system,
    system_functions,
    memories.reborrow(),
    &to_refund_recipient,
    &refund_recipient,   // ← no zero-address guard
    l1_chain_id,
    &mut inf_resources,
    tracer,
    validator,
)?;
``` [1](#0-0) 

`mint_base_token` calls `transfer_from_treasury`, which unconditionally credits the `to` address:

```rust
pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,          // ← accepted as-is, no zero check
    ...
``` [2](#0-1) 

The `refund_recipient` field in `ZKsyncL1Tx` is defined as a plain `Address` with no protocol-level constraint: [3](#0-2) 

The `L1TxBuilder` defaults `refund_recipient` to `Default::default()` (i.e., `address(0)`) when not explicitly set: [4](#0-3) 

The existing regression test `test_treasury_based_token_distribution_regression` explicitly uses `address(0)` as the refund recipient and asserts that the refund amount is successfully credited there — confirming the protocol accepts and processes this without any revert: [5](#0-4) [6](#0-5) 

### Impact Explanation
When `refund_recipient` is `address(0)`, the gas refund amount (computed as `(gas_limit - gas_used) * gas_price`) is transferred from the treasury to the zero address via `transfer_from_treasury`. These tokens are permanently inaccessible. Since the treasury is the source of all L1→L2 base token distributions, this constitutes a direct, irreversible loss of base tokens for every L1 transaction that leaves `refund_recipient` unset or explicitly zeroed. The loss scales with `gas_limit - gas_used`, which is non-zero for virtually every real transaction.

### Likelihood Explanation
The `L1TxBuilder` default for `refund_recipient` is `address(0)`. Any user or integration that submits an L1→L2 priority transaction without explicitly setting a refund recipient will silently burn their refund. This is a realistic, high-frequency scenario — not a contrived edge case. No privileged access is required; any unprivileged L1 transaction sender triggers this path.

### Recommendation
Add a zero-address guard before calling `mint_base_token` for the refund recipient. If `refund_recipient` is zero, fall back to the transaction sender (`from`) as the refund destination, consistent with how the value-mint step uses `from`:

```rust
let refund_recipient = {
    let r = u256_to_b160_checked(transaction.reserved[1].read());
    if r == B160::ZERO { transaction.from.read() } else { r }
};
```

Alternatively, reject L1 transactions with a zero `refund_recipient` at the L1 contract level, and add a defensive assertion in the bootloader.

### Proof of Concept
1. Submit an L1→L2 priority transaction with `refund_recipient = address(0)`, `gas_limit = 100_000`, `gas_price = 1_000`, and any calldata that uses fewer than 100,000 gas units.
2. `process_l1_transaction` computes `to_refund_recipient = (100_000 - gas_used) * 1_000 > 0`.
3. `refund_recipient = u256_to_b160_checked(transaction.reserved[1].read())` → `B160::ZERO`.
4. `mint_base_token(..., &to_refund_recipient, &B160::ZERO, ...)` → `transfer_from_treasury(..., &B160::ZERO, ...)` credits the zero address.
5. The refund tokens are permanently burned. The existing test `test_treasury_based_token_distribution_regression` (line 1843) demonstrates this exact scenario passes without error. [7](#0-6) [8](#0-7)

pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
{
    system_log!(
        system,
        "Transferring {nominal_token_value:?} tokens from treasury to {to:?}\n"
    );

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
            true, // true = subtract from balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Treasury transfer failed: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::TreasuryTransferFailed)
                }
                _ => wrap_error!(e),
            }
        })?;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            to,
            nominal_token_value,
            false, // false = add to balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Error while minting: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::MintingBalanceOverflow)
                }
                _ => wrap_error!(e),
            }
        })?;
