### Title
Staker Can Repeatedly Oscillate Commission Within Commitment Window to Drain Delegator Yield - (File: src/staking/staking.cairo)

### Summary
When a staker holds an active `CommissionCommitment`, the `set_commission` / `update_commission` path imposes **no directional restriction and no rate-limit** on commission changes. A staker can freely oscillate commission between 0 and `max_commission` an unlimited number of times within the commitment window. By raising commission to `max_commission` immediately before each attestation and lowering it again afterward, the staker redirects all pool rewards to themselves, stealing delegators' unclaimed yield.

### Finding Description

Without a commitment, `update_commission` enforces a strict one-way ratchet:

```rust
} else {
    assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
}
``` [1](#0-0) 

With an active commitment the guard is replaced by only two checks:

```rust
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
}
``` [2](#0-1) 

The first check allows **increases** up to `max_commission`; the second only prevents setting the same value twice in a row. There is no cooldown, no per-epoch change limit, and no minimum time between changes. The protocol's own comment acknowledges this gap:

```
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
``` [3](#0-2) 

Commission is read from storage at the moment `calculate_staker_pools_rewards` executes during attestation:

```rust
let commission = staker_pool_info.commission();
...
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [4](#0-3) 

So whatever commission value is stored at attestation time is the one applied to that epoch's rewards.

### Impact Explanation

A staker can:
1. Set `CommissionCommitment { max_commission: 10000, expiration_epoch: E }`.
2. Call `set_commission(0)` — commission drops to 0%, attracting delegators.
3. Delegators delegate, expecting 0% commission.
4. In the same block as (or just before) `attest()`, call `set_commission(10000)`.
5. Attestation fires: 100% of pool rewards flow to the staker as commission; delegators receive 0.
6. Call `set_commission(0)` again (allowed: `0 < 10000`, satisfies `commission != old_commission`).
7. Repeat every epoch for the entire commitment window (up to one year).

This is a direct, repeatable theft of delegators' unclaimed yield. The staker profits by the full pool reward amount each epoch; delegators receive nothing despite having their capital locked.

**Impact category: High — Theft of unclaimed yield.**

### Likelihood Explanation

- Any staker with a pool can execute this; no special privilege beyond owning the staker key is required.
- The commitment window can be up to one year (`expiration_epoch - current_epoch <= epochs_in_year()`), giving a long exploitation window.
- On Starknet, the staker controls both the staker address (for `set_commission`) and the operational address (for `attest`), so both transactions can be submitted in the same block with no race condition.
- The only on-chain signal is a `CommissionChanged` event, which delegators would need to monitor in real time to react.

### Recommendation

1. **Enforce a minimum time-lock between commission increases.** For example, require that at least one full epoch passes between any two `set_commission` calls that raise the value.
2. **Limit the number of commission increases per commitment window** (e.g., at most once per commitment).
3. **Alternatively, disallow commission increases entirely** even within a commitment window; the commitment's purpose is to cap the maximum, not to enable free oscillation. Increases could be permitted only when setting a new commitment after the old one expires.

### Proof of Concept

```
Epoch N:
  staker.set_commission_commitment(max_commission=10000, expiration_epoch=N+100)
  staker.set_commission(0)          // attract delegators

Epoch N+1:
  delegator.enter_delegation_pool(amount=X)

Epoch N+2 (attestation block):
  staker.set_commission(10000)      // raise to 100% — allowed by commitment
  staker.attest()                   // all pool rewards → staker commission
  staker.set_commission(0)          // lower back — allowed: 0 < 10000

Epoch N+3:
  delegator.claim_rewards()         // receives 0 STRK

Repeat from "Epoch N+2" for every epoch until expiration_epoch.
```

The `update_commission` function permits this because the only guard during an active commitment is `commission <= max_commission && commission != old_commission`. [5](#0-4)
