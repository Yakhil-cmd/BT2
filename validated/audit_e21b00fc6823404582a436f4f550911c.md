Gateway Stateful Validator Uses Stale Previous-Block L2 Gas Price for Resource Bounds Admission, Causing False Rejections and False Admissions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validator reads the **previous (latest committed) block's** L2 gas price to gate admission of incoming transactions. The batcher, however, builds the next block with the **actual current** L2 gas price determined by the L1 oracle at block-building time. Because these two prices can diverge, the gateway makes systematically wrong admission decisions: it rejects transactions that would succeed in the next block (false rejection) and admits transactions that will fail in the next block (false admission). The code even carries an explicit developer acknowledgement of the defect.

---

### Finding Description

In `validate_resource_bounds`, the gateway reads the gas price from the latest committed block and uses it as the reference for the admission threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold check rejects any `AllResources` transaction whose `l2_gas.max_price_per_unit` is below `previous_block_l2_gas_price × min_gas_price_percentage / 100`: [2](#0-1) 

Immediately after, `run_validate_entry_point` builds the `BlockContext` for the blockifier's `perform_pre_validation_stage` (which calls `check_fee_bounds` and `verify_can_pay_committed_bounds`) using the **same stale block info**, only bumping the block number:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [3](#0-2) 

The gas prices inside `block_info` are **never updated**; only the block number is incremented. The blockifier's `check_fee_bounds` therefore also runs against the previous block's prices: [4](#0-3) 

The batcher, by contrast, constructs its `BlockContext` from the L1 gas oracle at block-building time, so the actual execution price for the next block can differ arbitrarily from the previous block's price.

The `GatewayFixedBlockStateReader` implementation caches the block info in a `OnceCell`, so the stale value is frozen for the entire lifetime of the validator instance: [5](#0-4) 

---

### Impact Explanation

**False rejection (valid transaction denied):** When the next block's L2 gas price falls below the previous block's price, a user who sets `max_price_per_unit` equal to the actual next-block price passes the batcher's `check_fee_bounds` but fails the gateway's threshold check (`max_price < prev_price × threshold%`). The gateway returns `GAS_PRICE_TOO_LOW` and the transaction never reaches the mempool, even though it would have been executed successfully.

**False admission (invalid transaction accepted):** When the next block's L2 gas price rises above the previous block's price, a user who sets `max_price_per_unit` equal to the previous block's price passes both the gateway's threshold check and the gateway's blockifier pre-validation (both use stale prices), but the transaction will fail `check_fee_bounds` inside the batcher with `MaxGasPriceTooLow`. The transaction is admitted to the mempool and propagated to peers, consuming mempool capacity and network bandwidth, yet it cannot be sequenced.

Both outcomes match the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

Starknet's L2 gas price follows an EIP-1559-style update rule and changes every block. Any block in which the oracle-derived price differs from the previous committed block's price — which is the common case during periods of L1 fee volatility — triggers one of the two failure modes. The condition is unprivileged: any user submitting a V3 (`AllResources`) transaction with a `max_price_per_unit` calibrated to the actual next-block price rather than the previous block's price will be affected. No special access or timing attack is required.

### Recommendation

Replace the stale `get_block_info()` call with the **next block's** projected L2 gas price, as the TODO comment already acknowledges:

```rust

``` [6](#0-5) 

Concretely:
1. Expose the pending/next block's gas price through the `GatewayFixedBlockStateReader` trait (or a separate interface).
2. Use that price in both `validate_resource_bounds` and the `BlockContext` constructed inside `run_validate_entry_point`, so that the gateway's admission decision is consistent with the price the batcher will actually apply.
3. Ensure the `OnceCell` cache in `GatewayFixedBlockSyncStateClient` is invalidated or bypassed when the next-block price is needed.

### Proof of Concept

**False rejection scenario (valid transaction denied):**

1. Previous committed block has `strk l2_gas_price = 100`.
2. L1 oracle determines next block's `l2_gas_price = 80` (price fell 20%).
3. User submits a V3 invoke with `l2_gas.max_price_per_unit = 80`.
4. Gateway `validate_resource_bounds`: threshold = `100 × 100% = 100`; `80 < 100` → **rejected** with `GAS_PRICE_TOO_LOW`.
5. Batcher would have computed threshold = `80`; `80 >= 80` → **would have passed**.
6. A valid transaction is permanently denied at the gateway.

**False admission scenario (invalid transaction accepted):**

1. Previous committed block has `strk l2_gas_price = 100`.
2. L1 oracle determines next block's `l2_gas_price = 130` (price rose 30%).
3. User submits a V3 invoke with `l2_gas.max_price_per_unit = 100`.
4. Gateway `validate_resource_bounds`: threshold = `100 × 100% = 100`; `100 >= 100` → **passes**.
5. Gateway blockifier pre-validation (`check_fee_bounds`) uses stale price `100`; `100 >= 100` → **passes**.
6. Transaction is admitted to the mempool and propagated.
7. Batcher `check_fee_bounds` uses actual price `130`; `100 < 130` → **`MaxGasPriceTooLow`**, transaction cannot be sequenced.
8. An invalid transaction occupies mempool capacity and is never executable.

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
