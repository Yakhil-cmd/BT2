### Title
Double-Counting of Notional Fees in `collectFees` Drains LP Principal When Both Spread and Notional Fees Are Active - (File: metric-core/contracts/MetricOmmPool.sol)

---

### Summary

In `MetricOmmPool._executeSwap`, `binTotals` is updated using the pre-notional-fee `amount{0,1}DeltaScaled` value. The notional fee is then deducted from the trader's output *after* `binTotals` is written. This causes the notional fee amount to silently accumulate in the pool's token balance as part of the "surplus" (`balance × scaleMultiplier − binTotals`). When `collectFees` is later called, it distributes *both* the surplus (which already contains the notional fee) *and* the `notionalFeeToken{0,1}Scaled` accumulators, paying out the same notional fee twice and draining LP principal.

---

### Finding Description

**Step 1 — `binTotals` is written before the notional fee adjustment.**

In `_executeSwap`, the `binTotals` update block runs first (lines 732–748), using the raw `amount1DeltaScaled` (or `amount0DeltaScaled`) that was produced by the inner swap math: [1](#0-0) 

Immediately after, the notional fee block (lines 750–790) *reduces* the output delta that the trader actually receives and increments the accumulator: [2](#0-1) 

Because `binTotals.scaledToken1` was already decremented by the full (pre-notional) output, but the trader only receives `output − notionalFee`, the pool retains `notionalFeeScaled` extra tokens that are **not** reflected in `binTotals`. Those tokens therefore appear in the surplus:

```
surplus = balance × scaleMultiplier − binTotals.scaledToken1
        = spreadProtocolFee + notionalFee          ← notionalFee counted here
```

**Step 2 — `collectFees` pays out the surplus AND the notional accumulator.**

`collectFees` distributes:
- The full surplus (spread fees + notional fees already embedded)
- The `notionalFeeToken{0,1}Scaled` accumulators (the same notional fees again)

Total attempted payout = `spreadFee + 2 × notionalFee`, but only `spreadFee + notionalFee` is available as surplus. The excess `notionalFee` is taken from LP bin balances.

**Step 3 — The codebase's own test confirms the overallocation.**

The test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive` explicitly asserts: [3](#0-2) 

```solidity
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
```

This is not a test of a guard — it is a test that documents the broken invariant as observed behavior.

---

### Impact Explanation

When `collectFees` is called with both `spreadFeeE6 > 0` and `notionalFeeE8 > 0`, the function attempts to transfer `notionalFee` tokens a second time. Because the pool holds LP deposits in its balance, the transfer succeeds and the excess is taken from LP-owned bin balances. LPs who subsequently call `removeLiquidity` receive less than their entitled share, or the call reverts if the pool is sufficiently drained. This is a direct loss of LP principal proportional to the accumulated notional fees since the last `collectFees` call.

---

### Likelihood Explanation

The condition requires both fee types to be simultaneously active on a pool. The factory's `setPoolFees` function allows the pool admin to set both `spreadFeeE6` and `notionalFeeE8` independently. Any pool configured with both non-zero values is permanently vulnerable. The factory's `collectFees` path is a routine operational action (not a one-time setup), so the loss compounds with every collection cycle.

---

### Recommendation

`binTotals` must be updated *after* the notional fee adjustment, so that the notional fee amount is reflected in `binTotals` and excluded from the surplus. Concretely, the notional fee block (lines 750–790) should run **before** the `binTotals` write block (lines 732–748), or `binTotals` should be corrected by adding `notionalFeeScaled` back after the adjustment. Alternatively, `collectFees` should compute the notional fee payout exclusively from the `notionalFeeToken{0,1}Scaled` accumulators and compute the spread fee payout as `surplus − notionalFeeScaled`, never summing both against the raw surplus.

---

### Proof of Concept

1. Deploy a pool with `protocolSpreadFeeE6 = S > 0` and `protocolNotionalFeeE8 = N > 0`.
2. LP deposits liquidity into the pool.
3. Trader executes a swap; the pool accrues `spreadFee` in the surplus and `notionalFee` in both the surplus and `notionalFeeToken1Scaled`.
4. Factory calls `collectFees(S, 0, N, 0, dest)`.
5. `collectFees` computes:
   - `spread1ToProtocol = surplus1 × S / (S+0)` — includes `notionalFee`
   - `notional1ToProtocol = notionalFeeToken1Scaled × N / N = notionalFee`
   - Total transferred = `surplus1 + notionalFee` > `surplus1`
6. The excess `notionalFee` is taken from LP bin balances.
7. LP calls `removeLiquidity`; receives fewer tokens than deposited, confirming principal loss.

The test at `metric-core/test/MetricOmmPool.notionalFee.t.sol:211–265` reproduces steps 1–5 and asserts the overallocation. [4](#0-3)
