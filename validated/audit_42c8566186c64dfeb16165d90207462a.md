### Title
Unbounded Loop in `calculate_rewards` Over Growing Balance Trace Enables Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace since the last claim. Because each epoch in which a delegator modifies their balance appends a new entry to this trace, a delegator who accumulates many epochs of balance changes without claiming will eventually face a `claim_rewards` call that exceeds Starknet's block gas limit, permanently freezing their unclaimed yield.

### Finding Description
`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) iterates from `entry_to_claim_from` to `pool_member_trace_length`, reading each stored checkpoint from storage and calling `find_sigma` for every entry:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

The trace (`pool_member_epoch_balance`) is appended to by `set_member_balance`, which is called from `add_to_delegation_pool`, `exit_delegation_pool_intent`, `enter_delegation_pool`, and `switch_delegation_pool`. The `insert` logic in `trace.cairo` only appends a new checkpoint when the epoch key differs from the last entry's key — meaning at most one new entry per epoch. However, there is no cap on the total number of entries.

The `entry_to_claim_from` pointer (stored in `pool_member_info`) is only advanced when `claim_rewards` is successfully called. If a delegator makes balance changes across N epochs without ever claiming, the loop must process all N entries on the next `claim_rewards` invocation.

### Impact Explanation
Each loop iteration performs at least one storage read (`pool_member_trace.at(entry_to_claim_from)`) and one `find_sigma` call (which itself reads from `cumulative_rewards_trace`). As the trace grows, the gas cost of `claim_rewards` grows linearly. Once the trace is large enough, `claim_rewards` will exceed the block gas limit, making it impossible to ever collect the delegator's accumulated rewards. This constitutes **permanent freezing of unclaimed yield** (High impact).

### Likelihood Explanation
A delegator who makes balance changes (add/exit intent) across many epochs without claiming will accumulate trace entries at a rate of up to one per epoch. The code itself acknowledges the risk with the comment "This loop is unbounded but unlikely to exceed gas limits." The likelihood is low-to-medium: it requires many epochs of inactivity on the claim side, but is entirely within the reach of a long-term delegator who simply forgets to claim, or a delegator who deliberately avoids claiming to later grief themselves or demonstrate the issue. No privileged access is required — `claim_rewards` is callable by the pool member or their reward address.

### Recommendation
1. Introduce a maximum number of trace entries processed per `claim_rewards` call (a "batch size" cap), allowing partial claims that advance `entry_to_claim_from` incrementally.
2. Alternatively, enforce a maximum trace length by compacting old entries (e.g., merging entries older than the last claimed epoch into a single baseline checkpoint), preventing unbounded growth.
3. Add a protocol-level invariant or assertion that the trace length for any pool member cannot exceed a safe bound.

### Proof of Concept
1. Delegator calls `enter_delegation_pool` in epoch 0 — trace length = 1.
2. Each subsequent epoch, delegator calls `add_to_delegation_pool` (or `exit_delegation_pool_intent`) — each call to `set_member_balance` inserts at `epoch + K`, appending a new entry since the epoch key differs from the previous one. After N epochs: trace length = N+1.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After sufficiently many epochs, `claim_rewards` is called. The loop at line 859 iterates from 0 to N, reading N storage slots and calling `find_sigma` N times.
5. For large N, this exceeds the block gas limit, reverting the transaction. The delegator's accumulated rewards are permanently inaccessible.
