### Title
Missing Caller Restriction on `update_rewards` Allows Any User to Block Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the staking contract is specified as callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged user can call `update_rewards` with `disable_rewards: true`, consuming the single per-block reward slot without distributing rewards. Because the guard variable `last_reward_block` is global, this permanently blocks the legitimate sequencer call for that block, denying all stakers their consensus rewards.

### Finding Description
The protocol specification explicitly states:

> **access control**: Only starkware sequencer.

for `update_rewards`. However, the implementation at `src/staking/staking.cairo` lines 1449–1507 contains no `get_caller_address()` check against any sequencer address:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();          // only checks pause state
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validity checks ...
    self.last_reward_block.write(current_block_number);   // global write

    if disable_rewards || self.is_pre_consensus() {
        return;                            // exits without distributing rewards
    }
    // ... reward distribution ...
}
```

The critical design flaw is that `last_reward_block` is a **single global storage variable** for the entire contract. Calling `update_rewards` for any valid staker with `disable_rewards: true`:

1. Passes all guards (contract not paused, new block, valid active staker with non-zero balance).
2. Writes `last_reward_block = current_block_number`.
3. Returns immediately without distributing any rewards.

Any subsequent call to `update_rewards` in the same block — including the legitimate sequencer call — will revert with `REWARDS_ALREADY_UPDATED`. The attacker can repeat this every block, permanently preventing all stakers from earning any consensus rewards.

The `calculate_block_rewards` path in `reward_supplier.cairo` lines 166–187 is never reached, so `avg_block_duration` is never updated and `update_unclaimed_rewards_from_staking_contract` is never called, meaning the reward supplier's accounting is also never updated.

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users or protocol.**

The attacker earns nothing, but every staker in the protocol is denied their per-block consensus rewards for every block the attacker targets. If executed continuously, this permanently zeroes out all consensus-phase staking yield for the entire protocol. The `last_reward_block` global guard makes a single call sufficient to block rewards for all stakers in a given block.

### Likelihood Explanation
**Medium.** Any unprivileged address can call `update_rewards` — no stake, no delegation, no special role is required. The only practical constraint is that the attacker's transaction must be included before the sequencer's own `update_rewards` call in a given block. On Starknet the sequencer controls ordering, but:

- The sequencer may not call `update_rewards` every block (the function is designed to be called once per block, not mandatory).
- A determined attacker can flood the mempool with `update_rewards(valid_staker, disable_rewards: true)` transactions, making it statistically likely that at least some land before the sequencer's call.
- The attack is cheap: it only requires a valid staker address (publicly readable on-chain) and a single transaction per block.

### Recommendation
Add an explicit caller check matching the specification:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
}
```

Store the authorized sequencer address in contract storage and expose a governance-controlled setter. Alternatively, use Starknet's `get_execution_info` to verify the transaction originates from the sequencer account.

### Proof of Concept

1. Deploy the full staking system; activate consensus rewards via `set_consensus_rewards_first_epoch`.
2. Advance K epochs so a staker has non-zero effective balance.
3. At the start of each new block, attacker (any EOA) calls:
   ```
   staking.update_rewards(staker_address: <any_valid_staker>, disable_rewards: true)
   ```
4. `last_reward_block` is set to the current block number.
5. The sequencer's subsequent call `update_rewards(..., disable_rewards: false)` reverts with `REWARDS_ALREADY_UPDATED`.
6. No staker receives any consensus rewards for that block.
7. Repeat every block → all stakers earn zero consensus rewards indefinitely.

This is the direct analog of the external report: just as `setVote()` allowed a caller to manipulate a parameter (`voteGas` / `maxGasPrice`) to drain ETH from a guild, here a caller manipulates the `disable_rewards` parameter on a function that lacks the access control its specification requires, causing all stakers to lose their yield.
