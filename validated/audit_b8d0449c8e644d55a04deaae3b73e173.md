### Title
Presignature Not Consumed After Rerandomization Enables Nonce Reuse and Private Key Extraction — (File: `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

The `RerandomizedPresignOutput::rerandomize_presign` function accepts the `PresignOutput` by shared reference (`&PresignOutput`) rather than by value, and `PresignOutput` derives `Clone`. This means the library never consumes or invalidates the presignature after it is used to produce a rerandomized signing context. A malicious coordinator or a buggy caller can rerandomize the same presignature for multiple distinct messages, producing signatures whose nonces are multiplicatively related. Standard ECDSA nonce-reuse algebra then recovers the aggregate private key.

---

### Finding Description

`RerandomizedPresignOutput::rerandomize_presign` is defined as:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,       // ← shared reference, not moved
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) 

Because `presignature` is taken by `&` (not by value), the `PresignOutput` is never moved or dropped by the call. Additionally, `PresignOutput` derives `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
``` [2](#0-1) 

The rerandomization formula scales the nonce share by a message-dependent factor `delta`:

```
k_rerandomized  = k · delta⁻¹          (delta = HKDF(X, ε, h, R, ρ))
``` [3](#0-2) 

For two distinct messages `h₁` and `h₂`, the two rerandomized nonces satisfy:

```
k₁ = k · delta₁⁻¹
k₂ = k · delta₂⁻¹
⟹  k₁ / k₂ = delta₂ / delta₁   (publicly computable)
```

This is a multiplicatively related nonce pair. The signing document itself acknowledges this exact attack surface:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use **multiplicatively related nonces** and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [4](#0-3) 

The orchestration documentation further states:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [5](#0-4) 

Despite these documented requirements, the library API does not enforce one-time use at the type level. The presignature is never consumed, zeroed, or marked as spent after `rerandomize_presign` returns.

The same structural issue applies to `RerandomizedPresignOutput`, which also derives `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct RerandomizedPresignOutput { ... }
``` [6](#0-5) 

---

### Impact Explanation

Two signatures produced from the same underlying presignature nonce `k` (even after distinct rerandomizations) expose a linear relation between the nonce values. An attacker who observes both signatures and knows `delta₁`, `delta₂` (both publicly derivable from public inputs) can solve for the aggregate secret key `x` using standard ECDSA nonce-reuse equations. This constitutes **extraction of the aggregate private signing key** — a Critical impact under the allowed scope.

---

### Likelihood Explanation

A malicious coordinator is an explicitly in-scope attacker role. The coordinator controls which presignature is presented to each signing session. Because the library does not consume or invalidate the `PresignOutput` after use, the coordinator can:

1. Collect a presignature from honest participants during a legitimate presigning round.
2. Initiate signing session A for message `h₁` — honest participants contribute shares using their `PresignOutput`.
3. Immediately initiate signing session B for message `h₂` — honest participants have no library-enforced mechanism preventing them from reusing the same `PresignOutput`.
4. Recover the private key from the two resulting signatures.

Even without a malicious coordinator, a buggy integrator that retries a failed signing session with the same presignature triggers the same outcome. The library's own README warns "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions" but provides no enforcement. [7](#0-6) 

---

### Recommendation

1. **Consume `PresignOutput` by value** in `rerandomize_presign` so the Rust type system enforces one-time use:
   ```rust
   pub fn rerandomize_presign(
       presignature: PresignOutput,   // moved, not borrowed
       args: &RerandomizationArguments,
   ) -> Result<Self, ProtocolError>
   ```
2. **Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`** so callers cannot circumvent the move semantics by cloning.
3. Apply the same pattern to the FROST `PresignOutput<C>` (which contains `nonces: SigningNonces<C>`) and the robust ECDSA `PresignOutput`.

### Proof of Concept

```
// Attacker-controlled coordinator scenario:

let presign_out: PresignOutput = run_presign_protocol(...);

// Session 1 — message h1
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,          // presign_out is NOT consumed
    &args_for_h1,
)?;
let sig1 = run_sign(rerand1, h1, ...);

// Session 2 — message h2 (same presign_out, still valid)
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,          // reused — library does not prevent this
    &args_for_h2,
)?;
let sig2 = run_sign(rerand2, h2, ...);

// From sig1 = (r1, s1) and sig2 = (r2, s2):
//   s1 = (h1 + r1·x) · k1⁻¹,   k1 = k·delta1⁻¹
//   s2 = (h2 + r2·x) · k2⁻¹,   k2 = k·delta2⁻¹
//   k1/k2 = delta2/delta1  (known)
// Two equations, two unknowns (k, x) → solve for x (private key).
```
