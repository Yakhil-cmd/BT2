### Title
Global `last_reward_block` Lock Allows Anyone to Permanently Freeze All Staker Rewards via `update_rewards` with `disable_rewards: true` - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function enforces a one-update-per-block constraint using a single global `last_reward_block` variable. Because the function is publicly callable with no access control, any attacker can call `update_rewards(any_active_staker, disable_rewards: true)` every block to consume the global slot without distributing any rewards. This permanently freezes all staker yield at negligible cost.

### Finding Description
`update_rewards` is the V3 consensus-rewards entry point. It writes `last_reward_block` to the current block number **before** checking the `disable_rewards` flag:

```cairo
// src/staking/staking.cairo lines 1484-1489
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
```

The only gate is `general_prerequisites()`, which checks only that the contract is not paused and the caller is not the zero address:

```cairo
// src/staking/staking.cairo lines 1793-1797
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

There is no check that the caller is the staker, the staker's operational address, or any privileged role. The `last_reward_block` storage slot is a single global value shared across all stakers:

```cairo
// src/staking/staking.cairo line 187
last_reward_block: BlockNumber,
```

The guard that prevents double-updates in the same block is:

```cairo
// src/staking/staking.cairo lines 1454-1458
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

Because `last_reward_block` is written unconditionally before the `disable_rewards` branch, a call with `disable_rewards: true` consumes the slot for the entire block without distributing a single token to any staker.

**Attack path:**
1. Attacker monitors the chain. Each new block, attacker submits `update_rewards(any_active_staker_with_nonzero_balance, disable_rewards: true)`.
2. `last_reward_block` is set to the current block number.
3. The function returns early; no rewards are calculated or transferred.
4. Any legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeated every block, all stakers are permanently denied consensus rewards.

The attacker needs only to satisfy:
- The staker passed must exist and be active (`is_staker_active` check at line 1469).
- The staker must have a non-zero STRK balance at the current epoch (line 1482).

Both conditions are trivially satisfiable by observing any live staker on-chain.

### Impact Explanation
Every block's reward slot is consumed without distributing rewards. All stakers and all pool members are permanently denied their consensus-era yield. This matches the allowed impact: **Permanent freezing of unclaimed yield** (High).

The analog to the external report is direct: the external report describes how the fastest actors can exit locked markets while slower participants are left behind, and how the protocol favors speed. Here, the fastest caller each block wins the global `last_reward_block` slot. An attacker who wins every block permanently locks all other participants out of their yield, with no recourse until the protocol is paused or upgraded.

### Likelihood Explanation
Starknet transaction fees are low. The attacker needs one transaction per block with a trivially small calldata payload. No capital is required. The attack is permissionless, requires no privileged access, and is sustainable indefinitely. Likelihood is **High**.

### Recommendation
1. **Remove `disable_rewards` from the public interface** or restrict it to a privileged caller (e.g., the attestation contract or the staker's own operational address).
2. **Write `last_reward_block` only when rewards are actually distributed**, i.e., move the write to after the `disable_rewards` / `is_pre_consensus` guard:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
// Only consume the slot when rewards are actually distributed.
self.last_reward_block.write(current_block_number);
```

3. Alternatively, add an access-control check so only the staker themselves, their operational address, or the attestation contract can call `update_rewards`.

### Proof of Concept
```
Block N:
  Attacker calls: update_rewards(staker=<any_active_staker>, disable_rewards=true)
    → last_reward_block written to N
    → function returns early, zero rewards distributed

  Staker A calls: update_rewards(staker=A, disable_rewards=false)
    → assert!(N > N) → REVERTS with REWARDS_ALREADY_UPDATED

Block N+1:
  Attacker repeats.

Result: No staker ever receives consensus rewards.
```
