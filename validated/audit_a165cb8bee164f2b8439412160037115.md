### Title
Router's Accumulated Native ETH Silently Subsidizes Subsequent Users' WETH Swap Payments — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay()` function in `PeripheryPayments` reads `address(this).balance` — the router's **entire** native ETH balance — to settle WETH swap obligations before pulling from the actual payer. Any ETH left in the router from a prior user's `multicall` (who sent ETH but omitted `refundETH()`) is silently consumed to cover a subsequent user's WETH input. The subsequent user receives the full swap output without their WETH being pulled; the prior user's ETH is permanently lost.

---

### Finding Description

`pay()` is the internal settlement function called from every swap callback path (`_justPayCallback`, `_exactOutputIterateCallback`) and from the liquidity adder's `metricOmmModifyLiquidityCallback`. When `token == WETH` and `payer != address(this)`, the function branches on the router's current native balance: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer's WETH never touched
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

When `nativeBalance >= value`, **no `transferFrom` is issued against the payer at all**. The pool receives WETH (wrapped from the router's ETH), the payer's WETH balance is untouched, and the router's ETH is consumed.

ETH accumulates in the router because `multicall` is `payable`: [2](#0-1) 

Any user who sends ETH with `multicall` but omits `refundETH()` leaves a residue. The `receive()` guard only blocks bare ETH transfers (not ETH attached to function calls): [3](#0-2) 

The same vulnerability exists in `MetricOmmPoolLiquidityAdder`, which inherits `PeripheryPayments` and exposes `payable` liquidity functions: [4](#0-3) 

---

### Impact Explanation

- **Prior user** who left ETH in the router loses that ETH permanently — it is wrapped and transferred to the pool on behalf of a different user.
- **Subsequent user** (attacker) receives the full swap output or liquidity position without paying any WETH input. The pool is made whole (it receives WETH), so the pool itself is not harmed, but the attacker obtains value for free at the prior user's expense.
- The loss is bounded only by the ETH balance the router happens to hold, which can be significant in high-volume periods.

---

### Likelihood Explanation

- **Trigger condition**: a prior user must have left ETH in the router. This happens naturally whenever a user sends ETH with `multicall` and forgets (or is front-run before) calling `refundETH()`.
- **Attacker action**: call any WETH-input swap or liquidity function with 0 ETH. No special privilege is required.
- An attacker can also **deliberately induce** the condition: observe a pending `multicall{value: X}` in the mempool that lacks `refundETH()`, let it land, then immediately call `exactInputSingle` with WETH as `tokenIn` and 0 ETH.
- Likelihood is **Medium** — the precondition (stranded ETH) is not guaranteed but is a realistic and repeatable occurrence.

---

### Recommendation

Replace `address(this).balance` with `msg.value` (tracked per-call) so that only the ETH the **current caller** explicitly sent can be used to wrap WETH. A clean approach:

1. Pass `msg.value` into `pay()` as an explicit `nativeAvailable` parameter.
2. Use only that amount for wrapping; revert or pull from payer for any remainder.
3. After settlement, assert `address(this).balance` has not decreased beyond `msg.value` (or enforce a zero-balance invariant at the end of each top-level call).

Alternatively, enforce that the router's ETH balance is zero at the start of every top-level entry point (revert if `address(this).balance > msg.value`), preventing cross-user contamination entirely.

---

### Proof of Concept

```
1. User A calls router.multicall{value: 1 ether}([exactInputSingle(WETH→token1, amountIn=0.5e18, ...)])
   — no refundETH() step included.
   → Swap executes; 0.5 ETH is wrapped and sent to pool; 0.5 ETH remains in router.

2. User B calls router.exactInputSingle{value: 0}(
       pool=..., tokenIn=WETH, tokenOut=token1, amountIn=0.5e18, ...
   )
   → Pool triggers metricOmmSwapCallback → _justPayCallback → pay(WETH, userB, pool, 0.5e18).
   → address(this).balance == 0.5 ether >= value == 0.5e18.
   → Router wraps its own 0.5 ETH and transfers WETH to pool.
   → safeTransferFrom(userB, ...) is NEVER called.
   → User B receives token1 output; User B's WETH balance is unchanged.
   → User A's 0.5 ETH is gone.
``` [5](#0-4) [6](#0-5)
