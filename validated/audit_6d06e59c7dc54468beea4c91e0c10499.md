### Title
Gateway L2 Gas Price Threshold Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Incorrect Admission/Rejection Decisions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `validate_resource_bounds` function in `StatefulTransactionValidator` checks a transaction's `max_price_per_unit` against the **current block's** `l2_gas_price` field. However, the actual block being built will use `next_l2_gas_price` (the EIP-1559-adjusted price computed from the current block's gas consumption and stored in the block header). This mismatch causes the gateway to admit transactions that will fail `check_fee_bounds` during block building (when gas price is rising) and to reject transactions that would be valid for the next block (when gas price is falling). The code itself contains a `TODO` acknowledging the correct field to use.

### Finding Description

In `validate_resource_bounds`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

This reads `l2_gas_price` — the price that was used for transactions **in the already-committed block** — and uses it as the admission threshold. The block header also stores `next_l2_gas_price`, which is the EIP-1559-derived price for the **next** block:

```rust
pub next_l2_gas_price: GasPrice,
``` [2](#0-1) 

This `next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` and written into the block header by the consensus orchestrator:

```rust
next_l2_gas_price: self.l2_gas_price,
``` [3](#0-2) 

When the batcher builds the next block, it uses `self.l2_gas_price` (which equals the previous block's `next_l2_gas_price`) as the block context's gas price. The blockifier's `check_fee_bounds` inside `perform_pre_validation_stage` then validates the transaction's `max_price_per_unit` against this price:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(
        ResourceBoundsError::MaxGasPriceTooLow { ... }
    );
}
``` [4](#0-3) 

The `GatewayFixedBlockSyncStateClient.get_block_info_from_sync_client()` constructs a `BlockInfo` from the block header but only copies `l2_gas_price`, discarding `next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [5](#0-4) 

The two prices diverge whenever gas usage differs from the gas target. `calculate_next_base_gas_price` adjusts the price by up to `price / gas_price_max_change_denominator` per block:

```rust
let price_change = (price_u256 * gas_delta) / denominator;
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [6](#0-5) 

### Impact Explanation

**Case 1 — Invalid transaction admitted (gas price rising, `next_l2_gas_price > l2_gas_price`):**

An attacker submits a transaction with `max_price_per_unit = l2_gas_price`. The gateway threshold check passes (`max_price_per_unit >= threshold`). The transaction enters the mempool. During block building, `check_fee_bounds` compares `max_price_per_unit` against the actual block price (`next_l2_gas_price`), which is higher. The check fails; the transaction is rejected from every block until the gas price falls back. The mempool accumulates permanently unexecutable transactions, wasting sequencer resources and degrading throughput.

**Case 2 — Valid transaction rejected (gas price falling, `next_l2_gas_price < l2_gas_price`):**

A user submits a transaction with `max_price_per_unit = next_l2_gas_price`. This is sufficient for the next block, but the gateway threshold is `l2_gas_price` (higher). The transaction is rejected with `GAS_PRICE_TOO_LOW` even though it would execute successfully. Legitimate users are denied service.

Both cases match the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The EIP-1559 mechanism continuously adjusts `next_l2_gas_price` away from `l2_gas_price` whenever gas usage deviates from the target. In any active network, gas usage fluctuates, so the two prices routinely differ. The condition is unprivileged (any user submitting a transaction triggers it) and requires no special setup. The `min_gas_price_percentage` default of 100 means the threshold equals `l2_gas_price` exactly, maximising the window of divergence. [7](#0-6) 

### Recommendation

Replace the `l2_gas_price` read in `validate_resource_bounds` with `next_l2_gas_price` from the block header. This requires:

1. Extending `GatewayFixedBlockStateReader::get_block_info` (or adding a new method) to return `next_l2_gas_price` from `BlockHeaderWithoutHash`.
2. In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, propagate `block_header.next_l2_gas_price` into the returned value.
3. In `validate_resource_bounds`, use `next_l2_gas_price` as the threshold base instead of `l2_gas_price`.

This resolves the TODO already present in the code and aligns the gateway admission check with the price the batcher will actually enforce.

### Proof of Concept

1. Observe the latest committed block has `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (gas usage exceeded target).
2. Submit an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = P`.
3. Gateway `validate_resource_bounds` computes threshold = `P * 100% = P`; check passes.
4. Transaction is forwarded to the mempool and admitted.
5. Batcher builds the next block with `l2_gas_price = P'` in the block context.
6. `check_fee_bounds` evaluates `P < P'` → `MaxGasPriceTooLow` → transaction rejected from block.
7. Transaction remains in the mempool indefinitely, never executable, consuming mempool capacity and batcher processing cycles. [8](#0-7) [9](#0-8)
