### Title
Malicious Participant Can Corrupt CKD Output by Using an Attacker-Chosen `app_id` Without Detection - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` protocol function accepts `app_id` as a locally-supplied, unverified parameter from each participant. Because `app_id` is never included in the protocol messages exchanged between participants, the coordinator cannot detect whether a malicious participant used a different `app_id` than the one agreed upon. A single malicious MPC participant can silently corrupt the aggregated CKD output, causing honest parties to receive an unusable derived key.

### Finding Description

The `ckd()` entry point accepts `app_id` as a caller-supplied argument with no cross-participant consistency check: [1](#0-0) 

Each participant independently uses their local `app_id` inside `compute_signature_share()` to compute their contribution: [2](#0-1) 

The participant then sends only the computed group elements `(norm_big_y, norm_big_c)` to the coordinator — the `app_id` itself is never transmitted: [3](#0-2) 

The coordinator aggregates all received shares by simple addition, with no mechanism to verify that each participant used the same `app_id`: [4](#0-3) 

This is the direct analog to the external report: just as `atlasSolverCall()` accepted `sender` as a user-supplied variable that could not be guaranteed, here each participant supplies `app_id` locally and no protocol step enforces agreement across participants.

### Impact Explanation

The correct CKD output requires all participants to compute their share using the same `app_id`:

```
C_correct = Σ λᵢ · (xᵢ · H(pk, app_id) + yᵢ · A)
           = msk · H(pk, app_id) + y · A
```

If participant `j` uses `app_id_evil`:

```
C_corrupt = Σ_{i≠j} λᵢ · xᵢ · H(pk, app_id) + λⱼ · xⱼ · H(pk, app_id_evil) + y · A
```

This is not equal to `msk · H(pk, app_id) + y · A`. The app calling `unmask(app_sk)` will receive a wrong group element that does not correspond to any valid derived key. The corruption is silent — the coordinator returns a `CKDOutput` with no error, and honest parties have no way to distinguish a corrupted output from a valid one.

**Matched impact**: High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.

### Likelihood Explanation

- The attacker is a malicious MPC participant, which is explicitly within the documented threat model (the library is designed for settings with potentially faulty/Byzantine participants).
- The attack requires no privileged access: the malicious participant simply calls `ckd()` with a different `app_id` byte string.
- The `app_id` for a given session is known to all participants (it is agreed upon out-of-band), so the attacker knows exactly what value to deviate from.
- The attack is undetectable by the coordinator or any honest participant because `app_id` is never transmitted in any protocol message.

### Recommendation

Include `app_id` in the protocol transcript so that all participants commit to the same value before contributing their shares. Concretely:

1. **Broadcast `app_id` via echo-broadcast** at the start of the CKD round, so all participants agree on a single canonical `app_id` before computing their shares. Any participant sending a share computed under a different `app_id` would be detectable.
2. Alternatively, **include `app_id` in a session-binding hash** (similar to how `session_id` is used in DKG proofs of knowledge) so that each participant's contribution is cryptographically tied to the agreed `app_id`. [5](#0-4) 

The DKG already demonstrates the pattern: a `session_id` and `domain_separator` are hashed into every proof, binding each participant's contribution to the agreed session context. The same approach should be applied to `app_id` in the CKD protocol.

### Proof of Concept

1. Honest participants agree out-of-band on `app_id = b"legitimate_app"`.
2. Malicious participant `j` calls `ckd(..., app_id = b"evil_app", ...)` while all others call `ckd(..., app_id = b"legitimate_app", ...)`.
3. Participant `j` computes `big_s_j = H(pk, b"evil_app") * x_j` and sends `(λⱼ·Y_j, λⱼ·C_j)` to the coordinator.
4. The coordinator aggregates all shares without inspecting `app_id`: [6](#0-5) 

5. The coordinator returns a `CKDOutput` where `big_c` encodes a corrupted sum. The app calls `unmask(app_sk)` and receives a group element that is not `msk · H(pk, b"legitimate_app")`, making the derived key permanently unusable for the intended application — with no error or indication of tampering.

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
```

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```

```rust
/// Generates the challenge for the proof of knowledge
/// H(`domain_separator`, id, g^{secret} , R)
fn challenge<C: Ciphersuite>(
    domain_separator: &mut DomainSeparator,
    session_id: &HashOutput,
    id: Scalar<C>,
    vk_share: &CoefficientCommitment<C>,
    big_r: &Element<C>,
) -> Result<Challenge<C>, ProtocolError> {
    let mut preimage = vec![];
    let serialized_id = <C::Group as Group>::Field::serialize(&id);

    // Should not return Error
    // The function should not be called when the first coefficient is zero
    let serialized_vk_share = vk_share.serialize().map_err(|_| {
        ProtocolError::AssertionFailed(
            "The verification share
        could not be serialized as it is null"
                .to_string(),
        )
    })?;

    let serialized_big_r = <C::Group>::serialize(big_r).map_err(|_| {
        ProtocolError::AssertionFailed(
            "The group element R
        could not be serialized as it is the identity"
                .to_string(),
        )
    })?;

    preimage.extend_from_slice(&domain_separator.to_le_bytes());
    preimage.extend_from_slice(session_id.as_ref());
    preimage.extend_from_slice(serialized_id.as_ref());
    preimage.extend_from_slice(serialized_vk_share.as_ref());
    preimage.extend_from_slice(serialized_big_r.as_ref());

    domain_separator.increment();

    let hash = C::HDKG(&preimage[..]).ok_or(ProtocolError::DKGNotSupported)?;
    Ok(Challenge::from_scalar(hash))
}
```
