### Title
Gateway Skips `__validate__` (Signature Verification) for Invoke Transactions with Nonce=1 When Any Pending Transaction Exists for the Sender - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip `__validate__` only for the UX case where a user sends a `deploy_account` + `invoke(nonce=1)` pair simultaneously. However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** pending transaction from the sender — not exclusively a `deploy_account`. This means any invoke transaction with nonce=1 from an account whose on-chain nonce is 0 and which has **any** pending transaction in the mempool (e.g., a regular nonce=0 invoke) will bypass `__validate__` entirely at the gateway, allowing admission of transactions with invalid signatures.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` returns `true` (skip `__validate__`) when all of the following hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain nonce is 0).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

Condition 4 is satisfied by `Mempool::account_tx_in_pool_or_recent_block`, which returns `true` if the account has **any** transaction in the pool or is known from a recent committed block — it does not distinguish between a `deploy_account` and a regular `invoke`: [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`: [3](#0-2) 

This causes `StatefulValidator::perform_validations` to return `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling `__validate__`: [4](#0-3) 

The code comment in `skip_stateful_validations` claims the check is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is incorrect: an account with a regular nonce=0 invoke pending in the mempool satisfies condition 4 while being a fully deployed account — not a freshly deploying one.

### Impact Explanation

An attacker who observes that account `A` has a pending nonce=0 invoke in the mempool (on-chain nonce=0) can submit an invoke transaction with nonce=1 from account `A` carrying an **invalid signature**. The gateway will:

1. Pass `validate_nonce` (nonce=1 is within the allowed gap from account_nonce=0).
2. Pass `validate_by_mempool` (no duplicate hash, no fee escalation violation).
3. Call `skip_stateful_validations` → returns `true` because `account_tx_in_pool_or_recent_block` is `true`.
4. Skip `__validate__` entirely.
5. Admit the invalid transaction to the mempool.

The admitted transaction has an invalid signature and will fail when the batcher executes it. This constitutes **mempool admission of an invalid transaction** — the gateway's core admission invariant (every admitted invoke must have passed signature verification or be a known post-deploy UX exception) is broken.

### Likelihood Explanation

The attack requires only that the target account has a pending transaction in the mempool, which is observable from public mempool state. The attacker does not need to control the target account. The nonce=1 constraint is narrow but the window is open for the entire duration that the nonce=0 transaction is pending (which can be minutes to hours). Any deployed account that has submitted its first transaction is vulnerable during that window.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that only returns `true` when the pending transaction is of type `DeployAccount`. Alternatively, the skip condition should additionally verify that the sender address has no deployed class hash in the current state (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`), which would confirm the account is genuinely undeployed.

### Proof of Concept

1. Deploy account `A` (class hash set, on-chain nonce = 0).
2. Submit a legitimate invoke from `A` with nonce=0 and valid signature → it enters the mempool.
3. Craft an invoke from `A` with nonce=1 and an **invalid/arbitrary signature**.
4. Submit the crafted invoke to the gateway.
5. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓
   - `account_tx_in_pool_or_recent_block(A)` → `true` (nonce=0 tx is in pool) ✓
   - Returns `true` → `__validate__` is skipped.
6. The invalid invoke is admitted to the mempool without signature verification. [5](#0-4) [2](#0-1)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
