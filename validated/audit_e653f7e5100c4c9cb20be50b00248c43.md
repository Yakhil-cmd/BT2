### Title
WalletConnect Offer Confirmation Understates Total Royalty Cost for Multi-NFT Offers — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

`formatAmountWithRoyalties` in the WalletConnect command-display pipeline incorrectly divides the fungible amount by the number of NFTs before computing each royalty. The resulting "Total Amount with Royalties" shown in the WalletConnect approval dialog is systematically understated whenever an offer involves two or more royalty-bearing NFTs. The user approves a spend believing they will pay less than the blockchain will actually deduct.

---

### Finding Description

`parseCommandDisplay` is the function that builds the human-readable summary shown in the WalletConnect `Confirm` dialog before the user approves a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command. [1](#0-0) 

The faulty logic is:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← wrong
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Each NFT's royalty is computed on `amount / N` (where N = number of royalty-bearing NFTs) instead of on the full `amount`. The correct formula is:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The understated `amountWithRoyalties` value is then rendered in the WalletConnect confirmation dialog: [2](#0-1) 

The `withRoyaltyTotals` helper propagates this value to every fungible spending line: [3](#0-2) 

The royalty percentages fed into the function come from the NFTs on the opposite side of the trade: [4](#0-3) 

The existing test suite encodes and validates the buggy result (`0.00010255` instead of the correct `0.0001051`), confirming the defect is present and untested-against: [5](#0-4) 

---

### Impact Explanation

The WalletConnect confirmation dialog is the sole security gate between a dApp-initiated offer and the user's wallet. When the dialog shows an understated "Total Amount with Royalties," the user approves a transaction believing they will spend less XCH (or CAT) than the blockchain will actually deduct. The actual royalty amounts are enforced on-chain by the Chia wallet daemon and are always correct; only the display figure is wrong. The gap grows linearly with the number of NFTs and their royalty percentages.

This matches the allowed High impact: *"WalletConnect state that causes a user to approve… the wrong… amount."*

---

### Likelihood Explanation

Any unprivileged party can craft a valid offer containing two or more royalty-bearing NFTs and present it to a WalletConnect-connected wallet. No leaked keys, host compromise, or social engineering beyond the normal offer-sharing flow is required. Multi-NFT bundle offers are a documented and supported use case.

---

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be applied to the full fungible amount:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Each royalty is on the full amount, not on amount/N
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

Update the corresponding test expectations to reflect the corrected values.

---

### Proof of Concept

**Scenario:** A dApp presents a WalletConnect `take_offer` for two NFTs, each with a 5 % royalty (500 basis points), in exchange for 1 XCH (1 000 000 000 000 mojos).

**Buggy display (current code):**
```
splitAmount = 1_000_000_000_000 / 2 = 500_000_000_000
royaltyAmount = (500_000_000_000 × 500 / 10_000)
              + (500_000_000_000 × 500 / 10_000)
             = 25_000_000_000 + 25_000_000_000
             = 50_000_000_000
totalAmount  = 1_050_000_000_000 mojos  →  dialog shows "1.05 XCH"
```

**Correct calculation:**
```
royaltyAmount = (1_000_000_000_000 × 500 / 10_000)
              + (1_000_000_000_000 × 500 / 10_000)
             = 50_000_000_000 + 50_000_000_000
             = 100_000_000_000
totalAmount  = 1_100_000_000_000 mojos  →  should show "1.1 XCH"
```

The user sees **1.05 XCH** in the approval dialog and clicks Approve. The wallet daemon deducts **1.1 XCH**. The 0.05 XCH gap (5 % of the offer price) is invisible to the user at approval time and scales with both the number of NFTs and their royalty rates.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
