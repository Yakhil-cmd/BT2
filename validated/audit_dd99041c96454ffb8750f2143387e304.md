### Title
Missing Minimum Amount Enforcement in `add_to_delegation_pool` Allows Griefing via Event Flooding - (File: `src/pool/pool.cairo`)

---

### Summary

The `add_to_delegation_pool` function in `pool.cairo` accepts any non-zero amount without enforcing a minimum. A pool member holding even a trivial STRK balance can call this function thousands of times with `amount = 1`, emitting a `PoolMemberBalanceChanged` event on every call. This floods off-chain indexers and event consumers with noise at negligible cost to the attacker.

---

### Finding Description

`add_to_delegation_pool` validates the caller and amount with only two checks:

```cairo
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
``` [1](#0-0) 

There is no lower bound beyond `amount > 0`. After passing these checks the function transfers tokens, updates the balance trace, and unconditionally emits `PoolMemberBalanceChanged`:

```cairo
self.emit(
    Events::PoolMemberBalanceChanged {
        pool_member, old_delegated_stake, new_delegated_stake,
    },
);
``` [2](#0-1) 

The pool contract does store a `min_delegation_for_rewards` value (initialized from `STRK_CONFIG.min_for_rewards = 10^18`):

```cairo
pub(crate) const STRK_CONFIG: TokenRewardsConfig = TokenRewardsConfig {
    decimals: 18, min_for_rewards: 10_u128.pow(18), base_value: 10_u128.pow(28),
};
``` [3](#0-2) 

```cairo
self.min_delegation_for_rewards.write(config.min_for_rewards);
``` [4](#0-3) 

However, `min_delegation_for_rewards` is **only consulted during reward calculation**, never as a gate in `add_to_delegation_pool`. This is the direct analog of the LBTC `burnCommission` variable existing but not being enforced as a minimum at the entry point.

The same pattern exists in `increase_stake` in `staking.cairo`, which also only checks `amount.is_non_zero()`:

```cairo
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
``` [5](#0-4) 

By contrast, the initial `stake` call correctly enforces a minimum:

```cairo
assert!(amount >= self.min_stake.read(), "{}", Error::AMOUNT_LESS_THAN_MIN_STAKE);
``` [6](#0-5) 

This inconsistency means the minimum is enforced only at pool/staker creation, not on subsequent top-ups.

---

### Impact Explanation

An attacker who is already a pool member (an unprivileged role) can call `add_to_delegation_pool` with `amount = 1` (1 wei of STRK) in a tight loop. STRK has 18 decimals, so 1 STRK token = 10^18 units; a single STRK token is enough to sustain ~10^18 such calls (gas-limited in practice). Each call:

1. Emits a `PoolMemberBalanceChanged` event on-chain.
2. Writes a new checkpoint to the `pool_member_epoch_balance` trace.
3. Forces any off-chain indexer, monitoring service, or event-driven component to process the event.

This constitutes **griefing with no profit motive but damage to the protocol** (event log pollution, indexer overload, state bloat) — matching the allowed Medium impact: *Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption*.

---

### Likelihood Explanation

The entry path requires only that the attacker already be a pool member, which is an unprivileged, permissionless role. No special access, leaked key, or governance action is needed. The cost to the attacker is gas plus a negligible token amount (1 wei per call). The attack is therefore cheap, repeatable, and accessible to any delegator.

---

### Recommendation

Enforce a minimum amount in `add_to_delegation_pool` (and symmetrically in `increase_stake`) by reusing the already-stored `min_delegation_for_rewards` threshold or a dedicated minimum-top-up constant:

```cairo
assert!(
    amount >= self.min_delegation_for_rewards.read(),
    "{}",
    Error::AMOUNT_LESS_THAN_MIN_DELEGATION
);
```

This mirrors the fix applied to LBTC: initialize and enforce the threshold at every entry point that accepts an amount, not only at first registration.

---

### Proof of Concept

1. Deploy the staking system and have a staker open a STRK delegation pool.
2. Enter the pool as `attacker` with the minimum viable amount.
3. In a loop, call `add_to_delegation_pool(pool_member: attacker, amount: 1)` 10 000 times.
4. Observe that 10 000 `PoolMemberBalanceChanged` events are emitted, each with `new_delegated_stake` incremented by 1 wei, at a total token cost of 10 000 wei (≈ 10^-14 STRK).

```cairo
// Pseudocode PoC
let attacker = pool_member_address;
// Enter pool once with any valid amount
pool_dispatcher.enter_delegation_pool(reward_address: attacker, amount: MIN_STAKE);

// Spam tiny top-ups
let i: u32 = 0;
loop {
    if i == 10_000 { break; }
    pool_dispatcher.add_to_delegation_pool(pool_member: attacker, amount: 1);
    i += 1;
};
// 10_000 PoolMemberBalanceChanged events emitted; off-chain indexers flooded
``` [7](#0-6)
