### Title
COAOwnershipProof Signature Verification Missing Chain ID Binding Enables Cross-Network Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over an arbitrary `signedData` blob using the hardcoded domain separation tag `"FLOW-V0.0-user"`. This tag is identical across every Flow network (mainnet, testnet, previewnet, emulator). Neither the tag nor the protocol enforces that `signedData` encodes the chain ID. A valid COA ownership proof signature produced on one Flow network can therefore be replayed verbatim on any other Flow network, bypassing the chain-identity check that the Linea report identifies as the root class of vulnerability.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` (lines 1082–1086) calls `Crypto.KeyList.verify` with the fixed tag `"FLOW-V0.0-user"`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
``` [1](#0-0) 

The tag is a compile-time constant with no network component. The function's own NatSpec comment acknowledges a cross-address replay risk but says nothing about cross-network replay:

> "Note: this function does not enforce that `signedData` includes `evmAddress`." [2](#0-1) 

The `COAOwnershipProofInContext` struct that carries the proof contains `SignedData` as a raw byte slice with no chain-ID field:

```go
type COAOwnershipProofInContext struct {
    COAOwnershipProof
    SignedData SignedData
    EVMAddress Address
}
``` [3](#0-2) 

The EVM-side entry point is the `isValidSignature` (ERC-1271) function in the COA Solidity contract, which calls the `verifyCOAOwnershipProof` precompile with `address(this)` and the caller-supplied `_hash` and `_sig`:

```solidity
(bool ok, bytes memory data) = cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig));
``` [4](#0-3) 

The precompile decodes the proof and delegates to `coaOwnershipProofValidator`, which invokes `EVM.validateCOAOwnershipProof` on-chain: [5](#0-4) 

The `DecodeABIEncodedProof` function in `fvm/evm/precompiles/arch.go` reconstructs the proof from the raw ABI input with no chain-ID validation: [6](#0-5) 

Flow EVM chain IDs are distinct per network (`747` mainnet, `545` testnet, `646` previewnet) but are only used for EIP-155 transaction signing, not for COA ownership proof signing: [7](#0-6) 

### Impact Explanation

Any EVM contract or Cadence script that calls `verifyCOAOwnershipProof` / `EVM.validateCOAOwnershipProof` for authorization (e.g., a DeFi protocol using ERC-1271 for off-chain order approval, a bridge escrow, or an NFT marketplace) can be tricked into accepting a signature that was produced on a different Flow network. If the protocol does not embed the chain ID in its own message hash, the attacker can replay a testnet signature on mainnet to impersonate the victim and authorize withdrawals, trades, or other asset-moving actions.

### Likelihood Explanation

- Cadence account addresses are deterministically generated; the same address commonly exists on both testnet and mainnet for developers.
- Developers routinely register the same key pair on both networks for testing.
- Testnet transactions are public; an attacker can extract any `COAOwnershipProof` signature from testnet history.
- Protocols that hash only application-level data (without chain ID) produce the same `signedData` hash on both networks, making the replay directly applicable.
- The `"FLOW-V0.0-user"` tag is hardcoded and cannot be overridden by callers, so there is no opt-in mitigation available to protocol developers.

### Recommendation

Bind the domain separation tag to the network by appending the chain ID, for example `"FLOW-V0.0-user-<chainID>"`, or enforce at the protocol level that `signedData` must encode the chain ID before signatures are accepted. The analogous fix in the Linea report was to add `chainId` as a public input to the verifier; the equivalent here is to make the domain separation tag network-specific so that a proof signed on testnet is cryptographically invalid on mainnet.

### Proof of Concept

1. Victim holds a COA on both Flow Testnet and Flow Mainnet under the same Cadence account address with the same key pair.
2. A mainnet DeFi protocol uses `isValidSignature` (ERC-1271) for off-chain order authorization; its message hash does not include the chain ID.
3. Victim signs the same message hash on testnet to test the protocol integration; the resulting `COAOwnershipProof` is embedded in a public testnet EVM transaction.
4. Attacker extracts the encoded proof bytes from the testnet transaction.
5. Attacker submits a mainnet EVM transaction to the DeFi protocol, supplying the testnet `_hash` and the extracted testnet `_sig`.
6. The protocol calls `isValidSignature` on the victim's mainnet COA contract.
7. The COA contract calls `verifyCOAOwnershipProof(address(this), _hash, _sig)` via the Cadence Arch precompile.
8. `EVM.validateCOAOwnershipProof` verifies the signatures with `domainSeparationTag: "FLOW-V0.0-user"` — identical to the tag used on testnet — and confirms the victim's mainnet COA address matches; verification returns `true`.
9. The protocol executes the authorized action (e.g., releases escrowed assets) on behalf of the victim without the victim's mainnet consent.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
