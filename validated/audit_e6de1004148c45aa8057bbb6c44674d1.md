### Title
Cross-COA Signature Replay in `EVM.validateCOAOwnershipProof()` — (`File: fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof()` verifies that a Cadence account's keys signed `signedData` and that the account controls a COA at a given path with a given EVM address. However, the function does not enforce that `signedData` encodes the target `evmAddress`. A signature produced to prove ownership of one COA can be replayed by an attacker to prove ownership of a different COA owned by the same Cadence account, bypassing COA-specific authorization in any Cadence contract or off-chain flow that calls this function directly.

### Finding Description

`EVM.validateCOAOwnershipProof()` is an `access(all)` Cadence function that accepts caller-controlled parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. [1](#0-0) 

The function verifies the cryptographic signatures over `signedData` using the account's keys, then checks that the COA resource at `path` has the given `evmAddress`. Crucially, it does **not** verify that `signedData` encodes `evmAddress` or `path`. The code itself acknowledges this: [2](#0-1) 

The signature verification step only checks that the provided signatures are valid over `signedData` with the account's keys: [3](#0-2) 

The COA address check is separate and independent — it verifies that the COA at `path` matches `evmAddress`, but does not verify that the signer intended to authorize that specific COA: [4](#0-3) 

Because `validateCOAOwnershipProof` is `access(all)`, any unprivileged Cadence script or transaction can call it with attacker-controlled `path` and `evmAddress` parameters while supplying a previously observed valid signature over arbitrary `signedData`.

The `COAOwnershipProofInContext` type confirms that `SignedData` and `EVMAddress` are independent fields with no enforced binding: [5](#0-4) 

### Impact Explanation

An attacker who observes a valid COA ownership proof signature (e.g., from a public transaction or off-chain authentication flow) can replay it against a different COA owned by the same Cadence account. Any Cadence contract or off-chain system that calls `validateCOAOwnershipProof` directly for access control — for example, to gate asset withdrawals, bridge operations, or privileged EVM calls — can be deceived into accepting the replayed proof as valid authorization for a COA the signer never intended to authorize. This constitutes a transaction authorization bypass leading to unauthorized access to on-chain assets or privileged operations.

### Likelihood Explanation

The attack requires: (1) a Cadence account that owns more than one COA (stored at different paths), and (2) a caller that uses `validateCOAOwnershipProof` directly for access control rather than routing through the EVM precompile. The code comment acknowledges that "Flow wallets historically create at most one COA per account," which reduces likelihood today. However, the function is `access(all)` and the protocol places no technical barrier against multiple COAs per account. As COA usage grows and multi-COA patterns emerge, the attack surface increases. Any off-chain authentication flow built on top of this function is immediately vulnerable without additional application-level mitigations.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes the target `evmAddress` (and optionally `path`). The function should reconstruct the expected signed message as `hash(evmAddress || path || user_data)` and verify signatures against that bound message, rather than verifying signatures over an arbitrary caller-supplied `signedData`. This mirrors the fix pattern described in the external report: binding the signed payload to the specific context (COA address) to prevent cross-context replay.

### Proof of Concept

1. Alice owns a Cadence account with two COAs: `COA_A` at `/public/coaA` (EVM address `0xAAAA`) and `COA_B` at `/public/coaB` (EVM address `0xBBBB`).
2. Alice signs `signedData = hash("authenticate")` to prove ownership of `COA_A` for some application. The proof is submitted on-chain and becomes publicly observable.
3. An attacker observes Alice's `(keyIndices, signatures, signedData)` tuple from the transaction.
4. The attacker calls `EVM.validateCOAOwnershipProof(address: Alice, path: /public/coaB, signedData: hash("authenticate"), keyIndices: [...], signatures: [...], evmAddress: 0xBBBB)`.
5. The signature verification passes — Alice's keys did sign `signedData`. The COA address check passes — `COA_B` is at `/public/coaB` with address `0xBBBB`. The function returns `ValidationResult(isValid: true)`.
6. Any Cadence contract that uses this result to gate access to `COA_B`-controlled operations now grants the attacker unauthorized access, despite Alice never having authorized any action on `COA_B`.
