### Title
Declare Transactions Bypass `max_l2_gas_amount` Admission Check in `StatelessTransactionValidator` — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `max_l2_gas_amount` upper-bound check for `Declare` transactions. Any account with sufficient balance can submit a `Declare` transaction whose `l2_gas.max_amount` exceeds the configured gateway limit, bypassing the admission guard that exists for every other transaction type.

### Finding Description

In `validate_resource_bounds` the guard is written as an empty `if`-branch for `Declare`, followed by an `else if` that only fires for non-`Declare` types:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs  lines 78-85
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

This is structurally identical to the external bug's `if (transferTokensToSender && !unwrap)` pattern: a boolean/type condition silently drops the critical action for one specific case, while the rest of the code assumes the check ran.

The downstream stateful path provides no compensating guard. `StatefulTransactionValidator::validate_resource_bounds` only checks the L2 gas *price* against the previous block price, not the gas *amount*:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 223-243
async fn validate_resource_bounds(&self, executable_tx: &ExecutableTransaction) -> ... {
    if self.config.validate_resource_bounds {
        ...
        self.validate_tx_l2_gas_price_within_threshold(
            executable_tx.resource_bounds(),
            previous_block_l2_gas_price,
        )?;
    }
    Ok(())
}
```

The blockifier's `perform_pre_validation_stage` does call `verify_can_pay_committed_bounds`, which computes `max_possible_fee = max_l2_gas_amount × max_l2_gas_price`. For a normal user this would fail with `InsufficientFeeTokenBalance` when `max_l2_gas_amount` is enormous. However, for any account whose on-chain balance satisfies `balance ≥ max_l2_gas_amount × max_l2_gas_price`, the transaction clears every check and enters the mempool with an unbounded gas-amount field.

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 355-372
pub fn perform_pre_validation_stage<S: State + StateReader>(
    &self, state: &mut S, tx_context: &TransactionContext,
) -> TransactionPreValidationResult<()> {
    Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
    if self.execution_flags.charge_fee {
        self.check_fee_bounds(tx_context)?;
        verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
    }
    ...
}
```

### Impact Explanation

A well-funded account can submit a `Declare` transaction with `l2_gas.max_amount` set to any value above `config.max_l2_gas_amount` (up to `u64::MAX`). The gateway's stateless admission control — the only layer that enforces this per-transaction cap — is silently bypassed. The transaction is admitted to the mempool and forwarded to the batcher. The actual fee charged at execution time is based on real gas consumed (bounded by the validated class size), so no direct fund loss occurs, but the invariant "no transaction in the mempool may request more than `max_l2_gas_amount` L2 gas" is broken for `Declare`.

**Matching impact:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation

The trigger requires only a valid `Declare` transaction and an account balance large enough to satisfy `verify_can_pay_committed_bounds` for the chosen `max_l2_gas_amount`. No privileged access is needed. The code path is reachable by any external caller of the public gateway endpoint.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx {}` branch and apply the same `max_l2_gas_amount` check unconditionally, or add a separate, appropriately-sized limit for `Declare` transactions:

```rust
// Apply to all transaction types:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
```

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and a well-formed Sierra class that passes `validate_class_length` and `validate_entry_points_sorted_and_unique`.
2. Fund the sender account so that `balance ≥ u64::MAX × max_l2_gas_price`.
3. Submit via the gateway's `add_transaction` endpoint.
4. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for `Declare`.
5. The stateful validator also passes (only price, not amount, is checked).
6. `blockifier::StatefulValidator::perform_validations` calls `self.execute(tx)` for `Declare`, which calls `perform_pre_validation_stage`; `verify_can_pay_committed_bounds` passes because the account is funded.
7. The transaction enters the mempool with `max_l2_gas_amount = u64::MAX`, violating the gateway's admission invariant.

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
