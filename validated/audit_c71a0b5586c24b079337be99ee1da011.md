### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Staker Rewards - (File: src/staking/staking.cairo)

---

### Summary

The spec explicitly restricts `update_rewards` to "Only starkware sequencer," but the implementation enforces no such restriction. Because the function writes to a single global `last_reward_block` variable before any reward logic runs, any unprivileged caller can consume the per-block reward slot with `disable_rewards: true`, permanently blocking the sequencer from distributing consensus rewards to all stakers.

---

### Finding Description

`docs/spec.md` §`update_rewards` states:

> **access control:** Only starkware sequencer. [1](#0-0) 

The implementation, however, only calls `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero. There is no sequencer-identity check. [2](#0-1) 

The function then unconditionally writes `last_reward_block` to the current block number **before** the `disable_rewards` early-return guard: [3](#0-2) 

`last_reward_block` is a single global storage slot shared across all stakers: [4](#0-3) 

The `REWARDS_ALREADY_UPDATED` guard prevents any second call in the same block: [5](#0-4) 

Consequence: if an attacker calls `update_rewards(any_valid_staker, true)` in block N, `last_reward_block` becomes N and the function returns immediately without distributing rewards. Every subsequent call in block N — including the sequencer's calls for all other stakers — reverts with `REWARDS_ALREADY_UPDATED`. The sequencer's entire per-block reward distribution is silently voided.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

By submitting `update_rewards(any_valid_staker, disable_rewards: true)` once per block, an attacker prevents the sequencer from crediting `unclaimed_rewards_own` for every staker and from forwarding pool rewards to every delegation pool. No staker or pool member accrues any consensus-era yield for as long as the attack continues. Because Starknet block times are short and L2 gas costs are low, sustaining the attack is economically trivial relative to the yield denied to the entire staker set.

---

### Likelihood Explanation

The entry point is fully public — `general_prerequisites()` only rejects the zero address and a paused contract. Any EOA or contract can call `update_rewards` with an arbitrary valid `staker_address` and `disable_rewards: true`. No privileged role, leaked key, or external dependency is required. The attacker needs only to front-run the sequencer's transaction in each block, which is straightforward on Starknet where transaction ordering is visible in the mempool.

---

### Recommendation

Add an access-control guard at the top of `update_rewards` that restricts the caller to the Starkware sequencer address (obtainable via `starknet::get_sequencer_address()`) or to a dedicated role registered in `RolesComponent`. The guard must be placed **before** the `last_reward_block.write(...)` call so that an unauthorized call cannot consume the per-block slot.

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    // ADD: assert!(get_caller_address() == starknet::get_sequencer_address(), "...");
    let current_block_number = starknet::get_block_number();
    assert!(current_block_number > self.last_reward_block.read(), ...);
    ...
    self.last_reward_block.write(current_block_number); // only reached by sequencer
    ...
}
```

---

### Proof of Concept

1. Consensus rewards are active (`consensus_rewards_first_epoch` is set and current epoch ≥ it).
2. Attacker identifies any valid, active staker address `S`.
3. In block N, attacker submits `update_rewards(S, disable_rewards: true)`.
   - `general_prerequisites()` passes (contract unpaused, caller non-zero). [6](#0-5) 
   - `last_reward_block` is written to N. [7](#0-6) 
   - `disable_rewards == true` → early return, zero rewards distributed. [8](#0-7) 
4. Sequencer attempts `update_rewards(any_staker, false)` in block N → reverts `REWARDS_ALREADY_UPDATED` because `N > N` is false.
5. All stakers and pool members receive zero block rewards for block N.
6. Attacker repeats step 3 in every subsequent block → all stakers permanently frozen out of consensus-era yield.
