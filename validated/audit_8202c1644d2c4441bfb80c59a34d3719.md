### Title
Unbounded Loop in `calculate_rewards` Over Pool Member Balance Trace Enables Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an unbounded `while` loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace. Because each balance change in a distinct epoch appends a new checkpoint to this trace with no cap on its length, a pool member who makes frequent balance changes across many epochs without claiming rewards can grow the trace until `claim_rewards` always exhausts the Starknet per-transaction gas limit, permanently freezing their unclaimed yield. The developers themselves acknowledge the risk in a code comment: *"This loop is unbounded but unlikely to exceed gas limits."*

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:**

`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) iterates from `entry_to_claim_from` to `pool_member_trace_length`, calling `find_sigma` (a storage read) on every iteration:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

**How the trace grows:**

`set_member_balance` inserts a checkpoint at key `current_epoch + K` into `pool_member_epoch_balance`. The underlying `insert` only overwrites the last entry when the key matches; otherwise it appends a new checkpoint. Therefore, one new trace entry is created for every epoch in which the pool member modifies their balance. [2](#0-1) 

Balance-modifying entry points available to an unprivileged pool member:
- `enter_delegation_pool` → calls `set_member_balance` (line 201)
- `add_to_delegation_pool` → calls `increase_member_balance` → `set_member_balance` (line 242)
- `exit_delegation_pool_intent` → reduces balance via `set_member_balance` [3](#0-2) 

**Why `entry_to_claim_from` does not protect against this:**

`entry_to_claim_from` is stored in `pool_member_info` and is only advanced on a *successful* `claim_rewards` call (line 358). If the pool member accumulates N balance changes across N epochs without claiming, the next `claim_rewards` must iterate all N entries in a single transaction. If N is large enough to exceed the Starknet gas limit, the call reverts, `entry_to_claim_from` is never updated, and every subsequent `claim_rewards` attempt also reverts — permanently freezing the yield. [4](#0-3) 

**Callers of `calculate_rewards`:**

Both `claim_rewards` (line 349) and the view function `pool_member_info_v1` (line 532) invoke `calculate_rewards` with the full unclaimed range, so both are affected. [5](#0-4) 

### Impact Explanation

A pool member who makes balance changes in many distinct epochs without claiming rewards can grow their `pool_member_epoch_balance` trace without bound. Once the trace is large enough that a single `claim_rewards` call exceeds the Starknet per-transaction gas limit, the pool member's accumulated yield becomes permanently unclaimable. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

Any pool member who actively manages their delegation (e.g., a DeFi protocol that auto-compounds or adjusts stake each epoch) will naturally accumulate one trace entry per epoch. With Starknet epoch durations on the order of hours, a protocol running for months could accumulate hundreds of entries. The risk is elevated because the loop also performs a storage read (`find_sigma`) on every iteration, making each iteration expensive. The developers' own comment acknowledges the loop is unbounded.

### Recommendation

1. **Enforce a maximum trace length** or a maximum number of unclaimed epochs before a pool member must claim rewards.
2. **Alternatively**, restructure `calculate_rewards` to process only a bounded batch of entries per call and allow partial claiming, advancing `entry_to_claim_from` incrementally.
3. **At minimum**, add a hard cap on the number of loop iterations per transaction and revert with a descriptive error if the cap is reached, so the pool member is warned before the state becomes unrecoverable.

### Proof of Concept

1. Pool member `A` enters the delegation pool in epoch `E₀`.
2. In each subsequent epoch `Eᵢ`, `A` calls `add_to_delegation_pool` with a minimal amount. Each call appends one new entry to `pool_member_epoch_balance` because `get_epoch_plus_k()` returns a strictly increasing key.
3. `A` never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After `N` epochs (e.g., N ≈ 500–1000 depending on Starknet gas limits), `A` calls `claim_rewards`.
5. `calculate_rewards` enters the `while` loop and attempts to iterate all N entries, each requiring a `find_sigma` storage read. The transaction runs out of gas and reverts.
6. Because `entry_to_claim_from` is only written on success, every future `claim_rewards` call also reverts. `A`'s accumulated yield is permanently frozen. [1](#0-0) [6](#0-5)

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
