### Title
Irrecoverable Source-Coin Token Mapping Lock — No Deletion or Update Path for `cronos0x` Denoms — (File: `x/cronos/keeper/keeper.go`)

### Summary
`RegisterOrUpdateTokenMapping` handles source-coin denoms (`cronos0x...`) without any deletion or update path. Once a `cronos0x` mapping is written, it is permanently immutable: neither the admin nor governance can delete or replace it. If the mapped contract becomes non-functional after registration (e.g., self-destructed by its owner), all `ConvertCoinFromNativeToCRC21` and `ConvertCoinFromCRC21ToNative` flows for that token are permanently broken and unrecoverable, permanently stranding any native `cronos0x...` tokens held by users.

### Finding Description

In `RegisterOrUpdateTokenMapping`, the non-source-coin branch explicitly supports deletion by passing an empty `Contract` field:

```go
} else {
    if len(msg.Contract) == 0 {
        // delete existing mapping
        k.DeleteExternalContractForDenom(ctx, msg.Denom)
    }
``` [1](#0-0) 

The source-coin branch has no equivalent deletion path. It unconditionally calls `SetExternalContractForDenom`, which itself calls `ensureDenomNotMapped` and returns `ErrDenomAlreadyMapped` if the denom is already mapped:

```go
if types.IsSourceCoin(msg.Denom) {
    // ... validation ...
    if err := k.SetExternalContractForDenom(ctx, msg.Denom, contract); err != nil {
        return err   // always fails with ErrDenomAlreadyMapped on second call
    }
``` [2](#0-1) 

The governance path is also blocked. `TokenMappingChangeProposal.ValidateBasic()` hard-requires a valid hex contract address for source denoms, making it impossible to pass an empty contract to trigger deletion:

```go
if IsSourceCoin(tcp.Denom) {
    if !common.IsHexAddress(tcp.Contract) {
        return fmt.Errorf("invalid contract address for source denom: %s", tcp.Contract)
    }
}
``` [3](#0-2) 

The spec itself documents this gap: `"There's no way to delete a token mapping currently."` [4](#0-3) 

`DeleteExternalContractForDenom` exists in the keeper and correctly handles source denoms (confirmed by test at line 245), but it is never reachable for source coins through any exposed message or proposal handler. [5](#0-4) 

### Impact Explanation

When a source-coin mapping points to a dead contract, `ConvertCoinFromNativeToCRC21` (for source coins) burns the user's native `cronos0x...` tokens and then calls `transfer_from_cronos_module` on the dead contract — the EVM call fails, the entire tx reverts, and the user's tokens are permanently unspendable in the native layer. [6](#0-5) 

Symmetrically, `ConvertCoinFromCRC21ToNative` calls `transfer_by_cronos_module` on the dead contract before minting native coins — that call also fails, blocking the reverse direction. [7](#0-6) 

Because neither the admin (`MsgUpdateTokenMapping`) nor governance (`TokenMappingChangeProposal`) can delete or replace the stale mapping, the broken state is permanent. This satisfies the **High** impact criterion: *"Permanent or long-lived inability for honest users to process valid transactions, bridge/conversion flows"* and *"Corruption of token mappings, denom/contract binding... with direct security impact."*

### Likelihood Explanation

The attack path requires two steps:

1. **Admin registers a user-deployed contract** as a `cronos0x` source-coin mapping — a normal, expected operation.
2. **Contract owner (unprivileged) self-destructs the contract** after registration. `ensureContractCode` is only checked at registration time, not at conversion time, so a post-registration self-destruct is not prevented.

The contract owner needs no admin or governance access. Any user who deployed a CRC21 contract that was subsequently mapped by the admin can trigger this. Additionally, the admin themselves can accidentally create an irrecoverable state by mapping the wrong contract address, since there is no correction path.

### Recommendation

1. **Add a deletion path for source-coin mappings** in `RegisterOrUpdateTokenMapping`: when `msg.Contract` is empty and `IsSourceCoin(msg.Denom)`, call `DeleteExternalContractForDenom` (which already handles source denoms correctly).
2. **Update `TokenMappingChangeProposal.ValidateBasic()`** to allow an empty `Contract` field for source denoms (treating it as a delete request), mirroring the non-source-coin behavior.
3. **Consider adding a liveness check** in `ConvertCoinFromNativeToCRC21` / `ConvertCoinFromCRC21ToNative` that verifies contract code still exists before proceeding, to fail fast with a clear error rather than a silent revert.

### Proof of Concept

```
1. Admin calls MsgUpdateTokenMapping{Denom: "cronos0x<ADDR>", Contract: "<ADDR>"}
   → SetExternalContractForDenom writes the mapping. OK.

2. Contract owner calls selfdestruct() on <ADDR>.
   → Contract code is erased from EVM state.

3. User holds 100 cronos0x<ADDR> native tokens.
   User calls MsgConvertVouchers / ConvertCoinFromNativeToCRC21.
   → isSource=true → bankKeeper.BurnCoins succeeds (tokens burned)
   → CallModuleCRC21(contract, "transfer_from_cronos_module", ...) → EVM call to dead address → REVERT
   → Entire tx reverts. User's tokens are NOT burned (revert), but they can never successfully convert.

4. Admin attempts MsgUpdateTokenMapping{Denom: "cronos0x<ADDR>", Contract: "<NEW_ADDR>"}
   → SetExternalContractForDenom → ensureDenomNotMapped → ErrDenomAlreadyMapped. BLOCKED.

5. Admin attempts MsgUpdateTokenMapping{Denom: "cronos0x<ADDR>", Contract: ""}
   → RegisterOrUpdateTokenMapping source-coin branch: no deletion path, falls through to
     common.IsHexAddress("") == false → ErrInvalidAddress. BLOCKED.

6. Governance TokenMappingChangeProposal{Denom: "cronos0x<ADDR>", Contract: ""}
   → ValidateBasic: IsSourceCoin → !IsHexAddress("") → error. BLOCKED.

Result: mapping is permanently stuck pointing to dead contract.
        All conversion flows for cronos0x<ADDR> are permanently broken.
```

// DeleteExternalContractForDenom delete the external contract mapping for native denom,
// returns false if mapping not exists.
func (k Keeper) DeleteExternalContractForDenom(ctx sdk.Context, denom string) bool {
	store := ctx.KVStore(k.storeKey)
	contract, found := k.getExternalContractByDenom(ctx, denom)
	if !found {
		return false
	}
	store.Delete(types.DenomToExternalContractKey(denom))
	deleteReverseIfOwned(store, contract, denom)
	if auto, found := k.getAutoContractByDenom(ctx, denom); found {
		bz := store.Get(types.ContractToDenomKey(auto.Bytes()))
		if len(bz) == 0 {
			store.Set(types.ContractToDenomKey(auto.Bytes()), []byte(denom))
		} else if existingDenom := string(bz); existingDenom != denom {
			if k.contractOwnedByDenom(ctx, existingDenom, auto) {
				// auto address is already owned by another denom; drop local auto mapping
				store.Delete(types.DenomToAutoContractKey(denom))
			} else {
				// stale reverse entry
				store.Set(types.ContractToDenomKey(auto.Bytes()), []byte(denom))
			}
		}
	}
	return true
}

func (k Keeper) RegisterOrUpdateTokenMapping(ctx sdk.Context, msg *types.MsgUpdateTokenMapping) error {
	if types.IsSourceCoin(msg.Denom) {
		_, err := types.GetContractAddressFromDenom(msg.Denom)
		if err != nil {
			return err
		}

		if !common.IsHexAddress(msg.Contract) {
			return errors.Wrapf(sdkerrors.ErrInvalidAddress, "invalid contract address (%s)", msg.Contract)
		}
		contract := common.HexToAddress(msg.Contract)
		if err := k.ensureContractCode(ctx, contract); err != nil {
			return err
		}
		if err := validateContractAddressForSourceDenom(msg.Denom, contract, true); err != nil {
			return err
		}

		if err := k.SetExternalContractForDenom(ctx, msg.Denom, contract); err != nil {
			return err
		}
