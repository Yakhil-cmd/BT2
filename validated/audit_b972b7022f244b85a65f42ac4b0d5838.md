### Title
Uncopied Cache Reference Return in `BlockEndStateSnapshot.getFromCache` Allows Cache Corruption During Block Execution - (File: `engine/execution/storehouse/block_end_snapshot.go`)

---

### Summary

`BlockEndStateSnapshot.getFromCache` returns a direct `[]byte` reference into the internal `readCache` map without making a defensive copy. Because `flow.RegisterValue` is a type alias for `[]byte`, any caller that receives this reference and mutates the underlying array silently corrupts the cached entry. The same structural flaw exists in `LedgerStorageSnapshot.getFromCache` in `engine/execution/state/state.go`. Both snapshots are the base storage layer used during block execution on Execution nodes, and the `StorageSnapshot` interface contract explicitly requires idempotent, thread-safe reads — a guarantee that is broken when the backing bytes are mutated through the returned reference.

---

### Finding Description

`flow.RegisterValue` is declared as a type alias (`=`) for `[]byte`: [1](#0-0) 

`BlockEndStateSnapshot` holds a `readCache map[flow.RegisterID]flow.RegisterValue` and populates it on first access: [2](#0-1) 

`getFromCache` returns the map value directly — a reference to the same underlying byte array stored in the map: [3](#0-2) 

`Get` propagates this reference to every caller without interposing a copy: [4](#0-3) 

The identical pattern exists in `LedgerStorageSnapshot.getFromCache`: [5](#0-4) 

The `StorageSnapshot` interface contract explicitly requires thread-safety and idempotency: [6](#0-5) 

The returned reference travels up the full FVM call stack — through `storageState`, `spockState`, `ExecutionState.Get`, and finally `valueStore.GetValue` — before being handed to the Cadence runtime: [7](#0-6) [8](#0-7) 

At no point in this chain is a defensive copy made. The Cadence runtime receives the raw `[]byte` and may use it as a backing buffer during zero-copy CBOR decoding. Any in-place modification of that buffer — by the decoder, by a subsequent `append` that reuses the same backing array, or by any other FVM component — silently overwrites the entry in `readCache`.

---

### Impact Explanation

`BlockEndStateSnapshot` is constructed once per block and shared as the base snapshot across all transactions executing in that block: [9](#0-8) 

If a cached entry is mutated through the returned reference during transaction T₁'s execution, every subsequent transaction T₂…Tₙ in the same block that reads the same register ID will receive the corrupted value from cache — bypassing storage entirely. This breaks the idempotency guarantee of `StorageSnapshot.Get` and can cause:

- Incorrect register values fed into Cadence contract logic (balance reads, access-control checks, capability lookups).
- Divergent execution results between Execution nodes, since one node's in-memory cache may be corrupted while another's is not, producing different state commitments and breaking consensus.
- Silent state corruption that is not detectable at the Cadence level because the mutation happens below the FVM abstraction boundary.

---

### Likelihood Explanation

The structural defect is present in production code on the critical execution path for every block. Exploitation requires that some layer between `getFromCache` and the Cadence runtime mutates the returned `[]byte`. The Cadence CBOR decoder is known to support zero-copy modes that alias the input buffer; if such a mode is active, decoding a value and then writing a new value to the same register within the same transaction is sufficient to trigger corruption. A transaction sender needs no special privileges — any Cadence transaction that reads and then writes the same storage path exercises this path. Likelihood is **medium**: the structural precondition is always present; the triggering condition depends on whether the Cadence runtime's decoder aliases the buffer in the specific build configuration deployed.

---

### Recommendation

Return a defensive copy from both `getFromCache` implementations:

```go
// BlockEndStateSnapshot.getFromCache
func (s *BlockEndStateSnapshot) getFromCache(id flow.RegisterID) (flow.RegisterValue, bool) {
    s.mutex.RLock()
    defer s.mutex.RUnlock()
    value, ok := s.readCache[id]
    if !ok || value == nil {
        return value, ok
    }
    cp := make([]byte, len(value))
    copy(cp, value)
    return cp, true
}
```

Apply the same fix to `LedgerStorageSnapshot.getFromCache` in `engine/execution/state/state.go`. Alternatively, document and enforce a contract that callers must never mutate the returned slice, and audit every consumer in the FVM and Cadence integration layer to verify compliance.

---

### Proof of Concept

1. Deploy a Cadence contract that stores a value in account storage (register R).
2. Submit a transaction that:
   a. Reads register R via `getAccount(...).storage.load(...)` — this populates `readCache[R]` and returns the raw `[]byte` reference to the Cadence runtime.
   b. Writes a different value to register R via `save(...)` — the FVM's `storageState` write path may reuse or alias the same backing buffer depending on the CBOR encoder/decoder implementation.
3. Submit a second transaction in the same block that reads register R.
4. Observe that the second transaction receives the value written in step 2b from the cache rather than the original stored value, demonstrating that the cache entry was mutated through the uncopied reference returned in step 2a.
