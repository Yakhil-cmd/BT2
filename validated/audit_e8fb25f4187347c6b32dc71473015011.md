### Title
Incorrect Royalty Amount Calculation in WalletConnect `take_offer` Approval Dialog Causes User to Approve Understated Spend - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the base amount by the number of NFTs before computing each royalty, causing the displayed "total with royalties" in the WalletConnect `chia_wallet.take_offer` approval dialog to be understated by a factor of N (number of NFTs). A dApp can craft a multi-NFT offer where the approval dialog shows a significantly lower total cost than what the blockchain will actually charge, causing the user to approve a spend they would otherwise reject.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect `take_offer` command display to show the user their total XCH/CAT spend including NFT creator royalties: [1](#0-0) 

The function computes:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length); // divides by N first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This is mathematically wrong. For N NFTs with royalty percentages r₁…rN, the correct total royalty on `amount` is:

> `amount × (r₁ + r₂ + … + rN) / 10000`

But the function computes:

> `(amount / N) × (r₁ + r₂ + … + rN) / 10000`

The royalty component is understated by exactly 1/N. The existing test confirms this incorrect behavior is the expected output: [2](#0-1) 

For two NFTs with 500 bp (5%) and 10 bp (0.1%) royalties on 100,000,000 mojos (0.0001 XCH):
- **Displayed**: 0.00010255 XCH (royalty split by 2)
- **Correct**: 0.0001051 XCH (royalty applied to full amount per NFT)

The understated `amountWithRoyalties` is surfaced in the `DisplayWalletDelta` returned by `walletDeltaToDisplay` and rendered in the WalletConnect approval dialog: [3](#0-2) 

The approval flow is triggered by `parseCommandDisplay` for `chia_wallet.take_offer`: [4](#0-3) 

### Impact Explanation

A WalletConnect-connected dApp sends `chia_wallet.take_offer` with an offer containing multiple NFTs that have royalties. The approval dialog shows the user a total spend of X XCH (understated). The user approves. The blockchain charges the correct (higher) royalties, so the user actually spends Y XCH where Y > X. For 10 NFTs each with 10% royalty, the displayed royalty is 1/10 of the actual royalty — the user could approve a 2 XCH spend while the dialog shows 1.1 XCH. This is a direct, unprivileged-attacker-reachable balance loss caused by incorrect amount display in the WalletConnect signing approval context.

### Likelihood Explanation

Any dApp with an established WalletConnect session can send `chia_wallet.take_offer`. The attacker only needs to construct an offer with multiple NFTs that carry royalties. The user has no other signal in the approval dialog to detect the understatement — the displayed `amountWithRoyalties` is the only total shown.

### Recommendation

Remove the erroneous division by `royaltyPercentages.length`. Each royalty percentage should be applied to the full `amount`:

```typescript
// Before (incorrect):
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);

// After (correct):
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The test expectations in `parseCommandDisplay.test.ts` must also be updated to reflect the correct totals.

### Proof of Concept

1. dApp establishes a WalletConnect session with the Chia GUI wallet.
2. dApp constructs an offer: user spends 1 XCH to receive 10 NFTs, each with 1000 bp (10%) royalty.
3. dApp sends `chia_wallet.take_offer` via WalletConnect.
4. `parseCommandDisplay` calls `formatAmountWithRoyalties(line, 1_000_000_000_000n, [1000,1000,1000,1000,1000,1000,1000,1000,1000,1000])`.
5. `splitAmount = 1_000_000_000_000n / 10n = 100_000_000_000n`
6. `royaltyAmount = 10 × (100_000_000_000n × 1000n / 10_000n) = 10 × 10_000_000_000n = 100_000_000_000n`
7. Dialog displays: **1.1 XCH** total.
8. Correct total: `1_000_000_000_000n + 10 × (1_000_000_000_000n × 1000n / 10_000n) = 2_000_000_000_000n` → **2 XCH**.
9. User approves at 1.1 XCH displayed; blockchain deducts 2 XCH.

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
