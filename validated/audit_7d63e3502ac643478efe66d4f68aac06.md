### Title
Missing Lower-Bound Validation of `threshold` in `assert_sign_inputs` Allows Signing with Threshold = 1 — (`src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates the upper bound of the `threshold` parameter (must not exceed participant count) but omits the lower-bound check (`threshold >= 2`) that is consistently enforced in every other validation function in the library. A caller supplying `threshold = 1` passes all guards and drives the FROST signing protocol with a cryptographically invalid threshold, corrupting the aggregated signature output.

---

### Finding Description

`assert_sign_inputs` (used by both FROST EdDSA and FROST RedJubjub signing) performs the following threshold checks:

```rust
// src/frost/mod.rs  lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
```

There is **no corresponding lower-bound check** (`threshold < 2`). Compare this with every other validation helper in the library:

- `assert_key_invariants` (`src/dkg.rs`, lines 580-582):
  ```rust
  if threshold < 2 {
      return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
  }
  ```
- `validate_triple_inputs` (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`, lines 699-703):
  ```rust
  if threshold_value < 2 {
      return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
  }
  ```

The same gap exists in the FROST `presign` function (lines 71-77 of `src/frost/mod.rs`), which also validates only the upper bound.

`ReconstructionLowerBound` is a plain `usize` newtype with no internal invariant enforcing a minimum of 2:

```rust
// src/thresholds.rs  lines 9-12
pub struct ReconstructionLowerBound(usize);
```

So `ReconstructionLowerBound(1)` (or `ReconstructionLowerBound(0)`) is a valid value that passes through `assert_sign_inputs` unchallenged.

---

### Impact Explanation

In FROST, the signing threshold governs:
1. How many signature shares the coordinator must collect before aggregation.
2. The Lagrange coefficients used to combine those shares into the final scalar `s`.

When `threshold = 1` is accepted:
- The coordinator collects only one share `s_i`.
- The Lagrange coefficient for a single-element set evaluates to 1, so the aggregated scalar is simply `s = s_i` — a raw partial share, not the reconstructed group secret.
- The resulting `(R, s)` pair does not satisfy the FROST verification equation and is therefore a **corrupted, unusable signature output**.
- Honest participants who completed the protocol accept this output as the protocol result, unaware that the threshold parameter was invalid.

This maps to: **High — Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

`assert_sign_inputs` is a public library function. Any library caller — including a misconfigured application, a malicious coordinator constructing signing parameters, or a participant who controls the threshold argument — can supply `threshold = 1`. There is no type-level or runtime barrier preventing this. The keygen path correctly rejects `threshold < 2`, so the inconsistency is non-obvious and easy to miss during integration.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs`:

```rust
// src/frost/mod.rs — inside assert_sign_inputs, after the upper-bound check
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the identical fix to the `presign` function in the same file. Consider also encoding the invariant directly in `ReconstructionLowerBound::new` so it is impossible to construct a value below 2 at the type level.

---

### Proof of Concept

1. Run FROST EdDSA keygen with `threshold = 2`, `participants = [P1, P2, P3]` — succeeds normally.
2. Call `assert_sign_inputs(&[P1, P2, P3], /*threshold=*/1usize, P1, P1)` — **returns `Ok`** because the only threshold check is the upper-bound guard (`1 <= 3`).
3. Proceed to FROST signing with `threshold = 1`; the coordinator collects only `P1`'s share.
4. Lagrange interpolation over a single point yields coefficient 1; aggregated `s = s_P1`.
5. The resulting signature fails external FROST verification, demonstrating corrupted output accepted by the honest protocol participants. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
