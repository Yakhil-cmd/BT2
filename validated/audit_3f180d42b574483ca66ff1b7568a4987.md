### Title
EVM Cadence Arch `revertibleRandom()` Precompile and Cadence `revertibleRandom<UInt64>()` Share the Same PRG Stream — (`fvm/evm/handler/precompiles.go`)

---

### Summary

The EVM Cadence Arch precompile's `revertibleRandom()` and Cadence's native `revertibleRandom<UInt64>()` both drain bytes from the **same single `randomGenerator.prg` instance** that backs the FVM execution environment for a given transaction or script. An attacker who controls an EVM contract can call `revertibleRandom()` N times within an EVM transaction submitted via `EVM.run()`, observe the N returned values from the transaction result, and deterministically predict the next output that Cadence's `revertibleRandom<UInt64>()` will produce in the same outer Flow transaction.

---

### Finding Description

**Shared PRG construction path:**

`EVMInternalEVMContractValue` in `fvm/runtime/cadence_function_declarations.go` constructs the EVM backend by wrapping the same FVM environment that services all Cadence calls:

```go
evmBackend := backends.NewWrappedEnvironment(fvmEnv)   // line 94
``` [1](#0-0) 

`WrappedEnvironment.ReadRandom` is a thin pass-through to the underlying FVM environment:

```go
func (we *WrappedEnvironment) ReadRandom(buffer []byte) error {
    err := we.env.ReadRandom(buffer)   // same env as Cadence
    return handleEnvironmentError(err)
}
``` [2](#0-1) 

The EVM Arch precompile's `revertibleRandomGenerator` calls `backend.ReadRandom` directly:

```go
func revertibleRandomGenerator(backend backends.Backend) func() (uint64, error) {
    return func() (uint64, error) {
        rand := make([]byte, 8)
        err := backend.ReadRandom(rand)   // advances the shared PRG
        ...
    }
}
``` [3](#0-2) 

Both paths ultimately reach the single `randomGenerator` struct, which holds **one** `prg` field initialized once per transaction/script:

```go
type randomGenerator struct {
    ...
    prg          random.Rand
    isPRGCreated bool
}
``` [4](#0-3) 

```go
func (gen *randomGenerator) ReadRandom(buf []byte) error {
    if !gen.isPRGCreated {
        newPRG, err := gen.createPRG()
        ...
        gen.prg = newPRG
        gen.isPRGCreated = true
    }
    gen.prg.Read(buf)   // single sequential stream
    return nil
}
``` [5](#0-4) 

There is no separate PRG instance, no domain-separation tag, and no fork between the EVM and Cadence consumers. Every call to either `revertibleRandom()` (EVM) or `revertibleRandom<UInt64>()` (Cadence) advances the same cursor in the same CSPRG output stream.

---

### Impact Explanation

An attacker who controls both an EVM contract and the surrounding Cadence transaction can:

1. Submit an EVM transaction that calls `revertibleRandom()` N times and returns (or emits) the values.
2. Read those N values from `res.data` / EVM logs within the same outer Cadence transaction — this is confirmed by the existing test at `fvm/evm/evm_test.go:5307-5314` which reads the EVM random value back from `output.Value`.
3. Because the PRG is sequential and deterministic, the next 8-byte read — which is exactly what Cadence's `revertibleRandom<UInt64>()` will consume — is the (N+1)-th output of the same stream.

The attacker therefore learns the exact value that any subsequent Cadence `revertibleRandom<UInt64>()` call in the same transaction will return, breaking the unpredictability guarantee of Cadence randomness for any on-chain lottery, commit-reveal scheme, or randomized game that mixes EVM and Cadence calls in one transaction. [6](#0-5) 

---

### Likelihood Explanation

The attack requires only:
- Deploying an EVM contract (permissionless on Flow EVM).
- Submitting a standard Flow transaction that calls `EVM.run(attackerTx)` followed by `revertibleRandom<UInt64>()`.

No privileged access, no staked node, no key compromise. The call path is fully supported in production. The `staticcall` restriction in EVM only prevents EVM-state writes; it does not prevent the FVM PRG from being advanced, so even a `staticcall` to the Arch precompile advances the shared cursor.

---

### Recommendation

Introduce a domain-separated PRG fork for EVM randomness. In `revertibleRandomGenerator`, instead of calling `backend.ReadRandom` directly on the shared environment PRG, derive a child PRG seeded with an EVM-specific customizer (e.g., using `prg.New(source, prg.EVMExecutionEnvironment, salt)` analogous to how `createPRG` uses `prg.ExecutionEnvironment`). This ensures EVM and Cadence random streams are cryptographically independent even though they share the same entropy source. [7](#0-6) 

---

### Proof of Concept

```
// Solidity contract (attacker-deployed on Flow EVM)
contract AttackRandom {
    address constant ARCH = 0x0000000000000000000000010000000000000001;
    function drainN(uint n) external returns (uint64[] memory) {
        uint64[] memory vals = new uint64[](n);
        for (uint i = 0; i < n; i++) {
            (bool ok, bytes memory data) = ARCH.staticcall(
                abi.encodeWithSignature("revertibleRandom()")
            );
            require(ok);
            vals[i] = abi.decode(data, (uint64));
        }
        return vals;
    }
}

// Cadence transaction (attacker-controlled)
transaction {
    execute {
        // Step 1: run EVM tx that calls drainN(1), observe result
        let res = EVM.run(tx: attackerTxBytes, coinbase: coinbase)
        let evmRand: UInt64 = decodeUInt64(res.data)  // PRG output #1

        // Step 2: Cadence call gets PRG output #2 — predictable from #1
        let cadenceRand = revertibleRandom<UInt64>()

        // Attacker knows cadenceRand == PRG(seed, 2nd output)
        // because evmRand == PRG(seed, 1st output) and both use the same stream
    }
}
```

The attacker fixes the outer transaction ID (salt) and the block's random source (known at execution time), seeds the same CSPRG locally, skips N outputs (matching the EVM calls), and reads the next output — which equals the Cadence `revertibleRandom<UInt64>()` value.

func (gen *randomGenerator) createPRG() (random.Rand, error) {
	// Use the protocol state source of randomness [SoR] for the current block's
	// execution
	source, err := gen.entropySource.RandomSource()
	// `RandomSource` does not error in normal operations.
	// Any error should be treated as an exception.
	if err != nil {
		return nil, fmt.Errorf("reading random source from state failed: %w", err)
	}

	// Use the state/protocol PRG derivation from the source of randomness:
	//  - for the transaction execution case, the PRG used must be a CSPRG
	//  - use the state/protocol/prg customizer defined for the execution environment
	//  - use the salt as an extra diversifier of the CSPRG. Although this
	//    does not add any extra entropy to the output, it allows creating an independent
	//    PRG for each transaction or script.
	csprg, err := prg.New(source, prg.ExecutionEnvironment, gen.salt)
	if err != nil {
		return nil, fmt.Errorf("failed to create a CSPRG from source: %w", err)
	}

	return csprg, nil
```

func (gen *randomGenerator) ReadRandom(buf []byte) error {
	defer gen.tracer.StartExtensiveTracingChildSpan(
		trace.FVMEnvRandom).End()

	// PRG creation is only done once.
	if !gen.isPRGCreated {
		newPRG, err := gen.createPRG()
		if err != nil {
			return err
		}
		gen.prg = newPRG
		gen.isPRGCreated = true
	}

	gen.prg.Read(buf)
	return nil
