### Title
`uint32` Overflow in `_afterTimelock()` Allows Pool Admin to Bypass Stop-Loss Timelock — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock()` computes `uint32(block.timestamp + timelock)`. When the pool admin sets `timelock = type(uint32).max`, the addition overflows `uint32` and the stored `executeAfter` value is already in the past. Every subsequent `propose*` call stores a stale deadline, and the matching `execute*` call passes `_requireElapsed` immediately — the LP-protection timelock is fully bypassed.

---

### Finding Description

`_afterTimelock` is the single source of every `executeAfter` timestamp written by the four propose functions:

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

The result is cast to `uint32` **after** the addition is performed in `uint256`. When `timelock = type(uint32).max = 4 294 967 295`:

```
block.timestamp (≈ 1 753 000 000)  +  4 294 967 295  =  6 047 967 295
uint32(6 047 967 295)  =  6 047 967 295 − 4 294 967 296  =  1 752 999 999
```

The wrapped value `1 752 999 999` is **one second before** the current `block.timestamp`. The enforcement gate:

```solidity
// line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

compares `uint256 block.timestamp` against the promoted `uint32` value. Because `1 753 000 000 ≥ 1 752 999 999`, the revert is never reached and every `execute*` call succeeds immediately.

All four propose paths call `_afterTimelock` and write the corrupted deadline: [3](#0-2) [4](#0-3) [5](#0-4) 

There is no validation of the `timelock` value anywhere — neither in `initialize` nor in `executeOracleStopLossTimelock`: [6](#0-5) 

Compare with the validated fields: `_validateDrawdown` and `_validateDecay` both enforce upper bounds, but no equivalent `_validateTimelock` exists.

---

### Impact Explanation

The `OracleValueStopLossExtension` timelock is the only mechanism that gives LPs advance notice before the pool admin can weaken or disable the stop-loss. Once the timelock is set to `type(uint32).max`:

1. The admin calls `proposeOracleStopLossDrawdown(pool, 1e6)` — setting drawdown to 100 % makes `floorMultiplier = 0`, so `_applyWatermark` can never report a breach and the stop-loss is permanently disabled. [7](#0-6) [8](#0-7) 

2. `executeOracleStopLossDrawdown` passes `_requireElapsed` in the same block.
3. With the guard disabled, swaps that drain LP principal through a stale or manipulated oracle price are no longer blocked.

This is a direct LP principal loss path: the admin-boundary invariant (pool admin constrained by timelock) is broken by an arithmetic overflow, not by a privileged owner action.

---

### Likelihood Explanation

The pool admin is a semi-trusted role. The attack requires two steps:

1. **Set `timelock = type(uint32).max`** — either at pool creation (via `initialize` init-data, which is a trusted setup action) or post-deployment by proposing and executing a timelock change after the current timelock elapses. The second path is fully reachable by a semi-trusted admin on any live pool.
2. **Immediately execute any subsequent proposal** — no waiting required once the overflowed timelock is active.

No special permissions beyond `onlyPoolAdmin` are needed. The value `type(uint32).max` is a valid `uint32` and passes all existing checks.

---

### Recommendation

Validate the timelock value to prevent overflow, either in `_afterTimelock` or in the setter:

```solidity
// Option A: guard in _afterTimelock
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 timelock = oracleStopLossConfig[pool_].timelock;
    uint256 executeAfter = block.timestamp + timelock;
    if (executeAfter > type(uint32).max) revert TimelockOverflow();
    return uint32(executeAfter);
}

// Option B: validate on write (initialize + executeOracleStopLossTimelock)
function _validateTimelock(uint32 timelock) private pure {
    // block.timestamp < 2^31 for the next ~68 years; cap timelock to prevent wrap
    if (timelock > type(uint32).max / 2) revert TimelockTooLarge();
}
```
