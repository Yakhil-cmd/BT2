### Title
ETH Fee Permanently Locked in `RewardSupplier` When `tick()` Called With No Pending Mint Requests - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary
`RewardSupplier.tick()` is `payable` and requires ETH to cover L1â†’L2 messaging fees. However, the entire fee-forwarding logic is gated inside `if (amountToMint > 0)`. When `tick()` is called with `msg.value > 0` but no pending mint requests exist, all sent ETH is permanently locked in the contract with no refund path and no ETH withdrawal mechanism.

---

### Finding Description
`tick()` is declared `external payable` and callers must supply ETH to cover two downstream payable calls:

1. `bridge().depositWithMessage{value: msgFee}(...)` â€” `IBridge.depositWithMessage` is `payable`
2. `messagingContract().sendMessageToL2{value: msgFee}(...)` â€” `IStarknetMessaging.sendMessageToL2` is `payable`

Both calls are inside the block `if (amountToMint > 0)`: [1](#0-0) 

When `requiredMinting()` returns `(0, 0)` â€” i.e., no L2 mint-request messages are pending â€” the `if` block is skipped entirely. Any ETH sent with the call is accepted by the `payable` function but never forwarded and never refunded. [2](#0-1) 

`RewardSupplier.sol` contains no `receive()`, `fallback()`, or ETH-withdrawal function, so the ETH has no exit path. [3](#0-2) 

The `sendMessageToL2` interface confirms the fee is mandatory and payable: [4](#0-3) 

And `IBridge.depositWithMessage` is likewise `payable`: [5](#0-4) 

---

### Impact Explanation
Any ETH sent to `tick()` when `amountToMint == 0` is permanently locked in the `RewardSupplier` contract. Because `tick()` is intended to be called by automated keeper bots that may not know in advance whether pending messages exist (or that check off-chain but are front-run), repeated calls accumulate irretrievable ETH. This constitutes **permanent freezing of funds** (keeper/protocol-owned ETH) and **griefing with damage to the protocol** â€” bots lose gas-fee capital, which can deter future `tick()` calls and delay reward distribution.

---

### Likelihood Explanation
`tick()` is a public, permissionless entry point callable by any address. A keeper bot that calls `tick()` periodically (the intended usage pattern) will routinely send ETH. An adversary can front-run any pending `tick()` call by calling `tick()` themselves first with `msg.value = 0` (which succeeds when `amountToMint == 0` and costs nothing), draining the pending message queue so the legitimate caller's ETH-bearing call finds `amountToMint == 0` and locks their ETH. Even without adversarial front-running, any honest bot that sends ETH on a no-op tick loses those funds permanently.

---

### Recommendation
Add an ETH refund for the unused fee when `amountToMint == 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund unused fee to caller
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, remove the `payable` modifier and require callers to pre-fund the contract separately, or split `tick()` into a view check and a separate execution function.

---

### Proof of Concept

1. L2 `RewardSupplier` has no pending mint-request messages (normal idle state).
2. Keeper bot calls `RewardSupplier.tick{value: 0.01 ether}()`.
3. `requiredMinting()` returns `(0, 0)` â†’ `amountToMint == 0`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. `0.01 ether` is now held by `RewardSupplier` with no withdrawal path.
6. Repeated calls accumulate locked ETH indefinitely.

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
