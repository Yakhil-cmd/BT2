### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` internal function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace. Because the trace grows by one entry per epoch in which the pool member changes their balance, and because there is no cap on trace size or a mechanism to process rewards in batches, a pool member who makes balance changes across sufficiently many epochs without claiming rewards will eventually find that `claim_rewards` permanently reverts due to gas exhaustion, freezing their unclaimed yield forever.

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:**

`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) iterates over every entry in the pool member's `pool_member_epoch_balance` trace from `entry_to_claim_from` up to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
```

The developers themselves flag this as unbounded. [1](#0-0) 

**How the trace grows:**

Every call to `set_member_balance` inserts a new entry into the trace keyed by `current_epoch + K`:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

`set_member_balance` is called by:
- `enter_delegation_pool` (line 201) [3](#0-2) 
- `exit_delegation_pool_intent` (line 278) [4](#0-3) 
- `increase_member_balance` → called by `add_to_delegation_pool` (line 242) and `enter_delegation_pool_from_staking_contract` (line 456) [5](#0-4) 

Because the key is `current_epoch + K`, each call in a **different epoch** appends a distinct new entry. There is no deduplication across epochs and no cap on trace length.

**How `claim_rewards` is affected:**

`claim_rewards` (lines 335–377) calls `calculate_rewards` directly and only advances `entry_to_claim_from` on a **successful** transaction:

```cairo
let (mut rewards, updated_entry_to_claim_from) = self
    .calculate_rewards(
        :pool_member,
        from_checkpoint: pool_member_info.reward_checkpoint,
        :until_checkpoint,
        entry_to_claim_from: pool_member_info.entry_to_claim_from,
    );
...
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [6](#0-5) 

If the transaction reverts (gas exhaustion), `entry_to_claim_from` is never updated. Every subsequent call to `claim_rewards` restarts the loop from the same position and hits the same gas limit. The pool member's accumulated rewards are permanently frozen.

**Attacker-controlled entry path:**

`add_to_delegation_pool` (lines 221–254) allows the pool member **or their `reward_address`** to add stake:

```cairo
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
``` [7](#0-6) 

A malicious `reward_address` can call `add_to_delegation_pool` with a minimal amount in each successive epoch, growing the victim pool member's trace entry-by-entry until `claim_rewards` is permanently broken. The cost to the attacker is only the minimum token amount per epoch; the victim loses all accumulated yield.

Even without a malicious third party, a legitimate pool member who makes balance adjustments across many epochs and defers claiming will eventually self-lock their rewards with no recovery path.

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace is large enough that the `calculate_rewards` loop exceeds the Starknet transaction gas limit, every call to `claim_rewards` reverts. Because `entry_to_claim_from` is only persisted on success, the pointer never advances. The pool member's entire accumulated unclaimed yield is permanently inaccessible. There is no batching mechanism, no partial-claim path, and no administrative recovery function.

### Likelihood Explanation

**Likelihood: Medium.**

- A pool member who actively manages their delegation (adding/removing stake across many epochs) and infrequently claims rewards will organically grow their trace. Over hundreds of epochs this becomes a realistic concern.
- A malicious `reward_address` can deliberately accelerate trace growth by calling `add_to_delegation_pool` with dust amounts each epoch, requiring only token dust per epoch to execute the attack.
- The developers themselves acknowledge the loop is unbounded (the comment at line 858 reads "This loop is unbounded but unlikely to exceed gas limits"), confirming awareness of the risk but no mitigation.

### Recommendation

1. **Batch / paginated claiming**: Add a `claim_rewards_up_to(entry_limit: VecIndex)` variant that processes at most `entry_limit` trace entries per call and persists the updated `entry_to_claim_from` even for partial runs.
2. **Cap trace growth**: In `set_member_balance`, if the new key equals the last key already in the trace, update the existing entry in-place rather than appending. This limits the trace to at most one new entry per epoch.
3. **Enforce a minimum balance-change interval**: Require that balance changes are separated by at least one epoch, preventing rapid trace inflation.

### Proof of Concept

1. Pool member `P` joins pool with reward_address `R` (controlled by attacker).
2. Each epoch, `R` calls `add_to_delegation_pool(pool_member: P, amount: 1)` — one new entry is appended to `P`'s `pool_member_epoch_balance` trace.
3. After `N` epochs (where `N` is large enough to exhaust the block gas limit inside the `while` loop at line 859), `P` calls `claim_rewards(pool_member: P)`.
4. The transaction reverts due to gas exhaustion. `entry_to_claim_from` is not updated.
5. Every subsequent `claim_rewards` call by `P` or `R` also reverts. `P`'s accumulated yield is permanently frozen.
