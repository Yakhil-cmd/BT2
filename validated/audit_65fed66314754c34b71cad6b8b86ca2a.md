### Title
Malicious Participant Can Corrupt CKD Output by Supplying Mismatched `app_pk` Without Cross-Participant Consistency Enforcement — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry point accepts both `app_id` (the application identifier) and `app_pk` (the application's ElGamal public key) as independent, locally-supplied parameters. No protocol step broadcasts or cross-validates that every participant used the same `app_pk`. A single malicious participant can silently substitute an attacker-controlled public key, producing an aggregated CKD ciphertext that is not a valid ElGamal encryption under any single key, permanently rendering the CKD output undecryptable by the legitimate application.

---

### Finding Description

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts `app_id` and `app_pk` as two independent caller-supplied inputs:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,          // ← independent, locally-supplied, never cross-checked
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError>
```

Each participant independently computes their share in `compute_signature_share`:

```rust
// H(pk || app_id) when H is a random oracle
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
// S <- x . H(app_id)
let big_s = hash_point * private_share.to_scalar();
// C <- S + y . A          ← app_pk used here, never verified against app_id
let big_c = big_s + app_pk * y.0;
```

The coordinator then blindly aggregates every participant's `(big_y, big_c)` contribution with no consistency check on `app_pk`:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

For an honest run with all participants using the same `app_pk = A`:

```
big_C = msk · H(pk ‖ app_id) + A · Σ(λᵢ · yᵢ)
```

This is a valid ElGamal ciphertext decryptable by the holder of `app_sk`.

When malicious participant `P_m` substitutes `app_pk_attacker = A'`:

```
big_C = msk · H(pk ‖ app_id)
      + A  · Σ_{j≠m}(λⱼ · yⱼ)   ← honest blinding under legitimate key
      + A' · (λₘ · yₘ)           ← attacker's blinding under attacker key
```

This is not a valid ElGamal ciphertext under any single key. Neither the legitimate application (with `app_sk`) nor the attacker can recover `msk · H(pk ‖ app_id)` from this output.

The root cause is structurally identical to the reference report: two semantically coupled inputs (`app_id` and `app_pk`) are accepted independently, and the protocol never verifies their mutual consistency across participants.

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an unusable cryptographic output.**

The aggregated `(big_y, big_c)` pair output by the coordinator is a malformed ciphertext. The legitimate application cannot decrypt it to obtain its confidential derived key. Every honest participant accepts this output as the protocol result without detecting the corruption, because the coordinator performs no per-share verification. The CKD invocation must be entirely re-run (with the malicious participant excluded) to recover, and the corrupted output cannot be distinguished from a valid one by the coordinator or honest participants.

---

### Likelihood Explanation

**Medium.** Any single participant in the CKD session can trigger this by passing an attacker-controlled `app_pk` to their local `ckd()` call. No threshold of colluders is required — one malicious participant suffices. The attack requires no cryptographic break, no leaked keys, and no external dependency. It is directly reachable through the public library API by any participant who controls their own node.

---

### Recommendation

Add a broadcast-and-agree step for `app_pk` before any participant computes their share. Each participant broadcasts a commitment to (or the raw bytes of) their `app_pk`; the coordinator verifies all received values are identical before proceeding. If any participant's `app_pk` differs, abort with `ProtocolError::MaliciousParticipant`. This mirrors the existing `broadcast_success` pattern used in `src/dkg.rs` and the commitment-hash exchange already present in the DKG round structure.

Alternatively, include `app_pk` in the domain-separated hash input alongside `app_id` and the master public key, so that any `app_pk` mismatch produces a detectable inconsistency in the final output verification step (if one is added).

### Proof of Concept

```rust
// Honest participants call ckd() with the legitimate app_pk:
let legitimate_app_pk: ElementG1 = ElementG1::generator() * app_sk;

// Malicious participant P_m calls ckd() with an attacker-controlled key:
let attacker_sk = Scalar::random(&mut rng);
let attacker_pk  = ElementG1::generator() * attacker_sk;

// P_m's local call — substitutes attacker_pk for legitimate_app_pk:
let protocol_malicious = ckd(
    &participants,
    coordinator,
    p_m,
    key_pair_m,
    app_id.clone(),
    attacker_pk,   // ← mismatched app_pk, never detected by the protocol
    rng_m,
).unwrap();

// The coordinator aggregates all shares (including P_m's corrupted share)
// and produces a CKDOutput that is not a valid ElGamal ciphertext under
// legitimate_app_pk.  The call below will return a wrong value:
let corrupted_output = /* result of run_protocol(...) */;
let recovered = corrupted_output.unmask(app_sk);
// recovered ≠ msk · H(pk ‖ app_id)  — CKD output is silently corrupted.
```

The coordinator in `do_ckd_coordinator` has no mechanism to detect that `P_m` used a different `app_pk`; it simply sums `big_y` and `big_c` from all participants and returns the result as a valid `CKDOutput`.
