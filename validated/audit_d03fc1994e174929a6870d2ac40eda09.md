### Title
`calculateNFTRoyalties` Returns Inflated `nftSellerNetAmount`, Causing NFT Sellers to Approve Offers Displaying Wrong Proceeds - (`packages/gui/src/components/offers/utils.ts`)

### Summary
`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` hard-codes `nftSellerNetAmount = amount` (the full offered amount) instead of `amount - royaltyAmount - makerFee`. The correct subtraction is present but commented out. This wrong value propagates into both the "Net Proceeds" label and the "You will receive" override shown to an NFT seller at offer-acceptance time, causing the seller to see an inflated proceeds figure and potentially accept an offer under false pretenses about what they will actually receive.

### Finding Description

In `calculateNFTRoyalties`:

```ts
// packages/gui/src/components/offers/utils.ts  line 314-317
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The correct formula (`amount - royaltyAmount - makerFee`) is commented out, leaving `nftSellerNetAmount` equal to the raw offered amount regardless of royalties.

This value is consumed in two places in `NFTOfferViewer.tsx`:

**1. `overrideNFTSellerAmount` — overrides the "You will receive" row**

```ts
// NFTOfferViewer.tsx  lines 397-402
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)   // = full amount in mojos
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
```

This is passed as `overrideNFTSellerAmount` to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it replaces the on-chain amount in the "You will receive" display:

```ts
// OfferSummaryRow.tsx  line 146
const amount = overrideNFTSellerAmount ?? originalAmount;
```

**2. "Net Proceeds" label**

```tsx
// NFTOfferViewer.tsx  lines 595-597
<FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} /> {displayName}
```

The tooltip for this field explicitly states: *"The net proceeds include the asking price, minus any associated creator fees."* — but the value shown is the full asking price, not the asking price minus royalties.

**3. `showNegativeAmountWarning` is permanently suppressed**

```ts
// NFTOfferEditor.tsx  line 271
const showNegativeAmountWarning = (nftSellerNetAmount ?? 0) < 0;
```

Because `nftSellerNetAmount = amount` (always ≥ 0), this warning never fires even when royalties would consume the entire offered amount.

### Impact Explanation

When a buyer creates a `TokenForNFT` offer (e.g., "I offer 10 XCH for your NFT") and the NFT carries a 20% royalty, the NFT seller who opens that offer in `NFTOfferViewer` sees:

- **"You will receive: 10 XCH"** (overridden by `overrideNFTSellerAmount = chiaToMojo(10)`)
- **"Net Proceeds: 10 XCH"** (from `nftSellerNetAmount = 10`)

The seller actually receives **8 XCH** (2 XCH goes to the creator). The seller is shown the wrong amount at the exact moment they decide whether to click "Accept Offer." This directly causes a user to approve an offer displaying the wrong amount — matching the High impact criterion.

### Likelihood Explanation

Any NFT with a non-zero `royaltyPercentage` triggers this path. The `TokenForNFT` exchange type (buyer-initiated offer) is a standard workflow. An attacker can craft an offer for an NFT with a high royalty percentage, share the offer file, and the seller's GUI will display inflated proceeds, increasing the probability of acceptance.

### Recommendation

Restore the commented-out calculation in `calculateNFTRoyalties`:

```ts
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

This will correctly populate the "Net Proceeds" and "You will receive" displays, and will allow `showNegativeAmountWarning` to fire when royalties + fees exceed the offered amount.

### Proof of Concept

1. Mint an NFT with `royaltyPercentage = 2000` (20%).
2. As a buyer, create a `TokenForNFT` offer: "I offer 10 XCH for NFT X."
3. Export the offer file and open it in the seller's GUI via `NFTOfferViewer`.
4. Observe: "You will receive: 10 XCH" and "Net Proceeds: 10 XCH."
5. Accept the offer.
6. Actual on-chain result: seller receives 8 XCH; 2 XCH goes to the royalty address.

The discrepancy between the displayed 10 XCH and the actual 8 XCH is caused entirely by `nftSellerNetAmount = amount` at `packages/gui/src/components/offers/utils.ts` line 314, with the correct subtraction commented out at lines 315–317. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
