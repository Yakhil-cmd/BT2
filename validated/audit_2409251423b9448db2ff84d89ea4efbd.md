### Title
Insufficient Authorization Check in `skip_stateful_validations` Allows Signature Bypass for Nonce-1 Invoke Transactions — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator uses `account_tx_in_pool_or_recent_block` to decide whether to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 targeting an undeployed account. The check is too broad: it returns `true` for **any** transaction from the sender's address in the mempool, not exclusively a `deploy_account` transaction. An unprivileged attacker who observes a pending `deploy_account` transaction in the mempool can submit a crafted invoke transaction with nonce=1 for the victim's address, carrying an invalid or arbitrary signature, and have it admitted to the mempool without signature verification.

---

### Finding Description

`skip_stateful_validations` is a UX shortcut that allows a user to submit a `deploy_account` + nonce-1 `invoke` pair atomically, before the deploy is committed on-chain. The gateway skips the `__validate__` entry-point call for the invoke when three conditions hold simultaneously:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

Condition 3 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type from that address — not only `deploy_account`. The comment in the code acknowledges this imprecision:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**." [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, which causes `StatefulValidator::perform_validations` to return `Ok(())` without ever calling `__validate__`: [4](#0-3) [5](#0-4) 

The gateway then calls `mempool_client.add_tx` with the unauthenticated transaction: [6](#0-5) 

---

### Impact Explanation

**Mempool/gateway admission accepts invalid transactions before sequencing (High).**

An attacker who observes a pending `deploy_account` transaction for victim address `A` in the mempool can:

1. Submit an `Invoke V3` transaction with `sender_address = A`, `nonce = 1`, arbitrary `calldata`, and an invalid/forged `signature`.
2. The gateway's `validate_nonce` passes (0 ≤ 1 ≤ max_allowed_nonce_gap=200).
3. `validate_by_mempool` passes (no duplicate nonce=1 for `A` yet).
4. `skip_stateful_validations` returns `true` because `A` has a tx in the pool.
5. `__validate__` is skipped; the transaction is admitted to the mempool.

Concrete consequences:
- **Front-run / DoS**: The attacker's malicious nonce=1 tx occupies the nonce slot. The victim's legitimate nonce=1 invoke is rejected with `DuplicateNonce` (or must pay fee escalation to displace it).
- **Fee drain**: When the block is built, the batcher executes the malicious tx with `validate = true`. `__validate__` is called, fails (invalid signature), and the transaction reverts. The victim's newly-deployed account is charged fees for a transaction they never signed.
- **Mempool pollution**: Repeated submissions flood the mempool with unauthenticated transactions for any address with a pending deploy.

---

### Likelihood Explanation

The attack requires no special privileges. The attacker only needs to:
- Monitor the public mempool for `deploy_account` transactions (observable via the P2P propagation layer or the RPC).
- Submit a crafted invoke transaction before the deploy is committed.

The window is the entire time the `deploy_account` transaction sits in the mempool (potentially many blocks). The `max_allowed_nonce_gap` of 200 means the attacker can also submit nonce=1 through nonce=200 transactions, amplifying the DoS surface.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, not just the address presence. Alternatively, the gateway should track the `deploy_account` transaction hash submitted in the same request batch and verify it matches the pending pool entry before skipping validation.

The `max_nonce_for_validation_skip` field in `StatefulTransactionValidatorConfig` (default `Nonce(Felt::ONE)`) is defined but **not used** in `skip_stateful_validations` — the function hardcodes `Nonce(Felt::ONE)`. This field should be wired in to at least bound the skip window, and the type-specific check should be the primary guard. [7](#0-6) 

---

### Proof of Concept

```
1. Alice submits deploy_account tx for address A (nonce=0).
   → Gateway validates it, admits it to mempool.
   → mempool.tx_pool.contains_account(A) == true

2. Attacker submits Invoke V3 tx:
     sender_address = A
     nonce          = 1
     calldata       = [arbitrary]
     signature      = [0x0, 0x0]   // invalid

3. Gateway stateful validation for attacker's tx:
   a. get_nonce_from_state(A) → 0   (A not yet on-chain)
   b. validate_nonce: 0 ≤ 1 ≤ 200  → OK
   c. validate_by_mempool: no dup nonce=1 for A → OK
   d. skip_stateful_validations:
        tx.nonce() == 1  ✓
        account_nonce == 0  ✓
        account_tx_in_pool_or_recent_block(A) == true  ✓  (Alice's deploy is in pool)
      → returns true  →  __validate__ SKIPPED
   e. run_validate_entry_point: execution_flags.validate = false → Ok(())

4. Attacker's tx is added to mempool with invalid signature.

5. Alice's legitimate nonce=1 invoke tx is now rejected:
   MempoolError::DuplicateNonce { address: A, nonce: 1 }

6. During block execution:
   - Alice's deploy_account (nonce=0) executes → A is deployed.
   - Attacker's invoke (nonce=1) executes → __validate__ called → fails (bad sig) → REVERTED.
   - Alice's account is charged fees for the reverted tx she never signed.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
