### Title
TOCTOU: Pool Operator Can Redirect Farming Rewards by Swapping `targetPuzzleHash` Between Display and Submit — (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

When a user joins or changes a pool via the PlotNFT UI, the pool's `targetPuzzleHash` is fetched **twice** from the pool's HTTP endpoint: once to display pool details to the user, and again independently at transaction-submit time. A malicious pool operator can serve a legitimate `targetPuzzleHash` for the display fetch, then swap it to an attacker-controlled address for the submit fetch. The resulting `pw_join_pool` RPC call permanently points the user's PlotNFT at the attacker's puzzle hash, redirecting all future farming rewards (XCH) to the attacker.

---

### Finding Description

In `PlotNFTSelectBase`, `usePoolInfo(poolUrl)` fetches and renders pool details (including `targetPuzzleHash`) as the user types the URL: [1](#0-0) 

When the user clicks submit, `handleSubmit` in `PlotNFTSelectPool` calls `prepareSubmitData`, which makes a **completely independent, fresh HTTP call** to `getPoolInfo`: [2](#0-1) 

The `targetPuzzleHash` and `relativeLockHeight` returned by this second fetch — not the values the user reviewed — are what get passed directly to `pwJoinPool`: [3](#0-2) 

There is no comparison between the displayed values and the values used in the transaction. The same code path is triggered both for initial pool join (`PlotNFTSelectPool` standalone) and for pool changes (`PlotNFTChangePool`): [4](#0-3) 

---

### Impact Explanation

`targetPuzzleHash` is the address to which all pooled farming rewards are paid. By swapping it between the display fetch and the submit fetch, the attacker causes the user's PlotNFT to permanently point at an attacker-controlled puzzle hash. Every block reward earned by that PlotNFT thereafter is paid to the attacker. This is a **Critical** unauthorized transfer of XCH farming rewards.

---

### Likelihood Explanation

The attack requires the pool operator to control the pool's HTTP endpoint (i.e., the attacker *is* the pool operator, or has compromised it). A user who deliberately joins a malicious pool is the target. The window between the display fetch and the submit fetch is small but entirely within the attacker's control — the pool server can serve different responses to the same endpoint on successive requests. No cryptographic break, leaked key, or host compromise is required.

---

### Recommendation

At submit time, do not re-fetch pool info from the network. Instead, use the already-fetched and user-reviewed values from the form state or from the `usePoolInfo` hook result. If a re-fetch is considered necessary, compare the `targetPuzzleHash` (and `relativeLockHeight`) from the submit-time fetch against the values already displayed to the user, and abort with an explicit warning if they differ.

---

### Proof of Concept

1. Attacker operates a pool at `https://evil-pool.example/`.
2. The pool's `/pool_info` endpoint initially returns a legitimate `targetPuzzleHash` (e.g., `0xabc...`).
3. Victim opens the Chia GUI, navigates to **Join a Pool**, and enters `https://evil-pool.example/`. The UI fetches and displays `targetPuzzleHash: 0xabc...` via `usePoolInfo`.
4. Victim reviews the displayed pool details and clicks **Create**.
5. `handleSubmit` → `prepareSubmitData` → `getPoolInfo("https://evil-pool.example/")` fires a second HTTP request.
6. The attacker's server detects the second request (e.g., by timing, user-agent, or a race triggered by the first request) and responds with `targetPuzzleHash: 0xATTACKER_ADDRESS`.
7. `prepareSubmitData` returns `initialTargetState.targetPuzzleHash = 0xATTACKER_ADDRESS`.
8. `pwJoinPool` is called with the attacker's puzzle hash.
9. The victim's PlotNFT is now permanently configured to pay all farming rewards to `0xATTACKER_ADDRESS`.
