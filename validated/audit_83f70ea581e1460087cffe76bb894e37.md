### Title
Presignature Not Consumed After Rerandomization Enables Nonce Reuse and Secret Key Extraction - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both `RerandomizedPresignOutput::rerandomize_presign()` implementations (OT-based ECDSA and Robust ECDSA) accept the `PresignOutput` by shared reference (`&PresignOutput`) rather than by value. The presignature is therefore never consumed or invalidated after rerandomization. Combined with `PresignOutput` deriving `Clone`, the library provides no type-level enforcement of the one-time-use invariant that its own security documentation declares mandatory. A malicious coordinator can rerandomize the same `PresignOutput` with two different signing contexts `(h₁, ε₁, ρ₁)` and `(h₂, ε₂, ρ₂)`, run two parallel signing sessions, and recover the aggregate secret key via standard ECDSA nonce-reuse algebra.

---

### Finding Description

**Root cause — OT-based ECDSA (`src/ecdsa/ot_based_ecdsa/mod.rs`):**

```rust
// line 40 — PresignOutput is Clone; it is never consumed
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }

// line 66 — takes presignature by shared reference; original is not invalidated
pub fn rerandomize_presign(
    presignature: &PresignOutput,      // ← by reference, not by value
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

**Root cause — Robust ECDSA (`src/ecdsa/robust_ecdsa/mod.rs`):**

```rust
// line 26 — same Clone derivation
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }

// line 55 — same by-reference signature
pub fn rerandomize_presign(
    presignature: &PresignOutput,      // ← by reference, not by value
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

Because `rerandomize_presign` borrows the presignature rather than consuming it, a coordinator can legally call it twice on the same `PresignOutput`:

```rust
let r1 = RerandomizedPresignOutput::rerandomize_presign(&presign, &args_for_msg1)?;
let r2 = RerandomizedPresignOutput::rerandomize_presign(&presign, &args_for_msg2)?;
// Both r1 and r2 share the same underlying nonce R = g^k
```

The security documentation (`docs/ecdsa/robust_ecdsa/signing.md`, lines 147–177) explicitly states:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."
> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions."

The library documents the invariant but does not enforce it. The `rerandomize_presign` API is the exact entry point through which the invariant is violated.

The analog to the external report is direct: just as `_calculateRewards()` always re-includes the current `epochId` because `lastStakedEpoch` is never updated after a claim, `rerandomize_presign` never invalidates the `PresignOutput` after use, so the same presignature can be rerandomized and signed against an unbounded number of times.

---

### Impact Explanation

**Critical — Extraction/reconstruction of private signing shares.**

Two signatures produced from the same presignature nonce `R = g^k` under different message hashes `h₁ ≠ h₂` satisfy:

```
s₁ = k⁻¹(h₁ + r·x)
s₂ = k⁻¹(h₂ + r·x)
```

Subtracting: `s₁ - s₂ = k⁻¹(h₁ - h₂)`, so `k = (h₁ - h₂)/(s₁ - s₂)`, and then `x = (s₁·k - h₁)/r`. The full aggregate secret key is recovered from two publicly observable signatures. For the Robust ECDSA variant, the documentation further notes that the modified linearization step reduces the required presigning participants to as few as `2t + 2`, making the attack even easier to mount.

---

### Likelihood Explanation

The coordinator role is the natural attacker. The coordinator:
1. Holds the `PresignOutput` (it is the output of a protocol the coordinator participates in).
2. Calls `rerandomize_presign` twice with different `RerandomizationArguments` — two calls to a public API function, no cryptographic capability required.
3. Runs two signing sessions (possibly with overlapping or identical participant sets) and collects two valid signatures.
4. Applies standard nonce-reuse algebra offline to recover the secret key.

No privileged access beyond the coordinator role is needed. The coordinator role is explicitly part of the library's threat model (it is a named parameter in every `sign()` call).

---

### Recommendation

Enforce one-time use at the type level by consuming the `PresignOutput` by value in `rerandomize_presign`:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,       // consume by value — prevents reuse
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

Additionally, remove the `Clone` derivation from `PresignOutput` in both ECDSA variants so that callers cannot circumvent the move semantics by cloning before passing. The same hardening should be applied to `RerandomizedPresignOutput` and to `PresignOutput<C>` in `src/frost/mod.rs` (which also derives `Clone` and contains secret `SigningNonces`).

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

## Discarding information

Each phase can be run many times in advance, recording the information
public information produced, as well as the list of parties which produced it.

It's **critical** that the output is then destroyed, so that no other

use a certain output.
Otherwise, you might have two independent groups of parties trying
to use the same output, which is bad.
