### Title
Echo Broadcast Equivocation Protection Bypassed for n ≤ 3 Participants Due to Zero Thresholds — (File: src/protocol/echo_broadcast.rs)

---

### Summary

When the DKG participant count is 2 or 3, `echo_ready_thresholds` returns `(0, 0)` for both the echo and ready thresholds. Because every threshold comparison is strictly-greater-than (`> echo_t`, `> ready_t`, `> 2 * ready_t`), a single vote (count = 1) satisfies all three conditions immediately. This collapses the three-round echo broadcast into a trivial one-round protocol with no equivocation protection. A malicious participant can send different polynomial commitments to different honest parties during DKG, causing them to compute and accept inconsistent master public keys.

---

### Finding Description

**Root cause — `echo_ready_thresholds` returns `(0, 0)` for n ≤ 3:**

```rust
// src/protocol/echo_broadcast.rs, lines 67–78
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // echo_t = 0, ready_t = 0
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

These zero values are consumed at three decision points in `reliable_broadcast_receive_all`:

| Line | Condition | Effect when threshold = 0 |
|------|-----------|--------------------------|
| 225 | `count > echo_t` | Satisfied after **1 echo** (own simulated echo) |
| 282 | `count > ready_t` | Satisfied after **1 ready** (own simulated ready) |
| 295 | `count > 2 * ready_t` | Satisfied after **1 ready** (same message) |

Because `finish_echo` is set to `true` on the first Ready transition, each honest party locks in the **first value it echoes** — which is the value the malicious sender chose to send *to that specific party*.

**Attack path — DKG with n = 3, threshold = 2, one malicious participant P3:**

The DKG (`do_keygen` → `do_keyshare`) calls `do_broadcast` for two critical rounds:
- **Round 1** (step 1.3): reliable broadcast of session identifiers `sid_i`
- **Round 3** (step 3.2): reliable broadcast of `(C_i, R_i, s_i)` — the polynomial commitment and proof of knowledge

For the commitment equivocation attack (P3 honest in Round 1, malicious in Round 3):

1. P3 chooses two distinct degree-(threshold−1) polynomials `f_A` and `f_B` with `f_A(0) ≠ f_B(0)`.
2. **SEND phase of echo broadcast (Round 3):** P3 sends `C_A = f_A·G` to P1 and `C_B = f_B·G` to P2.
3. P1 echoes `C_A` (simulated); `count(C_A) = 1 > 0 = echo_t` → P1 sends `Ready(C_A)`, sets `finish_echo = true`.
4. P2 echoes `C_B` (simulated); `count(C_B) = 1 > 0 = echo_t` → P2 sends `Ready(C_B)`, sets `finish_echo = true`.
5. P1 receives `Ready(C_A)` (simulated); `count = 1 > 0 = ready_t` → amplify; `count = 1 > 0 = 2·ready_t` → **deliver `C_A`**.
6. P2 receives `Ready(C_B)` (simulated); same logic → **deliver `C_B`**.
7. P1 and P2 now hold different commitments for P3's polynomial.
8. **Round 2 (send_many, no broadcast protection):** P3 sends `h_3 = H(3, C_A, sid)` to P1 and `h_3 = H(3, C_B, sid)` to P2. Both pass the hash check in step 4.4.
9. **Round 5 (private shares):** P3 sends `f_A(1)` to P1 and `f_B(2)` to P2. Both pass the share-consistency check in step 5.2.
10. P1 computes `pk₁ = C₁(0) + C₂(0) + C_A(0)`, P2 computes `pk₂ = C₁(0) + C₂(0) + C_B(0)`. Since `C_A(0) ≠ C_B(0)`, `pk₁ ≠ pk₂`.

There is no subsequent check in the fresh DKG that forces all parties to agree on the master public key (the `pk == OldPK` assertion exists only in reshare).

---

### Impact Explanation

**High — Corruption of DKG output: honest parties accept inconsistent public keys.**

P1 and P2 each complete the DKG successfully from their own perspective, but hold different master public keys. Any threshold signing session will fail because the parties disagree on the verification key. The key material produced by the DKG is permanently unusable for its intended purpose.

---

### Likelihood Explanation

The DKG initialization check (`assert_key_invariants`) permits `participants.len() >= 2` and `threshold >= 2`, so a 3-participant DKG is a documented, supported configuration. A single malicious participant — the minimum required to mount this attack — is within the stated adversary model for DKG with threshold = 2. No special capabilities beyond controlling one participant's network messages are required.

---

### Recommendation

The `echo_ready_thresholds` function must not return `(0, 0)` for participant counts that the DKG allows with a non-zero malicious-party assumption. Concretely:

1. **Enforce the BFT precondition at the DKG entry point.** The echo broadcast requires `n ≥ 3f + 1`. Since the DKG allows up to `threshold − 1` malicious parties, add a check: `participants.len() >= 3 * (threshold - 1) + 1`. Reject configurations that violate this.
2. **Alternatively**, remove the `n <= 3` special case and let the formula `(n-1)/3` govern all sizes. For `n = 2` this yields `broadcast_threshold = 0`, which is correct (no tolerance), but the echo/ready thresholds should still be at least 1 to require a quorum from the other party.
3. **Document clearly** that the echo broadcast provides no equivocation protection when `n ≤ 3`, and that callers must not assume Byzantine fault tolerance in that regime.

pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
