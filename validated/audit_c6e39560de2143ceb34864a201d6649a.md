### Title
Malicious CKD Participant Can Corrupt Protocol Output Without Any Share Verification — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `do_ckd_coordinator` function in the CKD protocol blindly aggregates elliptic-curve group elements received from every participant with no cryptographic verification that those elements are consistent with the sender's committed public key share. A single malicious participant can therefore substitute arbitrary curve points, causing the coordinator to output a corrupted `CKDOutput` that does not encode `msk · H(pk, app_id)`. This is the direct analog of the reported `NonceVoter` bug: a function that is supposed to enforce multi-party integrity instead unconditionally accepts and accumulates whatever a single party sends.

---

### Finding Description

The CKD coordinator path is implemented in `do_ckd_coordinator`: [1](#0-0) 

```rust
async fn do_ckd_coordinator(...) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();   // ← no verification
        norm_big_c += participant_output.big_c();   // ← no verification
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
```

Each honest participant is supposed to send:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · A)
```

so that the coordinator's sum equals `(Y, C)` with `C = msk · H(pk, app_id) + Y · A`.

The participant path sends these values with no attached proof: [2](#0-1) 

```rust
fn do_ckd_participant(...) -> Result<CKDOutputOption, ProtocolError> {
    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
    Ok(None)
}
```

There is no zero-knowledge proof, commitment, or any other mechanism that binds `norm_big_c_i` to the participant's known public key share `x_i · G2` or to `norm_big_y_i`. The coordinator has no way to distinguish a correct share from an arbitrary pair of group elements.

Contrast this with the DKG protocol, which enforces a proof-of-knowledge for every commitment before accepting it: [3](#0-2) 

No equivalent check exists in the CKD aggregation loop.

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an unusable or wrong cryptographic output.**

A malicious participant substitutes `(norm_big_y', norm_big_c')` for its correct share. The coordinator computes:

```
Y'  = Y_honest + norm_big_y'
C'  = C_honest + norm_big_c'
```

The application unmasks as `C' − app_sk · Y'`, which equals:

```
msk · H(pk, app_id) + norm_big_c' − app_sk · norm_big_y'
```

Because the malicious participant does not know `app_sk`, it cannot make this equal to the correct key, but it can make it equal to any arbitrary group element by choosing `norm_big_c'` freely. Every honest party that relies on the coordinator's output will derive a wrong confidential key, silently, with no error returned. There is no post-aggregation consistency check analogous to the signature verification present in the OT-based ECDSA signing path: [4](#0-3) 

The CKD path has no equivalent guard.

---

### Likelihood Explanation

**High.** Any single participant in the CKD session — a role reachable by any library caller who holds a key share — can trigger this by sending two arbitrary `ElementG1` values instead of its honest contribution. No special privilege beyond participation is required. The attack is silent: the coordinator returns `Ok(Some(ckd_output))` with a corrupted value and no error.

---

### Recommendation

Require each participant to attach a zero-knowledge proof (e.g., a Schnorr proof of discrete-log equality) demonstrating that `norm_big_c_i − norm_big_y_i · A = λ_i · x_i · H(pk, app_id)`, binding the masked share to the participant's committed public key share. The coordinator must verify this proof before accumulating each participant's contribution, mirroring the proof-of-knowledge verification already enforced in the DKG path at `src/dkg.rs`.

---

### Proof of Concept

1. Honest setup: run DKG for `n = 3` participants with master secret key `msk`.
2. Participant `P_2` (malicious) overrides its `do_ckd_participant` to send `(norm_big_y' = G, norm_big_c' = G)` — two arbitrary generator points — instead of its correct share.
3. The coordinator aggregates without error and returns `CKDOutput { Y', C' }`.
4. The application computes `C' − app_sk · Y'`, which differs from `msk · H(pk, app_id)` by `G − app_sk · G = (1 − app_sk) · G`.
5. The derived confidential key is wrong; no error is raised anywhere in the protocol. [5](#0-4)
