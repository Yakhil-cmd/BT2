### Title
Blacklisted `adminFeeDestination` DoS on `collectFees`, `setPoolAdminFees`, and `setPoolProtocolFee` — (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`collectFees` in `MetricOmmPool` uses `safeTransfer` to push tokens directly to `adminFeeDestination_`. Three factory-level functions — `collectPoolFees`, `setPoolAdminFees`, and `setPoolProtocolFee` — call `collectFees` as a mandatory, non-skippable first step. If `adminFeeDestination` is or becomes a USDC/USDT blacklisted address, all three functions permanently revert until the destination is manually rotated, locking accrued protocol and admin fees in the pool and breaking fee-rate management for both the pool admin and the protocol owner.

---

### Finding Description

`MetricOmmPool.collectFees` transfers the admin share of accrued fees directly to `adminFeeDestination_` using `safeTransfer`:

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // safeTransfer — reverts on blacklist
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);
}
``` [1](#0-0) 

`transferToken0` / `transferToken1` are thin wrappers around `IERC20.safeTransfer`:

```solidity
function transferToken0(address to, uint256 amount) internal {
    IERC20(TOKEN0).safeTransfer(to, amount);
}
``` [2](#0-1) 

There is no try/catch and no fallback path. If `adminFeeDestination_` is blacklisted by USDC or USDT, `safeTransfer` reverts and the entire `collectFees` call reverts.

`collectFees` is called as a mandatory prerequisite — with no guard to skip it — inside all three of the following factory functions:

**`collectPoolFees`** (permissionless): [3](#0-2) 

**`setPoolAdminFees`** (pool admin): [4](#0-3) 

**`setPoolProtocolFee`** (protocol owner): [5](#0-4) 

The only recovery path is `setPoolAdminFeeDestination`, which does **not** call `collectFees` and therefore is not itself blocked: [6](#0-5) 

However, until the pool admin explicitly rotates the destination, all accrued fees are frozen in the pool and neither the pool admin nor the protocol owner can update fee rates.

---

### Impact Explanation

During the DoS window:

1. **Protocol and admin fees are locked in the pool.** `collectPoolFees` reverts; no fees can be extracted by either party.
2. **`setPoolAdminFees` is blocked.** The pool admin cannot change spread or notional fee rates.
3. **`setPoolProtocolFee` is blocked.** The protocol owner (factory owner) cannot update protocol fee components or clamp admin fees that exceed new caps — an admin-boundary break where a semi-trusted pool admin's destination choice neutralizes a privileged protocol-owner action.

Fees are not permanently lost (recovery via `setPoolAdminFeeDestination` is possible), but the DoS on fee management is real and the locked fees represent direct, quantifiable protocol revenue at risk during the window.

---

### Likelihood Explanation

- USDC and USDT are explicitly in scope per the allowed impact gate.
- `setPoolAdminFeeDestination` accepts any non-zero address with no further validation: [7](#0-6) 
- The pool admin is semi-trusted. They may set `adminFeeDestination` to a multisig, DAO treasury, or exchange deposit address that is later blacklisted by USDC/USDT without any on-chain warning.
- No monitoring or circuit-breaker exists in the contract to detect or handle a failed transfer.

---

### Recommendation

Decouple fee collection from fee-rate management. Two options:

1. **Wrap admin transfers in try/catch.** If the transfer to `adminFeeDestination_` fails, accumulate the owed amount in a claimable mapping rather than reverting. This mirrors the pull-over-push pattern.
2. **Separate `collectFees` from `setPoolAdminFees` / `setPoolProtocolFee`.** Do not force a fee collection inside fee-rate update functions. Instead, snapshot the current rates and update them atomically without requiring a live token transfer to succeed.

---

### Proof of Concept

1. Pool is deployed with USDC as `token0` and a valid `adminFeeDestination` address `D`.
2. Swaps accrue spread and notional fees; `notionalFeeToken0Scaled > 0`.
3. USDC blacklists address `D` (e.g., regulatory action on a DAO treasury).
4. Anyone calls `factory.collectPoolFees(pool)`:
   - `collectFees` is entered; `totalFee0ToAdmin > 0`.
   - `transferToken0(D, totalFee0ToAdmin)` → `USDC.safeTransfer(D, …)` → **reverts** (blacklisted).
   - `collectPoolFees` reverts. All accrued fees remain in the pool.
5. Pool admin calls `factory.setPoolAdminFees(pool, newFee, 0)`:
   - Same path → same revert. Fee rates cannot be updated.
6. Protocol owner calls `factory.setPoolProtocolFee(pool, newFee, 0)`:
   - Same path → same revert. Protocol cannot adjust its own fee component.
7. Pool admin calls `factory.setPoolAdminFeeDestination(pool, newD)` with a valid address → succeeds (no `collectFees` call inside). After this, all three functions above work again and fees can be collected.
