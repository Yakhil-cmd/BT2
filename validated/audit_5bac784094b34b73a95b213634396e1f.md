### Title
Malicious Participant Can Corrupt CKD Output via Unverified Contribution Accumulation — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly accumulates `(norm_big_y, norm_big_c)` contributions from all participants without any cryptographic verification that each contribution is consistent with the sender's actual key share. A single malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted derived key that honest parties will accept as valid. This is the direct analog of the reported `principalSharesDeposited` bug: state is incremented based on unverified participant-supplied values, creating a divergence between the tracked aggregate and the true cryptographic state.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55), the coordinator receives a `CKDOutput` from every other participant and unconditionally adds it to its running totals:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation performed by `compute_signature_share` (lines 148–181) produces:

- `norm_big_y = λᵢ · yᵢ · G₁` (Lagrange-weighted random nonce)
- `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)` (Lagrange-weighted key-share contribution) [2](#0-1) 

There is **no zero-knowledge proof, no consistency check, and no binding** between the received `(norm_big_y, norm_big_c)` and the sender's public key share (which is known from the DKG output). A malicious participant is free to substitute any pair of group elements.

The structural analog to the original bug is exact:

| Original (`Tranche.sol`) | This codebase (`protocol.rs`) |
|---|---|
| `principalSharesDeposited[receiver] += shares` incremented without actual share transfer | `norm_big_y += participant_output.big_y()` accumulated without verifying the contribution encodes the participant's real key share |
| Mapping reflects ownership that does not exist | Aggregate reflects a key-share sum that does not correspond to the real master secret |
| Downstream withdrawal fails or is incorrect | Downstream `unmask` produces a wrong derived key |

---

### Impact Explanation

The coordinator finalises the CKD output as:

```rust
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [3](#0-2) 

If participant `j` sends `(Ŷⱼ, Ĉⱼ)` instead of the correct `(λⱼ·yⱼ·G, λⱼ·(xⱼ·H+yⱼ·app_pk))`, the coordinator computes:

```
Y_out  = Σᵢ≠ⱼ λᵢ·yᵢ·G  +  Ŷⱼ
C_out  = Σᵢ≠ⱼ λᵢ·(xᵢ·H + yᵢ·app_pk)  +  Ĉⱼ
```

The `unmask(app_sk)` operation yields `C_out − app_sk·Y_out ≠ msk·H(pk, app_id)`. The coordinator outputs this corrupted value as the valid derived key. No honest party can detect the corruption within the protocol because the correct derived key is never independently computed or verified.

This matches the allowed **High** impact: *"Corruption of CKD outputs so honest parties accept inconsistent… unusable cryptographic outputs."*

---

### Likelihood Explanation

Any participant in the CKD protocol can mount this attack with no privileged access. The attacker need only deviate from `compute_signature_share` and send crafted group elements. The protocol provides no attribution or detection mechanism. The attack is single-round, requires no interaction beyond normal protocol participation, and is deterministically effective.

---

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` with a Chaum-Pedersen proof of discrete-log equality demonstrating that the same scalar `λᵢ·xᵢ` was used in both the public key share (known from DKG) and the `norm_big_c` term. Concretely, the participant proves knowledge of `λᵢ·xᵢ` such that:

```
λᵢ·xᵢ·G₂  =  (public key share from DKG)
λᵢ·xᵢ·H(pk, app_id)  =  norm_big_c − λᵢ·yᵢ·app_pk
```

The coordinator must verify this proof before accumulating each contribution. This is the standard fix for unauthenticated additive secret-sharing aggregation.

---

### Proof of Concept

1. Run the CKD protocol with `N = 3` participants; participant `P₂` is malicious.
2. `P₂` replaces its call to `compute_signature_share` and instead sends `(ElementG1::identity(), ElementG1::identity())` — the additive identity — to the coordinator.
3. The coordinator accumulates:
   - `norm_big_y = λ₁·y₁·G + 0 + λ₃·y₃·G` (missing `P₂`'s nonce)
   - `norm_big_c = λ₁·(x₁·H+y₁·app_pk) + 0 + λ₃·(x₃·H+y₃·app_pk)` (missing `P₂`'s key-share term)
4. `unmask(app_sk)` yields `(λ₁·x₁ + λ₃·x₃)·H(pk, app_id)` instead of `(λ₁·x₁ + λ₂·x₂ + λ₃·x₃)·H(pk, app_id) = msk·H(pk, app_id)`.
5. The coordinator outputs this wrong value as the valid `CKDOutput`. Any downstream consumer that relies on this derived key receives a cryptographically incorrect result with no indication of failure.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
