### Title
Single Malicious Participant Can Abort DKG/Reshare/Refresh for All Honest Parties via `broadcast_success` False Vote — (File: `src/dkg.rs`)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` is the final gate of the DKG, reshare, and refresh protocols. It requires **all** participants to broadcast `(true, session_id)`. A single malicious participant who broadcasts `(false, session_id)` causes every honest party to abort with `ProtocolError::AssertionFailed` and discard their already-computed key shares. This is a non-Byzantine-fault-tolerant check bolted onto an otherwise Byzantine-fault-tolerant protocol, and it can be triggered by any one participant at will.

---

### Finding Description

`do_keyshare` — the shared implementation of DKG, reshare, and refresh — calls `broadcast_success` as its final step (line 531):

```
broadcast_success(&mut chan, &participants, me, session_id).await?;
``` [1](#0-0) 

`broadcast_success` unconditionally broadcasts `(true, session_id)` on behalf of the calling party, then collects the votes of all other participants via `do_broadcast` (which internally uses the echo-broadcast protocol):

```rust
let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
``` [2](#0-1) 

After the broadcast completes, it checks that **every single entry** in the returned vote list is `(true, session_id)`:

```rust
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant seems to have failed its checks. Aborting Protocol!"
    ));
}
``` [3](#0-2) 

The echo-broadcast protocol (`reliable_broadcast_receive_all`) is designed to tolerate up to `f` Byzantine faults where `n ≥ 3f+1`. It guarantees consistent delivery — if a malicious participant broadcasts `(false, session_id)`, every honest party will receive exactly that value. The echo-broadcast protocol does **not** filter or suppress a consistently-broadcast false vote; it faithfully delivers it. [4](#0-3) 

The result is that the Byzantine-fault-tolerant broadcast layer correctly delivers the malicious `false` vote to all honest parties, and then the unanimity check at line 329 causes all honest parties to abort — discarding key shares they have already computed.

`do_keyshare` is invoked by both `do_keygen` and `do_reshare`, so the vulnerability covers DKG, reshare, and refresh: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**High — Permanent denial of key generation, reshare, and refresh for honest parties.**

A single malicious participant can repeatedly abort every DKG/reshare/refresh session. Honest parties complete all expensive cryptographic rounds (polynomial generation, commitment broadcasts, share distribution, share verification), reach the final step, and are then forced to discard all outputs. Because the attacker can repeat this indefinitely at zero marginal cost (they only need to send one false vote per session), honest parties can never successfully complete key generation or key rotation.

---

### Likelihood Explanation

Any participant enrolled in a DKG, reshare, or refresh session can trigger this. No special privilege, no prior key material, and no cryptographic capability is required — the attacker simply sends `(false, session_id)` instead of `(true, session_id)` in the final broadcast round. The attack is trivially repeatable and requires no coordination.

---

### Recommendation

Replace the unanimity check with a threshold-tolerant check. After the echo-broadcast delivers all votes, the protocol should proceed if at least `n - f` participants broadcast `(true, session_id)`, where `f` is the Byzantine fault tolerance bound already used by the echo-broadcast layer (`(n-1)/3`). Participants who broadcast `false` should be identified and excluded from the output participant set (or flagged), rather than causing a global abort. This mirrors the fault-tolerance guarantee already provided by the rest of the protocol.

---

### Proof of Concept

1. Start a DKG session with `n = 4` participants (threshold `f = 1`).
2. Participants `P1`, `P2`, `P3` are honest; `P4` is malicious.
3. All four participants complete rounds 1–5 of `do_keyshare` honestly: session IDs are exchanged, commitments are broadcast, secret shares are distributed and verified.
4. At line 531, `broadcast_success` is called. `P1`, `P2`, `P3` call `do_broadcast(…, (true, session_id))`. `P4` instead calls `do_broadcast(…, (false, session_id))`.
5. The echo-broadcast protocol delivers `P4`'s `(false, session_id)` consistently to `P1`, `P2`, and `P3`.
6. Each of `P1`, `P2`, `P3` reaches line 329, finds `boolean == false` for `P4`'s entry, and returns `Err(ProtocolError::AssertionFailed("A participant seems to have failed its checks. Aborting Protocol!"))`.
7. All honest parties discard their `KeygenOutput`. No usable key shares exist. `P4` repeats this in every subsequent session, permanently denying key generation. [7](#0-6)

async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                broadcast the wrong session id. Aborting Protocol!"
                .to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
}

pub async fn do_reshare<C: Ciphersuite>(
    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
    // either extract the share and linearize it or set it to zero
    let secret = old_signing_key
        .map(|x_i| {
            intersection
                .lagrange::<C>(me)
                .map(|lambda| lambda * x_i.to_scalar())
        })
        .transpose()?
        .unwrap_or_else(<C::Group as Group>::Field::zero);

    let old_reshare_package = Some((old_public_key, old_participants));
    let keygen_output = do_keyshare::<C>(
        chan,
        participants,
        me,
        threshold,
        secret,
        old_reshare_package,
        &mut rng,
    )
    .await?;

    Ok(keygen_output)

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
