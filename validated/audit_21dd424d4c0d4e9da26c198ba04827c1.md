### Title
XCC Router `add_batch_actions` Always Panics When `all-promise-actions` Feature Is Disabled, Permanently Freezing Attached NEAR - (File: `etc/xcc-router/src/lib.rs`)

### Summary
The XCC router contract's `add_batch_actions` function is not implemented in the default (production) build. When a user submits a cross-contract call containing a `SimpleNearPromise::Batch` action, the router panics unconditionally. Any NEAR that was already unwrapped from wNEAR and forwarded to the router as part of the XCC flow becomes permanently frozen.

### Finding Description

In `etc/xcc-router/src/lib.rs`, the `add_batch_actions` function has two conditional compilations: [1](#0-0) 

When the `all-promise-actions` feature is **not** enabled (the default production configuration), calling `add_batch_actions` unconditionally panics with `unimplemented!`. The full implementation only exists behind the feature flag: [2](#0-1) 

`add_batch_actions` is called from `recursive_promise_create` whenever a `SimpleNearPromise::Batch` variant is encountered: [3](#0-2) 

`recursive_promise_create` is invoked from `promise_create`, which is called by both the public `execute` and `execute_scheduled` methods: [4](#0-3) [5](#0-4) 

The user-controlled `PromiseArgs` (including `PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch(...)))`) is deserialized from the XCC precompile input and borsh-serialized into the args of the `execute` call dispatched to the router: [6](#0-5) 

Before the router's `execute` is called, the engine's `handle_precompile_promise` has already unwrapped the user's wNEAR into real NEAR and forwarded it to the router via `withdraw_wnear_to_router`: [7](#0-6) 

When the router panics, the NEAR already deposited into the router sub-account has no recovery path. The router exposes no general withdrawal function; `send_refund` only returns the fixed 2 NEAR storage-staking amount and requires the parent (Aurora engine) as caller with no failed promises: [8](#0-7) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who attaches NEAR to a batch XCC call (e.g., to fund a cross-contract transfer) has their wNEAR burned/transferred in the EVM execution phase. The engine then unwraps it to real NEAR and sends it to the router sub-account. When the router panics on `add_batch_actions`, the NEAR is permanently locked in the router with no recovery mechanism.

### Likelihood Explanation

**Medium.** The XCC precompile is a documented Aurora feature accessible to any EVM user. A user or integrating protocol that constructs a `PromiseArgs::Recursive` containing a `SimpleNearPromise::Batch` action — a valid and expected use case given the type system — will trigger this panic. The `all-promise-actions` feature is not enabled in the default build, as confirmed by the `unimplemented!` stub and its accompanying comment directing users to file a GitHub issue.

### Recommendation

Implement `add_batch_actions` unconditionally (remove the feature gate), or add an explicit input validation step in the XCC precompile and/or the router's `execute`/`execute_scheduled` methods that rejects `PromiseArgs::Recursive` containing `SimpleNearPromise::Batch` actions before any NEAR is transferred, returning a clean error instead of panicking after funds are already in the router.

### Proof of Concept

1. User crafts EVM calldata targeting the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a borsh-encoded `CrossContractCallArgs::Eager(PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch(...))))` and attaches NEAR (via wNEAR `transferFrom`).
2. The XCC precompile validates and emits a promise log targeting the router's `execute` method.
3. `handle_precompile_promise` in `engine/src/xcc.rs` calls `withdraw_wnear_to_router`, unwrapping the user's wNEAR to real NEAR and depositing it into the router sub-account.
4. The NEAR promise chain calls `execute` on the router with the batch `PromiseArgs`.
5. `execute` → `promise_create` → `recursive_promise_create` → `add_batch_actions` → `unimplemented!` panic.
6. The NEAR deposited in step 3 is permanently frozen in the router sub-account with no recovery path.
