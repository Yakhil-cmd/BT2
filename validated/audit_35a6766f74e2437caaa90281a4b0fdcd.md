### Title
Hardcoded NEAR Gas Amounts in Exit Precompiles Can Permanently Break Token Bridging - (`engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` and `ExitToEthereum` precompiles attach hardcoded NEAR gas amounts to every cross-contract call they schedule. If NEAR protocol gas costs increase, these calls will fail with insufficient gas after the user's tokens have already been burned or locked on the Aurora side, causing a temporary (or permanent, absent a timely upgrade) freeze of bridged funds.

### Finding Description

In `engine-precompiles/src/native.rs`, the `costs` module defines four compile-time NEAR gas constants:

```rust
// TODO(#483): Determine the correct amount of gas
pub(super) const EXIT_TO_NEAR_GAS: EthGas = EthGas::new(0);
// TODO(#483): Determine the correct amount of gas
pub(super) const EXIT_TO_ETHEREUM_GAS: EthGas = EthGas::new(0);
pub(super) const FT_TRANSFER_GAS: NearGas = NearGas::new(10_000_000_000_000);
pub(super) const FT_TRANSFER_CALL_GAS: NearGas = NearGas::new(70_000_000_000_000);
pub(super) const EXIT_TO_NEAR_CALLBACK_GAS: NearGas = NearGas::new(10_000_000_000_000);
// TODO(#332): Determine the correct amount of gas
pub(super) const WITHDRAWAL_GAS: NearGas = NearGas::new(100_000_000_000_000);
``` [1](#0-0) 

These constants are used directly as `attached_gas` in the NEAR cross-contract call promises constructed inside `ExitToNear::run()`:

```rust
let attached_gas = if method == "ft_transfer_call" {
    costs::FT_TRANSFER_CALL_GAS
} else {
    costs::FT_TRANSFER_GAS
};
let transfer_promise = PromiseCreateArgs {
    ...
    attached_gas,
};
``` [2](#0-1) 

And in `ExitToEthereum::run()`:

```rust
let withdraw_promise = PromiseCreateArgs {
    ...
    attached_gas: costs::WITHDRAWAL_GAS,
};
``` [3](#0-2) 

The callback promise for the `error_refund` path also uses a hardcoded constant:

```rust
attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,
``` [4](#0-3) 

These are identical in nature to Solidity's `transfer` stipend: a fixed gas budget baked into the contract bytecode that cannot be adjusted without a full contract upgrade. The two `TODO` comments (`#483`, `#332`) in the source explicitly acknowledge that the correct values have not been determined.

### Impact Explanation

When a user calls `ExitToNear` (precompile `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) or `ExitToEthereum` (precompile `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`), the EVM side of the bridge immediately burns or locks the user's tokens. The NEAR-side transfer (`ft_transfer`, `ft_transfer_call`, or `withdraw`) is then scheduled as a cross-contract call with the hardcoded gas budget. If that budget is insufficient — because NEAR protocol gas costs have increased — the NEAR-side call fails. The tokens are gone from the EVM state but never arrive at the destination. Without a timely contract upgrade to raise the constants, the funds are frozen.

**Impact: High — Temporary (potentially permanent) freezing of user funds.**

### Likelihood Explanation

NEAR protocol has repriced operations in the past. The `TODO` annotations in the source code confirm the developers themselves are uncertain whether the current values are correct. The value for `FT_TRANSFER_GAS` is described as "determined experimentally," and `WITHDRAWAL_GAS` carries an open TODO. Any future NEAR protocol upgrade that increases the cost of storage reads, cross-contract dispatch, or fungible-token operations could push the actual gas requirement above these ceilings. Any unprivileged EVM user who calls either exit precompile after such an upgrade would lose their tokens.

### Recommendation

Replace the compile-time constants with runtime-configurable values stored in contract state (analogous to replacing `transfer` with `call` in Solidity). Provide owner-only setter functions so the gas budgets can be updated without a full contract redeployment when NEAR gas costs change. Until that is done, at minimum resolve the open TODOs by benchmarking the actual gas consumption of each cross-contract call path and setting conservative upper bounds.

### Proof of Concept

1. User holds Aurora ETH or an ERC-20 token on Aurora.
2. User calls the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with a valid NEAR recipient.
3. The precompile burns the user's tokens on the EVM side and schedules an `ft_transfer` NEAR cross-contract call with `attached_gas = FT_TRANSFER_GAS` (10 TGas).
4. A NEAR protocol upgrade has increased the minimum gas required for `ft_transfer` beyond 10 TGas.
5. The NEAR-side call fails with `GasExceeded`; the callback (if `error_refund` is not enabled, or if `EXIT_TO_NEAR_CALLBACK_GAS` is also insufficient) does not execute.
6. The user's tokens are permanently lost from the EVM state and never credited on NEAR.

The same scenario applies to `ExitToEthereum` via `WITHDRAWAL_GAS` at line 982, and to the callback path via `EXIT_TO_NEAR_CALLBACK_GAS` at line 480. [1](#0-0) [2](#0-1) [3](#0-2)
