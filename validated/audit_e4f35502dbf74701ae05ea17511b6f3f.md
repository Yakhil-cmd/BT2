### Title
`SwapAllowlistExtension` checks router address instead of actual swapper, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router address**, not the actual user. The extension therefore checks the router against the allowlist, not the real swapper. Any user can bypass a per-user allowlist by going through the router if the router is allowlisted, or allowlisted users are silently blocked from using the router if it is not.

---

### Finding Description

`SwapAllowlistExtension` is a production periphery extension designed to restrict which addresses may trade on a curated pool.

Its `beforeSwap` hook receives `sender` as the first argument:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

`sender` is populated by `ExtensionCalling._beforeSwap`, which passes `msg.sender` of the pool's `swap()` call:

```solidity
// metric-core/contracts/MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
``` [2](#0-1) 

```solidity
// metric-core/contracts/ExtensionCalling.sol
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, ...)   // sender = pool's msg.sender
    )
);
``` [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle(...)`, the router calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

The router stores the original user's address in transient callback context for payment settlement, but **never forwards it to the pool as the swap initiator**. The pool sees `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` — the router's allowlist status — not the actual user's.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the position owner, explicitly passed by the liquidity adder), not `sender`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [5](#0-4) 

The swap extension has no equivalent mechanism to receive the real user identity through the router path.

---

### Impact Explanation

**Scenario A — Allowlist bypass (High):** Pool admin allowlists Alice and Bob individually, and also allowlists the router so those users can trade via the standard periphery. Charlie (not allowlisted) calls `router.exactInputSingle(...)`. The extension sees `sender = router`, finds the router is allowlisted, and permits the swap. Charlie bypasses the curated pool's access control entirely.

**Scenario B — Broken core functionality (High):** Pool admin allowlists Alice and Bob but does not allowlist the router. Alice calls `router.exactInputSingle(...)`. The extension sees `sender = router`, finds the router is not allowlisted, and reverts. Alice cannot use the primary swap interface even though she is individually permitted. The pool's swap flow is broken for all allowlisted users who rely on the router.

Both outcomes are direct consequences of the wrong actor being checked in the guard.

---

### Likelihood Explanation

The router is the primary user-facing swap interface. Any pool that deploys `SwapAllowlistExtension` to restrict trading will encounter this mismatch the moment a user (or the admin) attempts a router-mediated swap. The bypass requires no special privileges — any address can call the router. The broken-functionality path requires no attacker at all; it manifests on the first legitimate router swap.

---

### Recommendation

The pool must convey the original user's identity to the extension. Two approaches:

1. **Explicit swapper parameter**: Add an optional `swapper` field to the swap call that the router populates with `msg.sender` before calling the pool. The pool passes this to `_beforeSwap` instead of (or alongside) its own `msg.sender`.

2. **Extension-data convention**: Define a standard encoding where the router prepends the real user address to `extensionData`, and `SwapAllowlistExtension` decodes it when `sender` is a known router. This is opt-in and backward-compatible but requires router cooperation.

Either way, the extension must check the economically attributed actor (the user who initiated and will pay for the swap), not the intermediary contract that called the pool.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is allowlisted.
3. Pool admin calls `setAllowedToSwap(pool, address(router), true)` — router is allowlisted so Alice can use it.
4. Charlie (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(msg.sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true`.
8. Charlie's swap succeeds. The allowlist is bypassed.
