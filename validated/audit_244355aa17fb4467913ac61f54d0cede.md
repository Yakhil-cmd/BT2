### Title
`protocolUnpausePool` Sets Pause Level to `1` (Admin-Paused) Instead of `0` (Active), Leaving Pool Permanently Paused After Protocol Unpause - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` contains a copy/paste error: it calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). After the factory owner invokes this function to lift a protocol pause, the pool remains paused at level 1, blocking all swaps and liquidity operations indefinitely unless the pool admin separately calls `unpausePool`.

### Finding Description

The three pause levels are:
- `0` = active
- `1` = admin-paused
- `2` = protocol-paused

`protocolPausePool` correctly transitions to level `2`: [1](#0-0) 

`protocolUnpausePool` is supposed to reverse that, but instead transitions to level `1`: [2](#0-1) 

Both the guard (`revert InvalidPauseTransition(cur, 1)`) and the state write (`setPause(1)`) use `1` where `0` is required. This is a direct copy/paste analog of the external splay-tree bug: the wrong variable/literal was carried over from the sibling function.

The `swap` function is gated by `whenNotPaused`, which blocks execution at any non-zero pause level: [3](#0-2) 

`addLiquidity` and `removeLiquidity` are similarly gated. After `protocolUnpausePool` sets level to `1`, all three remain blocked.

### Impact Explanation

Every call to `protocolUnpausePool` leaves the pool stuck at admin-paused (level 1). Swaps, liquidity additions, and liquidity removals are all blocked. LPs cannot withdraw funds; traders cannot execute. The pool is functionally inoperable after any protocol-pause/unpause cycle. This matches the allowed impact gate: **broken core pool functionality causing unusable withdraw/swap/liquidity flows**.

### Likelihood Explanation

The trigger is the factory owner calling `protocolUnpausePool` — a routine, legitimate governance action. No adversarial setup is required. Every protocol-pause event followed by an unpause attempt will reproduce the bug deterministically.

### Recommendation

Change `protocolUnpausePool` to target level `0`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-   if (cur != 2) revert InvalidPauseTransition(cur, 1);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   if (cur != 2) revert InvalidPauseTransition(cur, 0);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

### Proof of Concept

1. Deploy pool (pause level = 0, active).
2. Factory owner calls `protocolPausePool(pool)` → level becomes `2`.
3. Factory owner calls `protocolUnpausePool(pool)` → function checks `cur != 2` (passes), then calls `setPause(1)`.
4. Pool pause level is now `1` (admin-paused), not `0`.
5. Any call to `swap(...)` reverts via `whenNotPaused`.
6. Any call to `addLiquidity(...)` or `removeLiquidity(...)` reverts via the same guard.
7. Pool is inoperable; LPs cannot withdraw principal. [2](#0-1)
