### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Bridge Insolvency and Permanent Fund Freeze - (`engine/src/engine.rs`)

### Summary

`receive_erc20_tokens` mints ERC-20 tokens equal to `args.amount` — the amount the NEP-141 token contract *claims* was transferred — without verifying the actual balance change in Aurora's account. For fee-on-transfer NEP-141 tokens, Aurora receives fewer tokens than it mints, making the bridge insolvent. The last users to withdraw will find Aurora cannot fulfill the `ft_transfer` call because it does not hold enough NEP-141 tokens.

### Finding Description

The NEP-141 to ERC-20 bridge flow is:

1. A user calls `ft_transfer_call(receiver_id=aurora, amount=N, msg=<evm_address>)` on a NEP-141 token contract.
2. The NEP-141 contract transfers tokens to Aurora (for a fee-on-transfer token: `N - fee` tokens actually arrive).
3. The NEP-141 contract calls `ft_on_transfer(sender_id, amount=N, msg)` on Aurora — passing the *gross* amount, not the net received amount.
4. Aurora's `ft_on_transfer` handler dispatches to `receive_erc20_tokens`.

Inside `receive_erc20_tokens`:

```rust
let amount = args.amount.as_u128();   // ← gross amount N, not actual received N-fee
// ...
setup_receive_erc20_tokens_input(&recipient, amount),  // mints N ERC-20 tokens
``` [1](#0-0) 

`setup_receive_erc20_tokens_input` encodes a `mint(recipient, amount)` call to the `EvmErc20` contract:

```rust
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);
    [selector, tail.as_slice()].concat()
}
``` [2](#0-1) 

The `EvmErc20.mint` function mints exactly `amount` ERC-20 tokens:

```solidity
function mint(address account, uint256 amount) public onlyAdmin {
    _mint(account, amount);
}
``` [3](#0-2) 

On withdrawal, `EvmErc20.withdrawToNear` burns the ERC-20 amount and calls the `ExitToNear` precompile with that same amount:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    // ... calls ExitToNear precompile with `amount`
}
``` [4](#0-3) 

The `ExitToNear` precompile schedules an `ft_transfer` of `exit_params.amount` NEP-141 tokens from Aurora's account to the recipient:

```rust
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()
),
``` [5](#0-4) 

**The accounting gap**: Aurora minted `N` ERC-20 tokens but only holds `N - fee` NEP-141 tokens. The ERC-20 total supply exceeds the NEP-141 backing. When the last `fee` worth of ERC-20 tokens are withdrawn, the `ft_transfer` call on the NEP-141 contract will fail because Aurora's balance is insufficient.

### Impact Explanation

**Critical — Insolvency and Permanent Fund Freeze.**

- The ERC-20 total supply permanently exceeds the NEP-141 balance held by Aurora for that token.
- The last users to call `withdrawToNear` will have their ERC-20 tokens burned (irreversible on-chain) but the corresponding `ft_transfer` promise will fail because Aurora does not hold enough NEP-141 tokens.
- Those users permanently lose their funds with no recovery path — the ERC-20 tokens are already burned and the NEP-141 tokens are not transferred.
- The deficit compounds with every deposit of a fee-on-transfer token.

### Likelihood Explanation

**Medium.** The NEP-141 standard does not prohibit fee-on-transfer tokens. Any unprivileged user can deploy a fee-on-transfer NEP-141 token, call `deploy_erc20_token` to register it on Aurora, and then bridge tokens via `ft_transfer_call`. No admin action or special privilege is required. The `ft_on_transfer` entry point is a public NEAR contract method callable by any NEP-141 token contract. [6](#0-5) 

### Recommendation

In `receive_erc20_tokens`, measure the actual balance change rather than trusting `args.amount`:

```
uint256 balBefore = nep141.balanceOf(aurora);
// ft_transfer_call executes
uint256 balAfter = nep141.balanceOf(aurora);
uint256 actualReceived = balAfter - balBefore;
// mint actualReceived ERC-20 tokens, not args.amount
```

At the NEAR/Rust level this means: before minting, query Aurora's NEP-141 balance for the token, compare it to the stored pre-transfer balance, and mint only the difference. Alternatively, document and enforce that only non-fee-on-transfer NEP-141 tokens may be bridged, and add a validation step in `deploy_erc20_token` or `ft_on_transfer` to reject known fee-on-transfer tokens.

### Proof of Concept

1. Deploy a NEP-141 token that charges a 1% fee on every transfer (deducted from the receiver's credited amount, but passes the gross `amount` to `ft_on_transfer`).
2. Call `deploy_erc20_token` on Aurora to register the NEP-141 → ERC-20 mapping.
3. Call `ft_transfer_call(receiver_id=aurora, amount=1000, msg=<evm_address>)`.
   - Aurora receives 990 NEP-141 tokens.
   - `ft_on_transfer` is called with `amount=1000`.
   - Aurora mints 1000 ERC-20 tokens. [7](#0-6) 
4. Repeat until the cumulative deficit is significant.
5. All ERC-20 holders attempt `withdrawToNear`.
   - The first 99% of withdrawals succeed.
   - The last 1% (equal to the cumulative fee) have their ERC-20 tokens burned by `EvmErc20.withdrawToNear` [4](#0-3)  but the `ft_transfer` promise fails — Aurora's NEP-141 balance is zero.
   - Those users permanently lose their funds.

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
