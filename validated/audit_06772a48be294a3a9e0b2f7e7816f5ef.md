### Title
`setPoolAdminFeeDestination` Redirects All Accrued Admin Fees to New Destination Without Prior Collection — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first flushing accrued fees at the old destination. Every other fee-mutating path in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before changing any fee parameter. Because `collectPoolFees` reads `poolAdminFeeDestination[pool]` at collection time, all fees that accrued before the destination change are silently redirected to the new address on the next collection call — a direct analog to the external bug where state is deleted while pending operations remain unprocessed.

---

### Finding Description

Two fee-changing functions in `MetricOmmPoolFactory` explicitly flush accrued fees before modifying state:

**`setPoolAdminFees`** calls `collectFees` with the current destination before updating rates: [1](#0-0) 

**`setPoolProtocolFee`** does the same before updating protocol components: [2](#0-1) 

**`setPoolAdminFeeDestination`** does neither — it simply overwrites the destination with no prior collection: [3](#0-2) 

The pool's `collectFees` function receives `adminFeeDestination_` as a live parameter from the factory, which reads `poolAdminFeeDestination[pool]` at the moment of collection: [4](#0-3) 

All fees that accrued before the destination change — both the spread-fee surplus tokens (`balance - binTotals - notionalAccumulators`) and the explicit `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` accumulators — are sent to the new destination on the next `collectPoolFees` call: [5](#0-4) 

The permissionless `collectPoolFees` on the factory reads the current (already-changed) destination: [6](#0-5) 

---

### Impact Explanation

The old fee destination loses its entire accrued fee balance. In the most direct scenario involving the two-step admin transfer:

1. Old admin A operates the pool with `poolAdminFeeDestination[pool] = treasuryA`; fees accrue over many swaps.
2. Admin A calls `proposePoolAdminTransfer(pool, adminB)`.
3. Admin B calls `acceptPoolAdmin(pool)` — B is now the pool admin. [7](#0-6) 
4. Admin B immediately calls `setPoolAdminFeeDestination(pool, treasuryB)` — no collection occurs, `poolAdminFeeDestination[pool]` is now `treasuryB`.
5. Anyone calls `collectPoolFees(pool)` — all fees accrued under admin A are transferred to `treasuryB`.
6. `treasuryA` receives nothing despite having earned those fees.

The loss is proportional to swap volume since the last collection and the admin fee rate. Both spread-fee surplus and notional-fee accumulators are affected.

---

### Likelihood Explanation

The trigger is a valid semi-trusted action: any pool admin (including a newly accepted admin after a transfer) can call `setPoolAdminFeeDestination` at any time with no timelock and no cap check. The inconsistency with `setPoolAdminFees` and `setPoolProtocolFee` — which both collect first — indicates this is a design flaw rather than intentional behavior. The scenario is realistic whenever admin roles are transferred between distinct parties (e.g., protocol handover, DAO governance change, or sale of a pool's admin rights).

---

### Recommendation

Add a `collectFees` call before updating `poolAdminFeeDestination[pool]`, consistent with the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the OLD destination before switching
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // <-- old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

```
1. Deploy pool: admin = A, adminFeeDestination = treasuryA, adminSpreadFeeE6 = 5000
2. Execute N swaps → spread-fee surplus and notional accumulators grow in the pool
3. A calls proposePoolAdminTransfer(pool, B)
4. B calls acceptPoolAdmin(pool)          // B is now pool admin; no fees collected
5. B calls setPoolAdminFeeDestination(pool, treasuryB)  // no collectFees called
6. Anyone calls collectPoolFees(pool)
   → factory reads poolAdminFeeDestination[pool] == treasuryB
   → all accrued admin fees (earned under A) transferred to treasuryB
7. treasuryA balance delta = 0  (loss of all accrued admin fees)
   treasuryB balance delta = full accrued admin fee amount
```
