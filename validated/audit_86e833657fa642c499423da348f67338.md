### Title
XCC Router Version Not Recorded on Callback Failure Permanently Freezes Funds - (`engine/src/xcc.rs`)

### Summary

In `handle_precompile_promise` and `fund_xcc_sub_account`, when a new XCC router sub-account is deployed for a user's EVM address, the engine schedules a `factory_update_address_version` callback to record the deployment version in storage. This callback calls `require_running` before checking the promise result. If the engine is paused between the deploy batch succeeding and the callback executing, `require_running` causes the callback to fail, leaving the router deployed on-chain but the engine's version record absent (`None`). Every subsequent XCC attempt for that address then tries to re-create the already-existing account, permanently failing. The 2 NEAR (`STORAGE_AMOUNT`) transferred to the router sub-account is permanently frozen.

### Finding Description

In `handle_precompile_promise` (`engine/src/xcc.rs`), when `AddressVersionStatus::DeployNeeded { create_needed: true }`, the engine constructs a batch promise: [1](#0-0) 

This batch transfers `STORAGE_AMOUNT` (2 NEAR) to the new sub-account and deploys the router. A callback is then attached: [2](#0-1) 

In `factory_update_address_version` (`engine/src/contract_methods/xcc.rs`), the function calls `require_running` **before** checking the promise result: [3](#0-2) 

If the engine is paused (a legitimate admin action) between the deploy batch succeeding and the callback executing, `require_running` returns an error. The callback fails, `set_code_version_of_address` is never called, and the engine's storage still shows `None` for that address's version.

The `AddressVersionStatus::new` logic maps `None` version to `DeployNeeded { create_needed: true }`: [4](#0-3) 

Every subsequent XCC call for that address will attempt `CreateAccount` again. Since the account already exists, the NEAR runtime rejects the batch. The callback is then invoked with `Some(false)` from `promise_result_check`, causing `factory_update_address_version` to return `ERR_ROUTER_DEPLOY_FAILED` — the version is still never set. This is an unrecoverable loop without a contract upgrade.

The same pattern exists in `fund_xcc_sub_account`: [5](#0-4) 

### Impact Explanation

- The `STORAGE_AMOUNT` = 2 NEAR transferred to the router sub-account during the deploy batch is permanently frozen. The router's `execute` and `schedule` methods enforce `require_caller(&parent)`, so only the engine can call them — and the engine will never successfully reach that point for this address again.
- The user's XCC functionality for that EVM address is permanently disabled.
- No recovery path exists without a contract upgrade, because `factory_update_address_version` requires `assert_private_call()` and cannot be invoked directly.

**Impact: High — Permanent freezing of funds (2 NEAR per affected address) and permanent loss of XCC capability.**

### Likelihood Explanation

- The engine being paused is a routine admin operation (maintenance windows, upgrades). The CHANGES.md shows the engine has a pause mechanism actively used.
- The timing window between the deploy batch completing and the callback executing is a single NEAR block boundary — small but real in production.
- A secondary trigger: if `VERSION_UPDATE_GAS` is misconfigured and insufficient for the callback's execution (storage write + hashchain update), the callback fails due to out-of-gas without any admin involvement.

**Likelihood: Low — Requires specific timing with an engine pause or a gas misconfiguration, but both are realistic production scenarios.**

### Recommendation

Remove `require_running` from `factory_update_address_version`, or move it to execute only when the promise result is `Some(false)` (i.e., only gate the version-write on the deploy result, not on the engine's running state). Since this is a private callback that must always complete to maintain consistency, it should not be gated on the engine's pause state. Alternatively, add a recovery path (e.g., an owner-callable function) that can set the version for an address whose account already exists on-chain.

### Proof of Concept

1. User calls the XCC precompile for the first time (no router deployed for their address).
2. `handle_precompile_promise` detects `AddressVersionStatus::DeployNeeded { create_needed: true }`.
3. Engine schedules: `[CreateAccount + Transfer(2 NEAR) + DeployContract + Initialize]` → callback `factory_update_address_version`.
4. The deploy batch executes successfully: router sub-account created, 2 NEAR transferred, contract deployed.
5. Admin pauses the engine for maintenance (legitimate operation).
6. `factory_update_address_version` callback executes; `require_running` fails → callback returns error → `set_code_version_of_address` never called.
7. Engine storage: `get_code_version_of_address(user_address)` = `None`.
8. Admin unpauses the engine. User retries XCC.
9. `AddressVersionStatus::new` sees `None` → returns `DeployNeeded { create_needed: true }`.
10. Engine schedules `CreateAccount` again → NEAR runtime rejects (account exists) → batch fails.
11. `factory_update_address_version` called with `promise_result_check() = Some(false)` → returns `ERR_ROUTER_DEPLOY_FAILED`.
12. Version remains `None`. Step 9–11 repeats forever. 2 NEAR permanently frozen in router sub-account; user permanently unable to use XCC for this address.
