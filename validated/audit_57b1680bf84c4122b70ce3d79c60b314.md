### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Rewards - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over all balance-change entries in a pool member's `pool_member_epoch_balance` trace since their last reward claim. A pool member who modifies their delegation balance across many epochs without claiming rewards will accumulate a large trace, eventually causing `claim_rewards` to revert due to gas exhaustion, permanently freezing their unclaimed yield.

---

### Finding Description

The `calculate_rewards` internal function iterates over the `pool_member_epoch_balance` trace for a given pool member: [1](#0-0) 

The code itself explicitly acknowledges the risk:

> `// **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.`

The trace (`pool_member_epoch_balance`) grows by one entry per epoch in which the pool member modifies their balance. This happens via:

- `enter_delegation_pool` → calls `set_member_balance`
- `add_to_delegation_pool` → calls `increase_member_balance` → `set_member_balance`
- `exit_delegation_pool_intent` → calls `set_member_balance`
- `enter_delegation_pool_from_staking_contract` → calls `set_member_balance` or `increase_member_balance` [2](#0-1) 

Each call to `set_member_balance` inserts a new entry keyed at `current_epoch + K`. Because the key differs each epoch, repeated balance modifications across distinct epochs each append a new entry to the trace. [2](#0-1) 

The loop in `calculate_rewards` starts from `entry_to_claim_from` (a stored index updated only when `claim_rewards` is successfully called) and iterates forward to the current trace length: [3](#0-2) 

If a pool member never calls `claim_rewards` (or defers it), `entry_to_claim_from` stays at its initial value while the trace grows without bound. When `claim_rewards` is eventually called, the loop must traverse every accumulated entry.

`calculate_rewards` is invoked from two public-facing paths:

1. `claim_rewards` (lines 349–355) — the only way to withdraw accumulated rewards.
2. `pool_member_info_v1` (lines 532–538) — a view function also calling `calculate_rewards`. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

If the `pool_member_epoch_balance` trace grows large enough that the loop in `calculate_rewards` exceeds the Starknet transaction gas limit, every call to `claim_rewards` will revert. Because `claim_rewards` is the only mechanism to withdraw accumulated pool rewards, the pool member's unclaimed yield becomes permanently frozen and unrecoverable. This matches the **High** impact category: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The trace grows by at most one entry per epoch per pool member. A pool member who actively adjusts their delegation (e.g., calling `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch) and defers claiming rewards for a long period will accumulate a proportionally large trace. Given that epochs are protocol-defined time windows and the protocol is designed for long-term staking, a sufficiently active and patient pool member (or one who simply forgets to claim) can reach a trace size that causes gas exhaustion. The likelihood is **Medium**: it requires sustained activity over many epochs without claiming, but no privileged access or external dependency is needed — any pool member can trigger this for themselves.

---

### Recommendation

1. **Cap the loop iterations**: Introduce a maximum number of entries processed per `claim_rewards` call (e.g., process at most `MAX_ENTRIES` per call and allow partial claims, advancing `entry_to_claim_from` incrementally).
2. **Force reward settlement on balance changes**: Before inserting a new entry into the trace (in `set_member_balance`), require or automatically settle pending rewards up to the current epoch, keeping `entry_to_claim_from` current and bounding the loop to a small constant.
3. **Document and enforce a maximum trace depth**: Add a hard cap on the number of unclaimed balance-change entries (e.g., revert `add_to_delegation_pool` if the unclaimed trace depth exceeds a threshold), forcing users to claim before accumulating further.

---

### Proof of Concept

1. Pool member Alice calls `enter_delegation_pool` at epoch 1. Trace length = 1, `entry_to_claim_from = 0`.
2. Alice calls `add_to_delegation_pool` once per epoch for epochs 2 through N (e.g., N = 10,000), never calling `claim_rewards`. Each call inserts a new entry at `current_epoch + K`, growing the trace to length N.
3. At epoch N+1, Alice (or anyone) calls `claim_rewards(alice)`. This invokes `calculate_rewards` with `entry_to_claim_from = 0` and `pool_member_trace_length = N`.
4. The `while` loop at line 859 iterates N times. For large N, this exceeds the Starknet block gas limit, causing the transaction to revert.
5. Every subsequent call to `claim_rewards(alice)` also reverts. Alice's accumulated rewards are permanently frozen. [6](#0-5)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
