### Title
Deprecated `.transfer()` in `WSEI.withdraw()` Permanently Locks ETH for Contract Callers — (`contracts/src/WSEI.sol`)

---

### Summary
`WSEI.sol`'s `withdraw()` function uses the deprecated `payable(msg.sender).transfer(wad)` pattern, which forwards only 2300 gas. Any smart contract whose `receive`/`fallback` function consumes more than 2300 gas will be permanently unable to withdraw its deposited ETH, locking funds in the WSEI contract with no recovery path.

---

### Finding Description
`WSEI.sol` is the official Wrapped SEI contract in the sei-chain repository. Its `withdraw()` function decrements the caller's balance and then attempts to send ETH using `.transfer()`:

```solidity
// contracts/src/WSEI.sol, line 27–32
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    payable(msg.sender).transfer(wad);   // ← deprecated, 2300 gas stipend
    emit Withdrawal(msg.sender, wad);
}
```

Since the EIP-1884 (Istanbul) hardfork, `SLOAD` costs 800 gas instead of 200. Many standard contract patterns (multisigs, DeFi vaults, proxy contracts) require more than 2300 gas in their `receive`/`fallback` to execute even a single storage read. When such a contract calls `withdraw()`, `.transfer()` reverts the entire transaction — including the balance decrement — so the WSEI balance is never reduced and the ETH remains locked in the contract forever.

A secondary instance exists in `MultiSender.sol` (lines 19, 41) using `.send()`, but that only causes batch-level DoS (transaction reverts, no permanent lock).

---

### Impact Explanation
Any smart contract that:
1. Calls `deposit()` or sends ETH directly to WSEI (triggering the `fallback`/`receive`), and
2. Has a `receive`/`fallback` that uses >2300 gas (e.g., emits an event, reads storage, calls another contract)

…will have its ETH permanently frozen inside the WSEI contract. There is no admin recovery function. The ETH cannot be retrieved by any means.

Scope match: **fund freeze** — severity scales with the amount deposited. Deposits ≥ $5k qualify as **Critical**.

---

### Likelihood Explanation
- The path is fully unprivileged: `deposit()` and `withdraw()` are public with no access control.
- Modern DeFi integrations (yield aggregators, multisigs, proxy wallets) routinely use contract accounts with non-trivial fallback logic.
- WSEI is the canonical wrapped-native token on Sei EVM; it is a natural integration target for DeFi protocols.
- The failure is silent from the depositor's perspective until withdrawal is attempted.

---

### Recommendation
Replace `.transfer()` with a low-level `.call()` and check the return value, following the Checks-Effects-Interactions pattern:

```solidity
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    (bool success, ) = payable(msg.sender).call{value: wad}("");
    require(success, "WSEI: ETH transfer failed");
    emit Withdrawal(msg.sender, wad);
}
```

Apply the same fix to `MultiSender.sol` lines 19 and 41 (replace `.send()` with `.call()`).

---

### Proof of Concept

```solidity
// Attacker/victim contract
contract VaultWithEvent {
    WSEI wsei = WSEI(<deployed_address>);

    // This fallback uses >2300 gas (emits event + storage write)
    uint256 public receiveCount;
    receive() external payable {
        receiveCount++;  // SSTORE: ~5000 gas
    }

    function depositAndWithdraw() external payable {
        // Step 1: deposit succeeds — ETH enters WSEI
        wsei.deposit{value: msg.value}();

        // Step 2: withdraw always reverts — ETH permanently locked
        wsei.withdraw(msg.value);  // reverts: out of gas in .transfer()
    }
}
```
