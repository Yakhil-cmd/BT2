### Title
Partial State Transition in `transfer_from_treasury` — Treasury Decremented Without Recipient Credit on Second Balance Update Failure - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`transfer_from_treasury` performs two sequential, non-atomic balance updates: it first decrements the treasury's balance, then increments the recipient's balance. If the first update succeeds but the second fails (e.g., `MintingBalanceOverflow`), the treasury is permanently decremented with no corresponding credit to the recipient and no rollback of the first update. This is the direct ZKsync OS analog of the `Bet::cancel` partial-transfer bug.

---

### Finding Description

`transfer_from_treasury` in `process_l1_transaction.rs` implements a two-step token movement:

```rust
// Step 1 — subtract from treasury
system.io.update_account_nominal_token_balance(
    treasury_address, nominal_token_value,
    true,  // subtract
    ...
)?;  // → TreasuryTransferFailed on error

// Step 2 — add to recipient
system.io.update_account_nominal_token_balance(
    to, nominal_token_value,
    false, // add
    ...
)?;  // → MintingBalanceOverflow on error
``` [1](#0-0) 

If Step 1 succeeds and Step 2 returns `MintingBalanceOverflow`, the `?` operator propagates the error upward. There is **no rollback of Step 1**. The treasury has been decremented by `nominal_token_value` but the recipient has not been credited.

This function is called from `mint_base_token`, which is invoked twice in the post-execution phase of `process_l1_transaction` — **outside any rollback frame**:

1. **Operator fee payment** (lines 290–309): `mint_base_token(pay_to_operator, coinbase, ...)`
2. **Refund recipient payment** (lines 336–359): `mint_base_token(to_refund_recipient, refund_recipient, ...)` [2](#0-1) [3](#0-2) 

Compare this with the L2 transaction flow, where `refund_and_commit_fee` is wrapped in an explicit rollback frame that undoes all balance changes if any step fails: [4](#0-3) 

No equivalent frame exists around the post-execution L1 token operations.

Additionally, `mint_base_token` calls `notify_l2_asset_tracker` **before** `transfer_from_treasury`. If the asset-tracker notification succeeds (its frame is committed) but `transfer_from_treasury` subsequently fails, the L2AssetTracker's `totalSuccessfulDepositsFromL1` counter is incremented for a token movement that never fully completed, creating a secondary accounting inconsistency. [5](#0-4) 

---

### Impact Explanation

If Step 2 of `transfer_from_treasury` fails after Step 1 succeeds:

- The treasury (`BASE_TOKEN_HOLDER_ADDRESS`) is permanently decremented by `nominal_token_value`.
- The intended recipient (operator or refund recipient) receives nothing.
- The error propagates as a fatal `BootloaderSubsystemError`, halting block processing.
- Because L1 transactions cannot be invalidated (they are committed on L1 in the priority queue), the block cannot be finalized and the chain stalls.
- The L2AssetTracker's accounting diverges from actual balances.

This matches the external report's impact class: a multi-step operation partially succeeds, state is committed as if fully complete, and there is no recovery path.

---

### Likelihood Explanation

The `MintingBalanceOverflow` path requires the recipient's balance to satisfy `balance + nominal_token_value > U256::MAX`. With real-world token amounts this is practically unreachable. However:

- The **refund recipient** is `transaction.reserved[1]`, a field set by the L1 transaction submitter — an unprivileged external actor.
- If a future protocol change introduces a minting mechanism that can produce near-`U256::MAX` balances (e.g., a separate minting bug), this path becomes directly exploitable.
- The structural defect (non-atomic two-step update without rollback) is present regardless of current token supply constraints.

Likelihood is **low** under current token economics, but the root cause is a latent structural vulnerability reachable via an externally controlled field.

---

### Recommendation

Make `transfer_from_treasury` atomic by wrapping both balance updates in a single snapshot frame, or by reversing Step 1 before returning the error if Step 2 fails:

```rust
// Option A: reverse Step 1 on Step 2 failure
let step1_result = system.io.update_account_nominal_token_balance(
    treasury_address, nominal_token_value, true, ...
)?;

system.io.update_account_nominal_token_balance(
    to, nominal_token_value, false, ...
).map_err(|e| {
    // Reverse the treasury decrement before propagating
    let _ = system.io.update_account_nominal_token_balance(
        treasury_address, nominal_token_value, false, ...
    );
    // map error ...
})?;
```

Or, preferably, wrap the entire post-execution fee/refund sequence for L1 transactions in a global rollback frame (analogous to the `refund_rollback_handle` used for L2 transactions), so that any failure in the sequence atomically reverts all partial balance changes.

---

### Proof of Concept

1. Attacker submits an L1→L2 priority transaction with `reserved[1]` (refund recipient) set to address `A`.
2. Address `A` has balance `B` such that `B + to_refund_recipient > U256::MAX`.
3. `process_l1_transaction` executes:
   - Operator fee mint succeeds: treasury decremented by `pay_to_operator`, coinbase credited.
   - Refund mint begins: `notify_l2_asset_tracker` succeeds (L2AssetTracker counter incremented).
   - `transfer_from_treasury` Step 1: treasury decremented by `to_refund_recipient` — **succeeds**.
   - `transfer_from_treasury` Step 2: `A.balance + to_refund_recipient` overflows — **fails** with `MintingBalanceOverflow`.
4. Error propagates via `?` through `mint_base_token` → `process_l1_transaction` → block runner.
5. Block processing halts. Treasury has lost `to_refund_recipient` tokens with no recipient credited. L2AssetTracker accounting is inconsistent. The priority transaction is stuck. [6](#0-5) [7](#0-6)

pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
{
    system_log!(
        system,
        "Transferring {nominal_token_value:?} tokens from treasury to {to:?}\n"
    );

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
            true, // true = subtract from balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Treasury transfer failed: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::TreasuryTransferFailed)
                }
                _ => wrap_error!(e),
            }
        })?;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            to,
            nominal_token_value,
            false, // false = add to balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Error while minting: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::MintingBalanceOverflow)
                }
                _ => wrap_error!(e),
            }
        })?;

    Ok(())
}
