### Title
Pool `change_reward_address` Incomplete Token Sentinel Allows Permanent Yield Freeze in BTC Pools — (`src/pool/pool.cairo`)

---

### Summary

The `change_reward_address` function in the Pool contract guards against setting the reward address to the pool's own delegation token, but does **not** guard against `STRK_TOKEN_ADDRESS` — the hardcoded address used to pay all pool rewards. In a BTC delegation pool, a pool member can set their `reward_address` to `STRK_TOKEN_ADDRESS`. Any subsequent `claim_rewards` call then transfers STRK directly into the STRK token contract itself, permanently locking the yield.

---

### Finding Description

The Pool contract's `change_reward_address` performs an incomplete sentinel check:

```rust
// src/pool/pool.cairo:505-510
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
``` [1](#0-0) 

This guard only prevents setting the reward address to the pool's own delegation token (e.g., BTC). It does **not** check against `STRK_TOKEN_ADDRESS`, which is the token used to pay all pool rewards:

```rust
// src/pool/pool.cairo:365-366
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [2](#0-1) 

`STRK_TOKEN_ADDRESS` is a hardcoded constant:

```
0x04718f5a0fc34cc1af16a1cdee98ffb20c31f5cd61d6ab07201858f4287c938d
``` [3](#0-2) 

For a BTC delegation pool, `self.token_dispatcher.contract_address.read()` returns the BTC token address. Since `BTC_TOKEN_ADDRESS != STRK_TOKEN_ADDRESS`, the assertion passes when a pool member calls `change_reward_address(STRK_TOKEN_ADDRESS)`.

By contrast, the staking contract's `change_reward_address` uses `does_token_exist()`, which checks against **all** known tokens (STRK + all BTC tokens):

```rust
// src/staking/staking.cairo:520-524
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [4](#0-3) 

Where `does_token_exist` is:

```rust
fn does_token_exist(self: @ContractState, token_address: ContractAddress) -> bool {
    token_address == STRK_TOKEN_ADDRESS || self.btc_tokens.read(token_address).is_some()
}
``` [5](#0-4) 

The pool contract lacks this comprehensive check. The design intent is clearly to block all token addresses as reward destinations (evidenced by the `REWARD_ADDRESS_IS_TOKEN` error name and the staking contract's full check), but the pool contract only enforces half of that invariant.

---

### Impact Explanation

When `claim_rewards` is called after the reward address has been set to `STRK_TOKEN_ADDRESS`, the STRK rewards are transferred to the STRK token contract itself. Standard ERC20 contracts have no mechanism to recover tokens sent to their own address. The yield is **permanently frozen** — matching the "Permanent freezing of unclaimed yield" impact category.

---

### Likelihood Explanation

Any pool member in a BTC delegation pool can call `change_reward_address(STRK_TOKEN_ADDRESS)` at any time without restriction. The call succeeds silently. The pool member may do this accidentally (e.g., copy-pasting the wrong address) or through a front-end bug. Once set and `claim_rewards` is triggered (which can be called by anyone on behalf of the pool member per the access control), the yield is gone. The staking contract's own `change_reward_address` correctly blocks this, making the pool's omission a concrete, reachable inconsistency.

---

### Recommendation

The pool's `change_reward_address` must also reject `STRK_TOKEN_ADDRESS`:

```rust
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    assert!(
        reward_address != STRK_TOKEN_ADDRESS,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
```

A more future-proof fix is to query the staking contract's `does_token_exist` (or maintain a local reference to all known token addresses), mirroring the staking contract's comprehensive check.

---

### Proof of Concept

1. Token admin deploys a BTC token and calls `add_token` / `enable_token` on the staking contract.
2. A staker calls `set_open_for_delegation(btc_token_address)`, deploying a BTC delegation pool.
3. A pool member delegates BTC to the pool.
4. The pool member calls `pool.change_reward_address(STRK_TOKEN_ADDRESS)`.
   - Check: `self.token_dispatcher.contract_address.read()` → `BTC_TOKEN_ADDRESS`
   - `BTC_TOKEN_ADDRESS != STRK_TOKEN_ADDRESS` → `true` → assertion **passes**
   - `reward_address` is now stored as `STRK_TOKEN_ADDRESS`.
5. Rewards accumulate over epochs via `update_rewards_from_staking_contract`.
6. Anyone calls `pool.claim_rewards(pool_member)`.
7. Execution reaches:
   ```rust
   let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
   reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards.into());
   ``` [6](#0-5) 
8. STRK rewards are transferred **into the STRK token contract itself** and are permanently unrecoverable.
