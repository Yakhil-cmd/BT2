### Title
Malicious Participant Equivocation Triggers Permanent Protocol Abort in `reliable_broadcast_receive_all` — (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

A malicious participant can equivocate during the Echo-Broadcast Send phase, causing an honest party to hit an early-abort path inside `reliable_broadcast_receive_all`. Because this early abort terminates the **entire** function (covering all `n` concurrent broadcast sessions), the DKG, reshare, and refresh protocols are permanently denied for all honest parties. The malicious participant can repeat this attack on every retry as long as they remain in the participant set.

---

### Finding Description

`reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs` runs `n` simultaneous broadcast instances — one per participant — inside a single loop. When processing Echo messages for session `sid`, the code performs an early-termination check:

```rust
// lines 241–263
else if !state_sid.finish_amplification {
    let received_echo_cnt = state_sid.data_echo.get_sum_counters();
    let non_received_echo_cnt = n - received_echo_cnt;
    let mut is_enough = false;
    for (_, cnt) in state_sid.data_echo.iter() {
        if cnt + non_received_echo_cnt > echo_t {
            is_enough = true;
            break;
        }
    }
    if !is_enough {
        return Err(ProtocolError::AssertionFailed(format!(
            "The original sender in session {sid:?} is malicious! ..."
        )));
    }
}
``` [1](#0-0) 

When this fires, it issues a `return Err(...)` that exits the **entire** `reliable_broadcast_receive_all` function, not just the failing session `sid`. All other sessions — including those belonging to honest participants — are abandoned simultaneously.

**Attack scenario (n = 4: H1, H2, H3 honest; M malicious):**

- `echo_threshold = usize::midpoint(4, 1) = 2`; strictly more than 2 echo votes are required (i.e., ≥ 3).
- M equivocates in its own broadcast session: sends `Send(true)` to H1 and H2, and `Send(false)` to H3.
- H1 and H2 echo `true`; H3 echoes `false`.
- H3 collects: 2 echos for `true` (from H1, H2) + 1 echo for `false` (self-simulated). With all `n = 4` echo slots consumed and no value exceeding the threshold of 2, `is_enough = false`.
- H3's instance of `reliable_broadcast_receive_all` returns `Err`, aborting the entire DKG session for H3.
- Since H3 cannot produce a key share, the DKG fails for all honest parties.
- M can repeat this equivocation on every retry, permanently blocking DKG.

The test in the same file explicitly confirms this behavior:

```rust
// lines 557–563
let result = broadcast_dishonest(..., do_broadcast_dishonest_consume_version_1);
assert_eq!(result, Err(ProtocolError::AssertionFailed(
    "The original sender in session 3 is malicious! Could not collect enough echo votes to meet the threshold"
    .to_string())));
``` [2](#0-1) 

This abort propagates up through `do_broadcast` → `do_keyshare` → `do_keygen` / `do_reshare`, terminating the entire DKG/reshare/refresh protocol. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**High — Permanent denial of key generation, reshare, and refresh for honest parties.**

A single malicious participant within the documented fault-tolerance bound (`f < n/3`) can permanently prevent all honest parties from completing DKG, reshare, or refresh. Because the early abort exits the entire `reliable_broadcast_receive_all` call rather than isolating the offending session, all `n` concurrent broadcast instances are cancelled. Since the malicious participant remains in the participant set across retries, the attack is repeatable and permanent.

---

### Likelihood Explanation

**High.** The attack requires only that a malicious participant send different `Send` messages to different subsets of honest parties — a trivial network-level action requiring no cryptographic capability. The equivocation is undetectable before the echo phase completes, and the triggering condition (split echo counts that cannot reach the threshold) is reliably achievable with a single malicious party in any group of size ≥ 4. The test suite already demonstrates this path is reachable.

---

### Recommendation

Replace the global `return Err(...)` at lines 259–263 with a per-session abort that marks session `sid` as permanently failed (e.g., by setting all `finish_*` flags for that session to `true` and recording an error for it) and continues processing the remaining sessions. The outer function should only return `Err` if the local participant's own session (`sid == participants.index(me)?`) is the one that failed, or after all sessions have been processed and the results collected. This mirrors the fix described in the reference report: introduce a more permissive mode that skips individual failing items rather than aborting the entire batch. [5](#0-4)

The existing test `test_three_honest_one_dihonest` (version 1) in `src/protocol/echo_broadcast.rs` already demonstrates the root cause: [6](#0-5) 

To confirm the DKG-level impact, run `do_keygen` with 3 honest participants and 1 participant that equivocates in its `reliable_broadcast_send` call (sending different values to different peers). The `do_broadcast` call inside `do_keyshare` at line 362 will propagate the `Err` returned by `reliable_broadcast_receive_all`, causing `do_keygen` to return an error for all honest parties.
