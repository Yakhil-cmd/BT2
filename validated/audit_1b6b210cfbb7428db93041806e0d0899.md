### Title
Gateway Stateful Validator Checks Transaction L2 Gas Price Against Wrong Block Field (`l2_gas_price` Instead of `next_l2_gas_price`) — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful admission check for L2 gas price compares a transaction's `max_price_per_unit` against the **current block's** `l2_gas_price` field, but the correct reference is the **next block's** `next_l2_gas_price` field (the EIP-1559-computed price that will actually govern the block the transaction is included in). The developer acknowledged this with a TODO comment but left the wrong field in place. The result is a systematic mismatch between the gateway admission decision and the blockifier execution decision: during congestion, transactions with a price between `l2_gas_price` and `next_l2_gas_price` are admitted to the mempool but then rejected at block-building time; during under-congestion, valid transactions are rejected at the gateway.

---

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` reads the L2 gas price reference from `get_block_info().gas_prices.strk_gas_prices.l2_gas_price`:

```rust

let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

`get_block_info()` in the batcher maps the stored block header field `header.l2_gas_price.price_in_fri` into `strk_gas_prices.l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    l1_gas_price: convert_price(header.l1_gas_price.price_in_fri)?,
    l1_data_gas_price: convert_price(header.l1_data_gas_price.price_in_fri)?,
    l2_gas_price: convert_price(header.l2_gas_price.price_in_fri)?,
},
``` [2](#0-1) 

However, `BlockHeaderWithoutHash` carries **two distinct** L2 gas price fields:

```rust
pub l2_gas_price: GasPricePerToken,      // price used in THIS block
pub l2_gas_consumed: GasAmount,
pub next_l2_gas_price: GasPrice,          // EIP-1559 price for the NEXT block
``` [3](#0-2) 

`next_l2_gas_price` is computed by `calculate_next_base_gas_price` (EIP-1559) and stored in the block header. It is the price that the batcher will use when building the next block. The gateway never reads this field; `get_block_info()` does not expose it.

The threshold comparison in `validate_tx_l2_gas_price_within_threshold` therefore uses the stale current-block price:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...);
}
``` [4](#0-3) 

The EIP-1559 formula allows the price to change by up to `price / gas_price_max_change_denominator` per block: [5](#0-4) 

With a denominator of 8 (Ethereum-style), this is up to 12.5% per block. The gap between `l2_gas_price` and `next_l2_gas_price` is therefore non-trivial and attacker-observable.

---

### Impact Explanation

**Congested block (next_l2_gas_price > l2_gas_price):**
A transaction with `max_price_per_unit` in the range `[l2_gas_price, next_l2_gas_price)` passes the gateway's `validate_resource_bounds` check (using the lower stale price) and enters the mempool. When the batcher builds the next block, it uses the correct `next_l2_gas_price` from the proposal init. The blockifier's `check_fee_bounds` then rejects the transaction with `MaxGasPriceTooLow`. The transaction occupies a mempool slot and wastes a batcher execution attempt without ever being included.

**Under-congested block (next_l2_gas_price < l2_gas_price):**
A transaction with `max_price_per_unit` in the range `[next_l2_gas_price, l2_gas_price)` is rejected by the gateway even though it would be accepted and executed correctly by the batcher. Valid user transactions are dropped at the admission gate.

Both directions are reachable by any unprivileged user who can submit RPC transactions. The congestion direction is the more dangerous one: it allows systematic mempool pollution with transactions that will never be included, degrading sequencer throughput.

---

### Likelihood Explanation

The EIP-1559 price update runs every block. Any block that is more than half-full causes `next_l2_gas_price > l2_gas_price`. On a live network with non-trivial load this condition is routine. The attacker needs only to observe the current block's gas usage (public information) and submit transactions priced just above `l2_gas_price` but below `next_l2_gas_price`. No special privileges, keys, or contract deployments are required.

---

### Recommendation

Replace the `get_block_info()` call with a dedicated accessor that returns `next_l2_gas_price` from the block header. The `GatewayFixedBlockStateReader` trait should expose this field, and `validate_resource_bounds` should use it as the reference price:

```rust
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_next_l2_gas_price()   // reads header.next_l2_gas_price
    .await?;
```

The `get_block_info()` path in the batcher (`crates/apollo_batcher/src/batcher.rs` lines 699–717) should also be extended to surface `header.next_l2_gas_price` so the gateway can consume it without a separate RPC call.

---

### Proof of Concept

1. Observe that the previous committed block consumed more than half its gas budget (public via RPC).
2. Compute `next_l2_gas_price` using the public EIP-1559 formula: `next = current + current * gas_delta / (gas_target * denominator)`.
3. Submit an `InvokeV3` transaction with `l2_gas.max_price_per_unit = current_l2_gas_price` (one unit below `next_l2_gas_price`).
4. The gateway's `validate_tx_l2_gas_price_within_threshold` compares against `current_l2_gas_price` and passes.
5. The transaction enters the mempool.
6. The batcher builds the next block with `next_l2_gas_price` as the block's L2 gas price.
7. The blockifier's `check_fee_bounds` fires `MaxGasPriceTooLow` and the transaction is dropped.
8. Repeat at scale to fill the mempool with permanently-unexecutable transactions. [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
