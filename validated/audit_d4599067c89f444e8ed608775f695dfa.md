### Title
Fee-on-Transfer NEP-141 Token Over-Minting in `receive_erc20_tokens` - (File: engine/src/engine.rs)

### Summary

The `receive_erc20_tokens` function in the Aurora Engine mints ERC-20 tokens equal to `args.amount` — the amount declared in the `ft_on_transfer` callback — without verifying the actual NEP-141 balance received by the engine. If a bridged NEP-141 token implements a fee-on-transfer mechanism, Aurora will mint more ERC-20 tokens than it holds in NEP-141 backing, creating a permanent insolvency.

### Finding Description

When a NEP-141 token is transferred to Aurora via `ft_transfer_call`, the NEP-141 standard calls `ft_on_transfer` on the receiver (Aurora) with the `amount` field set to the amount specified by the sender — not the amount actually credited to Aurora after any token-level fees.

In `engine/src/engine.rs`, the `receive_erc20_tokens` function reads this caller-supplied `amount` directly from the callback arguments and uses it to mint ERC-20 tokens:

The `setup_receive_erc20_tokens_input` call triggers an EVM `mint` on the ERC-20 mirror contract for exactly `amount` tokens. No balance snapshot is taken before and after the NEP-141 transfer to confirm what was actually received. [2](#0-1) 

The `ft_on_transfer` entry point in `engine/src/contract_methods/connector.rs` dispatches to `receive_erc20_tokens` for any registered NEP-141 predecessor: [3](#0-2) 

NEP-141 token registration via `deploy_erc20_token` is permissionless — any caller can register any NEP-141 token as an ERC-20 on Aurora: [4](#0-3) 

### Impact Explanation

**Critical — Insolvency.**

If a fee-on-transfer NEP-141 token is registered with Aurora, every bridge-in operation over-mints ERC-20 tokens relative to the NEP-141 backing held by the engine. The cumulative shortfall grows with each deposit. When users attempt to exit back to NEAR via `withdrawToNear` / `ExitToNear`, the engine calls `ft_transfer` on the NEP-141 contract for the full ERC-20 balance. The last users to exit will find Aurora holds insufficient NEP-141 tokens to cover their ERC-20 balance, permanently freezing those funds. [5](#0-4) 

### Likelihood Explanation

**Medium.** The `deploy_erc20_token` function is callable by any NEAR account, so any existing or newly deployed fee-on-transfer NEP-141 token can be registered. An attacker can deploy such a token themselves, register it with Aurora, and repeatedly bridge tokens in to accumulate an over-minted ERC-20 supply. No privileged access is required beyond a standard NEAR account.

### Recommendation

After the `ft_on_transfer` callback is received, compute the actual amount credited by taking the difference of Aurora's NEP-141 balance before and after the transfer (or use the amount returned by the NEP-141 contract's balance query), and use that actual received amount — not `args.amount` — as the quantity to mint. Alternatively, explicitly reject NEP-141 tokens that implement fee-on-transfer semantics during the `deploy_erc20_token` registration step.

### Proof of Concept

1. Deploy a fee-on-transfer NEP-141 token on NEAR that deducts 10% on every transfer (i.e., receiver gets 90% of the specified amount).
2. Call `deploy_erc20_token` on Aurora to register this token — this succeeds permissionlessly.
3. Call `ft_transfer_call` on the NEP-141 token with `receiver_id = aurora`, `amount = 1000`, `msg = <recipient_evm_address>`.
4. The NEP-141 contract transfers 900 tokens to Aurora (after 10% fee) and calls `aurora.ft_on_transfer(amount=1000, ...)`.
5. Aurora's `receive_erc20_tokens` mints 1000 ERC-20 tokens to the recipient.
6. Repeat N times. After N iterations, Aurora holds `900*N` NEP-141 tokens but has minted `1000*N` ERC-20 tokens.
7. The first `900*N / 1000` users can exit successfully; the remaining users' ERC-20 balances are permanently frozen because Aurora cannot cover the shortfall.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

#[named]
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
