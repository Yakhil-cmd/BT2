Audit Report

## Title
Unbounded `CANISTER_DISPLAY_NAMES` HashMap and Prometheus Label Growth via Attacker-Controlled `network_identifier.network` Causes OOM DoS — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
The `RosettaMetricsMiddleware` extracts the raw, unvalidated `network_identifier.network` string from every incoming POST body and unconditionally inserts it into the process-global `CANISTER_DISPLAY_NAMES` HashMap and into the Prometheus default registry as a new label set. Neither structure has a size cap or eviction policy. An unauthenticated attacker can exhaust the ICRC-1 Rosetta process heap by sending requests with distinct strings in that field, causing an OOM crash and complete denial of service of the Rosetta node.

## Finding Description
`CANISTER_DISPLAY_NAMES` is declared as a process-global `lazy_static` with no bound: [1](#0-0) 

`RosettaMetrics::new` unconditionally inserts into it on every call: [2](#0-1) 

`RosettaMetricsMiddleware::call` invokes `RosettaMetrics::new` for every request that carries a `network_identifier.network` field. `get_display_name_from_canister_id` returns the attacker-supplied string itself when it is not already registered, so the attacker-controlled value becomes both the key and the value inserted into the map: [3](#0-2) 

`extract_canister_id` performs no length or character validation — it accepts any JSON string of arbitrary length: [4](#0-3) 

The body is buffered with `usize::MAX` as the size limit, meaning a single request body can be arbitrarily large: [5](#0-4) 

A compounding vector exists: every unique `token_display_name` passed to `inc_api_status_count` and `observe_request_duration` creates a new Prometheus time-series label set in the default registry, which also has no eviction: [6](#0-5) [7](#0-6) 

The metrics middleware is applied to all routes, and the server binds to `0.0.0.0` with no authentication: [8](#0-7) 

## Impact Explanation
This is an application/platform-level DoS of the ICRC-1 Rosetta node — an in-scope financial integration component. Each unique attacker-supplied string permanently occupies heap memory in two places: the `CANISTER_DISPLAY_NAMES` HashMap and the Prometheus registry. With no body size limit and no entry cap, the Rosetta process heap grows without bound until the OS OOM-killer terminates it, causing a complete denial of service. This matches the allowed High impact: **"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."** This is not a raw volumetric DDoS — it is a targeted memory exhaustion attack exploiting a specific unbounded data structure in production code.

## Likelihood Explanation
The attack requires only unauthenticated HTTP POST access to any Rosetta endpoint (e.g., `/network/status`). No credentials, no privileged role, and no protocol-level access are needed. The default bind address is `0.0.0.0`, making the port reachable from any network-adjacent host. The attack is trivially scriptable and permanent — each request irreversibly grows the heap. A modest number of requests with long strings (e.g., 4 MB bodies) can exhaust memory rapidly.

## Recommendation
1. **Cap `CANISTER_DISPLAY_NAMES`**: enforce a maximum entry count (e.g., 1,000) and silently drop insertions beyond that limit.
2. **Validate the extracted canister ID**: accept only strings matching the canonical IC principal text format (e.g., `[a-z0-9]{5}(-[a-z0-9]{5}){4}`) before inserting into the map or using as a Prometheus label.
3. **Do not construct new `RosettaMetrics` instances per request**: the middleware should look up an existing, pre-registered `RosettaMetrics` by canister ID and fall back to the default rather than calling `RosettaMetrics::new` (which inserts into both the HashMap and the Prometheus registry) for every unrecognised ID.
4. **Apply a request body size limit**: replace `usize::MAX` in `to_bytes(body, usize::MAX)` with a reasonable cap (e.g., 4 MB, consistent with the ICP Actix server).

## Proof of Concept
```python
import requests, random, string

TARGET = "http://<rosetta-host>:8080/network/status"

for i in range(100_000):
    unique_id = ''.join(random.choices(string.ascii_lowercase, k=64)) + str(i)
    requests.post(TARGET, json={
        "network_identifier": {"blockchain": "Internet Computer", "network": unique_id},
        "metadata": {}
    })
    if i % 1000 == 0:
        print(f"Sent {i} requests")
# RSS of the rosetta process grows proportionally to i
```

A deterministic integration test can assert the invariant directly:
```rust
for i in 0..N {
    // simulate middleware call with unique network field
    let _ = RosettaMetrics::new(format!("fake-{i}"), format!("fake-{i}"));
}
let map = CANISTER_DISPLAY_NAMES.lock().unwrap();
assert_eq!(map.len(), N + 1); // holds for arbitrary N, demonstrating unbounded growth
```
