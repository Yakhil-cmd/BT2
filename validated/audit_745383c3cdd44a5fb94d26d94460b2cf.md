### Title
Missing Callback on `ExitToEthereum::run` Withdraw Promise Permanently Freezes User ETH — (`engine-precompiles/src/native.rs`)

---

### Summary

`ExitToEthereum::run` (flag `0x0`) schedules a bare `PromiseArgs::Create` for the eth-connector `withdraw` call with no attached callback. If that promise fails for any reason, the ETH already deducted from the caller's EVM balance and credited to the precompile address is permanently unrecoverable. The symmetric `ExitToNear` path has an explicit `exit_to_near_precompile_callback` refund mechanism; `ExitToEthereum` has none.

---

### Finding Description

**Step 1 — ETH is deducted and credited to the precompile address.**

When a user calls the `ExitToEthereum` precompile (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) with `msg.value = V`, the EVM deducts `V` from the caller's balance and credits it to the precompile address. The precompile returns `Ok(PrecompileOutput)`, so the EVM commits the transfer unconditionally.

**Step 2 — Only a bare `Create` promise is scheduled, no callback.** [1](#0-0) 

```rust
let withdraw_promise = PromiseCreateArgs { ... method: "withdraw" ... };
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

There is no `PromiseArgs::Callback` wrapping, no `exit_to_ethereum_precompile_callback`, and no refund args of any kind — unconditionally, regardless of any feature flag.

**Step 3 — Contrast with `ExitToNear`, which has a full refund path.**

`ExitToNear::run` conditionally wraps its transfer promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`: [2](#0-1) 

That callback calls `engine::refund_on_error`, which transfers ETH back from `exit_to_near::ADDRESS` to the original sender: [3](#0-2) 

The refund path is hardcoded to `exit_to_near::ADDRESS`. There is no equivalent branch for `exit_to_ethereum::ADDRESS`. A grep for any `ExitToEthereum` callback or refund pattern returns zero matches across the entire codebase.

**Step 4 — `filter_promises_from_logs` schedules the bare promise with no safety net.** [4](#0-3) 

A `PromiseArgs::Create` is simply scheduled. If the NEAR-side `withdraw` call panics or returns an error, NEAR rolls back that receipt but the Aurora EVM state (balance already deducted, precompile address already credited) is already finalised and is not rolled back.

---

### Impact Explanation

ETH value `V` is permanently frozen at `exit_to_ethereum::ADDRESS` (`0xb0bd02f6…`). The user's EVM balance is reduced by `V` with no recovery path. No admin function, no callback, and no on-chain mechanism exists to drain that address back to the user. This matches **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The `withdraw` call on the eth-connector can fail in several realistic, non-compromise scenarios:

- The eth-connector contract is **paused** (a routine operational action) while the `ExitToEthereum` precompile itself is not paused — two independent pause gates.
- The eth-connector is **upgraded** or has a bug that causes `withdraw` to panic.
- The eth-connector's NEP-141 accounting is in an inconsistent state (e.g., after a partial migration).

The `ExitToNear` path already acknowledges this risk and ships a refund callback precisely because promise failures are considered realistic. The identical risk exists for `ExitToEthereum` but the mitigation was never implemented.

The test suite explicitly documents the gap: [5](#0-4) 

```rust
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

That comment ("no refund in the EVM") applies permanently to `ExitToEthereum` — there is no `error_refund` gate that would ever enable a refund for it.

---

### Recommendation

Mirror the `ExitToNear` pattern for `ExitToEthereum`:

1. Accept a `refund_address` in the flag-`0x0` input (as `ExitToNear` does under `error_refund`).
2. Wrap `withdraw_promise` in a `PromiseArgs::Callback` pointing to a new `exit_to_ethereum_precompile_callback` method on the engine contract.
3. In that callback, if the promise result is not `Successful`, call `engine::refund_on_error` using `exit_to_ethereum::ADDRESS` as the source address (analogous to the `exit_to_near::ADDRESS` branch at line 1206 of `engine.rs`).

---

### Proof of Concept

```rust
// Pseudocode unit test
let mut engine = setup_engine();
let user = address(0xABCD);
engine.set_balance(user, Wei::new_u64(100));

// Call ExitToEthereum flag=0x0 with value=50
let input = [0x00u8].iter().chain(eth_recipient.as_bytes()).copied().collect();
let result = engine.call_precompile(exit_to_ethereum::ADDRESS, input, Wei::new_u64(50), user);
assert!(result.is_ok()); // precompile succeeds, ETH deducted

// Simulate the scheduled withdraw promise failing
simulate_promise_failure("withdraw");

// Assert: user balance is 50, not 100 — ETH is gone
assert_eq!(engine.get_balance(user), Wei::new_u64(50));
// Assert: no refund promise was ever created
assert!(engine.scheduled_promises().iter().all(|p| p.method != "exit_to_ethereum_precompile_callback"));
// ETH is permanently frozen at the precompile address
assert_eq!(engine.get_balance(exit_to_ethereum::ADDRESS), Wei::new_u64(50));
```
