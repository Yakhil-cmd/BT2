### Title
Unbounded `stakers` Vec Growth Due to Missing Removal on `unstake_action` Causes Unbounded Gas Consumption in `get_stakers` - (File: src/staking/staking.cairo)

### Summary
Every call to `stake()` unconditionally appends the staker address to the `self.stakers` Vec, but `unstake_action()` / `remove_staker()` never removes the address from that Vec. The `get_stakers` function iterates over the **full range** of the Vec on every call. An unprivileged actor can bloat the Vec with dead entries by cycling through minimum-stake accounts, making `get_stakers` increasingly expensive until it becomes impractical or exceeds gas limits, breaking the consensus layer's ability to query staker information.

### Finding Description

In `stake()`, every new staker is unconditionally pushed to the persistent `stakers` Vec: [1](#0-0) 

When a staker fully exits via `unstake_action()`, the internal `remove_staker()` helper clears `staker_info`, zeroes the own balance, and clears the operational address mapping — but **never removes the address from `self.stakers`**: [2](#0-1) 

Meanwhile, `get_stakers` iterates over `self.stakers.into_iter_full_range()` — the **entire** Vec including all dead entries — performing storage reads for each: [3](#0-2) 

The `is_staker_active` filter skips inactive stakers logically, but every dead entry still costs a storage read. Because `assert_staker_address_not_reused` prevents address reuse, each attack iteration requires a fresh address, but the Vec entry is permanent. [4](#0-3) 

### Impact Explanation

`get_stakers` is the primary interface used by the consensus layer to determine which stakers should attest and their relative staking power for a given epoch. As the `stakers` Vec grows with dead entries, the gas cost of `get_stakers` grows linearly. A sufficiently large Vec makes the function prohibitively expensive or causes it to exceed gas limits, preventing the consensus layer from querying staker information. This constitutes **unbounded gas consumption** (Medium impact per the allowed scope), and in the worst case could prevent stakers from attesting, freezing their unclaimed yield.

### Likelihood Explanation

Any unprivileged address can call `stake()` with the minimum stake amount, then call `unstake_intent()` followed by `unstake_action()` after the exit window to recover their funds. The only cost is the minimum stake locked for the exit window duration and transaction fees. By cycling through many fresh addresses, an attacker can bloat the Vec at relatively low cost. The attack is permissionless and requires no privileged access.

### Recommendation

Remove the staker address from `self.stakers` inside `remove_staker()` when the staker fully exits. Since `Vec` in Cairo does not support O(1) removal, a common pattern is swap-and-pop (swap the target with the last element, then truncate). Alternatively, maintain a separate mapping from staker address to its index in the Vec to enable efficient removal. This mirrors the fix recommended in the external report: do not allow dead entries to accumulate in iterated collections.

### Proof of Concept

1. Deploy the staking contract with minimum stake `M`.
2. For `N` fresh addresses `[a_1, ..., a_N]`:
   - Fund each with `M` STRK.
   - Call `stake(reward_address, operational_address, amount=M)` from each address.
   - Each call executes `self.stakers.push(staker_address)` at line 348.
3. After the exit window, call `unstake_intent()` then `unstake_action()` for each address to recover funds.
4. `remove_staker()` is called for each, but **none** are removed from `self.stakers`.
5. Call `get_stakers(epoch_id)`. The function iterates over all `N` dead entries plus any live stakers, performing a storage read per entry. With large `N`, the call becomes prohibitively expensive.

The `stakers` Vec length can be verified to equal `N` (all entries retained) while `staker_info` for each address returns `VInternalStakerInfo::None`, confirming the entries are dead but still iterated.
