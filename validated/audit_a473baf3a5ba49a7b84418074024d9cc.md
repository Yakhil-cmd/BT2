### Title
Gateway L2 Gas Price Admission Uses Stale Current-Block Price Instead of Pre-Committed `next_l2_gas_price` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the **current block's** `l2_gas_price` from `GatewayFixedBlockStateReader::get_block_info()` to gate-keep incoming transactions. The block header already stores a separate `next_l2_gas_price` field — the EIP-1559-derived price that the batcher will actually use when it executes the next block. Because the gateway and the batcher reference different prices, the gateway's admission decision is systematically wrong: it accepts transactions that will fail `check_fee_bounds` during execution (when the price is rising) and rejects transactions that would be valid (when the price is falling). The code itself acknowledges the defect with a `TODO` comment at the exact line.

---

### Finding Description

**Root cause — wrong price field in `GatewayFixedBlockSyncStateClient`**

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` from the latest committed block header. It copies `block_header.l2_gas_price` into `gas_prices.strk_gas_prices.l2_gas_price`, but the `StorageBlockHeader` also carries a distinct `next_l2_gas_price` field (the EIP-1559-computed price for the *next* block). That field is silently dropped and never surfaced to the gateway. [1](#0-0) 

**Stale price used for admission**

`validate_resource_bounds` reads the price from the `BlockInfo` returned above and calls `validate_tx_l2_gas_price_within_threshold`, which enforces `tx.max_price_per_unit >= (min_gas_price_percentage / 100) * previous_block_l2_gas_price`. The variable is even named `previous_block_l2_gas_price`, and the adjacent TODO comment explicitly states the correct fix:

```
// TODO(Arni): getnext_l2_gas_price from the block header.
``` [2](#0-1) 

The threshold function itself: [3](#0-2) 

**`run_validate_entry_point` inherits the same stale price**

The blockifier validation step also calls `get_block_info()` and builds its `BlockContext` from the same stale `l2_gas_price`. So `check_fee_bounds` inside the gateway's blockifier run passes, giving a false green light. [4](#0-3) 

**Batcher uses the correct `next_l2_gas_price`**

The consensus orchestrator computes the next block's L2 gas price via EIP-1559 (`calculate_next_base_gas_price`) and passes it to the batcher in `propose_block_input.block_info`. The batcher then propagates this price to the mempool and uses it as the block context for actual execution. [5](#0-4) [6](#0-5) 

**`next_l2_gas_price` is stored in the block header but never read by the gateway** [7](#0-6) 

---

### Impact Explanation

Two divergent failure modes arise from the price mismatch:

**1. Gateway admits transactions that will fail `check_fee_bounds` during execution (price rising)**

When the network is congested, EIP-1559 raises `next_l2_gas_price` above `l2_gas_price` by up to `price / gas_price_max_change_denominator` per block (denominator = 48, ≈ 2 % per block). A transaction with `max_price_per_unit` in the range `[l2_gas_price, next_l2_gas_price)` passes the gateway's threshold check but fails `check_fee_bounds` when the batcher executes it against the higher price. The transaction enters the mempool, occupies a slot, and is then reverted — the user pays fees for a failed transaction they had no reason to expect would fail.

**2. Gateway rejects transactions that are valid for the next block (price falling)**

When blocks are under-utilized, EIP-1559 lowers `next_l2_gas_price` below `l2_gas_price`. A transaction with `max_price_per_unit` in the range `[next_l2_gas_price, l2_gas_price)` is rejected at the gateway with `GAS_PRICE_TOO_LOW` even though it would satisfy `check_fee_bounds` in the next block. This is a direct "rejects valid transactions before sequencing" outcome.

Both outcomes match the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The condition is triggered on every block where the EIP-1559 mechanism adjusts the price — which is the normal operating state of any non-trivially loaded network. No special attacker capability is required; any ordinary user whose wallet sets `max_price_per_unit` equal to the current block's L2 gas price (the natural, visible reference) will hit the mismatch. The default `min_gas_price_percentage = 100` makes the boundary exact, maximising the affected range. [8](#0-7) 

---

### Recommendation

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a companion method) to also return `next_l2_gas_price` from `StorageBlockHeader`.
2. In `validate_resource_bounds`, replace the read of `gas_prices.strk_gas_prices.l2_gas_price` with `next_l2_gas_price` — exactly what the existing TODO comment requests.
3. Pass the same `next_l2_gas_price` into the `BlockContext` constructed inside `run_validate_entry_point` so that the blockifier's `check_fee_bounds` during gateway validation is consistent with what the batcher will enforce.

---

### Proof of Concept

1. Observe the current L2 gas price from the latest committed block header: call it `P_current`.
2. The EIP-1559 formula will set `P_next = P_current * (1 + gas_used / (gas_target * 48))`. For a full block (`gas_used = max_block_size = 5.8 × 10⁹`), `P_next ≈ P_current * 1.0208`.
3. Submit an invoke V3 transaction with `l2_gas.max_price_per_unit = P_current` (exactly the current block price, a natural wallet choice).
4. The gateway reads `previous_block_l2_gas_price = P_current` and computes `threshold = 1.0 * P_current`. The transaction satisfies `P_current >= P_current` → **admitted**.
5. The batcher builds the next block with `l2_gas_price = P_next > P_current`. `check_fee_bounds` evaluates `max_price_per_unit (P_current) < actual_gas_price (P_next)` → **`MaxGasPriceTooLow` error, transaction reverts**.
6. The user is charged fees for a revert that the gateway's own admission check should have prevented. [9](#0-8) [10](#0-9) [11](#0-10)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
