### Title
Malicious CKD Participant Can Corrupt CKD Output Without Detection - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function accepts participant contributions `(norm_big_y, norm_big_c)` and sums them into the final `CKDOutput` without any cryptographic verification that each contribution is correctly formed. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a corrupted `CKDOutput` that the client silently accepts, deriving an incorrect confidential key.

### Finding Description
In `do_ckd_coordinator`, the coordinator collects each participant's `CKDOutput` and accumulates it directly:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

Each participant is supposed to compute:
- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ∥ app_id) + y_i · app_pk)`

as implemented in `compute_signature_share`: [2](#0-1) 

However, the coordinator performs **no verification** that the received `(big_y, big_c)` pair satisfies this relationship. There is no zero-knowledge proof of correct formation, no consistency check against the participant's public key share, and no cross-check against `app_pk`. Any group element pair is accepted unconditionally.

The client later calls `CKDOutput::unmask(app_sk)` which computes `big_c − app_sk · big_y`: [3](#0-2) 

There is no verification at the client side either. The client has no mechanism to detect that the output was corrupted.

**Attack path:**
1. A malicious participant, instead of sending their correct `(norm_big_y_i, norm_big_c_i)`, sends arbitrary group elements `(P, Q)` to the coordinator.
2. The coordinator sums all contributions including the malicious ones.
3. The resulting `CKDOutput` is `(Y_honest + P, C_honest + Q)` instead of the correct `(Y, C)`.
4. The client computes `(C_honest + Q) − app_sk · (Y_honest + P)`, which equals `msk · H(pk ∥ app_id) + Q − app_sk · P` — an incorrect key.
5. Neither the coordinator nor the client detects the corruption.

This is structurally identical to the external report's root cause: unverified external values (participant contributions here, pool reserves there) are used directly in a critical computation without any integrity check.

### Impact Explanation
A malicious participant causes honest parties (the client) to silently accept and use an incorrect confidential derived key. The client has no way to distinguish a corrupted output from a correct one. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation
The CKD protocol requires all `n` participants to contribute. A single malicious participant among them is sufficient to corrupt the output. No special privilege is required beyond being a registered participant in the protocol. The attack requires only sending a malformed message to the coordinator, which is a trivial deviation from the protocol.

### Recommendation
Each participant should accompany their `(norm_big_y, norm_big_c)` contribution with a zero-knowledge proof of correct formation — specifically, a proof that `norm_big_c_i − norm_big_y_i · (app_pk / G) = λ_i · x_i · H(pk ∥ app_id)` relative to the participant's public key share. The coordinator must verify all proofs before summing contributions. Alternatively, the coordinator can verify consistency of the summed output against the aggregate public key before returning it.

### Proof of Concept
```
Setup: 3 participants, coordinator = P1, malicious = P2.

Honest execution:
  P2 computes norm_big_y_2 = λ_2 · y_2 · G
  P2 computes norm_big_c_2 = λ_2 · (x_2 · H + y_2 · app_pk)
  P2 sends (norm_big_y_2, norm_big_c_2) to P1

Attack:
  P2 instead sends (G, G) — the generator point for both fields

Coordinator result:
  Y_final = (λ_1·y_1·G + λ_3·y_3·G) + G   ← corrupted
  C_final = (λ_1·(x_1·H+y_1·app_pk) + λ_3·(x_3·H+y_3·app_pk)) + G  ← corrupted

Client unmask:
  C_final − app_sk · Y_final
  = msk·H(pk‖app_id) + G − app_sk·G
  = msk·H(pk‖app_id) + (1 − app_sk)·G   ← wrong key, silently accepted
```

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
