### Title
Deposit Refund Misdirected to Delegate Sender Instead of Relayer on Inner-Receipt Failure in Meta Transactions - (File: runtime/runtime/src/actions.rs)

### Summary

In NEAR's meta transaction (NEP-366) implementation, when a relayer submits a `DelegateAction` with an attached deposit and the inner action fails on the receiver's shard, the deposit refund is sent to the delegate sender (`Alice`) instead of the relayer who actually paid. An unprivileged user can exploit this by crafting a `DelegateAction` with a large attached deposit designed to fail, causing the relayer to lose the deposit while the sender receives it.

### Finding Description

In `apply_delegate_action` (`runtime/runtime/src/actions.rs`), the inner receipt spawned from a `DelegateAction` is constructed as:

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),          // Alice (delegate sender)
    receiver_id: delegate_action.receiver_id().clone(),
    receipt: ReceiptEnum::Action(ActionReceipt {
        signer_id: action_receipt.signer_id().clone(), // Relayer
        ...
    }),
});
``` [1](#0-0) 

The code comment at lines 471–475 explicitly acknowledges the mismatch:

> "If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction. Gas is refunded to the signer, this is Relayer." [2](#0-1) 

When this inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` in `runtime/runtime/src/lib.rs` calls `receipt.balance_refund_receiver()` to determine where to send the deposit refund: [3](#0-2) 

`balance_refund_receiver()` is defined as:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
``` [4](#0-3) 

`refund_to()` returns `&None` for `ReceiptEnum::Action` (the variant used in the inner receipt), so the fallback is always `predecessor_id` — which is Alice, not the relayer. [5](#0-4) 

The `refund_to` override field only exists in `ReceiptEnum::ActionV2`, which is never used when constructing the inner receipt from a `DelegateAction`.

The protocol documentation explicitly acknowledges this as an exploitable condition:

> "there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [6](#0-5) 

### Impact Explanation

The relayer pays the full attached deposit for inner actions (as required by NEP-366). If the inner action fails on the receiver's shard, the deposit is refunded to Alice (the `predecessor_id` of the inner receipt), not to the relayer who funded it. The exact corrupted value is the full `deposit` amount attached to the inner actions — transferred to Alice's account balance instead of the relayer's. This is a direct, permanent financial loss for the relayer with no protocol-level recovery mechanism.

**Impact: High** — arbitrary NEAR token amounts can be redirected from the relayer to the delegate sender.

### Likelihood Explanation

Any unprivileged user (Alice) can exploit this without any special privileges:
1. Alice constructs a `DelegateAction` with a large attached deposit targeting a call that will fail (e.g., a non-existent method, a contract that panics, or a receiver that does not exist).
2. Alice signs it and forwards it off-chain to the relayer.
3. The relayer, unable to guarantee execution success on a remote shard, submits the meta transaction.
4. The inner action fails; the deposit is refunded to Alice.

The relayer cannot fully prevent this without either refusing all meta transactions with attached deposits or simulating execution — neither of which is guaranteed to be accurate due to state changes between simulation and execution.

**Likelihood: Medium** — requires a willing relayer to accept the meta transaction, but relayers are designed to accept user-submitted `DelegateAction`s and cannot always predict failure.

### Recommendation

When constructing the inner receipt from a `DelegateAction` in `apply_delegate_action`, use `ReceiptEnum::ActionV2` (i.e., `ActionReceiptV2`) and set the `refund_to` field to `action_receipt.signer_id()` (the relayer). This ensures that if the inner action fails, the deposit refund is directed to the relayer who paid for it, mirroring how gas refunds already correctly target the relayer via `signer_id`. [7](#0-6) 

### Proof of Concept

**Actors:**
- `relayer.near` — the relayer, submits the meta transaction and pays all costs
- `alice.near` — the delegate sender, crafts the exploit
- `victim-contract.near` — a contract with no method named `nonexistent`

**Steps:**

1. Alice constructs a `DelegateAction`:
   - `sender_id`: `alice.near`
   - `receiver_id`: `victim-contract.near`
   - `actions`: `[FunctionCall { method_name: "nonexistent", deposit: 100_000_000_000_000_000_000_000_000 /* 100 NEAR */, gas: 30 TGas }]`
   - Signs with her key.

2. Alice sends the signed `DelegateAction` to the relayer off-chain.

3. The relayer wraps it in a transaction (signer: `relayer.near`, receiver: `alice.near`) and submits it. The relayer's account is debited 100 NEAR for the deposit plus gas costs.

4. On `alice.near`'s shard, `apply_delegate_action` creates the inner receipt:
   - `predecessor_id = alice.near`
   - `signer_id = relayer.near`

5. On `victim-contract.near`'s shard, the function call fails (`MethodNotFound`).

6. `refund_unspent_gas_and_deposits` fires:
   - `deposit_refund = 100 NEAR`
   - `receipt.balance_refund_receiver()` → `predecessor_id` → `alice.near`
   - `Receipt::new_balance_refund(&alice.near, 100 NEAR)` is emitted.

7. Alice's balance increases by 100 NEAR. The relayer's balance is permanently reduced by 100 NEAR (plus gas). Alice has stolen 100 NEAR from the relayer at zero cost to herself.
