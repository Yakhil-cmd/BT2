### Title
Single Malicious Participant Can Permanently Abort DKG, Reshare, and Refresh via False Success Broadcast - (File: src/dkg.rs)

### Summary
The `broadcast_success` function in `src/dkg.rs` requires unanimous agreement from **all** participants in the final round of DKG, reshare, and refresh. A single malicious participant can broadcast `(false, session_id)` instead of `(true, session_id)`, causing every honest participant to abort the protocol with an unrecoverable error. There is no threshold tolerance in this final round: even if the protocol is configured to tolerate `t-1` malicious parties, a single adversary can permanently deny key generation, resharing, or refresh.

### Finding Description

`broadcast_success` is the final step of `do_keyshare`, which is the shared implementation backing `keygen`, `reshare`, and `refresh`:

```
// src/dkg.rs:307-338
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    ...
    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant broadcast the wrong session id. Aborting Protocol!".to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant seems to have failed its checks. Aborting Protocol!".to_string(),
        ));
    }
    Ok(())
}
```

Every honest participant always broadcasts `(true, session_id)`. However, the checks at lines 321 and 329 require that **every** received vote is `true` and carries the correct session ID. A malicious participant who instead broadcasts `(false, session_id)` will cause all honest parties to hit the `all(|&(boolean, _)| boolean)` check and return `Err`.

Because `do_broadcast` uses the reliable echo-broadcast protocol (`reliable_broadcast_receive_all`), the malicious `false` vote is guaranteed to be delivered consistently to all honest parties — there is no way for honest parties to ignore or filter it out. The abort is therefore total and deterministic.

`broadcast_success` is called unconditionally at line 531 of `do_keyshare`:

```
// src/dkg.rs:531
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

`do_keyshare` is the shared core of all three public-facing protocols:
- `do_keygen` → `keygen` (line 552)
- `do_reshare` → `reshare` (line 623) and `refresh` (line 173 of `src/lib.rs`)

All three are permanently blocked by a single malicious participant.

### Impact Explanation

**High — Permanent denial of key generation, reshare, or refresh for honest parties.**

Any participant included in the participant list can abort DKG, reshare, or refresh at will by broadcasting `false` in the final round. Because the abort is triggered by the reliable broadcast output (not a local check), all honest parties receive the same `false` vote and all abort simultaneously. No subset of honest parties can proceed to obtain a usable `KeygenOutput`. The key material is never produced, and the protocol must be restarted from scratch — with the same attacker still present and able to repeat the attack indefinitely.

### Likelihood Explanation

**High.** The attacker needs only to be a listed participant (no special privilege beyond inclusion in the participant set). The attack requires no cryptographic capability: the malicious party simply sends `(false, session_id)` in the final broadcast round instead of `(true, session_id)`. The attacker can participate honestly through all prior rounds to avoid early detection, then abort at the last moment after all other parties have already committed their secret shares. The attack is repeatable at zero marginal cost.

### Recommendation

Replace the unanimous-agreement check with a threshold-tolerant check. The final success broadcast should abort only if **fewer than `threshold`** participants report success, consistent with the fault tolerance already assumed by the rest of the protocol. Participants who report failure (or send a wrong session ID) should be identified and excluded from the output rather than causing a global abort. Concretely:

1. Count how many participants broadcast `(true, session_id)`.
2. Abort only if that count falls below the reconstruction threshold.
3. Identify and log the specific participants who broadcast `false` or a mismatched session ID, so callers can exclude them from future sessions.

### Proof of Concept

1. Set up a DKG session with participants `[P0, P1, P2]` and threshold `2`. `P2` is malicious.
2. `P2` participates honestly through rounds 1–5 (commitment broadcast, proof-of-knowledge, share distribution).
3. In the `broadcast_success` round, `P2` uses the underlying channel to broadcast `(false, session_id)` instead of `(true, session_id)`.
4. The reliable echo-broadcast protocol delivers `(false, session_id)` consistently to `P0` and `P1`.
5. Both `P0` and `P1` reach the check at `src/dkg.rs:329`:
   ```
   if !vote_list.iter().all(|&(boolean, _)| boolean) {
       return Err(ProtocolError::AssertionFailed(...));
   }
   ```
