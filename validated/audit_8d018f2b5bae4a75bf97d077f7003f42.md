### Title
Missing Minimum Threshold Enforcement in FROST Signing Validation Allows Threshold-1 Signing — (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` enforce only an upper-bound check on the `threshold` parameter but omit the lower-bound check (`threshold >= 2`) that is explicitly enforced in `assert_key_invariants` in `src/dkg.rs`. Any library caller can pass `threshold = 1` (or `0`) to the FROST signing entry points, causing the signing protocol to run with a cryptographically invalid threshold and producing unusable, unverifiable signatures.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
// src/dkg.rs lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

In contrast, `assert_sign_inputs` — the public validation gate for FROST EdDSA and RedJubjub signing — only checks the upper bound:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard. The same omission exists in the FROST `presign` function:

```rust
// src/frost/mod.rs lines 71-77
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no invariant enforced at construction time, so `threshold = 1` (or `0`) is a valid value that passes all existing checks: [4](#0-3) 

---

### Impact Explanation

FROST key generation enforces `threshold >= 2`, so every key share is an evaluation of a degree-`(t-1)` polynomial where `t >= 2`. When signing is later invoked with `threshold = 1`:

1. The signing protocol treats the session as a 1-of-N scheme.
2. Lagrange interpolation is performed over a single point, yielding a coefficient of `1` for that participant.
3. The participant's share is a degree-`(t-1 >= 1)` polynomial evaluation — **not** the secret key itself.
4. The combined signature scalar is therefore wrong, and the resulting ECDSA/EdDSA signature fails to verify against the public key.

Honest participants complete the protocol without error, accept the output, and only discover the failure at external verification time. This matches:

> **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The entry path is direct and requires no special privilege. Any library caller invoking the FROST EdDSA or RedJubjub signing API controls the `threshold` argument. Passing `threshold = 1` satisfies all existing guards (`participants.len() >= 2` and `threshold <= participants.len()`), so the protocol runs to completion and silently produces a broken signature. A misconfigured or malicious coordinator can trigger this on every signing session.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already applies, in both `assert_sign_inputs` and `presign`:

```rust
// src/frost/mod.rs — assert_sign_inputs and presign
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing pattern in `src/dkg.rs` lines 580-582 and closes the inconsistency between DKG and signing validation. [5](#0-4) 

---

### Proof of Concept

1. Run DKG with `participants = [A, B, C]`, `threshold = 2` — succeeds, each party holds a degree-1 polynomial share.
2. Call `assert_sign_inputs(&[A, B, C], /*threshold=*/1usize, A, A)` — **passes all checks** (1 ≤ 3, participants.len() ≥ 2).
3. Call `frost::presign` with `args.threshold = 1` — **passes the upper-bound check** (1 ≤ 3).
4. Call the EdDSA/RedJubjub `sign` function with `threshold = 1` and only participant A contributing.
5. Lagrange interpolation over a single share produces the wrong scalar; the resulting signature fails `verify()` against the public key generated in step 1.
6. All honest parties accepted the protocol output — the corruption is silent until external verification. [6](#0-5) [7](#0-6)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
