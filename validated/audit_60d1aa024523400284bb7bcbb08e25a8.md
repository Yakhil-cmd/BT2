### Title
Missing Caller Authorization in `BankContract.transfer` Enables Unauthorized Native Token Theft — (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts the `sender` address as a user-supplied calldata argument and uses it directly as the `from` address in `bankKeeper.SendCoins`, without ever verifying that `sender == contract.Caller()`. Any contract can pass an arbitrary victim address as `sender` and drain that victim's `evm/<callerContract>`-denominated native tokens to any recipient.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)   // ← taken from calldata, never verified
    recipient := args[1].(common.Address)
    amount := args[2].(*big.Int)
    ...
    from := sdk.AccAddress(sender.Bytes())
    to   := sdk.AccAddress(recipient.Bytes())
    if err := bc.checkBlockedAddr(to); err != nil { ... }
    denom := EVMDenom(contract.Caller())  // denom = "evm/<callingContract>"
    ...
    bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `denom` is correctly scoped to the calling contract (`contract.Caller()`), but the `from` address is taken verbatim from the ABI-encoded input. There is no guard of the form `if sender != contract.Caller() { return error }`. The only check performed is `checkBlockedAddr(to)`, which validates the recipient, not the sender. [2](#0-1) 

The intended usage pattern (shown in `TestBank.nativeTransfer`) is that the calling contract enforces authorization at the ERC-20 layer before invoking the precompile:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);          // ERC-20 auth
    return bank.transfer(msg.sender, recipient, amount); // bank precompile
}
``` [3](#0-2) 

The bank precompile itself, however, performs no such check. A malicious contract can skip the ERC-20 authorization step entirely and call `bank.transfer(victim, attacker, amount)` directly.

The same missing check exists in the `burn` case: `recipient` (user-supplied) is passed directly to `bankKeeper.SendCoinsFromAccountToModule` without verifying the caller is authorized to burn from that address. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized transfer of `evm/<contract>`-denominated native assets.**

Any contract `M` that has distributed `evm/M` tokens to users (via `bank.mint` or prior `bank.transfer` calls) can subsequently call `bank.transfer(victim, attacker, amount)` to move those tokens out of the victim's account without the victim's consent. This constitutes an unauthorized balance change for precompile-controlled assets, matching the Critical impact tier.

The `burn` variant allows the same contract to destroy any holder's balance, which is also an unauthorized accounting change.

---

### Likelihood Explanation

**High.** The bank precompile is deployed at a fixed address (`0x0000000000000000000000000000000000000064`) and is callable by any contract. [5](#0-4) 

Any contract that has ever minted `evm/<itself>` tokens to external users can exploit this at will. No special privileges, leaked keys, or cryptographic breaks are required — only the ability to deploy a contract and call the precompile.

---

### Recommendation

Add a caller-identity check at the top of the `TransferMethodName` (and `BurnMethodName`) case, analogous to how the relayer precompile verifies the signer matches `contract.Caller()`:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender is not authenticated: must equal caller")
    }
    ...
``` [6](#0-5) 

This mirrors the pattern already used in `exec()` (the relayer precompile utility), where the signer extracted from the message is compared against `e.caller` before any state mutation is allowed.

---

### Proof of Concept

1. Attacker deploys `MaliciousToken` contract at address `M`.
2. `MaliciousToken` calls `bankPrecompile.mint(victim, 1000)` — victim now holds 1000 `evm/M` native tokens.
3. Attacker calls `MaliciousToken.steal(victim, attacker, 1000)`, which internally executes:
   ```solidity
   bank.transfer(victim, attacker, 1000);
   ```
4. The bank precompile sets `from = victim`, `denom = "evm/M"`, and calls `bankKeeper.SendCoins(ctx, victim, attacker, 1000 evm/M)`.
5. Victim's 1000 `evm/M` tokens are transferred to the attacker with no signature or approval from the victim. [7](#0-6)
