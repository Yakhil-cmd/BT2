### Title
`evmAddress` Not Included in Signed Digest Enables Cross-COA Signature Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `evmAddress` is committed to inside `signedData`. A signature produced for one Cadence Owned Account (COA) can be replayed against a different COA controlled by the same Cadence account, causing `isValidSignature` (ERC-1271) to return valid for the wrong COA. The code itself documents this gap in a comment but treats it as low-risk; the analysis below shows a concrete, unprivileged, cross-COA replay path with real EVM-side impact.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies that the provided `signatures` are valid over `signedData` using the Cadence account's keys, and then separately checks that the COA resource stored at `path` has an EVM address matching the supplied `evmAddress` argument.

The critical gap: the cryptographic check (`keyList.verify`) operates only on `signedData`. The `evmAddress` is never required to be encoded inside `signedData`. The function explicitly documents this:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

The signature verification step: [2](#0-1) 

The `evmAddress` check happens only *after* the signature is accepted as valid: [3](#0-2) 

The `validateCOAOwnershipProof` function is invoked by the `proofVerifier` precompile (`verifyCOAOwnershipProof(address,bytes32,bytes)`) registered in the Cadence Arch contract, which is callable from any EVM transaction: [4](#0-3) 

Every COA implements ERC-1271 by calling this precompile with `address(this)` as the EVM address: [5](#0-4) 

The `COAOwnershipProofInContext` struct separates `SignedData` from `EVMAddress`, confirming they are independent inputs with no enforced binding: [6](#0-5) 

---

### Impact Explanation

If a Cadence account controls two COAs (COA_A at EVM address `0xAAA` and COA_B at EVM address `0xBBB`), a valid ownership proof produced for COA_A over some `signedData = H` can be submitted verbatim to COA_B's `isValidSignature(H, proof)`. The signature check passes (same Cadence keys, same `signedData`), and the `evmAddress` check passes (the precompile supplies `address(this)` = `0xBBB`, and COA_B's stored address is `0xBBB`). The function returns `isValid: true` for COA_B using a proof that was never authorized for COA_B.

Concrete consequences:
- **Unauthorized ERC-1271 approvals**: EVM DeFi protocols (e.g., Permit2, Gnosis Safe, OpenSea Seaport) that call `isValidSignature` to authorize token transfers or order fills would accept the replayed proof for COA_B, enabling unauthorized asset movement from COA_B.
- **DAO/governance replay**: A vote or delegation signature produced for COA_A can be replayed to cast an unauthorized vote from COA_B.
- **NFT marketplace exploitation**: A listing signature for COA_A can be replayed to authorize a sale from COA_B.

---

### Likelihood Explanation

**Preconditions**:
1. The victim Cadence account must own at least two COAs. The protocol does not prevent this; `EVM.createCadenceOwnedAccount()` can be called multiple times from the same account.
2. The attacker must observe a valid ownership proof (signature + `signedData`) for COA_A. This is realistic: proofs are submitted in EVM transactions (public on-chain data) or in off-chain authentication flows.
3. The attacker must find a context where the same `signedData` value is meaningful for COA_B (e.g., the same hash `H` is used as an ERC-1271 digest for a different operation on COA_B).

The comment acknowledges the risk is "low" because wallets historically create at most one COA per account. However, the protocol imposes no such limit, multi-COA patterns are valid and documented, and the attack requires no privileged access — only the ability to submit an EVM transaction. As multi-COA usage grows (e.g., treasury management, protocol-owned liquidity), the likelihood increases.

---

### Recommendation

Enforce that `evmAddress` is committed to inside `signedData` at the protocol level. The simplest fix is to require `signedData` to be a structured encoding that includes `evmAddress` (e.g., `keccak256(abi.encode(evmAddress, applicationData))`), and verify this binding inside `validateCOAOwnershipProof` before accepting the proof. Alternatively, the function can construct the message to sign as `hash(evmAddress || signedData)` internally, removing the burden from callers. [2](#0-1) 

---

### Proof of Concept

**Setup**: Cadence account `Alice` creates two COAs:
- COA_A at EVM address `0xAAA`
- COA_B at EVM address `0xBBB`

**Step 1 – Obtain a valid proof for COA_A**:
Alice signs `signedData = H` (e.g., a hash of some message) with her Cadence key. This produces a `COAOwnershipProof` with `Address = Alice`, `CapabilityPath = /public/coaA`, `Signatures = [sig_H]`, `KeyIndices = [0]`.

This proof is submitted in an EVM transaction calling `COA_A.isValidSignature(H, encodedProof)` — it succeeds and is visible on-chain.

**Step 2 – Replay against COA_B**:
An attacker constructs a new `encodedProof` with the same `Address`, `Signatures`, `KeyIndices`, but `CapabilityPath = /public/coaB`.

The attacker submits an EVM transaction calling `COA_B.isValidSignature(H, newEncodedProof)`.

**Step 3 – Trace through the code**:
- `COA_B.isValidSignature` calls `verifyCOAOwnershipProof(0xBBB, H, newEncodedProof)` via the Cadence Arch precompile. [7](#0-6) 
- `DecodeABIEncodedProof` extracts `EVMAddress = 0xBBB`, `SignedData = H`, and the proof. [8](#0-7) 
- `validateCOAOwnershipProof` verifies `sig_H` over `signedData = H` using Alice's key — **passes**, because Alice did sign `H`. [2](#0-1) 
- The function borrows COA_B from `/public/coaB` and checks `coaRef.address() == 0xBBB` — **passes**. [3](#0-2) 
- Returns `ValidationResult(isValid: true)`.
- `COA_B.isValidSignature` returns `0x1626ba7e` (valid). [9](#0-8) 

**Result**: Any EVM protocol that trusts `COA_B.isValidSignature(H, ...)` now accepts an action authorized by COA_B for hash `H`, even though Alice only ever authorized `H` in the context of COA_A.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)

func NewCOAOwnershipProofInContext(sd []byte, addr Address, encodedProof []byte) (*COAOwnershipProofInContext, error) {
	proof, err := COAOwnershipProofFromEncoded(encodedProof)
