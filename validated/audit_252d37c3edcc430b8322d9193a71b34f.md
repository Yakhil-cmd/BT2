### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` in the Staking contract is documented as callable only by the Starknet sequencer, but the implementation enforces no such restriction. Any unprivileged caller can invoke it with `disable_rewards=true`, which advances the global `last_reward_block` checkpoint without distributing any rewards. Because the checkpoint is global and monotonically consumed, the rewards for that block are permanently lost to all stakers and delegators.

---

### Finding Description

`update_rewards` is the consensus-era reward distribution entry point. Its spec states access is restricted to "Only starkware sequencer," but the actual guard is `general_prerequisites`, which only checks that the contract is unpaused and the caller is non-zero: [1](#0-0) 

No sequencer-role check exists anywhere in the function body. The function then unconditionally writes `current_block_number` to the global `last_reward_block` **before** branching on `disable_rewards`: [2](#0-1) 

When `disable_rewards=true`, execution returns immediately after that write, skipping the entire reward calculation and distribution pipeline (`_update_rewards`, pool transfers, reward-supplier accounting): [3](#0-2) 

Because `last_reward_block` is a single global storage slot shared across all stakers: [4](#0-3) 

a subsequent legitimate sequencer call for the same block will revert with `REWARDS_ALREADY_UPDATED`: [5](#0-4) 

The rewards for that block are irrecoverably lost. The attacker only needs to call once per block with any valid active staker address.

---

### Impact Explanation

Every block for which the attacker front-runs the sequencer's `update_rewards` call results in zero STRK rewards being credited to `unclaimed_rewards_own` for the targeted staker and zero STRK being transferred to any delegation pools. Because the reward-supplier's `unclaimed_rewards` is also never incremented for those blocks, the yield is permanently destroyed rather than deferred. This constitutes **permanent freezing of unclaimed yield** for all stakers and delegators.

---

### Likelihood Explanation

`update_rewards` is a public, permissionless entry point (only `general_prerequisites` is checked). The attacker needs only a valid active staker address (publicly observable on-chain) and enough gas to call once per block. On Starknet, the sequencer's own `update_rewards` transaction is visible before finality, making front-running straightforward. The cost to the attacker is a small gas fee per block; the damage is total loss of consensus-era rewards for the entire protocol.

---

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, analogous to how `assert_caller_is_attestation_contract` is used in `update_rewards_from_attestation_contract`: [6](#0-5) 

Introduce a stored `sequencer_address` (or reuse an existing operator role) and assert `get_caller_address() == sequencer_address` before any state mutation. Alternatively, move the `self.last_reward_block.write(current_block_number)` line to **after** the `disable_rewards` branch so that a skipped-reward call does not consume the block slot.

---

### Proof of Concept

```
Preconditions:
  - Consensus rewards are active (consensus_rewards_first_epoch has passed).
  - At least one staker `S` is active with non-zero balance.

Attack (once per block):
  1. Attacker observes the sequencer is about to call
       update_rewards(S, disable_rewards=false)
  2. Attacker front-runs with:
       update_rewards(S, disable_rewards=true)
     This succeeds because general_prerequisites only checks
     is_paused() and caller != 0.
  3. last_reward_block is set to current_block_number.
  4. Sequencer's call reverts: REWARDS_ALREADY_UPDATED.
  5. No rewards are credited to S.unclaimed_rewards_own.
  6. No STRK is transferred to any delegation pool.
  7. reward_supplier.unclaimed_rewards is never incremented.

Result: All stakers and delegators receive zero rewards for every
block the attacker front-runs. Repeated each block → permanent
freezing of all consensus-era yield.
```
