### Title
Missing Triple Participant-Set Validation in OT-Based ECDSA Presign Allows Malicious Coordinator to Permanently Deny Signing - (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in the OT-based ECDSA module explicitly omits the check that the presign participant set is a subset of the participant set that generated the Beaver triples. The `TriplePub` struct carries a `participants` field recording exactly who generated each triple, but this field is never validated against the `participants` argument passed to `presign()`. As a result, `presign()` returns `Ok(protocol)` — succeeding at initialization — even when the participant sets are incompatible, causing the protocol to fail later with a cryptographic verification error and permanently denying signing to honest parties when a malicious coordinator exploits this.

---

### Finding Description

`TriplePub` (defined in `src/ecdsa/ot_based_ecdsa/triples/mod.rs`) carries two fields that describe the context in which a triple was generated:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    /// The threshold which will be able to reconstruct it.
    pub threshold: ReconstructionLowerBound,
}
``` [1](#0-0) 

The `presign()` function in `src/ecdsa/ot_based_ecdsa/presign.rs` validates only the `threshold` field against the presign threshold, and explicitly skips the participant-set check with a comment:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [2](#0-1) 

The `TriplePub.participants` field is populated during triple generation and is the authoritative record of which participants hold valid shares. Because `presign()` never compares `args.triple0.1.participants` or `args.triple1.1.participants` against the presign `participants` slice, the function returns `Ok(protocol)` — initialization succeeds — even when the two sets are entirely disjoint.

The protocol then proceeds into `do_presign`, where each participant linearizes their triple shares using Lagrange coefficients computed over the **presign** participant set, not the **triple-generation** participant set. A participant who was not present for triple generation will use an invalid or fabricated `TripleShare`. The mismatch is only caught later, inside the running protocol, at the cryptographic consistency checks:

```rust
if big_e != (ProjectivePoint::GENERATOR * e).to_affine() {
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of kd".to_string(),
    ));
}
``` [3](#0-2) 

```rust
if (ProjectivePoint::GENERATOR * alpha != big_k + big_a)
    || (ProjectivePoint::GENERATOR * beta != big_x + big_b)
{
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of additive triple phase.".to_string(),
    ));
}
``` [4](#0-3) 

These checks will always fail when the participant sets are mismatched, but they fire only after all parties have already committed network and computation resources to the protocol run.

---

### Impact Explanation

A malicious coordinator controls which `PresignArguments` are supplied to each honest participant. By providing `PresignArguments` whose `triple0` and `triple1` were generated by a participant set that does not include one or more of the current presign participants, the coordinator causes every presign attempt to abort with a cryptographic verification error. Because the missing check is at initialization — not inside the protocol — honest parties have no way to detect the mismatch before expending resources. The coordinator can repeat this indefinitely, permanently preventing honest parties from completing any presign and therefore any signature.

This maps to: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

The coordinator is a designated role in the signing pipeline and is the natural entity that assembles `PresignArguments`. The `TriplePub.participants` field is a public, serializable struct field that any caller can inspect or forge. No privilege beyond being the coordinator is required to supply mismatched triples. The omission is explicitly acknowledged in the source code comment, confirming the check was considered and deliberately deferred to a trust assumption that is not enforced by the library.

---

### Recommendation

Add an explicit check in `presign()` that the presign `participants` slice is a subset of `triple0.1.participants` and `triple1.1.participants` before constructing the protocol:

```rust
// Enforce that every presign participant was present for triple generation.
for p in participants.iter() {
    if !args.triple0.1.participants.contains(p) {
        return Err(InitializationError::BadParameters(format!(
            "Participant {p:?} was not present for triple0 generation"
        )));
    }
    if !args.triple1.1.participants.contains(p) {
        return Err(InitializationError::BadParameters(format!(
            "Participant {p:?} was not present for triple1 generation"
        )));
    }
}
```

This mirrors the existing threshold check and converts a late, confusing cryptographic failure into an early, descriptive initialization error — consistent with the library's pattern of validating all inputs before starting a protocol.

### Proof of Concept

1. Run triple generation with participants `{P1, P2, P3}` at threshold 2, producing `(triple_share_i, triple_pub)` for each `P_i`. `triple_pub.participants == [P1, P2, P3]`.
2. Assemble `PresignArguments` for a presign session with participants `{P1, P2, P4}` (P4 was absent from triple generation), using the same `triple_pub` (threshold still matches: 2 == 2).
3. Call `presign(&[P1, P2, P4], P1, args)`. The threshold check at line 43 passes. The omitted participant-set check at lines 38–40 is skipped. `presign()` returns `Ok(protocol)`.
4. All three parties execute the protocol. P4 supplies an arbitrary `TripleShare`. The sum `e = e_1 + e_2 + e_4` does not equal the committed `big_e` from the original triple. The check at line 127 fires: `"received incorrect shares of kd"`.
5. The presign aborts for all honest parties. The coordinator repeats step 2–4 indefinitely, permanently denying signing. [2](#0-1) [1](#0-0)
