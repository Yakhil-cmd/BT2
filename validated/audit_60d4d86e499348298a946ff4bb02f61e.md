### Title
Incomplete Balance Sufficiency Check for ZK Transactions with Blobs Omits Blob Gas Fee - (`basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

### Summary
The `required_balance()` function for ABI-encoded (ZK) transactions does not include blob gas fees in its calculation. This function is used as the sole pre-execution balance sufficiency check during ZK transaction validation. However, the actual fee amount charged (`fee_to_prepay`) does include blob gas fees. A sender whose balance satisfies the incomplete check but falls short of the full fee causes `precharge_fee` to reach an `unreachable!()` assertion, panicking the bootloader.

### Finding Description

**Root cause — `required_balance()` omits blob gas:**

`basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs` defines `required_balance()` for ABI-encoded ZK transactions as:

```rust
pub fn required_balance(&self) -> Option<U256> {
    let fee_amount = self
        .max_fee_per_gas
        .read()
        .checked_mul(U256::from(self.gas_limit.read()))?;
    self.value.read().checked_add(U256::from(fee_amount))
}
```

This computes only `value + max_fee_per_gas × gas_limit`. Blob gas fees are entirely absent. [1](#0-0) 

Compare with the RLP-encoded EIP-4844 path, which correctly adds blob fees:

```rust
RlpEncodedTxInner::EIP4844(tx, _) => {
    let gas_fee = self.max_fee_per_gas().checked_mul(U256::from(self.gas_limit()))?;
    let blob_gas = GAS_PER_BLOB.checked_mul(tx.blob_versioned_hashes.count as u64)?;
    let blob_fee = tx.max_fee_per_blob_gas.checked_mul(U256::from(blob_gas))?;
    self.value().checked_add(blob_fee)?.checked_add(gas_fee)
}
``` [2](#0-1) 

**Balance check uses the incomplete function:**

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the ZK validation path calls `transaction.required_balance()` as the only balance sufficiency gate:

```rust
let Some(total_required_balance) = transaction.required_balance() else { ... };
if total_required_balance > originator_account_data.nominal_token_balance.0 {
    return Err(TxError::Validation(InvalidTransaction::LackOfFundForMaxFee { ... }));
}
``` [3](#0-2) 

**Actual fee deduction includes blob gas:**

Immediately after the balance check, the same validation code computes `fee_to_prepay` that includes blob gas:

```rust
let fee_to_prepay = gas_fee_amount
    .checked_add(fee_for_blob_gas)
    .ok_or(internal_error!("gas fee + blob gas fee"))?;
``` [4](#0-3) 

**`precharge_fee` panics on insufficient balance:**

`precharge_fee` in `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs` deducts `context.fee_to_prepay` (which includes blob gas) and treats any `InsufficientBalance` error as `unreachable!`, because it assumes the balance was already verified:

```rust
SubsystemError::LeafUsage(interface_error) => {
    unreachable!(
        "balance should be pre-verified, but received error {:?}",
        interface_error
    );
}
``` [5](#0-4) 

The ZK validation code explicitly handles blobs for ZK transactions (the comment reads "No need to feature gate this part, as `blobs()` should return an empty list for non-EIP4844 transactions"), confirming that ZK transactions can carry blobs when EIP-4844 is enabled. [6](#0-5) 

### Impact Explanation

An unprivileged sender whose balance satisfies `value + max_fee_per_gas × gas_limit ≤ balance < value + max_fee_per_gas × gas_limit + blob_gas_fee` can submit a ZK transaction carrying blobs. The incomplete balance check passes, but `precharge_fee` then attempts to subtract the larger `fee_to_prepay` (which includes blob gas). The subtraction returns `InsufficientBalance`, which hits the `unreachable!()` branch and panics the bootloader. In the proving mode (RISC-V bare-metal), this aborts proof generation for the entire block; in forward mode (x86 sequencer), it crashes the sequencer process. Either outcome constitutes a liveness/availability failure for the chain.

### Likelihood Explanation

EIP-4844 blob support is present in the ZK transaction validation path and is gated only by the `eip-4844` feature flag. When that feature is active, any sender can craft a ZK transaction with blobs and set their balance to the exact window described above. No privileged access, oracle manipulation, or external dependency is required — only a standard transaction submission.

### Recommendation

Include blob gas fees in `required_balance()` for ABI-encoded ZK transactions, mirroring the RLP-encoded EIP-4844 handling. The function should compute:

```
required_balance = value + max_fee_per_gas × gas_limit + max_fee_per_blob_gas × (num_blobs × GAS_PER_BLOB)
```

Alternatively, perform the balance check after `fee_to_prepay` is fully computed (including blob gas) rather than relying on the `required_balance()` helper.

### Proof of Concept

1. Enable the `eip-4844` feature. Fund an account with balance `B = value + max_fee_per_gas × gas_limit` (exactly the amount `required_balance()` returns).
2. Submit a ZK (ABI-encoded) transaction with one blob, `max_fee_per_blob_gas > 0`.
3. Validation: `required_balance()` returns `B`; balance check passes (`B == B`).
4. `fee_to_prepay` is set to `gas_fee_amount + fee_for_blob_gas > B`.
5. `precharge_fee` calls `update_account_nominal_token_balance(..., fee_to_prepay, subtract=true)`.
6. Storage returns `BalanceError::InsufficientBalance` → `SubsystemError::LeafUsage(...)`.
7. The `unreachable!()` branch fires → bootloader panics → block processing aborts. [1](#0-0) [3](#0-2)
