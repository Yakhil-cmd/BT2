### Title
Missing Cross-Participant Consensus on `app_id` in CKD Protocol Allows Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol accepts `app_id` as a caller-supplied parameter on each participant node independently, with no cross-participant broadcast or consensus step to verify that all nodes are computing shares for the same application identity. A single malicious participant can silently substitute a different `app_id`, causing the coordinator to aggregate mismatched shares and produce a cryptographically invalid CKD output that the requesting application cannot use.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, the public entry point `ckd()` accepts `app_id: impl Into<AppId>` as a plain caller-supplied argument:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,   // ← no cross-participant verification
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError>
``` [1](#0-0) 

This `app_id` is forwarded directly into `compute_signature_share`, where it is hashed with the master public key to produce the BLS point `H(pk, app_id)` that each node multiplies by its private share:

```rust
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
let big_s = hash_point * private_share.to_scalar();
let big_c = big_s + app_pk * y.0;
``` [2](#0-1) 

Each participant then sends only the computed elliptic-curve points `(norm_big_y, norm_big_c)` to the coordinator — the `app_id` itself is **never included in the message**:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

The coordinator simply sums all received pairs without any check on which `app_id` each participant used:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [4](#0-3) 

There is no broadcast round, no commitment to `app_id`, and no echo-broadcast consensus step (unlike the DKG protocol, which broadcasts session IDs via `do_broadcast` before any secret material is computed). [5](#0-4) 

---

### Impact Explanation

The CKD output is defined as:

```
C = Σ λᵢ · (xᵢ · H(pk, app_id) + yᵢ · A)
  = msk · H(pk, app_id) + a · Y
```

If malicious participant `j` substitutes `app_id_wrong`, their contribution becomes:

```
λⱼ · (xⱼ · H(pk, app_id_wrong) + yⱼ · A)
```

The coordinator's aggregated `C` then equals:

```
msk · H(pk, app_id_correct)
  + λⱼ · xⱼ · (H(pk, app_id_wrong) − H(pk, app_id_correct))
  + a · Y
```

When the application computes `sig = C − a·Y`, it obtains a value that is neither `msk · H(pk, app_id_correct)` nor `msk · H(pk, app_id_wrong)`. BLS verification against the master public key fails, and the application cannot derive its confidential key. The coordinator produces and returns this corrupted output without detecting any anomaly.

**Matched impact**: *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

A malicious MPC node needs only to pass a different byte string as `app_id` when invoking `ckd()`. No cryptographic capability, no leaked key, and no privileged access is required. The attack is a single-line change in the node's integration code. Because the protocol sends only opaque curve points to the coordinator, the substitution is completely undetectable by any other participant or by the coordinator itself.

---

### Recommendation

Add a consensus step for `app_id` (and `app_pk`) at the start of the CKD protocol, mirroring the session-ID broadcast already present in `do_keyshare`:

1. Before computing any share, each participant broadcasts a commitment (e.g., a hash) to their `(app_id, app_pk)` inputs via the existing `do_broadcast` / echo-broadcast infrastructure.
2. Each participant verifies that all received commitments match their own before proceeding to `compute_signature_share`.
3. Abort with `ProtocolError::MaliciousParticipant` if any mismatch is detected.

This is the same pattern used in `do_keyshare` for session IDs: [6](#0-5)

### Proof of Concept

```
Setup: 3 participants P0 (coordinator), P1 (honest), P2 (malicious).
       Agreed app_id = b"correct_app".

Step 1: P0 calls ckd(..., app_id=b"correct_app", ...)
        P1 calls ckd(..., app_id=b"correct_app", ...)
        P2 calls ckd(..., app_id=b"wrong_app",   ...)   ← substitution

Step 2: P1 sends (λ₁·Y₁, λ₁·C₁) computed with H(pk, "correct_app") to P0.
        P2 sends (λ₂·Y₂, λ₂·C₂) computed with H(pk, "wrong_app")   to P0.

Step 3: P0 sums all three contributions (including its own for "correct_app")
        and returns CKDOutput = (Y, C_corrupted).

Step 4: Application computes sig = C_corrupted − a·Y.
        sig ≠ msk·H(pk, "correct_app").
        BLS verification fails; key derivation is impossible.
        No participant or the coordinator detected the mismatch.
```
