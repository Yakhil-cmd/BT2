### Title
Bank Precompile `transfer` Allows Unauthorized Token Drain from Any Holder — (`File: x/cronos/keeper/precompiles/bank.go`)

### Summary
The `transfer` method of the bank precompile (`0x0000000000000000000000000000000000000064`) accepts an arbitrary `sender` address from calldata and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `sender == contract.Caller()`. Any deployed contract can therefore drain `evm/<itself>` tokens from any holder's balance without the holder's consent.

### Finding Description
In `bank.go`, the `TransferMethodName` case unpacks three arguments — `sender`, `recipient`, `amount` — and constructs `from = sdk.AccAddress(sender.Bytes())`. The denom is `EVMDenom(contract.Caller())` = `"evm/" + callerHex`. The precompile then calls `bankKeeper.SendCoins(ctx, from, to, coins)` with no check that `from` equals the caller:

```go
sender := args[0].(common.Address)          // arbitrary address from calldata
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())       // no equality check against contract.Caller()
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())         // denom tied to calling contract
amt  := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))   // transfers from arbitrary `from`
```

The only guard present is `readonly` (blocks staticcall) and `checkBlockedAddr(to)` (module-account blocklist). Neither prevents the calling contract from specifying a victim as `sender`.

### Impact Explanation
Any contract `M` deployed by an attacker can call `bank.transfer(victimAddress, attackerAddress, victimBalance)`. The bank precompile will execute `SendCoins` moving `evm/<M>` tokens from the victim to the attacker with no authorization from the victim. This is an **unauthorized transfer of precompile-controlled assets**, matching the Critical impact tier. Users who hold `evm/<M>` tokens — acquired through airdrops, DeFi rewards, or purchases — have no on-chain mechanism to prevent the issuing contract from draining their balance at will.

### Likelihood Explanation
The attack requires only that (1) the attacker deploys a contract and (2) users hold that contract's `evm/<contract>` tokens. Both conditions are routine in any DeFi deployment. No special role, leaked key, or governance action is needed. The entry path is a standard EVM transaction calling the bank precompile at `0x64`.

### Recommendation
Add a sender-authorization check in the `TransferMethodName` case. The simplest fix is to require that the `sender` argument equals `contract.Caller()`, so the contract can only move tokens from its own balance:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the caller")
}
```

Alternatively, adopt the standard ERC-20 allowance model: require an explicit approval from `sender` before the calling contract may transfer on their behalf.

### Proof of Concept
1. Attacker deploys contract `M` containing:
   ```solidity
   address constant BANK = 0x0000000000000000000000000000000000000064;
   function steal(address victim, address to, uint256 amt) external {
       (bool ok,) = BANK.call(
           abi.encodeWithSignature("transfer(address,address,uint256)", victim, to, amt)
       );
       require(ok);
   }
   ```
2. Users legitimately acquire `evm/<M>` tokens (e.g., as liquidity-mining rewards issued by `M`).
3. Attacker calls `M.steal(victimAddress, attackerAddress, victimBalance)`.
4. The bank precompile executes `SendCoins(ctx, victim, attacker, {denom:"evm/<M>", amount:victimBalance})` — no victim signature or allowance required.
5. Victim's entire `evm/<M>` balance is transferred to the attacker in a single transaction. [1](#0-0) [2](#0-1)
