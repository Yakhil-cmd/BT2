### Title
Missing Lower-Bound Validation on `threshold` in FROST `assert_sign_inputs` Allows Corruption of Signing Outputs — (File: src/frost/mod.rs)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` enforces only an upper-bound check on the `threshold` parameter (`threshold <= participants.len()`) but omits the lower-bound check (`threshold >= 2`) that is explicitly enforced in `assert_key_invariants` in `src/dkg.rs`. Any library caller or malicious coordinator can pass `threshold = 1` (or `threshold = 0`) to any FROST signing entry point, causing the signing protocol to proceed with a cryptographically insecure threshold value and produce an unusable or invalid signature output — the direct analog of the unrestricted config-parameter update in the reference report.

---

### Finding Description

`assert_sign_inputs` is the sole validation gate for all FROST signing functions. It checks:

```
if threshold.value() > participants.len() { return Err(ThresholdTooLarge) }
```

but contains **no lower-bound check**. [1](#0-0) 

In contrast, `assert_key_invariants` — the validation gate for DKG — explicitly rejects `threshold < 2`:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [2](#0-1) 

Every FROST signing entry point delegates entirely to `assert_sign_inputs` for threshold validation:

- `sign_v1` and `sign_v2` in `src/frost/eddsa/sign.rs` [3](#0-2) 
- `sign` in `src/frost/redjubjub/sign.rs` [4](#0-3) 

None of these functions add an independent lower-bound check. The `presign` function in `src/frost/mod.rs` has the same omission: [5](#0-4) 

When `threshold = 1` is accepted, `construct_key_package` is called with `threshold = 1`, creating a `KeyPackage` whose threshold is inconsistent with the degree-1 polynomial used during key generation (which required `threshold = 2`). [6](#0-5) 

The `aggregate` call then either produces a signature that fails external verification (because the Lagrange interpolation is performed over the wrong polynomial degree) or returns a `ProtocolError::AssertionFailed` — in both cases yielding an unusable cryptographic output for all honest participants who correctly completed their rounds. [7](#0-6) 

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

With `threshold = 1` injected by a malicious caller or coordinator:
- The `KeyPackage` is constructed with `min_signers = 1`, mismatching the degree-1 polynomial committed to during DKG.
- Honest participants complete all protocol rounds and send valid signature shares, but the coordinator aggregates them under the wrong threshold, producing a signature that either fails the internal FROST verification or does not verify against the established public key.
- All honest parties have expended their nonces (which are one-time-use in FROST) and receive an unusable output with no ability to detect the root cause without out-of-band coordination.

---

### Likelihood Explanation

**Low.** The attacker must be the library caller or the coordinator role in a signing session and must deliberately supply `threshold = 1`. No cryptographic break or key compromise is required. The inconsistency between `assert_key_invariants` (which rejects `threshold < 2`) and `assert_sign_inputs` (which does not) makes this a straightforward parameter-manipulation path for any participant who controls the signing invocation.
