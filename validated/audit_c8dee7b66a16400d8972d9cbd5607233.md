### Title
Attacker-Controlled `extra_conditions` Passed to `take_offer` RPC Without Semantic Display in Confirmation Dialog — (`packages/api/src/services/WalletService.ts`, `packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

A WalletConnect dApp with `chia_takeOffer` permission can inject arbitrary `extraConditions` (e.g., `CREATE_COIN` to an attacker-controlled puzzle hash) into the `take_offer` RPC call. The confirmation dialog's prominent "You Spend / You Receive" section is derived solely from the offer blob's `get_offer_summary` and does **not** account for `extra_conditions`. The raw JSON blob is shown in a secondary `rows` section under the label "Extra Conditions," but it is completely opaque to a normal user (raw opcodes and puzzle hashes). The user approves what appears to be a normal offer exchange while unknowingly signing a spend that also transfers assets to the attacker.

---

### Finding Description

**Step 1 — `extra_conditions` is in the `chia_takeOffer` dApp allowlist**

`DappCommands.ts` builds the `chia_takeOffer` schema by inheriting all `params` from `chia_wallet.take_offer` when the dApp entry does not override them: [1](#0-0) 

The `chia_wallet.take_offer` command declares `extra_conditions` as an optional `json` param: [2](#0-1) 

The TODO comment on line 356 (`// TODO verify rest if needed for DAPP, if not use for dapp separate params`) shows the developers themselves were uncertain whether `extra_conditions` should be exposed to dApps.

**Step 2 — `parseDappParams` passes `extra_conditions` through without semantic validation**

`parseDappParams` validates that each key is in the allowlist, then for `json`-typed params, applies no transformation — the raw attacker-controlled value passes through as-is: [3](#0-2) [4](#0-3) 

**Step 3 — Confirmation dialog's `walletDelta` display ignores `extra_conditions`**

`parseCommandDisplay` for `chia_wallet.take_offer` calls `getOfferSummary(params.offer)` and builds the "You Spend / You Receive" display from the offer blob alone. It never reads `params.extra_conditions`: [5](#0-4) 

**Step 4 — `extra_conditions` is shown only as an opaque raw JSON blob**

`humanizeParams` includes `extra_conditions` in the `rows` because it is defined in the data. `humanizeParamValue` for `json` type serializes it with `JSONBig.stringify`: [6](#0-5) 

The dialog renders the prominent `walletDelta` section first ("You Spend / You Receive"), then the `rows` section below it: [7](#0-6) 

A user sees `[{"opcode":51,"args":["0xattacker_ph",1000000000000]}]` under the label "Extra Conditions" — completely uninterpretable without Chialisp knowledge.

**Step 5 — `extra_conditions` is forwarded verbatim to the `take_offer` RPC**

After user confirmation, `sendCommand` is called with `parsedParams` (which includes `extra_conditions`): [8](#0-7) 

`WalletService.takeOffer` forwards it directly: [9](#0-8) 

---

### Impact Explanation

The user's confirmation is based on the `walletDelta` display (e.g., "You Spend: 1 XCH, You Receive: 1 CAT"), which is accurate for the offer itself. The `extra_conditions` appends additional spend conditions (e.g., `CREATE_COIN` to the attacker's puzzle hash for an arbitrary amount) that are **not reflected in the walletDelta**. The user signs and broadcasts a spend that transfers assets beyond what they understood they were approving. Direct asset loss results.

---

### Likelihood Explanation

Any dApp that has been granted `chia_takeOffer` permission can exploit this. The dApp constructs a legitimate-looking offer blob (so the offer summary looks normal) and appends `extraConditions` with attacker-controlled spend conditions. The user sees a normal offer confirmation and approves it. No special user action beyond approving the offer is required.

---

### Recommendation

1. **Strip `extra_conditions` from the `chia_takeOffer` dApp command entirely** — remove it from the inherited params or override `params` in the dApp entry to exclude it. The TODO comment at line 356 already flags this as unresolved.
2. If `extra_conditions` must be supported for dApps, **decode and humanize each condition** (opcode → human name, args → bech32m addresses and XCH amounts) and include the resulting asset transfers in the `walletDelta` "You Spend / You Receive" display.
3. Add a unit test asserting that `parseDappParams('chia_takeOffer', ...)` rejects any payload containing `extra_conditions`.

---

### Proof of Concept

```
1. Pair a dApp with chia_takeOffer permission.
2. Send a chia_takeOffer WC request:
   {
     offer: "<valid_offer_blob>",   // e.g., 1 XCH for 1 CAT — looks legitimate
     fee: "0",
     extraConditions: [
       { opcode: 51, args: ["0xattacker_puzzle_hash", 1000000000000] }
     ]
   }
3. parseDappParams accepts extra_conditions (it's in the allowlist, type 'json').
4. Confirmation dialog shows:
   - "You Spend: 1 XCH / You Receive: 1 CAT"  ← from offer summary only
   - "Extra Conditions: [{"opcode":51,"args":["0xattacker...","1000000000000"]}]"  ← opaque
5. User clicks Accept.
6. take_offer RPC receives extra_conditions → daemon appends CREATE_COIN(attacker_ph, 1 XCH)
   to the spend bundle.
7. Transaction broadcasts: user loses 1 XCH to attacker in addition to the offer exchange.
```

    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
