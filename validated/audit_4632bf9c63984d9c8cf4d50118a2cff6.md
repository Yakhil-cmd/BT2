### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both the staking contract and the pool contract expose a `change_reward_address` function that accepts a new `ContractAddress` for reward routing without validating that the address is non-zero. A staker or pool member can set their reward address to `0`, causing all subsequent reward claims to transfer tokens to the zero address, permanently destroying the unclaimed yield.

---

### Finding Description

In `src/staking/staking.cairo`, the `change_reward_address` function performs only one validation — that the new address is not a registered token address — and then unconditionally writes the caller-supplied value to storage: [1](#0-0) 

The only guard present is:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

There is no `assert!(reward_address.is_non_zero(), ...)` check.

The identical pattern exists in `src/pool/pool.cairo`: [2](#0-1) 

The same omission is present at the point of initial registration. In `stake()`: [3](#0-2) 

And in `enter_delegation_pool()`: [4](#0-3) 

Neither function checks `reward_address.is_non_zero()`.

---

### Impact Explanation

When rewards are claimed, the contract transfers STRK tokens to the stored `reward_address`. On Starknet, an ERC-20 `transfer` to address `0` succeeds silently — the tokens are credited to the zero address and are unrecoverable. Any staker or pool member who has set (or initially registered) a zero reward address will have all accrued yield permanently burned upon the next `claim_rewards` call. This matches the allowed impact: **Permanent freezing of unclaimed yield (High)**.

---

### Likelihood Explanation

The entry path is directly reachable by any unprivileged staker or pool member with no preconditions beyond having an active position. The risk is realistic in two scenarios:

1. **Accidental**: A front-end bug, scripting error, or copy-paste mistake passes `0` as the reward address.
2. **Griefing via social engineering**: An attacker tricks a user into calling `change_reward_address(0)` (e.g., via a malicious dApp UI).

No privileged role, key leak, or external dependency compromise is required.

---

### Recommendation

Add an explicit non-zero check in both `change_reward_address` implementations and in the `stake` / `enter_delegation_pool` entry points:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the pattern already used elsewhere in the codebase (e.g., `add_token` validates `token_address` for zero): [5](#0-4) 

---

### Proof of Concept

1. Alice is a registered staker with unclaimed STRK rewards accruing.
2. Alice (or a malicious dApp acting on her behalf) calls:
   ```cairo
   staking_dispatcher.change_reward_address(reward_address: 0.try_into().unwrap());
   ```
3. The call succeeds — no revert, no event indicating danger.
4. Alice's `staker_info.reward_address` is now `0`.
5. When `claim_rewards` is subsequently called (by anyone), the contract executes:
   ```cairo
   token_dispatcher.checked_transfer(recipient: staker_info.reward_address, amount: rewards);
   ```
   transferring all accrued STRK to address `0`.
6. The tokens are permanently unrecoverable. Alice's unclaimed yield is frozen forever.

The same flow applies to pool members via `pool.cairo`'s `change_reward_address`. [6](#0-5) [7](#0-6)

            // Prerequisites and asserts.
            self.general_prerequisites();

                !self.does_token_exist(token_address: reward_address),
                "{}",
                GenericError::REWARD_ADDRESS_IS_TOKEN,
            );
            let staker_address = get_caller_address();

            // Update reward_address and commit to storage.

            self.write_staker_info(:staker_address, :staker_info);

            // Emit event.

                .emit(
                    Events::StakerRewardAddressChanged {
                        staker_address, new_address: reward_address, old_address,
                    },
                );
