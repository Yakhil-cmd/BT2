### Title
Hardcoded `DepositCallGasLimit` of 23,300 gas prevents FLOW token deposits to EVM smart contracts with non-trivial `receive()` functions - (File: `fvm/evm/types/call.go`)

### Summary
`NewDepositCall` and `NewTransferCall` in `fvm/evm/types/call.go` use a hardcoded gas limit of `DefaultGasLimitForTokenTransfer = 21_000 + 2_300 = 23_300`. After the 21,000 intrinsic transaction cost, only 2,300 gas remains for the recipient contract's `receive()` or `fallback()` function. Any EVM smart contract whose `receive()` function consumes more than 2,300 gas will always fail to receive FLOW token deposits via the bridge.

### Finding Description
`DepositCallGasLimit` and `WithdrawCallGasLimit` are both set to `DefaultGasLimitForTokenTransfer`:

```go
// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300

DepositCallGasLimit  = DefaultGasLimitForTokenTransfer  // 23_300
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer  // 23_300
``` [1](#0-0) 

`NewDepositCall` uses this hardcoded limit unconditionally:

```go
func NewDepositCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: DepositCallGasLimit,
    }
}
``` [2](#0-1) 

`Account.Deposit()` in the handler calls `NewDepositCall` and then panics if the result is failed:

```go
func (a *Account) Deposit(v *types.FLOWTokenVault) {
    ...
    res, err := a.fch.executeAndHandleCall(
        types.NewDepositCall(bridge, a.address, v.Balance(), bridgeAccount.Nonce()),
        v.Balance(), false,
    )
    panicOnErrorOrInvalidOrFailedState(res, err)
}
``` [3](#0-2) 

`panicOnErrorOrInvalidOrFailedState` panics on a failed EVM result, reverting the entire Cadence transaction:

```go
func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {
    if res != nil && res.Failed() {
        panic(fvmErrors.NewEVMError(res.VMError))
    }
    ...
}
``` [4](#0-3) 

The `mintTo` procedure in the emulator confirms that when the internal call fails (e.g., out-of-gas), the state is reset and the failed result is returned — which then triggers the panic: [5](#0-4) 

The 2,300 gas stipend is the traditional Ethereum `.transfer()`/`.send()` stipend, which Solidity itself deprecated precisely because it is insufficient for contracts that do anything beyond a trivial log. A single cold `SSTORE` costs 20,000 gas; emitting a non-trivial event or calling another contract also exceeds 2,300 gas.

The `Transfer` call used for COA-to-EVM transfers has the same problem: [6](#0-5) 

### Impact Explanation
Any EVM smart contract deployed on Flow EVM whose `receive()` or `fallback()` function performs storage writes, emits events, or calls other contracts will be permanently unable to receive FLOW token deposits via `EVMAddress.deposit()` or `CadenceOwnedAccount.deposit()`. The deposit transaction always reverts (funds are returned to the sender, not lost), but the contract is effectively bricked as a FLOW token recipient. DeFi protocols, reward distributors, vaults, or staking contracts that rely on receiving FLOW via the bridge are entirely non-functional.

### Likelihood Explanation
High. The 2,300 gas stipend is well-known to be insufficient for any contract that does more than emit a single cheap event. Any protocol that deploys a FLOW-receiving contract with even minimal logic in `receive()` (e.g., incrementing a counter, recording a deposit timestamp, or forwarding to another address) will hit this limit. This is the exact reason Ethereum deprecated `.transfer()` and `.send()`.

### Recommendation
Remove the hardcoded 2,300 gas addition from `DefaultGasLimitForTokenTransfer`. For `DepositCallGasLimit`, use a higher limit (e.g., the full block gas limit or a configurable value) so that contracts with non-trivial `receive()` functions can accept deposits. The `WithdrawCallGasLimit` (which sends to the bridge address, a protocol-controlled EOA) is less affected but should also be reviewed.

### Proof of Concept
1. Deploy an EVM smart contract on Flow EVM with a storage-writing `receive()`:
   ```solidity
   contract Receiver {
       uint256 public depositCount;
       receive() external payable {
           depositCount++; // cold SSTORE: 20,000 gas — far exceeds 2,300
       }
   }
   ```
2. From a Cadence transaction, call `EVMAddress.deposit(from: <-vault)` targeting the deployed contract address.
3. The deposit call is issued with `GasLimit = 23_300`. After the 21,000 intrinsic cost, only 2,300 gas remains. The `SSTORE` in `receive()` runs out of gas.
4. The emulator's `mintTo` detects `res.Failed()`, resets state, and returns the failed result.
5. `panicOnErrorOrInvalidOrFailedState` panics with the VMError, reverting the Cadence transaction.
6. The deposit always fails; the contract can never receive FLOW tokens via the bridge.

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
