### Title
Stakers Vec Grows Unboundedly as Unstaked Stakers Are Never Removed, Enabling Griefing via Unbounded Gas Consumption in `get_stakers` - (File: `src/staking/staking.cairo`)

### Summary
The `stakers: Vec<ContractAddress>` storage variable in the `Staking` contract is append-only. Stakers are appended when they call `stake`, but are **never removed** when they call `unstake_action`. Because `remove_staker` only zeroes out `staker_info` and related mappings without touching `stakers`, the Vec grows monotonically. When a staker re-stakes after unstaking, the `stake` function finds `staker_info == VInternalStakerInfo::None` and appends the address again, creating duplicates. The `get_stakers` consensus function iterates over the **full range** of this Vec on every call. An unprivileged attacker can repeatedly stake and unstake to bloat the Vec, making `get_stakers` consume unbounded gas and eventually become uncallable.

### Finding Description

`src/staking/staking.cairo` line 167–169 explicitly documents the invariant:

```cairo
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

`remove_staker` (the internal function called by `unstake_action`) zeroes out `staker_info`, the operational address mapping, commission, and commission commitment — but never touches `stakers`:

```cairo
fn remove_staker(...) {
    self.insert_staker_own_balance(:staker_address, own_balance: Zero::zero());
    self.staker_info.write(staker_address, VInternalStakerInfo::None);
    self.operational_address_to_staker_address.write(operational_address, Zero::zero());
    staker_pool_info.commission.write(Option::None);
    staker_pool_info.commission_commitment.write(Option::None);
    ...
}
``` [2](#0-1) 

After `unstake_action` completes, `staker_info` is `VInternalStakerInfo::None`. If the same address calls `stake` again, the contract treats it as a brand-new staker and appends the address to `stakers` a second time. Each stake-unstake-restake cycle adds one more entry to the Vec.

`get_stakers` (the consensus-layer function) iterates over the **entire** `stakers` Vec on every invocation:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
``` [3](#0-2) 

Every inactive/duplicate entry still costs a storage read and an `is_staker_active` check. As the Vec grows, so does the gas cost of every `get_stakers` call.

### Impact Explanation

`get_stakers` is the function the consensus layer calls to determine the active validator set for a given epoch. If the `stakers` Vec is bloated to a sufficient size, `get_stakers` will exceed the Starknet block gas limit and revert on every call. This permanently freezes the ability of the consensus layer to read the validator set, which constitutes **permanent freezing of unclaimed yield** (validators cannot be identified, so rewards cannot be distributed) and **griefing with damage to the protocol**.

Impact category: **High / Medium** — permanent freezing of unclaimed yield / unbounded gas consumption.

### Likelihood Explanation

Any address holding the minimum stake amount can execute this attack. The attacker calls `stake` → waits for `exit_wait_window` → calls `unstake_action` → repeats. Each cycle costs only the minimum stake (temporarily locked, then returned) plus gas. The attack is permissionless, requires no privileged access, and is economically feasible for a well-funded griever. The `exit_wait_window` slows the attack but does not prevent it.

### Recommendation

1. **Remove stakers from the Vec on unstake**: When `remove_staker` is called, swap-and-pop the address from `stakers` (or use an `IterableMap` instead of a `Vec` so entries can be deleted).
2. **Alternatively, deduplicate on re-stake**: Before appending to `stakers`, check whether the address is already present (e.g., via a separate `Map<ContractAddress, bool>` tracking membership).
3. **Cap the Vec size or add a re-entry guard**: Prevent a previously-unstaked address from being appended again by tracking whether it was ever added.

### Proof of Concept

```
1. Attacker holds 2 × min_stake STRK.
2. Attacker calls stake(amount=min_stake, ...).
   → stakers Vec: [attacker]  (length 1)
3. Attacker calls unstake_intent().
4. Attacker waits exit_wait_window seconds.
5. Attacker calls unstake_action(attacker).
   → staker_info[attacker] = VInternalStakerInfo::None
   → stakers Vec: [attacker]  (still length 1, address NOT removed)
6. Attacker calls stake(amount=min_stake, ...) again.
   → staker_info[attacker] is None → treated as new staker → appended again
   → stakers Vec: [attacker, attacker]  (length 2)
7. Repeat steps 3–6 N times.
   → stakers Vec length = N+1
8. Anyone calls get_stakers(epoch_id=current_epoch).
   → Iterates N+1 entries, each costing a storage read + is_staker_active check.
   → At large N, call reverts with out-of-gas.
   → Consensus layer cannot read the validator set.
```
