### Title
Asymmetric Native-Resource-to-Gas Reconciliation Silently Overcharges Users on Negative Delta — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` adjusts `gas_used` **upward** when native resource consumption exceeds EVM gas consumption, but performs **no corresponding downward adjustment** when native consumption is lower than gas consumption. The excess fee is silently transferred to the operator instead of being refunded to the sender. A `TODO` comment in the code explicitly acknowledges this asymmetry.

---

### Finding Description

In `compute_gas_refund`, after computing EVM `gas_used` from remaining ergs and native `native_used` from remaining native resources, the function computes a reconciliation delta:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native consumption > gas consumption → charge more gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

When `delta_gas < 0` — meaning `native_used / native_per_gas < gas_used`, i.e., the transaction consumed fewer native resources than its EVM gas consumption implies — `gas_used` is **not decreased**. The user is therefore charged for more gas than the actual native computational cost warrants, and the excess is never returned.

The refund disbursed to the sender is:

```rust
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
``` [2](#0-1) 

Because `gas_used` is inflated (not reduced when `delta_gas < 0`), `token_to_refund` is smaller than it should be. The operator receives the difference:

```rust
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
    ...
``` [3](#0-2)

### Impact Explanation

A user submitting a ZK-format L2 transaction (or an L1→L2 priority transaction) whose native resource consumption is lower than its EVM gas consumption will be overcharged. The overcharge amount is:

```
overcharge = gas_price × |delta_gas|
           = gas_price × (gas_used − native_used / native_per_gas)
```

This excess is credited to the operator (coinbase) rather than refunded to the sender. The sender permanently loses funds proportional to the gap between EVM gas and native resource consumption. This is a direct, unprivileged-user-triggered funds-loss path: any transaction sender can be affected without any special role or external dependency.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever a transaction's EVM gas consumption exceeds its native resource consumption (in gas-equivalent units). This is plausible for transactions that execute many EVM operations that are expensive in gas but relatively cheap in the ZK proof system's native resource model — for example, cold `SLOAD` (2100 gas) or `CALL` overhead. The `native_per_gas` ratio is set per block context; any block where this ratio does not perfectly track EVM gas for all opcode types will produce transactions where `delta_gas < 0`. The `TODO` comment confirms the developers are aware this case is unhandled.

---

### Recommendation

Implement the downward adjustment symmetrically with the upward one:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption is lower than gas consumption.
    // Reduce gas_used so the sender is refunded the difference.
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Still enforce the minimal_gas_used floor (already applied above).
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

This ensures the user is never charged more than the actual computational cost in either resource dimension.

---

### Proof of Concept

1. Submit a ZK-format L2 transaction that performs many cold `SLOAD` operations (high EVM gas, relatively low native resource cost).
2. After execution, `compute_gas_refund` computes `gas_used` from ergs and `native_used` from native resources.
3. Because `native_used / native_per_gas < gas_used`, `delta_gas < 0`.
4. The `if delta_gas > 0` branch is not taken; `gas_used` remains at the higher EVM-gas-derived value.
5. The refund `gas_price × (gas_limit − gas_used)` is smaller than it should be.
6. The operator receives `gas_price × gas_used` instead of `gas_price × (native_used / native_per_gas)`.
7. The sender loses `gas_price × (gas_used − native_used / native_per_gas)` wei with no recourse. [5](#0-4)
