### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Forged Signatures When a Deploy-Account Is Pending — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the blockifier `__validate__` entry-point for any invoke transaction with `nonce = 1` when `account_tx_in_pool_or_recent_block` returns `true` for the sender. Because that check returns `true` for **any** account that has a transaction in the mempool pool or staged/committed state — not specifically a deploy-account transaction — an unprivileged attacker can submit an invoke transaction with a completely forged signature for any account that has a pending deploy-account transaction, and the gateway will admit it to the mempool without ever verifying the signature.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the nonce and resource-bound checks pass:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
``` [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `validate = false` on the `ExecutionFlags`, so the blockifier's `__validate__` entry point is **never invoked**: [3](#0-2) 

The `account_tx_in_pool_or_recent_block` predicate is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

`state.contains_account` returns `true` if the address appears in the `staged` or `committed` maps — i.e., for **any** account that has ever had a transaction staged or committed within the retention window, not exclusively one that has a deploy-account transaction pending: [5](#0-4) 

The code comment in `skip_stateful_validations` acknowledges this looseness ("it means that either it has a deploy_account transaction **or** transactions with future nonces that passed validations"), but does not account for the case where a third-party attacker exploits the skip by submitting a forged-signature invoke transaction **after** a legitimate deploy-account transaction appears in the mempool.

**Attack path:**

1. Account A submits a `deploy_account` transaction (nonce = 0). It enters the mempool pool.
2. Attacker observes account A's address in the public mempool.
3. Attacker crafts an `invoke` transaction for account A with `nonce = 1` and an arbitrary/forged signature.
4. Gateway's `validate_nonce` passes: `0 ≤ 1 ≤ max_allowed_nonce_gap`. [6](#0-5) 

5. `validate_by_mempool` passes: no duplicate hash, nonce not too old, nonce-1 slot is empty.
6. `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(A)` is `true` (the deploy-account is in the pool).
7. `run_validate_entry_point` is called with `validate = false`; the `__validate__` entry point is **not called**.
8. The forged-signature invoke transaction is forwarded to the mempool via `mempool_client.add_tx`. [7](#0-6) 

9. Account A's legitimate invoke transaction with `nonce = 1` is subsequently rejected with `MempoolError::DuplicateNonce` (or requires a fee-escalation replacement). [8](#0-7) 

---

### Impact Explanation

The gateway admits an invalid transaction — one whose `__validate__` entry point would fail — into the mempool. Concrete consequences:

- **Invalid transaction accepted before sequencing**: The mempool holds a transaction with a forged signature, violating the invariant that all admitted transactions carry a valid account signature (or are legitimately exempt).
- **Nonce-slot occupation / griefing**: The attacker occupies the `nonce = 1` slot for account A, blocking the legitimate invoke transaction unless the owner pays a higher fee for replacement.
- **Batcher resource waste**: The batcher picks up the forged transaction, runs `__validate__` (which fails), and discards it — consuming execution resources without producing useful work.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The trigger is fully unprivileged:

- The attacker needs only account A's address and the knowledge that a deploy-account transaction is pending — both are publicly observable from the mempool.
- No private key material, no privileged role, and no special network position is required.
- The attack is repeatable: every time account A submits a new deploy-account transaction, the attacker can re-occupy the `nonce = 1` slot.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction exists for the sender address in the mempool. Alternatively, add a new mempool API `deploy_account_tx_in_pool(address) -> bool` that only returns `true` when a `DeployAccount` transaction is present for the given address, and use that in `skip_stateful_validations`.

If a more targeted API is not feasible, the skip should be removed and the UX concern addressed differently (e.g., by having the batcher tolerate a missing account during the deploy-account block and re-validate the invoke in the next block).

---

#    validate_nonce(account_nonce=0, tx_nonce=1) → OK (0 ≤ 1 ≤ gap)

    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
