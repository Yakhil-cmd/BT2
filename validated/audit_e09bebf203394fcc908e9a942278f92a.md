### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing/Presigning Corrupts Sign Outputs — (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate only the upper bound of the caller-supplied `threshold` parameter (`threshold ≤ participants.len()`), but omit the lower-bound check (`threshold ≥ 2`) that is consistently enforced in DKG and triple generation. A caller passing `threshold = 1` (or `0`) bypasses all validation, proceeds into the signing computation with wrong Lagrange coefficients, and produces an unusable/invalid signature with no error returned.

---

### Finding Description

**Root cause — missing lower-bound guard in `assert_sign_inputs`:**

```rust
// src/frost/mod.rs  lines 144–150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← NO check: threshold.value() < 2
``` [1](#0-0) 

The same gap exists in the FROST `presign` entry-point:

```rust
// src/frost/mod.rs  lines 71–77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← NO check: args.threshold.value() < 2
``` [2](#0-1) 

**Contrast with every other protocol entry-point in the library:**

DKG (`assert_key_invariants`, called from `keygen` / `reshare`) explicitly rejects `threshold < 2`: [3](#0-2) 

Triple generation (`validate_triple_inputs`) also rejects `threshold < 2`: [4](#0-3) 

The `ThresholdTooSmall` error variant exists precisely for this purpose: [5](#0-4) 

**Exploit path:**

1. A library caller (or a malicious coordinator that controls the signing session setup) invokes `assert_sign_inputs` or `frost::presign` with `threshold = 1`.
2. Both functions accept the value — the only guard that fires is `threshold > participants.len()`, which is satisfied.
3. The signing computation proceeds using Lagrange coefficients derived for a degree-0 polynomial, while the actual secret shares were generated from a degree-(`t`−1) polynomial (where `t ≥ 2` from keygen).
4. Every participant computes a signature share with the wrong interpolation basis; the aggregated signature is cryptographically invalid.
5. No error is surfaced — the protocol returns `Ok(…)` with a silently corrupted output.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

When `threshold = 1` is accepted at signing time but keygen used `threshold ≥ 2`, the Lagrange interpolation is performed over the wrong polynomial degree. The resulting aggregate signature fails verification against the public key produced by DKG. Honest participants complete the protocol without error yet hold an invalid, unusable signature. This directly matches the allowed High impact: *"Corruption of … sign … outputs so honest parties accept … unusable cryptographic outputs."*

---

### Likelihood Explanation

**Medium.** The entry-point is a public library function callable by any integrator. A misconfigured or adversarial caller can trivially supply `threshold = 1` — no privileged access, leaked key, or cryptographic break is required. The inconsistency with DKG and triple generation (which both enforce the lower bound) makes accidental misconfiguration plausible in addition to deliberate abuse.

---

### Recommendation

Add the same lower-bound guard that DKG and triple generation already enforce, in both `assert_sign_inputs` and `frost::presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place this check immediately after the `threshold.into()` conversion, before the upper-bound check, mirroring `validate_triple_inputs` at `src/ecdsa/ot_based_ecdsa/triples/generation.rs` lines 699–703. [6](#0-5) 

---

### Proof of Concept

```rust
// Pseudocode — no privileged access required
let participants = generate_participants(3);
// Keygen was run with threshold = 2 (degree-1 polynomial)
let keygen_out = run_keygen(&participants, /*threshold=*/2, &mut rng);

// Malicious/misconfigured caller passes threshold = 1 to signing
let args = frost::PresignArguments {
    keygen_out: keygen_out[0].1.clone(),
    threshold: ReconstructionLowerBound::from(1usize), // ← accepted, no error
};
let protocol = frost::presign(&participants, participants[0], &args, rng);
// protocol is Ok(…) — no InitializationError returned

// Downstream: assert_sign_inputs also accepts threshold = 1
let result = assert_sign_inputs(&participants, 1usize, participants[0], participants[1]);
// result is Ok(…)

// The signing session completes but produces an invalid signature
// because Lagrange coefficients are computed for degree-0 instead of degree-1.
```
