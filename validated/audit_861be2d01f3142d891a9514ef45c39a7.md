### Title
`skip_stateful_validations` admits unauthorized invoke transactions by checking any pending mempool entry instead of a deploy-account specifically — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX shortcut for the deploy-account + invoke pattern skips the `__validate__` entry-point call for an invoke transaction with nonce = 1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. That helper returns `true` for **any** transaction from that address in the pool or a recent block, not exclusively a `DeployAccount` transaction. An unprivileged attacker who observes a victim's pending `DeployAccount` in the mempool can therefore submit an invoke with nonce = 1 from the victim's address carrying an invalid (or empty) signature, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements the following logic:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [1](#0-0) 

The code comment explains the intent: "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is circular — those future-nonce transactions may themselves have had validation skipped through this same path.

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for **any** transaction type from that address. There is no check that the matching mempool entry is specifically a `DeployAccount`.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags` and passes the transaction to `StatefulValidator::perform_validations`. For an `Invoke` transaction, `perform_pre_validation_stage` (nonce + fee checks) still runs, but the `__validate__` entry-point call is skipped entirely:
