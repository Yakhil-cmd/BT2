### Title
Missing `max_l2_gas_amount` Upper-Bound Check for `Declare` Transactions in Stateless Validator - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `l2_gas.max_amount` upper-bound check for `Declare` transactions. Any other transaction type (`Invoke`, `DeployAccount`) is rejected if `l2_gas.max_amount > config.max_l2_gas_amount` (default 1,210,000,000). A `Declare` transaction with an arbitrarily large `l2_gas.max_amount` passes the stateless gate, is admitted to the mempool, and is later executed with that inflated value as its Sierra gas limit via `TransactionContext::initial_sierra_gas()`.

### Finding Description

In `validate_resource_bounds`, the upper-bound guard is:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The companion test `valid_l2_gas_amount_on_declare` explicitly asserts that a `Declare` with `max_amount: GasAmount(200)` passes even when `max_l2_gas_amount: 100`, confirming the bypass is intentional but unguarded: [2](#0-1) 

The `max_l2_gas_amount` field exists in `StatelessTransactionValidatorConfig` precisely to cap this value: [3](#0-2) 

When the transaction reaches execution, `TransactionContext::initial_sierra_gas()` returns `l2_gas.max_amount` verbatim for `AllResources` transactions: [4](#0-3) 

This value is used as the gas counter seed for the `__validate_declare__` and `__execute__` entry points. A `Declare` with `l2_gas.max_amount = u64::MAX` would start execution with `GasAmount(u64::MAX)` as its Sierra gas budget.

The only downstream guard that could stop an oversized `Declare` is `verify_can_pay_committed_bounds`, which computes `max_possible_fee` using saturating arithmetic: [5](#0-4) 

For a moderately oversized value (e.g., `max_amount = 2_000_000_000`, `max_price = 8_000_000_000`), `max_possible_fee = 16 × 10^18` STRK — a large but reachable balance for a funded attacker. The balance check passes, the transaction is admitted, and it executes with a gas limit 65% above the intended ceiling.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A `Declare` transaction that violates the operator-configured `max_l2_gas_amount` policy is admitted through the stateless gateway gate without rejection. The transaction then executes with a Sierra gas limit higher than the sequencer operator intended, bypassing the resource-bound admission invariant that applies to every other transaction type.

### Likelihood Explanation

Any unprivileged user can craft a `Declare` V3 transaction with `l2_gas.max_amount` set above `max_l2_gas_amount`. The only prerequisite is holding enough STRK to satisfy `verify_can_pay_committed_bounds` for the inflated `max_possible_fee`. No special access, no privileged role, no dependency on external behavior.

### Recommendation

Apply the same upper-bound check to `Declare` transactions. Remove the type-based branch and enforce the limit uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If a higher limit is legitimately needed for `Declare` transactions (e.g., because class compilation is more expensive), introduce a separate `max_l2_gas_amount_declare` config field rather than leaving the bound entirely absent.

### Proof of Concept

1. Construct an `RpcDeclareTransactionV3` with:
   - `l2_gas.max_amount = GasAmount(2_000_000_000)` (above the default limit of 1,210,000,000)
   - `l2_gas.max_price_per_unit = GasPrice(8_000_000_000)` (at the minimum gas price)
   - A valid Sierra class and `compiled_class_hash`
2. Submit to the gateway's `add_tx` endpoint.
3. Observe: `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for `Declare`.
4. The transaction proceeds to stateful validation. If the sender holds ≥ 16 × 10^18 STRK (to cover `max_possible_fee`), it is admitted to the mempool.
5. When the batcher executes the transaction, `initial_sierra_gas()` returns `GasAmount(2_000_000_000)`, giving the `__validate_declare__` and `__execute__` entry points a gas budget 65% above the operator-configured ceiling.
