### Title
NFT Seller Net Proceeds Displayed as Full Amount Regardless of Royalty — (`packages/gui/src/components/offers/utils.ts`)

### Summary

`calculateNFTRoyalties` in `utils.ts` hardcodes `nftSellerNetAmount = amount` (the full offer amount) instead of `amount - royaltyAmount`, with the correct subtraction explicitly commented out. This causes the GUI to display the full offer amount as "Net Proceeds" to an NFT seller even when the NFT carries a 100% royalty, misleading the seller into accepting an offer where they receive nothing.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `royaltyAmount` correctly but then assigns `nftSellerNetAmount = amount` unconditionally: [1](#0-0) 

The correct deduction (`amount - royaltyAmount - makerFee`) is commented out directly below. This value propagates through two display paths in `NFTOfferDetails` (`NFTOfferViewer.tsx`):

**Path 1 — `overrideNFTSellerAmount`** fed into `OfferSummaryTokenRow`: [2](#0-1) 

`nftSaleInfo?.nftSellerNetAmount` is always `amount` (not reduced), so `overrideNFTSellerAmount = chiaToMojo(amount)` — the full amount — is passed to `OfferSummaryTokenRow`, which uses it directly as the displayed amount: [3](#0-2) 

**Path 2 — "Net Proceeds" label** in the offer details panel: [4](#0-3) 

Both surfaces show the full offer amount as what the seller will receive, regardless of royalty.

### Impact Explanation

An attacker mints an NFT with `royaltyPercentage = 10000` (100%) and their own royalty address. After the NFT reaches a victim's wallet (via sale or transfer), the victim reviews a `TokenForNFT` offer in the GUI. The "Net Proceeds" display shows the full offer amount (e.g., 10 XCH) instead of 0. The victim accepts the offer. The blockchain enforces the 100% royalty: the entire 10 XCH is routed to the attacker's royalty address; the victim receives 0. This is a direct, concrete financial loss caused by the GUI displaying the wrong amount at the offer-acceptance decision point.

This fits the **High** impact category: NFT metadata (`royaltyPercentage`) causes the GUI to display the wrong amount, causing a user to approve a transaction with incorrect proceeds expectations.

### Likelihood Explanation

- Minting an NFT with arbitrary `royaltyPercentage` is a standard, permissionless on-chain operation — no special privileges required.
- The `showRoyaltyWarning` flag fires at ≥ 20% and shows a tooltip icon, but does **not** correct the "Net Proceeds" figure; the misleading number remains.
- The victim only needs to own the NFT and open any `TokenForNFT` offer for it in the GUI.
- The commented-out correct formula confirms this was a known, intentional (or accidentally regressed) code change, not a theoretical edge case.

### Recommendation

Restore the correct `nftSellerNetAmount` calculation in `calculateNFTRoyalties`:

```ts
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

Additionally, add a guard in the offer-acceptance UI that blocks or prominently warns when `nftSellerNetAmount <= 0`.

### Proof of Concept

```ts
import { calculateNFTRoyalties } from './utils';
import { NFTOfferExchangeType } from '../../@types/NFTOfferExchangeType';

const result = calculateNFTRoyalties(
  10,        // 10 XCH offer
  0,         // no maker fee
  100,       // 100% royalty (royaltyPercentage=10000 → convertRoyaltyToPercentage → 100)
  NFTOfferExchangeType.TokenForNFT,
);

// Bug: nftSellerNetAmount === 10 (full amount)
// Expected: nftSellerNetAmount === 0 (amount - royaltyAmount)
assert(result.nftSellerNetAmount === 0); // FAILS — returns 10
assert(result.royaltyAmount === 10);     // PASSES — royalty is correctly 10
```

The GUI then calls `chiaToMojo(10)` and displays it as "Net Proceeds", while the blockchain sends all 10 XCH to the attacker's royalty address.
