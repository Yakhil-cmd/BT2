Looking at the external report's vulnerability class — **missing bounds validation on a configuration parameter causing a state-checking function to return incorrect results, leading to downstream protocol failures** — I need to find an analog in the threshold-signatures codebase.

Let me examine the key validation functions across the codebase.

**Key observation**: `assert_key_invariants` in `src/dkg.rs` enforces **both** a lower bound and an upper bound on `threshold`:

The same gap exists in `src/ecdsa/ot_based_ecdsa/sign.rs` (`sign` function), which checks `participants.len() < threshold` but has no `threshold < 2` guard.

### Title
Missing Lower Bound Validation on `threshold` in `assert_sign_inputs` Allows Malicious Coordinator to Corrupt FROST Signing — (File: `src/frost/mod.rs`)

### Summary
`assert_sign_inputs` validates that `threshold` does not exceed the participant count but omits the lower bound check (`threshold >= 2`) that `assert_key_invariants` enforces. A malicious coordinator can supply `threshold = 1` (or `0`) to the FROST sign entry point; the function returns `Ok` instead of an error, the signing protocol proceeds with a degenerate threshold, Lagrange interpolation uses only one participant's share from a degree-`(t-1)` polynomial, and the resulting signature is cryptographically invalid — permanently denying signing for honest parties in that session.

### Finding Description
`assert_key_invariants` (`src/dkg.rs`, lines 572–582) enforces both bounds:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { threshold, max: participants.len() });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` (`src/frost/mod.rs`, lines 144–150) only enforces the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

There is no corresponding `ThresholdTooSmall` guard. Passing `threshold = 1` returns `Ok(participants)` without error. [3](#0-2) 

The same omission exists in the OT-based ECDSA `sign` entry point, which checks `participants.len() < threshold` but never checks `threshold < 2`: [4](#0-3) 

Key shares produced by DKG are evaluations of a degree-`(t-1)` polynomial (where `t` is the keygen threshold, `t >= 2`). When the signing call uses `threshold = 1`, the Lagrange interpolation collects only one share and assigns it coefficient `1`, reconstructing a value that is **not** the secret key scalar. The resulting signature fails verification. [5](#0-4) 

### Impact Explanation
**High — Corruption of sign outputs / Permanent denial of signing.** Honest participants complete the FROST signing round and accept the coordinator's combined output, but the signature is cryptographically invalid and will be rejected by any verifier. The session is unrecoverable without restarting with a correct threshold. This matches the allowed impact: *"Corruption of … sign … outputs so honest parties accept … unusable cryptographic outputs"* and *"Permanent denial of signing … for honest parties under valid protocol inputs."*

### Likelihood Explanation
**Medium.** The coordinator is the party that supplies the `threshold` argument to the sign entry point. A malicious or compromised coordinator can trivially pass `threshold = 1` (or `0`). No cryptographic capability is required — only the ability to call the public API with an out-of-range integer. The `ReconstructionLowerBound` wrapper type imposes no minimum at construction time: [6](#0-5) 

### Recommendation
Add the same lower bound guard present in `assert_key_invariants` to `assert_sign_inputs` and to the OT-based ECDSA `sign` function:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This makes the signing entry points consistent with the keygen entry points and prevents degenerate threshold values from silently propagating into the protocol.

### Proof of Concept
1. Run DKG with `threshold = 3`, `N = 5` participants → produces valid key shares (degree-2 polynomial evaluations).
2. Call `assert_sign_inputs(participants, /*threshold=*/1, me, coordinator)` → returns `Ok(participants)` with no error (the `ThresholdTooSmall` branch is absent).
3. The FROST signing protocol executes with `threshold = 1`; the coordinator collects one signature share and applies Lagrange coefficient `1`.
4. The combined signature `z = z_i` does not equal the correct aggregate `z = Σ λ_i · z_i` over all signers.
5. Signature verification fails; honest parties have produced an unusable output and cannot sign the message in this session. [3](#0-2) [7](#0-6)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
