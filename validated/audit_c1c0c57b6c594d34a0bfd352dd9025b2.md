### Title
Unbacked ERC-20 Token Minting via Unauthenticated Direct `ft_on_transfer` Call - (File: engine/src/contract_methods/connector.rs)

### Summary
Any NEAR account can call `ft_on_transfer` directly on the Aurora Engine to mint ERC-20 mirror tokens without ever depositing the corresponding NEP-141 tokens. Because `deploy_erc20_token` also has no caller access control, an attacker can register their own NEAR account as a NEP-141 token and then call `ft_on_transfer` directly to mint an arbitrary quantity of the corresponding ERC-20 tokens on Aurora, creating an unbacked supply and enabling insolvency of that token's bridge accounting.

### Finding Description

**Root cause 1 — `ft_on_transfer` performs no transfer verification.**

`ft_on_transfer` is the NEP-141 receiver callback. In the intended flow it is invoked by a NEP-141 token contract *after* it has already moved tokens to Aurora. Aurora Engine, however, places no restriction on who may call this entry point. The only guard is `require_running`:

```rust
// engine/src/contract_methods/connector.rs  lines 62-109
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;          // only liveness check
        let predecessor_account_id = env.predecessor_account_id();
        ...
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(          // mints ERC-20 tokens
                &predecessor_account_id,          // treated as the NEP-141 token contract
                &args,
                &current_account_id,
                handler,
            )
        };
```

`predecessor_account_id` is the NEAR runtime-supplied caller identity. Any NEAR account that is registered as a NEP-141 token on Aurora will cause `receive_erc20_tokens` to be executed, minting ERC-20 tokens equal to `args.amount` — a value supplied entirely by the caller — with no on-chain proof that any NEP-141 tokens were actually transferred.

**Root cause 2 — `deploy_erc20_token` has no access control.**

The function that registers a NEP-141 ↔ ERC-20 mapping on Aurora carries only a liveness check; there is no `require_owner_only` guard:

    io: I, env: &E, handler: &mut H,
