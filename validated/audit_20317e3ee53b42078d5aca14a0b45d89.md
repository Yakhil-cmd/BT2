### Title
Zero Native Resource Charge for All EIP-2537 BLS12-381 Precompiles Enables Prover Resource Exhaustion - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

All seven EIP-2537 BLS12-381 precompile implementations hardcode `cost_native = 0`, charging zero native (proving) resource for computationally expensive elliptic-curve operations. This is structurally analogous to the reported proxy vulnerability: just as the proxy forwarded the HTTP `Link` header without filtering (bypassing CSP), ZKsync OS forwards BLS12-381 precompile calls through the resource accounting layer without charging the native dimension, bypassing the native resource limit that is the primary guard against prover overload.

---

### Finding Description

ZKsync OS implements a **double resource accounting** model: every operation charges both EVM gas (ergs) and a **native resource** that models actual RISC-V proving cost. The native limit per transaction is `nativeLimit = gasLimit × nativePerGas`. If a transaction exhausts its native budget, it reverts; if the block exhausts `MAX_NATIVE_COMPUTATIONAL` (`2^35`), proving fails.

All other computationally heavy precompiles correctly charge native resources:

| Precompile | Native cost |
|---|---|
| `ecrecover` | `native_with_delegations!(350_000, 43_000, 0)` |
| `BN254 ecadd` | `native_with_delegations!(46_000, 1_650, 0)` |
| `BN254 ecmul` | `native_with_delegations!(600_000, 41_000, 0)` |
| `BN254 pairing` | `native_with_delegations!(13_000_000, 500_000, 0)` per pair |
| `KZG point evaluation` | `native_with_delegations!(49_900_000, 3_301_000, 0)` | [1](#0-0) 

However, every BLS12-381 precompile sets `cost_native = 0`:

**G1 Addition** (`Bls12381G1AdditionPrecompile`): [2](#0-1) 

**G2 Addition** (`Bls12381G2AdditionPrecompile`): [3](#0-2) 

**G1 MSM** (`Bls12381G1MSMPrecompile`): [4](#0-3) 

**G2 MSM** (`Bls12381G2MSMPrecompile`): [5](#0-4) 

**Pairing Check** (`Bls12381PairingCheckPrecompile`): [6](#0-5) 

**G1 Mapping** (`Bls12381G1MappingPrecompile`): [7](#0-6) 

**G2 Mapping** (`Bls12381G2MappingPrecompile`): [8](#0-7) 

The native resource is the mechanism that ensures the prover's actual RISC-V cycle budget is not exceeded. BLS12-381 pairing and MSM are among the most expensive elliptic-curve operations in existence — significantly more expensive than BN254 pairing, which already costs `~13M + 500K×4 = 15M` native units per pair. With `cost_native = 0`, the prover must execute the actual BLS12-381 arithmetic without any native budget being consumed, creating a **forward/proving divergence**: the forward system (sequencer) accepts the transaction as within resource limits, but the prover must spend far more RISC-V cycles than the native accounting records.

The native resource model is documented as the primary guard: [9](#0-8) 

The block-level cap `MAX_NATIVE_COMPUTATIONAL = 2^35` is the ceiling: [10](#0-9) 

---

### Impact Explanation

An attacker submits a transaction that calls BLS12-381 pairing check with the maximum number of pairs that fit within the block gas limit (~30M gas). The EVM gas cost per pair is `32,600 gas`, so approximately `(30M − 37,700) / 32,600 ≈ 919` pairs can be packed into a single transaction. Each pair requires a full BLS12-381 Miller loop and final exponentiation in RISC-V — operations that are substantially more expensive than BN254 pairing. Since `cost_native = 0`, the transaction's native budget is untouched, and the block's `MAX_NATIVE_COMPUTATIONAL` counter is not incremented by these operations.

The prover must execute all 919 pairings. If each BLS12-381 pairing costs even half of what BN254 pairing costs (~7.5M native units), 919 pairs = ~6.9B native units — roughly 20% of the entire block native budget consumed by a single transaction that the accounting system recorded as consuming zero native resources. A block filled with such transactions would force the prover to execute orders of magnitude more work than the native accounting permits, causing:

1. **Valid-execution unprovability**: blocks accepted by the sequencer cannot be proven within the prover's cycle budget.
2. **Prover DoS**: sustained submission of BLS12-381-heavy transactions forces the prover into an overloaded state.
3. **Block finality delay**: if proving fails or takes excessive time, L2 finality is disrupted.

---

### Likelihood Explanation

- BLS12-381 precompiles (EIP-2537) are live in the Prague hardfork and callable by any unprivileged EOA or contract.
- No special role, key, or governance access is required — a standard EVM `CALL` to addresses `0x0b`–`0x11` suffices.
- The attacker only needs to pay EVM gas (which is correctly charged); the native resource is free.
- The attack is repeatable every block and requires no coordination.

---

### Recommendation

Benchmark BLS12-381 operations on the RISC-V target and assign non-zero native costs analogous to the BN254 precompiles. At minimum, the pairing check and MSM operations (the most expensive) must have native costs proportional to the number of pairs/points. The existing `native_with_delegations!` macro and the pattern used in `BN254_PAIRING_BASE_NATIVE_COST` / `BN254_PAIRING_PER_PAIR_NATIVE_COST` should be applied to all seven BLS12-381 precompiles.

The existing TODO comment in `cost_constants.rs` (`// TODO(EVM-1178) Add more vectors and benchmark cost better`) for `POINT_EVALUATION_NATIVE_COST` confirms the team is aware of the need to benchmark and assign native costs — the same process must be applied to BLS12-381. [11](#0-10)

### Proof of Concept

```
Attacker EOA → CALL 0x0f (BLS12_PAIRING_CHECK)
  input: 919 × (G1_point || G2_point)  [valid on-curve points]
  gas:   30_000_000
  
Forward system result:
  EVM gas charged:  37,700 + 919 × 32,600 = 29,999,100  ✓ (within limit)
  Native charged:   0                                     ← BUG
  
Prover actual work:
  919 BLS12-381 Miller loops + final exponentiations
  Estimated RISC-V cycles: >> MAX_NATIVE_COMPUTATIONAL / block_tx_count
  
Block filled with N such transactions:
  Native accounted: 0
  Native actually consumed by prover: N × ~6.9B units
  → Prover cycle budget exceeded → proving failure / DoS
```

```markdown
## Native resource

The native resource models the offchain cost of processing a transaction. Currently, this is dominated by proving and publishing data. A good intuition for it is "how many RISC-V cycles it takes to prove a given computation".

If a transaction execution runs out of native resources, the entire transaction is reverted. If the same happens during transaction validation, the transaction is considered invalid.

The native resources are passed fully from frame to frame, a call cannot set a limit on how much of it the callee can spend.
```

/// Maximum amount of computational native resources a full program
/// execution (be it block or batch) can spend.
/// Actual limit is the largest multiple of 2^22 - 1 (segment limit)
/// that is less than 2^36. To be safe, in case our native model
/// is not upper-bounding for some corner cases, we set this limit
/// conservatively to 2^35.
///
pub const MAX_NATIVE_COMPUTATIONAL: u64 = 1 << 35;
