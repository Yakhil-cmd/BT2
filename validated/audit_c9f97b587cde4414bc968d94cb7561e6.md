### Title
`overrideNFTSellerAmount` Applied to All Token Rows in Multi-Asset NFT Offers, Enabling Amount Spoofing — (`packages/gui/src/components/offers/OfferSummaryRow.tsx`, `packages/gui/src/components/offers/NFTOfferViewer.tsx`)

---

### Summary

`NFTOfferSummaryRow` passes `overrideNFTSellerAmount` to every `OfferSummaryTokenRow` it renders, regardless of which asset is the NFT price asset. When a crafted offer contains multiple token assets (e.g., CAT + XCH) on the buyer's side, the override — computed from the CAT amount in CAT-mojos — is applied verbatim to the XCH row, which interprets those mojos as XCH-mojos. The resulting display can show a dramatically inflated XCH amount, causing the NFT seller to accept an offer for far less XCH than displayed.

---

### Finding Description

**Root cause — `OfferSummaryTokenRow` receives the override unconditionally:** [1](#0-0) 

Every `CHIA` or `TOKEN` asset in the summary gets the same `overrideNFTSellerAmount` prop.

**Root cause — `OfferSummaryTokenRow` replaces its amount with the override without checking `assetId`:** [2](#0-1) 

`const amount = overrideNFTSellerAmount ?? originalAmount;` — no guard on which asset this applies to.

**How `overrideNFTSellerAmount` is computed:** [3](#0-2) 

It is set only for `TokenForNFT` exchange type. The value is `catToMojo(nftSaleInfo.nftSellerNetAmount)` when the price asset is a CAT.

**How the price asset is selected — TOKEN is preferred over CHIA:** [4](#0-3) 

`getNFTPriceWithoutRoyalties` iterates `[TOKEN, CHIA]` and returns the first match. In a multi-token offer (CAT + XCH), the CAT is always selected as the price asset.

**`nftSellerNetAmount` equals the raw price amount (royalty subtraction is commented out):** [5](#0-4) 

So `overrideNFTSellerAmount = catToMojo(CAT_amount)`.

---

### Impact Explanation

**Concrete attack scenario:**

1. Attacker (buyer) crafts an offer file: NFT (requested) for **1,000,000 CAT + 0.001 XCH** (offered).
2. Victim (NFT seller) imports the offer. They own the NFT, so `useNFT` returns it and `nftSaleInfo` is computed.
3. `getNFTPriceWithoutRoyalties` picks the CAT asset (TOKEN preferred over CHIA).
4. `overrideNFTSellerAmount = catToMojo(1,000,000) = 1,000,000,000 mojos`.
5. Both the CAT row and the XCH row receive this override:
   - CAT row: `mojoToCATLocaleString(1,000,000,000)` → **1,000,000 CAT** ✓ (correct)
   - XCH row: `mojoToChiaLocaleString(1,000,000,000)` → **1 XCH** ✗ (actual: 0.001 XCH)
6. Victim sees "You will receive: 1,000,000 CAT + **1 XCH**" and accepts.
7. Victim actually receives 1,000,000 CAT + **0.001 XCH** — a loss of 0.999 XCH.

The attacker controls the ratio of CAT to XCH in the offer to tune how inflated the XCH display appears. The entrypoint is a standard offer file import, requiring no special privileges.

This satisfies the **High** impact criterion: spoofing of offer state causes the user to display the wrong amount and approve a transaction for the wrong value.

---

### Likelihood Explanation

- Crafting a multi-asset offer file is trivial; the Chia protocol supports it natively.
- The victim only needs to import and accept the offer through the standard GUI workflow.
- The bug triggers whenever the victim owns the NFT and the offer has ≥2 token assets on the buyer's side.
- No leaked keys, local compromise, or social engineering beyond sharing an offer file is required.

---

### Recommendation

In `NFTOfferSummaryRow`, pass `overrideNFTSellerAmount` only to the row whose `assetId` matches the price asset returned by `getNFTPriceWithoutRoyalties`. All other token rows should receive `undefined` (falling back to `originalAmount`).

```tsx
// NFTOfferViewer.tsx — NFTOfferSummaryRow
case OfferAsset.CHIA:
case OfferAsset.TOKEN:
  return (
    <OfferSummaryTokenRow
      assetId={assetId}
      amount={summaryData[assetId]}
      overrideNFTSellerAmount={
        assetId === nftPriceAssetId   // <-- guard added
          ? overrideNFTSellerAmount
          : undefined
      }
    />
  );
```

`nftPriceAssetId` should be derived from `getNFTPriceWithoutRoyalties(summary)?.assetId` and threaded through `NFTOfferSummaryRow`'s props.

---

### Proof of Concept

Render `NFTOfferSummary` with:
- `summary.offered = { "<cat_asset_id>": 1_000_000_000 /* 1M CAT mojos */, "xch": 1_000_000_000 /* 0.001 XCH */ }`
- `summary.requested = { "<nft_launcher_id>": 1 }`
- `overrideNFTSellerAmount = 1_000_000_000` (catToMojo(1000) for example)

Assert: the XCH `OfferSummaryTokenRow` renders `mojoToChia(1_000_000_000) = 0.000000001 XCH` instead of `mojoToChia(1_000_000_000) = 0.001 XCH` — demonstrating the override is applied to the wrong row.

export function getNFTPriceWithoutRoyalties(
  summary: OfferSummaryRecord,
): GetNFTPriceWithoutRoyaltiesResult | undefined {
  for (const assetType of [OfferAsset.TOKEN, OfferAsset.CHIA]) {
    const assetId = offerAssetIdForAssetType(assetType, summary);
    if (assetId) {
      const amountInMojos = offerAssetAmountForAssetId(assetId, summary);
      if (amountInMojos) {
        const amountInTokens = assetType === OfferAsset.CHIA ? mojoToChia(amountInMojos) : mojoToCAT(amountInMojos);
        return { amount: amountInTokens.toNumber(), assetId, assetType };
      }
    }
  }

  return undefined;
}
