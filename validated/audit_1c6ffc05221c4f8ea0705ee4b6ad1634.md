### Title
`estimate_minimal_gas_vector` omits `n_compiled_class_hash_updates` for Declare V2/V3, underestimating DA gas and admitting transactions that will fail during sequencing — (`crates/blockifier/src/fee/gas_usage.rs`)

---

### Summary

`estimate_minimal_gas_vector` hard-codes `n_compiled_class_hash_updates: 0` for every `Declare` transaction variant. Declare V2 and V3 transactions actually write one compiled-class-hash entry to the DA segment (`n_compiled_class_hash_updates: 1`), which contributes **2 additional felts** to the on-chain data segment. The undercount causes `check_fee_bounds` — the pre-validation gate used by both the gateway and the sequencer — to accept Declare V2/V3 transactions whose resource bounds are below the true minimum. Those transactions are admitted to the mempool, pass the sequencer's own pre-validation, and then fail at post-execution with the sequencer undercompensated.

---

### Finding Description

**Root cause — wrong state-change count in the estimator**

`estimate_minimal_gas_vector` builds a `StateChangesCount` for each transaction type and uses it to compute the minimum DA gas and KZG-DA VM steps:

```rust
// crates/blockifier/src/fee/gas_usage.rs  lines 168-174
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,   // ← wrong for V2/V3
    n_modified_contracts: 1,
},
```

The TODO comment is the developer's own acknowledgement of the bug. The correct value for Declare V2/V3 is `1`, as confirmed by the test helper that records the *actual* state changes produced by execution:

```rust
// crates/blockifier/src/transaction/transactions_test.rs  lines 1846-1852
} else if version == TransactionVersion::TWO || version == TransactionVersion::THREE {
    StateChangesCount {
        n_storage_updates: 1,
        n_modified_contracts: 1,
        n_compiled_class_hash_updates: 1, // Also set compiled class hash.
        ..StateChangesCount::default()
    }
```

**Propagation through the gas calculation**

`get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` felts to the DA segment:

```rust
// crates/blockifier/src/fee/gas_usage.rs  lines 33-34
// For each compiled class updated (through declare): class_hash, compiled_class_hash
onchain_data_segment_length += state_changes_count.n_compiled_class_hash_updates * 2;
```

With `n_compiled_class_hash_updates: 0`, the segment is 2 felts shorter than reality. This feeds into two downstream calculations inside `estimate_minimal_gas_vector`:

1. `get_da_gas_cost(&state_changes_by_account_tx, block_info.use_kzg_da)` — the blob-gas (KZG DA) or calldata-gas cost is underestimated by `2 × DATA_GAS_PER_FIELD_ELEMENT` (KZG) or `2 × GAS_PER_MEMORY_WORD` (calldata).
2. `versioned_constants.os_kzg_da_resources(data_segment_length).n_steps` — the VM-step overhead for the KZG commitment is also underestimated.

**How the underestimate reaches admission**

`check_fee_bounds` calls `estimate_minimal_gas_vector` and rejects a transaction only when `minimal_gas_amount > resource_bounds.max_amount`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 374-382
fn check_fee_bounds(&self, tx_context: &TransactionContext) -> ... {
    let minimal_gas_amount_vector = estimate_minimal_gas_vector(
        &tx_context.block_context,
        self,
        &tx_context.get_gas_vector_computation_mode(),
    );
    ...
```

`check_fee_bounds` is called from `perform_pre_validation_stage`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 363-366
if self.execution_flags.charge_fee {
    self.check_fee_bounds(tx_context)?;
    verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
}
```

The gateway's `run_validate_entry_point` constructs an `AccountTransaction` and calls `blockifier_validator.validate(account_tx)`, which internally calls `perform_pre_validation_stage`. Because the minimum is underestimated, a Declare V2/V3 transaction whose `l1_data_gas.max_amount` (KZG DA path) or `l1_gas.max_amount` (calldata DA path) sits between the underestimated minimum and the true minimum passes this gate and is admitted to the mempool.

**Execution outcome**

Declare transactions are non-revertible (`is_non_revertible` returns `true` for `Transaction::Declare`). When the actual gas vector computed during execution exceeds the resource bounds, `PostExecutionReport` detects the overdraft. The fee charged is capped at the resource bounds — not the actual gas consumed — so the sequencer is undercompensated for the DA work it performed.

---

### Impact Explanation

**High — mempool/gateway admission accepts invalid Declare V2/V3 transactions.**  
A transaction whose resource bounds are insufficient to cover the true DA cost of a Declare V2/V3 passes `check_fee_bounds` at both the gateway and the sequencer. The transaction is sequenced, fails at post-execution, and the sequencer collects less fee than the actual cost of the operation. This is a direct analog to the MozBridge underestimation: the fee-check uses a shorter/incomplete payload (missing the compiled-class-hash DA entry) instead of the actual one.

---

### Likelihood Explanation

Any unprivileged user can submit a Declare V2/V3 transaction. The window of exploitable resource bounds is narrow (2 felts of DA gas), but it is deterministic and reproducible. The TODO comment confirms the team is aware the value is wrong, meaning the bug is present in the current production code path.

---

### Recommendation

In `estimate_minimal_gas_vector`, distinguish Declare V2/V3 from V0/V1 and set `n_compiled_class_hash_updates: 1` for the former:

```rust
Transaction::Declare(tx) => {
    let n_compiled_class_hash_updates =
        if tx.version() >= TransactionVersion::TWO { 1 } else { 0 };
    StateChangesCount {
        n_storage_updates: 1,
        n_class_hash_updates: 0,
        n_compiled_class_hash_updates,
        n_modified_contracts: if tx.version() == TransactionVersion::ZERO { 0 } else { 1 },
    }
}
```

This aligns the estimator with `declare_expected_state_changes_count` and removes the gap that allows under-resourced Declare V2/V3 transactions to pass pre-validation.

---

### Proof of Concept

1. Observe that `estimate_minimal_gas_vector` returns a `GasVector` with `l1_data_gas` underestimated by `2 × DATA_GAS_PER_FIELD_ELEMENT` for any Declare V2/V3 transaction (KZG DA path).
2. Construct a Declare V3 transaction with `l1_data_gas.max_amount` set to exactly the value returned by `estimate_minimal_gas_vector` (the underestimated minimum).
3. Submit it to the gateway. `check_fee_bounds` passes because `minimal_gas_amount == resource_bounds.max_amount`.
4. The transaction is admitted to the mempool and forwarded to the sequencer.
5. During execution, `TransactionReceipt::from_account_tx` computes the true DA gas using the actual `StateChangesCount` (which includes `n_compiled_class_hash_updates: 1`), producing a `gas.l1_data_gas` that is 2 units higher than the bound.
6. `PostExecutionReport` detects the overdraft; the fee charged equals `l1_data_gas.max_amount × l1_data_gas_price`, not the true cost — the sequencer is undercompensated by `2 × DATA_GAS_PER_FIELD_ELEMENT × l1_data_gas_price`.
