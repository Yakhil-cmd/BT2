### Title
Commented-Out Royalty Subtraction in `calculateNFTRoyalties` Causes `nftSellerNetAmount` to Equal Full Pre-Royalty Amount, Misleading NFT Sellers into Accepting Underpaying Offers — (`packages/gui/src/components/offers/utils.ts`)

### Summary

The root cause is a commented-out subtraction in `calculateNFTRoyalties`. `nftSellerNetAmount` is assigned `amount` (the full offered price) instead of `amount - royalties`. This incorrect value propagates to two distinct display surfaces in `NFTOfferDetails`: the "You will receive" summary row (via `overrideNFTSellerAmount`) and the explicitly-labeled "Net Proceeds" section. Both tell the seller they will receive the full pre-royalty amount, when they will actually receive less.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The subtraction of `royaltyAmountString` and `makerFee` is commented out, so `nftSellerNetAmount === amount` unconditionally, regardless of royalty percentage.

This value flows into `NFTOfferDetails` via `nftSaleInfo`:

```typescript
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)  // equals full amount
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

`overrideNFTSellerAmount` is passed to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it replaces the displayed token amount:

```typescript
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [3](#0-2) 

Additionally, the "Net Proceeds" section in `NFTOfferDetails` renders `nftSaleInfo?.nftSellerNetAmount` under a label whose tooltip explicitly states *"The net proceeds include the asking price, minus any associated creator fees"* — but the value shown is the full pre-royalty amount: [4](#0-3) 

### Impact Explanation

For a `TokenForNFT` offer with a 10% royalty on a 1 XCH offer, the seller sees:

| UI Label | Displayed | Actual |
|---|---|---|
| "You will receive" (summary row) | 1 XCH | 0.9 XCH |
| "Net Proceeds" (explicit label) | 1 XCH | 0.9 XCH |
| "Creator Fee (10%)" | 0.1 XCH | 0.1 XCH ✓ |

The seller is shown two separate UI surfaces both asserting they will receive the full 1 XCH. The "Net Proceeds" label carries an explicit tooltip promising the royalty-subtracted value. A seller who trusts this UI accepts the offer and receives 0.9 XCH — a direct, concrete financial loss of 0.1 XCH relative to what the UI represented.

### Likelihood Explanation

Any buyer can craft a `TokenForNFT` offer for an NFT with a nonzero royalty. The seller views the offer through the standard import/accept flow. No special privileges or local access are required. The higher the royalty percentage, the larger the discrepancy between displayed and actual proceeds.

### Recommendation

Restore the commented-out subtraction in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString)).toFixed(12),
);
``` [1](#0-0) 

The `makerFee` subtraction should be evaluated separately — for `TokenForNFT`, the maker fee is paid by the buyer on top of the offer, not deducted from the seller's proceeds, so only `royaltyAmount` should be subtracted from `nftSellerNetAmount`.

### Proof of Concept

Render `NFTOfferDetails` with:
- `exchangeType = TokenForNFT`
- `amount = 1` XCH
- `nft.royaltyPercentage = 1000` (10%)

Assert that the "Net Proceeds" typography and the `OfferSummaryTokenRow` both display `0.9 XCH`. With the current code, both display `1 XCH` because `nftSellerNetAmount = amount = 1` due to the commented-out subtraction.
