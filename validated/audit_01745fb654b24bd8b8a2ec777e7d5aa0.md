### Title
Missing Caller Validation in `update_rewards` Allows Monopolization of Block Reward Slots, Permanently Freezing Unclaimed Yield for All Other Stakers - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract performs no check on who the caller is. A global `last_reward_block` guard allows only one reward-update call per block. An attacker who registers as a staker with the minimum required stake can call `update_rewards` for their own staker address in every block, consuming the single allowed slot and permanently preventing all other stakers from receiving their consensus-era block rewards.

---

### Finding Description

`update_rewards` is the V3 (consensus rewards) entry point for distributing per-block staking rewards. It is declared as a public ABI function with no role restriction and no check that the caller is the staker or the staker's operational address. [1](#0-0) 

The only guard against repeated calls is a **global** `last_reward_block` storage variable: [2](#0-1) 

This variable is written once per successful call and is shared across all stakers: [3](#0-2) 

Because the guard is global and the function accepts an arbitrary `staker_address` parameter, any caller can invoke `update_rewards(attacker_staker_address, false)` in every block. Each such call:

1. Passes all validation checks (staker exists, is active, has non-zero balance).
2. Writes `current_block_number` into `last_reward_block`.
3. Distributes block rewards to the attacker's staker via `_update_rewards`.

Every subsequent call in the same block from any other staker reverts with `REWARDS_ALREADY_UPDATED`. The reward slot for that block is consumed.

The reward calculation in V3 uses the **staker's own total balance** (not the global total stake) as the denominator: [4](#0-3) 

So the attacker receives `block_rewards × own_balance / staker_total_balance` per block — their proportional share within their own stake — while all other stakers receive zero for every block the attacker calls first.

The V2 path (`update_rewards_from_attestation_contract`) is correctly gated by both an attestation-contract caller check and a `is_pre_consensus()` guard: [5](#0-4) 

The V3 path has neither.

---

### Impact Explanation

**Permanent freezing of unclaimed yield** (High).

Once the attacker begins calling `update_rewards` every block, all other stakers' block rewards are never credited. The `unclaimed_rewards_own` field of every other staker remains at zero indefinitely. Because the attacker has an economic incentive (they receive their own proportional rewards), they will sustain the attack indefinitely. The frozen yield is never recovered unless the attacker voluntarily stops or the contract is paused/upgraded.

---

### Likelihood Explanation

**High.** The attack requires only:
- Registering as a staker with the minimum stake (`min_stake`).
- Calling `update_rewards(attacker_address, false)` once per block via a simple bot.

No privileged role, no leaked key, no external dependency. The attacker's cost is the minimum stake plus gas; their reward is their proportional block rewards. The attack is profitable and trivially automated.

---

### Recommendation

Add a caller restriction to `update_rewards` so that only the staker's registered operational address (or the staker address itself) may invoke it for a given `staker_address`. This mirrors the access pattern already enforced in `update_rewards_from_attestation_contract` via `assert_caller_is_attestation_contract`.

Alternatively, replace the global `last_reward_block` with a per-staker `last_reward_block` mapping, so that each staker's reward slot is independent and cannot be consumed by another staker's call.

---

### Proof of Concept

1. Attacker calls `stake(reward_addr, op_addr, min_stake)` to register as a staker. After K epochs, their balance becomes effective.
2. Each block N (after `consensus_rewards_first_epoch`), attacker calls `update_rewards(attacker_staker_address, false)`.
3. The check `current_block_number > last_reward_block` passes; `last_reward_block` is set to N.
4. `_update_rewards` credits `block_rewards × own_balance / staker_total_balance` to the attacker's `unclaimed_rewards_own`.
5. Any legitimate staker who calls `update_rewards` in block N receives `REWARDS_ALREADY_UPDATED` and gets zero rewards.
6. Repeated every block: all other stakers' `unclaimed_rewards_own` remain permanently at zero. [6](#0-5)

    impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
