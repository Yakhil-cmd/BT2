### Title
`skip_stateful_validations` admits invoke transactions with arbitrary signatures for any undeployed account that has a deploy-account tx in the mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX shortcut `skip_stateful_validations` is intended to let a user submit a `deploy_account` + `invoke` (nonce=1) pair simultaneously, skipping `__validate__` on the invoke because the account does not yet exist on-chain. The guard that enables the skip — `account_tx_in_pool_or_recent_block` — only checks whether *any* transaction from that address is present in the mempool or a recent committed block. It does not verify that the caller of the invoke is the same party that submitted the deploy-account. An unprivileged attacker who observes a victim's `deploy_account` tx in the mempool can therefore submit an invoke tx with nonce=1 for the victim's address carrying an **arbitrary/invalid signature**, and the gateway will admit it to the mempool without ever calling `__validate__`.

---

### Finding Description

**Exact code path**

`extract_state_nonce_and_run_validations` reads the on-chain nonce, calls `run_pre_validation_checks`, and then calls `run_validate_entry_point(executable_tx, skip_validate)`. [1](#0-0) 

`run_pre_validation_checks` calls `skip_stateful_validations`, which returns `true` (skip `__validate__`) when all three conditions hold:

```
tx is Invoke  AND  tx.nonce() == 1  AND  account_nonce == 0
AND  mempool.account_tx_in_pool_or_recent_block(sender) == true
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the address has *any* tx in the pool or any committed nonce in the mempool state — it does not check who submitted that tx or whether it is a deploy-account: [3](#0-2) 

When `skip_validate=true`, `run_validate_entry_point` sets `validate=false` in the `ExecutionFlags`, so the blockifier's `StatefulValidator::validate` is called with `validate=false` and `__validate__` is never executed: [4](#0-3) 

The mempool's own `validate_tx` only checks nonce ordering and fee escalation — it never inspects the signature: [5](#0-4) 

**Attack steps**

1. Victim V submits a valid `deploy_account` tx for address A. It enters the mempool pool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker submits an `Invoke` tx for address A with `nonce=1` and an **invalid/arbitrary signature**.
3. Gateway stateful validation:
   - `account_nonce` from state = 0 (A not yet deployed).
   - `validate_nonce`: `0 ≤ 1 ≤ 0 + max_gap` → passes.
   - `validate_by_mempool`: nonce not too old → passes.
   - `skip_stateful_validations`: nonce=1, account_nonce=0, account_in_pool=true → returns `true`.
   - `run_validate_entry_point` with `validate=false` → `__validate__` **not called**.
4. The invalid invoke tx is admitted to the mempool and occupies the nonce=1 slot for address A.

**Broken invariant**

Every invoke transaction admitted to the mempool must have its `__validate__` entry point successfully executed (signature verified). The skip-validate path breaks this invariant for any undeployed account whose deploy-account tx is visible in the mempool.

---

### Impact Explanation

The admitted invalid tx will fail during actual block execution because `AccountTransaction::new_for_sequencing` always sets `validate=true`: [6](#0-5) 

Concrete consequences:

- **Nonce-slot squatting**: The attacker occupies the nonce=1 slot for the victim's address. If the victim also submits a legitimate invoke tx with nonce=1, the victim must pay a higher fee (fee escalation) to replace the attacker's tx, or wait for the attacker's tx to be evicted.
- **Sequencer resource waste**: The invalid tx is pulled from the mempool by the batcher, executed, fails at `__validate__`, and is reverted — consuming sequencer CPU and block space.
- **Griefing at scale**: An attacker can monitor the mempool for all incoming `deploy_account` transactions and immediately front-run each one with an invalid nonce=1 invoke, systematically degrading the UX for every new account deployment.

This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- Triggering the condition requires only that a `deploy_account` tx for the target address is visible in the mempool — a public, observable event.
- No privileged access, no special contract, no on-chain funds required.
- The attacker only needs to craft a well-formed `InvokeV3` transaction with nonce=1 and any bytes in the signature field.
- The attack is repeatable for every new account deployment observed in the mempool.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a targeted check that verifies a **deploy-account** transaction specifically for the sender address is present in the mempool (not just any transaction). For example:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use:
mempool_client.deploy_account_tx_in_pool_or_recent_block(tx.sender_address())
```

This ensures the skip-validate shortcut is only granted when the account's own deploy-account tx is pending, not when an unrelated transaction happens to reference the same address.

Alternatively, require that the invoke tx's `tx_hash` is submitted together with the deploy-account tx hash as a linked pair, so the gateway can verify both originate from the same submitter.

---

### Proof of Concept

```
// Setup: victim V submits a valid deploy_account for address A.
// A is now in the mempool pool; account_tx_in_pool_or_recent_block(A) == true.
// On-chain nonce for A == 0.

// Attacker submits:
RpcInvokeTransactionV3 {
    sender_address: A,          // victim's undeployed address
    nonce: 1,                   // triggers skip_stateful_validations
    signature: [0xdead, 0xbeef], // arbitrary invalid signature
    resource_bounds: <valid>,
    calldata: <anything>,
    ...
}

// Gateway flow:
// 1. account_nonce = get_nonce_from_state(A) = 0
// 2. validate_nonce: 0 <= 1 <= 0 + max_gap  -> OK
// 3. validate_by_mempool: nonce not too old  -> OK
// 4. skip_stateful_validations:
//      tx.nonce() == 1  AND  account_nonce == 0
//      AND  account_tx_in_pool_or_recent_block(A) == true  -> returns true
// 5. run_validate_entry_point(tx, skip_validate=true):
//      ExecutionFlags { validate: false, ... }
//      __validate__ is NOT called
// 6. tx admitted to mempool  <-- invalid signature never checked

// Result: attacker's invalid invoke occupies nonce=1 slot for A.
// Victim must fee-escalate to replace it, or the invalid tx
// wastes sequencer resources when executed during block production.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
