### Title
Unauthenticated `LedgerService.Set()` RPC Allows Any Network Caller to Corrupt Execution State - (File: `cmd/ledger/main.go`, `ledger/remote/service.go`)

---

### Summary

The standalone ledger service (`cmd/ledger`) exposes a gRPC `Set()` RPC that writes key-value pairs directly into the execution-state Merkle trie. The gRPC server is instantiated with no TLS credentials, no authentication interceptors, and no authorization middleware. The `Set()` handler in `ledger/remote/service.go` performs no caller-identity check before delegating to `ledger.Set()`. When the service is bound to a TCP address — as explicitly documented and supported — any network-reachable caller can invoke `Set()` to write arbitrary data into the execution state, permanently corrupting the ledger.

---

### Finding Description

**Root cause — unauthenticated gRPC server**

In `cmd/ledger/main.go` the gRPC server is created with only message-size options:

```go
grpcServer := grpc.NewServer(
    grpc.MaxRecvMsgSize(int(*maxRequestSize)),
    grpc.MaxSendMsgSize(int(*maxResponseSize)),
)
```

No `grpc.Creds(...)`, no unary/stream interceptor for authentication or authorization is added. The `LedgerService` is registered directly on this unauthenticated server. [1](#0-0) 

**Root cause — no authorization in `Set()` handler**

`ledger/remote/service.go` defines the `Service` struct with only a `ledger.Ledger` field and a logger — no caller-identity or token field — and the `Set()` method (not shown in the snippet but registered via `ledgerpb.RegisterLedgerServiceServer`) delegates directly to `ledger.Set()` with no authorization gate. [2](#0-1) 

The proto definition confirms `Set()` is a plain unary RPC with no metadata-based auth contract: [3](#0-2) 

**Attacker-controlled entry path**

The README explicitly documents and encourages binding to `0.0.0.0:9000`:

```
./flow-ledger-service \
  -triedir /path/to/trie \
  -ledger-service-tcp 0.0.0.0:9000
``` [4](#0-3) 

The flag definition confirms TCP binding is a first-class supported mode: [5](#0-4) 

TCP listener creation with no TLS: [6](#0-5) 

Unlike the main node admin server (`admin/command_runner.go`), which at least offers optional mTLS via `WithTLS(config)`, the ledger service has **no authentication option at all** — there is no flag, no interceptor hook, and no TLS path in the entire `cmd/ledger` binary. [7](#0-6) 

---

### Impact Explanation

`Set()` writes key-value pairs directly into the execution-state Merkle trie — the same trie that backs all Flow account balances, contract storage, and protocol state. An attacker who can reach the TCP port can:

1. Overwrite arbitrary register values (account balances, contract bytecode, capability storage).
2. Produce a forged state root that the execution node will sign and propagate as a valid execution result.
3. Cause permanent, unrecoverable corruption of the ledger, since WAL entries are written immediately and checkpoints persist the corrupted trie to disk.

This maps directly to the report's impact class: **unauthorized state mutation** of a privileged data store, reachable by any unprivileged caller.

---

### Likelihood Explanation

The service is documented to bind on `0.0.0.0:9000`. Any attacker within the same cloud VPC, data-center network, or (if misconfigured at the firewall level) the public internet can send a well-formed gRPC `Set` request with no credentials. No keys, no tokens, no prior authentication step is required. The gRPC wire format is public and client libraries exist in every major language.

---

### Recommendation

1. Add mandatory mTLS to the ledger gRPC server, mirroring the pattern already used by the main node admin server (`admin/command_runner.go` `WithTLS`). Reject connections that do not present a certificate signed by the execution node's CA.
2. Add a unary server interceptor that verifies the peer certificate's identity before dispatching any RPC, especially `Set()`.
3. Document that the TCP listener must never be exposed outside the private execution-node network segment, and enforce this with a startup-time warning or hard error when no TLS config is provided.

---

### Proof of Concept

```bash
# No credentials required — any host that can reach port 9000 can corrupt state
grpcurl -plaintext \
  -d '{"state":{"hash":"<valid_32_byte_state_hash_hex>"},
       "keys":[{"parts":[{"type":0,"value":"<account_register_key>"}]}],
       "values":[{"data":"<arbitrary_bytes>","is_nil":false}]}' \
  <ledger-service-host>:9000 \
  ledger.LedgerService/Set
# Returns a new state hash; the trie now contains the attacker-supplied value.
# The execution node will use this corrupted trie for all subsequent block executions.
```
