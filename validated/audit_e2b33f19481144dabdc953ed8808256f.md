### Title
FIFO Mempool Silently Orphans Higher-Nonce Staged Transactions When Lowest-Nonce Sibling Is Rejected — (`crates/apollo_mempool/src/fifo_transaction_queue.rs`)

### Summary

In `FifoTransactionQueue::collect_txs_to_rewind`, the no-committed-txs branch decides whether to rewind an entire address solely by checking whether the **lowest-nonce** staged transaction was rejected. When it was, the address is excluded from `addresses_to_rewind`. The subsequent `staged_txs.clear()` then silently drops all higher-nonce staged transactions for that address from the queue's tracking structures, while those transactions remain in `tx_pool` with no path back to the queue.

### Finding Description

**Execution path:**

`commit_block` → `rewind_txs` → `FifoTransactionQueue::rewind_txs` → `collect_txs_to_rewind`

**State entering `commit_block`:**
- Address A has staged nonces [5, 6, 7] (all returned by prior `get_txs` calls, moved from `queue` to `staged_txs`, and present in `tx_pool`)
- `address_to_nonce` does **not** contain A (no committed txs for A in this block)
- `rejected_tx_hashes` = `{hash_of_nonce_5}`

**Step-by-step:**

1. The loop at `commit_block` lines 644–677 skips A entirely (not in `address_to_nonce`). No pool cleanup for A.

2. `state.commit(address_to_nonce)` — A is in `state.staged` but not in `address_to_nonce`, so A enters `addresses_to_rewind` (the `MempoolState`-level list). [1](#0-0) 

3. `rewind_txs` is called in FIFO mode, delegating entirely to `FifoTransactionQueue::rewind_txs`. [2](#0-1) 

4. Inside `collect_txs_to_rewind`, for address A with no committed nonce, the else-branch fires:
   - `first_tx` = nonce 5 (minimum nonce)
   - `!rejected_tx_hashes.contains(&hash_of_nonce_5)` = `false`
   - A is **not** added to `addresses_to_rewind` (the queue-level set) [3](#0-2) 

5. Step 3 of `collect_txs_to_rewind` filters staged txs to only those whose address is in `addresses_to_rewind`. Since A is absent, nonces 6 and 7 are **not** included in `txs_to_rewind`. [4](#0-3) 

6. Back in `rewind_txs`: `rewound_hashes` is empty. `staged_txs.clear()` executes unconditionally, dropping nonces 5, 6, and 7 from `staged_txs`. Nonces 6 and 7 are **never pushed back to `self.queue`**. [5](#0-4) 

7. `remove_rejected_txs` processes `hash_of_nonce_5`: removes it from `tx_pool`, then calls `tx_queue.remove_by_address(A)`. At this point `staged_txs` is already empty and nonces 6/7 are not in `self.queue`, so `remove_by_address` is a no-op for them. [6](#0-5) 

**Final state:**
| Tx | `tx_pool` | `queue` | `staged_txs` |
|----|-----------|---------|--------------|
| nonce 5 | removed | — | cleared |
| nonce 6 | **present** | absent | cleared |
| nonce 7 | **present** | absent | cleared |

Nonces 6 and 7 are permanently orphaned: they exist in `tx_pool` but have no entry in `queue` or `staged_txs`, and no code path re-queues them.

### Impact Explanation

Valid transactions (nonces 6 and 7) are permanently excluded from sequencing. They consume pool memory indefinitely and can never be processed. The mempool invariant — every non-committed, non-rejected staged transaction must be re-queued after `commit_block` — is violated. This maps to **High: mempool rejects valid transactions before sequencing**.

The question's "Critical economic misorder" framing is slightly overstated: the direct effect is liveness loss (stuck transactions), not a wrong fee charge or balance mutation. However, the pool memory leak is unbounded if this pattern repeats across many accounts.

### Likelihood Explanation

In FIFO/Echonet mode, a transaction fails the `__validate__` entry point and is included in `rejected_tx_hashes` whenever the account's validation logic rejects it. Any user who submits a transaction at nonce N that fails validation, while also having submitted transactions at nonces N+1, N+2, … (all staged in the same proposal), triggers this path. No privileged access is required.

### Recommendation

In the no-committed-txs branch of `collect_txs_to_rewind`, rejecting the lowest-nonce tx should not silently abandon higher-nonce siblings. Two correct options:

1. **Remove higher-nonce siblings from `tx_pool` as well** — since they cannot execute without nonce 5, they are also invalid and should be purged.
2. **Re-queue higher-nonce siblings** — if the protocol allows them to be retried after nonce 5 is re-submitted, push nonces 6 and 7 back to the front of the queue.

The current code does neither, leaving them stranded.
