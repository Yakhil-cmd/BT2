### Title
WalletConnect `create_offer_for_ids` Receiving-Side NFT Displayed as CAT in Approval Dialog — (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
In the WalletConnect approval flow for `chia_wallet.create_offer_for_ids`, any bytes32 asset ID on the **receiving** (positive-value) side of the offer that has no corresponding `driver_dict` entry is unconditionally classified as a CAT token. A malicious dApp can exploit this to make an NFT launcher ID appear as a tiny CAT amount in the user-facing confirmation dialog, causing the user to approve an offer whose receiving asset type is entirely misrepresented.

### Finding Description

`parseCommandDisplay` in `packages/gui/src/electron/commands/parseCommandDisplay.ts` is the function that builds the human-readable approval summary shown to the user before a WalletConnect command is executed.

For `chia_wallet.create_offer_for_ids`, it calls `createOfferAssetKinds`, which classifies each asset in the offer: [1](#0-0) 

The spending side (negative values, line 218–223) calls `assetKindForDriverDictAssetId` and returns `undefined` when no driver dict entry exists — which then causes `parseWalletDeltaItem` to throw `'Asset type is not valid'`, failing closed.

The **receiving side** (positive values, line 226–233) applies `?? 'cat'` as a fallback:

```typescript
assetKinds.receiving[assetId] = assetKindForDriverDictAssetId(assetId, driverDict) ?? 'cat';
```

This is explicitly documented as "Backend legacy behavior treats positive requested bytes32 assets with no explicit driver as CATs." The test suite confirms this behavior is intentional: [2](#0-1) 

When `assetKind === 'cat'`, `parseWalletDeltaItem` calls `catAssetIdToName` and renders the asset as a fungible token with a mojo-to-CAT amount: [3](#0-2) 

An NFT has an amount of `1` mojo, which `mojoToCATLocaleString` renders as `0.000000001`. The confirmation dialog therefore shows the NFT as `0.000000001 [unknown CAT]` instead of `NFT [nftId]`. [4](#0-3) 

### Impact Explanation

A WalletConnect-connected dApp with the `chia_createOfferForIds` permission sends:

```json
{
  "offer": {
    "1": "-1000000000000",
    "<nft_launcher_id_hex>": "1"
  }
}
```

with no `driver_dict` for the NFT launcher ID. The approval dialog shows:

- **Spending:** `1 XCH` ✓ (correct)
- **Receiving:** `0.000000001 [unknown CAT]` ✗ (should be `NFT <nftId>`)

The user, seeing a negligible CAT amount, may approve what appears to be a test or dust transaction. The actual offer created by the backend requests the NFT. A colluding party (or the dApp operator themselves) immediately accepts the offer, transferring the user's 1 XCH in exchange for a worthless NFT. The user has no recourse once the offer is accepted on-chain.

This is a **High** impact finding: spoofing of WalletConnect offer state that causes a user to approve the wrong asset type and amount, resulting in direct XCH loss.

### Likelihood Explanation

- Any dApp that has been granted `chia_createOfferForIds` permission via WalletConnect can trigger this.
- No special privileges, leaked keys, or cryptographic breaks are required.
- The dApp simply omits the `driver_dict` for the NFT asset — a trivially crafted payload.
- The user has no visual cue that the displayed asset type is wrong; the dialog renders a plausible-looking (if tiny) CAT amount.

### Recommendation

The receiving side of `createOfferAssetKinds` should not silently fall back to `'cat'` for unresolved bytes32 assets. Options:

1. **Fail closed** (symmetric with the spending side): throw `'Asset type is not valid'` when a bytes32 receiving asset has no driver dict entry, forcing dApps to always supply a `driver_dict`.
2. **Display as unknown**: render unresolved bytes32 receiving assets as `kind: 'unknown'` with a prominent warning, rather than silently treating them as CATs.

The spending side already fails closed for the same scenario: [5](#0-4) 

The receiving side should apply the same discipline.

### Proof of Concept

A malicious WalletConnect dApp with `chia_createOfferForIds` permission sends:

```json
{
  "command": "chia_wallet.create_offer_for_ids",
  "params": {
    "offer": {
      "1": "-1000000000000",
      "deadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef": "1"
    }
  }
}
```

`parseCommandDisplay` is called with these params. `createOfferAssetKinds` classifies `deadbeef...` (a bytes32 NFT launcher ID) as `'cat'` via the `?? 'cat'` fallback at line 232. The confirmation dialog renders:

- Spending: `1 XCH`
- Receiving: `0.000000001` (no symbol, since `catAssetIdToName` returns `{ name: undefined }` for an unknown asset)

The user approves. The backend creates a valid offer locking 1 XCH in exchange for the NFT. The attacker accepts the offer immediately, taking the user's XCH. [1](#0-0) [6](#0-5)
