### Title
STRK Reward Tokens Permanently Frozen in Pool Contract When Pool Balance Is Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When a delegation pool's total balance is below `min_delegation_for_rewards` (1 STRK = 10^18 wei), the staking contract still physically transfers STRK reward tokens to the pool contract, but the pool's internal accounting (`cumulative_rewards_trace`) is not updated. The transferred tokens are permanently unclaimable by pool members, with no recovery mechanism.

---

### Finding Description

The reward distribution path in `_update_rewards` (`src/staking/staking.cairo`) proceeds as follows:

1. Pool rewards are calculated via `calculate_staker_pools_rewards` — this calculation does **not** check `min_delegation_for_rewards`; it returns non-zero `pool_rewards` as long as the pool has any balance.
2. `claim_from_reward_supplier` transfers the total pool rewards from `RewardSupplier` to the `Staking` contract.
3. `update_pool_rewards` iterates over each pool and:
   - Calls `send_rewards_to_delegation_pool`, which **unconditionally** transfers STRK tokens to the pool contract.
   - Calls `pool.update_rewards_from_staking_contract(rewards, pool_balance)`. [1](#0-0) 

Inside `update_rewards_from_staking_contract` in the pool contract, `compute_rewards_per_unit` is called: [2](#0-1) 

`compute_rewards_per_unit` contains the following guard:

```cairo
// Return zero if the total stake is too small, to avoid overflow below.
if total_stake < self.min_delegation_for_rewards.read() {
    return Zero::zero();
}
``` [3](#0-2) 

When `pool_balance < min_delegation_for_rewards`, the function returns `Zero::zero()`. The `cumulative_rewards_trace` is therefore updated with **zero** additional rewards for that epoch. The STRK tokens already transferred to the pool contract are now unaccounted for in the trace.

The code comment itself acknowledges this:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [4](#0-3) 

The `min_delegation_for_rewards` for STRK pools is set to `10^18` (1 STRK): [5](#0-4) 

The only token-exit paths in the pool contract are `claim_rewards` (reads from `cumulative_rewards_trace`, which shows zero for the affected epoch), `exit_delegation_pool_action` (returns staked principal only), and `switch_delegation_pool` (moves principal). None can recover the stuck reward tokens. [6](#0-5) 

---

### Impact Explanation

STRK reward tokens are physically transferred to the pool contract but are permanently unclaimable. There is no admin recovery function in the pool contract. This constitutes **permanent freezing of unclaimed yield**, which is in the allowed impact scope.

---

### Likelihood Explanation

A pool with total delegated balance between 0 and 1 STRK (exclusive) triggers this condition. This is realistic in the following scenarios:
- A pool is newly opened and has only one or a few small delegators.
- Most delegators have exited via `exit_delegation_pool_intent`/`exit_delegation_pool_action`, leaving a residual balance below 1 STRK.
- A delegator intentionally keeps a tiny delegation to grief the pool.

The condition is reachable by any unprivileged pool member or staker without any privileged access.

---

### Recommendation

In `update_pool_rewards` (`src/staking/staking.cairo`), before calling `send_rewards_to_delegation_pool`, check whether `pool_balance < min_delegation_for_rewards`. If so, skip the token transfer (or redirect the tokens to a recoverable location such as the staker's reward address or a protocol treasury). This mirrors the fix suggested in M-03: ensure accounting and token custody are always consistent. [7](#0-6) 

---

### Proof of Concept

1. Staker calls `set_open_for_delegation` to create a STRK pool.
2. A pool member delegates `0.5 STRK` (5×10^17 wei < `min_delegation_for_rewards` = 10^18).
3. Epoch advances; `update_rewards` (or `update_rewards_from_attestation_contract`) is triggered.
4. `calculate_staker_pools_rewards` computes non-zero `pool_rewards` for this pool.
5. `claim_from_reward_supplier` pulls `pool_rewards` STRK into the staking contract.
6. `send_rewards_to_delegation_pool` transfers `pool_rewards` STRK to the pool contract. ✓ tokens arrive.
7. `pool.update_rewards_from_staking_contract(rewards: pool_rewards, pool_balance: 0.5 STRK)` is called.
8. `compute_rewards_per_unit` returns `0` because `0.5 STRK < min_delegation_for_rewards`.
9. `cumulative_rewards_trace` is updated with `last + 0 = last` — no new rewards recorded.
10. Pool member calls `claim_rewards` → receives `0` STRK.
11. The `pool_rewards` STRK tokens sit in the pool contract forever with no recovery path. [3](#0-2) [8](#0-7)

        fn update_pool_rewards(
            ref self: ContractState,
            staker_address: ContractAddress,
            pools_rewards_data: Array<(ContractAddress, ContractAddress, NormalizedAmount, Amount)>,
        ) -> Array<(ContractAddress, Amount)> {
            let mut pool_rewards_list = array![];
            let strk_token_dispatcher = strk_token_dispatcher();
            for (pool_contract, token_address, pool_balance, pool_rewards) in pools_rewards_data {
                let pool_dispatcher = IPoolDispatcher { contract_address: pool_contract };
                // Rewards are always in STRK.
                self

                        :staker_address,

                        token_dispatcher: strk_token_dispatcher,
                    );
                let decimals = self.get_token_decimals(:token_address);
                pool_dispatcher
                    .update_rewards_from_staking_contract(
                        rewards: pool_rewards,
                        pool_balance: pool_balance.to_native_amount(:decimals),
                    );
                pool_rewards_list.append((pool_contract, pool_rewards));
            }
            pool_rewards_list
        }

            self.assert_caller_is_staking_contract();

            // `rewards_info` is initialized in the constructor or in the upgrade proccess,

            let (_, last) = self.cumulative_rewards_trace.last().unwrap();
            self
                .cumulative_rewards_trace
                .insert(
                    key: self.get_current_epoch(),
                    value: last
                        + self
                            .compute_rewards_per_unit(
                                staking_rewards: rewards, total_stake: pool_balance,
                            ),
                );
        }

        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
