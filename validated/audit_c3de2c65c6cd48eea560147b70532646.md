Based on my investigation of the Cronos codebase, I can identify a valid analog to the fee-on-transfer (FOT) accounting vulnerability.

---

### Title
EVM Hook Handlers Use Event-Emitted `amount` Without Accounting for Fee-on-Transfer Deductions — (`x/cronos/keeper/evmhandlers/send_to_account.go`, `send_to_ibc.go`, `send_to_ibc_v2.go`)

### Summary

When an external CRC20/CRC21 contract registered via `MsgUpdateTokenMapping` implements a fee-on-transfer mechanism, the Cronos EVM hook handlers process the `amount` field from the emitted event log verbatim. Because the `module_address` actually receives `amount - fee` tokens (not `amount`), the hook either reverts due to insufficient balance or — if the module account holds a residual balance from prior operations — silently bridges/transfers more tokens than were deposited, draining the module account.

### Finding Description

The `ModuleCRC21.sol` contract's bridge functions transfer tokens to `module_address` and emit an event carrying the caller-specified `amount`:

```solidity
// contracts/src/ModuleCRC21.sol
function send_to_ibc(..., uint amount, ...) external {
    if (isSource) {
        transferFrom(msg.sender, module_address, amount);   // FOT: module receives amount - fee
    } else {
        unsafe_burn(msg.sender, amount);
    }
    emit __CronosSendToIbc(msg.sender, channel_id, recipient, amount, extraData); // emits full amount
}
``` [1](#0-0) 

The same pattern applies to `send_to_evm_chain` and `send_to_account`. [2](#0-1) 

The EVM hook infrastructure (`evm_hooks.go`) dispatches these log events to the handlers in `evmhandlers/`. Each handler unpacks the `amount` field from the event and passes it directly to keeper functions such as `ConvertCoinFromCRC21ToNative` / `IbcTransferCoins` / bank send — without re-reading the module address's actual post-transfer balance. [3](#0-2) 

The same trust-the-event-amount pattern is visible in the IBC conversion middleware, which reads `data.Token.Amount` directly from the packet without any balance reconciliation:

```go
transferAmount, ok := sdkmath.NewIntFromString(data.Token.Amount)
...
token := sdk.NewCoin(denom, transferAmount)
im.cronoskeeper.ConvertVouchersToEvmCoins(cacheCtx, data.Receiver, sdk.NewCoins(token))
``` [4](#0-3) 

### Impact Explanation

**Scenario A — Revert / DoS (High):** If the module account holds no residual balance of the FOT token, the keeper call to burn or transfer `amount` tokens fails because only `amount - fee` was deposited. Every bridge/conversion transaction for that token reverts, permanently blocking honest users from using the bridge/conversion flow for that registered token.

**Scenario B — Unauthorized Drain (Critical):** If the module account has accumulated a residual balance (e.g., from prior partial operations or dust), the hook succeeds and transfers `amount` to the recipient while only `amount - fee` was deposited. The difference (`fee`) is silently taken from the module's accumulated balance — an unauthorized accounting change for CRC20/IBC-voucher assets controlled by the module account.

### Likelihood Explanation

The trigger requires an external CRC20 contract with a transfer fee to be registered via `MsgUpdateTokenMapping` (an admin action). Once registered, any unprivileged user calling `send_to_ibc`, `send_to_ibc_v2`, or `send_to_account` on that contract activates the bug. Fee-on-transfer tokens are a well-known token pattern and are explicitly mentioned as a supported use-case in the contest README that prompted the original report.

### Recommendation

In each EVM handler (`send_to_account.go`, `send_to_ibc.go`, `send_to_ibc_v2.go`), replace the event-emitted `amount` with the actual balance delta of `module_address` measured before and after the transfer:

```go
balanceBefore := erc20.BalanceOf(moduleAddress)
// ... event already executed the transferFrom ...
balanceAfter  := erc20.BalanceOf(moduleAddress)
actualAmount  := balanceAfter - balanceBefore   // use this, not event amount
```

This mirrors the Uniswap V2 `swapExactTokensForTokensSupportingFeeOnTransferTokens` pattern cited in the original report.

### Proof of Concept

1. Admin registers an external CRC20 contract that charges a 5 % transfer fee via `MsgUpdateTokenMapping`.
2. Attacker (any user) calls `send_to_account(recipient, 1000)` on the registered FOT CRC20.
3. The contract executes `transferFrom(attacker, module_address, 1000)` → module receives **950**.
4. The contract emits `__CronosSendToAccount(attacker, recipient, 1000)`.
5. The EVM hook reads `amount = 1000` from the event and calls the keeper to send **1000** native tokens to `recipient`.
6. If the module account holds ≥ 1000 native tokens (from prior operations), the send succeeds and `recipient` receives 1000 while only 950 was deposited — **50 tokens drained from the module account** (Critical: unauthorized balance change).
7. If the module account holds < 1000 tokens, the transaction reverts — **DoS for all users of that token** (High: permanent inability to process valid bridge transactions). [5](#0-4) [6](#0-5)
