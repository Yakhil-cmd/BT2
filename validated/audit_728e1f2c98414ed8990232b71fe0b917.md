### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Non-Deploy-Account Mempool Presence — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the `__validate__` entry-point call (i.e., account signature verification) for an invoke transaction with nonce=1 when a deploy_account for the same sender is pending in the mempool. However, the actual guard used — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the sender address in the mempool, not exclusively a deploy_account. An attacker can exploit this by first placing a valid invoke with nonce=0 in the mempool for an account whose on-chain nonce is 0, then submitting a second invoke with nonce=1 carrying an **invalid signature**. The gateway skips `__validate__` and admits the invalid transaction to the mempool.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` and returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)`
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The code comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is incorrect. `account_tx_in_pool_or_recent_block` returns `true` whenever the sender address appears in either the mempool pool or the committed-block state: [2](#0-1) 

It does **not** verify that the pooled transaction is a deploy_account. A valid invoke with nonce=0 from the same sender is sufficient to satisfy condition 3.

When `skip_validate=true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `AccountTransaction::validate_tx`, when `execution_flags.validate` is `false`, the function returns `Ok(None)` immediately without executing the `__validate__` entry point: [4](#0-3) 

**Attack scenario:**

1. Target account `A` has on-chain nonce=0 (e.g., deployed via a factory/deploy syscall, not via deploy_account, so its nonce was never incremented by the OS).
2. Attacker submits a valid invoke with nonce=0 for `A`. This passes all checks including `__validate__` (account exists, nonce matches). It enters the mempool.
3. Attacker submits a second invoke with nonce=1 for `A` carrying an **arbitrary/invalid signature**.
   - Stateless validation: passes (only checks signature *length*, not cryptographic validity). [5](#0-4) 
   - `validate_nonce`: nonce=1 is within `[0, 0+200]`, passes. [6](#0-5) 
   - `validate_by_mempool`: checks nonce/duplicate only, no signature check.
   - `skip_stateful_validations`: nonce=1, account_nonce=0, nonce=0 invoke is in pool → returns `true`.
   - `run_validate_entry_point(skip_validate=true)`: `__validate__` is **not called**.
4. The invalid-signature invoke is admitted to the mempool.

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

The invalid transaction occupies a nonce=1 slot in the mempool for account `A`. Because the mempool enforces one transaction per (address, nonce) slot, any subsequent valid nonce=1 invoke from the legitimate account owner is either rejected as a duplicate or must pay a higher fee to replace it (fee escalation). This effectively blocks or delays the legitimate owner's next transaction.

The invalid transaction will fail at `__validate__` when the batcher attempts execution (blockifier uses `validate: true, strict_nonce_check: true` for sequencing), so no state corruption occurs. The impact is admission-level: an unauthorized transaction occupies a mempool slot.

---

### Likelihood Explanation

**Medium.** The precondition — an account with on-chain nonce=0 that was deployed via a mechanism other than deploy_account (e.g., a factory contract using the `deploy` syscall) — is realistic in production. The attacker needs no special privileges: they only need to know the target account address and submit two transactions via the public RPC endpoint.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction is pending for the sender address. Add a dedicated mempool query such as `deploy_account_tx_in_pool(sender_address)` that inspects the transaction type, or filter the pool result to only match `InternalRpcTransactionWithoutTxHash::DeployAccount` entries for the given address.

Alternatively, enforce the invariant in the comment: only skip `__validate__` when the account's on-chain nonce is 0 **and** a deploy_account transaction (not just any transaction) is present in the mempool for that address.

---

  "nonce": "0x0",
  "signature": [<valid_sig>],
  ...

// → Admitted to mempool. account_tx_in_pool_or_recent_block(A) now returns true.

POST /add_transaction
{
  "type": "INVOKE",
  "sender_address": A,
  "nonce": "0x1",
  "signature": [0xdeadbeef],   // arbitrary invalid signature
  ...
}
// Gateway flow:
//   stateless_tx_validator.validate() → Ok (length check only)
//   convert_rpc_tx_to_internal() → Ok (computes tx_hash, no sig check)
//   extract_state_nonce_and_run_validations():
//     get_nonce_from_state(A) → Nonce(0)
//     run_pre_validation_checks():
//       validate_nonce: 0 <= 1 <= 200 → Ok
//       validate_by_mempool: no dup → Ok
//       skip_stateful_validations: nonce==1 && account_nonce==0 && in_pool==true → true
//     run_validate_entry_point(skip_validate=true):
//       execution_flags.validate = false
//       validate_tx() → Ok(None)  // __validate__ NOT called
// → Invalid-signature invoke admitted to mempool at nonce=1 slot for A.

// Consequence: legitimate owner's valid nonce=1 invoke is now blocked

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
