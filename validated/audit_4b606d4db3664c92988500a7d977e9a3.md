### Title
Stale KV-Stored Permissions Persist After `CronosAdmin` Rotation via `UpdateParams` — (File: x/cronos/keeper/msg_server.go)

---

### Summary

`MsgUpdateParams` replaces the `CronosAdmin` parameter but never clears the KV-stored per-address permission entries previously written by the old admin. Any address that was granted `CanChangeTokenMapping` (or `CanTurnBridge`) by the old admin retains those permissions indefinitely, because `HasPermission` checks both the live `CronosAdmin` param **and** the KV store. There is no enumeration primitive for the permission store, so the new admin cannot discover and revoke stale grants. A holder of a stale `CanChangeTokenMapping` grant can delete a live denom→contract binding and replace it with a malicious contract, corrupting all future bridge/conversion flows for that denom.

---

### Finding Description

**Setter that does not migrate dependent state**

`UpdateParams` calls `SetParams`, which writes only the `Params` struct to the KV store: [1](#0-0) 

It performs no scan or deletion of the `prefixAdminToPermissions` key space. The permission entries written by the old admin remain intact.

`HasPermission` evaluates two independent conditions: [2](#0-1) 

Condition 1 (line 39) checks the *current* `CronosAdmin` param. Condition 2 (lines 42–44) checks the KV-stored permission bitmask for the caller. After an admin rotation, condition 1 no longer passes for the old admin's grantees, but condition 2 still passes because the KV entries were never cleared.

`UpdatePermissions` — the only way to revoke a grant — is gated to the *current* admin: [3](#0-2) 

The new admin can call `UpdatePermissions` to revoke a specific address, but only if it knows which addresses hold grants. There is no iterator or enumeration function over `KeyPrefixAdminToPermissions`: [4](#0-3) 

A grep across the entire keeper confirms no `GetAllPermissions`, `IteratePermissions`, or prefix-scan helper exists. The new admin has no on-chain mechanism to discover stale grantees.

**Exploitation path via `UpdateTokenMapping`**

`UpdateTokenMapping` is the privileged action gated by `CanChangeTokenMapping`: [5](#0-4) 

Inside `RegisterOrUpdateTokenMapping`, passing an empty `Contract` field deletes the existing mapping; passing a non-empty address sets a new one: [6](#0-5) 

`SetExternalContractForDenom` enforces `ensureDenomNotMapped`, so a two-step sequence is required: delete first, then re-register with a malicious contract. Both steps pass the `HasPermission` check using the stale KV grant.

After the remap, every call to `ConvertVouchersToEvmCoins` for that denom resolves to the malicious contract and calls `mint_by_cronos_module` on it: [7](#0-6) 

The malicious contract can mint to an arbitrary address instead of the legitimate receiver, or refuse to mint at all, while the user's native IBC vouchers have already been burned/escrowed.

---

### Impact Explanation

**High — Bypass of Cronos admin permission checks; corruption of denom/contract binding with direct security impact.**

A stale grantee can unilaterally delete and replace any denom→contract mapping. All subsequent `ConvertVouchersToEvmCoins` and IBC-middleware conversion flows for the affected denom will call into the attacker-controlled contract. Users' IBC vouchers are consumed while CRC20 tokens are minted to the attacker or not minted at all, constituting an unauthorized accounting change for IBC vouchers and CRC20 assets.

---

### Likelihood Explanation

Requires two preconditions: (1) the old admin granted `CanChangeTokenMapping` to at least one address before the rotation, and (2) that address is adversarial or compromised after the rotation. Admin rotations via governance are a documented, expected operation. The absence of an enumeration primitive makes silent retention of stale grants the default outcome of every rotation.

---

### Recommendation

`SetParams` should detect a `CronosAdmin` change and either (a) clear all `prefixAdminToPermissions` entries atomically, or (b) emit an on-chain event listing all current grantees so the new admin can revoke them in the same governance proposal. Additionally, add a `GetAllPermissions` / `IteratePermissions` keeper method so the new admin can enumerate and revoke stale grants programmatically.

---

```go
	if err := params.Validate(); err != nil {
		return err
	}

	store.Set(types.ParamsKey, bz)

}
```

```go
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```

```go
func (k msgServer) UpdateTokenMapping(goCtx context.Context, msg *types.MsgUpdateTokenMapping) (*types.MsgUpdateTokenMappingResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)

	// check permission
	if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
		return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
	}

	// msg is already validated
	if err := k.RegisterOrUpdateTokenMapping(ctx, msg); err != nil {
		return nil, err
	}
	return &types.MsgUpdateTokenMappingResponse{}, nil
}
```

```go
func (k msgServer) UpdatePermissions(goCtx context.Context, msg *types.MsgUpdatePermissions) (*types.MsgUpdatePermissionsResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)
	admin := k.Keeper.GetParams(ctx).CronosAdmin
	// if admin is empty, no sender could be equal to it
	if admin != msg.From {
		return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
	}
	acc, err := sdk.AccAddressFromBech32(msg.Address)
	if err != nil {
		return nil, err
	}
	k.SetPermissions(ctx, acc, msg.Permissions)

	return &types.MsgUpdatePermissionsResponse{}, nil
}
```
