### Title
Gateway stateless validator unconditionally skips `max_l2_gas_amount` upper-bound check for Declare transactions, admitting unbounded resource-bound declarations - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` contains an explicit early-return branch that makes the `max_l2_gas_amount` upper-bound check a no-op for every `RpcTransaction::Declare`. Any user can therefore submit a Declare transaction whose `l2_gas.max_amount` is set to `u64::MAX` (or any value exceeding the configured limit of `1,210,000,000`), and the transaction will pass the gateway's stateless admission gate. A companion test, `valid_l2_gas_amount_on_declare`, explicitly asserts this behaviour is accepted.

---

### Finding Description

In `validate_resource_bounds`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production configuration sets `max_l2_gas_amount = 1_210_000_000` for Invoke and DeployAccount transactions. [2](#0-1) 

For Declare transactions the branch is empty, so `l2_gas.max_amount = u64::MAX` passes without error. The test `valid_l2_gas_amount_on_declare` confirms this is the current behaviour: [3](#0-2) 

The Starknet OS reads `l2_gas.max_amount` directly as the initial gas budget for the transaction:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [4](#0-3) 

The downstream blockifier pre-validation stage (`perform_pre_validation_stage`) only runs `check_fee_bounds` and `verify_can_pay_committed_bounds` when `execution_flags.charge_fee == true`: [5](#0-4) 

When `charge_fee = false` — the path taken during RPC simulation (`starknet_simulateTransactions`, `starknet_estimateFee`) — those guards are entirely skipped, and the transaction executes with `u64::MAX` as its OS-level gas limit.

---

### Impact Explanation

**Gateway admission (High):** A Declare transaction with `l2_gas.max_amount = u64::MAX` passes `StatelessTransactionValidator::validate` and is forwarded to the mempool. The gateway's own admission invariant — that no transaction may declare more than `max_l2_gas_amount` L2 gas — is broken for the Declare type. This matches the impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

**RPC simulation / fee estimation (High):** When the RPC layer simulates a Declare transaction with `charge_fee = false`, all fee-bound checks are bypassed. The OS receives `u64::MAX` as the initial gas budget. A contract crafted to consume gas up to the block bouncer ceiling will run to that ceiling under simulation, causing the RPC node to perform maximum-cost execution for every such request — a denial-of-service vector against the simulation endpoint. This matches: *"RPC execution, fee estimation, tracing, simulation … returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

The trigger requires no special privilege. Any user can craft a V3 Declare transaction with `l2_gas.max_amount = u64::MAX` and a non-zero `l1_gas` bound (to pass the `ZeroResourceBounds` check). The stateless validator accepts it unconditionally. The TODO comment in the source confirms the gap is known but unresolved.

---

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx { }` guard and apply the same `max_l2_gas_amount` upper-bound check uniformly to all transaction types, including Declare:

```rust
// Remove the Declare exemption:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas ceiling (e.g., for large Sierra programs), introduce a separate `max_l2_gas_amount_declare` configuration parameter with an explicit, bounded value rather than leaving the field unchecked.

---

### Proof of Concept

The existing test `valid_l2_gas_amount_on_declare` already demonstrates the issue: a Declare transaction with `l2_gas.max_amount = 200` passes validation even when the configured limit is `100`. [3](#0-2) 

To reach the RPC simulation impact, submit the following to `starknet_simulateTransactions` with `simulation_flags: ["SKIP_FEE_CHARGE"]`:

```json
{
  "type": "DECLARE",
  "version": "0x3",
  "resource_bounds": {
    "l1_gas": { "max_amount": "0x1", "max_price_per_unit": "0x1" },
    "l2_gas": { "max_amount": "0xffffffffffffffff", "max_price_per_unit": "0x1" },
    "l1_data_gas": { "max_amount": "0x0", "max_price_per_unit": "0x0" }
  },
  ...
}
```

The transaction passes `StatelessTransactionValidator::validate` (the `max_l2_gas_amount` branch is skipped for Declare), is forwarded to the blockifier with `charge_fee = false`, and the OS executes it with `u64::MAX` as the initial gas budget. A gas-consuming contract will run to the block bouncer ceiling on every such simulation call.
