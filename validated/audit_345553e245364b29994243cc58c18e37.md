### Title
`calculateNFTRoyalties` Always Returns Full Offered Amount as "Net Proceeds" — Misleads NFT Seller into Accepting Offers at Understated Proceeds - (File: `packages/gui/src/components/offers/utils.ts`)

---

### Summary

In `calculateNFTRoyalties`, the `nftSellerNetAmount` field is hardcoded to `amount` (the full offered amount) because the actual deduction (`amount - royaltyAmount - makerFee`) is commented out. This is the direct GUI analog of the external `min(X+Y, X)` bug: a value that should vary based on royalty percentage always returns the constant input `amount`. The result is displayed as **"Net Proceeds"** in the NFT offer viewer when a buyer makes a `TokenForNFT` offer, misleading the NFT seller about what they will actually receive before they accept.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes royalty amounts and net proceeds for NFT offers:

```typescript
export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;          // ← always the full amount
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );                                              // ← correct deduction commented out
  ...
}
``` [1](#0-0) 

`nftSellerNetAmount` is always `amount` regardless of `royaltyPercentage` or `makerFee`. The correct formula — `amount - royaltyAmount - makerFee` — is commented out.

This value flows into two places in `NFTOfferViewer.tsx`:

1. **"Net Proceeds" display** (line 596): shown prominently to the NFT seller when reviewing a `TokenForNFT` offer. The tooltip at lines 588–591 explicitly promises this figure represents "the asking price, minus any associated creator fees." [2](#0-1) 

2. **`overrideNFTSellerAmount`** (lines 397–402): derived from `nftSellerNetAmount` and passed to `NFTOfferSummary` to override the "You will receive" amount shown in the purchase summary panel. [3](#0-2) 

Both displays show the full offered amount instead of the actual post-royalty proceeds.

---

### Impact Explanation

In the `TokenForNFT` flow, a buyer offers `amount` tokens for an NFT. Royalties are paid **from** the offered amount to the creator; the seller receives `amount - royaltyAmount`. The GUI's own tooltip confirms this model: *"those creator fees will be paid from the offered purchase price."* [4](#0-3) 

Because `nftSellerNetAmount` is always `amount`, the seller sees:
- **"Net Proceeds" = 100 XCH** (displayed)
- **Actual receipt = 80 XCH** (for a 20% royalty NFT)

The seller accepts the offer under a materially false representation of what they will receive. This constitutes display of the wrong amount in the offer-acceptance flow, directly causing the user to approve a transaction at understated proceeds.

---

### Likelihood Explanation

- Any unprivileged user can create a `TokenForNFT` offer for any royalty-bearing NFT.
- The victim (NFT seller) opens the offer file or link in the GUI — a standard workflow.
- NFTs with non-trivial royalties (5–20%) are common; the discrepancy is proportional to the royalty rate and the offer size.
- No special access, leaked keys, or social engineering beyond sharing a valid offer file is required.

---

### Recommendation

Restore the commented-out net proceeds calculation in `calculateNFTRoyalties`:

```typescript
// Before (broken):
const nftSellerNetAmount: number = amount;

// After (correct):
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [5](#0-4) 

The deduction should only apply in the `TokenForNFT` case (where royalties come from the offered amount). For `NFTForToken`, the seller receives the full asking price and the buyer pays royalties on top, so `nftSellerNetAmount` should remain `amount` for that branch. Add an `exchangeType` guard accordingly.

---

### Proof of Concept

1. Mint or obtain an NFT with a 20% royalty (`royaltyPercentage = 2000` in mojos representation).
2. As an attacker/buyer, create a `TokenForNFT` offer: offer 100 XCH for the NFT.
3. Share the offer file with the NFT owner (seller).
4. Seller opens the offer in the Chia GUI → `NFTOfferDetails` renders.
5. `calculateNFTRoyalties(100, 0, 20, TokenForNFT)` is called.
6. `nftSellerNetAmount = 100` (should be `80`).
7. GUI displays: **Net Proceeds: 100 XCH** and **"You will receive" override = 100 XCH**.
8. Seller accepts the offer.
9. On-chain settlement: creator receives 20 XCH royalty; seller receives **80 XCH** — 20 XCH less than displayed. [6](#0-5) [7](#0-6)
