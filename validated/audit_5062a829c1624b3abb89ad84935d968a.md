### Title
Missing Zero-Address Validation for `reward_address` Enables Permanent Freezing of Staker/Delegator Funds — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `staking.cairo::change_reward_address` and `pool.cairo::change_reward_address` (as well as the initial `stake` and `enter_delegation_pool` entry points) accept `reward_address = 0x0` without reverting. When rewards are later claimed or funds are unstaked, the protocol unconditionally calls `checked_transfer(recipient: reward_address, ...)` with the zero address, which reverts under the standard OpenZeppelin Cairo ERC20 implementation. Because `unstake_action` calls `send_rewards_to_staker` **before** returning the principal, a revert there permanently freezes the staker's principal stake inside the contract.

---

### Finding Description

**Vulnerability class**: Token-freeze bug — missing zero-address validation on a reward-destination field.

**Affected entry points (production files only)**:

1. `src/staking/staking.cairo` — `stake()` (line 288) and `change_reward_address()` (line 517)
2. `src/pool/pool.cairo` — `enter_delegation_pool()` (line 182) and `change_reward_address()` (line 505)

In `staking.cairo::change_reward_address`, the only guard is that `reward_address` is not a registered token address:

```cairo
// src/staking/staking.cairo:517-524
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No check: reward_address != 0
``` [1](#0-0) 

The same pattern exists in `pool.cairo::change_reward_address`:

```cairo
// src/pool/pool.cairo:505-510
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No check: reward_address != 0
``` [2](#0-1) 

And in `pool.cairo::enter_delegation_pool`:

```cairo
// src/pool/pool.cairo:195
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
// No check: reward_address != 0
``` [3](#0-2) 

When rewards are later distributed, `send_rewards_to_staker` unconditionally calls `checked_transfer` to whatever `reward_address` is stored:

```cairo
// src/staking/staking.cairo:1620-1625
let reward_address = staker_info.reward_address;
let amount = staker_info.unclaimed_rewards_own;
let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
claim_from_reward_supplier(:reward_supplier_dispatcher, :amount, :token_dispatcher);
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [4](#0-3) 

`unstake_action` calls `send_rewards_to_staker` **before** returning the principal to the staker:

```cairo
// src/staking/staking.cairo:494-506
let token_dispatcher = strk_token_dispatcher();
self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
// ...
token_dispatcher.checked_transfer(recipient: staker_address, amount: staker_amount.into());
``` [5](#0-4) 

If `reward_address` is `0x0`, the `checked_transfer` call inside `send_rewards_to_staker` reverts (OpenZeppelin Cairo ERC20 rejects zero-address recipients). Because this revert happens before the principal transfer, the staker's principal is permanently locked in the staking contract with no recovery path.

The same freeze applies to pool members: `pool.cairo::claim_rewards` calls `checked_transfer(recipient: reward_address, ...)` unconditionally. [6](#0-5) 

---

### Impact Explanation

- **Permanent freezing of unclaimed yield**: Any accrued STRK rewards are unclaimable once `reward_address = 0x0`.
- **Permanent freezing of principal stake**: Because `unstake_action` calls `send_rewards_to_staker` first and only then returns the principal, a revert in the reward transfer also blocks principal recovery. The staker's locked STRK is permanently inaccessible.

This matches the allowed impact: **High — Permanent freezing of unclaimed yield; Temporary/permanent freezing of funds**.

---

### Likelihood Explanation

Any staker or pool member can call `change_reward_address(0x0)` directly — it is an unprivileged, self-service call. A user may do this accidentally (e.g., passing an uninitialized variable) or as a result of a front-end bug. The initial `stake` and `enter_delegation_pool` calls also accept `reward_address = 0x0` without error, so the freeze can be set up at registration time. Likelihood is **Low** (requires user error or front-end bug), but the consequence is irreversible.

---

### Recommendation

Add a non-zero address assertion in all four entry points:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

This should be added to:
- `staking.cairo::stake` (before storing `reward_address`)
- `staking.cairo::change_reward_address` (alongside the existing token check)
- `pool.cairo::enter_delegation_pool` (alongside the existing token check)
- `pool.cairo::change_reward_address` (alongside the existing token check)

---

### Proof of Concept

1. Staker calls `staking.stake(reward_address: 0x0, operational_address: X, amount: MIN_STAKE)` — succeeds, no revert.
2. Protocol accrues rewards for the staker over several epochs.
3. Staker calls `staking.unstake_intent()` — succeeds.
4. After the exit window, staker calls `staking.unstake_action(staker_address)`.
5. Inside `unstake_action`, `send_rewards_to_staker` is called, which calls `checked_transfer(recipient: 0x0, amount: rewards)`.
6. The ERC20 `checked_transfer` reverts because `0x0` is an invalid recipient.
7. The entire `unstake_action` transaction reverts.
8. The staker's principal (`staker_amount`) is never returned — permanently frozen in the staking contract.
9. No privileged role can recover the funds; there is no admin withdrawal path for individual staker balances.
