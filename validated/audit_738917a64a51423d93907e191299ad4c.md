### Title
Participant-Set Mismatch Between `RerandomizationArguments` and `sign()` Causes Silent Signing Failure — (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

The `RerandomizationArguments` struct includes a `participants` field that is hashed into the HKDF-derived rerandomization scalar `delta`. However, neither `rerandomize_presign()` nor `sign()` validates that the `participants` used during rerandomization matches the `participants` passed to `sign()`. A malicious participant can supply a different participant set in their own `RerandomizationArguments`, producing an inconsistent `delta` and therefore an invalid signature share. The coordinator's final `sig.verify()` fails, the presignature is consumed (one-time use per documentation), and honest parties must restart from expensive presignature generation.

**Phase 1 — Rerandomization:**

`derive_randomness()` hashes the `participants` list into the HKDF input alongside `pk`, `tweak`, `msg_hash`, `big_r`, and `entropy`: [1](#0-0) 

This means any difference in the `participants` field produces a completely different scalar `delta`.

`rerandomize_presign()` validates only that `presignature.big_r == args.big_r`; it does **not** validate that `args.participants` matches the participant set that will be used in `sign()`: [2](#0-1) [3](#0-2) 

**Phase 2 — Signing:**

`sign()` accepts `participants` and `presignature: RerandomizedPresignOutput` as independent arguments. `RerandomizedPresignOutput` carries no record of which `participants` were used during rerandomization, and `sign()` performs no cross-check: [4](#0-3) [5](#0-4) 

Each participant independently computes their Lagrange coefficient from the `sign()` participant list and multiplies it against their `k`/`sigma` (OT-based) or `alpha`/`beta` (robust) shares, which were already scaled by their own `delta`: [6](#0-5) [7](#0-6) 

If a malicious participant used a different `participants` in their `RerandomizationArguments`, their `delta` differs from every honest participant's `delta`. Their share is therefore inconsistent with the rest, the summed `s` is wrong, and the coordinator's final assertion fails: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

**Impact: High — Permanent denial of signing for honest parties.**

The presignature is documented as one-time use: [10](#0-9) [11](#0-10) 

Once the signing round completes (even with failure), the presignature must be discarded. For OT-based ECDSA, regenerating a presignature requires a full new Beaver triple generation protocol — a multi-round, bandwidth-intensive offline phase. A malicious participant present in every signing attempt can permanently block signing for the honest majority by repeating this attack on every presignature, with no cryptographic detection mechanism to identify the culprit (the error is a generic `"signature failed to verify"`).

---

### Likelihood Explanation

**Likelihood: Medium.**

Any single participant in the signing set can mount this attack unilaterally. The attacker only needs to supply a different `participants` slice when constructing their own `RerandomizationArguments` before calling `rerandomize_presign()`. No special privilege, leaked key, or external assumption is required. The attack is invisible to the coordinator and other participants — the protocol simply aborts with a verification failure.

---

### Recommendation

1. **Bind `participants` into `RerandomizedPresignOutput`**: Store the sorted participant bytes used during rerandomization inside `RerandomizedPresignOutput` so the signing phase can verify consistency.

2. **Cross-check in `sign()`**: At the start of `sign()`, assert that the `participants` argument matches the participant set embedded in the `RerandomizedPresignOutput`.

3. **Alternative — derive `participants` inside `sign()`**: Remove `participants` from `RerandomizationArguments` entirely and instead pass the signing `ParticipantList` directly into `rerandomize_presign()` (or derive `delta` inside `sign()` after the participant set is fixed), eliminating the gap between the two phases.

---

### Proof of Concept

```
Setup:
  - Participants: {P1, P2, P3}, threshold = 2
  - Run presign → each Pi holds PresignOutput{big_r, k_i, sigma_i}

Attack (P3 is malicious):
  - P1, P2 create RerandomizationArguments{participants = {P1,P2,P3}, ...}
    → derive delta_correct
    → RerandomizedPresignOutput with k_i / delta_correct, sigma_i / delta_correct

  - P3 creates RerandomizationArguments{participants = {P1,P2,P4}, ...}  ← wrong set
    → derive delta_wrong  (delta_wrong ≠ delta_correct because participants differ)
    → RerandomizedPresignOutput with k_3 / delta_wrong, sigma_3 / delta_wrong

  - All three call sign(participants={P1,P2,P3}, ...)
    → P1, P2 compute correct shares (lambda_i * k_i / delta_correct)
    → P3 computes wrong share    (lambda_3 * k_3 / delta_wrong)

  - Coordinator sums shares → s is wrong
  - sig.verify() fails → ProtocolError::AssertionFailed("signature failed to verify")
  - Presignature consumed; honest parties must regenerate triples + presignature
  - P3 repeats on every subsequent presignature → permanent denial of signing
```

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<Scalar, ProtocolError> {
    // Round 1
    // Linearize ki
    // Spec 1.1
    let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
    let k_i = lambda * presignature.k;

    // Linearize sigmai
    // Spec 1.2
    let sigma_i = lambda * presignature.sigma;

    // Compute si = h * ki + Rx * sigmai
    // Spec 1.3
    let r = x_coordinate(&presignature.big_r);
    Ok(msg_hash * k_i + r * sigma_i)
}

    coordinator: Participant,

    msg_hash: Scalar,

    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))

/// A common computation done by both the coordinator and the other participants
fn compute_signature_share(
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
    participants: &ParticipantList,
    me: Participant,
) -> Result<SerializableScalar<C>, ProtocolError> {
    // (beta_i + tweak * k_i) * delta^{-1}
    let big_r = presignature.big_r;
    let big_r_x_coordinate = x_coordinate(&big_r);
    // beta * Rx + e
    let beta = presignature.beta * big_r_x_coordinate + presignature.e;

    let s_me = msg_hash * presignature.alpha + beta;
    // lambda_i * s_i
    let linearized_s_me = s_me * participants.lagrange::<C>(me)?;
    Ok(SerializableScalar::<C>(linearized_s_me))
}
