### Title
Gateway Admits Invoke Transactions Without Signature Verification via `skip_stateful_validations` UX Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validation path contains an "event-based" admission shortcut for the deploy-account + invoke UX flow. When an invoke transaction carries `nonce == 1` and the account's on-chain nonce is `0`, the gateway checks only whether any transaction for that account exists in the mempool or a recent block — a boolean state query — and if so, skips the `__validate__` entry-point call entirely. This is the direct Sequencer analog of the Uniswap "event-based vs. result-based" authentication finding: the admission decision is decoupled from the cryptographic check it is supposed to guard.

---

### Finding Description

**The bypass path** is in `skip_stateful_validations`: [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The account's current on-chain nonce is `Nonce(Felt::ZERO)` (not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The result is consumed in `run_pre_validation_checks`: [2](#0-1) 

And passed directly into `run_validate_entry_point` as the `validate` flag: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling the account's `__validate__` entry point: [4](#0-3) 

**The attacker-controlled trigger** requires no privilege:

1. Alice submits a `DeployAccount` for address `A`; it enters the mempool.
2. Attacker (Bob) observes `A` in the mempool and immediately submits an `Invoke` from `A` with `nonce = 1` and an **arbitrary/invalid signature**.
3. The gateway evaluates: `nonce == 1` ✓, `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block(A) == true` ✓ → `skip_validate = true`.
4. `run_validate_entry_point` builds `ExecutionFlags { validate: false, … }` and calls `blockifier_validator.validate(account_tx)`.
5. `perform_validations` hits the early-return branch and returns `Ok(())` — no `__validate__` call, no signature check.
6. Bob's invalid invoke is forwarded to the mempool and admitted.

The condition `account_nonce == Nonce(Felt::ZERO)` is read from the **gateway's fixed-block state reader** (a snapshot of the last committed block), not from the live mempool state. This window is open for the entire time between Alice's `DeployAccount` entering the mempool and the block containing it being committed.

---

### Impact Explanation

**Mempool/gateway admission accepts invalid transactions before sequencing** (High per scope).

- Bob's invalid invoke occupies Alice's `nonce = 1` slot in the mempool. Depending on the mempool's replacement policy, Alice's legitimate `nonce = 1` invoke may be rejected as a duplicate or require a fee bump to replace it.
- The batcher fetches and attempts to execute the invalid transaction. During sequencing, `AccountTransaction::new_for_sequencing` sets `validate: true`: [5](#0-4) 

  The `__validate__` entry point is then called and fails, causing the transaction to revert. The batcher wastes execution resources and the block slot is consumed by a provably-invalid transaction.
- An attacker can repeat this for every new `DeployAccount` observed in the mempool, creating a sustained DoS on the deploy-account + invoke UX path and on batcher throughput.

---

### Likelihood Explanation

The trigger is **unprivileged and fully observable**: `DeployAccount` transactions are public mempool entries. The attacker needs only to watch the mempool and race-submit an invoke with `nonce = 1` before the deploy-account is committed. The window is the full block time (seconds to minutes). No special knowledge of the victim's private key or contract is required.

---

### Recommendation

Replace the event-based boolean check with a result-based guard. Two options:

1. **Remove the UX shortcut entirely.** Require the `DeployAccount` to be committed before the `Invoke` is accepted. This is the safest option.

2. **Bind the skip to a verifiable invariant.** If the UX shortcut must be kept, require that the `DeployAccount` transaction in the mempool was submitted by the same sender address (i.e., the deploy-account's `contract_address` matches the invoke's `sender_address` AND the deploy-account's signature was already verified). Do not skip `__validate__` for the invoke; instead, defer it to a lightweight pre-execution check that does not require the account to be deployed yet (e.g., verify the signature against the public key embedded in the deploy-account's constructor calldata).

3. **At minimum**, ensure the mempool's replacement policy allows the legitimate owner to always displace an invalid nonce-1 transaction without a fee penalty.

---

### Proof of Concept

```

2. Bob observes A in mempool via any public RPC/P2P endpoint.

3. Bob submits Invoke{sender=A, nonce=1, calldata=[transfer all funds to Bob],
                      signature=[0x0, 0x0]}  ← invalid signature

4. Gateway stateful validator:
   account_nonce = get_nonce_from_state(A) = 0   (A not yet on-chain)
   skip_stateful_validations(invoke_nonce=1, account_nonce=0,

   ExecutionFlags { validate: false }
   StatefulValidator::perform_validations → Ok(()) without __validate__ call

5. Bob's invoke is added to mempool at nonce slot 1 for address A.

6. Alice submits her own Invoke{sender=A, nonce=1, calldata=[...], signature=[valid]}
   → mempool rejects or requires replacement fee (nonce slot already occupied).

7. Batcher executes Bob's invoke with validate:true → __validate__ fails → revert.
   Alice's legitimate nonce-1 transaction is blocked or delayed.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
