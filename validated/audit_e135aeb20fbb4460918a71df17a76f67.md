### Title
Unimplemented `frost_core::Ciphersuite` Hash Methods in `BLS12381SHA256` Cause Unconditional Panic on Signing — (File: `src/confidential_key_derivation/ciphersuite.rs`)

---

### Summary

`BLS12381SHA256` implements `frost_core::Ciphersuite` but leaves six required trait methods — `H1`, `H2`, `H3`, `H4`, `H5`, and `HID` — as `unimplemented!()` stubs. Because `BLS12381SHA256` satisfies all trait bounds required by the public `frost::presign` function, a library caller can invoke `presign::<BLS12381SHA256>()`. This call reaches `frost_core::round1::commit`, which calls `C::H4()` for nonce generation. Since `H4` is `unimplemented!()`, the runtime panics unconditionally, permanently denying the signing operation for any participant using this ciphersuite with the FROST signing path.

---

Inside `do_presign`, `frost_core::round1::commit` is called at line 101:

```rust

let (nonces, commitments) = commit(&signing_share, &mut rng);
```

`frost_core::round1::commit` internally calls `C::H4()` to generate the hiding nonce via `Nonce::nonce_generate_from_random_bytes`. Since `BLS12381SHA256::H4` is `unimplemented!()`, this triggers an unconditional `panic!` at runtime — not a recoverable `Err`, but a hard abort of the entire protocol execution.

This is a direct structural analog to the external report: a required interface method is absent (stubbed), and a reachable protocol call path invokes it, causing the operation to permanently fail.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

Any invocation of `frost::presign::<BLS12381SHA256>()` will panic unconditionally at the nonce-generation step. Because the panic is not wrapped in a `Result`, it cannot be caught by the caller's error-handling logic in normal Rust (without `std::panic::catch_unwind`). The presigning phase — and therefore the entire signing protocol — is permanently denied for any participant that reaches this code path. Since `BLS12381SHA256` is the ciphersuite for the CKD key material (`KeygenOutput = crate::KeygenOutput<BLS12381SHA256>`), a caller who attempts to use CKD-derived key material with the FROST signing path will find signing permanently broken.

---

### Likelihood Explanation

**Medium-Low.** `BLS12381SHA256` is intended for the CKD protocol, not FROST signing. However, there is no compile-time seal, marker trait, or type-system restriction preventing its use with `frost::presign`. The `#[allow(unused)]` attributes on the stubs indicate the developers are aware these methods are not called in the CKD flow, but the public generic API exposes the gap. A library integrator who holds `BLS12381SHA256`-based `KeygenOutput` and attempts to use the generic `frost::presign` entry point — a plausible mistake given the shared `KeygenOutput<C>` type — will trigger the panic with no compile-time warning.

---

### Recommendation

1. **Seal the type from FROST signing use**: Introduce a marker trait (e.g., `FrostSigningCiphersuite`) implemented only by ciphersuites with complete `H1`–`H5` implementations, and bound `frost::presign` on it instead of the bare `Ciphersuite`.
2. **Replace `unimplemented!()` with `compile_error!` or a sealed trait**: This converts a runtime panic into a compile-time error, making misuse impossible.
3. **Implement the missing methods**: If `BLS12381SHA256` is ever intended to support FROST signing, provide correct implementations of `H1`–`H5` and `HID` following the FROST specification for BLS12-381.
