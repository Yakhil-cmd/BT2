### Title
Missing Minimum Threshold Bound in `assert_sign_inputs` Allows Denial of FROST Signing — (`src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that the threshold does not exceed the participant count but omits the symmetric lower-bound check (`threshold >= 2`). Passing `threshold = 1` (or `0`) silently passes initialization, then causes a `ProtocolError::InvalidInterpolationArguments` deep inside Lagrange coefficient computation during the actual signing round, permanently denying signature generation for all honest participants in that session.

---

### Finding Description

`assert_sign_inputs` enforces only an upper bound on the threshold:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
``` [1](#0-0) 

There is no corresponding lower-bound guard. Compare this with `validate_triple_inputs` in the OT-based ECDSA path, which explicitly rejects `threshold < 2`:

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
``` [2](#0-1) 

The same omission exists in the FROST `presign` function, which also only checks the upper bound: [3](#0-2) 

When `threshold = 1` reaches `compute_lagrange_coefficient`, the function immediately returns an error because it requires at least two points:

```rust
if points_set.len() <= 1 {
    return Err(ProtocolError::InvalidInterpolationArguments);
}
``` [4](#0-3) 

The same guard exists in `batch_compute_lagrange_coefficients`: [5](#0-4) 

`assert_sign_inputs` is the shared entry-point called by both the EdDSA and RedJubjub signing paths: [6](#0-5) 

---

### Impact Explanation

Any caller — including a malicious coordinator or a misconfigured application — who passes `threshold = 1` (or `0`) to the EdDSA or RedJubjub `sign` function will receive `Ok(participants)` from `assert_sign_inputs`, proceed into the signing protocol, and then receive `ProtocolError::InvalidInterpolationArguments` when Lagrange reconstruction is attempted. Because the error surfaces inside the async protocol execution rather than at initialization, all honest participants in the session have already committed nonces and sent messages before the failure is observed. The signing session is aborted with no valid signature produced.

This matches the allowed impact: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

The `ReconstructionLowerBound` wrapper type accepts any `usize`, including `0` and `1`: [7](#0-6) 

No type-level or API-level constraint prevents a caller from constructing `ReconstructionLowerBound(1)`. A malicious coordinator controlling the threshold argument, or an honest application that accidentally passes `1` instead of the keygen threshold, will trigger the failure. The keygen path correctly rejects `threshold < 2` via `assert_key_invariants`, creating a false sense of safety that the signing path is equally guarded. [8](#0-7) 

---

### Recommendation

Add the missing lower-bound check inside `assert_sign_inputs` (and symmetrically inside `presign`):

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the guard already present in `validate_triple_inputs` and makes the FROST signing path consistent with the keygen and OT-ECDSA triple-generation paths.

---

### Proof of Concept

1. Call `assert_sign_inputs(&[p1, p2], ReconstructionLowerBound(1), p1, p2)`.
2. Observe: the function returns `Ok(participants)` — no error.
3. Proceed to run the EdDSA or RedJubjub `sign` protocol with `threshold = 1`.
4. During the signing round, `compute_lagrange_coefficient` or `batch_compute_lagrange_coefficients` is invoked with a single-element point set.
5. Both functions return `Err(ProtocolError::InvalidInterpolationArguments)` because `points_set.len() <= 1`.
6. The signing session terminates with an error; no signature is produced for any honest participant.

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
