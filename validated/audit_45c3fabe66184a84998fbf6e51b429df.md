### Title
Zero `priceProviderTimelock` Allows Pool Admin to Atomically Rotate Price Provider and Execute Swaps at Manipulated Prices — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`MetricOmmPoolFactory._validatePoolParameters` does not require `priceProviderTimelock > 0` for mutable-oracle pools. A pool deployed with `priceProviderTimelock = 0` lets the pool admin call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` sequentially in a single transaction, bypassing the intended delay entirely and replacing the live oracle with an arbitrary price provider before any LP or swapper can react.

---

### Finding Description

**Root cause — missing validation in `_validatePoolParameters`:**

`createPool` stores `params.priceProviderTimelock` directly without any lower-bound check for the mutable-oracle path:

```solidity
// MetricOmmPoolFactory.sol line 164
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
// ...
priceProviderTimelock[pool] = params.priceProviderTimelock;   // line 213 — stored verbatim, no > 0 guard
``` [1](#0-0) [2](#0-1) 

`_validatePoolParameters` checks tokens, admin, price-provider token match, fee caps, initial amounts, and minimal liquidity — but never validates `priceProviderTimelock`: [3](#0-2) 

**How the timelock is computed and enforced:**

`proposePoolPriceProvider` sets:
```solidity
uint256 executeAfter = block.timestamp + timelock;   // line 487
```
With `timelock = 0`, `executeAfter = block.timestamp`. [4](#0-3) 

`executePoolPriceProviderUpdate` enforces:
```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);   // line 499
```
When `execAfter == block.timestamp`, the condition `block.timestamp < block.timestamp` is `false`, so the check passes immediately. [5](#0-4) 

**Why both calls fit in one transaction:**

The factory inherits OpenZeppelin's `ReentrancyGuardTransient`, whose `nonReentrant` modifier sets a transient flag, executes the function body, then explicitly clears the flag via `_nonReentrantAfter`. Sequential (non-nested) calls in the same transaction are therefore permitted — the guard is cleared after each call completes. [6](#0-5) 

---

### Impact Explanation

A pool admin who deployed (or controls) a pool with `priceProviderTimelock = 0` can, in a single atomic transaction:

1. Call `proposePoolPriceProvider(pool, maliciousProvider)` — `maliciousProvider` must satisfy `_validatePriceProvider` (correct `token0()`/`token1()` return values) but can return any bid/ask prices.
2. Call `executePoolPriceProviderUpdate(pool)` — succeeds immediately because `block.timestamp < block.timestamp` is false.
3. Execute one or more swaps against the pool while the malicious oracle is active, extracting value from LP reserves at an off-market price.
4. Optionally restore the original provider in the same transaction.

LPs suffer direct loss of principal proportional to the price deviation the attacker injects. Swappers in the same block receive worse execution than the real market price. Because pool creation is permissionless, any attacker can deploy such a pool, attract liquidity, and execute this sequence.

The contest README explicitly scopes this class: *"Pool Admin — semi-trusted, bounded … Cannot exceed caps or bypass timelocks. If they can exceed caps or bypass timelocks and it leads to Medium or higher severity issue, then it can be valid."* [7](#0-6) 

---

### Likelihood Explanation

Pool creation is fully permissionless once `poolDeployer` is set. An attacker needs only to:
- Deploy a pool with `priceProviderTimelock = 0` (no factory guard prevents this).
- Deploy a price-provider contract that returns the correct token pair but manipulated bid/ask values.
- Attract LP deposits (e.g., by seeding the pool with initial liquidity and advertising it).

The attack requires no external flash loan, no oracle manipulation, and no privileged factory-owner cooperation. The only prerequisite is LP capital in the pool.

---

### Recommendation

In `_validatePoolParameters`, add a lower-bound check for the mutable-oracle path:

```solidity
// in MetricOmmPoolFactory._validatePoolParameters
if (params.priceProviderTimelock != type(uint256).max && params.priceProviderTimelock == 0) {
    revert InvalidPriceProviderTimelock();
}
```

This mirrors the pattern already applied to `minimalMintableLiquidity` and `initialAmount*PerShareE18`, where zero values are rejected because they would silently break a safety invariant. [8](#0-7)

### Proof of Concept

```solidity
// Attacker contract (simplified)
contract Exploit {
    IMetricOmmPoolFactory factory;
    address pool;
    MaliciousProvider malProvider; // implements IPriceProvider, returns manipulated bid/ask

    function setup() external {
        // 1. Deploy pool with priceProviderTimelock = 0 (mutable, zero delay)
        PoolParameters memory p = ...; // priceProviderTimelock = 0
        pool = factory.createPool(p);
        // 2. Seed pool with liquidity to attract LPs
    }

    function attack() external {
        // All in one transaction:
        // Step A: propose malicious provider (executeAfter = block.timestamp)
        factory.proposePoolPriceProvider(pool, address(malProvider));
        // Step B: execute immediately — block.timestamp < block.timestamp is false
        factory.executePoolPriceProviderUpdate(pool);
        // Step C: swap at manipulated price, draining LP reserves
        IMetricOmmPool(pool).swap(...);
        // Step D: restore original provider (optional, to hide the attack)
        factory.proposePoolPriceProvider(pool, originalProvider);
        factory.executePoolPriceProviderUpdate(pool);
    }
}
```

The `_validatePriceProvider` call inside `proposePoolPriceProvider` only checks that `malProvider.token0()` and `malProvider.token1()` match the pool's tokens — it does not constrain the bid/ask values `malProvider.getBidAndAskPrice()` returns. [9](#0-8)
