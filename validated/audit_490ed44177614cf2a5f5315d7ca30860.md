### Title
Wrong Total Stake Passed to `_update_rewards` in `update_rewards`, Causing Inflated Reward Allocation - (File: `src/staking/staking.cairo`)

### Summary

`StakingRewardsManagerImpl::update_rewards` passes the **individual staker's balance** (`staker_total_strk_balance`) as the `strk_total_stake` / `btc_total_stake` arguments to `_update_rewards`, instead of the **protocol-wide total staking power**. The sibling function `update_rewards_from_attestation_contract` correctly fetches the global total via `self.get_current_total_staking_power()` for the same parameters. Because `_update_rewards` uses these values as the denominator when computing each staker's proportional share of block rewards, passing the staker's own balance inflates their reward fraction to 100 % of the epoch's block rewards.

### Finding Description

`update_rewards` (consensus-rewards path) computes the staker's balance and then passes it directly as the total-stake denominator:

```cairo
// src/staking/staking.cairo  lines 1474-1506
let (staker_total_strk_balance, staker_total_btc_balance) = self
    .get_staker_total_strk_btc_balance_at_epoch(
        :staker_address, :staker_pool_info, epoch_id: curr_epoch,
    );
...
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_block_rewards,
    btc_total_rewards: btc_block_rewards,
    strk_total_stake: staker_total_strk_balance,   // ← staker's own balance, NOT total
    btc_total_stake:  staker_total_btc_balance,    // ← staker's own balance, NOT total
    ...
);
``` [1](#0-0) 

The attestation-rewards path in the same file does it correctly:

```cairo
// src/staking/staking.cairo  lines 1406-1422
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
...
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_epoch_rewards,
    btc_total_rewards:  btc_epoch_rewards,
    :strk_total_stake,   // ← correct: protocol-wide total
    :btc_total_stake,    // ← correct: protocol-wide total
    ...
);
``` [2](#0-1) 

`_update_rewards` uses `strk_total_stake` as the denominator in the proportional reward split (analogous to `getVestedFraction` in the external report). When the denominator equals the numerator (staker's own balance), the staker receives the full block reward rather than their proportional share.

### Impact Explanation

`_update_rewards` computes each staker's reward as:

```
staker_reward ≈ staker_balance / strk_total_stake × total_block_rewards
```

With `strk_total_stake = staker_balance`, the ratio becomes 1, so the staker claims **100 % of the block rewards** for that block. Because `last_reward_block` is written before the reward computation, only one `update_rewards` call succeeds per block. A malicious staker who front-runs this call each block drains the entire consensus-reward stream, stealing unclaimed yield that belongs to all other stakers.

**Impact category: High — Theft of unclaimed yield.**

### Likelihood Explanation

`update_rewards` is a fully public function gated only by `general_prerequisites()` (unpaused + non-zero caller). [3](#0-2) 

Any registered staker can call it once per block. On Starknet, block times are short and predictable, making front-running straightforward. No privileged role, leaked key, or external dependency is required.

### Recommendation

Replace the staker-scoped balance variables with the protocol-wide total staking power, mirroring `update_rewards_from_attestation_contract`:

```cairo
// Replace:
let (staker_total_strk_balance, staker_total_btc_balance) = self
    .get_staker_total_strk_btc_balance_at_epoch(...);

// With:
let (strk_total_stake, btc_total_stake) = self
    .get_total_staking_power_at_epoch(epoch_id: curr_epoch);
```

Then pass `strk_total_stake` / `btc_total_stake` to `_update_rewards` instead of the per-staker values. The staker's individual balance is still needed only for the non-zero assertion and can be fetched separately.

### Proof of Concept

1. Alice is a staker with 1 000 STRK staked. Total protocol stake is 1 000 000 STRK.
2. At the start of block N, Alice calls `update_rewards(alice_address, false)`.
3. `staker_total_strk_balance` = 1 000 (Alice's balance).
4. `strk_block_rewards` = 500 STRK (protocol block reward).
5. `_update_rewards` receives `strk_total_stake = 1 000` (Alice's balance, not 1 000 000).
6. Alice's computed reward = `1 000 / 1 000 × 500 = 500 STRK` instead of the correct `1 000 / 1 000 000 × 500 = 0.5 STRK`.
7. Alice claims 1 000× her fair share; all other stakers receive nothing for block N.
8. Repeated every block, Alice drains the entire reward stream.
