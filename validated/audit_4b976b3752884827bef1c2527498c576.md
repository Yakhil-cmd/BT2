Audit Report

## Title
Index-out-of-bounds panic via empty `EnvelopePair` vec in `construction_parse` crashes Rosetta process - (File: `rs/rosetta-api/icp/src/request_handler/construction_parse.rs`)

## Summary
The `construction_parse` handler unconditionally indexes `updates[0]` on a `Vec<EnvelopePair>` that is fully attacker-controlled. Because `SignedTransaction` deserialization performs no structural validation beyond CBOR well-formedness, an attacker can supply a signed transaction containing a `Request` with an empty envelope pair list. Rust panics with an index-out-of-bounds, terminating the Rosetta process and making it unavailable until restarted.

## Finding Description
`Request` is defined as `(RequestType, Vec<EnvelopePair>)` with no minimum-length constraint. [1](#0-0) 

`SignedTransaction::from_str` decodes the hex-encoded bytes via `serde_cbor::from_slice` and, on failure, retries as a `LegacySignedTransaction`. Neither path validates that any inner `Vec<EnvelopePair>` is non-empty. [2](#0-1) 

`ParsedTransaction::try_from` similarly only checks CBOR decodability for the signed branch, returning `Ok(ParsedTransaction::Signed(...))` for any structurally valid CBOR regardless of envelope content. [3](#0-2) 

In `construction_parse`, the signed branch immediately indexes `updates[0]` with no bounds check: [4](#0-3) 

If `updates` is an empty `Vec`, Rust panics with `index out of bounds: the len is 0 but the index is 0`, which is an unrecoverable panic that terminates the Rosetta process.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The Rosetta API is an in-scope financial integration component. A single crafted HTTP request crashes the entire Rosetta process, dropping all in-flight requests and making the service unavailable until an operator manually restarts it.

## Likelihood Explanation
The `/construction/parse` endpoint is public and unauthenticated. The CBOR encoding of `SignedTransaction` is straightforward and documented. No volume, no authentication, no privileged role, and no victim interaction is required. The exploit is deterministic and repeatable with a single request.

## Recommendation
Add a bounds check before accessing `updates[0]` in the signed branch of `construction_parse`. Return `ApiError::invalid_request` if the `Vec<EnvelopePair>` is empty:

```rust
ParsedTransaction::Signed(signed_transaction) => signed_transaction
    .requests
    .iter()
    .map(|(request_type, updates)| {
        let first = updates.first().ok_or_else(|| {
            ApiError::invalid_request("Empty envelope pair list".to_string())
        })?;
        match first.update.content.clone() {
            HttpCallContent::Call { update } => Ok((request_type.clone(), update)),
        }
    })
    .collect::<Result<Vec<_>, _>>()?,
```

## Proof of Concept
```rust
// Build a SignedTransaction with one Request containing an empty EnvelopePair vec
let signed_tx = SignedTransaction {
    requests: vec![(RequestType::Send, vec![])],  // empty Vec<EnvelopePair>
};
let cbor_bytes = serde_cbor::to_vec(&signed_tx).unwrap();
let hex_str = hex::encode(&cbor_bytes);

// POST to /construction/parse:
// { "network_identifier": {...}, "signed": true, "transaction": "<hex_str>" }
//
// -> ParsedTransaction::try_from succeeds (valid CBOR)
// -> construction_parse iterates requests, hits updates[0] on empty vec
// -> thread panics: index out of bounds: the len is 0 but the index is 0
// -> Rosetta process terminates
```
