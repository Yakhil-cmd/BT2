### Title
Pool Rewards Permanently Frozen When Pool Balance Falls Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When a delegation pool's total balance is below `min_delegation_for_rewards`, the staking contract still calculates non-zero pool rewards and **transfers the STRK tokens to the pool contract**, but the pool contract's `compute_rewards_per_unit` silently returns zero, leaving the cumulative rewards trace unchanged. Delegators can never claim these rewards; the STRK tokens are permanently locked inside the pool contract.

---

### Finding Description

**Root cause — two-step mismatch between reward transfer and reward accounting:**

**Step 1 — Staking contract calculates and sends rewards regardless of pool size.**

In `calculate_staker_pools_rewards` (`src/staking/staking.cairo` lines 1969–1999), pool rewards are computed as:

```
pool_rewards_including_commission = total_rewards * pool_balance_curr_epoch / total_stake
```

No check is made against `min_delegation_for_rewards` here. If `pool_rewards > 0`, the entry is appended to `pool_rewards_array` and `update_pool_rewards` is called.

Inside `update_pool_rewards` (lines 1865–1891), for every entry the staking contract:
1. Calls `send_rewards_to_delegation_pool` — **actual STRK token transfer to the pool contract**.
2. Calls `pool_dispatcher.update_rewards_from_staking_contract(rewards: pool_rewards, pool_balance: pool_balance.to_native_amount(:decimals))`.

**Step 2 — Pool contract silently drops the accounting update.**

In `update_rewards_from_staking_contract` (`src/pool/pool.cairo` lines 569–587), the pool calls `compute_rewards_per_unit(staking_rewards: rewards, total_stake: pool_balance)`.

In `compute_rewards_per_unit` (lines 967–978):

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    ...
}
```

When `pool_balance < min_delegation_for_rewards`, the function returns `0`. The cumulative rewards trace is updated with `last + 0 = last` — **no change**. The STRK tokens already transferred in Step 1 are now stranded in the pool contract with no accounting entry to back them.

The code comment itself acknowledges this: *"Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case."*

**Step 3 — Delegators receive nothing.**

When a delegator calls `claim_rewards` (`src/pool/pool.cairo` lines 335–377), rewards are computed from the cumulative rewards trace via `calculate_rewards`. Because the trace was not updated, the computed reward is zero. The STRK tokens remain locked in the pool contract with no recovery path.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.**

STRK reward tokens are transferred into the pool contract by the staking contract but are irrecoverable: no `rescue`, `sweep`, or administrative withdrawal function exists in the pool contract. Every reward distribution cycle where `pool_balance < min_delegation_for_rewards` silently burns that epoch's pool rewards into the contract's balance, permanently inaccessible to delegators.

This matches the allowed impact: **"Permanent freezing of unclaimed yield or unclaimed royalties"** (High).

---

### Likelihood Explanation

**Moderate — reachable by any unprivileged delegator.**

- For STRK pools: `min_delegation_for_rewards = 10^18` (1 STRK). A pool whose total delegated balance is less than 1 STRK triggers the bug. This is realistic for newly opened pools or pools with a single small delegator.
- For BTC pools (8 decimals): `min_delegation_for_rewards = 10^3` (1000 satoshis = 0.00001 BTC). Even easier to trigger.
- No privileged role is required. `enter_delegation_pool` only checks `amount.is_non_zero()` — there is no minimum delegation floor.
- The condition persists across every reward epoch until the total pool balance is raised above the threshold, but rewards from past epochs are already permanently lost.

---

### Recommendation

In `calculate_staker_pools_rewards` (`src/staking/staking.cairo`), before appending to `pool_rewards_array`, check whether the pool balance meets the minimum threshold. If it does not, skip the reward transfer entirely rather than sending tokens that cannot be accounted for:

```cairo
let native_pool_balance = pool_balance_curr_epoch.to_native_amount(:decimals);
// Skip pools whose balance is below the minimum required for reward accounting.
let pool_dispatcher = IPoolDispatcher { contract_address: pool_contract };
if native_pool_balance < pool_dispatcher.get_min_delegation_for_rewards() {
    continue;
}
```

Alternatively, expose `min_delegation_for_rewards` as a view on the pool contract and gate the reward calculation in the staking contract before any token transfer occurs.
