### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Threshold-Bypass Corruption — (`File: src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that the caller-supplied `threshold` does not exceed the participant count, but **never enforce a minimum value of 2**. A malicious coordinator or caller can supply `threshold = 0` or `threshold = 1`, bypassing the quorum requirement enforced at key-generation time and causing the signing protocol to produce cryptographically invalid (corrupted) output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
// src/dkg.rs lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` only enforces the upper bound:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
// ← no lower-bound check; threshold = 0 or 1 passes silently
``` [2](#0-1) 

The same omission exists in the `presign` function in the same file:

```rust
// src/frost/mod.rs lines 71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforced at construction time, so any value including 0 can be passed in: [4](#0-3) 

The DKG invariant (`threshold >= 2`) is therefore not re-checked at signing time, creating an implicit relationship between the keygen threshold and the signing threshold that is never enforced.

---

### Impact Explanation

When `threshold = 1` is supplied to the FROST signing path:

- Lagrange interpolation is performed over only one participant's share, with coefficient 1 (trivially).
- The aggregated signature scalar `s` equals a single participant's share contribution, not the correct reconstruction of the distributed secret.
- The resulting `(R, s)` pair fails verification against the public key that was generated with the true threshold ≥ 2.
- Every honest participant's signing round completes without error, yet the output signature is cryptographically invalid and unusable.

When `threshold = 0`:

- Lagrange interpolation over an empty set is attempted, which can produce a panic or a degenerate zero-polynomial result, causing the protocol to abort or emit a zero signature.

Both cases corrupt the signing output and permanently deny honest parties a usable signature for that session.

**Matched impact:** *High — Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs / Permanent denial of signing for honest parties under valid protocol inputs.*

---

### Likelihood Explanation

The entry point is the public `assert_sign_inputs` function and the `presign` function, both callable by any library user acting as coordinator. No special privilege beyond being a protocol participant is required. A malicious coordinator who controls the `threshold` argument (a plain `usize` cast to `ReconstructionLowerBound`) can trivially pass 1 without triggering any error. The inconsistency with `assert_key_invariants` makes this an easy mistake for an integrator to miss.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Consider also enforcing the invariant inside `ReconstructionLowerBound::from` / a dedicated constructor so that an out-of-range value can never be constructed in the first place, mirroring the defense-in-depth approach used elsewhere.

---
