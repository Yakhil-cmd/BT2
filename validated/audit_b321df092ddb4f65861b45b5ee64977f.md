### Title
`uint32` Truncation in `_afterTimelock` Allows Pool Admin to Bypass LP-Protection Timelock — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` casts the sum `block.timestamp + timelock` from `uint256` to `uint32`. When the pool admin sets `timelock` to any value that pushes the sum past `type(uint32).max` (~4.29 billion), the cast silently wraps to a timestamp already in the past. Every subsequent `_requireElapsed` check passes immediately, letting the admin execute drawdown/decay/watermark changes with zero waiting time — defeating the only LP-protection mechanism in the extension.

---

### Finding Description

`_afterTimelock` computes the deadline for all timelocked parameter changes:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32` (max ≈ 4.29 × 10⁹ s). The addition is performed in `uint256` space, then truncated to `uint32`. At the current epoch (~1.75 × 10⁹), any `timelock` value above ≈ 2.54 × 10⁹ s (≈ 80 years) causes the sum to exceed `type(uint32).max`, wrapping to a small number that is already in the past.

The returned `executeAfter` value is stored directly into the schedule struct:

```solidity
uint32 executeAfter = _afterTimelock(pool_);
sched.pendingTimelockExecuteAfter = executeAfter;
``` [2](#0-1) 

The guard that enforces the delay compares `block.timestamp` (uint256) against the stored `uint32`:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [3](#0-2) 

Because the wrapped value is already less than `block.timestamp`, the revert never fires and the change is executable in the same transaction it was proposed.

The same `_afterTimelock` call is the sole deadline source for every timelocked action — drawdown, decay, watermarks, and the timelock value itself: [4](#0-3) [5](#0-4) [6](#0-5) 

No cap or validation is applied to the `newTimelock` parameter anywhere in the proposal path:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // ← overflow if current timelock already large
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
``` [7](#0-6) 

The `PoolStopLossConfig` and `PoolStopLossSchedule` structs confirm all deadline fields are `uint32`: [8](#0-7) 

---

### Impact Explanation

`OracleValueStopLossExtension` is the sole mechanism protecting LPs from a malicious or compromised pool admin silently weakening or disabling the per-bin value stop-loss. The timelock is explicitly described as the window in which LPs can react and exit. Bypassing it lets the admin:

1. Set `drawdownE6 = 0` (disable stop-loss entirely) with no LP warning window.
2. Set `decayPerSecondE8` to an arbitrarily high value, causing watermarks to decay to zero instantly, again disabling protection.
3. Raise watermarks to `type(uint104).max`, making the floor unreachable and permanently suppressing stop-loss triggers.

Any of these actions can be executed atomically after the one-time cost of waiting out the initial (legitimate) timelock to install the overflowing value. After that, all future proposals bypass the delay. LPs who trusted the published timelock as their exit window lose that guarantee entirely, with direct risk of principal loss through unprotected value-draining swaps.

---

### Likelihood Explanation

The trigger requires the pool admin to first successfully execute a timelock change to a value ≥ 2.54 × 10⁹ s. This requires waiting out the pool's current timelock once — a one-time cost. After that, the overflow is permanent and every subsequent proposal is immediately executable. The pool admin role is semi-trusted by design; the timelock is the only cap on their power over stop-loss parameters. A pool admin who turns malicious (or whose key is compromised) has a clear, low-complexity path to exploit this.

---

### Recommendation

Replace the unsafe cast with a checked addition and an explicit cap:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + uint256(oracleStopLossConfig[pool_].timelock);
    require(result <= type(uint32).max, "Timelock overflow");
    return uint32(result);
}
```

Additionally, add a maximum cap on `newTimelock` in `proposeOracleStopLossTimelock` (e.g., 365 days) so that no value that could cause overflow can ever be stored:

```solidity
uint256 constant MAX_TIMELOCK = 365 days;

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    if (newTimelock > MAX_TIMELOCK) revert TimelockTooLarge(newTimelock);
    ...
}
```

### Proof of Concept

```solidity
// Precondition: pool admin controls a pool with OracleValueStopLossExtension attached.
// Current block.timestamp ≈ 1_750_000_000.
// type(uint32).max = 4_294_967_295.
// Overflow threshold: 4_294_967_295 - 1_750_000_000 = 2_544_967_295 seconds.

// Step 1: Admin proposes timelock = 2_600_000_000 (valid uint32, ~82 years).
//         Current timelock is T (e.g., 2 days). Admin waits T seconds.
extension.proposeOracleStopLossTimelock(pool, 2_600_000_000);
vm.warp(block.timestamp + 2 days);
extension.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock == 2_600_000_000

// Step 2: Admin proposes drawdown = 0 (disable stop-loss).
//         _afterTimelock computes uint32(1_750_000_000 + 2_600_000_000)
//                                = uint32(4_350_000_000)
//                                = 4_350_000_000 - 4_294_967_296 = 55_032_704  ← past timestamp
extension.proposeOracleStopLossDrawdown(pool, 0);

// Step 3: Admin immediately executes — no waiting required.
//         _requireElapsed(55_032_704): block.timestamp (1.75B) < 55_032_704 → false → no revert.
extension.executeOracleStopLossDrawdown(pool);
// drawdownE6 == 0: stop-loss permanently disabled, LPs unprotected.
```
