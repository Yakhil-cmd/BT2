### Title
Unregistered NEP-141 Recipient Causes Permanent ERC-20 Token Loss in `ExitToNear` Precompile — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns a user's ERC-20 tokens in the EVM before confirming that the NEP-141 `ft_transfer` to the recipient NEAR account will succeed. If the recipient NEAR account is not registered with the NEP-141 contract (i.e., has not performed `storage_deposit`), the `ft_transfer` cross-contract call fails. When the `error_refund` compile-time feature is **not** enabled, the `exit_to_near_precompile_callback` issues no refund, permanently destroying the user's ERC-20 tokens with no recovery path.

---

### Finding Description

**Vulnerability class:** Connector/bridge accounting bug → permanent freezing of funds.

The `ExitToNear::run` function in `engine-precompiles/src/native.rs` constructs a `ft_transfer` (or `ft_transfer_call`) promise targeting the NEP-141 contract, with the user-supplied NEAR account ID as the recipient. The ERC-20 tokens are burned atomically inside the EVM as part of the same transaction that schedules this promise. The promise result is handled asynchronously in `exit_to_near_precompile_callback` in `engine/src/contract_methods/connector.rs`.

The callback's refund branch is entirely gated on the `error_refund` compile-time feature:

```rust
// engine-precompiles/src/native.rs
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
```

When `error_refund` is absent, `refund` is always `None`. In the callback:

```rust
// engine/src/contract_methods/connector.rs
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // ft_transfer succeeded
    None
} else if let Some(args) = args.refund {
    // refund branch — never reached when error_refund is disabled
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op on ft_transfer failure
};
```

When `ft_transfer` fails (because the recipient is unregistered with the NEP-141 contract), the callback silently returns `None`. The ERC-20 tokens are already burned; the NEP-141 tokens remain locked in Aurora's account. There is no recovery mechanism.

The test suite explicitly documents this behavior:

```rust
// engine-tests/src/tests/erc20_connector.rs
exit_to_near(
    &ft_owner,
    // The ft_transfer will fail because this account is not registered with the NEP-141
    "unregistered.near",
    FT_EXIT_AMOUNT,
    &erc20,
    &aurora,
).await.unwrap();

#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `error_refund` is not enabled in the production build:

- The user's ERC-20 tokens are **burned** (removed from the EVM state) atomically.
- The NEP-141 `ft_transfer` to the unregistered recipient **fails** on the NEAR side.
- The NEP-141 tokens remain in Aurora's account with no mechanism to return them to the user.
- The user has permanently lost their bridged assets with no recourse.

---

### Likelihood Explanation

**Medium.** Any EVM user who calls `withdrawToNear` on an ERC-20 contract and specifies a NEAR account that has not performed `storage_deposit` on the corresponding NEP-141 contract will trigger this path. This is a realistic scenario because:

1. NEAR's storage-staking model requires explicit `storage_deposit` registration, which is unfamiliar to Ethereum-native users.
2. A user may specify a freshly created NEAR account that has not yet registered with the NEP-141 contract.
3. A user may specify a contract account (e.g., a multisig or DAO) that has not registered.
4. The EVM-side transaction succeeds (ERC-20 burn succeeds), giving the user no on-chain signal that the NEAR-side transfer will fail.

---

### Recommendation

1. **Enable `error_refund` unconditionally in production** so that a failed `ft_transfer` always triggers `refund_on_error`, re-minting the burned ERC-20 tokens to the original sender.
2. Alternatively, **remove the feature gate** and make the refund path unconditional in `ExitToNear::run` and `exit_to_near_precompile_callback`.
3. As a defense-in-depth measure, consider validating recipient registration before burning tokens (though this is difficult cross-contract in a single atomic step).

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora, obtaining ERC-20 tokens at address `erc20`.
3. Call `withdrawToNear(bytes,uint256)` on `erc20` with `dest = "new-unregistered.near"` and `amount = N`.
4. Observe: ERC-20 balance decreases by `N` (tokens burned in EVM). NEP-141 `ft_transfer` to `new-unregistered.near` fails (account not registered). `exit_to_near_precompile_callback` fires with `promise_result(0)` = failure and `args.refund = None`. Callback returns `None` silently.
5. Result: user's `N` ERC-20 tokens are permanently destroyed; NEP-141 tokens remain locked in Aurora's account.

This matches the test at `engine-tests/src/tests/erc20_connector.rs:623–666`, which explicitly confirms that without `error_refund`, `erc20_balance` is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (i.e., the exit amount is lost with no refund). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
