### Title
No Parameter Limits on `TxRateLimitCommand` `set_config` — Allows Disabling Transaction Rate Limiting on Collection Nodes - (File: `admin/commands/collection/tx_rate_limiter.go`)

### Summary

The `TxRateLimitCommand` admin command handler for `set_config` accepts arbitrary `limit` and `burst` values with no lower-bound validation. A node operator (or anyone with access to the admin server) can set `limit=0` and `burst=0`, which silently disables all per-address transaction rate limiting on the Collection Node's ingest engine, or set arbitrarily large values that make the rate limiter meaningless. The `Validator` method is a no-op stub that always returns `nil`.

### Finding Description

`TxRateLimitCommand` is the admin command registered as `"ingest-tx-rate-limit"` on Collection Nodes. Its `set_config` sub-command reads `limit` and `burst` as raw `float64` values from the JSON request and calls `AddressRateLimiter.SetLimitConfig` directly with no bounds checking:

```go
// admin/commands/collection/tx_rate_limiter.go
func (s *TxRateLimitCommand) Validator(req *admin.CommandRequest) error {
    return nil   // ← no validation at all
}

// Handler, set_config branch:
limit, ok := dataLimit.(float64)
// ...
burst, ok := dataBurst.(float64)
// ...
s.limiter.SetLimitConfig(rate.Limit(limit), int(burst))
```

`AddressRateLimiter.SetLimitConfig` also performs no validation:

```go
// engine/collection/ingest/rate_limiter.go
func (r *AddressRateLimiter) SetLimitConfig(limit rate.Limit, burst int) {
    r.mu.Lock()
    defer r.mu.Unlock()
    for address := range r.limiters {
        r.limiters[address] = rate.NewLimiter(limit, burst)
    }
    r.limit = limit
    r.burst = burst
}
```

The `golang.org/x/time/rate` package treats `rate.Limit(0)` as "zero events per second" (i.e., no tokens ever replenish), and `burst=0` means no burst is ever allowed. Together, `limit=0, burst=0` causes `limiter.Allow()` to always return `false`, meaning every transaction from every rate-limited address is permanently blocked. Conversely, setting `limit=math.MaxFloat64` and `burst=math.MaxInt` effectively disables rate limiting entirely.

Contrast this with other updatable configs in the same codebase that do enforce bounds:

- `SetMinSealingLag` / `SetMaxSealingLag` call `validation.ValidateMinMaxSealingLag`
- `SetHalvingInterval` calls `validation.ValidateHalvingInterval` (rejects zero)
- `WithSkipNewProposalsThreshold` enforces a `MinSkipNewProposalsThreshold`

### Impact Explanation

An operator with access to the admin HTTP endpoint (default `localhost:9002`) can issue:

```
curl localhost:9002/admin/run_command -d '{"commandName":"ingest-tx-rate-limit","data":{"command":"set_config","limit":0,"burst":0}}'
```

This permanently blocks all transactions from every address currently in the rate-limited set, or:

```
curl localhost:9002/admin/run_command -d '{"commandName":"ingest-tx-rate-limit","data":{"command":"set_config","limit":1e308,"burst":2147483647}}'
```

This disables rate limiting entirely, allowing spam transactions to flood the Collection Node's mempool unchecked. The impact is disruption of the Collection Node's transaction ingestion behavior — either blocking legitimate transactions or enabling mempool flooding — without any on-chain authorization or protocol-level check.

**Impact: 3** (Collection Node operational disruption; no direct asset loss, but can degrade or halt transaction processing on the node)

### Likelihood Explanation

The admin server binds to `localhost:9002` by default, so exploitation requires local access or a misconfigured network exposure. However, the vulnerability requires no special cryptographic key or staked-node compromise — only network access to the admin port. Any process or user on the same host, or any attacker who has gained shell access, can trigger this.

**Likelihood: 2**

### Recommendation

Add bounds validation in `TxRateLimitCommand.Validator` (or in `AddressRateLimiter.SetLimitConfig`) to reject nonsensical values:

1. Reject `limit <= 0` (zero or negative rate disables the limiter entirely or blocks all traffic).
2. Reject `burst <= 0` (zero burst means no token is ever granted).
3. Optionally enforce a reasonable upper bound to prevent accidental disabling via extremely large values.

Example fix in `Validator`:

```go
func (s *TxRateLimitCommand) Validator(req *admin.CommandRequest) error {
    input, ok := req.Data.(map[string]any)
    if !ok { ... }
    if cmd == "set_config" {
        limit := input["limit"].(float64)
        burst := input["burst"].(float64)
        if limit <= 0 {
            return admin.NewInvalidAdminReqParameterError("limit", "must be > 0", limit)
        }
        if burst <= 0 {
            return admin.NewInvalidAdminReqParameterError("burst", "must be > 0", burst)
        }
    }
    return nil
}
```
