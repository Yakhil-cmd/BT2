### Title
Gateway Admits Invoke Transactions with Invalid Signatures via Unconstrained `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature, intended to allow a user to submit a deploy_account + invoke pair atomically, does not restrict which sender may trigger the skip. Any unprivileged attacker who observes a pending deploy_account for address X in the mempool can submit an invoke with nonce=1 for X carrying an arbitrary (invalid) signature. The gateway admits the transaction without running `__validate__`, so the invalid invoke enters the mempool. When the batcher later executes it, `__validate__` runs, the transaction reverts, the account nonce is permanently incremented to 2, and the legitimate user's invoke with nonce=1 is rendered stale.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip `__validate__`) whenever three conditions hold simultaneously:

1. The incoming transaction is an Invoke with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

Condition 3 is satisfied by the *presence of any transaction for that address in the mempool*, regardless of who submitted it. The check does not verify that the caller of the current invoke is the same party who submitted the deploy_account. [2](#0-1) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point. [3](#0-2) 

`perform_pre_validation_stage` checks nonce ordering and fee bounds but does **not** verify the transaction signature; that is exclusively the responsibility of `__validate__`. [4](#0-3) 

The transaction is then forwarded to the mempool with the nonce returned from state (0), and the mempool's own `validate_tx` only checks for duplicate nonces and fee escalation — no signature check. [5](#0-4) 

At batcher execution time the transaction is run with `execution_flags.validate = true` (the default), so `__validate__` is called, the invalid signature is detected, and the transaction reverts. However, the nonce increment performed in `perform_pre_validation_stage` is **not** rolled back on revert, permanently advancing the account nonce to 2.

### Impact Explanation

An attacker can grief any account that has a pending deploy_account in the mempool:

- The attacker's invalid invoke is admitted to the mempool (gateway admission bypass).
- After execution the account nonce is 2; the legitimate user's invoke with nonce=1 is stale and will be rejected by the mempool.
- The legitimate user must resubmit with nonce=2, paying an additional fee and suffering a delayed deployment.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The attack requires only that the attacker observe a deploy_account transaction in the mempool (publicly visible) and submit a crafted invoke before the deploy_account is committed. No privileged access, no special account class, and no cryptographic capability is required. The window is the entire time the deploy_account sits in the mempool before being included in a block.

### Recommendation

Restrict the skip-validate path so that only the owner of the pending deploy_account can benefit from it. Concretely, `account_tx_in_pool_or_recent_block` should be replaced (or supplemented) with a check that a **deploy_account** transaction specifically exists for the sender address, and that the incoming invoke's transaction hash is signed by the key that will control the deployed account. Alternatively, the skip should be gated on the gateway having already accepted a deploy_account from the same sender in the same request batch (i.e., tie the skip to the deploy_account tx hash, as the `PyValidator` path already does via `deploy_account_tx_hash`). [6](#0-5) 

The `PyValidator` path already threads a `deploy_account_tx_hash` parameter and checks `deploy_account_tx_hash.is_some()` — the gateway path should adopt the same binding rather than relying solely on the mempool presence check.

### Proof of Concept

1. Legitimate user submits `deploy_account` for address X (nonce=0). It enters the mempool.
2. Attacker observes X in the mempool via `account_tx_in_pool_or_recent_block`.
3. Attacker submits `invoke` for address X, nonce=1, with a garbage signature `[0x41, 0x41]`.
4. Gateway evaluates: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → `skip_validate=true`.
5. `run_validate_entry_point` sets `validate=false`; `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`.
6. Invalid invoke is forwarded to the mempool and accepted (no duplicate nonce for X at nonce=1).
7. Batcher executes deploy_account (nonce=0) → X is deployed.
8. Batcher executes attacker's invoke (nonce=1) → `__validate__` runs → signature invalid → transaction reverts; nonce of X advances to 2.
9. Legitimate user's invoke with nonce=1 is now rejected as `NonceTooOld`. [7](#0-6) [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
