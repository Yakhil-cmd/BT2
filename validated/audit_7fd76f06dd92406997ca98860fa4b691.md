### Title
Gateway `skip_stateful_validations` admits invoke transactions with invalid signatures, enabling fee-escalation displacement of legitimate pending invokes — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point (the only place where the account contract verifies the cryptographic signature) for any Invoke transaction with `nonce == 1` when the sender account has `account_nonce == 0` and any transaction for that address is already in the mempool. Because the mempool's own `validate_tx` path checks only nonce ordering and fee escalation — never the signature — an attacker can submit an Invoke with a completely forged signature that passes every gateway check and is admitted to the mempool. With fee escalation enabled in production (`enable_fee_escalation: true`, `fee_escalation_percentage: 10`), the attacker can simultaneously evict the victim's legitimate pending Invoke by offering a marginally higher fee.

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

When all three conditions hold — the transaction is `Invoke`, `tx.nonce() == 1`, `account_nonce == 0`, and `account_tx_in_pool_or_recent_block` returns `true` — the function returns `true` (skip).

**How the skip propagates**

`run_pre_validation_checks` returns the `skip_validate` boolean: [2](#0-1) 

`run_validate_entry_point` then sets `validate: !skip_validate`: [3](#0-2) 

When `validate == false`, `AccountTransaction::validate_tx` returns `Ok(None)` immediately without calling the account's `__validate__` entry point: [4](#0-3) 

**The mempool does not check signatures**

`validate_by_mempool` (called before `skip_stateful_validations`) only checks nonce ordering and fee escalation: [5](#0-4) 

`validate_incoming_tx` checks only that `tx_nonce >= account_nonce`: [6](#0-5) 

**Fee escalation replaces the victim's transaction**

Production config has `enable_fee_escalation: true` and `fee_escalation_percentage: 10`: [7](#0-6) 

`validate_fee_escalation` allows a same-nonce replacement when the incoming tip and max-L2-gas-price each exceed the existing transaction's values by the configured percentage: [8](#0-7) 

**`account_tx_in_pool_or_recent_block` is not deploy-account-specific**

The check returns `true` for any account that has *any* transaction in the pool or in a recently committed block — not specifically a `DeployAccount`: [9](#0-8) 

The comment in `skip_stateful_validations` acknowledges this: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."* This broadens the attack surface beyond the deploy-account UX scenario.

### Impact Explanation

An attacker who observes a victim's `DeployAccount` transaction in the mempool can:

1. Craft an `Invoke` with `nonce=1`, the victim's sender address, an **arbitrary/invalid signature**, and a tip/gas-price ≥ 110 % of the victim's pending Invoke.
2. Submit it to the gateway. `skip_stateful_validations` returns `true` → `__validate__` is never called → the forged signature is never checked.
3. The mempool's fee-escalation logic accepts the replacement and **evicts the victim's legitimate Invoke**.
4. When the batcher builds the block, `DeployAccount` executes (account nonce advances to 1), then the attacker's Invoke is attempted. The blockifier now calls `__validate__` (the account contract exists), the forged signature fails, and the transaction is rejected.
5. The victim's account is left at nonce 1 with no pending Invoke; the victim must resubmit.

The attacker can repeat this indefinitely, causing a persistent DoS against any account in the deploy-account + invoke UX flow. No funds are required beyond the gas to submit the replacement transaction (which itself is rejected at execution time, so no fee is charged from the victim).

Matching impact: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- The attack window is the period between a `DeployAccount` entering the mempool and being committed to a block (typically seconds to minutes).
- The attacker needs only to monitor the public mempool/gateway for `DeployAccount` transactions and submit a replacement Invoke with a marginally higher fee.
- No privileged access, no special contract, no cryptographic capability is required.
- Fee escalation is enabled by default in production.

### Recommendation

1. **Verify the deploy-account type specifically.** Instead of checking `account_tx_in_pool_or_recent_block` (which is true for any account with any pending transaction), query the mempool for a pending `DeployAccount` transaction for the sender address. This closes the broader surface noted in the comment.

2. **Bind the skip to the deploy-account transaction hash.** Require the caller to supply the hash of the pending `DeployAccount` (as `PyValidator::should_run_stateful_validations` already does via `deploy_account_tx_hash`), and verify it exists in the mempool before skipping validation.

3. **Use `max_nonce_for_validation_skip` from config.** The `StatefulTransactionValidatorConfig` already defines `max_nonce_for_validation_skip` (default `Nonce(Felt::ONE)`), but `skip_stateful_validations` hardcodes the nonce-1 check and ignores this field. Align the gateway path with the `PyValidator` path.

### Proof of Concept

```
1. Victim broadcasts:
     DeployAccount { sender: 0xVICTIM, nonce: 0, ... }   → mempool accepts it
     Invoke        { sender: 0xVICTIM, nonce: 1, sig: VALID_SIG, tip: 100, max_l2_gas: 100 }
       → skip_stateful_validations returns true (deploy_account in pool)
       → __validate__ skipped → mempool accepts it

2. Attacker observes DeployAccount in mempool, then broadcasts:
     Invoke { sender: 0xVICTIM, nonce: 1, sig: [0xDEAD, 0xBEEF], tip: 111, max_l2_gas: 111 }
       → skip_stateful_validations returns true (same condition)
       → __validate__ skipped → forged signature never checked
       → validate_fee_escalation: 111 >= 100 * 1.10 → replacement accepted
       → victim's Invoke evicted from mempool

3. Block N is built:
     DeployAccount executes → account deployed, nonce = 1
     Attacker's Invoke attempted → __validate__ called → forged sig fails → REJECTED

4. Victim's account: nonce = 1, no pending Invoke.
   Attacker repeats from step 2 whenever victim resubmits.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }
