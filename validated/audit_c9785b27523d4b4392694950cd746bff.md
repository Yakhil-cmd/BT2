### Title
Attacker Bypasses `__validate__` Signature Check for Invoke Transactions via Overly Broad `skip_stateful_validations` Mempool Presence Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway is designed to allow the deploy-account + invoke UX by skipping the `__validate__` entry point for an invoke with nonce=1 when the account is undeployed and a deploy-account is pending. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the mempool for that address, not exclusively a `deploy_account`. Once a victim submits their `deploy_account`, an attacker can submit a forged invoke (nonce=1, arbitrary invalid signature, higher fee) for the victim's address. The gateway skips `__validate__`, accepts the forged transaction, and the mempool's fee-escalation logic can then evict the victim's legitimate invoke.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain). [1](#0-0) 

When all three hold, the function calls `account_tx_in_pool_or_recent_block` and returns its result as the `skip_validate` flag. [2](#0-1) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`state.contains_account` returns `true` if the address appears in either the `staged` or `committed` nonce maps — i.e., for **any** transaction type that was ever accepted for that address. [4](#0-3) 

The code comment claims this is sufficient because the address "either has a deploy_account transaction or transactions with future nonces that passed validations." That reasoning is circular: the only way a future-nonce invoke for an undeployed account passes gateway validation is if `account_tx_in_pool_or_recent_block` already returned `true` — which requires the victim's own `deploy_account` to be present. Once the victim's `deploy_account` is in the mempool, the invariant "presence implies deploy_account pending" holds for the victim's own subsequent invokes, but it also holds for **any attacker-crafted invoke** targeting the same address.

The `skip_validate` flag is then forwarded to `run_validate_entry_point`, which sets `execution_flags.validate = !skip_validate`: [5](#0-4) 

When `validate = false`, `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling the `__validate__` entry point: [6](#0-5) 

The `validate_by_mempool` call that precedes the skip check only performs a read-only nonce/fee-escalation check; it does not verify the signature: [7](#0-6) 

**Attack steps:**

1. Victim submits `deploy_account` for address `V`. It is accepted into the mempool.
2. `account_tx_in_pool_or_recent_block(V)` now returns `true` (the deploy_account is in `tx_pool`).
3. Attacker crafts an `Invoke` with `sender_address = V`, `nonce = 1`, an arbitrary invalid signature, and a fee higher than the victim's expected invoke.
4. Gateway stateless checks pass (valid address format, resource bounds, DA modes).
5. `validate_nonce` passes: `0 ≤ 1 ≤ max_allowed_nonce_gap`.
6. `validate_by_mempool` passes: no duplicate hash, nonce not too old.
7. `skip_stateful_validations` returns `true` → `run_validate_entry_point` is skipped entirely.
8. The forged invoke is forwarded to `mempool_client.add_tx`.
9. The mempool's fee-escalation logic evicts the victim's legitimate invoke (lower fee) and inserts the forged one.
10. At execution time the blockifier calls `__validate__` on the forged invoke; it fails (invalid signature or wrong return value); the transaction is reverted.
11. The victim's legitimate invoke has been permanently evicted from the mempool.

### Impact Explanation

The gateway accepts an invalid transaction — an invoke with an arbitrary, unverified signature — before sequencing. This satisfies the **High** impact criterion: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing." The concrete harm is that the attacker can use the mempool's fee-escalation replacement to evict the victim's legitimate nonce-1 invoke, forcing the victim to resubmit after their account is already deployed.

### Likelihood Explanation

The attack requires the attacker to observe a `deploy_account` transaction in the mempool (publicly visible) and race to submit a forged invoke before the victim's own invoke is processed. The window is the time between the victim's `deploy_account` being accepted and the victim's invoke being accepted. In the common UX pattern of submitting both transactions together, this window is narrow but non-zero due to network propagation and processing latency. The attacker pays a fee for the reverted forged invoke, making the attack economically costly but not impossible for a motivated adversary.

### Recommendation

`skip_stateful_validations` must verify that the pending transaction for the account is specifically a `DeployAccount` transaction, not merely that any transaction exists for that address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type stored in `tx_pool` for the given address. The current `account_tx_in_pool_or_recent_block` check is too coarse for a security-sensitive bypass of signature verification.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
