### Title
Pool Admin Bypasses Spread-Fee Cap via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

The factory enforces `maxAdminSpreadFeeE6` on `setPoolAdminFees` but applies **no equivalent cap** to `setPoolBinAdditionalFees`. A pool admin can set per-bin `addFeeBuyE6` / `addFeeSellE6` to any value up to `uint16` max (65,535 in E6 = 6.5535%) without restriction. During every swap through the affected bin, that uncapped additional fee is added on top of the already-capped base spread fee, causing traders to pay more than the documented hard ceiling and resulting in direct loss of trader funds.

### Finding Description

**Root cause — two bounds, one enforced:**

The factory defines a hard ceiling for spread fees:

```solidity
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000; // 20%
``` [1](#0-0) 

`setPoolAdminFees` correctly validates against `maxAdminSpreadFeeE6` (which is itself bounded by `HARD_MAX_SPREAD_FEE_E6`):

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [2](#0-1) 

But `setPoolBinAdditionalFees` — also callable by the pool admin — passes the values straight through with **no cap check**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool-level `setBinAdditionalFees` only validates the bin index, not the fee magnitude:

```solidity
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [4](#0-3) 

**How the additional fee reaches swap math:**

During every swap the bin additional fee is added directly on top of the oracle-derived base fee before computing the trader's cost:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [5](#0-4) 

The same pattern appears in the sell direction and in `getSellAndBuyPrices`:

```solidity
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [6](#0-5) 

**The two-bound gap (analog to the external report):**

| Bound | Value | Enforced on |
|---|---|---|
| `maxAdminSpreadFeeE6` ≤ `HARD_MAX_SPREAD_FEE_E6` | ≤ 200,000 (20%) | `setPoolAdminFees` |
| `uint16` max for `addFeeBuyE6`/`addFeeSellE6` | 65,535 (6.5535%) | **nowhere** |

The `BinState` struct stores both fields as `uint16`:

```solidity
uint16 addFeeBuyE6;   // Additional fee for buying token0; 1e6 = 100%
uint16 addFeeSellE6;  // Additional fee for buying token1; 1e6 = 100%
``` [7](#0-6) 

The same gap exists at pool creation time: `_unpackAndValidateBinStates` validates only bin distances, never the `buyFee`/`sellFee` fields unpacked from the packed bin arrays:

```solidity
(uint16 length, uint16 buyFee, uint16 sellFee) = binData.unpack();
// length validated; buyFee and sellFee stored without any cap check
nonNegativeBinStates[k] = BinState({..., addFeeBuyE6: buyFee, addFeeSellE6: sellFee});
``` [8](#0-7) 

### Impact Explanation

A pool admin (semi-trusted, bounded by caps) can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65_535` (6.5535%) on the active bin. Every trader who swaps through that bin pays a total effective spread fee of `spreadFeeE6 + 6.5535%`, which can reach up to ~46.5% (20% protocol + 20% admin + 6.5535% bin additional), far exceeding the documented 20% hard cap per component. The excess fee is extracted from traders as direct principal loss with no recourse. The pool admin can also set this at pool creation time via the packed bin arrays, making it invisible to traders who only inspect `spreadFeeE6`.

**Impact category:** Admin-boundary break — pool admin exceeds the documented fee cap through an uncapped path, causing direct loss of trader funds above Sherlock thresholds.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly scoped to operate only within the fee caps. The `setPoolBinAdditionalFees` function is a normal operational function (not a privileged emergency path), callable at any time by the pool admin without a timelock. The bypass requires only a single transaction. Any pool whose admin is malicious or compromised is immediately exploitable.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and symmetrically in `_unpackAndValidateBinStates` at creation time) bounding `addFeeBuyE6` and `addFeeSellE6` to `maxAdminSpreadFeeE6`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Apply the same guard inside `_unpackAndValidateBinStates` when iterating over packed bin arrays at `createPool` time.

### Proof of Concept

```solidity
// Pool admin sets bin 0 additional buy fee to uint16 max (6.5535%)
vm.prank(admin);
factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535);
// No revert — no cap check exists

// Trader swaps token1 → token0 through bin 0
// Effective fee = spreadFeeE6 (e.g. 20%) + 6.5535% = 26.5535%
// Trader pays 6.5535% more than the documented hard cap allows
// The excess is captured by the pool (LP fee bucket), not refunded
```

The `setPoolBinAdditionalFees` call succeeds unconditionally for any `uint16` value, and the swap math applies the full uncapped additional fee to every trade through the affected bin. [3](#0-2) [9](#0-8) [10](#0-9)
