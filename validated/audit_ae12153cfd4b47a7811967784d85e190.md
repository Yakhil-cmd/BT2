### Title
`Declare` Transactions Bypass `max_l2_gas_amount` Admission Check in `StatelessTransactionValidator` - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `max_l2_gas_amount` upper-bound check for `RpcTransaction::Declare` transactions. An unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount` set to an arbitrarily large value (up to `u64::MAX`), bypassing the gateway's admission control that is correctly enforced for `Invoke` and `DeployAccount` transactions.

---

### Finding Description

In `validate_resource_bounds`, the `max_l2_gas_amount` ceiling check is guarded by a type match that silently skips `Declare` transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1_210_000_000`. [2](#0-1) 

For `Invoke` and `DeployAccount` transactions, any transaction with `l2_gas.max_amount > 1_210_000_000` is rejected at the gateway stateless validation stage. For `Declare` transactions, this check is entirely absent. The full `validate` call sequence is:

```
validate_contract_address  ✓ (all types)
validate_empty_account_deployment_data  ✓ (all types)
validate_empty_paymaster_data  ✓ (all types)
validate_resource_bounds  ✓ (all types) — but max_l2_gas_amount skipped for Declare
validate_tx_size  ✓ (all types)
validate_nonce_data_availability_mode  ✓ (all types)
validate_fee_data_availability_mode  ✓ (all types)
``` [3](#0-2) 

After passing stateless validation, the transaction proceeds to `extract_state_nonce_and_run_validations`, which calls `validate_state_preconditions` → `validate_resource_bounds` (stateful). The stateful resource-bounds check only validates the L2 gas *price* against the previous block's price, not the gas *amount*: [4](#0-3) 

The `validate_tx_l2_gas_price_within_threshold` function only inspects `max_price_per_unit`, never `max_amount`: [5](#0-4) 

Therefore a `Declare` transaction with `l2_gas.max_amount = u64::MAX` and a valid gas price clears both the stateless and stateful gateway checks and is forwarded to the mempool via `add_tx`. [6](#0-5) 

The existing test suite confirms the gap: `test_invalid_max_l2_gas_amount` only exercises `TransactionType::DeployAccount` and `TransactionType::Invoke`, with `Declare` explicitly absent from the parameterized cases. [7](#0-6) 

---

### Impact Explanation

The gateway's admission invariant — "no transaction with `l2_gas.max_amount > max_l2_gas_amount` enters the mempool" — is broken for `Declare` transactions. An attacker can:

1. Craft a syntactically valid `Declare` transaction (valid Sierra class, valid `compiled_class_hash`) with `l2_gas.max_amount = u64::MAX`.
2. Submit it to the gateway; it passes all stateless and stateful checks.
3. The transaction is admitted to the mempool.

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."* The mempool now holds a transaction that violates the operator-configured admission policy, consuming a mempool slot and potentially interfering with gas-price-threshold bookkeeping that the mempool maintains per transaction reference.

---

### Likelihood Explanation

The trigger requires no privilege. Any user who can submit a `Declare` transaction (subject only to the `block_declare` flag and the authorized-declarer allowlist, both of which are operator-controlled and may be open) can reproduce this with a single crafted RPC call. The TODO comment in the source confirms the gap is known but unmitigated.

---

### Recommendation

Remove the `Declare` exception in `validate_resource_bounds` and apply the `max_l2_gas_amount` ceiling uniformly to all transaction types. If a different (higher) ceiling is appropriate for `Declare` transactions, introduce a dedicated `max_l2_gas_amount_declare` config field and enforce it explicitly rather than leaving the check absent.

```rust
// Remove the Declare exception:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Add a corresponding test case for `TransactionType::Declare` in `test_invalid_max_l2_gas_amount`.

---

### Proof of Concept

Submit the following to the gateway's `add_transaction` endpoint (all other fields valid):

```json
{
  "type": "DECLARE",
  "resource_bounds": {
    "l2_gas": {
      "max_amount": "0xffffffffffffffff",
      "max_price_per_unit": "<valid_price_above_min_gas_price>"
    },
    ...
  },
  "contract_class": { ... },
  "compiled_class_hash": "<matching_hash>",
  ...
}
```

Expected (broken) result: gateway returns `200 OK` with a transaction hash and the transaction appears in the mempool.

Expected (correct) result: gateway returns `MaxGasAmountTooHigh` error, identical to what `Invoke`/`DeployAccount` would receive for the same `max_amount`.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
