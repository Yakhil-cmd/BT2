### Title
Circular `skip_stateful_validations` Guard Allows Indefinite Admission of Unvalidated Invoke Transactions for Non-Existent Accounts - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to allow a single invoke with nonce=1 to bypass the `__validate__` entry point when a `deploy_account` is pending in the mempool (UX feature). However, the guard it uses — `account_tx_in_pool_or_recent_block` — checks for **any** account transaction in the pool, not specifically a `deploy_account`. Once the deploy_account is evicted or expires, the invoke with nonce=1 that was already admitted remains in the pool and itself satisfies the guard. This allows an attacker to perpetually refresh unvalidated invoke transactions for a non-existent account via fee escalation, bypassing `__validate__` (signature verification) indefinitely.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when an incoming invoke has `tx.nonce() == 1` and `account_nonce == 0`: [1](#0-0) 

The guard delegates to `account_tx_in_pool_or_recent_block`: [2](#0-1) 

Which checks `MempoolState::contains_account` (staged or committed) **or** `tx_pool.contains_account`: [3](#0-2) 

The code comment asserts this is safe because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is circular: the invoke with nonce=1 that was admitted via the skip itself satisfies `tx_pool.contains_account`, so after the deploy_account is evicted or TTL-expired, the invoke remains in the pool and bootstraps the guard for every subsequent fee-escalation replacement.

The mempool's TTL expiry removes transactions when a new transaction triggers cleanup: [4](#0-3) 

And capacity overflow eviction targets gap accounts: [5](#0-4) 

Neither eviction path removes the invoke with nonce=1 from the pool (it is not a gap account — it is the only transaction for that address). So after the deploy_account is gone, `tx_pool.contains_account(address)` remains `true` due to the invoke itself.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [6](#0-5) 

This means `validate_tx` is never called, so the account's `__validate__` entry point (which performs signature verification) is never executed at gateway admission time.

### Impact Explanation

The gateway accepts invoke transactions for a non-existent account address, bypassing the `__validate__` entry point (signature check). The attacker can submit invokes with arbitrary calldata and an arbitrary (invalid) signature for an address that has no deployed contract. These transactions are admitted to the mempool and forwarded to the batcher. During batcher execution, `get_class_hash_at` returns `ClassHash::default()` (zero) for the non-existent account, causing the `__validate__` call to fail and the transaction to be rejected. The nonce increment from `perform_pre_validation_stage` is rolled back via `TransactionalState`. No funds are lost and no state is corrupted, but the batcher wastes execution resources on transactions that will always fail, and the mempool is polluted with permanently invalid transactions. This matches the **High** impact scope: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

### Likelihood Explanation

The attacker needs only to:
1. Submit one valid `deploy_account` to bootstrap the mechanism.
2. Wait for the deploy_account to be evicted (TTL default is configurable; capacity overflow is attacker-triggerable by flooding the mempool with other transactions).
3. Thereafter, use fee escalation to keep one invoke with nonce=1 alive in the pool indefinitely.

This is fully unprivileged and requires no special access. The deploy_account does not need to succeed or even be executed.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type of the pooled entry at nonce=0 for the given address. Alternatively, the gateway can check the transaction type of the pooled transaction at `(address, nonce=0)` before granting the skip.

### Proof of Concept

1. Attacker deterministically computes address `A` from `(class_hash, salt, constructor_calldata)`.
2. Attacker submits `deploy_account` for `A` → enters mempool; `tx_pool.contains_account(A) = true`.
3. Attacker submits `invoke(nonce=1, sender=A, signature=GARBAGE)` → `skip_stateful_validations` queries `account_tx_in_pool_or_recent_block(A)` → `true` (deploy_account in pool) → `skip_validate = true` → `__validate__` skipped → invoke admitted.
4. Attacker floods the mempool with large transactions from other addresses, triggering capacity overflow eviction of the deploy_account (which is a gap account at nonce=0 relative to the invoke at nonce=1).
5. After eviction: `tx_pool.contains_account(A) = true` (the invoke at nonce=1 is still in the pool); `state.contains_account(A) = false` (no committed block for A).
6. Attacker submits `invoke(nonce=1, sender=A, signature=GARBAGE, tip=higher)` → fee escalation → `account_tx_in_pool_or_recent_block(A)` → `true` (the previous invoke is in the pool) → `skip_validate = true` → `__validate__` skipped → new invoke admitted, replaces old one.
7. Steps 5–6 repeat indefinitely. The attacker maintains a permanently unvalidated invoke in the mempool for a non-existent account, bypassing signature verification at the gateway.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
