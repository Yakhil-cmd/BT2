### Title
Echo Broadcast Threshold Returns Zero for `n ≤ 3`, Enabling Equivocation in DKG/Reshare with Minimum Participant Count — (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

`echo_ready_thresholds` in `src/protocol/echo_broadcast.rs` unconditionally returns `(0, 0)` for any participant count `n ≤ 3`. This means the echo broadcast provides **zero equivocation protection** for the minimum valid DKG configuration (n=3, threshold=2). A malicious participant can send different data to different honest parties across any broadcast round, causing honest parties to compute divergent session IDs, commitment hashes, or public keys — corrupting or permanently denying the DKG/reshare output.

---

### Finding Description

`echo_ready_thresholds` computes the echo and ready thresholds for the BFT reliable broadcast:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // ← hardcoded zero for n=3
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

When both thresholds are `0`, the delivery condition inside `reliable_broadcast_receive_all` becomes `count > 0`, meaning **a single Ready message from any participant immediately delivers**:

- Echo-to-Ready transition: `count > echo_t` → `count > 0` (line 225)
- Ready amplification: `count > ready_t` → `count > 0` (line 282)
- Delivery: `count > 2 * ready_t` → `count > 0` (line 295)

The DKG protocol (`do_keyshare`) calls `do_broadcast` — which internally calls `reliable_broadcast_receive_all` — for three critical values:

1. **Session IDs** (Round 1, line 362 of `src/dkg.rs`)
2. **Commitment hashes** (Round 2.8)
3. **Commitments** (Round 3.2)

With `n=3` and thresholds `(0,0)`, a malicious participant `M` can equivocate in any of these broadcasts: send `data_M1` to honest party `A` and `data_M2` to honest party `B`. Because the delivery threshold is `0`, each honest party delivers the value it received first (from its own simulated echo), which differs between `A` and `B`. The `finish_echo` flag is set after the first echo is processed, so subsequent echoes carrying the other value are silently dropped.

The public API `assert_key_invariants` permits `n=3, threshold=2` without any restriction:

```rust
if participants.len() < 2 { ... }   // allows n=3
if threshold < 2 { ... }            // allows threshold=2
```

There is no enforcement of the BFT broadcast requirement `n ≥ 3f + 1` (where `f = threshold − 1 = 1` malicious party), which demands `n ≥ 4`. The mismatch is silent and undetected by callers.

---

### Impact Explanation

**High — Corruption of DKG/reshare outputs so honest parties accept inconsistent public keys or transcripts.**

If `M` equivocates in the session ID broadcast (Round 1), honest parties `A` and `B` compute different `session_id` values:

- `A`: `session_id = H(sid_A, sid_B, sid_M1)`
- `B`: `session_id = H(sid_A, sid_B, sid_M2)`

All subsequent hash computations (proof-of-knowledge challenges, commitment hashes) diverge. Honest parties will reject each other's proofs or commitments, causing the DKG to abort for all honest parties — **permanent denial of key generation** for a valid 2-of-3 configuration.

Alternatively, if `M` equivocates in the commitment broadcast (Round 3), `A` and `B` accept different commitments for `M`, compute different aggregate public keys, and the DKG produces **inconsistent public keys** across honest parties.

---

### Likelihood Explanation

`n=3, threshold=2` (2-of-3) is the smallest and most commonly deployed threshold configuration. The attack requires only that one of the three participants sends different byte strings to the two honest parties in any broadcast round — a trivial network-layer manipulation requiring no cryptographic capability. No special setup or leaked material is needed.

---

### Recommendation

Enforce the BFT broadcast requirement at the API boundary. In `assert_key_invariants` (and `assert_reshare_keys_invariants`), add:

```rust
// BFT broadcast requires n >= 3*(threshold-1) + 1
let min_bft_participants = 3 * (threshold - 1) + 1;
if participants.len() < min_bft_participants {
    return Err(InitializationError::NotEnoughParticipants { ... });
}
```

Alternatively, update `echo_ready_thresholds` to compute correct non-zero thresholds for `n=3` when malicious parties are possible, or document and enforce that `n ≤ 3` configurations assume all parties are honest.

---

### Proof of Concept

1. Caller invokes DKG with participants `[A, B, M]`, `threshold=2`. This passes all checks in `assert_key_invariants`.
2. `do_keyshare` calls `do_broadcast` for session IDs. `echo_ready_thresholds(3)` returns `(0, 0)`.
3. `M` sends `sid_M1 = [0x01; 32]` to `A` and `sid_M2 = [0x02; 32]` to `B` in the Send phase.
4. `A` processes its own simulated echo of `sid_M1` first: `count(sid_M1) = 1 > 0` → sends `Ready(sid_M1)`, sets `finish_echo = true`. Subsequent echo of `sid_M2` from `B` is dropped.
5. `B` symmetrically delivers `sid_M2`.
6. `A` computes `session_id_A = H(..., sid_M1)`, `B` computes `session_id_B = H(..., sid_M2)`.
7. In Round 2, `A` and `B` compute different proof-of-knowledge challenges. Each rejects the other's proof in Round 4 (`verify_proof_of_knowledge` fails), causing both honest parties to abort — permanent denial of key generation for a valid protocol invocation.
