### Title
Single-Step Reward Address Change With No Zero-Address Validation Permanently Freezes or Misdirects Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
Both `Staking.change_reward_address` and `Pool.change_reward_address` perform a single-step, immediately-effective update of the address to which all future (and pending) rewards are sent. There is no zero-address guard and no two-step pending/confirm pattern. A staker or pool member who passes a wrong address — including the zero address — will have their unclaimed yield permanently misdirected or frozen, with no protocol-level recovery path for already-transferred funds.

### Finding Description
`change_reward_address` in `src/staking/staking.cairo` performs only one validation before overwriting `staker_info.reward_address`:

```cairo
// src/staking/staking.cairo  lines 517-540
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;

    staker_info.reward_address = reward_address;   // ← immediate, single-step
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
```

The only guard is `REWARD_ADDRESS_IS_TOKEN`; there is no `is_non_zero()` check on `reward_address`. The identical pattern exists in `src/pool/pool.cairo` lines 505-526 for pool members.

Once the address is overwritten, every downstream reward-transfer path reads it directly:

```cairo
// src/staking/staking.cairo  lines 1620-1625
let reward_address = staker_info.reward_address;
...
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

This path is exercised by both `claim_rewards` (line 411) and `unstake_action` (line 495), meaning a wrong reward address blocks the staker from completing an unstake as well.

The pool contract mirrors this in `claim_rewards` (pool.cairo line 366):
```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

### Impact Explanation
**Scenario A – zero address:** If a staker or pool member calls `change_reward_address(0)`, the Starknet OpenZeppelin ERC20 `transfer` to the zero address reverts. Every subsequent call to `claim_rewards` and (for stakers) `unstake_action` will revert, **temporarily freezing all unclaimed yield and the staker's principal** until the address is corrected. Because `unstake_action` is the only way to recover staked principal, a staker who also has unclaimed rewards is effectively locked out of their funds for the duration of the mistake.

**Scenario B – wrong non-zero address:** If the staker passes a valid but uncontrolled address (e.g., a typo), any `claim_rewards` call — which can be triggered by *anyone* for any staker — immediately and **permanently transfers unclaimed yield to the wrong address**. The staker can subsequently call `change_reward_address` again to redirect future rewards, but the already-transferred amount is irrecoverable.

Both scenarios fall within the allowed impact scope:
- *High: Permanent freezing of unclaimed yield* (Scenario A)
- *High: Theft of unclaimed yield* (Scenario B)

### Likelihood Explanation
Low. The root cause requires the staker or pool member to supply an incorrect address. This mirrors the external report's own likelihood assessment ("requires an error on the admin side"). The risk is elevated by the fact that `claim_rewards` is callable by any address on behalf of any staker, meaning the window between a mistaken `change_reward_address` call and the rewards being dispatched to the wrong address can be very short.

### Recommendation
1. **Add a zero-address guard** in both `change_reward_address` implementations:
   ```cairo
   assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
   ```
2. **Adopt a two-step pattern**: store a `pending_reward_address` on the first call; require a second transaction from the *new* address (or the staker) to confirm, before the live `reward_address` is updated. This eliminates the risk of a typo causing immediate, irreversible fund loss.

### Proof of Concept
1. Staker Alice stakes and accumulates unclaimed rewards.
2. Alice calls `staking.change_reward_address(0x0)` — passes the only guard (`REWARD_ADDRESS_IS_TOKEN`) since `0x0` is not a token address.
3. `staker_info.reward_address` is immediately set to `0x0`.
4. Bob (any address) calls `staking.claim_rewards(alice_address)`.
5. `send_rewards_to_staker` attempts `checked_transfer(recipient: 0x0, ...)` → reverts.
6. Alice cannot claim rewards and cannot execute `unstake_action` (which also calls `send_rewards_to_staker`) until she corrects the address.

For Scenario B, replace step 2 with `change_reward_address(attacker_address)`. Step 4 then succeeds and all of Alice's unclaimed STRK is permanently transferred to the attacker. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        fn unstake_action(ref self: ContractState, staker_address: ContractAddress) -> Amount {
            // Prerequisites and asserts.
            self.general_prerequisites();
            let mut staker_info = self.internal_staker_info(:staker_address);
            let unstake_time = staker_info
                .unstake_time
                .expect_with_err(Error::MISSING_UNSTAKE_INTENT);
            assert!(Time::now() >= unstake_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Send rewards to staker's reward address.
            // It must be part of this function's flow because staker_info is about to be erased.
            let token_dispatcher = strk_token_dispatcher();
            self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
            // Update staker info to storage (it will be erased later).
            // This is done here to avoid re-entrancy.
            self.write_staker_info(:staker_address, :staker_info);

            let staker_amount = self.get_own_balance(:staker_address).to_strk_native_amount();
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            self.remove_staker(:staker_address, :staker_info, :staker_pool_info);

            // Return stake to staker.
            token_dispatcher
                .checked_transfer(recipient: staker_address, amount: staker_amount.into());
            // Return delegated stake to pools and zero their balances.
            self
                .transfer_to_pools_when_unstake(
                    :staker_address, staker_pool_info: staker_pool_info.as_non_mut(),
                );
            // Clear staker pools.
            staker_pool_info.pools.clear();
            staker_amount
        }

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
