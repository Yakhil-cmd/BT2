### Title
Missing Silo Whitelist Check in `call` Function Allows Whitelist Bypass in Silo Mode - (File: engine/src/contract_methods/evm_transactions.rs)

### Summary

The `call` function in `engine/src/contract_methods/evm_transactions.rs` is missing the silo whitelist validation (`assert_access`) that the `submit` function enforces. In silo mode with whitelists enabled, any unprivileged NEAR account can invoke `call` to interact with EVM contracts, bypassing the `Account` and `Address` whitelist restrictions entirely.

### Finding Description

Aurora Engine's silo mode provides a whitelist system with four kinds: `Admin`, `EvmAdmin`, `Account`, and `Address`. When enabled, the `Account` and `Address` whitelists gate who may submit EVM transactions.

The `submit` (and `submit_with_args`) entry point enforces this via `assert_access`:

```
submit → engine::submit → submit_transaction → assert_access
```

`assert_access` calls `silo::is_allow_submit` (for calls) or `silo::is_allow_deploy` (for deploys), which checks both the NEAR predecessor account and the derived EVM address against their respective whitelists. [1](#0-0) 

The `call` function, however, goes directly to `engine.call_with_args` without invoking `assert_access`: [2](#0-1) 

The only guard present is `require_running`, which checks the global pause flag — not the silo whitelist: [3](#0-2) 

The `call` NEAR entry point is fully public: [4](#0-3) 

The silo whitelist helpers that `assert_access` uses: [5](#0-4) 

### Impact Explanation

In silo mode with `Account` and/or `Address` whitelists enabled, the operator's intent is to restrict EVM interaction to approved NEAR accounts and EVM addresses. The `call` bypass allows any NEAR account — regardless of whitelist status — to call arbitrary EVM contracts. If those contracts hold ETH or ERC-20 token balances, an attacker can invoke value-transferring functions (e.g., ERC-20 `transfer`, direct ETH sends) from a non-whitelisted identity, leading to unauthorized fund movement. This constitutes a **High** impact: theft of funds held in EVM contracts or temporary freezing of the whitelist protection mechanism.

### Likelihood Explanation

The `call` method is a standard public NEAR contract method. Any NEAR account can call it with no special privileges. The only prerequisite is that the engine is not globally paused. In any silo deployment that relies on the whitelist to enforce access control, this bypass is trivially reachable by any external NEAR account.

### Recommendation

Add the silo whitelist check to the `call` function, mirroring the `assert_access` call used in `submit_transaction`. Specifically, after parsing `CallArgs`, derive the normalized transaction representation and call `assert_access` (or an equivalent inline check using `silo::is_allow_submit` / `silo::is_allow_deploy`) before proceeding to `engine.call_with_args`.

### Proof of Concept

1. Deploy Aurora Engine in silo mode.
2. Enable the `Account` and `Address` whitelists via `set_whitelist_status`.
3. Do **not** add the attacker's NEAR account or EVM address to any whitelist.
4. Confirm that calling `submit` with a signed Ethereum transaction from the attacker's account fails with `NotAllowed`.
5. Call the `call` NEAR method directly with a `CallArgs` payload targeting an EVM contract that holds tokens.
6. Observe that the call succeeds and the EVM state is mutated — the whitelist was not checked. [2](#0-1) [1](#0-0) [6](#0-5)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

#[named]
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}

/// Check if a user has the right to submit transactions.
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}

/// Check if a user has the right to receive erc20 tokens.
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}

fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Admin);
    !list.is_enabled() || list.is_exist(account_id)
}

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::EvmAdmin);

}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {

}
