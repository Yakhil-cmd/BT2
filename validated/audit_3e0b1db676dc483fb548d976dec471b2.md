Audit Report

## Title
Unconditional Neuron ID Removal and Infallible `Ok(())` in `AccountState::transfer` Permanently Orphans Failed-Transfer Neurons — (`rs/nns/gtc/src/lib.rs`)

## Summary

`AccountState::transfer` removes each neuron ID from `self.neuron_ids` unconditionally at line 192, regardless of whether the corresponding `transfer_gtc_neuron` inter-canister call succeeded or failed, and always returns `Ok(())` at line 209. Because `forward_whitelisted_unclaimed_accounts` is callable by any principal after 188 days and sets `has_forwarded = true` whenever `transfer` returns `Ok`, any transient failure of the Governance canister during a forwarding call permanently orphans the affected neurons: they are removed from the GTC's canonical neuron list, the account is locked, and no retry path exists in the GTC interface.

## Finding Description

**Root cause — unconditional `retain` (line 192):**

```rust
self.neuron_ids.retain(|id| id != &neuron_id);   // executes before result is inspected
```

The `retain` call at line 192 runs immediately after the `await`, before the `match result` block at line 200. Whether `GovernanceCanister::transfer_gtc_neuron` returns `Ok` or `Err`, the neuron ID is evicted from `self.neuron_ids`. Failed neurons are pushed to `failed_transferred_neurons` (line 204) but are no longer reachable through the account's canonical list.

**Infallible return (line 209):**

```rust
Ok(())   // returned even when every transfer failed
```

`transfer` never propagates partial or total failure to its callers.

**`donate_account` — permanent flag set unconditionally (lines 89–90):**

```rust
account.transfer(custodian_neuron_id).await?;   // ? never fires; transfer always returns Ok
account.has_donated = true;                      // always reached
```

**`forward_whitelisted_unclaimed_accounts` — permanent flag set on `Ok(_)` (lines 118–119):**

```rust
match account.transfer(custodian_neuron_id).await {
    Ok(_) => account.has_forwarded = true,   // always reached
    Err(error) => { println!(...); }
}
```

Since `transfer` always returns `Ok(())`, `has_forwarded = true` is always set.

**Re-entry permanently blocked (lines 175–183):**

```rust
} else if self.has_donated {
    return Err("Account has already donated its funds".to_string());
} else if self.has_forwarded {
    return Err("Account has already forwarded its funds".to_string());
}
```

Once either flag is set, `transfer` is permanently closed. There is no administrative escape hatch in the GTC canister interface.

**Failure points in `transfer_gtc_neuron` (governance.rs lines 1873–1911):**

- `with_neuron` returning `NotFound` if the neuron does not exist
- Arithmetic at line 1900: `donor_cached_neuron_stake_e8s - transaction_fee` (underflow if stake < fee)
- `transfer_funds` returning a ledger error (line 1911, propagated via `?`)
- Canister-level rejection during a Governance canister upgrade (caught by `call` and returned as `Err`)

## Impact Explanation

Neurons that fail to transfer remain in the Governance canister under GTC control. The GTC has no method to manage, dissolve, or retry them. The ICP staked in those neurons is permanently inaccessible: the account owner cannot claim them (blocked by `has_donated`/`has_forwarded`), the GTC cannot retry the transfer (same guards), and no governance proposal can reassign them without a canister upgrade. This constitutes permanent loss of ICP staked in NNS neurons — an in-scope NNS governance asset — matching the **High** impact category: significant NNS security impact with concrete user and protocol harm involving permanent loss of canister-controlled funds.

## Likelihood Explanation

`forward_whitelisted_unclaimed_accounts` is callable by any principal after `SECONDS_UNTIL_FORWARD_WHITELISTED_UNCLAIMED_ACCOUNTS_CAN_BE_CALLED` (188 days post-genesis), a window that has long elapsed on mainnet. The attacker does not need to cause a Governance canister failure — they only need to submit the ingress call during a naturally occurring transient failure window: a Governance canister upgrade (which rejects in-flight inter-canister calls), a ledger error, or a neuron whose `cached_neuron_stake_e8s` is below the transaction fee. NNS canister upgrades are a routine, publicly observable event. The attack is repeatable across all whitelisted unclaimed accounts in a single call.

## Recommendation

1. Move `self.neuron_ids.retain(|id| id != &neuron_id)` inside the `Ok(_)` arm of the `match result` block, so neuron IDs are only removed on confirmed success.
2. Return an `Err` from `transfer` if any individual transfer failed, or at minimum if all transfers failed, so callers can decide whether to set the permanent flag.
3. Do not set `has_donated` / `has_forwarded` unless `transfer` reports complete success.
4. Consider retaining failed neuron IDs in `neuron_ids` (or a dedicated retry list) so a subsequent call can re-attempt only the failed subset.
