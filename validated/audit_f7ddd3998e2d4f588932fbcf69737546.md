### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures by Bypassing `__validate__` for Nonce-1 Transactions - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the Apollo Gateway's stateful transaction validator unconditionally skips the `__validate__` entry-point (signature verification) for any Invoke V3 transaction whose nonce equals 1 and whose sender account has not yet been deployed on-chain, provided the account address appears in the mempool. An unprivileged attacker who observes a pending `DeployAccount` transaction in the mempool can immediately submit a crafted Invoke transaction for that same address with an arbitrary (invalid) signature. The gateway admits this transaction to the mempool without ever verifying the signature, violating the invariant that every admitted transaction must pass account-level validation.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the nonce and resource-bounds checks pass:

```
run_pre_validation_checks
  └─ validate_state_preconditions   (nonce + resource bounds)
  └─ validate_by_mempool            (duplicate / nonce-gap check only)
  └─ skip_stateful_validations      ← returns true → __validate__ is skipped
``` [1](#0-0) 

The skip condition is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
``` [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate = false`, so `blockifier_validator.validate(account_tx)` — the only place the account's `__validate__` Cairo entry point is called — is never reached: [3](#0-2) 

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in the transaction pool **or** in the committed-block state: [4](#0-3) 

The check does **not** verify that the submitter of the Invoke transaction is the same party who submitted the `DeployAccount` transaction, nor does it verify any cryptographic relationship between the two transactions.

The `validate_by_mempool` call that precedes the skip check only validates nonce ordering and duplicate detection — it performs no signature check: [5](#0-4) 

The mempool's `validate_tx` similarly only checks for duplicate hashes and nonce gaps: [6](#0-5) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an Invoke transaction with a completely arbitrary (invalid) signature into the mempool for any account that has a pending `DeployAccount` transaction. Concretely:

1. Alice broadcasts a `DeployAccount` transaction (nonce=0) for a fresh address `A`. It enters the mempool.
2. Bob observes `A` in the mempool and immediately submits an Invoke transaction for `A` with nonce=1 and a forged/empty signature.
3. The gateway sees: `tx_nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` → skips `__validate__` → admits Bob's transaction.
4. Bob's invalid transaction now occupies the nonce=1 slot for address `A` in the mempool.
5. When Alice subsequently submits her legitimate Invoke transaction with nonce=1, the mempool rejects it with `DuplicateNonce`.

The batcher will eventually reject Bob's transaction during execution (because `__validate__` runs with `validate=true` in `new_for_sequencing`), but until then Alice's first post-deploy transaction is blocked. The attacker can repeat this indefinitely to permanently grief any newly-deploying account. [7](#0-6) 

---

### Likelihood Explanation

- **Unprivileged trigger**: Any observer of the public mempool can execute this attack. No special permissions or keys are required.
- **Low cost**: The attacker only needs to submit a single Invoke transaction (paying the gateway's resource-bounds check, but no actual fee since the tx will be rejected before execution).
- **Reliable**: The skip condition is deterministic and always fires when `tx_nonce == 1 && account_nonce == 0 && account_in_mempool`.
- **Visible target**: `DeployAccount` transactions are publicly visible in the mempool before they are sequenced.

---

### Recommendation

The skip logic should be restricted to the legitimate UX case: the submitter of the Invoke transaction must be the same party who submitted the `DeployAccount` transaction. Two concrete options:

1. **Require a matching `deploy_account_tx_hash` field**: Only skip `__validate__` when the Invoke transaction explicitly references the hash of the pending `DeployAccount` transaction (as the `PyValidator` path already does via `deploy_account_tx_hash`). Verify that the referenced hash is actually in the mempool for the same sender address.

2. **Remove the skip entirely at the gateway level**: Accept that the first Invoke after a `DeployAccount` will fail gateway validation and require the user to wait for the `DeployAccount` to be committed before submitting the Invoke. This is the safer option and eliminates the attack surface. [8](#0-7) [9](#0-8) 

---

### Proof of Concept

```
1. Deploy a fresh account address A (attacker controls the private key, or simply observes
   any pending DeployAccount tx for address A in the public mempool).

2. Submit a DeployAccount tx for A (nonce=0, valid signature). It enters the mempool.
   → account_tx_in_pool_or_recent_block(A) now returns true.

3. Submit an Invoke tx for A:
     sender_address = A
     nonce          = 1
     signature      = [0xdeadbeef, 0xdeadbeef]   // arbitrary garbage
     calldata       = <anything>
     resource_bounds = <meets stateless threshold>

4. Gateway stateful validation path:
     get_nonce_from_state(A)  → Nonce(0)          // A not yet on-chain
     validate_state_preconditions:
       validate_resource_bounds → passes
       validate_nonce(nonce=1, account_nonce=0) → passes (0 ≤ 1 ≤ 0+200)
     validate_by_mempool → passes (no duplicate)
     skip_stateful_validations:
       tx.nonce() == 1 && account_nonce == 0 → true
       account_tx_in_pool_or_recent_block(A) → true
       returns true  ← __validate__ is SKIPPED
     run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       blockifier_validator.validate() is called but __validate__ entry point
       is not executed → returns Ok(())

5. Gateway calls mempool.add_tx with the invalid-signature Invoke tx.
   The Invoke tx is now in the mempool occupying nonce=1 for address A.

6. Alice (the legitimate owner of A) submits her valid Invoke tx with nonce=1.
   Mempool returns: DuplicateNonce { address: A, nonce: 1 }
   Alice's transaction is rejected.
```

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
