### Title
`_afterTimelock` uint32 Truncation Bypasses Stop-Loss Timelock â€” (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary

`_afterTimelock` computes `uint32(block.timestamp + timelock)`. When `timelock` is set to a value large enough that the sum exceeds `type(uint32).max`, the result wraps to a past timestamp. Every subsequent `_requireElapsed` check then passes immediately, letting the pool admin execute drawdown, decay, or high-watermark changes with zero delay â€” defeating the LP-protection timelock entirely.

### Finding Description

`_afterTimelock` returns the deadline for any pending stop-loss parameter change:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

The addition is performed in `uint256` space (no overflow), but the result is then **truncated to `uint32`**. `type(uint32).max = 4,294,967,295`. With `block.timestamp â‰ˆ 1,753,000,000` (July 2025), any `timelock > 2,541,967,295` seconds (~80.6 years) causes the truncated result to wrap below the current timestamp.

Concretely, with `timelock = type(uint32).max`:

```
block.timestamp + type(uint32).max
= 1,753,000,000 + 4,294,967,295
= 6,047,967,295

uint32(6,047,967,295) = 6,047,967,295 mod 4,294,967,296 = 1,752,999,999
                      = block.timestamp âˆ’ 1
```

`_requireElapsed` then checks:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

`block.timestamp < (block.timestamp âˆ’ 1)` is `false`, so the revert never fires. The pending change is immediately executable.

There is **no validation** on the `newTimelock` value in `proposeOracleStopLossTimelock`:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
```

### Impact Explanation

The `OracleValueStopLossExtension` timelock is the sole mechanism preventing the pool admin from instantly changing drawdown, decay rate, or high-watermark values. Once bypassed, the pool admin can:

1. Set `drawdownE6 = E6` (100%) â†’ `floorMultiplier = 0` â†’ stop-loss never triggers â†’ all LP value-loss protection is silently disabled.
2. Set `decayPerSecondE8 = E8` (100%/s) â†’ all watermarks decay to zero in one second â†’ stop-loss never triggers.

With the stop-loss disabled, swaps can drain LP bins at unfavorable oracle prices without any circuit-breaker, causing direct LP principal loss.

### Likelihood Explanation

The attack requires the pool admin to first change the timelock to a large value (itself timelocked by the current timelock). If the initial timelock is short (e.g., 1 hour, which is valid since no minimum is enforced), the entire attack completes within hours. Setting a very large timelock superficially resembles a security improvement, making it unlikely to trigger LP alarm. The pool admin is semi-trusted and constrained by timelocks; this path bypasses that constraint.

### Recommendation

Compute and store the deadline in `uint256`, or validate that `timelock` cannot cause overflow before truncation:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 deadline = block.timestamp + oracleStopLossConfig[pool_].timelock;
    require(deadline <= type(uint32).max, "Timelock overflow");
    return uint32(deadline);
}
```

Alternatively, cap `newTimelock` in `proposeOracleStopLossTimelock` to a safe maximum (e.g., 365 days), and store deadlines as `uint64` or `uint256`.
