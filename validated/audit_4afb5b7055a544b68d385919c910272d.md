### Title
Malicious Participant Permanently Stalls DKG for All Honest Parties via Selective Invalid Share — (`src/dkg.rs`)

### Summary

In `do_keyshare`, if a participant fails share validation (caused by a malicious peer sending a targeted invalid share), that participant exits via `?` before ever reaching `broadcast_success`. All remaining honest participants who successfully validated their shares then enter `broadcast_success`, which calls `reliable_broadcast_receive_all` and loops until every participant's broadcast session completes. Because the failed participant never sends its SEND message, that session never completes, and the loop runs forever. There is no abort signal, no timeout, and no recovery path in the `Protocol` trait.

### Finding Description

**Root cause — `broadcast_success` only broadcasts success, never failure:**

`broadcast_success` in `src/dkg.rs` unconditionally broadcasts `(true, session_id)`: [1](#0-0) 

The function contains a dead-code check for `false` votes (line 329–334), implying the design intended participants to signal failure. But the implementation never broadcasts `false`. Participants who fail before this call never reach it at all.

**Failure path — share validation exits early:**

In `do_keyshare`, share validation uses the `?` operator: [2](#0-1) 

If `validate_received_share` returns `Err` (e.g., because a malicious participant sent a crafted invalid share), the entire `do_keyshare` future returns immediately. The participant never reaches line 531: [3](#0-2) 

**Stuck state — `reliable_broadcast_receive_all` waits for all N sessions:**

`broadcast_success` calls `do_broadcast`, which calls `reliable_broadcast_receive_all`. That function only returns when every participant's broadcast session has completed: [4](#0-3) 

The failed participant's session (`state[sid].finish_ready`) is never set to `true` because no SEND message ever arrives for it. The early-exit "malicious sender" check: [5](#0-4) 

only fires after at least one ECHO message is received for that session. If the SEND message never arrives, no ECHOs are generated, `data_echo` stays empty, the loop body for that session never executes, and the early-exit check never triggers. The outer loop spins forever on `chan.recv(wait).await`.

**No abort mechanism in the `Protocol` trait:** [6](#0-5) 

`poke()` returns `Action::Wait` indefinitely when the protocol is blocked. There is no `abort()`, no cancellation token, and no timeout primitive exposed by the library.

### Impact Explanation

**High — Permanent denial of key generation for honest parties.**

A single malicious participant M can permanently stall DKG for all other honest participants:

1. M participates honestly through Rounds 1–3 (session IDs, commitment hashes, commitments + proofs).
2. In Round 4, M sends a valid share to every participant except one honest target H; M sends H a share that fails `validate_received_share`.
3. H's `do_keyshare` returns `ProtocolError::InvalidSecretShare(M)` and exits before `broadcast_success`.
4. Every other honest participant successfully validates their shares and enters `broadcast_success`.
5. `reliable_broadcast_receive_all` waits for H's broadcast session to complete. H never sends a SEND message. The loop runs forever.
6. DKG is permanently stuck for all participants except H. No key is ever generated. No recovery is possible without external process termination and a full protocol restart.

### Likelihood Explanation

The attack requires exactly one malicious participant in the DKG session — within the documented threat model. Share distribution uses `send_private` (point-to-point), so M can trivially send different shares to different recipients. No privileged access, no key leakage, and no cryptographic break are required. The attack is deterministic and reproducible.

### Recommendation

1. **Broadcast failure explicitly.** Restructure `do_keyshare` to catch errors from share validation, then call a `broadcast_result(success: bool)` variant that sends `(false, session_id)` on failure. Other participants receiving a `false` vote can then abort cleanly via the existing check at line 329.

2. **Add an abort/cancel mechanism to the `Protocol` trait.** Expose a method (e.g., `abort()`) that causes `poke()` to return a terminal error, allowing the external caller to cleanly terminate a stuck protocol instance.

3. **Document the timeout requirement.** At minimum, document that callers must impose an external timeout and that a `Action::Wait` response persisting beyond that timeout should be treated as a protocol failure requiring restart.

### Proof of Concept

```
Participants: [A (malicious), B (honest), C (honest)]
Threshold: 2

Round 1–3: A participates honestly (valid session IDs, commitment hashes, commitments).

Round 4 (private share distribution):
  A → B: valid share (passes validate_received_share)
  A → C: crafted invalid share (fails validate_received_share)

Result:
  C: validate_received_share returns Err(InvalidSecretShare(A))
     do_keyshare returns early — broadcast_success is never called
     C's protocol instance exits with an error

  B: validate_received_share passes for all shares
     B reaches broadcast_success → do_broadcast → reliable_broadcast_receive_all
     B waits for C's broadcast session (state[C_sid].finish_ready == false)
     C never sends a SEND message → no ECHOs generated → early-exit check never fires
     B's loop runs forever → DKG permanently stalled for B

  A: same as B — also stuck waiting for C's session

Outcome: DKG produces no key. B and A are permanently blocked.
         No recovery without external process kill and full restart.
```

    type Output;

    /// Poke the protocol, receiving a new action.

    /// The idea is that the protocol should be poked until it returns an error,
