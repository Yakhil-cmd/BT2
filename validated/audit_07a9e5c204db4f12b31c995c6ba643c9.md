Audit Report

## Title
Voting Rewards Permanently Lost When No Neurons Vote on Settled Proposals in SNS Governance - (File: rs/sns/governance/src/types.rs, rs/sns/governance/src/governance.rs)

## Summary
In the SNS governance canister's `distribute_rewards` function, when proposals are settled in a reward round but no eligible votes were cast (`total_reward_shares == 0`), the entire `rewards_purse_e8s` for that round is silently discarded. The rollover condition (`rewards_rolled_over()`) checks only whether `settled_proposals` is empty, not whether rewards were actually distributed, so the purse is neither paid out nor carried forward to the next round. The NNS governance canister contains the identical structural flaw.

## Finding Description
In `rs/sns/governance/src/governance.rs`, `distribute_rewards` builds `rewards_purse_e8s` starting from `self.latest_reward_event().e8s_equivalent_to_be_rolled_over()` (line 5854–5858). It then collects `considered_proposals` from `ready_to_be_settled_proposal_ids()` (line 5822–5823) and tallies `total_reward_shares` by summing voting power of ballots with `Vote::Yes` or `Vote::No` (lines 5894–5931). When `total_reward_shares == dec!(0)` (line 5946), the code logs a warning and leaves `distributed_e8s_equivalent = 0`. The `RewardEvent` is then written (lines 6084–6092) with `settled_proposals: considered_proposals` (non-empty) and `distributed_e8s_equivalent: 0`.

In the next reward round, `e8s_equivalent_to_be_rolled_over()` (`rs/sns/governance/src/types.rs:2054–2060`) calls `rewards_rolled_over()` (line 2065–2067), which returns `true` only when `self.settled_proposals.is_empty()`. Because the prior event has non-empty `settled_proposals`, `rewards_rolled_over()` returns `false`, `e8s_equivalent_to_be_rolled_over()` returns `0`, and the purse from the prior round is permanently gone. The NNS governance has the identical flaw in `rs/nns/governance/src/reward/calculation.rs:120–147`.

Existing guards do not address this: the `total_reward_shares == 0` branch only logs a warning and does not trigger any rollover path. The comment at line 5824–5825 explicitly documents that rollover is intended only for the no-proposals case, confirming the no-votes case was not considered.

## Impact Explanation
Voting rewards (maturity e8s equivalent) legitimately accrued during a reward round are permanently destroyed when no eligible votes were cast on settled proposals. The total maturity in the SNS system becomes permanently lower than the protocol intends. Future stakers receive a smaller share of the cumulative reward pool than they are entitled to, and the loss is irreversible once the `RewardEvent` is written. This constitutes a concrete, permanent financial loss to SNS participants and a protocol accounting invariant violation, matching the High impact category: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The condition is reachable without any special privileges. The trigger is the unprivileged periodic `run_periodic_tasks` heartbeat. Realistic scenarios include: (1) an early SNS lifecycle where a proposal reaches `ReadyToSettle` before any neuron has configured followees or cast a direct vote; (2) a round where all neurons abstain (`Vote::Unspecified`) on every proposal; (3) neurons that voted dissolve and are deleted between proposal settlement and the reward distribution heartbeat. No attacker action is required — the condition arises from normal protocol operation. The loss repeats every time such a round occurs.

## Recommendation
Change `rewards_rolled_over()` in `rs/sns/governance/src/types.rs` (and the equivalent in `rs/nns/governance/src/reward/calculation.rs`) to key on whether rewards were actually distributed, not solely on whether proposals were settled:

```rust
pub(crate) fn rewards_rolled_over(&self) -> bool {
    self.distributed_e8s_equivalent == 0
}
```

Or equivalently, change `e8s_equivalent_to_be_rolled_over` directly:

```rust
pub(crate) fn e8s_equivalent_to_be_rolled_over(&self) -> u64 {
    if self.distributed_e8s_equivalent == 0 {
        self.total_available_e8s_equivalent.unwrap_or_default()
    } else {
        0
    }
}
```

This ensures that any round where no maturity was distributed — whether because there were no proposals or because there were no eligible votes — carries the full purse forward.

## Proof of Concept
1. Deploy an SNS with `VotingRewardsParameters` configured (non-zero reward rate, finite `round_duration_seconds`).
2. Submit a proposal. Ensure no neuron votes directly or via followees — all ballots remain `Vote::Unspecified`.
3. Wait for the proposal's voting period to expire → proposal enters `ReadyToSettle`.
4. Wait for `round_duration_seconds` to elapse → `should_distribute_rewards()` returns `true`.
5. `run_periodic_tasks` fires `distribute_rewards(supply)`.
6. Assert: `considered_proposals` is non-empty; `total_reward_shares == 0`; `distributed_e8s_equivalent == 0`; `RewardEvent.total_available_e8s_equivalent == X` (non-zero).
7. Wait for the next reward round to complete (with or without proposals/votes).
8. Assert: the second `RewardEvent.total_available_e8s_equivalent` equals only the new round's accrual, not `X + new_accrual`.

This is directly testable as a deterministic integration test or PocketIC test by inspecting the emitted `RewardEvent` fields after each round.
