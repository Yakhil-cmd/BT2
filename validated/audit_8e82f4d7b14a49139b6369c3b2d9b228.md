### Title
Signature Bypass via `skip_stateful_validations` Allows Unsigned Invoke Transactions to Enter Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The `skip_stateful_validations` function in the gateway's stateful validation pipeline unconditionally skips the `__validate__` entry point (the account's on-chain signature check) for any invoke transaction with `nonce=1` when the sender address has **any** transaction in the mempool and the on-chain account nonce is zero. Because the check is not restricted to the legitimate account owner, an unprivileged attacker who observes a `deploy_account` transaction in the mempool can submit an invoke with an arbitrary/invalid signature for that same address, bypass signature verification, and occupy the `nonce=1` slot — blocking or griefing the legitimate user.

### Finding Description

The gateway's stateful validation path is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range, resource bounds)
       ├─ validate_by_mempool            (duplicate nonce / fee escalation — no sig check)
       └─ skip_stateful_validations      ← decides whether __validate__ runs
  └─ run_validate_entry_point(skip_validate)
```

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

```rust

    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}

**Attack path:**

1. Alice broadcasts `deploy_account` for address `A` (nonce=0). It enters the mempool; `tx_pool.contains_account(A)` becomes `true`.
2. Attacker observes the mempool (public P2P data), constructs `Invoke(sender=A, nonce=1, signature=[0xdead, 0xbeef])` with an invalid signature.
3. Gateway stateless check passes (size, resource bounds, DA modes — no sig check).
4. `convert_rpc_tx_to_internal` computes the tx hash; no signature verification.
5. `extract_state_nonce_and_run_validations`: `account_nonce=0`, `tx_nonce=1`, `account_tx_in_pool=true` → `skip_validate=true`.
6. `run_validate_entry_point` with `validate: false` → `__validate__` is **never called**. Invalid invoke is admitted.
7. Alice submits her legitimate `Invoke(sender=A, nonce=1)`. `validate_by_mempool` returns `DuplicateNonce` — Alice is blocked unless she pays a fee-escalation premium.
8. At execution time the attacker's tx reaches the blockifier, `__validate__` runs, fails (invalid sig), tx reverts; fee is charged to attacker.

### Impact Explanation

The gateway's invariant — *every transaction in the mempool has passed the account's own `__validate__` signature check* — is broken for invoke transactions with `nonce=1` targeting any address with a pending `deploy_account`. An attacker can:

- Inject signature-invalid transactions into the mempool (High: admission accepts invalid transactions).
- Occupy the `nonce=1` slot for any undeployed account, forcing the legitimate user to pay a fee-escalation premium or be rejected.
- At scale, flood the mempool with invalid transactions targeting all observed `deploy_account` submissions.

### Likelihood Explanation

`deploy_account` transactions are broadcast over the P2P network and are publicly observable. No privileged access is required. The attacker only needs to submit a well-formed invoke (valid resource bounds, correct nonce=1, any signature) before the legitimate user's invoke arrives. The economic cost to the attacker is the fee charged on the failing execution, which is bounded and predictable.

### Recommendation

1. **Restrict the skip to the account owner.** The skip should only apply when the submitter can demonstrate ownership — e.g., require a lightweight ECDSA pre-check on the raw signature fields even when skipping the full VM execution.
2. **Check specifically for a `deploy_account` in the pool.** Expose a `has_deploy_account_in_pool(address)` predicate on the mempool that inspects the transaction type, rather than the generic `account_tx_in_pool_or_recent_block`.
3. **Alternatively, remove the skip entirely** and require users to submit `deploy_account` first, wait for it to be included, then submit the invoke — accepting the UX trade-off in exchange for the security invariant.

### Proof of Concept

```

POST /gateway/add_transaction  { type: DEPLOY_ACCOUNT, sender: A, nonce: 0, sig: [valid] }

# 2. Attacker submits invoke with INVALID signature for the same address

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
