Audit Report

## Title
ckBTC Minter Permanently Destroys User Funds on `AmountTooLow` Without Reimbursement - (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary

The ckBTC minter burns user ckBTC at `retrieve_btc` request time, but builds the Bitcoin transaction asynchronously using a freshly-fetched fee rate. When `build_unsigned_transaction` returns `BuildTxError::AmountTooLow` due to a fee spike between the two phases, the minter finalizes the request as `FinalizedStatus::AmountTooLow` without minting ckBTC back to the user. The reimbursement infrastructure already exists and is used for `BuildTxError::InvalidTransaction`, but is not wired into the `AmountTooLow` branch, causing permanent, unrecoverable fund loss.

## Finding Description

**Phase 1 — Burn at request time** (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, L166–170): The minter reads `fee_based_retrieve_btc_min_amount` from state and rejects if `args.amount < min_retrieve_amount`. If the check passes, `burn_ckbtcs` is called and the request is queued as `Pending`. [1](#0-0) [2](#0-1) 

**Phase 2 — Fee update and transaction build** (`rs/bitcoin/ckbtc/minter/src/lib.rs`, L358–360): `submit_pending_requests` calls `estimate_fee_per_vbyte`, which fetches the live Bitcoin network fee and **overwrites `fee_based_retrieve_btc_min_amount` in state** before `build_unsigned_transaction` is called. [3](#0-2) [4](#0-3) 

**Missing reimbursement** (`rs/bitcoin/ckbtc/minter/src/lib.rs`, L412–434): When `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, the minter iterates the batch and calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. No reimbursement is triggered. [5](#0-4) 

**Contrast with `InvalidTransaction`** (`rs/bitcoin/ckbtc/minter/src/lib.rs`, L400–410): The `InvalidTransaction` branch calls `reimburse_canceled_requests`, which invokes `state::audit::reimburse_withdrawal` and mints ckBTC back to the user's `reimbursement_account`. [6](#0-5) 

**Eventlog replay confirms no reimbursement event** (`rs/bitcoin/ckbtc/minter/src/state/eventlog.rs`, L405–417): `RemovedRetrieveBtcRequest` only pushes `FinalizedStatus::AmountTooLow` — no `WillReimburse` or `Reimbursed` event is emitted. [7](#0-6) 

**50,000-satoshi fee tier jumps** (`rs/bitcoin/ckbtc/minter/src/fees/mod.rs`, L130–147): `fee_based_minimum_withdrawal_amount` rounds up to the nearest 50,000-satoshi multiple, meaning a single fee-rate tick can raise the minimum by up to 50,000 satoshis, easily invalidating a request that was accepted moments earlier. [8](#0-7) 

**Terminal state with no rescue** (`rs/bitcoin/ckbtc/minter/src/state.rs`, L329–330): `AmountTooLow` is a terminal `RetrieveBtcStatus` variant. `WillReimburse` and `Reimbursed` variants exist in the same enum but are never reached from this path. [9](#0-8) 

## Impact Explanation

This is a **High** severity finding. ckBTC is a production chain-key Bitcoin twin token. The bug causes a direct ledger conservation violation: ckBTC is burned (destroyed) but no BTC is ever sent and no ckBTC is ever minted back. Every affected user permanently loses their funds with no recovery path. This matches the allowed impact: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."* The per-user loss is bounded by the minimum withdrawal amount (~100,000–150,000 satoshis), but the vulnerability is systematic and repeatable across any number of users. [10](#0-9) 

## Likelihood Explanation

Bitcoin network fees are highly volatile. During inscription/ordinal activity or mempool congestion, fees can spike by multiples within minutes. `estimate_fee_per_vbyte` is called at the start of every `submit_pending_requests` invocation, so the effective minimum can increase between the user's `retrieve_btc` call and the minter's processing loop. The vulnerability is reachable by any unprivileged user via the public `retrieve_btc` or `retrieve_btc_with_approval` endpoints. The real-world ckBTC upgrade proposal from 2025-06-27 confirms that fee-related withdrawal failures have already occurred in production. [4](#0-3) 

## Recommendation

In the `BuildTxError::AmountTooLow` branch of `submit_pending_requests` (`rs/bitcoin/ckbtc/minter/src/lib.rs`, L412–434), replace the bare `remove_retrieve_btc_request` loop with a call to `reimburse_canceled_requests`, mirroring the `InvalidTransaction` branch. The reimbursement should use `WithdrawalReimbursementReason` (e.g., a new `AmountTooLow` variant), deduct a small penalty fee via `reimbursement_fee_for_pending_withdrawal_requests`, and mint ckBTC back to `request.reimbursement_account`. The `WillReimburse`/`Reimbursed` status variants and the `reimburse_withdrawal` audit function are already implemented and only need to be wired into this path. [11](#0-10) 

## Proof of Concept

1. Bitcoin fees are low: `fee_based_retrieve_btc_min_amount = 100_000 sats`.
2. Alice calls `retrieve_btc(amount = 100_000)`. The check at L169 passes; `burn_ckbtcs` destroys 100,000 satoshis of ckBTC. Request queued as `Pending`.
3. Bitcoin fees spike. `estimate_fee_per_vbyte` runs inside `submit_pending_requests` and updates `fee_based_retrieve_btc_min_amount = 150_000 sats` (one 50,000-satoshi tier jump).
4. `build_unsigned_transaction` is called with Alice's request (amount = 100,000 sats) and the new fee rate. The amount is insufficient → returns `BuildTxError::AmountTooLow`.
5. The minter executes the L426–432 loop: `remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)`. No reimbursement event is emitted.
6. Alice queries `retrieve_btc_status_v2` → `AmountTooLow` (terminal). Her 100,000 satoshis of ckBTC are permanently destroyed.

A deterministic integration test can reproduce this by: (a) setting a low fee rate, (b) submitting a `retrieve_btc` at the minimum, (c) updating the mocked fee rate to the next tier before the processing loop runs, (d) asserting that `retrieve_btc_status_v2` returns `AmountTooLow` and that the ckBTC ledger total supply decreased without a corresponding BTC transaction. [5](#0-4) [8](#0-7)

fn reimburse_canceled_requests<R: CanisterRuntime>(
    state: &mut state::CkBtcMinterState,
    requests: BTreeSet<state::RetrieveBtcRequest>,
    reason: WithdrawalReimbursementReason,
    total_fee: u64,
    runtime: &R,
) {
    assert!(!requests.is_empty());
    let fees = distribute(total_fee, requests.len() as u64);
    // This assertion makes sure the fee is smaller than each request amount
    assert!(
        fees[0] <= state.retrieve_btc_min_amount,
        "BUG: fees {fees:?} for {} withdrawal requests are larger than `retrieve_btc_min_amount` {}",
        requests.len(),
        state.retrieve_btc_min_amount
    );
    for (request, fee) in requests.into_iter().zip(fees.into_iter()) {
        if let Some(account) = request.reimbursement_account {
            let amount = request.amount.saturating_sub(fee);
            if amount > 0 {
                state::audit::reimburse_withdrawal(
                    state,
                    request.block_index,
                    amount,
                    account,
                    reason.clone(),
                    runtime,
                );
            }
        } else {
            log!(
                Priority::Info,
                "[reimburse_canceled_requests]: account is not found for retrieve_btc request ({:?})",
                request
            );
        }
    }
}
