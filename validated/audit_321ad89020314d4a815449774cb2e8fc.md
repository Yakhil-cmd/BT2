### Title
One-Sided Participant-State Check in `assert_reshare_keys_invariants` Allows New Participant to Inject Arbitrary Secret into Reshare — (File: `src/dkg.rs`)

---

### Summary

`assert_reshare_keys_invariants` validates only one direction of the participant/key-presence invariant: it rejects an **old** participant that omits their signing key, but it never rejects a **new** participant that supplies one. A malicious new participant can therefore pass a non-`None` `old_signing_key` through validation unchallenged, causing `do_reshare` to derive and contribute a non-zero secret on their behalf, corrupting the aggregate reshared key.

---

### Finding Description

`assert_reshare_keys_invariants` is the public-API guard for the reshare protocol. Its own inline comment documents the intended invariant:

```
// if me is not in the old participant set then ensure that old_signing_key is None
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(...)
}
``` [1](#0-0) 

The comment describes **two** symmetric conditions that must be enforced:

| Condition | Expected behaviour | Implemented? |
|---|---|---|
| `me` ∈ old set **and** key is `None` | Reject | ✅ |
| `me` ∉ old set **and** key is `Some(_)` | Reject | ❌ |

The second branch is entirely absent. The function returns `Ok` when a new participant supplies a non-`None` `old_signing_key`.

`do_reshare` then consumes that key without any further guard:

```rust
let intersection = old_participants.intersection(&participants);
let secret = old_signing_key
    .map(|x_i| {
        intersection
            .lagrange::<C>(me)          // me is NOT in intersection
            .map(|lambda| lambda * x_i.to_scalar())
    })
    .transpose()?
    .unwrap_or_else(<C::Group as Group>::Field::zero);
``` [2](#0-1) 

`ParticipantList::lagrange` computes the Lagrange basis polynomial evaluated at `me`'s scalar, regardless of whether `me` is a member of the list:

```rust
pub fn lagrange<C: Ciphersuite>(&self, p: Participant) -> Result<Scalar<C>, ProtocolError> {
    let p = p.scalar::<C>();
    let identifiers: Vec<Scalar<C>> = self.participants().iter()
        .map(Participant::scalar::<C>).collect();
    Ok(compute_lagrange_coefficient::<C>(&identifiers, &p, None)?.0)
}
``` [3](#0-2) 

Because `me` is not equal to any element of the intersection, the denominator `(p − x_j)` is non-zero for every `j`, so the computation succeeds and returns a non-zero scalar. The product `lambda * x_i` is therefore a non-zero secret that is silently injected as the constant term of the new participant's polynomial inside `do_keyshare`. [4](#0-3) 

Inside `do_keyshare`, the proof-of-knowledge verification for a new participant only checks commitment length; it does **not** verify that the constant term is zero:

```rust
None => {
    if old_participants.is_none_or(|p| p.contains(participant)) {
        return Err(ProtocolError::MaliciousParticipant(participant));
    }
    if commitment.coefficients().len() != threshold - 1 {
        return Err(ProtocolError::IncorrectNumberOfCommitments);
    }
    Ok(())   // constant term is never checked
}
``` [5](#0-4) 

The injected non-zero constant term therefore propagates undetected into the aggregate signing share.

---

### Impact Explanation

A new participant who injects a non-zero secret `δ` causes the aggregate reshared key to become `(old_secret + δ) · G` instead of `old_secret · G`. All honest parties accept the resulting key shares (no cross-check against the old verification key is performed at the share-combination step), but the new public key is inconsistent with the old one. Any subsequent signing attempt will produce signatures that fail to verify against the original public key, permanently denying signing capability to honest parties.

**Impact category**: High — Corruption of reshare outputs so honest parties accept inconsistent public keys and unusable cryptographic outputs.

---

### Likelihood Explanation

The attacker is a new participant (not in `old_participants`) who controls their own call to the public `reshare` API. They simply pass any non-`None` `SigningShare` value. No privileged access, no key leakage, and no cryptographic break is required. The missing check is in the public validation entry point, so the attack is reachable by any unprivileged library caller participating in a reshare.

---

### Recommendation

Add the symmetric guard immediately after the existing check in `assert_reshare_keys_invariants`:

```rust
// Existing check (old participant must supply a key)
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is present in the old participant list but provided no share"
    )));
}

// Missing check (new participant must NOT supply a key)
if !old_participants.contains(me) && old_signing_key.is_some() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is not present in the old participant list but provided a share"
    )));
}
``` [6](#0-5) 

---

### Proof of Concept

1. Run a DKG with old participants `{P1, P2, P3}` and threshold 2.
2. Introduce a new participant `P4` (not in the old set) for a reshare to `{P1, P2, P3, P4}`.
3. `P4` calls `reshare` with `old_signing_key = Some(arbitrary_share)`.
4. `assert_reshare_keys_invariants` returns `Ok` — the missing branch is never reached.
5. `do_reshare` computes `secret = lagrange_intersection(P4) * arbitrary_share ≠ 0`.
6. `P4`'s polynomial has a non-zero constant term; `do_keyshare` accepts it (proof-of-knowledge check for new participants only validates commitment length).
7. The aggregate reshared key is `old_key + δ·G` where `δ ≠ 0`; all honest parties store shares of this corrupted key and signing against the original public key becomes permanently impossible.
