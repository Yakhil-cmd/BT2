### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Deny Block Rewards to Stakers - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `StakingRewardsManagerImpl` lacks the "Only starkware sequencer" access control mandated by the protocol specification. Any unprivileged caller can invoke `update_rewards(staker_address, disable_rewards: true)` to consume the global per-block reward slot without distributing rewards, permanently denying the targeted staker their yield for that block.

---

### Finding Description

The protocol specification at `docs/spec.md` lines 1626â€“1652 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` lines 1449â€“1507 contains no such restriction:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks: not paused, non-zero caller
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // global slot consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // no rewards distributed
    }
    ...
``` [1](#0-0) 

`general_prerequisites()` only checks that the contract is unpaused and the caller is non-zero â€” there is no sequencer identity check: [2](#0-1) 

The storage variable `last_reward_block` is **global** (not per-staker): [3](#0-2) 

Once `last_reward_block` is set to block N, no further reward update can occur for block N â€” the next call reverts with `REWARDS_ALREADY_UPDATED`. If an attacker calls `update_rewards(any_staker, disable_rewards: true)` before the sequencer in block N, the sequencer's legitimate call for block N is permanently blocked, and **all stakers lose their consensus rewards for that block**.

The specification confirms this is the intended reward distribution path: [4](#0-3) 

---

### Impact Explanation

**High â€” Permanent freezing of unclaimed yield.**

When `disable_rewards: true` is passed and the slot is consumed, the staker's `unclaimed_rewards_own` is never incremented for that block. The rewards are not deferred â€” they are permanently lost. An attacker who calls this function in every block before the sequencer can continuously deny all stakers their consensus rewards indefinitely.

This matches the allowed impact: *"High: Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**Medium.** On Starknet's current centralized sequencer, the sequencer controls transaction ordering and would normally process its own `update_rewards` call first. However:

1. The function is a public ABI entry point with no on-chain enforcement of the sequencer restriction â€” the spec's access control is entirely unenforced.
2. If the sequencer's call is submitted as a regular user transaction (not a protocol-level forced transaction), any user transaction submitted in a prior block that lands before the sequencer's call in the next block can consume the slot.
3. The attack requires no special privilege, no leaked key, and no external dependency â€” only the ability to submit a transaction.

---

### Recommendation

Enforce the "Only starkware sequencer" access control on-chain. Add a stored `sequencer_address` and assert `get_caller_address() == self.sequencer_address.read()` at the top of `update_rewards`, or use an existing role (e.g., a `SEQUENCER_ROLE` via the `RolesComponent`) to gate the function.

---

### Proof of Concept

1. Consensus rewards are active (current epoch â‰Ą `consensus_rewards_first_epoch`).
2. Attacker submits a transaction calling `staking.update_rewards(victim_staker_address, disable_rewards: true)` at block N.
3. Inside `update_rewards`: `general_prerequisites()` passes (contract unpaused, caller non-zero); `current_block_number (N) > last_reward_block` passes; `last_reward_block` is written to N; the `if disable_rewards` branch returns early â€” **no rewards are distributed**.
4. The sequencer's legitimate `update_rewards` call for block N now reverts with `REWARDS_ALREADY_UPDATED` because `N > N` is false.
5. The victim staker's `unclaimed_rewards_own` is not incremented for block N â€” the yield is permanently lost.
6. Repeating this every block permanently freezes all staker rewards. [5](#0-4) [6](#0-5)

            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();

            assert!(
                current_block_number > self.last_reward_block.read(),

            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();

                "{}",

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
