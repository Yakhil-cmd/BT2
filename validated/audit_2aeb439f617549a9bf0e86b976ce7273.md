### Title
Gateway Stateful Validator Uses Stale Previous-Block L2 Gas Price for Resource Bound Admission Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`validate_resource_bounds` and `run_validate_entry_point` inside `StatefulTransactionValidator` both derive their gas-price reference from the **previous** finalized block via `gateway_fixed_block_state_reader.get_block_info()`. The current block's L2 gas price — which is what the blockifier will actually enforce during sequencing — is never consulted. This creates a systematic mismatch between the gateway's admission decision and the blockifier's execution-time fee check, causing the gateway to accept transactions that will be reverted on execution (when the price rises) or to reject transactions that would succeed (when the price falls).

---

Only `block_number` is advanced; all three gas-price vectors (`l1_gas_price`, `l1_data_gas_price`, `l2_gas_price`) remain from the previous block. The `BlockContext` built here is passed to `StatefulValidator::perform_validations`, which calls `AccountTransaction::perform_pre_validation_stage` → `check_fee_bounds` and `verify_can_pay_committed_bounds`. Both checks therefore use the previous block's prices, not the prices that will be in force when the transaction is actually sequenced. [2](#0-1) 

**`perform_pre_validation_stage` uses the block context's gas prices**

During actual block production the batcher builds a fresh `BlockContext` with the **current** block's gas prices. The same `check_fee_bounds` and `verify_can_pay_committed_bounds` are re-run there. Any transaction whose `max_price_per_unit` sits between the previous block's price and the current block's price will pass the gateway but fail (revert) during sequencing.

---

### Impact Explanation

**Admission of transactions that will revert (price increase scenario):** When the L2 gas price rises from block N to block N+1, a transaction with `max_price_per_unit = price_N` satisfies the gateway threshold (`price_N >= 100% × price_N`) and passes `check_fee_bounds` against the stale context. During sequencing the batcher uses `price_{N+1} > price_N`; `check_fee_bounds` fails and the transaction is reverted. The gateway has admitted an invalid transaction.

**Rejection of valid transactions (price decrease scenario):** When the L2 gas price falls from block N to block N+1, a transaction with `max_price_per_unit = price_{N+1}` is rejected by the gateway (`price_{N+1} < 100% × price_N`) even though it would succeed during sequencing. The gateway has rejected a valid transaction.

Both outcomes match the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The L2 gas price is recalculated every block by the consensus orchestrator. Any non-trivial price movement between consecutive blocks (which is the normal operating condition) triggers the mismatch. The trigger requires no special privilege: any user submitting an `AllResources` (V3) transaction is affected. The `L1Gas` path is explicitly exempted from the check (`ValidResourceBounds::L1Gas(_) => {}`), so only V3 transactions are affected, but V3 is the current standard format. [4](#0-3) 

---

### Recommendation

1. In `validate_resource_bounds`, replace `previous_block_l2_gas_price` with the **next** block's projected L2 gas price, as the TODO comment already acknowledges (`// TODO(Arni): getnext_l2_gas_price from the block header`). The next block's price should be obtained from the same source the batcher uses when building the block context.

2. In `run_validate_entry_point`, after incrementing `block_number`, also update the gas-price vectors in `block_info` to reflect the prices that will be used for the next block, so that `check_fee_bounds` and `verify_can_pay_committed_bounds` simulate the actual execution environment rather than the previous block's environment. [5](#0-4) [6](#0-5) 

---

### Proof of Concept

1. Observe the current L2 gas price from the latest finalized block: `price_N`.
2. Craft an `AllResources` (V3) invoke transaction with `l2_gas.max_price_per_unit = price_N`.
3. Submit the transaction to the gateway immediately after a block whose L2 gas price is `price_{N+1} > price_N`.
4. **Gateway admission**: `validate_resource_bounds` reads `price_N` (the previous block), computes threshold = `100% × price_N = price_N`, checks `price_N >= price_N` → **passes**.
5. `run_validate_entry_point` builds a `BlockContext` with `price_N` gas prices and block number `N+1`; `check_fee_bounds` checks `price_N >= price_N` → **passes**; transaction is admitted to the mempool.
6. **Sequencing**: the batcher builds a `BlockContext` with `price_{N+1}`; `check_fee_bounds` checks `price_N >= price_{N+1}` → **fails**; transaction is reverted.
7. The gateway has admitted a transaction that the sequencer cannot execute successfully, demonstrating the stale-state admission flaw. [7](#0-6) [8](#0-7) [9](#0-8)

#[derive(Clone, Debug, Serialize, Deserialize, Validate, PartialEq)]
pub struct StatefulTransactionValidatorConfig {
    // If true, ensures the max L2 gas price exceeds (a configurable percentage of) the base gas
    // price of the previous block.
    pub validate_resource_bounds: bool,
    pub max_allowed_nonce_gap: u32,
    pub reject_future_declare_txs: bool,
    pub max_nonce_for_validation_skip: Nonce,
    pub versioned_constants_overrides: Option<VersionedConstantsOverrides>,
    // Minimum gas price as percentage of threshold to accept transactions.
    pub min_gas_price_percentage: u8, // E.g., 80 to require 80% of threshold.
}

impl Default for StatefulTransactionValidatorConfig {
    fn default() -> Self {
        StatefulTransactionValidatorConfig {
            validate_resource_bounds: true,
            max_allowed_nonce_gap: 200,
            reject_future_declare_txs: true,
            max_nonce_for_validation_skip: Nonce(Felt::ONE),
            min_gas_price_percentage: 100,
            versioned_constants_overrides: None,
        }
    }
}
