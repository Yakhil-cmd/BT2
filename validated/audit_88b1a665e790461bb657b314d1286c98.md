### Title
Stale `target_shard_index` from pre-resharding epoch used to index post-resharding `outcome_roots`, producing wrong `outcome_root_proof` — (`chain/client/src/view_client_actor.rs`)

---

### Summary

`Handler<GetExecutionOutcome>` computes `target_shard_index` once from the **outcome block's** epoch shard layout, then calls `get_next_block_hash_with_new_chunk` which may return a block `h` belonging to a **later epoch with a different (larger) shard layout**. The stale index is then used to subscript `outcome_roots` collected from block `h`, producing an `outcome_root_proof` that is a valid Merkle path to the **wrong shard's** `prev_outcome_root`. Light-client proof verification therefore fails for any execution outcome that straddles a resharding boundary.

---

### Finding Description

The handler computes `target_shard_index` at lines 1155–1158 using `shard_layout` derived from the outcome block's `epoch_id` (epoch E):

```rust
let shard_layout =
    self.epoch_manager.get_shard_layout(&epoch_id).into_chain_error()?;
let target_shard_id =
    account_id_to_shard_id(self.epoch_manager.as_ref(), &account_id, &epoch_id)
        .into_chain_error()?;
let target_shard_index = shard_layout
    .get_shard_index(target_shard_id)
    .map_err(Into::into)
    .into_chain_error()?;
``` [1](#0-0) 

`get_next_block_hash_with_new_chunk` is then called. When a resharding event has occurred between the outcome block and the next-chunk block, this function internally expands `shard_ids` to the child shards of the new epoch and returns `(h, child_shard_id)` where `h` is in epoch E+1 with a different shard layout: [2](#0-1) 

The returned `target_shard_id` (now a child shard in epoch E+1) shadows the original via `if let Some((h, target_shard_id)) = res`, but **`target_shard_index` is never recomputed**: [3](#0-2) 

`outcome_roots` is collected from block `h` (epoch E+1, with 2N chunk headers), and `merklize(&outcome_roots).1[target_shard_index]` is returned as the proof. Because `target_shard_index` was derived from epoch E's N-shard layout, it indexes the **wrong slot** in the 2N-entry `outcome_roots` vector.

The only guard present is a bounds check (`target_shard_index >= outcome_roots.len()`), which does not fire when the new epoch has *more* shards than the old one — the stale index is still in-range but points to a different shard's root: [4](#0-3) 

---

### Impact Explanation

The returned `outcome_root_proof` is a Merkle path from the block's `outcome_root` to the **wrong shard's** `prev_outcome_root`. The light-client verification step:

```
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
```

will not reconstruct the correct `block_outcome_root` because the proof leaf is the wrong shard's root. Every light-client that calls `EXPERIMENTAL_light_client_proof` for a receipt/transaction whose outcome block is in epoch E and whose next-chunk block is in epoch E+1 (post-resharding) will receive an unverifiable proof. The `outcome_root_proof` field of `GetExecutionOutcomeResponse` is the exact corrupted value. [5](#0-4) 

---

### Likelihood Explanation

The condition requires a resharding event between the outcome block and the next-chunk block. NEAR's sharding roadmap makes this a planned, production occurrence. Any receipt whose `receiver_id` maps to a shard that is split will trigger the bug automatically — no special attacker action is needed beyond submitting a normal receipt and querying the public RPC endpoint afterward.

---

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` using the shard layout of block `h`'s epoch rather than the outcome block's epoch:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let next_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let next_shard_layout =
        self.epoch_manager.get_shard_layout(&next_epoch_id).into_chain_error()?;
    let target_shard_index = next_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... collect outcome_roots and merklize as before
}
```

This ensures `target_shard_index` always corresponds to the shard layout of the block whose `outcome_roots` are being Merkle-ized.

---

### Proof of Concept

Property test outline:

1. Construct a two-epoch chain: epoch E with N shards, epoch E+1 with 2N shards (one shard splits).
2. Execute a receipt whose `receiver_id` maps to the split shard in epoch E.
3. Call `GetExecutionOutcome` for that receipt after the resharding block is finalized.
4. Assert `verify_path(block_outcome_root, outcome_root_proof, chunk_outcome_root)` — this assertion **fails** because `outcome_root_proof` indexes the wrong slot.
5. Recompute `target_shard_index` from epoch E+1's shard layout and re-run `merklize`; assert the corrected proof verifies — this assertion **passes**.
