### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent ERC-20 Token Loss — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then invoking the Aurora exit precompile via an inline assembly `call`. The return value of that assembly `call` is captured in a local variable `res` but is **never inspected or acted upon**. If the precompile call fails for any reachable reason (invalid NEAR account ID, amount overflow, etc.), the ERC-20 tokens are permanently destroyed while no corresponding NEP-141 transfer is ever scheduled on the NEAR side, resulting in irreversible loss of bridged assets.

---

### Finding Description

`EvmErc20.sol` `withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no `if iszero(res) { revert(0,0) }`
    }
}

The identical pattern appears in `EvmErc20V2.sol`: [2](#0-1) 

And in `withdrawToEthereum` in both files: [3](#0-2) 

These contracts are the production ERC-20 bytecode embedded directly into the Aurora Engine and deployed for every bridged NEP-141 token: [4](#0-3) 

The `ExitToNear` precompile (`0xe9217bc7…`) returns a non-fatal `ExitError` (which causes the EVM sub-`call` to return 0 without reverting the outer frame) in several reachable conditions:

- **Invalid NEAR recipient account ID** — `ERR_INVALID_RECEIVER_ACCOUNT_ID` is returned when the `recipient` bytes do not parse as a valid NEAR account ID.
- **Amount exceeds `u128::MAX`** — `ERR_INVALID_AMOUNT` is returned when the 32-byte big-endian amount exceeds `u128::MAX`. [5](#0-4) [6](#0-5) 

Because the EVM `call` opcode only propagates a `Fatal` precompile failure upward; `ExitError` failures merely set the sub-call return value to 0. Since `res` is never tested, the outer `withdrawToNear` / `withdrawToEthereum` frame completes successfully after the burn, and no NEAR-side promise is ever created.

---

### Impact Explanation

**Critical — Permanent freezing / loss of funds.**

A user's ERC-20 tokens (representing bridged NEP-141 assets) are irreversibly burned inside the EVM. Because the precompile failure is silent, no `ft_transfer` or `ft_transfer_call` promise is scheduled on NEAR. The NEP-141 tokens remain locked in the Aurora engine account forever with no mechanism to recover them. The user has lost the full `amount` of bridged assets.

---

### Likelihood Explanation

**Medium.** Every user who calls `withdrawToNear` with a recipient string that is not a valid NEAR account ID (e.g., contains uppercase letters, exceeds 64 characters, uses disallowed characters, or is empty) will silently lose their tokens. This is a realistic user-error path that the contract is supposed to guard against by reverting. Additionally, any future condition that causes the precompile to return a non-fatal error (e.g., a new validation rule added to the precompile) would silently trigger the same loss for all callers.

---

### Recommendation

Check the return value of every assembly `call` to an exit precompile and revert if it fails:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures the `_burn` is atomically rolled back whenever the precompile rejects the call, preserving the invariant that ERC-20 supply and NEP-141 custody remain in sync.

---

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora, creating an `EvmErc20` instance.
2. Acquire a non-zero ERC-20 balance on Aurora.
3. Call `withdrawToNear(bytes("INVALID ACCOUNT!!"), balance)` — the recipient string contains spaces and uppercase letters, which are forbidden in NEAR account IDs.
4. The `ExitToNear` precompile returns `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`.
5. The assembly `call` returns `res = 0`; since `res` is unchecked, `withdrawToNear` returns without reverting.
6. Observe: the caller's ERC-20 balance is now 0; no NEP-141 transfer was made; the NEP-141 tokens remain locked in the Aurora engine account with no recovery path.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
