### Title
Permissionless `deploy_erc20_token` Allows Front-Running to Permanently Poison the NEP-141→ERC-20 Bridge Mapping — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `deploy_erc20_token` NEAR contract method has no access-control guard. Any NEAR account can call it with any NEP-141 account ID. Because the NEP-141→ERC-20 mapping is write-once (no update or removal path exists), an attacker who observes a legitimate `deploy_erc20_token` call in the NEAR transaction pool can front-run it, permanently registering a different ERC-20 address for the target NEP-141 token and making the legitimate registration impossible forever.

---

### Finding Description

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` is the public NEAR entry point that bridges a NEP-141 token to Aurora by deploying a standard ERC-20 contract and recording the `nep141 → erc20_address` mapping:

```rust
// engine/src/contract_methods/connector.rs  lines 111-158
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;   // ← only liveness check
        // NO require_owner_only / caller validation
        ...
        engine::deploy_erc20_token(nep141, None, io, env, handler)?;
        ...
    })
}
``` [1](#0-0) 

The inner `engine::deploy_erc20_token` deploys the ERC-20 bytecode and then calls `register_token`, which is a **write-once** operation:

```rust
// engine/src/engine.rs  lines 722-741
pub fn register_token(&mut self, erc20_token: Address, nep141_token: AccountId)
    -> Result<(), RegisterTokenError>
{
    match get_erc20_from_nep141(&self.io, &nep141_token) {
        Err(GetErc20FromNep141Error::Nep141NotFound) => (),
        Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered),  // ← no update path
        ...
    }
    nep141_erc20_map(self.io).insert(&nep141_token, &erc20_token);
    Ok(())
}
``` [2](#0-1) 

Once a NEP-141 account ID is registered, it can never be re-registered or corrected. There is no `unregister_token` or `update_token` function anywhere in the codebase.

The ERC-20 address produced by `deploy_erc20_token` is deterministic: it is derived from the EVM nonce of `near_account_to_evm_address(predecessor_account_id)`. A different NEAR predecessor produces a different EVM origin address and therefore a different ERC-20 deployment address. [3](#0-2) 

---

### Impact Explanation

**Impact: High — Permanent freezing of bridged funds.**

Attack scenario:

1. A legitimate operator submits `deploy_erc20_token("usdc.near")` to register the canonical USDC bridge on Aurora.
2. An attacker observes the pending NEAR transaction and submits the same call from their own account with higher priority.
3. The attacker's transaction executes first. An ERC-20 is deployed at address `A_attacker` (derived from the attacker's NEAR account nonce) and the mapping `usdc.near → A_attacker` is written permanently.
4. The legitimate operator's transaction fails with `ERR_NEP141_TOKEN_ALREADY_REGISTERED`. The mapping cannot be corrected.
5. All subsequent `ft_on_transfer` calls from the `usdc.near` NEP-141 contract will mint tokens at `A_attacker` — an address that no existing DeFi protocol, wallet, or integration knows about.
6. Users who bridge `usdc.near` receive ERC-20 tokens at `A_attacker`. Any protocol that pre-integrated the expected canonical address (or that the legitimate operator intended to deploy at) cannot interact with these tokens. The bridged NEP-141 value is permanently locked in the Aurora engine while the corresponding ERC-20 tokens are stranded at an unrecognized address. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: Medium.**

NEAR transactions are visible in the mempool before finalization. An attacker running a NEAR node or monitoring the RPC can detect a `deploy_erc20_token` call and submit a competing transaction. No special privilege, leaked key, or governance capture is required — only a funded NEAR account. High-value tokens (USDC, USDT, wBTC) are the most attractive targets because their ERC-20 addresses are widely integrated.

---

### Recommendation

Add a caller-identity check to `deploy_erc20_token`. The simplest fix is to restrict the call to the contract owner (consistent with how other privileged connector operations are guarded):

```rust
require_owner_only(&state::get_state(&io)?, &env.predecessor_account_id())?;
```

Alternatively, validate that the caller is the NEP-141 contract itself (i.e., `predecessor_account_id == nep141`), which ties the registration to the token's own account and eliminates the front-running surface entirely. [1](#0-0) 

---

### Proof of Concept

1. Alice (legitimate operator) broadcasts:
   ```
   aurora.deploy_erc20_token(args=borsh("usdc.near"))
   ```
   from NEAR account `alice.near`.

2. Bob (attacker) sees the pending transaction and broadcasts the identical call from `bob.near` with a higher gas price / earlier shard slot.

3. Bob's transaction is included first. The engine executes:
   ```rust
   // predecessor = bob.near → evm_origin = near_account_to_evm_address(b"bob.near")
   // nonce of that evm_origin = 0
   // ERC-20 deployed at address = keccak(rlp([evm_origin, 0]))[12:]  → A_bob
   nep141_erc20_map.insert("usdc.near", A_bob)   // permanent
   ```

4. Alice's transaction executes next and hits:
   ```rust
   Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered)
   ```
   Alice's call reverts. The mapping `usdc.near → A_bob` is now immutable.

5. Any user who calls `usdc.near::ft_transfer_call(receiver_id="aurora", ...)` will have their NEP-141 tokens locked in Aurora and receive ERC-20 tokens at `A_bob` — an address unknown to any existing integration. [5](#0-4) [6](#0-5) [7](#0-6)
