### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Unclaimed Pool Rewards - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in `pool.cairo` contains an unbounded `while` loop that iterates over every entry in a pool member's balance trace since their last reward claim. Because `exit_delegation_pool_intent(0)` appends a new trace entry each epoch at zero token cost, a pool member (or a griefing attacker who is themselves a pool member) can grow this trace without bound. Once the trace is large enough, `claim_rewards` will always revert due to gas exhaustion, permanently freezing the pool member's unclaimed yield.

---

### Finding Description

`calculate_rewards` in `pool.cairo` iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current trace length:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The developers themselves acknowledge the loop is unbounded. The trace grows by one entry per epoch in which `set_member_balance` is called:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    ...
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

`set_member_balance` is called unconditionally at the end of `exit_delegation_pool_intent`, with no minimum-amount guard:

```cairo
fn exit_delegation_pool_intent(ref self: ContractState, amount: Amount) {
    ...
    let new_delegated_stake = total_amount - amount;
    self.write_pool_member_info(:pool_member, :pool_member_info);
    // Update the pool member's balance checkpoint.
    self.set_member_balance(:pool_member, amount: new_delegated_stake);
``` [3](#0-2) 

Calling `exit_delegation_pool_intent(0)` is valid (only `amount <= total_amount` is checked), costs nothing beyond transaction gas, and inserts a new checkpoint keyed to `current_epoch + K`. Repeating this call once per epoch across many epochs inflates the trace indefinitely.

`entry_to_claim_from` is only advanced on a *successful* `claim_rewards` execution:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [4](#0-3) 

Once the trace is large enough that the loop exceeds the Starknet transaction gas limit, every subsequent `claim_rewards` call reverts, `entry_to_claim_from` is never advanced, and the rewards are permanently inaccessible.

---

### Impact Explanation

A pool member whose `pool_member_epoch_balance` trace has grown beyond the per-transaction gas budget can never successfully execute `claim_rewards`. Because there is no partial-range claim and no way to advance `entry_to_claim_from` outside of a successful claim, all accrued unclaimed yield is permanently frozen in the contract.

**Impact: High — Permanent freezing of unclaimed yield.**

---

### Likelihood Explanation

- `exit_delegation_pool_intent(0)` is a valid, zero-cost (beyond gas) call available to any pool member.
- A pool member who calls it once per epoch over a few hundred epochs (a realistic multi-year horizon) will accumulate enough trace entries to push `claim_rewards` past the gas limit.
- Even without deliberate abuse, a legitimate user who frequently adjusts their delegation intent across many epochs without claiming rewards can reach this state organically.
- No privileged role, bridge compromise, or external dependency is required.

**Likelihood: Medium** (requires sustained activity over many epochs, but is reachable by any pool member).

---

### Recommendation

1. **Short term:** Add a `max_entries` parameter to `claim_rewards` (or an internal helper) so the loop processes at most N trace entries per call, advancing `entry_to_claim_from` incrementally. This allows users to drain the backlog across multiple transactions.
2. **Short term:** Add a guard in `exit_delegation_pool_intent` that skips the `set_member_balance` call when `amount == 0` and the balance is unchanged, preventing zero-cost trace inflation.
3. **Long term:** Cap the number of balance-trace entries that can be written per pool member, or enforce a minimum meaningful change before inserting a new checkpoint.

---

### Proof of Concept

1. Alice enters the delegation pool with a small amount.
2. Alice calls `exit_delegation_pool_intent(0)` once per epoch for N epochs (e.g., N = 500). Each call appends one entry to `pool_member_epoch_balance` at zero token cost.
3. Alice never calls `claim_rewards` during this period.
4. After N epochs, Alice calls `claim_rewards`. The `calculate_rewards` loop must iterate over all N entries. If N × (gas per iteration) > Starknet block gas limit, the transaction reverts.
5. Because `entry_to_claim_from` is only updated on success, every future `claim_rewards` call also reverts. Alice's accumulated rewards are permanently frozen. [5](#0-4)

                );

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
