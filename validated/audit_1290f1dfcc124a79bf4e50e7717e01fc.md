### Title
Pool Member Exit Intent Cancellation Can Be Front-Run - (File: `src/pool/pool.cairo`)

---

### Summary

A pool member who has initiated an exit intent and subsequently wishes to cancel it by calling `exit_delegation_pool_intent(amount: 0)` can be front-run by any address calling `exit_delegation_pool_action(pool_member)`. This forces the pool member out of their staking position against their will, causing loss of future staking rewards and requiring them to re-enter the pool.

---

### Finding Description

The pool contract implements a two-step exit process for pool members:

1. **Intent phase**: A pool member calls `exit_delegation_pool_intent(amount)` to signal intent to exit. This deducts `amount` from their active staked balance and starts the `exit_wait_window` timer.

2. **Cancellation mechanism**: Calling `exit_delegation_pool_intent(amount: 0)` is the only way to cancel an existing intent. When `amount == 0`, the function sets `pool_member_info.unpool_time = Option::None` and clears the intent in the staking contract's `pool_exit_intents` map.

3. **Action phase**: `exit_delegation_pool_action(pool_member)` finalizes the exit by transferring `unpool_amount` back to the pool member. Critically, **any address can call this function** — there is no access control restriction.

The race condition:
- Pool member broadcasts `exit_delegation_pool_intent(amount: 0)` to cancel their intent after the `exit_wait_window` has elapsed.
- Before that transaction is included in a block, any observer can call `exit_delegation_pool_action(pool_member)` to finalize the exit.
- The pool member's `unpool_amount` is returned to them, but their staking position is permanently exited.

### Impact Explanation

When `exit_delegation_pool_action` is front-run:

- The pool member's `unpool_amount` tokens are returned to their address (no principal loss).
- Their staking position is exited: `unpool_amount` and `unpool_time` are zeroed out.
- The pool member loses all **future staking rewards** they would have earned by remaining in the pool.
- The pool member must pay gas to re-enter the pool via `enter_delegation_pool` or `add_to_delegation_pool`.
- Accumulated but unclaimed rewards (stored in `pool_member_info`) are **not** lost and remain claimable via `claim_rewards`.

This constitutes **griefing with no profit motive but damage to users**: the attacker gains nothing, but the victim suffers disruption of their staking position and loss of future yield.

**Impact: Medium** — Griefing with no profit motive but damage to users or protocol.

---

### Likelihood Explanation

- Starknet's sequencer currently processes transactions in a defined order. A mempool observer (or the sequencer itself) can detect a pending `exit_delegation_pool_intent(amount: 0)` transaction and submit `exit_delegation_pool_action` ahead of it.
- The attack window is narrow but real: it exists between the pool member broadcasting the cancellation and the transaction being finalized.
- The attacker requires no special privileges, no funds, and no prior relationship with the victim — only the ability to submit a transaction.
- The attack is cheap (low gas) and requires only knowledge of the pool member's address and their pending cancellation transaction.

**Likelihood: Low-Medium** — Requires mempool monitoring and transaction ordering, but no privileged access.

---

### Recommendation

Restrict `exit_delegation_pool_action` so that only the pool member themselves (or their reward address) can trigger it, similar to how `exit_delegation_pool_intent` is restricted to the pool member:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let caller = get_caller_address();
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    assert!(
        caller == pool_member || caller == pool_member_info.reward_address,
        "{}",
        Error::UNAUTHORIZED_CALLER,
    );
    // ... rest of function
}
```

Alternatively, document this as intentional behavior with an explicit comment, as was done in the referenced original report.

### Proof of Concept

1. Pool member `Alice` calls `exit_delegation_pool_intent(amount: 100)`. The `exit_wait_window` starts.
2. The `exit_wait_window` elapses. Alice's `unpool_time` is now in the past.
3. Alice changes her mind and broadcasts `exit_delegation_pool_intent(amount: 0)` to cancel.
4. Attacker `Bob` observes Alice's pending cancellation transaction in the mempool.
5. Bob submits `exit_delegation_pool_action(pool_member: Alice)` with higher priority.
6. Bob's transaction is included first: Alice's 100 tokens are transferred back to her address, her staking position is exited.
7. Alice's cancellation transaction then executes but finds `unpool_time = None` (already cleared), effectively becoming a no-op or reverting with `MISSING_UNDELEGATE_INTENT`.
8. Alice has lost her staking position and must re-enter the pool to resume earning rewards.

```text
        fn exit_delegation_pool_action(
            ref self: ContractState, pool_member: ContractAddress,
        ) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let unpool_time = pool_member_info
                .unpool_time
                .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
            assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberExitAction {
                        pool_member, unpool_amount: pool_member_info.unpool_amount,
                    },
                );

            // Perform removal action in the staking contract, receiving funds if needed.
            // Note that if the intent was done after the staker was removed (unstake_action),
            // the funds will already be in the pool contract, and the following call will do
            // nothing.
            let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
            staking_pool_dispatcher
                .remove_from_delegation_pool_action(identifier: pool_member.into());

            let unpool_amount = pool_member_info.unpool_amount;
            pool_member_info.unpool_amount = Zero::zero();
            pool_member_info.unpool_time = Option::None;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer delegated amount to the pool member.
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());

            unpool_amount
        }
```
