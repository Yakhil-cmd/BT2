### Title
Stale Program Cache on Access Node: `programInvalidator.ShouldInvalidateEntries()` Ignores `invalidated` Map, Silently Skipping Cache Invalidation for Specific Contract Updates — (`module/state_synchronization/indexer/util.go`)

---

### Summary

`programInvalidator.ShouldInvalidateEntries()` returns only `inv.invalidateAll`, completely ignoring the populated `inv.invalidated` map. Because `TableTransaction.AddInvalidator` uses `ShouldInvalidateEntries()` as a gate, when `invalidateAll=false` and `invalidated` is non-empty (the common case: a user contract was updated but no service-account transaction was present in the block), the invalidator is silently dropped. The program cache on the access node is never cleared for those contracts, and all subsequent script executions against the access node use the pre-update Cadence program indefinitely.

---

### Finding Description

**Root cause — `ShouldInvalidateEntries()` ignores `invalidated`:** [1](#0-0) 

```go
func (inv *programInvalidator) ShouldInvalidateEntries() bool {
    return inv.invalidateAll   // ← inv.invalidated is never consulted
}
```

`ShouldInvalidateEntry` (the per-entry check) is correctly implemented: [2](#0-1) 

```go
func (inv *programInvalidator) ShouldInvalidateEntry(...) bool {
    _, ok := inv.invalidated[location]
    return inv.invalidateAll || ok   // ← correct, but never reached
}
```

**Gate in `AddInvalidator` that silently drops the invalidator:** [3](#0-2) 

```go
func (txn *TableTransaction[TKey, TVal]) AddInvalidator(invalidator ...) {
    if invalidator == nil || !invalidator.ShouldInvalidateEntries() {
        return   // ← returns here when invalidateAll=false, even if invalidated is non-empty
    }
    ...
}
```

**Call site that constructs the invalidator:** [4](#0-3) 

```go
tx.AddInvalidator(&accessInvalidator{
    programs: &programInvalidator{
        invalidated:   updatedContracts,                              // populated
        invalidateAll: hasAuthorizedTransaction(collections, c.serviceAddress), // false when no service-account tx
    },
    ...
})
```

**Commit path that never fires invalidation:** [5](#0-4) 

Because `txn.invalidators` is empty (the invalidator was never appended), `txn.invalidators.ShouldInvalidateEntries()` returns `false` at commit time, so no cache entries are marked invalid and no entries are deleted from `table.items`.

**Exact scenario:**

1. Block N contains a user-deployed contract update for contract `C` (no service-account transaction → `invalidateAll=false`).
2. `findContractUpdates` correctly populates `updatedContracts = {C: {}}`.
3. `programInvalidator{invalidateAll:false, invalidated:{C:{}}}` is constructed.
4. `AddInvalidator` calls `ShouldInvalidateEntries()` → `false` → returns immediately.
5. `tx.Commit()` sees an empty invalidator list → does not touch the cache.
6. The cached entry for `C` remains live in `table.items` indefinitely.
7. Every subsequent script execution on the access node that imports `C` receives the pre-update compiled program.

---

### Impact Explanation

Access nodes execute Cadence scripts locally using the program cache. If a contract update removes or restricts a public function (e.g., one that previously returned private resource fields or bypassed a capability check), the stale cached program retains that function. Any script submitted to the access node can call it and read data or invoke capabilities that the contract owner explicitly revoked. Because `invalidateAll` is only `true` when a service-account transaction is present, ordinary user-contract updates — the common case — never trigger invalidation. The stale cache persists across all future blocks until a service-account transaction happens to appear, which may be arbitrarily far in the future or never on a given access node.

---

### Likelihood Explanation

Every block that contains a user contract update without a service-account transaction triggers this path. This is the normal case for application-layer contract upgrades on mainnet. No special attacker capability is required beyond the ability to submit a script to any public access node after a contract update has been sealed.

---

### Recommendation

Fix `ShouldInvalidateEntries()` to also return `true` when `inv.invalidated` is non-empty:

```go
func (inv *programInvalidator) ShouldInvalidateEntries() bool {
    return inv.invalidateAll || len(inv.invalidated) > 0
}
```

This mirrors the correct logic already present in `ShouldInvalidateEntry` and matches the analogous pattern used in the FVM's own `ProgramInvalidator`: [6](#0-5) 

```go
func (invalidator ProgramInvalidator) ShouldInvalidateEntries() bool {
    return invalidator.ExecutionParametersUpdated ||
        invalidator.ContractUpdates.Any()
}
```

### Proof of Concept

1. Deploy contract `C` with a public function `leak()` that returns a private field value.
2. Update `C` to remove `leak()` in a block that contains no service-account transaction (`hasAuthorizedTransaction` → `false`).
3. On the access node, `updateProgramCache` is called; `updatedContracts = {C:{}}`, `invalidateAll=false`.
4. `AddInvalidator` silently returns; cache entry for `C` is untouched.
5. Submit a script `import C from 0x...; pub fun main(): AnyStruct { return C.leak() }` to the access node.
6. The script executes against the stale program and returns the private field value, even though `leak()` no longer exists in the on-chain contract.
7. Verify by inspecting `DerivedDataTable.items` before and after step 2: the entry for `C` is still present and `isInvalid=false`.
