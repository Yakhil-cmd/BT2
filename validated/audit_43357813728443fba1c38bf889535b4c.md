### Title
Pool Admin Can Bypass `OracleValueStopLossExtension` Timelock via `uint32` Overflow in `_afterTimelock`, Silently Disabling LP Stop-Loss Protection — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` truncates `block.timestamp + timelock` to `uint32`. When the pool admin sets `timelock = type(uint32).max` (4,294,967,295 s ≈ 136 yr), the addition overflows `uint32` and wraps to a timestamp **already in the past**. Every subsequent `_requireElapsed` check passes immediately, letting the admin atomically propose-and-execute any stop-loss parameter change (drawdown, decay, watermarks) with **zero delay**, defeating the LP-protection guarantee the timelock exists to enforce.

---

### Finding Description

`_afterTimelock` computes the execution deadline for all timelocked stop-loss mutations:

```solidity
// OracleValueStopLossExtension.sol:297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space, then **silently truncated** to `uint32`. With `block.timestamp ≈ 1,753,000,000` (July 2026) and `timelock = type(uint32).max = 4,294,967,295`:

```
1,753,000,000 + 4,294,967,295 = 6,047,967,295
uint32(6,047,967,295) = 6,047,967,295 - 4,294,967,296 = 1,752,999,999
```

`executeAfter = 1,752,999,999 < block.timestamp = 1,753,000,000` → the elapsed check passes immediately.

`_requireElapsed` compares `uint256 block.timestamp` against the `uint32 executeAfter` (implicitly widened to `uint256`), so the wrapped value is never re-inflated:

```solidity
// OracleValueStopLossExtension.sol:301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

`proposeOracleStopLossTimelock` accepts any `uint32` value with **no upper-bound validation**:

```solidity
// OracleValueStopLossExtension.sol:78-84
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
}
```

Compare: `_validateDrawdown` and `_validateDecay` both enforce caps; `timelock` has none.

**Attack path:**

1. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`.
2. Waits for the *current* timelock to elapse (e.g., 3 days — the only window LPs have).
3. Calls `executeOracleStopLossTimelock(pool)` → `oracleStopLossConfig[pool].timelock = type(uint32).max`.
4. Immediately calls `proposeOracleStopLossDrawdown(pool, 0)` → `_afterTimelock` returns a past timestamp.
5. Immediately calls `executeOracleStopLossDrawdown(pool)` → `_requireElapsed` passes; `drawdownE6 = 0`.
6. Stop-loss is disabled. All subsequent swaps skip the watermark check (`if (drawdown == 0) return`).
7. Repeat for decay and watermarks with zero delay.

---

### Impact Explanation

`OracleValueStopLossExtension` is the sole on-chain mechanism guaranteeing that LP value-per-share cannot fall faster than `drawdown + decay × t`. Once the admin disables it (drawdown = 0) without notice, the pool can drain LP principal through adverse oracle moves or MEV with no circuit-breaker. This is a direct loss of LP assets: the extension's invariant — "value per share at oracle marks cannot fall faster than drawdown" — is silently broken. The admin-boundary break is concrete: the timelock that caps admin power is itself rendered ineffective by the overflow.

---

### Likelihood Explanation

The pool admin is a semi-trusted role. The attack requires only two transactions separated by the *current* timelock duration (which may be as short as 0 if initialized at zero). No external conditions, special tokens, or oracle manipulation are needed. Any pool using `OracleValueStopLossExtension` with a mutable admin is exposed.

---

### Recommendation

Add an upper-bound check in `proposeOracleStopLossTimelock` (and mirror it in `initialize`) to prevent overflow:

```solidity
uint32 constant MAX_TIMELOCK = 365 days; // or any safe bound

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    if (newTimelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(newTimelock);
    ...
}
```

Alternatively, perform the addition in `uint256` and store/compare as `uint256` throughout, eliminating the truncation entirely.

### Proof of Concept

```solidity
// Foundry test sketch
function test_timelockOverflowBypassesGuard() public {
    // Pool initialized with timelock = 0 (immediate execution allowed)
    vm.startPrank(admin);

    // Step 1: propose type(uint32).max as new timelock
    extension.proposeOracleStopLossTimelock(pool, type(uint32).max);
    // With current timelock = 0, executeAfter = uint32(block.timestamp) <= block.timestamp
    extension.executeOracleStopLossTimelock(pool);

    // Step 2: timelock is now type(uint32).max
    // _afterTimelock returns uint32(block.timestamp + type(uint32).max) which wraps to past

    // Step 3: propose drawdown = 0 (disable stop-loss)
    extension.proposeOracleStopLossDrawdown(pool, 0);
    // executeAfter is in the past → no warp needed
    extension.executeOracleStopLossDrawdown(pool);

    vm.stopPrank();

    (uint32 dd,,,) = extension.oracleStopLossConfig(pool);
    assertEq(dd, 0); // stop-loss disabled with zero LP notice
}
```
