### Title
Unbounded `pool_member_epoch_balance` Trace Loop in `calculate_rewards` Enables Unbounded Gas Consumption - (File: `src/pool/pool.cairo`)

### Summary
The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a delegator's entire `pool_member_epoch_balance` trace. There is no cap on how many entries can accumulate in this trace. A delegator who makes balance changes across many epochs without claiming rewards will cause `claim_rewards` to iterate over an ever-growing number of storage reads, eventually making the transaction fail due to gas exhaustion and permanently freezing their unclaimed yield.

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to `pool_member_trace_length`: [1](#0-0) 

The developers themselves annotate this loop as unbounded: [2](#0-1) 

The trace grows via `set_member_balance`, which inserts a new checkpoint at key `current_epoch + K` whenever the key differs from the last entry: [3](#0-2) 

`set_member_balance` is called by both `increase_member_balance` (used in `add_to_delegation_pool`) and directly in `exit_delegation_pool_intent`: [4](#0-3) [5](#0-4) 

Because the trace insert only deduplicates within the same epoch key, each epoch in which a delegator modifies their balance adds exactly one new entry. There is no maximum trace length enforced anywhere.

The `entry_to_claim_from` cursor is stored in `pool_member_info` and advanced after each successful `claim_rewards` call. If a delegator never claims (or claims infrequently), the loop must iterate over all accumulated entries since the last claim. Over a sufficiently long period of active balance changes without claiming, the loop's gas cost grows without bound.

### Impact Explanation

A delegator who makes one balance change per epoch and defers claiming for a large number of epochs will eventually trigger a `claim_rewards` call whose gas cost exceeds Starknet's per-transaction gas limit. At that point, `claim_rewards` will always revert, permanently freezing the delegator's accumulated unclaimed yield with no recovery path (since the only way to reduce the loop range is to successfully execute `claim_rewards`).

This maps to: **Permanent freezing of unclaimed yield** (High) and **Unbounded gas consumption** (Medium).

### Likelihood Explanation

Likelihood is **low**. The trace grows by at most one entry per epoch. At typical epoch lengths (e.g., one week), reaching a trace size large enough to exhaust Starknet's gas budget would require hundreds to thousands of epochs of continuous activity without claiming. However:

- The developers explicitly acknowledge the loop is unbounded.
- No protocol-level cap exists.
- The condition is entirely self-inflicted by a delegator who simply never calls `claim_rewards`, which is a realistic user behavior (e.g., a long-term passive delegator).
- Once the threshold is crossed, the freeze is permanent and unrecoverable.

### Recommendation

1. Enforce a maximum trace length per pool member, analogous to the `MAX_MSG_SIZE` pattern in the reference report. Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` calls that would push the trace beyond a safe bound.
2. Alternatively, cap the number of entries processed per `claim_rewards` call (partial claiming), advancing `entry_to_claim_from` by at most `MAX_ENTRIES_PER_CLAIM` per invocation, so the function can always make progress.
3. Add a constant such as `MAX_BALANCE_TRACE_ENTRIES: u64 = <safe_limit>` and assert it in `set_member_balance`.

### Proof of Concept

1. Delegator calls `enter_delegation_pool` in epoch 0 → trace length = 1.
2. Each subsequent epoch, delegator calls `add_to_delegation_pool` with a minimal amount → one new entry appended per epoch via `set_member_balance` → `pool_member_epoch_balance` trace grows by 1.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After `N` epochs, `claim_rewards` triggers `calculate_rewards`, which loops over all `N` entries, each requiring a storage read from `pool_member_epoch_balance`.
5. For sufficiently large `N`, the transaction exceeds the gas limit and reverts. All accumulated rewards are permanently inaccessible.

The root cause — no size cap on `pool_member_epoch_balance` and an unbounded loop in `calculate_rewards` — is directly analogous to the missing `MAX_MSG_SIZE` check in the reference report. [6](#0-5) [7](#0-6)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }

#[generate_trait]
pub impl MutablePoolMemberBalanceTraceImpl of MutablePoolMemberBalanceTraceTrait {
    /// Inserts a (`key`, `value`) pair into a Trace so that it is stored as the checkpoint.
    /// This is done by either inserting a new checkpoint, or updating the last one.
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
