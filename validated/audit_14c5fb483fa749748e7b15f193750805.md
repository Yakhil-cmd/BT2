Audit Report

## Title
Self-Address Guard Bypassed via P2SH Address with Identical Hash — (`rs/dogecoin/ckdoge/minter/src/lib.rs`, `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

The ckDOGE minter's self-address guard in `retrieve_btc_with_approval` compares the caller-supplied destination address against the minter's own address using plain string equality. Because the minter's address is always P2PKH (prefix byte 30, base58 string starting with `D`), an attacker can construct a P2SH address (prefix byte 22, base58 string starting with `A`) encoding the same 20-byte hash, pass it as the withdrawal destination, bypass the guard, and cause the minter to permanently lock DOGE at an address it can never spend.

## Finding Description

**Step 1 — String-only guard.**

`retrieve_btc_with_approval` in `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` performs a single string equality check:

```rust
let main_address_str = state::read_state(|s| runtime.derive_minter_address_str(s));
if args.address == main_address_str {
    ic_cdk::trap("illegal retrieve_btc target");
}
``` [1](#0-0) 

**Step 2 — Minter address is always P2PKH.**

`derive_minter_address_str` (lib.rs L223–231) calls `account_to_p2pkh_address_from_state`, which calls `account_to_p2pkh_address`, which calls `p2pkh_from_public_key` — always returning `DogecoinAddress::P2pkh(hash160(pubkey))`. [2](#0-1) [3](#0-2) [4](#0-3) 

`display()` encodes it with `DOGE_MAINNET_P2PKH_PREFIX = 30`, producing a string starting with `D` on mainnet. [5](#0-4) 

**Step 3 — P2SH with the same hash produces a different string.**

`DOGE_MAINNET_P2SH_PREFIX = 22` produces a base58 string starting with `A`. A P2SH address built from the same 20-byte hash H is a completely different string; the `==` comparison returns `false` and the guard does not fire. [6](#0-5) 

**Step 4 — `parse_address` accepts P2SH addresses.**

The ckDOGE runtime's `parse_address` accepts both P2PKH and P2SH addresses and maps them to `BitcoinAddress::P2sh(bytes)`. [7](#0-6) 

**Step 5 — Transaction builder emits a P2SH output.**

`into_bitcoin_transaction` emits `ScriptBuf::new_p2sh` for `BitcoinAddress::P2sh`, sending DOGE to the P2SH address on-chain. [8](#0-7) 

**Step 6 — Minter cannot spend from that P2SH address.**

The transaction signer only produces P2PKH `script_sig` (signature + compressed public key). Spending a P2SH UTXO requires a redeem script whose `hash160` equals H. The minter holds only the ECDSA private key for the public key whose `hash160` is H; it has no redeem script and the assert at line 60–63 confirms only P2PKH inputs are ever signed. [9](#0-8) 

## Impact Explanation

Every successful exploit burns the caller's ckDOGE and causes the minter to broadcast a transaction sending the corresponding DOGE to a P2SH address it can never spend. The DOGE is permanently unspendable on-chain. The minter's spendable UTXO pool shrinks without any possibility of recovery or consolidation. Repeated calls proportionally drain the minter's spendable DOGE reserve, eventually preventing all legitimate withdrawals. This constitutes a significant, concrete, permanent loss of in-scope chain-key/ck-token assets — matching the **High** impact class: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The attack requires no privilege. Any non-anonymous principal holding ckDOGE with an ICRC-2 approval can execute it in a single `retrieve_doge_with_approval` call. The minter's P2PKH address is publicly queryable via `get_doge_address`. Constructing the corresponding P2SH address is trivial: decode the base58 P2PKH address, swap the version byte from 30 to 22, recompute the double-SHA256 checksum, and re-encode. No off-chain infrastructure, no privileged access, and no threshold corruption is required. The attack is repeatable up to the attacker's ckDOGE balance.

## Recommendation

Replace the string equality guard with a structural comparison. After parsing `args.address` into a `BitcoinAddress`, compare it against the parsed minter address returned by `derive_minter_address` (which already exists and returns `BitcoinAddress`):

```rust
let minter_address = state::read_state(|s| runtime.derive_minter_address(s));
let parsed_address = runtime
    .parse_address(&args.address, btc_network)
    .map_err(RetrieveBtcWithApprovalError::MalformedAddress)?;
if parsed_address == minter_address {
    ic_cdk::trap("illegal retrieve_btc target");
}
```

This rejects any address — regardless of type — whose underlying 20-byte hash matches the minter's hash, because `BitcoinAddress::P2pkh(H) != BitcoinAddress::P2sh(H)` structurally but both share the same hash, and the comparison should be on the hash bytes directly (or the guard should check `parsed_address.script_hash() == minter_address.script_hash()`).

## Proof of Concept

```rust
// 1. Query the minter's P2PKH address, e.g. "DPubKeyHashXXX..."
let p2pkh_str = minter.get_doge_address(...);

// 2. Decode and extract the 20-byte hash H
let decoded = bs58::decode(&p2pkh_str).into_vec().unwrap(); // 25 bytes
let hash: [u8; 20] = decoded[1..21].try_into().unwrap();

// 3. Re-encode as P2SH (version byte 22 for mainnet)
let mut payload = vec![22u8];
payload.extend_from_slice(&hash);
let checksum = sha256d(&payload)[..4].to_vec();
payload.extend_from_slice(&checksum);
let p2sh_str = bs58::encode(&payload).into_string();

// 4. p2sh_str != p2pkh_str  →  guard does not fire
// 5. DogecoinAddress::parse(p2sh_str, Mainnet) → Ok(P2sh(H))
// 6. Call retrieve_doge_with_approval({amount, address: p2sh_str, from_subaccount: None})
//    → ckDOGE burned, DOGE sent to P2SH address, permanently locked.
assert_ne!(p2sh_str, p2pkh_str);
assert!(DogecoinAddress::parse(&p2sh_str, &Network::Mainnet).is_ok());
```

A deterministic integration test using PocketIC can confirm: (a) the guard does not trap on the P2SH address, (b) the burn succeeds, (c) the resulting transaction output is a P2SH script, and (d) no signing path in `DogecoinTransactionSigner` can produce a valid `script_sig` for that P2SH UTXO.
