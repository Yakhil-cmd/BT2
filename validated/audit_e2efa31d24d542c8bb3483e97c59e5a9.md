### Title
Pool Admin Can Permanently Block Protocol Fee Collection via Reverting `adminFeeDestination` — (`metric-core/contracts/MetricOmmPoolFactory.sol`, `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`setPoolAdminFeeDestination` accepts any non-zero address with no timelock and no transfer-ability check. If the pool admin sets the destination to a contract that reverts on ERC20 `transfer`, every subsequent call to `collectFees` reverts before the protocol share is transferred, permanently locking both admin and protocol fee revenue in the pool.

---

### Finding Description

**Entry point — no timelock, no validation beyond non-zero:**

`setPoolAdminFeeDestination` stores any non-zero address immediately:

```solidity
if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
poolAdminFeeDestination[pool] = newAdminFeeDestination;
``` [1](#0-0) 

There is no timelock, no two-step confirmation, and no check that the address can receive tokens.

**Transfer ordering in `collectFees` — admin first, protocol second, reset last:**

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // ← reverts here
}
...
if (totalFee0ToProtocol > 0) {
    transferToken0(FACTORY, totalFee0ToProtocol);             // ← never reached
}
...
notionalFeeToken0Scaled = 0;                                  // ← never reached
notionalFeeToken1Scaled = 0;
``` [2](#0-1) 

`transferToken0` / `transferToken1` use `safeTransfer`, which propagates the revert from the recipient. [3](#0-2) 

Because the admin transfer is attempted first and the notional accumulator reset (`notionalFeeToken0Scaled = 0`) is placed after all four transfers, a revert on the admin leg:

1. Prevents the protocol from receiving its share.
2. Leaves `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` non-zero, so every future `collectFees` call re-enters the same revert path.

**All three `collectFees` call sites are affected:**

- `collectPoolFees` (permissionless keeper path) [4](#0-3) 
- `setPoolAdminFees` (admin fee-change path) [5](#0-4) 
- `setPoolProtocolFee` (owner fee-change path) [6](#0-5) 

All three pass `poolAdminFeeDestination[pool]` directly. The factory owner has no function to override `poolAdminFeeDestination`; only the pool admin can change it. [7](#0-6) 

---

### Impact Explanation

- Protocol fee revenue for the affected pool is permanently frozen — the factory owner cannot collect it via `collectTokens` because the tokens never leave the pool.
- `setPoolProtocolFee` (owner-only) also calls `collectFees` first; it too reverts, so the owner cannot even update protocol fee rates for the pool.
- The pool's swap and liquidity functions remain operational, so fees continue to accumulate in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` with no path to extraction.
- Recovery requires the malicious pool admin to voluntarily restore a valid destination — the factory owner has no override.

This satisfies the allowed impact gate: **direct loss of protocol fee revenue** and **broken core pool functionality** (fee collection).

---

### Likelihood Explanation

The pool admin is explicitly classified as **semi-trusted only inside caps and timelocks** (Smart Audit Pivots, admin path). `setPoolAdminFeeDestination` has neither a cap nor a timelock, placing it outside the boundary where the admin is trusted. A compromised or adversarial pool admin key is a realistic post-deployment scenario; the two-step admin transfer mechanism means the admin role can change hands after creation.

---

### Recommendation

Two independent fixes, either sufficient:

1. **Separate the transfers**: Move the protocol transfers before or independently of the admin transfer, so a reverting admin destination cannot block the protocol leg. Reset `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` after the protocol transfers regardless of admin transfer success.

2. **Add a timelock to `setPoolAdminFeeDestination`**: Mirror the price-provider rotation pattern — require a proposal + delay before the new destination takes effect, giving the protocol time to detect and respond to a malicious change.

---

### Proof of Concept

```solidity
// 1. Deploy pool normally; admin is attacker.
// 2. Deploy RevertOnTransfer — a contract whose fallback/receive and
//    ERC20 transfer hook always revert.
// 3. Attacker calls:
factory.setPoolAdminFeeDestination(pool, address(revertOnTransfer));

// 4. Run swaps to accumulate notionalFeeToken0Scaled > 0.
pool.swap(...);

// 5. Anyone (keeper, protocol) calls:
factory.collectPoolFees(pool);
// → collectFees → transferToken0(revertOnTransfer, adminAmt) → REVERT
// Protocol receives nothing; notionalFeeToken0Scaled unchanged.

// 6. Factory owner attempts recovery:
factory.setPoolProtocolFee(pool, newRate, newRate);
// → also calls collectFees internally → same REVERT
// Owner cannot update rates or collect fees.

// Assert: IERC20(token0).balanceOf(address(factory)) == 0 (no protocol fees received).
// Assert: pool.notionalFeeToken0Scaled() == original non-zero value (accumulator stuck).
```
