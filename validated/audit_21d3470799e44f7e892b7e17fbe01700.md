### Title
`addLiquidity()` and `removeLiquidity()` Missing `whenNotPaused` Guard Allows LP Fund Withdrawal During Emergency Pause - (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.addLiquidity()` and `removeLiquidity()` are missing the `whenNotPaused` modifier that is applied to `swap()`. When a pool is paused at level 1 (admin) or level 2 (protocol), any LP can still call `removeLiquidity()` to withdraw their principal, defeating the entire purpose of the pause mechanism.

---

### Finding Description

`MetricOmmPool` defines a three-level pause system stored in `pauseLevel` (0 = active, 1 = admin-paused, 2 = protocol-paused) and a `whenNotPaused` modifier that reverts when `pauseLevel != 0`. [1](#0-0) 

`swap()` correctly applies this guard: [2](#0-1) 

However, `addLiquidity()` and `removeLiquidity()` carry only `nonReentrant` — `whenNotPaused` is absent from both: [3](#0-2) [4](#0-3) 

The factory's pause transitions confirm that pausing is an emergency control: `pausePool()` (admin) moves 0→1, `protocolPausePool()` (owner) moves 0/1→2, and critically `protocolUnpausePool()` can only move 2→1 (not directly to 0), meaning a protocol-level pause is a serious, multi-step recovery event. [5](#0-4) 

Despite this, any LP can call `removeLiquidity()` at any pause level and receive token0/token1 from the pool's bins.

---

### Impact Explanation

The pause mechanism is the protocol's primary emergency brake. Its purpose is to freeze all pool state changes — including LP withdrawals — while an incident (oracle failure, price manipulation, accounting exploit) is investigated and resolved. Because `removeLiquidity()` bypasses the pause, LPs can drain their principal from a paused pool before the team can assess or remediate the situation. This constitutes broken core pool functionality and direct loss of LP assets that the pause was intended to protect.

---

### Likelihood Explanation

Any pause event (admin or protocol) immediately exposes this path. No special privilege is required — any LP with an existing position can call `removeLiquidity()` the moment a pause is set. The factory's pause functions are callable by the pool admin (semi-trusted) or the protocol owner, both of whom are expected to act in genuine emergencies, making the trigger realistic.

---

### Recommendation

Add `whenNotPaused` to both `addLiquidity()` and `removeLiquidity()`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

This mirrors the guard already applied to `swap()` and ensures the pause level uniformly halts all state-changing pool operations.

---

### Proof of Concept

1. Pool is active (`pauseLevel == 0`). Alice adds liquidity via `addLiquidity()`.
2. An oracle manipulation or accounting exploit is detected. The pool admin calls `pausePool(pool)` → `pauseLevel` becomes 1.
3. `swap()` now reverts with `PoolPaused`.
4. Alice calls `removeLiquidity()` with her full share position. The function has no `whenNotPaused` check, executes `LiquidityLib.removeLiquidity()`, and transfers token0/token1 out of the pool.
5. The pool's bin balances are reduced while the incident is still open, removing assets the protocol intended to freeze. [6](#0-5)
