### Title
Unchecked Exit Precompile Call Return Value Causes Silent Token Burn Without Bridge Withdrawal - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary
In both `EvmErc20.sol` and `EvmErc20V2.sol`, the `withdrawToNear` and `withdrawToEthereum` functions burn the caller's ERC-20 tokens first, then invoke the Aurora exit precompile via inline assembly. The return value of the `call` opcode is captured in `res` but is never checked. If the precompile call fails, the burn is committed and the transaction succeeds, permanently destroying the user's tokens with no corresponding NEP-141 release on the NEAR side.

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` follow this pattern:

```solidity
_burn(_msgSender(), amount);   // tokens destroyed — irreversible

assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    // res is NEVER checked — silent failure is possible
}
``` [1](#0-0) [2](#0-1) 

The identical pattern is present in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The exit precompiles (`ExitToNear` at `0xe9217bc7...` and `ExitToEthereum` at `0xb0bd02f6...`) can return failure (EVM `call` returns `0`) for multiple documented reasons in `engine-precompiles/src/native.rs`:

- `ERR_TARGET_TOKEN_NOT_FOUND` — the NEP-141 ↔ ERC-20 mapping is absent
- `ERR_INVALID_RECEIVER_ACCOUNT_ID` — malformed NEAR account ID in the recipient bytes
- `ERR_INVALID_INPUT` — input size out of bounds
- `ERR_PAUSED` — precompile is administratively paused [5](#0-4) [6](#0-5) 

When any of these conditions cause the precompile to return an error, the EVM `call` opcode returns `0`. Because `res` is never inspected and no `if iszero(res) { revert(0,0) }` guard exists, the outer transaction does not revert. The `_burn` that already executed is committed to state. The user's ERC-20 tokens are gone, and no NEAR-side `ft_transfer` or `withdraw` promise is ever scheduled.

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` with input that causes the precompile to fail (e.g., a recipient bytes payload that is syntactically valid Solidity but produces an invalid NEAR account ID inside the precompile) will have their ERC-20 tokens burned with no recovery path. The NEP-141 tokens on the NEAR side are never released. The total supply of the bridged asset on Aurora decreases without a corresponding credit on NEAR, constituting permanent, irrecoverable loss of user funds.

### Likelihood Explanation

**Medium.** Any token holder can trigger this by passing a recipient value that passes Solidity-level encoding but is rejected by the precompile's NEAR account ID parser (e.g., an account ID exceeding 64 bytes, containing invalid characters, or referencing a NEP-141 mapping that has not been registered). The `withdrawToNear` and `withdrawToEthereum` functions are public, externally callable, and require no special privilege. The failure mode is silent — the transaction emits no revert and the user receives no on-chain indication that the withdrawal failed.

### Recommendation

Add a return-value check in every assembly block that calls an exit precompile. Revert if the call fails so that the preceding `_burn` is rolled back:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to all four affected assembly blocks across `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures that if the precompile rejects the call for any reason, the burn is atomically rolled back and the user retains their tokens.

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) with a valid NEP-141 mapping registered.
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear(recipient_bytes, amount)` where `recipient_bytes` encodes a NEAR account ID that is syntactically valid UTF-8 but exceeds 64 characters (rejected by `parse_recipient` inside the precompile with `ERR_INVALID_RECEIVER_ACCOUNT_ID`).
4. The precompile call returns `0`; `res` is ignored; the function returns without reverting.
5. Observe: `alice`'s ERC-20 balance is reduced by `amount`; no `ExitToNear` promise is scheduled; no NEP-141 tokens are credited on NEAR. The tokens are permanently destroyed. [7](#0-6)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
