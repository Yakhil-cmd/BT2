Audit Report

## Title
ckETH Minter Indefinitely Queues Withdrawal Requests Without Reimbursement When Gas Fees Exceed Withdrawal Amount - (`File: rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

When `create_transactions_batch` encounters `CreateTransactionError::InsufficientTransactionFee` for a `WithdrawalRequest::CkEth` request, it calls `reschedule_withdrawal_request` to move the request to the back of the queue without ever scheduling a reimbursement. Because the user's ckETH was already irreversibly burned before the request was enqueued, and because no `FailedEthWithdrawalRequest` event type or equivalent reimbursement path exists for this case, a sustained Ethereum gas spike above the static `cketh_minimum_withdrawal_amount` causes the burned ckETH to be permanently unrecoverable.

## Finding Description

**Burn path** (`rs/ethereum/cketh/minter/src/main.rs`, lines 291–336): `withdraw_eth` checks `amount >= cketh_minimum_withdrawal_amount`, then calls `client.burn_from(...)`. On success, it emits `EventType::AcceptedEthWithdrawalRequest` and enqueues the request. The burn is irreversible at this point.

**Transaction creation path** (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 249–294): `create_transactions_batch` iterates pending requests and calls `create_transaction`. For `WithdrawalRequest::CkEth`, `create_transaction` computes `max_transaction_fee = gas_fee_estimate.to_price(gas_limit).max_transaction_fee()` and returns `Err(CreateTransactionError::InsufficientTransactionFee)` if `withdrawal_amount < max_transaction_fee` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 1122–1134).

**Missing reimbursement** (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 281–291): The error arm calls only `mutate_state(|s| s.eth_transactions.reschedule_withdrawal_request(request))`. `reschedule_withdrawal_request` (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`, lines 470–483) simply removes the request from its current queue position and pushes it to the back — no retry counter, no maximum age, no reimbursement event.

**No FailedEthWithdrawalRequest event type** (`rs/ethereum/cketh/minter/src/state/event.rs`, lines 17–174): The `EventType` enum contains `FailedErc20WithdrawalRequest` (line 140) but has no equivalent `FailedEthWithdrawalRequest`. The `process_reimbursement` function (`rs/ethereum/cketh/minter/src/withdraw.rs`, lines 46–95) only processes entries in `reimbursement_requests`, which are never populated for this case.

**Static minimum** (`rs/ethereum/cketh/minter/src/lifecycle/init.rs`, line 92): `cketh_minimum_withdrawal_amount` is set once at canister initialization from `InitArg.minimum_withdrawal_amount` and is not dynamically adjusted to track real-time gas fee estimates.

**Contrast with ckERC20** (`rs/ethereum/cketh/minter/src/main.rs`, lines 506–531): When the ckERC20 burn fails, the minter explicitly constructs a `ReimbursementRequest` and emits `EventType::FailedErc20WithdrawalRequest`, which `process_reimbursement` later processes to mint back the burned ckETH. No equivalent path exists for the ckETH `InsufficientTransactionFee` case.

## Impact Explanation

This is a **High** severity finding matching the allowed impact class: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."* A user's ckETH is burned on the IC ledger (permanently reducing their balance). No ETH is ever sent on Ethereum. No ckETH reimbursement is ever minted back. The loss is proportional to the user's withdrawal amount and is permanent as long as gas fees remain elevated. Any unprivileged IC principal can be affected; no attacker privilege is required to trigger the condition.

## Likelihood Explanation

The trigger requires Ethereum gas fees to exceed `cketh_minimum_withdrawal_amount` at the time `create_transactions_batch` runs. Ethereum gas spikes are a well-documented, recurring phenomenon (e.g., NFT mint events, network congestion). The minimum withdrawal amount provides a buffer but is a static deployment-time value. A user submitting a request at exactly the minimum amount during a gas spike — a realistic scenario — satisfies the condition. The condition persists for every timer tick until gas fees drop or the canister is upgraded with a higher minimum. Likelihood is **Medium**.

## Recommendation

1. **Immediate fix**: In the `Err(CreateTransactionError::InsufficientTransactionFee)` arm of `create_transactions_batch` (`withdraw.rs` lines 281–291), add a retry counter or maximum queue age to the `EthWithdrawalRequest`. Once the threshold is exceeded, emit a new `EventType::FailedEthWithdrawalRequest(ReimbursementRequest { ... })` event and remove the request from the queue, mirroring the `FailedErc20WithdrawalRequest` pattern in `main.rs` lines 516–530.
2. **Structural**: Extend `process_reimbursement` to handle a `ReimbursementIndex::CkEth` case that was triggered by `InsufficientTransactionFee` (distinct from a finalized-but-failed transaction), so the burned ckETH is minted back to the user.
3. **Preventive**: Dynamically gate `withdraw_eth` acceptance against the current gas fee estimate so the burn is rejected before it occurs if fees already exceed the withdrawal amount at acceptance time.

## Proof of Concept

1. Alice calls `withdraw_eth` with `amount = cketh_minimum_withdrawal_amount` (e.g., `30_000_000_000_000_000` wei). The minter burns this amount from Alice's ckETH ledger account (`main.rs` lines 301–312). Burn succeeds; `AcceptedEthWithdrawalRequest` is emitted.
2. A gas spike occurs. The timer fires `process_retrieve_eth_requests` → `create_transactions_batch`. `create_transaction` is called for Alice's request. `max_transaction_fee > withdrawal_amount` → returns `Err(CreateTransactionError::InsufficientTransactionFee)` (`transactions/mod.rs` lines 1125–1133).
3. The error arm at `withdraw.rs` lines 281–291 calls `reschedule_withdrawal_request`. No reimbursement event is emitted. Alice's burned ckETH is not returned.
4. Every subsequent timer tick repeats step 2–3. `process_reimbursement` is never invoked for Alice's request because no `ReimbursementRequest` was ever recorded in `reimbursement_requests`.
5. A deterministic integration test or PocketIC test can reproduce this by: (a) initializing the minter with a known `minimum_withdrawal_amount`; (b) mocking the gas fee estimate to return a value exceeding that amount; (c) calling `withdraw_eth` at the minimum; (d) triggering the timer; (e) asserting that Alice's ckETH balance remains reduced and no reimbursement mint occurred after N timer ticks.
