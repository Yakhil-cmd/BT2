### Title
Unbounded L1 Transaction Calldata Bypasses Native Resource Accounting via `FORMAL_INFINITE` Fallback — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

L1→L2 transactions carry no explicit calldata size limit enforced by ZKsync OS. When calldata is large enough to exhaust native resources during transaction-hash computation, the bootloader silently falls back to `FORMAL_INFINITE` resources and recomputes the keccak hash outside of any resource budget. The prover must still perform O(calldata\_size) keccak work, but that work is invisible to the native resource accounting system, creating a resource accounting bug and a valid-execution unprovability path.

---

### Finding Description

**Step 1 — No calldata size limit for L1 transactions.**

`AbiEncodedTransaction::validate_structure` enforces that `signature` is empty, `paymaster_input` is empty, and `reserved_dynamic` is empty for L1/upgrade transactions, but places **no bound on the `data` (calldata) field**. [1](#0-0) 

The test-layer struct comment confirms this is intentional: `"data: An unlimited size byte array specifying the input data of the message call."` [2](#0-1) 

**Step 2 — L1 transaction hash computation over the full buffer.**

`l1_tx_calculate_hash` calls `charge_keccak(32 + self.underlying_buffer.len(), resources)` and then hashes the entire buffer. For a ~1.875 MB calldata payload (the practical Ethereum block-gas-limit ceiling), the native cost is approximately `(1_875_032 / 136) * 17_500 + 2_500 ≈ 241 million` native units — well above `MAX_NATIVE_COMPUTATIONAL`. [3](#0-2) 

**Step 3 — `FORMAL_INFINITE` fallback silently bypasses resource accounting.**

In `process_l1_transaction`, when `calculate_hash` fails because native resources are exhausted, the code logs the event, exhausts ergs, and **recomputes the hash with `S::Resources::FORMAL_INFINITE`**. The keccak computation over the full buffer still executes in the prover, but the native resource tracker never sees it. [4](#0-3) 

**Step 4 — L1ResourcesPolicy saturates rather than rejects.**

Even if the gas limit is far below the intrinsic cost for large calldata, `L1ResourcesPolicy::handle_arithmetic_error` logs and saturates to `0` instead of returning a validation error, so the transaction is never rejected at the resource-creation stage. [5](#0-4) 

---

### Impact Explanation

The RISC-V prover binary has a fixed cycle budget. When a block contains an L1 transaction with maximum-size calldata, the prover must execute the full keccak loop over the large buffer (inside the `FORMAL_INFINITE` fallback path) without that work being reflected in native resource accounting. This can cause the prover to exceed its actual cycle budget, making the block **unprovable** even though the forward (sequencer) execution succeeded. The result is a **forward/proving divergence** and **valid-execution unprovability**: the sequencer accepts and finalises the block, but no ZK proof can be generated, halting the chain.

---

### Likelihood Explanation

Any unprivileged user can submit an L1→L2 priority transaction through the L1 bridge contract with calldata up to the Ethereum block gas limit (~1.875 MB at 16 gas/non-zero byte). No special privilege is required. The L1 bridge enforces payment in ETH, but the cost of a single large-calldata transaction is modest compared to the damage of halting the proving pipeline. The attack is therefore realistic and low-cost relative to impact.

---

### Recommendation

1. **Enforce an explicit calldata size cap for L1 transactions inside `validate_structure`** (or in `process_l1_transaction` before hash computation), analogous to `MAX_INITCODE_SIZE` for deployment transactions. A reasonable limit is the maximum calldata that can be hashed within the native resource budget allocated to L1 transaction intrinsic work.

2. **Remove the `FORMAL_INFINITE` fallback for hash computation** or, if it must be retained for chain-liveness reasons, ensure the native cost of hashing is pre-charged from a dedicated, bounded budget before the fallback is triggered, so the prover's actual work is always accounted for.

3. **Add a prover-side assertion** that the total native resources consumed by L1 transaction intrinsic work never exceed `MAX_NATIVE_COMPUTATIONAL`, failing fast with a clear error rather than silently overrunning the budget.

---

### Proof of Concept

1. Construct an L1→L2 transaction (`tx_type = 0x7F`) with `data` set to `vec![0xAA; 1_875_000]` (≈1.875 MB of non-zero calldata) and `gas_limit = 0`.
2. Submit it through the L1 bridge; it enters the priority queue.
3. The sequencer (forward mode) processes it: `prepare_and_check_resources` saturates to 0 via `L1ResourcesPolicy`; `calculate_hash` exhausts native resources; the fallback recomputes the hash with `FORMAL_INFINITE`; `preparation_out_of_resources = true`; execution is skipped; the transaction is recorded as failed but the block is finalised.
4. The prover (RISC-V mode) attempts to reproduce the same execution. At the `FORMAL_INFINITE` fallback, it must execute `keccak256` over ~1.875 MB. The native resource tracker shows no charge for this work, but the RISC-V cycle counter advances by ~13,787 keccak rounds × (cycles per round). The prover exceeds its cycle budget and cannot generate a proof.
5. The block is finalised on the sequencer but unprovable, halting the ZK rollup. [4](#0-3) [3](#0-2) [5](#0-4)

    fn l1_tx_calculate_hash<R: Resources>(&self, resources: &mut R) -> Result<[u8; 32], TxError> {
        charge_keccak(32 + self.underlying_buffer.len(), resources)?;
        let mut hasher = Keccak256::new();
        // Note, that the correct ABI encoding of the Transaction structure starts with 0x20
        hasher.update(&U256::from(0x20).to_be_bytes::<32>());
        hasher.update(&self.underlying_buffer.as_slice());
        Ok(hasher.finalize())
    }
