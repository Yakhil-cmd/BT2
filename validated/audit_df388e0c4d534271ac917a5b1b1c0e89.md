### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Silently Overcharges Users When Native Cost < Gas Cost - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

`compute_gas_refund` adjusts `gas_used` upward when native resource consumption exceeds gas consumption (`delta_gas > 0`), but never adjusts it downward when native consumption is lower than gas consumption (`delta_gas < 0`). A developer-acknowledged `TODO` comment marks the missing branch. The result is that every transaction whose native resource footprint is lighter than its EVM gas footprint pays more in fees than it should, with the surplus silently redirected to the operator.

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`, `compute_gas_refund` reconciles EVM gas accounting with the ZKsync-native resource model:

```rust
#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };

    if delta_gas > 0 {
        // native cost > gas cost → increase gas_used (user pays more)
        gas_used += delta_gas as u64;
    }
    // TODO: return delta_gas to gas_used?   ← missing else branch
}
``` [1](#0-0) 

`delta_gas` is defined as `(native_used / native_per_gas) - gas_used`. When this value is **negative** (native cost < gas cost), the code takes no action and `gas_used` remains at its higher EVM-derived value. The developer-left `TODO` on line 80 explicitly acknowledges the missing symmetric adjustment.

The final `gas_used` value is then consumed in two places:

1. **User refund** (ZK flow): `token_to_refund = gas_price * (tx_gas_limit - gas_used)` — a higher `gas_used` means a smaller refund.
2. **Operator payment** (ZK flow): `token_to_pay_operator = gas_used * gas_price_for_operator` — a higher `gas_used` means a larger operator payment. [2](#0-1) 

The same `gas_used` from `RefundInfo` drives the L1-transaction operator payment:

```rust
let pay_to_operator = U256::from(gas_used)
    .checked_mul(U256::from(gas_price))
    ...
``` [3](#0-2) 

### Impact Explanation

When `delta_gas < 0` (native resource consumption is lower than EVM gas consumption), `gas_used` is not reduced to reflect the actual cost. The difference `|delta_gas| * gas_price` is:

- **Withheld from the user's refund** — the user receives fewer tokens back than they are owed.
- **Transferred to the operator** — the operator is paid for gas units that were never consumed.

This is a resource accounting bug that causes a systematic, silent transfer of user funds to the operator on every transaction where native resource usage is lighter than EVM gas usage (e.g., computation-heavy but storage-light transactions). The magnitude scales with `gas_price * |delta_gas|` per transaction.

### Likelihood Explanation

The `delta_gas < 0` path is reachable by any unprivileged L2 or L1→L2 transaction sender. Transactions that perform heavy EVM computation (loops, hashing) but touch few storage slots will naturally have `native_used / native_per_gas < gas_used`. No special privileges, governance access, or external oracle manipulation are required. The `#[cfg(not(feature = "unlimited_native"))]` guard means this path is active in the standard (non-unlimited-native) build configuration used in production.

### Recommendation

Apply the symmetric downward adjustment when `delta_gas < 0`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // Native cost is lower than gas cost; return the difference to the user.
    gas_used = gas_used.saturating_sub((-delta_gas) as u64);
    // Enforce the minimal gas floor after adjustment.
    gas_used = core::cmp::max(gas_used, minimal_gas_used);
}
```

The `minimal_gas_used` floor must be re-applied after the downward adjustment to preserve the existing invariant that validation costs are always charged.

### Proof of Concept

1. Submit a ZK L2 transaction with a high `gas_limit` (e.g., 1,000,000 gas) and a `gas_price` of 1 gwei.
2. The transaction body performs heavy EVM computation (many arithmetic opcodes) but writes to zero storage slots, resulting in low `native_used`.
3. After execution, `compute_gas_refund` computes `delta_gas = (native_used / native_per_gas) - gas_used`. Because native consumption is low, `delta_gas < 0`.
4. The `if delta_gas > 0` branch is not taken; `gas_used` remains at the higher EVM-derived value.
5. `refund_and_commit_fee` computes `token_to_refund = gas_price * (gas_limit - gas_used)` — the user receives a smaller refund than owed.
6. The operator receives `gas_used * gas_price` — more than the actual resource cost of the transaction.
7. The difference `|delta_gas| * gas_price` has been silently transferred from the user to the operator with no corresponding resource consumption. [4](#0-3)
