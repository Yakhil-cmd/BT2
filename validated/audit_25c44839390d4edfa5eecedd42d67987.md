### Title
Malicious Participant Sends Malformed Message at `sign_waitpoint`, Permanently Aborting Coordinator's Signing Session — (`src/frost/redjubjub/sign.rs`)

### Summary
A single malicious participant included in the signing set can permanently abort the coordinator's FROST/RedJubjub signing session by sending a message with a valid protocol header but an invalid (malformed) body at the `sign_waitpoint`. The deserialization error propagates unconditionally through `recv_from_others` → `do_sign_coordinator`, with no per-sender error isolation, retry, or exclusion mechanism.

### Finding Description

The coordinator's `do_sign_coordinator` collects signature shares via `recv_from_others::<SignatureShare>` at the `sign_waitpoint`: [1](#0-0) 

`recv_from_others` loops calling `chan.recv(waitpoint).await?`, propagating any error immediately: [2](#0-1) 

`chan.recv` deserializes the raw bytes with `rmp_serde`. If deserialization fails, it returns `Err(ProtocolError::DeserializationError(...))`: [3](#0-2) 

The `?` at line 20 of `helpers.rs` propagates this error out of the entire `while !seen.full()` loop before the sender-identity check (`seen.put(from)`) is ever reached. The `?` at line 165 of `sign.rs` then propagates it out of `do_sign_coordinator`, permanently aborting the session.

The `push_message` function only validates the `MessageHeader` prefix; it does not validate the message body type: [4](#0-3) 

A malicious participant who knows the protocol structure (and therefore the `sign_waitpoint` counter value) can craft a message with a valid header but a body that is not a valid `SignatureShare` (e.g., a serialized `u64`). This message passes `push_message`'s header check, enters the incoming buffer, and causes `rmp_serde` to fail on deserialization.

### Impact Explanation
**High — Permanent denial of signing.** A single malicious participant in the signing set can unconditionally abort every signing attempt that includes them. There is no retry path, no per-sender error skip, and no mechanism to exclude the offending participant and re-run with the remaining honest parties. The `ProtocolError::DeserializationError` is terminal.

### Likelihood Explanation
Any participant in the signing set can execute this attack with trivial effort: send one message with a valid header and garbage body bytes. No cryptographic capability is required. The attack is repeatable across every signing invocation that includes the malicious participant.

### Recommendation
In `recv_from_others`, catch deserialization errors per-sender rather than propagating them immediately. Options include:
- Skip (and optionally log) messages that fail to deserialize, treating the sender as having sent nothing (allowing the loop to continue waiting for a valid message or timing out).
- Return a structured error that identifies the offending participant, allowing the caller to exclude them and retry with the remaining honest set.

The `chan.recv` function in `internal.rs` should also be considered for a variant that returns `Option<T>` or a per-message `Result` so callers can decide how to handle individual deserialization failures without aborting the entire round.
