### Title
Gateway Admits Signature-Bypassed Invoke Transactions via Overly Broad `skip_stateful_validations` Check - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the account's `__validate__` entry point (signature verification) for nonce=1 invoke transactions when `account_tx_in_pool_or_recent_block` returns true. An attacker who observes a pending `deploy_account` for a target account can submit a fake-signature nonce=1 invoke that is admitted to the mempool without any signature check, violating the admission invariant that all accepted transactions must carry a valid account signature.

### Finding Description

In `skip_stateful_validations` at lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the gateway skips `__validate__` when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage` — without ever calling `__validate__`: [3](#0-2) 

`perform_pre_validation_stage` only checks nonce ordering, fee bounds, and proof facts — it does **not** verify the account signature: [4](#0-3) 

The code comment in `skip_stateful_validations` states: *"it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This assumption is correct for an undeployed account in isolation, but it ignores the **race window**: once a legitimate `deploy_account` is visible in the mempool, any third party can submit a nonce=1 invoke with a fabricated signature before the legitimate owner does.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and fee escalation — it performs no signature check: [5](#0-4) 

### Impact Explanation

The gateway admits an invalid transaction — one whose signature has never been verified — into the mempool. Concretely:

- The attacker's fake nonce=1 invoke occupies the nonce=1 slot for the target account.
- The legitimate owner's nonce=1 invoke is subsequently rejected as a duplicate nonce, or the owner is forced to pay a higher fee to displace the fake transaction via fee escalation.
- The fake invoke eventually fails at batcher execution time when the blockifier runs `__validate__` (non-revertible failure, no nonce increment), but the damage to the admission queue has already occurred.

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

### Likelihood Explanation

The attack requires:
1. Observing a pending `deploy_account` in the public mempool (zero privilege required).
2. Deriving the target account address from the `deploy_account` fields (deterministic, public).
3. Submitting a fake nonce=1 invoke before the legitimate owner does (a race window that can span seconds to minutes depending on user behaviour).
4. The target account must be pre-funded (standard practice before deploying an account).

All four conditions are routinely satisfied in normal Starknet account-creation flows.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a specific `has_pending_deploy_account_in_pool(address)` query that returns `true` only when a `deploy_account` transaction — not an arbitrary transaction — is present for the given address. This closes the race window without breaking the intended UX for simultaneous `deploy_account + invoke` submission.

### Proof of Concept

```
1. Account X pre-funds address A (sends STRK to A).
2. X submits deploy_account for A (passes __validate_deploy__, enters mempool).
3. Attacker observes deploy_account in mempool, derives A.
4. Attacker constructs Invoke V3 from A, nonce=1, arbitrary calldata,
   valid resource bounds (≥ min_gas_price), signature = [0x41, 0x41] (garbage).
5. Gateway stateless check: passes (valid address, non-zero bounds, size OK).
6. TransactionConverter: calculates tx_hash, no signature check.
7. StatefulTransactionValidator::extract_state_nonce_and_run_validations:
     - account_nonce = 0  (A not deployed)
     - validate_state_preconditions: nonce 0 ≤ 1 ≤ max_allowed_nonce_gap → OK
                                     verify_can_pay_committed_bounds → OK (A is funded)
     - validate_by_mempool: no nonce=1 tx in pool yet → OK
     - skip_stateful_validations: nonce==1 && account_nonce==0
                                  && account_tx_in_pool_or_recent_block(A)==true
                                  → returns true (SKIP __validate__)
     - run_validate_entry_point(validate=false): returns Ok(()) immediately
8. Fake invoke is added to mempool at nonce=1 for account A.
9. X submits legitimate nonce=1 invoke → MempoolError::DuplicateNonce
   (or forced to pay higher fee to escalate past the fake tx).
``` [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
