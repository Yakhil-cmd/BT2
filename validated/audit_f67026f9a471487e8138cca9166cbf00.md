### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last reward claim. A pool member who makes balance changes across many epochs without claiming rewards will grow this trace without bound. Once the trace is large enough, a single `claim_rewards` call will exceed the Starknet transaction step limit, permanently freezing all accumulated unclaimed yield with no recovery path.

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:**

The loop at `pool.cairo:859–877` iterates from `entry_to_claim_from` to `pool_member_trace_length`, processing every balance-change checkpoint that falls before `until_epoch`. The developers themselves flag this:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

**How the trace grows:**

Every call to `add_to_delegation_pool` (pool.cairo:242) or `exit_delegation_pool_intent` (pool.cairo:278) calls `set_member_balance` / `increase_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` implementation in `trace.cairo:152–175` appends a **new** checkpoint whenever the key (epoch) differs from the last entry's key:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
```

Each epoch in which the pool member changes their balance therefore appends one new entry to the trace.

**Why `entry_to_claim_from` does not help:**

`entry_to_claim_from` is stored in `pool_member_info` and is only advanced when `claim_rewards` **succeeds** (pool.cairo:358). If the transaction reverts due to gas exhaustion, the pointer is never updated. The user is permanently stuck: the loop is too large to complete, yet there is no partial-claim mechanism to advance the pointer in smaller steps.

**No external path needed — the pool member is the only actor required:**

`add_to_delegation_pool` enforces `caller == pool_member || caller == reward_address` (pool.cairo:228–232), and `exit_delegation_pool_intent` uses `get_caller_address()` as the pool member (pool.cairo:258). The trace therefore grows solely through the pool member's own legitimate actions, with no privileged access required.

### Impact Explanation

Once the `pool_member_epoch_balance` trace is large enough that iterating over it in a single transaction exceeds the Starknet step limit, `claim_rewards` will always revert. Because:

1. The trace is append-only (entries are never pruned).
2. `entry_to_claim_from` only advances on a successful `claim_rewards`.
3. There is no batched or partial-claim interface.

The pool member's entire accumulated unclaimed yield is **permanently frozen** — matching the allowed High impact: *"Permanent freezing of unclaimed yield."*

The same loop is also executed inside the view function `pool_member_info_v1` (pool.cairo:532–538), which would also become permanently unusable for the affected member.

### Likelihood Explanation

The precondition is a pool member who makes balance changes in many distinct epochs without claiming rewards. This can occur:

- **Organically**: a delegator who regularly tops up their stake (e.g., via DCA) but rarely claims rewards.
- **Deliberately**: a pool member who intentionally avoids claiming to accumulate a large trace (self-harm, but the freeze is irreversible once triggered).

The number of epochs required depends on the Starknet step budget per transaction and the per-iteration cost (multiple storage reads + arithmetic per iteration). The code comment acknowledges the loop is unbounded and relies on the assumption that gas limits will not be reached in practice — an assumption that weakens as the protocol ages and epoch counts grow. Likelihood is **Medium**: not trivially triggered, but reachable through normal long-term usage patterns.

### Recommendation

1. **Add a `max_iterations` cap** to `calculate_rewards` and allow `claim_rewards` to be called multiple times to process the trace in bounded chunks, advancing `entry_to_claim_from` incrementally.
2. **Alternatively**, compact the trace on each `claim_rewards` call by removing all processed entries (those with `epoch < reward_checkpoint.epoch`), keeping the trace size proportional to unclaimed epochs rather than total lifetime epochs.
3. At minimum, document a safe upper bound on how many epochs a pool member may skip between reward claims before the function becomes uncallable.

### Proof of Concept

1. Pool member calls `enter_delegation_pool` with a small amount.
2. For each of N epochs (N large enough to exceed the per-transaction step budget):
   - Advance to the next epoch (epoch counter increments).
   - Call `add_to_delegation_pool` with `amount = 1` — this appends one new entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert`.
   - Do **not** call `claim_rewards` (so `entry_to_claim_from` stays at 0).
3. After N epochs, call `claim_rewards(pool_member)`.
4. **Expected result**: the transaction reverts with an out-of-gas / step-limit error because `calculate_rewards` must iterate over all N entries.
5. All subsequent `claim_rewards` calls also revert. The pool member's accumulated yield is permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```text
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
```
