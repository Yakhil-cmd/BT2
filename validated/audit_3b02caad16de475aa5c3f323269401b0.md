### Title
Wrong Address Used in Token Transfer in `add_to_delegation_pool` - (File: src/pool/pool.cairo)

### Summary

In `pool.cairo`, the `add_to_delegation_pool` function allows either the `pool_member` or their `reward_address` to call it. However, when the `reward_address` is the caller, the function incorrectly transfers tokens **from `caller_address` (the reward address)** instead of from `pool_member`. This directly contradicts the interface specification, which states that `pool_member` must have sufficient funds and approval. The result is broken functionality: the reward address path always reverts unless the reward address itself holds tokens and has approved the pool — an unintended and undocumented requirement.

---

### Finding Description

In `add_to_delegation_pool`, the access control check correctly permits either `pool_member` or `pool_member_info.reward_address` as the caller: [1](#0-0) 

However, the subsequent token transfer uses `caller_address` as the source: [2](#0-1) 

`transfer_from_delegator` is defined as: [3](#0-2) 

So when `caller_address == reward_address`, the call becomes `checked_transfer_from(sender: reward_address, ...)`. But the pool member's balance is still credited: [4](#0-3) 

The interface specification explicitly states the precondition is on `pool_member`, not the caller: [5](#0-4) 

This is the same vulnerability class as the referenced report: a token operation uses the wrong address (caller instead of the intended token holder), causing the operation to revert or behave incorrectly.

---

### Impact Explanation

When `reward_address` calls `add_to_delegation_pool(pool_member, amount)`:

1. The access control check passes (reward_address is authorized).
2. The function attempts `transfer_from(sender: reward_address, ...)` instead of `transfer_from(sender: pool_member, ...)`.
3. Since the reward address is a reward *recipient*, it will not ordinarily hold staking tokens or have approved the pool contract.
4. The call **reverts**, permanently breaking the reward-address-initiated top-up path that the interface explicitly advertises.

Additionally, if a reward address happens to hold tokens and has approved the pool (e.g., a yield-aggregator contract), those tokens are silently drained into the pool and credited to `pool_member` — an unintended fund movement from the reward address.

**Impact category:** Griefing with damage to users (broken protocol feature); in the token-drain sub-case, direct theft of user funds from the reward address.

---

### Likelihood Explanation

Any pool member whose reward address attempts to call `add_to_delegation_pool` on their behalf triggers the broken path. The interface explicitly permits this call pattern, so users following the documented API will encounter the revert. The likelihood is **medium**: the reward-address caller path is a documented feature, and any integrator or smart-contract reward address that tries to use it will be affected.

---

### Recommendation

Replace `caller_address` with `pool_member` in the `transfer_from_delegator` call:

```cairo
// Before (buggy):
transfer_from_delegator(pool_member: caller_address, :amount, :token_dispatcher);

// After (correct):
transfer_from_delegator(pool_member: pool_member, :amount, :token_dispatcher);
```

This aligns the implementation with the interface specification, which states that `pool_member` must hold the funds and provide the approval.

---

### Proof of Concept

1. Pool member `P` joins the pool via `enter_delegation_pool`, setting `reward_address = R`.
2. `P` approves the pool contract to spend their tokens (as the interface requires).
3. `R` calls `add_to_delegation_pool(pool_member: P, amount: X)`.
4. Access control passes: `caller_address (R) == pool_member_info.reward_address`.
5. `transfer_from_delegator(pool_member: R, amount: X, ...)` is called.
6. The ERC-20 `transfer_from(sender: R, ...)` reverts because `R` has not approved the pool contract and/or does not hold the staking token.
7. The entire transaction reverts, even though `P` had the correct approval in place and `R` was a legitimately authorized caller per the interface. [6](#0-5) [3](#0-2)
