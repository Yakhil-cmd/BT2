### Title
Missing Zero-Divisor Check in `i256_mod` Causes Panic on SMOD Opcode with Zero Divisor - (`evm_interpreter/src/i256.rs`)

### Summary
`i256_mod` does not guard against a zero divisor before calling `div_rem`, which panics/aborts on zero input. The EVM specification requires `SMOD` with a zero divisor to return 0. Any unprivileged user can craft a transaction containing the `SMOD` opcode with a zero divisor to trigger this panic, halting forward execution or aborting proof generation.

### Finding Description

`i256_div` correctly guards against a zero divisor: [1](#0-0) 

```rust
pub fn i256_div(dividend: &mut U256, divisor_or_quotient: &mut U256) {
    let divisor_sign = i256_sign::<true>(divisor_or_quotient);
    if divisor_sign == Sign::Zero {
        *divisor_or_quotient = U256::ZERO;
        return;
    }
```

`i256_mod`, however, calls `i256_sign::<true>` on the divisor but **discards the result** and proceeds directly to `div_rem` without any zero check: [2](#0-1) 

```rust
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }
    let _ = i256_sign::<true>(divisor_or_remainder);  // result discarded
    // this is unsigned division of moduli
    let (_q, r) = dividend.div_rem(*divisor_or_remainder);  // panics if divisor == 0
```

Both the naive and RISC-V `div_rem` implementations explicitly panic/assert on a zero divisor: [3](#0-2) [4](#0-3) 

The test suite confirms this behavior: [5](#0-4) 

### Impact Explanation

The EVM Yellow Paper specifies that `SMOD(a, 0) = 0`. When an EVM transaction executes `SMOD` with a zero divisor, `i256_mod` reaches `div_rem` with a zero divisor and panics. In the forward system (sequencer, x86), this unwinds the process. In the proving system (RISC-V, `proof_running_system`), the `assert!(is_zero == false)` in `risc_v::U256::div_rem` aborts proof generation entirely. This is both an **EVM semantic mismatch** (wrong result vs. spec) and a **valid-execution unprovability** issue (a valid EVM transaction cannot be proven).

### Likelihood Explanation

Any unprivileged user can submit a transaction containing the `SMOD` opcode with a zero second operand. This is a single-opcode, zero-cost-to-construct attack vector. No special privileges, oracle manipulation, or governance access is required.

### Recommendation

Add a zero-divisor guard to `i256_mod` mirroring the existing guard in `i256_div`:

```rust
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    let divisor_sign = i256_sign::<true>(divisor_or_remainder);
    if divisor_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }
    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }
    let (_q, r) = dividend.div_rem(*divisor_or_remainder);
    ...
}
```

### Proof of Concept

1. Deploy any EVM contract containing the bytecode sequence: `PUSH1 0x00` (divisor = 0), `PUSH1 0x01` (dividend = 1), `SMOD`.
2. Submit a transaction calling this contract from any EOA.
3. The bootloader dispatches to the EVM interpreter, which calls `i256_mod(&mut U256::ONE, &mut U256::ZERO)`.
4. `dividend_sign` is `Sign::Plus` (non-zero), so the early return is skipped.
5. `i256_sign::<true>(&mut U256::ZERO)` returns `Sign::Zero` but the result is discarded.
6. `U256::div_rem(&mut U256::ONE, &mut U256::ZERO)` hits `assert!(is_zero == false)` → panic/abort.
7. In the proving system, proof generation aborts; in the forward system, the sequencer process panics.
