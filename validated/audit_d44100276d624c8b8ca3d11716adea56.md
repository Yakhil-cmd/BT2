### Title
Reentrancy in `claim_rewards` Allows Double-Claiming of Staker Rewards - (File: `src/staking/staking.cairo`)

### Summary
The `claim_rewards` function in `staking.cairo` transfers tokens to the staker's `reward_address` before persisting the zeroed-out `unclaimed_rewards_own` to storage. A malicious contract set as `reward_address` can reenter `claim_rewards` during the token transfer, read the stale storage state (rewards still non-zero), and claim the same rewards a second time.

### Finding Description

In `claim_rewards` (lines 411–431), the execution order is:

1. Read `staker_info` from storage — `unclaimed_rewards_own = X` [1](#0-0) 
2. Call `send_rewards_to_staker`, which internally:
   - Calls `claim_from_reward_supplier` (decrements `reward_supplier.unclaimed_rewards`) [2](#0-1) 
   - **Transfers tokens to `reward_address`** — external call, reentry point [3](#0-2) 
   - Sets `staker_info.unclaimed_rewards_own = 0` **in memory only** [4](#0-3) 
3. **Only then** writes the updated `staker_info` to storage [5](#0-4) 

Because the storage write at step 3 has not yet occurred when the token transfer fires at step 2, a reentrant call to `claim_rewards` from `reward_address` reads the old on-chain `staker_info` (still showing `unclaimed_rewards_own = X`) and successfully claims the same amount again.

The developers were aware of this class of bug in `unstake_action`, where they explicitly write `staker_info` to storage **before** any token transfer, with the comment *"This is done here to avoid re-entrancy"*: [6](#0-5) 

That same protection was not applied to `claim_rewards`.

The caller check in `claim_rewards` permits `reward_address` to call the function directly: [7](#0-6) 

So a malicious contract registered as `reward_address` is a fully authorized caller for the reentrant invocation.

### Impact Explanation

**High — Theft of unclaimed yield.**

The `reward_supplier.unclaimed_rewards` pool represents the aggregate rewards owed to all stakers. In a live system this pool is large. Each reentrant iteration drains an additional `X` from it, transferring funds that belong to other stakers to the attacker. The attacker can nest the reentry as many times as the reward_supplier balance allows, multiplying their payout by an arbitrary factor within a single transaction.

### Likelihood Explanation

**Medium.** The attack requires the adversary to be a registered staker (permissionless) and to set a malicious contract as their `reward_address` via `change_reward_address` (also permissionless). No privileged role, leaked key, or external dependency is needed. The only prerequisite is that the staker has accumulated non-zero `unclaimed_rewards_own`, which happens automatically over time.

### Recommendation

Apply the checks-effects-interactions pattern: write the zeroed `staker_info` to storage **before** transferring tokens to `reward_address`, exactly as `unstake_action` already does.

```cairo
// In claim_rewards — fix: write state before external transfer
self.write_staker_info(:staker_address, :staker_info);          // <-- move here
self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
// remove the write_staker_info call that currently follows
```

Alternatively, add a reentrancy guard (a boolean lock in storage) that is set before and cleared after the function body.

### Proof of Concept

1. Attacker deploys `MaliciousReward` contract that, upon receiving STRK, calls `staking.claim_rewards(staker_address)` once more.
2. Attacker calls `staking.stake(...)` with `reward_address = address(MaliciousReward)`.
3. Attacker waits for `unclaimed_rewards_own` to accumulate to `X`.
4. `MaliciousReward` calls `staking.claim_rewards(staker_address)`.
5. Inside `send_rewards_to_staker`:
   - `reward_supplier.unclaimed_rewards` is decremented by `X`.
   - `X` STRK is transferred to `MaliciousReward` → triggers reentry.
6. Reentrant call reads `staker_info` from storage: `unclaimed_rewards_own` is still `X` (write not yet committed).
7. Reentrant call decrements `reward_supplier.unclaimed_rewards` by `X` again and transfers another `X` to `MaliciousReward`.
8. Reentrant call writes `staker_info` (zeroed).
9. Outer call writes `staker_info` (zeroed) — no error, no revert.

**Result:** Attacker receives `2X` STRK; the extra `X` is stolen from the shared `reward_supplier` pool, reducing rewards available to all other stakers.
