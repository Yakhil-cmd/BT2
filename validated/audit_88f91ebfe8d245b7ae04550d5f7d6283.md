### Title
Threshold Value Duplicated in `PresignArguments` Without Binding to `KeygenOutput`, Enabling Mismatch That Corrupts Sign Outputs — (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/frost/mod.rs`)

---

### Summary

`KeygenOutput` does not persist the threshold used during DKG. Both `PresignArguments` structs (OT-based ECDSA and FROST) require the caller to re-supply the threshold as a separate field. This is an exact structural analog to the FuelToken duplicate-data bug: a value that has an authoritative source (the DKG run) is instead hardcoded/re-specified by the caller, creating a window for the two copies to diverge. When they diverge downward (caller-supplied threshold < DKG threshold), the signing protocol completes but produces a cryptographically unusable signature, corrupting the sign output accepted by honest parties.

---

### Finding Description

`KeygenOutput<C>` is defined as:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
}
``` [1](#0-0) 

It stores only the private share and the public key. The threshold that governed the DKG is **not** preserved.

Both `PresignArguments` structs then require the caller to re-supply the threshold independently:

**OT-based ECDSA:**
```rust
pub struct PresignArguments {
    pub triple0: (TripleShare, TriplePub),
    pub triple1: (TripleShare, TriplePub),
    pub keygen_out: KeygenOutput,
    /// The desired threshold for the presignature, which must match the original threshold
    pub threshold: ReconstructionLowerBound,
}
``` [2](#0-1) 

**FROST (EdDSA / RedJubjub):**
```rust
pub struct PresignArguments<C: Ciphersuite> {
    pub keygen_out: KeygenOutput<C>,
    /// The threshold for the scheme
    pub threshold: ReconstructionLowerBound,
}
``` [3](#0-2) 

The comment "which must match the original threshold" documents the invariant but provides no enforcement mechanism. There is no runtime check that `args.threshold` equals the threshold embedded in the DKG run that produced `args.keygen_out`.

In the FROST presign path, the only guard is:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [4](#0-3) 

This check only prevents the threshold from exceeding the participant count; it does **not** verify the threshold matches the one used during DKG. The same pattern appears in `assert_sign_inputs`: [5](#0-4) 

For the OT-based ECDSA path, the presign function checks the supplied threshold against the triple thresholds:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(...);
}
``` [6](#0-5) 

But again, there is no check that `args.threshold` matches the DKG threshold embedded in `args.keygen_out`. If triples were generated with a threshold lower than the DKG threshold, the check passes silently.

---

### Impact Explanation

When the caller-supplied `threshold` in `PresignArguments` is **lower** than the actual DKG threshold:

- The validation gates pass (the supplied value is ≤ `participants.len()`).
- The FROST signing protocol proceeds with a participant set smaller than the DKG polynomial degree requires.
- Lagrange interpolation over an under-sized participant set yields incorrect partial-signature aggregation.
- The coordinator assembles and broadcasts a signature that fails external verification.
- Honest parties have completed a full signing round and accepted an **unusable cryptographic output** — matching the allowed High impact: *"Corruption of sign outputs so honest parties accept unusable cryptographic outputs."*

When the caller-supplied threshold is **higher** than the DKG threshold, `assert_sign_inputs` rejects valid signing sessions, causing permanent denial of signing for honest parties under valid inputs — also a High impact.

---

### Likelihood Explanation

The threshold is not stored in `KeygenOutput`, so every caller of `presign()` must manually re-derive and re-supply it. In a multi-phase pipeline (keygen → triple generation → presign → sign), the threshold value must be threaded through by the application layer. A malicious coordinator orchestrating the signing session can supply a mismatched threshold to honest participants' `PresignArguments`. Alternatively, an honest implementation error (e.g., using a stale or wrong threshold after a reshare that changed the threshold) silently passes all guards and corrupts the sign output. The reshare path explicitly allows changing the threshold: [7](#0-6) 

After a reshare, the old threshold stored by the application layer is stale, and there is no mechanism in `KeygenOutput` to detect this.

---

### Recommendation

Store the threshold inside `KeygenOutput` so it is always derived from the authoritative DKG output rather than re-specified by the caller:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
    pub threshold: ReconstructionLowerBound, // authoritative source
}
```

Then remove the `threshold` field from both `PresignArguments` structs and derive it directly from `args.keygen_out.threshold`. Add an assertion in `presign()` that the derived threshold matches the triple thresholds (OT-based ECDSA) and the participant count (FROST), eliminating the duplicate entirely.

---

### Proof of Concept

1. Run DKG with 5 participants and threshold = 3, producing `KeygenOutput` for each participant.
2. Construct `PresignArguments` for FROST with `threshold = 2` (below the DKG threshold of 3).
3. Call `frost::presign()` with only 2 participants. The guard `args.threshold.value() > participants.len()` evaluates `2 > 2 = false`, so no error is returned. [4](#0-3) 
4. The presign protocol completes successfully.
5. Call `frost::sign()` with the same 2-participant set and `threshold = 2`. `assert_sign_inputs` passes. [5](#0-4) 
6. The aggregated signature is computed via Lagrange interpolation over 2 shares of a degree-2 polynomial (requiring 3 shares). The result is cryptographically incorrect.
7. Honest parties receive and accept a signature that fails all external ECDSA/EdDSA verification — a corrupted, unusable sign output.
