### Title
Incomplete Threshold Lower-Bound Validation in FROST Signing Allows Invalid Threshold to Bypass Checks - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate only the upper bound of the threshold parameter (`threshold > participants.len()`), omitting the lower-bound check (`threshold < 2`) that every other validation function in the codebase enforces. A malicious coordinator or unprivileged library caller can supply `threshold = 0` or `threshold = 1`, pass validation, and drive the FROST signing protocol into producing a corrupted or unusable signature.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` and `validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` both enforce a two-sided range on the threshold:

```rust
// src/dkg.rs – assert_key_invariants
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`validate_triple_inputs` mirrors this pattern exactly. [2](#0-1) 

By contrast, both public FROST entry-points perform only the upper-bound check:

```rust
// src/frost/mod.rs – presign (line 72)
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no ThresholdTooSmall check
``` [3](#0-2) 

```rust
// src/frost/mod.rs – assert_sign_inputs (line 144)
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no ThresholdTooSmall check
``` [4](#0-3) 

The DKG spec (docs/dkg.md line 50) explicitly requires `1 < threshold < N`, so `threshold = 1` is an invalid configuration that the signing layer silently accepts. [5](#0-4) 

### Impact Explanation

When `threshold = 1` (or `0`) is accepted by `assert_sign_inputs` or `presign`, the FROST signing protocol runs with a reconstruction lower bound that is inconsistent with the key material produced during DKG (which enforced `threshold >= 2`). The coordinator collects only one signature share and attempts aggregation; the resulting Lagrange interpolation is evaluated over a single point, producing a scalar that does not correspond to the shared secret. The aggregated Schnorr signature is cryptographically invalid and will fail external verification. Honest participants have no way to detect this mismatch before the protocol completes, so they accept a useless output.

This maps to: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

`assert_sign_inputs` is a public API function callable by any library user or coordinator. The threshold is a plain caller-supplied integer with no type-level enforcement of the lower bound (`ReconstructionLowerBound` is a transparent `usize` wrapper). A malicious coordinator constructing the signing session simply passes `threshold = 1`; no cryptographic capability or key material is required. The missing check is a single omitted `if` branch, making accidental misconfiguration equally plausible.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already use:

```rust
// In both presign() and assert_sign_inputs()
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place this check immediately after the upper-bound check in both functions.

### Proof of Concept

1. Key generation runs with `threshold = 2`, producing shares split across 3 participants.
2. A malicious coordinator calls `assert_sign_inputs(participants, 1u8, me, coordinator)`. The function returns `Ok(participants)` because `1 <= 3` passes the only guard.
3. The coordinator drives the FROST signing round with `threshold = 1`, collecting a single signature share from participant `P1`.
4. FROST aggregation interpolates over one point: the resulting `z` scalar equals `P1`'s share scalar, not the true secret. The produced `(R, z)` pair fails `verify(public_key, message, signature)`.
5. All honest participants have completed the protocol and accepted the output, but the signature is unusable.

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
