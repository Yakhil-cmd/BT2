### Title
L1 Priority Transaction Exclusion via Block Resource Limits Causes Indefinite Fund Lock - (File: `basic_bootloader/src/bootloader/block_flow/zk/mod.rs`)

### Summary
ZKsync OS applies block resource limits (gas, native, pubdata, and L2→L1 logs) uniformly to both L2 and L1 priority transactions. When an L1 priority transaction causes any block limit to be exceeded, it is silently reverted and excluded from the block — identical to how an L2 transaction is dropped. Since L1 priority transactions represent funds already locked on L1, and ZKsync OS contains no mechanism to guarantee their inclusion in a subsequent block, an unprivileged attacker can fill block resources (particularly the 16,384-log cap) to indefinitely defer L1 priority transactions, leaving user funds stuck on L1.

### Finding Description

**Root cause — `check_for_block_limits` treats L1 priority transactions identically to L2 transactions:** [1](#0-0) 

The function returns `Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)` (or any other block-limit error) for any transaction — including L1 priority ops — that pushes the block over a limit.

**Root cause — the tx loop reverts and discards the L1 priority transaction on any block-limit error:** [2](#0-1) 

When `check_for_block_limits` returns an error, the entire transaction (including its `emit_l1_l2_tx_log` side-effect) is rolled back via `finish_global_frame(Some(&pre_tx_rollback_handle))`. The priority transaction hash is **never** added to `enforced_transaction_hashes_accumulator`: [3](#0-2) 

**Root cause — every L1 priority transaction unconditionally emits a log:** [4](#0-3) 

`emit_l1_l2_tx_log` pushes one entry into `logs_storage`. The block-level log cap is: [5](#0-4) 

**Root cause — the settlement layer only verifies the hash of *included* L1 txs, not that all pending ones were processed:** [6](#0-5) 

The `priority_operations_hash` committed to the public input is built only from transactions the operator chose to include: [7](#0-6) 

There is no ZKsync OS-level enforcement that all pending L1 priority transactions must be processed within any bounded number of blocks.

**Attack path (unprivileged sender):**

1. Attacker submits up to 16,384 L2 transactions that each call the L1 Messenger (`0x8008`) with `sendToL1(bytes)`, filling the per-block log capacity.
2. The victim's L1 priority transaction (with deposited base tokens) is fed to the block by the operator.
3. `process_l1_transaction` executes the tx body and calls `emit_l1_l2_tx_log`, pushing the log count past `MAX_NUMBER_OF_LOGS`.
4. `check_for_block_limits` returns `BlockL2ToL1LogsLimitReached`.
5. The entire L1 priority transaction is rolled back; its hash is never added to `enforced_transaction_hashes_accumulator`.
6. The settlement layer accepts the batch without the victim's L1 tx; the victim's funds remain locked on L1.
7. The attacker repeats in every subsequent block, indefinitely deferring the victim's transaction.

### Impact Explanation

L1 priority transactions carry base-token deposits (`reserved[0]` = `to_mint`) that are minted on L2 only upon successful execution. If the transaction is perpetually excluded from blocks, the deposited funds are permanently locked on L1 with no cancel or expiry path in ZKsync OS. This is a direct public-funds-loss / indefinite-lock impact. [8](#0-7) 

### Likelihood Explanation

The L2→L1 log limit (16,384) is reachable within a single block by any user willing to pay L2 gas for 16,384 small transactions. The cost is bounded and predictable. Because ZKsync OS has no priority ordering that places L1 transactions ahead of L2 transactions in the block-limit check, and because the settlement layer imposes no deadline on L1 tx inclusion, the attack can be sustained across blocks at modest cost. The victim has no recourse: there is no cancel, no expiry, and no escape hatch in the ZKsync OS state-transition function.

### Recommendation

1. **Exempt L1 priority transactions from block-limit exclusion.** When `check_for_block_limits` would exclude an L1 priority transaction, the block should be sealed *before* that transaction rather than discarding it. The L1 tx must then be the first transaction in the next block.
2. **Enforce a maximum deferral window.** The settlement layer (or the ZKsync OS public-input structure) should commit to a deadline by which every pending L1 priority transaction must appear in the `priority_operations_hash`, halting the chain if the operator violates it.
3. **Separate the L2→L1 log budget.** Reserve a portion of `MAX_NUMBER_OF_LOGS` exclusively for L1 priority transaction result logs so that L2 user messages cannot crowd them out.

fn check_for_block_limits<S: EthereumLikeTypes>(
    system: &mut System<S>,
    gas_used: u64,
    computational_native_used: u64,
    pubdata_used: u64,
    logs_used: u64,
    blob_gas_used: u64,
) -> Result<(), InvalidTransaction>
where
    S::IO: IOSubsystemExt,
    <S as SystemTypes>::Metadata: ZkSpecificPricingMetadata,
{
    if gas_used > system.get_gas_limit() {
        system_log!(
            system,
            "Block gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockGasLimitReached)
    } else if blob_gas_used > system.get_blob_gas_limit() {
        system_log!(
            system,
            "Block blob gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockBlobGasLimitReached)
    } else if !cfg!(feature = "resources_for_tester")
        && computational_native_used > MAX_NATIVE_COMPUTATIONAL
    {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block native limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockNativeLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && pubdata_used > system.get_pubdata_limit() {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block pubdata limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockPubdataLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block logs limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
    } else {
        Ok(())
    }
}
