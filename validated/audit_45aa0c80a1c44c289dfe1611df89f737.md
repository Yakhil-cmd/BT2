### Title
`skip_stateful_validations` Accepts Invoke Transactions with Arbitrary Signatures for Undeployed Accounts - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function bypasses the `__validate__` entry-point check for any invoke transaction with `nonce == 1` sent from an address whose on-chain nonce is `0` and whose address appears in `account_tx_in_pool_or_recent_block`. Because that check is not restricted to deploy-account transactions, an unprivileged attacker who observes a victim's pending `deploy_account` transaction in the mempool can submit a competing invoke transaction (nonce=1) carrying an arbitrary/invalid signature from the victim's address. The gateway accepts it without ever calling `__validate__`, inserting it into the mempool and allowing it to displace the victim's legitimate invoke transaction via fee escalation.

### Finding Description

`skip_stateful_validations` is the UX shortcut that lets a user broadcast `deploy_account` + `invoke` (nonce=1) simultaneously before the account exists on-chain:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:429-460
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}

When `skip_validate = true`, `run_validate_entry_point` sets `validate: !skip_validate = false`, so the blockifier's `StatefulValidator::validate` runs `perform_pre_validation_stage` (nonce increment + fee-bound checks) but **never calls the `__validate__` entry point**:

```rust

The guard `account_tx_in_pool_or_recent_block` is implemented as:

    self.state.contains_account(account_address)

`tx_pool.contains_account` returns `true` whenever **any** transaction from that address is in the pool — it is not restricted to `DeployAccount` transactions:

```rust
// crates/apollo_mempool/src/transaction_pool.rs:201-203
pub fn contains_account(&self, address: ContractAddress) -> bool {
    self.txs_by_account.contains(address)
}
``` [4](#0-3) 

The inline comment in `skip_stateful_validations` claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." That reasoning is circular: the very first invoke (nonce=1) that triggers the skip is the one whose signature is never checked.

**Attack path:**

1. Victim pre-funds their deterministic account address and submits a `deploy_account` tx (nonce=0). It enters the mempool; `tx_pool.contains_account(victim)` becomes `true`.
2. Attacker observes the pending `deploy_account` tx (via P2P or RPC).
3. Attacker crafts an invoke tx: `sender_address = victim`, `nonce = 1`, arbitrary/invalid signature, resource bounds set high enough that `enforce_fee` returns `true` and `verify_can_pay_committed_bounds` passes (the victim's pre-funded balance covers it).
4. Gateway evaluates `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` (on-chain, victim not deployed) ✓
   - `account_tx_in_pool_or_recent_block(victim)` → `true` ✓
5. `run_validate_entry_point` is called with `validate = false`; `__validate__` is **never invoked**.
6. `perform_pre_validation_stage` passes: nonce `0 ≤ 1` with `strict_nonce_check=false`, fee bounds satisfied, balance sufficient.
7. Attacker's tx is inserted into the mempool.
8. If the victim also submitted a legitimate invoke (nonce=1), the attacker's higher-fee tx replaces it via fee escalation (`enable_fee_escalation = true` by default).
9. Batcher executes `deploy_account` (victim deployed), then executes attacker's invoke → `__validate__` fails → tx reverts. Victim's invoke is gone. [5](#0-4) 

### Impact Explanation

The gateway admits an invoke transaction whose signature was never verified. This satisfies: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."** The victim's legitimate nonce-1 invoke is displaced from the mempool by the attacker's signature-less tx. The attacker can repeat this every time the victim re-submits, creating a persistent, targeted DoS against any account in the deploy-account-pending window.

### Likelihood Explanation

- Pending `deploy_account` transactions are publicly visible via P2P propagation and RPC.
- The attacker needs only to know the victim's address (derivable from the `deploy_account` tx itself) and submit a single invoke with a marginally higher fee.
- No privileged access, no special contract, no cryptographic secret is required.
- The default production config has `enable_fee_escalation = true` and `fee_escalation_percentage = 10`, making displacement straightforward. [6](#0-5) 

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`DeployAccount` transaction** for the sender address is present in the pool. Add a dedicated `deploy_account_tx_in_pool(address)` query to the mempool that inspects `InternalRpcTransactionWithoutTxHash::DeployAccount` entries, and use that in `skip_stateful_validations` instead of the current address-presence check.

```rust
/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
```
