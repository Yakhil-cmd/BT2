### Title
Silent Token-Type Substitution in `exit_erc20_token_to_near`: `:unwrap` on Non-wNEAR ERC-20 Silently Delivers NEP-141 Instead of Native NEAR, Enabling Permanent Fund Freeze - (File: engine-precompiles/src/native.rs)

---

### Summary

When a user calls the `exitToNear` precompile with the `:unwrap` suffix on a non-wNEAR ERC-20 token, the code silently falls through to a regular `ft_transfer` instead of reverting. The user explicitly requested native NEAR but receives NEP-141 tokens instead. If the recipient NEAR account is a smart contract that cannot handle NEP-141 tokens, the tokens are permanently locked. This is a direct structural analog to the HubPool `sendEth=true` / WETH-to-contract silent substitution bug.

---

### Finding Description

In `exit_erc20_token_to_near` (`engine-precompiles/src/native.rs`), the match on `exit_params.message` has three arms:

```rust
let (nep141_account_id, args, method, transfer_near_args, event) = match exit_params.message {
    // Arm 1: wNEAR unwrap — only fires when erc20_address == wnear address
    Some(Message::UnwrapWnear) if erc20_address == get_wnear_address(io).raw() => {
        // calls near_withdraw → native NEAR transfer
    }
    // Arm 2: Omni message
    Some(Message::Omni(msg)) => { /* ft_transfer_call */ }
    // Arm 3: catch-all — silently handles BOTH None AND UnwrapWnear on non-wNEAR tokens
    _ => {
        // ft_transfer — NEP-141 tokens, NOT native NEAR
    }
};
``` [1](#0-0) 

The `_` arm catches `Some(Message::UnwrapWnear)` whenever the ERC-20 token is not the registered wNEAR address. The code comment explicitly acknowledges this: *"P.S. We use underscore here instead of `None` to handle the case when a user could add the `unwrap` suffix for non wNEAR ERC-20 token by mistake."* [2](#0-1) 

This creates a critical inconsistency: the base-token (ETH) path in `exit_base_token_to_near` correctly returns `ERR_INVALID_MESSAGE` for `Message::UnwrapWnear`, while the ERC-20 path silently substitutes `ft_transfer`. [3](#0-2) 

The full execution path is:

1. EVM user calls an ERC-20 contract's `withdrawToNear(bytes recipient, uint256 amount)` with `recipient = "target.near:unwrap"`.
2. The ERC-20 contract burns the tokens and calls the `exitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x01`.
3. `ExitToNear::run` dispatches to `exit_erc20_token_to_near`.
4. `parse_recipient` parses the `:unwrap` suffix and sets `message = Some(Message::UnwrapWnear)`.
5. The guard `erc20_address == get_wnear_address(io).raw()` is **false** (non-wNEAR token).
6. Falls through to `_` arm → method becomes `"ft_transfer"`, `transfer_near_args = None`.
7. A `PromiseArgs::Create` (no callback) is emitted calling `ft_transfer` on the NEP-141 contract.
8. The recipient receives NEP-141 tokens, not native NEAR. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

The user's ERC-20 tokens are irreversibly burned on Aurora. The equivalent NEP-141 tokens are sent via `ft_transfer` to the specified NEAR account. If that account is a NEAR smart contract that:

1. Has storage registered with the NEP-141 contract (so `ft_transfer` succeeds and does not revert), **and**
2. Has no internal mechanism to transfer or use NEP-141 tokens (e.g., a native-NEAR-only escrow, a multisig that only handles `Transfer` actions, or any contract not implementing NEP-141 outbound logic),

then the NEP-141 tokens are permanently inaccessible. The user's funds are frozen: burned on the EVM side, delivered to a contract that cannot move them on the NEAR side.

This is structurally identical to the HubPool bug: the user explicitly signals a desired output format (native NEAR via `:unwrap`), the protocol silently delivers a different format (NEP-141 tokens), and if the recipient cannot handle that format, funds are permanently frozen.

**Impact:** Permanent freezing of funds (Critical).

---

### Likelihood Explanation

**Medium-low.** Two conditions must hold simultaneously:

1. **User mistake or confusion:** A user calls `exitToNear` with `:unwrap` on a non-wNEAR ERC-20 token. This is realistic — the `:unwrap` suffix is a documented feature for wNEAR, and users integrating multiple ERC-20 tokens may apply it incorrectly. The code itself acknowledges this is an expected user mistake.

2. **Vulnerable recipient:** The recipient NEAR account is a smart contract with NEP-141 storage registered but no outbound NEP-141 transfer capability. This is less common but realistic for contracts designed around native NEAR (e.g., simple escrows, DAO treasuries, or contracts that only implement `ft_on_transfer` for a specific token).

The combination is low-frequency but not theoretical. Any EVM user can trigger this path with no special privileges.

---

### Recommendation

Apply one of the following fixes to `exit_erc20_token_to_near`:

**Option A (preferred — consistent with base token behavior):** Return an explicit error when `Message::UnwrapWnear` is set but the ERC-20 is not the registered wNEAR token:

```rust
Some(Message::UnwrapWnear) => {
    return Err(ExitError::Other(Cow::from("ERR_UNWRAP_NOT_WNEAR")));
}
```

**Option B:** Explicitly match `None` in the fallthrough arm and return an error for `Some(Message::UnwrapWnear)` on non-wNEAR tokens, making the silent substitution impossible.

This aligns the ERC-20 path with the base-token path, which already returns `ERR_INVALID_MESSAGE` for `Message::UnwrapWnear`. [1](#0-0) 

---

### Proof of Concept

1. Deploy Aurora with a registered non-wNEAR NEP-141 token and its corresponding ERC-20 mirror.
2. Deploy a NEAR smart contract `vault.near` that has storage registered with the NEP-141 contract but implements no outbound NEP-141 transfer logic (e.g., a native-NEAR-only escrow).
3. As an EVM user, hold ERC-20 tokens and call `withdrawToNear("vault.near:unwrap", amount)` on the ERC-20 contract.
4. The ERC-20 contract burns the tokens and calls the `exitToNear` precompile with `flag=0x01` and recipient `"vault.near:unwrap"`.
5. `parse_recipient` sets `message = Some(Message::UnwrapWnear)`.
6. `exit_erc20_token_to_near` evaluates `erc20_address == get_wnear_address(io).raw()` → **false**.
7. Falls through to `_` arm: `method = "ft_transfer"`, `transfer_near_args = None`.
8. A `PromiseArgs::Create` is emitted calling `ft_transfer` on the NEP-141 contract with `receiver_id = "vault.near"`.
9. `vault.near` receives NEP-141 tokens (not native NEAR) and has no way to move them.
10. The ERC-20 tokens are permanently burned; the NEP-141 tokens are permanently frozen in `vault.near`. [6](#0-5) [7](#0-6) [8](#0-7)

    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
