Audit Report

## Title
`AccountState::transfer()` Unconditionally Removes Neuron IDs and Returns `Ok(())` on Partial Failure, Permanently Locking Genesis Participants Out of `claim_neurons` - (File: rs/nns/gtc/src/lib.rs)

## Summary
`AccountState::transfer()` removes each neuron from `self.neuron_ids` and returns `Ok(())` unconditionally, even when individual `transfer_gtc_neuron` inter-canister calls fail. Because `forward_whitelisted_unclaimed_accounts` — callable by any unprivileged principal after 188 days — sets `has_forwarded = true` on an `Ok(())` return, a single transient failure during the per-neuron loop permanently locks the account owner out of `claim_neurons`, while the failed neurons remain stuck under GTC control in the Governance canister, transferred to neither the owner nor the custodian.

## Finding Description
In `rs/nns/gtc/src/lib.rs`, `AccountState::transfer()` (lines 188–209) iterates over all neuron IDs and calls `GovernanceCanister::transfer_gtc_neuron` for each. At line 192, `self.neuron_ids.retain(|id| id != &neuron_id)` executes unconditionally — before the result is inspected — so the neuron ID is removed from the account regardless of whether the inter-canister call succeeded or failed. Failed neurons are pushed to `self.failed_transferred_neurons` (line 204) with no recovery path. The function then returns `Ok(())` at line 209 regardless of how many neurons failed.

The caller `forward_whitelisted_unclaimed_accounts` (lines 118–119) treats this `Ok(())` as full success and sets `account.has_forwarded = true`. This function is exposed as an unprivileged update call (canister.rs lines 174–182) with no caller restriction; any principal may invoke it once `SECONDS_UNTIL_FORWARD_WHITELISTED_UNCLAIMED_ACCOUNTS_CAN_BE_CALLED` (188 days) has elapsed since genesis, a window that has long passed on mainnet.

Once `has_forwarded = true` is set, `claim_neurons` is permanently blocked (lines 58–60). Because `neuron_ids` is also emptied unconditionally, even removing the `has_forwarded` guard would leave nothing to claim. The same flaw applies to `donate_account` (lines 89–90): `transfer()` is called with `?`, but since it always returns `Ok(())`, `has_donated = true` is always set even on partial failure.

## Impact Explanation
Genesis participants whose whitelisted accounts experience even a single transient `transfer_gtc_neuron` failure during forwarding permanently lose access to their neurons. The failed neurons remain under GTC control in the Governance canister — not transferred to the custodian, not claimable by the owner — with no on-chain recovery mechanism. This constitutes permanent loss of governance assets (NNS neurons representing staked ICP) for affected genesis participants, matching the High bounty impact: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds" and "Significant NNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The 188-day window has elapsed on mainnet, so any principal can trigger `forward_whitelisted_unclaimed_accounts` at any time. No special role or privilege is required. A transient Governance canister error — message queue saturation, temporary unavailability, or a canister trap during `transfer_gtc_neuron` — during any one of the per-neuron inter-canister calls is sufficient. Seed Round accounts have 48 neurons each, meaning 48 sequential inter-canister calls are made per account; the cumulative probability of at least one failure across 48 calls is materially higher than for a single call. An adversary who can induce Governance canister message queue pressure (e.g., by flooding it with other calls) can increase the probability of triggering the partial-failure path on demand.

## Recommendation
Move `self.neuron_ids.retain(|id| id != &neuron_id)` inside the `Ok(_)` arm so that failed neurons remain in `self.neuron_ids` and are retryable:

```rust
match result {
    Ok(_) => {
        self.neuron_ids.retain(|id| id != &neuron_id);
        self.successfully_transferred_neurons.push(donated_neuron);
    }
    Err(e) => {
        donated_neuron.error = Some(e.to_string());
        self.failed_transferred_neurons.push(donated_neuron);
    }
}
```

Additionally, `transfer()` should return `Err(...)` if any neuron failed (or check `self.failed_transferred_neurons.is_empty()` before returning `Ok(())`), so that callers do not set `has_forwarded`/`has_donated` on partial failure. The `has_forwarded`/`has_donated` flags should only be set after confirmed full success.

## Proof of Concept
1. Confirm mainnet: 188 days since IC genesis has elapsed; `forward_whitelisted_unclaimed_accounts` is callable by any principal.
2. Any principal sends an ingress `forward_whitelisted_unclaimed_accounts` call to the GTC canister.
3. The GTC iterates over whitelisted accounts and calls `AccountState::transfer()` for each.
4. During the loop over neuron IDs for a target account, the Governance canister's `transfer_gtc_neuron` call fails for neuron N (e.g., due to message queue saturation induced by concurrent calls, or a transient canister error).
5. Line 192 executes: `self.neuron_ids` no longer contains N. N is pushed to `failed_transferred_neurons`.
6. The loop continues; `transfer()` returns `Ok(())` at line 209.
7. Line 119 executes: `account.has_forwarded = true`.
8. The account owner later calls `claim_neurons` → receives `"Account has previously forwarded its funds"` (line 59).
9. Neuron N remains under GTC control in the Governance canister: not transferred to the custodian, not claimable by the owner, with no recovery path.

A deterministic integration test can reproduce this by mocking `GovernanceCanister::transfer_gtc_neuron` to return `Err(...)` for one neuron ID mid-loop, then asserting that `has_forwarded == true`, `neuron_ids` is empty, and `failed_transferred_neurons` is non-empty, while `claim_neurons` returns the forwarded-funds error.
