### Title
Missing Minimum Threshold Lower-Bound Validation in FROST Signing Allows Corrupt Signing Outputs - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that the threshold does not exceed the participant count but omits the minimum-threshold lower-bound check (`threshold >= 2`) that is consistently enforced in every other protocol entry point in the library. A caller (including a malicious coordinator) can invoke FROST EdDSA or RedJubjub signing with `threshold = 0` or `threshold = 1`, causing the signing protocol to proceed and produce a cryptographically invalid, unusable signature that honest participants accept as a completed protocol run.

### Finding Description

Every other protocol entry point in the library enforces a minimum threshold of 2:

- `assert_key_invariants` (DKG/refresh/reshare) in `src/dkg.rs` explicitly rejects `threshold < 2`: [1](#0-0) 

- `validate_triple_inputs` (OT-based ECDSA triple generation) in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` explicitly rejects `threshold_value < 2`: [2](#0-1) 

However, `assert_sign_inputs` in `src/frost/mod.rs`, which gates both EdDSA and RedJubjub signing, only checks the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

There is **no** corresponding lower-bound check: [3](#0-2) 

This function is the sole validation gate for:
- `sign_v1` (EdDSA) in `src/frost/eddsa/sign.rs`: [4](#0-3) 
- `sign` (RedJubjub) in `src/frost/redjubjub/sign.rs`: [5](#0-4) 

When `threshold = 1` is passed, the threshold value is forwarded into `construct_key_package` which calls `KeyPackage::new()` with `threshold = 1`: [6](#0-5) 

With `threshold = 1`, the frost_core library accepts a signing package containing only a single signer's commitment. The Lagrange coefficient for a single-participant set is 1, so the signature share computed is `z_i = nonce_i + 1 * x_i * challenge`, where `x_i` is a Shamir share of the secret key (not the full secret key). The resulting aggregated signature is cryptographically invalid and fails verification, yet the protocol completes without error from the library's perspective — honest participants have accepted and contributed to a signing session that produces an unusable output.

### Impact Explanation

This maps to **High: Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs**. Honest participants complete the signing protocol, contribute their nonces and signature shares, and receive no error — but the coordinator produces a signature that fails standard Ed25519 or RedJubjub verification. The signing session is consumed (nonces are spent, presign material is used), and the honest parties have no indication the output is invalid until external verification fails. Under `threshold = 0`, the `KeyPackage::new()` call receives `0u16`, which may trigger undefined behavior or a panic inside frost_core, causing a denial-of-signing for all honest participants.

### Likelihood Explanation

The `ReconstructionLowerBound` type is a plain `usize` newtype with no enforced minimum: [7](#0-6) 

Any library caller — including a malicious coordinator who controls the signing invocation — can construct `ReconstructionLowerBound::from(1usize)` or `ReconstructionLowerBound::from(0usize)` and pass it directly to `sign_v1` or `sign`. No privilege is required beyond being a participant who initiates a signing session. The inconsistency with DKG and triple-generation validation makes this an easy mistake for an integrator to trigger accidentally as well.

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs` to `assert_sign_inputs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This check should be inserted immediately after the `participants.len() < 2` guard in `assert_sign_inputs`: [8](#0-7) 

Additionally, consider enforcing the minimum at the type level by making `ReconstructionLowerBound::new()` a validated constructor that rejects values below 2, consistent with the library's documented invariant.

### Proof of Concept

```rust
use threshold_signatures::{
    frost::eddsa::sign::sign_v1,
    ReconstructionLowerBound,
    // ... other imports
};

// Assume valid DKG was run with threshold = 3, participants = [A, B, C]
// Malicious coordinator calls sign_v1 with threshold = 1:
let result = sign_v1(
    &participants,
    ReconstructionLowerBound::from(1usize), // bypasses lower-bound check
    me,
    coordinator,
    keygen_output,
    message,
    rng,
);
// result is Ok(...) — no error returned
// Protocol runs to completion; coordinator aggregates with Lagrange coefficient = 1
// applied to a single Shamir share, producing z = nonce + x_i * challenge
// The resulting signature fails Ed25519 verification.
// Nonces are consumed; the signing session is permanently wasted.
```
