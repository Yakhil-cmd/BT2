### Title
WalletConnect `cat_spend` Confirmation Dialog Omits CAT Token Identity, Enabling Asset Confusion Attack — (`packages/gui/src/electron/commands/humanizeParamValue.ts`)

### Summary
A dApp connected via WalletConnect that has been granted `chia_wallet.cat_spend` permission can issue a spend against any `wallet_id` (including a high-value CAT wallet). The confirmation dialog displays the amount as a bare number with no token name or symbol, and the `wallet_id` as a raw integer. The user has no reliable way to identify which specific CAT is being spent before approving.

### Finding Description

The `formatMojoCat` function in `humanizeParamValue.ts` is responsible for rendering the `amount` field in the `cat_spend` confirmation dialog. The code reads `wallet_id` from the request data but then unconditionally returns only the formatted mojo-to-CAT number — the `lookupCat` call that would append the token display name is explicitly commented out: [1](#0-0) 

The `chia_wallet.cat_spend` command schema in `Commands.ts` marks `amount` with `humanize: 'mojo-to-cat'` (which routes through `formatMojoCat`) and shows `wallet_id` as a plain `type: 'number'` with no `hide: true`: [2](#0-1) 

The result is that the confirmation dialog presents:
- **Wallet Id:** `2` (a raw integer; users do not know which CAT this maps to)
- **Amount:** `1.000000` (no token name or symbol)

The `lookupCat` import is already commented out at the top of the file, confirming the feature was never completed: [3](#0-2) 

### Impact Explanation

A dApp that has been granted `chia_wallet.cat_spend` permission (checked at line 33 of `useWalletConnectCommand.tsx`) can freely choose any `wallet_id` value in its request: [4](#0-3) 

There is no guard that cross-validates the supplied `wallet_id` against the CAT type or name before presenting the dialog. The user sees only a number and a bare decimal amount. A malicious dApp can target `wallet_id=2` (a high-value CAT such as USDS or SBX) while the user believes they are approving a spend of a low-value or test CAT. If the user approves, the spend executes against the actual high-value wallet.

This satisfies the **High** impact criterion: *WalletConnect state causes a user to approve the wrong asset.*

### Likelihood Explanation

- The attacker only needs a WalletConnect pairing with `chia_wallet.cat_spend` in the approved command list — a normal, user-initiated pairing flow.
- The omission is structural (a commented-out TODO), not a transient race or edge case.
- The attack is reproducible locally with any two CAT wallets and a WalletConnect-capable dApp.

### Recommendation

Implement `lookupCat` (the commented-out TODO) to resolve `wallet_id` to a CAT display name and asset ID before rendering the confirmation dialog. At minimum, the `amount` field should display `<formatted_amount> <TOKEN_SYMBOL>` and the `wallet_id` field should be resolved to the wallet's human-readable name. Until `lookupCat` is implemented, the `wallet_id` field should not carry `hide: true` (it currently does not), but it should be resolved to a name rather than shown as a raw integer.

### Proof of Concept

1. Open Chia GUI with two CAT wallets: wallet_id=1 (low-value test CAT) and wallet_id=2 (high-value CAT, e.g., USDS).
2. Connect a WalletConnect dApp and grant it `chia_wallet.cat_spend` permission.
3. From the dApp, issue:
   ```json
   { "method": "chia_wallet.cat_spend", "params": { "wallet_id": 2, "address": "<attacker_address>", "amount": "1000000", "fee": "0" } }
   ```
4. Observe the confirmation dialog: it shows **Wallet Id: 2** and **Amount: 1.000000** with no token name.
5. A user who expected to spend their low-value CAT (wallet_id=1) cannot distinguish this from the high-value CAT spend and may approve.
6. On approval, `cat_spend` executes against wallet_id=2, transferring the high-value CAT to the attacker's address.
