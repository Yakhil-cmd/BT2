### Title
Period Mint Cap Exhaustion Causes `tick()` to Revert, Temporarily Freezing Reward Distribution — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The L1 `RewardSupplier.tick()` function processes L2→L1 mint-request messages atomically: it either mints the full computed batch or reverts entirely. Because `MintManager.mintRequest()` internally calls `PeriodMintLimit.checkAndUpdatePeriodicalQuota()`, any call to `tick()` whose batch amount would push the running period total past `PERIOD_MINT_CAP` (6.5 M STRK) causes the entire transaction to revert. The L2 mint-request messages are not consumed, and no tokens are bridged to the L2 reward supplier. Stakers' unclaimed rewards are frozen until the next weekly period begins.

---

### Finding Description

`tick()` in `RewardSupplier.sol` follows this sequence:

1. Compute `amountToMint = TOKENS_PER_MINT_REQUEST × numMsgsToConsume` (capped at `MAX_MESSAGES_TO_PROCESS_PER_TICK`).
2. Consume the L2→L1 messages via `consumeMessageFromL2`.
3. Call `mintManager().mintRequest(token(), amountToMint)`.
4. Deposit the minted tokens to the bridge. [1](#0-0) 

Inside `mintRequest`, `checkAndUpdatePeriodicalQuota` enforces:

```solidity
require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
``` [2](#0-1) 

Because Solidity transactions are atomic, a revert in step 3 rolls back steps 1–2 as well. The messages remain pending, but `tick()` will continue to revert for every call during the remainder of the current weekly period whenever the batch amount exceeds the remaining cap headroom.

The protocol does not attempt to process a smaller sub-batch that would fit within the remaining cap. `requiredMinting()` always returns the maximum consumable batch: [3](#0-2) 

So if `mintedThisPeriodBefore = 5 M` and `amountToMint = 2 M`, every subsequent `tick()` call reverts with `EXCEED_PERIOD_MINTING` until the period rolls over (up to 7 days).

---

### Impact Explanation

Stakers' rewards accumulate on L2 but cannot be funded because the L2 reward supplier never receives the bridged tokens. The `unclaimed_rewards` balance in the L2 reward supplier is not incremented, so any call to `transfer_rewards` that requires those funds will also fail. [4](#0-3) 

This constitutes **temporary freezing of unclaimed yield** for all stakers — up to one full week — matching the "High: Temporary freezing of funds / Permanent freezing of unclaimed yield" impact tier.

---

### Likelihood Explanation

The scenario is realistic under normal protocol operation:

- The L2 reward supplier sends mint requests every epoch. If multiple epochs' worth of requests accumulate (e.g., due to a period of `tick()` inactivity), a single `tick()` call may attempt to mint an amount that, combined with earlier mints in the same week, exceeds `PERIOD_MINT_CAP = 6_500_000 × 10^18`.
- `tick()` is a public, permissionless function — any caller can trigger it, but no caller can force it to succeed once the cap headroom is exhausted.
- The cap is fixed in code; there is no admin function to raise it mid-period. [5](#0-4) 

---

### Recommendation

Implement partial-batch processing in `tick()`: compute the maximum number of messages whose aggregate mint amount fits within the remaining period cap, and process only that sub-batch. This mirrors the fix recommended in the external report — allowing the operation to succeed partially rather than reverting entirely — and prevents rewards from being frozen when the cap is nearly exhausted.

---

### Proof of Concept

1. Assume `TOKENS_PER_MINT_REQUEST = 1_000_000 × 10^18` and `MAX_MESSAGES_TO_PROCESS_PER_TICK = 10`.
2. In week W, `tick()` is called twice successfully, minting 5 M STRK total. Remaining cap = 1.5 M.
3. Six more L2 mint-request messages accumulate (6 M STRK worth).
4. Any call to `tick()` now computes `numMsgsToConsume = min(6, 10) = 6`, `amountToMint = 6 M`.
5. `checkAndUpdatePeriodicalQuota`: `5 M + 6 M = 11 M > 6.5 M` → revert `EXCEED_PERIOD_MINTING`.
6. The 1.5 M remaining cap headroom is wasted; all 6 pending messages stay unconsumed.
7. The L2 reward supplier receives no tokens; `unclaimed_rewards` is not updated; staker reward claims revert until week W+1. [2](#0-1) [6](#0-5)

        uint256[] memory messagePayload = new uint256[](1);
        messagePayload[0] = TOKENS_PER_MINT_REQUEST;

        bytes32 msgHash = l2ToL1MsgHash(mintRequestSource(), address(this), messagePayload);
        // Limit the number of msgs to consume to limit.
        uint256 numMsgsToConsume = Math.min(
            messagingContract().l2ToL1Messages(msgHash),
            MAX_MESSAGES_TO_PROCESS_PER_TICK
        );

        return (TOKENS_PER_MINT_REQUEST * numMsgsToConsume, numMsgsToConsume);
    }

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }

    function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
        bytes32 periodSlot = periodAccountingSlot(token);
        uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
        uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
        require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
        periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;
    }
