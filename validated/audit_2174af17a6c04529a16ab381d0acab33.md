### Title
Unverified Participant Contributions in CKD Protocol Allow Malicious Participant to Corrupt the Derived Key — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly aggregates participant-supplied group elements `(norm_big_y, norm_big_c)` without any cryptographic verification. A single malicious participant can substitute arbitrary curve points, silently corrupting the final `CKDOutput` that the application uses to reconstruct its confidential derived key.

---

### Finding Description

In `do_ckd_coordinator` the coordinator collects each participant's share of the blinded derived key and sums them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();   // no check
    norm_big_c += participant_output.big_c();   // no check
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

The values that *should* arrive are computed by `compute_signature_share`:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
``` [2](#0-1) 

There is no zero-knowledge proof, no consistency check against the participant's public key share, and no other mechanism that would let the coordinator detect a deviation. The `KeygenOutput` passed to every participant contains only the aggregate `public_key` and the participant's own `private_share`; individual public shares (`x_i · G`) are not stored and are therefore unavailable to the coordinator for verification. [3](#0-2) 

---

### Impact Explanation

The final output is:

```
Y   = Σ norm_big_y_i
C   = Σ norm_big_c_i
```

The application recovers the derived key as `C − app_sk · Y`. If a malicious participant substitutes `(norm_big_y_i + Δ_Y, norm_big_c_i + Δ_C)` for arbitrary curve points `Δ_Y, Δ_C`, the coordinator produces:

```
Y'  = Y_correct + Δ_Y
C'  = C_correct + Δ_C
```

The application then computes:

```
C' − app_sk · Y' = msk · H(pk ‖ app_id) + Δ_C − app_sk · Δ_Y
```

The derived key is silently wrong. The application has no way to detect the corruption because `CKDOutput` carries no authentication tag and `unmask` performs no validity check. This matches the allowed impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs (High)**. [4](#0-3) 

---

### Likelihood Explanation

Every participant in the CKD session is an unprivileged library caller who supplies their own `(norm_big_y, norm_big_c)` pair. No special access, no leaked keys, and no external dependency is required. Because the protocol sums *all* participants' contributions (there is no threshold sub-selection at the aggregation step), a single corrupted participant is sufficient to produce a wrong output. The attack is trivially reachable by any participant who deviates from the honest computation. [5](#0-4) 

---

### Recommendation

1. **Store per-participant public key shares** (`x_i · G_2`) as part of `KeygenOutput` or a separate public key package, so the coordinator has the material needed to verify contributions.
2. **Require a proof of correct encryption** alongside each `(norm_big_y_i, norm_big_c_i)`. A standard approach is a Chaum–Pedersen DLEQ proof showing that the discrete-log relationship between `norm_big_y_i` and `norm_big_c_i − λ_i · x_i · H(pk ‖ app_id)` with respect to `G` and `app_pk` is consistent with the participant's known public share.
3. **Reject and abort** if any participant's proof fails to verify before aggregating.

---

### Proof of Concept

```
Setup: n participants, coordinator C, application with key pair (app_sk, app_pk).
       Honest participants hold shares x_i of msk.

Attack (malicious participant M):
  1. M runs compute_signature_share honestly to obtain (norm_big_y_M, norm_big_c_M).
  2. M picks arbitrary non-zero Δ_C ∈ G1.
  3. M sends (norm_big_y_M, norm_big_c_M + Δ_C) to the coordinator instead.

Coordinator:
  4. Aggregates all contributions without verification.
  5. Returns CKDOutput { Y = Y_correct, C = C_correct + Δ_C }.

Application:
  6. Calls ckd_output.unmask(app_sk):
       result = (C_correct + Δ_C) − app_sk · Y_correct
              = msk · H(pk ‖ app_id) + Δ_C   ← wrong key, silently accepted
  7. Application uses the wrong derived key; no error is raised.
``` [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
