### Title
State Committed Instead of Rolled Back on `TxError::Internal` in Transaction Loop - (`basic_bootloader/src/bootloader/block_flow/zk/tx_loop.rs`)

---

### Summary

When `process_transaction` returns `TxError::Internal`, the ZK transaction loop calls `system.finish_global_frame(None)` — which **commits** the pre-transaction snapshot frame — instead of rolling it back via `finish_global_frame(Some(&pre_tx_rollback_handle))`. Any state changes made during the failed transaction are therefore permanently committed into the block's in-memory state, even though the transaction did not complete successfully. A developer-acknowledged `TODO` comment at the exact line confirms this is an unresolved design question.

---

### Finding Description

In `basic_bootloader/src/bootloader/block_flow/zk/tx_loop.rs`, before each transaction is processed, a rollback snapshot is taken:

```rust
let pre_tx_rollback_handle = system.start_global_frame()?;
``` [1](#0-0) 

This handle exists precisely so that the entire transaction's state changes can be reverted if the transaction must be invalidated (e.g., block limits exceeded). The validation-error and block-limit paths both use it correctly:

```rust
// Validation error → rollback
system.finish_global_frame(Some(&pre_tx_rollback_handle))?;

// Block limit exceeded → rollback
system.finish_global_frame(Some(&pre_tx_rollback_handle))?;
``` [2](#0-1) [3](#0-2) 

However, the **internal error path** does not roll back:

```rust
Err(TxError::Internal(err)) => {
    system_log!(system, "Tx execution result: Internal error = {err:?}\n",);
    // Finish the frame opened before processing the tx
    system.finish_global_frame(None)?; // TODO should we use pre_tx_rollback_handle here?
    return Err(err);
}
``` [4](#0-3) 

`finish_global_frame(None)` **commits** the frame. The `pre_tx_rollback_handle` is dropped unused. All storage writes, nonce increments, balance changes, log emissions, and event emissions that occurred inside `process_transaction` before the internal error are merged into the parent frame as if the transaction had succeeded.

The `finish_io_frame` implementation confirms that passing `None` means no rollback occurs across all sub-systems (storage, transient storage, logs, events, interop roots): [5](#0-4) 

The `LogsStorage` and `EventsStorage` both support rollback via their `finish_frame(rollback_handle)` methods, but these are never invoked with the snapshot in this path: [6](#0-5) 

---

### Impact Explanation

The committed-but-invalid state changes persist in the in-memory block state. Concretely:

- **Nonce increments** from the failed transaction remain, making the sender's account appear to have a higher nonce than it should.
- **Balance changes** (fee pre-charges, value transfers) from the failed transaction remain committed.
- **L2→L1 logs** emitted before the internal error remain in `logs_storage`, inflating the log count and corrupting the Merkle tree root used in the block public input.
- **Storage slot writes** from the failed transaction remain, corrupting the state root.

Because the block then returns `Err(err)`, the sequencer (forward system) must decide how to proceed. If it attempts to seal the block or start the next block from the now-corrupted in-memory state, subsequent transactions execute against incorrect account nonces, balances, and storage values. This is a **state-transition correctness bug**: the proven state root would not match the true post-execution state. [7](#0-6) 

---

### Likelihood Explanation

`TxError::Internal` wraps `InternalError`, which is documented as "should not be triggered by user input." However, several code paths inside `process_transaction` convert resource-exhaustion or arithmetic edge cases into internal errors (e.g., `internal_error!("gp*gl")` for overflow in `gas_price * gas_limit`). A transaction with carefully chosen `max_fee_per_gas` and `gas_limit` values that overflow `U256` multiplication could reach such a path. Additionally, the `TODO` comment at the exact line signals that the developers themselves have not resolved whether this is correct, raising the probability that the behavior is unintentional. [8](#0-7) 

---

### Recommendation

Replace the commit with a rollback in the internal-error arm:

```rust
Err(TxError::Internal(err)) => {
    system_log!(system, "Tx execution result: Internal error = {err:?}\n",);
    // Roll back all state changes made during this transaction
    system.finish_global_frame(Some(&pre_tx_rollback_handle))?;
    return Err(err);
}
```

This mirrors the handling of validation errors and block-limit errors, ensuring that no partial state from a failed transaction leaks into the block state regardless of the error category.

---

### Proof of Concept

1. Craft an L1→L2 (priority) transaction with `max_fee_per_gas = U256::MAX` and `gas_limit = 2`, so that `gas_price * gas_limit` overflows `U256` inside `process_l1_transaction`, triggering `internal_error!("gp*gl")`.
2. Before the overflow is reached, the transaction has already incremented the sender's nonce (via `increment_nonce` during validation or intrinsic processing) and potentially emitted a partial log.
3. `process_transaction` returns `TxError::Internal`.
4. `tx_loop.rs` line 111 calls `finish_global_frame(None)`, committing the nonce increment and any partial log.
5. The block returns an error, but the in-memory state now has the sender's nonce incremented and a spurious log entry.
6. If the sequencer starts the next block from this state, the sender's nonce is off-by-one, causing all subsequent transactions from that sender to fail nonce validation, and the L2→L1 log Merkle root is incorrect. [4](#0-3) [8](#0-7)
