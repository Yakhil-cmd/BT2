### Title
Unauthorized Transfer of Native Bank Tokens via Missing Caller Authorization in Bank Precompile `transfer` - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The `transfer` method of the bank precompile (`BankContract.Run`) accepts an arbitrary `sender` address as a caller-supplied argument and uses it directly as the `from` account in `bankKeeper.SendCoins`, without verifying that the supplied `sender` matches `contract.Caller()`. Any EVM contract can therefore drain native bank tokens (denom `evm/<contract_address>`) from any user who holds them, without the user's consent or approval.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case of `BankContract.Run` reads:

```go
sender    := args[0].(common.Address)   // attacker-controlled
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
...
from  := sdk.AccAddress(sender.Bytes())
denom := EVMDenom(contract.Caller())    // "evm/<calling_contract>"
amt   := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The denom is correctly scoped to `evm/<contract.Caller()>`, but `from` is taken verbatim from `args[0]` â€” a value the calling contract fully controls. There is no guard of the form `require sender == contract.Caller()`. The `checkBlockedAddr` check only validates the **recipient**, not the sender. [2](#0-1) 

The `IBankModule` interface exposes `transfer(address, address, uint256)` as a payable function callable by any EVM contract: [3](#0-2) 

The bank precompile is registered at the fixed address `0x0000000000000000000000000000000000000064` and is reachable by any unprivileged EVM contract. [4](#0-3) 

### Impact Explanation

**Critical â€” Unauthorized transfer of CRC20/native bank assets.**

A malicious contract deployed at address `0xMalicious` manages the native denom `evm/0xMalicious`. Any user who holds tokens of that denom (e.g., received via the contract's `mint` path) can have their entire balance drained by the contract owner calling:

```solidity
bank.transfer(victim, attacker, victimBalance);
```

The bank keeper executes `SendCoins(ctx, victim, attacker, coins)` with no authorization from `victim`. This is a direct, unauthorized transfer of user funds with no recourse.

The `TestBank.sol` reference contract illustrates the intended pattern â€” passing `msg.sender` as the sender â€” but the precompile enforces nothing: [5](#0-4) 

### Likelihood Explanation

Any unprivileged actor can deploy an EVM contract and call the bank precompile's `transfer` method. No admin key, governance vote, or special permission is required. The only precondition is that victims hold tokens of the denom `evm/<attacker_contract>`, which is satisfied whenever the attacker's contract has previously minted tokens to users (a normal operational flow for any token contract using the bank precompile).

### Recommendation

Add an authorization check in the `TransferMethodName` case to ensure the supplied `sender` matches the EVM caller:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the caller")
}
```

This mirrors the authorization model used by `mint` and `burn`, where the denom is derived from `contract.Caller()` and the operation is implicitly authorized by the calling contract acting on its own behalf. [6](#0-5) 

### Proof of Concept

1. Attacker deploys `MaliciousToken` at `0xMalicious`. The native denom is `evm/0xMalicious`.
2. `MaliciousToken` calls `bank.mint(alice, 1000)` â€” Alice now holds 1000 `evm/0xMalicious` tokens.
3. Attacker calls `bank.transfer(alice, attacker, 1000)` from `MaliciousToken`.
4. The precompile sets `from = alice`, `denom = "evm/0xMalicious"`, and executes `bankKeeper.SendCoins(ctx, alice, attacker, 1000 evm/0xMalicious)`.
5. Alice's balance is drained to zero with no signature or approval from Alice.

The root cause is identical in structure to the Flan `safeTransfer` bug: the transfer function uses the wrong identity â€” a caller-supplied address instead of the actual authenticated caller â€” leading to unauthorized fund movement. [7](#0-6)
