### Title
Unrestricted `update_rewards` Allows Any Caller to Steal or Permanently Freeze Block Rewards - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in `src/staking/staking.cairo` is documented to be callable only by the Starknet sequencer, but the implementation contains no access-control check. Any unprivileged caller can invoke it once per block for any valid staker, either redirecting block rewards to an undeserving staker or permanently destroying the block's reward allocation by passing `disable_rewards: true`.

### Finding Description
The protocol spec explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` line 1449 only calls `general_prerequisites()` (which exclusively checks the pause flag) and then proceeds with no caller-identity verification:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks is_paused
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... no role / sequencer check anywhere
``` [1](#0-0) 

The function is gated only by a per-block call limit (`last_reward_block`). Once it is called for any valid staker in a given block, `last_reward_block` is updated and no further call is possible for that block:

```cairo
self.last_reward_block.write(current_block_number);
if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [2](#0-1) 

Compare with `update_rewards_from_attestation_contract`, which correctly adds an explicit caller check after `general_prerequisites()`:

```cairo
self.assert_caller_is_attestation_contract();
``` [3](#0-2) 

No equivalent guard exists in `update_rewards`.

The spec's access-control requirement is documented but unenforced: [4](#0-3) 

### Impact Explanation
Two distinct high-severity attack paths exist:

**Path A – Permanent freezing of block rewards (griefing)**
An attacker calls `update_rewards(staker_address: any_valid_active_staker, disable_rewards: true)`. The function writes the current block number to `last_reward_block` and returns immediately without distributing any rewards. The legitimate sequencer's subsequent call for the same block reverts with `REWARDS_ALREADY_UPDATED`. The entire block's reward allocation is permanently lost — no staker receives it, and it cannot be recovered.

**Path B – Theft of unclaimed yield (reward misrouting)**
An attacker who is themselves a valid staker calls `update_rewards(staker_address: attacker_staker, disable_rewards: false)` before the sequencer's call for the legitimate block producer. The attacker's staker receives the block rewards proportional to their stake, while the legitimate block producer receives nothing for that block. The rewards are transferred out of the `RewardSupplier` to the attacker's staker and their pool:

```cairo
staker_info.unclaimed_rewards_own += staker_rewards;
// ...
self.write_staker_info(:staker_address, :staker_info);
``` [5](#0-4) 

Both paths match the allowed impact scope: **High: Theft of unclaimed yield** and **High: Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The current Starknet sequencer is centralized (StarkWare), which means the sequencer controls transaction ordering and can place its own `update_rewards` call first in each block. This reduces the practical exploitability under normal operation. However:

1. The contract itself provides zero enforcement — the vulnerability is fully reachable by any external caller with no preconditions beyond holding a valid staker position (Path B) or simply knowing any active staker address (Path A).
2. Any future move toward a decentralized or permissionless sequencer would immediately make this exploitable at scale.
3. Path A (griefing with `disable_rewards: true`) requires no staker position at all — only knowledge of one active staker address.

**Likelihood: Low (2/10)** under the current centralized sequencer; escalates significantly if sequencer decentralization proceeds.

### Recommendation
Add an explicit sequencer/operator role check at the top of `update_rewards`, mirroring the pattern used in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.roles.only_operator();   // <-- add this guard
    ...
}
```

Alternatively, introduce a dedicated `SEQUENCER` role and enforce it here. The fix must be at the contract level; relying on sequencer-level transaction ordering is not a security guarantee.

### Proof of Concept
1. Deploy the staking system and advance to the consensus-rewards epoch.
2. Stake as `attacker_staker` with any non-zero amount; wait K epochs for the balance to become effective.
3. At the start of a new block (before the sequencer's `update_rewards` call), call:
   ```
   staking.update_rewards(staker_address: attacker_staker, disable_rewards: false)
   ```
4. Observe that `attacker_staker.unclaimed_rewards_own` increases by the full block reward.
5. The sequencer's subsequent `update_rewards` call for the legitimate block producer reverts with `REWARDS_ALREADY_UPDATED`.
6. The legitimate block producer's `unclaimed_rewards_own` is unchanged — their block reward is permanently lost.

For the griefing variant, replace step 3 with:
```
staking.update_rewards(staker_address: any_active_staker, disable_rewards: true)
```
No staker position is required; the block's entire reward allocation is destroyed.
