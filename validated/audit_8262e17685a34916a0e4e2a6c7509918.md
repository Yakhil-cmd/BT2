### Title
`skip_stateful_validations` Bypasses `__validate__` for Nonce-1 Invoke When Account Has Any Mempool Transaction, Not Only a `deploy_account` - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function is intended to skip the `__validate__` entry-point call for an invoke with nonce=1 only when the account's `deploy_account` transaction is still pending in the mempool (UX improvement). However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from that address — including future-nonce invokes that passed their own validation. An attacker who already has a valid future-nonce invoke in the mempool can therefore submit a second invoke with nonce=1 carrying an **invalid signature** (or any payload that would fail `__validate__`) and have it admitted to the mempool without the account's validation entry point ever being called.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` and its return value is forwarded directly to `run_validate_entry_point` as the `skip_validate` flag:

```
run_pre_validation_checks
  └─ skip_stateful_validations  →  bool
       └─ run_validate_entry_point(skip_validate = true)
            └─ ExecutionFlags { validate: !skip_validate }   // __validate__ NOT called
```

The skip condition:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  L437-L456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...;
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  L697-L700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` whenever the address appears in the pool or in a recently committed block — regardless of transaction type. The code comment acknowledges this explicitly:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**." [3](#0-2) 

The flaw is that `__validate__` is **transaction-specific**: a valid signature for a nonce-5 invoke does not imply a valid signature for a nonce-1 invoke. Skipping `__validate__` for the nonce-1 invoke because a nonce-5 invoke passed its own `__validate__` breaks the per-transaction signature invariant.

When `skip_validate = true` is propagated into `run_validate_entry_point`, the `ExecutionFlags` are built with `validate: false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  L308-L312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };
``` [4](#0-3) 

`StatefulValidator::perform_validations` then short-circuits before calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  L79-L81
if !tx.execution_flags.validate {
    return Ok(());
}
``` [5](#0-4) 

The transaction is forwarded to the mempool without any account-level signature check.

---

### Impact Explanation

An attacker controlling an already-deployed account (on-chain nonce = 0, contract exists) can:

1. Submit a valid invoke with nonce = 5 — passes `__validate__` normally; the account now appears in `account_tx_in_pool_or_recent_block`.
2. Submit a second invoke with nonce = 1 carrying an **invalid signature** (or any calldata that would fail `__validate__`).

Step 2 satisfies all three skip conditions (`Invoke`, `tx.nonce() == 1`, `account_nonce == 0`) and `account_tx_in_pool_or_recent_block` returns `true` due to step 1. The gateway admits the invalid transaction to the mempool without calling `__validate__`.

The admitted transaction will be reverted during block execution (the batcher calls `__validate__` unconditionally), but the mempool admission invariant — *every accepted transaction has passed the account's validation entry point* — is broken. This matches the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The trigger is fully unprivileged: any account holder can exploit it. The only prerequisite is having one valid future-nonce invoke already in the mempool, which is a normal user action. The nonce-1 window is narrow but deterministic and repeatable across different accounts.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that inspects transaction type, not merely address presence. Alternatively, restrict the skip to cases where `account_nonce == 0` **and** the account contract does not yet exist in state (i.e., `get_class_hash_at(address) == ClassHash::default()`), which is the only scenario where skipping `__validate__` is semantically safe.

---

### Proof of Concept

```
1. Deploy account A on-chain  →  on-chain nonce = 0, contract exists.

2. POST /add_transaction  (invoke, sender=A, nonce=5, valid_signature)
   Gateway:
     validate_nonce:            0 ≤ 5 ≤ 200  ✓
     skip_stateful_validations: nonce≠1       → false
     run_validate_entry_point:  __validate__  called, passes  ✓
   Mempool: account A now in pool.

3. POST /add_transaction  (invoke, sender=A, nonce=1, INVALID_signature)
   Gateway:
     validate_nonce:            0 ≤ 1 ≤ 200  ✓
     validate_by_mempool:       no duplicate  ✓
     skip_stateful_validations: nonce==1 ∧ account_nonce==0
                                account_tx_in_pool_or_recent_block(A) == true  (step 2)
                                → returns true  (SKIP)
     run_validate_entry_point:  validate=false  →  __validate__ NOT called
   Mempool: invalid transaction admitted  ✗

4. Batcher executes block:
     __validate__ called with validate=true  →  fails  →  tx reverted.
     Invalid transaction was in the mempool without signature verification.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
