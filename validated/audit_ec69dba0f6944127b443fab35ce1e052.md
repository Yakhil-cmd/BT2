### Title
Missing `msg_hash == 0` Rejection in OT-Based ECDSA `sign` Allows Coordinator to Reconstruct Presign Secret `σ = k·x` — (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `sign` function accepts a zero `msg_hash` without rejection. When `msg_hash = 0`, each participant's signature share degenerates to `s_i = r · λ_i · σ_i`, directly exposing their linearized share of the presign secret `σ = k·x`. A malicious coordinator can reconstruct `σ` from threshold-many shares, constituting disclosure of presign secrets. The robust ECDSA `sign` function has an explicit guard against this; the OT-based variant does not.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/sign.rs`, the public entry point `sign` accepts any `Scalar` as `msg_hash`, including zero: [1](#0-0) 

The signature share computation in `compute_signature_share` is:

```
s_i = msg_hash * k_i + r * sigma_i
``` [2](#0-1) 

When `msg_hash = 0`, this collapses to:

```
s_i = r * sigma_i
```

Since `r` (the x-coordinate of `R`) is public, the coordinator can compute `σ_i = s_i / r` for each received share, then use Lagrange interpolation over threshold-many shares to reconstruct the full presign secret `σ = k·x`.

By contrast, the robust ECDSA `sign` function explicitly rejects `msg_hash = 0`: [3](#0-2) 

The security documentation for the robust scheme explains the rationale: [4](#0-3) 

The OT-based ECDSA `sign` function carries only a documentation warning ("Allowing the signing of arbitrary scalars *is* a security risk") but no enforcement: [5](#0-4) 

The `PresignOutput` struct confirms that `sigma` is a secret field that must not be disclosed: [6](#0-5) 

---

### Impact Explanation

`σ = k·x` is a presign secret: it is the product of the ephemeral nonce `k` and the long-term secret key `x`. Disclosing `σ` to the coordinator falls directly under the Critical impact category — *disclosure of presign secrets and nonce material*. Additionally, if the coordinator can reuse the same presignature in a second session with `h ≠ 0`, they obtain both `σ = k·x` and `s = h·k + r·σ`, from which `k = (s − r·σ)/h` and then `x = σ/k`, fully extracting the aggregate secret key.

---

### Likelihood Explanation

The coordinator is the direct caller of `sign` and supplies `msg_hash` as a free parameter. No validation in the function body, no type-level constraint, and no caller-side enforcement prevents passing `Scalar::ZERO`. The attack requires no special privileges beyond being the designated coordinator — a role that is explicitly trusted to be honest in the protocol but is not cryptographically constrained from passing a zero hash.

---

### Recommendation

Add a zero-hash guard in `src/ecdsa/ot_based_ecdsa/sign.rs` immediately after the participant-count check, mirroring the existing guard in the robust ECDSA variant:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0: signing with a zero hash exposes presign secret σ".to_string(),
    ));
}
``` [7](#0-6) 

---

### Proof of Concept

1. A malicious coordinator calls:
   ```rust
   sign(participants, coordinator, threshold, me, public_key, presignature, Scalar::ZERO)
   ```
2. Each honest participant executes `compute_signature_share` and produces:
   ```
   s_i = 0 * (λ_i * k) + r * (λ_i * σ) = r * λ_i * σ_i
   ```
3. The coordinator receives all `s_i` values and computes `σ_i = s_i / r` for each participant (since `r` is public from `presignature.big_r`).
4. The coordinator applies Lagrange interpolation over the threshold-many `(participant_id, σ_i)` pairs to reconstruct `σ = k·x`.
5. `σ` is a presign secret. If the coordinator also possesses a second signing session over the same presignature with any `h ≠ 0`, they solve `k = (s − r·σ)/h` and `x = σ/k`, fully recovering the long-term secret key. [2](#0-1) [8](#0-7)

            role: "coordinator",
            participant: coordinator,

        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),

    me: Participant,
    presignature: &RerandomizedPresignOutput,
