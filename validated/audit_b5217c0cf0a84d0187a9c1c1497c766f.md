### Title
Missing Zero-Address Validation in `change_reward_address` Permanently Freezes Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both the staking contract and the delegation pool contract expose a `change_reward_address` function that accepts a `ContractAddress` parameter with no zero-address guard. A staker or pool member who accidentally (or via a malformed front-end call) passes the zero address will silently overwrite their reward address with `0x0`. All subsequent reward claims will transfer tokens to the zero address, permanently freezing unclaimed yield with no recovery path.

---

### Finding Description

**Staking contract** — `src/staking/staking.cairo`, `change_reward_address` (line 517):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;
    staker_info.reward_address = reward_address;   // ← zero address accepted
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
```

The only validation is that `reward_address` is not a registered token address. There is no `assert!(reward_address.is_non_zero(), ...)` guard.

**Pool contract** — `src/pool/pool.cairo`, `change_reward_address` (line 505):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let pool_member = get_caller_address();
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    ...
    pool_member_info.reward_address = reward_address;   // ← zero address accepted
    self.write_pool_member_info(:pool_member, :pool_member_info);
    ...
}
```

Same pattern: only the token-address check, no zero-address check.

The codebase already recognises zero-address callers as invalid — `assert_caller_is_not_zero` exists in `src/staking/utils.cairo` (line 62-64) and `Error::CALLER_IS_ZERO_ADDRESS` is defined in `src/staking/errors.cairo` (line 100) — but this protection is never applied to the *destination* reward address.

---

### Impact Explanation

When `claim_rewards` is later called (by the staker or pool member, or by anyone on their behalf), the contract transfers accumulated yield to `reward_address`. If that address is `0x0`, the ERC-20 `transfer` call sends tokens to the zero address. On StarkNet, the zero address is not a burner contract with a recovery mechanism; the tokens are permanently inaccessible. This constitutes **permanent freezing of unclaimed yield**, which is a High-severity impact under the allowed scope.

---

### Likelihood Explanation

The likelihood is **medium-high**:

- `change_reward_address` is a routine self-service function callable by any staker or pool member — no privileged role required.
- Front-ends that omit or fail to populate the address field commonly default to the zero address (exactly the scenario described in the reference report).
- Scripts or CLI tools that pass an uninitialized `felt252` / `ContractAddress` variable will silently submit `0x0`.
- There is no on-chain warning, confirmation step, or time-lock; the change is immediate and irreversible.
- The existing codebase test at `src/staking/tests/test.cairo` line 1851 demonstrates that `declare_operational_address(staker_address: Zero::zero())` is accepted without error, showing zero-address inputs are not universally rejected.

---

### Recommendation

Add a non-zero assertion at the top of both `change_reward_address` implementations, mirroring the existing `assert_caller_is_not_zero` pattern:

```cairo
// src/staking/staking.cairo  — change_reward_address
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);

// src/pool/pool.cairo  — change_reward_address
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);
```

A corresponding `REWARD_ADDRESS_IS_ZERO` error variant should be added to `src/staking/errors.cairo`.

---

### Proof of Concept

1. Staker `A` is registered with `reward_address = 0xABCD`.
2. Staker `A` calls `staking.change_reward_address(reward_address: 0x0)` — perhaps via a front-end that left the field blank.
3. The only check (`does_token_exist(0x0)`) returns `false`; the call succeeds.
4. `staker_info.reward_address` is now `0x0` in storage.
5. Staker `A` (or anyone) later calls `claim_rewards`; the contract transfers all accumulated STRK yield to `0x0`.
6. Tokens are permanently lost. Staker `A` cannot recover them and cannot re-stake or re-delegate to a new reward address without first calling `change_reward_address` again — but the already-transferred yield is gone forever.

The same sequence applies to any pool member via `pool.change_reward_address`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
