### Title
Unbounded `stakers` Vec Growth Causes Unbounded Gas Consumption in `get_stakers()` - (File: src/staking/staking.cairo)

### Summary

The `stakers` storage Vec in `staking.cairo` is append-only: staker addresses are pushed on `stake()` but **never removed** on `unstake_action()`. The `get_stakers()` function iterates over the entire Vec on every call. Any unprivileged actor can bloat this Vec by repeatedly staking with fresh addresses and completing the unstake cycle, causing `get_stakers()` to consume unbounded gas.

### Finding Description

In `staking.cairo`, the `stakers` field is declared as a `Vec<ContractAddress>` with an explicit design note:

> `/// **Note**: Stakers are not removed from this vector when they unstake.` [1](#0-0) 

On every `stake()` call, the staker address is appended: [2](#0-1) 

On `unstake_action()`, the staker's `staker_info` record is deleted and funds are returned, but the address is **never removed** from `self.stakers`: [3](#0-2) 

The `get_stakers()` function iterates over the **full range** of the Vec on every invocation, calling `is_staker_active()` to skip inactive entries — but the iteration cost itself is proportional to the total number of entries ever added, including all stale (unstaked) ones: [4](#0-3) 

`is_staker_active()` returns `false` for stakers whose `staker_info` has been deleted, so they are skipped in the output — but the storage reads still occur for every entry: [5](#0-4) 

The flow test `DiverseStakerVecFlow` explicitly confirms that a staker who completed `unstake_action` remains in the Vec: [6](#0-5) 

### Impact Explanation

`get_stakers()` is part of the `IStakingConsensus` interface and is the primary mechanism for the consensus layer to determine the active validator set. As the Vec grows without bound, the gas cost of iterating it grows linearly with the total number of historical stakers. With a sufficiently large Vec, the function will exceed Starknet's per-transaction resource limits, permanently preventing the consensus layer from querying the active staker set. This matches the **Unbounded gas consumption** impact in the allowed scope.

### Likelihood Explanation

Any unprivileged actor with access to the minimum stake amount can execute this attack. The attacker creates N fresh addresses, stakes the minimum amount from each, waits for the `exit_wait_window`, and calls `unstake_action` for each. Each cycle permanently adds one entry to the `stakers` Vec. The attack is economically bounded only by the `min_stake` parameter and the exit wait window, but requires no privileged access. The `assert_staker_address_not_reused` guard prevents reuse of the same address, but fresh addresses are trivially generated on Starknet. [7](#0-6) 

### Recommendation

Remove staker addresses from the `stakers` Vec when `unstake_action` completes, or replace the append-only `Vec` with a data structure that supports deletion (e.g., a swap-and-pop pattern or an `IterableMap`). Alternatively, maintain a separate active-staker count and enforce an upper bound on the Vec size to prevent griefing.

### Proof of Concept

```
Step 0: Attacker generates N fresh addresses: ADDR_1, ADDR_2, ..., ADDR_N.

Step 1: For each ADDR_i:
  - Call stake(reward_address, operational_address, min_stake) from ADDR_i.
  - stakers Vec length increases by 1.

Step 2: For each ADDR_i:
  - Call unstake_intent() from ADDR_i.
  - Wait exit_wait_window seconds.
  - Call unstake_action(ADDR_i).
  - staker_info[ADDR_i] is deleted, but ADDR_i remains in stakers Vec.

Step 3: After N cycles, stakers Vec has length N (all stale entries).

Step 4: Any call to get_stakers(epoch_id) must iterate all N entries,
  performing a storage read + is_staker_active() check for each.
  For large N, this exceeds Starknet's per-transaction gas/step limits,
  causing get_stakers() to revert and permanently blocking the
  consensus layer from querying the active validator set.
```
