### Title
Permanent ERC-20 Token Loss When `exitToNear` NEAR-Side Transfer Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary

The `exitToNear` precompile burns ERC-20 tokens on Aurora and initiates a NEAR-side NEP-141 transfer. When the NEAR-side transfer fails permanently (e.g., unregistered recipient), the burned ERC-20 tokens are irrecoverable because the `error_refund` compile-time feature — which gates the entire refund mechanism — is absent from the production build configuration. There is no admin retry, no user rescue path, and no fallback.

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` constructs `ExitToNearPrecompileCallbackArgs` with a compile-time conditional:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a standard ERC-20 exit (non-wNEAR), `transfer_near` is also `None`, so `callback_args` equals the `Default` value. The code then skips attaching any callback entirely:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

This means when the NEAR-side `ft_transfer` fails, there is no `exit_to_near_precompile_callback` invocation at all. Even in the wNEAR case where a callback is attached (because `transfer_near` is `Some`), the callback's failure branch is:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← taken when refund is None; no action
};
``` [3](#0-2) 

The `error_refund` feature is **not** a default feature and is **not** included in the production build. The `Makefile.toml` defines the production build features as:

```
CARGO_FEATURES_BUILD = "contract"
``` [4](#0-3) 

And `engine/Cargo.toml` confirms `error_refund` is a non-default, opt-in feature:

```toml
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [5](#0-4) 

The `refund_on_error` function, which would re-mint the burned ERC-20 tokens, is therefore never reachable in production: [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls `exitToNear` for an ERC-20 token:
1. The ERC-20 tokens are burned on Aurora (irreversible within the EVM).
2. A NEAR-side `ft_transfer` promise is dispatched.
3. If the promise fails (e.g., recipient not registered with the NEP-141 contract), no callback fires (for non-wNEAR) or the callback fires but takes the `else { None }` branch (for wNEAR).
4. The burned ERC-20 tokens are gone. The NEP-141 tokens remain in Aurora's account. The user has no mechanism to recover either.

The test suite explicitly documents this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

### Likelihood Explanation

**Medium.** The `exitToNear` precompile is a standard, user-facing bridge operation. A NEAR-side transfer failure is a realistic and documented scenario — the test itself uses an unregistered recipient to trigger it. Any user who specifies a recipient NEAR account that is not registered with the target NEP-141 contract will permanently lose their tokens. This requires no special privilege; any EVM token holder can trigger it.

### Recommendation

Enable the `error_refund` feature in the production build by adding it to `CARGO_FEATURES_BUILD` in `Makefile.toml`:

```
CARGO_FEATURES_BUILD = "contract,error_refund"
```

Alternatively, remove the feature gate entirely and unconditionally populate `refund` in `ExitToNearPrecompileCallbackArgs`, making the refund path always active. The `refund_call_args` function already exists and is correct; it is only excluded by the compile-time gate. [8](#0-7) 

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora as an ERC-20.
2. As an EVM user holding the ERC-20, call the `exitToNear` precompile specifying a recipient NEAR account that is **not registered** with the NEP-141 contract.
3. The ERC-20 tokens are burned. The NEAR-side `ft_transfer` fails.
4. Because `CARGO_FEATURES_BUILD = "contract"` (no `error_refund`), `callback_args.refund` is `None`, `callback_args == Default`, and no callback is attached to the promise chain.
5. Observe: ERC-20 balance is reduced by the exit amount; the NEP-141 tokens remain in Aurora's account; the user has no recovery path.

This is confirmed by the existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs`, which asserts the reduced balance under `#[cfg(not(feature = "error_refund"))]`. [9](#0-8)

fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}
