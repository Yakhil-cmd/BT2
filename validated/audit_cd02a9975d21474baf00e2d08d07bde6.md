### Title
Commission Can Be Increased Immediately via `set_commission_commitment` Without Delay Period for Delegators to Exit — (File: `src/staking/staking.cairo`)

---

### Summary

A staker can atomically call `set_commission_commitment` (setting `max_commission = 10000`) followed by `set_commission` (setting `commission = 10000`) to raise their pool commission to 100% with no delay. Because the commission change is written to storage immediately and delegators are subject to the `exit_wait_window` lock-up, delegators cannot exit before the new commission rate applies to their rewards. The staker captures all delegator yield for the duration of the lock-up window.

---

### Finding Description

The `set_commission_commitment` mechanism was designed to give delegators predictability: a staker commits to a `max_commission` ceiling, and `set_commission` is then constrained to stay at or below that ceiling while the commitment is active.

However, the implementation contains no delay between setting the commitment and applying a commission increase. The `update_commission` internal function, called by `set_commission`, only checks:

```
commission <= commitment.max_commission   // upper bound
commission != old_commission              // no-op guard
```

There is no lower-bound check and no epoch-delay before the new commission takes effect. The new value is written to storage immediately:

```cairo
// staking.cairo L1600
staker_pool_info.commission.write(Option::Some(commission));
```

The codebase itself acknowledges this gap with a comment directly above `set_commission_commitment`:

> **Note**: Current commission increase safeguards still allow for sudden commission changes.

A staker can therefore execute the following two calls in the same block:

1. `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`
2. `set_commission(commission: 10000)`

This is confirmed by the existing test at `src/staking/tests/test.cairo` lines 2099–2111, which demonstrates a staker increasing commission to `max_commission` in the same epoch the commitment is set.

The commission is read by the pool contract at reward-distribution time via `contract_parameters_v1()`, so the new 100% commission applies to all rewards distributed after the change — including rewards earned by delegators who are already locked in the `exit_wait_window`.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Delegators who have staked are subject to the `exit_wait_window` (at minimum K epochs). A malicious staker can raise commission to 100% immediately, capturing all delegator yield for the entire lock-up period. Delegators have no mechanism to avoid this: `exit_delegation_pool_intent` starts the timer but does not stop reward accrual at the old commission rate — rewards continue to accrue at the new commission rate until the intent window expires and funds are returned.

---

### Likelihood Explanation

Any staker who has attracted delegators can execute this attack in two sequential transactions. No privileged access, leaked key, or external dependency is required. The only prerequisite is that the staker has an active pool (i.e., has previously called `set_open_for_delegation`). The attack is economically rational: the staker profits by extracting delegator yield.

---

### Recommendation

Introduce an epoch delay between when a commission increase is committed and when it becomes effective. Specifically:

- When `set_commission` is called with a value **higher** than the current commission, the new commission should be scheduled to take effect at `current_epoch + K` (analogous to how `enable_token` / `disable_token` use `get_epoch_plus_k()`), giving delegators sufficient time to observe the change and exit.
- Alternatively, prohibit commission increases entirely (only allow decreases), and require stakers to unstake and re-stake with a new commission if they wish to increase it.

---

### Proof of Concept

```
// Staker has commission = 500 (5%), many delegators locked in exit_wait_window

// Step 1: Set commitment with max = 100%
staking.set_commission_commitment(
    max_commission: 10000,          // 100%
    expiration_epoch: current_epoch + 1
);
// → CommissionCommitmentSet event emitted, commitment stored immediately
// → No delay, no notification to delegators

// Step 2: Immediately raise commission to 100%
staking.set_commission(commission: 10000);
// → update_commission checks: 10000 <= 10000 ✓, 10000 != 500 ✓
// → staker_pool_info.commission.write(Some(10000))  [L1600]
// → CommissionChanged event emitted

// All delegators are now paying 100% commission.
// They cannot exit until exit_wait_window elapses.
// All yield earned during that window flows to the staker.
```

        /// **Note**: Current commission increase safeguards still allow for sudden commission
        /// changes.
        /// **Note**: Updating epoch info can impact the commission commitment expiration date.
        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );
        }
