### Title
Silo Address Whitelist Not Enforced for Base-Token (`ETH`) Deposits via `receive_base_tokens` - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, `receive_erc20_tokens` correctly redirects deposits to a fallback address when the intended recipient is not on the Address whitelist. The parallel function `receive_base_tokens`, which handles bridged ETH (base-token) deposits, performs no such check. Any NEAR account can bridge ETH to a non-whitelisted EVM address, bypassing the Silo access-control policy entirely.

---

### Finding Description

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to one of two paths depending on whether the predecessor is the ETH-connector account: [1](#0-0) 

When the predecessor **is** the ETH connector, `receive_base_tokens` is called. That function parses the recipient address from the message and immediately credits the balance — with no whitelist gate: [2](#0-1) 

By contrast, `receive_erc20_tokens` — which handles every other NEP-141 token — explicitly checks `is_allow_receive_erc20_tokens` and redirects to the configured fallback address when the recipient is not whitelisted: [3](#0-2) 

The helper `is_allow_receive_erc20_tokens` simply delegates to the Address whitelist: [4](#0-3) 

The whitelist itself is disabled by default and only enforced when explicitly enabled by the Silo operator: [5](#0-4) 

The Silo parameters include an `erc20_fallback_address` specifically to catch non-whitelisted recipients: [6](#0-5) 

`receive_base_tokens` never consults either the fallback address or the whitelist, so the protection that exists for ERC-20 tokens is entirely absent for ETH.

---

### Impact Explanation

When a Silo operator enables the Address whitelist and sets a fallback address, the intended invariant is: *tokens sent to a non-whitelisted EVM address must be redirected to the fallback address*. This invariant holds for every NEP-141 token except ETH. Any NEAR user can call `ft_transfer_call` on the ETH-connector contract, targeting Aurora with a non-whitelisted EVM address in the message, and the ETH will be credited directly to that address. Because the Account whitelist (if also enabled) prevents non-whitelisted accounts from submitting EVM transactions, the ETH credited to a non-whitelisted address may be unspendable — a **temporary freeze of funds** — and the fallback account that was supposed to receive and manage those tokens does not.

**Impact: High — Temporary freezing of funds / whitelist bypass.**

---

### Likelihood Explanation

The entry point (`ft_transfer_call` on the ETH connector) is a standard, publicly callable NEP-141 interface. No special privilege is required. The condition is simply that the Silo operator has enabled the Address whitelist and set a fallback address, which is the normal production configuration for a Silo deployment. Any token holder can trigger this path.

**Likelihood: High.**

---

### Recommendation

Apply the same fallback-redirect guard in `receive_base_tokens` that already exists in `receive_erc20_tokens`:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut recipient = message_data.recipient;

    // Mirror the guard used in receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
    {
        recipient = fallback_address;
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    sdk::log!("Mint {amount} base tokens for: {}", recipient.encode());
    Ok(None)
}
```

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a non-zero `fixed_gas` and a `fallback_address` (e.g. `fallback.near`'s EVM address).
2. Enable the Address whitelist: call `set_whitelist_status` with `WhitelistKind::Address, active: true`.
3. Do **not** add victim EVM address `V` to the whitelist.
4. From any NEAR account, call `ft_transfer_call` on the ETH-connector NEP-141 contract:
   - `receiver_id`: Aurora Engine account
   - `amount`: any ETH amount
   - `msg`: hex-encoded address of `V`
5. Aurora's `ft_on_transfer` is invoked; `predecessor == connector_account_id`, so `receive_base_tokens` is called.
6. Observe that `V`'s EVM balance increases by the bridged amount — the fallback address receives nothing.
7. Repeat with `receive_erc20_tokens` (any other NEP-141 token, same non-whitelisted `V`) and observe the tokens go to `fallback_address` instead — confirming the asymmetry. [7](#0-6) [8](#0-7)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
