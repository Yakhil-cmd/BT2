### Title
Unchecked Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Silent Token Burn Without Exit Completion ŌĆö (`File: etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then calling the exit precompile via inline assembly. The return value of the assembly `call` is never checked. If the precompile call fails for any reason, the tokens are permanently destroyed with no corresponding NEP-141 or Ethereum-side transfer, permanently freezing the user's funds.

---

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` follow this pattern:

1. `_burn(_msgSender(), amount)` ŌĆö irreversibly destroys the caller's ERC-20 tokens.
2. An inline assembly `call` to the exit precompile address is made.
3. The return value `res` is captured but **never inspected**. No `require(res != 0)` or revert path exists.

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked ŌĆö silent failure
    }
}
```

The same pattern appears in `withdrawToEthereum` (lines 65ŌĆō76) and identically in `EvmErc20V2.sol` (lines 53ŌĆō77).

The exit-to-NEAR precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) can legitimately fail ŌĆö for example, when the recipient string is not a valid NEAR account ID, when the precompile encounters a state error, or when the ERC-20 contract is called from a context where the precompile rejects the caller. When the EVM `call` opcode returns `0` (failure), the Solidity function does not revert, so the burn is committed and the transaction succeeds from the caller's perspective, while no NEP-141 tokens are ever transferred.

---

### Impact Explanation

**Critical ŌĆö Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 balance is permanently reduced (tokens burned).
- No NEP-141 tokens are credited on the NEAR side.
- No ETH is credited on the Ethereum side (for `withdrawToEthereum`).
- The transaction does not revert, so the user has no on-chain indication of failure.
- There is no recovery path: the burned tokens cannot be re-minted by the user, and the precompile never scheduled a NEP-141 transfer.

The user's bridged assets are permanently destroyed.

---

### Likelihood Explanation

**Medium.** Any holder of a bridged ERC-20 token can trigger this path by calling `withdrawToNear` or `withdrawToEthereum` directly. The precompile can fail with an invalid recipient (e.g., a recipient byte string that is not a valid NEAR account ID, which is not validated in Solidity before the burn). A user who mistypes a recipient or passes a malformed byte string will silently lose their tokens. This is an unprivileged, externally reachable call with no preconditions beyond holding a token balance.

---

### Recommendation

Reverse the order of operations: call the precompile first and only burn tokens if the precompile call succeeds. Alternatively, check the return value and revert on failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
_burn(_msgSender(), amount);
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. User holds 1000 units of a bridged ERC-20 token on Aurora.
2. User calls `withdrawToNear(invalidRecipientBytes, 1000)` where `invalidRecipientBytes` is not a valid NEAR account ID.
3. `_burn(msg.sender, 1000)` executes ŌĆö user's balance drops to 0.
4. The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` rejects the call (invalid account ID) and returns `0`.
5. The assembly block captures `res = 0` but does not revert.
6. The transaction succeeds. The user's ERC-20 tokens are gone. No NEP-141 tokens are credited. Funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
