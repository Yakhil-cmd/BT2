### Title
Secret Triple Scalars Disclosed via `Debug` Trait Derivation on `TripleShare` — (File: `src/ecdsa/ot_based_ecdsa/triples/mod.rs`)

---

### Summary

`TripleShare` — which holds the secret Beaver triple scalars `(a, b, c)` used in OT-based ECDSA presigning — unconditionally derives `Debug`. The module's own documentation explicitly states that leaking these scalars allows recovery of the private signing key. Any caller, logging framework, or error handler that formats a `TripleShare` with `{:?}` will silently emit all three secret scalars in plaintext. The same pattern is repeated across `KeygenOutput<C>`, both `PresignOutput` types, and both `RerandomizedPresignOutput` types.

---

### Finding Description

`TripleShare` is defined in `src/ecdsa/ot_based_ecdsa/triples/mod.rs` with an unconditional `Debug` derive:

```rust
#[derive(Clone, Debug, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct TripleShare {
    pub a: Scalar,
    pub b: Scalar,
    pub c: Scalar,
}
```

The module-level documentation at the top of the same file explicitly warns:

> "It's important that the value of the underlying scalars in the triple is kept secret, otherwise the private key used to create a signature with that triple could be recovered."

The `Debug` derive on `TripleShare` directly contradicts this requirement. The fields `a`, `b`, `c` are raw `k256` scalars; `k256`'s `Debug` implementation emits the full scalar bytes in hex. Any `{:?}` formatting of a `TripleShare` — in a log line, a test assertion failure, a `dbg!()` call, or an error message — will emit all three secret scalars.

The same structural flaw appears across the codebase:

### Impact Explanation

**`TripleShare` — Critical.** The module documentation explicitly states that leaking `(a, b, c)` allows recovery of the private signing key. An attacker who obtains a `TripleShare` debug dump can reconstruct the master secret key and forge arbitrary signatures.

**`KeygenOutput<C>` — Critical.** `private_share` is a `SigningShare<C>` holding the participant's secret scalar. Leaking it via `Debug` is direct disclosure of a private signing share. Collecting threshold-many shares allows full private key reconstruction.

**`PresignOutput` / `RerandomizedPresignOutput` (both ECDSA variants) — Critical.** The secret nonce shares `k`, `sigma`, `alpha`, `beta`, `c`, `e` are presign secrets. Leaking them enables nonce recovery and, combined with the signing equation, private key extraction.

All three impact categories map to: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

This is a library. Callers are expected to store, pass, and log these types. The `Debug` derive is part of the public API contract — it signals to callers that formatting with `{:?}` is safe and supported. Realistic exposure paths include:

- Structured logging frameworks (e.g., `tracing`, `log`) that automatically format fields with `Debug`
- Test assertion failures that print the full struct via `assert_eq!` or `insta` snapshots (the snapshot files in the repo already serialize `private_share` in plaintext)
- Error propagation that wraps these types in `anyhow`/`thiserror` contexts with `{:?}` formatting
- Any `dbg!(triple_share)` call during development that is accidentally left in

The `ZeroizeOnDrop` derive on these types shows the authors are aware of secret-material hygiene, making the unguarded `Debug` derive an inconsistency rather than an intentional design choice.

---

### Recommendation

1. **Remove `Debug` from all types holding secret scalars.** Replace with a manual `Debug` implementation that redacts secret fields:
   ```rust
   impl fmt::Debug for TripleShare {
       fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
           f.debug_struct("TripleShare")
            .field("a", &"[REDACTED]")
            .field("b", &"[REDACTED]")
            .field("c", &"[REDACTED]")
            .finish()
       }
   }
   ```
2. Apply the same pattern to `KeygenOutput<C>` (redact `private_share`), all `PresignOutput` variants (redact all scalar fields), and all `RerandomizedPresignOutput` variants.
3. Consider using the [`secrecy`](https://crates.io/crates/secrecy) crate to wrap secret scalars, which provides `Debug` redaction automatically and prevents accidental exposure through `Display` as well.
4. Audit all snapshot test files — several already contain plaintext `private_share` values — and ensure they use test-only dummy values, not values derived from a real RNG seed that could be confused with production material.
