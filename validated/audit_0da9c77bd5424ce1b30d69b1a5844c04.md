### Title
`DeleteAccount` with `beneficiary_id == account_id` Permanently Burns Account Balance — (File: `runtime/runtime/src/action_validation.rs`)

---

### Summary

`validate_delete_action` does not check that `beneficiary_id != account_id`. When an account submits `DeleteAccount { beneficiary_id: self }`, the runtime creates a balance-refund receipt addressed to the account it just deleted. Because the receiver no longer exists, the refund receipt fails, and per the documented refund semantics the entire balance is permanently burnt.

---

### Finding Description

**Root cause — missing identity guard in `validate_delete_action`:**

`validate_delete_action` only verifies that `beneficiary_id` is a syntactically valid account ID; it never checks that the beneficiary differs from the account being deleted. [1](#0-0) 

**Execution path — balance sent to a deleted account:**

`action_delete_account` unconditionally creates a `Receipt::new_balance_refund` addressed to `delete_account.beneficiary_id`, then immediately removes the account from state. When `beneficiary_id == account_id`, the refund receipt is routed to an account that no longer exists. [2](#0-1) 

**Refund failure semantics — balance is burnt:**

`Receipt::new_balance_refund` sets `predecessor_id = "system"`, making it a refund receipt. The documented invariant for refund receipts is explicit: [3](#0-2) 

When the refund receipt's `Transfer` action executes against the now-absent account, `check_account_existence` returns `AccountDoesNotExist`, the receipt fails, and the entire balance is burnt — not returned to any party. [4](#0-3) 

**No existing guard:**

`check_actor_permissions` for `DeleteAccount` only verifies `actor_id == account_id` and that the account is not staking; it does not inspect `beneficiary_id`. [5](#0-4) 

---

### Impact Explanation

Any account owner can submit a `DeleteAccount` action with `beneficiary_id` equal to their own `account_id`. The action passes all validation and execution checks, the account is removed, and the balance refund receipt is silently burnt. The corrupted value is the account's entire liquid balance (`account_ref.amount()`), which is permanently destroyed with no recovery path. [6](#0-5) 

This falls squarely in the **accounting** target scope: the conservation invariant "deleted account balance is transferred to beneficiary" is broken when beneficiary equals the deleted account.

---

### Likelihood Explanation

Likelihood is **Low**. The attacker must control the signing key for the account and must deliberately (or accidentally) supply `beneficiary_id = own account_id`. There is no third-party exploitation path; the loss is self-inflicted. However, the absence of any guard means the scenario is reachable by any unprivileged user with no special privileges.

---

### Recommendation

Add an identity check inside `validate_delete_action` (or equivalently inside `action_delete_account`) that rejects the action when `beneficiary_id == account_id`:

```rust
fn validate_delete_action(
    action: &DeleteAccountAction,
    account_id: &AccountId,          // pass the receiver/account being deleted
) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    if action.beneficiary_id == *account_id {
        return Err(ActionsValidationError::InvalidBeneficiaryId {
            account_id: account_id.to_string(),
        });
    }
    Ok(())
}
```

The check must be enforced at the static-validation layer so it is applied both to transactions and to receipts generated by smart contracts via `promise_batch_action_delete_account`. [1](#0-0) 

---

### Proof of Concept

1. Account `alice` holds balance `X > 0`.
2. Alice signs and submits:
   ```
   Transaction {
     signer_id:   "alice",
     receiver_id: "alice",
     actions: [DeleteAccount { beneficiary_id: "alice" }],
   }
   ```
3. `validate_delete_action` passes — `"alice"` is a valid account ID format.
4. `action_delete_account` executes:
   - Pushes `Receipt::new_balance_refund("alice", X)` onto `result.new_receipts`.
   - Calls `remove_account(state_update, "alice")` — alice's state entry is gone.
5. The balance-refund receipt (`predecessor_id = "system"`, `receiver_id = "alice"`) is processed in a subsequent block.
6. `check_account_existence` for the `Transfer` action finds no account for `"alice"` and returns `AccountDoesNotExist`.
7. The refund receipt fails; per `docs/RuntimeSpec/Refunds.md` line 12, the amount `X` is **burnt**.
8. Alice's entire balance `X` is permanently destroyed. [7](#0-6) [8](#0-7) [9](#0-8)

```rust
pub(crate) fn action_delete_account(
    state_update: &mut TrieUpdate,
    account: &mut Option<Account>,
    actor_id: &mut AccountId,
    receipt: &Receipt,
    result: &mut ActionResult,
    account_id: &AccountId,
    delete_account: &DeleteAccountAction,
    config: &RuntimeConfig,
    current_protocol_version: ProtocolVersion,
) -> Result<(), StorageError> {
    let account_ref = account.as_ref().unwrap();
    let account_storage_usage = if ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage
        .enabled(current_protocol_version)
    {
        let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
        account_ref.storage_usage().saturating_sub(contract_storage)
    } else {
        // Legacy behavior: only subtracts local contract code, misses the
        // global contract identifier overhead.
        let account_storage_usage = account_ref.storage_usage();
        let code_len = get_code_len_or_default(
            state_update,
            account_id.clone(),
            account_ref.local_contract_hash().unwrap_or_default(),
        )?;
        debug_assert!(
            code_len == 0 || account_storage_usage > code_len,
            "account storage usage should be larger than code size. storage usage: {}, code size: {}",
            account_storage_usage,
            code_len
        );
        account_storage_usage.saturating_sub(code_len)
    };
    if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
        result.result =
            Err(ActionErrorKind::DeleteAccountWithLargeState { account_id: account_id.clone() }
                .into());
        return Ok(());
    }
    let gas_key_balance_to_burn = compute_gas_key_balance_sum(state_update, account_id)?;
    if gas_key_balance_to_burn > GasKeyInfo::MAX_BALANCE_TO_BURN {
        result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh {
            account_id: account_id.clone(),
            public_key: None,
            balance: gas_key_balance_to_burn,
        }
        .into());
        return Ok(());
    }
    // We use current amount as a pay out to beneficiary.
    let account_balance = account_ref.amount();
    if account_balance > Balance::ZERO {
        result
            .new_receipts
            .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
    }
    let remove_result = remove_account(state_update, account_id)?;
    result.tokens_burnt =
        result.tokens_burnt.checked_add(gas_key_balance_to_burn).ok_or_else(|| {
            StorageError::StorageInconsistentState("tokens_burnt overflow".to_string())
        })?;
    if remove_result.gas_key_nonce_count > 0 {
        let compute = storage_removes_compute(
            &config.wasm_config.ext_costs,
            remove_result.gas_key_nonce_count,
            remove_result.gas_key_nonce_total_key_bytes,
            AccessKey::NONCE_VALUE_LEN * remove_result.gas_key_nonce_count,
        );
        result.compute_usage = safe_add_compute(result.compute_usage, compute).map_err(|_| {
            StorageError::StorageInconsistentState("compute_usage overflow".to_string())
        })?;
    }
    *actor_id = receipt.predecessor_id().clone();
    *account = None;
    Ok(())
}
```

**File:** docs/RuntimeSpec/Refunds.md (L10-13)
```markdown
Refund receipts are identified by having `predecessor_id == "system"`. They are also special because they don't cost any gas to generate or execute. As a result, they also do not contribute to the block gas limit.

If the execution of a refund fails, the refund amount is burnt.
The refund receipt is an `ActionReceipt` that consists of a single action `Transfer` with the `deposit` amount of the refund.
```
