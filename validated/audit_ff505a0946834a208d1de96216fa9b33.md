### Title
Missing `msg_hash == 0` Validation in OT-Based ECDSA Sign Enables Split-View Attack Leading to Private Key Extraction - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

### Summary

The OT-based ECDSA `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` accepts a zero `msg_hash` scalar without rejection. The robust ECDSA `sign` function in `src/ecdsa/robust_ecdsa/sign.rs` explicitly rejects `msg_hash == 0` to prevent a documented split-view attack. The OT-based ECDSA scheme shares the same algebraic structure and the same `N >= threshold` (not `N = threshold`) constraint, making it equally susceptible. A malicious coordinator who presents `msg_hash = 0` to participants can exploit this to extract the aggregate private signing key.

### Finding Description

**Root cause — missing validation in `src/ecdsa/ot_based_ecdsa/sign.rs`:**

The `sign` function performs several input checks (participant count, duplicates, coordinator membership, threshold bound) but never checks whether `msg_hash` is zero: [1](#0-0) 

Compare with the robust ECDSA `sign` function, which explicitly rejects zero: [2](#0-1) 

The security documentation explicitly identifies this as a required guard: [3](#0-2) 

**Algebraic mechanism:**

Each participant's signature share is computed as: [4](#0-3) 

When `msg_hash = 0`, the share collapses to `s_i = r * sigma_i`, which is completely independent of the message. The coordinator aggregates these to obtain `S₀ = r * sigma`.

**Split-view attack path:**

The OT-based ECDSA `sign` function enforces only `N >= threshold`, not `N = threshold`: [5](#0-4) 

This means `N > threshold` is a valid configuration. A malicious coordinator can therefore run two overlapping signing sessions with the same presignature (same `k`, `sigma`):

- **Session 1** — coordinator presents `msg_hash = 0` to all participants → each participant computes `s_i(0) = r * sigma_i`; coordinator collects `S₀ = r * sigma`.
- **Session 2** — coordinator presents `msg_hash = m` to all participants → each participant computes `s_i(m) = m * k_i + r * sigma_i`; coordinator collects `S_m = m * k + r * sigma`.

From the two aggregated values the coordinator recovers:

```
k     = (S_m - S₀) / m
sigma = S₀ / r
x     = sigma / k          ← aggregate private key
```

This is a standard ECDSA nonce-reuse key-extraction attack. The `msg_hash = 0` input is the enabling condition because it lets the coordinator isolate `r * sigma` without the message-dependent term, making the two-equation system solvable.

### Impact Explanation

**Critical — extraction of the aggregate private signing key.**

A malicious coordinator who controls which `msg_hash` value is passed to each participant's local `sign` call can reuse a single presignature across two sessions (one with `msg_hash = 0`, one with any nonzero `msg_hash`) and algebraically recover the full private key `x`. All future and past signatures produced under that key are compromised. This falls squarely within the allowed Critical impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

### Likelihood Explanation

The coordinator role is reachable by any participant in the protocol (the coordinator is simply one of the signing participants). A malicious coordinator can supply `msg_hash = 0` as a local argument to its own `sign` call and simultaneously instruct other participants to use `msg_hash = 0` (or craft a session where it controls the message-hash input). No cryptographic primitive break or external key compromise is required — only control of the coordinator role and the ability to reuse a presignature across two sessions, which the library does not prevent at the protocol level.

### Recommendation

Add the same zero-hash guard that the robust ECDSA `sign` function already enforces, immediately before the protocol is started in `src/ecdsa/ot_based_ecdsa/sign.rs`:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
```

This mirrors the existing check in `src/ecdsa/robust_ecdsa/sign.rs` at lines 91–95 and closes the algebraic split-view attack path.

### Proof of Concept

1. Complete a presigning session to obtain `PresignOutput { big_r, k_shares, sigma_shares }`.
2. Rerandomize the presignature into `RerandomizedPresignOutput` with `msg_hash_bytes = [0u8; 32]` (zero hash) to produce `args0`.
3. Call `ot_based_ecdsa::sign::sign(participants, coordinator, threshold, me, pk, rerandomized_presig_0, Scalar::ZERO)` — the function accepts this without error because no zero-hash check exists.
4. The coordinator collects all `s_i(0) = r * sigma_i` and sums them to obtain `S₀ = r * sigma`.
5. Rerandomize the **same** raw `PresignOutput` again with a nonzero `msg_hash_bytes = m_bytes` to produce `args_m` (same `big_r`, same presignature nonce).
6. Call `sign` again with `msg_hash = m` — participants produce `s_i(m) = m * k_i + r * sigma_i`; coordinator sums to `S_m = m * k + r * sigma`.
7. Compute `k = (S_m - S₀) / m`, `sigma = S₀ / r`, `x = sigma / k`.
8. Verify: `ProjectivePoint::GENERATOR * x == public_key`. The private key is recovered. [1](#0-0) [6](#0-5) [7](#0-6)

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

4. **Do not sign with $h = 0$** (the zero message hash).
   This input enables a related algebraic split-view attack in the modified scheme when
   $N_1 > 2t + 1$.
