### Title
`MetricOmmPoolFactory.setPoolBinAdditionalFees` Allows Pool Admin to Impose Per-Bin Fees Exceeding Factory-Defined Spread Cap - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

The factory enforces `maxAdminSpreadFeeE6` as the hard ceiling on admin-controlled spread fees. Every other admin fee setter validates against this cap. However, `setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap check, allowing a pool admin to impose per-bin effective fees that exceed the factory-defined bound.

### Finding Description

`MetricOmmPoolFactory` stores `maxAdminSpreadFeeE6` (up to `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%) as the ceiling for admin-controlled spread fees. `setPoolAdminFees` enforces this:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` performs no analogous check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index:

```solidity
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap, the bin additional fee is added directly on top of the base spread fee:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so their maximum value is 65 535 E6 ≈ 6.55%. A pool admin can set this on any bin at any time, making the effective per-bin fee `spreadFeeE6 + addFeeE6`, which can reach ≈ 26.55% — well above the 20% hard cap the factory is supposed to enforce. [5](#0-4) 

### Impact Explanation

Traders swapping through an affected bin pay a higher fee than the protocol cap permits. The excess fee is retained in the pool as LP surplus (spread fee accounting), meaning traders lose principal beyond the protocol-sanctioned maximum. This is a direct loss of user funds triggered by a semi-trusted actor (pool admin) exploiting an unchecked path — an admin-boundary break matching the contest's allowed impact gate.

### Likelihood Explanation

Any pool admin can call `setPoolBinAdditionalFees` at any time with no timelock. The only prerequisite is holding the `poolAdmin` role for the target pool. Because pool admins are semi-trusted (not fully trusted), this path is reachable by a malicious or compromised admin without any privileged protocol-owner action.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, analogous to the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
+   if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
+   if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce that `spreadFeeE6 + addFeeE6 <= maxAdminSpreadFeeE6` to bound the total effective per-bin fee.

### Proof of Concept

```solidity
// Pool created with maxAdminSpreadFeeE6 = 200_000 (20%)
// Admin sets base admin spread fee at the cap
factory.setPoolAdminFees(pool, 200_000, 0); // passes — at cap

// Admin then sets bin additional fee to uint16 max (65_535 ≈ 6.55%)
// No revert — no cap check exists
factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535);

// Effective buy fee for bin 0 during swap:
// baseFeeX64 (from spreadFeeE6=200_000) + mulDiv(65_535, ONE_X64, 1e6)
// = 20% + 6.55% = 26.55% — exceeds the 20% factory cap
``` [2](#0-1) [6](#0-5)
