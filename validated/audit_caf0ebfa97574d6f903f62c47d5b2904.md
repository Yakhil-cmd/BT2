### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Any Address with a Pending Pool Transaction, Not Just Deploy-Account Pairs - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce 1 when the account's deploy-account transaction is still pending in the mempool (a UX feature). However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from that address in the pool — not specifically a deploy-account transaction. This allows an unprivileged third party to submit an invoke transaction with an invalid signature for any address that has **any** pending transaction in the mempool, bypassing the account's `__validate__` entry point entirely at the gateway admission stage.

### Finding Description

In `skip_stateful_validations`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

The condition `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` if **any** transaction from the address is in the pool — including a previously admitted invoke with nonce 1 that itself was admitted via the skip mechanism. The code comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is circular: those "future nonce transactions that passed validations" may have passed precisely because of the skip mechanism, not because the account was genuinely deployed.

When `skip_validate = true` is returned, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, and `StatefulValidator::perform_validations` returns early without calling the `__validate__` entry point:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
```

The pre-validation stage (`perform_pre_validation_stage`) still runs nonce and fee-bound checks, but the account's custom signature verification logic is entirely skipped.

**Concrete attack path:**

1. Alice submits a `deploy_account` transaction for address X (nonce 0). It enters the mempool.
2. Bob (who does not control X) submits an `invoke` transaction with nonce 1 from address X, carrying an **invalid signature** and minimal but non-zero resource bounds.
3. Gateway nonce check: `account_nonce(0) ≤ tx_nonce(1) ≤ max_allowed_nonce(200)` — passes.
4. `skip_stateful_validations`: `tx.nonce() == 1 && account_nonce == 0` — true. `account_tx_in_pool_or_recent_block(X)` — true (Alice's deploy-account is in the pool).
5. `skip_validate = true` → `run_validate_entry_point` is called with `validate = false` → `__validate__` is **not called**.
6. Bob's invalid invoke is admitted to the mempool.

### Impact Explanation

An unprivileged attacker can inject invoke transactions with invalid signatures (or arbitrary calldata that the account's `__validate__` would reject) into the mempool for any address that has a pending deploy-account transaction. This breaks the admission invariant: **every invoke transaction from an undeployed account must have its `__validate__` entry point verified before mempool admission**. The admitted invalid transactions consume mempool capacity and, if included in a block, consume block space before reverting during execution.

### Likelihood Explanation

Any address that submits a deploy-account + invoke pair (a common UX pattern explicitly supported and tested by the codebase) creates a window during which a third party can inject invalid invokes for that address. The attack requires no special privileges, only knowledge of a pending deploy-account transaction (observable from the public mempool).

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction is pending for the address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address)` that only returns `true` when a deploy-account transaction (not any transaction) is present. Alternatively, the gateway can track the deploy-account transaction hash explicitly and verify it is still pending before granting the skip.

### Proof of Concept

1. Alice calls `add_tx` with a valid `deploy_account` for address `0xALICE` (nonce 0). Mempool accepts it; `account_tx_in_pool_or_recent_block(0xALICE)` now returns `true`.
2. Bob calls `add_tx` with an `invoke` (nonce 1, sender `0xALICE`) carrying a garbage signature `[0xDEAD, 0xBEEF]` and `l2_gas.max_amount = 1, l2_gas.max_price_per_unit = 1`.
3. Stateless validator passes (non-zero resource bounds, valid address format).
4. Stateful validator: `validate_nonce` passes (`0 ≤ 1 ≤ 200`). `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(0xALICE)` is `true`.
5. `run_validate_entry_point` is called with `validate = false`; `__validate__` is never invoked.
6. Bob's invalid invoke is now in the mempool. When the batcher includes it and Alice's deploy-account executes first, Bob's invoke will call `__validate__`, fail signature verification, and revert — but it was admitted and occupies mempool and block resources. async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
