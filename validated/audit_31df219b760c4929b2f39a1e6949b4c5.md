### Title
Missing Maximum Commission Guard in `switch_delegation_pool()` Allows Staker to Front-Run Delegator with Commission Increase — (`src/pool/pool.cairo`)

---

### Summary

`switch_delegation_pool()` accepts no `max_commission` parameter. A staker who holds an active `CommissionCommitment` can legally raise their commission to any value ≤ `max_commission` at any time. A delegator who inspects the target pool's commission before submitting a switch can have that commission raised between their intent and their switch call, silently redirecting a larger share of their future yield to the staker.

---

### Finding Description

The `switch_delegation_pool()` entry point in `src/pool/pool.cairo` is:

```cairo
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
) -> Amount {
``` [1](#0-0) 

There is no `max_commission` (or equivalent slippage guard) parameter. The function unconditionally moves the delegator's funds into the target pool at whatever commission the staker currently holds.

The commission system allows a staker to **increase** commission while a `CommissionCommitment` is active. The `update_commission` internal function enforces only that the new value is ≤ `max_commission` and ≠ the old value:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission, ...
    );
    assert!(commission != old_commission, ...);
``` [2](#0-1) 

So a staker with `current_commission = 200` (2 %) and `max_commission = 5000` (50 %) can call `set_commission(5000)` at any time while the commitment is active.

The commission is read at reward-calculation time from live storage:

```cairo
let commission = staker_pool_info.commission();
...
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [3](#0-2) 

Because the commission is applied at attestation time (not locked at delegation time), any increase that occurs after the delegator switches but before the next attestation silently reduces the delegator's yield.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

After the switch the delegator's funds are locked in the target pool. The staker can immediately raise commission to `max_commission`, capturing a larger fraction of every future epoch's rewards at the delegator's expense. The delegator has no on-chain recourse other than initiating a new exit intent and waiting out the full `exit_wait_window` (default one week, up to twelve weeks).

The yield diverted to the staker is yield that would otherwise have accrued to the delegator — a direct, quantifiable transfer of unclaimed rewards.

---

### Likelihood Explanation

**Medium.**

1. The staker must have previously called `set_commission_commitment()` with a `max_commission` meaningfully above the current commission. This is a normal, documented protocol action.
2. On Starknet the sequencer controls transaction ordering; a staker who also operates (or influences) the sequencer can trivially front-run. Even without sequencer access, the staker can simply call `set_commission` in the same block or the block immediately before the delegator's `switch_delegation_pool` lands.
3. The scenario also arises in normal (non-adversarial) flows: a staker may legitimately raise commission between the moment a delegator reads the pool parameters and the moment the switch transaction is included.

---

### Recommendation

Add a `max_commission: Commission` parameter to `switch_delegation_pool()` (and symmetrically to `enter_delegation_pool()`). Before completing the switch, assert that the target staker's current commission does not exceed the caller-supplied bound:

```cairo
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
    max_commission: Commission,   // <-- new
) -> Amount {
    // ... existing checks ...
    let current_commission = IStakingDispatcher { contract_address: staking_contract }
        .staker_pool_info(to_staker).commission.unwrap();
    assert!(current_commission <= max_commission, Error::COMMISSION_TOO_HIGH);
    // ... rest of logic ...
}
```

This mirrors the slippage-protection pattern already present in the `set_commission_commitment` flow and gives delegators the same MEV protection the original Size protocol was advised to implement.

---

### Proof of Concept

```
1. Staker S stakes and calls set_commission(200)          // 2 %
2. Staker S calls set_commission_commitment(
       max_commission: 5000, expiration_epoch: now+10)    // commitment active
3. Delegator D sees commission = 2 %, decides to switch.
4. D calls exit_delegation_pool_intent(amount) on old pool.
   -- tx lands in mempool / sequencer queue --
5. Staker S calls set_commission(5000)                    // raises to 50 %
   -- this tx is ordered before D's switch --
6. D's switch_delegation_pool(to_staker: S, ...) executes.
   D is now in S's pool at 50 % commission.
7. Next attestation: 50 % of D's share of pool rewards
   flows to S instead of D.
```

The delegator expected ≤ 2 % commission; they silently received 50 %. No parameter in `switch_delegation_pool()` would have caused the transaction to revert. [4](#0-3) [5](#0-4) [6](#0-5)
