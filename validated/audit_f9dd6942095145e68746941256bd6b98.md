### Title
Token Balance Used as Credit in `request_funds` Allows Suppression of L1 Mint Requests, Temporarily Freezing Unclaimed Rewards - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

`RewardSupplier.request_funds()` reads the contract's live STRK token balance via `balance_of(get_contract_address())` to compute `credit` and decide whether to send L1 mint requests. Because any unprivileged caller can transfer STRK directly to the `RewardSupplier` address, an attacker can artificially inflate `balance`, suppress all L1 mint requests, and create a window during which `balance < unclaimed_rewards`, causing every `claim_rewards` call to revert.

---

### Finding Description

In `request_funds()`, the credit side of the solvency check is:

```
credit = balance_of(get_contract_address()) + l1_pending_requested_amount
debit  = unclaimed_rewards
if credit < debit + threshold { send L1 mint request }
``` [1](#0-0) 

`balance_of` returns the raw ERC-20 balance of the contract, which includes any tokens sent directly by an external party — not just tokens that arrived via the StarkGate bridge. The `on_receive` hook (the only path that reduces `l1_pending_requested_amount`) is only triggered by the bridge: [2](#0-1) 

A direct ERC-20 transfer to the contract increases `balance` without touching `l1_pending_requested_amount`, so `credit` is inflated by the full donated amount. As long as `credit >= debit + threshold`, no L1 mint request is emitted.

The attacker's donated tokens are real and sit in the contract's balance, so they are consumed as rewards are paid out via `claim_rewards`: [3](#0-2) 

Once those tokens are exhausted, an L1 request is finally triggered. However, L1→L2 message delivery on Starknet takes hours to days. During that gap, `unclaimed_rewards` (an internal counter) can exceed the actual `balance`. The `checked_transfer` on line 219 will then panic, causing every staker and delegator reward claim to revert until the L1 tokens arrive.

The staking contract's `send_rewards_to_staker` calls `claim_from_reward_supplier` which calls `RewardSupplier.claim_rewards`: [4](#0-3) 

This means `claim_rewards`, `unstake_action`, and any pool reward distribution that triggers a reward claim all revert during the freeze window.

---

### Impact Explanation

**Temporary freezing of unclaimed yield.** All stakers and delegators are unable to claim earned rewards for the duration of the L1→L2 message delivery delay after the attacker's donated tokens are exhausted. The `unclaimed_rewards` accounting variable correctly reflects owed rewards, but the actual token balance is insufficient to fulfill transfers.

---

### Likelihood Explanation

The attack is fully permissionless — any address can call a standard ERC-20 `transfer` to the `RewardSupplier` contract address. The attacker's cost equals the donated tokens, which are consumed paying legitimate rewards (the attacker does not recover them). This makes it a pure griefing attack. The cost scales with the reward rate and the desired freeze duration. If STRK price drops significantly, the attack becomes economically viable for a motivated adversary. The L1→L2 delay (the freeze window) is a protocol-level constant that cannot be shortened by the protocol.

---

### Recommendation

Replace the live `balance_of` call in `request_funds` with a dedicated internal accounting variable (e.g., `available_balance`) that is incremented only when tokens arrive via `on_receive` and decremented when tokens are transferred out via `claim_rewards`. This mirrors the fix applied to the Connext analog: maintain custodied amounts through controlled entry/exit points rather than reading the raw token balance.

```
// Instead of:
let balance = token_dispatcher.balance_of(account: get_contract_address());

// Use:
let balance = self.available_balance.read();
```

`available_balance` should be:
- Incremented in `on_receive` by the received amount.
- Decremented in `claim_rewards` by the transferred amount.

This ensures that tokens sent directly to the contract by an attacker do not affect the L1 mint request logic.

---

### Proof of Concept

1. Deploy the system normally. Let `unclaimed_rewards = 0`, `balance = B`, `l1_pending_requested_amount = 0`.
2. Attacker calls `STRK.transfer(recipient: reward_supplier_address, amount: X)` where `X` is large enough that `B + X >= expected_total_rewards_over_N_epochs + threshold`.
3. Over the next N epochs, attestations fire, rewards accumulate (`unclaimed_rewards` grows), and `request_funds` is called each time. Because `credit = B + X >= debit + threshold` throughout, **no L1 mint request is ever sent**.
4. Rewards are paid out from the contract's balance (consuming `B + X`). Eventually `balance ≈ 0` and `unclaimed_rewards = R > 0`.
5. `request_funds` now sends an L1 mint request. `l1_pending_requested_amount = base_mint_amount`.
6. During the L1→L2 delivery delay, any call to `staking.claim_rewards`, `staking.unstake_action`, or pool reward distribution triggers `RewardSupplier.claim_rewards(amount)` where `amount <= unclaimed_rewards` but `amount > balance ≈ 0`. The `checked_transfer` panics.
7. All stakers and delegators are frozen out of their earned rewards until the L1 message is processed. [5](#0-4) [6](#0-5) [7](#0-6)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
