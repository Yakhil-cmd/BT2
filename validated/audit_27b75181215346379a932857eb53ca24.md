Audit Report

## Title
`last_fee` Not Updated for `Approve` Operations Causes Stale Fee Fallback and Incorrect Balance Accounting in ICRC-1 Index-NG - (File: rs/ledger_suite/icrc1/index-ng/src/main.rs)

## Summary
The `last_fee` state variable in the ICRC-1 index-ng canister is updated by `Burn`, `Mint`, and `Transfer` operations but is never updated in the `Approve` branch of `process_balance_changes`, even when the resolved fee is `Some`. When a legacy mainnet `Approve` block with no fee fields is encountered ‚ÄĒ a condition the code itself acknowledges at lines 1092‚Äď1095 ‚ÄĒ the fallback reads the stale `last_fee`. If the ledger fee changed between the last Transfer/Burn/Mint and those legacy blocks, the wrong fee is used to debit the `from` account, either silently corrupting the index balance state or causing a trap in `debit` that permanently halts block ingestion for the index canister.

## Finding Description
In `process_balance_changes` (line 1021), `last_fee` is updated in every fee-bearing branch except `Approve`:

- **Burn** (line 1037): `mutate_state(|s| s.last_fee = Some(fee))` ‚ÄĒ when `effective_fee` is `Some`
- **Mint** (line 1053): `mutate_state(|s| s.last_fee = Some(fee))` ‚ÄĒ when `effective_fee` is `Some`
- **Transfer** (line 1072): `mutate_state(|s| s.last_fee = Some(fee))` ‚ÄĒ unconditionally

The `Approve` branch (lines 1087‚Äď1121) resolves the fee at line 1090‚Äď1091 via `fee.or(block.effective_fee)` but never calls `mutate_state` to persist it. When the resolved fee is `Some`, it is used locally to `debit` the `from` account at line 1116, but `last_fee` remains unchanged.

The fallback at lines 1096‚Äď1107 is reached for legacy mainnet `Approve` blocks where both `fee` and `effective_fee` are `None`. It returns whatever `last_fee` was set by the most recent Transfer/Burn/Mint, which may reflect a different fee epoch. The `debit` function at lines 1139‚Äď1144 traps with an underflow panic if the debited amount exceeds the tracked balance, permanently halting block ingestion.

Exploit flow:
1. Transfer at fee=F‚āĀ ‚Üí `last_fee = F‚āĀ`
2. Ledger fee changes to F‚āā (via NNS governance)
3. Approve with `fee=Some(F‚āā)` ‚Üí debits F‚āā correctly, but `last_fee` stays F‚āĀ
4. Legacy Approve with `fee=None`, `effective_fee=None` ‚Üí fallback uses stale `last_fee=F‚āĀ`
5. If F‚āĀ > tracked balance: `debit` traps ‚Üí index halts permanently
6. If F‚āĀ ‚Č† F‚āā: index balance for `from` diverges from true ledger balance

## Impact Explanation
This matches two allowed High-severity ICP bounty impacts:

1. **Application/platform-level DoS**: When `last_fee` is higher than the account's tracked balance, `debit` traps at line 1142, halting all block ingestion for the index canister. All subsequent `get_account_transactions` and `get_blocks` queries return permanently stale data for every account, not just the affected one.

2. **Significant ledger/infrastructure security impact with concrete user harm**: When `last_fee` is lower than the true fee, the index over-reports the `from` account's balance. Wallets and DeFi integrations querying the index canister receive incorrect balances, enabling misinformed transactions.

## Likelihood Explanation
The legacy fee-less `Approve` blocks are immutable historical facts on mainnet, explicitly acknowledged in the code comment at lines 1092‚Äď1095. The ICP ledger fee has been modified via NNS governance proposals. Any index canister ‚ÄĒ including a freshly deployed one pointed at the affected ledger ‚ÄĒ that processes a chain segment where the fee changed between the last Transfer/Burn/Mint and a legacy fee-less `Approve` block will trigger this bug. No privileged access is required; deploying a canister on the IC is permissionless.

## Recommendation
In the `Approve` branch, update `last_fee` whenever the resolved fee is `Some`, mirroring the pattern used in `Burn` and `Mint`:

```rust
Operation::Approve { from, fee, spender, .. } => {
    let fee = match fee.or(block.effective_fee) {
        Some(fee) => {
            mutate_state(|s| s.last_fee = Some(fee)); // ADD THIS
            fee
        }
        None => match with_state(|state| state.last_fee) { ... }
    };
    ...
}
```

## Proof of Concept
Deterministic integration test plan using PocketIC or a local replica:

1. Initialize an ICRC-1 ledger with fee = 10_000.
2. Submit a `Transfer` block ‚Üí index processes it, `last_fee = 10_000`.
3. Change ledger fee to 100_000 via admin call.
4. Submit an `Approve` block with `fee = Some(100_000)` ‚Üí index debits 100_000 correctly; assert `last_fee` is still 10_000 (bug confirmed).
5. Inject a synthetic legacy `Approve` block with `fee = None` and `effective_fee = None` for an account whose index-tracked balance is 50_000.
6. Observe: index uses `last_fee = 10_000`, debits 10_000 instead of 100_000 ‚Üí balance over-reported by 90_000 (corruption scenario).
7. Repeat step 5 for an account whose index-tracked balance is 5_000 (less than `last_fee = 10_000`) ‚Üí `debit` traps ‚Üí index halts (DoS scenario).
