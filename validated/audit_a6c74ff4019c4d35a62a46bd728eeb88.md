### Title
Unrestricted `disable_rewards` Flag in `update_rewards` Allows Any Caller to Permanently Block Consensus Reward Accrual — (File: `src/staking/staking.cairo`)

---

### Summary
The `update_rewards` function accepts a caller-supplied `disable_rewards: bool` parameter with no access control beyond "caller is not zero address." When `true`, the function advances the global `last_reward_block` checkpoint to the current block without distributing any rewards, consuming the block's single reward slot. Any unprivileged address can call this every block, silently starving all stakers of consensus-mode yield indefinitely.

---

### Finding Description
`update_rewards` is gated only by `general_prerequisites`, which enforces two conditions: the contract is not paused, and the caller is not the zero address. [1](#0-0) 

No role check, no staker-ownership check, and no restriction on the `disable_rewards` value is applied.

Inside the function, after validating that the staker is active and has non-zero STRK balance, the global `last_reward_block` is unconditionally written to the current block number: [2](#0-1) 

Immediately after, if `disable_rewards` is `true`, the function returns without distributing any rewards: [3](#0-2) 

The `REWARDS_ALREADY_UPDATED` guard at the top of the function ensures only one call per block can succeed: [4](#0-3) 

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` once per block therefore:
1. Advances `last_reward_block` to the current block.
2. Distributes zero rewards.
3. Causes every subsequent `update_rewards` call in the same block to revert with `REWARDS_ALREADY_UPDATED`.

The attacker needs only a valid (active, non-zero-balance) staker address as the argument — any staker in the `stakers` vector satisfies this, and the vector is public. [5](#0-4) 

---

### Impact Explanation
Consensus rewards are the sole yield mechanism once `consensus_rewards_first_epoch` is set (i.e., `is_pre_consensus()` returns `false`). Attestation-based rewards are explicitly blocked in that mode: [6](#0-5) 

By consuming every block's reward slot with `disable_rewards = true`, the attacker prevents all stakers and delegators from accruing any yield. Unclaimed yield is permanently frozen for the duration of the attack. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

---

### Likelihood Explanation
- Entry point is fully public; no role, no signature, no stake required.
- The attacker only needs to know one valid staker address (readable from the public `stakers` vector).
- Starknet transaction fees are low, making sustained per-block calls economically viable.
- The attack is silent — no on-chain event distinguishes a legitimate `disable_rewards: true` call from a malicious one.

---

### Recommendation
Restrict the `disable_rewards = true` path to a privileged role (e.g., `only_app_governor`
