### Title
`msg.sender`-Anchored Burns in CRC20/CRC21 Bridge Functions Cause Unauthorized Token Burns and Break Aggregator Bridge Flows - (File: `contracts/src/ModuleCRC20.sol`, `contracts/src/ModuleCRC21.sol`)

---

### Summary

`ModuleCRC20.send_to_ibc`, `ModuleCRC20.send_to_ethereum`, `ModuleCRC21.send_to_ibc`, and `ModuleCRC21.send_to_evm_chain` unconditionally burn or pull tokens from `msg.sender`. When an aggregator or router contract calls these functions on behalf of a user (a standard composability pattern), the aggregator's own CRC20/CRC21 balance is burned instead of the user's, causing unauthorized token destruction and a broken bridge/conversion flow.

---

### Finding Description

`ModuleCRC20.send_to_ibc` and `send_to_ethereum` call `unsafe_burn(msg.sender, amount)` and emit the event with `msg.sender` as the `sender` field: [1](#0-0) 

`ModuleCRC21.send_to_ibc` and `send_to_evm_chain` do the same — for non-source tokens they call `unsafe_burn(msg.sender, amount)`, and for source tokens they call `transferFrom(msg.sender, module_address, amount)`, both anchored to `msg.sender`: [2](#0-1) 

The EVM hook `SendToIbcHandler.handle` then reads the `sender` address directly from the event data and uses it as the IBC transfer originator: [3](#0-2) [4](#0-3) 

`SendToIbcV2Handler` reads `sender` from `topics[1]`, which is also set to `msg.sender` by the contract: [5](#0-4) 

There is no mechanism for an aggregator to specify the actual token holder as the burn/transfer source. The entire bridge pipeline — EVM burn, event emission, and Cosmos-side IBC transfer — is anchored to `msg.sender`.

---

### Impact Explanation

**Scenario A — Aggregator has no CRC20 balance:**
1. User approves aggregator to spend their CRC20 tokens.
2. Aggregator calls `send_to_ibc` to bridge the user's tokens.
3. `unsafe_burn(msg.sender, amount)` reverts with `ds-token-insufficient-balance`.
4. The bridge/conversion flow is permanently broken for any aggregator that does not pre-hold the tokens.

**Scenario B — Aggregator holds CRC20 balance:**
1. Aggregator calls `send_to_ibc` intending to bridge a user's tokens.
2. `unsafe_burn(msg.sender, amount)` burns the **aggregator's** tokens, not the user's.
3. The event emits the aggregator's address as `sender`.
4. The EVM hook transfers IBC coins from the contract address to the **aggregator's** Cosmos address and initiates IBC transfer from the aggregator.
5. The user's CRC20 tokens are **not** burned — a permanent accounting mismatch.
6. The aggregator suffers an unauthorized burn of its own CRC20/CRC21 assets.

This constitutes:
- An unauthorized burn of CRC20/CRC21 assets (Critical impact class).
- A permanent inability for honest aggregator contracts to process valid bridge/conversion flows (High impact class).

---

### Likelihood Explanation

Any smart contract that acts as a router, aggregator, or relayer for CRC20/CRC21 bridge operations is affected. This is a standard composability pattern in EVM ecosystems (e.g., DEX routers, cross-chain aggregators). The Cronos ecosystem explicitly supports proxy/aggregator patterns as evidenced by `ModuleCRC20Proxy` and `ModuleCRC21` proxy contracts in the codebase: [6](#0-5) 

The proxy contracts work around this by using `move`/`transferFrom` to pull tokens into themselves first, but the canonical `ModuleCRC20` and `ModuleCRC21` contracts have no such mechanism, and any third-party aggregator calling them directly is affected.

---

### Recommendation

Introduce an explicit `from` parameter to `send_to_ibc`, `send_to_ethereum`, and `send_to_evm_chain`, or require the caller to first transfer tokens to itself before calling the bridge function (as the proxy contracts do). At minimum, the event `sender` field and the burn target must always refer to the same address that actually holds the tokens being bridged, and that address must have authorized the operation.

For `ModuleCRC21` (source tokens), replace `transferFrom(msg.sender, module_address, amount)` with a pattern that allows a designated `from` address, validated via allowance, so aggregators can bridge on behalf of users without burning their own balance.

---

### Proof of Concept

```solidity
// Aggregator contract
contract BridgeAggregator {
    ModuleCRC20 crc20;

    // User calls this, having approved the aggregator to spend their CRC20
    function bridgeForUser(
        address user,
        string calldata ibcRecipient,
        uint256 amount
    ) external {
        // Pull user tokens into aggregator
        crc20.transferFrom(user, address(this), amount);
        // Now call send_to_ibc — msg.sender is the aggregator
        // unsafe_burn(msg.sender, amount) burns AGGREGATOR's tokens, not user's
        // The user's tokens remain in the aggregator (unburned, accounting mismatch)
        crc20.send_to_ibc(ibcRecipient, amount);
    }
}
```

After `bridgeForUser` executes:
- The aggregator's CRC20 balance is reduced by `amount` (unauthorized burn).
- The user's tokens transferred to the aggregator are **not** burned.
- The IBC transfer is initiated from the aggregator's Cosmos address, not the user's.
- Total CRC20 supply is reduced by `amount` but the corresponding IBC vouchers are credited to the aggregator, not the user — a permanent accounting divergence. [7](#0-6) [8](#0-7) [9](#0-8)
