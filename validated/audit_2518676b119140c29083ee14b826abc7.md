### Title
Unbounded Loop Over Pool Member Balance Trace Enables Permanent Freezing of Unclaimed Rewards - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace since their last claim. A delegator who makes many balance changes across different epochs without claiming rewards can grow this trace to an arbitrary size, eventually causing any reward-claiming or exit operation to run out of gas, permanently freezing their unclaimed yield.

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over the `pool_member_epoch_balance` trace to compute accumulated rewards: [1](#0-0) 

The code itself acknowledges the risk:

> **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.

Each entry in the trace is appended by `set_member_balance` (called via `increase_member_balance`) whenever a pool member's balance changes in a new epoch: [2](#0-1) 

The trace deduplicates within the same epoch key (if `last.key == key`, it updates in place), but inserts a **new** checkpoint for every distinct epoch in which a balance change occurs: [3](#0-2) 

Because `set_member_balance` always uses `get_epoch_plus_k()` as the key, one balance change per epoch produces one new trace entry per epoch. A delegator who calls `increase_delegate` (or `enter_delegation_pool` / partial-exit / re-enter) once per epoch, without ever calling `claim_rewards`, will accumulate one trace entry per epoch indefinitely.

The `entry_to_claim_from` cursor stored in the pool member's checkpoint advances only when rewards are claimed: [4](#0-3) 

So the loop length on the next claim equals the number of balance changes since the **last** claim. If the delegator never claims, this grows without bound.

### Impact Explanation

When the trace is large enough, any call that invokes `calculate_rewards` for that pool member — including `claim_rewards`, `exit_delegation_pool_action`, and `switch_delegation_pool` — will exhaust the Starknet transaction gas limit and revert. The delegator's accumulated rewards become permanently unclaimable, and their principal may also be locked if the exit action cannot complete.

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

A delegator needs only to make one small balance increase per epoch (the minimum non-zero amount) while deferring reward claims. Epochs are time-bounded, so after a sufficient number of epochs the trace grows large enough to exceed the gas limit. No privileged access is required; any pool member can reach this state through normal public entry points (`enter_delegation_pool`, `add_to_delegation_pool`).

### Recommendation

Enforce a maximum number of unclaimed balance-change entries per pool member, or require that rewards are claimed (and `entry_to_claim_from` advanced) before a new balance change is recorded. Alternatively, cap the number of trace entries that `calculate_rewards` processes in a single call and store the cursor so that reward claiming can be split across multiple transactions.

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with the minimum amount in epoch `E`.
2. In each subsequent epoch `E+1, E+2, …, E+N`, the delegator calls `add_to_delegation_pool` with the minimum non-zero amount. Each call appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `increase_member_balance`.
3. The delegator never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After `N` epochs, `pool_member_trace_length == N`.
5. The delegator (or anyone triggering an exit action) calls `claim_rewards` or `exit_delegation_pool_action`. The `calculate_rewards` loop executes `N` iterations, each reading storage and calling `find_sigma`. For sufficiently large `N`, the transaction exceeds the Starknet step limit and reverts.
6. All subsequent attempts to claim or exit also revert. The delegator's unclaimed yield is permanently frozen. [1](#0-0) [2](#0-1)
