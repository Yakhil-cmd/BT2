### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze All Stakers' Consensus Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `Staking` contract is intended to be callable only by the Starkware sequencer (per the protocol specification), but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` every block, permanently preventing all stakers from ever receiving consensus block rewards.

### Finding Description
The spec at `docs/spec.md` lines 1644–1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` performs no caller validation whatsoever: [1](#0-0) 

The function only checks that the current block number is greater than `last_reward_block` (a single **global** storage slot, not per-staker): [2](#0-1) 

It then unconditionally writes the current block number to `last_reward_block` before checking `disable_rewards`: [3](#0-2) 

Because `last_reward_block` is a single global value shared across all stakers: [4](#0-3) 

calling `update_rewards(any_valid_staker, disable_rewards: true)` once per block consumes the block's reward slot for **every** staker simultaneously. Any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

The spec confirms this is the intended sequencer-only mechanism for distributing per-block consensus rewards: [5](#0-4) 

### Impact Explanation
An attacker who calls `update_rewards(staker_address, disable_rewards: true)` at the start of every block:

1. Marks the block as "already processed" by writing `last_reward_block = current_block`.
2. Skips reward distribution because `disable_rewards` is `true`.
3. Blocks every other staker from receiving rewards for that block — the slot is consumed globally.

Since block rewards are never retroactively recalculated, the rewards for each skipped block are **permanently lost** for all stakers and their delegators. This constitutes **permanent freezing of unclaimed yield** for the entire protocol.

**Impact: High** — Permanent freezing of unclaimed yield / unclaimed royalties for all stakers and pool members.

### Likelihood Explanation
The entry point is fully public with no token holdings, staking, or privileged role required. Any EOA on Starknet can call it. The only cost is the gas for one transaction per block. A motivated attacker (e.g., a competing validator or someone shorting STRK) can sustain this indefinitely at minimal cost, completely halting consensus reward distribution.

### Recommendation
Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the designated Starkware sequencer address (or a stored `sequencer_address` role), consistent with the spec's stated access control. This mirrors the pattern already used in `update_rewards_from_attestation_contract`, which correctly asserts the caller is the attestation contract: [6](#0-5) 

### Proof of Concept

1. Consensus rewards are activated (`consensus_rewards_first_epoch` is set and passed).
2. A valid staker exists with non-zero balance at the current epoch.
3. Attacker (any address) calls:
   ```
   IStakingRewardsManager(staking_contract).update_rewards(
       staker_address: <any_valid_staker>,
       disable_rewards: true
   )
   ```
   at the first transaction of every block.
4. `last_reward_block` is set to the current block; no rewards are distributed.
5. Any legitimate sequencer call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. All stakers and their pool delegators receive zero consensus rewards for every block the attacker front-runs, permanently.
