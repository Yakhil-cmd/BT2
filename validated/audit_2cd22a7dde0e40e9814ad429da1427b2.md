### Title
Incorrect Multi-NFT Royalty Calculation in WalletConnect Approval Dialog Causes Understated Spend Amount - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in the WalletConnect command display pipeline incorrectly divides the fungible payment amount by the number of NFTs before computing each NFT's royalty. This causes the "Total Amount with Royalties" shown in the WalletConnect confirmation dialog to be systematically lower than the amount that will actually be spent on-chain, because Chia's protocol enforces each NFT's royalty against the full purchase price, not a split portion.

### Finding Description

In `packages/gui/src/electron/commands/parseCommandDisplay.ts`, `formatAmountWithRoyalties` computes the displayed royalty total as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The `splitAmount` division is wrong. Each NFT's royalty is independently owed on the **full** fungible amount, not on `amount / N`. The correct formula is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The buggy formula instead computes:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
```

which underestimates total royalties by a factor of N when all royalties are equal, and produces an incorrect result in the general case.

The existing test fixture encodes the wrong expected value (`'0.00010255'`) for a two-NFT offer (royalties 500 and 10 basis points on 100,000,000 mojos), confirming the bug is baked into the test suite: [2](#0-1) 

Correct calculation for that fixture:
- NFT1 royalty: `100,000,000 × 500 / 10,000 = 5,000,000` mojos  
- NFT2 royalty: `100,000,000 × 10 / 10,000 = 100,000` mojos  
- Correct total: `105,100,000` mojos = **0.0001051 XCH**  
- Displayed total (buggy): `102,550,000` mojos = **0.00010255 XCH**

The `amountWithRoyalties` value produced by this function is rendered directly in the WalletConnect confirmation dialog under the "You Spend" / "Total Amount with Royalties" label: [3](#0-2) 

This dialog is the sole security gate a user sees before approving a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` WalletConnect command: [4](#0-3) 

### Impact Explanation

An unprivileged dApp connected via WalletConnect can craft a `take_offer` payload containing multiple NFTs with non-trivial royalties. The confirmation dialog will display a "Total Amount with Royalties" that is materially lower than what the Chia protocol will actually deduct. Because NFT royalties are enforced at the protocol level, the user's wallet will spend the correct (higher) amount regardless of what the GUI showed. The user approved based on a false lower figure, constituting a WalletConnect state spoofing issue that causes the user to approve the wrong spend amount.

This maps to the **High** allowed impact: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation

Any WalletConnect-connected dApp can trigger this by including two or more NFTs with royalties in a single offer. No special privileges, leaked keys, or social engineering are required beyond a normal WalletConnect pairing. The discrepancy grows with the number of NFTs and the magnitude of royalties.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be calculated against the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

Update the corresponding test expectation in `parseCommandDisplay.test.ts` to `'0.0001051'`.

### Proof of Concept

Given a WalletConnect `chia_wallet.take_offer` command for an offer where:
- The taker pays **0.0001 XCH** (100,000,000 mojos)
- NFT A has `royalty_percentage = 500` (5 %)
- NFT B has `royalty_percentage = 10` (0.1 %)

The confirmation dialog displays **"Total Amount with Royalties: 0.00010255 XCH"** (the buggy value), while the Chia protocol will actually deduct **0.0001051 XCH** — a difference of 0.0000025 XCH per transaction. With higher royalties or more NFTs the gap scales proportionally. [5](#0-4) [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
