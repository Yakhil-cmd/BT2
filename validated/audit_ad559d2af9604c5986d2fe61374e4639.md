### Title
Missing Verification of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Confidential Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly sums cryptographic contributions `(norm_big_y, norm_big_c)` received from each participant without any proof of correctness. A single malicious participant can send arbitrary group elements, silently corrupting the derived confidential key for all honest parties. No final output verification exists to detect this.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `CKDOutput` and adds the components together unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is expected to compute:

- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

where `x_i` is the participant's private share and `y_i` is a fresh random scalar. [2](#0-1) 

The coordinator has no mechanism to verify that `norm_big_c_i` was computed using the participant's actual private share `x_i` and the same `y_i` used for `norm_big_y_i`. There is no zero-knowledge proof, no commitment binding `y_i` to both components, and no individual public-key-share check (the `KeygenOutput` only stores the aggregate public key, not per-participant verification shares).

After aggregation, the output is returned directly:

```rust
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [3](#0-2) 

The application then calls `unmask(app_sk)` which computes `C − app_sk · Y`. If either `C` or `Y` is corrupted, the result silently diverges from `msk · H(pk ‖ app_id)` with no error raised. The test confirms there is no internal verification in `unmask`:

```rust
let confidential_key = ckd_output.unmask(app_sk);
let expected_confidential_key = hash_app_id_with_pk(&pk, &app_id) * msk;
assert_eq!(confidential_key, expected_confidential_key, "Keys should be equal");
``` [4](#0-3) 

Contrast this with the OT-based ECDSA presign, which explicitly verifies that received shares are consistent with public commitments (`E =?= e·G`, `α·G =?= K + A`, `β·G =?= X + B`) before proceeding: [5](#0-4) 

The CKD protocol has no equivalent consistency checks.

---

### Impact Explanation

A single malicious participant sends `norm_big_c = 0` (or any arbitrary `ElementG1` value) while sending a valid `norm_big_y`. The coordinator sums all contributions and produces a `CKDOutput` where `C` is shifted by the malicious delta. Every honest party that calls `unmask(app_sk)` receives a wrong confidential key. Because there is no output verification step, honest parties silently accept the corrupted result.

This maps directly to: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. The attacker needs only to deviate from the honest `do_ckd_participant` computation and send a crafted `CKDOutput` over the channel. No special privilege, leaked key, or cryptographic break is required. The coordinator's `recv_from_others` loop accepts any well-typed message from any listed participant. [6](#0-5) 

---

### Recommendation

Add a zero-knowledge proof of correct contribution. Specifically, each participant should prove in zero-knowledge that:

1. `norm_big_y_i = λ_i · y_i · G` for some `y_i` (a Schnorr PoK of `y_i`).
2. `norm_big_c_i − norm_big_y_i · app_pk = λ_i · x_i · H(pk ‖ app_id)` (i.e., the "secret" component of `norm_big_c_i` is consistent with the participant's public key share).

Alternatively, store per-participant verification shares in `KeygenOutput` (analogous to FROST's `PublicKeyPackage`) so the coordinator can verify `norm_big_c_i − norm_big_y_i · app_pk =?= λ_i · X_i · H(pk ‖ app_id)` where `X_i = x_i · G` is the participant's public share.

---

### Proof of Concept

```
1. Honest participants P1, P2 and malicious participant P3 run ckd().
2. P3 computes norm_big_y correctly but sends norm_big_c = ElementG1::identity() (zero).
3. do_ckd_coordinator sums all contributions:
       Y = norm_big_y_1 + norm_big_y_2 + norm_big_y_3   (correct)
       C = norm_big_c_1 + norm_big_c_2 + 0              (missing P3's share)
4. ckd_output = CKDOutput::new(Y, C) is returned with no error.
5. Application calls ckd_output.unmask(app_sk):
       result = C - app_sk * Y
              = (lambda_1*x_1 + lambda_2*x_2) * H(pk, app_id)   ≠   msk * H(pk, app_id)
6. Honest parties silently receive a wrong confidential key with no indication of failure.
```
