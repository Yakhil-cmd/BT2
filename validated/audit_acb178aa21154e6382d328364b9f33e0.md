### Title
Unchecked Promise Result in `ExitToEthereum` Precompile Causes Permanent Fund Loss - (File: `engine-precompiles/src/native.rs`)

### Summary
The `ExitToEthereum` precompile burns ERC-20 tokens in the EVM and schedules a fire-and-forget `withdraw` NEAR promise to the eth-connector contract with no callback to handle failure. If the `withdraw` promise fails, the burned ERC-20 tokens are permanently lost with no refund path.

### Finding Description
In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run()` function unconditionally creates a `PromiseArgs::Create(withdraw_promise)` — a promise with no attached callback: [1](#0-0) 

This is structurally identical to the reported vulnerability: a cross-contract call whose return value (success/failure) is never checked. The execution sequence is:

1. An ERC-20 contract burns user tokens and calls the `ExitToEthereum` precompile.
2. The precompile schedules a `withdraw` call on the eth-connector NEP-141 contract.
3. The EVM transaction commits successfully — the token burn is final.
4. The NEAR `withdraw` promise executes asynchronously. If it fails for any reason, there is **no callback, no re-mint, and no refund path**.

Contrast this with `ExitToNear::run()`, which conditionally attaches an `exit_to_near_precompile_callback` to handle failures and re-mint tokens: [2](#0-1) 

The callback `exit_to_near_precompile_callback` explicitly handles the failure case by calling `engine::refund_on_error`: [3](#0-2) 

`ExitToEthereum` has no equivalent mechanism whatsoever.

### Impact Explanation
**Critical — Permanent freezing of funds.** ERC-20 tokens are burned in the EVM (state is committed and irreversible), but the corresponding ETH is never released on Ethereum if the `withdraw` promise fails. The user loses their tokens with no recovery path.

### Likelihood Explanation
The `withdraw` promise can fail due to:
- The eth-connector contract being paused or in an error state at the time the promise executes.
- Insufficient attached gas (`WITHDRAWAL_GAS = 100_000_000_000_000` TGas) for the eth-connector's internal logic under load.
- Any runtime panic or assertion failure inside the eth-connector's `withdraw` handler.

The fact that the Aurora team already implemented `error_refund` logic for `ExitToNear` (guarded by a feature flag) demonstrates awareness that promise failures are a realistic concern on this code path. The `ExitToEthereum` path has no analogous protection at all. [4](#0-3) 

### Recommendation
Add a callback to `ExitToEthereum` analogous to `exit_to_near_precompile_callback`. The callback should check `handler.promise_result(0)` and, on failure, call `engine::refund_on_error` to re-mint the burned ERC-20 tokens to the original sender. The `refund_on_error` function already supports the ERC-20 re-mint path: [5](#0-4) 

### Proof of Concept
1. User holds ERC-20 tokens on Aurora representing bridged ETH.
2. User calls the ERC-20 contract's `withdraw(amount, eth_recipient)` function, which burns `amount` tokens and calls the `ExitToEthereum` precompile (flag `0x01`).
3. The precompile schedules `PromiseArgs::Create(withdraw_promise)` targeting the eth-connector's `withdraw` method — no callback is attached.
4. The EVM transaction succeeds; ERC-20 tokens are burned.
5. The eth-connector's `withdraw` promise fails (e.g., connector is paused, or gas exhausted).
6. No callback fires. No re-mint occurs. The user's ERC-20 tokens are gone and no ETH is released on Ethereum. [6](#0-5)
