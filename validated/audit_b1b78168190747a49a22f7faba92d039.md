### Title
Signature Validation Bypass via `skip_stateful_validations` for Already-Deployed Accounts with Pending Nonce-0 Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry-point call for any invoke transaction with `nonce == 1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The check is intended to cover only the deploy-account + invoke UX flow, but `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from that address in the pool — including a legitimate nonce-0 invoke from an already-deployed account. An unprivileged attacker can therefore inject an invoke transaction carrying an arbitrary (invalid) signature into the mempool without passing signature validation.

### Finding Description

`skip_stateful_validations` is called from `run_pre_validation_checks` and returns `true` (skip) when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)`.
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` constructs an `AccountTransaction` with `execution_flags.validate = false` and passes it to `StatefulValidator::validate` (which delegates to `perform_validations`). Inside `perform_validations`, the invoke branch calls `perform_pre_validation_stage` (nonce/fee checks) and then immediately returns `Ok(())` without ever calling `validate_tx` / the `__validate__` entry point: [2](#0-1) [3](#0-2) 

The comment in `skip_stateful_validations` justifies the check as follows:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy-account transaction **or transactions with future nonces that passed validations**."

However, `account_tx_in_pool_or_recent_block` is implemented as: [4](#0-3) 

This returns `true` for **any** transaction from the address — including a nonce-0 invoke submitted by the legitimate account owner for an already-deployed account (on-chain nonce = 0). The presence of a nonce-0 invoke in the pool does **not** imply a deploy-account is pending; the account may already be fully deployed.

**Attack scenario:**

1. Account `A` is deployed on-chain; its nonce is `0`.
2. The legitimate owner submits a nonce-0 invoke for `A`. It passes full stateful validation (signature checked) and enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. An attacker submits a nonce-1 invoke for `A` with an **invalid signature** (e.g., all-zero signature, or a signature for a different account).
4. Gateway stateful validation:
   - `validate_nonce`: `0 ≤ 1 ≤ 200` — passes.
   - `validate_by_mempool`: no duplicate nonce — passes.
   - `skip_stateful_validations`: nonce=1, account_nonce=0, account in pool → returns `true`.
   - `run_validate_entry_point`: `validate = false` → `__validate__` **never called**.
5. The attacker's invalid-signature invoke is admitted to the mempool.

During block execution the batcher creates transactions via `AccountTransaction::new_for_sequencing`, which always sets `validate: true`: [5](#0-4) 

The invalid transaction is therefore rejected at execution time (non-revertible `PanicInValidate` / `InvalidValidateReturnData` error), but it was already admitted to the mempool without signature verification.

### Impact Explanation

The gateway's stateful validator — the last line of defence before mempool admission — accepts a transaction whose signature has never been verified. This directly satisfies the **High** impact criterion:

> "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

An attacker can spam the mempool with signature-invalid invokes targeting any account that has a pending nonce-0 transaction, bypassing the `__validate__` entry-point check entirely. The invalid transactions consume mempool slots, can displace legitimate future-nonce transactions, and force the batcher to waste execution resources on transactions that will be hard-rejected.

### Likelihood Explanation

The precondition — an already-deployed account with on-chain nonce 0 and a pending nonce-0 invoke in the mempool — is common during normal operation (any account sending its very first transaction after deployment). The attacker needs only to observe the mempool for such accounts (or predict them) and submit a nonce-1 invoke with a garbage signature. No privileged access, no key material, and no special tooling are required.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction is pending for the address. Alternatively, restrict the skip to the case where the account does **not** yet exist on-chain (i.e., `get_class_hash_at(sender_address)` returns the zero class hash), which is the only situation where running `__validate__` would fail for a legitimate post-deploy invoke. A minimal fix:

```rust
// Only skip validation when the account contract does not yet exist on-chain.
// This is the sole legitimate reason the __validate__ entry point cannot run.
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    let class_hash = state_reader.get_class_hash_at(tx.sender_address())?;
    if class_hash == ClassHash::default() {
        // Account not yet deployed; check that a deploy_account is pending.
        return mempool_client
            .account_tx_in_pool_or_recent_block(tx.sender_address())
            .await ...;
    }
}
```

2. Fund A with enough STRK to pass fee checks.
3. Submit invoke_v3 from A with nonce=0, valid signature → accepted, enters mempool.
   account_tx_in_pool_or_recent_block(A) == true.
4. Submit invoke_v3 from A with nonce=1, signature=[0x0, 0x0] (invalid).

     run_validate_entry_point: validate=false → __validate__ NOT called  ✓
   → Transaction admitted to mempool with invalid signature.
5. Batcher picks up both transactions.
   nonce-0 invoke executes; account nonce advances to 1.
   nonce-1 invoke: __validate__ called → signature invalid → PanicInValidate → rejected.
   Invalid transaction was in the mempool without ever having its signature checked.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
