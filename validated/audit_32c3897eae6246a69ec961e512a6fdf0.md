### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`, `metric-core/contracts/MetricOmmPool.sol`)

### Summary
`setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` (both `uint16`, max 65535 = 6.5535% in E6) directly to the pool with **no cap validation**, while the parallel global admin spread fee path (`setPoolAdminFees`) enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin fees to the `uint16` maximum on the active bin, charging traders up to 6.5535% additional fee on top of the base oracle spread, bypassing the factory-enforced cap entirely.

### Finding Description

**Factory — `setPoolAdminFees` (capped):** [1](#0-0) 

**Factory — `setPoolBinAdditionalFees` (no cap):** [2](#0-1) 

**Pool — `setBinAdditionalFees` (only checks bin index, not fee value):** [3](#0-2) 

**Swap math — per-bin fee applied additively to base fee:** [4](#0-3) [5](#0-4) 

The `BinState.addFeeBuyE6` field is `uint16`, so the pool admin can write any value 0–65535 to any bin's additional fee. The swap math adds `Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)` directly to `baseFeeX64` before computing the required token1 input. There is no check anywhere in the factory or pool that bounds this value against `maxAdminSpreadFeeE6` or any other cap.

### Impact Explanation
If `maxAdminSpreadFeeE6` is set to, say, 1000 (0.1%), the pool admin is still free to call `setPoolBinAdditionalFees(pool, curBinIdx, 65535, 65535)`, writing 6.5535% additional buy/sell fees onto the active bin. Every subsequent swap through that bin pays 6.5535% more than the oracle-anchored price permits, with the excess accruing to LPs (not the admin directly, but the admin controls the pool). Traders receive materially less output than the oracle price allows, constituting bad-price execution above Sherlock thresholds.

### Likelihood Explanation
The pool admin is a semi-trusted role constrained by caps and timelocks. The Smart Audit Pivots explicitly flag `setBinAdditionalFees` as an admin-boundary check point. The asymmetry — global spread fee capped, per-bin fee uncapped — is a code-level design gap, not a documentation guideline. Any pool admin (including a compromised multisig) can exploit this immediately with a single transaction, no timelock required.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Or introduce a dedicated `maxAdminBinAdditionalFeeE6` cap. Either way, the cap must be enforced in the factory before the pool call, mirroring the pattern in `setPoolAdminFees`. [6](#0-5)
