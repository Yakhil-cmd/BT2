### Title
Missing Participant-Set Binding Between Triple Generation and Presigning Allows Permanent Denial of Signing - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary
The OT-based ECDSA `presign()` function explicitly omits the check that the presigning participant set matches the participant set that generated the Beaver triples. The `TriplePub` struct records the generating participants, but `presign()` never compares them against the active presigning participants. A malicious coordinator can supply triples generated by a different participant set, causing the internal consistency check to fail and permanently denying signing for honest parties.

### Finding Description
The protocol specification (`docs/ecdsa/ot_based_ecdsa/signing.md`) requires that the two Beaver triples used in presigning were **generated by the same set of parties P₀** as the presigning session. The `TriplePub` struct stores this information in its `participants` field:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    pub threshold: ReconstructionLowerBound,
}
```

However, `presign()` explicitly skips this check with a comment acknowledging the omission:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
```

Only the `threshold` field is validated against the triple:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
```

The `participants` field of `TriplePub` is never compared to the presigning `participants` argument. A caller can pass `PresignArguments` where `triple0.1.participants` and `triple1.1.participants` differ from the presigning participant list, and `presign()` will accept the call and begin the protocol.

When the protocol executes, each party linearizes their triple shares using Lagrange coefficients derived from the **presigning** participant set (not the triple-generation set):

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
```

Because the Lagrange coefficients are wrong (computed for the wrong participant set), the summed values `alpha` and `beta` will not satisfy the commitment check:

```rust
if (ProjectivePoint::GENERATOR * alpha != big_k + big_a)
    || (ProjectivePoint::GENERATOR * beta != big_x + big_b)
{
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of additive triple phase.".to_string(),
    ));
}
```

Every honest party aborts. No presignature is produced.

### Impact Explanation
A malicious coordinator who controls which `TriplePub` values are distributed to participants can supply triples generated by a different participant set (e.g., from a prior session with a different membership). Because the library does not validate the participant-set binding, the presigning protocol always aborts at the consistency check. As long as the coordinator continues to supply mismatched triples, honest parties can never complete presigning and therefore can never produce a signature. This is a **permanent denial of signing** for honest parties.

### Likelihood Explanation
The coordinator role is explicitly modeled in the signing protocol and is a documented trust boundary. A malicious coordinator has direct control over which `TriplePub` values are handed to each participant (the coordinator orchestrates the session setup). The attack requires no cryptographic break, no leaked keys, and no external assumptions — only the ability to supply a `TriplePub` whose `participants` field differs from the presigning participant list, which the library accepts without complaint.

### Recommendation
Add an explicit check in `presign()` that the presigning participant set matches the participant set recorded in both `TriplePub` values before the protocol begins:

```rust
// Enforce the protocol requirement that triples were generated by the same
// participant set as the presigning session.
if args.triple0.1.participants.as_slice() != participants.participants()
    || args.triple1.1.participants.as_slice() != participants.participants()
{
    return Err(InitializationError::BadParameters(
        "Triple participants must exactly match presigning participants".to_string(),
    ));
}
```

This mirrors the threshold check already present and closes the gap acknowledged by the existing comment.

### Proof of Concept
1. Run triple generation with participant set P₀ = {A, B, C}, producing `TriplePub` T₀ (with `T₀.participants = [A, B, C]`) and per-party shares.
2. Initiate a presigning session with participant set P₁ = {A, B, D} (D ∉ P₀), but supply T₀ as the `TriplePub` argument to all participants.
3. Each participant calls `presign(&[A, B, D], me, PresignArguments { triple0: (share, T₀), ... })`. The library accepts this call — `T₀.threshold` matches, and `T₀.participants` is never checked.
4. Inside `do_presign`, each party computes `lambda_me = participants.lagrange(me)` using P₁'s identifiers, then linearizes their P₀ shares with those wrong coefficients.
5. The summed `alpha` and `beta` do not satisfy `alpha·G = K + A` (commitments from T₀), so every honest party hits the `AssertionFailed` error at line 162–168 of `presign.rs` and aborts.
6. The coordinator repeats this with any mismatched triple, permanently preventing presigning from completing. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
