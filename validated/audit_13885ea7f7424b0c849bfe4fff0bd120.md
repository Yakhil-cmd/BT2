### Title
NFT Metadata Accepted Without Integrity Verification When `metadataHash` Is Absent — (`packages/gui/src/hooks/useFetchAndProcessMetadata.ts`)

### Summary

`useFetchAndProcessMetadata` skips the SHA-256 integrity check on fetched NFT metadata whenever the on-chain `metadataHash` field is falsy (empty string or `undefined`). Because the Chia NFT standard permits minting without a metadata hash, an attacker who controls the metadata server for such an NFT can silently serve arbitrary metadata content. The GUI displays this unverified metadata — including NFT name, description, and preview images — in offer-acceptance flows, enabling a bait-and-switch attack that causes a victim to accept an offer for a worthless NFT while believing they are receiving a valuable one.

### Finding Description

**Root cause — `useFetchAndProcessMetadata.ts` line 23:**

```typescript
if (hash && !compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
```

The guard is `hash && …`. When `metadataHash` is `''` or `undefined` (both falsy), the entire checksum comparison is bypassed and the function returns the fetched metadata as trusted. [1](#0-0) 

**Call site — `useMetadataData.ts` line 76–83:**

```typescript
const { metadataUris = [], metadataHash } = nft;
const [firstUri] = metadataUris;
const metadata = await fetchAndProcessMetadata(firstUri, metadataHash);
```

`metadataHash` is taken directly from the RPC response. The `NFTInfo` type declares it as `string`, but the GUI itself guards `nft.metadataHash && nft.metadataHash !== '0x'` before displaying it, confirming that falsy values are a real runtime possibility. [2](#0-1) [3](#0-2) 

**No user-visible warning for missing metadata hash.** The `useNFTVerifyHash` hook (which drives `NFTHashStatus`) checks the *data* hash (`dataHash` / `dataUris`), not the *metadata* hash. When `dataHash` is also absent it returns `undefined` (unverified), but when only `metadataHash` is absent the data-hash indicator may still show "verified" while the metadata JSON itself is completely unprotected. [4](#0-3) 

**Metadata flows into offer-acceptance UI.** `useMetadataData` feeds `useNFTMetadata`, which is consumed by `NotificationOffer`, `NotificationPreviewOffer`, and `OfferDetails` — all rendered inside the offer-acceptance dialog. The NFT name and preview image shown to the user when deciding whether to accept an offer come from this unverified metadata. [5](#0-4) 

**Exploit path:**

1. Attacker mints an NFT with `metadataHash` omitted (or set to `""`) and `metadataUris[0]` pointing to an attacker-controlled server.
2. Initially the server returns legitimate-looking metadata (e.g., `{"name":"Common Pebble","description":"..."}`).
3. When targeting a victim, the attacker switches the server response to `{"name":"Rare Dragon #1","description":"Floor price 500 XCH","preview_image_uris":["https://attacker.io/dragon.png"]}`.
4. The attacker sends the victim an offer notification (via the on-chain notification mechanism or WalletConnect `chia_showNotification`) for this NFT.
5. The GUI calls `fetchAndProcessMetadata(firstUri, undefined)` — the `if (hash && …)` guard is false, the checksum is never compared, and the fake metadata is stored and rendered.
6. The victim sees "Rare Dragon #1" with a convincing preview image and accepts the offer.
7. The victim receives the worthless NFT; the attacker receives the victim's XCH or CATs.

### Impact Explanation

A victim is caused to **accept an offer for the wrong asset** — they believe they are receiving a high-value NFT but receive a worthless one. This is a direct, irreversible financial loss. It maps to the allowed High impact: *"unsafe trust of … NFT metadata … that causes a user to approve … the wrong asset, identity."*

### Likelihood Explanation

- Minting an NFT without a metadata hash is valid per the Chia NFT standard and requires only a small XCH fee — no special privilege.
- The attacker controls their own metadata server; no third-party compromise is needed.
- The offer notification mechanism (on-chain messages or WalletConnect `chia_showNotification`) is available to any unprivileged actor.
- The victim receives no warning that the displayed metadata is unverified (the data-hash indicator is separate and may show "verified" for the image file even when the metadata JSON is unprotected).

### Recommendation

Replace the falsy guard with an explicit `undefined`/`null` check and treat a missing hash as an unverifiable state that is surfaced to the user rather than silently accepted:

```typescript
if (hash == null || hash === '' || hash === '0x') {
  // No hash provided — metadata integrity cannot be verified.
  // Either throw, or return the metadata tagged as unverified
  // and let the caller decide whether to display a warning.
  throw new Error('Metadata hash is missing; integrity cannot be verified');
}
if (!compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
```

Additionally, expose a `metadataHashVerified` flag in the NFT metadata state so that offer-acceptance dialogs can display a prominent warning when metadata integrity is unconfirmed. [6](#0-5) 

### Proof of Concept

1. Mint an NFT on testnet with `metadata_uris=["https://attacker.io/meta.json"]` and no `metadata_hash` field (or `metadata_hash=""`).
2. Have `https://attacker.io/meta.json` initially return `{"name":"Pebble"}`.
3. Open the Chia GUI and confirm the NFT displays "Pebble".
4. Switch the server to return `{"name":"Rare Dragon #1","preview_image_uris":["https://attacker.io/dragon.png"]}`.
5. Send an offer notification for this NFT to a second wallet.
6. In the second wallet, observe that the offer notification displays "Rare Dragon #1" with the dragon preview image — no integrity-failure warning is shown.
7. Confirm that `fetchAndProcessMetadata` was called with `hash = undefined` and that the `if (hash && …)` branch was never entered (add a `console.log` or use the debugger).
