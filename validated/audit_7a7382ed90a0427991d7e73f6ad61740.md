### Title
Unregistered EVM Contract Can Force-Revert Any Transaction via Crafted Bridge Event Log — (`x/cronos/keeper/evm_hooks.go`, `x/cronos/keeper/evmhandlers/send_to_account.go`)

### Summary

`PostTxProcessing` propagates errors from `SendToAccountHandler.Handle` (and `SendToIbcHandler.Handle`) without distinguishing between "unregistered contract" and "legitimate failure." Any unprivileged EVM contract can emit a properly ABI-encoded `__CronosSendToAccount` log to force-revert its own transaction at the post-processing stage, even though EVM execution succeeded.

### Finding Description

**Step 1 — `PostTxProcessing` propagates handler errors unconditionally**

In `x/cronos/keeper/evm_hooks.go`, `PostTxProcessing` iterates over all receipt logs. When `Topics[0]` matches a registered handler, it calls `handler.Handle(...)` and immediately returns any non-nil error: [1](#0-0) 

**Step 2 — `SendToAccountHandler.Handle` returns an error for unregistered contracts with valid data**

In `x/cronos/keeper/evmhandlers/send_to_account.go`:

- Lines 69–73: If `Unpack` fails (nil/malformed data), the handler logs and returns `nil` — safe.
- Lines 76–79: If `Unpack` **succeeds** but `GetDenomByContract` returns `!found`, the handler returns a hard error: [2](#0-1) 

The same pattern exists in `SendToIbcHandler.handle`: [3](#0-2) 

**Step 3 — The test suite confirms the error path**

The existing test `"not enough balance, expect fail"` in `evmhandlers_test.go` explicitly confirms that an unregistered contract with valid ABI-encoded data produces the error `"contract 0x... is not connected to native token"`: [4](#0-3) 

**Step 4 — The attack path**

1. Attacker deploys a contract that emits `__CronosSendToAccount(someAddress, someAmount)` with valid ABI encoding (so `Unpack` succeeds).
2. The contract address is not in the denom-contract mapping.
3. A victim calls the attacker's contract (directly or via a DEX/router).
4. EVM execution succeeds; the log is appended to the receipt.
5. `PostTxProcessing` picks up the log, calls `SendToAccountHandler.Handle`.
6. `Unpack` succeeds → `GetDenomByContract` returns `!found` → returns error.
7. `PostTxProcessing` returns the error → the entire EVM transaction is reverted.

The distinction from the "invalid log data, but still success" test case is critical: that test uses `nil` data (Unpack fails → nil returned). The attacker must supply valid ABI-encoded data to bypass the nil-data guard. [5](#0-4) 

### Impact Explanation

Any unprivileged contract can permanently cause every transaction that calls it to be reverted at post-processing, despite EVM execution succeeding. This breaks the EVM execution/finality invariant and enables:

- Griefing of any user or protocol that interacts with the malicious contract (e.g., via a DEX router, callback, or composable call).
- Selective transaction blocking: an attacker can deploy a contract that conditionally emits the event to selectively revert specific callers.

This fits **High: Permanent or long-lived inability for honest users or validators to process valid transactions** and **High: Corruption of EVM receipt/log processing with direct security impact**.

### Likelihood Explanation

Exploitability is trivial: deploy a contract, emit a valid ABI-encoded `__CronosSendToAccount` log. No privileged access, no leaked keys, no external assumptions required. The PoC is a single Solidity contract.

### Recommendation

In `SendToAccountHandler.Handle` (and `SendToIbcHandler.handle`), when `GetDenomByContract` returns `!found`, **log and return `nil`** instead of returning an error — consistent with how decode failures are already handled:

```go
denom, found := h.cronosKeeper.GetDenomByContract(ctx, contract)
if !found {
    h.cronosKeeper.Logger(ctx).Error("contract not connected to native token, skipping", "contract", contract)
    return nil  // was: return fmt.Errorf(...)
}
```

This matches the existing pattern for malformed data (lines 70–73 of `send_to_account.go`) and restores the invariant that unregistered contracts cannot affect transaction finality.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract MaliciousEmitter {
    // Matches __CronosSendToAccount(address,uint256) ABI signature
    event __CronosSendToAccount(address recipient, uint256 amount);

    function attack(address recipient) external {
        // Emit valid ABI-encoded log from unregistered address
        emit __CronosSendToAccount(recipient, 1);
    }
}
```

Deploy `MaliciousEmitter` (address not in denom-contract mapping). Call `attack(anyAddress)`. Assert: EVM execution succeeds, but `PostTxProcessing` returns `"contract <MaliciousEmitter> is not connected to native token"`, reverting the transaction. The existing test at `evmhandlers_test.go:44–59` already demonstrates this error path with an unregistered contract.
