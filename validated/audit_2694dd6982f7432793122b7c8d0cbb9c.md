### Title
Staker Can Instantly Increase Commission to 100% via `set_commission_commitment` + `set_commission` in the Same Block, Stealing Delegator Yield - (File: `src/staking/staking.cairo`)

---

### Summary
A staker can atomically call `set_commission_commitment` followed by `set_commission` in the same block to raise their commission from any value up to 100% with zero delay. The new commission takes effect immediately for the next reward calculation, allowing the staker to front-run delegators and redirect all pool rewards to themselves.

---

### Finding Description

The `set_commission_commitment` mechanism was introduced to allow stakers to increase their commission (normally only decreases are allowed). However, there is no mandatory waiting period between setting the commitment and using it to raise the commission.

The flow is:

1. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`. The only constraint is `max_commission >= current_commission` and `expiration_epoch > current_epoch`.
2. In the **same block**, the staker calls `set_commission(commission: 10000)`. Because an active commitment exists, the check at line 1584–1589 only requires `commission <= commitment.max_commission` and `commission != old_commission` — no time lock, no delay.
3. The commission is written to storage immediately at line 1600.
4. At the next attestation, `calculate_staker_pools_rewards` reads the commission at line 1964 and applies it to split rewards. Delegators receive `pool_rewards_including_commission * (1 - commission/10000)`, which is now 0.

The codebase itself acknowledges this gap with the comment at line 745–746:

> `/// **Note**: Current commission increase safeguards still allow for sudden commission changes.` [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker with delegators can raise commission to 10000 (100%) immediately before attestation. The `split_rewards_with_commission` call at line 1989–1991 then assigns the entire pool reward to the staker as commission, leaving delegators with 0 rewards for that epoch. Delegators have no on-chain mechanism to exit before the next attestation fires, because the exit intent + wait window spans multiple epochs. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** Any staker who has accumulated delegators has a direct financial incentive to execute this attack at the moment their pool holds the most unclaimed yield. The two-transaction sequence (`set_commission_commitment` then `set_commission`) can be submitted in the same Starknet block. No special access, leaked key, or external dependency is required — only the staker's own private key.

---

### Recommendation

Enforce a mandatory delay between setting a commission commitment and being allowed to increase the commission under that commitment. For example:

- Require that `set_commission` with an upward change can only be called at least `N` epochs after `set_commission_commitment` was recorded.
- Alternatively, make commission increases take effect only at the start of the epoch **after** the one in which `set_commission` was called (mirror the epoch-delayed balance trace pattern already used for stake changes).

---

### Proof of Concept

```
// Staker starts with commission = 500 (5%)
// Step 1 — same block, tx 1:
staking.set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1);

// Step 2 — same block, tx 2 (or next tx in same block):
staking.set_commission(commission: 10000);  // passes: 10000 <= 10000 && 10000 != 500

// Step 3 — attestation fires in the same or next epoch:
// calculate_staker_pools_rewards reads commission = 10000
// split_rewards_with_commission → commission_rewards = pool_rewards_including_commission
//                               → pool_rewards = 0
// Delegators receive 0 STRK/BTC rewards; staker captures 100%.

            // Prerequisites and asserts.

                    );

                self.emit(Events::CommissionInitialized { staker_address, commission });

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );
        }
