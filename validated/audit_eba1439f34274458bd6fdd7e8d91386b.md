### Title
Unchecked Precompile Call Return Value in `withdrawToNear` and `withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** invoking the Aurora exit precompile via inline assembly. The return value of the assembly `call` is captured in a local variable (`res`) but is **never checked**. If the precompile call fails for any reason, the function returns normally without reverting, leaving the user's tokens permanently burned with no corresponding withdrawal on the NEAR side.

---

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // (1) tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is assigned but never read; no revert on failure
    }
}

The identical pattern appears in `EvmErc20V2.sol`: [2](#0-1) 

And in `withdrawToEthereum` in both contracts: [3](#0-2) [4](#0-3) 

The Aurora `ExitToNear` precompile (at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) and `ExitToEthereum` precompile (at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) can and do return failure (EVM `call` returns `0`) under documented error conditions. Looking at the precompile implementation, it explicitly returns `ExitError` variants for conditions such as invalid recipient address, wrong input size, or ETH attached to an ERC-20 exit: [5](#0-4) [6](#0-5) 

When the EVM `call` to a precompile fails, it returns `0`. Because neither contract checks `res` and neither reverts on `res == 0`, execution continues past the assembly block. The `_burn` at step (1) has already permanently destroyed the tokens from the user's balance, but no NEAR-side `ft_transfer` or Ethereum-side withdrawal is ever initiated.

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` with input that causes the precompile to fail (e.g., a malformed recipient, insufficient gas forwarded to the precompile, or any other precompile-level error) will have their ERC-20 tokens burned with zero recourse. The NEP-141 backing tokens remain locked in the connector contract, but the EVM-side mirror tokens are gone. The user suffers a total, irreversible loss of the withdrawn amount.

---

### Likelihood Explanation

**Medium.** Every token holder who calls `withdrawToNear` or `withdrawToEthereum` is exposed. The precompile can fail on:

- A recipient byte string that does not decode to a valid NEAR account ID (user-supplied `recipient` in `withdrawToNear`).
- Insufficient gas reaching the precompile (the `gas()` opcode forwards all remaining gas, but if the outer transaction is submitted with a tight gas limit the precompile may run out).
- Any future precompile-level validation added to the engine that rejects previously accepted inputs.

No special privilege is required; any token holder can trigger this path by calling the public `withdrawToNear` or `withdrawToEthereum` functions.

---

### Recommendation

Check the return value of the assembly `call` and revert if it indicates failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to all four assembly blocks across `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions so the `_burn` is only executed **after** a successful precompile call, or use a two-step commit/reveal pattern.

---

### Proof of Concept

1. Deploy `EvmErc20` with a NEP-141 backing token.
2. Mint tokens to address `Alice`.
3. Alice calls `withdrawToNear(invalidRecipient, amount)` where `invalidRecipient` is a byte string that the `ExitToNear` precompile rejects (e.g., an empty byte string or one that fails NEAR account-ID validation).
4. The precompile returns failure (`call` returns `0`); the assembly block does not revert.
5. `withdrawToNear` returns normally.
6. Alice's ERC-20 balance is now `0` (tokens burned at step 1 of the function).
7. No `ft_transfer` was issued to the NEAR connector; Alice receives nothing on the NEAR side.
8. Funds are permanently lost. [1](#0-0) [2](#0-1)
