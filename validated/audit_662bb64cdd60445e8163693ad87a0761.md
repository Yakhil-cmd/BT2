Looking at the critical code path:

### Title
`_afterTimelock` uint32 Truncation Allows Pool Admin to Bypass Stop-Loss Timelocks — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` computes `uint32(block.timestamp + timelock)`. When `timelock = type(uint32).max`, the addition is performed in `uint256` space (no revert), then the explicit `uint32(...)` downcast silently truncates the result to `block.timestamp - 1` — a past timestamp. Because `_requireElapsed` passes whenever `block.timestamp >= executeAfter`, the pool admin can immediately execute any drawdown, decay, or watermark change with zero delay, fully bypassing the LP-protection timelock.

---

### Finding Description

`_afterTimelock` is the single function that stamps every pending-change deadline:

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32` and is implicitly promoted to `uint256` for the addition. The addition itself cannot overflow `uint256`. However, the explicit `uint32(...)` downcast is a **truncating cast** — Solidity 0.8+ overflow protection does not apply to explicit casts. With `timelock = type(uint32).max = 4_294_967_295` and `block.timestamp ≈ 1_753_142_400` (current era):

```
block.timestamp + type(uint32).max = 6_048_109_695
uint32(6_048_109_695)              = 6_048_109_695 mod 2^32
                                   = 1_753_142_399   ← block.timestamp - 1
```

`pendingDrawdownExecuteAfter` (and equivalently `pendingDecayExecuteAfter`, `pendingTimelockExecuteAfter`, `PendingHighWatermarks.executeAfter`) is set to a timestamp **one second in the past**.

The guard:

```solidity
// line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

evaluates `block.timestamp < block.timestamp - 1` → `false` → no revert. The execute call succeeds in the same block as the propose call.

There is **no validation** on the `timelock` value in either `initialize` or `proposeOracleStopLossTimelock`:

### Impact Explanation

The timelock is the sole mechanism bounding the pool admin's power over LP-protection parameters (drawdown floor, decay rate, high watermarks). With the bypass active, the pool admin can:

1. Immediately raise `drawdownE6` to `1e6` (100% drawdown = stop-loss disabled), removing all value-loss protection for LPs.
2. Immediately raise `decayPerSecondE8` to `1e8` (100%/second decay), collapsing watermarks to zero within one second and disabling the stop-loss in practice.
3. Immediately reset high watermarks to zero, erasing accumulated LP-protection state.

All three actions are supposed to require the full timelock delay so LPs can observe and exit. The bypass eliminates that window.

The deceptive variant is particularly harmful: the pool admin proposes `timelock = type(uint32).max` (which appears to LPs as a ~136-year maximum-protection timelock), waits for the current timelock to elapse, executes the change, and then immediately executes any drawdown/decay/watermark change with zero delay — while LPs believe they have maximum protection.

---

### Likelihood Explanation

- The pool admin is a semi-trusted role explicitly in scope for timelock-bypass analysis.
- `type(uint32).max` is a valid `uint32` value; no on-chain guard rejects it.
- The two-step path (set timelock → exploit overflow) requires only pool-admin access and patience equal to the current timelock.
- The deceptive variant (appearing to set maximum protection) gives LPs no signal to exit before the bypass is active.

---

### Recommendation

Cap the timelock to a safe maximum (e.g., `365 days = 31_536_000`, well below `type(uint32).max = 4_294_967_295`) in both `initialize` and `proposeOracleStopLossTimelock`, and use safe arithmetic in `_afterTimelock`:

```solidity
uint256 private constant MAX_TIMELOCK = 365 days;

function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 tl = oracleStopLossConfig[pool_].timelock;
    uint256 result = block.timestamp + tl;
    require(result <= type(uint32).max, "timelock overflow");
    return uint32(result);
}
```

Or equivalently, validate `newTimelock <= MAX_TIMELOCK` at proposal/initialization time so the overflow can never be reached.
