### Title
Unprivileged Caller Can Suppress Consensus Reward Distribution via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

---

### Summary
The `update_rewards` function in `IStakingRewardsManager` has no access control beyond a basic pause/zero-address check. Any public caller can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` to the current block without distributing any rewards. Because `last_reward_block` is a single global value, this blocks all stakers from receiving consensus rewards for that block. Repeating this once per block permanently freezes unclaimed yield for the entire protocol.

---

### Finding Description

`update_rewards` is exposed as a public function with no role restriction: [1](#0-0) 

The only guard is `general_prerequisites()`, which only checks the pause flag and that the caller is non-zero: [2](#0-1) 

Inside `update_rewards`, the global `last_reward_block` is unconditionally written to the current block **before** the `disable_rewards` branch is evaluated: [3](#0-2) 

If `disable_rewards` is `true`, the function returns immediately without distributing any rewards: [4](#0-3) 

Because `last_reward_block` is a single global storage slot shared across all stakers: [5](#0-4) 

any subsequent call to `update_rewards` in the same block will fail the guard: [6](#0-5) 

This means one call per block with `disable_rewards: true` is sufficient to prevent every staker from receiving consensus block rewards for that block.

By contrast, the attestation-based reward path (`update_rewards_from_attestation_contract`) is correctly gated: [7](#0-6) 

The `disable_rewards` parameter is a privileged control knob (it governs whether rewards are actually distributed) but it is exposed to any public caller with no role separation.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker calling `update_rewards(any_active_staker, disable_rewards: true)` once per block:
- Advances `last_reward_block` to the current block with zero reward distribution.
- Blocks every other staker from receiving consensus rewards for that block (the global lock prevents any further `update_rewards` call in the same block).
- Costs only gas; requires no stake, no special role, and no funds.

Sustained over time, this permanently freezes all consensus-era unclaimed yield for all stakers and delegators in the protocol.

---

### Likelihood Explanation

**High.** The function is publicly callable with no access control. The attacker only needs to know any active staker address (readable from on-chain events or `get_stakers()`). The attack is cheap (one transaction per block) and requires no capital.

---

### Recommendation

1. **Restrict `update_rewards` to a trusted caller** (e.g., the attestation contract, a dedicated keeper role, or the staker themselves). Add an `assert_caller_is_attestation_contract()` guard or a dedicated role check analogous to the one already present in `update_rewards_from_attestation_contract`.
2. **Separate the `disable_rewards` control path** from the public interface. If `disable_rewards: true` is needed for governance/migration purposes, gate it behind `only_app_governor()` or `only_token_admin()`.
3. **Write `last_reward_block` only after confirming rewards will be distributed**, or use a per-staker reward block tracker instead of a single global one.

---

### Proof of Concept

```
// Attacker (any EOA, no stake required):
// 1. Read any active staker address from on-chain events.
let victim_staker = <any active staker with non-zero balance>;

// 2. Once per block, call:
staking_contract.update_rewards(victim_staker, disable_rewards: true);

// Effect:
// - last_reward_block is set to current_block_number.
// - No rewards are distributed to victim_staker or any other staker.
// - All subsequent update_rewards calls in this block revert with REWARDS_ALREADY_UPDATED.
// - Repeating this every block permanently freezes all consensus reward accrual.
```
