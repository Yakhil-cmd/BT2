### Title
Malicious CKD Participant Can Corrupt the Derived Confidential Key Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator blindly accumulates each participant's `(norm_big_y, norm_big_c)` contribution by simple addition, with no zero-knowledge proof or commitment binding those values to the participant's actual key share. A single malicious participant can substitute arbitrary group elements, silently steering the coordinator to output a wrong confidential derived key.

### Finding Description
In `do_ckd_coordinator` the coordinator collects every participant's share and sums them unconditionally:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-56
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

An honest participant P_i is supposed to compute (in `compute_signature_share`):

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
``` [2](#0-1) 

The coordinator sums these to obtain `(Y, C)`, and the caller later recovers the confidential key via `C − app_sk · Y = msk · H(pk ‖ app_id)`. [3](#0-2) 

There is **no proof of correct formation** attached to any participant's message. A malicious participant P_m can send `(Y_m', C_m')` — two completely arbitrary `G1` points — instead of the correctly derived values. The coordinator then computes:

```
Y' = Y + (Y_m' − norm_big_y_m)
C' = C + (C_m' − norm_big_c_m)
```

Unmasking yields `C' − app_sk · Y' = msk · H(pk ‖ app_id) + δ` where `δ ≠ 0`, producing a wrong confidential key. No error is raised; the coordinator silently accepts the corrupted `CKDOutput`.

Because `do_ckd_participant` is a library function the caller drives, a malicious node can bypass it entirely and inject arbitrary bytes through the communication layer, or simply modify the values before they are sent.

### Impact Explanation
The coordinator accepts and returns a `CKDOutput` whose `unmask(app_sk)` call produces an incorrect group element `≠ msk · H(pk ‖ app_id)`. Every downstream consumer of the derived confidential key receives a wrong value. This is a direct, undetectable corruption of CKD output — matching the allowed **High** impact: *"Corruption of CKD outputs so honest parties accept unusable cryptographic outputs."*

### Likelihood Explanation
Any single participant who deviates from the protocol can trigger this. The attack requires no cryptographic capability: the adversary only needs to send two arbitrary `G1` points instead of the correct ones. The coordinator has no mechanism to identify or reject the malformed contribution. In a deployment where even one node is compromised (e.g., a TEE attestation failure), this attack is immediately reachable.

### Recommendation
Attach a non-interactive zero-knowledge proof to each participant's message that binds `(norm_big_y_i, norm_big_c_i)` to the participant's committed public share. Concretely, each participant should prove knowledge of `(x_i, y_i)` such that:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

with `x_i · G2 = public_share_i` (verifiable from the DKG output). A standard Sigma / Chaum-Pedersen proof over `G1` suffices. The coordinator must verify all proofs before accumulating any contribution.

### Proof of Concept
1. Three participants: P1 (coordinator), P2 (malicious), P3 (honest). All hold valid DKG key shares summing to `msk`.
2. P3 correctly computes and sends `(norm_big_y_3, norm_big_c_3)` to P1.
3. P2 sends `(G1::generator(), G1::generator())` — two generator points — instead of its correct share.
4. P1 accumulates: `Y' = norm_big_y_1 + G + norm_big_y_3`, `C' = norm_big_c_1 + G + norm_big_c_3`.
5. P1 returns `CKDOutput { big_y: Y', big_c: C' }` with no error.
6. Caller invokes `ckd_output.unmask(app_sk)` and receives `C' − app_sk · Y'`, which differs from `msk · H(pk ‖ app_id)` by the injected delta.
7. No protocol-level check detects the corruption; the wrong key is silently used.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
