### Title
COA Ownership Proof Signature Replay Across Multiple COAs Owned by the Same Cadence Account - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced by a Cadence account key over arbitrary data `D` to prove ownership of COA_A can be replayed by any unprivileged caller to assert ownership of COA_B — a different COA stored under the same Cadence account — because the cryptographic check and the EVM-address binding check are independent and the signed payload is not required to commit to the specific COA being proven.

---

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` is an `access(all)` function callable from any Cadence transaction or script. It accepts six parameters: `address` (Cadence account), `path` (capability path), `signedData`, `keyIndices`, `signatures`, and `evmAddress`. [1](#0-0) 

The function performs two independent checks:

1. It verifies that the supplied signatures over `signedData` are valid for the keys at `address`.
2. It borrows the COA capability from `path` on `address` and checks that the COA's EVM address matches `evmAddress`. [2](#0-1) 

Critically, there is **no enforcement that `signedData` commits to `evmAddress`**. The code itself acknowledges this gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [3](#0-2) 

Because the two checks are decoupled, an attacker who obtains a valid signature from Alice over any `signedData = D` can supply a different `path` and `evmAddress` pointing to a different COA owned by Alice, and the function will return `isValid: true`.

---

### Impact Explanation

The `validateCOAOwnershipProof` function is the trust anchor for COA ownership in the Flow EVM environment. It is invoked:

- Directly from any Cadence transaction or script (it is `access(all)`).
- Via the `verifyCOAOwnershipProof` Cadence Arch precompile, which is callable from any EVM transaction. [4](#0-3) 

The COA's ERC-1271 implementation (`isValidSignature`) delegates to this precompile: [5](#0-4) 

Any EVM contract that uses ERC-1271 for authorization (token approvals, DAO votes, permit-style flows) and accepts a COA as a signer is exposed. An attacker who intercepts or observes a valid COA ownership proof (e.g., from an off-chain authentication flow or a prior on-chain call) can replay it against a different COA owned by the same Cadence account to impersonate that COA in ERC-1271 checks, potentially authorizing token transfers, governance actions, or other signature-gated operations without the account holder's consent.

---

### Likelihood Explanation

- The function is `access(all)` — any unprivileged Cadence transaction or EVM transaction can invoke it with attacker-controlled arguments.
- The `COAOwnershipProof` struct (containing the Cadence address, path, key indices, and signatures) is RLP-encoded and passed as opaque `bytes` in EVM calls, making it trivially substitutable by an attacker. [6](#0-5) 

- The precondition — Alice having multiple COAs — is supported by the protocol and will become more common as the ecosystem matures.
- The precondition — attacker obtaining a valid signature — is realistic: signatures are produced for off-chain authentication, ERC-1271 checks in other contracts, or any other proof-of-ownership flow, and the signed payload `D` is not secret.

---

### Recommendation

Enforce that `signedData` commits to the specific `evmAddress` being proven. The simplest fix is to require that `signedData` is the ABI- or RLP-encoding of `evmAddress` (or a hash thereof), and to verify this binding inside `validateCOAOwnershipProof` before accepting the proof as valid. Alternatively, include `evmAddress` as a mandatory prefix or suffix of `signedData` and reject proofs where the binding is absent. This mirrors the SeaDrop recommendation of including a unique identifier (token ID / `minterNumMinted`) in the signed payload to prevent reuse across contexts.

---

### Proof of Concept

**Setup**: Alice's Cadence account (`0xAlice`) has two COAs:
- COA_A stored at `/public/coaA`, with EVM address `0xCOA_A`
- COA_B stored at `/public/coaB`, with EVM address `0xCOA_B`

**Step 1 — Legitimate proof generation**: Alice signs arbitrary data `D` (e.g., a challenge nonce) with her account key index 0 to prove ownership of COA_A for some off-chain service. The resulting `COAOwnershipProof` is:
```
{ Address: 0xAlice, CapabilityPath: "coaA", KeyIndices: [0], Signatures: [sig_D] }
```

**Step 2 — Replay attack**: An attacker constructs a new proof targeting COA_B:
```
{ Address: 0xAlice, CapabilityPath: "coaB", KeyIndices: [0], Signatures: [sig_D] }
```
and calls:
```cadence
EVM.validateCOAOwnershipProof(
    address: 0xAlice,
    path: /public/coaB,
    signedData: D,          // same data Alice signed for COA_A
    keyIndices: [0],
    signatures: [sig_D],    // Alice's signature, unchanged
    evmAddress: 0xCOA_B     // different COA
)
```

**Step 3 — Result**: The function verifies `sig_D` over `D` using Alice's key 0 (valid), then borrows COA_B from `/public/coaB` and confirms its address is `0xCOA_B` (valid). It returns `ValidationResult(isValid: true)` — even though Alice never authorized a proof for COA_B. [7](#0-6) 

**Step 4 — ERC-1271 exploitation**: The attacker encodes the replayed proof as `bytes` and calls `COA_B.isValidSignature(D, encoded_replayed_proof)`. The precompile passes `address(this) = 0xCOA_B` as `evmAddress`, which matches the replayed proof's target, so `isValidSignature` returns `ValidERC1271Signature` — granting the attacker the ability to impersonate COA_B as a signer in any ERC-1271-gated EVM contract. [8](#0-7)
