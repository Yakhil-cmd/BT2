### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Initialization — (`File: src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs`, the shared initialization guard for all FROST EdDSA and RedJubjub signing entry points, validates that `threshold ≤ participants.len()` but never checks that `threshold ≥ 2`. A caller can pass `threshold = 1` (or `threshold = 0`) to `sign_v1`, `sign_v2`, or `redjubjub::sign`. This causes each participant's `KeyPackage` to be constructed with `min_signers = 1`, allowing the FROST `aggregate` function to accept a single signature share as a complete threshold signature — effectively reducing the scheme to single-party signing.

---

### Finding Description

`assert_sign_inputs` is the sole validation gate called by every FROST signing entry point:

- `src/frost/eddsa/sign.rs` → `sign_v1` and `sign_v2`
- `src/frost/redjubjub/sign.rs` → `sign`

The function performs the following checks: [1](#0-0) 

It checks `threshold.value() > participants.len()` (upper bound) but has **no lower-bound check** (`threshold >= 2`). Contrast this with `assert_key_invariants` in `src/dkg.rs`, which explicitly enforces the minimum: [2](#0-1) 

And `validate_triple_inputs` in the OT-based ECDSA triple generation, which also enforces it: [3](#0-2) 

The signing path does not apply this same guard. When `threshold = 1` is passed, `construct_key_package` converts it directly to `min_signers = 1` in the FROST `KeyPackage`: [4](#0-3) 

With `min_signers = 1`, FROST's `round2::sign` computes a Lagrange-weighted signature share where the Lagrange coefficient for a single participant evaluates to 1 (the identity scalar), meaning the share equals the full secret contribution. FROST's `aggregate` then accepts this single share as a complete, verifiable threshold signature.

The same `construct_key_package` pattern exists in the RedJubjub path: [5](#0-4) 

The `presign` function in `src/frost/mod.rs` also lacks the lower-bound check on `args.threshold`: [6](#0-5) 

---

### Impact Explanation

**Impact: Critical — Unauthorized creation of a valid threshold signature.**

With `threshold = 1`, a single participant (e.g., the coordinator) can:

1. Call `sign_v1` or `sign_v2` with `threshold = 1` — accepted without error by `assert_sign_inputs`.
2. Receive their own `KeyPackage` with `min_signers = 1`.
3. Compute a signature share via `round2::sign`; with threshold-1 Lagrange interpolation, this share encodes the full secret contribution.
4. Pass this single share to `aggregate`, which verifies and returns a valid, externally-verifiable FROST signature.

This bypasses the threshold security property entirely: a key generated with `threshold = 5` can be signed against with `threshold = 1`, requiring only 1 of 5 participants.

---

### Likelihood Explanation

The `threshold` parameter is passed by the library caller at signing time and is not derived from or cross-checked against the `KeygenOutput`. Any caller — including a malicious coordinator orchestrating a signing session — can supply an arbitrary `threshold` value. The DKG output (`KeygenOutput`) stores only `private_share` and `public_key`, not the original threshold: [7](#0-6) 

There is no runtime binding between the threshold used at key generation and the threshold supplied at signing. A malicious coordinator controlling the signing invocation can trivially pass `threshold = 1`.

---

### Recommendation

Add a lower-bound check in `assert_sign_inputs` mirroring the check already present in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same fix to the `presign` function in `src/frost/mod.rs` for the `args.threshold` field.

---

### Proof of Concept

1. Run DKG with 5 participants and `threshold = 3` using `keygen` — this succeeds because `assert_key_invariants` enforces `threshold >= 2`.
2. Call `sign_v1` with the same 5 participants but `threshold = 1`. `assert_sign_inputs` accepts this without error (no lower-bound check).
3. Each participant constructs a `KeyPackage` with `min_signers = 1` via `construct_key_package`.
4. The coordinator computes its own `round2::sign` share. With `min_signers = 1`, the Lagrange coefficient is 1, so the share equals the full secret contribution.
5. The coordinator calls `aggregate` with only its own share. FROST's `aggregate` verifies the resulting signature against the public key and returns `Ok(Some(signature))`.
6. The returned signature is a valid, externally-verifiable EdDSA signature produced by a single participant, bypassing the 3-of-5 threshold requirement.
