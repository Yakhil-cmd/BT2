### Title
Hardcoded `rejectUnauthorized: false` Disables TLS Server Verification on Daemon WebSocket — Enables RPC Spoofing Leading to Wrong Transaction Approval - (File: packages/api/src/Client.ts)

---

### Summary
The Chia GUI's WebSocket connections to the Chia daemon are established with `rejectUnauthorized: false` hardcoded in three separate production files. This disables server certificate verification, allowing a network-positioned attacker to impersonate the daemon, inject spoofed RPC responses, and cause the user to approve, send, or sign transactions for the wrong asset, amount, or destination.

---

### Finding Description
In all three production WebSocket connection paths, the TLS option `rejectUnauthorized: false` is hardcoded unconditionally:

**`packages/api/src/Client.ts` (line 141–145):**
```ts
const ws = new WebSocket(url, {
  key,
  cert,
  rejectUnauthorized: false,
});
``` [1](#0-0) 

**`packages/gui/src/electron/api/sendCommand.ts` (line 22–26):**
```ts
const socket = new WebSocket(url, {
  key,
  cert,
  rejectUnauthorized: false,
});
``` [2](#0-1) 

**`packages/gui/src/electron/utils/webSocketBridge.ts` (line 61–65):**
```ts
const socket = new WebSocket(url, {
  key,
  cert,
  rejectUnauthorized: false,
});
``` [3](#0-2) 

The daemon URL is constructed from the config file field `ui.daemon_host`, which defaults to `'localhost'` but is user-configurable to any hostname or IP address:

```ts
const selfHostname = get(config, 'ui.daemon_host', 'localhost');
const url = `wss://${selfHostname}:${daemonPort}`;
``` [4](#0-3) 

When `daemon_host` is set to a remote hostname (a supported and documented Chia configuration for users running a separate farming node), a network-positioned attacker can intercept the WSS connection and present any TLS certificate — which is unconditionally accepted because `rejectUnauthorized: false` is hardcoded. No `ca` option is passed to constrain which certificate authorities are trusted.

---

### Impact Explanation
The daemon is the single authority for all wallet operations: sending XCH/CAT transactions, signing, key management, offer acceptance, and clawback. All three affected code paths carry live wallet RPC traffic. A successful MITM allows the attacker to:

- Return spoofed RPC responses (e.g., fake balances, fake transaction confirmations, fake offer states) causing the user to approve a transaction to the wrong destination or for the wrong amount.
- Intercept outbound wallet commands (e.g., `send_transaction`, `create_offer_for_ids`) and observe or suppress them.
- Inject fabricated daemon events that trigger UI state changes (e.g., fake sync completion, fake offer acceptance).

This maps directly to the **High** allowed impact: *"Corruption, spoofing, or unsafe trust of RPC … state that causes a user to approve, import, sign, send, revoke, burn, join, or display the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation
The attack is reachable without any privileged access when `daemon_host` is configured to a non-loopback address — a common setup for users running a dedicated farming machine. In that scenario, any attacker on the same LAN or controlling a network hop between the GUI host and the daemon host can perform the MITM. Even in the default localhost configuration, if `daemon_host` is set to a domain name rather than `127.0.0.1`, DNS rebinding is a viable vector. The vulnerability is hardcoded (not conditional), so it is always active regardless of environment.

---

### Recommendation
- **Short term:** Remove `rejectUnauthorized: false` from all three WebSocket connection sites. Pass the daemon's CA certificate (available in the Chia SSL directory alongside the client cert/key) as the `ca` option so the client verifies the server's certificate against the known Chia CA.
- **Long term:** Add a CI lint rule (e.g., via ESLint or Semgrep with a `tls-rejectunauthorized-false` rule) to prevent `rejectUnauthorized: false` from being reintroduced in any WebSocket or HTTPS connection.

---

### Proof of Concept
1. Configure `~/.chia/mainnet/config/config.yaml` to set `ui.daemon_host` to a hostname (e.g., `chia-daemon.local`).
2. On the network path between the GUI host and the daemon host, run a TLS-intercepting proxy (e.g., `mitmproxy`) presenting a self-signed certificate for `chia-daemon.local`.
3. Launch the Chia GUI. The WebSocket connection in `Client.ts` / `webSocketBridge.ts` / `sendCommand.ts` completes successfully despite the invalid certificate because `rejectUnauthorized: false` suppresses all certificate validation.
4. The proxy now has full read/write access to the daemon RPC channel. Inject a spoofed `send_transaction` response confirming a transaction to an attacker-controlled address, or return a fake balance to manipulate the user's next send action.
