### Title
Unbounded `pool_member_epoch_balance` Trace Causes Unbounded Gas Consumption in `calculate_rewards`, Permanently Freezing Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in `pool.cairo` iterates over every entry in a delegator's `pool_member_epoch_balance` trace without any cap. Because each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends a new checkpoint to this trace, a delegator who makes balance changes across many epochs without claiming rewards will eventually cause `claim_rewards` (and `pool_member_info_v1`) to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

### Finding Description
`calculate_rewards` in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
``` [1](#0-0) 

The loop iterates from `entry_to_claim_from` (the index saved at the last `claim_rewards` call) up to the current trace length. Each iteration reads a storage slot and calls `find_sigma` (a binary search on `cumulative_rewards_trace`), making each iteration non-trivial in gas cost.

The trace grows via `set_member_balance` → `trace.insert`. The `insert` function only appends a **new** checkpoint when the epoch key differs from the last entry's key: [2](#0-1) 

This means every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a **different epoch** appends one new entry. The trace is never pruned or capped.

`add_to_delegation_pool` is callable by the pool member or their reward address with any non-zero amount: [3](#0-2) 

`exit_delegation_pool_intent` similarly calls `set_member_balance` on every invocation: [4](#0-3) 

`entry_to_claim_from` is only advanced when `claim_rewards` succeeds: [5](#0-4) 

If the delegator never calls `claim_rewards` (or it always reverts once the limit is hit), `entry_to_claim_from` stays at 0 and every subsequent call must iterate the entire trace from the beginning.

### Impact Explanation
Once the trace is large enough that a single `claim_rewards` call exceeds the Starknet transaction gas limit, the call always reverts. Because `entry_to_claim_from` is only updated on success, the delegator's unclaimed yield is **permanently frozen** — it can never be extracted. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation
- `add_to_delegation_pool` requires only a non-zero token amount per call; a delegator can call it once per epoch with a dust amount.
- Starknet epochs are on the order of days to weeks. Over months to years of active delegation with frequent balance changes and infrequent reward claims, the trace grows to hundreds or thousands of entries.
- A delegator running an automated strategy (e.g., a smart contract that rebalances delegation each epoch) could reach the gas limit within a year.
- The developers themselves acknowledge the risk in the comment at line 858, indicating awareness but no mitigation.

Likelihood: **Medium** (requires sustained activity over many epochs, but is realistic for long-lived automated delegators).

### Recommendation
1. **Enforce a maximum trace length**: Reject `add_to_delegation_pool` and `exit_delegation_pool_intent` if the trace length for the caller exceeds a safe cap (e.g., 500 entries).
2. **Partial claim with pagination**: Allow `claim_rewards` to accept an optional `max_entries` parameter so a delegator can drain their rewards in multiple transactions.
3. **Automatic checkpoint on balance change**: When `set_member_balance` is called, automatically trigger a partial reward settlement up to the current epoch, resetting `entry_to_claim_from` and preventing unbounded accumulation.

### Proof of Concept
1. Delegator calls `enter_delegation_pool` with `amount = MIN_DELEGATION`.
2. For each of N epochs, delegator calls `add_to_delegation_pool(pool_member, amount: 1)`. Each call in a new epoch appends one entry to `pool_member_epoch_balance` (confirmed by `trace.insert` logic at line 166–172 of `pool_member_balance_trace/trace.cairo`).
3. After N epochs without calling `claim_rewards`, `pool_member_info.entry_to_claim_from` remains 0 and `pool_member_epoch_balance` has N entries.
4. Delegator calls `claim_rewards`. `calculate_rewards` enters the loop at line 859 and iterates N times, each time reading storage and calling `find_sigma`.
5. For sufficiently large N (determined by Starknet's per-transaction gas cap divided by per-iteration gas cost), the transaction reverts with out-of-gas.
6. Every subsequent `claim_rewards` call also reverts because `entry_to_claim_from` was never updated. The delegator's accumulated yield is permanently inaccessible.
