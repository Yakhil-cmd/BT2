### Title
Zero `priceProviderTimelock` Allows Pool Admin to Atomically Swap In Malicious Price Provider and Drain LP Funds — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`createPool` accepts `params.priceProviderTimelock = 0` without enforcing any minimum. When the timelock is zero, `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` can be called in the same transaction, giving LPs zero time to react and withdraw before a manipulated oracle price is activated.

---

### Finding Description

`createPool` stores `params.priceProviderTimelock` directly into `priceProviderTimelock[pool]` with no lower-bound check: [1](#0-0) [2](#0-1) 

`_validatePoolParameters` validates fees, tokens, admin, and bin geometry but contains no check on `priceProviderTimelock`: [3](#0-2) 

In `proposePoolPriceProvider`, `executeAfter` is computed as:

```
uint256 executeAfter = block.timestamp + timelock;   // = block.timestamp when timelock == 0
``` [4](#0-3) 

In `executePoolPriceProviderUpdate`, the guard is:

```
if (block.timestamp < execAfter) revert ...;
// With timelock == 0: block.timestamp < block.timestamp → always false → passes immediately
``` [5](#0-4) 

The only validation on the new provider is a token-pair match check, which a malicious provider trivially satisfies while returning any price: [6](#0-5) 

---

### Impact Explanation

A pool admin who deployed with `priceProviderTimelock = 0` can, in a single atomic transaction after LPs have deposited:

1. Call `proposePoolPriceProvider` with a malicious oracle that returns an extreme bid/ask.
2. Immediately call `executePoolPriceProviderUpdate` (timelock check passes because `block.timestamp < block.timestamp` is false).
3. Call `swap` on the pool — `_getBidAndAskPriceX64` now reads the malicious oracle price.
4. Receive LP-owned tokens at a price far from fair value, draining LP principal.

The pool's swap path reads the active price provider on every call with no staleness or sanity check beyond `bid < ask` and `bid != 0`: [7](#0-6) 

This is a direct loss of LP principal — the exact impact class listed in the allowed gate.

---

### Likelihood Explanation

Any pool creator can set `priceProviderTimelock = 0`. The attack requires no external resources beyond deploying a malicious `IPriceProvider` that returns the correct token pair. The entire attack is executable in one transaction with no observable warning to LPs (the `PoolPriceProviderChangeProposed` event and the `PoolPriceProviderUpdated` event are emitted in the same block, leaving no reaction window).

---

### Recommendation

Enforce a protocol-level minimum timelock in `_validatePoolParameters`:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 2 days; // example

// In _validatePoolParameters:
if (params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert TimelockTooShort();
}
```

This ensures LPs always have a meaningful window to observe a pending provider change and exit before it takes effect.

---

### Proof of Concept

```solidity
// Attacker-controlled malicious price provider
contract MaliciousProvider {
    address public token0;
    address public token1;
    constructor(address t0, address t1) { token0 = t0; token1 = t1; }
    // Returns extreme price: 1 token0 = 1e18 token1
    function getBidAndAskPrice() external pure returns (uint128, uint128) {
        return (uint128(1e18 * 2**64), uint128(1e18 * 2**64 + 1));
    }
}

// Step 1: Admin creates pool with priceProviderTimelock = 0
PoolParameters memory params = ...;
params.priceProviderTimelock = 0;  // no minimum enforced
address pool = factory.createPool(params);

// Step 2: LPs deposit liquidity (pool appears legitimate)
pool.addLiquidity(...);

// Step 3: Admin atomically drains LPs in one transaction
MaliciousProvider evil = new MaliciousProvider(token0, token1);
factory.proposePoolPriceProvider(pool, address(evil));
// executeAfter = block.timestamp + 0 = block.timestamp → check passes immediately
factory.executePoolPriceProviderUpdate(pool);
// Pool now prices token1 at 1e18x fair value
pool.swap(attacker, false, largeAmount, 0, ...);
// Attacker receives token0 at near-zero cost, draining LP token0 balances
```
