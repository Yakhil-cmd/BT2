### Title
EVM-to-Cadence Withdrawal Granularity Mismatch Permanently Traps Sub-UFix64 attoFlow Balances - (`fvm/evm/types/balance.go`, `fvm/evm/emulator/emulator.go`, `fvm/evm/impl/impl.go`)

---

### Summary

EVM accounts on Flow can accumulate balances at attoflow (1e-18 FLOW) granularity through EVM-level operations, but the only mechanism to recover those funds to Cadence — `CadenceOwnedAccount.withdraw` — enforces UFix64 (1e-8 FLOW = 1e10 attoflow) granularity. Any sub-1e10 attoflow remainder in an EVM account balance is permanently irrecoverable to Cadence, constituting a cross-VM asset loss.

---

### Finding Description

**Two different granularity constraints govern the same asset:**

**EVM-level operations** (EVM transactions, Solidity contract calls, direct calls) operate at attoflow (1e-18 FLOW) precision. Any amount of attoflow can be transferred between EVM addresses. This is the "lotSize" analog.

**Cadence withdrawal** (`CadenceOwnedAccount.withdraw`) enforces UFix64 precision (1e-8 FLOW = 1e10 attoflow minimum). This is the "tradingLotSize" analog.

The withdrawal check in `fvm/evm/types/balance.go` is:

```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
``` [1](#0-0) 

Where `UFixToAttoConversionMultiplier = 10^10`. This check only requires `bal >= 1e10`; it does **not** require `bal % 1e10 == 0`.

The actual withdrawal in `fvm/evm/emulator/emulator.go` (`withdrawFrom`) enforces this check:

```go
if !types.AttoFlowBalanceIsValidForFlowVault(call.Value) {
    return types.NewInvalidResult(
        call.Type,
        call.Hash(),
        types.ErrWithdrawBalanceRounding,
    ), nil
}
``` [2](#0-1) 

And in `fvm/evm/impl/impl.go` (`newInternalEVMTypeWithdrawFunction`), the division truncates silently:

```go
value := new(big.Int).Div(amountValue.BigInt, types.UFixToAttoConversionMultiplier)
amount := types.NewBalanceFromUFix64(cadence.UFix64(value.Uint64()))
``` [3](#0-2) 

The `deposit` path in `fvm/evm/stdlib/contract.cdc` (`EVMAddress.deposit`) applies **no** granularity check — it accepts any `FlowToken.Vault` amount and converts it to attoflow: [4](#0-3) 

And `newInternalEVMTypeDepositFunction` in `fvm/evm/impl/impl.go` similarly applies no minimum-granularity check: [5](#0-4) 

**The mismatch scenario:**

1. An EVM account (COA or EOA) receives `X` attoflow where `X % 1e10 != 0` via an EVM-level transfer (e.g., a Solidity contract sends 5 attoflow, or gas refunds leave a sub-1e10 remainder).
2. The user can withdraw `floor(X / 1e10) * 1e10` attoflow to Cadence.
3. The remainder `X % 1e10` (up to 9,999,999,999 attoflow) is permanently trapped in the EVM environment. It can be moved between EVM addresses but **can never be withdrawn to Cadence** because any withdrawal attempt of an amount `< 1e10` fails with `ErrWithdrawBalanceRounding`. [6](#0-5) 

The `CadenceOwnedAccount.withdraw` documentation in `contract.cdc` acknowledges this limitation: [7](#0-6) 

---

### Impact Explanation

Any sub-1e10 attoflow remainder (up to 9,999,999,999 attoflow ≈ 0.00000001 FLOW) in an EVM account is permanently irrecoverable to Cadence. While individually small, this constitutes a **cross-VM asset loss**: funds that exist in the EVM state cannot be bridged back to Cadence. The funds are not burned — they remain in the EVM state indefinitely — but no Cadence-level mechanism can retrieve them. This is the direct analog to the Mai Protocol issue where a liquidated position's remainder below `tradingLotSize` could never be closed.

---

### Likelihood Explanation

**Medium-Low.** Any EVM-level operation that transfers a non-UFix64-aligned amount of attoflow to a COA or EOA creates this condition. This includes:
- Solidity contracts sending arbitrary attoflow amounts (e.g., `payable` functions, `transfer`, `send`)
- EVM gas refunds that produce sub-1e10 attoflow remainders
- Any EVM transaction where the value field is not a multiple of 1e10

This is reachable by any unprivileged EVM transaction sender without special permissions.

---

### Recommendation

Either:
1. Restrict EVM-level FLOW transfers to multiples of 1e10 attoflow (the UFix64 minimum unit) at the protocol level, or
2. Provide a mechanism for EVM accounts to recover sub-1e10 attoflow remainders (e.g., accumulate them in a protocol-level dust collector that periodically burns or redistributes them), or
3. Change `AttoFlowBalanceIsValidForFlowVault` to also check divisibility (`bal % 1e10 == 0`) and document that EVM-level transfers must be UFix64-aligned to be fully recoverable.

---

### Proof of Concept

1. Deploy a Solidity contract on Flow EVM that sends 5 attoflow to a COA address.
2. The COA's EVM balance is now 5 attoflow.
3. Attempt `cadenceOwnedAccount.withdraw(balance: EVM.Balance(attoflow: 5))` from Cadence.
4. The transaction panics with `ErrWithdrawBalanceRounding` ("withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow").
5. The 5 attoflow is permanently trapped — no Cadence transaction can recover it.

This is confirmed by the existing test in `fvm/evm/emulator/emulator_test.go`: [8](#0-7) 

And by `fvm/evm/handler/handler_test.go`: [9](#0-8)

func newInternalEVMTypeDepositFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeDepositFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			// Get from vault

			fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amountValue, ok := fromValue.GetField(
				context,
				fungibleTokenVaultTypeBalanceFieldName,
			).(interpreter.UFix64Value)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))

			// Get to address

			toAddressValue, ok := invocation.Arguments[1].(*interpreter.ArrayValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			toAddress, err := AddressBytesArrayValueToEVMAddress(context, toAddressValue)
			if err != nil {
				panic(err)
			}

			// NOTE: We're intentionally not destroying the vault here,
			// because the value of it is supposed to be "kept alive".
			// Destroying would incorrectly be equivalent to a burn and decrease the total supply,
			// and a withdrawal would then have to perform an actual mint of new tokens.

			// Deposit

			const isAuthorized = false
			account := handler.AccountByAddress(toAddress, isAuthorized)
			account.Deposit(types.NewFlowTokenVault(amount))

			return interpreter.Void
		},
	)
