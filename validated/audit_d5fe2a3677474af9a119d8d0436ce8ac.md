## Exact Semantic Proof

`StateNumber` is defined as:

    StateNumber(block_number)   // StateNumber(N) IS right_before_block(N)

`StateNumber(N)` is literally `right_before_block(N)` — it represents the state **before** block N executes, i.e., it sees only blocks 0..N-1.

`get_deprecated_class_definition_at` filters with:

```rust
if state_number.is_before(value.block_number) {
    return Ok(None);

``` [3](#0-2) 

For a class declared at `BlockNumber(N)` with `latest_block = N`:

```

= StateNumber(N).is_before(BlockNumber(N))
= N <= N
= true  →  returns None

The class is invisible. The correct call would be `StateNumber::unchecked_right_after_block(self.latest_block)` = `StateNumber(N+1)`, giving `N+1 <= N` = false → class found.

## Verdict

### Title
Cairo-0 class declared at `latest_block` returns `UndeclaredClassHash` due to off-by-one in `StateNumber` — (`crates/apollo_state_reader/src/apollo_state.rs`)

### Summary
`read_deprecated_casm` constructs `StateNumber(self.latest_block)`, which is semantically `right_before_block(latest_block)` and excludes the state changes of `latest_block` itself. Any Cairo-0 class declared in the last finalized block is invisible to the pending block's execution, causing `get_compiled_class_from_db` to return `StateError::UndeclaredClassHash` for a class that is actually declared.

### Finding Description
In `ApolloReader::read_deprecated_casm` (line 166), the state number is constructed as `StateNumber(self.latest_block)`. By the definition of `StateNumber`, this equals `right_before_block(latest_block)` — the state **before** block N executes. The filter in `get_deprecated_class_definition_at` uses `state_number.is_before(value.block_number)` which evaluates `N <= N = true` and returns `None`. The correct construction is `StateNumber::unchecked_right_after_block(self.latest_block)` = `StateNumber(N+1)`, which gives `N+1 <= N = false` and returns the class. The Cairo 1 path (`is_declared`) uses `block_number <= state_number.0` = `N <= N = true` and correctly finds the class, creating an asymmetry.

### Impact Explanation
When `class_reader = None` (the `native_blockifier` / `ApolloReader::new` path), any transaction in the pending block that invokes a Cairo-0 class declared in the immediately preceding finalized block will receive `StateError::UndeclaredClassHash` from the blockifier. The transaction is reverted with wrong execution semantics — the wrong compiled class (None) is selected for a class that is actually declared and should be executable.

### Likelihood Explanation
The scenario requires a Cairo-0 class to be declared in block N and used in block N+1. Cairo-0 declarations are still valid on Starknet. The off-by-one is deterministic and always triggers for this one-block window. It only affects the `class_reader = None` code path; if the production sequencer always supplies a `ClassReader`, the bug is dormant.

### Recommendation
Change line 166 in `crates/apollo_state_reader/src/apollo_state.rs` from:
```rust
let state_number = StateNumber(self.latest_block);
```
to:
```rust
let state_number = StateNumber::unchecked_right_after_block(self.latest_block);
```
This matches the semantics used everywhere else in the codebase (RPC, state sync, re-execution) when the intent is "state visible after block N has been applied."

### Proof of Concept
The existing test at `crates/apollo_storage/src/state/state_test.rs:73` already proves the boundary: `get_deprecated_class_definition_at(StateNumber(0), dc0_declared_at_block_0)` returns `None`. A unit test for `ApolloReader` would:
