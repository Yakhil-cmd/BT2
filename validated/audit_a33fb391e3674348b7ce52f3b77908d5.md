### Title
Malicious Participant Can Corrupt CKD Output Without Cryptographic Verification — (`src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function blindly sums `(norm_big_y, norm_big_c)` shares received from participants with no cryptographic proof of correctness. A single malicious participant can supply an arbitrary or wrong private share, causing the coordinator to accept and output a corrupted confidential derived key. No detection or attribution mechanism exists.

### Finding Description

In `do_ckd_coordinator` (lines 35–58 of `src/confidential_key_derivation/protocol.rs`), the coordinator receives each participant's `(norm_big_y, norm_big_c)` pair and accumulates them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

The expected invariant is that participant `i` sends:

- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

so that summing over all participants yields `big_C − app_sk · big_Y = msk · H(pk ‖ app_id)`.

The `compute_signature_share` function that produces these values performs no self-consistency check and accepts whatever `private_share` is in the caller-supplied `KeygenOutput`:

```rust
let private_share = Zeroizing::new(key_pair.private_share);
// ...
let big_s = hash_point * private_share.to_scalar();
let big_c = big_s + app_pk * y.0;
``` [2](#0-1) 

Because `KeygenOutput` is caller-supplied and the coordinator never verifies that a participant's contribution is consistent with the agreed public key, a malicious participant can:

1. Pass a fabricated `private_share` (e.g., a random scalar) to `ckd(...)`.
2. The library faithfully computes and sends a wrong `(norm_big_y, norm_big_c)` to the coordinator.
3. The coordinator sums all contributions, producing an incorrect `CKDOutput`.
4. No error is raised; the coordinator returns `Ok(Some(corrupted_ckd_output))`.

There is no zero-knowledge proof, commitment, or consistency check anywhere in the protocol to bind a participant's contribution to their public key share. [3](#0-2) 

The only test for the CKD protocol (`test_ckd`) exercises the honest-participant happy path exclusively and does not cover any malicious-participant scenario: [4](#0-3) 

This mirrors the external report's finding: critical protocol code has only happy-path tests, leaving the malicious-participant branch entirely unexercised and the missing validation undetected.

### Impact Explanation

A single malicious participant causes the coordinator to accept and output a corrupted `CKDOutput`. The confidential derived key `msk · H(pk ‖ app_id)` is silently wrong. Any downstream consumer (e.g., a TEE application) that relies on this key will operate on incorrect secret material. This maps directly to:

> **High: Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

Because the coordinator is the sole recipient of the final output and has no way to re-derive or cross-check the correct value, the corruption is undetectable and unrecoverable without re-running the protocol with a different participant set.

### Likelihood Explanation

The attack requires only that one participant in the CKD session supplies a wrong `private_share` to the `ckd()` API — a capability available to any library caller with a valid participant identity. No cryptographic break, key leakage, or privileged access is needed. The `ckd` function is a public API with no enforcement of TEE attestation or share correctness at the library level. [5](#0-4) 

### Recommendation

Add a zero-knowledge proof of correct share contribution. Each participant should prove, alongside `(norm_big_y, norm_big_c)`, that their contribution is consistent with the public verification key share derived from the DKG output. Concretely, a DLog-equality proof (already implemented in `src/crypto/proofs/dlogeq.rs`) can bind `norm_big_c` to the participant's known public share commitment, allowing the coordinator to reject and attribute any malformed contribution before accumulating it. [6](#0-5) 

### Proof of Concept

```
Setup:
  participants = [P0 (coordinator), P1 (malicious), P2 (honest)]
  Correct msk, app_id, app_pk established via DKG.

Attack:
  P1 calls ckd(...) with KeygenOutput {
      public_key: correct_pk,          // matches DKG output
      private_share: random_scalar,    // wrong share
  }

  P1's compute_signature_share computes:
      big_s  = H(pk, app_id) * random_scalar   // wrong
      big_c  = big_s + app_pk * y              // wrong
      norm_big_c = lambda_1 * big_c            // wrong

  Coordinator accumulates:
      total_big_c = lambda_0*correct_c0 + lambda_1*wrong_c1 + lambda_2*correct_c2

  Coordinator outputs CKDOutput with corrupted big_C.
  unmask(app_sk) yields:  big_C - app_sk*big_Y  ≠  msk * H(pk, app_id)

  No error is raised. The coordinator silently accepts the wrong key.
```

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    Ok(e == proof.e.0)
}
