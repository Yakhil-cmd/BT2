## Code Path Trace

**`skip_stateful_validations`** fires when:
1. The tx is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

**`account_tx_in_pool_or_recent_block`** returns `true` if the address has **any** transaction in the pool or any committed nonce in the mempool state — it does **not** check whether that transaction is a `DeployAccount`: [2](#0-1) 

The comment in the code claims this is safe because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This assumption is **incorrect** for deployed accounts with on-chain nonce=0. [3](#0-2) 

**`validate_nonce`** for an Invoke with `tx_nonce=1`, `account_nonce=0` passes as long as `max_allowed_nonce_gap >= 1`: [4](#0-3) 

When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`, and the blockifier's `perform_validations` returns `Ok(())` immediately without calling `__validate__`: [5](#0-4) [6](#0-5)

## Concrete Attack Scenario

The assumption that `account_nonce == 0` implies "account not yet deployed" is wrong. A **deployed** account that has never sent a transaction also has `account_nonce=0`.

**Step 1.** Victim has a deployed account at address `A` with on-chain nonce=0. Victim submits `Invoke(nonce=0, valid_sig)`. This passes all gateway checks (nonce=0 matches account_nonce=0, `skip_stateful_validations` does not fire because tx_nonce≠1, `__validate__` runs and succeeds). Address `A` is now in the mempool pool.

**Step 2.** Attacker submits `Invoke(sender=A, nonce=1, INVALID_sig)`:
- `validate_nonce`: `0 <= 1 <= max_gap` → passes
- `validate_by_mempool`: no duplicate, `tx_nonce=1 >= account_nonce=0` → passes
- `skip_stateful_validations`: tx_nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)` = `true` (because victim's nonce=0 Invoke is in pool) → returns `true`
- `run_validate_entry_point`: `skip_validate=true` → `__validate__` **not called**
- Transaction admitted to mempool with invalid signature

**Step 3.** The attacker's invalid-signature nonce=1 transaction now occupies the `(A, nonce=1)` slot in the mempool. When the victim tries to submit their legitimate nonce=1 transaction, the mempool rejects it as `DuplicateNonce` (or requires fee escalation to replace it). [7](#0-6)

## Why the Batcher Does Not Fully Mitigate This

When the batcher eventually executes the attacker's transaction, `__validate__` is called and fails, so the transaction is not included in a block. However:
- The invalid transaction **was admitted to the mempool**, consuming the `(A, nonce=1)` slot
- The victim's legitimate nonce=1 transaction is blocked until the attacker's transaction is evicted (TTL expiry or fee escalation)
- This is a targeted, low-cost griefing attack against any deployed account with nonce=0 that has a pending transaction

### Title
`skip_stateful_validations` bypasses `__validate__` for deployed accounts with nonce=0 when any (non-DeployAccount) transaction exists in the mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
`skip_stateful_validations` uses `account_tx_in_pool_or_recent_block` as a proxy for "a DeployAccount transaction exists for this undeployed account." The function returns `true` for **any** transaction type in the pool. A deployed account with on-chain nonce=0 that has a pending Invoke(nonce=0) in the mempool satisfies all skip conditions, allowing an attacker to inject an Invoke(nonce=1) with an invalid signature that bypasses gateway `__validate__` and is admitted to the mempool.

### Finding Description
In `skip_stateful_validations`, the guard condition is:
```
tx is Invoke AND tx.nonce == 1 AND account_nonce == 0 AND account_tx_in_pool_or_recent_block == true
```
The intent is to detect the "deploy_account + invoke" UX pattern where the account is not yet deployed. However, `account_nonce == 0` does not imply "undeployed" — a deployed account that has never sent a transaction also has nonce=0. For such an account, a valid Invoke(nonce=0) can be admitted to the mempool (passing `__validate__`), which causes `account_tx_in_pool_or_recent_block` to return `true`. A subsequent Invoke(nonce=1) with an arbitrary (invalid) signature then satisfies all four conditions and has its `__validate__` skipped at the gateway.

### Impact Explanation
An attacker can inject an invalid-signature Invoke(nonce=1) into the mempool for any deployed account with on-chain nonce=0 that has a pending transaction. This occupies the `(address, nonce=1)` mempool slot, blocking the legitimate account owner's nonce=1 transaction (rejected as `DuplicateNonce` or requiring fee escalation to displace the attacker's entry). The gateway's signature verification — the primary admission control — is bypassed.

### Likelihood Explanation
Any deployed account that has just been set up (nonce=0) and submits its first transaction is vulnerable for the duration that transaction remains in the mempool. An attacker monitoring the mempool can trigger this for any such account. No privileged access is required.

### Recommendation
Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction exists for the address in the mempool. Alternatively, add a check that the account has no deployed class hash at the sender address (i.e., `get_class_hash_at(sender) == ClassHash::default()`), which is the true invariant the UX feature requires.

### Proof of Concept
1. Deploy account at address `A` (on-chain nonce becomes 0, class hash set).
2. Submit `Invoke(sender=A, nonce=0, valid_sig)` → admitted to mempool (passes `__validate__`).
3. Submit `Invoke(sender=A, nonce=1, sig=[0xff; 32])` (garbage signature).
4. In `skip_stateful_validations`: `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
5. `run_validate_entry_point` sets `execution_flags.validate=false` → `__validate__` not called.
6. Transaction admitted to mempool. Victim's legitimate nonce=1 transaction is now rejected as `DuplicateNonce`. [8](#0-7) [2](#0-1) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
