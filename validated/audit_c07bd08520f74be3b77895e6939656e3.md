### Title
Per-Swap Reference Price Update in `PriceVelocityGuardExtension.beforeSwap` Allows Cumulative Intra-Block Velocity Bypass — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` is designed to cap how fast the oracle mid-price can move **per block**. However, because `lastMidPriceX64` is overwritten with the current oracle price **before** the velocity check is evaluated, the guard is effectively per-swap, not per-block. An attacker who can push multiple signed Pyth Lazer payloads (each showing a small, individually-compliant price increment) and interleave them with pool swaps within a single transaction can move the oracle price by an arbitrarily large cumulative amount — far beyond the configured `maxChangePerBlockE18` — without ever triggering `PriceVelocityExceeded`.

---

### Finding Description

In `PriceVelocityGuardExtension.beforeSwap`, the storage state is updated unconditionally before the bound is checked:

```solidity
// Lines 57-58: reference price is overwritten FIRST
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

// Lines 60-75: check runs against the ALREADY-UPDATED reference
if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;
        uint256 delta = midPrice > prevMid ? ...;
        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
        if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
    }
}
``` [1](#0-0) 

After the first swap in a block, `lastUpdateBlock` equals `block.number`. Every subsequent swap in the same block therefore computes `blockDiff = 0`, so `allowedSq = maxChange² × 1`. The check becomes a pure per-swap bound, not a cumulative per-block bound. The reference price `prevMid` for each successive swap is the price from the immediately preceding swap, not the price at the start of the block.

The `PythOracle.fallback()` is a permissionless public function — anyone can push a Lazer-signed payload to update the oracle:

```solidity
fallback() payable external override {
    ...
    _verifyAndStore(oracleData, updateFeedIds, priceUpdate);
}
``` [2](#0-1) 

The only constraint is that each payload must carry a `FeedUpdateTimestamp` strictly newer than the stored one (monotonicity). Pyth Lazer publishes high-frequency signed updates; an attacker can collect N consecutive signed payloads from the public feed, each showing a small price increment, and replay them in a single transaction.

`MetricOmmPool.swap()` carries a `nonReentrant` guard that uses transient storage — it prevents reentrancy within one call but does **not** prevent multiple sequential `swap` calls in the same transaction:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
    ...
    (uint128 bidPriceX64, uint128 askPriceX64) = _getBidAndAskPriceX64();
    _beforeSwap(..., bidPriceX64, askPriceX64, extensionData);
    ...
}
``` [3](#0-2) 

---

### Impact Explanation

The `PriceVelocityGuardExtension` is the only on-chain mechanism that limits how fast the oracle price can reach the pool's swap math within a single block. Its purpose is to give LPs a reaction window — if the oracle price jumps sharply, the guard blocks swaps until enough blocks have elapsed, allowing LPs to remove liquidity before being adversely selected.

When bypassed, an attacker can execute a sequence of swaps at a cumulatively far-moved price within a single block. LPs have no on-chain recourse: the pool settles each swap at the oracle price, and the velocity guard — the only protection — has already been stepped past. The result is direct loss of LP principal through adverse selection at a price that the guard was explicitly configured to block.

This maps to the allowed impact gate: **bad-price execution** (unclamped oracle price reaches pool swap) and **direct loss of LP assets**.

---

### Likelihood Explanation

- The `PythOracle.fallback()` is permissionless; no special role is required to push oracle updates.
- Pyth Lazer publishes signed price updates at sub-second frequency. During any period of meaningful price movement, an attacker can collect N consecutive signed payloads, each within `maxChangePerBlockE18` of the previous, that together span a large total move.
- The attack is executable in a single atomic transaction with no privileged setup.
- The only prerequisite is that the pool uses `PriceVelocityGuardExtension` with a non-zero `maxChangePerBlockE18` and a Pyth-backed price provider — the exact configuration the extension is designed for.

Likelihood: **Medium** (requires active market volatility to collect sufficient incremental payloads, but no privileged access).

---

### Recommendation

Track the **block-start reference price** separately from the per-swap update. On the first swap of a new block, snapshot the current oracle price as the block anchor. On all subsequent swaps within the same block, compare against that anchor — not against the previous swap's price.

Concretely, add a `blockAnchorMidPriceX64` field to `PriceVelocityState`. When `block.number > lastUpdateBlock`, set `blockAnchorMidPriceX64 = prevMid` (the price from the previous block) and update `lastUpdateBlock`. When `block.number == lastUpdateBlock`, keep `blockAnchorMidPriceX64` unchanged. Always compute `changeE18` against `blockAnchorMidPriceX64`, not `prevMid`. This ensures the cumulative intra-block move is bounded by `maxChangePerBlockE18 × sqrt(1)` regardless of how many swaps or oracle updates occur within the block.

### Proof of Concept

```
Setup:
  maxChangePerBlockE18 = 0.02e18  (2% per block)
  Initial oracle price: P0 = 1.00 (Q64)
  lastMidPriceX64 = P0, lastUpdateBlock = B-1

Attack (all steps in block B, single transaction):

Step 1: Push Pyth payload → oracle price = 1.02 (P0 × 1.02)
        pool.swap(...)
        beforeSwap: prevMid=P0, midPrice=1.02
          blockDiff = B - (B-1) = 1
          allowedSq = (0.02e18)² × 2
          changeE18 = 0.02e18  →  actualSq = (0.02e18)² ≤ allowedSq  ✓
          lastMidPriceX64 = 1.02, lastUpdateBlock = B

Step 2: Push Pyth payload → oracle price = 1.0404 (1.02 × 1.02)
        pool.swap(...)
        beforeSwap: prevMid=1.02, midPrice=1.0404
          blockDiff = B - B = 0
          allowedSq = (0.02e18)² × 1
          changeE18 = 0.02e18  →  actualSq = (0.02e18)² ≤ allowedSq  ✓
          lastMidPriceX64 = 1.0404, lastUpdateBlock = B

Step 3: Push Pyth payload → oracle price = 1.0612
        pool.swap(...)  → passes (same logic)

... repeat N times ...

After N=50 steps: oracle price ≈ 1.02^50 ≈ 2.69 (+169%)
  Intended per-block cap at blockDiff=1: 2% × sqrt(2) ≈ 2.83%
  Actual cumulative move allowed: 169%
```

Each individual swap passes the velocity check. The cumulative intra-block price move is unbounded, limited only by the number of available signed Pyth payloads and the gas limit. [4](#0-3) [5](#0-4) [2](#0-1) [6](#0-5)
