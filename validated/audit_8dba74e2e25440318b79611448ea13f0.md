### Title
Missing Length Validation on Received Vector Before Index Access Causes Panic in Triple Generation — (`File: src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

The OT-based ECDSA triple generation protocol indexes into vectors received from remote participants using a compile-time constant `N` without first verifying that the received vectors contain at least `N` elements. A single malicious participant can send a short (e.g., empty) vector to trigger an index-out-of-bounds panic in every honest party's process, permanently aborting triple generation and all downstream signing.

---

### Finding Description

In `do_generation_many`, two separate receive loops index into participant-supplied vectors using `i in 0..N` with no length guard:

**Location 1 — commitment collection loop:**

```rust
// src/ecdsa/ot_based_ecdsa/triples/generation.rs ~L176-183
while all_commitments_vec.iter().any(|c| !c.full()) {
    let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
    for i in 0..N {
        all_commitments_vec[i].put(from, commitments[i]); // panics if commitments.len() < N
    }
}
```

`commitments` is a `Vec<_>` deserialized from a network message. There is no check that `commitments.len() >= N` before the loop body executes. [1](#0-0) 

**Location 2 — polynomial commitment verification loop:**

```rust
// src/ecdsa/ot_based_ecdsa/triples/generation.rs ~L331-342
for (from, their) in recv_from_others::<PolynomialCommitmentsMessageMany>(...).await? {
    for i in 0..N {
        let their_big_e   = &their.big_e_v[i];   // panics if big_e_v.len() < N
        let their_big_f   = &their.big_f_v[i];
        let their_big_l   = &their.big_l_v[i];
        let their_randomizer  = &their.randomizer_v[i];
        let their_phi_proof0  = &their.phi_proof0_v[i];
        let their_phi_proof1  = &their.phi_proof1_v[i];
        if their_big_e.degree() != threshold.value() - 1  // degree check is AFTER the index
```

The degree/length check at line 343 only executes *after* the six indexing operations above. A malicious participant sending a `PolynomialCommitmentsMessageMany` with any of its `Vec` fields shorter than `N` will panic before the check is ever reached. [2](#0-1) 

The struct whose fields are indexed without length validation:

```rust
struct PolynomialCommitmentsMessageMany {
    big_e_v: Vec<PolynomialCommitment>,
    big_f_v: Vec<PolynomialCommitment>,
    big_l_v: Vec<PolynomialCommitment>,
    randomizer_v: Vec<Randomness>,
    phi_proof0_v: Vec<dlog::Proof<Secp256K1Sha256>>,
    phi_proof1_v: Vec<dlog::Proof<Secp256K1Sha256>>,
}
``` [3](#0-2) 

The generic parameter `N` is a compile-time constant (e.g., `N=2` in the benchmarks), but the *received* vector length is fully attacker-controlled at runtime. [4](#0-3) 

---

### Impact Explanation

In Rust, an out-of-bounds index panics and unwinds (or aborts) the thread. Because `do_generation_many` is an `async fn` driving the triple generation state machine, a panic propagates out of the protocol executor and terminates the protocol for the honest party. Since Beaver triples are a prerequisite for OT-based ECDSA presigning and signing, aborting triple generation permanently denies signing capability to all honest parties for that session. There is no recovery path: the protocol must be restarted from scratch, and the malicious participant can repeat the attack indefinitely.

This matches the allowed impact: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

Any participant in the triple generation protocol can send a crafted message. The attacker only needs to be a registered participant (a documented trust assumption boundary). No cryptographic break, key leak, or external compromise is required. The attack requires sending a single malformed message and is trivially repeatable. The vulnerable code path is exercised on every triple generation run.

---

### Recommendation

Before the inner `for i in 0..N` loop, validate that every received vector has exactly `N` elements and return a `ProtocolError` (not a panic) if not:

```rust
// Location 1
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(
        format!("commitments from {from:?} has wrong length: expected {N}, got {}", commitments.len())
    ));
}

// Location 2
if their.big_e_v.len() != N || their.big_f_v.len() != N
    || their.big_l_v.len() != N || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(
        format!("PolynomialCommitmentsMessageMany from {from:?} has wrong vector lengths")
    ));
}
```

This converts a panic (unrecoverable crash) into a graceful protocol error that can be attributed to the malicious sender.

---

### Proof of Concept

A malicious participant `M` participates in `generate_triple_many::<2>` with honest parties. During the commitment collection phase (wait0), instead of sending a `Vec<Commitment>` of length 2, `M` sends an empty `Vec`. Every honest party receiving this message executes `commitments[0]` and panics, aborting their triple generation protocol. The same attack applies at the `wait2` phase by sending a `PolynomialCommitmentsMessageMany` with any empty `Vec` field.
