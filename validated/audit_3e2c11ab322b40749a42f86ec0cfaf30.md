### Title
Unprotected `new()` Initialization Allows Any Caller to Seize Engine Ownership - (File: engine/src/contract_methods/admin.rs)

### Summary

The Aurora Engine's `new()` initialization function contains no caller restriction. Any NEAR account that calls `new()` before the legitimate deployer does will become the `owner_id` of the engine, gaining full administrative control. Because the deployer must first deploy the contract and then call `new()` in a separate transaction, a front-running attacker can intercept this window and permanently seize ownership of the engine.

### Finding Description

The `new()` function in `engine/src/contract_methods/admin.rs` (lines 55–88) is the sole initialization path for the Aurora Engine's `EngineState`. Its only guard is a re-initialization check:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ... reads args, sets state including owner_id
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

There is no check on `env.predecessor_account_id()` — no `require_owner_only`, no `assert_private_call`, no comparison against `env.current_account_id()`. The `owner_id` field written into `EngineState` is taken entirely from the attacker-supplied `NewCallArgs` input:

```rust
let args = NewCallArgs::deserialize(&input)...;
let state: EngineState = args.into();
state::set_state(&mut io, &state)?;
``` [2](#0-1) 

The `EngineState.owner_id` field is the root of all administrative authority in the engine: [3](#0-2) 

Every privileged operation — `set_owner`, `upgrade`, `stage_upgrade`, `set_upgrade_delay_blocks`, `resume_precompiles`, `factory_update`, `factory_set_wnear_address`, `attach_full_access_key`, `set_key_manager` — is gated by `require_owner_only(&state, &env.predecessor_account_id())`: [4](#0-3) 

The public NEAR entrypoint is exported with no additional guard: [5](#0-4) 

On NEAR Protocol, contract deployment and initialization are two separate transactions. An attacker monitoring the chain can observe the deployment transaction, then immediately submit a `new()` call with their own account as `owner_id` before the legitimate deployer's initialization transaction is processed.

### Impact Explanation

**Critical — Auth bypass leading to direct theft of all bridged user funds.**

The attacker-controlled `owner_id` can:
1. Call `upgrade()` / `stage_upgrade()` to replace the engine bytecode with a malicious contract that drains all ETH and ERC-20 balances held by the engine.
2. Call `attach_full_access_key()` to add a full-access key to the engine account, giving unrestricted on-chain control.
3. Call `set_eth_connector_contract_account()` to redirect all deposit/withdrawal flows to an attacker-controlled connector.
4. Call `factory_set_wnear_address()` to corrupt the XCC wNEAR mapping, breaking cross-contract calls for all users.

All bridged ETH and ERC-20 tokens held in the Aurora Engine are at risk of permanent theft or freeze.

### Likelihood Explanation

**High.** NEAR Protocol processes transactions sequentially within a shard. The window between contract deployment and the `new()` call is observable on-chain. An attacker running a NEAR node or monitoring the RPC can detect the deployment and submit a competing `new()` call in the same or next block. No privileged access, leaked keys, or social engineering is required — only a standard NEAR account and a crafted function call.

### Recommendation

Add a caller restriction to `new()` so that only the contract account itself (i.e., `env.current_account_id() == env.predecessor_account_id()`) can initialize the engine. On NEAR, this is the standard pattern for ensuring initialization is only callable by the deploying account:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Add this guard:
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    // ...
}
```

Alternatively, combine deployment and initialization into a single NEAR batch transaction so no window exists between the two operations.

### Proof of Concept

1. Attacker monitors the NEAR chain for a transaction deploying the Aurora Engine WASM binary to account `aurora.near`.
2. Before the deployer's `new()` call is processed, attacker submits:
   ```
   near call aurora.near new '{"chain_id": [...], "owner_id": "attacker.near", "upgrade_delay_blocks": 0}' --accountId attacker.near
   ```
3. `state::get_state(&io)` returns `Err` (state not yet set), so the `ERR_ALREADY_INITIALIZED` guard passes.
4. No caller check exists; `state::set_state` writes `owner_id = "attacker.near"` to storage.
5. The legitimate deployer's subsequent `new()` call fails with `ERR_ALREADY_INITIALIZED`.
6. Attacker now calls `attach_full_access_key` or `upgrade` as `owner_id = "attacker.near"` to take full control of the engine and drain all bridged funds. [1](#0-0) [6](#0-5)

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
