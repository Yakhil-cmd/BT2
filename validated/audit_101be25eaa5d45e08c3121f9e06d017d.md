### Title
`evmAddress` Not Committed to in `signedData` Enables Cross-COA Ownership Proof Replay — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account owns a specific COA (Cadence-Owned Account) by checking signatures over caller-supplied `signedData`. However, the `evmAddress` parameter — which identifies *which* COA is being proven — is not required to be included in `signedData`. A signature produced to prove ownership of one COA can be replayed to prove ownership of a different COA belonging to the same Cadence account, without the account holder's consent.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The function verifies the signatures over `signedData` and then checks that the COA at `path` has an EVM address matching `evmAddress`. The critical flaw is that `evmAddress` is never required to be encoded inside `signedData` — the two are completely independent.

The code itself documents this explicitly:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The signature verification step only checks `signedData`: [2](#0-1) 

The `evmAddress` check is a separate, post-signature step that verifies the COA at `path` matches the supplied `evmAddress` — but since `evmAddress` is not bound to the signature, a caller can freely substitute any `evmAddress` they choose: [3](#0-2) 

The function is `access(all)` and callable by any unprivileged Cadence script or transaction. The Go-side precompile dispatcher (`coaOwnershipProofValidator`) passes all proof fields including the caller-controlled `evmAddress` directly to this function: [4](#0-3) 

The `COAOwnershipProofInContext` struct separates `SignedData` and `EVMAddress` as independent fields with no enforcement that one encodes the other: [5](#0-4) 

---

### Impact Explanation

If a Cadence account holds two COAs — COA_A (holding valuable assets) and COA_B (empty) — and a Cadence DApp requests a signature from the account holder to prove ownership of COA_B, the resulting signature over `signedData` (which does not encode COA_B's address) can be replayed by an attacker against COA_A. Any Cadence contract that calls `EVM.validateCOAOwnershipProof` and uses the returned `isValid: true` to authorize COA-specific operations (e.g., withdrawing bridged assets, executing calls on behalf of a COA) would incorrectly authorize the attacker's chosen `evmAddress`. This constitutes unauthorized cross-VM asset access — the exact impact class of the original report.

The COA's ERC-1271 `isValidSignature` implementation in `coa.sol` always passes `address(this)` as the `evmAddress`, so that specific path is safe: [6](#0-5) 

However, the Cadence-side `EVM.validateCOAOwnershipProof` function is a public protocol primitive callable by any Cadence contract or script, and any consumer that does not independently enforce `evmAddress ∈ signedData` is vulnerable.

---

### Likelihood Explanation

The precondition is that a victim Cadence account holds more than one COA. While the comment notes wallets "historically create at most one COA per account," the protocol imposes no such limit, and multi-COA accounts are a valid and expected use case as the EVM bridge matures. The attacker entry path requires only submitting a Cadence script or transaction — no staked node, no admin key, no social engineering. The signature to replay can be obtained from any context where the victim was asked to prove ownership of any one of their COAs (e.g., a DApp login flow).

---

### Recommendation

Enforce at the protocol level that `signedData` encodes `evmAddress`. The simplest fix is to add a pre-condition inside `validateCOAOwnershipProof` that checks whether the first 20 bytes (or a structured prefix) of `signedData` match `evmAddress`, or to document and enforce a canonical encoding scheme (e.g., `signedData = evmAddress ++ applicationData`) and reject proofs that do not conform. Alternatively, the function can be made internal-only and replaced with a version that constructs `signedData` itself, binding the EVM address into the signed message before verification.

---

### Proof of Concept

1. Alice holds two COAs: `COA_A` (with bridged NFTs) and `COA_B` (empty). Both are stored under `/public/coa` and `/public/coa2` respectively on Alice's Cadence account.
2. A DApp asks Alice to sign `signedData = "login-nonce-xyz"` to prove ownership of `COA_B`. Alice signs with her account key.
3. An attacker observes Alice's signature (e.g., from a public transaction or off-chain channel).
4. The attacker calls `EVM.validateCOAOwnershipProof(address: Alice, path: /public/coa, signedData: "login-nonce-xyz", keyIndices: [0], signatures: [aliceSig], evmAddress: COA_A.bytes)` from a Cadence script.
5. The function verifies Alice's signature over `"login-nonce-xyz"` — valid. It then checks that the COA at `/public/coa` has address `COA_A` — also valid. It returns `isValid: true`.
6. Any Cadence contract that trusts this result to authorize operations on `COA_A` (e.g., `withdrawNFT` from the bridge escrow on behalf of `COA_A`) now incorrectly grants the attacker authorization over `COA_A`'s assets. [7](#0-6) [8](#0-7)
