### Title
Non-Atomic Eviction in `handle_capacity_overflow` Allows Permanent Removal of Valid Gap-Account Transactions Without Admitting the Incoming Transaction - (File: crates/apollo_mempool/src/mempool.rs)

### Summary

`Mempool::handle_capacity_overflow` calls `try_make_space`, which permanently evicts gap-account transactions from the pool. If `try_make_space` frees *some* but not *enough* space, it returns `false`, and `handle_capacity_overflow` returns `Err(MempoolFull)`. The evictions are never rolled back. An unprivileged attacker can craft a transaction large enough to trigger eviction of all gap-account transactions yet too large to fit even after eviction, permanently removing valid third-party transactions from the mempool while their own transaction is also rejected.

### Finding Description

The analog to the external `WhitelistRegistry` bug is structural: in the Solidity case, `AddressSet.add()` silently returns `false` for an already-registered caller, and the minimum registrant is removed without the caller being admitted. Here, `try_make_space` silently returns `false` when it cannot free enough space, and the evicted transactions are removed without the incoming transaction being admitted.

The relevant code path in `add_tx_validations`:

```rust
if self.exceeds_capacity(tx, freed_bytes) {
    self.handle_capacity_overflow(tx, account_nonce, freed_bytes)?;
}
``` [1](#0-0) 

Inside `handle_capacity_overflow`:

``` [2](#0-1) 

`try_make_space` iterates over `accounts_with_gap`, removes their transactions from the pool and queue, and returns `false` if the total freed bytes never reached `required_space`: [3](#0-2) 

When `try_make_space` returns `false`, the evictions that already occurred inside the loop are **not reverted**. The test suite explicitly documents this as current behavior:

```

// We do not revert the eviction attempt even if adding large_tx ultimately fails.
assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")));
``` [4](#0-3) 

The gateway's `validate_tx` call (which precedes `add_tx`) does **not** check capacity, so it passes cleanly:

``` [5](#0-4) 

The gateway calls `validate_tx` (via `validate_by_mempool`) and then separately calls `add_tx`: [6](#0-5) 

### Impact Explanation

An attacker with a valid account can submit a transaction whose `total_bytes()` exceeds the combined size of all gap-account transactions currently in the mempool. The sequence:

1. `validate_tx` passes (no capacity check).
2. `add_tx` → `add_tx_validations` → `handle_capacity_overflow` → `try_make_space` evicts all gap-account transactions.
3. `try_make_space` returns `false` (attacker's transaction is still too large).
4. `add_tx` returns `MempoolFull` to the attacker.
5. All evicted gap-account transactions are **permanently gone** from the mempool.

Valid transactions from other accounts are rejected before sequencing without ever being included in a block. The attacker's own transaction is also rejected, but the attacker pays no on-chain cost for the attack (gateway validation does not charge fees).

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The attack requires:
1. A valid account with sufficient balance to pass `verify_can_pay_committed_bounds`.
2. A transaction with a large signature (up to `max_signature_length` felts, checked by `validate_tx_signature_size`).
3. Gap-account transactions in the mempool whose combined size is less than the attacker's transaction size.

Condition 3 is easily satisfied when the mempool contains many small gap-account transactions. The attacker can repeat the attack to progressively drain all gap-account transactions. No privileged access is required; any user can submit transactions through the public gateway RPC. [7](#0-6) 

### Recommendation

Make eviction atomic with admission. Before calling `try_make_space`, compute whether the total evictable bytes are sufficient to accommodate the incoming transaction. If not, return `MempoolFull` immediately without evicting anything. Alternatively, collect the list of transactions to evict, verify the total freed space is sufficient, and only then commit the removals.

```rust
fn handle_capacity_overflow(...) -> Result<(), MempoolError> {
    let required_space = tx.total_bytes().saturating_sub(freed_bytes);
    if !creating_gap {
        // Pre-check: total evictable bytes must cover required_space before evicting.
        let total_evictable = self.total_evictable_bytes();
        if total_evictable >= required_space && self.try_make_space(required_space) {
            return Ok(());
        }
    }
    Err(MempoolError::MempoolFull)
}
```

Result:
  - gap_tx_1 and gap_tx_2 are permanently removed from the mempool
  - Attacker's transaction is rejected
  - Mempool now has 40 bytes free (200 - 160 = 40), but gap_tx_1 and gap_tx_2 are gone

    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space

```rust
    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }
