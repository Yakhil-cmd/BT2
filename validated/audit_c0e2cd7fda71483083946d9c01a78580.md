### Title
Destroying a `CadenceOwnedAccount` Resource With a Non-Zero EVM Balance Permanently Locks FLOW Tokens - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `EVM.CadenceOwnedAccount` (COA) Cadence resource is the **sole controller** of its associated EVM address. The resource has no destructor that withdraws or redirects the EVM-side FLOW token balance before the resource is destroyed. Any user who destroys their COA resource while it holds a non-zero EVM balance permanently and irrecoverably locks those FLOW tokens, because the EVM address has no private key and no alternative control path exists.

---

### Finding Description

A `CadenceOwnedAccount` resource is created via `EVM.createCadenceOwnedAccount()`. The resulting resource owns a deterministically derived EVM address (keyed to the resource's `uuid`). The EVM contract documentation explicitly states:

> "Because of this, users are never able to access a key for their account, meaning that they cannot control their COA's address on other EVM blockchains." [1](#0-0) 

The `CadenceOwnedAccount` resource definition contains no `destroy` function (destructor). In Cadence, a resource without a destructor can still be destroyed with `destroy`, but no cleanup logic runs. The EVM-side balance is stored in the EVM state trie, not inside the Cadence resource itself, so destroying the resource does **not** touch the EVM balance. [2](#0-1) 

The only way to move FLOW tokens out of a COA's EVM address back to Cadence is through the `withdraw` function, which requires the `Owner | Withdraw` entitlement — i.e., it requires the resource to still exist: [3](#0-2) 

Once the resource is destroyed, `withdraw` can never be called again. The EVM address retains its balance in the EVM state trie permanently, with no key and no Cadence resource to authorize a withdrawal.

The existing test `TestCOAAddressDeposit` in `fvm/evm/evm_test.go` demonstrates this exact scenario — it deposits 1.23 FLOW into a COA and then immediately destroys the resource without withdrawing, and the test passes without error: [4](#0-3) 

The COA address is derived from the resource `uuid` via `InternalEVM.createCadenceOwnedAccount(uuid: acc.uuid)`: [5](#0-4) 

Cadence `uuid` values are globally unique and never reused, so no future COA can ever be assigned the same EVM address. The locked funds are permanently inaccessible.

---

### Impact Explanation

Any FLOW tokens held in a COA's EVM address at the time the COA resource is destroyed are permanently locked. There is no recovery path:

- The EVM address has no private key (by design).
- The COA resource is the only authorized caller of `InternalEVM.withdraw`.
- The `uuid`-derived EVM address cannot be re-assigned to a new COA.
- EVM-side transfers into the address remain possible but cannot be bridged back to Cadence.

This is a **cross-VM asset loss**: the Cadence-side resource is destroyed, but the EVM-side FLOW token balance persists in a permanently inaccessible state. The impact is direct, irreversible financial loss of FLOW tokens for the resource owner. [6](#0-5) 

---

### Likelihood Explanation

The entry path is reachable by any unprivileged transaction sender who owns a COA resource. No special privileges, leaked keys, or node compromise are required. Scenarios include:

1. A user intentionally destroys a COA without first withdrawing its balance (e.g., believing the resource destruction also clears the EVM balance).
2. A Cadence contract or transaction script destroys a COA as part of cleanup logic without checking `balance()` first.
3. A user deposits FLOW into a COA via the `access(all)` `deposit` function, then the COA owner destroys the resource.

The `deposit` function is `access(all)`, meaning any party can fund a COA: [7](#0-6) 

The likelihood is **medium**: the behavior is non-obvious (destroying a Cadence resource does not intuitively imply permanent EVM-side asset loss), and the pattern of creating, funding, and destroying a COA in a single transaction is demonstrated in the existing test suite without any warning.

---

### Recommendation

Add a `destroy` function (destructor) to the `CadenceOwnedAccount` resource in `fvm/evm/stdlib/contract.cdc` that checks the EVM balance and either:

1. **Panics** if the COA holds a non-zero EVM balance, forcing the caller to explicitly withdraw before destroying, or
2. **Automatically withdraws** the remaining EVM balance to a caller-specified Cadence address before the resource is destroyed.

Option 1 is the safer approach as it makes the invariant explicit and prevents accidental loss.

---

### Proof of Concept

The following Cadence transaction, submittable by any unprivileged user, permanently locks FLOW tokens:

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin
        )!
        let minter <- admin.createNewMinter(allowedAmount: 10.0)
        let vault <- minter.mintTokens(amount: 10.0)
        destroy minter

        // Step 1: Create a COA — gets a unique EVM address
        let coa <- EVM.createCadenceOwnedAccount()

        // Step 2: Deposit 10 FLOW into the COA's EVM address
        coa.deposit(from: <-vault)

        // Step 3: Destroy the COA resource WITHOUT withdrawing
        // The EVM address still holds 10 FLOW with no key and no resource to authorize withdrawal
        destroy coa

        // 10 FLOW are now permanently locked in the EVM state trie.
        // No private key exists. No new COA can ever claim this address.
        // The funds are irrecoverably lost.
    }
}
```

This is confirmed by the existing passing test at `fvm/evm/evm_test.go` lines 2182–2214, which performs exactly this sequence and expects no error. [8](#0-7)
