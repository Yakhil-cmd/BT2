### Title
Off-by-One in `SliceVec::resize` Causes Spurious OOG on Full-Capacity Heap Growth - (File: `zk_ee/src/memory/slice_vec.rs`)

---

### Summary

`SliceVec::resize` uses a `>=` guard instead of `>` when checking whether the requested new length exceeds the backing buffer. This off-by-one causes the heap to silently fail to grow to exactly the backing buffer boundary, returning `Err(())` which `resize_heap_implementation` maps to `EvmError::OutOfGas`. Any EVM transaction that causes the heap to need exactly `self.memory.len()` bytes receives a spurious OOG revert that would not occur on a reference EVM.

---

### Finding Description

In `zk_ee/src/memory/slice_vec.rs`, the `resize` method guards against out-of-bounds growth with:

```rust
pub fn resize(&mut self, new_length: usize, padding: T) -> Result<(), ()> {
    if new_length >= self.memory.len() {   // ← BUG: should be `>`
        return Err(());
    }
    ...
}
``` [1](#0-0) 

The backing slice `self.memory` has `N` slots (indices `0..N-1`). Resizing to exactly `N` elements is valid — all slots are used — but the `>=` condition rejects it. The correct guard is `new_length > self.memory.len()`.

This is inconsistent with the `TryExtend` implementation in the same file, which correctly allows filling all `cap` slots:

```rust
if idx == cap {   // ← correct: only fails when truly out of space
    return Err(());
}
``` [2](#0-1) 

The call site in `resize_heap_implementation` maps any `Err(())` from `resize` directly to `EvmError::OutOfGas`:

```rust
heap.resize(new_heap_size, 0)
    .map_err(|_| ExitCode::EvmError(EvmError::OutOfGas))?;
``` [3](#0-2) 

Gas for memory growth is charged **before** the `resize` call:

```rust
gas.pay_for_memory_growth(current_heap_size, new_heap_size)?;
heap.resize(new_heap_size, 0)
    .map_err(|_| ExitCode::EvmError(EvmError::OutOfGas))?;
``` [4](#0-3) 

So when `new_heap_size == self.memory.len()`, gas is consumed and then the transaction is reverted with OOG — even though the backing buffer has exactly enough space.

The heap backing buffer is allocated as `MAX_HEAP_BUFFER_SIZE = 1 << 27` (128 MB) per block:

```rust
pub const MAX_HEAP_BUFFER_SIZE: usize = 1 << 27; // 128 MB
...
let mut heaps = Box::new_uninit_slice_in(MAX_HEAP_BUFFER_SIZE, system.get_allocator());
``` [5](#0-4) [6](#0-5) 

Per the memory subsystem design, all EE heaps share this single array, with the most recent heap growing into the remaining tail. The off-by-one therefore applies at every call-stack depth: whenever any EE's heap slice needs to grow to exactly the remaining tail length, it gets a spurious OOG.

`resize_heap_implementation` is called from every memory-touching opcode: `MLOAD`, `MSTORE`, `MSTORE8`, `MCOPY`, `CALLDATACOPY`, `CODECOPY`, `EXTCODECOPY`, `RETURNDATACOPY`, `CREATE`, `CREATE2`. [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9) 

---

### Impact Explanation

**Classification**: EVM semantic mismatch / resource accounting bug / valid-execution unprovability.

A transaction that:
1. Has sufficient gas to pay for memory growth up to the backing buffer boundary, and
2. Issues any memory opcode whose `new_heap_size` lands exactly on `self.memory.len()`

...will be reverted with `OutOfGas` on ZKsync OS but would succeed on a reference EVM. This constitutes a state-transition divergence: the ZKsync OS block state differs from what Ethereum would produce for the same transaction. Contracts that deliberately use the maximum available heap (e.g., large in-memory data processing, memory-heavy CREATE deployments) are permanently broken on ZKsync OS. Gas is also incorrectly consumed before the spurious revert.

---

### Likelihood Explanation

The heap grows in 32-byte multiples (`max_offset.next_multiple_of(32)`). The backing buffer is 128 MB = `134,217,728` bytes, which is itself a multiple of 32. Any transaction that grows the heap to exactly 128 MB (or to exactly the remaining tail of the shared heap array at any call depth) triggers the bug. Because the backing buffer size is a round power-of-two multiple of 32, the boundary is reachable by a straightforward `MSTORE` at offset `134,217,696` (= 128 MB − 32). An attacker or a legitimate contract can craft this deliberately or encounter it organically in memory-heavy workloads.

---

### Recommendation

Change the guard in `SliceVec::resize` from `>=` to `>`:

```rust
// Before (buggy):
if new_length >= self.memory.len() {
    return Err(());
}

// After (correct):
if new_length > self.memory.len() {
    return Err(());
}
``` [1](#0-0) 

Add a regression test asserting that `resize` to exactly `self.memory.len()` succeeds and that the resulting slice contains all elements.

---

### Proof of Concept

```
Backing buffer: [MaybeUninit<u8>; 128]   (self.memory.len() == 128)
Current heap length: 96

EVM transaction issues: MSTORE at offset 96  (needs heap size = 128)
  → resize_heap_implementation computes new_heap_size = 128
  → gas.pay_for_memory_growth(96, 128) succeeds (gas consumed)
  → heap.resize(128, 0) called
      → 128 >= 128  →  Err(())   ← off-by-one fires
  → mapped to EvmError::OutOfGas
  → transaction reverted with OOG

Reference EVM: MSTORE at offset 96 with heap size 128 succeeds normally.
```

The same trigger applies at any call depth where the remaining tail of the shared heap array is exactly `new_heap_size` bytes. [11](#0-10) [12](#0-11)

    pub fn resize(&mut self, new_length: usize, padding: T) -> Result<(), ()> {
        if new_length >= self.memory.len() {
            return Err(());
        }

        if new_length > self.length {
            for x in &mut self.memory[self.length..new_length] {
                x.write(padding.clone());
            }
        }
        if new_length < self.length {
            unsafe {
                assert_unchecked(self.length <= self.memory.len());
                ptr::drop_in_place(
                    &mut self.memory[new_length..self.length] as *mut [MaybeUninit<T>]
                        as *mut [T],
                );
            }
        }
        self.length = new_length;

        Ok(())
    }

    pub(crate) fn resize_heap_implementation<'a>(
        heap: &mut SliceVec<'a, u8>,
        gas: &mut Gas<S>,
        offset: usize,
        len: usize,
    ) -> Result<(), ExitCode> {
        let max_offset = offset.saturating_add(len);
        let new_heap_size = if max_offset > ((u32::MAX - 31) as usize) {
            return Err(ExitCode::EvmError(EvmError::MemoryLimitOOG));
        } else {
            max_offset.next_multiple_of(32)
        };
        let current_heap_size = heap.len();
        if new_heap_size > current_heap_size {
            gas.pay_for_memory_growth(current_heap_size, new_heap_size)?;

            heap.resize(new_heap_size, 0)
                .map_err(|_| ExitCode::EvmError(EvmError::OutOfGas))?;
        }

        Ok(())
    }
