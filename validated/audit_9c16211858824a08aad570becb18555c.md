### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Enables Corrupted Signature Aggregation - (File: src/frost/mod.rs)

---

### Summary

Both `assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that the caller-supplied `threshold` does not exceed the participant count, but neither enforces a minimum value of 2. The DKG entry point (`assert_key_invariants` in `src/dkg.rs`) correctly rejects `threshold < 2`, but this lower-bound guard is absent from the FROST signing path. A malicious coordinator can supply `threshold = 1` (or `threshold = 0`) to the signing API, causing the protocol to aggregate signature shares with incorrect Lagrange coefficients and produce a cryptographically invalid, unusable output.

---

### Finding Description

**Root cause — missing lower-bound check in `assert_sign_inputs`:**

`src/frost/mod.rs` lines 144–150 validate only the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [1](#0-0) 

There is no corresponding lower-bound guard. The same omission exists in `presign`:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

**Contrast with DKG — lower bound IS enforced there:**

`assert_key_invariants` in `src/dkg.rs` explicitly rejects `threshold < 2`:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [3](#0-2) 

The same guard exists in `validate_triple_inputs` for OT-based ECDSA triple generation: [4](#0-3) 

**Type provides no enforcement:**

`ReconstructionLowerBound` is a plain `usize` newtype with `From`/`Into` derives and no constructor-level minimum: [5](#0-4) 

Any `usize` — including 0 or 1 — converts freely into `ReconstructionLowerBound`.

**Protocol impact of `threshold = 1`:**

In FROST, the signing aggregation computes Lagrange coefficients over the set of participating signers. With `threshold = 1`, only one signer's share is collected. The Lagrange coefficient for a single point is 1, so the aggregated signature scalar becomes the raw share of that one participant — not the Lagrange-weighted combination of the correct threshold of shares. Because the group public key was established at DKG with `threshold ≥ 2`, the resulting aggregated value does not satisfy the FROST verification equation. Honest participants complete the protocol and consume their presignature, but the output is cryptographically invalid.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

A malicious coordinator passes `threshold = 1` to the FROST sign API. All honest participants execute their signing rounds, consume their one-time presignature nonces, and hand their shares to the coordinator. The coordinator aggregates with incorrect Lagrange weights, producing a signature that fails standard EdDSA/RedJubjub verification. The presignature is permanently spent; the signing session cannot be retried with the same nonce material. Honest parties have no way to detect the coordinator's misbehavior before the aggregation step.

---

### Likelihood Explanation

**Medium.** The coordinator is an explicitly documented attacker profile in `RESEARCHER.md` ("Malicious peer/integrator/oracle only where that role is reachable without privileged assumptions"). [6](#0-5)  The entry point is the public signing API (`sign_v1`/`sign_v2` in `src/frost/eddsa/sign.rs` and the equivalent in `src/frost/redjubjub/`), which calls `assert_sign_inputs`. No key material or out-of-band access is required — only the ability to act as coordinator and supply the `threshold` argument.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place this check immediately after the upper-bound check in both functions, mirroring the pattern at `src/dkg.rs:580–582`.

---

### Proof of Concept

1. Run DKG with `participants = [A, B, C]`, `threshold = 2`. All parties obtain valid key shares.
2. Run FROST presign with `participants = [A, B, C]`, `threshold = 2`. Presign outputs are produced.
3. Malicious coordinator invokes the FROST sign function with `participants = [A, B, C]`, **`threshold = 1`**, `coordinator = A`, `me = A`.
4. `assert_sign_inputs` evaluates: `1 > 3` → false (no upper-bound error). No lower-bound check exists. Returns `Ok(participants)`.
5. The signing protocol proceeds. The coordinator collects only its own signature share (threshold = 1 means one share suffices).
6. Aggregation computes Lagrange coefficient = 1 for the single share; the aggregated `s` equals A's raw share scalar.
7. The resulting `(R, s)` pair fails EdDSA verification against the group public key established at DKG with `threshold = 2`.
8. The presignature nonces are consumed and cannot be reused; the signing session is permanently lost.
