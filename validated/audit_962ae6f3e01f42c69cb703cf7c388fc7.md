### Title
Gateway Proof-Facts Block-Number Boundary Is One Block Ahead of Available State, Causing Valid Invoke V3 Transactions to Be Rejected — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`run_validate_entry_point` advances the block number by one (`unchecked_next()`) to simulate the next block, but the state reader still reflects the latest committed block. `validate_proof_block_number` therefore permits proof block numbers up to `(latest_block + 1) − STORED_BLOCK_HASH_BUFFER`, while `validate_proof_block_hash` looks up the stored hash in the state at `latest_block`, where that hash has not yet been written. Any invoke V3 transaction carrying a proof that references exactly block `latest_block − STORED_BLOCK_HASH_BUFFER + 1` passes the number check but fails the hash lookup with "Block hash mismatch", causing the gateway to reject a transaction that would succeed during actual block execution.

---

### Finding Description

**Step 1 — Block number is incremented in the gateway.**

In `run_validate_entry_point`, the gateway fetches the latest committed block info and bumps the block number before building the `BlockContext`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // ← latest_block + 1
let block_context = BlockContext::new(block_info, ...);
```

The state reader, however, is still rooted at `latest_block`. [1](#0-0) 

**Step 2 — `validate` dispatches to `perform_validations` → `perform_pre_validation_stage`.**

`StatefulValidatorTrait::validate` is implemented as a thin wrapper:

```rust
fn validate(&mut self, account_tx: AccountTransaction) -> StatefulValidatorResult<()> {
    self.perform_validations(account_tx)
}
``` [2](#0-1) 

For invoke transactions, `perform_validations` calls `perform_pre_validation_stage` with the same `block_context` (block number = `latest_block + 1`) and the state at `latest_block`: [3](#0-2) 

**Step 3 — `validate_proof_facts` computes the allowed range from the incremented block number.**

`perform_pre_validation_stage` calls `validate_proof_facts`, which calls `validate_proof_block_number` with `block_context.block_info.block_number` = `latest_block + 1`: [4](#0-3) 

`validate_proof_block_number` computes:

```rust
let max_allowed =
    current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
// max_allowed = (latest_block + 1) − STORED_BLOCK_HASH_BUFFER
//             = latest_block − STORED_BLOCK_HASH_BUFFER + 1
``` [5](#0-4) 

A proof referencing block `latest_block − STORED_BLOCK_HASH_BUFFER + 1` satisfies `proof_block_number ≤ max_allowed` and passes this check.

**Step 4 — `validate_proof_block_hash` reads from state at `latest_block`, where the hash is not yet present.**

```rust
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

if stored_block_hash != proof_block_hash {
    return Err(TransactionPreValidationError::InvalidProofFacts(
        format!("Block hash mismatch for block {proof_block_number}. ...")
    ));
}
``` [6](#0-5) 

The OS writes the hash for block `N − STORED_BLOCK_HASH_BUFFER` during block `N` processing (`pre_process_block` → `write_block_number_to_block_hash_mapping`). Therefore the hash for block `latest_block − STORED_BLOCK_HASH_BUFFER + 1` is written during block `latest_block + 1` processing — **after** the gateway validation snapshot. At validation time `stored_block_hash = Felt::ZERO`, so `stored_block_hash ≠ proof_block_hash` and the transaction is rejected. [7](#0-6) 

**Step 5 — The same transaction succeeds during actual execution.**

During block `latest_block + 1` execution, `pre_process_block` runs before any transaction and writes the hash for block `(latest_block + 1) − STORED_BLOCK_HASH_BUFFER`. The proof hash check therefore passes, and the transaction executes successfully — contradicting the gateway's rejection.

---

### Impact Explanation

This matches the allowed impact: **"Mempool/gateway/RPC admission rejects valid transactions before sequencing."**

Any user submitting a client-side proving invoke V3 transaction with a proof referencing the most recent permissible block (`latest_block − STORED_BLOCK_HASH_BUFFER + 1`) — a natural choice to minimise proof staleness — will have their transaction rejected at the gateway with a spurious `ValidateFailure` / "Block hash mismatch" error. The transaction is valid and would be accepted by the blockifier during actual block execution.

---

### Likelihood Explanation

The trigger is unprivileged and requires no special access: any user of the client-side proving feature who picks the freshest allowed block number hits the bug. The boundary block is exactly one block behind the incremented block number used by the gateway, so the off-by-one is hit every time a user targets the most recent proof-eligible block.

---

### Recommendation

In `run_validate_entry_point`, do **not** advance the block number before computing the proof block number bound. The proof block hash lookup must be consistent with the state snapshot used. Either:

1. Use the non-incremented block number for the `BlockContext` passed to `validate_proof_facts` (while keeping `unchecked_next()` for other purposes such as nonce/fee checks that genuinely need the next-block context), or
2. Simulate the `pre_process_block` hash-write for block `latest_block − STORED_BLOCK_HASH_BUFFER + 1` into the cached state before running `validate_proof_block_hash`.

Option 1 is simpler and consistent with the invariant stated in `validate_proof_block_number`'s own comment: *"Stored block hashes are guaranteed only up to: current − STORED_BLOCK_HASH_BUFFER."* The "current" block for the purpose of hash availability is `latest_block`, not `latest_block + 1`.

---

### Proof of Concept

Let `N = latest_block`, `B = STORED_BLOCK_HASH_BUFFER`.

1. Craft an invoke V3 transaction with `proof_facts.block_number = N − B + 1` and the correct block hash for that block.
2. Submit to the gateway.
3. Gateway calls `run_validate_entry_point` → sets `block_number = N + 1`.
4. `validate_proof_block_number`: `max_allowed = (N+1) − B = N − B + 1`. Check passes (`N−B+1 ≤ N−B+1`).
5. `validate_proof_block_hash`: reads `stored_block_hash` for block `N−B+1` from state at block `N`. Hash not yet written → `stored_block_hash = 0x0`.
6. `0x0 ≠ proof_block_hash` → error: `"Block hash mismatch for block N−B+1"`.
7. Gateway returns `ValidateFailure`. Transaction rejected.
8. If the same transaction is included directly in block `N+1` (bypassing the gateway), `pre_process_block` writes the hash for block `N−B+1` first, the check passes, and the transaction executes successfully.
