### Title
Missing Sender Authorization in Bank Precompile `transfer` Enables Unauthorized Transfer of Native EVM Tokens - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer(sender, recipient, amount)` method does not validate that the `sender` argument equals `contract.Caller()`. Any unprivileged EVM contract can transfer `evm/<callerAddress>` native tokens from any address to any address without the token holder's consent, constituting an unauthorized transfer of precompile-controlled assets.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case in `BankContract.Run()` unpacks `sender` from the ABI-decoded call arguments and passes it directly as the `from` address to `bankKeeper.SendCoins()`:

```go
case TransferMethodName:
    args, err := method.Inputs.Unpack(contract.Input[4:])
    sender := args[0].(common.Address)          // attacker-controlled
    recipient := args[1].(common.Address)
    amount := args[2].(*big.Int)
    from := sdk.AccAddress(sender.Bytes())       // no check: sender == contract.Caller()
    to := sdk.AccAddress(recipient.Bytes())
    denom := EVMDenom(contract.Caller())         // evm/<callerAddress>
    // ...
    bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The denom is correctly scoped to `evm/<contract.Caller()>`, but the `from` address is taken verbatim from the call input with no authorization check. There is no requirement that `sender == contract.Caller()`, no allowance mechanism, and no consent check of any kind.

The analog to the external report is exact: just as the KYC upload accepted arbitrary file types because the type field was never validated, the bank precompile accepts an arbitrary `sender` address because the field is never validated against the actual caller.

### Impact Explanation
**Critical — Unauthorized transfer of precompile-controlled assets.**

Any EVM contract at address `0xC` can call `bank.transfer(victimAddress, attackerAddress, amount)` and cause `bankKeeper.SendCoins` to move `evm/0xC` tokens from `victimAddress` to `attackerAddress` with no authorization from `victimAddress`. The `evm/<address>` denom is the native-layer representation of CRC20/CRC21 tokens; holders of these tokens can convert them back to EVM-side tokens. Stealing them is a direct, irreversible balance change on a precompile-controlled asset, which falls squarely within the Critical impact class.

### Likelihood Explanation
The bank precompile is a globally accessible precompile reachable by any deployed EVM contract. No privileged key, governance action, or admin permission is required. The attacker only needs to deploy a contract and have users hold `evm/<attackerContract>` tokens — achievable by calling `bank.mint(victimAddress, amount)` from the same contract beforehand, or by operating a CRC20 contract that users interact with normally (calling `burn` on the CRC20 side creates `evm/<contract>` native tokens on the Cosmos side).

### Recommendation
Add an explicit check that `sender == contract.Caller()` before executing the transfer:

```go
case TransferMethodName:
    // ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("transfer: sender must be the caller")
    }
    // ... rest of the logic
```

Alternatively, redesign the method to always use `contract.Caller()` as the `from` address, consistent with how `mint` and `burn` derive the target address from the first argument but scope the denom to the caller.

### Proof of Concept

1. Attacker deploys `MaliciousToken` contract at `0xMalicious`.
2. Attacker calls `bank.mint(victimAddress, 1000)` from `0xMalicious`.
   - `bankKeeper.MintCoins` creates 1000 `evm/0xMalicious` tokens and sends them to `victimAddress`.
   - Victim now holds 1000 `evm/0xMalicious` native tokens.
3. Attacker calls `bank.transfer(victimAddress, attackerAddress, 1000)` from `0xMalicious`.
   - `sender = victimAddress` (attacker-supplied argument, never validated).
   - `denom = "evm/0xMalicious"` (derived from `contract.Caller()`).
   - `bankKeeper.SendCoins(ctx, victimAddr, attackerAddr, [1000 evm/0xMalicious])` executes unconditionally.
4. Victim's balance of `evm/0xMalicious` drops to 0; attacker holds 1000 `evm/0xMalicious` tokens.
5. Attacker converts them back to EVM-side CRC20 tokens via the standard conversion path.

No victim action, no admin key, no governance proposal required. [1](#0-0) [2](#0-1) [3](#0-2)
