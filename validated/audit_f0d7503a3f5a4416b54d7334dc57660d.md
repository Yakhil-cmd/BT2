### Title
Predictable `PREVRANDAO` via `keccak256(block.timestamp)` Enables Randomness Manipulation in EVM Contracts - (`x/evm/keeper/keeper.go`, `giga/deps/xevm/keeper/keeper.go`)

---

### Summary

Both EVM keeper implementations in Sei set the `PREVRANDAO` opcode value to `keccak256(block.timestamp)`. Since `block.timestamp` is a low-entropy, predictable value (set by the block proposer and observable before execution), any EVM contract on Sei that relies on `block.prevrandao` as a source of randomness is trivially exploitable by an unprivileged attacker who can predict the value in advance.

---

### Finding Description

In `GetVMBlockContext`, both keepers compute the `Random` field (which backs the EVM `PREVRANDAO` / `0x44` opcode) as:

```go
// Use hash of block timestamp as info for PREVRANDAO
r, err := ctx.BlockHeader().Time.MarshalBinary()
rh := crypto.Keccak256Hash(r)
...
Random: &rh,
``` [1](#0-0) [2](#0-1) 

The `Random` field is what the go-ethereum EVM runtime exposes as `block.prevrandao` (opcode `PREVRANDAO`). On Ethereum post-merge, this is the beacon chain RANDAO value — a cryptographically unpredictable commitment. On Sei, it is simply `keccak256(binary(block.timestamp))`.

`block.timestamp` on Sei is the Tendermint block time, which:
1. Is set by the block proposer and is known to all nodes before the block is finalized.
2. Follows a regular, predictable cadence (Sei's ~400ms block time).
3. Is observable by any participant watching the mempool/chain.

Because `keccak256` is a deterministic function, `PREVRANDAO = keccak256(block.timestamp)` is fully computable by anyone who knows (or can predict) the block timestamp.

---

### Impact Explanation

Any EVM contract deployed on Sei that uses `block.prevrandao` as a source of randomness — a common and recommended pattern post-merge on Ethereum — is vulnerable. Concrete examples:

- **On-chain lotteries / prize draws**: attacker predicts the winning ticket index and only submits when they win.
- **NFT trait/rarity assignment**: attacker times minting to land on desired traits.
- **Gaming contracts**: attacker predicts dice rolls, card draws, etc.

The sei-chain EVM keeper is the necessary vulnerable step: it is the infrastructure that supplies a predictable value where contracts expect an unpredictable one. Fund loss in deployed contracts is the direct consequence.

Impact: **Medium** (fund loss/freeze in EVM contracts; amount depends on contract TVL, but the mechanism is fully attacker-controlled).

---

### Likelihood Explanation

- **No special privileges required**: any user can observe block timestamps and compute `keccak256(binary(timestamp))` off-chain before submitting a transaction.
- **Widely used pattern**: `block.prevrandao` is the standard post-merge randomness source recommended by the Ethereum ecosystem; many contracts use it.
- **Fully deterministic**: the attacker does not need to brute-force anything — the value is directly computable.
- **Sei's fast block time** (~400ms) makes timestamp prediction even easier since the window is narrow and regular.

Likelihood: **High**.

---

### Recommendation

Replace the timestamp-derived value with a source that is not known before block finalization. Options in order of preference:

1. **Use the Tendermint block hash** (`ctx.HeaderHash()`) as the `Random` seed. The block hash commits to all transactions and validator signatures and is not known until the block is finalized.
2. **Use a VRF-based beacon** (e.g., integrate with a decentralized randomness oracle).
3. At minimum, mix in additional entropy: `keccak256(block.timestamp ++ block.height ++ last_block_hash)` — still manipulable by the proposer but raises the bar for unprivileged attackers.

---

### Proof of Concept

An attacker can deploy the following contract on Sei and call `attack()` to predict the outcome of any victim contract that uses `block.prevrandao`:

```solidity
interface IVictimLottery {
    function enter() external payable;
    function draw() external; // uses block.prevrandao % numTickets
}

contract PrevrandaoAttack {
    IVictimLottery public victim;

    constructor(address _victim) { victim = _victim; }

    function predictPrevrandao() public view returns (bytes32) {
        // Sei sets PREVRANDAO = keccak256(abi.encode(block.timestamp))
        // block.timestamp is known before the tx executes
        return keccak256(abi.encode(block.timestamp));
    }

    function attack(uint256 numTickets, uint256 wantedTicket) external payable {
        bytes32 predicted = predictPrevrandao();
        uint256 winner = uint256(predicted) % numTickets;
        require(winner == wantedTicket, "not our turn");
        victim.enter{value: msg.value}();
        victim.draw();
    }
}
```

The attacker calls `attack()` repeatedly (or simulates off-chain) until `block.timestamp` produces the desired `PREVRANDAO` value, then submits the real transaction. Because Sei's block time is ~400ms and timestamps are predictable, the attacker can pre-compute the exact block to target.
