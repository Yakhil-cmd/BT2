### Title
`MsgTurnBridge` Handler Is a No-Op Stub: Bridge Cannot Be Disabled and Permission Check Is Bypassed - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` gRPC message handler in `x/cronos/keeper/msg_server.go` is implemented as a complete no-op (`return nil, nil`). It performs no permission check and never updates any bridge state. As a result: (1) any unprivileged user can call `MsgTurnBridge` and it silently succeeds, bypassing the `CanTurnBridge` permission system; and (2) the bridge circuit-breaker can never actually disable bridge operations, permanently defeating the emergency shutdown mechanism.

### Finding Description

The `MsgTurnBridge` message is a fully defined, reachable Cosmos SDK transaction surface. It has a proto definition, a CLI command (`CmdTurnBridge`), a `MsgTurnBridge` struct with `Sender` and `Enable` fields, and is registered in the `MsgServer` interface. The permission system explicitly reserves `CanTurnBridge` as a distinct permission bit for this operation.

However, the actual server-side handler is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly enforces its permission:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

The permission constants show `CanTurnBridge` is a defined, intended guard: [3](#0-2) 

The CLI command is fully wired and callable by any user: [4](#0-3) 

### Impact Explanation

**Two distinct High-severity impacts:**

1. **Permission bypass**: Any unprivileged address can submit `MsgTurnBridge` and receive a success response (`nil, nil`). The `CanTurnBridge` permission bit — which exists precisely to restrict this operation to authorized addresses — is never checked. This is a direct bypass of Cronos's bridge authorization system.

2. **Bridge circuit-breaker permanently broken**: The bridge emergency shutdown mechanism is completely non-functional. No call to `TurnBridge` — whether from an authorized or unauthorized address — will ever change bridge state. This is a permanent inability to disable bridge/conversion flows under any network condition, including emergencies.

This maps to the allowed High impacts:
- *Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks*
- *Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows*

### Likelihood Explanation

**High.** The entry path is a standard, publicly documented Cosmos SDK transaction (`MsgTurnBridge`) with a working CLI command. Any user can submit it. The no-op stub means the disabling functionality has never worked since it was introduced. The integration test `test_gravity_turn_bridge` in `integration_tests/test_gravity.py` explicitly tests that turning the bridge off should block bridge operations — but this test exercises the gravity module's own `bridge_active` parameter path, not the Cronos `MsgTurnBridge` handler, masking the bug. [5](#0-4) 

### Recommendation

Implement `TurnBridge` with the same pattern as `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` if it fails.
2. Persist the `Enable` flag to module params (e.g., a `BridgeActive` param) or a dedicated KV store key.
3. Enforce the stored flag in all bridge-related handlers (`SendToIbcHandler`, `IbcTransferCoins`, EVM log handlers for gravity) before processing any bridge operation.

### Proof of Concept

Any user can submit the following transaction and receive a success response, with no state change:

```bash
# Unprivileged address calls TurnBridge false — succeeds silently
cronosd tx cronos turn-bridge false --from any_unprivileged_user --chain-id cronos_777-1

# Query bridge state — unchanged, bridge is still active
# Subsequent bridge operations proceed normally, circuit-breaker has no effect
```

				return err

				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	return cmd
}
