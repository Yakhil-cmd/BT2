### Title
Gateway Stateful Validator Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` for Resource Bounds Admission — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads `gas_prices.strk_gas_prices.l2_gas_price` from the latest committed block header to gate admission. The correct reference is `next_l2_gas_price` — the price that will actually govern the next block — which is a distinct field in the same block header. The code itself carries an explicit TODO acknowledging the wrong field is used. Because the two values diverge whenever the EIP-1559-style fee market adjusts, the gateway either admits transactions whose `max_price_per_unit` is below the next block's actual price (rising-price case) or rejects transactions that would succeed (falling-price case).

### Finding Description

In `validate_resource_bounds` the validator fetches block info and extracts the current block's STRK L2 gas price:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

That value is then passed to `validate_tx_l2_gas_price_within_threshold`, which enforces:

```
tx_l2_gas_price  >=  (min_gas_price_percentage / 100)  *  current_block_l2_gas_price
``` [2](#0-1) 

The block header stored in `StorageBlockHeader` carries two distinct fields:

- `l2_gas_price: GasPricePerToken` — the price **used in the current block**
- `next_l2_gas_price: GasPrice` — the price **that will be applied to the next block** [3](#0-2) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` from `block_header.l2_gas_price.price_in_fri` and never reads `next_l2_gas_price`: [4](#0-3) 

The `run_validate_entry_point` path also builds its block context from the same stale prices (incrementing only the block number, not the gas price): [5](#0-4) 

The consensus orchestrator writes `next_l2_gas_price` into every committed block header via `update_state_sync_with_new_block`: [6](#0-5) 

That value is computed by `calculate_next_l2_gas_price` / `calculate_next_l2_gas_price_for_fin` and can differ materially from the current block's price whenever gas usage deviates from the target. [7](#0-6) 

### Impact Explanation

**Rising-price scenario** (`next_l2_gas_price > current_l2_gas_price`): A transaction whose `max_price_per_unit` sits between the two values passes the gateway's threshold check (anchored to the stale lower price) and is admitted to the mempool. When the batcher executes it against the next block's actual gas price, the blockifier's `check_fee_bounds` / `InsufficientResourceBounds` check rejects it. The gateway has admitted a transaction that cannot be sequenced.

**Falling-price scenario** (`next_l2_gas_price < current_l2_gas_price`): A transaction whose `max_price_per_unit` is above the next block's price but below the stale threshold is rejected by the gateway even though it would execute successfully. Valid transactions are incorrectly excluded from the mempool.

Both outcomes match: **High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The L2 gas price is recalculated after every block via the EIP-1559-style `calculate_next_base_gas_price`. During any period of sustained high throughput the price rises monotonically block-by-block. Any user who submits a transaction with `max_price_per_unit` calibrated to the current block's price (e.g., by reading it from the latest block header via RPC) will land in the gap whenever the price has already moved. No special privilege is required; the trigger is a normal transaction submission.

### Recommendation

1. Extend `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` (or add a dedicated accessor) to surface `block_header.next_l2_gas_price` alongside the existing gas-price fields.
2. Replace the `gas_prices.strk_gas_prices.l2_gas_price` read in `validate_resource_bounds` with the `next_l2_gas_price` value, resolving the acknowledged TODO.
3. Apply the same correction to the block context constructed in `run_validate_entry_point` so that the blockifier validation during gateway admission is also anchored to the next block's price.

### Proof of Concept

1. Read the latest committed block header; note `l2_gas_price_fri = P` and `next_l2_gas_price = P'` where `P' > P` (rising market).
2. Construct an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = P` (exactly the current price, above the `min_gas_price_percentage` threshold anchored to `P`).
3. Submit via the gateway. `validate_resource_bounds` computes `threshold = (min_gas_price_percentage/100) * P ≤ P`, so the check passes and the transaction enters the mempool.
4. The batcher builds the next block using `next_l2_gas_price = P'`. The blockifier's `check_fee_bounds` finds `max_price_per_unit (P) < block_l2_gas_price (P')` and raises `ResourceBoundsError::MaxGasPriceTooLow` for L2 gas.
5. The transaction is dropped from the block; the gateway has admitted a transaction that cannot be sequenced.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

/// Compute the next L2 gas price (for the fin or for updating state). Respects override when set.
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
