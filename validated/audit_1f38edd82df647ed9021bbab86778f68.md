### Title
`skip_stateful_validations` Bypasses `__validate__` for Nonce-1 Invoke When Any Future-Nonce Transaction Exists in Pool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX feature that skips `__validate__` for a nonce-1 invoke (to allow deploy-account + invoke to be submitted simultaneously) uses an overly broad membership check. `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the account in the pool, not only a `deploy_account` transaction. For a deployed account with on-chain nonce 0 that already has a future-nonce invoke (nonce ≥ 2) in the pool, a second nonce-1 invoke with a completely invalid signature will have its `__validate__` entry point silently skipped and will be admitted into the mempool.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` to decide whether to set `validate = false` before the blockifier validation step: [1](#0-0) 

The skip condition fires when `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)`, and the decision is delegated entirely to: [2](#0-1) 

`account_tx_in_pool_or_recent_block` is implemented as: [3](#0-2) 

It returns `true` if the account has **any** transaction in the pool or any transaction in a recent committed block — it does not distinguish between a `deploy_account` transaction and a future-nonce invoke transaction.

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, an invoke with `validate = false` returns `Ok(())` immediately without calling `__validate__`: [5](#0-4) 

The code comment in `skip_stateful_validations` states the intended invariant:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of this disjunction is the flaw. A future-nonce invoke (nonce = 2) passes `__validate__` with **its own signature**. That says nothing about whether a separately submitted nonce-1 invoke carries a valid signature. The two transactions are independent; the nonce-1 invoke's signature is never verified.

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction.**

An invoke transaction whose `__validate__` entry point would reject it (e.g., invalid or forged signature) is accepted by the gateway and inserted into the mempool. This breaks the admission invariant that every accepted invoke must have passed account-level signature/authorization validation. Consequences include:

- Mempool and batcher resources are consumed by transactions that will revert at execution time.
- For account contracts whose `__validate__` is the sole authorization gate, the attacker can inject an unauthorized call that reaches the batcher's execution stage.
- The attacker can grief a specific account by flooding its nonce-1 slot with invalid transactions, preventing the legitimate nonce-1 transaction from being sequenced first.

---

### Likelihood Explanation

**Medium.** The preconditions are:

1. The target account is deployed and has on-chain nonce = 0 (freshly deployed, no executed transactions yet).
2. The attacker (or anyone) has already submitted a valid future-nonce invoke (nonce ≥ 2) for that account, causing `tx_pool.contains_account` to return `true`.
3. The attacker submits a nonce-1 invoke with an invalid/arbitrary signature.

Condition 1 is common for newly deployed accounts. Condition 2 can be self-arranged by the attacker (submit a valid nonce-2 invoke first). Condition 3 is trivially satisfied. No privileged access is required.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address)` that only returns `true` when a `DeployAccount` transaction for that address is present. This preserves the intended UX feature while closing the bypass for deployed accounts with future-nonce invokes in the pool.

Alternatively, add a secondary guard: only skip `__validate__` if the account contract does **not** yet exist in state (i.e., `get_class_hash_at(sender_address)` returns the zero class hash), which is the true precondition the feature was designed for.

---

### Proof of Concept

```
1. Deploy account A on-chain. On-chain nonce for A = 0.

2. Attacker submits InvokeV3 for A with nonce=2, valid signature.
   → Gateway: tx_nonce=2, account_nonce=0, skip_stateful_validations returns false
     (condition tx_nonce==1 is not met), __validate__ is called, passes.
   → tx_pool.contains_account(A) = true.

3. Attacker submits InvokeV3 for A with nonce=1, INVALID/FORGED signature.
   → Gateway stateful validator:
       account_nonce = get_nonce_from_state(A) = 0
       skip_stateful_validations:
         tx.nonce() == Nonce(ONE) ✓
         account_nonce == Nonce(ZERO) ✓
         account_tx_in_pool_or_recent_block(A) = true  ← due to nonce-2 tx
         → returns true (skip_validate = true)
       run_validate_entry_point: execution_flags.validate = false
       StatefulValidator::perform_validations: returns Ok(()) immediately
   → Invalid-signature nonce-1 invoke is accepted into the mempool.
```

The broken invariant: `account_tx_in_pool_or_recent_block` at [3](#0-2)  does not distinguish deploy-account transactions from invoke transactions, causing `skip_stateful_validations` at [1](#0-0)  to suppress `__validate__` in a scenario it was not designed to handle.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
