### Title
Wrong Boolean Operator Causes Signature Validation to Never Skip in Ethereum Transaction Flow — (`basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`)

---

### Summary

The Ethereum-path transaction validator uses a bitwise OR (`|`) instead of a logical OR (`||`) to combine two boolean config flags. Due to Rust's operator precedence, the expression `!Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION` is evaluated as `(!Config::VALIDATE_EOA_SIGNATURE) | Config::SIMULATION`, which is semantically correct in isolation — but the use of the non-short-circuiting bitwise `|` instead of logical `||` is a latent correctness hazard. More critically, the analogous pattern in the ZK transaction path correctly uses `Config::SIMULATION == false` to gate the EIP-3607 check, but the Ethereum path **does not gate EIP-3607 at all** — it always enforces `RejectCallerWithCode` regardless of whether `SIMULATION` is `true`. This means that for Ethereum-type transactions processed through `BasicBootloaderForwardETHLikeConfig` (or any config where `SIMULATION = false`), a contract address can never be the `from` field even in simulation/`eth_call` contexts that use the Ethereum-like flow.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`, the signature-skip guard at line 201 reads:

```rust
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // No native for Eth STF
} else {
    // ... ecrecover signature validation ...
}
```

The intent is: skip signature validation when either `VALIDATE_EOA_SIGNATURE` is `false` OR `SIMULATION` is `true`. The bitwise `|` on `bool` values in Rust is non-short-circuiting but produces the same boolean result as `||` — so the signature skip itself works correctly for the existing configs.

However, immediately after the signature block, the EIP-3607 check at lines 269–272 is **unconditional**:

```rust
// EIP-3607: Reject transactions from senders with deployed code modulo delegations
if originator_account_data.is_contract() {
    return Err(InvalidTransaction::RejectCallerWithCode.into());
}
```

Compare this to the ZK transaction path in `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 324, which correctly gates the same check:

```rust
// EIP-3607: Reject transactions from senders with deployed code modulo delegations
// We skip it for simulation to allow simulate calls between contracts
if Config::SIMULATION == false && originator_account_data.is_contract() {
    return Err(InvalidTransaction::RejectCallerWithCode.into());
}
```

The ZK path explicitly documents that simulation must bypass EIP-3607 to allow contract-to-contract `eth_call` simulation. The Ethereum path omits this bypass entirely.

The `BasicBootloaderCallSimulationConfig` sets `SIMULATION = true` and is used by `simulate_tx` / `simulate_block`. When an Ethereum-type transaction (RLP-encoded) is submitted via simulation with a contract address as `from`, the ZK path would allow it, but the Ethereum path will always reject it with `RejectCallerWithCode`.

---

### Impact Explanation

Any caller using `eth_call` or `eth_estimateGas` with an Ethereum-type transaction where `from` is a contract address will receive a spurious `RejectCallerWithCode` validation error. This is a broken permissionless path: the simulation API is intended to be callable by anyone with any `from` address (including contracts), but the Ethereum validation path incorrectly enforces a production-only restriction in simulation mode.

This is directly analogous to the M-19 report: a function intended to be permissionless (simulation) passes the wrong flag/omits the bypass, causing it to reject callers that should be allowed.

Concrete impact:
- `eth_call` from a contract address fails for Ethereum-type transactions.
- `eth_estimateGas` for contract-initiated calls fails.
- Broken tooling/integration for any system that simulates contract-originated Ethereum transactions.

---

### Likelihood Explanation

Any node operator or user calling `eth_call` or `eth_estimateGas` with `from` set to a contract address (a common pattern for simulating contract interactions) will trigger this. The `simulate_tx` entry point is a public API surface. Likelihood is high for any protocol that simulates contract-to-contract calls using Ethereum-type transactions.

---

### Recommendation

Mirror the ZK path fix in the Ethereum validation path. Gate the EIP-3607 check with `Config::SIMULATION`:

```rust
// EIP-3607: Reject transactions from senders with deployed code modulo delegations
// Skip for simulation to allow simulate calls between contracts
if !Config::SIMULATION && originator_account_data.is_contract() {
    return Err(InvalidTransaction::RejectCallerWithCode.into());
}
```

Additionally, replace the bitwise `|` with logical `||` in the signature-skip guard for clarity and to prevent future confusion:

```rust
if !Config::VALIDATE_EOA_SIGNATURE || Config::SIMULATION {
    // skip signature validation
} else {
    // ... ecrecover ...
}
```
