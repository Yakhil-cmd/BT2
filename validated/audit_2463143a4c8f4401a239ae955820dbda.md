### Title
Missing Zero-Address Validation in `ExitToEthereum` Precompile Allows Permanent Destruction of User Funds - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile and the `EvmErc20`/`EvmErc20V2` withdrawal contracts accept `address(0)` as a valid recipient without any zero-address guard. Because the ERC-20 burn is executed before the precompile call, and the precompile schedules a NEAR promise with no refund callback, any user who supplies `address(0)` as the Ethereum recipient permanently destroys their own funds with no recovery path.

---

### Finding Description

`ExitToEthereum::run()` handles two withdrawal paths:

**Flag `0x0` – ETH base-token exit** (lines 889–914):

```rust
let recipient_address: Address = input
    .try_into()
    .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECIPIENT_ADDRESS")))?;
// ← no check: recipient_address != Address::zero()
``` [1](#0-0) 

**Flag `0x1` – ERC-20 exit** (lines 938–965):

```rust
let recipient_address = Address::try_from_slice(input)
    .map_err(|_| ExitError::Other(Cow::from("ERR_WRONG_ADDRESS")))?;
// ← no check: recipient_address != Address::zero()
``` [2](#0-1) 

After parsing, the precompile unconditionally schedules a NEAR `withdraw` promise to the ETH connector with the zero address as recipient:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,          // contains recipient = 0x0000...
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [3](#0-2) 

Critically, unlike `ExitToNear`, the `ExitToEthereum` precompile uses a bare `PromiseArgs::Create` with **no callback**. There is no `refund_on_error` path that can re-mint burned tokens if the downstream NEAR promise fails.

On the Solidity side, `EvmErc20.withdrawToEthereum` burns the caller's tokens *before* the precompile call, and the assembly `call` return value is never checked:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed first
    bytes20 recipient_b = bytes20(recipient);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient_b);
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [4](#0-3) 

The same pattern exists in `EvmErc20V2`: [5](#0-4) 

---

### Impact Explanation

**Permanent freezing / destruction of funds (Critical).**

Two concrete loss paths:

1. **ERC-20 path**: A user calls `withdrawToEthereum(address(0), amount)`. The ERC-20 tokens are burned on the Aurora EVM. The precompile schedules a NEAR `withdraw` promise to the ETH connector with `recipient = 0x0000...0000`. Whether the ETH connector accepts or rejects the zero address, the ERC-20 tokens are already gone and there is no refund callback to re-mint them. Funds are permanently destroyed.

2. **ETH base-token path**: A user calls the `ExitToEthereum` precompile directly (flag `0x0`) with `recipient = address(0)` and attaches ETH value. The ETH is deducted from the caller's EVM balance and a `withdraw` promise is scheduled to the ETH connector with the zero Ethereum address. Again, no refund callback exists. Funds are permanently destroyed.

---

### Likelihood Explanation

**Medium.** The entry point is a standard public ERC-20 function callable by any token holder without any privilege. Accidental misuse (e.g., a front-end bug, a copy-paste error in a script, or a contract that passes an uninitialized `address` variable) is a realistic scenario. The absence of a zero-address guard is a well-known class of omission that has caused real losses in production protocols.

---

### Recommendation

Add a zero-address guard in `ExitToEthereum::run()` for both the ETH and ERC-20 paths, immediately after parsing `recipient_address`:

```rust
if recipient_address == Address::zero() {
    return Err(ExitError::Other(Cow::from("ERR_ZERO_RECIPIENT_ADDRESS")));
}
```

Apply the same guard in `EvmErc20.withdrawToEthereum` and `EvmErc20V2.withdrawToEthereum`:

```solidity
require(recipient != address(0), "ERR_ZERO_RECIPIENT_ADDRESS");
```

### Proof of Concept

**ERC-20 path (Solidity):**

```solidity
// Attacker holds some ERC-20 tokens on Aurora
uint256 balance = token.balanceOf(attacker);  // e.g. 1000

// Call withdraw with zero address – no revert, tokens are burned
token.withdrawToEthereum(address(0), balance);

// Post-state: attacker's ERC-20 balance is 0, no ETH received on Ethereum
assert(token.balanceOf(attacker) == 0);
// Funds are permanently destroyed; no refund callback exists in ExitToEthereum
```

**ETH base-token path (low-level):**

```solidity
// Encode flag 0x00 + 20 zero bytes as recipient
bytes memory input = abi.encodePacked(bytes1(0x00), bytes20(address(0)));
// Call ExitToEthereum precompile with ETH value
(bool ok,) = address(0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab)
    .call{value: 1 ether}(input);
// ok == true, ETH deducted from caller, withdrawal scheduled to address(0)
// No refund path; ETH is permanently lost
```

The root cause in both cases is the absence of a zero-address check in `ExitToEthereum::run()`: [6](#0-5) [7](#0-6)
