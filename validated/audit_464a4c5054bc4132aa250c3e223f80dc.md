### Title
`setPoolAdminFeeDestination` Redirects Accumulated Admin Fees Without Prior Collection, Causing Old Destination to Forfeit Owed Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees`. Every other fee-state-mutating function in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) collects accrued fees at the old rates/destination before making any change. The omission means all previously accumulated admin fees — both the spread surplus and the notional fee accumulators (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) — are silently redirected to the new destination on the next `collectPoolFees` call, and the old destination forfeits them.

---

### Finding Description

`setPoolAdminFees` explicitly collects fees before mutating fee state: [1](#0-0) 

`setPoolProtocolFee` does the same: [2](#0-1) 

`setPoolAdminFeeDestination` does not: [3](#0-2) 

`collectPoolFees` is permissionless and passes the current `poolAdminFeeDestination[pool]` directly to `collectFees`: [4](#0-3) 

Inside `collectFees` on the pool, the spread surplus (all tokens held above `binTotals`) and the notional accumulators are split and transferred to whichever `adminFeeDestination_` is passed in at call time: [5](#0-4) 

Because `setPoolAdminFeeDestination` writes the new address before any collection occurs, the entire backlog of earned admin fees is delivered to the new address on the very next `collectPoolFees` invocation.

---

### Impact Explanation

The old admin fee destination (which may be a separate treasury, DAO, or LP-revenue contract distinct from the admin key itself) loses all fees that accrued under its tenure. Both fee streams are affected:

- **Spread fees** — the entire token surplus above `binTotals.scaledToken0/1` is split by the current rates and sent to the new destination.
- **Notional fees** — `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` are zeroed and transferred to the new destination.

The loss is bounded only by how long fees have been accumulating since the last `collectPoolFees` call. For active pools with high volume and infrequent collection, this can represent a material amount of protocol-owed tokens.

---

### Likelihood Explanation

The pool admin is semi-trusted and can call `setPoolAdminFeeDestination` at any time with no timelock and no cap constraint. The trigger is a single admin transaction. The most realistic scenarios are:

1. **Treasury migration** — admin updates the fee destination to a new multisig/treasury without realising accumulated fees will follow.
2. **Admin role handover** — after `acceptPoolAdmin` transfers the admin role (which also does not collect fees), the new admin immediately calls `setPoolAdminFeeDestination` to point fees at their own address, capturing the old admin's accumulated balance. [6](#0-5) 

`acceptPoolAdmin` does not collect fees either, making the two-step handover a compounded exposure window.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: collect fees at the current destination before overwriting `poolAdminFeeDestination[pool]`.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Collect accrued fees to the OLD destination before redirecting
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

Apply the same fix to `acceptPoolAdmin` if the incoming admin is expected to inherit only future fees.

---

### Proof of Concept

1. Pool is active; significant spread surplus and notional accumulators have built up (e.g., `notionalFeeToken0Scaled = 1000e18`).
2. Admin calls `setPoolAdminFeeDestination(pool, newTreasury)`.
   - `poolAdminFeeDestination[pool]` is overwritten; no `collectFees` is called.
3. Anyone (keeper, bot, or the new admin) calls `collectPoolFees(pool)`.
   - `collectFees` is invoked with `adminFeeDestination_ = newTreasury`.
   - The entire spread surplus and notional accumulator are split and sent to `newTreasury`.
4. The old treasury receives **zero** tokens despite having earned fees during the pool's operation.

The exact corrupted value is `poolAdminFeeDestination[pool]` being updated before the accumulated `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` and spread surplus are flushed to the old address, causing a full forfeiture of owed admin fees to the old destination. [3](#0-2)
