### Title
Missing Zero-Address Validation on `reward_address` Enables Temporary Freezing of Staker and Pool Member Funds - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `staking.cairo::change_reward_address` and `pool.cairo::change_reward_address` accept any `ContractAddress` as the new reward destination, validating only that it is not a registered token address. There is no check that the supplied address is non-zero. Setting `reward_address` to `0` causes every subsequent call to `claim_rewards` and `unstake_action` to revert (because the underlying ERC-20 `checked_transfer` to address zero panics), temporarily freezing the staker's or pool member's funds until a corrective `change_reward_address` call is made.

### Impact Explanation

If `reward_address` is set to zero:

1. **`claim_rewards` reverts** — the staker (or pool member) cannot collect accrued yield.
2. **`unstake_action` reverts** — because it calls `send_rewards_to_staker` before returning the principal, the staker's entire staked principal is locked in the contract until they issue a corrective `change_reward_address` call.

This constitutes **temporary freezing of funds** (principal + unclaimed yield). Recovery is possible only by calling `change_reward_address` with a valid address, but the staker must first realize the mistake — which may not be immediate, especially since the contract continues to function normally for all other operations (attestation, delegation, etc.) after the bad address is set.

This maps to the allowed impact: **High — Temporary freezing of funds**.

---

### Likelihood Explanation

- Any active staker or pool member can trigger this unilaterally with a single transaction.
- Zero-address is a common accidental input (uninitialized variable, scripting error, copy-paste mistake).
- The contract emits a `StakerRewardAddressChanged` event but provides no on-chain warning that the new address is zero.
- The error only manifests at the next `claim_rewards` or `unstake_action` call, which may be much later, after significant rewards have accrued.

---

### Recommendation

Add an explicit non-zero check in all four entry points:

1. `staking.cairo::change_reward_address`
2. `pool.cairo::change_reward_address`
3. `staking.cairo::stake`
4. `pool.cairo::enter_delegation_pool`

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

Long-term, consider a two-step reward-address change (propose + accept) so that the new address must actively confirm ownership before the change takes effect, preventing both zero-address and wrong-address mistakes.

---

### Proof of Concept

1. Alice stakes via `staking.cairo::stake(reward_address: valid, ...)`.
2. Alice calls `staking.cairo::change_reward_address(reward_address: 0)` — succeeds, no revert.
3. Alice earns rewards over several epochs via attestation.
4. Alice calls `staking.cairo::claim_rewards(staker_address: alice)` — **reverts** because `checked_transfer(recipient: 0, ...)` panics in the STRK ERC-20.
5. Alice calls `staking.cairo::unstake_intent()` — succeeds.
6. After the exit window, Alice calls `staking.cairo::unstake_action(staker_address: alice)` — **reverts** because `send_rewards_to_staker` is called first and panics on the zero-address transfer.
7. Alice's principal remains locked in the staking contract until she calls `change_reward_address` with a valid address and retries `unstake_action`.
