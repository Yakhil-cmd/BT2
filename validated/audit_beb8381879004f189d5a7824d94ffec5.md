### Title
Zero `priceProviderTimelock` Bypasses Timelock Guard, Enabling Instant Price-Provider Swap by Pool Admin — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.createPool` stores `params.priceProviderTimelock` directly into `priceProviderTimelock[pool]` without validating it is non-zero for mutable-provider pools. Because `priceProviderTimelock[pool]` is never updated after creation, a pool deployed with `priceProviderTimelock = 0` permanently removes the only delay that separates a price-provider proposal from its execution. The pool admin can atomically propose and execute a malicious price provider in a single transaction, execute a swap at the manipulated price, and drain LP funds.

---

### Finding Description

`_validatePoolParameters` enforces many invariants (non-zero tokens, admin, fee destination, initial amounts, minimal mintable liquidity) but contains no guard against a zero timelock for mutable-provider pools:

```solidity
// MetricOmmPoolFactory.sol – _validatePoolParameters (lines 548-563)
function _validatePoolParameters(PoolParameters calldata params) internal view {
    if (params.token0 == address(0) || ...) revert InvalidTokenConfig();
    if (params.admin == address(0)) revert InvalidAdmin();
    _validatePriceProvider(params.token0, params.token1, params.priceProvider);
    if (params.adminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // ... fee cap checks, initial amount checks ...
    // ❌ NO check: params.priceProviderTimelock > 0 when not immutable
}
```

After creation the timelock is written once and never changed:

```solidity
// createPool (line 213)
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`proposePoolPriceProvider` computes `executeAfter = block.timestamp + timelock`. When `timelock = 0`, `executeAfter = block.timestamp`. The enforcement check in `executePoolPriceProviderUpdate` is:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
```

`block.timestamp < block.timestamp` is always `false`, so the guard never fires. Both `propose` and `execute` can be called sequentially in the same transaction through a custom contract, atomically replacing the price provider with no delay.

---

### Impact Explanation

The price provider is the sole source of bid/ask prices for every swap. `MetricOmmPool.swap` calls `_getBidAndAskPriceX64()` which delegates entirely to the active provider:

```solidity
// MetricOmmPool.sol (lines 804-813)
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) { revert PriceProviderFailed(reason); }
}
```

A malicious provider returning an extreme bid/ask causes the pool to execute swaps at prices far from fair value, transferring LP token balances to the attacker. This is a **direct loss of LP principal** — the pool's `binTotals` are permanently reduced by the manipulated swap.

---

### Likelihood Explanation

- Any pool created with `priceProviderTimelock = 0` and `priceProviderTimelock != type(uint256).max` (i.e., a mutable provider) is permanently vulnerable.
- The factory imposes no minimum timelock, so this is a valid deployment configuration.
- The pool admin is the only required actor; no external flash loan or oracle manipulation is needed.
- The attack is atomic (single transaction), leaving no window for LPs to react.

---

### Recommendation

Add a validation in `_validatePoolParameters` that rejects a zero timelock for mutable-provider pools:

```solidity
bool immutable_ = params.priceProviderTimelock == type(uint256).max;
if (!immutable_ && params.priceProviderTimelock == 0) {
    revert ZeroPriceProviderTimelock();
}
```

This mirrors the recommended fix from the external report (`require(_minLockPeriod > 0, 'zero minLockPeriod')`) applied to the analogous immutable-once parameter in this codebase.

---

### Proof of Concept

1. Pool admin deploys a pool via `createPool` with `params.priceProviderTimelock = 0` and a legitimate price provider. `_validatePoolParameters` passes without error.
2. LPs add liquidity to the pool.
3. Pool admin deploys `MaliciousPriceProvider` that returns `bid = 1, ask = 2` (extreme undervaluation of token1).
4. Pool admin calls `proposePoolPriceProvider(pool, maliciousProvider)`. `executeAfter = block.timestamp + 0 = block.timestamp`.
5. In the same transaction, pool admin calls `executePoolPriceProviderUpdate(pool)`. The check `block.timestamp < block.timestamp` is `false`, so it passes. `setPriceProvider(maliciousProvider)` is called on the pool.
6. Pool admin (or accomplice) calls `swap(zeroForOne=false, ...)`. `_getBidAndAskPriceX64()` returns the manipulated prices. The swap drains token0 from LP bins at a fraction of fair value.
7. Pool admin calls `proposePoolPriceProvider` + `executePoolPriceProviderUpdate` again to restore the original provider, concealing the manipulation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
