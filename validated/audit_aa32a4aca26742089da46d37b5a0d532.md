### Title
DSToken-Based `approve` Front-Running Enables Double-Spend of CRC20/CRC21 Allowances - (File: contracts/src/ModuleCRC20.sol, contracts/src/ModuleCRC21.sol)

### Summary
Both `ModuleCRC20` and `ModuleCRC21` — Cronos's auto-deployed contracts for IBC and gravity-bridge tokens — inherit from `DSToken`, whose `approve` function unconditionally overwrites the allowance. No `increaseAllowance` or `decreaseAllowance` exists anywhere in the codebase. A previously approved spender can front-run an allowance-change transaction and drain both the old and new allowance, resulting in an unauthorized transfer of CRC20/CRC21 assets.

### Finding Description
`ModuleCRC20` and `ModuleCRC21` both import and inherit `DSToken`:

```solidity
// contracts/src/ModuleCRC20.sol
import "ds-token/token.sol";
contract ModuleCRC20 is DSToken { ... }

// contracts/src/ModuleCRC21.sol
import "ds-token/token.sol";
contract ModuleCRC21 is DSToken { ... }
```

`DSToken`'s `approve` function (inherited, not overridden) directly sets `allowance[msg.sender][guy] = wad` without checking whether a prior non-zero allowance exists. The ABI for both contracts confirms the presence of `approve(address guy, uint256 wad)` and `transferFrom(address src, address dst, uint256 wad)` with no `increaseAllowance` or `decreaseAllowance` counterparts — confirmed by a full-codebase grep returning zero matches for those identifiers.

**Attack sequence:**

1. Alice calls `approve(Bob, 100)` on a `ModuleCRC20` or `ModuleCRC21` token. Bob now has allowance 100.
2. Alice later decides to reduce Bob's allowance and broadcasts `approve(Bob, 50)`.
3. Bob monitors the Cronos mempool. Before Alice's second `approve` is included, Bob submits `transferFrom(Alice, Bob, 100)` with a higher gas price.
4. Because Cronos (like all EVM chains) orders transactions by gas price, Bob's `transferFrom` executes first, draining the original 100 tokens.
5. Alice's `approve(Bob, 50)` then confirms, granting Bob a fresh allowance of 50.
6. Bob calls `transferFrom(Alice, Bob, 50)` again, stealing the second tranche.

Net result: Bob transfers 150 tokens when Alice intended him to have access to only 50.

### Impact Explanation
This is an unauthorized transfer of CRC20/CRC21 assets. `ModuleCRC20` and `ModuleCRC21` are the canonical auto-deployed token contracts for every IBC voucher and gravity-bridge token on Cronos. Any token holder who attempts to reduce a spender's allowance from a non-zero value to a lower non-zero value is vulnerable to having both the original and the new allowance drained. The stolen amount is bounded by `old_allowance + new_allowance`, which can be arbitrarily large depending on the token and the amounts involved.

This maps directly to the allowed Critical impact: **Unauthorized transfer of CRC20/CRC21 assets**.

### Likelihood Explanation
The precondition is that the victim must be changing an existing non-zero allowance to another non-zero value. This is a common real-world pattern (e.g., adjusting a DeFi protocol's spending limit). The attacker must be the previously approved spender and must monitor the mempool — both are realistic for a motivated adversary. The attack requires no special privileges, no key compromise, and no cryptographic assumptions.

### Recommendation
Add `increaseAllowance` and `decreaseAllowance` functions to both `ModuleCRC20` and `ModuleCRC21`, or override `approve` to require the current allowance to be zero before setting a new non-zero value (forcing a two-step reset through zero). The OpenZeppelin `ERC20` implementation's `increaseAllowance`/`decreaseAllowance` pattern is the standard mitigation.

```solidity
function increaseAllowance(address guy, uint256 addedValue) public returns (bool) {
    approve(guy, add(allowance[msg.sender][guy], addedValue));
    return true;
}

function decreaseAllowance(address guy, uint256 subtractedValue) public returns (bool) {
    approve(guy, sub(allowance[msg.sender][guy], subtractedValue));
    return true;
}
```

### Proof of Concept

**Setup:** Alice holds 200 tokens of a `ModuleCRC20` IBC voucher (e.g., `ibc/...` denom auto-deployed on Cronos). Bob is a DeFi contract or EOA.

1. Alice: `ModuleCRC20.approve(Bob, 100)` → `allowance[Alice][Bob] = 100`
2. Alice decides to lower Bob's allowance: broadcasts `ModuleCRC20.approve(Bob, 50)` with gas price G.
3. Bob sees the pending tx in the mempool. Bob submits `ModuleCRC20.transferFrom(Alice, Bob, 100)` with gas price G+1.
4. Block is produced. Bob's `transferFrom` executes first: Alice loses 100 tokens, Bob gains 100.
5. Alice's `approve(Bob, 50)` executes: `allowance[Alice][Bob] = 50`.
6. Bob submits `ModuleCRC20.transferFrom(Alice, Bob, 50)`: Alice loses 50 more tokens.

**Final state:** Bob has stolen 150 tokens; Alice intended to grant access to only 50. The same attack applies identically to `ModuleCRC21` tokens (IBC source tokens and gravity-bridge tokens). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
