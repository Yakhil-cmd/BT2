### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Suppress Block Rewards - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in the Staking contract is specified to be callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any non-zero address can call it with `disable_rewards: true`, consuming the single global `last_reward_block` slot for that block and permanently preventing reward distribution for all stakers in that block.

### Finding Description
The protocol specification at `docs/spec.md:1644-1645` explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` at `src/staking/staking.cairo:1449-1507` only calls `self.general_prerequisites()` (line 1452), which checks two things: the contract is not paused, and the caller is not the zero address. There is no check that the caller is the authorized sequencer.

The function signature accepts a caller-controlled `disable_rewards: bool` parameter:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
)
```

The critical execution path is:

1. **Line 1485**: `self.last_reward_block.write(current_block_number);` — the global `last_reward_block` is unconditionally updated to the current block, regardless of `disable_rewards`.
2. **Lines 1487–1488**: `if disable_rewards || self.is_pre_consensus() { return; }` — when `disable_rewards` is `true`, the function returns early without distributing any rewards.

`last_reward_block` is a single global storage variable (not per-staker), declared at `src/staking/staking.cairo:187`:

```rust
last_reward_block: BlockNumber,
```

The guard at lines 1454–1458 enforces that only one call per block succeeds:

```rust
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

**Attack path**: An attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)` in a block before the sequencer's legitimate reward-distribution call. This:
- Marks the block as "already updated" in `last_reward_block`
- Distributes zero rewards
- Causes the sequencer's subsequent call to revert with `REWARDS_ALREADY_UPDATED`

The rewards for that block are permanently lost — they are never credited to `unclaimed_rewards_own` of any staker, and never forwarded to pool contracts.

This is confirmed by the test at `src/staking/staking.cairo` (flow test lines 2882–2885):
```
// Disable rewards = true with consensus on - no rewards
system.update_rewards(:staker, disable_rewards: true);
let rewards = system.staker_claim_rewards(:staker);
assert!(rewards.is_zero());
```
The test demonstrates that calling with `disable_rewards: true` produces zero rewards, and a subsequent call in the same block panics with `REWARDS_ALREADY_UPDATED`.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

For every block in which the attacker front-runs the sequencer's `update_rewards` call, all stakers and pool members permanently lose the block rewards they would have earned. The rewards are never minted into `unclaimed_rewards_own` and never forwarded to delegation pools. The loss is irreversible — there is no mechanism to retroactively credit missed block rewards. Sustained over many blocks, this drains all consensus-era yield from the protocol.

### Likelihood Explanation
On Starknet, the sequencer controls transaction ordering, which limits classical mempool front-running. However:
1. The sequencer is not guaranteed to call `update_rewards` as the first transaction in every block.
2. Any user transaction included before the sequencer's reward call in a block exploits this.
3. The attacker only needs to pay gas for each griefing call — there is no capital requirement.
4. The attack is permissionless and requires no privileged role, leaked key, or external dependency.

The vulnerability is reachable by any non-zero address once consensus rewards are active (`!is_pre_consensus()`).

### Recommendation
Add an access control check to `update_rewards` restricting callers to the authorized sequencer address, consistent with the specification. For example, store the sequencer address in contract storage and assert it at the start of `update_rewards`:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
}
```

### Proof of Concept

1. Consensus rewards are active (`consensus_rewards_first_epoch` has passed).
2. A valid staker `S` exists with non-zero balance at the current epoch.
3. Attacker (any non-zero address `A`) calls:
   ```
   staking_contract.update_rewards(staker_address: S, disable_rewards: true)
   ```
4. Inside `update_rewards`:
   - `general_prerequisites()` passes (contract not paused, `A != 0`).
   - `internal_staker_info(S)` succeeds (staker exists).
   - `is_staker_active(S, curr_epoch)` returns `true`.
   - `staker_total_strk_balance.is_non_zero()` passes.
   - `last_reward_block` is written to `current_block_number` (line 1485).
   - `disable_rewards == true` → early return, zero rewards distributed (line 1487).
5. The sequencer's legitimate call to `update_rewards(S, disable_rewards: false)` in the same block hits:
   ```
   assert!(current_block_number > self.last_reward_block.read(), ...)
   ```
   and reverts with `REWARDS_ALREADY_UPDATED`.
