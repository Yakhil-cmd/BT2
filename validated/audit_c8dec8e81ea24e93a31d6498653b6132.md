### Title
EIP-7702 Authorization with `chain_id = 0` Enables Cross-Chain Replay of Delegation Signatures — (`engine-transactions/src/eip_7702.rs`)

### Summary

Aurora Engine's EIP-7702 implementation explicitly accepts authorization tuples whose `chain_id` field is zero, treating them as valid on any chain. An authorization signed with `chain_id = 0` on Ethereum mainnet (or any other EVM chain) becomes public once broadcast, and an attacker can replay the identical signature bytes on Aurora Engine to install an unexpected code delegation on the authority's Aurora account. If the delegated contract address resolves to malicious bytecode on Aurora, the authority's ETH balance can be drained.

### Finding Description

In `SignedTransaction7702::authorization_list()`, each `AuthorizationTuple` is validated with the following chain-ID check:

```rust
(auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
    && auth.parity <= U256::one()
``` [1](#0-0) 

When `auth.chain_id` is zero, the condition short-circuits to `true` regardless of which chain the transaction is being processed on. The authorization hash that is then verified is:

```
keccak256(0x05 || rlp([chain_id=0, address, nonce]))
``` [2](#0-1) 

Because `chain_id = 0` is encoded literally into the hash preimage, the resulting hash — and therefore the signature — is **identical across every EVM chain**. There is no Aurora-specific context (no NEAR account ID, no Aurora chain ID) bound into the authorization hash when `chain_id = 0`.

The code comment at line 169–172 acknowledges that the outer transaction's `chain_id` is validated separately by the engine's `submit_transaction`, but that validation only guards the *outer* EIP-7702 transaction, not the inner per-authority authorization tuples. [3](#0-2) 

The engine's `submit_transaction` validates the outer transaction chain ID: [4](#0-3) 

But this does not prevent a valid `chain_id = 0` authorization tuple from being replayed inside a new outer EIP-7702 transaction whose outer `chain_id` matches Aurora's chain ID.

**Attack path:**

1. Authority signs an EIP-7702 authorization with `chain_id = 0`, `address = ContractX`, `nonce = N` on Ethereum mainnet. The signed authorization is broadcast and becomes public.
2. `ContractX` exists on Aurora with different (attacker-controlled or malicious) bytecode.
3. The authority's Aurora account has nonce `N` (e.g., a fresh account with nonce 0).
4. Attacker constructs a new EIP-7702 transaction on Aurora with `chain_id = Aurora_chain_id`, embedding the replayed authorization tuple (same `chain_id=0`, `address`, `nonce`, `r`, `s`, `parity`).
5. `authorization_list()` accepts the tuple as valid (`chain_id.is_zero()` → `true`), recovers the authority's address via `ecrecover`, and marks `is_valid = true`.
6. The EVM installs the `EF0100 || ContractX` delegation designator on the authority's Aurora account.
7. Any subsequent call to the authority's account executes `ContractX`'s malicious code in the authority's context, draining its ETH balance.

### Impact Explanation

**Impact: High — Theft of user funds.**

The authority's ETH balance on Aurora is at risk. Once the delegation designator is installed, any caller can trigger the malicious delegated code in the authority's context. The authority's entire ETH balance can be transferred out. This is direct theft of at-rest funds.

### Likelihood Explanation

**Likelihood: Low.**

Three conditions must align: (a) the authority must have signed a `chain_id = 0` authorization on another chain, (b) the delegated contract address must exist on Aurora with exploitable bytecode, and (c) the authority's Aurora nonce must match the nonce in the authorization. Condition (c) is most easily satisfied for fresh Aurora accounts (nonce = 0), which is common for users who primarily operate on Ethereum. The authorization being public (on-chain on Ethereum) satisfies the "revealed hash" precondition automatically.

### Recommendation

When `auth.chain_id` is zero, additionally verify that the recovered authority address is the expected signer and consider whether Aurora Engine should reject `chain_id = 0` authorizations entirely, or bind an Aurora-specific domain separator into the authorization hash. At minimum, document clearly that `chain_id = 0` authorizations signed on any other chain are valid on Aurora, so users are aware of the cross-chain replay risk.

### Proof of Concept

```
1. On Ethereum mainnet, authority signs:
   auth = { chain_id: 0, address: ContractX, nonce: 0, parity, r, s }
   This is broadcast in an Ethereum EIP-7702 tx and becomes public.

2. On Aurora (chain_id = 1313161554), attacker submits:
   EIP-7702 tx {
     chain_id: 1313161554,   // passes outer chain_id check in engine.rs
     authorization_list: [
       { chain_id: 0, address: ContractX, nonce: 0, parity, r, s }
       // ^^^ identical bytes from step 1
     ]
   }

3. In authorization_list() [eip_7702.rs:181]:
   auth.chain_id.is_zero() == true  →  is_valid = true
   ecrecover recovers authority's address correctly (same hash, same sig)
   Authorization { authority: authority_addr, address: ContractX, is_valid: true }

4. EVM installs EF0100||ContractX on authority's Aurora account.
   ContractX on Aurora contains SELFDESTRUCT or ETH-transfer-to-attacker logic.

5. Attacker calls authority's Aurora account → malicious code executes in
   authority's context → authority's ETH balance is stolen.
``` [5](#0-4)

    pub fn authorization_list(&self) -> Result<Vec<Authorization>, Error> {
        if self.transaction.authorization_list.is_empty() {
            return Err(Error::EmptyAuthorizationList);
        }
        let current_tx_chain_id = U256::from(self.transaction.chain_id);
        let mut authorization_list = Vec::with_capacity(self.transaction.authorization_list.len());
        let mut rlp_stream = RlpStream::new();
        let mut message_bytes = vec![MAGIC; 1];
        // According to EIP-7702, we should validate each authorization. We shouldn't skip any of them.
        // And just put the `is_valid` flag to `false` if any of them is invalid. It's related to
        // gas calculation, as each `authorization_list` must be charged, even if it's invalid.
        for auth in &self.transaction.authorization_list {
            // According to EIP-7702 step 1. validation, we should verify it as
            // `chain_id = 0 || current_chain_id`.
            // AS `current_chain_id` we used `transaction.chain_id` as we will validate `chain_id` in
            // Engine `submit_transaction` method.

            // Step 2 - validation logic inside EVM itself.
            // Step 3. Checking: authority = ecrecover(keccak(MAGIC || rlp([chain_id, address, nonce])), y_parity, r, s])
            // Validate the signature, as in tests it is possible to have invalid signature values.
            // Value `v` shouldn't be greater than 1
            let mut is_valid = if auth.s > SECP256K1N_HALF {
                false
            } else {
                (auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
                    && auth.parity <= U256::one()
            };

            let auth_address = if is_valid {
                rlp_stream.begin_list(3);
                rlp_stream.append(&auth.chain_id);
                rlp_stream.append(&auth.address);
                rlp_stream.append(&auth.nonce);

                message_bytes.extend_from_slice(rlp_stream.as_raw());

                let signature_hash = aurora_engine_sdk::keccak(&message_bytes);
                // U256::as_u32() is safe because here we're sure that the parity <= 1.
                let v = u8::try_from(auth.parity.as_u32()).unwrap_or(u8::MAX);
                let auth_address = ecrecover(signature_hash, &super::vrs_to_arr(v, auth.r, auth.s))
                    .unwrap_or_else(|_| {
                        is_valid = false;
                        Address::default()
                    });

                message_bytes.truncate(1);
                rlp_stream.clear();

                auth_address
            } else {
                Address::default()
            };

            // Validations steps 2,4-9 0f EIP-7702 provided by EVM itself.
            authorization_list.push(Authorization {
                authority: auth_address.raw(),
                address: auth.address,
                nonce: auth.nonce,
                is_valid,
            });
        }
        Ok(authorization_list)
    }
