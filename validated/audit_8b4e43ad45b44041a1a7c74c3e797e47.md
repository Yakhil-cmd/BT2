### Title
Negative `delta_gas` Not Applied in `compute_gas_refund` Causes Permanent User Fund Loss — (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

ZKsync OS implements a double resource accounting model tracking both EVM gas (ergs) and native resource. After execution, `compute_gas_refund` computes `delta_gas = (native_used / native_per_gas) - gas_used`. When `delta_gas > 0`, it is added to `gas_used` so the user pays for native resource costs. When `delta_gas < 0` (native resource cheaper than EVM gas implies), the adjustment is silently dropped. The code itself marks this with `// TODO: return delta_gas to gas_used?`. As a result, `gas_used` is permanently inflated and the user's refund is smaller than it should be — the excess fee is irrecoverably transferred to the operator.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` performs the following asymmetric adjustment:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native cost > EVM gas cost → charge extra gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?   ← acknowledged omission
``` [1](#0-0) 

The design intent, documented in `docs/double_resource_accounting.md`, is:

> `deltaGas := (nativeUsed / nativePerGas) - gasUsed`
> If `deltaGas > 0`, we add it to `gasUsed` … Finally, any remaining gas left is refunded as usual. [2](#0-1) 

The documentation implies symmetry — "any remaining gas left is refunded as usual" — but the implementation only handles the positive branch. When `delta_gas < 0`, `gas_used` is not reduced, so the user is charged for EVM gas that was never backed by equivalent native resource consumption. The resulting inflated `gas_used` is then used directly to compute the operator payment and the user refund:

```rust
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
``` [3](#0-2) 

```rust
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
    ...
``` [4](#0-3) 

The excess `|delta_gas| × gas_price` is permanently transferred to the operator and is never returned to the user. There is no mechanism for the user to reclaim it — the transaction is already finalized.

This same `compute_gas_refund` function is called for both L2 transactions (ZK flow) and L1→L2 transactions: [5](#0-4) 

---

### Impact Explanation

Every transaction where `native_used / native_per_gas < gas_used` results in a permanent, irrecoverable financial loss for the user. The magnitude of the loss per transaction is:

```
loss = |delta_gas| × gas_price
     = (gas_used − native_used / native_per_gas) × gas_price
```

The excess is silently paid to the operator. The user has no way to detect this at submission time, no way to prevent it, and no way to reclaim it after the fact. This is a direct, quantifiable loss of user funds on every affected transaction.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever EVM gas consumption exceeds native resource consumption on a per-unit basis. This is a normal operating condition, not an edge case:

- `native_per_gas = gas_price / native_price`. When a user pays a higher gas price (e.g., to get faster inclusion), `native_per_gas` increases, making `native_used / native_per_gas` smaller and `delta_gas` more negative.
- EVM opcodes such as `SLOAD` (2100 gas cold) and `SSTORE` (20,000 gas) are expensive in EVM gas but their native resource cost may be proportionally lower, widening the gap.
- Any transaction that uses a priority fee above the base fee increases `gas_price`, directly increasing the magnitude of the loss.

The condition is reachable by any unprivileged transaction sender submitting a standard EVM transaction. No special access or governance role is required.

---

### Recommendation

Apply the negative `delta_gas` symmetrically, subject to the existing `minimal_gas_used` floor:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
``` [6](#0-5) 

This ensures the user is refunded for the portion of EVM gas that was not backed by native resource consumption, consistent with the documented intent that "any remaining gas left is refunded as usual."

### Proof of Concept

**Setup:**
- `gas_limit = 100,000`
- `gas_price = 1,000 wei`
- `native_price = 10` (operator-set)
- `native_per_gas = gas_price / native_price = 100`
- `native_limit = gas_limit × native_per_gas = 10,000,000`

**Execution result:**
- EVM gas consumed → `gas_used = 50,000` (from ergs remaining)
- Native resource consumed → `native_used = 2,000,000`

**`compute_gas_refund` behavior:**
```
delta_gas = (2,000,000 / 100) − 50,000 = 20,000 − 50,000 = −30,000
```
Since `delta_gas < 0`, the branch is skipped. `gas_used` stays at `50,000`.

**Actual outcome:**
- User refund: `(100,000 − 50,000) × 1,000 = 50,000,000 wei`
- Operator payment: `50,000 × 1,000 = 50,000,000 wei`

**Correct outcome (with fix):**
- `gas_used` should be `20,000`
- User refund: `(100,000 − 20,000) × 1,000 = 80,000,000 wei`
- Operator payment: `20,000 × 1,000 = 20,000,000 wei`

**User loss: `30,000 × 1,000 = 30,000,000 wei` per transaction, permanently transferred to the operator.**

The existing test `test_delta_gas` in `tests/instances/transactions/src/native_charging.rs` only tests the `delta_gas > 0` path (low `native_per_gas` ratio forcing the user to add priority fee). It does not cover the `delta_gas < 0` path where the user overpays. [7](#0-6)
