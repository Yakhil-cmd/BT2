### Title
Attacker-Triggered XCC Router Pre-Creation Causes Permanent wNEAR Loss for First-Time XCC Users — (`engine/src/contract_methods/xcc.rs`, `engine-precompiles/src/xcc.rs`, `engine/src/xcc.rs`)

---

### Summary

The `fund_xcc_sub_account` NEAR entry point has no access control when `wnear_account_id` is `None`, allowing any unprivileged NEAR account to create the XCC router sub-account for any victim EVM address. If an attacker does this after a victim's EVM XCC precompile call has already deducted wNEAR from the victim's EVM balance but before the engine's NEAR promise batch executes, the victim's `CreateAccount` batch action fails (account already exists), the entire promise chain aborts, and the victim's wNEAR is permanently stranded in the engine's implicit EVM address with no refund path.

---

### Finding Description

**Step 1 — wNEAR is deducted synchronously inside the EVM.**

In `engine-precompiles/src/xcc.rs`, when a user calls the XCC precompile for the first time (no router deployed), `required_near` is set to `attached_near + STORAGE_AMOUNT` (2 NEAR): [1](#0-0) 

A synchronous EVM-level `transferFrom` immediately moves that wNEAR from the caller's EVM balance to the engine's implicit address: [2](#0-1) 

The precompile then emits a log encoding the promise; the EVM transaction succeeds and the wNEAR deduction is final.

**Step 2 — The NEAR promise batch is scheduled asynchronously.**

After EVM execution, `handle_precompile_promise` in `engine/src/xcc.rs` schedules a NEAR batch receipt targeting `{hex_address}.aurora` with `CreateAccount`, `Transfer(STORAGE_AMOUNT)`, `DeployContract`, and `FunctionCall(initialize)`: [3](#0-2) 

This batch receipt is processed in a subsequent NEAR block — there is a window between the EVM execution and the batch execution.

**Step 3 — `fund_xcc_sub_account` is open to any caller.**

`engine/src/contract_methods/xcc.rs` only enforces owner-only access when `wnear_account_id` is `Some`. When it is `None`, any NEAR account may call it with any arbitrary `target` EVM address: [4](#0-3) 

The `FundXccArgs.target` field is a free `Address` parameter with no binding to the caller: [5](#0-4) 

**Step 4 — Attacker pre-creates the router; victim's batch fails.**

If the attacker submits `fund_xcc_sub_account(target=victim_address, wnear_account_id=None)` with ≥ 2 NEAR attached in the same NEAR block as the victim's EVM transaction, and the attacker's receipt is processed before the victim's promise batch receipt in the next block, the attacker's call successfully creates `{victim_address}.aurora` and calls `factory_update_address_version` to record the version in engine storage.

When the victim's promise batch then executes, `CreateAccount` fails because the sub-account already exists. The entire batch fails. The `factory_update_address_version` callback sees `promise_result_check() = Some(false)` and aborts: [6](#0-5) 

The downstream `withdraw_wnear_to_router` and `send_refund` calls are never reached. The victim's wNEAR — already transferred to the engine's implicit address in Step 1 — is stranded there with no automated refund path.

---

### Impact Explanation

The victim permanently loses `STORAGE_AMOUNT + attached_near` worth of wNEAR (minimum 2 NEAR per attack). The wNEAR sits in the engine's implicit EVM address, which has no self-service withdrawal mechanism for the victim. Recovery requires privileged admin intervention. This constitutes permanent freezing of user funds (Critical) or at minimum temporary freezing requiring admin action (High).

---

### Likelihood Explanation

- `fund_xcc_sub_account` is a public NEAR method callable by any account with no authentication when `wnear_account_id = None`.
- The victim's EVM address is public on-chain; an attacker can monitor for first-time XCC precompile calls.
- NEAR receipt ordering within a block is deterministic and observable; an attacker can craft their transaction to be processed before the victim's promise batch.
- The attacker's cost is ≥ 2 NEAR to fund the router, but the router is legitimately deployed for the victim's address, so the attacker's NEAR is not destroyed — it is locked in the router sub-account. The victim's wNEAR loss is the net damage.
- Likelihood: **Medium** — requires mempool monitoring and NEAR receipt ordering knowledge, but no privileged access.

---

### Recommendation

1. **Bind `fund_xcc_sub_account` to the target address's owner.** Require that the NEAR predecessor account is the implicit NEAR account corresponding to `args.target` (i.e., `{hex(args.target)}.aurora`), or require a signed proof from the EVM key controlling that address.
2. **Add a wNEAR refund callback on promise failure.** In `handle_precompile_promise`, attach a failure-path callback that returns the deducted wNEAR from the engine's implicit address back to the original EVM caller if the `CreateAccount` batch fails.
3. **Check account existence before deducting wNEAR.** In the XCC precompile, verify on-chain that the sub-account does not already exist before charging `STORAGE_AMOUNT`, or defer the wNEAR deduction to the NEAR promise layer where failure can be handled atomically.

---

### Proof of Concept

1. Victim (`0xVICTIM`) calls the XCC precompile for the first time. The precompile deducts `2 NEAR + call_near` of wNEAR from `0xVICTIM`'s EVM balance and transfers it to the engine's implicit address. A NEAR promise batch receipt is scheduled to create `{hex(0xVICTIM)}.aurora`.

2. Attacker observes the victim's EVM transaction on-chain and immediately submits:
   ```
   fund_xcc_sub_account({ target: 0xVICTIM, wnear_account_id: None })
   ```
   with 2 NEAR attached, targeting the Aurora engine contract.

3. In the next NEAR block, if the attacker's receipt is processed first, `{hex(0xVICTIM)}.aurora` is created and `factory_update_address_version` records the version.

4. The victim's promise batch then executes. `CreateAccount` fails (account exists). The batch aborts. `withdraw_wnear_to_router` and `send_refund` are never called.

5. The victim's `2 NEAR + call_near` of wNEAR remains in the engine's implicit EVM address. The victim has no mechanism to recover it. The attack cost to the attacker is near-zero (their 2 NEAR is locked in the router, not destroyed).

#[named]
pub fn fund_xcc_sub_account<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;

        let args: xcc::FundXccArgs = io.read_input_borsh()?;

        // If a specific wNEAR account is specified then this transaction must
        // come from a trusted user. The wNEAR account must be accurate for the
        // XCC sub-account to work properly.
        // This method can be public when `args.wnear_account_id.is_none()`
        // because then the Engine figures out the correct wNEAR account on
        // its own.
        if args.wnear_account_id.is_some() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        xcc::fund_xcc_sub_account(&io, handler, env, args)?;
        Ok(())
    })
