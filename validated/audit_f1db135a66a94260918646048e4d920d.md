### Title
Pool admin can bypass factory fee caps via uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
The factory enforces hard caps on all pool-admin fee parameters in `setPoolAdminFees`, but the parallel admin entry-point `setPoolBinAdditionalFees` forwards per-bin additional fees to the pool with **zero cap validation**. A pool admin can set `addFeeBuyE6` / `addFeeSellE6` to the full `uint16` range (65 535 = 6.5535 % in E6 units) regardless of the current `maxAdminSpreadFeeE6` value, breaking the invariant that the factory bounds all admin-controlled fee parameters.

### Finding Description

`setPoolAdminFees` correctly enforces caps:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`setPoolBinAdditionalFees` performs **no such check** before forwarding to the pool:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
```

These bin-level fees are then added directly on top of the oracle-derived base fee during every swap through that bin:

```solidity
// MetricOmmPool.sol ~L910 (swap path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

The factory owner can lower `maxAdminSpreadFeeE6` (e.g., to 1 000 = 0.1 %) via `setFeeCaps`. After that reduction, `setPoolAdminFees` correctly rejects any `adminSpreadFeeE6 > 1 000`. But `setPoolBinAdditionalFees` still accepts `addFeeBuyE6 = 65 535` (6.5535 %) for every bin, silently exceeding the cap the owner just enforced.

### Impact Explanation

Every swap routed through a bin with inflated additional fees pays more than the protocol-enforced maximum. The excess accrues as spread surplus and is collected by the admin via `collectFees`. This is a direct, per-swap extraction of user funds above the factory-enforced ceiling — an admin-boundary break with continuous fund loss proportional to swap volume through the affected bin.

### Likelihood Explanation

Medium. The trigger is a pool admin (semi-trusted, not fully privileged) calling a single permissioned function with no special preconditions. No flash loan, no oracle manipulation, and no recovery-mode equivalent is required. The impact is amplified whenever the factory owner has tightened `maxAdminSpreadFeeE6` below 65 535, which is the normal governance action for reducing protocol risk.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept

1. Factory owner calls `setFeeCaps(..., newMaxAdminSpreadFeeE6 = 1_000, ...)` to tighten the admin spread cap to 0.1 %.
2. Pool admin calls `setPoolAdminFees(pool, 1_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; bin 0 now carries an additional 6.5535 % buy/sell fee.
4. Any user swapping through bin 0 pays 6.5535 % extra on top of the oracle spread, far above the 0.1 % cap the owner intended.
5. The surplus accumulates and is drained by the admin via `collectPoolFees`. [1](#0-0) [2](#0-1) [3](#0-2)
