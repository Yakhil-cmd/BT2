### Title
Cross-Network Signature Replay in `EVM.validateCOAOwnershipProof` via Chain-ID-Free Domain Tag - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence account signatures over an attacker-supplied `signedData` blob using a hardcoded domain separation tag `"FLOW-V0.0-user"` that contains no chain ID or network identifier. A signature produced on one Flow network (e.g., testnet) is therefore cryptographically valid on any other Flow network (e.g., mainnet) for the same Cadence account keys. This enables cross-network replay of COA ownership proofs, which are consumed by the ERC-1271 `isValidSignature` entry point on every deployed COA contract.

### Finding Description
`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts caller-supplied `signedData`, builds a `Crypto.KeyList`, and calls `keyList.verify` with the fixed tag `"FLOW-V0.0-user"`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"   // ← no chain ID
)
``` [1](#0-0) 

The code's own comment acknowledges a related replay gap (cross-address replay) but does not address cross-network replay:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [2](#0-1) 

The `evmAddress` equality check that follows only confirms the COA resource address matches the on-chain capability; it does not bind the signature to any specific network. [3](#0-2) 

This function is the backend for the `verifyCOAOwnershipProof` Cadence Arch precompile, invoked via `coaOwnershipProofValidator` in `fvm/evm/handler/precompiles.go`: [4](#0-3) 

The precompile is in turn called by the COA Solidity contract's ERC-1271 `isValidSignature`:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig));
    ...
}
``` [5](#0-4) 

The `_hash` passed to `isValidSignature` becomes the `signedData` verified by `validateCOAOwnershipProof`. Because the domain tag `"FLOW-V0.0-user"` is identical on every Flow network (mainnet, testnet, emulator, sandboxnet — all defined in `model/flow/chain.go`), a signature over any `_hash` is valid on all networks simultaneously. [6](#0-5) 

### Impact Explanation
Any EVM smart contract on Flow EVM that uses ERC-1271 to authorize actions by a COA (e.g., a DEX order book, a lending protocol, an NFT marketplace) can be deceived into accepting a signature that was produced on a different Flow network. Concretely:

1. A victim signs `hash = keccak256(orderData)` on testnet to authorize a test trade.
2. An attacker submits an EVM transaction on mainnet to the same contract (deployed at the same address), supplying the victim's testnet signature as the COA ownership proof.
3. `isValidSignature` calls `verifyCOAOwnershipProof` → `validateCOAOwnershipProof`.
4. The Cadence account exists on mainnet with the same keys (common for users who mirror testnet setups), so the signature verifies under `"FLOW-V0.0-user"`.
5. The mainnet contract accepts the proof and executes the action on behalf of the victim's COA — unauthorized asset transfer, order execution, or privilege escalation depending on the consuming contract.

### Likelihood Explanation
- Flow users routinely create accounts with the same keys on testnet and mainnet for development/testing purposes.
- COA EVM addresses are deterministic per account storage slot; a user who creates their first COA on both networks may receive the same EVM address.
- The attacker needs only to observe a valid proof on one network (e.g., by monitoring the public Access API or mempool) and replay it on another — no privileged access is required.
- The attack surface grows with every EVM DeFi protocol that integrates ERC-1271 for COA-based authorization.

### Recommendation
Bind the domain separation tag to the specific Flow network by incorporating the chain ID. Replace the hardcoded `"FLOW-V0.0-user"` tag with a chain-scoped variant, for example:

```cadence
// Construct a chain-specific tag, e.g. "FLOW-V0.0-user-flow-mainnet"
let chainTag = "FLOW-V0.0-user-".concat(chainID)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: chainTag
)
```

Alternatively, mandate that `signedData` encodes both the `evmAddress` and the chain ID before signature verification, enforcing this at the protocol level rather than leaving it as a caller responsibility. This mirrors the EIP-712 domain separator pattern (`chainId` + `verifyingContract`).

### Proof of Concept
**Setup:** Alice has a Cadence account `0xALICE` with key index 0 on both testnet and mainnet. She deploys a COA on both networks. A DEX contract is deployed at the same address on both networks and uses `isValidSignature` to authorize limit orders.

**Step 1 – Obtain testnet signature:**
Alice signs `hash = keccak256(abi.encode("sell 100 FLOW at 1 USDC"))` on testnet with tag `"FLOW-V0.0-user"` to place a test order. The resulting `(sig, proof)` is broadcast in a testnet EVM transaction and is publicly observable.

**Step 2 – Replay on mainnet:**
Attacker constructs a mainnet EVM transaction calling the DEX's `fillOrder` function, which internally calls `coaAddress.isValidSignature(hash, encodedProof)`.

**Step 3 – Verification path:**
`isValidSignature` → `verifyCOAOwnershipProof` precompile [7](#0-6) 
→ `coaOwnershipProofValidator` [4](#0-3) 
→ `EVM.validateCOAOwnershipProof` with `signedData = hash`, `domainSeparationTag = "FLOW-V0.0-user"` [1](#0-0) 

**Step 4 – Result:**
Alice's mainnet account has the same key, so `keyList.verify` returns `true`. The DEX executes the order on mainnet, draining Alice's mainnet COA balance without her authorization.
