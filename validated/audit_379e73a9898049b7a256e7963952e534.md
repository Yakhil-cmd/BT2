### Title
Unrestricted `update_rewards` Allows Any Caller to Permanently Freeze Consensus Staking Rewards via `disable_rewards=true` - (File: src/staking/staking.cairo)

---

### Summary
The `update_rewards` function in the Staking contract carries no access-control check. It enforces at most one execution per block through a global `last_reward_block` variable. Any unprivileged caller can invoke `update_rewards(any_active_staker, disable_rewards: true)` in every block, consuming the block's single reward-distribution slot without distributing any rewards, permanently preventing all stakers from receiving consensus rewards.

---

### Finding Description
`update_rewards` (consensus-rewards path) is the sole mechanism for distributing per-block staking rewards once the consensus epoch is active. Its guard against double-execution is a global storage variable `last_reward_block`:

```cairo
// src/staking/staking.cairo  lines 1449-1507
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // slot consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // exits without distributing
    }
    ...
}
``` [1](#0-0) 

There is no role check — `general_prerequisites()` only asserts the contract is not paused. The `last_reward_block` write happens **before** the `disable_rewards` branch, so calling the function with `disable_rewards=true` permanently consumes the block's reward slot while distributing nothing.

Because `last_reward_block` is a **global** (not per-staker) variable, a single call per block locks out every other caller for that block. Any active staker address is publicly observable from on-chain `Staked` events.

---

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

Stakers are entitled to per-block consensus rewards. If an attacker calls `update_rewards(any_active_staker, disable_rewards: true)` in every block after `consensus_rewards_first_epoch`, the `last_reward_block` slot is consumed with no reward distribution. There is no retroactive recovery mechanism: rewards for a missed block are permanently lost. Continuous execution of this attack means stakers receive zero consensus rewards indefinitely. [2](#0-1) 

---

### Likelihood Explanation
**Medium.** The attacker must supply a valid active staker address (trivially obtained from public events) and pay gas for one call per block. The cost is non-trivial but well within reach of a motivated adversary (e.g., a competing validator or protocol saboteur). No privileged key, bridge access, or external dependency is required.

---

### Recommendation
1. **Restrict `disable_rewards=true` to a privileged role** (e.g., `only_app_governor` or `only_security_agent`), or
2. **Do not write `last_reward_block` when `disable_rewards=true`**, so the slot remains available for a legitimate caller in the same block, or
3. **Add an explicit caller whitelist** (e.g., only the staker themselves or a designated rewards-manager address may call `update_rewards`). [3](#0-2) 

---

### Proof of Concept

1. Consensus rewards become active (current epoch ≥ `consensus_rewards_first_epoch`).
2. Attacker reads any active staker address `S` from on-chain events.
3. In every block `N`, attacker calls:
   ```
   update_rewards(staker_address: S, disable_rewards: true)
   ```
4. Inside the call: `last_reward_block` is written to `N`; the function returns early — no rewards distributed.
5. Any subsequent legitimate call to `update_rewards` in block `N` hits:
   ```
   assert!(current_block_number > self.last_reward_block.read(), REWARDS_ALREADY_UPDATED)
   ```
   and reverts.
6. Stakers receive zero consensus rewards for block `N`.
7. Repeating steps 3–6 every block → all stakers permanently receive zero consensus rewards. [4](#0-3) [5](#0-4)
