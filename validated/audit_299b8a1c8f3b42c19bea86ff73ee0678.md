### Title
Signature Validation Bypass via `skip_stateful_validations` for Factory-Deployed Accounts — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call for an invoke transaction when the account's on-chain nonce is 0 and the account has *any* transaction in the mempool. The guard is intended exclusively for the deploy-account + invoke UX pattern, but the mempool check `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type, not only `DeployAccount`. For accounts deployed via a factory contract (on-chain nonce = 0, contract already exists), a legitimate invoke with nonce 0 can already be in the mempool. An attacker who observes that transaction can immediately submit a second invoke with nonce 1 and an **invalid signature**; the gateway skips `__validate__`, admits the transaction, and forwards it to the mempool. The invalid transaction is later included in a block, fails `__validate__` during execution, and drains the victim account's balance through the charged validation fee.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all four conditions hold:

1. Transaction type is `Invoke`
2. Transaction nonce = 1
3. Account on-chain nonce = 0
4. `account_tx_in_pool_or_recent_block(sender)` returns `true`

Condition 4 is implemented as: [2](#0-1) 

This returns `true` for **any** transaction type from that address — not only `DeployAccount`.

**Why the comment's reasoning is wrong for factory-deployed accounts**

The code comment at line 440–443 states: *"it is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning fails for accounts deployed via the `deploy` syscall (factory pattern). Such accounts have on-chain nonce = 0 but the contract already exists. A legitimate invoke with nonce 0 for such an account passes `__validate__` (the contract exists) and enters the mempool. Once that transaction is in the mempool, condition 4 is satisfied for any subsequent invoke with nonce 1 — including one with a forged signature.

**Effect on the validation pipeline**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the early-return path is taken and `__validate__` is never called: [4](#0-3) 

The transaction therefore bypasses signature verification at the gateway and is forwarded to the mempool.

**Mempool's `validate_tx` does not check signatures**

`validate_by_mempool` only checks nonce ordering and duplicate detection: [5](#0-4) [6](#0-5) 

No signature or `__validate__` check is performed here.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

The gateway admits an invoke transaction with an invalid (attacker-forged) signature. During block execution the batcher calls `__validate__` with `validate = true`; the transaction fails, the nonce is incremented, and the validation fee is charged from the victim account's balance. The attacker pays nothing (they only need to submit a transaction for someone else's address). The victim loses one validation fee per exploit window.

---

### Likelihood Explanation

Factory-deployed accounts (deployed via `deploy` syscall from a factory contract) are common in Starknet (e.g., account-factory patterns, multi-sig deployers). Any such account with a pending transaction in the mempool is a valid target. The mempool is observable by any node participant. The attack requires no privileged access and no on-chain funds from the attacker.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction is pending for the sender address. Alternatively, add a check that the account contract does not yet exist on-chain (class hash at the address is zero) before skipping validation — this is the true precondition the UX feature requires:

```rust
// Only skip validation if the account contract does not yet exist on-chain.
// This is the correct invariant: the account is being deployed for the first time.
let class_hash = state.get_class_hash_at(tx.sender_address())?;
if class_hash != ClassHash::default() {
    return Ok(false); // account already exists, do not skip
}
```

### Proof of Concept

1. Deploy contract `A` via a factory contract. `A` has on-chain nonce = 0 and a valid `__validate__` implementation.
2. The legitimate owner of `A` submits `invoke(sender=A, nonce=0, valid_sig)`. This passes `__validate__` at the gateway and enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits `invoke(sender=A, nonce=1, forged_sig)` to the gateway.
4. Gateway evaluates `skip_stateful_validations`: nonce=1 ✓, account_nonce=0 ✓, `account_tx_in_pool_or_recent_block(A)=true` ✓ → returns `true`.
5. `run_validate_entry_point` sets `execution_flags.validate = false`; `__validate__` is never called. The transaction passes all gateway checks and is forwarded to the mempool.
6. Batcher selects both transactions for a block. `invoke(nonce=0)` executes normally. `invoke(nonce=1)` runs `__validate__` with `validate=true`, fails (invalid signature), nonce is incremented to 1, and the validation fee is deducted from `A`'s balance.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
