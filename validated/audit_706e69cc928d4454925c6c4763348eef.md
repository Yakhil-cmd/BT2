### Title
`starknet_call` Always Executes Under Zeroed Deprecated v0 Transaction Semantics, Returning Wrong `get_execution_info` Data for Any Contract — (`crates/apollo_rpc_execution/src/lib.rs`)

---

### Summary

`execute_call`, which backs the `starknet_call` RPC endpoint, unconditionally constructs a `TransactionInfo::Deprecated(DeprecatedTransactionInfo::default())` for every invocation regardless of the target contract's version. Any Cairo 1 contract that reads its own transaction context via the `get_execution_info` syscall will receive all-zero deprecated v0 fields instead of the correct v3 fields, causing `starknet_call` to return an authoritative-looking but concretely wrong simulation result.

---

### Finding Description

In `execute_call`:

```rust
// TODO(yair): fix when supporting v3 transactions
let tx_info = TransactionInfo::Deprecated(DeprecatedTransactionInfo::default());
``` [1](#0-0) 

This `tx_info` is immediately wrapped in a `TransactionContext` and passed into `EntryPointExecutionContext::new_invoke`:

```rust
let mut context = EntryPointExecutionContext::new_invoke(
    Arc::new(TransactionContext { block_context: Arc::new(block_context), tx_info }),
    ...
);
``` [2](#0-1) 

`DeprecatedTransactionInfo::default()` derives `Default`, so every field is zeroed:

```rust
#[derive(Clone, Debug, Default, Eq, PartialEq)]
pub struct DeprecatedTransactionInfo {
    pub common_fields: CommonAccountFields,  // all zero
    pub max_fee: Fee,                        // Fee(0)
}
``` [3](#0-2) 

`CommonAccountFields::default()` yields `version = TransactionVersion(0)`, `sender_address = ContractAddress(0)`, `transaction_hash = TransactionHash(0)`, `nonce = Nonce(0)`, `signature = []`. [4](#0-3) 

When the blockifier executes the entry point and the contract issues a `get_execution_info` syscall, the syscall handler reads `tx_context.tx_info` directly. For a Cairo 1 contract using `get_execution_info_v2` or `get_execution_info_v3`, the returned `TxInfo` struct will contain:

| Field | Expected (v3) | Actual (hardcoded default) |
|---|---|---|
| `version` | `3` | `0` |
| `account_contract_address` | caller | `0` |
| `transaction_hash` | real hash | `0` |
| `nonce` | real nonce | `0` |
| `resource_bounds` | L1/L2 gas bounds | absent (deprecated variant) |
| `tip` | real tip | absent |
| `max_fee` | `0` (v3 field) | `0` (but wrong variant) |

The `gas_mode()` method on `TransactionInfo` also returns `GasVectorComputationMode::NoL2Gas` for the `Deprecated` variant, affecting gas accounting within the simulated call: [5](#0-4) 

---

### Impact Explanation

`starknet_call` is the primary RPC endpoint dApps use to simulate contract reads and off-chain view logic before submitting transactions. Any contract that branches on `tx_info.version`, checks `account_contract_address`, reads `resource_bounds`, or inspects `transaction_hash` / `nonce` via `get_execution_info` will receive entirely wrong data. The simulation result returned to the caller is concretely corrupted and authoritative-looking — there is no error, no revert, and no indication that the transaction context is synthetic. This maps directly to: **High — RPC execution/simulation returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

This affects every single invocation of `starknet_call` against any contract that uses `get_execution_info`. No special attacker capability is required — any unprivileged user calling the public RPC endpoint triggers it. The TODO comment confirms the code path is intentionally incomplete, not a transient bug.

---

### Recommendation

Replace the hardcoded default with a properly constructed `TransactionInfo::Current(CurrentTransactionInfo::create_default_unlimited())` (already available in the codebase for testing purposes) or derive a synthetic v3 `TransactionInfo` from the block context's fee token and gas price data. At minimum, the `version` and `sender_address` fields must be non-zero and semantically correct for the call to simulate faithfully.

---

### Proof of Concept

A Rust integration test against `execute_call` targeting a Cairo 1 contract that calls `get_execution_info_v3` and asserts on the returned `tx_info.version` field:

```rust
// In crates/apollo_rpc_execution/src/execution_test.rs
#[test]
fn test_execute_call_uses_deprecated_tx_info_for_cairo1_contract() {
    // Deploy a Cairo 1 contract that calls get_execution_info_v3
    // and returns tx_info.version as its retdata.
    let result = execute_call(
        storage_reader,
        None,
        &chain_id,
        state_number,
        block_number,
        &cairo1_contract_address,
        get_execution_info_selector,
        Calldata::default(),
        &ExecutionConfig::default(),
        false,
        None,
    ).unwrap();

    // The contract returns tx_info.version from get_execution_info_v3.
    // Expected: Felt::from(3u64)  (v3 semantics)
    // Actual:   Felt::from(0u64)  (DeprecatedTransactionInfo::default().version == 0)
    assert_eq!(result.retdata.0[0], Felt::from(3u64)); // FAILS: returns 0
}
```

The test will fail because `execute_call` hardcodes `TransactionInfo::Deprecated(DeprecatedTransactionInfo::default())`, causing `get_execution_info` to report `version = 0` instead of `3`. [6](#0-5)
