Audit Report

## Title
SNS Governance `perform_execute_generic_nervous_system_function` Lacks Re-entrancy Guard During Inter-Canister Proposal Execution - (File: `rs/sns/governance/src/canister_control.rs`)

## Summary

`perform_execute_generic_nervous_system_function_call` makes an unguarded `await` to an arbitrary community-registered external canister with no in-progress lock. While governance is suspended at that await point, the IC runtime can deliver and fully process any other ingress or inter-canister message — including a `manage_neuron / RegisterVote` call issued by the target canister itself. This allows the target canister to trigger execution of a second proposal (targeting the same canister) before the first proposal's execution has completed, violating the intended ordering and enabling double-execution of governance-authorized actions.

## Finding Description

**Root cause — no re-entrancy guard:**

`perform_execute_generic_nervous_system_function_call` in `rs/sns/governance/src/canister_control.rs` makes an unguarded `await` to an arbitrary external canister:

```rust
let result = env
    .call_canister(
        valid_function.target_canister_id,
        &valid_function.target_method,
        call.payload,
    )
    .await;   // governance is suspended here; other messages are processed
``` [1](#0-0) 

Its caller, `perform_execute_generic_nervous_system_function`, sets no in-progress flag and acquires no lock before delegating: [2](#0-1) 

No per-function-id or global execution lock exists for `ExecuteGenericNervousSystemFunction`. The `upgrade_in_progress` and `finalize_swap_in_progress` guards that exist elsewhere in the codebase are scoped to upgrade and swap operations only — they do not cover generic function execution.

**Re-entrancy path:**

While governance is suspended at the `await`, the IC runtime delivers and processes any queued message. The public `#[update]` endpoint `manage_neuron` is reachable by any canister: [3](#0-2) 

After casting a vote, `register_vote` synchronously calls `process_proposal`: [4](#0-3) 

`process_proposal` checks only whether the proposal's status is `Open` and whether quorum is reached. It does not check whether any other proposal is currently mid-execution: [5](#0-4) 

If quorum is reached, `start_proposal_execution` is called, which uses `unsafe { std::mem::transmute(self) }` to obtain a `'static` reference and spawns a new async task: [6](#0-5) 

The spawned future calls `perform_action` → `perform_execute_generic_nervous_system_function` → `perform_execute_generic_nervous_system_function_call`, issuing a second inter-canister call to the same target canister before the first has returned: [7](#0-6) 

**Existing checks are insufficient:**

The `process_proposal` guard at L1961 only prevents re-execution of the *same* proposal (it checks `status() != Open`). A *different* proposal B that is still `Open` passes this check freely. There is no cross-proposal execution lock.

## Impact Explanation

This is a **High** severity finding. A malicious or compromised target canister can:

1. **Violate execution ordering**: Proposal B executes and completes *inside* proposal A's execution window. If A sets up state that B depends on, B observes stale state.
2. **Double-execute governance-authorized actions**: The target canister receives two governance-authorized calls in a single round-trip, in an order it fully controls. Any side-effect performed on each call (token minting, permission grants, state transitions) is doubled.
3. **Exploit intermediate governance state**: The target canister can observe governance state between the two calls and craft its second response to exploit the intermediate state.

This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* (High, $2,000–$10,000). It does not reach Critical because the financial impact is bounded by what the registered `GenericNervousSystemFunction` is authorized to do, and exploitation requires meaningful preconditions.

## Likelihood Explanation

The attack requires the attacker to control:
1. A canister registered as a `NervousSystemFunction` target.
2. A neuron whose vote is the deciding one for a second pending proposal.

Both conditions are realistic in SNS DAOs: dapp canisters commonly hold developer neurons at SNS launch, and proposals can be pre-voted to threshold minus one. No privileged role, key compromise, or subnet-majority is required. The attacker-controlled entry path is a standard inter-canister `manage_neuron` call issued by the target canister during its own execution of proposal A's call.

## Recommendation

1. **Add a per-function-id execution lock**: Before the `await` in `perform_execute_generic_nervous_system_function`, insert the `function_id` into a `BTreeSet<u64>` stored in governance state. Remove it after the `await` completes (success or failure). Reject any concurrent `ExecuteGenericNervousSystemFunction` call for the same `function_id` with `ErrorType::ResourceExhausted`.

2. **Alternatively, add a global execution-in-progress flag** (analogous to the existing `upgrade_in_progress` flag) that blocks new `ExecuteGenericNervousSystemFunction` executions while any such execution is awaiting a reply.

3. **At minimum, document** that target canisters must not call back into governance's `manage_neuron` during execution, and that SNS DAOs should not register canisters holding voting neurons as `GenericNervousSystemFunction` targets.

## Proof of Concept

**Setup:**
- SNS has `NervousSystemFunction` id=1000 targeting canister X.
- Canister X controls neuron N with enough voting power to be the deciding vote.
- Proposal A: `ExecuteGenericNervousSystemFunction { function_id: 1000, payload: b"action_a" }` — adopted, awaiting execution.
- Proposal B: `ExecuteGenericNervousSystemFunction { function_id: 1000, payload: b"action_b" }` — open, all neurons except N have voted Yes.

**Steps:**
1. Governance heartbeat triggers `process_proposals` → proposal A adopted → `start_proposal_execution(A)` → `perform_execute_generic_nervous_system_function` → `env.call_canister(X, "execute", b"action_a")` → governance suspends at `await`.
2. Canister X's `execute` method receives `b"action_a"`. Before replying, it issues an inter-canister call to governance: `manage_neuron { subaccount: N.id, command: RegisterVote { proposal: B, vote: Yes } }`.
3. Governance processes the `manage_neuron` message: `register_vote` casts N's vote on B → `process_proposal(B)` → B reaches quorum → `start_proposal_execution(B)` → `perform_execute_generic_nervous_system_function` → `env.call_canister(X, "execute", b"action_b")` → governance suspends again.
4. Canister X's `execute` method receives `b"action_b"` and replies immediately.
5. Governance resumes proposal B's execution → B marked `Executed`.
6. Governance's `manage_neuron` reply returns to canister X.
7. Canister X replies to governance's original call for proposal A.
8. Governance resumes proposal A's execution → A marked `Executed`.

**Result:** Canister X received `b"action_b"` before `b"action_a"` completed. The intended execution order is violated, and canister X had full control over the interleaving.

A deterministic integration test using PocketIC can reproduce this by: (a) setting up the described SNS state, (b) implementing canister X to issue the `manage_neuron` callback before replying, and (c) asserting that proposal B's `executed_timestamp_seconds` is set before proposal A's.

    fn start_proposal_execution(&mut self, proposal_id: u64, action: Action) {
        // `perform_action` is an async method of &mut self.
        //
        // Starting it and letting it run in the background requires knowing that
        // the `self` reference will last until the future has completed.
        //
        // The compiler cannot know that, but this is actually true:
        //
        // - in unit tests, all futures are immediately ready, because no real async
        //   call is made. In this case, the transmutation to a static ref is abusive,
        //   but it's still ok since the future will immediately resolve.
        //
        // - in prod, "self" is a reference to the GOVERNANCE static variable, which is
        //   initialized only once (in canister_init or canister_post_upgrade)
        let governance: &'static mut Governance = unsafe { std::mem::transmute(self) };
        spawn_in_canister_env(governance.perform_action(proposal_id, action));
    }
