The code confirms the vulnerability is real. `proposePoolPriceProvider` has **no guard** checking whether a pending proposal already exists before overwriting both `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]`.

---

### Title
Pool admin can silently overwrite a pending price-provider proposal, resetting the timelock and indefinitely deferring or redirecting oracle rotation — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary
`proposePoolPriceProvider` unconditionally overwrites `pendingPriceProvider[pool]` and resets `pendingPriceProviderExecuteAfter[pool]` to `block.timestamp + timelock` with no check for an already-pending proposal. A pool admin can call it again at any time — including one second before the original timelock elapses — to silently replace the pending oracle and restart the clock, making the timelock protection meaningless.

---

### Finding Description
In `MetricOmmPoolFactory.proposePoolPriceProvider`: [1](#0-0) 

There is no check of the form `if (pendingPriceProvider[pool] != address(0)) revert PendingProviderExists()`. The function unconditionally writes:

```solidity
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
```

Because there is also no `cancelPoolPriceProviderProposal` function, re-proposing is the only mechanism available to the admin to change a pending entry — but it silently resets the clock rather than requiring the original delay to elapse first. [2](#0-1) 

---

### Impact Explanation
The timelock is the sole on-chain protection constraining the pool admin's oracle rotation power. It is supposed to give LPs and traders a guaranteed observation window before a new price provider takes effect. Two concrete harms follow from the bypass:

1. **Indefinite deferral**: The admin re-proposes the same (or any valid) provider one block before `executeAfter`, resetting the clock to `now + T`. Repeated indefinitely, `executePoolPriceProviderUpdate` can never be called successfully, permanently locking the oracle rotation mechanism.

2. **Silent provider substitution**: The admin proposes provider A (publicly announced), waits until `T-1`, then proposes provider B (a different oracle). LPs monitoring the event log for provider A see a new `PoolPriceProviderChangeProposed` event for B, but the original timelock window for A is gone. If provider B has different pricing characteristics, swaps executed after `executePoolPriceProviderUpdate` run against an oracle that users had no effective advance notice of, satisfying the "bad-price execution" impact gate.

This is an explicit admin-boundary break: the timelock is the cap on the pool admin's oracle rotation authority, and it is bypassed entirely. [3](#0-2) 

---

### Likelihood Explanation
The pool admin is a semi-trusted role. The contest scope explicitly states the admin is trusted *only inside caps and timelocks* and directs auditors to look for bypasses in provider proposals. The bypass requires only repeated calls to a function the admin already controls — no external dependency, no special token behavior, no off-chain oracle manipulation.

---

### Recommendation
Add a guard at the top of `proposePoolPriceProvider` that reverts if a pending proposal already exists:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PendingProviderExists();
```

Alternatively, add a separate `cancelPoolPriceProviderProposal` function that explicitly clears the pending entry (with an event), so replacement requires a deliberate two-step action that is visible on-chain and cannot silently reset the clock.

---

### Proof of Concept

```solidity
// 1. Admin proposes provider A; timelock T = 2 days
factory.proposePoolPriceProvider(pool, providerA);
// pendingPriceProvider[pool] == providerA
// pendingPriceProviderExecuteAfter[pool] == block.timestamp + 2 days

// 2. Warp to T-1 (one second before timelock elapses)
vm.warp(block.timestamp + 2 days - 1);

// 3. Admin proposes provider B — no revert, clock resets
factory.proposePoolPriceProvider(pool, providerB);
// pendingPriceProvider[pool] == providerB  (A silently dropped)
// pendingPriceProviderExecuteAfter[pool] == block.timestamp + 2 days (future again)

// 4. executePoolPriceProviderUpdate now reverts — timelock not elapsed
vm.expectRevert(PriceProviderTimelockNotElapsed.selector);
factory.executePoolPriceProviderUpdate(pool);

// 5. Admin can repeat step 3 indefinitely, making the timelock unenforceable
``` [4](#0-3)
