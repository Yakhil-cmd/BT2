### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Both `Staking::change_reward_address` and `Pool::change_reward_address` accept a `reward_address` parameter and persist it without checking `reward_address != 0`. The same gap exists in `Staking::stake` and `Pool::enter_delegation_pool`. Once a zero reward address is stored, every subsequent call to `claim_rewards` will attempt `checked_transfer(recipient: 0x0, ...)`, which reverts under the OpenZeppelin ERC20 implementation used for STRK, permanently freezing all accumulated yield for that staker or pool member.

---

### Finding Description

**`Staking::stake` and `Staking::change_reward_address`** — the only guard on `reward_address` is:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [1](#0-0) [2](#0-1) 

`address(0)` is not a registered token, so `does_token_exist(0)` returns `false` and the assertion passes. The zero address is then written to storage as the staker's `reward_address`.

**`Pool::enter_delegation_pool` and `Pool::change_reward_address`** — the only guard is:

```cairo
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [3](#0-2) [4](#0-3) 

Again, `address(0)` is not the token address, so the check passes and zero is stored.

**Reward disbursement paths that will revert:**

`Staking::send_rewards_to_staker` (called by both `claim_rewards` and `unstake_action`):

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [5](#0-4) 

`Pool::claim_rewards`:

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [6](#0-5) 

The OpenZeppelin ERC20 `transfer` implementation on Starknet panics when `recipient == 0`. Because `reward_address` is read directly from storage with no downstream zero-check, every call to `claim_rewards` (and `unstake_action` for stakers) will revert once a zero reward address is stored.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once `reward_address` is set to zero:
- `Staking::claim_rewards` reverts on every call for that staker.
- `Staking::unstake_action` also calls `send_rewards_to_staker` before returning principal, so the staker cannot exit the protocol and retrieve their staked tokens either — compounding the freeze to include principal as well.
- `Pool::claim_rewards` reverts on every call for that pool member.

Accumulated rewards are permanently locked in the reward supplier / staking contract with no recovery path, because `change_reward_address` requires the caller to be the existing staker/pool member — if that account is compromised or the user has lost access, there is no admin override.

---

### Likelihood Explanation

**Low-to-Medium.** The action is reachable by any unprivileged staker or pool member with a single public call. Realistic triggers include:

1. **User error**: a developer or UI bug passes `0` as `reward_address` during `stake()` or `enter_delegation_pool()`.
2. **Deliberate self-harm / griefing**: a user intentionally calls `change_reward_address(0)` to permanently freeze their own yield (e.g., to grief a protocol metric or as part of a social-engineering scenario).
3. **Scripted migration**: an automated script that "clears" the reward address before setting a new one, passing zero as an intermediate value.

No privileged access, bridge compromise, or external dependency is required.

---

### Recommendation

Add an explicit non-zero assertion at the top of all four entry points:

```cairo
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);
```

Apply this to:
- `Staking::stake` — immediately after the existing token-address check [1](#0-0) 
- `Staking::change_reward_address` — immediately after the existing token-address check [2](#0-1) 
- `Pool::enter_delegation_pool` — immediately after the existing token-address check [3](#0-2) 
- `Pool::change_reward_address` — immediately after the existing token-address check [4](#0-3) 

---

### Proof of Concept

1. A staker calls `Staking::change_reward_address(reward_address: 0)`.
2. The only guard — `does_token_exist(0)` — returns `false`; the assertion passes.
3. `staker_info.reward_address` is written as `0` to storage. [7](#0-6) 
4. The staker (or anyone) calls `Staking::claim_rewards(staker_address)`.
5. `send_rewards_to_staker` reads `reward_address = 0` and calls `checked_transfer(recipient: 0, amount: ...)`. [8](#0-7) 
6. The OZ ERC20 panics on `recipient == 0`; the transaction reverts.
7. The staker's accumulated rewards are permanently unclaimable. Because `unstake_action` also calls `send_rewards_to_staker` before returning principal, the staker's principal is also frozen. [9](#0-8) 

The identical path applies to pool members via `Pool::change_reward_address(0)` → `Pool::claim_rewards`. [10](#0-9) [11](#0-10)
