### Title
Fatal `LeafDefect` Error Triggered by Malformed `InteropRootAdded` Event Data — (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

`interop_root_reporter_event_hook` returns a `SystemError::LeafDefect` (fatal internal error) when the ABI-encoded event data length is not exactly 96 bytes. Because the hook filters events by topic signature rather than by emitting contract address, any unprivileged contract can emit an `InteropRootAdded`-shaped event with malformed data to trigger this fatal error path, analogous to an unexpected revert from `abi.decode` on malformed input.

---

### Finding Description

`interop_root_reporter_event_hook` is an event hook that processes `InteropRootAdded(uint256,uint256,bytes32[])` events. Its first guard filters by topic signature:

```rust
if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
    return Ok(());
}
``` [1](#0-0) 

This filtering is done **inside the hook function itself**, not in the dispatch layer. This is the canonical pattern for a globally-registered event hook that receives all events and self-filters — meaning any contract that emits an event whose first topic equals `INTEROP_ROOT_ADDED_EVENT_SIG` will cause this hook to execute.

Once the topic check passes, the hook immediately validates the data length:

```rust
if data.len() != 96 {
    return Err(internal_error!("Interop root reporter event hook received bad data").into());
}
``` [2](#0-1) 

`internal_error!(...)` produces a `SystemError::LeafDefect`. This is a **fatal internal error**, not a graceful revert. The comment on line 32 reads "Internal error if the data supplied doesn't match the expected value" — confirming the developers assumed only the trusted `MessageRoot`
