### Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any User to Bypass Per-User Allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is the caller, `sender` is always the router address. If the pool admin allowlists the router (a natural configuration to enable router-based trading), every end user — including those explicitly not allowlisted — can trade through the router, defeating the extension's purpose entirely.

### Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230
_beforeSwap(msg.sender, recipient, ...);
```

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making the router the `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol:72-80
IMetricOmmPoolActions(params.pool).swap(params.recipient, ...);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = router. The check resolves to `allowedSwapper[pool][router]`. The actual end user's address is never evaluated. Any user who calls `exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`) through the router inherits the router's allowlist status.

### Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict trading to a specific set of users (e.g., KYC-verified addresses, whitelisted counterparties) and allowlists the router to support standard router-based swaps inadvertently grants every user on-chain access to the gated pool. The allowlist invariant — that only individually approved addresses may swap — is silently broken. Non-allowlisted users can trade freely, violating the pool's access control model.

### Likelihood Explanation
The scenario requires the pool admin to allowlist the router address. This is the expected and natural configuration for any pool that wants to support `MetricOmmSimpleRouter` while also using the allowlist extension. The two features are designed to be composed, making this a likely misconfiguration that any operator would fall into.

### Recommendation
`SwapAllowlistExtension` should not rely solely on the `sender` argument passed by the pool. Two viable fixes:

1. **Pass the original initiator through `extensionData`**: The router encodes `msg.sender` (the end user) into `extensionData` before calling `pool.swap()`, and the extension decodes and checks it. This requires a convention between router and extension.
2. **Check `tx.origin` as a fallback** (weaker, not recommended for general use): Only appropriate in controlled environments.
3. **Preferred — router-aware allowlist**: Add a separate `allowedInitiator` mapping that the extension checks when `sender` is a known router, using the decoded initiator from `extensionData`.

### Proof of Concept
```solidity
// Setup: pool admin allowlists the router, NOT the end user
extension.setAllowedToSwap(pool, address(router), true);
// allowedSwapper[pool][user] = false (default)

// Attack: non-allowlisted user swaps through the router
vm.prank(nonAllowlistedUser);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: tokenIn,
    ...
}));
// Succeeds: allowedSwapper[pool][router] = true, user check never happens
```
