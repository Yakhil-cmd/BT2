Audit Report

## Title
Gzip Footer Spoofing Inflates `compilation_cost` in `create_execution_state`, Exhausting Round Instruction Budget — (`rs/execution_environment/src/hypervisor.rs`, `rs/embedders/src/wasm_utils/decoding.rs`)

## Summary

`decoded_wasm_size` reads the gzip ISIZE footer field directly as a `usize` with no cap, despite its own doc-comment warning the value is attacker-controlled. `create_execution_state` uses this uncapped value to compute `compilation_cost` before calling `wasm_executor.create_execution_state`. When the executor rejects the module as `ModuleTooLarge`, the error path subtracts the inflated cost (~25.7 trillion instructions) from `round_limits.instructions`, driving it deeply negative and causing `instructions_reached()` to return `true`, which breaks the scheduler's subnet message processing loop for the remainder of that round.

## Finding Description

**Root cause — uncapped gzip footer in `decoded_wasm_size`:**

`wasm_encoding_and_size` reads the last 4 bytes of any gzip stream as a `u32` and casts directly to `usize` with no bound check: [1](#0-0) 

The function's own doc-comment at lines 11–14 explicitly warns this value "cannot be trusted" and "could have been manipulated," yet no cap is applied before returning it. [2](#0-1) 

**Inflated `compilation_cost` computed before `decode_wasm` is called:**

In `create_execution_state`, `decoded_wasm_size` is called first, and the result is used to compute `compilation_cost` before the executor (which calls `decode_wasm`) is invoked: [3](#0-2) 

With a spoofed footer of `0xFFFFFFFF`, `wasm_size = max(4_294_967_295, tiny_compressed_len) = 4_294_967_295`. With `DEFAULT_COST_TO_COMPILE_WASM_INSTRUCTION = 6_000`: [4](#0-3) 

`compilation_cost = 6_000 × 4_294_967_295 = 25,769,803,770,000`.

**`decode_wasm` rejects the module as `ModuleTooLarge`:**

Inside `wasm_executor.create_execution_state`, `decode_wasm` checks `uncompressed_size > max_size` (100 MiB = 104,857,600). Since `4,294,967,295 > 104,857,600`, it returns `Err(ModuleTooLarge)` immediately: [5](#0-4) 

**Error path subtracts the inflated cost:**

The `Err` arm of the match in `create_execution_state` uses the `compilation_cost` variable computed from the spoofed footer (still in scope from line 160): [6](#0-5) 

`as_round_instructions` converts to `i64` with `unwrap_or(i64::MAX)`. Since 25.7 trillion fits in `i64::MAX` (~9.2 × 10^18), no saturation occurs: [7](#0-6) 

**Scheduler stops processing subnet messages:**

`instructions_reached()` returns `true` when `instructions <= 0`: [8](#0-7) 

The scheduler's subnet message loop breaks immediately: [9](#0-8) 

All remaining `install_code`, `update_settings`, and other subnet messages queued in the same round are skipped.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

One crafted `install_code` call per round starves all other subnet management operations (install_code, update_settings, etc.) for that round. The attack is repeatable every round, creating a sustained DoS on subnet management. The attacker's call fails with `ModuleTooLarge` (no canister state change), so the cost is only the cycles for a rejected call.

## Likelihood Explanation

The exploit requires no special privileges. Any canister developer can submit an `install_code` ingress message. The crafted gzip is trivially constructable: valid gzip magic (`\x1f\x8b\x08`) + any deflate body ≥ 8 bytes + footer `\xff\xff\xff\xff` + any 4-byte CRC32. The path is deterministic and locally reproducible. No key material, social engineering, or network-level attack is required.

## Recommendation

Cap `wasm_size` to `WASM_MAX_SIZE` before computing `compilation_cost` in `create_execution_state`:

```rust
// rs/execution_environment/src/hypervisor.rs
let wasm_size_result = decoded_wasm_size(canister_module.as_slice());
let wasm_size = match wasm_size_result {
    Ok(size) => std::cmp::max(size, canister_module.len()),
    Err(_) => canister_module.len(),
};
// Add this cap before computing compilation_cost:
let wasm_size = wasm_size.min(config.wasm_max_size.get() as usize);
let compilation_cost = self.cost_to_compile_wasm_instruction * wasm_size as u64;
```

This mirrors the guard already present in `decode_wasm` and ensures the pre-decompression cost estimate cannot exceed the cost of a legitimately maximum-sized module.

## Proof of Concept

```rust
// Craft a minimal gzip with ISIZE footer = 0xFFFFFFFF
let wasm: Vec<u8> = vec![
    0x1f, 0x8b, 0x08, 0x00, // magic + method
    0x00, 0x00, 0x00, 0x00, // mtime
    0x00, 0xff,             // xfl + OS
    0x03, 0x00,             // empty deflate stream (BFINAL=1, BTYPE=00)
    0x00, 0x00, 0x00, 0x00, // CRC32 (any value)
    0xff, 0xff, 0xff, 0xff, // ISIZE = u32::MAX
];
// Submit via install_code ingress message.
// Expected: round_limits.instructions drops by ~25.7 trillion.
// All other install_code/update_settings messages in the same round are not executed.
// The install_code itself fails with ModuleTooLarge.
```

A deterministic integration test using PocketIC or a local replica can confirm that after submitting this payload, a second `install_code` message queued in the same round is not executed (remains pending until the next round).
