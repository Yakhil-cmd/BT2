### Title
Permanent Loss of ERC-20 Tokens When `ft_transfer` Fails for Unregistered Recipient — (`engine-precompiles/src/native.rs`)

---

### Summary

In the production build of Aurora Engine (compiled with `contract` features only, without `error_refund`), when a user exits ERC-20 tokens to a NEAR recipient that is not registered with the NEP-141 token contract, the ERC-20 tokens are permanently burned with no corresponding NEP-141 credit and no refund. A 64-byte all-lowercase-hex implicit NEAR account ID is syntactically valid and passes all engine-side validation, but such an account may have no storage deposit with the NEP-141 contract, causing `ft_transfer` to fail silently with no recovery path.

---

### Finding Description

**Step 1 — Production build excludes `error_refund`.**

The `Makefile.toml` defines the production build features as:

```
CARGO_FEATURES_BUILD = "contract"
``` [1](#0-0) 

The `error_refund` feature is a separate opt-in flag not included in the production binary: [2](#0-1) 

**Step 2 — `AccountId::validate` accepts 64-byte hex implicit accounts.**

A 64-character all-lowercase-hex string (e.g., `e523efec9b66c4c8f6e708f1cf56be1399181e5b7c1e35f845670429faf143c2`) passes `AccountId::validate` because every character is in `[a-z0-9]`, length is exactly 64 (≤ `MAX_ACCOUNT_ID_LEN`), and no separator rules are violated: [3](#0-2) 

`parse_recipient` in `native.rs` accepts such an account ID without any additional existence or registration check: [4](#0-3) 

**Step 3 — ERC-20 tokens are burned before the NEAR-side promise executes.**

The ERC-20 exit flow burns tokens in the EVM first, then schedules an `ft_transfer` promise on the NEP-141 contract. The promise is a fire-and-forget `PromiseArgs::Create` when no callback is needed: [5](#0-4) 

**Step 4 — Without `error_refund`, `refund` is always `None` and no callback is registered for regular ERC-20 exits.**

When `error_refund` is not compiled in, `refund` is hardcoded to `None`: [6](#0-5) 

For a regular ERC-20 exit (non-wNEAR), `transfer_near_args` is also `None`: [7](#0-6) 

Because both fields are `None`, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the promise is a bare `PromiseArgs::Create` — no callback is attached: [8](#0-7) 

**Step 5 — If `ft_transfer` fails, the callback `exit_to_near_precompile_callback` is never called, and even if it were, `args.refund` is `None` so no refund executes.**

The callback only refunds when `args.refund` is `Some`: [9](#0-8) 

**Step 6 — The existing test explicitly confirms permanent token loss without `error_refund`.**

The test `test_exit_to_near_refund` exits to `"unregistered.near"` (not registered with the NEP-141 contract) and asserts:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [10](#0-9) 

This confirms that without `error_refund`, the ERC-20 balance is permanently reduced even when `ft_transfer` fails.

---

### Impact Explanation

When a user exits ERC-20 tokens to a 64-byte hex implicit NEAR account that has no storage deposit with the NEP-141 contract:

- The ERC-20 tokens are burned in the EVM (balance decremented).
- The `ft_transfer` promise fails on NEAR (recipient not registered).
- No callback is registered; no refund is issued.
- The NEP-141 balance of the Aurora contract is unchanged.
- The user's ERC-20 tokens are permanently destroyed with no NEAR-side credit.

**Impact: Critical — Permanent freezing/destruction of funds.**

---

### Likelihood Explanation

- Any user can trigger this by calling the ERC-20 `withdraw`/exit function with a 64-byte hex recipient that has no NEP-141 storage deposit.
- Implicit NEAR accounts are commonly generated from public keys and are not automatically registered with every NEP-141 contract.
- No admin compromise or special privilege is required.
- The production build (`CARGO_FEATURES_BUILD = "contract"`) does not include `error_refund`, so this affects all production deployments built with the standard Makefile target.

---

### Recommendation

Enable the `error_refund` feature in the production build (`CARGO_FEATURES_BUILD = "contract,error_refund"`), or add a pre-flight check in `exit_erc20_token_to_near` / `parse_recipient` that verifies the recipient is registered with the target NEP-141 contract before burning ERC-20 tokens. At minimum, document clearly that exits to unregistered accounts result in permanent token loss.

---

### Proof of Concept

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` already proves the behavior: [10](#0-9) 

To reproduce with a 64-byte hex implicit account:

1. Build Aurora Engine without `error_refund` (standard production build: `CARGO_FEATURES_BUILD = "contract"`).
2. Deploy a NEP-141 token and bridge it to an ERC-20 on Aurora.
3. Call the ERC-20 exit function with recipient `e523efec9b66c4c8f6e708f1cf56be1399181e5b7c1e35f845670429faf143c2` (a valid 64-byte hex implicit account with no storage deposit on the NEP-141 contract).
4. Observe: ERC-20 balance decremented to zero; NEP-141 balance of Aurora contract unchanged; no refund issued.

    pub fn validate(account_id: &str) -> Result<(), ParseAccountError> {
        if account_id.len() < MIN_ACCOUNT_ID_LEN {
            Err(ParseAccountError::TooShort)
        } else if account_id.len() > MAX_ACCOUNT_ID_LEN {
            Err(ParseAccountError::TooLong)
        } else {
            // Adapted from https://github.com/near/near-sdk-rs/blob/fd7d4f82d0dfd15f824a1cf110e552e940ea9073/near-sdk/src/environment/env.rs#L819

            // NOTE: We don't want to use Regex here, because it requires extra time to compile it.
            // The valid account ID regex is /^(([a-z\d]+[-_])*[a-z\d]+\.)*([a-z\d]+[-_])*[a-z\d]+$/
            // Instead the implementation is based on the previous character checks.

            // We can safely assume that last char was a separator.
            let mut last_char_is_separator = true;

            for c in account_id.bytes() {
                let current_char_is_separator = match c {
                    b'a'..=b'z' | b'0'..=b'9' => false,
                    b'-' | b'_' | b'.' => true,
                    _ => {
                        return Err(ParseAccountError::Invalid);
                    }
                };
                if current_char_is_separator && last_char_is_separator {
                    return Err(ParseAccountError::Invalid);
                }
                last_char_is_separator = current_char_is_separator;
            }

            (!last_char_is_separator)
                .then_some(())
                .ok_or(ParseAccountError::Invalid)
        }
    }

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}

    async fn test_exit_to_near_refund() {

            ft_owner,

            nep_141,
